HagiCode docs screenshot analysis context

Use this file as auxiliary context for screenshots managed under repos/docs.
Treat every statement here as a hint, not as ground truth. When the visible image
conflicts with this context, trust the pixels, on-screen text, layout, icons, and
other directly observable evidence from the screenshot.

Stable domain guidance:
- Screenshots usually come from HagiCode desktop, web, docs, or supporting tooling.
- Many docs screenshots specifically come from repos/web (PCode.Client), so expect a
  React web app with sidebar, header actions, drawers, tabs, cards, badges, and
  dashboard-style panels.
- Many other docs screenshots come from repos/hagicode-desktop, so expect an
  Electron desktop application with sidebar navigation, settings tabs, version and
  dependency cards, onboarding wizards, dialogs, and embedded web service controls.
- Common flows include installation, onboarding, project creation, proposal review,
  AI coding sessions, commit or workflow confirmation, settings, and service status.
- Interfaces may show dashboards, sidebars, tab bars, modal dialogs, forms, tables,
  progress indicators, code editors, terminals, markdown content, or browser pages.
- UI labels can be bilingual. English and Chinese labels may appear together or in
  separate screenshots for the same product area.
- Product screenshots often emphasize user actions such as creating a project,
  selecting a repository, configuring a model, starting a session, or confirming
  that a task completed successfully.

Stable repos/web style clues:
- The app commonly uses light, dark, and seasonal themes. Screenshots may have
  neutral light backgrounds, dark charcoal panels, or spring-like green accents.
- Primary actions frequently use vivid green accents; proposal and MonoSpecs related
  actions can appear in violet or purple tones; execute actions often lean teal or
  green; archive or warning actions can lean amber; destructive states use red.
- Validation and status surfaces often map to success, warning, error, and info
  colors rather than plain grayscale.
- Screens often contain rounded cards, bordered sidebars, color-coded badges, project
  avatar dots, and gradient AI action buttons instead of flat monochrome controls.

Stable repos/hagicode-desktop style clues:
- The desktop app commonly uses light and dark themes with teal-green primary accents,
  neutral cards, bordered sidebars, and rounded shadcn-style controls.
- Status surfaces and charts often use a green-to-teal family for healthy or active
  states, with red destructive/error actions and muted gray secondary surfaces.
- Some screens use glassmorphism-like panels with translucent cards, soft borders,
  blur, and subtle glow/shimmer animations rather than flat blocks.
- The renderer often uses dense dashboard cards, status badges, progress bars,
  installation logs, wizard step trackers, and sidebar-driven layouts optimized for
  desktop windows rather than narrow mobile views.

Recognition boundaries:
- Prefer literal, visible UI text over inferred product knowledge.
- Do not invent buttons, page names, or steps that are not visible.
- Do not assume a specific feature variant, release version, or tenant unless the
  screenshot itself provides evidence.
- Summaries should describe the main screen purpose, the key visible action, and the
  most relevant state shown in the image.
- Tags should stay at a stable feature or workflow level, such as installation,
  settings, desktop, session, project, workflow, editor, or confirmation.

Keep this context stable over time:
- Include durable semantics that help many screenshots in the docs repository.
- Exclude one-off debugging notes, ticket-specific instructions, temporary campaign
  names, or guesses that only apply to a single screenshot batch.
