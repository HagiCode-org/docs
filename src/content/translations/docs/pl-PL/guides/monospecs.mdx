## Przegląd

Monospecs to podejście HagiCode do zarządzania wieloma repozytoriami. Śledzi i zarządza podrepozytoriami `monospecs.yaml`i działa w przypadku każdego projektu, który potrzebuje jednego miejsca do koordynowania wielu niezależnych repozytoriów.

### Podstawowe pojęcia

**Plik konfiguracyjny**

`monospecs.yaml` stanowi rdzeń całego rozwiązania do zarządzania, definiując:
- Lista i lokalizacja podrepozytoriów
- Adres URL Git dla każdego repozytorium
- Wyświetlana nazwa i ikona
- Zachowanie integracyjne z OpenSpec

**Różnice w stosunku do zwykłych repozytoriów**

| Funkcja | Regularne repozytorium | Projekt zarządzany przez firmę Monospecs |
|----------|-------------------|---------------------------|
| **Lokalizacja OpenSpec** | Katalog główny `openspec/` | Katalog główny głównego repozytorium `openspec/` |
| **Zmień zakres** | Ograniczone do bieżącego repozytorium | Może obejmować wiele podrepozytoriów |
| **Czystość podrepozytorium** | Specyfikacje zmieszane z kodem | Specyfikacje ustalone w głównym repozytorium i podrepozytoriach są czystsze |
| **Automatyczne zatwierdzanie archiwum** | Wymaga ręcznego zatwierdzenia specyfikacji | `commit_when_archive: true` automatycznie zatwierdza specyfikacje do głównego repozytorium |

### Dlaczego warto używać Monospeców?

- **Ujednolicone zarządzanie**: Zarządzaj wszystkimi podrepozytoriami w jednym pliku konfiguracyjnym
- **Wsparcie automatyzacji**: zapewnia metadane repozytorium dla skryptów i narzędzi
- **Przyjazność AI**: Ustrukturyzowana konfiguracja pomaga agentowi AI zrozumieć relacje z repozytorium
- **Integracja OpenSpec**: Bezproblemowa integracja z przepływem pracy związanym z zarządzaniem zmianami
- **Czystsze podrepozytoria**: Specyfikacje są przechowywane w głównym repozytorium, podrepozytoria zawierają tylko rzeczywisty kod

### Obowiązujące scenariusze

Najpierw użyj Monospecs w takich scenariuszach:

**1. Wspólny rozwój obejmujący wiele repozytoriów**
Kiedy projekt jest podzielony na wiele niezależnych repozytoriów:
- Aplikacja frontendowa, usługa backendowa i aplikacja komputerowa opracowane osobno
- Należy koordynować wersje i zależności między repozytoriami
- Dokumentacja i projekty pomocnicze są oddzielone od kodu głównego

**2. Pomocnicze zarządzanie projektami**
Użyj Monospecs do zarządzania:
- Projekt dokumentacji
- Oficjalna strona internetowa
- Twórz narzędzia i skrypty
- Każde repozytorium utrzymuje własną kontrolę wersji

**3. Rozwój funkcji między repozytoriami**
Funkcja wymagająca zmian w wielu repozytoriach:
- Należy śledzić zmiany w jakich repozytoriach
- Automatycznie zatwierdzaj specyfikacje, aby poprawić repozytorium docelowe
- Koordynuj aktualizacje wersji i zależności

## Szybki start

### Inicjalizacja

Wykonaj poniższe kroki, aby dokończyć inicjalizację monospeców:

#### 1. Utwórz puste repozytorium git

Utwórz pusty `git` folder w głównym repozytorium (uwaga: to jest `git`, nie `repos/` - to jest konwencja monospecs):

```bash
mkdir git
cd git
git init
cd ..
```

#### 2. Skonfiguruj plik monospecs

Utwórz `monospecs.yaml` plik konfiguracyjny w katalogu głównym repozytorium głównego:

```yaml
version: "1.0"
commit_when_archive: true

repositories:
  - path: "repos/frontend"
    url: "https://github.com/your-org/frontend.git"
    displayName: "Frontend App"
    icon: "🌐"

  - path: "repos/backend"
    url: "https://github.com/your-org/backend.git"
    displayName: "Backend Service"
    icon: "⚙️"
```

**Elementy konfiguracji**:
| Element konfiguracji | Wpisz | Wymagane | Opis |
|----------------|------|----------|-------------|
| `version` | ciąg | Tak | Wersja pliku konfiguracyjnego |
| `commit_when_archive` | wartość logiczna | Nie | Czy automatycznie zatwierdzać specyfikacje podczas archiwizacji (nie zatwierdzaj kodu sub-repo) |
| `repositories` | tablica | Tak | Lista konfiguracji repozytorium |
| `repositories[].path` | ciąg | Tak | Ścieżka repozytorium (względem katalogu głównego projektu) |
| `repositories[].url` | ciąg | Tak | Adres URL repozytorium Git |
| `repositories[].displayName` | ciąg | Nie | Nazwa wyświetlana repozytorium |
| `repositories[].icon` | ciąg | Nie | Ikona repozytorium (emoji) |

### Dodaj podrepozytoria do głównego repozytorium

Dodaj wszystkie podrepozytoria do pliku `repos/` katalog. Ten katalog musi zostać **wykluczony z kontroli wersji Git w głównym repozytorium**, aby uniknąć śledzenia zawartości podrepozytorium:

```plaintext
# Exclude all sub-repositories
repos/
```

#### 3. Dodaj repozytorium nadrzędne do HagiCode

Jeśli korzystasz z aplikacji komputerowej HagiCode, otwórz aplikację i dodaj swoje główne repozytorium jako projekt. Kliknij przycisk „Dodaj projekt” lub „+” na ekranie głównym, wybierz główny folder repozytorium, a zobaczysz wszystkie podrepozytoria i informacje o ich statusie.

#### 4. Klonuj podrepozytoria

Skorzystaj z `monospecs.yaml` konfiguracja, aby sklonować wszystkie skonfigurowane podrepozytoria do `repos/` katalog:

Spowoduje to sklonowanie wszystkich podrepozytoriów na podstawie adresów URL określonych w pliku konfiguracyjnym.

## Struktura pliku konfiguracyjnego

### Konfiguracja na poziomie głównym

```yaml
version: "1.0"
commit_when_archive: true

repositories:
  # ... repository configurations
```

### Konfiguracja na poziomie repozytorium

Każde podrepozytorium może posiadać:
- Folder zmian OpenSpec (`openspec/`)
- Niezależny `.git/` katalog
- Opcjonalne `AGENTS.md` do konfiguracji agenta AI

## Operacje zarządzania repozytorium

### Dodaj nowe repozytorium

Podczas dodawania nowego podrepozytorium do pliku `monospecs.yaml` `repositories` tablica:

```yaml
repositories:
  - path: "repos/new-service"
    url: "https://github.com/HagiCode-org/new-service.git"
    displayName: "New Service"
    icon: "🆕"
```

### Zaktualizuj konfigurację repozytorium

Gdy zmienią się adresy URL repozytorium lub metadane:
1. Edytuj `monospecs.yaml` aby zaktualizować odpowiednie wpisy
2. Sprawdź, czy składnia YAML jest poprawna
3. Jeśli synchronizujesz zmiany, ręcznie zaktualizuj konfigurację lokalnego repozytorium

## Obsługa konfiguracji agenta AI

### AGENTÓW.md

The `AGENTS.md` plik zawiera wskazówki dotyczące repozytorium dla agenta AI:

**Informacje techniczne dotyczące stosu**
- Framework i narzędzia do budowania
- Konwencje kodowe i praktyki nazewnictwa
- Zarządzanie wersjami i konfiguracja wdrożenia

**Zachowania specyficzne dla projektu**
- Rozszerzenia konfiguracji i wymagania specjalne
- Wzorce integracji OpenSpec
- Potrzeby koordynacji między repozytoriami

### Korzyści ze sztucznej inteligencji

Używanie `AGENTS.md` umożliwia agentowi AI:
- Zrozumienie relacji i zależności w repozytorium
- Generuj kod i refaktoryzuj w wielu repozytoriach
- Zachowaj spójność w kontroli wersji i konfiguracjach

## Integracja z OpenSpecem

### Jak Monospecs współpracuje z OpenSpec

Niezależnie od tego, czy zarządzasz projektem zarządzanym przez monospecs, czy zwykłym repozytorium, możesz skorzystać z przepływu pracy OpenSpec HagiCode:

**Zwykłe repozytorium:**
- OpenSpec w katalogu głównym `openspec/` katalog
- Zmiany dotyczą tylko bieżącego repozytorium

**Projekt zarządzany przez firmę Monospecs:**
- OpenSpec czyta `monospecs.yaml` konfiguracja
- Zmiany śledzone przez podrepozytorium
- Specyfikacje są automatycznie przydzielane do właściwych celów

### Zachowanie archiwalne

Kiedy `commit_when_archive: true`:
- Specyfikacje są automatycznie przesyłane do głównego repozytorium
- Kod podrepozytoria nie jest zatwierdzony (pozostaje oddzielny)
- Upraszcza zarządzanie specyfikacjami — nie ma potrzeby ręcznego zatwierdzania specyfikacji dla podrepozytoriów

### Ręczne zatwierdzenia podrepozytorium

Podrepozytoria utrzymują niezależną kontrolę wersji:
- Zatwierdzaj rzeczywiste zmiany kodu we własnych repozytoriach
- Specyfikacje są zatwierdzane osobno (z głównego repozytorium podczas archiwizacji)

## Najlepsze praktyki

### Konwencje nazewnictwa repozytoriów

- Użyj kebab-case (małe litery i łączniki) jako ścieżki repozytorium
- Używaj prostych, opisowych nazw wyświetlanych w języku chińskim
- Używaj odpowiednich emoji jako ikon

### Wytyczne dotyczące nazw wyświetlanych

- Zachowaj zwięzłość i opis
- Używaj spójnych nazewnictwa w podobnych repozytoriach
- Wybierz ikony reprezentujące cel repozytorium

### Konserwacja konfiguracji

**Synchronizuj konfigurację z aktualnym stanem**
- Regularnie sprawdzaj `monospecs.yaml` odzwierciedla rzeczywistą strukturę repozytorium
- Aktualizuj lub usuwaj wpisy po dodaniu lub usunięciu repozytoriów
- Testuj zmiany, aby upewnić się, że konfiguracja jest poprawna

**Kontrola wersji konfiguracji**
- Śledź zmiany w plikach konfiguracyjnych za pomocą gita
- Udokumentuj przyczyny głównych zmian konfiguracji
- Oznacz wersje plików konfiguracyjnych, aby ułatwić ich przywrócenie

## Rozwiązywanie problemów

### Nie znaleziono pliku konfiguracyjnego

Jeśli `monospecs.yaml` nie znaleziono pliku w katalogu głównym projektu:
1. Sprawdź, czy plik istnieje we właściwej lokalizacji
2. Sprawdź, czy znajdujesz się we właściwym katalogu roboczym
3. Sprawdź, czy w nazwie pliku konfiguracyjnego nie ma literówek

### Błędy składni YAML

Typowe problemy ze składnią YAML:
- Nieprawidłowe wcięcie (użyj spacji, a nie tabulatora)
- Brakujące lub dodatkowe cudzysłowy wokół ciągów
- Nieprawidłowe typy danych (ciągi wymagają cudzysłowów, liczby nie)
- Brakujące wymagane pola (np `path` lub `url`)

### Nie wykryto repozytorium

Jeśli nowo dodane repozytorium nie pojawia się:
1. Sprawdź `path` jest poprawny w `monospecs.yaml`
2. Sprawdź, czy repozytorium zostało pomyślnie sklonowane
3. Sprawdź, czy repozytorium istnieje w oczekiwanej lokalizacji

### Błędy klonowania

Jeśli klonowanie repozytorium nie powiedzie się:
1. Sprawdź połączenie sieciowe
2. Sprawdź, czy adres URL repozytorium Git jest poprawny
3. Sprawdź problemy z uwierzytelnianiem
4. Przejrzyj stan kontenera Docker
5. Sprawdź dostępność miejsca na dysku
