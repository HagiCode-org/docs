## Budowanie problemów z GitHubem Integracja od podstaw: praktyka bezpośredniego połączenia frontendowego HagiCode

> Ten artykuł dokumentuje pełny proces integracji GitHub Issues z platformą HagiCode. Zbadamy, jak wykorzystać architekturę „bezpośrednie połączenie frontendu + minimalny backend”, aby osiągnąć bezpieczne uwierzytelnianie OAuth i efektywną synchronizację problemów przy jednoczesnym zachowaniu lekkości backendu.

## Tło: Dlaczego warto zintegrować GitHub?

Jako platforma programistyczna wspomagana sztuczną inteligencją, podstawowa wartość HagiCode polega na łączeniu pomysłów z realizacją. Jednak w praktyce odkryliśmy, że gdy użytkownicy wypełnią propozycję w HagiCode, często muszą ręcznie skopiować zawartość do GitHub Issues w celu śledzenia projektu.

Stwarza to kilka oczywistych problemów:

1.  **Rozdrobniony przepływ pracy**: Użytkownicy muszą przełączać się między dwoma systemami. Działanie nie jest płynne, a kluczowe informacje można łatwo utracić podczas kopiowania i wklejania.
2.  **Niewygodna współpraca**: Inni członkowie zespołu są przyzwyczajeni do sprawdzania zadań w GitHub i nie mogą bezpośrednio widzieć postępu propozycji w HagiCode.
3.  **Powtarzająca się praca ręczna**: Za każdym razem, gdy propozycja jest aktualizowana, ktoś musi ręcznie zaktualizować odpowiedni problem w GitHub, co zwiększa niepotrzebne koszty utrzymania.

Aby rozwiązać ten problem, postanowiliśmy wprowadzić funkcję **GitHub Issues Integration**, łączącą sesje HagiCode z repozytoriami GitHub, aby umożliwić „synchronizację jednym kliknięciem”.

## Wybór techniczny: bezpośrednie połączenie frontonu vs serwer proxy zaplecza

Projektując podejście integracyjne, mieliśmy przed sobą dwie opcje: tradycyjny „model proxy backendu” i bardziej agresywny „model bezpośredniego połączenia frontendowego”.

### Porównanie rozwiązań

W tradycyjnym **modelu proxy backendu** każde żądanie z frontendu musi najpierw przejść przez nasz backend, który następnie wywołuje API GitHub. Centralizuje to logikę, ale powoduje również znaczne obciążenie backendu:

1.  **Rozdęty backend**: musielibyśmy napisać dedykowane opakowanie klienta API GitHub, a także obsłużyć złożoną maszynę stanu OAuth.
2.  **Ryzyko tokena**: token użytkownika GitHub musiałby być przechowywany w bazie danych zaplecza. Nawet w przypadku szyfrowania zwiększa to powierzchnię bezpieczeństwa.
3.  **Koszt rozwoju**: potrzebowalibyśmy migracji baz danych do przechowywania tokenów i dodatkowej usługi synchronizacji do utrzymania.

**Model bezpośredniego połączenia frontendowego** jest znacznie lżejszy. W tym podejściu używamy backendu tylko dla najbardziej wrażliwego kroku „tajnej wymiany” (wywołanie zwrotne OAuth). Po uzyskaniu tokena przechowujemy go bezpośrednio w przeglądarce `localStorage`. Późniejsze operacje, takie jak tworzenie problemów i aktualizowanie komentarzy, są wysyłane bezpośrednio z frontendu do GitHub za pośrednictwem protokołu HTTP.

| Wymiar porównawczy | Model proxy zaplecza | Model bezpośredniego połączenia frontonu |
| :--- | :--- | :--- |
| **Złożoność backendu** | Wymaga pełnej usługi OAuth i klienta API GitHub | Wymaga tylko punktu końcowego wywołania zwrotnego OAuth |
| **Zarządzanie tokenami** | Muszą być zaszyfrowane i przechowywane w bazie danych, co wiąże się z ryzykiem wycieku | Zapisywane w przeglądarce i widoczne tylko dla użytkownika |
| **Koszt wdrożenia** | Wymaga migracji baz danych i rozwoju wielu usług | Głównie prace frontendowe |
| **Doświadczenie użytkownika** | Scentralizowana logika, ale opóźnienia serwera mogą być nieco większe | Niezwykle szybka reakcja dzięki bezpośredniej interakcji z GitHubem |

Ponieważ zależało nam na szybkiej integracji i minimalnych zmianach w backendie, **ostatecznie wybraliśmy „model bezpośredniego połączenia frontendowego”**. To jakby dać przeglądarce „tymczasową przepustkę”. Po uzyskaniu zgody przeglądarka może sama obsługiwać GitHub, bez konieczności każdorazowego proszenia administratora backendu o zgodę.

## Podstawowy projekt: przepływ danych i bezpieczeństwo

Po ustaleniu architektury musieliśmy zaprojektować konkretny przepływ danych. Istotą procesu synchronizacji jest bezpieczne zdobycie tokena i efektywne jego wykorzystanie.

### Ogólny schemat architektury

Cały system można podzielić na trzy role: przeglądarkę (frontend), backend HagiCode i GitHub.

```text
+--------------+        +--------------+        +--------------+
| Frontend     |        |   Backend    |        |   GitHub     |
| React        |        |   ASP.NET    |        |   REST API   |
|              |        |              |        |              |
|  +--------+  |        |              |        |              |
|  | OAuth  |--+--------> /callback    |        |              |
|  | Flow   |  |        |              |        |              |
|  +--------+  |        |              |        |              |
|              |        |              |        |              |
|  +--------+  |        |  +--------+  |        |  +--------+  |
|  | GitHub |  +------------> Session |  +----------> Issues |  |
|  | API    |  |        |  |Metadata|  |        |  |        |  |
|  | Direct |  |        |  +--------+  |        |  +--------+  |
|  +--------+  |        |              |        |              |
+--------------+        +--------------+        +--------------+
```

**Kluczową kwestią jest**: tylko jeden mały krok w OAuth (wymiana kodu na token) musi przejść przez backend. Następnie najcięższe prace (tworzenie problemów) są obsługiwane bezpośrednio pomiędzy frontendem a GitHubem.

### Szczegóły synchronizacji przepływu danych

Kiedy użytkownik kliknie przycisk „Synchronizuj z GitHub” w interfejsie użytkownika HagiCode, następuje seria złożonych działań:

```text
User clicks "Sync to GitHub"
         │
         ▼
1. Frontend checks localStorage for the GitHub token
         │
         ▼
2. Format issue content (convert the Proposal into Markdown)
         │
         ▼
3. Frontend directly calls the GitHub API to create/update the issue
         │
         ▼
4. Call the HagiCode backend API to update Session.metadata (store issue URL and other info)
         │
         ▼
5. Backend broadcasts the SessionUpdated event via SignalR
         │
         ▼
6. Frontend receives the event and updates the UI to show the "Synced" state
```

### Projekt zabezpieczeń

Bezpieczeństwo jest zawsze najwyższym priorytetem podczas integracji usług stron trzecich. Podjęliśmy następujące rozważania:

1.  **Broń się przed atakami CSRF**: Generuj losowo `state` parametr podczas przekierowania OAuth i zapisz go w `sessionStorage`. Ściśle sprawdzaj stan wywołania zwrotnego, aby zapobiec sfałszowanym żądaniom.
2.  **Izolowane przechowywanie tokenów**: Token jest przechowywany tylko w przeglądarce `localStorage`. Dzięki Polityce Same-Origin tylko skrypty HagiCode mogą ją odczytać, co pozwala uniknąć ryzyka wycieku bazy danych po stronie serwera, który wpłynie na użytkowników.
3.  **Granice błędów**: Zaprojektowaliśmy dedykowaną obsługę typowych błędów API GitHub (takich jak wygasły token 401, błąd sprawdzania poprawności 422 i ograniczenie szybkości 429), dzięki czemu użytkownicy otrzymują przyjazną informację zwrotną.

## W praktyce: szczegóły implementacji w kodzie

Teoria sięga tylko tak daleko. Przyjrzyjmy się, jak faktycznie działa kod.

### 1. Minimalne zmiany w backendie

Backend musi zrobić tylko dwie rzeczy: przechowywać informacje o synchronizacji i obsługiwać wywołanie zwrotne OAuth.

**Zmiany w bazie danych**
Musimy tylko dodać a `Metadata` kolumna do `Sessions` tabela do przechowywania danych rozszerzenia w formacie JSON.

### 2. Przepływ OAuth frontonu

To jest punkt wejścia połączenia. Używamy standardowego przepływu kodu autoryzacyjnego.

```typescript
// src/HagiCode.Client/src/services/githubOAuth.ts

// Generate the authorization URL and redirect
export async function generateAuthUrl(): Promise<string> {
  const state = generateRandomString(); // Generate a random string for CSRF protection
  sessionStorage.setItem('hagicode_github_state', state);
  
  const params = new URLSearchParams({
    client_id: clientId,
    redirect_uri: window.location.origin + '/settings?tab=github&oauth=callback',
    scope: ['repo', 'public_repo'].join(' '),
    state: state,
  });
  
  return `https://github.com/login/oauth/authorize?${params.toString()}`;
}

// Handle the code-to-token exchange on the callback page
export async function exchangeCodeForToken(code: string, state: string): Promise<GitHubToken> {
  // 1. Validate state to prevent CSRF
  const savedState = sessionStorage.getItem('hagicode_github_state');
  if (state !== savedState) throw new Error('Invalid state parameter');

  // 2. Call the backend API to exchange the token
  // Note: this must go through the backend because ClientSecret cannot be exposed to the frontend
  const response = await fetch('/api/GitHubOAuth/callback', {
    method: 'POST',
    headers: { 'Content-Type': 'application/json' },
    body: JSON.stringify({ code, state, redirectUri: window.location.origin + '/settings?tab=github&oauth=callback' }),
  });

  if (!response.ok) throw new Error('Failed to exchange token');
  
  const token = await response.json();
  
  // 3. Save into LocalStorage
  saveToken(token);
  return token;
}
```

### 3. Opakowanie klienta API GitHub

Kiedy już będziemy mieli token, potrzebujemy solidnego narzędzia do wywołania API GitHuba.

```typescript
// src/HagiCode.Client/src/services/githubApiClient.ts

const GITHUB_API_BASE = 'https://api.github.com';

// Core request wrapper
async function githubApi<T>(endpoint: string, options: RequestInit = {}): Promise<T> {
  const token = localStorage.getItem('gh_token');
  if (!token) throw new Error('Not connected to GitHub');
  
  const response = await fetch(`${GITHUB_API_BASE}${endpoint}`, {
    ...options,
    headers: {
      ...options.headers,
      Authorization: `Bearer ${token}`,
      Accept: 'application/vnd.github.v3+json', // Specify the API version
    },
  });
  
  // Error handling logic
  if (!response.ok) {
    if (response.status === 401) throw new Error('GitHub token expired, please reconnect');
    if (response.status === 403) throw new Error('No permission to access this repository or rate limit exceeded');
    if (response.status === 422) throw new Error('Issue validation failed, the title may be duplicated');
    throw new Error(`GitHub API Error: ${response.statusText}`);
  }
  
  return response.json();
}

// Create issue
export async function createIssue(owner: string, repo: string, data: { title: string, body: string, labels: string[] }) {
  return githubApi(`/repos/${owner}/${repo}/issues`, {
    method: 'POST',
    body: JSON.stringify(data),
  });
}
```

### 4. Formatowanie i synchronizacja treści

Ostatnim krokiem jest konwersja danych sesji HagiCode do formatu zgłoszenia GitHub. To trochę jak z pracą tłumaczeniową.

```typescript
// Convert a Session object into a Markdown string
function formatIssueForSession(session: Session): string {
  let content = `# ${session.title}\n\n`;
  content += `**> HagiCode Session:** #${session.code}\n`;
  content += `**> Status:** ${session.status}\n\n`;
  content += `## Description\n\n${session.description || 'No description provided.'}\n\n`;
  
  // If this is a Proposal session, add extra fields
  if (session.type === 'proposal') {
    content += `## Chief Complaint\n\n${session.chiefComplaint || ''}\n\n`;
    // Add a deep link so users can jump back from GitHub to HagiCode
    content += `---\n\n**[View in HagiCode](hagicode://sessions/${session.id})**\n`;
  }
  
  return content;
}

// Main logic when clicking the sync button
const handleSync = async (session: Session) => {
  try {
    const repoInfo = parseRepositoryFromUrl(session.repoUrl); // Parse the repository URL
    if (!repoInfo) throw new Error('Invalid repository URL');

    toast.loading('Syncing to GitHub...');
    
    // 1. Format content
    const issueBody = formatIssueForSession(session);
    
    // 2. Call API
    const issue = await githubApiClient.createIssue(repoInfo.owner, repoInfo.repo, {
      title: `[HagiCode] ${session.title}`,
      body: issueBody,
      labels: ['hagicode', 'proposal', `status:${session.status}`],
    });
    
    // 3. Update Session Metadata (save the issue link)
    await SessionsService.patchApiSessionsSessionId(session.id, {
      metadata: {
        githubIssue: {
          owner: repoInfo.owner,
          repo: repoInfo.repo,
          issueNumber: issue.number,
          issueUrl: issue.html_url,
          syncedAt: new Date().toISOString(),
        }
      }
    });

    toast.success('Synced successfully!');
  } catch (err) {
    console.error(err);
    toast.error('Sync failed, please check your token or network');
  }
};
```

## Podsumowanie i perspektywy

Dzięki podejściu „bezpośredniego połączenia frontendowego” osiągnęliśmy bezproblemową integrację problemów z GitHub przy najmniejszej możliwej liczbie kodu backendu.

### Kluczowe zyski
1.  **Wysoka wydajność programowania**: Zmiany w backendie są minimalne, głównie jedno dodatkowe pole bazy danych i prosty punkt końcowy wywołania zwrotnego OAuth. Większość logiki jest realizowana na interfejsie użytkownika.
2.  **Silne bezpieczeństwo**: Token nie przechodzi przez bazę danych serwera, co zmniejsza ryzyko wycieku.
3.  **Doskonałe doświadczenie użytkownika**: Żądania są inicjowane bezpośrednio z frontendu, więc szybkość reakcji jest szybka i nie ma potrzeby przekazywania dalej z backendu.

### Rzeczy, na które należy zwrócić uwagę
Podczas prawdziwego wdrażania należy pamiętać o kilku pułapkach:
- **Ustawienia aplikacji OAuth**: Pamiętaj o wpisaniu poprawnych danych `Authorization callback URL` w ustawieniach aplikacji GitHub OAuth (zazwyczaj `http://localhost:3000/settings?tab=github&oauth=callback`).
- **Limity szybkości**: API GitHub dość ściśle ogranicza nieuwierzytelnione żądania, ale w przypadku tokena limit jest zwykle wystarczający (5000 żądań/godzinę).
- **Przetwarzanie adresów URL**: użytkownicy wprowadzają wszelkiego rodzaju adresy URL repozytoriów, więc upewnij się, że wyrażenie regularne pasuje do `.git` sufiksy, formaty SSH i podobne przypadki.

### Przyszłe ulepszenia
Obecną funkcją jest nadal synchronizacja jednokierunkowa (HagiCode -> GitHub). W przyszłości planujemy wdrożyć synchronizację dwukierunkową za pośrednictwem webhooków GitHub. Na przykład, jeśli problem zostanie zamknięty w GitHubie, stan sesji po stronie HagiCode może również zostać zaktualizowany automatycznie. Będzie to wymagało od nas udostępnienia punktu końcowego webhooka na zapleczu, co będzie interesującym kolejnym krokiem.
