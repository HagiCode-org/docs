## Od wywołań CLI do integracji SDK: najlepsze praktyki dotyczące GitHub Copilot w projektach .NET

> Ścieżka aktualizacji od wywołania wiersza poleceń do oficjalnej integracji z pakietem SDK była długą podróżą. Dzisiaj chcę podzielić się pułapkami, na jakie wpadliśmy i czego się nauczyliśmy podczas tworzenia HagiCode.

## Tło

Po oficjalnym wydaniu pakietu SDK GitHub Copilot SDK w 2025 r. zaczęliśmy integrować go z naszą warstwą możliwości sztucznej inteligencji. Wcześniej w projekcie korzystano głównie z GitHub Copilot, bezpośrednio wywołując narzędzie wiersza poleceń Copilot CLI, ale takie podejście wiązało się z kilkoma oczywistymi problemami:

- **Złożone zarządzanie procesami**: Musieliśmy ręcznie zarządzać cyklem życia procesu CLI, limitami czasu uruchamiania i czyszczeniem procesów. Procesy mogą ulec awarii bez ostrzeżenia.
- **Niekompletna obsługa zdarzeń**: Nieprzetworzone wywołanie CLI utrudnia przechwytywanie szczegółowych zdarzeń na podstawie rozumowania modelu i wykonywania narzędzi. To tak, jakby widzieć tylko wynik bez procesu myślenia.
- **Trudne zarządzanie sesją**: Nie istniał skuteczny mechanizm ponownego wykorzystania i odzyskiwania sesji, więc każda interakcja musiała zaczynać się od nowa.
- **Problemy ze zgodnością**: Argumenty CLI często się zmieniały, co oznaczało, że musieliśmy stale utrzymywać logikę zgodności dla tych parametrów.

Kwestie te stawały się coraz bardziej widoczne w codziennym rozwoju, zwłaszcza gdy musieliśmy śledzić rozumowanie modelu (`thinking`) i status wykonania narzędzia w czasie rzeczywistym. W tym momencie stało się jasne, że potrzebujemy pełniejszego podejścia do integracji na niższym poziomie.

## Projekt Architektury

Projekt wykorzystuje architekturę warstwową, aby rozwiązać ograniczenia wywoływania CLI:

```
┌─────────────────────────────────────────────────────────┐
│  hagicode-core (Orleans Grains + AI Provider Layer)    │
│  - CopilotAIProvider: Converts AIRequest to CopilotOptions │
│  - GitHubCopilotGrain: Orleans distributed execution interface │
└─────────────────────────────────────────────────────────┘
                          ↓
┌─────────────────────────────────────────────────────────┐
│  HagiCode.Libs (Shared Provider Layer)                 │
│  - CopilotProvider: CLI Provider interface implementation │
│  - ICopilotSdkGateway: SDK invocation abstraction      │
│  - GitHubCopilotSdkGateway: SDK session management and event dispatch │
└─────────────────────────────────────────────────────────┘
                          ↓
┌─────────────────────────────────────────────────────────┐
│  GitHub Copilot SDK (Official .NET SDK)                │
│  - CopilotClient: SDK client                           │
│  - CopilotSession: Session management                  │
│  - SessionEvent: Event stream                          │
└─────────────────────────────────────────────────────────┘
```

Ta warstwowa konstrukcja niesie ze sobą kilka praktycznych zalet technicznych:

1. **Oddzielenie problemów**: podstawowa logika biznesowa jest oddzielona od szczegółów implementacji pakietu SDK.
2. **Testowalność**: The `ICopilotSdkGateway` interfejs sprawia, że testowanie jednostkowe jest proste.
3. **Ponowne użycie**: `HagiCode.Libs` można odwoływać się do wielu projektów.
4. **Łatwość konserwacji**: aktualizacje SDK wymagają jedynie zmian w warstwie Gateway, podczas gdy górne warstwy pozostają niezmienione.

## Implementacja rdzenia

### Przepływ uwierzytelniania

Uwierzytelnianie jest pierwszym i najważniejszym krokiem integracji SDK. Jeśli uwierzytelnienie nie powiedzie się, nic innego się nie liczy. Zaprojektowaliśmy elastyczną konfigurację uwierzytelniania, która obsługuje wiele źródeł uwierzytelniania:

```csharp
// CopilotProvider.cs - Authentication source configuration
public class CopilotOptions
{
    public bool UseLoggedInUser { get; set; } = true;
    public string? GitHubToken { get; set; }
    public string? CliUrl { get; set; }
}

// Convert to SDK request
return new CopilotSdkRequest(
    GitHubToken: options.AuthSource == CopilotAuthSource.GitHubToken
        ? options.GitHubToken
        : null,
    UseLoggedInUser: options.AuthSource != CopilotAuthSource.GitHubToken
);
```

Korzyści z tego projektu są dość oczywiste:

- Obsługuje tryb zalogowanego użytkownika bez konieczności posiadania tokena, co dobrze pasuje do scenariuszy komputerów stacjonarnych.
- Obsługuje tryb GitHub Token, który jest odpowiedni do wdrożeń po stronie serwera i scentralizowanego zarządzania.
- Obsługuje zastępowanie adresu URL interfejsu CLI oprogramowania Copilot, co ułatwia konfigurację serwera proxy w przedsiębiorstwie.

W praktyce ten elastyczny model uwierzytelniania znacznie uprościł konfigurację w różnych scenariuszach wdrażania. Klient stacjonarny może używać własnego stanu logowania Copilot każdego użytkownika, podczas gdy strona serwera może centralnie zarządzać uwierzytelnianiem za pomocą tokenów.

### Obsługa strumienia zdarzeń

Jedną z najpotężniejszych możliwości pakietu SDK jest pełne przechwytywanie strumienia zdarzeń. Wdrożyliśmy system wysyłki zdarzeń, który jest w stanie obsłużyć w czasie rzeczywistym wszelkiego rodzaju zdarzenia SDK:

```csharp
// GitHubCopilotSdkGateway.cs - Core logic for event dispatch
internal static SessionEventDispatchResult DispatchSessionEvent(
    SessionEvent evt, bool sawDelta)
{
    switch (evt)
    {
        case AssistantReasoningEvent reasoningEvent:
            // Capture the model reasoning process
            events.Add(new CopilotSdkStreamEvent(
                CopilotSdkStreamEventType.ReasoningDelta,
                Content: reasoningEvent.Data.Content));
            break;

        case ToolExecutionStartEvent toolStartEvent:
            // Capture the start of a tool invocation
            events.Add(new CopilotSdkStreamEvent(
                CopilotSdkStreamEventType.ToolExecutionStart,
                ToolName: toolStartEvent.Data.ToolName,
                ToolCallId: toolStartEvent.Data.ToolCallId));
            break;

        case ToolExecutionCompleteEvent toolCompleteEvent:
            // Capture tool completion and its result
            events.Add(new CopilotSdkStreamEvent(
                CopilotSdkStreamEventType.ToolExecutionEnd,
                Content: ExtractToolExecutionContent(toolCompleteEvent)));
            break;

        default:
            // Preserve unhandled events as RawEvent
            events.Add(new CopilotSdkStreamEvent(
                CopilotSdkStreamEventType.RawEvent,
                RawEventType: evt.GetType().Name));
            break;
    }
}
```

Wartość tego wdrożenia jest znacząca:

- **Pełne ujęcie procesu rozumowania modelu** (`thinking`): użytkownicy mogą zobaczyć, jak AI rozumuje, a nie tylko ostateczną odpowiedź.
- **Śledzenie stanu wykonania narzędzia w czasie rzeczywistym**: wiemy, które narzędzia są uruchomione, kiedy się kończą i co zwracają.
- **Zerowa strata na zdarzeniach**: poprzez powrót do `RawEvent`, dbamy o to, aby każde wydarzenie zostało zarejestrowane.

W HagiCode te szczegółowe zdarzenia pomagają użytkownikom zrozumieć, jak wewnętrznie działa sztuczna inteligencja, szczególnie podczas debugowania złożonych zadań.

### Obsługa zgodności z CLI

Po migracji z wywoływania CLI do zestawu SDK odkryliśmy, że niektóre istniejące parametry CLI nie są już stosowane w zestawie SDK. Aby zachować kompatybilność wsteczną, wdrożyliśmy system filtrowania parametrów:

```csharp
// CopilotCliCompatibility.cs - Argument filtering
private static readonly Dictionary<string, string> RejectedFlags = new()
{
    ["--headless"] = "Unsupported startup argument",
    ["--model"] = "Passed through an SDK-native field",
    ["--prompt"] = "Passed through an SDK-native field",
    ["--interactive"] = "Interaction is managed by the provider",
};

public static CopilotCliArgumentBuildResult BuildCliArgs(CopilotOptions options)
{
    // Filter out unsupported arguments and keep compatible ones
    // Generate diagnostic information
}
```

Daje nam to kilka korzyści:

- Automatycznie filtruje niezgodne argumenty CLI, aby zapobiec błędom w czasie wykonywania.
- Generuje jasne komunikaty diagnostyczne, które pomagają programistom szybko zlokalizować problemy.
- Utrzymuje stabilność pakietu SDK i izoluje go od zmian parametrów CLI.

Podczas aktualizacji ten mechanizm zgodności pomógł nam w płynnym przejściu. Można było nadal korzystać z istniejących plików konfiguracyjnych i wymagać jedynie stopniowych dostosowań w oparciu o wyniki diagnostyki.

### Łączenie środowiska wykonawczego

Tworzenie sesji Copilot SDK jest stosunkowo drogie, a zbyt częste tworzenie i niszczenie sesji pogarsza wydajność. Aby rozwiązać ten problem, wdrożyliśmy system zarządzania pulą sesji:

```csharp
// CopilotProvider.cs - Session pool management
await using var lease = await _poolCoordinator.AcquireCopilotRuntimeAsync(
    request,
    async ct => await _gateway.CreateRuntimeAsync(sdkRequest, ct),
    cancellationToken);

if (lease.IsWarmLease)
{
    // Reuse an existing session
    yield return CreateSessionReusedMessage();
}

await foreach (var eventData in lease.Entry.Resource.SendPromptAsync(...))
{
    yield return MapEvent(eventData);
}
```

Korzyści z łączenia sesji obejmują:

- **Ponowne wykorzystanie sesji**: żądania z tym samym `sessionId` może ponownie wykorzystać istniejące sesje i zmniejszyć obciążenie związane z uruchamianiem.
- **Wsparcie odzyskiwania sesji**: po przerwie w działaniu sieci można przywrócić poprzedni stan sesji.
- **Automatyczne zarządzanie pulami**: wygasłe sesje są automatycznie czyszczone, aby uniknąć wycieków zasobów.

W HagiCode łączenie sesji zauważalnie poprawiło responsywność, szczególnie w przypadku ciągłych rozmów.

### Integracja Orleanu

HagiCode używa Orleans jako rozproszonego frameworka, a my zintegrowaliśmy Copilot SDK z Orleans Grains:

```csharp
// GitHubCopilotGrain.cs - Distributed execution
public async IAsyncEnumerable<GitHubCopilotResponse> ExecuteCommandStreamAsync(
    string command,
    CancellationToken token = default)
{
    var provider = await aiProviderFactory.GetProviderAsync(AIProviderType.GitHubCopilot);

    await foreach (var chunk in provider.SendMessageAsync(request, null, token))
    {
        // Map to the unified response format
        yield return BuildChunkResponse(chunk, startedAt);
    }
}
```

Korzyści z integracji Orleanu są znaczne:

- **Atrakcja ujednoliconego dostawcy AI**: przełączanie między różnymi dostawcami AI staje się łatwe.
- **Izolacja wielu dzierżawców**: Sesje Copilot dla różnych użytkowników pozostają od siebie odizolowane.
- **Stały stan sesji**: stan sesji można przywrócić nawet po ponownym uruchomieniu serwera.

W scenariuszach, które wymagają obsługi dużej liczby jednoczesnych żądań, Orleans zapewnia dużą skalowalność.

## Praktyczny przewodnik

### Przykład konfiguracji

Oto pełny przykład konfiguracji:

```json
{
  "AI": {
    "Providers": {
      "Providers": {
        "GitHubCopilot": {
          "Enabled": true,
          "ExecutablePath": "copilot",
          "Model": "gpt-5",
          "WorkingDirectory": "/path/to/project",
          "Timeout": 7200,
          "StartupTimeout": 30,
          "UseLoggedInUser": true,
          "NoAskUser": true,
          "Permissions": {
            "AllowAllTools": false,
            "AllowedTools": ["Read", "Bash", "Grep"],
            "DeniedTools": ["Edit"]
          }
        }
      }
    }
  }
}
```

### Uwagi dotyczące użytkowania

W praktyce podsumowaliśmy kilka punktów, na które warto zwrócić uwagę:

**Konfiguracja limitu czasu uruchamiania**: Pierwsze uruchomienie Copilot CLI może zająć stosunkowo dużo czasu, dlatego zalecamy ustawienie `StartupTimeout` do co najmniej 30 sekund. Jeśli jest to pierwsze logowanie, może to potrwać jeszcze dłużej.

**Zarządzanie uprawnieniami**: W środowiskach produkcyjnych unikaj używania `AllowAllTools: true`. Skorzystaj z `AllowedTools` listę dozwolonych, aby kontrolować, które narzędzia są dostępne i korzystać z `DeniedTools` lista odrzuconych, aby blokować niebezpieczne operacje. To skutecznie zapobiega wykonywaniu przez sztuczną inteligencję ryzykownych poleceń.

**Zarządzanie sesją**: Żądania z tym samym `sessionId` automatycznie ponownie wykorzystuje sesje. Stan sesji jest utrzymywany `ProviderSessionId`. Anulowanie jest propagowane za pośrednictwem `CancellationTokenSource`.

**Wyjście diagnostyczne**: Niekompatybilne argumenty CLI generują komunikaty typu `diagnostic`. Surowe zdarzenia SDK są zachowywane jako `event.raw`. Komunikaty o błędach obejmują takie kategorie, jak przekroczenie limitu czasu uruchamiania i niezgodność argumentów, co ułatwia rozwiązywanie problemów.

### Najlepsze praktyki

W oparciu o nasze praktyczne doświadczenie, oto kilka najlepszych praktyk:

**1. Użyj listy dozwolonych narzędzi**

```csharp
var request = new AIRequest
{
    Prompt = "Analyze this file",
    AllowedTools = new[] { "Read", "Grep", "Bash(git:*)" }
};
```

Jawne określenie dozwolonych narzędzi na liście dozwolonych pomaga zapobiegać nieoczekiwanym działaniom sztucznej inteligencji. Jest to szczególnie ważne w przypadku narzędzi posiadających uprawnienia do zapisu, takich jak `Edit`, z którymi należy obchodzić się ze szczególną ostrożnością.

**2. Ustaw rozsądne limity czasu**

```csharp
options.Timeout = 3600;  // 1 hour
options.StartupTimeout = 60;  // 1 minute
```

Ustaw odpowiednie wartości limitu czasu w oparciu o złożoność zadania. Jeśli wartość będzie za krótka, zadania mogą zostać przerwane. Jeśli jest zbyt długi, zasoby mogą zostać zmarnowane w oczekiwaniu na żądania, które nie odpowiadają.

**3. Włącz ponowne wykorzystanie sesji**

```csharp
options.SessionId = "my-session-123";
```

Używając tego samego `sessionId` dla powiązanych zadań pozwala ponownie wykorzystać kontekst poprzedniej sesji i poprawić szybkość reakcji.

**4. Obsługuj odpowiedzi przesyłane strumieniowo**

```csharp
await foreach (var chunk in provider.StreamAsync(request))
{
    switch (chunk.Type)
    {
        case StreamingChunkType.ThinkingDelta:
            // Handle the reasoning process
            break;
        case StreamingChunkType.ToolCallDelta:
            // Handle tool invocation
            break;
        case StreamingChunkType.ContentDelta:
            // Handle text output
            break;
    }
}
```

Odpowiedzi przesyłane strumieniowo umożliwiają pokazywanie postępu przetwarzania AI w czasie rzeczywistym, co poprawia komfort użytkownika. Jest to szczególnie cenne w przypadku długotrwałych zadań.

**5. Obsługa błędów i ponowne próby**

```csharp
try
{
    await foreach (var chunk in provider.StreamAsync(request))
    {
        // Handle the response
    }
}
catch (CopilotSessionException ex)
{
    // Handle session exceptions
    logger.LogError(ex, "Copilot session failed");
    // Decide whether to retry based on the exception type
}
```

Właściwa obsługa błędów i logika ponawiania poprawiają ogólną stabilność systemu.

Aktualizacja z wywoływania CLI do integracji SDK zapewniła projektowi HagiCode znaczną wartość:

- **Poprawiona stabilność**: pakiet SDK zapewnia bardziej stabilny interfejs, na który nie mają wpływu zmiany wersji CLI.
- **Większa funkcjonalność**: przechwytuje pełny strumień zdarzeń, w tym rozumowanie i stan wykonania narzędzia.
- **Wyższa wydajność programowania**: bezpieczny dla typu interfejs SDK sprawia, że programowanie jest bardziej wydajne i redukuje błędy w czasie wykonywania.
- **Lepsze doświadczenie użytkownika**: informacje zwrotne o zdarzeniach w czasie rzeczywistym pozwalają użytkownikom lepiej zrozumieć, co robi sztuczna inteligencja.

To uaktualnienie nie polegało jedynie na zastąpieniu wdrożenia technicznego. Była to także optymalizacja architektoniczna całej warstwy możliwości AI. Dzięki warstwowemu projektowaniu i interfejsom abstrakcji zyskaliśmy lepszą łatwość konserwacji i rozszerzalność.
