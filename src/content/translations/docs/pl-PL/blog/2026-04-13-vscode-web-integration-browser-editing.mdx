## VSCode i serwer kodu: wybór rozwiązania do edycji kodu w przeglądarce

> Tworząc możliwości edycji kodu w przeglądarce, programiści stają przed kluczowym wyborem: użyj oficjalnego narzędzia VSCode `code serve-web` funkcję lub przyjąć funkcję kierowaną przez społeczność `code-server` rozwiązanie? Decyzja ta wpływa nie tylko na architekturę techniczną, ale także na zgodność z licencjami i elastyczność wdrażania.

## Tło

Wybór techniczny jest bardzo podobny do wyboru ścieżki życiowej. Kiedy już wybierzesz jedną, zwykle musisz iść dalej, a późniejsza zmiana może być bardzo kosztowna.

W dobie programowania wspomaganego sztuczną inteligencją edycja kodu w przeglądarce staje się coraz ważniejsza. Użytkownicy oczekują, że po zakończeniu analizy kodu przez asystenta AI będą mogli od razu otworzyć edytor w tej samej sesji przeglądarki i wprowadzić zmiany bez konieczności przełączania aplikacji. Tego rodzaju bezproblemowe działanie powinno być po prostu dostępne, kiedy tego potrzebujesz.

Jednak wdrażając tę funkcję, programiści stają przed krytycznym wyborem technicznym: czy powinni używać oficjalnego VSCode `code serve-web` funkcja lub kierowana przez społeczność `code-server` rozwiązanie?

Każda opcja ma swoje mocne strony i kompromisy, a zły wybór może później spowodować wiele problemów. Licencjonowanie jest jednym z przykładów: jeśli dopiero po wprowadzeniu na rynek odkryjesz, że Twój produkt nie jest zgodny z licencją, jest już za późno. Wdrożenie to kolejna sprawa: rozwiązanie może doskonale działać w fazie rozwojowej, a po przeniesieniu do kontenerów napotkać różnego rodzaju problemy. Właśnie takich pułapek zespoły chcą uniknąć.

To jest najbardziej zasadnicza różnica między obydwoma rozwiązaniami i pierwszy czynnik, który wzięliśmy pod uwagę podczas oceny. Dokonując wyboru technicznego, ważne jest, aby od razu zrozumieć ryzyko prawne.

- Można go bezpłatnie używać komercyjnie, modyfikować i rozpowszechniać
- Brak ograniczeń dotyczących scenariuszy użytkowania

**VSCod `code serve-web`**
- Część produktu Microsoft VSCode
- Korzysta z licencji Microsoftu (licencja VS Code zawiera ograniczenia dotyczące użytku komercyjnego)

Z punktu widzenia licencji serwer kodu jest bardziej przyjazny dla projektów komercyjnych. Należy to przemyśleć podczas planowania produktu, ponieważ późniejsza migracja może okazać się bardzo kosztowna.

## Różnice we wdrażaniu

Po ustaleniu kwestii licencjonowania następną kwestią jest wdrożenie. Ma to bezpośredni wpływ na koszty operacyjne i projekt architektoniczny.

**serwer kodu**
- Samodzielna aplikacja Node.js, którą można wdrożyć niezależnie
- Obsługuje wiele źródeł środowiska wykonawczego:
  - Bezpośrednie określenie ścieżki pliku wykonywalnego
  - Sprawdzam to poprzez systemową PATH
  - Automatyczne wykrywanie środowiska NVM Node.js 22.x
- Nie ma potrzeby instalowania aplikacji komputerowej VSCode na serwerze
- Łatwiejsze wdrażanie w kontenerach

**VSCod `code serve-web`**
- Musi zależeć od lokalnie zainstalowanego interfejsu CLI VSCode
- Wymaga dostępnego `code` polecenie na komputerze hosta
- System odfiltrowuje opakowania CLI VS Code Remote
- Zaprojektowane przede wszystkim dla lokalnych scenariuszy rozwoju

code-server lepiej nadaje się do scenariuszy wdrażania serwerów i kontenerów. Jeśli Twój produkt musi działać w Dockerze lub Twoi użytkownicy nie mają zainstalowanego VSCode, serwer kodu jest zazwyczaj właściwym wyborem.

## Różnice parametrów funkcji

Obydwa rozwiązania różnią się także kilkoma parametrami funkcjonalnymi. Różnice nie są ogromne, ale mogą powodować trudności w integracji w rzeczywistym użyciu.

| Funkcja | serwer kodowy | kod serwujący-web |
|------|-------------|----------------|
| Publiczna ścieżka bazowa | `/` (konfigurowalne) | `/vscode-server` (naprawiono) |
| Uwierzytelnianie | `--auth` parametr z wieloma trybami | `--connection-token` / `--without-connection-token` |
| Katalog danych | `{DataDir}/code-server` | `{DataDir}/vscode-serve-web` |
| Telemetria | Domyślnie wyłączone z `--disable-telemetry` | Zależy od ustawień VSCode |
| Aktualizuj kontrole | Można wyłączyć za pomocą `--disable-update-check` | Zależy od ustawień VSCode |

Różnice te wymagają szczególnej uwagi podczas integracji. Na przykład różne ścieżki URL oznaczają, że kod frontendu wymaga dedykowanej obsługi.

## Różnice w wykrywaniu dostępności

Podczas implementowania przełączania edytorów logika wykrywania dostępności również jest inna.

**serwer kodu**
- Zawsze zwracany jako widoczna implementacja
- Nadal wyświetlany, nawet jeśli jest niedostępny, z ikoną `install-required` stan
- Obsługuje automatyczne wykrywanie środowiska NVM Node.js 22.x

**kod serwujący-web**
- Widoczne tylko wtedy, gdy jesteś lokalny `code` Wykryto CLI
- Jeśli jest niedostępna, frontend automatycznie ukrywa tę opcję
- Zależy od lokalnego stanu instalacji VSCode

Ta różnica bezpośrednio wpływa na doświadczenie użytkownika. code-server jest bardziej przejrzysty: użytkownicy mogą zobaczyć tę opcję i zrozumieć, że instalacja jest nadal wymagana. codeserv-web jest bardziej ukryty: użytkownicy mogą nawet nie zdawać sobie sprawy z istnienia takiej opcji. To, które podejście jest lepsze, zależy od pozycjonowania produktu.

## Architektura podwójnej implementacji HagiCode

Po dogłębnej analizie w projekcie HagiCode przyjęto architekturę o podwójnej implementacji, która obsługuje oba rozwiązania na poziomie architektury.

### Domyślnie serwer kodu

```csharp
// The default active implementation is code-server
// If an explicit activeImplementation is saved, try that implementation first
// If the requested implementation is unavailable, the resolver tries the other one
// If a fallback occurs, return fallbackReason
```

Domyślnie wybieramy serwer kodu głównie ze względu na elastyczność licencjonowania i wdrażania. Jednak dla użytkowników, którzy mają już lokalne środowisko VSCode, codeserv-web jest również solidną opcją.

### Selektor implementacji

`CodeServerImplementationResolver` jest odpowiedzialny za ujednolicenie:
- Wybór implementacji podczas rozgrzewki rozruchowej
- Wybór implementacji podczas odczytywania statusu
- Wybór wdrożenia przy otwieraniu projektów
- Wybór implementacji podczas otwierania skarbców

Taka konstrukcja pozwala systemowi elastycznie reagować na różne scenariusze, a użytkownicy mogą wybrać implementację, która najlepiej pasuje do ich środowiska.

### Zasady adaptacji frontendu

```typescript
// When localCodeAvailable=false, do not show code serve-web
// When localCodeAvailable=true, show the code serve-web configuration
```

Frontend automatycznie pokazuje dostępne opcje w zależności od środowiska, dzięki czemu użytkownicy nie są zdezorientowani funkcjami, z których nie mogą skorzystać.

## Praktyczny przewodnik

Po całej tej teorii, na co należy zwrócić uwagę podczas faktycznego wdrażania? Ostatecznie liczy się wdrożenie.

### Zalecenie dotyczące wdrożenia platformy Docker

W przypadku wdrażania kontenerowego lepszym wyborem jest serwer kodu:

```dockerfile
# Use the official code-server image directly
FROM codercom/code-server:latest

# Or install through npm
RUN npm install -g code-server
```

Rozwiązuje to problem w jednej warstwie, bez konieczności dodatkowej instalacji VSCode.

### Przykłady konfiguracji

**konfiguracja serwera kodu**
```json
{
  "vscodeServer": {
    "enabled": true,
    "activeImplementation": "code-server",
    "codeServer": {
      "host": "0.0.0.0",
      "port": 8080,
      "executablePath": "",
      "authMode": "none"
    }
  }
}
```

**konfiguracja kodu serwującego przez Internet**
```json
{
  "vscodeServer": {
    "enabled": true,
    "activeImplementation": "serve-web",
    "serveWeb": {
      "host": "0.0.0.0",
      "port": 8080,
      "executablePath": "/usr/local/bin/code"
    }
  }
}
```

Konfiguracja może być nieco żmudna za pierwszym razem, ale gdy już zostanie wdrożona, wszystko staje się znacznie łatwiejsze w utrzymaniu.

### Różnice w budowie adresów URL

**serwer kodu**
```
http://localhost:8080/?folder=/path/to/project&vscode-lang=zh-CN
```

**kod serwujący-web**
```
http://localhost:8080/vscode-server/?folder=/path/to/project&tkn=xxx&vscode-lang=zh-CN
```

Zwróć uwagę na różnice w ścieżkach i parametrach. Podczas integracji musisz zająć się nimi oddzielnie.

### Implementacje przełączania

System obsługuje przełączanie runtime i automatycznie zatrzymuje poprzednią implementację podczas przełączania:

```csharp
// VsCodeServerManager automatically handles mutual exclusion
// When switching activeImplementation, the old implementation will not keep running in the background
```

Dzięki takiemu projektowi użytkownicy mogą w dowolnym momencie wypróbować różne implementacje i znaleźć opcję, która będzie dla nich najlepsza.

### Monitorowanie stanu

```typescript
const { settings, runtime } = await getVsCodeServerSettings();

// runtime.activeImplementation: "code-server" | "serve-web"
// runtime.fallbackReason: reason for switching
// runtime.status: "running" | "starting" | "stopped" | "unhealthy"
```

Gdy status jest widoczny, użytkownicy mogą szybko określić, czy problem pochodzi po stronie serwera, czy z jego własnej działalności.
