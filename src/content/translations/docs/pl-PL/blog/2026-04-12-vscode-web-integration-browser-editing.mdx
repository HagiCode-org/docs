## Szybka edycja kodu w przeglądarce: Integracja sieciowa VSCode w praktyce

> Jak po zakończeniu analizy kodu przez AI od razu otworzyć edytor w przeglądarce i rozpocząć wprowadzanie zmian? W tym artykule podzieliliśmy się naszymi praktycznymi doświadczeniami w integracji serwera kodu w projekcie HagiCode w celu stworzenia płynnego pomostu pomiędzy asystentem AI a doświadczeniem w edycji kodu.

## Tło

W dobie programowania wspomaganego sztuczną inteligencją programiści często muszą szybko sprawdzać i edytować kod. Tradycyjny przepływ pracy jest prosty: otwórz projekt w środowisku IDE na komputerze, zlokalizuj plik, edytuj go i zapisz. Ale w niektórych sytuacjach ten przepływ zawsze wydaje się nieco zaburzony.

Scenariusz pierwszy: zdalny rozwój. Podczas korzystania z asystenta AI, takiego jak HagiCode, backend może działać na zdalnym serwerze lub w kontenerze, a maszyny lokalne nie mogą bezpośrednio uzyskać dostępu do plików projektu. Za każdym razem, gdy musisz sprawdzić lub zmodyfikować kod, musisz połączyć się przez SSH lub inną metodę, a doświadczenie wydaje się fragmentaryczne. To jak chcieć spotkać kogoś przez grubą szybę: możesz go zobaczyć, ale nie możesz do niego dotrzeć.

Scenariusz drugi: szybkie podglądy. Po tym jak asystent AI przeanalizuje kod, użytkownik może chcieć jedynie szybko przejrzeć plik lub wprowadzić drobną zmianę. Uruchomienie pełnego środowiska IDE na komputerze stacjonarnym wydaje się ciężkie, podczas gdy lekki edytor w przeglądarce lepiej odpowiada potrzebie „szybkiego spojrzenia”. W końcu kto chce zmobilizować cały zestaw narzędzi tylko po to, żeby rzucić okiem?

Scenariusz trzeci: współpraca na różnych urządzeniach. Podczas pracy na różnych urządzeniach edytor oparty na przeglądarce zapewnia ujednolicony punkt dostępu bez konieczności konfigurowania każdego komputera w środowisku programistycznym. Już samo to pozwala zaoszczędzić wielu kłopotów. Życie jest krótkie; po co powtarzać w kółko tę samą pracę konfiguracyjną?

Aby rozwiązać te problemy, zintegrowaliśmy VSCode Web z projektem HagiCode. Dzięki temu asystent AI i środowisko edycji kodu płynnie się łączą: po przeanalizowaniu kodu przez sztuczną inteligencję użytkownicy mogą natychmiast otworzyć edytor i wprowadzić zmiany w tej samej sesji przeglądarki, bez przełączania aplikacji. Jest to rodzaj doświadczenia, które po prostu jest tam, gdzie go potrzebujesz.

## Wybór techniczny: Dlaczego serwer kodu?

Spośród wielu dostępnych rozwiązań internetowych VSCode wybraliśmy serwer kodu. Za tą decyzją stało kilka konkretnych powodów.

Kompletność funkcji. code-server jest internetową wersją VSCode i obsługuje większość funkcji pulpitu, w tym system rozszerzeń, inteligentne sugestie, debugowanie i inne. Oznacza to, że użytkownicy mogą uzyskać możliwość edycji w przeglądarce bardzo zbliżonej do wersji komputerowej. W końcu kto naprawdę chce iść na kompromis w kwestii funkcjonalności?

Elastyczne wdrożenie. code-server może działać jako niezależna usługa, a także obsługuje wdrażanie oparte na Dockerze, co dobrze pasuje do architektury HagiCode. Nasz backend jest napisany w C#, frontend wykorzystuje React, a oba komunikują się z usługą serwera kodu poprzez API REST. To jak budowanie z klocków: każdy element ma swoje miejsce.

Bezpieczne uwierzytelnianie. code-server zawiera wbudowany moduł `connection-token` mechanizm zapobiegający nieupoważnionemu dostępowi. Każda sesja posiada unikalny token, dzięki czemu tylko autoryzowani użytkownicy mogą otworzyć edytor. Bezpieczeństwo to jedna z tych rzeczy, które w pełni doceniasz dopiero wtedy, gdy je masz.

## Projekt Architektury

Integracja sieciowa VSCode HagiCode wykorzystuje oddzielną architekturę front-end/back-end.

### Warstwa usług frontendowych

Frontend obejmuje interakcje z backendem `vscodeServerService.ts`:

```typescript
// Open project
export async function openProjectInCodeServer(
  id: string,
  currentInterfaceLanguage?: string,
): Promise<VsCodeServerLaunchResponseDto>

// Open vault
export async function openVaultInCodeServer(
  id: string,
  path?: string,
  currentInterfaceLanguage?: string,
): Promise<VsCodeServerLaunchResponseDto>
```

Różnica między tymi dwiema metodami jest prosta: `openProjectInCodeServer` otwiera cały projekt, natomiast `openVaultInCodeServer` otwiera określoną ścieżkę wewnątrz skarbca. W przypadku projektów MonoSpecs obejmujących wiele repozytoriów system automatycznie tworzy plik obszaru roboczego. Jasno określone obowiązki często wystarczą, jeśli każda część dobrze wykonuje swoją pracę.

### Warstwa usług backendu

Zaplecze `VaultAppService.cs` implementuje podstawową logikę:

```csharp
public async Task<VsCodeServerLaunchResponseDto> OpenInCodeServerAsync(
    string id,
    string? relativePath = null,
    string? currentInterfaceLanguage = null,
    CancellationToken cancellationToken = default)
{
    // 1. Get settings and check whether the feature is enabled
    var settings = await _vsCodeServerSettingsService.GetResolvedSettingsAsync(cancellationToken);
    if (!settings.Enabled) {
        throw new BusinessException(VsCodeServerErrorCodes.Disabled, "VSCode Server is disabled.");
    }

    // 2. Get vault and resolve the launch directory
    var vault = await RequireVaultAsync(id, cancellationToken);
    var launchDirectory = ResolveLaunchDirectory(vault, relativePath);

    // 3. Ensure code-server is running and get runtime info
    var runtime = await _vsCodeServerManager.EnsureStartedAsync(settings, cancellationToken);

    // 4. Resolve language settings
    var language = _vsCodeServerSettingsService.ResolveLaunchLanguage(
        settings.Language,
        currentInterfaceLanguage);

    // 5. Build launch URL
    return new VsCodeServerLaunchResponseDto {
        LaunchUrl = AppendQueryString(runtime.BaseUrl, new Dictionary<string, string?> {
            ["folder"] = launchDirectory,
            ["tkn"] = runtime.ConnectionToken,
            ["vscode-lang"] = language,
        }),
        ConnectionToken = runtime.ConnectionToken,
        OpenMode = "folder",
        Runtime = VsCodeServerSettingsService.MapRuntime(
            await _vsCodeServerManager.GetRuntimeSnapshotAsync(cancellationToken)),
    };
}
```

Ta metoda ma bardzo jasną odpowiedzialność: sprawdź ustawienia, rozwiąż ścieżki, uruchom usługę i zbuduj adres URL. Wśród nich `ResolveLaunchDirectory` Metoda przeprowadza kontrolę bezpieczeństwa ścieżki, aby zapobiec atakom polegającym na przechodzeniu ścieżki. Kod może przypominać trochę poezję, gdy każda linijka ma swój cel.

### Automatyczne zarządzanie czasem działania

Backend zarządza procesem serwera kodu `VsCodeServerManager`:

- Sprawdź stan procesu
- Automatycznie uruchamiaj zatrzymane usługi
- Zwróć migawki środowiska wykonawczego, takie jak port, identyfikator procesu i czas rozpoczęcia

Taka konstrukcja pozwala systemowi automatycznie obsługiwać cykl życia serwera kodu, dzięki czemu użytkownicy nie muszą ręcznie zarządzać procesami serwisowymi. Życie jest już wystarczająco skomplikowane; wszystko, co można zautomatyzować, powinno takie być.

### Synchronizacja języka

HagiCode obsługuje wielojęzyczny interfejs i serwer kodu musi przestrzegać tego ustawienia. System obsługuje trzy tryby językowe:

- `follow`: postępuj zgodnie z bieżącym językiem interfejsu
- `zh-CN`: naprawiono język chiński
- `en-US`: naprawiono język angielski

Ustawienie jest przekazywane do serwera kodu poprzez `vscode-lang` Parametr URL, dzięki czemu język edytora pozostaje spójny z interfejsem HagiCode. Język czuje się najlepiej, gdy jest ujednolicony.

### Obszar roboczy wielu repozytoriów MonoSpecs

W przypadku projektów MonoSpecs, które zawierają wiele podrepozytoriów w jednym repozytorium, system automatycznie tworzy plik `.code-workspace` plik:

```csharp
private async Task<string> CreateWorkspaceFileAsync(Project project, Guid projectId)
{
    var folders = await ResolveWorkspaceFoldersAsync(project.Path);
    var workspaceDocument = new {
        folders = folders.Select(path => new { path }).ToArray(),
    };
    // Generate workspace file...
}
```

Umożliwia to edycję wielu podrepozytoriów w tej samej instancji serwera kodu, co jest szczególnie praktyczne w przypadku dużych projektów monorepo. Wiele repozytoriów w jednym oknie może sprawiać wrażenie wielu historii zebranych w tej samej książce.

## Integracja frontendowa

Frontend HagiCode wykorzystuje React + TypeScript, a integracja serwera kodu nie jest szczególnie skomplikowana.

### Przycisk szybkiego działania

Dodaj przycisk Code Server do karty projektu:

```tsx
// QuickActionsZone.tsx
<Button
  size="sm"
  variant="default"
  onClick={() => onAction({ type: 'open-code-server' })}
>
  <Globe className="h-3 w-3 mr-1" />
  <span className="text-xs">{t('project.openCodeServer')}</span>
</Button>
```

Ten przycisk wyzwala akcję otwierania i wywołuje interfejs API zaplecza w celu uzyskania adresu URL uruchamiania. Jeden przycisk, jedna akcja, bezpośrednia i prosta.

### Obsługa akcji otwartej

```tsx
const handleAction = async (action: ProjectAction) => {
  if (action.type === 'open-code-server') {
    const response = await openProjectInCodeServer(project.id, i18n.language);
    window.open(response.launchUrl, '_blank', 'noopener,noreferrer');
  }
};
```

Użyj `window.open` aby otworzyć serwer kodu w nowej karcie. The `noopener,noreferrer` parametry zapewniają dodatkowe bezpieczeństwo. Jeśli chodzi o bezpieczeństwo, nie ma czegoś takiego jak zbytnia ostrożność.

### Wpis edycji skarbca

Dodaj podobny przycisk edycji na liście Vault:

```tsx
const handleEditVault = async (vault: VaultItemDto) => {
  const response = await openVaultInCodeServer(vault.id);
  window.open(response.launchUrl, '_blank', 'noopener,noreferrer');
};
```

Projekty i skarbce korzystają z tego samego otwartego mechanizmu, który zapewnia spójność interakcji. Spójność ma prawie takie samo znaczenie jak sama funkcja.

## Logika konstrukcji adresu URL

Format adresu URL serwera kodu zawiera kilka szczegółów, na które warto zwrócić uwagę.

**Tryb folderu**:
```
http://{host}:{port}/?folder={path}&tkn={token}&vscode-lang={lang}
```

**Tryb roboczy**:
```
http://{host}:{port}/?workspace={workspacePath}&tkn={token}&vscode-lang={lang}
```

Tutaj, `tkn` to token połączenia. Jest generowany automatycznie przy każdym uruchomieniu serwera kodu, zapewniając bezpieczny dostęp. The `vscode-lang` parametr kontroluje język interfejsu edytora. Każdy z tych parametrów ma swoją rolę do odegrania.

## Scenariusze użycia

### Scenariusz pierwszy: przegląd kodu wspomagany sztuczną inteligencją

Użytkownik rozmawia z HagiCode, sztuczna inteligencja analizuje kod projektu i znajduje potencjalny problem, a następnie użytkownik klika przycisk „Otwórz w serwerze kodu”, aby otworzyć edytor bezpośrednio w przeglądarce, sprawdzić plik, którego dotyczy problem, naprawić go i wrócić do HagiCode, aby kontynuować rozmowę. Cały przepływ odbywa się w przeglądarce bez przełączania aplikacji. Jest gładka, tak jak płynąca woda jest gładka.

### Scenariusz drugi: edytowanie materiałów do nauki w repozytorium

Użytkownik tworzy repozytorium do studiowania projektu open source i chce dodać notatki do nauki w sekcji `docs/` katalog. Dzięki serwerowi kodu mogą edytować pliki Markdown bezpośrednio w przeglądarce, zapisywać je i pozwolić HagiCode natychmiastowo przeczytać zaktualizowane notatki. Jest to szczególnie przydatne przy budowaniu osobistej bazy wiedzy. Wiedza staje się tym cenniejsza, im więcej ją gromadzisz.

### Scenariusz trzeci: rozwój wielu repozytoriów MonoSpecs

Projekt MonoSpecs zawiera wiele podrepozytoriów, a serwer kodu automatycznie tworzy wielofolderowy obszar roboczy. W przeglądarce użytkownicy mogą edytować kod w kilku repozytoriach jednocześnie, a następnie zatwierdzać zmiany z powrotem do odpowiednich repozytoriów Git. Ten przepływ pracy szczególnie dobrze nadaje się do zmian, które muszą obejmować wiele repozytoriów. Wspólna edycja kilku repozytoriów wymaga trochę techniki, podobnie jak jednoczesne wykonywanie wielu zadań.

## Względy bezpieczeństwa

Podczas wdrażania integracji serwera kodu z serwerem kodu na szczególną uwagę zasługuje bezpieczeństwo. Jeśli bezpieczeństwo zawiedzie, zawsze zauważysz to za późno.

### Token połączenia

The `connection-token` jest generowany losowo i nie należy go ujawniać. Najlepiej stosować go w ramach protokołu HTTPS, aby zapobiec przechwyceniu tokena przez człowieka pośrodku. Wrażliwe informacje warto odpowiednio chronić.

### Bezpieczeństwo ścieżki pliku

Backend implementuje kontrole przejścia ścieżki:

```csharp
private static string ResolveLaunchDirectory(VaultRegistryEntry vault, string? relativePath)
{
    var vaultRoot = EnsureTrailingSeparator(Path.GetFullPath(vault.PhysicalPath));
    var combinedPath = Path.GetFullPath(Path.Combine(vaultRoot, relativePath ?? "."));
    if (!combinedPath.StartsWith(vaultRoot, StringComparison.OrdinalIgnoreCase))
    {
        throw new BusinessException(VaultRelativePathTraversalCode, "Relative path traversal detected.");
    }
    return combinedPath;
}
```

Ten kod gwarantuje, że użytkownicy nie będą mogli z niego korzystać `../` lub podobne wzorce umożliwiające dostęp do plików poza katalogiem Vault. Kontrole graniczne zawsze lepiej przeprowadzić niż pominąć.

### Kontrola uprawnień

Proces serwera kodu powinien działać z odpowiednimi uprawnieniami użytkownika, aby nie mógł uzyskać dostępu do wrażliwych plików systemowych. Najlepiej jest uruchomić usługę serwera kodów pod dedykowanym użytkownikiem. Kontrola uprawnień to jedna z tych podstaw, o których należy zawsze pamiętać.

## Optymalizacja wydajności

code-server zużywa zasoby serwera, dlatego oto kilka sugestii optymalizacyjnych:

- Monitoruj wykorzystanie procesora i pamięci oraz w razie potrzeby dostosuj limity zasobów
- Duże projekty mogą wymagać dłuższych limitów czasu
- Zaimplementuj automatyczne czyszczenie limitu czasu sesji, aby zwolnić zasoby
- Rozważ buforowanie, aby ograniczyć powtarzające się obliczenia

HagiCode zapewnia API monitorowania stanu środowiska wykonawczego, które frontend może wywołać `getVsCodeServerSettings()` aby odzyskać bieżący stan:

```typescript
const { settings, runtime } = await getVsCodeServerSettings();
// runtime.status: 'disabled' | 'stopped' | 'starting' | 'running' | 'unhealthy'
// runtime.baseUrl: "http://localhost:8080"
// runtime.processId: 12345
```

Taka konstrukcja pozwala użytkownikom jasno zrozumieć stan serwera kodu i szybko zlokalizować problemy, gdy coś pójdzie nie tak. Kiedy status jest widoczny, ludzie czują się bardziej pod kontrolą.

## Szczegóły doświadczenia użytkownika

Podczas wdrożenia odkryliśmy kilka szczegółów, które zauważalnie wpływają na doświadczenia użytkownika i zasługują na szczególną uwagę.

Otwarcie serwera kodu po raz pierwszy może wymagać oczekiwania na uruchomienie, a opóźnienie to może wynosić od kilku sekund do pół minuty. Dobrym pomysłem jest pokazanie stanu ładowania w interfejsie, aby użytkownicy wiedzieli, że system nadal działa. Oczekiwanie jest łatwiejsze, gdy pojawia się informacja zwrotna.

Przeglądarki mogą blokować wyskakujące okienka, dlatego użytkownicy powinni zostać poproszeni o ręczne zezwolenie. Przy pierwszym uruchomieniu HagiCode wyświetla wskazówki wyjaśniające, jak przyznać niezbędne uprawnienia przeglądarki. Doświadczenia użytkownika często zależą właśnie od tych drobnych szczegółów.

Dobrym pomysłem jest także wyświetlanie stanu środowiska wykonawczego, takiego jak uruchomienie, działanie lub błąd, aby w przypadku wystąpienia problemów użytkownicy mogli szybko stwierdzić, czy problem leży po stronie serwera, czy w ich własnym działaniu. Wiedza o tym, gdzie leży problem, daje przynajmniej punkt wyjścia.

## Przykład konfiguracji

Konfiguracja serwera kodu nie jest skomplikowana:

```json
{
  "vscodeServer": {
    "enabled": true,
    "host": "0.0.0.0",
    "port": 8080,
    "language": "follow"
  }
}
```

`enabled` kontroluje, czy funkcja jest włączona, `host` i `port` zdefiniuj adres nasłuchiwania oraz `language` ustawia tryb językowy. Ustawienia te można modyfikować za pomocą interfejsu użytkownika i zaczynają obowiązywać natychmiast. Proste rzeczy są często najłatwiejsze w użyciu.
