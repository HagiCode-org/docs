## Ochrona kodu .NET w praktyce: od zaciemniania do ochrony maszyn wirtualnych

> W tym artykule wyjaśniono, jak wdrożyć wielowarstwową strategię ochrony kodu w projektach .NET, obejmującą pełną ścieżkę od podstawowego zaciemniania do profesjonalnej ochrony maszyny wirtualnej.

## Tło

W przypadku tworzenia aplikacji .NET ochrona podstawowego kodu, np. sprawdzania poprawności licencji, logiki biznesowej i wrażliwej konfiguracji przed dekompilacją i inżynierią wsteczną, jest, szczerze mówiąc, tematem, którego nie można uniknąć. W miarę dojrzewania ekosystemu .NET programiści uzyskali dostęp do szeregu opcji ochrony, od wbudowanych atrybutów zaciemniających po profesjonalne narzędzia ochrony oparte na wirtualizacji.

Jako złożony, wielojęzyczny projekt monorepo, HagiCode obejmuje aplikacje komputerowe, systemy kompilacji i możliwości zarządzania licencjami. Kod nieuchronnie zawiera logikę sprawdzania licencji, wrażliwą konfigurację, taką jak klucze API i identyfikatory produktów, oraz logikę o znaczeniu krytycznym dla biznesu. Te części wymagają poważnej ochrony, ponieważ nikt nie chce, aby ich ciężka praca została tak łatwo narażona.

W tym artykule opisano podejście do ochrony kodu, które faktycznie przyjęliśmy w projekcie HagiCode, i podsumowano całą drogę od wczesnych pułapek do późniejszej optymalizacji. Mamy nadzieję, że podsunie Ci kilka przydatnych pomysłów.

## Treść podstawowa

### 1. Wbudowany atrybut zaciemniania firmy Microsoft

.NET Framework udostępnia wbudowaną platformę .NET Framework `[ObfuscationAttribute]`, który jest najbardziej podstawowym i powszechnie używanym znacznikiem zaciemniania kodu. Ten atrybut żyje w `System.Reflection` przestrzeni nazw i umożliwia zastosowanie podstawowej ochrony kodu bez konieczności wprowadzania narzędzi innych firm.

**Podstawowe funkcje**:
- `Feature` właściwość: Określa funkcję zaciemniania, taką jak `"ultra"` (duże zaciemnienie) lub `"all"` (pełne zaciemnienie)
- `Exclude` nieruchomość: `true` oznacza wykluczenie z zaciemniania, oraz `false` oznacza zastosowanie zaciemnienia
- Można zastosować do klas, metod, właściwości i innych elementów członkowskich typu

W projekcie HagiCode można go zobaczyć w następujący sposób:

```csharp
[Obfuscation(Feature = "ultra", Exclude = false)]
public async Task<LicenseValidationResult?> ValidateLicenseAsync(...)
```

**Zalety** tego podejścia są dość oczywiste:
- Żadnych dodatkowych zależności; jest wbudowany w .NET Framework od razu po wyjęciu z pudełka
- Mogą być rozpoznawane i przetwarzane przez narzędzia zaciemniające stron trzecich
- Nie zwiększa znacząco rozmiaru skompilowanego zestawu

To powiedziawszy, ma to również ograniczenia. To tylko znacznik, a faktyczny wynik zaciemniania zależy od implementacji narzędzia. Nie może zapewnić poziomu bezpieczeństwa maszyny wirtualnej.

### 2. VMP (ochrona maszyny wirtualnej)

VMP to profesjonalne narzędzie do ochrony kodu, które zapewnia wysoki poziom ochrony poprzez kompilację kodu do instrukcji maszyny wirtualnej. W przeciwieństwie do prostego zaciemniania nazw, VMP faktycznie przekształca logikę kodu w formę, której konwencjonalne dekompilatory nie są w stanie odtworzyć.

**Klasyfikacja poziomu ochrony**:

| Poziom | Wirtualizacja | Mutacja | Zapobieganie debugowaniu | Szyfrowanie ciągów | Przypadki użycia |
|------|--------|------|--------|------------|----------|
| WYSOKI | pełny | wysoki | włączone | włączone | Weryfikacja licencji, współbieżność sesji, wrażliwe stałe |
| ŚREDNI | częściowe | średni | włączone | włączone | Logika biznesowa, modele dziedzinowe |
| NISKI | żaden | niski | wyłączone | wyłączone | Klasy użytkowe, kod niekrytyczny |

Projekt HagiCode definiuje deklaratywny system atrybutów do oznaczania kodu wymagającego ochrony:

```csharp
// High-priority protection
[VmProtect(VmProtectionPriority.High, Reason = "Contains license verification logic")]
public class KeygenClient { ... }

// Exclude from protection
[VmExclude(Reason = "Public API that must remain unchanged")]
public class PublicApi { ... }

// Inherited protection
[VmProtect(Priority.High, ProtectDerived = true)]
public class BaseLicenseValidator { ... }
```

### 3. Strategia ochrony na czas budowy

Ochrona VMP ma znaczenie nie tylko w czasie wykonywania. Należy go również zautomatyzować w ramach procesu kompilacji, ponieważ robienie tego ręcznie byłoby zbyt nudne. System kompilacji HagiCode obsługuje kilka trybów:

- **Natywny tryb Windows**: Bezpośrednie wywołanie narzędzia VMProtect
- **Tryb kontenera Linux Docker**: Uruchom VMP w kontenerze, aby rozwiązać problemy ze zgodnością między platformami
- **Skanowanie atrybutów**: Automatyczne wykrywanie znaczników ochrony w kodzie
- **Mechanizm sprawdzania**: Potwierdź, że ochrona została pomyślnie zastosowana

Podsumowując, funkcje te znacznie ułatwiają zarządzanie procesem.

## Rozwiązanie

### 1. Korzystanie z wbudowanego atrybutu zaciemniania firmy Microsoft

Zastosuj `ObfuscationAttribute` bezpośrednio w kodzie:

```csharp
using System.Reflection;

[Obfuscation(Feature = "ultra", Exclude = false)]
public class LicenseService
{
    [Obfuscation(Feature = "ultra", Exclude = false)]
    public async Task<bool> ValidateLicenseAsync(string key)
    {
        // License validation logic
    }

    [Obfuscation(Feature = "flow", Exclude = false)]
    private string DecryptToken(string encrypted)
    {
        // Decryption logic
    }
}
```

Czasami trzeba zezwolić zespołom testowym na dostęp do elementów wewnętrznych, zachowując jednocześnie bezpieczeństwo kodu produkcyjnego:

```csharp
// AssemblyInfo.cs
[assembly: InternalsVisibleTo("HagiCode.Application.Tests")]
[assembly: InternalsVisibleTo("DynamicProxyGenAssembly2")] // for Moq
```

Dzięki temu testowanie jest znacznie wygodniejsze, ponieważ kod nadal wymaga prawidłowego przetestowania.

### 2. Niestandardowe definicje atrybutów dla ochrony VMP

Utwórz niestandardowe atrybuty ochrony, aby kontrolować zachowanie VMP:

```csharp
using System;

[AttributeUsage(AttributeTargets.Class | AttributeTargets.Method | AttributeTargets.Property)]
public class VmProtectAttribute : Attribute
{
    public VmProtectionPriority Priority { get; set; }
    public string? Reason { get; set; }
    public bool ProtectDerived { get; set; }
}

[AttributeUsage(AttributeTargets.Class | AttributeTargets.Method | AttributeTargets.Property)]
public class VmExcludeAttribute : Attribute
{
    public string? Reason { get; set; }
}

public enum VmProtectionPriority
{
    None = 0,
    Low = 1,
    Medium = 2,
    High = 3
}
```

Atrybuty niestandardowe są często łatwiejsze w użyciu, ponieważ bezpośrednio odzwierciedlają Twoje własne wymagania dotyczące ochrony.

### 3. Plik konfiguracyjny VMP

```yaml
# vmp_config.yml
protection:
  priority_mode: "attribute"  # Attribute-based priority
  default_level: "medium"

tools:
  - name: "vmprotect"
    path: "C:\\Program Files\\VMProtect Ultimate\\VMProtect.exe"

protection_levels:
  high:
    virtualization: "full"
    mutation: "high"
    anti_debug: true
    anti_dump: true
    encrypt_strings: true
    encrypt_resources: true

  medium:
    virtualization: "partial"
    mutation: "medium"
    anti_debug: true
    encrypt_strings: true

  low:
    virtualization: "none"
    mutation: "low"
    anti_debug: false
```

Przejrzysta konfiguracja znacznie ułatwia późniejszą konserwację systemu.

## Praktyczny przewodnik

### 1. Praktyki zabezpieczające komponenty krytyczne

Zgodnie ze specyfikacją ochrony kodu HagiCode, następujące komponenty muszą korzystać z ochrony o WYSOKIM priorytecie:

```csharp
// Production constants - must be encrypted and protected by VMP
[VmProtect(VmProtectionPriority.High, Reason = "Production constants")]
public static class ProductionConstants
{
    // Encrypted string accessor, protected by VMP
    [VmProtect(VmProtectionPriority.High)]
    public static string GetLicenseServerUrl(IOptions<LicenseOptions> options) => ...;
}

// License validation logic
[VmProtect(VmProtectionPriority.High, Reason = "License verification logic")]
public class KeygenClient : IKeygenClient
{
    [Obfuscation(Feature = "ultra", Exclude = false)]
    public async Task<LicenseValidationResult?> ValidateLicenseAsync(...) { ... }
}

// Machine fingerprint service
[VmProtect(VmProtectionPriority.High)]
public class MachineFingerprintService : IMachineFingerprintService { ... }
```

Kod krytyczny zasługuje na silniejszą ochronę, ponieważ ujawnienie podstawowej logiki spowodowałoby realne problemy.

### 2. Szyfrowanie ciągów i deszyfrowanie w czasie wykonywania

Szyfruj ciągi znaków w czasie kompilacji i odszyfruj je w czasie wykonywania:

```csharp
public static class StringDecryption
{
    [VmProtect(VmProtectionPriority.High, Reason = "CRITICAL SECURITY")]
    public static string DecryptString(byte[] encryptedData, byte[] key, byte[] iv)
    {
        using var aes = Aes.Create();
        aes.Key = key;
        aes.IV = iv;

        using var decryptor = aes.CreateDecryptor();
        using var ms = new MemoryStream(encryptedData);
        using var cs = new CryptoStream(ms, decryptor, CryptoStreamMode.Read);
        using var reader = new StreamReader(cs);

        return reader.ReadToEnd();
    }
}

// Production constant accessor (lazy loading + caching)
public static class ProductionConstants
{
    private static string? _cachedLicenseServerUrl;

    public static string GetLicenseServerUrl(IOptions<LicenseOptions> options)
    {
        if (_cachedLicenseServerUrl == null)
        {
            var encrypted = GetEncryptedLicenseServerUrl();
#if DEBUG
            _cachedLicenseServerUrl = options.Value.PrimaryServer.Url;
#else
            _cachedLicenseServerUrl = StringDecryption.DecryptString(
                encrypted,
                GetEncryptionKey(),
                GetEncryptionIV());
#endif
        }
        return _cachedLicenseServerUrl;
    }
}
```

Ten krok ma znaczenie, ponieważ poufnych informacji nigdy nie należy pozostawiać w postaci zwykłego tekstu.

### 3. Weryfikacja ochrony VMP

Po kompilacji należy sprawdzić, czy ochrona została pomyślnie zastosowana; w przeciwnym razie nie możesz być pewien, że rzeczywiście działa:

```csharp
// Example verification script
public bool VerifyProtection(string assemblyPath)
{
    // 1. Check the VMP signature
    var bytes = File.ReadAllBytes(assemblyPath);
    var vmpSignature = Encoding.ASCII.GetBytes("VMProtect");
    if (bytes.Any(b => vmpSignature.Contains(b)))
    {
        return true;
    }

    // 2. Check for file size changes (the protected file is usually larger)
    var originalInfo = new FileInfo(assemblyPath.Replace(".dll", ".bak"));
    if (originalInfo.Exists)
    {
        var sizeRatio = (double)new FileInfo(assemblyPath).Length / originalInfo.Length;
        return sizeRatio > 1.1;
    }

    return false;
}
```

Weryfikację zawsze warto przeprowadzić, gdyż w przeciwnym razie problemy mogą przemknąć niezauważone.

### 4. Uwagi i zastrzeżenia

Jest tu kilka pułapek, które zasługują na szczególną uwagę:

1. **Nie zaciemniaj całego kodu**: Publiczne interfejsy API, definicje interfejsów i klasy DTO zwykle nie wymagają ochrony. Nadmierne zaciemnianie może zaszkodzić wydajności i debugowaniu. Projekt HagiCode nauczył się tego na własnej skórze.

2. **Chroń osoby uzyskujące dostęp do kluczy**: Metody odzyskiwania kluczy szyfrowania muszą otrzymać ten sam lub wyższy poziom ochrony niż same zaszyfrowane dane; w przeciwnym razie cała konfiguracja straci na wartości.

3. **Testowanie równowagi i produkcja**: Kompilacje DEBUG powinny pomijać szyfrowanie, aby ułatwić programowanie i debugowanie, natomiast kompilacje RELEASE powinny umożliwiać pełną ochronę. Pamiętaj, aby oddzielić je kompilacją warunkową, np `#if DEBUG`.

4. **Rozważ środowisko Docker**: Uruchamianie VMP w systemie Linux wymaga podejścia kontenerowego, aby zapewnić kompatybilność narzędzi. HagiCode wykorzystuje rozwiązanie kontenerowe Wine + VMP, aby rozwiązać problem międzyplatformowy.

5. **Weryfikacja jest obowiązkowa**: Po zakończeniu kompilacji należy sprawdzić, czy ochrona została pomyślnie zastosowana. W przeciwnym razie wrażliwy kod może nadal zostać ujawniony, a pokazany wcześniej kod weryfikacyjny istnieje właśnie w tym celu.

## Wniosek

Dzięki tej wielowarstwowej strategii ochrony HagiCode zbudował kompleksowy system bezpieczeństwa kodu, który obejmuje od zaciemniania linii bazowej po ochronę maszyny wirtualnej:

- **Warstwa 1**: Użyj `ObfuscationAttribute` do oznaczania linii bazowej i udostępniania wskazówek narzędziom innych firm
- **Warstwa 2**: Użyj niestandardowego `VmProtectAttribute` oświadczenia wyrażające zamiar i pierwszeństwo ochrony
- **Warstwa 3**: Użyj ochrony maszyny wirtualnej VMP, aby przekształcić krytyczny kod w nieodwracalne instrukcje maszyny wirtualnej
- **Warstwa 4**: Automatycznie skanuj i zastosuj ochronę podczas kompilacji, a następnie zweryfikuj wynik

Takie podejście może oprzeć się zwykłym narzędziom dekompilacyjnym, a jednocześnie lepiej przeciwstawić się zaawansowanym atakom inżynierii wstecznej. Jeśli tworzysz aplikację .NET, która wymaga ochrony kodu, mam nadzieję, że da ci to przynajmniej przydatny punkt odniesienia.
