## Przewodnik po wdrażaniu obsługi słów kluczowych w rozpoznawaniu mowy Doubao

> W tym artykule szczegółowo wyjaśniono, jak zaimplementować obsługę słów kluczowych w rozpoznawaniu mowy Doubao w projekcie HagiCode. Używając zarówno niestandardowych słów kluczowych, jak i tabel słów kluczowych platformy, możesz znacznie poprawić dokładność rozpoznawania słownictwa specyficznego dla domeny.

## Tło

Technologia rozpoznawania mowy rozwija się od wielu lat, jednak jeden problem nie daje spokoju programistom. Modele rozpoznawania mowy ogólnego przeznaczenia mogą obejmować język potoczny, ale często zawodzą, jeśli chodzi o profesjonalną terminologię, nazwy produktów i nazwiska. Pomyśl o tym: asystent głosowy w branży medycznej musi dokładnie rozpoznawać terminy takie jak „nadciśnienie”, „cukrzyca” i „choroba niedokrwienna serca”; system prawny musi precyzyjnie uchwycić takie terminy, jak „podstawa powództwa”, „obrona” i „ciężar dowodu”. W takich scenariuszach model ogólnego przeznaczenia daje z siebie wszystko, ale często to nie wystarcza.

Natknęliśmy się na to samo wyzwanie w projekcie HagiCode. Jako wielofunkcyjny asystent kodowania AI, HagiCode musi obsługiwać rozpoznawanie mowy w szerokim zakresie terminologii technicznej. Jednakże interfejs API rozpoznawania mowy Doubao w swojej domyślnej konfiguracji nie mógł w pełni spełnić naszych wymagań dotyczących dokładności w przypadku terminów specjalistycznych. To nie tak, że Doubao nie jest wystarczająco dobry; raczej każda dziedzina ma swój własny system terminologii. Po przeprowadzeniu badań i eksploracji technicznej odkryliśmy, że interfejs API rozpoznawania mowy Doubao faktycznie obsługuje słowa-klucze. Dzięki prostej konfiguracji może znacznie poprawić dokładność rozpoznawania określonego słownictwa. W pewnym sensie, gdy już powiesz mu, na które słowa ma zwrócić uwagę, słucha ich uważniej.

W tym artykule przedstawiono kompletne rozwiązanie, którego użyliśmy w projekcie HagiCode do wdrożenia słów-kluczy rozpoznawania mowy Doubao. Dostępne są oba tryby, niestandardowe słowa kluczowe i tabele słów kluczowych platformy, które można również łączyć. Dzięki temu rozwiązaniu programiści mogą elastycznie konfigurować słowa-klucze w oparciu o scenariusze biznesowe, dzięki czemu system rozpoznawania mowy może lepiej „rozpoznawać” profesjonalne, nietypowe, ale krytyczne słownictwo.

## Implementacja rdzenia

### Zrozumienie dwóch trybów słów kluczowych

Interfejs API rozpoznawania mowy Doubao zapewnia dwa sposoby konfigurowania słów-kluczy, a każdy z nich ma swoje własne idealne przypadki użycia i zalety.

**Niestandardowy tryb słów kluczowych** pozwala nam przekazywać tekst słów kluczowych bezpośrednio przez `corpus.context` pole. To podejście jest szczególnie odpowiednie w scenariuszach, w których trzeba szybko skonfigurować niewielką liczbę słów-kluczy, na przykład tymczasowo rozpoznawać nazwę produktu lub imię i nazwisko osoby. W implementacji HagiCode analizujemy wielowierszowy tekst słowa kluczowego wprowadzony przez użytkownika na listę ciągów, a następnie formatujemy go w `context_data` tablica wymagana przez API Doubao. To podejście jest bardzo bezpośrednie: po prostu mówisz systemowi, na które słowa ma zwrócić uwagę, a on dokładnie to robi.

**Tryb tabeli słów kluczowych na platformie** wykorzystuje `corpus.boosting_table_id` pole umożliwiające odniesienie do wstępnie skonfigurowanej tabeli słów kluczowych na platformie samouczącej się Doubao. To podejście jest odpowiednie w scenariuszach, w których trzeba zarządzać dużą liczbą słów-kluczy. Możemy tworzyć i utrzymywać tabele słów kluczowych na platformie samouczącej się Doubao, a następnie odwoływać się do nich według identyfikatora. W przypadku projektów takich jak HagiCode, w których specjalistyczne terminy muszą być stale aktualizowane i utrzymywane, ten tryb zapewnia znacznie lepsze zarządzanie. Gdy liczba słów kluczowych wzrośnie, posiadanie scentralizowanego miejsca do zarządzania nimi jest znacznie lepsze niż wpisywanie ich za każdym razem ręcznie.

Co ciekawe, te dwa tryby można także stosować łącznie. Interfejs API Doubao obsługuje dołączanie zarówno niestandardowych słów-kluczy, jak i identyfikatora tabeli słów-kluczy platformy w tym samym żądaniu, przy czym strategia łączenia jest kontrolowana przez `combine_mode` parametr. Ta elastyczność pozwala HagiCode sprostać szerokiemu zakresowi złożonych potrzeb w zakresie rozpoznawania profesjonalnej terminologii. Czasami połączenie wielu podejść daje lepsze rezultaty.

### Definicje typów frontendu i weryfikacja

W implementacji frontendowej HagiCode zdefiniowaliśmy kompletny zestaw typów konfiguracji słów kluczowych i logiki walidacji. Pierwsza część to definicja typu:

```typescript
export interface HotwordConfig {
  contextText: string;           // Multi-line hotword text
  boostingTableId: string;      // Doubao platform hotword table ID
  combineMode: boolean;          // Whether to use both together
}
```

Ten prosty interfejs zawiera wszystkie elementy konfiguracji funkcji słowa-klucza. Wśród nich `contextText` to część, z którą użytkownicy wchodzą w bezpośrednią interakcję: pozwalamy użytkownikom na wprowadzenie jednego słowa kluczowego w każdym wierszu, co jest bardzo intuicyjne. Poproszenie użytkowników o wprowadzenie jednego terminu w każdym wierszu jest znacznie łatwiejsze niż umożliwienie im zrozumienia skomplikowanego formatu konfiguracji.

Następna jest funkcja sprawdzania poprawności. W oparciu o ograniczenia API Doubao zdefiniowaliśmy rygorystyczne zasady walidacji: maksymalnie 100 linii tekstu-klucza, do 50 znaków w linii i nie więcej niż 5000 znaków łącznie; `boosting_table_id` może mieć maksymalnie 200 znaków i może zawierać tylko litery, cyfry, podkreślenia i łączniki. Limity te nie są arbitralne; pochodzą bezpośrednio z oficjalnej dokumentacji Doubao. Limity API to limity API i musimy ich przestrzegać.

```typescript
export function validateContextText(contextText: string): HotwordValidationResult {
  if (!contextText || contextText.trim().length === 0) {
    return { isValid: true, errors: [] };
  }

  const lines = contextText.split('\n').filter(line => line.trim().length > 0);
  const errors: string[] = [];

  if (lines.length > 100) {
    errors.push(`Hotword line count cannot exceed 100 lines; current count is ${lines.length}`);
  }

  const totalChars = contextText.length;
  if (totalChars > 5000) {
    errors.push(`Total hotword character count cannot exceed 5000; current count is ${totalChars}`);
  }

  for (let i = 0; i < lines.length; i++) {
    if (lines[i].length > 50) {
      errors.push(`Hotword on line ${i + 1} exceeds the 50-character limit`);
    }
  }

  return { isValid: errors.length === 0, errors };
}

export function validateBoostingTableId(boostingTableId: string): HotwordValidationResult {
  if (!boostingTableId || boostingTableId.trim().length === 0) {
    return { isValid: true, errors: [] };
  }

  const errors: string[] = [];

  if (boostingTableId.length > 200) {
    errors.push(`boosting_table_id cannot exceed 200 characters; current count is ${boostingTableId.length}`);
  }

  if (!/^[a-zA-Z0-9_-]+$/.test(boostingTableId)) {
    errors.push('boosting_table_id can contain only letters, numbers, underscores, and hyphens');
  }

  return { isValid: errors.length === 0, errors };
}
```

Te funkcje sprawdzania poprawności uruchamiają się natychmiast po skonfigurowaniu przez użytkownika słów-kluczy, zapewniając wykrycie problemów tak wcześnie, jak to możliwe. Z punktu widzenia doświadczenia użytkownika tego rodzaju natychmiastowa informacja zwrotna jest bardzo ważna. Zawsze lepiej jest, aby użytkownicy wiedzieli, co jest nie tak, podczas pisania, a nie po przesłaniu.

### Trwałość konfiguracji frontendu

W implementacji frontendu HagiCode zdecydowaliśmy się na użycie przeglądarki `localStorage` do przechowywania konfiguracji słów kluczowych. Za tą decyzją projektową przemawiało kilka czynników. Po pierwsze, konfiguracja słów kluczowych jest wysoce spersonalizowana, a różni użytkownicy mogą mieć różne potrzeby specyficzne dla domeny. Po drugie, takie podejście upraszcza implementację zaplecza, ponieważ nie wymaga dodatkowych tabel bazy danych ani punktów końcowych API. Wreszcie, po jednorazowej konfiguracji ustawień w przeglądarce, ustawienia mogą zostać załadowane automatycznie przy kolejnych użyciach, co jest bardzo wygodne. Krótko mówiąc, jest to najłatwiejsze podejście.

```typescript
const HOTWORD_STORAGE_KEYS = {
  contextText: 'hotword-context-text',
  boostingTableId: 'hotword-boosting-table-id',
  combineMode: 'hotword-combine-mode',
} as const;

export const DEFAULT_HOTWORD_CONFIG: HotwordConfig = {
  contextText: '',
  boostingTableId: '',
  combineMode: false,
};

// Load hotword configuration
export function loadHotwordConfig(): HotwordConfig {
  const contextText = localStorage.getItem(HOTWORD_STORAGE_KEYS.contextText) || '';
  const boostingTableId = localStorage.getItem(HOTWORD_STORAGE_KEYS.boostingTableId) || '';
  const combineMode = localStorage.getItem(HOTWORD_STORAGE_KEYS.combineMode) === 'true';

  return { contextText, boostingTableId, combineMode };
}

// Save hotword configuration
export function saveHotwordConfig(config: HotwordConfig): void {
  localStorage.setItem(HOTWORD_STORAGE_KEYS.contextText, config.contextText);
  localStorage.setItem(HOTWORD_STORAGE_KEYS.boostingTableId, config.boostingTableId);
  localStorage.setItem(HOTWORD_STORAGE_KEYS.combineMode, String(config.combineMode));
}
```

Logika w tym kodzie jest prosta i jasna. Czytamy z `localStorage` podczas ładowania konfiguracji i napisz do `localStorage` podczas zapisywania. Udostępniamy również konfigurację domyślną, dzięki czemu system może nadal działać poprawnie, nawet jeśli nie istnieje jeszcze żadna konfiguracja. W końcu musi istnieć rozsądne rozwiązanie domyślne.

### Rozszerzenie konfiguracji zestawu SDK zaplecza

W implementacji backendu HagiCode musieliśmy dodać właściwości związane ze słowami-kluczami do klasy konfiguracyjnej SDK. Biorąc pod uwagę charakterystykę i wzorce użycia języka C#, zastosowaliśmy `List<string>` do przechowywania niestandardowych kontekstów słów kluczowych:

```csharp
public class DoubaoVoiceConfig
{
    /// <summary>
    /// App ID
    /// </summary>
    public string AppId { get; set; } = string.Empty;

    /// <summary>
    /// Access token
    /// </summary>
    public string AccessToken { get; set; } = string.Empty;

    /// <summary>
    /// Service URL
    /// </summary>
    public string ServiceUrl { get; set; } = string.Empty;

    /// <summary>
    /// Custom hotword context list
    /// </summary>
    public List<string>? HotwordContexts { get; set; }

    /// <summary>
    /// Doubao platform hotword table ID
    /// </summary>
    public string? BoostingTableId { get; set; }
}
```

Projekt tej klasy konfiguracyjnej jest zgodny ze zwykłym, zwięzłym stylem HagiCode. `HotwordContexts` jest typem listy dopuszczającym wartość null, oraz `BoostingTableId` jest ciągiem dopuszczającym wartość null, więc jeśli nie ma konfiguracji słowa-klucza, te właściwości nie mają żadnego wpływu na żądanie. Jeśli nie korzystasz z tej funkcji, powinna ona pozostać na uboczu.

### Logika konstrukcji ładunku

Konstrukcja ładunku jest podstawą całej funkcji słowa kluczowego. Kiedy już mamy konfigurację słów-kluczy, musimy sformatować je do struktury JSON wymaganej przez API Doubao. Ten proces ma miejsce, zanim pakiet SDK wyśle żądanie:

```csharp
private void AddCorpusToRequest(Dictionary<string, object> request)
{
    var corpus = new Dictionary<string, object>();

    // Add custom hotwords
    if (Config.HotwordContexts != null && Config.HotwordContexts.Count > 0)
    {
        corpus["context"] = new Dictionary<string, object>
        {
            ["context_type"] = "dialog_ctx",
            ["context_data"] = Config.HotwordContexts
                .Select(text => new Dictionary<string, object> { ["text"] = text })
                .ToList()
        };
    }

    // Add platform hotword table ID
    if (!string.IsNullOrEmpty(Config.BoostingTableId))
    {
        corpus["boosting_table_id"] = Config.BoostingTableId;
    }

    // Add corpus to the request only when it is not empty
    if (corpus.Count > 0)
    {
        request["corpus"] = corpus;
    }
}
```

Ten kod pokazuje, jak dynamicznie konstruować plik `corpus` pole w oparciu o konfigurację. Kluczową kwestią jest to, że dodamy `corpus` pole tylko wtedy, gdy faktycznie istnieje konfiguracja słowa kluczowego. Ten projekt zapewnia kompatybilność wsteczną: gdy nie skonfigurowano żadnych słów-kluczy, struktura żądań pozostaje dokładnie taka sama jak poprzednio. Kompatybilność wsteczna ma znaczenie; dodanie funkcji nie powinno zakłócać istniejącej logiki.

### Przekazywanie parametrów protokołu WebSocket

Pomiędzy frontendem a backendem parametry słów kluczowych są przesyłane za pośrednictwem komunikatów kontrolnych protokołu WebSocket. HagiCode został zaprojektowany w taki sposób, że gdy frontend rozpoczyna nagrywanie, ładuje konfigurację słowa kluczowego `localStorage` i wysyła go do backendu za pośrednictwem wiadomości WebSocket.

```typescript
const controlMessage = {
  type: 'control',
  payload: {
    command: 'StartRecognition',
    contextText: '高血压\n糖尿病\n冠心病',
    boosting_table_id: 'medical_table',
    combineMode: false
  }
};
```

Należy tu zwrócić uwagę na jeden szczegół: frontend przekazuje tekst wielowierszowy oddzielony znakami nowej linii, a backend musi go przeanalizować. Procedura obsługi WebSocket zaplecza analizuje te parametry i przekazuje je do zestawu SDK:

```csharp
private async Task HandleControlMessageAsync(
    string connectionId,
    DoubaoSession session,
    ControlMessage message)
{
    if (message.Payload is SessionControlRequest controlRequest)
    {
        // Parse hotword parameters
        string? contextText = controlRequest.ContextText;
        string? boostingTableId = controlRequest.BoostingTableId;
        bool? combineMode = controlRequest.CombineMode;

        // Parse multi-line text into a hotword list
        if (!string.IsNullOrEmpty(contextText))
        {
            var hotwords = contextText
                .Split('\n', StringSplitOptions.RemoveEmptyEntries)
                .Select(s => s.Trim())
                .Where(s => s.Length > 0)
                .ToList();

            session.HotwordContexts = hotwords;
        }

        session.BoostingTableId = boostingTableId;
    }
}
```

Dzięki temu projektowi przekazywanie konfiguracji słów kluczowych z frontendu do backendu staje się przejrzyste i wydajne. Nie ma w tym nic szczególnie tajemniczego; dane są po prostu przekazywane warstwa po warstwie.

## Praktyczny przewodnik

### Skonfiguruj niestandardowe słowa kluczowe

W rzeczywistym użyciu konfigurowanie niestandardowych słów-kluczy jest bardzo proste. Otwórz stronę ustawień rozpoznawania mowy w HagiCode i znajdź sekcję „Konfiguracja słów kluczowych”. W polu tekstowym „Niestandardowy tekst słowa kluczowego” wpisz jedno słowo kluczowe w każdym wierszu.

Na przykład, jeśli tworzysz aplikację związaną z medycyną, możesz ją skonfigurować w następujący sposób:

```
高血压
糖尿病
冠心病
心绞痛
心肌梗死
心力衰竭
```

Po zapisaniu konfiguracji te słowa kluczowe są automatycznie przekazywane do API Doubao przy każdym uruchomieniu rozpoznawania mowy. W naszych testach po skonfigurowaniu słów kluczowych zauważalnie wzrosła dokładność rozpoznawania powiązanych terminów zawodowych. Poprawa jest realna i wyraźnie lepsza niż wcześniej.

### Skonfiguruj tabelę słów kluczowych platformy

Jeśli chcesz zarządzać dużą liczbą słów-kluczy lub jeśli słowa-klucze wymagają częstych aktualizacji, lepiej sprawdzi się tryb tabeli słów-kluczy na platformie. Najpierw utwórz tabelę słów kluczowych na platformie samouczącej się Doubao i uzyskaj wygenerowaną tabelę `boosting_table_id`, następnie wprowadź ten identyfikator na stronie ustawień HagiCode.

Platforma samoucząca się Doubao zapewnia takie możliwości, jak masowy import i kategoryzowane zarządzanie słowami-kluczami, co jest bardzo praktyczne dla zespołów, które muszą zarządzać dużymi zbiorami specjalistycznej terminologii. Zarządzając słowami-kluczami na platformie, możesz zarządzać nimi centralnie i konsekwentnie wprowadzać aktualizacje. Gdy lista słów kluczowych stanie się duża, posiadanie jednego miejsca do zarządzania nią jest znacznie bardziej praktyczne niż ręczne wprowadzanie za każdym razem.

### Korzystanie z trybu kombinowanego

W niektórych złożonych scenariuszach może być konieczne jednoczesne użycie niestandardowych słów kluczowych i tabeli słów kluczowych platformy. W takim przypadku po prostu skonfiguruj oba w HagiCode i włącz przełącznik „Tryb kombinowany”.

W trybie kombinowanym interfejs API Doubao uwzględnia oba źródła słów kluczowych jednocześnie, więc dokładność rozpoznawania jest zwykle wyższa niż w przypadku korzystania z każdego źródła osobno. Warto jednak zaznaczyć, że tryb kombinowany zwiększa złożoność żądania, dlatego decyzję o włączeniu go najlepiej podjąć po praktycznych testach. Większa złożoność jest tego warta tylko wtedy, gdy uzasadniają to wyniki w świecie rzeczywistym.

### Przykłady integracji kodu

Integracja funkcji słów kluczowych z projektem HagiCode jest bardzo prosta. Oto kilka często używanych fragmentów kodu:

```typescript
import {
  loadHotwordConfig,
  saveHotwordConfig,
  validateHotwordConfig,
  parseContextText,
  getEffectiveHotwordMode,
  type HotwordConfig
} from '@/types/hotword';

// Load and validate configuration
const config = loadHotwordConfig();
const validation = validateHotwordConfig(config);

if (!validation.isValid) {
  console.error('Hotword configuration validation failed:', validation.errors);
  return;
}

// Parse hotword text
const hotwords = parseContextText(config.contextText);
console.log('Parsed hotwords:', hotwords);

// Get effective hotword mode
const mode = getEffectiveHotwordMode(config);
console.log('Current hotword mode:', mode);
```

Użycie backendu jest podobnie zwięzłe:

```csharp
var config = new DoubaoVoiceConfig
```

## Rzeczy, o których warto pamiętać

Istnieje kilka punktów, które zasługują na szczególną uwagę podczas wdrażania i używania funkcji słów kluczowych.

Po pierwsze, jest to limit znaków. Interfejs API Doubao ma ścisłe ograniczenia dotyczące słów kluczowych, w tym liczby wierszy, znaków w wierszu i całkowitej liczby znaków. Jeśli którykolwiek limit zostanie przekroczony, interfejs API zwróci błąd. W implementacji frontendowej HagiCode sprawdzamy te ograniczenia podczas wprowadzania danych przez użytkownika za pomocą funkcji walidacyjnych, co zapobiega wysyłaniu nieprawidłowych konfiguracji do backendu. Wczesne wykrycie problemów jest zawsze lepsze niż czekanie, aż interfejs API ulegnie awarii.

Po drugie, format `boosting_table_id`. W tym polu dozwolone są tylko litery, cyfry, podkreślenia i łączniki. Nie może ono zawierać spacji ani innych znaków specjalnych. Tworząc tabelę słów kluczowych na platformie samouczącej się Doubao, pamiętaj o przestrzeganiu zasad nazewnictwa. Tego rodzaju ścisła walidacja formatu jest powszechna w przypadku interfejsów API.

Trzecia to kompatybilność wsteczna. Parametry słowa kluczowego są całkowicie opcjonalne. Jeśli nie skonfigurowano żadnych słów-kluczy, system zachowuje się dokładnie tak, jak poprzednio. Taka konstrukcja zapewnia, że ​​nie ma to żadnego wpływu na istniejących użytkowników, a także ułatwia stopniową migrację i aktualizacje. Dodanie funkcji nie powinno zakłócać dotychczasowej logiki.

Na koniec pozostaje obsługa błędów. Gdy konfiguracja słowa kluczowego jest nieprawidłowa, interfejs API Doubao zwraca odpowiednie komunikaty o błędach. Implementacja HagiCode rejestruje szczegółowe logi, aby pomóc programistom w rozwiązywaniu problemów. Jednocześnie frontend wyświetla błędy walidacyjne w interfejsie użytkownika, aby pomóc użytkownikom poprawić konfigurację. Dobra obsługa błędów w naturalny sposób prowadzi do lepszego doświadczenia użytkownika.

## Wniosek

W tym artykule szczegółowo omówiliśmy kompletne rozwiązanie umożliwiające implementację słów kluczowych Doubao w projekcie HagiCode. To rozwiązanie obejmuje cały proces, od analizy wymagań i wyboru technicznego po implementację kodu, dając programistom praktyczny przykład, z którego mogą skorzystać w celach informacyjnych.

Kluczowe punkty można podsumować w następujący sposób. Po pierwsze, interfejs API Doubao obsługuje zarówno niestandardowe słowa kluczowe, jak i tabele słów kluczowych platformy i można ich używać niezależnie lub w kombinacji. Po drugie, frontend wykorzystuje `localStorage` do przechowywania konfiguracji w prosty i efektywny sposób. Po trzecie, backend przekazuje parametry słów-kluczy, dynamicznie konstruując plik `corpus` dziedzinie, zachowując silną kompatybilność wsteczną. Po czwarte, kompleksowa logika walidacji zapewnia poprawność konfiguracji i pozwala uniknąć nieprawidłowych żądań. Ogólnie rozwiązanie nie jest skomplikowane; po prostu dokładnie przestrzega wymagań API.
