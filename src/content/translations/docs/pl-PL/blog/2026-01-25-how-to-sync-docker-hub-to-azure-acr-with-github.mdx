## Automatyzacja synchronizacji obrazów z Docker Hub do Azure ACR

> W tym artykule wyjaśniono, jak używać akcji GitHub i narzędzia do synchronizacji obrazów w celu automatyzacji synchronizacji obrazów z usługi Docker Hub do Azure Container Registry, rozwiązując problem powolnego dostępu do usługi Docker Hub w Chinach i niektórych regionach świadczenia usługi Azure, jednocześnie poprawiając dostępność obrazów i efektywność wdrażania w środowiskach platformy Azure.

## Tło / Wprowadzenie

Projekt HagiCode wykorzystuje obrazy Dockera jako podstawowe komponenty środowiska wykonawczego, a główne obrazy są hostowane w Docker Hub. W miarę rozwoju projektu i rosnących potrzeb w zakresie wdrażania platformy Azure napotkaliśmy następujące problemy:

- Niskie prędkości pobierania obrazów, a dostęp do centrum Docker Hub jest ograniczony w Chinach i niektórych regionach świadczenia usługi Azure
- Pojedyncze źródło obrazu tworzy pojedynczy punkt ryzyka awarii
- Azure Container Registry zapewnia lepszą wydajność sieci i integrację podczas działania na platformie Azure

Aby rozwiązać te problemy, potrzebowaliśmy zautomatyzowanego mechanizmu synchronizacji obrazów, aby regularnie synchronizować obrazy z Docker Hub z usługą Azure ACR, zapewniając użytkownikom większą prędkość pobierania obrazów i wyższą dostępność w środowiskach platformy Azure.

## Porównanie rozwiązań technicznych

Definiując rozwiązanie, porównaliśmy wiele opcji technicznych:

### 1. synchronizator obrazu (ostateczny wybór)
- Synchronizacja przyrostowa: synchronizuje tylko zmienione warstwy obrazu, znacznie zmniejszając transfer sieciowy
- Wznów obsługę: synchronizacja może być kontynuowana po przerwach w działaniu sieci
- Kontrola współbieżności: konfigurowalne wątki robocze poprawiają wydajność synchronizacji dużych obrazów
- Solidna obsługa błędów: wbudowany mechanizm ponawiania w przypadku niepowodzeń (domyślnie 3 próby)
- Lekkie wdrożenie: pojedynczy plik binarny bez zależności
- Obsługa wielu rejestrów: zgodna z Docker Hub, Azure ACR, Harbour i nie tylko

### 2. Interfejs wiersza polecenia Dockera
- Brak synchronizacji przyrostowej: za każdym razem należy pobrać pełną zawartość obrazu
- Niższa wydajność: duży wolumen transferu sieciowego i dłuższy czas realizacji
- Prosty w użyciu: oparty na znanych `docker pull` i `docker push` polecenia

### 3. Interfejs wiersza polecenia platformy Azure
- Większa złożoność: należy skonfigurować uwierzytelnianie interfejsu wiersza polecenia platformy Azure
- Ograniczona funkcjonalność: `az acr import` ma stosunkowo wąski zakres
- Integracja natywna: dobrze integruje się z usługami Azure

### 1. Przygotowanie
- Utwórz lub potwierdź rejestr kontenerów platformy Azure w witrynie Azure Portal
- Utwórz dane uwierzytelniające dostępu ACR (nazwa użytkownika i hasło)

### 2. Skonfiguruj sekrety GitHub
Dodaj następujące wpisy tajne w ustawieniach repozytorium GitHub:
- AZURE_ACR_USERNAME: nazwa użytkownika usługi Azure ACR
- AZURE_ACR_PASSWORD: hasło usługi Azure ACR

### 3. Utwórz przepływ pracy GitHub Actions
Skonfiguruj przepływ pracy w `.github/workflows/sync-docker-acr.yml`:
- Zaplanowane wyzwalanie: codziennie o 00:00 UTC
- Ręczny spust: obsługuje `workflow_dispatch`
- Dodatkowy spust: działa po naciśnięciu przycisku `publish` gałąź do szybkiej synchronizacji

### 4. Przebieg wykonywania przepływu pracy

| Krok | Uczestnik | Akcja | Opis |
| --- | --- | --- | --- |
| 1 | Akcje GitHuba | Uruchom przepływ pracy | Wywoływane przez harmonogram, wykonanie ręczne lub wypychanie do `publish` oddział |
| 2 | Akcje GitHub → synchronizator obrazu | Pobierz i uruchom narzędzie do synchronizacji | Wejście w fazę właściwej synchronizacji |
| 3 | synchronizator obrazu → Docker Hub | Pobierz manifesty obrazów i listę tagów | Odczytuje metadane z repozytorium źródłowego |
| 4 | synchronizator obrazów → Azure ACR | Pobierz istniejące dane obrazu z repozytorium docelowego | Określa bieżący stan strony docelowej |
| 5 | synchronizator obrazu | Porównaj różnice w źródle i celu | Identyfikuje warstwy obrazu wymagające synchronizacji |
| 6 | synchronizator obrazu → Docker Hub | Wyciągnij zmienione warstwy obrazu | Przesyła tylko zawartość wymagającą aktualizacji |
| 7 | synchronizator obrazów → Azure ACR | Wciśnij zmienione warstwy obrazu | Kończy synchronizację przyrostową |

## Implementacja przepływu pracy akcji GitHub

Poniżej znajduje się rzeczywista używana konfiguracja przepływu pracy (`.github/workflows/sync-docker-acr.yml`):

```yaml
name: Sync Docker Image to Azure ACR

on:
  schedule:
    - cron: "0 0 * * *" # Every day at 00:00 UTC
  workflow_dispatch: # Manual trigger
  push:
    branches: [publish]

permissions:
  contents: read

jobs:
  sync:
    runs-on: ubuntu-latest

    steps:
      - name: Checkout code
        uses: actions/checkout@v4

      - name: Download image-syncer
        run: |
          # Download the image-syncer binary
          wget https://github.com/AliyunContainerService/image-syncer/releases/download/v1.5.5/image-syncer-v1.5.5-linux-amd64.tar.gz
          tar -zxvf image-syncer-v1.5.5-linux-amd64.tar.gz
          chmod +x image-syncer

      - name: Create auth config
        run: |
          # Generate the authentication config file (YAML format)
          cat > auth.yaml <<EOF
          hagicode.azurecr.io:
            username: "${{ secrets.AZURE_ACR_USERNAME }}"
            password: "${{ secrets.AZURE_ACR_PASSWORD }}"
          EOF

      - name: Create images config
        run: |
          # Generate the image sync config file (YAML format)
          cat > images.yaml <<EOF
          docker.io/newbe36524/hagicode: hagicode.azurecr.io/hagicode
          EOF

      - name: Run image-syncer
        run: |
          # Run synchronization (using the newer --auth and --images parameters)
          ./image-syncer --auth=./auth.yaml --images=./images.yaml --proc=10 --retries=3

      - name: Upload logs
        if: always()
        uses: actions/upload-artifact@v4
        with:
          name: sync-logs
          path: image-syncer-*.log
          retention-days: 7
```

## Szczegóły konfiguracji

### 1. Warunki wyzwalania
- Zaplanowany wyzwalacz: `cron: "0 0 * * *"` kursuje codziennie o godzinie 00:00 UTC
- Ręczny wyzwalacz: `workflow_dispatch` pozwala użytkownikom uruchomić go ręcznie z interfejsu użytkownika GitHub
- Naciśnij spust: `push: branches: [publish]` wyzwala po wypchnięciu do gałęzi wydania w celu szybkiej synchronizacji

### 2. Konfiguracja uwierzytelniania (`auth.yaml`)
```yaml
hagicode.azurecr.io:
  username: "${{ secrets.AZURE_ACR_USERNAME }}"
  password: "${{ secrets.AZURE_ACR_PASSWORD }}"
```

### 3. Konfiguracja synchronizacji obrazu
```yaml
docker.io/newbe36524/hagicode: hagicode.azurecr.io/hagicode
```
Ta konfiguracja oznacza, że wszystkie tagi z `docker.io/newbe36524/hagicode` zostanie zsynchronizowany z `hagicode.azurecr.io/hagicode`.

### 4. parametry synchronizacji obrazu
- `--auth=./auth.yaml`: ścieżka do pliku konfiguracyjnego uwierzytelniania
- `--images=./images.yaml`: ścieżka do pliku konfiguracyjnego synchronizacji obrazu
- `--proc=10`: użyj `10` współpracownicy
- `--retries=3`: niepowodzenie ponownych prób `3` razy

## Uwagi i najlepsze praktyki

### 1. Zalecenia dotyczące bezpieczeństwa
- Regularnie zmieniaj hasła usługi Azure ACR (zalecane co 90 dni)
- Użyj dedykowanego konta usługi ACR z uprawnieniami tylko do wysyłania
- Monitoruj dzienniki dostępu ACR, aby w porę wykryć nieprawidłowy dostęp
- Nie zapisuj poświadczeń w dziennikach
- Nie przekazuj poświadczeń do repozytorium kodu

### 2. Optymalizacja wydajności
- Dostosuj `--proc` parametr: dostosuj współbieżność w oparciu o przepustowość sieci (zalecane `5-20`)
- Monitoruj czas trwania synchronizacji: jeśli synchronizacja trwa zbyt długo, rozważ zmniejszenie współbieżności
- Regularnie czyść dzienniki: ustaw rozsądne `retention-days` wartość (`7` obecnie)

### 3. Rozwiązywanie problemów

#### Problem 1: Błąd uwierzytelnienia
```
Error: failed to authenticate to hagicode.azurecr.io
```
Rozwiązanie:
1. Sprawdź, czy sekrety GitHub są poprawnie skonfigurowane
2. Sprawdź, czy hasło usługi Azure ACR wygasło
3. Upewnij się, że uprawnienia konta usługi ACR są prawidłowe

#### Problem 2: Przekroczono limit czasu sieci
```
Error: timeout waiting for response
```
Rozwiązanie:
1. Sprawdź łączność sieciową
2. Zmniejsz liczbę jednoczesnych pracowników (tj `--proc` parametr)
3. Poczekaj, aż sieć się zregeneruje i ponownie uruchom przepływ pracy

#### Problem 3: Niekompletna synchronizacja obrazu
```
Warning: some tags failed to sync
```
Rozwiązanie:
1. Sprawdź dzienniki synchronizacji, aby zidentyfikować nieudane tagi
2. Ręcznie uruchom ponownie przepływ pracy, aby ponownie przeprowadzić synchronizację
3. Sprawdź, czy obraz źródłowy w Docker Hub jest w dobrej kondycji

### 4. Monitorowanie i alarmowanie
- Regularnie sprawdzaj stronę Akcje, aby potwierdzić status przepływu pracy
- Włącz powiadomienia GitHub, aby awarie przepływu pracy były natychmiast zgłaszane
- Monitoruj użycie magazynu Azure ACR
- Regularnie sprawdzaj spójność tagów w rejestrach

## Często zadawane pytania i rozwiązania

### P1: Jak zsynchronizować określone tagi zamiast wszystkich?

Zmodyfikuj `images.yaml` plik konfiguracyjny:
```yaml
# Sync only the latest and v1.0 tags
docker.io/newbe36524/hagicode:latest: hagicode.azurecr.io/hagicode:latest
docker.io/newbe36524/hagicode:v1.0: hagicode.azurecr.io/hagicode:v1.0
```

### P2: Jak zsynchronizować wiele repozytoriów obrazów?

Dodaj wiele linii do `images.yaml`:
```yaml
docker.io/newbe36524/hagicode: hagicode.azurecr.io/hagicode
docker.io/newbe36524/another-image: hagicode.azurecr.io/another-image
```

### P3: Jak ponowić próbę po nieudanej synchronizacji?
- Automatyczne ponawianie prób: program Image-Syncer zawiera wbudowany mechanizm ponawiania prób (`3` domyślnie ponawia próby)
- Ręczna ponowna próba: kliknij `Re-run all jobs` na stronie Akcje w GitHubie

### P4: Jak mogę wyświetlić szczegółowy postęp synchronizacji?
- Przeglądaj dzienniki w czasie rzeczywistym na stronie Działania
- Pobierz `sync-logs` artefakt umożliwiający sprawdzenie pełnych plików dziennika
- Pliki dziennika obejmują stan synchronizacji i prędkość transferu dla każdego znacznika

### P5: Jak długo trwa synchronizacja?
- Początkowa pełna synchronizacja: zwykle `10-30` minut, w zależności od rozmiaru obrazu
- Synchronizacja przyrostowa: zwykle `2-5` minut, jeśli zmiany obrazu są niewielkie
- Czas zależy od przepustowości sieci, rozmiaru obrazu i ustawień współbieżności

## Sugerowane rozszerzenia

### 1. Dodaj powiadomienia o synchronizacji
Dodaj krok powiadomienia do przepływu pracy:
```yaml
- name: Notify on success
  if: success()
  run: |
    echo "Docker images synced successfully to Azure ACR"
```

### 2. Zaimplementuj filtrowanie tagów obrazu
Dodaj logikę filtrowania tagów do przepływu pracy:
```yaml
- name: Filter tags
  run: |
    # Sync only tags prefixed with v
    echo "docker.io/newbe36524/hagicode:v* : hagicode.azurecr.io/hagicode:v*" > images.yaml
```

### 3. Dodaj raport statystyk synchronizacji
```yaml
- name: Generate report
  if: always()
  run: |
    echo "## Sync Report" >> $GITHUB_STEP_SUMMARY
    echo "- Total tags: $(grep -c 'synced' image-syncer-*.log)" >> $GITHUB_STEP_SUMMARY
    echo "- Sync time: ${{ steps.sync.outputs.duration }}" >> $GITHUB_STEP_SUMMARY
```
