## Jak zautomatyzować wydania Steam za pomocą akcji GitHub

> W tym artykule przedstawiono kompletne rozwiązanie, które wdrożyliśmy dla automatycznych wydań Steam w projekcie HagiCode Desktop, obejmujące kompleksowy przepływ automatyzacji z GitHub Release do platformy Steam, w tym kluczowe szczegóły techniczne, takie jak uwierzytelnianie Steam Guard i przesyłanie do wieloplatformowego magazynu.

## Tło

Proces wydawania aplikacji na platformie Steam różni się od tradycyjnej dystrybucji aplikacji. Steam ma własny, kompletny system dostarczania aktualizacji. Programiści muszą używać SteamCMD do przesyłania artefaktów kompilacji do sieci CDN Steam, zamiast po prostu umieszczać link do pobrania, jak na innych platformach.

Projekt HagiCode Desktop przygotowuje się do wydania na platformie Steam, co wprowadziło kilka nowych wyzwań do naszego przepływu pracy przy wydawaniu:

1. Musieliśmy przekonwertować istniejące artefakty kompilacji na format zgodny ze Steam.
2. Musieliśmy przesłać je na Steam przez SteamCMD.
3. Musieliśmy także zająć się uwierzytelnianiem Steam Guard.
4. Musieliśmy obsługiwać wieloplatformowe przesyłanie Depot dla systemów Linux, Windows i macOS.
5. Chcieliśmy w pełni zautomatyzowanego przepływu z GitHub Release do Steam.

W projekcie wdrożono już „tryb wersji przenośnej”, który umożliwia aplikacji wykrywanie ładunków usług stałych spakowanych w formacie `extra` informator. Naszym celem była płynna integracja trybu wersji przenośnej z dystrybucją Steam.

## Projekt Architektury

Rdzeniem całego procesu wydawania wersji Steam jest przepływ pracy GitHub Actions, który dzieli proces na trzy główne etapy:

```
┌─────────────────────────────────────────────────────────────┐
│ GitHub Actions Workflow (Steam Release)                    │
├─────────────────────────────────────────────────────────────┤
│ 1. Preparation Stage:                                      │
│    - Check out portable-version code                       │
│    - Download build artifacts from GitHub Release          │
│    - Extract and prepare the Steam content directory       │
│                                                             │
│ 2. SteamCMD Setup:                                         │
│    - Install or reuse SteamCMD                             │
│    - Authenticate with Steam Guard                         │
│                                                             │
│ 3. Release Stage:                                          │
│    - Generate Depot VDF configuration files                │
│    - Generate App Build VDF configuration files            │
│    - Invoke SteamCMD to upload to Steam                    │
└─────────────────────────────────────────────────────────────┘
```

Ten projekt ma kilka zalet:
- Wykorzystuje ponownie istniejące artefakty wydania GitHub i pozwala uniknąć ponownego tworzenia tych samych wyników.
- Korzysta z hostowanych modułów biegaczy w celu zapewnienia bezpiecznej izolacji.
- Obsługuje przełączanie między trybem podglądu a formalnymi gałęziami wydań.
- Obejmuje pełną obsługę błędów i rejestrowanie, co ułatwia diagnozowanie awarii.

## Implementacja przepływu pracy

### Projekt parametru wyzwalacza

Nasz przepływ pracy obsługuje następujące kluczowe parametry:

```yaml
inputs:
  release:           # Portable Version release tag
    description: 'Version tag to release (for example v1.0.0)'
    required: true
  steam_preview:     # Whether to generate a preview build
    description: 'Whether to use preview mode'
    required: false
    default: 'false'
  steam_branch:      # Steam branch to set live
    description: 'Target Steam branch'
    required: false
    default: 'preview'
  steam_description: # Build description override
    description: 'Build description'
    required: false
```

### Konfiguracja modułu uruchamiającego na własnym serwerze

Ze względów bezpieczeństwa używamy hostowanego na własnym serwerze biegacza z rozszerzeniem `steam` etykieta:

```yaml
runs-on:
  - self-hosted
  - Linux
  - X64
  - steam
```

Dzięki temu wydania Steam działają na dedykowanym serwerze i bezpiecznie izolują poufne dane uwierzytelniające.

### Kontrola współbieżności

Aby zapobiec wzajemnemu zakłócaniu się wydań tej samej wersji, skonfigurowaliśmy kontrolę współbieżności:

```yaml
concurrency:
  group: portable-version-steam-${{ github.event.inputs.release }}
  cancel-in-progress: false
```

Zauważ to `cancel-in-progress: false` jest tutaj ustawione, ponieważ wydania Steam mogą zająć trochę czasu, a nie chcemy, aby nowo uruchomione uruchomienie anulowało już przesyłane.

## Implementacja podstawowego skryptu

### Przygotowanie danych wejściowych do wydania

The `prepare-steam-release-input.mjs` skrypt jest odpowiedzialny za przygotowanie danych wejściowych wymaganych do wydania:

```javascript
// Download the GitHub Release build manifest and artifact inventory
const buildManifest = await downloadBuildManifest(releaseTag);
const artifactInventory = await downloadArtifactInventory(releaseTag);

// Download compressed archives for each platform
for (const platform of ['linux-x64', 'win-x64', 'osx-universal']) {
  const artifactUrl = getArtifactUrl(artifactInventory, platform);
  await downloadArtifact(artifactUrl, platform);
}

// Extract into the Steam content directory structure
await extractToSteamContent(sources, contentRoot);
```

### Uwierzytelnianie Steam Guard

Steam wymaga, aby konta były chronione za pomocą Steam Guard, dlatego wdrożyliśmy algorytm generowania kodu oparty na współdzielonym tajemnicy:

```javascript
function generateSteamGuardCode(sharedSecret, timestamp = Date.now()) {
  const secret = decodeSharedSecret(sharedSecret);
  const time = Math.floor(timestamp / 1000 / 30);

  const timeBuffer = Buffer.alloc(8);
  timeBuffer.writeBigUInt64BE(BigInt(time));

  // Use HMAC-SHA1 to generate a time-based one-time code
  const hash = crypto.createHmac('sha1', secret)
    .update(timeBuffer)
    .digest();

  // Convert it into a 5-character Steam Guard code
  const code = steamGuardCode(hash);
  return code;
}
```

Ta implementacja opiera się na mechanizmie TOTP (Time-Based One-Time Password) firmy Steam Guard, generującym nowy kod weryfikacyjny co 30 sekund.

### Generowanie konfiguracji VDF

VDF (Valve Data Format) to format konfiguracji używany przez Steam. Musimy wygenerować dwa typy plików VDF:

**Depot VDF** służy do konfiguracji treści dla każdej platformy:

```javascript
function buildDepotVdf(depotId, contentRoot) {
  return [
    '"DepotBuildConfig"',
    '{',
    `  "DepotID" "${escapeVdf(depotId)}"`,
    `  "ContentRoot" "${escapeVdf(contentRoot)}"`,
    '  "FileMapping"',
    '  {',
    '    "LocalPath" "*"',
    '    "DepotPath" "."',
    '    "recursive" "1"',
    '  }',
    '}'
  ].join('\n');
}
```

**App Build VDF** służy do konfiguracji całej kompilacji aplikacji:

```javascript
function buildAppBuildVdf(appId, depotBuilds, description, setLive) {
  const vdf = [
    '"appbuild"',
    '{',
    `  "appid" "${appId}"`,
    `  "desc" "${escapeVdf(description)}"`,
    `  "contentroot" "${escapeVdf(contentRoot)}"`,
    '  "buildoutput" "build_output"',
    '  "depots"',
    '  {'
  ];

  for (const [depotId, depotVdfPath] of Object.entries(depotBuilds)) {
    vdf.push(`    "${depotId}" "${depotVdfPath}"`);
  }

  if (setLive) {
    vdf.push(`  }`);
    vdf.push(`  "setlive" "${setLive}"`);
  }

  vdf.push('}');
  return vdf.join('\n');
}
```

### Wywołanie SteamCMD

Na koniec przesyłanie odbywa się poprzez wywołanie SteamCMD:

```javascript
await runCommand(steamcmdPath, [
  '+login', steamUsername, steamPassword, steamGuardCode,
  '+run_app_build', appBuildPath,
  '+quit'
]);
```

To ostatni skok w całym przepływie pracy. Kiedy się to uda, wydanie zostanie zakończone.

## Wieloplatformowa obsługa magazynów

Steam korzysta z systemu Depot do zarządzania zawartością dla różnych platform. Obsługujemy trzy główne Depoty:

| Platforma | Identyfikator magazynu | Wsparcie architektury |
|-----|-----------|---------|
| Linuksa | `linux-x64` | x64_64 |
| Okna | `win-x64` | x64_64 |
| macOS | `osx-universal` | uniwersalny, x64_64, ramię64 |

Każdy magazyn ma swój własny katalog zawartości i plik konfiguracyjny VDF. Dzięki temu użytkownicy na różnych platformach pobierają tylko te treści, których faktycznie potrzebują.

## Zwolnij przepływ pracy

### Krok 1: Przygotuj wersję GitHub

Najpierw utwórz wydanie GitHub w pliku `portable-version` repozytorium zawierające:
- Skompresowane archiwa dla każdej platformy
- Zbuduj manifest (`{tag}.build-manifest.json`)
- Inwentarz artefaktów (`{tag}.artifact-inventory.json`)

### Krok 2: Uruchom proces uwalniania Steam

Uruchom przepływ pracy ręcznie za pomocą GitHub Actions i uzupełnij wymagane parametry:
- `release`: znacznik wersji do opublikowania, np `v1.0.0`
- `steam_branch`: gałąź docelowa, np `preview` lub `public`
- `steam_preview`: czy używać trybu podglądu

### Krok 3: Automatycznie wykonaj proces wydania

Przepływ pracy automatycznie wykonuje następujące kroki:

1. Pobierz i wyodrębnij artefakty wydania GitHub.
2. Zainstaluj lub zaktualizuj SteamCMD.
3. Wygeneruj pliki konfiguracyjne Steam VDF.
4. Uwierzytelnij się za pomocą Steam Guard.
5. Prześlij zawartość do Steam CDN.
6. Ustaw określoną gałąź jako aktywną.

Po zakończeniu tej sekwencji cała ścieżka wydania jest objęta.

## Przewodnik konfiguracji

### Wymagana konfiguracja wpisów tajnych

Skonfiguruj następujące wpisy tajne w ustawieniach repozytorium GitHub:

| Sekretne imię | Opis |
|-----------|------|
| `STEAM_USERNAME` | Nazwa użytkownika konta Steam |
| `STEAM_PASSWORD` | Hasło do konta Steam |
| `STEAM_SHARED_SECRET` | Wspólny sekret Steam Guard (opcjonalnie) |
| `STEAM_GUARD_CODE` | Kod Steam Guard (opcjonalnie) |
| `STEAM_APP_ID` | Identyfikator aplikacji Steam |
| `STEAM_DEPOT_ID_LINUX` | Identyfikator magazynu systemu Linux |
| `STEAM_DEPOT_ID_WINDOWS` | Identyfikator magazynu systemu Windows |
| `STEAM_DEPOT_ID_MACOS` | Identyfikator magazynu macOS |

W tych ustawieniach nie ma nic szczególnie niezwykłego. Po prostu potrzebujesz wszystkich oczekiwanych wartości na swoim miejscu.

### Konfiguracja zmiennej środowiskowej

| Nazwa zmiennej | Opis | Wartość domyślna |
|---------|------|--------|
| `PORTABLE_VERSION_STEAMCMD_ROOT` | Katalog instalacyjny SteamCMD | `~/.local/share/portable-version/steamcmd` |

## Najlepsze praktyki

### Zarządzanie uwierzytelnianiem Steam Guard

Przy pierwszym uruchomieniu musisz ręcznie wprowadzić kod Steam Guard. Następnie zaleca się skonfigurowanie wspólnego sekretu, aby kod mógł być generowany automatycznie. Pozwala to uniknąć ręcznej interwencji przy każdej wersji.

SteamCMD zapisuje token logowania i może go ponownie wykorzystać przy późniejszych uruchomieniach. Nadal musisz uważać na wygaśnięcie tokena, ponieważ po wygaśnięciu tokena wymagane jest ponowne uwierzytelnienie.

### Struktura katalogu treści

Upewnij się, że struktura katalogów zawartości Steam jest poprawna:

```
steam-content/
├── linux-x64/     # Linux platform content
├── win-x64/       # Windows platform content
└── osx-universal/ # macOS universal binary content
```

Każdy katalog powinien zawierać kompletne pliki aplikacji dla odpowiedniej platformy.

### Użycie trybu podglądu

Tryb podglądu nie uruchamia żadnej gałęzi, dlatego nadaje się do testowania i walidacji:

```yaml
if [ "$STEAM_PREVIEW_INPUT" = 'true' ]; then
  cmd+=(--preview)
fi
```

Dzięki temu możesz najpierw przesłać do Steam w celu weryfikacji, a następnie przejść do formalnej gałęzi, gdy wszystko zostanie sprawdzone.

### Obsługa błędów i rejestrowanie

Skrypty obejmują pełną obsługę błędów i rejestrowanie:
- Sprawdź, czy istnieje wydanie GitHub.
- Sprawdź wymagane pliki metadanych.
- Upewnij się, że zawartość platformy jest obecna.
- Wygeneruj raport podsumowujący działania GitHub.

Informacje te są bardzo cenne zarówno przy debugowaniu, jak i inspekcji.

### Zarządzanie artefaktami

Przepływ pracy generuje dwa rodzaje artefaktów:
- `portable-steam-release-preparation-{tag}`: metadane przygotowania wydania
- `portable-steam-build-metadata-{tag}`: Metadane kompilacji Steam

Te artefakty można wykorzystać do późniejszej inspekcji i debugowania. Praktycznym rozwiązaniem domyślnym jest okres przechowywania wynoszący 30 dni.

## Wykorzystanie w świecie rzeczywistym

W projekcie HagiCode ten zautomatyzowany przepływ pracy został już pomyślnie uruchomiony w wielu wersjach. Cała droga z GitHub Release do platformy Steam jest w pełni zautomatyzowana i nie wymaga ręcznej interwencji.

To znacznie poprawiło zarówno wydajność, jak i niezawodność wydania. W przeszłości ręczne publikowanie jednej wersji zajmowało ponad 30 minut. Teraz cały proces trwa zaledwie kilka minut.

Co ważniejsze, zautomatyzowany przepływ pracy zmniejsza ryzyko błędu ludzkiego. Każde wydanie podlega temu samemu ustandaryzowanemu procesowi, dzięki czemu wyniki są bardziej przewidywalne.
