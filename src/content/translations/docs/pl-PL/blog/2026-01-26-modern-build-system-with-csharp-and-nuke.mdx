## Żegnaj piekło skryptowe: dlaczego wybraliśmy C# dla nowoczesnego systemu kompilacji

> Dokładne omówienie sposobu, w jaki projekt HagiCode wykorzystuje Nuke do tworzenia bezpiecznego dla typów, wieloplatformowego i wysoce rozszerzalnego potoku zautomatyzowanej kompilacji, w pełni rozwiązującego problemy związane z konserwacją tradycyjnych skryptów kompilacji.

## Tło

Na długiej drodze tworzenia oprogramowania słowo „kompilacja” zwykle inspiruje zarówno miłość, jak i frustrację. Kocham, bo jednym kliknięciem kod zamienia się w produkt, a to jeden z najbardziej satysfakcjonujących momentów w programowaniu. Frustracja, ponieważ utrzymywanie stosu niechlujnych skryptów kompilacji może być koszmarem.

W wielu projektach jesteśmy przyzwyczajeni do pisania skryptów w Pythonie lub bazowania na plikach konfiguracyjnych XML. Wyobraź sobie strach przed byciem rządzonym `<property>` bloki. Jednak w miarę wzrostu złożoności projektu, zwłaszcza w przypadku projektu takiego jak HagiCode, który obejmuje pracę frontendową i backendową, wiele platform i wiele języków, tradycyjne podejścia do kompilacji zaczynają pokazywać swoje ograniczenia. Rozproszona logika skryptu, brak sprawdzania typu, słaba obsługa IDE... każdy z tych problemów staje się małą pułapką, która od czasu do czasu natyka się na zespół.

Aby rozwiązać te problemy, w projekcie HagiCode wprowadzono **Nuke**, nowoczesny system kompilacji oparty na języku C#. To nie tylko narzędzie. Jest to inny sposób myślenia o przepływach pracy podczas kompilacji. Dzisiaj przyjrzymy się, dlaczego go wybraliśmy i jak radykalnie poprawiło to nasze doświadczenie w zakresie programowania.

## Analiza podstawowa: dlaczego Nuke?

Być może zastanawiasz się: „Istnieje już tak wiele systemów kompilacji, jak Make, Gradle, a nawet zwykłe skrypty Shell. Po co robić wszystko, aby używać skryptu opartego na C#?”

To jest słuszne pytanie. Główną zaletą Nuke jest to, że przenosi funkcje języka programowania, które już dobrze znamy, do świata skryptów kompilacji.

### 1. Modularyzacja procesów kompilacji: sztuka celów

Nuke ma bardzo jasną filozofię projektowania: **wszystko jest celem**.

W tradycyjnych skryptach łatwo jest skończyć z setkami splątanych ze sobą linii liniowej logiki wykonania. W Nuke dzielimy proces kompilacji na niezależny `Target`s. Każdy cel robi dokładnie jedną rzecz, na przykład:

- `Clean`: wyczyść katalog wyjściowy
- `Restore`: przywróć pakiety zależności
- `Compile`: skompiluj kod
- `Test`: uruchom testy jednostkowe

Projekt ten bardzo dobrze wpisuje się w zasadę pojedynczej odpowiedzialności. Podobnie jak elementy konstrukcyjne, cele te można dowolnie łączyć. Co ważniejsze, Nuke pozwala nam zdefiniować zależności pomiędzy Celami. Na przykład, jeśli chcesz biegać `Test`, system automatycznie sprawdzi, czy `Compile` zostało już wykonane. Jeśli chcesz `Compile`, wtedy `Restore` naturalnie jest na pierwszym miejscu.

Ten wykres zależności sprawia, że logika jest znacznie jaśniejsza i znacznie poprawia wydajność wykonywania, ponieważ Nuke może automatycznie analizować optymalną ścieżkę wykonania.

### 2. Bezpieczeństwo typu: pożegnaj koszmar literówek

Każdy, kto pisał skrypty kompilacji w Pythonie, prawdopodobnie napotkał tę niezręczną sytuację: skrypt działa przez pięć minut, a następnie kończy się niepowodzeniem, ponieważ `Confi.guration` został błędnie napisany lub ponieważ przekazano ciąg znaków w miejscu, w którym oczekiwano parametru numerycznego.

Największą zaletą pisania skryptów kompilacji w C# jest **bezpieczeństwo typu**. To oznacza:

- **Kontrola w czasie kompilacji**: IDE może powiedzieć Ci, co jest nie tak już podczas pisania, zamiast kazać czekać do czasu wykonania.
- **Bezpieczna refaktoryzacja**: jeśli chcesz zmienić nazwę zmiennej lub metody, IDE może obsłużyć to w jednym kroku, bez zmuszania Cię do ryzykownego globalnego wyszukiwania i zamiany.
- **Inteligentne uzupełnianie**: wydajna technologia IntelliSense może automatycznie uzupełniać kod, dzięki czemu nie musisz przeglądać dokumentacji, aby zapamiętać niejasne interfejsy API.

### 3. Obsługa wielu platform: jedno spójne środowisko kompilacji

W przeszłości pisaliśmy `.bat` w systemie Windows i `.sh` w systemie Linux, a następnie być może dodano skrypt w języku Python, aby wypełnić lukę. Teraz wszędzie tam, gdzie można uruchomić platformę .NET Core lub dzisiejszą platformę .NET 5+, Nuke też może działać.

Oznacza to, że członkowie zespołu mogą używać systemu Windows, Linux lub macOS i niezależnie od tego, czy wolą Visual Studio, VS Code czy Rider, wszyscy realizują dokładnie tę samą logikę. To znacznie zmniejsza problemy spowodowane różnicami w środowisku, szczególnie klasyczny problem „to działa na moim komputerze”.

### 4. Zarządzanie parametrami i konfiguracją

Nuke zapewnia bardzo elegancki mechanizm analizowania parametrów. Nie ma potrzeby ręcznego analizowania `string[] args`. Wystarczy zdefiniować właściwość i opatrzyć ją adnotacją `[Parameter]`, a Nuke automatycznie obsługuje mapowanie z argumentów wiersza poleceń i plików konfiguracyjnych.

Na przykład możemy bardzo łatwo zdefiniować konfigurację kompilacji:

```csharp
[Parameter("Configuration to build - Default is 'Debug'")]
readonly Configuration BuildConfiguration = IsLocalBuild ? Configuration.Debug : Configuration.Release;

Target Compile => _ => _
    .DependsOn(Restore)
    .Executes(() =>
    {
        // Use BuildConfiguration here; it is type-safe
        DotNetBuild(s => s
            .SetConfiguration(BuildConfiguration)
            .SetProjectFile(SolutionFile));
    });
```

Styl ten jest jednocześnie intuicyjny i odporny na błędy.

## Poradnik praktyczny: jak zastosować to w praktyce w projekcie

Sama teoria donikąd nas nie zaprowadzi. Przyjrzyjmy się, jak faktycznie zastosowaliśmy to podejście w projekcie HagiCode.

### 1. Zaplanuj strukturę projektu

Nie chcieliśmy, aby skrypty kompilacji zaśmiecały katalog główny projektu, a także nie chcieliśmy, aby struktura katalogów była tak głęboka, aby przypominała niektóre projekty Java. Dlatego umieściliśmy wszystkie pliki kompilacji związane z Nuke w jednym `nukeBuild/` katalog.

Korzyści są proste:

- Katalog główny projektu pozostaje czysty.
- Logika budowania pozostaje spójna i łatwiejsza w zarządzaniu.
- Nowi członkowie zespołu mogą natychmiast zobaczyć: „Tutaj leży logika związana z kompilacją”.

### 2. Zaprojektuj przejrzysty łańcuch zależności celu

Projektując Target, kierowaliśmy się jedną zasadą: **atomowość plus przepływ zależności**.

Każdy Cel powinien być na tyle mały, aby mógł wykonać dokładnie jedną rzecz. Na przykład `Clean` powinien usuwać jedynie pliki i nie powinien spokojnie zajmować się opakowaniami z boku.

Zalecany przepływ zależności wygląda mniej więcej tak:

`Clean` -> `Restore` -> `Compile` -> `Test` -> `Pack`

Oczywiście nie jest to absolutne. Na przykład, jeśli chcesz tylko uruchomić testy i nie chcesz niczego pakować, Nuke pozwala na wykonanie `nuke Test` bezpośrednio i automatycznie obsłuży wymagane `Restore` i `Compile` pierwsze kroki.

### 3. Solidna obsługa błędów i rejestrowanie

Co jest najbardziej frustrujące w skryptach kompilacji? Niejasne komunikaty o błędach. Nieudana kompilacja, która rejestruje tylko „Błąd: 1”, wystarczy, aby doprowadzić każdego do szaleństwa.

W Nuke, ponieważ możemy bezpośrednio korzystać z obsługi wyjątków w C#, możemy przechwytywać i raportować błędy ze znacznie większą precyzją.

```csharp
Target Publish => _ => _
    .DependsOn(Test)
    .Executes(() =>
    {
        try 
        {
            // Try publishing to NuGet
            DotNetNuGetPush(s => s
                .SetTargetPath(ArtifactPath)
                .SetSource("https://api.nuget.org/v3/index.json")
                .SetApiKey(ApiKey));
        }
        catch (Exception ex)
        {
            Log.Error($"Publishing failed. Team, please check whether the key is correct: {ex.Message}");
            throw; // Make sure the build process exits with a non-zero exit code
        }
    });
```

### 4. Używaj testów, aby chronić jakość

Skrypty kompilacji również są kodem, co oznacza, że ​​również wymagają testów. Nuke pozwala nam pisać testy dla przepływów pracy kompilacji, dzięki czemu gdy zmienimy logikę kompilacji, nie uszkodzimy przypadkowo istniejących procesów wydawania. Jest to szczególnie ważne w potokach ciągłej integracji.

## Podsumowanie

Dzięki wprowadzeniu Nuke proces kompilacji HagiCode stał się płynniejszy niż kiedykolwiek wcześniej. Nie była to tylko wymiana narzędzi. Był to postęp w sposobie myślenia inżynierów.

**Co zyskaliśmy?**

- **Łatwość konserwacji**: konfiguracja w formie kodu, jaśniejsza logika i szybsza ścieżka wdrażania nowych współpracowników.
- **Stabilność**: mocne pisanie eliminuje ponad 90% błędów niskiego poziomu.
- **Spójność**: ujednolicone doświadczenie międzyplatformowe, które eliminuje dryfowanie środowiska.

Jeśli pisanie skryptów kompilacji było jak „przemierzanie ciemności”, to używanie Nuke przypomina „chodzenie nocą przy włączonym świetle”. Jeśli masz dość utrzymywania trudnych do debugowania skryptów, rozważ przeniesienie logiki kompilacji do świata C#. Być może odkryjesz, że kompilacje mogą być również eleganckie.
