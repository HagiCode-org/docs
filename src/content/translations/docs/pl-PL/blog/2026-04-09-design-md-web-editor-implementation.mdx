## Edytuj DESIGN.md bezpośrednio w interfejsie internetowym: od pomysłu do wdrożenia

> W systemie zarządzania projektami MonoSpecs, `DESIGN.md` wykonuje projekt architektoniczny i decyzje techniczne projektu. Jednak tradycyjny proces edycji zmusza użytkowników do przejścia do zewnętrznego edytora. To fragmentaryczne doświadczenie jest jak przerwa w czytaniu wiersza: znika inspiracja i nastrój. W tym artykule opisujemy rozwiązanie, które wdrożyliśmy w projekcie HagiCode: edycję `DESIGN.md` bezpośrednio w interfejsie internetowym, z możliwością importowania szablonów z witryny projektowej online. W końcu kto nie lubi poczucia, że ​​wszystko można załatwić za jednym razem?

## Tło

Jako główny nośnik dokumentów projektowych, `DESIGN.md` zawiera kluczowe informacje, takie jak projekt architektury, decyzje techniczne i wytyczne dotyczące wdrożenia. Jednak tradycyjne podejście do edycji wymaga od użytkowników przejścia do zewnętrznego edytora, takiego jak VS Code, ręcznego zlokalizowania ścieżki fizycznej, a następnie edycji pliku. Nie jest to specjalnie skomplikowane, ale po kilkukrotnym powtórzeniu staje się męczące.

Problemy objawiają się głównie w następujący sposób:

- **Rozdrobniony przepływ pracy**: użytkownicy muszą stale przełączać się między internetowym interfejsem zarządzania a lokalnym edytorem, przerywając ciągłość przepływu pracy, podobnie jak w przypadku przerywania muzyki w środku utworu.
- **Trudne do ponownego użycia**: witryna projektowa publikuje już bogatą bibliotekę szablonów projektów, ale nie można ich zintegrować bezpośrednio z procesem edycji projektu. Dobre rzeczy istnieją, ale nadal nie można ich używać tam, gdzie są potrzebne.
- **Brakująca pętla doświadczenia**: nie ma zamkniętej pętli dla „podgląd-wybierz-import”, więc użytkownicy muszą ręcznie kopiować i wklejać, co zwiększa ryzyko błędów.
- **Problemy we współpracy**: synchronizacja dokumentów projektowych i implementacji kodu staje się procesem wymagającym wielu tarć, który negatywnie wpływa na wydajność zespołu.

Aby rozwiązać te problemy, zdecydowaliśmy się dodać obsługę bezpośredniej edycji `DESIGN.md` w interfejsie internetowym i umożliwia import szablonów jednym kliknięciem z witryny projektowej online. To nie była decyzja, która wstrząsnęłaby ziemią. Chcieliśmy po prostu sprawić, że programowanie będzie płynniejsze.

## Rozwiązanie techniczne

### Ogólna architektura

To rozwiązanie wykorzystuje architekturę oddzieloną od frontendu i backendu z serwerem proxy tego samego pochodzenia, składającą się głównie z następujących warstw. W praktyce projekt można podsumować jako „każda część wykonuje swoją pracę”:

**1. Warstwa edytora frontendu**
```typescript
// Core component: DesignMdManagementDrawer
// Location: repos/web/src/components/project/DesignMdManagementDrawer.tsx
// Responsibility: handle editing, saving, version conflict detection, and import flow
```

**2. Warstwa usług backendu**
```csharp
// Core service: ProjectAppService.DesignMd
// Location: repos/hagicode-core/src/PCode.Application/ProjectAppService.DesignMd.cs
// Responsibility: path resolution, file read/write, and version management
```

**3. Warstwa proxy tego samego pochodzenia**
```csharp
// Proxy service: ProjectAppService.DesignMdSiteIndex
// Location: repos/hagicode-core/src/PCode.Application/ProjectAppService.DesignMdSiteIndex.cs
// Responsibility: proxy design site resources, preview image caching, and security validation
```

### Kluczowe decyzje techniczne

#### Decyzja 1: Globalny wzór szuflady

Używamy pojedynczej globalnej szuflady zamiast lokalnych warstw wyskakujących, z możliwością zarządzania stanem `layoutSlice`, co zapewnia użytkownikom spójne wrażenia w różnych widokach (`classic` i `kanban`). Niezależnie od tego, z jakiego widoku użytkownik otworzy edytor, otrzymuje ten sam model interakcji. Spójne doświadczenie sprawia, że ​​ludzie czują się bardziej swobodnie, zamiast dezorientować się przy zmianie poglądów.

#### Decyzja 2: API o zasięgu projektu

Montowaliśmy `DESIGN.md`-powiązane punkty końcowe w ramach `ProjectController`, ponownie wykorzystując istniejące granice uprawnień projektu i unikając złożoności związanej z dodawaniem oddzielnego kontrolera. Dzięki temu obsługa uprawnień jest jaśniejsza i jest zgodna z organizacją zasobów RESTful. Czasami ponowne użycie ma większe znaczenie niż tworzenie czegoś nowego od zera.

#### Decyzja 3: Wykrywanie konfliktów wersji

Wyprowadzamy nieprzezroczystą wersję z systemu plików `LastWriteTimeUtc`, co zapewnia nam lekką, optymistyczną kontrolę współbieżności. Gdy wielu użytkowników jednocześnie edytuje ten sam plik, system może wykryć konflikty i poprosić użytkownika o odświeżenie. Taka konstrukcja nie blokuje edycji, jednocześnie chroniąc spójność danych.

#### Decyzja 4: Wzorzec proxy tego samego pochodzenia

Używamy `IHttpClientFactory` zastępować zewnętrzne zasoby miejsca projektowania, unikając zarówno problemów związanych z różnymi źródłami, jak i ryzyka SSRF. Dzięki temu system jest bezpieczny, a jednocześnie upraszcza połączenia z interfejsem użytkownika. Trudno być zbyt ostrożnym, jeśli chodzi o bezpieczeństwo.

## Implementacja rdzenia

### 1. Edytuj DESIGN.md bezpośrednio

#### Implementacja backendu

Backend jest głównie odpowiedzialny za rozpoznawanie ścieżek, odczyt/zapis plików i zarządzanie wersjami. Zadania te są podstawowe, ale niezbędne, jak fundament domu:

```csharp
// Path resolution and security validation
private Task<string> ResolveDesignDocumentDirectoryAsync(string projectPath, string? repositoryPath)
{
    if (string.IsNullOrWhiteSpace(repositoryPath))
    {
        return Task.FromResult(Path.GetFullPath(projectPath));
    }
    return ValidateSubPathAsync(projectPath, repositoryPath);
}

// Version generation (based on file system timestamp and size)
private static string BuildDesignDocumentVersion(string path)
{
    var fileInfo = new FileInfo(path);
    fileInfo.Refresh();
    return string.Create(
        CultureInfo.InvariantCulture,
        $"{fileInfo.LastWriteTimeUtc.Ticks:x}-{fileInfo.Length:x}");
}
```

Projekt wersji jest interesujący w swojej prostocie: wykorzystujemy czas i rozmiar ostatniej modyfikacji pliku, aby wygenerować unikalny identyfikator wersji. Jest lekki i niezawodny, nie wymaga obsługi dodatkowej bazy danych wersji. Proste rozwiązania są często najskuteczniejsze.

#### Implementacja frontendu

Na froncie wdrażamy wykrywanie stanu brudnego i zapisujemy logikę. Ten projekt pomaga użytkownikom zrozumieć, czy ich zmiany zostały zapisane i zmniejsza niepokój „co, jeśli je stracę?”:

```typescript
// Dirty-state detection and save logic
const [draft, setDraft] = useState('');
const [savedDraft, setSavedDraft] = useState('');
const isDirty = draft !== savedDraft;

const handleSave = useCallback(async () => {
    const result = await saveProjectDesignMdDocument({
        ...activeTarget,
        content: draft,
        expectedVersion: document.version, // optimistic concurrency control
    });
    setSavedDraft(draft); // update saved state
}, [activeTarget, document, draft]);
```

W tej implementacji utrzymujemy dwa elementy stanu: `draft` to treść aktualnie edytowana, podczas gdy `savedDraft` to zapisana treść. Porównanie ich mówi nam, czy istnieją niezapisane zmiany. Projekt jest prosty, ale zapewnia ludziom spokój ducha. Nikt nie chce, aby to, nad czym ciężko pracował, zniknęło.

#### Implementacja serwera proxy tego samego pochodzenia

Aby zapewnić bezpieczeństwo, backend przeprowadza ścisłą weryfikację dostępu do witryny projektu. Jeśli chodzi o bezpieczeństwo, nie można być zbyt ostrożnym:

```csharp
// Safe slug validation
private static readonly Regex SafeDesignSiteSlugRegex =
    new("^[A-Za-z0-9](?:[A-Za-z0-9._-]{0,127})$", RegexOptions.Compiled);

private static string NormalizeDesignSiteSlug(string slug)
{
    var normalizedSlug = slug?.Trim() ?? string.Empty;
    if (!IsSafeDesignSiteSlug(normalizedSlug))
    {
        throw new BusinessException(
            ProjectDesignSiteIndexErrorCodes.InvalidSlug,
            "Design site slug must be a single safe path segment.");
    }
    return normalizedSlug;
}

// Preview image caching (OS temp directory)
private static string ComputePreviewCacheKey(string slug, string theme, string previewUrl)
{
    var raw = $"{slug}|{theme}|{previewUrl}";
    var bytes = SHA256.HashData(Encoding.UTF8.GetBytes(raw));
    return Convert.ToHexString(bytes).ToLowerInvariant();
}
```

Robimy tutaj dwie rzeczy: po pierwsze, sprawdzamy format ślimaka ściśle za pomocą wyrażenia regularnego, aby zapobiec atakom polegającym na przechodzeniu ścieżki; po drugie, buforujemy obrazy podglądu, aby zmniejszyć obciążenie witryny zewnętrznej. Pierwsza to ochrona, druga to optymalizacja, a jedno i drugie ma znaczenie.

### 3. Pełny przepływ importu

```typescript
// 1. Open the import drawer
const handleRequestImportDrawer = useCallback(() => {
    setIsImportDrawerOpen(true);
}, []);

// 2. Select and import
const handleImportRequest = useCallback((entry) => {
    if (isDirty) {
        setPendingImportEntry(entry);
        setConfirmMode('import'); // overwrite confirmation
        return;
    }
    void executeImport(entry);
}, [isDirty]);

// 3. Execute import
const executeImport = useCallback(async (entry) => {
    const result = await getProjectDesignMdSiteImportDocument(
        activeTarget.projectId,
        entry.slug
    );
    setDraft(result.content); // replace editor text only, do not save automatically
    setIsImportDrawerOpen(false);
}, [activeTarget?.projectId]);
```

Proces importu opiera się na zasadzie „potwierdzenia użytkownika”: po imporcie aktualizowana jest tylko zawartość edytora i nic nie jest zapisywane automatycznie. Użytkownicy mogą sprawdzić zaimportowaną treść i zapisać ją ręcznie dopiero po potwierdzeniu, że wygląda prawidłowo. Ostateczna decyzja powinna pozostać w rękach użytkownika.

## Praktyczne przykłady

### Scenariusz 1: Tworzenie DESIGN.md w katalogu głównym projektu

Kiedy `DESIGN.md` nie istnieje, backend zwraca stan dokumentu wirtualnego. Dzięki temu frontend może uniknąć specjalnej obsługi przypadku „plik nie istnieje”, a ujednolicony interfejs API upraszcza logikę kodu:

```csharp
return new ProjectDesignDocumentDto
{
    Path = targetPath,
    Exists = false,  // virtual document state
    Content = string.Empty,
    Version = null
};

// Automatically create the file on first save
public async Task<SaveProjectDesignDocumentResultDto> SaveDesignDocumentAsync(...)
{
    Directory.CreateDirectory(targetDirectory);
    await File.WriteAllTextAsync(targetPath, input.Content);
    return new SaveProjectDesignDocumentResultDto { Created = !exists };
}
```

Zaletą tego projektu jest to, że frontend nie potrzebuje logiki specjalnych przypadków dla brakujących plików. Ukrywając tę ​​złożoność w backendzie, frontend może łatwiej skupić się na doświadczeniach użytkownika.

### Scenariusz 2: Zaimportuj szablon z witryny projektowej

Po wybraniu przez użytkownika szablonu projektu „Liniowy” w szufladzie importu, system pobiera plik `DESIGN.md` zawartość za pośrednictwem serwera proxy zaplecza. Cały proces jest przejrzysty dla użytkownika: wybiera on jedynie szablon, a system automatycznie obsługuje żądania sieciowe i transformację danych.

```typescript
// 1. The system fetches DESIGN.md content through the backend proxy
GET /api/project/{id}/design-md/site-index/linear.app

// 2. The backend validates the slug and fetches content from upstream
var entry = FindDesignSiteEntry(catalog, "linear.app");
using var upstreamResponse = await httpClient.SendAsync(request);
var content = await upstreamResponse.Content.ReadAsStringAsync();

// 3. The frontend replaces the editor text
setDraft(result.content);
// The user reviews it and then saves it manually to disk
```

Cały przepływ pozostaje przejrzysty dla użytkownika. Po prostu wybierają szablon, a system zajmuje się tworzeniem sieci i transformacją za kulisami. Właśnie takiego doświadczenia chcemy: prostego, ale potężnego.

### Scenariusz 3: Obsługa konfliktów wersji

Gdy wielu użytkowników edytuje ten sam plik `DESIGN.md` jednocześnie system wykrywa konflikty wersji. Ten optymistyczny mechanizm kontroli współbieżności zachowuje spójność danych bez blokowania edycji użytkownika:

```csharp
if (!string.Equals(currentVersion, expectedVersion, StringComparison.Ordinal))
{
    throw new BusinessException(
        ProjectDesignDocumentErrorCodes.VersionConflict,
        $"DESIGN.md at '{targetPath}' changed on disk.");
}
```

Frontend wychwytuje ten błąd i pyta użytkownika:

```typescript
// Frontend prompts the user to refresh and retry
<Alert>
    <AlertTitle>Version conflict</AlertTitle>
    <AlertDescription>
        The file was modified by another process. Please refresh to get the latest version and try again.
    </AlertDescription>
</Alert>
```

Ten optymistyczny mechanizm kontroli współbieżności utrzymuje spójność danych bez blokowania użytkowników podczas pracy. Konflikty są nieuniknione, ale przynajmniej użytkownicy powinni wiedzieć, co się stało, zamiast po cichu tracić wprowadzone zmiany.

## Uwagi i najlepsze praktyki

### 1. Bezpieczeństwo ścieżki

Zawsze sprawdzaj `repositoryPath` aby zapobiec atakom polegającym na przechodzeniu ścieżki. Jeśli chodzi o bezpieczeństwo, nigdy nie można zrobić za wiele:

```csharp
// Always validate repositoryPath to prevent path traversal attacks
return ValidateSubPathAsync(projectPath, repositoryPath);
// Reject dangerous inputs such as "../" and absolute paths
```

### 2. Strategia pamięci podręcznej

Buforuj obrazy podglądu przez 24 godziny, maksymalnie 160 plików. Umiarkowane buforowanie poprawia wydajność, ale równowaga nadal ma znaczenie:

```csharp
// Cache preview images for 24 hours, with a maximum of 160 files
private static readonly TimeSpan PreviewCacheTtl = TimeSpan.FromHours(24);
private const int PreviewCacheMaxFiles = 160;
// Periodically clean up expired cache
```

### 3. Obsługa błędów

Płynnie degraduj, gdy strona nadrzędna jest niedostępna. Taka konstrukcja gwarantuje, że nawet w przypadku awarii zależności zewnętrznej podstawowa funkcja edycji nadal będzie działać normalnie:

```typescript
// Gracefully degrade when the upstream site is unavailable
try {
    const catalog = await getProjectDesignMdSiteImportCatalog(projectId);
} catch (error) {
    toast.error(t('project.designMd.siteImport.feedback.catalogLoadFailed'));
    // The main editing drawer remains available
}
```

Ta płynna degradacja zapewnia, że ​​nawet gdy zewnętrzne zależności są niedostępne, podstawowa funkcja edycji nadal działa. System powinien być odporny, a nie zapadać się w momencie, gdy coś pójdzie nie tak.

### 4. Optymalizacja doświadczenia użytkownika

Potwierdź nadpisanie przed importem i nie zapisuj automatycznie po imporcie. Użytkownicy powinni zachować kontrolę nad swoimi działaniami:

```typescript
// Confirm overwrite before import
if (isDirty) {
    setConfirmMode('import');
    return;
}

// Do not save automatically after import; let the user confirm
setDraft(result.content); // update draft only
// The content is written to disk only after the user reviews it and clicks Save
```

### 5. Względy wydajności

Użyj fabryki klientów HTTP, aby uniknąć tworzenia zbyt wielu połączeń. Zarządzanie zasobami może wydawać się niewielkie, ale dobre wykonanie może mieć duże znaczenie:

```csharp
// Use an HTTP client factory to avoid creating too many connections
private const string DesignSiteProxyClientName = "ProjectDesignSiteProxy";
private static readonly TimeSpan DesignSiteProxyTimeout = TimeSpan.FromSeconds(8);
```

## Sugerowane rozszerzenia

1. **Ulepszenie Markdown**: obecnie używamy podstawowego `Textarea`, ale moglibyśmy uaktualnić do CodeMirror w celu podświetlania składni i skrótów klawiaturowych. Kiedy redaktor czuje się lepiej, pisanie dokumentacji też jest lepsze.
2. **Tryb podglądu**: dodaj podgląd Markdown w czasie rzeczywistym, aby poprawić jakość edycji. Zasada „to, co widzisz, to to, co dostajesz” zawsze dodaje ludziom większej pewności.
3. **Scalanie różnicowe**: zaimplementuj inteligentny algorytm scalania zamiast prostego zastępowania pełnotekstowego. Konflikty są nieuniknione, ale proces ich rozwiązywania nie musi być bolesny.
4. **Buforowanie lokalne**: pamięć podręczna `design.json` w bazie danych, aby zmniejszyć zależność od witryny zewnętrznej. Im mniej zależności ma system, tym jest on stabilniejszy.

## Podsumowanie

W projekcie HagiCode wdrożyliśmy kompletne rozwiązanie do edycji i importu online `DESIGN.md` poprzez współpracę frontend-backend. Podstawowa wartość tego rozwiązania leży w następujących punktach:

- **Wyższa wydajność**: nie ma potrzeby zmiany narzędzi; edytowanie i importowanie dokumentów projektowych może odbywać się w jednym, ujednoliconym interfejsie internetowym.
- **Niższa bariera wejścia**: import szablonów projektów jednym kliknięciem pomaga szybko rozpocząć nowe projekty.
- **Bezpieczny i niezawodny**: sprawdzanie poprawności ścieżki, wykrywanie konfliktów wersji i mechanizmy płynnej degradacji zapewniają stabilność systemu.
- **Lepsze doświadczenie użytkownika**: globalna szuflada, wykrywanie stanu złego i okna dialogowe z potwierdzeniem poprawiają ogólne wrażenia z interakcji.
