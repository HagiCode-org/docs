## Budowanie eleganckiego wdrażania nowych użytkowników w projektach React: praktyka HagiCode Driver.js

> Czy użytkownicy otwierający Twój produkt po raz pierwszy naprawdę wiedzą, od czego zacząć? W tym artykule chcę trochę opowiedzieć o tym, jak wykorzystaliśmy sterownik.js do wdrażania nowych użytkowników w projekcie HagiCode. Potraktuj to jako mały praktyczny przykład, od którego można rozpocząć rozmowę.

## Tło

Czy kiedykolwiek spotkałeś się z taką sytuacją? Nowy użytkownik rejestruje się w Twoim produkcie, otwiera stronę i od razu wygląda na zagubionego. Rozglądają się, nie wiedząc, co kliknąć i co dalej. Jako programiści często zakładamy, że użytkownicy będą po prostu „odkrywać rzeczy samodzielnie”, ponieważ w końcu ludzka ciekawość jest nieograniczona. Rzeczywistość jest jednak inna: większość użytkowników po cichu opuszcza witrynę w ciągu kilku minut, jeśli nie może znaleźć odpowiedniego punktu wejścia, tak jakby historia zaczynała się nagle i kończyła równie naturalnie.

Wdrożenie nowego użytkownika jest ważnym sposobem rozwiązania tego problemu, ale jego dobre zbudowanie nie jest takie proste. Dobry system onboardingu musi:

- Precyzyjnie lokalizuj elementy strony i zaznacz je
- Obsługuj wieloetapowe procesy wdrażania
- Zapamiętaj wybór użytkownika (ukończ lub pomiń)
- Unikaj wpływu na wydajność strony i normalną interakcję
- Staraj się, aby struktura kodu była przejrzysta i łatwa w utrzymaniu

Budując HagiCode, stanęliśmy przed tym samym wyzwaniem. HagiCode to asystent kodowania AI, a jego podstawowy przepływ pracy to przepływ pracy OpenSpec, który wygląda następująco: „użytkownik tworzy propozycję -> sztuczna inteligencja generuje plan -> użytkownik go przegląda -> sztuczna inteligencja go wykonuje. Dla użytkowników, którzy po raz pierwszy spotykają się z tą koncepcją, przepływ pracy jest zupełnie nowy, dlatego potrzebują solidnego wdrożenia, aby szybko rozpocząć pracę. Przyzwyczajenie się do nowych rzeczy zawsze wymaga trochę czasu.

## Dlaczego wybraliśmy sterownik.js

Na etapie oceny technicznej przyjrzeliśmy się kilku głównym nurtom bibliotek wprowadzających. Każdy miał swoje mocne strony:

- **Intro.js**: Potężny, ale stosunkowo duży, a dostosowywanie stylu jest nieco skomplikowane
- **Shepherd.js**: Dobrze zaprojektowane API, ale trochę za „ciężkie” dla naszego przypadku użycia
- **driver.js**: Lekki, zwięzły, intuicyjny interfejs API i dobrze działa w ekosystemie React

Ostatecznie wybraliśmy sterownik.js. Nie było żadnego szczególnie dramatycznego powodu. Wybór padł głównie na następujące względy:

1. **Lekki**: Biblioteka podstawowa jest mała i nie zwiększa znacząco rozmiaru pakietu
2. **Proste API**: Konfiguracja jest przejrzysta i intuicyjna, więc łatwo ją opanować
3. **Elastyczny**: Obsługuje niestandardowe pozycjonowanie, stylizację i zachowanie interakcji
4. **Import dynamiczny**: Można załadować na żądanie bez wpływu na wydajność pierwszego ekranu

W przypadku wyboru technologii rzadko istnieje uniwersalnie najlepsza odpowiedź. Zwykle dostępna jest tylko opcja, która najlepiej pasuje.

## Realizacja techniczna

### Konfiguracja rdzenia

Driver.js posiada bardzo intuicyjny model konfiguracji. Oto podstawowa konfiguracja, której używamy w projekcie HagiCode:

```typescript
import { driver } from 'driver.js';
import 'driver.js/dist/driver.css';

const newConversationDriver = driver({
  allowClose: true,           // Allow users to close the guide
  animate: true,              // Enable animations
  overlayClickBehavior: 'close', // Close the guide when the overlay is clicked
  disableActiveInteraction: false, // Keep elements interactive
  showProgress: false,        // Do not show the progress bar (we manage progress ourselves)
  steps: guideSteps           // Array of guide steps
});
```

Uzasadnienie tych ustawień jest następujące:

- `allowClose: true` - Szanuj wybór użytkownika i nie zmuszaj go do ukończenia poradnika
- `disableActiveInteraction: false` - Niektóre kroki wymagają rzeczywistych działań użytkownika, takich jak wpisywanie danych, więc interakcji nie można wyłączyć
- `overlayClickBehavior: 'close'` - Daj użytkownikom szybki sposób na wyjście

### Zarządzanie Państwem

Trwały stan wdrożenia jest krytyczny. Nie chcemy restartować przewodnika przy każdym odświeżeniu strony, bo to szybko staje się irytujące. Używa HagiCode `localStorage` aby zarządzać stanem przewodnika:

```typescript
export type GuideState = 'pending' | 'dismissed' | 'completed';

export interface UserGuideState {
  session: GuideState;
  detailGuides: Record<string, GuideState>;
}

// Read state
export const getUserGuideState = (): UserGuideState => {
  const state = localStorage.getItem('userGuideState');
  return state ? JSON.parse(state) : { session: 'pending', detailGuides: {} };
};

// Update state
export const setUserGuideState = (state: UserGuideState) => {
  localStorage.setItem('userGuideState', JSON.stringify(state));
};
```

Definiujemy trzy stany:

- `pending`: Przewodnik jest nadal w toku, a użytkownik go nie ukończył lub pominął
- `dismissed`: użytkownik aktywnie zamknął przewodnik
- `completed`: użytkownik wykonał wszystkie kroki

W przypadku dołączania do strony szczegółów oferty obsługujemy również bardziej szczegółowe śledzenie stanu za pośrednictwem `detailGuides` mapie, ponieważ jedna oferta może przejść przez wiele etapów — wersję roboczą, przegląd i wykonanie — a każdy etap wymaga innych wskazówek. Sytuacja stale się zmienia, a onboarding powinien to odzwierciedlać.

### Wybór elementu docelowego

Driver.js używa selektorów CSS do lokalizowania elementów docelowych. HagiCode kieruje się prostą konwencją: użyj niestandardowego `data-guide` atrybut do oznaczania celów onboardingu:

```typescript
const steps = [
  {
    element: '[data-guide="launch"]',
    popover: {
      title: 'Start a New Conversation',
      description: 'Click here to create a new conversation session...'
    }
  }
];
```

W komponentach wygląda to tak:

```typescript
<button data-guide="launch" onClick={handleLaunch}>
  New Conversation
</button>
```

Korzyści z tego podejścia są następujące:

- Unikaj konfliktów z nazwami klas stylizacji biznesowej
- Przejrzysta semantyka, dzięki czemu od razu można rozpoznać, że element jest powiązany z onboardingiem
- Łatwiejsze zarządzanie i konsekwentna konserwacja

### Dynamiczna optymalizacja importu

Ponieważ funkcja wprowadzenia jest potrzebna tylko w określonych scenariuszach, takich jak pierwsza wizyta użytkownika, korzystamy z importu dynamicznego, aby zoptymalizować wydajność początkowego ładowania:

```typescript
const initNewUserGuide = async () => {
  // Dynamically import driver.js
  const { driver } = await import('driver.js');
  await import('driver.js/dist/driver.css');

  // Initialize the guide
  const newConversationDriver = driver({
    // ...configuration
  });

  newConversationDriver.drive();
};
```

W ten sposób plik Driver.js i jego arkusz stylów są ładowane tylko wtedy, gdy jest to potrzebne i nie wpływają na wydajność pierwszego ekranu. Niewiele osób lubi czekać na coś, czego nawet jeszcze nie potrzebują.

## Projekt przepływu wdrożenia

HagiCode implementuje dwie ścieżki wdrożenia, które obejmują podstawowe scenariusze użytkownika.

### Wdrożenie sesji (10 kroków)

Ta ścieżka wprowadzenia pomaga użytkownikom przejść cały proces, od utworzenia rozmowy do przesłania pierwszej kompletnej oferty:

1. **uruchom** - Uruchom przewodnik i wprowadź przycisk „Nowa rozmowa”.
2. **utwórz** — poinstruuj użytkownika, jak wpisać żądanie w polu wprowadzania
3. **wyślij** — poinstruuj użytkownika, jak kliknąć przycisk wysyłania
4. **proposal-launch-readme** — poinstruuj użytkownika, jak utworzyć propozycję README
5. **proposal-compose-readme** — poinstruuj użytkownika, jak edytować treść żądania README
6. **proposal-submit-readme** — poinstruuj użytkownika, jak przesłać propozycję README
7. **proposal-launch-agents** — poinstruuj użytkownika, jak utworzyć `AGENTS.md` Propozycja
8. **proposal-compose-agents** — poinstruuj użytkownika, jak edytować plik `AGENTS.md` prośba
9. **agenci składania wniosków** — poinstruuj użytkownika, jak przesłać wniosek `AGENTS.md` Propozycja
10. **proposal-wait** – Wyjaśnij, że sztuczna inteligencja jest przetwarzana i użytkownik powinien chwilę poczekać

Ideą tej ścieżki jest umożliwienie użytkownikom bezpośredniego poznania podstawowego przepływu pracy HagiCode poprzez dwa rzeczywiste zadania tworzenia propozycji, jedno dla `README` i jeden dla `AGENTS.md`. Istnieje duża różnica pomiędzy usłyszeniem o przepływie pracy a samodzielnym przejściem przez niego.

### Wdrożenie szczegółów propozycji (3 kroki)

Kiedy użytkownicy wchodzą na stronę szczegółów propozycji, HagiCode uruchamia odpowiedni przewodnik w oparciu o bieżący stan propozycji:

1. **przygotowanie** (etap wersji roboczej) — poprowadź użytkownika do przeglądu planu wygenerowanego przez sztuczną inteligencję
2. **przegląd** (etap przeglądu) – Poinstruuj użytkownika, jak wykonać plan
3. **executionCompleted** (etap ukończony) — poinstruuj użytkownika, jak zarchiwizować plan

Cechą charakterystyczną tego przewodnika jest to, że jest on sterowany stanem: dynamicznie decyduje, który etap wdrożenia ma zostać wyświetlony, w oparciu o rzeczywisty stan propozycji. Rzeczy się zmieniają, a onboarding powinien zmieniać się wraz z nimi.

### Mechanizm ponawiania renderowania elementu

W aplikacjach React docelowy element dołączający mógł jeszcze nie zakończyć renderowania, na przykład dlatego, że asynchroniczne dane wciąż się ładują. Aby sobie z tym poradzić, HagiCode implementuje mechanizm ponawiania prób:

```typescript
const waitForElement = (selector: string, maxRetries = 10, interval = 100) => {
  let retries = 0;
  return new Promise<HTMLElement>((resolve, reject) => {
    const checkElement = () => {
      const element = document.querySelector(selector) as HTMLElement;
      if (element) {
        resolve(element);
      } else if (retries < maxRetries) {
        retries++;
        setTimeout(checkElement, interval);
      } else {
        reject(new Error(`Element not found: ${selector}`));
      }
    };
    checkElement();
  });
};
```

Wywołaj tę funkcję przed zainicjowaniem przewodnika, aby upewnić się, że element docelowy już istnieje. Czasami warto poczekać trochę dłużej.

## Podsumowanie najlepszych praktyk

W oparciu o praktyczne doświadczenie HagiCode, oto kilka kluczowych najlepszych praktyk:

### 1. Wdrożenie powinno być łatwe do zakończenia

Nie zmuszaj użytkowników do dokończenia wdrożenia. Niektórzy użytkownicy są z natury odkrywcami i wolą odkrywać nowe rzeczy samodzielnie. Zapewnij wyraźny przycisk „Pomiń”, zapamiętaj jego wybór i nie przeszkadzaj mu następnym razem.

### 2. Treść wprowadzająca powinna być krótka i ostra

Każdy etap wdrażania powinien koncentrować się na jednym celu:

- **Tytuł**: Krótki i jasny, najlepiej nie dłuższy niż kilka słów
- **Opis**: Przejdź od razu do rzeczy i powiedz użytkownikowi, co to jest i dlaczego jest to ważne

Unikaj rozwlekłych wyjaśnień. Uwaga użytkownika jest ograniczona podczas onboardingu, a im więcej powiesz, tym mniejsze jest prawdopodobieństwo, że to przeczyta.

### 3. Utrzymuj selektory w stabilności

Używaj stabilnych znaczników elementów, które nie zmieniają się często. Zwyczaj `data-guide` atrybut to dobry wybór. Unikaj polegania na nazwach klas lub strukturze DOM, ponieważ można je łatwo zmienić podczas refaktoryzacji. Kod zmienia się cały czas, ale niektóre kotwice powinny pozostać stabilne, jeśli to możliwe.

### 4. Przetestuj swoje wdrożenie

HagiCode zawiera kompletne przypadki testowe dla funkcji onboardingu:

```typescript
describe('NewUserConversationGuide', () => {
  it('should initialize guide state correctly', () => {
    const state = getUserGuideState();
    expect(state.session).toBe('pending');
  });

  it('should update guide state correctly', () => {
    setUserGuideState({ session: 'completed', detailGuides: {} });
    const state = getUserGuideState();
    expect(state.session).toBe('completed');
  });
});
```

Testy pomagają upewnić się, że refaktoryzacja nie zakłóca przypadkowo zachowania podczas wdrażania. Nikt nie chce, aby niewielka zmiana kodu po cichu uszkodziła wcześniej działającą funkcjonalność.

### 5. Optymalizacja wydajności

- Użyj importu dynamicznego, aby leniwie załadować bibliotekę wprowadzającą
- Unikaj inicjowania logiki dołączania, gdy użytkownik ukończył już przewodnik
- Weź pod uwagę wpływ animacji na wydajność i w razie potrzeby wyłącz je na słabszych urządzeniach

Wydajność, podobnie jak wiele rzeczy w życiu, zasługuje na odrobinę ostrożnego planowania budżetu.

## Wniosek

Wdrażanie nowych użytkowników jest ważną częścią poprawy doświadczenia użytkownika produktu. W projekcie HagiCode wykorzystaliśmy sterownik.js do zbudowania kompletnego systemu onboardingowego, który obejmuje pełny przepływ pracy od utworzenia sesji do wykonania propozycji.

Najważniejsze punkty, którymi mamy nadzieję podzielić się w tym artykule, to:

1. **Wybory techniczne powinny odpowiadać rzeczywistym potrzebom**: Driver.js nie jest najpotężniejszą opcją, ale dla nas najlepiej pasuje
2. **Zarządzanie stanem ma kluczowe znaczenie**: Użyj `localStorage` aby utrzymać stan wdrożenia i unikać wielokrotnego przerywania użytkownikom
3. **Projekt wdrożenia powinien skupiać się**: Każdy krok powinien rozwiązać jeden problem i nie więcej
4. **Struktura kodu powinna pozostać przejrzysta**: Oddzielna konfiguracja wdrożeniowa, zarządzanie stanem i logika interfejsu użytkownika w celu ułatwienia konserwacji

Jeśli dodajesz nowego użytkownika do swojego projektu, mam nadzieję, że praktyczne doświadczenia opisane w tym artykule okażą się pomocne. W tego rodzaju technologii nie ma nic szczególnie mistycznego. Próbuj dalej, podsumowuj to, czego się nauczyłeś, a wszystko stopniowo stanie się łatwiejsze.
