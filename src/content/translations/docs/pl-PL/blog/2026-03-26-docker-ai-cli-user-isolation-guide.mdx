## Uruchamianie narzędzi AI CLI w kontenerach Docker: praktyczny przewodnik po izolacji użytkowników i trwałych woluminach

> Integracja narzędzi do kodowania AI, takich jak Claude Code, Codex i OpenCode, ze środowiskami kontenerowymi wydaje się prosta, ale wszędzie kryją się ukryte komplikacje. W tym artykule szczegółowo omówiono, w jaki sposób projekt HagiCode rozwiązuje podstawowe wyzwania we wdrożeniach Dockera, w tym uprawnienia użytkowników, trwałość konfiguracji i zarządzanie wersjami, dzięki czemu można uniknąć typowych pułapek.

## Tło

Kiedy zdecydowaliśmy się uruchomić narzędzia CLI do kodowania AI w kontenerach Dockera, najbardziej intuicyjną myślą było prawdopodobnie: „Czy kontenery nie są po prostu rootowane? Dlaczego nie zainstalować wszystkiego bezpośrednio i nie zakończyć tego?” W rzeczywistości za tym pozornie prostym pomysłem kryje się kilka podstawowych problemów, które należy rozwiązać.

Po pierwsze, **ograniczenia bezpieczeństwa są pierwszą przeszkodą**. Weźmy na przykład Claude CLI: wyraźnie zabrania on uruchamiania jako użytkownik root. Jest to obowiązkowa kontrola bezpieczeństwa i jeśli zostanie wykryty root, odmówi uruchomienia. Możesz pomyśleć, czy nie mogę po prostu zmienić użytkowników za pomocą `USER` dyrektywa? To nie jest takie proste. Nadal występuje problem z mapowaniem pomiędzy użytkownikiem innym niż root w kontenerze a uprawnieniami użytkownika na komputerze-hoście.

Po drugie, **utrzymywanie stanu jest drugą pułapką**. Claude Code wymaga logowania, Codex ma własną konfigurację, a OpenCode ma także katalog pamięci podręcznej. Jeśli przy każdym ponownym uruchomieniu kontenera trzeba wszystko konfigurować na nowo, cała idea „automatyzacji” traci sens. Potrzebujemy, aby te konfiguracje przetrwały dłużej niż cykl życia kontenera.

Trzecim problemem jest **spójność uprawnień**. Czy procesy wewnątrz kontenera mogą uzyskać dostęp do plików konfiguracyjnych utworzonych przez użytkownika hosta? Niedopasowania UID/GID często powodują błędy uprawnień do plików, co jest niezwykle częste w rzeczywistych wdrożeniach.

Problemy te mogą wydawać się niezależne, ale w praktyce są ze sobą ściśle powiązane. Podczas rozwoju HagiCode stopniowo wypracowywaliśmy praktyczne rozwiązanie. Następnie podzielę się szczegółami technicznymi i wnioskami wyciągniętymi z tych pułapek.

## Dlaczego nie możemy po prostu użyć roota?

Występuje tu powszechne nieporozumienie: kontenery Dockera domyślnie działają jako root, więc dlaczego nie po prostu zainstalować narzędzi jako root? Jeśli tak myślisz, Claude CLI szybko nauczy Cię, że jest inaczej.

```bash
# Run Claude CLI directly as root? No.
docker run --rm -it --user root myimage claude
# Output: Error: This command cannot be run as root user
```

Jest to twarde ograniczenie bezpieczeństwa w Claude CLI. Powód jest prosty: te narzędzia CLI odczytują i zapisują wrażliwą konfigurację użytkownika, w tym tokeny API, lokalne pamięci podręczne, a nawet skrypty napisane przez użytkownika. Uruchamianie ich z uprawnieniami roota wiąże się ze zbyt dużym ryzykiem.

Powstaje zatem pytanie: w jaki sposób możemy spełnić wymagania bezpieczeństwa CLI, zachowując jednocześnie elastyczność zarządzania kontenerami? Musimy zmienić sposób, w jaki o tym myślimy: **zamiast przełączać użytkowników w czasie wykonywania, utwórz dedykowanego użytkownika na etapie tworzenia obrazu**.

## Tworzenie dedykowanego użytkownika: więcej niż tylko zmiana nazwy

Możesz pomyśleć, że dodanie pojedynczego `USER` wystarczy linia do pliku Dockerfile. To rzeczywiście najprostsza metoda, ale nie jest ona wystarczająco solidna.

### Tworzenie statyczne a mapowanie dynamiczne

Podejście HagiCode polega na stworzeniu pliku `hagicode` użytkownik z UID 1000, który zwykle odpowiada domyślnemu użytkownikowi na większości komputerów-hostów:

```dockerfile
RUN groupadd -o -g 1000 hagicode && \
    useradd -o -u 1000 -g 1000 -s /bin/bash -m hagicode && \
    mkdir -p /home/hagicode/.claude && \
    chown -R hagicode:hagicode /home/hagicode
```

Ale to rozwiązuje tylko wbudowanego użytkownika wewnątrz obrazu. Co się stanie, jeśli użytkownik hosta ma identyfikator UID 1001? Nadal musisz obsługiwać mapowanie dynamiczne podczas uruchamiania kontenera.

docker-entrypoint.sh zawiera kluczową logikę:

```bash
if [ -n "$PUID" ] && [ -n "$PGID" ]; then
    if ! id hagicode >/dev/null 2>&1; then
        groupadd -g "$PGID" hagicode
        useradd -u "$PUID" -g "$PGID" -s /bin/bash -m hagicode
    fi
fi
```

Zaleta tego projektu jest jasna: **użyj domyślnego UID 1000 w czasie tworzenia obrazu, a następnie dostosuj dynamicznie w czasie wykonywania poprzez `PUID` i `PGID` zmienne środowiskowe**. Bez względu na UID użytkownika hosta, własność plików konfiguracyjnych pozostaje prawidłowa.

## Filozofia projektowania trwałych woluminów

Każde narzędzie AI CLI ma swój własny preferowany katalog konfiguracyjny, dlatego należy je mapować jeden po drugim:

| Narzędzie CLI | Ścieżka w kontenerze | Nazwany wolumin |
|---------|-----------|--------|
| Klaudiusz | `/home/hagicode/.claude` | `claude-data` |
| Kodeks | `/home/hagicode/.codex` | `codex-data` |
| Otwarty kod | `/home/hagicode/.config/opencode` | `opencode-config-data` |

Po co używać nazwanych woluminów zamiast montowania powiązań? Trzy powody:

1. **Prostsze zarządzanie**: Nazwane woluminy są zarządzane automatycznie przez Docker, więc nie musisz ręcznie tworzyć katalogów hosta.
2. **Izolacja uprawnień**: Początkowa zawartość woluminów jest tworzona przez użytkownika w kontenerze, co pozwala uniknąć konfliktów uprawnień z hostem.
3. **Niezależna migracja**: Woluminy mogą istnieć niezależnie od kontenerów, więc dane nie zostaną utracone podczas aktualizacji obrazów.

docker-compose-builder-web automatycznie generuje odpowiednią konfigurację woluminu:

```yaml
volumes:
  claude-data:
  codex-data:
  opencode-config-data:

services:
  hagicode:
    volumes:
      - claude-data:/home/hagicode/.claude
      - codex-data:/home/hagicode/.codex
      - opencode-config-data:/home/hagicode/.config/opencode
    user: "${PUID:-1000}:${PGID:-1000}"
```

Zwróć uwagę na `user` pole tutaj: `PUID` i `PGID` są wstrzykiwane poprzez zmienne środowiskowe, aby zapewnić, że procesy wewnątrz kontenera będą działać z tożsamością pasującą do użytkownika hosta. Ten szczegół ma znaczenie, ponieważ problemy z uprawnieniami są trudne do debugowania, gdy się pojawią.

## Zarządzanie wersjami: wersje wbudowane z nadpisaniami w czasie wykonywania

Przypinanie wersji obrazu platformy Docker jest niezbędne dla odtwarzalności. Ale w prawdziwym rozwoju często musimy przetestować nowszą wersję lub pilnie naprawić błąd. Gdybyśmy za każdym razem musieli odbudowywać obraz, przepływ pracy byłby zdecydowanie zbyt nieefektywny.

Strategia HagiCode to **domyślne wersje stałe, z nadpisaniami w czasie wykonywania jako mechanizmem rozszerzenia**. Jest to pragmatyczny kompromis inżynieryjny pomiędzy stabilnością i elastycznością.

Wersje pinów Dockerfile.template tutaj:

```dockerfile
USER hagicode
WORKDIR /home/hagicode

# Configure the global npm install path
RUN mkdir -p /home/hagicode/.npm-global && \
    npm config set prefix '/home/hagicode/.npm-global'

# Install CLI tools using pinned versions
RUN npm install -g @anthropic-ai/claude-code@2.1.71 && \
    npm install -g @openai/codex@0.112.0 && \
    npm install -g opencode-ai@1.2.25 && \
    npm cache clean --force
```

docker-entrypoint.sh obsługuje zastąpienia środowiska wykonawczego:

```bash
install_cli_override_if_needed() {
    local package_name="$2"
    local override_version="$5"

    if [ -n "$override_version" ]; then
        gosu hagicode npm install -g "${package_name}@${override_version}"
    fi
}

# Example usage
install_cli_override_if_needed "" "@anthropic-ai/claude-code" "" "" "${CLAUDE_CODE_CLI_VERSION}"
```

Pozwala to przetestować nową wersję za pomocą zmiennej środowiskowej bez konieczności przebudowy obrazu:

```bash
docker run -e CLAUDE_CODE_CLI_VERSION=2.2.0 myimage
```

Ten projekt jest praktyczny, ponieważ nikt nie chce odbudowywać obrazu za każdym razem, gdy testuje nową funkcję.

## Automatyczny wtrysk konfiguracji

Oprócz ręcznego konfigurowania narzędzi CLI niektóre scenariusze wymagają automatycznego wstrzykiwania konfiguracji. Najbardziej typowym przykładem jest token API.

```bash
if [ -n "$ANTHROPIC_AUTH_TOKEN" ]; then
    mkdir -p /home/hagicode/.claude
    cat > /home/hagicode/.claude/settings.json <<EOF
{
  "env": {
    "ANTHROPIC_AUTH_TOKEN": "${ANTHROPIC_AUTH_TOKEN}"
  }
}
EOF
    chown -R hagicode:hagicode /home/hagicode/.claude
fi
```

Liczą się tutaj dwie rzeczy: **przekazuj wrażliwe informacje przez zmienne środowiskowe, zamiast wpisywać je na stałe w obrazie** i upewnij się, że własność plików konfiguracyjnych jest ustawiona poprawnie, w przeciwnym razie narzędzia CLI nie będą mogły ich odczytać.

## Najlepsze praktyki i lista kontrolna pułapek

### Problemy z niezgodnością uprawnień

To pułapka, w którą najłatwiej wpaść. Użytkownik hosta ma UID 1001, natomiast kontener używa 1000, zatem nie można uzyskać dostępu do plików utworzonych po jednej stronie po drugiej stronie.

```bash
# Correct approach: make the container match the host user
docker run \
    -e PUID=$(id -u) \
    -e PGID=$(id -g) \
    myimage
```

Ten problem jest bardzo powszechny i może być frustrujący, gdy napotkasz go po raz pierwszy.

### Konfiguracja znika po ponownym uruchomieniu kontenera

Jeśli po każdym ponownym uruchomieniu logujesz się ponownie, sprawdź, czy nie zapomniałeś zamontować woluminu stałego:

```yaml
volumes:
  - claude-data:/home/hagicode/.claude
```

Nie ma nic bardziej frustrującego niż staranne skonfigurowanie konfiguracji tylko po to, by zobaczyć, jak znika.

### Właściwy sposób aktualizacji wersji

Nie biegaj `npm install -g` bezpośrednio w działającym kontenerze. Prawidłowe podejścia to:

1. Ustaw zmienną środowiskową, aby uruchomić instalację zastępującą.
2. Lub odbuduj wizerunek.

```bash
# Option 1: runtime override
docker run -e CLAUDE_CODE_CLI_VERSION=2.2.0 myimage

# Option 2: rebuild the image
docker build -t myimage:v2 .
```

Do Rzymu prowadzi więcej niż jedna droga, ale niektóre są gładsze od innych.

### Lista kontrolna wzmacniania zabezpieczeń

- Przekazuj tokeny API przez zmienne środowiskowe zamiast zapisywać je w obrazie.
- Ustaw uprawnienia pliku konfiguracyjnego na `600`.
- Zawsze uruchamiaj aplikację jako użytkownik inny niż root.
- Regularnie aktualizuj wersje CLI, aby naprawić luki w zabezpieczeniach.

Bezpieczeństwo jest zawsze ważne, ale prawdziwym wyzwaniem jest konsekwentne egzekwowanie go w praktyce.

## Wniosek

Uruchamianie narzędzi AI CLI w kontenerach Docker wiąże się z trzema głównymi wyzwaniami: **uprawnieniami użytkowników, trwałością konfiguracji i zarządzaniem wersjami**. Łącząc dedykowanych użytkowników, izolację nazwanych woluminów i zastąpienia oparte na zmiennych środowiskowych, w ramach projektu HagiCode zbudowano architekturę wdrażania, która jest zarówno bezpieczna, jak i elastyczna.

Kluczowe punkty projektowe:

- **Izolacja użytkownika**: Utwórz dedykowanego użytkownika na etapie tworzenia obrazu, z obsługą dynamiki w czasie wykonywania `PUID`/`PGID` mapowanie.
- **Strategia trwałości**: Każde narzędzie CLI otrzymuje swój własny nazwany wolumin, więc ponowne uruchomienie nie ma wpływu na konfigurację.
- **Elastyczność wersji**: Stałe ustawienia domyślne zapewniają powtarzalność, natomiast zastąpienie w czasie wykonywania zapewnia miejsce na testowanie.
- **Konfiguracja automatyczna**: Wrażliwa konfiguracja może zostać wprowadzona automatycznie poprzez zmienne środowiskowe.

Rozwiązanie to działa stabilnie w projekcie HagiCode już od jakiegoś czasu i mam nadzieję, że będzie stanowić przydatny punkt odniesienia dla programistów o podobnych potrzebach.
