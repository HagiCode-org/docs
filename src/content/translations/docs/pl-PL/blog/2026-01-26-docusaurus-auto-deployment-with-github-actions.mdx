## Dodanie obsługi automatycznego wdrażania stron GitHub do HagiCode

> Projekt ten pierwotnie nosił nazwę kodową PCode, a obecnie został oficjalnie przemianowany na HagiCode. W tym artykule opisano, jak wprowadziliśmy automatyczne wdrażanie witryn statycznych, dzięki czemu publikowanie treści stało się tak proste, jak picie wody.

## Tło / Wprowadzenie

Podczas opracowywania HagiCode napotkaliśmy bardzo praktyczny problem: w miarę jak nasza dokumentacja i propozycje stale się rozrastały, efektywne zarządzanie i prezentowanie tej treści stawało się coraz pilniejsze. Zdecydowaliśmy się używać GitHub Pages do hostowania naszej statycznej witryny, ale ręczne budowanie i wdrażanie było po prostu zbyt kłopotliwe. Każda zmiana wymagała lokalnej kompilacji, opakowania, a następnie ręcznego przeniesienia do pliku `gh-pages` oddział. Było to nie tylko nieefektywne, ale także łatwe do popełnienia błędu.

Aby rozwiązać ten problem, głównie dlatego, że chcieliśmy zaoszczędzić trochę wysiłku, potrzebowaliśmy zautomatyzowanego przepływu pracy podczas wdrażania. W tym artykule szczegółowo opisano, jak dodaliśmy obsługę automatycznego wdrażania opartą na GitHub Actions do projektu HagiCode, dzięki czemu możemy skupić się na pisaniu treści i pozwolić automatyzacji zająć się resztą.

## Analiza celu

Zanim przejdziemy do implementacji, musimy najpierw wyjaśnić, co właściwie ma osiągnąć to zadanie. Przecież ostrzenie siekiery nie opóźnia rąbania drewna opałowego.

### Podstawowe wymagania

1.  **Automatyczna kompilacja**: automatycznie uruchamiaj przepływ pracy kompilacji, gdy kod jest przesyłany do `main` oddział.
2.  **Automatyczne wdrażanie**: po pomyślnej kompilacji automatycznie wdrażaj wygenerowane pliki statyczne na stronach GitHub.
3.  ** Spójność środowiska**: Upewnij się, że środowisko CI pasuje do lokalnego środowiska kompilacji, aby uniknąć klasycznego zawstydzenia w postaci „działa lokalnie, ale zawodzi w trybie online”.

### Wybór techniczny

Ponieważ HagiCode jest zbudowany na Docusaurusie, bardzo popularnym generatorze stron statycznych React, możemy użyć GitHub Actions, aby osiągnąć ten cel.

## Konfigurowanie przepływu pracy akcji GitHub

GitHub Actions to usługa CI/CD świadczona przez GitHub. Definiując pliki przepływu pracy w formacie YAML wewnątrz repozytorium, możemy dostosować wszelkiego rodzaju zadania automatyzacji.

### Tworzenie pliku przepływu pracy

Musimy utworzyć nowy plik konfiguracyjny w formacie `.github/workflows` na przykład katalog w katalogu głównym projektu `deploy.yml`. Jeśli katalog jeszcze nie istnieje, utwórz go najpierw.

Podstawowa logika tego pliku konfiguracyjnego jest następująca:

1.  **Warunek wyzwalania**: Słuchaj `push` wydarzenia na `main` oddział.
2.  **Środowisko wykonawcze**: Najnowsza wersja Ubuntu.
3.  **Kroki tworzenia**:
    -   Sprawdź kod
    -   Zainstaluj Node.js
    -   Zainstaluj zależności (`npm install`)
    -   Zbuduj pliki statyczne (`npm run build`)
4.  **Etap wdrożenia**: Użyj oficjalnego `action-gh-pages` akcja, aby wypchnąć dane wyjściowe kompilacji do `gh-pages` oddział.

### Kluczowy kod konfiguracji

Poniżej znajduje się szablon konfiguracji, który ostatecznie przyjęliśmy:

```yaml
name: Deploy to GitHub Pages

# Trigger condition: run when code is pushed to the main branch
on:
  push:
    branches:
      - main
    # Add path filters if needed, for example only build when docs change
    # paths:
    #   - 'docs/**'
    #   - 'package.json'

# Set permissions, which is important for GitHub Pages deployment
permissions:
  contents: read
  pages: write
  id-token: write

# Concurrency control: cancel older builds for the same branch
concurrency:
  group: "pages"
  cancel-in-progress: false

jobs:
  build:
    runs-on: ubuntu-latest
    steps:
      - name: Checkout
        uses: actions/checkout@v4
        # Note: fetch-depth: 0 is required, otherwise the build version may be inaccurate
        with:
          fetch-depth: 0

      - name: Setup Node
        uses: actions/setup-node@v4
        with:
          node-version: 20 # Keep this consistent with the local development environment
          cache: 'npm'     # Enabling cache can speed up the build process

      - name: Install dependencies
        run: npm ci
        # Use npm ci instead of npm install because it is faster and stricter, which makes it better for CI

      - name: Build website
        run: npm run build
        env:
          # Configure environment variables here if your site build requires them
          # NODE_ENV: production
          # PUBLIC_URL: /your-repo-name
          
      - name: Upload artifact
        uses: actions/upload-pages-artifact@v3
        with:
          path: ./build # Default Docusaurus output directory

  deploy:
    environment:
      name: github-pages
      url: ${{ steps.deployment.outputs.page_url }}
    runs-on: ubuntu-latest
    needs: build
    steps:
      - name: Deploy to GitHub Pages
        id: deployment
        uses: actions/deploy-pages@v4
```

## Typowe pułapki podczas wdrażania

W praktyce napotkaliśmy kilka problemów. Dzielę się nimi tutaj w nadziei, że inni będą mogli ich uniknąć lub przynajmniej przygotować rozwiązania z wyprzedzeniem.

### 1. Problemy z uprawnieniami do tokena GitHub

Kiedy po raz pierwszy wszystko konfigurowaliśmy, wdrożenie nadal kończyło się niepowodzeniem i wyświetlało się komunikat 403 (Zabronione). Po wielu poszukiwaniach odkryliśmy, że GitHub jest domyślny `GITHUB_TOKEN` nie miał uprawnień do pisania w Pages.

**Rozwiązanie**: W repozytorium `Settings` -> `Actions` -> `General` -> `Workflow permissions`, pamiętaj o wybraniu **„Uprawnienia do odczytu i zapisu”**.

### 2. Nieprawidłowa ścieżka do katalogu kompilacji

Docusaurus umieszcza zbudowane pliki statyczne w formacie `build` domyślnie katalog. Jednak niektóre projekty mogą być skonfigurowane inaczej. Na przykład domyślnie opcja Utwórz aplikację React ma wartość `build`, podczas gdy Vite domyślnie `dist`. Jeśli Actions zgłosi, że nie może znaleźć plików wyjściowych, sprawdź ścieżkę wyjściową `docusaurus.config.js`.

### 3. Problemy z podścieżką

Jeśli Twoje repozytorium nie jest stroną główną użytkownika, to znaczy, że nią nie jest `username.github.io`, ale zamiast tego strona projektu, taka jak `username.github.io/project-name`, następnie musisz skonfigurować `baseUrl`.

w `docusaurus.config.js`:

```javascript
module.exports = {
  // ...
  url: 'https://hagicode.com', // Your Hagicode URL
  baseUrl: '/', // Deploy from the root path
  // ...
};
```
Ten szczegół łatwo przeoczyć. Jeśli jest skonfigurowana nieprawidłowo, strona może zostać załadowana jako pusty ekran, ponieważ nie można rozpoznać ścieżek zasobów.

## Weryfikacja wyniku

Po skonfigurowaniu wszystkiego i wciśnięciu kodu możemy przejść do zakładki **Akcje** w repozytorium GitHub i obejrzeć pokaz.

Podczas wykonywania przepływu pracy zobaczysz żółte kółko, a gdy zmieni kolor na zielony, będzie to oznaczać sukces. Jeśli zmieni kolor na czerwony, kliknij i sprawdź logi. W większości przypadków możesz tam prześledzić problem, zazwyczaj jest to literówka lub nieprawidłowa konfiguracja ścieżki.

Gdy kompilacja się powiedzie, odwiedź `https://<your-username>.github.io/<repository-name>/` i zobaczysz świeżo wdrożoną witrynę.

## Podsumowanie

Wprowadzając GitHub Actions, pomyślnie zautomatyzowaliśmy wdrażanie witryny z dokumentacją HagiCode. To nie tylko oszczędza czas spędzony wcześniej na ręcznych operacjach, ale, co ważniejsze, standaryzuje przepływ pracy przy wydawaniu. Teraz niezależnie od tego, który członek zespołu zaktualizuje dokumentację, po włączeniu zmiany `main`, najnowsza zawartość będzie dostępna kilka minut później.

**Podstawowe korzyści**:
- **Wyższa wydajność**: Przeszliśmy z „ręcznego pakowania i ręcznego przesyłania” na „kod równa się wydaniu”.
- **Mniej błędów**: Wyeliminowaliśmy możliwość błędu ludzkiego podczas operacji ręcznych.
- **Lepsze doświadczenie dla programistów**: programiści mogą bardziej skoncentrować się na jakości treści, zamiast rozpraszać się żmudnymi etapami wdrażania.

Chociaż konfiguracja CI/CD jest na początku nieco kłopotliwa, szczególnie w przypadku wszystkich uprawnień i szczegółów ścieżki, jest to jednorazowa inwestycja, która zapewnia duże długoterminowe zyski. Zdecydowanie zalecam, aby każdy projekt strony statycznej stosował podobny zautomatyzowany przepływ pracy.
