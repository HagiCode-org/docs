## Głęboka integracja i praktyczne wykorzystanie StreamJsonRpc w HagiCode

> W tym artykule szczegółowo wyjaśniono, jak projekt HagiCode (dawniej PCode) pomyślnie zintegrował bibliotekę komunikacyjną StreamJsonRpc firmy Microsoft, zastępując jej oryginalną, niestandardową implementację JSON-RPC, rozwiązując jednocześnie problemy techniczne i wyzwania architektoniczne napotykane podczas procesu integracji.

## Tło

StreamJsonRpc to oficjalnie obsługiwana biblioteka komunikacyjna JSON-RPC firmy Microsoft dla .NET i TypeScript, znana z wysokiego bezpieczeństwa typów, automatycznego generowania serwerów proxy i dojrzałych mechanizmów obsługi wyjątków. W projekcie HagiCode zespół zdecydował się zintegrować StreamJsonRpc w celu komunikacji z zewnętrznymi narzędziami AI, takimi jak iflow CLI i OpenCode CLI poprzez ACP (Agent Communication Protocol), eliminując jednocześnie koszty utrzymania i potencjalne błędy wprowadzone przez wcześniejszą niestandardową implementację JSON-RPC. Jednak proces integracji napotkał wyzwania specyficzne dla przesyłania strumieniowego JSON-RPC, zwłaszcza podczas obsługi wiązania celu proxy i rozpoznawania parametrów ogólnych.

Aby rozwiązać te problemy, podjęliśmy odważną decyzję: przebudować cały system kompilacji od podstaw. Skutki tej decyzji mogą być jeszcze większe, niż się spodziewasz, co wkrótce wyjaśnię szczegółowo.

## Analiza

Obecny projekt znajduje się w krytycznym etapie integracji protokołu ACP i stoi przed następującymi problemami technicznymi i wyzwaniami architektonicznymi:

### 1. Ograniczenia wdrożenia niestandardowego
Oryginalna implementacja JSON-RPC znajduje się w `src/HagiCode.ClaudeHelper/AcpImp/` i zawiera elementy takie jak `JsonRpcEndpoint` i `ClientSideConnection`. Utrzymanie tej niestandardowej bazy kodu jest kosztowne i brakuje w niej zaawansowanych możliwości dojrzałej biblioteki, takich jak raportowanie postępu i obsługa anulowania.

### 2. Przeszkody w integracji StreamJsonRpc
Podczas próby migracji istniejących `CallbackProxyTarget` wzorzec do StreamJsonRpc, odkryliśmy, że plik `_rpc.AddLocalRpcTarget(target)` metoda nie mogła rozpoznać obiektów docelowych utworzonych za pomocą wzorca proxy. W szczególności StreamJsonRpc nie mógł automatycznie podzielić właściwości typu ogólnego `T` w parametry metody RPC, powodując awarię po stronie serwera podczas przetwarzania wywołań metod inicjowanych przez klienta.

### 3. Zagmatwane nawarstwianie architektoniczne
Istniejący `ClientSideConnection` łączy warstwę transportową (WebSocket/Stdio), warstwę protokołu (JSON-RPC) i warstwę biznesową (interfejs agenta ACP), co prowadzi do niejasnych obowiązków. Problem dotyczy również brakujących powiązań metod w `AcpAgentCallbackRpcAdapter`.

### 4. Brakujące logi
Warstwa transportowa protokołu WebSocket nie rejestruje nieprzetworzonej zawartości JSON, co utrudnia określenie podczas debugowania RPC, czy problem pochodzi z serializacji, czy z sieci.

## Rozwiązanie

Aby rozwiązać powyższe problemy, przyjęliśmy następujące systematyczne rozwiązanie, optymalizując w trzech wymiarach: refaktoryzacja architektury, integracja bibliotek i ulepszone debugowanie.

### 1. Dokonaj pełnej migracji do StreamJsonRpc

#### Usuń starszy kod
Usuń `JsonRpcEndpoint.cs`, `AgentSideConnection.cs`i powiązane niestandardowe konwertery serializacji, takie jak `JsonRpcMessageJsonConverter`.

#### Zintegruj oficjalną bibliotekę
Przedstaw `StreamJsonRpc` Pakiet NuGet i użyj jego `JsonRpc` klasa do obsługi podstawowej logiki komunikacji.

#### Streszczenie warstwy transportowej
Zdefiniuj `IAcpTransport` interfejs do obsługi obu `WebSocket` i `Stdio` trybów transportu w ujednolicony sposób, zapewniając oddzielenie warstwy protokołu od warstwy transportowej.

```csharp
// Definition of the IAcpTransport interface
public interface IAcpTransport
{
    Task SendAsync(string message, CancellationToken cancellationToken = default);
    Task<string> ReceiveAsync(CancellationToken cancellationToken = default);
    Task CloseAsync(CancellationToken cancellationToken = default);
}

// WebSocket transport implementation
public class WebSocketTransport : IAcpTransport
{
    private readonly WebSocket _webSocket;

    public WebSocketTransport(WebSocket webSocket)
    {
        _webSocket = webSocket;
    }

    // Implement send and receive methods
    // ...
}

// Stdio transport implementation
public class StdioTransport : IAcpTransport
{
    private readonly StreamReader _reader;
    private readonly StreamWriter _writer;

    public StdioTransport(StreamReader reader, StreamWriter writer)
    {
        _reader = reader;
        _writer = writer;
    }

    // Implement send and receive methods
    // ...
}
```

### 2. Napraw problemy z rozpoznawaniem celów proxy

#### Analizuj `CallbackProxyTarget`
Sprawdź istniejącą logikę generowania dynamicznego serwera proxy i zidentyfikuj główną przyczynę, dla której StreamJsonRpc nie może jej rozpoznać. W większości przypadków dzieje się tak, ponieważ obiekt proxy nie udostępnia publicznie rzeczywistych podpisów metod lub używa typów parametrów nieobsługiwanych przez StreamJsonRpc.

#### Przekazywanie parametrów refaktoryzacji
Podziel właściwości ogólne na jawne parametry metody RPC. Zamiast polegać na właściwościach dynamicznych, zdefiniuj konkretne DTO żądania/odpowiedzi (obiekty przesyłania danych), aby StreamJsonRpc mógł poprawnie rozpoznać sygnatury metod poprzez odbicie.

```csharp
// Original generic property approach
public class CallbackProxyTarget<T>
{
    public Func<T, Task> Callback { get; set; }
}

// Refactored concrete method approach
public class ReadTextFileRequest
{
    public string FilePath { get; set; }
}

public class ReadTextFileResponse
{
    public string Content { get; set; }
}

public interface IAcpAgentCallback
{
    Task<ReadTextFileResponse> ReadTextFileAsync(ReadTextFileRequest request);
    // Other methods...
}
```

#### Użyj `Attach` Zamiast `AddLocalRpcTarget`
W niektórych złożonych scenariuszach ręczne proxy `JsonRpc` przedmiot i obsługa `RpcConnection` może być bardziej elastyczne niż bezpośrednie dodanie celu lokalnego.

### 3. Zaimplementuj wiązanie metod i silniejsze rejestrowanie

#### Wdrożyć `AcpAgentCallbackRpcAdapter`
Upewnij się, że ten komponent jawnie implementuje interfejs proxy StreamJsonRpc i odwzorowuje metody zdefiniowane przez protokół ACP, takie jak `ReadTextFileAsync`, do programów obsługi wywołań zwrotnych StreamJsonRpc.

#### Zintegruj rejestrowanie
Przechwytuj i zapisuj nieprzetworzony tekst żądań i odpowiedzi JSON-RPC w potoku przetwarzania komunikatów WebSocket lub Stdio. Użyj `ILogger` aby wyprowadzić surowy ładunek przed analizą i po serializacji, dzięki czemu można łatwiej zdiagnozować problemy z formatowaniem.

```csharp
// Transport wrapper with enhanced logging
public class LoggingAcpTransport : IAcpTransport
{
    private readonly IAcpTransport _innerTransport;
    private readonly ILogger<LoggingAcpTransport> _logger;

    public LoggingAcpTransport(IAcpTransport innerTransport, ILogger<LoggingAcpTransport> logger)
    {
        _innerTransport = innerTransport;
        _logger = logger;
    }

    public async Task SendAsync(string message, CancellationToken cancellationToken = default)
    {
        _logger.LogTrace("Sending message: {Message}", message);
        await _innerTransport.SendAsync(message, cancellationToken);
    }

    public async Task<string> ReceiveAsync(CancellationToken cancellationToken = default)
    {
        var message = await _innerTransport.ReceiveAsync(cancellationToken);
        _logger.LogTrace("Received message: {Message}", message);
        return message;
    }

    public async Task CloseAsync(CancellationToken cancellationToken = default)
    {
        _logger.LogDebug("Closing connection");
        await _innerTransport.CloseAsync(cancellationToken);
    }
}
```

### 4. Refaktoryzacja warstw architektonicznych

#### Warstwa transportowa (`AcpRpcClient`)
Zawiń połączenie StreamJsonRpc i powierz mu odpowiedzialność `InvokeAsync` zarządzanie połączeniami i cyklem życia połączeń.

```csharp
public class AcpRpcClient : IDisposable
{
    private readonly JsonRpc _rpc;
    private readonly IAcpTransport _transport;

    public AcpRpcClient(IAcpTransport transport)
    {
        _transport = transport;
        _rpc = new JsonRpc(new StreamRpcTransport(transport));
        _rpc.StartListening();
    }

    public async Task<TResponse> InvokeAsync<TResponse>(string methodName, object parameters)
    {
        return await _rpc.InvokeAsync<TResponse>(methodName, parameters);
    }

    public void Dispose()
    {
        _rpc.Dispose();
        _transport.Dispose();
    }

    // StreamRpcTransport is the StreamJsonRpc adapter for IAcpTransport
    private class StreamRpcTransport : IDuplexPipe
    {
        // Implement the IDuplexPipe interface
        // ...
    }
}
```

#### Warstwa protokołu (`IAcpAgentClient` / `IAcpAgentCallback`)
Zdefiniuj przejrzyste interfejsy klient-agent i agent-klient. Usuń cykliczny wzór fabryczny `Func<IAcpAgent, IAcpClient>` i zastąp go wstrzykiwaniem zależności lub bezpośrednią rejestracją wywołania zwrotnego.

## Ćwicz

W oparciu o najlepsze praktyki StreamJsonRpc i doświadczenia projektowe, poniżej przedstawiono kluczowe zalecenia dotyczące wdrożenia:

### 1. DTO o silnym typie są lepsze niż obiekty dynamiczne
Podstawową zaletą StreamJsonRpc jest silne pisanie. Nie używać `dynamic` lub `JObject` do przekazywania parametrów. Zamiast tego zdefiniuj jawne klasy C# POCO jako parametry dla każdej metody RPC. To nie tylko rozwiązuje problem rozpoznawania celu proxy, ale także wychwytuje błędy typów w czasie kompilacji.

Przykład: zamień właściwości ogólne w `CallbackProxyTarget` z klasami konkretnymi, takimi jak `ReadTextFileRequest` i `WriteTextFileRequest`.

### 2. Jawnie zadeklaruj nazwy metod
Skorzystaj z `[JsonRpcMethod]` atrybut, aby jawnie określić nazwy metod RPC zamiast polegać na domyślnym mapowaniu nazw metod. Zapobiega to błędom wywołań spowodowanym różnicami w stylu nazewnictwa, takimi jak PascalCase i camelCase.

```csharp
public interface IAcpAgentCallback
{
    [JsonRpcMethod("readTextFile")]
    Task<ReadTextFileResponse> ReadTextFileAsync(ReadTextFileRequest request);
    
    [JsonRpcMethod("writeTextFile")]
    Task WriteTextFileAsync(WriteTextFileRequest request);
}
```

### 3. Skorzystaj z wywołań zwrotnych stanu połączenia
StreamJsonRpc zapewnia `JsonRpc.ConnectionLost` wydarzenie. Należy bezwzględnie nasłuchiwać tego zdarzenia, aby obsłużyć nieoczekiwane wyjścia procesów lub rozłączenia sieci, co jest bardziej aktualne niż poleganie wyłącznie na wykrywaniu awarii Orleans Grain.

```csharp
_rpc.ConnectionLost += (sender, e) =>
{
    _logger.LogError("RPC connection lost: {Reason}", e.ToString());
    // Handle reconnection logic or notify the user
};
```

### 4. Rejestruj dzienniki według warstw
- **Poziom śledzenia**: zapisz pełny surowy ładunek żądania/odpowiedzi JSON.
- **Poziom debugowania**: Rejestruj stosy wywołań metod i podsumowania parametrów.
- **Uwaga**: Upewnij się, że dzienniki nie zawierają wrażliwych tokenów autoryzacji ani treści dużych plików zakodowanych w formacie Base64.

### 5. Radź sobie ze szczególnym charakterem transportu strumieniowego
StreamJsonRpc natywnie obsługuje `IAsyncEnumerable`. Implementując odpowiedzi na monity dotyczące przesyłania strumieniowego dla ACP, użyj `IAsyncEnumerable` bezpośrednio zamiast niestandardowej logiki stronicowania. Może to znacznie uprościć ilość kodu potrzebnego do przetwarzania strumieniowego.

```csharp
public interface IAcpAgentCallback
{
    [JsonRpcMethod("streamText")]
    IAsyncEnumerable<string> StreamTextAsync(StreamTextRequest request);
}
```

### 6. Wzór adaptera
Zachowaj `ACPSession` i `ClientSideConnection` oddzielne. `ACPSession` powinien skupiać się na zarządzaniu stanem w Orleanie i logice biznesowej, takiej jak kolejkowanie wiadomości, i powinien używać obiektu połączenia StreamJsonRpc poprzez kompozycję, a nie dziedziczenie.

## Wniosek

Dzięki kompleksowej integracji StreamJsonRpc projekt HagiCode z powodzeniem uporał się z wysokimi kosztami utrzymania, ograniczeniami funkcjonalnymi i zamieszaniem związanym z warstwami architektonicznymi w oryginalnej niestandardowej implementacji. Najważniejsze ulepszenia obejmują:

1. Zastąpienie właściwości dynamicznych silnie typowanymi DTO, poprawiając łatwość konserwacji i niezawodność
2. Implementacja abstrakcji warstwy transportowej i separacji warstwy protokołu, poprawiająca przejrzystość architektury
3. Wzmocnienie możliwości rejestrowania, aby ułatwić diagnozowanie problemów komunikacyjnych
4. Przedstawiamy obsługę przesyłania strumieniowego w celu uproszczenia implementacji przesyłania strumieniowego

Te ulepszenia zapewniają HagiCode bardziej stabilną i wydajniejszą podstawę komunikacji, umożliwiając lepszą interakcję z zewnętrznymi narzędziami AI i kładąc solidny fundament pod przyszłą rozbudowę funkcji.
