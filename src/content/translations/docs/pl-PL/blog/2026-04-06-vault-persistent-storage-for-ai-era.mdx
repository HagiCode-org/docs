## Jak odtwarzać projekty w erze sztucznej inteligencji: Vault, system trwałego przechowywania danych między projektami

> Jak w dobie rozwoju wspomaganego sztuczną inteligencją możemy pomóc asystentom AI lepiej zrozumieć nasze zasoby edukacyjne? W ramach projektu HagiCode zbudowano system Vault jako ujednoliconą warstwę abstrakcji przechowywania wiedzy, zrozumiałą dla sztucznej inteligencji, znacznie poprawiającą efektywność uczenia się poprzez reprodukcję projektów.

## Tło

W erze sztucznej inteligencji sposób, w jaki programiści uczą się nowych technologii i architektur, ulega głębokim zmianom. „Odtwarzanie projektów” — czyli głębokie studiowanie i uczenie się na podstawie kodu, architektury i wzorców projektowych doskonałych projektów open source — stało się skutecznym sposobem nauki. W porównaniu z tradycyjnymi metodami, takimi jak czytanie książek lub oglądanie filmów, bezpośrednie czytanie i uruchamianie wysokiej jakości projektów open source pomaga znacznie szybciej zrozumieć rzeczywiste praktyki inżynieryjne.

Jednak ta metoda uczenia się wiąże się z wieloma wyzwaniami.

Materiały do nauki są zbyt rozproszone. Twoje notatki mogą znajdować się w Obsidianie, repozytoria kodu mogą być rozproszone w różnych folderach, a historia rozmów Twojego asystenta AI stanie się kolejną odizolowaną wyspą danych. Jeśli chcesz, aby sztuczna inteligencja pomogła w analizie projektu, musisz ręcznie skopiować fragmenty kodu i uporządkować kontekst, co jest dość żmudne.

Jeszcze bardziej kłopotliwy jest zepsuty kontekst. Asystenci AI nie mają bezpośredniego dostępu do lokalnych zasobów edukacyjnych, dlatego w każdej rozmowie musisz ponownie podawać podstawowe informacje. Co więcej, powielone repozytoria kodu aktualizują się szybko, ręczna synchronizacja jest podatna na błędy, a dzielenie się wiedzą w wielu projektach edukacyjnych jest trudne.

U podstaw wszystkich tych problemów leżą „wyspy danych”. Gdyby istniała ujednolicona warstwa abstrakcji pamięci, która umożliwiłaby asystentom AI zrozumienie wszystkich zasobów edukacyjnych i dostęp do nich, problem zostałby rozwiązany.

HagiCode to projekt asystenta kodowania AI i w naszej codziennej pracy programistycznej często musimy studiować i odwoływać się do wielu różnych projektów open source. Aby pomóc asystentom AI lepiej zrozumieć te zasoby edukacyjne, zaprojektowaliśmy Vault, system trwałego przechowywania obejmujący wiele projektów.

To rozwiązanie zostało już sprawdzone w HagiCode w rzeczywistym użyciu. Jeśli stoisz przed podobnymi wyzwaniami w zakresie zarządzania wiedzą, mam nadzieję, że te doświadczenia mogą stanowić inspirację. W końcu, jeśli sam wpadniesz w kilka dołów, powinieneś zostawić coś po sobie następnej osobie.

## Filozofia projektowania systemu Vault

Podstawowa idea systemu Vault jest prosta: utwórz ujednoliconą warstwę abstrakcji magazynu wiedzy zrozumiałą dla sztucznej inteligencji. Z punktu widzenia wdrożenia system ma kilka kluczowych cech.

### Obsługa wielu typów

System obsługuje cztery typy skarbców, każdy odpowiadający innemu scenariuszowi użytkowania:

```typescript
// folder: general-purpose folder type
export const DEFAULT_VAULT_TYPE = 'folder';

// coderef: a type specifically for reproduced code projects
export const CODEREF_VAULT_TYPE = 'coderef';

// obsidian: integrated with Obsidian note-taking software
export const OBSIDIAN_VAULT_TYPE = 'obsidian';

// system-managed: vault automatically managed by the system
export const SYSTEM_MANAGED_VAULT_TYPE = 'system-managed';
```

Wśród nich typ **coderef** jest najczęściej używany w HagiCode. Został specjalnie zaprojektowany do projektów powielanego kodu, zapewniając ujednoliconą strukturę katalogów i opisy metadanych czytelne dla AI.

### Trwały mechanizm przechowywania

Rejestr Vault jest przechowywany trwale w formacie JSON, co gwarantuje, że konfiguracja pozostanie dostępna po ponownym uruchomieniu aplikacji:

```csharp
public class VaultRegistryStore : IVaultRegistryStore
{
    private readonly string _registryFilePath;

    public VaultRegistryStore(IConfiguration configuration, ILogger<VaultRegistryStore> logger)
    {
        var dataDir = configuration["DataDir"] ?? "./data";
        var absoluteDataDir = Path.IsPathRooted(dataDir)
            ? dataDir
            : Path.GetFullPath(Path.Combine(Directory.GetCurrentDirectory(), dataDir));

        _registryFilePath = Path.Combine(absoluteDataDir, "personal-data", "vaults", "registry.json");
    }
}
```

Zaletą tej konstrukcji jest to, że jest prosta i niezawodna. JSON jest czytelny dla człowieka, co ułatwia debugowanie i ręczną edycję; przechowywanie w systemie plików pozwala uniknąć złożoności bazy danych i zmniejsza zależności systemowe. W końcu czasami najprostsza opcja jest naprawdę najlepsza.

### Integracja kontekstu AI

Co najważniejsze, system może automatycznie wstrzykiwać informacje o skarbcu w kontekście propozycji AI:

```typescript
export function buildTargetVaultsText(
  vaults: VaultForText[],
  template: VaultPromptTemplate = DEFAULT_VAULT_PROMPT_TEMPLATE,
): string {
  const readOnlyVaults = vaults.filter((vault) => vault.accessType === 'read');
  const editableVaults = vaults.filter((vault) => vault.accessType === 'write');

  if (readOnlyVaults.length === 0 && editableVaults.length === 0) {
    return '';
  }

  const sections = [
    buildVaultSection(readOnlyVaults, template.reference),
    buildVaultSection(editableVaults, template.editable),
  ].filter(Boolean);

  return `\n\n### ${template.heading}\n\n${sections.join('\n')}`;
}
```

Zapewnia to ważną możliwość: asystenci AI mogą automatycznie zrozumieć dostępne zasoby edukacyjne bez konieczności ręcznego podawania kontekstu przez użytkowników. Można powiedzieć, że liczy się to jako rodzaj milczącego zrozumienia.

## Standaryzowana struktura CodeRef Vault

Dla skarbca typu coderef HagiCode zapewnia standardową strukturę katalogów:

```
my-coderef-vault/
├── index.yaml          # vault metadata description
├── AGENTS.md           # operating guide for AI assistants
├── docs/               # stores study notes and documents
└── repos/              # manages reproduced code repositories through Git submodules
```

Podczas tworzenia skarbca system automatycznie inicjuje tę strukturę:

```csharp
private async Task EnsureCodeRefStructureAsync(
    string vaultName,
    string physicalPath,
    ICollection<VaultBootstrapDiagnosticDto> diagnostics,
    CancellationToken cancellationToken)
{
    Directory.CreateDirectory(physicalPath);

    var indexPath = Path.Combine(physicalPath, CodeRefIndexFileName);
    var docsPath = Path.Combine(physicalPath, CodeRefDocsDirectoryName);
    var reposPath = Path.Combine(physicalPath, CodeRefReposDirectoryName);

    // Create the standard directory structure
    if (!Directory.Exists(docsPath))
    {
        Directory.CreateDirectory(docsPath);
    }

    if (!Directory.Exists(reposPath))
    {
        Directory.CreateDirectory(reposPath);
    }

    // Create the AGENTS.md guide
    await EnsureCodeRefAgentsDocumentAsync(physicalPath, cancellationToken);

    // Create the index.yaml metadata
    await WriteCodeRefIndexDocumentAsync(indexPath, mergedDocument, cancellationToken);
}
```

Ta konstrukcja jest również starannie zaprojektowana:

- **docs/** przechowuje notatki z nauki, w których możesz zapisać zrozumienie kodu, analizę architektury, wyciągnięte wnioski itd. w Markdown
- **repos/** zarządza reprodukowanymi repozytoriami poprzez podmoduły Git zamiast bezpośrednio kopiować kod, co zapewnia synchronizację kodu i oszczędza miejsce
- **index.yaml** zawiera metadane skarbca, dzięki czemu asystenci AI mogą szybko zrozumieć cel i zawartość skarbca
- **AGENTS.md** to przewodnik napisany specjalnie dla asystentów AI, wyjaśniający, jak postępować z zawartością skarbca

Być może zorganizowana w ten sposób sztuczna inteligencja będzie w stanie łatwiej zrozumieć, co masz na myśli.

## Automatyczna inicjalizacja skarbców zarządzanych przez system

Oprócz ręcznie tworzonych skarbców, HagiCode obsługuje także skarbce zarządzane przez system:

```csharp
public async Task<IReadOnlyList<VaultRegistryEntry>> EnsureAllSystemManagedVaultsAsync(
    CancellationToken cancellationToken = default)
{
    var definitions = GetAllResolvedDefinitions();
    var entries = new List<VaultRegistryEntry>(definitions.Count);

    foreach (var definition in definitions)
    {
        entries.Add(await EnsureResolvedSystemManagedVaultAsync(definition, cancellationToken));
    }

    return entries;
}
```

System automatycznie tworzy i zarządza następującymi skarbcami:

- **hagiprojectdata**: przechowywanie danych projektu używane do zapisywania konfiguracji i stanu projektu
- **dane osobowe**: przechowywanie danych osobowych służące do zapisywania preferencji użytkownika
- **hbsprompt**: biblioteka szablonów podpowiedzi używana do zarządzania często używanymi podpowiedziami AI

Skarbce te są inicjowane automatycznie podczas uruchamiania systemu, więc użytkownicy nie muszą ich konfigurować ręcznie. Niektóre rzeczy po prostu lepiej pozostawić systemowi, zamiast martwić się o nie ludzi.

## Mechanizm kontroli dostępu

Ważną częścią projektu jest kontrola dostępu. System dzieli skarbce na dwa typy dostępu:

```typescript
export interface VaultForText {
  id: string;
  name: string;
  type: string;
  physicalPath: string;
  accessType: 'read' | 'write';  // Key: distinguish read-only from editable
}
```

- **odniesienie (tylko do odczytu)**: sztuczna inteligencja służy wyłącznie do analizy i zrozumienia i nie może modyfikować treści. Nadaje się do referencyjnych projektów open source, dokumentów i podobnych materiałów
- **edytowalne (edytowalne)**: sztuczna inteligencja może modyfikować treść zgodnie z potrzebami zadania. Nadaje się do notatek, szkiców i podobnych materiałów

To rozróżnienie ma znaczenie. Informuje sztuczną inteligencję, która treść jest „odniesieniem tylko do odczytu”, a którą „można bezpiecznie edytować”, zmniejszając ryzyko przypadkowych zmian. W końcu nikt nie chce, aby jego ciężka praca zniknęła z powodu niezamierzonej edycji.

## W praktyce: tworzenie i używanie Vault

Skoro omówiliśmy już te pomysły, przyjrzyjmy się, jak to działa w praktyce.

### Utwórz magazyn CodeRef

Oto kompletny przykład wywołania frontendowego:

```typescript
const createCodeRefVault = async () => {
  const response = await VaultService.postApiVaults({
    requestBody: {
      name: "React Learning Vault",
      type: "coderef",
      physicalPath: "/Users/developer/vaults/react-learning",
      gitUrl: "https://github.com/facebook/react.git"
    }
  });

  // The system will automatically:
  // 1. Clone the React repository into vault/repos/react
  // 2. Create the docs/ directory for notes
  // 3. Generate the index.yaml metadata
  // 4. Create the AGENTS.md guide file

  return response;
};
```

To wywołanie API kończy serię działań: utworzenie struktury katalogów, inicjalizację podmodułów Git, wygenerowanie plików metadanych i wiele więcej. Wystarczy podać podstawowe informacje, a resztą zajmie się system. Szczerze mówiąc, jest to dość bezproblemowe podejście.

### Użyj Vault w propozycji AI

Po utworzeniu skarbca możesz odwoływać się do niego w propozycji AI:

```typescript
const proposal = composeProposalChiefComplaint({
  chiefComplaint: "Help me analyze React's concurrent rendering mechanism",
  repositories: [
    { id: "react", gitUrl: "https://github.com/facebook/react.git" }
  ],
  vaults: [
    {
      id: "react-learning",
      name: "React Learning Vault",
      type: "coderef",
      physicalPath: "/vaults/react-learning",
      accessType: "read"  // AI can only read, not modify
    }
  ],
  quickRequestText: "Focus on the Fiber architecture and scheduler implementation"
});
```

System automatycznie wprowadza informacje o skarbcu do kontekstu sztucznej inteligencji, informując sztuczną inteligencję, jakie zasoby edukacyjne są dostępne. Kiedy sztuczna inteligencja potrafi zrozumieć, co masz na myśli, trudno o taki rodzaj milczącego zrozumienia.

## Najlepsze praktyki i rzeczy, na które warto zwrócić uwagę

Korzystając z systemu Vault, podsumowaliśmy kilka wyciągniętych wniosków.

### Bezpieczeństwo ścieżki

System ściśle sprawdza ścieżki, aby zapobiec atakom polegającym na przechodzeniu ścieżki:

```csharp
private static string ResolveFilePath(string vaultRoot, string relativePath)
{
    var rootPath = EnsureTrailingSeparator(Path.GetFullPath(vaultRoot));
    var combinedPath = Path.GetFullPath(Path.Combine(rootPath, relativePath));
    if (!combinedPath.StartsWith(rootPath, StringComparison.OrdinalIgnoreCase))
    {
        throw new BusinessException(VaultRelativePathTraversalCode,
            "Vault file paths must stay inside the registered vault root.");
    }
    return combinedPath;
}
```

To jest ważne. Jeśli dostosujesz ścieżkę skarbca, upewnij się, że mieści się ona w dozwolonym zakresie, w przeciwnym razie system odrzuci operację. Naprawdę nie można przeceniać bezpieczeństwa.

### Zarządzanie submodułami Git

CodeRef Vault zaleca podmoduły Git zamiast bezpośredniego kopiowania kodu:

```csharp
private static string BuildCodeRefAgentsContent()
{
    return """
    # CodeRef Vault Guide

    Repositories under `repos/` should be maintained through Git submodules
    rather than copied directly into the vault root.

    Keep this structure stable so assistants and tools can understand the vault quickly.
    """ + Environment.NewLine;
}
```

Daje to kilka korzyści: synchronizację kodu z wcześniejszym kodem, oszczędność miejsca na dysku i ułatwienie zarządzania wieloma wersjami kodu. W końcu kto chce ciągle pobierać to samo?

### Limity podglądu plików

Aby zapobiec problemom z wydajnością, system ogranicza rozmiar i typ pliku:

```csharp
private const int FileEnumerationLimit = 500;
private const int PreviewByteLimit = 256 * 1024;  // 256KB
```

Jeśli repozytorium zawiera dużą liczbę plików lub bardzo duże pliki, może to mieć wpływ na wydajność podglądu. W takim wypadku można rozważyć przetwarzanie plików partiami lub skorzystanie ze specjalistycznych narzędzi wyszukiwania. Czasami, gdy coś staje się zbyt duże, trudniej jest sobie z tym poradzić, a nie łatwiej.

### Informacje diagnostyczne

Podczas tworzenia skarbca system zwraca informacje diagnostyczne pomocne w debugowaniu:

```csharp
List<VaultBootstrapDiagnosticDto> bootstrapDiagnostics = [];

if (IsCodeRefVaultType(normalizedType))
{
    bootstrapDiagnostics = await EnsureCodeRefBootstrapAsync(
        normalizedName,
        normalizedPhysicalPath,
        normalizedGitUrl,
        cancellationToken);
}
```

Jeśli utworzenie nie powiedzie się, możesz sprawdzić informacje diagnostyczne, aby poznać konkretną przyczynę. Kiedy coś pójdzie nie tak, sprawdzenie diagnostyki jest często najbardziej bezpośrednim rozwiązaniem.

## Podsumowanie

Dzięki ujednoliconej warstwie abstrakcji pamięci system Vault rozwiązuje kilka podstawowych problemów związanych z reprodukcją projektów w erze sztucznej inteligencji:

- **Scentralizowane zarządzanie wiedzą**: wszystkie zasoby edukacyjne są gromadzone w jednym miejscu, a nie rozproszone po całym świecie
- **Automatyczne wstrzykiwanie kontekstu AI**: asystenci AI mogą automatycznie zrozumieć dostępne zasoby edukacyjne bez ręcznej konfiguracji kontekstu
- **Ponowne wykorzystanie wiedzy między projektami**: wiedza może być udostępniana i ponownie wykorzystywana w wielu projektach edukacyjnych
- **Standardowa struktura katalogów**: spójny układ katalogów skraca czas uczenia się

Rozwiązanie to zostało już sprawdzone w projekcie HagiCode. Jeśli budujesz także narzędzia związane z rozwojem wspomaganym sztuczną inteligencją lub borykasz się z podobnymi problemami z zarządzaniem wiedzą, mam nadzieję, że te doświadczenia mogą posłużyć jako przydatne odniesienie.

Tak naprawdę wartość rozwiązania technicznego nie polega na tym, jak bardzo jest ono skomplikowane, ale na tym, czy rozwiązuje rzeczywiste problemy. Podstawowa idea systemu Vault jest bardzo prosta: zbuduj ujednoliconą warstwę przechowywania wiedzy zrozumiałą dla sztucznej inteligencji. Jednak to właśnie ta prosta abstrakcja znacznie poprawiła efektywność naszego rozwoju.

Czasami proste podejście jest naprawdę najlepsze. W końcu skomplikowane rzeczy często kryją jeszcze więcej pułapek...
