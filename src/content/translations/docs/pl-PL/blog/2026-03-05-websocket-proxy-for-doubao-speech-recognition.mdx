## Rozwiązywanie problemów związanych z uwierzytelnianiem za pomocą protokołu WebSocket przeglądarki: praktyczny wzorzec serwera proxy do rozpoznawania mowy Doubao

> Interfejs API WebSocket przeglądarki nie obsługuje niestandardowych nagłówków HTTP, co stanowi wyzwanie dla usług rozpoznawania mowy, które wymagają danych uwierzytelniających w nagłówkach. W tym artykule opisano, jak rozwiązaliśmy ten problem w projekcie HagiCode za pomocą wzorca proxy zaplecza oraz jak podejście ewoluowało od eksperymentów na placu zabaw do zastosowań produkcyjnych.


## Tło

Kiedy zaczynaliśmy budować rozpoznawanie mowy dla projektu HagiCode, śmiało wybraliśmy usługę rozpoznawania mowy Doubao firmy ByteDance. Początkowy projekt był prosty: interfejs powinien łączyć się bezpośrednio z usługą WebSocket firmy Doubao. Jak trudne to może być? Po prostu otwórz połączenie i wyślij trochę danych, prawda?

Potem przyszła niespodzianka: interfejs API Doubao wymaga przekazywania informacji uwierzytelniających przez nagłówki HTTP, w tym takich jak `accessToken` i `secretKey`. Od razu stało się to niezręczne, ponieważ interfejs API WebSocket przeglądarki po prostu nie obsługuje ustawiania niestandardowych nagłówków.

Co więc zrobić, gdy przeglądarka nie pozwala na ich wysłanie?

Rozważaliśmy wówczas dwie opcje:

1. Umieść dane uwierzytelniające w parametrach zapytania adresu URL - proste i tępe
2. Dodaj warstwę proxy na backendzie - więcej pracy na pierwszy rzut oka

Pierwsza opcja udostępnia poświadczenia bezpośrednio w kodzie frontonu i magazynie lokalnym. Czy to naprawdę bezpieczne? Nie czułem się z tym komfortowo. Co więcej, niektóre interfejsy API wymagają weryfikacji opartej na nagłówkach, więc takie podejście nie jest nawet wykonalne.

Ostatecznie wybraliśmy drugą opcję: zaimplementuj proxy WebSocket na backendzie. Przypadkowo ten wzór został po raz pierwszy sprawdzony na naszym placu zabaw i dopiero po potwierdzeniu jego stabilności wprowadziliśmy go do produkcji. W końcu nikt nie chce, aby produkcja pełniła funkcję eksperymentu laboratoryjnego.

## Analiza wyzwań technicznych

### Ograniczenia protokołu WebSocket przeglądarki

Standardowy interfejs API WebSocket wygląda cudownie prosto:

```javascript
const ws = new WebSocket('wss://example.com/ws');
```

Ale właśnie w tej prostocie leży problem — przekazuje ona jedynie parametry w adresie URL i nie może ustawić nagłówków w sposób, w jaki może to zrobić żądanie HTTP:

```javascript
// This is not supported in the WebSocket API
const ws = new WebSocket('wss://example.com/ws', {
  headers: {
    'Authorization': 'Bearer token'
  }
});
```

I to jest kluczowa kwestia. W przypadku usług takich jak rozpoznawanie mowy Doubao, które zależą od uwierzytelniania opartego na nagłówkach, to ograniczenie stanowi twardą blokadę.

Kiedy zaakceptujesz to ograniczenie, architektura musi się zmienić.

### Decyzje projektowe architektoniczne

Projektując rozwiązanie, dokładnie porównaliśmy kompromisy.

**Decyzja 1: Wybór wzorca proxy**

Porównaliśmy dwa podejścia:

| Opcja | Plusy | Wady | Decyzja |
|------|------|------|------|
| Natywny protokół sieciowy | Lekkie, proste, bezpośrednie przekazywanie | Zarządzanie połączeniami musi być obsługiwane ręcznie | Wybrany |
| SygnałR | Automatyczne ponowne połączenie, silne pisanie | Zbyt skomplikowane, dodatkowe zależności | Odrzucony |

Ostatecznie wybraliśmy natywny protokół WebSocket. Szczerze mówiąc, była to najlżejsza opcja i lepiej pasowała do prostego dwukierunkowego przesyłania strumienia binarnego. Wyciągnięcie sygnału sygnalizującego w tym przypadku użycia wydawałoby się przesadą i mogłoby spowodować dodatkowe opóźnienie.

**Decyzja 2: Strategia zarządzania połączeniami**

Przyjęliśmy model „jedno połączenie, jedna sesja” — każde połączenie frontonu WebSocket jest mapowane na własne, niezależne połączenie zaplecza Doubao.

Korzyści są proste:
- Proste do wdrożenia i zgodne z powszechnym wzorcem użytkowania
- Łatwiejsze debugowanie i rozwiązywanie problemów
- Dobra izolacja zasobów, zapobiegająca zakłóceniom między sesjami

Krótko mówiąc, bezpośrednie rozwiązanie jest czasami najlepsze. Złożoność nie czyni automatycznie projektu lepszym.

**Decyzja 3: Przechowywanie danych uwierzytelniających**

Poświadczenia są przechowywane w plikach konfiguracyjnych zaplecza (`appsettings.yml` lub zmienne środowiskowe) i ładowane poprzez wstrzykiwanie zależności:

- Prosty model konfiguracji, który pasuje do istniejących konwencji backendu
- Wrażliwe dane nigdy nie docierają do interfejsu użytkownika
- Obsługuje konfigurację wielośrodowiskową na potrzeby programowania, testowania i produkcji

Ten poziom separacji ma znaczenie. Nikt nie chce, aby referencje wisiały w miejscach, w których nie powinny się znajdować.

### Projekt przepływu danych

Ogólny przepływ danych wygląda następująco:

```
Frontend (browser)
  │
  │ ws://backend/api/voice/ws
  │ WebSocket (binary)
  ▼
Backend (proxy)
  │
  │ wss://openspeech.bytedance.com/
  │ (with authentication headers)
  ▼
Doubao API
```

Sam przepływ nie jest skomplikowany:
1. Frontend łączy się z serwerem proxy zaplecza za pośrednictwem protokołu WebSocket
2. Serwer proxy zaplecza odbiera dane audio i łączy się z interfejsem API Doubao za pomocą uwierzytelnionych nagłówków
3. API Doubao zwraca wyniki rozpoznania, a proxy przekazuje je do frontendu
4. Cały proces pozostaje w pełni asynchroniczny przy dwukierunkowym przesyłaniu strumieniowym

Po wyraźnym podziale obowiązków projekt staje się całkiem naturalny.

## Implementacja podstawowego komponentu

### 1. Konfiguracja punktu końcowego WebSocket

```csharp
app.Map("/ws", async context =>
{
    if (context.WebSockets.IsWebSocketRequest)
    {
        // Read configuration from query parameters
        var appId = context.Request.Query["appId"];
        var accessToken = context.Request.Query["accessToken"];

        // Validate required parameters
        if (string.IsNullOrEmpty(appId) || string.IsNullOrEmpty(accessToken))
        {
            context.Response.StatusCode = 400;
            return;
        }

        // Accept the WebSocket connection
        using var webSocket = await context.WebSockets.AcceptWebSocketAsync();

        // Message handling loop
        var buffer = new byte[4096];
        while (!webSocket.CloseStatus.HasValue)
        {
            var result = await webSocket.ReceiveAsync(buffer, CancellationToken.None);

            if (result.MessageType == WebSocketMessageType.Close)
            {
                await webSocket.CloseAsync(
                    result.CloseStatus.Value,
                    result.CloseStatusDescription,
                    CancellationToken.None);
                break;
            }

            // Process audio data
            await HandleAudioDataAsync(buffer, result.Count);
        }
    }
});
```

### 2. Zarządzanie sesją

```csharp
public class DoubaoSessionManager : IDoubaoSessionManager
{
    private readonly ConcurrentDictionary<string, DoubaoSession> _sessions = new();

    public DoubaoSession CreateSession(string connectionId)
    {
        var session = new DoubaoSession(connectionId);
        _sessions[connectionId] = session;
        return session;
    }

    public async Task SendAudioAsync(string connectionId, byte[] audioData)
    {
        if (_sessions.TryGetValue(connectionId, out var session))
        {
            await session.SendAudioAsync(audioData);
        }
    }

    public void RemoveSession(string connectionId)
    {
        if (_sessions.TryRemove(connectionId, out var session))
        {
            session.Dispose();
        }
    }
}
```

Używanie `ConcurrentDictionary` do zarządzania sesjami oznacza, że bezpieczeństwo wątków jest w dużej mierze obsługiwane przez nas. Każde połączenie przychodzące otrzymuje własną sesję, a czyszczenie odbywa się automatycznie po rozłączeniu.

## Integracja frontendowa w praktyce

### Połączenie WebSocket

```javascript
class DoubaoVoiceClient {
    constructor(config) {
        this.config = config;
        this.ws = null;
    }

    async connect() {
        const url = new URL(this.config.wsUrl);
        // Add query parameters
        Object.entries(this.config.params).forEach(([key, value]) => {
            url.searchParams.set(key, value);
        });

        this.ws = new WebSocket(url);

        return new Promise((resolve, reject) => {
            this.ws.onopen = () => {
                console.log('[DoubaoVoice] Connected');
                resolve();
            };

            this.ws.onmessage = (event) => {
                this._handleMessage(JSON.parse(event.data));
            };

            this.ws.onerror = reject;
        });
    }

    _handleMessage(message) {
        switch (message.type) {
            case 'status':
                this._handleStatus(message.payload);
                break;
            case 'result':
                this.onResult?.(message.payload);
                break;
            case 'error':
                console.error('[DoubaoVoice] Error:', message.payload);
                break;
        }
    }
}

// Usage example
const client = new DoubaoVoiceClient({
    wsUrl: 'ws://localhost:5000/ws',
    params: {
        appId: 'your-app-id',
        accessToken: 'your-access-token',
        sampleRate: 16000,
        bitsPerSample: 16,
        channels: 1
    }
});
```

### Przechwytywanie i transmisja dźwięku

Używanie `AudioWorkletNode` do przetwarzania dźwięku zapewnia lepszą wydajność:

```javascript
// audio-worklet.js
class AudioProcessorWorklet extends AudioWorkletProcessor {
    process(inputs, outputs, parameters) {
        const input = inputs[0]?.[0];
        if (!input) return true;

        // Convert to 16-bit PCM
        const pcm = new Int16Array(input.length);
        for (let i = 0; i < input.length; i++) {
            pcm[i] = Math.max(-32768, Math.min(32767, input[i] * 32767));
        }

        this.port.postMessage({
            type: 'audioData',
            data: pcm.buffer
        }, [pcm.buffer]);

        return true;
    }
}

registerProcessor('audio-processor', AudioProcessorWorklet);

// Main thread code
async function startAudioRecording() {
    const stream = await navigator.mediaDevices.getUserMedia({
        audio: {
            echoCancellation: true,
            noiseSuppression: true,
            autoGainControl: true,
            sampleRate: 48000
        }
    });

    const audioContext = new AudioContext();
    const audioSource = audioContext.createMediaStreamSource(stream);

    await audioContext.audioWorklet.addModule('/audio-worklet.js');
    const audioWorkletNode = new AudioWorkletNode(audioContext, 'audio-processor');

    audioWorkletNode.port.onmessage = (event) => {
        if (event.data.type === 'audioData' && ws?.readyState === WebSocket.OPEN) {
            ws.send(event.data.data); // Send binary data directly
        }
    };

    audioSource.connect(audioWorkletNode);
}
```

`AudioWorklet` działa znacznie lepiej niż `ScriptProcessorNode` i pozwala uniknąć problemów z zacinaniem się dźwięku, które często wprowadzają starsze ścieżki przetwarzania.

## Uwagi i najlepsze praktyki

### 1. Monitorowanie połączenia

- Regularnie rejestruj stan sesji, aby prześledzić pełny cykl życia połączenia
- Monitoruj liczbę i czas trwania segmentów audio, aby wykryć nieprawidłowe połączenia
- Rejestruj status połączenia i zachowanie podczas ponownego łączenia dla usługi Doubao

Są to podstawowe praktyki operacyjne, ale mają one realny wpływ na produkcję.

### 2. Obsługa błędów

- Przechwytuj i rejestruj wszystkie wyjątki protokołu WebSocket
- Użyj `IAsyncDisposable` aby zapewnić oczyszczenie zasobów
- Zaimplementuj płynne zamykanie połączeń i obsługę przekroczeń limitu czasu

Krótko mówiąc, preferuj solidność.

### 3. Wymagania dotyczące formatu audio

- Częstotliwość próbkowania: 16000 Hz (zalecana) lub 8000 Hz
- Głębia bitowa: 16-bitowa
- Kanały: mono
- Kodowanie: PCM (surowe)

Jeśli format jest nieprawidłowy, rozpoznawanie może nie zostać rozpoznane lub jakość może ulec znacznemu pogorszeniu. Te wymagania mają znaczenie.

### 4. Względy bezpieczeństwa

- Zachowaj poufne poświadczenia tylko w konfiguracji zaplecza
- Egzekwuj limity połączeń, aby zapobiec wyczerpaniu zasobów
- Użyj protokołu HTTPS/WSS w środowisku produkcyjnym

Bezpieczeństwo nigdy nie jest sprawą drugorzędną.

### 5. Optymalizacja wydajności

- Użyj operacji asynchronicznych, aby uniknąć blokowania
- Dostosuj rozmiary buforów według potrzeb (domyślnie: 4096 bajtów)
- Rozważ strategie łączenia połączeń i ponownego wykorzystania

Zastosuj te optymalizacje tam, gdzie mają one sens w kontekście Twojego obciążenia pracą.

## Zalecenia dotyczące wdrożenia

1. **Wdrożenie Dockera**: Spakuj usługę proxy jako kontener, aby ułatwić skalowanie i zarządzanie
2. **Równoważenie obciążenia**: Użyj Nginx lub Envoy jako odwrotnego proxy dla ruchu WebSocket
3. **Kontrola stanu**: wdrażaj kontrole oparte na pulsie, aby monitorować dostępność usług
4. **Agregacja logów**: Wysyłaj logi do scentralizowanego systemu rejestrowania, takiego jak ELK lub Loki

Wdrożenie może być proste lub złożone, w zależności od zespołu i środowiska, dlatego należy odpowiednio dostosować.

## Podsumowanie

Wzorzec proxy WebSocket rozwiązuje podstawowy problem polegający na tym, że interfejs API WebSocket przeglądarki nie obsługuje niestandardowych nagłówków. W projekcie HagiCode wzorzec ten okazał się wykonalny i stabilny po przejściu od walidacji na placu zabaw do wdrożenia produkcyjnego.

Kluczowe wnioski:
- Serwer proxy zaplecza może bezpiecznie przekazywać informacje uwierzytelniające
- Natywny protokół WebSocket jest lekki i wydajny w przypadku prostych scenariuszy
- Model „jedno połączenie, jedna sesja” upraszcza zarówno wdrażanie, jak i debugowanie
- Protokół frontend-backend powinien oddzielać sygnały sterujące od danych audio

Mam nadzieję, że jeśli tworzysz funkcję, która zależy również od uwierzytelniania WebSocket, ten wzorzec będzie przydatnym punktem wyjścia.
