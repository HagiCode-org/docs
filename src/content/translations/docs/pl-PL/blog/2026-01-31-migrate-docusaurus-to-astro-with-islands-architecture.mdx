## Od Docusaurus 3.x do Astro 5.x: Retrospektywa migracji witryny HagiCode

> W tym artykule wspominamy pełną migrację oficjalnej strony HagiCode z Docusaurus 3.x do Astro 5.x. Przyjrzymy się bliżej temu, jak architektura Islands firmy Astro pomogła nam rozwiązać wąskie gardła wydajności, zachowując jednocześnie istniejące zasoby komponentów React, zapewniając poprawę zarówno szybkości kompilacji, jak i wydajności ładowania.

## Tło

W styczniu 2026 r. przeprowadziliśmy „przeszczep serca” na oficjalnej stronie HagiCode, dokonując pełnej migracji jej podstawowego środowiska z Docusaurus 3.x do Astro 5.x. Nie była to impulsywna zmiana, ale starannie przemyślana decyzja techniczna.

Przed migracją nasza witryna była funkcjonalnie kompletna, ale zaczęły pojawiać się w niej klasyczne „luksusowe problemy”: rozdęte artefakty kompilacji, nadmierne obciążenie JavaScriptem i niezbyt idealna prędkość ładowania stron na złożonych stronach dokumentacji. Jako projekt asystenta kodowania AI, HagiCode potrzebuje częstych aktualizacji dokumentacji i funkcji, więc wydajność kompilacji bezpośrednio wpływa na szybkość wydawania. Jednocześnie chcieliśmy, aby strona była bardziej przyjazna dla wyszukiwarek (SEO), aby więcej programistów mogło odkryć projekt.

Aby rozwiązać te problemy, podjęliśmy odważną decyzję: przebudować cały system na Astro. Skutki tej decyzji mogą być jeszcze większe, niż się spodziewasz. Wkrótce przejdę do szczegółów.

## O HagiCodzie

Podejście do migracji witryny przedstawione w tym artykule wynika z naszego praktycznego doświadczenia w zakresie [Kod Hagi](https://github.com/HagiCode-org/site) projekt.

HagiCode to asystent kodowania AI skupiony na poprawie efektywności programowania. Dbamy nie tylko o iterację podstawowych funkcji, ale także o doświadczenie programisty. Ten refaktor witryny miał również zapewnić użytkownikom najszybsze możliwe doświadczenie podczas przeglądania naszych dokumentów i oficjalnej witryny internetowej.

## Po co opuszczać dojrzały ekosystem Docusaurus?

W ekosystemie React Docusaurus od dawna jest „standardową odpowiedzią” dla witryn z dokumentacją. Działa od razu po wyjęciu z pudełka, oferuje bogaty ekosystem wtyczek i ma aktywną społeczność. Jednak w miarę jak HagiCode zyskiwał więcej funkcji, poczuliśmy także jego ograniczenia:

1.  **Wąskie gardła w wydajności**: Docusaurus jest zasadniczo React SPA (aplikacja jednostronicowa). Nawet jeśli piszesz tylko strony statyczne, klient i tak musi załadować środowisko wykonawcze React i nawodnić stronę, co jest niepotrzebnie obciążające w przypadku prostych stron dokumentów.
2.  **Duży rozmiar zasobu**: Nawet jeśli strona zawiera bardzo mało treści, dołączony rozmiar JS pozostaje względnie stały. Nie jest to idealne rozwiązanie dla użytkowników mobilnych lub złych warunków sieciowych.
3.  **Ograniczona elastyczność**: chociaż można ją rozszerzyć, chcieliśmy mieć większą kontrolę niskiego poziomu nad potokiem kompilacji.

Astro przybyło dokładnie w odpowiednim momencie, aby rozwiązać te problemy. Wprowadzono nową „architekturę wysp”: domyślnie Astro generuje statyczny kod HTML z zerowym JavaScriptem, a tylko komponenty wymagające interaktywności są „aktywowane” i ładują JS. Oznacza to, że większość naszej witryny staje się czystym kodem HTML i ładuje się niezwykle szybko.

## Podstawowa strategia migracji: płynna transformacja architektoniczna

Migracja to nie tylko kopiowanie i wklejanie. Wymagało to zmiany sposobu myślenia. Przeszliśmy z modelu „all React” firmy Docusaurus na model „Core + Islands” firmy Astro.

### 1. Przebudowa systemu konfiguracyjnego

Najpierw musieliśmy się ruszyć `docusaurus.config.ts` do `astro.config.mjs`. Nie była to tylko zmiana nazwy pliku, ale przepisanie routingu i logiki kompilacji.

W Docusaurus wszystko jest wtyczką. W Astro wszystko jest integracją. Musieliśmy na nowo zdefiniować ścieżkę podstawową witryny, zbudować tryb wyjściowy (statyczny vs SSR) i strategię optymalizacji zasobów.

**Przed migracją:**
```typescript
// docusaurus.config.ts
export default {
  title: 'HagiCode',
  url: 'https://hagicode.com',
  baseUrl: '/',
  // ... more configuration
};
```

**Po migracji:**
```javascript
// astro.config.mjs
import { defineConfig } from 'astro/config';
import react from '@astrojs/react';

export default defineConfig({
  integrations: [react()],
  site: 'https://hagicode.com',
  base: '/',
  // Optimization settings for static assets
  build: {
    inlineStylesheets: 'auto',
  },
});
```

### 2. Co zachować i co refaktoryzować w komponentach React

To była najbardziej bolesna część migracji. Nasza istniejąca witryna zawierała wiele komponentów React, takich jak zakładki, podświetlanie kodu, przyciski opinii i inne. Wyrzucenie ich byłoby marnotrawstwem, ale trzymanie wszystkiego spowodowałoby, że ładunek JavaScript byłby zbyt duży.

HagiCode przyjął strategię **stopniowego nawodnienia**:
*   **Czysto statyczne komponenty**: W przypadku treści prezentacyjnych, takich jak nagłówki, stopki i dokumentacja w postaci zwykłego tekstu, przepisaliśmy je jako komponenty Astro (`.astro` pliki) i renderowałem je bezpośrednio do formatu HTML w czasie kompilacji.
*   **Wyspy interaktywne**: W przypadku komponentów, które muszą pozostać interaktywne, takich jak przełączniki motywów, przełączanie kart i przyciski kopiowania bloków kodu, zachowaliśmy implementację React i dodaliśmy `client:load` lub `client:visible` dyrektywy.

Na przykład nasz powszechnie używany komponent Tabs w dokumentacji:

```jsx
// src/components/Tabs.jsx
import { useState } from 'react';
import './Tabs.css'; // Import styles

export default function Tabs({ items }) {
  const [activeIndex, setActiveIndex] = useState(0);
  // ... state logic
  return (
    <div className="tabs-wrapper">
      {/* Rendering logic */}
    </div>
  );
}
```

W przypadku użycia w Markdown wyraźnie mówimy Astro: „Ten komponent potrzebuje JS”.

```markdown
// src/content/docs/example.mdx
import Tabs from '../../components/Tabs.jsx';

<!-- Load JS only when the component enters the viewport -->
<Tabs client:visible items={...} />
```

W ten sposób interaktywne komponenty znajdujące się poza rzutnią nie konkurują o przepustowość, co znacznie poprawia prędkość ładowania pierwszego ekranu.

### 3. Dostosowanie systemu stylizacji: od modułów CSS do CSS o określonym zakresie

Docusaurus domyślnie obsługuje moduły CSS, podczas gdy Astro zachęca do korzystania z CSS o określonym zakresie poprzez `<style>` znacznik. Podstawową ideą obu jest izolacja stylu, ale składnia jest inna.

Podczas migracji HagiCode przekonwertowaliśmy najbardziej złożone moduły CSS na style Astro. Okazało się to naprawdę dobrą rzeczą, ponieważ w `.astro` pliki style i szablony znajdują się w tym samym pliku, co sprawia, że obsługa jest bardziej intuicyjna.

**Przed refaktoryzacją:**
```css
/* Tabs.module.css */
.wrapper { background: var(--ifm-background-color); }
```

**Po refaktoryzacji (w zakresie astronomicznym):**
```html
<!-- Tabs.astro -->
<div class="tabs-wrapper">
  <slot />
</div>

<style>
  .tabs-wrapper {
    /* Use CSS variables directly to adapt to the theme */
    background: var(--bg-color);
    padding: 1rem;
  }
</style>
```

Jednocześnie ujednoliciliśmy globalny system zmiennych CSS i wykorzystaliśmy możliwości Astro uwzględniające środowisko, aby zapewnić płynne przełączanie trybu ciemnego na stronach.

## Pułapki, na które natrafiamy w praktyce i jak je rozwiązaliśmy

Podczas migracji HagiCode napotkaliśmy sporo problemów. Oto kilka najbardziej typowych.

### 1. Zmienne problemy związane ze ścieżką i środowiskiem

HagiCode obsługuje wdrażanie podścieżek, na przykład wdrażanie w podkatalogu GitHub Pages. W Docusaurusie, `baseUrl` jest obsługiwany automatycznie. W Astro musimy jednak zachować większą ostrożność podczas obsługi łączy do obrazów i żądań API.

Aby konsekwentnie zarządzać tym, wprowadziliśmy mechanizm zmiennych środowiskowych:

```javascript
// Handle paths in the build script
const getBasePath = () => import.meta.env.VITE_SITE_BASE || '/';
```

Pamiętaj, aby nie kodować ścieżek zaczynających się od `/` w swoim kodzie. W fazie rozwoju a produkcji lub po skonfigurowaniu `base` path, może to spowodować błąd 404 dla zasobów.

### 2. Zgodność ze skryptem CommonJS

Nasza stara witryna zawierała kilka skryptów Node.js używanych do takich zadań, jak automatyczne pobieranie danych metryk i aktualizowanie mapy witryny, i zostały one napisane w języku CommonJS (`require`). Astro i nowoczesne narzędzia do kompilacji w pełni uwzględniają moduły ES (`import`/`export`).

Jeśli masz również podobne skrypty, pamiętaj o refaktoryzacji ich wszystkich do modułów ES. To jest kierunek, w którym zmierza ekosystem, a im szybciej dokonasz zmiany, tym mniej problemów będziesz miał później.

```javascript
// Old way
const fs = require('fs');

// New way
import fs from 'fs';
```

### 3. Nie zapomnij o SEO i przekierowaniach

Wyszukiwarki już zaindeksowały stare strony Docusaurus HagiCode. Jeśli przejdziesz bezpośrednio na Astro i zmieni się struktura adresu URL, możesz otrzymać dużą liczbę błędów 404 i znaczny spadek rankingu wyszukiwania.

Skonfigurowaliśmy reguły przekierowań w Astro:

```javascript
// astro.config.mjs
export default defineConfig({
  redirects: {
    '/docs/old-path': '/docs/new-path',
    // Map old links to new links in bulk
  }
});
```

Możesz też zająć się tym w warstwie konfiguracji serwera. Upewnij się, że stare linki można przekierować 301 na nowe adresy, ponieważ ma to kluczowe znaczenie dla SEO.

## Podsumowanie

Dla HagiCode migracja z Docusaurus do Astro nie była tylko aktualizacją frameworka. Było to także praktyczne wdrożenie filozofii „najpierw wydajność”.

**Co zyskaliśmy:**
*   **Wybitne wyniki Lighthouse**: Po migracji wynik wydajności witryny HagiCode z łatwością osiągnął wynik doskonały.
*   **Większa szybkość kompilacji**: Możliwości przyrostowej kompilacji Astro skracają czas wydawania aktualizacji dokumentacji o połowę.
*   **Zachowana elastyczność**: Dzięki Islands Architecture nie zrezygnowaliśmy z żadnych funkcji interaktywnych i nadal mogliśmy używać React tam, gdzie było to potrzebne.

Jeśli prowadzisz także witrynę zorientowaną na dokumentację i masz problemy z rozmiarem pakietu lub szybkością ładowania, warto wypróbować Astro. Chociaż proces migracji wymaga pewnych operacji, takich jak zmiana nazwy PCode na HagiCode i przenoszenie komponentów jeden po drugim, jedwabiście gładka obsługa, jaką otrzymujesz w zamian, sprawia, że ​​jest to absolutnie tego warte.

System kompilacji udostępniony w tym artykule jest dokładnie tym samym podejściem, które opracowaliśmy metodą prób i błędów podczas tworzenia HagiCode. Jeśli uznasz to podejście za wartościowe, mówi to coś o naszej sile inżynieryjnej, a samemu HagiCode prawdopodobnie warto przyjrzeć się bliżej.
