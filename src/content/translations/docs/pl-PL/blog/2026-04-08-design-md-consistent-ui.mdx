## Design.md: rozwiązanie do spójnego projektowania interfejsu użytkownika opartego na sztucznej inteligencji

> Jak w dobie rozwoju frontendu wspomaganego sztuczną inteligencją zapewnić spójność interfejsów użytkownika generowanych przez sztuczną inteligencję? W tym artykule dzielimy się naszym praktycznym doświadczeniem w tworzeniu witryny galerii projektów w oparciu o Awesome-design-md, a także opisujemy, jak stworzyć ustrukturyzowaną witrynę `design.md` aby poprowadzić sztuczną inteligencję w stronę ustandaryzowanego projektu interfejsu użytkownika.

## Tło

Każdy, kto używał sztucznej inteligencji do pisania kodu frontendowego, prawdopodobnie miał podobne doświadczenia: poproś sztuczną inteligencję o wygenerowanie tej samej strony kilka razy, a każdy wynik będzie inny. Czasem rogi są zaokrąglone, czasem ostre. Czasami odstępy wynoszą 8 pikseli, a innym razem 16 pikseli. Nawet ten sam przycisk może wyglądać inaczej w różnych rozmowach.

To nie jest odosobniony problem. W miarę jak rozwój wspomagany sztuczną inteligencją staje się coraz bardziej powszechny, **brak spójności w interfejsie użytkownika generowanym przez sztuczną inteligencję** stał się powszechnym problemem. Różni asystenci AI, różne podpowiedzi, a nawet ten sam asystent w różnych rozmowach mogą skutkować radykalnie różnymi projektami interfejsów. Stwarza to ogromne koszty utrzymania podczas iteracji produktu.

Podstawowa przyczyna jest w rzeczywistości prosta: nie ma **miarodajnego dokumentu referencyjnego projektu**. Tradycyjne arkusze stylów CSS mogą informować programistów, „jak coś zaimplementować”, ale nie mogą w pełni przekazać, „dlaczego zostało to zaprojektowane w ten sposób” ani „który wzorzec projektowy powinien zostać zastosowany w jakim scenariuszu”. W przypadku sztucznej inteligencji jasny i uporządkowany opis jest jeszcze ważniejszy dla zrozumienia konwencji projektowych.

Jednocześnie społeczność open source oferuje już doskonałe zasoby. The [VoltAgent/awesome-design-md](https://github.com/VoltAgent/awesome-design-md) projekt gromadzi dokumentację projektową systemu od wielu znanych firm. Każdy katalog zawiera plik `README.md`, za `DESIGN.md`i podgląd HTML. Jednak wszystko to jest rozproszone po całym repozytorium nadrzędnym, co utrudnia szybkie przeglądanie i porównywanie.

Czy zatem możemy skonsolidować te zasoby w łatwą do przeglądania galerię projektów, jednocześnie tworząc ustrukturyzowaną `design.md` do wykorzystania przez sztuczną inteligencję?

Odpowiedź brzmi: tak. Następnie opowiem o naszym podejściu.

## Analiza

Zanim napiszemy kod, przeanalizujmy wyzwania techniczne stojące za tym problemem.

### Zarządzanie treścią źródłową: jak ujednolicić rozproszone zasoby projektowe?

Repozytorium Awesome-design-md zawiera dużą liczbę dokumentów projektowych, ale potrzebowaliśmy sposobu na przeniesienie ich do naszego własnego projektu.

**Rozwiązanie: użyj podmodułu git**

```
awesome-design-md-site
└── vendor/awesome-design-md          # Upstream resources (git submodule)
```

Daje nam to kilka korzyści:
- Kontrola wersji: możemy przypiąć konkretną wersję nadrzędną
- Kompilacje offline: nie ma potrzeby żądania zewnętrznych interfejsów API podczas kompilacji
- Przegląd treści: konkretne zmiany są widoczne w PR

### Normalizacja danych: jak standaryzować różne struktury dokumentów?

Różne firmy w różny sposób konstruują swoje dokumenty projektowe. W niektórych brakuje plików podglądu, a niektóre używają niespójnych nazewnictwa. Musimy je znormalizować podczas procesu kompilacji.

**Rozwiązanie: skanuj i generuj znormalizowane wpisy w czasie kompilacji**

Podstawowym modułem jest `awesomeDesignCatalog.ts`, odpowiedzialny za:
1. Skanowanie `vendor/awesome-design-md/design-md/*` katalog
2. Sprawdzanie, czy każdy wpis zawiera wymagane pliki (`README.md`, `DESIGN.md`i co najmniej jeden plik podglądu)
3. Wyodrębnianie i renderowanie treści Markdown do formatu HTML
4. Generowanie znormalizowanych danych wejściowych

### Statyczna architektura witryny: jak zapewnić dynamiczne wyszukiwanie, pozostając w pełni statycznym?

Ponieważ jest to galeria designu, wyszukiwanie jest koniecznością. Ale Astro to generator stron statycznych, więc jak wdrożyć wyszukiwanie w czasie rzeczywistym?

**Rozwiązanie: React Island + synchronizacja parametrów zapytania URL**

```typescript
// src/components/gallery/SearchToolbar.tsx

export function SearchToolbar() {

  // Sync with the URL
  useEffect(() => {
    const params = new URLSearchParams(window.location.search);

  const filtered = entries.filter(entry =>
    entry.searchText.includes(query)

```

Zaletą tego podejścia jest to, że pozwala na zachowanie możliwości wdrożenia witryny statycznej, co oznacza, że można ją wdrożyć w dowolnej usłudze hostingu statycznego, zapewniając jednocześnie natychmiastowe filtrowanie.

### Dokumentacja projektowa: jak pomóc sztucznej inteligencji zrozumieć i przestrzegać standardów projektowych?

To jest sedno całego rozwiązania. Musimy stworzyć strukturę `design.md` które sztuczna inteligencja może zrozumieć i zastosować.

**Rozwiązanie: zapożycz strukturę ClickHouse `DESIGN.md`**

[Kliknij House'a `DESIGN.md`](https://github.com/ClickHouse/ClickHouse/blob/master/DESIGN.md) to doskonałe odniesienie. Obejmuje:
- Motyw wizualny i atmosfera
- Paleta kolorów i role
- Zasady typografii
- Stylizacje komponentów
- Zasady układu
- Głębokość i wysokość
- Co robić i czego nie robić
- Responsywne zachowanie
- Przewodnik po podpowiedziach agenta

Nasze podejście jest następujące: **wykorzystaj ponownie strukturę, przepisz treść**. Zachowujemy strukturę sekcyjną ClickHouse `DESIGN.md`, ale zastąp zawartość rzeczywistymi tokenami projektu i konwencjami komponentów używanymi w naszym własnym projekcie.

## Rozwiązanie

Z powyższej analizy wynika, że nasze rozwiązanie składa się z czterech podstawowych modułów.

### 1. Potok przetwarzania treści

Stanowi to podstawę całego systemu, odpowiedzialnego za wydobywanie i normalizację treści z zasobów nadrzędnych.

```typescript
// src/lib/content/awesomeDesignCatalog.ts

export async function scanSourceEntries(): Promise<DesignEntry[]> {
  const designDir = 'vendor/awesome-design-md/design-md';
  const entries: DesignEntry[] = [];

  for (const dir of await fs.readdir(designDir)) {
    const entryPath = path.join(designDir, dir);
    if (await isValidDesignEntry(entryPath)) {
      const entry = await normalizeDesignEntry(entryPath);
      entries.push(entry);
    }
  }

  return entries;
}

async function isValidDesignEntry(dir: string): Promise<boolean> {
  const requiredFiles = ['README.md', 'DESIGN.md'];
  for (const file of requiredFiles) {
    if (!(await fileExists(path.join(dir, file)))) {
      return false;
    }
  }
  return true;
}
```

### 2. Interfejs przeglądania galerii

Interfejs galerii składa się z trzech głównych części:

**Strona główna**: wyświetla siatkę kart ze wszystkimi wpisami projektów, a każda karta zawiera:
- Tytuł wpisu dotyczącego projektu i podsumowanie
- Obraz podglądu, jeśli jest dostępny
- Podświetlanie szybkiego wyszukiwania

**Strona szczegółów**: gromadzi pełne informacje dla pojedynczego wpisu projektu:
- `README` dokument
- `DESIGN` dokument
- Podgląd z przełączaniem motywu jasny/ciemny
- Nawigacja do sąsiadujących wpisów

**Nawigacja**: umożliwia powrót do galerii i przeglądanie sąsiadujących wpisów

Galeria na stronie głównej wykorzystuje układ kart o dużej gęstości, spłaszczony `design.md` wpisy z różnych źródeł w ujednoliconą strukturę wizualną, dzięki czemu zespoły mogą szybko porównywać style marki, wzory przycisków i rytm typograficzny.

Po otwarciu konkretnego wpisu strona szczegółów umieszcza podsumowanie projektu i podgląd na żywo na tej samej stronie, redukując koszty przełączania pomiędzy dokumentacją, podglądami i kodem źródłowym.

### 3. Funkcjonalność wyszukiwania

Funkcja wyszukiwania opiera się na filtrowaniu po stronie klienta, a stan jest zachowywany poprzez parametry zapytania adresu URL:

```typescript
// src/components/gallery/SearchToolbar.tsx

function SearchToolbar({ entries }: { entries: DesignEntry[] }) {
  const [query, setQuery] = useState('');
  const [results, setResults] = useState(entries);

  useEffect(() => {
    const params = new URLSearchParams(window.location.search);
    const initialQuery = params.get('q') || '';
    setQuery(initialQuery);
    filterEntries(initialQuery);
  }, []);

  const filterEntries = (searchQuery: string) => {
    const filtered = entries.filter(entry =>
      entry.searchText.toLowerCase().includes(searchQuery.toLowerCase())
    );
    setResults(filtered);
  };

  const handleChange = (e: React.ChangeEvent<HTMLInputElement>) => {
    const value = e.target.value;
    setQuery(value);
    filterEntries(value);

    // Update the URL without triggering a page refresh
    const newUrl = value
      ? `${window.location.pathname}?q=${encodeURIComponent(value)}`
      : window.location.pathname;
    window.history.replaceState({}, '', newUrl);
  };

  return (
    <div className="search-toolbar">
      <input
        type="text"
        value={query}
        onChange={handleChange}
        placeholder="Search design entries..."
      />
      <span className="result-count">{results.length} results</span>
    </div>
  );
}
```

Oprócz surowego `design.md` treści konsumowanych przez sztuczną inteligencję, umieszczamy również zarówno `README` i `DESIGN` dokumenty do tego samego interfejsu do czytania, co ułatwia użytkownikom sprawdzanie, kopiowanie fragmentów i porównywanie ich z wynikami podglądu.

## Ćwicz

Skoro omówiliśmy już rozwiązanie, jak je właściwie wdrożyć?

### Kroki wdrożenia

**Krok 1: Zainicjuj submoduł**

```bash
# Add the upstream repository as a submodule
git submodule add https://github.com/VoltAgent/awesome-design-md.git vendor/awesome-design-md

# Initialize and update the submodule
git submodule update --init --recursive
```

**Krok 2: Utwórz potok treści**

Wdrożyć `awesomeDesignCatalog.ts`, w tym:
- Logika skanowania i sprawdzania poprawności plików
- Renderowanie Markdown przy użyciu wbudowanego modułu renderującego Astro
- Ekstrakcja danych wejściowych

**Krok 3: Zbuduj interfejs galerii**

Użyj wysp Astro + React, aby stworzyć:
- Układ galerii na stronie głównej (siatka kart)
- Elementy karty projektu
- Pasek narzędzi wyszukiwania
- Szczegółowy układ strony

**Krok 4: Napisz dokument projektowy**

Oparty na strukturze ClickHouse `DESIGN.md`, wypełnij rzeczywiste żetony projektu z własnego projektu. Aktualizacja `README.md` i dodaj link do `design.md`.

### Notatki

**Bezpieczeństwo**: Renderowanie Markdown wymaga filtrowania niebezpiecznego kodu HTML. Wbudowane filtry renderujące Astro `script` tagi domyślnie, ale nadal musisz uważać na ryzyko XSS.

**Wydajność**: Duża liczba podglądów elementów iframe może mieć wpływ na wydajność pierwszego malowania. Zaleca się stosowanie `loading="lazy"` aby leniwie ładować zawartość podglądu.

**Łatwość konserwacji**: `design.md` musi być zsynchronizowany z implementacją kodu. Zaleca się dodanie kontroli CI, aby zapewnić spójność zmiennych CSS pomiędzy dokumentacją i kodem.

**Dostępność**: Upewnij się, że kontrast kolorów jest zgodny ze standardem WCAG AA (co najmniej 4,5:1).

### Przewodnik użytkowania AI

Po utworzeniu `design.md`Jak sprawić, by sztuczna inteligencja faktycznie z niej korzystała? Oto kilka praktycznych wskazówek:

**Wskazówka 1: Odwołaj się do tego wyraźnie w monicie**

```
Please refer to the design.md file in the project root and use the design conventions defined there to implement the following components:
- Buttons: use the primary color with an 8px border radius
- Cards: use the elevation-2 shadow level
```

**Wskazówka 2: Wymagaj, aby sztuczna inteligencja odwoływała się do określonych zmiennych CSS**

```
Implement a navigation bar with the following requirements:
- Use --color-bg-primary for the background color
- Use --color-border-subtle for borders
- Use --text-color-primary for text
```

**Wskazówka 3: Uwzględnij `design.md` treść w wierszu poleceń**

Jeśli Twoje narzędzie AI obsługuje niestandardowe monity systemowe, możesz dodać podstawową zawartość `design.md` bezpośrednio do niego.

### Strategia testowania

**Testowanie potoku treści**:
- Scenariusze braku plików (missing `README.md` lub `DESIGN.md`)
- Scenariusze błędów formatowania (niepowodzenie analizy Markdown)
- Scenariusze z pustym katalogiem

**Testowanie funkcji wyszukiwania**:
- Obsługa pustych wyników
- Znaki specjalne, takie jak chiński i emoji
- Weryfikacja synchronizacji adresu URL

**Testowanie komponentów interfejsu użytkownika**:
- Przełączanie motywu jasny/ciemny
- Responsywny układ
- Podgląd stanów ładowania

## Wniosek

Niespójność w interfejsach użytkownika generowanych przez sztuczną inteligencję, którą HagiCode napotkał podczas programowania, była w swej istocie spowodowana brakiem ustrukturyzowanego dokumentu referencyjnego projektu. Budując witrynę galerii projektów i tworząc ustandaryzowany `design.md`, pomyślnie rozwiązaliśmy ten problem.

Podstawową wartością tego rozwiązania jest:
- **Ujednolicone zasoby**: konsolidacja rozproszonej dokumentacji systemu projektowego
- **Standardy strukturalne**: wyrażające konwencje projektowe w formacie zrozumiałym dla sztucznej inteligencji
- **Ciągła konserwacja**: ciągłe aktualizowanie treści `git submodule`

Jeśli wykorzystujesz sztuczną inteligencję również do programowania frontendowego, warto wypróbować to podejście. Tworzenie struktury `design.md` nie tylko poprawia spójność kodu generowanego przez sztuczną inteligencję, ale także pomaga Twojemu zespołowi zachować wewnętrznie ujednolicone standardy projektowe.
