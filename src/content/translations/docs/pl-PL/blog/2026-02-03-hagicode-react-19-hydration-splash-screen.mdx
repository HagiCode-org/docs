## Projektowanie 12 wyjątkowych doświadczeń startowych dla HagiCode: od minimalizmu po cyberpunk

> Krótka przerwa między pobraniem aplikacji React 19 a zakończeniem nawadniania to doskonała okazja dla użytkowników, aby poczuć osobowość Twojej marki. W tym artykule udostępniamy kompletny system stylu startowego, który zbudowaliśmy dla projektu HagiCode przy użyciu HTML/CSS/JS.

## Tło

Jako nowoczesna aplikacja zbudowana przy użyciu ASP.NET Core 10 i React 19 (Vite), HagiCode wykorzystuje architekturę wdrażania rozdzieloną frontendem i backendem. Dane wyjściowe kompilacji frontendu są pakowane w backend `wwwroot/` katalog i hostowany przez ASP.NET Core.

Jednak ta architektura wprowadza klasyczny problem związany z UX: kiedy użytkownicy odwiedzają stronę, przeglądarka najpierw ładuje kod HTML, następnie pobiera duży pakiet JS, a na koniec pozwala Reactowi wykonać Hydration. Podczas tego „okresu próżni”, który trwa od kilkuset milisekund do kilku sekund, użytkownicy widzą albo pusty ekran, albo martwą, statyczną stronę.

Aby wypełnić tę lukę i nadać charakter marce HagiCode, musieliśmy zaprojektować system w stylu start-up, zaimplementowany w całości z wbudowanym kodem `index.html`.

## O HagiCodzie

Podejście do projektowania ekranów powitalnych przedstawione w tym artykule wynika z naszego praktycznego doświadczenia w branży [Kod Hagi](https://github.com/HagiCode-org/site) projekt. Jako asystent kodowania AI, HagiCode dba nie tylko o efektywność generowania kodu, ale także o wrażenia wizualne programisty. Ten system startowy jest jednym z rezultatów naszego dążenia do najwyższej wydajności frontendu.

## Podstawowe wyzwania i projektowanie architektury

Zanim rozpoczęliśmy projektowanie, musieliśmy najpierw wyjaśnić ograniczenia techniczne. Ponieważ wszystko musiało zostać zaimplementowane inline in `index.html`, nie mogliśmy załadować żadnych zewnętrznych plików CSS ani JS innych niż własny pakiet React.

### Analiza ograniczeń technicznych

1.  **Zasada zerowej zależności**: Wszystkie style muszą znajdować się w pliku a `<style>` tag, a cała logika musi znajdować się wewnątrz pliku `<script>` znacznik.
2.  **Defensywny CSS**: Aby zapobiec zanieczyszczaniu ekranu powitalnego stylami globalnymi po zamontowaniu aplikacji React, zdecydowaliśmy się opakować wszystkie style startowe prefiksem identyfikatora o wysokim priorytecie, takim jak `#boot-screen`.
3.  **Najważniejsza wydajność**: Animacje powinny używać CSS `transform` i `opacity` tam, gdzie to możliwe, aby uniknąć ponownego przepływu i upewnić się, że główny wątek pozostaje odblokowany.
4.  **Spójność wizualna**: Kolory i czcionki muszą być zgodne z konfiguracją Tailwind HagiCode.

### Wzór architektury: obudowa i wtryskiwacz

Przyjęliśmy **wzorzec wariantowy**. Podstawowa logika jest zawarta w natychmiast wywoływanym wyrażeniu funkcyjnym (IIFE), podczas gdy specyficzna logika renderowania jest wstrzykiwana poprzez konfigurację. Dzięki temu możemy przełączać się między różnymi stylami poprzez prostą konfigurację, zamiast wielokrotnie przepisywać logikę manipulacji DOM.

Oto podstawowy kod architektury:

```html
<!-- Inline in index.html -->
<div id="boot-root"></div>

<script>
(function() {
  const BootSequence = {
    config: {
      theme: 'terminal', // Configurable as 'minimal', 'skeleton', 'code-rain', etc.
      color: '#3b82f6'   // Brand color
    },
    
    // Core lifecycle
    init() {
      this.render();
      this.listenForMount();
    },

    // Render the currently selected style
    render() {
      const root = document.getElementById('boot-root');
      if (this.variants[this.config.theme]) {
        root.innerHTML = this.variants[this.config.theme].render();
      }
    },

    // Listen for successful React mount and exit gracefully
    listenForMount() {
      window.addEventListener('hagicode:ready', () => {
        const screen = document.getElementById('boot-root');
        // Fade out first, then remove the DOM to avoid flicker
        screen.style.opacity = '0';
        screen.style.transition = 'opacity 0.3s ease';
        setTimeout(() => screen.remove(), 300);
      });
    },

    // The implementation logic for all 12 styles lives here
    variants: {
      // ...see details below
    }
  };

  BootSequence.init();
})();
</script>
```

## Lista kontrolna 12 projektów w stylu startowym

Pogrupowaliśmy te 12 stylów w sześć głównych kategorii, aby zaspokoić różne scenariusze i preferencje estetyczne.

### A. Minimalizm

> „Mniej znaczy więcej”. W przypadku scenariuszy wymagających najwyższej prędkości ładowania zapewniamy najlżejsze możliwe opcje.

#### 1. Minimalistyczna kropka
Na środku ekranu znajduje się prosta kropka połączona z animacją oddychania.
*   **Wdrożenie**: CSS `@keyframes` kontroluje skalę i przezroczystość.
*   **Najlepsze dla**: W każdym przypadku, gdy strona musi pozostać absolutnie czysta.

#### 2. Ujawnienie marki
Korzystanie z SVG `stroke-dasharray` animacja symuluje ręcznie rysowane linie logo HagiCode, po których następuje zanikanie tekstu.
*   **Technika**: animacja ścieżki SVG o bardzo dopracowanym wyglądzie.

### B. Mimikra ekranu szkieletu

> „Sztuka oszukiwania oka”. Symulując prawdziwy układ interfejsu użytkownika, użytkownicy mają wrażenie, że strona jest już w połowie załadowana.

#### 3. Szkielet czatu na pasku bocznym
To może być najbardziej praktyczna opcja. Ręcznie zbudowaliśmy układ w HTML, który odzwierciedla React `Sidebar` i `ChatInput` dokładnie elementy, a następnie nałóż na nie szarą, migoczącą animację.
*   **Wartość**: Po zakończeniu nawodnienia React szkielet natychmiast staje się prawdziwym składnikiem, a użytkownicy ledwo zauważają zmianę.

#### 4. Szkielet stosu kart
Symuluje ruch ułożonych kart propozycji podczas ładowania, wykorzystując transformacje 3D, aby karty subtelnie unosiły się w powietrzu.

### C. Abstrakcja i sztuka

> Pochwal się DNA maniaka HagiCode.

#### 5. Kształt geometryczny
Na środku ekranu renderowany jest geometryczny kształt (kwadrat), który z biegiem czasu płynnie przechodzi w okrąg, trójkąt, a na końcu w logo.
*   **Technologia**: Płynne przejścia dzięki CSS `border-radius`.

#### 6. Kod deszczu
Hołd dla *Matrixa*. Używając czcionki JetBrains Mono, w tle pojawiają się słabe strumienie znaków.
*   **Uwaga**: Aby zapewnić wydajność, strumienie znaków muszą znajdować się na mniejszym obszarze lub używać niższej częstotliwości odświeżania.

#### 7. Impuls neonowy
Świecący pierścień w stylu cyberpunkowym, który wykorzystuje wiele `box-shadow` warstwy, aby stworzyć mocny neonowy blask.

### D. Branding i motywy

> Spraw, aby system poczuł się żywy.

#### 8. Motyw sezonowy
To jest dynamiczny moduł ładujący. Sprawdza aktualną datę świąt, takich jak Nowy Rok Księżycowy czy Boże Narodzenie i ładuje odpowiednią animację SVG.
*   **Przykład**: Podczas Księżycowego Nowego Roku czerwone latarnie delikatnie kołyszą się na dole ekranu.

#### 9. Przepływ gradientowy
Tło wykorzystuje płynny gradient oparty na kolorach marki HagiCode. W połączeniu z animacją `background-size` i `background-position`, tworzy wrażenie ruchu przypominające zorzę polarną.

### E. Wyczucie techniczne

> Pozdrowienia dla deweloperów.

#### 10. Rozruch terminala
Symuluje wyjście konsoli. Linie kodu przewijają się szybko:
```text
> Initializing HagiCode Core...
> Loading models...
> Connecting to neural network...
```
To od razu wydaje się znajome każdemu programiście.

#### 11. Pasek postępu
U góry ekranu pojawi się cienki pasek postępu, z wartością procentową pokazaną po prawej stronie. Chociaż nie mamy dostępu do rzeczywistego postępu pobierania, możemy użyć timera, aby zasymulować „wiarygodny” proces ładowania: szybki przez pierwsze 80%, a następnie stopniowo wolniejszy przez ostatnie 20%.

### F. Kreatywność

#### 12. Montaż pikseli
To bardzo ciekawy pomysł. Małe kwadraty są rozproszone po ekranie, następnie zbiegają się w kierunku środka i stopniowo łączą się w ikonę logo HagiCode. Symbolizuje proces budowania kodu.

## Najlepsze praktyki i pułapki

W prawdziwej pracy rozwojowej HagiCode podsumowaliśmy kilka niezwykle ważnych szczegółów implementacji.

### 1. Defensywny CSS jest obowiązkowy
Nigdy nie bądź leniwy i pomiń przedrostek. Kiedyś zapomnieliśmy określić zakres stylów ekranu powitalnego za pomocą identyfikatora i globalnego `div` style po zamontowaniu Reacta nieoczekiwanie wpłynęły na ekran powitalny, psując układ.
**Wyciągnięta lekcja**: Umieść każdy selektor CSS pod `#boot-screen`i użyj `!important` aby podnieść priorytet, jeśli to konieczne, ale tylko wewnątrz CSS ekranu powitalnego.

### 2. Pełne wdzięku przejścia
Po pomyślnym zamontowaniu Reacta nie rób tego bezpośrednio `remove()` ekran powitalny DOM.
**Właściwe podejście**:
1.  Reaguj wyzwalacze `window.dispatchEvent(new Event('hagicode:ready'))`.
2.  Ekran powitalny nasłuchuje wydarzenia i pierwszych setów `opacity: 0`.
3.  Poczekaj 300 ms, co odpowiada czasowi przejścia CSS, i zadzwoń `.remove()` dopiero gdy ekran będzie całkowicie niewidoczny.

### 3. Synchronizacja zmiennych tematycznych
Wartości kolorów ekranu powitalnego są zakodowane na stałe `index.html`. Jeśli zmienimy podstawowy kolor Tailwinda, będziemy musieli zaktualizować także ekran powitalny.
**Optymalizacja**: Napisz prostą wtyczkę w skrypcie kompilacji Vite, aby ją przeczytać `tailwind.config.js` i wstrzyknij zmienne koloru do pliku `index.html` zmienne szablonowe, tworząc jedno źródło prawdy.

### 4. Wstępne ładowanie czcionek
Na ekranach powitalnych często konieczne jest użycie czcionki marki, ale jeśli czcionka ładuje się powoli, może pojawić się FOUT (Flash of Unstyled Text).
**Rozwiązanie**: Dodaj `<link rel="preload" href="/fonts/JetBrainsMono.woff2" as="font" type="font/woff2" crossorigin>` wewnątrz `<head>`. Jest to niedrogi i przynoszący wysoki zwrot sposób na poprawę doświadczenia.

### 5. Monitorowanie wydajności
Wstrzyknęliśmy `performance.mark('boot-start')` na dole `index.html`i oznaczone `boot-end` kiedy React pomyślnie się zamontował.
**Dlaczego to ma znaczenie**: Zbierając te dane za pomocą Application Insights, możemy bezpośrednio zmierzyć, w jakim stopniu ekran powitalny skraca postrzegany czas oczekiwania. Dane pokazują, że doskonały szkieletowy ekran może poprawić tolerancję użytkowników na „wolną sieć” o ponad 50%.

## Podsumowanie

Dobry ekran powitalny to coś więcej niż tylko „dekoracja na czas oczekiwania”. Jest to sygnał uścisku dłoni już w pierwszej interakcji pomiędzy produktem a użytkownikiem. W projekcie HagiCode ten system startowy oparty na **wzorze wariantów** pozwala nam elastycznie zmieniać style w zależności od świąt i wydań, znacznie zwiększając poczucie zabawy i profesjonalizmu produktu.

Rozwiązanie przedstawione w tym artykule jest zbudowane w całości na natywnych standardach sieciowych, bez wprowadzania jakichkolwiek ciężkich zależności, co odzwierciedla dążenie HagiCode do bycia „lekkim, ale potężnym”.
