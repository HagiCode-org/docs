HagiCode Desktop jest dostarczany z zestawem narzędzi Node zarządzanym za pomocą komputera stacjonarnego. Nie przejmuje globalnej instalacji Node na Twoim komputerze. Jego celem jest zapewnienie stabilnej i weryfikowalnej podstawy środowiska wykonawczego podczas uruchamiania pulpitu i przepływów związanych z jego uruchamianiem.

Przeczytaj to FAQ, jeśli widzisz `portable-fixed/toolchain` w diagnostyce, muszę wiedzieć które `node` Desktop jest używany lub zastanawiasz się, dlaczego wersja węzła systemu nie definiuje bezpośrednio zachowania środowiska wykonawczego Desktop.

## Dlaczego Desktop zawiera Node

Desktop musi obsługiwać te same przepływy obsługi w różnych systemach operacyjnych, kanałach instalacyjnych i środowiskach użytkowników. Gdyby zależało to tylko od istniejącej instalacji Node użytkownika, pomoc techniczna musiałaby zająć się jednocześnie nieaktualnymi wersjami, brakującym npm, nieprawidłową kolejnością ścieżek, zanieczyszczonymi pakietami globalnymi i zasadami rejestru specyficznymi dla organizacji.

Desktop opisuje to bardziej szczegółowo jako izolowane środowisko wykonawcze Node/npm dla zarządzanych interfejsów CLI zamiast ponownego wykorzystania środowiska ogólnosystemowego. Ten projekt ma następujące główne zalety:

- **Spójne wersje**: Desktop korzysta z przypiętej umowy wykonawczej Node 22, zamiast zgadywać lokalną wersję główną użytkownika.
- **Kontrolowane przepływy uruchamiania**: Polecenia zarządzane na komputerze stacjonarnym mogą preferować pakiety `node` i `npm` wpisy.
- **Wyraźne granice konfiguracji**: konfiguracja związana z interfejsem CLI jest oddzielona od konfiguracji globalnej obejmującej cały host, dzięki czemu aktualizacje, przełączanie i odzyskiwanie są mniej podatne na zakłócenia spowodowane różnicami w środowisku systemowym.
- **Zredukowane konflikty**: Dołączone ścieżki dotyczą środowisk poleceń utworzonych przez program Desktop i nie zastępują globalnego węzła użytkownika w jego terminalu.
- **Nieinwazyjne zachowanie hosta**: Ta izolacja pozwala uniknąć zapisywania stanu łańcucha narzędzi w istniejącym środowisku Node/npm użytkownika, a odinstalowanie Desktop nie zakłóca węzła, npm ani konfiguracji już obecnej na komputerze.

## Jak Desktop wybiera ścieżkę węzła

Gdy normalnie uruchamiasz program Desktop, woli on zainstalować zestaw narzędzi Node z aplikacją, zamiast najpierw czytać instalację Node z systemu. Dzięki temu wersja Node w terminalu jest oddzielona od wersji Node, której oczekuje Desktop podczas działania.

Gdy pakiet narzędzi jest włączony, środowiska poleceń utworzone przez program Desktop umieszczają ścieżki pakietów narzędzi na początku `PATH`. W rezultacie, nawet jeśli w systemie zainstalowana jest inna wersja Node, przepływy zarządzane przez Desktop są najpierw rozpoznawane przez dołączony do Desktop Node i npm.

Dotyczy to tylko środowisk wykonawczych utworzonych przez program Desktop. Jeśli biegniesz `node`, `npm`lub inne polecenia bezpośrednio w terminalu systemowym, nadal korzystają z normalnego środowiska systemowego i konfiguracji powłoki.

W środowiskach diagnostycznych lub procesach podrzędnych uruchamianych przez program Desktop możesz zobaczyć zmienną podobną do tej:

```bash
HAGICODE_PORTABLE_TOOLCHAIN_ROOT=/path/to/portable-fixed/toolchain
```

Ta zmienna oznacza, że środowisko wykonawcze zarządzane przez program Desktop rozpoznało katalog główny dołączonego łańcucha narzędzi. Widzenie tej ścieżki nie oznacza, że ​​Desktop zmienił węzeł systemowy; oznacza to tylko, że Desktop korzysta ze środowiska Node dostarczonego z aplikacją w jej własnym zakresie wykonawczym.

## Gdzie Desktop instaluje zarządzane pakiety npm

Desktop przechowuje pliki wykonawcze i modyfikowalne pliki pakietów npm w różnych lokalizacjach. Dołączona ścieżka łańcucha narzędzi, taka jak `portable-fixed/toolchain/node` jest niezmiennym źródłem dla zarządzanego pulpitu `node` i `npm` polecenia. Zamiast tego zarządzane pakiety globalne są instalowane w katalogu danych użytkownika Electron:

```text
userData/node22/npmGlobal
```

Jeśli program Desktop zostanie później przeniesiony do węzła 24, użyje osobnego drzewa pakietów, takiego jak `userData/node24/npmGlobal`. Pakiety zainstalowane dla węzła 22 pozostają na dysku, ale nie są traktowane jako zainstalowane dla węzła 24, dopóki nie zostaną ponownie zainstalowane z nowym przedrostkiem węzła głównego.

Stan zależności pulpitu sprawdza tylko aktywny `userData/node<major>/npmGlobal` przedrostek. Pliki pakietów poza tym prefiksem są ignorowane przez zarządzanie zależnościami.
