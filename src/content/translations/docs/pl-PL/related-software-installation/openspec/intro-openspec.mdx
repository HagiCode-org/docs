# Wprowadzenie do OpenSpec

OpenSpec to narzędzie do tworzenia oprogramowania sterowane specyfikacjami, które pomaga zespołom zarządzać procesem rozwoju oprogramowania za pomocą ustrukturyzowanych propozycji i specyfikacji. Ten artykuł zawiera kompleksowe wprowadzenie do podstawowych koncepcji, przebiegów pracy i sposobów użycia OpenSpec.

## Czym jest OpenSpec?

OpenSpec to narzędzie wiersza poleceń służące do zarządzania propozycjami (Proposals), specyfikacjami (Specifications) i zmianami (Changes) w procesie rozwoju oprogramowania. Standaryzuje proces rozwoju, zapewniając, że każda funkcja przechodzi przez gruntowne projektowanie i przegląd — od pomysłu do implementacji.

### Główne założenia

OpenSpec opiera się na następujących zasadach:

- **Specyfikacje najpierw**: Zdefiniuj jasne specyfikacje przed napisaniem kodu
- **Ustrukturyzowana współpraca**: Ułatwiaj współpracę zespołową poprzez ustandaryzowane formaty dokumentów
- **Śledzalność**: Rejestruj cały proces od propozycji po wdrożenie
- **Zapewnienie jakości**: Poprzez walidację specyfikacji zapewnij, że kod odpowiada zamierzeniom projektu

### Główne funkcje

OpenSpec oferuje następujące funkcje:

1. **Zarządzanie propozycjami**: Twórz, przeglądaj i śledź propozycje funkcji
2. **Definiowanie specyfikacji**: Twórz i utrzymuj dokumenty specyfikacji technicznych
3. **Śledzenie zmian**: Rejestruj historię wszystkich zrecenzowanych zmian
4. **Walidacja specyfikacji**: Zapewnij, że zmiany w kodzie spełniają zdefiniowane specyfikacje
5. **Współpraca zespołowa**: Wspieraj procesy rozwoju i przeglądu z udziałem wielu osób

## Przebieg pracy w OpenSpec

Typowy przebieg pracy w OpenSpec obejmuje następujące etapy:

### 1. Utworzenie propozycji (Proposal)

Gdy pojawi się nowa funkcja lub wymaganie znaczącej zmiany, najpierw utwórz propozycję:

```bash
openspec proposal create
```

Propozycja zawiera:
- **Opis problemu**: Problem lub wymaganie do rozwiązania
- **Rozwiązanie**: Przegląd proponowanego rozwiązania
- **Zakres wpływu**: Wpływ na istniejący system
- **Alternatywy**: Inne rozważane rozwiązania

### 2. Spisanie specyfikacji (Specification)

Po zatwierdzeniu propozycji napisz szczegółową specyfikację techniczną:

```bash
openspec spec create
```

Specyfikacja zawiera:
- **Projekt techniczny**: Szczegółowy plan implementacji technicznej
- **Definicja interfejsów**: Projekt API, struktur danych itp.
- **Szczegóły implementacji**: Kluczowa logika implementacji i algorytmy
- **Plan testów**: Jak zweryfikować poprawność implementacji

### 3. Utworzenie zmiany (Change)

Po rozpoczęciu implementacji utwórz zmianę, aby śledzić postęp prac:

```bash
openspec change create
```

Zmiana zawiera:
- **Powiązana specyfikacja**: Specyfikacja realizowana przez tę zmianę
- **Zadania implementacyjne**: Lista konkretnych zadań deweloperskich
- **Śledzenie postępu**: Status wykonania zadań
- **Zmiany w kodzie**: Rzeczywiste modyfikacje kodu

### 4. Walidacja i archiwizacja

Po zakończeniu prac zweryfikuj, czy zmiana spełnia specyfikację, a następnie zarchiwizuj:

```bash
openspec change validate
openspec change archive
```

## Struktura katalogów OpenSpec

OpenSpec tworzy katalog `openspec/` w katalogu głównym projektu o następującej strukturze:

```
openspec/
├── proposals/           # katalog z propozycjami
│   ├── PROPOSAL-001.md
│   └── PROPOSAL-002.md
├── specs/              # katalog specyfikacji
│   ├── SPEC-001.md
│   └── SPEC-002.md
├── changes/            # katalog zmian
│   ├── feature-name/
│   │   ├── proposal.md
│   │   ├── design.md
│   │   ├── specs.md
│   │   └── tasks.md
│   └── another-feature/
└── archived/           # zarchiwizowane zmiany
    └── feature-name/
```

### Wyjaśnienie katalogów

- **proposals/**: Przechowuje wszystkie dokumenty propozycji, każda propozycja w osobnym pliku Markdown
- **specs/**: Przechowuje dokumenty specyfikacji technicznych odpowiadające propozycjom
- **changes/**: Przechowuje bieżące zmiany, każda w osobnym podkatalogu
- **archived/**: Przechowuje zakończone i zarchiwizowane zmiany

## Podstawowe polecenia

### Polecenia związane z propozycjami

```bash
# Utwórz nową propozycję
openspec proposal create

# Wyświetl listę wszystkich propozycji
openspec proposal list

# Pokaż szczegóły propozycji
openspec proposal view <proposal-id>

# Zatwierdź propozycję
openspec proposal approve <proposal-id>
```

### Polecenia związane ze specyfikacjami

```bash
# Utwórz nową specyfikację
openspec spec create

# Wyświetl listę wszystkich specyfikacji
openspec spec list

# Pokaż szczegóły specyfikacji
openspec spec view <spec-id>

# Zaktualizuj specyfikację
openspec spec update <spec-id>
```

### Polecenia związane ze zmianami

```bash
# Utwórz nową zmianę
openspec change create

# Wyświetl listę wszystkich zmian
openspec change list

# Pokaż szczegóły zmiany
openspec change view <change-name>

# Dodaj zadanie
openspec change add-task <change-name>

# Zwaliduj zmianę
openspec change validate <change-name>

# Zarchiwizuj zmianę
openspec change archive <change-name>
```

### Polecenia ogólne

```bash
# Wyświetl wersję
openspec --version

# Wyświetl pomoc
openspec --help

# Wyświetl pomoc dla konkretnego polecenia
openspec <command> --help
```

## Użycie OpenSpec w projekcie HagiCode

### Integracja projektu

W projekcie HagiCode OpenSpec jest zintegrowany z workflowem deweloperskim:

1. **Sprawdzenie przy inicjalizacji**: kreator uruchomienia sprawdza, czy OpenSpec jest zainstalowany
2. **Rozwój oparty na propozycjach**: Wszystkie nowe funkcje muszą przejść przez proces propozycji OpenSpec
3. **Przegląd kodu**: Zmiany muszą spełniać specyfikacje OpenSpec, aby mogły zostać zmergowane
4. **Synchronizacja dokumentacji**: Dokumentacja OpenSpec jest utrzymywana w synchronizacji z implementacją kodu

### Wymagania wersji

Projekt HagiCode wymaga użycia wersji **OpenSpec >=1.0.0 <2.0.0**.

Polecenie instalacji:

```bash
npm install -g @fission-ai/openspec@1
```

### Przykład przebiegu pracy

Poniżej pełny przykład użycia OpenSpec w projekcie HagiCode:

#### Krok 1: Utwórz propozycję

```bash
cd /path/to/hagicode-project
openspec proposal create
```

Wprowadź w interaktywnej podpowiedzi:
- Tytuł propozycji: Dodanie funkcji uwierzytelniania użytkowników
- Opis problemu: System wymaga logowania użytkowników i zarządzania uprawnieniami
- Rozwiązanie: Wdrożenie systemu uwierzytelniania opartego na JWT

#### Krok 2: Spisanie specyfikacji

Po zatwierdzeniu propozycji utwórz szczegółową specyfikację:

```bash
openspec spec create
```

W specyfikacji zdefiniuj:
- Projekt procesu uwierzytelniania
- Definicje interfejsów API
- Struktury tabel w bazie danych
- Aspekty bezpieczeństwa

#### Krok 3: Utwórz zmianę

Rozpocznij implementację:

```bash
openspec change create
```

Wprowadź nazwę zmiany: `add-user-authentication`

#### Krok 4: Dodaj zadania

```bash
openspec change add-task add-user-authentication
```

Dodaj konkretne zadania:
- Implementacja endpointu rejestracji użytkownika
- Implementacja endpointu logowania
- Implementacja middleware JWT
- Pisanie testów jednostkowych

#### Krok 5: Implementacja i walidacja

Po zakończeniu kodowania zweryfikuj zmianę:

```bash
openspec change validate add-user-authentication
```

#### Krok 6: Archiwizacja

Po pomyślnej walidacji zarchiwizuj zmianę:

```bash
openspec change archive add-user-authentication
```

## Najlepsze praktyki

### Pisanie propozycji

1. **Jasne określenie problemu**: Wyraźnie zdefiniuj problem do rozwiązania — unikaj niejasnych opisów
2. **Rozważ alternatywy**: Wypisz i porównaj różne rozwiązania
3. **Oceń wpływ**: Przeanalizuj wpływ i ryzyka dla istniejącego systemu
4. **Bądź zwięzły**: Propozycje powinny być krótkie i jasne — szczegóły pozostaw specyfikacji

### Pisanie specyfikacji

1. **Szczegółowe, ale nie nadmierne**: Podaj wystarczające detale, unikaj nadmiernego projektowania
2. **Używaj diagramów**: Wykorzystuj diagramy przepływu, diagramy architektury itd. do ilustracji
3. **Zdefiniuj interfejsy**: Wyraźnie zdefiniuj wszystkie publiczne interfejsy i struktury danych
4. **Rozważ przypadki brzegowe**: Wymień sytuacje wyjątkowe i obsługę błędów

### Zarządzanie zmianami

1. **Iteruj w małych krokach**: Podziel duże funkcje na wiele małych zmian
2. **Aktualizuj na bieżąco**: Utrzymuj spójność stanu zadań i dokumentacji
3. **Przegląd kodu**: Wykonaj przegląd kodu po zakończeniu zmiany
4. **Archiwizuj na czas**: Archiwizuj zakończone zmiany, utrzymuj porządek w katalogu roboczym

### Współpraca zespołowa

1. **Regularne przeglądy**: Regularnie przeglądaj propozycje i specyfikacje, aby zapewnić konsensus w zespole
2. **Dokumentacja na pierwszym miejscu**: Dokończ dokumentację przed rozpoczęciem kodowania
3. **Utrzymuj komunikację**: Zapisuj dyskusje i decyzje w propozycjach i specyfikacjach
4. **Kontrola wersji**: Dołącz dokumentację OpenSpec do kontroli wersji

## Typowe scenariusze użycia

### Scenariusz 1: Dodanie nowej funkcji

1. Utwórz propozycję opisującą wymagania nowej funkcji
2. Zespół dyskutuje i akceptuje propozycję
3. Napisz szczegółową specyfikację techniczną
4. Utwórz zmianę i zaimplementuj funkcję
5. Zwaliduj i zarchiwizuj zmianę

### Scenariusz 2: Refaktoryzacja istniejącego kodu

1. Utwórz propozycję wyjaśniającą powody i cele refaktoryzacji
2. Zdefiniuj plan i kroki refaktoryzacji w specyfikacji
3. Utwórz zmianę i refaktoryzuj stopniowo
4. Zapewnij pokrycie testami, zweryfikuj wyniki refaktoryzacji
5. Zarchiwizuj zmianę

### Scenariusz 3: Naprawa błędu

Dla prostych błędów:
- Można naprawić bez pełnego procesu OpenSpec

Dla złożonych błędów:
1. Utwórz propozycję analizującą przyczynę źródłową błędu
2. Zdefiniuj rozwiązanie naprawcze w specyfikacji
3. Utwórz zmianę i zaimplementuj naprawę
4. Dodaj testy zapobiegające regresji
5. Zarchiwizuj zmianę

### Scenariusz 4: Zarządzanie długu technicznego

1. Utwórz propozycję wymieniającą pozycje długu technicznego
2. Zdefiniuj plan poprawy w specyfikacji
3. Utwórz zmianę, aby stopniowo spłacać dług techniczny
4. Śledź postęp i efekty
5. Zarchiwizuj ukończone ulepszenia

## Integracja z innymi narzędziami

### Integracja z Git

OpenSpec integruje się bezproblemowo z Git:

```bash
# Dokumentacja OpenSpec powinna być uwzględniona w kontroli wersji
git add openspec/
git commit -m "Add OpenSpec proposal for feature X"
```

### Integracja z CI/CD

Weryfikuj specyfikacje OpenSpec w procesie CI/CD:

```yaml
# .github/workflows/openspec.yml
name: OpenSpec Validation
on: [pull_request]
jobs:
  validate:
    runs-on: ubuntu-latest
    steps:
      - uses: actions/checkout@v2
      - name: Install OpenSpec
        run: npm install -g @fission-ai/openspec@1
      - name: Validate changes
        run: openspec change validate
```

### Integracja z IDE

Zaleca się używanie edytora obsługującego Markdown do edycji dokumentów OpenSpec:
- VS Code + wtyczka Markdown
- JetBrains IDEs + obsługa Markdown
- Vim/Neovim + wtyczka Markdown

## Rozwiązywanie problemów

### Częste problemy

#### Problem: Polecenie OpenSpec nie zostało znalezione

**Rozwiązanie**:
1. Potwierdź, że OpenSpec jest zainstalowany: `npm list -g @fission-ai/openspec`
2. Sprawdź, czy zmienna środowiskowa PATH zawiera globalny katalog bin npm
3. Ponownie zainstaluj: `npm install -g @fission-ai/openspec@1`

#### Problem: Nie można utworzyć propozycji lub zmiany

**Rozwiązanie**:
1. Potwierdź, że bieżący katalog to katalog główny projektu
2. Sprawdź, czy istnieje katalog `openspec/`, jeśli nie, utwórz go
3. Potwierdź, że masz uprawnienia do zapisu

#### Problem: Walidacja nie powiodła się

**Rozwiązanie**:
1. Sprawdź, czy dokumentacja w katalogu zmian jest kompletna
2. Potwierdź, że wszystkie wymagane zadania zostały ukończone
3. Przejrzyj szczegółowe informacje o błędzie: `openspec change validate <change-name> --verbose`

#### Problem: Archiwizacja nie powiodła się

**Rozwiązanie**:
1. Upewnij się, że zmiana przeszła walidację
2. Sprawdź, czy katalog `openspec/archived/` istnieje
3. Potwierdź, że nie ma niezatwierdzonych zmian w kodzie

## Zaawansowane użycie

### Szablony niestandardowe

OpenSpec obsługuje niestandardowe szablony propozycji i specyfikacji:

```bash
# Utwórz katalog szablonów w katalogu głównym projektu
mkdir -p .openspec/templates

# Utwórz niestandardowy szablon propozycji
cat > .openspec/templates/proposal.md << 'EOF'
# Proposal: {{title}}

## Problem Statement
{{problem}}

## Proposed Solution
{{solution}}

## Custom Section
Your custom content here
EOF
```

### Plik konfiguracyjny

Utwórz plik konfiguracyjny `.openspec.json` w katalogu głównym projektu:

```json
{
  "version": "1.0",
  "templates": {
    "proposal": ".openspec/templates/proposal.md",
    "spec": ".openspec/templates/spec.md"
  },
  "validation": {
    "requireTests": true,
    "requireDocs": true
  },
  "hooks": {
    "pre-archive": "npm test"
  }
}
```

### Hooki (Hooks)

OpenSpec obsługuje uruchamianie niestandardowych skryptów przed i po określonych operacjach:

```json
{
  "hooks": {
    "pre-create": "scripts/check-permissions.sh",
    "post-archive": "scripts/notify-team.sh"
  }
}
```
