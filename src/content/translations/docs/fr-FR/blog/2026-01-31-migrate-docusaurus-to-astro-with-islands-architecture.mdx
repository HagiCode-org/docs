## De Docusaurus 3.x à Astro 5.x : retour sur la migration du site HagiCode

> Cet article revient sur notre migration complète du site officiel HagiCode de Docusaurus 3.x vers Astro 5.x. Nous plongerons en profondeur dans la manière dont l'architecture Islands d'Astro nous a permis de résoudre les goulets d'étranglement en matière de performances tout en préservant nos composants React existants, offrant des améliorations tant en vitesse de build qu'en performances de chargement.

## Contexte

En janvier 2026, nous avons effectué une « greffe cardiaque » sur le site officiel HagiCode en migrant entièrement son framework principal de Docusaurus 3.x vers Astro 5.x. Il ne s'agissait pas d'une réécriture impulsive, mais d'une décision technique mûrement réfléchie.

Avant la migration, notre site était fonctionnellement complet, mais il commençait à montrer quelques « problèmes de luxe » classiques : des artefacts de build volumineux, des payloads JavaScript excessifs et une vitesse de chargement de page moins qu'idéale sur les pages de documentation complexes. En tant que projet d'assistant de codage IA, HagiCode nécessite des mises à jour fréquentes de la documentation et des fonctionnalités, la vitesse de build affectant donc directement la vitesse de publication. Parallèlement, nous voulions que le site soit plus favorable au référencement (SEO) afin que davantage de développeurs puissent découvrir le projet.

Pour résoudre ces problèmes, nous avons pris une décision audacieuse : reconstruire l'ensemble du système sur Astro. L'impact de cette décision s'est révélé plus important que prévu. Nous y reviendrons en détail.

## À propos de HagiCode

L'approche de migration de site partagée dans cet article provient de notre expérience pratique dans le projet [HagiCode](https://github.com/HagiCode-org/site).

HagiCode est un assistant de codage IA axé sur l'amélioration de l'efficacité du développement. Nous nous soucions non seulement de l'itération des fonctionnalités principales, mais aussi de l'expérience développeur. Cette refonte du site visait également à offrir aux utilisateurs l'expérience la plus rapide possible lors de la consultation de notre documentation et de notre site officiel.

## Pourquoi quitter l'écosystème mature de Docusaurus ?

Au sein de l'écosystème React, Docusaurus est depuis longtemps la « réponse standard » pour les sites de documentation. Il fonctionne hors de la boîte, offre un riche écosystème de plugins et possède une communauté active. Mais au fil de l'ajout de fonctionnalités à HagiCode, nous avons aussi ressenti ses limites :

1.  **Goulots d'étranglement en matière de performances** : Docusaurus est fondamentalement une SPA (application monopage) React. Même si vous n'écrivez que des pages statiques, le client doit toujours charger le runtime React et effectuer l'hydratation de la page, ce qui est inutilement lourd pour de simples pages de documentation.
2.  **Taille importante des assets** : même lorsqu'une page contient très peu de contenu, la taille du JS bundle reste relativement fixe. Ce n'est pas idéal pour les utilisateurs mobiles ou les conditions réseau médiocres.
3.  **Flexibilité limitée** : bien qu'il soit extensible, nous voulions un contrôle plus fin du pipeline de build.

Astro est arrivé exactement au bon moment pour résoudre ces problèmes. Il a introduit une nouvelle « architecture Islands » : par défaut, Astro génère du HTML statique avec zéro JavaScript, et seuls les composants nécessitant de l'interactivité sont « activés » et chargent du JS. Cela signifie que la majeure partie de notre site devient du HTML pur et se charge extrêmement rapidement.

## Stratégie de migration principale : une transition architecturale fluide

La migration n'était pas un simple copier-coller. Elle nécessitait un changement de paradigme. Nous sommes passés du modèle « tout React » de Docusaurus au modèle « Core + Islands » d'Astro.

### 1. Reconstruction du système de configuration

Il a d'abord fallu passer de `docusaurus.config.ts` à `astro.config.mjs`. Ce n'était pas qu'un simple renommage de fichier, mais une réécriture de la logique de routage et de build.

Dans Docusaurus, tout est un plugin. Dans Astro, tout est une intégration. Nous devions redéfinir le chemin de base du site, le mode de sortie du build (statique vs SSR) et la stratégie d'optimisation des assets.

**Avant la migration :**
```typescript
// docusaurus.config.ts
export default {
  title: 'HagiCode',
  url: 'https://hagicode.com',
  baseUrl: '/',
  // ... plus de configuration
};
```

**Après la migration :**
```javascript
// astro.config.mjs
import { defineConfig } from 'astro/config';
import react from '@astrojs/react';

export default defineConfig({
  integrations: [react()],
  site: 'https://hagicode.com',
  base: '/',
  // Paramètres d'optimisation pour les assets statiques
  build: {
    inlineStylesheets: 'auto',
  },
});
```

### 2. Ce qu'il faut conserver et ce qu'il faut refactorer dans les composants React

Cela a été la partie la plus douloureuse de la migration. Notre site existant comportait de nombreux composants React, tels que des onglets, la coloration syntaxique, des boutons de feedback, etc. Les jeter aurait été du gaspillage, mais tout conserver aurait rendu le payload JavaScript trop lourd.

HagiCode a adopté une stratégie d'**hydratation progressive** :
*   **Composants purement statiques** : pour le contenu de présentation tel que les en-têtes, pieds de page et la documentation en texte brut, nous les avons réécrits en composants Astro (fichiers `.astro`) et les avons rendus directement en HTML au moment du build.
*   **Îlots interactifs** : pour les composants qui doivent rester interactifs, tels que les sélecteurs de thème, la commutation d'onglets et les boutons de copie des blocs de code, nous avons conservé l'implémentation React et ajouté les directives `client:load` ou `client:visible`.

Par exemple, notre composant Tabs couramment utilisé dans la documentation :

```jsx
// src/components/Tabs.jsx
import { useState } from 'react';
import './Tabs.css'; // Import des styles

export default function Tabs({ items }) {
  const [activeIndex, setActiveIndex] = useState(0);
  // ... logique d'état
  return (
    <div className="tabs-wrapper">
      {/* Logique de rendu */}
    </div>
  );
}
```

Lorsqu'il est utilisé dans Markdown, nous indiquons explicitement à Astro : « Ce composant a besoin de JS. »

```markdown
// src/content/docs/example.mdx
import Tabs from '../../components/Tabs.jsx';

<!-- Charger le JS uniquement lorsque le composant entre dans le viewport -->
<Tabs client:visible items={...} />
```

De cette façon, les composants interactifs hors du viewport ne consomment pas de bande passante, ce qui améliore considérablement la vitesse de chargement de l'écran initial.

### 3. Adaptation du système de style : des CSS Modules au CSS Scoped

Docusaurus prend en charge les CSS Modules par défaut, tandis qu'Astro encourage le CSS Scoped via la balise `<style>`. L'idée fondamentale derrière les deux est l'isolation des styles, mais la syntaxe est différente.

Lors de la migration de HagiCode, nous avons converti la plupart des CSS Modules complexes en styles scoped d'Astro. Cela s'est avéré être une bonne chose, car dans les fichiers `.astro`, les styles et les modèles résident dans le même fichier, ce qui rend la maintenance plus intuitive.

**Avant la refactorisation :**
```css
/* Tabs.module.css */
.wrapper { background: var(--ifm-background-color); }
```

**Après la refactorisation (Astro Scoped) :**
```html
<!-- Tabs.astro -->
<div class="tabs-wrapper">
  <slot />
</div>

<style>
  .tabs-wrapper {
    /* Utiliser directement les variables CSS pour s'adapter au thème */
    background: var(--bg-color);
    padding: 1rem;
  }
</style>
```

Parallèlement, nous avons unifié le système de variables CSS globales et utilisé les capacités sensibles à l'environnement d'Astro pour garantir que le mode sombre bascule fluidement d'une page à l'autre.

## Pièges rencontrés en pratique et comment nous les avons résolus

Lors de la migration réelle de HagiCode, nous avons rencontré pas mal de problèmes. Voici plusieurs des plus typiques.

### 1. Problèmes de chemins et de variables d'environnement

HagiCode prend en charge le déploiement en sous-chemin, par exemple sous un sous-répertoire GitHub Pages. Dans Docusaurus, le `baseUrl` est géré automatiquement. Dans Astro, cependant, nous devons être plus prudents dans la gestion des liens d'images et des requêtes API.

Nous avons introduit un mécanisme de variables d'environnement pour gérer cela de manière cohérente :

```javascript
// Gérer les chemins dans le script de build
const getBasePath = () => import.meta.env.VITE_SITE_BASE || '/';
```

Veillez à ne pas coder en dur les chemins commençant par `/` dans votre code. En développement comme en production, ou après avoir configuré un chemin `base`, cela peut provoquer des erreurs 404 pour les assets.

### 2. Compatibilité des scripts CommonJS

Notre ancien site comportait des scripts Node.js utilisés pour des tâches telles que la récupération automatique de données Metrics et la mise à jour du sitemap, écrits en CommonJS (`require`). Astro et les outils de build modernes ont pleinement adopté les ES Modules (`import`/`export`).

Si vous avez des scripts similaires, n'oubliez pas de les refactorer tous en ES Modules. C'est la direction que prend l'écosystème, et plus tôt vous effectuerez le changement, moins vous aurez de problèmes par la suite.

```javascript
// Ancienne méthode
const fs = require('fs');

// Nouvelle méthode
import fs from 'fs';
```

### 3. Ne pas oublier le SEO et les redirections

Les moteurs de recherche ont déjà indexé les anciennes pages Docusaurus de HagiCode. Si vous passez directement à Astro et que la structure des URL change, vous risquez de vous retrouver avec un grand nombre de 404 et une chute significative du classement de recherche.

Nous avons configuré des règles de redirection dans Astro :

```javascript
// astro.config.mjs
export default defineConfig({
  redirects: {
    '/docs/old-path': '/docs/new-path',
    // Mapper les anciens liens vers les nouveaux en masse
  }
});
```

Vous pouvez également gérer cela au niveau de la configuration du serveur. Assurez-vous que les anciens liens peuvent être redirigés en 301 vers les nouvelles adresses, car cela est crucial pour le SEO.

## Conclusion

Pour HagiCode, la migration de Docusaurus vers Astro n'était pas qu'une simple mise à niveau de framework. C'était aussi la mise en pratique d'une philosophie « performances d'abord ».

**Ce que nous avons gagné :**
*   **Scores Lighthouse exceptionnels** : après la migration, le score de performance du site HagiCode s'est facilement approché du score parfait.
*   **Vitesse de build plus rapide** : les capacités de build incrémental d'Astro ont réduit de moitié le temps de publication des mises à jour de la documentation.
*   **Flexibilité préservée** : avec l'architecture Islands, nous n'avons sacrifié aucune fonctionnalité interactive et pouvions toujours utiliser React là où c'était nécessaire.

Si vous maintenez également un site orienté documentation et que vous luttez contre la taille du bundle ou la vitesse de chargement, Astro vaut vraiment la peine d'être essayé. Bien que le processus de migration nécessite quelques ajustements, l'expérience utilisateur fluide que vous obtenez en retour en vaut absolument la peine.

Le système de build partagé dans cet article est exactement l'approche que nous avons développée par essais et erreurs réels lors de la construction de HagiCode. Si vous trouvez cette approche précieuse, cela en dit long sur notre force d'ingénierie, et HagiCode lui-même mérite probablement aussi un examen plus attentif.
