## De TypeScript à C# : pratique de portage inter-langage pour le SDK Codex

> Pour faire simple, cet article est aussi un peu notre bébé : il documente le processus complet de portage du SDK TypeScript officiel de Codex vers C#. L'appeler un « portage » donne presque l'impression que c'était trop facile — c'était plutôt une longue aventure, car ces deux langages ont des personnalités très différentes, et nous avons dû trouver un moyen de les faire coopérer.

## Contexte

Codex est l'outil CLI d'Agent IA publié par OpenAI, et il est vraiment puissant. L'équipe officielle fournit un SDK TypeScript dans le package `@openai/codex`. Il interagit avec le CLI Codex en appelant la commande `codex exec --experimental-json` et en analysant un flux d'événements JSONL.

Le problème, c'est que dans le projet HagiCode, nous devons l'utiliser dans un environnement .NET pur — spécifiquement dans des services backend C# et des applications de bureau. Il n'était pas raisonnable d'introduire un runtime Node.js dans un projet .NET juste pour appeler un outil CLI. Cela aurait été bien trop lourd.

Nous avions donc deux choix : maintenir une couche de pont Node.js complexe, ou construire nous-mêmes un SDK C# natif.

Nous avons choisi la seconde option.

## À propos de HagiCode

Cet article provient également directement de notre expérience pratique dans le projet [HagiCode](https://hagicode.com). HagiCode est un projet d'assistant de codage IA open source. En termes simples, cela signifie maintenir plusieurs composants à la fois : une extension VSCode côté frontend, des services IA côté backend, et un client de bureau multiplateforme. Cette complexité multi-langage et multiplateforme est exactement la raison pour laquelle nous avions besoin d'un SDK C# natif — nous ne voulions vraiment pas ajouter Node.js à cette pile technologique déjà complexe.

## Pourquoi un portage natif plutôt qu'un pont inter-processus

Il existe de nombreuses approches pour permettre à C# d'appeler du code TypeScript :

1. **Pont via HTTP** : encapsuler le SDK TypeScript dans un service HTTP, et C# appelle ce service via REST.
2. **Pont via CLI** : C# appelle directement le CLI Codex et analyse la sortie.
3. **SDK natif C#** : réécrire entièrement le SDK en C#.

Le pont HTTP et le pont CLI sont les approches les plus simples au départ. Mais ils ont des problèmes fondamentaux :

- **Dépendance à Node.js** : chaque déploiement nécessite l'installation de Node.js, ce qui augmente la complexité opérationnelle.
- **Latence supplémentaire** : chaque appel traverse une couche de processus supplémentaire.
- **Difficulté de débogage** : lorsque quelque chose ne fonctionne pas, il faut déboguer à travers deux langages et deux runtimes.

Le SDK natif C# résout ces problèmes de manière fondamentale :

- **Zéro dépendance à Node.js** : fonctionne dans n'importe quel environnement .NET.
- **Appels directs** : pas de surcharge inter-processus.
- **Expérience de développement unifiée** : le débogage, l'intellisense et le refactoring fonctionnent tous dans un seul langage.

## Différences clés entre TypeScript et C# et comment les gérer

### 1. Système de types

TypeScript utilise un typage structurel, tandis que C# utilise un typage nominal. En pratique, cela signifie que deux objets TypeScript avec les mêmes propriétés sont considérés comme du même type, même s'ils n'ont pas de relation d'héritage.

En C#, nous devons définir explicitement des interfaces ou des classes. Nous avons adopté une approche systématique : pour chaque type TypeScript, nous avons créé une classe C# correspondante.

```csharp
// TypeScript: interface CodexEvent { type: string; data: any; }
// C# equivalent:
public class CodexEvent
{
    [JsonPropertyName("type")]
    public string Type { get; set; } = string.Empty;
    
    [JsonPropertyName("data")]
    public JsonElement? Data { get; set; }
}
```

### 2. Gestion asynchrone

TypeScript utilise des Promises et async/await, ce qui est conceptuellement similaire à C#. Mais il y a des différences subtiles :

- TypeScript n'a pas de `CancellationToken`, tandis que C# l'utilise largement pour l'annulation.
- TypeScript utilise `Promise.all()` pour l'exécution parallèle, C# utilise `Task.WhenAll()`.

Nous avons introduit `CancellationToken` dans toutes les méthodes asynchrones du SDK C#, permettant aux appelants d'annuler des opérations longues.

### 3. Gestion des erreurs

TypeScript utilise `try/catch` avec des types d'erreur personnalisés. C# a un système d'exceptions plus riche avec des exceptions personnalisées.

```csharp
public class CodexException : Exception
{
    public CodexErrorCode Code { get; }
    
    public CodexException(CodexErrorCode code, string message) 
        : base(message)
    {
        Code = code;
    }
}
```

### 4. Flux d'événements JSONL

C'est le cœur du SDK. Codex CLI produit un flux JSONL (un objet JSON par ligne). Le SDK TypeScript le traite en utilisant des readline et des event emitters.

En C#, nous avons utilisé `StreamReader.ReadLineAsync()` avec un modèle producteur-consommateur :

```csharp
public async IAsyncEnumerable<CodexEvent> ReadEventsAsync(
    Stream stream,
    [EnumeratorCancellation] CancellationToken cancellationToken = default)
{
    using var reader = new StreamReader(stream);
    while (!reader.EndOfStream)
    {
        cancellationToken.ThrowIfCancellationRequested();
        var line = await reader.ReadLineAsync(cancellationToken);
        if (line == null) break;
        
        var evt = JsonSerializer.Deserialize<CodexEvent>(line);
        if (evt != null) yield return evt;
    }
}
```

## Pièges de portage et solutions

### Piège 1 : le JSONL n'est pas du JSON standard

Le flux JSONL de Codex peut contenir des lignes qui ne sont pas du JSON valide (comme des messages de debug). Nous devons gérer gracieusement les erreurs de parsing.

**Solution** : envelopper chaque ligne dans un try-catch et journaliser les lignes invalides sans interrompre le flux.

### Piège 2 : les types union TypeScript

TypeScript utilise des types union (`string | number`), qui n'existent pas directement en C#. Nous avons utilisé des types nullable et des pattern matching pour gérer ces cas.

### Piège 3 : les fonctions de rappel (callbacks)

Le SDK TypeScript utilise largement les callbacks. En C#, nous avons converti les callbacks en événements C# standard (`event EventHandler<T>`), ce qui est plus idiomatique.

## Tests et vérification

Pour nous assurer que le SDK C# se comporte de manière identique au SDK TypeScript, nous avons adopté les stratégies de test suivantes :

1. **Tests d'intégration** : exécuter le même scénario avec les deux SDK et comparer les résultats.
2. **Tests de compatibilité** : vérifier que le SDK C# peut analyser tous les types d'événements produits par Codex CLI.
3. **Tests de performance** : comparer la latence et l'utilisation mémoire entre les deux SDK.

## Conclusion

Le portage du SDK TypeScript Codex vers C# a été un défi technique intéressant. Les différences entre les deux langages nous ont obligés à repenser certaines conceptions, mais le résultat final — un SDK C# natif sans dépendance à Node.js — en valait la peine.

Si vous êtes confronté à un besoin de portage inter-langage similaire, nos recommandations sont :

1. **Comprendre la philosophie de chaque langage** plutôt que de traduire mot à mot.
2. **Utiliser les idiomes du langage cible** — le code C# devrait ressembler à du C#, pas à du TypeScript transposé.
3. **Investir dans les tests** — la vérification de compatibilité est cruciale.
4. **Documenter les écarts** — notez clairement les différences de comportement entre l'original et le portage.
