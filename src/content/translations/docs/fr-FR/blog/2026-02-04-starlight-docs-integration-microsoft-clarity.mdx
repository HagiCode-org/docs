## Choix technique et exploration

Au début, nous avons discuté de plusieurs approches d'intégration lors de la phase de Proposal. Puisque nous utilisions Starlight, un framework de documentation basé sur Astro, l'idée la plus évidente était d'utiliser les Hooks Astro.

Nous avons d'abord essayé de modifier `astro.config.mjs` et prévu d'injecter le script Clarity pendant le build. Bien que cette approche puisse garantir une couverture globale, elle manquait de flexibilité : nous ne pouvions pas charger ou décharger dynamiquement le script en fonction de la préférence de l'utilisateur.

En tenant compte de l'expérience utilisateur et du contrôle de la vie privée, nous avons finalement choisi une approche de **surcharge de composant**. Starlight permet aux développeurs de surcharger ses composants internes, ce qui signifie que nous pouvions reprendre la logique de rendu pour `<footer>` ou `<head>` et contrôler précisément quand Clarity est chargé.

Il y a aussi eu un petit détour : à l'origine, nous voulions créer un wrapper de layout nommé `StarlightWrapper.astro`. Mais lors du débogage réel, nous avons découvert que le mécanisme de routage de Starlight n'invoque pas automatiquement ce wrapper personnalisé, ce qui faisait échouer le script sur certaines pages. C'était un piège classique d'« hypothèse évidente » qui nous a rappelé que **nous devons comprendre en profondeur le flux de rendu du framework plutôt que d'appliquer aveuglément des patterns de framework génériques**.

## Solution principale : surcharger le composant Footer

Pour garantir que le script Clarity se charge sur toutes les pages, y compris les docs et les articles de blog, sans casser la structure de page originale, nous avons choisi de surcharger le composant `Footer` de Starlight.

### Pourquoi le Footer ?

1.  **Présence globale** : le Footer apparaît sur presque toutes les pages standard.
2.  **Non intrusif** : placer le script dans la zone du Footer, qui est en fait rendue près du bas du body, ne bloque pas le chemin de rendu critique de la page (LCP), l'impact sur les performances est donc minimal.
3.  **Logique centralisée** : la logique de consentement aux cookies peut être gérée en un seul endroit à l'intérieur du composant.

**Points clés expliqués** :
- `is:inline` : indique à Astro de ne pas traiter le contenu de cette balise script et de le sortir directement dans le HTML. C'est critique pour les scripts d'analytics tiers ; sinon, l'optimisation de bundling d'Astro peut faire en sorte que le script cesse de fonctionner.
- `define:vars` : une fonctionnalité d'Astro 3+ qui permet d'injecter des variables de manière sécurisée dans la portée.
- `import.meta.env.PROD` : garantit qu'aucune analytics inutile n'est générée pendant le développement local, sauf si nécessaire pour le débogage, gardant vos données propres.

## Leçons apprises et pièges à éviter

Lors du déploiement de cette solution dans HagiCode, nous avons résumé plusieurs détails faciles à négliger :

1.  **`StarlightWrapper.astro` est un piège** :
    Comme mentionné précédemment, n'essayez pas de créer un Wrapper global pour injecter le script. Cette approche ne fonctionne pas dans Starlight. La solution correcte est de surcharger un composant spécifique comme `StarlightFooter.astro` ou `StarlightHead.astro`.

2.  **Considérations de performance pour le placement du script** :
    Bien que Clarity recommande de placer le script dans `<head>` pour garantir une précision maximale des données, pour un site de documentation, la vitesse de chargement de l'écran initial (LCP) affecte directement le SEO et la rétention des utilisateurs. Nous avons choisi de le placer dans le Footer, près du bas du body. Cela peut manquer légèrement une petite quantité de données utilisateur de type « rebond en quelques secondes », mais cela nous donne une expérience de chargement de page plus rapide, ce qui en vaut la peine.

3.  **Interférence de l'environnement de développement** :
    Veillez à ajouter la vérification `import.meta.env.PROD`. En mode développement, vous rafraîchirez les pages fréquemment, ce qui générerait sinon une grande quantité de données de test inutiles et polluerait votre tableau de bord Clarity.

## Vérification du résultat

Après le déploiement, vous pouvez consulter les données en temps réel dans la console Clarity. Habituellement en quelques minutes, vous commencerez à voir les heatmaps et les enregistrements des utilisateurs.

Pour HagiCode, ces données nous ont aidés à découvrir que :
- de nombreux utilisateurs revisitent repeatedly la section « Démarrage rapide », suggérant que notre guide d'installation n'est peut-être pas encore assez intuitif.
- la page « Référence API » a le temps de séjour le plus long, confirmant les besoins de notre groupe d'utilisateurs principal.

## Conclusion

L'intégration de Microsoft Clarity ne nécessite pas de modifications complexes côté serveur, ni l'introduction d'un SDK lourd.

En exploitant le mécanisme de surcharge de composants de Starlight, nous avons réalisé une analytics à l'échelle du site avec rien de plus qu'un composant léger `StarlightFooter.astro`. Ce type de « micro-intégration » garde la base de code propre tout en nous donnant la capacité de comprendre le comportement des utilisateurs.

Si vous exploitez également un projet technique, surtout un projet comme **HagiCode** où la documentation doit continuer d'évoluer, je recommande vivement d'essayer Clarity. Les données vous diront où les utilisateurs rencontrent vraiment des difficultés.
