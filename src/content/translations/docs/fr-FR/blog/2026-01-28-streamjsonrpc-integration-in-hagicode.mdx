## Intégration approfondie et utilisation pratique de StreamJsonRpc dans HagiCode

> Cet article présente en détail comment le projet HagiCode (anciennement PCode) a réussi à intégrer la bibliothèque de communication StreamJsonRpc de Microsoft pour remplacer l'implémentation JSON-RPC personnalisée d'origine, et comment les défis techniques et les problèmes d'architecture rencontrés lors de l'intégration ont été résolus.

## Contexte

StreamJsonRpc est une bibliothèque de communication JSON-RPC officiellement maintenue par Microsoft pour .NET et TypeScript, reconnue pour sa forte sécurité de type, sa génération automatique de proxys et son mécanisme mature de gestion des exceptions. Dans le projet HagiCode, afin de communiquer avec des outils IA externes (comme iflow CLI, OpenCode CLI) via l'ACP (Agent Communication Protocol), et d'éliminer les coûts de maintenance et les bugs potentiels de l'implémentation JSON-RPC personnalisée antérieure, le projet a décidé d'intégrer StreamJsonRpc. Cependant, l'intégration a rencontré des défis spécifiques au JSON-RPC en flux continu, en particulier dans le traitement de la liaison de cibles proxy et de l'identification des paramètres génériques.

## Analyse

Le projet se trouve actuellement à un stade clé de l'intégration du protocole ACP, confronté à plusieurs défis techniques et architecturaux :

### 1. Limites de l'implémentation personnalisée
L'implémentation JSON-RPC d'origine se trouve dans `src/HagiCode.ClaudeHelper/AcpImp/` et comprend des composants comme `JsonRpcEndpoint` et `ClientSideConnection`. Le coût de maintenance de ce code personnalisé est élevé et il lui manque les fonctionnalités avancées d'une bibliothèque mature (comme les rapports de progression et le support d'annulation).

### 2. Obstacles à l'intégration de StreamJsonRpc
Lors de la tentative de migration du modèle `CallbackProxyTarget` existant vers StreamJsonRpc, il s'est avéré que la méthode `_rpc.AddLocalRpcTarget(target)` ne pouvait pas identifier les cibles créées via le mode proxy. Plus précisément, StreamJsonRpc ne pouvait pas décomposer automatiquement les propriétés du type générique `T` en paramètres de méthode RPC.

### 3. Confusion dans la stratification de l'architecture
Le `ClientSideConnection` existant mélangeait la couche de transport (WebSocket/Stdio), la couche de protocole (JSON-RPC) et la couche métier (interface ACP Agent), entraînant un manque de clarté des responsabilités.

### 4. Absence de journalisation
La couche de transport WebSocket manquait de sortie de journal pour le contenu JSON brut, rendant difficile le diagnostic des problèmes de communication RPC.

## Solution

Pour résoudre ces problèmes, nous avons adopté une solution systématique, optimisant sur trois dimensions : refactorisation architecturale, intégration de bibliothèque et amélioration du débogage.

### 1. Migration complète vers StreamJsonRpc

#### Suppression de l'ancien code
Suppression de `JsonRpcEndpoint.cs`, `AgentSideConnection.cs` et des convertisseurs de sérialisation personnalisés associés.

#### Intégration de la bibliothèque officielle
Introduction du package NuGet `StreamJsonRpc`, utilisant sa classe `JsonRpc` pour la logique de communication principale.

#### Abstraction de la couche de transport
Définition de l'interface `IAcpTransport` pour unifier les deux modes de transport `WebSocket` et `Stdio`, assurant le découplage entre la couche de protocole et la couche de transport.

```csharp
// Définition de l'interface IAcpTransport
public interface IAcpTransport
{
    Task SendAsync(string message, CancellationToken cancellationToken = default);
    Task<string> ReceiveAsync(CancellationToken cancellationToken = default);
    Task CloseAsync(CancellationToken cancellationToken = default);
}

// Implémentation du transport WebSocket
public class WebSocketTransport : IAcpTransport
{
    private readonly WebSocket _webSocket;

    public WebSocketTransport(WebSocket webSocket)
    {
        _webSocket = webSocket;
    }
}

// Implémentation du transport Stdio
public class StdioTransport : IAcpTransport
{
    private readonly StreamReader _reader;
    private readonly StreamWriter _writer;

    public StdioTransport(StreamReader reader, StreamWriter writer)
    {
        _reader = reader;
        _writer = writer;
    }
}
```

### 2. Correction du problème d'identification des cibles proxy

#### Analyse de `CallbackProxyTarget`
Examen de la logique existante de génération de proxy dynamique pour déterminer la cause fondamentale de l'incapacité de StreamJsonRpc à les identifier.

#### Refactorisation du passage de paramètres
Décomposition des propriétés génériques en paramètres de méthode RPC explicites. Au lieu de dépendre de propriétés dynamiques, nous définissons des DTO Request/Response spécifiques pour que StreamJsonRpc puisse identifier les signatures de méthode par réflexion.

```csharp
// Ancienne approche avec propriétés génériques
public class CallbackProxyTarget<T>
{
    public Func<T, Task> Callback { get; set; }
}

// Nouvelle approche avec méthodes spécifiques
public class ReadTextFileRequest
{
    public string FilePath { get; set; }
}

public class ReadTextFileResponse
{
    public string Content { get; set; }
}

public interface IAcpAgentCallback
{
    Task<ReadTextFileResponse> ReadTextFileAsync(ReadTextFileRequest request);
}
```

### 3. Implémentation de la liaison de méthodes et amélioration de la journalisation

#### Implémentation de `AcpAgentCallbackRpcAdapter`
Assurer que ce composant implémente explicitement l'interface proxy de StreamJsonRpc, mappant les méthodes définies par le protocole ACP aux gestionnaires de callback de StreamJsonRpc.

#### Intégration de la journalisation
Interception et enregistrement du texte brut des requêtes et réponses JSON-RPC dans le pipeline de traitement des messages WebSocket ou Stdio.

```csharp
// Wrapper de transport avec journalisation
public class LoggingAcpTransport : IAcpTransport
{
    private readonly IAcpTransport _innerTransport;
    private readonly ILogger<LoggingAcpTransport> _logger;

    public LoggingAcpTransport(IAcpTransport innerTransport, ILogger<LoggingAcpTransport> logger)
    {
        _innerTransport = innerTransport;
        _logger = logger;
    }

    public async Task SendAsync(string message, CancellationToken cancellationToken = default)
    {
        _logger.LogTrace("Sending message: {Message}", message);
        await _innerTransport.SendAsync(message, cancellationToken);
    }

    public async Task<string> ReceiveAsync(CancellationToken cancellationToken = default)
    {
        var message = await _innerTransport.ReceiveAsync(cancellationToken);
        _logger.LogTrace("Received message: {Message}", message);
        return message;
    }

    public async Task CloseAsync(CancellationToken cancellationToken = default)
    {
        _logger.LogDebug("Closing connection");
        await _innerTransport.CloseAsync(cancellationToken);
    }
}
```

### 4. Refactorisation de la stratification de l'architecture

#### Couche de transport (`AcpRpcClient`)
Encapsulation de la connexion StreamJsonRpc, responsable de `InvokeAsync` et de la gestion du cycle de vie de la connexion.

#### Couche de protocole (`IAcpAgentClient` / `IAcpAgentCallback`)
Définition d'interfaces claires client-vers-agent et agent-vers-client, en supprimant les dépendances circulaires.

## Bonnes pratiques

### 1. DTO fortement typés plutôt que des objets dynamiques
Ne pas utiliser `dynamic` ou `JObject` pour passer des paramètres. Définissez des classes C# POCO spécifiques pour chaque méthode RPC.

### 2. Déclaration explicite des noms de méthode
Utilisez l'attribut `[JsonRpcMethod]` pour spécifier explicitement le nom de la méthode RPC.

```csharp
public interface IAcpAgentCallback
{
    [JsonRpcMethod("readTextFile")]
    Task<ReadTextFileResponse> ReadTextFileAsync(ReadTextFileRequest request);

    [JsonRpcMethod("writeTextFile")]
    Task WriteTextFileAsync(WriteTextFileRequest request);
}
```

### 3. Utiliser les callbacks d'état de connexion
Surveillez l'événement `JsonRpc.ConnectionLost` pour gérer les sorties de processus inattendues ou les déconnexions réseau.

### 4. Journalisation en couches
- **Niveau Trace** : enregistrer le texte JSON Request/Response complet.
- **Niveau Debug** : enregistrer la pile d'appels de méthode et le résumé des paramètres.

### 5. Gérer les spécificités du streaming
StreamJsonRpc prend en charge nativement `IAsyncEnumerable`. Utilisez-le directement pour les réponses Prompt en flux continu de l'ACP.

```csharp
public interface IAcpAgentCallback
{
    [JsonRpcMethod("streamText")]
    IAsyncEnumerable<string> StreamTextAsync(StreamTextRequest request);
}
```

## Conclusion

Grâce à l'intégration complète de StreamJsonRpc, le projet HagiCode a résolu avec succès les problèmes de l'implémentation personnalisée d'origine. Les améliorations clés incluent :

1. Adoption de DTO fortement typés remplaçant les propriétés dynamiques, améliorant la maintenabilité et la fiabilité du code.
2. Implémentation de l'abstraction de la couche de transport et de la séparation de la couche de protocole.
3. Amélioration de la journalisation pour faciliter le diagnostic des problèmes de communication.
4. Introduction du support du streaming pour simplifier l'implémentation du traitement en flux continu.
