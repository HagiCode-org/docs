## Concevoir 12 expériences de démarrage exceptionnelles pour HagiCode : du minimalisme au cyberpunk

> Le bref laps de temps entre le téléchargement d'une application React 19 et la fin de l'hydratation est une opportunité en or pour que les utilisateurs ressentent la personnalité de votre marque. Dans cet article, nous partageons un système complet de styles de démarrage que nous avons construit pour le projet HagiCode en HTML/CSS/JS.

## Contexte

En tant qu'application moderne construite avec ASP.NET Core 10 et React 19 (Vite), HagiCode utilise une architecture de déploiement séparée entre frontend et backend. Le résultat du build frontend est empaqueté dans le répertoire `wwwroot/` du backend et servi par ASP.NET Core.

Cependant, cette architecture introduit un point de douleur UX classique : lorsque les utilisateurs visitent la page, le navigateur charge d'abord le HTML, puis télécharge le gros bundle JS, et enfin laisse React effectuer l'hydratation. Pendant cette « période de vide » qui dure de quelques centaines de millisecondes à plusieurs secondes, les utilisateurs voient soit un écran blanc, soit une page statique sans vie.

Pour combler ce vide et insuffler la personnalité de la marque HagiCode, nous devions concevoir un système de styles de démarrage entièrement implémenté en code inline dans `index.html`.

## À propos de HagiCode

L'approche de conception d'écran de démarrage partagée dans cet article provient de notre expérience pratique dans le projet [HagiCode](https://github.com/HagiCode-org/site). En tant qu'assistant de codage IA, HagiCode se soucie non seulement de l'efficacité de la génération de code, mais aussi de l'expérience visuelle du développeur. Ce système de démarrage est l'un des résultats de notre quête de performances frontend ultimes.

## Défis principaux et conception de l'architecture

Avant de commencer la conception, nous devions d'abord clarifier les contraintes techniques. Puisque tout devait être implémenté en inline dans `index.html`, nous ne pouvions charger aucun fichier CSS ou JS externe en dehors du propre bundle de React.

### Analyse des contraintes techniques

1.  **Principe zéro dépendance** : tous les styles doivent résider dans une balise `<style>`, et toute la logique doit résider dans une balise `<script>`.
2.  **CSS défensif** : pour empêcher les styles globaux de polluer l'écran de démarrage après le montage de l'application React, nous avons décidé d'envelopper tous les styles de démarrage avec un préfixe ID de haute priorité tel que `#boot-screen`.
3.  **Performances d'abord** : les animations devraient utiliser `transform` et `opacity` de CSS autant que possible pour éviter les reflows et garantir que le thread principal reste non bloqué.
4.  **Cohérence visuelle** : les couleurs et les polices doivent rester alignées avec la configuration Tailwind de HagiCode.

### Modèle d'architecture : Shell & Injector

Nous avons adopté un **pattern de variantes**. La logique principale est encapsulée dans une expression de fonction invoquée immédiatement (IIFE), tandis que la logique de rendu spécifique est injectée via la configuration. Cela nous permet de basculer entre différents styles par une simple configuration plutôt que de réécrire la logique de manipulation du DOM à plusieurs reprises.

Voici le code de l'architecture principale :

```html
<!-- Inline in index.html -->
<div id="boot-root"></div>

<script>
(function() {
  const BootSequence = {
    config: {
      theme: 'terminal', // Configurable as 'minimal', 'skeleton', 'code-rain', etc.
      color: '#3b82f6'   // Brand color
    },
    
    // Core lifecycle
    init() {
      this.render();
      this.listenForMount();
    },

    // Render the currently selected style
    render() {
      const root = document.getElementById('boot-root');
      if (this.variants[this.config.theme]) {
        root.innerHTML = this.variants[this.config.theme].render();
      }
    },

    // Listen for successful React mount and exit gracefully
    listenForMount() {
      window.addEventListener('hagicode:ready', () => {
        const screen = document.getElementById('boot-root');
        // Fade out first, then remove the DOM to avoid flicker
        screen.style.opacity = '0';
        screen.style.transition = 'opacity 0.3s ease';
        setTimeout(() => screen.remove(), 300);
      });
    },

    // The implementation logic for all 12 styles lives here
    variants: {
      // ...see details below
    }
  };

  BootSequence.init();
})();
</script>
```

## Inventaire de 12 conceptions de styles de démarrage

Nous avons regroupé ces 12 styles en six grandes catégories pour satisfaire différents scénarios et préférences esthétiques.

### A. Minimalisme

> « Moins, c'est plus. » Pour les scénarios qui recherchent une vitesse de chargement ultime, nous fournissons les options les plus légères possibles.

#### 1. Point minimaliste
Un simple point se trouve au centre de l'écran, associé à une animation de respiration.
*   **Implémentation** : `@keyframes` CSS contrôle l'échelle et l'opacité.
*   **Idéal pour** : tout cas où la page doit rester absolument épurée.

#### 2. Révélation de marque
En utilisant l'animation SVG `stroke-dasharray`, elle simule un dessin à la main des lignes du logo HagiCode, suivi d'un fondu du texte.
*   **Technique** : animation de chemin SVG avec un rendu très soigné.

### B. Imitation d'écran squelette

> « L'art de tromper l'œil. » En simulant une véritable mise en page d'interface, les utilisateurs ont l'impression que la page est déjà à moitié chargée.

#### 3. Écran squelette avec barre latérale et chat
C'est peut-être l'option la plus pratique. Nous avons construit manuellement une mise en page en HTML qui reflète exactement les composants React `Sidebar` et `ChatInput`, puis l'avons recouverte d'une animation de scintillement gris.
*   **Valeur** : lorsque l'hydratation React est terminée, le squelette devient instantanément le vrai composant, et les utilisateurs perçoivent à peine le changement.

#### 4. Écran squelette empilage de cartes
Il simule le mouvement empilé des cartes de proposition pendant le chargement, en utilisant des transformations 3D pour faire flotter subtilement les cartes.

### C. Abstraction et art

> Montrez l'ADN geek de HagiCode.

#### 5. Transformation géométrique
Une forme géométrique (un carré) est rendue au centre de l'écran, puis se transforme au fil du temps en cercle, en triangle, et enfin en logo.
*   **Technologie** : transitions fluides avec `border-radius` CSS.

#### 6. Pluie de code
Un hommage à *Matrix*. En utilisant la police JetBrains Mono, de faibles flux de caractères tombent en arrière-plan.
*   **Note** : pour les performances, les flux de caractères doivent rester dans une zone plus petite ou utiliser un taux de rafraîchissement plus faible.

#### 7. Impulsion néon
Un anneau lumineux de style cyberpunk qui utilise plusieurs couches de `box-shadow` pour créer une puissante lueur néon.

### D. Marque et thèmes

> Donnez vie au système.

#### 8. Thème saisonnier
C'est un chargeur dynamique. Il vérifie la date actuelle pour les jours fériés tels que le Nouvel An lunaire ou Noël et charge l'animation SVG correspondante.
*   **Exemple** : pendant le Nouvel An lunaire, de petites lanternes rouges se balancent doucement en bas de l'écran.

#### 9. Flux de dégradé
L'arrière-plan utilise un dégradé fluide basé sur les couleurs de marque de HagiCode. Combiné avec `background-size` et `background-position` animés, il crée un mouvement semblable à une aurore boréale.

### E. Atmosphère technique

> Un salut aux développeurs.

#### 10. Démarrage terminal
Il simule la sortie de la console. Des lignes de code défilent rapidement :
```text
> Initializing HagiCode Core...
> Loading models...
> Connecting to neural network...
```
C'est instantanément familier pour tout développeur.

#### 11. Barre de progression
Une fine barre de progression apparaît en haut de l'écran, avec un pourcentage affiché à droite. Bien que nous ne puissions pas accéder à la progression réelle du téléchargement, nous pouvons utiliser un minuteur pour simuler un processus de chargement « crédible » : rapide pour les 80 premiers pourcents, puis progressivement plus lent pour les 20 derniers pourcents.

### F. Créativité

#### 12. Assemblage de pixels
C'est une idée très intéressante. De petits carrés sont dispersés sur l'écran, puis convergent vers le centre et s'assemblent progressivement en icône du logo HagiCode. Cela symbolise le processus de construction du code.

## Bonnes pratiques et pièges à éviter

Dans le développement réel de HagiCode, nous avons résumé plusieurs détails d'implémentation critiques.

### 1. Le CSS défensif est obligatoire
Ne soyez jamais paresseux et ne sautez pas le préfixe. Une fois, nous avons oublié de délimiter les styles de l'écran de démarrage avec un ID, et les styles `div` globaux après le montage de React ont affecté de manière inattendue l'écran de démarrage, cassant la mise en page.
**Leçon apprise** : placez chaque sélecteur CSS sous `#boot-screen`, et utilisez `!important` pour augmenter la priorité si nécessaire, mais uniquement dans le CSS de l'écran de démarrage.

### 2. Transitions gracieuses
Après le montage réussi de React, ne supprimez pas directement le DOM de l'écran de démarrage avec `remove()`.
**Approche correcte** :
1.  React déclenche `window.dispatchEvent(new Event('hagicode:ready'))`.
2.  L'écran de démarrage écoute l'événement et définit d'abord `opacity: 0`.
3.  Attendez 300ms, ce qui correspond à la durée de la transition CSS, et appelez `.remove()` uniquement après que l'écran est complètement invisible.

### 3. Synchronisation des variables de thème
Les valeurs de couleur de l'écran de démarrage sont codées en dur dans `index.html`. Si nous changeons la couleur primaire de Tailwind, nous devons aussi mettre à jour l'écran de démarrage.
**Optimisation** : écrivez un simple plugin dans le script de build Vite pour lire `tailwind.config.js` et injecter les variables de couleur dans les variables de modèle `index.html`, créant ainsi une source unique de vérité.

### 4. Préchargement des polices
Les écrans de démarrage ont souvent besoin d'utiliser une police de marque, mais si la police se charge lentement, un FOUT (Flash of Unstyled Text) peut apparaître.
**Solution** : ajoutez `<link rel="preload" href="/fonts/JetBrainsMono.woff2" as="font" type="font/woff2" crossorigin>` dans `<head>`. C'est un moyen peu coûteux et à haut rendement d'améliorer l'expérience.

### 5. Surveillance des performances
Nous avons injecté `performance.mark('boot-start')` en bas de `index.html`, et marqué `boot-end` lorsque React s'est monté avec succès.
**Pourquoi c'est important** : en collectant ces données via Application Insights, nous pouvons mesurer directement combien l'écran de démarrage raccourcit le temps d'attente perçu. Les données montrent qu'un excellent écran squelette peut améliorer la tolérance des utilisateurs pour un « réseau lent » de plus de 50 %.

## Conclusion

Un bon écran de démarrage n'est pas qu'un « décor pendant l'attente ». C'est le signal de poignée de main dans la toute première interaction entre le produit et l'utilisateur. Dans le projet HagiCode, ce système de démarrage basé sur le **pattern de variantes** nous permet de basculer flexiblement les styles lors des fêtes et des releases, améliorant considérablement le côté ludique et le professionnalisme du produit.

La solution partagée dans cet article est entièrement construite sur les standards web natifs sans introduire de dépendances lourdes, ce qui reflète la quête de HagiCode d'être « léger mais puissant ».
