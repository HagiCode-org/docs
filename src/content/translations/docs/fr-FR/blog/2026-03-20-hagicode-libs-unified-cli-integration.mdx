## HagiCode.Libs : pratique d'ingénierie pour l'intégration unifiée de multiples CLIs d'assistants de codage IA

> Pendant le développement du projet HagiCode, nous devions intégrer plusieurs CLIs d'assistants de codage IA en même temps, notamment Claude Code, Codex et CodeBuddy. Chaque CLI a des interfaces, des paramètres et des formats de sortie différents, et le code d'intégration répétitif rendait le projet de plus en plus difficile à maintenir. Dans cet article, nous partageons comment nous avons construit une couche d'abstraction unifiée avec HagiCode.Libs pour résoudre ce problème d'ingénierie. On pourrait aussi dire que c'est simplement de l'expérience durement acquise à travers les pièges que nous avons déjà rencontrés.

## Contexte

Le marché des assistants de codage IA est plutôt animé en ce moment. Outre Claude Code, il y a aussi Codex d'OpenAI, CodeBuddy de Zhipu, et bien d'autres. En tant que projet d'assistant de codage IA, HagiCode doit intégrer ces différents outils CLI à travers plusieurs sous-projets, notamment le bureau, le backend et le web.

Au début, le problème était gérable. Intégrer un CLI ne représentait que quelques centaines de lignes de code. Mais au fur et à mesure que le nombre de CLIs que nous devions supporter ne cessait de croître, les choses ont commencé à devenir chaotiques.

Chaque CLI a son propre format d'arguments en ligne de commande, des exigences variables en matière de variables d'environnement, et une grande variété de formats de sortie. Certains produisent du JSON, d'autres du JSON en flux continu, et d'autres du texte brut. En plus de cela, il y a les problèmes de compatibilité multiplateforme. La découverte des exécutables et la gestion des processus fonctionnent très différemment entre Windows et les systèmes Unix, de sorte que la duplication de code ne cessait d'augmenter. En vérité, c'était devenu ingérable.

## Pourquoi avons-nous besoin d'une abstraction unifiée ?

Le cœur du problème est que chaque CLI a ses propres conventions, et les intégrer directement signifie réécrire essentiellement la même logique encore et encore. Nous avions besoin d'une couche d'abstraction qui :

1. Fournit une interface commune pour tous les CLIs
2. Gère les différences de format de sortie
3. Prend en charge la découverte automatique des exécutables
4. Gère les différences multiplateformes
5. Fournit un mécanisme d'extension pour les nouveaux CLIs

## Architecture de HagiCode.Libs

L'architecture suit un pattern de conception classique : l'interface définit le contrat, et chaque implémentation concrète adapte un CLI spécifique.

```csharp
// Interface commune pour tous les CLIs
public interface IAICliAdapter
{
    string CliName { get; }
    Task<bool> IsAvailableAsync(CancellationToken ct = default);
    Task<AICliResult> ExecuteAsync(AICliRequest request, CancellationToken ct = default);
}
```

Chaque implémentation (ClaudeCodeAdapter, CodexAdapter, CodeBuddyAdapter, etc.) encapsule les spécificités de son CLI respectif.

## Conclusion

HagiCode.Libs résout le problème de l'intégration de multiples CLIs IA en fournissant une couche d'abstraction unifiée. Cela rend le code plus maintenable, plus testable et plus extensible.
