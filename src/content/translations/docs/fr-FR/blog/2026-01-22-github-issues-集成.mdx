## Construire une intégration GitHub Issues de zéro : la pratique de connexion frontend directe de HagiCode

> Cet article documente l'intégration complète de GitHub Issues dans la plateforme HagiCode. Nous explorerons comment utiliser une architecture « connexion frontend directe + backend minimal » pour réaliser une authentification OAuth sécurisée et une synchronisation efficace des Issues, tout en conservant un backend léger.

## Contexte : pourquoi intégrer GitHub ?

En tant que plateforme de développement assisté par IA, la valeur fondamentale de HagiCode réside dans la connexion entre les idées et leur réalisation. Mais dans la pratique, nous avons constaté qu'après avoir complété un Proposal dans HagiCode, les utilisateurs devaient souvent copier manuellement le contenu dans GitHub Issues pour le suivi de projet.

Cela crée plusieurs problèmes évidents :

1.  **Flux de travail fragmenté** : les utilisateurs doivent naviguer constamment entre deux systèmes. L'expérience n'est pas fluide et des informations clés peuvent facilement se perdre lors des copier-coller.
2.  **Collaboration difficile** : les autres membres de l'équipe ont l'habitude de consulter les tâches sur GitHub et ne peuvent pas voir directement l'avancement des propositions dans HagiCode.
3.  **Travail répétitif** : chaque fois qu'une proposition est mise à jour, il faut manuellement mettre à jour l'Issue correspondante sur GitHub, ce qui augmente les coûts de maintenance inutiles.

Pour résoudre ce problème, nous avons décidé d'introduire la fonctionnalité **GitHub Issues Integration**, connectant les sessions HagiCode aux dépôts GitHub pour permettre une « synchronisation en un clic ».

## Choix technique : connexion frontend directe vs proxy backend

Lors de la conception de l'approche d'intégration, deux options s'offraient à nous : le modèle traditionnel de « proxy backend » et le modèle plus audacieux de « connexion frontend directe ».

### Comparaison des solutions

Dans le modèle traditionnel de **proxy backend**, toutes les requêtes du frontend doivent d'abord passer par notre backend, qui appelle ensuite l'API GitHub. Bien que la logique soit centralisée, cela alourdit considérablement le backend :

1.  **Backend gonflé** : il faut écrire un encapsuleur client dédié pour l'API GitHub et gérer la machine à états complexe d'OAuth.
2.  **Risque de Token** : le token GitHub de l'utilisateur doit être stocké dans la base de données du backend. Même avec le chiffrement, cela augmente la surface d'attaque de sécurité.
3.  **Coût de développement** : il faut des migrations de base de données pour stocker les tokens et un service de synchronisation supplémentaire à maintenir.

Le modèle de **connexion frontend directe** est beaucoup plus léger. Dans cette approche, nous n'utilisons le backend que pour l'étape la plus sensible de « l'échange de clés » (le callback OAuth). Une fois le token obtenu, il est stocké directement dans le `localStorage` du navigateur. Les opérations ultérieures telles que la création d'Issues et la mise à jour des commentaires sont envoyées directement du frontend à GitHub via HTTP.

| Dimension de comparaison | Modèle proxy backend | Modèle connexion frontend directe |
| :--- | :--- | :--- |
| **Complexité backend** | Nécessite un service OAuth complet et un client API GitHub | Nécessite uniquement un endpoint de callback OAuth |
| **Gestion des tokens** | Doit être chiffré et stocké en base de données, avec risque de fuite | Stocké dans le navigateur, visible uniquement par l'utilisateur |
| **Coût d'implémentation** | Nécessite des migrations de base de données et le développement de plusieurs services | Principalement du travail frontend |
| **Expérience utilisateur** | Logique centralisée, mais la latence serveur peut être légèrement supérieure | Réponse extrêmement rapide, interaction directe avec GitHub |

Puisque nous recherchions une intégration rapide et des modifications minimales du backend, **nous avons finalement opté pour le modèle de « connexion frontend directe »**. C'est comme donner au navigateur un « passe temporaire » : une fois le passe obtenu, le navigateur peut aller s'occuper des choses sur GitHub par lui-même, sans avoir à demander l'approbation de l'administrateur backend à chaque fois.

## Conception principale : flux de données et sécurité

Après avoir fixé l'architecture, nous devions concevoir le flux de données spécifique. Le cœur du processus de synchronisation réside dans la façon d'obtenir le token de manière sécurisée et de l'utiliser efficacement.

### Diagramme d'architecture globale

L'ensemble du système peut être abstrait en trois acteurs : le navigateur (frontend), le backend HagiCode et GitHub.

```text
+--------------+        +--------------+        +--------------+
|  Frontend    |        |   Backend    |        |   GitHub     |
|  React       |        |   ASP.NET    |        |   REST API   |
|              |        |              |        |              |
|  +--------+  |        |              |        |              |
|  | OAuth  |--+--------> /callback    |        |              |
|  | Flux   |  |        |              |        |              |
|  +--------+  |        |              |        |              |
|              |        |              |        |              |
|  +--------+  |        |  +--------+  |        |  +--------+  |
|  | GitHub |  +------------> Session |  +----------> Issues |  |
|  | API    |  |        |  |Metadata|  |        |  |        |  |
|  | Direct |  |        |  +--------+  |        |  +--------+  |
|  +--------+  |        |              |        |              |
+--------------+        +--------------+        +--------------+
```

**Le point clé est** : seule une petite étape d'OAuth (l'échange du code contre un token) doit passer par le backend. Ensuite, le gros du travail (la création d'Issues) est géré directement entre le frontend et GitHub.

### Flux de données de synchronisation en détail

Lorsque l'utilisateur clique sur le bouton « Sync to GitHub » dans l'interface HagiCode, une série d'actions complexes se déclenche :

```text
L'utilisateur clique sur « Sync to GitHub »
         │
         ▼
1. Le frontend vérifie le localStorage pour le token GitHub
         │
         ▼
2. Formatage du contenu de l'Issue (conversion du Proposal en Markdown)
         │
         ▼
3. Le frontend appelle directement l'API GitHub pour créer/mettre à jour l'Issue
         │
         ▼
4. Appel de l'API backend HagiCode pour mettre à jour Session.metadata (stockage de l'URL de l'Issue, etc.)
         │
         ▼
5. Le backend diffuse l'événement SessionUpdated via SignalR
         │
         ▼
6. Le frontend reçoit l'événement et met à jour l'interface pour afficher l'état « Synchronisé »
```

### Conception de la sécurité

La sécurité est toujours la priorité absolue lors de l'intégration de services tiers. Nous avons pris les précautions suivantes :

1.  **Protection contre les attaques CSRF** : lors de la redirection OAuth, un paramètre `state` aléatoire est généré et stocké dans `sessionStorage`. Le state est vérifié rigoureusement lors du callback pour empêcher les requêtes falsifiées.
2.  **Stockage isolé des tokens** : le token est stocké uniquement dans le `localStorage` du navigateur. Grâce à la politique de même origine (Same-Origin Policy), seuls les scripts HagiCode peuvent le lire, évitant qu'une fuite de base de données côté serveur n'affecte les utilisateurs.
3.  **Gestion des erreurs** : nous avons conçu une gestion dédiée pour les erreurs courantes de l'API GitHub (telles que token expiré 401, échec de validation 422 et limitation de débit 429), offrant ainsi un retour convivial aux utilisateurs.

## En pratique : détails d'implémentation

La théorie ne suffit pas. Voyons comment le code fonctionne réellement.

### 1. Modifications minimales du backend

Le backend n'a besoin de faire que deux choses : stocker les informations de synchronisation et gérer le callback OAuth.

**Modifications de la base de données**
Nous devons simplement ajouter une colonne `Metadata` à la table `Sessions` pour stocker les données d'extension au format JSON.

```sql
-- Ajouter la colonne metadata à la table Sessions
ALTER TABLE "Sessions" ADD COLUMN "Metadata" text NULL;
```

**Définitions des entités et DTO**

### 2. Flux OAuth du frontend

C'est le point d'entrée de la connexion. Nous utilisons le flux d'autorisation standard (Authorization Code Flow).

```typescript
// src/HagiCode.Client/src/services/githubOAuth.ts

// Générer l'URL d'autorisation et rediriger
export async function generateAuthUrl(): Promise<string> {
  const state = generateRandomString(); // Générer une chaîne aléatoire pour la protection CSRF
  sessionStorage.setItem('hagicode_github_state', state);

  const params = new URLSearchParams({
    client_id: clientId,
    redirect_uri: window.location.origin + '/settings?tab=github&oauth=callback',
    scope: ['repo', 'public_repo'].join(' '),
    state: state,
  });

  return `https://github.com/login/oauth/authorize?${params.toString()}`;
}

// Gérer l'échange code-token sur la page de callback
export async function exchangeCodeForToken(code: string, state: string): Promise<GitHubToken> {
  // 1. Valider le state pour empêcher le CSRF
  const savedState = sessionStorage.getItem('hagicode_github_state');
  if (state !== savedState) throw new Error('Invalid state parameter');

  // 2. Appeler l'API backend pour échanger le token
  // Note : cela doit passer par le backend car le ClientSecret ne peut pas être exposé au frontend
  const response = await fetch('/api/GitHubOAuth/callback', {
    method: 'POST',
    headers: { 'Content-Type': 'application/json' },
    body: JSON.stringify({ code, state, redirectUri: window.location.origin + '/settings?tab=github&oauth=callback' }),
  });

  if (!response.ok) throw new Error('Failed to exchange token');

  const token = await response.json();

  // 3. Sauvegarder dans le LocalStorage
  saveToken(token);
  return token;
}
```

### 3. Encapsuleur client de l'API GitHub

Une fois le token obtenu, nous avons besoin d'un outil solide pour appeler l'API GitHub.

```typescript
// src/HagiCode.Client/src/services/githubApiClient.ts

const GITHUB_API_BASE = 'https://api.github.com';

// Encapsuleur de requête principal
async function githubApi<T>(endpoint: string, options: RequestInit = {}): Promise<T> {
  const token = localStorage.getItem('gh_token');
  if (!token) throw new Error('Not connected to GitHub');

  const response = await fetch(`${GITHUB_API_BASE}${endpoint}`, {
    ...options,
    headers: {
      ...options.headers,
      Authorization: `Bearer ${token}`,
      Accept: 'application/vnd.github.v3+json', // Spécifier la version de l'API
    },
  });

  // Logique de gestion des erreurs
  if (!response.ok) {
    if (response.status === 401) throw new Error('Le token GitHub a expiré, veuillez vous reconnecter');
    if (response.status === 403) throw new Error('Pas d\'accès à ce dépôt ou limite de débit dépassée');
    if (response.status === 422) throw new Error('La validation de l\'Issue a échoué, le titre est peut-être en doublon');
    throw new Error(`GitHub API Error: ${response.statusText}`);
  }

  return response.json();
}

// Créer une Issue
export async function createIssue(owner: string, repo: string, data: { title: string, body: string, labels: string[] }) {
  return githubApi(`/repos/${owner}/${repo}/issues`, {
    method: 'POST',
    body: JSON.stringify(data),
  });
}
```

### 4. Formatage du contenu et synchronisation

La dernière étape consiste à convertir les données de la session HagiCode au format d'une Issue GitHub. C'est un peu un travail de « traduction ».

```typescript
// Convertir un objet Session en chaîne Markdown
function formatIssueForSession(session: Session): string {
  let content = `# ${session.title}\n\n`;
  content += `**> HagiCode Session:** #${session.code}\n`;
  content += `**> Status:** ${session.status}\n\n`;
  content += `## Description\n\n${session.description || 'No description provided.'}\n\n`;

  // Si c'est une session de type Proposal, ajouter des champs supplémentaires
  if (session.type === 'proposal') {
    content += `## Chief Complaint\n\n${session.chiefComplaint || ''}\n\n`;
    // Ajouter un lien profond pour permettre de revenir de GitHub vers HagiCode
    content += `---\n\n**[Voir dans HagiCode](hagicode://sessions/${session.id})**\n`;
  }

  return content;
}

// Logique principale lors du clic sur le bouton de synchronisation
const handleSync = async (session: Session) => {
  try {
    const repoInfo = parseRepositoryFromUrl(session.repoUrl); // Analyser l'URL du dépôt
    if (!repoInfo) throw new Error('Invalid repository URL');

    toast.loading('Synchronisation vers GitHub...');

    // 1. Formater le contenu
    const issueBody = formatIssueForSession(session);

    // 2. Appeler l'API
    const issue = await githubApiClient.createIssue(repoInfo.owner, repoInfo.repo, {
      title: `[HagiCode] ${session.title}`,
      body: issueBody,
      labels: ['hagicode', 'proposal', `status:${session.status}`],
    });

    // 3. Mettre à jour les métadonnées de la Session (sauvegarder le lien de l'Issue)
    await SessionsService.patchApiSessionsSessionId(session.id, {
      metadata: {
        githubIssue: {
          owner: repoInfo.owner,
          repo: repoInfo.repo,
          issueNumber: issue.number,
          issueUrl: issue.html_url,
          syncedAt: new Date().toISOString(),
        }
      }
    });

    toast.success('Synchronisation réussie !');
  } catch (err) {
    console.error(err);
    toast.error('Échec de la synchronisation, veuillez vérifier le token ou le réseau');
  }
};
```

## Bilan et perspectives

Grâce à cette approche de « connexion frontend directe », nous avons réalisé l'intégration transparente de GitHub Issues avec un minimum de code backend.

### Acquis
1.  **Haute efficacité de développement** : les modifications du backend sont minimales, principalement un champ de base de données supplémentaire et un simple endpoint de callback OAuth. La majeure partie de la logique est réalisée côté frontend.
2.  **Bonne sécurité** : le token ne transite pas par la base de données du serveur, réduisant le risque de fuite.
3.  **Excellente expérience utilisateur** : les requêtes sont initiées directement depuis le frontend, la vitesse de réponse est rapide et il n'est pas nécessaire de passer par le backend.

### Points de vigilance
Lors du déploiement réel, voici quelques pièges à garder à l'esprit :
- **Paramètres de l'OAuth App** : n'oubliez pas d'entrer le bon `Authorization callback URL` dans les paramètres de votre OAuth App GitHub (généralement `http://localhost:3000/settings?tab=github&oauth=callback`).
- **Limites de débit** : l'API GitHub limite strictement les requêtes non authentifiées, mais avec un token, le quota est généralement suffisant (5000 requêtes/heure).
- **Analyse des URL** : les utilisateurs entrent toutes sortes d'URL de dépôts, assurez-vous que votre expression régulière peut gérer les suffixes `.git`, les formats SSH, etc.

### Améliorations futures
La fonctionnalité actuelle est encore une synchronisation unidirectionnelle (HagiCode -> GitHub). À l'avenir, nous prévoyons de mettre en œuvre une synchronisation bidirectionnelle via les Webhooks GitHub. Par exemple, si une Issue est fermée sur GitHub, l'état de la session côté HagiCode pourrait également se mettre à jour automatiquement. Cela nécessitera d'exposer un endpoint de webhook sur le backend, ce qui sera une étape intéressante à venir.
