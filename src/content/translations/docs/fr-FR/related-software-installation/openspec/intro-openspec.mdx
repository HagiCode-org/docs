# Introduction à OpenSpec

OpenSpec est un outil de développement piloté par des spécifications qui aide les équipes à gérer le processus de développement logiciel via des propositions et des spécifications structurées. Ce guide fournit une introduction complète aux concepts clés, aux flux de travail et à l'utilisation d'OpenSpec.

## Qu'est-ce qu'OpenSpec ?

OpenSpec est un outil en ligne de commande pour gérer les Proposals, Specifications et Changes tout au long du processus de développement logiciel. Il standardise le flux de travail de développement et garantit que chaque fonctionnalité passe par une conception et une révision approfondies, de l'idée à l'implémentation.

### Philosophie centrale

OpenSpec repose sur les principes suivants :

- **Spécification d'abord** : Définir des spécifications claires avant d'écrire du code
- **Collaboration structurée** : Favoriser la collaboration d'équipe via des formats de documents standardisés
- **Traçabilité** : Enregistrer le processus complet de la proposition à l'implémentation
- **Assurance qualité** : Garantir que le code est conforme à l'intention de conception via la validation des spécifications

### Fonctionnalités principales

OpenSpec fournit les fonctionnalités principales suivantes :

1. **Gestion des propositions** : Créer, réviser et suivre les propositions de fonctionnalités
2. **Définition des spécifications** : Rédiger et maintenir des documents de spécification technique
3. **Suivi des changements** : Enregistrer tout l'historique des modifications révisées
4. **Validation des spécifications** : S'assurer que les modifications de code respectent les spécifications prédéfinies
5. **Collaboration d'équipe** : Prise en charge du développement et de la revue collaboratifs

## Flux de travail OpenSpec

Le flux de travail type d'OpenSpec comprend les étapes suivantes :

### 1. Créer une proposition

Lorsqu'une nouvelle fonctionnalité ou un changement majeur est requis, créez d'abord une proposition :

```bash
openspec proposal create
```

Une proposition contient :
- **Description du problème** : Le problème ou le besoin à résoudre
- **Solution** : Aperçu de la solution proposée
- **Portée d'impact** : Impact sur les systèmes existants
- **Alternatives** : Autres solutions envisagées

### 2. Rédiger une spécification

Après approbation de la proposition, rédigez une spécification technique détaillée :

```bash
openspec spec create
```

Une spécification contient :
- **Conception technique** : Plan d'implémentation technique détaillé
- **Définition des interfaces** : API, structures de données et autres définitions d'interface
- **Détails d'implémentation** : Logique d'implémentation clé et algorithmes
- **Plan de tests** : Comment vérifier la correction de l'implémentation

### 3. Créer un changement

Lorsque vous commencez l'implémentation, créez un change pour suivre le processus de développement :

```bash
openspec change create
```

Un change contient :
- **Spécification associée** : La spécification que ce change implémente
- **Tâches d'implémentation** : Liste des tâches de développement spécifiques
- **Suivi de progression** : Statut d'achèvement des tâches
- **Modifications de code** : Modifications réelles du code

### 4. Valider et archiver

Une fois le développement terminé, validez que le change est conforme à la spécification, puis archivez-le :

```bash
openspec change validate
openspec change archive
```

## Structure du répertoire OpenSpec

OpenSpec crée un répertoire `openspec/` à la racine du projet avec la structure suivante :

```
openspec/
├── proposals/           # Répertoire des propositions
│   ├── PROPOSAL-001.md
│   └── PROPOSAL-002.md
├── specs/              # Répertoire des spécifications
│   ├── SPEC-001.md
│   └── SPEC-002.md
├── changes/            # Répertoire des changements
│   ├── feature-name/
│   │   ├── proposal.md
│   │   ├── design.md
│   │   ├── specs.md
│   │   └── tasks.md
│   └── another-feature/
└── archived/           # Changements archivés
    └── feature-name/
```

### Description des répertoires

- **proposals/** : Stocke tous les documents de proposition, un fichier Markdown indépendant par proposition
- **specs/** : Stocke les documents de spécification technique correspondant aux propositions
- **changes/** : Stocke les changements en cours, un sous-répertoire par changement
- **archived/** : Stocke les changements terminés et archivés

## Commandes principales

### Commandes liées aux propositions

```bash
# Créer une nouvelle proposition
openspec proposal create

# Lister toutes les propositions
openspec proposal list

# Voir les détails d'une proposition
openspec proposal view <proposal-id>

# Approuver une proposition
openspec proposal approve <proposal-id>
```

### Commandes liées aux spécifications

```bash
# Créer une nouvelle spécification
openspec spec create

# Lister toutes les spécifications
openspec spec list

# Voir les détails d'une spécification
openspec spec view <spec-id>

# Mettre à jour une spécification
openspec spec update <spec-id>
```

### Commandes liées aux changements

```bash
# Créer un nouveau change
openspec change create

# Lister tous les changes
openspec change list

# Voir les détails d'un change
openspec change view <change-name>

# Ajouter une tâche
openspec change add-task <change-name>

# Valider un change
openspec change validate <change-name>

# Archiver un change
openspec change archive <change-name>
```

### Commandes générales

```bash
# Afficher la version
openspec --version

# Afficher l'aide
openspec --help

# Afficher l'aide pour une commande spécifique
openspec <command> --help
```

## Utilisation d'OpenSpec dans les projets HagiCode

### Intégration au projet

Dans les projets HagiCode, OpenSpec est intégré au flux de travail de développement :

1. **Vérification d'initialisation** : L'assistant de démarrage vérifie si OpenSpec est installé
2. **Développement piloté par proposition** : Toutes les nouvelles fonctionnalités doivent passer par le processus de propositions d'OpenSpec
3. **Revue de code** : Les changements doivent respecter les spécifications OpenSpec pour être fusionnés
4. **Synchronisation de la documentation** : La documentation OpenSpec reste synchronisée avec l'implémentation du code

### Exigences de version

Les projets HagiCode requièrent la plage de versions **OpenSpec >=1.0.0 <2.0.0**.

Commande d'installation :

```bash
npm install -g @fission-ai/openspec@1
```

### Exemple de workflow

Voici un exemple complet d'utilisation d'OpenSpec dans un projet HagiCode :

#### Étape 1 : Créer une proposition

```bash
cd /path/to/hagicode-project
openspec proposal create
```

Saisissez dans l'invite interactive :
- Titre de la proposition : Ajouter la fonctionnalité d'authentification utilisateur
- Description du problème : Le système nécessite une connexion et la gestion des permissions
- Solution : Mettre en œuvre un système d'authentification basé sur JWT

#### Étape 2 : Rédiger une spécification

Après approbation de la proposition, créez une spécification détaillée :

```bash
openspec spec create
```

Définissez dans la spécification :
- Conception du flux d'authentification
- Définition de l'interface API
- Structure des tables de la base de données
- Considérations de sécurité

#### Étape 3 : Créer un change

Commencez l'implémentation :

```bash
openspec change create
```

Entrez le nom du change : `add-user-authentication`

#### Étape 4 : Ajouter des tâches

```bash
openspec change add-task add-user-authentication
```

Ajoutez des tâches spécifiques :
- Implémenter le endpoint d'inscription utilisateur
- Implémenter le endpoint de connexion
- Implémenter le middleware JWT
- Rédiger des tests unitaires

#### Étape 5 : Implémenter et valider

Une fois le code terminé, validez le change :

```bash
openspec change validate add-user-authentication
```

#### Étape 6 : Archiver

Après validation réussie, archivez le change :

```bash
openspec change archive add-user-authentication
```

## Bonnes pratiques

### Rédiger une proposition

1. **Énoncé clair du problème** : Définissez clairement le problème à résoudre et évitez les descriptions vagues
2. **Examiner les alternatives** : Listez et comparez différentes solutions
3. **Évaluer l'impact** : Analysez l'impact et les risques pour les systèmes existants
4. **Rester concis** : Les propositions doivent être succinctes ; les détails appartiennent à la spécification

### Rédiger des spécifications

1. **Détaillé mais non redondant** : Fournissez suffisamment de détails, évitez la suringénierie
2. **Utiliser des diagrammes** : Utilisez des diagrammes de flux, des diagrammes d'architecture, etc. pour clarifier
3. **Définir les interfaces** : Définissez clairement toutes les interfaces publiques et structures de données
4. **Prendre en compte les cas limites** : Énumérez les situations exceptionnelles et la gestion des erreurs

### Gestion des changements

1. **Petites itérations** : Fractionnez les grandes fonctionnalités en plusieurs petits changes
2. **Mises à jour en temps utile** : Maintenez la synchronisation des statuts des tâches et de la documentation
3. **Revue de code** : Effectuez des revues de code après l'achèvement des changes
4. **Archivage en temps opportun** : Archivez les changes terminés pour garder l'arborescence de travail propre

### Collaboration d'équipe

1. **Revues régulières** : Révisez régulièrement les propositions et les spécifications pour assurer un consensus d'équipe
2. **Documentation d'abord** : Complétez la documentation avant de commencer le codage
3. **Maintenir la communication** : Enregistrez les discussions et décisions dans les propositions et spécifications
4. **Contrôle de version** : Incluez les documents OpenSpec dans le contrôle de version

## Cas d'utilisation courants

### Scénario 1 : Ajouter de nouvelles fonctionnalités

1. Créez une proposition décrivant les exigences de la nouvelle fonctionnalité
2. Discutez et approuvez la proposition en équipe
3. Rédigez une spécification technique détaillée
4. Créez un change et implémentez la fonctionnalité
5. Validez et archivez le change

### Scénario 2 : Refactorisation du code existant

1. Créez une proposition expliquant les raisons et objectifs de la refactorisation
2. Définissez le plan et les étapes de refactorisation dans la spécification
3. Créez un change et effectuez la refactorisation par étapes
4. Assurez la couverture de tests et validez les résultats de la refactorisation
5. Archivez le change

### Scénario 3 : Correction de bugs

Pour les bugs simples :
- Ils peuvent être corrigés directement sans passer par l'ensemble du processus OpenSpec

Pour les bugs complexes :
1. Créez une proposition analysant la cause profonde du bug
2. Définissez la correction dans la spécification
3. Créez un change et implémentez la correction
4. Ajoutez des tests pour éviter les régressions
5. Archivez le change

### Scénario 4 : Gestion de la dette technique

1. Créez une proposition listant les éléments de dette technique
2. Définissez un plan d'amélioration dans la spécification
3. Créez des changes pour réduire progressivement la dette
4. Suivez les progrès et l'efficacité
5. Archivez les améliorations terminées

## Intégration avec d'autres outils

### Intégration Git

OpenSpec s'intègre parfaitement à Git :

```bash
# Les documents OpenSpec doivent être inclus dans le contrôle de version
git add openspec/
git commit -m "Add OpenSpec proposal for feature X"
```

### Intégration CI/CD

Validez les spécifications OpenSpec dans vos pipelines CI/CD :

```yaml
# .github/workflows/openspec.yml
name: OpenSpec Validation
on: [pull_request]
jobs:
  validate:
    runs-on: ubuntu-latest
    steps:
      - uses: actions/checkout@v2
      - name: Install OpenSpec
        run: npm install -g @fission-ai/openspec@1
      - name: Validate changes
        run: openspec change validate
```

### Intégration IDE

Éditeurs recommandés avec support Markdown pour éditer les documents OpenSpec :
- VS Code + plugins Markdown
- IDE JetBrains + support Markdown
- Vim/Neovim + plugins Markdown

## Dépannage

### Problèmes courants

#### Problème : commande OpenSpec introuvable

**Solution** :
1. Vérifiez qu'OpenSpec est installé : `npm list -g @fission-ai/openspec`
2. Vérifiez que la variable d'environnement PATH inclut le répertoire bin global de npm
3. Réinstallez : `npm install -g @fission-ai/openspec@1`

#### Problème : impossible de créer des propositions ou des changes

**Solution** :
1. Vérifiez que le répertoire courant est la racine du projet
2. Vérifiez l'existence du répertoire `openspec/`; si absent, créez-le
3. Vérifiez les permissions d'écriture

#### Problème : la validation échoue

**Solution** :
1. Vérifiez que les documents du répertoire change sont complets
2. Assurez-vous que toutes les tâches requises sont accomplies
3. Consultez les erreurs détaillées : `openspec change validate <change-name> --verbose`

#### Problème : l'archivage échoue

**Solution** :
1. Assurez-vous que le change a passé la validation
2. Vérifiez l'existence du répertoire `openspec/archived/`
3. Assurez-vous qu'il n'y a pas de modifications de code non commises

## Usage avancé

### Templates personnalisés

OpenSpec prend en charge les templates personnalisés pour les proposals et les spécifications :

```bash
# Créez un répertoire de templates à la racine du projet
mkdir -p .openspec/templates

# Créez un template de proposition personnalisé
cat > .openspec/templates/proposal.md << 'EOF'
# Proposal: {{title}}

## Problem Statement
{{problem}}

## Proposed Solution
{{solution}}

## Custom Section
Your custom content here
EOF
```

### Fichier de configuration

Créez un fichier de configuration `.openspec.json` à la racine du projet :

```json
{
  "version": "1.0",
  "templates": {
    "proposal": ".openspec/templates/proposal.md",
    "spec": ".openspec/templates/spec.md"
  },
  "validation": {
    "requireTests": true,
    "requireDocs": true
  },
  "hooks": {
    "pre-archive": "npm test"
  }
}
```

### Hooks

OpenSpec prend en charge l'exécution de scripts personnalisés avant et après certaines opérations :

```json
{
  "hooks": {
    "pre-create": "scripts/check-permissions.sh",
    "post-archive": "scripts/notify-team.sh"
  }
}
```
