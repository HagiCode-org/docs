# Вступ до OpenSpec

OpenSpec — це інструмент розробки, керований специфікаціями, який допомагає командам керувати процесом розробки програмного забезпечення через структуровані пропозиції та специфікації. Ця стаття надає всебічний вступ до основних концепцій, робочих процесів та способів використання OpenSpec.

## Що таке OpenSpec?

OpenSpec — це інструмент командного рядка, який використовується для керування пропозиціями (Proposals), специфікаціями (Specifications) та змінами (Changes) у процесі розробки програмного забезпечення. Він стандартизує процес розробки та гарантує, що кожна функція проходить детальне проєктування та перегляд від ідеї до впровадження.

### Основні принципи

OpenSpec базується на таких основних принципах:

- **Специфікації першими**: Визначайте чіткі специфікації перед тим, як писати код
- **Структурована співпраця**: Спрощуйте командну співпрацю за допомогою стандартизованих форматів документів
- **Відстежуваність**: Фіксуйте весь процес від пропозиції до впровадження
- **Забезпечення якості**: Перевіряйте, щоб код відповідав задуму проєкту за допомогою валідації специфікацій

### Основні можливості

OpenSpec надає такі основні можливості:

1. **Керування пропозиціями**: Створюйте, переглядайте та відстежуйте пропозиції функціоналу
2. **Визначення специфікацій**: Пишіть та підтримуйте документи технічних специфікацій
3. **Відстеження змін**: Фіксуйте історію всіх переглянутих змін
4. **Валідація специфікацій**: Переконайтеся, що зміни в коді відповідають заздалегідь визначеним специфікаціям
5. **Командна співпраця**: Підтримуйте процеси розробки та перегляду з участю кількох осіб

## Робочий процес OpenSpec

Типовий робочий процес OpenSpec включає такі етапи:

### 1. Створення пропозиції (Proposal)

Якщо зʼявляється нова функція або вимога до значної зміни, спочатку створюють пропозицію:

```bash
openspec proposal create
```

Пропозиція містить:
- **Опис проблеми**: Проблема або вимога, яку потрібно вирішити
- **Рішення**: Огляд запропонованого рішення
- **Обсяг впливу**: Вплив на існуючу систему
- **Альтернативні рішення**: Інші розглянуті варіанти

### 2. Написання специфікації (Specification)

Після затвердження пропозиції напишіть детальну технічну специфікацію:

```bash
openspec spec create
```

Специфікація містить:
- **Технічний дизайн**: Детальний план технічної реалізації
- **Визначення інтерфейсів**: Дизайн інтерфейсів, таких як API, структури даних тощо
- **Деталі реалізації**: Ключова логіка реалізації та алгоритми
- **План тестування**: Як перевірити правильність реалізації

### 3. Створення зміни (Change)

Коли починається реалізація, створіть зміну для відстеження процесу розробки:

```bash
openspec change create
```

Зміна містить:
- **Повʼязана специфікація**: Специфікація, що реалізується цією зміною
- **Завдання реалізації**: Список конкретних задач розробки
- **Відстеження прогресу**: Статус виконання задач
- **Зміни в коді**: Фактичні модифікації коду

### 4. Валідація та архівування

Після завершення розробки перевірте, чи відповідна зміна відповідає специфікації, а потім архівуйте її:

```bash
openspec change validate
openspec change archive
```

## Структура директорій OpenSpec

OpenSpec створює директорію `openspec/` у корені проекту зі структурою:

```
openspec/
├── proposals/           # Директорія з пропозиціями
│   ├── PROPOSAL-001.md
│   └── PROPOSAL-002.md
├── specs/              # Директорія зі специфікаціями
│   ├── SPEC-001.md
│   └── SPEC-002.md
├── changes/            # Директорія зі змінами
│   ├── feature-name/
│   │   ├── proposal.md
│   │   ├── design.md
│   │   ├── specs.md
│   │   └── tasks.md
│   └── another-feature/
└── archived/           # Архівовані зміни
    └── feature-name/
```

### Пояснення директорій

- **proposals/**: Зберігає всі документи з пропозиціями, кожна пропозиція — у окремому Markdown-файлі
- **specs/**: Зберігає документацію технічних специфікацій, що відповідають пропозиціям
- **changes/**: Зберігає поточні зміни, кожна зміна у власній піддиректорії
- **archived/**: Зберігає завершені та архівовані зміни

## Основні команди

### Команди, повʼязані з пропозиціями

```bash
# Створити нову пропозицію
openspec proposal create

# Перелік усіх пропозицій
openspec proposal list

# Переглянути деталі пропозиції
openspec proposal view <proposal-id>

# Затвердити пропозицію
openspec proposal approve <proposal-id>
```

### Команди, повʼязані зі специфікаціями

```bash
# Створити нову специфікацію
openspec spec create

# Перелік усіх специфікацій
openspec spec list

# Переглянути деталі специфікації
openspec spec view <spec-id>

# Оновити специфікацію
openspec spec update <spec-id>
```

### Команди, повʼязані зі змінами

```bash
# Створити нову зміну
openspec change create

# Перелік усіх змін
openspec change list

# Переглянути деталі зміни
openspec change view <change-name>

# Додати завдання
openspec change add-task <change-name>

# Валідувати зміну
openspec change validate <change-name>

# Архівувати зміну
openspec change archive <change-name>
```

### Загальні команди

```bash
# Переглянути версію
openspec --version

# Переглянути довідку
openspec --help

# Переглянути довідку для конкретної команди
openspec <command> --help
```

## Використання OpenSpec у проекті HagiCode

### Інтеграція з проектом

У проекті HagiCode OpenSpec інтегрований у робочий процес розробки:

1. **Перевірка при ініціалізації**: Майстер запуску перевіряє, чи встановлено OpenSpec
2. **Розробка, керована пропозиціями**: Усі нові функції повинні пройти процес пропозицій OpenSpec
3. **Перегляд коду**: Зміни повинні відповідати специфікаціям OpenSpec, щоб бути злитими
4. **Синхронізація документації**: Документацію OpenSpec підтримують синхронізованою з реалізацією коду

### Вимоги до версії

Проект HagiCode вимагає використання версії **OpenSpec >=1.0.0 {'<'}2.0.0**.

Команда встановлення:

```bash
npm install -g @fission-ai/openspec@1
```

### Приклад робочого процесу

Нижче наведено повний приклад використання OpenSpec у проекті HagiCode:

#### Крок 1: Створити пропозицію

```bash
cd /path/to/hagicode-project
openspec proposal create
```

У інтерактивному запиті введіть:
- Назва пропозиції: Додати функціонал автентифікації користувача
- Опис проблеми: Системі потрібна автентифікація користувача і управління правами доступу
- Рішення: Реалізувати систему автентифікації на основі JWT

#### Крок 2: Написати специфікацію

Після затвердження пропозиції створіть детальну специфікацію:

```bash
openspec spec create
```

Визначте у специфікації:
- Дизайн процесу автентифікації
- Визначення API
- Структура таблиць бази даних
- Питання безпеки

#### Крок 3: Створити зміну

Почніть реалізацію:

```bash
openspec change create
```

Введіть назву зміни: `add-user-authentication`

#### Крок 4: Додати завдання

```bash
openspec change add-task add-user-authentication
```

Додайте конкретні завдання:
- Реалізувати endpoint для реєстрації користувача
- Реалізувати endpoint для входу
- Реалізувати JWT middleware
- Написати unit-тести

#### Крок 5: Реалізація та валідація

Після завершення коду валідуйте зміну:

```bash
openspec change validate add-user-authentication
```

#### Крок 6: Архівування

Після успішної валідації архівуйте зміну:

```bash
openspec change archive add-user-authentication
```

## Найкращі практики

### Написання пропозицій

1. **Чітке формулювання проблеми**: Ясно визначте проблему для вирішення; уникайте невизначених описів
2. **Розглянути альтернативи**: Перерахуй та порівняй різні рішення
3. **Оцінити вплив**: Проаналізуй вплив і ризики для наявної системи
4. **Будьте лаконічними**: Пропозиції повинні бути короткими та зрозумілими; деталі залиште в специфікації

### Написання специфікацій

1. **Детальні, але не надлишкові**: Надайте достатньо деталей, уникайте over-design
2. **Використовуйте діаграми**: Використовуйте блок-схеми, архітектурні діаграми тощо для ілюстрації
3. **Визначте інтерфейси**: Чітко визначте всі публічні інтерфейси та структури даних
4. **Розгляньте крайні випадки**: Перерахуйте виняткові ситуації та обробку помилок

### Управління змінами

1. **Невеликі ітерації**: Розділіть великі функції на декілька невеликих змін
2. **Оновлюйте вчасно**: Підтримуйте стан завдань і документацію синхронізованими
3. **Перегляд коду**: Проведіть code review після завершення зміни
4. **Архівуйте вчасно**: Архівуйте завершені зміни вчасно і підтримуйте робочий каталог чистим

### Командна співпраця

1. **Регулярні перегляди**: Регулярно переглядайте пропозиції та специфікації для забезпечення консенсусу в команді
2. **Документація перш за все**: Завершіть документацію перед початком кодування
3. **Підтримуйте комунікацію**: Фіксуйте дискусії та рішення в пропозиціях та специфікаціях
4. **Контроль версій**: Включіть OpenSpec-документацію до системи контролю версій

## Звичайні сценарії використання

### Сценарій 1: Додавання нової функції

1. Створіть пропозицію, що описує вимоги до нової функції
2. Команда обговорює та затверджує пропозицію
3. Напишіть детальну технічну специфікацію
4. Створіть зміну та реалізуйте функцію
5. Валідуйте та архівуйте зміну

### Сценарій 2: Рефакторинг існуючого коду

1. Створіть пропозицію, яка пояснює причини та цілі рефакторингу
2. Визначте план та кроки рефакторингу у специфікації
3. Створіть зміну і рефакторуйте поступово
4. Забезпечте тестове покриття, валідуйте результати рефакторингу
5. Архівуйте зміну

### Сценарій 3: Виправлення помилки

Для простих помилок:
- Можна виправити напряму, без повного процесу OpenSpec

Для складних помилок:
1. Створіть пропозицію для аналізу кореневої причини помилки
2. Визначіть рішення у специфікації
3. Створіть зміну та реалізуйте виправлення
4. Додайте тести для уникнення регресій
5. Архівуйте зміну

### Сценарій 4: Управління технічним боргом

1. Створіть пропозицію, що перераховує елементи технічного боргу
2. Визначте план покращення у специфікації
3. Створіть зміну для поступового погашення технічного боргу
4. Відслідковуйте прогрес та вплив
5. Архівуйте завершені покращення

## Інтеграція з іншими інструментами

### Інтеграція з Git

OpenSpec інтегрується без проблем з Git:

```bash
# Документація OpenSpec має бути включена в контроль версій
git add openspec/
git commit -m "Add OpenSpec proposal for feature X"
```

### Інтеграція з CI/CD

Валідуйте специфікації OpenSpec у CI/CD процесі:

```yaml
# .github/workflows/openspec.yml
name: OpenSpec Validation
on: [pull_request]
jobs:
  validate:
    runs-on: ubuntu-latest
    steps:
      - uses: actions/checkout@v2
      - name: Install OpenSpec
        run: npm install -g @fission-ai/openspec@1
      - name: Validate changes
        run: openspec change validate
```

### Інтеграція з IDE

Рекомендується використовувати редактор з підтримкою Markdown для редагування документів OpenSpec:
- VS Code + Markdown плагін
- JetBrains IDEs + підтримка Markdown
- Vim/Neovim + Markdown плагін

## Вирішення проблем

### Поширені проблеми

#### Проблема: Команду OpenSpec не знайдено

**Рішення**:
1. Переконайтеся, що OpenSpec встановлено: `npm list -g @fission-ai/openspec`
2. Перевірте, чи змінна середовища PATH містить глобальний npm bin каталог
3. Перевстановіть: `npm install -g @fission-ai/openspec@1`

#### Проблема: Неможливо створити пропозицію або зміну

**Рішення**:
1. Переконайтеся, що поточний каталог — корінь проекту
2. Перевірте наявність директорії `openspec/`; якщо її немає — створіть її
3. Переконайтеся, що у вас є права на запис

#### Проблема: Валідація не вдалася

**Рішення**:
1. Перевірте, чи документація в каталозі змін повна
2. Переконайтеся, що всі необхідні завдання виконані
3. Перегляньте докладну інформацію про помилку: `openspec change validate <change-name> --verbose`

#### Проблема: Архівування не вдалося

**Рішення**:
1. Переконайтеся, що зміна пройшла валідацію
2. Перевірте наявність директорії `openspec/archived/`
3. Переконайтеся, що немає невідправлених змін у коді

## Просунуте використання

### Кастомні шаблони

OpenSpec підтримує кастомні шаблони для пропозицій та специфікацій:

```bash
# Створіть каталог шаблонів у корені проекту
mkdir -p .openspec/templates

# Створіть кастомний шаблон пропозиції
cat > .openspec/templates/proposal.md << 'EOF'
# Proposal: {{title}}

## Problem Statement
{{problem}}

## Proposed Solution
{{solution}}

## Custom Section
Your custom content here
EOF
```

### Файл конфігурації

Створіть конфігураційний файл `.openspec.json` у корені проекту:

```json
{
  "version": "1.0",
  "templates": {
    "proposal": ".openspec/templates/proposal.md",
    "spec": ".openspec/templates/spec.md"
  },
  "validation": {
    "requireTests": true,
    "requireDocs": true
  },
  "hooks": {
    "pre-archive": "npm test"
  }
}
```

### Hooks

OpenSpec підтримує виконання кастомних скриптів до та після певних операцій:

```json
{
  "hooks": {
    "pre-create": "scripts/check-permissions.sh",
    "post-archive": "scripts/notify-team.sh"
  }
}
```
