## Повний посібник із аналізу повідомлень консолі Codex SDK

> У цій статті пояснюється механізм потоку подій Codex SDK, аналіз типів повідомлень і найкращі практики в реальних проектах, допомагаючи розробникам швидко освоїти основні навички, що стоять за службами виконання ШІ.

## Фон

Створюючи службу виконання штучного інтелекту на основі Codex SDK, ми неминуче стикаємося з практичним питанням: як нам обробляти потокові повідомлення про події, які повертає Codex? Ці повідомлення містять важливу інформацію, таку як стан виконання, вихідний вміст і деталі помилки, тому вони заслуговують на обережне поводження.

У рамках проекту HagiCode нам потрібен був надійний виконавець сценаріїв помічника кодування ШІ. Саме тому ми вирішили детально вивчити механізм потоку подій Codex SDK. Зрештою, лише зрозумівши, як працюють базові повідомлення, ми зможемо побудувати платформу виконання ШІ справді корпоративного рівня.

Codex SDK — це пакет SDK для програмування, випущений OpenAI. Він повертає результати виконання через потік подій. На відміну від традиційної моделі запит-відповідь, Codex використовує потокові події, щоб ми могли:

- Отримуйте хід виконання в реальному часі
- Оперативно виправляйте помилки
- Отримати детальну статистику використання токенів
- Підтримуйте довгострокові складні завдання

Розуміння цих типів подій і їх правильний аналіз є важливими для реалізації повноцінного виконавця ШІ. Зрештою, ніхто не хоче працювати з чорним ящиком.

## Про HagiCode

Рішення, наведене в цій статті, походить з нашого практичного досвіду в [HagiCode](https://hagicode.com) проект. HagiCode — це проект помічника в кодуванні штучного інтелекту з відкритим кодом, призначений для надання розробникам підтримки інтелектуального кодування. Під час розробки нам потрібно було створити надійну службу виконання AI для обробки запитів на виконання коду користувача, що є прямою причиною, чому ми представили Codex SDK.

Як помічник кодування штучного інтелекту, HagiCode має працювати з різноманітними складними сценаріями виконання коду: отримувати хід виконання в режимі реального часу, оперативно обробляти помилки та збирати детальну статистику використання маркерів. Глибоко розуміючи механізм потоку подій Codex SDK, ми можемо створити виконавець, який відповідає вимогам виробничого середовища. Зрештою, будь то програмне забезпечення чи реальне життя, усе виграє від постійного накопичення та вдосконалення.

## Механізм потоку подій

### Основні поняття

Codex SDK використовує `thread.runStreamed()` метод для повернення асинхронного ітератора події:

```typescript
import { Codex } from '@openai/codex-sdk';

const client = new Codex({
  apiKey: process.env.CODEX_API_KEY,
  baseUrl: process.env.CODEX_BASE_URL,
});

const thread = client.startThread({
  workingDirectory: '/path/to/project',
  skipGitRepoCheck: false,
});

const { events } = await thread.runStreamed('your prompt here', {
  outputSchema: {
    type: 'object',
    properties: {
      output: { type: 'string' },
      status: { type: 'string', enum: ['ok', 'action_required'] },
    },
    required: ['output', 'status'],
  },
});

for await (const event of events) {
  // Handle each event
}
```

### Детальне пояснення типів подій

| Тип події | опис | Ключові дані |
|---------|------|----------|
| `thread.started` | Потік успішно розпочато | `thread_id` |
| `item.updated` | Вміст повідомлення оновлено | `item.text` |
| `item.completed` | Повідомлення завершено | `item.text` |
| `turn.completed` | Виконання завершено | `usage` (використання маркера) |
| `turn.failed` | Виконання не вдалося | `error.message` |
| `error` | Подія помилки | `message` |

У реальних проектах компонент виконавця HagiCode побудовано на основі цих типів подій. Нам потрібно ретельно обробляти кожну подію, щоб забезпечити безперебійну роботу користувачів. Хороші системи будуються, серйозно ставлячись до деталей.

## Реалізація розбору повідомлень

### Витяг вмісту повідомлення

Вміст повідомлення витягується через обробник подій:

```typescript
private handleThreadEvent(event: ThreadEvent, onMessage: (content: string) => void): void {
  // Only handle message update and completion events
  if (event.type !== 'item.updated' && event.type !== 'item.completed') {
    return;
  }

  // Only handle agent message content
  if (event.item.type !== 'agent_message') {
    return;
  }

  // Extract text content
  onMessage(event.item.text);
}
```

Ключові моменти:
- Тільки ручка `item.updated` і `item.completed` події
- Обробляти лише вміст типу `agent_message`
- Зміст повідомлення знаходиться в `event.item.text` поле

### Розбір структурованого виводу

Codex підтримує структурований вихід JSON. Ви можете вказати формат повернення через `outputSchema` параметр:

```typescript
const DEFAULT_OUTPUT_SCHEMA = {
  type: 'object',
  properties: {
    output: { type: 'string' },
    status: { type: 'string', enum: ['ok', 'action_required'] },
  },
  required: ['output', 'status'],
  additionalProperties: false,
} as const;
```

Функція аналізу намагається проаналізувати JSON, і якщо це не вдається, вона повертається до необробленого тексту.

```typescript
function toStructuredOutput(raw: string): StructuredOutput {
  try {
    const parsed = JSON.parse(raw) as Partial<StructuredOutput>;
    if (typeof parsed.output === 'string') {
      return {
        output: parsed.output,
        status: parsed.status === 'action_required' ? 'action_required' : 'ok',
      };
    }
  } catch {
    // JSON parsing failed, fall back to the raw text
  }

  return {
    output: raw,
    status: 'ok',
  };
}
```

### Повний процес обробки подій

```typescript
private async runWithStreaming(
  thread: Thread,
  input: CodexStageExecutionInput
): Promise<{ output: string; usage: Usage | null }> {
  const abortController = new AbortController();
  const timeoutHandle = setTimeout(() => {
    abortController.abort();
  }, Math.max(1000, input.timeoutMs));

  let latestMessage = '';
  let usage: Usage | null = null;
  let emittedLength = 0;

  try {
    const { events } = await thread.runStreamed(input.prompt, {
      outputSchema: DEFAULT_OUTPUT_SCHEMA,
      signal: abortController.signal,
    });

    for await (const event of events) {
      // Handle message content
      this.handleThreadEvent(event, (nextContent) => {
        const delta = nextContent.slice(emittedLength);
        if (delta.length > 0) {
          emittedLength = nextContent.length;
          input.callbacks?.onChunk?.(delta);  // Streaming callback
        }
        latestMessage = nextContent;
      });

      // Process different data based on the event type
      if (event.type === 'thread.started') {
        this.threadId = event.thread_id;
      } else if (event.type === 'turn.completed') {
        usage = event.usage;
      } else if (event.type === 'turn.failed') {
        throw new CodexExecutorError('gateway_unavailable', event.error.message, true);
      } else if (event.type === 'error') {
        throw new CodexExecutorError('gateway_unavailable', event.message, true);
      }
    }
  } catch (error) {
    if (abortController.signal.aborted) {
      throw new CodexExecutorError(
        'upstream_timeout',
        `Codex stage timed out after ${input.timeoutMs}ms`,
        true
      );
    }
    throw error;
  } finally {
    clearTimeout(timeoutHandle);
  }

  const structured = toStructuredOutput(latestMessage);
  return { output: structured.output, usage };
}
```

## Стратегія обробки помилок

### Відображення коду помилки

Зіставте конкретні шаблони помилок на конкретні коди помилок, щоб верхні рівні могли легше з ними впоратися:

```typescript
function mapError(error: unknown): CodexExecutorError {
  if (error instanceof CodexExecutorError) {
    return error;
  }

  const message = error instanceof Error ? error.message : String(error);
  const normalized = message.toLowerCase();

  // Authentication errors - not retryable
  if (normalized.includes('401') ||
      normalized.includes('403') ||
      normalized.includes('api key') ||
      normalized.includes('auth')) {
    return new CodexExecutorError('auth_invalid', message, false);
  }

  // Rate limit errors - retryable
  if (normalized.includes('429') || normalized.includes('rate limit')) {
    return new CodexExecutorError('rate_limited', message, true);
  }

  // Timeout errors - retryable
  if (normalized.includes('timeout') || normalized.includes('aborted')) {
    return new CodexExecutorError('upstream_timeout', message, true);
  }

  // Default error
  return new CodexExecutorError('gateway_unavailable', message, true);
}
```

### Визначення типів помилок

```typescript
export type CodexErrorCode =
  | 'auth_invalid'      // Authentication failure
  | 'upstream_timeout'  // Upstream timeout
  | 'rate_limited'      // Rate limited
  | 'gateway_unavailable'; // Gateway unavailable

export class CodexExecutorError extends Error {
  readonly code: CodexErrorCode;
  readonly retryable: boolean;

  constructor(code: CodexErrorCode, message: string, retryable: boolean) {
    super(message);
    this.name = 'CodexExecutorError';
    this.code = code;
    this.retryable = retryable;
  }
}
```

## Конфігурація робочого каталогу та середовища

### Перевірка робочого каталогу

Codex SDK вимагає, щоб робочий каталог був дійсним репозиторієм Git.

```typescript
export function validateWorkingDirectory(
  workingDirectory: string,
  skipGitRepoCheck: boolean
): void {
  const resolvedWorkingDirectory = path.resolve(workingDirectory);

  if (!existsSync(resolvedWorkingDirectory)) {
    throw new CodexExecutorError(
      'gateway_unavailable',
      'Working directory does not exist.',
      false
    );
  }

  if (!statSync(resolvedWorkingDirectory).isDirectory()) {
    throw new CodexExecutorError(
      'gateway_unavailable',
      'Working directory is not a directory.',
      false
    );
  }

  if (skipGitRepoCheck) {
    return;
  }

  const gitDir = path.join(resolvedWorkingDirectory, '.git');
  if (!existsSync(gitDir)) {
    throw new CodexExecutorError(
      'gateway_unavailable',
      'Working directory is not a git repository.',
      false
    );
  }
}
```

### Завантаження змінних середовища

Codex SDK має завантажити змінні середовища з оболонки входу, щоб агент AI міг отримати доступ до системних команд:

```typescript
function parseEnvironmentOutput(output: Buffer): Record<string, string> {
  const parsed: Record<string, string> = {};

  for (const entry of output.toString('utf8').split('\0')) {
    if (!entry) continue;

    const separatorIndex = entry.indexOf('=');
    if (separatorIndex <= 0) continue;

    const key = entry.slice(0, separatorIndex);
    const value = entry.slice(separatorIndex + 1);
    if (key.length > 0) {
      parsed[key] = value;
    }
  }

  return parsed;
}

function tryLoadEnvironmentFromShell(shellPath: string): Record<string, string> | null {
  const result = spawnSync(shellPath, ['-ilc', 'env -0'], {
    env: process.env,
    stdio: ['ignore', 'pipe', 'pipe'],
    timeout: 5000,
  });

  if (result.error || result.status !== 0) {
    return null;
  }

  return parseEnvironmentOutput(result.stdout);
}

export function createExecutorEnvironment(
  envOverrides: Record<string, string> = {}
): Record<string, string> {
  // Load environment variables from the login shell
  const consoleEnv = loadConsoleEnvironmentFromShell();

  return {
    ...process.env,
    ...consoleEnv,
    ...envOverrides,
  };
}
```

## Повний приклад використання

### Основне використання

У проекті HagiCode ми використовуємо такий підхід для ініціалізації клієнта Codex і виконання завдань:

```typescript
import { Codex } from '@openai/codex-sdk';

async function executeWithCodex(prompt: string, workingDir: string) {
  const client = new Codex({
    apiKey: process.env.CODEX_API_KEY,
    env: { PATH: process.env.PATH },
  });

  const thread = client.startThread({
    workingDirectory: workingDir,
  });

  const { events } = await thread.runStreamed(prompt);

  let result = '';
  for await (const event of events) {
    if (event.type === 'item.updated' && event.item.type === 'agent_message') {
      result = event.item.text;
    }
    if (event.type === 'turn.completed') {
      console.log('Token usage:', event.usage);
    }
  }

  // Try to parse JSON output
  try {
    const parsed = JSON.parse(result);
    return parsed.output;
  } catch {
    return result;
  }
}
```

### Повне впровадження з повторними спробами

```typescript
export class CodexSdkExecutor {
  private readonly config: CodexRuntimeConfig;
  private readonly client: Codex;
  private threadId: string | null = null;

  async executeStage(input: CodexStageExecutionInput): Promise<CodexStageExecutionResult> {
    const maxAttempts = Math.max(1, this.config.retryCount + 1);
    let attempt = 0;
    let lastError: CodexExecutorError | null = null;

    while (attempt < maxAttempts) {
      attempt += 1;

      try {
        const thread = this.getThread(input.workingDirectory);
        const { output, usage } = await this.runWithStreaming(thread, input);

        return {
          output,
          usage,
          threadId: this.threadId!,
          attempts: attempt,
          latencyMs: Date.now() - startedAt,
        };
      } catch (error) {
        const mappedError = mapError(error);
        lastError = mappedError;

        // Non-retryable error or max retry attempts reached
        if (!mappedError.retryable || attempt >= maxAttempts) {
          throw mappedError;
        }

        // Wait before retrying
        await new Promise(resolve => setTimeout(resolve, 1000 * attempt));
      }
    }

    throw lastError!;
  }
}
```

## Найкращі практики

### 1. Вимоги до робочого каталогу

- Переконайтеся, що робочий каталог є дійсним сховищем Git
- Використовуйте `PROJECT_ROOT` змінна середовища, щоб вказати її явно
- Під час налагодження можна встановити `CODEX_SKIP_GIT_REPO_CHECK=true` щоб пропустити перевірку

### 2. Конфігурація змінної середовища

- Передайте лише необхідні змінні середовища через механізм білого списку
- Використовуйте оболонку входу, щоб завантажити повне середовище
- Уникайте передачі конфіденційної інформації

### 3. Тайм-аут і стратегія повторної спроби

- Встановіть розумні тайм-аути залежно від складності завдання
- Застосуйте експоненціальне відставання для повторних помилок
- Записуйте кількість повторних спроб і причини

### 4. Обробка помилок

- Розрізняйте повторні та неповторні помилки
- Надайте чіткі повідомлення про помилки та пропозиції
- Використовуйте уніфіковані коди помилок, щоб верхні рівні могли їх узгоджено обробляти

### 5. Потоковий вихід

- Застосуйте зворотні виклики інкрементного виведення, щоб покращити взаємодію з користувачем
- Правильно обробляйте інкрементні оновлення повідомлень
- Записуйте використання токенів для аналізу витрат
