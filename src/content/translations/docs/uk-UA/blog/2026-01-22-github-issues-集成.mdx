## Створення GitHub Проблеми з інтеграцією з нуля: практика прямого з’єднання інтерфейсу HagiCode

> У цій статті описано повний процес інтеграції GitHub Issues у платформу HagiCode. Ми розглянемо, як використовувати архітектуру «пряме з’єднання інтерфейсу + мінімальний сервер» для досягнення безпечної автентифікації OAuth і ефективної синхронізації проблем, зберігаючи легкий сервер.

## Довідкова інформація: навіщо інтегрувати GitHub?

Основна цінність HagiCode як платформи розробки за допомогою штучного інтелекту полягає в поєднанні ідей із реалізацією. Але під час фактичного використання ми виявили, що після того, як користувачі заповнюють Пропозицію в HagiCode, їм часто потрібно вручну скопіювати вміст у GitHub Issues для відстеження проекту.

Це створює кілька очевидних больових точок:

1.  **Фрагментований робочий процес**: користувачам потрібно перемикатися між двома системами. Робота не є гладкою, і ключову інформацію можна легко втратити під час копіювання та вставлення.
2.  **Незручна співпраця**: інші учасники команди звикли перевіряти завдання на GitHub і не можуть безпосередньо бачити прогрес пропозиції в HagiCode.
3.  **Повторна ручна робота**: кожного разу, коли пропозиція оновлюється, хтось повинен вручну оновлювати відповідну проблему на GitHub, додаючи непотрібні витрати на обслуговування.

Щоб вирішити цю проблему, ми вирішили представити функцію **GitHub Issues Integration**, з’єднуючи сеанси HagiCode зі сховищами GitHub, щоб увімкнути «синхронізацію одним клацанням миші».

## Технічний вибір: пряме з’єднання переднього інтерфейсу проти проксі-сервера

Розробляючи підхід до інтеграції, ми мали два варіанти: традиційну «модель серверного проксі» та більш агресивну «модель прямого з’єднання з інтерфейсом».

### Рішення Порівняння

У традиційній **моделі бекенд-проксі** кожен запит із інтерфейсу має спочатку проходити через наш бекенд, який потім викликає API GitHub. Це централізує логіку, але також створює значне навантаження на серверну частину:

1.  **Роздутий бекенд**: нам потрібно буде написати спеціальну оболонку клієнта GitHub API, а також обробляти складну машину стану OAuth.
2.  **Ризик маркера**: Маркер GitHub користувача повинен зберігатися у серверній базі даних. Навіть із шифруванням це все одно підвищує рівень безпеки.
3.  **Вартість розробки**: нам знадобиться міграція бази даних для зберігання токенів і додаткова служба синхронізації для підтримки.

**Модель із прямим підключенням** набагато легша. У цьому підході ми використовуємо серверну частину лише для найбільш чутливого етапу «секретного обміну» (зворотний виклик OAuth). Після отримання токена ми зберігаємо його безпосередньо в браузері `localStorage`. Подальші операції, такі як створення проблем і оновлення коментарів, надсилаються безпосередньо з інтерфейсу на GitHub через HTTP.

| Вимір порівняння | Модель серверного проксі | Модель прямого з’єднання інтерфейсу |
| :--- | :--- | :--- |
| **Складність серверної частини** | Потрібна повна служба OAuth і клієнт API GitHub | Потрібна лише кінцева точка зворотного виклику OAuth |
| **Керування маркерами** | Має бути зашифровано та збережено в базі даних із ризиком витоку | Зберігається в браузері та видимий лише для користувача |
| **Вартість впровадження** | Потрібна міграція бази даних і мультисервісна розробка | В основному фронтенд робота |
| **Досвід користувача** | Централізована логіка, але затримка сервера може бути трохи вищою | Надзвичайно швидка відповідь завдяки прямій взаємодії GitHub |

Оскільки ми хотіли швидкої інтеграції та мінімальних змін серверної частини, **зрештою ми обрали «модель прямого з’єднання з інтерфейсом»**. Це як надання браузеру "тимчасового пропуску". Отримавши дозвіл, браузер може самостійно керувати справами на GitHub, не запитуючи кожного разу схвалення у адміністратора серверної частини.

## Основний дизайн: потік даних і безпека

Визначившись з архітектурою, нам потрібно було спроектувати конкретний потік даних. Суть процесу синхронізації полягає в тому, як безпечно отримати маркер і ефективно його використовувати.

### Загальна схема архітектури

Всю систему можна розділити на три ролі: браузер (фронтенд), бекенд HagiCode і GitHub.

```text
+--------------+        +--------------+        +--------------+
| Frontend     |        |   Backend    |        |   GitHub     |
| React        |        |   ASP.NET    |        |   REST API   |
|              |        |              |        |              |
|  +--------+  |        |              |        |              |
|  | OAuth  |--+--------> /callback    |        |              |
|  | Flow   |  |        |              |        |              |
|  +--------+  |        |              |        |              |
|              |        |              |        |              |
|  +--------+  |        |  +--------+  |        |  +--------+  |
|  | GitHub |  +------------> Session |  +----------> Issues |  |
|  | API    |  |        |  |Metadata|  |        |  |        |  |
|  | Direct |  |        |  +--------+  |        |  +--------+  |
|  +--------+  |        |              |        |              |
+--------------+        +--------------+        +--------------+
```

**Ключовий момент**: лише один маленький крок в OAuth (обмін коду на маркер) повинен пройти через серверну частину. Після цього важка робота (створення проблем) обробляється безпосередньо між інтерфейсом і GitHub.

### Потік даних синхронізації в деталях

Коли користувач натискає кнопку «Синхронізувати з GitHub» в інтерфейсі користувача HagiCode, виконується ряд складних дій:

```text
User clicks "Sync to GitHub"
         │
         ▼
1. Frontend checks localStorage for the GitHub token
         │
         ▼
2. Format issue content (convert the Proposal into Markdown)
         │
         ▼
3. Frontend directly calls the GitHub API to create/update the issue
         │
         ▼
4. Call the HagiCode backend API to update Session.metadata (store issue URL and other info)
         │
         ▼
5. Backend broadcasts the SessionUpdated event via SignalR
         │
         ▼
6. Frontend receives the event and updates the UI to show the "Synced" state
```

### Дизайн безпеки

Безпека завжди є головним пріоритетом під час інтеграції сторонніх служб. Ми врахували наступне:

1.  **Захист від атак CSRF**: генеруйте випадковий `state` під час переспрямування OAuth і збережіть його в `sessionStorage`. Суворо перевіряйте стан у зворотному виклику, щоб запобігти підробленим запитам.
2.  **Ізольоване сховище маркерів**: маркер зберігається лише в браузері `localStorage`. Завдяки політиці Same-Origin тільки сценарії HagiCode можуть читати його, уникаючи ризику витоку бази даних на стороні сервера, що впливає на користувачів.
3.  **Межі помилок**: ми розробили спеціальну обробку поширених помилок GitHub API (таких як 401 прострочений маркер, 422 помилка перевірки та 429 обмеження частоти), щоб користувачі отримували дружні відгуки.

### 2. Інтерфейсний потік OAuth

Це точка входу підключення. Ми використовуємо стандартний код авторизації.

```typescript
// src/HagiCode.Client/src/services/githubOAuth.ts

// Generate the authorization URL and redirect
export async function generateAuthUrl(): Promise<string> {
  const state = generateRandomString(); // Generate a random string for CSRF protection
  sessionStorage.setItem('hagicode_github_state', state);
  
  const params = new URLSearchParams({
    client_id: clientId,
    redirect_uri: window.location.origin + '/settings?tab=github&oauth=callback',
    scope: ['repo', 'public_repo'].join(' '),
    state: state,
  });
  
  return `https://github.com/login/oauth/authorize?${params.toString()}`;
}

// Handle the code-to-token exchange on the callback page
export async function exchangeCodeForToken(code: string, state: string): Promise<GitHubToken> {
  // 1. Validate state to prevent CSRF
  const savedState = sessionStorage.getItem('hagicode_github_state');
  if (state !== savedState) throw new Error('Invalid state parameter');

  // 2. Call the backend API to exchange the token
  // Note: this must go through the backend because ClientSecret cannot be exposed to the frontend
  const response = await fetch('/api/GitHubOAuth/callback', {
    method: 'POST',
    headers: { 'Content-Type': 'application/json' },
    body: JSON.stringify({ code, state, redirectUri: window.location.origin + '/settings?tab=github&oauth=callback' }),
  });

  if (!response.ok) throw new Error('Failed to exchange token');
  
  const token = await response.json();
  
  // 3. Save into LocalStorage
  saveToken(token);
  return token;
}
```

### 3. GitHub API Client Wrapper

Коли ми отримаємо токен, нам знадобиться надійний інструмент для виклику API GitHub.

```typescript
// src/HagiCode.Client/src/services/githubApiClient.ts

const GITHUB_API_BASE = 'https://api.github.com';

// Core request wrapper
async function githubApi<T>(endpoint: string, options: RequestInit = {}): Promise<T> {
  const token = localStorage.getItem('gh_token');
  if (!token) throw new Error('Not connected to GitHub');
  
  const response = await fetch(`${GITHUB_API_BASE}${endpoint}`, {
    ...options,
    headers: {
      ...options.headers,
      Authorization: `Bearer ${token}`,
      Accept: 'application/vnd.github.v3+json', // Specify the API version
    },
  });
  
  // Error handling logic
  if (!response.ok) {
    if (response.status === 401) throw new Error('GitHub token expired, please reconnect');
    if (response.status === 403) throw new Error('No permission to access this repository or rate limit exceeded');
    if (response.status === 422) throw new Error('Issue validation failed, the title may be duplicated');
    throw new Error(`GitHub API Error: ${response.statusText}`);
  }
  
  return response.json();
}

// Create issue
export async function createIssue(owner: string, repo: string, data: { title: string, body: string, labels: string[] }) {
  return githubApi(`/repos/${owner}/${repo}/issues`, {
    method: 'POST',
    body: JSON.stringify(data),
  });
}
```

### 4. Форматування та синхронізація вмісту

Останнім кроком є перетворення даних сеансу HagiCode у формат випуску GitHub. Це трохи схоже на перекладацьку роботу.

```typescript
// Convert a Session object into a Markdown string
function formatIssueForSession(session: Session): string {
  let content = `# ${session.title}\n\n`;
  content += `**> HagiCode Session:** #${session.code}\n`;
  content += `**> Status:** ${session.status}\n\n`;
  content += `## Description\n\n${session.description || 'No description provided.'}\n\n`;
  
  // If this is a Proposal session, add extra fields
  if (session.type === 'proposal') {
    content += `## Chief Complaint\n\n${session.chiefComplaint || ''}\n\n`;
    // Add a deep link so users can jump back from GitHub to HagiCode
    content += `---\n\n**[View in HagiCode](hagicode://sessions/${session.id})**\n`;
  }
  
  return content;
}

// Main logic when clicking the sync button
const handleSync = async (session: Session) => {
  try {
    const repoInfo = parseRepositoryFromUrl(session.repoUrl); // Parse the repository URL
    if (!repoInfo) throw new Error('Invalid repository URL');

    toast.loading('Syncing to GitHub...');
    
    // 1. Format content
    const issueBody = formatIssueForSession(session);
    
    // 2. Call API
    const issue = await githubApiClient.createIssue(repoInfo.owner, repoInfo.repo, {
      title: `[HagiCode] ${session.title}`,
      body: issueBody,
      labels: ['hagicode', 'proposal', `status:${session.status}`],
    });
    
    // 3. Update Session Metadata (save the issue link)
    await SessionsService.patchApiSessionsSessionId(session.id, {
      metadata: {
        githubIssue: {
          owner: repoInfo.owner,
          repo: repoInfo.repo,
          issueNumber: issue.number,
          issueUrl: issue.html_url,
          syncedAt: new Date().toISOString(),
        }
      }
    });

    toast.success('Synced successfully!');
  } catch (err) {
    console.error(err);
    toast.error('Sync failed, please check your token or network');
  }
};
```

## Резюме та прогноз

Завдяки такому підходу «прямого підключення до зовнішнього інтерфейсу» ми досягли бездоганної інтеграції GitHub Issues із мінімальною кількістю коду внутрішньої частини.

### Ключові переваги
1.  **Висока ефективність розробки**: зміни в серверній частині мінімальні, переважно одне додаткове поле бази даних і проста кінцева точка зворотного виклику OAuth. Більшість логіки завершено на інтерфейсі.
2.  **Надійна безпека**: маркер не проходить через базу даних сервера, що зменшує ризик витоку.
3.  **Чудова взаємодія з користувачем**: запити ініціюються безпосередньо з інтерфейсу, тому швидкість відповіді висока, і немає потреби у переадресації на сервері.

### Речі, на які варто звернути увагу
Є кілька підводних каменів, про які слід пам’ятати під час реального розгортання:
- **Налаштування програми OAuth**: не забувайте вводити правильні дані `Authorization callback URL` у налаштуваннях програми GitHub OAuth (зазвичай `http://localhost:3000/settings?tab=github&oauth=callback`).
- **Обмеження швидкості**: API GitHub досить суворо обмежує неавтентифіковані запити, але з токеном квоти зазвичай достатньо (5000 запитів на годину).
- **Аналіз URL-адреси**: користувачі вводять усі типи URL-адрес сховища, тому переконайтеся, що ваш регулярний вираз збігається `.git` суфікси, формати SSH і подібні випадки.

### Майбутні вдосконалення
Поточна функція все ще одностороння синхронізація (HagiCode -> GitHub). У майбутньому ми плануємо реалізувати двосторонню синхронізацію через GitHub Webhooks. Наприклад, якщо проблему закрито на GitHub, стан сеансу на стороні HagiCode також може автоматично оновлюватися. Це вимагатиме від нас виставити кінцеву точку вебхуку на серверній частині, що буде цікавим наступним кроком.
