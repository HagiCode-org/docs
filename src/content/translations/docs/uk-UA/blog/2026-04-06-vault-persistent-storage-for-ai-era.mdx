## Як відтворити проекти в епоху штучного інтелекту: Vault, міжпроектна система постійного зберігання

> Як ми можемо допомогти помічникам зі штучним інтелектом краще зрозуміти наші навчальні ресурси в епоху розвитку за допомогою ШІ? Проект HagiCode створив систему Vault як уніфікований рівень абстракції для зберігання знань, який може зрозуміти штучний інтелект, що значно покращує ефективність навчання завдяки відтворенню проекту.

## Фон

В епоху ШІ спосіб, у який розробники вивчають нові технології та архітектури, кардинально змінюється. «Відтворення проектів» — тобто глибоке вивчення коду, архітектури та шаблонів дизайну чудових проектів із відкритим кодом і навчання на них — стало ефективним способом навчання. У порівнянні з традиційними методами, такими як читання книг або перегляд відео, безпосереднє читання та запуск високоякісних проектів з відкритим кодом допомагає вам набагато швидше зрозуміти реальні інженерні практики.

Однак цей метод навчання пов’язаний із чимало труднощів.

Навчальні матеріали надто розпорошені. Ваші нотатки можуть зберігатися в Obsidian, сховища кодів можуть бути розкидані по різних папках, а історія розмов вашого AI-помічника стає ще одним ізольованим островом даних. Якщо ви хочете, щоб штучний інтелект допоміг проаналізувати проект, вам доведеться вручну копіювати фрагменти коду та впорядковувати контекст, що досить виснажливо.

Що ще більше турбує, так це порушений контекст. Помічники зі штучним інтелектом не можуть отримати прямий доступ до ваших локальних навчальних ресурсів, тому ви повинні знову надавати довідкову інформацію під час кожної розмови. Крім того, репозиторії відтвореного коду швидко оновлюються, ручна синхронізація є схильною до помилок, а знаннями важко поділитися між кількома навчальними проектами.

У корені всі ці проблеми походять від «острівців даних». Якби існував уніфікований абстрактний рівень зберігання, який дозволяв би помічникам зі штучним інтелектом розуміти та отримувати доступ до всіх ваших навчальних ресурсів, проблема була б чітко вирішена.

## Філософія проектування системи сховища

Основна ідея системи Vault проста: створити уніфікований рівень абстракції для зберігання знань, зрозумілий ШІ. З точки зору реалізації, система має кілька ключових характеристик.

### Багатотипова підтримка

Система підтримує чотири типи сховищ, кожен з яких відповідає різному сценарію використання:

```typescript
// folder: general-purpose folder type
export const DEFAULT_VAULT_TYPE = 'folder';

// coderef: a type specifically for reproduced code projects
export const CODEREF_VAULT_TYPE = 'coderef';

// obsidian: integrated with Obsidian note-taking software
export const OBSIDIAN_VAULT_TYPE = 'obsidian';

// system-managed: vault automatically managed by the system
export const SYSTEM_MANAGED_VAULT_TYPE = 'system-managed';
```

Серед них тип **coderef** є найбільш часто використовуваним у HagiCode. Він спеціально розроблений для проектів із відтвореним кодом, забезпечуючи стандартизовану структуру каталогів і описи метаданих, які читаються ШІ.

### Механізм постійного зберігання

Реєстр Vault постійно зберігається у форматі JSON, гарантуючи, що конфігурація залишається доступною після перезапуску програми:

```csharp
public class VaultRegistryStore : IVaultRegistryStore
{
    private readonly string _registryFilePath;

    public VaultRegistryStore(IConfiguration configuration, ILogger<VaultRegistryStore> logger)
    {
        var dataDir = configuration["DataDir"] ?? "./data";
        var absoluteDataDir = Path.IsPathRooted(dataDir)
            ? dataDir
            : Path.GetFullPath(Path.Combine(Directory.GetCurrentDirectory(), dataDir));

        _registryFilePath = Path.Combine(absoluteDataDir, "personal-data", "vaults", "registry.json");
    }
}
```

Перевага такої конструкції в тому, що вона проста і надійна. JSON читається людиною, що полегшує налагодження та редагування вручну; Зберігання файлової системи дозволяє уникнути складності бази даних і зменшити системні залежності. Зрештою, іноді найпростіший варіант дійсно є найкращим.

### Інтеграція контексту ШІ

Найважливіше те, що система може автоматично вводити інформацію про сховище в контекст пропозицій ШІ:

```typescript
export function buildTargetVaultsText(
  vaults: VaultForText[],
  template: VaultPromptTemplate = DEFAULT_VAULT_PROMPT_TEMPLATE,
): string {
  const readOnlyVaults = vaults.filter((vault) => vault.accessType === 'read');
  const editableVaults = vaults.filter((vault) => vault.accessType === 'write');

  if (readOnlyVaults.length === 0 && editableVaults.length === 0) {
    return '';
  }

  const sections = [
    buildVaultSection(readOnlyVaults, template.reference),
    buildVaultSection(editableVaults, template.editable),
  ].filter(Boolean);

  return `\n\n### ${template.heading}\n\n${sections.join('\n')}`;
}
```

Це забезпечує важливу можливість: помічники штучного інтелекту можуть автоматично розуміти доступні навчальні ресурси без користувача вручну вводити контекст. Можна сказати, що це вважається різновидом мовчазного розуміння.

## Стандартизована структура CodeRef Vault

Для типу сховища coderef HagiCode надає стандартизовану структуру каталогів:

```
my-coderef-vault/
├── index.yaml          # vault metadata description
├── AGENTS.md           # operating guide for AI assistants
├── docs/               # stores study notes and documents
└── repos/              # manages reproduced code repositories through Git submodules
```

Під час створення сховища система автоматично ініціалізує цю структуру:

```csharp
private async Task EnsureCodeRefStructureAsync(
    string vaultName,
    string physicalPath,
    ICollection<VaultBootstrapDiagnosticDto> diagnostics,
    CancellationToken cancellationToken)
{
    Directory.CreateDirectory(physicalPath);

    var indexPath = Path.Combine(physicalPath, CodeRefIndexFileName);
    var docsPath = Path.Combine(physicalPath, CodeRefDocsDirectoryName);
    var reposPath = Path.Combine(physicalPath, CodeRefReposDirectoryName);

    // Create the standard directory structure
    if (!Directory.Exists(docsPath))
    {
        Directory.CreateDirectory(docsPath);
    }

    if (!Directory.Exists(reposPath))
    {
        Directory.CreateDirectory(reposPath);
    }

    // Create the AGENTS.md guide
    await EnsureCodeRefAgentsDocumentAsync(physicalPath, cancellationToken);

    // Create the index.yaml metadata
    await WriteCodeRefIndexDocumentAsync(indexPath, mergedDocument, cancellationToken);
}
```

Ця структура також ретельно розроблена:

- **docs/** зберігає ваші навчальні нотатки, де ви можете записати своє розуміння коду, аналіз архітектури, отримані уроки тощо в Markdown
- **repos/** керує відтвореними репозиторіями через підмодулі Git замість того, щоб копіювати код безпосередньо, що підтримує синхронізацію коду та економить місце
- **index.yaml** містить метадані сховища, щоб помічники ШІ могли швидко зрозуміти призначення та вміст сховища
- **AGENTS.md** – це посібник, написаний спеціально для помічників ШІ, який пояснює, як працювати з вмістом сховища

Організований таким чином, можливо, ШІ зможе трохи легше зрозуміти, що ви маєте на увазі.

## Автоматична ініціалізація для керованих системою сховищ

Окрім створених вручну сховищ, HagiCode також підтримує сховища, керовані системою:

```csharp
public async Task<IReadOnlyList<VaultRegistryEntry>> EnsureAllSystemManagedVaultsAsync(
    CancellationToken cancellationToken = default)
{
    var definitions = GetAllResolvedDefinitions();
    var entries = new List<VaultRegistryEntry>(definitions.Count);

    foreach (var definition in definitions)
    {
        entries.Add(await EnsureResolvedSystemManagedVaultAsync(definition, cancellationToken));
    }

    return entries;
}
```

Система автоматично створює та керує такими сховищами:

- **hagiprojectdata**: сховище даних проекту, яке використовується для збереження конфігурації та стану проекту
- **personaldata**: сховище персональних даних, яке використовується для збереження налаштувань користувача
- **hbsprompt**: бібліотека шаблонів підказок, яка використовується для керування типовими підказками AI

Ці сховища ініціалізуються автоматично під час запуску системи, тому користувачам не потрібно налаштовувати їх вручну. Деякі речі просто краще залишити системі, а не людям турбуватися про них.

## Механізм контролю доступу

Важливою частиною дизайну є контроль доступу. Система поділяє сховища на два типи доступу:

```typescript
export interface VaultForText {
  id: string;
  name: string;
  type: string;
  physicalPath: string;
  accessType: 'read' | 'write';  // Key: distinguish read-only from editable
}
```

- **довідка (тільки для читання)**: AI використовується лише для аналізу та розуміння та не може змінювати вміст. Підходить для посилань на проекти з відкритим кодом, документи та подібні матеріали
- **редагований (з можливістю редагування)**: штучний інтелект може змінювати вміст за потреби для виконання завдання. Підходить для ваших нотаток, чернеток і подібних матеріалів

Ця відмінність має значення. Він повідомляє штучному інтелекту, який вміст є «посиланням лише для читання», а який — «безпечним для редагування», зменшуючи ризик випадкових змін. Зрештою, ніхто не хоче, щоб його важка робота зникла через ненавмисне редагування.

## На практиці: створення та використання Vault

Тепер, коли ми розглянули ідеї, давайте подивимося, як це працює на практиці.

### Створіть сховище CodeRef

Ось повний приклад зовнішнього виклику:

```typescript
const createCodeRefVault = async () => {
  const response = await VaultService.postApiVaults({
    requestBody: {
      name: "React Learning Vault",
      type: "coderef",
      physicalPath: "/Users/developer/vaults/react-learning",
      gitUrl: "https://github.com/facebook/react.git"
    }
  });

  // The system will automatically:
  // 1. Clone the React repository into vault/repos/react
  // 2. Create the docs/ directory for notes
  // 3. Generate the index.yaml metadata
  // 4. Create the AGENTS.md guide file

  return response;
};
```

Цей виклик API завершує серію дій: створення структури каталогів, ініціалізація підмодулів Git, генерація файлів метаданих тощо. Вам потрібно надати лише основну інформацію та дозволити системі впоратися з рештою. Чесно кажучи, це досить безтурботний підхід.

Система автоматично вводить інформацію зі сховища в контекст ШІ, повідомляючи ШІ, які навчальні ресурси доступні. Коли штучний інтелект може зрозуміти, що ви маєте на увазі, такого мовчазного розуміння важко досягти.

## Найкращі практики та речі, на які варто звернути увагу

Використовуючи систему Vault, ми узагальнили кілька отриманих уроків.

### Безпека шляху

Система суворо перевіряє шляхи, щоб запобігти атакам обходу шляху:

```csharp
private static string ResolveFilePath(string vaultRoot, string relativePath)
{
    var rootPath = EnsureTrailingSeparator(Path.GetFullPath(vaultRoot));
    var combinedPath = Path.GetFullPath(Path.Combine(rootPath, relativePath));
    if (!combinedPath.StartsWith(rootPath, StringComparison.OrdinalIgnoreCase))
    {
        throw new BusinessException(VaultRelativePathTraversalCode,
            "Vault file paths must stay inside the registered vault root.");
    }
    return combinedPath;
}
```

Це важливо. Якщо ви налаштовуєте шлях до сховища, переконайтеся, що він залишається в межах дозволеного діапазону, інакше система відхилить операцію. Ви дійсно не можете переоцінити безпеку.

### Керування підмодулем Git

CodeRef Vault рекомендує підмодулі Git замість прямого копіювання коду:

```csharp
private static string BuildCodeRefAgentsContent()
{
    return """
    # CodeRef Vault Guide

    Repositories under `repos/` should be maintained through Git submodules
    rather than copied directly into the vault root.

    Keep this structure stable so assistants and tools can understand the vault quickly.
    """ + Environment.NewLine;
}
```

Це дає кілька переваг: синхронізація коду з попереднім потоком, економія місця на диску та спрощення керування кількома версіями коду. Зрештою, хто хоче завантажувати те саме знову і знову?

### Обмеження попереднього перегляду файлів

Щоб уникнути проблем з продуктивністю, система обмежує розмір і тип файлу:

```csharp
private const int FileEnumerationLimit = 500;
private const int PreviewByteLimit = 256 * 1024;  // 256KB
```

Якщо ваше сховище містить велику кількість файлів або дуже великі файли, продуктивність попереднього перегляду може вплинути. У цьому випадку ви можете розглянути можливість обробки файлів пакетами або за допомогою спеціальних інструментів пошуку. Іноді, коли щось стає занадто великим, це стає важче, а не легше.

### Діагностична інформація

Під час створення сховища система повертає діагностичну інформацію, щоб допомогти з налагодженням:

```csharp
List<VaultBootstrapDiagnosticDto> bootstrapDiagnostics = [];

if (IsCodeRefVaultType(normalizedType))
{
    bootstrapDiagnostics = await EnsureCodeRefBootstrapAsync(
        normalizedName,
        normalizedPhysicalPath,
        normalizedGitUrl,
        cancellationToken);
}
```

Якщо створити помилку, ви можете переглянути діагностичну інформацію, щоб зрозуміти конкретну причину. Коли щось йде не так, перевірка діагностики часто є найпрямішим шляхом.

## Резюме

За допомогою уніфікованого рівня абстракції сховища система Vault вирішує кілька основних проблемних моментів відтворення проектів в епоху ШІ:

- **Централізоване управління знаннями**: усі навчальні ресурси зібрані в одному місці, а не розкидані скрізь
- **Автоматичне введення контексту AI**: помічники AI можуть автоматично розуміти доступні навчальні ресурси без налаштування контексту вручну
- **Повторне використання міжпроектних знань**: знання можна поділитися та повторно використовувати в кількох навчальних проектах
- **Стандартизована структура каталогів**: узгоджене розташування каталогів спрощує навчання

Це рішення вже перевірено в проекті HagiCode. Якщо ви також створюєте інструменти, пов’язані з розробкою за допомогою штучного інтелекту, або стикаєтеся з подібними проблемами управління знаннями, я сподіваюся, що цей досвід може стати корисною довідкою.

Насправді цінність технічного рішення полягає не в тому, наскільки воно складне, а в тому, чи вирішує воно реальні проблеми. Основна ідея системи Vault дуже проста: створити уніфікований рівень зберігання знань, який може зрозуміти ШІ. Проте саме ця проста абстракція значно підвищила ефективність нашої розробки.

Іноді простий підхід дійсно є найкращим. Адже складні речі часто приховують ще більше підводних каменів...
