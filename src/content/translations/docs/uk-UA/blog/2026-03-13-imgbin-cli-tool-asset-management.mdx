## Дизайн інструменту ImgBin CLI: підхід HagiCode до керування ресурсами зображень

> У цій статті пояснюється, як створити автоматизований конвеєр ресурсів зображень з нуля, охоплюючи дизайн інструменту CLI, архітектуру адаптера постачальника та стратегії керування метаданими.

## Фон

Чесно кажучи, я не очікував, що управління іміджевими активами буде так довго заплутувати нас.

Під час розробки HagiCode ми зіткнулися з проблемою, яка виглядала простою на поверхні, але була напрочуд складною на практиці: генерування ресурсів зображення та керування ними. У певному сенсі це було схоже на драми підліткового віку — спокійно зовні, бурхливе всередині.

Оскільки проект накопичував більше документації та маркетингових матеріалів, нам знадобилася велика кількість допоміжних зображень. Деякі мали бути створені ШІ, деякі мали бути обрані з існуючої бібліотеки ресурсів, а інші потребували розпізнавання ШІ та автоматичного маркування. Проблема полягала в тому, що все це довгий час оброблялося за допомогою розрізнених сценаріїв і вручну. Щоразу, коли ми створювали зображення, нам доводилося вручну запускати сценарій, вручну впорядковувати метадані та вручну створювати мініатюри. Одне це вже досить дратувало, але більшою проблемою було те, що все було розкидано всюди. Коли ми хотіли щось знайти, ми не могли. Коли нам потрібно було щось використати повторно, ми не могли.

Больові точки були конкретними:

1. **Немає уніфікованої точки входу**: логіка генерації зображень була розподілена між різними сценаріями, тому пакетне виконання було практично неможливим.
2. **Відсутні метадані**: створені зображення не були уніфіковані `metadata.json`, що означало відсутність надійного пошуку чи відстеження.
3. **Високі витрати на організацію вручну**: заголовки та теги потрібно було сортувати один за одним вручну, що було неефективно.
4. **Без автоматизації**: автоматичне створення візуальних ресурсів у конвеєрі CI/CD? Немає шансів.

Ми думали просто залишити це в спокої. Але проекти ще мають рухатися вперед. Оскільки ми не могли уникнути проблеми, ми вирішили, що також могли б її вирішити. Тому ми вирішили оновити ImgBin із набору розрізнених сценаріїв у конвеєр ресурсів зображень, який можна виконувати автоматично. Зрештою, деякі проблеми не зникають лише тому, що ви відводите погляд.

## Про HagiCode

Підхід, опублікований у цій статті, базується на нашому практичному досвіді в [HagiCode](https://hagicode.com) проект. HagiCode — це проект помічника в кодуванні штучного інтелекту, який одночасно обслуговує кілька компонентів, включаючи розширення VSCode, серверні служби штучного інтелекту та кросплатформний настільний клієнт. У такому складному, багатомовному, міжплатформному середовищі, як це, стандартизоване керування ресурсами зображень стає ключовою частиною підвищення ефективності розробки.

Можна сказати, що це був один із тих невеликих труднощів розвитку HagiCode. У кожному проекті є такі моменти: незначна проблема, яка виглядає несуттєвою, але якимось чином займає половину дня.

Система збирання HagiCode базується на екосистемі TypeScript + Node.js, тому ImgBin, природно, прийняв той самий стек технологій, щоб підтримувати технічну послідовність проекту. Коли ви звикли до одного стека, перемикання на щось інше просто здається непотрібним.

## Основний дизайн

### Загальна архітектура

ImgBin використовує багаторівневу архітектуру, яка чітко відокремлює команди CLI, служби додатків, сторонні адаптери API та рівень інфраструктури:

```
Component hierarchy
├── CLI Entry (cli.ts)              Global argument parsing, command routing
├── Commands (commands/*)           generate | batch | annotate | thumbnail
├── Application Services            job-runner | metadata | thumbnail | asset-writer
├── Provider Adapters               image-api-provider | vision-api-provider
└── Infrastructure Layer            config | logger | paths | schema
```

Перевагою цього багаторівневого дизайну є чіткі межі відповідальності. Це також спрощує тестування, оскільки зовнішні залежності можна чітко імітувати. На практиці це просто означає, що кожен шар виконує свою роботу, не заважаючи іншим, тому, коли щось ламається, легше зрозуміти, чому.

### Модель каталогу з одним ресурсом

ImgBin використовує модель «один ресурс, один каталог». Кожного разу, коли створюється зображення, воно створює таку структуру:

```
library/
└── 2026-03/
    └── orange-dashboard/
        ├── original.png      # Original image
        ├── thumbnail.webp    # 512x512 thumbnail
        └── metadata.json     # Structured metadata
```

Перевагами цієї моделі є:

1. **Автономний**: усі файли для одного активу знаходяться в одному каталозі, що робить міграцію та резервне копіювання зручними.
2. **Відстежується**: `metadata.json` дозволяє відстежувати час генерації, підказку, модель та інші деталі.
3. **Розширюваний**: якщо пізніше знадобляться додаткові варіанти, наприклад мініатюри різних розмірів, ми можемо просто додати нові файли в той самий каталог.

Красивими речами не завжди потрібно володіти. Іноді достатньо, щоб вони залишалися красивими, і щоб ви могли їх спокійно оцінити. Це може здатися трохи далеким, але логіка все ще актуальна: коли зображення зберігаються разом, на них приємніше дивитися, і їх набагато легше знайти.

### Багатошарове зберігання метаданих

`metadata.json` є ядром усієї системи. Він використовує багатошарову стратегію зберігання, яка розділяє поля на три категорії:

```json
{
  "schemaVersion": 2,
  "assetId": "orange-dashboard",
  "slug": "orange-dashboard",
  "title": "Orange Dashboard",
  "tags": ["dashboard", "hero", "orange"],

  "source": { "type": "generated" },

  "paths": {
    "assetDir": "library/2026-03/orange-dashboard",
    "original": "original.png",
    "thumbnail": "thumbnail.webp"
  },

  "generated": {
    "prompt": "orange dashboard for docs hero",
    "provider": "azure-openai-image-api",
    "model": "gpt-image-1.5"
  },

  "recognized": {
    "title": "Orange Dashboard",
    "tags": ["dashboard", "ui", "orange"],
    "description": "A modern orange dashboard with charts and metrics"
  },

  "status": {
    "generation": "succeeded",
    "recognition": "succeeded",
    "thumbnail": "succeeded"
  },

  "timestamps": {
    "createdAt": "2026-03-11T04:01:19.570Z",
    "updatedAt": "2026-03-11T04:02:09.132Z"
  }
}
```

- **generated**: записує оригінальну інформацію зі створення зображення, таку як підказка, постачальник і модель.
- **розпізнано**: зберігає результати розпізнавання штучним інтелектом, наприклад автоматично створені заголовки, теги та описи.
- **вручну**: зберігає підібрані вручну результати. Дані в цій області мають найвищий пріоритет і не будуть перезаписані розпізнаванням ШІ.

Ця багаторівнева стратегія вирішує один із наших попередніх ключових конфліктів: коли розпізнавання штучного інтелекту та ручне курування розходяться, який із них має виграти? Відповідь - ручне введення. ШІ-розпізнавання призначене для того, щоб допомагати, а не приймати рішення. З часом це питання також стало зрозумілішим – машини залишаються машинами, і зрештою людям все одно потрібно телефонувати.

## Шаблон адаптера постачальника

Іншою основною частиною ImgBin є шаблон адаптера постачальника. Ми абстрагуємо зовнішні API за уніфікованим інтерфейсом, тому навіть якщо ми змінимо постачальника послуг ШІ, нам не потрібно змінювати бізнес-логіку.

У певному сенсі це трохи схоже на стосунки – зовнішній вигляд може змінюватися, але важливо, щоб внутрішня структура залишалася незмінною. Як тільки інтерфейс буде виправлено, внутрішня реалізація може вільно змінюватися.

### Постачальник генерації зображень

```typescript
interface ImageGenerationProvider {
  // Generate an image and return its Buffer
  generate(options: GenerateOptions): Promise<Buffer>;

  // Get the list of supported models
  getSupportedModels(): Promise<string[]>;
}

interface GenerateOptions {
  prompt: string;
  model?: string;
  size?: '1024x1024' | '1792x1024' | '1024x1792';
  quality?: 'standard' | 'hd';
  format?: 'png' | 'webp' | 'jpeg';
}
```

### Постачальник систем розпізнавання зору

```typescript
interface VisionRecognitionProvider {
  // Recognize image content and return structured metadata
  recognize(imageBuffer: Buffer): Promise<RecognitionResult>;

  // Get the list of supported models
  getSupportedModels(): Promise<string[]>;
}

interface RecognitionResult {
  title?: string;
  tags: string[];
  description?: string;
  confidence: number;
}
```

Перевагами цього дизайну інтерфейсу є:

1. **Можливий для перевірки**: у модульних тестах ми можемо передати імітаційних постачальників замість здійснення реальних зовнішніх викликів API.
2. **Розширюваний**: для додавання нового постачальника потрібно лише реалізувати інтерфейс; код абонента змінювати не потрібно.
3. **Замінний**: виробництво може використовувати Azure OpenAI, тоді як тестування може використовувати локальну модель, при цьому змінюється лише конфігурація.

Іноді робота над проектом також здається подібною. Зовні виглядає так, ніби ми щойно поміняли API, але внутрішня логіка залишається незмінною, і це робить усе набагато менш страшним.

## Дизайн команд CLI

ImgBin надає чотири основні команди для різних сценаріїв використання:

### генерувати: створення одного зображення

```bash
# Simplest usage
imgbin generate --prompt "orange dashboard for docs hero"

# Generate a thumbnail and AI annotations at the same time
imgbin generate --prompt "orange dashboard" --annotate --thumbnail

# Specify an output directory
imgbin generate --prompt "orange dashboard" --output ./library
```

### пакет: пакетні завдання

Пакетні завдання визначаються за допомогою файлів маніфесту YAML або JSON, що робить їх придатними для робочих процесів CI/CD:

```yaml
# assets/jobs/launch.yaml
defaults:
  annotate: true
  thumbnail: true
  libraryRoot: ./library

jobs:
  - prompt: "orange dashboard hero"
    slug: orange-dashboard
    tags: [dashboard, hero, orange]

  - prompt: "pricing grid for docs"
    slug: pricing-grid
    tags: [pricing, grid, docs]
```

Виконайте команду:

```bash
imgbin batch assets/jobs/launch.yaml
```

Дизайн пакетного завдання підтримує ізоляцію помилок: елементи в маніфесті обробляються один за одним, і помилка в одному елементі не впливає на інші. Ви також можете попередньо переглянути роботу за допомогою `--dry-run` фактично не виконавши його.

І найкраща частина полягає в тому, що вона точно розповідає вам, що вдалося, а що не вдалося. На відміну від деяких речей у житті, де трапляються невдачі, і ви навіть не знаєте, як це сталося.

### анотація: анотація ШІ

Запустіть розпізнавання AI на існуючих зображеннях, щоб автоматично генерувати заголовки, теги та описи:

```bash
# Annotate a single image
imgbin annotate ./library/2026-03/orange-dashboard

# Annotate an entire directory in batch
imgbin annotate ./library/2026-03/
```

### мініатюра: створення мініатюр

Створення мініатюр для існуючих зображень:

```bash
# Generate a thumbnail
imgbin thumbnail ./library/2026-03/orange-dashboard
```

## Дизайн маніфесту пакетного завдання

Формат маніфесту для пакетних завдань підтримує гнучку конфігурацію. Стандартні параметри можна встановити глобально, а окремі завдання можуть замінити їх:

```yaml
# Global defaults
defaults:
  annotate: true        # Enable AI annotation by default
  thumbnail: true       # Generate thumbnails by default
  libraryRoot: ./library
  model: gpt-image-1.5

jobs:
  # Minimal configuration: only provide a prompt
  - prompt: "first image"

  # Full configuration
  - prompt: "second image"
    slug: custom-slug
    tags: [tag1, tag2]
    annotate: false     # Do not run AI annotation for this job
    model: dall-e-3    # Use a different model for this job
```

Після виконання ImgBin обробляє завдання одне за іншим. Результат кожної роботи записується у відповідник `metadata.json`. Навіть якщо одна робота зазнає невдачі, інші це не вплине. Після виконання всіх завдань CLI видає підсумковий звіт:

```
✓ orange-dashboard (succeeded)
✓ pricing-grid (succeeded)
✗ hero-banner (failed: API rate limit exceeded)

2/3 succeeded, 1 failed
```

Деякі речі не можна поспішати. Взяти їх по черзі часто є стабільнішим шляхом. Можливо, це і є філософія пакетних робіт.

## Конфігурація змінної середовища

ImgBin підтримує гнучку конфігурацію за допомогою змінних середовища:

```bash
# ImgBin working directory
IMGBIN_WORKDIR=/path/to/imgbin

# Executable path (for invocation inside scripts)
IMGBIN_EXECUTABLE=/path/to/imgbin/dist/cli.js

# Asset library root
IMGBIN_LIBRARY_ROOT=./.imgbin-library

# Azure OpenAI configuration (if using the Azure provider)
AZURE_OPENAI_ENDPOINT=https://your-resource.openai.azure.com/
AZURE_OPENAI_API_KEY=***
AZURE_OPENAI_IMAGE_DEPLOYMENT=gpt-image-1
```

Конфігурація — це одна з тих речей, які водночас можуть здаватися важливими і не дуже важливими. Зрештою, те, що здається зручним і найкраще підходить для вашого робочого процесу, зазвичай є правильним вибором.

## Примітки щодо реалізації

Під час впровадження ми підсумували кілька ключових моментів:

### Дизайн інтерфейсу провайдера

Визначення інтерфейсу мають бути чіткими та повними, включаючи параметри введення, значення, що повертаються, і обробку помилок. Також доцільно надати як синхронний, так і асинхронний стилі виклику для різних сценаріїв.

Це маленький шматочок важко заробленого досвіду. Після встановлення інтерфейсу ніхто не хоче продовжувати змінювати його пізніше.

### Стратегія поводження з невдачами

Коли один елемент виходить з ладу в пакетному завданні, CLI має:

1. Запишіть детальну інформацію про помилку в окремий файл журналу.
2. Продовжуйте виконувати інші завдання замість того, щоб переривати весь процес.
3. Поверніть ненульовий код виходу в кінці, щоб вказати, що деякі завдання не вдалося виконати.
4. Чітке відображення результатів виконання кожного завдання в підсумковому звіті.

Деякі невдачі - це просто невдачі. Немає сенсу вдавати протилежне. Краще визнати їх відкрито, а потім придумати, як їх вирішити. Така ж логіка стосується проектів і життя.

### Стратегія злиття метаданих

Результати розпізнавання записуються до `recognized` за замовчуванням, тоді як поля, відредаговані вручну, позначені `manual`. Оновлення метаданих дотримуються стратегії лише додавання: якщо `--force` явно передається, існуючі підібрані вручну результати не перезаписуються.

Цей момент також став зрозумілим – деякі речі, перезаписані, просто зникають. Часто їх краще зберегти, оскільки сам запис має цінність.

### Створення каталогу Atomicity

використання `fs.mkdir({ recursive: true })` щоб гарантувати, що створення каталогу залишається атомарним і щоб уникнути умов змагання в одночасних сценаріях.

Можливо, це те, що відчуває безпека: бути стабільним, коли стабільність має значення, рухатися швидко, коли швидкість має значення, і ніколи не застрягати на передбачуваних здогадах.

## Висновок

Як основний інструмент для керування ресурсами зображень у проекті HagiCode, ImgBin вирішує наші проблеми за допомогою таких варіантів дизайну:

1. **Уніфікована точка входу**: CLI охоплює створення, анотації, мініатюри та всі інші основні операції.
2. **На основі метаданих**: кожен актив має повну інформацію `metadata.json`, що забезпечує пошук і відстеження.
3. **Адаптер постачальника**: гнучка абстракція для зовнішніх API, що полегшує тестування та розширення.
4. **Підтримка пакетних завдань**: створення пакетних зображень можна автоматизувати в робочих процесах CI/CD.

Усе інше, можливо, зникло, але цей підхід справді виявився корисним.

Це рішення не тільки покращує ефективність власної розробки HagiCode, але й формує структуру багаторазового використання для керування ресурсами зображень. Якщо ви створюєте подібний багатокомпонентний проект, я вважаю, що дизайнерські ідеї ImgBin можуть дати вам натхнення.
