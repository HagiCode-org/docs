## До побачення, пекло сценаріїв: чому ми вибрали C# для сучасної системи збирання

> Глибоке занурення в те, як проект HagiCode використовує Nuke для створення надійного, кросплатформного та високорозширюваного автоматизованого конвеєра збірки, повністю вирішуючи проблеми обслуговування традиційних сценаріїв збірки.

## Фон

Протягом довгого шляху розробки програмного забезпечення слово «збірка» викликає як любов, так і розчарування. Любов, тому що одним клацанням миші код перетворюється на продукт, і це один із найприємніших моментів у програмуванні. Розчарування, тому що підтримувати купу брудних сценаріїв збірки може бути кошмаром.

У багатьох проектах ми звикли писати сценарії на Python або покладатися на файли конфігурації XML. Тільки уявіть собі страх перед тим, як бути керованим `<property>` блоки. Але в міру того як складність проекту зростає, особливо в такому проекті, як HagiCode, який охоплює зовнішню та бекендову роботу, кілька платформ і кілька мов, традиційні підходи до збірки починають показувати свої межі. Розрізнена логіка сценаріїв, відсутність перевірки типів, слабка підтримка IDE... кожна з цих проблем стає невеликою пасткою, яка час від часу ставить команду на спотикання.

Щоб вирішити ці проблеми, проект HagiCode представив **Nuke**, сучасну систему збирання на основі C#. Це не просто інструмент. Це інший спосіб мислення про створення робочих процесів. Сьогодні ми розглянемо, чому ми обрали його та як це значно покращило наш досвід розробки.

## Про HagiCode

> Короткий вступ до того, що ми будуємо

Ми створюємо **HagiCode**, помічника кодування на основі ШІ, який робить процес розробки розумнішим, зручнішим і приємнішим.

**Розумний**: допомога ШІ доступна протягом усього процесу, від ідей до коду, що підвищує ефективність розробки. **Зручно**: багатопотокові одночасні операції повною мірою використовують ресурси та забезпечують плавність робочих процесів. **Забава**: гейміфікація та система досягнень роблять кодування менш нудним і набагато кориснішим.

Проект швидко розвивається. Якщо ви зацікавлені в технічному написанні, управлінні знаннями або розробці за допомогою ШІ, не соромтеся відвідати [GitHub](https://github.com/HagiCode-org/site).

## Основний аналіз: чому Nuke?

Ви можете запитати: «Існує вже так багато систем збирання, як-от Make, Gradle або навіть звичайні сценарії Shell. Навіщо йти від шляху, щоб використовувати систему на основі C#?»

Це справедливе запитання. Основна привабливість Nuke полягає в тому, що він переносить функції мови програмування, які ми вже добре знаємо, у світ сценаріїв збірки.

### 1. Робочі процеси модульної побудови: Мистецтво цілей

Nuke має дуже чітку філософію дизайну: **усе є мішенню**.

У традиційних сценаріях легко отримати сотні рядків лінійної логіки виконання, сплутаних разом. У Nuke ми розбиваємо процес збирання на незалежний `Target`с. Кожна мета виконує одну дію, наприклад:

- `Clean`: очистити вихідний каталог
- `Restore`: відновити пакети залежностей
- `Compile`: скомпілювати код
- `Test`: запустити модульні тести

Цей дизайн дуже добре узгоджується з принципом єдиної відповідальності. Подібно до будівельних блоків, ці цілі можна вільно комбінувати. Що ще важливіше, Nuke дозволяє нам визначати залежності між цілями. Наприклад, якщо ви хочете побігати `Test`, система автоматично перевірить, чи `Compile` вже виконано. Якщо хочеш `Compile`, потім `Restore` природно стоїть на першому місці.

Цей графік залежностей робить логіку набагато зрозумілішою та значно покращує ефективність виконання, оскільки Nuke може автоматично аналізувати оптимальний шлях виконання.

### 2. Безпека шрифту: до побачення, кошмар друкарських помилок

Кожен, хто писав сценарії збірки на Python, напевно, стикався з такою незручною ситуацією: сценарій виконується п’ять хвилин, а потім виходить з ладу, тому що `Confi.guration` було введено з помилкою або тому, що передано рядок там, де очікувався числовий параметр.

Найбільшою перевагою написання сценаріїв збірки на C# є **безпека типів**. Це означає:

- **Перевірки під час компіляції**: IDE може повідомити вам, що не так, поки ви вводите текст, замість того, щоб змушувати вас чекати до виконання.
- **Безпечний рефакторинг**: якщо ви хочете перейменувати змінну або метод, IDE може впоратися з цим за один крок, не змушуючи вас виконувати ризикований глобальний пошук і заміну.
- **Розумне завершення**: потужний IntelliSense може автоматично завершувати код за вас, тож вам не потрібно постійно гортати документацію, щоб запам’ятати незрозумілі API.

### 3. Кросплатформна підтримка: єдиний досвід збірки

У минулому ми писали `.bat` на Windows і `.sh` на Linux, а потім, можливо, додав сценарій Python, щоб подолати розрив. Тепер скрізь, де може працювати .NET Core або .NET 5+ сьогодні, Nuke також може працювати.

Це означає, що члени команди можуть використовувати Windows, Linux або macOS, і незалежно від того, віддають перевагу Visual Studio, VS Code або Rider, усі вони виконують однакову логіку. Це значно зменшує проблеми, спричинені відмінностями середовищ, особливо класичну проблему «це працює на моїй машині».

### 4. Управління параметрами та конфігурацією

Nuke забезпечує дуже елегантний механізм аналізу параметрів. Вам не потрібно розбирати вручну `string[] args`. Ви просто визначаєте властивість і додаєте до неї анотації `[Parameter]`, і Nuke автоматично обробляє відображення з аргументів командного рядка та файлів конфігурації.

Наприклад, ми можемо дуже легко визначити конфігурацію збірки:

```csharp
[Parameter("Configuration to build - Default is 'Debug'")]
readonly Configuration BuildConfiguration = IsLocalBuild ? Configuration.Debug : Configuration.Release;

Target Compile => _ => _
    .DependsOn(Restore)
    .Executes(() =>
    {
        // Use BuildConfiguration here; it is type-safe
        DotNetBuild(s => s
            .SetConfiguration(BuildConfiguration)
            .SetProjectFile(SolutionFile));
    });
```

Цей стиль інтуїтивно зрозумілий і стійкий до помилок.

## Практичний посібник: як це реалізувати на практиці в проекті

Сама лише теорія нікуди не приведе. Давайте подивимося, як ми фактично застосували цей підхід у проекті HagiCode.

### 1. Сплануйте структуру проекту

Ми не хотіли, щоб скрипти збірки захаращували корінь проекту, і ми також не хотіли, щоб структура каталогів була настільки глибокою, щоб вона нагадувала певні проекти Java. Тому ми розмістили всі файли збірки, пов’язані з Nuke, в одному `nukeBuild/` каталог.

Переваги очевидні:

- Корінь проекту залишається чистим.
- Логіка побудови залишається згуртованою та легшою в управлінні.
- Нові учасники команди одразу бачать: «Ось де живе логіка, пов’язана з розробкою».

### 2. Створіть чіткий ланцюжок цільових залежностей

Розробляючи Targets, ми дотримувалися одного принципу: **атомність плюс потік залежностей**.

Кожна ціль має бути достатньо маленькою, щоб виконувати лише одну дію. Наприклад, `Clean` слід лише видаляти файли і не слід тихо обробляти упаковку збоку.

Рекомендований потік залежностей виглядає приблизно так:

`Clean` -> `Restore` -> `Compile` -> `Test` -> `Pack`

Звичайно, це не є абсолютним. Наприклад, якщо ви хочете лише запускати тести і не хочете нічого пакувати, Nuke дозволяє вам виконати `nuke Test` безпосередньо, і він автоматично обробить необхідні `Restore` і `Compile` кроки перші.

### 3. Надійна обробка помилок і журналювання

Що найбільше засмучує у сценаріях збірки? Нечіткі повідомлення про помилки. Невдалої збірки, яка реєструє лише «Помилка: 1», достатньо, щоб звести будь-кого з розуму.

У Nuke, оскільки ми можемо безпосередньо використовувати обробку винятків C#, ми можемо фіксувати та повідомляти про помилки з набагато більшою точністю.

```csharp
Target Publish => _ => _
    .DependsOn(Test)
    .Executes(() =>
    {
        try 
        {
            // Try publishing to NuGet
            DotNetNuGetPush(s => s
                .SetTargetPath(ArtifactPath)
                .SetSource("https://api.nuget.org/v3/index.json")
                .SetApiKey(ApiKey));
        }
        catch (Exception ex)
        {
            Log.Error($"Publishing failed. Team, please check whether the key is correct: {ex.Message}");
            throw; // Make sure the build process exits with a non-zero exit code
        }
    });
```

### 4. Використовуйте тести для захисту якості

Сценарії збірки також є кодом, а це означає, що вони також потребують тестів. Nuke дозволяє нам писати тести для робочих процесів збирання, щоб, коли ми змінюємо логіку збирання, ми випадково не порушили існуючі процеси випуску. Це особливо важливо в конвеєрах безперервної інтеграції.

## Резюме

З появою Nuke процес збирання HagiCode став плавнішим, ніж будь-коли раніше. Це була не просто заміна інструменту. Це було оновлення інженерного мислення.

**Що ми отримали?**

- **Ремонтопридатність**: конфігурація як код, чіткіша логіка та швидший шлях адаптації для нових учасників.
- **Стабільність**: чітке введення усуває понад 90% помилок низького рівня.
- **Узгодженість**: уніфікований міжплатформний досвід, який усуває дрейф середовища.

Якщо раніше написання сценаріїв збірки було схожим на «проникнення в темряві», то використання Nuke — наче «прогулянка вночі з увімкненим світлом». Якщо ви втомилися підтримувати сценарії, які важко налагоджувати, подумайте про те, щоб перенести свою логіку збірки у світ C#. Ви можете виявити, що конструкції також можуть бути елегантними.
