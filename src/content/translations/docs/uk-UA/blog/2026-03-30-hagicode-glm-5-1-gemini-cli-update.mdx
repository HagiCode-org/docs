## Фон

Час дійсно плине швидко. Розвиток великих мовних моделей піднімався, як бамбук навесні. Не так давно ми все ще вболівали за «ШІ, який може писати код». Зараз ми вже живемо в епоху співпраці з кількома моделями та інтеграції з кількома інструментами. Це захоплююче? Можливо. Зрештою, розробникам ніколи не потрібен був лише сам інструмент, а легкість адаптації до різних сценаріїв і гнучкого перемикання за потреби.

Як платформа кодування з підтримкою ШІ, HagiCode нещодавно привітала дві важливі розробки: по-перше, повну інтеграцію моделі GLM-5.1 Zhipu AI; по-друге, офіційне додавання Gemini CLI як десятого підтримуваного Agent CLI. Можливо, ці два оновлення не звучать карколомно, але вони, безсумнівно, є гарною новиною для подальшого розвитку платформи.

GLM-5.1 — остання флагманська модель Zhipu AI. У порівнянні з GLM-5.0, він пропонує сильніші міркування, глибше розуміння коду та плавніший виклик інструментів. Що ще важливіше, це перша модель GLM, яка підтримує введення зображень. Що це означає? Це означає, що користувачі можуть дозволити штучному інтелекту дивитися прямо на знімок екрана, замість того, щоб намагатися описати проблему словами. Скориставшись цією зручністю, ви відразу розумієте її цінність.

Водночас через в `HagiCode.Libs.Providers` архітектури, HagiCode успішно інтегрував Gemini CLI в платформу. Це вже десятий CLI агента. Чесно кажучи, досягнення цього моменту приносить скромне відчуття виконаного завдання.

Варто також згадати, що функція завантаження зображень HagiCode дозволяє користувачам спілкуватися з ШІ безпосередньо через скріншоти. Навіть під час роботи GLM 4.7 платформа все ще працює добре і вже допомогла виконати багато важливих завдань зі створення. Що стосується GLM-5.1, то, природно, він йде ще далі.

## Про HagiCode

Підхід, поширений у цій статті, базується на нашому практичному досвіді в [HagiCode](https://hagicode.com) проект. HagiCode — це платформа кодування з відкритим вихідним кодом із підтримкою штучного інтелекту, розроблена для надання розробникам гнучкого та потужного помічника з програмування штучного інтелекту через багатомодельну архітектуру з кількома CLI. ## Дизайн архітектури Multi-CLI

Однією з основних сильних сторін HagiCode є підтримка кількох інструментів CLI програмування штучного інтелекту через уніфікований рівень абстракції. Перевага цього дизайну насправді досить проста: нові інструменти можуть надходити, старі інструменти можуть залишатися, а кодова база не перетворюється на хаос. Чесно кажучи, кожен би хотів, щоб життя складалося так.

### AIProviderType enum

Платформа визначає підтримувані типи постачальників CLI через `AIProviderType` enum:

```csharp
public enum AIProviderType
{
    ClaudeCodeCli = 0,    // Claude Code CLI
    CodexCli = 1,          // GitHub Copilot Codex
    GitHubCopilot = 2,     // GitHub Copilot
    CodebuddyCli = 3,     // Codebuddy CLI
    OpenCodeCli = 4,      // OpenCode CLI
    IFlowCli = 5,         // IFlow CLI
    HermesCli = 6,        // Hermes CLI
    QoderCli = 7,         // Qoder CLI
    KiroCli = 8,          // Kiro CLI
    KimiCli = 9,          // Kimi CLI
    GeminiCli = 10,       // Gemini CLI (new)
}
```

Як бачите, Gemini CLI приєднується до цього сімейства як десятий член. Кожен CLI має свої відмінні характеристики та сценарії використання, тому користувачі можуть гнучко вибирати відповідно до своїх потреб. Адже до Риму веде багато доріг; деякі просто легші, ніж інші.

### Архітектура провайдера

`HagiCode.Libs.Providers` забезпечує уніфікований інтерфейс постачальника, який робить кожну інтеграцію CLI стандартизованою та лаконічною. Беручи Gemini CLI як приклад:

```csharp
public class GeminiProvider : ICliProvider<GeminiOptions>
{
    private static readonly string[] DefaultExecutableCandidates = ["gemini", "gemini-cli"];
    private const string ManagedBootstrapArgument = "--acp";

    public string Name => "gemini";
    public bool IsAvailable => _executableResolver.ResolveFirstAvailablePath(DefaultExecutableCandidates) is not null;
}
```

Перевагами цієї конструкції є:
- Інтеграція нового CLI вимагає лише реалізації одного класу постачальника
- Уніфіковане керування життєвим циклом і об’єднання сеансів
- Автоматизоване визначення псевдонімів і виявлення виконуваних файлів

Простіше кажучи, цей дизайн перетворює складні речі на простіші та полегшує життя.

### Реєстр провайдерів

Реєстр постачальників автоматично обробляє зіставлення псевдонімів і реєстрацію:

```csharp
if (provider is GeminiProvider)
{
    registry.Register(provider.Name, provider, ["gemini-cli"]);
    continue;
}
```

Це означає, що користувачі можуть викликати Gemini CLI будь-яким із них `gemini` або `gemini-cli`, і система розпізнає його автоматично. Це як друг із офіційним ім’ям і псевдонімом – у будь-якому випадку люди знають, кого ви маєте на увазі.

## Підтримка моделі GLM-5.1

GLM-5.1 — остання флагманська модель Zhipu AI, і HagiCode завершила її повну підтримку.

### Каталог середніх професій

HagiCode керує всіма підтримуваними моделями через каталог вторинних професій. Ось конфігурація серії GLM:

| ID моделі | Ім'я | Підтримує зображення | Сумісні сімейства CLI |
|----------|------|----------------|-------------------------|
| `glm-4.7` | GLM 4.7 | - | claude, codebuddy, hermes, qoder, kiro |
| `glm-5` | GLM 5 | - | claude, codebuddy, hermes, qoder, kiro |
| `glm-5-turbo` | GLM 5 Турбо | - | claude, codebuddy, hermes, qoder, kiro |
| `glm-5.0` | GLM 5.0 (застарілий) | - | claude, codebuddy, hermes, qoder, kiro |
| `glm-5.1` | GLM 5.1 | **правда** | claude, codebuddy, hermes, qoder, kiro |

Основні характеристики GLM-5.1 можна підсумувати наступним чином:
- Окремий ідентифікатор версії без застарілого багажу
- Перша модель GLM, яка підтримує введення зображень
- Покращене міркування та розуміння коду
- Широка сумісність з кількома командами командного рядка

### GLM-5.1 проти GLM-5.0

На рівні коду тут показано ключову різницю між GLM-5.1 і GLM-5.0:

```csharp
// GLM-5.0 (Legacy) - contains special retention logic
private const string Glm50CodebuddySecondaryProfessionId = "secondary-glm-5-codebuddy";
private const string Glm50CodebuddyModelValue = "glm-5.0";

// GLM-5.1 - standalone new model identifier
private const string Glm51SecondaryProfessionId = "secondary-glm-5-1";
private const string Glm51ModelValue = "glm-5.1";
```

GLM-5.0 має позначку «Legacy», оскільки це ідентифікатор старої версії, збережений для зворотної сумісності. GLM-5.1, навпаки, є абсолютно новою автономною версією без історичного тягаря. Деякі речі залишаються в минулому; інші подорожують легше і рухаються швидше.

### Налаштувати GLM-5.1

Ось приклад конфігурації для використання GLM-5.1 у HagiCode:

```json
{
  "primaryProfessionId": "profession-claude-code",
  "secondaryProfessionId": "secondary-glm-5-1",
  "model": "glm-5.1",
  "reasoning": "high"
}
```

## Функція завантаження зображень

Підтримка зображень HagiCode реалізована через `SupportsImage` власність на `SecondaryProfession`:

```csharp
public class HeroSecondaryProfessionSettingDto
{
    public bool SupportsImage { get; set; }
}
```

У каталозі вторинних професій конфігурація GLM-5.1 виглядає так:

```json
{
  "id": "secondary-glm-5-1",
  "supportsImage": true
}
```

Це означає, що користувачі можуть завантажувати знімки екрана безпосередньо для аналізу ШІ, наприклад:
- Скріншоти повідомлень про помилки
- Проблеми на екрані інтерфейсу користувача
- Діаграми візуалізації даних
- Результати виконання коду

Більше не потрібно все описувати вручну - просто завантажте скріншот. Зручність цієї функції очевидна, коли ви нею скористаєтеся. Іноді один погляд говорить більше, ніж довге пояснення.

## Інтеграція Gemini CLI

Як десятий Agent CLI, Gemini CLI інтегровано в HagiCode через стандартну архітектуру Provider.

### Параметри конфігурації

Gemini CLI підтримує широкий набір параметрів конфігурації:

```csharp
public class GeminiOptions
{
    public string? ExecutablePath { get; set; }
    public string? WorkingDirectory { get; set; }
    public string? SessionId { get; set; }
    public string? Model { get; set; }
    public string? AuthenticationMethod { get; set; }
    public string? AuthenticationToken { get; set; }
    public Dictionary<string, string?> AuthenticationInfo { get; set; }
    public Dictionary<string, string?> EnvironmentVariables { get; set; }
    public string[] ExtraArguments { get; set; }
    public TimeSpan? StartupTimeout { get; set; }
    public CliPoolSettings? PoolSettings { get; set; }
}
```

Ці параметри охоплюють усе, від базового налаштування до розширених функцій, надаючи користувачам гнучкість налаштування CLI відповідно до власних потреб. Робочий процес у кожного різний, тому трохи гнучкості завжди вітається.

### Протокол зв'язку ACP

Gemini CLI підтримує ACP (Agent Communication Protocol), який є уніфікованим стандартом зв’язку HagiCode CLI. Через ACP різні CLI можуть узгоджено взаємодіяти з платформою, що значно спрощує роботу з інтеграції. Коротше кажучи, він стандартизує складні частини, щоб кожному було легше працювати.

## Власна практика збирання HagiCode

Якщо говорити про реальну практику, то найкращим прикладом є власний робочий процес збірки платформи HagiCode. У процесі розробки HagiCode вже повною мірою використано можливості ШІ.

### Добре працює навіть з GLM 4.7

Конструкція платформи HagiCode добре оптимізована, тому вона все ще може забезпечити хороший досвід розробки навіть з GLM 4.7. Платформа вже допомогла завершити кілька важливих будівельних проектів, зокрема:
- Інтеграція кількох постачальників CLI
- Реалізація функції завантаження зображень
- Формування документації та публікація контенту

Це насправді добре. Не всім постійно потрібні нові речі. Те, що вам найбільше підходить, часто є найважливішим.

### GLM-5.1 забезпечує більше з меншими зусиллями

Після оновлення до GLM-5.1 ці можливості стають ще сильнішими:
- Краще розуміння коду, скорочення зворотнього зв’язку
- Більш точний аналіз залежностей, який одразу вказує у правильному напрямку
- Більш ефективна діагностика помилок, швидше визначення проблем
- Підтримка введення зображень, прискорення опису проблем

Це як пересісти з велосипеда на автомобіль. Ви все ще можете дістатися до того самого пункту призначення, але швидкість і комфорт вже не ті.

## Найкращі методи інтеграції з кількома командами командного рядка

`HagiCode.Libs.Providers` забезпечує уніфікований механізм реєстрації та використання:

```csharp
services.AddHagiCodeLibs();

var gemini = serviceProvider.GetRequiredService<ICliProvider<GeminiOptions>>();
var codebuddy = serviceProvider.GetRequiredService<ICliProvider<CodebuddyOptions>>();
var hermes = serviceProvider.GetRequiredService<ICliProvider<HermesOptions>>();
```

Цей дизайн ін’єкції залежностей зберігає використання в різних CLI дуже лаконічним, а також робить модульне тестування та моделювання зручнішим. Чистий код – це спосіб бути відповідальним перед собою.

### Примітки

Під час фактичного використання слід пам’ятати про кілька речей:

1. **Конфігурація ключа API**: переконайтеся `ANTHROPIC_AUTH_TOKEN` встановлено правильно, або модель не може бути викликана
2. **Доступність моделі**: GLM-5.1 має бути ввімкнено відповідним постачальником моделі
3. **Функція зображення**: Лише моделі з `supportsImage: true` можна використовувати завантаження зображень
4. **Встановлення CLI**: переконайтеся, що перед використанням Gemini CLI `gemini` або `gemini-cli` знаходиться в системі PATH

Це можуть бути дрібні деталі, але дрібні деталі, якщо їх не врахувати, можуть обернутися великими проблемами, тому на них варто звернути увагу.

## Висновок

Завдяки повній підтримці GLM-5.1 і успішній інтеграції Gemini CLI, HagiCode ще більше розширює свої можливості як багатомодельної платформи програмування штучного інтелекту з кількома CLI. Ці оновлення не тільки дають користувачам більше можливостей, але й демонструють передову архітектуру та масштабованість HagiCode.

Підтримка зображень GLM-5.1 у поєднанні з функцією завантаження скріншотів HagiCode дозволяє штучному інтелекту «розуміти зображення» та значно знижує вартість опису проблем. А завдяки підтримці десяти командних команд користувачі можуть гнучко вибрати помічника програмування ШІ, який найкраще відповідає їхнім уподобанням і сценаріям. Більший вибір – майже завжди добре.

Найважливіше те, що власна практика створення HagiCode доводить, що платформа вже може добре працювати та виконувати складні завдання навіть із GLM 4.7, тоді як оновлення до GLM-5.1 може ще більше підвищити ефективність розробки. Життя також часто буває таким: вам не завжди потрібно абсолютне найкраще, лише те, що вам підходить. Звичайно, якщо те, що вам підходить, може стати ще кращим, то тим краще.

Якщо вас цікавить мультимодельна платформа програмування штучного інтелекту з кількома командами командного рядка, спробуйте HagiCode — з відкритим вихідним кодом, безкоштовно та все ще розвивається. Спроба нічого не коштує, і це може виявитися саме тим, що вам потрібно.
