## Від Docusaurus 3.x до Astro 5.x: ретроспектива міграції сайту HagiCode

> У цій статті розглядається наша повна міграція офіційного веб-сайту HagiCode з Docusaurus 3.x на Astro 5.x. Ми детально зануримося в те, як Astro's Islands Architecture допомогла нам усунути вузькі місця продуктивності, зберігаючи наявні активи компонентів React, забезпечуючи покращення як швидкості збірки, так і продуктивності завантаження.

## Фон

У січні 2026 року ми здійснили «пересадку серця» на офіційному сайті HagiCode, повністю перенісши його основну структуру з Docusaurus 3.x на Astro 5.x. Це було не імпульсивне переписування, а ретельно виважене технічне рішення.

Перед міграцією наш сайт був функціонально завершеним, але він почав виявляти деякі класичні «проблеми розкоші»: роздуті артефакти збірки, надмірне корисне навантаження JavaScript і менш ніж ідеальна швидкість завантаження сторінок на сторінках складної документації. Як проект помічника кодування AI, HagiCode потребує частих оновлень документації та функцій, тому ефективність створення безпосередньо впливає на швидкість випуску. У той же час ми хотіли, щоб сайт був більш зручним для пошукових систем (SEO), щоб більше розробників могли дізнатися про проект.

Щоб вирішити ці проблеми, ми прийняли сміливе рішення: перебудувати всю систему на Astro. Вплив цього рішення може бути навіть більшим, ніж ви очікуєте. Незабаром я розберусь у подробиці.

## Про HagiCode

Підхід до міграції сайту, про який ми розповідаємо в цій статті, базується на нашому практичному досвіді в [HagiCode](https://github.com/HagiCode-org/site) проект.

HagiCode — це помічник кодування штучного інтелекту, спрямований на підвищення ефективності розробки. Ми дбаємо не лише про ітерацію основних функцій, але й про досвід розробників. Цей рефакторинг сайту також мав на меті надати користувачам найшвидший досвід під час перегляду наших документів і офіційного веб-сайту.

## Навіщо залишати зрілу екосистему Docusaurus?

В екосистемі React Docusaurus довгий час був «стандартною відповіддю» для сайтів документації. Він працює з коробки, пропонує багату екосистему плагінів і має активну спільноту. Але оскільки HagiCode отримав більше функцій, ми також відчули його обмеження:

1.  **Вузькі місця у продуктивності**: Docusaurus — це, по суті, React SPA (односторінкова програма). Навіть якщо ви пишете лише статичні сторінки, клієнту все одно потрібно завантажити середовище виконання React і гідратувати сторінку, що є невиправдано важким для простих сторінок документів.
2.  **Великий розмір ресурсу**: навіть якщо сторінка містить дуже мало вмісту, розмір пакету JS залишається відносно фіксованим. Це не ідеально для мобільних користувачів або поганих умов мережі.
3.  **Обмежена гнучкість**: хоча він розширюваний, ми хотіли більше низькорівневого контролю над конвеєром збірки.

Astro прибув саме в потрібний час, щоб вирішити ці проблеми. Він представив нову «архітектуру островів»: за замовчуванням Astro генерує статичний HTML без JavaScript, і лише компоненти, які потребують інтерактивності, «активуються» та завантажують JS. Це означає, що більша частина нашого сайту стає чистим HTML і завантажується надзвичайно швидко.

## Основна стратегія міграції: плавний архітектурний перехід

Міграція була не просто копіюванням і вставкою. Це вимагало зміни мислення. Ми перейшли від моделі Docusaurus «all React» до моделі Astro «Core + Islands».

### 1. Перебудова системи конфігурації

По-перше, ми повинні були переїхати з `docusaurus.config.ts` до `astro.config.mjs`. Це було не просто перейменування файлу, а перепис логіки маршрутизації та збірки.

У Docusaurus все є плагіном. В Astro все є інтеграцією. Нам потрібно було перевизначити базовий шлях сайту, створити режим виведення (статичний проти SSR) і стратегію оптимізації активів.

**Перед міграцією:**
```typescript
// docusaurus.config.ts
export default {
  title: 'HagiCode',
  url: 'https://hagicode.com',
  baseUrl: '/',
  // ... more configuration
};
```

**Після міграції:**
```javascript
// astro.config.mjs
import { defineConfig } from 'astro/config';
import react from '@astrojs/react';

export default defineConfig({
  integrations: [react()],
  site: 'https://hagicode.com',
  base: '/',
  // Optimization settings for static assets
  build: {
    inlineStylesheets: 'auto',
  },
});
```

### 2. Що залишити, а що змінити в компонентах React

Це була найболючіша частина міграції. Наш існуючий сайт мав багато компонентів React, таких як вкладки, підсвічування коду, кнопки зворотного зв’язку тощо. Викинути їх було б марнотратно, але зберегти все це зробило б корисне навантаження JavaScript занадто важким.

HagiCode застосував стратегію **прогресивного зволоження**:
*   **Чисто статичні компоненти**: для презентаційного вмісту, такого як верхні, нижні колонтитули та звичайна текстова документація, ми переписали їх як компоненти Astro (`.astro` файли) і відтворювали їх безпосередньо в HTML під час створення.
*   **Інтерактивні острівці**: для компонентів, які мають залишатися інтерактивними, таких як перемикач тем, перемикання вкладок і кнопки копіювання блоку коду, ми зберегли реалізацію React і додали `client:load` або `client:visible` директиви.

Наприклад, наш часто використовуваний компонент Tabs у документації:

```jsx
// src/components/Tabs.jsx
import { useState } from 'react';
import './Tabs.css'; // Import styles

export default function Tabs({ items }) {
  const [activeIndex, setActiveIndex] = useState(0);
  // ... state logic
  return (
    <div className="tabs-wrapper">
      {/* Rendering logic */}
    </div>
  );
}
```

При використанні в Markdown ми явно повідомляємо Astro: «Цьому компоненту потрібен JS».

```markdown
// src/content/docs/example.mdx
import Tabs from '../../components/Tabs.jsx';

<!-- Load JS only when the component enters the viewport -->
<Tabs client:visible items={...} />
```

Таким чином, інтерактивні компоненти за межами вікна перегляду не конкурують за пропускну здатність, що значно покращує швидкість завантаження першого екрана.

### 3. Адаптація системи стилів: від CSS-модулів до Scoped CSS

Docusaurus за замовчуванням підтримує модулі CSS, тоді як Astro підтримує Scoped CSS через `<style>` тег. Основною ідеєю обох є ізоляція стилів, але синтаксис різний.

Під час міграції HagiCode ми перетворили найскладніші модулі CSS у стилі Astro. Це насправді виявилося добре, тому що в `.astro` стилі та шаблони знаходяться в одному файлі, що робить обслуговування більш інтуїтивно зрозумілим.

**Перед рефакторингом:**
```css
/* Tabs.module.css */
.wrapper { background: var(--ifm-background-color); }
```

**Після рефакторингу (Astro Scoped):**
```html
<!-- Tabs.astro -->
<div class="tabs-wrapper">
  <slot />
</div>

<style>
  .tabs-wrapper {
    /* Use CSS variables directly to adapt to the theme */
    background: var(--bg-color);
    padding: 1rem;
  }
</style>
```

У той же час ми уніфікували глобальну систему змінних CSS і використали можливості Astro з урахуванням середовища, щоб забезпечити плавне перемикання темного режиму між сторінками.

## Підводні камені, з якими ми стикалися на практиці, і як ми їх вирішували

Під час фактичної міграції HagiCode ми зіткнулися з кількома проблемами. Ось кілька найбільш типових.

### 1. Больові точки змінної шляху та середовища

HagiCode підтримує розгортання підшляху, наприклад розгортання в підкаталозі GitHub Pages. У Docusaurus, `baseUrl` обробляється автоматично. Проте в Astro нам потрібно бути обережнішими під час обробки посилань на зображення та запитів API.

Ми запровадили механізм змінної середовища, щоб керувати цим послідовно:

```javascript
// Handle paths in the build script
const getBasePath = () => import.meta.env.VITE_SITE_BASE || '/';
```

Переконайтеся, що шляхи не починаються з `/` у вашому коді. У розробці проти виробництва або після налаштування a `base` шляху, це може спричинити помилку 404 для активів.

### 2. Сумісність зі сценарієм CommonJS

На нашому старому сайті було кілька сценаріїв Node.js, які використовувалися для таких завдань, як автоматичне отримання даних показників і оновлення карти сайту, і вони були написані на CommonJS (`require`). Astro та сучасні інструменти збірки повністю охопили модулі ES (`import`/`export`).

Якщо у вас також є подібні сценарії, не забудьте переробити їх усі до модулів ES. Саме в цьому напрямку рухається екосистема, і чим раніше ви внесете зміни, тим менше у вас буде проблем згодом.

```javascript
// Old way
const fs = require('fs');

// New way
import fs from 'fs';
```

### 3. Не забувайте про SEO та перенаправлення

Пошукові системи вже проіндексували старі сторінки Docusaurus HagiCode. Якщо ви перейдете безпосередньо на Astro і структура URL-адреси зміниться, ви можете отримати велику кількість 404 і значне падіння рейтингу в пошуку.

Ми налаштували правила перенаправлення в Astro:

```javascript
// astro.config.mjs
export default defineConfig({
  redirects: {
    '/docs/old-path': '/docs/new-path',
    // Map old links to new links in bulk
  }
});
```

Або ви можете вирішити це на рівні конфігурації сервера. Переконайтеся, що старі посилання можуть бути 301 перенаправлені на нові адреси, тому що це критично для SEO.

## Резюме

Для HagiCode перехід із Docusaurus на Astro був не просто оновленням фреймворку. Це також була практична реалізація філософії «спочатку продуктивність».

**Що ми отримали:**
*   **Видатні оцінки Lighthouse**: після міграції оцінка продуктивності сайту HagiCode легко наблизилася до ідеальної оцінки.
*   **Швидше збирання**: можливості поступового збирання Astro скорочують час випуску оновлень документації вдвічі.
*   **Збережена гнучкість**: з Islands Architecture ми не пожертвували жодними інтерактивними функціями та могли використовувати React там, де це було потрібно.

Якщо ви також підтримуєте сайт, орієнтований на документацію, і маєте проблеми з розміром пакета або швидкістю завантаження, Astro варто спробувати. Незважаючи на те, що процес міграції вимагає деяких операцій, наприклад перейменування PCode на HagiCode і переміщення компонентів один за одним, плавна робота користувача, яку ви отримуєте натомість, робить його абсолютно вартим.

Система збірки, про яку йдеться в цій статті, — це точний підхід, який ми розробили шляхом реальних спроб і помилок під час створення HagiCode. Якщо ви вважаєте цей підхід цінним, це говорить про нашу інженерну силу, і сам HagiCode, ймовірно, також варто розглянути ближче.
