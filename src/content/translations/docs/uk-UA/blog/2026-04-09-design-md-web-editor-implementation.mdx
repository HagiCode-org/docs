## Редагуйте DESIGN.md безпосередньо у веб-інтерфейсі: від ідеї до реалізації

> У системі управління проектами MonoSpecs `DESIGN.md` здійснює архітектурно-технічне рішення проекту. Але традиційний процес редагування змушує користувачів переходити до зовнішнього редактора. Цей фрагментарний досвід схожий на те, що його переривають посеред читання вірша: натхнення зникло, настрій теж. У цій статті розповідається про рішення, яке ми застосували на практиці в проекті HagiCode: редагування `DESIGN.md` безпосередньо у веб-інтерфейсі з підтримкою імпорту шаблонів із сайту онлайн-дизайну. Зрештою, хто не насолоджується відчуттям завершення всього одним потоком?

## Фон

Як основний носій проектно-технічної документації, `DESIGN.md` зберігає ключову інформацію, таку як проект архітектури, технічні рішення та вказівки щодо реалізації. Однак традиційний підхід редагування вимагає від користувачів перейти до зовнішнього редактора, такого як VS Code, вручну знайти фізичний шлях, а потім відредагувати файл. Це не особливо складно, але після кількаразового повторення процесу стає втомливим.

Проблеми в основному проявляються такими способами:

- **Роздроблений робочий процес**: користувачі повинні постійно перемикатися між веб-інтерфейсом керування та локальним редактором, порушуючи безперервність робочого процесу, подібно до того, як музика обривається посеред пісні.
- **Важко повторно використовувати**: сайт дизайну вже публікує багату бібліотеку шаблонів дизайну, але їх неможливо інтегрувати безпосередньо в робочий процес редагування проекту. Хороші речі існують, але ви все одно не можете використовувати їх там, де вам це потрібно.
- **Відсутній цикл досвіду**: немає замкнутого циклу для «попереднього перегляду-вибору-імпорту», тому користувачі повинні копіювати та вставляти вручну, що збільшує ризик помилок.
- **Проблеми співпраці**: синхронізація проектних документів і впровадження коду стає складним процесом, що негативно впливає на ефективність команди.

Щоб вирішити ці проблеми, ми вирішили додати підтримку прямого редагування `DESIGN.md` у веб-інтерфейсі та дозволяють імпортувати шаблон одним клацанням миші з сайту онлайн-дизайну. Це не було якесь карколомне рішення. Ми просто хотіли зробити процес розробки плавнішим.

### Ключові технічні рішення

#### Рішення 1: Глобальний шаблон ящика

Ми використовуємо єдиний глобальний ящик замість локальних спливаючих шарів із керуванням станом `layoutSlice`, що надає користувачам узгоджену взаємодію з різними переглядами (`classic` і `kanban`). Незалежно від того, з якого перегляду користувач відкриває редактор, вони отримують однакову модель взаємодії. Послідовний досвід дозволяє людям почуватися більш вільно, а не дезорієнтуватися, коли вони змінюють погляди.

#### Рішення 2: Проектний API

Ми змонтували `DESIGN.md`пов'язані кінцеві точки під `ProjectController`, повторно використовуючи існуючі межі дозволів проекту та уникаючи складності додавання окремого контролера. Це робить обробку дозволів більш зрозумілою, а також узгоджується з організацією ресурсів RESTful. Іноді повторне використання є більш значущим, ніж створення чогось нового з нуля.

#### Рішення 3: Виявлення конфлікту версій

Ми отримуємо непрозору версію з файлової системи `LastWriteTimeUtc`, що дає нам легкий оптимістичний контроль паралельності. Коли кілька користувачів редагують один і той самий файл одночасно, система може виявити конфлікти та запропонувати користувачеві оновити файл. Ця конструкція не блокує редагування, водночас захищаючи послідовність даних.

#### Рішення 4: проксі-шаблон того самого походження

Ми використовуємо `IHttpClientFactory` для проксі зовнішніх ресурсів сайту дизайну, уникаючи як проблем перехресного походження, так і ризиків SSRF. Це забезпечує безпеку системи, а також спрощує зовнішні виклики. Навряд чи ви можете бути надто обережними з безпекою.

## Основна реалізація

### 1. Безпосередньо відредагуйте DESIGN.md

#### Впровадження бекенда

Сервер головним чином відповідає за вирішення шляхів, читання/запис файлів і керування версіями. Ці завдання базові, але незамінні, як фундамент будинку:

```csharp
// Path resolution and security validation
private Task<string> ResolveDesignDocumentDirectoryAsync(string projectPath, string? repositoryPath)
{
    if (string.IsNullOrWhiteSpace(repositoryPath))
    {
        return Task.FromResult(Path.GetFullPath(projectPath));
    }
    return ValidateSubPathAsync(projectPath, repositoryPath);
}

// Version generation (based on file system timestamp and size)
private static string BuildDesignDocumentVersion(string path)
{
    var fileInfo = new FileInfo(path);
    fileInfo.Refresh();
    return string.Create(
        CultureInfo.InvariantCulture,
        $"{fileInfo.LastWriteTimeUtc.Ticks:x}-{fileInfo.Length:x}");
}
```

Дизайн версії цікавий своєю простотою: ми використовуємо час і розмір останньої зміни файлу для створення унікального ідентифікатора версії. Він легкий і надійний, не потребує підтримки додаткової бази даних версії. Прості рішення часто є найефективнішими.

#### Реалізація інтерфейсу

У інтерфейсі ми реалізуємо виявлення брудного стану та зберігаємо логіку. Цей дизайн допомагає користувачам зрозуміти, чи збережено їхні зміни, і зменшує хвилювання «а що, якщо я втрачу це?»:

```typescript
// Dirty-state detection and save logic
const [draft, setDraft] = useState('');
const [savedDraft, setSavedDraft] = useState('');
const isDirty = draft !== savedDraft;

const handleSave = useCallback(async () => {
    const result = await saveProjectDesignMdDocument({
        ...activeTarget,
        content: draft,
        expectedVersion: document.version, // optimistic concurrency control
    });
    setSavedDraft(draft); // update saved state
}, [activeTarget, document, draft]);
```

У цій реалізації ми підтримуємо дві частини стану: `draft` це вміст, який зараз редагується, а `savedDraft` це збережений вміст. Їх порівняння повідомляє нам, чи є незбережені зміни. Дизайн простий, але він дає людям спокій. Ніхто не хоче, щоб те, над чим він важко працював, зникло.

#### Формат даних індексу

Індексний файл на сайті дизайну визначає всі доступні шаблони. За допомогою цього індексу користувачі можуть вибрати потрібний шаблон так само легко, як замовити з меню:

```json
{
  "entries": [
    {
      "slug": "linear.app",
      "title": "Linear Inspired Design System",
      "summary": "AI Product / Dark Feel",
      "detailUrl": "/designs/linear.app/",
      "designDownloadUrl": "/designs/linear.app/DESIGN.md",
      "previewLightImageUrl": "...",
      "previewDarkImageUrl": "..."
```

Кожен запис містить основну інформацію про шаблон і посилання для завантаження. Сервер зчитує список доступних шаблонів із цього індексу та надає їх для вибору користувачеві. Це робить вибір інтуїтивно зрозумілим замість того, щоб змушувати людей промацувати дорогу в темряві.

#### Реалізація проксі-сервера того самого походження

Для забезпечення безпеки серверна частина виконує сувору перевірку доступу до сайту розробки. Ви не можете бути занадто обережними щодо безпеки:

```csharp
// Safe slug validation
private static readonly Regex SafeDesignSiteSlugRegex =
    new("^[A-Za-z0-9](?:[A-Za-z0-9._-]{0,127})$", RegexOptions.Compiled);

private static string NormalizeDesignSiteSlug(string slug)
{
    var normalizedSlug = slug?.Trim() ?? string.Empty;
    if (!IsSafeDesignSiteSlug(normalizedSlug))
    {
        throw new BusinessException(
            ProjectDesignSiteIndexErrorCodes.InvalidSlug,
            "Design site slug must be a single safe path segment.");
    }
    return normalizedSlug;
}

// Preview image caching (OS temp directory)
private static string ComputePreviewCacheKey(string slug, string theme, string previewUrl)
{
    var raw = $"{slug}|{theme}|{previewUrl}";
    var bytes = SHA256.HashData(Encoding.UTF8.GetBytes(raw));
    return Convert.ToHexString(bytes).ToLowerInvariant();
}
```

Тут ми робимо дві речі: по-перше, ми перевіряємо формат слаг строго за допомогою регулярного виразу, щоб запобігти атакам проходження шляху; по-друге, ми кешуємо зображення попереднього перегляду, щоб зменшити тиск на зовнішній сайт. Перше – це захист, друге – оптимізація, і обидва важливі.

### 3. Повний потік імпорту

```typescript
// 1. Open the import drawer
const handleRequestImportDrawer = useCallback(() => {
    setIsImportDrawerOpen(true);
}, []);

// 2. Select and import
const handleImportRequest = useCallback((entry) => {
    if (isDirty) {
        setPendingImportEntry(entry);
        setConfirmMode('import'); // overwrite confirmation
        return;
    }
    void executeImport(entry);
}, [isDirty]);

// 3. Execute import
const executeImport = useCallback(async (entry) => {
    const result = await getProjectDesignMdSiteImportDocument(
        activeTarget.projectId,
        entry.slug
    );
    setDraft(result.content); // replace editor text only, do not save automatically
    setIsImportDrawerOpen(false);
}, [activeTarget?.projectId]);
```

Потік імпорту відбувається за принципом «підтвердження користувача»: після імпорту оновлюється лише вміст редактора, і нічого не зберігається автоматично. Користувачі можуть перевірити імпортований вміст і зберегти його вручну лише після підтвердження, що він виглядає правильно. Остаточне рішення має залишатися за користувачем.

## Практичні приклади

### Сценарій 1: Створення DESIGN.md у корені проекту

Коли `DESIGN.md` не існує, серверна частина повертає стан віртуального документа. Це дозволяє інтерфейсу уникнути спеціальної обробки випадку «файл не існує», а уніфікований API спрощує логіку коду:

```csharp
return new ProjectDesignDocumentDto
{
    Path = targetPath,
    Exists = false,  // virtual document state
    Content = string.Empty,
    Version = null
};

// Automatically create the file on first save
public async Task<SaveProjectDesignDocumentResultDto> SaveDesignDocumentAsync(...)
{
    Directory.CreateDirectory(targetDirectory);
    await File.WriteAllTextAsync(targetPath, input.Content);
    return new SaveProjectDesignDocumentResultDto { Created = !exists };
}
```

Перевагою цього дизайну є те, що інтерфейс не потребує спеціальної логіки для відсутніх файлів. Приховуючи цю складність у серверній частині, зовнішня частина може легше зосередитися на взаємодії з користувачем.

### Сценарій 2. Імпортуйте шаблон із сайту дизайну

Після того, як користувач вибирає шаблон дизайну «Лінійний» у панелі імпорту, система отримує `DESIGN.md` вміст через серверний проксі-сервер. Весь процес прозорий для користувача: він лише вибирає шаблон, а система автоматично обробляє мережеві запити та перетворення даних.

```typescript
// 1. The system fetches DESIGN.md content through the backend proxy
GET /api/project/{id}/design-md/site-index/linear.app

// 2. The backend validates the slug and fetches content from upstream
var entry = FindDesignSiteEntry(catalog, "linear.app");
using var upstreamResponse = await httpClient.SendAsync(request);
var content = await upstreamResponse.Content.ReadAsStringAsync();

// 3. The frontend replaces the editor text
setDraft(result.content);
// The user reviews it and then saves it manually to disk
```

Весь потік залишається прозорим для користувача. Вони просто вибирають шаблон, а система займається створенням мережі та трансформацією за лаштунками. Це той досвід, який ми хочемо: простий, але потужний.

### Сценарій 3: вирішення конфліктів версій

Коли кілька користувачів редагують те саме `DESIGN.md` в той же час система виявляє конфлікти версій. Цей оптимістичний механізм керування паралелізмом зберігає узгодженість даних, не блокуючи редагування користувача:

```csharp
if (!string.Equals(currentVersion, expectedVersion, StringComparison.Ordinal))
{
    throw new BusinessException(
        ProjectDesignDocumentErrorCodes.VersionConflict,
        $"DESIGN.md at '{targetPath}' changed on disk.");
}
```

Інтерфейс виявляє цю помилку та запитує користувача:

```typescript
// Frontend prompts the user to refresh and retry
<Alert>
    <AlertTitle>Version conflict</AlertTitle>
    <AlertDescription>
        The file was modified by another process. Please refresh to get the latest version and try again.
    </AlertDescription>
</Alert>
```

Цей оптимістичний механізм керування паралелізмом підтримує узгодженість даних, не блокуючи користувачів під час їх роботи. Конфліктів не уникнути, але принаймні користувачі повинні знати, що сталося, а не мовчки втрачати свої зміни.

## Примітки та найкращі практики

### 1. Безпека шляху

Завжди перевіряйте `repositoryPath` щоб запобігти атакам проходження шляху. Ви ніколи не можете зробити занадто багато, коли справа стосується безпеки:

```csharp
// Always validate repositoryPath to prevent path traversal attacks
return ValidateSubPathAsync(projectPath, repositoryPath);
// Reject dangerous inputs such as "../" and absolute paths
```

### 2. Стратегія кешу

Кешувати зображення попереднього перегляду протягом 24 годин, максимум 160 файлів. Помірне кешування покращує продуктивність, але баланс все ще має значення:

```csharp
// Cache preview images for 24 hours, with a maximum of 160 files
private static readonly TimeSpan PreviewCacheTtl = TimeSpan.FromHours(24);
private const int PreviewCacheMaxFiles = 160;
// Periodically clean up expired cache
```

### 3. Обробка помилок

Витончено знижувати, коли вихідний сайт недоступний. Ця конструкція гарантує, що навіть якщо зовнішня залежність виходить з ладу, основна функція редагування все одно працює нормально:

```typescript
// Gracefully degrade when the upstream site is unavailable
try {
    const catalog = await getProjectDesignMdSiteImportCatalog(projectId);
} catch (error) {
    toast.error(t('project.designMd.siteImport.feedback.catalogLoadFailed'));
    // The main editing drawer remains available
}
```

Ця витончена деградація гарантує, що навіть якщо зовнішні залежності недоступні, основна функція редагування продовжує працювати. Система має бути стійкою, а не руйнуватися, коли щось йде не так.

### 4. Оптимізація взаємодії з користувачем

Підтвердьте перезапис перед імпортом і не зберігайте автоматично після імпорту. Користувачі повинні контролювати власні дії:

```typescript
// Confirm overwrite before import
if (isDirty) {
    setConfirmMode('import');
    return;
}

// Do not save automatically after import; let the user confirm
setDraft(result.content); // update draft only
// The content is written to disk only after the user reviews it and clicks Save
```

### 5. Питання продуктивності

Використовуйте фабрику клієнтів HTTP, щоб уникнути створення занадто великої кількості з’єднань. Управління ресурсами може здатися дрібницею, але вдале його виконання може мати велике значення:

## Пропоновані розширення

1. **Покращення уцінки**: зараз ми використовуємо базовий `Textarea`, але ми могли б оновити до CodeMirror для підсвічування синтаксису та комбінацій клавіш. Коли редактор почувається краще, писати документацію теж стає краще.
2. **Режим попереднього перегляду**: додайте попередній перегляд Markdown у реальному часі, щоб покращити процес редагування. Те, що ви бачите, те й отримуєте, завжди додає людям більше впевненості.
3. **Злиття відмінностей**: реалізуйте інтелектуальний алгоритм злиття замість простої заміни повного тексту. Конфлікти неминучі, але процес вирішення конфлікту не повинен бути болючим.
4. **Локальне кешування**: кеш `design.json` у базі даних, щоб зменшити залежність від зовнішнього сайту. Чим менше залежностей система має, тим стабільнішою вона, як правило, є.

## Резюме

У проекті HagiCode ми реалізували повне рішення для онлайн-редагування та імпорту `DESIGN.md` за допомогою взаємодії інтерфейсу та бекенда. Основна цінність цього рішення полягає в наступних моментах:

- **Вища ефективність**: не потрібно міняти інструменти; редагування та імпорт проектних документів може відбуватися в одному єдиному веб-інтерфейсі.
- **Нижчий бар’єр входу**: імпорт шаблону дизайну одним клацанням миші допомагає швидко розпочати нові проекти.
- **Безпечний і надійний**: перевірка шляху, виявлення конфліктів версій і зручні механізми деградації забезпечують стабільність системи.
- **Покращена взаємодія з користувачем**: глобальна панель, діалогові вікна виявлення забруднень і підтвердження покращують загальну взаємодію.

Це рішення вже запущено в проекті HagiCode і вирішило проблемні моменти команди щодо керування проектними документами. Якщо ви зіткнулися з подібними проблемами, сподіваюся, ця стаття дасть вам корисні ідеї.
