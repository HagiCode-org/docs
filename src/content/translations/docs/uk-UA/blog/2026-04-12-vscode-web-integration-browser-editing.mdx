## Швидке редагування коду в браузері: веб-інтеграція VSCode на практиці

> Після того як штучний інтелект завершить аналіз коду, як відразу відкрити редактор у браузері та почати вносити зміни? У цій статті ми ділимося нашим практичним досвідом інтеграції сервера коду в проект HagiCode, щоб створити бездоганний міст між помічником ШІ та досвідом редагування коду.

## Фон

В еру програмування за допомогою ШІ розробникам часто доводиться швидко перевіряти та редагувати код. Традиційний робочий процес простий: відкрийте проект у настільному IDE, знайдіть файл, відредагуйте його та збережіть. Але в деяких ситуаціях цей потік завжди здається дещо збитим.

Сценарій перший: віддалена розробка. Під час використання помічника штучного інтелекту, такого як HagiCode, бекенд може працювати на віддаленому сервері або всередині контейнера, і локальні машини не можуть отримати прямий доступ до файлів проекту. Кожного разу, коли вам потрібно перевірити або змінити код, вам потрібно підключитися через SSH або інший метод, і досвід здається фрагментованим. Це як бажання зустрітися з кимось крізь товсте скло: ви бачите їх, але не можете до них дотягнутися.

Другий сценарій: швидкий попередній перегляд. Після того як помічник штучного інтелекту проаналізує код, користувач може лише захотіти швидко переглянути файл або внести невеликі зміни. Запуск повноцінної IDE для настільного комп’ютера здається важким, тоді як легкий редактор у браузері краще відповідає потребам у «швидкому перегляді». Зрештою, хто хоче мобілізувати весь ланцюжок інструментів, щоб просто кинути погляд?

Сценарій третій: співпраця між пристроями. Під час роботи на різних пристроях редактор на основі браузера забезпечує уніфіковану точку доступу, не вимагаючи налаштування середовища розробки на кожній машині. Тільки це позбавляє від багатьох проблем. Життя коротке; навіщо повторювати ту саму роботу з налаштування знову і знову?

Щоб вирішити ці проблеми, ми інтегрували VSCode Web у проект HagiCode. Це дозволяє помічнику штучного інтелекту та редагування коду легко з’єднуватися: після того, як штучний інтелект проаналізує код, користувачі можуть негайно відкрити редактор і внести зміни в тому ж сеансі браузера, не перемикаючи програми. Це такий досвід, де, коли він вам потрібен, він просто є.

Під час розробки ми виявили, що користувачам часто потрібно швидко перемикатися між аналізом штучного інтелекту та редагуванням коду, що спонукало нас дослідити, як інтегрувати редактор безпосередньо у веб-переглядач.

## Технічний вибір: чому саме сервер коду?

Серед багатьох доступних веб-рішень VSCode ми вибрали сервер коду. За таким рішенням було кілька конкретних причин.

Повнота функції. code-server — це веб-версія VSCode і підтримує більшість функцій настільного комп’ютера, включаючи систему розширення, інтелектуальні пропозиції, налагодження тощо. Це означає, що користувачі можуть отримати досвід редагування у браузері, який дуже схожий на настільну версію. Зрештою, хто насправді хоче йти на компроміс щодо функціональності?

Гнучке розгортання. code-server може працювати як незалежна служба, а також підтримує розгортання на основі Docker, що добре підходить до архітектури HagiCode. Наш бекенд написаний на C#, інтерфейс використовує React, і обидва взаємодіють зі службою кодового сервера через REST API. Це як будувати з блоків: кожна деталь має своє місце.

Безпечна автентифікація. код-сервер містить вбудований `connection-token` механізм запобігання несанкціонованому доступу. Кожен сеанс має унікальний маркер, тому лише авторизовані користувачі можуть відкрити редактор. Безпека — це одна з тих речей, які ви повністю цінуєте лише тоді, коли її маєте.

## Архітектурний дизайн

Веб-інтеграція VSCode від HagiCode використовує архітектуру, розділену на зовнішній і внутрішній.

### Рівень зовнішньої служби

Інтерфейс обгортає взаємодію з серверною частиною `vscodeServerService.ts`:

```typescript
// Open project
export async function openProjectInCodeServer(
  id: string,
  currentInterfaceLanguage?: string,
): Promise<VsCodeServerLaunchResponseDto>

// Open vault
export async function openVaultInCodeServer(
  id: string,
  path?: string,
  currentInterfaceLanguage?: string,
): Promise<VsCodeServerLaunchResponseDto>
```

Різниця між цими двома методами проста: `openProjectInCodeServer` відкриває весь проект, поки `openVaultInCodeServer` відкриває певний шлях усередині сховища. Для проектів MonoSpecs із кількома сховищами система автоматично створює файл робочої області. Часто достатньо чітких обов’язків, коли кожна частина добре виконує свою роботу.

### Рівень серверної служби

Бекенд `VaultAppService.cs` реалізує основну логіку:

```csharp
public async Task<VsCodeServerLaunchResponseDto> OpenInCodeServerAsync(
    string id,
    string? relativePath = null,
    string? currentInterfaceLanguage = null,
    CancellationToken cancellationToken = default)
{
    // 1. Get settings and check whether the feature is enabled
    var settings = await _vsCodeServerSettingsService.GetResolvedSettingsAsync(cancellationToken);
    if (!settings.Enabled) {
        throw new BusinessException(VsCodeServerErrorCodes.Disabled, "VSCode Server is disabled.");
    }

    // 2. Get vault and resolve the launch directory
    var vault = await RequireVaultAsync(id, cancellationToken);
    var launchDirectory = ResolveLaunchDirectory(vault, relativePath);

    // 3. Ensure code-server is running and get runtime info
    var runtime = await _vsCodeServerManager.EnsureStartedAsync(settings, cancellationToken);

    // 4. Resolve language settings
    var language = _vsCodeServerSettingsService.ResolveLaunchLanguage(
        settings.Language,
        currentInterfaceLanguage);

    // 5. Build launch URL
    return new VsCodeServerLaunchResponseDto {
        LaunchUrl = AppendQueryString(runtime.BaseUrl, new Dictionary<string, string?> {
            ["folder"] = launchDirectory,
            ["tkn"] = runtime.ConnectionToken,
            ["vscode-lang"] = language,
        }),
        ConnectionToken = runtime.ConnectionToken,
        OpenMode = "folder",
        Runtime = VsCodeServerSettingsService.MapRuntime(
            await _vsCodeServerManager.GetRuntimeSnapshotAsync(cancellationToken)),
    };
}
```

Цей метод має чітку відповідальність: перевірити налаштування, визначити шляхи, запустити службу та створити URL-адресу. Серед них, `ResolveLaunchDirectory` метод виконує перевірку безпеки шляху, щоб запобігти атакам проходження шляху. Код може виглядати трохи як поезія, коли кожен рядок має мету.

### Автоматичне керування часом виконання

Сервер керує процесом сервера коду `VsCodeServerManager`:

- Перевірте статус процесу
- Автоматично запускати зупинені служби
- Повертає знімки часу виконання, такі як порт, ідентифікатор процесу та час початку

Така конструкція дозволяє системі автоматично обробляти життєвий цикл кодового сервера, тому користувачам не потрібно керувати процесами обслуговування вручну. Життя вже досить складне; все, що можна автоматизувати, має бути автоматизованим.

### Синхронізація мови

HagiCode підтримує багатомовний інтерфейс, і сервер коду повинен дотримуватися цього налаштування. Система підтримує три мовні режими:

- `follow`: слідувати поточній мові інтерфейсу
- `zh-CN`: виправлено на китайську мову
- `en-US`: виправлено на англійську мову

Налаштування передається на сервер коду через `vscode-lang` Параметр URL-адреси, щоб мова редактора залишалася узгодженою з інтерфейсом HagiCode. Мова почувається найкраще, коли вона уніфікована.

Це дає змогу редагувати кілька підрепозиторіїв в одному екземплярі сервера коду, що особливо зручно для великих проектів монорепо. Кілька репозиторіїв в одному вікні можуть виглядати як кілька історій, зібраних в одній книзі.

## Інтеграція зовнішнього інтерфейсу

Інтерфейс HagiCode використовує React + TypeScript, а інтеграція кодового сервера не є особливо складною.

### Кнопка швидкої дії

Додайте кнопку сервера коду до картки проекту:

```tsx
// QuickActionsZone.tsx
<Button
  size="sm"
  variant="default"
  onClick={() => onAction({ type: 'open-code-server' })}
>
  <Globe className="h-3 w-3 mr-1" />
  <span className="text-xs">{t('project.openCodeServer')}</span>
</Button>
```

Ця кнопка запускає дію відкриття та викликає серверний API, щоб отримати URL-адресу запуску. Одна кнопка, одна дія, прямо та просто.

### Обробка відкритої дії

```tsx
const handleAction = async (action: ProjectAction) => {
  if (action.type === 'open-code-server') {
    const response = await openProjectInCodeServer(project.id, i18n.language);
    window.open(response.launchUrl, '_blank', 'noopener,noreferrer');
  }
};
```

використання `window.open` щоб відкрити кодовий сервер у новій вкладці. The `noopener,noreferrer` параметри забезпечують додаткову безпеку. Коли мова йде про безпеку, немає такого поняття, як бути надто обережним.

### Запис редагування сховища

Додайте схожу кнопку редагування в список сховища:

```tsx
const handleEditVault = async (vault: VaultItemDto) => {
  const response = await openVaultInCodeServer(vault.id);
  window.open(response.launchUrl, '_blank', 'noopener,noreferrer');
};
```

Проекти та сховища використовують той самий відкритий механізм, який забезпечує послідовність взаємодії. Узгодженість має значення майже так само, як і сама функція.

## Логіка побудови URL

Формат URL-адреси для кодового сервера має кілька деталей, які варто відзначити.

**Режим папки**:
```
http://{host}:{port}/?folder={path}&tkn={token}&vscode-lang={lang}
```

**Режим робочої області**:
```
http://{host}:{port}/?workspace={workspacePath}&tkn={token}&vscode-lang={lang}
```

тут, `tkn` є маркером підключення. Він генерується автоматично кожного разу, коли запускається сервер коду, забезпечуючи безпечний доступ. The `vscode-lang` Параметр керує мовою інтерфейсу редактора. Кожен із цих параметрів відіграє свою роль.

## Сценарії використання

### Сценарій перший: перевірка коду за допомогою ШІ

Користувач спілкується з HagiCode, штучний інтелект аналізує код проекту та знаходить потенційну проблему, а потім користувач натискає кнопку «Відкрити на сервері коду», щоб відкрити редактор безпосередньо в браузері, перевірити уражений файл, виправити його та повернутися до HagiCode, щоб продовжити розмову. Весь процес відбувається в браузері без перемикання програм. Він відчувається гладко, як гладка вода.

### Другий сценарій: редагування навчальних матеріалів у сховищі

Користувач створює сховище для вивчення проекту з відкритим вихідним кодом і хоче додати навчальні нотатки під `docs/` каталог. За допомогою сервера коду вони можуть редагувати файли Markdown безпосередньо в браузері, зберігати їх і дозволяти HagiCode негайно читати оновлені примітки. Це особливо корисно для створення особистої бази знань. Знання стають тим ціннішими, чим більше ви їх накопичуєте.

### Сценарій третій: розробка кількох сховищ MonoSpecs

Проект MonoSpecs містить кілька підрепозиторіїв, а сервер коду автоматично створює робочу область із кількома папками. У браузері користувачі можуть редагувати код у кількох сховищах одночасно, а потім повертати зміни до відповідних сховищ Git. Цей робочий процес особливо добре підходить для змін, які мають охопити кілька сховищ. Спільне редагування кількох сховищ потребує певної техніки, як і виконання кількох завдань одночасно.

## Міркування щодо безпеки

При реалізації інтеграції код-сервера безпека заслуговує на особливу увагу. Якщо безпека йде не так, ви завжди помічаєте це надто пізно.

### Маркер підключення

The `connection-token` генерується випадковим чином і не повинен бути відкритим. Його найкраще використовувати під протоколом HTTPS, щоб запобігти перехопленню маркера людиною посередині. Конфіденційну інформацію варто захищати належним чином.

### Безпека шляху до файлу

Сервер реалізує перевірку проходження шляху:

```csharp
private static string ResolveLaunchDirectory(VaultRegistryEntry vault, string? relativePath)
{
    var vaultRoot = EnsureTrailingSeparator(Path.GetFullPath(vault.PhysicalPath));
    var combinedPath = Path.GetFullPath(Path.Combine(vaultRoot, relativePath ?? "."));
    if (!combinedPath.StartsWith(vaultRoot, StringComparison.OrdinalIgnoreCase))
    {
        throw new BusinessException(VaultRelativePathTraversalCode, "Relative path traversal detected.");
    }
    return combinedPath;
}
```

Цей код гарантує, що користувачі не можуть використовувати `../` або подібні шаблони для доступу до файлів поза каталогом Vault. Перевірки кордонів завжди краще зробити, ніж пропустити.

### Контроль дозволів

Процес сервера коду має працювати з відповідними дозволами користувача, щоб він не мав доступу до конфіденційних системних файлів. Найкраще запускати службу кодового сервера під керуванням виділеного користувача. Контроль дозволів — це одна з тих основ, яку слід завжди дотримуватися.

## Оптимізація продуктивності

code-server споживає ресурси сервера, тому ось кілька пропозицій щодо оптимізації:

- Відстежуйте використання процесора та пам’яті та за потреби регулюйте обмеження ресурсів
- Для великих проектів може знадобитися довший тайм-аут
- Реалізуйте автоматичне очищення часу очікування сеансу, щоб звільнити ресурси
- Розгляньте можливість кешування, щоб зменшити кількість повторних обчислень

HagiCode надає API для моніторингу стану виконання, і інтерфейс може викликати `getVsCodeServerSettings()` щоб отримати поточний стан:

```typescript
const { settings, runtime } = await getVsCodeServerSettings();
// runtime.status: 'disabled' | 'stopped' | 'starting' | 'running' | 'unhealthy'
// runtime.baseUrl: "http://localhost:8080"
// runtime.processId: 12345
```

Цей дизайн дозволяє користувачам чітко розуміти стан працездатності сервера кодів і швидко виявляти проблеми, коли щось йде не так. Коли статус видно, люди відчувають більше контролю.

## Деталі взаємодії з користувачем

Під час впровадження ми виявили кілька деталей, які помітно впливають на взаємодію з користувачем і заслуговують на додаткову увагу.

Перше відкриття сервера коду може вимагати очікування запуску, і ця затримка може становити від кількох секунд до півхвилини. Було б гарною ідеєю показати стан завантаження у інтерфейсі, щоб користувачі знали, що система все ще працює. Чекати легше, коли є зворотний зв’язок.

Браузери можуть блокувати спливаюче вікно, тому користувачам слід запропонувати дозволити це вручну. Під час першого запуску HagiCode відображає вказівки, які пояснюють, як надати необхідні дозволи браузеру. Користувацький досвід часто живе саме в цих дрібних деталях.

Також доцільно відображати статус виконання, як-от запуск, запуск або помилка, щоб у разі виникнення проблем користувачі могли швидко визначити, на стороні сервера чи в їхній власній роботі. Знаючи, де є проблема, ви принаймні зможете почати.

## Приклад конфігурації

Конфігурація код-сервера не складна:

```json
{
  "vscodeServer": {
    "enabled": true,
    "host": "0.0.0.0",
    "port": 8080,
    "language": "follow"
  }
}
```

`enabled` контролює, чи ввімкнено функцію, `host` і `port` визначте адресу прослуховування та `language` встановлює мовний режим. Ці параметри можна змінити за допомогою інтерфейсу користувача, і вони негайно набудуть чинності. Прості речі часто найлегші у використанні.

## Висновок

Веб-інтеграція VSCode від HagiCode забезпечує елегантне рішення: воно дозволяє безпроблемно з’єднати помічника зі штучним інтелектом і редагування коду. Інтегрувавши сервер коду в браузер, користувачі можуть швидко діяти на основі результатів аналізу штучного інтелекту та завершити весь процес від аналізу до редагування в тому самому сеансі браузера.

Це рішення має кілька ключових переваг: уніфікований досвід, оскільки проекти та сховища використовують той самий відкритий механізм; підтримка кількох репозиторіїв, оскільки проекти MonoSpecs автоматично створюють робочі області; і керована безпека завдяки моніторингу стану під час виконання та перевіркам безпеки шляху.

Підхід, опублікований у цій статті, — це те, що HagiCode виділило з реальних розробників. Якщо ви вважаєте це рішення цінним, це свідчить про те, що наша інженерна практика робить щось правильно, і сам HagiCode, можливо, варто розглянути ближче.
