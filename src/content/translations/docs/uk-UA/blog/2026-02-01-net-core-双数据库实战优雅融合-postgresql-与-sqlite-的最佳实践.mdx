## Двоядерна база даних .NET Core на практиці: нехай PostgreSQL і SQLite мирно співіснують

> Створюючи сучасні програми, ми часто стикаємося з таким компромісом: середовища розробки хочуть чогось легкого та зручного, а виробничі середовища вимагають високого паралелізму та високої доступності. У цій статті розповідається про те, як елегантно підтримувати PostgreSQL і SQLite у проекті .NET Core і реалізовувати найкращі практики «SQLite для розробки, PostgreSQL для виробництва».


## Фон

У розробці програмного забезпечення відмінності між середовищами завжди були однією з найскладніших проблем для команд інженерів. Візьмемо для прикладу платформу **HagiCode**, яку ми створюємо: це система розробки за допомогою штучного інтелекту на основі ASP.NET Core 10 і React з внутрішньою інтеграцією Orleans для розподіленого управління станом. Стек сучасний і досить складний.

На початку проекту ми зіткнулися з класичною інженерною проблемою: розробники хотіли, щоб локальне середовище працювало з коробки, без необхідності встановлювати та налаштовувати важку базу даних PostgreSQL. Але у виробництві нам потрібно було обробляти високопаралелізований запис і складні запити JSON, і саме тут легкий SQLite починає демонструвати свої межі.

Як ми можемо зберегти єдину кодову базу, дозволяючи програмі використовувати переваги переносимості SQLite, як настільна програма, а також використовувати потужну продуктивність PostgreSQL, як сервіс корпоративного рівня? Це головне питання, яке досліджує ця стаття.

## Основна тема 1: Дизайн архітектури та уніфікована абстракція

Для підтримки двох баз даних у .NET Core ключовою ідеєю є залежність від абстракцій, а не конкретних реалізацій. Нам потрібно відокремити вибір бази даних від бізнес-коду та дозволити рівні конфігурації вирішити.

### Підхід до проектування

1.  **Уніфікований інтерфейс**: уся бізнес-логіка має залежати від `DbContext` базовий клас або спеціальні інтерфейси, а не певний `PostgreSqlDbContext`.
2.  **На основі конфігурації**: використовуйте елементи конфігурації в `appsettings.json` щоб динамічно вирішувати, який постачальник бази даних завантажувати під час запуску програми.
3.  **Ізоляція функцій**: додайте логіку адаптації для спеціальних можливостей PostgreSQL, таких як JSONB, щоб програма все ще могла плавно погіршуватися на SQLite.

### Реалізація коду: конфігурація динамічного контексту

У ASP.NET Core `Program.cs`, ми не повинні жорстко кодувати `UseNpgsql` або `UseSqlite`. Замість цього ми повинні прочитати конфігурацію та прийняти рішення динамічно.

Спочатку визначте клас конфігурації:

```csharp
public class DatabaseSettings
{
    public const string SectionName = "Database";
    
    // Database type: PostgreSQL or SQLite
    public string DbType { get; set; } = "PostgreSQL"; 
    
    // Connection string
    public string ConnectionString { get; set; } = string.Empty;
}
```

Потім зареєструйте послугу в `Program.cs` на основі конфігурації:

```csharp
// Read configuration
var databaseSettings = builder.Configuration.GetSection(DatabaseSettings.SectionName).Get<DatabaseSettings>();

// Register DbContext
builder.Services.AddDbContext<ApplicationDbContext>(options =>
{
    if (databaseSettings?.DbType?.ToLower() == "sqlite")
    {
        // SQLite configuration
        options.UseSqlite(databaseSettings.ConnectionString);
        
        // Handling SQLite's concurrent write limitations
        // Note: in production, enabling WAL mode is recommended to improve concurrency performance
    }
    else
    {
        // PostgreSQL configuration (default)
        options.UseNpgsql(databaseSettings.ConnectionString, npgsqlOptions =>
        {
            // Enable JSONB support, which is very useful when handling AI conversation records
            npgsqlOptions.UseJsonNet(); 
        });
        
        // Configure connection pool retry policy
        options.EnableRetryOnFailure(3);
    }
});
```

## Основна тема 2: Подолання розбіжностей і стратегія міграції

Хоча PostgreSQL і SQLite підтримують стандарт SQL, вони суттєво відрізняються конкретними можливостями та поведінкою. Якщо не врахувати ці відмінності, ви можете легко потрапити в незручну ситуацію, коли все працює локально, але дає збій після розгортання.

### 1. Обробка типів JSON

У HagiCode нам потрібно зберігати велику кількість підказок і метаданих AI, які зазвичай містять стовпці JSON.
- **PostgreSQL**: має нативну версію `JSONB` типу з чудовою продуктивністю запитів.
- **SQLite**: не має рідного типу JSON (новіші версії включають розширення JSON1, але відображення об’єктів усе ще відрізняється), тому дані зазвичай зберігаються як ТЕКСТ.

**Рішення**:
У відображенні сутності EF Core ми налаштовуємо її як конвертований тип.

```csharp
protected override void OnModelCreating(ModelBuilder modelBuilder)
{
    base.OnModelCreating(modelBuilder);

    // Configure entity
    modelBuilder.Entity<PromptTemplate>(entity => 
    {
        entity.Property(e => e.Metadata)
              .HasColumnType("jsonb") // PG uses jsonb
              .HasConversion(
                  v => JsonSerializer.Serialize(v, (JsonSerializerOptions)null),
                  v => JsonSerializer.Deserialize<Dictionary<string, object>>(v, (JsonSerializerOptions)null)
              ); 
    });
}
```

Коли використовується SQLite, `HasColumnType("jsonb")` може бути проігнорований або викликати попередження. Однак тому що `HasConversion` налаштовано, дані все ще правильно серіалізуються та десеріалізуються як рядки, що зберігаються в полі TEXT, що забезпечує сумісність.

### 2. Розділення стратегій міграції

Ніколи не намагайтеся змусити один набір сценаріїв міграції працювати як для PostgreSQL, так і для SQLite одночасно. Відмінності в стратегіях генерації первинного ключа, синтаксисі індексів та інших деталях бази даних неминуче призведуть до збоїв.

**Рекомендована практика**:
Підтримувати дві міграційні гілки або проекти. У робочому процесі розробки HagiCode ми обробляємо це таким чином:

1.  **Стадія розробки**: Працюйте переважно з SQLite. використання `Add-Migration Init_Sqlite -OutputDir Migrations/Sqlite`.
2.  **Етап адаптації**: після розробки функції переключіть рядок підключення на PostgreSQL і запустіть `Add-Migration Init_Postgres -OutputDir Migrations/Postgres`.
3.  **Сценарії автоматизації**: напишіть простий сценарій PowerShell або Bash для автоматичного застосування правильної міграції на основі поточної змінної середовища.

```bash
# Pseudocode for simple deployment logic
if [ "$DATABASE_PROVIDER" = "PostgreSQL" ]; then
    dotnet ef database update --project Migrations.Postgres
else
    dotnet ef database update --project Migrations.Sqlite
fi
```

## Основна тема 3: Уроки, отримані з HagiCode у виробництві

Під час рефакторингу **HagiCode** з моделі з однією базою даних на підтримку двох баз даних ми натрапили на кілька підводних каменів і зібрали кілька важливих уроків, які можуть допомогти вам уникнути тих самих помилок.

### 1. Відмінності в паралелізмі та транзакціях

PostgreSQL використовує архітектуру сервер-клієнт і підтримує високопаралелізований запис із потужними рівнями ізоляції транзакцій. SQLite використовує блокування файлів, тому операції запису блокують весь файл бази даних, якщо не ввімкнено режим WAL.

**Рекомендація**:
Під час написання бізнес-логіки, яка включає часті записи, такі як збереження в режимі реального часу стану редагування користувача, ви повинні взяти до уваги модель блокування SQLite. Під час розробки модуля співпраці OpenSpec у **HagiCode** ми запровадили механізм «злиття перед записом», щоб зменшити частоту прямих записів до бази даних, дозволяючи нам підтримувати хорошу продуктивність обох баз даних.

### 2. Управління життєвим циклом рядків підключення

Встановлення з’єднання PostgreSQL є відносно дорогим і залежить від пулу з’єднань. З’єднання SQLite дуже легкі, але якщо їх не звільнити негайно, блокування файлів може призвести до завершення пізніших операцій.

в `Program.cs`, ми можемо точно налаштувати поведінку кожної бази даних:

```csharp
if (databaseSettings?.DbType?.ToLower() == "sqlite")
{
    // SQLite: keeping connections open can improve performance, but watch out for file locks
    options.UseSqlite(connectionString, sqliteOptions =>
    {
        // Set command timeout
        sqliteOptions.CommandTimeout(30);
    });
}
else
{
    // PG: make full use of connection pooling
    options.UseNpgsql(connectionString, npgsqlOptions =>
    {
        npgsqlOptions.MaxBatchSize(100);
        npgsqlOptions.CommandTimeout(30);
    });
}
```

### 3. Важливість тестового покриття

Багато розробників, у тому числі деякі перші члени нашої команди, схильні робити одну поширену помилку: запускати модульні тести лише в середовищі розробки, яким зазвичай є SQLite.

У конвеєрі CI/CD HagiCode ми застосували крок GitHub Actions, щоб переконатися, що кожен запит на отримання виконує тести інтеграції PostgreSQL.

```yaml
# Example snippet from .github/workflows/test.yml
- name: Run Integration Tests (PostgreSQL)
  run: |
    docker-compose up -d db_postgres
    dotnet test --filter "Category=Integration"
```

Це допомогло нам виявити незліченні помилки, пов’язані з відмінностями синтаксису SQL і чутливістю до регістру.

Щоб підсумувати ключові моменти:
1.  **Спочатку абстракція**: бізнес-код не повинен залежати від конкретних реалізацій бази даних.
2.  **Окрема конфігурація**: використовуйте різні `appsettings.json` файли для розробки та виробництва.
3.  **Окремі міграції**: не намагайтеся змусити один набір міграції працювати всюди.
4.  **Погіршення функцій**: надайте пріоритет сумісності в SQLite та продуктивності в PostgreSQL.
