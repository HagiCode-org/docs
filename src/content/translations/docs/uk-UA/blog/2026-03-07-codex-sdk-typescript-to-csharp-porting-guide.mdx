## Від TypeScript до C#: практика міжмовного перенесення для Codex SDK

> Простіше кажучи, ця стаття також є нашою немовлятою: у ній описано повний процес перенесення офіційного TypeScript Codex SDK на C#. Називаючи це «портом», це майже звучить надто легко — це було більше схоже на довгу пригоду, оскільки ці дві мови мають дуже різні особливості, і ми повинні були знайти спосіб змусити їх співпрацювати.

## Фон

Codex — це інструмент CLI агента AI, випущений OpenAI, і він справді потужний. Офіційна команда надає TypeScript SDK у `@openai/codex` пакет. Він взаємодіє з Codex CLI, викликаючи `codex exec --experimental-json` і розбір потоку подій JSONL.

Проблема полягає в тому, що в проекті HagiCode нам потрібно використовувати його в чистому середовищі .NET - зокрема, у базових службах C# і настільних програмах. Ми не могли ввести середовище виконання Node.js у проект .NET просто для виклику інструменту CLI. Це було б надто громіздко.

Тож у нас було два вибори: підтримувати складний рівень мосту Node.js або самостійно створювати нативний C# SDK.

Ми вибрали останнє.

Ця стаття також походить безпосередньо з нашого практичного досвіду в [HagiCode](https://hagicode.com) проект. HagiCode — це проект помічника кодування ШІ з відкритим кодом. Простими словами, це означає підтримку кількох компонентів одночасно: розширення VSCode на інтерфейсі, служби ШІ на сервері та кросплатформний клієнт для настільного комп’ютера. Ця багатомовна та багатоплатформна складність саме тому, чому нам потрібен рідний C# SDK — ми справді не хотіли запускати Node.js у проекті .NET.

## Основний вміст

### Порівняння архітектурного дизайну

Перш ніж перекладати код один до одного, ми спочатку повинні були зрозуміти архітектурний дизайн обох SDK. Вам потрібно зрозуміти обидві сторони, перш ніж ви зможете їх добре перенести.

Основна архітектура TypeScript SDK виглядає так:

```
Codex (entry class)
  └── CodexExec (executor, manages child processes)
      └── Thread (conversation thread)
          ├── run() / runStreamed() (synchronous/asynchronous execution)
          └── event stream parsing
```

C# SDK зберігає ті самі архітектурні рівні, але адаптує деталі реалізації. Загальна ідея проста: зберегти узгодженість API, повністю використовуючи функції мови C# у реалізації.

### Перетворення типу системи

Це найважливіша і водночас найважливіша частина роботи. Якщо основа слабка, все, що йде далі, стає твердішим.

Система типів TypeScript більш гнучка, ніж C#, і це просто факт. Нам потрібно було знайти відповідну стратегію відображення:

| TypeScript | C# | Примітки |
|------------|-----|------|
| `interface` / `type` | `record` | Використання C# `record` для незмінних структур даних |
| `рядок \| null` | `string?` | Nullable тип посилання |
| `логічний \| невизначений` | `bool?` | Логічне значення з можливістю нульового значення |
| `AsyncGenerator` | `IAsyncEnumerable` | Асинхронний ітератор |

Типовим прикладом є **система типів подій**. TypeScript використовує об’єднані типи для визначення подій:

```typescript
export type ThreadEvent =
  | ThreadStartedEvent
  | TurnStartedEvent
  | TurnCompletedEvent
  | ...
```

У C# ми використовуємо ієрархію успадкування та зіставлення шаблонів, щоб досягти подібного ефекту:

```csharp
public abstract record ThreadEvent(string Type);

public sealed record ThreadStartedEvent(string ThreadId) : ThreadEvent("thread.started");
public sealed record TurnStartedEvent() : ThreadEvent("turn.started");
public sealed record TurnCompletedEvent(Usage Usage) : ThreadEvent("turn.completed");
// ...
```

Ми вибрали `record` замість `class` оскільки об’єкти подій мають бути незмінними, що відповідає наміру використання простих об’єктів у TypeScript. The `sealed` Ключове слово також запобігає додатковому успадкуванню та дає компілятору простір для оптимізації.

### Ключові моменти портування

#### 1. Парсер подій

Синтаксичний аналіз подій є ядром усього SDK, оскільки він визначає, чи зможемо ми правильно зрозуміти кожне повідомлення, повернуте Codex CLI. Якщо синтаксичний аналіз неправильний, усе після цього буде витраченим зусиллям.

Версія TypeScript використовує `JSON.parse()` щоб розібрати кожен рядок JSON:

```typescript
export function parseEvent(line: string): ThreadEvent {
  const data = JSON.parse(line);
  // Handle different event types...
}
```

Версія C# використовує `System.Text.Json.JsonDocument` замість цього:

```csharp
public static ThreadEvent Parse(string line)
{
    using var document = JsonDocument.Parse(line);
    var root = document.RootElement;
    var type = GetRequiredString(root, "type", "event.type");

    return type switch
    {
        "thread.started" => new ThreadStartedEvent(GetRequiredString(root, "thread_id", ...)),
        "turn.started" => new TurnStartedEvent(),
        "turn.completed" => new TurnCompletedEvent(ParseUsage(...)),
        // ...
        _ => new UnknownThreadEvent(type, root.Clone()),
    };
}
```

Тут є одна маленька, але важлива хитрість: `root.Clone()` є обов’язковим, оскільки елементи з `JsonDocument` стає недійсним після утилізації документа. Нам потрібно зберегти копію для невідомих типів подій. Це просто одна з відмінностей між обробкою C# JSON і JavaScript.

#### 2. Відмінності в управлінні процесами

У цьому два SDK відрізняються найбільше. Node.js і .NET мають різні угоди про час виконання, тому реалізацію потрібно адаптувати.

TypeScript використовує Node.js `spawn()` функція:

```typescript
const child = spawn(this.executablePath, commandArgs, { env, signal });
```

C# використовує .NET `System.Diagnostics.Process`:

```csharp
using var process = new Process { StartInfo = startInfo };
process.Start();

// stdin/stdout/stderr must be managed manually
```

Точніше, версія C# повинна налаштувати процес таким чином:

```csharp
var startInfo = new ProcessStartInfo
{
    FileName = _executablePath,
    RedirectStandardInput = true,
    RedirectStandardOutput = true,
    RedirectStandardError = true,
    UseShellExecute = false,
    CreateNoWindow = true,
};
```

Найбільшою відмінністю є механізм скасування. TypeScript використовує `AbortSignal`, який є частиною Web API і дуже зручний для роботи:

```typescript
const child = spawn(cmd, args, { signal: cancellationSignal });
```

Використання C# `CancellationToken` замість цього:

```csharp
public async IAsyncEnumerable<string> RunAsync(
    CodexExecArgs args,
    [EnumeratorCancellation] CancellationToken cancellationToken = default)
{
    // Check cancellation status inside the loop
    while (!cancellationToken.IsCancellationRequested)
    {
        // Process output...
    }

    // Terminate the process when cancellation is requested
    if (cancellationToken.IsCancellationRequested)
    {
        try { process.Kill(entireProcessTree: true); } catch { }
    }
}
```

На високому рівні це лише ще один приклад різниці між екосистемою Web API та екосистемою .NET.

#### 3. Збереження серіалізації конфігурації

Обидва SDK реалізують логіку, яка перетворює конфігурацію JSON у конфігурацію TOML, оскільки Codex CLI приймає перевизначення конфігурації у форматі TOML. Ця частина має залишатися повністю узгодженою, інакше та сама конфігурація поводитиметься по-різному в двох SDK.

Це та деталь, з якою ви не можете піти на компроміс. Успіх чи невдача часто зводяться до таких деталей.

### Деталі реалізації

#### Структура проекту

Ми створили таку структуру проекту:

```
CodexSdk/
├── CodexSdk.csproj
├── Codex.cs           # Entry class
├── CodexThread.cs     # Conversation thread
├── CodexExec.cs       # Executor
├── Events.cs          # Event type definitions
├── Items.cs           # Item type definitions
├── EventParser.cs     # Event parser
├── OutputSchemaTempFile.cs  # Temporary file management
└── ...
```

Це досить чиста структура, і це дуже допомогло під час порту.

#### Приклад використання

Основне використання залишається узгодженим із TypeScript SDK:

```csharp
using CodexSdk;

// Create a Codex instance
var codex = new Codex();
var thread = codex.StartThread();

// Execute a query
var result = await thread.RunAsync("Summarize this repository.");
Console.WriteLine(result.FinalResponse);
```

Обробка потокових подій використовує переваги зіставлення шаблонів C#:

```csharp
await foreach (var @event in thread.RunStreamedAsync("Analyze the code."))
{
    switch (@event)
    {
        case ItemCompletedEvent itemCompleted
            when itemCompleted.Item is AgentMessageItem msg:
            Console.WriteLine($"Assistant: {msg.Text}");
            break;
        case TurnCompletedEvent completed:
            Console.WriteLine($"Tokens: in={completed.Usage.InputTokens}");
            break;
        case CommandExecutionItem command:
            Console.WriteLine($"Command: {command.Command}");
            break;
    }
}
```

### Примітки

Під час впровадження ми зібрали кілька практичних уроків:

1. **Керування процесом**: версія C# повинна керувати повним життєвим циклом процесу вручну, включаючи завершення процесу під час скасування. використання `Kill(entireProcessTree: true)` щоб переконатися, що дочірні процеси також очищені.

2. **Обробка помилок**: ми використовуємо `InvalidOperationException` щоб викликати помилки аналізу, зберігаючи стиль обробки помилок, подібний до TypeScript SDK.

3. **Очищення ресурсу**: `OutputSchemaTempFile` інвентар `IAsyncDisposable` щоб забезпечити правильне очищення тимчасових файлів.

4. **Змінні середовища**: версія C# підтримує повне перевизначення змінних середовища процесу `CodexOptions.Env`. Це невелика функція, але дуже практична.

5. **Відмінності платформи**: версія C# не включає логіку версії TypeScript для автоматичного пошуку двійкових файлів у пакетах npm. Оскільки проекти .NET зазвичай не залежать від npm, шлях до `codex` виконуваний файл має бути вказано через `CODEX_EXECUTABLE` змінна середовища або `CodexPathOverride`.

## Висновок

Перенесення зрілого TypeScript SDK на C# — це не лише питання перетворення синтаксису — це також вимагає розуміння філософії дизайну обох мов. Гнучкість TypeScript і такі функції екосистеми JavaScript, як `AbortSignal` потрібні відповідні аналоги в C#.

Ключовий висновок такий: **підтримка узгодженості API важливіша, ніж підтримка узгодженості на рівні реалізації**. Користувачів цікавить те, чи простий у використанні інтерфейс, а не те, чи ідентична внутрішня реалізація. Це звучить просто, але прийняття цих компромісів вимагає розсудливості.

Якщо ви працюєте над подібним міжмовним портом, наш досвід полягає в тому, щоб спочатку повністю зрозуміти оригінальну архітектуру SDK, потім перекласти її модуль за модулем і, нарешті, використати повний набір тестів для забезпечення узгодженості поведінки. Таку роботу не можна поспішати.

Зрештою все вийде.
