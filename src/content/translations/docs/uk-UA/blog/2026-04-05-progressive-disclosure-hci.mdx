## Поступове розкриття інформації: покращення взаємодії людини та комп’ютера в продуктах штучного інтелекту за принципом «менше — краще»

> У розробці продукту штучного інтелекту якість введення користувачами часто визначає якість результату. У цій статті розповідається про підхід взаємодії «поступового розкриття інформації», який ми застосовували в проекті HagiCode. Завдяки покроковим інструкціям, інтелектуальному завершенню та миттєвому зворотному зв’язку він перетворює короткі та розпливчасті введення користувачів у структуровані технічні пропозиції, значно покращуючи ефективність взаємодії людини з комп’ютером.

## Фон

Будь-хто, хто розробляє продукти штучного інтелекту, напевно, бачив таку ситуацію: користувач відкриває вашу програму та з ентузіазмом набирає однорядковий запит, лише для того, щоб штучний інтелект повернув щось зовсім не цільове. Справа не в тому, що ШІ недостатньо розумний. Користувач просто не надав достатньо інформації. Читання думок важко для будь-кого.

Ця проблема стала особливо очевидною під час створення HagiCode. HagiCode — це помічник кодування на основі ШІ, де користувачі описують вимоги природною мовою для створення технічних пропозицій і сеансів. Під час фактичного використання ми виявили, що введення користувачами часто має такі проблеми:

- **Невідповідна якість введення**: деякі користувачі вводять лише кілька слів, наприклад «оптимізувати вхід» або «виправити помилку», без необхідного контексту
- **Непослідовна технічна термінологія**: різні користувачі використовують різні терміни для того самого; одні кажуть "frontend", а інші кажуть "FE"
- **Відсутня структурована інформація**: немає передісторії проекту, обсягу сховища чи масштабу впливу, навіть якщо це важливі деталі
- **Повторювані проблеми**: одні і ті ж типи запитів з'являються знову і знову, і щоразу їх потрібно пояснювати з нуля

Прямий результат передбачуваний: штучному інтелекту важче зрозуміти запит, якість пропозиції стає нестабільною, а взаємодія з користувачем страждає. Користувачі думають: «Цей штучний інтелект не дуже хороший», а ми почуваємося несправедливо звинуваченими. Якщо ви дасте мені лише одне речення, як я можу здогадатися, чого ви насправді хочете?

Насправді це зрозуміло. Навіть людям потрібен час, щоб зрозуміти один одного, і машини не є винятком.

Щоб вирішити ці проблеми, ми прийняли сміливе рішення: запровадили принцип «поступового розкриття інформації» для покращення взаємодії людини з комп’ютером. Зміни, які це принесло, ймовірно, були більшими, ніж ви могли собі уявити. Чесно кажучи, тоді ми не очікували, що це буде настільки ефективно.

## Що таке прогресивне розкриття

«Прогресивне розкриття» — це принцип дизайну з області HCI (взаємодія людина-комп’ютер). Його основна ідея проста: **не показувати користувачам всю інформацію та параметри одночасно. Натомість розкривайте лише те, що необхідно крок за кроком, виходячи з дій і потреб користувача**.

Цей принцип особливо підходить для продуктів штучного інтелекту, оскільки взаємодія штучного інтелекту природно прогресивна. Користувач говорить трохи, ШІ розуміє трохи, потім користувач додає більше, а ШІ розуміє більше. Це дуже схоже на те, як люди спілкуються один з одним: розуміння зазвичай розвивається поступово.

У сценарії HagiCode ми застосували прогресивне розкриття інформації в чотирьох областях:

### 1. Механізм оптимізації опису: нехай штучний інтелект допоможе вам сказати речі більш чітко

Коли користувач вводить короткий опис, ми не надсилаємо його безпосередньо ШІ для інтерпретації. Натомість ми спочатку запускаємо потік «оптимізації опису». Основою цього потоку є «структурований вихід»: перетворення вільного тексту користувача у стандартний формат. Це схоже на нанизування розсипаних перлин у намисто, щоб усе було легше зрозуміти.

Оптимізований опис повинен містити такі стандартні розділи:

- **Історія**: фон і контекст проблеми
- **Аналіз**: технічний аналіз і міркування
- **Рішення**: рішення та кроки впровадження
- **Практика**: конкретні приклади коду та примітки

У той же час ми автоматично генеруємо таблицю Markdown із такою інформацією, як цільове сховище, шляхи та дозволи на редагування, що полегшує наступні операції ШІ. Зрозумілий каталог завжди полегшує пошук.

Ось фактична реалізація:

```csharp
// Core method in ProposalDescriptionMemoryService.cs
public async Task<string> OptimizeDescriptionAsync(
    string title,
    string description,
    string locale = "zh-CN",
    DescriptionOptimizationMemoryContext? memoryContext = null,
    CancellationToken cancellationToken = default)
{
    // Build query parameters
    var queryContext = BuildQueryContext(title, description);

    // Retrieve historical context
    var memoryContext = await RetrieveHistoricalContextAsync(queryContext, cancellationToken);

    // Generate a structured prompt
    var prompt = await BuildOptimizationPromptAsync(
        title,
        description,
        memoryContext,
        cancellationToken);

    // Call AI for optimization
    return await _aiService.CompleteAsync(prompt, cancellationToken);
}
```

Ключем до цього потоку є «ін’єкція пам’яті». Ми впроваджуємо історичний контекст, такий як умовності проекту, подібні випадки та негативні шаблони в підказку, дозволяючи ШІ посилатися на минулий досвід під час оптимізації. Досвід не повинен пропадати даремно.

**Примітки:**
- Переконайтеся, що поточний вхід має пріоритет над історичною пам’яттю, тому явно вказана інформація користувача не перевизначається
- Посилання на HagIndex слід розглядати як фактичні джерела і не повинні змінюватися історичними випадками
- Пропозиції виправлення з низьким рівнем достовірності не слід вводити як сильні обмеження

### 2. Можливість голосового введення: говорити природніше, ніж друкувати

Окрім введення тексту, ми також підтримуємо голосове введення. Це особливо корисно для опису складних вимог. Введення технічного запиту може зайняти кілька хвилин, тоді як виголошення його вголос може зайняти лише кілька десятків секунд.

Основним фокусом розробки для голосового введення є «управління станом». Користувачі повинні чітко знати, в якому стані зараз перебуває система. Ми визначили такі стани:

- **В режимі очікування**: система готова і можна починати запис
- **Очікування вгорі**: система підключається до серверної служби
- **Запис**: записується голос користувача
- **Обробка**: мова перетворюється на текст
- **Помилка**: сталася помилка, яка потребує уваги користувача

Модель зовнішнього стану виглядає приблизно так:

```typescript
interface VoiceInputState {
  status: 'idle' | 'waiting-upstream' | 'recording' | 'processing' | 'error';
  duration: number;
  error?: string;
  deletedSet: Set<string>; // Fingerprint set of deleted results
}

// State transition when recording starts
const handleVoiceInputStart = async () => {
  // Enter waiting state first and show a loading animation
  setState({ status: 'waiting-upstream' });

  // Wait for backend readiness confirmation
  const isReady = await waitForBackendReady();
  if (!isReady) {
    setState({ status: 'error', error: 'Backend service is not ready' });
    return;
  }

  // Start recording
  setState({ status: 'recording', startTime: Date.now() });
};

// Handle recognition results
const handleRecognitionResult = (result: RecognitionResult) => {
  const fingerprint = normalizeFingerprint(result.text);

  // Check whether it has already been deleted
  if (state.deletedSet.has(fingerprint)) {
    return; // Skip deleted content
  }

  // Merge the result into the text box
  appendResult(result);
};
```

Тут є важлива деталь: ми використовуємо «набір відбитків пальців» для керування синхронізацією видалення. Якщо розпізнавання мовлення повертає кілька результатів, користувачі можуть видалити деякі з них. Ми зберігаємо відбитки пальців видаленого вмісту, щоб, якщо той самий вміст з’явився знову пізніше, його було автоматично пропущено. По суті, це спосіб запам’ятати те, що користувач уже відхилив.

### 3. Система швидкого управління: зовнішній «мозок» ШІ

HagiCode має гнучку систему керування підказками, у якій усі підказки зберігаються у вигляді файлів:

```
prompts/
├── metadata/
│   ├── optimize-description.zh-CN.json
│   └── optimize-description.en-US.json
└── templates/
    ├── optimize-description.zh-CN.hbs
    └── optimize-description.en-US.hbs
```

Кожна підказка складається з двох частин:
- **Файл метаданих (`.json`)**: визначає таку інформацію, як сценарій запиту, версія та параметри
- **Файл шаблону (`.hbs`)**: фактичний вміст підказки, написаний за допомогою синтаксису Handlebars

Формат файлу метаданих виглядає так:

```json
{
  "scenario": "optimize-description",
  "locale": "zh-CN",
  "version": "1.0.0",
  "syntax": "handlebars",
  "syntaxVersion": "1.0",
  "parameters": [
    {
      "name": "title",
      "type": "string",
      "required": true,
      "description": "Proposal title"
    },
    {
      "name": "description",
      "type": "string",
      "required": true,
      "description": "Original description"
    }
  ],
  "author": "HagiCode Team",
  "description": "Optimize the user's technical proposal description",
  "lastModified": "2026-04-05",
  "tags": ["optimization", "nlp"]
}
```

Файл шаблону використовує синтаксис Handlebars і підтримує впровадження параметрів:

```handlebars
You are a technical proposal expert.

<task>
Generate a structured technical proposal description based on the following information.
</task>

<input>
<title>{{title}}</title>
<description>{{description}}</description>
{{#if memoryContext}}
<memory_context>
{{memoryContext}}
</memory_context>
{{/if}}
</input>

<output_format>
## Background
[Describe the problem background and context, including project information, repository scope, and so on]

## Analysis
[Provide the technical analysis and reasoning process, and explain why this change is needed]

## Solution
[Provide the solution and implementation steps, listing the key code locations]

## Practice
[Provide concrete code examples and notes]
</output_format>
```

Переваги цієї конструкції очевидні:
- підказки можуть керуватися версіями, як і код
- підтримуються кілька мов, які можна автоматично перемикати залежно від уподобань користувача
- параметризований дизайн дозволяє динамічно вводити контекст
- повноту можна перевірити під час запуску, уникаючи помилок під час виконання

Якщо знання залишаються тільки в голові, їх легко втратити. Записати його структуровано з самого початку набагато безпечніше.

### 4. Прогресивний майстер: розділіть складні завдання на маленькі кроки

Для складних завдань, таких як перша інсталяція та конфігурація, ми використовуємо багатоетапний дизайн майстра. Кожен крок вимагає лише необхідної інформації та надає чіткі індикатори прогресу. Великі завдання стають набагато легшими, якщо виконувати їх крок за кроком.

Модель стану майстра:

```typescript
interface WizardState {
  currentStep: number;           // 0-3, corresponding to 4 steps
  steps: WizardStep[];
  canGoNext: boolean;
  canGoBack: boolean;
  isLoading: boolean;
  error: string | null;
}

interface WizardStep {
  id: number;
  title: string;
  description: string;
  completed: boolean;
}

// Step navigation logic
const goToNextStep = () => {
  if (wizardState.currentStep < wizardState.steps.length - 1) {
    // Validate input for the current step
    if (validateCurrentStep()) {
      wizardState.currentStep++;
      wizardState.steps[wizardState.currentStep - 1].completed = true;
    }
  }
};

const goToPreviousStep = () => {
  if (wizardState.currentStep > 0) {
    wizardState.currentStep--;
  }
};
```

Кожен крок має власну логіку перевірки, а виконані кроки отримують чіткі візуальні позначки. Скасування відкриває діалогове вікно підтвердження, щоб користувачі не втратили прогрес через випадкові дії.

## Висновок

Озираючись на прогресивну практику розкриття HagiCode, ми можемо підсумувати кілька основних принципів:

1. **Покрокові інструкції**: розділіть складні завдання на менші етапи та запитуйте лише необхідну інформацію на кожному етапі
2. **Розумне завершення**: використовуйте історичний контекст і знання проекту для автоматичного заповнення інформації
3. **Миттєвий зворотній зв’язок**: надайте кожній дії чіткий візуальний зворотний зв’язок і підказки щодо стану
4. **Механізми відмовостійкості**: дозволяють користувачам скасовувати та скидати налаштування, щоб помилки не призвели до незворотних втрат
5. **Різноманітність введення**: підтримка кількох методів введення, таких як текст і голос

У HagiCode практичний результат цього підходу був очевидним: середня довжина введеного користувачем зросла з менш ніж 20 символів до структурованого опису в 200-300 символів, якість пропозицій, створених ШІ, значно покращилася, а разом з цим зросла задоволеність користувачів.

Це й не дивно. Чим більше інформації надають користувачі, тим точніше ШІ може зрозуміти їх і тим кращі результати він може повернути. У цьому сенсі воно мало чим відрізняється від спілкування між людьми.

Якщо ви також створюєте продукти, пов’язані зі штучним інтелектом, я сподіваюся, що цей досвід стане для вас корисним натхненням. Пам’ятайте: користувачі не обов’язково відмовляються надати інформацію. Частіше продукт ще не ставив правильних запитань належним чином. Основою поступового розкриття є пошук найкращого часу та форми для цих запитань.
