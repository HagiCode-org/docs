## Технічний аналіз системи навичок HagiCode: створення масштабованої платформи керування навичками AI

> У цій статті детально розглядається архітектура та реалізація системи керування навичками в проекті HagiCode, охоплюючи технічні деталі, що стоять за чотирма основними можливостями: локальне глобальне управління, пошук на ринку, інтелектуальні рекомендації та керування надійними постачальниками.

## Фон

У сфері помічників кодування ШІ питання про те, як розширити межі можливостей ШІ, завжди було ключовим. Сам Claude Code вже сильний у допомозі з програмуванням коду, але різні команди розробників і різні стеки технологій часто потребують спеціалізованих можливостей для конкретних сценаріїв, таких як розгортання Docker, оптимізація бази даних або генерація зовнішніх компонентів. Саме тут система навичок стає особливо важливою.

Під час розробки проекту HagiCode ми зіткнулися з подібною проблемою: як ми дозволимо Клоду Коду «вивчити» нові професійні навички, як це зробить людина, зберігаючи при цьому стабільний досвід користувача та хорошу інженерну придатність до обслуговування? Ця проблема по-своєму складна і проста. Навколо цього питання ми розробили та впровадили повну систему управління навичками.

У цій статті детально описано технічну архітектуру та основну реалізацію системи. Він призначений для розробників, зацікавлених у розширюваності AI та інтеграції інструментів командного рядка. Це може бути корисним для вас, а може й ні, але принаймні воно зараз записано.

## Про HagiCode

Підхід, поширений у цій статті, базується на нашому практичному досвіді в [HagiCode](https://hagicode.com) проект. HagiCode — це проект помічника в кодуванні штучного інтелекту з відкритим вихідним кодом, розроблений, щоб допомогти командам розробників підвищити ефективність розробки. Стек проекту включає ASP.NET Core, розподілену структуру Orleans, інтерфейс TanStack Start + React і підсистему керування навичками, представлену в цій статті.

Репозиторій GitHub є [HagiCode-org/сайт](https://github.com/HagiCode-org/site). Якщо ви вважаєте технічний підхід у цій статті цінним, не соромтеся поставити йому зірку. Зрештою, більше зірок покращує настрій.

## Огляд архітектури системи

Система Skill використовує архітектуру, розділену на інтерфейс і сервер. Нічого особливо загадкового в цьому немає.

**Frontend** використовує TanStack Start + React для створення інтерфейсу користувача з Redux Toolkit для керування станом. Чотири основні можливості відображаються безпосередньо на чотирьох компонентах вкладки: локальні навички, галерея навичок, інтелектуальні рекомендації та надійні постачальники. Зрештою, дизайн здебільшого спрямований на покращення взаємодії з користувачем.

**Бекенд** базується на ASP.NET Core + ABP Framework, використовуючи Orleans Grain для розподіленого керування станом. Онлайн-клієнт API обертає `IOnlineApiClient` інтерфейс для зв’язку з віддаленою службою каталогу навичок.

Загальний архітектурний принцип полягає в тому, щоб відокремити виконання команд від бізнес-логіки. Через шаблон адаптера деталі реалізації виконання команди npm/npx приховані всередині незалежних модулів. Зрештою, ніхто насправді не хоче, щоб виклики командного рядка були розкидані по всій кодовій базі.

## Основна можливість 1: локальне глобальне управління

Локальне глобальне управління є основним модулем. Він відповідає за перелік встановлених навичок і підтримку операцій видалення. Тут немає нічого надскладного; це здебільшого про те, щоб добре виконувати основи.

### Технічний підхід

Реалізація живе `LocalSkillsTab.tsx` і `LocalSkillCommandAdapter.cs`. Основна ідея полягає в тому, щоб загорнути `npx skills` команду, проаналізувати її вихідні дані JSON і перетворити їх на внутрішні структури даних. Це звучить просто, і на практиці це переважно так.

```csharp
public async Task<IReadOnlyList<LocalSkillInventoryResponseDto>> GetLocalSkillsAsync(
    CancellationToken cancellationToken = default)
{
    var result = await _commandAdapter.ListGlobalSkillsAsync(cancellationToken);
    return result.Skills.Select(skill => new LocalSkillInventoryResponseDto
    {
        Name = skill.Name,
        Version = skill.Version,
        Source = skill.Source,
        InstalledPath = skill.InstalledPath,
        Description = skill.Description
    }).ToList();
}
```

Потік даних дуже зрозумілий: інтерфейс надсилає запит -> `SkillGalleryAppService` отримує його -> `LocalSkillCommandAdapter` виконує команду npx -> аналізується результат JSON -> повертається DTO. Кожен крок природно випливає з попереднього.

Для видалення навичок використовується `npx skills remove -g <skillName> -y` і система автоматично обробляє залежності та очищення. Метадані встановлення зберігаються в `managed-install.json` у каталозі навичок, записуючи таку інформацію, як час встановлення та вихідну версію для наступних оновлень і аудиту. Деякі речі просто варто записати.

### Детальний процес встановлення

Установка навику вимагає кількох скоординованих кроків. Насправді це не особливо складно:

```csharp
public async Task<SkillInstallResultDto> InstallAsync(
    SkillInstallRequestDto request,
    CancellationToken cancellationToken = default)
{
    // 1. Normalize the installation reference
    var normalized = _referenceNormalizer.Normalize(
        request.SkillId,
        request.Source,
        request.SkillSlug,
        request.Version);

    // 2. Check prerequisites
    await _prerequisiteChecker.CheckAsync(cancellationToken);

    // 3. Acquire installation lock
    using var installLock = await _lockProvider.AcquireAsync(normalized.SkillId);

    // 4. Execute installation command
    var result = await _installCommandRunner.ExecuteAsync(
        new SkillInstallCommandExecutionRequest
        {
            Command = $"npx skills add {normalized.FullReference} -g -y",
            Timeout = TimeSpan.FromMinutes(4)
        },
        cancellationToken);

    // 5. Persist installation metadata
    await _metadataStore.WriteAsync(normalized.SkillPath, request);

    return new SkillInstallResultDto { Success = result.Success };
}
```

Тут використовується кілька ключових шаблонів проектування: **еталонний нормалізатор** перетворює різні формати введення, наприклад `tanweai/pua` і `@opencode/docker-skill`, в єдине внутрішнє представництво; **механізм блокування інсталяції** гарантує, що лише одна операція інсталяції може бути виконана для однієї навички одночасно; а **потоковий вихід** передає хід встановлення до інтерфейсу в режимі реального часу за допомогою подій, надісланих сервером, тож користувачі можуть переглядати журнали, схожі на термінальні, щойно вони відбуваються.

Зрештою, усі ці шаблони існують з однією метою: зробити систему простішою у використанні та обслуговуванні.

## Основна можливість 2: пошук на ринку

Пошук на ринку дозволяє користувачам знаходити та встановлювати навички спільноти. Можливості однієї людини завжди обмежені; колективне знання йде набагато далі.

### Технічний підхід

Функція пошуку покладається на онлайн-API `https://api.hagicode.com/v1/skills/search`. Для підвищення швидкості відповіді система реалізує кешування. Кеш трохи схожий на пам'ять: якщо ви зберігаєте корисні речі, вам не доведеться так сильно думати наступного разу.

```csharp
private async Task<IReadOnlyList<SkillGallerySkillDto>> SearchCatalogAsync(
    string query,
    CancellationToken cancellationToken,
    IReadOnlySet<string>? allowedSources = null)
{
    var cacheKey = $"skill_search:{query}:{string.Join(",", allowedSources ?? Array.Empty<string>())}";

    if (_memoryCache.TryGetValue(cacheKey, out var cached))
        return (IReadOnlyList<SkillGallerySkillDto>)cached!;

    var response = await _onlineApiClient.SearchAsync(
        new SearchSkillsRequest
        {
            Query = query,
            Limit = _options.LimitPerQuery,
        },
        cancellationToken);

    var results = response.Skills
        .Where(skill => allowedSources is null || allowedSources.Contains(skill.Source))
        .Select(skill => new SkillGallerySkillDto { ... })
        .ToList();

    _memoryCache.Set(cacheKey, results, TimeSpan.FromMinutes(10));
    return results;
}
```

Результати пошуку підтримують фільтрацію за надійними джерелами, тому користувачі бачать лише джерела навичок, яким довіряють. Початкові запити, такі як `popular` і `recent` використовуються для ініціалізації каталогу, що дозволяє користувачам бачити рекомендовані популярні навички під час першого відкриття. Перше враження все ще має значення.

## Основна можливість 3: розумні рекомендації

Інтелектуальні рекомендації є найскладнішою частиною системи. Вони можуть автоматично рекомендувати найбільш підходящі навички на основі поточного контексту проекту. Який би складний він не був, його все ж варто побудувати.

### Потік рекомендацій

Повний потік рекомендацій поділяється на п’ять етапів:

```
1. Build project context
   ↓
2. AI generates search queries
   ↓
3. Search the online catalog in parallel
   ↓
4. AI ranks the candidates
   ↓
5. Return the recommendation list
```

По-перше, система аналізує такі характеристики, як стек технологій проекту, мови програмування та доменну структуру, щоб створити «профіль проекту». Цей профіль трохи схожий на резюме, де записуються ключові риси проекту.

Потім AI Grain використовується для створення цільових пошукових запитів. Цей дизайн насправді досить цікавий: замість того, щоб напряму запитувати ШІ: «Які навички я повинен порекомендувати?», ми спочатку просимо його подумати про те, «Які терміни пошуку, ймовірно, знайдуть відповідні навички?» Іноді те, як ви ставите запитання, має більше значення, ніж сама відповідь:

```csharp
var queryGeneration = await aiGrain.GenerateSkillRecommendationQueriesAsync(
    projectContext,      // Project context
    locale,              // User language preference
    maxQueries,           // Maximum number of queries
    effectiveSearchHero); // AI model selection
```

Потім ці пошукові запити виконуються паралельно, щоб зібрати список навичок кандидата. Зрештою, паралельна обробка — це лише спосіб заощадити час.

Нарешті, інше AI Grain оцінює навички кандидатів. На цьому етапі враховуються такі фактори, як релевантність навичок проекту, статус довіри та історичні вподобання користувача:

```csharp
var ranking = await aiGrain.RankSkillRecommendationsAsync(
    projectContext,
    candidates,
    installedSkillNames,
    locale,
    maxRecommendations,
    effectiveRankingHero);

response.Items = MergeRecommendations(projectContext, candidates, ranking, maxRecommendations);
```

### Запасний механізм

Моделі AI можуть реагувати повільно або тимчасово бути недоступними. Навіть найкращі системи іноді спотикаються. З цієї причини система включає детермінований резервний механізм: коли служба штучного інтелекту недоступна, вона використовує евристичний алгоритм на основі правил для створення рекомендацій, таких як висновок про ймовірні необхідні навички на основі залежностей у `package.json`.

Простіше кажучи, цей резервний механізм є просто резервним планом для системи.

## Основна можливість 4: Керування довіреним постачальником

Керування довіреними постачальниками дозволяє користувачам контролювати, які джерела навичок вважаються надійними. Довіра – це все ще те, що користувачі повинні мати можливість визначити для себе.

### Правила відповідності

Надійні постачальники підтримують два правила відповідності: **точна відповідність** (`exact`) і **збіг префікса** (`prefix`).

```csharp
public static TrustedSkillProviderResolutionSnapshot Resolve(
    TrustedSkillProviderSnapshot snapshot,
    string source)
{
    var normalizedSource = Normalize(source);

    foreach (var entry in snapshot.Entries.OrderBy(e => e.SortOrder))
    {
        if (!entry.IsEnabled) continue;

        foreach (var rule in entry.MatchRules)
        {
            bool isMatch = rule.MatchType switch
            {
                TrustedSkillProviderMatchRuleType.Exact
                    => string.Equals(normalizedSource, Normalize(rule.Value),
                        StringComparison.OrdinalIgnoreCase),
                TrustedSkillProviderMatchRuleType.Prefix
                    => normalizedSource.StartsWith(Normalize(rule.Value) + "/",
                        StringComparison.OrdinalIgnoreCase),
                _ => false
            };

            if (isMatch)
                return new TrustedSkillProviderResolutionSnapshot
                {
                    IsTrustedSource = true,
                    ProviderId = entry.ProviderId,
                    DisplayName = entry.DisplayName
                };
        }
    }

    return new TrustedSkillProviderResolutionSnapshot { IsTrustedSource = false };
}
```

Вбудовані надійні постачальники включають відомі організації та проекти, такі як Vercel, Azure, anthropics, Microsoft і використання браузера. Користувацькі постачальники можна додавати за допомогою файлів конфігурації, вказавши ідентифікатор постачальника, відображуване ім’я, мітку значка, правила відповідності тощо. Світ настільки великий, що довіряти лише кільком вбудованим компонентам ніколи не буде достатньо.

### Реалізація наполегливості

Довірена конфігурація зберігається за допомогою Orleans Grain:

```csharp
public class TrustedSkillProviderGrain : Grain<TrustedSkillProviderState>,
    ITrustedSkillProviderGrain
{
    public async Task UpdateConfigurationAsync(TrustedSkillProviderSnapshot snapshot)
    {
        State.Snapshot = snapshot;
        await WriteStateAsync();
    }

    public Task<TrustedSkillProviderSnapshot> GetConfigurationAsync()
    {
        return Task.FromResult(State.Snapshot);
    }
}
```

Перевага цього підходу полягає в тому, що зміни конфігурації автоматично синхронізуються на всіх вузлах без необхідності оновлювати кеші вручну. Зрештою, автоматизація полягає в тому, щоб дати людям менше хвилюватися.

## Ключовий технічний дизайн

### Шаблон адаптера виконання команд

Система навичок повинна виконувати різні команди npx. Якби ця логіка була розкидана всюди, код швидко стало б важко підтримувати. Саме тому ми розробили інтерфейс адаптера. Шаблони проектування, зрештою, існують, щоб полегшити підтримку коду:

```csharp
public interface ISkillInstallCommandRunner
{
    Task<SkillInstallCommandExecutionResult> ExecuteAsync(
        SkillInstallCommandExecutionRequest request,
        CancellationToken cancellationToken = default);
}
```

Різні команди мають різні реалізації виконавців, але всі вони реалізують однаковий інтерфейс, що робить тестування та заміну простими.

### Потоковий вихід SSE

Хід інсталяції передається на інтерфейс у реальному часі за допомогою подій, надісланих сервером:

```csharp
public async Task InstallWithProgressAsync(
    SkillInstallRequestDto request,
    IServerStreamWriter<SkillInstallProgressEventDto> stream,
    CancellationToken cancellationToken)
{
    var process = new Process
    {
        StartInfo = new ProcessStartInfo
        {
            FileName = "npx",
            Arguments = $"skills add {request.FullReference} -g -y",
            RedirectStandardOutput = true,
            RedirectStandardError = true,
            UseShellExecute = false
        }
    };

    process.OutputDataReceived += async (sender, e) =>
    {
        await stream.WriteAsync(new SkillInstallProgressEventDto
        {
            EventType = "output",
            Data = e.Data ?? string.Empty
        });
    };

    process.Start();
    process.BeginOutputReadLine();
    await process.WaitForExitAsync(cancellationToken);
}
```

На інтерфейсі користувачі можуть бачити вихід, подібний до терміналу, у режимі реального часу, що робить роботу дуже інтуїтивно зрозумілою. Зворотний зв’язок у реальному часі допомагає людям почуватися невимушено.

## Практичний посібник

### Встановлення навичок спільноти

Візьміть установку `pua` навичка як приклад (це популярна навичка спільноти):

1. Відкрийте ящик навичок і перейдіть до `Skill Gallery` вкладка
2. Введіть `pua` у вікні пошуку
3. Натисніть результат пошуку, щоб переглянути деталі навичок
4. Натисніть на `Install` кнопку
5. Переключитися на `Local Skills` вкладку, щоб підтвердити успішне встановлення

Команда встановлення така `npx skills add tanweai/pua -g -y`, і система автоматично обробляє всі деталі. Насправді кроків не так вже й багато, якщо їх робити один за одним.

### Додавання спеціального довіреного джерела

Якщо ваша команда має власне сховище навичок, ви можете додати його як надійне джерело:

```yaml
providerId: "my-team"
displayName: "My Team Skills"
badgeLabel: "MyTeam"
isEnabled: true
sortOrder: 100
matchRules:
  - matchType: "prefix"
    value: "my-team/"
  - matchType: "exact"
    value: "my-team/special-skill"
```

Таким чином, усі навички вашої команди відображатимуть позначку довіреного, що полегшить їх встановлення для користувачів. Мітки та сигнали справді допомагають людям почуватися впевненіше.

### Основи розвитку навичок

Для створення спеціального навику потрібна така структура:

```
my-skill/
├── SKILL.md          # Skill metadata (YAML front matter)
├── index.ts          # Skill entry point
├── agents/           # Supported agent configuration
└── references/       # Reference resources
```

приклад `SKILL.md` формат:

## Примітки

1. **Мережеві вимоги**: для пошуку навичок і встановлення потрібен доступ `api.hagicode.com` і реєстр npm
2. **Версія Node.js**: рекомендується Node.js 18 або новішої версії
3. **Вимоги до дозволів**: потрібні глобальні дозволи на встановлення npm
4. **Контроль паралелізму**: лише одна операція встановлення або видалення може виконуватися для однієї навички одночасно
5. **Налаштування часу очікування**: час очікування за замовчуванням для встановлення становить 4 хвилини, але складні сценарії можуть потребувати коригування

Ці примітки існують, зрештою, щоб допомогти все пройти гладко.

## Висновок

У цій статті представлено повну реалізацію системи управління навичками в проекті HagiCode. Завдяки архітектурі, розділеній між інтерфейсом і сервером, шаблону адаптера, розподіленому управлінню станом на основі Орлеана та відповідним методам система забезпечує:

- **Локальне глобальне керування**: уніфікований інтерфейс керування навичками, створений за допомогою упаковки `npx skills` команди
- **Пошук на ринку**: швидке виявлення навичок спільноти за допомогою онлайн-API та механізмів кешування
- **Розумні рекомендації**: рекомендації щодо навичок на основі штучного інтелекту на основі контексту проекту
- **Керування довірою**: гнучка система конфігурації, яка дозволяє користувачам контролювати межі довіри

Цей підхід до розробки застосовний не лише до управління навичками. Він також корисний як довідник для будь-якого сценарію, який потребує інтеграції інструментів командного рядка під час балансування локального сховища та онлайн-служб.
