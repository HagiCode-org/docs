## VSCode і сервер коду: вибір рішення для редагування коду на основі браузера

> Розробляючи можливості редагування коду на основі браузера, розробники стикаються з ключовим вибором: використовувати офіційний VSCode `code serve-web` функцію або прийняти керовану спільнотою `code-server` рішення? Це рішення впливає не лише на технічну архітектуру, але й на відповідність ліцензії та гнучкість розгортання.

## Фон

Технічний відбір дуже схожий на вибір шляху в житті. Як тільки ви виберете один, вам зазвичай доведеться продовжувати його ходити, а перехід пізніше може стати дуже дорогим.

В еру програмування за допомогою штучного інтелекту редагування коду на основі браузера стає все більш важливим. Користувачі очікують, що після того, як помічник AI завершить аналіз коду, вони зможуть негайно відкрити редактор у тому ж сеансі браузера та внести зміни, не перемикаючи програми. Такий бездоганний досвід просто має бути поруч, коли вам це потрібно.

Однак, реалізуючи цю функцію, розробники стикаються з критичним технічним вибором: чи використовувати їм офіційний VSCode `code serve-web` функція або керована спільнотою `code-server` рішення?

Кожен варіант має свої сильні сторони та недоліки, і неправильний вибір може створити багато проблем згодом. Ліцензування — один із прикладів: якщо лише після запуску ви дізнаєтеся, що ваш продукт не відповідає вимогам ліцензії, це вже занадто пізно. Розгортання — це інше: рішення може ідеально працювати під час розробки, а потім зіткнутися з різними проблемами, коли його перемістять у контейнери. Це саме ті підводні камені, яких команди хочуть уникнути.

## Відмінності в ліцензуванні (найважливіші)

Це найбільш фундаментальна відмінність між двома рішеннями та перший фактор, який ми врахували під час оцінки. Роблячи технічний вибір, важливо заздалегідь розуміти юридичні ризики.

**код-сервер**
- Ліцензія MIT, повністю відкритий код
- Підтримується Coder.com з активною спільнотою
- Безкоштовно для комерційного використання, модифікації та розповсюдження
- Жодних обмежень щодо сценаріїв використання

**VSCode `code serve-web`**
- Частина продукту Microsoft VSCode
- Використовує ліцензію Microsoft (ліцензія VS Code містить обмеження на комерційне використання)
- В першу чергу призначений для індивідуальних розробників
- Корпоративне розгортання може потребувати додаткової перевірки комерційної ліцензії

З точки зору ліцензування, сервер коду більш дружній до комерційних проектів. Це те, про що вам потрібно подумати під час планування продукту, оскільки пізніша міграція може бути дуже дорогою.

## Відмінності розгортання

Після того, як ліцензування вирішено, наступним питанням є розгортання. Це безпосередньо впливає на ваші операційні витрати та архітектурний дизайн.

**код-сервер**
- Окрема програма Node.js, яку можна розгортати незалежно
- Підтримує кілька джерел виконання:
  - Безпосереднє вказівку шляху до виконуваного файлу
  - Перегляд через системний PATH
  - Автоматичне виявлення середовища NVM Node.js 22.x
- Немає необхідності встановлювати настільну програму VSCode на сервері
- Легше розгортати в контейнерах

**VSCode `code serve-web`**
- Має залежати від локально встановленого VSCode CLI
- Потрібна наявність `code` на головній машині
- Система відфільтровує обгортки VS Code Remote CLI
- В першу чергу розроблено для сценаріїв місцевого розвитку

code-server краще підходить для сценаріїв розгортання серверів і контейнерів. Якщо ваш продукт має працювати в Docker або у ваших користувачів не встановлено VSCode, сервер коду зазвичай є правильним вибором.

## Відмінності параметрів функції

Ці два рішення також відрізняються декількома параметрами. Відмінності невеликі, але вони можуть спричинити тертя інтеграції в реальному використанні.

| Особливість | кодовий сервер | code serve-web |
|------|-------------|----------------|
| Загальнодоступний базовий шлях | `/` (настроюється) | `/vscode-server` (виправлено) |
| Аутентифікація | `--auth` параметр з кількома режимами | `--connection-token` / `--without-connection-token` |
| Каталог даних | `{DataDir}/code-server` | `{DataDir}/vscode-serve-web` |
| Телеметрія | Вимкнено за замовчуванням за допомогою `--disable-telemetry` | Залежить від налаштувань VSCode |
| Оновлення чеків | Можна відключити за допомогою `--disable-update-check` | Залежить від налаштувань VSCode |

Ці відмінності потребують особливої ​​уваги під час інтеграції. Наприклад, різні URL-шляхи означають, що ваш зовнішній код потребує спеціальної обробки.

## Відмінності виявлення доступності

При реалізації перемикання редакторів логіка визначення доступності також відрізняється.

**код-сервер**
- Завжди повертається як видима реалізація
- Все ще відображається, навіть якщо він недоступний, з an `install-required` статус
- Підтримує автоматичне виявлення середовища NVM Node.js 22.x

**code serve-web**
- Видно лише місцевим `code` CLI виявлено
- Якщо вона недоступна, інтерфейс автоматично приховує цю опцію
- Залежить від локального стану встановлення VSCode

Ця різниця безпосередньо впливає на досвід користувача. код-сервер більш прозорий: користувачі можуть побачити параметр і зрозуміти, що інсталяція все ще потрібна. code serve-web є більш прихованим: користувачі можуть навіть не усвідомлювати існування цієї опції. Який підхід кращий, залежить від позиціонування продукту.

## Архітектура подвійної реалізації HagiCode

Після поглибленого аналізу проект HagiCode прийняв архітектуру подвійної реалізації, яка підтримує обидва рішення на архітектурному рівні.

### Сервер коду за замовчуванням

```csharp
// The default active implementation is code-server
// If an explicit activeImplementation is saved, try that implementation first
// If the requested implementation is unavailable, the resolver tries the other one
// If a fallback occurs, return fallbackReason
```

За замовчуванням ми використовуємо сервер коду головним чином через ліцензування та гнучкість розгортання. Однак для користувачів, які вже мають локальне середовище VSCode, веб-сервер коду також є надійним варіантом.

### Селектор реалізації

`CodeServerImplementationResolver` відповідає за уніфікацію:
- Вибір впровадження під час розігріву запуску
- Вибір реалізації під час читання статусу
- Вибір реалізації при відкритті проектів
- Вибір реалізації під час відкриття сховищ

Така конструкція дозволяє системі гнучко реагувати на різні сценарії, а користувачі можуть вибрати реалізацію, яка найкраще відповідає їхньому середовищу.

### Правила адаптації інтерфейсу

```typescript
// When localCodeAvailable=false, do not show code serve-web
// When localCodeAvailable=true, show the code serve-web configuration
```

Інтерфейс автоматично показує доступні параметри на основі середовища, щоб користувачів не бентежили функції, якими вони не можуть користуватися.

## Практичний посібник

Після всієї цієї теорії, на що слід звернути увагу під час фактичного розгортання? Зрештою, важлива реалізація.

### Рекомендація щодо розгортання Docker

Для контейнерного розгортання кращим вибором є сервер коду:

```dockerfile
# Use the official code-server image directly
FROM codercom/code-server:latest

# Or install through npm
RUN npm install -g code-server
```

Це вирішує проблему на одному рівні, не вимагаючи додаткового встановлення VSCode.

### Приклади конфігурації

**конфігурація сервера коду**
```json
{
  "vscodeServer": {
    "enabled": true,
    "activeImplementation": "code-server",
    "codeServer": {
      "host": "0.0.0.0",
      "port": 8080,
      "executablePath": "",
      "authMode": "none"
    }
  }
}
```

**конфігурація служби коду-веб**
```json
{
  "vscodeServer": {
    "enabled": true,
    "activeImplementation": "serve-web",
    "serveWeb": {
      "host": "0.0.0.0",
      "port": 8080,
      "executablePath": "/usr/local/bin/code"
    }
  }
}
```

Конфігурація може бути трохи стомлюючою в перший раз, але як тільки вона встановлена, усе стає набагато легше підтримувати.

### Відмінності побудови URL

**код-сервер**
```
http://localhost:8080/?folder=/path/to/project&vscode-lang=zh-CN
```

**code serve-web**
```
http://localhost:8080/vscode-server/?folder=/path/to/project&tkn=xxx&vscode-lang=zh-CN
```

Зверніть увагу на відмінності в шляхах і параметрах. Вам потрібно обробляти їх окремо під час інтеграції.

### Перемикання реалізацій

Система підтримує перемикання під час виконання та автоматично зупиняє попередню реалізацію під час перемикання:

```csharp
// VsCodeServerManager automatically handles mutual exclusion
// When switching activeImplementation, the old implementation will not keep running in the background
```

Цей дизайн дозволяє користувачам у будь-який час спробувати різні реалізації та знайти варіант, який їм найкраще підходить.

### Моніторинг стану

```typescript
const { settings, runtime } = await getVsCodeServerSettings();

// runtime.activeImplementation: "code-server" | "serve-web"
// runtime.fallbackReason: reason for switching
// runtime.status: "running" | "starting" | "stopped" | "unhealthy"
```

Коли статус відображається, користувачі можуть швидко визначити, чи проблема виникла на стороні сервера чи через їх власну роботу.

## Висновок

| Вимір порівняння | кодовий сервер | code serve-web | Рекомендація |
|---------|-------------|----------------|------|
| Ліцензія | MIT (комерційний) | Microsoft (з обмеженнями) | кодовий сервер |
| Гнучкість розгортання | Самостійне розгортання | Залежить від локального VSCode | кодовий сервер |
| Придатність сервера | Призначений для серверів | В основному для місцевого розвитку | кодовий сервер |
| Контейнерізація | Рідна підтримка | Потрібна інсталяція VSCode | кодовий сервер |
| Повнота функції | Близький до настільного видання | Офіційна повна версія | code serve-web |
| Технічна діяльність | Активна спільнота | Офіційно підтримується Microsoft | Обидва мають сильні сторони |

**Рекомендована стратегія**: спочатку використовуйте сервер коду, а якщо вам потрібна повна офіційна функціональність і вже є локальне середовище VSCode, розгляньте веб-сервер коду.
