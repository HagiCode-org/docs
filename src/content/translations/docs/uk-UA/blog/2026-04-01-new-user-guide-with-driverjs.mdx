## Створення елегантної адаптації нових користувачів у проектах React: практика HagiCode щодо driver.js

> Чи знають вони, з чого почати, коли користувачі відкривають ваш продукт уперше? У цій статті я хочу трохи поговорити про те, як ми використовували driver.js для адаптації нових користувачів у проекті HagiCode. Вважайте це невеликим практичним прикладом для початку розмови.

## Фон

Ви коли-небудь потрапляли в таку ситуацію? Новий користувач реєструється на вашому продукті, відкриває сторінку й одразу виглядає втраченим. Вони сканують навколо, не знаючи, що натиснути чи що робити далі. Як розробники, ми часто припускаємо, що користувачі просто «досліджуватимуть самі», оскільки, зрештою, людська цікавість безмежна. Але реальність інша: більшість користувачів тихо виходять протягом декількох хвилин, якщо не можуть знайти потрібну точку входу, ніби історія починається раптово і закінчується так само природно.

Реєстрація нових користувачів — важливий спосіб вирішення цієї проблеми, але створити його не так просто. Хороша система адаптації повинна:

- Точне розташування елементів сторінки та виділення їх
- Підтримка багатоетапних процесів адаптації
- Запам'ятати вибір користувача (заповнити або пропустити)
- Уникайте впливу на продуктивність сторінки та нормальну взаємодію
- Зберігайте структуру коду чіткою та легкою для обслуговування

Створюючи HagiCode, ми зіткнулися з тією ж проблемою. HagiCode — це помічник кодування штучного інтелекту, і його основний робочий процес — робочий процес OpenSpec, який виглядає так: «користувач створює пропозицію -> штучний інтелект створює план -> користувач переглядає його -> штучний інтелект виконує». Для користувачів, які вперше стикаються з цією концепцією, робочий процес абсолютно новий, тому їм потрібна надійна підготовка, щоб швидко розпочати роботу. До нових речей завжди потрібен час, щоб звикнути.

## Про HagiCode

Підхід, опублікований у цій статті, базується на нашому практичному досвіді в [HagiCode](https://hagicode.com) демонструвати. HagiCode — це помічник кодування на базі Claude, який допомагає розробникам ефективніше виконувати завдання кодування через робочий процес OpenSpec. Ви можете переглянути наш відкритий вихідний код на [GitHub](https://github.com/HagiCode-org/site).

## Чому ми вибрали driver.js

Під час етапу технічної оцінки ми розглянули кілька основних бібліотек адаптації. Кожен мав свої сильні сторони:

- **Intro.js**: потужний, але відносно великий, і налаштування стилю є дещо складним
- **Shepherd.js**: добре розроблений API, але занадто «важкий» для нашого випадку використання
- **driver.js**: легкий, лаконічний, інтуїтивно зрозумілий API, добре працює в екосистемі React

Зрештою, ми вибрали driver.js. Особливо драматичної причини не було. Вибір в основному зводився до таких міркувань:

1. **Легкий**: базова бібліотека невелика і не значно збільшує розмір комплекту
2. **Простий API**: конфігурація зрозуміла та інтуїтивно зрозуміла, тому її легко підібрати
3. **Гнучкий**: підтримує спеціальне позиціонування, стиль і поведінку взаємодії
4. **Динамічний імпорт**: можна завантажувати за запитом, не впливаючи на продуктивність першого екрана

Що стосується вибору технології, рідко існує універсальна найкраща відповідь. Зазвичай є лише той варіант, який найкраще підходить.

## Технічна реалізація

### Конфігурація ядра

driver.js має дуже інтуїтивно зрозумілу модель конфігурації. Ось основна конфігурація, яку ми використовуємо в проекті HagiCode:

```typescript
import { driver } from 'driver.js';
import 'driver.js/dist/driver.css';

const newConversationDriver = driver({
  allowClose: true,           // Allow users to close the guide
  animate: true,              // Enable animations
  overlayClickBehavior: 'close', // Close the guide when the overlay is clicked
  disableActiveInteraction: false, // Keep elements interactive
  showProgress: false,        // Do not show the progress bar (we manage progress ourselves)
  steps: guideSteps           // Array of guide steps
});
```

Причини цих налаштувань:

- `allowClose: true` - Поважайте вибір користувача та не змушуйте його закінчувати посібник
- `disableActiveInteraction: false` - Для деяких кроків потрібні реальні дії користувача, наприклад введення тексту, тому взаємодію неможливо вимкнути
- `overlayClickBehavior: 'close'` - Дайте користувачам можливість швидко вийти

### Державне управління

Постійний стан підключення є критичним. Ми не хочемо перезапускати посібник кожного разу, коли сторінка оновлюється, тому що це швидко дратує. HagiCode використовує `localStorage` щоб керувати станом довідника:

```typescript
export type GuideState = 'pending' | 'dismissed' | 'completed';

export interface UserGuideState {
  session: GuideState;
  detailGuides: Record<string, GuideState>;
}

// Read state
export const getUserGuideState = (): UserGuideState => {
  const state = localStorage.getItem('userGuideState');
  return state ? JSON.parse(state) : { session: 'pending', detailGuides: {} };
};

// Update state
export const setUserGuideState = (state: UserGuideState) => {
  localStorage.setItem('userGuideState', JSON.stringify(state));
};
```

Визначаємо три стани:

- `pending`: Посібник ще розробляється, і користувач не завершив або пропустив його
- `dismissed`: користувач завчасно закрив посібник
- `completed`: користувач виконав усі кроки

Для реєстрації сторінки з інформацією про пропозицію ми також підтримуємо більш детальне відстеження стану через `detailGuides` карту, тому що одна Пропозиція може пройти кілька етапів – чернетка, перегляд і завершення виконання – і кожен етап потребує різних вказівок. Стан речей постійно змінюється, і адаптація має відображати це.

### Вибір цільового елемента

driver.js використовує селектори CSS для пошуку цільових елементів. HagiCode дотримується простої угоди: використовуйте спеціальний код `data-guide` атрибут для позначення цілей реєстрації:

```typescript
const steps = [
  {
    element: '[data-guide="launch"]',
    popover: {
      title: 'Start a New Conversation',
      description: 'Click here to create a new conversation session...'
    }
  }
];
```

У складових це виглядає так:

```typescript
<button data-guide="launch" onClick={handleLaunch}>
  New Conversation
</button>
```

Переваги цього підходу:

- Уникайте конфліктів із назвами класів бізнес-стилю
- Чітка семантика, тож ви можете одразу визначити, що елемент пов’язаний із адаптацією
- Простіше в управлінні та постійному обслуговуванні

### Динамічна оптимізація імпорту

Оскільки функція адаптації потрібна лише в певних ситуаціях, наприклад під час першого візиту користувача, ми використовуємо динамічний імпорт для оптимізації продуктивності початкового завантаження:

```typescript
const initNewUserGuide = async () => {
  // Dynamically import driver.js
  const { driver } = await import('driver.js');
  await import('driver.js/dist/driver.css');

  // Initialize the guide
  const newConversationDriver = driver({
    // ...configuration
  });

  newConversationDriver.drive();
};
```

Таким чином, driver.js і його таблиця стилів завантажуються лише за потреби й не впливають на продуктивність першого екрана. Мало кому подобається чекати чогось, що їм ще навіть не потрібно.

## Дизайн потоку адаптації

HagiCode реалізує два шляхи реєстрації, які охоплюють основні сценарії користувача.

### Підготовка до сеансу (10 кроків)

Цей шлях адаптації допомагає користувачам пройти весь процес від створення бесіди до подання своєї першої повної пропозиції:

1. **запуск** – запустіть посібник і введіть кнопку «Нова розмова».
2. **написати** – скерувати користувача до введення запиту у полі введення
3. **надіслати** – вказівка користувача натиснути кнопку надсилання
4. **proposal-launch-readme** – допоможе користувачеві створити пропозицію README
5. **proposal-compose-readme** – допоможе користувачеві редагувати вміст запиту README
6. **proposal-submit-readme** – вказує користувачу надіслати пропозицію README
7. **proposal-launch-agents** – допоможе користувачеві створити `AGENTS.md` Пропозиція
8. **proposal-compose-agents** - Наказує користувачеві редагувати `AGENTS.md` запит
9. **proposal-submit-agents** – направляє користувача щодо подання `AGENTS.md` Пропозиція
10. **пропозиція-чекайте** – поясніть, що штучний інтелект обробляє, і користувачеві слід трохи почекати

Ідея цього шляху полягає в тому, щоб дозволити користувачам відчути основний робочий процес HagiCode з перших вуст за допомогою двох справжніх завдань зі створення пропозиції, одне для `README` і один для `AGENTS.md`. Існує велика різниця між тим, щоб почути про робочий процес і пройти через нього самостійно.

### Реєстрація деталей пропозиції (3 кроки)

Коли користувачі переходять на сторінку деталей пропозиції, HagiCode запускає відповідний посібник на основі поточного стану пропозиції:

1. **складання** (стадія чернетки) – скеруйте користувача до перегляду створеного ШІ плану
2. **перегляд** (стадія перегляду) – скеровуйте користувача до виконання плану
3. **executionCompleted** (завершений етап) – наказує користувачеві архівувати план

Визначальною характеристикою цього посібника є те, що він керується станом: він динамічно вирішує, який етап реєстрації показувати на основі фактичного стану пропозиції. Речі змінюються, і адаптація повинна змінюватися разом з ними.

На знімку екрана нижче показано сторінку з інформацією про пропозицію в стані підключення на етапі розробки:

На цьому етапі керівництво зосереджує увагу користувача на ключовій дії створення плану, щоб він не задавався питанням, що робити в першу чергу, коли вперше заходить на сторінку з деталями.

### Механізм повторної спроби візуалізації елемента

У додатках React цільовий інтеграційний елемент може ще не завершити рендеринг, наприклад, тому що асинхронні дані все ще завантажуються. Щоб впоратися з цим, HagiCode реалізує механізм повторної спроби:

```typescript
const waitForElement = (selector: string, maxRetries = 10, interval = 100) => {
  let retries = 0;
  return new Promise<HTMLElement>((resolve, reject) => {
    const checkElement = () => {
      const element = document.querySelector(selector) as HTMLElement;
      if (element) {
        resolve(element);
      } else if (retries < maxRetries) {
        retries++;
        setTimeout(checkElement, interval);
      } else {
        reject(new Error(`Element not found: ${selector}`));
      }
    };
    checkElement();
  });
};
```

Викличте цю функцію перед ініціалізацією посібника, щоб переконатися, що цільовий елемент уже існує. Іноді варто почекати трохи довше.

## Резюме найкращих практик

Базуючись на практичному досвіді HagiCode, ось кілька ключових найкращих практик:

### 1. Вийти з реєстрації має бути легко

Не змушуйте користувачів завершувати адаптацію. Деякі користувачі за своєю природою дослідники і вважають за краще з’ясовувати речі самостійно. Надайте чітку кнопку «Пропустити», запам’ятайте їхній вибір і не переривайте їх наступного разу.

### 2. Зберігайте короткий і чіткий контент про адаптацію

Кожен етап адаптації має бути зосереджений на одній меті:

- **Назва**: Короткий і зрозумілий, в ідеалі не більше ніж кілька слів
- **Опис**: перейдіть безпосередньо до суті та скажіть користувачеві, що це таке та чому це важливо

Уникайте розлогих пояснень. Увага користувача обмежена під час адаптації, і чим більше ви говорите, тим менша ймовірність, що вони це прочитають.

### 3. Тримайте селектори стабільними

Використовуйте стабільні маркери елементів, які не часто змінюються. Звичай `data-guide` атрибут є хорошим вибором. Уникайте залежності від назв класів або структури DOM, тому що їх легко змінити під час рефакторів. Код постійно змінюється, але деякі прив’язки мають залишатися стабільними, коли це можливо.

### 4. Перевірте свою адаптацію

HagiCode містить повні тестові приклади для функції адаптації:

```typescript
describe('NewUserConversationGuide', () => {
  it('should initialize guide state correctly', () => {
    const state = getUserGuideState();
    expect(state.session).toBe('pending');
  });

  it('should update guide state correctly', () => {
    setUserGuideState({ session: 'completed', detailGuides: {} });
    const state = getUserGuideState();
    expect(state.session).toBe('completed');
  });
});
```

Тести допомагають переконатися, що рефакторинг випадково не порушує поведінку підключення. Ніхто не хоче, щоб невелика зміна коду тихо пошкодила раніше працюючий функціонал.

### 5. Оптимізація продуктивності

- Використовуйте динамічний імпорт для відкладеного завантаження адаптаційної бібліотеки
- Уникайте ініціалізації логіки підключення після того, як користувач уже пройшов посібник
- Враховуйте вплив анімацій на продуктивність і за потреби вимикайте їх на пристроях нижчого рівня

Ефективність, як і багато речей у житті, заслуговує на ретельний бюджет.

## Висновок

Реєстрація нових користувачів є важливою частиною покращення взаємодії з продуктом. У проекті HagiCode ми використовували driver.js для створення повної системи адаптації, яка охоплює весь робочий процес від створення сеансу до виконання пропозиції.

Основні моменти, якими ми сподіваємося поділитися в цій статті:

1. **Технічний вибір має відповідати реальним потребам**: driver.js не є найпотужнішим варіантом, але він найкраще підходить для нас
2. **Управління державою має вирішальне значення**: використовуйте `localStorage` щоб зберегти активний стан і уникнути повторного переривання користувачів
3. **Розробка адаптації має залишатися зосередженою**: кожен крок має вирішувати одну проблему, і не більше
4. **Структура коду має залишатися зрозумілою**: окрема конфігурація реєстрації, керування станом і логіка інтерфейсу користувача, щоб спростити обслуговування
