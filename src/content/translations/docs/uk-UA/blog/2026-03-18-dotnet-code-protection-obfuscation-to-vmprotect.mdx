## Захист коду .NET на практиці: від обфускації до захисту віртуальної машини

> У цій статті пояснюється, як реалізувати багаторівневу стратегію захисту коду в проектах .NET, охоплюючи повний шлях від базової обфускації до професійного захисту віртуальної машини.

## Фон

У розробці додатків .NET захист основного коду, такого як перевірка ліцензії, бізнес-логіка та конфіденційна конфігурація, від декомпіляції та зворотного проектування — це, відверто кажучи, тема, яку неможливо уникнути. У міру розвитку екосистеми .NET розробники отримали доступ до ряду варіантів захисту, від вбудованих атрибутів обфускації до професійних засобів захисту на основі віртуалізації.

Будучи складним багатомовним проектом монорепо, HagiCode включає програми для робочого столу, системи збірки та можливості керування ліцензіями. Код неминуче містить логіку перевірки ліцензії, конфіденційну конфігурацію, таку як ключі API та ідентифікатори продукту, а також критичну для бізнесу логіку. Ці деталі потребують серйозного захисту, тому що ніхто не хоче, щоб їх важка робота була так легко відкрита.

У цій статті розповідається про підхід до захисту коду, який ми фактично застосували в проекті HagiCode, і підсумовується повний шлях від ранніх помилок до подальшої оптимізації. Сподіваюся, це дасть вам корисні ідеї.

## Основний вміст

### 1. Вбудований атрибут обфускації Microsoft

.NET Framework забезпечує вбудований `[ObfuscationAttribute]`, який є основним і часто використовуваним маркером обфускації коду. Цей атрибут живе в `System.Reflection` простір імен і дозволяє застосовувати базовий захист до коду без використання сторонніх інструментів.

**Основні функції**:
- `Feature` властивість: визначає функцію обфускації, наприклад `"ultra"` (висока обфускація) або `"all"` (повна обфускація)
- `Exclude` властивість: `true` означає виключити з обфускації, і `false` означає застосування обфускації
- Може застосовуватися до класів, методів, властивостей та інших членів типу

У проекті HagiCode ви можете побачити, як це використовується так:

```csharp
[Obfuscation(Feature = "ultra", Exclude = false)]
public async Task<LicenseValidationResult?> ValidateLicenseAsync(...)
```

**Переваги** цього підходу досить очевидні:
- Без додаткових залежностей; він вбудований у .NET Framework з коробки
- Може бути розпізнано та оброблено сторонніми інструментами обфускації
- Значно не збільшує розмір скомпільованої збірки

Тим не менш, він також має обмеження. Це лише маркер, а фактичний результат обфускації залежить від реалізації інструменту. Він не може забезпечити безпеку рівня захисту віртуальної машини.

### 2. VMP (захист віртуальної машини)

VMP — це професійний інструмент захисту коду, який забезпечує захист високого рівня шляхом компіляції коду в інструкції віртуальної машини. На відміну від простої обфускації імен, VMP фактично перетворює логіку коду у форму, яку звичайні декомпілятори не можуть реконструювати.

**Класифікація рівня захисту**:

| Рівень | Віртуалізація | Мутація | Антиналагодження | Шифрування рядків | Випадки використання |
|------|--------|------|--------|------------|----------|
| ВИСОКА | повний | висока | включено | включено | Перевірка ліцензії, паралелізм сесії, чутливі константи |
| СЕРЕДНІЙ | частковий | середній | включено | включено | Бізнес-логіка, моделі домену |
| НИЗЬКИЙ | немає | низький | вимкнено | вимкнено | Класи корисності, некритичний код |

Проект HagiCode визначає декларативну систему атрибутів для позначення коду, який потребує захисту:

```csharp
// High-priority protection
[VmProtect(VmProtectionPriority.High, Reason = "Contains license verification logic")]
public class KeygenClient { ... }

// Exclude from protection
[VmExclude(Reason = "Public API that must remain unchanged")]
public class PublicApi { ... }

// Inherited protection
[VmProtect(Priority.High, ProtectDerived = true)]
public class BaseLicenseValidator { ... }
```

### 3. Стратегія захисту під час створення

Захист VMP має значення не лише під час виконання. Його також потрібно автоматизувати як частину конвеєра збірки, оскільки робити це вручну було б надто стомлююче. Система збирання HagiCode підтримує декілька режимів:

- **Рідний режим Windows**: безпосередньо викликайте інструмент VMProtect
- **Режим контейнера Linux Docker**: запустіть VMP усередині контейнера, щоб вирішити проблеми сумісності між платформами
- **Сканування атрибутів**: автоматично виявляти маркери захисту в коді
- **Механізм перевірки**: підтвердьте, що захист застосовано успішно

У сукупності ці можливості значно полегшують керування процесом.

## Рішення

### 1. Використання вбудованого атрибута обфускації Microsoft

Застосувати `ObfuscationAttribute` безпосередньо в коді:

```csharp
using System.Reflection;

[Obfuscation(Feature = "ultra", Exclude = false)]
public class LicenseService
{
    [Obfuscation(Feature = "ultra", Exclude = false)]
    public async Task<bool> ValidateLicenseAsync(string key)
    {
        // License validation logic
    }

    [Obfuscation(Feature = "flow", Exclude = false)]
    private string DecryptToken(string encrypted)
    {
        // Decryption logic
    }
}
```

Іноді вам потрібно дозволити тестовим збіркам доступ до внутрішніх членів, зберігаючи безпеку робочого коду:

```csharp
// AssemblyInfo.cs
[assembly: InternalsVisibleTo("HagiCode.Application.Tests")]
[assembly: InternalsVisibleTo("DynamicProxyGenAssembly2")] // for Moq
```

Це робить тестування набагато зручнішим, оскільки код все ще потрібно перевірити належним чином.

### 2. Визначення спеціальних атрибутів для захисту VMP

Створіть власні атрибути захисту для керування поведінкою VMP:

```csharp
using System;

[AttributeUsage(AttributeTargets.Class | AttributeTargets.Method | AttributeTargets.Property)]
public class VmProtectAttribute : Attribute
{
    public VmProtectionPriority Priority { get; set; }
    public string? Reason { get; set; }
    public bool ProtectDerived { get; set; }
}

[AttributeUsage(AttributeTargets.Class | AttributeTargets.Method | AttributeTargets.Property)]
public class VmExcludeAttribute : Attribute
{
    public string? Reason { get; set; }
}

public enum VmProtectionPriority
{
    None = 0,
    Low = 1,
    Medium = 2,
    High = 3
}
```

З настроюваними атрибутами часто легше працювати, оскільки вони безпосередньо відображають ваші власні вимоги до захисту.

### 3. Файл конфігурації VMP

```yaml

```

Більш чітка конфігурація значно полегшує подальше обслуговування системи.

## Практичний посібник

### 1. Методи захисту критичних компонентів

Відповідно до специфікації захисту коду HagiCode наступні компоненти повинні використовувати захист з ВИСОКИМ пріоритетом:

```csharp
// Production constants - must be encrypted and protected by VMP
[VmProtect(VmProtectionPriority.High, Reason = "Production constants")]
public static class ProductionConstants
{
    // Encrypted string accessor, protected by VMP
    [VmProtect(VmProtectionPriority.High)]
    public static string GetLicenseServerUrl(IOptions<LicenseOptions> options) => ...;
}

// License validation logic
[VmProtect(VmProtectionPriority.High, Reason = "License verification logic")]
public class KeygenClient : IKeygenClient
{
    [Obfuscation(Feature = "ultra", Exclude = false)]
    public async Task<LicenseValidationResult?> ValidateLicenseAsync(...) { ... }
}

// Machine fingerprint service
[VmProtect(VmProtectionPriority.High)]
public class MachineFingerprintService : IMachineFingerprintService { ... }
```

Критичний код заслуговує на сильніший захист, оскільки розкриття основної логіки спричинило б реальні проблеми.

### 2. Шифрування рядків і дешифрування під час виконання

Шифруйте рядки під час збирання та розшифруйте їх під час виконання:

```csharp
public static class StringDecryption
{
    [VmProtect(VmProtectionPriority.High, Reason = "CRITICAL SECURITY")]
    public static string DecryptString(byte[] encryptedData, byte[] key, byte[] iv)
    {
        using var aes = Aes.Create();
        aes.Key = key;
        aes.IV = iv;

        using var decryptor = aes.CreateDecryptor();
        using var ms = new MemoryStream(encryptedData);
        using var cs = new CryptoStream(ms, decryptor, CryptoStreamMode.Read);
        using var reader = new StreamReader(cs);

        return reader.ReadToEnd();
    }
}

// Production constant accessor (lazy loading + caching)
public static class ProductionConstants
{
    private static string? _cachedLicenseServerUrl;

    public static string GetLicenseServerUrl(IOptions<LicenseOptions> options)
    {
        if (_cachedLicenseServerUrl == null)
        {
            var encrypted = GetEncryptedLicenseServerUrl();
#if DEBUG
            _cachedLicenseServerUrl = options.Value.PrimaryServer.Url;
#else
            _cachedLicenseServerUrl = StringDecryption.DecryptString(
                encrypted,
                GetEncryptionKey(),
                GetEncryptionIV());
#endif
        }
        return _cachedLicenseServerUrl;
    }
}
```

Цей крок має значення, оскільки конфіденційну інформацію ніколи не можна залишати відкритим текстом.

### 3. Перевірка захисту VMP

Після збірки ви повинні перевірити, чи успішно застосовано захист; інакше ви не можете бути впевнені, що він справді працює:

```csharp
// Example verification script
public bool VerifyProtection(string assemblyPath)
{
    // 1. Check the VMP signature
    var bytes = File.ReadAllBytes(assemblyPath);
    var vmpSignature = Encoding.ASCII.GetBytes("VMProtect");
    if (bytes.Any(b => vmpSignature.Contains(b)))
    {
        return true;
    }

    // 2. Check for file size changes (the protected file is usually larger)
    var originalInfo = new FileInfo(assemblyPath.Replace(".dll", ".bak"));
    if (originalInfo.Exists)
    {
        var sizeRatio = (double)new FileInfo(assemblyPath).Length / originalInfo.Length;
        return sizeRatio > 1.1;
    }

    return false;
}
```

Перевірку варто робити завжди, оскільки інакше проблеми можуть проскочити непомітно.

### 4. Примітки та застереження

Тут є кілька підводних каменів, які заслуговують на особливу увагу:

1. **Не обфусцуйте весь код**: публічні API, визначення інтерфейсу та класи DTO зазвичай не потребують захисту. Надмірна обфускація може погіршити продуктивність і налагодження. Проект HagiCode навчився цьому на важкому шляху.

2. **Захист засобів доступу до ключів**: методи, які отримують ключі шифрування, повинні отримувати той самий або вищий рівень захисту, ніж самі зашифровані дані; інакше вся установка втрачає свою цінність.

3. **Баланс тестування та виробництва**: збірки DEBUG повинні пропускати шифрування, щоб спростити розробку та налагодження, тоді як збірки RELEASE повинні забезпечувати повний захист. Не забудьте розділити їх за допомогою умовної компіляції, наприклад `#if DEBUG`.

4. **Розгляньте середовище Docker**: для забезпечення сумісності інструментів для роботи VMP у Linux потрібен контейнерний підхід. HagiCode використовує контейнерне рішення Wine + VMP для вирішення кросплатформної проблеми.

5. **Перевірка є обов’язковою**: після завершення збірки ви повинні переконатися, що захист застосовано успішно. Інакше конфіденційний код все ще може бути розкритий, а показаний раніше код перевірки існує саме для цієї мети.

## Висновок

За допомогою цієї багаторівневої стратегії захисту HagiCode створив комплексну систему захисту коду, яка охоплює від базової обфускації до захисту віртуальної машини:

- **Рівень 1**: Використовуйте `ObfuscationAttribute` для розмітки базової лінії та надання підказок стороннім інструментам
- **Рівень 2**: Використовувати власний `VmProtectAttribute` декларації для вираження наміру захисту та пріоритету
- **Рівень 3**: використовуйте захист віртуальної машини VMP для перетворення критичного коду в незворотні інструкції віртуальної машини
- **Рівень 4**: автоматично скануйте та застосовуйте захист під час збірки, а потім перевірте результат

Цей підхід може протистояти звичайним інструментам декомпіляції, а також краще протистояти передовим атакам зворотного проектування. Якщо ви створюєте програму .NET, яка потребує захисту коду, я сподіваюся, що це дасть вам принаймні корисну точку відліку.
