## Hagicode.Libs: Інженерна практика для уніфікованої інтеграції кількох команд командного рядка AI Coding Assistant

> Під час розробки проекту HagiCode нам потрібно було інтегрувати кілька CLI-помічників кодування штучного інтелекту одночасно, включаючи Claude Code, Codex і CodeBuddy. Кожен CLI має різні інтерфейси, параметри та вихідні формати, а повторюваний інтеграційний код ускладнював підтримку проекту. У цій статті ми розповідаємо, як ми створили уніфікований рівень абстракції за допомогою HagiCode.Libs, щоб вирішити цю інженерну проблему. Ви також можете сказати, що це просто важко зароблений досвід, отриманий з пасток, з якими ми вже стикалися.

## Фон

Ринок помічників кодування ШІ зараз досить жвавий. Окрім Claude Code, є також Codex OpenAI, CodeBuddy Zhipu тощо. Як проект помічника кодування штучного інтелекту, HagiCode має інтегрувати ці різні інструменти CLI в кілька підпроектів, включаючи робочий стіл, серверну частину та веб.

Спочатку проблему можна було вирішити. Інтеграція одного CLI займала лише кілька сотень рядків коду. Але оскільки кількість CLI, яку нам потрібно підтримувати, зростала, все почало ставати безладним.

Кожен CLI має власний формат аргументів командного рядка, різні вимоги до змінних середовища та широкий вибір форматів виводу. Деякі виводять JSON, деякі виводять потоковий JSON, а деякі виводять простий текст. Крім того, існують проблеми сумісності між платформами. Виявлення виконуваних файлів і керування процесами працюють дуже по-різному в системах Windows і Unix, тому дублювання коду постійно зростало. По правді кажучи, це було трохи більше Ctrl+C і Ctrl+V, але обслуговування швидко стало болючим.

Найбільше розчарування полягало в тому, що кожного разу, коли ми хотіли додати підтримку нової можливості CLI, нам доводилося змінювати той самий код у кількох проектах. Очевидно, що такий підхід не був стійким у довгостроковій перспективі. У кодексу також є характер; дублюйте його занадто багато разів, і це починає створювати проблеми.

## Про HagiCode

Підхід, поширений у цій статті, базується на нашому практичному досвіді в [HagiCode](https://hagicode.com) проект. HagiCode — це проект помічника кодування штучного інтелекту з відкритим вихідним кодом, якому потрібно підтримувати кілька підпроектів одночасно, включаючи інтерфейсне розширення VSCode, серверні служби штучного інтелекту та кросплатформний клієнт для настільних ПК. У певному сенсі саме це складне, багатомовне, мультиплатформене середовище призвело до народження HagiCode.Libs. Можна сказати, що нас змусили, і нехай буде так.

## Аналіз: пошук спільної мови

Незважаючи на те, що кожен з цих CLI помічників кодування AI має свої особливості, з технічної точки зору вони мають кілька очевидних рис:

**Схожі моделі взаємодії**: усі вони запускають процес CLI, надсилають запит, отримують потокові відповіді, аналізують повідомлення, а потім завершують або продовжують сеанс. Зрештою, весь потік відбувається за тією ж основною формою.

**Схожі потреби конфігурації**: усі вони потребують автентифікації ключа API, налаштування робочого каталогу, вибору моделі, контролю дозволів інструменту та керування сеансами. Зрештою, всі заробляють на життя API; відмінності здебільшого залежать від смаку.

**Ті самі крос-платформні виклики**: усі вони мають вирішити шлях до виконуваного файлу (`claude` проти `claude.exe` проти `/usr/local/bin/claude`), запуск процесу та обробка змінних середовища, екранування команд оболонки та побудова аргументів. Кросплатформна робота – це боляче, як би ви це не описували. Тільки люди, які потрапили в пастки, дійсно розуміють різницю між Windows і Unix.

Виходячи з цього аналізу, нам потрібен був уніфікований рівень абстракції, який міг би забезпечити послідовний інтерфейс, інкапсулювати логіку виявлення кросплатформного CLI, обробляти синтаксичний аналіз потокового виводу та підтримувати як сценарії впровадження залежностей, так і сценарії без DI. Це така проблема, що голова болить від однієї лише думки про неї, але вам все одно доведеться з нею зіткнутися. Зрештою, це наш власний проект, тож ми маємо його завершити, навіть якщо нам доведеться через нього проплакатися.

## Рішення: HagiCode.Libs

Ми створили **HagiCode.Libs**, легкий робочий простір бібліотеки .NET 10, випущений за ліцензією MIT і зараз опублікований на [GitHub](https://github.com/HagiCode-org/Hagicode.Libs). Можливо, це не якийсь шедевр, який потряс світ, але він справді корисний для вирішення реальних проблем.

### Структура проекту

```
HagiCode.Libs/
├── src/
│   ├── HagiCode.Libs.Core/           # Core capabilities
│   │   ├── Discovery/                 # CLI executable discovery
│   │   ├── Process/                   # Cross-platform process management
│   │   ├── Transport/                 # Streaming message transport
│   │   └── Environment/               # Runtime environment resolution
│   ├── HagiCode.Libs.Providers/       # Provider implementations
│   │   ├── ClaudeCode/                # Claude Code provider
│   │   ├── Codex/                     # Codex provider
│   │   └── Codebuddy/                 # CodeBuddy provider
│   ├── HagiCode.Libs.ConsoleTesting/  # Testing framework
│   ├── HagiCode.Libs.ClaudeCode.Console/
│   ├── HagiCode.Libs.Codex.Console/
│   └── HagiCode.Libs.Codebuddy.Console/
└── tests/                             # xUnit tests
```

### Цілі проектування

Розробляючи HagiCode.Libs, ми дотримувалися кількох принципів. Усі вони прийшли з уроків, засвоєних важким шляхом:

**Нуль важких залежностей фреймворку**: він не залежить від ABP чи будь-якої іншої великої фреймворку, що робить його легким. У наші дні, чим менше у вас залежностей, тим менше у вас головних болів. Більшість людей вже хоча б раз потрапили в пекло залежності.

**Кросплатформна підтримка**: вбудована підтримка Windows, macOS і Linux без написання окремого коду для різних платформ. Одна кодова база, яка працює скрізь, — це дуже добре.

**Потокова обробка**: вихід CLI обробляється за допомогою асинхронних потоків, що набагато краще відповідає сучасним шаблонам програмування .NET. Часи міняються, і асинхронний – це король.

**Гнучка інтеграція**: підтримує сценарії впровадження залежностей, а також дозволяє пряме створення екземплярів. У різних людей різні вподобання, тому ми хотіли, щоб це було зручно в будь-якому випадку.

## Як ним користуватися

### Через ін’єкцію залежності

Якщо ваш проект уже використовує ін’єкцію залежностей, як-от ASP.NET Core або загальний хост, ви можете інтегрувати його безпосередньо. Це дрібниця, але добре поводиться:

```csharp
using HagiCode.Libs.Providers;
using Microsoft.Extensions.DependencyInjection;

var services = new ServiceCollection();
services.AddHagiCodeLibs();

await using var provider = services.BuildServiceProvider();
var claude = provider.GetRequiredService<ICliProvider<ClaudeCodeOptions>>();

var options = new ClaudeCodeOptions
{
    ApiKey = "your-api-key",
    Model = "claude-sonnet-4-20250514"
};

await foreach (var message in claude.ExecuteAsync(options, "Hello, Claude!"))
{
    Console.WriteLine($"{message.Type}: {message.Content}");
}
```

### Пряма інстанціація

Якщо ви пишете простий сценарій або працюєте в сценарії без DI, створення екземпляра безпосередньо також працює. Простіше кажучи, це залежить від ваших особистих уподобань:

```csharp
var claude = new ClaudeCodeProvider();
var options = new ClaudeCodeOptions
{
    ApiKey = "sk-ant-xxx",
    Model = "claude-sonnet-4-20250514"
};

await foreach (var message in claude.ExecuteAsync(options, "Help me write a quicksort"))
{
    // Handle messages
}
```

Обидва підходи використовують ту саму базову реалізацію, тому ви можете вибрати стиль інтеграції, який найкраще підходить для вашого проекту. У цьому світі немає універсальної правильної відповіді. Те, що вам підходить, найкращий варіант. Це може звучати кліше, але це правда.

## Практичний досвід

### 1. Виділені консолі тестування

Кожен постачальник має власний спеціальний проект консолі тестування, що полегшує незалежну перевірку інтеграції. Тестування – це одна з тих речей, де, якщо ви збираєтеся це робити, ви повинні робити це належним чином:

```bash
# Claude Code tests
dotnet run --project src/HagiCode.Libs.ClaudeCode.Console -- --test-provider
dotnet run --project src/HagiCode.Libs.ClaudeCode.Console -- --test-all claude

# CodeBuddy tests
dotnet run --project src/HagiCode.Libs.Codebuddy.Console -- --test-provider codebuddy-cli

# Codex tests
dotnet run --project src/HagiCode.Libs.Codex.Console -- --test-provider codex-cli
```

Сценарії тестування охоплюють кілька ключових випадків:
- **Ping**: перевірка працездатності для підтвердження наявності CLI
- **Simple Prompt**: базовий тест підказок
- **Складна підказка**: багатоповоротний тест розмови
- **Відновлення/відновлення сеансу**: тест відновлення сеансу
- **Аналіз сховища**: тест аналізу сховища

Цей автономний дизайн консолі тестування особливо корисний під час налагодження, оскільки він дозволяє нам швидко визначити, чи є проблема в рівні HagiCode.Libs, чи в самому CLI. Насправді налагодження полягає лише в пошуку проблеми. Якщо напрямок правильний, ви вже на півдорозі.

### 2. Міжплатформна перевірка CI/CD

Сумісність між платформами є однією з основних цілей HagiCode.Libs. Ми налаштували робочий процес GitHub Actions `.github/workflows/cli-discovery-cross-platform.yml` щоб запустити реальну перевірку виявлення CLI `ubuntu-latest`, `macos-latest`, і `windows-latest`.

Це гарантує, що кожна зміна коду не порушує кросплатформну сумісність. Під час локальної розробки ви також можете відтворити його за допомогою наступних команд. Зрештою, ви не можете просити CI взяти провину за все. Ваше локальне середовище також повинно мати можливість запускати його:

```bash
npm install --global @anthropic-ai/claude-code@2.1.79
HAGICODE_REAL_CLI_TESTS=1 dotnet test --filter "Category=RealCli"
```

### 3. Обробка потоку повідомлень

HagiCode.Libs використовує асинхронні потоки для обробки вихідних даних CLI. У порівнянні з традиційними підходами зворотного виклику або подіями, це набагато краще відповідає стилю асинхронного програмування сучасної .NET. Зрештою, це просто те, як технології рухаються вперед, подобається це комусь чи ні:

```csharp
public async IAsyncEnumerable<CliMessage> ExecuteAsync(
    TOptions options,
    string prompt,
    [EnumeratorCancellation] CancellationToken cancellationToken = default)
{
    // Start the CLI process
    // Parse streaming JSON output
    // Yield the CliMessage sequence
}
```

Типи повідомлень включають:
- `user`: повідомлення користувача
- `assistant`: відповідь помічника
- `tool_use`: виклик інструменту
- `result`: завершення сесії

Ця конструкція дозволяє абонентам гнучко керувати потоковим виведенням, чи то для відображення в реальному часі, буферизованої постобробки чи переадресації до інших служб. Навіщо хвилюватися, чи небо сонячне, чи хмарне? Важливо те, що коли ідея відкривається, ви можете використовувати її як завгодно.

### 4. Дослідження репозиторію Git

Модуль HagiCode.Libs.Exploration забезпечує виявлення сховища Git і перевірку стану, що особливо корисно в сценаріях аналізу сховища. Ця функція також народилася через необхідність, оскільки HagiCode потрібно аналізувати репозиторії:

```csharp
// Discover Git repositories
var repositories = await GitRepositoryDiscovery.DiscoverAsync("/path/to/search");

// Get repository information
var info = await GitRepository.GetInfoAsync(repoPath);
Console.WriteLine($"Branch: {info.Branch}, Remote: {info.RemoteUrl}");
Console.WriteLine($"Has uncommitted changes: {info.HasUncommittedChanges}");
```

Можливості аналізу коду HagiCode використовують цей модуль для визначення структури проекту та стану Git. Це хороший приклад повного використання того, що ми створили.

## Варто звернути увагу

Виходячи з нашої практики в проекті HagiCode, є кілька моментів, які заслуговують на особливу увагу. Усі вони реальні проблеми, які потребують обережності:

**Безпека ключа API**: не закодуйте ключі API у своєму коді. Натомість використовуйте змінні середовища або керування конфігурацією. HagiCode.Libs підтримує передачу конфігурації через об’єкти Options, що полегшує інтеграцію з різними джерелами конфігурації. Коли мова йде про безпеку, немає такого поняття, як бути надто обережним.

**Закріплення версії CLI**: у CI/CD ми закріплюємо певні версії, наприклад `@anthropic-ai/claude-code@2.1.79`, щоб зменшити невизначеність, спричинену дрейфом версій. Також доцільно використовувати фіксовані версії в локальній розробці. Визначення версій може бути болючим. Якщо ви не закріпите версії, проблема дуже швидко навчить вас уроку.

**Категоризація тестів**: тести за замовчуванням використовують фальшиві постачальники, щоб підтримувати їх детермінованість і швидкість, тоді як справжні тести CLI мають бути активовані явно. Це забезпечує швидкий зворотній зв’язок CI, але при необхідності дозволяє перевірку в реальному середовищі. Встановити цей баланс ніколи не буває легко. Швидкість і стабільність завжди вимагають компромісів.

**Керування сеансами**: різні CLI мають різні механізми відновлення сеансів. Клод Код використовує `.claude/` каталог для зберігання сеансів, тоді як Codex і CodeBuddy мають власні підходи. Використовуючи їх, обов’язково перевірте відповідну документацію та зрозумійте деталі їхніх механізмів збереження сеансу. Немає шкоди в чіткому розумінні цього.
