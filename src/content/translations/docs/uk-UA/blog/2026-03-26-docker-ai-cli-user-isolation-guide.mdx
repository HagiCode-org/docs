## Запуск інструментів AI CLI в контейнерах Docker: практичний посібник із ізоляції користувачів і постійних томів

> Інтеграція таких інструментів кодування штучного інтелекту, як Claude Code, Codex і OpenCode, у контейнерні середовища звучить просто, але всюди є приховані складності. У цій статті детально розповідається про те, як проект HagiCode вирішує основні проблеми в розгортанні Docker, включаючи дозволи користувачів, збереження конфігурації та керування версіями, щоб ви могли уникнути поширених пасток.

## Фон

Коли ми вирішили запустити інструменти кодування штучного інтелекту CLI всередині контейнерів Docker, найбільш інтуїтивно зрозумілою думкою було, напевно, таке: «Хіба контейнери не просто root? Чому б не встановити все безпосередньо і назвати це готовим?» Насправді ця, здавалося б, проста ідея приховує кілька основних проблем, які необхідно вирішити.

По-перше, **обмеження безпеки є першою перешкодою**. Візьміть Claude CLI як приклад: він явно забороняє працювати від імені користувача root. Це обов'язкова перевірка безпеки, і якщо root виявлено, він відмовляється запускатися. Ви можете подумати, чи не можу я просто переключити користувачів за допомогою `USER` директива? Це не так просто. Досі існує проблема зіставлення між некореневим користувачем усередині контейнера та дозволами користувача на головній машині.

По-друге, **наполегливість держави — друга пастка**. Claude Code вимагає входу, Codex має власну конфігурацію, а OpenCode також має каталог кешу. Якщо вам доведеться переналаштовувати все щоразу, коли контейнер перезавантажується, вся ідея «автоматизації» втрачає сенс. Нам потрібні ці конфігурації, щоб зберігатися після життєвого циклу контейнера.

Третя проблема — **послідовність дозволів**. Чи можуть процеси всередині контейнера отримати доступ до файлів конфігурації, створених користувачем хосту? Невідповідність UID/GID часто спричиняє помилки дозволу на файл, і це надзвичайно часто в реальних розгортаннях.

Ці проблеми можуть виглядати незалежними, але на практиці вони тісно пов'язані. Під час розробки HagiCode ми поступово виробили практичне рішення. Далі я поділюся технічними деталями та уроками, отриманими з цих пасток.

## Чому ми не можемо просто використовувати root?

Тут є поширене непорозуміння: контейнери Docker за замовчуванням запускаються від імені root, тож чому б просто не встановити інструменти від імені root? Якщо ви так думаєте, Клод КЛІ швидко навчить вас інакше.

```bash
# Run Claude CLI directly as root? No.
docker run --rm -it --user root myimage claude
# Output: Error: This command cannot be run as root user
```

Це жорстке обмеження безпеки в CLAude CLI. Причина проста: ці інструменти CLI читають і записують конфіденційну конфігурацію користувача, включаючи маркери API, локальні кеші та навіть сценарії, написані користувачем. Їх запуск із привілеями root створює занадто великий ризик.

Тож виникає запитання: як ми можемо задовольнити вимоги безпеки CLI, зберігаючи при цьому гнучке керування контейнерами? Нам потрібно змінити своє ставлення до цього: **замість того, щоб змінювати користувачів під час виконання, створіть спеціального користувача на етапі створення образу**.

## Створення спеціального користувача: більше, ніж просто зміна імені

Можна подумати, що додавання одного `USER` рядка до Dockerfile достатньо. Це дійсно найпростіший підхід, але він недостатньо надійний.

### Статичне створення проти динамічного відображення

Підхід HagiCode полягає у створенні a `hagicode` користувач з UID 1000, який зазвичай відповідає користувачеві за замовчуванням на більшості хост-машин:

```dockerfile
RUN groupadd -o -g 1000 hagicode && \
    useradd -o -u 1000 -g 1000 -s /bin/bash -m hagicode && \
    mkdir -p /home/hagicode/.claude && \
    chown -R hagicode:hagicode /home/hagicode
```

Але це вирішує лише проблему вбудованого користувача в образ. Що робити, якщо користувач хосту має UID 1001? Ви все ще повинні підтримувати динамічне відображення під час запуску контейнера.

docker-entrypoint.sh містить ключову логіку:

```bash
if [ -n "$PUID" ] && [ -n "$PGID" ]; then
    if ! id hagicode >/dev/null 2>&1; then
        groupadd -g "$PGID" hagicode
        useradd -u "$PUID" -g "$PGID" -s /bin/bash -m hagicode
    fi
fi
```

Перевага цього дизайну очевидна: **використовуйте UID за замовчуванням 1000 під час створення образу, а потім динамічно коригуйте під час виконання за допомогою `PUID` і `PGID` змінні середовища**. Незалежно від того, який UID має користувач хосту, право власності на файли конфігурації залишається правильним.

## Філософія дизайну постійних томів

Кожен інструмент AI CLI має власний бажаний каталог конфігурації, тому їх потрібно зіставляти по одному:

| Інструмент CLI | Шлях у контейнері | Названий том |
|---------|-----------|--------|
| Клод | `/home/hagicode/.claude` | `claude-data` |
| Кодекс | `/home/hagicode/.codex` | `codex-data` |
| OpenCode | `/home/hagicode/.config/opencode` | `opencode-config-data` |

Навіщо використовувати іменовані томи замість прив’язуваних монтувань? Три причини:

1. **Простіше керування**: Docker автоматично керує іменованими томами, тому вам не потрібно створювати каталоги хостів вручну.
2. **Ізоляція дозволів**: початковий вміст томів створюється користувачем усередині контейнера, щоб уникнути конфліктів дозволів із хостом.
3. **Незалежна міграція**: томи можуть існувати незалежно від контейнерів, тому дані не втрачаються під час оновлення зображень.

docker-compose-builder-web автоматично генерує відповідну конфігурацію тому:

```yaml
volumes:
  claude-data:
  codex-data:
  opencode-config-data:

services:
  hagicode:
    volumes:
      - claude-data:/home/hagicode/.claude
      - codex-data:/home/hagicode/.codex
      - opencode-config-data:/home/hagicode/.config/opencode
    user: "${PUID:-1000}:${PGID:-1000}"
```

Зверніть увагу на `user` поле тут: `PUID` і `PGID` вводяться через змінні середовища, щоб гарантувати, що процеси всередині контейнера виконуються з ідентифікатором, який відповідає користувачеві хосту. Ця деталь важлива, оскільки проблеми з дозволами важко виправляти, коли вони з’являються.

## Управління версіями: вбудовані версії зі змінами часу виконання

Закріплення версій зображень Docker є важливим для відтворюваності. Але в реальній розробці нам часто потрібно протестувати новішу версію або терміново виправити помилку. Якби нам доводилося кожного разу перебудовувати зображення, робочий процес був би надто неефективним.

Стратегією HagiCode є **фіксовані версії за замовчуванням із заміною часу виконання як механізмом розширення**. Це прагматичний інженерний компроміс між стабільністю та гнучкістю.

Dockerfile.template закріплює версії тут:

```dockerfile
USER hagicode
WORKDIR /home/hagicode

# Configure the global npm install path
RUN mkdir -p /home/hagicode/.npm-global && \
    npm config set prefix '/home/hagicode/.npm-global'

# Install CLI tools using pinned versions
RUN npm install -g @anthropic-ai/claude-code@2.1.71 && \
    npm install -g @openai/codex@0.112.0 && \
    npm install -g opencode-ai@1.2.25 && \
    npm cache clean --force
```

docker-entrypoint.sh підтримує перевизначення часу виконання:

```bash
install_cli_override_if_needed() {
    local package_name="$2"
    local override_version="$5"

    if [ -n "$override_version" ]; then
        gosu hagicode npm install -g "${package_name}@${override_version}"
    fi
}

# Example usage
install_cli_override_if_needed "" "@anthropic-ai/claude-code" "" "" "${CLAUDE_CODE_CLI_VERSION}"
```

Це дозволяє тестувати нову версію за допомогою змінної середовища без перебудови образу:

```bash
docker run -e CLAUDE_CODE_CLI_VERSION=2.2.0 myimage
```

Цей дизайн практичний, оскільки ніхто не хоче перебудовувати зображення щоразу, коли тестує нову функцію.

## Автоматичне введення конфігурації

На додаток до налаштування інструментів CLI вручну, деякі сценарії вимагають автоматичного введення конфігурації. Найбільш типовим прикладом є маркер API.

```bash
if [ -n "$ANTHROPIC_AUTH_TOKEN" ]; then
    mkdir -p /home/hagicode/.claude
    cat > /home/hagicode/.claude/settings.json <<EOF
{
  "env": {
    "ANTHROPIC_AUTH_TOKEN": "${ANTHROPIC_AUTH_TOKEN}"
  }
}
EOF
    chown -R hagicode:hagicode /home/hagicode/.claude
fi
```

Тут важливі дві речі: **передайте конфіденційну інформацію через змінні середовища замість того, щоб жорстко кодувати її в образ**, і переконайтеся, що право власності на файли конфігурації встановлено правильно, інакше інструменти CLI не зможуть їх прочитати.

## Найкращі практики та контрольний список підводних каменів

### Проблеми з невідповідністю дозволів

Це найпростіша пастка, в яку можна потрапити. Користувач хоста має UID 1001, тоді як контейнер використовує 1000, тому файли, створені з одного боку, не можуть бути доступні з іншого.

```bash
# Correct approach: make the container match the host user
docker run \
    -e PUID=$(id -u) \
    -e PGID=$(id -g) \
    myimage
```

Ця проблема дуже поширена, і вона може засмучувати, коли ви з нею стикаєтеся вперше.

### Конфігурація зникає після перезапуску контейнера

Якщо після кожного перезапуску ви знову входите в систему, перевірте, чи не забули ви підключити постійний том:

```yaml
volumes:
  - claude-data:/home/hagicode/.claude
```

Немає нічого більшого розчарування, ніж ретельне налаштування конфігурації, щоб побачити, як вона зникає.

### Правильний спосіб оновлення версій

Не біжи `npm install -g` безпосередньо всередині працюючого контейнера. Правильними підходами є:

1. Встановіть змінну середовища, щоб ініціювати інсталяцію заміни.
2. Або перебудувати імідж.

```bash
# Option 1: runtime override
docker run -e CLAUDE_CODE_CLI_VERSION=2.2.0 myimage

# Option 2: rebuild the image
docker build -t myimage:v2 .
```

До Риму веде кілька доріг, але деякі дороги більш гладкі, ніж інші.

### Контрольний список посилення безпеки

- Передайте маркери API через змінні середовища, а не записуйте їх у зображення.
- Установіть дозволи файлу конфігурації на `600`.
- Завжди запускайте програму як користувач без права root.
- Регулярно оновлюйте версії CLI, щоб усунути вразливості безпеки.

Безпека завжди важлива, але справжньою проблемою є її послідовне забезпечення на практиці.

## Розширення підтримки нових інструментів CLI

Якщо ви хочете підтримувати новий інструмент CLI у майбутньому, виконайте лише три кроки:

1. **Dockerfile.template**: додайте крок встановлення.
2. **docker-entrypoint.sh**: додайте логіку заміни версії.
3. **docker-compose-builder-web**: додайте постійне зіставлення томів.

Цей шаблонний дизайн робить розширення простим без зміни основної логіки.

## Висновок

Запуск інструментів AI CLI в контейнерах Docker передбачає три основні завдання: **дозволи користувача, збереження конфігурації та керування версіями**. Поєднуючи виділених користувачів, ізоляцію іменованих томів і заміни на основі змінних середовища, проект HagiCode створив безпечну та гнучку архітектуру розгортання.

Основні моменти дизайну:

- **Ізоляція користувача**: створіть виділеного користувача на етапі створення образу з підтримкою динамічного виконання `PUID`/`PGID` картографування.
- **Стратегія збереження**: кожен інструмент CLI отримує власний іменований том, тому перезапуски не впливають на конфігурацію.
- **Гнучкість версій**: фіксовані параметри за замовчуванням забезпечують відтворюваність, тоді як заміни часу виконання забезпечують простір для тестування.
- **Автоматизована конфігурація**: чутливу конфігурацію можна вводити автоматично за допомогою змінних середовища.

Це рішення стабільно працює в проекті HagiCode протягом певного часу, і я сподіваюся, що воно пропонує корисні орієнтири для розробників із подібними потребами.
