## Автоматизація синхронізації зображень із Docker Hub до Azure ACR

> У цій статті пояснюється, як використовувати дії GitHub і інструмент синхронізації зображень для автоматизації синхронізації зображень із Docker Hub до реєстру контейнерів Azure, вирішуючи повільний доступ до Docker Hub у Китаї та деяких регіонах Azure, одночасно покращуючи доступність зображень і ефективність розгортання в середовищах Azure.

## Передумови / Вступ

Проект HagiCode використовує зображення Docker як основні компоненти середовища виконання, а основні зображення розміщено на Docker Hub. Оскільки проект розширювався, а потреби в розгортанні Azure збільшувалися, ми зіткнулися з такими проблемними моментами:

- Низька швидкість отримання зображень, доступ до Docker Hub обмежений у Китаї та деяких регіонах Azure
- Єдине джерело зображення створює єдину точку ризику відмови
- Реєстр контейнерів Azure забезпечує кращу продуктивність мережі та інтеграцію під час роботи в Azure

Щоб вирішити ці проблеми, нам потрібен був автоматичний механізм синхронізації зображень для регулярної синхронізації зображень із Docker Hub у Azure ACR, гарантуючи, що користувачі зможуть отримувати більшу швидкість отримання зображень і вищу доступність у середовищах Azure.

## Порівняння технічних рішень

При визначенні рішення ми порівнювали кілька технічних варіантів:

### 1. image-syncer (остаточний вибір)
- Поступова синхронізація: синхронізує лише змінені шари зображення, що значно зменшує передачу по мережі
- Відновити підтримку: синхронізація може продовжуватися після перебоїв у мережі
- Контроль паралелізму: настроювані робочі потоки покращують ефективність синхронізації великих зображень
- Надійна обробка помилок: вбудований механізм повторних спроб у випадку збоїв (3 повторних спроби за замовчуванням)
- Спрощене розгортання: єдиний двійковий файл без залежностей
- Підтримка кількох реєстрів: сумісна з Docker Hub, Azure ACR, Harbor тощо

### 2. Docker CLI
- Немає додаткової синхронізації: щоразу необхідно витягувати весь вміст зображення
- Нижча ефективність: великий обсяг передачі по мережі та довший час виконання
- Простий у використанні: на основі знайомих `docker pull` і `docker push` команди

### 3. Azure CLI
- Вища складність: необхідно налаштувати автентифікацію Azure CLI
- Обмежена функціональність: `az acr import` має відносно вузький обсяг
- Рідна інтеграція: добре інтегрується зі службами Azure

## Етапи реалізації

### 1. Підготовка
- Створіть або підтвердьте реєстр контейнерів Azure на порталі Azure
- Створення облікових даних доступу до ACR (ім’я користувача та пароль)
- Підтвердьте права доступу до сховища образів Docker Hub

### 2. Налаштуйте секрети GitHub
Додайте такі секрети в налаштуваннях сховища GitHub:
- AZURE_ACR_USERNAME: ім’я користувача Azure ACR
- AZURE_ACR_PASSWORD: пароль Azure ACR

### 3. Створіть робочий процес GitHub Actions
Налаштуйте робочий процес у `.github/workflows/sync-docker-acr.yml`:
- Тригер за розкладом: щодня о 00:00 UTC
- Ручний курок: підтримує `workflow_dispatch`
- Додатковий тригер: запускається на поштовхи до `publish` гілка для швидкої синхронізації

### 4. Потік виконання робочого процесу

| Крок | Учасник | Дія | опис |
| --- | --- | --- | --- |
| 1 | Дії GitHub | Запустіть робочий процес | Ініціюється розкладом, ручним виконанням або надсиланням до `publish` відділення |
| 2 | Дії GitHub → image-syncer | Завантажте та запустіть інструмент синхронізації | Входить у фактичну фазу синхронізації |
| 3 | image-synccer → Docker Hub | Отримати маніфести зображень і список тегів | Читає метадані з вихідного сховища |
| 4 | image-synccer → Azure ACR | Отримайте наявні дані зображення з цільового сховища | Визначає поточний стан цільової сторони |
| 5 | імідж-синхрон | Порівняйте вихідні та цільові відмінності | Визначає шари зображення, які потребують синхронізації |
| 6 | image-synccer → Docker Hub | Витягнути змінені шари зображення | Передає лише вміст, який потребує оновлення |
| 7 | image-synccer → Azure ACR | Надішліть змінені шари зображення | Завершує поступову синхронізацію |
| 8 | image-syncer → Дії GitHub | Повернути статистику синхронізації | Містить результати, відмінності та інформацію про помилки |
| 9 | Дії GitHub | Записуйте журнали та завантажуйте артефакти | Підтримує подальший аудит і усунення несправностей |

## Реалізація робочого процесу GitHub Actions

Нижче наведено фактичну конфігурацію робочого процесу, що використовується (`.github/workflows/sync-docker-acr.yml`):

```yaml
name: Sync Docker Image to Azure ACR

on:
  schedule:
    - cron: "0 0 * * *" # Every day at 00:00 UTC
  workflow_dispatch: # Manual trigger
  push:
    branches: [publish]

permissions:
  contents: read

jobs:
  sync:
    runs-on: ubuntu-latest

    steps:
      - name: Checkout code
        uses: actions/checkout@v4

      - name: Download image-syncer
        run: |
          # Download the image-syncer binary
          wget https://github.com/AliyunContainerService/image-syncer/releases/download/v1.5.5/image-syncer-v1.5.5-linux-amd64.tar.gz
          tar -zxvf image-syncer-v1.5.5-linux-amd64.tar.gz
          chmod +x image-syncer

      - name: Create auth config
        run: |
          # Generate the authentication config file (YAML format)
          cat > auth.yaml <<EOF
          hagicode.azurecr.io:
            username: "${{ secrets.AZURE_ACR_USERNAME }}"
            password: "${{ secrets.AZURE_ACR_PASSWORD }}"
          EOF

      - name: Create images config
        run: |
          # Generate the image sync config file (YAML format)
          cat > images.yaml <<EOF
          docker.io/newbe36524/hagicode: hagicode.azurecr.io/hagicode
          EOF

      - name: Run image-syncer
        run: |
          # Run synchronization (using the newer --auth and --images parameters)
          ./image-syncer --auth=./auth.yaml --images=./images.yaml --proc=10 --retries=3

      - name: Upload logs
        if: always()
        uses: actions/upload-artifact@v4
        with:
          name: sync-logs
          path: image-syncer-*.log
          retention-days: 7
```

## Деталі конфігурації

### 1. Умови запуску
- Запланований тригер: `cron: "0 0 * * *"` курсує щодня о 00:00 UTC
- Ручний тригер: `workflow_dispatch` дозволяє користувачам запускати його вручну з інтерфейсу GitHub
- Натиснути тригер: `push: branches: [publish]` тригери при натисканні на гілку випуску для швидкої синхронізації

### 2. Конфігурація автентифікації (`auth.yaml`)
```yaml
hagicode.azurecr.io:
  username: "${{ secrets.AZURE_ACR_USERNAME }}"
  password: "${{ secrets.AZURE_ACR_PASSWORD }}"
```

### 3. Конфігурація синхронізації зображення
```yaml
docker.io/newbe36524/hagicode: hagicode.azurecr.io/hagicode
```
Ця конфігурація означає, що всі теги з `docker.io/newbe36524/hagicode` буде синхронізовано з `hagicode.azurecr.io/hagicode`.

### 4. Параметри зображення-синхронизатора
- `--auth=./auth.yaml`: шлях до конфігураційного файлу автентифікації
- `--images=./images.yaml`: шлях до конфігураційного файлу синхронізації зображень
- `--proc=10`: використання `10` одночасні працівники
- `--retries=3`: помилки повторної спроби `3` разів

## Примітки та найкращі практики

### 1. Рекомендації щодо безпеки
- Регулярно змінюйте паролі Azure ACR (рекомендовано кожні 90 днів)
- Використовуйте спеціальний обліковий запис служби ACR із дозволами лише для натискання
- Відстежуйте журнали доступу ACR, щоб вчасно виявити ненормальний доступ
- Не виводьте облікові дані в журналах
- Не надсилайте облікові дані до сховища коду

### 2. Оптимізація продуктивності
- Налаштуйте `--proc` параметр: налаштувати паралелізм на основі пропускної здатності мережі (рекомендовано `5-20`)
- Відстежуйте тривалість синхронізації: якщо синхронізація триває надто довго, подумайте про зниження паралельності
- Регулярно очищайте журнали: установіть розумний `retention-days` значення (`7` зараз)

### 3. Усунення несправностей

#### Проблема 1: Помилка автентифікації
```
Error: failed to authenticate to hagicode.azurecr.io
```
рішення:
1. Перевірте, чи правильно налаштовано секрети GitHub
2. Перевірте, чи не минув термін дії пароля Azure ACR
3. Переконайтеся, що дозволи облікового запису служби ACR правильні

#### Проблема 2. Час очікування мережі
```
Error: timeout waiting for response
```
рішення:
1. Перевірте підключення до мережі
2. Зменшіть кількість одночасних працівників ( `--proc` параметр)
3. Зачекайте, поки мережа відновиться, і повторно запустіть робочий процес

#### Проблема 3: Неповна синхронізація зображення
```
Warning: some tags failed to sync
```
рішення:
1. Перевірте журнали синхронізації, щоб визначити помилкові теги
2. Повторно запустіть робочий процес вручну, щоб знову синхронізувати
3. Переконайтеся, що вихідний образ на Docker Hub справний

### 4. Спостереження та оповіщення
- Регулярно перевіряйте сторінку «Дії», щоб підтвердити статус робочого процесу
- Увімкніть сповіщення GitHub, щоб негайно повідомляти про збої робочого процесу
- Відстежуйте використання сховища Azure ACR
- Регулярно перевіряйте узгодженість тегів у реєстрах

## Часті запитання та рішення

### Q1: Як синхронізувати певні теги замість усіх тегів?

Змінити `images.yaml` конфігураційний файл:
```yaml
# Sync only the latest and v1.0 tags
docker.io/newbe36524/hagicode:latest: hagicode.azurecr.io/hagicode:latest
docker.io/newbe36524/hagicode:v1.0: hagicode.azurecr.io/hagicode:v1.0
```

### Q2: Як мені синхронізувати кілька сховищ зображень?

Додати кілька рядків до `images.yaml`:
```yaml
docker.io/newbe36524/hagicode: hagicode.azurecr.io/hagicode
docker.io/newbe36524/another-image: hagicode.azurecr.io/another-image
```

### Q3: Як повторити спробу після збою синхронізації?
- Автоматична повторна спроба: image-syncer включає вбудований механізм повторної спроби (`3` повторює за замовчуванням)
- Повторна спроба вручну: натисніть `Re-run all jobs` на сторінці дій GitHub

### Q4: Як я можу переглянути детальний хід синхронізації?
- Перегляд журналів у реальному часі на сторінці Дії
- Завантажити `sync-logs` артефакт для перевірки повних файлів журналу
- Файли журналу містять статус синхронізації та швидкість передачі для кожного тегу

### Q5: Скільки часу займає синхронізація?
- Початкова повна синхронізація: зазвичай `10-30` хвилин залежно від розміру зображення
- Поступова синхронізація: зазвичай `2-5` хвилин, якщо зміни зображення незначні
- Час залежить від пропускної здатності мережі, розміру зображення та параметрів паралельності
