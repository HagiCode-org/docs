## Як автоматизувати випуски Steam за допомогою дій GitHub

> У цій статті представлено повне рішення, яке ми реалізували для автоматизованих випусків Steam у проекті HagiCode Desktop, охоплюючи наскрізний потік автоматизації від випуску GitHub до платформи Steam, включаючи ключові технічні деталі, такі як автентифікація Steam Guard і завантаження на багато платформи Depot.

## Фон

Процес випуску в Steam насправді значно відрізняється від традиційного розповсюдження додатків. Steam має власну повну систему доставки оновлень. Розробникам потрібно використовувати SteamCMD для завантаження артефактів збірки в мережу CDN Steam, а не просто вводити посилання для завантаження, як на інших платформах.

Проект HagiCode Desktop готується до випуску Steam, який представив кілька нових викликів для нашого робочого процесу випуску:

1. Нам потрібно було конвертувати існуючі артефакти збірки у формат, сумісний зі Steam.
2. Нам довелося завантажити їх у Steam через SteamCMD.
3. Нам також довелося впоратися з автентифікацією Steam Guard.
4. Нам потрібно було підтримувати мультиплатформене завантаження Depot для Linux, Windows і macOS.
5. Ми хотіли повністю автоматизувати перехід від GitHub Release до Steam.

У проекті вже було реалізовано «режим портативної версії», який дозволяє програмі виявляти корисні навантаження фіксованих служб, запаковані в `extra` каталог. Нашою метою було бездоганно інтегрувати цей режим портативної версії з розповсюдженням Steam.

## Про HagiCode

Рішення, наведене в цій статті, походить з нашого практичного досвіду в [HagiCode](https://hagicode.com) проект. HagiCode — це помічник кодування ШІ, який підтримує використання робочого столу. Оскільки ми активно працюємо над запуском у Steam, нам потрібно було створити надійний автоматизований робочий процес випуску.

## Архітектурний дизайн

Основою всього процесу випуску Steam є робочий процес GitHub Actions, який поділяє процес на три основні етапи:

```
┌─────────────────────────────────────────────────────────────┐
│ GitHub Actions Workflow (Steam Release)                    │
├─────────────────────────────────────────────────────────────┤
│ 1. Preparation Stage:                                      │
│    - Check out portable-version code                       │
│    - Download build artifacts from GitHub Release          │
│    - Extract and prepare the Steam content directory       │
│                                                             │
│ 2. SteamCMD Setup:                                         │
│    - Install or reuse SteamCMD                             │
│    - Authenticate with Steam Guard                         │
│                                                             │
│ 3. Release Stage:                                          │
│    - Generate Depot VDF configuration files                │
│    - Generate App Build VDF configuration files            │
│    - Invoke SteamCMD to upload to Steam                    │
└─────────────────────────────────────────────────────────────┘
```

Цей дизайн має кілька переваг:
- Він повторно використовує існуючі артефакти GitHub Release і уникає повторного створення тих самих результатів.
- Для безпечної ізоляції використовуються самостійні бігуни.
- Він підтримує перемикання між режимом попереднього перегляду та офіційними гілками випуску.
- Він включає повну обробку помилок і журналювання, що полегшує діагностику несправностей.

## Впровадження робочого процесу

### Дизайн параметрів тригера

Наш робочий процес підтримує такі ключові параметри:

```yaml
inputs:
  release:           # Portable Version release tag
    description: 'Version tag to release (for example v1.0.0)'
    required: true
  steam_preview:     # Whether to generate a preview build
    description: 'Whether to use preview mode'
    required: false
    default: 'false'
  steam_branch:      # Steam branch to set live
    description: 'Target Steam branch'
    required: false
    default: 'preview'
  steam_description: # Build description override
    description: 'Build description'
    required: false
```

### Саморозміщена конфігурація Runner

З міркувань безпеки ми використовуємо власний бігун із `steam` етикетка:

```yaml
runs-on:
  - self-hosted
  - Linux
  - X64
  - steam
```

Це гарантує, що випуски Steam запускаються на спеціальному програмному забезпеченні та зберігають конфіденційні облікові дані в безпечній ізоляції.

### Контроль паралельності

Щоб запобігти взаємодії випусків однієї версії одна з одною, ми налаштували керування паралелізмом:

```yaml
concurrency:
  group: portable-version-steam-${{ github.event.inputs.release }}
  cancel-in-progress: false
```

Зверніть увагу на це `cancel-in-progress: false` встановлено тут, оскільки випуски Steam можуть зайняти деякий час, і ми не хочемо, щоб нещодавно запущений запуск скасував той, який уже завантажується.

## Реалізація основного сценарію

### Підготовка вхідних даних випуску

The `prepare-steam-release-input.mjs` сценарій відповідає за підготовку вхідних даних, необхідних для випуску:

```javascript
// Download the GitHub Release build manifest and artifact inventory
const buildManifest = await downloadBuildManifest(releaseTag);
const artifactInventory = await downloadArtifactInventory(releaseTag);

// Download compressed archives for each platform
for (const platform of ['linux-x64', 'win-x64', 'osx-universal']) {
  const artifactUrl = getArtifactUrl(artifactInventory, platform);
  await downloadArtifact(artifactUrl, platform);
}

// Extract into the Steam content directory structure
await extractToSteamContent(sources, contentRoot);
```

### Аутентифікація Steam Guard

Steam вимагає, щоб облікові записи були захищені за допомогою Steam Guard, тому ми запровадили алгоритм генерації коду на основі загального секрету:

```javascript
function generateSteamGuardCode(sharedSecret, timestamp = Date.now()) {
  const secret = decodeSharedSecret(sharedSecret);
  const time = Math.floor(timestamp / 1000 / 30);

  const timeBuffer = Buffer.alloc(8);
  timeBuffer.writeBigUInt64BE(BigInt(time));

  // Use HMAC-SHA1 to generate a time-based one-time code
  const hash = crypto.createHmac('sha1', secret)
    .update(timeBuffer)
    .digest();

  // Convert it into a 5-character Steam Guard code
  const code = steamGuardCode(hash);
  return code;
}
```

Ця реалізація базується на механізмі TOTP (одноразовий пароль на основі часу) Steam Guard, який генерує новий код підтвердження кожні 30 секунд.

### Генерація конфігурації VDF

VDF (формат даних клапана) — це формат конфігурації, який використовує Steam. Нам потрібно створити два типи файлів VDF:

**Depot VDF** використовується для налаштування вмісту для кожної платформи:

```javascript
function buildDepotVdf(depotId, contentRoot) {
  return [
    '"DepotBuildConfig"',
    '{',
    `  "DepotID" "${escapeVdf(depotId)}"`,
    `  "ContentRoot" "${escapeVdf(contentRoot)}"`,
    '  "FileMapping"',
    '  {',
    '    "LocalPath" "*"',
    '    "DepotPath" "."',
    '    "recursive" "1"',
    '  }',
    '}'
  ].join('\n');
}
```

**App Build VDF** використовується для налаштування всієї збірки програми:

```javascript
function buildAppBuildVdf(appId, depotBuilds, description, setLive) {
  const vdf = [
    '"appbuild"',
    '{',
    `  "appid" "${appId}"`,
    `  "desc" "${escapeVdf(description)}"`,
    `  "contentroot" "${escapeVdf(contentRoot)}"`,
    '  "buildoutput" "build_output"',
    '  "depots"',
    '  {'
  ];

  for (const [depotId, depotVdfPath] of Object.entries(depotBuilds)) {
    vdf.push(`    "${depotId}" "${depotVdfPath}"`);
  }

  if (setLive) {
    vdf.push(`  }`);
    vdf.push(`  "setlive" "${setLive}"`);
  }

  vdf.push('}');
  return vdf.join('\n');
}
```

### Виклик SteamCMD

Нарешті, завантаження виконується за допомогою SteamCMD:

```javascript
await runCommand(steamcmdPath, [
  '+login', steamUsername, steamPassword, steamGuardCode,
  '+run_app_build', appBuildPath,
  '+quit'
]);
```

Це останній стрибок у всьому робочому процесі. Коли це вдасться, звільнення виконано.

## Мультиплатформне обслуговування складів

Steam використовує систему Depot для керування контентом для різних платформ. Ми підтримуємо три основні депо:

| Платформа | Ідентифікатор депо | Підтримка архітектури |
|-----|-----------|---------|
| Linux | `linux-x64` | x64_64 |
| вікна | `win-x64` | x64_64 |
| macOS | `osx-universal` | універсальний, x64_64, arm64 |

Кожен Depot має власний каталог вмісту та файл конфігурації VDF. Це гарантує, що користувачі на різних платформах завантажують лише той вміст, який їм дійсно потрібен.

## Робочий процес випуску

### Крок 1. Підготуйте випуск GitHub

Спочатку створіть випуск GitHub у `portable-version` репозиторій, який включає:
- Стиснуті архіви для кожної платформи
- Побудувати маніфест (`{tag}.build-manifest.json`)
- Інвентаризація артефактів (`{tag}.artifact-inventory.json`)

### Крок 2. Запустіть робочий процес випуску Steam

Запустіть робочий процес вручну через GitHub Actions і заповніть необхідні параметри:
- `release`: тег версії для публікації, наприклад `v1.0.0`
- `steam_branch`: цільова гілка, наприклад `preview` або `public`
- `steam_preview`: чи використовувати режим попереднього перегляду

### Крок 3: Виконайте потік випуску автоматично

Робочий процес автоматично виконує такі дії:

1. Завантажте та розпакуйте артефакти GitHub Release.
2. Встановіть або оновіть SteamCMD.
3. Створіть файли конфігурації Steam VDF.
4. Автентифікація за допомогою Steam Guard.
5. Завантажуйте вміст на CDN Steam.
6. Оживити вказану гілку.

Після завершення цієї послідовності буде пройдено весь шлях випуску.

## Керівництво з налаштування

### Необхідна конфігурація секретів

Налаштуйте такі секрети в налаштуваннях сховища GitHub:

| Таємне ім'я | опис |
|-----------|------|
| `STEAM_USERNAME` | Ім'я користувача облікового запису Steam |
| `STEAM_PASSWORD` | Пароль облікового запису Steam |
| `STEAM_SHARED_SECRET` | Спільний секрет Steam Guard (необов'язково) |
| `STEAM_GUARD_CODE` | Код Steam Guard (необов'язково) |
| `STEAM_APP_ID` | Ідентифікатор програми Steam |
| `STEAM_DEPOT_ID_LINUX` | ID депо Linux |
| `STEAM_DEPOT_ID_WINDOWS` | Windows Depot ID |
| `STEAM_DEPOT_ID_MACOS` | macOS Depot ID |

У цих налаштуваннях немає нічого особливо незвичайного. Вам просто потрібні всі очікувані значення.

### Конфігурація змінної середовища

## Найкращі практики

### Керування автентифікацією Steam Guard

Під час першого запуску вам потрібно вручну ввести код Steam Guard. Після цього рекомендується налаштувати спільний секрет, щоб код міг генеруватися автоматично. Це дозволяє уникнути ручного втручання під час кожного випуску.

SteamCMD зберігає маркер входу та може повторно використовувати його під час наступних запусків. Вам все одно потрібно стежити за закінченням терміну дії маркера, оскільки після закінчення терміну дії маркера потрібна повторна автентифікація.

### Структура каталогу вмісту

Переконайтеся, що структура каталогу вмісту Steam правильна:

```
steam-content/
├── linux-x64/     # Linux platform content
├── win-x64/       # Windows platform content
└── osx-universal/ # macOS universal binary content
```

Кожен каталог має містити повні файли програми для відповідної платформи.

### Використання режиму попереднього перегляду

Режим попереднього перегляду не активує жодну гілку, тому він підходить для тестування та перевірки:

```yaml
if [ "$STEAM_PREVIEW_INPUT" = 'true' ]; then
  cmd+=(--preview)
fi
```

Це дає вам змогу спочатку завантажити в Steam для перевірки, а потім перейти до офіційної гілки, коли все буде перевірено.

### Обробка помилок і журналювання

Сценарії включають повну обробку помилок і журналювання:
- Перевірте наявність випуску GitHub.
- Перевірте потрібні файли метаданих.
- Переконайтеся, що вміст платформи присутній.
- Створіть підсумковий звіт GitHub Actions.

Ця інформація дуже цінна як для налагодження, так і для аудиту.

### Керування артефактами

Робочий процес створює два види артефактів:
- `portable-steam-release-preparation-{tag}`: метадані підготовки випуску
- `portable-steam-build-metadata-{tag}`: метадані збірки Steam

Ці артефакти можна використовувати для подальшого аудиту та налагодження. Період зберігання 30 днів є практичним стандартом.

## Використання в реальному світі

У проекті HagiCode цей автоматизований робочий процес випуску вже успішно запущено в кількох версіях. Весь шлях від випуску GitHub до платформи Steam повністю автоматизований і не вимагає ручного втручання.

Це значно підвищило як ефективність випуску, так і надійність. Раніше публікація однієї версії вручну займала більше 30 хвилин. Тепер весь процес завершується всього за кілька хвилин.

Що ще важливіше, автоматизований робочий процес зменшує ймовірність людської помилки. Кожен випуск виконується за тим самим стандартизованим процесом, що робить результати більш передбачуваними.
