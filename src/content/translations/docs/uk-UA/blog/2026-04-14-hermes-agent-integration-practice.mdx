## Практика інтеграції Hermes Agent: від протоколу до виробництва

> Ділиться повним досвідом HagiCode щодо інтеграції Hermes Agent, включаючи основні уроки щодо адаптації протоколу ACP, керування сеансовим пулом і синхронізацію контрактів на передньому та внутрішньому рівнях.

## Фон

Під час створення HagiCode, платформи кодування за допомогою штучного інтелекту, нашій команді потрібно було інтегрувати фреймворк агента, який міг би працювати локально, а також масштабуватися до хмари. Після дослідження Hermes Agent від Nous Research було обрано базовим двигуном для наших можливостей агента загального призначення.

Насправді вибір технології не є ні особливо складним, ні особливо легким. На ринку є багато потужних фреймворків агентів, але Hermes виділявся тим, що його протокол ACP і система інструментів особливо добре відповідають високим вимогам HagiCode: локальна розробка, командна співпраця та хмарне розширення. Тим не менш, впровадження Hermes у реальну виробничу систему означало вирішення довгого списку інженерних проблем. Ця частина була зовсім не тривіальною.

Стек HagiCode використовує Orleans для побудови розподіленої системи, тоді як інтерфейс створено за допомогою React + TypeScript. Інтеграція Hermes означала збереження архітектурної послідовності, роблячи Hermes першокласним виконавцем поряд із ClaudeCode та OpenCode. Звучить досить просто, але реалізація завжди розповідає реальну історію.

Ця стаття ділиться нашим практичним досвідом інтеграції Hermes Agent у HagiCode, і ми сподіваємося, що вона запропонує корисний довідковий матеріал для команд, які стикаються з подібними потребами. Зрештою, коли ви впали в яму, немає причин дозволяти комусь іншому впасти в ту ж саму яму.

## Архітектурний дизайн

### Багаторівневий підхід до проектування

Інтеграція Hermes від HagiCode використовує чітку багаторівневу архітектуру, де кожен рівень зосереджений на власних обов’язках:

**Внутрішній базовий рівень**
- `HermesCliProvider`: реалізує `IAIProvider` інтерфейс як уніфікована точка входу постачальника AI
- `HermesPlatformConfiguration`: керує шляхом до виконуваного файлу Hermes, аргументами, автентифікацією та пов’язаними налаштуваннями
- `ICliProvider<HermesOptions>`: низькорівнева абстракція CLI, надана HagiCode.Libs для обробки життєвих циклів підпроцесів

**Транспортний шар**
- `StdioAcpTransport`: спілкується з підпроцесом Hermes ACP через стандартний вхід і вихід
- Методи протоколу ACP: `initialize`, `authenticate`, `session/new`, `session/prompt`

**Рівень виконання**
- `HermesGrain`: реалізація Orleans Grain, яка обробляє розподілене виконання сеансу
- `CliAcpSessionPool`: пул сеансів, який повторно використовує підпроцеси ACP, щоб уникнути частого запуску

**Перехідний рівень**
- `ExecutorAvatar`: Візуальний стиль і значок Hermes
- `executorTypeAdapter`: Логіка зіставлення типу постачальника
- Обмін повідомленнями в режимі реального часу SignalR: підтримує послідовність ідентичності Hermes у всьому потоці повідомлень

Ця багатошарова конструкція дозволяє кожному шару розвиватися незалежно. Наприклад, якщо ми захочемо в майбутньому додати новий транспортний механізм, такий як WebSocket, нам потрібно лише змінити транспортний рівень. Немає необхідності перевертати всю систему тільки тому, що змінюється один транспорт.

### Уніфікована абстракція інтерфейсу

Усі постачальники AI реалізують `IAIProvider` інтерфейс, який є одним із основних варіантів дизайну в архітектурі HagiCode:

```csharp
public interface IAIProvider
{
    string Name { get; }
    ProviderCapabilities Capabilities { get; }

    IAsyncEnumerable<AIStreamingChunk> StreamAsync(
        AIRequest request,
        CancellationToken cancellationToken = default);

    Task<AIResponse> ExecuteAsync(
        AIRequest request,
        CancellationToken cancellationToken = default);
}
```

`HermesCliProvider` реалізує цей інтерфейс і стоїть на рівних з ним `ClaudeCodeProvider`, `OpenCodeProvider`, та інші. До переваг такої конструкції можна віднести:

1. **Можливість заміни**: зміна постачальників не впливає на бізнес-логіку верхнього рівня
2. **Тестованість**: постачальників можна легко висміяти для модульного тестування
3. **Розширюваність**: для додавання нового постачальника потрібно лише реалізувати інтерфейс

Зрештою, інтерфейси дуже схожі на правила. Коли правила встановлені, кожен може гармонійно співіснувати, грати на користь своїх сильних сторін і не наступати один на одного. У цьому є якась елегантність.

## Основна реалізація

### Реалізація рівня провайдера

`HermesCliProvider` є ядром усієї інтеграції. Він координує різні компоненти, необхідні для завершення одного виклику AI:

```csharp
public sealed class HermesCliProvider : IAIProvider, IVersionedAIProvider
{
    private readonly ICliProvider<LibsHermesOptions> _provider;
    private readonly ConcurrentDictionary<string, string> _sessionBindings;

    public ProviderCapabilities Capabilities { get; } = new()
    {
        SupportsStreaming = true,
        SupportsTools = true,
        SupportsSystemMessages = true,
        SupportsArtifacts = false
    };

    public async IAsyncEnumerable<AIStreamingChunk> StreamAsync(
        AIRequest request,
        [EnumeratorCancellation] CancellationToken cancellationToken = default)
    {
        // 1. Resolve the session binding key
        var bindingKey = ResolveBindingKey(request.CessionId);

        // 2. Get or create a Hermes session through the session pool
        var options = new HermesOptions
        {
            ExecutablePath = _platformConfiguration.ExecutablePath,
            Arguments = _platformConfiguration.Arguments,
            SessionId = _sessionBindings.TryGetValue(bindingKey, out var sessionId) ? sessionId : null,
            WorkingDirectory = request.WorkingDirectory,
            Model = request.Model
        };

        // 3. Execute and collect the streaming response
        await foreach (var message in _provider.ExecuteAsync(options, request.Prompt, cancellationToken))
        {
            // 4. Map ACP messages to AIStreamingChunk
            if (_responseMapper.TryConvertToStreamingChunk(message, out var chunk))
            {
                yield return chunk;
            }
        }
    }
}
```

Тут особливо важливі кілька моментів дизайну:

1. **Прив’язка до сеансу**: використовує `CessionId` прив’язувати кілька запитів до одного підпроцесу Hermes, зберігаючи безперервність контексту в багаточергових розмовах
2. **Відображення відповіді**: перетворює формат повідомлення Hermes ACP в уніфікований `AIStreamingChunk` формат
3. **Підтримка потокового передавання**: використання `IAsyncEnumerable` для підтримки справжніх потокових відповідей

Прив’язка сеансу трохи нагадує людські стосунки. Після встановлення з’єднання майбутнє спілкування має контекст, тому вам не потрібно щоразу починати з нуля. Звичайно, ці стосунки все одно потрібно підтримувати.

### Адаптація протоколу ACP

Hermes використовує ACP (Agent Communication Protocol), який відрізняється від традиційного HTTP API. ACP — це протокол, заснований на стандартному введенні та виведенні, і він має кілька характеристик:

1. **Маркер запуску**: після запуску процесу Hermes він виводить `//ready` маркер
2. **Динамічна автентифікація**: методи автентифікації не є фіксованими, і їх потрібно обговорювати через протокол
3. **Повторне використання сеансу**: встановлені сеанси використовуються повторно `SessionId`
4. **Фрагментовані відповіді**: повна відповідь може бути розділена на кілька `session/update` сповіщення

HagiCode обробляє ці характеристики `StdioAcpTransport`:

```csharp
public class StdioAcpTransport
{
    public async Task InitializeAsync(CancellationToken cancellationToken)
    {
        // Wait for the //ready marker
        var readyLine = await _outputReader.ReadLineAsync(cancellationToken);
        if (readyLine != "//ready")
        {
            throw new InvalidOperationException("Hermes did not send ready signal");
        }

        // Send the initialize request
        await SendRequestAsync(new
        {
            jsonrpc = "2.0",
            id = 1,
            method = "initialize",
            @params = new
            {
                protocolVersion = "2024-11-05",
                capabilities = new { },
                clientInfo = new { name = "HagiCode", version = "1.0.0" }
            }
        }, cancellationToken);
    }
}
```

Протоколи трохи схожі на взаєморозуміння між людьми. Коли це розуміння є, спілкування протікає набагато плавніше. Для його створення потрібен час.

### Керування пулом сеансів

Частий запуск підпроцесів Hermes коштує дорого, тому ми реалізували механізм пулу сеансів:

```csharp
services.AddSingleton(static _ =>
{
    var registry = new CliProviderPoolConfigurationRegistry();
    registry.Register("hermes", new CliPoolSettings
    {
        MaxActiveSessions = 50,
        IdleTimeout = TimeSpan.FromMinutes(10)
    });
    return registry;
});
```

Ключові параметри пулу сеансів:
- `MaxActiveSessions`: контролює обмеження паралельності, щоб уникнути виснаження ресурсів
- `IdleTimeout`: тайм-аут простою, який балансує вартість запуску та використання пам’яті

На практиці ми виявили, що:
1. Якщо час простою занадто короткий, сеанси часто перезапускаються; якщо він надто довгий, пам'ять залишається зайнятою
2. Обмеження паралельності має бути налаштовано відповідно до фактичного навантаження, тому що встановлення занадто високого рівня може призвести до млявості системи
3. Використання пулу сеансів потребує моніторингу, щоб можна було вчасно налаштувати параметри

Це схоже на багато інших варіантів у житті: надто агресивна поведінка створює проблеми, а надто консервативна втрачає можливості. Мета – просто знайти правильний баланс.

### Синхронізація договору

Інтерфейс і бек-енд підтримують узгодженість своїх контрактів через створення OpenAPI. Задня частина визначає `AIProviderType` enum:

```csharp
public enum AIProviderType
{
    Unknown,
    ClaudeCode,
    OpenCode,
    HermesCli  // Newly added
}
```

Інтерфейс генерує відповідний тип TypeScript через OpenAPI, забезпечуючи узгодженість значень enum. Це ключ до уникнення зовнішнього відображення `Unknown`.

Контракт дуже схожий на обіцянку. Після того, як ви домовилися, це потрібно шанувати, інакше ви потрапите в незручні ситуації, як-от `Unknown`.

## Управління конфігурацією

Конфігурація Hermes управляється через `appsettings.json`:

```json
{
  "Providers": {
    "HermesCli": {
      "ExecutablePath": "hermes",
      "Arguments": "acp",
      "StartupTimeoutMs": 10000,
      "ClientName": "HagiCode",
      "Authentication": {
        "PreferredMethodId": "api-key",
        "MethodInfo": {
          "api-key": "your-api-key-here"
        }
      },
      "SessionDefaults": {
        "Model": "claude-sonnet-4-20250514",
        "ModeId": "default"
      }
    }
  }
}
```

Ця конструкція, керована конфігурацією, забезпечує гнучкість:
- Виконувані шляхи можна перевизначати, що зручно для розробки та тестування
- Аргументи запуску можна налаштувати відповідно до різних версій Hermes
- інформацію про автентифікацію можна налаштувати для підтримки кількох методів автентифікації

Конфігурація трохи схожа на питання з кількома варіантами відповіді в житті. Якщо доступно достатньо варіантів, зазвичай є той, який підходить. Тим не менш, занадто багато варіантів можуть самі по собі викликати втому від прийняття рішень.

## Практичний досвід

### Перевірки стану здоров'я

Створення надійного постачальника вимагає комплексних перевірок стану здоров’я:

```csharp
public async Task<ProviderTestResult> PingAsync(CancellationToken cancellationToken = default)
{
    var response = await ExecuteAsync(new AIRequest
    {
        Prompt = "Reply with exactly PONG.",
        CessionId = null,
        AllowedTools = Array.Empty<string>(),
        WorkingDirectory = ResolveWorkingDirectory(null)
    }, cancellationToken);

    var success = string.Equals(response.Content.Trim(), "PONG", StringComparison.OrdinalIgnoreCase);
    return new ProviderTestResult
    {
        ProviderName = Name,
        Success = success,
        ResponseTimeMs = stopwatch.ElapsedMilliseconds,
        ErrorMessage = success ? null : $"Unexpected Hermes ping response: '{response.Content}'."
    };
}
```

На що слід звернути увагу під час перевірки стану здоров’я:
1. Використовуйте прості тести та уникайте надто складних сценаріїв
2. Встановіть розумні значення часу очікування
3. Записуйте час відповіді для підтримки аналізу продуктивності

Подібно до того, як люди потребують фізичних оглядів, системи також потребують перевірки стану здоров’я. Чим швидше виявлені проблеми, тим легше їх виправити.

### Інструмент перевірки

HagiCode надає спеціальну консоль для перевірки інтеграції Hermes:

```bash
# Basic validation

# Full suite (including repository analysis)
HagiCode.Libs.Hermes.Console --test-provider-full --repo .

# Custom executable

```

Цей інструмент надзвичайно корисний під час розробки, оскільки він дозволяє нам швидко перевірити, чи правильна інтеграція. Зрештою, ніхто не хоче чекати, доки проблема виявиться, перш ніж згадати про тестування.

### Вирішення типових проблем

**Помилка автентифікації**
- Перевірте чи `Authentication.PreferredMethodId` відповідає методу автентифікації, який фактично підтримує Hermes
- Переконайтеся, що формат інформації автентифікації правильний, наприклад ключ API або маркер носія

**Тайм-аут сеансу**
- Збільшити `StartupTimeoutMs` значення
- Перевірте доступність сервера MCP
- Перегляньте використання системних ресурсів

**Неповна відповідь**
- Забезпечити `session/update` сповіщення та кінцевий результат агрегуються правильно
- Перевірте логіку скасування в обробці потокового передавання
- Перевірте завершення обробки помилок

**Передні дисплеї Невідомо**
- Підтвердьте, що покоління OpenAPI вже включає `HermesCli` значення enum
- Перевірте правильність зіставлення типу
- Очистити кеш браузера та відновити типи

Проблеми будуть завжди. При їх появі головне не панікувати. Відстежуйте причину крок за кроком, і зрештою більшість із них можна вирішити.

### Пропозиції щодо оптимізації продуктивності

1. **Використовуйте пул сеансів**: повторно використовуйте підпроцеси ACP, щоб зменшити витрати на запуск
2. **Встановіть час очікування належним чином**: збалансуйте використання пам’яті та вартість запуску
3. **Повторне використання ідентифікаторів сеансу**: використовуйте те саме `CessionId` для пакетних завдань
4. **Налаштуйте MCP на вимогу**: уникайте непотрібних викликів інструментів

Продуктивність багато в чому схожа на ефективність у повсякденному житті. Коли ви робите це правильно, ви досягаєте більшого з меншими витратами; коли ви помиляєтеся, зусилля збільшуються, а результати зменшуються. Щоб знайти «правильну» точку, потрібні і досвід, і удача.

## Висновок

Інтеграція Hermes Agent у виробничу систему вимагає розгляду проблем у кількох вимірах:

1. **Архітектура**: створіть уніфікований інтерфейс постачальника та реалізуйте архітектуру змінних компонентів
2. **Протокол**: правильно обробляти специфічну для ACP поведінку, таку як маркери запуску та динамічна автентифікація
3. **Продуктивність**: повторно використовуйте ресурси через пул сеансів і збалансуйте вартість запуску з використанням пам’яті
4. **Перехідна частина**: забезпечує синхронізацію контрактів і забезпечує послідовне візуальне враження

Досвід HagiCode показує, що за допомогою якісного багаторівневого дизайну та реалізації, керованої конфігурацією, складну систему агентів можна бездоганно інтегрувати в існуючу архітектуру.

Ці принципи звучать просто, якщо їх описати словами, але фактичне впровадження завжди створює багато різних проблем. Це нормально. Якщо проблема вирішується, це стає досвідом. Якщо цього не відбувається, це стає уроком. У будь-якому випадку, це все ще має цінність.

Красиві речі або люди не потребують володіння; поки вони залишаються красивими, достатньо спокійно цінувати цю красу. Технологія майже така сама. Якщо це допомагає покращити систему, то конкретна структура чи протокол мають набагато менше значення, ніж іноді думають люди.
