## Посібник із впровадження підтримки Hotword для розпізнавання мовлення Doubao

> У цій статті детально пояснюється, як реалізувати підтримку гарячих слів для розпізнавання мовлення Doubao у проекті HagiCode. Використовуючи як користувацькі гарячі слова, так і таблиці гарячих слів платформи, ви можете значно підвищити точність розпізнавання словника, що стосується конкретної області.

## Фон

Технологія розпізнавання мовлення розвивалася багато років, але одна проблема постійно турбує розробників. Моделі розпізнавання мовлення загального призначення можуть охоплювати повсякденну мову, але вони часто не враховують професійну термінологію, назви продуктів та особисті імена. Подумайте про це: голосовий помічник у сфері медицини повинен точно розпізнавати такі терміни, як «гіпертонія», «діабет» та «ішемічна хвороба серця»; правовій системі необхідно точно охоплювати такі терміни, як «підстава для позову», «захист» і «тягар доведення». У цих сценаріях модель загального призначення намагається зробити все можливе, але цього часто недостатньо.

Ми зіткнулися з такою ж проблемою в проекті HagiCode. Будучи багатофункціональним помічником кодування штучного інтелекту, HagiCode має розпізнавати мовлення для широкого спектру технічної термінології. Однак API розпізнавання мовлення Doubao у конфігурації за замовчуванням не міг повністю задовольнити наші вимоги щодо точності спеціальних термінів. Справа не в тому, що Doubao недостатньо хороший; радше, кожен домен має власну термінологічну систему. Після деяких досліджень і технічних досліджень ми виявили, що API розпізнавання мовлення Doubao справді забезпечує підтримку гарячих слів. Завдяки простій конфігурації він може значно підвищити точність розпізнавання певної лексики. У певному сенсі, як тільки ви говорите йому, на які слова звернути увагу, він уважно їх слухає.

Ця стаття ділиться повним рішенням, яке ми використовували в проекті HagiCode для впровадження гарячих слів розпізнавання мовлення Doubao. Доступні обидва режими, спеціальні гарячі слова та таблиці гарячих слів платформи, і їх також можна комбінувати. За допомогою цього рішення розробники можуть гнучко налаштовувати гарячі слова на основі бізнес-сценаріїв, щоб система розпізнавання мовлення могла краще «розпізнавати» професійну, незвичайну, але критичну лексику.

## Основна реалізація

### Розуміння двох режимів гарячих слів

API розпізнавання мовлення Doubao надає два способи налаштування гарячих слів, і кожен із них має свої ідеальні варіанти використання та переваги.

**Користувацький режим гарячого слова** дозволяє передавати текст гарячого слова безпосередньо через `corpus.context` поле. Цей підхід особливо підходить для сценаріїв, коли потрібно швидко налаштувати невелику кількість гарячих слів, наприклад тимчасове розпізнавання назви продукту або імені людини. У реалізації HagiCode ми аналізуємо багаторядковий текст гарячого слова, введений користувачем, у список рядків, а потім форматуємо його у `context_data` масив, необхідний API Doubao. Цей підхід дуже прямий: ви просто вказуєте системі, на які слова звернути увагу, і вона робить саме це.

**Режим таблиці гарячих слів платформи** використовує `corpus.boosting_table_id` поле для посилання на попередньо налаштовану таблицю гарячих слів у платформі самонавчання Doubao. Цей підхід підходить для сценаріїв, коли потрібно керувати великою кількістю гарячих слів. Ми можемо створювати та підтримувати таблиці гарячих слів на платформі самонавчання Doubao, а потім посилатися на них за ідентифікаторами. Для такого проекту, як HagiCode, де спеціалізовані терміни потрібно постійно оновлювати та підтримувати, цей режим пропонує набагато кращу керованість. Коли кількість гарячих слів зросте, мати централізоване місце для керування ними набагато краще, ніж щоразу вводити їх вручну.

Цікаво, що ці два режими також можна використовувати разом. API Doubao підтримує включення як користувацьких гарячих слів, так і ідентифікатора таблиці гарячих слів платформи в один запит, при цьому стратегія поєднання контролюється `combine_mode` параметр. Ця гнучкість дозволяє HagiCode обробляти широкий спектр складних потреб у розпізнаванні професійної термінології. Іноді поєднання кількох підходів дає кращі результати.

### Визначення типів зовнішнього інтерфейсу та перевірка

У реалізації інтерфейсу HagiCode ми визначили повний набір типів конфігурації гарячих слів і логіку перевірки. Перша частина — це визначення типу:

```typescript
export interface HotwordConfig {
  contextText: string;           // Multi-line hotword text
  boostingTableId: string;      // Doubao platform hotword table ID
  combineMode: boolean;          // Whether to use both together
}
```

Цей простий інтерфейс містить усі елементи конфігурації для функції гарячого слова. Серед них `contextText` це частина, з якою користувачі взаємодіють найбільш безпосередньо: ми дозволяємо користувачам вводити одну фразу гарячого слова на рядок, що дуже інтуїтивно зрозуміло. Попросити користувачів ввести один термін на рядок набагато легше, ніж змусити їх зрозуміти складний формат конфігурації.

Далі йде функція перевірки. Базуючись на обмеженнях API Doubao, ми визначили суворі правила перевірки: щонайбільше 100 рядків тексту гарячого слова, до 50 символів у рядку та не більше 5000 символів загалом; `boosting_table_id` може містити не більше 200 символів і може містити лише літери, цифри, підкреслення та дефіси. Ці обмеження не є довільними; вони походять безпосередньо з офіційної документації Doubao. Обмеження API є обмеженнями API, і ми маємо їх дотримуватися.

```typescript
export function validateContextText(contextText: string): HotwordValidationResult {
  if (!contextText || contextText.trim().length === 0) {
    return { isValid: true, errors: [] };
  }

  const lines = contextText.split('\n').filter(line => line.trim().length > 0);
  const errors: string[] = [];

  if (lines.length > 100) {
    errors.push(`Hotword line count cannot exceed 100 lines; current count is ${lines.length}`);
  }

  const totalChars = contextText.length;
  if (totalChars > 5000) {
    errors.push(`Total hotword character count cannot exceed 5000; current count is ${totalChars}`);
  }

  for (let i = 0; i < lines.length; i++) {
    if (lines[i].length > 50) {
      errors.push(`Hotword on line ${i + 1} exceeds the 50-character limit`);
    }
  }

  return { isValid: errors.length === 0, errors };
}

export function validateBoostingTableId(boostingTableId: string): HotwordValidationResult {
  if (!boostingTableId || boostingTableId.trim().length === 0) {
    return { isValid: true, errors: [] };
  }

  const errors: string[] = [];

  if (boostingTableId.length > 200) {
    errors.push(`boosting_table_id cannot exceed 200 characters; current count is ${boostingTableId.length}`);
  }

  if (!/^[a-zA-Z0-9_-]+$/.test(boostingTableId)) {
    errors.push('boosting_table_id can contain only letters, numbers, underscores, and hyphens');
  }

  return { isValid: errors.length === 0, errors };
}
```

Ці функції перевірки запускаються негайно, коли користувач налаштовує гарячі слова, гарантуючи, що проблеми виявляються якомога раніше. З точки зору взаємодії з користувачем, такий миттєвий зворотний зв’язок дуже важливий. Користувачам завжди краще знати, що не так, під час введення тексту, а не після надсилання.

### Постійність конфігурації зовнішнього інтерфейсу

У реалізації інтерфейсу HagiCode ми вирішили використовувати веб-переглядач `localStorage` для збереження конфігурації гарячого слова. За цим дизайнерським рішенням стояло кілька міркувань. По-перше, конфігурація гарячого слова дуже персоналізована, і різні користувачі можуть мати різні потреби в домені. По-друге, цей підхід спрощує реалізацію серверної частини, оскільки не вимагає додаткових таблиць бази даних або кінцевих точок API. Нарешті, після того, як користувачі налаштують його один раз у браузері, параметри можуть автоматично завантажуватися під час наступного використання, що дуже зручно. Простіше кажучи, це найпростіший підхід.

```typescript
const HOTWORD_STORAGE_KEYS = {
  contextText: 'hotword-context-text',
  boostingTableId: 'hotword-boosting-table-id',
  combineMode: 'hotword-combine-mode',
} as const;

export const DEFAULT_HOTWORD_CONFIG: HotwordConfig = {
  contextText: '',
  boostingTableId: '',
  combineMode: false,
};

// Load hotword configuration
export function loadHotwordConfig(): HotwordConfig {
  const contextText = localStorage.getItem(HOTWORD_STORAGE_KEYS.contextText) || '';
  const boostingTableId = localStorage.getItem(HOTWORD_STORAGE_KEYS.boostingTableId) || '';
  const combineMode = localStorage.getItem(HOTWORD_STORAGE_KEYS.combineMode) === 'true';

  return { contextText, boostingTableId, combineMode };
}

// Save hotword configuration
export function saveHotwordConfig(config: HotwordConfig): void {
  localStorage.setItem(HOTWORD_STORAGE_KEYS.contextText, config.contextText);
  localStorage.setItem(HOTWORD_STORAGE_KEYS.boostingTableId, config.boostingTableId);
  localStorage.setItem(HOTWORD_STORAGE_KEYS.combineMode, String(config.combineMode));
}
```

Логіка в цьому коді проста і зрозуміла. Читаємо з `localStorage` під час завантаження конфігурації та напишіть `localStorage` при його збереженні. Ми також надаємо конфігурацію за замовчуванням, щоб система все ще могла працювати належним чином, навіть якщо конфігурації ще немає. Зрештою, має бути розумний дефолт.

### Backend SDK Configuration Extension

У базовій реалізації HagiCode нам потрібно було додати властивості, пов’язані з гарячими словами, до класу конфігурації SDK. Беручи до уваги характеристики мови C# та моделі використання, ми використовували `List<string>` щоб зберегти спеціальні контексти гарячих слів:

```csharp
public class DoubaoVoiceConfig
{
    /// <summary>
    /// App ID
    /// </summary>
    public string AppId { get; set; } = string.Empty;

    /// <summary>
    /// Access token
    /// </summary>
    public string AccessToken { get; set; } = string.Empty;

    /// <summary>
    /// Service URL
    /// </summary>
    public string ServiceUrl { get; set; } = string.Empty;

    /// <summary>
    /// Custom hotword context list
    /// </summary>
    public List<string>? HotwordContexts { get; set; }

    /// <summary>
    /// Doubao platform hotword table ID
    /// </summary>
    public string? BoostingTableId { get; set; }
}
```

Дизайн цього класу конфігурації відповідає звичайному лаконічному стилю HagiCode. `HotwordContexts` є типом списку, що допускає значення NULL, і `BoostingTableId` є рядком із значенням nullable, тому, коли немає конфігурації гарячого слова, ці властивості взагалі не впливають на запит. Якщо ви не використовуєте цю функцію, вона не повинна заважати.

### Логіка побудови корисного навантаження

Побудова корисного навантаження є ядром усієї функції гарячого слова. Коли у нас є конфігурація гарячого слова, нам потрібно відформатувати його у структуру JSON, необхідну для API Doubao. Цей процес відбувається до того, як SDK надішле запит:

```csharp
private void AddCorpusToRequest(Dictionary<string, object> request)
{
    var corpus = new Dictionary<string, object>();

    // Add custom hotwords
    if (Config.HotwordContexts != null && Config.HotwordContexts.Count > 0)
    {
        corpus["context"] = new Dictionary<string, object>
        {
            ["context_type"] = "dialog_ctx",
            ["context_data"] = Config.HotwordContexts
                .Select(text => new Dictionary<string, object> { ["text"] = text })
                .ToList()
        };
    }

    // Add platform hotword table ID
    if (!string.IsNullOrEmpty(Config.BoostingTableId))
    {
        corpus["boosting_table_id"] = Config.BoostingTableId;
    }

    // Add corpus to the request only when it is not empty
    if (corpus.Count > 0)
    {
        request["corpus"] = corpus;
    }
}
```

Цей код показує, як динамічно створювати `corpus` поле на основі конфігурації. Ключовим моментом є те, що ми додаємо `corpus` лише тоді, коли фактично існує конфігурація гарячого слова. Така конструкція забезпечує зворотну сумісність: коли гарячі слова не налаштовано, структура запиту залишається такою ж, як і раніше. Зворотна сумісність має значення; додавання функції не повинно порушувати існуючу логіку.

### Передача параметрів WebSocket

Між інтерфейсом і сервером параметри гарячих слів передаються через контрольні повідомлення WebSocket. HagiCode розроблено таким чином, що коли зовнішній інтерфейс починає запис, він завантажує конфігурацію гарячого слова з `localStorage` і надсилає його серверній частині через повідомлення WebSocket.

```typescript
const controlMessage = {
  type: 'control',
  payload: {
    command: 'StartRecognition',
    contextText: '高血压\n糖尿病\n冠心病',
    boosting_table_id: 'medical_table',
    combineMode: false
  }
};
```

Тут слід звернути увагу на одну деталь: інтерфейс передає багаторядковий текст, розділений символами нового рядка, і бекенд повинен його розібрати. Серверний обробник WebSocket аналізує ці параметри та передає їх до SDK:

```csharp
private async Task HandleControlMessageAsync(
    string connectionId,
    DoubaoSession session,
    ControlMessage message)
{
    if (message.Payload is SessionControlRequest controlRequest)
    {
        // Parse hotword parameters
        string? contextText = controlRequest.ContextText;
        string? boostingTableId = controlRequest.BoostingTableId;
        bool? combineMode = controlRequest.CombineMode;

        // Parse multi-line text into a hotword list
        if (!string.IsNullOrEmpty(contextText))
        {
            var hotwords = contextText
                .Split('\n', StringSplitOptions.RemoveEmptyEntries)
                .Select(s => s.Trim())
                .Where(s => s.Length > 0)
                .ToList();

            session.HotwordContexts = hotwords;
        }

        session.BoostingTableId = boostingTableId;
    }
}
```

Завдяки такому дизайну передача конфігурації гарячих слів із інтерфейсу на сервер стає зрозумілою та ефективною. У цьому немає нічого особливо загадкового; дані просто передаються шар за шаром.

## Практичний посібник

### Налаштувати спеціальні гарячі слова

У реальному використанні налаштувати спеціальні гарячі слова дуже просто. Відкрийте сторінку налаштувань розпізнавання мовлення в HagiCode та знайдіть розділ «Конфігурація гарячого слова». У полі введення «Користувацький текст гарячого слова» введіть одну фразу гарячого слова на рядок.

Наприклад, якщо ви розробляєте програму, пов’язану з медициною, ви можете налаштувати її так:

```
高血压
糖尿病
冠心病
心绞痛
心肌梗死
心力衰竭
```

Після збереження конфігурації ці гарячі слова автоматично передаються в API Doubao кожного разу, коли починається розпізнавання мовлення. У наших тестах після налаштування гарячих слів точність розпізнавання пов’язаних професійних термінів помітно покращилася. Поліпшення реальне, і явно краще, ніж раніше.

### Налаштуйте таблицю гарячих слів платформи

Якщо вам потрібно керувати великою кількістю гарячих слів або якщо гарячі слова потребують частого оновлення, краще підійде режим таблиці гарячих слів платформи. Спочатку створіть таблицю гарячих слів на платформі самонавчання Doubao та отримайте згенерований `boosting_table_id`, а потім введіть цей ідентифікатор на сторінці налаштувань HagiCode.

Платформа самонавчання Doubao надає такі можливості, як масовий імпорт і категорізоване керування гарячими словами, що дуже зручно для команд, яким потрібно керувати великими наборами спеціалізованої термінології. Керуючи гарячими словами на платформі, ви можете підтримувати їх централізовано та постійно розгортати оновлення. Коли список гарячих слів стає великим, мати єдине місце для керування ним набагато практичніше, ніж щоразу вводити його вручну.

### Використання комбінованого режиму

У деяких складних сценаріях вам може знадобитися одночасно використовувати спеціальні гарячі слова та таблицю гарячих слів платформи. У такому випадку просто налаштуйте обидва в HagiCode та ввімкніть перемикач «Комбінований режим».

У комбінованому режимі API Doubao розглядає обидва джерела гарячих слів одночасно, тому точність розпізнавання зазвичай вища, ніж використання одного джерела окремо. Однак варто зазначити, що комбінований режим збільшує складність запиту, тому найкраще вирішити, чи вмикати його після практичного тестування. Більша складність того варта, лише якщо реальні результати виправдовують це.

### Приклади інтеграції коду

Інтегрувати функцію гарячого слова в проект HagiCode дуже просто. Ось деякі часто використовувані фрагменти коду:

```typescript
import {
  loadHotwordConfig,
  saveHotwordConfig,
  validateHotwordConfig,
  parseContextText,
  getEffectiveHotwordMode,
  type HotwordConfig
} from '@/types/hotword';

// Load and validate configuration
const config = loadHotwordConfig();
const validation = validateHotwordConfig(config);

if (!validation.isValid) {
  console.error('Hotword configuration validation failed:', validation.errors);
  return;
}

// Parse hotword text
const hotwords = parseContextText(config.contextText);
console.log('Parsed hotwords:', hotwords);

// Get effective hotword mode
const mode = getEffectiveHotwordMode(config);
console.log('Current hotword mode:', mode);
```

Використання бекенда так само стисле:

```csharp
var config = new DoubaoVoiceConfig
{
    AppId = "your_app_id",
    AccessToken = "your_access_token",
    ServiceUrl = "wss://openspeech.bytedance.com/api/v3/sauc/bigmodel_async",

    // Configure custom hotwords
    HotwordContexts = new List<string>
    {
        "高血压",
        "糖尿病",
        "冠心病"
    },

    // Configure platform hotword table
    BoostingTableId = "medical_table_v1"
};

var client = new DoubaoVoiceClient(config, logger);
await client.ConnectAsync();
await client.SendFullClientRequest();
```

## Про що слід пам’ятати

Є кілька моментів, які заслуговують на особливу увагу під час впровадження та використання функції гарячого слова.

По-перше, це обмеження кількості символів. API Doubao має суворі обмеження щодо гарячих слів, включаючи кількість рядків, кількість символів у рядку та загальну кількість символів. Якщо будь-який ліміт перевищено, API повертає помилку. У реалізації інтерфейсу HagiCode ми перевіряємо ці обмеження під час введення користувачем за допомогою функцій перевірки, що запобігає надсиланню недійсних конфігурацій на сервер. Виявляти проблеми на ранній стадії завжди краще, ніж чекати, поки API вийде з ладу.

По-друге, це формат `boosting_table_id`. У цьому полі допускаються лише літери, цифри, знаки підкреслення та дефіси, воно не може містити пробіли чи інші спеціальні символи. Створюючи таблицю гарячих слів на платформі для самостійного навчання Doubao, обов’язково дотримуйтеся правил іменування. Така сувора перевірка формату є звичайною для API.

По-третє, це зворотна сумісність. Параметри гарячого слова є необов’язковими. Якщо гарячі слова не налаштовано, система поводиться так само, як і раніше. Цей дизайн гарантує, що існуючі користувачі жодним чином не вплинуть, а також полегшує поступову міграцію та оновлення. Додавання функції не повинно порушувати попередню логіку.

Нарешті, є обробка помилок. Якщо конфігурація гарячого слова недійсна, Doubao API повертає відповідні повідомлення про помилку. Реалізація HagiCode записує детальні журнали, щоб допомогти розробникам усунути проблеми. У той же час інтерфейс відображає помилки перевірки в інтерфейсі користувача, щоб допомогти користувачам виправити конфігурацію. Хороша обробка помилок природно веде до кращої взаємодії з користувачем.

## Висновок

У цій статті ми надали детальний вступ до повного рішення для впровадження гарячих слів розпізнавання мовлення Doubao у проекті HagiCode. Це рішення охоплює весь процес від аналізу вимог і технічного вибору до впровадження коду, надаючи розробникам практичний приклад, який вони можуть використовувати для довідки.

Ключові моменти можна підсумувати таким чином. По-перше, Doubao API підтримує як спеціальні гарячі слова, так і таблиці гарячих слів платформи, і їх можна використовувати окремо або в комбінації. По-друге, інтерфейс використовує `localStorage` зберігати конфігурацію простим і ефективним способом. По-третє, серверна частина передає параметри гарячого слова шляхом динамічного створення `corpus` поля, зберігаючи сильну зворотну сумісність. По-четверте, комплексна логіка перевірки забезпечує правильність конфігурації та запобігає недійсним запитам. Загалом, рішення не є складним; він просто ретельно дотримується вимог API.

Впровадження функції гарячого слова ще більше посилює можливості HagiCode у сфері розпізнавання мовлення. Гнучко налаштовуючи пов’язані з бізнесом професійні терміни, розробники можуть допомогти системі розпізнавання мовлення краще розуміти вміст із певних доменів і, отже, надавати точніші послуги. Зрештою, технології мають задовольняти реальні потреби бізнесу, а вирішення практичних проблем є найважливішим.
