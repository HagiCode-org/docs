Під час розробки HagiCode ми зіткнулися з дуже практичною проблемою: оскільки наша документація та пропозиції постійно зростали, ефективне управління та представлення цього вмісту ставало все більш актуальним. Ми вирішили використовувати сторінки GitHub для розміщення нашого статичного сайту, але створення та розгортання його вручну було надто громіздким. Кожна зміна вимагала локальної збірки, упаковки, а потім ручного натискання на `gh-pages` відділення. Це було не тільки неефективно, але й легко помилитися.

Щоб вирішити цю проблему, головним чином тому, що ми хотіли заощадити свої зусилля, нам потрібен був автоматизований робочий процес розгортання. У цій статті детально описано, як ми додали підтримку автоматичного розгортання на основі GitHub Actions до проекту HagiCode, щоб ми могли зосередитися на написанні вмісту та дозволити автоматизації впоратися з іншим.

## Аналіз цілей

Перш ніж перейти до впровадження, нам спочатку потрібно з’ясувати, що насправді має виконувати це завдання. Адже заточення сокири не затримує рубку дров.

### Основні вимоги

1.  **Автоматизоване збирання**: автоматично запускає робочий процес збирання, коли код надсилається до `main` відділення.
2.  **Автоматичне розгортання**: після успішної збірки автоматично розгортайте згенеровані статичні файли на сторінках GitHub.
3.  **Узгодженість середовища**: переконайтеся, що середовище CI відповідає локальному середовищу збірки, щоб уникнути класичного збентеження «воно працює локально, але не працює в Інтернеті».

### Технічний вибір

Оскільки HagiCode побудовано на Docusaurus, дуже популярному генераторі статичних сайтів React, ми можемо використовувати GitHub Actions для досягнення цієї мети.

## Налаштування робочого циклу дій GitHub

GitHub Actions — це служба CI/CD, яку надає GitHub. Визначаючи файли робочого циклу у форматі YAML у сховищі, ми можемо налаштовувати всі види завдань автоматизації.

### Створення файлу робочого процесу

Нам потрібно створити новий файл конфігурації в `.github/workflows` наприклад, у кореневому каталозі проекту `deploy.yml`. Якщо каталог ще не існує, спочатку створіть його.

Основна логіка цього файлу конфігурації така:

1.  **Умова тригера**: Слухайте `push` події на в `main` відділення.
2.  **Середовище виконання**: останній випуск Ubuntu.
3.  **Кроки створення**:
    -   Перевірте код
    -   Встановіть Node.js
    -   Встановити залежності (`npm install`)
    -   Створіть статичні файли (`npm run build`)
4.  **Крок розгортання**: використовуйте офіційний `action-gh-pages` дія, щоб перемістити вихід збірки на `gh-pages` відділення.

### Код конфігурації ключа

Нижче наведено шаблон конфігурації, який ми остаточно прийняли:

```yaml
name: Deploy to GitHub Pages

# Trigger condition: run when code is pushed to the main branch
on:
  push:
    branches:
      - main
    # Add path filters if needed, for example only build when docs change
    # paths:
    #   - 'docs/**'
    #   - 'package.json'

# Set permissions, which is important for GitHub Pages deployment
permissions:
  contents: read
  pages: write
  id-token: write

# Concurrency control: cancel older builds for the same branch
concurrency:
  group: "pages"
  cancel-in-progress: false

jobs:
  build:
    runs-on: ubuntu-latest
    steps:
      - name: Checkout
        uses: actions/checkout@v4
        # Note: fetch-depth: 0 is required, otherwise the build version may be inaccurate
        with:
          fetch-depth: 0

      - name: Setup Node
        uses: actions/setup-node@v4
        with:
          node-version: 20 # Keep this consistent with the local development environment
          cache: 'npm'     # Enabling cache can speed up the build process

      - name: Install dependencies
        run: npm ci
        # Use npm ci instead of npm install because it is faster and stricter, which makes it better for CI

      - name: Build website
        run: npm run build
        env:
          # Configure environment variables here if your site build requires them
          # NODE_ENV: production
          # PUBLIC_URL: /your-repo-name
          
      - name: Upload artifact
        uses: actions/upload-pages-artifact@v3
        with:
          path: ./build # Default Docusaurus output directory

  deploy:
    environment:
      name: github-pages
      url: ${{ steps.deployment.outputs.page_url }}
    runs-on: ubuntu-latest
    needs: build
    steps:
      - name: Deploy to GitHub Pages
        id: deployment
        uses: actions/deploy-pages@v4
```

## Поширені підводні камені під час впровадження

На практиці ми зіткнулися з кількома проблемами. Я ділюся ними тут, сподіваючись, що інші зможуть уникнути їх або принаймні підготувати рішення заздалегідь.

### 1. Проблеми з дозволом на токен GitHub

Коли ми вперше налаштували щось, розгортання постійно виникало з ладу через помилку 403 (заборонено). Після тривалого копання ми виявили, що GitHub стандартний `GITHUB_TOKEN` не мав дозволу писати на Pages.

**Рішення**: у сховищі `Settings` -> `Actions` -> `General` -> `Workflow permissions`, переконайтеся, що вибрано **"Дозволи на читання та запис"**.

### 2. Неправильний шлях до каталогу збірки

Docusaurus розміщує створені статичні файли в `build` каталог за замовчуванням. Однак деякі проекти можуть бути налаштовані інакше. Наприклад, Create React App за замовчуванням `build`, тоді як Vite за замовчуванням `dist`. Якщо Actions повідомляє, що не може знайти вихідні файли, перевірте вихідний шлях `docusaurus.config.js`.

### 3. Проблеми підшляху

Якщо ваше сховище не є домашньою сторінкою користувача, це означає, що це не так `username.github.io`, але натомість сторінка проекту, наприклад `username.github.io/project-name`, то потрібно налаштувати `baseUrl`.

в `docusaurus.config.js`:

```javascript
module.exports = {
  // ...
  url: 'https://hagicode.com', // Your Hagicode URL
  baseUrl: '/', // Deploy from the root path
  // ...
};
```
Цю деталь легко не помітити. Якщо його налаштовано неправильно, сторінка може завантажуватися як порожній екран, оскільки шляхи активів не можуть бути визначені.

## Перевірка результату

Налаштувавши все та натиснувши код, ми можемо перейти на вкладку **Дії** репозиторію GitHub і подивитися шоу.

Під час виконання робочого процесу ви побачите жовте коло, і коли воно стане зеленим, це означає успіх. Якщо він стане червоним, натисніть і огляньте журнали. У більшості випадків ви можете відстежити проблему там, зазвичай це помилка або неправильна конфігурація шляху.

Коли збірка вдасться, відвідайте `https://<your-username>.github.io/<repository-name>/` і ви побачите щойно розгорнутий сайт.

## Резюме

Представивши GitHub Actions, ми успішно автоматизували розгортання для сайту документації HagiCode. Це не тільки економить час, який раніше витрачався на ручні операції, але, що більш важливо, стандартизує робочий процес випуску. Тепер, незалежно від того, хто з членів команди оновлює документацію, зміни об’єднуються `main`, останній вміст буде опубліковано через кілька хвилин.

**Основні переваги**:
- **Вища ефективність**: ми перейшли від «ручного пакування та ручного завантаження» до «код дорівнює випуску».
- **Менше помилок**: ми виключили ймовірність людської помилки під час ручних операцій.
- **Кращий досвід розробника**: розробники можуть більше зосередитися на якості вмісту, а не відволікатися на нудні етапи розгортання.

Хоча налаштування CI/CD спочатку є дещо складним, особливо з усіма дозволами та деталями шляху, це одноразова інвестиція з великим довгостроковим прибутком. Я наполегливо рекомендую, щоб кожен проект статичного сайту використовував подібний автоматизований робочий процес.
