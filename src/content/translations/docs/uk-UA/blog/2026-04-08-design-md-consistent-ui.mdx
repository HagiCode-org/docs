## Design.md: рішення для узгодженого дизайну зовнішнього інтерфейсу користувача на основі ШІ

> В епоху розробки інтерфейсу за допомогою штучного інтелекту, як ви можете підтримувати узгодженість інтерфейсів користувача, створених штучним інтелектом? У цій статті розповідається про наш практичний досвід створення сайту-галереї дизайну на основі awesome-design-md, а також про те, як створити структурований `design.md` спрямовувати ШІ до стандартизованого дизайну інтерфейсу користувача.

## Фон

Будь-хто, хто використовував штучний інтелект для написання зовнішнього коду, напевно, мав подібний досвід: попросіть штучний інтелект створити ту саму сторінку кілька разів, і кожен результат виходить в іншому стилі. Іноді кути закруглені, іноді гострі. Іноді інтервал становить 8 пікселів, а іноді – 16 пікселів. Навіть та сама кнопка може виглядати по-різному в різних бесідах.

Це не поодинока проблема. Оскільки розробка за допомогою штучного інтелекту стає все більш поширеною, **відсутність узгодженості створеного ШІ зовнішнього інтерфейсу користувача** стала широко поширеною проблемою. Різні помічники штучного інтелекту, різні підказки та навіть один і той самий помічник у різних розмовах можуть створювати кардинально різні дизайни інтерфейсу. Це створює величезні витрати на обслуговування під час ітерації продукту.

Основна причина насправді проста: не існує **авторитетного довідкового документу про дизайн**. Традиційні таблиці стилів CSS можуть підказати розробникам, «як реалізувати» щось, але вони не можуть повністю повідомити, «чому це розроблено таким чином» або «який шаблон дизайну слід використовувати в якому сценарії». Для ШІ чіткий і структурований опис ще важливіший для розуміння умов проектування.

У той же час спільнота з відкритим кодом вже пропонує кілька чудових ресурсів. The [VoltAgent/awesome-design-md](https://github.com/VoltAgent/awesome-design-md) project збирає системну документацію від багатьох відомих компаній. Кожен каталог містить a `README.md`, а `DESIGN.md`і попередній перегляд HTML. Однак усе це розкидано по репозиторію, що ускладнює швидкий перегляд і порівняння.

Отже, чи можемо ми об’єднати ці ресурси в зручну для перегляду галерею дизайну, водночас виділяючи структурований `design.md` для використання ШІ?

Відповідь - так. Далі я розповім про наш підхід.

### Керування вихідним вмістом: як уніфікувати розкидані ресурси дизайну?

Початковий репозиторій awesome-design-md містить велику кількість проектних документів, але нам потрібен був спосіб перенести їх у наш власний проект.

### Статична архітектура сайту: як забезпечити динамічний пошук, залишаючись повністю статичним?

Оскільки це галерея дизайну, пошук є обов’язковим. Але Astro — це генератор статичних сайтів, тож як реалізувати пошук у реальному часі?

**Рішення: острів React + синхронізація параметрів запиту URL-адреси**

```typescript
// src/components/gallery/SearchToolbar.tsx
```

Перевага цього підходу полягає в тому, що він зберігає можливість розгортання статичного сайту, тобто його можна розгорнути на будь-якій статичній службі хостингу, забезпечуючи миттєву фільтрацію.

### Проектна документація: як допомогти штучному інтелекту зрозуміти та дотримуватися стандартів проектування?

Це ядро всього рішення. Нам потрібно створити структурований `design.md` які ШІ може зрозуміти та застосувати.

**Рішення: запозичити структуру ClickHouse `DESIGN.md`**

[ClickHouse `DESIGN.md`](https://github.com/ClickHouse/ClickHouse/blob/master/DESIGN.md) є чудовою довідкою. Він включає:
- Візуальна тема та атмосфера
- Кольорова палітра та ролі
- Правила друкарні
- Стилі компонентів
- Принципи компонування
- Глибина та висота
- Що робити і чого не робити
- Чуйна поведінка
- Підказка агента

Наш підхід такий: **повторне використання структури, переписування вмісту**. Ми зберігаємо структуру розділів ClickHouse `DESIGN.md`, але замінити вміст фактичними маркерами дизайну та умовними умовами компонентів, які використовуються в нашому власному проекті.

## Рішення

Виходячи з аналізу вище, наше рішення складається з чотирьох основних модулів.

### 1. Конвеєр прийому вмісту

Це основа всієї системи, яка відповідає за вилучення та нормалізацію вмісту з вихідних ресурсів.

```typescript
// src/lib/content/awesomeDesignCatalog.ts

export async function scanSourceEntries(): Promise<DesignEntry[]> {
  const designDir = 'vendor/awesome-design-md/design-md';
  const entries: DesignEntry[] = [];

  for (const dir of await fs.readdir(designDir)) {
    const entryPath = path.join(designDir, dir);
    if (await isValidDesignEntry(entryPath)) {
      const entry = await normalizeDesignEntry(entryPath);
      entries.push(entry);
    }
  }

  return entries;
}

async function isValidDesignEntry(dir: string): Promise<boolean> {
  const requiredFiles = ['README.md', 'DESIGN.md'];
  for (const file of requiredFiles) {
    if (!(await fileExists(path.join(dir, file)))) {
      return false;
    }
  }
  return true;
}
```

Інтерфейс галереї складається з трьох основних частин:

### 3. Функціональність пошуку

Функція пошуку базується на фільтрації на стороні клієнта, стан якої зберігається через параметри запиту URL-адреси:

```typescript
// src/components/gallery/SearchToolbar.tsx

function SearchToolbar({ entries }: { entries: DesignEntry[] }) {
  const [query, setQuery] = useState('');
  const [results, setResults] = useState(entries);

  useEffect(() => {
    const params = new URLSearchParams(window.location.search);
    const initialQuery = params.get('q') || '';
    setQuery(initialQuery);
    filterEntries(initialQuery);
  }, []);

  const filterEntries = (searchQuery: string) => {
    const filtered = entries.filter(entry =>
      entry.searchText.toLowerCase().includes(searchQuery.toLowerCase())
    );
    setResults(filtered);
  };

  const handleChange = (e: React.ChangeEvent<HTMLInputElement>) => {
    const value = e.target.value;
    setQuery(value);
    filterEntries(value);

    // Update the URL without triggering a page refresh
    const newUrl = value
      ? `${window.location.pathname}?q=${encodeURIComponent(value)}`
      : window.location.pathname;
    window.history.replaceState({}, '', newUrl);
  };

  return (
    <div className="search-toolbar">
      <input
        type="text"
        value={query}
        onChange={handleChange}
        placeholder="Search design entries..."
      />
      <span className="result-count">{results.length} results</span>
    </div>
  );
}
```

### Примітки

**Безпека**: рендеринг Markdown вимагає фільтрації небезпечного HTML. Вбудовані фільтри рендерера Astro `script` теги за замовчуванням, але вам все одно потрібно стежити за ризиками XSS.

**Продуктивність**: велика кількість попередніх переглядів iframe може вплинути на продуктивність першого малювання. Рекомендується до використання `loading="lazy"` для відкладеного завантаження попереднього перегляду вмісту.

**Ремонтопридатність**: `design.md` має підтримувати синхронізацію з реалізацією коду. Рекомендується додати перевірки CI, щоб переконатися, що змінні CSS залишаються узгодженими між документацією та кодом.

**Доступність**: переконайтеся, що контраст кольорів відповідає стандарту WCAG AA (принаймні 4,5:1).

## Висновок

Неузгодженість створених штучним інтелектом інтерфейсів користувача, з якими HagiCode зіткнувся під час розробки, по суті, була спричинена відсутністю структурованого довідкового документа щодо дизайну. Побудувавши дизайн галереї сайту та створивши стандартизований `design.md`, ми успішно вирішили цю проблему.

Основна цінність цього рішення полягає в:
- **Уніфіковані ресурси**: консолідація розрізненої документації системи проектування
- **Структуровані стандарти**: вираження умов проектування у форматі, зрозумілому ШІ
- **Постійне обслуговування**: постійно оновлюйте вміст `git submodule`

Якщо ви також використовуєте ШІ для розробки інтерфейсу, цей підхід варто спробувати. Створення структурованого `design.md` не тільки покращує узгодженість коду, створеного штучним інтелектом, але й допомагає вашій команді підтримувати уніфіковані стандарти дизайну всередині.
