## Огляд

Monospecs — це підхід HagiCode до керування декількома сховищами. Він відстежує підсховища та керує ними `monospecs.yaml`, і це працює для будь-якого проекту, якому потрібно одне місце для координації кількох незалежних сховищ.

### Основні поняття

**Файл конфігурації**

`monospecs.yaml` є ядром всього рішення управління, що визначає:
- Перелік і розташування підсховищ
- Git URL для кожного сховища
- Відображуване ім'я та значок
- Поведінка інтеграції з OpenSpec

**Відмінності від звичайних сховищ**

| Особливість | Звичайний репозиторій | Керований проект Monospecs |
|----------|-------------------|---------------------------|
| **Розташування OpenSpec** | Кореневі каталоги `openspec/` | Кореневий каталог головного сховища `openspec/` |
| **Обсяг зміни** | Обмежено поточним сховищем | Може включати кілька підрепозиторіїв |
| **Чистота субрепозитарія** | Специфікації змішані з кодом | Специфікації встановлені в основному сховищі, підсховища чистіші |
| **Автоматична фіксація архіву** | Потрібні ручні специфікації | `commit_when_archive: true` автоматично фіксує специфікації в основному сховищі |

### Навіщо використовувати Monospec?

- **Уніфіковане керування**: керуйте всіма підрепозиторіями в одному конфігураційному файлі
- **Підтримка автоматизації**: надає метадані сховища для сценаріїв і інструментів
- **Згідність зі штучним інтелектом**: структурована конфігурація допомагає агенту штучного інтелекту розуміти взаємозв’язки між сховищами
- **Інтеграція OpenSpec**: повна інтеграція з робочим процесом керування змінами
- **Чистіші підсховища**: характеристики зберігаються в основному сховищі, підсховища містять лише фактичний код

### Застосовні сценарії

Спершу використовуйте Monospec у подібних сценаріях:

**1. Спільна розробка з кількома сховищами**
Коли проект розділено на кілька незалежних сховищ:
- Інтерфейсна програма, серверна служба та настільна програма розроблені окремо
- Потрібно координувати версії та залежності між сховищами
- Документація та допоміжні проекти окремо від основного коду

**2. Управління допоміжним проектом**
Використовуйте Monospec для керування:
- Проект документації
- Офіційний сайт
- Інструменти для створення та сценарії
- Кожен репозиторій підтримує власний контроль версій

**3. Розробка функцій крос-сховища**
Функція, яка вимагає змін у кількох сховищах:
- Потрібно відстежувати зміни в яких сховищах
- Автоматично фіксувати специфікації для правильного цільового сховища
- Координуйте оновлення версій і залежності

## Швидкий старт

### Ініціалізація

Виконайте такі кроки, щоб завершити ініціалізацію моноспекцій:

#### 1. Створіть порожній репозиторій git

Створіть порожній `git` у вашому основному сховищі (примітка: це `git`, ні `repos/` - це угода про моноспецифікації):

```bash
mkdir git
cd git
git init
cd ..
```

#### 2. Налаштуйте файл monospecs

Створити `monospecs.yaml` конфігураційний файл у кореневому каталозі головного сховища:

```yaml
version: "1.0"
commit_when_archive: true

repositories:
  - path: "repos/frontend"
    url: "https://github.com/your-org/frontend.git"
    displayName: "Frontend App"
    icon: "🌐"

  - path: "repos/backend"
    url: "https://github.com/your-org/backend.git"
    displayName: "Backend Service"
    icon: "⚙️"
```

**Елементи конфігурації**:
| Елемент конфігурації | Тип | Обов'язковий | опис |
|----------------|------|----------|-------------|
| `version` | рядок | так | Версія файлу конфігурації |
| `commit_when_archive` | логічний | немає | Чи автоматично фіксувати специфікації під час архівування (не фіксувати код підрепо) |
| `repositories` | масив | так | Список конфігурації сховища |
| `repositories[].path` | рядок | так | Шлях до сховища (відносно кореня проекту) |
| `repositories[].url` | рядок | так | URL-адреса сховища Git |
| `repositories[].displayName` | рядок | немає | Відображуване ім'я сховища |
| `repositories[].icon` | рядок | немає | Піктограма сховища (emoji) |

### Додайте підсховища до головного сховища

Додайте всі підсховища до `repos/` каталог. Цей каталог має бути **виключено з контролю версій Git головного сховища**, щоб уникнути відстеження вмісту підсховища:

```plaintext
# Exclude all sub-repositories
repos/
```

#### 3. Додайте батьківський репозиторій до HagiCode

Якщо ви використовуєте настільну програму HagiCode, відкрийте програму та додайте свій головний репозиторій як проект. Натисніть кнопку «Додати проект» або кнопку «+» на головному екрані, виберіть папку основного сховища, і ви побачите всі підсховища та інформацію про їхній статус.

#### 4. Підсховища клонів

Використовуйте `monospecs.yaml` конфігурації для клонування всіх налаштованих підрепозиторіїв у `repos/` каталог:

Це клонує всі підсховища на основі URL-адрес, указаних у файлі конфігурації.

## Структура файлу конфігурації

### Конфігурація кореневого рівня

```yaml
version: "1.0"
commit_when_archive: true

repositories:
  # ... repository configurations
```

### Конфігурація рівня сховища

Кожен підрепозиторій може мати:
- OpenSpec змінити папку (`openspec/`)
- Незалежний `.git/` каталог
- Додатково `AGENTS.md` для конфігурації AI Agent

## Операції з управління репозиторієм

### Додати новий репозиторій

Під час додавання нового підсховища до `monospecs.yaml` `repositories` масив:

```yaml
repositories:
  - path: "repos/new-service"
    url: "https://github.com/HagiCode-org/new-service.git"
    displayName: "New Service"
    icon: "🆕"
```

### Оновити конфігурацію сховища

Коли URL-адреси сховища або метадані змінюються:
1. Редагувати `monospecs.yaml` щоб оновити відповідні записи
2. Переконайтеся, що синтаксис YAML правильний
3. Якщо синхронізація зміниться, оновіть конфігурацію локального сховища вручну

## Підтримка конфігурації AI Agent

### AGENTS.md

The `AGENTS.md` файл надає вказівки щодо репозиторію для агента AI:

**Технічна інформація про стек**
- Інструменти фреймворку та збірки
- Кодові угоди та методи іменування
- Керування версіями та налаштування розгортання

**Специфічні для проекту поведінки**
- Розширення конфігурації та спеціальні вимоги
- Шаблони інтеграції OpenSpec
- Потреби координації між сховищами

### Переваги ШІ

Використання `AGENTS.md` дозволяє агенту AI:
- Зрозумійте зв’язки та залежності сховища
- Створюйте код і рефакторинг у кількох сховищах
- Підтримуйте узгодженість контролю версій і конфігурацій

## Інтеграція OpenSpec

### Як Monospec працює з OpenSpec

Незалежно від того, чи ви керуєте проектом, керованим моноспецифікацією, чи звичайним репозиторієм, ви можете використовувати робочий процес OpenSpec від HagiCode:

**Звичайний репозиторій:**
- OpenSpec в root `openspec/` каталог
- Зміни стосуються лише поточного сховища

**Керований проект Monospecs:**
- OpenSpec читає `monospecs.yaml` конфігурація
- Зміни відстежуються підрепозиторієм
- Специфікації автоматично встановлюються для правильних цілей

### Поведінка архіву

Коли `commit_when_archive: true`:
- Специфікації автоматично закріплюються в основному репозиторії
- Код підрепозиторіїв не закріплено (залишається окремим)
- Спрощує керування специфікаціями - для підсховищ не потрібно вводити специфікації вручну

### Коміти підсховища вручну

Підрепозиторії підтримують незалежний контроль версій:
- Закріпити фактичні зміни коду у своїх власних сховищах
- Специфікації фіксуються окремо (з головного репозиторію під час архівування)

## Найкращі практики

### Правила іменування сховищ

- Використовуйте kebab-case (малі літери та дефіси) для шляхів до сховища
- Використовуйте прості описові відображувані імена китайською мовою
- Використовуйте відповідні емодзі як значки

### Вказівки щодо відображуваного імені

- Зберігайте його лаконічним і описовим
- Використовуйте узгоджені імена в подібних сховищах
- Виберіть піктограми, які представляють призначення сховища

### Обслуговування конфігурації

**Синхронізація конфігурації з фактичним станом**
- Регулярно перевіряйте `monospecs.yaml` відображає фактичну структуру сховища
- Оновлення або видалення записів, коли сховища додаються або видаляються
- Перевірте зміни, щоб переконатися в дійсності конфігурації

**Керування версіями для конфігурації**
- Відстежуйте зміни файлу конфігурації за допомогою git
- Задокументуйте причини основних змін конфігурації
- Позначте версії файлу конфігурації для легкого відкоту

## Усунення несправностей

### Файл конфігурації не знайдено

Якщо `monospecs.yaml` файл не знайдено в корені проекту:
1. Перевірте, чи існує файл у правильному місці
2. Переконайтеся, що ви перебуваєте у правильному робочому каталозі
3. Перевірте назву файлу конфігурації на наявність помилок

### Синтаксичні помилки YAML

Поширені проблеми синтаксису YAML:
- Неправильний відступ (використовуйте пробіли, а не табуляцію)
- Відсутні або додаткові лапки навколо рядків
- Недійсні типи даних (рядки потребують лапок, а числа – ні)
- Відсутні обов’язкові поля (як `path` або `url`)

### Репозиторій не виявлено

Якщо щойно доданий репозиторій не відображається:
1. Перевірте `path` правильно в `monospecs.yaml`
2. Переконайтеся, що репозиторій успішно клоновано
3. Перевірте, чи існує сховище в очікуваному місці

### Помилки клонів

Якщо клонування сховища не вдається:
1. Перевірте підключення до мережі
2. Переконайтеся, що URL-адреса сховища Git правильна
3. Перевірте наявність проблем з автентифікацією
4. Перегляньте стан контейнера Docker
5. Перевірте наявність дискового простору
