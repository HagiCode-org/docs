HagiCode Desktop постачається з ланцюжком інструментів Node, керованим робочим столом. Він не бере на себе глобальне встановлення Node на вашому комп’ютері. Його мета полягає в тому, щоб надати запуск робочого столу та пов’язані з ним потоки стабільної та перевіреної основи виконання.

Прочитайте цей FAQ, якщо бачите `portable-fixed/toolchain` в діагностиці, потрібно знати, які `node` Desktop використовує або дивуєтеся, чому ваша системна версія вузла безпосередньо не визначає поведінку середовища виконання Desktop.

## Чому Desktop об’єднує Node

Desktop має запускати однакові допоміжні потоки в операційних системах, каналах інсталяції та середовищах користувачів. Якби це залежало лише від наявної інсталяції користувача Node, підтримці довелося б одночасно обробляти застарілі версії, відсутність npm, неправильне впорядкування шляхів, забруднені глобальні пакети та політики реєстру, специфічні для організації.

Desktop описує це більш конкретно як ізольоване середовище виконання Node/npm для керованих CLI замість повторного використання загальносистемного середовища. Цей дизайн має такі основні переваги:

- **Узгоджені версії**: Desktop використовує закріплений контракт середовища Node 22 замість вгадування локальної основної версії користувача.
- **Контрольовані потоки запуску**: команди, керовані настільним комп’ютером, можуть віддавати перевагу упакованим `node` і `npm` записи.
- **Чіткі межі конфігурації**: конфігурація, пов’язана з CLI, зберігається окремо від глобальної конфігурації на всьому хості, тому оновлення, перемикання та відновлення з меншою ймовірністю будуть порушені через відмінності системного середовища.
- **Зменшена кількість конфліктів**: пакетні шляхи застосовуються до командних середовищ, створених Desktop, і не замінюють глобальний вузол користувача в його терміналі.
- **Ненав’язлива поведінка хоста**: ця ізоляція дозволяє уникнути запису стану ланцюжка інструментів у наявне середовище Node/npm користувача, а видалення Desktop не порушує Node, npm або конфігурацію, яка вже є на машині.

## Як робочий стіл вибирає шлях вузла

Коли ви запускаєте Desktop у звичайний спосіб, Desktop віддає перевагу ланцюжку інструментів Node, встановленому разом із програмою, замість того, щоб спочатку читати інсталяцію Node у вашій системі. Це зберігає версію Node у вашому терміналі окремо від версії Node, яку очікує Desktop під час роботи.

Коли ввімкнено комплектний інструментальний ланцюжок, командні середовища, створені Desktop, розміщують пакетні інструментальні шляхи на початку `PATH`. У результаті, навіть якщо у вашій системі встановлено іншу версію Node, потоки, керовані настільним комп’ютером, спершу вирішуються до об’єднаного Node і npm Desktop.

Це впливає лише на середовища виконання, створені Desktop. Якщо ви біжите `node`, `npm`або інші команди безпосередньо у вашому системному терміналі, вони продовжуватимуть використовувати звичайне системне середовище та конфігурацію оболонки.

У середовищах діагностики або дочірніх процесів, запущених Desktop, ви можете побачити таку змінну:

```bash
HAGICODE_PORTABLE_TOOLCHAIN_ROOT=/path/to/portable-fixed/toolchain
```

Ця змінна означає, що середовище виконання, кероване настільним комп’ютером, розпізнало кореневий пакет інструментів. Бачення цього шляху не означає, що робочий стіл змінив ваш системний вузол; це лише означає, що Desktop використовує середовище Node, яке постачається разом із програмою, у власному діапазоні виконання.

## Де Desktop встановлює керовані пакети npm

Робочий стіл зберігає файли середовища виконання та змінні файли пакетів npm у різних місцях. Поєднаний шлях ланцюжка інструментів, наприклад `portable-fixed/toolchain/node` є незмінним джерелом для настільного ПК `node` і `npm` команди. Натомість керовані глобальні пакети встановлюються в каталозі даних користувача Electron:

```text
userData/node22/npmGlobal
```

Якщо Desktop пізніше переходить до Node 24, він використовує окреме дерево пакетів, наприклад `userData/node24/npmGlobal`. Пакети, встановлені для Node 22, залишаються на диску, але не розглядаються як встановлені для Node 24, доки їх не буде повторно встановлено в новому префіксі Node-major.

Статус залежності робочого столу перевіряє лише активний `userData/node<major>/npmGlobal` префікс. Файли пакетів поза цим префіксом ігноруються керуванням залежностями.
