## Επίλυση προκλήσεων ελέγχου ταυτότητας WebSocket προγράμματος περιήγησης: Ένα πρακτικό μοτίβο διακομιστή μεσολάβησης για την αναγνώριση ομιλίας Doubao

> Το πρόγραμμα περιήγησης WebSocket API δεν υποστηρίζει προσαρμοσμένες κεφαλίδες HTTP, γεγονός που δημιουργεί πρόκληση για υπηρεσίες αναγνώρισης ομιλίας που απαιτούν δεδομένα ελέγχου ταυτότητας στις κεφαλίδες. Αυτό το άρθρο περιγράφει πώς λύσαμε αυτό το πρόβλημα στο έργο HagiCode με ένα μοτίβο διακομιστή μεσολάβησης υποστήριξης και πώς η προσέγγιση εξελίχθηκε από τα πειράματα παιδικής χαράς στη χρήση παραγωγής.


## φόντο

Όταν ξεκινήσαμε τη δημιουργία αναγνώρισης ομιλίας για το έργο HagiCode, επιλέξαμε με σιγουριά την υπηρεσία αναγνώρισης ομιλίας Doubao της ByteDance. Ο αρχικός σχεδιασμός ήταν απλός: αφήστε το frontend να συνδεθεί απευθείας με την υπηρεσία WebSocket του Doubao. Πόσο δύσκολο μπορεί να είναι αυτό; Απλώς ανοίξτε μια σύνδεση και στείλτε κάποια δεδομένα, σωστά;

Στη συνέχεια ήρθε η έκπληξη: το API του Doubao απαιτεί τη διαβίβαση πληροφοριών ελέγχου ταυτότητας μέσω των κεφαλίδων HTTP, συμπεριλαμβανομένων στοιχείων όπως `accessToken` και `secretKey`. Αυτό έγινε αμέσως περίεργο, επειδή το API WebSocket του προγράμματος περιήγησης απλά δεν υποστηρίζει τη ρύθμιση προσαρμοσμένων κεφαλίδων.

Τι κάνετε λοιπόν όταν το πρόγραμμα περιήγησης δεν σας επιτρέπει να τα στείλετε;

Εκείνη την εποχή, σταθμίσαμε δύο επιλογές:

1. Τοποθετήστε τα διαπιστευτήρια στις παραμέτρους ερωτήματος URL - απλά και αμβλύ
2. Προσθέστε ένα επίπεδο διακομιστή μεσολάβησης στο backend - περισσότερη δουλειά με την πρώτη ματιά

Η πρώτη επιλογή εκθέτει τα διαπιστευτήρια απευθείας στον κώδικα διεπαφής και στον τοπικό χώρο αποθήκευσης. Είναι πραγματικά ασφαλές; Δεν ένιωθα άνετα με αυτό. Επιπλέον, ορισμένα API απαιτούν επαλήθευση βάσει κεφαλίδας, επομένως αυτή η προσέγγιση δεν είναι καν βιώσιμη.

Στο τέλος, επιλέξαμε τη δεύτερη επιλογή: να εφαρμόσουμε έναν διακομιστή μεσολάβησης WebSocket στο backend. Συμπτωματικά, αυτό το μοτίβο επικυρώθηκε για πρώτη φορά στο περιβάλλον της παιδικής χαράς μας και μόνο αφού επιβεβαιώσαμε τη σταθερότητά του, το μεταφέραμε στην παραγωγή. Σε τελική ανάλυση, κανείς δεν θέλει η παραγωγή να διπλασιαστεί ως εργαστηριακό πείραμα.

## Τεχνική Ανάλυση Προκλήσεων

### Περιορισμοί του προγράμματος περιήγησης WebSocket

Το τυπικό WebSocket API φαίνεται υπέροχα απλό:

```javascript
const ws = new WebSocket('wss://example.com/ws');
```

Αλλά αυτή η απλότητα είναι ακριβώς εκεί που βρίσκεται το πρόβλημα - μεταβιβάζει μόνο παραμέτρους στη διεύθυνση URL και δεν μπορεί να ορίσει κεφαλίδες με τον τρόπο που μπορεί ένα αίτημα HTTP:

```javascript
// This is not supported in the WebSocket API
const ws = new WebSocket('wss://example.com/ws', {
  headers: {
    'Authorization': 'Bearer token'
  }
});
```

Και αυτό είναι το βασικό ζήτημα. Για υπηρεσίες όπως η αναγνώριση ομιλίας Doubao που εξαρτώνται από τον έλεγχο ταυτότητας βάσει κεφαλίδας, αυτός ο περιορισμός αποτελεί σκληρό αποκλεισμό.

Μόλις αποδεχτείτε αυτόν τον περιορισμό, η αρχιτεκτονική πρέπει να αλλάξει.

### Αρχιτεκτονικές αποφάσεις σχεδιασμού

Κατά το σχεδιασμό της λύσης, συγκρίναμε προσεκτικά τις ανταλλαγές.

**Απόφαση 1: Επιλογή του μοτίβου διακομιστή μεσολάβησης**

Συγκρίναμε δύο προσεγγίσεις:

| Επιλογή | Πλεονεκτήματα | Μειονεκτήματα | Απόφαση |
|------|------|------|------|
| Native WebSocket | Ελαφρύ, απλό, άμεση προώθηση | Η διαχείριση της σύνδεσης πρέπει να γίνεται χειροκίνητα | Επιλεγμένο |
| SignalR | Αυτόματη επανασύνδεση, δυνατή πληκτρολόγηση | Υπερβολικά περίπλοκες, επιπλέον εξαρτήσεις | Απορρίφθηκε |

Επιλέξαμε τελικά το εγγενές WebSocket. Για να είμαι ειλικρινής, ήταν η πιο ελαφριά επιλογή και ταίριαζε καλύτερα για απλή αμφίδρομη προώθηση δυαδικής ροής. Το να τραβήξετε το SignalR θα ήταν σαν υπερ-μηχανική για αυτήν την περίπτωση χρήσης και θα μπορούσε να προσθέσει επιπλέον καθυστέρηση.

**Απόφαση 2: Στρατηγική διαχείρισης σύνδεσης**

Υιοθετήσαμε ένα μοντέλο "μία σύνδεση, μία συνεδρία" - κάθε σύνδεση WebSocket στο frontend αντιστοιχίζεται στη δική της ανεξάρτητη σύνδεση υποστήριξης Doubao.

Τα οφέλη είναι ξεκάθαρα:
- Απλό στην εφαρμογή και ευθυγραμμισμένο με το κοινό μοτίβο χρήσης
- Πιο εύκολο στον εντοπισμό σφαλμάτων και την αντιμετώπιση προβλημάτων
- Καλή απομόνωση πόρων, αποτρέποντας παρεμβολές μεταξύ των συνεδριών

Με απλά λόγια, η άμεση λύση είναι μερικές φορές η καλύτερη. Η πολυπλοκότητα δεν κάνει αυτόματα ένα σχέδιο καλύτερο.

**Απόφαση 3: Αποθήκευση δεδομένων ελέγχου ταυτότητας**

Τα διαπιστευτήρια αποθηκεύονται σε αρχεία διαμόρφωσης backend (`appsettings.yml` ή μεταβλητές περιβάλλοντος) και φορτώνονται μέσω της ένεσης εξάρτησης:

- Απλό μοντέλο διαμόρφωσης που ταιριάζει με τις υπάρχουσες συμβάσεις υποστήριξης
- Τα ευαίσθητα δεδομένα δεν φτάνουν ποτέ στο frontend
- Υποστηρίζει ρύθμιση πολλαπλού περιβάλλοντος για ανάπτυξη, δοκιμή και παραγωγή

Αυτό το επίπεδο διαχωρισμού έχει σημασία. Κανείς δεν θέλει τα διαπιστευτήρια να επιπλέουν σε μέρη που δεν θα έπρεπε.

### Σχεδιασμός ροής δεδομένων

Η συνολική ροή δεδομένων μοιάζει με αυτό:

```
Frontend (browser)
  │
  │ ws://backend/api/voice/ws
  │ WebSocket (binary)
  ▼
Backend (proxy)
  │
  │ wss://openspeech.bytedance.com/
  │ (with authentication headers)
  ▼
Doubao API
```

Η ίδια η ροή δεν είναι περίπλοκη:
1. Το frontend συνδέεται με τον διακομιστή μεσολάβησης υποστήριξης μέσω του WebSocket
2. Ο διακομιστής μεσολάβησης υποστήριξης λαμβάνει δεδομένα ήχου και συνδέεται με το API Doubao με επαληθευμένες κεφαλίδες
3. Το Doubao API επιστρέφει αποτελέσματα αναγνώρισης και ο διακομιστής μεσολάβησης τα προωθεί στο frontend
4. Η όλη διαδικασία παραμένει πλήρως ασύγχρονη με αμφίδρομη ροή

Μόλις κατανεμηθούν ξεκάθαρα οι αρμοδιότητες, ο σχεδιασμός γίνεται αρκετά φυσικός.

## Υλοποίηση βασικού στοιχείου

### 1. Διαμόρφωση τελικού σημείου WebSocket

```csharp
app.Map("/ws", async context =>
{
    if (context.WebSockets.IsWebSocketRequest)
    {
        // Read configuration from query parameters
        var appId = context.Request.Query["appId"];
        var accessToken = context.Request.Query["accessToken"];

        // Validate required parameters
        if (string.IsNullOrEmpty(appId) || string.IsNullOrEmpty(accessToken))
        {
            context.Response.StatusCode = 400;
            return;
        }

        // Accept the WebSocket connection
        using var webSocket = await context.WebSockets.AcceptWebSocketAsync();

        // Message handling loop
        var buffer = new byte[4096];
        while (!webSocket.CloseStatus.HasValue)
        {
            var result = await webSocket.ReceiveAsync(buffer, CancellationToken.None);

            if (result.MessageType == WebSocketMessageType.Close)
            {
                await webSocket.CloseAsync(
                    result.CloseStatus.Value,
                    result.CloseStatusDescription,
                    CancellationToken.None);
                break;
            }

            // Process audio data
            await HandleAudioDataAsync(buffer, result.Count);
        }
    }
});
```

### 2. Διαχείριση συνεδρίας

```csharp
public class DoubaoSessionManager : IDoubaoSessionManager
{
    private readonly ConcurrentDictionary<string, DoubaoSession> _sessions = new();

    public DoubaoSession CreateSession(string connectionId)
    {
        var session = new DoubaoSession(connectionId);
        _sessions[connectionId] = session;
        return session;
    }

    public async Task SendAudioAsync(string connectionId, byte[] audioData)
    {
        if (_sessions.TryGetValue(connectionId, out var session))
        {
            await session.SendAudioAsync(audioData);
        }
    }

    public void RemoveSession(string connectionId)
    {
        if (_sessions.TryRemove(connectionId, out var session))
        {
            session.Dispose();
        }
    }
}
```

Χρησιμοποιώντας `ConcurrentDictionary` για τη διαχείριση συνεδρίας σημαίνει ότι η ασφάλεια νημάτων αντιμετωπίζεται σε μεγάλο βαθμό για εμάς. Κάθε εισερχόμενη σύνδεση έχει τη δική της περίοδο λειτουργίας και η εκκαθάριση γίνεται αυτόματα κατά την αποσύνδεση.

## Ενσωμάτωση Frontend στην πράξη

### Σύνδεση WebSocket

```javascript
class DoubaoVoiceClient {
    constructor(config) {
        this.config = config;
        this.ws = null;
    }

    async connect() {
        const url = new URL(this.config.wsUrl);
        // Add query parameters
        Object.entries(this.config.params).forEach(([key, value]) => {
            url.searchParams.set(key, value);
        });

        this.ws = new WebSocket(url);

        return new Promise((resolve, reject) => {
            this.ws.onopen = () => {
                console.log('[DoubaoVoice] Connected');
                resolve();
            };

            this.ws.onmessage = (event) => {
                this._handleMessage(JSON.parse(event.data));
            };

            this.ws.onerror = reject;
        });
    }

    _handleMessage(message) {
        switch (message.type) {
            case 'status':
                this._handleStatus(message.payload);
                break;
            case 'result':
                this.onResult?.(message.payload);
                break;
            case 'error':
                console.error('[DoubaoVoice] Error:', message.payload);
                break;
        }
    }
}

// Usage example
const client = new DoubaoVoiceClient({
    wsUrl: 'ws://localhost:5000/ws',
    params: {
        appId: 'your-app-id',
        accessToken: 'your-access-token',
        sampleRate: 16000,
        bitsPerSample: 16,
        channels: 1
    }
});
```

### Λήψη και μετάδοση ήχου

Χρησιμοποιώντας `AudioWorkletNode` για επεξεργασία ήχου παρέχει καλύτερη απόδοση:

```javascript
// audio-worklet.js
class AudioProcessorWorklet extends AudioWorkletProcessor {
    process(inputs, outputs, parameters) {
        const input = inputs[0]?.[0];
        if (!input) return true;

        // Convert to 16-bit PCM
        const pcm = new Int16Array(input.length);
        for (let i = 0; i < input.length; i++) {
            pcm[i] = Math.max(-32768, Math.min(32767, input[i] * 32767));
        }

        this.port.postMessage({
            type: 'audioData',
            data: pcm.buffer
        }, [pcm.buffer]);

        return true;
    }
}

registerProcessor('audio-processor', AudioProcessorWorklet);

// Main thread code
async function startAudioRecording() {
    const stream = await navigator.mediaDevices.getUserMedia({
        audio: {
            echoCancellation: true,
            noiseSuppression: true,
            autoGainControl: true,
            sampleRate: 48000
        }
    });

    const audioContext = new AudioContext();
    const audioSource = audioContext.createMediaStreamSource(stream);

    await audioContext.audioWorklet.addModule('/audio-worklet.js');
    const audioWorkletNode = new AudioWorkletNode(audioContext, 'audio-processor');

    audioWorkletNode.port.onmessage = (event) => {
        if (event.data.type === 'audioData' && ws?.readyState === WebSocket.OPEN) {
            ws.send(event.data.data); // Send binary data directly
        }
    };

    audioSource.connect(audioWorkletNode);
}
```

`AudioWorklet` αποδίδει πολύ καλύτερα από `ScriptProcessorNode` και αποφεύγει τα προβλήματα τραύματος ήχου που συχνά εισάγουν παλαιότερες διαδρομές επεξεργασίας.

## Σημειώσεις και βέλτιστες πρακτικές

### 1. Παρακολούθηση σύνδεσης

- Καταγράφετε τακτικά την κατάσταση περιόδου λειτουργίας για να παρακολουθείτε τον πλήρη κύκλο ζωής της σύνδεσης
- Παρακολουθήστε τον αριθμό και τη διάρκεια των τμημάτων ήχου για να εντοπίσετε μη κανονικές συνδέσεις
- Καταγράψτε την κατάσταση σύνδεσης και τη συμπεριφορά επανασύνδεσης για την υπηρεσία Doubao

Αυτές είναι βασικές επιχειρησιακές πρακτικές, αλλά κάνουν πραγματική διαφορά στην παραγωγή.

### 2. Αντιμετώπιση σφαλμάτων

- Καταγράψτε και καταγράψτε όλες τις εξαιρέσεις του WebSocket
- Χρήση `IAsyncDisposable` για να εξασφαλιστεί η εκκαθάριση των πόρων
- Εφαρμόστε χαριτωμένο τερματισμό σύνδεσης και χειρισμό χρονικού ορίου

Εν ολίγοις, ευνοήστε τη στιβαρότητα.

### 3. Απαιτήσεις μορφής ήχου

- Ρυθμός δειγματοληψίας: 16000 Hz (συνιστάται) ή 8000 Hz
- Βάθος bit: 16-bit
- Κανάλια: μονοφωνικά
- Κωδικοποίηση: PCM (ακατέργαστο)

Εάν η μορφή είναι λάθος, η αναγνώριση μπορεί να αποτύχει ή η ποιότητα μπορεί να υποβαθμιστεί σημαντικά. Αυτές οι απαιτήσεις έχουν σημασία.

### 4. Θέματα ασφαλείας

- Διατηρήστε ευαίσθητα διαπιστευτήρια μόνο στη διαμόρφωση backend
- Επιβολή ορίων σύνδεσης για να αποφευχθεί η εξάντληση των πόρων
- Χρησιμοποιήστε HTTPS/WSS στην παραγωγή

Η ασφάλεια δεν είναι ποτέ δευτερεύουσας σημασίας.

### 5. Βελτιστοποίηση απόδοσης

- Χρησιμοποιήστε ασύγχρονες λειτουργίες για να αποφύγετε τον αποκλεισμό
- Συντονίστε μεγέθη buffer όπως απαιτείται (προεπιλογή: 4096 byte)
- Εξετάστε τις στρατηγικές συγκέντρωσης και επαναχρησιμοποίησης συνδέσεων

Εφαρμόστε αυτές τις βελτιστοποιήσεις όπου έχουν νόημα για τον φόρτο εργασίας σας.

## Συστάσεις ανάπτυξης

1. **Ανάπτυξη Docker**: Συσκευάστε την υπηρεσία διακομιστή μεσολάβησης ως κοντέινερ για ευκολότερη κλιμάκωση και διαχείριση
2. **Εξισορρόπηση φόρτου**: Χρησιμοποιήστε το Nginx ή το Envoy ως αντίστροφο διακομιστή μεσολάβησης για την κυκλοφορία WebSocket
3. **Έλεγχοι υγείας**: Εφαρμόστε ελέγχους με βάση τον καρδιακό παλμό για την παρακολούθηση της διαθεσιμότητας των υπηρεσιών
4. **Συγκέντρωση αρχείων καταγραφής**: Αποστολή αρχείων καταγραφής σε ένα κεντρικό σύστημα καταγραφής όπως το ELK ή το Loki

Η ανάπτυξη μπορεί να είναι απλή ή περίπλοκη ανάλογα με την ομάδα και το περιβάλλον, επομένως προσαρμόστε ανάλογα.

## Περίληψη

Το μοτίβο διακομιστή μεσολάβησης WebSocket επιλύει το βασικό πρόβλημα ότι το API WebSocket του προγράμματος περιήγησης δεν υποστηρίζει προσαρμοσμένες κεφαλίδες. Στο έργο HagiCode, αυτό το μοτίβο αποδείχθηκε εφικτό και σταθερό καθώς πέρασε από την επικύρωση παιδικής χαράς στην ανάπτυξη παραγωγής.

Βασικά φαγητά:
- Ένας διακομιστής μεσολάβησης υποστήριξης μπορεί να μεταβιβάσει με ασφάλεια πληροφορίες ελέγχου ταυτότητας
- Το Native WebSocket είναι ελαφρύ και αποτελεσματικό για απλά σενάρια
- Το μοντέλο "μία σύνδεση, μία συνεδρία" απλοποιεί τόσο την υλοποίηση όσο και τον εντοπισμό σφαλμάτων
- Το πρωτόκολλο frontend-backend θα πρέπει να διαχωρίζει τα σήματα ελέγχου από τα δεδομένα ήχου

Εάν δημιουργείτε μια δυνατότητα που εξαρτάται επίσης από τον έλεγχο ταυτότητας WebSocket, ελπίζω αυτό το μοτίβο να σας προσφέρει ένα χρήσιμο σημείο εκκίνησης.

Εάν έχετε ερωτήσεις, μη διστάσετε να τις συζητήσετε μαζί μας. Η τεχνική πρόοδος συμβαίνει πιο γρήγορα όταν οι άνθρωποι συγκρίνουν σημειώσεις.
