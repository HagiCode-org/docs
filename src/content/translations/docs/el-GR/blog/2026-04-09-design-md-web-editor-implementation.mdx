## Επεξεργαστείτε το DESIGN.md απευθείας στη διεπαφή Ιστού: Από την ιδέα στην υλοποίηση

> Στο σύστημα διαχείρισης έργου MonoSpecs, `DESIGN.md` φέρει τον αρχιτεκτονικό σχεδιασμό και τις τεχνικές αποφάσεις ενός έργου. Αλλά η παραδοσιακή ροή εργασιών επεξεργασίας αναγκάζει τους χρήστες να μεταβούν σε έναν εξωτερικό επεξεργαστή. Αυτή η κατακερματισμένη εμπειρία είναι σαν να σε διακόπτουν στη μέση της ανάγνωσης ενός ποιήματος: η έμπνευση έχει φύγει, το ίδιο και η διάθεση. Αυτό το άρθρο μοιράζεται τη λύση που εφαρμόσαμε στην πράξη στο έργο HagiCode: επεξεργασία `DESIGN.md` απευθείας στη διεπαφή ιστού, με υποστήριξη για εισαγωγή προτύπων από διαδικτυακό ιστότοπο σχεδιασμού. Τελικά ποιος δεν απολαμβάνει την αίσθηση να ολοκληρώνεις τα πάντα σε μια ροή;

## φόντο

Ως βασικός φορέας των εγγράφων σχεδιασμού έργου, `DESIGN.md` κατέχει βασικές πληροφορίες όπως σχεδιασμός αρχιτεκτονικής, τεχνικές αποφάσεις και οδηγίες υλοποίησης. Ωστόσο, η παραδοσιακή προσέγγιση επεξεργασίας απαιτεί από τους χρήστες να μεταβούν σε έναν εξωτερικό επεξεργαστή όπως ο κώδικας VS, να εντοπίσουν με μη αυτόματο τρόπο τη φυσική διαδρομή και, στη συνέχεια, να επεξεργαστούν το αρχείο. Δεν είναι ιδιαίτερα περίπλοκο, αλλά αφού επαναλάβετε τη διαδικασία μερικές φορές, γίνεται κουραστικό.

Τα προβλήματα εμφανίζονται κυρίως με τους εξής τρόπους:

- **Κατακερματισμένη ροή εργασίας**: οι χρήστες πρέπει να αλλάζουν συνεχώς μεταξύ της διεπαφής διαχείρισης ιστού και ενός τοπικού προγράμματος επεξεργασίας, παραβιάζοντας τη συνέχεια της ροής εργασίας τους, σαν να κόβουν τη μουσική στη μέση ενός τραγουδιού.
- **Δύσκολο στην επαναχρησιμοποίηση**: ο ιστότοπος σχεδίασης δημοσιεύει ήδη μια πλούσια βιβλιοθήκη προτύπων σχεδίασης, αλλά δεν μπορούν να ενσωματωθούν απευθείας στη ροή εργασιών επεξεργασίας έργου. Τα καλά πράγματα υπάρχουν, αλλά ακόμα δεν μπορείτε να τα χρησιμοποιήσετε όπου τα χρειάζεστε.
- **Λείπει βρόχος εμπειρίας**: δεν υπάρχει κλειστός βρόχος για "προεπισκόπηση-επιλογή-εισαγωγή", επομένως οι χρήστες πρέπει να αντιγράψουν και να επικολλήσουν με μη αυτόματο τρόπο, γεγονός που αυξάνει τον κίνδυνο λαθών.
- **Τριβή συνεργασίας**: η συγχρονισμός των εγγράφων σχεδιασμού και της εφαρμογής κώδικα γίνεται μια διαδικασία υψηλής τριβής, η οποία βλάπτει την αποτελεσματικότητα της ομάδας.

Για να λύσουμε αυτά τα σημεία πόνου, αποφασίσαμε να προσθέσουμε υποστήριξη άμεσης επεξεργασίας για `DESIGN.md` στη διεπαφή ιστού και επιτρέψτε την εισαγωγή προτύπου με ένα κλικ από έναν ιστότοπο σχεδιασμού στο διαδίκτυο. Δεν ήταν κάποια απόφαση που ταρακουνούσε τη γη. Θέλαμε απλώς να κάνουμε την εμπειρία ανάπτυξης πιο ομαλή.

### Βασικές Τεχνικές Αποφάσεις

#### Απόφαση 1: Καθολικό μοτίβο συρταριού

Χρησιμοποιούμε ένα μόνο καθολικό συρτάρι αντί για τοπικά αναδυόμενα επίπεδα, με διαχείριση κατάστασης `layoutSlice`, που παρέχει στους χρήστες μια συνεπή εμπειρία σε όλες τις προβολές (`classic` και `kanban`). Ανεξάρτητα από την προβολή από την οποία ο χρήστης ανοίγει το πρόγραμμα επεξεργασίας, έχει το ίδιο μοντέλο αλληλεπίδρασης. Μια σταθερή εμπειρία κάνει τους ανθρώπους να αισθάνονται πιο άνετα αντί να αποπροσανατολίζονται όταν αλλάζουν απόψεις.

#### Απόφαση 2: Project-Scoped API

Ανεβήκαμε `DESIGN.md`-σχετικά τελικά σημεία κάτω από `ProjectController`, επαναχρησιμοποιώντας το υπάρχον όριο άδειας έργου και αποφεύγοντας την πολυπλοκότητα της προσθήκης ξεχωριστού ελεγκτή. Αυτό καθιστά σαφέστερο τον χειρισμό αδειών και ευθυγραμμίζεται επίσης με την οργάνωση πόρων RESTful. Μερικές φορές η επαναχρησιμοποίηση έχει πιο νόημα από το να δημιουργείς κάτι νέο από την αρχή.

#### Απόφαση 3: Ανίχνευση σύγκρουσης έκδοσης

Εξάγουμε μια αδιαφανή έκδοση από το σύστημα αρχείων `LastWriteTimeUtc`, που μας δίνει ελαφρύ αισιόδοξο έλεγχο ταυτόχρονης χρήσης. Όταν πολλοί χρήστες επεξεργάζονται το ίδιο αρχείο ταυτόχρονα, το σύστημα μπορεί να εντοπίσει διενέξεις και να ζητήσει από τον χρήστη να ανανεώσει. Αυτό το σχέδιο δεν εμποδίζει την επεξεργασία, ενώ εξακολουθεί να προστατεύει τη συνοχή των δεδομένων.

#### Απόφαση 4: Μοτίβο διακομιστή μεσολάβησης ίδιας προέλευσης

χρησιμοποιούμε `IHttpClientFactory` για τη διαμεσολάβηση εξωτερικών πόρων της τοποθεσίας σχεδιασμού, αποφεύγοντας τόσο ζητήματα διασταυρούμενης προέλευσης όσο και κινδύνους SSRF. Αυτό διατηρεί το σύστημα ασφαλές ενώ ταυτόχρονα απλοποιεί τις κλήσεις διεπαφής. Δύσκολα μπορείς να είσαι πολύ προσεκτικός με την ασφάλεια.

## Βασική Υλοποίηση

### 1. Επεξεργαστείτε το DESIGN.md απευθείας

#### Εφαρμογή Backend

Το backend είναι κυρίως υπεύθυνο για την ανάλυση διαδρομής, την ανάγνωση/εγγραφή αρχείων και τη διαχείριση της έκδοσης. Αυτές οι εργασίες είναι βασικές, αλλά απαραίτητες, όπως η θεμελίωση ενός σπιτιού:

```csharp
// Path resolution and security validation
private Task<string> ResolveDesignDocumentDirectoryAsync(string projectPath, string? repositoryPath)
{
    if (string.IsNullOrWhiteSpace(repositoryPath))
    {
        return Task.FromResult(Path.GetFullPath(projectPath));
    }
    return ValidateSubPathAsync(projectPath, repositoryPath);
}

// Version generation (based on file system timestamp and size)
private static string BuildDesignDocumentVersion(string path)
{
    var fileInfo = new FileInfo(path);
    fileInfo.Refresh();
    return string.Create(
        CultureInfo.InvariantCulture,
        $"{fileInfo.LastWriteTimeUtc.Ticks:x}-{fileInfo.Length:x}");
}
```

Ο σχεδιασμός της έκδοσης είναι ενδιαφέρον στην απλότητά του: χρησιμοποιούμε τον χρόνο και το μέγεθος της τελευταίας τροποποίησης του αρχείου για να δημιουργήσουμε ένα μοναδικό αναγνωριστικό έκδοσης. Είναι ελαφρύ και αξιόπιστο, χωρίς επιπλέον βάση δεδομένων εκδόσεων για συντήρηση. Οι απλές λύσεις είναι συχνά οι πιο αποτελεσματικές.

#### Εφαρμογή Frontend

Στο μπροστινό μέρος, εφαρμόζουμε τον εντοπισμό βρώμικων καταστάσεων και αποθηκεύουμε τη λογική. Αυτός ο σχεδιασμός βοηθά τους χρήστες να κατανοήσουν εάν οι αλλαγές τους έχουν αποθηκευτεί και μειώνει το άγχος του "κι αν το χάσω;":

```typescript
// Dirty-state detection and save logic
const [draft, setDraft] = useState('');
const [savedDraft, setSavedDraft] = useState('');
const isDirty = draft !== savedDraft;

const handleSave = useCallback(async () => {
    const result = await saveProjectDesignMdDocument({
        ...activeTarget,
        content: draft,
        expectedVersion: document.version, // optimistic concurrency control
    });
    setSavedDraft(draft); // update saved state
}, [activeTarget, document, draft]);
```

Σε αυτήν την υλοποίηση, διατηρούμε δύο κομμάτια κατάστασης: `draft` είναι το περιεχόμενο που επεξεργάζεται αυτήν τη στιγμή, ενώ `savedDraft` είναι το αποθηκευμένο περιεχόμενο. Η σύγκριση τους μας λέει αν υπάρχουν μη αποθηκευμένες αλλαγές. Ο σχεδιασμός είναι απλός, αλλά δίνει στους ανθρώπους ηρεμία. Κανείς δεν θέλει να εξαφανιστεί αυτό για το οποίο δούλεψαν σκληρά.

#### Εφαρμογή διακομιστή μεσολάβησης ίδιας προέλευσης

Για να διατηρούνται τα πράγματα ασφαλή, το backend εκτελεί αυστηρή επικύρωση κατά την πρόσβαση στον ιστότοπο σχεδιασμού. Δεν μπορείτε να είστε πολύ προσεκτικοί σχετικά με την ασφάλεια:

```csharp
// Safe slug validation
private static readonly Regex SafeDesignSiteSlugRegex =
    new("^[A-Za-z0-9](?:[A-Za-z0-9._-]{0,127})$", RegexOptions.Compiled);

private static string NormalizeDesignSiteSlug(string slug)
{
    var normalizedSlug = slug?.Trim() ?? string.Empty;
    if (!IsSafeDesignSiteSlug(normalizedSlug))
    {
        throw new BusinessException(
            ProjectDesignSiteIndexErrorCodes.InvalidSlug,
            "Design site slug must be a single safe path segment.");
    }
    return normalizedSlug;
}

// Preview image caching (OS temp directory)
private static string ComputePreviewCacheKey(string slug, string theme, string previewUrl)
{
    var raw = $"{slug}|{theme}|{previewUrl}";
    var bytes = SHA256.HashData(Encoding.UTF8.GetBytes(raw));
    return Convert.ToHexString(bytes).ToLowerInvariant();
}
```

Κάνουμε δύο πράγματα εδώ: πρώτον, επικυρώνουμε τη μορφή slug αυστηρά με μια κανονική έκφραση για να αποτρέψουμε επιθέσεις διάβασης μονοπατιών. Δεύτερον, αποθηκεύουμε προσωρινά τις εικόνες προεπισκόπησης για να μειώσουμε την πίεση στον εξωτερικό ιστότοπο. Το πρώτο είναι προστασία, το δεύτερο είναι βελτιστοποίηση, και τα δύο έχουν σημασία.

### 3. Πλήρης ροή εισαγωγής

```typescript
// 1. Open the import drawer
const handleRequestImportDrawer = useCallback(() => {
    setIsImportDrawerOpen(true);
}, []);

// 2. Select and import
const handleImportRequest = useCallback((entry) => {
    if (isDirty) {
        setPendingImportEntry(entry);
        setConfirmMode('import'); // overwrite confirmation
        return;
    }
    void executeImport(entry);
}, [isDirty]);

// 3. Execute import
const executeImport = useCallback(async (entry) => {
```

Η ροή εισαγωγής ακολουθεί μια αρχή "επιβεβαίωσης χρήστη": μετά την εισαγωγή, ενημερώνεται μόνο το περιεχόμενο του επεξεργαστή και τίποτα δεν αποθηκεύεται αυτόματα. Οι χρήστες μπορούν να επιθεωρήσουν το εισαγόμενο περιεχόμενο και να το αποθηκεύσουν με μη αυτόματο τρόπο μόνο αφού επιβεβαιώσουν ότι φαίνεται σωστό. Η τελική απόφαση θα πρέπει να παραμείνει στα χέρια του χρήστη.

## Πρακτικά Παραδείγματα

### Σενάριο 1: Δημιουργία DESIGN.md στο Project Root

Πότε `DESIGN.md` δεν υπάρχει, το backend επιστρέφει μια κατάσταση εικονικού εγγράφου. Αυτό επιτρέπει στο frontend να αποφεύγει τον ειδικό χειρισμό για την περίπτωση "το αρχείο δεν υπάρχει" και ένα ενοποιημένο API απλοποιεί τη λογική του κώδικα:

```csharp
return new ProjectDesignDocumentDto
{
    Path = targetPath,
    Exists = false,  // virtual document state
    Content = string.Empty,
    Version = null
};

// Automatically create the file on first save
public async Task<SaveProjectDesignDocumentResultDto> SaveDesignDocumentAsync(...)
{
    Directory.CreateDirectory(targetDirectory);
    await File.WriteAllTextAsync(targetPath, input.Content);
    return new SaveProjectDesignDocumentResultDto { Created = !exists };
}
```

Το πλεονέκτημα αυτού του σχεδιασμού είναι ότι το frontend δεν χρειάζεται λογική ειδικής περίπτωσης για αρχεία που λείπουν. Με την απόκρυψη αυτής της πολυπλοκότητας στο backend, το frontend μπορεί να επικεντρωθεί πιο εύκολα στην εμπειρία χρήστη.

### Σενάριο 2: Εισαγωγή προτύπου από την τοποθεσία σχεδίασης

Αφού ο χρήστης επιλέξει το πρότυπο σχεδίασης "Γραμμικό" στο συρτάρι εισαγωγής, το σύστημα ανακτά το `DESIGN.md` περιεχόμενο μέσω του διακομιστή μεσολάβησης υποστήριξης. Η όλη διαδικασία είναι διαφανής για τον χρήστη: επιλέγει μόνο ένα πρότυπο και το σύστημα χειρίζεται αυτόματα τα αιτήματα δικτύου και τον μετασχηματισμό δεδομένων.

```typescript
// 1. The system fetches DESIGN.md content through the backend proxy
GET /api/project/{id}/design-md/site-index/linear.app

// 2. The backend validates the slug and fetches content from upstream
var entry = FindDesignSiteEntry(catalog, "linear.app");
using var upstreamResponse = await httpClient.SendAsync(request);
var content = await upstreamResponse.Content.ReadAsStringAsync();

// 3. The frontend replaces the editor text
setDraft(result.content);
// The user reviews it and then saves it manually to disk
```

Όλη η ροή παραμένει διαφανής στον χρήστη. Απλώς επιλέγουν ένα πρότυπο και το σύστημα χειρίζεται τη δικτύωση και τον μετασχηματισμό στα παρασκήνια. Αυτή είναι η εμπειρία που θέλουμε: απλή, αλλά ισχυρή.

### Σενάριο 3: Χειρισμός διενέξεων εκδόσεων

Όταν πολλοί χρήστες επεξεργάζονται το ίδιο `DESIGN.md` Ταυτόχρονα, το σύστημα εντοπίζει διενέξεις εκδόσεων. Αυτός ο αισιόδοξος μηχανισμός ελέγχου συγχρονισμού διατηρεί τη συνέπεια των δεδομένων χωρίς να μπλοκάρει τις αλλαγές του χρήστη:

```csharp
if (!string.Equals(currentVersion, expectedVersion, StringComparison.Ordinal))
{
    throw new BusinessException(
        ProjectDesignDocumentErrorCodes.VersionConflict,
        $"DESIGN.md at '{targetPath}' changed on disk.");
}
```

Το frontend εντοπίζει αυτό το σφάλμα και ζητά από τον χρήστη:

```typescript
// Frontend prompts the user to refresh and retry
<Alert>
    <AlertTitle>Version conflict</AlertTitle>
    <AlertDescription>
        The file was modified by another process. Please refresh to get the latest version and try again.
    </AlertDescription>
</Alert>
```

Αυτός ο αισιόδοξος μηχανισμός ελέγχου συγχρονισμού διατηρεί τα δεδομένα συνεπή χωρίς να εμποδίζει τους χρήστες ενώ εργάζονται. Οι συγκρούσεις είναι αναπόφευκτες, αλλά τουλάχιστον οι χρήστες θα πρέπει να γνωρίζουν τι συνέβη αντί να χάνουν σιωπηλά τις αλλαγές τους.

## Σημειώσεις και βέλτιστες πρακτικές

### 1. Ασφάλεια διαδρομής

Πάντα επικύρωση `repositoryPath` για την αποτροπή επιθέσεων διάβασης μονοπατιών. Δεν μπορείτε ποτέ να κάνετε πάρα πολλά όσον αφορά την ασφάλεια:

```csharp
// Always validate repositoryPath to prevent path traversal attacks
return ValidateSubPathAsync(projectPath, repositoryPath);
// Reject dangerous inputs such as "../" and absolute paths
```

### 2. Στρατηγική κρυφής μνήμης

Προσωρινή προεπισκόπηση εικόνων για 24 ώρες, με μέγιστο αριθμό 160 αρχείων. Η μέτρια προσωρινή αποθήκευση βελτιώνει την απόδοση, αλλά η ισορροπία εξακολουθεί να έχει σημασία:

```csharp
// Cache preview images for 24 hours, with a maximum of 160 files
private static readonly TimeSpan PreviewCacheTtl = TimeSpan.FromHours(24);
private const int PreviewCacheMaxFiles = 160;
// Periodically clean up expired cache
```

### 3. Χειρισμός σφαλμάτων

Υποβαθμίστε χαριτωμένα όταν η τοποθεσία ανάντη δεν είναι διαθέσιμη. Αυτός ο σχεδιασμός διασφαλίζει ότι ακόμη και αν αποτύχει μια εξωτερική εξάρτηση, η βασική λειτουργία επεξεργασίας εξακολουθεί να λειτουργεί κανονικά:

```typescript
// Gracefully degrade when the upstream site is unavailable
try {
    const catalog = await getProjectDesignMdSiteImportCatalog(projectId);
} catch (error) {
    toast.error(t('project.designMd.siteImport.feedback.catalogLoadFailed'));
    // The main editing drawer remains available
}
```

Αυτή η χαριτωμένη υποβάθμιση διασφαλίζει ότι ακόμη και όταν οι εξωτερικές εξαρτήσεις δεν είναι διαθέσιμες, η βασική λειτουργία επεξεργασίας συνεχίζει να λειτουργεί. Ένα σύστημα πρέπει να είναι ανθεκτικό αντί να καταρρέει τη στιγμή που κάτι πάει στραβά.

### 4. Βελτιστοποίηση εμπειρίας χρήστη

Επιβεβαιώστε τις αντικαταστάσεις πριν από την εισαγωγή και μην αποθηκεύονται αυτόματα μετά την εισαγωγή. Οι χρήστες θα πρέπει να διατηρούν τον έλεγχο των ενεργειών τους:

```typescript
// Confirm overwrite before import
if (isDirty) {
    setConfirmMode('import');
    return;
}

// Do not save automatically after import; let the user confirm
setDraft(result.content); // update draft only
// The content is written to disk only after the user reviews it and clicks Save
```

### 5. Θέματα απόδοσης

Χρησιμοποιήστε ένα εργοστάσιο πελάτη HTTP για να αποφύγετε τη δημιουργία πάρα πολλών συνδέσεων. Η διαχείριση των πόρων μπορεί να φαίνεται μικρή, αλλά η σωστή εκτέλεση μπορεί να κάνει μεγάλη διαφορά:

```csharp
// Use an HTTP client factory to avoid creating too many connections
private const string DesignSiteProxyClientName = "ProjectDesignSiteProxy";
private static readonly TimeSpan DesignSiteProxyTimeout = TimeSpan.FromSeconds(8);
```

## Προτεινόμενες επεκτάσεις

1. **Βελτίωση Markdown**: αυτήν τη στιγμή χρησιμοποιούμε ένα βασικό `Textarea`, αλλά θα μπορούσαμε να κάνουμε αναβάθμιση σε CodeMirror για επισήμανση σύνταξης και συντομεύσεις πληκτρολογίου. Όταν ο συντάκτης αισθάνεται καλύτερα, η σύνταξη τεκμηρίωσης είναι επίσης καλύτερη.
2. **Λειτουργία προεπισκόπησης**: προσθέστε προεπισκόπηση Markdown σε πραγματικό χρόνο για να βελτιώσετε την εμπειρία επεξεργασίας. Αυτό που-βλέπεις-είναι-ό,τι-παίρνεις δίνει πάντα στους ανθρώπους περισσότερη αυτοπεποίθηση.
3. **Συγχώνευση διαφορών**: εφαρμόστε έναν έξυπνο αλγόριθμο συγχώνευσης αντί για απλή αντικατάσταση πλήρους κειμένου. Οι συγκρούσεις είναι αναπόφευκτες, αλλά η διαδικασία επίλυσης των συγκρούσεων δεν χρειάζεται να είναι επώδυνη.
4. **Τοπική προσωρινή αποθήκευση**: προσωρινή μνήμη `design.json` στη βάση δεδομένων για τη μείωση της εξάρτησης από τον εξωτερικό ιστότοπο. Όσο λιγότερες εξαρτήσεις έχει ένα σύστημα, τόσο πιο σταθερό τείνει να είναι.

## Περίληψη

Στο έργο HagiCode, εφαρμόσαμε μια ολοκληρωμένη διαδικτυακή λύση επεξεργασίας και εισαγωγής για `DESIGN.md` μέσω της συνεργασίας frontend-backend. Η βασική αξία αυτής της λύσης βρίσκεται στα ακόλουθα σημεία:

- **Υψηλότερη απόδοση**: δεν χρειάζεται να αλλάξετε εργαλεία. Η επεξεργασία και η εισαγωγή εγγράφων σχεδιασμού μπορεί να πραγματοποιηθεί σε μια ενοποιημένη διεπαφή ιστού.
- **Χαμηλότερο εμπόδιο εισόδου**: Η εισαγωγή προτύπου σχεδίασης με ένα κλικ βοηθά τα νέα έργα να ξεκινήσουν γρήγορα.
- **Ασφαλές και αξιόπιστο**: η επικύρωση διαδρομής, ο εντοπισμός διένεξης έκδοσης και οι χαριτωμένοι μηχανισμοί υποβάθμισης διατηρούν το σύστημα σταθερό.
- **Καλύτερη εμπειρία χρήστη**: το καθολικό συρτάρι, ο εντοπισμός βρώμικων καταστάσεων και τα παράθυρα διαλόγου επιβεβαίωσης βελτιώνουν τη συνολική εμπειρία αλληλεπίδρασης.

Αυτή η λύση εκτελείται ήδη στο έργο HagiCode και έχει λύσει τα προβλήματα της ομάδας σχετικά με τη διαχείριση εγγράφων σχεδιασμού. Εάν αντιμετωπίζετε παρόμοια προβλήματα, ελπίζω αυτό το άρθρο να σας δώσει μερικές χρήσιμες ιδέες. Δεν υπάρχει καμία ιδιαίτερα βαθιά θεωρία εδώ, μόνο η πρακτική εργασία του να συναντήσετε ένα πρόβλημα και να βρείτε έναν τρόπο να το λύσετε.
