## Goodbye Script Hell: Γιατί επιλέξαμε το C# για ένα σύγχρονο σύστημα κατασκευής

> Μια βαθιά κατάδυση στον τρόπο με τον οποίο το έργο HagiCode χρησιμοποιεί τη Nuke για να δημιουργήσει έναν ασφαλή για τον τύπο, πολλαπλές πλατφόρμες και εξαιρετικά επεκτάσιμο αυτοματοποιημένο αγωγό κατασκευής, αντιμετωπίζοντας πλήρως τα σημεία πόνου συντήρησης των παραδοσιακών σεναρίων κατασκευής.

## φόντο

Σε όλο το μακρύ ταξίδι της ανάπτυξης λογισμικού, η λέξη "build" τείνει να εμπνέει αγάπη και απογοήτευση. Αγάπη, γιατί με ένα κλικ, ο κώδικας μετατρέπεται σε προϊόν, και αυτή είναι μια από τις πιο ικανοποιητικές στιγμές στον προγραμματισμό. Απογοήτευση, γιατί η διατήρηση ενός σωρού ακατάστατων σεναρίων κατασκευής μπορεί να είναι εφιάλτης.

Σε πολλά έργα, έχουμε συνηθίσει να γράφουμε σενάρια σε Python ή να βασιζόμαστε σε αρχεία διαμόρφωσης XML. Απλά φανταστείτε τον φόβο να σας κυβερνήσουν `<property>` μπλοκ. Αλλά καθώς η πολυπλοκότητα του έργου αυξάνεται, ειδικά σε ένα έργο όπως το HagiCode που εκτείνεται σε εργασίες frontend και backend, πολλαπλές πλατφόρμες και πολλές γλώσσες, οι παραδοσιακές προσεγγίσεις κατασκευής αρχίζουν να δείχνουν τα όριά τους. Διάσπαρτη λογική σεναρίου, χωρίς έλεγχο τύπου, αδύναμη υποστήριξη IDE... καθένα από αυτά τα ζητήματα γίνεται μια μικρή παγίδα που σκοντάφτει την ομάδα από καιρό σε καιρό.

Για να λύσει αυτά τα σημεία πόνου, το έργο HagiCode εισήγαγε το **Nuke**, ένα σύγχρονο σύστημα κατασκευής βασισμένο σε C#. Δεν είναι απλώς ένα εργαλείο. Είναι ένας διαφορετικός τρόπος σκέψης για τη δημιουργία ροών εργασίας. Σήμερα, θα εξετάσουμε γιατί το επιλέξαμε και πώς έχει βελτιώσει δραματικά την εμπειρία ανάπτυξής μας.

## Ανάλυση πυρήνα: Γιατί Nuke;

Ίσως αναρωτιέστε: "Υπάρχουν ήδη τόσα πολλά συστήματα κατασκευής, όπως τα σενάρια Make, Gradle ή ακόμα και απλά σενάρια Shell. Γιατί να ξεφύγετε από τη χρήση ενός που βασίζεται σε C#;"

Αυτή είναι μια δίκαιη ερώτηση. Η βασική απήχηση του Nuke είναι ότι φέρνει τα χαρακτηριστικά της γλώσσας προγραμματισμού που ήδη γνωρίζουμε καλά στον κόσμο των σεναρίων κατασκευής.

### 1. Modularizing Build Workflows: The Art of Targets

Η Nuke έχει μια πολύ ξεκάθαρη σχεδιαστική φιλοσοφία: **όλα είναι στόχος**.

Στα παραδοσιακά σενάρια, είναι εύκολο να καταλήξετε με εκατοντάδες γραμμές γραμμικής λογικής εκτέλεσης μπερδεμένες μεταξύ τους. Στο Nuke, χωρίζουμε τη διαδικασία κατασκευής σε ανεξάρτητη `Target`s. Κάθε στόχος κάνει ακριβώς ένα πράγμα, για παράδειγμα:

- `Clean`: καθαρίστε τον κατάλογο εξόδου
- `Restore`: επαναφορά πακέτων εξάρτησης
- `Compile`: μεταγλώττιση του κώδικα
- `Test`: εκτέλεση δοκιμών μονάδας

Αυτός ο σχεδιασμός ευθυγραμμίζεται πολύ καλά με την αρχή της ενιαίας ευθύνης. Όπως τα δομικά στοιχεία, αυτοί οι στόχοι μπορούν να συνδυαστούν ελεύθερα. Το πιο σημαντικό, το Nuke μας επιτρέπει να ορίζουμε εξαρτήσεις μεταξύ Στόχων. Για παράδειγμα, αν θέλετε να τρέξετε `Test`, το σύστημα θα ελέγξει αυτόματα αν `Compile` έχει ήδη εκτελεστεί. Αν θέλεις `Compile`, τότε `Restore` φυσικά έρχεται πρώτη.

Αυτό το γράφημα εξάρτησης κάνει τη λογική πολύ πιο ξεκάθαρη και βελτιώνει σημαντικά την αποτελεσματικότητα της εκτέλεσης, επειδή το Nuke μπορεί να αναλύσει αυτόματα τη βέλτιστη διαδρομή εκτέλεσης.

### 2. Type Safety: Αντίο στον Εφιάλτη του Typos

Όποιος έχει γράψει σενάρια κατασκευής στην Python πιθανότατα έχει αντιμετωπίσει αυτήν την άβολη κατάσταση: το σενάριο τρέχει για πέντε λεπτά και μετά αποτυγχάνει επειδή `Confi.guration` γράφτηκε λάθος ή επειδή πέρασε μια συμβολοσειρά όπου αναμενόταν μια αριθμητική παράμετρος.

Το μεγαλύτερο πλεονέκτημα της σύνταξης σεναρίων κατασκευής σε C# είναι η **ασφάλεια τύπου**. Αυτό σημαίνει:

- **Έλεγχοι χρόνου μεταγλώττισης**: το IDE μπορεί να σας πει τι είναι λάθος ενώ πληκτρολογείτε αντί να σας κάνει να περιμένετε μέχρι το χρόνο εκτέλεσης.
- **Ασφαλής ανακατασκευή**: εάν θέλετε να μετονομάσετε μια μεταβλητή ή μια μέθοδο, το IDE μπορεί να το χειριστεί σε ένα βήμα χωρίς να σας υποχρεώσει να κάνετε μια επικίνδυνη καθολική αναζήτηση και να την αντικαταστήσετε.
- **Έξυπνη ολοκλήρωση**: το ισχυρό IntelliSense μπορεί να ολοκληρώσει αυτόματα τον κώδικα για εσάς, επομένως δεν χρειάζεται να ξεφυλλίζετε συνεχώς την τεκμηρίωση για να θυμάστε σκοτεινά API.

### 3. Υποστήριξη μεταξύ πλατφορμών: Μία συνεπής εμπειρία κατασκευής

Στο παρελθόν γράφαμε `.bat` στα Windows και `.sh` στο Linux και, στη συνέχεια, ίσως προσθέσει ένα σενάριο Python για να γεφυρώσει το χάσμα. Τώρα, όπου μπορεί να τρέξει το .NET Core ή το .NET 5+ σήμερα, μπορεί να τρέξει και το Nuke.

Αυτό σημαίνει ότι τα μέλη της ομάδας μπορούν να χρησιμοποιούν Windows, Linux ή macOS και είτε προτιμούν το Visual Studio, το VS Code ή το Rider, όλα εκτελούν την ίδια ακριβώς λογική. Αυτό μειώνει σημαντικά τα προβλήματα που προκαλούνται από διαφορές στο περιβάλλον, ειδικά το κλασικό ζήτημα "λειτουργεί στο μηχάνημά μου".

### 4. Διαχείριση παραμέτρων και διαμόρφωσης

Το Nuke παρέχει έναν πολύ κομψό μηχανισμό ανάλυσης παραμέτρων. Δεν χρειάζεται να κάνετε χειροκίνητη ανάλυση `string[] args`. Απλώς ορίζετε μια ιδιότητα και την σχολιάζετε `[Parameter]`, και το Nuke χειρίζεται αυτόματα την αντιστοίχιση από ορίσματα γραμμής εντολών και αρχεία διαμόρφωσης.

Για παράδειγμα, μπορούμε να ορίσουμε τη διαμόρφωση της κατασκευής πολύ εύκολα:

```csharp
[Parameter("Configuration to build - Default is 'Debug'")]
readonly Configuration BuildConfiguration = IsLocalBuild ? Configuration.Debug : Configuration.Release;

Target Compile => _ => _
    .DependsOn(Restore)
    .Executes(() =>
    {
        // Use BuildConfiguration here; it is type-safe
        DotNetBuild(s => s
            .SetConfiguration(BuildConfiguration)
            .SetProjectFile(SolutionFile));
    });
```

Αυτό το στυλ είναι τόσο διαισθητικό όσο και ανθεκτικό στα λάθη.

## Πρακτικός οδηγός: Πώς να το εφαρμόσετε σε ένα έργο

Η θεωρία από μόνη της δεν μας οδηγεί πουθενά. Ας δούμε πώς εφαρμόσαμε αυτή την προσέγγιση στο έργο HagiCode.

### 1. Σχεδιάστε τη Δομή του Έργου

Δεν θέλαμε να δημιουργήσουμε σενάρια που να γεμίζουν τη ρίζα του έργου και επίσης δεν θέλαμε μια δομή καταλόγου τόσο βαθιά που να μοιάζει με ορισμένα έργα Java. Έτσι, τοποθετήσαμε όλα τα αρχεία κατασκευής που σχετίζονται με το Nuke σε ένα μόνο `nukeBuild/` καταλόγου.

Τα οφέλη είναι ξεκάθαρα:

- Η ρίζα του έργου παραμένει καθαρή.
- Η λογική κατασκευής παραμένει συνεκτική και πιο εύκολη στη διαχείριση.
- Τα νέα μέλη της ομάδας μπορούν να δουν αμέσως, "Εδώ ζει η λογική που σχετίζεται με την κατασκευή."

### 2. Σχεδιάστε μια καθαρή αλυσίδα εξάρτησης στόχου

Κατά το σχεδιασμό των Στόχων, ακολουθήσαμε μια αρχή: **ατομικότητα συν ροή εξάρτησης**.

Κάθε Στόχος πρέπει να είναι αρκετά μικρός για να κάνει ακριβώς ένα πράγμα. Για παράδειγμα, `Clean` θα πρέπει να αφαιρεί μόνο αρχεία και δεν θα πρέπει να χειρίζεται αθόρυβα τη συσκευασία στο πλάι.

Μια συνιστώμενη ροή εξάρτησης μοιάζει περίπου με αυτό:

`Clean` -> `Restore` -> `Compile` -> `Test` -> `Pack`

Αυτό βέβαια δεν είναι απόλυτο. Για παράδειγμα, εάν θέλετε να εκτελέσετε μόνο δοκιμές και δεν θέλετε να πακετάρετε τίποτα, το Nuke σας επιτρέπει να το εκτελέσετε `nuke Test` απευθείας και θα χειριστεί αυτόματα τα απαιτούμενα `Restore` και `Compile` βήματα πρώτα.

### 3. Ισχυρός χειρισμός και καταγραφή σφαλμάτων

Ποιο είναι το πιο απογοητευτικό πράγμα για τα σενάρια κατασκευής; Ασαφή μηνύματα σφάλματος. Μια αποτυχημένη κατασκευή που καταγράφει μόνο "Σφάλμα: 1" είναι αρκετή για να τρελάνει οποιονδήποτε.

Στο Nuke, επειδή μπορούμε να χρησιμοποιήσουμε απευθείας τον χειρισμό εξαιρέσεων C#, μπορούμε να καταγράψουμε και να αναφέρουμε σφάλματα με πολύ μεγαλύτερη ακρίβεια.

```csharp
Target Publish => _ => _
    .DependsOn(Test)
    .Executes(() =>
    {
        try 
        {
            // Try publishing to NuGet
            DotNetNuGetPush(s => s
                .SetTargetPath(ArtifactPath)
                .SetSource("https://api.nuget.org/v3/index.json")
                .SetApiKey(ApiKey));
        }
        catch (Exception ex)
        {
            Log.Error($"Publishing failed. Team, please check whether the key is correct: {ex.Message}");
            throw; // Make sure the build process exits with a non-zero exit code
        }
    });
```

### 4. Χρησιμοποιήστε δοκιμές για την προστασία της ποιότητας

Τα σενάρια κατασκευής είναι επίσης κώδικας, πράγμα που σημαίνει ότι χρειάζονται και δοκιμές. Το Nuke μας επιτρέπει να γράφουμε δοκιμές για ροές εργασιών έκδοσης, έτσι ώστε όταν αλλάζουμε τη λογική κατασκευής, να μην διακόπτουμε κατά λάθος τις υπάρχουσες διαδικασίες έκδοσης. Αυτό είναι ιδιαίτερα σημαντικό σε αγωγούς συνεχούς ολοκλήρωσης.

## Περίληψη

Με την εισαγωγή του Nuke, η διαδικασία κατασκευής του HagiCode έγινε πιο ομαλή από ποτέ. Αυτό δεν ήταν απλώς μια αντικατάσταση εργαλείου. Ήταν μια αναβάθμιση της νοοτροπίας της μηχανικής.

**Τι κερδίσαμε;**

- **Δυνατότητα συντήρησης**: διαμόρφωση ως κώδικα, σαφέστερη λογική και ταχύτερη διαδρομή ενσωμάτωσης για νέους συνεισφέροντες.
- **Σταθερότητα**: η δυνατή πληκτρολόγηση εξαλείφει περισσότερο από το 90% των λαθών χαμηλού επιπέδου.
- **Συνέπεια**: μια ενοποιημένη εμπειρία πολλαπλών πλατφορμών που εξαλείφει τη μετατόπιση του περιβάλλοντος.

Εάν η σύνταξη σεναρίων κατασκευής είχε την αίσθηση ότι «να νιώθεις το δρόμο σου μέσα στο σκοτάδι», τότε η χρήση του Nuke είναι σαν να «περπατάς τη νύχτα με τα φώτα αναμμένα».
