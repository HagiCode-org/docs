## Από 120 δευτερόλεπτα έως 45 δευτερόλεπτα: Ένας πρακτικός οδηγός για τη βελτιστοποίηση της απόδοσης του Vite Build με τα νήματα εργασίας

> Όταν εργάζεστε με μεγάλα έργα frontend, οι εκδόσεις παραγωγής μπορεί να είναι οδυνηρά αργές. Αυτό το άρθρο μοιράζεται τον τρόπο με τον οποίο χρησιμοποιήσαμε το Node.js Worker Threads για να μειώσουμε το στάδιο συσκότισης σε μια έκδοση Vite από 120 δευτερόλεπτα σε 45 δευτερόλεπτα, μαζί με τις λεπτομέρειες υλοποίησης και τα διδάγματα που αντλήθηκαν από το έργο HagiCode.

## φόντο

Στην πρακτική μας στη μηχανική frontend, τα ζητήματα απόδοσης κατασκευής έγιναν όλο και πιο εμφανή καθώς το έργο μεγάλωνε. Συγκεκριμένα, κατά τη διαδικασία δημιουργίας παραγωγής, συνήθως εισάγουμε εργαλεία συσκότισης JavaScript όπως π.χ `javascript-obfuscator` για την προστασία της λογικής του πηγαίου κώδικα. Αυτό το βήμα είναι απαραίτητο, αλλά είναι επίσης υπολογιστικά ακριβό και πολύ δεσμευμένο στην CPU.

Κατά τη διάρκεια του πρώιμου σταδίου ανάπτυξης του **HagiCode**, αντιμετωπίσαμε ένα πολύ δύσκολο πρόβλημα απόδοσης: οι χρόνοι κατασκευής της παραγωγής επιδεινώθηκαν γρήγορα καθώς η βάση κώδικα μεγάλωνε.

**Τα συγκεκριμένα σημεία πόνου ήταν**:
- Οι εργασίες συσκότισης εκτελούνταν σειριακά σε ένα μόνο νήμα, μεγιστοποιώντας έναν πυρήνα CPU ενώ οι άλλοι ήταν σε αδράνεια
- Ο χρόνος κατασκευής αυξήθηκε από τα 30 δευτερόλεπτα στα 110-120 δευτερόλεπτα
- Ο βρόχος επαλήθευσης κατασκευής μετά την αλλαγή έγινε εξαιρετικά μεγάλος, επιβραδύνοντας σοβαρά την επανάληψη της ανάπτυξης
- Στον αγωγό CI/CD, το στάδιο κατασκευής έγινε το πιο χρονοβόρο μέρος

**Γιατί το χρειαζόταν αυτό το HagiCode;**

Για να διασφαλίσουμε την ασφάλεια του βασικού μας κώδικα, επιβάλαμε συσκότιση υψηλής έντασης στις εκδόσεις παραγωγής. Αντιμέτωποι με αναμονή κατασκευής που πλησιάζει τα δύο λεπτά, αποφασίσαμε να πραγματοποιήσουμε μια βαθιά βελτιστοποίηση απόδοσης του συστήματος κατασκευής.

## Ανάλυση: Εύρεση του σημείου πρόοδος στο σημείο συμφόρησης απόδοσης

Πριν λύσουμε το ζήτημα της απόδοσης, έπρεπε πρώτα να ξεκαθαρίσουμε τη σκέψη μας και να εντοπίσουμε την καλύτερη τεχνική λύση.

### Βασική απόφαση: Γιατί να επιλέξετε τα νήματα των εργαζομένων;

Υπάρχουν τρεις κύριοι τρόποι για να επιτευχθεί παράλληλος υπολογισμός στο Node.js:
1. **child_process**: δημιουργία ανεξάρτητων θυγατρικών διεργασιών
2. **Web Workers**: χρησιμοποιείται κυρίως από την πλευρά του προγράμματος περιήγησης
3. **worker_threads**: εγγενής υποστήριξη πολλαπλών νημάτων στο Node.js

Μετά τη σύγκριση των επιλογών, το HagiCode επέλεξε τελικά τα **Worker Threads** για τους ακόλουθους λόγους:
- **Επιβάρυνση μηδενικής σειριοποίησης**: Τα νήματα εργασίας εκτελούνται με την ίδια διαδικασία και μπορούν να μοιράζονται τη μνήμη μέσω `SharedArrayBuffer` ή μεταβίβαση ιδιοκτησίας, αποφεύγοντας το βαρύ κόστος σειριοποίησης της επικοινωνίας μεταξύ των διεργασιών.
- **Εγγενής υποστήριξη**: ενσωματωμένη στο Node.js 12+ χωρίς να χρειάζονται επιπλέον βαριές εξαρτήσεις.
- **Ενοποιημένο πλαίσιο**: ο εντοπισμός σφαλμάτων και η καταγραφή είναι πιο βολικές από ό,τι με τις θυγατρικές διαδικασίες.

### Ευκρίνεια εργασιών: Πώς πρέπει να διαχωρίζονται οι εργασίες συσκότισης;

Είναι δύσκολο να παραλληλίσουμε τη συσκότιση ενός τεράστιου αρχείου δέσμης JS επειδή ο κώδικας έχει εξαρτήσεις, αλλά η έξοδος έκδοσης Vite αποτελείται από πολλά **κομμάτια**. Αυτό μας δίνει ένα φυσικό παράλληλο όριο:

- **Ανεξαρτησία**: μετά τη συσκευασία Vite, οι εξαρτήσεις μεταξύ διαφορετικών κομματιών έχουν ήδη αποσυνδεθεί, έτσι ώστε να μπορούν να υποβάλλονται σε επεξεργασία με ασφάλεια παράλληλα.
- **Κατάλληλη ευαισθησία**: τα έργα έχουν συνήθως 10-30 κομμάτια, που είναι μια εξαιρετική κλίμακα για παράλληλο προγραμματισμό.
- **Εύκολη ενσωμάτωση**: το `generateBundle` Τα πρόσθετα hook in Vite μας επιτρέπουν να υποκλέψουμε και να επεξεργαστούμε αυτά τα κομμάτια πριν από την εκπομπή των αρχείων.

### Αρχιτεκτονικός Σχεδιασμός

Σχεδιάσαμε ένα σύστημα παράλληλης επεξεργασίας με τέσσερα βασικά στοιχεία:

1. **Διαχωριστής εργασιών**: επαναλαμβάνεται πάνω από το αντικείμενο δέσμης του Vite, φιλτράρει αρχεία που δεν χρειάζονται συσκότιση, όπως κομμάτια προμηθευτή, και δημιουργεί μια ουρά εργασιών.
2. **Worker Pool Manager**: διαχειρίζεται τον κύκλο ζωής του Worker και χειρίζεται τη διανομή εργασιών, την ανακύκλωση και την επανάληψη σε περίπτωση αποτυχίας.
3. **Progress Reporter**: τα αποτελέσματα δημιουργούν πρόοδο σε πραγματικό χρόνο για να μειώσουν το άγχος αναμονής.
4. **ObfuscationWorker**: το νήμα Worker που στην πραγματικότητα εκτελεί τη λογική της συσκότισης.

## Λύση: Πρόχειρη εφαρμογή

Με βάση την παραπάνω ανάλυση, ξεκινήσαμε την εφαρμογή αυτού του παράλληλου συστήματος συσκότισης.

### 1. Διαμορφώστε το πρόσθετο Vite

Αρχικά, ενσωματώσαμε το πρόσθετο παράλληλης συσκότισης `vite.config.ts`. Η διαμόρφωση είναι απλή. Χρειάζεται μόνο να καθορίσετε τον αριθμό των Εργαζομένων και τους κανόνες συσκότισης.

```typescript
import { defineConfig } from 'vite'
import { parallelJavascriptObfuscator } from './buildTools/plugin'

export default defineConfig(({ mode }) => {
  const isProduction = mode === 'production'
  
  return {
    build: {
      rollupOptions: {
        ...(isProduction
          ? {
              plugins: [
                parallelJavascriptObfuscator({
                  enabled: true,
                  // Automatically adjust based on CPU core count; leave one core for the main thread
                  workerCount: 4, 
                  retryAttempts: 3,
                  fallbackToMainThread: true, // Automatically degrade to single-thread mode on failure
                  // Filter out vendor chunks; third-party libraries usually do not need obfuscation
                  isVendorChunk: (fileName: string) => fileName.includes('vendor-'),
                  obfuscationConfig: {
                    compact: true,
                    controlFlowFlattening: true,
                    deadCodeInjection: true,
                    disableConsoleOutput: true,
                    // ... more obfuscation options
                  },
                }),
              ],
            }
          : {}),
      },
    },
  }
})
```

### 2. Εφαρμόστε την Εργατική Λογική

Εργάτης είναι η μονάδα που εκτελεί εργασίες. Πρέπει να ορίσουμε με σαφήνεια τις δομές δεδομένων εισόδου και εξόδου.

**Σημείωση**: αν και ο κώδικας εδώ είναι απλός, υπάρχουν πολλές παγίδες που πρέπει να προσέξετε, όπως να ελέγξετε αν `parentPort` είναι μηδενικό και χειρίζεται σωστά τα σφάλματα. Στην υλοποίηση του HagiCode, διαπιστώσαμε ότι ορισμένα ειδικά μοτίβα σύνταξης ES6 θα μπορούσαν να προκαλέσουν συντριβή του επισκιαστή, οπότε προσθέσαμε `try-catch` προστασία.

```typescript
import { parentPort } from 'worker_threads'
import javascriptObfuscator from 'javascript-obfuscator'

export interface ObfuscationTask {
  chunkId: string
  code: string
  config: any
}

export interface ObfuscationResult {
  chunkId: string
  obfuscatedCode: string
  error?: string
}

// Listen for tasks sent from the main thread
if (parentPort) {
  parentPort.on('message', async (task: ObfuscationTask) => {
    try {
      // Perform obfuscation
      const obfuscated = javascriptObfuscator.obfuscate(task.code, task.config)
      const result: ObfuscationResult = {
        chunkId: task.chunkId,
        obfuscatedCode: obfuscated.getObfuscatedCode(),
      }
      // Send the result back to the main thread
      parentPort?.postMessage(result)
    } catch (error) {
      // Handle exceptions so one Worker crash does not block the whole build
      const result: ObfuscationResult = {
        chunkId: task.chunkId,
        obfuscatedCode: '',
        error: error instanceof Error ? error.message : 'Unknown error',
      }
      parentPort?.postMessage(result)
    }
  })
}
```

### 3. Worker Pool Manager

Αυτός είναι ο πυρήνας της όλης λύσης. Πρέπει να διατηρήσουμε μια ομάδα Εργαζομένων σταθερού μεγέθους και να προγραμματίσουμε εργασίες χρησιμοποιώντας μια στρατηγική **FIFO (first in, first out)**.

```typescript
import { Worker } from 'worker_threads'
import os from 'os'

export class WorkerPool {
  private workers: Worker[] = []
  private taskQueue: Array<{
    task: ObfuscationTask
    resolve: (result: ObfuscationResult) => void
    reject: (error: Error) => void
  }> = []
  
  constructor(options: WorkerPoolOptions = {}) {
    // Default to core count - 1 so the main thread still has some breathing room
    const workerCount = options.workerCount ?? Math.max(1, (os.cpus().length || 4) - 1)
    
    for (let i = 0; i < workerCount; i++) {
      this.createWorker()
    }
  }

  private createWorker() {
    const worker = new Worker('./worker.ts')
    
    worker.on('message', (result) => {
      // After one task finishes, take the next task from the queue
      const nextTask = this.taskQueue.shift()
      if (nextTask) {
        this.dispatchTask(worker, nextTask)
      } else {
        // If there are no pending tasks, mark the Worker as idle
        this.activeWorkers.delete(worker)
      }
    })
    
    this.workers.push(worker)
  }

  // Submit a task to the pool
  public runTask(task: ObfuscationTask): Promise<ObfuscationResult> {
    return new Promise((resolve, reject) => {
      const job = { task, resolve, reject }
      const idleWorker = this.workers.find(w => !this.activeWorkers.has(w))
      
      if (idleWorker) {
        this.dispatchTask(idleWorker, job)
      } else {
        this.taskQueue.push(job)
      }
    })
  }

  private dispatchTask(worker: Worker, job: any) {
    this.activeWorkers.set(worker, job.task)
    worker.postMessage(job.task)
  }
}
```

### 4. Αναφορά προόδου

Η αναμονή είναι επώδυνη, ειδικά όταν δεν έχεις ιδέα πόσο περισσότερο θα πάρει. Έτσι, προσθέσαμε έναν απλό ανταποκριτή προόδου για την παροχή σχολίων σε πραγματικό χρόνο σχετικά με την τρέχουσα κατάσταση.

```typescript
export class ProgressReporter {
  private completed = 0
  private readonly total: number
  private readonly startTime: number

  constructor(total: number) {
    this.total = total
    this.startTime = Date.now()
  }

  increment(): void {
    this.completed++
    this.report()
  }

  private report(): void {
    const now = Date.now()
    const elapsed = now - this.startTime
    const percentage = (this.completed / this.total) * 100
    
    // Simple ETA estimate
    const avgTimePerChunk = elapsed / this.completed
    const remaining = (this.total - this.completed) * avgTimePerChunk

    console.log(
      `[Parallel Obfuscation] ${this.completed}/${this.total} chunks completed (${percentage.toFixed(1)}%) | ETA: ${(remaining / 1000).toFixed(1)}s`
    )
  }
}
```

## Εξάσκηση: Αποτελέσματα και διδάγματα

Μετά την ανάπτυξη αυτής της λύσης, η απόδοση κατασκευής του έργου HagiCode βελτιώθηκε αμέσως.

### Δεδομένα συγκριτικής αξιολόγησης απόδοσης

Δοκιμάσαμε στο ακόλουθο περιβάλλον:
- CPU: Intel Core i7-12700K (12 πυρήνες / 20 νήματα)
- RAM: 32 GB DDR4
- Node.js: v18.17.0
- ΛΣ: Ubuntu 22.04

**Σύγκριση αποτελεσμάτων**:
- **Μονής κλωστή (πριν από τη βελτιστοποίηση)**: 118 δευτερόλεπτα
- **4 Εργαζόμενοι**: 55 δευτερόλεπτα (**53%** βελτίωση)
- **8 Εργαζόμενοι**: 48 δευτερόλεπτα (**60%** βελτίωση)
- **12 Εργαζόμενοι**: 45 δευτερόλεπτα (**62%** βελτίωση)

Όπως μπορείτε να δείτε, τα κέρδη δεν ήταν γραμμικά. Μόλις ο αριθμός Εργατών ξεπέρασε το 8, η βελτίωση έγινε μικρότερη. Αυτό περιορίστηκε κυρίως από την ομοιόμορφη κατανομή εργασιών και τα σημεία συμφόρησης στο εύρος ζώνης μνήμης.

### Κοινά Θέματα και Λύσεις

Στη χρήση του HagiCode στον πραγματικό κόσμο, συναντήσαμε επίσης πολλές παγίδες, επομένως εδώ είναι για αναφορά:

** Ε1: Ο χρόνος κατασκευής δεν μειώθηκε πολύ και έγινε ακόμη πιο αργός;**
- **Λόγος**: η δημιουργία Workers έχει τα δικά της έξοδα ή διαμορφώθηκαν πάρα πολλοί Workers, προκαλώντας συχνή εναλλαγή περιβάλλοντος.
- **Λύση**: συνιστούμε να ορίσετε τον αριθμό Εργαζομένων σε `CPU core count - 1`. Ελέγξτε επίσης εάν κάποιο μεμονωμένο κομμάτι είναι ιδιαίτερα μεγάλο, για παράδειγμα `> 5MB`. Αυτό το είδος αρχείου "τέρας" θα γίνει το σημείο συμφόρησης, επομένως ίσως χρειαστεί να βελτιστοποιήσετε τη στρατηγική διαχωρισμού κώδικα.

**Ε2: Οι εργαζόμενοι περιστασιακά συντρίβονται και προκαλούν αστοχίες κατασκευής;**
- **Λόγος**: ορισμένα ειδικά μοτίβα σύνταξης κώδικα ενδέχεται να προκαλέσουν εσωτερικά σφάλματα στο εσωτερικό του συσκότισης.
- **Λύση**: εφαρμόσαμε έναν **αυτόματο μηχανισμό υποβάθμισης**. Όταν ένας Worker φτάσει το όριο αποτυχίας, το πρόσθετο επανέρχεται αυτόματα σε λειτουργία μονού νήματος για να διασφαλίσει ότι η κατασκευή δεν θα σταματήσει. Ταυτόχρονα, καταγράφει το όνομα αρχείου που προκάλεσε το σφάλμα, ώστε να μπορεί να διορθωθεί αργότερα.

**Ε3: Η χρήση μνήμης είναι πολύ υψηλή (OOM);**
- **Λόγος**: κάθε Worker χρειάζεται το δικό του χώρο μνήμης για να φορτώσει τον επισκιαστή και να αναλύσει το AST.
- **Λύση**:
  - Μειώστε τον αριθμό των εργαζομένων.
  - Αυξήστε το όριο μνήμης Node.js: `NODE_OPTIONS="--max-old-space-size=4096" npm run build`.
  - Βεβαιωθείτε ότι οι εργαζόμενοι δεν κρατούν περιττές αναφορές σε μεγάλα αντικείμενα.

Ο πυρήνας αυτής της λύσης είναι:
1. **Διαχωρίστε σωστά τις εργασίες**: χρησιμοποιήστε κομμάτια Vite ως παράλληλη μονάδα.

3. **Σχεδιασμός για ανοχή σφαλμάτων**: ένας αυτόματος μηχανισμός υποβάθμισης εξασφαλίζει σταθερότητα κατασκευής.

Εάν δυσκολεύεστε επίσης με την αποτελεσματικότητα της κατασκευής του frontend ή το έργο σας εκτελεί επίσης βαριά επεξεργασία κώδικα, αυτή η λύση αξίζει να δοκιμάσετε. Φυσικά, θα συνιστούσαμε να ρίξετε μια άμεση ματιά στο HagiCode, όπου αυτές οι μηχανικές λεπτομέρειες είναι ήδη ενσωματωμένες.
