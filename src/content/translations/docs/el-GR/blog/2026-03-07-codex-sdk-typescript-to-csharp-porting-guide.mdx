## Από το TypeScript στο C#: Πρακτική μεταφοράς μεταξύ των γλωσσών για το Codex SDK

> Με απλά λόγια, αυτό το άρθρο είναι επίσης ένα μικρό δικό μας: καταγράφει την πλήρη διαδικασία μεταφοράς του επίσημου TypeScript Codex SDK σε C#. Το να το αποκαλούμε "λιμάνι" σχεδόν το κάνει να ακούγεται πολύ εύκολο - ήταν περισσότερο σαν μια μακρά περιπέτεια, γιατί αυτές οι δύο γλώσσες έχουν πολύ διαφορετικές προσωπικότητες και έπρεπε να βρούμε έναν τρόπο να τις κάνουμε να συνεργαστούν.

## φόντο

Το Codex είναι το εργαλείο AI Agent CLI που κυκλοφόρησε από την OpenAI και είναι πραγματικά ισχυρό. Η επίσημη ομάδα παρέχει ένα TypeScript SDK στο `@openai/codex` πακέτο. Αλληλεπιδρά με το Codex CLI καλώντας το `codex exec --experimental-json` εντολή και ανάλυση μιας ροής συμβάντων JSONL.

Το πρόβλημα είναι ότι στο έργο HagiCode, πρέπει να το χρησιμοποιήσουμε σε καθαρό περιβάλλον .NET - ειδικά σε υπηρεσίες υποστήριξης C# και εφαρμογές επιφάνειας εργασίας. Δεν μπορούσαμε εύλογα να εισαγάγουμε έναν χρόνο εκτέλεσης Node.js σε ένα έργο .NET απλώς για να καλέσουμε ένα εργαλείο CLI. Αυτό θα ήταν πολύ δυσκίνητο.

Έτσι, είχαμε δύο επιλογές: να διατηρήσουμε ένα σύνθετο επίπεδο γέφυρας Node.js ή να δημιουργήσουμε μόνοι μας ένα εγγενές C# SDK.

Εμείς επιλέξαμε το δεύτερο.

## Βασικό Περιεχόμενο

### Σύγκριση Αρχιτεκτονικού Σχεδιασμού

Πριν μεταφράσουμε τον κώδικα ένα προς ένα, έπρεπε πρώτα να κατανοήσουμε τον αρχιτεκτονικό σχεδιασμό και των δύο SDK. Πρέπει να κατανοήσετε και τις δύο πλευρές για να μπορέσετε να τις μεταφέρετε καλά.

Η βασική αρχιτεκτονική του TypeScript SDK μοιάζει με αυτό:

```
Codex (entry class)
  └── CodexExec (executor, manages child processes)
      └── Thread (conversation thread)
          ├── run() / runStreamed() (synchronous/asynchronous execution)
          └── event stream parsing
```

Το C# SDK διατηρεί την ίδια αρχιτεκτονική στρώση, αλλά προσαρμόζει τις λεπτομέρειες υλοποίησης. Η γενική ιδέα είναι απλή: διατηρήστε τη συνοχή του API ενώ αξιοποιείτε πλήρως τα χαρακτηριστικά της γλώσσας C# στην υλοποίηση.

### Τύπος Μετατροπή συστήματος

Αυτό είναι το πιο θεμελιώδες και επίσης το πιο σημαντικό μέρος της εργασίας. Αν τα θεμέλια είναι αδύναμα, όλα όσα ακολουθούν γίνονται πιο σκληρά.

Το σύστημα τύπων του TypeScript είναι πιο ευέλικτο από το C#, και αυτό είναι απλώς γεγονός. Χρειαζόταν να βρούμε μια κατάλληλη στρατηγική χαρτογράφησης:

| TypeScript | C# | Σημειώσεις |
|------------|-----|------|
| `interface` / `type` | `record` | Χρήσεις C# `record` για αμετάβλητες δομές δεδομένων |
| `string \| μηδενικό». | `string?` | Μηδενικός τύπος αναφοράς |
| "boolean"| απροσδιόριστο». | `bool?` | Nullable Boolean |
| `AsyncGenerator` | `IAsyncEnumerable` | Ασύγχρονος επαναλήπτης |

Το **σύστημα τύπου συμβάντος** είναι χαρακτηριστικό παράδειγμα. Το TypeScript χρησιμοποιεί τύπους ένωσης για να ορίσει συμβάντα:

```typescript
export type ThreadEvent =
  | ThreadStartedEvent
  | TurnStartedEvent
  | TurnCompletedEvent
  | ...
```

Στην C#, χρησιμοποιούμε μια ιεραρχία κληρονομικότητας και αντιστοίχιση προτύπων για να επιτύχουμε ένα παρόμοιο αποτέλεσμα:

```csharp
public abstract record ThreadEvent(string Type);

public sealed record ThreadStartedEvent(string ThreadId) : ThreadEvent("thread.started");
public sealed record TurnStartedEvent() : ThreadEvent("turn.started");
public sealed record TurnCompletedEvent(Usage Usage) : ThreadEvent("turn.completed");
// ...
```

Εμείς επιλέξαμε `record` αντί για `class` επειδή τα αντικείμενα συμβάντων θα πρέπει να είναι αμετάβλητα, κάτι που ταιριάζει με την πρόθεση πίσω από τη χρήση απλών αντικειμένων στο TypeScript. Το `sealed` Η λέξη-κλειδί αποτρέπει επίσης την πρόσθετη κληρονομικότητα και δίνει στον μεταγλωττιστή χώρο για βελτιστοποίηση.

### Βασικά σημεία μεταφοράς

#### 1. Αναλυτής συμβάντων

Η ανάλυση συμβάντων είναι ο πυρήνας ολόκληρου του SDK, επειδή καθορίζει εάν μπορούμε να κατανοήσουμε σωστά κάθε μήνυμα που επιστρέφεται από το Codex CLI. Εάν η ανάλυση είναι λάθος, όλα μετά από αυτό είναι χαμένη προσπάθεια.

Η έκδοση TypeScript χρησιμοποιεί `JSON.parse()` για να αναλύσετε κάθε γραμμή του JSON:

```typescript
export function parseEvent(line: string): ThreadEvent {
  const data = JSON.parse(line);
  // Handle different event types...
}
```

Η έκδοση C# χρησιμοποιεί `System.Text.Json.JsonDocument` αντί για:

```csharp
public static ThreadEvent Parse(string line)
{
    using var document = JsonDocument.Parse(line);
    var root = document.RootElement;
    var type = GetRequiredString(root, "type", "event.type");

    return type switch
    {
        "thread.started" => new ThreadStartedEvent(GetRequiredString(root, "thread_id", ...)),
        "turn.started" => new TurnStartedEvent(),
        "turn.completed" => new TurnCompletedEvent(ParseUsage(...)),
        // ...
        _ => new UnknownThreadEvent(type, root.Clone()),
    };
}
```

Υπάρχει ένα μικρό αλλά σημαντικό κόλπο εδώ: `root.Clone()` απαιτείται, διότι στοιχεία από `JsonDocument` καθίστανται άκυρα μετά την απόρριψη του εγγράφου. Πρέπει να διατηρήσουμε ένα αντίγραφο για άγνωστους τύπους συμβάντων. Αυτή είναι απλώς μια από τις διαφορές μεταξύ χειρισμού C# JSON και JavaScript.

#### 2. Διαφορές διαχείρισης διαδικασιών

Εδώ διαφέρουν περισσότερο τα δύο SDK. Το Node.js και το .NET έχουν διαφορετικές συμβάσεις χρόνου εκτέλεσης, επομένως η υλοποίηση πρέπει να προσαρμοστεί.

Το TypeScript χρησιμοποιεί Node.js's `spawn()` λειτουργία:

```typescript
const child = spawn(this.executablePath, commandArgs, { env, signal });
```

Το C# χρησιμοποιεί .NET's `System.Diagnostics.Process`:

```csharp
using var process = new Process { StartInfo = startInfo };
process.Start();

// stdin/stdout/stderr must be managed manually
```

Πιο συγκεκριμένα, η έκδοση C# πρέπει να διαμορφώσει τη διαδικασία ως εξής:

```csharp
var startInfo = new ProcessStartInfo
{
    FileName = _executablePath,
    RedirectStandardInput = true,
    RedirectStandardOutput = true,
    RedirectStandardError = true,
    UseShellExecute = false,
    CreateNoWindow = true,
};
```

Η μεγαλύτερη διαφορά είναι ο μηχανισμός ακύρωσης. Χρήσεις TypeScript `AbortSignal`, το οποίο είναι μέρος του Web API και είναι πολύ βολικό για εργασία με:

```typescript
const child = spawn(cmd, args, { signal: cancellationSignal });
```

Χρήσεις C# `CancellationToken` αντί για:

```csharp
public async IAsyncEnumerable<string> RunAsync(
    CodexExecArgs args,
    [EnumeratorCancellation] CancellationToken cancellationToken = default)
{
    // Check cancellation status inside the loop
    while (!cancellationToken.IsCancellationRequested)
    {
        // Process output...
    }

    // Terminate the process when cancellation is requested
    if (cancellationToken.IsCancellationRequested)
    {
        try { process.Kill(entireProcessTree: true); } catch { }
    }
}
```

Σε υψηλό επίπεδο, αυτό είναι απλώς ένα άλλο παράδειγμα της διαφοράς μεταξύ του οικοσυστήματος Web API και του οικοσυστήματος .NET.

#### 3. Διατήρηση σειριοποίησης διαμόρφωσης

Και τα δύο SDK εφαρμόζουν τη λογική που μετατρέπει τη διαμόρφωση JSON σε διαμόρφωση TOML, επειδή το Codex CLI δέχεται παρακάμψεις διαμόρφωσης σε μορφή TOML. Αυτό το τμήμα πρέπει να παραμείνει απόλυτα συνεπές, διαφορετικά η ίδια διαμόρφωση θα συμπεριφέρεται διαφορετικά στα δύο SDK.

Αυτό είναι το είδος της λεπτομέρειας που δεν μπορείτε να συμβιβαστείτε. Η επιτυχία ή η αποτυχία συχνά καταλήγουν σε λεπτομέρειες όπως αυτή.

### Στοιχεία Υλοποίησης

#### Δομή έργου

Δημιουργήσαμε την ακόλουθη δομή έργου:

```
CodexSdk/
├── CodexSdk.csproj
├── Codex.cs           # Entry class
├── CodexThread.cs     # Conversation thread
├── CodexExec.cs       # Executor
├── Events.cs          # Event type definitions
├── Items.cs           # Item type definitions
├── EventParser.cs     # Event parser
├── OutputSchemaTempFile.cs  # Temporary file management
└── ...
```

Είναι μια αρκετά καθαρή κατασκευή, και αυτό βοήθησε πολύ κατά τη διάρκεια του λιμανιού.

#### Παράδειγμα χρήσης

Η βασική χρήση παραμένει συνεπής με το TypeScript SDK:

```csharp
using CodexSdk;

// Create a Codex instance
var codex = new Codex();
var thread = codex.StartThread();

// Execute a query
var result = await thread.RunAsync("Summarize this repository.");
Console.WriteLine(result.FinalResponse);
```

Ο χειρισμός συμβάντων ροής εκμεταλλεύεται την αντιστοίχιση μοτίβων C#:

```csharp
await foreach (var @event in thread.RunStreamedAsync("Analyze the code."))
{
    switch (@event)
    {
        case ItemCompletedEvent itemCompleted
            when itemCompleted.Item is AgentMessageItem msg:
            Console.WriteLine($"Assistant: {msg.Text}");
            break;
        case TurnCompletedEvent completed:
            Console.WriteLine($"Tokens: in={completed.Usage.InputTokens}");
            break;
        case CommandExecutionItem command:
            Console.WriteLine($"Command: {command.Command}");
            break;
    }
}
```

### Σημειώσεις

Κατά την υλοποίηση, συγκεντρώσαμε αρκετά πρακτικά μαθήματα:

1. **Διαχείριση διαδικασίας**: Η έκδοση C# πρέπει να διαχειρίζεται τον πλήρη κύκλο ζωής της διαδικασίας με μη αυτόματο τρόπο, συμπεριλαμβανομένου του τερματισμού της διαδικασίας κατά την ακύρωση. Χρήση `Kill(entireProcessTree: true)` για να βεβαιωθείτε ότι καθαρίζονται και οι παιδικές διαδικασίες.

2. **Διαχείριση σφαλμάτων**: Χρησιμοποιούμε `InvalidOperationException` για τη δημιουργία σφαλμάτων ανάλυσης, διατηρώντας το στυλ χειρισμού σφαλμάτων παρόμοιο με το TypeScript SDK.

3. **Εκκαθάριση πόρων**: `OutputSchemaTempFile` υλοποιεί `IAsyncDisposable` για να διασφαλίσετε ότι τα προσωρινά αρχεία καθαρίζονται σωστά.

4. **Μεταβλητές περιβάλλοντος**: Η έκδοση C# υποστηρίζει πλήρως υπερισχύουσες μεταβλητές περιβάλλοντος διεργασίας μέσω `CodexOptions.Env`. Είναι ένα μικρό χαρακτηριστικό, αλλά πολύ πρακτικό.

5. **Διαφορές πλατφόρμας**: Η έκδοση C# δεν περιλαμβάνει τη λογική της έκδοσης TypeScript για τον αυτόματο εντοπισμό δυαδικών αρχείων μέσα σε πακέτα npm. Δεδομένου ότι τα έργα .NET συνήθως δεν εξαρτώνται από npm, η διαδρομή προς το `codex` εκτελέσιμο πρέπει να καθοριστεί μέσω του `CODEX_EXECUTABLE` μεταβλητή περιβάλλοντος ή `CodexPathOverride`.

## Συμπέρασμα

Η μεταφορά ενός ώριμου TypeScript SDK σε C# δεν είναι απλώς θέμα μετατροπής σύνταξης - απαιτεί επίσης κατανόηση των φιλοσοφιών σχεδιασμού και των δύο γλωσσών. Η ευελιξία του TypeScript και τα χαρακτηριστικά του οικοσυστήματος JavaScript όπως π.χ `AbortSignal` χρειάζονται κατάλληλα αντίστοιχα σε C#.

Η βασική λύση είναι η εξής: **η διατήρηση της συνέπειας του API έχει μεγαλύτερη σημασία από τη διατήρηση της συνέπειας σε επίπεδο υλοποίησης**. Οι χρήστες ενδιαφέρονται για το αν η διεπαφή είναι εύκολη στη χρήση, όχι εάν η εσωτερική υλοποίηση είναι πανομοιότυπη. Αυτό ακούγεται απλό, αλλά το να κάνεις αυτούς τους συμβιβασμούς χρειάζεται κρίση.

Εάν εργάζεστε σε μια παρόμοια θύρα πολλαπλών γλωσσών, η εμπειρία μας είναι να κατανοήσουμε πλήρως την αρχική αρχιτεκτονική SDK πρώτα, στη συνέχεια να τη μεταφράσουμε ενότητα προς ενότητα και, τέλος, να χρησιμοποιήσουμε μια πλήρη σουίτα δοκιμών για να εξασφαλίσουμε συνέπεια συμπεριφοράς. Αυτού του είδους η δουλειά δεν μπορεί να γίνει βιαστική.
