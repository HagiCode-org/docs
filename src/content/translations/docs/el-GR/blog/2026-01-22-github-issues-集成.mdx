## Η οικοδόμηση ενός GitHub Θέματα ενσωμάτωσης από την αρχή: Πρακτική απευθείας σύνδεσης στο Frontend της HagiCode

> Αυτό το άρθρο τεκμηριώνει την πλήρη διαδικασία ενσωμάτωσης ζητημάτων GitHub στην πλατφόρμα HagiCode. Θα διερευνήσουμε πώς να χρησιμοποιήσετε μια αρχιτεκτονική "απευθείας σύνδεση διεπαφής + ελάχιστο backend" για να επιτύχετε ασφαλή έλεγχο ταυτότητας OAuth και αποτελεσματικό συγχρονισμό προβλημάτων, διατηρώντας παράλληλα ελαφρύ το backend.

## Ιστορικό: Γιατί να ενσωματώσουμε το GitHub;

Ως πλατφόρμα ανάπτυξης υποβοηθούμενη από AI, η βασική αξία του HagiCode έγκειται στη σύνδεση των ιδεών με την υλοποίηση. Αλλά στην πραγματική χρήση, διαπιστώσαμε ότι αφού οι χρήστες ολοκληρώσουν μια πρόταση στο HagiCode, συχνά χρειάζεται να αντιγράψουν με μη αυτόματο τρόπο το περιεχόμενο στα ζητήματα GitHub για παρακολούθηση έργου.

Αυτό δημιουργεί πολλά εμφανή σημεία πόνου:

1.  **Κατακερματισμένη ροή εργασίας**: Οι χρήστες πρέπει να κάνουν εναλλαγή μεταξύ δύο συστημάτων. Η εμπειρία δεν είναι ομαλή και οι βασικές πληροφορίες μπορούν εύκολα να χαθούν κατά την αντιγραφή και επικόλληση.
2.  **Αβολική συνεργασία**: Τα άλλα μέλη της ομάδας έχουν συνηθίσει να ελέγχουν εργασίες στο GitHub και δεν μπορούν να δουν απευθείας την πρόοδο της πρότασης μέσα στο HagiCode.
3.  **Επαναλαμβανόμενη χειρωνακτική εργασία**: Κάθε φορά που ενημερώνεται μια πρόταση, κάποιος πρέπει να ενημερώνει μη αυτόματα το αντίστοιχο ζήτημα στο GitHub, προσθέτοντας περιττό κόστος συντήρησης.

Για να λύσουμε αυτό το πρόβλημα, αποφασίσαμε να εισαγάγουμε τη δυνατότητα **GitHub Issues Integration**, συνδέοντας συνεδρίες HagiCode με αποθετήρια GitHub για να ενεργοποιήσουμε τον "συγχρονισμό με ένα κλικ".

## Τεχνική επιλογή: Frontend Direct Connection vs Backend Proxy

Κατά το σχεδιασμό της προσέγγισης ενσωμάτωσης, είχαμε δύο επιλογές μπροστά μας: το παραδοσιακό "μοντέλο διακομιστή μεσολάβησης υποστήριξης" και το πιο επιθετικό "μοντέλο άμεσης σύνδεσης frontend".

### Σύγκριση λύσεων

Στο παραδοσιακό **μοντέλο διακομιστή μεσολάβησης υποστήριξης**, κάθε αίτημα από το frontend πρέπει πρώτα να περάσει από το backend μας, το οποίο στη συνέχεια καλεί το GitHub API. Αυτό συγκεντρώνει τη λογική, αλλά επιβαρύνει επίσης σημαντικά το backend:

1.  **Bloated backend**: Θα χρειαστεί να γράψουμε ένα αποκλειστικό περιτύλιγμα πελάτη GitHub API και επίσης να χειριστούμε τον περίπλοκο μηχάνημα κατάστασης OAuth.
2.  **Κίνδυνος Token**: Το διακριτικό GitHub του χρήστη θα πρέπει να αποθηκευτεί στη βάση δεδομένων υποστήριξης. Ακόμη και με την κρυπτογράφηση, αυτό εξακολουθεί να αυξάνει την επιφάνεια ασφαλείας.
3.  **Κόστος ανάπτυξης**: Θα χρειαζόμασταν μετεγκαταστάσεις βάσης δεδομένων για την αποθήκευση διακριτικών και μια πρόσθετη υπηρεσία συγχρονισμού για συντήρηση.

Το **μοντέλο άμεσης σύνδεσης στο frontend** είναι πολύ πιο ελαφρύ. Σε αυτήν την προσέγγιση, χρησιμοποιούμε το backend μόνο για το πιο ευαίσθητο βήμα "ανταλλαγής μυστικών" (την επιστροφή κλήσης OAuth). Αφού λάβουμε το διακριτικό, το αποθηκεύουμε απευθείας στο πρόγραμμα περιήγησης `localStorage`. Μεταγενέστερες λειτουργίες, όπως η δημιουργία προβλημάτων και η ενημέρωση σχολίων, αποστέλλονται απευθείας από το frontend στο GitHub μέσω HTTP.

| Διάσταση σύγκρισης | Μοντέλο διακομιστή μεσολάβησης υποστήριξης | Μοντέλο άμεσης σύνδεσης Frontend |
| :--- | :--- | :--- |
| **Πολυπλοκότητα backend** | Απαιτεί πλήρη υπηρεσία OAuth και πελάτη GitHub API | Χρειάζεται μόνο ένα τελικό σημείο επανάκλησης OAuth |
| **Διαχείριση διακριτικών** | Πρέπει να είναι κρυπτογραφημένο και αποθηκευμένο στη βάση δεδομένων, με κίνδυνο διαρροής | Αποθηκευμένο στο πρόγραμμα περιήγησης και ορατό μόνο στον χρήστη |
| **Κόστος υλοποίησης** | Απαιτεί μετεγκαταστάσεις βάσεων δεδομένων και ανάπτυξη πολλαπλών υπηρεσιών | Κυρίως εργασία frontend |
| **Εμπειρία χρήστη** | Κεντρική λογική, αλλά η καθυστέρηση διακομιστή μπορεί να είναι ελαφρώς υψηλότερη | Εξαιρετικά γρήγορη απόκριση με άμεση αλληλεπίδραση GitHub |

Επειδή θέλαμε γρήγορη ενοποίηση και ελάχιστες αλλαγές στο backend, **επιλέξαμε τελικά το "μοντέλο άμεσης σύνδεσης frontend"**. Είναι σαν να δίνετε στο πρόγραμμα περιήγησης ένα "προσωρινό πάσο". Μόλις λάβει το πάσο, το πρόγραμμα περιήγησης μπορεί να χειριστεί τα πράγματα στο GitHub από μόνο του χωρίς να ζητά έγκριση από τον διαχειριστή του backend κάθε φορά.

## Σχεδιασμός πυρήνα: Ροή δεδομένων και ασφάλεια

Αφού καταλήξαμε στην αρχιτεκτονική, έπρεπε να σχεδιάσουμε τη συγκεκριμένη ροή δεδομένων. Ο πυρήνας της διαδικασίας συγχρονισμού είναι πώς να αποκτήσετε το διακριτικό με ασφάλεια και να το χρησιμοποιήσετε αποτελεσματικά.

### Συνολικό διάγραμμα αρχιτεκτονικής

Ολόκληρο το σύστημα μπορεί να αναλυθεί σε τρεις ρόλους: το πρόγραμμα περιήγησης (frontend), το backend HagiCode και το GitHub.

```text
+--------------+        +--------------+        +--------------+
| Frontend     |        |   Backend    |        |   GitHub     |
| React        |        |   ASP.NET    |        |   REST API   |
|              |        |              |        |              |
|  +--------+  |        |              |        |              |
|  | OAuth  |--+--------> /callback    |        |              |
|  | Flow   |  |        |              |        |              |
|  +--------+  |        |              |        |              |
|              |        |              |        |              |
|  +--------+  |        |  +--------+  |        |  +--------+  |
|  | GitHub |  +------------> Session |  +----------> Issues |  |
|  | API    |  |        |  |Metadata|  |        |  |        |  |
|  | Direct |  |        |  +--------+  |        |  +--------+  |
|  +--------+  |        |              |        |              |
+--------------+        +--------------+        +--------------+
```

**Το βασικό σημείο είναι**: μόνο ένα μικρό βήμα στο OAuth (ανταλλαγή του κώδικα με διακριτικό) χρειάζεται να περάσει από το backend. Μετά από αυτό, η βαριά ανύψωση (δημιουργία προβλημάτων) αντιμετωπίζεται απευθείας μεταξύ του frontend και του GitHub.

### Λεπτομερής ροή δεδομένων συγχρονισμού

Όταν ο χρήστης κάνει κλικ στο κουμπί "Sync to GitHub" στο HagiCode UI, λαμβάνει χώρα μια σειρά από πολύπλοκες ενέργειες:

```text
User clicks "Sync to GitHub"
         │
         ▼
1. Frontend checks localStorage for the GitHub token
         │
         ▼
2. Format issue content (convert the Proposal into Markdown)
         │
         ▼
3. Frontend directly calls the GitHub API to create/update the issue
         │
         ▼
4. Call the HagiCode backend API to update Session.metadata (store issue URL and other info)
         │
         ▼
5. Backend broadcasts the SessionUpdated event via SignalR
         │
         ▼
6. Frontend receives the event and updates the UI to show the "Synced" state
```

### Σχεδιασμός Ασφαλείας

Η ασφάλεια είναι πάντα η κορυφαία προτεραιότητα κατά την ενσωμάτωση υπηρεσιών τρίτων. Κάναμε τις ακόλουθες σκέψεις:

1.  **Αμυνθείτε από επιθέσεις CSRF**: Δημιουργήστε ένα τυχαίο `state` παράμετρος κατά την ανακατεύθυνση OAuth και αποθηκεύστε την `sessionStorage`. Επικυρώστε αυστηρά την κατάσταση στην επιστροφή κλήσης για να αποτρέψετε πλαστά αιτήματα.
2.  **Απομονωμένη αποθήκευση διακριτικών**: Το διακριτικό αποθηκεύεται μόνο στο πρόγραμμα περιήγησης `localStorage`. Χρησιμοποιώντας την Πολιτική ίδιας προέλευσης, μόνο τα σενάρια HagiCode μπορούν να τη διαβάσουν, αποφεύγοντας τον κίνδυνο διαρροής βάσης δεδομένων από την πλευρά του διακομιστή που επηρεάζει τους χρήστες.
3.  **Όρια σφαλμάτων**: Σχεδιάσαμε αποκλειστικό χειρισμό για κοινά σφάλματα API του GitHub (όπως 401 ληγμένο διακριτικό, 422 αποτυχία επικύρωσης και 429 περιορισμός ρυθμού), ώστε οι χρήστες να λαμβάνουν φιλικά σχόλια.

### 2. Frontend OAuth Flow

Αυτό είναι το σημείο εισόδου της σύνδεσης. Χρησιμοποιούμε την τυπική ροή κώδικα εξουσιοδότησης.

```typescript
// src/HagiCode.Client/src/services/githubOAuth.ts

// Generate the authorization URL and redirect
export async function generateAuthUrl(): Promise<string> {
  const state = generateRandomString(); // Generate a random string for CSRF protection
  sessionStorage.setItem('hagicode_github_state', state);
  
  const params = new URLSearchParams({
    client_id: clientId,
    redirect_uri: window.location.origin + '/settings?tab=github&oauth=callback',
    scope: ['repo', 'public_repo'].join(' '),
    state: state,
  });
  
  return `https://github.com/login/oauth/authorize?${params.toString()}`;
}

// Handle the code-to-token exchange on the callback page
export async function exchangeCodeForToken(code: string, state: string): Promise<GitHubToken> {
  // 1. Validate state to prevent CSRF
  const savedState = sessionStorage.getItem('hagicode_github_state');
  if (state !== savedState) throw new Error('Invalid state parameter');

  // 2. Call the backend API to exchange the token
  // Note: this must go through the backend because ClientSecret cannot be exposed to the frontend
  const response = await fetch('/api/GitHubOAuth/callback', {
    method: 'POST',
    headers: { 'Content-Type': 'application/json' },
    body: JSON.stringify({ code, state, redirectUri: window.location.origin + '/settings?tab=github&oauth=callback' }),
  });

  if (!response.ok) throw new Error('Failed to exchange token');
  
  const token = await response.json();
  
  // 3. Save into LocalStorage
  saveToken(token);
  return token;
}
```

### 3. GitHub API Client Wrapper

Μόλις έχουμε το διακριτικό, χρειαζόμαστε ένα σταθερό εργαλείο για την κλήση του GitHub API.

```typescript
// src/HagiCode.Client/src/services/githubApiClient.ts

const GITHUB_API_BASE = 'https://api.github.com';

// Core request wrapper
async function githubApi<T>(endpoint: string, options: RequestInit = {}): Promise<T> {
  const token = localStorage.getItem('gh_token');
  if (!token) throw new Error('Not connected to GitHub');
  
  const response = await fetch(`${GITHUB_API_BASE}${endpoint}`, {
    ...options,
    headers: {
      ...options.headers,
      Authorization: `Bearer ${token}`,
      Accept: 'application/vnd.github.v3+json', // Specify the API version
    },
  });
  
  // Error handling logic
  if (!response.ok) {
    if (response.status === 401) throw new Error('GitHub token expired, please reconnect');
    if (response.status === 403) throw new Error('No permission to access this repository or rate limit exceeded');
    if (response.status === 422) throw new Error('Issue validation failed, the title may be duplicated');
    throw new Error(`GitHub API Error: ${response.statusText}`);
  }
  
  return response.json();
}

// Create issue
export async function createIssue(owner: string, repo: string, data: { title: string, body: string, labels: string[] }) {
  return githubApi(`/repos/${owner}/${repo}/issues`, {
    method: 'POST',
    body: JSON.stringify(data),
  });
}
```

### 4. Μορφοποίηση και συγχρονισμός περιεχομένου

Το τελευταίο βήμα είναι να μετατρέψετε τα δεδομένα περιόδου λειτουργίας HagiCode στη μορφή ενός ζητήματος GitHub. Είναι λίγο σαν μεταφραστική δουλειά.

```typescript
// Convert a Session object into a Markdown string
function formatIssueForSession(session: Session): string {
  let content = `# ${session.title}\n\n`;
  content += `**> HagiCode Session:** #${session.code}\n`;
  content += `**> Status:** ${session.status}\n\n`;
  content += `## Description\n\n${session.description || 'No description provided.'}\n\n`;
  
  // If this is a Proposal session, add extra fields
  if (session.type === 'proposal') {
    content += `## Chief Complaint\n\n${session.chiefComplaint || ''}\n\n`;
    // Add a deep link so users can jump back from GitHub to HagiCode
    content += `---\n\n**[View in HagiCode](hagicode://sessions/${session.id})**\n`;
  }
  
  return content;
}

// Main logic when clicking the sync button
const handleSync = async (session: Session) => {
  try {
    const repoInfo = parseRepositoryFromUrl(session.repoUrl); // Parse the repository URL
    if (!repoInfo) throw new Error('Invalid repository URL');

    toast.loading('Syncing to GitHub...');
    
    // 1. Format content
    const issueBody = formatIssueForSession(session);
    
    // 2. Call API
    const issue = await githubApiClient.createIssue(repoInfo.owner, repoInfo.repo, {
      title: `[HagiCode] ${session.title}`,
      body: issueBody,
      labels: ['hagicode', 'proposal', `status:${session.status}`],
    });
    
    // 3. Update Session Metadata (save the issue link)
    await SessionsService.patchApiSessionsSessionId(session.id, {
      metadata: {
        githubIssue: {
          owner: repoInfo.owner,
          repo: repoInfo.repo,
          issueNumber: issue.number,
          issueUrl: issue.html_url,
          syncedAt: new Date().toISOString(),
        }
      }
    });

    toast.success('Synced successfully!');
  } catch (err) {
    console.error(err);
    toast.error('Sync failed, please check your token or network');
  }
};
```

## Περίληψη και Outlook

Με αυτήν την προσέγγιση "απευθείας σύνδεσης στο frontend", επιτύχαμε απρόσκοπτη ενσωμάτωση ζητημάτων GitHub με τον λιγότερο δυνατό κώδικα υποστήριξης.

### Βασικά κέρδη
1.  **Υψηλή απόδοση ανάπτυξης**: Οι αλλαγές στο backend είναι ελάχιστες, κυρίως ένα επιπλέον πεδίο βάσης δεδομένων και ένα απλό τελικό σημείο επανάκλησης OAuth. Η περισσότερη λογική ολοκληρώνεται στο frontend.
2.  **Ισχυρή ασφάλεια**: Το διακριτικό δεν περνά από τη βάση δεδομένων του διακομιστή, μειώνοντας τον κίνδυνο διαρροής.
3.  **Εξαιρετική εμπειρία χρήστη**: Τα αιτήματα ξεκινούν απευθείας από το frontend, επομένως η ταχύτητα απόκρισης είναι γρήγορη και δεν υπάρχει ανάγκη για προώθηση backend.

### Πράγματα που πρέπει να προσέξετε
Υπάρχουν μερικές παγίδες που πρέπει να έχετε υπόψη κατά την πραγματική ανάπτυξη:
- **Ρυθμίσεις εφαρμογής OAuth**: Θυμηθείτε να εισαγάγετε τη σωστή `Authorization callback URL` στις ρυθμίσεις της εφαρμογής GitHub OAuth (συνήθως `http://localhost:3000/settings?tab=github&oauth=callback`).
- **Όρια ποσοστού**: Το GitHub API περιορίζει αρκετά αυστηρά αιτήματα χωρίς έλεγχο ταυτότητας, αλλά με ένα διακριτικό το όριο είναι συνήθως αρκετό (5000 αιτήματα/ώρα).
- **Ανάλυση URL**: Οι χρήστες εισάγουν όλα τα είδη URL αποθετηρίου, επομένως βεβαιωθείτε ότι το regex σας μπορεί να ταιριάζει `.git` επιθήματα, μορφές SSH και παρόμοιες περιπτώσεις.

### Μελλοντικές βελτιώσεις
Η τρέχουσα λειτουργία εξακολουθεί να είναι ο μονόδρομος συγχρονισμός (HagiCode -> GitHub). Στο μέλλον, σχεδιάζουμε να εφαρμόσουμε αμφίδρομο συγχρονισμό μέσω του GitHub Webhooks. Για παράδειγμα, εάν ένα ζήτημα κλείσει στο GitHub, η κατάσταση της περιόδου λειτουργίας στην πλευρά του HagiCode θα μπορούσε επίσης να ενημερώνεται αυτόματα. Αυτό θα απαιτήσει από εμάς να εκθέσουμε ένα τελικό σημείο webhook στο backend, το οποίο θα είναι ένα ενδιαφέρον επόμενο βήμα.

Ελπίζουμε αυτό το άρθρο να σας δώσει λίγη έμπνευση για τη δική σας ανάπτυξη ενοποίησης τρίτων. Εάν έχετε ερωτήσεις, μη διστάσετε να ανοίξετε ένα θέμα για συζήτηση [HagiCode GitHub](https://github.com/HagiCode-org/site).
