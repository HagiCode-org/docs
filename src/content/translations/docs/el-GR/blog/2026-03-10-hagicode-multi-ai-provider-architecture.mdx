## Πρακτική Αρχιτεκτονική Παρόχων πολλαπλών AI στην πλατφόρμα HagiCode

> Αυτό το άρθρο μοιράζεται την τεχνική προσέγγιση που χρησιμοποιήσαμε στην αρχιτεκτονική Orleans Grain για την ενοποίηση δύο εργαλείων τεχνητής νοημοσύνης, του iflow και του OpenCode, μέσω ενός ενοποιημένου `IAIProvider` διεπαφή και συγκρίνει λεπτομερώς τις διαφορές υλοποίησης μεταξύ της επικοινωνίας WebSocket και HTTP.

:::note[Χρονικό πλαίσιο]
Αυτό το άρθρο καταγράφει την πρακτική αρχιτεκτονικής παρόχου κατά τη στιγμή της δημοσίευσης. Για το τρέχον εύρος που υποστηρίζεται ενεργά, ανατρέξτε στο [Εγκατάσταση AI Agent CLI](/el-GR/related-software-installation/ai-agent-cli/) και το [Οδηγός εγκατάστασης](/el-GR/installation); `IFlowCli` διατηρείται πλέον μόνο ως τιμή συμβατότητας ιστορικού και δεν αποτελεί μέρος του τρέχοντος πεδίου ενεργής υποστήριξης.
:::

## φόντο

Δεν υπάρχει τίποτα ιδιαίτερα μυστηριώδες σε αυτό. Κατά τη δημιουργία του HagiCode, αντιμετωπίσαμε ένα πολύ πρακτικό πρόβλημα: οι χρήστες ήθελαν να εργαστούν με διαφορετικά εργαλεία AI. Αυτό δεν προκαλεί έκπληξη, αφού ο καθένας έχει τις δικές του συνήθειες. Κάποιοι προτιμούν τον Claude Code, κάποιοι αγαπούν το GitHub Copilot και ορισμένες ομάδες χρησιμοποιούν εργαλεία που ανέπτυξαν οι ίδιες.

Η αρχική μας λύση ήταν απλή και άμεση: γράψτε αποκλειστικό κώδικα ενοποίησης για κάθε εργαλείο AI. Αλλά τα μειονεκτήματα φάνηκαν γρήγορα. Η βάση κωδικών γέμισε με `if-else` υποκαταστήματα, κάθε αλλαγή απαιτούσε δοκιμές σε πολλαπλά σημεία και κάθε νέο εργαλείο σήμαινε να γράψω άλλο ένα σωρό λογικής από την αρχή.

Αργότερα, συνειδητοποίησα ότι θα ήταν καλύτερο να δημιουργήσω ένα ενιαίο `IAIProvider` διασύνδεση και αφαίρεση των δυνατοτήτων που μοιράζονται όλοι οι πάροχοι τεχνητής νοημοσύνης. Με αυτόν τον τρόπο, ανεξάρτητα από το εργαλείο που χρησιμοποιείται από κάτω, τα ανώτερα στρώματα μπορούν να το καλέσουν με τον ίδιο τρόπο.

Πρόσφατα, το έργο χρειάστηκε να ενσωματώσει δύο νέα εργαλεία: το iflow και το OpenCode. Και τα δύο υποστηρίζουν το πρωτόκολλο ACP, αλλά τα στυλ επικοινωνίας τους είναι διαφορετικά. Το iflow χρησιμοποιεί το WebSocket, ενώ το OpenCode χρησιμοποιεί ένα API HTTP. Αυτό έγινε ένα χρήσιμο αρχιτεκτονικό τεστ: προσαρμόστε δύο διαφορετικούς τρόπους μεταφοράς πίσω από μια ενοποιημένη διεπαφή.

## Σχετικά με το HagiCode

Η προσέγγιση που κοινοποιείται σε αυτό το άρθρο προέρχεται από την πρακτική εμπειρία μας στο [HagiCode](https://hagicode.com) έργο. Το HagiCode είναι μια πλατφόρμα ανάπτυξης υποβοηθούμενη από AI, βασισμένη στην αρχιτεκτονική Orleans Grain. Ενσωματώνεται με διαφορετικούς παρόχους AI μέσω ενός ενιαίου `IAIProvider` διεπαφή, επιτρέποντας στους χρήστες να επιλέγουν ευέλικτα τα εργαλεία AI που προτιμούν.

## Αρχιτεκτονικός Σχεδιασμός

### Αφαίρεση ενοποιημένης διεπαφής

Αρχικά, ορίσαμε το `IAIProvider` διεπαφή και αφαίρεσε τις δυνατότητες που χρειάζεται να εφαρμόσει κάθε πάροχος τεχνητής νοημοσύνης:

```csharp
public interface IAIProvider
{
    string Name { get; }
    bool SupportsStreaming { get; }
    ProviderCapabilities Capabilities { get; }

    Task<AIResponse> ExecuteAsync(AIRequest request, CancellationToken cancellationToken = default);
    IAsyncEnumerable<AIStreamingChunk> StreamAsync(AIRequest request, CancellationToken cancellationToken = default);
    Task<ProviderTestResult> PingAsync(CancellationToken cancellationToken = default);
    IAsyncEnumerable<AIStreamingChunk> SendMessageAsync(AIRequest request, string? embeddedCommandPrompt = null, CancellationToken cancellationToken = default);
}
```

Αυτή η διεπαφή περιλαμβάνει διάφορες βασικές μεθόδους:
- `ExecuteAsync`: εκτελέστε ένα αίτημα τεχνητής νοημοσύνης μίας λήψης
- `StreamAsync`: λάβετε απαντήσεις ροής για προβολή σε πραγματικό χρόνο
- `PingAsync`: πραγματοποιήστε έλεγχο υγείας για να επαληθεύσετε εάν ο πάροχος είναι διαθέσιμος
- `SendMessageAsync`: στείλτε ένα μήνυμα με υποστήριξη για ενσωματωμένες εντολές

### IFlowCliProvider: Μια εφαρμογή που βασίζεται σε WebSocket

Το iflow χρησιμοποιεί το WebSocket για επικοινωνία ACP. Η συνολική αρχιτεκτονική μοιάζει με αυτό:

```
IFlowCliProvider → ACPSessionManager → WebSocketAcpTransport → iflow CLI
                ↓
         Dynamic port allocation + process management
```

Η ροή του πυρήνα είναι επίσης αρκετά απλή:
1. `ACPSessionManager` δημιουργεί και διαχειρίζεται συνεδρίες ACP.
2. `WebSocketAcpTransport` χειρίζεται την επικοινωνία WebSocket.
3. Μια θύρα εκχωρείται δυναμικά και ξεκινά η διαδικασία iflow `iflow --experimental-acp --port`.
4. `IAIRequestToAcpMapper` και `IAcpToAIResponseMapper` μετατροπή αιτημάτων και απαντήσεων.

Εδώ είναι ο βασικός κώδικας:

```csharp
private async IAsyncEnumerable<AIStreamingChunk> StreamCoreAsync(
    AIRequest request,
    string? embeddedCommandPrompt,
    [EnumeratorCancellation] CancellationToken cancellationToken)
{
    // Resolve working directory
    var resolvedWorkingDirectory = ResolveWorkingDirectory(request);
    var effectiveRequest = ApplyEmbeddedCommandPrompt(request, embeddedCommandPrompt);

    // Create ACP session
    await using var session = await _sessionManager.CreateSessionAsync(
        Name,
        resolvedWorkingDirectory,
        cancellationToken,
        request.SessionId);

    // Send prompt
    var prompt = _requestMapper.ToPromptString(effectiveRequest);
    var promptResponse = await session.SendPromptAsync(prompt, cancellationToken);

    // Receive streaming response
    await foreach (var notification in session.ReceiveUpdatesAsync(cancellationToken))
    {
        if (_responseMapper.TryConvertToStreamingChunk(notification, out var chunk))
        {
            if (chunk.Type == StreamingChunkType.Metadata && chunk.IsComplete)
            {
                yield return chunk;
                yield break;
            }
            yield return chunk;
        }
    }
}
```

Υπάρχουν μερικά σημεία σχεδιασμού που αξίζει να επισημανθούν εδώ:
- Χρήση `await using` για να βεβαιωθείτε ότι η περίοδος λειτουργίας έχει κυκλοφορήσει σωστά και να αποφύγετε διαρροές πόρων.
- Επιστρέψτε τις απαντήσεις ροής μέσω `IAsyncEnumerable`, το οποίο υποστηρίζει φυσικά ασύγχρονες ροές.
- Χρήση `Metadata` κομμάτια για να προσδιορίσετε την ολοκλήρωση και να διασφαλίσετε ότι έχει ληφθεί η πλήρης απάντηση.

### OpenCodeCliProvider: Μια υλοποίηση που βασίζεται σε HTTP API

Το OpenCode παρέχει την υπηρεσία του μέσω ενός HTTP API, επομένως η αρχιτεκτονική είναι ελαφρώς διαφορετική:

```plaintext
OpenCodeCliProvider → OpenCodeRuntimeManager → OpenCodeClient → OpenCode HTTP API
                      ↓
                OpenCodeProcessManager → opencode process management
```

Ένα αξιοσημείωτο χαρακτηριστικό του OpenCode είναι ότι χρησιμοποιεί μια βάση δεδομένων SQLite για να διατηρήσει τις συνδέσεις συνεδρίας. Αυτό καθιστά δυνατή την ανάκτηση συνεδρίας και την ανάκτηση άμεσης απόκρισης:

```csharp
private async Task<OpenCodePromptExecutionResult> ExecutePromptAsync(
    AIRequest request,
    string? embeddedCommandPrompt,
    CancellationToken cancellationToken)
{
    var prompt = BuildPrompt(request, embeddedCommandPrompt);
    var resolvedWorkingDirectory = ResolveWorkingDirectory(request.WorkingDirectory);
    var client = await _runtimeManager.GetClientAsync(resolvedWorkingDirectory, cancellationToken);
    var bindingSessionId = request.SessionId;
    var boundSession = TryGetBinding(bindingSessionId, resolvedWorkingDirectory);

    // Try to use the already bound session
    if (boundSession is not null)
    {
        try
        {
            return await PromptSessionAsync(
                client,
                boundSession,
                BuildPromptRequest(request, prompt, CreatePromptMessageId()),
                request.Model ?? _settings.Model,
                cancellationToken);
        }
        catch (OpenCodeApiException ex) when (IsStaleBinding(ex))
        {
            // The session has expired, remove the binding
            RemoveBinding(bindingSessionId);
        }
    }

    // Create a new session
    var session = await client.Session.CreateAsync(new OpenCodeSessionCreateRequest
    {
        Title = BuildSessionTitle(request)
    }, cancellationToken);

    BindSession(bindingSessionId, session.Id, resolvedWorkingDirectory);
    return await PromptSessionAsync(client, session.Id, ...);
}
```

Αυτή η υλοποίηση έχει πολλά ενδιαφέροντα σημεία:
- Μηχανισμός δέσμευσης συνεδρίας: το ίδιο `SessionId` επαναχρησιμοποιεί την ίδια περίοδο λειτουργίας OpenCode, αποφεύγοντας την επαναλαμβανόμενη δημιουργία συνεδρίας.
- Χειρισμός λήξης: όταν διαπιστωθεί ότι μια περίοδος λειτουργίας έχει λήξει, το δέσιμο καθαρίζεται αυτόματα.
- Εμμονή βάσης δεδομένων: οι δεσμεύσεις αποθηκεύονται στο SQLite και παραμένουν ενεργές μετά την επανεκκίνηση.

### Συγκρίνοντας τις δύο προσεγγίσεις

| Όψη | IFlowCliProvider | OpenCodeCliProvider |
|------|-----------------|---------------------|
| Επικοινωνία | WebSocket (ACP) | HTTP API |
| Διαχείριση διαδικασίας | ACPSessionManager | OpenCodeProcessManager |
| Κατανομή λιμένων | Δυναμική θύρα | Χωρίς θύρα (χρησιμοποιεί HTTP) |
| Διαχείριση συνεδρίας | Συνεδρία ACPS | OpenCodeSession |
| Επιμονή | Προσωρινή μνήμη στη μνήμη | Βάση δεδομένων SQLite |
| Εντολή εκκίνησης | `iflow --experimental-acp --port` | `opencode` |
| Καθυστέρηση | Χαμηλότερη (μακροχρόνια σύνδεση) | Σχετικά υψηλότερο (αιτήματα HTTP) |

Ποια προσέγγιση θα επιλέξετε εξαρτάται κυρίως από τις ανάγκες σας. Το WebSocket είναι καλύτερο για σενάρια με υψηλές απαιτήσεις σε πραγματικό χρόνο, ενώ ένα HTTP API είναι απλούστερο και ευκολότερο στον εντοπισμό σφαλμάτων.

## Πρακτικός Οδηγός

### Διαμόρφωση παρόχων

Αρχικά, ενεργοποιήστε τους δύο παρόχους στο αρχείο ρυθμίσεων:

```yaml
AI:
  Providers:
    IFlowCli:
      Type: "IFlowCli"
      Enabled: true
      ExecutablePath: "iflow"
      Model: null
      WorkingDirectory: null
    OpenCodeCli:
      Type: "OpenCodeCli"
      Enabled: true
      ExecutablePath: "opencode"
      Model: "anthropic/claude-sonnet-4"
      WorkingDirectory: null

OpenCode:
  Enabled: true
  BaseUrl: "http://localhost:38376"
  ExecutablePath: "opencode"
  StartupTimeoutSeconds: 30
  RequestTimeoutSeconds: 120
```

### Χρησιμοποιήστε το IFlowCliProvider

```csharp
// Get provider through the factory
var provider = await _providerFactory.GetProviderAsync(AIProviderType.IFlowCli);

// Execute an AI request
var request = new AIRequest
{
    Prompt = "请帮我重构这个函数",
    WorkingDirectory = "/path/to/project",
    Model = "claude-sonnet-4"
};

// Get the complete response
var response = await provider.ExecuteAsync(request, cancellationToken);
Console.WriteLine(response.Content);

// Or use streaming responses
await foreach (var chunk in provider.StreamAsync(request, cancellationToken))
{
    if (chunk.Type == StreamingChunkType.ContentDelta)
    {
        Console.Write(chunk.Content);
    }
}
```

### Χρησιμοποιήστε το OpenCodeCliProvider

```csharp
// Get provider through the factory
var provider = await _providerFactory.GetProviderAsync(AIProviderType.OpenCodeCli);

var request = new AIRequest
{
    Prompt = "请帮我分析这个错误",
    WorkingDirectory = "/path/to/project",
    Model = "anthropic/claude-sonnet-4"
};

var response = await provider.ExecuteAsync(request, cancellationToken);
Console.WriteLine(response.Content);
```

### Έλεγχοι υγείας

Πριν από την εκκίνηση ή πριν από τη χρήση, μπορείτε να ελέγξετε εάν ο πάροχος είναι διαθέσιμος:

```csharp
var iflowResult = await iflowProvider.PingAsync(cancellationToken);
if (!iflowResult.Success)
{
    Console.WriteLine($"IFlow is unavailable: {iflowResult.ErrorMessage}");
    return;
}

var openCodeResult = await openCodeProvider.PingAsync(cancellationToken);
if (!openCodeResult.Success)
{
    Console.WriteLine($"OpenCode is unavailable: {openCodeResult.ErrorMessage}");
    return;
}
```

### Υποστήριξη ενσωματωμένης εντολής

Και οι δύο πάροχοι υποστηρίζουν ενσωματωμένες εντολές, όπως π.χ `/file:xxx`:

```csharp
var request = new AIRequest
{
    Prompt = "分析这个文件的问题",
    SystemMessage = "你是一个代码分析专家"
};

await foreach (var chunk in provider.SendMessageAsync(
    request,
    embeddedCommandPrompt: "/file:src/main.cs",
    cancellationToken))
{
    Console.Write(chunk.Content);
}
```

## Σημειώσεις και βέλτιστες πρακτικές

### Διαχείριση Πόρων

Το IFlow χρησιμοποιεί συνδέσεις WebSocket με μεγάλη διάρκεια ζωής, επομένως η διαχείριση πόρων αξίζει ιδιαίτερης προσοχής:
- Χρήση `await using` για να διασφαλίσετε ότι οι συνεδρίες απελευθερώνονται σωστά.
- Η ακύρωση ενεργοποιεί την εκκαθάριση της διαδικασίας.
- `ACPSessionManager` υποστηρίζει ένα μέγιστο όριο αριθμού περιόδων σύνδεσης.

Η διαχείριση της διαδικασίας OpenCode είναι σχετικά απλούστερη και `OpenCodeRuntimeManager` το χειρίζεται αυτόματα.

### Χειρισμός σφαλμάτων

Και οι δύο πάροχοι έχουν πλήρη διαχείριση σφαλμάτων:
- Τα σφάλματα IFlow διαδίδονται μέσω ενημερώσεων περιόδου λειτουργίας ACP.
- Τα σφάλματα OpenCode αποκαλύπτονται `OpenCodeApiException`.
- Συνιστάται ο καλών να πιάσει και να χειριστεί αυτές τις εξαιρέσεις.

### Θέματα απόδοσης

- Η επικοινωνία IFlow WebSocket έχει χαμηλότερο λανθάνοντα χρόνο από το HTTP.
- Η επαναχρησιμοποίηση της περιόδου λειτουργίας OpenCode μπορεί να μειώσει την επιβάρυνση των αιτημάτων HTTP.
- Ο εργοστασιακός μηχανισμός προσωρινής αποθήκευσης αποφεύγει την επανειλημμένη δημιουργία παρόχων.
- Σε σενάρια υψηλού συγχρονισμού, δώστε ιδιαίτερη προσοχή στα όρια για τον αριθμό διεργασιών και τον αριθμό των συνδέσεων.

### Επικύρωση διαμόρφωσης

Η εκτελέσιμη διαδρομή επικυρώνεται κατά την εκκίνηση, αλλά ενδέχεται να παρουσιαστούν προβλήματα χρόνου εκτέλεσης. `PingAsync` είναι ένα χρήσιμο εργαλείο για να επαληθεύσετε εάν η διαμόρφωση είναι σωστή:

```csharp
// Check at startup
var provider = await _providerFactory.GetProviderAsync(providerType);
var result = await provider.PingAsync(cancellationToken);
if (!result.Success)
{
    _logger.LogError("Provider {ProviderType} is unavailable: {Error}", providerType, result.ErrorMessage);
}
```

## Περίληψη

Αυτό το άρθρο μοιράζεται την τεχνική προσέγγιση που χρησιμοποιείται από την πλατφόρμα HagiCode κατά την ενσωμάτωση των δύο εργαλείων AI iflow και OpenCode. Μέσα από ένα ενιαίο `IAIProvider` διασύνδεση, προσαρμόσαμε διαφορετικά στυλ επικοινωνίας, WebSocket και HTTP, διατηρώντας ταυτόχρονα το μοτίβο κλήσης του ανώτερου επιπέδου.

Η βασική ιδέα είναι στην πραγματικότητα πολύ απλή:
1. Ορίστε μια αφαίρεση ενοποιημένης διεπαφής.
2. Δημιουργήστε επίπεδα προσαρμογέα για διαφορετικές υλοποιήσεις.
3. Διαχειριστείτε τα πάντα ομοιόμορφα μέσω του εργοστασιακού μοτίβου.

Αυτό δίνει στο σύστημα καλή επεκτασιμότητα. Όταν ένα νέο εργαλείο AI πρέπει να ενσωματωθεί αργότερα, το μόνο που χρειάζεται να κάνουμε είναι να το εφαρμόσουμε `IAIProvider` διεπαφή χωρίς να αλλάξει πάρα πολύ τον υπάρχοντα κώδικα.

Εάν εργάζεστε επίσης για την ενοποίηση πολλαπλών εργαλείων AI, ελπίζω αυτό το άρθρο να είναι χρήσιμο.
