## Αυτοματοποίηση συγχρονισμού εικόνας από το Docker Hub στο Azure ACR

> Αυτό το άρθρο εξηγεί τον τρόπο χρήσης του GitHub Actions και του εργαλείου συγχρονισμού εικόνας για την αυτοματοποίηση του συγχρονισμού εικόνων από το Docker Hub στο Μητρώο κοντέινερ Azure, αντιμετωπίζοντας την αργή πρόσβαση στο Docker Hub στην Κίνα και ορισμένες περιοχές Azure, βελτιώνοντας παράλληλα τη διαθεσιμότητα εικόνας και την αποτελεσματικότητα ανάπτυξης σε περιβάλλοντα Azure.

## Ιστορικό / Εισαγωγή

Το έργο HagiCode χρησιμοποιεί εικόνες Docker ως βασικά στοιχεία χρόνου εκτέλεσης, με τις κύριες εικόνες να φιλοξενούνται στο Docker Hub. Καθώς το έργο μεγάλωσε και οι ανάγκες ανάπτυξης του Azure αυξήθηκαν, αντιμετωπίσαμε τα ακόλουθα σημεία πόνου:

- Χαμηλές ταχύτητες έλξης εικόνας, με περιορισμένη πρόσβαση στο Docker Hub στην Κίνα και σε ορισμένες περιοχές Azure
- Μια μοναδική πηγή εικόνας δημιουργεί ένα μόνο σημείο κινδύνου αστοχίας
- Το Azure Container Registry παρέχει καλύτερη απόδοση και ενοποίηση δικτύου όταν εκτελείται στο Azure

Για να λύσουμε αυτά τα προβλήματα, χρειαζόμασταν έναν αυτοματοποιημένο μηχανισμό συγχρονισμού εικόνων για τον τακτικό συγχρονισμό εικόνων από το Docker Hub στο Azure ACR, διασφαλίζοντας ότι οι χρήστες μπορούν να έχουν μεγαλύτερες ταχύτητες έλξης εικόνας και μεγαλύτερη διαθεσιμότητα σε περιβάλλοντα Azure.

- Σταδιακός συγχρονισμός: συγχρονίζει μόνο αλλαγμένα επίπεδα εικόνας, μειώνοντας σημαντικά τη μεταφορά δικτύου

- Χωρίς αυξητικό συγχρονισμό: το πλήρες περιεχόμενο της εικόνας πρέπει να τραβιέται κάθε φορά

### 3. Azure CLI
- Μεγαλύτερη πολυπλοκότητα: Ο έλεγχος ταυτότητας CLI Azure πρέπει να διαμορφωθεί
- Περιορισμένη λειτουργικότητα: `az acr import` είναι σχετικά στενό σε εύρος
- Εγγενής ενσωμάτωση: ενσωματώνεται καλά με τις υπηρεσίες Azure

## Υλοποίηση ροής εργασίας GitHub Actions

Παρακάτω είναι η πραγματική διαμόρφωση ροής εργασίας σε χρήση (`.github/workflows/sync-docker-acr.yml`):

```yaml
name: Sync Docker Image to Azure ACR

on:
  schedule:
    - cron: "0 0 * * *" # Every day at 00:00 UTC
  workflow_dispatch: # Manual trigger
  push:
    branches: [publish]

permissions:
  contents: read

jobs:
  sync:
    runs-on: ubuntu-latest

    steps:
      - name: Checkout code
        uses: actions/checkout@v4

      - name: Download image-syncer
        run: |
          # Download the image-syncer binary
          wget https://github.com/AliyunContainerService/image-syncer/releases/download/v1.5.5/image-syncer-v1.5.5-linux-amd64.tar.gz
          tar -zxvf image-syncer-v1.5.5-linux-amd64.tar.gz
          chmod +x image-syncer

      - name: Create auth config
        run: |
          # Generate the authentication config file (YAML format)
          cat > auth.yaml <<EOF
          hagicode.azurecr.io:
            username: "${{ secrets.AZURE_ACR_USERNAME }}"
            password: "${{ secrets.AZURE_ACR_PASSWORD }}"
          EOF

      - name: Create images config
        run: |
          # Generate the image sync config file (YAML format)
          cat > images.yaml <<EOF
          docker.io/newbe36524/hagicode: hagicode.azurecr.io/hagicode
          EOF

      - name: Run image-syncer
        run: |
          # Run synchronization (using the newer --auth and --images parameters)
          ./image-syncer --auth=./auth.yaml --images=./images.yaml --proc=10 --retries=3

      - name: Upload logs
        if: always()
        uses: actions/upload-artifact@v4
        with:
          name: sync-logs
          path: image-syncer-*.log
          retention-days: 7
```

## Λεπτομέρειες διαμόρφωσης

### 1. Συνθήκες ενεργοποίησης
- Προγραμματισμένη ενεργοποίηση: `cron: "0 0 * * *"` τρέχει καθημερινά στις 00:00 UTC
- Χειροκίνητη ενεργοποίηση: `workflow_dispatch` επιτρέπει στους χρήστες να το εκτελούν χειροκίνητα από το GitHub UI
- Σκανδάλη ώθησης: `push: branches: [publish]` ενεργοποιεί τις ωθήσεις προς τον κλάδο απελευθέρωσης για γρήγορο συγχρονισμό

### 2. Διαμόρφωση ελέγχου ταυτότητας (`auth.yaml`)
```yaml
hagicode.azurecr.io:
  username: "${{ secrets.AZURE_ACR_USERNAME }}"
  password: "${{ secrets.AZURE_ACR_PASSWORD }}"
```

### 3. Διαμόρφωση συγχρονισμού εικόνας
```yaml
docker.io/newbe36524/hagicode: hagicode.azurecr.io/hagicode
```
Αυτή η διαμόρφωση σημαίνει ότι όλες οι ετικέτες από `docker.io/newbe36524/hagicode` θα συγχρονιστεί με `hagicode.azurecr.io/hagicode`.

### 4. παράμετροι συγχρονισμού εικόνας
- `--auth=./auth.yaml`: διαδρομή προς το αρχείο διαμόρφωσης ελέγχου ταυτότητας
- `--images=./images.yaml`: διαδρομή προς το αρχείο διαμόρφωσης συγχρονισμού εικόνας
- `--proc=10`: χρήση `10` ταυτόχρονοι εργαζόμενοι
- `--retries=3`: αποτυχίες επανάληψης `3` φορές

## Σημειώσεις και βέλτιστες πρακτικές

### 1. Συστάσεις ασφαλείας
- Εναλλάσσετε τακτικά τους κωδικούς πρόσβασης Azure ACR (συνιστάται κάθε 90 ημέρες)
- Χρησιμοποιήστε έναν αποκλειστικό λογαριασμό υπηρεσίας ACR με άδειες μόνο push
- Παρακολουθήστε τα αρχεία καταγραφής πρόσβασης ACR για να εντοπίσετε έγκαιρα μη φυσιολογική πρόσβαση
- Μην εξάγετε διαπιστευτήρια σε αρχεία καταγραφής
- Μην δεσμεύετε διαπιστευτήρια στο αποθετήριο κώδικα

### 2. Βελτιστοποίηση απόδοσης
- Ρυθμίστε το `--proc` παράμετρος: συντονισμός συγχρονισμού με βάση το εύρος ζώνης δικτύου (συνιστάται `5-20`)
- Παρακολούθηση διάρκειας συγχρονισμού: εάν ο συγχρονισμός διαρκεί πολύ, εξετάστε το ενδεχόμενο μείωσης του συγχρονισμού
- Καθαρίζετε τακτικά τα κούτσουρα: ορίστε ένα λογικό `retention-days` αξία (`7` επί του παρόντος)

### 3. Αντιμετώπιση προβλημάτων

#### Θέμα 1: Αποτυχία ελέγχου ταυτότητας
```
Error: failed to authenticate to hagicode.azurecr.io
```
Λύση:
1. Ελέγξτε εάν το GitHub Secrets έχει ρυθμιστεί σωστά
2. Βεβαιωθείτε ότι ο κωδικός πρόσβασης Azure ACR έχει λήξει
3. Επιβεβαιώστε ότι τα δικαιώματα λογαριασμού υπηρεσίας ACR είναι σωστά

#### Θέμα 2: Λήξη χρονικού ορίου δικτύου
```
Error: timeout waiting for response
```
Λύση:
1. Ελέγξτε τη συνδεσιμότητα δικτύου
2. Μειώστε τον αριθμό των ταυτόχρονων εργαζομένων (ο `--proc` παράμετρος)
3. Περιμένετε να επανέλθει το δίκτυο και εκτελέστε ξανά τη ροή εργασίας

#### Θέμα 3: Ατελής συγχρονισμός εικόνας
```
Warning: some tags failed to sync
```
Λύση:
1. Ελέγξτε τα αρχεία καταγραφής συγχρονισμού για να εντοπίσετε αποτυχημένες ετικέτες
2. Εκτελέστε ξανά μη αυτόματα τη ροή εργασίας για να συγχρονίσετε ξανά
3. Βεβαιωθείτε ότι η εικόνα προέλευσης στο Docker Hub είναι υγιής

### 4. Παρακολούθηση και ειδοποίηση
- Ελέγχετε τακτικά τη σελίδα Ενέργειες για να επιβεβαιώσετε την κατάσταση ροής εργασιών
- Ενεργοποιήστε τις ειδοποιήσεις GitHub, ώστε οι αποτυχίες ροής εργασιών να αναφέρονται αμέσως
- Παρακολουθήστε τη χρήση αποθήκευσης Azure ACR
- Επαληθεύστε τακτικά τη συνέπεια των ετικετών σε όλα τα μητρώα

## Συχνές Ερωτήσεις και Λύσεις

### Ε1: Πώς μπορώ να συγχρονίσω συγκεκριμένες ετικέτες αντί για όλες τις ετικέτες;

Τροποποιήστε το `images.yaml` αρχείο ρυθμίσεων:
```yaml
# Sync only the latest and v1.0 tags
docker.io/newbe36524/hagicode:latest: hagicode.azurecr.io/hagicode:latest
docker.io/newbe36524/hagicode:v1.0: hagicode.azurecr.io/hagicode:v1.0
```

### Ε2: Πώς μπορώ να συγχρονίσω πολλά αποθετήρια εικόνων;

Προσθήκη πολλαπλών γραμμών σε `images.yaml`:
```yaml
docker.io/newbe36524/hagicode: hagicode.azurecr.io/hagicode
docker.io/newbe36524/another-image: hagicode.azurecr.io/another-image
```

### Ε3: Πώς μπορώ να προσπαθήσω ξανά μετά την αποτυχία του συγχρονισμού;
- Αυτόματη επανάληψη: ο συγχρονιστής εικόνας περιλαμβάνει έναν ενσωματωμένο μηχανισμό επανάληψης (`3` επαναλαμβάνει από προεπιλογή)
- Μη αυτόματη επανάληψη: κάντε κλικ `Re-run all jobs` στη σελίδα Ενέργειες GitHub

### Ε4: Πώς μπορώ να δω τη λεπτομερή πρόοδο του συγχρονισμού;
- Προβολή αρχείων καταγραφής σε πραγματικό χρόνο στη σελίδα Ενέργειες
- Κατεβάστε το `sync-logs` artifact για να επιθεωρήσετε τα πλήρη αρχεία καταγραφής
- Τα αρχεία καταγραφής περιλαμβάνουν την κατάσταση συγχρονισμού και την ταχύτητα μεταφοράς για κάθε ετικέτα

### Ε5: Πόσος χρόνος διαρκεί ο συγχρονισμός;
- Αρχικός πλήρης συγχρονισμός: συνήθως `10-30` λεπτά ανάλογα με το μέγεθος της εικόνας
- Αυξητικός συγχρονισμός: συνήθως `2-5` λεπτά εάν οι αλλαγές στην εικόνα είναι μικρές
- Ο χρόνος εξαρτάται από το εύρος ζώνης δικτύου, το μέγεθος εικόνας και τις ρυθμίσεις ταυτόχρονης χρήσης

## Προτεινόμενες επεκτάσεις

### 1. Προσθέστε ειδοποιήσεις συγχρονισμού
Προσθέστε ένα βήμα ειδοποίησης στη ροή εργασίας:
```yaml
- name: Notify on success
  if: success()
  run: |
    echo "Docker images synced successfully to Azure ACR"
```

### 2. Εφαρμογή φιλτραρίσματος ετικετών εικόνας
Προσθήκη λογικής φιλτραρίσματος ετικετών στη ροή εργασίας:
```yaml
- name: Filter tags
  run: |
    # Sync only tags prefixed with v
    echo "docker.io/newbe36524/hagicode:v* : hagicode.azurecr.io/hagicode:v*" > images.yaml
```

### 3. Προσθέστε μια αναφορά στατιστικών στοιχείων συγχρονισμού
```yaml
- name: Generate report
  if: always()
  run: |
    echo "## Sync Report" >> $GITHUB_STEP_SUMMARY
    echo "- Total tags: $(grep -c 'synced' image-syncer-*.log)" >> $GITHUB_STEP_SUMMARY
    echo "- Sync time: ${{ steps.sync.outputs.duration }}" >> $GITHUB_STEP_SUMMARY
```

Αυτή η λύση συνδυάζει τους προγραμματισμένους και μη αυτόματους ενεργοποιητές του GitHub Actions με τους μηχανισμούς αυξητικού συγχρονισμού και χειρισμού σφαλμάτων του συγχρονισμού εικόνας για να διασφαλίσει τον έγκαιρο και συνεπή συγχρονισμό εικόνων.
