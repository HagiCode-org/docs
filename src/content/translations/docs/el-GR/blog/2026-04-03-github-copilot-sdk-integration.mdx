## Από κλήσεις CLI στην ενσωμάτωση SDK: Βέλτιστες πρακτικές για το GitHub Copilot σε έργα .NET

> Η διαδρομή αναβάθμισης από την επίκληση της γραμμής εντολών στην επίσημη ενσωμάτωση του SDK ήταν αρκετά μεγάλη διαδρομή. Σήμερα, θέλω να μοιραστώ τις παγίδες που συναντήσαμε και τι μάθαμε κατά τη δημιουργία του HagiCode.

## φόντο

Μετά την επίσημη κυκλοφορία του GitHub Copilot SDK το 2025, αρχίσαμε να το ενσωματώνουμε στο επίπεδο δυνατοτήτων AI. Πριν από αυτό, το έργο χρησιμοποιούσε κυρίως το GitHub Copilot επικαλούμενοι απευθείας το εργαλείο γραμμής εντολών Copilot CLI, αλλά αυτή η προσέγγιση είχε αρκετά προφανή ζητήματα:

- **Διαχείριση σύνθετης διαδικασίας**: Έπρεπε να διαχειριστούμε με μη αυτόματο τρόπο τον κύκλο ζωής της διαδικασίας CLI, τα χρονικά όρια εκκίνησης και την εκκαθάριση της διαδικασίας. Οι διαδικασίες μπορεί να διακοπούν χωρίς προειδοποίηση.
- **Ημιτελής χειρισμός συμβάντων**: Η ακατέργαστη επίκληση CLI καθιστά δύσκολη την καταγραφή λεπτομερών συμβάντων από τη συλλογιστική του μοντέλου και την εκτέλεση του εργαλείου. Είναι σαν να βλέπεις μόνο το αποτέλεσμα χωρίς τη διαδικασία σκέψης.
- **Δύσκολη διαχείριση συνεδρίας**: Δεν υπήρχε αποτελεσματικός μηχανισμός για την επαναχρησιμοποίηση και την ανάκτηση της περιόδου λειτουργίας, επομένως κάθε αλληλεπίδραση έπρεπε να ξεκινήσει από την αρχή.
- **Προβλήματα συμβατότητας**: Τα ορίσματα CLI άλλαζαν συχνά, πράγμα που σήμαινε ότι έπρεπε να διατηρούμε συνεχώς τη λογική συμβατότητας για αυτές τις παραμέτρους.

Αυτά τα ζητήματα έγιναν ολοένα και πιο εμφανή στην καθημερινή ανάπτυξη, ειδικά όταν χρειαζόταν να παρακολουθήσουμε τη λογική του μοντέλου (`thinking`) και την κατάσταση εκτέλεσης του εργαλείου σε πραγματικό χρόνο. Σε εκείνο το σημείο, κατέστη σαφές ότι χρειαζόμασταν μια προσέγγιση ολοκλήρωσης χαμηλότερου επιπέδου και πληρέστερης.

## Βασική Υλοποίηση

### Ροή ελέγχου ταυτότητας

Ο έλεγχος ταυτότητας είναι το πρώτο και πιο σημαντικό βήμα της ενοποίησης του SDK. Εάν ο έλεγχος ταυτότητας αποτύχει, τίποτα άλλο δεν έχει σημασία. Σχεδιάσαμε μια ευέλικτη διαμόρφωση ελέγχου ταυτότητας που υποστηρίζει πολλαπλές πηγές ελέγχου ταυτότητας:

```csharp
// CopilotProvider.cs - Authentication source configuration
public class CopilotOptions
{
    public bool UseLoggedInUser { get; set; } = true;
    public string? GitHubToken { get; set; }
    public string? CliUrl { get; set; }
}

// Convert to SDK request
return new CopilotSdkRequest(
    GitHubToken: options.AuthSource == CopilotAuthSource.GitHubToken
        ? options.GitHubToken
        : null,
    UseLoggedInUser: options.AuthSource != CopilotAuthSource.GitHubToken
);
```

Τα οφέλη αυτού του σχεδιασμού είναι αρκετά ξεκάθαρα:

- Υποστηρίζει τη λειτουργία συνδεδεμένου χρήστη χωρίς να απαιτείται διακριτικό, το οποίο ταιριάζει καλά σε σενάρια επιφάνειας εργασίας.
- Υποστηρίζει τη λειτουργία GitHub Token, η οποία είναι κατάλληλη για αναπτύξεις από την πλευρά του διακομιστή και κεντρική διαχείριση.
- Υποστηρίζει την παράκαμψη της διεύθυνσης URL CLI Copilot, η οποία βοηθά στη διαμόρφωση του εταιρικού διακομιστή μεσολάβησης.

Στην πράξη, αυτό το ευέλικτο μοντέλο ελέγχου ταυτότητας απλοποίησε σημαντικά τη διαμόρφωση σε διαφορετικά σενάρια ανάπτυξης. Ο υπολογιστής-πελάτης επιτραπέζιου υπολογιστή μπορεί να χρησιμοποιήσει τη δική του κατάσταση σύνδεσης Copilot του κάθε χρήστη, ενώ η πλευρά του διακομιστή μπορεί να διαχειρίζεται τον έλεγχο ταυτότητας κεντρικά μέσω διακριτικών.

### Διαχείριση ροής συμβάντων

Μία από τις πιο ισχυρές δυνατότητες του SDK είναι η πλήρης καταγραφή ροής συμβάντων. Υλοποιήσαμε ένα σύστημα αποστολής συμβάντων που μπορεί να χειριστεί όλα τα είδη συμβάντων SDK σε πραγματικό χρόνο:

```csharp
// GitHubCopilotSdkGateway.cs - Core logic for event dispatch
internal static SessionEventDispatchResult DispatchSessionEvent(
    SessionEvent evt, bool sawDelta)
{
    switch (evt)
    {
        case AssistantReasoningEvent reasoningEvent:
            // Capture the model reasoning process
            events.Add(new CopilotSdkStreamEvent(
                CopilotSdkStreamEventType.ReasoningDelta,
                Content: reasoningEvent.Data.Content));
            break;

        case ToolExecutionStartEvent toolStartEvent:
            // Capture the start of a tool invocation
            events.Add(new CopilotSdkStreamEvent(
                CopilotSdkStreamEventType.ToolExecutionStart,
                ToolName: toolStartEvent.Data.ToolName,
                ToolCallId: toolStartEvent.Data.ToolCallId));
            break;

        case ToolExecutionCompleteEvent toolCompleteEvent:
            // Capture tool completion and its result
            events.Add(new CopilotSdkStreamEvent(
                CopilotSdkStreamEventType.ToolExecutionEnd,
                Content: ExtractToolExecutionContent(toolCompleteEvent)));
            break;

        default:
            // Preserve unhandled events as RawEvent
            events.Add(new CopilotSdkStreamEvent(
                CopilotSdkStreamEventType.RawEvent,
                RawEventType: evt.GetType().Name));
            break;
    }
}
```

Η αξία αυτής της υλοποίησης είναι σημαντική:

- **Πλήρης αποτύπωση της διαδικασίας συλλογιστικής του μοντέλου** (`thinking`): οι χρήστες μπορούν να δουν πώς συλλογίζεται το AI, όχι μόνο την τελική απάντηση.
- **Παρακολούθηση της κατάστασης εκτέλεσης του εργαλείου σε πραγματικό χρόνο**: γνωρίζουμε ποια εργαλεία εκτελούνται, πότε τελειώνουν και τι επιστρέφουν.
- **Μηδενική απώλεια συμβάντος**: με επιστροφή σε `RawEvent`, διασφαλίζουμε ότι κάθε συμβάν καταγράφεται.

Στο HagiCode, αυτά τα λεπτομερή συμβάντα βοηθούν τους χρήστες να κατανοήσουν πώς λειτουργεί το AI εσωτερικά, ειδικά κατά τον εντοπισμό σφαλμάτων σύνθετων εργασιών.

Αυτό μας δίνει πολλά οφέλη:

- Δημιουργεί σαφή διαγνωστικά μηνύματα για να βοηθήσει τους προγραμματιστές να εντοπίσουν τα προβλήματα γρήγορα.

Κατά τη διάρκεια της αναβάθμισης, αυτός ο μηχανισμός συμβατότητας μας βοήθησε να μεταβούμε ομαλά. Τα υπάρχοντα αρχεία διαμόρφωσης θα μπορούσαν ακόμα να χρησιμοποιηθούν και χρειάζονταν μόνο σταδιακές προσαρμογές με βάση τη διαγνωστική έξοδο.

### Runtime Pooling

Η δημιουργία περιόδων λειτουργίας Copilot SDK είναι σχετικά ακριβή και η δημιουργία και η καταστροφή περιόδων σύνδεσης πολύ συχνά βλάπτει την απόδοση. Για να το λύσουμε αυτό, εφαρμόσαμε ένα σύστημα διαχείρισης συνεδριάσεων:

```csharp
// CopilotProvider.cs - Session pool management
await using var lease = await _poolCoordinator.AcquireCopilotRuntimeAsync(
    request,
    async ct => await _gateway.CreateRuntimeAsync(sdkRequest, ct),
    cancellationToken);

if (lease.IsWarmLease)
{
    // Reuse an existing session
    yield return CreateSessionReusedMessage();
}

await foreach (var eventData in lease.Entry.Resource.SendPromptAsync(...))
{
    yield return MapEvent(eventData);
}
```

Τα οφέλη από τη συγκέντρωση συνεδριών περιλαμβάνουν:

- **Επαναχρησιμοποίηση περιόδου λειτουργίας**: αιτήματα με το ίδιο `sessionId` μπορεί να επαναχρησιμοποιήσει υπάρχουσες συνεδρίες και να μειώσει τα έξοδα εκκίνησης.
- **Υποστήριξη ανάκτησης περιόδου λειτουργίας**: μετά από διακοπή δικτύου, η προηγούμενη κατάσταση περιόδου λειτουργίας μπορεί να αποκατασταθεί.
- **Αυτόματη διαχείριση συγκέντρωσης**: οι συνεδρίες που έχουν λήξει καθαρίζονται αυτόματα για να αποφευχθούν διαρροές πόρων.

Στο HagiCode, η συγκέντρωση συνεδριών βελτίωσε αισθητά την ανταπόκριση, ειδικά για συνεχείς συνομιλίες.

## Πρακτικός Οδηγός

### Παράδειγμα διαμόρφωσης

Ακολουθεί ένα πλήρες παράδειγμα διαμόρφωσης:

```json
{
  "AI": {
    "Providers": {
      "Providers": {
        "GitHubCopilot": {
          "Enabled": true,
          "ExecutablePath": "copilot",
          "Model": "gpt-5",
          "WorkingDirectory": "/path/to/project",
          "Timeout": 7200,
          "StartupTimeout": 30,
          "UseLoggedInUser": true,
          "NoAskUser": true,
          "Permissions": {
            "AllowAllTools": false,
            "AllowedTools": ["Read", "Bash", "Grep"],
            "DeniedTools": ["Edit"]
          }
        }
      }
    }
  }
}
```

### Σημειώσεις χρήσης

Στη χρήση του πραγματικού κόσμου, συνοψίσαμε πολλά σημεία στα οποία αξίζει να προσέξουμε:

**Διαμόρφωση χρονικού ορίου εκκίνησης**: Η πρώτη εκκίνηση του Copilot CLI μπορεί να διαρκέσει σχετικά μεγάλο χρονικό διάστημα, επομένως προτείνουμε τη ρύθμιση `StartupTimeout` σε τουλάχιστον 30 δευτερόλεπτα. Εάν αυτή είναι η πρώτη σύνδεση, μπορεί να διαρκέσει ακόμη περισσότερο.

**Διαχείριση αδειών**: Σε περιβάλλοντα παραγωγής, αποφύγετε τη χρήση `AllowAllTools: true`. Χρησιμοποιήστε το `AllowedTools` λίστα επιτρεπόμενων για να ελέγξετε ποια εργαλεία είναι διαθέσιμα και να χρησιμοποιήσετε το `DeniedTools` denylist για να μπλοκάρει επικίνδυνες λειτουργίες. Αυτό αποτρέπει αποτελεσματικά το AI από την εκτέλεση επικίνδυνων εντολών.

**Διαχείριση συνεδρίας**: Αιτήματα με το ίδιο `sessionId` αυτόματη επαναχρησιμοποίηση συνεδριών. Η κατάσταση της συνεδρίας διατηρείται έως `ProviderSessionId`. Η ακύρωση διαδίδεται μέσω `CancellationTokenSource`.

**Διαγνωστική έξοδος**: Μη συμβατά ορίσματα CLI δημιουργούν μηνύματα τύπου `diagnostic`. Τα συμβάντα Raw SDK διατηρούνται ως `event.raw`. Τα μηνύματα σφάλματος περιλαμβάνουν κατηγορίες όπως το χρονικό όριο εκκίνησης και η ασυμβατότητα ορισμάτων για να διευκολύνουν την αντιμετώπιση προβλημάτων.

### Βέλτιστες Πρακτικές

Με βάση την πρακτική εμπειρία μας, ακολουθούν μερικές βέλτιστες πρακτικές:

**1. Χρησιμοποιήστε μια λίστα επιτρεπόμενων εργαλείων**

```csharp
var request = new AIRequest
{
    Prompt = "Analyze this file",
    AllowedTools = new[] { "Read", "Grep", "Bash(git:*)" }
};
```

Ο ρητός καθορισμός των επιτρεπόμενων εργαλείων μέσω μιας λίστας επιτρεπόμενων συμβάλλει στην αποτροπή απροσδόκητων ενεργειών τεχνητής νοημοσύνης. Αυτό είναι ιδιαίτερα σημαντικό για εργαλεία με δικαιώματα εγγραφής, όπως π.χ `Edit`, το οποίο θα πρέπει να αντιμετωπίζεται με ιδιαίτερη προσοχή.

**2. Ορίστε εύλογα χρονικά διαστήματα**

```csharp
options.Timeout = 3600;  // 1 hour
options.StartupTimeout = 60;  // 1 minute
```

Ορίστε τις κατάλληλες τιμές χρονικού ορίου με βάση την πολυπλοκότητα της εργασίας. Εάν η τιμή είναι πολύ μικρή, οι εργασίες ενδέχεται να διακοπούν. Εάν είναι πολύ μεγάλο, οι πόροι μπορεί να σπαταληθούν περιμένοντας σε αιτήματα που δεν ανταποκρίνονται.

**3. Ενεργοποίηση επαναχρησιμοποίησης συνεδρίας**

```csharp
options.SessionId = "my-session-123";
```

Χρησιμοποιώντας το ίδιο `sessionId` για σχετικές εργασίες σάς επιτρέπει να επαναχρησιμοποιήσετε το πλαίσιο προηγούμενης συνεδρίας και να βελτιώσετε την ταχύτητα απόκρισης.

**4. Χειριστείτε τις απαντήσεις ροής**

```csharp
await foreach (var chunk in provider.StreamAsync(request))
{
    switch (chunk.Type)
    {
        case StreamingChunkType.ThinkingDelta:
            // Handle the reasoning process
            break;
        case StreamingChunkType.ToolCallDelta:
            // Handle tool invocation
            break;
        case StreamingChunkType.ContentDelta:
            // Handle text output
            break;
    }
}
```

Οι απαντήσεις ροής σάς επιτρέπουν να δείχνετε την πρόοδο επεξεργασίας AI σε πραγματικό χρόνο, γεγονός που βελτιώνει την εμπειρία του χρήστη. Αυτό είναι ιδιαίτερα πολύτιμο για μακροχρόνιες εργασίες.

**5. Διαχείριση σφαλμάτων και επανάληψη**

```csharp
try
{
    await foreach (var chunk in provider.StreamAsync(request))
    {
        // Handle the response
    }
}
catch (CopilotSessionException ex)
{
    // Handle session exceptions
    logger.LogError(ex, "Copilot session failed");
    // Decide whether to retry based on the exception type
}
```

Ο σωστός χειρισμός σφαλμάτων και η λογική επανάληψης της προσπάθειας βελτιώνουν τη συνολική σταθερότητα του συστήματος.

- **Καλύτερη εμπειρία χρήστη**: Η ανατροφοδότηση συμβάντων σε πραγματικό χρόνο δίνει στους χρήστες μια σαφέστερη κατανόηση του τι κάνει η τεχνητή νοημοσύνη.

Αυτή η αναβάθμιση δεν ήταν απλώς μια αντικατάσταση της τεχνικής υλοποίησης. Ήταν επίσης μια αρχιτεκτονική βελτιστοποίηση ολόκληρου του επιπέδου ικανότητας AI. Μέσω της πολυεπίπεδης σχεδίασης και των διεπαφών αφαίρεσης, αποκτήσαμε καλύτερη συντηρησιμότητα και επεκτασιμότητα.

Εάν σκέφτεστε να ενσωματώσετε το GitHub Copilot στο δικό σας έργο .NET, ελπίζω αυτά τα πρακτικά μαθήματα να σας βοηθήσουν να αποφύγετε περιττές παρακάμψεις. Το επίσημο SDK είναι πράγματι πιο σταθερό και πλήρες από την επίκληση CLI και αξίζει τον χρόνο να το κατανοήσετε και να το υιοθετήσετε.
