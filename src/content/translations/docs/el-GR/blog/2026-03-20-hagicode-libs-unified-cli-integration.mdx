## Hagicode.Libs: Engineering Practice for Unified Integration of Multiple AI Coding Assistant CLIs

> Κατά την ανάπτυξη του έργου HagiCode, χρειαζόμασταν να ενσωματώσουμε πολλαπλά βοηθητικά CLI κωδικοποίησης AI, συμπεριλαμβανομένων των Claude Code, Codex και CodeBuddy. Κάθε CLI έχει διαφορετικές διεπαφές, παραμέτρους και μορφές εξόδου και ο επαναλαμβανόμενος κώδικας ενσωμάτωσης έκανε το έργο όλο και πιο δύσκολο να διατηρηθεί. Σε αυτό το άρθρο, μοιραζόμαστε πώς δημιουργήσαμε ένα ενοποιημένο επίπεδο αφαίρεσης με το HagiCode.Libs για να λύσουμε αυτό το σημείο πόνου της μηχανικής. Θα μπορούσατε επίσης να πείτε ότι είναι απλώς μια σκληρή εμπειρία που συλλέγεται από τις παγίδες που έχουμε ήδη χτυπήσει.

## φόντο

Η αγορά για βοηθούς κωδικοποίησης AI είναι αρκετά ζωντανή τώρα. Εκτός από τον Claude Code, υπάρχουν επίσης το Codex του OpenAI, το CodeBuddy του Zhipu και άλλα. Ως έργο βοηθού κωδικοποίησης AI, το HagiCode πρέπει να ενσωματώσει αυτά τα διαφορετικά εργαλεία CLI σε πολλά υποέργα, συμπεριλαμβανομένων των επιτραπέζιων υπολογιστών, του backend και του ιστού.

Στην αρχή, το πρόβλημα ήταν διαχειρίσιμο. Η ενσωμάτωση ενός CLI ήταν μόνο μερικές εκατοντάδες γραμμές κώδικα. Αλλά καθώς ο αριθμός των CLI που χρειαζόμασταν να υποστηρίξουμε συνέχιζε να αυξάνεται, τα πράγματα άρχισαν να γίνονται ακατάστατα.

Κάθε CLI έχει τη δική του μορφή ορίσματος γραμμής εντολών, διαφορετικές απαιτήσεις μεταβλητών περιβάλλοντος και μεγάλη ποικιλία μορφών εξόδου. Κάποια εξάγουν JSON, μερικά εξάγουν JSON ροής και μερικά εξάγουν απλό κείμενο. Επιπλέον, υπάρχουν ζητήματα συμβατότητας μεταξύ πλατφορμών. Ο εντοπισμός εκτελέσιμων και η διαχείριση διεργασιών λειτουργούν πολύ διαφορετικά μεταξύ των συστημάτων Windows και Unix, επομένως η αντιγραφή κώδικα συνέχιζε να αυξάνεται. Στην πραγματικότητα, ήταν λίγο πιο Ctrl+C και Ctrl+V, αλλά η συντήρηση έγινε γρήγορα επώδυνη.

Το πιο απογοητευτικό μέρος ήταν ότι κάθε φορά που θέλαμε να προσθέσουμε υποστήριξη για μια νέα δυνατότητα CLI, έπρεπε να αλλάζουμε τον ίδιο κώδικα σε πολλά έργα. Αυτή η προσέγγιση σαφώς δεν ήταν βιώσιμη μακροπρόθεσμα. Ο κώδικας έχει επίσης χαρακτήρα. αντιγράψτε το πάρα πολλές φορές και αρχίζει να προκαλεί προβλήματα.

## Σχετικά με το HagiCode

Η προσέγγιση που κοινοποιείται σε αυτό το άρθρο προέρχεται από την πρακτική εμπειρία μας στο [HagiCode](https://hagicode.com) έργο. Το HagiCode είναι ένα έργο βοηθού κωδικοποίησης τεχνητής νοημοσύνης ανοιχτού κώδικα που πρέπει να διατηρεί ταυτόχρονα πολλαπλά υποέργα, συμπεριλαμβανομένης μιας επέκτασης VSCode frontend, υπηρεσιών AI backend και ενός προγράμματος-πελάτη για επιτραπέζιους υπολογιστές πολλαπλών πλατφορμών. Κατά κάποιο τρόπο, ήταν ακριβώς αυτό το πολύπλοκο, πολυγλωσσικό, πολυπλατφορμικό περιβάλλον που οδήγησε στη γέννηση του HagiCode.Libs. Θα μπορούσατε να πείτε ότι αναγκαστήκαμε σε αυτό, και έτσι θα είναι.

## Ανάλυση: Εύρεση κοινού εδάφους

Αν και αυτοί οι βοηθοί κωδικοποίησης AI CLI έχουν το καθένα τα δικά του χαρακτηριστικά, από τεχνική άποψη μοιράζονται πολλά προφανή χαρακτηριστικά:

**Παρόμοια μοτίβα αλληλεπίδρασης**: όλα ξεκινούν μια διαδικασία CLI, στέλνουν ένα μήνυμα προτροπής, λαμβάνουν απαντήσεις ροής, αναλύουν μηνύματα και, στη συνέχεια, τερματίζουν ή συνεχίζουν τη συνεδρία. Στο τέλος της ημέρας, όλη η ροή ακολουθεί το ίδιο βασικό καλούπι.

**Παρόμοιες ανάγκες διαμόρφωσης**: όλα χρειάζονται έλεγχο ταυτότητας κλειδιού API, ρύθμιση καταλόγου εργασίας, επιλογή μοντέλου, έλεγχο αδειών εργαλείου και διαχείριση συνεδρίας. Εξάλλου, όλοι ζουν από τα API. οι διαφορές είναι κυρίως θέμα γεύσης.

**Οι ίδιες προκλήσεις πολλαπλών πλατφορμών**: όλες πρέπει να λύσουν την ανάλυση εκτελέσιμων μονοπατιών (`claude` vs `claude.exe` vs `/usr/local/bin/claude`), εκκίνηση διεργασίας και χειρισμός μεταβλητών περιβάλλοντος, διαφυγή εντολών κελύφους και κατασκευή ορίσματος. Η εργασία μεταξύ πλατφορμών είναι επώδυνη όπως και να την περιγράφεις. Μόνο οι άνθρωποι που έχουν μπει στις παγίδες καταλαβαίνουν πραγματικά τη διαφορά μεταξύ των Windows και του Unix.

Με βάση αυτήν την ανάλυση, χρειαζόμασταν ένα ενοποιημένο επίπεδο αφαίρεσης που θα μπορούσε να παρέχει μια συνεπή διεπαφή, να ενσωματώνει τη λογική ανακάλυψης CLI μεταξύ πλατφορμών, να χειρίζεται την ανάλυση εξόδου ροής και να υποστηρίζει τόσο σενάρια εισαγωγής εξάρτησης όσο και σενάρια μη DI. Είναι το είδος του προβλήματος που πονάει το κεφάλι σου και μόνο που το σκέφτεσαι, αλλά πρέπει ακόμα να το αντιμετωπίσεις. Άλλωστε, είναι δικό μας έργο, οπότε πρέπει να το τελειώσουμε ακόμα κι αν πρέπει να κλάψουμε.

## Λύση: HagiCode.Libs

Δημιουργήσαμε το **HagiCode.Libs**, έναν ελαφρύ χώρο εργασίας βιβλιοθήκης .NET 10 που κυκλοφόρησε με άδεια MIT και τώρα δημοσιεύεται στις [GitHub](https://github.com/HagiCode-org/Hagicode.Libs). Μπορεί να μην είναι κάποιο αριστούργημα που συγκλονίζει τον κόσμο, αλλά είναι πραγματικά χρήσιμο για την επίλυση πραγματικών προβλημάτων.

### Δομή έργου

```
HagiCode.Libs/
├── src/
│   ├── HagiCode.Libs.Core/           # Core capabilities
│   │   ├── Discovery/                 # CLI executable discovery
│   │   ├── Process/                   # Cross-platform process management
│   │   ├── Transport/                 # Streaming message transport
│   │   └── Environment/               # Runtime environment resolution
│   ├── HagiCode.Libs.Providers/       # Provider implementations
│   │   ├── ClaudeCode/                # Claude Code provider
│   │   ├── Codex/                     # Codex provider
│   │   └── Codebuddy/                 # CodeBuddy provider
│   ├── HagiCode.Libs.ConsoleTesting/  # Testing framework
│   ├── HagiCode.Libs.ClaudeCode.Console/
│   ├── HagiCode.Libs.Codex.Console/
│   └── HagiCode.Libs.Codebuddy.Console/
└── tests/                             # xUnit tests
```

### Σχεδιαστικοί στόχοι

Κατά το σχεδιασμό του HagiCode.Libs, ακολουθήσαμε μερικές αρχές. Όλα προήλθαν από μαθήματα που πήραμε με τον δύσκολο τρόπο:

**Μηδενικές βαριές εξαρτήσεις πλαισίου**: δεν εξαρτάται από το ABP ή οποιοδήποτε άλλο μεγάλο πλαίσιο, το οποίο το διατηρεί ελαφρύ. Αυτές τις μέρες, όσο λιγότερες εξαρτήσεις έχετε, τόσο λιγότερους πονοκεφάλους έχετε. Οι περισσότεροι άνθρωποι έχουν ήδη χτυπηθεί από την κόλαση της εξάρτησης τουλάχιστον μία φορά.

**Υποστήριξη μεταξύ πλατφορμών**: εγγενής υποστήριξη για Windows, macOS και Linux, χωρίς εγγραφή ξεχωριστού κώδικα για διαφορετικές πλατφόρμες. Μια βάση κώδικα που τρέχει παντού είναι πολύ καλό.

**Επεξεργασία ροής**: Η έξοδος CLI γίνεται με ασύγχρονες ροές, που ταιριάζει πολύ καλύτερα σε σύγχρονα μοτίβα προγραμματισμού .NET. Οι καιροί αλλάζουν και το async είναι βασιλιάς.

**Ευέλικτη ενσωμάτωση**: υποστηρίζει σενάρια έγχυσης εξάρτησης, ενώ επιτρέπει επίσης την άμεση εγκατάσταση. Διαφορετικοί άνθρωποι έχουν διαφορετικές προτιμήσεις, επομένως θέλαμε να είναι βολικό με κάθε τρόπο.

## Πώς να το χρησιμοποιήσετε

### Μέσω της ένεσης εξάρτησης

Εάν το έργο σας χρησιμοποιεί ήδη ένεση εξάρτησης, όπως το ASP.NET Core ή τον γενικό κεντρικό υπολογιστή, μπορείτε να το ενσωματώσετε απευθείας. Είναι μικρό πράγμα, αλλά με καλή συμπεριφορά:

```csharp
using HagiCode.Libs.Providers;
using Microsoft.Extensions.DependencyInjection;

var services = new ServiceCollection();
services.AddHagiCodeLibs();

await using var provider = services.BuildServiceProvider();
var claude = provider.GetRequiredService<ICliProvider<ClaudeCodeOptions>>();

var options = new ClaudeCodeOptions
{
    ApiKey = "your-api-key",
    Model = "claude-sonnet-4-20250514"
};

await foreach (var message in claude.ExecuteAsync(options, "Hello, Claude!"))
{
    Console.WriteLine($"{message.Type}: {message.Content}");
}
```

### Άμεση παρουσίαση

Εάν γράφετε ένα απλό σενάριο ή εργάζεστε σε ένα σενάριο που δεν είναι DI, η δημιουργία ενός στιγμιότυπου απευθείας λειτουργεί επίσης. Με απλά λόγια, εξαρτάται από τις προσωπικές σας προτιμήσεις:

```csharp
var claude = new ClaudeCodeProvider();
var options = new ClaudeCodeOptions
{
    ApiKey = "sk-ant-xxx",
    Model = "claude-sonnet-4-20250514"
};

await foreach (var message in claude.ExecuteAsync(options, "Help me write a quicksort"))
{
    // Handle messages
}
```

Και οι δύο προσεγγίσεις χρησιμοποιούν την ίδια υποκείμενη υλοποίηση, ώστε να μπορείτε να επιλέξετε το στυλ ολοκλήρωσης που ταιριάζει καλύτερα στο έργο σας. Δεν υπάρχει καθολική σωστή απάντηση σε αυτόν τον κόσμο. Αυτό που σας ταιριάζει είναι η καλύτερη επιλογή. Μπορεί να ακούγεται κλισέ, αλλά είναι αλήθεια.

## Πρακτική εμπειρία

### 1. Ειδικές κονσόλες δοκιμών

Κάθε πάροχος έχει το δικό του αποκλειστικό έργο κονσόλας δοκιμών, καθιστώντας ευκολότερη την ανεξάρτητη επικύρωση της ενσωμάτωσης. Οι δοκιμές είναι ένα από εκείνα τα πράγματα όπου εάν πρόκειται να το κάνετε, θα πρέπει να το κάνετε σωστά:

```bash
# Claude Code tests
dotnet run --project src/HagiCode.Libs.ClaudeCode.Console -- --test-provider
dotnet run --project src/HagiCode.Libs.ClaudeCode.Console -- --test-all claude

# CodeBuddy tests
dotnet run --project src/HagiCode.Libs.Codebuddy.Console -- --test-provider codebuddy-cli

# Codex tests
dotnet run --project src/HagiCode.Libs.Codex.Console -- --test-provider codex-cli
```

Τα σενάρια δοκιμών καλύπτουν πολλές βασικές περιπτώσεις:
- **Ping**: έλεγχος υγείας για επιβεβαίωση ότι το CLI είναι διαθέσιμο
- **Απλή ερώτηση**: βασική δοκιμή προτροπής
- **Σύνθετη ερώτηση**: δοκιμή συνομιλίας πολλαπλών στροφών
- **Επαναφορά συνεδρίας/Συνέχιση**: δοκιμή ανάκτησης συνεδρίας
- **Ανάλυση αποθετηρίου**: δοκιμή ανάλυσης αποθετηρίου

Αυτός ο αυτόνομος σχεδιασμός κονσόλας δοκιμών είναι ιδιαίτερα χρήσιμος κατά τον εντοπισμό σφαλμάτων, επειδή μας επιτρέπει να προσδιορίσουμε γρήγορα εάν το πρόβλημα βρίσκεται στο επίπεδο HagiCode.Libs ή στο ίδιο το CLI. Ο εντοπισμός σφαλμάτων είναι στην πραγματικότητα απλώς να βρεις πού βρίσκεται το πρόβλημα. Μόλις η κατεύθυνση είναι σωστή, είστε ήδη στα μισά του δρόμου.

### 2. Επικύρωση CI/CD μεταξύ πλατφορμών

Η συμβατότητα μεταξύ πλατφορμών είναι ένας από τους βασικούς στόχους του HagiCode.Libs. Διαμορφώσαμε τη ροή εργασίας GitHub Actions `.github/workflows/cli-discovery-cross-platform.yml` για εκτέλεση πραγματικής επικύρωσης ανακάλυψης CLI `ubuntu-latest`, `macos-latest`, και `windows-latest`.

Αυτό διασφαλίζει ότι κάθε αλλαγή κώδικα δεν διαταράσσει τη συμβατότητα μεταξύ πλατφορμών. Κατά την τοπική ανάπτυξη, μπορείτε επίσης να το αναπαράγετε με τις ακόλουθες εντολές. Εξάλλου, δεν μπορείτε να ζητήσετε από την CI να αναλάβει την ευθύνη για όλα. Το τοπικό σας περιβάλλον θα πρέπει επίσης να μπορεί να το εκτελέσει:

```bash
npm install --global @anthropic-ai/claude-code@2.1.79
HAGICODE_REAL_CLI_TESTS=1 dotnet test --filter "Category=RealCli"
```

### 3. Επεξεργασία ροής μηνυμάτων

Το HagiCode.Libs χρησιμοποιεί ασύγχρονες ροές για την επεξεργασία της εξόδου CLI. Σε σύγκριση με τις παραδοσιακές προσεγγίσεις επανάκλησης ή με βάση συμβάντα, αυτό ταιριάζει πολύ καλύτερα στο στυλ ασύγχρονου προγραμματισμού του σύγχρονου .NET. Τελικά, έτσι απλά προχωρά η τεχνολογία, είτε αρέσει σε κάποιον είτε όχι:

```csharp
public async IAsyncEnumerable<CliMessage> ExecuteAsync(
    TOptions options,
    string prompt,
    [EnumeratorCancellation] CancellationToken cancellationToken = default)
{
    // Start the CLI process
    // Parse streaming JSON output
    // Yield the CliMessage sequence
}
```

Οι τύποι μηνυμάτων περιλαμβάνουν:
- `user`: μήνυμα χρήστη
- `assistant`: απάντηση βοηθού
- `tool_use`: επίκληση εργαλείου
- `result`: τέλος συνεδρίας

Αυτός ο σχεδιασμός επιτρέπει στους καλούντες να χειρίζονται την έξοδο ροής με ευελιξία, είτε για προβολή σε πραγματικό χρόνο, για μετα-επεξεργασία προσωρινής αποθήκευσης ή για προώθηση σε άλλες υπηρεσίες. Γιατί να ανησυχείτε αν ο ουρανός είναι ηλιόλουστος ή συννεφιασμένος; Αυτό που έχει σημασία είναι ότι μόλις ανοίξει η ιδέα, μπορείτε να τη χρησιμοποιήσετε όπως θέλετε.

### 4. Εξερεύνηση αποθετηρίου Git

Η ενότητα HagiCode.Libs.Exploration παρέχει εντοπισμό αποθετηρίου Git και έλεγχο κατάστασης, κάτι που είναι ιδιαίτερα χρήσιμο σε σενάρια ανάλυσης αποθετηρίου. Αυτή η δυνατότητα γεννήθηκε επίσης από ανάγκη, επειδή το HagiCode χρειάζεται να αναλύσει τα αποθετήρια:

```csharp
// Discover Git repositories
var repositories = await GitRepositoryDiscovery.DiscoverAsync("/path/to/search");

// Get repository information
var info = await GitRepository.GetInfoAsync(repoPath);
Console.WriteLine($"Branch: {info.Branch}, Remote: {info.RemoteUrl}");
Console.WriteLine($"Has uncommitted changes: {info.HasUncommittedChanges}");
```

Οι δυνατότητες ανάλυσης κώδικα του HagiCode χρησιμοποιούν αυτήν την ενότητα για τον προσδιορισμό της δομής του έργου και της κατάστασης Git. Είναι ένα καλό παράδειγμα για την πλήρη χρήση αυτού που χτίσαμε.

## Πράγματα που πρέπει να σημειωθούν

Με βάση την πρακτική μας στο έργο HagiCode, υπάρχουν αρκετά σημεία που αξίζουν ιδιαίτερης προσοχής. Είναι όλα πραγματικά ζητήματα που πρέπει να αντιμετωπιστούν προσεκτικά:

**Ασφάλεια κλειδιού API**: μην κωδικοποιείτε τα κλειδιά API στον κώδικά σας. Χρησιμοποιήστε μεταβλητές περιβάλλοντος ή διαχείριση διαμόρφωσης. Το HagiCode.Libs υποστηρίζει τη μετάδοση της διαμόρφωσης μέσω των αντικειμένων Options, καθιστώντας ευκολότερη την ενσωμάτωση με διαφορετικές πηγές διαμόρφωσης. Όσον αφορά την ασφάλεια, δεν υπάρχει τέτοιο πράγμα όπως να είσαι πολύ προσεκτικός.

**Καρφίτσωμα έκδοσης CLI**: στο CI/CD, καρφιτσώνουμε συγκεκριμένες εκδόσεις, όπως π.χ `@anthropic-ai/claude-code@2.1.79`, για μείωση της αβεβαιότητας που προκαλείται από την μετατόπιση έκδοσης. Είναι επίσης καλή ιδέα να χρησιμοποιείτε σταθερές εκδόσεις στην τοπική ανάπτυξη. Η έκδοση εκδόσεων μπορεί να είναι επώδυνη. Εάν δεν καρφιτσώσετε εκδόσεις, το πρόβλημα θα σας διδάξει ένα μάθημα πολύ γρήγορα.

**Κατηγοριοποίηση δοκιμών**: οι προεπιλεγμένες δοκιμές χρησιμοποιούν ψεύτικους παρόχους για να τους διατηρούν καθοριστικούς και γρήγορους, ενώ οι πραγματικές δοκιμές CLI πρέπει να είναι ρητά ενεργοποιημένες. Αυτό παρέχει γρήγορη ανατροφοδότηση CI, ενώ εξακολουθεί να επιτρέπει την επικύρωση σε πραγματικό περιβάλλον όταν χρειάζεται. Η επίτευξη αυτής της ισορροπίας δεν είναι ποτέ εύκολη. Η ταχύτητα και η σταθερότητα απαιτούν πάντα συμβιβασμούς.

**Διαχείριση περιόδου σύνδεσης**: διαφορετικά CLI έχουν διαφορετικούς μηχανισμούς ανάκτησης συνεδριών. Ο Κώδικας Claude χρησιμοποιεί το `.claude/` κατάλογο για αποθήκευση συνεδριών, ενώ το Codex και το CodeBuddy έχουν το καθένα τις δικές του προσεγγίσεις. Όταν τα χρησιμοποιείτε, φροντίστε να ελέγξετε την αντίστοιχη τεκμηρίωσή τους και να κατανοήσετε τις λεπτομέρειες των μηχανισμών επιμονής συνεδρίας. Δεν είναι κακό να το καταλάβουμε ξεκάθαρα.

## Περίληψη

Το HagiCode.Libs είναι το ενοποιημένο στρώμα αφαίρεσης που κατασκευάσαμε κατά την ανάπτυξη του HagiCode για να λύσουμε τις επαναλαμβανόμενες εργασίες μηχανικής που σχετίζονται με την ενοποίηση πολλαπλών CLI. Παρέχοντας μια συνεπή διεπαφή, ενσωματώνοντας λεπτομέρειες μεταξύ πλατφορμών και υποστηρίζοντας ευέλικτα μοτίβα ενοποίησης, μειώνει σημαντικά τη μηχανική πολυπλοκότητα της ενσωμάτωσης πολλαπλών βοηθών κωδικοποίησης AI. Πολλά μπορεί να εξαφανιστούν, αλλά η εμπειρία παραμένει.
