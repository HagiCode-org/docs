## Πώς να εφαρμόσετε την αυτόματη επανάληψη για CLI Agent όπως ο Κώδικας Claude και ο Codex

> Η φράση αυτόματη επανάληψη μοιάζει με έναν μικρό διακόπτη εναλλαγής, αλλά μόλις τη βάλετε σε ένα πραγματικό περιβάλλον μηχανικής, δεν είναι τίποτα τέτοιο. Γεια σε όλους, είμαι ο δημιουργός του HagiCode, Yu Kun. Σήμερα, δεν θέλω να ανταλλάξω άδειες κουβέντες. Θέλω να μιλήσω για το πώς θα πρέπει να γίνει στην πραγματικότητα η αυτόματη επανάληψη για Agent CLI όπως ο Claude Code και ο Codex, ώστε να μπορεί να ανακτήσει από τις εξαιρέσεις και να αποφύγει τη μεταφορά του συστήματος σε ατελείωτες επαναλαμβανόμενες εκτελέσεις.

## φόντο

Εάν εργάζεστε επίσης πρόσφατα με την κωδικοποίηση AI, πιθανότατα έχετε ήδη αντιμετωπίσει αυτού του είδους το πρόβλημα: η εργασία δεν αποτυγχάνει αμέσως, αλλά σπάει στα μισά της εκτέλεσης.

Σε ένα συνηθισμένο αίτημα HTTP, αυτό συχνά σημαίνει απλώς αποστολή του ξανά, ίσως με κάποια εκθετική υποχώρηση. Αλλά τα Agent CLI είναι διαφορετικά. Εργαλεία όπως ο Κώδικας Claude και ο Codex συνήθως εκτελούνται με τρόπο ροής, ωθώντας την έξοδο κομμάτι-κομμάτι. Κατά τη διάρκεια αυτής της διαδικασίας, ενδέχεται επίσης να συνδεθούν σε ένα νήμα, μια περίοδο λειτουργίας ή ένα διακριτικό συνέχισης. Με άλλα λόγια, το ερώτημα δεν είναι απλώς «Απέτυχε αυτό το αίτημα ή όχι;». Γίνεται:

- Εξακολουθεί να μετράει το περιεχόμενο που έχει ήδη εκδοθεί;
- Μπορεί το τρέχον πλαίσιο να συνεχίσει να λειτουργεί;
- Πρέπει να αποκατασταθεί αυτόματα αυτή η αποτυχία;
- Εάν πρέπει να ανακτηθεί, πόσο καιρό θα πρέπει να περιμένουμε πριν προσπαθήσουμε ξανά, τι πρέπει να στείλουμε κατά την επανάληψη και πρέπει να χρησιμοποιήσουμε ξανά το αρχικό περιβάλλον;

Την πρώτη φορά που πολλές ομάδες κατασκευάζουν αυτό το μέρος, γράφουν ενστικτωδώς την πιο αφελή εκδοχή: αν παρουσιαστεί κάποιο σφάλμα, δοκιμάστε άλλη μια φορά. Αυτή η ιδέα είναι απολύτως φυσική, αλλά μόλις φτάσει σε ένα πραγματικό έργο, το ένα πρόβλημα μετά το άλλο αρχίζει να εμφανίζεται.

- Ορισμένα σφάλματα είναι σαφώς προσωρινές αποτυχίες, αλλά αντιμετωπίζονται ως τελικές αποτυχίες
- Ορισμένα σφάλματα δεν αξίζει να τα ξαναδοκιμάσετε καθόλου, ωστόσο το σύστημα τα επαναλαμβάνει ξανά και ξανά
- Τα αιτήματα με νήμα και τα αιτήματα χωρίς νήμα αντιμετωπίζονται ακριβώς το ίδιο
- Η στρατηγική backoff δεν έχει όριο και τα αιτήματα παρασκηνίου υπερφορτώνονται

Κατά την ενσωμάτωση πολλών Agent CLI, το HagiCode μπήκε επίσης σε αυτές τις παγίδες. Ειδικότερα από την πλευρά του Codex, το πρώτο ζήτημα που αποκαλύψαμε ήταν ότι ένας συγκεκριμένος τύπος μηνύματος επανασύνδεσης δεν αναγνωρίστηκε ως κατάσταση τερματικού με δυνατότητα επανάληψης, επομένως ο μηχανισμός ανάκτησης δεν είχαμε ήδη ποτέ την ευκαιρία να τεθεί σε ισχύ. Για να το θέσω ξεκάθαρα, δεν ήταν ότι το σύστημα δεν είχε αυτόματη επανάληψη. Το σύστημα απλώς απέτυχε να αναγνωρίσει ότι αυτή η συγκεκριμένη αποτυχία άξιζε να επαναληφθεί.

Επομένως, το βασικό σημείο αυτού του άρθρου είναι πολύ σαφές: **η αυτόματη επανάληψη δεν είναι ένα κουμπί, αλλά μια πολυεπίπεδη σχεδίαση.**

## Γιατί η αυτόματη επανάληψη για τα CLI Agent είναι πιο δύσκολη από την κανονική επανάληψη

Αυτή είναι μια πολύ πρακτική ερώτηση, οπότε ας πάμε κατευθείαν στο συμπέρασμα: η δυσκολία της αυτόματης επανάληψης για τα Agent CLI δεν είναι "προσπαθήστε ξανά μετά από λίγα δευτερόλεπτα", αλλά "μπορεί να συνεχίσει ακόμα στο αρχικό πλαίσιο;"

Μπορείτε να το σκεφτείτε ως μια μακρά συζήτηση. Η κανονική επανάληψη του API μοιάζει περισσότερο με την επανάκληση όταν η τηλεφωνική γραμμή είναι κατειλημμένη. Η επανάληψη του Agent CLI μοιάζει περισσότερο με το σήμα που πέφτει ενώ το άλλο μέρος βρίσκεται στα μισά της πρότασης και, στη συνέχεια, πρέπει να αποφασίσετε εάν θα καλέσετε ξανά, εάν θα ξεκινήσετε από την αρχή όταν το κάνετε και εάν το άλλο μέρος εξακολουθεί να θυμάται πού σταμάτησε η συνομιλία. Αυτά δεν είναι καθόλου το ίδιο είδος μηχανικού προβλήματος.

Πιο συγκεκριμένα, υπάρχουν τέσσερις ιδιαίτερα τυπικές δυσκολίες.

### 1. Είναι σε ροή

Όταν η έξοδος έχει ήδη σταλεί στον χρήστη, δεν μπορείτε πλέον να χειριστείτε την αποτυχία με τον τρόπο που θα κάνατε με ένα συνηθισμένο αίτημα, όπου το καταπίνετε σιωπηλά και προσπαθείτε ξανά ήσυχα. Αυτό συμβαίνει επειδή το προηγούμενο περιεχόμενο έχει ήδη δει. Εάν η στρατηγική επανάληψης είναι λανθασμένη, η διεπαφή μπορεί εύκολα να εμφανίσει διπλό κείμενο και ασυνεπή κατάσταση και ο κύκλος ζωής των κλήσεων εργαλείων μπορεί επίσης να μπερδευτεί. Αυτό δεν είναι μεταφυσική. Είναι μηχανική.

### 2. Συνήθως συνδέεται με το πλαίσιο συνεδρίας

Οι πάροχοι όπως το Codex συνδέονται σε ένα νήμα και οι υλοποιήσεις όπως ο Κώδικας Claude έχουν επίσης έναν στόχο συνέχειας ή ένα ισοδύναμο περιβάλλον με δυνατότητα επαναφοράς. Η πραγματική προϋπόθεση για την αυτόματη επανάληψη δεν είναι απλώς το σφάλμα να μοιάζει με προσωρινή αποτυχία, αλλά και να υπάρχει ακόμα ένας φορέας που επιτρέπει τη συνέχιση αυτής της εκτέλεσης.

### 3. Δεν αξίζει κάθε σφάλμα να ξαναδοκιμάζεται

Το jitter δικτύου, το χρονικό όριο αδράνειας SSE και οι προσωρινές αποτυχίες ανάντη συνήθως αξίζουν άλλη μια προσπάθεια. Αλλά αν αυτό που αντιμετωπίζετε είναι αποτυχία ελέγχου ταυτότητας, χαμένο πλαίσιο ή ένας πάροχος που δεν έχει καθόλου δυνατότητα συνέχισης, τότε η επανάληψη συνήθως δεν είναι ανάκτηση. Είναι παραγωγή θορύβου.

### 4. Χρειάζεται όρια

Η απεριόριστη αυτόματη επανάληψη είναι σχεδόν πάντα λάθος. Οι τάσεις της τεχνολογίας μπορεί να είναι θορυβώδεις για λίγο, αλλά οι νόμοι της μηχανικής συχνά παραμένουν σταθεροί για πολλά χρόνια. Ένα από αυτά είναι ότι η αποκατάσταση αποτυχίας πρέπει να έχει όρια. Το σύστημα πρέπει να γνωρίζει πόσες φορές μπορεί να προσπαθήσει ξανά το πολύ, πόσο καιρό θα πρέπει να περιμένει κάθε φορά και πότε πρέπει να σταματήσει και να παραδεχτεί ότι αυτό πραγματικά δεν πρόκειται να ανακάμψει.

Εξαιτίας αυτών των χαρακτηριστικών, το HagiCode τελικά δεν εφάρμοσε την αυτόματη επανάληψη ως μερικές γραμμές `try/catch` μέσα σε συγκεκριμένο πάροχο. Αντίθετα, το εξάγαμε σε ένα επίπεδο κοινής δυνατότητας. Στο τέλος, τα προβλήματα μηχανικής πρέπει να επιλυθούν ακόμη με μηχανικές μεθόδους.

## Προσέγγιση της HagiCode: Τραβήξτε ξανά την προσπάθεια έξω από τον πάροχο

Η τρέχουσα υλοποίηση του HagiCode σε πραγματικό κόσμο μπορεί να συμπιεστεί σε μία πρόταση:

**Το κοινό επίπεδο διαχειρίζεται ομοιόμορφα τη ροή επανάληψης δοκιμής και κάθε συγκεκριμένος πάροχος είναι υπεύθυνος μόνο για την απάντηση σε δύο ερωτήσεις: αξίζει αυτή η κατάσταση τερματικού να επαναληφθεί και μπορεί να συνεχιστεί το τρέχον πλαίσιο;**

Αυτό δεν είναι περίπλοκο, αλλά είναι κρίσιμο. Μόλις διαχωριστούν οι αρμοδιότητες με αυτόν τον τρόπο, ο Κώδικας Claude, ο Codex και ακόμη και άλλα Agent CLI μπορούν να χρησιμοποιήσουν ξανά τον ίδιο σκελετό. Τα μοντέλα θα αλλάξουν, τα εργαλεία θα εξελιχθούν, οι ροές εργασίας θα αναβαθμιστούν, αλλά τα θεμέλια της μηχανικής παραμένει εκεί.

### Επίπεδο 1: Χρησιμοποιήστε έναν ενοποιημένο συντονιστή για τη διαχείριση του βρόχου επανάληψης

Το βασικό τμήμα υλοποίησης του έργου μοιάζει περίπου με αυτό:

```csharp
internal static class ProviderErrorAutoRetryCoordinator
{
    public static async IAsyncEnumerable<CliMessage> ExecuteAsync(
        string prompt,
        ProviderErrorAutoRetrySettings? settings,
        Func<string, IAsyncEnumerable<CliMessage>> executeAttemptAsync,
        Func<bool> canRetryInSameContext,
        Func<TimeSpan, CancellationToken, Task> delayAsync,
        Func<CliMessage, bool> isRetryableTerminalMessage,
        [EnumeratorCancellation] CancellationToken cancellationToken)
    {
        var normalizedSettings = ProviderErrorAutoRetrySettings.Normalize(settings);
        var retrySchedule = normalizedSettings.Enabled
            ? normalizedSettings.GetRetrySchedule()
            : [];

        for (var attempt = 0; ; attempt++)
        {
            var attemptPrompt = attempt == 0
                ? prompt
                : ProviderErrorAutoRetrySettings.ContinuationPrompt;

            CliMessage? terminalFailure = null;

            await foreach (var message in executeAttemptAsync(attemptPrompt)
                               .WithCancellation(cancellationToken))
            {
                if (isRetryableTerminalMessage(message))
                {
                    terminalFailure = message;
                    break;
                }

                yield return message;
            }

            if (terminalFailure is null)
            {
                yield break;
            }

            if (attempt >= retrySchedule.Count || !canRetryInSameContext())
            {
                yield return terminalFailure;
                yield break;
            }

            await delayAsync(retrySchedule[attempt], cancellationToken);
        }
    }
}
```

Αυτό που κάνει αυτός ο κώδικας είναι πραγματικά πολύ απλό, αλλά και πολύ αποτελεσματικό.

- Μην περνάτε απευθείας τις ενδιάμεσες αστοχίες στην αρχή. ο συντονιστής αποφασίζει εάν η ανάκτηση είναι ακόμα δυνατή
- Μόνο όταν εξαντληθεί ο προϋπολογισμός επανάληψης, η τελική αποτυχία επιστρέφει πραγματικά στο ανώτερο επίπεδο
- Ξεκινώντας από τη δεύτερη προσπάθεια, το αρχικό μήνυμα δεν αποστέλλεται πλέον. Αντ' αυτού, αποστέλλεται ομοιόμορφα μια προτροπή συνέχειας

Αυτός είναι ο λόγος για τον οποίο συνέχισα να τονίζω νωρίτερα ότι η αυτόματη επανάληψη δεν είναι απλώς "υποβολή του αιτήματος ξανά". Δεν είναι απλώς επιδιόρθωση ενός κλάδου εξαίρεσης. Διαχειρίζεται τον κύκλο ζωής μιας εκτέλεσης. Αυτό μπορεί να ακούγεται σαν γλώσσα διαχείρισης προϊόντος, αλλά με όρους μηχανικής, αυτό ακριβώς είναι.

### Επίπεδο 2: Στιγμιότυπο της πολιτικής επανάληψης

Ένα άλλο ζήτημα που είναι πολύ εύκολο να παραβλεφθεί είναι το εξής: ποιος αποφασίζει εάν είναι ενεργοποιημένη η αυτόματη επανάληψη για αυτό το αίτημα;

Η απάντηση του HagiCode δεν είναι να εξαρτάται από κάποια "τρέχουσα παγκόσμια διαμόρφωση", αλλά να μετατρέψει την πολιτική σε στιγμιότυπο και να την αφήσει να ταξιδέψει μαζί με αυτό το αίτημα. Με αυτόν τον τρόπο, η ουρά συνεδρίας, η επιμονή μηνυμάτων, η προώθηση εκτέλεσης και η προσαρμογή του παρόχου δεν θα χάσουν την πολιτική στην πορεία. Μια επιτυχημένη εκτέλεση δεν είναι σύστημα. Η διαρκής επιτυχία είναι ένα σύστημα.

Η βασική δομή μπορεί να απλοποιηθεί ως εξής:

```csharp
public sealed record ProviderErrorAutoRetrySnapshot
{
    public const string DefaultStrategy = "default";

    public bool Enabled { get; init; }

    public string Strategy { get; init; } = DefaultStrategy;

    public static ProviderErrorAutoRetrySnapshot Normalize(bool? enabled, string? strategy)
    {
        return new ProviderErrorAutoRetrySnapshot
        {
            Enabled = enabled ?? true,
            Strategy = string.IsNullOrWhiteSpace(strategy)
                ? DefaultStrategy
                : strategy.Trim()
        };
    }
}
```

Στη συνέχεια, στην πλευρά εκτέλεσης, αντιστοιχίζεται στο αντικείμενο ρυθμίσεων που καταναλώνεται πραγματικά από τον πάροχο. Η αξία αυτής της προσέγγισης είναι πολύ άμεση:

- Το επιχειρηματικό επίπεδο αποφασίζει εάν θα πρέπει να επιτρέπεται η επανάληψη
- Ο χρόνος εκτέλεσης αποφασίζει πώς πρέπει να γίνει η επανάληψη

Κάθε πλευρά διαχειρίζεται τη δική της ανησυχία χωρίς να συγκρούεται με την άλλη. Πολλά προβλήματα δεν είναι αδύνατο να λυθούν. Το κόστος τους απλά δεν έχει διευκρινιστεί. Η μετατροπή του συμβολαίου σε στιγμιότυπο είναι ουσιαστικά ένας τρόπος υπολογισμού αυτού του κόστους εκ των προτέρων.

### Επίπεδο 3: Οι πάροχοι αποφασίζουν μόνο τη βιωσιμότητα της κατάστασης του τερματικού και του περιβάλλοντος

Μόλις φτάσουμε στον συγκεκριμένο πάροχο Claude Code ή Codex, η ευθύνη εδώ στην πραγματικότητα γίνεται πολύ λεπτή. Μπορείτε να το σκεφτείτε ως βελτίωση, όχι ως αντικατάσταση.

Λαμβάνοντας ως παράδειγμα το Codex, όταν συνδέεται με τον κοινόχρηστο συντονιστή, χρειάζεται πραγματικά μόνο να παρέχει τρία πράγματα:

```csharp
await foreach (var message in ProviderErrorAutoRetryCoordinator.ExecuteAsync(
                   prompt,
                   options.ProviderErrorAutoRetry,
                   retryPrompt => ExecuteCodexAttemptAsync(...),
                   () => !string.IsNullOrWhiteSpace(resolvedThreadId),
                   DelayAsync,
                   IsRetryableTerminalFailure,
                   cancellationToken))
{
    yield return message;
}
```

Θα παρατηρήσετε ότι οι συγκεκριμένες αποφάσεις για τον πάροχο είναι στην πραγματικότητα μόνο αυτές οι δύο:

- `IsRetryableTerminalFailure`
- `canRetryInSameContext`

Το Codex ελέγχει εάν το νήμα μπορεί να συνεχίσει, ενώ ο Κώδικας Claude ελέγχει εάν ο στόχος συνέχειας εξακολουθεί να υπάρχει. Η πολιτική υποχώρησης, ο αριθμός επαναλήψεων προσπαθειών και οι προτροπές παρακολούθησης δεν πρέπει να επινοούνται εκ νέου από κάθε πάροχο ξεχωριστά.

Μόλις διαχωριστεί αυτό το επίπεδο, το κόστος της ενσωμάτωσης περισσότερων CLI στο HagiCode μειώνεται πολύ. Δεν χρειάζεται να αντιγράψετε ένα ολόκληρο μηχάνημα κατάστασης επανάληψης. Χρειάζεται μόνο να συνδέσετε τις οριακές συνθήκες αυτού του παρόχου. Το να γράφεις γρήγορα δεν είναι το ίδιο με το να γράφεις γερά. Το να μπορείς να συνδέσεις κάτι δεν είναι το ίδιο με το να το συνδέσεις καλά. Το να το θέσετε σε λειτουργία δεν είναι επίσης το ίδιο με το να το κάνετε συντηρήσιμο με την πάροδο του χρόνου.

## Ένα εύκολο λάθος: Μην αντιμετωπίζετε κάθε σφάλμα ως επαναδοκιμάσιμο

Σε αυτήν την ανάλυση, το σημείο που θέλω περισσότερο να ξεχωρίσω δεν είναι "πώς να εφαρμόσετε την επανάληψη", αλλά "πώς να αποφύγετε τις λανθασμένες επαναλήψεις".

Το αρχικό σημείο εισόδου στο πρόβλημα ήταν ότι το Codex απέτυχε να αναγνωρίσει ένα μήνυμα επανασύνδεσης. Με τη διαίσθηση, πολλοί άνθρωποι θα επέλεγαν τη μικρότερη δυνατή διόρθωση: προσθέστε ένα ακόμη πρόθεμα συμβολοσειράς στη λίστα επιτρεπόμενων. Αυτή η ιδέα δεν είναι ακριβώς λανθασμένη, αλλά μοιάζει περισσότερο με λύση επίδειξης παρά με μακροπρόθεσμη συντηρήσιμη λύση.

Από την τρέχουσα εφαρμογή του HagiCode, το σύστημα έχει ήδη κάνει ένα βήμα προς μια πιο ισχυρή κατεύθυνση. Δεν κοιτάζει πλέον μόνο μια κυριολεκτική χορδή. Αντίθετα, παραδίδει ομοιόμορφα τις ανακτήσιμες τερματικές καταστάσεις στον κοινόχρηστο συντονιστή. Τα οφέλη είναι προφανή:

- Είναι λιγότερο πιθανό να αποτύχει εντελώς λόγω μιας μικρής αλλαγής διατύπωσης σε ένα μήνυμα
- Η δοκιμαστική κάλυψη μπορεί να δημιουργηθεί γύρω από τον φάκελο κατάστασης τερματικού και όχι σε μια γραμμή κειμένου με σκληρό κώδικα
- Η λογική επανάληψης γίνεται πιο συνεπής στον ίδιο πάροχο

Φυσικά, πρέπει να υπάρχει ένα σταθερό όριο εδώ: το να είσαι πιο γενικός δεν σημαίνει να είσαι πιο επιτρεπτικός. **Εάν το τρέχον περιβάλλον δεν μπορεί να συνεχιστεί, τότε ακόμα κι αν το σφάλμα μοιάζει με προσωρινή αποτυχία, δεν θα πρέπει να επαναληφθεί στα τυφλά.**

Αυτό το σημείο είναι κρίσιμο. Αυτό που πραγματικά κάνει τους ανθρώπους να εμπιστεύονται ένα σύστημα δεν είναι ότι λειτουργεί περιστασιακά, αλλά ότι είναι αξιόπιστο τις περισσότερες φορές. Εάν μια ροή μπορεί να διατηρηθεί μόνο από ειδικούς, τότε απέχει ακόμη πολύ από την πραγματική υιοθέτηση.

## Τα τρία πιο πολύτιμα μαθήματα που πρέπει να κρατήσετε στην πράξη

Σε αυτό το σημείο, είναι λογικό να αρχίσουμε να επαναφέρουμε τη συζήτηση στην πρακτική υλοποίησης. Εάν σχεδιάζετε να δημιουργήσετε μια παρόμοια δυνατότητα στο δικό σας έργο, αυτοί είναι οι τρεις κανόνες που συνιστώ ανεπιφύλακτα να προστατεύσετε πρώτα.

### 1. Ο προϋπολογισμός επανάληψης πρέπει να έχει όρια

Ο τρέχων προεπιλεγμένος ρυθμός επιστροφής του HagiCode είναι:

- 10 δευτερόλεπτα
- 20 δευτερόλεπτα
- 60 δευτερόλεπτα

Αυτός ο ρυθμός μπορεί να μην ταιριάζει σε κάθε σύστημα, αλλά η ύπαρξη ορίων πρέπει να παραμείνει. Διαφορετικά, η αυτόματη επανάληψη παύει γρήγορα να είναι μηχανισμός ανάκτησης και μετατρέπεται σε ενισχυτή πρόσπτωσης. Μη βιαστείτε να του δώσετε ένα εντυπωσιακό όνομα. Πρώτα βεβαιωθείτε ότι το πράγμα μπορεί να επιβιώσει δύο επαναλήψεις μέσα σε μια πραγματική ομάδα.

### 2. Η προτροπή συνέχισης πρέπει να είναι ενοποιημένη

Το έργο χρησιμοποιεί μια σταθερή προτροπή συνέχισης, έτσι ώστε οι μεταγενέστερες προσπάθειες να ακολουθήσουν ξεκάθαρα το μονοπάτι της συνέχισης του τρέχοντος πλαισίου αντί να ξεκινήσουν ένα ολοκαίνουργιο πλήρες αίτημα. Αυτή η δυνατότητα δεν είναι εντυπωσιακή, αλλά όταν δημιουργείτε ένα πραγματικό έργο, δεν μπορείτε να το κάνετε χωρίς αυτό. Πολλά πράγματα που μοιάζουν με μαγικά είναι, μόλις διαλυθούν, απλώς μια γυαλισμένη μηχανική διαδικασία.

### 3. Τόσο η κοινόχρηστη βιβλιοθήκη όσο και το επίπεδο προσαρμογέα χρειάζονται αντικατοπτρισμένες δοκιμές

Θέλω ιδιαίτερα να πω λίγα περισσότερα για αυτό το σημείο. Πολλές ομάδες θα γράψουν ένα επίπεδο δοκιμών στον κοινόχρηστο χρόνο εκτέλεσης και πιστεύουν ότι μάλλον είναι αρκετό. Δεν είναι.

Ο λόγος που νιώθω σχετικά σίγουρος για την υλοποίηση του HagiCode είναι ότι και τα δύο επίπεδα έχουν δοκιμαστική κάλυψη:

- Ο κοινόχρηστος πάροχος ελέγχει εάν όντως συνέβη η αυτόματη συνέχιση
- Το επίπεδο προσαρμογέα ελέγχει εάν τα τελικά σφάλματα και τα μηνύματα ροής διατηρήθηκαν σωστά

Αυτή τη φορά ξαναέτρεξα επίσης δύο σχετικές ομάδες δοκιμών, και οι 31 περιπτώσεις εξετάσεων πέρασαν και στις δύο. Αυτό το αποτέλεσμα από μόνο του δεν αποδεικνύει ότι ο σχεδιασμός είναι τέλειος, αλλά αποδεικνύει τουλάχιστον ένα πράγμα: η τρέχουσα αυτόματη επανάληψη δεν είναι σχέδιο χαρτιού. Είναι μια δυνατότητα που περιορίζεται τόσο από κώδικα όσο και από δοκιμές. Η συζήτηση είναι φθηνή. Δείξε μου τον κωδικό. Εδώ ταιριάζει απόλυτα.

## Περίληψη

Αν ολόκληρο το άρθρο έπρεπε να συμπιεστεί σε μία πρόταση, θα ήταν η εξής:

**Για τα CLI Agent όπως ο Κώδικας Claude και ο Codex, η αυτόματη επανάληψη δεν θα πρέπει να εφαρμόζεται ως τοπικό τέχνασμα που κρύβεται μέσα σε έναν πάροχο. Θα πρέπει να κατασκευαστεί ως συνδυασμός ενός κοινόχρηστου συντονιστή, στιγμιότυπου πολιτικής, ελέγχων βιωσιμότητας περιβάλλοντος και αντικατοπτρισμένων δοκιμών.**

Τα πλεονεκτήματα του να το κάνεις με αυτόν τον τρόπο είναι πολύ πρακτικά:

- Η λογική γράφεται μία φορά και επαναχρησιμοποιείται σε πολλούς παρόχους
- Το αν επιτρέπεται σε ένα αίτημα για επανάληψη μπορεί να ταξιδέψει σταθερά με την αλυσίδα εκτέλεσης
- Συνεχίστε να εκτελείτε όταν υπάρχει πλαίσιο και σταματήστε εγκαίρως όταν δεν υπάρχει
- Αυτό που τελικά βλέπει το frontend είναι μια σταθερή ολοκληρωμένη κατάσταση ή μια αποτυχημένη κατάσταση, όχι ένα σωρό εγκαταλειμμένου ενδιάμεσου θορύβου

Αυτή η λύση γυαλίστηκε σιγά σιγά ενώ η HagiCode ενσωμάτωνε πολλαπλά Agent CLI σε πραγματικά σενάρια. Ποιος λέει ότι ο προγραμματισμός με τη βοήθεια AI δεν είναι η νέα εποχή του προγραμματισμού ζευγών; Τα μοντέλα σάς βοηθούν να ξεκινήσετε, να ολοκληρώσετε τον κώδικα και να διακλαδώσετε, αλλά αυτό που συχνά καθορίζει το ανώτερο όριο της εμπειρίας είναι ακόμα το πλαίσιο, η διαδικασία και οι περιορισμοί.
