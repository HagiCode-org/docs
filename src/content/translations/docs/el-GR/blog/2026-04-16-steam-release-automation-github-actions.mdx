## Πώς να αυτοματοποιήσετε τις εκδόσεις Steam με το GitHub Actions

> Αυτό το άρθρο κοινοποιεί την πλήρη λύση που εφαρμόσαμε για αυτοματοποιημένες εκδόσεις Steam στο έργο HagiCode Desktop, καλύπτοντας τη ροή αυτοματισμού από άκρο σε άκρο από το GitHub Release στην πλατφόρμα Steam, συμπεριλαμβανομένων βασικών τεχνικών λεπτομερειών όπως ο έλεγχος ταυτότητας Steam Guard και οι μεταφορτώσεις Depot σε πολλές πλατφόρμες.

## φόντο

Η ροή εργασίας απελευθέρωσης στο Steam είναι στην πραγματικότητα αρκετά διαφορετική από την παραδοσιακή διανομή εφαρμογών. Το Steam έχει το δικό του πλήρες σύστημα παράδοσης ενημερώσεων. Οι προγραμματιστές πρέπει να χρησιμοποιήσουν το SteamCMD για να ανεβάσουν τεχνουργήματα κατασκευής στο δίκτυο CDN του Steam, αντί να αποθέσουν απλώς έναν σύνδεσμο λήψης όπως σε άλλες πλατφόρμες.

Το έργο HagiCode Desktop ετοιμάζεται για μια κυκλοφορία στο Steam, η οποία εισήγαγε μερικές νέες προκλήσεις στη ροή εργασιών κυκλοφορίας μας:

1. Χρειαζόμασταν να μετατρέψουμε υπάρχοντα τεχνουργήματα κατασκευής σε μορφή συμβατή με το Steam.
2. Έπρεπε να τα ανεβάσουμε στο Steam μέσω του SteamCMD.
3. Έπρεπε επίσης να χειριστούμε τον έλεγχο ταυτότητας Steam Guard.
4. Χρειαζόταν να υποστηρίξουμε μεταφορτώσεις Depot σε πολλές πλατφόρμες για Linux, Windows και macOS.
5. Θέλαμε μια πλήρως αυτοματοποιημένη ροή από το GitHub Release στο Steam.

Το έργο είχε ήδη εφαρμόσει τη "λειτουργία φορητής έκδοσης", η οποία επιτρέπει στην εφαρμογή να ανιχνεύει σταθερά ωφέλιμα φορτία υπηρεσιών συσκευασμένα στο `extra` τηλεφωνικός κατάλογος. Στόχος μας ήταν να ενσωματώσουμε απρόσκοπτα αυτή τη λειτουργία φορητής έκδοσης με τη διανομή Steam.

Το HagiCode είναι ένας βοηθός κωδικοποίησης AI που υποστηρίζει τη χρήση επιτραπέζιων υπολογιστών. Δεδομένου ότι εργαζόμαστε ενεργά για την κυκλοφορία στο Steam, έπρεπε να δημιουργήσουμε μια αξιόπιστη αυτοματοποιημένη ροή εργασίας.

## Αρχιτεκτονικός Σχεδιασμός

Ο πυρήνας ολόκληρης της διαδικασίας έκδοσης Steam είναι μια ροή εργασίας GitHub Actions που χωρίζει τη διαδικασία σε τρία κύρια στάδια:

```
┌─────────────────────────────────────────────────────────────┐
│ GitHub Actions Workflow (Steam Release)                    │
├─────────────────────────────────────────────────────────────┤
│ 1. Preparation Stage:                                      │
│    - Check out portable-version code                       │
│    - Download build artifacts from GitHub Release          │
│    - Extract and prepare the Steam content directory       │
│                                                             │
│ 2. SteamCMD Setup:                                         │
│    - Install or reuse SteamCMD                             │
│    - Authenticate with Steam Guard                         │
│                                                             │
│ 3. Release Stage:                                          │
│    - Generate Depot VDF configuration files                │
│    - Generate App Build VDF configuration files            │
│    - Invoke SteamCMD to upload to Steam                    │
└─────────────────────────────────────────────────────────────┘
```

Αυτός ο σχεδιασμός προσφέρει πολλά πλεονεκτήματα:
- Επαναχρησιμοποιεί τα υπάρχοντα τεχνουργήματα του GitHub Release και αποφεύγει την εκ νέου κατασκευή των ίδιων εξόδων.
- Χρησιμοποιεί αυτο-φιλοξενούμενους δρομείς για ασφαλή απομόνωση.
- Υποστηρίζει εναλλαγή μεταξύ της λειτουργίας προεπισκόπησης και των κλάδων επίσημης έκδοσης.
- Περιλαμβάνει πλήρη χειρισμό σφαλμάτων και καταγραφή, γεγονός που καθιστά ευκολότερη τη διάγνωση των αστοχιών.

## Υλοποίηση ροής εργασιών

### Σχεδιασμός παραμέτρων ενεργοποίησης

Η ροή εργασίας μας υποστηρίζει τις ακόλουθες βασικές παραμέτρους:

```yaml
inputs:
  release:           # Portable Version release tag
    description: 'Version tag to release (for example v1.0.0)'
    required: true
  steam_preview:     # Whether to generate a preview build
    description: 'Whether to use preview mode'
    required: false
    default: 'false'
  steam_branch:      # Steam branch to set live
    description: 'Target Steam branch'
    required: false
    default: 'preview'
  steam_description: # Build description override
    description: 'Build description'
    required: false
```

### Αυτο-φιλοξενούμενη διαμόρφωση Runner

Για λόγους ασφαλείας, χρησιμοποιούμε αυτο-φιλοξενούμενο δρομέα με το `steam` ετικέτα:

```yaml
runs-on:
  - self-hosted
  - Linux
  - X64
  - steam
```

Αυτό διασφαλίζει ότι οι εκδόσεις Steam εκτελούνται σε έναν αποκλειστικό δρομέα και διατηρεί τα ευαίσθητα διαπιστευτήρια με ασφάλεια απομονωμένα.

### Έλεγχος Συγχρονισμού

Για να αποτρέψουμε την παρεμβολή των εκδόσεων της ίδιας έκδοσης μεταξύ τους, διαμορφώσαμε τον έλεγχο ταυτόχρονης χρήσης:

```yaml
concurrency:
  group: portable-version-steam-${{ github.event.inputs.release }}
  cancel-in-progress: false
```

Παρατηρήστε ότι `cancel-in-progress: false` έχει ρυθμιστεί εδώ επειδή οι εκδόσεις Steam μπορεί να διαρκέσουν λίγο και δεν θέλουμε μια νέα εκτέλεση για να ακυρωθεί μια που ανεβάζεται ήδη.

## Υλοποίηση βασικού σεναρίου

### Προετοιμασία εισόδων έκδοσης

Το `prepare-steam-release-input.mjs` Το σενάριο είναι υπεύθυνο για την προετοιμασία των εισόδων που απαιτούνται για την κυκλοφορία:

```javascript
// Download the GitHub Release build manifest and artifact inventory
const buildManifest = await downloadBuildManifest(releaseTag);
const artifactInventory = await downloadArtifactInventory(releaseTag);

// Download compressed archives for each platform
for (const platform of ['linux-x64', 'win-x64', 'osx-universal']) {
  const artifactUrl = getArtifactUrl(artifactInventory, platform);
  await downloadArtifact(artifactUrl, platform);
}

// Extract into the Steam content directory structure
await extractToSteamContent(sources, contentRoot);
```

### Έλεγχος ταυτότητας Steam Guard

Το Steam απαιτεί την προστασία των λογαριασμών με το Steam Guard, επομένως εφαρμόσαμε έναν αλγόριθμο δημιουργίας κώδικα που βασίζεται σε κοινό μυστικό:

```javascript
function generateSteamGuardCode(sharedSecret, timestamp = Date.now()) {
  const secret = decodeSharedSecret(sharedSecret);
  const time = Math.floor(timestamp / 1000 / 30);

  const timeBuffer = Buffer.alloc(8);
  timeBuffer.writeBigUInt64BE(BigInt(time));

  // Use HMAC-SHA1 to generate a time-based one-time code
  const hash = crypto.createHmac('sha1', secret)
    .update(timeBuffer)
    .digest();

  // Convert it into a 5-character Steam Guard code
  const code = steamGuardCode(hash);
  return code;
}
```

Αυτή η υλοποίηση βασίζεται στον μηχανισμό TOTP (Time-based One-Time Password) της Steam Guard, που δημιουργεί έναν νέο κωδικό επαλήθευσης κάθε 30 δευτερόλεπτα.

### Δημιουργία διαμόρφωσης VDF

VDF (Valve Data Format) είναι η μορφή διαμόρφωσης που χρησιμοποιείται από το Steam. Πρέπει να δημιουργήσουμε δύο τύπους αρχείων VDF:

Το **Depot VDF** χρησιμοποιείται για τη διαμόρφωση περιεχομένου για κάθε πλατφόρμα:

```javascript
function buildDepotVdf(depotId, contentRoot) {
  return [
    '"DepotBuildConfig"',
    '{',
    `  "DepotID" "${escapeVdf(depotId)}"`,
    `  "ContentRoot" "${escapeVdf(contentRoot)}"`,
    '  "FileMapping"',
    '  {',
    '    "LocalPath" "*"',
    '    "DepotPath" "."',
    '    "recursive" "1"',
    '  }',
    '}'
  ].join('\n');
}
```

Το **App Build VDF** χρησιμοποιείται για τη διαμόρφωση ολόκληρης της έκδοσης εφαρμογής:

```javascript
function buildAppBuildVdf(appId, depotBuilds, description, setLive) {
  const vdf = [
    '"appbuild"',
    '{',
    `  "appid" "${appId}"`,
    `  "desc" "${escapeVdf(description)}"`,
    `  "contentroot" "${escapeVdf(contentRoot)}"`,
    '  "buildoutput" "build_output"',
    '  "depots"',
    '  {'
  ];

  for (const [depotId, depotVdfPath] of Object.entries(depotBuilds)) {
    vdf.push(`    "${depotId}" "${depotVdfPath}"`);
  }

  if (setLive) {
    vdf.push(`  }`);
    vdf.push(`  "setlive" "${setLive}"`);
  }

  vdf.push('}');
  return vdf.join('\n');
}
```

### Επίκληση SteamCMD

Τέλος, η μεταφόρτωση πραγματοποιείται με την επίκληση του SteamCMD:

```javascript
await runCommand(steamcmdPath, [
  '+login', steamUsername, steamPassword, steamGuardCode,
  '+run_app_build', appBuildPath,
  '+quit'
]);
```

Αυτό είναι το τελευταίο άλμα σε ολόκληρη τη ροή εργασίας. Μόλις πετύχει, η απελευθέρωση ολοκληρώνεται.

## Διαχείριση αποθήκης πολλαπλών πλατφορμών

Το Steam χρησιμοποιεί το σύστημα Depot για τη διαχείριση περιεχομένου για διαφορετικές πλατφόρμες. Υποστηρίζουμε τρεις κύριες αποθήκες:

| Πλατφόρμα | Αναγνωριστικό αποθήκης | Υποστήριξη Αρχιτεκτονικής |
|-----|-----------|---------|
| Linux | `linux-x64` | x64_64 |
| Windows | `win-x64` | x64_64 |
| macOS | `osx-universal` | universal, x64_64, arm64 |

Κάθε Depot έχει τον δικό του κατάλογο περιεχομένου και το δικό του αρχείο διαμόρφωσης VDF. Αυτό διασφαλίζει ότι οι χρήστες σε διαφορετικές πλατφόρμες κατεβάζουν μόνο το περιεχόμενο που πραγματικά χρειάζονται.

## Απελευθέρωση ροής εργασιών

### Βήμα 1: Προετοιμάστε την έκδοση του GitHub

Πρώτα, δημιουργήστε μια έκδοση GitHub στο `portable-version` αποθετήριο που περιλαμβάνει:
- Συμπιεσμένα αρχεία για κάθε πλατφόρμα
- Δημιουργία δήλωσης (`{tag}.build-manifest.json`)
- απογραφή αντικειμένων (`{tag}.artifact-inventory.json`)

### Βήμα 2: Ενεργοποιήστε τη ροή εργασιών έκδοσης Steam

Ενεργοποιήστε τη ροή εργασίας με μη αυτόματο τρόπο μέσω των ενεργειών GitHub και συμπληρώστε τις απαιτούμενες παραμέτρους:
- `release`: η ετικέτα έκδοσης προς δημοσίευση, όπως π.χ `v1.0.0`
- `steam_branch`: ο κλάδος στόχος, όπως π.χ `preview` ή `public`
- `steam_preview`: εάν θα χρησιμοποιηθεί η λειτουργία προεπισκόπησης

### Βήμα 3: Εκτελέστε αυτόματα τη ροή απελευθέρωσης

Η ροή εργασίας εκτελεί αυτόματα τα ακόλουθα βήματα:

1. Λήψη και εξαγωγή τεχνουργημάτων GitHub Release.
2. Εγκαταστήστε ή ενημερώστε το SteamCMD.
3. Δημιουργήστε αρχεία διαμόρφωσης Steam VDF.
4. Έλεγχος ταυτότητας με το Steam Guard.
5. Ανεβάστε περιεχόμενο στο Steam CDN.
6. Ρυθμίστε ζωντανά τον καθορισμένο κλάδο.

Μόλις ολοκληρωθεί αυτή η ακολουθία, καλύπτεται ολόκληρη η διαδρομή απελευθέρωσης.

Διαμορφώστε τα ακόλουθα μυστικά στις ρυθμίσεις του αποθετηρίου GitHub:

| Μυστικό Όνομα | Περιγραφή |

Δεν υπάρχει τίποτα ιδιαίτερα ασυνήθιστο σε αυτές τις ρυθμίσεις. Χρειάζεστε απλώς όλες τις αναμενόμενες τιμές στη θέση τους.

## Βέλτιστες Πρακτικές

### Διαχείριση ελέγχου ταυτότητας Steam Guard

Κατά την πρώτη εκτέλεση, πρέπει να εισαγάγετε τον κωδικό Steam Guard με μη αυτόματο τρόπο. Μετά από αυτό, συνιστάται να διαμορφώσετε το κοινό μυστικό, ώστε ο κώδικας να μπορεί να δημιουργηθεί αυτόματα. Αυτό αποφεύγει τη χειροκίνητη παρέμβαση σε κάθε έκδοση.

Το SteamCMD αποθηκεύει το διακριτικό σύνδεσης και μπορεί να το χρησιμοποιήσει ξανά σε μεταγενέστερες εκτελέσεις. Πρέπει ακόμα να παρακολουθείτε τη λήξη του διακριτικού, επειδή απαιτείται εκ νέου έλεγχος ταυτότητας μετά τη λήξη του διακριτικού.

### Δομή καταλόγου περιεχομένου

Βεβαιωθείτε ότι η δομή του καταλόγου περιεχομένου Steam είναι σωστή:

```
steam-content/
├── linux-x64/     # Linux platform content
├── win-x64/       # Windows platform content
└── osx-universal/ # macOS universal binary content
```

Κάθε κατάλογος πρέπει να περιέχει τα πλήρη αρχεία εφαρμογής για την αντίστοιχη πλατφόρμα του.

### Χρήση λειτουργίας προεπισκόπησης

Η λειτουργία προεπισκόπησης δεν ρυθμίζει κανένα κλάδο ζωντανό, επομένως είναι κατάλληλος για δοκιμή και επικύρωση:

```yaml
if [ "$STEAM_PREVIEW_INPUT" = 'true' ]; then
  cmd+=(--preview)
fi
```

Αυτό σας επιτρέπει να ανεβάσετε πρώτα στο Steam για επαλήθευση και, στη συνέχεια, να μεταβείτε στο επίσημο υποκατάστημα αφού ολοκληρωθούν τα πάντα.

### Διαχείριση σφαλμάτων και καταγραφή

Τα σενάρια περιλαμβάνουν πλήρη χειρισμό και καταγραφή σφαλμάτων:
- Επιβεβαιώστε ότι υπάρχει η έκδοση GitHub.
- Ελέγξτε τα απαιτούμενα αρχεία μεταδεδομένων.
- Βεβαιωθείτε ότι υπάρχει περιεχόμενο πλατφόρμας.
- Δημιουργήστε μια συνοπτική αναφορά GitHub Actions.

Αυτές οι πληροφορίες είναι εξαιρετικά πολύτιμες τόσο για τον εντοπισμό σφαλμάτων όσο και για τον έλεγχο.

### Διαχείριση αντικειμένων

Η ροή εργασίας δημιουργεί δύο είδη τεχνουργημάτων:
- `portable-steam-release-preparation-{tag}`: μεταδεδομένα προετοιμασίας έκδοσης
- `portable-steam-build-metadata-{tag}`: Μεταδεδομένα δημιουργίας Steam

Αυτά τα τεχνουργήματα μπορούν να χρησιμοποιηθούν για μεταγενέστερο έλεγχο και εντοπισμό σφαλμάτων. Μια περίοδος διατήρησης 30 ημερών είναι μια πρακτική προεπιλογή.

## Χρήση σε πραγματικό κόσμο

Στο έργο HagiCode, αυτή η ροή εργασίας αυτοματοποιημένης έκδοσης έχει ήδη εκτελεστεί με επιτυχία σε πολλές εκδόσεις. Ολόκληρη η διαδρομή από το GitHub Release στην πλατφόρμα Steam είναι πλήρως αυτοματοποιημένη και δεν απαιτεί χειροκίνητη παρέμβαση.

Αυτό βελτίωσε σημαντικά τόσο την αποτελεσματικότητα όσο και την αξιοπιστία της απελευθέρωσης. Στο παρελθόν, η μη αυτόματη δημοσίευση μιας έκδοσης χρειαζόταν περισσότερα από 30 λεπτά. Τώρα η όλη διαδικασία ολοκληρώνεται σε λίγα μόνο λεπτά.

Το πιο σημαντικό, η αυτοματοποιημένη ροή εργασίας μειώνει την πιθανότητα ανθρώπινου λάθους. Κάθε έκδοση ακολουθεί την ίδια τυποποιημένη διαδικασία, η οποία κάνει τα αποτελέσματα πιο προβλέψιμα.
