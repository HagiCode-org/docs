## Σχεδιασμός Εργαλείου ImgBin CLI: Προσέγγιση διαχείρισης περιουσιακών στοιχείων εικόνας της HagiCode

> Αυτό το άρθρο εξηγεί πώς να δημιουργήσετε μια αυτοματοποιημένη διοχέτευση στοιχείων εικόνας από την αρχή, καλύπτοντας τη σχεδίαση εργαλείων CLI, μια αρχιτεκτονική προσαρμογέα παρόχου και στρατηγικές διαχείρισης μεταδεδομένων.

## φόντο

Ειλικρινά, δεν περίμενα ότι η διαχείριση περιουσιακών στοιχείων εικόνας θα μας κρατούσε μπερδεμένους για τόσο πολύ.

Κατά την ανάπτυξη του HagiCode, αντιμετωπίσαμε ένα πρόβλημα που φαινόταν απλό στην επιφάνεια, αλλά ήταν εκπληκτικά ακανθώδες στην πράξη: δημιουργία και διαχείριση στοιχείων εικόνας. Κατά κάποιο τρόπο, έμοιαζε με τα δράματα της εφηβείας - ήρεμη απ' έξω, ταραχώδης από κάτω.

Καθώς το έργο συγκέντρωνε περισσότερη τεκμηρίωση και υλικό μάρκετινγκ, χρειαζόμασταν μεγάλο αριθμό υποστηρικτικών εικόνων. Ορισμένα έπρεπε να δημιουργηθούν από AI, μερικά έπρεπε να επιλεγούν από μια υπάρχουσα βιβλιοθήκη στοιχείων και άλλα χρειάζονταν αναγνώριση τεχνητής νοημοσύνης και αυτόματη επισήμανση. Το πρόβλημα ήταν ότι όλα αυτά είχαν αντιμετωπιστεί εδώ και καιρό μέσω διάσπαρτων σεναρίων και χειροκίνητων βημάτων. Κάθε φορά που δημιουργούσαμε μια εικόνα, έπρεπε να εκτελέσουμε ένα σενάριο με το χέρι, να οργανώσουμε τα μεταδεδομένα με το χέρι και να δημιουργήσουμε μικρογραφίες με το χέρι. Αυτό από μόνο του ήταν αρκετά ενοχλητικό, αλλά το μεγαλύτερο θέμα ήταν ότι όλα ήταν διάσπαρτα παντού. Όταν θέλαμε να βρούμε κάτι, δεν μπορούσαμε. Όταν χρειαζόμασταν να επαναχρησιμοποιήσουμε κάτι, δεν μπορούσαμε.

Τα σημεία πόνου ήταν συγκεκριμένα:

1. **Δεν υπάρχει ενοποιημένο σημείο εισόδου**: η λογική για τη δημιουργία εικόνων διαδόθηκε σε διαφορετικά σενάρια, επομένως η εκτέλεση κατά παρτίδες ήταν βασικά αδύνατη.
2. **Λείπουν μεταδεδομένα**: οι εικόνες που δημιουργήθηκαν δεν είχαν ενοποιηθεί `metadata.json`, που σήμαινε ότι δεν υπήρχε αξιόπιστη δυνατότητα αναζήτησης ή ιχνηλασιμότητας.
3. **Υψηλό κόστος μη αυτόματης οργάνωσης**: οι τίτλοι και οι ετικέτες έπρεπε να ταξινομηθούν ένας προς έναν με το χέρι, κάτι που ήταν αναποτελεσματικό.
4. **Χωρίς αυτοματισμό**: αυτόματη δημιουργία οπτικών στοιχείων σε αγωγό CI/CD; Δεν υπάρχει ευκαιρία.

Σκεφτήκαμε να το αφήσουμε μόνο του. Όμως τα έργα πρέπει ακόμα να προχωρήσουν. Εφόσον δεν μπορούσαμε να αποφύγουμε το πρόβλημα, σκεφτήκαμε ότι θα μπορούσαμε επίσης να το λύσουμε. Έτσι αποφασίσαμε να αναβαθμίσουμε το ImgBin από ένα σύνολο διάσπαρτων σεναρίων σε μια διοχέτευση στοιχείων εικόνας που μπορεί να εκτελεστεί αυτόματα. Κάποια προβλήματα, άλλωστε, δεν εξαφανίζονται μόνο και μόνο επειδή κοιτάς αλλού.

## Σχετικά με το HagiCode

Η προσέγγιση που κοινοποιείται σε αυτό το άρθρο προέρχεται από την πρακτική εμπειρία μας στο [HagiCode](https://hagicode.com) έργο. Το HagiCode είναι ένα έργο βοηθού κωδικοποίησης AI που διατηρεί ταυτόχρονα πολλαπλά στοιχεία, συμπεριλαμβανομένης μιας επέκτασης VSCode, υπηρεσιών τεχνητής νοημοσύνης υποστήριξης και ενός προγράμματος-πελάτη επιτραπέζιου υπολογιστή μεταξύ πλατφορμών. ## Σχεδιασμός πυρήνα

### Συνολική Αρχιτεκτονική

Το ImgBin χρησιμοποιεί μια πολυεπίπεδη αρχιτεκτονική που διαχωρίζει καθαρά τις εντολές CLI, τις υπηρεσίες εφαρμογών, τους προσαρμογείς API τρίτων κατασκευαστών και το επίπεδο υποδομής:

```
Component hierarchy
├── CLI Entry (cli.ts)              Global argument parsing, command routing
├── Commands (commands/*)           generate | batch | annotate | thumbnail
├── Application Services            job-runner | metadata | thumbnail | asset-writer
├── Provider Adapters               image-api-provider | vision-api-provider
└── Infrastructure Layer            config | logger | paths | schema
```

Το πλεονέκτημα αυτού του πολυεπίπεδου σχεδιασμού είναι τα σαφή όρια ευθύνης. Κάνει επίσης τη δοκιμή ευκολότερη, επειδή οι εξωτερικές εξαρτήσεις μπορούν να κοροϊδευτούν καθαρά. Στην πράξη, σημαίνει απλώς ότι κάθε στρώμα κάνει τη δική του δουλειά χωρίς να εμποδίζει τα άλλα, οπότε όταν κάτι σπάει, είναι ευκολότερο να καταλάβουμε γιατί.

### Μοντέλο καταλόγου μεμονωμένων στοιχείων

Το ImgBin χρησιμοποιεί ένα μοντέλο "ένα στοιχείο, ένας κατάλογος". Κάθε φορά που δημιουργείται μια εικόνα, δημιουργεί μια δομή όπως αυτή:

```
library/
└── 2026-03/
    └── orange-dashboard/
        ├── original.png      # Original image
        ├── thumbnail.webp    # 512x512 thumbnail
        └── metadata.json     # Structured metadata
```

Τα πλεονεκτήματα αυτού του μοντέλου είναι:

1. **Αυτοτελές**: όλα τα αρχεία για ένα μόνο στοιχείο βρίσκονται στον ίδιο κατάλογο, καθιστώντας εύκολη τη μετεγκατάσταση και τη δημιουργία αντιγράφων ασφαλείας.
2. **Ιχνηλάσιμο**: `metadata.json` καθιστά δυνατή την ανίχνευση χρόνου παραγωγής, προτροπής, μοντέλου και άλλων λεπτομερειών.
3. **Επεκτάσιμο**: εάν χρειαστούν περισσότερες παραλλαγές αργότερα, όπως μικρογραφίες σε πολλά μεγέθη, μπορούμε απλώς να προσθέσουμε νέα αρχεία στον ίδιο κατάλογο.

Τα όμορφα πράγματα δεν χρειάζεται πάντα να τα κατέχεις. Μερικές φορές αρκεί να παραμένουν όμορφα και να μπορείς να τα εκτιμάς ήσυχα. Αυτό μπορεί να ακούγεται λίγο μακριά, αλλά η λογική εξακολουθεί να ισχύει εδώ: από τη στιγμή που οι εικόνες διατηρούνται μαζί, είναι πιο ευχάριστες να τις δούμε και πολύ πιο εύκολο να τις βρείτε.

### Πολυεπίπεδη αποθήκευση μεταδεδομένων

`metadata.json` είναι ο πυρήνας ολόκληρου του συστήματος. Χρησιμοποιεί μια στρατηγική αποθήκευσης σε επίπεδα που χωρίζει τα πεδία σε τρεις κατηγορίες:

```json
{
  "schemaVersion": 2,
  "assetId": "orange-dashboard",
  "slug": "orange-dashboard",
  "title": "Orange Dashboard",
  "tags": ["dashboard", "hero", "orange"],

  "source": { "type": "generated" },

  "paths": {
    "assetDir": "library/2026-03/orange-dashboard",
    "original": "original.png",
    "thumbnail": "thumbnail.webp"
  },

  "generated": {
    "prompt": "orange dashboard for docs hero",
    "provider": "azure-openai-image-api",
    "model": "gpt-image-1.5"
  },

  "recognized": {
    "title": "Orange Dashboard",
    "tags": ["dashboard", "ui", "orange"],
    "description": "A modern orange dashboard with charts and metrics"
  },

  "status": {
    "generation": "succeeded",
    "recognition": "succeeded",
    "thumbnail": "succeeded"
  },

  "timestamps": {
    "createdAt": "2026-03-11T04:01:19.570Z",
    "updatedAt": "2026-03-11T04:02:09.132Z"
  }
}
```

- **δημιουργήθηκε**: καταγράφει τις αρχικές πληροφορίες από τη δημιουργία εικόνων, όπως η προτροπή, ο πάροχος και το μοντέλο.
- **αναγνωρισμένο**: αποθηκεύει αποτελέσματα αναγνώρισης τεχνητής νοημοσύνης, όπως τίτλους, ετικέτες και περιγραφές που δημιουργούνται αυτόματα.
- **εγχειρίδιο**: αποθηκεύει μη αυτόματα επιλεγμένα αποτελέσματα. Τα δεδομένα σε αυτόν τον τομέα έχουν την υψηλότερη προτεραιότητα και δεν θα αντικατασταθούν από την αναγνώριση AI.

Αυτή η πολυεπίπεδη στρατηγική επιλύει μια από τις προηγούμενες βασικές διενέξεις μας: όταν η αναγνώριση τεχνητής νοημοσύνης και η μη αυτόματη επιμέλεια διαφωνούν, ποια πρέπει να κερδίσει; Η απάντηση είναι η χειροκίνητη εισαγωγή. Η αναγνώριση AI είναι εκεί για να βοηθήσει, όχι για να αποφασίσει. Αυτή η ερώτηση έγινε επίσης πιο ξεκάθαρη με την πάροδο του χρόνου - οι μηχανές εξακολουθούν να είναι μηχανές, και στο τέλος, οι άνθρωποι πρέπει ακόμα να κάνουν την κλήση.

## Μοτίβο προσαρμογέα παρόχου

Ένα άλλο βασικό μέρος του ImgBin είναι το μοτίβο του προσαρμογέα παρόχου. Αφαιρούμε εξωτερικά API πίσω από μια ενοποιημένη διεπαφή, έτσι ώστε ακόμα κι αν αλλάξουμε παρόχους υπηρεσιών τεχνητής νοημοσύνης, να μην χρειάζεται να αλλάξουμε την επιχειρηματική λογική.

Κατά κάποιο τρόπο, μοιάζει λίγο με τις σχέσεις - η εξωτερική εμφάνιση μπορεί να αλλάξει, αλλά αυτό που έχει σημασία είναι ότι η εσωτερική δομή παραμένει ίδια. Μόλις επιδιορθωθεί η διεπαφή, η εσωτερική υλοποίηση μπορεί να ποικίλλει ελεύθερα.

### Πάροχος δημιουργίας εικόνων

```typescript
interface ImageGenerationProvider {
  // Generate an image and return its Buffer
  generate(options: GenerateOptions): Promise<Buffer>;

  // Get the list of supported models
  getSupportedModels(): Promise<string[]>;
}

interface GenerateOptions {
  prompt: string;
  model?: string;
  size?: '1024x1024' | '1792x1024' | '1024x1792';
  quality?: 'standard' | 'hd';
  format?: 'png' | 'webp' | 'jpeg';
}
```

### Παροχέας Αναγνώρισης Οράματος

```typescript
interface VisionRecognitionProvider {
  // Recognize image content and return structured metadata
  recognize(imageBuffer: Buffer): Promise<RecognitionResult>;

  // Get the list of supported models
  getSupportedModels(): Promise<string[]>;
}

interface RecognitionResult {
  title?: string;
  tags: string[];
  description?: string;
  confidence: number;
}
```

Τα πλεονεκτήματα αυτού του σχεδιασμού διεπαφής είναι:

1. **Δοκιμή**: στις δοκιμές μονάδας, μπορούμε να περάσουμε σε εικονικούς παρόχους αντί να κάνουμε πραγματικές εξωτερικές κλήσεις API.
2. **Επεκτάσιμο**: η προσθήκη νέου παρόχου απαιτεί μόνο την υλοποίηση της διεπαφής. Ο κωδικός καλούντος δεν χρειάζεται να αλλάξει.
3. **Αντικαταστάσιμο**: η παραγωγή μπορεί να χρησιμοποιήσει το Azure OpenAI ενώ η δοκιμή μπορεί να χρησιμοποιήσει ένα τοπικό μοντέλο, με τη διαμόρφωση να είναι το μόνο πράγμα που αλλάζει.

Μερικές φορές η δουλειά του έργου μοιάζει κι αυτή. Επιφανειακά φαίνεται ότι μόλις ανταλλάξαμε ένα API, αλλά η εσωτερική λογική παραμένει ακριβώς η ίδια, και αυτό κάνει το όλο θέμα πολύ λιγότερο τρομακτικό.

## Σχεδίαση εντολών CLI

Το ImgBin παρέχει τέσσερις βασικές εντολές για την κάλυψη διαφορετικών σεναρίων χρήσης:

### δημιουργία: δημιουργία μιας εικόνας

```bash
# Simplest usage
imgbin generate --prompt "orange dashboard for docs hero"

# Generate a thumbnail and AI annotations at the same time
imgbin generate --prompt "orange dashboard" --annotate --thumbnail

# Specify an output directory
imgbin generate --prompt "orange dashboard" --output ./library
```

### παρτίδα: εργασίες παρτίδας

Οι ομαδικές εργασίες ορίζονται μέσω των αρχείων δήλωσης YAML ή JSON, γεγονός που τις καθιστά κατάλληλες για ροές εργασίας CI/CD:

```yaml
# assets/jobs/launch.yaml
defaults:
  annotate: true
  thumbnail: true
  libraryRoot: ./library

jobs:
  - prompt: "orange dashboard hero"
    slug: orange-dashboard
    tags: [dashboard, hero, orange]

  - prompt: "pricing grid for docs"
    slug: pricing-grid
    tags: [pricing, grid, docs]
```

Εκτελέστε την εντολή:

```bash
imgbin batch assets/jobs/launch.yaml
```

Ο σχεδιασμός ομαδικής εργασίας υποστηρίζει την απομόνωση αποτυχίας: τα στοιχεία στη δήλωση υποβάλλονται σε επεξεργασία ένα προς ένα και μια αποτυχία σε ένα στοιχείο δεν επηρεάζει τα άλλα. Μπορείτε επίσης να κάνετε προεπισκόπηση της εργασίας με `--dry-run` χωρίς ουσιαστικά να το εκτελέσει.

Και το καλύτερο είναι ότι σας λέει ακριβώς τι πέτυχε και τι απέτυχε. Σε αντίθεση με κάποια πράγματα στη ζωή, όπου η αποτυχία συμβαίνει και μένεις να μην ξέρεις καν πώς συνέβη.

### annotate: σχολιασμός AI

Εκτελέστε την αναγνώριση AI σε υπάρχουσες εικόνες για να δημιουργήσετε αυτόματα τίτλους, ετικέτες και περιγραφές:

```bash
# Annotate a single image
imgbin annotate ./library/2026-03/orange-dashboard

# Annotate an entire directory in batch
imgbin annotate ./library/2026-03/
```

### thumbnail: δημιουργία μικρογραφιών

Δημιουργήστε μικρογραφίες για υπάρχουσες εικόνες:

```bash
# Generate a thumbnail
imgbin thumbnail ./library/2026-03/orange-dashboard
```

## Σχεδιασμός Μανιφέστου Παρτίδας Εργασίας

Η μορφή δήλωσης για ομαδικές εργασίες υποστηρίζει ευέλικτη διαμόρφωση. Οι προεπιλογές μπορούν να οριστούν παγκοσμίως και μεμονωμένες εργασίες μπορούν να τις παρακάμψουν:

```yaml
# Global defaults
defaults:
  annotate: true        # Enable AI annotation by default
  thumbnail: true       # Generate thumbnails by default
  libraryRoot: ./library
  model: gpt-image-1.5

jobs:
  # Minimal configuration: only provide a prompt
  - prompt: "first image"

  # Full configuration
  - prompt: "second image"
    slug: custom-slug
    tags: [tag1, tag2]
    annotate: false     # Do not run AI annotation for this job
    model: dall-e-3    # Use a different model for this job
```

Όταν εκτελείται, το ImgBin επεξεργάζεται εργασίες μία προς μία. Το αποτέλεσμα κάθε εργασίας γράφεται στο αντίστοιχο `metadata.json`. Ακόμα κι αν μια δουλειά αποτύχει, οι άλλες δεν επηρεάζονται. Αφού ολοκληρωθούν όλες οι εργασίες, το CLI βγάζει μια συνοπτική αναφορά:

```
✓ orange-dashboard (succeeded)
✓ pricing-grid (succeeded)
✗ hero-banner (failed: API rate limit exceeded)

2/3 succeeded, 1 failed
```

Κάποια πράγματα δεν μπορούν να βιαστούν. Το να τα παίρνετε ένα-ένα είναι συχνά ο πιο σταθερός δρόμος. Ίσως αυτή είναι η φιλοσοφία πίσω από τις δουλειές παρτίδας.

## Διαμόρφωση μεταβλητής περιβάλλοντος

Το ImgBin υποστηρίζει ευέλικτη διαμόρφωση μέσω μεταβλητών περιβάλλοντος:

```bash
# ImgBin working directory
IMGBIN_WORKDIR=/path/to/imgbin

# Executable path (for invocation inside scripts)
IMGBIN_EXECUTABLE=/path/to/imgbin/dist/cli.js

# Asset library root
IMGBIN_LIBRARY_ROOT=./.imgbin-library

# Azure OpenAI configuration (if using the Azure provider)
AZURE_OPENAI_ENDPOINT=https://your-resource.openai.azure.com/
AZURE_OPENAI_API_KEY=***
AZURE_OPENAI_IMAGE_DEPLOYMENT=gpt-image-1
```

Η διαμόρφωση είναι ένα από εκείνα τα πράγματα που μπορεί να αισθάνονται τόσο σημαντικά όσο και όχι τόσο σημαντικά ταυτόχρονα. Στο τέλος, ό,τι αισθάνεστε άνετα και ταιριάζει καλύτερα στη ροή εργασίας σας είναι συνήθως η σωστή επιλογή.

## Σημειώσεις Εφαρμογής

Κατά την υλοποίηση, συνοψίσαμε μερικά βασικά σημεία:

### Σχεδιασμός διεπαφής παρόχου

Οι ορισμοί της διεπαφής πρέπει να είναι σαφείς και πλήρεις, συμπεριλαμβανομένων των παραμέτρων εισόδου, των τιμών επιστροφής και του χειρισμού σφαλμάτων. Είναι επίσης καλή ιδέα να παρέχετε τόσο σύγχρονο όσο και ασύγχρονο στυλ επίκλησης για διαφορετικά σενάρια.

Αυτό είναι ένα μικρό κομμάτι εμπειρίας που αποκτήθηκε με κόπο. Μόλις οριστεί μια διεπαφή, κανείς δεν θέλει να συνεχίσει να την αλλάζει αργότερα.

### Στρατηγική Αντιμετώπισης Αποτυχιών

Όταν ένα στοιχείο αποτυγχάνει σε μια εργασία δέσμης, το CLI θα πρέπει:

1. Γράψτε λεπτομερείς πληροφορίες σφάλματος σε ξεχωριστό αρχείο καταγραφής.
2. Συνεχίστε να εκτελείτε άλλες εργασίες αντί να διακόπτετε την όλη διαδικασία.
3. Επιστρέψτε έναν μη μηδενικό κωδικό εξόδου στο τέλος για να υποδείξετε ότι ορισμένες εργασίες απέτυχαν.
4. Εμφανίστε καθαρά το αποτέλεσμα εκτέλεσης κάθε εργασίας στη συνοπτική αναφορά.

Μερικές αποτυχίες είναι απλώς αποτυχίες. Δεν έχει νόημα να προσποιούμαστε το αντίθετο. Είναι καλύτερα να τα αναγνωρίσετε ανοιχτά και μετά να βρείτε πώς να τα λύσετε. Η ίδια λογική ισχύει και για τα έργα και για τη ζωή.

### Στρατηγική συγχώνευσης μεταδεδομένων

Τα αποτελέσματα της αναγνώρισης εγγράφονται στο `recognized` ενότητα από προεπιλογή, ενώ τα πεδία που έχουν υποστεί μη αυτόματη επεξεργασία επισημαίνονται `manual`. Οι ενημερώσεις μεταδεδομένων ακολουθούν μια στρατηγική μόνο για προσάρτημα: εκτός εάν `--force` έχει περάσει ρητά, τα υπάρχοντα μη αυτόματα επιμελημένα αποτελέσματα δεν αντικαθίστανται.

Αυτό το σημείο έγινε επίσης ξεκάθαρο - ορισμένα πράγματα, μόλις αντικατασταθούν, μόλις έχουν φύγει. Συχνά είναι καλύτερο να τα διατηρήσουμε, γιατί ο ίδιος ο δίσκος έχει αξία.

### Ατομικότητα δημιουργίας καταλόγου

Χρήση `fs.mkdir({ recursive: true })` για να διασφαλιστεί ότι η δημιουργία καταλόγου παραμένει ατομική και για να αποφευχθούν οι συνθήκες αγώνων σε ταυτόχρονα σενάρια.

Ίσως αυτό είναι το νόημα της ασφάλειας - να είσαι σταθερός όταν έχει σημασία η σταθερότητα, να κινείσαι γρήγορα όταν έχει σημασία η ταχύτητα και να μην κολλάς σε δεύτερη μοίρα.

## Συμπέρασμα

Ως το βασικό εργαλείο για τη διαχείριση περιουσιακών στοιχείων εικόνας στο έργο HagiCode, το ImgBin επιλύει τα προβλήματά μας μέσω των παρακάτω επιλογών σχεδίασης:

1. **Ενοποιημένο σημείο εισόδου**: το CLI καλύπτει τη δημιουργία, τον σχολιασμό, τις μικρογραφίες και όλες τις άλλες βασικές λειτουργίες.
2. **Βάσει μεταδεδομένων**: κάθε στοιχείο έχει ένα πλήρες `metadata.json`, επιτρέποντας την αναζήτηση και την ιχνηλασιμότητα.
3. **Προσαρμογέας παρόχου**: ευέλικτη αφαίρεση για εξωτερικά API, διευκολύνοντας τη δοκιμή και την επέκταση.
4. **Υποστήριξη ομαδικών εργασιών**: η δημιουργία ομαδικής εικόνας μπορεί να αυτοματοποιηθεί εντός των ροών εργασίας CI/CD.

Όλα τα άλλα μπορεί να έχουν ξεθωριάσει, αλλά αυτή η προσέγγιση τελικά αποδείχθηκε χρήσιμη.

Αυτή η λύση όχι μόνο βελτιώνει την αποδοτικότητα ανάπτυξης της ίδιας της HagiCode, αλλά σχηματίζει επίσης ένα επαναχρησιμοποιήσιμο πλαίσιο για τη διαχείριση περιουσιακών στοιχείων εικόνας. Εάν κατασκευάζετε ένα παρόμοιο έργο πολλών συστατικών, πιστεύω ότι οι σχεδιαστικές ιδέες του ImgBin μπορεί να σας δώσουν κάποια έμπνευση.

Η νεολαία είναι να δοκιμάζεις πράγματα και να κάνεις λίγο χάος. Αν δεν βάλετε ποτέ τον εαυτό σας σε αυτό, πώς θα ξέρετε τι είστε πραγματικά ικανοί;
