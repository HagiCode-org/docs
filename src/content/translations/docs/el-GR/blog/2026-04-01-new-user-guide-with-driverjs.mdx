## Δημιουργία κομψής ενσωμάτωσης νέου χρήστη σε React Projects: HagiCode's driver.js Practice

> Όταν οι χρήστες ανοίγουν το προϊόν σας για πρώτη φορά, ξέρουν πραγματικά από πού να ξεκινήσουν; Σε αυτό το άρθρο, θέλω να μιλήσω λίγο για το πώς χρησιμοποιήσαμε το driver.js για την ενσωμάτωση νέου χρήστη στο έργο HagiCode. Σκεφτείτε το ένα μικρό πρακτικό παράδειγμα για να ξεκινήσετε τη συζήτηση.

## φόντο

Έχετε βρεθεί ποτέ σε αυτή την κατάσταση; Ένας νέος χρήστης εγγράφεται στο προϊόν σας, ανοίγει τη σελίδα και φαίνεται αμέσως χαμένος. Σαρώνουν τριγύρω, δεν είναι σίγουροι τι να κάνουν κλικ ή τι να κάνουν στη συνέχεια. Ως προγραμματιστές, συχνά υποθέτουμε ότι οι χρήστες απλώς «θα εξερευνήσουν μόνοι τους», επειδή, τελικά, η ανθρώπινη περιέργεια είναι απεριόριστη. Αλλά η πραγματικότητα είναι διαφορετική: οι περισσότεροι χρήστες φεύγουν αθόρυβα μέσα σε λίγα λεπτά αν δεν μπορούν να βρουν το σωστό σημείο εισόδου, σαν η ιστορία να ξεκινά ξαφνικά και να τελειώνει το ίδιο φυσικά.

Η ενσωμάτωση νέου χρήστη είναι ένας σημαντικός τρόπος επίλυσης αυτού του προβλήματος, αλλά η σωστή δημιουργία του δεν είναι τόσο απλή. Ένα καλό σύστημα ενσωμάτωσης χρειάζεται:

- Εντοπίστε με ακρίβεια τα στοιχεία της σελίδας και επισημάνετε τα
- Υποστήριξη ροών ενσωμάτωσης πολλαπλών βημάτων
- Θυμηθείτε την επιλογή του χρήστη (ολοκλήρωση ή παράλειψη)
- Αποφύγετε να επηρεάσετε την απόδοση της σελίδας και την κανονική αλληλεπίδραση
- Διατηρήστε τη δομή του κώδικα σαφή και εύκολη στη συντήρηση

Κατά τη δημιουργία του HagiCode, αντιμετωπίσαμε την ίδια πρόκληση. Το HagiCode είναι ένας βοηθός κωδικοποίησης AI και η βασική ροή εργασίας του είναι μια ροή εργασίας OpenSpec που μοιάζει με αυτό: "ο χρήστης δημιουργεί μια πρόταση -> το AI δημιουργεί ένα σχέδιο -> ο χρήστης το εξετάζει -> το AI το εκτελεί." Για τους χρήστες που αντιμετωπίζουν αυτήν την ιδέα για πρώτη φορά, η ροή εργασίας είναι εντελώς νέα, επομένως χρειάζονται σταθερή ενσωμάτωση για να ξεκινήσουν γρήγορα. Τα νέα πράγματα χρειάζονται πάντα λίγο χρόνο για να τα συνηθίσεις.

## Γιατί επιλέξαμε το driver.js

Κατά τη φάση της τεχνικής αξιολόγησης, εξετάσαμε αρκετές βασικές ενσωματωμένες βιβλιοθήκες. Το καθένα είχε τα δικά του δυνατά σημεία:

- **Intro.js**: Ισχυρή, αλλά σχετικά μεγάλη και η προσαρμογή στυλ είναι κάπως περίπλοκη
- **Shepherd.js**: Καλά σχεδιασμένο API, αλλά λίγο πολύ "βαρύ" για την περίπτωση χρήσης μας
- **driver.js**: Ελαφρύ, συνοπτικό, διαισθητικό API και λειτουργεί καλά στο οικοσύστημα React

Στο τέλος, επιλέξαμε το driver.js. Δεν υπήρχε ιδιαίτερα δραματικός λόγος. Η επιλογή οφείλεται κυρίως σε αυτούς τους λόγους:

1. **Ελαφρύ**: Η βασική βιβλιοθήκη είναι μικρή και δεν αυξάνει σημαντικά το μέγεθος του πακέτου
2. **Απλό API**: Η διαμόρφωση είναι σαφής και διαισθητική, επομένως είναι εύκολο να το παραλάβετε
3. **Εύκαμπτο**: Υποστηρίζει προσαρμοσμένη τοποθέτηση, στυλ και συμπεριφορά αλληλεπίδρασης
4. **Δυναμική εισαγωγή**: Μπορεί να φορτωθεί κατά παραγγελία χωρίς να επηρεαστεί η απόδοση της πρώτης οθόνης

Με την επιλογή τεχνολογίας, σπάνια υπάρχει μια παγκοσμίως καλύτερη απάντηση. Συνήθως, υπάρχει μόνο η επιλογή που ταιριάζει καλύτερα.

## Τεχνική Υλοποίηση

### Διαμόρφωση πυρήνα

Το driver.js έχει ένα πολύ διαισθητικό μοντέλο διαμόρφωσης. Εδώ είναι η βασική διαμόρφωση που χρησιμοποιούμε στο έργο HagiCode:

```typescript
import { driver } from 'driver.js';
import 'driver.js/dist/driver.css';

const newConversationDriver = driver({
  allowClose: true,           // Allow users to close the guide
  animate: true,              // Enable animations
  overlayClickBehavior: 'close', // Close the guide when the overlay is clicked
  disableActiveInteraction: false, // Keep elements interactive
  showProgress: false,        // Do not show the progress bar (we manage progress ourselves)
  steps: guideSteps           // Array of guide steps
});
```

Το σκεπτικό πίσω από αυτές τις ρυθμίσεις είναι:

- `allowClose: true` - Σεβαστείτε την επιλογή του χρήστη και μην τον αναγκάσετε να ολοκληρώσει τον οδηγό
- `disableActiveInteraction: false` - Ορισμένα βήματα απαιτούν πραγματικές ενέργειες χρήστη, όπως η πληκτρολόγηση εισαγωγής, επομένως η αλληλεπίδραση δεν μπορεί να απενεργοποιηθεί
- `overlayClickBehavior: 'close'` - Δώστε στους χρήστες έναν γρήγορο τρόπο εξόδου

### Κρατική Διοίκηση

Η διατήρηση της κατάστασης ενσωμάτωσης είναι κρίσιμη. Δεν θέλουμε να κάνουμε επανεκκίνηση του οδηγού κάθε φορά που ανανεώνεται η σελίδα, γιατί αυτό γίνεται γρήγορα ενοχλητικό. Χρησιμοποιεί το HagiCode `localStorage` για τη διαχείριση της κατάστασης οδηγού:

```typescript
export type GuideState = 'pending' | 'dismissed' | 'completed';

export interface UserGuideState {
  session: GuideState;
  detailGuides: Record<string, GuideState>;
}

// Read state
export const getUserGuideState = (): UserGuideState => {
  const state = localStorage.getItem('userGuideState');
  return state ? JSON.parse(state) : { session: 'pending', detailGuides: {} };
};

// Update state
export const setUserGuideState = (state: UserGuideState) => {
  localStorage.setItem('userGuideState', JSON.stringify(state));
};
```

Ορίζουμε τρεις καταστάσεις:

- `pending`: Ο οδηγός είναι ακόμα σε εξέλιξη και ο χρήστης δεν τον έχει ολοκληρώσει ή δεν τον έχει παραλείψει
- `dismissed`: Ο χρήστης έκλεισε τον οδηγό προληπτικά
- `completed`: Ο χρήστης ολοκλήρωσε όλα τα βήματα

Για την ενσωμάτωση της σελίδας λεπτομερειών πρότασης, υποστηρίζουμε επίσης πιο λεπτομερή παρακολούθηση κατάστασης μέσω του `detailGuides` χάρτη, επειδή μια πρόταση μπορεί να περάσει από πολλά στάδια - το προσχέδιο, η αναθεώρηση και η εκτέλεση ολοκληρώθηκε - και κάθε στάδιο χρειάζεται διαφορετική καθοδήγηση. Η κατάσταση των πραγμάτων αλλάζει πάντα και η ενσωμάτωση θα πρέπει να αντικατοπτρίζει αυτό.

### Επιλογή στοιχείου στόχου

Το driver.js χρησιμοποιεί επιλογείς CSS για να εντοπίσει στοιχεία προορισμού. Το HagiCode ακολουθεί μια απλή σύμβαση: χρησιμοποιήστε μια προσαρμογή `data-guide` χαρακτηριστικό για την επισήμανση στόχων ενσωμάτωσης:

```typescript
const steps = [
  {
    element: '[data-guide="launch"]',
    popover: {
      title: 'Start a New Conversation',
      description: 'Click here to create a new conversation session...'
    }
  }
];
```

Στα εξαρτήματα, μοιάζει με αυτό:

```typescript
<button data-guide="launch" onClick={handleLaunch}>
  New Conversation
</button>
```

Τα οφέλη αυτής της προσέγγισης είναι:

- Αποφύγετε τις συγκρούσεις με τα ονόματα των κλάσεων επιχειρηματικού στυλ
- Ξεκαθαρίστε τη σημασιολογία, ώστε να μπορείτε να πείτε αμέσως ότι το στοιχείο σχετίζεται με την ενσωμάτωση
- Πιο εύκολο στη διαχείριση και διατήρηση με συνέπεια

### Δυναμική βελτιστοποίηση εισαγωγής

Επειδή η δυνατότητα ενσωμάτωσης απαιτείται μόνο σε συγκεκριμένα σενάρια, όπως η πρώτη επίσκεψη ενός χρήστη, χρησιμοποιούμε δυναμικές εισαγωγές για τη βελτιστοποίηση της αρχικής απόδοσης φόρτωσης:

```typescript
const initNewUserGuide = async () => {
  // Dynamically import driver.js
  const { driver } = await import('driver.js');
  await import('driver.js/dist/driver.css');

  // Initialize the guide
  const newConversationDriver = driver({
    // ...configuration
  });

  newConversationDriver.drive();
};
```

Με αυτόν τον τρόπο, το driver.js και το φύλλο στυλ του φορτώνονται μόνο όταν χρειάζεται και δεν επηρεάζουν την απόδοση της πρώτης οθόνης. Δεν είναι πολλοί οι άνθρωποι που απολαμβάνουν να περιμένουν κάτι που δεν χρειάζονται ακόμη.

## Σχεδιασμός ροής ενσωμάτωσης

Το HagiCode υλοποιεί δύο διαδρομές ενσωμάτωσης που καλύπτουν τα βασικά σενάρια του χρήστη.

### Ενσωμάτωση συνεδρίας (10 βήματα)

Αυτή η διαδρομή ενσωμάτωσης βοηθά τους χρήστες να ολοκληρώσουν ολόκληρη τη ροή από τη δημιουργία μιας συνομιλίας έως την υποβολή της πρώτης ολοκληρωμένης Πρότασής τους:

1. **εκκίνηση** - Ξεκινήστε τον οδηγό και εισάγετε το κουμπί "Νέα συνομιλία".
2. **σύνθεση** - Καθοδηγήστε τον χρήστη να πληκτρολογήσει ένα αίτημα στο πλαίσιο εισαγωγής
3. **αποστολή** - Καθοδηγήστε τον χρήστη να κάνει κλικ στο κουμπί αποστολής
4. **proposal-launch-readme** - Καθοδηγήστε τον χρήστη να δημιουργήσει μια πρόταση README
5. **proposal-compose-readme** - Καθοδηγήστε τον χρήστη να επεξεργαστεί το περιεχόμενο του αιτήματος README
6. **proposal-submit-readme** - Καθοδήγηση του χρήστη να υποβάλει την πρόταση README
7. **proposal-launch-agents** - Καθοδηγήστε τον χρήστη να δημιουργήσει ένα `AGENTS.md` Πρόταση
8. **proposal-compose-agents** - Καθοδηγήστε τον χρήστη να επεξεργαστεί το `AGENTS.md` αίτημα
9. **proposal-submit-agents** - Καθοδηγήστε τον χρήστη να υποβάλει το `AGENTS.md` Πρόταση
10. **proposal-wait** - Εξηγήστε ότι η τεχνητή νοημοσύνη βρίσκεται σε επεξεργασία και ο χρήστης πρέπει να περιμένει λίγο

Η ιδέα πίσω από αυτό το μονοπάτι είναι να επιτραπεί στους χρήστες να βιώσουν από πρώτο χέρι τη βασική ροή εργασίας του HagiCode μέσω δύο πραγματικών εργασιών δημιουργίας πρότασης, μία για `README` και ένα για `AGENTS.md`. Υπάρχει μεγάλη διαφορά ανάμεσα στο να ακούς για μια ροή εργασίας και να την κάνεις μόνος σου.

### Ενσωμάτωση λεπτομερειών πρότασης (3 βήματα)

Όταν οι χρήστες εισέρχονται στη σελίδα λεπτομερειών πρότασης, το HagiCode ενεργοποιεί τον αντίστοιχο οδηγό με βάση την τρέχουσα κατάσταση της Πρότασης:

1. **σύνταξη** (πρόχειρο στάδιο) - Καθοδηγήστε τον χρήστη να επανεξετάσει το σχέδιο που δημιουργήθηκε από την τεχνητή νοημοσύνη
2. **ανασκόπηση** (στάδιο αναθεώρησης) - Καθοδηγήστε τον χρήστη να εκτελέσει το σχέδιο
3. **executionCompleted** (ολοκληρώθηκε το στάδιο) - Καθοδηγήστε τον χρήστη να αρχειοθετήσει το σχέδιο

Το καθοριστικό χαρακτηριστικό αυτού του οδηγού είναι ότι βασίζεται στην κατάσταση: αποφασίζει δυναμικά ποιο βήμα επιβίβασης θα εμφανίσει με βάση την πραγματική κατάσταση της Πρότασης. Τα πράγματα αλλάζουν και η ενσωμάτωση θα πρέπει να αλλάξει μαζί τους.

Το παρακάτω στιγμιότυπο οθόνης δείχνει τη σελίδα λεπτομερειών της πρότασης σε κατάσταση ενσωμάτωσης κατά τη φάση σύνταξης:

Σε αυτό το στάδιο, ο οδηγός εστιάζει την προσοχή του χρήστη στη βασική ενέργεια της δημιουργίας ενός σχεδίου, ώστε να μην αναρωτιέται τι να κάνει πρώτα όταν εισέρχεται για πρώτη φορά στη σελίδα λεπτομερειών.

### Μηχανισμός επανάληψης απόδοσης στοιχείων

Στις εφαρμογές React, το στοιχείο ενσωμάτωσης προορισμού ενδέχεται να μην έχει ολοκληρώσει ακόμη την απόδοση, για παράδειγμα επειδή φορτώνονται ακόμη ασύγχρονα δεδομένα. Για να το χειριστεί αυτό, το HagiCode εφαρμόζει έναν μηχανισμό επανάληψης δοκιμής:

```typescript
const waitForElement = (selector: string, maxRetries = 10, interval = 100) => {
  let retries = 0;
  return new Promise<HTMLElement>((resolve, reject) => {
    const checkElement = () => {
      const element = document.querySelector(selector) as HTMLElement;
      if (element) {
        resolve(element);
      } else if (retries < maxRetries) {
        retries++;
        setTimeout(checkElement, interval);
      } else {
        reject(new Error(`Element not found: ${selector}`));
      }
    };
    checkElement();
  });
};
```

Καλέστε αυτήν τη συνάρτηση πριν αρχικοποιήσετε τον οδηγό για να βεβαιωθείτε ότι το στοιχείο προορισμού υπάρχει ήδη. Μερικές φορές το να περιμένεις λίγο περισσότερο αξίζει τον κόπο.

## Σύνοψη βέλτιστων πρακτικών

Με βάση την πρακτική εμπειρία της HagiCode, ακολουθούν μερικές βασικές βέλτιστες πρακτικές:

### 1. Η έξοδος από την ενσωμάτωση πρέπει να είναι εύκολη

Μην αναγκάζετε τους χρήστες να ολοκληρώσουν την ενσωμάτωση. Μερικοί χρήστες είναι εκ φύσεως εξερευνητές και προτιμούν να καταλαβαίνουν τα πράγματα μόνοι τους. Δώστε ένα καθαρό κουμπί "Παράλειψη", θυμηθείτε την επιλογή τους και μην τους διακόψετε ξανά την επόμενη φορά.

### 2. Διατηρήστε το περιεχόμενο ενσωμάτωσης σύντομο και ευκρινές

Κάθε βήμα ενσωμάτωσης θα πρέπει να επικεντρώνεται σε έναν μόνο στόχο:

- **Τίτλος**: Σύντομος και σαφής, ιδανικά όχι περισσότερες από λίγες λέξεις
- **Περιγραφή**: Πηγαίνετε κατευθείαν στο θέμα και πείτε στο χρήστη τι είναι αυτό και γιατί έχει σημασία

Αποφύγετε τις μακροσκελείς εξηγήσεις. Η προσοχή του χρήστη είναι περιορισμένη κατά την ενσωμάτωση και όσο περισσότερα λέτε, τόσο λιγότερες είναι οι πιθανότητες να το διαβάσουν.

### 3. Διατηρήστε σταθερούς τους επιλογείς

Χρησιμοποιήστε δείκτες σταθερών στοιχείων που δεν αλλάζουν συχνά. Το έθιμο `data-guide` το χαρακτηριστικό είναι μια καλή επιλογή. Αποφύγετε να εξαρτώνται από τα ονόματα κλάσεων ή τη δομή DOM, γιατί αυτά είναι εύκολο να αλλάξουν κατά τη διάρκεια των refactors. Ο κώδικας αλλάζει συνεχώς, αλλά ορισμένες αγκυρώσεις πρέπει να παραμένουν σταθερές όταν είναι δυνατόν.

### 4. Δοκιμάστε την Ενσωμάτωση

Το HagiCode περιλαμβάνει πλήρεις δοκιμές για τη λειτουργία ενσωμάτωσης:

```typescript
describe('NewUserConversationGuide', () => {
  it('should initialize guide state correctly', () => {
    const state = getUserGuideState();
    expect(state.session).toBe('pending');
  });

  it('should update guide state correctly', () => {
    setUserGuideState({ session: 'completed', detailGuides: {} });
    const state = getUserGuideState();
    expect(state.session).toBe('completed');
  });
});
```

Οι δοκιμές βοηθούν να διασφαλιστεί ότι η ανακατασκευή δεν θα παραβιάσει κατά λάθος τη συμπεριφορά ενσωμάτωσης. Κανείς δεν θέλει μια μικρή αλλαγή κώδικα για να βλάψει αθόρυβα τη λειτουργικότητα που λειτουργούσε προηγουμένως.

### 5. Βελτιστοποίηση απόδοσης

- Χρησιμοποιήστε δυναμικές εισαγωγές για να φορτώσετε αργά τη βιβλιοθήκη ενσωμάτωσης
- Αποφύγετε την προετοιμασία της λογικής ενσωμάτωσης αφού ο χρήστης έχει ήδη ολοκληρώσει τον οδηγό
- Λάβετε υπόψη τον αντίκτυπο στην απόδοση των κινούμενων εικόνων και απενεργοποιήστε τις σε συσκευές χαμηλότερης τεχνολογίας, εάν χρειάζεται

Η απόδοση, όπως και πολλά πράγματα στη ζωή, αξίζει λίγο προσεκτικό προϋπολογισμό.

## Συμπέρασμα

Η ενσωμάτωση νέου χρήστη αποτελεί σημαντικό μέρος της βελτίωσης της εμπειρίας χρήστη του προϊόντος. Στο έργο HagiCode, χρησιμοποιήσαμε το driver.js για να δημιουργήσουμε ένα πλήρες σύστημα ενσωμάτωσης που καλύπτει την πλήρη ροή εργασίας από τη δημιουργία συνεδρίας έως την εκτέλεση της πρότασης.

Τα βασικά σημεία που ελπίζουμε να μοιραστούμε μέσω αυτού του άρθρου είναι:

1. **Οι τεχνικές επιλογές πρέπει να ταιριάζουν με τις πραγματικές ανάγκες**: το driver.js δεν είναι η πιο ισχυρή επιλογή, αλλά είναι η καλύτερη επιλογή για εμάς
2. **Η κρατική διαχείριση είναι κρίσιμη**: Χρήση `localStorage` για να παραμείνει η κατάσταση ενσωμάτωσης και να αποφευχθεί η επανειλημμένη διακοπή των χρηστών
3. **Ο σχεδιασμός ενσωμάτωσης πρέπει να παραμένει εστιασμένος**: Κάθε βήμα θα πρέπει να λύνει ένα πρόβλημα και όχι περισσότερα
4. **Η δομή του κώδικα θα πρέπει να παραμένει σαφής**: Ξεχωριστή διαμόρφωση ενσωματωμένης λειτουργίας, διαχείριση κατάστασης και λογική διεπαφής χρήστη για να διευκολύνει τη συντήρηση

Εάν προσθέτετε την ενσωμάτωση νέου χρήστη στο δικό σας έργο, ελπίζω η πρακτική εμπειρία σε αυτό το άρθρο να σας βοηθήσει. Δεν υπάρχει τίποτα ιδιαίτερα μυστικιστικό σε αυτό το είδος τεχνολογίας. Συνεχίστε να προσπαθείτε, συνέχισε να συνοψίζεις όσα μαθαίνεις και σταδιακά τα πράγματα γίνονται πιο εύκολα.
