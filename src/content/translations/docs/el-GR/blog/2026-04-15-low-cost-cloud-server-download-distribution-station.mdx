## Πώς να δημιουργήσετε έναν σταθμό διανομής γρήγορης λήψης με διακομιστές Cloud χαμηλού κόστους

> Το εύρος ζώνης αποθήκευσης στο cloud είναι παράλογα ακριβό, η διασυνοριακή πρόσβαση είναι οδυνηρά αργή και η τιμολόγηση του CDN είναι αρκετή για να τρομάξει οποιονδήποτε... Εάν χειρίζεστε τη διανομή αρχείων, πιθανότατα γνωρίζετε καλά αυτά τα προβλήματα. Σε αυτήν την ανάρτηση, θέλω να μοιραστώ μια προσέγγιση χαμηλού κόστους που επεξεργαστήκαμε κατά την κατασκευή [HagiCode](https://hagicode.com): ένας διακομιστής cloud συν ένα επίπεδο προσωρινής αποθήκευσης Nginx. Το κόστος μειώθηκε περίπου στο μισό, ενώ η ταχύτητα λήψης βελτιώθηκε αρκετά, κάτι που ήταν τουλάχιστον λίγο ανακουφιστικό.

## φόντο

Όταν πρόκειται για το Διαδίκτυο, η ταχύτητα λήψης και η σταθερότητα είναι πραγματικά μέρος της εμπειρίας χρήστη. Είτε εκτελείτε ένα έργο ανοιχτού κώδικα είτε ένα εμπορικό προϊόν, θα πρέπει να παρέχετε στους χρήστες έναν αξιόπιστο τρόπο λήψης αρχείων.

Η λήψη αρχείων απευθείας από την αποθήκευση cloud, όπως το Azure Blob Storage ή το AWS S3, φαίνεται απλή, αλλά συνοδεύεται από αρκετά πρακτικά ζητήματα:

**Λάπαντος χρόνου δικτύου**: Η διασυνοριακή και διαπεριφερειακή πρόσβαση μπορεί να είναι αρκετά αργή ώστε να σας κάνει να θέλετε να σπάσετε το πληκτρολόγιό σας. Εάν οι χρήστες πρέπει να περιμένουν για πάντα, η εμπειρία προφανώς δεν θα είναι εξαιρετική.

**Κόστος εύρους ζώνης**: Η κίνηση εξόδου στο χώρο αποθήκευσης cloud είναι επώδυνα ακριβή. Η πρόσβαση στο Azure Blob Storage από την ηπειρωτική Κίνα κοστίζει περίπου 0,5 CNY ανά GB, που σημαίνει ότι 1 TB ανά μήνα ανέρχεται περίπου σε 500 CNY. Για μια μικρή ομάδα, αυτό δεν είναι ασήμαντο ποσό. Εξάλλου, τα λεφτά κανενός δεν προέρχονται από τον άνεμο.

**Περιορισμοί πρόσβασης**: Σε ορισμένες περιοχές, η πρόσβαση σε υπηρεσίες cloud στο εξωτερικό είναι ασταθής και μερικές φορές απλώς δεν είναι διαθέσιμη. Οι χρήστες θέλουν να κατεβάσουν τα αρχεία αλλά δεν μπορούν, κάτι που είναι απογοητευτικό για όλους.

**Κόστος CDN**: Τα εμπορικά CDN μπορούν να λύσουν αυτά τα προβλήματα, αλλά η τιμή είναι εξίσου πραγματική. Οι περισσότερες μικρές ομάδες απλά δεν μπορούν να το δικαιολογήσουν.

Υπάρχει λοιπόν λύση που να είναι και προσιτή και πρακτική; Ναί. Χρησιμοποιήστε έναν διακομιστή cloud, έναν αντίστροφο διακομιστή μεσολάβησης και ένα επίπεδο προσωρινής αποθήκευσης. Είναι μια απλή προσέγγιση, αλλά λειτουργεί. Το κόστος μειώνεται περίπου στο μισό και η ταχύτητα βελτιώνεται επίσης, κάτι που είναι μια αξιοπρεπής αντιστάθμιση.

## Αρχιτεκτονικός Σχεδιασμός

### Συνολική Αρχιτεκτονική

Ας ξεκινήσουμε με την πλήρη αρχιτεκτονική:

```
User request
    ↓
DNS resolution
    ↓
┌─────────────────────────────────────┐
│   Reverse proxy layer (Traefik/Bunker Web) │ ← SSL termination, routing, security protection
├─────────────────────────────────────┤
│   Ports: 80/443                     │
│   Features: Automatic Let's Encrypt certificates │
│             Host routing            │
└─────────────────────────────────────┘
    ↓
┌─────────────────────────────────────┐
│   Cache layer (Nginx)               │ ← File caching, Gzip compression
├─────────────────────────────────────┤
│   Ports: 8080(server) / 8081(desktop) │
│   Cache strategy:                   │
│   - index.json: 1 hour              │
│   - other files: 7 days             │
│   Cache size: 1GB                   │
└─────────────────────────────────────┘
    ↓
┌─────────────────────────────────────┐
│   Origin (Azure Blob Storage)       │ ← File storage
└─────────────────────────────────────┘
```

Η βασική ιδέα αυτής της αρχιτεκτονικής είναι απλή: **τοποθετήστε μια κρυφή μνήμη μεταξύ των χρηστών και του χώρου αποθήκευσης cloud**.

Τα αιτήματα των χρηστών φτάνουν πρώτα στο επίπεδο αντίστροφου διακομιστή μεσολάβησης στον διακομιστή σύννεφο και, στη συνέχεια, αναλαμβάνει το επίπεδο κρυφής μνήμης Nginx. Εάν το ζητούμενο αρχείο είναι ήδη αποθηκευμένο στην προσωρινή μνήμη, επιστρέφεται αμέσως. Εάν όχι, το Nginx το ανακτά από την αποθήκευση cloud και αποθηκεύει ένα τοπικό αντίγραφο ταυτόχρονα. Την επόμενη φορά που κάποιος θα ζητήσει το ίδιο αρχείο, η αποθήκευση cloud δεν χρειάζεται να εμπλακεί ξανά.

### Γιατί να επιλέξετε αυτή την αρχιτεκτονική;

**Πλεονεκτήματα των διακομιστών cloud**:
- Προβλέψιμο κόστος: πάροχοι όπως το Alibaba Cloud προσφέρουν χαμηλού κόστους διακομιστές cloud, με 1-2 περιπτώσεις vCPU και 2 GB RAM με τιμή περίπου 50-100 CNY ανά μήνα
- Ευέλικτη ανάπτυξη: μπορείτε να διαμορφώσετε ελεύθερα κανόνες αντίστροφου διακομιστή μεσολάβησης και πολιτικές αποθήκευσης στην προσωρινή μνήμη
- Ευέλικτη γεωγραφία: μπορείτε να επιλέξετε περιοχές διακομιστών πιο κοντά στους χρήστες σας
- Καλή επεκτασιμότητα: μπορείτε να αναβαθμίσετε τις προδιαγραφές διακομιστή καθώς αυξάνεται η επισκεψιμότητα

**Αρχιτεκτονική αντίστροφου διακομιστή μεσολάβησης + προσωρινής μνήμης**:
- Μειώστε την πίεση προέλευσης: αποθηκεύστε προσωρινά ζεστά αρχεία για να μειώσετε την άμεση πρόσβαση στο χώρο αποθήκευσης cloud
- Χαμηλότερο κόστος: η κίνηση διακομιστή cloud είναι πολύ φθηνότερη από την έξοδο αποθήκευσης cloud
- Βελτιώστε την ταχύτητα: η κοντινή πρόσβαση και το εύρος ζώνης διακομιστή είναι συνήθως καλύτερα από την άμεση παράδοση αποθήκευσης στο cloud

**Γιατί να επιλέξετε το Nginx ως επίπεδο κρυφής μνήμης;**

Αυτή δεν ήταν μια τυχαία επιλογή. Το Nginx έχει πολλά πραγματικά πλεονεκτήματα εδώ:

1. Υψηλή απόδοση: Το Nginx αναγνωρίζεται ευρέως για εξαιρετική απόδοση αντίστροφου διακομιστή μεσολάβησης
2. Ώριμη προσωρινή αποθήκευση: το ενσωματωμένο `proxy_cache` το χαρακτηριστικό είναι σταθερό και αξιόπιστο
3. Χαμηλή χρήση πόρων: μπορεί να τρέξει με μόλις 256 MB μνήμης
4. Ευέλικτη διαμόρφωση: μπορείτε να εφαρμόσετε διαφορετικές πολιτικές προσωρινής μνήμης σε διαφορετικούς τύπους αρχείων

## Reverse Proxy Layer: Traefik vs Bunker Web

Η λύση ανάπτυξης του HagiCode υποστηρίζει δύο επιλογές αντίστροφου διακομιστή μεσολάβησης και η καθεμία έχει τα δικά της πλεονεκτήματα:

| Επιλογή | Χαρακτηριστικά | Κατάλληλα σενάρια |
|------|------|----------|
| **Traefik** | Ελαφρύ, αυτόματο SSL, απλή διαμόρφωση | Βασική ανάπτυξη, σενάρια χαμηλής κυκλοφορίας |
| **Ιστός Bunker** | Ενσωματωμένο WAF, προστασία DDoS, προστασία anti-bot | Σενάρια υψηλής ασφάλειας, υψηλής κυκλοφορίας |

### Traefik: Η ελαφριά πρώτη επιλογή

Το Traefik είναι ένας σύγχρονος HTTP reverse proxy και load balancer. Το μεγαλύτερο πλεονέκτημά του είναι ότι η διαμόρφωση είναι απλή και μπορεί να αποκτήσει αυτόματα πιστοποιητικά Let's Encrypt.

Για αρχικές αναπτύξεις ή σενάρια χαμηλής επισκεψιμότητας, το Traefik είναι συχνά μια πολύ καλή επιλογή:
- Χρησιμοποιεί σχετικά λίγους πόρους. Αρκεί 1,5 CPU και 512 MB μνήμη
- Τα πιστοποιητικά SSL διαμορφώνονται αυτόματα, επομένως δεν χρειάζεται να τα διαχειρίζεστε μόνοι σας
- Η δρομολόγηση διαμορφώνεται μέσω των ετικετών Docker, κάτι που είναι αρκετά βολικό

### Bunker Web: Για σενάρια υψηλής ασφάλειας

Το Bunker Web είναι ένα τείχος προστασίας εφαρμογών ιστού που βασίζεται σε Nginx με πιο ολοκληρωμένη προστασία ασφαλείας.

Πότε πρέπει να σκεφτείτε να μεταβείτε στο Bunker Web; Συνήθως σε περιπτώσεις όπως αυτές:
- Βρίσκεστε υπό επίθεση DDoS
- Χρειάζεστε προστασία ModSecurity
- Θέλετε προστασία anti-bot
- Έχετε αυστηρότερες απαιτήσεις ασφαλείας

Το HagiCode παρέχει το `switch-deployment.sh` script ώστε να μπορείτε να κάνετε εναλλαγή μεταξύ των δύο επιλογών:

```bash
# Switch to Bunker Web
./switch-deployment.sh bunkerweb

# Switch back to Traefik
./switch-deployment.sh traefik

# Check current status
./switch-deployment.sh status
```

Το σενάριο εκτελεί προκαταρκτικούς ελέγχους, ελέγχους υγείας και αυτόματη επαναφορά, επομένως η διαδικασία εναλλαγής είναι αρκετά ασφαλής και αξιόπιστη.

## Διαμόρφωση επιπέδου κρυφής μνήμης Nginx

Το επίπεδο κρυφής μνήμης είναι ο πυρήνας ολόκληρης της αρχιτεκτονικής, επομένως η διαμόρφωση Nginx κάνει τεράστια διαφορά στην απόδοση της κρυφής μνήμης.

### Διαμόρφωση διαδρομής προσωρινής μνήμης

```nginx
# Cache path configuration
proxy_cache_path /var/cache/nginx levels=1:2 keys_zone=azure_cache:10m
                   max_size=1g inactive=7d use_temp_path=off;
```

Λεπτομέρειες παραμέτρων:
- `levels=1:2`: ιεραρχία καταλόγου κρυφής μνήμης με δύο επίπεδα για βελτίωση της αποτελεσματικότητας πρόσβασης στα αρχεία
- `keys_zone=azure_cache:10m`: ζώνη αποθήκευσης κλειδιών κρυφής μνήμης. 10 MB είναι αρκετά για μεγάλο αριθμό πλήκτρων
- `max_size=1g`: το μέγιστο μέγεθος προσωρινής μνήμης είναι 1 GB
- `inactive=7d`: διαγράψτε τα αποθηκευμένα αρχεία εάν δεν έχετε πρόσβαση σε αυτά για 7 ημέρες
- `use_temp_path=off`: γράψτε απευθείας στον κατάλογο της προσωρινής μνήμης για καλύτερη απόδοση

### Στρατηγική κρυφής μνήμης σε επίπεδα

Διαφορετικοί τύποι αρχείων χρειάζονται διαφορετικές στρατηγικές κρυφής μνήμης:

```nginx
# Server download service
server {
    listen 8080;

    # Short-term cache for index.json (to allow timely updates)
    location /index.json {
        proxy_cache azure_cache;
        proxy_cache_valid 200 1h;
        proxy_cache_key "$scheme$server_port$request_uri";
        add_header X-Cache-Status $upstream_cache_status;
        add_header Cache-Control "public, max-age=3600";

        # Reverse proxy to Azure OSS
        proxy_pass https://${SERVER_DL_HOST}/${SERVER_DL_CONTAINER}$uri?${SERVER_DL_SAS_TOKEN};
        proxy_ssl_server_name on;
        proxy_ssl_protocols TLSv1.2 TLSv1.3;
    }

    # Long-term cache for static files such as installation packages
    location / {
        proxy_cache azure_cache;
        proxy_cache_valid 200 7d;
        proxy_cache_key "$scheme$server_port$request_uri";
        add_header X-Cache-Status $upstream_cache_status;
        add_header Cache-Control "public, max-age=604800";

        proxy_pass https://${SERVER_DL_HOST}/${SERVER_DL_CONTAINER}$uri?${SERVER_DL_SAS_TOKEN};
        proxy_ssl_server_name on;
        proxy_ssl_protocols TLSv1.2 TLSv1.3;
    }
}
```

**Γιατί έχει σχεδιαστεί έτσι;**

`index.json` είναι το αρχείο ελέγχου έκδοσης, επομένως πρέπει να ενημερωθεί άμεσα. Με ένα παράθυρο κρυφής μνήμης 1 ώρας, οι χρήστες μπορούν να εντοπίσουν μια νέα έκδοση το πολύ εντός μίας ώρας μετά τη δημοσίευση.

Τα στατικά αρχεία, όπως τα πακέτα εγκατάστασης, αλλάζουν σπάνια, επομένως η προσωρινή αποθήκευση τους για 7 ημέρες μειώνει σημαντικά την πρόσβαση προέλευσης. Όταν απαιτείται ενημέρωση, μπορείτε απλώς να διαγράψετε την προσωρινή μνήμη με μη αυτόματο τρόπο.

**`X-Cache-Status` κεφαλίδα απάντησης**:

Αυτή η κεφαλίδα σάς βοηθά να επιθεωρήσετε τη συμπεριφορά επίσκεψης στην κρυφή μνήμη:
- `HIT`: χτύπημα κρυφής μνήμης
- `MISS`: απουσία κρυφής μνήμης, λήφθηκε από την προέλευση
- `EXPIRED`: η προσωρινή μνήμη έληξε, λήφθηκε ξανά από την αρχή
- `BYPASS`: παράκαμψη της προσωρινής μνήμης

Πώς να το ελέγξετε:
```bash
curl -I https://server.dl.hagicode.com/app.zip
```

## Συμβουλές λειτουργίας

### Διαχείριση προσωρινής μνήμης

Οι κρυφές μνήμες χρειάζονται επίσης συντήρηση από καιρό σε καιρό:

**Ελέγξτε τη χρήση του δίσκου της προσωρινής μνήμης**:
```bash
docker volume inspect docker_nginx-cache
du -sh /var/lib/docker/volumes/docker_nginx-cache/_data
```

**Εκκαθάριση της προσωρινής μνήμης με μη αυτόματο τρόπο**:
```bash
./clear-cache.sh
```

Ή εκτελέστε τις μη αυτόματες εντολές απευθείας εάν χρειάζεται:
```bash
docker exec nginx sh -c "rm -rf /var/cache/nginx/*"
docker restart nginx
```

### Όρια πόρων

Σε έναν διακομιστή 1 πυρήνα, 2 GB, η διαμόρφωση ορίου πόρων μοιάζει με αυτό:

```yaml
services:
  traefik:
    deploy:
      resources:
        limits:
          cpus: '1.50'
          memory: 512M

  nginx:
    deploy:
      resources:
        limits:
          cpus: '0.50'
          memory: 256M
```

Για να παρακολουθείτε τη χρήση πόρων, μπορείτε περιστασιακά να εκτελείτε:
```bash
docker stats
```

### SAS Token Security Practices

Το SAS Token είναι το διαπιστευτήριο που χρησιμοποιείται για την πρόσβαση στο Azure Blob Storage, επομένως η διαρροή του θα ήταν σοβαρή:

- Μην δεσμεύσετε το `.env` έλεγχος από αρχείο σε έκδοση. είναι ήδη μέσα `.gitignore`
- Ορίστε έναν κατάλληλο χρόνο λήξης SAS Token, με προτεινόμενο 1 έτος
- Περιορίστε τα δικαιώματα SAS Token μόνο για ανάγνωση
- Περιστρέψτε τακτικά τα SAS Tokens

### Παρακολούθηση και Ειδοποιήσεις

Το HagiCode ενσωματώνει ειδοποιήσεις Lark/Feishu Webhook, οι οποίες μπορούν να στείλουν ειδοποιήσεις για τα ακόλουθα συμβάντα:
- Επιτυχία ή αποτυχία ανάπτυξης
- Κατάσταση εκκαθάρισης προσωρινής μνήμης
- Εξαιρέσεις υπηρεσιών

Οι ειδοποιήσεις περιλαμβάνουν πληροφορίες διακομιστή, χρονικές σημάνσεις και λεπτομέρειες σφαλμάτων, κάνοντας την αντιμετώπιση προβλημάτων πολύ πιο γρήγορη.

### Επεκτάσεις υψηλής διαθεσιμότητας

Όταν ένας διακομιστής δεν είναι πλέον αρκετός, μπορείτε να εξετάσετε τα εξής:

1. **Οριζόντια κλιμάκωση**: αναπτύξτε πολλούς κόμβους και κατανείμετε την κυκλοφορία με DNS round-robin ή εξισορροπητή φορτίου
2. **CDN μπροστά**: τοποθετήστε ένα CDN μπροστά από τους διακομιστές cloud για ακόμα πιο γρήγορη πρόσβαση
3. **Θέρμανση προσωρινής μνήμης**: χρησιμοποιήστε δέσμες ενεργειών για προφόρτωση καυτών αρχείων στην κρυφή μνήμη

## Σημειώσεις

Υπάρχουν μερικά πράγματα που αξίζει να έχετε υπόψη σας:

1. **Πιστοποιητικά SSL**: Το Let's Encrypt έχει όρια ρυθμού, επομένως μην αλλάζετε τις αναπτύξεις πολύ συχνά διαφορετικά η έκδοση πιστοποιητικού μπορεί να αποτύχει
2. **Εκκαθάριση προσωρινής μνήμης**: μετά την ενημέρωση σημαντικών αρχείων, θυμηθείτε να διαγράψετε την προσωρινή μνήμη, διαφορετικά οι χρήστες μπορούν να κάνουν λήψη της παλιάς έκδοσης
3. **Διαχείριση αρχείων καταγραφής**: καθαρίστε τακτικά τα αρχεία καταγραφής Docker, διαφορετικά ο δίσκος μπορεί να γεμίσει
4. **Στρατηγική δημιουργίας αντιγράφων ασφαλείας**: δημιουργία αντιγράφων ασφαλείας αρχείων όπως π.χ `Traefik acme.json` και Bunker Web configuration
5. **Παρακολούθηση και ειδοποιήσεις**: διαμορφώστε τις ειδοποιήσεις Feishu, ώστε να μπορείτε να παρακολουθείτε την κατάσταση ανάπτυξης και να απαντάτε γρήγορα σε ζητήματα

## Συμπέρασμα

Ένας διακομιστής cloud συν ένα επίπεδο προσωρινής αποθήκευσης Nginx είναι το μόνο που χρειάζεται. Το HagiCode χρησιμοποιεί αυτή τη λύση με ένα αρκετά χαμηλό μηνιαίο κόστος, περίπου 60-100 CNY για τον διακομιστή, και τα αποτελέσματα ήταν πολύ καλά. Τα κύρια πλεονεκτήματα είναι:

- **Προβλέψιμο κόστος**: περίπου 50% φθηνότερο από τη χρήση απευθείας αποθήκευσης στο cloud ή την πληρωμή για ένα εμπορικό CDN
- **Ευέλικτη ανάπτυξη**: επιλέξτε Traefik ή Bunker Web ανάλογα με τις ανάγκες σας
- **Ισχυρή επεκτασιμότητα**: μπορείτε να κλιμακώσετε οριζόντια ή να προσθέσετε ένα CDN αργότερα, εάν χρειάζεται
- **Απλές λειτουργίες**: Τα σενάρια Shell συν το Ansible κάνουν την αυτοματοποιημένη ανάπτυξη απλή

Για μικρές ομάδες και ανεξάρτητους προγραμματιστές που χρειάζονται διανομή αρχείων, αυτή είναι σίγουρα μια πρακτική επιλογή που αξίζει να δοκιμάσετε.

Το HagiCode εκτελούσε αυτήν την αρχιτεκτονική σταθερά στην παραγωγή για λίγο και οι παγκόσμιες λήψεις χρηστών παρέμειναν αξιόπιστες. Αν ψάχνετε για μια παρόμοια λύση, αξίζει να δοκιμάσετε.
