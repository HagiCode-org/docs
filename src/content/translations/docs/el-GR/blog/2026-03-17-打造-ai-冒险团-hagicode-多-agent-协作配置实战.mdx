## Χτίζοντας ένα πάρτι περιπέτειας AI: Διαμόρφωση συνεργασίας πολλαπλών πρακτόρων HagiCode στην πράξη

> Στη σύγχρονη ανάπτυξη λογισμικού, ένας μόνο Πράκτορας AI δεν αρκεί πλέον για να καλύψει πολύπλοκες απαιτήσεις. Πώς μπορούν πολλοί βοηθοί τεχνητής νοημοσύνης από διαφορετικές εταιρείες να συνεργαστούν στο ίδιο έργο; Αυτό το άρθρο μοιράζεται την προσέγγιση διαμόρφωσης συνεργασίας πολλών πρακτόρων που ανέπτυξε το έργο HagiCode μέσω πρακτικής στον πραγματικό κόσμο.

## φόντο

Πολλοί προγραμματιστές είχαν πιθανώς αυτήν την εμπειρία: μετά την εισαγωγή ενός βοηθού AI σε ένα έργο, η παραγωγικότητα πραγματικά βελτιώνεται. Αλλά καθώς οι απαιτήσεις γίνονται όλο και πιο περίπλοκες, ένας Πράκτορας AI αρχίζει να αισθάνεται ανεπαρκής. Θέλετε να χειρίζεται την αναθεώρηση κώδικα, τη δημιουργία τεκμηρίωσης, τη δοκιμή μονάδων και άλλες εργασίες ταυτόχρονα, αλλά το αποτέλεσμα είναι συχνά ότι δεν μπορεί να κρατήσει τα πάντα ισορροπημένα και η ποιότητα εξόδου γίνεται ασυνεπής.

Αυτό που είναι ακόμη πιο απογοητευτικό είναι ότι μόλις προσπαθήσετε να φέρετε πολλούς βοηθούς AI, το πρόβλημα γίνεται πιο περίπλοκο. Κάθε Πράκτορας έχει τη δική του μέθοδο διαμόρφωσης, διεπαφή API και λογική εκτέλεσης και μπορεί ακόμη και να έρχεται σε σύγκρουση μεταξύ τους. Είναι σαν μια αθλητική ομάδα στην οποία κάθε παίκτης είναι ταλαντούχος, αλλά κανείς δεν ξέρει πώς να συνεργαστεί, οπότε ο αγώνας μετατρέπεται σε χάος.

Το έργο HagiCode αντιμετώπισε την ίδια πρόκληση κατά τη διάρκεια της ανάπτυξης. Ως ένα πολύπλοκο έργο που περιλαμβάνει μια επέκταση VSCode, υπηρεσίες τεχνητής νοημοσύνης υποστήριξης και έναν υπολογιστή-πελάτη για υπολογιστές πολλαπλών πλατφορμών, χρειαζόμασταν να συνδέσουμε πολλούς βοηθούς τεχνητής νοημοσύνης από διαφορετικές εταιρείες ταυτόχρονα: Claude Code, Codex, CodeBuddy, iFlow και πολλά άλλα. Το πώς να τους αφήσουμε να συνυπάρξουν αρμονικά στο ίδιο έργο και να αξιοποιήσουν στο έπακρο τις δυνάμεις τους έγινε βασικό πρόβλημα που έπρεπε να λύσουμε.

Αυτό και μόνο θα ήταν ήδη αρκετό πρόβλημα. Τελικά, ποιος θέλει να ασχολείται με ένα σωρό μαχητικά AI κάθε μέρα;

Η προσέγγιση που μοιράζεται σε αυτό το άρθρο είναι η πρακτική διαμόρφωσης συνεργασίας πολλών πρακτόρων που αναπτύξαμε στο έργο HagiCode μέσω πραγματικών δοκιμών και σφαλμάτων και επαναλαμβανόμενης βελτιστοποίησης. Εάν δυσκολεύεστε επίσης με πολλούς βοηθούς τεχνητής νοημοσύνης που συνεργάζονται, αυτό το άρθρο μπορεί να σας δώσει κάποια έμπνευση. Ισως. Κάθε έργο είναι άλλωστε διαφορετικό.

## Σχετικά με το HagiCode

[HagiCode](https://hagicode.com) είναι ένα έργο βοηθού κωδικοποίησης AI που υιοθετεί ένα μοντέλο "πάρτι περιπέτειας" στο οποίο συνεργάζονται πολλαπλοί κινητήρες τεχνητής νοημοσύνης. Αποθετήριο έργου: [github.com/HagiCode-org/site](https://github.com/HagiCode-org/site).

Η προσέγγιση διαμόρφωσης πολλαπλών παραγόντων που κοινοποιείται σε αυτό το άρθρο είναι μία από τις βασικές τεχνολογίες που επιτρέπει στο HagiCode να διατηρεί αποτελεσματική ανάπτυξη σε πολύπλοκα έργα. Δεν υπάρχει τίποτα ιδιαίτερα μαγικό σε αυτό. απλά μετατρέπει μια ομάδα AI σε ένα πάρτι περιπέτειας που μπορεί πραγματικά να συντονίσει.

## Σχεδιασμός Αρχιτεκτονικής Multi-Agent της HagiCode

### Από το "Going Solo" στο "Team Collaboration"

Στις πρώτες μέρες του έργου HagiCode, προσπαθήσαμε επίσης να χρησιμοποιήσουμε έναν μόνο πράκτορα AI για να χειριστούμε κάθε εργασία. Σύντομα ανακαλύψαμε ένα σαφές σημείο συμφόρησης σε αυτήν την προσέγγιση: διαφορετικές εργασίες απαιτούν διαφορετικά πλεονεκτήματα. Ορισμένες εργασίες χρειάζονται ισχυρότερη κατανόηση των συμφραζομένων, ενώ άλλες χρειάζονται πιο ακριβείς δυνατότητες τροποποίησης κώδικα. Ο One Agent δυσκολεύεται να διαπρέψει σε όλα.

Αυτό μας έκανε να συνειδητοποιήσουμε ότι πολλοί Πράκτορες έπρεπε να συνεργαστούν. Αλλά το πρόβλημα ήταν το εξής: πώς αφήνεις προϊόντα τεχνητής νοημοσύνης από διαφορετικές εταιρείες να συνυπάρχουν ειρηνικά στο ίδιο έργο; Χρειαζόταν να λύσουμε πολλά βασικά ζητήματα:

1. **Πολυπλοκότητα διαχείρισης διαμόρφωσης**: κάθε Agent έχει διαφορετικές μεθόδους διαμόρφωσης, διεπαφές API και τρόπους εκτέλεσης
2. **Ενοποιημένο πρωτόκολλο επικοινωνίας**: χρειαζόμαστε έναν τυποποιημένο τρόπο ανταλλαγής δεδομένων από διαφορετικούς πράκτορες
3. **Συντονισμός εργασιών και καταμερισμός της εργασίας**: πώς αναθέτουμε την εργασία εύλογα ώστε κάθε Πράκτορας να μπορεί να παίξει σύμφωνα με τις δυνάμεις του

Έχοντας αυτές τις ερωτήσεις στο μυαλό, ξεκινήσαμε να σχεδιάζουμε την αρχιτεκτονική πολλών πρακτόρων της HagiCode. Στην πραγματικότητα δεν ήταν τόσο περίπλοκο. έπρεπε απλώς να το σκεφτούμε καθαρά.

### Συνολική Αρχιτεκτονική με μια ματιά

Μετά από πολλές επαναλήψεις, αυτή είναι η αρχιτεκτονική στην οποία καταλήξαμε:

```
┌─────────────────────────────────────────────────────────────────┐
│                    AIProviderFactory                             │
│  (Factory pattern for unified management of all AI Providers)    │
├─────────────────────────────────────────────────────────────────┤
│  ClaudeCodeCli  │  CodexCli  │  CodebuddyCli  │  IFlowCli    │
│  (Anthropic)   │  (OpenAI)  │  (Zhipu GLM)    │  (Zhipu)     │
└─────────────────────────────────────────────────────────────────┘
```

Η βασική ιδέα είναι να επιτραπεί η διαχείριση διαφορετικών AI Agents από το ίδιο σύνολο κώδικα μέσω μιας ενοποιημένης διεπαφής παρόχου. Ταυτόχρονα, το εργοστασιακό μοτίβο χρησιμοποιείται για τη δυναμική δημιουργία και διαμόρφωση αυτών των Παρόχων, διασφαλίζοντας επεκτασιμότητα και ευελιξία σε όλο το σύστημα.

Είναι σαν καταμερισμός εργασίας στην καθημερινή ζωή. Ο καθένας έχει τον δικό του ρόλο. Εδώ απλώς μετατρέψαμε αυτήν την ιδέα σε αρχιτεκτονική κώδικα.

### Τύποι πρακτόρων και κατανομή αρμοδιοτήτων

Με βάση την πραγματική εμπειρία της HagiCode, αναθέσαμε διαφορετικές ευθύνες σε κάθε Πράκτορα:

| Πράκτορας | Πάροχος | Μοντέλο | Πρωτογενής Χρήση |
|-------|----------|-------|-------------|
| ClaudeCodeCli | Ανθρωπικό | glm-5-turbo | Δημιουργία τεχνικών λύσεων και προτάσεων |
| CodexCli | OpenAI/Zed | gpt-5.4 | Εκτελέστε ακριβείς αλλαγές κώδικα |
| CodebuddyCli | Ζιπού | glm-4,7 | Βελτιώστε τις περιγραφές και την τεκμηρίωση των προτάσεων |
| IFlowCli | Ζιπού | glm-4,7 | Αρχειακές προτάσεις και ιστορικές καταγραφές |
| OpenCodeCli | - | - | Επεξεργασία κώδικα γενικής χρήσης |
| GitHubCopilot | Microsoft | - | Υποβοηθούμενος προγραμματισμός και συμπλήρωση κώδικα |

Η λογική πίσω από αυτόν τον καταμερισμό εργασίας είναι απλή: κάθε Πράκτορας έχει τη δική του περιοχή ισχύος. Ο Claude Code έχει καλή απόδοση στην κατανόηση και την ανάλυση πολύπλοκων απαιτήσεων, επομένως χειρίζεται τον πρώιμο σχεδιασμό λύσεων. Το Codex είναι πιο ακριβές όταν τροποποιεί τον κώδικα, επομένως είναι πιο κατάλληλο για συγκεκριμένες εργασίες υλοποίησης. Το CodeBuddy προσφέρει ισχυρή απόδοση κόστους, γεγονός που το καθιστά ιδανικό για τη βελτίωση του κειμένου και της τεκμηρίωσης της πρότασης.

Άλλωστε το κατάλληλο εργαλείο για τη σωστή δουλειά είναι η καλύτερη επιλογή. Υπάρχουν πολλοί δρόμοι προς τη Ρώμη. μερικά είναι απλά πιο εύκολα στο περπάτημα από άλλα.

## Μηχανισμοί Διαμόρφωσης Πυρήνα

### Σχεδιασμός διεπαφής ενοποιημένου παρόχου

Για να διαχειριστούμε διαφορετικούς πράκτορες AI με ενοποιημένο τρόπο, πρέπει πρώτα να ορίσουμε μια κοινή διεπαφή. Στο HagiCode, αυτή η διεπαφή μοιάζει με αυτό:

```csharp
public interface IAIProvider
{
    // Unified Provider interface
    Task<IAIProvider?> GetProviderAsync(AIProviderType providerType);
    Task<IAIProvider?> GetProviderAsync(string providerName, CancellationToken cancellationToken);
}
```

Η διεπαφή φαίνεται απλή, αλλά είναι το θεμέλιο ολόκληρου του συστήματος πολλαπλών πρακτόρων. Με μια ενοποιημένη διεπαφή, μπορούμε να καλούμε προϊόντα τεχνητής νοημοσύνης από διαφορετικές εταιρείες με τον ίδιο τρόπο, ανεξάρτητα από το ποια εταιρεία βρίσκεται πίσω από αυτά.

Αυτό είναι στην πραγματικότητα απλώς για να κάνουμε τα σύνθετα πράγματα απλά. Άλλωστε το απλό είναι όμορφο.

### Εφαρμογή εργοστασιακών προτύπων παρόχου

Μόλις ενοποιηθεί η διεπαφή, η επόμενη ερώτηση είναι πώς να δημιουργήσετε αυτές τις παρουσίες Παρόχου. Το HagiCode χρησιμοποιεί το εργοστασιακό μοτίβο:

```csharp
private IAIProvider? CreateProvider(AIProviderType providerType, ProviderConfiguration config)
{
    return providerType switch
    {
        AIProviderType.ClaudeCodeCli =>
            ActivatorUtilities.CreateInstance<ClaudeCodeCliProvider>(_serviceProvider, Options.Create(config)),
        AIProviderType.CodebuddyCli =>
            ActivatorUtilities.CreateInstance<CodebuddyCliProvider>(_serviceProvider, Options.Create(config)),
        AIProviderType.CodexCli =>
            ActivatorUtilities.CreateInstance<CodexCliProvider>(_serviceProvider, Options.Create(config)),
        AIProviderType.IFlowCli =>
            ActivatorUtilities.CreateInstance<IFlowCliProvider>(_serviceProvider, Options.Create(config)),
        _ => null
    };
}
```

Αυτό χρησιμοποιεί ένεση εξάρτησης μέσω `ActivatorUtilities.CreateInstance`, το οποίο μπορεί να δημιουργήσει δυναμικά στιγμιότυπα Provider κατά το χρόνο εκτέλεσης ενώ εισάγει αυτόματα εξαρτήσεις. Το πλεονέκτημα αυτού του σχεδιασμού είναι ότι όταν προστίθεται ένας νέος τύπος Agent, χρειάζεται μόνο να προσθέσετε την αντίστοιχη κλάση Provider και στη συνέχεια να προσθέσετε έναν ακόμη `case` κλάδος στην εργοστασιακή μέθοδο. Δεν χρειάζεται καθόλου τροποποίηση του υπάρχοντος κώδικα.

Αυτός είναι αρκετός λόγος. Ποιος θέλει να ξαναγράφει ένα σωρό παλιού κώδικα κάθε φορά που προστίθεται ένα νέο χαρακτηριστικό;

### Ανάλυση δυναμικής διαμόρφωσης

Για να κάνουμε τη διαμόρφωση πιο ευέλικτη, εφαρμόσαμε επίσης έναν μηχανισμό αντιστοίχισης τύπων:

```csharp
public static AIProviderTypeExtensions
{
    private static readonly Dictionary<string, AIProviderType> _typeMap = new(
        StringComparer.OrdinalIgnoreCase)
    {
        ["ClaudeCodeCli"] = AIProviderType.ClaudeCodeCli,
        ["CodebuddyCli"] = AIProviderType.CodebuddyCli,
        ["CodexCli"] = AIProviderType.CodexCli,
        ["IFlowCli"] = AIProviderType.IFlowCli,
        // ...more type mappings
    };
}
```

Ο σκοπός αυτού του πίνακα αντιστοίχισης είναι να μετατρέψει τα ονόματα παρόχων μορφής συμβολοσειράς σε τύπους enum. Αυτό επιτρέπει στα αρχεία διαμόρφωσης να χρησιμοποιούν διαισθητικά ονόματα συμβολοσειρών, ενώ ο εσωτερικός κώδικας χρησιμοποιεί κωδικούς ασφαλείας τύπου για επεξεργασία.

Η διαμόρφωση πρέπει να είναι όσο το δυνατόν πιο διαισθητική. Κανείς δεν θέλει να απομνημονεύσει ένα σωρό περίπλοκων κωδικών ονομάτων.

### Παράδειγμα αρχείου διαμόρφωσης

Στην πράξη, όλα μπορούν να ρυθμιστούν `appsettings.json`:

```yaml
AI:
  Providers:
    Providers:
      ClaudeCodeCli:
        Enabled: true
        Model: glm-5-turbo
        WorkingDirectory: /path/to/project
      CodebuddyCli:
        Enabled: true
        Model: glm-4.7
      CodexCli:
        Enabled: true
        Model: gpt-5.4
      IFlowCli:
        Enabled: true
        Model: glm-4.7
```

Κάθε πάροχος μπορεί να διαμορφώσει ανεξάρτητα παραμέτρους όπως ενεργοποίηση, έκδοση μοντέλου και κατάλογο εργασίας. Αυτός ο σχεδιασμός διατηρεί την ευελιξία ενώ παραμένει εύκολος στη διαχείριση και τη συντήρηση.

Τα αρχεία διαμόρφωσης μοιάζουν λίγο με τις επιλογές της ζωής: μπορείτε να επιλέξετε να ενεργοποιήσετε ή να απενεργοποιήσετε ορισμένα πράγματα. Η μόνη διαφορά είναι ότι οι επιλογές κωδικών είναι πιο εύκολο να μετανιώσετε αργότερα.

## Ροή εργασιών πάρτι περιπέτειας

### The Art of Task Division

Με την ενοποιημένη τεχνική αρχιτεκτονική σε ισχύ, το επόμενο βήμα είναι η συνεργασία πολλών αντιπροσώπων. Η HagiCode σχεδίασε έναν μηχανισμό ροής εργασιών, έτσι ώστε διαφορετικοί Πράκτορες να μπορούν να χειριστούν διαφορετικά στάδια της εργασίας:

```
Proposal creation (user)
    │
    ▼
[Claude Code] ──generate proposal──▶ Proposal document
    │                               │
    │                               ▼
    │                      [Codebuddy] ──refine description──▶ Refined proposal
    │                               │
    │                               ▼
    │                      [Codex] ──execute changes──▶ Code changes
    │                               │
    │                               ▼
    └──────────────────────▶ [iFlow] ──archive──▶ Historical records
```

Το όφελος αυτού του καταμερισμού εργασίας είναι ότι κάθε Πράκτορας χρειάζεται μόνο να εστιάζει στις εργασίες που κάνει καλύτερα, αντί να προσπαθεί να κάνει τα πάντα. Ο Claude Code είναι υπεύθυνος για τη δημιουργία προτάσεων από την αρχή. Το Codebuddy κάνει τις περιγραφές προτάσεων πιο σαφείς. Το Codex μετατρέπει τις προτάσεις σε πραγματικές αλλαγές κώδικα. Το iFlow αρχειοθετεί και διατηρεί αυτές τις αλλαγές.

Αυτό είναι πραγματικά απλώς ομαδική εργασία, όπως στην καθημερινή ζωή. Ο καθένας έχει τον δικό του ρόλο και μόνο μαζί μπορεί να γίνει κάτι μεγάλο. Η μόνη διαφορά είναι ότι τα μέλη της ομάδας εδώ τυχαίνει να είναι AI.

### Βασικά πρακτικά συμπεράσματα

Στην πραγματική λειτουργία, συνοψίσαμε τα ακόλουθα μαθήματα:

**1. Η στρατηγική επιλογής αντιπροσώπου έχει σημασία**

Οι εργασίες δεν πρέπει να ανατίθενται τυχαία. θα πρέπει να ταιριάζουν με τα δυνατά σημεία κάθε Αντιπροσώπου:
- Δημιουργία προτάσεων: χρησιμοποιήστε τον Κώδικα Claude, επειδή κατανοεί καλύτερα τα συμφραζόμενα
- Εκτέλεση κώδικα: χρησιμοποιήστε το Codex, επειδή είναι πιο ακριβές για την τροποποίηση κώδικα
- Βελτίωση πρότασης: χρησιμοποιήστε το Codebuddy, επειδή προσφέρει ισχυρή απόδοση κόστους
- Αποθήκευση αρχείων: χρησιμοποιήστε το iFlow, γιατί είναι σταθερό και αξιόπιστο

Εξάλλου, το να βάζεις το σωστό άτομο στο σωστό έργο είναι μια διαχρονική αρχή.

**2. Η απομόνωση διαμόρφωσης εξασφαλίζει σταθερότητα**

Η διαχείριση της διαμόρφωσης κάθε πράκτορα γίνεται ανεξάρτητα, υποστηρίζει παρακάμψεις μεταβλητών περιβάλλοντος και χρησιμοποιεί ξεχωριστούς καταλόγους εργασίας. Ως αποτέλεσμα, ένα σφάλμα διαμόρφωσης σε έναν Πράκτορα δεν επηρεάζει τους άλλους.

Αυτό είναι σαν τα προσωπικά όρια στη ζωή. Ο καθένας χρειάζεται τον δικό του χώρο. η μη παρεμβολή καθιστά δυνατή την αρμονική συνύπαρξη.

**3. Μηχανισμός χειρισμού σφαλμάτων**

Μια αποτυχία σε έναν μόνο Πράκτορα δεν πρέπει να επηρεάζει τη συνολική ροή εργασίας. Εφαρμόσαμε μια εναλλακτική στρατηγική: όταν ένας Πράκτορας αποτυγχάνει, το σύστημα μπορεί αυτόματα να μεταβεί σε ένα εφεδρικό σχέδιο ή να παραλείψει αυτό το βήμα και να συνεχίσει με μεταγενέστερες εργασίες. Ταυτόχρονα, η πλήρης καταγραφή διευκολύνει την αντιμετώπιση προβλημάτων στη συνέχεια.

Κανείς δεν μπορεί να εγγυηθεί ότι τα λάθη δεν θα συμβούν ποτέ. Το κλειδί είναι πώς τα χειρίζεσαι. Η ζωή λειτουργεί σχεδόν με τον ίδιο τρόπο.

**4. Παρακολούθηση και παρατηρησιμότητα**

Μέσω του πρωτοκόλλου ACP (το προσαρμοσμένο μας πρωτόκολλο επικοινωνίας που βασίζεται στο JSON-RPC 2.0), μπορούμε να παρακολουθούμε την κατάσταση εκτέλεσης κάθε Agent. Η απομόνωση περιόδου λειτουργίας διασφαλίζει την ασφάλεια ταυτόχρονης χρήσης, ενώ η δυναμική προσωρινή αποθήκευση βελτιώνει την απόδοση.

Τα πράγματα που δεν μπορείτε να δείτε είναι συχνά αυτά που είναι πιο πιθανό να πάνε στραβά. Η ορατότητα είναι πάντα καλύτερη από το να πετάς στα τυφλά.

## Αποτελέσματα και οφέλη στον πραγματικό κόσμο

Μετά την υιοθέτηση αυτής της διαμόρφωσης συνεργασίας πολλαπλών πρακτόρων, η αποδοτικότητα ανάπτυξης του έργου HagiCode βελτιώθηκε σημαντικά. Συγκεκριμένα:

1. **Διπλασιάστηκε η ικανότητα χειρισμού εργασιών**: στο παρελθόν, ένας Πράκτορας έπρεπε να χειρίζεται πολλά είδη εργασιών ταυτόχρονα. Τώρα οι εργασίες μπορούν να επεξεργάζονται παράλληλα και η απόδοση έχει αυξηθεί δραματικά
2. **Πιο σταθερή ποιότητα εξόδου**: κάθε Πράκτορας εστιάζει μόνο σε αυτό που κάνει καλύτερα, επομένως η συνέπεια και η ποιότητα βελτιώνονται και οι δύο
3. **Χαμηλότερο κόστος συντήρησης**: οι ενοποιημένες διεπαφές και η διαχείριση διαμόρφωσης κάνουν όλο το σύστημα πιο εύκολο στη συντήρηση και επέκταση
4. **Η προσθήκη νέων πρακτόρων είναι απλή**: για να ενσωματώσετε ένα νέο προϊόν τεχνητής νοημοσύνης, χρειάζεται μόνο να εφαρμόσετε τη διεπαφή και να προσθέσετε διαμόρφωση, χωρίς να αλλάξετε τη βασική λογική

Αυτή η προσέγγιση όχι μόνο έλυσε τα προβλήματα της ίδιας της HagiCode, αλλά απέδειξε επίσης ότι η συνεργασία πολλών πρακτόρων είναι μια βιώσιμη αρχιτεκτονική επιλογή.

Τα κέρδη ήταν αρκετά αισθητά. Η διαδικασία ήταν απλώς μια μικρή ταλαιπωρία.

## Συμπέρασμα

Αυτό το άρθρο μοιράστηκε την πρακτική εμπειρία του έργου HagiCode με τη διαμόρφωση συνεργασίας πολλών πρακτόρων. Οι κύριες προτάσεις περιλαμβάνουν:

1. **Τυποποιημένες διεπαφές**: `IAIProvider` ενοποιεί τη συμπεριφορά διαφορετικών αντιπροσώπων, επιτρέποντας στον κώδικα να αγνοεί ποιο προϊόν της εταιρείας βρίσκεται από κάτω
2. **Εργοστασιακό μοτίβο**: `ActivatorUtilities.CreateInstance` δημιουργεί δυναμικά στιγμιότυπα Provider, υποστηρίζοντας τη διαμόρφωση χρόνου εκτέλεσης και την ένεση εξάρτησης
3. **Ενοποίηση πρωτοκόλλου**: το πρωτόκολλο ACP παρέχει τυποποιημένη επικοινωνία μεταξύ Agents μέσω ενός αμφίδρομου μηχανισμού που βασίζεται στο JSON-RPC 2.0
4. **Δρομολόγηση εργασιών**: αναθέστε εργασία εύλογα σε διαφορετικούς πράκτορες, ώστε ο καθένας να μπορεί να παίξει με τα δυνατά του σημεία, αντί να περιμένετε από έναν πράκτορα να κάνει τα πάντα

Αυτός ο σχεδιασμός όχι μόνο λύνει το πρόβλημα "πολλοί Πράκτορες που μάχονται μεταξύ τους", αλλά χρησιμοποιεί επίσης τον μηχανισμό ροής εργασιών πάρτι περιπέτειας για να κάνει τη διαδικασία ανάπτυξης πιο αυτοματοποιημένη και εξειδικευμένη.

Εάν σκέφτεστε επίσης να εισαγάγετε πολλούς βοηθούς τεχνητής νοημοσύνης, ελπίζω αυτό το άρθρο να σας δώσει μερικά χρήσιμα σημεία αναφοράς. Φυσικά, κάθε έργο είναι διαφορετικό και η συγκεκριμένη προσέγγιση πρέπει ακόμα να προσαρμοστεί στην πραγματική κατάσταση. Δεν υπάρχει λύση που να ταιριάζει σε όλους. η καλύτερη λύση είναι αυτή που σου ταιριάζει.

Τα όμορφα πράγματα ή οι άνθρωποι δεν χρειάζεται να κατέχονται. Όσο παραμένουν όμορφες, αρκεί απλώς να εκτιμάς αυτή την ομορφιά. Οι τεχνικές λύσεις είναι οι ίδιες: αυτή που σας ταιριάζει είναι η καλύτερη...
