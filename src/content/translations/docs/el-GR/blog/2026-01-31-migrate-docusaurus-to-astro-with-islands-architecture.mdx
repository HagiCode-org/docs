## From Docusaurus 3.x to Astro 5.x: A Retrospective on the HagiCode Site Migration

> Αυτό το άρθρο κάνει μια αναδρομή στην πλήρη μετεγκατάσταση του επίσημου ιστότοπου HagiCode από το Docusaurus 3.x στο Astro 5.x. Θα ρίξουμε μια βαθιά βουτιά στο πώς η Astro's Islands Architecture μας βοήθησε να λύσουμε τα σημεία συμφόρησης απόδοσης διατηρώντας παράλληλα τα υπάρχοντα στοιχεία React στοιχείων μας, παρέχοντας βελτιώσεις τόσο στην ταχύτητα κατασκευής όσο και στην απόδοση φόρτωσης.

## φόντο

Τον Ιανουάριο του 2026, πραγματοποιήσαμε μια "μεταμόσχευση καρδιάς" στον επίσημο ιστότοπο του HagiCode μεταφέροντας πλήρως το βασικό του πλαίσιο από το Docusaurus 3.x στο Astro 5.x. Αυτό δεν ήταν μια παρορμητική επανεγγραφή, αλλά μια προσεκτικά μελετημένη τεχνική απόφαση.

Πριν από τη μετεγκατάσταση, ο ιστότοπός μας ήταν λειτουργικά πλήρης, αλλά είχε αρχίσει να εμφανίζει κάποια κλασικά "προβλήματα πολυτέλειας": διογκωμένα τεχνουργήματα κατασκευής, υπερβολικά ωφέλιμα φορτία JavaScript και λιγότερο από ιδανική ταχύτητα φόρτωσης σελίδας σε πολύπλοκες σελίδες τεκμηρίωσης. Ως έργο βοηθού κωδικοποίησης AI, το HagiCode χρειάζεται συχνή τεκμηρίωση και ενημερώσεις χαρακτηριστικών, επομένως η απόδοση κατασκευής επηρεάζει άμεσα την ταχύτητα κυκλοφορίας. Ταυτόχρονα, θέλαμε ο ιστότοπος να είναι πιο φιλικός προς τις μηχανές αναζήτησης (SEO), ώστε περισσότεροι προγραμματιστές να μπορούν να ανακαλύψουν το έργο.

Για να λύσουμε αυτά τα σημεία πόνου, πήραμε μια τολμηρή απόφαση: να ξαναχτίσουμε ολόκληρο το σύστημα στο Astro. Ο αντίκτυπος αυτής της απόφασης μπορεί να είναι ακόμη μεγαλύτερος από ό,τι περιμένετε. Θα μπω σε λεπτομέρειες σύντομα.

## Σχετικά με το HagiCode

Η προσέγγιση μετεγκατάστασης ιστότοπου που κοινοποιείται σε αυτό το άρθρο προέρχεται από την πρακτική εμπειρία μας στο [HagiCode](https://github.com/HagiCode-org/site) έργο.

Το HagiCode είναι ένας βοηθός κωδικοποίησης AI που επικεντρώνεται στη βελτίωση της αποδοτικότητας ανάπτυξης. Μας ενδιαφέρει όχι μόνο η επανάληψη των βασικών δυνατοτήτων, αλλά και η εμπειρία προγραμματιστή. Αυτό το site refactor είχε επίσης σκοπό να προσφέρει στους χρήστες την ταχύτερη δυνατή εμπειρία κατά την περιήγηση στα έγγραφά μας και στον επίσημο ιστότοπο.

## Γιατί να εγκαταλείψουμε το ώριμο οικοσύστημα Docusaurus;

Εντός του οικοσυστήματος React, το Docusaurus είναι εδώ και καιρό η «τυπική απάντηση» για τοποθεσίες τεκμηρίωσης. Λειτουργεί ασυνήθιστα, προσφέρει ένα πλούσιο οικοσύστημα πρόσθετων και έχει μια ενεργή κοινότητα. Αλλά καθώς το HagiCode απέκτησε περισσότερες δυνατότητες, αισθανθήκαμε επίσης τους περιορισμούς του:

1.  **Συμφόρηση απόδοσης**: Το Docusaurus είναι ουσιαστικά μια React SPA (εφαρμογή μιας σελίδας). Ακόμα κι αν γράφετε μόνο στατικές σελίδες, ο πελάτης πρέπει να φορτώσει το χρόνο εκτέλεσης του React και να ενυδατώσει τη σελίδα, κάτι που είναι αδικαιολόγητα βαρύ για απλές σελίδες εγγράφων.
2.  **Μεγάλο μέγεθος στοιχείου**: Ακόμη και όταν μια σελίδα περιέχει πολύ λίγο περιεχόμενο, το ομαδοποιημένο μέγεθος JS παραμένει σχετικά σταθερό. Αυτό δεν είναι ιδανικό για χρήστες κινητών ή κακές συνθήκες δικτύου.
3.  **Περιορισμένη ευελιξία**: Αν και είναι επεκτάσιμο, θέλαμε περισσότερο έλεγχο χαμηλού επιπέδου στον αγωγό κατασκευής.

Το Astro έφτασε ακριβώς την κατάλληλη στιγμή για να λύσει αυτά τα προβλήματα. Εισήγαγε μια νέα "Islands Architecture": από προεπιλογή, το Astro δημιουργεί στατικό HTML με μηδενική JavaScript και μόνο τα στοιχεία που απαιτούν διαδραστικότητα "ενεργοποιούνται" και φορτώνουν JS. Αυτό σημαίνει ότι το μεγαλύτερο μέρος του ιστότοπού μας γίνεται καθαρό HTML και φορτώνεται εξαιρετικά γρήγορα.

## Core Migration Strategy: A Smooth Architectural Transition

Η μετανάστευση δεν ήταν απλώς αντιγραφή και επικόλληση. Απαιτούσε αλλαγή νοοτροπίας. Μεταφερθήκαμε από το μοντέλο "all React" του Docusaurus στο μοντέλο "Core + Islands" της Astro.

### 1. Ανακατασκευή του Συστήματος Διαμόρφωσης

Πρώτα, έπρεπε να φύγουμε από `docusaurus.config.ts` να `astro.config.mjs`. Αυτό δεν ήταν απλώς μια μετονομασία αρχείου, αλλά μια επανεγγραφή της λογικής δρομολόγησης και κατασκευής.

Στο Docusaurus, όλα είναι πρόσθετα. Στο Astro όλα είναι μια ενοποίηση. Χρειαζόταν να επαναπροσδιορίσουμε τη βασική διαδρομή του ιστότοπου, τη λειτουργία εξόδου δημιουργίας (στατική έναντι SSR) και τη στρατηγική βελτιστοποίησης στοιχείων.

**Πριν από τη μετανάστευση:**
```typescript
// docusaurus.config.ts
export default {
  title: 'HagiCode',
  url: 'https://hagicode.com',
  baseUrl: '/',
  // ... more configuration
};
```

**Μετά τη μετανάστευση:**
```javascript
// astro.config.mjs
import { defineConfig } from 'astro/config';
import react from '@astrojs/react';

export default defineConfig({
  integrations: [react()],
  site: 'https://hagicode.com',
  base: '/',
  // Optimization settings for static assets
  build: {
    inlineStylesheets: 'auto',
  },
});
```

### 2. Τι να κρατήσετε και τι να αναπαράγετε σε στοιχεία React

Αυτό ήταν το πιο οδυνηρό μέρος της μετανάστευσης. Ο υπάρχων ιστότοπός μας είχε πολλά στοιχεία React, όπως καρτέλες, επισήμανση κώδικα, κουμπιά σχολίων και πολλά άλλα. Η απόρριψή τους θα ήταν σπάταλη, αλλά η διατήρηση όλων θα έκανε το ωφέλιμο φορτίο JavaScript πολύ βαρύ.

Η HagiCode υιοθέτησε μια στρατηγική **προοδευτικής ενυδάτωσης**:
*   **Καθαρά στατικά στοιχεία**: Για περιεχόμενο παρουσίασης, όπως κεφαλίδες, υποσέλιδα και τεκμηρίωση απλού κειμένου, τα ξαναγράψαμε ως στοιχεία Astro (`.astro` αρχεία) και τα απέδωσε απευθείας σε HTML κατά το χρόνο κατασκευής.
*   **Διαδραστικά νησιά**: Για στοιχεία που πρέπει να παραμείνουν διαδραστικά, όπως εναλλάκτες θεμάτων, εναλλαγή καρτελών και κουμπιά αντιγραφής μπλοκ κώδικα, διατηρήσαμε την υλοποίηση του React και προσθέσαμε `client:load` ή `client:visible` οδηγίες.

Για παράδειγμα, το στοιχείο Tabs που χρησιμοποιείται συνήθως στην τεκμηρίωση:

```jsx
// src/components/Tabs.jsx
import { useState } from 'react';
import './Tabs.css'; // Import styles

export default function Tabs({ items }) {
  const [activeIndex, setActiveIndex] = useState(0);
  // ... state logic
  return (
    <div className="tabs-wrapper">
      {/* Rendering logic */}
    </div>
  );
}
```

Όταν χρησιμοποιείται στο Markdown, λέμε ρητά στην Astro: "Αυτό το στοιχείο χρειάζεται JS."

```markdown
// src/content/docs/example.mdx
import Tabs from '../../components/Tabs.jsx';

<!-- Load JS only when the component enters the viewport -->
<Tabs client:visible items={...} />
```

Με αυτόν τον τρόπο, τα διαδραστικά στοιχεία εκτός της θύρας προβολής δεν ανταγωνίζονται για το εύρος ζώνης, γεγονός που βελτιώνει σημαντικά την ταχύτητα φόρτωσης στην πρώτη οθόνη.

### 3. Προσαρμογή του συστήματος styling: Από μονάδες CSS σε Scoped CSS

Το Docusaurus υποστηρίζει λειτουργικές μονάδες CSS από προεπιλογή, ενώ το Astro ενθαρρύνει το Scoped CSS μέσω του `<style>` ετικέτα. Η βασική ιδέα πίσω από τα δύο είναι η απομόνωση στυλ, αλλά η σύνταξη είναι διαφορετική.

Κατά τη μετεγκατάσταση του HagiCode, μετατρέψαμε τις πιο σύνθετες μονάδες CSS σε στυλ εμβέλειας του Astro. Αυτό στην πραγματικότητα αποδείχτηκε καλό, γιατί μέσα `.astro` αρχεία τα στυλ και τα πρότυπα ζωντανά στο ίδιο αρχείο, γεγονός που καθιστά τη συντήρηση πιο διαισθητική.

**Πριν την ανακατασκευή:**
```css
/* Tabs.module.css */
.wrapper { background: var(--ifm-background-color); }
```

**Μετά την ανακατασκευή (Astro Scoped):**
```html
<!-- Tabs.astro -->
<div class="tabs-wrapper">
  <slot />
</div>

<style>
  .tabs-wrapper {
    /* Use CSS variables directly to adapt to the theme */
    background: var(--bg-color);
    padding: 1rem;
  }
</style>
```

Ταυτόχρονα, ενοποιήσαμε το παγκόσμιο σύστημα μεταβλητών CSS και χρησιμοποιήσαμε τις δυνατότητες της Astro με επίγνωση του περιβάλλοντος για να διασφαλίσουμε ότι η σκοτεινή λειτουργία εναλλάσσεται ομαλά στις σελίδες.

## Παγίδες που χτυπήσαμε στην πράξη και πώς τις λύσαμε

Κατά τη διάρκεια της πραγματικής μετεγκατάστασης του HagiCode, αντιμετωπίσαμε αρκετά προβλήματα. Εδώ είναι μερικά από τα πιο τυπικά.

### 1. Μεταβλητά σημεία πόνου διαδρομής και περιβάλλοντος

Το HagiCode υποστηρίζει την ανάπτυξη υποδιαδρομών, όπως την ανάπτυξη κάτω από έναν υποκατάλογο GitHub Pages. Στο Docusaurus, `baseUrl` χειρίζεται αυτόματα. Στο Astro, ωστόσο, πρέπει να είμαστε πιο προσεκτικοί όταν χειριζόμαστε συνδέσμους εικόνων και αιτήματα API.

Εισαγάγαμε έναν μηχανισμό μεταβλητής περιβάλλοντος για να το διαχειριζόμαστε με συνέπεια:

```javascript
// Handle paths in the build script
const getBasePath = () => import.meta.env.VITE_SITE_BASE || '/';
```

Βεβαιωθείτε ότι δεν έχετε κωδικοποιήσει μονοπάτια που ξεκινούν με `/` στον κωδικό σας. Στην ανάπτυξη έναντι της παραγωγής, ή μετά τη διαμόρφωση α `base` διαδρομή, κάτι τέτοιο μπορεί να προκαλέσει 404 για στοιχεία.

### 2. Συμβατότητα σεναρίου CommonJS

Ο παλιός μας ιστότοπος είχε ορισμένα σενάρια Node.js που χρησιμοποιήθηκαν για εργασίες όπως η αυτόματη ανάκτηση δεδομένων Metrics και η ενημέρωση του χάρτη ιστότοπου και ήταν γραμμένα σε CommonJS (`require`). Τα Astro και τα σύγχρονα εργαλεία κατασκευής έχουν αγκαλιάσει πλήρως τις μονάδες ES (`import`/`export`).

Εάν έχετε επίσης παρόμοια σενάρια, θυμηθείτε να τα επαναφέρετε όλα σε μονάδες ES. Αυτή είναι η κατεύθυνση που κινείται το οικοσύστημα και όσο πιο γρήγορα κάνετε την αλλαγή, τόσο λιγότερα προβλήματα θα έχετε αργότερα.

```javascript
// Old way
const fs = require('fs');

// New way
import fs from 'fs';
```

### 3. Μην ξεχνάτε το SEO και τις ανακατευθύνσεις

Οι μηχανές αναζήτησης έχουν ήδη ευρετηριάσει τις παλιές σελίδες Docusaurus του HagiCode. Εάν μεταβείτε απευθείας στο Astro και αλλάξει η δομή της διεύθυνσης URL, μπορεί να καταλήξετε με μεγάλο αριθμό 404 και σημαντική πτώση στην κατάταξη αναζήτησης.

Διαμορφώσαμε κανόνες ανακατεύθυνσης στο Astro:

```javascript
// astro.config.mjs
export default defineConfig({
  redirects: {
    '/docs/old-path': '/docs/new-path',
    // Map old links to new links in bulk
  }
});
```

Ή μπορείτε να το χειριστείτε αυτό στο επίπεδο διαμόρφωσης διακομιστή. Βεβαιωθείτε ότι οι παλιοί σύνδεσμοι μπορούν να ανακατευθυνθούν 301 στις νέες διευθύνσεις, επειδή αυτό είναι κρίσιμο για το SEO.

## Περίληψη

Για το HagiCode, η μετάβαση από το Docusaurus στο Astro δεν ήταν απλώς μια αναβάθμιση πλαισίου. Ήταν επίσης μια πρακτική εφαρμογή μιας φιλοσοφίας «performance first».

**Τι κερδίσαμε:**
*   **Εξαιρετικές βαθμολογίες Lighthouse**: Μετά τη μετεγκατάσταση, η βαθμολογία απόδοσης του ιστότοπου HagiCode πλησίασε εύκολα την τέλεια βαθμολογία.
*   **Ταχύτερη ταχύτητα κατασκευής**: Οι δυνατότητες σταδιακής κατασκευής του Astro μειώνουν στο μισό το χρόνο έκδοσης για ενημερώσεις τεκμηρίωσης.
*   **Διατηρημένη ευελιξία**: Με το Islands Architecture, δεν θυσιάσαμε καμία διαδραστική λειτουργία και μπορούσαμε να χρησιμοποιήσουμε το React όπου χρειαζόταν.

Εάν διατηρείτε επίσης έναν ιστότοπο προσανατολισμένο στην τεκμηρίωση και δυσκολεύεστε με το μέγεθος του πακέτου ή την ταχύτητα φόρτωσης, το Astro αξίζει να δοκιμάσετε. Παρόλο που η διαδικασία μετεγκατάστασης απαιτεί κάποια χειρουργική επέμβαση, όπως η μετονομασία του PCode σε HagiCode και η μετακίνηση εξαρτημάτων ένα προς ένα, η μεταξένια ομαλή εμπειρία χρήστη που λαμβάνετε σε αντάλλαγμα το καθιστά απολύτως χρήσιμο.

Το σύστημα κατασκευής που κοινοποιείται σε αυτό το άρθρο είναι η ακριβής προσέγγιση που αναπτύξαμε μέσω πραγματικών δοκιμών και σφαλμάτων κατά την κατασκευή του HagiCode. Εάν βρίσκετε αυτή την προσέγγιση πολύτιμη, αυτό λέει κάτι για τη μηχανική μας δύναμη, και το ίδιο το HagiCode ίσως αξίζει να το εξετάσετε προσεκτικότερα.
