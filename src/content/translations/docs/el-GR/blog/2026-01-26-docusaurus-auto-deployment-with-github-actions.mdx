## Προσθήκη υποστήριξης αυτόματης ανάπτυξης σελίδων GitHub στο HagiCode

> Αυτό το έργο είχε αρχικά την κωδική ονομασία PCode και τώρα έχει μετονομαστεί επίσημα σε HagiCode. Αυτό το άρθρο τεκμηριώνει τον τρόπο με τον οποίο εισαγάγαμε την αυτοματοποιημένη ανάπτυξη στατικών ιστότοπων, ώστε η δημοσίευση περιεχομένου να γίνεται τόσο εύκολη όσο το πόσιμο νερό.

## Ιστορικό / Εισαγωγή

Κατά τη διάρκεια της ανάπτυξης του HagiCode, αντιμετωπίσαμε ένα πολύ πρακτικό πρόβλημα: καθώς η τεκμηρίωση και οι προτάσεις μας συνέχιζαν να αυξάνονται, η αποτελεσματική διαχείριση και παρουσίαση αυτού του περιεχομένου έγινε όλο και πιο επείγουσα. Αποφασίσαμε να χρησιμοποιήσουμε τις Σελίδες GitHub για να φιλοξενήσουμε τον στατικό μας ιστότοπο, αλλά η δημιουργία και η μη αυτόματη ανάπτυξή του ήταν πολύ περίπλοκη. Κάθε αλλαγή απαιτούσε τοπική κατασκευή, συσκευασία και στη συνέχεια χειροκίνητη ώθηση στο `gh-pages` υποκατάστημα. Αυτό δεν ήταν μόνο αναποτελεσματικό, αλλά και εύκολο να γίνει λάθος.

Για να λύσουμε αυτό το πρόβλημα, κυρίως επειδή θέλαμε να εξοικονομήσουμε κάποια προσπάθεια, χρειαζόμασταν μια αυτοματοποιημένη ροή εργασιών ανάπτυξης. Αυτό το άρθρο καταγράφει λεπτομερώς πώς προσθέσαμε υποστήριξη αυτόματης ανάπτυξης βάσει GitHub Actions στο έργο HagiCode, ώστε να μπορούμε να παραμείνουμε συγκεντρωμένοι στη σύνταξη περιεχομένου και να αφήσουμε τον αυτοματισμό να χειριστεί τα υπόλοιπα.

## Ανάλυση Στόχων

Πριν προχωρήσουμε στην υλοποίηση, πρέπει πρώτα να διευκρινίσουμε τι υποτίθεται ότι θα επιτύχει πραγματικά αυτή η εργασία. Άλλωστε, το ακόνισμα του τσεκούρι δεν καθυστερεί τον τεμαχισμό των καυσόξυλων.

### Βασικές Απαιτήσεις

1.  **Αυτόματη κατασκευή**: Ενεργοποιεί αυτόματα τη ροή εργασιών κατασκευής όταν ο κώδικας προωθείται στο `main` υποκατάστημα.
2.  **Αυτόματη ανάπτυξη**: Μετά από μια επιτυχημένη κατασκευή, αναπτύξτε αυτόματα τα δημιουργημένα στατικά αρχεία στις Σελίδες GitHub.
3.  **Περιβαλλοντική συνέπεια**: Βεβαιωθείτε ότι το περιβάλλον CI ταιριάζει με το τοπικό περιβάλλον κατασκευής για να αποφύγετε την κλασική αμηχανία ότι "λειτουργεί τοπικά, αλλά αποτυγχάνει στο διαδίκτυο".

### Τεχνική Επιλογή

Δεδομένου ότι το HagiCode είναι χτισμένο στο Docusaurus, μια πολύ δημοφιλής δημιουργία στατικών τοποθεσιών React, μπορούμε να χρησιμοποιήσουμε το GitHub Actions για να πετύχουμε αυτόν τον στόχο.

## Διαμόρφωση της ροής εργασιών ενεργειών GitHub

Το GitHub Actions είναι η υπηρεσία CI/CD που παρέχεται από το GitHub. Ορίζοντας αρχεία ροής εργασίας σε μορφή YAML μέσα στο αποθετήριο, μπορούμε να προσαρμόσουμε όλα τα είδη εργασιών αυτοματισμού.

### Δημιουργία του αρχείου ροής εργασίας

Πρέπει να δημιουργήσουμε ένα νέο αρχείο ρυθμίσεων στο `.github/workflows` κατάλογο στη ρίζα του έργου, για παράδειγμα `deploy.yml`. Εάν ο κατάλογος δεν υπάρχει ακόμα, δημιουργήστε τον πρώτα.

Η βασική λογική αυτού του αρχείου διαμόρφωσης είναι η εξής:

1.  **Συνθήκη ενεργοποίησης**: Ακούστε για `push` εκδηλώσεις για την `main` υποκατάστημα.
2.  **Περιβάλλον χρόνου εκτέλεσης**: Η τελευταία έκδοση του Ubuntu.
3.  **Βήματα κατασκευής**:
    -   Ελέγξτε τον κωδικό
    -   Εγκαταστήστε το Node.js
    -   Εγκατάσταση εξαρτήσεων (`npm install`)
    -   Δημιουργήστε τα στατικά αρχεία (`npm run build`)
4.  **Βήμα ανάπτυξης**: Χρησιμοποιήστε το επίσημο `action-gh-pages` δράση για να ωθήσει την έξοδο κατασκευής στο `gh-pages` υποκατάστημα.

### Κωδικός διαμόρφωσης κλειδιού

Παρακάτω είναι το πρότυπο διαμόρφωσης που υιοθετήσαμε τελικά:

```yaml
name: Deploy to GitHub Pages

# Trigger condition: run when code is pushed to the main branch
on:
  push:
    branches:
      - main
    # Add path filters if needed, for example only build when docs change
    # paths:
    #   - 'docs/**'
    #   - 'package.json'

# Set permissions, which is important for GitHub Pages deployment
permissions:
  contents: read
  pages: write
  id-token: write

# Concurrency control: cancel older builds for the same branch
concurrency:
  group: "pages"
  cancel-in-progress: false

jobs:
  build:
    runs-on: ubuntu-latest
    steps:
      - name: Checkout
        uses: actions/checkout@v4
        # Note: fetch-depth: 0 is required, otherwise the build version may be inaccurate
        with:
          fetch-depth: 0

      - name: Setup Node
        uses: actions/setup-node@v4
        with:
          node-version: 20 # Keep this consistent with the local development environment
          cache: 'npm'     # Enabling cache can speed up the build process

      - name: Install dependencies
        run: npm ci
        # Use npm ci instead of npm install because it is faster and stricter, which makes it better for CI

      - name: Build website
        run: npm run build
        env:
          # Configure environment variables here if your site build requires them
          # NODE_ENV: production
          # PUBLIC_URL: /your-repo-name
          
      - name: Upload artifact
        uses: actions/upload-pages-artifact@v3
        with:
          path: ./build # Default Docusaurus output directory

  deploy:
    environment:
      name: github-pages
      url: ${{ steps.deployment.outputs.page_url }}
    runs-on: ubuntu-latest
    needs: build
    steps:
      - name: Deploy to GitHub Pages
        id: deployment
        uses: actions/deploy-pages@v4
```

## Συνήθεις παγίδες κατά την εφαρμογή

Στην πράξη αντιμετωπίσαμε πολλά ζητήματα. Τα μοιράζομαι εδώ με την ελπίδα ότι οι άλλοι μπορούν να τα αποφύγουν ή τουλάχιστον να προετοιμάσουν λύσεις εκ των προτέρων.

### 1. Ζητήματα άδειας GitHub Token

Όταν ρυθμίσαμε για πρώτη φορά τα πράγματα, η ανάπτυξη συνέχιζε να αποτυγχάνει με ένα 403 (Απαγορευμένο). Μετά από πολύ ψάξιμο, διαπιστώσαμε ότι η προεπιλογή του GitHub `GITHUB_TOKEN` δεν είχε άδεια να γράψει στο Pages.

**Λύση**: Στο αποθετήριο `Settings` -> `Actions` -> `General` -> `Workflow permissions`, φροντίστε να επιλέξετε **"Δικαιώματα ανάγνωσης και εγγραφής"**.

### 2. Λανθασμένη διαδρομή καταλόγου κατασκευής

Το Docusaurus τοποθετεί ενσωματωμένα στατικά αρχεία στο `build` κατάλογο από προεπιλογή. Ωστόσο, ορισμένα έργα ενδέχεται να διαμορφωθούν διαφορετικά. Για παράδειγμα, το Create React App έχει οριστεί ως προεπιλογή `build`, ενώ το Vite έχει οριστεί ως προεπιλογή `dist`. Εάν το Actions αναφέρει ότι δεν μπορεί να βρει τα αρχεία εξόδου, ελέγξτε τη διαδρομή εξόδου `docusaurus.config.js`.

### 3. Θέματα υποδιαδρομής

Εάν το αποθετήριο σας δεν είναι μια αρχική σελίδα χρήστη, σημαίνει ότι δεν είναι `username.github.io`, αλλά αντίθετα μια σελίδα έργου όπως `username.github.io/project-name`, τότε πρέπει να ρυθμίσετε τις παραμέτρους `baseUrl`.

Σε `docusaurus.config.js`:

```javascript
module.exports = {
  // ...
  url: 'https://hagicode.com', // Your Hagicode URL
  baseUrl: '/', // Deploy from the root path
  // ...
};
```
Αυτή η λεπτομέρεια είναι εύκολο να παραβλεφθεί. Εάν έχει ρυθμιστεί εσφαλμένα, η σελίδα μπορεί να φορτωθεί ως κενή οθόνη επειδή δεν είναι δυνατή η επίλυση των διαδρομών στοιχείων.

## Επαλήθευση του αποτελέσματος

Αφού ρυθμίσουμε τα πάντα και πατήσουμε τον κώδικα, μπορούμε να κατευθυνθούμε στην καρτέλα **Ενέργειες** του αποθετηρίου GitHub και να παρακολουθήσουμε την εκπομπή.

Θα δείτε έναν κίτρινο κύκλο ενώ εκτελείται η ροή εργασίας και μόλις γίνει πράσινος, σημαίνει επιτυχία. Εάν γίνει κόκκινο, κάντε κλικ και επιθεωρήστε τα αρχεία καταγραφής. Στις περισσότερες περιπτώσεις, μπορείτε να παρακολουθήσετε το πρόβλημα εκεί κάτω, συνήθως ένα τυπογραφικό λάθος ή μια εσφαλμένη διαμόρφωση διαδρομής.

Μόλις η κατασκευή πετύχει, επισκεφθείτε `https://<your-username>.github.io/<repository-name>/` και θα δείτε τον πρόσφατα αναπτυγμένο ιστότοπο.

## Περίληψη

Με την εισαγωγή του GitHub Actions, αυτοματοποιήσαμε με επιτυχία την ανάπτυξη για τον ιστότοπο τεκμηρίωσης HagiCode. Αυτό όχι μόνο εξοικονομεί τον χρόνο που δαπανήθηκε προηγουμένως για χειροκίνητες λειτουργίες, αλλά το πιο σημαντικό τυποποιεί τη ροή εργασιών απελευθέρωσης. Τώρα, ανεξάρτητα από το ποιο μέλος της ομάδας ενημερώνει την τεκμηρίωση, μόλις συγχωνευθεί η αλλαγή `main`, το πιο πρόσφατο περιεχόμενο θα είναι ζωντανό λίγα λεπτά αργότερα.

**Βασικά οφέλη**:
- **Υψηλότερη απόδοση**: Μετακινηθήκαμε από τη "μη αυτόματη συσκευασία και τη μη αυτόματη μεταφόρτωση" στο "κώδικας ισούται με απελευθέρωση".
- **Λιγότερα λάθη**: Καταργήσαμε την πιθανότητα ανθρώπινου λάθους κατά τη διάρκεια χειροκίνητων χειρισμών.
- **Καλύτερη εμπειρία προγραμματιστή**: Οι προγραμματιστές μπορούν να επικεντρωθούν περισσότερο στην ποιότητα περιεχομένου αντί να αποσπώνται από κουραστικά βήματα ανάπτυξης.

Αν και η διαμόρφωση του CI/CD είναι λίγο ενοχλητική στην αρχή, ειδικά με όλα τα δικαιώματα και τις λεπτομέρειες διαδρομής, είναι μια εφάπαξ επένδυση με σημαντικές μακροπρόθεσμες αποδόσεις. Συνιστώ ανεπιφύλακτα κάθε έργο στατικής τοποθεσίας να υιοθετεί μια παρόμοια αυτοματοποιημένη ροή εργασίας.
