## Running AI CLI Tools in Docker Containers: A Practical Guide to User Isolation and Persistent Volumes

> Η ενσωμάτωση εργαλείων κωδικοποίησης AI όπως ο Claude Code, ο Codex και το OpenCode σε περιβάλλοντα με κοντέινερ ακούγεται απλή, αλλά υπάρχουν κρυμμένες πολυπλοκότητες παντού. Αυτό το άρθρο εξετάζει τον τρόπο με τον οποίο το έργο HagiCode επιλύει βασικές προκλήσεις στις αναπτύξεις Docker, συμπεριλαμβανομένων των δικαιωμάτων χρήστη, της επιμονής της διαμόρφωσης και της διαχείρισης εκδόσεων, ώστε να μπορείτε να αποφύγετε τις κοινές παγίδες.

## φόντο

Όταν αποφασίσαμε να τρέξουμε εργαλεία CLI κωδικοποίησης τεχνητής νοημοσύνης μέσα σε κοντέινερ Docker, η πιο διαισθητική σκέψη ήταν πιθανώς: "Τα κοντέινερ δεν είναι απλώς root; Γιατί να μην εγκαταστήσετε τα πάντα απευθείας και να τα ονομάσετε ολοκληρωμένα;" Στην πραγματικότητα, αυτή η φαινομενικά απλή ιδέα κρύβει πολλά βασικά προβλήματα που πρέπει να επιλυθούν.

Πρώτον, **οι περιορισμοί ασφαλείας είναι το πρώτο εμπόδιο**. Πάρτε το Claude CLI ως παράδειγμα: απαγορεύει ρητά την εκτέλεση ως χρήστης root. Αυτός είναι ένας υποχρεωτικός έλεγχος ασφαλείας και εάν εντοπιστεί root, αρνείται να ξεκινήσει. Ίσως σκεφτείτε, δεν μπορώ απλώς να αλλάξω χρήστες με το `USER` οδηγία; Δεν είναι τόσο απλό. Υπάρχει ακόμα ένα πρόβλημα αντιστοίχισης μεταξύ του μη ριζικού χρήστη μέσα στο κοντέινερ και των δικαιωμάτων χρήστη στον κεντρικό υπολογιστή.

Δεύτερον, **η επιμονή της κατάστασης είναι η δεύτερη παγίδα**. Ο Claude Code απαιτεί σύνδεση, το Codex έχει τη δική του διαμόρφωση και το OpenCode έχει επίσης έναν κατάλογο προσωρινής μνήμης. Εάν πρέπει να ρυθμίζετε εκ νέου τα πάντα κάθε φορά που το κοντέινερ επανεκκινείται, η όλη ιδέα του "αυτοματισμού" χάνει το νόημά της. Χρειαζόμαστε αυτές τις διαμορφώσεις για να διατηρηθούν πέρα ​​από τον κύκλο ζωής του κοντέινερ.

Το τρίτο πρόβλημα είναι η **συνέπεια άδειας**. Μπορούν οι διεργασίες μέσα στο κοντέινερ να έχουν πρόσβαση στα αρχεία διαμόρφωσης που έχουν δημιουργηθεί από τον κεντρικό χρήστη; Οι αναντιστοιχίες UID/GID προκαλούν συχνά σφάλματα δικαιωμάτων αρχείων και αυτό είναι εξαιρετικά συνηθισμένο σε πραγματικές αναπτύξεις.

Αυτά τα προβλήματα μπορεί να φαίνονται ανεξάρτητα, αλλά στην πράξη είναι στενά συνδεδεμένα. Κατά τη διάρκεια της ανάπτυξης του HagiCode, επεξεργαστήκαμε σταδιακά μια πρακτική λύση. Στη συνέχεια, θα μοιραστώ τις τεχνικές λεπτομέρειες και τα διδάγματα που αντλήθηκαν από αυτές τις παγίδες.

## Γιατί δεν μπορούμε να χρησιμοποιήσουμε απλώς root;

Υπάρχει μια κοινή παρεξήγηση εδώ: Τα κοντέινερ Docker εκτελούνται ως root από προεπιλογή, οπότε γιατί να μην εγκαταστήσετε απλώς τα εργαλεία ως root; Εάν σκέφτεστε έτσι, ο Claude CLI θα σας διδάξει γρήγορα το αντίθετο.

```bash
# Run Claude CLI directly as root? No.
docker run --rm -it --user root myimage claude
# Output: Error: This command cannot be run as root user
```

Αυτός είναι ένας αυστηρός περιορισμός ασφαλείας στο Claude CLI. Ο λόγος είναι απλός: αυτά τα εργαλεία CLI διαβάζουν και γράφουν ευαίσθητες ρυθμίσεις παραμέτρων χρήστη, συμπεριλαμβανομένων διακριτικών API, τοπικές κρυφές μνήμες, ακόμη και σενάρια γραμμένα από τον χρήστη. Η εκτέλεσή τους με δικαιώματα root δημιουργεί υπερβολικό κίνδυνο.

Επομένως, το ερώτημα είναι: πώς μπορούμε να ικανοποιήσουμε τις απαιτήσεις ασφαλείας του CLI διατηρώντας παράλληλα ευέλικτη τη διαχείριση εμπορευματοκιβωτίων; Πρέπει να αλλάξουμε τον τρόπο που το σκεφτόμαστε: **αντί να αλλάζουμε χρήστες κατά το χρόνο εκτέλεσης, δημιουργήστε έναν αποκλειστικό χρήστη κατά το στάδιο δημιουργίας εικόνας**.

## Δημιουργία αποκλειστικού χρήστη: κάτι περισσότερο από την απλή αλλαγή ονόματος

Ίσως σκεφτείτε ότι προσθέτοντας ένα σινγκλ `USER` αρκεί η γραμμή στο Dockerfile. Αυτή είναι πράγματι η απλούστερη προσέγγιση, αλλά δεν είναι αρκετά ισχυρή.

### Στατική δημιουργία έναντι δυναμικής χαρτογράφησης

Η προσέγγιση του HagiCode είναι να δημιουργήσει ένα `hagicode` χρήστης με UID 1000, το οποίο συνήθως ταιριάζει με τον προεπιλεγμένο χρήστη στους περισσότερους κεντρικούς υπολογιστές:

```dockerfile
RUN groupadd -o -g 1000 hagicode && \
    useradd -o -u 1000 -g 1000 -s /bin/bash -m hagicode && \
    mkdir -p /home/hagicode/.claude && \
    chown -R hagicode:hagicode /home/hagicode
```

Αλλά αυτό λύνει μόνο τον ενσωματωμένο χρήστη μέσα στην εικόνα. Τι γίνεται αν ο κεντρικός χρήστης είναι UID 1001; Πρέπει ακόμα να υποστηρίζετε τη δυναμική αντιστοίχιση όταν ξεκινά το κοντέινερ.

Το docker-entrypoint.sh περιέχει τη βασική λογική:

```bash
if [ -n "$PUID" ] && [ -n "$PGID" ]; then
    if ! id hagicode >/dev/null 2>&1; then
        groupadd -g "$PGID" hagicode
        useradd -u "$PUID" -g "$PGID" -s /bin/bash -m hagicode
    fi
fi
```

Το πλεονέκτημα αυτού του σχεδιασμού είναι σαφές: **χρησιμοποιήστε το προεπιλεγμένο UID 1000 κατά το χρόνο δημιουργίας της εικόνας και, στη συνέχεια, προσαρμόστε δυναμικά κατά το χρόνο εκτέλεσης μέσω του `PUID` και `PGID` μεταβλητές περιβάλλοντος**. Ανεξάρτητα από το UID που έχει ο οικοδεσπότης, η ιδιοκτησία των αρχείων διαμόρφωσης παραμένει σωστή.

## Η σχεδιαστική φιλοσοφία των επίμονων όγκων

Κάθε εργαλείο AI CLI έχει τον δικό του κατάλογο προτιμώμενων ρυθμίσεων, επομένως πρέπει να αντιστοιχιστούν ένα προς ένα:

| Εργαλείο CLI | Διαδρομή στο κοντέινερ | Ονομασμένος τόμος |
|---------|-----------|--------|
| Ο Κλοντ | `/home/hagicode/.claude` | `claude-data` |
| Codex | `/home/hagicode/.codex` | `codex-data` |
| OpenCode | `/home/hagicode/.config/opencode` | `opencode-config-data` |

Γιατί να χρησιμοποιήσετε επώνυμους τόμους αντί για βάσεις σύνδεσης; Τρεις λόγοι:

1. **Απλότερη διαχείριση**: Οι επώνυμοι τόμοι διαχειρίζονται αυτόματα από το Docker, επομένως δεν χρειάζεται να δημιουργείτε καταλόγους κεντρικού υπολογιστή με μη αυτόματο τρόπο.
2. **Απομόνωση άδειας**: Τα αρχικά περιεχόμενα των τόμων δημιουργούνται από τον χρήστη μέσα στο κοντέινερ, αποφεύγοντας τις διενέξεις αδειών με τον κεντρικό υπολογιστή.
3. **Ανεξάρτητη μετεγκατάσταση**: Οι τόμοι μπορούν να υπάρχουν ανεξάρτητα από τα κοντέινερ, επομένως τα δεδομένα δεν χάνονται όταν αναβαθμίζονται οι εικόνες.

Το docker-compose-builder-web δημιουργεί αυτόματα την αντίστοιχη διαμόρφωση τόμου:

```yaml
volumes:
  claude-data:
  codex-data:
  opencode-config-data:

services:
  hagicode:
    volumes:
      - claude-data:/home/hagicode/.claude
      - codex-data:/home/hagicode/.codex
      - opencode-config-data:/home/hagicode/.config/opencode
    user: "${PUID:-1000}:${PGID:-1000}"
```

Δώστε προσοχή στο `user` πεδίο εδώ: `PUID` και `PGID` εγχέονται μέσω μεταβλητών περιβάλλοντος για να διασφαλιστεί ότι οι διεργασίες μέσα στο κοντέινερ εκτελούνται με ταυτότητα που ταιριάζει με τον κεντρικό χρήστη. Αυτή η λεπτομέρεια έχει σημασία επειδή τα ζητήματα αδειών είναι επώδυνο να εντοπιστούν σφάλματα μόλις εμφανιστούν.

## Διαχείριση εκδόσεων: ψημένες εκδόσεις με παρακάμψεις χρόνου εκτέλεσης

Το καρφίτσωμα των εκδόσεων εικόνων Docker είναι απαραίτητο για την αναπαραγωγιμότητα. Αλλά στην πραγματική ανάπτυξη, συχνά χρειάζεται να δοκιμάσουμε μια νεότερη έκδοση ή να διορθώσουμε επειγόντως ένα σφάλμα. Αν έπρεπε να ξαναφτιάχνουμε την εικόνα κάθε φορά, η ροή εργασίας θα ήταν πολύ αναποτελεσματική.

Η στρατηγική του HagiCode είναι **σταθερές εκδόσεις ως προεπιλογή, με παρακάμψεις χρόνου εκτέλεσης ως μηχανισμό επέκτασης**. Είναι ένας ρεαλιστικός μηχανικός συμβιβασμός μεταξύ σταθερότητας και ευελιξίας.

Dockerfile.template εκδόσεις pins εδώ:

```dockerfile
USER hagicode
WORKDIR /home/hagicode

# Configure the global npm install path
RUN mkdir -p /home/hagicode/.npm-global && \
    npm config set prefix '/home/hagicode/.npm-global'

# Install CLI tools using pinned versions
RUN npm install -g @anthropic-ai/claude-code@2.1.71 && \
    npm install -g @openai/codex@0.112.0 && \
    npm install -g opencode-ai@1.2.25 && \
    npm cache clean --force
```

Το docker-entrypoint.sh υποστηρίζει παρακάμψεις χρόνου εκτέλεσης:

```bash
install_cli_override_if_needed() {
    local package_name="$2"
    local override_version="$5"

    if [ -n "$override_version" ]; then
        gosu hagicode npm install -g "${package_name}@${override_version}"
    fi
}

# Example usage
install_cli_override_if_needed "" "@anthropic-ai/claude-code" "" "" "${CLAUDE_CODE_CLI_VERSION}"
```

Αυτό σας επιτρέπει να δοκιμάσετε μια νέα έκδοση μέσω μιας μεταβλητής περιβάλλοντος χωρίς να δημιουργήσετε ξανά την εικόνα:

```bash
docker run -e CLAUDE_CODE_CLI_VERSION=2.2.0 myimage
```

Αυτός ο σχεδιασμός είναι πρακτικός γιατί κανείς δεν θέλει να ξαναφτιάχνει μια εικόνα κάθε φορά που δοκιμάζει ένα νέο χαρακτηριστικό.

## Αυτόματη έγχυση διαμόρφωσης

Εκτός από τη μη αυτόματη ρύθμιση παραμέτρων των εργαλείων CLI, ορισμένα σενάρια απαιτούν αυτόματη έγχυση διαμόρφωσης. Το πιο χαρακτηριστικό παράδειγμα είναι ένα διακριτικό API.

```bash
if [ -n "$ANTHROPIC_AUTH_TOKEN" ]; then
    mkdir -p /home/hagicode/.claude
    cat > /home/hagicode/.claude/settings.json <<EOF
{
  "env": {
    "ANTHROPIC_AUTH_TOKEN": "${ANTHROPIC_AUTH_TOKEN}"
  }
}
EOF
    chown -R hagicode:hagicode /home/hagicode/.claude
fi
```

Δύο πράγματα έχουν σημασία εδώ: **διαβιβάστε ευαίσθητες πληροφορίες μέσω μεταβλητών περιβάλλοντος αντί να τις κωδικοποιήσετε σκληρά στην εικόνα** και βεβαιωθείτε ότι η ιδιοκτησία των αρχείων διαμόρφωσης έχει ρυθμιστεί σωστά, διαφορετικά τα εργαλεία CLI δεν θα μπορούν να τα διαβάσουν.

## Βέλτιστες πρακτικές και λίστα ελέγχου παγίδων

### Προβλήματα αναντιστοιχίας αδειών

Αυτή είναι η πιο εύκολη παγίδα για να πέσεις. Ο κεντρικός χρήστης έχει UID 1001, ενώ το κοντέινερ χρησιμοποιεί 1000, επομένως δεν είναι δυνατή η πρόσβαση στα αρχεία που δημιουργούνται στη μία πλευρά από την άλλη.

```bash
# Correct approach: make the container match the host user
docker run \
    -e PUID=$(id -u) \
    -e PGID=$(id -g) \
    myimage
```

Αυτό το ζήτημα είναι πολύ κοινό και μπορεί να είναι απογοητευτικό την πρώτη φορά που το αντιμετωπίζετε.

### Η διαμόρφωση εξαφανίζεται μετά την επανεκκίνηση του κοντέινερ

Εάν διαπιστώσετε ότι συνδέεστε ξανά μετά από κάθε επανεκκίνηση, ελέγξτε αν ξεχάσατε να προσαρτήσετε έναν μόνιμο τόμο:

```yaml
volumes:
  - claude-data:/home/hagicode/.claude
```

Τίποτα δεν είναι πιο απογοητευτικό από το να ρυθμίσετε προσεκτικά μια διαμόρφωση μόνο για να την δείτε να εξαφανίζεται.

### Ο σωστός τρόπος αναβάθμισης εκδόσεων

Μην τρέχετε `npm install -g` απευθείας μέσα σε ένα τρεχούμενο δοχείο. Οι σωστές προσεγγίσεις είναι:

1. Ορίστε μια μεταβλητή περιβάλλοντος για να ενεργοποιήσει την εγκατάσταση παράκαμψης.
2. Ή ξαναχτίστε την εικόνα.

```bash
# Option 1: runtime override
docker run -e CLAUDE_CODE_CLI_VERSION=2.2.0 myimage

# Option 2: rebuild the image
docker build -t myimage:v2 .
```

Υπάρχουν περισσότεροι από ένας δρόμοι προς τη Ρώμη, αλλά μερικοί δρόμοι είναι πιο ομαλοί από άλλους.

### Λίστα ελέγχου σκλήρυνσης ασφαλείας

- Περάστε διακριτικά API μέσω μεταβλητών περιβάλλοντος αντί να τα γράψετε στην εικόνα.
- Ορίστε τα δικαιώματα του αρχείου διαμόρφωσης σε `600`.
- Πάντοτε να εκτελείτε την εφαρμογή ως χρήστης χωρίς root.
- Ενημερώνετε τακτικά τις εκδόσεις CLI για να διορθώνετε ευπάθειες ασφαλείας.

Η ασφάλεια είναι πάντα σημαντική, αλλά η πραγματική πρόκληση είναι η συνεχής επιβολή της στην πράξη.

## Συμπέρασμα

Η εκτέλεση εργαλείων AI CLI σε κοντέινερ Docker περιλαμβάνει τρεις βασικές προκλήσεις: **δικαιώματα χρήστη, επιμονή διαμόρφωσης και διαχείριση έκδοσης**. Συνδυάζοντας αποκλειστικούς χρήστες, απομόνωση επώνυμου τόμου και παρακάμψεις με βάση τις μεταβλητές περιβάλλοντος, το έργο HagiCode δημιούργησε μια αρχιτεκτονική ανάπτυξης που είναι ασφαλής και ευέλικτη.

Βασικά σημεία σχεδιασμού:

- **Απομόνωση χρήστη**: Δημιουργήστε έναν αποκλειστικό χρήστη κατά τη διάρκεια του σταδίου δημιουργίας εικόνας, με υποστήριξη χρόνου εκτέλεσης για δυναμική `PUID`/`PGID` χαρτογράφηση.
- **Στρατηγική εμμονής**: Κάθε εργαλείο CLI λαμβάνει τον δικό του επώνυμο τόμο, επομένως οι επανεκκινήσεις δεν επηρεάζουν τη διαμόρφωση.
- **Ευελιξία έκδοσης**: Οι σταθερές προεπιλογές εξασφαλίζουν αναπαραγωγιμότητα, ενώ οι παρακάμψεις χρόνου εκτέλεσης παρέχουν χώρο για δοκιμές.
- **Αυτοματοποιημένη διαμόρφωση**: Η ευαίσθητη διαμόρφωση μπορεί να εγχυθεί αυτόματα μέσω μεταβλητών περιβάλλοντος.
