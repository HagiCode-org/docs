# Εισαγωγή στο OpenSpec

Το OpenSpec είναι ένα εργαλείο ανάπτυξης που καθοδηγείται από προδιαγραφές και βοηθά τις ομάδες να διαχειρίζονται τη διαδικασία ανάπτυξης λογισμικού μέσω δομημένων προτάσεων και προδιαγραφών. Αυτό το άρθρο παρέχει μια ολοκληρωμένη εισαγωγή στις βασικές έννοιες, τη ροή εργασίας και τον τρόπο χρήσης του OpenSpec.

## Τι είναι το OpenSpec;

Το OpenSpec είναι ένα εργαλείο γραμμής εντολών που χρησιμοποιείται για τη διαχείριση προτάσεων (Proposals), προδιαγραφών (Specifications) και αλλαγών (Changes) στη διαδικασία ανάπτυξης λογισμικού. Τυποποιεί τη διαδικασία ανάπτυξης και διασφαλίζει ότι κάθε λειτουργία περνά από λεπτομερή σχεδίαση και ανασκόπηση από την ιδέα μέχρι την υλοποίηση.

### Θεμελιώδεις αρχές

Το OpenSpec βασίζεται στις ακόλουθες θεμελιώδεις αρχές:

- **Προδιαγραφή πρώτα**: Ορίστε σαφείς προδιαγραφές πριν γράψετε κώδικα
- **Δομημένη συνεργασία**: Προωθήστε τη συνεργασία της ομάδας μέσω τυποποιημένων μορφών εγγράφων
- **Ιχνηλασιμότητα**: Καταγράψτε ολόκληρη τη διαδικασία από την πρόταση έως την υλοποίηση
- **Διασφάλιση ποιότητας**: Εξασφαλίστε ότι ο κώδικας συμμορφώνεται με το σχεδιαστικό σκοπό μέσω της επικύρωσης των προδιαγραφών

### Κύρια χαρακτηριστικά

Το OpenSpec προσφέρει τα ακόλουθα κύρια χαρακτηριστικά:

1. **Διαχείριση προτάσεων**: Δημιουργήστε, αναθεωρήστε και παρακολουθήστε προτάσεις λειτουργιών
2. **Ορισμός προδιαγραφών**: Γράψτε και διατηρήστε έγγραφα τεχνικών προδιαγραφών
3. **Παρακολούθηση αλλαγών**: Καταγράψτε όλο το ιστορικό των αναθεωρημένων αλλαγών
4. **Επικύρωση προδιαγραφών**: Εξασφαλίστε ότι οι αλλαγές κώδικα συμμορφώνονται με τις προκαθορισμένες προδιαγραφές
5. **Συνεργασία ομάδας**: Υποστηρίξτε διαδικασίες ανάπτυξης και αναθεώρησης με πολλούς συμμετέχοντες

## Ροή εργασίας OpenSpec

Η τυπική ροή εργασίας του OpenSpec περιλαμβάνει τα εξής στάδια:

### 1. Δημιουργία πρότασης (Proposal)

Όταν υπάρχει νέα λειτουργία ή απαίτηση για σημαντική αλλαγή, δημιουργήστε πρώτα μια πρόταση:

```bash
openspec proposal create
```

Η πρόταση περιλαμβάνει:
- **Περιγραφή του προβλήματος**: Το πρόβλημα ή η απαίτηση που πρέπει να λυθεί
- **Λύση**: Επισκόπηση της προτεινόμενης λύσης
- **Πεδίο επιρροής**: Η επίδραση στο υπάρχον σύστημα
- **Εναλλακτικές λύσεις**: Άλλες εξεταζόμενες λύσεις

### 2. Σύνταξη προδιαγραφής (Specification)

Μετά την έγκριση της πρότασης, γράψτε λεπτομερή τεχνική προδιαγραφή:

```bash
openspec spec create
```

Η προδιαγραφή περιλαμβάνει:
- **Τεχνικός σχεδιασμός**: Λεπτομερές σχέδιο τεχνικής υλοποίησης
- **Ορισμός διεπαφών**: Σχεδιασμός διεπαφών όπως API, δομές δεδομένων κ.λπ.
- **Λεπτομέρειες υλοποίησης**: Κύρια λογική υλοποίησης και αλγόριθμοι
- **Σχέδιο δοκιμών**: Πώς να επαληθεύσετε την ορθότητα της υλοποίησης

### 3. Δημιουργία αλλαγής (Change)

Όταν ξεκινήσετε την υλοποίηση, δημιουργήστε μια αλλαγή για να παρακολουθήσετε τη διαδικασία ανάπτυξης:

```bash
openspec change create
```

Η αλλαγή περιλαμβάνει:
- **Σχετική προδιαγραφή**: Η προδιαγραφή που υλοποιείται από αυτήν την αλλαγή
- **Εργασίες υλοποίησης**: Λίστα συγκεκριμένων εργασιών ανάπτυξης
- **Παρακολούθηση προόδου**: Κατάσταση ολοκλήρωσης των εργασιών
- **Αλλαγές κώδικα**: Οι πραγματικές τροποποιήσεις του κώδικα

### 4. Επικύρωση και αρχειοθέτηση

Μετά την ολοκλήρωση της ανάπτυξης, επικυρώστε εάν η αλλαγή συμμορφώνεται με την προδιαγραφή και, στη συνέχεια, αρχειοθετήστε την:

```bash
openspec change validate
openspec change archive
```

## Δομή καταλόγων OpenSpec

Το OpenSpec δημιουργεί έναν κατάλογο `openspec/` στο ριζικό κατάλογο του έργου με την ακόλουθη δομή:

```
openspec/
├── proposals/           # Κατάλογος προτάσεων
│   ├── PROPOSAL-001.md
│   └── PROPOSAL-002.md
├── specs/              # Κατάλογος προδιαγραφών
│   ├── SPEC-001.md
│   └── SPEC-002.md
├── changes/            # Κατάλογος αλλαγών
│   ├── feature-name/
│   │   ├── proposal.md
│   │   ├── design.md
│   │   ├── specs.md
│   │   └── tasks.md
│   └── another-feature/
└── archived/           # Αρχειοθετημένες αλλαγές
    └── feature-name/
```

### Επεξήγηση καταλόγων

- **proposals/**: Αποθηκεύει όλα τα έγγραφα προτάσεων, κάθε πρόταση σε ξεχωριστό αρχείο Markdown
- **specs/**: Αποθηκεύει έγγραφα τεχνικών προδιαγραφών που αντιστοιχούν στις προτάσεις
- **changes/**: Αποθηκεύει τις τρέχουσες αλλαγές, κάθε αλλαγή σε υποκατάλογο
- **archived/**: Αποθηκεύει τις ολοκληρωμένες και αρχειοθετημένες αλλαγές

## Βασικές εντολές

### Εντολές σχετικές με προτάσεις

```bash
# Δημιουργία νέας πρότασης
openspec proposal create

# Λίστα όλων των προτάσεων
openspec proposal list

# Προβολή λεπτομερειών πρότασης
openspec proposal view <proposal-id>

# Έγκριση πρότασης
openspec proposal approve <proposal-id>
```

### Εντολές σχετικές με προδιαγραφές

```bash
# Δημιουργία νέας προδιαγραφής
openspec spec create

# Λίστα όλων των προδιαγραφών
openspec spec list

# Προβολή λεπτομερειών προδιαγραφής
openspec spec view <spec-id>

# Ενημέρωση προδιαγραφής
openspec spec update <spec-id>
```

### Εντολές σχετικές με αλλαγές

```bash
# Δημιουργία νέας αλλαγής
openspec change create

# Λίστα όλων των αλλαγών
openspec change list

# Προβολή λεπτομερειών αλλαγής
openspec change view <change-name>

# Προσθήκη εργασίας
openspec change add-task <change-name>

# Επικύρωση αλλαγής
openspec change validate <change-name>

# Αρχειοθέτηση αλλαγής
openspec change archive <change-name>
```

### Γενικές εντολές

```bash
# Προβολή έκδοσης
openspec --version

# Προβολή βοήθειας
openspec --help

# Προβολή βοήθειας για συγκεκριμένη εντολή
openspec <command> --help
```

## Χρήση του OpenSpec στο έργο HagiCode

### Ενσωμάτωση έργου

Στο έργο HagiCode, το OpenSpec έχει ενσωματωθεί στη ροή εργασίας ανάπτυξης:

1. **Έλεγχος αρχικοποίησης**: Ο οδηγός εκκίνησης ελέγχει εάν το OpenSpec είναι εγκατεστημένο
2. **Ανάπτυξη με βάση προτάσεις**: Όλες οι νέες λειτουργίες πρέπει να περάσουν από τη ροή προτάσεων του OpenSpec
3. **Έλεγχος κώδικα**: Οι αλλαγές πρέπει να συμμορφώνονται με τις προδιαγραφές OpenSpec για να συγχωνευτούν
4. **Συγχρονισμός τεκμηρίωσης**: Η τεκμηρίωση OpenSpec διατηρείται συγχρονισμένη με την υλοποίηση του κώδικα

### Απαιτήσεις έκδοσης

Το έργο HagiCode απαιτεί τη χρήση της έκδοσης **OpenSpec >=1.0.0 <2.0.0**.

Εντολή εγκατάστασης:

```bash
npm install -g @fission-ai/openspec@1
```

### Παράδειγμα ροής εργασίας

Ακολουθεί ένα πλήρες παράδειγμα χρήσης του OpenSpec στο έργο HagiCode:

#### Βήμα 1: Δημιουργία πρότασης

```bash
cd /path/to/hagicode-project
openspec proposal create
```

Στην αλληλεπιδραστική προτροπή εισάγετε:
- Τίτλος πρότασης: Προσθήκη λειτουργίας πιστοποίησης χρηστών
- Περιγραφή προβλήματος: Το σύστημα χρειάζεται σύνδεση χρηστών και διαχείριση δικαιωμάτων
- Λύση: Υλοποίηση συστήματος πιστοποίησης με βάση το JWT

#### Βήμα 2: Σύνταξη προδιαγραφής

Μετά την έγκριση της πρότασης, δημιουργήστε λεπτομερή προδιαγραφή:

```bash
openspec spec create
```

Ορίστε στην προδιαγραφή:
- Σχεδιασμός της ροής πιστοποίησης
- Ορισμός διεπαφής API
- Δομή πινάκων βάσης δεδομένων
- Ασφαλείς παράγοντες

#### Βήμα 3: Δημιουργία αλλαγής

Ξεκινήστε την υλοποίηση:

```bash
openspec change create
```

Εισαγάγετε όνομα αλλαγής: `add-user-authentication`

#### Βήμα 4: Προσθήκη εργασιών

```bash
openspec change add-task add-user-authentication
```

Προσθέστε συγκεκριμένες εργασίες:
- Υλοποίηση διεπαφής εγγραφής χρηστών
- Υλοποίηση διεπαφής σύνδεσης
- Υλοποίηση JWT middleware
- Γράψιμο unit tests

#### Βήμα 5: Υλοποίηση και επικύρωση

Μετά την ολοκλήρωση του κώδικα, επικυρώστε την αλλαγή:

```bash
openspec change validate add-user-authentication
```

#### Βήμα 6: Αρχειοθέτηση

Μετά την επιτυχή επικύρωση, αρχειοθετήστε την αλλαγή:

```bash
openspec change archive add-user-authentication
```

## Καλές πρακτικές

### Σύνταξη προτάσεων

1. **Σαφής διατύπωση του προβλήματος**: Ορίστε με σαφήνεια το πρόβλημα που πρέπει να λυθεί, αποφύγετε ασαφείς περιγραφές
2. **Εξετάστε εναλλακτικές λύσεις**: Καταγράψτε και συγκρίνετε διαφορετικές λύσεις
3. **Αξιολόγηση επιπτώσεων**: Αναλύστε την επίδραση και τους κινδύνους στο υπάρχον σύστημα
4. **Διατηρήστε συντομία**: Οι προτάσεις πρέπει να είναι συνοπτικές και σαφείς — αφήστε τις λεπτομέρειες για την προδιαγραφή

### Σύνταξη προδιαγραφών

1. **Λεπτομερής αλλά όχι περιττός**: Παρέχετε αρκετές λεπτομέρειες, αλλά αποφύγετε το over-design
2. **Χρησιμοποιήστε διαγράμματα**: Χρησιμοποιήστε διαγράμματα ροής, αρχιτεκτονικά διαγράμματα κ.λπ. για επεξήγηση
3. **Ορίστε διεπαφές**: Ορίστε με σαφήνεια όλες τις δημόσιες διεπαφές και δομές δεδομένων
4. **Σκεφτείτε ακριαίες περιπτώσεις**: Καταγράψτε εξαιρετικές καταστάσεις και χειρισμό σφαλμάτων

### Διαχείριση αλλαγών

1. **Μικρά βήματα**: Διαχωρίστε μεγάλες λειτουργίες σε πολλά μικρά changes
2. **Ενημέρωση εγκαίρως**: Διατηρήστε συγχρονισμένη κατάσταση εργασιών και τεκμηρίωσης
3. **Κώδικας ανασκόπηση**: Διεξάγετε ανασκόπηση κώδικα μετά την ολοκλήρωση της αλλαγής
4. **Αρχειοθέτηση εγκαίρως**: Αρχειοθετήστε τις ολοκληρωμένες αλλαγές εγκαίρως, διατηρήστε το workspace καθαρό

### Συνεργασία ομάδας

1. **Τακτικές ανασκοπήσεις**: Επαναλαμβανόμενες ανασκοπήσεις προτάσεων και προδιαγραφών για εξασφάλιση ομοφωνίας
2. **Τεκμηρίωση πρώτα**: Ολοκληρώστε την τεκμηρίωση πριν ξεκινήσετε την κωδικοποίηση
3. **Διατηρήστε επικοινωνία**: Καταγράψτε συζητήσεις και αποφάσεις σε προτάσεις και προδιαγραφές
4. **Έλεγχος έκδοσης**: Συμπεριλάβετε τεκμηρίωση OpenSpec στον έλεγχο έκδοσης

## Κοινά σενάρια χρήσης

### Σενάριο 1: Προσθήκη νέας λειτουργίας

1. Δημιουργήστε μια πρόταση που περιγράφει τις απαιτήσεις της νέας λειτουργίας
2. Η ομάδα συζητά και εγκρίνει την πρόταση
3. Συντάξτε λεπτομερή τεχνική προδιαγραφή
4. Δημιουργήστε μια αλλαγή και υλοποιήστε τη λειτουργία
5. Επικυρώστε και αρχειοθετήστε την αλλαγή

### Σενάριο 2: Αναδιάρθρωση υπάρχοντος κώδικα

1. Δημιουργήστε μια πρόταση που εξηγεί τους λόγους και τους στόχους της αναδιάρθρωσης
2. Ορίστε το πλάνο αναδιάρθρωσης και τα βήματα στην προδιαγραφή
3. Δημιουργήστε αλλαγή και αναδιαρθρώστε σταδιακά
4. Διασφαλίστε την κάλυψη με τεστ, επικυρώστε τα αποτελέσματα της αναδιάρθρωσης
5. Αρχειοθετήστε την αλλαγή

### Σενάριο 3: Επιδιόρθωση σφάλματος

Για απλά σφάλματα:
- Μπορούν να διορθωθούν άμεσα, δεν απαιτείται πλήρης διαδικασία OpenSpec

Για σύνθετα σφάλματα:
1. Δημιουργήστε μια πρόταση για την ανάλυση της ριζικής αιτίας του σφάλματος
2. Ορίστε τη λύση διόρθωσης στην προδιαγραφή
3. Δημιουργήστε αλλαγή και υλοποιήστε τη διόρθωση
4. Προσθέστε τεστ για να αποτρέψετε παλινδρόμηση
5. Αρχειοθετήστε την αλλαγή

### Σενάριο 4: Διαχείριση τεχνικού χρέους

1. Δημιουργήστε μια πρόταση που απαριθμεί τα στοιχεία τεχνικού χρέους
2. Ορίστε ένα σχέδιο βελτίωσης στην προδιαγραφή
3. Δημιουργήστε αλλαγή για σταδιακή αποπληρωμή του τεχνικού χρέους
4. Παρακολουθήστε την πρόοδο και το αποτέλεσμα
5. Αρχειοθετήστε τις ολοκληρωμένες βελτιώσεις

## Ενσωμάτωση με άλλα εργαλεία

### Git ενσωμάτωση

Το OpenSpec ενσωματώνεται ομαλά με το Git:

```bash
# Η τεκμηρίωση OpenSpec πρέπει να συμπεριληφθεί στον έλεγχο έκδοσης
git add openspec/
git commit -m "Add OpenSpec proposal for feature X"
```

### CI/CD ενσωμάτωση

Επικυρώστε τις προδιαγραφές OpenSpec στη ροή CI/CD:

```yaml
# .github/workflows/openspec.yml
name: OpenSpec Validation
on: [pull_request]
jobs:
  validate:
    runs-on: ubuntu-latest
    steps:
      - uses: actions/checkout@v2
      - name: Install OpenSpec
        run: npm install -g @fission-ai/openspec@1
      - name: Validate changes
        run: openspec change validate
```

### Ενσωμάτωση IDE

Συνιστάται να χρησιμοποιείτε έναν επεξεργαστή που υποστηρίζει Markdown για την επεξεργασία εγγράφων OpenSpec:
- VS Code + Markdown plugin
- JetBrains IDEs + υποστήριξη Markdown
- Vim/Neovim + Markdown plugin

## Αντιμετώπιση προβλημάτων

### Συνήθη προβλήματα

#### Πρόβλημα: Η εντολή OpenSpec δεν βρέθηκε

**Λύση**:
1. Επιβεβαιώστε ότι το OpenSpec είναι εγκατεστημένο: `npm list -g @fission-ai/openspec`
2. Ελέγξτε εάν η μεταβλητή περιβάλλοντος PATH περιλαμβάνει τον παγκόσμιο κατάλογο bin του npm
3. Επανεγκαταστήστε: `npm install -g @fission-ai/openspec@1`

#### Πρόβλημα: Δεν είναι δυνατή η δημιουργία πρότασης ή αλλαγής

**Λύση**:
1. Επιβεβαιώστε ότι ο τρέχων κατάλογος είναι ο ριζικός κατάλογος του έργου
2. Ελέγξτε εάν υπάρχει κατάλογος `openspec/`, αν όχι δημιουργήστε τον
3. Επιβεβαιώστε ότι έχετε δικαιώματα εγγραφής

#### Πρόβλημα: Η επικύρωση απέτυχε

**Λύση**:
1. Ελέγξτε εάν τα έγγραφα στον κατάλογο αλλαγών είναι πλήρη
2. Επιβεβαιώστε ότι όλες οι απαιτούμενες εργασίες έχουν ολοκληρωθεί
3. Δείτε λεπτομερείς πληροφορίες σφάλματος: `openspec change validate <change-name> --verbose`

#### Πρόβλημα: Η αρχειοθέτηση απέτυχε

**Λύση**:
1. Βεβαιωθείτε ότι η αλλαγή έχει περάσει την επικύρωση
2. Ελέγξτε εάν υπάρχει ο κατάλογος `openspec/archived/`
3. Επιβεβαιώστε ότι δεν υπάρχουν μη δεσμευμένες αλλαγές κώδικα

## Προχωρημένη χρήση

### Προσαρμοσμένα πρότυπα

Το OpenSpec υποστηρίζει προσαρμοσμένα πρότυπα για προτάσεις και προδιαγραφές:

```bash
# Δημιουργήστε έναν κατάλογο προτύπων στο ριζικό κατάλογο του έργου
mkdir -p .openspec/templates

# Δημιουργήστε ένα προσαρμοσμένο πρότυπο πρότασης
cat > .openspec/templates/proposal.md << 'EOF'
# Proposal: {{title}}

## Problem Statement
{{problem}}

## Proposed Solution
{{solution}}

## Custom Section
Your custom content here
EOF
```

### Αρχείο ρυθμίσεων

Δημιουργήστε ένα αρχείο ρυθμίσεων `.openspec.json` στο ριζικό κατάλογο του έργου:

```json
{
  "version": "1.0",
  "templates": {
    "proposal": ".openspec/templates/proposal.md",
    "spec": ".openspec/templates/spec.md"
  },
  "validation": {
    "requireTests": true,
    "requireDocs": true
  },
  "hooks": {
    "pre-archive": "npm test"
  }
}
```

### Hooks

Το OpenSpec υποστηρίζει την εκτέλεση προσαρμοσμένων σεναρίων πριν και μετά από συγκεκριμένες ενέργειες:

```json
{
  "hooks": {
    "pre-create": "scripts/check-permissions.sh",
    "post-archive": "scripts/notify-team.sh"
  }
}
```
