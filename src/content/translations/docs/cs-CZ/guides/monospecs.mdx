## Přehled

Monospecs je přístup HagiCode ke správě více úložišť. Sleduje a spravuje dílčí úložiště `monospecs.yaml`a funguje pro jakýkoli projekt, který potřebuje jedno místo pro koordinaci více nezávislých úložišť.

### Základní koncepty

**Konfigurační soubor**

`monospecs.yaml` je jádrem celého řešení pro správu a definuje:
- Seznam a umístění dílčích úložišť
- Git URL pro každé úložiště
- Zobrazovaný název a ikona
- Chování při integraci s OpenSpec

**Odlišnosti od běžných úložišť**

| Funkce | Běžné úložiště | Projekt řízený Monospecs |
|----------|-------------------|---------------------------|
| **Umístění OpenSpec** | Kořenový adresář `openspec/` | Hlavní kořenový adresář úložiště `openspec/` |
| **Změnit rozsah** | Omezeno na aktuální úložiště | Může zahrnovat více dílčích úložišť |
| **Čistota dílčího úložiště** | Specifikace smíchané s kódem | Specifikace stanovené v hlavním úložišti, dílčí repozitáře jsou čistší |
| **Automatické potvrzení archivu** | Vyžaduje ruční potvrzení specifikace | `commit_when_archive: true` automaticky odevzdá specifikace do hlavního úložiště |

### Proč používat Monospecs?

- **Sjednocená správa**: Spravujte všechna dílčí úložiště v jednom konfiguračním souboru
- **Podpora automatizace**: Poskytuje metadata úložiště pro skripty a nástroje
- **Přívětivost AI**: Strukturovaná konfigurace pomáhá agentovi AI porozumět vztahům s úložištěm
- **OpenSpec Integration**: Bezproblémová integrace s pracovním postupem správy změn
- **Čistící dílčí repozitáře**: Specifikace jsou uchovávány v hlavním repozitáři, dílčí repozitáře obsahují pouze skutečný kód

### Použitelné scénáře

Nejprve použijte Monospecs ve scénářích, jako jsou tyto:

**1. Kolaborativní vývoj ve více úložištích**
Když je projekt rozdělen do více nezávislých úložišť:
- Frontendová aplikace, backendová služba a desktopová aplikace vyvinuté samostatně
- Potřeba koordinovat verze a závislosti napříč repozitáři
- Dokumentace a pomocné projekty oddělené od hlavního kódu

**2. Pomocný projektový management**
Použijte Monospecs ke správě:
- Dokumentační projekt
- Oficiální stránky
- Vytvářejte nástroje a skripty
- Každé úložiště si udržuje vlastní správu verzí

**3. Vývoj funkcí napříč úložišti**
Funkce, která vyžaduje změny ve více úložištích:
- Je třeba sledovat změny ve kterých úložištích
- Automaticky potvrdit specifikace pro opravu cílového úložiště
- Koordinujte aktualizace verzí a závislosti

## Rychlý start

### Inicializace

Pro dokončení inicializace monospecs postupujte takto:

#### 1. Vytvořte prázdné úložiště git

Vytvořte prázdnou `git` složku ve vašem hlavním úložišti (poznámka: toto je `git`, ne `repos/` - toto je konvence monospec):

```bash
mkdir git
cd git
git init
cd ..
```

#### 2. Nakonfigurujte soubor monospecs

Vytvořte `monospecs.yaml` konfigurační soubor v kořenovém adresáři hlavního úložiště:

```yaml
version: "1.0"
commit_when_archive: true

repositories:
  - path: "repos/frontend"
    url: "https://github.com/your-org/frontend.git"
    displayName: "Frontend App"
    icon: "🌐"

  - path: "repos/backend"
    url: "https://github.com/your-org/backend.git"
    displayName: "Backend Service"
    icon: "⚙️"
```

**Položky konfigurace**:
| Položka konfigurace | Typ | Povinné | Popis |
|----------------|------|----------|-------------|
| `version` | řetězec | Ano | Verze konfiguračního souboru |
| `commit_when_archive` | booleovský | ne | Zda při archivaci automaticky potvrdit specifikace (nepotvrdit kód sub-repo) |
| `repositories` | pole | Ano | Seznam konfigurace úložiště |
| `repositories[].path` | řetězec | Ano | Cesta k úložišti (vzhledem ke kořenu projektu) |
| `repositories[].url` | řetězec | Ano | Adresa URL úložiště Git |
| `repositories[].displayName` | řetězec | ne | Zobrazovaný název úložiště |
| `repositories[].icon` | řetězec | ne | Ikona úložiště (emoji) |

### Přidejte dílčí úložiště do hlavního úložiště

Přidejte všechna dílčí úložiště do `repos/` adresář. Tento adresář musí být **vyloučen z kontroly verzí Git v hlavním úložišti**, aby se zabránilo sledování obsahu dílčího úložiště:

```plaintext
# Exclude all sub-repositories
repos/
```

#### 3. Přidejte nadřazené úložiště do HagiCode

Pokud používáte desktopovou aplikaci HagiCode, otevřete aplikaci a přidejte své hlavní úložiště jako projekt. Klikněte na tlačítko "Přidat projekt" nebo "+" na domovské obrazovce, vyberte hlavní složku úložiště a uvidíte všechna dílčí úložiště a jejich stavové informace.

#### 4. Klonování dílčích repozitářů

Použijte `monospecs.yaml` konfigurace pro klonování všech nakonfigurovaných dílčích repozitářů do `repos/` adresář:

Tím se naklonují všechna dílčí úložiště na základě adres URL uvedených v konfiguračním souboru.

## Struktura konfiguračního souboru

### Konfigurace kořenové úrovně

```yaml
version: "1.0"
commit_when_archive: true

repositories:
  # ... repository configurations
```

### Konfigurace úrovně úložiště

Každé dílčí úložiště může mít:
- OpenSpec změnit složku (`openspec/`)
- Nezávislý `.git/` adresář
- Volitelné `AGENTS.md` pro konfiguraci agenta AI

## Operace správy úložiště

### Přidat nové úložiště

Při přidávání nového dílčího úložiště do `monospecs.yaml` `repositories` pole:

```yaml
repositories:
  - path: "repos/new-service"
    url: "https://github.com/HagiCode-org/new-service.git"
    displayName: "New Service"
    icon: "🆕"
```

### Aktualizujte konfiguraci úložiště

Když se změní adresy URL úložiště nebo metadata:
1. Upravit `monospecs.yaml` pro aktualizaci příslušných záznamů
2. Ověřte správnost syntaxe YAML
3. Pokud se synchronizace změní, ručně aktualizujte konfiguraci místního úložiště

## Podpora konfigurace agenta AI

### AGENTS.md

The `AGENTS.md` poskytuje specifické pokyny pro úložiště pro agenta AI:

**Technické informace o zásobníku**
- Rámcové a sestavovací nástroje
- Kódové konvence a postupy pojmenování
- Správa verzí a konfigurace nasazení

**Chování specifické pro projekt**
- Rozšíření konfigurace a speciální požadavky
- Vzory integrace OpenSpec
- Potřeby koordinace napříč úložišti

### Výhody AI

Použití `AGENTS.md` umožňuje agentovi AI:
- Pochopte vztahy a závislosti úložiště
- Generujte kód a refaktor napříč více úložišti
- Udržujte konzistenci správy verzí a konfigurací

## Integrace OpenSpec

### Jak Monospecs funguje s OpenSpec

Ať už spravujete projekt spravovaný monospec nebo běžné úložiště, můžete použít pracovní postup OpenSpec HagiCode:

**Běžné úložiště:**
- OpenSpec v rootu `openspec/` adresář
- Změny se týkají pouze aktuálního úložiště

**Monospecs Managed Project:**
- OpenSpec čte `monospecs.yaml` konfigurace
- Změny sledované dílčím úložištěm
- Specifikace se automaticky zavazují ke správným cílům

### Archivační chování

Kdy `commit_when_archive: true`:
- Specifikace jsou automaticky odevzdány do hlavního úložiště
- Kód dílčích úložišť není potvrzen (zůstává oddělený)
- Zjednodušuje správu specifikací – pro dílčí repozitáře není potřeba žádné ruční potvrzení specifikací

### Manuální závazky dílčího úložiště

Dílčí repozitáře udržují nezávislou správu verzí:
- Potvrdit skutečné změny kódu do svých vlastních úložišť
- Specifikace jsou potvrzeny odděleně (z hlavního úložiště při archivaci)

## Nejlepší postupy

### Konvence pojmenování úložišť

- Pro cesty k úložišti použijte malá písmena (malá písmena a pomlčky).
- Používejte jednoduché, popisné zobrazované názvy v čínštině
- Použijte relevantní emotikony jako ikony

### Pokyny pro zobrazované názvy

- Udržujte to stručné a popisné
- Používejte konzistentní pojmenování napříč podobnými repozitáři
- Vyberte ikony, které představují účel úložiště

### Údržba konfigurace

**Konfigurace synchronizace se skutečným stavem**
- Pravidelně ověřovat `monospecs.yaml` odráží skutečnou strukturu úložiště
- Aktualizujte nebo odeberte položky při přidání nebo odebrání úložišť
- Otestujte změny, abyste zajistili platnost konfigurace

**Ovládání verzí pro konfiguraci**
- Sledujte změny konfiguračního souboru pomocí git
- Zdokumentujte důvody velkých změn konfigurace
- Verze konfiguračních souborů značek pro snadné vrácení zpět

## Odstraňování problémů

### Konfigurační soubor nenalezen

Pokud `monospecs.yaml` soubor nebyl nalezen v kořenovém adresáři projektu:
1. Zkontrolujte, zda soubor existuje ve správném umístění
2. Ověřte, že jste ve správném pracovním adresáři
3. Zkontrolujte překlepy v názvu konfiguračního souboru

### Chyby syntaxe YAML

Běžné problémy se syntaxí YAML:
- Nesprávné odsazení (použijte mezery, nikoli tabulátory)
- Chybějící nebo nadbytečné uvozovky kolem řetězců
- Neplatné typy dat (řetězce potřebují uvozovky, čísla ne)
- Chybí povinná pole (např `path` nebo `url`)

### Úložiště nezjištěno

Pokud se nově přidané úložiště nezobrazuje:
1. Zkontrolujte `path` je správně v `monospecs.yaml`
2. Ověřte, že úložiště bylo úspěšně naklonováno
3. Zkontrolujte, zda úložiště existuje v očekávaném umístění

### Selhání klonování

Pokud se klonování úložiště nezdaří:
1. Zkontrolujte připojení k síti
2. Ověřte správnost adresy URL úložiště Git
3. Zkontrolujte problémy s ověřením
4. Zkontrolujte stav kontejneru Docker
5. Zkontrolujte dostupnost místa na disku
