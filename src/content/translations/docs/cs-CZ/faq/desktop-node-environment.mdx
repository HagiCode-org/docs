HagiCode Desktop je dodáván s desktopem spravovaným Node toolchainem. Nepřebírá globální instalaci Node na vašem počítači. Jeho účelem je poskytnout spouštění Desktopu a tokům souvisejícím s běhovým prostředím stabilní a ověřitelný základ běhu.

Přečtěte si tento FAQ, pokud vidíte `portable-fixed/toolchain` v diagnostice, třeba vědět, které `node` Desktop používá, nebo se divte, proč verze vašeho systémového uzlu přímo nedefinuje běhové chování plochy.

## Proč Desktop balí Node

Desktop musí spouštět stejné podpůrné toky napříč operačními systémy, instalačními kanály a uživatelskými prostředími. Pokud by to záviselo pouze na existující instalaci uzlu uživatele, podpora by musela zvládnout zastaralé verze, chybějící npm, nesprávné řazení cest, znečištěné globální balíčky a zásady registru specifické pro organizaci najednou.

Desktop to konkrétněji popisuje jako izolovaný Node/npm runtime pro spravovaná CLI namísto opětovného použití systémového prostředí. Tento design má tyto hlavní výhody:

- **Konzistentní verze**: Desktop používá připnutou smlouvu runtime Node 22 namísto hádání místní hlavní verze uživatele.
- **Řízené toky spouštění**: Příkazy spravované počítačem mohou preferovat zabalené `node` a `npm` záznamy.
- **Jasné hranice konfigurace**: Konfigurace související s CLI je uchovávána odděleně od globální konfigurace celého hostitele, takže je méně pravděpodobné, že aktualizace, přepínání a obnova budou narušeny rozdíly v systémovém prostředí.
- **Snížené konflikty**: Připojené cesty se vztahují na příkazová prostředí vytvořená Desktopem a nenahrazují globální uzel uživatele v jejich terminálu.
- **Nerušivé chování hostitele**: Tato izolace zabraňuje zápisu stavu toolchain do stávajícího prostředí Node/npm uživatele a odinstalování Desktopu nenaruší uzel, npm nebo konfiguraci, která je již na počítači přítomna.

## Jak Desktop vybere cestu k uzlu

Když spustíte Desktop normálně, Desktop upřednostňuje sadu nástrojů Node nainstalovanou s aplikací místo toho, aby nejprve načetl instalaci Node z vašeho systému. To udržuje verzi uzlu ve vašem terminálu oddělenou od verze uzlu, kterou Desktop očekává, když běží.

Když je sdružená sada nástrojů povolena, příkazová prostředí vytvořená aplikací Desktop umístí cesty sdružených řetězců nástrojů na začátek `PATH`. Výsledkem je, že i když má váš systém nainstalovanou jinou verzi uzlu, toky spravované desktopem se nejprve přenesou na přibalený Node a npm Desktopu.

To má vliv pouze na běhová prostředí vytvořená Desktopem. Pokud běžíš `node`, `npm`, nebo jiné příkazy přímo ve vašem systémovém terminálu, budou nadále používat vaše normální systémové prostředí a konfiguraci shellu.

V diagnostických prostředích nebo prostředích podřízených procesů spouštěných Desktopem můžete vidět proměnnou jako je tato:

```bash
HAGICODE_PORTABLE_TOOLCHAIN_ROOT=/path/to/portable-fixed/toolchain
```

Tato proměnná znamená, že běhové prostředí spravované desktopem rozpoznalo kořen přidruženého řetězce nástrojů. Zobrazení této cesty neznamená, že Desktop změnil váš systémový uzel; Znamená to pouze, že Desktop používá prostředí Node dodané s aplikací v rámci vlastního běhového rozsahu.

## Kde Desktop instaluje spravované balíčky npm

Desktop uchovává soubory runtime a měnitelné soubory balíčků npm na různých místech. Připojená dráha toolchain, jako je např `portable-fixed/toolchain/node` je neměnný zdroj pro desktop spravovaný `node` a `npm` příkazy. Spravované globální balíčky se místo toho instalují do adresáře uživatelských dat Electron:

```text
userData/node22/npmGlobal
```

Pokud se Desktop později přesune do Node 24, použije samostatný strom balíčků, jako je např `userData/node24/npmGlobal`. Balíčky nainstalované pro Node 22 zůstávají na disku, ale nejsou považovány za nainstalované pro Node 24, dokud nejsou přeinstalovány do nové předpony Node-major.

Stav závislosti na ploše kontroluje pouze aktivní `userData/node<major>/npmGlobal` předpona. Soubory balíčků mimo tuto předponu jsou správou závislostí ignorovány.
