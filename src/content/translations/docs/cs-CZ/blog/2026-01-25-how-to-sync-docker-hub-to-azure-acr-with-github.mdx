## Automatizace synchronizace obrázků z centra Docker do Azure ACR

> Tento článek vysvětluje, jak používat akce GitHub a nástroj pro synchronizaci obrázků k automatizaci synchronizace obrázků z Docker Hub do Azure Container Registry, řešící pomalý přístup k Docker Hub v Číně a některých oblastech Azure a zároveň zlepšovat dostupnost image a efektivitu nasazení v prostředích Azure.

## Pozadí / Úvod

Projekt HagiCode používá obrazy Docker jako své základní runtime komponenty, přičemž primární obrazy jsou hostovány na Docker Hub. Jak projekt rostl a potřeby nasazení Azure vzrostly, narazili jsme na následující problémy:

- Nízká rychlost stahování obrazu s omezeným přístupem k Docker Hub v Číně a některých oblastech Azure
- Jediný zdroj obrazu vytváří jediný bod rizika selhání
- Azure Container Registry poskytuje lepší výkon sítě a integraci při spuštění v Azure

Abychom tyto problémy vyřešili, potřebovali jsme automatizovaný mechanismus synchronizace obrazů, který by pravidelně synchronizoval obrazy z centra Docker do Azure ACR, což uživatelům zajistilo rychlejší vytahování obrazu a vyšší dostupnost v prostředích Azure.

## Porovnání technických řešení

Při definování řešení jsme porovnávali několik technických možností:

### 1. image-syncer (konečná volba)
- Přírůstková synchronizace: synchronizuje pouze změněné vrstvy obrazu, což výrazně snižuje přenos po síti
- Obnovit podporu: synchronizace může pokračovat po přerušení sítě
- Řízení souběžnosti: konfigurovatelná pracovní vlákna zlepšují efektivitu synchronizace velkých obrázků
- Robustní zpracování chyb: vestavěný mechanismus opakování pro selhání (ve výchozím nastavení 3 opakování)
- Odlehčené nasazení: jediný binární soubor bez závislostí
- Podpora více registrů: kompatibilní s Docker Hub, Azure ACR, Harbor a dalšími

### 2. Docker CLI
- Žádná přírůstková synchronizace: pokaždé je nutné stáhnout celý obsah obrázku
- Nižší účinnost: velký objem síťových přenosů a delší doba provádění
- Jednoduché použití: založené na známém `docker pull` a `docker push` příkazy

### 3. Azure CLI
- Vyšší složitost: Musí být nakonfigurováno ověřování Azure CLI
- Omezená funkčnost: `az acr import` je svým rozsahem poměrně úzký
- Nativní integrace: dobře se integruje se službami Azure

## Kroky implementace

### 1. Příprava
- Vytvořte nebo potvrďte Azure Container Registry v Azure Portal
- Vytvořte přístupové údaje ACR (uživatelské jméno a heslo)
- Potvrďte přístupová oprávnění pro úložiště obrázků Docker Hub

### 2. Nakonfigurujte tajemství GitHubu
Přidejte následující tajemství do nastavení úložiště GitHub:
- AZURE_ACR_USERNAME: Uživatelské jméno Azure ACR
- AZURE_ACR_PASSWORD: Heslo Azure ACR

### 3. Vytvořte pracovní postup akcí GitHub
Nakonfigurujte pracovní postup v `.github/workflows/sync-docker-acr.yml`:
- Plánované spuštění: každý den v 00:00 UTC
- Manuální spoušť: podporuje `workflow_dispatch`
- Dodatečná spoušť: běží po stisknutí tlačítka `publish` větev pro rychlou synchronizaci

### 4. Tok provádění pracovního postupu

| Krok | Účastník | Akce | Popis |
| --- | --- | --- | --- |
| 1 | Akce GitHubu | Spusťte pracovní postup | Spuštěno plánem, ručním spuštěním nebo přesunem do `publish` větev |
| 2 | Akce GitHubu → synchronizace obrázků | Stáhněte a spusťte nástroj synchronizace | Vstupuje do skutečné fáze synchronizace |
| 3 | image-syncer → Docker Hub | Načte manifesty obrázků a seznam značek | Čte metadata ze zdrojového úložiště |
| 4 | image-syncer → Azure ACR | Načíst existující obrazová data z cílového úložiště | Určuje aktuální stav cílové strany |
| 5 | synchronizátor obrazu | Porovnejte zdrojové a cílové rozdíly | Identifikuje vrstvy obrazu, které potřebují synchronizaci |
| 6 | image-syncer → Docker Hub | Vytáhněte změněné vrstvy obrázku | Přenese pouze obsah, který potřebuje aktualizaci |
| 7 | image-syncer → Azure ACR | Push změněné vrstvy obrazu | Dokončí přírůstkovou synchronizaci |
| 8 | image-syncer → Akce GitHubu | Vrátit statistiky synchronizace | Zahrnuje výsledky, rozdíly a informace o chybách |
| 9 | Akce GitHubu | Zaznamenávejte protokoly a nahrávejte artefakty | Podporuje pozdější auditování a odstraňování problémů |

## Implementace pracovního postupu akcí GitHub

Níže je skutečná používaná konfigurace pracovního postupu (`.github/workflows/sync-docker-acr.yml`):

```yaml
name: Sync Docker Image to Azure ACR

on:
  schedule:
    - cron: "0 0 * * *" # Every day at 00:00 UTC
  workflow_dispatch: # Manual trigger
  push:
    branches: [publish]

permissions:
  contents: read

jobs:
  sync:
    runs-on: ubuntu-latest

    steps:
      - name: Checkout code
        uses: actions/checkout@v4

      - name: Download image-syncer
        run: |
          # Download the image-syncer binary
          wget https://github.com/AliyunContainerService/image-syncer/releases/download/v1.5.5/image-syncer-v1.5.5-linux-amd64.tar.gz
          tar -zxvf image-syncer-v1.5.5-linux-amd64.tar.gz
          chmod +x image-syncer

      - name: Create auth config
        run: |
          # Generate the authentication config file (YAML format)
          cat > auth.yaml <<EOF
          hagicode.azurecr.io:
            username: "${{ secrets.AZURE_ACR_USERNAME }}"
            password: "${{ secrets.AZURE_ACR_PASSWORD }}"
          EOF

      - name: Create images config
        run: |
          # Generate the image sync config file (YAML format)
          cat > images.yaml <<EOF
          docker.io/newbe36524/hagicode: hagicode.azurecr.io/hagicode
          EOF

      - name: Run image-syncer
        run: |
          # Run synchronization (using the newer --auth and --images parameters)
          ./image-syncer --auth=./auth.yaml --images=./images.yaml --proc=10 --retries=3

      - name: Upload logs
        if: always()
        uses: actions/upload-artifact@v4
        with:
          name: sync-logs
          path: image-syncer-*.log
          retention-days: 7
```

## Podrobnosti konfigurace

### 1. Spouštěcí podmínky
- Naplánované spuštění: `cron: "0 0 * * *"` běží každý den v 00:00 UTC
- Manuální spoušť: `workflow_dispatch` umožňuje uživatelům spustit jej ručně z uživatelského rozhraní GitHub
- Stiskněte spoušť: `push: branches: [publish]` spouští se při stisknutí uvolňovací větve pro rychlou synchronizaci

### 2. Konfigurace ověřování (`auth.yaml`)
```yaml
hagicode.azurecr.io:
  username: "${{ secrets.AZURE_ACR_USERNAME }}"
  password: "${{ secrets.AZURE_ACR_PASSWORD }}"
```

### 3. Konfigurace synchronizace obrazu
```yaml
docker.io/newbe36524/hagicode: hagicode.azurecr.io/hagicode
```
Tato konfigurace znamená, že všechny značky z `docker.io/newbe36524/hagicode` budou synchronizovány s `hagicode.azurecr.io/hagicode`.

### 4. parametry synchronizace obrazu
- `--auth=./auth.yaml`: cesta ke konfiguračnímu souboru ověřování
- `--images=./images.yaml`: cesta ke konfiguračnímu souboru synchronizace obrazu
- `--proc=10`: použití `10` souběžní pracovníci
- `--retries=3`: selhání opakování `3` časy

## Poznámky a doporučené postupy

### 1. Bezpečnostní doporučení
- Pravidelně střídejte hesla Azure ACR (doporučeno každých 90 dní)
- Použijte vyhrazený účet služby ACR s oprávněními pouze pro push
- Sledujte protokoly přístupu ACR, abyste včas odhalili abnormální přístup
- Nevypisujte přihlašovací údaje do protokolů
- Nezavazujte přihlašovací údaje do úložiště kódu

### 2. Optimalizace výkonu
- Upravte `--proc` parametr: ladění souběžnosti na základě šířky pásma sítě (doporučeno `5-20`)
- Sledovat dobu trvání synchronizace: pokud synchronizace trvá příliš dlouho, zvažte snížení souběžnosti
- Pravidelně čistěte protokoly: nastavte rozumnou hodnotu `retention-days` hodnota (`7` aktuálně)

### 3. Odstraňování problémů

#### Problém 1: Selhání ověření
```
Error: failed to authenticate to hagicode.azurecr.io
```
Řešení:
1. Zkontrolujte, zda jsou tajemství GitHub správně nakonfigurována
2. Ověřte, zda vypršela platnost hesla Azure ACR
3. Potvrďte, že oprávnění účtu služby ACR jsou správná

#### Problém 2: Časový limit sítě
```
Error: timeout waiting for response
```
Řešení:
1. Zkontrolujte připojení k síti
2. Snižte počet souběžných pracovníků ( `--proc` parametr)
3. Počkejte, až se síť obnoví, a znovu spusťte pracovní postup

#### Problém 3: Neúplná synchronizace obrazu
```
Warning: some tags failed to sync
```
Řešení:
1. Zkontrolujte protokoly synchronizace a identifikujte neúspěšné značky
2. Ručně znovu spusťte pracovní postup a synchronizujte jej znovu
3. Ověřte, zda je zdrojový obraz na Docker Hub v pořádku

### 4. Monitorování a varování
- Stav pracovního postupu pravidelně kontrolujte na stránce Akce
- Povolte oznámení GitHubu, aby byla selhání pracovního postupu okamžitě hlášena
- Monitorujte využití úložiště Azure ACR
- Pravidelně ověřujte konzistenci značek napříč registry

## Často kladené otázky a řešení

### Q1: Jak mohu synchronizovat konkrétní značky místo všech značek?

Upravte `images.yaml` konfigurační soubor:
```yaml
# Sync only the latest and v1.0 tags
docker.io/newbe36524/hagicode:latest: hagicode.azurecr.io/hagicode:latest
docker.io/newbe36524/hagicode:v1.0: hagicode.azurecr.io/hagicode:v1.0
```

### Q2: Jak mohu synchronizovat více úložišť obrázků?

Přidat více řádků do `images.yaml`:
```yaml
docker.io/newbe36524/hagicode: hagicode.azurecr.io/hagicode
docker.io/newbe36524/another-image: hagicode.azurecr.io/another-image
```

### Q3: Jak mohu opakovat po selhání synchronizace?
- Automatické opakování: synchronizace obrazu obsahuje vestavěný mechanismus opakování (`3` ve výchozím nastavení opakování)
- Ruční opakování: klikněte `Re-run all jobs` na stránce Akce GitHubu

### Q4: Jak mohu zobrazit podrobný průběh synchronizace?
- Prohlédněte si protokoly v reálném čase na stránce Akce
- Stáhněte si `sync-logs` artefakt pro kontrolu úplných souborů protokolu
- Soubory protokolu obsahují stav synchronizace a přenosovou rychlost pro každou značku

### Q5: Jak dlouho trvá synchronizace?
- Počáteční úplná synchronizace: obvykle `10-30` minut v závislosti na velikosti snímku
- Přírůstková synchronizace: obvykle `2-5` minut, pokud jsou změny obrazu malé
- Načasování závisí na šířce pásma sítě, velikosti obrazu a nastavení souběžnosti

## Navrhovaná rozšíření

### 1. Přidejte oznámení o synchronizaci
Přidejte do pracovního postupu krok oznámení:
```yaml
- name: Notify on success
  if: success()
  run: |
    echo "Docker images synced successfully to Azure ACR"
```

### 2. Implementujte filtrování značek obrázků
Přidejte do pracovního postupu logiku filtrování značek:
```yaml
- name: Filter tags
  run: |
    # Sync only tags prefixed with v
    echo "docker.io/newbe36524/hagicode:v* : hagicode.azurecr.io/hagicode:v*" > images.yaml
```

### 3. Přidejte zprávu o statistice synchronizace
```yaml
- name: Generate report
  if: always()
  run: |
    echo "## Sync Report" >> $GITHUB_STEP_SUMMARY
    echo "- Total tags: $(grep -c 'synced' image-syncer-*.log)" >> $GITHUB_STEP_SUMMARY
    echo "- Sync time: ${{ steps.sync.outputs.duration }}" >> $GITHUB_STEP_SUMMARY
```
