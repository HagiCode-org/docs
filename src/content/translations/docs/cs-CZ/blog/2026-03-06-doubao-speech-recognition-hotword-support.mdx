## Průvodce implementací podpory klíčových slov pro rozpoznávání řeči Doubao

> Tento článek podrobně vysvětluje, jak implementovat podporu klíčových slov pro rozpoznávání řeči Doubao v projektu HagiCode. Použitím vlastních klíčových slov a tabulek klíčových slov platformy můžete výrazně zlepšit přesnost rozpoznávání slovní zásoby specifické pro doménu.

## Pozadí

Technologie rozpoznávání řeči se vyvíjela mnoho let, přesto jeden problém neustále trápí vývojáře. Univerzální modely rozpoznávání řeči mohou pokrýt běžný jazyk, ale často zaostávají, pokud jde o odbornou terminologii, názvy produktů a osobní jména. Přemýšlejte o tom: hlasový asistent v lékařské oblasti musí přesně rozpoznat pojmy jako "hypertenze", "diabetes" a "ischemická choroba srdeční"; právní systém musí přesně vystihovat pojmy jako „příčina jednání“, „obrana“ a „důkazní břemeno“. V těchto scénářích se univerzální model snaží ze všech sil, ale to často nestačí.

Na stejnou výzvu jsme narazili v projektu HagiCode. Jako multifunkční asistent kódování AI potřebuje HagiCode zvládnout rozpoznávání řeči pro širokou škálu technické terminologie. Rozhraní API pro rozpoznávání řeči Doubao však ve výchozí konfiguraci nemohlo plně splnit naše požadavky na přesnost pro specializované výrazy. Není to tak, že by Doubao nebyl dost dobrý; spíše, každá doména má svůj vlastní terminologický systém. Po nějakém výzkumu a technickém průzkumu jsme zjistili, že rozhraní API pro rozpoznávání řeči Doubao ve skutečnosti poskytuje podporu klíčových slov. Díky jednoduché konfiguraci může výrazně zlepšit přesnost rozpoznávání konkrétní slovní zásoby. V jistém smyslu, jakmile mu řeknete, kterým slovům má věnovat pozornost, bude jim naslouchat pečlivěji.

Tento článek sdílí kompletní řešení, které jsme použili v projektu HagiCode k implementaci klíčových slov pro rozpoznávání řeči Doubao. K dispozici jsou oba režimy, vlastní klíčová slova a tabulky klíčových slov platformy, a lze je také kombinovat. S tímto řešením mohou vývojáři flexibilně konfigurovat klíčová slova na základě obchodních scénářů, takže systém rozpoznávání řeči může lépe „rozpoznat“ profesionální, neobvyklou, ale kritickou slovní zásobu.

## Implementace jádra

### Porozumění dvěma režimům klíčových slov

Rozhraní API pro rozpoznávání řeči Doubao poskytuje dva způsoby konfigurace klíčových slov a každý z nich má své vlastní ideální případy použití a výhody.

**Vlastní režim klíčových slov** nám umožňuje předávat text klíčových slov přímo přes `corpus.context` pole. Tento přístup je vhodný zejména pro scénáře, kdy potřebujete rychle nakonfigurovat malý počet klíčových slov, například dočasně rozpoznat název produktu nebo jméno osoby. V implementaci HagiCode analyzujeme víceřádkový text klíčového slova zadaný uživatelem do seznamu řetězců a poté jej naformátujeme do `context_data` pole vyžadované rozhraním Doubao API. Tento přístup je velmi přímý: jednoduše řeknete systému, kterým slovům má věnovat pozornost, a on to přesně udělá.

**Režim tabulky klíčových slov platformy** používá `corpus.boosting_table_id` pole pro odkaz na předem nakonfigurovanou tabulku klíčových slov v samoučící se platformě Doubao. Tento přístup je vhodný pro scénáře, kde potřebujete spravovat velké množství klíčových slov. Můžeme vytvářet a udržovat tabulky klíčových slov na samoučící platformě Doubao a poté na ně odkazovat pomocí ID. Pro projekt, jako je HagiCode, kde je třeba neustále aktualizovat a udržovat specializované termíny, nabízí tento režim mnohem lepší správu. Jakmile počet klíčových slov naroste, mít centralizované místo pro jejich správu je mnohem lepší, než je pokaždé zadávat ručně.

Zajímavé je, že tyto dva režimy lze používat i společně. Doubao API podporuje zahrnutí jak vlastních klíčových slov, tak ID tabulky klíčových slov platformy do stejného požadavku, přičemž kombinační strategie je řízena `combine_mode` parametr. Tato flexibilita umožňuje HagiCode zvládnout širokou škálu komplexních požadavků na rozpoznání odborné terminologie. Někdy kombinace více přístupů přináší lepší výsledky.

### Definice typů frontendu a ověření

V implementaci frontendu HagiCode jsme definovali kompletní sadu typů konfigurace klíčových slov a logiky ověřování. První část je definice typu:

```typescript
export interface HotwordConfig {
  contextText: string;           // Multi-line hotword text
  boostingTableId: string;      // Doubao platform hotword table ID
  combineMode: boolean;          // Whether to use both together
}
```

Toto jednoduché rozhraní obsahuje všechny konfigurační položky pro funkci klíčového slova. Mezi nimi `contextText` je část, se kterou uživatelé interagují nejpříměji: umožňujeme uživatelům zadat jedno klíčové slovo na řádek, což je velmi intuitivní. Požádat uživatele, aby zadali jeden výraz na řádek, je mnohem snazší, než aby pochopili složitý konfigurační formát.

Následuje funkce ověření. Na základě omezení Doubao API jsme definovali přísná pravidla ověřování: maximálně 100 řádků textu klíčových slov, až 50 znaků na řádek a ne více než 5000 znaků celkem; `boosting_table_id` může mít maximálně 200 znaků a může obsahovat pouze písmena, čísla, podtržítka a spojovníky. Tyto limity nejsou libovolné; pocházejí přímo z oficiální dokumentace Doubao. Limity API jsou limity API a musíme je dodržovat.

```typescript
export function validateContextText(contextText: string): HotwordValidationResult {
  if (!contextText || contextText.trim().length === 0) {
    return { isValid: true, errors: [] };
  }

  const lines = contextText.split('\n').filter(line => line.trim().length > 0);
  const errors: string[] = [];

  if (lines.length > 100) {
    errors.push(`Hotword line count cannot exceed 100 lines; current count is ${lines.length}`);
  }

  const totalChars = contextText.length;
  if (totalChars > 5000) {
    errors.push(`Total hotword character count cannot exceed 5000; current count is ${totalChars}`);
  }

  for (let i = 0; i < lines.length; i++) {
    if (lines[i].length > 50) {
      errors.push(`Hotword on line ${i + 1} exceeds the 50-character limit`);
    }
  }

  return { isValid: errors.length === 0, errors };
}

export function validateBoostingTableId(boostingTableId: string): HotwordValidationResult {
  if (!boostingTableId || boostingTableId.trim().length === 0) {
    return { isValid: true, errors: [] };
  }

  const errors: string[] = [];

  if (boostingTableId.length > 200) {
    errors.push(`boosting_table_id cannot exceed 200 characters; current count is ${boostingTableId.length}`);
  }

  if (!/^[a-zA-Z0-9_-]+$/.test(boostingTableId)) {
    errors.push('boosting_table_id can contain only letters, numbers, underscores, and hyphens');
  }

  return { isValid: errors.length === 0, errors };
}
```

Tyto ověřovací funkce se spustí okamžitě, když uživatel nakonfiguruje klíčová slova, což zajišťuje, že problémy budou zachyceny co nejdříve. Z hlediska uživatelské zkušenosti je tento druh okamžité zpětné vazby velmi důležitý. Pro uživatele je vždy lepší vědět, co je špatně během psaní, než až po odeslání.

### Stálost konfigurace frontendu

V implementaci frontendu HagiCode jsme se rozhodli použít prohlížeč `localStorage` pro uložení konfigurace klíčových slov. Za tímto návrhovým rozhodnutím stálo několik úvah. Za prvé, konfigurace klíčových slov je vysoce personalizovaná a různí uživatelé mohou mít různé potřeby specifické pro doménu. Za druhé, tento přístup zjednodušuje implementaci backendu, protože nevyžaduje další databázové tabulky nebo koncové body API. Poté, co jej uživatelé jednou nakonfigurují v prohlížeči, lze nastavení automaticky načíst při dalších použitích, což je velmi pohodlné. Jednoduše řečeno, je to nejjednodušší způsob.

```typescript
const HOTWORD_STORAGE_KEYS = {
  contextText: 'hotword-context-text',
  boostingTableId: 'hotword-boosting-table-id',
  combineMode: 'hotword-combine-mode',
} as const;

export const DEFAULT_HOTWORD_CONFIG: HotwordConfig = {
  contextText: '',
  boostingTableId: '',
  combineMode: false,
};

// Load hotword configuration
export function loadHotwordConfig(): HotwordConfig {
  const contextText = localStorage.getItem(HOTWORD_STORAGE_KEYS.contextText) || '';
  const boostingTableId = localStorage.getItem(HOTWORD_STORAGE_KEYS.boostingTableId) || '';
  const combineMode = localStorage.getItem(HOTWORD_STORAGE_KEYS.combineMode) === 'true';

  return { contextText, boostingTableId, combineMode };
}

// Save hotword configuration
export function saveHotwordConfig(config: HotwordConfig): void {
  localStorage.setItem(HOTWORD_STORAGE_KEYS.contextText, config.contextText);
  localStorage.setItem(HOTWORD_STORAGE_KEYS.boostingTableId, config.boostingTableId);
  localStorage.setItem(HOTWORD_STORAGE_KEYS.combineMode, String(config.combineMode));
}
```

Logika tohoto kódu je přímá a jasná. Čteme z `localStorage` při načítání konfigurace a zapisovat do `localStorage` při jeho ukládání. Poskytujeme také výchozí konfiguraci, takže systém může stále správně fungovat, když ještě žádná konfigurace neexistuje. Koneckonců musí existovat rozumné výchozí nastavení.

### Backend SDK Configuration Extension

V backendové implementaci HagiCode jsme potřebovali přidat vlastnosti související s klíčovým slovem do konfigurační třídy SDK. S ohledem na charakteristiky jazyka C# a vzorce použití jsme použili `List<string>` pro uložení kontextů vlastních klíčových slov:

```csharp
public class DoubaoVoiceConfig
{
    /// <summary>
    /// App ID
    /// </summary>
    public string AppId { get; set; } = string.Empty;

    /// <summary>
    /// Access token
    /// </summary>
    public string AccessToken { get; set; } = string.Empty;

    /// <summary>
    /// Service URL
    /// </summary>
    public string ServiceUrl { get; set; } = string.Empty;

    /// <summary>
    /// Custom hotword context list
    /// </summary>
    public List<string>? HotwordContexts { get; set; }

    /// <summary>
    /// Doubao platform hotword table ID
    /// </summary>
    public string? BoostingTableId { get; set; }
}
```

Design této třídy konfigurace navazuje na obvyklý stručný styl HagiCode. `HotwordContexts` je typ seznamu s povolenou hodnotou null a `BoostingTableId` je řetězec s možnou hodnotou null, takže když neexistuje žádná konfigurace klíčového slova, tyto vlastnosti nemají na požadavek vůbec žádný vliv. Pokud tuto funkci nepoužíváte, měla by zůstat stranou.

### Logika konstrukce užitečného zatížení

Konstrukce užitečného zatížení je jádrem celé funkce klíčového slova. Jakmile máme konfiguraci klíčového slova, musíme jej naformátovat do struktury JSON požadované rozhraním Doubao API. K tomuto procesu dojde předtím, než SDK odešle požadavek:

```csharp
private void AddCorpusToRequest(Dictionary<string, object> request)
{
    var corpus = new Dictionary<string, object>();

    // Add custom hotwords
    if (Config.HotwordContexts != null && Config.HotwordContexts.Count > 0)
    {
        corpus["context"] = new Dictionary<string, object>
        {
            ["context_type"] = "dialog_ctx",
            ["context_data"] = Config.HotwordContexts
                .Select(text => new Dictionary<string, object> { ["text"] = text })
                .ToList()
        };
    }

    // Add platform hotword table ID
    if (!string.IsNullOrEmpty(Config.BoostingTableId))
    {
        corpus["boosting_table_id"] = Config.BoostingTableId;
    }

    // Add corpus to the request only when it is not empty
    if (corpus.Count > 0)
    {
        request["corpus"] = corpus;
    }
}
```

Tento kód ukazuje, jak dynamicky konstruovat `corpus` pole na základě konfigurace. Klíčovým bodem je, že přidáme `corpus` pole pouze v případě, že konfigurace klíčového slova skutečně existuje. Tento design zajišťuje zpětnou kompatibilitu: když nejsou nakonfigurována žádná klíčová slova, struktura požadavku zůstává přesně stejná jako dříve. Na zpětné kompatibilitě záleží; přidání funkce by nemělo narušit stávající logiku.

### Předávání parametrů WebSocket

Mezi frontendem a backendem jsou parametry klíčových slov předávány prostřednictvím řídicích zpráv WebSocket. HagiCode je navržen tak, že když frontend začne nahrávat, načte konfiguraci klíčového slova z `localStorage` a odešle jej do backendu prostřednictvím zprávy WebSocket.

```typescript
const controlMessage = {
  type: 'control',
  payload: {
    command: 'StartRecognition',
    contextText: '高血压\n糖尿病\n冠心病',
    boosting_table_id: 'medical_table',
    combineMode: false
  }
};
```

Zde je třeba poznamenat jeden detail: frontend předává víceřádkový text oddělený znaky nového řádku a backend jej potřebuje analyzovat. Backendový obslužný program WebSocket analyzuje tyto parametry a předá je SDK:

```csharp
private async Task HandleControlMessageAsync(
    string connectionId,
    DoubaoSession session,
    ControlMessage message)
{
    if (message.Payload is SessionControlRequest controlRequest)
    {
        // Parse hotword parameters
        string? contextText = controlRequest.ContextText;
        string? boostingTableId = controlRequest.BoostingTableId;
        bool? combineMode = controlRequest.CombineMode;

        // Parse multi-line text into a hotword list
        if (!string.IsNullOrEmpty(contextText))
        {
            var hotwords = contextText
                .Split('\n', StringSplitOptions.RemoveEmptyEntries)
                .Select(s => s.Trim())
                .Where(s => s.Length > 0)
                .ToList();

            session.HotwordContexts = hotwords;
        }

        session.BoostingTableId = boostingTableId;
    }
}
```

Díky tomuto návrhu je předávání konfigurace klíčových slov z frontendu do backendu jasné a efektivní. Není na tom nic zvlášť tajemného; data jednoduše procházejí vrstvou po vrstvě.

## Praktický průvodce

### Konfigurace vlastních klíčových slov

V reálném použití je konfigurace vlastních klíčových slov velmi jednoduchá. Otevřete stránku nastavení rozpoznávání řeči v HagiCode a najděte sekci "Konfigurace klíčových slov". Do vstupního pole „Text vlastního klíčového slova“ zadejte jednu frázi klíčového slova na řádek.

Pokud například vyvíjíte lékařskou aplikaci, můžete ji nakonfigurovat takto:

```
高血压
糖尿病
冠心病
心绞痛
心肌梗死
心力衰竭
```

Po uložení konfigurace jsou tato klíčová slova automaticky předána rozhraní API Doubao při každém spuštění rozpoznávání řeči. V našich testech se po nakonfigurování klíčových slov znatelně zlepšila přesnost rozpoznávání souvisejících odborných výrazů. Zlepšení je skutečné a jednoznačně lepší než dříve.

### Nakonfigurujte tabulku klíčových slov platformy

Pokud potřebujete spravovat velký počet klíčových slov nebo pokud klíčová slova vyžadují časté aktualizace, je vhodnější režim tabulky klíčových slov platformy. Nejprve vytvořte tabulku klíčových slov na samoučící platformě Doubao a získejte vygenerovanou `boosting_table_id`a poté zadejte toto ID na stránce nastavení HagiCode.

Samoučící se platforma Doubao poskytuje funkce, jako je hromadný import a kategorizovaná správa klíčových slov, což je velmi praktické pro týmy, které potřebují spravovat velké sady specializované terminologie. Správou klíčových slov na platformě je můžete spravovat centrálně a konzistentně zavádět aktualizace. Jakmile se seznam klíčových slov zvětší, mít jedno místo pro jeho správu je mnohem praktičtější než pokaždé ruční zadávání.

### Použití kombinovaného režimu

V některých složitých scénářích může být nutné použít vlastní klíčová slova i tabulku klíčových slov platformy současně. V takovém případě jednoduše nakonfigurujte obojí v HagiCode a povolte přepínač „Kombinovaný režim“.

V kombinovaném režimu Doubao API bere v úvahu oba zdroje klíčových slov současně, takže přesnost rozpoznávání je obvykle vyšší než při použití jednoho zdroje samostatně. Je však třeba poznamenat, že kombinovaný režim zvyšuje složitost požadavků, takže je nejlepší se rozhodnout, zda jej povolit po praktickém testování. Větší složitost stojí za to, pouze pokud to ospravedlňují výsledky v reálném světě.

## Věci, které je třeba mít na paměti

Existuje několik bodů, které si zaslouží zvláštní pozornost při implementaci a používání funkce klíčových slov.

První je limit počtu znaků. Doubao API má přísná omezení na klíčová slova, včetně počtu řádků, znaků na řádek a celkového počtu znaků. Pokud je překročen jakýkoli limit, API vrátí chybu. V implementaci frontendu HagiCode kontrolujeme tato omezení během uživatelského vstupu prostřednictvím ověřovacích funkcí, což zabraňuje odesílání neplatných konfigurací do backendu. Včasné zachycení problémů je vždy lepší než čekat na selhání API.

Druhým je formát `boosting_table_id`. Toto pole povoluje pouze písmena, čísla, podtržítka a spojovníky a nemůže obsahovat mezery ani jiné speciální znaky. Při vytváření tabulky klíčových slov na samoučící platformě Doubao nezapomeňte dodržovat pravidla pro pojmenování. Tento druh přísného ověřování formátu je pro API běžný.

Třetí je zpětná kompatibilita. Parametry klíčových slov jsou zcela volitelné. Pokud nejsou nakonfigurována žádná klíčová slova, systém se chová přesně jako předtím. Tento design zajišťuje, že stávající uživatelé nejsou žádným způsobem ovlivněni, a také usnadňuje postupnou migraci a upgrady. Přidání funkce by nemělo narušit předchozí logiku.

Nakonec je zde řešení chyb. Když je konfigurace klíčového slova neplatná, rozhraní API Doubao vrátí odpovídající chybové zprávy. Implementace HagiCode zaznamenává podrobné protokoly, které vývojářům pomáhají řešit problémy. Frontend zároveň zobrazuje chyby ověření v uživatelském rozhraní, aby uživatelům pomohl opravit konfiguraci. Dobré zpracování chyb přirozeně vede k lepší uživatelské zkušenosti.

## Závěr

Prostřednictvím tohoto článku jsme poskytli podrobný úvod do kompletního řešení pro implementaci klíčových slov pro rozpoznávání řeči Doubao v projektu HagiCode. Toto řešení pokrývá celý proces od analýzy požadavků a technického výběru až po implementaci kódu a poskytuje vývojářům praktický příklad, který mohou použít jako referenci.

Klíčové body lze shrnout následovně. Za prvé, Doubao API podporuje jak vlastní klíčová slova, tak tabulky klíčových slov platformy a lze je používat samostatně nebo v kombinaci. Za druhé, frontend používá `localStorage` uložit konfiguraci jednoduchým a efektivním způsobem. Za třetí, backend předává parametry klíčových slov dynamickou konstrukcí `corpus` pole při zachování silné zpětné kompatibility. Za čtvrté, komplexní logika ověřování zajišťuje správnost konfigurace a zabraňuje neplatným požadavkům. Celkově není řešení složité; prostě pečlivě dodržuje požadavky API.

Implementace funkce klíčových slov dále posiluje schopnosti HagiCode v oblasti rozpoznávání řeči. Flexibilní konfigurací odborných termínů souvisejících s podnikáním mohou vývojáři pomoci systému rozpoznávání řeči lépe porozumět obsahu z konkrétních domén, a proto poskytovat přesnější služby. Technologie by nakonec měla sloužit skutečným obchodním potřebám a řešení praktických problémů je tím nejdůležitějším.
