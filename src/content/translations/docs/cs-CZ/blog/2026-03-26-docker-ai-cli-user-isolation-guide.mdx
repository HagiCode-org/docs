## Spouštění nástrojů AI CLI v kontejnerech Docker: Praktický průvodce izolací uživatelů a trvalými svazky

> Integrace nástrojů pro kódování AI, jako jsou Claude Code, Codex a OpenCode, do kontejnerových prostředí zní jednoduše, ale všude jsou skryté složitosti. Tento článek se hluboce ponoří do toho, jak projekt HagiCode řeší základní problémy při nasazení Dockeru, včetně uživatelských oprávnění, zachování konfigurace a správy verzí, abyste se mohli vyhnout běžným nástrahám.

## Pozadí

Když jsme se rozhodli spustit nástroje CLI pro kódování AI v kontejnerech Docker, nejintuitivnější myšlenka byla pravděpodobně: "Nejsou kontejnery pouze root? Proč nenainstalovat vše přímo a nenazvat to hotové?" Ve skutečnosti tato zdánlivě jednoduchá myšlenka skrývá několik zásadních problémů, které je třeba vyřešit.

Za prvé, **bezpečnostní omezení jsou první překážkou**. Vezměte si Claude CLI jako příklad: výslovně zakazuje spouštění jako uživatel root. Toto je povinná bezpečnostní kontrola a pokud je detekován root, odmítne se spustit. Možná si říkáte, nemohu jednoduše přepnout uživatele pomocí `USER` směrnice? Není to tak jednoduché. Stále existuje problém s mapováním mezi uživatelem bez oprávnění root v kontejneru a uživatelskými oprávněními na hostitelském počítači.

Za druhé, **stálost stavu je druhou pastí**. Claude Code vyžaduje přihlášení, Codex má vlastní konfiguraci a OpenCode má také adresář cache. Pokud musíte při každém restartu kontejneru vše přenastavovat, ztrácí celá myšlenka „automatizace“ smysl. Potřebujeme, aby tyto konfigurace přetrvávaly i po skončení životního cyklu kontejneru.

Třetím problémem je **konzistence oprávnění**. Mohou procesy uvnitř kontejneru přistupovat ke konfiguračním souborům vytvořeným hostitelským uživatelem? Neshody UID/GID často způsobují chyby oprávnění k souboru, což je ve skutečných nasazeních extrémně běžné.

Tyto problémy mohou vypadat nezávisle, ale v praxi spolu úzce souvisí. Během vývoje HagiCode jsme se postupně dopracovali k praktickému řešení. Dále se podělím o technické detaily a poučení z těchto úskalí.

## Proč prostě nemůžeme použít root?

Dochází zde k běžnému nedorozumění: Kontejnery Docker běží ve výchozím nastavení jako root, tak proč neinstalovat nástroje jako root? Pokud si to myslíte, Claude CLI vás rychle naučí opak.

```bash
# Run Claude CLI directly as root? No.
docker run --rm -it --user root myimage claude
# Output: Error: This command cannot be run as root user
```

Toto je tvrdé bezpečnostní omezení v Claude CLI. Důvod je jednoduchý: tyto nástroje CLI čtou a zapisují citlivou uživatelskou konfiguraci, včetně tokenů API, lokálních mezipamětí a dokonce i skriptů napsaných uživatelem. Jejich spouštění s právy root představuje příliš velké riziko.

Otázka tedy zní: jak můžeme uspokojit bezpečnostní požadavky CLI a zároveň zachovat flexibilitu správy kontejnerů? Musíme změnit způsob, jakým o tom přemýšlíme: **místo přepínání uživatelů za běhu vytvořte vyhrazeného uživatele během fáze vytváření obrazu**.

## Vytvoření vyhrazeného uživatele: více než jen změna jména

Možná si myslíte, že přidání singlu `USER` stačí řádek do Dockerfile. To je skutečně nejjednodušší přístup, ale není dostatečně robustní.

### Statické vytváření vs. dynamické mapování

Přístup HagiCode je vytvořit a `hagicode` uživatel s UID 1000, který obvykle odpovídá výchozímu uživateli na většině hostitelských počítačů:

```dockerfile
RUN groupadd -o -g 1000 hagicode && \
    useradd -o -u 1000 -g 1000 -s /bin/bash -m hagicode && \
    mkdir -p /home/hagicode/.claude && \
    chown -R hagicode:hagicode /home/hagicode
```

To ale řeší pouze vestavěný uživatel uvnitř obrázku. Co když je hostitelský uživatel UID 1001? Při spuštění kontejneru musíte stále podporovat dynamické mapování.

docker-entrypoint.sh obsahuje klíčovou logiku:

```bash
if [ -n "$PUID" ] && [ -n "$PGID" ]; then
    if ! id hagicode >/dev/null 2>&1; then
        groupadd -g "$PGID" hagicode
        useradd -u "$PUID" -g "$PGID" -s /bin/bash -m hagicode
    fi
fi
```

Výhoda tohoto návrhu je zřejmá: **použijte výchozí UID 1000 při vytváření obrazu a poté dynamicky upravte za běhu prostřednictvím `PUID` a `PGID` proměnné prostředí**. Bez ohledu na to, jaké UID má hostitelský uživatel, vlastnictví konfiguračních souborů zůstává správné.

## Filozofie designu trvalých objemů

Každý nástroj AI CLI má svůj vlastní preferovaný konfigurační adresář, takže je třeba je namapovat jeden po druhém:

| Nástroj CLI | Cesta v kontejneru | Jmenovaný svazek |
|---------|-----------|--------|
| Claude | `/home/hagicode/.claude` | `claude-data` |
| Codex | `/home/hagicode/.codex` | `codex-data` |
| OpenCode | `/home/hagicode/.config/opencode` | `opencode-config-data` |

Proč používat pojmenované svazky místo připojení připojení? Tři důvody:

1. **Jednodušší správa**: Pojmenované svazky jsou spravovány automaticky pomocí Dockeru, takže hostitelské adresáře nemusíte vytvářet ručně.
2. **Izolace oprávnění**: Počáteční obsah svazků vytváří uživatel uvnitř kontejneru, čímž se zabrání konfliktům oprávnění s hostitelem.
3. **Nezávislá migrace**: Svazky mohou existovat nezávisle na kontejnerech, takže při upgradu obrazů nedochází ke ztrátě dat.

docker-compose-builder-web automaticky vygeneruje odpovídající konfiguraci svazku:

```yaml
volumes:
  claude-data:
  codex-data:
  opencode-config-data:

services:
  hagicode:
    volumes:
      - claude-data:/home/hagicode/.claude
      - codex-data:/home/hagicode/.codex
      - opencode-config-data:/home/hagicode/.config/opencode
    user: "${PUID:-1000}:${PGID:-1000}"
```

Věnujte pozornost `user` pole zde: `PUID` a `PGID` jsou vkládány prostřednictvím proměnných prostředí, aby bylo zajištěno, že procesy uvnitř kontejneru běží s identitou, která odpovídá hostitelskému uživateli. Na tomto detailu záleží, protože problémy s oprávněními se ladí, jakmile se objeví.

## Správa verzí: zapracované verze s přepisy za běhu

Připnutí verzí obrázků Dockeru je zásadní pro reprodukovatelnost. V reálném vývoji ale často potřebujeme otestovat novější verzi nebo urychleně opravit chybu. Pokud bychom museli obraz pokaždé znovu sestavit, pracovní postup by byl příliš neefektivní.

Strategií HagiCode jsou **pevné verze jako výchozí, s přepisy za běhu jako mechanismem rozšíření**. Je to pragmatický technický kompromis mezi stabilitou a flexibilitou.

Verze pinů Dockerfile.template zde:

```dockerfile
USER hagicode
WORKDIR /home/hagicode

# Configure the global npm install path
RUN mkdir -p /home/hagicode/.npm-global && \
    npm config set prefix '/home/hagicode/.npm-global'

# Install CLI tools using pinned versions
RUN npm install -g @anthropic-ai/claude-code@2.1.71 && \
    npm install -g @openai/codex@0.112.0 && \
    npm install -g opencode-ai@1.2.25 && \
    npm cache clean --force
```

docker-entrypoint.sh podporuje přepsání za běhu:

```bash
install_cli_override_if_needed() {
    local package_name="$2"
    local override_version="$5"

    if [ -n "$override_version" ]; then
        gosu hagicode npm install -g "${package_name}@${override_version}"
    fi
}

# Example usage
install_cli_override_if_needed "" "@anthropic-ai/claude-code" "" "" "${CLAUDE_CODE_CLI_VERSION}"
```

To vám umožní otestovat novou verzi prostřednictvím proměnné prostředí bez opětovného sestavení obrazu:

```bash
docker run -e CLAUDE_CODE_CLI_VERSION=2.2.0 myimage
```

Tento design je praktický, protože nikdo nechce znovu vytvářet obraz pokaždé, když testuje novou funkci.

## Automatická konfigurace konfigurace

Kromě ruční konfigurace nástrojů CLI vyžadují některé scénáře automatické vložení konfigurace. Nejtypičtějším příkladem je token API.

```bash
if [ -n "$ANTHROPIC_AUTH_TOKEN" ]; then
    mkdir -p /home/hagicode/.claude
    cat > /home/hagicode/.claude/settings.json <<EOF
{
  "env": {
    "ANTHROPIC_AUTH_TOKEN": "${ANTHROPIC_AUTH_TOKEN}"
  }
}
EOF
    chown -R hagicode:hagicode /home/hagicode/.claude
fi
```

Zde záleží na dvou věcech: **předávejte citlivé informace prostřednictvím proměnných prostředí namísto jejich pevného kódování do obrazu** a ujistěte se, že vlastnictví konfiguračních souborů je nastaveno správně, jinak je nástroje CLI nebudou schopny číst.

## Osvědčené postupy a kontrolní seznam nástrah

### Problémy s nesouladem oprávnění

To je ta nejsnadnější past, do které lze spadnout. Hostitel má UID 1001, zatímco kontejner používá 1000, takže k souborům vytvořeným na jedné straně nelze přistupovat na straně druhé.

```bash
# Correct approach: make the container match the host user
docker run \
    -e PUID=$(id -u) \
    -e PGID=$(id -g) \
    myimage
```

Tento problém je velmi častý a může být frustrující, když se s ním poprvé setkáte.

### Konfigurace zmizí po restartu kontejneru

Pokud zjistíte, že se po každém restartu znovu přihlašujete, zkontrolujte, zda jste nezapomněli připojit trvalý svazek:

```yaml
volumes:
  - claude-data:/home/hagicode/.claude
```

Nic není více frustrující než pečlivé nastavení konfigurace, abyste viděli, jak zmizí.

### Správný způsob upgradu verzí

Neutíkej `npm install -g` přímo uvnitř běžícího kontejneru. Správné přístupy jsou:

1. Nastavte proměnnou prostředí, která spustí přepisovací instalaci.
2. Nebo předělat obrázek.

```bash
# Option 1: runtime override
docker run -e CLAUDE_CODE_CLI_VERSION=2.2.0 myimage

# Option 2: rebuild the image
docker build -t myimage:v2 .
```

Do Říma vede více cest, ale některé jsou hladší než jiné.

### Kontrolní seznam zabezpečení

- Předávejte tokeny API prostřednictvím proměnných prostředí, místo abyste je zapisovali do obrazu.
- Nastavte oprávnění konfiguračního souboru na `600`.
- Aplikaci vždy spouštějte jako uživatel bez oprávnění root.
- Pravidelně aktualizujte verze rozhraní CLI, abyste opravili chyby zabezpečení.

Bezpečnost je vždy důležitá, ale skutečnou výzvou je její důsledné prosazování v praxi.

## Rozšíření podpory pro nové nástroje CLI

Pokud chcete v budoucnu podporovat nový nástroj CLI, existují pouze tři kroky:

1. **Dockerfile.template**: přidejte krok instalace.
2. **docker-entrypoint.sh**: přidat logiku přepsání verze.
3. **docker-compose-builder-web**: přidejte trvalé mapování svazků.

## Závěr

Spouštění nástrojů AI CLI v kontejnerech Docker zahrnuje tři hlavní výzvy: **uživatelská oprávnění, stálost konfigurace a správa verzí**. Kombinací vyhrazených uživatelů, izolace pojmenovaného svazku a přepsání na základě proměnných prostředí vytvořil projekt HagiCode architekturu nasazení, která je bezpečná i flexibilní.

Klíčové body designu:

- **Izolace uživatelů**: Vytvořte vyhrazeného uživatele během fáze vytváření bitové kopie s podporou dynamiky za běhu `PUID`/`PGID` mapování.
- **Strategie perzistence**: Každý nástroj CLI získá svůj vlastní pojmenovaný svazek, takže restarty neovlivní konfiguraci.
- **Flexibilita verzí**: Pevné výchozí hodnoty zajišťují reprodukovatelnost, zatímco přepisy za běhu poskytují prostor pro testování.
- **Automatická konfigurace**: Citlivou konfiguraci lze vložit automaticky prostřednictvím proměnných prostředí.

Toto řešení již nějakou dobu stabilně běží v projektu HagiCode a doufám, že nabízí užitečné referenční body pro vývojáře s podobnými potřebami.
