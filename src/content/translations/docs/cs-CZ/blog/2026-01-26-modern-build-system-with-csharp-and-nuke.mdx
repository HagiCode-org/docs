## Goodbye Script Hell: Proč jsme zvolili C# pro moderní sestavovací systém

> Hluboký ponor do toho, jak projekt HagiCode využívá Nuke k vytvoření typově bezpečného, multiplatformního a vysoce rozšiřitelného automatizovaného sestavení kanálu, který plně řeší problémy s údržbou tradičních sestavení skriptů.

## Pozadí

Během dlouhé cesty vývoje softwaru slovo „budovat“ inspiruje lásku i frustraci. Láska, protože jedním kliknutím se kód promění v produkt, a to je jeden z nejuspokojivějších okamžiků v programování. Frustrace, protože udržovat hromadu chaotických sestavovacích skriptů může být noční můra.

V mnoha projektech jsme zvyklí psát skripty v Pythonu nebo se spoléhat na XML konfigurační soubory. Jen si představte ten strach z toho, že vám vládne `<property>` bloky. Ale jak roste složitost projektu, zejména v projektu jako HagiCode, který zahrnuje frontendovou a backendovou práci, více platforem a více jazyků, tradiční přístupy k sestavování začínají ukazovat své limity. Logika rozptýleného skriptu, žádná kontrola typu, slabá podpora IDE... každý z těchto problémů se stává malým úskalím, které čas od času zakopne tým.

K vyřešení těchto bolestivých bodů představil projekt HagiCode **Nuke**, moderní sestavovací systém založený na C#. Není to jen nástroj. Je to jiný způsob uvažování o pracovních postupech vytváření. Dnes se podíváme na to, proč jsme si jej vybrali a jak dramaticky zlepšil naše zkušenosti s vývojem.

## Základní analýza: Proč Nuke?

Možná se ptáte: "Už existuje tolik systémů sestavení, jako jsou Make, Gradle nebo dokonce prosté skripty Shell. Proč se vyhýbat používání skriptů založených na C#?"

To je správná otázka. Základní přitažlivost Nuke spočívá v tom, že přináší funkce programovacího jazyka, které již dobře známe, do světa sestavovacích skriptů.

### 1. Modularizace pracovních postupů sestavování: Umění cílů

Nuke má velmi jasnou filozofii designu: **všechno je cíl**.

V tradičních skriptech je snadné skončit se stovkami řádků lineární logiky provádění zamotaných dohromady. V Nuke rozdělujeme proces sestavení na nezávislý `Target`s. Každý cíl dělá přesně jednu věc, například:

- `Clean`: vyčistit výstupní adresář
- `Restore`: obnovení závislých balíčků
- `Compile`: zkompilujte kód
- `Test`: spustit testy jednotek

Tento návrh je velmi dobře v souladu s principem jediné odpovědnosti. Stejně jako stavební kameny lze tyto cíle libovolně kombinovat. Ještě důležitější je, že nám Nuke umožňuje definovat závislosti mezi cíli. Pokud chcete například běhat `Test`, systém automaticky zkontroluje, zda `Compile` již byla provedena. Pokud chcete `Compile`, pak `Restore` přirozeně na prvním místě.

Tento graf závislosti dělá logiku mnohem jasnější a výrazně zlepšuje efektivitu provádění, protože Nuke dokáže automaticky analyzovat optimální cestu provádění.

### 2. Bezpečnost typu: Sbohem noční můře překlepů

Každý, kdo psal sestavení skriptů v Pythonu, se pravděpodobně dostal do této nepříjemné situace: skript běží pět minut a pak selže, protože `Confi.guration` byl chybně napsán nebo protože byl předán řetězec tam, kde byl očekáván číselný parametr.

Největší výhodou psaní sestavovacích skriptů v C# je **bezpečnost typu**. To znamená:

- **Kontrola během kompilace**: IDE vám může říct, co je špatně, když píšete, místo toho, abyste museli čekat až do běhu.
- **Bezpečný refaktoring**: pokud chcete přejmenovat proměnnou nebo metodu, IDE to zvládne v jednom kroku, aniž by vás nutilo provádět riskantní globální vyhledávání a nahrazování.
- **Inteligentní dokončování**: výkonný IntelliSense dokáže dokončit kód za vás automaticky, takže nemusíte neustále listovat dokumentací, abyste si pamatovali obskurní API.

### 3. Podpora napříč platformami: Jedno konzistentní sestavení

V minulosti jsme si psali `.bat` na Windows a `.sh` na Linuxu a pak možná přidal skript Python, aby překlenul mezeru. Nyní, kdekoli může běžet .NET Core nebo dnes .NET 5+, může běžet i Nuke.

To znamená, že členové týmu mohou používat Windows, Linux nebo macOS, a ať už dávají přednost Visual Studiu, VS Code nebo Rideru, všichni vykonávají přesně stejnou logiku. To výrazně snižuje problémy způsobené rozdíly v prostředí, zejména klasický problém „funguje to na mém počítači“.

### 4. Parametry a správa konfigurace

Nuke poskytuje velmi elegantní mechanismus analýzy parametrů. Nemusíte ručně analyzovat `string[] args`. Jednoduše definujete vlastnost a označíte ji `[Parameter]`a Nuke automaticky zpracovává mapování z argumentů příkazového řádku a konfiguračních souborů.

Můžeme například velmi snadno definovat konfiguraci sestavení:

```csharp
[Parameter("Configuration to build - Default is 'Debug'")]
readonly Configuration BuildConfiguration = IsLocalBuild ? Configuration.Debug : Configuration.Release;

Target Compile => _ => _
    .DependsOn(Restore)
    .Executes(() =>
    {
        // Use BuildConfiguration here; it is type-safe
        DotNetBuild(s => s
            .SetConfiguration(BuildConfiguration)
            .SetProjectFile(SolutionFile));
    });
```

Tento styl je intuitivní a odolný vůči chybám.

## Praktický průvodce: Jak to uvést do praxe v projektu

Samotná teorie nás nikam nedostane. Podívejme se, jak jsme tento přístup vlastně aplikovali v projektu HagiCode.

### 1. Naplánujte strukturu projektu

Nechtěli jsme vytvářet skripty, které by zaplňovaly kořen projektu, a také jsme nechtěli adresářovou strukturu tak hlubokou, aby připomínala určité Java projekty. Takže jsme umístili všechny soubory sestavení související s Nuke do jednoho `nukeBuild/` adresář.

Výhody jsou přímočaré:

- Kořen projektu zůstane čistý.
- Logika sestavení zůstává soudržná a snáze se spravuje.
- Noví členové týmu mohou okamžitě vidět: "Tady žije logika související se stavbou."

### 2. Navrhněte jasný cílový závislostní řetězec

Při navrhování cílů jsme se řídili jedním principem: **atomicita plus tok závislostí**.

Každý cíl by měl být dostatečně malý, aby dělal přesně jednu věc. např. `Clean` by měl pouze odstraňovat soubory a neměl by tiše manipulovat s obaly na straně.

Doporučený tok závislosti vypadá zhruba takto:

`Clean` -> `Restore` -> `Compile` -> `Test` -> `Pack`

Samozřejmě to není absolutní. Například, pokud chcete pouze spouštět testy a nechcete nic balit, Nuke vám umožní provést `nuke Test` přímo a automaticky zpracuje požadované `Restore` a `Compile` nejprve kroky.

### 3. Robustní zpracování chyb a protokolování

Co je nejvíce frustrující na sestavovacích skriptech? Nejasné chybové zprávy. Nepovedené sestavení, které zaznamenává pouze „Chyba: 1“, stačí k tomu, aby se někdo zbláznil.

V Nuke, protože můžeme přímo používat zpracování výjimek C#, můžeme zachytit a hlásit chyby s mnohem větší přesností.

```csharp
Target Publish => _ => _
    .DependsOn(Test)
    .Executes(() =>
    {
        try 
        {
            // Try publishing to NuGet
            DotNetNuGetPush(s => s
                .SetTargetPath(ArtifactPath)
                .SetSource("https://api.nuget.org/v3/index.json")
                .SetApiKey(ApiKey));
        }
        catch (Exception ex)
        {
            Log.Error($"Publishing failed. Team, please check whether the key is correct: {ex.Message}");
            throw; // Make sure the build process exits with a non-zero exit code
        }
    });
```

### 4. Použijte testy k ochraně kvality

Sestavovací skripty jsou také kód, což znamená, že také potřebují testy. Nuke nám umožňuje psát testy pro pracovní postupy sestavení, takže když změníme logiku sestavení, náhodně neporušíme stávající procesy vydání. To je zvláště důležité u kontinuálních integračních potrubí.

## Shrnutí

Zavedením Nuke se proces sestavování HagiCode stal plynulejším než kdykoli předtím. Nešlo jen o výměnu nářadí. Byl to upgrade v inženýrském myšlení.

**Co jsme získali?**

- **Udržovatelnost**: konfigurace jako kód, jasnější logika a rychlejší cesta pro nové přispěvatele.
- **Stabilita**: silné psaní eliminuje více než 90 % chyb na nízké úrovni.
- **Konzistence**: sjednocené prostředí napříč platformami, které odstraňuje posun prostředí.

Pokud vám psaní skriptů pro sestavení připadalo jako „cítit se ve tmě“, pak vám používání Nuke připadalo jako „chodit v noci se zapnutými světly“. Pokud vás nebaví udržovat obtížně laditelné skripty, zvažte přesunutí logiky sestavení do světa C#. Možná zjistíte, že i stavby mohou být elegantní.
