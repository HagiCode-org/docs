## Rychlá úprava kódu v prohlížeči: Webová integrace VSCode v praxi

> Když AI dokončí analýzu kódu, jak okamžitě otevřete editor v prohlížeči a začnete provádět změny? Tento článek sdílí naše praktické zkušenosti s integrací kódového serveru do projektu HagiCode, abychom vytvořili bezproblémový most mezi asistentem AI a úpravou kódu.

## Pozadí

V éře programování podporovaného umělou inteligencí potřebují vývojáři často rychle kontrolovat a upravovat kód. Tradiční pracovní postup je jednoduchý: otevřete projekt v desktopovém IDE, vyhledejte soubor, upravte jej a uložte. Ale v některých situacích je tento tok vždy trochu mimo.

První scénář: vzdálený vývoj. Při použití asistenta AI, jako je HagiCode, může backend běžet na vzdáleném serveru nebo uvnitř kontejneru a místní počítače nemají přímý přístup k souborům projektu. Pokaždé, když potřebujete zkontrolovat nebo upravit kód, musíte se připojit přes SSH nebo jinou metodu a zážitek je roztříštěný. Je to jako chtít se s někým setkat přes tlustou tabuli skla: můžete ho vidět, ale nemůžete na něj dosáhnout.

Druhý scénář: rychlé náhledy. Poté, co asistent AI analyzuje kód, může uživatel chtít pouze rychle procházet soubor nebo provést malou změnu. Spuštění plného desktopového IDE se zdá těžké, zatímco lehký editor v prohlížeči lépe vyhovuje potřebě „rychlého vzhledu“. Koneckonců, kdo chce mobilizovat celý nástroj, jen aby se podíval?

Třetí scénář: spolupráce mezi zařízeními. Při práci na různých zařízeních poskytuje editor založený na prohlížeči jednotný přístupový bod, aniž by bylo nutné na každém počítači konfigurovat vývojové prostředí. To samo o sobě ušetří spoustu problémů. Život je krátký; proč opakovat stejné nastavení znovu a znovu?

K vyřešení těchto bolestivých bodů jsme integrovali VSCode Web do projektu HagiCode. To umožňuje bezproblémové propojení asistenta AI a úpravy kódu: poté, co AI analyzuje kód, mohou uživatelé okamžitě otevřít editor a provádět změny ve stejné relaci prohlížeče, aniž by museli přepínat aplikace. Je to ten typ zážitku, kdy, když ho potřebujete, prostě tam je.

## O HagiCode

Řešení sdílené v tomto článku pochází z našich praktických zkušeností v [HagiCode](https://hagicode.com) projekt. HagiCode je asistent kódování řízený umělou inteligencí navržený ke zlepšení efektivity vývoje prostřednictvím interakce přirozeného jazyka. Během vývoje jsme zjistili, že uživatelé často potřebují rychle přepínat mezi analýzou AI a úpravou kódu, což nás přimělo prozkoumat, jak integrovat editor přímo do prohlížeče.

Projektové úložiště: [github.com/HagiCode-org/site](https://github.com/HagiCode-org/site)

## Technická volba: Proč kódový server?

Mezi mnoha dostupnými webovými řešeními VSCode jsme vybrali kódový server. Za tímto rozhodnutím stálo několik konkrétních důvodů.

Úplnost funkce. code-server je webová verze VSCode a podporuje většinu funkcí desktopu, včetně systému rozšíření, inteligentních návrhů, ladění a dalších. To znamená, že uživatelé mohou získat zkušenosti s úpravami v prohlížeči, které se velmi blíží verzi pro stolní počítače. Koneckonců, kdo opravdu chce dělat kompromisy ve funkčnosti?

Flexibilní nasazení. code-server může běžet jako nezávislá služba a také podporuje nasazení založené na Dockeru, které dobře zapadá do architektury HagiCode. Náš backend je napsán v C#, frontend používá React a oba komunikují se službou kódového serveru prostřednictvím REST API. Je to jako stavět z kostek: každý kousek má své místo.

Bezpečná autentizace. kódový server obsahuje vestavěný `connection-token` mechanismus zabraňující neoprávněnému přístupu. Každá relace má jedinečný token, takže editor mohou otevřít pouze oprávnění uživatelé. Bezpečnost je jednou z věcí, kterou plně oceníte, až když ji máte.

## Architektura Design

Webová integrace HagiCode VSCode využívá architekturu oddělenou front-end/back-end.

### Frontend Service Layer

Frontend propojuje interakce s backendem `vscodeServerService.ts`:

```typescript
// Open project
export async function openProjectInCodeServer(
  id: string,
  currentInterfaceLanguage?: string,
): Promise<VsCodeServerLaunchResponseDto>

// Open vault
export async function openVaultInCodeServer(
  id: string,
  path?: string,
  currentInterfaceLanguage?: string,
): Promise<VsCodeServerLaunchResponseDto>
```

Rozdíl mezi těmito dvěma metodami je jasný: `openProjectInCodeServer` otevře celý projekt, zatímco `openVaultInCodeServer` otevře konkrétní cestu uvnitř trezoru. U projektů s více úložišti MonoSpecs systém automaticky vytvoří soubor pracovního prostoru. Jasná odpovědnost často stačí, když každá část dělá dobře svou vlastní práci.

### Backend Service Layer

Backend `VaultAppService.cs` implementuje základní logiku:

```csharp
public async Task<VsCodeServerLaunchResponseDto> OpenInCodeServerAsync(
    string id,
    string? relativePath = null,
    string? currentInterfaceLanguage = null,
    CancellationToken cancellationToken = default)
{
    // 1. Get settings and check whether the feature is enabled
    var settings = await _vsCodeServerSettingsService.GetResolvedSettingsAsync(cancellationToken);
    if (!settings.Enabled) {
        throw new BusinessException(VsCodeServerErrorCodes.Disabled, "VSCode Server is disabled.");
    }

    // 2. Get vault and resolve the launch directory
    var vault = await RequireVaultAsync(id, cancellationToken);
    var launchDirectory = ResolveLaunchDirectory(vault, relativePath);

    // 3. Ensure code-server is running and get runtime info
    var runtime = await _vsCodeServerManager.EnsureStartedAsync(settings, cancellationToken);

    // 4. Resolve language settings
    var language = _vsCodeServerSettingsService.ResolveLaunchLanguage(
        settings.Language,
        currentInterfaceLanguage);

    // 5. Build launch URL
    return new VsCodeServerLaunchResponseDto {
        LaunchUrl = AppendQueryString(runtime.BaseUrl, new Dictionary<string, string?> {
            ["folder"] = launchDirectory,
            ["tkn"] = runtime.ConnectionToken,
            ["vscode-lang"] = language,
        }),
        ConnectionToken = runtime.ConnectionToken,
        OpenMode = "folder",
        Runtime = VsCodeServerSettingsService.MapRuntime(
            await _vsCodeServerManager.GetRuntimeSnapshotAsync(cancellationToken)),
    };
}
```

Tato metoda má velmi jasnou odpovědnost: zkontrolovat nastavení, vyřešit cesty, spustit službu a vytvořit adresu URL. Mezi nimi, `ResolveLaunchDirectory` metoda provádí kontroly zabezpečení cesty, aby se zabránilo útokům s přecházením cesty. Kód může působit trochu jako poezie, když každý řádek má svůj účel.

### Automatická správa běhu

Backend řídí proces kódového serveru `VsCodeServerManager`:

- Zkontrolujte stav procesu
- Automaticky spustit zastavené služby
- Vraťte snímky za běhu, jako je port, ID procesu a čas spuštění

Tento návrh umožňuje systému automaticky zpracovávat životní cyklus kódového serveru, takže uživatelé nemusí řídit servisní procesy ručně. Život je už dost komplikovaný; vše, co lze automatizovat, by mělo být.

### Synchronizace jazyka

HagiCode podporuje vícejazyčné rozhraní a kódový server musí toto nastavení dodržovat. Systém podporuje tři jazykové režimy:

- `follow`: postupujte podle aktuálního jazyka rozhraní
- `zh-CN`: opraveno na čínštinu
- `en-US`: opraveno na angličtinu

Nastavení je předáno kódovému serveru přes `vscode-lang` Parametr URL, aby jazyk editoru zůstal konzistentní s rozhraním HagiCode. Jazyk se cítí nejlépe, když je jednotný.

### MonoSpecs Multi-Repository Workspace

Pro projekty MonoSpecs, které obsahují více dílčích úložišť uvnitř jednoho monorepa, systém automaticky vytvoří `.code-workspace` soubor:

```csharp
private async Task<string> CreateWorkspaceFileAsync(Project project, Guid projectId)
{
    var folders = await ResolveWorkspaceFoldersAsync(project.Path);
    var workspaceDocument = new {
        folders = folders.Select(path => new { path }).ToArray(),
    };
    // Generate workspace file...
}
```

To umožňuje upravovat více dílčích repozitářů ve stejné instanci kódového serveru, což je praktické zejména u velkých monorepo projektů. Více úložišť v jednom okně může působit jako více příběhů shromážděných ve stejné knize.

## Integrace frontendu

Frontend HagiCode používá React + TypeScript a integrace kódového serveru není nijak zvlášť složitá.

### Tlačítko rychlé akce

Přidejte tlačítko Kódový server na kartu projektu:

```tsx
// QuickActionsZone.tsx
<Button
  size="sm"
  variant="default"
  onClick={() => onAction({ type: 'open-code-server' })}
>
  <Globe className="h-3 w-3 mr-1" />
  <span className="text-xs">{t('project.openCodeServer')}</span>
</Button>
```

Toto tlačítko spouští akci otevření a volá backend API, aby získalo spouštěcí adresu URL. Jedno tlačítko, jedna akce, přímá a jednoduchá.

### Zpracování otevřené akce

```tsx
const handleAction = async (action: ProjectAction) => {
  if (action.type === 'open-code-server') {
    const response = await openProjectInCodeServer(project.id, i18n.language);
    window.open(response.launchUrl, '_blank', 'noopener,noreferrer');
  }
};
```

Použijte `window.open` otevřete kódový server na nové kartě. The `noopener,noreferrer` parametry poskytují extra bezpečnost. Pokud jde o bezpečnost, neexistuje nic jako být příliš opatrný.

### Položka pro úpravy trezoru

Přidejte podobné tlačítko úprav do seznamu úschovny:

```tsx
const handleEditVault = async (vault: VaultItemDto) => {
  const response = await openVaultInCodeServer(vault.id);
  window.open(response.launchUrl, '_blank', 'noopener,noreferrer');
};
```

Projekty a Vaulty používají stejný otevřený mechanismus, který udržuje interakci konzistentní. Na konzistenci záleží téměř stejně jako na samotné funkci.

## Logika konstrukce URL

Formát URL pro kódový server má několik podrobností, které stojí za zmínku.

**Režim složky**:
```
http://{host}:{port}/?folder={path}&tkn={token}&vscode-lang={lang}
```

**Režim pracovního prostoru**:
```
http://{host}:{port}/?workspace={workspacePath}&tkn={token}&vscode-lang={lang}
```

tady `tkn` je token připojení. Generuje se automaticky při každém spuštění kódového serveru, což zajišťuje bezpečný přístup. The `vscode-lang` parametr řídí jazyk uživatelského rozhraní editoru. Každý z těchto parametrů má svou roli.

## Scénáře použití

### Scénář 1: Kontrola kódu za pomoci umělé inteligence

Uživatel mluví s HagiCode, AI analyzuje kód projektu a najde potenciální problém, a poté uživatel klikne na tlačítko „Otevřít v kódovém serveru“, čímž otevře editor přímo v prohlížeči, prohlédne si postižený soubor, opraví jej a vrátí se do HagiCode, aby mohl pokračovat v konverzaci. Celý tok probíhá v prohlížeči bez přepínání aplikací. Působí hladce jako tekoucí voda.

### Scénář 2: Úprava studijních materiálů v trezoru

Uživatel vytvoří Vault pro studium projektu s otevřeným zdrojovým kódem a chce do něj přidat studijní poznámky `docs/` adresář. Pomocí code-serveru mohou upravovat soubory Markdown přímo v prohlížeči, ukládat je a nechat HagiCode okamžitě číst aktualizované poznámky. To je užitečné zejména pro budování osobní znalostní báze. Znalosti se stávají cennějšími, čím více jich nashromáždíte.

### Scénář 3: Vývoj víceúložišť MonoSpecs

Projekt MonoSpecs obsahuje více dílčích úložišť a kódový server automaticky vytváří vícesložkový pracovní prostor. V prohlížeči mohou uživatelé upravovat kód v několika úložištích najednou a poté zasílat změny zpět do svých příslušných úložišť Git. Tento pracovní postup je zvláště vhodný pro změny, které se musí týkat více úložišť. Úpravy několika úložišť společně vyžadují trochu techniky, stejně jako zpracování více úkolů současně.

## Bezpečnostní aspekty

Při implementaci integrace kód-server si bezpečnost zaslouží zvláštní pozornost. Pokud se zabezpečení pokazí, vždy si toho všimnete příliš pozdě.

### Token připojení

The `connection-token` je generován náhodně a neměl by být vystaven. Nejlépe se používá pod protokolem HTTPS, aby se zabránilo zachycení tokenu mužem ve středu. Citlivé informace se vyplatí náležitě chránit.

### Zabezpečení cesty souboru

Backend implementuje kontroly procházení cesty:

```csharp
private static string ResolveLaunchDirectory(VaultRegistryEntry vault, string? relativePath)
{
    var vaultRoot = EnsureTrailingSeparator(Path.GetFullPath(vault.PhysicalPath));
    var combinedPath = Path.GetFullPath(Path.Combine(vaultRoot, relativePath ?? "."));
    if (!combinedPath.StartsWith(vaultRoot, StringComparison.OrdinalIgnoreCase))
    {
        throw new BusinessException(VaultRelativePathTraversalCode, "Relative path traversal detected.");
    }
    return combinedPath;
}
```

Tento kód zajišťuje, že uživatelé nemohou používat `../` nebo podobné vzory pro přístup k souborům mimo adresář Vault. Hraniční kontroly jsou vždy lepší než je přeskočit.

### Kontrola oprávnění

Proces kódového serveru by měl běžet s příslušnými uživatelskými oprávněními, aby neměl přístup k citlivým systémovým souborům. Nejlepší je spouštět službu kódového serveru pod vyhrazeným uživatelem. Kontrola oprávnění je jedním z těch základů, které byste měli mít vždy na svém místě.

## Optimalizace výkonu

code-server spotřebovává zdroje serveru, takže zde je několik optimalizačních návrhů:

- Sledujte využití procesoru a paměti a v případě potřeby upravte limity zdrojů
- Velké projekty mohou vyžadovat delší časové limity
- Pro uvolnění prostředků implementujte automatické čištění časového limitu relace
- Zvažte ukládání do mezipaměti, abyste omezili opakované výpočty

HagiCode poskytuje rozhraní API pro monitorování stavu běhu a frontend může volat `getVsCodeServerSettings()` pro načtení aktuálního stavu:

```typescript
const { settings, runtime } = await getVsCodeServerSettings();
// runtime.status: 'disabled' | 'stopped' | 'starting' | 'running' | 'unhealthy'
// runtime.baseUrl: "http://localhost:8080"
// runtime.processId: 12345
```

Tento design umožňuje uživatelům jasně porozumět zdravotnímu stavu kódového serveru a rychle lokalizovat problémy, když se něco pokazí. Když je stav viditelný, lidé se cítí více pod kontrolou.

## Podrobnosti uživatelské zkušenosti

Během implementace jsme objevili několik detailů, které znatelně ovlivňují uživatelský zážitek a zaslouží si zvláštní pozornost.

První otevření kódového serveru může vyžadovat čekání na spuštění a toto zpoždění se může pohybovat od několika sekund do půl minuty. Je dobré zobrazit stav načítání na frontendu, aby uživatelé věděli, že systém stále funguje. Čekání je jednodušší, když existuje zpětná vazba.

Prohlížeče mohou vyskakovací okno blokovat, takže uživatelé by měli být vyzváni k ručnímu povolení. Při prvním spuštění HagiCode zobrazí pokyny, které vysvětlují, jak udělit potřebná oprávnění prohlížeče. Uživatelská zkušenost často žije přesně v těchto malých detailech.

Je také dobré zobrazit stav běhu, jako je spuštění, běh nebo chyba, aby uživatelé v případě problémů mohli rychle zjistit, zda je problém na straně serveru nebo v jejich vlastní operaci. Vědět, kde je problém, vám alespoň poskytne místo, kde začít.

## Příklad konfigurace

Konfigurace pro kódový server není složitá:

```json
{
  "vscodeServer": {
    "enabled": true,
    "host": "0.0.0.0",
    "port": 8080,
    "language": "follow"
  }
}
```

`enabled` kontroluje, zda je funkce zapnutá, `host` a `port` definovat naslouchací adresu a `language` nastaví jazykový režim. Tato nastavení lze upravit prostřednictvím uživatelského rozhraní a okamžitě se projeví. Jednoduché věci se často používají nejsnáze.
