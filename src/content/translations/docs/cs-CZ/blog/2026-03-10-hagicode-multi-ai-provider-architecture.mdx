## Praktická architektura poskytovatelů multi-AI na platformě HagiCode

> Tento článek sdílí technický přístup, který jsme použili v rámci architektury Orleans Grain k integraci dvou nástrojů AI, iflow a OpenCode, prostřednictvím jednotného `IAIProvider` a podrobně porovnává implementační rozdíly mezi WebSocket a HTTP komunikací.

:::poznámka[Časový kontext]
Tento článek zaznamenává praxi architektury poskytovatele v době publikace. Aktuální aktivně podporovaný rozsah viz [Instalace AI Agent CLI](/cs-CZ/related-software-installation/ai-agent-cli/) a [Průvodce instalací](/cs-CZ/installation); `IFlowCli` je nyní zachována pouze jako historická hodnota kompatibility a není součástí aktuálního rozsahu aktivní podpory.
:::

## Pozadí

Není na tom nic zvlášť tajemného. Při vytváření HagiCode jsme narazili na velmi praktický problém: uživatelé chtěli pracovat s různými nástroji AI. To není překvapující, protože každý má své vlastní zvyky. Někdo preferuje Claude Code, někdo miluje GitHub Copilot a některé týmy používají nástroje, které si sami vyvinuli.

Naše počáteční řešení bylo jednoduché a přímé: napsat vyhrazený integrační kód pro každý nástroj AI. Nevýhody se ale rychle ukázaly. Kódová základna se zaplnila `if-else` větve, každá změna vyžadovala testování na více místech a každý nový nástroj znamenal napsat další hromadu logiky od začátku.

Později jsem si uvědomil, že by bylo lepší vytvořit jednotný `IAIProvider` rozhraní a abstrahují možnosti sdílené všemi poskytovateli AI. Tímto způsobem, bez ohledu na to, který nástroj je použit pod ním, horní vrstvy jej mohou nazývat stejným způsobem.

Projekt nedávno potřeboval integrovat dva nové nástroje: iflow a OpenCode. Oba podporují protokol ACP, ale jejich komunikační styly se liší. iflow používá WebSocket, zatímco OpenCode používá HTTP API. To se stalo užitečným architektonickým testem: přizpůsobte dva různé způsoby dopravy za jedno jednotné rozhraní.

## O HagiCode

Přístup sdílený v tomto článku vychází z našich praktických zkušeností v [HagiCode](https://hagicode.com) projekt. HagiCode je vývojová platforma podporovaná umělou inteligencí postavená na architektuře Orleans Grain. Integruje se s různými poskytovateli AI prostřednictvím jednotného `IAIProvider` rozhraní, které uživatelům umožňuje flexibilně vybírat nástroje umělé inteligence, které preferují.

## Architektura Design

### Abstrakce jednotného rozhraní

Nejprve jsme definovali `IAIProvider` rozhraní a abstrahovali možnosti, které každý poskytovatel AI potřebuje implementovat:

```csharp
public interface IAIProvider
{
    string Name { get; }
    bool SupportsStreaming { get; }
    ProviderCapabilities Capabilities { get; }

    Task<AIResponse> ExecuteAsync(AIRequest request, CancellationToken cancellationToken = default);
    IAsyncEnumerable<AIStreamingChunk> StreamAsync(AIRequest request, CancellationToken cancellationToken = default);
    Task<ProviderTestResult> PingAsync(CancellationToken cancellationToken = default);
    IAsyncEnumerable<AIStreamingChunk> SendMessageAsync(AIRequest request, string? embeddedCommandPrompt = null, CancellationToken cancellationToken = default);
}
```

Toto rozhraní obsahuje několik klíčových metod:
- `ExecuteAsync`: provedení jednorázového požadavku AI
- `StreamAsync`: získat odpovědi ze streamování pro zobrazení v reálném čase
- `PingAsync`: proveďte kontrolu stavu, abyste ověřili, zda je poskytovatel dostupný
- `SendMessageAsync`: odeslat zprávu s podporou vložených příkazů

### IFlowCliProvider: Implementace založená na WebSocket

iflow používá WebSocket pro komunikaci ACP. Celková architektura vypadá takto:

```
IFlowCliProvider → ACPSessionManager → WebSocketAcpTransport → iflow CLI
                ↓
         Dynamic port allocation + process management
```

Tok jádra je také poměrně přímočarý:
1. `ACPSessionManager` vytváří a spravuje relace AKT.
2. `WebSocketAcpTransport` zpracovává komunikaci WebSocket.
3. Port je přidělován dynamicky a proces iflow je zahájen `iflow --experimental-acp --port`.
4. `IAIRequestToAcpMapper` a `IAcpToAIResponseMapper` převádět požadavky a odpovědi.

Zde je základní kód:

```csharp
private async IAsyncEnumerable<AIStreamingChunk> StreamCoreAsync(
    AIRequest request,
    string? embeddedCommandPrompt,
    [EnumeratorCancellation] CancellationToken cancellationToken)
{
    // Resolve working directory
    var resolvedWorkingDirectory = ResolveWorkingDirectory(request);
    var effectiveRequest = ApplyEmbeddedCommandPrompt(request, embeddedCommandPrompt);

    // Create ACP session
    await using var session = await _sessionManager.CreateSessionAsync(
        Name,
        resolvedWorkingDirectory,
        cancellationToken,
        request.SessionId);

    // Send prompt
    var prompt = _requestMapper.ToPromptString(effectiveRequest);
    var promptResponse = await session.SendPromptAsync(prompt, cancellationToken);

    // Receive streaming response
    await foreach (var notification in session.ReceiveUpdatesAsync(cancellationToken))
    {
        if (_responseMapper.TryConvertToStreamingChunk(notification, out var chunk))
        {
            if (chunk.Type == StreamingChunkType.Metadata && chunk.IsComplete)
            {
                yield return chunk;
                yield break;
            }
            yield return chunk;
        }
    }
}
```

Zde je několik návrhových bodů, které stojí za zmínku:
- Použijte `await using` abyste zajistili správné uvolnění relace a zabránili úniku prostředků.
- Vraťte streamované odpovědi prostřednictvím `IAsyncEnumerable`, který přirozeně podporuje asynchronní streamy.
- Použijte `Metadata` kousky, abyste určili dokončení a zajistili přijetí úplné odpovědi.

### OpenCodeCliProvider: Implementace založená na HTTP API

OpenCode poskytuje své služby prostřednictvím HTTP API, takže architektura je mírně odlišná:

```plaintext
OpenCodeCliProvider → OpenCodeRuntimeManager → OpenCodeClient → OpenCode HTTP API
                      ↓
                OpenCodeProcessManager → opencode process management
```

Pozoruhodnou vlastností OpenCode je, že používá databázi SQLite k zachování vazeb relací. To umožňuje obnovu relace a rychlou odezvu:

```csharp
private async Task<OpenCodePromptExecutionResult> ExecutePromptAsync(
    AIRequest request,
    string? embeddedCommandPrompt,
    CancellationToken cancellationToken)
{
    var prompt = BuildPrompt(request, embeddedCommandPrompt);
    var resolvedWorkingDirectory = ResolveWorkingDirectory(request.WorkingDirectory);
    var client = await _runtimeManager.GetClientAsync(resolvedWorkingDirectory, cancellationToken);
    var bindingSessionId = request.SessionId;
    var boundSession = TryGetBinding(bindingSessionId, resolvedWorkingDirectory);

    // Try to use the already bound session
    if (boundSession is not null)
    {
        try
        {
            return await PromptSessionAsync(
                client,
                boundSession,
                BuildPromptRequest(request, prompt, CreatePromptMessageId()),
                request.Model ?? _settings.Model,
                cancellationToken);
        }
        catch (OpenCodeApiException ex) when (IsStaleBinding(ex))
        {
            // The session has expired, remove the binding
            RemoveBinding(bindingSessionId);
        }
    }

    // Create a new session
    var session = await client.Session.CreateAsync(new OpenCodeSessionCreateRequest
    {
        Title = BuildSessionTitle(request)
    }, cancellationToken);

    BindSession(bindingSessionId, session.Id, resolvedWorkingDirectory);
    return await PromptSessionAsync(client, session.Id, ...);
}
```

Tato implementace má několik zajímavých předností:
- Mechanismus vazby relace: stejný `SessionId` znovu používá stejnou relaci OpenCode, čímž se vyhne opakovanému vytváření relace.
- Zpracování vypršení platnosti: když se zjistí, že relace vypršela, vazba se automaticky vyčistí.
- Perzistence databáze: vazby jsou uloženy v SQLite a zůstávají účinné po restartu.

### Porovnání dvou přístupů

| Aspekt | IFlowCliProvider | OpenCodeCliProvider |
|------|-----------------|---------------------|
| Komunikace | WebSocket (ACP) | HTTP API |
| Řízení procesů | ACPSessionManager | OpenCodeProcessManager |
| Přidělení portů | Dynamický port | Žádný port (používá HTTP) |
| Správa relace | ACPSsession | OpenCodeSession |
| Vytrvalost | Mezipaměť v paměti | SQLite databáze |
| Příkaz pro spuštění | `iflow --experimental-acp --port` | `opencode` |
| Latence | Nižší (dlouhotrvající připojení) | Relativně vyšší (požadavky HTTP) |

Jaký přístup zvolíte, záleží především na vašich potřebách. WebSocket je lepší pro scénáře s vysokými požadavky v reálném čase, zatímco HTTP API je jednodušší a snáze se ladí.

## Praktický průvodce

### Konfigurace poskytovatelů

Nejprve povolte dva poskytovatele v konfiguračním souboru:

```yaml
AI:
  Providers:
    IFlowCli:
      Type: "IFlowCli"
      Enabled: true
      ExecutablePath: "iflow"
      Model: null
      WorkingDirectory: null
    OpenCodeCli:
      Type: "OpenCodeCli"
      Enabled: true
      ExecutablePath: "opencode"
      Model: "anthropic/claude-sonnet-4"
      WorkingDirectory: null

OpenCode:
  Enabled: true
  BaseUrl: "http://localhost:38376"
  ExecutablePath: "opencode"
  StartupTimeoutSeconds: 30
  RequestTimeoutSeconds: 120
```

### Použijte IFlowCliProvider

```csharp
// Get provider through the factory
var provider = await _providerFactory.GetProviderAsync(AIProviderType.IFlowCli);

// Execute an AI request
var request = new AIRequest
{
    Prompt = "请帮我重构这个函数",
    WorkingDirectory = "/path/to/project",
    Model = "claude-sonnet-4"
};

// Get the complete response
var response = await provider.ExecuteAsync(request, cancellationToken);
Console.WriteLine(response.Content);

// Or use streaming responses
await foreach (var chunk in provider.StreamAsync(request, cancellationToken))
{
    if (chunk.Type == StreamingChunkType.ContentDelta)
    {
        Console.Write(chunk.Content);
    }
}
```

### Použijte OpenCodeCliProvider

```csharp
// Get provider through the factory
var provider = await _providerFactory.GetProviderAsync(AIProviderType.OpenCodeCli);

var request = new AIRequest
{
    Prompt = "请帮我分析这个错误",
    WorkingDirectory = "/path/to/project",
    Model = "anthropic/claude-sonnet-4"
};

var response = await provider.ExecuteAsync(request, cancellationToken);
Console.WriteLine(response.Content);
```

### Zdravotní kontroly

Před spuštěním nebo před použitím můžete zkontrolovat, zda je poskytovatel dostupný:

```csharp
var iflowResult = await iflowProvider.PingAsync(cancellationToken);
if (!iflowResult.Success)
{
    Console.WriteLine($"IFlow is unavailable: {iflowResult.ErrorMessage}");
    return;
}

var openCodeResult = await openCodeProvider.PingAsync(cancellationToken);
if (!openCodeResult.Success)
{
    Console.WriteLine($"OpenCode is unavailable: {openCodeResult.ErrorMessage}");
    return;
}
```

### Vestavěná podpora příkazů

Oba poskytovatelé podporují vestavěné příkazy, jako např `/file:xxx`:

```csharp
var request = new AIRequest
{
    Prompt = "分析这个文件的问题",
    SystemMessage = "你是一个代码分析专家"
};

await foreach (var chunk in provider.SendMessageAsync(
    request,
    embeddedCommandPrompt: "/file:src/main.cs",
    cancellationToken))
{
    Console.Write(chunk.Content);
}
```

## Poznámky a doporučené postupy

### Správa zdrojů

IFlow používá dlouhotrvající připojení WebSocket, takže správa zdrojů si zaslouží zvláštní pozornost:
- Použijte `await using` abyste zajistili, že se relace uvolní správně.
- Zrušení spustí proces čištění.
- `ACPSessionManager` podporuje maximální limit počtu relací.

Správa procesů OpenCode je relativně jednodušší a `OpenCodeRuntimeManager` zvládá to automaticky.

### Zpracování chyb

Oba poskytovatelé mají kompletní zpracování chyb:
- Chyby IFlow se šíří prostřednictvím aktualizací relace ACP.
- Chyby OpenCode jsou prohozeny `OpenCodeApiException`.
- Doporučuje se, aby volající zachytil a zpracoval tyto výjimky.

### Úvahy o výkonu

- Komunikace IFlow WebSocket má nižší latenci než HTTP.
- Opětovné použití relace OpenCode může snížit režii požadavků HTTP.
- Mechanismus tovární mezipaměti zabraňuje opakovanému vytváření poskytovatelů.
- Ve scénářích s vysokou souběžností věnujte velkou pozornost limitům počtu procesů a počtu připojení.

### Ověření konfigurace

Cesta ke spustitelnému souboru je ověřena při spuštění, ale přesto se mohou vyskytnout problémy s runtime. `PingAsync` je užitečný nástroj pro ověření správnosti konfigurace:

```csharp
// Check at startup
var provider = await _providerFactory.GetProviderAsync(providerType);
var result = await provider.PingAsync(cancellationToken);
if (!result.Success)
{
    _logger.LogError("Provider {ProviderType} is unavailable: {Error}", providerType, result.ErrorMessage);
}
```

## Shrnutí

Tento článek sdílí technický přístup používaný platformou HagiCode při integraci dvou nástrojů AI iflow a OpenCode. Prostřednictvím jednotného `IAIProvider` přizpůsobili jsme různé komunikační styly, WebSocket a HTTP, přičemž jsme zachovali konzistentní vzor volání horní vrstvy.

Základní myšlenka je vlastně docela jednoduchá:
1. Definujte abstrakci jednotného rozhraní.
2. Vytvářejte vrstvy adaptérů pro různé implementace.
3. Spravujte vše jednotně prostřednictvím továrního vzoru.

To dává systému dobrou rozšiřitelnost. Když je později potřeba integrovat nový nástroj umělé inteligence, vše, co musíme udělat, je implementovat `IAIProvider` rozhraní bez přílišné změny stávajícího kódu.

Pokud také pracujete na integraci více nástrojů AI, doufám, že vám tento článek pomůže.
