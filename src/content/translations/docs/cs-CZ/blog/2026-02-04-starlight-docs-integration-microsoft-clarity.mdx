Při provozu **HagiCode** jsme neustále naráželi na problém „černé skříňky“: vytvářeli jsme obsah, ale neměli jsme jasnou představu o tom, jak jej uživatelé ve skutečnosti čtou. GitHub Stars může ukázat nějaký signál, ale tato zpětná vazba přichází příliš pozdě. Co jsme opravdu potřebovali vědět, bylo:

- Opravdu uživatelé dočtou naše návody?

Na trhu existuje mnoho analytických nástrojů, jako je Google Analytics (GA) a Microsoft Clarity. GA je výkonná, ale složitější na konfiguraci a je také přísněji omezena předpisy o ochraně osobních údajů, jako je GDPR. Clarity, jako bezplatný nástroj teplotní mapy od společnosti Microsoft, se nejen intuitivně používá, ale také se s ním relativně snadněji pracuje z hlediska ochrany osobních údajů, takže se skvěle hodí pro stránky s technickou dokumentací.

**Náš cíl byl velmi jasný**: hladce integrovat Clarity do dokumentačního webu **HagiCode** tak, aby fungoval na všech stránkách a zároveň poskytoval uživatelům právo odmítnout dodržování zásad ochrany osobních údajů.

## Technický výběr a průzkum

Nejprve jsme ve fázi návrhu diskutovali o několika integračních přístupech. Vzhledem k tomu, že jsme používali Starlight, dokumentační rámec postavený na Astro, nejzřejmější myšlenkou bylo použití Astro Hooks.

Nejprve jsme se pokusili upravit `astro.config.mjs` a plánoval vložit skript Clarity během sestavení. Ačkoli tento přístup mohl zaručit globální pokrytí, postrádal flexibilitu: nemohli jsme dynamicky načítat nebo uvolnit skript na základě uživatelských preferencí.

S ohledem na uživatelskou zkušenost a kontrolu soukromí jsme nakonec zvolili přístup **přepisování komponent**. Starlight umožňuje vývojářům přepsat jeho vnitřní komponenty, což znamená, že bychom mohli převzít logiku vykreslování `<footer>` nebo `<head>` a přesné ovládání při načtení Clarity.

Zde byla také malá odbočka: původně jsme chtěli vytvořit obálku rozvržení s názvem `StarlightWrapper.astro`. Ale během skutečného ladění jsme zjistili, že směrovací mechanismus Starlight automaticky nevyvolá tento vlastní obal, což způsobuje selhání skriptu na některých stránkách. Bylo to klasické „zřejmé předpoklady“ úskalí a připomnělo nám, že **musíme hluboce porozumět toku vykreslování rámce namísto slepého používání obecných vzorců rámců**.

## Základní řešení: Přepsání komponenty zápatí

Abychom zajistili, že se skript Clarity načte na všech stránkách, včetně dokumentů a blogových příspěvků, aniž by došlo k porušení původní struktury stránky, rozhodli jsme se přepsat Starlight `Footer` komponenta.

### Proč zápatí?

1.  **Globální přítomnost**: Zápatí se objevuje téměř na všech normostranách.
2.  **Neobtěžující**: umístění skriptu do oblasti zápatí, která je ve skutečnosti vykreslena blízko spodní části těla, neblokuje kritickou cestu vykreslování stránky (LCP), takže dopad na výkon je minimální.
3.  **Centralizovaná logika**: Logiku souhlasu se soubory cookie lze zpracovávat na jednom místě uvnitř komponenty.

Vytvořte soubor s názvem `StarlightFooter.astro` pod `src/components/`. Starlight jej automaticky rozpozná a přepíše výchozí zápatí.

Logika základního kódu je následující:

```astro
---
// src/components/StarlightFooter.astro
// 1. Import the original component to preserve its default behavior
import DefaultFooter from '@astrojs/starlight/components/StarlightFooter.astro';

// 2. Read the environment variable
const clarityId = import.meta.env.PUBLIC_CLARITY_ID;

// 3. Define a simple injection script (inline approach)
// Note: in production, it is recommended to move this logic into a separate .js file to take advantage of caching
const initScript = `
(function(c,l,a,r,i,t,y){
    c[a]=c[a]||function(){(c[a].q=c[a].q||[]).push(arguments)};
    t=l.createElement(r);t.async=1;t.src="https://www.clarity.ms/tag/"+i;
    y=l.getElementsByTagName(r)[0];y.parentNode.insertBefore(t,y);
})(window, document, "clarity", "script", "${clarityId}");
`;
---

<DefaultFooter {...Astro.props} />

{/* Inject the script only in production and only when the ID exists */}
{import.meta.env.PROD && clarityId && (
  <script is:inline define:vars={{ clarityId }}>
    {initScript}
  </script>
)}
```

**Vysvětlení klíčových bodů**:
- `is:inline`: říká Astrovi, aby nezpracovával obsah této značky skriptu a aby jej vydal přímo do HTML. To je důležité pro analytické skripty třetích stran; jinak může optimalizace sdružování Astro způsobit, že skript přestane fungovat.
- `define:vars`: funkce Astro 3+, která umožňuje bezpečné vstřikování proměnných v rámci rozsahu.
- `import.meta.env.PROD`: zajišťuje, že během místního vývoje nejsou generovány žádné nesmyslné analýzy, pokud to není výslovně nutné pro ladění, a vaše data tak zůstávají čistá.

## Poučení a úskalí

Když jsme toto řešení zavedli v HagiCode, shrnuli jsme několik podrobností, které lze snadno přehlédnout:

1.  **`StarlightWrapper.astro` je past**:
    Jak bylo zmíněno dříve, nepokoušejte se vytvořit globální Wrapper pro vložení skriptu. Tento přístup ve Starlight nefunguje. Správným řešením je přepsat konkrétní komponentu jako např `StarlightFooter.astro` nebo `StarlightHead.astro`.

2.  **Úvahy o výkonu pro umístění skriptu**:
    Přestože Clarity doporučuje umístit skript do `<head>` Aby byla zajištěna maximální přesnost dat, u dokumentačního webu rychlost načítání první obrazovky (LCP) přímo ovlivňuje SEO a udržení uživatelů. Rozhodli jsme se jej umístit do zápatí, poblíž spodní části těla. To může trochu vynechat malé množství uživatelských dat „odskoků během několika sekund“, ale poskytuje nám to rychlejší načítání stránek, což je cenný kompromis.

3.  **Zásah z vývojového prostředí**:
    Nezapomeňte přidat `import.meta.env.PROD` zkontrolovat. Ve vývojovém režimu budete stránky často obnovovat, což by jinak generovalo velké množství nesmyslných testovacích dat a znečišťovalo váš dashboard Clarity.

## Závěr

Integrace Microsoft Clarity nevyžaduje složité změny na straně serveru ani nevyžaduje zavedení těžké sady SDK.

Využitím mechanismu přepisování komponent Starlight jsme dosáhli analýzy na celém webu s ničím jiným než lehkým `StarlightFooter.astro` komponenta. Tento druh „mikrointegrace“ udržuje kódovou základnu čistou a zároveň nám dává možnost porozumět chování uživatelů.
