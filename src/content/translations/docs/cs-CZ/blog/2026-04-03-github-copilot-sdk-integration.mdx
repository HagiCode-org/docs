## Od volání CLI k integraci SDK: Nejlepší postupy pro GitHub Copilot v projektech .NET

> Cesta upgradu od vyvolání příkazového řádku k oficiální integraci SDK byla docela dlouhá cesta. Dnes se chci podělit o úskalí, na která jsme narazili, a co jsme se naučili při budování HagiCode.

## Pozadí

Poté, co byla sada GitHub Copilot SDK oficiálně vydána v roce 2025, začali jsme ji integrovat do naší vrstvy schopností AI. Předtím projekt používal hlavně GitHub Copilot přímým vyvoláním nástroje příkazového řádku Copilot CLI, ale tento přístup měl několik zjevných problémů:

- **Složitá správa procesů**: Museli jsme ručně spravovat životní cyklus procesu CLI, časové limity spouštění a čištění procesů. Procesy mohou selhat bez varování.
- **Neúplné zpracování událostí**: Vyvolání nezpracovaného CLI ztěžuje zachycení jemnozrnných událostí z uvažování modelu a provádění nástroje. Je to jako vidět pouze výsledek bez procesu myšlení.
- **Obtížná správa relace**: Neexistoval žádný účinný mechanismus pro opětovné použití a obnovu relace, takže každá interakce musela začít znovu.
- **Problémy s kompatibilitou**: Argumenty CLI se často měnily, což znamenalo, že jsme neustále museli udržovat logiku kompatibility pro tyto parametry.

Tyto problémy byly stále zjevnější v každodenním vývoji, zvláště když jsme potřebovali sledovat uvažování modelu (`thinking`) a stav provádění nástroje v reálném čase. V tu chvíli bylo jasné, že potřebujeme nižší a úplnější integrační přístup.

## O HagiCode

Přístup sdílený v tomto článku vychází z našich praktických zkušeností v [HagiCode](https://hagicode.com) projekt. HagiCode je projekt asistenta kódování umělé inteligence s otevřeným zdrojovým kódem a během vývoje jsme potřebovali hlubokou integraci s funkcemi GitHub Copilot, od základního dokončování kódu až po složité víceotáčkové konverzace a volání nástrojů. Tyto skutečné požadavky nás donutily přejít z vyvolání CLI na oficiální SDK.

Pokud se vám tato implementace zdá užitečná, je velká šance, že vám naše inženýrské zkušenosti pomohou. V takovém případě může stát za to vyzkoušet i samotný projekt HagiCode. Na konci tohoto článku můžete dokonce najít další informace o projektu a odkazy.

## Architektura Design

Projekt využívá vrstvenou architekturu k řešení omezení vyvolání CLI:

```
┌─────────────────────────────────────────────────────────┐
│  hagicode-core (Orleans Grains + AI Provider Layer)    │
│  - CopilotAIProvider: Converts AIRequest to CopilotOptions │
│  - GitHubCopilotGrain: Orleans distributed execution interface │
└─────────────────────────────────────────────────────────┘
                          ↓
┌─────────────────────────────────────────────────────────┐
│  HagiCode.Libs (Shared Provider Layer)                 │
│  - CopilotProvider: CLI Provider interface implementation │
│  - ICopilotSdkGateway: SDK invocation abstraction      │
│  - GitHubCopilotSdkGateway: SDK session management and event dispatch │
└─────────────────────────────────────────────────────────┘
                          ↓
┌─────────────────────────────────────────────────────────┐
│  GitHub Copilot SDK (Official .NET SDK)                │
│  - CopilotClient: SDK client                           │
│  - CopilotSession: Session management                  │
│  - SessionEvent: Event stream                          │
└─────────────────────────────────────────────────────────┘
```

Tento vrstvený design přináší několik praktických technických výhod:

1. **Oddělení obav**: Základní obchodní logika je oddělena od podrobností implementace SDK.
2. **Testovatelnost**: The `ICopilotSdkGateway` rozhraní usnadňuje testování jednotek.
3. **Opětovné použití**: `HagiCode.Libs` může být odkazováno více projekty.
4. **Údržba**: Upgrady SDK vyžadují pouze změny ve vrstvě Gateway, zatímco horní vrstvy zůstávají nedotčeny.

## Implementace jádra

### Tok autentizace

Autentizace je prvním a nejdůležitějším krokem integrace SDK. Pokud se ověření nezdaří, na ničem jiném nezáleží. Navrhli jsme flexibilní konfiguraci ověřování, která podporuje více zdrojů ověřování:

```csharp
// CopilotProvider.cs - Authentication source configuration
public class CopilotOptions
{
    public bool UseLoggedInUser { get; set; } = true;
    public string? GitHubToken { get; set; }
    public string? CliUrl { get; set; }
}

// Convert to SDK request
return new CopilotSdkRequest(
    GitHubToken: options.AuthSource == CopilotAuthSource.GitHubToken
        ? options.GitHubToken
        : null,
    UseLoggedInUser: options.AuthSource != CopilotAuthSource.GitHubToken
);
```

Výhody tohoto designu jsou poměrně jasné:

- Podporuje režim přihlášeného uživatele bez potřeby tokenu, což se dobře hodí pro scénáře na ploše.
- Podporuje režim GitHub Token, který je vhodný pro nasazení na straně serveru a centralizovanou správu.
- Podporuje přepsání adresy URL rozhraní CLI Copilot, což pomáhá s konfigurací podnikového proxy serveru.

V praxi tento flexibilní model ověřování značně zjednodušil konfiguraci v různých scénářích nasazení. Desktopový klient může používat vlastní stav přihlášení Copilot každého uživatele, zatímco serverová strana může řídit ověřování centrálně prostřednictvím tokenů.

### Zpracování toku událostí

Jednou z nejvýkonnějších funkcí sady SDK je úplné zachycení streamu událostí. Implementovali jsme systém odesílání událostí, který dokáže zpracovat všechny druhy událostí SDK v reálném čase:

```csharp
// GitHubCopilotSdkGateway.cs - Core logic for event dispatch
internal static SessionEventDispatchResult DispatchSessionEvent(
    SessionEvent evt, bool sawDelta)
{
    switch (evt)
    {
        case AssistantReasoningEvent reasoningEvent:
            // Capture the model reasoning process
            events.Add(new CopilotSdkStreamEvent(
                CopilotSdkStreamEventType.ReasoningDelta,
                Content: reasoningEvent.Data.Content));
            break;

        case ToolExecutionStartEvent toolStartEvent:
            // Capture the start of a tool invocation
            events.Add(new CopilotSdkStreamEvent(
                CopilotSdkStreamEventType.ToolExecutionStart,
                ToolName: toolStartEvent.Data.ToolName,
                ToolCallId: toolStartEvent.Data.ToolCallId));
            break;

        case ToolExecutionCompleteEvent toolCompleteEvent:
            // Capture tool completion and its result
            events.Add(new CopilotSdkStreamEvent(
                CopilotSdkStreamEventType.ToolExecutionEnd,
                Content: ExtractToolExecutionContent(toolCompleteEvent)));
            break;

        default:
            // Preserve unhandled events as RawEvent
            events.Add(new CopilotSdkStreamEvent(
                CopilotSdkStreamEventType.RawEvent,
                RawEventType: evt.GetType().Name));
            break;
    }
}
```

Hodnota této implementace je značná:

- **Kompletní zachycení procesu uvažování modelu** (`thinking`): uživatelé mohou vidět, jak AI uvažuje, nejen konečnou odpověď.
- **Sledování stavu provádění nástroje v reálném čase**: víme, které nástroje běží, kdy skončí a co vrátí.
- **Nulová ztráta události**: návratem k `RawEvent`, zajišťujeme záznam každé události.

V HagiCode tyto jemnozrnné události pomáhají uživatelům pochopit, jak AI interně funguje, zejména při ladění složitých úkolů.

### Manipulace s kompatibilitou CLI

Po migraci z vyvolání rozhraní CLI na sadu SDK jsme zjistili, že některé existující parametry rozhraní CLI již v sadě SDK neplatí. Pro zachování zpětné kompatibility jsme implementovali systém filtrování parametrů:

```csharp
// CopilotCliCompatibility.cs - Argument filtering
private static readonly Dictionary<string, string> RejectedFlags = new()
{
    ["--headless"] = "Unsupported startup argument",
    ["--model"] = "Passed through an SDK-native field",
    ["--prompt"] = "Passed through an SDK-native field",
    ["--interactive"] = "Interaction is managed by the provider",
};

public static CopilotCliArgumentBuildResult BuildCliArgs(CopilotOptions options)
{
    // Filter out unsupported arguments and keep compatible ones
    // Generate diagnostic information
}
```

To nám přináší několik výhod:

- Automaticky filtruje nekompatibilní argumenty CLI, aby se zabránilo chybám za běhu.
- Generuje jasné diagnostické zprávy, které vývojářům pomáhají rychle lokalizovat problémy.
- Udržuje SDK stabilní a izolované od změn parametrů CLI.

Během upgradu nám tento mechanismus kompatibility pomohl hladce přejít. Stávající konfigurační soubory mohly být nadále používány a vyžadovaly pouze postupné úpravy na základě diagnostického výstupu.

### Runtime Pooling

Vytváření relací sady Copilot SDK je poměrně nákladné a příliš časté vytváření a rušení relací snižuje výkon. Abychom to vyřešili, implementovali jsme systém správy fondu relací:

```csharp
// CopilotProvider.cs - Session pool management
await using var lease = await _poolCoordinator.AcquireCopilotRuntimeAsync(
    request,
    async ct => await _gateway.CreateRuntimeAsync(sdkRequest, ct),
    cancellationToken);

if (lease.IsWarmLease)
{
    // Reuse an existing session
    yield return CreateSessionReusedMessage();
}

await foreach (var eventData in lease.Entry.Resource.SendPromptAsync(...))
{
    yield return MapEvent(eventData);
}
```

Mezi výhody sdružování relací patří:

- **Opětovné použití relace**: požadavky se stejným `sessionId` může znovu použít existující relace a snížit režii při spuštění.
- **Podpora obnovení relace**: po přerušení sítě lze obnovit stav předchozí relace.
- **Automatická správa sdružování**: relace, jejichž platnost vypršela, jsou automaticky vyčištěny, aby nedocházelo k únikům zdrojů.

V HagiCode sdružování relací znatelně zlepšilo odezvu, zejména u nepřetržitých konverzací.

### Orleánská integrace

HagiCode používá Orleans jako svůj distribuovaný rámec a do Orleans Grains jsme integrovali Copilot SDK:

```csharp
// GitHubCopilotGrain.cs - Distributed execution
public async IAsyncEnumerable<GitHubCopilotResponse> ExecuteCommandStreamAsync(
    string command,
    CancellationToken token = default)
{
    var provider = await aiProviderFactory.GetProviderAsync(AIProviderType.GitHubCopilot);

    await foreach (var chunk in provider.SendMessageAsync(request, null, token))
    {
        // Map to the unified response format
        yield return BuildChunkResponse(chunk, startedAt);
    }
}
```

Výhody integrace Orleans jsou značné:

- **Unifikovaná abstrakce poskytovatelů AI**: Přepínání mezi různými poskytovateli umělé inteligence je snadné.
- **Izolace více tenantů**: Relace druhého pilota pro různé uživatele zůstávají navzájem izolované.
- **Trvalý stav relace**: stav relace lze obnovit i po restartu serveru.

Pro scénáře, které potřebují zpracovat velký počet souběžných požadavků, poskytuje Orleans silnou škálovatelnost.

## Praktický průvodce

### Příklad konfigurace

Zde je kompletní příklad konfigurace:

```json
{
  "AI": {
    "Providers": {
      "Providers": {
        "GitHubCopilot": {
          "Enabled": true,
          "ExecutablePath": "copilot",
          "Model": "gpt-5",
          "WorkingDirectory": "/path/to/project",
          "Timeout": 7200,
          "StartupTimeout": 30,
          "UseLoggedInUser": true,
          "NoAskUser": true,
          "Permissions": {
            "AllowAllTools": false,
            "AllowedTools": ["Read", "Bash", "Grep"],
            "DeniedTools": ["Edit"]
          }
        }
      }
    }
  }
}
```

### Poznámky k použití

V reálném světě jsme shrnuli několik bodů, kterým stojí za to věnovat pozornost:

**Konfigurace časového limitu spouštění**: První spuštění Copilot CLI může trvat relativně dlouho, proto doporučujeme nastavit `StartupTimeout` na alespoň 30 sekund. Pokud se jedná o první přihlášení, může to trvat ještě déle.

**Správa oprávnění**: Nepoužívejte v produkčním prostředí `AllowAllTools: true`. Použijte `AllowedTools` seznam povolených pro kontrolu, které nástroje jsou k dispozici, a použití `DeniedTools` denylist k blokování nebezpečných operací. To účinně zabraňuje AI ve vykonávání riskantních příkazů.

**Správa relace**: Žádosti se stejným `sessionId` automaticky znovu použít relace. Stav relace trvá `ProviderSessionId`. Zrušení je propagováno prostřednictvím `CancellationTokenSource`.

**Diagnostický výstup**: Nekompatibilní argumenty CLI generují zprávy typu `diagnostic`. Nezpracované události SDK jsou zachovány jako `event.raw`. Chybové zprávy zahrnují kategorie, jako je časový limit spouštění a nekompatibilita argumentů, aby se usnadnilo odstraňování problémů.

### Nejlepší postupy

Na základě našich praktických zkušeností uvádíme několik osvědčených postupů:

**1. Použít seznam povolených nástrojů**

```csharp
var request = new AIRequest
{
    Prompt = "Analyze this file",
    AllowedTools = new[] { "Read", "Grep", "Bash(git:*)" }
};
```

Explicitní určení povolených nástrojů prostřednictvím seznamu povolených pomáhá předcházet neočekávaným akcím AI. To je důležité zejména u nástrojů s oprávněním k zápisu, jako je např `Edit`, se kterými je třeba zacházet se zvýšenou opatrností.

**2. Nastavte přiměřené časové limity**

```csharp
options.Timeout = 3600;  // 1 hour
options.StartupTimeout = 60;  // 1 minute
```

Nastavte vhodné hodnoty časového limitu na základě složitosti úlohy. Pokud je hodnota příliš krátká, úlohy mohou být přerušeny. Pokud je příliš dlouhý, může dojít k plýtvání zdroji čekáním na nereagující požadavky.

**3. Povolit opětovné použití relace**

```csharp
options.SessionId = "my-session-123";
```

Pomocí stejného `sessionId` pro související úlohy umožňuje znovu použít kontext předchozí relace a zlepšit rychlost odezvy.

**4. Zpracovat odpovědi streamování**

```csharp
await foreach (var chunk in provider.StreamAsync(request))
{
    switch (chunk.Type)
    {
        case StreamingChunkType.ThinkingDelta:
            // Handle the reasoning process
            break;
        case StreamingChunkType.ToolCallDelta:
            // Handle tool invocation
            break;
        case StreamingChunkType.ContentDelta:
            // Handle text output
            break;
    }
}
```

Odezvy streamování vám umožňují zobrazit průběh zpracování AI v reálném čase, což zlepšuje uživatelský dojem. To je zvláště cenné pro dlouhodobé úkoly.

**5. Zpracování chyb a opakování**

```csharp
try
{
    await foreach (var chunk in provider.StreamAsync(request))
    {
        // Handle the response
    }
}
catch (CopilotSessionException ex)
{
    // Handle session exceptions
    logger.LogError(ex, "Copilot session failed");
    // Decide whether to retry based on the exception type
}
```

Správné zpracování chyb a logika opakování zlepšují celkovou stabilitu systému.

## Závěr

Upgrade z vyvolání CLI na integraci SDK přinesl projektu HagiCode podstatnou hodnotu:

- **Vylepšená stabilita**: SDK poskytuje stabilnější rozhraní, které není ovlivněno změnami verze CLI.
- **Ucelenější funkce**: zachycuje úplný proud událostí, včetně zdůvodnění a stavu provádění nástroje.
- **Vyšší efektivita vývoje**: typově bezpečné rozhraní SDK zefektivňuje vývoj a snižuje chyby za běhu.
- **Lepší uživatelská zkušenost**: zpětná vazba událostí v reálném čase poskytuje uživatelům jasnější pochopení toho, co AI dělá.

Tento upgrade nebyl jen náhradou technické implementace. Jednalo se také o architektonickou optimalizaci celé vrstvy schopností AI. Prostřednictvím vrstveného designu a rozhraní abstrakce jsme získali lepší udržovatelnost a rozšiřitelnost.

Pokud uvažujete o integraci GitHub Copilot do svého vlastního projektu .NET, doufám, že vám tyto praktické lekce pomohou vyhnout se zbytečným oklikám. Oficiální SDK je skutečně stabilnější a úplnější než vyvolání CLI a stojí za to mu porozumět a přijmout ho.
