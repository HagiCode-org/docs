## Od Docusaura 3.x k Astro 5.x: Retrospektiva migrace webu HagiCode

> Tento článek se ohlíží zpět na naši úplnou migraci oficiálního webu HagiCode z Docusaurus 3.x na Astro 5.x. Hluboce se ponoříme do toho, jak nám Astro's Islands Architecture pomohla vyřešit problémová místa ve výkonu při zachování našich stávajících komponent React a přinesla vylepšení jak v rychlosti sestavení, tak ve výkonu načítání.

## Pozadí

V lednu 2026 jsme na oficiálních stránkách HagiCode provedli „transplantaci srdce“ úplnou migrací jeho základního rámce z Docusaurus 3.x na Astro 5.x. Nešlo o impulzivní přepis, ale o pečlivě zvážené technické rozhodnutí.

Před migrací byly naše stránky funkčně kompletní, ale začaly vykazovat některé klasické „luxusní problémy“: nabubřelé artefakty sestavení, nadměrné zatížení JavaScriptu a méně než ideální rychlost načítání stránek na složitých dokumentačních stránkách. Jako projekt asistenta kódování AI potřebuje HagiCode častou dokumentaci a aktualizace funkcí, takže efektivita sestavení přímo ovlivňuje rychlost vydání. Zároveň jsme chtěli, aby byl web přívětivější pro vyhledávače (SEO), aby projekt mohlo objevit více vývojářů.

Abychom vyřešili tyto bolestivé body, učinili jsme odvážné rozhodnutí: přestavět celý systém na Astro. Dopad tohoto rozhodnutí může být ještě větší, než očekáváte. Brzy se dostanu do podrobností.

## O HagiCode

Přístup k migraci stránek sdílený v tomto článku vychází z našich praktických zkušeností v oblasti [HagiCode](https://github.com/HagiCode-org/site) projekt.

HagiCode je asistent kódování AI zaměřený na zlepšení efektivity vývoje. Záleží nám nejen na opakování základních funkcí, ale také na zkušenostech vývojáře. Účelem tohoto refaktoru stránek bylo také poskytnout uživatelům nejrychlejší možný zážitek při procházení našich dokumentů a oficiálních webových stránek.

## Proč opustit ekosystém zralého docusaura?

V rámci ekosystému React byl Docusaurus dlouho „standardní odpovědí“ pro dokumentační weby. Funguje hned po vybalení, nabízí bohatý ekosystém pluginů a má aktivní komunitu. Ale jak HagiCode získal více funkcí, pocítili jsme také jeho omezení:

1.  **Úzká místa výkonu**: Docusaurus je v podstatě React SPA (jednostránková aplikace). I když píšete pouze statické stránky, klient stále potřebuje načíst běhové prostředí React a stránku hydratovat, což je pro jednoduché stránky dokumentů zbytečně těžké.
2.  **Velká velikost podkladu**: I když stránka obsahuje velmi málo obsahu, velikost přibaleného JS zůstává relativně pevná. To není ideální pro mobilní uživatele nebo špatné podmínky sítě.
3.  **Omezená flexibilita**: Přestože je rozšiřitelná, chtěli jsme více nízkoúrovňové kontroly nad procesem sestavení.

Společnost Astro dorazila přesně v pravý čas, aby tyto problémy vyřešila. Představila novou „Islands Architecture“: ve výchozím nastavení Astro generuje statické HTML s nulovým JavaScriptem a pouze komponenty, které vyžadují interaktivitu, jsou „aktivovány“ a načítají JS. To znamená, že většina našich stránek se stává čistým HTML a načítá se extrémně rychle.

## Základní strategie migrace: Hladký architektonický přechod

Migrace nebyla jen kopírování a vkládání. Chtělo to změnu myšlení. Přešli jsme z modelu "all React" od Docusaura k modelu "Core + Islands" společnosti Astro.

### 1. Přestavba konfiguračního systému

Nejprve jsme se museli odstěhovat `docusaurus.config.ts` do `astro.config.mjs`. Nešlo jen o přejmenování souboru, ale o přepsání logiky směrování a sestavení.

V Docusauru je vše plugin. V Astro je vše integrací. Potřebovali jsme předefinovat základní cestu webu, sestavit výstupní režim (statický vs SSR) a strategii optimalizace aktiv.

**Před migrací:**
```typescript
// docusaurus.config.ts
export default {
  title: 'HagiCode',
  url: 'https://hagicode.com',
  baseUrl: '/',
  // ... more configuration
};
```

**Po migraci:**
```javascript
// astro.config.mjs
import { defineConfig } from 'astro/config';
import react from '@astrojs/react';

export default defineConfig({
  integrations: [react()],
  site: 'https://hagicode.com',
  base: '/',
  // Optimization settings for static assets
  build: {
    inlineStylesheets: 'auto',
  },
});
```

### 2. Co si ponechat a co refaktorovat v React Components

To byla nejbolestivější část migrace. Náš stávající web měl mnoho komponent React, jako jsou karty, zvýrazňování kódu, tlačítka zpětné vazby a další. Jejich vyhazování by bylo plýtvání, ale ponechání všeho by způsobilo příliš velkou zátěž JavaScriptu.

HagiCode přijal strategii **progresivní hydratace**:
*   **Čistě statické komponenty**: Pro prezentační obsah, jako jsou záhlaví, zápatí a dokumentace ve formátu prostého textu, jsme je přepsali jako komponenty Astro (`.astro` soubory) a vykreslil je přímo do HTML v době sestavování.
*   **Interaktivní ostrovy**: U komponent, které musí zůstat interaktivní, jako jsou přepínače témat, přepínání karet a tlačítka pro kopírování bloku kódu, jsme ponechali implementaci React a přidali `client:load` nebo `client:visible` směrnice.

Například naše běžně používaná komponenta Tabs v dokumentaci:

```jsx
// src/components/Tabs.jsx
import { useState } from 'react';
import './Tabs.css'; // Import styles

export default function Tabs({ items }) {
  const [activeIndex, setActiveIndex] = useState(0);
  // ... state logic
  return (
    <div className="tabs-wrapper">
      {/* Rendering logic */}
    </div>
  );
}
```

Při použití v Markdown výslovně říkáme Astro: "Tato komponenta potřebuje JS."

```markdown
// src/content/docs/example.mdx
import Tabs from '../../components/Tabs.jsx';

<!-- Load JS only when the component enters the viewport -->
<Tabs client:visible items={...} />
```

Tímto způsobem interaktivní komponenty mimo výřez nesoutěží o šířku pásma, což výrazně zvyšuje rychlost načítání na první obrazovce.

### 3. Přizpůsobení systému stylů: Od modulů CSS k CSS s rozsahem

Docusaurus ve výchozím nastavení podporuje moduly CSS, zatímco Astro podporuje Scoped CSS prostřednictvím `<style>` tag. Základní myšlenkou obou je izolace stylu, ale syntaxe je odlišná.

Během migrace HagiCode jsme převedli nejsložitější moduly CSS do rozsahových stylů Astro. To se vlastně ukázalo jako dobrá věc, protože v `.astro` soubory, styly a šablony žijí ve stejném souboru, díky čemuž je údržba intuitivnější.

**Před refaktorizací:**
```css
/* Tabs.module.css */
.wrapper { background: var(--ifm-background-color); }
```

**Po refaktorizaci (Astro Scoped):**
```html
<!-- Tabs.astro -->
<div class="tabs-wrapper">
  <slot />
</div>

<style>
  .tabs-wrapper {
    /* Use CSS variables directly to adapt to the theme */
    background: var(--bg-color);
    padding: 1rem;
  }
</style>
```

Zároveň jsme sjednotili globální systém proměnných CSS a využili možnosti Astro s ohledem na prostředí, abychom zajistili plynulé přepínání tmavého režimu mezi stránkami.

## Nástrahy, na které jsme narazili v praxi, a jak jsme je vyřešili

Během skutečné migrace HagiCode jsme narazili na několik problémů. Zde je několik nejtypičtějších.

### 1. Proměnné body bolesti dráhy a prostředí

HagiCode podporuje nasazení podcesty, jako je nasazení v podadresáři GitHub Pages. V Docusauru, `baseUrl` je zpracováno automaticky. V Astro však musíme být opatrnější při zpracování odkazů na obrázky a požadavků API.

Zavedli jsme mechanismus proměnných prostředí, abychom to důsledně spravovali:

```javascript
// Handle paths in the build script
const getBasePath = () => import.meta.env.VITE_SITE_BASE || '/';
```

Ujistěte se, že nekódujete cesty začínající na `/` ve vašem kódu. Ve vývoji versus výroba nebo po konfiguraci a `base` cestu, může to způsobit 404 pro aktiva.

### 2. Kompatibilita se skripty CommonJS

Na našem starém webu byly některé skripty Node.js používané pro úkoly, jako je automatické načítání dat metrik a aktualizace mapy webu, a byly napsány v CommonJS (`require`). Astro a moderní nástroje pro sestavení plně přijaly moduly ES (`import`/`export`).

Pokud máte také podobné skripty, nezapomeňte je všechny zrefaktorovat na moduly ES. To je směr, kterým se ekosystém ubírá, a čím dříve změnu provedete, tím méně problémů budete mít později.

```javascript
// Old way
const fs = require('fs');

// New way
import fs from 'fs';
```

### 3. Nezapomeňte na SEO a přesměrování

Vyhledávače již indexovaly staré stránky HagiCode Docusaurus. Pokud přejdete přímo na Astro a změní se struktura URL, můžete skončit s velkým počtem 404 a velkým poklesem hodnocení ve vyhledávání.

V Astro jsme nakonfigurovali pravidla přesměrování:

```javascript
// astro.config.mjs
export default defineConfig({
  redirects: {
    '/docs/old-path': '/docs/new-path',
    // Map old links to new links in bulk
  }
});
```

Nebo to můžete zvládnout na vrstvě konfigurace serveru. Ujistěte se, že staré odkazy lze přesměrovat 301 na nové adresy, protože to je pro SEO zásadní.

## Shrnutí

Pro HagiCode nebyla migrace z Docusaurus na Astro jen upgradem rámce. Jednalo se také o praktickou realizaci filozofie „performance first“.

**Co jsme získali:**
*   **Vynikající skóre Lighthouse**: Po migraci se skóre výkonu webu HagiCode snadno přiblížilo perfektnímu skóre.
*   **Rychlejší sestavení**: Možnosti inkrementálního sestavení Astro zkracují dobu vydání aktualizací dokumentace na polovinu.
*   **Zachovaná flexibilita**: Díky Islands Architecture jsme neobětovali žádné interaktivní funkce a mohli jsme stále používat React tam, kde bylo potřeba.

Pokud také udržujete web orientovaný na dokumentaci a potýkáte se s velikostí balíčku nebo rychlostí načítání, Astro stojí za vyzkoušení. Přestože proces migrace vyžaduje určitý chirurgický zákrok, jako je přejmenování PCode na HagiCode a přesouvání komponent po jedné, díky hedvábně hladkému uživatelskému dojmu, který získáte, se to absolutně vyplatí.

Systém sestavování sdílený v tomto článku je přesný přístup, který jsme vyvinuli skutečnými pokusy a omyly při sestavování HagiCode. Pokud považujete tento přístup za hodnotný, vypovídá to něco o naší technické síle a samotný HagiCode pravděpodobně také stojí za bližší prozkoumání.
