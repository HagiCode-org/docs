## Hagicode.Libs: Inženýrská praxe pro sjednocenou integraci více rozhraní AI Coding Assistant CLI

> Během vývoje projektu HagiCode jsme potřebovali integrovat více CLI asistenta kódování AI současně, včetně Claude Code, Codex a CodeBuddy. Každé CLI má různá rozhraní, parametry a výstupní formáty a opakovaný integrační kód ztěžoval údržbu projektu. V tomto článku sdílíme, jak jsme vytvořili jednotnou abstrakční vrstvu s HagiCode.Libs, abychom vyřešili tento technický problém. Dalo by se také říci, že je to prostě nějaká těžce nasbíraná zkušenost z nástrah, na které jsme už narazili.

## Pozadí

Trh s asistenty kódování AI je nyní docela živý. Kromě Claude Code existují také OpenAI's Codex, Zhipu's CodeBuddy a další. Jako projekt asistenta kódování AI potřebuje HagiCode tyto různé nástroje CLI integrovat do více dílčích projektů, včetně desktopu, backendu a webu.

Zpočátku se problém dal zvládnout. Integrace jednoho CLI představovala pouze několik stovek řádků kódu. Ale jak počet CLI, které jsme potřebovali podporovat, neustále rostl, věci začaly být zamotané.

Každé CLI má svůj vlastní formát argumentů příkazového řádku, různé požadavky na proměnné prostředí a širokou škálu výstupních formátů. Některé vydávají JSON, některé streamované JSON a některé prostý text. Kromě toho existují problémy s kompatibilitou mezi platformami. Zjišťování spustitelných souborů a správa procesů fungují mezi systémy Windows a Unix velmi odlišně, takže duplikace kódu neustále narůstala. Ve skutečnosti to bylo jen o něco více Ctrl+C a Ctrl+V, ale údržba se rychle stala bolestivou.

Nejvíce frustrující bylo, že pokaždé, když jsme chtěli přidat podporu pro novou schopnost CLI, museli jsme v několika projektech změnit stejný kód. Tento přístup zjevně nebyl dlouhodobě udržitelný. Kód má také temperament; duplikovat to příliš mnohokrát a začne to způsobovat potíže.

## O HagiCode

Přístup sdílený v tomto článku vychází z našich praktických zkušeností v [HagiCode](https://hagicode.com) projekt. HagiCode je projekt asistenta kódování AI s otevřeným zdrojovým kódem, který potřebuje spravovat více podprojektů současně, včetně rozšíření frontendu VSCode, backendových služeb AI a desktopového klienta pro více platforem. Svým způsobem to bylo přesně toto komplexní, vícejazyčné a multiplatformní prostředí, které vedlo ke zrodu HagiCode.Libs. Dalo by se říct, že jsme k tomu byli donuceni, a tak to bude.

## Analýza: Hledání společného základu

Ačkoli každý z těchto CLI asistenta kódování AI má své vlastní charakteristiky, z technického hlediska sdílejí několik zjevných rysů:

**Podobné vzorce interakce**: všechny zahájí proces CLI, odešlou výzvu, přijmou odezvy ze streamování, analyzují zprávy a poté buď ukončí, nebo v relaci pokračují. Na konci dne celý tok sleduje stejnou základní formu.

**Podobné potřeby konfigurace**: všechny potřebují autentizaci pomocí klíče API, nastavení pracovního adresáře, výběr modelu, kontrolu oprávnění nástrojů a správu relací. Koneckonců, každý se živí API; rozdíly jsou většinou věcí chuti.

**Stejné výzvy napříč platformami**: všechny potřebují vyřešit spustitelné řešení cesty (`claude` vs `claude.exe` vs `/usr/local/bin/claude`), spouštění procesu a zpracování proměnných prostředí, escapování příkazů shellu a konstrukce argumentů. Práce napříč platformami je bolestivá bez ohledu na to, jak ji popíšete. Pouze lidé, kteří vstoupili do pasti, skutečně chápou rozdíl mezi Windows a Unixem.

Na základě této analýzy jsme potřebovali jednotnou abstraktní vrstvu, která by mohla poskytnout konzistentní rozhraní, zapouzdřit logiku zjišťování CLI napříč platformami, zvládnout analýzu streamovaného výstupu a podporovat vkládání závislostí i scénáře bez DI. Je to ten druh problému, ze kterého vás bolí hlava už jen při pomyšlení na to, ale musíte mu stále čelit. Koneckonců je to náš vlastní projekt, takže ho musíme dokončit, i kdybychom si ho museli probrečet.

## Řešení: HagiCode.Libs

Vytvořili jsme **HagiCode.Libs**, odlehčený pracovní prostor knihovny .NET 10 vydaný pod licencí MIT a nyní zveřejněný na [GitHub](https://github.com/HagiCode-org/Hagicode.Libs). Možná to není nějaké světoborné mistrovské dílo, ale je skutečně užitečné pro řešení skutečných problémů.

### Struktura projektu

```
HagiCode.Libs/
├── src/
│   ├── HagiCode.Libs.Core/           # Core capabilities
│   │   ├── Discovery/                 # CLI executable discovery
│   │   ├── Process/                   # Cross-platform process management
│   │   ├── Transport/                 # Streaming message transport
│   │   └── Environment/               # Runtime environment resolution
│   ├── HagiCode.Libs.Providers/       # Provider implementations
│   │   ├── ClaudeCode/                # Claude Code provider
│   │   ├── Codex/                     # Codex provider
│   │   └── Codebuddy/                 # CodeBuddy provider
│   ├── HagiCode.Libs.ConsoleTesting/  # Testing framework
│   ├── HagiCode.Libs.ClaudeCode.Console/
│   ├── HagiCode.Libs.Codex.Console/
│   └── HagiCode.Libs.Codebuddy.Console/
└── tests/                             # xUnit tests
```

### Cíle designu

Při navrhování HagiCode.Libs jsme se řídili několika zásadami. Všechny vzešly z lekcí, které se naučily těžce:

**Nulové závislosti na frameworku**: nezávisí na ABP ani na žádném jiném velkém frameworku, díky čemuž je lehký. V dnešní době, čím méně závislostí máte, tím méně vás bolí hlava. Většina lidí už byla alespoň jednou zbita peklem závislosti.

**Podpora napříč platformami**: nativní podpora pro Windows, macOS a Linux bez psaní samostatného kódu pro různé platformy. Jedna kódová základna, která běží všude, je docela dobrá věc.

**Zpracování streamování**: Výstup CLI je zpracován pomocí asynchronních proudů, které mnohem lépe zapadají do moderních programovacích vzorů .NET. Časy se mění a async je králem.

**Flexibilní integrace**: podporuje scénáře vkládání závislostí a zároveň umožňuje přímou konkretizaci. Různí lidé mají různé preference, takže jsme chtěli, aby to bylo pohodlné v obou směrech.

## Jak jej používat

### Prostřednictvím injekce závislosti

Pokud váš projekt již používá vkládání závislostí, jako je ASP.NET Core nebo obecný hostitel, můžete jej integrovat přímo. Je to maličkost, ale slušně vychovaná:

```csharp
using HagiCode.Libs.Providers;
using Microsoft.Extensions.DependencyInjection;

var services = new ServiceCollection();
services.AddHagiCodeLibs();

await using var provider = services.BuildServiceProvider();
var claude = provider.GetRequiredService<ICliProvider<ClaudeCodeOptions>>();

var options = new ClaudeCodeOptions
{
    ApiKey = "your-api-key",
    Model = "claude-sonnet-4-20250514"
};

await foreach (var message in claude.ExecuteAsync(options, "Hello, Claude!"))
{
    Console.WriteLine($"{message.Type}: {message.Content}");
}
```

### Přímá konkretizace

Pokud píšete jednoduchý skript nebo pracujete ve scénáři bez DI, funguje také přímé vytvoření instance. Jednoduše řečeno, záleží na vašich osobních preferencích:

```csharp
var claude = new ClaudeCodeProvider();
var options = new ClaudeCodeOptions
{
    ApiKey = "sk-ant-xxx",
    Model = "claude-sonnet-4-20250514"
};

await foreach (var message in claude.ExecuteAsync(options, "Help me write a quicksort"))
{
    // Handle messages
}
```

Oba přístupy používají stejnou základní implementaci, takže si můžete vybrat styl integrace, který nejlépe vyhovuje vašemu projektu. V tomto světě neexistuje žádná univerzální správná odpověď. To, co vám vyhovuje, je nejlepší varianta. Může to znít jako klišé, ale je to tak.

## Praktické zkušenosti

### 1. Vyhrazené testovací konzole

Každý poskytovatel má svůj vlastní projekt testovací konzoly, což usnadňuje nezávislé ověření integrace. Testování je jednou z věcí, kde pokud to chcete udělat, měli byste to udělat správně:

```bash
# Claude Code tests
dotnet run --project src/HagiCode.Libs.ClaudeCode.Console -- --test-provider
dotnet run --project src/HagiCode.Libs.ClaudeCode.Console -- --test-all claude

# CodeBuddy tests
dotnet run --project src/HagiCode.Libs.Codebuddy.Console -- --test-provider codebuddy-cli

# Codex tests
dotnet run --project src/HagiCode.Libs.Codex.Console -- --test-provider codex-cli
```

Testovací scénáře pokrývají několik klíčových případů:
- **Ping**: kontrola stavu pro potvrzení dostupnosti CLI
- **Simple Prompt**: základní rychlý test
- **Složitá výzva**: test konverzace s více otáčkami
- **Obnovení/obnovení relace**: test obnovení relace
- **Analýza úložiště**: test analýzy úložiště

Tento design samostatné testovací konzoly je užitečný zejména při ladění, protože nám umožňuje rychle zjistit, zda je problém ve vrstvě HagiCode.Libs nebo v samotném CLI. Ladění je opravdu jen o hledání toho, kde je problém. Jakmile je směr správný, jste již v polovině.

### 2. Ověření CI/CD napříč platformami

Kompatibilita mezi platformami je jedním z hlavních cílů HagiCode.Libs. Nakonfigurovali jsme pracovní postup GitHub Actions `.github/workflows/cli-discovery-cross-platform.yml` spustit skutečné ověřování zjišťování CLI napříč `ubuntu-latest`, `macos-latest`a `windows-latest`.

To zajišťuje, že každá změna kódu nenaruší kompatibilitu napříč platformami. Během místního vývoje jej můžete také reprodukovat pomocí následujících příkazů. Nemůžete přece chtít, aby CI vzala vinu za všechno. Vaše místní prostředí by to také mělo umět spustit:

```bash
npm install --global @anthropic-ai/claude-code@2.1.79
HAGICODE_REAL_CLI_TESTS=1 dotnet test --filter "Category=RealCli"
```

### 3. Zpracování toku zpráv

HagiCode.Libs používá ke zpracování výstupu CLI asynchronní proudy. Ve srovnání s tradičními zpětnými voláními nebo přístupy založenými na událostech to mnohem lépe vyhovuje asynchronnímu programovacímu stylu moderního .NET. Nakonec to je prostě to, jak se technologie posouvá kupředu, ať se to někomu líbí nebo ne:

```csharp
public async IAsyncEnumerable<CliMessage> ExecuteAsync(
    TOptions options,
    string prompt,
    [EnumeratorCancellation] CancellationToken cancellationToken = default)
{
    // Start the CLI process
    // Parse streaming JSON output
    // Yield the CliMessage sequence
}
```

Mezi typy zpráv patří:
- `user`: uživatelská zpráva
- `assistant`: odpověď asistenta
- `tool_use`: vyvolání nástroje
- `result`: konec relace

Tento design umožňuje volajícím flexibilně zpracovávat streamovaný výstup, ať už pro zobrazení v reálném čase, následné zpracování s vyrovnávací pamětí nebo přeposílání na jiné služby. Proč se obávat, zda je obloha slunečná nebo zatažená? Důležité je, že jakmile se nápad otevře, můžete jej použít, jak chcete.

### 4. Průzkum úložiště Git

Modul HagiCode.Libs.Exploration poskytuje zjišťování a kontrolu stavu úložiště Git, což je užitečné zejména ve scénářích analýzy úložiště. Tato funkce se také zrodila z nutnosti, protože HagiCode potřebuje analyzovat úložiště:

```csharp
// Discover Git repositories
var repositories = await GitRepositoryDiscovery.DiscoverAsync("/path/to/search");

// Get repository information
var info = await GitRepository.GetInfoAsync(repoPath);
Console.WriteLine($"Branch: {info.Branch}, Remote: {info.RemoteUrl}");
Console.WriteLine($"Has uncommitted changes: {info.HasUncommittedChanges}");
```

Schopnosti analýzy kódu HagiCode využívají tento modul k identifikaci struktury projektu a stavu Git. Je to dobrý příklad plného využití toho, co jsme vybudovali.

## Věci k poznámce

Na základě naší praxe v projektu HagiCode existuje několik bodů, které si zaslouží zvláštní pozornost. Všechno jsou to skutečné problémy, které je třeba pečlivě řešit:

**Zabezpečení klíče API**: nezadávejte do kódu klíče API napevno. Místo toho použijte proměnné prostředí nebo správu konfigurace. HagiCode.Libs podporuje předávání konfigurace přes objekty Options, což usnadňuje integraci s různými zdroji konfigurace. Pokud jde o bezpečnost, neexistuje nic jako být příliš opatrný.

**Připnutí verze CLI**: v CI/CD připínáme konkrétní verze, jako např `@anthropic-ai/claude-code@2.1.79`, aby se snížila nejistota způsobená posunem verze. V lokálním vývoji je také dobré používat pevné verze. Verze může být bolestivá. Pokud verze nepřipnete, problém vás velmi rychle poučí.

**Kategorizace testů**: výchozí testy používají falešné poskytovatele, aby byly deterministické a rychlé, zatímco skutečné testy CLI musí být explicitně povoleny. To poskytuje rychlou zpětnou vazbu CI a zároveň umožňuje v případě potřeby ověření v reálném prostředí. Dosáhnout této rovnováhy není nikdy snadné. Rychlost a stabilita vždy vyžadují kompromisy.

**Správa relací**: různá rozhraní CLI mají různé mechanismy obnovy relací. Claude Code používá `.claude/` adresář pro ukládání relací, zatímco Codex a CodeBuddy mají každý svůj vlastní přístup. Při jejich používání si nezapomeňte zkontrolovat jejich příslušnou dokumentaci a porozumět podrobnostem jejich mechanismů perzistence relace. Není na škodu tomu jasně porozumět.
