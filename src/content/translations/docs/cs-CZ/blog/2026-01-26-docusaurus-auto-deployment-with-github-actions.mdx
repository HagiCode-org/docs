## Přidání podpory automatického nasazení stránek GitHub do HagiCode

> Tento projekt měl původně kódové označení PCode a nyní byl oficiálně přejmenován na HagiCode. Tento článek dokumentuje, jak jsme zavedli automatizované nasazování statických stránek, takže publikování obsahu je snadné jako pití vody.

## Pozadí / Úvod

Během vývoje HagiCode jsme narazili na velmi praktický problém: jak se naše dokumentace a návrhy neustále rozrůstaly, efektivní správa a prezentace tohoto obsahu byla stále naléhavější. Rozhodli jsme se použít GitHub Pages k hostování našeho statického webu, ale vytvoření a nasazení ručně bylo příliš těžkopádné. Každá změna vyžadovala místní sestavení, zabalení a poté ruční vložení `gh-pages` větev. To bylo nejen neefektivní, ale také snadné se mýlit.

Abychom tento problém vyřešili, především proto, že jsme si chtěli ušetřit určité úsilí, potřebovali jsme automatizovaný pracovní postup nasazení. Tento článek podrobně zaznamenává, jak jsme do projektu HagiCode přidali podporu automatického nasazení na bázi GitHub Actions, takže se můžeme soustředit na psaní obsahu a nechat automatizaci, aby se postarala o zbytek.

## Analýza cílů

Než se pustíme do implementace, musíme si nejprve ujasnit, co má tento úkol vlastně plnit. Ostření sekery totiž nezdržuje sekání palivového dříví.

### Základní požadavky

1.  **Automatické sestavení**: Automaticky spouštět pracovní postup sestavení, když je kód vložen do `main` větev.
2.  **Automatické nasazení**: Po úspěšném sestavení automaticky nasaďte vygenerované statické soubory na stránky GitHub.
3.  **Konzistence prostředí**: Ujistěte se, že prostředí CI odpovídá místnímu prostředí sestavení, abyste se vyhnuli klasickým rozpakům typu „funguje to lokálně, ale selže online“.

### Technická volba

Vzhledem k tomu, že HagiCode je postaven na Docusauru, velmi oblíbeném generátoru statických stránek React, můžeme k dosažení tohoto cíle použít akce GitHub.

## Konfigurace pracovního postupu akcí GitHub

GitHub Actions je služba CI/CD poskytovaná GitHubem. Definováním souborů pracovního postupu ve formátu YAML uvnitř úložiště můžeme přizpůsobit všechny druhy automatizačních úloh.

### Vytvoření souboru pracovního postupu

Musíme vytvořit nový konfigurační soubor v `.github/workflows` adresář v kořenovém adresáři projektu, například `deploy.yml`. Pokud adresář ještě neexistuje, nejprve jej vytvořte.

Základní logika tohoto konfiguračního souboru je následující:

1.  **Spouštěcí stav**: Poslouchejte `push` události na `main` větev.
2.  **Běhové prostředí**: Nejnovější vydání Ubuntu.
3.  **Postup sestavení**:
    -   Podívejte se na kód
    -   Nainstalujte Node.js
    -   Nainstalujte závislosti (`npm install`)
    -   Vytvořte statické soubory (`npm run build`)
4.  **Krok nasazení**: Použijte oficiální `action-gh-pages` akce, která posune výstup sestavení do `gh-pages` větev.

### Konfigurační kód klíče

Níže je konfigurační šablona, kterou jsme nakonec přijali:

```yaml
name: Deploy to GitHub Pages

# Trigger condition: run when code is pushed to the main branch
on:
  push:
    branches:
      - main
    # Add path filters if needed, for example only build when docs change
    # paths:
    #   - 'docs/**'
    #   - 'package.json'

# Set permissions, which is important for GitHub Pages deployment
permissions:
  contents: read
  pages: write
  id-token: write

# Concurrency control: cancel older builds for the same branch
concurrency:
  group: "pages"
  cancel-in-progress: false

jobs:
  build:
    runs-on: ubuntu-latest
    steps:
      - name: Checkout
        uses: actions/checkout@v4
        # Note: fetch-depth: 0 is required, otherwise the build version may be inaccurate
        with:
          fetch-depth: 0

      - name: Setup Node
        uses: actions/setup-node@v4
        with:
          node-version: 20 # Keep this consistent with the local development environment
          cache: 'npm'     # Enabling cache can speed up the build process

      - name: Install dependencies
        run: npm ci
        # Use npm ci instead of npm install because it is faster and stricter, which makes it better for CI

      - name: Build website
        run: npm run build
        env:
          # Configure environment variables here if your site build requires them
          # NODE_ENV: production
          # PUBLIC_URL: /your-repo-name
          
      - name: Upload artifact
        uses: actions/upload-pages-artifact@v3
        with:
          path: ./build # Default Docusaurus output directory

  deploy:
    environment:
      name: github-pages
      url: ${{ steps.deployment.outputs.page_url }}
    runs-on: ubuntu-latest
    needs: build
    steps:
      - name: Deploy to GitHub Pages
        id: deployment
        uses: actions/deploy-pages@v4
```

## Běžná úskalí při implementaci

V praxi jsme narazili na několik problémů. Sdílím je zde v naději, že se jim ostatní mohou vyhnout, nebo si alespoň připravit řešení předem.

### 1. Problémy s oprávněním k tokenu GitHub

Když jsme věci poprvé nastavili, nasazení stále selhávalo s 403 (Zakázáno). Po dlouhém hledání jsme zjistili, že výchozí nastavení GitHubu `GITHUB_TOKEN` neměl oprávnění zapisovat do Pages.

**Řešení**: V úložišti `Settings` -> `Actions` -> `General` -> `Workflow permissions`, nezapomeňte vybrat **"Oprávnění ke čtení a zápisu"**.

### 2. Chybná cesta k adresáři sestavení

Docusaurus umístí vestavěné statické soubory do `build` adresář ve výchozím nastavení. Některé projekty však mohou být nakonfigurovány odlišně. Například Create React App je výchozí `build`, zatímco Vite je výchozí `dist`. Pokud Actions hlásí, že nemůže najít výstupní soubory, zkontrolujte výstupní cestu `docusaurus.config.js`.

### 3. Problémy s podcestou

Pokud vaše úložiště není domovskou stránkou uživatele, znamená to, že není `username.github.io`, ale místo toho stránka projektu jako např `username.github.io/project-name`, pak musíte nakonfigurovat `baseUrl`.

In `docusaurus.config.js`:

```javascript
module.exports = {
  // ...
  url: 'https://hagicode.com', // Your Hagicode URL
  baseUrl: '/', // Deploy from the root path
  // ...
};
```
Tento detail lze snadno přehlédnout. Pokud je nakonfigurován nesprávně, stránka se může načíst jako prázdná obrazovka, protože cesty k aktivům nelze vyřešit.

## Ověření výsledku

Po konfiguraci všeho a natlačení kódu můžeme zamířit na kartu **Akce** v repozitáři GitHub a sledovat show.

Během pracovního postupu uvidíte žlutý kruh, a jakmile se změní na zelenou, znamená to úspěch. Pokud se změní na červenou, klikněte dovnitř a prohlédněte si protokoly. Ve většině případů můžete problém sledovat tam, obvykle jde o překlep nebo nesprávnou konfiguraci cesty.

Jakmile se stavba podaří, navštivte `https://<your-username>.github.io/<repository-name>/` a uvidíte čerstvě nasazený web.

## Shrnutí

Zavedením GitHub Actions jsme úspěšně zautomatizovali nasazení pro web s dokumentací HagiCode. To nejen šetří čas dříve strávený ručními operacemi, ale co je důležitější, standardizuje pracovní postup vydání. Nyní, bez ohledu na to, který člen týmu aktualizuje dokumentaci, jakmile je změna začleněna `main`, nejnovější obsah bude zveřejněn o několik minut později.

**Základní výhody**:
- **Vyšší efektivita**: Přešli jsme z „ručního balení a ručního nahrávání“ na „kód se rovná uvolnění“.
- **Méně chyb**: Eliminovali jsme možnost lidské chyby při ručních operacích.
- **Lepší vývojářský zážitek**: Vývojáři se mohou více zaměřit na kvalitu obsahu, místo aby byli rozptylováni únavnými kroky nasazení.

Ačkoli je konfigurace CI/CD zpočátku trochu problematická, zejména se všemi oprávněními a podrobnostmi o cestě, jde o jednorázovou investici s velkou dlouhodobou návratností. Důrazně doporučuji, aby každý projekt statického webu přijal podobný automatizovaný pracovní postup.
