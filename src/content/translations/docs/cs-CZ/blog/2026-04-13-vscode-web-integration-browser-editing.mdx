## VSCode a kódový server: Výběr řešení pro úpravu kódu založeného na prohlížeči

> Při vytváření možností úpravy kódu v prohlížeči stojí vývojáři před klíčovou volbou: použít oficiální verzi VSCode `code serve-web` nebo přijmout komunitu řízenou `code-server` řešení? Toto rozhodnutí ovlivňuje nejen technickou architekturu, ale také soulad s licencí a flexibilitu nasazení.

## Pozadí

Technický výběr je hodně podobný výběru životní cesty. Jakmile si jeden vyberete, musíte s ním obvykle chodit a pozdější přechod se může velmi prodražit.

V éře programování podporovaného umělou inteligencí je stále důležitější úprava kódu pomocí prohlížeče. Uživatelé očekávají, že poté, co asistent AI dokončí analýzu kódu, mohou okamžitě otevřít editor ve stejné relaci prohlížeče a provádět změny bez přepínání aplikací. Tento druh bezproblémového zážitku by prostě měl být k dispozici, když ho potřebujete.

Při implementaci této funkce však vývojáři čelí kritické technické volbě: měli by použít oficiální verzi VSCode `code serve-web` nebo komunitou řízený `code-server` řešení?

Každá možnost má své silné stránky a kompromisy a špatná volba může později způsobit spoustu problémů. Jedním z příkladů je licencování: pokud až po spuštění zjistíte, že váš produkt není v souladu s licencí, je již příliš pozdě. Nasazení je něco jiného: řešení může ve vývoji fungovat perfektně a poté, co se přesunete do kontejnerů, narazí na nejrůznější problémy. To jsou přesně ty druhy nástrah, kterým se týmy chtějí vyhnout.

## Rozdíly v licencích (nejdůležitější)

Toto je nejzásadnější rozdíl mezi oběma řešeními a první faktor, který jsme při hodnocení zvažovali. Při technické volbě je důležité předem porozumět právním rizikům.

**kódový server**
- Licence MIT, plně open source
- Spravuje Coder.com s aktivní komunitou
- Zdarma ke komerčnímu použití, úpravám a distribuci
- Žádná omezení scénářů použití

**VSCode `code serve-web`**
- Část produktu Microsoft VSCode
- Používá licenci společnosti Microsoft (licence VS Code zahrnuje omezení komerčního použití)
- Primárně určeno pro jednotlivé vývojáře
- Podnikové nasazení může vyžadovat další kontrolu komerčních licencí

Z hlediska licencování je kódový server přátelštější ke komerčním projektům. To je něco, co musíte promyslet během plánování produktu, protože pozdější migrace může být velmi nákladná.

## Rozdíly v nasazení

Po vyřízení licencí je dalším problémem nasazení. To přímo ovlivňuje vaše provozní náklady a architektonický návrh.

**kódový server**
- Samostatná aplikace Node.js, kterou lze nasadit nezávisle
- Podporuje více zdrojů runtime:
  - Přímé zadání cesty ke spustitelnému souboru
  - Vyhledejte to přes systém PATH
  - Automatická detekce prostředí NVM Node.js 22.x
- Není třeba instalovat desktopovou aplikaci VSCode na server
- Snadnější nasazení v kontejnerech

**VSCode `code serve-web`**
- Musí záviset na místně nainstalovaném rozhraní CLI VSCode
- Vyžaduje k dispozici `code` příkaz na hostitelském počítači
- Systém odfiltruje obaly VS Code Remote CLI
- Primárně navržený pro scénáře místního rozvoje

code-server se lépe hodí pro scénáře nasazení serverů a kontejnerů. Pokud váš produkt potřebuje běžet v Dockeru nebo vaši uživatelé nemají nainstalovaný VSCode, kódový server je obvykle tou správnou volbou.

## Rozdíly v parametrech funkce

Obě řešení se také liší v několika parametrech funkcí. Rozdíly nejsou velké, ale mohou vytvářet integrační tření v reálném světě.

| Funkce | kódový server | kód server-web |
|------|-------------|----------------|
| Cesta veřejné základny | `/` (konfigurovatelné) | `/vscode-server` (opraveno) |
| Autentizace | `--auth` parametr s více režimy | `--connection-token` / `--without-connection-token` |
| Datový adresář | `{DataDir}/code-server` | `{DataDir}/vscode-serve-web` |
| Telemetrie | Ve výchozím nastavení zakázáno s `--disable-telemetry` | Závisí na nastavení VSCode |
| Kontroly aktualizací | Lze deaktivovat pomocí `--disable-update-check` | Závisí na nastavení VSCode |

Tyto rozdíly vyžadují při integraci zvláštní pozornost. Například různé cesty URL znamenají, že váš kód frontendu vyžaduje speciální zpracování.

## Rozdíly v detekci dostupnosti

Při implementaci přepínání editorů se také liší logika zjišťování dostupnosti.

**kódový server**
- Vždy vráceno jako viditelná implementace
- Stále zobrazeno, i když není k dispozici, s `install-required` stavu
- Podporuje automatickou detekci prostředí NVM Node.js 22.x

**code-servis-web**
- Viditelné pouze u místního `code` Je detekováno CLI
- Pokud není k dispozici, frontend tuto možnost automaticky skryje
- Závisí na místním stavu instalace VSCode

Tento rozdíl přímo ovlivňuje uživatelský dojem. code-server je transparentnější: uživatelé vidí možnost a chápou, že je stále nutná instalace. code serve-web je skrytější: uživatelé si možná ani neuvědomují, že tato možnost existuje. Který přístup je lepší závisí na umístění produktu.

## Architektura duální implementace HagiCode

Po hloubkové analýze přijal projekt HagiCode architekturu duální implementace, která podporuje obě řešení na architektonické úrovni.

### Výchozí kódový server

```csharp
// The default active implementation is code-server
// If an explicit activeImplementation is saved, try that implementation first
// If the requested implementation is unavailable, the resolver tries the other one
// If a fallback occurs, return fallbackReason
```

Standardně používáme kódový server hlavně kvůli flexibilitě licencování a nasazení. Nicméně pro uživatele, kteří již mají místní prostředí VSCode, je code serve-web také solidní volbou.

### Selektor implementace

`CodeServerImplementationResolver` je zodpovědný za sjednocení:
- Výběr implementace během zahřívání při spuštění
- Výběr implementace při čtení stavu
- Výběr implementace při otevírání projektů
- Výběr implementace při otevírání Vaultů

Tento návrh umožňuje systému pružně reagovat na různé scénáře a uživatelé si mohou vybrat implementaci, která nejlépe odpovídá jejich prostředí.

### Pravidla pro úpravu frontendu

```typescript
// When localCodeAvailable=false, do not show code serve-web
// When localCodeAvailable=true, show the code serve-web configuration
```

Frontend automaticky zobrazuje dostupné možnosti na základě prostředí, takže uživatelé nejsou zmateni funkcemi, které nemohou používat.

## Praktický průvodce

Po vší té teorii, čemu byste měli věnovat pozornost při skutečném nasazení? Nakonec je důležitá implementace.

### Doporučení k nasazení Dockeru

Pro kontejnerizované nasazení je lepší volbou kódový server:

```dockerfile
# Use the official code-server image directly
FROM codercom/code-server:latest

# Or install through npm
RUN npm install -g code-server
```

To řeší problém v jediné vrstvě bez nutnosti další instalace VSCode.

### Příklady konfigurace

**konfigurace kódového serveru**
```json
{
  "vscodeServer": {
    "enabled": true,
    "activeImplementation": "code-server",
    "codeServer": {
      "host": "0.0.0.0",
      "port": 8080,
      "executablePath": "",
      "authMode": "none"
    }
  }
}
```

**konfigurace webu pro poskytování kódu**
```json
{
  "vscodeServer": {
    "enabled": true,
    "activeImplementation": "serve-web",
    "serveWeb": {
      "host": "0.0.0.0",
      "port": 8080,
      "executablePath": "/usr/local/bin/code"
    }
  }
}
```

Konfigurace může být napoprvé trochu únavná, ale jakmile je na svém místě, údržba věcí je mnohem snazší.

### Konstrukční rozdíly URL

**kódový server**
```
http://localhost:8080/?folder=/path/to/project&vscode-lang=zh-CN
```

**code-servis-web**
```
http://localhost:8080/vscode-server/?folder=/path/to/project&tkn=xxx&vscode-lang=zh-CN
```

Věnujte pozornost rozdílům v cestách a parametrech. Při integraci s nimi musíte zacházet samostatně.

### Spínací implementace

Systém podporuje přepínání za běhu a automaticky zastaví předchozí implementaci při přepínání:

```csharp
// VsCodeServerManager automatically handles mutual exclusion
// When switching activeImplementation, the old implementation will not keep running in the background
```

Tento návrh umožňuje uživatelům kdykoli vyzkoušet různé implementace a najít možnost, která jim nejlépe vyhovuje.

### Sledování stavu

```typescript
const { settings, runtime } = await getVsCodeServerSettings();

// runtime.activeImplementation: "code-server" | "serve-web"
// runtime.fallbackReason: reason for switching
// runtime.status: "running" | "starting" | "stopped" | "unhealthy"
```

Když je stav viditelný, uživatelé mohou rychle určit, zda problém pochází ze strany serveru nebo z jejich vlastní operace.
