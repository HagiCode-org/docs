> Fráze automatické opakování vypadá jako malý přepínač, ale jakmile jej vložíte do skutečného inženýrského prostředí, nic takového není. Ahoj všichni, jsem tvůrce HagiCode Yu Kun. Dnes nechci obchodovat s prázdnými řečmi. Chci mluvit o tom, jak by se ve skutečnosti mělo provádět automatické opakování pro Agent CLI, jako je Claude Code a Codex, aby se mohl zotavit z výjimek a vyhnout se přetahování systému do nekonečného opakovaného provádění.

## Pozadí

Pokud jste také v poslední době pracovali na kódování AI, pravděpodobně jste již narazili na tento druh problému: úloha se nezdaří okamžitě, ale přeruší se v polovině provádění.

V běžném požadavku HTTP to často znamená jeho opětovné odeslání, možná s nějakým exponenciálním zpožděním. Ale Agent CLI jsou jiné. Nástroje jako Claude Code a Codex se obvykle spouštějí způsobem streamování a vytlačují výstup kousek po kousku. Během tohoto procesu se mohou také vázat na vlákno, relaci nebo token obnovení. Jinými slovy, otázka nezní jednoduše: "Prošel tento požadavek nebo ne?" Stává se:

- Počítá se obsah, který již byl vydán?
- Může aktuální kontext pokračovat?
- Měla by být tato porucha automaticky obnovena?
- Pokud by měl být obnoven, jak dlouho bychom měli čekat, než to zopakujeme, co bychom měli během opakování odeslat a měli bychom stále znovu použít původní kontext?

Když mnoho týmů staví tuto část poprvé, instinktivně napíše tu nejnaivnější verzi: pokud dojde k chybě, zkuste to ještě jednou. Tato myšlenka je naprosto přirozená, ale jakmile dojde k reálnému projektu, začíná se na povrch vynořovat jeden problém za druhým.

- Některé chyby jsou jasně dočasná selhání, ale jsou považovány za konečné chyby
- Některé chyby se vůbec nevyplatí zkoušet, přesto je systém znovu a znovu přehrává
- S požadavky s vláknem a požadavky bez vlákna se zachází úplně stejně
- Strategie ústupu nemá žádné hranice a požadavky na pozadí se samy přetěžují

Při integraci více Agent CLI se HagiCode také vložil do těchto pastí. Zejména na straně Codexu byl prvním problémem, který jsme odhalili, to, že určitý typ zprávy o opětovném připojení nebyl rozpoznán jako opakovatelný terminálový stav, takže mechanismus obnovy, který jsme již neměli šanci, se projevil. Jednoduše řečeno, nebylo to tím, že by systému chybělo automatické opakování. Systém prostě nedokázal rozpoznat, že toto konkrétní selhání stojí za to zkusit znovu.

Stěžejní bod tohoto článku je tedy velmi jasný: **automatické opakování není tlačítko, ale vrstvený design.**

## Proč je automatické opakování pro agenta CLI těžší než běžné opakování

Toto je velmi praktická otázka, takže pojďme rovnou k závěru: obtížnost automatického opakování pro Agent CLI není "zkusit to znovu po několika sekundách", ale "může to stále pokračovat v původním kontextu?"

Můžete si to představit jako dlouhý rozhovor. Běžné opakování API je spíše jako opakované vytáčení, když je telefonní linka obsazená. Agent CLI retry je spíše jako výpadek signálu, když je druhá strana v polovině věty, a pak se musíte rozhodnout, zda zavoláte zpět, zda začnete znovu, když to uděláte, a zda si druhá strana stále pamatuje, kde se konverzace zastavila. Vůbec se nejedná o stejný druh technických problémů.

Konkrétněji, existují čtyři obzvláště typické obtíže.

### 1. Je to streamování

Jakmile je výstup odeslán uživateli, již nezvládnete selhání tak, jako byste to udělali s běžným požadavkem, kdy jej mlčky spolknete a v tichosti zkusíte znovu. Je to proto, že dřívější obsah již byl viděn. Pokud je strategie přehrávání chybná, frontend může snadno zobrazit duplicitní text a nekonzistentní stav a také se může zamotat životní cyklus volání nástrojů. To není metafyzika. Je to strojírenství.

### 2. Obvykle se váže na kontext relace

Poskytovatelé jako Codex se vážou k vláknu a implementace jako Claude Code mají také cíl pokračování nebo ekvivalentní obnovitelný kontext. Skutečným předpokladem pro automatické opakování není jen to, že chyba vypadá jako dočasné selhání, ale také to, že stále existuje nosič, který umožňuje pokračování tohoto provádění.

### 3. Ne každá chyba se vyplatí opakovat

Jitter sítě, časový limit nečinnosti SSE a dočasné výpadky upstreamu obvykle stojí za další pokus. Pokud však čelíte selhání ověřování, ztracenému kontextu nebo poskytovateli, který nemá vůbec žádnou možnost obnovení, pak opakování obvykle neznamená obnovení. Je to generování hluku.

### 4. Potřebuje hranice

Neomezené automatické opakování je téměř vždy špatné. Technologické trendy mohou být na chvíli hlučné, ale inženýrské zákony často zůstávají stabilní po mnoho let. Jedním z nich je, že zotavení po selhání musí mít hranice. Systém musí vědět, kolikrát se může maximálně pokusit, jak dlouho by měl pokaždé čekat a kdy se má zastavit a přiznat, že tohle se opravdu nevzpamatuje.

Kvůli těmto vlastnostem HagiCode nakonec neimplementoval automatické opakování jako několik řádků `try/catch` uvnitř konkrétního poskytovatele. Místo toho jsme jej extrahovali do vrstvy sdílených schopností. Koneckonců, inženýrské problémy je stále třeba řešit inženýrskými metodami.

## Přístup HagiCode: Opakujte pokus od poskytovatele

Aktuální implementaci HagiCode v reálném světě lze zkomprimovat do jedné věty:

**Sdílená vrstva spravuje tok opakování jednotně a každý konkrétní poskytovatel je odpovědný pouze za zodpovězení dvou otázek: stojí tento stav terminálu za opakování a může aktuální kontext stále pokračovat?**

Není to složité, ale je to kritické. Jakmile jsou odpovědnosti takto odděleny, Claude Code, Codex a dokonce i další Agent CLI mohou znovu použít stejnou kostru. Modely se budou měnit, nástroje se budou vyvíjet, pracovní postupy budou upgradovány, ale inženýrský základ tam zůstává.

### Vrstva 1: Ke správě smyčky opakování použijte jednotného koordinátora

Fragment základní implementace v projektu vypadá zhruba takto:

```csharp
internal static class ProviderErrorAutoRetryCoordinator
{
    public static async IAsyncEnumerable<CliMessage> ExecuteAsync(
        string prompt,
        ProviderErrorAutoRetrySettings? settings,
        Func<string, IAsyncEnumerable<CliMessage>> executeAttemptAsync,
        Func<bool> canRetryInSameContext,
        Func<TimeSpan, CancellationToken, Task> delayAsync,
        Func<CliMessage, bool> isRetryableTerminalMessage,
        [EnumeratorCancellation] CancellationToken cancellationToken)
    {
        var normalizedSettings = ProviderErrorAutoRetrySettings.Normalize(settings);
        var retrySchedule = normalizedSettings.Enabled
            ? normalizedSettings.GetRetrySchedule()
            : [];

        for (var attempt = 0; ; attempt++)
        {
            var attemptPrompt = attempt == 0
                ? prompt
                : ProviderErrorAutoRetrySettings.ContinuationPrompt;

            CliMessage? terminalFailure = null;

            await foreach (var message in executeAttemptAsync(attemptPrompt)
                               .WithCancellation(cancellationToken))
            {
                if (isRetryableTerminalMessage(message))
                {
                    terminalFailure = message;
                    break;
                }

                yield return message;
            }

            if (terminalFailure is null)
            {
                yield break;
            }

            if (attempt >= retrySchedule.Count || !canRetryInSameContext())
            {
                yield return terminalFailure;
                yield break;
            }

            await delayAsync(retrySchedule[attempt], cancellationToken);
        }
    }
}
```

To, co tento kód dělá, je ve skutečnosti velmi přímočaré, ale také velmi efektivní.

- Mezilehlé poruchy nejprve neprocházejte přímo; koordinátor rozhodne, zda je vymáhání stále možné
- Teprve po vyčerpání rozpočtu na opakování se konečné selhání skutečně vrátí do horní vrstvy
- Počínaje druhým pokusem se již neodesílá původní výzva; místo toho je jednotně odeslána výzva k pokračování

Proto jsem dříve zdůrazňoval, že automatické opakování není jednoduše „zadat požadavek znovu“. Není to jen záplatování větve výjimky. Řídí životní cyklus exekuce. Může to znít jako jazyk produktového manažera, ale z technického hlediska je to přesně ono.

### Vrstva 2: Snímek zásady opakování

Další problém, který lze velmi snadno přehlédnout, je tento: kdo rozhoduje, zda je pro tento požadavek povoleno automatické opakování?

Odpověď HagiCode není záviset na nějaké „aktuální globální konfiguraci“, ale přeměnit politiku na snímek a nechat ji cestovat společně s tímto požadavkem. Tímto způsobem řazení relací, perzistence zpráv, přesměrování provádění a přizpůsobení poskytovatele neztratí zásady. Jeden úspěšný běh není systém. Trvalý úspěch je systém.

Základní strukturu lze zjednodušit na:

```csharp
public sealed record ProviderErrorAutoRetrySnapshot
{
    public const string DefaultStrategy = "default";

    public bool Enabled { get; init; }

    public string Strategy { get; init; } = DefaultStrategy;

    public static ProviderErrorAutoRetrySnapshot Normalize(bool? enabled, string? strategy)
    {
        return new ProviderErrorAutoRetrySnapshot
        {
            Enabled = enabled ?? true,
            Strategy = string.IsNullOrWhiteSpace(strategy)
                ? DefaultStrategy
                : strategy.Trim()
        };
    }
}
```

Poté se na straně provádění mapuje do objektu nastavení skutečně spotřebovaného poskytovatelem. Hodnota tohoto přístupu je velmi přímá:

- O tom, zda má být povoleno opakování, rozhoduje obchodní vrstva
- Běhové prostředí rozhoduje o tom, jak se má opakovat

Každá strana řídí svůj vlastní problém, aniž by se střetla s tou druhou. Mnoho problémů není nemožné vyřešit. Jejich cena prostě nebyla explicitně uvedena. Přeměna zásady na snímek je v podstatě způsob, jak tyto náklady předem zaúčtovat.

### Vrstva 3: Poskytovatelé rozhodují pouze o stavu terminálu a životaschopnosti kontextu

Jakmile se dostaneme ke konkrétnímu poskytovateli Claude Code nebo Codex, odpovědnost zde ve skutečnosti velmi ztenčí. Můžete to považovat za vylepšení, ne jako náhradu.

Vezmeme-li jako příklad Codex, když se zapojí do sdíleného koordinátora, potřebuje ve skutečnosti pouze tři věci:

```csharp
await foreach (var message in ProviderErrorAutoRetryCoordinator.ExecuteAsync(
                   prompt,
                   options.ProviderErrorAutoRetry,
                   retryPrompt => ExecuteCodexAttemptAsync(...),
                   () => !string.IsNullOrWhiteSpace(resolvedThreadId),
                   DelayAsync,
                   IsRetryableTerminalFailure,
                   cancellationToken))
{
    yield return message;
}
```

Všimnete si, že rozhodnutí specifická pro poskytovatele jsou ve skutečnosti pouze tato dvě:

- `IsRetryableTerminalFailure`
- `canRetryInSameContext`

Codex kontroluje, zda vlákno může stále pokračovat, zatímco Claude Code kontroluje, zda cíl pokračování stále existuje. Zásady stažení, počet opakování a následné výzvy by neměly znovu vymýšlet každý poskytovatel samostatně.

Jakmile je tato vrstva oddělena, náklady na integraci více CLI do HagiCode značně klesnou. Nemusíte duplikovat celý stavový stroj opakování. Musíte pouze zapojit okrajové podmínky daného poskytovatele. Psát rychle není totéž jako psát robustně. Umět něco propojit není totéž jako to propojit dobře. Zprovoznit jej také není totéž jako zajistit jeho údržbu v průběhu času.

## Snadná chyba: Nepovažujte každou chybu za opakovatelnou

V této analýze není bod, který bych chtěl nejvíce vyzdvihnout, „jak implementovat opakování“, ale „jak se vyhnout nesprávným pokusům“.

Původním vstupním bodem problému bylo, že Codex nedokázal rozpoznat jednu zprávu o opětovném připojení. Podle intuice by si mnoho lidí vybralo tu nejmenší možnou opravu: přidat do whitelistu další předponu řetězce. Tato myšlenka není úplně špatná, ale připadá mi to spíše jako řešení v demo fázi než jako dlouhodobě udržovatelné.

Od současné implementace HagiCode již systém udělal krok robustnějším směrem. Už nezírá jen na jednu doslovnou strunu. Místo toho jednotně předává obnovitelné terminálové stavy sdílenému koordinátorovi. Výhody jsou zřejmé:

- Je méně pravděpodobné, že úplně selže kvůli malé změně formulace v jedné zprávě
- Testovací pokrytí může být postaveno kolem obálky terminálového stavu spíše než jednoho pevně zakódovaného textového řádku
- Logika opakování bude v rámci stejného poskytovatele konzistentnější

Samozřejmě zde musí existovat pevná hranice: být obecnější neznamená být povolnější. **Pokud aktuální kontext nemůže pokračovat, pak i když chyba vypadá jako dočasné selhání, neměla by být přehrána naslepo.**

Tento bod je kritický. To, co skutečně přiměje lidi důvěřovat systému, není to, že občas funguje, ale to, že je většinou spolehlivý. Pokud tok mohou udržovat pouze odborníci, pak má ke skutečnému přijetí ještě hodně daleko.

## Tři nejcennější lekce, které je třeba mít v praxi

V tuto chvíli má smysl začít přivádět diskusi zpět k implementační praxi. Pokud plánujete vybudovat podobnou schopnost ve svém vlastním projektu, toto jsou tři pravidla, která důrazně doporučuji nejprve chránit.

### 1. Rozpočet pro opakování musí mít hranice

Aktuální výchozí rytmus ústupu HagiCode je:

- 10 sekund
- 20 sekund
- 60 sekund

Tento rytmus nemusí vyhovovat každému systému, ale existence hranic musí zůstat. V opačném případě automatické opakování rychle přestane být mechanismem obnovy a změní se na zesilovač incidentů. Nespěchejte, abyste mu dali působivé jméno. Nejprve se ujistěte, že věc může přežít dvě iterace uvnitř skutečného týmu.

### 2. Výzva k pokračování by měla být sjednocena

Projekt používá pevnou výzvu k pokračování, takže pozdější pokusy jasně sledují cestu pokračování v aktuálním kontextu, spíše než zahájení zcela nového úplného požadavku. Tato schopnost není okázalá, ale když stavíte skutečný projekt, neobejdete se bez ní. Mnoho věcí, které vypadají jako magie, jsou po rozpadu jen vyleštěným inženýrským procesem.

### 3. Sdílená knihovna i vrstva adaptéru potřebují zrcadlené testy

Zvláště bych k tomuto bodu chtěl říci něco více. Mnoho týmů napíše jednu vrstvu testů ve sdíleném runtime a myslí si, že to pravděpodobně stačí. není.

Důvod, proč se cítím relativně jistý implementací HagiCode, je ten, že obě vrstvy mají testovací pokrytí:

- Sdílený poskytovatel testuje, zda k automatickému pokračování skutečně došlo
- Vrstva adaptéru testuje, zda byly správně zachovány finální chyby a streamingové zprávy

Tentokrát jsem také znovu provedl dvě související testovací skupiny a všech 31 testovacích případů prošlo v obou z nich. Tento výsledek sám o sobě nedokazuje, že návrh je dokonalý, ale dokazuje alespoň jednu věc: současné automatické opakování není papírový návrh. Je to schopnost omezená kódem i testy. Diskuse je levná. Ukaž mi kód. Skvěle se to sem hodí.

Výhody tohoto provedení jsou velmi praktické:

- Logika je zapsána jednou a znovu použita u více poskytovatelů

- Pokračujte v běhu, když kontext existuje, a zastavte se v čase, když neexistuje
- To, co frontend nakonec vidí, je stabilní dokončený stav nebo stav selhání, nikoli hromada opuštěného mezilehlého šumu
