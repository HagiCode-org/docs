## Řešení problémů s autentizací WebSocket prohlížeče: Praktický vzor proxy pro rozpoznávání řeči Doubao

> Prohlížeč WebSocket API nepodporuje vlastní hlavičky HTTP, což představuje výzvu pro služby rozpoznávání řeči, které v hlavičkách vyžadují ověřovací data. Tento článek uvádí, jak jsme tento problém vyřešili v projektu HagiCode pomocí backendového proxy vzoru a jak se tento přístup vyvinul od experimentů na hřišti k produkčnímu použití.


## Pozadí

Když jsme začali budovat rozpoznávání řeči pro projekt HagiCode, s jistotou jsme si vybrali službu rozpoznávání řeči Doubao od ByteDance. Původní návrh byl přímočarý: nechat frontend připojit se přímo ke službě WebSocket společnosti Doubao. Jak těžké to může být? Stačí otevřít připojení a odeslat nějaká data, že?

Pak přišlo překvapení: API Doubao vyžaduje, aby byly ověřovací informace předávány přes HTTP hlavičky, včetně věcí jako `accessToken` a `secretKey`. To se okamžitě stalo nepříjemné, protože rozhraní WebSocket API v prohlížeči prostě nepodporuje nastavení vlastních záhlaví.

Co tedy dělat, když vám je prohlížeč nedovolí odeslat?

V té době jsme zvažovali dvě možnosti:

1. Vložte přihlašovací údaje do parametrů URL dotazu – jednoduché a neomalené
2. Přidejte proxy vrstvu na backend – více práce na první pohled

První možnost zpřístupňuje přihlašovací údaje přímo v kódu frontendu a místním úložišti. Je to opravdu bezpečné? Nebylo mi to příjemné. Navíc některá rozhraní API vyžadují ověření na základě hlavičky, takže tento přístup není ani životaschopný.

Nakonec jsme zvolili druhou možnost: implementovat WebSocket proxy na backend. Shodou okolností byl tento vzor poprvé ověřen v prostředí našeho hřiště a teprve poté, co jsme potvrdili jeho stabilitu, jsme jej přesunuli do výroby. Nikdo koneckonců nechce, aby výroba fungovala jako laboratorní experiment.

## Analýza technické výzvy

### Omezení WebSocket prohlížeče

Standardní rozhraní WebSocket API vypadá úžasně jednoduše:

```javascript
const ws = new WebSocket('wss://example.com/ws');
```

Ale právě v této jednoduchosti je problém – předává parametry pouze v URL a neumí nastavit hlavičky tak, jak to dokáže požadavek HTTP:

```javascript
// This is not supported in the WebSocket API
const ws = new WebSocket('wss://example.com/ws', {
  headers: {
    'Authorization': 'Bearer token'
  }
});
```

A to je hlavní problém. U služeb, jako je rozpoznávání řeči Doubao, které závisí na ověřování založeném na záhlaví, je toto omezení tvrdým blokátorem.

Jakmile toto omezení přijmete, architektura se musí změnit.

### Rozhodnutí o architektonickém návrhu

Při navrhování řešení jsme pečlivě porovnávali kompromisy.

**Rozhodnutí 1: Výběr vzoru proxy**

Porovnali jsme dva přístupy:

| Možnost | Pros | Nevýhody | Rozhodnutí |
|------|------|------|------|
| Nativní WebSocket | Lehké, jednoduché, přímé předávání | Správa připojení musí být řešena ručně | Vyvolený |
| SignálR | Automatické opětovné připojení, silné psaní | Příliš složité, další závislosti | Odmítnuto |

Nakonec jsme zvolili nativní WebSocket. Abych byl upřímný, byla to nejlehčí varianta a lépe se hodila pro jednoduché obousměrné přesměrování binárního proudu. Zatažení SignalR by pro tento případ použití vypadalo jako přehnané inženýrství a mohlo by to přidat další latenci.

**Rozhodnutí 2: Strategie správy připojení**

Přijali jsme model „jedno připojení, jedna relace“ – každé frontendové připojení WebSocket se mapuje na své vlastní nezávislé backendové připojení Doubao.

Výhody jsou přímočaré:
- Jednoduchá implementace a sladění s běžným vzorem používání
- Snazší ladění a odstraňování problémů
- Dobrá izolace zdrojů, která zabraňuje rušení mezi relacemi

Jednoduše řečeno, přímé řešení je někdy to nejlepší. Složitost automaticky nedělá návrh lepším.

**Rozhodnutí 3: Uložení ověřovacích dat**

Pověření jsou uložena v konfiguračních souborech backendu (`appsettings.yml` nebo proměnné prostředí) a načte se pomocí vkládání závislostí:

- Jednoduchý konfigurační model, který odpovídá stávajícím backendovým konvencím
- Citlivá data se nikdy nedostanou na frontend
- Podporuje nastavení pro více prostředí pro vývoj, testování a produkci

Na této úrovni separace záleží. Nikdo nechce, aby se přihlašovací údaje vznášely na místech, kde by neměly být.

### Návrh datového toku

Celkový datový tok vypadá takto:

```
Frontend (browser)
  │
  │ ws://backend/api/voice/ws
  │ WebSocket (binary)
  ▼
Backend (proxy)
  │
  │ wss://openspeech.bytedance.com/
  │ (with authentication headers)
  ▼
Doubao API
```

Samotný tok není složitý:
1. Frontend se připojuje k backendovému proxy přes WebSocket
2. Backend proxy přijímá zvuková data a připojuje se k Doubao API pomocí ověřených hlaviček
3. Doubao API vrací výsledky rozpoznávání a proxy je předává frontendu
4. Celý proces zůstává plně asynchronní s obousměrným streamováním

Jakmile jsou povinnosti jasně rozděleny, design se stává zcela přirozeným.

## Implementace základní komponenty

### 1. Konfigurace koncového bodu WebSocket

```csharp
app.Map("/ws", async context =>
{
    if (context.WebSockets.IsWebSocketRequest)
    {
        // Read configuration from query parameters
        var appId = context.Request.Query["appId"];
        var accessToken = context.Request.Query["accessToken"];

        // Validate required parameters
        if (string.IsNullOrEmpty(appId) || string.IsNullOrEmpty(accessToken))
        {
            context.Response.StatusCode = 400;
            return;
        }

        // Accept the WebSocket connection
        using var webSocket = await context.WebSockets.AcceptWebSocketAsync();

        // Message handling loop
        var buffer = new byte[4096];
        while (!webSocket.CloseStatus.HasValue)
        {
            var result = await webSocket.ReceiveAsync(buffer, CancellationToken.None);

            if (result.MessageType == WebSocketMessageType.Close)
            {
                await webSocket.CloseAsync(
                    result.CloseStatus.Value,
                    result.CloseStatusDescription,
                    CancellationToken.None);
                break;
            }

            // Process audio data
            await HandleAudioDataAsync(buffer, result.Count);
        }
    }
});
```

### 2. Řízení relace

```csharp
public class DoubaoSessionManager : IDoubaoSessionManager
{
    private readonly ConcurrentDictionary<string, DoubaoSession> _sessions = new();

    public DoubaoSession CreateSession(string connectionId)
    {
        var session = new DoubaoSession(connectionId);
        _sessions[connectionId] = session;
        return session;
    }

    public async Task SendAudioAsync(string connectionId, byte[] audioData)
    {
        if (_sessions.TryGetValue(connectionId, out var session))
        {
            await session.SendAudioAsync(audioData);
        }
    }

    public void RemoveSession(string connectionId)
    {
        if (_sessions.TryRemove(connectionId, out var session))
        {
            session.Dispose();
        }
    }
}
```

Použití `ConcurrentDictionary` pro správu relací znamená, že bezpečnost vláken je z velké části řešena za nás. Každé příchozí připojení získá svou vlastní relaci a při odpojení proběhne automatické čištění.

## Integrace frontendu v praxi

### Připojení WebSocket

```javascript
class DoubaoVoiceClient {
    constructor(config) {
        this.config = config;
        this.ws = null;
    }

    async connect() {
        const url = new URL(this.config.wsUrl);
        // Add query parameters
        Object.entries(this.config.params).forEach(([key, value]) => {
            url.searchParams.set(key, value);
        });

        this.ws = new WebSocket(url);

        return new Promise((resolve, reject) => {
            this.ws.onopen = () => {
                console.log('[DoubaoVoice] Connected');
                resolve();
            };

            this.ws.onmessage = (event) => {
                this._handleMessage(JSON.parse(event.data));
            };

            this.ws.onerror = reject;
        });
    }

    _handleMessage(message) {
        switch (message.type) {
            case 'status':
                this._handleStatus(message.payload);
                break;
            case 'result':
                this.onResult?.(message.payload);
                break;
            case 'error':
                console.error('[DoubaoVoice] Error:', message.payload);
                break;
        }
    }
}

// Usage example
const client = new DoubaoVoiceClient({
    wsUrl: 'ws://localhost:5000/ws',
    params: {
        appId: 'your-app-id',
        accessToken: 'your-access-token',
        sampleRate: 16000,
        bitsPerSample: 16,
        channels: 1
    }
});
```

### Snímání a přenos zvuku

Použití `AudioWorkletNode` pro zpracování zvuku poskytuje lepší výkon:

```javascript
// audio-worklet.js
class AudioProcessorWorklet extends AudioWorkletProcessor {
    process(inputs, outputs, parameters) {
        const input = inputs[0]?.[0];
        if (!input) return true;

        // Convert to 16-bit PCM
        const pcm = new Int16Array(input.length);
        for (let i = 0; i < input.length; i++) {
            pcm[i] = Math.max(-32768, Math.min(32767, input[i] * 32767));
        }

        this.port.postMessage({
            type: 'audioData',
            data: pcm.buffer
        }, [pcm.buffer]);

        return true;
    }
}

registerProcessor('audio-processor', AudioProcessorWorklet);

// Main thread code
async function startAudioRecording() {
    const stream = await navigator.mediaDevices.getUserMedia({
        audio: {
            echoCancellation: true,
            noiseSuppression: true,
            autoGainControl: true,
            sampleRate: 48000
        }
    });

    const audioContext = new AudioContext();
    const audioSource = audioContext.createMediaStreamSource(stream);

    await audioContext.audioWorklet.addModule('/audio-worklet.js');
    const audioWorkletNode = new AudioWorkletNode(audioContext, 'audio-processor');

    audioWorkletNode.port.onmessage = (event) => {
        if (event.data.type === 'audioData' && ws?.readyState === WebSocket.OPEN) {
            ws.send(event.data.data); // Send binary data directly
        }
    };

    audioSource.connect(audioWorkletNode);
}
```

`AudioWorklet` funguje mnohem lépe než `ScriptProcessorNode` a předchází problémům se zasekáváním zvuku, které často přináší starší cesty zpracování.

## Poznámky a osvědčené postupy

### 1. Monitorování připojení

- Pravidelně protokolujte stav relace, abyste mohli sledovat celý životní cyklus připojení
- Sledujte počet a trvání zvukových segmentů, abyste zjistili abnormální připojení
- Zaznamenejte stav připojení a chování při opětovném připojení pro službu Doubao

Toto jsou základní provozní postupy, které však ve výrobě představují skutečný rozdíl.

### 2. Ošetření chyb

- Zachyťte a protokolujte všechny výjimky WebSocket
- Použijte `IAsyncDisposable` zajistit vyčištění zdrojů
- Implementujte plynulé ukončení připojení a zpracování časového limitu

Stručně řečeno, upřednostněte robustnost.

### 3. Požadavky na formát zvuku

- Vzorkovací frekvence: 16000 Hz (doporučeno) nebo 8000 Hz
- Bitová hloubka: 16-bit
- Kanály: mono
- Kódování: PCM (raw)

Pokud je formát nesprávný, rozpoznání může selhat nebo se může výrazně snížit kvalita. Na těchto požadavcích záleží.

### 4. Bezpečnostní aspekty

- Uchovávejte citlivé přihlašovací údaje pouze v konfiguraci backendu
- Vynucujte limity připojení, abyste zabránili vyčerpání zdrojů
- Použijte HTTPS/WSS v produkci

Bezpečnost není nikdy podřadným problémem.

### 5. Optimalizace výkonu

- Abyste se vyhnuli blokování, používejte asynchronní operace
- Vylaďte velikosti vyrovnávací paměti podle potřeby (výchozí: 4096 bajtů)
- Zvažte sdružování připojení a strategie opětovného použití

Použijte tyto optimalizace tam, kde mají smysl pro vaši pracovní zátěž.

## Doporučení k nasazení

1. **Nasazení dockeru**: Zabalte službu proxy jako kontejner pro snadnější škálování a správu
2. **Vyrovnávání zátěže**: Použijte Nginx nebo Envoy jako reverzní proxy pro provoz WebSocket
3. **Kontrola stavu**: Implementujte kontroly založené na prezenčním signálu pro sledování dostupnosti služeb
4. **Agregace protokolů**: Odesílejte protokoly do centralizovaného systému protokolování, jako je ELK nebo Loki

Nasazení může být jednoduché nebo složité v závislosti na týmu a prostředí, takže se tomu přizpůsobte.

## Shrnutí

Vzor proxy WebSocket řeší hlavní problém, že rozhraní WebSocket API v prohlížeči nepodporuje vlastní záhlaví. V projektu HagiCode se tento vzor ukázal jako proveditelný a stabilní, když se přesunul z ověřování na hřišti do produkčního nasazení.

Klíčové věci:
- Backend proxy může bezpečně předávat ověřovací informace
- Nativní WebSocket je lehký a efektivní pro jednoduché scénáře
- Model „jedno připojení, jedna relace“ zjednodušuje implementaci i ladění
- Protokol frontend-backend by měl oddělovat řídicí signály od zvukových dat

Pokud vytváříte funkci, která také závisí na autentizaci WebSocket, doufám, že vám tento vzor poskytne užitečný výchozí bod.
