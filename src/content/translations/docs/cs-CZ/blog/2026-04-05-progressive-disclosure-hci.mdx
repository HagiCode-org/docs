## Postupné odhalování: Zlepšení interakce člověk-počítač v produktech umělé inteligence pomocí filozofie „méně je více“

> V produktovém designu AI kvalita vstupu uživatele často určuje kvalitu výstupu. Tento článek sdílí přístup k interakci „progresivního zveřejňování“, který jsme praktikovali v projektu HagiCode. Prostřednictvím podrobného vedení, inteligentního dokončování a okamžité zpětné vazby mění stručné a vágní vstupy uživatelů do strukturovaných technických návrhů, což výrazně zlepšuje efektivitu interakce mezi člověkem a počítačem.

## Pozadí

Každý, kdo vytváří produkty AI, pravděpodobně viděl tuto situaci: uživatel otevře vaši aplikaci a nadšeně zadá jednořádkový požadavek, pouze aby AI vrátila něco zcela mimo cíl. Neznamená to, že by umělá inteligence nebyla dostatečně chytrá. Uživatel jednoduše neposkytl dostatek informací. Čtení myšlenek je těžké pro každého.

Tento problém se stal obzvláště zřejmým, když jsme budovali HagiCode. HagiCode je asistent kódování řízený umělou inteligencí, kde uživatelé popisují požadavky přirozeným jazykem pro vytváření technických návrhů a relací. Při skutečném použití jsme zjistili, že uživatelský vstup má často tyto problémy:

- **Nekonzistentní kvalita vstupu**: někteří uživatelé zadají pouze několik slov, například „optimalizovat přihlášení“ nebo „opravit chybu“, bez potřebného kontextu
- **Nekonzistentní technická terminologie**: různí uživatelé používají různé výrazy pro stejnou věc; někteří říkají „frontend“, zatímco jiní říkají „FE“
- **Chybí strukturované informace**: chybí pozadí projektu, rozsah úložiště nebo rozsah dopadu, i když se jedná o kritické detaily
- **Opakované problémy**: stejné typy požadavků se objevují znovu a znovu a pokaždé, když je třeba je vysvětlit od začátku

Přímý výsledek je předvídatelný: AI má obtížnější pochopit požadavek, kvalita návrhu se stává nestabilní a uživatelská zkušenost trpí. Uživatelé si myslí: „Tato umělá inteligence není moc dobrá“, zatímco my se cítíme nespravedlivě obviňováni. Když mi dáš jen jednu větu, jak mám uhodnout, co vlastně chceš?

Ve skutečnosti je to pochopitelné. I lidé potřebují čas, aby si porozuměli, a stroje nejsou výjimkou.

Abychom vyřešili tyto bolestivé body, učinili jsme odvážné rozhodnutí: zavést princip návrhu „progresivního zveřejňování“, abychom zlepšili interakci člověka s počítačem. Změny, které to přineslo, byly pravděpodobně větší, než byste si představovali. Abych byl upřímný, v té době jsme nečekali, že bude tak efektivní.

## Co je progresivní zveřejňování

„Progressive disclosure“ je konstrukční princip z oblasti HCI (interakce člověka s počítačem). Jeho základní myšlenka je jednoduchá: **nezobrazovat uživatelům všechny informace a možnosti najednou. Místo toho odhalujte krok za krokem pouze to, co je nezbytné, na základě akcí a potřeb uživatele**.

Tento princip je vhodný zejména pro produkty AI, protože interakce AI je přirozeně progresivní. Uživatel říká málo, AI trochu rozumí, pak uživatel přidává více a AI rozumí více. Je to velmi podobné tomu, jak spolu lidé komunikují: porozumění se obvykle vyvíjí postupně.

Ve scénáři HagiCode jsme aplikovali progresivní zveřejňování ve čtyřech oblastech:

### 1. Mechanismus optimalizace popisu: AI vám pomůže říkat věci jasněji

Když uživatel zadá krátký popis, neposíláme jej přímo do AI k interpretaci. Místo toho nejprve spustíme tok „optimalizace popisu“. Jádrem tohoto toku je „strukturovaný výstup“: převod volného textu uživatele do standardního formátu. Je to jako navlékání volných perel do náhrdelníku, takže je vše snáze pochopitelné.

Optimalizovaný popis musí obsahovat následující standardní části:

- **Pozadí**: pozadí a kontext problému
- **Analýza**: technická analýza a zdůvodnění
- **Řešení**: řešení a kroky implementace
- **Cvičení**: konkrétní příklady kódu a poznámky

Současně automaticky generujeme tabulku Markdown zobrazující informace, jako je cílové úložiště, cesty a oprávnění k úpravám, což usnadňuje následné operace AI. Přehledný adresář vždy usnadní hledání věcí.

Zde je skutečná implementace:

```csharp
// Core method in ProposalDescriptionMemoryService.cs
public async Task<string> OptimizeDescriptionAsync(
    string title,
    string description,
    string locale = "zh-CN",
    DescriptionOptimizationMemoryContext? memoryContext = null,
    CancellationToken cancellationToken = default)
{
    // Build query parameters
    var queryContext = BuildQueryContext(title, description);

    // Retrieve historical context
    var memoryContext = await RetrieveHistoricalContextAsync(queryContext, cancellationToken);

    // Generate a structured prompt
    var prompt = await BuildOptimizationPromptAsync(
        title,
        description,
        memoryContext,
        cancellationToken);

    // Call AI for optimization
    return await _aiService.CompleteAsync(prompt, cancellationToken);
}
```

Klíčem k tomuto toku je „injekce paměti“. Do výzvy vkládáme historický kontext, jako jsou projektové konvence, podobné případy a negativní vzorce, což umožňuje AI odkazovat na minulé zkušenosti během optimalizace. Zkušenosti by neměly přijít nazmar.

**Poznámky:**
- Ujistěte se, že aktuální vstup má přednost před historickou pamětí, takže explicitně zadané informace o uživateli nebudou přepsány
- S odkazy HagIndex je třeba zacházet jako s faktickými zdroji a nesmí být měněny historickými případy
- Návrhy korekce s nízkou spolehlivostí by neměly být aplikovány jako silná omezení

### 2. Možnost hlasového vstupu: mluvení je přirozenější než psaní

Kromě zadávání textu podporujeme také hlasové zadávání. To je užitečné zejména pro popis složitých požadavků. Napsání technického požadavku může trvat minuty, zatímco jeho vyslovení nahlas může trvat jen několik desítek sekund.

Klíčovým designovým zaměření pro hlasový vstup je „správa stavu“. Uživatelé musí jasně vědět, v jakém stavu se systém aktuálně nachází. Definovali jsme následující stavy:

- **Idle**: systém je připraven a nahrávání může začít
- **Čekání upstream**: systém se připojuje k backendové službě
- **Nahrávání**: Nahrává se hlas uživatele
- **Zpracování**: řeč se převádí na text
- **Chyba**: došlo k chybě a vyžaduje pozornost uživatele

Model stavu frontendu vypadá zhruba takto:

```typescript
interface VoiceInputState {
  status: 'idle' | 'waiting-upstream' | 'recording' | 'processing' | 'error';
  duration: number;
  error?: string;
  deletedSet: Set<string>; // Fingerprint set of deleted results
}

// State transition when recording starts
const handleVoiceInputStart = async () => {
  // Enter waiting state first and show a loading animation
  setState({ status: 'waiting-upstream' });

  // Wait for backend readiness confirmation
  const isReady = await waitForBackendReady();
  if (!isReady) {
    setState({ status: 'error', error: 'Backend service is not ready' });
    return;
  }

  // Start recording
  setState({ status: 'recording', startTime: Date.now() });
};

// Handle recognition results
const handleRecognitionResult = (result: RecognitionResult) => {
  const fingerprint = normalizeFingerprint(result.text);

  // Check whether it has already been deleted
  if (state.deletedSet.has(fingerprint)) {
    return; // Skip deleted content
  }

  // Merge the result into the text box
  appendResult(result);
};
```

Je zde důležitý detail: ke správě synchronizace mazání používáme „sada otisků prstů“. Když rozpoznávání řeči vrátí více výsledků, uživatelé mohou některé z nich odstranit. Ukládáme otisky prstů smazaného obsahu, takže pokud se stejný obsah objeví znovu později, je automaticky přeskočen. Je to v podstatě způsob, jak si zapamatovat, co uživatel již odmítl.

### 3. Systém rychlého řízení: externalizujte „mozek“ AI

HagiCode má flexibilní systém správy výzev, ve kterém jsou všechny výzvy uloženy jako soubory:

```
prompts/
├── metadata/
│   ├── optimize-description.zh-CN.json
│   └── optimize-description.en-US.json
└── templates/
    ├── optimize-description.zh-CN.hbs
    └── optimize-description.en-US.hbs
```

Každá výzva se skládá ze dvou částí:
- **Soubor metadat (`.json`)**: definuje informace, jako je scénář výzvy, verze a parametry
- **Soubor šablony (`.hbs`)**: skutečný obsah výzvy napsaný pomocí syntaxe Handlebars

Formát souboru metadat vypadá takto:

```json
{
  "scenario": "optimize-description",
  "locale": "zh-CN",
  "version": "1.0.0",
  "syntax": "handlebars",
  "syntaxVersion": "1.0",
  "parameters": [
    {
      "name": "title",
      "type": "string",
      "required": true,
      "description": "Proposal title"
    },
    {
      "name": "description",
      "type": "string",
      "required": true,
      "description": "Original description"
    }
  ],
  "author": "HagiCode Team",
  "description": "Optimize the user's technical proposal description",
  "lastModified": "2026-04-05",
  "tags": ["optimization", "nlp"]
}
```

Soubor šablony používá syntaxi Handlebars a podporuje vkládání parametrů:

```handlebars
You are a technical proposal expert.

<task>
Generate a structured technical proposal description based on the following information.
</task>

<input>
<title>{{title}}</title>
<description>{{description}}</description>
{{#if memoryContext}}
<memory_context>
{{memoryContext}}
</memory_context>
{{/if}}
</input>

<output_format>
## Background
[Describe the problem background and context, including project information, repository scope, and so on]

## Analysis
[Provide the technical analysis and reasoning process, and explain why this change is needed]

## Solution
[Provide the solution and implementation steps, listing the key code locations]

## Practice
[Provide concrete code examples and notes]
</output_format>
```

Výhody tohoto designu jsou jasné:
- výzvy mohou být řízeny verzí stejně jako kód
- je podporováno více jazyků a lze je automaticky přepínat podle preferencí uživatele
- parametrizovaný design umožňuje dynamické vkládání kontextu
- úplnost lze ověřit při spuštění, čímž se zabrání chybám za běhu

Pokud znalosti zůstanou jen v něčí hlavě, je snadné je ztratit. Nahrávání od začátku strukturovaně je mnohem bezpečnější.

### 4. Progresivní průvodce: rozdělte složité úkoly do malých kroků

Pro složité úkoly, jako je první instalace a konfigurace, používáme vícekrokový návrh průvodce. Každý krok vyžaduje pouze nezbytné informace a poskytuje jasné ukazatele pokroku. Velké úkoly se stávají mnohem lépe zvládnutelné, když je zvládáte jeden krok po druhém.

Model stavu průvodce:

```typescript
interface WizardState {
  currentStep: number;           // 0-3, corresponding to 4 steps
  steps: WizardStep[];
  canGoNext: boolean;
  canGoBack: boolean;
  isLoading: boolean;
  error: string | null;
}

interface WizardStep {
  id: number;
  title: string;
  description: string;
  completed: boolean;
}

// Step navigation logic
const goToNextStep = () => {
  if (wizardState.currentStep < wizardState.steps.length - 1) {
    // Validate input for the current step
    if (validateCurrentStep()) {
      wizardState.currentStep++;
      wizardState.steps[wizardState.currentStep - 1].completed = true;
    }
  }
};

const goToPreviousStep = () => {
  if (wizardState.currentStep > 0) {
    wizardState.currentStep--;
  }
};
```

Každý krok má svou vlastní logiku ověřování a dokončené kroky dostávají jasné vizuální značky. Zrušením se otevře potvrzovací dialog, který zabrání tomu, aby uživatelé ztratili pokrok v důsledku náhodných akcí.

## Závěr

Když se podíváme zpět na postup HagiCode v oblasti zveřejňování, můžeme shrnout několik základních principů:

1. **Pokyny krok za krokem**: rozdělte složité úkoly na menší kroky a v každé fázi si vyžádejte pouze nezbytné informace
2. **Inteligentní vyplňování**: použijte historický kontext a znalosti projektu k automatickému vyplňování informací
3. **Okamžitá zpětná vazba**: poskytněte každé akci jasnou vizuální zpětnou vazbu a stavové rady
4. **Mechanismy odolnosti proti chybám**: umožňují uživatelům vrátit zpět a resetovat, takže chyby nevedou k nevratné ztrátě
5. **Vstupní rozmanitost**: podpora více metod zadávání, jako je text a hlas

V HagiCode byl praktický výsledek tohoto přístupu jasný: průměrná délka uživatelského vstupu se zvýšila z méně než 20 znaků na strukturované popisy 200–300 znaků, kvalita návrhů generovaných umělou inteligencí se výrazně zlepšila a spolu s tím vzrostla spokojenost uživatelů.

To není překvapivé. Čím více informací uživatelé poskytnou, tím přesněji jim AI porozumí a tím lepší výsledky může vrátit. V tomto smyslu se příliš neliší od komunikace mezi lidmi.

Pokud také vytváříte produkty související s umělou inteligencí, doufám, že tyto zkušenosti přinesou užitečnou inspiraci. Pamatujte: uživatelé nemusí nutně odmítat poskytnout informace. Produkt ještě častěji nepoložil správné otázky správným způsobem. Jádrem progresivního odhalování je nalezení nejlepšího načasování a formy pro tyto otázky.
