## Upravit DESIGN.md Přímo ve webovém rozhraní: Od nápadu k realizaci

> V systému řízení projektů MonoSpecs `DESIGN.md` nese architektonický návrh a technická rozhodnutí projektu. Ale tradiční pracovní postup úprav nutí uživatele skočit do externího editoru. Ten roztříštěný zážitek je jako být přerušen uprostřed čtení básně: inspirace je pryč a nálada také. Tento článek sdílí řešení, které jsme uvedli do praxe v projektu HagiCode: úpravy `DESIGN.md` přímo ve webovém rozhraní s podporou importu šablon z online designového webu. Koneckonců, kdo by si neužil pocit, že vše dokončí jedním pohybem?

## Pozadí

Jako hlavní nosič projektové dokumentace, `DESIGN.md` obsahuje klíčové informace, jako je návrh architektury, technická rozhodnutí a pokyny k implementaci. Tradiční přístup k úpravám však vyžaduje, aby uživatelé přešli na externí editor, jako je VS Code, ručně našli fyzickou cestu a poté soubor upravili. Není to nijak zvlášť složité, ale po několikanásobném opakování je to únavné.

Problémy se projevují především následujícími způsoby:

- **Fragmentovaný pracovní postup**: uživatelé musí neustále přepínat mezi webovým rozhraním pro správu a místním editorem, čímž narušují kontinuitu jejich pracovního postupu, podobně jako když mají hudbu vystřiženou uprostřed skladby.
- **Obtížné opětovné použití**: designový web již publikuje bohatou knihovnu návrhových šablon, ale nelze je integrovat přímo do pracovního postupu úprav projektu. Dobré věci existují, ale stále je nemůžete použít tam, kde je potřebujete.
- **Chybí smyčka zkušeností**: neexistuje žádná uzavřená smyčka pro „preview-select-import“, takže uživatelé musí kopírovat a vkládat ručně, což zvyšuje riziko chyb.
- **Tření ve spolupráci**: synchronizace návrhových dokumentů a implementace kódu se stává velmi náročným procesem, který snižuje efektivitu týmu.

K vyřešení těchto bolestivých bodů jsme se rozhodli přidat přímou podporu úprav `DESIGN.md` ve webovém rozhraní a umožňují import šablony jedním kliknutím z online designového webu. Nebylo to nějaké převratné rozhodnutí. Jednoduše jsme chtěli usnadnit vývoj.

## Technické řešení

### Celková architektura

Toto řešení využívá architekturu oddělenou mezi frontendem a backendem s proxy se stejným původem, která se skládá hlavně z následujících vrstev. V praxi lze design shrnout jako „každá část dělá svou vlastní práci“:

**1. Vrstva frontend editoru**
```typescript
// Core component: DesignMdManagementDrawer
// Location: repos/web/src/components/project/DesignMdManagementDrawer.tsx
// Responsibility: handle editing, saving, version conflict detection, and import flow
```

**2. Vrstva backendové služby**
```csharp
// Core service: ProjectAppService.DesignMd
// Location: repos/hagicode-core/src/PCode.Application/ProjectAppService.DesignMd.cs
// Responsibility: path resolution, file read/write, and version management
```

**3. Vrstva proxy se stejným původem**
```csharp
// Proxy service: ProjectAppService.DesignMdSiteIndex
// Location: repos/hagicode-core/src/PCode.Application/ProjectAppService.DesignMdSiteIndex.cs
// Responsibility: proxy design site resources, preview image caching, and security validation
```

### Klíčová technická rozhodnutí

#### Rozhodnutí 1: Globální vzor zásuvky

Místo lokálních vyskakovacích vrstev používáme jeden globální šuplík se stavem spravovaným prostřednictvím `layoutSlice`, která uživatelům poskytuje konzistentní zkušenost napříč zhlédnutími (`classic` a `kanban`). Bez ohledu na to, ze kterého pohledu uživatel otevře editor, získá stejný model interakce. Díky konzistentnímu zážitku se lidé cítí pohodlněji, místo aby byli dezorientovaní, když změní názory.

#### Rozhodnutí 2: Project-Scoped API

Namontovali jsme `DESIGN.md`- související koncové body pod `ProjectController`, opětovně použít stávající hranici oprávnění k projektu a vyhnout se složitosti přidávání samostatného ovladače. Díky tomu je manipulace s oprávněními jasnější a také je v souladu s organizací zdrojů RESTful. Někdy je opětovné použití smysluplnější než vytváření něčeho nového od začátku.

#### Rozhodnutí 3: Zjištění konfliktu verzí

Odvozujeme neprůhlednou verzi ze souborového systému `LastWriteTimeUtc`, což nám poskytuje odlehčenou optimistickou kontrolu souběžnosti. Když více uživatelů upravuje stejný soubor najednou, systém může detekovat konflikty a vyzvat uživatele k aktualizaci. Tento návrh neblokuje úpravy a přitom stále chrání konzistenci dat.

#### Rozhodnutí 4: Vzor proxy se stejným původem

Používáme `IHttpClientFactory` k proxy externím zdrojům design-site, čímž se vyhnete problémům s křížovým původem a rizikům SSRF. To udržuje systém v bezpečí a zároveň zjednodušuje frontendové hovory. S bezpečností můžete být jen stěží příliš opatrní.

## Implementace jádra

### 1. Přímo upravte DESIGN.md

#### Implementace backendu

Backend je zodpovědný hlavně za rozlišení cest, čtení/zápis souborů a správu verzí. Tyto úkoly jsou základní, ale nepostradatelné, jako založení domu:

```csharp
// Path resolution and security validation
private Task<string> ResolveDesignDocumentDirectoryAsync(string projectPath, string? repositoryPath)
{
    if (string.IsNullOrWhiteSpace(repositoryPath))
    {
        return Task.FromResult(Path.GetFullPath(projectPath));
    }
    return ValidateSubPathAsync(projectPath, repositoryPath);
}

// Version generation (based on file system timestamp and size)
private static string BuildDesignDocumentVersion(string path)
{
    var fileInfo = new FileInfo(path);
    fileInfo.Refresh();
    return string.Create(
        CultureInfo.InvariantCulture,
        $"{fileInfo.LastWriteTimeUtc.Ticks:x}-{fileInfo.Length:x}");
}
```

Návrh verze je zajímavý svou jednoduchostí: používáme čas a velikost poslední změny souboru ke generování jedinečného identifikátoru verze. Je lehký a spolehlivý a nevyžaduje údržbu žádné další databáze verzí. Jednoduchá řešení jsou často nejúčinnější.

#### Implementace frontendu

Na frontendu implementujeme detekci nečistého stavu a logiku ukládání. Tento design pomáhá uživatelům pochopit, zda byly jejich změny uloženy, a snižuje obavy z „co když to ztratím?“:

```typescript
// Dirty-state detection and save logic
const [draft, setDraft] = useState('');
const [savedDraft, setSavedDraft] = useState('');
const isDirty = draft !== savedDraft;

const handleSave = useCallback(async () => {
    const result = await saveProjectDesignMdDocument({
        ...activeTarget,
        content: draft,
        expectedVersion: document.version, // optimistic concurrency control
    });
    setSavedDraft(draft); // update saved state
}, [activeTarget, document, draft]);
```

V této implementaci udržujeme dva stavy: `draft` je obsah, který je aktuálně upravován, zatímco `savedDraft` je uložený obsah. Jejich porovnáním zjistíme, zda existují neuložené změny. Design je jednoduchý, ale dává lidem klid. Nikdo nechce, aby zmizela věc, na které tvrdě pracoval.

### 2. Importujte soubory návrhu z online zdroje

#### Struktura adresáře

```
repos/index/
└── src/data/public/design.json    # Design template index

repos/awesome-design-md-site/
├── vendor/awesome-design-md/       # Upstream design templates
│   └── design-md/
│       ├── clickhouse/
│       │   └── DESIGN.md
│       ├── linear/
│       │   └── DESIGN.md
│       └── ...
└── src/lib/content/
    └── awesomeDesignCatalog.ts     # Content pipeline
```

#### Formát dat indexu

Indexový soubor na webu návrhu definuje všechny dostupné šablony. Pomocí tohoto indexu si uživatelé mohou vybrat šablonu, kterou chtějí, stejně snadno jako objednání z nabídky:

```json
{
  "entries": [
    {
      "slug": "linear.app",
      "title": "Linear Inspired Design System",
      "summary": "AI Product / Dark Feel",
      "detailUrl": "/designs/linear.app/",
      "designDownloadUrl": "/designs/linear.app/DESIGN.md",
      "previewLightImageUrl": "...",
      "previewDarkImageUrl": "..."
    }
  ]
}
```

Každý záznam obsahuje základní informace o šabloně a odkazy ke stažení. Backend čte seznam dostupných šablon z tohoto indexu a předkládá je uživateli, aby si z nich mohl vybrat. Díky tomu je výběr intuitivní, místo aby lidi nutil, aby se ve tmě orientovali.

#### Implementace proxy se stejným původem

Aby byly věci zabezpečeny, backend provádí přísné ověřování přístupu k webu návrhu. Nemůžete být příliš opatrní ohledně zabezpečení:

```csharp
// Safe slug validation
private static readonly Regex SafeDesignSiteSlugRegex =
    new("^[A-Za-z0-9](?:[A-Za-z0-9._-]{0,127})$", RegexOptions.Compiled);

private static string NormalizeDesignSiteSlug(string slug)
{
    var normalizedSlug = slug?.Trim() ?? string.Empty;
    if (!IsSafeDesignSiteSlug(normalizedSlug))
    {
        throw new BusinessException(
            ProjectDesignSiteIndexErrorCodes.InvalidSlug,
            "Design site slug must be a single safe path segment.");
    }
    return normalizedSlug;
}

// Preview image caching (OS temp directory)
private static string ComputePreviewCacheKey(string slug, string theme, string previewUrl)
{
    var raw = $"{slug}|{theme}|{previewUrl}";
    var bytes = SHA256.HashData(Encoding.UTF8.GetBytes(raw));
    return Convert.ToHexString(bytes).ToLowerInvariant();
}
```

Děláme zde dvě věci: za prvé ověřujeme formát slug striktně pomocí regulárního výrazu, abychom zabránili útokům s procházením cesty; za druhé, ukládáme do mezipaměti náhledové obrázky, abychom snížili tlak na externí web. To první je ochrana, to druhé je optimalizace a na obojím záleží.

### 3. Úplný tok importu

```typescript
// 1. Open the import drawer
const handleRequestImportDrawer = useCallback(() => {
    setIsImportDrawerOpen(true);
}, []);

// 2. Select and import
const handleImportRequest = useCallback((entry) => {
    if (isDirty) {
        setPendingImportEntry(entry);
        setConfirmMode('import'); // overwrite confirmation
        return;
    }
    void executeImport(entry);
}, [isDirty]);

// 3. Execute import
const executeImport = useCallback(async (entry) => {
    const result = await getProjectDesignMdSiteImportDocument(
        activeTarget.projectId,
        entry.slug
    );
    setDraft(result.content); // replace editor text only, do not save automatically
    setIsImportDrawerOpen(false);
}, [activeTarget?.projectId]);
```

Postup importu se řídí principem „potvrzení uživatelem“: po importu se aktualizuje pouze obsah editoru a nic se automaticky neukládá. Uživatelé mohou importovaný obsah zkontrolovat a ručně uložit pouze po potvrzení, že vypadá správně. Konečné rozhodnutí by mělo zůstat v rukou uživatele.

## Praktické příklady

### Scénář 1: Vytvoření DESIGN.md v kořenovém adresáři projektu

Kdy `DESIGN.md` neexistuje, backend vrátí stav virtuálního dokumentu. To umožňuje frontendu vyhnout se zvláštnímu zacházení pro případ „soubor neexistuje“ a jednotné rozhraní API zjednodušuje logiku kódu:

```csharp
return new ProjectDesignDocumentDto
{
    Path = targetPath,
    Exists = false,  // virtual document state
    Content = string.Empty,
    Version = null
};

// Automatically create the file on first save
public async Task<SaveProjectDesignDocumentResultDto> SaveDesignDocumentAsync(...)
{
    Directory.CreateDirectory(targetDirectory);
    await File.WriteAllTextAsync(targetPath, input.Content);
    return new SaveProjectDesignDocumentResultDto { Created = !exists };
}
```

Výhodou tohoto návrhu je, že frontend nepotřebuje speciální logiku pro chybějící soubory. Skrytím této složitosti v backendu se frontend může snadněji zaměřit na uživatelskou zkušenost.

### Scénář 2: Importujte šablonu z webu návrhu

Poté, co uživatel v zásuvce importu vybere šablonu návrhu "Lineární", systém načte šablonu `DESIGN.md` obsah přes backend proxy. Celý proces je pro uživatele transparentní: vybere si pouze šablonu a systém automaticky zpracovává síťové požadavky a transformaci dat.

```typescript
// 1. The system fetches DESIGN.md content through the backend proxy
GET /api/project/{id}/design-md/site-index/linear.app

// 2. The backend validates the slug and fetches content from upstream
var entry = FindDesignSiteEntry(catalog, "linear.app");
using var upstreamResponse = await httpClient.SendAsync(request);
var content = await upstreamResponse.Content.ReadAsStringAsync();

// 3. The frontend replaces the editor text
setDraft(result.content);
// The user reviews it and then saves it manually to disk
```

Celý tok zůstává pro uživatele transparentní. Stačí si vybrat šablonu a systém se postará o vytváření sítí a transformaci v zákulisí. To je zážitek, který chceme: jednoduchý, ale silný.

### Scénář 3: Řešení konfliktů verzí

Když totéž upravuje více uživatelů `DESIGN.md` současně systém detekuje konflikty verzí. Tento optimistický mechanismus kontroly souběžnosti zachovává konzistenci dat, aniž by blokoval úpravy uživatele:

```csharp
if (!string.Equals(currentVersion, expectedVersion, StringComparison.Ordinal))
{
    throw new BusinessException(
        ProjectDesignDocumentErrorCodes.VersionConflict,
        $"DESIGN.md at '{targetPath}' changed on disk.");
}
```

Frontend zachytí tuto chybu a vyzve uživatele:

```typescript
// Frontend prompts the user to refresh and retry
<Alert>
    <AlertTitle>Version conflict</AlertTitle>
    <AlertDescription>
        The file was modified by another process. Please refresh to get the latest version and try again.
    </AlertDescription>
</Alert>
```

Tento optimistický mechanismus kontroly souběžnosti udržuje data konzistentní bez blokování uživatelů při práci. Konfliktům se nelze vyhnout, ale uživatelé by alespoň měli vědět, co se stalo, místo aby tiše přišli o své změny.

## Poznámky a doporučené postupy

### 1. Zabezpečení cesty

Vždy potvrdit `repositoryPath` aby se zabránilo útokům přes cestu. Pokud jde o bezpečnost, nikdy toho nemůžete udělat příliš mnoho:

```csharp
// Always validate repositoryPath to prevent path traversal attacks
return ValidateSubPathAsync(projectPath, repositoryPath);
// Reject dangerous inputs such as "../" and absolute paths
```

### 2. Strategie mezipaměti

Ukládat do mezipaměti náhledy obrázků po dobu 24 hodin, maximálně 160 souborů. Mírné ukládání do mezipaměti zlepšuje výkon, ale na vyvážení stále záleží:

```csharp
// Cache preview images for 24 hours, with a maximum of 160 files
private static readonly TimeSpan PreviewCacheTtl = TimeSpan.FromHours(24);
private const int PreviewCacheMaxFiles = 160;
// Periodically clean up expired cache
```

### 3. Zpracování chyb

Pěkně degradujte, když je upstream místo nedostupné. Tento návrh zajišťuje, že i když selže externí závislost, základní funkce úprav stále funguje normálně:

```typescript
// Gracefully degrade when the upstream site is unavailable
try {
    const catalog = await getProjectDesignMdSiteImportCatalog(projectId);
} catch (error) {
    toast.error(t('project.designMd.siteImport.feedback.catalogLoadFailed'));
    // The main editing drawer remains available
}
```

Tato půvabná degradace zajišťuje, že i v případě nedostupnosti externích závislostí bude funkce úpravy jádra nadále fungovat. Systém by měl být odolný, místo aby se zhroutil v okamžiku, kdy se něco pokazí.

### 4. Optimalizace uživatelské zkušenosti

Před importem potvrďte přepsání a po importu automaticky neukládejte. Uživatelé by měli mít kontrolu nad svými vlastními akcemi:

```typescript
// Confirm overwrite before import
if (isDirty) {
    setConfirmMode('import');
    return;
}

// Do not save automatically after import; let the user confirm
setDraft(result.content); // update draft only
// The content is written to disk only after the user reviews it and clicks Save
```

### 5. Úvahy o výkonu

Použijte továrnu HTTP klienta, abyste se vyhnuli vytváření příliš mnoha připojení. Správa zdrojů se může zdát malá, ale její dobré provedení může znamenat velký rozdíl:

```csharp
// Use an HTTP client factory to avoid creating too many connections
private const string DesignSiteProxyClientName = "ProjectDesignSiteProxy";
private static readonly TimeSpan DesignSiteProxyTimeout = TimeSpan.FromSeconds(8);
```

## Navrhovaná rozšíření

1. **Vylepšení Markdown**: v současné době používáme základní `Textarea`, ale mohli jsme upgradovat na CodeMirror pro zvýraznění syntaxe a klávesové zkratky. Když se editor cítí lépe, psaní dokumentace je také lepší.
2. **Režim náhledu**: přidejte náhled Markdown v reálném čase, abyste zlepšili možnosti úprav. To, co-vidíte-je-co-dostanete, lidem vždy dodá větší sebevědomí.
3. **Rozdílné sloučení**: implementujte inteligentní slučovací algoritmus namísto jednoduchého nahrazování plného textu. Konflikty jsou nevyhnutelné, ale proces jejich řešení nemusí být bolestivý.
4. **Místní ukládání do mezipaměti**: mezipaměť `design.json` v databázi, aby se snížila závislost na externím webu. Čím méně závislostí má systém, tím bývá stabilnější.

## Shrnutí

V projektu HagiCode jsme implementovali kompletní online řešení editace a importu pro `DESIGN.md` prostřednictvím spolupráce frontend-backend. Hlavní hodnota tohoto řešení spočívá v následujících bodech:

- **Vyšší účinnost**: není třeba měnit nástroje; editace a import návrhových dokumentů může probíhat v jednom jednotném webovém rozhraní.
- **Nižší překážka vstupu**: Import šablony návrhu jedním kliknutím pomáhá novým projektům rychle začít.
- **Bezpečné a spolehlivé**: ověření cesty, detekce konfliktu verzí a elegantní mechanismy degradace udržují systém stabilní.
- **Lepší uživatelský dojem**: Globální šuplík, detekce nečistého stavu a potvrzovací dialogy zdokonalují celkový zážitek z interakce.

Toto řešení již běží v projektu HagiCode a vyřešilo problémy týmu týkající se správy návrhových dokumentů.
