## Vytváření integrace problémů GitHubu od nuly: Praxe přímého připojení frontendu HagiCode

> Tento článek dokumentuje celý proces integrace problémů GitHub do platformy HagiCode. Prozkoumáme, jak používat architekturu „přímé frontendové připojení + minimální backend“ k dosažení bezpečné autentizace OAuth a efektivní synchronizace problémů při zachování lehkého backendu.

## Pozadí: Proč integrovat GitHub?

Jako vývojová platforma podporovaná umělou inteligencí spočívá hlavní hodnota HagiCode v propojení nápadů s implementací. Ale při skutečném používání jsme zjistili, že poté, co uživatelé dokončí návrh v HagiCode, často potřebují ručně zkopírovat obsah do GitHub Issues pro sledování projektu.

To vytváří několik zjevných bolestivých bodů:

1.  **Fragmentovaný pracovní postup**: Uživatelé musí mezi dvěma systémy přepínat tam a zpět. Práce není plynulá a klíčové informace se mohou snadno ztratit během kopírování a vkládání.
2.  **Nepohodlná spolupráce**: Ostatní členové týmu jsou zvyklí kontrolovat úkoly na GitHubu a nemohou přímo vidět průběh návrhu v HagiCode.
3.  **Opakovaná ruční práce**: Při každé aktualizaci návrhu musí někdo ručně aktualizovat odpovídající problém na GitHubu, což zvyšuje zbytečné náklady na údržbu.

Abychom tento problém vyřešili, rozhodli jsme se zavést funkci **GitHub Issues Integration**, která spojuje relace HagiCode s repozitáři GitHub a umožňuje „synchronizaci jedním kliknutím“.

## Technická volba: Frontend Direct Connection vs Backend Proxy

Při navrhování integračního přístupu jsme měli před sebou dvě možnosti: tradiční „backend proxy model“ a agresivnější „frontendový model přímého připojení“.

### Srovnání řešení

V tradičním **backend proxy modelu** musí každý požadavek z frontendu nejprve projít naším backendem, který pak volá GitHub API. To centralizuje logiku, ale také to značně zatěžuje backend:

1.  **Nadupaný backend**: Potřebovali bychom napsat vyhrazený GitHub API klient wrapper a také zvládnout složitý stavový stroj OAuth.
2.  **Riziko tokenu**: Token uživatele GitHub by musel být uložen v backendové databázi. I při šifrování to stále zvyšuje úroveň zabezpečení.
3.  **Náklady na vývoj**: Potřebovali bychom migraci databáze pro ukládání tokenů a další synchronizační službu, kterou bychom měli udržovat.

**model přímého připojení frontend** je mnohem lehčí. V tomto přístupu používáme backend pouze pro nejcitlivější krok „tajné výměny“ (zpětné volání OAuth). Po získání tokenu jej uložíme přímo do prohlížeče `localStorage`. Pozdější operace, jako je vytváření problémů a aktualizace komentářů, jsou odesílány přímo z frontendu na GitHub přes HTTP.

| Srovnávací rozměr | Backendový proxy model | Model přímého připojení frontendu |
| :--- | :--- | :--- |
| **Složitost backendu** | Vyžaduje plnou službu OAuth a klienta GitHub API | Vyžaduje pouze koncový bod zpětného volání OAuth |
| **Správa tokenů** | Musí být zašifrované a uložené v databázi s rizikem úniku | Uloženo v prohlížeči a viditelné pouze pro uživatele |
| **Implementation cost** | Vyžaduje migraci databáze a vývoj více služeb | Primárně frontendová práce |
| **Uživatelská zkušenost** | Centralizovaná logika, ale latence serveru může být o něco vyšší | Extrémně rychlá odezva s přímou interakcí GitHub |

Protože jsme chtěli rychlou integraci a minimální změny backendu, **konečně jsme zvolili „model přímého připojení frontendu**. Je to jako dát prohlížeči „dočasné povolení“. Jakmile prohlížeč získá povolení, může si věci na GitHubu vyřídit sám, aniž by pokaždé žádal administrátora backendu o schválení.

## Základní design: tok dat a zabezpečení

Po usazení na architektuře jsme potřebovali navrhnout konkrétní datový tok. Základem procesu synchronizace je to, jak bezpečně získat token a efektivně jej používat.

### Celkový diagram architektury

Celý systém lze abstrahovat do tří rolí: prohlížeč (frontend), backend HagiCode a GitHub.

```text
+--------------+        +--------------+        +--------------+
| Frontend     |        |   Backend    |        |   GitHub     |
| React        |        |   ASP.NET    |        |   REST API   |
|              |        |              |        |              |
|  +--------+  |        |              |        |              |
|  | OAuth  |--+--------> /callback    |        |              |
|  | Flow   |  |        |              |        |              |
|  +--------+  |        |              |        |              |
|              |        |              |        |              |
|  +--------+  |        |  +--------+  |        |  +--------+  |
|  | GitHub |  +------------> Session |  +----------> Issues |  |
|  | API    |  |        |  |Metadata|  |        |  |        |  |
|  | Direct |  |        |  +--------+  |        |  +--------+  |
|  +--------+  |        |              |        |              |
+--------------+        +--------------+        +--------------+
```

**Klíčový bod je**: pouze jeden malý krok v OAuth (výměna kódu za token) musí projít backendem. Poté se těžké zvedání (vytváření problémů) řeší přímo mezi frontendem a GitHubem.

### Synchronizace datového toku v detailu

Když uživatel klikne na tlačítko „Sync to GitHub“ v uživatelském rozhraní HagiCode, proběhne řada složitých akcí:

```text
User clicks "Sync to GitHub"
         │
         ▼
1. Frontend checks localStorage for the GitHub token
         │
         ▼
2. Format issue content (convert the Proposal into Markdown)
         │
         ▼
3. Frontend directly calls the GitHub API to create/update the issue
         │
         ▼
4. Call the HagiCode backend API to update Session.metadata (store issue URL and other info)
         │
         ▼
5. Backend broadcasts the SessionUpdated event via SignalR
         │
         ▼
6. Frontend receives the event and updates the UI to show the "Synced" state
```

### Návrh zabezpečení

Bezpečnost je vždy nejvyšší prioritou při integraci služeb třetích stran. Provedli jsme následující úvahy:

1.  **Obraňte se před útoky CSRF**: Vygenerujte náhodný `state` parametr během přesměrování OAuth a uložte jej `sessionStorage`. Přísně ověřte stav ve zpětném volání, abyste zabránili podvrženým požadavkům.
2.  **Izolované úložiště tokenů**: Token je uložen pouze v prohlížeči `localStorage`. Pomocí zásad stejného původu jej mohou číst pouze skripty HagiCode, čímž se zabrání riziku úniku databáze na straně serveru, který by ovlivnil uživatele.
3.  **Hranice chyb**: Navrhli jsme vyhrazené řešení pro běžné chyby GitHub API (jako je token 401, jehož platnost vypršela, selhání ověření 422 a omezení rychlosti 429), takže uživatelé dostávají přátelskou zpětnou vazbu.

## V praxi: Podrobnosti implementace v kódu

Teorie jde jen tak daleko. Podívejme se, jak kód ve skutečnosti funguje.

### 1. Minimální změny backendu

Backend potřebuje udělat pouze dvě věci: uložit informace o synchronizaci a zpracovat zpětné volání OAuth.

**Změny databáze**
Potřebujeme přidat pouze a `Metadata` sloupec k `Sessions` tabulky pro ukládání dat rozšíření ve formátu JSON.

### 2. Frontend OAuth Flow

Toto je vstupní bod připojení. Používáme standardní tok autorizačních kódů.

```typescript
// src/HagiCode.Client/src/services/githubOAuth.ts

// Generate the authorization URL and redirect
export async function generateAuthUrl(): Promise<string> {
  const state = generateRandomString(); // Generate a random string for CSRF protection
  sessionStorage.setItem('hagicode_github_state', state);
  
  const params = new URLSearchParams({
    client_id: clientId,
    redirect_uri: window.location.origin + '/settings?tab=github&oauth=callback',
    scope: ['repo', 'public_repo'].join(' '),
    state: state,
  });
  
  return `https://github.com/login/oauth/authorize?${params.toString()}`;
}

// Handle the code-to-token exchange on the callback page
export async function exchangeCodeForToken(code: string, state: string): Promise<GitHubToken> {
  // 1. Validate state to prevent CSRF
  const savedState = sessionStorage.getItem('hagicode_github_state');
  if (state !== savedState) throw new Error('Invalid state parameter');

  // 2. Call the backend API to exchange the token
  // Note: this must go through the backend because ClientSecret cannot be exposed to the frontend
  const response = await fetch('/api/GitHubOAuth/callback', {
    method: 'POST',
    headers: { 'Content-Type': 'application/json' },
    body: JSON.stringify({ code, state, redirectUri: window.location.origin + '/settings?tab=github&oauth=callback' }),
  });

  if (!response.ok) throw new Error('Failed to exchange token');
  
  const token = await response.json();
  
  // 3. Save into LocalStorage
  saveToken(token);
  return token;
}
```

### 3. GitHub API Client Wrapper

Jakmile máme token, potřebujeme solidní nástroj pro volání GitHub API.

```typescript
// src/HagiCode.Client/src/services/githubApiClient.ts

const GITHUB_API_BASE = 'https://api.github.com';

// Core request wrapper
async function githubApi<T>(endpoint: string, options: RequestInit = {}): Promise<T> {
  const token = localStorage.getItem('gh_token');
  if (!token) throw new Error('Not connected to GitHub');
  
  const response = await fetch(`${GITHUB_API_BASE}${endpoint}`, {
    ...options,
    headers: {
      ...options.headers,
      Authorization: `Bearer ${token}`,
      Accept: 'application/vnd.github.v3+json', // Specify the API version
    },
  });
  
  // Error handling logic
  if (!response.ok) {
    if (response.status === 401) throw new Error('GitHub token expired, please reconnect');
    if (response.status === 403) throw new Error('No permission to access this repository or rate limit exceeded');
    if (response.status === 422) throw new Error('Issue validation failed, the title may be duplicated');
    throw new Error(`GitHub API Error: ${response.statusText}`);
  }
  
  return response.json();
}

// Create issue
export async function createIssue(owner: string, repo: string, data: { title: string, body: string, labels: string[] }) {
  return githubApi(`/repos/${owner}/${repo}/issues`, {
    method: 'POST',
    body: JSON.stringify(data),
  });
}
```

### 4. Formátování a synchronizace obsahu

Posledním krokem je převod dat relace HagiCode do formátu problému na GitHubu. Je to trochu jako překladatelská práce.

```typescript
// Convert a Session object into a Markdown string
function formatIssueForSession(session: Session): string {
  let content = `# ${session.title}\n\n`;
  content += `**> HagiCode Session:** #${session.code}\n`;
  content += `**> Status:** ${session.status}\n\n`;
  content += `## Description\n\n${session.description || 'No description provided.'}\n\n`;
  
  // If this is a Proposal session, add extra fields
  if (session.type === 'proposal') {
    content += `## Chief Complaint\n\n${session.chiefComplaint || ''}\n\n`;
    // Add a deep link so users can jump back from GitHub to HagiCode
    content += `---\n\n**[View in HagiCode](hagicode://sessions/${session.id})**\n`;
  }
  
  return content;
}

// Main logic when clicking the sync button
const handleSync = async (session: Session) => {
  try {
    const repoInfo = parseRepositoryFromUrl(session.repoUrl); // Parse the repository URL
    if (!repoInfo) throw new Error('Invalid repository URL');

    toast.loading('Syncing to GitHub...');
    
    // 1. Format content
    const issueBody = formatIssueForSession(session);
    
    // 2. Call API
    const issue = await githubApiClient.createIssue(repoInfo.owner, repoInfo.repo, {
      title: `[HagiCode] ${session.title}`,
      body: issueBody,
      labels: ['hagicode', 'proposal', `status:${session.status}`],
    });
    
    // 3. Update Session Metadata (save the issue link)
    await SessionsService.patchApiSessionsSessionId(session.id, {
      metadata: {
        githubIssue: {
          owner: repoInfo.owner,
          repo: repoInfo.repo,
          issueNumber: issue.number,
          issueUrl: issue.html_url,
          syncedAt: new Date().toISOString(),
        }
      }
    });

    toast.success('Synced successfully!');
  } catch (err) {
    console.error(err);
    toast.error('Sync failed, please check your token or network');
  }
};
```

## Shrnutí a Outlook

S tímto přístupem „frontend direct connection“ jsme dosáhli bezproblémové integrace problémů GitHubu s co nejmenším backendovým kódem.

### Klíčové zisky
1.  **Vysoká efektivita vývoje**: Změny backendu jsou minimální, hlavně jedno pole databáze navíc a jednoduchý koncový bod zpětného volání OAuth. Většina logiky je dokončena na frontendu.
2.  **Silné zabezpečení**: Token neprochází databází serveru, což snižuje riziko úniku.
3.  **Skvělá uživatelská zkušenost**: Požadavky jsou iniciovány přímo z frontendu, takže rychlost odezvy je rychlá a není potřeba backend forwarding.

### Na co si dát pozor
Při reálném nasazení je třeba mít na paměti několik úskalí:
- **Nastavení aplikace OAuth**: Nezapomeňte zadat správné `Authorization callback URL` v nastavení aplikace GitHub OAuth (obvykle `http://localhost:3000/settings?tab=github&oauth=callback`).
- **Rate limity**: GitHub API omezuje neautentizované požadavky poměrně přísně, ale s tokenem je kvóta obvykle dostatečná (5000 požadavků/hodinu).
- ** Analýza adres URL**: Uživatelé zadávají všechny druhy adres URL úložiště, takže se ujistěte, že se váš regulární výraz může shodovat `.git` přípony, formáty SSH a podobné případy.

### Budoucí vylepšení
Aktuální funkcí je stále jednosměrná synchronizace (HagiCode -> GitHub). V budoucnu plánujeme implementaci obousměrné synchronizace prostřednictvím GitHub Webhooks. Pokud je například problém uzavřen na GitHubu, stav relace na straně HagiCode by se také mohl automaticky aktualizovat. To bude vyžadovat, abychom vystavili koncový bod webhooku na backendu, což bude zajímavý další krok.
