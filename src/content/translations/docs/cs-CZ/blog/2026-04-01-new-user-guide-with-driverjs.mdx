## Vytváření elegantního začleňování nových uživatelů do projektů React: Praxe driver.js HagiCode

> Když uživatelé poprvé otevřou váš produkt, opravdu vědí, kde začít? V tomto článku chci trochu pohovořit o tom, jak jsme použili driver.js pro nové uživatele v projektu HagiCode. Považujte to za malý praktický příklad pro zahájení konverzace.

## Pozadí

Už jste se někdy dostali do této situace? Nový uživatel se zaregistruje k vašemu produktu, otevře stránku a okamžitě vypadá ztraceně. Skenují kolem, nejsou si jisti, na co kliknout nebo co dělat dál. Jako vývojáři často předpokládáme, že uživatelé budou jen „prozkoumávat sami“, protože koneckonců lidská zvědavost je neomezená. Ale realita je jiná: většina uživatelů tiše odejde během několika minut, pokud nemohou najít správný vstupní bod, jako by příběh náhle začal a skončil stejně přirozeně.

Začlenění nového uživatele je důležitý způsob, jak tento problém vyřešit, ale dobře jej vytvořit není tak jednoduché. Dobrý onboardingový systém potřebuje:

- Přesně lokalizujte prvky stránky a zvýrazněte je
- Podporujte vícekrokové toky onboardingu
- Zapamatujte si volbu uživatele (dokončit nebo přeskočit)
- Vyvarujte se ovlivnění výkonu stránky a běžné interakce
- Udržujte strukturu kódu jasnou a snadno udržovatelnou

Při budování HagiCode jsme narazili na stejnou výzvu. HagiCode je asistent kódování AI a jeho základním pracovním postupem je pracovní postup OpenSpec, který vypadá takto: "uživatel vytvoří návrh -> AI vygeneruje plán -> uživatel jej zkontroluje -> AI jej provede." Pro uživatele, kteří se s tímto konceptem setkávají poprvé, je pracovní postup zcela nový, takže potřebují solidní integraci, aby mohli rychle začít. Na nové věci je potřeba si vždy chvíli zvykat.

## O HagiCode

Přístup sdílený v tomto článku vychází z našich praktických zkušeností v oblasti [HagiCode](https://hagicode.com) projekt. HagiCode je asistent kódování AI založený na Claude, který pomáhá vývojářům dokončit úlohy kódování efektivněji prostřednictvím pracovního postupu OpenSpec. Náš open-source kód si můžete prohlédnout na [GitHub](https://github.com/HagiCode-org/site).

## Proč jsme zvolili driver.js

Během fáze technického hodnocení jsme se podívali na několik běžných onboardingových knihoven. Každý z nich měl své silné stránky:

- **Intro.js**: Výkonný, ale relativně velký a přizpůsobení stylu je poněkud složité
- **Shepherd.js**: Dobře navržené API, ale trochu příliš „těžké“ pro náš případ použití
- **driver.js**: Lehké, stručné, intuitivní API a dobře funguje v ekosystému React

Nakonec jsme zvolili driver.js. Nebyl tam žádný zvlášť dramatický důvod. Výběr se zakládal hlavně na těchto úvahách:

1. **Lehká**: Základní knihovna je malá a výrazně nezvětšuje velikost balíku
2. **Jednoduché API**: Konfigurace je jasná a intuitivní, takže ji lze snadno vyzvednout
3. **Flexibilní**: Podporuje vlastní umístění, styl a chování při interakci
4. **Dynamický import**: Lze načíst na vyžádání bez ovlivnění výkonu na první obrazovce

Při výběru technologie jen zřídka existuje univerzálně nejlepší odpověď. Obvykle existuje pouze možnost, která nejlépe vyhovuje.

## Technická realizace

### Konfigurace jádra

driver.js má velmi intuitivní konfigurační model. Zde je základní konfigurace, kterou používáme v projektu HagiCode:

```typescript
import { driver } from 'driver.js';
import 'driver.js/dist/driver.css';

const newConversationDriver = driver({
  allowClose: true,           // Allow users to close the guide
  animate: true,              // Enable animations
  overlayClickBehavior: 'close', // Close the guide when the overlay is clicked
  disableActiveInteraction: false, // Keep elements interactive
  showProgress: false,        // Do not show the progress bar (we manage progress ourselves)
  steps: guideSteps           // Array of guide steps
});
```

Důvodem těchto nastavení je:

- `allowClose: true` - Respektujte volbu uživatele a nenuťte jej dokončit průvodce
- `disableActiveInteraction: false` - Některé kroky vyžadují skutečné akce uživatele, jako je zadávání vstupu, takže interakci nelze zakázat
- `overlayClickBehavior: 'close'` - Poskytněte uživatelům rychlý způsob ukončení

### Řízení státu

Přetrvávání stavu registrace je kritické. Nechceme restartovat průvodce pokaždé, když se stránka obnoví, protože to rychle otravuje. HagiCode používá `localStorage` pro správu stavu průvodce:

```typescript
export type GuideState = 'pending' | 'dismissed' | 'completed';

export interface UserGuideState {
  session: GuideState;
  detailGuides: Record<string, GuideState>;
}

// Read state
export const getUserGuideState = (): UserGuideState => {
  const state = localStorage.getItem('userGuideState');
  return state ? JSON.parse(state) : { session: 'pending', detailGuides: {} };
};

// Update state
export const setUserGuideState = (state: UserGuideState) => {
  localStorage.setItem('userGuideState', JSON.stringify(state));
};
```

Definujeme tři stavy:

- `pending`: Příručka stále probíhá a uživatel ji nedokončil nebo nepřeskočil
- `dismissed`: Uživatel průvodce proaktivně zavřel
- `completed`: Uživatel dokončil všechny kroky

Pro přihlášení na stránku s podrobnostmi o návrhu také podporujeme podrobnější sledování stavu prostřednictvím `detailGuides` mapa, protože jeden návrh může projít několika fázemi – návrh, revize a realizace jsou dokončeny – a každá fáze potřebuje jiné vedení. Stav věcí se neustále mění a onboarding by to měl odrážet.

### Výběr cílového prvku

driver.js používá CSS selektory k nalezení cílových prvků. HagiCode se řídí jednoduchou konvencí: použijte vlastní `data-guide` atribut pro označení cílů pro přihlášení:

```typescript
const steps = [
  {
    element: '[data-guide="launch"]',
    popover: {
      title: 'Start a New Conversation',
      description: 'Click here to create a new conversation session...'
    }
  }
];
```

V komponentách to vypadá takto:

```typescript
<button data-guide="launch" onClick={handleLaunch}>
  New Conversation
</button>
```

Výhody tohoto přístupu jsou:

- Vyhněte se konfliktům s názvy tříd obchodního stylu
- Jasná sémantika, takže okamžitě poznáte, že prvek souvisí s onboardingem
- Snadnější správa a důsledná údržba

### Dynamická optimalizace importu

Protože je funkce registrace potřebná pouze ve specifických scénářích, jako je první návštěva uživatele, používáme dynamické importy k optimalizaci výkonu počátečního načítání:

```typescript
const initNewUserGuide = async () => {
  // Dynamically import driver.js
  const { driver } = await import('driver.js');
  await import('driver.js/dist/driver.css');

  // Initialize the guide
  const newConversationDriver = driver({
    // ...configuration
  });

  newConversationDriver.drive();
};
```

Tímto způsobem se driver.js a jeho šablona stylů načtou pouze v případě potřeby a neovlivní výkon na první obrazovce. Málokoho baví čekat na něco, co ještě ani nepotřebuje.

## Onboarding Flow Design

HagiCode implementuje dvě vstupní cesty, které pokrývají základní scénáře uživatele.

### Registrace relace (10 kroků)

Tato cesta k registraci pomáhá uživatelům dokončit celý proces od vytvoření konverzace až po odeslání první kompletní nabídky:

1. **spustit** – Spusťte průvodce a zadejte tlačítko „Nová konverzace“.
2. **compose** – Navede uživatele k zadání požadavku do vstupního pole
3. **odeslat** – Navede uživatele ke kliknutí na tlačítko Odeslat
4. **proposal-launch-readme** – Provede uživatele vytvořením návrhu README
5. **proposal-compose-readme** – Provede uživatele úpravou obsahu požadavku README
6. **proposal-submit-readme** – Navede uživatele k odeslání návrhu README
7. **proposal-launch-agents** – Průvodce uživatele k vytvoření `AGENTS.md` Návrh
8. **proposal-compose-agents** – Průvodce uživatele k úpravě `AGENTS.md` žádost
9. **proposal-submit-agents** – Průvodce uživatele k odeslání `AGENTS.md` Návrh
10. **proposal-wait** – Vysvětlete, že AI zpracovává a uživatel by měl chvíli počkat

Smyslem této cesty je umožnit uživatelům vyzkoušet si základní pracovní postup HagiCode z první ruky prostřednictvím dvou skutečných úkolů vytváření návrhů, z nichž jeden `README` a jeden pro `AGENTS.md`. Je velký rozdíl slyšet o pracovním postupu a projít si ho sami.

### Registrace podrobností nabídky (3 kroky)

Když uživatelé vstoupí na stránku podrobností návrhu, HagiCode spustí odpovídající průvodce na základě aktuálního stavu návrhu:

1. **návrh** (fáze návrhu) – Proveďte uživatele ke kontrole plánu vygenerovaného umělou inteligencí
2. **recenze** (fáze kontroly) – Vede uživatele k provedení plánu
3. **executionCompleted** (dokončená fáze) – Navede uživatele k archivaci plánu

Charakteristickým rysem této příručky je, že je řízena stavem: dynamicky rozhoduje o tom, který vstupní krok se zobrazí, na základě skutečného stavu návrhu. Věci se mění a onboarding by se měl měnit s nimi.

V této fázi průvodce zaměřuje pozornost uživatele na klíčovou akci generování plánu, takže se při prvním vstupu na stránku s podrobnostmi nemusí divit, co udělat jako první.

### Mechanismus opakování vykreslení prvku

V aplikacích React nemusí cílový onboarding element ještě dokončit vykreslování, například proto, že se stále načítají asynchronní data. Aby se to vyřešilo, HagiCode implementuje mechanismus opakování:

```typescript
const waitForElement = (selector: string, maxRetries = 10, interval = 100) => {
  let retries = 0;
  return new Promise<HTMLElement>((resolve, reject) => {
    const checkElement = () => {
      const element = document.querySelector(selector) as HTMLElement;
      if (element) {
        resolve(element);
      } else if (retries < maxRetries) {
        retries++;
        setTimeout(checkElement, interval);
      } else {
        reject(new Error(`Element not found: ${selector}`));
      }
    };
    checkElement();
  });
};
```

Tuto funkci zavolejte před inicializací průvodce, abyste se ujistili, že cílový prvek již existuje. Někdy se vyplatí počkat o něco déle.

## Shrnutí osvědčených postupů

Na základě praktických zkušeností HagiCode je zde několik klíčových osvědčených postupů:

### 1. Výstup by měl být snadný

Nenuťte uživatele dokončit registraci. Někteří uživatelé jsou od přírody průzkumníci a raději na věci přicházejí sami. Poskytněte jasné tlačítko „Přeskočit“, zapamatujte si jejich volbu a příště už je nepřerušujte.

### 2. Udržujte obsah na palubě krátký a ostrý

Každý vstupní krok by se měl zaměřit na jeden cíl:

- **Nadpis**: Krátký a jasný, ideálně ne více než pár slov
- **Popis**: Přejděte přímo k věci a řekněte uživateli, co to je a proč na tom záleží

Vyvarujte se sáhodlouhých vysvětlování. Pozornost uživatelů je během onboardingu omezená a čím více toho řeknete, tím menší je pravděpodobnost, že si to přečtou.

### 3. Udržujte voliče stabilní

Používejte stabilní značky prvků, které se často nemění. zvyk `data-guide` atribut je dobrá volba. Vyhněte se závislosti na názvech tříd nebo struktuře DOM, protože ty lze snadno změnit během refaktorů. Kód se neustále mění, ale některé kotvy by měly zůstat stabilní, pokud je to možné.

### 4. Otestujte svou registraci

HagiCode obsahuje kompletní testovací případy pro funkci onboarding:

```typescript
describe('NewUserConversationGuide', () => {
  it('should initialize guide state correctly', () => {
    const state = getUserGuideState();
    expect(state.session).toBe('pending');
  });

  it('should update guide state correctly', () => {
    setUserGuideState({ session: 'completed', detailGuides: {} });
    const state = getUserGuideState();
    expect(state.session).toBe('completed');
  });
});
```

Testy pomáhají zajistit, že refaktoring náhodně nenaruší chování při začleňování. Nikdo nechce, aby malá změna kódu tiše poškodila dříve fungující funkčnost.

### 5. Optimalizace výkonu

- Použijte dynamické importy k pomalému načítání onboardingové knihovny
- Vyhněte se inicializaci logiky onboardingu poté, co uživatel již průvodce dokončil
- Zvažte dopad animací na výkon a v případě potřeby je na zařízeních nižší třídy deaktivujte

Výkon, jako mnoho věcí v životě, si zaslouží trochu pečlivého rozpočtu.

## Závěr

Začlenění nových uživatelů je důležitou součástí zlepšování uživatelské zkušenosti produktu. V projektu HagiCode jsme použili driver.js k vybudování kompletního onboardingového systému, který pokrývá celý pracovní postup od vytvoření relace až po provedení návrhu.

Hlavní body, o které se chceme podělit prostřednictvím tohoto článku, jsou:

1. **Technické možnosti by měly odpovídat skutečným potřebám**: driver.js není nejvýkonnější možností, ale je pro nás nejvhodnější
2. **Správa státu je kritická**: Použijte `localStorage` Chcete-li přetrvat stav registrace a vyhnout se opakovanému vyrušování uživatelů
3. **Návrh registrace by měl zůstat soustředěný**: Každý krok by měl vyřešit jeden problém, nic víc
4. **Struktura kódu by měla zůstat jasná**: Oddělená konfigurace pro přihlášení, správa stavu a logika uživatelského rozhraní pro snazší údržbu
