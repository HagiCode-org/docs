## Duální databáze .NET Core v praxi: Nechte PostgreSQL a SQLite koexistovat v klidu

> Při vytváření moderních aplikací často čelíme tomuto kompromisu: vývojová prostředí chtějí něco lehkého a pohodlného, zatímco produkční prostředí vyžadují vysokou souběžnost a vysokou dostupnost. Tento článek uvádí, jak elegantně podporovat PostgreSQL i SQLite v projektu .NET Core a implementovat osvědčený postup „SQLite pro vývoj, PostgreSQL pro produkci“.

Ale ve výrobě jsme potřebovali zvládnout vysoce souběžné zápisy a složité dotazy JSON, a to je přesně místo, kde odlehčený SQLite začíná ukazovat své limity.

## Hlavní téma 1: Návrh architektury a jednotná abstrakce

Pro podporu dvou databází v .NET Core je klíčovou myšlenkou záviset spíše na abstrakcích než na konkrétních implementacích. Musíme oddělit výběr databáze od obchodního kódu a nechat rozhodnout konfigurační vrstvu.

### Designový přístup

1.  **Jednotné rozhraní**: Veškerá obchodní logika by měla záviset na `DbContext` základní třída nebo vlastní rozhraní, spíše než konkrétní `PostgreSqlDbContext`.
2.  **Řízené konfigurací**: Používejte konfigurační položky v `appsettings.json` dynamicky rozhodnout, kterého poskytovatele databáze načíst při spuštění aplikace.
3.  **Izolace funkcí**: Přidejte adaptační logiku pro funkce specifické pro PostgreSQL, jako je JSONB, takže aplikace může stále plynule degradovat na SQLite.

### Implementace kódu: Konfigurace dynamického kontextu

V ASP.NET Core `Program.cs`, neměli bychom napevno kódovat `UseNpgsql` nebo `UseSqlite`. Místo toho bychom měli číst konfiguraci a rozhodovat se dynamicky.

Nejprve definujte konfigurační třídu:

```csharp
public class DatabaseSettings
{
    public const string SectionName = "Database";
    
    // Database type: PostgreSQL or SQLite
    public string DbType { get; set; } = "PostgreSQL"; 
    
    // Connection string
    public string ConnectionString { get; set; } = string.Empty;
}
```

Poté službu zaregistrujte `Program.cs` na základě konfigurace:

```csharp
// Read configuration
var databaseSettings = builder.Configuration.GetSection(DatabaseSettings.SectionName).Get<DatabaseSettings>();

// Register DbContext
builder.Services.AddDbContext<ApplicationDbContext>(options =>
{
    if (databaseSettings?.DbType?.ToLower() == "sqlite")
    {
        // SQLite configuration
        options.UseSqlite(databaseSettings.ConnectionString);
        
        // Handling SQLite's concurrent write limitations
        // Note: in production, enabling WAL mode is recommended to improve concurrency performance
    }
    else
    {
        // PostgreSQL configuration (default)
        options.UseNpgsql(databaseSettings.ConnectionString, npgsqlOptions =>
        {
            // Enable JSONB support, which is very useful when handling AI conversation records
            npgsqlOptions.UseJsonNet(); 
        });
        
        // Configure connection pool retry policy
        options.EnableRetryOnFailure(3);
    }
});
```

## Hlavní téma 2: Řešení rozdílů a strategie migrace

Přestože PostgreSQL i SQLite podporují standard SQL, výrazně se liší ve specifických schopnostech a chování. Pokud se s těmito rozdíly nezachází opatrně, můžete se snadno dostat do nepříjemné situace, kdy vše funguje lokálně, ale po nasazení selže.

### 1. Práce s typy JSON

V HagiCode potřebujeme ukládat velké množství výzev a metadat AI, což obvykle zahrnuje sloupce JSON.
- **PostgreSQL**: Má nativní `JSONB` typu s vynikajícím výkonem dotazů.
- **SQLite**: Nemá nativní typ JSON (novější verze obsahují rozšíření JSON1, ale mapování objektů se stále liší), takže data jsou obvykle uložena jako TEXT.

**Řešení**:
V mapování entity EF Core jej nakonfigurujeme jako konvertibilní typ.

```csharp
protected override void OnModelCreating(ModelBuilder modelBuilder)
{
    base.OnModelCreating(modelBuilder);

    // Configure entity
    modelBuilder.Entity<PromptTemplate>(entity => 
    {
        entity.Property(e => e.Metadata)
              .HasColumnType("jsonb") // PG uses jsonb
              .HasConversion(
                  v => JsonSerializer.Serialize(v, (JsonSerializerOptions)null),
                  v => JsonSerializer.Deserialize<Dictionary<string, object>>(v, (JsonSerializerOptions)null)
              ); 
    });
}
```

Když se používá SQLite, `HasColumnType("jsonb")` mohou být ignorovány nebo vyvolat varování. Nicméně, protože `HasConversion` je nakonfigurován, data jsou stále správně serializována a deserializována jako řetězce uložené v poli TEXT, což zajišťuje kompatibilitu.

### 2. Oddělování migračních strategií

Nikdy se nesnažte, aby jedna sada migračních skriptů fungovala současně pro PostgreSQL i SQLite. Rozdíly ve strategiích generování primárního klíče, syntaxi indexu a dalších podrobnostech databáze nevyhnutelně způsobí selhání.

**Doporučený postup**:
Udržujte dvě migrační větve nebo projekty. Ve vývojovém pracovním postupu HagiCode to řešíme takto:

1.  **Fáze vývoje**: Práce hlavně s SQLite. Použijte `Add-Migration Init_Sqlite -OutputDir Migrations/Sqlite`.
2.  **Fáze adaptace**: Po vyvinutí funkce přepněte připojovací řetězec na PostgreSQL a spusťte `Add-Migration Init_Postgres -OutputDir Migrations/Postgres`.
3.  **Automatizační skripty**: Napište jednoduchý skript PowerShell nebo Bash, který automaticky použije správnou migraci na základě aktuální proměnné prostředí.

```bash
# Pseudocode for simple deployment logic
if [ "$DATABASE_PROVIDER" = "PostgreSQL" ]; then
    dotnet ef database update --project Migrations.Postgres
else
    dotnet ef database update --project Migrations.Sqlite
fi
```

## Hlavní téma 3: Poučení z HagiCode v produkci

Při refaktorování **HagiCode** z modelu s jednou databází na podporu dvou databází jsme narazili na několik úskalí a shromáždili jsme několik důležitých lekcí, které vám mohou pomoci vyhnout se stejným chybám.

### 1. Rozdíly v souběhu a transakcích

PostgreSQL používá architekturu server-klient a podporuje vysoce souběžné zápisy s výkonnými úrovněmi izolace transakcí. SQLite používá zamykání souborů, takže operace zápisu zamykají celý databázový soubor, pokud není povolen režim WAL.

**Doporučení**:
Při psaní obchodní logiky, která zahrnuje časté zápisy, jako je ukládání stavu úprav uživatele v reálném čase, musíte vzít v úvahu model zamykání SQLite. Při navrhování modulu spolupráce OpenSpec v **HagiCode** jsme zavedli mechanismus „sloučení před zápisem“, abychom snížili frekvenci přímých zápisů do databáze, což nám umožňuje zachovat dobrý výkon na obou databázích.

### 2. Správa životního cyklu spojovacích řetězců

Navázání připojení PostgreSQL je poměrně nákladné a závisí na sdružování připojení. Připojení SQLite jsou velmi lehká, ale pokud nejsou okamžitě uvolněna, zámky souborů mohou způsobit časový limit pozdějších operací.

In `Program.cs`, můžeme doladit chování pro každou databázi:

```csharp
if (databaseSettings?.DbType?.ToLower() == "sqlite")
{
    // SQLite: keeping connections open can improve performance, but watch out for file locks
    options.UseSqlite(connectionString, sqliteOptions =>
    {
        // Set command timeout
        sqliteOptions.CommandTimeout(30);
    });
}
else
{
    // PG: make full use of connection pooling
    options.UseNpgsql(connectionString, npgsqlOptions =>
    {
        npgsqlOptions.MaxBatchSize(100);
        npgsqlOptions.CommandTimeout(30);
    });
}
```

Mnoho vývojářů, včetně některých prvních členů našeho týmu, má tendenci dělat jednu běžnou chybu: spouštět testy jednotek pouze ve vývojovém prostředí, což je obvykle SQLite.
