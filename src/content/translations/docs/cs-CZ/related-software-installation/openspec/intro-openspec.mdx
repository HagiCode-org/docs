# Úvod do OpenSpec

OpenSpec je nástroj pro vývoj řízený specifikacemi, který pomáhá týmům spravovat proces vývoje softwaru prostřednictvím strukturovaných návrhů a specifikací. Tento článek poskytuje komplexní úvod do základních konceptů, pracovních postupů a metod používání OpenSpec.

## Co je OpenSpec?

OpenSpec je nástroj příkazového řádku používaný ke správě návrhů (Proposals), specifikací (Specifications) a změn (Changes) v procesu vývoje softwaru. Standardizuje vývojový proces a zajišťuje, že každá funkce prochází důkladným návrhem a kontrolou od nápadu po implementaci.

### Základní principy

OpenSpec je založen na následujících základních principech:

- **Specifikace na prvním místě**: Definujte jasné specifikace před psaním kódu
- **Strukturovaná spolupráce**: Podporujte týmovou spolupráci prostřednictvím standardizovaných formátů dokumentů
- **Sledovatelnost**: Zaznamenávejte celý proces od návrhu po implementaci
- **Zajištění kvality**: Zajistěte, že kód odpovídá záměru návrhu prostřednictvím validace specifikací

### Hlavní funkce

OpenSpec poskytuje následující hlavní funkce:

1. **Správa návrhů**: Vytvářejte, kontrolujte a sledujte návrhy funkcí
2. **Definice specifikací**: Pište a udržujte dokumenty technických specifikací
3. **Sledování změn**: Zaznamenávejte celou historii kontrolovaných změn
4. **Validace specifikací**: Zajistěte, že změny kódu odpovídají předem definovaným specifikacím
5. **Týmová spolupráce**: Podporujte procesy vývoje a kontroly s účastí více lidí

## Pracovní postup OpenSpec

Typický pracovní postup OpenSpec zahrnuje následující fáze:

### 1. Vytvoření návrhu (Proposal)

Když existuje nová funkce nebo požadavek na významnou změnu, nejprve vytvořte návrh:

```bash
openspec proposal create
```

Návrh zahrnuje:
- **Popis problému**: Problém nebo požadavek, který je třeba vyřešit
- **Řešení**: Přehled navrhovaného řešení
- **Rozsah dopadu**: Dopad na stávající systém
- **Alternativní řešení**: Další zvažovaná řešení

### 2. Psaní specifikace (Specification)

Po schválení návrhu napište podrobnou technickou specifikaci:

```bash
openspec spec create
```

Specifikace zahrnuje:
- **Technický návrh**: Podrobný plán technické implementace
- **Definice rozhraní**: Návrh rozhraní jako API, datové struktury atd.
- **Detaily implementace**: Klíčová logika implementace a algoritmy
- **Testovací plán**: Jak ověřit správnost implementace

### 3. Vytvoření změny (Change)

Když začnete s implementací, vytvořte změnu pro sledování vývojového procesu:

```bash
openspec change create
```

Změna zahrnuje:
- **Přidružená specifikace**: Specifikace implementovaná touto změnou
- **Úkoly implementace**: Seznam konkrétních vývojových úkolů
- **Sledování pokroku**: Stav dokončení úkolů
- **Změny kódu**: Skutečné úpravy kódu

### 4. Validace a archivace

Po dokončení vývoje ověřte, zda změna odpovídá specifikaci, a poté ji archivujte:

```bash
openspec change validate
openspec change archive
```

## Struktura adresářů OpenSpec

OpenSpec vytváří adresář `openspec/` v kořenovém adresáři projektu s následující strukturou:

```
openspec/
├── proposals/           # Adresář návrhů
│   ├── PROPOSAL-001.md
│   └── PROPOSAL-002.md
├── specs/              # Adresář specifikací
│   ├── SPEC-001.md
│   └── SPEC-002.md
├── changes/            # Adresář změn
│   ├── feature-name/
│   │   ├── proposal.md
│   │   ├── design.md
│   │   ├── specs.md
│   │   └── tasks.md
│   └── another-feature/
└── archived/           # Archivované změny
    └── feature-name/
```

### Popis adresářů

- **proposals/**: Ukládá všechny dokumenty návrhů, každý návrh v samostatném souboru Markdown
- **specs/**: Ukládá dokumenty technických specifikací odpovídající návrhům
- **changes/**: Ukládá probíhající změny, každá změna v podadresáři
- **archived/**: Ukládá dokončené a archivované změny

## Základní příkazy

### Příkazy související s návrhy

```bash
# Vytvoření nového návrhu
openspec proposal create

# Seznam všech návrhů
openspec proposal list

# Zobrazení detailů návrhu
openspec proposal view <proposal-id>

# Schválení návrhu
openspec proposal approve <proposal-id>
```

### Příkazy související se specifikacemi

```bash
# Vytvoření nové specifikace
openspec spec create

# Seznam všech specifikací
openspec spec list

# Zobrazení detailů specifikace
openspec spec view <spec-id>

# Aktualizace specifikace
openspec spec update <spec-id>
```

### Příkazy související se změnami

```bash
# Vytvoření nové změny
openspec change create

# Seznam všech změn
openspec change list

# Zobrazení detailů změny
openspec change view <change-name>

# Přidání úkolu
openspec change add-task <change-name>

# Validace změny
openspec change validate <change-name>

# Archivace změny
openspec change archive <change-name>
```

### Obecné příkazy

```bash
# Zobrazení verze
openspec --version

# Zobrazení nápovědy
openspec --help

# Zobrazení nápovědy pro konkrétní příkaz
openspec <command> --help
```

## Použití OpenSpec v projektu HagiCode

### Integrace projektu

V projektu HagiCode je OpenSpec integrován do vývojového pracovního postupu:

1. **Kontrola při inicializaci**: Průvodce spuštěním kontroluje, zda je OpenSpec nainstalován
2. **Vývoj řízený návrhy**: Všechny nové funkce musí projít procesem návrhů OpenSpec
3. **Kontrola kódu**: Změny musí odpovídat specifikacím OpenSpec, aby mohly být sloučeny
4. **Synchronizace dokumentace**: Dokumentace OpenSpec je udržována synchronizovaná s implementací kódu

### Požadavky na verzi

Projekt HagiCode vyžaduje použití verze **OpenSpec >=1.0.0 {'<'}2.0.0**.

Příkaz pro instalaci:

```bash
npm install -g @fission-ai/openspec@1
```

### Příklad pracovního postupu

Zde je kompletní příklad použití OpenSpec v projektu HagiCode:

#### Krok 1: Vytvoření návrhu

```bash
cd /path/to/hagicode-project
openspec proposal create
```

Zadejte do interaktivní výzvy:
- Název návrhu: Přidání funkce ověřování uživatelů
- Popis problému: Systém potřebuje přihlášení uživatelů a správu oprávnění
- Řešení: Implementace ověřovacího systému založeného na JWT

#### Krok 2: Psaní specifikace

Po schválení návrhu vytvořte podrobnou specifikaci:

```bash
openspec spec create
```

Definujte ve specifikaci:
- Návrh procesu ověřování
- Definice rozhraní API
- Struktura tabulek databáze
- Bezpečnostní úvahy

#### Krok 3: Vytvoření změny

Začněte s implementací:

```bash
openspec change create
```

Zadejte název změny: `add-user-authentication`

#### Krok 4: Přidání úkolů

```bash
openspec change add-task add-user-authentication
```

Přidejte konkrétní úkoly:
- Implementace rozhraní pro registraci uživatelů
- Implementace rozhraní pro přihlášení
- Implementace JWT middleware
- Psaní jednotkových testů

#### Krok 5: Implementace a validace

Po dokončení kódu ověřte změnu:

```bash
openspec change validate add-user-authentication
```

#### Krok 6: Archivace

Po úspěšné validaci archivujte změnu:

```bash
openspec change archive add-user-authentication
```

## Osvědčené postupy

### Psaní návrhů

1. **Jasné vyjádření problému**: Jasně definujte problém, který je třeba vyřešit, vyhněte se nejasným popisům
2. **Zvažte alternativní řešení**: Uveďte a porovnejte různá řešení
3. **Vyhodnoťte dopad**: Analyzujte dopad a rizika pro stávající systém
4. **Buďte struční**: Návrhy by měly být stručné a jasné, podrobnosti nechte pro specifikaci

### Psaní specifikací

1. **Podrobně, ale ne nadbytečně**: Poskytněte dostatek detailů, ale vyhněte se nadměrnému návrhu
2. **Používejte diagramy**: Používejte vývojové diagramy, architektonické diagramy atd. pro ilustraci
3. **Definujte rozhraní**: Jasně definujte všechna veřejná rozhraní a datové struktury
4. **Zvažte hraniční případy**: Uveďte výjimečné situace a zpracování chyb

### Správa změn

1. **Iterujte v malých krocích**: Rozdělte velké funkce na více malých změn
2. **Aktualizujte včas**: Udržujte synchronizovaný stav úkolů a dokumentaci
3. **Kontrola kódu**: Proveďte kontrolu kódu po dokončení změny
4. **Archivujte včas**: Archivujte dokončené změny včas, udržujte pracovní adresář čistý

### Týmová spolupráce

1. **Pravidelné kontroly**: Pravidelně kontrolujte návrhy a specifikace, abyste zajistili konsenzus v týmu
2. **Dokumentace na prvním místě**: Dokončete dokumentaci před zahájením kódování
3. **Udržujte komunikaci**: Zaznamenávejte diskuse a rozhodnutí v návrzích a specifikacích
4. **Správa verzí**: Zahrňte dokumentaci OpenSpec do správy verzí

## Běžné scénáře použití

### Scénář 1: Přidání nové funkce

1. Vytvořte návrh popisující požadavky na novou funkci
2. Tým diskutuje a schvaluje návrh
3. Napište podrobnou technickou specifikaci
4. Vytvořte změnu a implementujte funkci
5. Ověřte a archivujte změnu

### Scénář 2: Refaktoring existujícího kódu

1. Vytvořte návrh vysvětlující důvody a cíle refaktoringu
2. Definujte plán a kroky refaktoringu ve specifikaci
3. Vytvořte změnu a postupně refaktorujte
4. Zajistěte pokrytí testy, ověřte výsledky refaktoringu
5. Archivujte změnu

### Scénář 3: Oprava chyby

Pro jednoduché chyby:
- Lze opravit přímo, není nutný úplný proces OpenSpec

Pro složité chyby:
1. Vytvořte návrh pro analýzu hlavní příčiny chyby
2. Definujte řešení opravy ve specifikaci
3. Vytvořte změnu a implementujte opravu
4. Přidejte testy pro prevenci regrese
5. Archivujte změnu

### Scénář 4: Správa technického dluhu

1. Vytvořte návrh uvádějící položky technického dluhu
2. Definujte plán zlepšení ve specifikaci
3. Vytvořte změnu pro postupné splácení technického dluhu
4. Sledujte pokrok a efekt
5. Archivujte dokončená zlepšení

## Integrace s dalšími nástroji

### Integrace s Git

OpenSpec se bezproblémově integruje s Git:

```bash
# Dokumentace OpenSpec by měla být zahrnuta do správy verzí
git add openspec/
git commit -m "Add OpenSpec proposal for feature X"
```

### Integrace s CI/CD

Ověřte specifikace OpenSpec v procesu CI/CD:

```yaml
# .github/workflows/openspec.yml
name: OpenSpec Validation
on: [pull_request]
jobs:
  validate:
    runs-on: ubuntu-latest
    steps:
      - uses: actions/checkout@v2
      - name: Install OpenSpec
        run: npm install -g @fission-ai/openspec@1
      - name: Validate changes
        run: openspec change validate
```

### Integrace s IDE

Doporučuje se používat editor podporující Markdown pro úpravu dokumentů OpenSpec:
- VS Code + plugin Markdown
- JetBrains IDEs + podpora Markdown
- Vim/Neovim + plugin Markdown

## Řešení problémů

### Časté problémy

#### Problém: Příkaz OpenSpec nebyl nalezen

**Řešení**:
1. Potvrďte, že OpenSpec je nainstalován: `npm list -g @fission-ai/openspec`
2. Zkontrolujte, zda proměnná prostředí PATH obsahuje globální bin adresář npm
3. Přeinstalujte: `npm install -g @fission-ai/openspec@1`

#### Problém: Nelze vytvořit návrh nebo změnu

**Řešení**:
1. Potvrďte, že aktuální adresář je kořenový adresář projektu
2. Zkontrolujte, zda existuje adresář `openspec/`, pokud ne, vytvořte jej
3. Potvrďte, že máte oprávnění k zápisu

#### Problém: Validace selhala

**Řešení**:
1. Zkontrolujte, zda je dokumentace v adresáři změn úplná
2. Potvrďte, že všechny požadované úkoly jsou dokončeny
3. Prohlédněte si podrobné informace o chybě: `openspec change validate <change-name> --verbose`

#### Problém: Archivace selhala

**Řešení**:
1. Ujistěte se, že změna prošla validací
2. Zkontrolujte, zda adresář `openspec/archived/` existuje
3. Potvrďte, že nejsou žádné nepotvrzené změny kódu

## Pokročilé použití

### Vlastní šablony

OpenSpec podporuje vlastní šablony pro návrhy a specifikace:

```bash
# Vytvořte adresář šablon v kořenovém adresáři projektu
mkdir -p .openspec/templates

# Vytvořte vlastní šablonu návrhu
cat > .openspec/templates/proposal.md << 'EOF'
# Proposal: {{title}}

## Problem Statement
{{problem}}

## Proposed Solution
{{solution}}

## Custom Section
Your custom content here
EOF
```

### Konfigurační soubor

Vytvořte konfigurační soubor `.openspec.json` v kořenovém adresáři projektu:

```json
{
  "version": "1.0",
  "templates": {
    "proposal": ".openspec/templates/proposal.md",
    "spec": ".openspec/templates/spec.md"
  },
  "validation": {
    "requireTests": true,
    "requireDocs": true
  },
  "hooks": {
    "pre-archive": "npm test"
  }
}
```

### Háčky (Hooks)

OpenSpec podporuje spouštění vlastních skriptů před a po konkrétních operacích:

```json
{
  "hooks": {
    "pre-create": "scripts/check-permissions.sh",
    "post-archive": "scripts/notify-team.sh"
  }
}
```
