# Introductie tot OpenSpec

OpenSpec is een specificatiegestuurd ontwikkelhulpmiddel dat teams helpt het softwareontwikkelproces te beheren via gestructureerde voorstellen en specificaties. Dit artikel biedt een volledige introductie tot de kernconcepten, workflows en gebruikswijzen van OpenSpec.

## Wat is OpenSpec?

OpenSpec is een opdrachtregeltool die wordt gebruikt om voorstellen (Proposals), specificaties (Specifications) en wijzigingen (Changes) te beheren in het softwareontwikkelproces. Het standaardiseert de ontwikkelworkflow en zorgt ervoor dat elke functie een grondig ontwerp en review doorloopt van idee tot implementatie.

### Kernprincipes

OpenSpec is gebaseerd op de volgende kernprincipes:

- **Specificaties eerst**: Definieer duidelijke specificaties voordat u code schrijft
- **Gestructureerde samenwerking**: Stimuleer teamsamenwerking via gestandaardiseerde documentformaten
- **Traceerbaarheid**: Leg het volledige proces van voorstel tot implementatie vast
- **Kwaliteitsborging**: Zorg ervoor dat code voldoet aan de ontwerpsintentie door specificaties te valideren

### Belangrijkste functies

OpenSpec biedt de volgende belangrijkste functies:

1. **Voorstelbeheer**: Maak, beoordeel en volg functieverzoeken
2. **Specificatiedefinitie**: Schrijf en onderhoud technische specificatiedocumenten
3. **Wijzigingsregistratie**: Registreer de volledige geschiedenis van beoordeelde wijzigingen
4. **Specificatievalidatie**: Zorg ervoor dat codewijzigingen voldoen aan vooraf gedefinieerde specificaties
5. **Teamsamenwerking**: Ondersteun ontwikkeling- en beoordelingsprocessen met meerdere deelnemers

## OpenSpec workflow

De typische OpenSpec workflow omvat de volgende fasen:

### 1. Maak een voorstel (Proposal)

Wanneer er een nieuwe functie is of een grote wijziging vereist is, maak eerst een voorstel:

```bash
openspec proposal create
```

Het voorstel bevat:
- **Probleembeschrijving**: Het probleem of de vereiste die moet worden opgelost
- **Oplossing**: Overzicht van de voorgestelde oplossing
- **Omvang van impact**: Impact op het bestaande systeem
- **Alternatieve oplossingen**: Andere overwogen oplossingen

### 2. Schrijf de specificatie (Specification)

Nadat het voorstel is goedgekeurd, schrijf een gedetailleerde technische specificatie:

```bash
openspec spec create
```

De specificatie bevat:
- **Technisch ontwerp**: Gedetailleerd plan voor technische implementatie
- **Interface-definitie**: Ontwerp van interfaces zoals API's, datastructuren enz.
- **Implementatiedetails**: Kernimplementatielogica en algoritmen
- **Testplan**: Hoe de correctheid van de implementatie te verifi√Ђren

### 3. Maak een wijziging (Change)

Wanneer u begint met implementeren, maak een wijziging om het ontwikkelproces bij te houden:

```bash
openspec change create
```

De wijziging bevat:
- **Gerelateerde specificatie**: De specificatie die door deze wijziging wordt ge√ѓmplementeerd
- **Implementatietaken**: Lijst met concrete ontwikkeltaken
- **Voortgangsregistratie**: Status van voltooiing van taken
- **Codewijzigingen**: De daadwerkelijke codewijzigingen

### 4. Validatie en archivering

Na voltooiing van de ontwikkeling, valideer of de wijziging aan de specificatie voldoet en archiveer deze:

```bash
openspec change validate
openspec change archive
```

## Kerncommando's

### Voorstelgerelateerde commando's

```bash
# Maak een nieuw voorstel
openspec proposal create

# Lijst van alle voorstellen
openspec proposal list

# Bekijk voorstel details
openspec proposal view <proposal-id>

# Keur voorstel goed
openspec proposal approve <proposal-id>
```

### Specificatiegerelateerde commando's

```bash
# Maak een nieuwe specificatie
openspec spec create

# Lijst van alle specificaties
openspec spec list

# Bekijk specificatiedetails
openspec spec view <spec-id>

# Werk specificatie bij
openspec spec update <spec-id>
```

### Wijzigingsgerelateerde commando's

```bash
# Maak een nieuwe wijziging
openspec change create

# Lijst van alle wijzigingen
openspec change list

# Bekijk wijzigingsdetails
openspec change view <change-name>

# Voeg taak toe
openspec change add-task <change-name>

# Valideer wijziging
openspec change validate <change-name>

# Archiveer wijziging
openspec change archive <change-name>
```

### Algemene commando's

```bash
# Bekijk versie
openspec --version

# Bekijk help
openspec --help

# Bekijk help voor een specifiek commando
openspec <command> --help
```

### Workflowvoorbeeld

Hier is een volledig voorbeeld van het gebruik van OpenSpec in het HagiCode-project:

```bash
cd /path/to/hagicode-project

```

```bash

```

Definieer in de specificatie:
- Ontwerp van authenticatieflyt
- API interface definitie
- Database tabelstructuur
- Veiligheidsconsideraties

#### Stap 3: Maak een wijziging

Begin de implementatie:

```bash
openspec change create
```

Voer de wijzigingsnaam in: `add-user-authentication`

#### Stap 4: Voeg taken toe

```bash
openspec change add-task add-user-authentication
```

Voeg concrete taken toe:
- Implementeer gebruikersregistratie endpoint
- Implementeer login endpoint
- Implementeer JWT middleware
- Schrijf unit tests

#### Stap 5: Implementatie en validatie

Wanneer de code klaar is, valideer de wijziging:

```bash
openspec change validate add-user-authentication
```

#### Stap 6: Archiveren

Na vellykket validatie, archiveer de wijziging:

```bash
openspec change archive add-user-authentication
```
