## Snelle codebewerking in de browser: VSCode-webintegratie in de praktijk

> Hoe open je, nadat AI klaar is met het analyseren van de code, onmiddellijk een editor in de browser en begin je met het aanbrengen van wijzigingen? Dit artikel deelt onze praktische ervaring met het integreren van de codeserver in het HagiCode-project om een ​​naadloze brug te creëren tussen de AI-assistent en de codebewerkingservaring.

## Achtergrond

In het tijdperk van AI-ondersteund programmeren moeten ontwikkelaars vaak code snel inspecteren en bewerken. De traditionele workflow is eenvoudig: open het project in een desktop-IDE, zoek het bestand, bewerk het en sla het op. Maar in sommige situaties voelt die stroom altijd een beetje vreemd aan.

Scenario één: ontwikkeling op afstand. Wanneer je een AI-assistent zoals HagiCode gebruikt, draait de backend mogelijk op een externe server of in een container, en hebben lokale machines geen directe toegang tot de projectbestanden. Elke keer dat je code moet inspecteren of wijzigen, moet je verbinding maken via SSH of een andere methode, en de ervaring voelt gefragmenteerd aan. Het is alsof je iemand door een dikke ruit wilt ontmoeten: je kunt hem zien, maar je kunt hem niet bereiken.

Scenario twee: snelle voorbeelden. Nadat de AI-assistent de code heeft geanalyseerd, wil de gebruiker mogelijk alleen snel door een bestand bladeren of een kleine wijziging aanbrengen. Het starten van een volledige desktop-IDE voelt zwaar, terwijl een lichtgewicht in-browser-editor beter aansluit bij de behoefte aan een "snelle blik". Wie wil immers een hele gereedschapsketen mobiliseren om slechts één blik te werpen?

Scenario drie: samenwerking tussen apparaten. Bij het werken op verschillende apparaten biedt een browsergebaseerde editor een uniform toegangspunt zonder dat elke machine met een ontwikkelomgeving hoeft te worden geconfigureerd. Dat alleen al scheelt een hoop ellende. Het leven is kort; waarom hetzelfde installatiewerk keer op keer herhalen?

Om deze pijnpunten op te lossen, hebben we VSCode Web geïntegreerd in het HagiCode-project. Hierdoor kunnen de AI-assistent en de codebewerkingservaring naadloos op elkaar aansluiten: nadat AI de code heeft geanalyseerd, kunnen gebruikers onmiddellijk een editor openen en wijzigingen aanbrengen in dezelfde browsersessie, zonder van applicatie te wisselen. Het is het soort ervaring waarbij het, wanneer je het nodig hebt, er gewoon is.

## Over HagiCode

De oplossing die in dit artikel wordt gedeeld, komt voort uit onze praktijkervaring in de [HagiCode](https://hagicode.com) project. HagiCode is een AI-gestuurde codeerassistent die is ontworpen om de ontwikkelingsefficiëntie te verbeteren door middel van natuurlijke taalinteractie. Tijdens de ontwikkeling ontdekten we dat gebruikers vaak snel moeten schakelen tussen AI-analyse en codebewerking, wat ons ertoe aanzette te onderzoeken hoe we de editor rechtstreeks in de browser konden integreren.

Projectopslagplaats: [github.com/HagiCode-org/site](https://github.com/HagiCode-org/site)

## Technische keuze: waarom codeserver?

Uit de vele beschikbare VSCode-weboplossingen hebben we gekozen voor codeserver. Er lagen enkele concrete redenen achter dat besluit.

Volledigheid van functies. code-server is de webversie van VSCode en ondersteunt de meeste desktopfuncties, waaronder het uitbreidingssysteem, intelligente suggesties, foutopsporing en meer. Dat betekent dat gebruikers een bewerkingservaring in de browser kunnen krijgen die heel dicht bij de desktopversie ligt. Wie wil er tenslotte echt concessies doen aan de functionaliteit?

Flexibele inzet. code-server kan als een onafhankelijke service draaien en ondersteunt ook Docker-gebaseerde implementatie, wat goed aansluit bij de architectuur van HagiCode. Onze backend is geschreven in C#, de frontend gebruikt React en de twee communiceren met de codeserverservice via REST API's. Het is als bouwen met blokken: elk stuk heeft zijn plaats.

Veilige authenticatie. code-server bevat een ingebouwde `connection-token` mechanisme om ongeoorloofde toegang te voorkomen. Elke sessie heeft een uniek token zodat alleen geautoriseerde gebruikers de editor kunnen openen. Beveiliging is een van die dingen die je pas ten volle kunt waarderen als je die eenmaal hebt.

## Architectuurontwerp

HagiCode's VSCode-webintegratie maakt gebruik van een front-end/back-end gescheiden architectuur.

### Frontend-servicelaag

De frontend wikkelt interacties met de backend door `vscodeServerService.ts`:

```typescript
// Open project
export async function openProjectInCodeServer(
  id: string,
  currentInterfaceLanguage?: string,
): Promise<VsCodeServerLaunchResponseDto>

// Open vault
export async function openVaultInCodeServer(
  id: string,
  path?: string,
  currentInterfaceLanguage?: string,
): Promise<VsCodeServerLaunchResponseDto>
```

Het verschil tussen deze twee methoden is eenvoudig: `openProjectInCodeServer` opent het hele project, terwijl `openVaultInCodeServer` opent een specifiek pad binnen een kluis. Voor MonoSpecs-projecten met meerdere opslagplaatsen maakt het systeem automatisch een werkruimtebestand. Duidelijke verantwoordelijkheden zijn vaak voldoende als elk onderdeel zijn eigen werk goed doet.

### Backend-servicelaag

De achterkant `VaultAppService.cs` implementeert de kernlogica:

```csharp
public async Task<VsCodeServerLaunchResponseDto> OpenInCodeServerAsync(
    string id,
    string? relativePath = null,
    string? currentInterfaceLanguage = null,
    CancellationToken cancellationToken = default)
{
    // 1. Get settings and check whether the feature is enabled
    var settings = await _vsCodeServerSettingsService.GetResolvedSettingsAsync(cancellationToken);
    if (!settings.Enabled) {
        throw new BusinessException(VsCodeServerErrorCodes.Disabled, "VSCode Server is disabled.");
    }

    // 2. Get vault and resolve the launch directory
    var vault = await RequireVaultAsync(id, cancellationToken);
    var launchDirectory = ResolveLaunchDirectory(vault, relativePath);

    // 3. Ensure code-server is running and get runtime info
    var runtime = await _vsCodeServerManager.EnsureStartedAsync(settings, cancellationToken);

    // 4. Resolve language settings
    var language = _vsCodeServerSettingsService.ResolveLaunchLanguage(
        settings.Language,
        currentInterfaceLanguage);

    // 5. Build launch URL
    return new VsCodeServerLaunchResponseDto {
        LaunchUrl = AppendQueryString(runtime.BaseUrl, new Dictionary<string, string?> {
            ["folder"] = launchDirectory,
            ["tkn"] = runtime.ConnectionToken,
            ["vscode-lang"] = language,
        }),
        ConnectionToken = runtime.ConnectionToken,
        OpenMode = "folder",
        Runtime = VsCodeServerSettingsService.MapRuntime(
            await _vsCodeServerManager.GetRuntimeSnapshotAsync(cancellationToken)),
    };
}
```

Deze methode heeft een heel duidelijke verantwoordelijkheid: instellingen controleren, paden oplossen, de service starten en de URL bouwen. Onder hen zijn de `ResolveLaunchDirectory` methode voert padbeveiligingscontroles uit om padtraversale aanvallen te voorkomen. Code kan een beetje op poëzie lijken als elke regel een doel heeft.

### Automatisch runtimebeheer

De backend beheert het codeserverproces via `VsCodeServerManager`:

- Controleer de processtatus
- Start gestopte services automatisch
- Retourneer runtime-snapshots, zoals poort, proces-ID en starttijd

Dankzij dit ontwerp kan het systeem de levenscyclus van de codeserver automatisch afhandelen, zodat gebruikers de serviceprocessen niet handmatig hoeven te beheren. Het leven is al ingewikkeld genoeg; alles wat geautomatiseerd kan worden, zou dat ook moeten zijn.

### Taalsynchronisatie

HagiCode ondersteunt een meertalige interface en de codeserver moet die instelling volgen. Het systeem ondersteunt drie taalmodi:

- `follow`: volg de huidige interfacetaal
- `zh-CN`: vast op Chinees
- `en-US`: vast ingesteld op Engels

De instelling wordt doorgegeven aan de codeserver via de `vscode-lang` URL-parameter zodat de editortaal consistent blijft met de HagiCode-interface. Taal voelt het beste als het verenigd is.

### MonoSpecs werkruimte met meerdere opslagplaatsen

Voor MonoSpecs-projecten, die meerdere subrepository's binnen één monorepo bevatten, maakt het systeem automatisch een `.code-workspace` bestand:

```csharp
private async Task<string> CreateWorkspaceFileAsync(Project project, Guid projectId)
{
    var folders = await ResolveWorkspaceFoldersAsync(project.Path);
    var workspaceDocument = new {
        folders = folders.Select(path => new { path }).ToArray(),
    };
    // Generate workspace file...
}
```

Dit maakt het mogelijk om meerdere subrepository's in dezelfde codeserverinstantie te bewerken, wat vooral praktisch is voor grote monorepo-projecten. Meerdere opslagplaatsen in één venster kunnen het gevoel geven dat er meerdere verhalen in hetzelfde boek zijn verzameld.

## Frontend-integratie

De HagiCode-frontend maakt gebruik van React + TypeScript en het integreren van de codeserver is niet bijzonder ingewikkeld.

### Snelle actieknop

Voeg een Code Server-knop toe aan de projectkaart:

```tsx
// QuickActionsZone.tsx
<Button
  size="sm"
  variant="default"
  onClick={() => onAction({ type: 'open-code-server' })}
>
  <Globe className="h-3 w-3 mr-1" />
  <span className="text-xs">{t('project.openCodeServer')}</span>
</Button>
```

Deze knop activeert de openactie en roept de backend-API aan om de start-URL te verkrijgen. Eén knop, één actie, direct en eenvoudig.

### Het afhandelen van de open actie

```tsx
const handleAction = async (action: ProjectAction) => {
  if (action.type === 'open-code-server') {
    const response = await openProjectInCodeServer(project.id, i18n.language);
    window.open(response.launchUrl, '_blank', 'noopener,noreferrer');
  }
};
```

Gebruik `window.open` om codeserver in een nieuw tabblad te openen. De `noopener,noreferrer` parameters zorgen voor extra veiligheid. Als het om veiligheid gaat, bestaat er niet zoiets als te voorzichtig zijn.

### Kluisbewerkingsinvoer

Voeg een soortgelijke bewerkingsknop toe aan de kluislijst:

```tsx
const handleEditVault = async (vault: VaultItemDto) => {
  const response = await openVaultInCodeServer(vault.id);
  window.open(response.launchUrl, '_blank', 'noopener,noreferrer');
};
```

Projecten en Vaults gebruiken hetzelfde open mechanisme, waardoor de interactie consistent blijft. Consistentie is bijna net zo belangrijk als de functie zelf.

## URL-constructielogica

Het URL-formaat voor codeserver heeft een paar details die het vermelden waard zijn.

**Mapmodus**:
```
http://{host}:{port}/?folder={path}&tkn={token}&vscode-lang={lang}
```

**Werkruimtemodus**:
```
http://{host}:{port}/?workspace={workspacePath}&tkn={token}&vscode-lang={lang}
```

Hier, `tkn` is het verbindingstoken. Het wordt automatisch gegenereerd elke keer dat de codeserver start, waardoor veilige toegang wordt gegarandeerd. De `vscode-lang` parameter bepaalt de taal van de gebruikersinterface van de editor. Elk van deze parameters speelt een rol.

## Beveiligingsoverwegingen

Bij het implementeren van code-serverintegratie verdient beveiliging speciale aandacht. Als de beveiliging misgaat, merk je dat altijd te laat.

### Verbindingstoken

De `connection-token` wordt willekeurig gegenereerd en mag niet worden blootgesteld. Het kan het beste worden gebruikt onder HTTPS om te voorkomen dat het token wordt onderschept door een man-in-the-middle. Gevoelige informatie is de moeite waard om goed te beschermen.

### Beveiliging van bestandspaden

De backend implementeert controles op het doorlopen van paden:

```csharp
private static string ResolveLaunchDirectory(VaultRegistryEntry vault, string? relativePath)
{
    var vaultRoot = EnsureTrailingSeparator(Path.GetFullPath(vault.PhysicalPath));
    var combinedPath = Path.GetFullPath(Path.Combine(vaultRoot, relativePath ?? "."));
    if (!combinedPath.StartsWith(vaultRoot, StringComparison.OrdinalIgnoreCase))
    {
        throw new BusinessException(VaultRelativePathTraversalCode, "Relative path traversal detected.");
    }
    return combinedPath;
}
```

Deze code zorgt ervoor dat gebruikers er geen gebruik van kunnen maken `../` of vergelijkbare patronen om toegang te krijgen tot bestanden buiten de Vault-map. Grenscontroles zijn altijd beter gedaan dan overgeslagen.

### Toestemmingscontrole

Het codeserverproces moet worden uitgevoerd met de juiste gebruikersmachtigingen, zodat het geen toegang heeft tot gevoelige systeembestanden. Het is het beste om de codeserverservice onder een speciale gebruiker uit te voeren. Toestemmingscontrole is een van de fundamentele zaken die u altijd moet handhaven.

## Prestatie-optimalisatie

code-server verbruikt serverbronnen, dus hier zijn een paar optimalisatiesuggesties:

- Houd het CPU- en geheugengebruik in de gaten en pas indien nodig de resourcelimieten aan
- Bij grote projecten kunnen langere time-outs nodig zijn
- Implementeer het automatisch opschonen van sessietime-outs om bronnen vrij te maken
- Overweeg caching om herhaalde berekeningen te verminderen

HagiCode biedt een API voor runtimestatusmonitoring, en de frontend kan bellen `getVsCodeServerSettings()` om de huidige status op te halen:

```typescript
const { settings, runtime } = await getVsCodeServerSettings();
// runtime.status: 'disabled' | 'stopped' | 'starting' | 'running' | 'unhealthy'
// runtime.baseUrl: "http://localhost:8080"
// runtime.processId: 12345
```

Dankzij dit ontwerp kunnen gebruikers de gezondheidsstatus van de codeserver duidelijk begrijpen en snel problemen lokaliseren als er iets misgaat. Wanneer de status zichtbaar is, voelen mensen zich meer in controle.

## Details van gebruikerservaring

Tijdens de implementatie zijn we een aantal details tegengekomen die de gebruikerservaring merkbaar beïnvloeden en extra aandacht verdienen.

Als u de codeserver voor de eerste keer opent, kan het nodig zijn dat u moet wachten voordat u opstart, en die vertraging kan variëren van een paar seconden tot een halve minuut. Het is een goed idee om een ​​laadstatus in de frontend weer te geven, zodat gebruikers weten dat het systeem nog steeds werkt. Wachten is gemakkelijker als er feedback is.

Browsers kunnen de pop-up blokkeren, dus gebruikers moeten worden gevraagd deze handmatig toe te staan. Bij de eerste keer opstarten geeft HagiCode richtlijnen weer waarin wordt uitgelegd hoe u de benodigde browsermachtigingen kunt verlenen. Gebruikerservaring zit vaak in precies deze kleine details.

Het is ook een goed idee om de runtimestatus, zoals starten, actief of fouten, weer te geven, zodat gebruikers, wanneer zich problemen voordoen, snel kunnen zien of het probleem aan de serverzijde ligt of aan hun eigen werking ligt. Als u weet waar het probleem zit, heeft u in ieder geval een startpunt.

## Configuratievoorbeeld

De configuratie voor codeserver is niet ingewikkeld:

```json
{
  "vscodeServer": {
    "enabled": true,
    "host": "0.0.0.0",
    "port": 8080,
    "language": "follow"
  }
}
```

`enabled` bepaalt of de functie is ingeschakeld, `host` en `port` het luisteradres definiëren, en `language` stelt de taalmodus in. Deze instellingen kunnen via de gebruikersinterface worden gewijzigd en worden onmiddellijk van kracht. Simpele dingen zijn vaak het gemakkelijkst te gebruiken.

HagiCode's VSCode-webintegratie biedt een elegante oplossing: het laat de AI-assistent en de codebewerkingservaring naadloos op elkaar aansluiten.
