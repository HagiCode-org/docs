## Van CLI-aanroepen tot SDK-integratie: best practices voor GitHub Copilot in .NET-projecten

> Het upgradetraject van het aanroepen van de opdrachtregel tot de officiële SDK-integratie was een behoorlijk traject. Vandaag wil ik de valkuilen delen die we tegenkwamen en wat we hebben geleerd tijdens het bouwen van HagiCode.

## Achtergrond

Nadat de GitHub Copilot SDK officieel werd uitgebracht in 2025, zijn we begonnen deze te integreren in onze AI-mogelijkhedenlaag. Voordien gebruikte het project voornamelijk GitHub Copilot door rechtstreeks de Copilot CLI-opdrachtregeltool aan te roepen, maar die aanpak had een aantal voor de hand liggende problemen:

- **Complex procesbeheer**: we moesten de levenscyclus van het CLI-proces, de opstarttime-outs en het opschonen van processen handmatig beheren. Processen kunnen zonder waarschuwing crashen.
- **Onvolledige afhandeling van gebeurtenissen**: De ruwe CLI-aanroep maakt het moeilijk om fijnmazige gebeurtenissen vast te leggen op basis van modelredenering en uitvoering van tools. Het is alsof je alleen het resultaat ziet zonder het denkproces.
- **Moeilijk sessiebeheer**: Er was geen effectief mechanisme voor hergebruik en herstel van sessies, dus elke interactie moest opnieuw beginnen.
- **Compatibiliteitsproblemen**: CLI-argumenten veranderden regelmatig, wat betekende dat we voortdurend de compatibiliteitslogica voor die parameters moesten handhaven.

Deze problemen werden steeds duidelijker in de dagelijkse ontwikkeling, vooral toen we de modelredenering moesten volgen (`thinking`) en de status van de gereedschapsuitvoering in realtime. Op dat moment werd het duidelijk dat we een meer complete integratieaanpak op een lager niveau nodig hadden.

## Over HagiCode

De aanpak die in dit artikel wordt gedeeld, komt voort uit onze praktijkervaring in de [HagiCode](https://hagicode.com) project. HagiCode is een open-source AI-coderingsassistentproject en tijdens de ontwikkeling hadden we een diepgaande integratie nodig met de mogelijkheden van GitHub Copilot, van het voltooien van de basiscode tot complexe multi-turn-gesprekken en het aanroepen van tools. Deze reële vereisten hebben ons ertoe aangezet om van CLI-aanroep over te stappen naar de officiële SDK.

Als deze implementatie u nuttig in de oren klinkt, is de kans groot dat onze technische ervaring u kan helpen. In dat geval is het HagiCode-project zelf wellicht ook de moeite van het bekijken waard. Mogelijk vindt u zelfs meer projectinformatie en links aan het einde van dit artikel.

## Architectuurontwerp

Het project maakt gebruik van een gelaagde architectuur om de beperkingen van CLI-aanroep aan te pakken:

```
┌─────────────────────────────────────────────────────────┐
│  hagicode-core (Orleans Grains + AI Provider Layer)    │
│  - CopilotAIProvider: Converts AIRequest to CopilotOptions │
│  - GitHubCopilotGrain: Orleans distributed execution interface │
└─────────────────────────────────────────────────────────┘
                          ↓
┌─────────────────────────────────────────────────────────┐
│  HagiCode.Libs (Shared Provider Layer)                 │
│  - CopilotProvider: CLI Provider interface implementation │
│  - ICopilotSdkGateway: SDK invocation abstraction      │
│  - GitHubCopilotSdkGateway: SDK session management and event dispatch │
└─────────────────────────────────────────────────────────┘
                          ↓
┌─────────────────────────────────────────────────────────┐
│  GitHub Copilot SDK (Official .NET SDK)                │
│  - CopilotClient: SDK client                           │
│  - CopilotSession: Session management                  │
│  - SessionEvent: Event stream                          │
└─────────────────────────────────────────────────────────┘
```

Dit gelaagde ontwerp brengt verschillende praktische technische voordelen met zich mee:

1. **Scheiding van zorgen**: de kernlogica van het bedrijf is losgekoppeld van de details van de SDK-implementatie.
2. **Testbaarheid**: De `ICopilotSdkGateway` interface maakt het testen van eenheden eenvoudig.
3. **Herbruikbaarheid**: `HagiCode.Libs` Er kan door meerdere projecten naar worden verwezen.
4. **Onderhoudbaarheid**: SDK-upgrades vereisen alleen wijzigingen in de Gateway-laag, terwijl de bovenste lagen onaangeroerd blijven.

## Kernimplementatie

### Authenticatiestroom

Authenticatie is de eerste en belangrijkste stap van SDK-integratie. Als de authenticatie mislukt, doet niets anders er toe. We hebben een flexibele authenticatieconfiguratie ontworpen die meerdere authenticatiebronnen ondersteunt:

```csharp
// CopilotProvider.cs - Authentication source configuration
public class CopilotOptions
{
    public bool UseLoggedInUser { get; set; } = true;
    public string? GitHubToken { get; set; }
    public string? CliUrl { get; set; }
}

// Convert to SDK request
return new CopilotSdkRequest(
    GitHubToken: options.AuthSource == CopilotAuthSource.GitHubToken
        ? options.GitHubToken
        : null,
    UseLoggedInUser: options.AuthSource != CopilotAuthSource.GitHubToken
);
```

De voordelen van dit ontwerp zijn vrij duidelijk:

- Het ondersteunt de ingelogde gebruikersmodus zonder dat er een token nodig is, wat goed past bij desktopscenario's.
- Het ondersteunt de GitHub Token-modus, die geschikt is voor server-side implementaties en gecentraliseerd beheer.
- Het ondersteunt het overschrijven van de Copilot CLI-URL, wat helpt bij de proxyconfiguratie van de onderneming.

In de praktijk heeft dit flexibele authenticatiemodel de configuratie in verschillende implementatiescenario's aanzienlijk vereenvoudigd. De desktopclient kan de eigen Copilot-inlogstatus van elke gebruiker gebruiken, terwijl de server de authenticatie centraal kan beheren via tokens.

### Afhandeling van gebeurtenisstreams

Een van de krachtigste mogelijkheden van de SDK is het volledig vastleggen van gebeurtenisstromen. We hebben een systeem voor het verzenden van gebeurtenissen geïmplementeerd dat allerlei soorten SDK-gebeurtenissen in realtime kan afhandelen:

```csharp
// GitHubCopilotSdkGateway.cs - Core logic for event dispatch
internal static SessionEventDispatchResult DispatchSessionEvent(
    SessionEvent evt, bool sawDelta)
{
    switch (evt)
    {
        case AssistantReasoningEvent reasoningEvent:
            // Capture the model reasoning process
            events.Add(new CopilotSdkStreamEvent(
                CopilotSdkStreamEventType.ReasoningDelta,
                Content: reasoningEvent.Data.Content));
            break;

        case ToolExecutionStartEvent toolStartEvent:
            // Capture the start of a tool invocation
            events.Add(new CopilotSdkStreamEvent(
                CopilotSdkStreamEventType.ToolExecutionStart,
                ToolName: toolStartEvent.Data.ToolName,
                ToolCallId: toolStartEvent.Data.ToolCallId));
            break;

        case ToolExecutionCompleteEvent toolCompleteEvent:
            // Capture tool completion and its result
            events.Add(new CopilotSdkStreamEvent(
                CopilotSdkStreamEventType.ToolExecutionEnd,
                Content: ExtractToolExecutionContent(toolCompleteEvent)));
            break;

        default:
            // Preserve unhandled events as RawEvent
            events.Add(new CopilotSdkStreamEvent(
                CopilotSdkStreamEventType.RawEvent,
                RawEventType: evt.GetType().Name));
            break;
    }
}
```

De waarde van deze implementatie is aanzienlijk:

- **Volledige weergave van het redeneerproces van het model** (`thinking`): gebruikers kunnen zien hoe de AI redeneert, en niet alleen het uiteindelijke antwoord.
- **Realtime volgen van de uitvoeringsstatus van tools**: we weten welke tools actief zijn, wanneer ze klaar zijn en wat ze retourneren.
- **Nul gebeurtenisverlies**: door terug te vallen op `RawEvent`, zorgen wij ervoor dat elke gebeurtenis wordt vastgelegd.

In HagiCode helpen deze fijnmazige gebeurtenissen gebruikers te begrijpen hoe de AI intern werkt, vooral bij het debuggen van complexe taken.

### Afhandeling van CLI-compatibiliteit

Na de migratie van CLI-aanroep naar de SDK hebben we vastgesteld dat sommige bestaande CLI-parameters niet langer van toepassing zijn in de SDK. Om achterwaartse compatibiliteit te behouden, hebben we een parameterfiltersysteem geïmplementeerd:

```csharp
// CopilotCliCompatibility.cs - Argument filtering
private static readonly Dictionary<string, string> RejectedFlags = new()
{
    ["--headless"] = "Unsupported startup argument",
    ["--model"] = "Passed through an SDK-native field",
    ["--prompt"] = "Passed through an SDK-native field",
    ["--interactive"] = "Interaction is managed by the provider",
};

public static CopilotCliArgumentBuildResult BuildCliArgs(CopilotOptions options)
{
    // Filter out unsupported arguments and keep compatible ones
    // Generate diagnostic information
}
```

Dit levert ons verschillende voordelen op:

- Het filtert automatisch incompatibele CLI-argumenten om runtimefouten te voorkomen.
- Het genereert duidelijke diagnostische berichten zodat ontwikkelaars problemen snel kunnen opsporen.
- Het houdt de SDK stabiel en geïsoleerd tegen wijzigingen in CLI-parameters.

Tijdens de upgrade heeft dit compatibiliteitsmechanisme ons geholpen de overgang soepel te laten verlopen. Bestaande configuratiebestanden konden nog steeds worden gebruikt en hadden slechts geleidelijke aanpassingen nodig op basis van de diagnostische uitvoer.

### Runtime-pooling

Het maken van Copilot SDK-sessies is relatief duur, en het te vaak aanmaken en vernietigen van sessies heeft een negatieve invloed op de prestaties. Om dat op te lossen, hebben we een sessiepoolbeheersysteem geïmplementeerd:

```csharp
// CopilotProvider.cs - Session pool management
await using var lease = await _poolCoordinator.AcquireCopilotRuntimeAsync(
    request,
    async ct => await _gateway.CreateRuntimeAsync(sdkRequest, ct),
    cancellationToken);

if (lease.IsWarmLease)
{
    // Reuse an existing session
    yield return CreateSessionReusedMessage();
}

await foreach (var eventData in lease.Entry.Resource.SendPromptAsync(...))
{
    yield return MapEvent(eventData);
}
```

De voordelen van sessiepooling zijn onder meer:

- **Sessiehergebruik**: verzoeken met hetzelfde `sessionId` kan bestaande sessies hergebruiken en de opstartoverhead verminderen.
- **Ondersteuning voor sessieherstel**: na een netwerkonderbreking kan de vorige sessiestatus worden hersteld.
- **Automatisch poolingbeheer**: verlopen sessies worden automatisch opgeschoond om lekken van bronnen te voorkomen.

In HagiCode verbeterde sessiepooling de responsiviteit merkbaar, vooral voor continue gesprekken.

### Orléans Integratie

HagiCode gebruikt Orleans als gedistribueerd raamwerk en we hebben de Copilot SDK geïntegreerd in Orleans Grains:

```csharp
// GitHubCopilotGrain.cs - Distributed execution
public async IAsyncEnumerable<GitHubCopilotResponse> ExecuteCommandStreamAsync(
    string command,
    CancellationToken token = default)
{
    var provider = await aiProviderFactory.GetProviderAsync(AIProviderType.GitHubCopilot);

    await foreach (var chunk in provider.SendMessageAsync(request, null, token))
    {
        // Map to the unified response format
        yield return BuildChunkResponse(chunk, startedAt);
    }
}
```

De voordelen van Orleans-integratie zijn aanzienlijk:

- **Unified AI Provider-abstractie**: het wordt eenvoudig om tussen verschillende AI-providers te schakelen.
- **Multi-tenant isolatie**: Copilot-sessies voor verschillende gebruikers blijven van elkaar geïsoleerd.
- **Persistente sessiestatus**: de sessiestatus kan worden hersteld, zelfs nadat de server opnieuw is opgestart.

Voor scenario's die een groot aantal gelijktijdige verzoeken moeten verwerken, biedt Orleans een sterke schaalbaarheid.

### Gebruiksnotities

Bij gebruik in de echte wereld hebben we een aantal punten samengevat die de moeite waard zijn om op te letten:

**Opstarttime-outconfiguratie**: De eerste keer opstarten van Copilot CLI kan relatief lang duren, daarom raden wij u aan deze instelling in te stellen `StartupTimeout` tot minimaal 30 seconden. Als dit de eerste login is, kan het nog langer duren.

**Toestemmingsbeheer**: Vermijd het gebruik in productieomgevingen `AllowAllTools: true`. Gebruik de `AllowedTools` toelatingslijst om te bepalen welke tools beschikbaar zijn, en gebruik de `DeniedTools` denylist om gevaarlijke operaties te blokkeren. Dit voorkomt effectief dat de AI risicovolle opdrachten uitvoert.

**Sessiebeheer**: Verzoeken met hetzelfde `sessionId` sessies automatisch opnieuw gebruiken. De sessiestatus blijft behouden `ProviderSessionId`. Annulering vindt plaats via `CancellationTokenSource`.

**Diagnostische uitvoer**: Incompatibele CLI-argumenten genereren berichten van het type `diagnostic`. Raw SDK-gebeurtenissen worden bewaard als `event.raw`. Foutberichten omvatten categorieën zoals opstarttime-out en incompatibiliteit van argumenten om het oplossen van problemen eenvoudiger te maken.

### Beste praktijken

Op basis van onze praktijkervaring volgen hier enkele best practices:

**1. Gebruik een toelatingslijst voor tools**

```csharp
var request = new AIRequest
{
    Prompt = "Analyze this file",
    AllowedTools = new[] { "Read", "Grep", "Bash(git:*)" }
};
```

Het expliciet specificeren van de toegestane tools via een toelatingslijst helpt onverwachte AI-acties te voorkomen. Dit is vooral belangrijk voor tools met schrijfrechten, zoals `Edit`, waar extra voorzichtig mee moet worden omgegaan.

**2. Stel redelijke time-outs in**

```csharp
options.Timeout = 3600;  // 1 hour
options.StartupTimeout = 60;  // 1 minute
```

Stel de juiste time-outwaarden in op basis van de complexiteit van de taak. Als de waarde te kort is, kunnen taken worden onderbroken. Als het te lang duurt, kunnen er middelen worden verspild door te wachten op niet-reagerende verzoeken.

**3. Hergebruik van sessie inschakelen**

```csharp
options.SessionId = "my-session-123";
```

Hetzelfde gebruiken `sessionId` voor gerelateerde taken kunt u de context van eerdere sessies hergebruiken en de reactiesnelheid verbeteren.

**4. Streamingreacties verwerken**

```csharp
await foreach (var chunk in provider.StreamAsync(request))
{
    switch (chunk.Type)
    {
        case StreamingChunkType.ThinkingDelta:
            // Handle the reasoning process
            break;
        case StreamingChunkType.ToolCallDelta:
            // Handle tool invocation
            break;
        case StreamingChunkType.ContentDelta:
            // Handle text output
            break;
    }
}
```

Met streamingreacties kunt u de voortgang van de AI-verwerking in realtime weergeven, wat de gebruikerservaring verbetert. Dit is vooral waardevol bij langdurige taken.

**5. Foutafhandeling en nieuwe pogingen**

```csharp
try
{
    await foreach (var chunk in provider.StreamAsync(request))
    {
        // Handle the response
    }
}
catch (CopilotSessionException ex)
{
    // Handle session exceptions
    logger.LogError(ex, "Copilot session failed");
    // Decide whether to retry based on the exception type
}
```

Een juiste foutafhandeling en logica voor opnieuw proberen verbeteren de algehele systeemstabiliteit.

## Conclusie

Het upgraden van CLI-aanroep naar SDK-integratie leverde substantiële waarde op voor het HagiCode-project:

- **Verbeterde stabiliteit**: de SDK biedt een stabielere interface die niet wordt beïnvloed door wijzigingen in de CLI-versie.
- **Volledige functionaliteit**: het legt de volledige gebeurtenisstroom vast, inclusief de redenering en de uitvoeringsstatus van de tool.
- **Hogere ontwikkelingsefficiëntie**: de typeveilige SDK-interface maakt de ontwikkeling efficiënter en vermindert runtime-fouten.
- **Betere gebruikerservaring**: realtime feedback over gebeurtenissen geeft gebruikers een beter inzicht in wat de AI doet.

Deze upgrade was niet alleen een vervanging van de technische implementatie. Het was ook een architecturale optimalisatie van de gehele AI-capaciteitslaag. Door gelaagd ontwerp en abstractie-interfaces hebben we een betere onderhoudbaarheid en uitbreidbaarheid bereikt.

Als u overweegt GitHub Copilot in uw eigen .NET-project te integreren, hoop ik dat deze praktische lessen u helpen onnodige omwegen te vermijden. De officiële SDK is inderdaad stabieler en completer dan CLI-aanroep, en het is de moeite waard om deze te begrijpen en over te nemen.
