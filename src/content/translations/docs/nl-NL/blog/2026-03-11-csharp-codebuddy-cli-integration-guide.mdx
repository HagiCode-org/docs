## Praktische gids voor het integreren van CodeBuddy CLI in een C#-backend

> In dit artikel wordt een volledige aanpak beschreven voor het integreren van CodeBuddy CLI in een C#-backend-project, zodat u de mogelijkheden van AI-coderingsassistenten van begin tot eind kunt leveren.

:::note[Tijdcontext]
Dit artikel bespreekt het CodeBuddy-integratieontwerp en de providerstructuur zoals deze bestonden op het moment van publicatie. Raadpleeg voor het huidige permanent ondersteunde bereik [AI Agent CLI-installatie](/nl-NL/related-software-installation/ai-agent-cli/) en de [Installatiehandleiding](/nl-NL/installation); `IFlowCli` wordt nu alleen behouden als historische compatibiliteitswaarde en maakt geen deel uit van het huidige actieve ondersteuningsbereik.
:::

## Achtergrond

Bij de moderne ontwikkeling van AI-coderingsassistenten kan één enkele AI-provider vaak niet voldoen aan complexe en veranderende ontwikkelingsscenario’s. HagiCode moet als multifunctionele AI-coderingsassistent meerdere AI-providers ondersteunen om een ​​betere gebruikerservaring te bieden. Gebruikers moeten voldoende keuzevrijheid hebben. Begin 2026 stond het project voor een belangrijke beslissing: hoe de integratiemogelijkheden van CodeBuddy ACP (Agent Communication Protocol) in de C#-backend konden worden hersteld.

Het project had eerder CodeBuddy-integratie geïmplementeerd, maar de gerelateerde code werd tijdens een refactor verwijderd. Er valt daar niet veel te klagen; tijdens iteratieve ontwikkeling blijft er altijd iets achter. Het doel van deze technische oplossing was om die mogelijkheid volledig te herstellen en de architectuur te verbeteren, zodat deze robuuster en onderhoudbaarder zou zijn.

Als u ook overweegt om meerdere AI-codeerassistenten aan uw eigen project te koppelen, kan de onderstaande aanpak u wellicht op ideeën brengen. Het weerspiegelt de lessen die we hebben samengevat nadat we in veel valkuilen waren gestapt, en misschien kan het je helpen een paar omwegen te vermijden.

De aanpak die in dit artikel wordt gedeeld, komt voort uit onze praktijkervaring in de [HagiCode](https://hagicode.com) project. ## Architectuurontwerp

### Overzicht van gelaagde architectuur

De integratie tussen C# en CodeBuddy maakt gebruik van een duidelijke gelaagde architectuur. Dit ontwerp maakt verantwoordelijkheden expliciet en maakt langdurig onderhoud veel eenvoudiger:

```
┌─────────────────────────────────────────────┐
│           Provider Contract Layer           │
│   AIProviderType enum + extension methods   │
├─────────────────────────────────────────────┤
│           Provider Factory Layer            │
│   AIProviderFactory dependency injection factory │
├─────────────────────────────────────────────┤
│           Provider Implementation Layer     │
│   CodebuddyCliProvider concrete implementation │
├─────────────────────────────────────────────┤
│           ACP Infrastructure Layer          │
│  ACPSessionManager / StdioAcpTransport      │
│  AcpRpcClient / AcpAgentClient              │
└─────────────────────────────────────────────┘
```

Wat zijn de voordelen van deze gelaagdheid? Simpel gezegd: elke laag blijft uit de buurt van de anderen. Als we later het communicatiemechanisme willen veranderen, bijvoorbeeld van `stdio` voor WebSocket hoeven we alleen de onderste laag aan te passen en blijft de bedrijfslogica daarboven onaangeroerd. Niemand wil dat er een communicatieverandering door de hele codebase gaat.

### Uitsplitsing van de kerncomponenten

De **Provider-contractlaag** vormt de basis van de gehele architectuur. Wij definiëren de `AIProviderType` enum, waar `CodebuddyCli = 3` wordt gebruikt als de enumwaarde, en implementeert bidirectionele mapping tussen strings en enums via uitbreidingsmethoden. Hierdoor kunnen strings in configuratiebestanden gemakkelijk worden geconverteerd naar enums, en kunnen enums weer worden geconverteerd naar strings voor het debuggen van de uitvoer.

De **Provider-fabriekslaag** is verantwoordelijk voor het maken van de overeenkomstige Provider-instantie op basis van de configuratie. Het maakt gebruik van .NET-afhankelijkheidsinjectie samen met `ActivatorUtilities.CreateInstance` voor dynamische creatie. Het voordeel van het fabriekspatroon is dat u bij het toevoegen van een nieuwe Provider alleen de aanmaaklogica hoeft toe te voegen in plaats van de bestaande code te wijzigen.

De **Provider-implementatielaag** is waar het daadwerkelijke werk gebeurt. `CodebuddyCliProvider` implementeert de `IAIProvider` interface en biedt twee aanroepmodi: `ExecuteAsync` voor niet-streaming oproepen en `StreamAsync` voor het streamen van oproepen.

De **ACP-infrastructuurlaag** vormt de onderliggende communicatiebasis. Deze laag verwerkt alle protocoldetails, inclusief procesbeheer, berichtserialisatie en het parseren van antwoorden. Het is de basis die alles erboven stabiel houdt.

## Communicatiemechanisme

### Stdio-transportmodus

CodeBuddy gebruikt **Stdio (standaard invoer/uitvoer)** om met externe processen te communiceren. De opstartopdracht is eenvoudig:

```bash
codebuddy --acp
```

Daarna worden JSON-RPC-berichten uitgewisseld via standaard invoer en uitvoer. Deze aanpak heeft verschillende voordelen:

1. **Snelle opstart**: lokale procescommunicatie vermijdt netwerklatentie
2. **Eenvoudige configuratie**: u hoeft alleen het uitvoerbare pad op te geven
3. **Omgevingsisolatie**: elke sessie verloopt in een onafhankelijk proces, zodat ze elkaar niet beïnvloeden

Injectie van omgevingsvariabelen wordt ondersteund tijdens communicatie. Veel voorkomende voorbeelden zijn:

- `CODEBUDDY_API_KEY`: API-sleutelverificatie
- `CODEBUDDY_INTERNET_ENVIRONMENT`: configuratie van de netwerkomgeving

Net als bij communicatie tussen mensen helpt het om eerst een geschikt kanaal te kiezen.

### Berichtenprotocol

ACP is gebaseerd op JSON-RPC 2.0. Het berichtformaat ziet er ongeveer als volgt uit:

```json
// Request message
{
  "jsonrpc": "2.0",
  "id": 1,
  "method": "agent/prompt",
  "params": {
    "prompt": "Help me write a sorting algorithm",
    "sessionId": "session-123"
  }
}

// Response message
{
  "jsonrpc": "2.0",
  "id": 1,
  "result": {
    "content": "Here is the AI response..."
  }
}
```

In de echte implementatie kapselen we al deze protocoldetails in, zodat de bovenste bedrijfslaag zich alleen maar hoeft te bekommeren om de prompt en het antwoord.

## Kernimplementatie

### 1. Herstel het providercontract

Herstel eerst het CodeBuddy-type in het enum-bestand:

private static readonly Dictionary<string, AIProviderType> _typeMap = new(
    StringComparer.OrdinalIgnoreCase)
{
    ["CodebuddyCli"] = AIProviderType.CodebuddyCli,

    ["codebuddy"] = AIProviderType.CodebuddyCli,

### 2. Integreer de Provider Factory

Voeg een CodeBuddy-creatievertakking toe aan de fabrieksklasse:

```csharp
// AIProviderFactory.cs
private IAIProvider? CreateProvider(AIProviderType providerType, ProviderConfiguration config)
{
    return providerType switch
    {
        AIProviderType.CodebuddyCli =>
            ActivatorUtilities.CreateInstance<CodebuddyCliProvider>(
                _serviceProvider,
                Options.Create(config)),
        // ... Other providers
        _ => throw new NotSupportedException($"Provider {providerType} not supported")
    };
}
```

Hierbij wordt gebruik gemaakt van afhankelijkheidsinjectie `ActivatorUtilities`, dat automatisch de injectie van constructorparameters afhandelt en erg handig is.

### 3. Voltooi de implementatie van de provider

Hieronder vindt u de kernimplementatie van `CodebuddyCliProvider`, die zowel streaming- als niet-streaming-aanroepmodi omvat:

```csharp
public class CodebuddyCliProvider : IAIProvider
{
    private readonly ILogger<CodebuddyCliProvider> _logger;
    private readonly IACPSessionManager _sessionManager;
    private readonly ProviderConfiguration _config;

    public string Name => "CodebuddyCli";
    public bool SupportsStreaming => true;
    public ProviderCapabilities Capabilities { get; }

    public CodebuddyCliProvider(
        ILogger<CodebuddyCliProvider> logger,
        IACPSessionManager sessionManager,
        IOptions<ProviderConfiguration> config)
    {
        _logger = logger;
        _sessionManager = sessionManager;
        _config = config.Value;

        // Define the capabilities of the current Provider
        Capabilities = new ProviderCapabilities
        {
            SupportsStreaming = true,
            SupportsTools = true,
            SupportsSystemMessages = true,
            SupportsArtifacts = false,
            MaxTokens = 8192
        };
    }

    // Non-streaming call: return all results together after completion
    public async Task<AIResponse> ExecuteAsync(
        AIRequest request,
        CancellationToken cancellationToken = default)
    {
        // Create an independent session for the request
        var session = await _sessionManager.CreateSessionAsync(
            "CodebuddyCli",
            request.WorkingDirectory,
            cancellationToken,
            request.SessionId);

        try
        {
            var fullPrompt = BuildPrompt(request);
            await session.SendPromptAsync(fullPrompt, cancellationToken);

            var responseBuilder = new StringBuilder();
            var toolCalls = new List<AIToolCall>();

            // Collect all response chunks
            await foreach (var chunk in StreamFromSession(session, cancellationToken))
            {
                if (!string.IsNullOrEmpty(chunk.Content))
                {
                    responseBuilder.Append(chunk.Content);
                }
                // Handle tool calls...
            }

            return new AIResponse
            {
                Content = AIResultContentSanitizer.SanitizeResultContent(
                    responseBuilder.ToString()),
                ToolCalls = toolCalls,
                Provider = Name,
                Model = string.Empty
            };
        }
        finally
        {
            // Release session resources
            await session.DisposeAsync();
        }
    }

    // Streaming call: return response chunks in real time
    public async IAsyncEnumerable<AIStreamingChunk> StreamAsync(
        AIRequest request,
        [EnumeratorCancellation] CancellationToken cancellationToken = default)
    {
        var session = await _sessionManager.CreateSessionAsync(
            "CodebuddyCli",
            request.WorkingDirectory,
            cancellationToken);

        try
        {
            var fullPrompt = BuildPrompt(request);
            await session.SendPromptAsync(fullPrompt, cancellationToken);

            await foreach (var chunk in StreamFromSession(session, cancellationToken))
            {
                yield return chunk;
            }
        }
        finally
        {
            await session.DisposeAsync();
        }
    }

    private async IAsyncEnumerable<AIStreamingChunk> StreamFromSession(
        IACPSession session,
        [EnumeratorCancellation] CancellationToken cancellationToken)
    {
        // Iterate through all updates in the session
        await foreach (var notification in session.ReceiveUpdatesAsync(cancellationToken))
        {
            switch (notification.Update)
            {
                case AgentMessageChunkSessionUpdate agentMessage:
                    // Handle text content chunks
                    if (agentMessage.Content is AcpImp.TextContentBlock textContent)
                    {
                        yield return new AIStreamingChunk
                        {
                            Content = textContent.Text,
                            Type = StreamingChunkType.ContentDelta,
                            IsComplete = false
                        };
                    }
                    break;

                case ToolCallSessionUpdate toolCall:
                    // Handle tool calls
                    yield return new AIStreamingChunk
                    {
                        Content = string.Empty,
                        Type = StreamingChunkType.ToolCallDelta,
                        ToolCallDelta = new AIToolCallDelta
                        {
                            Id = toolCall.ToolCallId,
                            Name = toolCall.Kind.ToString(),
                            Arguments = toolCall.RawInput?.ToString()
                        }
                    };
                    break;

                case AcpImp.PromptCompletedSessionUpdate:
                    // Response complete
                    yield break;
            }
        }
    }

    // Build the full prompt
    private string BuildPrompt(AIRequest request, string? embeddedCommandPrompt = null)
    {
        var sb = new StringBuilder();

        // Embedded command prompt, if present
        if (!string.IsNullOrEmpty(embeddedCommandPrompt))
        {
            sb.AppendLine(embeddedCommandPrompt);
            sb.AppendLine();
        }

        // System message
        if (!string.IsNullOrEmpty(request.SystemMessage))
        {
            sb.AppendLine(request.SystemMessage);
            sb.AppendLine();
        }

        // User prompt
        sb.Append(request.Prompt);
        return sb.ToString();
    }
}
```

Er zijn verschillende belangrijke punten in deze code:

1. **Sessiebeheer**: elk verzoek creëert een onafhankelijke sessie en geeft bronnen vrij nadat het verzoek is voltooid. Dit is een les die we met vallen en opstaan ​​hebben geleerd. Als het hergebruik van sessies niet goed wordt aangepakt, ontstaat er gemakkelijk staatsvervuiling.

2. **Streamingverwerking**: `IAsyncEnumerable` zorgt ervoor dat het antwoord kan worden geretourneerd terwijl het nog wordt gegenereerd, in plaats van te wachten tot alle inhoud is voltooid. Dit is vooral belangrijk voor scenario's met lange tekst en verbetert de gebruikerservaring aanzienlijk.

3. **Tooloproepen**: CodeBuddy ondersteunt tooloproepen (Function Calling), afgehandeld via `ToolCallSessionUpdate`. Deze mogelijkheid is van cruciaal belang voor complexe codebewerkingstaken.

4. **Inhoudsfiltering**: `AIResultContentSanitizer` wordt gebruikt om de inhoud van Think-blokken te filteren en de uitvoer schoon te houden.

### 4. Configuratie van afhankelijkheidsinjectie

Voeg de gerelateerde services toe tijdens de moduleregistratie:

```csharp
// PCodeClaudeHelperModule.cs
public void ConfigureModule(IServiceCollection context)
{
    // Register Provider
    context.Services.AddTransient<CodebuddyCliProvider>();

    // Register ACP infrastructure
    context.Services.AddSingleton<IACPSessionManager, ACPSessionManager>();
    context.Services.AddSingleton<IAcpPlatformConfigurationResolver, AcpPlatformConfigurationResolver>();
    context.Services.AddSingleton<IAIRequestToAcpMapper, AIRequestToAcpMapper>();
    context.Services.AddSingleton<IAcpToAIResponseMapper, AcpToAIResponseMapper>();
}
```

## Configuratievoorbeeld

### Configuratiebestand

Voeg CodeBuddy-gerelateerde configuratie toe aan `appsettings.json`:

```yaml
AI:
  # Default Provider to use
  DefaultProvider: "CodebuddyCli"

  # Provider configuration
  Providers:
    CodebuddyCli:
      Type: "CodebuddyCli"
      WorkingDirectory: "C:/projects/my-app"
      ExecutablePath: "C:/tools/codebuddy.cmd"

  # Platform-specific configuration
  PlatformConfigurations:
    CodebuddyCli:
      ExecutablePath: "C:/tools/codebuddy.cmd"
      Arguments: "--acp"
      StartupTimeoutMs: 5000
      EnvironmentVariables:
        CODEBUDDY_API_KEY: "${CODEBUDDY_API_KEY}"
        CODEBUDDY_INTERNET_ENVIRONMENT: "production"
```

### Configuratiemodel

De bijbehorende configuratiemodeldefinitie:

```csharp
public class CodebuddyPlatformConfiguration : IAcpPlatformConfiguration
{

    public AcpTransportType TransportType => AcpTransportType.Stdio;

    public string ExecutablePath { get; set; } = "codebuddy";
    public string Arguments { get; set; } = "--acp";
    public int StartupTimeoutMs { get; set; } = 5000;

    public Dictionary<string, string?>? EnvironmentVariables { get; set; }
}
```

## Praktische lessen geleerd

### Valkuillogboek

We zijn tijdens de implementatie verschillende typische valkuilen tegengekomen, en als u deze hier deelt, kunnen anderen dezelfde omwegen vermijden:

1. **Probleem met sessielekken**: in eerste instantie werden sessies niet correct vrijgegeven, waardoor de procesbronnen uitgeput raakten. Gebruiken was de oplossing `try-finally` om ervoor te zorgen dat voor elk verzoek middelen worden vrijgegeven.

2. **Doorgeven van omgevingsvariabelen**: Windows en Linux gebruiken verschillende syntaxis van omgevingsvariabelen, dus we hebben dit later gestandaardiseerd `Dictionary<string, string?>` om dit af te handelen.

3. **Time-outconfiguratie**: CLI-opstarten kost tijd, daarom hebben we een opstarttime-out van 5 seconden ingesteld om snelle aanvraagfouten te voorkomen.

4. **Coderingsproblemen**: op Windows kan de standaardcodering verminkte Chinese tekst veroorzaken, dus UTF-8-codering wordt expliciet opgegeven bij het starten van het proces.

### Prestatie-optimalisatie

1. **Sessiepool**: voor frequente korte verzoeken kunt u overwegen een sessiepool te implementeren om processen te hergebruiken
2. **Verbindingscache**: de fabrieksklasse ondersteunt al het cachen van Provider-instanties
3. **Async eerst**: gebruik overal asynchrone programmering om te voorkomen dat threads worden geblokkeerd

Prestaties zijn altijd de moeite waard om te optimaliseren. Hoe langer gebruikers wachten, hoe slechter de ervaring wordt.
