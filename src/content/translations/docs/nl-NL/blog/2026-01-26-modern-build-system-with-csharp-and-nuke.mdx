## Vaarwel Script Hell: waarom we C# kozen voor een modern bouwsysteem

> Een diepe duik in hoe het HagiCode-project Nuke gebruikt om een typeveilige, platformonafhankelijke en zeer uitbreidbare geautomatiseerde bouwpijplijn te bouwen, waarbij de onderhoudspijnpunten van traditionele bouwscripts volledig worden aangepakt.

## Achtergrond

Gedurende het lange traject van softwareontwikkeling roept het woord 'build' zowel liefde als frustratie op. Liefs, want met één klik verandert code in een product, en dat is een van de meest bevredigende momenten bij het programmeren. Frustratie, omdat het onderhouden van een stapel rommelige bouwscripts een nachtmerrie kan zijn.

In veel projecten zijn we gewend scripts in Python te schrijven of te vertrouwen op XML-configuratiebestanden. Stel je de angst eens voor om geregeerd te worden door `<property>` blokken. Maar naarmate de complexiteit van projecten toeneemt, vooral in een project als HagiCode dat frontend- en backend-werk, meerdere platforms en meerdere talen omvat, beginnen traditionele bouwbenaderingen hun grenzen te tonen. Verspreide scriptlogica, geen typecontrole, zwakke IDE-ondersteuning... elk van deze problemen wordt een kleine valkuil waar het team van tijd tot tijd over struikelt.

Om deze pijnpunten op te lossen introduceerde het HagiCode-project **Nuke**, een modern bouwsysteem gebaseerd op C#. Het is niet alleen een hulpmiddel. Het is een andere manier van denken over bouwworkflows. Vandaag zullen we bekijken waarom we ervoor hebben gekozen en hoe het onze ontwikkelingservaring dramatisch heeft verbeterd.

## Over HagiCode

> Een korte introductie van wat we bouwen

We bouwen **HagiCode**, een door AI aangedreven codeerassistent die de ontwikkelervaring slimmer, handiger en leuker maakt.

## Kernanalyse: waarom kernwapens?

Je vraagt je misschien af: "Er zijn al zoveel build-systemen, zoals Make-, Gradle- of zelfs gewone Shell-scripts. Waarom zou je je uiterste best doen om een op C# gebaseerd systeem te gebruiken?"

Dat is een terechte vraag. De kernaantrekkingskracht van Nuke is dat het de programmeertaalfuncties die we al goed kennen, in de wereld van build-scripts brengt.

### 1. Het modulair maken van bouwworkflows: de kunst van het stellen van doelen

Nuke heeft een heel duidelijke ontwerpfilosofie: **alles is een doelwit**.

In traditionele scripts is het gemakkelijk om honderden regels lineaire uitvoeringslogica in de war te krijgen. Bij Nuke splitsen we het bouwproces op in een onafhankelijk proces `Target`s. Elk doelwit doet precies één ding, bijvoorbeeld:

- `Clean`: maak de uitvoermap schoon
- `Restore`: herstel afhankelijkheidspakketten
- `Compile`: compileer de code
- `Test`: unit-tests uitvoeren

Dit ontwerp sluit zeer goed aan bij het principe van één verantwoordelijkheid. Net als bouwstenen kunnen deze doelstellingen vrij worden gecombineerd. Belangrijker nog is dat Nuke ons in staat stelt de afhankelijkheden tussen doelen te definiëren. Bijvoorbeeld als je wilt hardlopen `Test`, controleert het systeem automatisch of `Compile` is al uitgevoerd. Als je wilt `Compile`, dan `Restore` komt uiteraard op de eerste plaats.

Deze afhankelijkheidsgrafiek maakt de logica veel duidelijker en verbetert de efficiëntie van de uitvoering aanzienlijk, omdat Nuke automatisch het optimale uitvoeringspad kan analyseren.

### 2. Typeveiligheid: afscheid van de nachtmerrie van typefouten

Iedereen die build-scripts in Python heeft geschreven, is waarschijnlijk deze lastige situatie tegengekomen: het script loopt vijf minuten en mislukt dan omdat `Confi.guration` verkeerd gespeld is, of omdat er een tekenreeks is doorgegeven waar een numerieke parameter werd verwacht.

Het grootste voordeel van het schrijven van build-scripts in C# is **typeveiligheid**. Dat betekent:

- **Compile-time controles**: de IDE kan u vertellen wat er mis is terwijl u typt, in plaats van u te laten wachten tot runtime.
- **Veilige refactoring**: als u een variabele of methode wilt hernoemen, kan de IDE dit in één stap afhandelen zonder u te dwingen een riskant globaal zoeken en vervangen uit te voeren.
- **Intelligente voltooiing**: krachtige IntelliSense kan de code automatisch voor u voltooien, zodat u niet door de documentatie hoeft te bladeren om obscure API's te onthouden.

### 3. Ondersteuning voor meerdere platforms: één consistente bouwervaring

Vroeger schreven we `.bat` op Windows en `.sh` op Linux, en daarna misschien een Python-script toegevoegd om de kloof te overbruggen. Waar .NET Core, of .NET 5+ vandaag de dag, ook kan draaien, kan Nuke ook draaien.

Dat betekent dat teamleden Windows, Linux of macOS kunnen gebruiken, en of ze nu de voorkeur geven aan Visual Studio, VS Code of Rider, ze voeren allemaal exact dezelfde logica uit. Dit vermindert de problemen die worden veroorzaakt door omgevingsverschillen aanzienlijk, vooral het klassieke probleem "het werkt op mijn machine".

### 4. Parameters en configuratiebeheer

Nuke biedt een zeer elegant mechanisme voor het parseren van parameters. U hoeft niet handmatig te parseren `string[] args`. U definieert eenvoudigweg een eigenschap en annoteert deze met `[Parameter]`, en Nuke verwerkt automatisch de toewijzing van opdrachtregelargumenten en configuratiebestanden.

We kunnen de build-configuratie bijvoorbeeld heel eenvoudig definiëren:

```csharp
[Parameter("Configuration to build - Default is 'Debug'")]
readonly Configuration BuildConfiguration = IsLocalBuild ? Configuration.Debug : Configuration.Release;

Target Compile => _ => _
    .DependsOn(Restore)
    .Executes(() =>
    {
        // Use BuildConfiguration here; it is type-safe
        DotNetBuild(s => s
            .SetConfiguration(BuildConfiguration)
            .SetProjectFile(SolutionFile));
    });
```

Deze stijl is zowel intuïtief als bestand tegen fouten.

## Praktische gids: hoe u dit in een project in de praktijk kunt brengen

Met alleen theorie komen we nergens. Laten we eens kijken hoe we deze aanpak daadwerkelijk hebben toegepast in het HagiCode-project.

### 1. Plan de projectstructuur

We wilden niet dat buildscripts de hoofdmap van het project rommelig zouden maken, en we wilden ook geen directorystructuur die zo diep was dat deze op bepaalde Java-projecten leek. Daarom hebben we alle Nuke-gerelateerde buildbestanden in één enkele map geplaatst `nukeBuild/` map.

De voordelen zijn eenvoudig:

- De projectroot blijft schoon.
- Bouwlogica blijft samenhangend en gemakkelijker te beheren.
- Nieuwe teamleden kunnen meteen zien: "Dit is waar de bouwgerelateerde logica leeft."

### 2. Ontwerp een duidelijke doelafhankelijkheidsketen

Bij het ontwerpen van Targets hebben we één principe gevolgd: **atomiciteit plus afhankelijkheidsstroom**.

Elk doel moet klein genoeg zijn om precies één ding te doen. Bijvoorbeeld `Clean` mag alleen bestanden verwijderen en mag de verpakking niet stilletjes aan de zijkant hanteren.

Een aanbevolen afhankelijkheidsstroom ziet er ongeveer als volgt uit:

`Clean` -> `Restore` -> `Compile` -> `Test` -> `Pack`

Uiteraard is dit niet absoluut. Als u bijvoorbeeld alleen tests wilt uitvoeren en niets wilt verpakken, kunt u met Nuke uitvoeren `nuke Test` direct, en het zal automatisch de vereiste afhandeling afhandelen `Restore` en `Compile` eerst stappen.

### 3. Robuuste foutafhandeling en registratie

Wat is het meest frustrerende aan buildscripts? Onduidelijke foutmeldingen. Een mislukte build die alleen "Error: 1" registreert, is genoeg om iedereen gek te maken.

Omdat we in Nuke rechtstreeks C#-uitzonderingsafhandeling kunnen gebruiken, kunnen we fouten met veel grotere precisie vastleggen en rapporteren.

```csharp
Target Publish => _ => _
    .DependsOn(Test)
    .Executes(() =>
    {
        try 
        {
            // Try publishing to NuGet
            DotNetNuGetPush(s => s
                .SetTargetPath(ArtifactPath)
                .SetSource("https://api.nuget.org/v3/index.json")
                .SetApiKey(ApiKey));
        }
        catch (Exception ex)
        {
            Log.Error($"Publishing failed. Team, please check whether the key is correct: {ex.Message}");
            throw; // Make sure the build process exits with a non-zero exit code
        }
    });
```

### 4. Gebruik tests om de kwaliteit te beschermen

Build-scripts zijn ook code, wat betekent dat ze ook tests nodig hebben. Met Nuke kunnen we tests schrijven voor build-workflows, zodat we bij het wijzigen van de build-logica niet per ongeluk bestaande releaseprocessen verbreken. Dit is vooral belangrijk in pijplijnen voor continue integratie.

## Samenvatting

Door de introductie van Nuke werd het bouwproces van HagiCode soepeler dan ooit tevoren. Dit was niet alleen een vervanging van gereedschap. Het was een upgrade van de technische mentaliteit.

**Wat hebben we gewonnen?**

- **Onderhoudbaarheid**: configuratie als code, duidelijkere logica en een sneller onboardingpad voor nieuwe bijdragers.
- **Stabiliteit**: krachtig typen elimineert meer dan 90% van de fouten op laag niveau.
- **Consistentie**: een uniforme platformonafhankelijke ervaring die omgevingsafwijkingen wegneemt.

Als het schrijven van build-scripts vroeger voelde als 'je een weg banen door het donker', dan voelt het gebruik van Nuke als 's nachts lopen met de lichten aan'. Als je het beu bent om moeilijk te debuggen scripts te onderhouden, overweeg dan om je buildlogica naar de wereld van C# te verplaatsen. Misschien ontdek je dat bouwwerken ook elegant kunnen zijn.
