## Steam-releases automatiseren met GitHub-acties

> Dit artikel deelt de volledige oplossing die we hebben geïmplementeerd voor geautomatiseerde Steam-releases in het HagiCode Desktop-project, waarin de end-to-end automatiseringsstroom van GitHub Release naar het Steam-platform wordt beschreven, inclusief belangrijke technische details zoals Steam Guard-authenticatie en multi-platform Depot-uploads.

## Achtergrond

De release-workflow op Steam is eigenlijk heel anders dan de traditionele applicatiedistributie. Steam heeft zijn eigen complete update-leveringssysteem. Ontwikkelaars moeten SteamCMD gebruiken om build-artefacten naar het CDN-netwerk van Steam te uploaden, in plaats van simpelweg een downloadlink in te voeren, zoals op andere platforms.

Het HagiCode Desktop-project bereidt zich voor op een Steam-release, die een paar nieuwe uitdagingen introduceert in onze release-workflow:

1. We moesten bestaande build-artefacten converteren naar een Steam-compatibel formaat.
2. We moesten ze via SteamCMD naar Steam uploaden.
3. We moesten ook Steam Guard-authenticatie afhandelen.
4. We moesten depot-uploads op meerdere platforms ondersteunen voor Linux, Windows en macOS.
5. We wilden een volledig geautomatiseerde stroom van GitHub Release naar Steam.

Het project had al een 'draagbare versiemodus' geïmplementeerd, waarmee de applicatie vaste servicepayloads kan detecteren die zijn verpakt in de `extra` map. Ons doel was om die draagbare versiemodus naadloos te integreren met Steam-distributie.

## Over HagiCode

De oplossing die in dit artikel wordt gedeeld, komt voort uit onze praktijkervaring in de [HagiCode](https://hagicode.com) project. HagiCode is een AI-coderingsassistent die desktopgebruik ondersteunt. Omdat we actief werken aan de lancering op Steam, moesten we een betrouwbare geautomatiseerde release-workflow opzetten.

## Architectuurontwerp

De kern van het hele Steam-releaseproces is een GitHub Actions-workflow die het proces in drie hoofdfasen verdeelt:

```
┌─────────────────────────────────────────────────────────────┐
│ GitHub Actions Workflow (Steam Release)                    │
├─────────────────────────────────────────────────────────────┤
│ 1. Preparation Stage:                                      │
│    - Check out portable-version code                       │
│    - Download build artifacts from GitHub Release          │
│    - Extract and prepare the Steam content directory       │
│                                                             │
│ 2. SteamCMD Setup:                                         │
│    - Install or reuse SteamCMD                             │
│    - Authenticate with Steam Guard                         │
│                                                             │
│ 3. Release Stage:                                          │
│    - Generate Depot VDF configuration files                │
│    - Generate App Build VDF configuration files            │
│    - Invoke SteamCMD to upload to Steam                    │
└─────────────────────────────────────────────────────────────┘
```

Dit ontwerp biedt verschillende voordelen:
- Het hergebruikt bestaande GitHub Release-artefacten en vermijdt het opnieuw opbouwen van dezelfde uitvoer.
- Het maakt gebruik van zelfgehoste hardlopers voor veilige isolatie.
- Het ondersteunt het schakelen tussen preview-modus en formele release-takken.
- Het omvat volledige foutafhandeling en logboekregistratie, waardoor fouten gemakkelijker kunnen worden gediagnosticeerd.

## Implementatie van werkstromen

### Triggerparameterontwerp

Onze workflow ondersteunt de volgende belangrijke parameters:

```yaml
inputs:
  release:           # Portable Version release tag
    description: 'Version tag to release (for example v1.0.0)'
    required: true
  steam_preview:     # Whether to generate a preview build
    description: 'Whether to use preview mode'
    required: false
    default: 'false'
  steam_branch:      # Steam branch to set live
    description: 'Target Steam branch'
    required: false
    default: 'preview'
  steam_description: # Build description override
    description: 'Build description'
    required: false
```

### Zelf-gehoste Runner-configuratie

Om veiligheidsredenen gebruiken we een zelfgehoste runner met de `steam` etiket:

```yaml
runs-on:
  - self-hosted
  - Linux
  - X64
  - steam
```

Dit zorgt ervoor dat Steam-releases op een speciale runner draaien en dat gevoelige inloggegevens veilig geïsoleerd blijven.

### Gelijktijdigheidscontrole

Om te voorkomen dat releases van dezelfde versie elkaar storen, hebben we gelijktijdigheidscontrole geconfigureerd:

```yaml
concurrency:
  group: portable-version-steam-${{ github.event.inputs.release }}
  cancel-in-progress: false
```

Merk dat op `cancel-in-progress: false` is hier ingesteld omdat Steam-releases een tijdje kunnen duren, en we willen niet dat een nieuw geactiveerde run een run annuleert die al aan het uploaden is.

## Implementatie van kernscripts

### Vrijgave-invoer voorbereiden

De `prepare-steam-release-input.mjs` script is verantwoordelijk voor het voorbereiden van de invoer die nodig is voor vrijgave:

```javascript
// Download the GitHub Release build manifest and artifact inventory
const buildManifest = await downloadBuildManifest(releaseTag);

for (const platform of ['linux-x64', 'win-x64', 'osx-universal']) {

// Extract into the Steam content directory structure
await extractToSteamContent(sources, contentRoot);
```

### Steam Guard-authenticatie

Steam vereist dat accounts worden beschermd met Steam Guard, daarom hebben we een algoritme voor het genereren van code op basis van gedeelde geheimen geïmplementeerd:

```javascript
function generateSteamGuardCode(sharedSecret, timestamp = Date.now()) {
  const secret = decodeSharedSecret(sharedSecret);
  const time = Math.floor(timestamp / 1000 / 30);

  const timeBuffer = Buffer.alloc(8);
  timeBuffer.writeBigUInt64BE(BigInt(time));

  // Use HMAC-SHA1 to generate a time-based one-time code
  const hash = crypto.createHmac('sha1', secret)
    .update(timeBuffer)
    .digest();

  // Convert it into a 5-character Steam Guard code
  const code = steamGuardCode(hash);
  return code;
}
```

Deze implementatie is gebaseerd op het TOTP-mechanisme (Time-based One-Time Password) van Steam Guard, waarbij elke 30 seconden een nieuwe verificatiecode wordt gegenereerd.

### Generatie van VDF-configuratie

VDF (Valve Data Format) is het configuratieformaat dat door Steam wordt gebruikt. We moeten twee soorten VDF-bestanden genereren:

**Depot VDF** wordt gebruikt om inhoud voor elk platform te configureren:

```javascript
function buildDepotVdf(depotId, contentRoot) {
  return [
    '"DepotBuildConfig"',
    '{',
    `  "DepotID" "${escapeVdf(depotId)}"`,
    `  "ContentRoot" "${escapeVdf(contentRoot)}"`,
    '  "FileMapping"',
    '  {',
    '    "LocalPath" "*"',
    '    "DepotPath" "."',
    '    "recursive" "1"',
    '  }',
    '}'
  ].join('\n');
}
```

**App Build VDF** wordt gebruikt om de gehele applicatie-build te configureren:

```javascript
function buildAppBuildVdf(appId, depotBuilds, description, setLive) {
  const vdf = [
    '"appbuild"',
    '{',
    `  "appid" "${appId}"`,
    `  "desc" "${escapeVdf(description)}"`,
    `  "contentroot" "${escapeVdf(contentRoot)}"`,
    '  "buildoutput" "build_output"',
    '  "depots"',
    '  {'
  ];

  for (const [depotId, depotVdfPath] of Object.entries(depotBuilds)) {
    vdf.push(`    "${depotId}" "${depotVdfPath}"`);
  }

  if (setLive) {
    vdf.push(`  }`);
    vdf.push(`  "setlive" "${setLive}"`);
  }

  vdf.push('}');
  return vdf.join('\n');
}
```

### SteamCMD-aanroep

Ten slotte wordt het uploaden uitgevoerd door SteamCMD aan te roepen:

```javascript
await runCommand(steamcmdPath, [
  '+login', steamUsername, steamPassword, steamGuardCode,
  '+run_app_build', appBuildPath,
  '+quit'
]);
```

Dit is de laatste sprong in de hele workflow. Zodra dit lukt, is de release voltooid.

## Depotafhandeling op meerdere platforms

Steam gebruikt het Depot-systeem om inhoud voor verschillende platforms te beheren. Wij ondersteunen drie belangrijke depots:

| Platform | Depot-identificatie | Ondersteuning van architectuur |
|-----|-----------|---------|
| Linux | `linux-x64` | x64_64 |
| Ramen | `win-x64` | x64_64 |
| macOS | `osx-universal` | universeel, x64_64, arm64 |

Elk depot heeft zijn eigen inhoudsmap en VDF-configuratiebestand. Dit zorgt ervoor dat gebruikers op verschillende platforms alleen de inhoud downloaden die ze daadwerkelijk nodig hebben.

## Werkstroom vrijgeven

### Stap 1: Bereid de GitHub-release voor

Maak eerst een GitHub-release in de `portable-version` opslagplaats met:
- Gecomprimeerde archieven voor elk platform
- Maak een manifest (`{tag}.build-manifest.json`)
- Artefactinventaris (`{tag}.artifact-inventory.json`)

### Stap 2: Activeer de Steam Release-workflow

Activeer de workflow handmatig via GitHub Actions en vul de vereiste parameters in:
- `release`: de versietag die moet worden gepubliceerd, zoals `v1.0.0`
- `steam_branch`: de doelvertakking, zoals `preview` of `public`
- `steam_preview`: of u de voorbeeldmodus wilt gebruiken

### Stap 3: Voer de vrijgavestroom automatisch uit

De workflow voert automatisch de volgende stappen uit:

1. Download en extraheer GitHub Release-artefacten.
2. Installeer of update SteamCMD.
3. Genereer Steam VDF-configuratiebestanden.
4. Authenticeer met Steam Guard.
5. Upload inhoud naar het Steam CDN.
6. Zet de opgegeven branch live.

Zodra deze reeks is voltooid, is het hele releasepad bedekt.

## Beste praktijken

### Steam Guard-authenticatiebeheer

Bij de eerste run moet je de Steam Guard-code handmatig invoeren. Daarna wordt aanbevolen om het gedeelde geheim te configureren, zodat de code automatisch kan worden gegenereerd. Dit vermijdt handmatige tussenkomst bij elke release.

SteamCMD bewaart het inlogtoken en kan het bij latere uitvoeringen hergebruiken. U moet nog steeds de vervaldatum van het token in de gaten houden, omdat herauthenticatie vereist is nadat het token is verlopen.

### Inhoudsmapstructuur

Zorg ervoor dat de Steam-inhoudsmapstructuur correct is:

```
steam-content/
├── linux-x64/     # Linux platform content
├── win-x64/       # Windows platform content
└── osx-universal/ # macOS universal binary content
```

Elke map moet de volledige toepassingsbestanden voor het bijbehorende platform bevatten.

### Gebruik van voorbeeldmodus

De preview-modus stelt geen enkele vertakking live in, dus deze is geschikt voor testen en validatie:

```yaml
if [ "$STEAM_PREVIEW_INPUT" = 'true' ]; then
  cmd+=(--preview)
fi
```

Hiermee kun je eerst uploaden naar Steam ter verificatie en vervolgens overschakelen naar de formele tak nadat alles is uitgecheckt.

### Foutafhandeling en logboekregistratie

De scripts omvatten volledige foutafhandeling en logboekregistratie:
- Valideer dat de GitHub-release bestaat.
- Controleer de vereiste metadatabestanden.
- Zorg ervoor dat platforminhoud aanwezig is.
- Genereer een samenvattend rapport over GitHub-acties.

Deze informatie is zeer waardevol voor zowel foutopsporing als auditing.

### Beheer van artefacten

De workflow genereert twee soorten artefacten:
- `portable-steam-release-preparation-{tag}`: metagegevens voor voorbereiding vrijgeven
- `portable-steam-build-metadata-{tag}`: Metagegevens van Steam-build

Deze artefacten kunnen worden gebruikt voor latere audits en foutopsporing. Een bewaartermijn van 30 dagen is een praktische standaard.

## Gebruik in de echte wereld

In het HagiCode-project is deze geautomatiseerde release-workflow al succesvol uitgevoerd in meerdere versies. Het hele traject van GitHub Release naar het Steam-platform is volledig geautomatiseerd en vereist geen handmatige tussenkomst.

Dit verbeterde zowel de release-efficiëntie als de betrouwbaarheid aanzienlijk. In het verleden duurde het handmatig publiceren van één versie meer dan 30 minuten. Nu is het hele proces in slechts een paar minuten voltooid.

Belangrijker nog is dat de geautomatiseerde workflow de kans op menselijke fouten verkleint. Elke release volgt hetzelfde gestandaardiseerde proces, waardoor de resultaten voorspelbaarder worden.
