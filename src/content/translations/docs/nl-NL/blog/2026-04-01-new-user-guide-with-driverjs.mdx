## Elegante onboarding van nieuwe gebruikers bouwen in React Projects: HagiCode's driver.js-praktijk

> Wanneer gebruikers uw product voor de eerste keer openen, weten ze dan echt waar ze moeten beginnen? In dit artikel wil ik iets vertellen over hoe we driver.js hebben gebruikt voor de onboarding van nieuwe gebruikers in het HagiCode-project. Beschouw het als een klein praktijkvoorbeeld om het gesprek op gang te brengen.

## Achtergrond

Bent u ooit in deze situatie terechtgekomen? Een nieuwe gebruiker meldt zich aan voor uw product, opent de pagina en ziet er meteen verdwaald uit. Ze scannen rond en weten niet zeker waar ze op moeten klikken of wat ze vervolgens moeten doen. Als ontwikkelaars gaan we er vaak van uit dat gebruikers gewoon “zelf op verkenning gaan”, omdat de menselijke nieuwsgierigheid immers grenzeloos is. Maar de realiteit is anders: de meeste gebruikers vertrekken stilletjes binnen enkele minuten als ze het juiste beginpunt niet kunnen vinden, alsof het verhaal plotseling begint en net zo natuurlijk eindigt.

Het onboarden van nieuwe gebruikers is een belangrijke manier om dit probleem op te lossen, maar het goed opbouwen ervan is niet zo eenvoudig. Een goed onboardingsysteem moet:

- Lokaliseer pagina-elementen nauwkeurig en markeer ze
- Ondersteun onboardingstromen in meerdere stappen
- Onthoud de keuze van de gebruiker (aanvullen of overslaan)
- Voorkom dat de paginaprestaties en normale interactie worden beïnvloed
- Houd de codestructuur duidelijk en gemakkelijk te onderhouden

Tijdens het bouwen van HagiCode kwamen we dezelfde uitdaging tegen. HagiCode is een AI-coderingsassistent en de kernworkflow is een OpenSpec-workflow die er als volgt uitziet: "de gebruiker maakt een voorstel -> de AI genereert een plan -> de gebruiker beoordeelt het -> de AI voert het uit." Voor gebruikers die dit concept voor het eerst tegenkomen, is de workflow compleet nieuw, dus ze hebben een solide onboarding nodig om snel aan de slag te gaan. Nieuwe dingen hebben altijd wat tijd nodig om te wennen.

## Waarom we voor driver.js kozen

Tijdens de technische evaluatiefase hebben we gekeken naar verschillende reguliere onboarding-bibliotheken. Ieder had zijn eigen sterke punten:

- **Intro.js**: krachtig, maar relatief groot, en stijlaanpassing is enigszins complex
- **Shepherd.js**: goed ontworpen API, maar een beetje te "zwaar" voor ons gebruik
- **driver.js**: lichtgewicht, beknopte, intuïtieve API en werkt goed in het React-ecosysteem

Uiteindelijk kozen we voor driver.js. Er was geen bijzonder dramatische reden. De keuze kwam vooral neer op deze overwegingen:

1. **Lichtgewicht**: de kernbibliotheek is klein en vergroot de bundelgrootte niet aanzienlijk
2. **Eenvoudige API**: De configuratie is duidelijk en intuïtief, dus gemakkelijk op te pikken
3. **Flexibel**: Ondersteunt aangepaste positionering, stijl en interactiegedrag
4. **Dynamische import**: kan op aanvraag worden geladen zonder de prestaties op het eerste scherm te beïnvloeden

Bij technologieselectie is er zelden een universeel beste antwoord. Meestal is er alleen de optie die het beste past.

## Technische implementatie

### Kernconfiguratie

driver.js heeft een zeer intuïtief configuratiemodel. Dit is de kernconfiguratie die we gebruiken in het HagiCode-project:

```typescript
import { driver } from 'driver.js';
import 'driver.js/dist/driver.css';

const newConversationDriver = driver({
  allowClose: true,           // Allow users to close the guide
  animate: true,              // Enable animations
  overlayClickBehavior: 'close', // Close the guide when the overlay is clicked
  disableActiveInteraction: false, // Keep elements interactive
  showProgress: false,        // Do not show the progress bar (we manage progress ourselves)
  steps: guideSteps           // Array of guide steps
});
```

De redenering achter deze instellingen is:

- `allowClose: true` - Respecteer de keuze van de gebruiker en dwing hem niet om de gids af te maken
- `disableActiveInteraction: false` - Sommige stappen vereisen echte gebruikersacties, zoals het typen van invoer, dus interactie kan niet worden uitgeschakeld
- `overlayClickBehavior: 'close'` - Geef gebruikers een snelle manier om af te sluiten

### Staatsbeheer

Een aanhoudende onboardingstatus is van cruciaal belang. We willen de gids niet elke keer opnieuw opstarten als de pagina wordt vernieuwd, omdat dat snel vervelend wordt. HagiCode gebruikt `localStorage` om de gidsstatus te beheren:

```typescript
export type GuideState = 'pending' | 'dismissed' | 'completed';

export interface UserGuideState {
  session: GuideState;
  detailGuides: Record<string, GuideState>;
}

// Read state
export const getUserGuideState = (): UserGuideState => {
  const state = localStorage.getItem('userGuideState');
  return state ? JSON.parse(state) : { session: 'pending', detailGuides: {} };
};

// Update state
export const setUserGuideState = (state: UserGuideState) => {
  localStorage.setItem('userGuideState', JSON.stringify(state));
};
```

We definiëren drie toestanden:

- `pending`: De handleiding is nog bezig en de gebruiker heeft deze niet voltooid of overgeslagen
- `dismissed`: De gebruiker heeft de gids proactief gesloten
- `completed`: De gebruiker heeft alle stappen voltooid

Voor de onboarding van voorsteldetailpagina's ondersteunen we ook een gedetailleerdere statustracking via de `detailGuides` kaart, omdat één voorstel meerdere fasen kan doorlopen (ontwerp, beoordeling en uitvoering voltooid) en elke fase andere begeleiding nodig heeft. De stand van zaken verandert voortdurend en de onboarding moet dat weerspiegelen.

### Selectie van doelelementen

driver.js gebruikt CSS-selectors om doelelementen te lokaliseren. HagiCode volgt een eenvoudige conventie: gebruik een custom `data-guide` attribuut om onboardingdoelen te markeren:

```typescript
const steps = [
  {
    element: '[data-guide="launch"]',
    popover: {
      title: 'Start a New Conversation',
      description: 'Click here to create a new conversation session...'
    }
  }
];
```

In componenten ziet het er als volgt uit:

```typescript
<button data-guide="launch" onClick={handleLaunch}>
  New Conversation
</button>
```

De voordelen van deze aanpak zijn:

- Vermijd conflicten met namen van bedrijfsstijlklassen
- Duidelijke semantiek, zodat u meteen kunt zien dat het element verband houdt met onboarding
- Gemakkelijker te beheren en consistent te onderhouden

### Dynamische importoptimalisatie

Omdat de onboarding-functie alleen nodig is in specifieke scenario's, zoals het eerste bezoek van een gebruiker, gebruiken we dynamische imports om de initiële laadprestaties te optimaliseren:

```typescript
const initNewUserGuide = async () => {
  // Dynamically import driver.js
  const { driver } = await import('driver.js');
  await import('driver.js/dist/driver.css');

  // Initialize the guide
  const newConversationDriver = driver({
    // ...configuration
  });

  newConversationDriver.drive();
};
```

Op deze manier worden driver.js en zijn stylesheet alleen geladen wanneer dat nodig is en hebben ze geen invloed op de prestaties op het eerste scherm. Niet veel mensen vinden het leuk om te wachten op iets dat ze nog niet eens nodig hebben.

## Onboarding-flowontwerp

HagiCode implementeert twee onboarding-paden die de kernscenario's van de gebruiker bestrijken.

### Sessie-onboarding (10 stappen)

Dit onboarding-traject helpt gebruikers het hele proces te voltooien, van het opzetten van een gesprek tot het indienen van hun eerste volledige voorstel:

1. **lanceren** - Start de gids en introduceer de knop "Nieuw gesprek".
2. **opstellen** - Begeleid de gebruiker bij het typen van een verzoek in het invoervak
3. **verzenden** - Laat de gebruiker op de verzendknop klikken
4. **proposal-launch-readme** - Begeleid de gebruiker bij het maken van een README-voorstel
5. **proposal-compose-readme** - Begeleid de gebruiker bij het bewerken van de README-aanvraaginhoud
6. **proposal-submit-readme** - Begeleid de gebruiker bij het indienen van het README-voorstel
7. **proposal-launch-agents** - Begeleid de gebruiker bij het maken van een `AGENTS.md` Voorstel
8. **proposal-compose-agents** - Begeleid de gebruiker bij het bewerken van het `AGENTS.md` verzoek
9. **proposal-submit-agents** - Begeleid de gebruiker bij het indienen van het `AGENTS.md` Voorstel
10. **voorstel-wachten** - Leg uit dat de AI bezig is met verwerken en dat de gebruiker even moet wachten

Het idee achter dit pad is om gebruikers de kernworkflow van HagiCode uit de eerste hand te laten ervaren door middel van twee echte taken voor het maken van voorstellen, één voor `README` en één voor `AGENTS.md`. Er is een groot verschil tussen horen over een workflow en er zelf doorheen gaan.

### Voorsteldetail onboarding (3 stappen)

Wanneer gebruikers de detailpagina van het voorstel openen, activeert HagiCode de bijbehorende gids op basis van de huidige status van het voorstel:

1. **opstellen** (conceptfase) - Begeleid de gebruiker bij het beoordelen van het door AI gegenereerde plan
2. **reviewing** (reviewfase) - Begeleid de gebruiker bij het uitvoeren van het plan
3. **executionCompleted** (voltooide fase) - Begeleid de gebruiker bij het archiveren van het plan

Het bepalende kenmerk van deze gids is dat deze staatsgestuurd is: hij beslist dynamisch welke onboarding-stap moet worden weergegeven op basis van de feitelijke status van het voorstel. Dingen veranderen en de onboarding moet mee veranderen.

### Element Render-mechanisme voor opnieuw proberen

In React-applicaties is het doelonboarding-element mogelijk nog niet klaar met renderen, bijvoorbeeld omdat asynchrone gegevens nog worden geladen. Om dit aan te pakken, implementeert HagiCode een mechanisme voor opnieuw proberen:

```typescript
const waitForElement = (selector: string, maxRetries = 10, interval = 100) => {
  let retries = 0;
  return new Promise<HTMLElement>((resolve, reject) => {
    const checkElement = () => {
      const element = document.querySelector(selector) as HTMLElement;
      if (element) {
        resolve(element);
      } else if (retries < maxRetries) {
        retries++;
        setTimeout(checkElement, interval);
      } else {
        reject(new Error(`Element not found: ${selector}`));
      }
    };
    checkElement();
  });
};
```

Roep deze functie aan voordat u de guide initialiseert om er zeker van te zijn dat het doelelement al bestaat. Soms is wat langer wachten de moeite waard.

## Samenvatting van beste praktijken

Gebaseerd op de praktijkervaring van HagiCode, volgen hier enkele belangrijke best practices:

### 1. Onboarding moet gemakkelijk te beëindigen zijn

Forceer gebruikers niet om de onboarding te voltooien. Sommige gebruikers zijn van nature ontdekkingsreizigers en zoeken het liefst zelf dingen uit. Zorg voor een duidelijke 'Overslaan'-knop, onthoud hun keuze en onderbreek ze de volgende keer niet opnieuw.

### 2. Houd de introductie-inhoud kort en scherp

Elke onboarding-stap moet zich op één doel richten:

- **Titel**: Kort en duidelijk, idealiter niet meer dan een paar woorden
- **Beschrijving**: Kom meteen ter zake en vertel de gebruiker wat dit is en waarom het ertoe doet

Vermijd langdradige uitleg. De aandacht van gebruikers is beperkt tijdens de introductie, en hoe meer u zegt, hoe kleiner de kans dat ze het lezen.

### 3. Houd de selectors stabiel

Gebruik stabiele elementmarkeringen die niet vaak veranderen. De gewoonte `data-guide` attribuut is een goede keuze. Vermijd afhankelijk te zijn van klassenamen of DOM-structuur, omdat deze tijdens refactoren gemakkelijk kunnen worden gewijzigd. De code verandert voortdurend, maar sommige ankers moeten indien mogelijk stabiel blijven.

### 4. Test uw introductie

HagiCode bevat complete testcases voor de onboarding-functie:

```typescript
describe('NewUserConversationGuide', () => {
  it('should initialize guide state correctly', () => {
    const state = getUserGuideState();
    expect(state.session).toBe('pending');
  });

  it('should update guide state correctly', () => {
    setUserGuideState({ session: 'completed', detailGuides: {} });
    const state = getUserGuideState();
    expect(state.session).toBe('completed');
  });
});
```

Tests helpen ervoor te zorgen dat refactoring het onboarding-gedrag niet per ongeluk verbreekt. Niemand wil dat een kleine codewijziging stilletjes de eerder werkende functionaliteit beschadigt.

### 5. Prestatieoptimalisatie

- Gebruik dynamische import om de onboardingbibliotheek lui te laden
- Vermijd het initialiseren van onboardinglogica nadat de gebruiker de handleiding al heeft voltooid
- Houd rekening met de prestatie-impact van animaties en schakel deze indien nodig uit op lagere apparaten

Prestaties verdienen, zoals veel dingen in het leven, een beetje zorgvuldige budgettering.

## Conclusie

Het onboarden van nieuwe gebruikers is een belangrijk onderdeel van het verbeteren van de gebruikerservaring van producten. In het HagiCode-project hebben we driver.js gebruikt om een ​​compleet onboardingsysteem te bouwen dat de volledige workflow omvat, van het maken van een sessie tot de uitvoering van het voorstel.

De kernpunten die we via dit artikel hopen te delen zijn:

1. **Technische keuzes moeten overeenkomen met de werkelijke behoeften**: driver.js is niet de krachtigste optie, maar past het beste bij ons
2. **Staatsbeheer is van cruciaal belang**: Gebruik `localStorage` om de onboardingstatus te behouden en te voorkomen dat gebruikers herhaaldelijk worden onderbroken
3. **Onboarding-ontwerp moet gefocust blijven**: elke stap moet één probleem oplossen, en niet meer
4. **De codestructuur moet duidelijk blijven**: afzonderlijke onboardingconfiguratie, statusbeheer en UI-logica om het onderhoud eenvoudiger te maken

Als u de onboarding van nieuwe gebruikers aan uw eigen project toevoegt, hoop ik dat de praktische ervaring in dit artikel helpt. Er is niets bijzonders mystieks aan dit soort technologie. Blijf proberen, blijf samenvatten wat je leert, en de dingen worden geleidelijk gemakkelijker.
