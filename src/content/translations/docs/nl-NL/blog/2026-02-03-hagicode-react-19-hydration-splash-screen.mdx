## 12 uitzonderlijke startup-ervaringen ontwerpen voor HagiCode: van minimalisme tot cyberpunk

> De korte periode tussen het downloaden van een React 19-app en het voltooien van Hydration is een gouden kans voor gebruikers om uw merkpersoonlijkheid te voelen. In dit artikel delen we een compleet opstartsysteem dat we voor het HagiCode-project hebben gebouwd met behulp van HTML/CSS/JS.

## Achtergrond

Als moderne applicatie gebouwd met ASP.NET Core 10 en React 19 (Vite), gebruikt HagiCode een frontend-backend gescheiden implementatiearchitectuur. De output van de frontend-build wordt verpakt in de backend `wwwroot/` directory en gehost door ASP.NET Core.

Deze architectuur introduceert echter een klassiek UX-pijnpunt: wanneer gebruikers de pagina bezoeken, laadt de browser eerst de HTML, downloadt vervolgens de grote JS-bundel en laat React ten slotte Hydratatie uitvoeren. Tijdens deze ‘vacuümperiode’, die enkele honderden milliseconden tot enkele seconden duurt, zien gebruikers een leeg scherm of een levenloze statische pagina.

Om die leemte op te vullen en de merkpersoonlijkheid van HagiCode te injecteren, moesten we een systeem in startup-stijl ontwerpen dat volledig was geïmplementeerd met inline code erin. `index.html`.

## Over HagiCode

De benadering van het splash-schermontwerp die in dit artikel wordt gedeeld, komt voort uit onze praktische ervaring in de [HagiCode](https://github.com/HagiCode-org/site) project. ## Kernuitdagingen en architectuurontwerp

Voordat we begonnen met ontwerpen, moesten we eerst de technische beperkingen ophelderen. Omdat alles inline in geïmplementeerd moest worden `index.html`, konden we geen andere externe CSS- of JS-bestanden laden dan de eigen bundel van React.

### Technische beperkingsanalyse

1.  **Nul-afhankelijkheidsprincipe**: Alle stijlen moeten binnen een `<style>` tag, en alle logica moet in a leven `<script>` label.
2.  **Defensieve CSS**: om te voorkomen dat globale stijlen het opstartscherm vervuilen nadat de React-app is geactiveerd, hebben we besloten om alle opstartstijlen te verpakken met een ID-voorvoegsel met hoge prioriteit, zoals `#boot-screen`.
3.  **Prestaties eerst**: Animaties moeten CSS gebruiken `transform` en `opacity` waar mogelijk om terugstromen te voorkomen en ervoor te zorgen dat de rode draad niet geblokkeerd blijft.
4.  **Visuele consistentie**: Kleuren en lettertypen moeten uitgelijnd blijven met de Tailwind-configuratie van HagiCode.

### Architectuurpatroon: Shell & Injector

We hebben een **variantpatroon** aangenomen. De kernlogica is ingekapseld in een onmiddellijk aangeroepen functie-expressie (IIFE), terwijl de specifieke weergavelogica via configuratie wordt geïnjecteerd. Hierdoor kunnen we schakelen tussen verschillende stijlen door middel van eenvoudige configuratie in plaats van de DOM-manipulatielogica herhaaldelijk te herschrijven.

Hier is de kernarchitectuurcode:

```html
<!-- Inline in index.html -->
<div id="boot-root"></div>

<script>
(function() {
  const BootSequence = {
    config: {
      theme: 'terminal', // Configurable as 'minimal', 'skeleton', 'code-rain', etc.
      color: '#3b82f6'   // Brand color
    },
    
    // Core lifecycle
    init() {
      this.render();
      this.listenForMount();
    },

    // Render the currently selected style
    render() {
      const root = document.getElementById('boot-root');
      if (this.variants[this.config.theme]) {
        root.innerHTML = this.variants[this.config.theme].render();
      }
    },

    // Listen for successful React mount and exit gracefully
    listenForMount() {
      window.addEventListener('hagicode:ready', () => {
        const screen = document.getElementById('boot-root');
        // Fade out first, then remove the DOM to avoid flicker
        screen.style.opacity = '0';
        screen.style.transition = 'opacity 0.3s ease';
        setTimeout(() => screen.remove(), 300);
      });
    },

    // The implementation logic for all 12 styles lives here
    variants: {
      // ...see details below
    }
  };

  BootSequence.init();
})();
</script>
```

## Een checklist met 12 startup-stijlontwerpen

We hebben deze twaalf stijlen in zes hoofdcategorieën gegroepeerd om aan verschillende scenario's en esthetische voorkeuren te voldoen.

### A. Minimalisme

> "Minder is meer." Voor scenario's die ultieme laadsnelheid nastreven, bieden wij de lichtst mogelijke opties.

#### 1. Minimalistische stip
Een eenvoudige stip bevindt zich in het midden van het scherm, gecombineerd met een ademhalingsanimatie.
*   **Implementatie**: CSS `@keyframes` regelt de schaal en de dekking.
*   **Beste voor**: elk geval waarin de pagina absoluut schoon moet blijven.

#### 2. Merk onthullen
SVG gebruiken `stroke-dasharray` animatie simuleert het een handgetekende onthulling van de HagiCode-logolijnen, gevolgd door een tekstfade-in.
*   **Techniek**: SVG-padanimatie met een zeer gepolijst gevoel.

### B. Mimicry van skeletschermen

> "De kunst van het bedriegen van het oog." Door een echte UI-indeling te simuleren, hebben gebruikers het gevoel dat de pagina al half geladen is.

#### 3. Chatskelet in de zijbalk
Dit is misschien wel de meest praktische optie. We hebben handmatig een lay-out in HTML gebouwd die de React weerspiegelt `Sidebar` en `ChatInput` componenten precies en legde er vervolgens een grijze glansanimatie overheen.
*   **Waarde**: wanneer de React-hydratatie is voltooid, wordt het skelet onmiddellijk het echte onderdeel en kunnen gebruikers de schakelaar nauwelijks waarnemen.

#### 4. Kaartstapelskelet
Het simuleert de gestapelde beweging van voorstelkaarten tijdens het laden, met behulp van 3D-transformaties om de kaarten subtiel te laten zweven.

### C. Abstractie en kunst

> Pronk met het nerd-DNA van HagiCode.

#### 5. Geometrische morphing
In het midden van het scherm wordt een geometrische vorm (een vierkant) weergegeven, die na verloop van tijd soepel verandert in een cirkel, een driehoek en ten slotte het logo.
*   **Technologie**: vloeiende overgangen met CSS `border-radius`.

#### 6. Codeer regen
Een eerbetoon aan *De Matrix*. Bij gebruik van het JetBrains Mono-lettertype vallen vage tekens op de achtergrond.
*   **Opmerking**: voor prestaties moeten de karakterstreams binnen een kleiner gebied blijven of een lagere vernieuwingsfrequentie gebruiken.

#### 7. Neonpuls
Een gloeiende ring in cyberpunkstijl die meerdere gebruikt `box-shadow` lagen om een krachtige neongloed te creëren.

### D. Branding en thema's

> Zorg ervoor dat het systeem levend aanvoelt.

#### 8. Seizoensthema
Dit is een dynamische lader. Het controleert de huidige datum voor feestdagen zoals Chinees Nieuwjaar of Kerstmis en laadt de bijbehorende SVG-animatie.
*   **Voorbeeld**: tijdens het nieuwe maanjaar zwaaien rode lantaarns zachtjes aan de onderkant van het scherm.

#### 9. Gradiëntstroom
De achtergrond maakt gebruik van een vloeiend verloop gebaseerd op de merkkleuren van HagiCode. Gecombineerd met animatie `background-size` en `background-position`creëert het een aurora-achtig bewegingsgevoel.

### E. Technisch gevoel

> Een groet aan ontwikkelaars.

#### 10. Terminalopstarten
Het simuleert console-uitvoer. Coderegels scrollen snel voorbij:
```text
> Initializing HagiCode Core...
> Loading models...
> Connecting to neural network...
```
Dat voelt voor iedere ontwikkelaar meteen vertrouwd.

#### 11. Voortgangsbalk
Er verschijnt een dunne voortgangsbalk bovenaan het scherm, met aan de rechterkant een percentage. Hoewel we geen toegang hebben tot de werkelijke downloadvoortgang, kunnen we een timer gebruiken om een ​​"geloofwaardig" laadproces te simuleren: snel voor de eerste 80%, daarna geleidelijk langzamer voor de laatste 20%.

### F. Creativiteit

#### 12. Pixelassemblage
Dit is een heel interessant idee. Kleine vierkantjes zijn verspreid over het scherm, convergeren vervolgens naar het midden en vormen geleidelijk het HagiCode-logopictogram. Het symboliseert het proces van het bouwen van code.

## Beste praktijken en valkuilen

In het echte ontwikkelingswerk van HagiCode hebben we een aantal cruciaal belangrijke implementatiedetails samengevat.

### 1. Defensieve CSS is verplicht
Word nooit lui en sla het voorvoegsel over. Ooit vergaten we om de opstartschermstijlen te voorzien van een ID en globaal `div` stijlen nadat React was gemonteerd, hadden onverwacht invloed op het opstartscherm, waardoor de lay-out werd verbroken.
**Les geleerd**: plaats elke CSS-selector eronder `#boot-screen`, en gebruik `!important` om de prioriteit te verhogen wanneer dat nodig is, maar alleen binnen de CSS van het opstartscherm.

### 2. Sierlijke overgangen
Nadat React succesvol is geactiveerd, doe dit niet direct `remove()` het opstartscherm DOM.
**Juiste aanpak**:
1.  Reageer op triggers `window.dispatchEvent(new Event('hagicode:ready'))`.
2.  Het opstartscherm luistert naar de gebeurtenis en de eerste sets `opacity: 0`.
3.  Wacht 300 ms, wat overeenkomt met de duur van de CSS-overgang, en bel `.remove()` pas nadat het scherm volledig onzichtbaar is.

### 3. Synchronisatie van themavariabelen
De kleurwaarden van het opstartscherm zijn hardgecodeerd `index.html`. Als we de primaire kleur van Tailwind wijzigen, moeten we ook het opstartscherm bijwerken.
**Optimalisatie**: schrijf een eenvoudige plug-in in het Vite-buildscript om te lezen `tailwind.config.js` en injecteer kleurvariabelen in het `index.html` sjabloonvariabelen, waardoor één enkele bron van waarheid ontstaat.

### 4. Lettertype vooraf laden
Opstartschermen moeten vaak een merklettertype gebruiken, maar als het lettertype langzaam laadt, kan FOUT (Flash of Unstyled Text) verschijnen.
**Oplossing**: toevoegen `<link rel="preload" href="/fonts/JetBrainsMono.woff2" as="font" type="font/woff2" crossorigin>` binnen `<head>`. Dit is een goedkope manier met een hoog rendement om de ervaring te verbeteren.

### 5. Prestatiemonitoring
Wij injecteerden `performance.mark('boot-start')` aan de onderkant van `index.html`, en gemarkeerd `boot-end` wanneer React succesvol is gemonteerd.
**Waarom het ertoe doet**: door deze gegevens te verzamelen via Application Insights, kunnen we direct meten hoeveel het opstartscherm de waargenomen wachttijd verkort. Uit de gegevens blijkt dat een uitstekend skeletscherm de tolerantie van gebruikers voor een ‘traag netwerk’ met meer dan 50% kan verbeteren.
