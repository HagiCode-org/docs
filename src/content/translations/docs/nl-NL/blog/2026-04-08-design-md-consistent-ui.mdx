## Design.md: een oplossing voor consistent AI-gestuurd frontend UI-ontwerp

> Hoe kun je in het tijdperk van AI-ondersteunde frontend-ontwikkeling de door AI gegenereerde gebruikersinterfaces consistent houden? Dit artikel deelt onze praktijkervaring met het bouwen van een ontwerpgalerijsite op basis van awesome-design-md, samen met hoe u een gestructureerde website kunt maken `design.md` om AI naar een gestandaardiseerd UI-ontwerp te leiden.

## Achtergrond

Iedereen die AI heeft gebruikt om frontend-code te schrijven, heeft waarschijnlijk een soortgelijke ervaring gehad: vraag AI meerdere keren dezelfde pagina te genereren, en elk resultaat komt in een andere stijl uit. Soms zijn de hoeken afgerond, soms zijn ze scherp. Soms is de afstand 8px en soms wordt deze 16px. Zelfs dezelfde knop kan er in verschillende gesprekken anders uitzien.

Dit is geen geïsoleerd probleem. Naarmate AI-ondersteunde ontwikkeling steeds gebruikelijker wordt, is het **gebrek aan consistentie in de door AI gegenereerde frontend-UI** een wijdverbreid probleem geworden. Verschillende AI-assistenten, verschillende aanwijzingen en zelfs dezelfde assistent in verschillende gesprekken kunnen dramatisch verschillende interface-ontwerpen opleveren. Dat zorgt voor enorme onderhoudskosten tijdens productiteratie.

De oorzaak is eigenlijk eenvoudig: er is geen **gezaghebbend ontwerpreferentiedocument**. Traditionele CSS-stylesheets kunnen ontwikkelaars vertellen ‘hoe ze iets moeten implementeren’, maar ze kunnen niet volledig communiceren ‘waarom het op deze manier is ontworpen’ of ‘welk ontwerppatroon in welk scenario moet worden gebruikt’. Voor AI is een duidelijke en gestructureerde beschrijving nog belangrijker om ontwerpconventies te begrijpen.

Tegelijkertijd biedt de open-sourcegemeenschap al een aantal uitstekende bronnen. De [VoltAgent/awesome-design-md](https://github.com/VoltAgent/awesome-design-md) project verzamelt ontwerpsysteemdocumentatie van veel bekende bedrijven. Elke map bevat een `README.md`, een `DESIGN.md`en bekijk een voorbeeld van HTML. Dit alles is echter verspreid over de upstream-repository, waardoor het moeilijk is om snel te bladeren en te vergelijken.

Kunnen we deze bronnen dus consolideren in een gemakkelijk te doorzoeken ontwerpgalerij, terwijl we tegelijkertijd een gestructureerde `design.md` voor AI om te gebruiken?

Het antwoord is ja. Laat me vervolgens onze aanpak doornemen.

## Analyse

Laten we, voordat we code schrijven, de technische uitdagingen achter dit probleem opsplitsen.

### Source Content Management: hoe verenigt u verspreide ontwerpbronnen?

De upstream awesome-design-md repository bevat een groot aantal ontwerpdocumenten, maar we hadden een manier nodig om ze in ons eigen project te brengen.

**Oplossing: gebruik git-submodule**

```
awesome-design-md-site
└── vendor/awesome-design-md          # Upstream resources (git submodule)
```

Dit levert ons verschillende voordelen op:
- Versiebeheer: we kunnen een specifieke upstream-versie vastzetten
- Offline builds: het is niet nodig om externe API's aan te vragen tijdens de build
- Content review: specifieke veranderingen zijn zichtbaar in PR’s

### Statische site-architectuur: hoe zorg je voor dynamisch zoeken terwijl je volledig statisch blijft?

Omdat dit een designgalerij is, is zoeken een must-have. Maar Astro is een statische sitegenerator, dus hoe implementeer je realtime zoeken?

**Oplossing: Reageereiland + synchronisatie van URL-queryparameters**

```typescript
// src/components/gallery/SearchToolbar.tsx

export function SearchToolbar() {
  const [query, setQuery] = useState('');

  // Sync with the URL
```

Het voordeel van deze aanpak is dat de inzetbaarheid van een statische site intact blijft, wat betekent dat deze kan worden ingezet op elke statische hostingservice, terwijl er toch een onmiddellijke filterervaring wordt geboden.

### Ontwerpdocumentatie: hoe helpt u AI ontwerpnormen te begrijpen en te volgen?

Dit is de kern van de hele oplossing. We moeten een structuur creëren `design.md` die AI kan begrijpen en toepassen.

**Oplossing: leen de structuur van ClickHouse `DESIGN.md`**

[ClickHouse's `DESIGN.md`](https://github.com/ClickHouse/ClickHouse/blob/master/DESIGN.md) is een uitstekende referentie. Het omvat:
- Visueel thema en sfeer
- Kleurenpalet en rollen
- Typografieregels
- Component-stijlen
- Lay-outprincipes
- Diepte en hoogte
- Do's en don'ts
- Responsief gedrag
- Agentpromptgids

Onze aanpak is: **hergebruik de structuur, herschrijf de inhoud**. Wij behouden de sectiestructuur van ClickHouse `DESIGN.md`, maar vervang de inhoud door de daadwerkelijke ontwerptokens en componentconventies die in ons eigen project worden gebruikt.

## Oplossing

Op basis van de bovenstaande analyse bestaat onze oplossing uit vier kernmodules.

### 1. Pijplijn voor inhoudsopname

Dit is de basis van het hele systeem, dat verantwoordelijk is voor het extraheren en normaliseren van inhoud uit upstream-bronnen.

```typescript
// src/lib/content/awesomeDesignCatalog.ts

export async function scanSourceEntries(): Promise<DesignEntry[]> {
  const designDir = 'vendor/awesome-design-md/design-md';
  const entries: DesignEntry[] = [];

  for (const dir of await fs.readdir(designDir)) {
    const entryPath = path.join(designDir, dir);
    if (await isValidDesignEntry(entryPath)) {
      const entry = await normalizeDesignEntry(entryPath);
      entries.push(entry);
    }
  }

  return entries;
}

async function isValidDesignEntry(dir: string): Promise<boolean> {
  const requiredFiles = ['README.md', 'DESIGN.md'];
  for (const file of requiredFiles) {
    if (!(await fileExists(path.join(dir, file)))) {
      return false;
    }
  }
  return true;
}
```

### 3. Zoekfunctionaliteit

De zoekfunctie is gebaseerd op filtering aan de clientzijde, waarbij de status wordt bewaard via URL-queryparameters:

```typescript
// src/components/gallery/SearchToolbar.tsx

function SearchToolbar({ entries }: { entries: DesignEntry[] }) {
  const [query, setQuery] = useState('');
  const [results, setResults] = useState(entries);

  useEffect(() => {
    const params = new URLSearchParams(window.location.search);
    const initialQuery = params.get('q') || '';
    setQuery(initialQuery);
    filterEntries(initialQuery);
  }, []);

  const filterEntries = (searchQuery: string) => {
    const filtered = entries.filter(entry =>
      entry.searchText.toLowerCase().includes(searchQuery.toLowerCase())
    );
    setResults(filtered);
  };

  const handleChange = (e: React.ChangeEvent<HTMLInputElement>) => {
    const value = e.target.value;
    setQuery(value);
    filterEntries(value);

    // Update the URL without triggering a page refresh
    const newUrl = value
      ? `${window.location.pathname}?q=${encodeURIComponent(value)}`
      : window.location.pathname;
    window.history.replaceState({}, '', newUrl);
  };

  return (
    <div className="search-toolbar">
      <input
        type="text"
        value={query}
        onChange={handleChange}
        placeholder="Search design entries..."
      />
      <span className="result-count">{results.length} results</span>
    </div>
  );
}
```

## Oefen

Nu we de oplossing hebben besproken, hoe implementeer je deze eigenlijk?

### Implementatiestappen

**Stap 1: Initialiseer de submodule**

```bash
# Add the upstream repository as a submodule
git submodule add https://github.com/VoltAgent/awesome-design-md.git vendor/awesome-design-md

# Initialize and update the submodule
git submodule update --init --recursive
```

**Stap 2: Maak de contentpijplijn**

Implementeren `awesomeDesignCatalog.ts`, inclusief:
- Logica voor het scannen van bestanden en validatie
- Markdown-rendering met behulp van de ingebouwde renderer van Astro
- Extractie van invoergegevens

**Stap 3: Bouw de gebruikersinterface van de galerij**

Gebruik Astro + React-eilanden om het volgende te creëren:
- Indeling van de galerij van de startpagina (kaartraster)
- Ontwerp kaartcomponenten
- Zoekwerkbalk
- Detailpagina-indeling

**Stap 4: Schrijf het ontwerpdocument**

Gebaseerd op de structuur van ClickHouse `DESIGN.md`, vul de daadwerkelijke ontwerptokens van uw eigen project in. Bijwerken `README.md` en voeg een link toe aan `design.md`.

### Opmerkingen

**Beveiliging**: Markdown-weergave vereist het filteren van onveilige HTML. Astro's ingebouwde rendererfilters `script` standaard tags, maar u moet nog steeds opletten voor XSS-risico's.

**Prestaties**: een groot aantal iframe-voorbeelden kan de prestaties van de eerste verf beïnvloeden. Het wordt aanbevolen om te gebruiken `loading="lazy"` om preview-inhoud lui te laden.

**Onderhoudbaarheid**: `design.md` moet synchroon blijven met de code-implementatie. Het wordt aanbevolen om CI-controles toe te voegen om ervoor te zorgen dat CSS-variabelen consistent blijven tussen documentatie en code.

**Toegankelijkheid**: Zorg ervoor dat het kleurcontrast voldoet aan de WCAG AA-norm (minstens 4,5:1).

### AI-gebruiksgids

Na het creëren `design.md`, hoe zorg je ervoor dat AI het daadwerkelijk gebruikt? Hier zijn een paar praktische tips:

**Tip 1: Verwijs er expliciet naar in de prompt**

```
Please refer to the design.md file in the project root and use the design conventions defined there to implement the following components:
- Buttons: use the primary color with an 8px border radius
- Cards: use the elevation-2 shadow level
```

**Tip 2: Vereisen dat AI verwijst naar specifieke CSS-variabelen**

```
Implement a navigation bar with the following requirements:
- Use --color-bg-primary for the background color
- Use --color-border-subtle for borders
- Use --text-color-primary for text
```

**Tip 3: Voeg toe `design.md` inhoud in de systeemprompt**

Als uw AI-tool aangepaste systeemprompts ondersteunt, kunt u de kerninhoud van toevoegen `design.md` er rechtstreeks naar toe.

## Conclusie

De inconsistentie in door AI gegenereerde gebruikersinterfaces die HagiCode tijdens de ontwikkeling tegenkwam, werd in de kern veroorzaakt door het ontbreken van een gestructureerd ontwerpreferentiedocument. Door een ontwerpgalerijsite te bouwen en een gestandaardiseerde website te creëren `design.md`hebben we dit probleem met succes opgelost.

De kernwaarde van deze oplossing ligt in:
- **Unified resources**: consolidatie van verspreide ontwerpsysteemdocumentatie
- **Gestructureerde standaarden**: ontwerpconventies uitdrukken in een formaat dat AI kan begrijpen
- **Continu onderhoud**: inhoud up-to-date houden `git submodule`

Als je AI ook gebruikt voor frontend-ontwikkeling, is deze aanpak het proberen waard. Het creëren van een gestructureerde `design.md` verbetert niet alleen de consistentie van door AI gegenereerde code, maar helpt uw team ook intern uniforme ontwerpnormen te handhaven.
