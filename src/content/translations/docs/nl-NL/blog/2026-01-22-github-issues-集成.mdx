## Het bouwen van een GitHub-probleem Integratie vanaf nul: HagiCode's Frontend Direct Connection-praktijk

> Dit artikel documenteert het volledige proces van het integreren van GitHub Issues in het HagiCode-platform. We zullen onderzoeken hoe we een 'frontend directe verbinding + minimale backend'-architectuur kunnen gebruiken om veilige OAuth-authenticatie en efficiënte probleemsynchronisatie te bereiken, terwijl de backend lichtgewicht blijft.

## Achtergrond: Waarom GitHub integreren?

Als AI-ondersteund ontwikkelingsplatform ligt de kernwaarde van HagiCode in het verbinden van ideeën met implementatie. Maar bij feitelijk gebruik ontdekten we dat nadat gebruikers een voorstel in HagiCode hebben voltooid, ze de inhoud vaak handmatig naar GitHub Issues moeten kopiëren voor het volgen van projecten.

Dit creëert verschillende voor de hand liggende pijnpunten:

1.  **Gefragmenteerde workflow**: Gebruikers moeten heen en weer schakelen tussen twee systemen. De ervaring verloopt niet soepel en belangrijke informatie kan gemakkelijk verloren gaan tijdens het kopiëren en plakken.
2.  **Onhandige samenwerking**: Andere teamleden zijn gewend om taken op GitHub te controleren en kunnen de voortgang van het voorstel niet direct zien in HagiCode.
3.  **Herhaald handmatig werk**: elke keer dat een voorstel wordt bijgewerkt, moet iemand het bijbehorende probleem handmatig bijwerken op GitHub, wat onnodige onderhoudskosten met zich meebrengt.

Om dit probleem op te lossen, hebben we besloten de functie **GitHub Issues Integration** te introduceren, waarmee HagiCode-sessies worden verbonden met GitHub-opslagplaatsen om 'synchronisatie met één klik' mogelijk te maken.

## Technische keuze: directe verbinding met frontend versus backend-proxy

Bij het ontwerpen van de integratieaanpak hadden we twee opties voor ogen: het traditionele 'backend proxy-model' en het agressievere 'frontend directe verbindingsmodel'.

### Oplossingsvergelijking

In het traditionele **backend proxy-model** moet elk verzoek van de frontend eerst via onze backend gaan, die vervolgens de GitHub API aanroept. Dit centraliseert de logica, maar legt ook een aanzienlijke last op de backend:

1.  **Opgeblazen backend**: we zouden een speciale GitHub API-clientwrapper moeten schrijven en ook de complexe OAuth-statusmachine moeten afhandelen.
2.  **Tokenrisico**: het GitHub-token van de gebruiker moet worden opgeslagen in de backend-database. Zelfs met encryptie vergroot dit nog steeds het beveiligingsoppervlak.
3.  **Ontwikkelingskosten**: we hebben databasemigraties nodig om tokens op te slaan en een extra synchronisatieservice om te onderhouden.

Het **frontend directe verbindingsmodel** is veel lichter. Bij deze aanpak gebruiken we de backend alleen voor de meest gevoelige stap voor 'geheime uitwisseling' (de OAuth-callback). Nadat we het token hebben verkregen, slaan we het rechtstreeks in de browser op `localStorage`. Latere bewerkingen, zoals het maken van problemen en het bijwerken van opmerkingen, worden rechtstreeks van de frontend via HTTP naar GitHub verzonden.

| Vergelijkingsdimensie | Backend-proxymodel | Frontend directe verbindingsmodel |
| :--- | :--- | :--- |
| **Backend-complexiteit** | Vereist een volledige OAuth-service en GitHub API-client | Heeft alleen een OAuth-callback-eindpunt nodig |
| **Tokenbeheer** | Moet worden gecodeerd en opgeslagen in de database, met risico op lekkage | Opgeslagen in de browser en alleen zichtbaar voor de gebruiker |
| **Implementatiekosten** | Vereist databasemigraties en ontwikkeling van meerdere services | Voornamelijk frontend werk |
| **Gebruikerservaring** | Gecentraliseerde logica, maar de serverlatentie kan iets hoger zijn | Extreem snelle respons met directe GitHub-interactie |

Omdat we een snelle integratie en minimale veranderingen aan de backend wilden, **kozen we uiteindelijk voor het "frontend directe verbindingsmodel"**. Het is alsof je de browser een ‘tijdelijke pas’ geeft. Zodra de browser de pas heeft gekregen, kan hij zelf dingen op GitHub afhandelen zonder de backend-beheerder elke keer om goedkeuring te vragen.

## Kernontwerp: gegevensstroom en beveiliging

Nadat we de architectuur hadden bepaald, moesten we de specifieke gegevensstroom ontwerpen. De kern van het synchronisatieproces is hoe u het token veilig kunt verkrijgen en efficiënt kunt gebruiken.

### Algemeen architectuurdiagram

Het hele systeem kan worden samengevat in drie rollen: de browser (frontend), de HagiCode-backend en GitHub.

```text
+--------------+        +--------------+        +--------------+
| Frontend     |        |   Backend    |        |   GitHub     |
| React        |        |   ASP.NET    |        |   REST API   |
|              |        |              |        |              |
|  +--------+  |        |              |        |              |
|  | OAuth  |--+--------> /callback    |        |              |
|  | Flow   |  |        |              |        |              |
|  +--------+  |        |              |        |              |
|              |        |              |        |              |
|  +--------+  |        |  +--------+  |        |  +--------+  |
|  | GitHub |  +------------> Session |  +----------> Issues |  |
|  | API    |  |        |  |Metadata|  |        |  |        |  |
|  | Direct |  |        |  +--------+  |        |  +--------+  |
|  +--------+  |        |              |        |              |
+--------------+        +--------------+        +--------------+
```

**Het belangrijkste punt is**: slechts één kleine stap in OAuth (het uitwisselen van de code voor een token) hoeft via de backend te gaan. Daarna wordt het zware werk (het creëren van problemen) rechtstreeks tussen de frontend en GitHub afgehandeld.

### Synchronisatiegegevensstroom in detail

Wanneer de gebruiker op de knop "Synchroniseren met GitHub" in de HagiCode-gebruikersinterface klikt, vindt een reeks complexe acties plaats:

```text
User clicks "Sync to GitHub"
         │
         ▼
1. Frontend checks localStorage for the GitHub token
         │
         ▼
2. Format issue content (convert the Proposal into Markdown)
         │
         ▼
3. Frontend directly calls the GitHub API to create/update the issue
         │
         ▼
4. Call the HagiCode backend API to update Session.metadata (store issue URL and other info)
         │
         ▼
5. Backend broadcasts the SessionUpdated event via SignalR
         │
         ▼
6. Frontend receives the event and updates the UI to show the "Synced" state
```

### Beveiligingsontwerp

Beveiliging heeft altijd de hoogste prioriteit bij het integreren van services van derden. Wij hebben de volgende overwegingen gemaakt:

1.  **Verdedig tegen CSRF-aanvallen**: genereer een willekeurige `state` parameter tijdens de OAuth-omleiding en sla deze op `sessionStorage`. Valideer de status in de callback strikt om vervalste verzoeken te voorkomen.
2.  **Geïsoleerde tokenopslag**: het token wordt alleen in de browser opgeslagen `localStorage`. Met behulp van het Same-Origin-beleid kunnen alleen HagiCode-scripts het lezen, waardoor het risico wordt vermeden dat een databaselek op de server gebruikers treft.
3.  **Foutgrenzen**: we hebben speciale afhandeling ontworpen voor veelvoorkomende GitHub API-fouten (zoals 401 verlopen token, 422-validatiefout en 429-snelheidslimiet), zodat gebruikers vriendelijke feedback ontvangen.

### 2. Frontend OAuth-stroom

Dit is het toegangspunt van de verbinding. Wij gebruiken de standaard Autorisatiecodestroom.

```typescript
// src/HagiCode.Client/src/services/githubOAuth.ts

// Generate the authorization URL and redirect
export async function generateAuthUrl(): Promise<string> {
  const state = generateRandomString(); // Generate a random string for CSRF protection
  sessionStorage.setItem('hagicode_github_state', state);
  
  const params = new URLSearchParams({
    client_id: clientId,
    redirect_uri: window.location.origin + '/settings?tab=github&oauth=callback',
    scope: ['repo', 'public_repo'].join(' '),
    state: state,
  });
  
  return `https://github.com/login/oauth/authorize?${params.toString()}`;
}

// Handle the code-to-token exchange on the callback page
export async function exchangeCodeForToken(code: string, state: string): Promise<GitHubToken> {
  // 1. Validate state to prevent CSRF
  const savedState = sessionStorage.getItem('hagicode_github_state');
  if (state !== savedState) throw new Error('Invalid state parameter');

  // 2. Call the backend API to exchange the token
  // Note: this must go through the backend because ClientSecret cannot be exposed to the frontend
  const response = await fetch('/api/GitHubOAuth/callback', {
    method: 'POST',
    headers: { 'Content-Type': 'application/json' },
    body: JSON.stringify({ code, state, redirectUri: window.location.origin + '/settings?tab=github&oauth=callback' }),
  });

  if (!response.ok) throw new Error('Failed to exchange token');
  
  const token = await response.json();
  
  // 3. Save into LocalStorage
  saveToken(token);
  return token;
}
```

### 3. GitHub API-clientwrapper

Zodra we het token hebben, hebben we een solide tool nodig om de GitHub API aan te roepen.

```typescript
// src/HagiCode.Client/src/services/githubApiClient.ts

const GITHUB_API_BASE = 'https://api.github.com';

// Core request wrapper
async function githubApi<T>(endpoint: string, options: RequestInit = {}): Promise<T> {
  const token = localStorage.getItem('gh_token');
  if (!token) throw new Error('Not connected to GitHub');
  
  const response = await fetch(`${GITHUB_API_BASE}${endpoint}`, {
    ...options,
    headers: {
      ...options.headers,
      Authorization: `Bearer ${token}`,
      Accept: 'application/vnd.github.v3+json', // Specify the API version
    },
  });
  
  // Error handling logic
  if (!response.ok) {
    if (response.status === 401) throw new Error('GitHub token expired, please reconnect');
    if (response.status === 403) throw new Error('No permission to access this repository or rate limit exceeded');
    if (response.status === 422) throw new Error('Issue validation failed, the title may be duplicated');
    throw new Error(`GitHub API Error: ${response.statusText}`);
  }
  
  return response.json();
}

// Create issue
export async function createIssue(owner: string, repo: string, data: { title: string, body: string, labels: string[] }) {
  return githubApi(`/repos/${owner}/${repo}/issues`, {
    method: 'POST',
    body: JSON.stringify(data),
  });
}
```

### 4. Inhoudsopmaak en synchronisatie

De laatste stap is het converteren van HagiCode-sessiegegevens naar het formaat van een GitHub-probleem. Het lijkt een beetje op vertaalwerk.

```typescript
// Convert a Session object into a Markdown string
function formatIssueForSession(session: Session): string {
  let content = `# ${session.title}\n\n`;
  content += `**> HagiCode Session:** #${session.code}\n`;
  content += `**> Status:** ${session.status}\n\n`;
  content += `## Description\n\n${session.description || 'No description provided.'}\n\n`;
  
  // If this is a Proposal session, add extra fields
  if (session.type === 'proposal') {
    content += `## Chief Complaint\n\n${session.chiefComplaint || ''}\n\n`;
    // Add a deep link so users can jump back from GitHub to HagiCode
    content += `---\n\n**[View in HagiCode](hagicode://sessions/${session.id})**\n`;
  }
  
  return content;
}

// Main logic when clicking the sync button
const handleSync = async (session: Session) => {
  try {
    const repoInfo = parseRepositoryFromUrl(session.repoUrl); // Parse the repository URL
    if (!repoInfo) throw new Error('Invalid repository URL');

    toast.loading('Syncing to GitHub...');
    
    // 1. Format content
    const issueBody = formatIssueForSession(session);
    
    // 2. Call API
    const issue = await githubApiClient.createIssue(repoInfo.owner, repoInfo.repo, {
      title: `[HagiCode] ${session.title}`,
      body: issueBody,
      labels: ['hagicode', 'proposal', `status:${session.status}`],
    });
    
    // 3. Update Session Metadata (save the issue link)
    await SessionsService.patchApiSessionsSessionId(session.id, {
      metadata: {
        githubIssue: {
          owner: repoInfo.owner,
          repo: repoInfo.repo,
          issueNumber: issue.number,
          issueUrl: issue.html_url,
          syncedAt: new Date().toISOString(),
        }
      }
    });

    toast.success('Synced successfully!');
  } catch (err) {
    console.error(err);
    toast.error('Sync failed, please check your token or network');
  }
};
```

## Samenvatting en vooruitzichten

Met deze ‘frontend directe verbinding’-aanpak hebben we een naadloze GitHub Issues-integratie bereikt met zo min mogelijk backend-code.

### Belangrijkste winsten
1.  **Hoge ontwikkelingsefficiëntie**: wijzigingen in de backend zijn minimaal, voornamelijk één extra databaseveld en een eenvoudig OAuth-callback-eindpunt. De meeste logica is voltooid op de frontend.
2.  **Sterke beveiliging**: Het token passeert de serverdatabase niet, waardoor het risico op lekkage wordt verminderd.
3.  **Geweldige gebruikerservaring**: Verzoeken worden rechtstreeks vanuit de frontend geïnitieerd, dus de reactiesnelheid is snel en er is geen noodzaak voor het doorsturen van de backend.

### Dingen om op te letten
Er zijn een aantal valkuilen waarmee u rekening moet houden tijdens de echte implementatie:
- **OAuth-app-instellingen**: vergeet niet de juiste gegevens in te voeren `Authorization callback URL` in uw GitHub OAuth-app-instellingen (meestal `http://localhost:3000/settings?tab=github&oauth=callback`).
- **Prijslimieten**: GitHub API beperkt niet-geverifieerde verzoeken vrij strikt, maar met een token is het quotum meestal voldoende (5000 verzoeken/uur).
- **URL-parsing**: gebruikers voeren allerlei soorten repository-URL's in, dus zorg ervoor dat uw regex overeenkomt `.git` achtervoegsels, SSH-formaten en soortgelijke gevallen.

### Toekomstige verbeteringen
De huidige functie is nog steeds eenrichtingssynchronisatie (HagiCode -> GitHub). In de toekomst zijn we van plan tweerichtingssynchronisatie te implementeren via GitHub Webhooks. Als een probleem bijvoorbeeld op GitHub wordt gesloten, kan de sessiestatus aan de HagiCode-kant ook automatisch worden bijgewerkt. Daarvoor moeten we een webhook-eindpunt op de backend beschikbaar stellen, wat een interessante volgende stap zal zijn.
