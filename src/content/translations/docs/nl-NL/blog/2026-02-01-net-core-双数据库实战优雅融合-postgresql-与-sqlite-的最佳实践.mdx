## .NET Core Dual-Database in de praktijk: laat PostgreSQL en SQLite vreedzaam naast elkaar bestaan

> Bij het bouwen van moderne applicaties worden we vaak geconfronteerd met deze afweging: ontwikkelomgevingen willen iets lichts en handigs, terwijl productieomgevingen een hoge mate van gelijktijdigheid en hoge beschikbaarheid vereisen. In dit artikel wordt beschreven hoe u zowel PostgreSQL als SQLite op elegante wijze kunt ondersteunen in een .NET Core-project en hoe u de best practice van "SQLite voor ontwikkeling, PostgreSQL voor productie" kunt implementeren.


## Achtergrond

Bij softwareontwikkeling zijn verschillen tussen omgevingen altijd een van de moeilijkste problemen geweest voor technische teams. Neem het **HagiCode**-platform dat we aan het bouwen zijn als voorbeeld: het is een AI-ondersteund ontwikkelingssysteem gebaseerd op ASP.NET Core 10 en React, waarbij Orleans intern is geïntegreerd voor gedistribueerd staatsbeheer. De stapel is modern en redelijk verfijnd.

Al vroeg in het project kwamen we een klassiek technisch pijnpunt tegen: ontwikkelaars wilden dat de lokale omgeving out-of-the-box werkte, zonder een zware PostgreSQL-database te hoeven installeren en configureren. Maar in de productie moesten we schrijfbewerkingen met hoge gelijktijdigheid en complexe JSON-query's afhandelen, en dat is precies waar lichtgewicht SQLite zijn grenzen begint te tonen.

Hoe kunnen we één enkele codebase behouden en tegelijkertijd de applicatie laten profiteren van de draagbaarheid van SQLite zoals een desktop-app, en ook de krachtige prestaties van PostgreSQL benutten als een dienst op ondernemingsniveau? Dat is de kernvraag die dit artikel onderzoekt.

## Over HagiCode

De aanpassingsaanpak met dubbele databases die in dit artikel wordt gedeeld, komt rechtstreeks voort uit onze praktijkervaring in het **HagiCode**-project. Juist om de ervaring van ontwikkelaars in evenwicht te brengen met productiestabiliteit zijn we tot dit beproefde architecturale patroon gekomen.

## Kernonderwerp 1: Architectuurontwerp en uniforme abstractie

Om twee databases in .NET Core te ondersteunen, is het belangrijkste idee om afhankelijk te zijn van abstracties in plaats van concrete implementaties. We moeten de databaseselectie scheiden van de bedrijfscode en de configuratielaag laten beslissen.

### Ontwerpbenadering

1.  **Unified interface**: Alle bedrijfslogica moet afhankelijk zijn van de `DbContext` basisklasse of aangepaste interfaces, in plaats van een specifieke `PostgreSqlDbContext`.
2.  **Configuratiegestuurd**: gebruik configuratie-items in `appsettings.json` om dynamisch te beslissen welke databaseprovider moet worden geladen bij het opstarten van de applicatie.
3.  **Functie-isolatie**: Voeg aanpassingslogica toe voor PostgreSQL-specifieke mogelijkheden, zoals JSONB, zodat de applicatie nog steeds netjes kan degraderen op SQLite.

### Code-implementatie: dynamische contextconfiguratie

In ASP.NET Core's `Program.cs`, moeten we niet hardcoderen `UseNpgsql` of `UseSqlite`. In plaats daarvan moeten we de configuratie lezen en dynamisch beslissen.

Definieer eerst de configuratieklasse:

```csharp
public class DatabaseSettings
{
    public const string SectionName = "Database";
    
    // Database type: PostgreSQL or SQLite
    public string DbType { get; set; } = "PostgreSQL"; 
    
    // Connection string
    public string ConnectionString { get; set; } = string.Empty;
}
```

Meld de dienst dan aan `Program.cs` gebaseerd op configuratie:

```csharp
// Read configuration
var databaseSettings = builder.Configuration.GetSection(DatabaseSettings.SectionName).Get<DatabaseSettings>();

// Register DbContext
builder.Services.AddDbContext<ApplicationDbContext>(options =>
{
    if (databaseSettings?.DbType?.ToLower() == "sqlite")
    {
        // SQLite configuration
        options.UseSqlite(databaseSettings.ConnectionString);
        
        // Handling SQLite's concurrent write limitations
        // Note: in production, enabling WAL mode is recommended to improve concurrency performance
    }
    else
    {
        // PostgreSQL configuration (default)
        options.UseNpgsql(databaseSettings.ConnectionString, npgsqlOptions =>
        {
            // Enable JSONB support, which is very useful when handling AI conversation records
            npgsqlOptions.UseJsonNet(); 
        });
        
        // Configure connection pool retry policy
        options.EnableRetryOnFailure(3);
    }
});
```

## Kernonderwerp 2: Omgaan met verschillen en migratiestrategie

Hoewel PostgreSQL en SQLite beide de SQL-standaard ondersteunen, verschillen ze aanzienlijk in specifieke mogelijkheden en gedrag. Als er niet zorgvuldig met deze verschillen wordt omgegaan, kun je gemakkelijk in de lastige situatie terechtkomen waarin alles lokaal werkt, maar na de implementatie mislukt.

### 1. Omgaan met JSON-typen

In HagiCode moeten we een grote hoeveelheid prompts en AI-metagegevens opslaan, waarbij meestal JSON-kolommen betrokken zijn.
- **PostgreSQL**: heeft een native `JSONB` type met uitstekende queryprestaties.
- **SQLite**: Heeft geen native JSON-type (nieuwere versies bevatten de JSON1-extensie, maar de objecttoewijzing verschilt nog steeds), dus gegevens worden meestal opgeslagen als TEKST.

**Oplossing**:
Bij het in kaart brengen van EF Core-entiteiten configureren we het als een converteerbaar type.

```csharp
protected override void OnModelCreating(ModelBuilder modelBuilder)
{
    base.OnModelCreating(modelBuilder);

    // Configure entity
    modelBuilder.Entity<PromptTemplate>(entity => 
    {
        entity.Property(e => e.Metadata)
              .HasColumnType("jsonb") // PG uses jsonb
              .HasConversion(
                  v => JsonSerializer.Serialize(v, (JsonSerializerOptions)null),
                  v => JsonSerializer.Deserialize<Dictionary<string, object>>(v, (JsonSerializerOptions)null)
              ); 
    });
}
```

Wanneer SQLite wordt gebruikt, `HasColumnType("jsonb")` kan worden genegeerd of een waarschuwing activeren. Echter, omdat `HasConversion` is geconfigureerd, worden de gegevens nog steeds correct geserialiseerd en gedeserialiseerd als tekenreeksen die zijn opgeslagen in een TEXT-veld, waardoor compatibiliteit wordt gegarandeerd.

### 2. Het scheiden van migratiestrategieën

Probeer nooit één set migratiescripts tegelijkertijd voor zowel PostgreSQL als SQLite te laten werken. Verschillen in strategieën voor het genereren van primaire sleutels, indexsyntaxis en andere databasedetails zullen onvermijdelijk fouten veroorzaken.

**Aanbevolen praktijk**:
Onderhoud twee migratietakken of projecten. In de HagiCode-ontwikkelingsworkflow gaan we er als volgt mee om:

1.  **Ontwikkelingsfase**: Werk voornamelijk met SQLite. Gebruik `Add-Migration Init_Sqlite -OutputDir Migrations/Sqlite`.
2.  **Aanpassingsfase**: schakel na het ontwikkelen van een functie de verbindingsreeks over naar PostgreSQL en voer deze uit `Add-Migration Init_Postgres -OutputDir Migrations/Postgres`.
3.  **Automatiseringsscripts**: schrijf een eenvoudig PowerShell- of Bash-script om automatisch de juiste migratie toe te passen op basis van de huidige omgevingsvariabele.

```bash
# Pseudocode for simple deployment logic
if [ "$DATABASE_PROVIDER" = "PostgreSQL" ]; then
    dotnet ef database update --project Migrations.Postgres
else
    dotnet ef database update --project Migrations.Sqlite
fi
```

## Kernonderwerp 3: Lessen die zijn geleerd van HagiCode in de productie

Bij het herstructureren van **HagiCode** van een model met één database naar ondersteuning voor twee databases, kwamen we een aantal valkuilen tegen en verzamelden we enkele belangrijke lessen die u kunnen helpen dezelfde fouten te voorkomen.

### 1. Verschillen in gelijktijdigheid en transacties

PostgreSQL maakt gebruik van een server-clientarchitectuur en ondersteunt schrijfbewerkingen met hoge gelijktijdigheid met krachtige transactie-isolatieniveaus. SQLite maakt gebruik van bestandsvergrendeling, dus schrijfbewerkingen vergrendelen het gehele databasebestand, tenzij de WAL-modus is ingeschakeld.

**Aanbeveling**:
Bij het schrijven van bedrijfslogica waarbij veelvuldig schrijven nodig is, zoals het in realtime opslaan van de bewerkingsstatus van een gebruiker, moet u rekening houden met het vergrendelingsmodel van SQLite. Bij het ontwerpen van de OpenSpec-samenwerkingsmodule in **HagiCode** hebben we een 'samenvoegen vóór schrijven'-mechanisme geïntroduceerd om de frequentie van directe databaseschrijfbewerkingen te verminderen, waardoor we goede prestaties op beide databases kunnen behouden.

### 2. Levenscyclusbeheer van verbindingsreeksen

Het opzetten van een PostgreSQL-verbinding is relatief duur en is afhankelijk van het poolen van verbindingen. SQLite-verbindingen zijn erg licht, maar als ze niet snel worden vrijgegeven, kunnen bestandsvergrendelingen ervoor zorgen dat latere bewerkingen een time-out krijgen.

In `Program.cs`, kunnen we het gedrag voor elke database verfijnen:

```csharp
if (databaseSettings?.DbType?.ToLower() == "sqlite")
{
    // SQLite: keeping connections open can improve performance, but watch out for file locks
    options.UseSqlite(connectionString, sqliteOptions =>
    {
        // Set command timeout
        sqliteOptions.CommandTimeout(30);
    });
}
else
{
    // PG: make full use of connection pooling
    options.UseNpgsql(connectionString, npgsqlOptions =>
    {
        npgsqlOptions.MaxBatchSize(100);
        npgsqlOptions.CommandTimeout(30);
    });
}
```

### 3. Het belang van testdekking

Veel ontwikkelaars, waaronder enkele vroege leden van ons team, maken vaak één veelgemaakte fout: unit-tests alleen uitvoeren in de ontwikkelomgeving, meestal SQLite.

In de CI/CD-pijplijn van HagiCode hebben we een GitHub Actions-stap afgedwongen om ervoor te zorgen dat elke pull-aanvraag PostgreSQL-integratietests uitvoert.

```yaml
# Example snippet from .github/workflows/test.yml
- name: Run Integration Tests (PostgreSQL)
  run: |
    docker-compose up -d db_postgres
    dotnet test --filter "Category=Integration"
```

Dit heeft ons geholpen talloze bugs op te sporen die verband houden met verschillen in de SQL-syntaxis en hoofdlettergevoeligheid.

## Conclusie

Door het introduceren van een abstractielaag en configuratiegestuurde afhankelijkheidsinjectie hebben we met succes een tweesporige PostgreSQL- en SQLite-installatie geïmplementeerd in het **HagiCode**-project. Dit verlaagde niet alleen de onboardingbarrière voor nieuwe ontwikkelaars aanzienlijk door de noodzaak om PostgreSQL te installeren weg te nemen, maar zorgde ook voor sterke prestatiegaranties voor de productie.

Om de belangrijkste punten samen te vatten:
1.  **Abstractie eerst**: Bedrijfscode mag niet afhankelijk zijn van concrete database-implementaties.
2.  **Afzonderlijke configuratie**: Gebruik verschillende `appsettings.json` bestanden voor ontwikkeling en productie.
3.  **Afzonderlijke migraties**: Probeer niet om één migratieset overal te laten werken.
4.  **Degradatie van functies**: geef prioriteit aan compatibiliteit in SQLite en prestaties in PostgreSQL.

Dit architecturale patroon is niet alleen geschikt voor HagiCode, maar voor elk .NET-project dat een balans moet vinden tussen lichtgewicht ontwikkeling en zware productie.
