## AI CLI-tools uitvoeren in Docker-containers: een praktische gids voor gebruikersisolatie en persistente volumes

> Het integreren van AI-coderingstools zoals Claude Code, Codex en OpenCode in containeromgevingen klinkt eenvoudig, maar er zijn overal verborgen complexiteiten. In dit artikel wordt dieper ingegaan op de manier waarop het HagiCode-project de belangrijkste uitdagingen in Docker-implementaties oplost, inclusief gebruikersrechten, configuratiepersistentie en versiebeheer, zodat u de veelvoorkomende valkuilen kunt vermijden.

## Achtergrond

Toen we besloten om CLI-tools voor AI-codering in Docker-containers uit te voeren, was de meest intuïtieve gedachte waarschijnlijk: "Zijn containers niet gewoon root? Waarom installeren we niet alles rechtstreeks en noemen we het klaar?" In werkelijkheid verbergt dat ogenschijnlijk eenvoudige idee verschillende kernproblemen die opgelost moeten worden.

Ten eerste: **veiligheidsbeperkingen vormen de eerste hindernis**. Neem Claude CLI als voorbeeld: het verbiedt expliciet om als rootgebruiker te werken. Dit is een verplichte veiligheidscontrole en als root wordt gedetecteerd, weigert deze te starten. Je denkt misschien: kan ik niet gewoon van gebruiker wisselen met de `USER` richtlijn? Het is niet zo eenvoudig. Er is nog steeds een toewijzingsprobleem tussen de niet-rootgebruiker in de container en de gebruikersmachtigingen op de hostmachine.

Ten tweede is **statuspersistentie de tweede valkuil**. Claude Code vereist inloggen, Codex heeft zijn eigen configuratie en OpenCode heeft ook een cachemap. Als je elke keer dat de container opnieuw opstart alles opnieuw moet configureren, verliest het hele idee van "automatisering" zijn betekenis. We hebben deze configuraties nodig om ook na de levenscyclus van de container te blijven bestaan.

Het derde probleem is **consistentie van de toestemming**. Kunnen processen binnen de container toegang krijgen tot configuratiebestanden die door de hostgebruiker zijn gemaakt? Niet-overeenkomende UID/GID's veroorzaken vaak fouten in de bestandsrechten, en dit komt zeer vaak voor bij echte implementaties.

Deze problemen lijken misschien onafhankelijk, maar in de praktijk zijn ze nauw met elkaar verbonden. Tijdens de ontwikkeling van HagiCode hebben we gaandeweg een praktische oplossing uitgewerkt. Vervolgens zal ik de technische details delen en de lessen die ik uit deze valkuilen heb geleerd.

## Waarom kunnen we niet gewoon root gebruiken?

Er bestaat hier een veelvoorkomend misverstand: Docker-containers draaien standaard als root, dus waarom installeert u de tools niet gewoon als root? Als u zo denkt, zal Claude CLI u snel het tegendeel leren.

```bash
# Run Claude CLI directly as root? No.
docker run --rm -it --user root myimage claude
# Output: Error: This command cannot be run as root user
```

Dit is een harde beveiligingsbeperking in Claude CLI. De reden is simpel: deze CLI-tools lezen en schrijven gevoelige gebruikersconfiguraties, inclusief API-tokens, lokale caches en zelfs scripts die door de gebruiker zijn geschreven. Als u ze met rootrechten uitvoert, brengt dit te veel risico met zich mee.

De vraag wordt dus: hoe kunnen we voldoen aan de beveiligingseisen van de CLI en tegelijkertijd het containerbeheer flexibel houden? We moeten de manier waarop we erover denken veranderen: **in plaats van tijdens runtime van gebruiker te wisselen, kunt u een speciale gebruiker aanmaken tijdens de fase van het maken van de image**.

## Een toegewijde gebruiker aanmaken: meer dan alleen een naam wijzigen

Je zou kunnen denken dat het toevoegen van een enkele `USER` regel naar het Dockerbestand is voldoende. Dat is inderdaad de eenvoudigste aanpak, maar niet robuust genoeg.

### Statische creatie versus dynamische mapping

De aanpak van HagiCode is het creëren van een `hagicode` gebruiker met UID 1000, wat meestal overeenkomt met de standaardgebruiker op de meeste hostmachines:

```dockerfile
RUN groupadd -o -g 1000 hagicode && \
    useradd -o -u 1000 -g 1000 -s /bin/bash -m hagicode && \
    mkdir -p /home/hagicode/.claude && \
    chown -R hagicode:hagicode /home/hagicode
```

Maar dit lost alleen de ingebouwde gebruiker in de afbeelding op. Wat moet ik doen als de hostgebruiker UID 1001 is? U moet nog steeds dynamische toewijzing ondersteunen wanneer de container start.

docker-entrypoint.sh bevat de sleutellogica:

```bash
if [ -n "$PUID" ] && [ -n "$PGID" ]; then
    if ! id hagicode >/dev/null 2>&1; then
        groupadd -g "$PGID" hagicode
        useradd -u "$PUID" -g "$PGID" -s /bin/bash -m hagicode
    fi
fi
```

Het voordeel van dit ontwerp is duidelijk: **gebruik de standaard UID 1000 tijdens het maken van de image en pas deze vervolgens dynamisch aan tijdens runtime via de `PUID` en `PGID` omgevingsvariabelen**. Ongeacht welke UID de hostgebruiker heeft, het eigendom van configuratiebestanden blijft correct.

## De ontwerpfilosofie van aanhoudende volumes

Elke AI CLI-tool heeft zijn eigen voorkeursconfiguratiemap, dus deze moeten één voor één in kaart worden gebracht:

| CLI-tool | Pad in container | Genoemd volume |
|---------|-----------|--------|
| Claude | `/home/hagicode/.claude` | `claude-data` |
| Codex | `/home/hagicode/.codex` | `codex-data` |
| OpenCode | `/home/hagicode/.config/opencode` | `opencode-config-data` |

Waarom benoemde volumes gebruiken in plaats van bindmounts? Drie redenen:

1. **Eenvoudiger beheer**: Benoemde volumes worden automatisch beheerd door Docker, dus u hoeft hostmappen niet handmatig aan te maken.
2. **Permissie-isolatie**: De initiële inhoud van de volumes wordt door de gebruiker in de container gemaakt, waardoor toestemmingsconflicten met de host worden vermeden.
3. **Onafhankelijke migratie**: volumes kunnen onafhankelijk van containers bestaan, zodat er geen gegevens verloren gaan wanneer afbeeldingen worden geüpgraded.

docker-compose-builder-web genereert automatisch de bijbehorende volumeconfiguratie:

```yaml
volumes:
  claude-data:
  codex-data:
  opencode-config-data:

services:
  hagicode:
    volumes:
      - claude-data:/home/hagicode/.claude
      - codex-data:/home/hagicode/.codex
      - opencode-config-data:/home/hagicode/.config/opencode
    user: "${PUID:-1000}:${PGID:-1000}"
```

Let op de `user` veld hier: `PUID` en `PGID` worden geïnjecteerd via omgevingsvariabelen om ervoor te zorgen dat processen in de container worden uitgevoerd met een identiteit die overeenkomt met de hostgebruiker. Dit detail is van belang omdat toestemmingsproblemen lastig op te lossen zijn zodra ze zich voordoen.

## Versiebeheer: ingebouwde versies met runtime-overschrijvingen

Het vastzetten van Docker-imageversies is essentieel voor reproduceerbaarheid. Maar bij echte ontwikkeling moeten we vaak een nieuwere versie testen of dringend een bug oplossen. Als we de afbeelding elke keer opnieuw zouden moeten opbouwen, zou de workflow veel te inefficiënt zijn.

De strategie van HagiCode is **vaste versies als standaard, met runtime-overschrijvingen als uitbreidingsmechanisme**. Het is een pragmatisch technisch compromis tussen stabiliteit en flexibiliteit.

Dockerfile.template pint versies hier:

```dockerfile
USER hagicode
WORKDIR /home/hagicode

# Configure the global npm install path
RUN mkdir -p /home/hagicode/.npm-global && \
    npm config set prefix '/home/hagicode/.npm-global'

# Install CLI tools using pinned versions
RUN npm install -g @anthropic-ai/claude-code@2.1.71 && \
    npm install -g @openai/codex@0.112.0 && \
    npm install -g opencode-ai@1.2.25 && \
    npm cache clean --force
```

docker-entrypoint.sh ondersteunt runtime-overschrijvingen:

```bash
install_cli_override_if_needed() {
    local package_name="$2"
    local override_version="$5"

    if [ -n "$override_version" ]; then
        gosu hagicode npm install -g "${package_name}@${override_version}"
    fi
}

# Example usage
install_cli_override_if_needed "" "@anthropic-ai/claude-code" "" "" "${CLAUDE_CODE_CLI_VERSION}"
```

Hiermee kunt u een nieuwe versie testen via een omgevingsvariabele zonder de afbeelding opnieuw op te bouwen:

```bash
docker run -e CLAUDE_CODE_CLI_VERSION=2.2.0 myimage
```

Dit ontwerp is praktisch omdat niemand elke keer dat hij een nieuwe functie test een afbeelding opnieuw wil opbouwen.

## Automatische configuratie-injectie

Naast het handmatig configureren van CLI-tools, vereisen sommige scenario's automatische configuratie-injectie. Het meest typische voorbeeld is een API-token.

```bash
if [ -n "$ANTHROPIC_AUTH_TOKEN" ]; then
    mkdir -p /home/hagicode/.claude
    cat > /home/hagicode/.claude/settings.json <<EOF
{
  "env": {
    "ANTHROPIC_AUTH_TOKEN": "${ANTHROPIC_AUTH_TOKEN}"
  }
}
EOF
    chown -R hagicode:hagicode /home/hagicode/.claude
fi
```

Hier zijn twee dingen van belang: **geef gevoelige informatie door via omgevingsvariabelen in plaats van deze hard te coderen in de afbeelding**, en zorg ervoor dat het eigendom van configuratiebestanden correct is ingesteld, anders kunnen de CLI-tools ze niet lezen.

## Best practices en een checklist voor valkuilen

### Problemen met niet-overeenkomende machtigingen

Dit is de gemakkelijkste valkuil waar je in kunt trappen. De hostgebruiker heeft UID 1001, terwijl de container 1000 gebruikt, dus bestanden die aan de ene kant zijn gemaakt, zijn aan de andere kant niet toegankelijk.

```bash
# Correct approach: make the container match the host user
docker run \
    -e PUID=$(id -u) \
    -e PGID=$(id -g) \
    myimage
```

Dit probleem komt veel voor en kan frustrerend zijn als u er de eerste keer mee te maken krijgt.

### De configuratie verdwijnt nadat de container opnieuw is opgestart

Als u merkt dat u na elke herstart opnieuw inlogt, controleer dan of u bent vergeten een persistent volume te koppelen:

```yaml
volumes:
  - claude-data:/home/hagicode/.claude
```

Niets is frustrerender dan het zorgvuldig opzetten van een configuratie om deze vervolgens te zien verdwijnen.

### De juiste manier om versies te upgraden

Niet rennen `npm install -g` direct in een lopende container. De juiste benaderingen zijn:

1. Stel een omgevingsvariabele in om de overschrijvingsinstallatie te activeren.
2. Of herbouw het beeld.

```bash
# Option 1: runtime override
docker run -e CLAUDE_CODE_CLI_VERSION=2.2.0 myimage

# Option 2: rebuild the image
docker build -t myimage:v2 .
```

Er is meer dan één weg naar Rome, maar sommige wegen zijn gladder dan andere.

### Controlelijst voor het versterken van de beveiliging

- Geef API-tokens door via omgevingsvariabelen in plaats van ze in de afbeelding te schrijven.
- Stel de machtigingen voor configuratiebestanden in op `600`.
- Voer de applicatie altijd uit als niet-rootgebruiker.
- Update CLI-versies regelmatig om beveiligingsproblemen op te lossen.

Beveiliging is altijd belangrijk, maar de echte uitdaging is het consequent afdwingen ervan in de praktijk.

## Ondersteuning voor nieuwe CLI-tools uitgebreid

Als u in de toekomst een nieuwe CLI-tool wilt ondersteunen, zijn er slechts drie stappen:

1. **Dockerfile.template**: voeg de installatiestap toe.
2. **docker-entrypoint.sh**: voeg de logica voor het overschrijven van de versie toe.
3. **docker-compose-builder-web**: voeg de persistente volumetoewijzing toe.

Dit op sjablonen gebaseerde ontwerp maakt uitbreiding eenvoudig zonder de kernlogica te veranderen.

## Conclusie

Het uitvoeren van AI CLI-tools in Docker-containers brengt drie kernuitdagingen met zich mee: **gebruikersrechten, configuratiepersistentie en versiebeheer**. Door toegewijde gebruikers, isolatie op naamvolume en op omgevingsvariabelen gebaseerde overschrijvingen te combineren, heeft het HagiCode-project een implementatiearchitectuur gebouwd die zowel veilig als flexibel is.

Belangrijkste ontwerppunten:

- **Gebruikersisolatie**: maak een speciale gebruiker tijdens de fase van het bouwen van de image, met runtime-ondersteuning voor dynamic `PUID`/`PGID` in kaart brengen.
- **Persistentiestrategie**: elke CLI-tool krijgt zijn eigen benoemde volume, dus opnieuw opstarten heeft geen invloed op de configuratie.
- **Versieflexibiliteit**: Vaste standaardwaarden zorgen voor reproduceerbaarheid, terwijl runtime-overschrijvingen ruimte bieden voor testen.
- **Geautomatiseerde configuratie**: Gevoelige configuratie kan automatisch worden geïnjecteerd via omgevingsvariabelen.

Deze oplossing draait al een tijdje stabiel in het HagiCode-project en ik hoop dat het nuttige referentiepunten biedt voor ontwikkelaars met vergelijkbare behoeften.
