## Diepe integratie en praktisch gebruik van StreamJsonRpc in HagiCode

> In dit artikel wordt in detail uitgelegd hoe het HagiCode-project (voorheen PCode) met succes de StreamJsonRpc-communicatiebibliotheek van Microsoft integreerde ter vervanging van de oorspronkelijke aangepaste JSON-RPC-implementatie, terwijl ook de technische pijnpunten en architectonische uitdagingen werden opgelost die tijdens het integratieproces tegenkwamen.

## Achtergrond

StreamJsonRpc is de officieel onderhouden JSON-RPC-communicatiebibliotheek van Microsoft voor .NET en TypeScript, bekend om zijn sterke typeveiligheid, automatische proxy-generatie en volwassen mechanismen voor het afhandelen van uitzonderingen. In het HagiCode-project besloot het team StreamJsonRpc te integreren om te communiceren met externe AI-tools zoals ifflow CLI en OpenCode CLI via ACP (Agent Communication Protocol), terwijl ook de onderhoudskosten en potentiële bugs werden geëlimineerd die werden geïntroduceerd door de eerdere aangepaste JSON-RPC-implementatie. Het integratieproces stuitte echter op uitdagingen die specifiek zijn voor het streamen van JSON-RPC, vooral bij het afhandelen van proxy-doelbinding en generieke parameterherkenning.

Om deze pijnpunten aan te pakken, hebben we een gedurfde beslissing genomen: het hele bouwsysteem vanaf de grond opnieuw opbouwen. De impact van dat besluit kan zelfs groter zijn dan je verwacht, en ik zal het binnenkort in detail uitleggen.

## Analyse

Het huidige project bevindt zich in een kritieke fase van de integratie van het ACP-protocol en wordt geconfronteerd met de volgende technische pijnpunten en architecturale uitdagingen:

### 1. Beperkingen van de Maatwerkimplementatie
De oorspronkelijke JSON-RPC-implementatie bevindt zich in `src/HagiCode.ClaudeHelper/AcpImp/` en bevat componenten zoals `JsonRpcEndpoint` en `ClientSideConnection`. Het onderhouden van deze aangepaste codebase is kostbaar en mist de geavanceerde mogelijkheden van een volwassen bibliotheek, zoals voortgangsrapportage en ondersteuning voor annuleringen.

### 2. StreamJsonRpc-integratieobstakels
Wanneer u probeert het bestaande `CallbackProxyTarget` patroon naar StreamJsonRpc, ontdekten we dat de `_rpc.AddLocalRpcTarget(target)` methode kon geen doelen herkennen die via het proxypatroon waren gemaakt. Met name kon StreamJsonRpc eigenschappen van het generieke type niet automatisch splitsen `T` in RPC-methodeparameters, waardoor de serverkant faalt bij het verwerken van methodeaanroepen die door de client zijn geïnitieerd.

### 3. Verwarde architecturale gelaagdheid
De bestaande `ClientSideConnection` combineert de transportlaag (WebSocket/Stdio), protocollaag (JSON-RPC) en bedrijfslaag (ACP Agent-interface), wat leidt tot onduidelijke verantwoordelijkheden. Het heeft ook last van ontbrekende methodebindingen `AcpAgentCallbackRpcAdapter`.

### 4. Ontbrekende logboeken
De WebSocket-transportlaag ontbeert ruwe JSON-inhoudsregistratie, waardoor het moeilijk is om tijdens RPC-foutopsporing te bepalen of een probleem afkomstig is van serialisatie of van het netwerk.

## Oplossing

Om de bovenstaande problemen aan te pakken, hebben we de volgende systematische oplossing aangenomen, waarbij we optimaliseerden vanuit drie dimensies: architecturale refactoring, bibliotheekintegratie en verbeterde foutopsporing.

### 1. Volledig migreren naar StreamJsonRpc

#### Verouderde code verwijderen
Verwijderen `JsonRpcEndpoint.cs`, `AgentSideConnection.cs`, en gerelateerde aangepaste serialisatieconverters zoals `JsonRpcMessageJsonConverter`.

#### Integreer de officiële bibliotheek
Introduceer de `StreamJsonRpc` NuGet-pakket en gebruik het `JsonRpc` klasse om de kerncommunicatielogica af te handelen.

#### Samenvatting van de transportlaag
Definieer de `IAcpTransport` interface om beide te verwerken `WebSocket` en `Stdio` transportmodi op een uniforme manier, waardoor de protocollaag wordt losgekoppeld van de transportlaag.

```csharp
// Definition of the IAcpTransport interface
public interface IAcpTransport
{
    Task SendAsync(string message, CancellationToken cancellationToken = default);
    Task<string> ReceiveAsync(CancellationToken cancellationToken = default);
    Task CloseAsync(CancellationToken cancellationToken = default);
}

// WebSocket transport implementation
public class WebSocketTransport : IAcpTransport
{
    private readonly WebSocket _webSocket;

    public WebSocketTransport(WebSocket webSocket)
    {
        _webSocket = webSocket;
    }

    // Implement send and receive methods
    // ...
}

// Stdio transport implementation
public class StdioTransport : IAcpTransport
{
    private readonly StreamReader _reader;
    private readonly StreamWriter _writer;

    public StdioTransport(StreamReader reader, StreamWriter writer)
    {
        _reader = reader;
        _writer = writer;
    }

    // Implement send and receive methods
    // ...
}
```

### 2. Problemen met proxydoelherkenning oplossen

#### Analyseer `CallbackProxyTarget`
Inspecteer de bestaande logica voor het genereren van dynamische proxy's en identificeer de hoofdoorzaak waarom StreamJsonRpc deze niet kan herkennen. In de meeste gevallen gebeurt dit omdat het proxyobject de feitelijke methodehandtekeningen niet publiekelijk openbaar maakt, of omdat het parametertypen gebruikt die niet door StreamJsonRpc worden ondersteund.

#### Refactor-parameter doorgeven
Splits generieke eigenschappen op in expliciete RPC-methodeparameters. In plaats van te vertrouwen op dynamische eigenschappen, definieert u concrete Request/Response DTO's (objecten voor gegevensoverdracht), zodat StreamJsonRpc methodehandtekeningen correct kan herkennen door middel van reflectie.

```csharp
// Original generic property approach
public class CallbackProxyTarget<T>
{
    public Func<T, Task> Callback { get; set; }
}

// Refactored concrete method approach
public class ReadTextFileRequest
{
    public string FilePath { get; set; }
}

public class ReadTextFileResponse
{
    public string Content { get; set; }
}

public interface IAcpAgentCallback
{
    Task<ReadTextFileResponse> ReadTextFileAsync(ReadTextFileRequest request);
    // Other methods...
}
```

#### Gebruik `Attach` In plaats van `AddLocalRpcTarget`
In sommige complexe scenario's kan het handmatig proxyeren van de `JsonRpc` object en behandeling `RpcConnection` kan flexibeler zijn dan het rechtstreeks toevoegen van een lokaal doel.

### 3. Implementeer methodebinding en sterkere houtkap

#### Implementeren `AcpAgentCallbackRpcAdapter`
Zorg ervoor dat dit onderdeel expliciet de StreamJsonRpc-proxyinterface implementeert en methoden toewijst die zijn gedefinieerd door het ACP-protocol, zoals `ReadTextFileAsync`, naar StreamJsonRpc-callback-handlers.

#### Integreer logboekregistratie
Onderschep en registreer de onbewerkte tekst van JSON-RPC-aanvragen en -antwoorden in de WebSocket- of Stdio-berichtverwerkingspijplijn. Gebruik `ILogger` om de onbewerkte payload uit te voeren vóór het parseren en na de serialisatie, zodat opmaakproblemen gemakkelijker kunnen worden gediagnosticeerd.

```csharp
// Transport wrapper with enhanced logging
public class LoggingAcpTransport : IAcpTransport
{
    private readonly IAcpTransport _innerTransport;
    private readonly ILogger<LoggingAcpTransport> _logger;

    public LoggingAcpTransport(IAcpTransport innerTransport, ILogger<LoggingAcpTransport> logger)
    {
        _innerTransport = innerTransport;
        _logger = logger;
    }

    public async Task SendAsync(string message, CancellationToken cancellationToken = default)
    {
        _logger.LogTrace("Sending message: {Message}", message);
        await _innerTransport.SendAsync(message, cancellationToken);
    }

    public async Task<string> ReceiveAsync(CancellationToken cancellationToken = default)
    {
        var message = await _innerTransport.ReceiveAsync(cancellationToken);
        _logger.LogTrace("Received message: {Message}", message);
        return message;
    }

    public async Task CloseAsync(CancellationToken cancellationToken = default)
    {
        _logger.LogDebug("Closing connection");
        await _innerTransport.CloseAsync(cancellationToken);
    }
}
```

### 4. Herstructureer de architecturale gelaagdheid

#### Transportlaag (`AcpRpcClient`)
Verpak de StreamJsonRpc-verbinding en maak deze verantwoordelijk voor `InvokeAsync` beheer van oproepen en levenscyclus van verbindingen.

```csharp
public class AcpRpcClient : IDisposable
{
    private readonly JsonRpc _rpc;
    private readonly IAcpTransport _transport;

    public AcpRpcClient(IAcpTransport transport)
    {
        _transport = transport;
        _rpc = new JsonRpc(new StreamRpcTransport(transport));
        _rpc.StartListening();
    }

    public async Task<TResponse> InvokeAsync<TResponse>(string methodName, object parameters)
    {
        return await _rpc.InvokeAsync<TResponse>(methodName, parameters);
    }

    public void Dispose()
    {
        _rpc.Dispose();
        _transport.Dispose();
    }

    // StreamRpcTransport is the StreamJsonRpc adapter for IAcpTransport
    private class StreamRpcTransport : IDuplexPipe
    {
        // Implement the IDuplexPipe interface
        // ...
    }
}
```

#### Protocollaag (`IAcpAgentClient` / `IAcpAgentCallback`)
Definieer duidelijke client-naar-agent- en agent-naar-client-interfaces. Verwijder het cyclische fabriekspatroon `Func<IAcpAgent, IAcpClient>` en vervang het door afhankelijkheidsinjectie of directe callback-registratie.

## Oefen

Op basis van de best practices en projectervaring van StreamJsonRpc zijn dit de belangrijkste aanbevelingen voor implementatie:

### 1. Sterk getypeerde DTO's zijn beter dan dynamische objecten
Het belangrijkste voordeel van StreamJsonRpc ligt in krachtig typen. Niet gebruiken `dynamic` of `JObject` parameters doorgeven. Definieer in plaats daarvan expliciete C# POCO-klassen als parameters voor elke RPC-methode. Dit lost niet alleen het probleem met de proxy-doelherkenning op, maar spoort ook typefouten op tijdens het compileren.

Voorbeeld: vervang de generieke eigenschappen in `CallbackProxyTarget` met concrete klassen zoals `ReadTextFileRequest` en `WriteTextFileRequest`.

### 2. Verklaar expliciet de namen van de methoden
Gebruik de `[JsonRpcMethod]` attribuut om expliciet RPC-methodenamen op te geven in plaats van te vertrouwen op de standaardmethode-naamtoewijzing. Dit voorkomt aanroepfouten die worden veroorzaakt door verschillen in naamgevingsstijl, zoals PascalCase versus camelCase.

```csharp
public interface IAcpAgentCallback
{
    [JsonRpcMethod("readTextFile")]
    Task<ReadTextFileResponse> ReadTextFileAsync(ReadTextFileRequest request);
    
    [JsonRpcMethod("writeTextFile")]
    Task WriteTextFileAsync(WriteTextFileRequest request);
}
```

### 3. Profiteer van callbacks van de verbindingsstatus
StreamJsonRpc biedt de `JsonRpc.ConnectionLost` evenement. U moet absoluut op deze gebeurtenis letten om onverwachte procesafsluitingen of netwerkverbrekingen af te handelen, wat beter opportuun is dan alleen te vertrouwen op de foutdetectie van Orleans Grain.

```csharp
_rpc.ConnectionLost += (sender, e) =>
{
    _logger.LogError("RPC connection lost: {Reason}", e.ToString());
    // Handle reconnection logic or notify the user
};
```

### 4. Registreer logboeken per laag
- **Traceniveau**: registreer de volledige onbewerkte JSON-verzoek/antwoord-payload.
- **Debug-niveau**: Methodeaanroepstapels en parametersamenvattingen opnemen.
- **Opmerking**: Zorg ervoor dat de logboeken geen gevoelige autorisatietokens of Base64-gecodeerde inhoud van grote bestanden bevatten.

### 5. Omgaan met de bijzondere aard van stromend transport
StreamJsonRpc ondersteunt native `IAsyncEnumerable`. Gebruik bij het implementeren van streaming-promptreacties voor ACP `IAsyncEnumerable` rechtstreeks in plaats van aangepaste pagineringslogica. Dit kan de hoeveelheid code die nodig is voor streamingverwerking aanzienlijk vereenvoudigen.

```csharp
public interface IAcpAgentCallback
{
    [JsonRpcMethod("streamText")]
    IAsyncEnumerable<string> StreamTextAsync(StreamTextRequest request);
}
```

### 6. Adapterpatroon
Houd `ACPSession` en `ClientSideConnection` gescheiden. `ACPSession` zou zich moeten concentreren op het staatsbeheer en de bedrijfslogica van Orleans, zoals het in de wachtrij plaatsen van berichten, en zou het StreamJsonRpc-verbindingsobject moeten gebruiken via compositie in plaats van overerving.

Door StreamJsonRpc uitgebreid te integreren, heeft het HagiCode-project met succes de hoge onderhoudskosten, functionele beperkingen en verwarring over de architecturale gelaagdheid van de oorspronkelijke aangepaste implementatie aangepakt. De belangrijkste verbeteringen zijn onder meer:

1. Vervanging van dynamische eigenschappen door sterk getypeerde DTO's, waardoor de onderhoudbaarheid en betrouwbaarheid worden verbeterd
2. Implementatie van abstractie van transportlagen en scheiding van protocollagen, waardoor de architecturale duidelijkheid wordt verbeterd
