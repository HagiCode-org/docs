## Bewerk DESIGN.md rechtstreeks in de webinterface: van idee tot implementatie

> In het MonoSpecs-projectmanagementsysteem `DESIGN.md` draagt het architectonisch ontwerp en de technische beslissingen van een project. Maar de traditionele bewerkingsworkflow dwingt gebruikers om naar een externe editor te springen. Die gefragmenteerde ervaring is alsof je wordt onderbroken tijdens het lezen van een gedicht: de inspiratie is weg, net als de stemming. Dit artikel deelt de oplossing die we in het HagiCode-project in de praktijk hebben gebracht: bewerken `DESIGN.md` rechtstreeks in de webinterface, met ondersteuning voor het importeren van sjablonen van een online ontwerpsite. Wie geniet immers niet van het gevoel alles in één keer af te ronden?

## Achtergrond

Als de belangrijkste drager van projectontwerpdocumenten, `DESIGN.md` bevat belangrijke informatie zoals architectuurontwerp, technische beslissingen en implementatiebegeleiding. De traditionele bewerkingsaanpak vereist echter dat gebruikers overschakelen naar een externe editor zoals VS Code, handmatig het fysieke pad lokaliseren en vervolgens het bestand bewerken. Het is niet bijzonder ingewikkeld, maar na een paar keer herhalen van het proces wordt het vermoeiend.

De problemen manifesteren zich voornamelijk op de volgende manieren:

- **Gefragmenteerde workflow**: gebruikers moeten voortdurend schakelen tussen de webbeheerinterface en een lokale editor, waardoor de continuïteit van hun workflow wordt verbroken, net zoals wanneer de muziek midden in een nummer wordt weggesneden.
- **Moeilijk opnieuw te gebruiken**: de ontwerpsite publiceert al een rijke bibliotheek met ontwerpsjablonen, maar deze kunnen niet rechtstreeks in de projectbewerkingsworkflow worden geïntegreerd. De goede dingen bestaan, maar je kunt het nog steeds niet gebruiken waar je het nodig hebt.
- **Ontbrekende ervaringslus**: er is geen gesloten lus voor "preview-select-import", dus gebruikers moeten handmatig kopiëren en plakken, wat het risico op fouten vergroot.
- **Samenwerkingsfrictie**: het synchroon houden van ontwerpdocumenten en code-implementatie wordt een proces met veel wrijving, wat de teamefficiëntie schaadt.

Om deze pijnpunten op te lossen, hebben we besloten ondersteuning voor directe bewerking toe te voegen `DESIGN.md` in de webinterface en maak het importeren van sjablonen met één klik vanaf een online ontwerpsite mogelijk. Het was geen wereldschokkend besluit. We wilden eenvoudigweg de ontwikkelingservaring soepeler maken.

## Technische oplossing

### Algemene architectuur

Deze oplossing maakt gebruik van een frontend-backend-gescheiden architectuur met een proxy van dezelfde oorsprong, die voornamelijk uit de volgende lagen bestaat. In de praktijk kan het ontwerp worden samengevat als "elk onderdeel doet zijn eigen werk":

**1. Frontend-editorlaag**

```typescript
// Responsibility: handle editing, saving, version conflict detection, and import flow
```

**2. Backend-servicelaag**

```csharp
// Core service: ProjectAppService.DesignMd
// Location: repos/hagicode-core/src/PCode.Application/ProjectAppService.DesignMd.cs
// Responsibility: path resolution, file read/write, and version management
```

**3. Proxylaag van dezelfde oorsprong**

```csharp
// Proxy service: ProjectAppService.DesignMdSiteIndex
// Location: repos/hagicode-core/src/PCode.Application/ProjectAppService.DesignMdSiteIndex.cs
// Responsibility: proxy design site resources, preview image caching, and security validation
```

### Belangrijke technische beslissingen

#### Beslissing 1: Globaal ladepatroon

We gebruiken een enkele globale lade in plaats van lokale pop-uplagen, waarbij de status wordt beheerd `layoutSlice`, waardoor gebruikers een consistente ervaring krijgen in verschillende weergaven (`classic` en `kanban`). Vanuit welke weergave de gebruiker de editor ook opent, hij of zij krijgt hetzelfde interactiemodel. Een consistente ervaring zorgt ervoor dat mensen zich meer op hun gemak voelen in plaats van gedesoriënteerd te raken als ze van mening veranderen.

#### Beslissing 2: Projectgerichte API

Wij zijn gemonteerd `DESIGN.md`-gerelateerde eindpunten onder `ProjectController`, waarbij de bestaande projecttoestemmingsgrens wordt hergebruikt en de complexiteit van het toevoegen van een afzonderlijke controller wordt vermeden. Dit maakt de afhandeling van machtigingen duidelijker en sluit ook aan bij de RESTful-resourceorganisatie. Soms is hergebruik zinvoller dan iets nieuws helemaal opnieuw creëren.

#### Beslissing 3: Versieconflictdetectie

We leiden een ondoorzichtige versie af van die van het bestandssysteem `LastWriteTimeUtc`, wat ons een lichtgewicht, optimistische gelijktijdigheidscontrole geeft. Wanneer meerdere gebruikers hetzelfde bestand tegelijk bewerken, kan het systeem conflicten detecteren en de gebruiker vragen om te vernieuwen. Dit ontwerp blokkeert het bewerken niet, terwijl de gegevensconsistentie nog steeds wordt beschermd.

#### Beslissing 4: Proxypatroon van dezelfde oorsprong

Wij gebruiken `IHttpClientFactory` om externe bronnen van ontwerpsites te proxyen, waarbij zowel cross-origin-problemen als SSRF-risico's worden vermeden. Dit houdt het systeem veilig en vereenvoudigt tegelijkertijd frontend-gesprekken. Je kunt nauwelijks voorzichtig genoeg zijn met beveiliging.

## Kernimplementatie

### 1. Bewerk DESIGN.md rechtstreeks

#### Backend-implementatie

De backend is voornamelijk verantwoordelijk voor padresolutie, lezen/schrijven van bestanden en versiebeheer. Deze taken zijn fundamenteel, maar onmisbaar, zoals de fundering van een huis:

```csharp
// Path resolution and security validation
private Task<string> ResolveDesignDocumentDirectoryAsync(string projectPath, string? repositoryPath)
{
    if (string.IsNullOrWhiteSpace(repositoryPath))
    {
        return Task.FromResult(Path.GetFullPath(projectPath));
    }
    return ValidateSubPathAsync(projectPath, repositoryPath);
}

// Version generation (based on file system timestamp and size)
private static string BuildDesignDocumentVersion(string path)
{
    var fileInfo = new FileInfo(path);
    fileInfo.Refresh();
    return string.Create(
        CultureInfo.InvariantCulture,
        $"{fileInfo.LastWriteTimeUtc.Ticks:x}-{fileInfo.Length:x}");
}
```

Het versieontwerp is interessant door zijn eenvoud: we gebruiken het tijdstip en de grootte van de laatste wijziging van het bestand om een unieke versie-identificatie te genereren. Het is lichtgewicht en betrouwbaar, zonder dat er een extra versiedatabase hoeft te worden onderhouden. Simpele oplossingen zijn vaak het meest effectief.

#### Frontend-implementatie

Aan de frontend implementeren we detectie van vuile toestanden en slaan we logica op. Dit ontwerp helpt gebruikers te begrijpen of hun wijzigingen zijn opgeslagen en vermindert de angst van "wat als ik het verlies?":

```typescript
// Dirty-state detection and save logic
const [draft, setDraft] = useState('');
const [savedDraft, setSavedDraft] = useState('');
const isDirty = draft !== savedDraft;

const handleSave = useCallback(async () => {
    const result = await saveProjectDesignMdDocument({
        ...activeTarget,
        content: draft,
        expectedVersion: document.version, // optimistic concurrency control
    });
    setSavedDraft(draft); // update saved state
}, [activeTarget, document, draft]);
```

Bij deze implementatie behouden we twee staatsdelen: `draft` is de inhoud die momenteel wordt bewerkt, terwijl `savedDraft` is de opgeslagen inhoud. Als u ze vergelijkt, kunt u zien of er niet-opgeslagen wijzigingen zijn. Het ontwerp is eenvoudig, maar het geeft mensen gemoedsrust. Niemand wil dat datgene waar ze hard aan gewerkt hebben, verdwijnt.

#### Indexgegevensformaat

Het indexbestand op de ontwerpsite definieert alle beschikbare sjablonen. Met deze index kunnen gebruikers de gewenste sjabloon net zo eenvoudig kiezen als bestellen via een menu:

```json
{
  "entries": [
    {
      "slug": "linear.app",
      "title": "Linear Inspired Design System",
      "summary": "AI Product / Dark Feel",
      "detailUrl": "/designs/linear.app/",
      "designDownloadUrl": "/designs/linear.app/DESIGN.md",
      "previewLightImageUrl": "...",
      "previewDarkImageUrl": "..."
    }
  ]
}
```

Elk item bevat de basisinformatie van de sjabloon en downloadlinks. De backend leest de lijst met beschikbare sjablonen uit deze index en presenteert deze waaruit de gebruiker kan kiezen. Dat maakt selectie intuïtief in plaats van mensen te dwingen op de tast in het donker te zoeken.

#### Proxy-implementatie van dezelfde oorsprong

Om de zaken veilig te houden, voert de backend strikte validatie uit op de toegang tot de ontwerpsite. Je kunt niet voorzichtig genoeg zijn als het om beveiliging gaat:

```csharp
// Safe slug validation
private static readonly Regex SafeDesignSiteSlugRegex =
    new("^[A-Za-z0-9](?:[A-Za-z0-9._-]{0,127})$", RegexOptions.Compiled);

private static string NormalizeDesignSiteSlug(string slug)
{
    var normalizedSlug = slug?.Trim() ?? string.Empty;
    if (!IsSafeDesignSiteSlug(normalizedSlug))
    {
        throw new BusinessException(
            ProjectDesignSiteIndexErrorCodes.InvalidSlug,
            "Design site slug must be a single safe path segment.");
    }
    return normalizedSlug;
}

// Preview image caching (OS temp directory)
private static string ComputePreviewCacheKey(string slug, string theme, string previewUrl)
{
    var raw = $"{slug}|{theme}|{previewUrl}";
    var bytes = SHA256.HashData(Encoding.UTF8.GetBytes(raw));
    return Convert.ToHexString(bytes).ToLowerInvariant();
}
```

We doen hier twee dingen: ten eerste valideren we het slug-formaat strikt met een reguliere expressie om path traversal-aanvallen te voorkomen; ten tweede cachen we voorbeeldafbeeldingen om de druk op de externe site te verminderen. De eerste is bescherming, de laatste is optimalisatie, en beide zijn van belang.

### 3. Volledige importstroom

```typescript
// 1. Open the import drawer
const handleRequestImportDrawer = useCallback(() => {
    setIsImportDrawerOpen(true);
}, []);

// 2. Select and import
const handleImportRequest = useCallback((entry) => {
    if (isDirty) {
        setPendingImportEntry(entry);
        setConfirmMode('import'); // overwrite confirmation
        return;
    }
    void executeImport(entry);
}, [isDirty]);

// 3. Execute import
const executeImport = useCallback(async (entry) => {
    const result = await getProjectDesignMdSiteImportDocument(
        activeTarget.projectId,
        entry.slug
    );
    setDraft(result.content); // replace editor text only, do not save automatically
    setIsImportDrawerOpen(false);
}, [activeTarget?.projectId]);
```

De importstroom volgt het principe van "gebruikersbevestiging": na het importeren wordt alleen de editorinhoud bijgewerkt en wordt niets automatisch opgeslagen. Gebruikers kunnen de geïmporteerde inhoud alleen inspecteren en handmatig opslaan nadat ze hebben bevestigd dat deze er goed uitziet. De uiteindelijke beslissing moet in de handen van de gebruiker blijven.

## Praktische voorbeelden

### Scenario 1: DESIGN.md maken in de projecthoofdmap

Wanneer `DESIGN.md` bestaat niet, retourneert de backend een virtuele documentstatus. Hierdoor kan de frontend speciale afhandeling voor het geval "bestand bestaat niet" vermijden, en een uniforme API vereenvoudigt de codelogica:

```csharp
return new ProjectDesignDocumentDto
{
    Path = targetPath,
    Exists = false,  // virtual document state
    Content = string.Empty,
    Version = null
};

// Automatically create the file on first save
public async Task<SaveProjectDesignDocumentResultDto> SaveDesignDocumentAsync(...)
{
    Directory.CreateDirectory(targetDirectory);
    await File.WriteAllTextAsync(targetPath, input.Content);
    return new SaveProjectDesignDocumentResultDto { Created = !exists };
}
```

Het voordeel van dit ontwerp is dat de frontend geen speciale logica nodig heeft voor ontbrekende bestanden. Door die complexiteit in de backend te verbergen, kan de frontend zich makkelijker richten op de gebruikerservaring.

### Scenario 2: Importeer een sjabloon van de ontwerpsite

Nadat de gebruiker de "Lineaire" ontwerpsjabloon in de importlade heeft geselecteerd, haalt het systeem de `DESIGN.md` inhoud via de backend-proxy. Het hele proces is transparant voor de gebruiker: hij kiest alleen een sjabloon en het systeem handelt de netwerkverzoeken en gegevenstransformatie automatisch af.

```typescript
// 1. The system fetches DESIGN.md content through the backend proxy
GET /api/project/{id}/design-md/site-index/linear.app

// 2. The backend validates the slug and fetches content from upstream
var entry = FindDesignSiteEntry(catalog, "linear.app");
using var upstreamResponse = await httpClient.SendAsync(request);
var content = await upstreamResponse.Content.ReadAsStringAsync();

// 3. The frontend replaces the editor text
setDraft(result.content);
// The user reviews it and then saves it manually to disk
```

De hele stroom blijft transparant voor de gebruiker. Ze kiezen gewoon een sjabloon en het systeem zorgt achter de schermen voor het netwerken en de transformatie. Dat is de ervaring die we willen: eenvoudig, maar krachtig.

### Scenario 3: versieconflicten afhandelen

Wanneer meerdere gebruikers hetzelfde bewerken `DESIGN.md` tegelijkertijd detecteert het systeem versieconflicten. Dit optimistische controlemechanisme voor gelijktijdigheid waarborgt de gegevensconsistentie zonder de bewerkingen van de gebruiker te blokkeren:

```csharp
if (!string.Equals(currentVersion, expectedVersion, StringComparison.Ordinal))
{
    throw new BusinessException(
        ProjectDesignDocumentErrorCodes.VersionConflict,
        $"DESIGN.md at '{targetPath}' changed on disk.");
}
```

De frontend vangt deze fout op en vraagt de gebruiker:

```typescript
// Frontend prompts the user to refresh and retry
<Alert>
    <AlertTitle>Version conflict</AlertTitle>
    <AlertDescription>
        The file was modified by another process. Please refresh to get the latest version and try again.
    </AlertDescription>
</Alert>
```

Dit optimistische controlemechanisme voor gelijktijdigheid houdt gegevens consistent zonder gebruikers te blokkeren terwijl ze werken. Conflicten zijn onvermijdelijk, maar gebruikers moeten in ieder geval weten wat er is gebeurd, in plaats van hun wijzigingen stilletjes kwijt te raken.

## Opmerkingen en beste praktijken

### 1. Padbeveiliging

Valideer altijd `repositoryPath` om path traversal-aanvallen te voorkomen. Op het gebied van veiligheid kun je nooit teveel doen:

```csharp
// Always validate repositoryPath to prevent path traversal attacks
return ValidateSubPathAsync(projectPath, repositoryPath);
// Reject dangerous inputs such as "../" and absolute paths
```

### 2. Cachestrategie

Cache voorbeeldafbeeldingen gedurende 24 uur, met een maximum van 160 bestanden. Matige caching verbetert de prestaties, maar balans is nog steeds belangrijk:

```csharp
// Cache preview images for 24 hours, with a maximum of 160 files
private static readonly TimeSpan PreviewCacheTtl = TimeSpan.FromHours(24);
private const int PreviewCacheMaxFiles = 160;
// Periodically clean up expired cache
```

### 3. Foutafhandeling

Degradeer op een elegante manier wanneer de upstream-site niet beschikbaar is. Dit ontwerp zorgt ervoor dat zelfs als een externe afhankelijkheid faalt, de kernbewerkingsfunctionaliteit nog steeds normaal werkt:

```typescript
// Gracefully degrade when the upstream site is unavailable
try {
    const catalog = await getProjectDesignMdSiteImportCatalog(projectId);
} catch (error) {
    toast.error(t('project.designMd.siteImport.feedback.catalogLoadFailed'));
    // The main editing drawer remains available
}
```

Deze sierlijke degradatie zorgt ervoor dat zelfs wanneer externe afhankelijkheden niet beschikbaar zijn, de kernbewerkingsfunctie blijft werken. Een systeem moet veerkrachtig zijn en niet instorten zodra er iets misgaat.

### 4. Optimalisatie van gebruikerservaring

Bevestig overschrijvingen vóór het importeren en sla na het importeren niet automatisch op. Gebruikers moeten de controle houden over hun eigen acties:

```typescript
// Confirm overwrite before import
if (isDirty) {
    setConfirmMode('import');
    return;
}

// Do not save automatically after import; let the user confirm
setDraft(result.content); // update draft only
// The content is written to disk only after the user reviews it and clicks Save
```

### 5. Prestatieoverwegingen

Gebruik een HTTP-clientfabriek om te voorkomen dat er te veel verbindingen worden gemaakt. Resourcemanagement lijkt misschien klein, maar als u het goed doet, kan dit een groot verschil maken:

```csharp
// Use an HTTP client factory to avoid creating too many connections
private const string DesignSiteProxyClientName = "ProjectDesignSiteProxy";
private static readonly TimeSpan DesignSiteProxyTimeout = TimeSpan.FromSeconds(8);
```

## Voorgestelde extensies

1. **Markdown-verbetering**: we gebruiken momenteel een basis `Textarea`, maar we zouden kunnen upgraden naar CodeMirror voor syntaxisaccentuering en sneltoetsen. Als de redacteur zich beter voelt, voelt het schrijven van documentatie ook beter.
2. **Voorbeeldmodus**: voeg een realtime Markdown-voorbeeld toe om de bewerkingservaring te verbeteren. Wat-je-ziet-is-wat-je-krijgt, geeft mensen altijd meer zelfvertrouwen.
3. **Diff merge**: implementeer een intelligent samenvoegalgoritme in plaats van eenvoudige volledige tekstvervanging. Conflicten zijn onvermijdelijk, maar het conflictoplossingsproces hoeft niet pijnlijk te zijn.
4. **Lokale caching**: cache `design.json` in de database om de afhankelijkheid van de externe site te verminderen. Hoe minder afhankelijkheden een systeem heeft, hoe stabieler het doorgaans is.
