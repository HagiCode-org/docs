## Automatisering van afbeeldingssynchronisatie van Docker Hub naar Azure ACR

> In dit artikel wordt uitgelegd hoe u GitHub Actions en de image-syncer-tool kunt gebruiken om de synchronisatie van afbeeldingen van Docker Hub naar Azure Container Registry te automatiseren, waarbij de trage toegang tot Docker Hub in China en sommige Azure-regio's wordt aangepakt en tegelijkertijd de beschikbaarheid van afbeeldingen en de implementatie-efficiëntie in Azure-omgevingen worden verbeterd.

## Achtergrond / Inleiding

Het HagiCode-project gebruikt Docker-images als belangrijkste runtime-componenten, waarbij de primaire images worden gehost op Docker Hub. Naarmate het project groeide en de behoeften voor Azure-implementatie toenamen, kwamen we de volgende pijnpunten tegen:

- Lage snelheden voor het ophalen van afbeeldingen, waarbij de toegang tot Docker Hub beperkt is in China en sommige Azure-regio's
- Eén enkele beeldbron creëert een single point of Failure-risico
- Azure Container Registry biedt betere netwerkprestaties en integratie bij uitvoering in Azure

Om deze problemen op te lossen hadden we een geautomatiseerd mechanisme voor beeldsynchronisatie nodig om regelmatig afbeeldingen van Docker Hub naar Azure ACR te synchroniseren, zodat gebruikers snellere beeldophaalsnelheden en een hogere beschikbaarheid in Azure-omgevingen kunnen krijgen.

## Vergelijking van technische oplossingen

Bij het definiëren van de oplossing hebben we meerdere technische opties vergeleken:

### 1. beeldsynchronisatie (definitieve keuze)
- Incrementele synchronisatie: synchroniseert alleen gewijzigde beeldlagen, waardoor de netwerkoverdracht aanzienlijk wordt verminderd
- Ondersteuning hervatten: de synchronisatie kan doorgaan na netwerkonderbrekingen
- Gelijktijdigheidscontrole: configureerbare werkthreads verbeteren de efficiëntie van de synchronisatie van grote afbeeldingen
- Robuuste foutafhandeling: ingebouwd mechanisme voor opnieuw proberen bij fouten (standaard 3 nieuwe pogingen)
- Lichtgewicht implementatie: één binair bestand zonder afhankelijkheden
- Ondersteuning voor meerdere registers: compatibel met Docker Hub, Azure ACR, Harbor en meer

### 2. Docker-CLI
- Geen incrementele synchronisatie: de volledige afbeeldingsinhoud moet elke keer worden opgehaald
- Lagere efficiëntie: groot netwerkoverdrachtsvolume en langere uitvoeringstijd
- Eenvoudig in gebruik: gebaseerd op bekend `docker pull` en `docker push` opdrachten

### 3. Azure CLI
- Hogere complexiteit: Azure CLI-verificatie moet worden geconfigureerd
- Beperkte functionaliteit: `az acr import` heeft een relatief beperkte reikwijdte
- Native integratie: integreert goed met Azure-services

## Architectuurontwerpbeslissingen

### Beslissing 1: Stel de synchronisatiefrequentie in op dagelijks om 00:00 UTC
- Brengt de frisheid van het beeld en het verbruik van hulpbronnen in evenwicht
- Vermijdt zakelijke piekuren en vermindert de impact op andere activiteiten
- Docker Hub-images worden doorgaans bijgewerkt na dagelijkse builds

### Beslissing 2: Synchroniseer alle afbeeldingstags
- Behoudt volledige consistentie met Docker Hub
- Geeft gebruikers flexibele versiekeuzes
- Vereenvoudigt de synchronisatielogica en vermijdt complexe tagfilterregels

### Beslissing 3: gebruik GitHub Secrets om inloggegevens op te slaan
- Ondersteuning voor native GitHub-acties met krachtige beveiliging
- Eenvoudige configuratie en eenvoudig onderhoud
- Ondersteunt toegangscontrole op repositoryniveau

## Risicobeoordeling en -beperking

### Risico 1: lekken van Azure ACR-referenties
- Bewaar inloggegevens versleuteld in GitHub Secrets
- Wissel ACR-wachtwoorden regelmatig af
- Beperk het ACR-account tot alleen-push-machtigingen
- Bewaak ACR-toegangslogboeken

### Risico 2: Synchronisatiefouten veroorzaken inconsistentie in het beeld
- image-syncer bevat een ingebouwd incrementeel synchronisatiemechanisme
- Automatische nieuwe pogingen bij falen (standaard 3)
- Gedetailleerde foutenlogboeken en foutmeldingen
- Hervat vermogen

### Risico 3: Overmatig verbruik van hulpbronnen
- Incrementele synchronisatie vermindert de netwerkoverdracht
- Configureerbare gelijktijdigheid (`10` in de huidige configuratie)
- Controleer het aantal en de grootte van gesynchroniseerde beelden
- Voer synchronisatie uit tijdens de daluren

## Kernoplossing

We gebruiken een geautomatiseerde oplossing op basis van GitHub Actions + image-syncer om afbeeldingen van Docker Hub naar Azure ACR te synchroniseren.

## Implementatiestappen

### 1. Voorbereiding
- Maak of bevestig een Azure Container Registry in Azure Portal
- ACR-toegangsreferenties aanmaken (gebruikersnaam en wachtwoord)
- Bevestig de toegangsrechten voor de Docker Hub-imagerepository

### 2. Configureer GitHub-geheimen
Voeg de volgende geheimen toe in de GitHub-repository-instellingen:
- AZURE_ACR_USERNAME: Azure ACR-gebruikersnaam
- AZURE_ACR_PASSWORD: Azure ACR-wachtwoord

### 3. Maak de GitHub Actions-workflow
Configureer de workflow in `.github/workflows/sync-docker-acr.yml`:
- Geplande trigger: elke dag om 00:00 UTC
- Handmatige trigger: ondersteunt `workflow_dispatch`
- Extra trigger: loopt op duwt naar de `publish` branch voor snelle synchronisatie

### 4. Uitvoeringsstroom van de workflow

| Stap | Deelnemer | Actie | Beschrijving |
| --- | --- | --- | --- |
| 1 | GitHub-acties | Activeer de werkstroom | Geactiveerd door planning, handmatige uitvoering of push-ups naar de `publish` tak |
| 2 | GitHub-acties → afbeeldingssynchronisatie | Download de synchronisatietool en voer deze uit | Gaat naar de daadwerkelijke synchronisatiefase |
| 3 | afbeeldingssynchronisatie → Docker Hub | Haal afbeeldingsmanifesten en taglijst op | Leest metagegevens uit de bronrepository |
| 4 | beeldsynchronisatie → Azure ACR | Haal bestaande afbeeldingsgegevens op uit de doelrepository | Bepaalt de huidige status van de doelzijde |
| 5 | beeldsynchronisatie | Vergelijk bron- en doelverschillen | Identificeert de afbeeldingslagen die synchronisatie nodig hebben |
| 6 | afbeeldingssynchronisatie → Docker Hub | Trek gewijzigde afbeeldingslagen op | Brengt alleen de inhoud over die moet worden bijgewerkt |
| 7 | beeldsynchronisatie → Azure ACR | Push gewijzigde afbeeldingslagen | Voltooit de incrementele synchronisatie |
| 8 | afbeeldingssynchronisatie → GitHub-acties | Synchronisatiestatistieken retourneren | Bevat resultaten, verschillen en foutinformatie |
| 9 | GitHub-acties | Registreer logboeken en upload artefacten | Ondersteunt latere audits en probleemoplossing |

## GitHub Actions-werkstroomimplementatie

Hieronder vindt u de daadwerkelijke workflowconfiguratie die in gebruik is (`.github/workflows/sync-docker-acr.yml`):

```yaml
name: Sync Docker Image to Azure ACR

on:
  schedule:
    - cron: "0 0 * * *" # Every day at 00:00 UTC
  workflow_dispatch: # Manual trigger
  push:
    branches: [publish]

permissions:
  contents: read

jobs:
  sync:
    runs-on: ubuntu-latest

    steps:
      - name: Checkout code
        uses: actions/checkout@v4

      - name: Download image-syncer
        run: |
          # Download the image-syncer binary
          wget https://github.com/AliyunContainerService/image-syncer/releases/download/v1.5.5/image-syncer-v1.5.5-linux-amd64.tar.gz
          tar -zxvf image-syncer-v1.5.5-linux-amd64.tar.gz
          chmod +x image-syncer

      - name: Create auth config
        run: |
          # Generate the authentication config file (YAML format)
          cat > auth.yaml <<EOF
          hagicode.azurecr.io:
            username: "${{ secrets.AZURE_ACR_USERNAME }}"
            password: "${{ secrets.AZURE_ACR_PASSWORD }}"
          EOF

      - name: Create images config
        run: |
          # Generate the image sync config file (YAML format)
          cat > images.yaml <<EOF
          docker.io/newbe36524/hagicode: hagicode.azurecr.io/hagicode
          EOF

      - name: Run image-syncer
        run: |
          # Run synchronization (using the newer --auth and --images parameters)
          ./image-syncer --auth=./auth.yaml --images=./images.yaml --proc=10 --retries=3

      - name: Upload logs
        if: always()
        uses: actions/upload-artifact@v4
        with:
          name: sync-logs
          path: image-syncer-*.log
          retention-days: 7
```

## Configuratiedetails

### 1. Voorwaarden activeren
- Geplande trigger: `cron: "0 0 * * *"` loopt elke dag om 00:00 UTC
- Handmatige trigger: `workflow_dispatch` stelt gebruikers in staat het handmatig uit te voeren vanuit de GitHub-gebruikersinterface
- Druk op de trekker: `push: branches: [publish]` wordt geactiveerd bij push-ups naar de release-tak voor snelle synchronisatie

### 2. Authenticatieconfiguratie (`auth.yaml`)
```yaml
hagicode.azurecr.io:
  username: "${{ secrets.AZURE_ACR_USERNAME }}"
  password: "${{ secrets.AZURE_ACR_PASSWORD }}"
```

### 3. Configuratie van beeldsynchronisatie
```yaml
docker.io/newbe36524/hagicode: hagicode.azurecr.io/hagicode
```
Deze configuratie betekent dat alle tags van `docker.io/newbe36524/hagicode` zal worden gesynchroniseerd `hagicode.azurecr.io/hagicode`.

### 4. parameters voor beeldsynchronisatie
- `--auth=./auth.yaml`: pad naar het authenticatieconfiguratiebestand
- `--images=./images.yaml`: pad naar het configuratiebestand voor beeldsynchronisatie
- `--proc=10`: gebruik `10` gelijktijdige werknemers
- `--retries=3`: mislukte pogingen `3` keer

## Opmerkingen en beste praktijken

### 1. Beveiligingsaanbevelingen
- Roteer Azure ACR-wachtwoorden regelmatig (aanbevolen elke 90 dagen)
- Gebruik een speciaal ACR-serviceaccount met alleen-push-machtigingen
- Bewaak ACR-toegangslogboeken om abnormale toegang tijdig te detecteren
- Voer geen inloggegevens uit in logboeken
- Leg geen inloggegevens vast in de coderepository

### 2. Prestatieoptimalisatie
- Pas de aan `--proc` parameter: stem gelijktijdigheid af op basis van netwerkbandbreedte (aanbevolen `5-20`)
- Houd de synchronisatieduur in de gaten: als de synchronisatie te lang duurt, kunt u overwegen de gelijktijdigheid te verlagen
- Ruim logboeken regelmatig op: stel een redelijk bedrag in `retention-days` waarde (`7` momenteel)

### 3. Problemen oplossen

#### Probleem 1: verificatiefout
```
Error: failed to authenticate to hagicode.azurecr.io
```
Oplossing:
1. Controleer of GitHub Secrets correct zijn geconfigureerd
2. Controleer of het Azure ACR-wachtwoord is verlopen
3. Controleer of de machtigingen voor het ACR-serviceaccount correct zijn

#### Probleem 2: Netwerktime-out
```
Error: timeout waiting for response
```
Oplossing:
1. Controleer de netwerkconnectiviteit
2. Verminder het aantal gelijktijdige werknemers (de `--proc` parameter)
3. Wacht tot het netwerk is hersteld en voer de workflow opnieuw uit

#### Probleem 3: Onvolledige beeldsynchronisatie
```
Warning: some tags failed to sync
```
Oplossing:
1. Controleer de synchronisatielogboeken om mislukte tags te identificeren
2. Voer de workflow handmatig opnieuw uit om opnieuw te synchroniseren
3. Controleer of de bronimage op Docker Hub in orde is

### 4. Monitoring en alarmering
- Controleer regelmatig de pagina Acties om de workflowstatus te bevestigen
- Schakel GitHub-meldingen in, zodat workflowfouten onmiddellijk worden gerapporteerd
- Bewaak het gebruik van Azure ACR-opslag
- Controleer regelmatig de consistentie van de tags in alle registers

## Veelgestelde vragen en oplossingen

### V1: Hoe synchroniseer ik specifieke tags in plaats van alle tags?

Wijzig de `images.yaml` configuratiebestand:
```yaml
# Sync only the latest and v1.0 tags
docker.io/newbe36524/hagicode:latest: hagicode.azurecr.io/hagicode:latest
docker.io/newbe36524/hagicode:v1.0: hagicode.azurecr.io/hagicode:v1.0
```

### Vraag 2: Hoe synchroniseer ik meerdere afbeeldingsopslagplaatsen?

Voeg meerdere regels toe `images.yaml`:
```yaml
docker.io/newbe36524/hagicode: hagicode.azurecr.io/hagicode
docker.io/newbe36524/another-image: hagicode.azurecr.io/another-image
```

### V3: Hoe kan ik het opnieuw proberen nadat de synchronisatie is mislukt?
- Automatisch opnieuw proberen: image-synchronisatie bevat een ingebouwd mechanisme voor opnieuw proberen (`3` standaard nieuwe pogingen)
- Handmatig opnieuw proberen: klik `Re-run all jobs` op de pagina GitHub-acties

### V4: Hoe kan ik de gedetailleerde voortgang van de synchronisatie bekijken?
- Bekijk realtime logboeken op de pagina Acties
- Download de `sync-logs` artefact om de volledige logbestanden te inspecteren
- De logbestanden bevatten de synchronisatiestatus en overdrachtsnelheid voor elke tag

### Vraag 5: Hoe lang duurt de synchronisatie?
- Initiële volledige synchronisatie: meestal `10-30` minuten, afhankelijk van de beeldgrootte
- Incrementele synchronisatie: meestal `2-5` minuten als de beeldveranderingen klein zijn
- De timing is afhankelijk van de netwerkbandbreedte, de afbeeldingsgrootte en de gelijktijdigheidsinstellingen

## Voorgestelde extensies

### 1. Voeg synchronisatiemeldingen toe
Voeg een meldingsstap toe aan de workflow:
```yaml
- name: Notify on success
  if: success()
  run: |
    echo "Docker images synced successfully to Azure ACR"
```

### 2. Implementeer filtering van afbeeldingstags
Voeg tagfilterlogica toe aan de workflow:
```yaml
- name: Filter tags
  run: |
    # Sync only tags prefixed with v
    echo "docker.io/newbe36524/hagicode:v* : hagicode.azurecr.io/hagicode:v*" > images.yaml
```

### 3. Voeg een rapport met synchronisatiestatistieken toe
```yaml
- name: Generate report
  if: always()
  run: |
    echo "## Sync Report" >> $GITHUB_STEP_SUMMARY
    echo "- Total tags: $(grep -c 'synced' image-syncer-*.log)" >> $GITHUB_STEP_SUMMARY
    echo "- Sync time: ${{ steps.sync.outputs.duration }}" >> $GITHUB_STEP_SUMMARY
```
