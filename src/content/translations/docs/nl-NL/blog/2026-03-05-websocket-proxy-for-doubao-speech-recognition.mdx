## Problemen met browser-websocket-authenticatie oplossen: een praktisch proxypatroon voor Doubao-spraakherkenning

> De browser WebSocket API ondersteunt geen aangepaste HTTP-headers, wat een uitdaging vormt voor spraakherkenningsservices die authenticatiegegevens in headers vereisen. In dit artikel wordt beschreven hoe we dat probleem in het HagiCode-project hebben opgelost met een backend-proxypatroon, en hoe de aanpak is geëvolueerd van speeltuinexperimenten naar productiegebruik.


## Achtergrond

Toen we begonnen met het bouwen van spraakherkenning voor het HagiCode-project, kozen we vol vertrouwen voor de Doubao-spraakherkenningsservice van ByteDance. Het oorspronkelijke ontwerp was eenvoudig: laat de frontend rechtstreeks verbinding maken met de WebSocket-service van Doubao. Hoe moeilijk kan dat zijn? Open gewoon een verbinding en stuur wat gegevens, toch?

Toen kwam de verrassing: de API van Doubao vereist dat authenticatie-informatie wordt doorgegeven via HTTP-headers, inclusief zaken als `accessToken` en `secretKey`. Dat werd meteen lastig, omdat de browser WebSocket API simpelweg geen ondersteuning biedt voor het instellen van custom headers.

Dus wat doe je als de browser je niet toestaat ze te verzenden?

We hebben toen twee opties afgewogen:

1. Zet de inloggegevens in URL-queryparameters - eenvoudig en duidelijk
2. Voeg een proxylaag toe aan de backend - meer werk op het eerste gezicht

De eerste optie maakt inloggegevens rechtstreeks zichtbaar in de frontendcode en lokale opslag. Is dat echt veilig? Ik voelde me er niet prettig bij. Bovendien vereisen sommige API's headergebaseerde verificatie, dus deze aanpak is niet eens haalbaar.

Uiteindelijk hebben we voor de tweede optie gekozen: een WebSocket-proxy op de backend implementeren. Toevallig werd dit patroon voor het eerst gevalideerd in onze speeltuinomgeving, en pas nadat we de stabiliteit ervan hadden bevestigd, hebben we het in productie genomen. Niemand wil immers dat de productie ook dienst doet als laboratoriumexperiment.

## Analyse van technische uitdagingen

### Browser WebSocket-beperkingen

De standaard WebSocket API ziet er wonderbaarlijk eenvoudig uit:

```javascript
const ws = new WebSocket('wss://example.com/ws');
```

Maar die eenvoud is precies waar het probleem ligt: het geeft alleen parameters door in de URL, en het kan geen headers instellen zoals een HTTP-verzoek dat kan:

```javascript
// This is not supported in the WebSocket API
const ws = new WebSocket('wss://example.com/ws', {
  headers: {
    'Authorization': 'Bearer token'
  }
});
```

En dat is het kernprobleem. Voor diensten zoals Doubao-spraakherkenning die afhankelijk zijn van op headers gebaseerde authenticatie, is deze beperking een harde blokkering.

Zodra je die beperking accepteert, moet de architectuur veranderen.

### Architectonische ontwerpbeslissingen

Bij het ontwerpen van de oplossing hebben we de afwegingen zorgvuldig vergeleken.

**Beslissing 1: Het proxypatroon kiezen**

We vergeleken twee benaderingen:

| Optie | Pluspunten | Nadelen | Besluit |
|------|------|------|------|
| Native WebSocket | Lichtgewicht, eenvoudig, direct doorsturen | Het verbindingsbeheer moet handmatig worden afgehandeld | Gekozen |
| SignaalR | Automatische herverbinding, krachtig typen | Te complex, extra afhankelijkheden | Afgewezen |

We hebben uiteindelijk gekozen voor native WebSocket. Om eerlijk te zijn, het was de lichtste optie en beter geschikt voor eenvoudige bidirectionele binaire stream forwarding. Het inschakelen van SignalR zou voor dit gebruik als over-engineering hebben gevoeld, en het zou extra latentie kunnen toevoegen.

**Beslissing 2: Strategie voor verbindingsbeheer**

We hebben een 'één verbinding, één sessie'-model aangenomen: elke frontend WebSocket-verbinding wordt toegewezen aan zijn eigen onafhankelijke Doubao-backend-verbinding.

De voordelen zijn eenvoudig:
- Eenvoudig te implementeren en afgestemd op het gangbare gebruikspatroon
- Gemakkelijker te debuggen en problemen op te lossen
- Goede isolatie van bronnen, waardoor interferentie tussen sessies wordt voorkomen

Simpel gezegd: de directe oplossing is soms de beste. Complexiteit maakt een ontwerp niet automatisch beter.

**Beslissing 3: Authenticatiegegevens opslaan**

Inloggegevens worden opgeslagen in backend-configuratiebestanden (`appsettings.yml` of omgevingsvariabelen) en geladen via afhankelijkheidsinjectie:

- Eenvoudig configuratiemodel dat aansluit bij bestaande backend-conventies
- Gevoelige gegevens bereiken nooit de frontend
- Ondersteunt het opzetten van meerdere omgevingen voor ontwikkeling, testen en productie

Dat niveau van scheiding is van belang. Niemand wil dat inloggegevens rondzweven op plaatsen waar ze niet mogen zijn.

### Ontwerp van gegevensstromen

De algemene gegevensstroom ziet er als volgt uit:

```
Frontend (browser)
  │
  │ ws://backend/api/voice/ws
  │ WebSocket (binary)
  ▼
Backend (proxy)
  │
  │ wss://openspeech.bytedance.com/
  │ (with authentication headers)
  ▼
Doubao API
```

De stroom zelf is niet ingewikkeld:
1. De frontend maakt via WebSocket verbinding met de backend-proxy
2. De backend-proxy ontvangt audiogegevens en maakt verbinding met de Doubao API met geverifieerde headers
3. De Doubao API retourneert herkenningsresultaten en de proxy stuurt deze door naar de frontend
4. Het hele proces blijft volledig asynchroon bij bidirectionele streaming

Zodra de verantwoordelijkheden duidelijk zijn verdeeld, wordt het ontwerp heel natuurlijk.

## Implementatie van kerncomponenten

### 1. WebSocket-eindpuntconfiguratie

```csharp
app.Map("/ws", async context =>
{
    if (context.WebSockets.IsWebSocketRequest)
    {
        // Read configuration from query parameters
        var appId = context.Request.Query["appId"];
        var accessToken = context.Request.Query["accessToken"];

        // Validate required parameters
        if (string.IsNullOrEmpty(appId) || string.IsNullOrEmpty(accessToken))
        {
            context.Response.StatusCode = 400;
            return;
        }

        // Accept the WebSocket connection
        using var webSocket = await context.WebSockets.AcceptWebSocketAsync();

        // Message handling loop
        var buffer = new byte[4096];
        while (!webSocket.CloseStatus.HasValue)
        {
            var result = await webSocket.ReceiveAsync(buffer, CancellationToken.None);

            if (result.MessageType == WebSocketMessageType.Close)
            {
                await webSocket.CloseAsync(
                    result.CloseStatus.Value,
                    result.CloseStatusDescription,
                    CancellationToken.None);
                break;
            }

            // Process audio data
            await HandleAudioDataAsync(buffer, result.Count);
        }
    }
});
```

### 2. Sessiebeheer

```csharp
public class DoubaoSessionManager : IDoubaoSessionManager
{
    private readonly ConcurrentDictionary<string, DoubaoSession> _sessions = new();

    public DoubaoSession CreateSession(string connectionId)
    {
        var session = new DoubaoSession(connectionId);
        _sessions[connectionId] = session;
        return session;
    }

    public async Task SendAudioAsync(string connectionId, byte[] audioData)
    {
        if (_sessions.TryGetValue(connectionId, out var session))
        {
            await session.SendAudioAsync(audioData);
        }
    }

    public void RemoveSession(string connectionId)
    {
        if (_sessions.TryRemove(connectionId, out var session))
        {
            session.Dispose();
        }
    }
}
```

Gebruiken `ConcurrentDictionary` voor sessiebeheer betekent dat de draadveiligheid grotendeels voor ons wordt afgehandeld. Elke inkomende verbinding krijgt zijn eigen sessie en het opschonen gebeurt automatisch wanneer de verbinding wordt verbroken.

## Frontend-integratie in de praktijk

### WebSocket-verbinding

```javascript
class DoubaoVoiceClient {
    constructor(config) {
        this.config = config;
        this.ws = null;
    }

    async connect() {
        const url = new URL(this.config.wsUrl);
        // Add query parameters
        Object.entries(this.config.params).forEach(([key, value]) => {
            url.searchParams.set(key, value);
        });

        this.ws = new WebSocket(url);

        return new Promise((resolve, reject) => {
            this.ws.onopen = () => {
                console.log('[DoubaoVoice] Connected');
                resolve();
            };

            this.ws.onmessage = (event) => {
                this._handleMessage(JSON.parse(event.data));
            };

            this.ws.onerror = reject;
        });
    }

    _handleMessage(message) {
        switch (message.type) {
            case 'status':
                this._handleStatus(message.payload);
                break;
            case 'result':
                this.onResult?.(message.payload);
                break;
            case 'error':
                console.error('[DoubaoVoice] Error:', message.payload);
                break;
        }
    }
}

// Usage example
const client = new DoubaoVoiceClient({
    wsUrl: 'ws://localhost:5000/ws',
    params: {
        appId: 'your-app-id',
        accessToken: 'your-access-token',
        sampleRate: 16000,
        bitsPerSample: 16,
        channels: 1
    }
});
```

### Audio-opname en -overdracht

Gebruiken `AudioWorkletNode` voor audioverwerking geeft betere prestaties:

```javascript
// audio-worklet.js
class AudioProcessorWorklet extends AudioWorkletProcessor {
    process(inputs, outputs, parameters) {
        const input = inputs[0]?.[0];
        if (!input) return true;

        // Convert to 16-bit PCM
        const pcm = new Int16Array(input.length);
        for (let i = 0; i < input.length; i++) {
            pcm[i] = Math.max(-32768, Math.min(32767, input[i] * 32767));
        }

        this.port.postMessage({
            type: 'audioData',
            data: pcm.buffer
        }, [pcm.buffer]);

        return true;
    }
}

registerProcessor('audio-processor', AudioProcessorWorklet);

// Main thread code
async function startAudioRecording() {
    const stream = await navigator.mediaDevices.getUserMedia({
        audio: {
            echoCancellation: true,
            noiseSuppression: true,
            autoGainControl: true,
            sampleRate: 48000
        }
    });

    const audioContext = new AudioContext();
    const audioSource = audioContext.createMediaStreamSource(stream);

    await audioContext.audioWorklet.addModule('/audio-worklet.js');
    const audioWorkletNode = new AudioWorkletNode(audioContext, 'audio-processor');

    audioWorkletNode.port.onmessage = (event) => {
        if (event.data.type === 'audioData' && ws?.readyState === WebSocket.OPEN) {
            ws.send(event.data.data); // Send binary data directly
        }
    };

    audioSource.connect(audioWorkletNode);
}
```

`AudioWorklet` presteert veel beter dan `ScriptProcessorNode` en vermijdt de problemen met het stotteren van audio die oudere verwerkingspaden vaak met zich meebrengen.

## Opmerkingen en best practices

### 1. Verbindingsbewaking

- Registreer regelmatig de sessiestatus om de volledige levenscyclus van de verbinding te volgen
- Controleer het aantal en de duur van audiosegmenten om abnormale verbindingen te detecteren
- Registreer de verbindingsstatus en het herverbindingsgedrag voor de Doubao-service

Dit zijn fundamentele operationele praktijken, maar ze maken echt een verschil in de productie.

### 2. Foutafhandeling

- Leg alle WebSocket-uitzonderingen vast en registreer ze
- Gebruik `IAsyncDisposable` om ervoor te zorgen dat de hulpbronnen worden opgeruimd
- Implementeer een correcte afsluiting van de verbinding en afhandeling van time-outs

Kortom, geef de voorkeur aan robuustheid.

### 3. Vereisten voor audioformaten

- Bemonsteringsfrequentie: 16000 Hz (aanbevolen) of 8000 Hz
- Bitdiepte: 16-bit
- Kanalen: mono
- Codering: PCM (ruw)

Als het formaat verkeerd is, kan de herkenning mislukken of kan de kwaliteit aanzienlijk achteruitgaan. Deze eisen zijn van belang.

### 4. Beveiligingsoverwegingen

- Bewaar gevoelige inloggegevens alleen in de backend-configuratie
- Dwing verbindingslimieten af om uitputting van bronnen te voorkomen
- Gebruik HTTPS/WSS in productie

Veiligheid is nooit een klein probleem.

### 5. Prestatieoptimalisatie

- Gebruik asynchrone bewerkingen om blokkering te voorkomen
- Pas de buffergroottes indien nodig aan (standaard: 4096 bytes)
- Overweeg strategieën voor het poolen van verbindingen en hergebruik

Pas deze optimalisaties toe waar ze zinvol zijn voor uw werklast.

## Aanbevelingen voor implementatie

1. **Docker-implementatie**: verpak de proxyservice als container voor eenvoudiger schalen en beheer
2. **Taakverdeling**: gebruik Nginx of Envoy als omgekeerde proxy voor WebSocket-verkeer
3. **Gezondheidscontroles**: Implementeer op hartslag gebaseerde controles om de beschikbaarheid van services te controleren
4. **Logboekaggregatie**: Stuur logs naar een gecentraliseerd logsysteem zoals ELK of Loki

Implementatie kan eenvoudig of complex zijn, afhankelijk van het team en de omgeving, dus pas dienovereenkomstig aan.

## Samenvatting

Het WebSocket-proxypatroon lost het kernprobleem op dat de WebSocket-API van de browser geen aangepaste headers ondersteunt. In het HagiCode-project bleek dit patroon zowel haalbaar als stabiel toen het van speeltuinvalidatie naar productie-implementatie ging.

Belangrijkste afhaalrestaurants:
- Een backend-proxy kan authenticatie-informatie veilig doorgeven
- Native WebSocket is lichtgewicht en efficiënt voor eenvoudige scenario's
- Het "één verbinding, één sessie"-model vereenvoudigt zowel de implementatie als het debuggen
- Het frontend-backend-protocol moet besturingssignalen scheiden van audiogegevens

Als u een functie bouwt die ook afhankelijk is van WebSocket-authenticatie, hoop ik dat dit patroon u een nuttig startpunt biedt.

Als u vragen heeft, kunt u deze gerust met ons bespreken. Technische vooruitgang gaat sneller als mensen aantekeningen vergelijken.
