## Ondersteuning voor automatische implementatie van GitHub-pagina's toegevoegd aan HagiCode

> Dit project had oorspronkelijk de codenaam PCode en is nu officieel omgedoopt tot HagiCode. Dit artikel documenteert hoe we de geautomatiseerde implementatie van statische sites hebben geïntroduceerd, zodat het publiceren van inhoud net zo eenvoudig wordt als het drinken van water.

## Achtergrond / Inleiding

Tijdens de ontwikkeling van HagiCode kwamen we een heel praktisch probleem tegen: naarmate onze documentatie en voorstellen bleven groeien, werd het efficiënt beheren en presenteren van die inhoud steeds urgenter. We besloten GitHub Pages te gebruiken om onze statische site te hosten, maar het handmatig bouwen en implementeren ervan was gewoon te omslachtig. Elke verandering vereiste een lokale build, verpakking en vervolgens een handmatige push naar de `gh-pages` tak. Dat was niet alleen inefficiënt, maar het was ook gemakkelijk om fouten te maken.

Om dit probleem op te lossen, vooral omdat we onszelf wat moeite wilden besparen, hadden we een geautomatiseerde implementatieworkflow nodig. In dit artikel wordt gedetailleerd beschreven hoe we op GitHub Actions gebaseerde ondersteuning voor automatische implementatie hebben toegevoegd aan het HagiCode-project, zodat we ons kunnen blijven concentreren op het schrijven van inhoud en de automatisering de rest kan laten doen.

## Doelanalyse

Voordat we overgaan tot implementatie, moeten we eerst duidelijk maken wat deze taak eigenlijk moet bereiken. Het slijpen van de bijl vertraagt ​​immers niet het hakken van brandhout.

### Kernvereisten

1.  **Geautomatiseerde build**: Activeer automatisch de build-workflow wanneer code naar de `main` tak.
2.  **Geautomatiseerde implementatie**: na een succesvolle build implementeert u automatisch de gegenereerde statische bestanden op GitHub-pagina's.
3.  **Consistentie van de omgeving**: Zorg ervoor dat de CI-omgeving overeenkomt met de lokale bouwomgeving om de klassieke schaamte van 'het werkt lokaal, maar mislukt online' te voorkomen.

### Technische keuze

Omdat HagiCode is gebouwd op Docusaurus, een zeer populaire statische sitegenerator van React, kunnen we GitHub Actions gebruiken om dit doel te bereiken.

## De GitHub-actiesworkflow configureren

GitHub Actions is de CI/CD-service die wordt aangeboden door GitHub. Door workflowbestanden in YAML-formaat in de repository te definiëren, kunnen we allerlei automatiseringstaken aanpassen.

### Het workflowbestand maken

We moeten een nieuw configuratiebestand maken in de `.github/workflows` map in de hoofdmap van het project, bijvoorbeeld `deploy.yml`. Als de directory nog niet bestaat, maak deze dan eerst aan.

De kernlogica van dit configuratiebestand is als volgt:

1.  **Triggervoorwaarde**: Luister naar `push` evenementen op de `main` tak.
2.  **Runtime-omgeving**: de nieuwste Ubuntu-release.
3.  **Bouwstappen**:
    -   Bekijk de code
    -   Installeer Node.js
    -   Afhankelijkheden installeren (`npm install`)
    -   Bouw de statische bestanden (`npm run build`)
4.  **Implementatiestap**: Gebruik de official `action-gh-pages` actie om de build-uitvoer naar de `gh-pages` tak.

### Sleutelconfiguratiecode

Hieronder vindt u de configuratiesjabloon die we uiteindelijk hebben overgenomen:

```yaml
name: Deploy to GitHub Pages

# Trigger condition: run when code is pushed to the main branch
on:
  push:
    branches:
      - main
    # Add path filters if needed, for example only build when docs change
    # paths:
    #   - 'docs/**'
    #   - 'package.json'

# Set permissions, which is important for GitHub Pages deployment
permissions:
  contents: read
  pages: write
  id-token: write

# Concurrency control: cancel older builds for the same branch
concurrency:
  group: "pages"
  cancel-in-progress: false

jobs:
  build:
    runs-on: ubuntu-latest
    steps:
      - name: Checkout
        uses: actions/checkout@v4
        # Note: fetch-depth: 0 is required, otherwise the build version may be inaccurate
        with:
          fetch-depth: 0

      - name: Setup Node
        uses: actions/setup-node@v4
        with:
          node-version: 20 # Keep this consistent with the local development environment
          cache: 'npm'     # Enabling cache can speed up the build process

      - name: Install dependencies
        run: npm ci
        # Use npm ci instead of npm install because it is faster and stricter, which makes it better for CI

      - name: Build website
        run: npm run build
        env:
          # Configure environment variables here if your site build requires them
          # NODE_ENV: production
          # PUBLIC_URL: /your-repo-name
          
      - name: Upload artifact
        uses: actions/upload-pages-artifact@v3
        with:
          path: ./build # Default Docusaurus output directory

  deploy:
    environment:
      name: github-pages
      url: ${{ steps.deployment.outputs.page_url }}
    runs-on: ubuntu-latest
    needs: build
    steps:
      - name: Deploy to GitHub Pages
        id: deployment
        uses: actions/deploy-pages@v4
```

## Veelvoorkomende valkuilen tijdens de implementatie

In de praktijk kwamen we verschillende problemen tegen. Ik deel ze hier in de hoop dat anderen ze kunnen vermijden, of op zijn minst vooraf oplossingen kunnen voorbereiden.

### 1. Problemen met toestemming van GitHub-token

Toen we de zaken voor het eerst instelden, mislukte de implementatie steeds met een 403 (Verboden). Na veel graafwerk ontdekten we dat dit de standaard van GitHub was `GITHUB_TOKEN` had geen toestemming om naar Pages te schrijven.

**Oplossing**: in de repository `Settings` -> `Actions` -> `General` -> `Workflow permissions`Zorg ervoor dat u **"Lees- en schrijfrechten"** selecteert.

### 2. Verkeerd pad voor buildmap

Docusaurus plaatst gebouwde statische bestanden in het `build` standaardmap. Sommige projecten kunnen echter anders worden geconfigureerd. Create React App is bijvoorbeeld standaard ingesteld op `build`, terwijl Vite dit standaard doet `dist`. Als Actions meldt dat het de uitvoerbestanden niet kan vinden, controleer dan het uitvoerpad `docusaurus.config.js`.

### 3. Problemen met subpaden

Als uw repository geen startpagina van een gebruiker is, betekent dit dat dit niet het geval is `username.github.io`, maar in plaats daarvan een projectpagina zoals `username.github.io/project-name`, dan moet u configureren `baseUrl`.

In `docusaurus.config.js`:

```javascript
module.exports = {
  // ...
  url: 'https://hagicode.com', // Your Hagicode URL
  baseUrl: '/', // Deploy from the root path
  // ...
};
```
Dit detail wordt gemakkelijk over het hoofd gezien. Als het verkeerd is geconfigureerd, wordt de pagina mogelijk als een leeg scherm geladen omdat de itempaden niet kunnen worden opgelost.

## Het resultaat verifiëren

Nadat we alles hebben geconfigureerd en de code hebben gepusht, kunnen we naar het tabblad **Acties** van de GitHub-repository gaan en de show bekijken.

U ziet een gele cirkel terwijl de workflow actief is, en zodra deze groen wordt, betekent dit succes. Als het rood wordt, klikt u erin en inspecteert u de boomstammen. In de meeste gevallen kunt u het probleem daar beneden traceren, meestal een typefout of een onjuiste padconfiguratie.

Zodra de bouw is gelukt, bezoek je `https://<your-username>.github.io/<repository-name>/` en u zult de nieuw geïmplementeerde site zien.

## Samenvatting

Door GitHub Actions te introduceren, hebben we met succes de implementatie van de HagiCode-documentatiesite geautomatiseerd. Dit bespaart niet alleen de tijd die voorheen aan handmatige handelingen werd besteed, maar, nog belangrijker, het standaardiseert de release-workflow. Ongeacht welk teamlid de documentatie bijwerkt, zodra de wijziging is samengevoegd `main`, de nieuwste inhoud zal een paar minuten later live zijn.

**Kernvoordelen**:
- **Hogere efficiëntie**: we zijn overgestapt van 'handmatig verpakken en handmatig uploaden' naar 'code staat gelijk aan vrijgave'.
- **Minder fouten**: We hebben de mogelijkheid van menselijke fouten tijdens handmatige handelingen geëlimineerd.
- **Betere ontwikkelaarservaring**: Ontwikkelaars kunnen zich meer concentreren op de kwaliteit van de inhoud in plaats van te worden afgeleid door vervelende implementatiestappen.

Hoewel het configureren van CI/CD in het begin wat lastig is, vooral met alle rechten en paddetails, is het een eenmalige investering met een groot rendement op de lange termijn. Ik raad ten zeerste aan dat elk statisch siteproject een vergelijkbare geautomatiseerde workflow gebruikt.
