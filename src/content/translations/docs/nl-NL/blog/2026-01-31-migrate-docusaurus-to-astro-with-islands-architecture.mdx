## Van Docusaurus 3.x naar Astro 5.x: een terugblik op de HagiCode-sitemigratie

> Dit artikel blikt terug op onze volledige migratie van de officiële HagiCode-website van Docusaurus 3.x naar Astro 5.x. We zullen dieper ingaan op de manier waarop Astro's Islands Architecture ons heeft geholpen prestatieknelpunten op te lossen en tegelijkertijd onze bestaande React-componenten te behouden, waardoor verbeteringen zijn bereikt in zowel de bouwsnelheid als de laadprestaties.

## Achtergrond

In januari 2026 voerden we een "harttransplantatie" uit op de officiële HagiCode-site door het kernframework volledig te migreren van Docusaurus 3.x naar Astro 5.x. Dit was geen impulsieve herschrijving, maar een zorgvuldig overwogen technische beslissing.

Vóór de migratie was onze site functioneel voltooid, maar hij begon enkele klassieke 'luxeproblemen' te vertonen: opgeblazen bouwartefacten, buitensporige JavaScript-payloads en een minder dan ideale laadsnelheid van pagina's op complexe documentatiepagina's. Als AI-coderingsassistentproject heeft HagiCode regelmatig documentatie en functie-updates nodig, zodat de bouwefficiëntie rechtstreeks van invloed is op de releasesnelheid. Tegelijkertijd wilden we dat de site zoekmachinevriendelijker (SEO) zou worden, zodat meer ontwikkelaars het project konden ontdekken.

Om deze pijnpunten op te lossen, hebben we een moedige beslissing genomen: het hele systeem opnieuw opbouwen op Astro. De impact van die beslissing kan zelfs groter zijn dan je verwacht. Ik zal binnenkort op de details ingaan.

## Over HagiCode

De aanpak voor sitemigratie die in dit artikel wordt gedeeld, komt voort uit onze praktijkervaring in de [HagiCode](https://github.com/HagiCode-org/site) project.

HagiCode is een AI-coderingsassistent gericht op het verbeteren van de ontwikkelingsefficiëntie. Het gaat ons niet alleen om het herhalen van kernfuncties, maar ook om de ontwikkelaarservaring. Deze refactor van de site was ook bedoeld om gebruikers de snelst mogelijke ervaring te bieden bij het bladeren door onze documenten en de officiële website.

## Waarom het volwassen Docusaurus-ecosysteem verlaten?

Binnen het React-ecosysteem is Docusaurus lange tijd het ‘standaardantwoord’ geweest voor documentatiesites. Het werkt out-of-the-box, biedt een rijk plug-in-ecosysteem en heeft een actieve community. Maar naarmate HagiCode meer functies kreeg, voelden we ook de beperkingen ervan:

1.  **Prestatieknelpunten**: Docusaurus is in wezen een React SPA (applicatie van één pagina). Zelfs als u alleen statische pagina's schrijft, moet de client nog steeds de React-runtime laden en de pagina hydrateren, wat onnodig zwaar is voor eenvoudige documentpagina's.
2.  **Grote itemgrootte**: zelfs als een pagina heel weinig inhoud bevat, blijft de gebundelde JS-grootte relatief vast. Dat is niet ideaal voor mobiele gebruikers of slechte netwerkomstandigheden.
3.  **Beperkte flexibiliteit**: Hoewel het uitbreidbaar is, wilden we meer controle op laag niveau over de build-pijplijn.

Astro kwam precies op het juiste moment om deze problemen op te lossen. Het introduceerde een nieuwe "Islands Architecture": Astro genereert standaard statische HTML zonder JavaScript, en alleen componenten die interactiviteit vereisen worden "geactiveerd" en laden JS. Dat betekent dat het grootste deel van onze site pure HTML wordt en extreem snel laadt.

## Kernmigratiestrategie: een soepele architecturale transitie

Migratie was niet alleen maar kopiëren en plakken. Het vereiste een mentaliteitsverandering. We zijn overgestapt van het "all React" -model van Docusaurus naar het "Core + Islands" -model van Astro.

### 1. Het configuratiesysteem opnieuw opbouwen

Eerst moesten we verhuizen `docusaurus.config.ts` aan `astro.config.mjs`. Dit was niet alleen een bestandshernoeming, maar een herschrijving van de routerings- en bouwlogica.

In Docusaurus is alles een plug-in. In Astro is alles een integratie. We moesten het basispad van de site, de uitvoermodus (statisch versus SSR) en de strategie voor activa-optimalisatie opnieuw definiëren.

**Vóór migratie:**
```typescript
// docusaurus.config.ts
export default {
  title: 'HagiCode',
  url: 'https://hagicode.com',
  baseUrl: '/',
  // ... more configuration
};
```

**Na migratie:**
```javascript
// astro.config.mjs
import { defineConfig } from 'astro/config';
import react from '@astrojs/react';

export default defineConfig({
  integrations: [react()],
  site: 'https://hagicode.com',
  base: '/',
  // Optimization settings for static assets
  build: {
    inlineStylesheets: 'auto',
  },
});
```

### 2. Wat u moet behouden en wat u moet herstructureren in React-componenten

Dit was het meest pijnlijke deel van de migratie. Onze bestaande site had veel React-componenten, zoals tabbladen, codemarkering, feedbackknoppen en meer. Ze weggooien zou verspilling zijn, maar alles bewaren zou de JavaScript-payload te zwaar maken.

HagiCode heeft een **progressieve hydratatie**-strategie aangenomen:
*   **Pure statische componenten**: voor presentatie-inhoud zoals kopteksten, voetteksten en documentatie in platte tekst hebben we deze herschreven als Astro-componenten (`.astro` bestanden) en deze tijdens het bouwen rechtstreeks naar HTML weergegeven.
*   **Interactieve eilanden**: voor componenten die interactief moeten blijven, zoals themaschakelaars, schakelen tussen tabbladen en knoppen voor het kopiëren van codeblokken, hebben we de React-implementatie behouden en toegevoegd `client:load` of `client:visible` richtlijnen.

Bijvoorbeeld onze veelgebruikte Tabs-component in de documentatie:

```jsx
// src/components/Tabs.jsx
import { useState } from 'react';
import './Tabs.css'; // Import styles

export default function Tabs({ items }) {
  const [activeIndex, setActiveIndex] = useState(0);
  // ... state logic
  return (
    <div className="tabs-wrapper">
      {/* Rendering logic */}
    </div>
  );
}
```

Bij gebruik in Markdown vertellen we Astro expliciet: "Dit onderdeel heeft JS nodig."

```markdown
// src/content/docs/example.mdx
import Tabs from '../../components/Tabs.jsx';

<!-- Load JS only when the component enters the viewport -->
<Tabs client:visible items={...} />
```

Op deze manier concurreren interactieve componenten buiten de viewport niet om bandbreedte, wat de laadsnelheid van het eerste scherm aanzienlijk verbetert.

### 3. Het stylingsysteem aanpassen: van CSS-modules tot scoped CSS

Docusaurus ondersteunt standaard CSS-modules, terwijl Astro Scoped CSS aanmoedigt via de `<style>` label. Het kernidee achter beide is stijlisolatie, maar de syntaxis is anders.

Tijdens de HagiCode-migratie hebben we de meest complexe CSS-modules omgezet naar de scoped-stijlen van Astro. Dit bleek eigenlijk maar goed ook, want in `.astro` bestanden de stijlen en sjablonen bevinden zich in hetzelfde bestand, wat het onderhoud intuïtiever maakt.

**Vóór refactoring:**
```css
/* Tabs.module.css */
.wrapper { background: var(--ifm-background-color); }
```

**Na refactoring (Astro Scoped):**
```html
<!-- Tabs.astro -->
<div class="tabs-wrapper">
  <slot />
</div>

<style>
  .tabs-wrapper {
    /* Use CSS variables directly to adapt to the theme */
    background: var(--bg-color);
    padding: 1rem;
  }
</style>
```

Tegelijkertijd hebben we het wereldwijde CSS-variabelensysteem verenigd en de omgevingsbewuste mogelijkheden van Astro gebruikt om ervoor te zorgen dat de donkere modus soepel over de pagina's schakelt.

## Valkuilen die we in de praktijk tegenkomen en hoe we ze hebben opgelost

Tijdens de daadwerkelijke HagiCode-migratie kwamen we nogal wat problemen tegen. Hier zijn enkele van de meest typische.

### 1. Pad- en omgevingsvariabele pijnpunten

HagiCode ondersteunt subpadimplementatie, zoals implementatie onder een GitHub Pages-submap. In Docusaurus, `baseUrl` wordt automatisch afgehandeld. In Astro moeten we echter voorzichtiger zijn bij het omgaan met afbeeldingslinks en API-verzoeken.

We hebben een mechanisme voor omgevingsvariabelen geïntroduceerd om dit consistent te beheren:

```javascript
// Handle paths in the build script
const getBasePath = () => import.meta.env.VITE_SITE_BASE || '/';
```

Zorg ervoor dat u paden die beginnen met `/` in je code. In ontwikkeling versus productie, of na het configureren van een `base` pad, kan dit 404's voor activa veroorzaken.

### 2. Compatibiliteit met CommonJS-scripts

Onze oude site had enkele Node.js-scripts die werden gebruikt voor taken zoals het automatisch ophalen van metrische gegevens en het bijwerken van de sitemap, en deze waren geschreven in CommonJS (`require`). Astro- en moderne bouwtools hebben ES-modules volledig omarmd (`import`/`export`).

Als u ook soortgelijke scripts heeft, vergeet dan niet om ze allemaal te herstructureren naar ES-modules. Dat is de richting waarin het ecosysteem beweegt, en hoe eerder je de verandering doorvoert, hoe minder problemen je later zult hebben.

```javascript
// Old way
const fs = require('fs');

// New way
import fs from 'fs';
```

### 3. Vergeet SEO en omleidingen niet

Zoekmachines hebben de oude Docusaurus-pagina's van HagiCode al geïndexeerd. Als u rechtstreeks naar Astro overschakelt en de URL-structuur verandert, kunt u eindigen met een groot aantal 404's en een grote daling in de zoekresultaten.

We hebben omleidingsregels geconfigureerd in Astro:

```javascript
// astro.config.mjs
export default defineConfig({
  redirects: {
    '/docs/old-path': '/docs/new-path',
    // Map old links to new links in bulk
  }
});
```

Of u kunt dit op de serverconfiguratielaag afhandelen. Zorg ervoor dat oude links kunnen worden omgeleid naar de nieuwe adressen, omdat dit van cruciaal belang is voor SEO.

## Samenvatting

Voor HagiCode was de migratie van Docusaurus naar Astro niet alleen een upgrade van het raamwerk. Het was ook een praktische implementatie van een ‘prestatie eerst’-filosofie.

**Wat we hebben gewonnen:**
*   **Uitstekende Lighthouse-scores**: Na de migratie benaderde de prestatiescore van de HagiCode-site gemakkelijk een perfecte score.
*   **Hogere bouwsnelheid**: Astro's incrementele bouwmogelijkheden halveerden de releasetijd voor documentatie-updates.
*   **Bewaarde flexibiliteit**: Met Islands Architecture hebben we geen interactieve functies opgeofferd en konden we React nog steeds gebruiken waar dat nodig was.

Als je ook een documentatiegerichte site onderhoudt en worstelt met de bundelgrootte of laadsnelheid, is Astro de moeite van het proberen waard. Hoewel het migratieproces enige operatie vereist, zoals het hernoemen van PCode naar HagiCode en het één voor één verplaatsen van componenten, maakt de zijdezachte gebruikerservaring die u ervoor terugkrijgt het absoluut de moeite waard.

Het bouwsysteem dat in dit artikel wordt gedeeld, is de exacte aanpak die we met vallen en opstaan hebben ontwikkeld tijdens het bouwen van HagiCode. Als je deze aanpak waardevol vindt, zegt dat iets over onze technische kracht, en HagiCode zelf is waarschijnlijk ook de moeite van het bekijken waard.
