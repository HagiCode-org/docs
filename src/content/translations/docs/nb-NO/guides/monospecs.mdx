## Oversikt

Monospecs er HagiCodes multi-repository management tilnærming. Den sporer og administrerer underlager gjennom `monospecs.yaml`, og det fungerer for ethvert prosjekt som trenger ett sted for å koordinere flere uavhengige depoter.

### Kjernekonsepter

**Konfigurasjonsfil**

`monospecs.yaml` er kjernen i hele administrasjonsløsningen, og definerer:
- Liste og plassering av underdepoter
- Git URL for hvert depot
- Visningsnavn og ikon
- Integrasjonsatferd med OpenSpec

**Forskjeller fra vanlige depoter**

| Funksjon | Vanlig depot | Monospecs Managed Project |
|----------|-------------------|---------------------------|
| **OpenSpec-plassering** | Rotkataloger `openspec/` | Hovedlagerets rotkataloger `openspec/` |
| **Endre omfang** | Begrenset til nåværende depot | Kan involvere flere underlager |
| **Renhet under depot** | Spesifikasjoner blandet med kode | Spesifikasjoner etablert i hovedlageret, underlager er renere |
| **Arkiv automatisk forpliktelse** | Trenger manuelle spesifikasjoner | `commit_when_archive: true` forplikter automatisk spesifikasjoner til hovedlageret |

### Hvorfor bruke monospecs?

- **Unified Management**: Administrer alle underlager i én konfigurasjonsfil
- **Automasjonsstøtte**: Gir lagermetadata for skript og verktøy
- **AI-vennlighet**: Strukturert konfigurasjon hjelper AI-agenten med å forstå repositoryrelasjoner
- **OpenSpec-integrasjon**: Sømløs integrasjon med arbeidsflyten for endringsadministrasjon
- **Renere underdepoter**: Spesifikasjonene holdes i hovedrepoen, underdepotene inneholder kun faktisk kode

### Gjeldende scenarier

Bruk Monospecs først i scenarier som disse:

**1. Multi-Repository Collaborative Development**
Når et prosjekt er delt opp i flere uavhengige depoter:
- Frontend-applikasjon, backend-tjeneste og desktop-app utviklet separat
- Trenger å koordinere versjoner og avhengigheter på tvers av depoter
- Dokumentasjon og hjelpeprosjekter atskilt fra hovedkode

**2. Hjelpeprosjektledelse**
Bruk Monospecs for å administrere:
- Dokumentasjonsprosjekt
- Offisiell nettside
- Bygg verktøy og skript
- Hvert depot opprettholder sin egen versjonskontroll

**3. Funksjonsutvikling på tvers av arkiv**
En funksjon som krever endringer på tvers av flere depoter:
- Trenger å spore endringer i hvilke depoter
- Forplikter automatisk spesifikasjoner til riktig mållager
- Koordinere versjonsoppdateringer og avhengigheter

## Rask start

### Initialisering

Følg disse trinnene for å fullføre initialiseringen av monospecs:

#### 1. Opprett et tomt git-lager

Lag en tom `git` mappe i hovedlageret ditt (merk: dette er `git`, ikke `repos/` - dette er monospecs-konvensjonen):

```bash
mkdir git
cd git
git init
cd ..
```

#### 2. Konfigurer monospecs-fil

Opprett `monospecs.yaml` konfigurasjonsfilen i hoveddepotets rotkatalog:

```yaml
version: "1.0"
commit_when_archive: true

repositories:
  - path: "repos/frontend"
    url: "https://github.com/your-org/frontend.git"
    displayName: "Frontend App"
    icon: "🌐"

  - path: "repos/backend"
    url: "https://github.com/your-org/backend.git"
    displayName: "Backend Service"
    icon: "⚙️"
```

**Konfigurasjonselementer**:
| Konfigurasjonselement | Type | Obligatorisk | Beskrivelse |
|----------------|------|----------|-------------|
| `version` | streng | Ja | Konfigurasjonsfilversjon |
| `commit_when_archive` | boolsk | Nei | Hvorvidt spesifikasjoner skal foretas automatisk ved arkivering (ikke forplikte sub-repo-kode) |
| `repositories` | array | Ja | Depotkonfigurasjonsliste |
| `repositories[].path` | streng | Ja | Depotbane (i forhold til prosjektrot) |
| `repositories[].url` | streng | Ja | Git repository URL |
| `repositories[].displayName` | streng | Nei | Visningsnavn for depot |
| `repositories[].icon` | streng | Nei | Lagringsikon (emoji) |

### Legg til underlager til hovedlageret

Legg til alle underlager til `repos/` katalog. Denne katalogen må **ekskluderes fra hovedlagerets Git-versjonskontroll** for å unngå sporing av innhold fra underlager:

```plaintext
# Exclude all sub-repositories
repos/
```

#### 3. Legg til Parent Repository til HagiCode

Hvis du bruker HagiCode-skrivebordsapplikasjonen, åpner du appen og legger til hovedlageret ditt som et prosjekt. Klikk på "Legg til prosjekt" eller "+"-knappen på startskjermen, velg hoveddepotmappen din, og du vil se alle underlagre og deres statusinformasjon.

#### 4. Klone underlager

Bruk `monospecs.yaml` konfigurasjon for å klone alle konfigurerte underlager til `repos/` katalog:

Dette vil klone alle underlager basert på URL-ene spesifisert i konfigurasjonsfilen.

## Konfigurasjonsfilstruktur

### Konfigurasjon av rotnivå

```yaml
version: "1.0"
commit_when_archive: true

repositories:
  # ... repository configurations
```

### Lagringsnivåkonfigurasjon

Hvert dellager kan ha:
- OpenSpec endre mappe (`openspec/`)
- Uavhengig `.git/` katalog
- Valgfritt `AGENTS.md` for AI Agent-konfigurasjon

## Lagerstyringsoperasjoner

### Legg til nytt depot

Når du legger til et nytt underlager til `monospecs.yaml` `repositories` array:

```yaml
repositories:
  - path: "repos/new-service"
    url: "https://github.com/HagiCode-org/new-service.git"
    displayName: "New Service"
    icon: "🆕"
```

### Oppdater depotkonfigurasjon

Når depot-URL-er eller metadata endres:
1. Rediger `monospecs.yaml` for å oppdatere tilsvarende oppføringer
2. Kontroller at YAML-syntaksen er riktig
3. Hvis du synkroniserer endringer, må du manuelt oppdatere konfigurasjonen for det lokale depotet

## Støtte for AI Agent-konfigurasjon

### AGENTER.md

Den `AGENTS.md` filen gir depotspesifikk veiledning for AI-agenten:

**Teknisk stabelinformasjon**
- Ramme og bygge verktøy
- Kodekonvensjoner og navnepraksis
- Versjonsadministrasjon og distribusjonskonfigurasjon

**Prosjektspesifikk atferd**
- Konfigurasjonsutvidelser og spesielle krav
- OpenSpec-integrasjonsmønstre
- Koordineringsbehov på tvers av depoter

### AI-fordeler

Bruker `AGENTS.md` gjør det mulig for AI-agenten å:
- Forstå depotrelasjoner og avhengigheter
- Generer kode og refaktorer på tvers av flere depoter
- Oppretthold konsistens i versjonskontroll og konfigurasjoner

## OpenSpec-integrasjon

### Hvordan Monospecs fungerer med OpenSpec

Enten du administrerer et monospecs-administrert prosjekt eller et vanlig depot, kan du bruke HagiCodes OpenSpec-arbeidsflyt:

**Vanlig depot:**
- OpenSpec i root `openspec/` katalog
- Endringer påvirker kun gjeldende depot

**Monospecs Managed Project:**
- OpenSpec leser `monospecs.yaml` konfigurasjon
- Endringer spores av underlager
- Spesifikasjoner automatisk forpliktet til å rette mål

### Arkivatferd

Når `commit_when_archive: true`:
- Spesifikasjoner blir automatisk forpliktet til hovedlageret
- Koden til underdepotene er ikke forpliktet (forblir atskilt)
- Forenkler administrasjon av spesifikasjoner - ingen manuelle spesifikasjonsbekreftelser er nødvendige for underlager

### Manuelle forpliktelser for underlager

Underlagre opprettholder uavhengig versjonskontroll:
- Foreta faktiske kodeendringer til sine egne depoter
- Spesifikasjoner forpliktes separat (fra hovedlageret ved arkivering)

## Beste praksis

### Navnekonvensjoner for depoter

- Bruk kebab-store bokstaver (små bokstaver og bindestreker) for depotstier
- Bruk enkle, beskrivende visningsnavn på kinesisk
- Bruk relevante emojier som ikoner

### Retningslinjer for visningsnavn

- Hold det kortfattet og beskrivende
- Bruk konsistent navn på tvers av lignende depoter
- Velg ikoner som representerer depotets formål

### Konfigurasjonsvedlikehold

**Synkroniser konfigurasjon med faktisk status**
- Kontroller regelmessig `monospecs.yaml` reflekterer faktisk depotstruktur
- Oppdater eller fjern oppføringer når depoter legges til eller fjernes
- Test endringer for å sikre konfigurasjonsgyldighet

**Versjonskontroll for konfigurasjon**
- Spor konfigurasjonsfilendringer med git
- Dokumenter årsaker til store konfigurasjonsendringer
- Tag-konfigurasjonsfilversjoner for enkel tilbakeføring

## Feilsøking

### Finner ikke konfigurasjonsfil

Hvis `monospecs.yaml` filen er ikke funnet i prosjektroten:
1. Sjekk om filen finnes på riktig plassering
2. Bekreft at du er i riktig arbeidskatalog
3. Se etter skrivefeil i konfigurasjonsfilnavnet

### YAML-syntaksfeil

Vanlige YAML-syntaksproblemer:
- Feil innrykk (bruk mellomrom, ikke tabulatorer)
- Manglende eller ekstra anførselstegn rundt strenger
- Ugyldige datatyper (strenger trenger anførselstegn, tall ikke)
- Mangler obligatoriske felt (som `path` eller `url`)

### Depot ikke oppdaget

Hvis et nylig lagt til depot ikke vises:
1. Sjekk `path` er riktig i `monospecs.yaml`
2. Bekreft at depotet ble klonet
3. Sjekk om depotet finnes på forventet plassering

### Klonefeil

Hvis depotkloning mislykkes:
1. Sjekk nettverkstilkoblingen
2. Bekreft at nettadressen til Git-depotet er riktig
3. Se etter autentiseringsproblemer
4. Gjennomgå Docker-beholderstatus
5. Sjekk tilgjengeligheten av diskplass
