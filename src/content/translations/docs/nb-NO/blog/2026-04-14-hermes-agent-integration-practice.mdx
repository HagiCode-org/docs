## Hermes Agent Integration Practice: Fra protokoll til produksjon

> Dele den komplette HagiCode-opplevelsen med å integrere Hermes Agent, inkludert kjerneleksjoner rundt ACP-protokolltilpasning, administrasjon av øktpooler og synkronisering av front-end/back-end kontrakt.

## Bakgrunn

Mens vi bygde HagiCode, en AI-assistert kodeplattform, trengte teamet vårt å integrere et Agent-rammeverk som kunne kjøres lokalt og også skaleres til skyen. Etter forskning ble Hermes Agent fra Nous Research valgt som den underliggende motoren for våre generelle agentfunksjoner.

I sannhet er teknologivalg verken spesielt vanskelig eller spesielt enkelt. Det er mange sterke Agent-rammeverk på markedet, men Hermes skilte seg ut fordi ACP-protokollen og verktøysystemet passet spesielt godt til HagiCodes krevende krav: lokal utvikling, teamsamarbeid og skyutvidelse. Likevel, å bringe Hermes inn i et ekte produksjonssystem betydde å løse en lang rekke tekniske problemer. Denne delen var alt annet enn triviell.

HagiCodes stack bruker Orleans til å bygge et distribuert system, mens frontend er bygget med React + TypeScript. Å integrere Hermes innebar å bevare arkitektonisk konsistens samtidig som Hermes ble en førsteklasses eksekutør sammen med ClaudeCode og OpenCode. Det høres enkelt nok ut, men implementeringen forteller alltid den virkelige historien.

Denne artikkelen deler vår praktiske erfaring med å integrere Hermes Agent i HagiCode, og vi håper den tilbyr nyttig referansemateriale for team som har lignende behov. Tross alt, når du først har falt i en grop, er det ingen grunn til å la noen andre falle i den samme.

## Om HagiCode

Løsningen beskrevet i denne artikkelen kommer fra vårt praktiske arbeid i [HagiCode](https://hagicode.com) prosjekt. HagiCode er en AI-drevet kodehjelpsplattform som støtter enhetlig tilgang til og administrasjon av flere AI-leverandører. Under Hermes Agent-integrasjonen designet vi et generisk Provider-abstraksjonslag slik at nye Agenttyper sømløst kunne kobles til det eksisterende systemet.

Hvis du er interessert i HagiCode, besøk gjerne [GitHub](https://github.com/HagiCode-org/site) for å lære mer. Jo flere som tar hensyn, jo sterkere momentum.

## Arkitektur design

### Lagdelt designtilnærming

HagiCodes Hermes-integrasjon bruker en tydelig lagdelt arkitektur, med hvert lag fokusert på sitt eget ansvar:

**Back-end kjernelag**
- `HermesCliProvider`: implementerer `IAIProvider` grensesnitt som det enhetlige AI-leverandørens inngangspunkt
- `HermesPlatformConfiguration`: administrerer Hermes kjørbar bane, argumenter, autentisering og relaterte innstillinger
- `ICliProvider<HermesOptions>`: lavnivå CLI-abstraksjonen levert av HagiCode.Libs for håndtering av underprosesslivssykluser

**Transportlag**
- `StdioAcpTransport`: kommuniserer med Hermes ACP-delprosessen gjennom standard input og output
- ACP-protokollmetoder: `initialize`, `authenticate`, `session/new`, `session/prompt`

**Kjøretidslag**
- `HermesGrain`: Orleans Grain-implementering som håndterer distribuert øktkjøring
- `CliAcpSessionPool`: sesjonspool som gjenbruker ACP-underprosesser for å unngå hyppige oppstartskostnader

**Frontend-lag**
- `ExecutorAvatar`: Hermes visuelle identitet og ikon
- `executorTypeAdapter`: Tilordningslogikk av leverandørtype
- SignalR sanntidsmeldinger: opprettholder Hermes identitetskonsistens gjennom hele meldingsstrømmen

Denne lagdelte designen lar hvert lag utvikle seg uavhengig. Hvis vi for eksempel ønsker å legge til en ny transportmekanisme i fremtiden, for eksempel WebSocket, trenger vi bare å modifisere transportlaget. Det er ikke nødvendig å snu hele systemet bare fordi én transport endres.

### Unified interface abstraksjon

Alle AI-leverandører implementerer `IAIProvider` grensesnitt, som er et av kjernedesignvalgene i HagiCodes arkitektur:

```csharp
public interface IAIProvider
{
    string Name { get; }
    ProviderCapabilities Capabilities { get; }

    IAsyncEnumerable<AIStreamingChunk> StreamAsync(
        AIRequest request,
        CancellationToken cancellationToken = default);

    Task<AIResponse> ExecuteAsync(
        AIRequest request,
        CancellationToken cancellationToken = default);
}
```

`HermesCliProvider` implementerer dette grensesnittet og står på lik linje med `ClaudeCodeProvider`, `OpenCodeProvider`, og andre. Fordelene med dette designet inkluderer:

1. **Utskiftbarhet**: Bytting av leverandør påvirker ikke forretningslogikken i det øvre laget
2. **Testbarhet**: Leverandører kan lett bli hånet for enhetstesting
3. **Utvidbarhet**: å legge til en ny leverandør krever bare implementering av grensesnittet

Til slutt er grensesnitt mye som regler. Når reglene er på plass, kan alle sameksistere harmonisk, spille etter sine styrker og unngå å tråkke på hverandre. Det er en viss eleganse i det.

## Kjerneimplementering

### Implementering av leverandørlag

`HermesCliProvider` er kjernen i hele integreringen. Den koordinerer de ulike komponentene som trengs for å fullføre en enkelt AI-påkalling:

```csharp
public sealed class HermesCliProvider : IAIProvider, IVersionedAIProvider
{
    private readonly ICliProvider<LibsHermesOptions> _provider;
    private readonly ConcurrentDictionary<string, string> _sessionBindings;

    public ProviderCapabilities Capabilities { get; } = new()
    {
        SupportsStreaming = true,
        SupportsTools = true,
        SupportsSystemMessages = true,
        SupportsArtifacts = false
    };

    public async IAsyncEnumerable<AIStreamingChunk> StreamAsync(
        AIRequest request,
        [EnumeratorCancellation] CancellationToken cancellationToken = default)
    {
        // 1. Resolve the session binding key
        var bindingKey = ResolveBindingKey(request.CessionId);

        // 2. Get or create a Hermes session through the session pool
        var options = new HermesOptions
        {
            ExecutablePath = _platformConfiguration.ExecutablePath,
            Arguments = _platformConfiguration.Arguments,
            SessionId = _sessionBindings.TryGetValue(bindingKey, out var sessionId) ? sessionId : null,
            WorkingDirectory = request.WorkingDirectory,
            Model = request.Model
        };

        // 3. Execute and collect the streaming response
        await foreach (var message in _provider.ExecuteAsync(options, request.Prompt, cancellationToken))
        {
            // 4. Map ACP messages to AIStreamingChunk
            if (_responseMapper.TryConvertToStreamingChunk(message, out var chunk))
            {
                yield return chunk;
            }
        }
    }
}
```

Flere designpunkter er spesielt viktige her:

1. **Sesjonsbinding**: bruker `CessionId` å binde flere forespørsler til den samme Hermes-delprosessen, og bevare kontekstkontinuitet på tvers av samtaler med flere svinger
2. **Responsmapping**: konverterer Hermes ACP-meldingsformat til det enhetlige `AIStreamingChunk` format
3. **Streamingstøtte**: bruker `IAsyncEnumerable` for å støtte ekte strømmesvar

Sesjonsbinding er litt som menneskelige relasjoner. Når en forbindelse er etablert, har fremtidig kommunikasjon kontekst, så du trenger ikke starte fra null hver gang. Selvfølgelig må det forholdet fortsatt opprettholdes.

### ACP-protokolltilpasning

Hermes bruker ACP (Agent Communication Protocol), som skiller seg fra en tradisjonell HTTP API. ACP er en protokoll basert på standard input og output, og den har flere egenskaper:

1. **Oppstartsmarkør**: etter at Hermes-prosessen starter, sender den ut `//ready` markør
2. **Dynamisk autentisering**: autentiseringsmetoder er ikke faste og må forhandles gjennom protokollen
3. **gjenbruk av økter**: etablerte økter gjenbrukes gjennom `SessionId`
4. **Fragmenterte svar**: et komplett svar kan deles på flere `session/update` varsler

HagiCode håndterer disse egenskapene gjennom `StdioAcpTransport`:

```csharp
public class StdioAcpTransport
{
    public async Task InitializeAsync(CancellationToken cancellationToken)
    {
        // Wait for the //ready marker
        var readyLine = await _outputReader.ReadLineAsync(cancellationToken);
        if (readyLine != "//ready")
        {
            throw new InvalidOperationException("Hermes did not send ready signal");
        }

        // Send the initialize request
        await SendRequestAsync(new
        {
            jsonrpc = "2.0",
            id = 1,
            method = "initialize",
            @params = new
            {
                protocolVersion = "2024-11-05",
                capabilities = new { },
                clientInfo = new { name = "HagiCode", version = "1.0.0" }
            }
        }, cancellationToken);
    }
}
```

Protokoller er litt som gjensidig forståelse mellom mennesker. Når den forståelsen er der, flyter kommunikasjonen mye mer jevnt. Å bygge det tar bare tid.

### Ledelse av øktbasseng

Å starte Hermes-underprosesser ofte er dyrt, så vi implementerte en session pool-mekanisme:

```csharp
services.AddSingleton(static _ =>
{
    var registry = new CliProviderPoolConfigurationRegistry();
    registry.Register("hermes", new CliPoolSettings
    {
        MaxActiveSessions = 50,
        IdleTimeout = TimeSpan.FromMinutes(10)
    });
    return registry;
});
```

Viktige sesjonspoolparametere:
- `MaxActiveSessions`: kontrollerer samtidighetsgrensen for å unngå utmattende ressurser
- `IdleTimeout`: inaktiv timeout som balanserer oppstartskostnad mot minnebruk

I praksis fant vi at:
1. Hvis tidsavbruddet for inaktivitet er for kort, starter øktene på nytt ofte; hvis den er for lang, forblir minnet opptatt
2. Samtidighetsgrensen må justeres i henhold til faktisk belastning, fordi å sette den for høyt kan gjøre systemet tregt
3. Utnyttelsen av øktpoolen trenger overvåking slik at parametere kan justeres i tide

Dette er omtrent som mange valg i livet: å være for aggressiv skaper problemer, mens å være for konservativ går glipp av muligheter. Målet er rett og slett å finne den rette balansen.

## Front-end integrasjon

### Type kartlegging

Frontenden må identifisere Hermes-leverandøren på riktig måte og vise de tilsvarende visuelle elementene:

```typescript
// executorTypeAdapter.ts
export const resolveExecutorVisualTypeFromProviderType = (
  providerType: PCode_Models_AIProviderType | null | undefined

  switch (providerType) {
    case PCode_Models_AIProviderType.HERMES_CLI:
      return 'Hermes';
    default:
      return 'Unknown';
  }
};
```

Tross alt fortjener vakre ting vakker presentasjon. Å sørge for at skjønnhet faktisk er synlig avhenger fortsatt av front-end håndverk.

### Kontraktssynkronisering

Front- og bakenden holder kontrakten på linje gjennom OpenAPI-generering. Bakenden definerer `AIProviderType` enum:

```csharp
public enum AIProviderType
{
    Unknown,
    ClaudeCode,
    OpenCode,
    HermesCli  // Newly added
}
```

Grensesnittet genererer den tilsvarende TypeScript-typen gjennom OpenAPI, og sikrer at enum-verdiene forblir konsistente. Dette er nøkkelen til å unngå at frontenden vises `Unknown`.

En kontrakt er mye som et løfte. Når det er avtalt, må det respekteres, ellers havner du i vanskelige situasjoner som `Unknown`.

Hermes-konfigurasjonen administreres gjennom `appsettings.json`:

```json
{
  "Providers": {
    "HermesCli": {
      "ExecutablePath": "hermes",
      "Arguments": "acp",
      "StartupTimeoutMs": 10000,
      "ClientName": "HagiCode",
      "Authentication": {
        "PreferredMethodId": "api-key",
        "MethodInfo": {
          "api-key": "your-api-key-here"
        }
      },
      "SessionDefaults": {
        "Model": "claude-sonnet-4-20250514",
        "ModeId": "default"
      }
    }
  }
}
```

Denne konfigurasjonsdrevne designen gir fleksibilitet:
- kjørbare baner kan overstyres, noe som er praktisk for utvikling og testing
- oppstartsargumenter kan tilpasses for å matche forskjellige Hermes-versjoner
- autentiseringsinformasjon kan konfigureres til å støtte flere autentiseringsmetoder

Konfigurasjon er litt som flervalgsspørsmål i livet. Hvis nok alternativer er tilgjengelige, er det vanligvis en som passer. Når det er sagt, kan for mange alternativer skape sin egen beslutningstrøtthet.

## Praktisk erfaring

### Helsesjekker

Å bygge en pålitelig leverandør krever omfattende helsesjekker:

```csharp
public async Task<ProviderTestResult> PingAsync(CancellationToken cancellationToken = default)
{
    var response = await ExecuteAsync(new AIRequest
    {
        Prompt = "Reply with exactly PONG.",
        CessionId = null,
        AllowedTools = Array.Empty<string>(),
        WorkingDirectory = ResolveWorkingDirectory(null)
    }, cancellationToken);

    var success = string.Equals(response.Content.Trim(), "PONG", StringComparison.OrdinalIgnoreCase);
    return new ProviderTestResult
    {
        ProviderName = Name,
        Success = success,
        ResponseTimeMs = stopwatch.ElapsedMilliseconds,
        ErrorMessage = success ? null : $"Unexpected Hermes ping response: '{response.Content}'."
    };
}
```

Punkter å se i helsesjekker:
1. Bruk enkle testtilfeller og unngå altfor komplekse scenarier
2. Angi rimelige tidsavbruddsverdier
3. Registrer responstid for å støtte ytelsesanalyse

Akkurat som folk trenger fysiske kontroller, trenger systemene også helsesjekker. Jo raskere problemer blir funnet, desto lettere er de å fikse.

### Valideringsverktøy

HagiCode tilbyr en dedikert konsoll for å validere Hermes-integrasjon:

```bash
# Basic validation
HagiCode.Libs.Hermes.Console --test-provider

# Full suite (including repository analysis)
HagiCode.Libs.Hermes.Console --test-provider-full --repo .

# Custom executable

```

Dette verktøyet er ekstremt nyttig under utvikling fordi det lar oss raskt verifisere om integrasjonen er riktig. Tross alt, ingen ønsker å vente til et problem dukker opp før de husker å teste.

### Håndtering av vanlige problemer

**Autentiseringsfeil**
- Sjekk om `Authentication.PreferredMethodId` samsvarer med autentiseringsmetoden Hermes faktisk støtter
- Bekreft at autentiseringsinformasjonsformatet er riktig, for eksempel API Key eller Bearer Token

**Tidsavbrudd for økten**
- Øk `StartupTimeoutMs` verdi
- Sjekk MCP-serverens tilgjengelighet
- Gjennomgå systemressursutnyttelsen

**Ufullstendig svar**
- Sørg for `session/update` varsler og det endelige resultatet er aggregert riktig
- Sjekk kanselleringslogikk i strømmehåndtering
- Bekreft at feilhåndteringen er fullført

**Frontend viser Ukjent**
- Bekreft at OpenAPI-genereringen allerede inkluderer `HermesCli` enum verdi
- Sjekk om typetilordning er riktig
- Tøm nettleserbufferen og regenerer typer

Problemer vil alltid eksistere. Når de dukker opp, er det viktig å ikke få panikk. Spor årsaken trinn for trinn, og til slutt kan de fleste av dem løses.

### Forslag til ytelsesoptimalisering

1. **Bruk øktpoolen**: gjenbruk ACP-underprosesser for å redusere oppstartskostnader
2. **Angi tidsavbrudd på riktig måte**: balanser minnebruk mot oppstartskostnad
3. **Gjenbruk økt-ID-er**: bruk det samme `CessionId` for batchoppgaver
4. **Konfigurer MCP på forespørsel**: unngå unødvendige verktøyanrop

Ytelse er mye som effektivitet i dagliglivet. Når du får det til, oppnår du mer med mindre; når du tar feil, multipliseres innsatsen mens resultatene krymper. Å finne det "riktige" punktet krever både erfaring og flaks.

## Konklusjon

Å integrere Hermes Agent i et produksjonssystem krever å vurdere problemer på tvers av flere dimensjoner:

1. **Arkitektur**: utform et enhetlig leverandørgrensesnitt og implementer en utskiftbar komponentarkitektur
2. **Protokoll**: håndtere ACP-spesifikk oppførsel på riktig måte som oppstartsmarkører og dynamisk autentisering
3. **Ytelse**: gjenbruk ressurser gjennom øktpoolen og balanser oppstartskostnad mot minnebruk
4. **Frontend**: sikre kontraktssynkronisering og gi en konsistent visuell opplevelse

HagiCodes erfaring viser at med god lagdelt design og konfigurasjonsdrevet implementering kan et komplekst Agent-system integreres sømløst i en eksisterende arkitektur.

Disse prinsippene høres enkle ut når de beskrives i ord, men faktisk implementering introduserer alltid mange forskjellige typer problemer. Det er greit. Hvis et problem blir løst, blir det erfaring. Hvis det ikke gjør det, blir det en leksjon. Uansett har det fortsatt verdi.

Vakre ting eller mennesker trenger ikke å være besatt; så lenge de forblir vakre, er det nok å stille pris på den skjønnheten. Teknologi er mye det samme. Hvis det bidrar til å gjøre systemet bedre, betyr det spesifikke rammeverket eller protokollen langt mindre enn folk noen ganger tror.
