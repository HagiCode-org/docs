## Veiledning for implementering av hotword-støtte for Doubao talegjenkjenning

> Denne artikkelen forklarer i detalj hvordan du implementerer kommandoordstøtte for Doubao-talegjenkjenning i HagiCode-prosjektet. Ved å bruke både tilpassede kommandoord og plattformordordtabeller kan du forbedre gjenkjenningsnøyaktigheten betydelig for domenespesifikke ordforrådene.

## Bakgrunn

Talegjenkjenningsteknologi har utviklet seg i mange år, men ett problem har konsekvent plaget utviklere. Generelle talegjenkjenningsmodeller kan dekke dagligdags språk, men de kommer ofte til kort når det kommer til faglig terminologi, produktnavn og personnavn. Tenk på det: en stemmeassistent i det medisinske feltet må gjenkjenne begreper som "hypertensjon", "diabetes" og "koronar hjertesykdom" nøyaktig; et rettssystem trenger å fange opp begreper som «saksak», «forsvar» og «bevisbyrde». I disse scenariene prøver en generell modell sitt beste, men det er ofte ikke nok.

Vi møtte den samme utfordringen i HagiCode-prosjektet. Som en multifunksjonell AI-kodingsassistent må HagiCode håndtere talegjenkjenning for et bredt spekter av teknisk terminologi. Doubao talegjenkjennings-API, i sin standardkonfigurasjon, kunne imidlertid ikke fullt ut oppfylle våre nøyaktighetskrav for spesialiserte termer. Det er ikke det at Doubao ikke er god nok; snarere har hvert domene sitt eget terminologisystem. Etter litt forskning og teknisk utforskning fant vi ut at Doubao talegjenkjennings-API faktisk gir kommandoordstøtte. Med en enkel konfigurasjon kan den forbedre gjenkjenningsnøyaktigheten til et spesifikt vokabular betydelig. På en måte, når du forteller det hvilke ord du skal ta hensyn til, lytter det mer nøye etter dem.

Det denne artikkelen deler er den komplette løsningen vi brukte i HagiCode-prosjektet for å implementere Doubao talegjenkjenningskommandoer. Begge modusene, egendefinerte kommandoord og plattformordordtabeller, er tilgjengelige, og de kan også kombineres. Med denne løsningen kan utviklere fleksibelt konfigurere kommandoord basert på forretningsscenarier, slik at talegjenkjenningssystemet bedre kan "gjenkjenne" profesjonelt, uvanlig, men likevel kritisk vokabular.

## Kjerneimplementering

### Forstå de to hotword-modusene

Doubao talegjenkjennings-API gir to måter å konfigurere kommandoord på, og hver av dem har sine egne ideelle brukstilfeller og fordeler.

**Egendefinert kommandoordmodus** lar oss sende kommandoordtekst direkte gjennom `corpus.context` felt. Denne tilnærmingen er spesielt egnet for scenarier der du raskt må konfigurere et lite antall kommandoord, for eksempel midlertidig gjenkjenne et produktnavn eller en persons navn. I HagiCodes implementering analyserer vi flerlinjers hotword-teksten som er skrevet inn av brukeren til en liste med strenger, og formaterer den deretter til `context_data` array kreves av Doubao API. Denne tilnærmingen er veldig direkte: du forteller ganske enkelt systemet hvilke ord du skal være oppmerksom på, og det gjør akkurat det.

**Plattform hotword-tabellmodus** bruker `corpus.boosting_table_id` feltet for å referere til en forhåndskonfigurert kommandoordtabell i Doubao selvlæringsplattform. Denne tilnærmingen er egnet for scenarier der du trenger å administrere et stort antall kommandoord. Vi kan opprette og vedlikeholde kommandoordtabeller på Doubao selvlæringsplattform, og deretter referere dem med ID. For et prosjekt som HagiCode, hvor spesialiserte termer må oppdateres og vedlikeholdes kontinuerlig, tilbyr denne modusen mye bedre administrasjon. Når antallet kommandoord vokser, er det langt bedre å ha et sentralisert sted å administrere dem enn å legge dem inn manuelt hver gang.

Interessant nok kan disse to modusene også brukes sammen. Doubao API støtter å inkludere både egendefinerte hotwords og en plattform hotword-tabell-ID i samme forespørsel, med kombinasjonsstrategien kontrollert av `combine_mode` parameter. Denne fleksibiliteten gjør at HagiCode kan håndtere et bredt spekter av komplekse behov for profesjonell terminologigjenkjenning. Noen ganger gir det bedre resultater å kombinere flere tilnærminger.

### Grensesnitttypedefinisjoner og validering

I HagiCodes frontend-implementering definerte vi et komplett sett med kommandoordkonfigurasjonstyper og valideringslogikk. Den første delen er typedefinisjonen:

```typescript
export interface HotwordConfig {
  contextText: string;           // Multi-line hotword text
  boostingTableId: string;      // Doubao platform hotword table ID
  combineMode: boolean;          // Whether to use both together
}
```

Dette enkle grensesnittet inneholder alle konfigurasjonselementer for kommandoord-funksjonen. Blant dem, `contextText` er den delen brukerne samhandler mest direkte med: vi tillater brukere å skrive inn én hotword-frase per linje, noe som er veldig intuitivt. Å be brukere om å skrive inn ett begrep per linje er mye enklere enn å få dem til å forstå et komplisert konfigurasjonsformat.

Deretter kommer valideringsfunksjonen. Basert på Doubao API-begrensningene, definerte vi strenge valideringsregler: maksimalt 100 linjer med kommandoordtekst, opptil 50 tegn per linje, og ikke mer enn 5000 tegn totalt; `boosting_table_id` kan være på maksimalt 200 tegn og kan bare inneholde bokstaver, tall, understreker og bindestreker. Disse grensene er ikke vilkårlige; de kommer direkte fra den offisielle Doubao-dokumentasjonen. API-grenser er API-grenser, og vi må følge dem.

```typescript
export function validateContextText(contextText: string): HotwordValidationResult {
  if (!contextText || contextText.trim().length === 0) {
    return { isValid: true, errors: [] };
  }

  const lines = contextText.split('\n').filter(line => line.trim().length > 0);
  const errors: string[] = [];

  if (lines.length > 100) {
    errors.push(`Hotword line count cannot exceed 100 lines; current count is ${lines.length}`);
  }

  const totalChars = contextText.length;
  if (totalChars > 5000) {
    errors.push(`Total hotword character count cannot exceed 5000; current count is ${totalChars}`);
  }

  for (let i = 0; i < lines.length; i++) {
    if (lines[i].length > 50) {
      errors.push(`Hotword on line ${i + 1} exceeds the 50-character limit`);
    }
  }

  return { isValid: errors.length === 0, errors };
}

export function validateBoostingTableId(boostingTableId: string): HotwordValidationResult {
  if (!boostingTableId || boostingTableId.trim().length === 0) {
    return { isValid: true, errors: [] };
  }

  const errors: string[] = [];

  if (boostingTableId.length > 200) {
    errors.push(`boosting_table_id cannot exceed 200 characters; current count is ${boostingTableId.length}`);
  }

  if (!/^[a-zA-Z0-9_-]+$/.test(boostingTableId)) {
    errors.push('boosting_table_id can contain only letters, numbers, underscores, and hyphens');
  }

  return { isValid: errors.length === 0, errors };
}
```

Disse valideringsfunksjonene kjøres umiddelbart når brukeren konfigurerer kommandoord, og sikrer at problemer fanges opp så tidlig som mulig. Fra et brukeropplevelsesperspektiv er denne typen umiddelbar tilbakemelding veldig viktig. Det er alltid bedre for brukere å vite hva som er galt mens de skriver i stedet for etter at de har sendt inn.

### Frontend-konfigurasjon Persistens

I HagiCodes frontend-implementering valgte vi å bruke nettleserens `localStorage` for å lagre kommandoordkonfigurasjon. Det var flere hensyn bak denne designbeslutningen. For det første er kommandoordkonfigurasjonen svært personlig tilpasset, og forskjellige brukere kan ha forskjellige domenespesifikke behov. For det andre forenkler denne tilnærmingen backend-implementeringen fordi den ikke krever ekstra databasetabeller eller API-endepunkter. Til slutt, etter at brukere har konfigurert det én gang i nettleseren, kan innstillingene lastes automatisk ved senere bruk, noe som er veldig praktisk. Enkelt sagt er det den enkleste tilnærmingen.

```typescript
const HOTWORD_STORAGE_KEYS = {
  contextText: 'hotword-context-text',
  boostingTableId: 'hotword-boosting-table-id',
  combineMode: 'hotword-combine-mode',
} as const;

export const DEFAULT_HOTWORD_CONFIG: HotwordConfig = {
  contextText: '',
  boostingTableId: '',
  combineMode: false,
};

// Load hotword configuration
export function loadHotwordConfig(): HotwordConfig {
  const contextText = localStorage.getItem(HOTWORD_STORAGE_KEYS.contextText) || '';
  const boostingTableId = localStorage.getItem(HOTWORD_STORAGE_KEYS.boostingTableId) || '';
  const combineMode = localStorage.getItem(HOTWORD_STORAGE_KEYS.combineMode) === 'true';

  return { contextText, boostingTableId, combineMode };
}

// Save hotword configuration
export function saveHotwordConfig(config: HotwordConfig): void {
  localStorage.setItem(HOTWORD_STORAGE_KEYS.contextText, config.contextText);
  localStorage.setItem(HOTWORD_STORAGE_KEYS.boostingTableId, config.boostingTableId);
  localStorage.setItem(HOTWORD_STORAGE_KEYS.combineMode, String(config.combineMode));
}
```

Logikken i denne koden er grei og klar. Vi leser fra `localStorage` når du laster inn konfigurasjon, og skriv til `localStorage` når du lagrer den. Vi tilbyr også en standardkonfigurasjon slik at systemet fortsatt kan fungere som det skal når det ikke finnes noen konfigurasjon ennå. Det må tross alt være en fornuftig standard.

### Backend SDK-konfigurasjonsutvidelse

I HagiCodes backend-implementering trengte vi å legge til hotword-relaterte egenskaper til SDK-konfigurasjonsklassen. Vi tok i betraktning C#-språkegenskaper og bruksmønstre `List<string>` for å lagre tilpassede kommandoordkontekster:

```csharp
public class DoubaoVoiceConfig
{
    /// <summary>
    /// App ID
    /// </summary>
    public string AppId { get; set; } = string.Empty;

    /// <summary>
    /// Access token
    /// </summary>
    public string AccessToken { get; set; } = string.Empty;

    /// <summary>
    /// Service URL
    /// </summary>
    public string ServiceUrl { get; set; } = string.Empty;

    /// <summary>
    /// Custom hotword context list
    /// </summary>
    public List<string>? HotwordContexts { get; set; }

    /// <summary>
    /// Doubao platform hotword table ID
    /// </summary>
    public string? BoostingTableId { get; set; }
}
```

Utformingen av denne konfigurasjonsklassen følger HagiCodes vanlige konsise stil. `HotwordContexts` er en nullbar listetype, og `BoostingTableId` er en nullbar streng, så når det ikke er noen kommandoordkonfigurasjon, har disse egenskapene ingen effekt på forespørselen i det hele tatt. Hvis du ikke bruker funksjonen, bør den holde seg unna.

### Nyttelastkonstruksjonslogikk

Nyttelastkonstruksjon er kjernen i hele kommandoord-funksjonen. Når vi har konfigurasjon av kommandoord, må vi formatere det til JSON-strukturen som kreves av Doubao API. Denne prosessen skjer før SDK-en sender forespørselen:

```csharp
private void AddCorpusToRequest(Dictionary<string, object> request)
{
    var corpus = new Dictionary<string, object>();

    // Add custom hotwords
    if (Config.HotwordContexts != null && Config.HotwordContexts.Count > 0)
    {
        corpus["context"] = new Dictionary<string, object>
        {
            ["context_type"] = "dialog_ctx",
            ["context_data"] = Config.HotwordContexts
                .Select(text => new Dictionary<string, object> { ["text"] = text })
                .ToList()
        };
    }

    // Add platform hotword table ID
    if (!string.IsNullOrEmpty(Config.BoostingTableId))
    {
        corpus["boosting_table_id"] = Config.BoostingTableId;
    }

    // Add corpus to the request only when it is not empty
    if (corpus.Count > 0)
    {
        request["corpus"] = corpus;
    }
}
```

Denne koden viser hvordan du dynamisk konstruerer `corpus` felt basert på konfigurasjon. Hovedpoenget er at vi legger til `corpus` feltet bare når kommandoordkonfigurasjonen faktisk eksisterer. Denne designen sikrer bakoverkompatibilitet: når ingen kommandoord er konfigurert, forblir forespørselsstrukturen nøyaktig den samme som før. Bakoverkompatibilitet er viktig; å legge til en funksjon bør ikke forstyrre eksisterende logikk.

### WebSocket-parameter passerer

Mellom frontend og backend sendes kommandoordparametere gjennom WebSocket-kontrollmeldinger. HagiCode er designet slik at når grensesnittet starter opptaket, laster den kommandoordkonfigurasjonen fra `localStorage` og sender den til backend gjennom en WebSocket-melding.

```typescript
const controlMessage = {
  type: 'control',
  payload: {
    command: 'StartRecognition',
    contextText: '高血压\n糖尿病\n冠心病',
    boosting_table_id: 'medical_table',
    combineMode: false
  }
};
```

Det er én detalj å merke seg her: grensesnittet sender tekst med flere linjer atskilt med nylinjetegn, og bakenden må analysere den. Backend WebSocket-behandleren analyserer disse parameterne og sender dem til SDK:en:

```csharp
private async Task HandleControlMessageAsync(
    string connectionId,
    DoubaoSession session,
    ControlMessage message)
{
    if (message.Payload is SessionControlRequest controlRequest)
    {
        // Parse hotword parameters
        string? contextText = controlRequest.ContextText;
        string? boostingTableId = controlRequest.BoostingTableId;
        bool? combineMode = controlRequest.CombineMode;

        // Parse multi-line text into a hotword list
        if (!string.IsNullOrEmpty(contextText))
        {
            var hotwords = contextText
                .Split('\n', StringSplitOptions.RemoveEmptyEntries)
                .Select(s => s.Trim())
                .Where(s => s.Length > 0)
                .ToList();

            session.HotwordContexts = hotwords;
        }

        session.BoostingTableId = boostingTableId;
    }
}
```

Med denne designen blir overføring av kommandoordkonfigurasjon fra frontend til backend tydelig og effektiv. Det er ikke noe spesielt mystisk med det; dataene sendes ganske enkelt gjennom lag for lag.

## Praktisk veiledning

### Konfigurer egendefinerte kommandoord

Ved reell bruk er det veldig enkelt å konfigurere tilpassede kommandoord. Åpne innstillingssiden for talegjenkjenning i HagiCode og finn delen "Hotword Configuration". I inntastingsboksen "Tilpasset kommandoordtekst" skriver du inn én kommandoordfrase per linje.

Hvis du for eksempel utvikler en medisinsk-relatert applikasjon, kan du konfigurere den slik:

```
高血压
糖尿病
冠心病
心绞痛
心肌梗死
心力衰竭
```

Etter at du har lagret konfigurasjonen, sendes disse kommandoordene automatisk til Doubao API hver gang talegjenkjenning starter. I testene våre ble gjenkjennelsesnøyaktigheten for relaterte profesjonelle termer merkbart forbedret når kommandoord ble konfigurert. Forbedringen er reell, og klart bedre enn før.

### Konfigurer en plattformordordtabell

Hvis du trenger å administrere et stort antall kommandoord, eller hvis kommandoordene trenger hyppige oppdateringer, passer plattformens kommandoordtabell-modus bedre. Lag først en kommandoordtabell på Doubao selvlæringsplattform og skaff den genererte `boosting_table_id`, skriv deretter inn denne ID-en på HagiCode-innstillingssiden.

Doubao-selvlæringsplattformen gir funksjoner som bulkimport og kategorisert administrasjon for hotwords, noe som er veldig praktisk for team som trenger å administrere store sett med spesialisert terminologi. Ved å administrere kommandoord på plattformen kan du vedlikeholde dem sentralt og rulle ut oppdateringer konsekvent. Når kommandoordlisten blir stor, er det mye mer praktisk å ha et enkelt sted å administrere den enn manuell inntasting hver gang.

### Bruke kombinasjonsmodus

I noen komplekse scenarier kan det hende du må bruke både egendefinerte kommandoord og en plattformordordtabell samtidig. I så fall er det bare å konfigurere begge i HagiCode og aktivere "Combination Mode"-bryteren.

I kombinasjonsmodus vurderer Doubao API begge kommandoordkildene samtidig, så gjenkjenningsnøyaktigheten er vanligvis høyere enn å bruke begge kildene alene. Det er imidlertid verdt å merke seg at kombinasjonsmodus øker forespørselskompleksiteten, så det er best å bestemme seg for om du vil aktivere den etter praktisk testing. Mer kompleksitet er bare verdt det hvis resultatene fra den virkelige verden rettferdiggjør det.

### Eksempler på kodeintegrering

Det er veldig enkelt å integrere kommandoord-funksjonen i HagiCode-prosjektet. Her er noen ofte brukte kodebiter:

```typescript
import {
  loadHotwordConfig,
  saveHotwordConfig,
  validateHotwordConfig,
  parseContextText,
  getEffectiveHotwordMode,
  type HotwordConfig

// Load and validate configuration
const config = loadHotwordConfig();
const validation = validateHotwordConfig(config);

if (!validation.isValid) {
  console.error('Hotword configuration validation failed:', validation.errors);
  return;

// Parse hotword text
const hotwords = parseContextText(config.contextText);
console.log('Parsed hotwords:', hotwords);

// Get effective hotword mode
const mode = getEffectiveHotwordMode(config);
console.log('Current hotword mode:', mode);
```

## Ting å huske på

Det er flere punkter som fortjener spesiell oppmerksomhet når du implementerer og bruker kommandoord-funksjonen.

Først er tegngrensen. Doubao API har strenge begrensninger på hotwords, inkludert linjeantall, tegn per linje og totalt antall tegn. Hvis en grense overskrides, returnerer APIen en feil. I HagiCodes frontend-implementering sjekker vi disse begrensningene under brukerinndata gjennom valideringsfunksjoner, som forhindrer at ugyldige konfigurasjoner sendes til backend. Å fange opp problemer tidlig er alltid bedre enn å vente på at API-en skal mislykkes.

For det andre er formatet på `boosting_table_id`. Dette feltet tillater bare bokstaver, tall, understrek og bindestreker, og det kan ikke inneholde mellomrom eller andre spesialtegn. Når du oppretter en kommandoordstabell på Doubao selvlæringsplattform, sørg for å følge navneglene. Den typen streng formatvalidering er vanlig for APIer.

For det tredje er bakoverkompatibilitet. Hotword-parametere er helt valgfrie. Hvis ingen kommandoord er konfigurert, oppfører systemet seg nøyaktig som det gjorde før. Denne designen sikrer at eksisterende brukere ikke påvirkes på noen måte, og det gjør også gradvis migrering og oppgraderinger enklere. Å legge til en funksjon bør ikke forstyrre den forrige logikken.

Til slutt er det feilhåndtering. Når kommandoordkonfigurasjonen er ugyldig, returnerer Doubao API tilsvarende feilmeldinger. HagiCodes implementering registrerer detaljerte logger for å hjelpe utviklere med å feilsøke problemer. Samtidig viser grensesnittet valideringsfeil i brukergrensesnittet for å hjelpe brukere med å korrigere konfigurasjonen. God feilhåndtering fører naturligvis til en bedre brukeropplevelse.
