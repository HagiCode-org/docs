## Fra CLI-anrop til SDK-integrasjon: Beste praksis for GitHub Copilot i .NET-prosjekter

> Oppgraderingsveien fra kommandolinjepåkallelse til offisiell SDK-integrasjon har vært litt av en reise. I dag vil jeg dele fallgruvene vi havnet i og hva vi lærte mens vi bygde HagiCode.

## Bakgrunn

Etter at GitHub Copilot SDK ble offisielt utgitt i 2025, begynte vi å integrere den i AI-kapasitetslaget vårt. Før det brukte prosjektet hovedsakelig GitHub Copilot ved å direkte påkalle Copilot CLI-kommandolinjeverktøyet, men den tilnærmingen hadde flere åpenbare problemer:

- **Kompleks prosessadministrasjon**: Vi måtte manuelt administrere CLI-prosessens livssyklus, oppstartstidsavbrudd og prosessopprydding. Prosesser kan krasje uten forvarsel.
- **Ufullstendig hendelseshåndtering**: Rå CLI-anrop gjør det vanskelig å fange opp finkornede hendelser fra modellresonnement og verktøyutførelse. Det er som å bare se resultatet uten tankeprosessen.
- **Vanskelig øktadministrasjon**: Det fantes ingen effektiv mekanisme for gjenbruk og gjenoppretting av økter, så hver interaksjon måtte starte på nytt.
- **Kompatibilitetsproblemer**: CLI-argumenter endret seg ofte, noe som gjorde at vi hele tiden måtte opprettholde kompatibilitetslogikk for disse parameterne.

Disse problemene ble stadig tydeligere i den daglige utviklingen, spesielt når vi trengte å spore modellresonnement (`thinking`) og verktøyutførelsesstatus i sanntid. På det tidspunktet ble det klart at vi trengte en integreringstilnærming på lavere nivå og mer fullstendig.

## Arkitektur design

Prosjektet bruker en lagdelt arkitektur for å adressere begrensningene ved CLI-påkallelse:

```
┌─────────────────────────────────────────────────────────┐
│  hagicode-core (Orleans Grains + AI Provider Layer)    │
│  - CopilotAIProvider: Converts AIRequest to CopilotOptions │
│  - GitHubCopilotGrain: Orleans distributed execution interface │
└─────────────────────────────────────────────────────────┘
                          ↓
┌─────────────────────────────────────────────────────────┐
│  HagiCode.Libs (Shared Provider Layer)                 │
│  - CopilotProvider: CLI Provider interface implementation │
│  - ICopilotSdkGateway: SDK invocation abstraction      │
│  - GitHubCopilotSdkGateway: SDK session management and event dispatch │
└─────────────────────────────────────────────────────────┘
                          ↓
┌─────────────────────────────────────────────────────────┐
│  GitHub Copilot SDK (Official .NET SDK)                │
│  - CopilotClient: SDK client                           │
│  - CopilotSession: Session management                  │
│  - SessionEvent: Event stream                          │
└─────────────────────────────────────────────────────────┘
```

Denne lagdelte designen gir flere praktiske tekniske fordeler:

1. **Separasjon av bekymringer**: Kjernevirksomhetslogikk er koblet fra SDK-implementeringsdetaljer.
2. **Testbarhet**: Den `ICopilotSdkGateway` grensesnitt gjør enhetstesting enkel.
3. **Gjenbrukbarhet**: `HagiCode.Libs` kan refereres av flere prosjekter.
4. **Vedlikehold**: SDK-oppgraderinger krever bare endringer i Gateway-laget, mens de øvre lagene forblir urørt.

## Kjerneimplementering

### Autentiseringsflyt

Autentisering er det første og viktigste trinnet i SDK-integrasjon. Hvis autentiseringen mislykkes, betyr ingenting annet. Vi utviklet en fleksibel autentiseringskonfigurasjon som støtter flere autentiseringskilder:

```csharp
// CopilotProvider.cs - Authentication source configuration
public class CopilotOptions
{
    public bool UseLoggedInUser { get; set; } = true;
    public string? GitHubToken { get; set; }
    public string? CliUrl { get; set; }
}

// Convert to SDK request
return new CopilotSdkRequest(
    GitHubToken: options.AuthSource == CopilotAuthSource.GitHubToken
        ? options.GitHubToken
        : null,
    UseLoggedInUser: options.AuthSource != CopilotAuthSource.GitHubToken
);
```

Fordelene med dette designet er ganske klare:

- Den støtter innlogget brukermodus uten å kreve et token, noe som passer godt til skrivebordsscenarier.
- Den støtter GitHub Token-modus, som er egnet for distribusjoner på serversiden og sentralisert administrasjon.
- Den støtter overstyring av Copilot CLI URL, som hjelper med konfigurasjon av bedriftens proxy.

I praksis forenklet denne fleksible autentiseringsmodellen konfigurasjonen betydelig på tvers av ulike distribusjonsscenarier. Desktop-klienten kan bruke hver brukers egen Copilot-påloggingstilstand, mens serversiden kan administrere autentisering sentralt gjennom tokens.

### Håndtering av hendelsesstrømmer

En av de kraftigste egenskapene til SDK er fullstendig hendelsesstrømopptak. Vi implementerte et hendelsesforsendelsessystem som kan håndtere alle slags SDK-hendelser i sanntid:

```csharp
// GitHubCopilotSdkGateway.cs - Core logic for event dispatch
internal static SessionEventDispatchResult DispatchSessionEvent(
    SessionEvent evt, bool sawDelta)
{
    switch (evt)
    {
        case AssistantReasoningEvent reasoningEvent:
            // Capture the model reasoning process
            events.Add(new CopilotSdkStreamEvent(
                CopilotSdkStreamEventType.ReasoningDelta,
                Content: reasoningEvent.Data.Content));
            break;

        case ToolExecutionStartEvent toolStartEvent:
            // Capture the start of a tool invocation
            events.Add(new CopilotSdkStreamEvent(
                CopilotSdkStreamEventType.ToolExecutionStart,
                ToolName: toolStartEvent.Data.ToolName,
                ToolCallId: toolStartEvent.Data.ToolCallId));
            break;

        case ToolExecutionCompleteEvent toolCompleteEvent:
            // Capture tool completion and its result
            events.Add(new CopilotSdkStreamEvent(
                CopilotSdkStreamEventType.ToolExecutionEnd,
                Content: ExtractToolExecutionContent(toolCompleteEvent)));
            break;

        default:
            // Preserve unhandled events as RawEvent
            events.Add(new CopilotSdkStreamEvent(
                CopilotSdkStreamEventType.RawEvent,
                RawEventType: evt.GetType().Name));
            break;
    }
}
```

Verdien av denne implementeringen er betydelig:

- **Fullstendig fangst av modellresonneringsprosessen** (`thinking`): brukere kan se hvordan AI resonnerer, ikke bare det endelige svaret.
- **Sanntidssporing av verktøyutførelsesstatus**: vi vet hvilke verktøy som kjører, når de er ferdige og hva de returnerer.
- **Null hendelsestap**: ved å falle tilbake til `RawEvent`, sørger vi for at alle hendelser blir registrert.

I HagiCode hjelper disse finkornede hendelsene brukere med å forstå hvordan AI fungerer internt, spesielt når de feilsøker komplekse oppgaver.

### CLI-kompatibilitetshåndtering

Etter migrering fra CLI-påkalling til SDK, fant vi ut at noen eksisterende CLI-parametere ikke lenger ble brukt i SDK. For å bevare bakoverkompatibilitet implementerte vi et parameterfiltreringssystem:

```csharp
};

public static CopilotCliArgumentBuildResult BuildCliArgs(CopilotOptions options)
{
    // Filter out unsupported arguments and keep compatible ones
    // Generate diagnostic information
}
```

Dette gir oss flere fordeler:

- Den filtrerer automatisk inkompatible CLI-argumenter for å forhindre kjøretidsfeil.
- Den genererer klare diagnostiske meldinger for å hjelpe utviklere med å finne problemer raskt.
- Det holder SDK stabil og isolert fra endringer i CLI-parametere.

Under oppgraderingen hjalp denne kompatibilitetsmekanismen oss med en jevn overgang. Eksisterende konfigurasjonsfiler kunne fortsatt brukes og trengte bare gradvise justeringer basert på diagnostisk utdata.

### Runtime Pooling

Å lage Copilot SDK-økter er relativt dyrt, og å lage og ødelegge økter for ofte skader ytelsen. For å løse det implementerte vi et administrasjonssystem for øktbasseng:

```csharp
// CopilotProvider.cs - Session pool management
await using var lease = await _poolCoordinator.AcquireCopilotRuntimeAsync(
    request,
    async ct => await _gateway.CreateRuntimeAsync(sdkRequest, ct),
    cancellationToken);

if (lease.IsWarmLease)
{
    // Reuse an existing session
    yield return CreateSessionReusedMessage();
}

await foreach (var eventData in lease.Entry.Resource.SendPromptAsync(...))
{
    yield return MapEvent(eventData);
}
```

Fordelene med samling av økter inkluderer:

- **gjenbruk av økter**: forespørsler med det samme `sessionId` kan gjenbruke eksisterende økter og redusere oppstartskostnader.
- **Støtte for øktgjenoppretting**: etter et nettverksavbrudd kan forrige øktstatus gjenopprettes.
- **Automatisk samlingsadministrasjon**: utløpte økter ryddes opp automatisk for å unngå ressurslekkasjer.

I HagiCode, økte sammenslåing merkbart forbedret respons, spesielt for kontinuerlige samtaler.

### Orleans integrasjon

HagiCode bruker Orleans som sitt distribuerte rammeverk, og vi integrerte Copilot SDK i Orleans Grains:

```csharp
// GitHubCopilotGrain.cs - Distributed execution
public async IAsyncEnumerable<GitHubCopilotResponse> ExecuteCommandStreamAsync(
    string command,
    CancellationToken token = default)
{
    var provider = await aiProviderFactory.GetProviderAsync(AIProviderType.GitHubCopilot);

    await foreach (var chunk in provider.SendMessageAsync(request, null, token))
    {
        // Map to the unified response format
        yield return BuildChunkResponse(chunk, startedAt);
    }
}
```

Fordelene med Orleans-integrasjon er betydelige:

- **Unified AI Provider abstraksjon**: det blir enkelt å bytte mellom forskjellige AI-leverandører.
- **Isolering av flere leietakere**: Kopilotøkter for forskjellige brukere forblir isolert fra hverandre.
- **Vedvarende øktstatus**: Sesjonstilstand kan gjenopprettes selv etter at serveren har startet på nytt.

For scenarier som trenger å håndtere et stort antall samtidige forespørsler, tilbyr Orleans sterk skalerbarhet.

## Praktisk veiledning

### Konfigurasjonseksempel

Her er et fullstendig konfigurasjonseksempel:

```json
{
  "AI": {
    "Providers": {
      "Providers": {
        "GitHubCopilot": {
          "Enabled": true,
          "ExecutablePath": "copilot",
          "Model": "gpt-5",
          "WorkingDirectory": "/path/to/project",
          "Timeout": 7200,
          "StartupTimeout": 30,
          "UseLoggedInUser": true,
          "NoAskUser": true,
          "Permissions": {
            "AllowAllTools": false,
            "AllowedTools": ["Read", "Bash", "Grep"],
            "DeniedTools": ["Edit"]
          }
        }
      }
    }
  }
}
```

### Bruksnotater

Ved bruk i den virkelige verden oppsummerte vi flere punkter det er verdt å være oppmerksom på:

**Konfigurasjon av tidsavbrudd for oppstart**: Den første oppstarten av Copilot CLI kan ta relativt lang tid, så vi anbefaler å stille inn `StartupTimeout` til minst 30 sekunder. Hvis dette er første pålogging, kan det ta enda lengre tid.

**Tillatelsesbehandling**: Unngå bruk i produksjonsmiljøer `AllowAllTools: true`. Bruk `AllowedTools` godkjenningsliste for å kontrollere hvilke verktøy som er tilgjengelige, og bruke `DeniedTools` denylist for å blokkere farlige operasjoner. Dette forhindrer effektivt AI fra å utføre risikable kommandoer.

**Session management**: Forespørsler med samme `sessionId` automatisk gjenbruk økter. Sesjonstilstanden opprettholdes `ProviderSessionId`. Kansellering forplantes via `CancellationTokenSource`.

**Diagnostisk utdata**: Inkompatible CLI-argumenter genererer meldinger av typen `diagnostic`. Raw SDK-hendelser er bevart som `event.raw`. Feilmeldinger inkluderer kategorier som oppstartstidsavbrudd og argumentinkompatibilitet for å gjøre feilsøkingen enklere.

### Beste praksis

Basert på vår praktiske erfaring, her er noen gode fremgangsmåter:

**1. Bruk en godkjenningsliste for verktøy**

```csharp
var request = new AIRequest
{
    Prompt = "Analyze this file",
    AllowedTools = new[] { "Read", "Grep", "Bash(git:*)" }
};
```

Å eksplisitt spesifisere de tillatte verktøyene gjennom en godkjenningsliste bidrar til å forhindre uventede AI-handlinger. Dette er spesielt viktig for verktøy med skriverettigheter, som f.eks `Edit`, som bør håndteres med ekstra forsiktighet.

**2. Angi rimelige tidsavbrudd**

```csharp
options.Timeout = 3600;  // 1 hour
options.StartupTimeout = 60;  // 1 minute
```

Angi passende tidsavbruddsverdier basert på oppgavens kompleksitet. Hvis verdien er for kort, kan oppgaver bli avbrutt. Hvis det er for lang tid, kan ressurser bli kastet bort på å vente på forespørsler som ikke svarer.

**3. Aktiver gjenbruk av økter**

```csharp
options.SessionId = "my-session-123";
```

Bruker det samme `sessionId` for relaterte oppgaver lar deg gjenbruke tidligere øktkontekst og forbedre responshastigheten.

**4. Håndter strømmesvar**

```csharp
await foreach (var chunk in provider.StreamAsync(request))
{
    switch (chunk.Type)
    {
        case StreamingChunkType.ThinkingDelta:
            // Handle the reasoning process
            break;
        case StreamingChunkType.ToolCallDelta:
            // Handle tool invocation
            break;
        case StreamingChunkType.ContentDelta:
            // Handle text output
            break;
    }
}
```

Streaming-svar lar deg vise AI-behandlingsfremgang i sanntid, noe som forbedrer brukeropplevelsen. Dette er spesielt verdifullt for langvarige oppgaver.

**5. Feilhåndtering og gjenforsøk**

```csharp
try
{
    await foreach (var chunk in provider.StreamAsync(request))
    {
        // Handle the response
    }
}
catch (CopilotSessionException ex)
{
    // Handle session exceptions
    logger.LogError(ex, "Copilot session failed");
    // Decide whether to retry based on the exception type
}
```

Riktig feilhåndtering og gjenforsøkslogikk forbedrer den generelle systemstabiliteten.

## Konklusjon

Oppgradering fra CLI-påkalling til SDK-integrasjon ga betydelig verdi til HagiCode-prosjektet:

- **Forbedret stabilitet**: SDK gir et mer stabilt grensesnitt som ikke påvirkes av CLI-versjonsendringer.
- **Mer komplett funksjonalitet**: den fanger opp hele hendelsesstrømmen, inkludert resonnement og verktøyutførelsesstatus.
- **Høyere utviklingseffektivitet**: det typesikre SDK-grensesnittet gjør utviklingen mer effektiv og reduserer kjøretidsfeil.
- **Bedre brukeropplevelse**: tilbakemeldinger om hendelser i sanntid gir brukerne en klarere forståelse av hva AI gjør.

Denne oppgraderingen var ikke bare en erstatning for teknisk implementering. Det var også en arkitektonisk optimalisering av hele AI-evnelaget. Gjennom lagdelte design- og abstraksjonsgrensesnitt, fikk vi bedre vedlikeholdsmuligheter og utvidbarhet.

Hvis du vurderer å integrere GitHub Copilot i ditt eget .NET-prosjekt, håper jeg disse praktiske leksjonene hjelper deg å unngå unødvendige omveier. Den offisielle SDK-en er faktisk mer stabil og komplett enn CLI-påkallelse, og det er verdt tiden å forstå og ta i bruk den.
