## Teknisk analyse av HagiCode Skill System: Bygge en skalerbar AI Skill Management Platform

> Denne artikkelen tar en grundig titt på arkitekturen og implementeringen av Skill Management-systemet i HagiCode-prosjektet, og dekker de tekniske detaljene bak fire kjernefunksjoner: lokal global administrasjon, markedsplasssøk, intelligente anbefalinger og pålitelig leverandøradministrasjon.

## Bakgrunn

Når det gjelder AI-kodingsassistenter, har det alltid vært et kjernespørsmål hvordan man utvider grensene for AI-evner. Claude Code selv er allerede sterk på kodeassistanse, men forskjellige utviklingsteam og forskjellige teknologistabler trenger ofte spesialiserte evner for spesifikke scenarier, for eksempel håndtering av Docker-distribusjoner, databaseoptimalisering eller frontend-komponentgenerering. Det er akkurat der et ferdighetssystem blir spesielt viktig.

Under utviklingen av HagiCode-prosjektet møtte vi en lignende utfordring: hvordan lar vi Claude Code «lære» nye faglige ferdigheter som en person ville, samtidig som vi opprettholder en solid brukeropplevelse og god teknisk vedlikehold? Dette problemet er både vanskelig og enkelt på sin egen måte. Rundt det spørsmålet designet og implementerte vi et komplett ferdighetsstyringssystem.

Denne artikkelen går gjennom den tekniske arkitekturen og kjerneimplementeringen av systemet i detalj. Den er beregnet på utviklere som er interessert i AI-utvidbarhet og kommandolinjeverktøyintegrasjon. Det kan være nyttig for deg, eller kanskje ikke, men det er i det minste skrevet ned nå.

## Om HagiCode

Tilnærmingen som deles i denne artikkelen kommer fra vår praktiske erfaring i [HagiCode](https://hagicode.com) prosjekt. HagiCode er et åpen kildekode AI-kodeassistentprosjekt designet for å hjelpe utviklingsteam med å forbedre ingeniøreffektiviteten. Prosjektets stabel inkluderer ASP.NET Core, Orleans-distribuerte rammeverk, en TanStack Start + React-frontend og Skill Management-delsystemet introdusert i denne artikkelen.

GitHub-depotet er [HagiCode-org/side](https://github.com/HagiCode-org/site). Hvis du finner den tekniske tilnærmingen i denne artikkelen verdifull, gi den gjerne en stjerne. Flere stjerner har en tendens til å forbedre stemningen, tross alt.

## Oversikt over systemarkitektur

Skill-systemet bruker en frontend-backend-separert arkitektur. Det er ikke noe spesielt mystisk med det.

**Frontend** bruker TanStack Start + React for å bygge brukergrensesnittet, med Redux Toolkit-administrasjonstilstand. De fire hovedfunksjonene tilordnes direkte til fire fanekomponenter: lokale ferdigheter, ferdighetsgalleri, intelligente anbefalinger og pålitelige leverandører. Til syvende og sist handler designet mest om å gjøre brukeropplevelsen bedre.

**Backend** er basert på ASP.NET Core + ABP Framework, og bruker Orleans Grain for distribuert statlig styring. Den elektroniske API-klienten omslutter `IOnlineApiClient` grensesnitt for å kommunisere med den eksterne ferdighetskatalogtjenesten.

Det overordnede arkitektoniske prinsippet er å skille kommandoutførelse fra forretningslogikk. Gjennom adaptermønsteret er implementeringsdetaljene for npm/npx-kommandoutførelse skjult inne i uavhengige moduler. Tross alt er det ingen som virkelig vil ha kommandolinjeanrop spredt over hele kodebasen.

## Kjerneevne 1: Lokal global ledelse

Lokal global ledelse er den mest grunnleggende modulen. Det er ansvarlig for å liste opp installerte ferdigheter og støtte avinstalleringsoperasjoner. Det er ikke noe altfor komplisert her; det handler mest om å gjøre det grunnleggende godt.

### Teknisk tilnærming

Gjennomføringen lever inn `LocalSkillsTab.tsx` og `LocalSkillCommandAdapter.cs`. Kjerneideen er å pakke inn `npx skills` kommandoen, analyser dens JSON-utgang og konverter den til interne datastrukturer. Det høres enkelt ut, og i praksis er det stort sett det.

```csharp
public async Task<IReadOnlyList<LocalSkillInventoryResponseDto>> GetLocalSkillsAsync(
    CancellationToken cancellationToken = default)
{
    var result = await _commandAdapter.ListGlobalSkillsAsync(cancellationToken);
    return result.Skills.Select(skill => new LocalSkillInventoryResponseDto
    {
        Name = skill.Name,
        Version = skill.Version,
        Source = skill.Source,
        InstalledPath = skill.InstalledPath,
        Description = skill.Description
    }).ToList();
}
```

Dataflyten er veldig tydelig: frontend sender en forespørsel -> `SkillGalleryAppService` mottar det -> `LocalSkillCommandAdapter` utfører npx-kommandoen -> JSON-resultatet blir analysert -> en DTO returneres. Hvert trinn følger naturlig fra det forrige.

Avinstallering av ferdigheter bruker `npx skills remove -g <skillName> -y` kommando, og systemet håndterer automatisk avhengigheter og opprydding. Installasjonsmetadata lagres i `managed-install.json` inne i ferdighetskatalogen, registrerer informasjon som installasjonstid og kildeversjon for senere oppdateringer og revisjon. Noen ting er rett og slett verdt å ta opp.

### Installasjonsflyt i detalj

Installasjon av ferdigheter krever flere koordinerte trinn. I sannhet er det ikke spesielt komplisert:

```csharp
public async Task<SkillInstallResultDto> InstallAsync(
    SkillInstallRequestDto request,
    CancellationToken cancellationToken = default)
{
    // 1. Normalize the installation reference
    var normalized = _referenceNormalizer.Normalize(
        request.SkillId,
        request.Source,
        request.SkillSlug,
        request.Version);

    // 2. Check prerequisites
    await _prerequisiteChecker.CheckAsync(cancellationToken);

    // 3. Acquire installation lock
    using var installLock = await _lockProvider.AcquireAsync(normalized.SkillId);

    // 4. Execute installation command
    var result = await _installCommandRunner.ExecuteAsync(
        new SkillInstallCommandExecutionRequest
        {
            Command = $"npx skills add {normalized.FullReference} -g -y",
            Timeout = TimeSpan.FromMinutes(4)
        },
        cancellationToken);

    // 5. Persist installation metadata
    await _metadataStore.WriteAsync(normalized.SkillPath, request);

    return new SkillInstallResultDto { Success = result.Success };
}
```

Her brukes flere nøkkeldesignmønstre: **referansenormalisatoren** konverterer forskjellige inndataformater, som f.eks. `tanweai/pua` og `@opencode/docker-skill`, til en enhetlig intern representasjon; **installasjonslåsmekanismen** sikrer at bare én installasjonsoperasjon kan kjøres for samme ferdighet om gangen; og **streaming output** presser installasjonsfremdriften til frontend i sanntid gjennom Server-Sent Events, slik at brukere kan se terminallignende logger etter hvert som de skjer.

Til syvende og sist er alle disse mønstrene der for ett formål: å gjøre systemet enklere å bruke og vedlikeholde.

## Kjernefunksjon 2: Markedssøk

Marketplace-søk lar brukere oppdage og installere ferdigheter fra fellesskapet. En persons evne er alltid begrenset; kollektiv kunnskap går mye lenger.

### Teknisk tilnærming

Søkefunksjonen er avhengig av online API `https://api.hagicode.com/v1/skills/search`. For å forbedre responshastigheten implementerer systemet caching. Cache er litt som minne: Hvis du har nyttige ting rundt deg, trenger du ikke tenke så hardt neste gang.

```csharp
private async Task<IReadOnlyList<SkillGallerySkillDto>> SearchCatalogAsync(
    string query,
    CancellationToken cancellationToken,
    IReadOnlySet<string>? allowedSources = null)
{
    var cacheKey = $"skill_search:{query}:{string.Join(",", allowedSources ?? Array.Empty<string>())}";

    if (_memoryCache.TryGetValue(cacheKey, out var cached))
        return (IReadOnlyList<SkillGallerySkillDto>)cached!;

    var response = await _onlineApiClient.SearchAsync(
        new SearchSkillsRequest
        {
            Query = query,
            Limit = _options.LimitPerQuery,
        },
        cancellationToken);

    var results = response.Skills
        .Where(skill => allowedSources is null || allowedSources.Contains(skill.Source))
        .Select(skill => new SkillGallerySkillDto { ... })
        .ToList();

    _memoryCache.Set(cacheKey, results, TimeSpan.FromMinutes(10));
    return results;
}
```

Søkeresultater støtter filtrering etter pålitelige kilder, slik at brukere bare ser ferdighetskilder de stoler på. Frøspørringer som f.eks `popular` og `recent` brukes til å initialisere katalogen, slik at brukere kan se anbefalte populære ferdigheter første gang de åpner den. Førsteinntrykket betyr fortsatt noe.

## Kjernekapasitet 3: Intelligente anbefalinger

Intelligente anbefalinger er den mest komplekse delen av systemet. De kan automatisk anbefale de best egnede ferdighetene basert på den aktuelle prosjektkonteksten. Kompleks som det er, er det fortsatt verdt å bygge.

### Anbefalingsflyt

Hele anbefalingsflyten er delt inn i fem stadier:

```
1. Build project context
   ↓
2. AI generates search queries
   ↓
3. Search the online catalog in parallel
   ↓
4. AI ranks the candidates
   ↓
5. Return the recommendation list
```

Først analyserer systemet egenskaper som prosjektets teknologistabel, programmeringsspråk og domenestruktur for å bygge en «prosjektprofil». Den profilen er litt som en CV, som registrerer hovedtrekkene til prosjektet.

Deretter brukes en AI Grain til å generere målrettede søk. Denne designen er faktisk ganske interessant: i stedet for å spørre AI direkte, "Hvilke ferdigheter bør jeg anbefale?", ber vi den først tenke på "Hvilke søkeord vil sannsynligvis finne relevante ferdigheter?" Noen ganger betyr måten du stiller spørsmålet på mer enn selve svaret:

```csharp
var queryGeneration = await aiGrain.GenerateSkillRecommendationQueriesAsync(
    projectContext,      // Project context
    locale,              // User language preference
    maxQueries,           // Maximum number of queries
    effectiveSearchHero); // AI model selection
```

Deretter utføres disse søkene parallelt for å samle en liste over kandidatferdigheter. Parallell behandling er på slutten av dagen bare en måte å spare tid på.

Til slutt rangerer en annen AI Grain kandidatens ferdigheter. Dette trinnet tar hensyn til faktorer som relevans for ferdigheter for prosjektet, tillitsstatus og brukerhistoriske preferanser:

```csharp
var ranking = await aiGrain.RankSkillRecommendationsAsync(
    projectContext,
    candidates,
    installedSkillNames,
    locale,
    maxRecommendations,
    effectiveRankingHero);

response.Items = MergeRecommendations(projectContext, candidates, ranking, maxRecommendations);
```

### Tilbakeslagsmekanisme

AI-modeller kan reagere sakte eller bli midlertidig utilgjengelige. Selv de beste systemene snubler noen ganger. Av den grunn inkluderer systemet en deterministisk reservemekanisme: når AI-tjenesten er utilgjengelig, bruker den en regelbasert heuristisk algoritme for å generere anbefalinger, for eksempel å utlede sannsynlig nødvendige ferdigheter fra avhengigheter i `package.json`.

Enkelt sagt er denne reservemekanismen ganske enkelt en backupplan for systemet.

## Kjerneevne 4: Trusted Provider Management

Betrodd leverandøradministrasjon lar brukere kontrollere hvilke ferdighetskilder som anses som pålitelige. Tillit er fortsatt noe brukere bør kunne definere selv.

### Matchende regler

Klarerte leverandører støtter to samsvarsregler: **eksakt samsvar** (`exact`) og **prefiksmatch** (`prefix`).

```csharp
public static TrustedSkillProviderResolutionSnapshot Resolve(
    TrustedSkillProviderSnapshot snapshot,
    string source)
{
    var normalizedSource = Normalize(source);

    foreach (var entry in snapshot.Entries.OrderBy(e => e.SortOrder))
    {
        if (!entry.IsEnabled) continue;

        foreach (var rule in entry.MatchRules)
        {
            bool isMatch = rule.MatchType switch
            {
                TrustedSkillProviderMatchRuleType.Exact
                    => string.Equals(normalizedSource, Normalize(rule.Value),
                        StringComparison.OrdinalIgnoreCase),
                TrustedSkillProviderMatchRuleType.Prefix
                    => normalizedSource.StartsWith(Normalize(rule.Value) + "/",
                        StringComparison.OrdinalIgnoreCase),
                _ => false
            };

            if (isMatch)
                return new TrustedSkillProviderResolutionSnapshot
                {
                    IsTrustedSource = true,
                    ProviderId = entry.ProviderId,
                    DisplayName = entry.DisplayName
                };
        }
    }

    return new TrustedSkillProviderResolutionSnapshot { IsTrustedSource = false };
}
```

Innebygde pålitelige leverandører inkluderer kjente organisasjoner og prosjekter som Vercel, Azure, anthropics, Microsoft og nettleserbruk. Egendefinerte leverandører kan legges til gjennom konfigurasjonsfiler ved å spesifisere en leverandør-ID, visningsnavn, merkeetikett, samsvarsregler og mer. Verden er stor nok til at det aldri vil være nok å stole på noen få innebygde enheter.

### Utholdenhetsimplementering

Klarert konfigurasjon opprettholdes med en Orleans Grain:

```csharp
public class TrustedSkillProviderGrain : Grain<TrustedSkillProviderState>,
    ITrustedSkillProviderGrain
{
    public async Task UpdateConfigurationAsync(TrustedSkillProviderSnapshot snapshot)
    {
        State.Snapshot = snapshot;
        await WriteStateAsync();
    }

    public Task<TrustedSkillProviderSnapshot> GetConfigurationAsync()
    {
        return Task.FromResult(State.Snapshot);
    }
}
```

Fordelen med denne tilnærmingen er at konfigurasjonsendringer automatisk synkroniseres på tvers av alle noder, uten behov for å oppdatere cacher manuelt. Automatisering handler til syvende og sist om å la folk bekymre seg mindre.

## Nøkkel teknisk design

### Adaptermønster for kommandoutførelse

Skill-systemet må utføre ulike npx-kommandoer. Hvis den logikken var spredt overalt, ville koden raskt blitt vanskelig å vedlikeholde. Derfor utviklet vi et adaptergrensesnitt. Designmønstre eksisterer til syvende og sist for å gjøre koden enklere å vedlikeholde:

```csharp
public interface ISkillInstallCommandRunner
{
    Task<SkillInstallCommandExecutionResult> ExecuteAsync(
        SkillInstallCommandExecutionRequest request,
        CancellationToken cancellationToken = default);
}
```

Ulike kommandoer har forskjellige eksekveringsimplementeringer, men alle implementerer det samme grensesnittet, noe som gjør testing og utskifting enkel.

### SSE Streaming Output

Installasjonsfremdriften skyves til frontend i sanntid gjennom Server-Sent Events:

```csharp
public async Task InstallWithProgressAsync(
    SkillInstallRequestDto request,
    IServerStreamWriter<SkillInstallProgressEventDto> stream,
    CancellationToken cancellationToken)
{
    var process = new Process
    {
        StartInfo = new ProcessStartInfo
        {
            FileName = "npx",
            Arguments = $"skills add {request.FullReference} -g -y",
            RedirectStandardOutput = true,
            RedirectStandardError = true,
            UseShellExecute = false
        }
    };

    process.OutputDataReceived += async (sender, e) =>
    {
        await stream.WriteAsync(new SkillInstallProgressEventDto
        {
            EventType = "output",
            Data = e.Data ?? string.Empty
        });
    };

    process.Start();
    process.BeginOutputReadLine();
    await process.WaitForExitAsync(cancellationToken);
}
```

På frontend kan brukere se terminallignende utdata i sanntid, noe som gjør opplevelsen veldig intuitiv. Tilbakemelding i sanntid hjelper folk til å føle seg vel.

## Praktisk veiledning

### Installere en fellesskapsferdighet

Ta å installere `pua` ferdighet som et eksempel (det er en populær fellesskapsferdighet):

1. Åpne ferdighetsskuffen og bytt til `Skill Gallery` fanen
2. Gå inn `pua` i søkefeltet
3. Klikk på søkeresultatet for å se ferdighetsdetaljene
4. Klikk på `Install` knappen
5. Bytt til `Local Skills` for å bekrefte at installasjonen var vellykket

Installasjonskommandoen er `npx skills add tanweai/pua -g -y`, og systemet håndterer alle detaljene automatisk. Det er egentlig ikke så mange trinn når du tar dem en etter en.

### Legge til en tilpasset pålitelig kilde

Hvis teamet ditt har sitt eget ferdighetslager, kan du legge det til som en pålitelig kilde:

```yaml
providerId: "my-team"
displayName: "My Team Skills"
badgeLabel: "MyTeam"
isEnabled: true
sortOrder: 100
matchRules:
  - matchType: "prefix"
    value: "my-team/"
  - matchType: "exact"
    value: "my-team/special-skill"
```

På denne måten vil alle ferdigheter fra teamet ditt vise et klarert merke, noe som gjør brukerne mer komfortable med å installere dem. Etiketter og signaler hjelper folk til å føle seg mer selvsikre.

### Grunnleggende ferdighetsutvikling

Å lage en egendefinert ferdighet krever følgende struktur:

```
my-skill/
├── SKILL.md          # Skill metadata (YAML front matter)
├── index.ts          # Skill entry point
├── agents/           # Supported agent configuration
└── references/       # Reference resources
```

Et eksempel `SKILL.md` format:

```yaml
---
name: my-skill
description: A brief description of what this skill does
---

# My Skill

Detailed documentation...
```

## Notater

1. **Nettverkskrav**: ferdighetssøk og installasjon krever tilgang til `api.hagicode.com` og npm-registeret
2. **Node.js-versjon**: Node.js 18 eller nyere anbefales
3. **Tillatelseskrav**: globale npm installasjonstillatelser kreves
4. **Samtidighetskontroll**: bare én installasjons- eller avinstalleringsoperasjon kan kjøres for samme ferdighet om gangen
5. **Innstillinger for tidsavbrudd**: standard tidsavbrudd for installasjon er 4 minutter, men komplekse scenarier kan kreve justering

Disse notatene eksisterer til syvende og sist for å hjelpe ting til å gå greit.

## Konklusjon

Denne artikkelen introduserte den komplette implementeringen av Skill Management-systemet i HagiCode-prosjektet. Gjennom en frontend-backend-separert arkitektur, adaptermønsteret, Orleans-basert distribuert statsadministrasjon og relaterte teknikker, leverer systemet:

- **Lokal global ledelse**: et enhetlig grensesnitt for ferdighetsadministrasjon bygget ved innpakning `npx skills` kommandoer
- **Markedplace-søk**: rask oppdagelse av fellesskapsferdigheter gjennom online API og hurtigbuffermekanismer
- **Intelligente anbefalinger**: AI-drevne ferdighetsanbefalinger basert på prosjektkontekst
- **Tillitadministrasjon**: et fleksibelt konfigurasjonssystem som lar brukere kontrollere tillitsgrenser

Denne designtilnærmingen gjelder ikke bare for ferdighetsledelse. Det er også nyttig som referanse for ethvert scenario som trenger å integrere kommandolinjeverktøy mens du balanserer lokal lagring og onlinetjenester.
