## Progressiv avsløring: Forbedring av menneske-datamaskin-interaksjon i AI-produkter med "Less Is More"-filosofien

> I AI-produktdesign bestemmer kvaliteten på brukerinndata ofte kvaliteten på resultatet. Denne artikkelen deler en "progressiv avsløring"-interaksjonstilnærming vi praktiserte i HagiCode-prosjektet. Gjennom trinn-for-trinn-veiledning, intelligent fullføring og umiddelbar tilbakemelding, gjør den brukernes korte og vage innspill til strukturerte tekniske forslag, noe som forbedrer effektiviteten i menneske-datamaskin-interaksjonen betydelig.

## Bakgrunn

Alle som bygger AI-produkter har sannsynligvis sett denne situasjonen: en bruker åpner appen din og skriver entusiastisk inn en forespørsel på én linje, bare for at AI skal returnere noe helt utenfor målet. Det er ikke det at AI ikke er smart nok. Brukeren ga rett og slett ikke nok informasjon. Tankelesing er vanskelig for alle.

Dette problemet ble spesielt tydelig mens vi bygde HagiCode. HagiCode er en AI-drevet kodeassistent der brukere beskriver krav på naturlig språk for å lage tekniske forslag og økter. Ved faktisk bruk fant vi ut at brukerinndata ofte har disse problemene:

- **Inkonsekvent inndatakvalitet**: noen brukere skriver bare noen få ord, for eksempel "optimaliser pålogging" eller "fiks feil", uten nødvendig kontekst
- **Inkonsekvent teknisk terminologi**: forskjellige brukere bruker forskjellige termer for det samme; noen sier "frontend" mens andre sier "FE"
- **Mangler strukturert informasjon**: det er ingen prosjektbakgrunn, depotomfang eller påvirkningsomfang, selv om dette er kritiske detaljer
- **Gjentatte problemer**: de samme typene forespørsler dukker opp igjen og igjen, og hver gang må de forklares fra bunnen av

Det direkte resultatet er forutsigbart: AI har vanskeligere for å forstå forespørselen, forslagskvaliteten blir ustabil, og brukeropplevelsen lider. Brukere tenker «Denne kunstige intelligensen er ikke særlig bra», mens vi føler oss urettferdig klandret. Hvis du bare gir meg én setning, hvordan skal jeg da gjette hva du egentlig vil?

I sannhet er dette forståelig. Selv mennesker trenger tid til å forstå hverandre, og maskiner er intet unntak.

For å løse disse smertepunktene tok vi en dristig beslutning: introduser designprinsippet "progressiv avsløring" for å forbedre menneske-datamaskin-interaksjon. Endringene dette medførte var sannsynligvis større enn du kunne forestille deg. For å være ærlig, forventet vi ikke at det skulle være så effektivt på det tidspunktet.

## Hva er progressiv avsløring

"Progressive disclosure" er et designprinsipp fra HCI-feltet (human-computer interaction). Kjerneideen er enkel: **ikke vis brukere all informasjon og alternativer samtidig. Avslør i stedet bare det som er nødvendig steg for steg, basert på brukerens handlinger og behov**.

Dette prinsippet er spesielt egnet for AI-produkter fordi AI-interaksjon er naturlig progressiv. Brukeren sier litt, AI-en forstår litt, så legger brukeren til mer, og AI-en forstår mer. Det er veldig likt hvordan mennesker kommuniserer med hverandre: forståelsen utvikler seg vanligvis gradvis.

I HagiCodes scenario brukte vi progressiv avsløring på fire områder:

### 1. Beskrivelsesoptimeringsmekanisme: la AI hjelpe deg å si ting tydeligere

Når en bruker skriver inn en kort beskrivelse, sender vi den ikke direkte til AI for tolkning. I stedet utløser vi først en "beskrivelsesoptimalisering"-flyt. Kjernen i denne flyten er "strukturert utgang": konvertering av brukerens frie tekst til et standardformat. Det er som å strenge løse perler inn i et kjede så alt blir lettere å forstå.

Den optimaliserte beskrivelsen må inneholde følgende standardseksjoner:

- **Bakgrunn**: problembakgrunnen og konteksten
- **Analyse**: teknisk analyse og resonnement
- **Løsning**: løsnings- og implementeringstrinn
- **Praksis**: konkrete kodeeksempler og notater

Samtidig genererer vi automatisk en Markdown-tabell som viser informasjon som mållageret, stier og redigeringstillatelser, noe som gjør påfølgende AI-operasjoner enklere. En oversiktlig katalog gjør alltid ting lettere å finne.

Her er den faktiske implementeringen:

```csharp
// Core method in ProposalDescriptionMemoryService.cs
public async Task<string> OptimizeDescriptionAsync(
    string title,
    string description,
    string locale = "zh-CN",
    DescriptionOptimizationMemoryContext? memoryContext = null,
    CancellationToken cancellationToken = default)
{
    // Build query parameters
    var queryContext = BuildQueryContext(title, description);

    // Retrieve historical context
    var memoryContext = await RetrieveHistoricalContextAsync(queryContext, cancellationToken);

    // Generate a structured prompt
    var prompt = await BuildOptimizationPromptAsync(
        title,
        description,
        memoryContext,
        cancellationToken);

    // Call AI for optimization
    return await _aiService.CompleteAsync(prompt, cancellationToken);
}
```

Nøkkelen til denne flyten er "minneinjeksjon". Vi injiserer historisk kontekst som prosjektkonvensjoner, lignende tilfeller og negative mønstre i ledeteksten, slik at AI kan referere tidligere erfaringer under optimalisering. Erfaring skal ikke gå til spille.

**Merknader:**
- Sørg for at gjeldende inndata prioriteres over historisk minne, slik at eksplisitt spesifisert brukerinformasjon ikke overstyres
- HagIndex-referanser må behandles som faktakilder og må ikke endres av historiske saker
- Korreksjonsforslag med lav tillit bør ikke injiseres som sterke begrensninger

### 2. Mulighet for taleinndata: Å snakke er mer naturlig enn å skrive

I tillegg til tekstinntasting støtter vi også taleinntasting. Dette er spesielt nyttig for å beskrive komplekse krav. Det kan ta minutter å skrive en teknisk forespørsel, mens det å si det høyt kan ta bare noen få dusin sekunder.

Det sentrale designfokuset for stemmeinndata er "statsstyring". Brukere må tydelig vite hvilken tilstand systemet er i for øyeblikket. Vi definerte følgende tilstander:

- **Inaktiv**: systemet er klart og opptaket kan starte
- **Venter oppstrøms**: systemet kobler til backend-tjenesten
- **Opptak**: brukerens stemme blir tatt opp
- **Behandler**: tale blir konvertert til tekst
- **Feil**: en feil oppstod og trenger brukerens oppmerksomhet

Frontend-tilstandsmodellen ser omtrent slik ut:

```typescript
interface VoiceInputState {
  status: 'idle' | 'waiting-upstream' | 'recording' | 'processing' | 'error';
  duration: number;
  error?: string;
  deletedSet: Set<string>; // Fingerprint set of deleted results
}

// State transition when recording starts
const handleVoiceInputStart = async () => {
  // Enter waiting state first and show a loading animation
  setState({ status: 'waiting-upstream' });

  // Wait for backend readiness confirmation
  const isReady = await waitForBackendReady();
  if (!isReady) {
    setState({ status: 'error', error: 'Backend service is not ready' });
    return;
  }

  // Start recording
  setState({ status: 'recording', startTime: Date.now() });
};

// Handle recognition results
const handleRecognitionResult = (result: RecognitionResult) => {
  const fingerprint = normalizeFingerprint(result.text);

  // Check whether it has already been deleted
  if (state.deletedSet.has(fingerprint)) {
    return; // Skip deleted content
  }

  // Merge the result into the text box
  appendResult(result);
};
```

Det er en viktig detalj her: vi bruker et "fingeravtrykksett" for å administrere slettesynkronisering. Når talegjenkjenning returnerer flere resultater, kan brukere slette noen av dem. Vi lagrer fingeravtrykk av slettet innhold slik at hvis det samme innholdet dukker opp igjen senere, hoppes det over automatisk. Det er i hovedsak en måte å huske hva brukeren allerede har avvist.

### 3. Rask styringssystem: eksternaliser AIs "hjerne"

HagiCode har et fleksibelt ledeteksthåndteringssystem der alle spørsmål lagres som filer:

```
prompts/
├── metadata/
│   ├── optimize-description.zh-CN.json
│   └── optimize-description.en-US.json
└── templates/
    ├── optimize-description.zh-CN.hbs
    └── optimize-description.en-US.hbs
```

Hver forespørsel består av to deler:
- **Metadatafil (`.json`)**: definerer informasjon som spørsmålsscenario, versjon og parametere
- **Malfil (`.hbs`)**: det faktiske ledetekstinnholdet, skrevet med Handlebars-syntaks

Metadatafilformatet ser slik ut:

```json
{
  "scenario": "optimize-description",
  "locale": "zh-CN",
  "version": "1.0.0",
  "syntax": "handlebars",
  "syntaxVersion": "1.0",
  "parameters": [
    {
      "name": "title",
      "type": "string",
      "required": true,
      "description": "Proposal title"
    },
    {
      "name": "description",
      "type": "string",
      "required": true,
      "description": "Original description"
    }
  ],
  "author": "HagiCode Team",
  "description": "Optimize the user's technical proposal description",
  "lastModified": "2026-04-05",
  "tags": ["optimization", "nlp"]
}
```

Malfilen bruker Handlebars-syntaks og støtter parameterinjeksjon:

```handlebars
You are a technical proposal expert.

<task>
Generate a structured technical proposal description based on the following information.
</task>

<input>
<title>{{title}}</title>
<description>{{description}}</description>
{{#if memoryContext}}
<memory_context>
{{memoryContext}}
</memory_context>
{{/if}}
</input>

<output_format>
## Background
[Describe the problem background and context, including project information, repository scope, and so on]

## Analysis
[Provide the technical analysis and reasoning process, and explain why this change is needed]

## Solution
[Provide the solution and implementation steps, listing the key code locations]

## Practice
[Provide concrete code examples and notes]
</output_format>
```

Fordelene med dette designet er klare:
- ledetekster kan versjonskontrolleres akkurat som kode
- flere språk støttes og kan byttes automatisk basert på brukerens preferanser
- den parameteriserte designen gjør at kontekst kan injiseres dynamisk
- fullstendighet kan valideres ved oppstart, og unngår kjøretidsfeil

Hvis kunnskap bare forblir i noens hode, er det lett å miste. Å spille det inn på en strukturert måte fra begynnelsen er mye tryggere.

### 4. Progressiv veiviser: del opp komplekse oppgaver i små trinn

For komplekse oppgaver, som førstegangsinstallasjon og konfigurasjon, bruker vi en flertrinns veiviserdesign. Hvert trinn ber bare om nødvendig informasjon og gir klare fremdriftsindikatorer. Store oppgaver blir mye mer håndterbare når de håndteres ett trinn av gangen.

Veivisertilstandsmodellen:

```typescript
interface WizardState {
  currentStep: number;           // 0-3, corresponding to 4 steps
  steps: WizardStep[];
  canGoNext: boolean;
  canGoBack: boolean;
  isLoading: boolean;
  error: string | null;
}

interface WizardStep {
  id: number;
  title: string;
  description: string;
  completed: boolean;
}

// Step navigation logic
const goToNextStep = () => {
  if (wizardState.currentStep < wizardState.steps.length - 1) {
    // Validate input for the current step
    if (validateCurrentStep()) {
      wizardState.currentStep++;
      wizardState.steps[wizardState.currentStep - 1].completed = true;
    }
  }
};

const goToPreviousStep = () => {
  if (wizardState.currentStep > 0) {
    wizardState.currentStep--;
  }
};
```

Hvert trinn har sin egen valideringslogikk, og fullførte trinn får tydelige visuelle markører. Avbryting åpner en bekreftelsesdialog for å forhindre at brukere mister fremgang gjennom utilsiktede handlinger.

## Konklusjon

Når vi ser tilbake på HagiCodes progressive avsløringspraksis, kan vi oppsummere flere kjerneprinsipper:

1. **Trinn-for-trinn veiledning**: del komplekse oppgaver i mindre trinn og be om kun nødvendig informasjon på hvert trinn
2. **Intelligent fullføring**: bruk historisk kontekst og prosjektkunnskap for å fylle ut informasjon automatisk
3. **Øyeblikkelig tilbakemelding**: gi hver handling tydelig visuell tilbakemelding og statushint
4. **Feiltoleransemekanismer**: tillat brukere å angre og tilbakestille slik at feil ikke fører til irreversibelt tap
5. **Inndatamangfold**: støtter flere inndatametoder som tekst og stemme

I HagiCode var det praktiske resultatet av denne tilnærmingen tydelig: den gjennomsnittlige lengden på brukerinndata økte fra færre enn 20 tegn til strukturerte beskrivelser på 200-300 tegn, kvaliteten på AI-genererte forslag ble betydelig forbedret, og brukertilfredsheten økte samtidig.

Dette er ikke overraskende. Jo mer informasjon brukerne gir, jo mer nøyaktig kan AI forstå dem, og jo bedre resultater kan den returnere. Sånn sett er det ikke veldig forskjellig fra kommunikasjon mellom mennesker.

Hvis du også bygger AI-relaterte produkter, håper jeg disse erfaringene gir nyttig inspirasjon. Husk: brukere nekter ikke nødvendigvis å gi informasjon. Oftere har produktet ennå ikke stilt de riktige spørsmålene på riktig måte. Kjernen i progressiv avsløring er å finne den beste timingen og formen for disse spørsmålene.
