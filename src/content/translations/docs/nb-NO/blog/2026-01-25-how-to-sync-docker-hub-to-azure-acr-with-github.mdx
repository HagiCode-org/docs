## Automatisering av bildesynkronisering fra Docker Hub til Azure ACR

> Denne artikkelen forklarer hvordan du bruker GitHub Actions og bildesynkroniseringsverktøyet til å automatisere bildesynkronisering fra Docker Hub til Azure Container Registry, og adresserer den langsomme tilgangen til Docker Hub i Kina og enkelte Azure-regioner, samtidig som bildetilgjengelighet og distribusjonseffektivitet i Azure-miljøer forbedres.

HagiCode-prosjektet bruker Docker-bilder som sine kjernekjøringskomponenter, med de primære bildene som vert på Docker Hub. Etter hvert som prosjektet har vokst og Azure-implementeringsbehovet har økt, møtte vi følgende smertepunkter:

- Lave bildetrekkhastigheter, med Docker Hub-tilgang begrenset i Kina og noen Azure-regioner
- En enkelt bildekilde skaper et enkelt punkt for feilrisiko
- Azure Container Registry gir bedre nettverksytelse og integrasjon når du kjører i Azure

For å løse disse problemene trengte vi en automatisert bildesynkroniseringsmekanisme for regelmessig å synkronisere bilder fra Docker Hub til Azure ACR, for å sikre at brukere kan få raskere bildetrekkhastigheter og høyere tilgjengelighet i Azure-miljøer.

## Sammenligning av tekniske løsninger

Når vi definerte løsningen, sammenlignet vi flere tekniske alternativer:

### 1. bildesynkronisering (endelig valg)
- Inkrementell synkronisering: synkroniserer bare endrede bildelag, noe som reduserer nettverksoverføringen betydelig
- Gjenoppta støtte: synkroniseringen kan fortsette etter nettverksavbrudd
- Samtidig kontroll: konfigurerbare arbeidertråder forbedrer effektiviteten ved synkronisering av store bilder
- Robust feilhåndtering: innebygd prøvemekanisme for feil (3 forsøk som standard)
- Lettvekts distribusjon: en enkelt binær uten avhengigheter
- Støtte for flere register: kompatibel med Docker Hub, Azure ACR, Harbor og mer

### 2. Docker CLI
- Ingen inkrementell synkronisering: hele bildeinnholdet må hentes hver gang
- Lavere effektivitet: stort nettverksoverføringsvolum og lengre utførelsestid
- Enkel å bruke: basert på kjent `docker pull` og `docker push` kommandoer

### 3. Azure CLI
- Høyere kompleksitet: Azure CLI-autentisering må konfigureres
- Begrenset funksjonalitet: `az acr import` er relativt snevert i omfang
- Innebygd integrasjon: integreres godt med Azure-tjenester

## Implementeringstrinn

### 1. Forberedelse
- Opprett eller bekreft et Azure Container Registry i Azure Portal
- Opprett ACR-tilgangslegitimasjon (brukernavn og passord)
- Bekreft tilgangstillatelser for Docker Hub-bildelageret

### 2. Konfigurer GitHub Secrets
Legg til følgende hemmeligheter i GitHub-depotinnstillingene:
- AZURE_ACR_USERNAME: Azure ACR-brukernavn
- AZURE_ACR_PASSWORD: Azure ACR-passord

### 3. Lag arbeidsflyten for GitHub Actions
Konfigurer arbeidsflyten i `.github/workflows/sync-docker-acr.yml`:
- Planlagt trigger: hver dag kl. 00:00 UTC
- Manuell utløser: støtter `workflow_dispatch`
- Ekstra utløser: kjører på trykk til `publish` gren for rask synkronisering

### 4. Utførelsesflyt for arbeidsflyt

| Trinn | Deltaker | Handling | Beskrivelse |
| --- | --- | --- | --- |
| 1 | GitHub-handlinger | Utløs arbeidsflyten | Utløses av tidsplan, manuell utførelse eller push til `publish` gren |
| 2 | GitHub-handlinger → bildesynkronisering | Last ned og utfør synkroniseringsverktøyet | Går inn i selve synkroniseringsfasen |
| 3 | bildesynkronisering → Docker Hub | Hent bildemanifester og merkeliste | Leser metadata fra kildelageret |
| 4 | bildesynkronisering → Azure ACR | Hent eksisterende bildedata fra mållageret | Bestemmer gjeldende tilstand for målsiden |
| 5 | bildesynkronisering | Sammenlign kilde- og målforskjeller | Identifiserer bildelagene som trenger synkronisering |
| 6 | bildesynkronisering → Docker Hub | Trekk endrede bildelag | Overfører kun innholdet som trenger oppdatering |
| 7 | bildesynkronisering → Azure ACR | Trykk på endrede bildelag | Fullfører inkrementell synkronisering |
| 8 | bildesynkronisering → GitHub-handlinger | Returner synkroniseringsstatistikk | Inkluderer resultater, forskjeller og feilinformasjon |
| 9 | GitHub-handlinger | Registrer logger og last opp artefakter | Støtter senere revisjon og feilsøking |

## GitHub Actions arbeidsflytimplementering

Nedenfor er den faktiske arbeidsflytkonfigurasjonen som er i bruk (`.github/workflows/sync-docker-acr.yml`):

```yaml
name: Sync Docker Image to Azure ACR

on:
  schedule:
    - cron: "0 0 * * *" # Every day at 00:00 UTC
  workflow_dispatch: # Manual trigger
  push:
    branches: [publish]

permissions:
  contents: read

jobs:
  sync:
    runs-on: ubuntu-latest

    steps:
      - name: Checkout code
        uses: actions/checkout@v4

      - name: Download image-syncer
        run: |
          # Download the image-syncer binary
          wget https://github.com/AliyunContainerService/image-syncer/releases/download/v1.5.5/image-syncer-v1.5.5-linux-amd64.tar.gz
          tar -zxvf image-syncer-v1.5.5-linux-amd64.tar.gz
          chmod +x image-syncer

      - name: Create auth config
        run: |
          # Generate the authentication config file (YAML format)
          cat > auth.yaml <<EOF
          hagicode.azurecr.io:
            username: "${{ secrets.AZURE_ACR_USERNAME }}"
            password: "${{ secrets.AZURE_ACR_PASSWORD }}"
          EOF

      - name: Create images config
        run: |
          # Generate the image sync config file (YAML format)
          cat > images.yaml <<EOF
          docker.io/newbe36524/hagicode: hagicode.azurecr.io/hagicode
          EOF

      - name: Run image-syncer
        run: |
          # Run synchronization (using the newer --auth and --images parameters)
          ./image-syncer --auth=./auth.yaml --images=./images.yaml --proc=10 --retries=3

      - name: Upload logs
        if: always()
        uses: actions/upload-artifact@v4
        with:
          name: sync-logs
          path: image-syncer-*.log
          retention-days: 7
```

## Konfigurasjonsdetaljer

### 1. Triggerforhold
- Planlagt utløser: `cron: "0 0 * * *"` kjører hver dag kl. 00:00 UTC
- Manuell utløser: `workflow_dispatch` lar brukere kjøre det manuelt fra GitHub UI
- Trykk avtrekkeren: `push: branches: [publish]` utløses ved trykk til utløsergrenen for rask synkronisering

### 2. Autentiseringskonfigurasjon (`auth.yaml`)
```yaml
hagicode.azurecr.io:
  username: "${{ secrets.AZURE_ACR_USERNAME }}"
  password: "${{ secrets.AZURE_ACR_PASSWORD }}"
```

### 3. Konfigurasjon av bildesynkronisering
```yaml
docker.io/newbe36524/hagicode: hagicode.azurecr.io/hagicode
```
Denne konfigurasjonen betyr at alle tagger fra `docker.io/newbe36524/hagicode` vil bli synkronisert til `hagicode.azurecr.io/hagicode`.

### 4. bildesynkroniseringsparametere
- `--auth=./auth.yaml`: bane til autentiseringskonfigurasjonsfilen
- `--images=./images.yaml`: bane til konfigurasjonsfilen for bildesynkronisering
- `--proc=10`: bruk `10` samtidige arbeidere
- `--retries=3`: mislykket forsøk på nytt `3` ganger

## Notater og beste praksis

### 1. Sikkerhetsanbefalinger
- Roter Azure ACR-passord regelmessig (anbefalt hver 90. dag)
- Bruk en dedikert ACR-tjenestekonto med push-only tillatelser
- Overvåk ACR-tilgangslogger for å oppdage unormal tilgang i tide
- Ikke skriv ut legitimasjon i logger
- Ikke overfør legitimasjon til kodelageret

### 2. Ytelsesoptimalisering
- Juster `--proc` parameter: juster samtidighet basert på nettverksbåndbredde (anbefalt `5-20`)
- Overvåk synkroniseringsvarigheten: hvis synkroniseringen tar for lang tid, bør du vurdere å redusere samtidigheten
- Rydd opp logger regelmessig: angi en rimelig `retention-days` verdi (`7` for øyeblikket)

### 3. Feilsøking

#### Problem 1: Autentiseringsfeil
```
Error: failed to authenticate to hagicode.azurecr.io
```
Løsning:
1. Sjekk om GitHub Secrets er riktig konfigurert
2. Bekreft om Azure ACR-passordet er utløpt
3. Bekreft at tillatelsene for ACR-tjenestekontoen er riktige

#### Problem 2: Nettverkstidsavbrudd
```
Error: timeout waiting for response
```
Løsning:
1. Sjekk nettverkstilkoblingen
2. Reduser antall samtidige arbeidere (den `--proc` parameter)
3. Vent til nettverket gjenoppretter seg og kjør arbeidsflyten på nytt

#### Problem 3: Ufullstendig bildesynkronisering
```
Warning: some tags failed to sync
```
Løsning:
1. Sjekk synkroniseringsloggene for å identifisere feilslåtte tagger
2. Kjør arbeidsflyten på nytt manuelt for å synkronisere igjen
3. Bekreft at kildebildet på Docker Hub er sunt

### 4. Overvåking og varsling
- Sjekk siden Handlinger regelmessig for å bekrefte arbeidsflytstatus
- Aktiver GitHub-varsler slik at arbeidsflytfeil rapporteres umiddelbart
- Overvåk Azure ACR-lagringsbruk
- Kontroller regelmessig tagkonsistens på tvers av registre

## Ofte stilte spørsmål og løsninger

### Spørsmål 1: Hvordan synkroniserer jeg spesifikke tagger i stedet for alle tagger?

Endre `images.yaml` konfigurasjonsfil:
```yaml
# Sync only the latest and v1.0 tags
docker.io/newbe36524/hagicode:latest: hagicode.azurecr.io/hagicode:latest
docker.io/newbe36524/hagicode:v1.0: hagicode.azurecr.io/hagicode:v1.0
```

### Q2: Hvordan synkroniserer jeg flere bildelager?

Legg til flere linjer til `images.yaml`:
```yaml
docker.io/newbe36524/hagicode: hagicode.azurecr.io/hagicode
docker.io/newbe36524/another-image: hagicode.azurecr.io/another-image
```

### Spørsmål 3: Hvordan prøver jeg på nytt etter at synkroniseringen mislykkes?
- Automatisk forsøk på nytt: bildesynkronisering inkluderer en innebygd prøvemekanisme (`3` prøver på nytt som standard)
- Manuelt forsøk på nytt: klikk `Re-run all jobs` på GitHub Actions-siden

### Q4: Hvordan kan jeg se detaljert synkroniseringsfremgang?
- Se sanntidslogger på Handlinger-siden
- Last ned `sync-logs` artefakt for å inspisere de fullstendige loggfilene
- Loggfilene inkluderer synkroniseringsstatus og overføringshastighet for hver tag

### Q5: Hvor lang tid tar synkronisering?
- Innledende full synkronisering: vanligvis `10-30` minutter avhengig av bildestørrelse
- Inkrementell synkronisering: vanligvis `2-5` minutter hvis bildeendringene er små
- Timing avhenger av nettverksbåndbredde, bildestørrelse og samtidighetsinnstillinger
