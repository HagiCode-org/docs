## Rask koderedigering i nettleseren: VSCode-nettintegrering i praksis

> Etter at AI er ferdig med å analysere kode, hvordan åpner du umiddelbart en editor i nettleseren og begynner å gjøre endringer? Denne artikkelen deler vår praktiske erfaring med å integrere kodeserver i HagiCode-prosjektet for å skape en sømløs bro mellom AI-assistenten og koderedigeringsopplevelsen.

## Bakgrunn

I en tid med AI-assistert programmering må utviklere ofte inspisere og redigere kode raskt. Den tradisjonelle arbeidsflyten er enkel: åpne prosjektet i en skrivebords-IDE, finn filen, rediger den og lagre. Men i noen situasjoner føles den flyten alltid litt dårlig.

Scenario 1: fjernutvikling. Når du bruker en AI-assistent som HagiCode, kan backend kjøre på en ekstern server eller inne i en container, og lokale maskiner kan ikke få direkte tilgang til prosjektfilene. Hver gang du trenger å inspisere eller endre kode, må du koble til via SSH eller en annen metode, og opplevelsen føles fragmentert. Det er som å ville møte noen gjennom en tykk glassrute: du kan se dem, men du kan ikke nå dem.

Scenario to: raske forhåndsvisninger. Etter at AI-assistenten har analysert koden, vil brukeren kanskje bare raskt bla gjennom en fil eller gjøre en liten endring. Å lansere en full desktop IDE føles tung, mens en lett editor i nettleseren passer bedre behovet for et "rask utseende". Tross alt, hvem vil mobilisere en hel verktøykjede bare for å ta et blikk?

Scenario tre: samarbeid på tvers av enheter. Når du arbeider på tvers av forskjellige enheter, gir en nettleserbasert editor et enhetlig tilgangspunkt uten at hver maskin må konfigureres med et utviklingsmiljø. Det alene sparer mye trøbbel. Livet er kort; hvorfor gjenta det samme oppsettarbeidet om og om igjen?

For å løse disse smertepunktene, integrerte vi VSCode Web i HagiCode-prosjektet. Dette lar AI-assistenten og koderedigeringsopplevelsen kobles sømløst sammen: etter at AI analyserer koden, kan brukere umiddelbart åpne en editor og gjøre endringer i samme nettleserøkt, uten å bytte applikasjon. Det er den typen opplevelse der, når du trenger den, er den rett og slett der.

## Om HagiCode

HagiCode er en AI-drevet kodeassistent designet for å forbedre utviklingseffektiviteten gjennom naturlig språkinteraksjon. Under utviklingen fant vi ut at brukere ofte må bytte raskt mellom AI-analyse og koderedigering, noe som presset oss til å utforske hvordan vi kan integrere editoren direkte i nettleseren.

Prosjektlager: [github.com/HagiCode-org/site](https://github.com/HagiCode-org/site)

## Teknisk valg: Hvorfor kodeserver?

Blant de mange VSCode Web-løsningene som er tilgjengelige, valgte vi kodeserver. Det var noen få konkrete årsaker bak denne avgjørelsen.

Funksjons fullstendighet. kodeserver er nettversjonen av VSCode og støtter de fleste skrivebordsfunksjoner, inkludert utvidelsessystemet, intelligente forslag, feilsøking og mer. Det betyr at brukere kan få en redigeringsopplevelse i nettleseren som er veldig nær desktopversjonen. Tross alt, hvem vil egentlig gå på akkord med funksjonalitet?

Fleksibel distribusjon. kodeserver kan kjøres som en uavhengig tjeneste og støtter også Docker-basert distribusjon, som passer godt med HagiCodes arkitektur. Backend vår er skrevet i C#, frontend bruker React, og de to kommuniserer med kodeservertjenesten gjennom REST APIer. Det er som å bygge med blokker: hver brikke har sin plass.

Sikker autentisering. kodeserver inkluderer en innebygd `connection-token` mekanisme for å forhindre uautorisert tilgang. Hver økt har et unikt token slik at bare autoriserte brukere kan åpne redigeringsprogrammet. Sikkerhet er en av de tingene du først setter full pris på når du har den.

## Arkitektur design

HagiCodes VSCode Web-integrasjon bruker en front-end/back-end-separert arkitektur.

### Frontend Service Layer

Frontend bryter interaksjoner med backend gjennom `vscodeServerService.ts`:

```typescript
// Open project
export async function openProjectInCodeServer(
  id: string,
  currentInterfaceLanguage?: string,
): Promise<VsCodeServerLaunchResponseDto>

// Open vault
export async function openVaultInCodeServer(
  id: string,
  path?: string,
  currentInterfaceLanguage?: string,
): Promise<VsCodeServerLaunchResponseDto>
```

Forskjellen mellom disse to metodene er enkel: `openProjectInCodeServer` åpner hele prosjektet, mens `openVaultInCodeServer` åpner en bestemt bane inne i et hvelv. For MonoSpecs multi-repository-prosjekter oppretter systemet automatisk en arbeidsområdefil. Klart ansvar er ofte nok når hver del gjør sin egen jobb godt.

### Backend Service Layer

Bakenden `VaultAppService.cs` implementerer kjernelogikken:

```csharp
public async Task<VsCodeServerLaunchResponseDto> OpenInCodeServerAsync(
    string id,
    string? relativePath = null,
    string? currentInterfaceLanguage = null,
    CancellationToken cancellationToken = default)
{
    // 1. Get settings and check whether the feature is enabled
    var settings = await _vsCodeServerSettingsService.GetResolvedSettingsAsync(cancellationToken);
    if (!settings.Enabled) {
        throw new BusinessException(VsCodeServerErrorCodes.Disabled, "VSCode Server is disabled.");
    }

    // 2. Get vault and resolve the launch directory
    var vault = await RequireVaultAsync(id, cancellationToken);
    var launchDirectory = ResolveLaunchDirectory(vault, relativePath);

    // 3. Ensure code-server is running and get runtime info
    var runtime = await _vsCodeServerManager.EnsureStartedAsync(settings, cancellationToken);

    // 4. Resolve language settings
    var language = _vsCodeServerSettingsService.ResolveLaunchLanguage(
        settings.Language,
        currentInterfaceLanguage);

    // 5. Build launch URL
    return new VsCodeServerLaunchResponseDto {
        LaunchUrl = AppendQueryString(runtime.BaseUrl, new Dictionary<string, string?> {
            ["folder"] = launchDirectory,
            ["tkn"] = runtime.ConnectionToken,
            ["vscode-lang"] = language,
        }),
        ConnectionToken = runtime.ConnectionToken,
        OpenMode = "folder",
        Runtime = VsCodeServerSettingsService.MapRuntime(
            await _vsCodeServerManager.GetRuntimeSnapshotAsync(cancellationToken)),
    };
}
```

Denne metoden har et veldig klart ansvar: sjekk innstillinger, løs stier, start tjenesten og bygg URL. Blant dem `ResolveLaunchDirectory` metoden utfører stisikkerhetskontroller for å forhindre banegjennomgangsangrep. Kode kan føles litt som poesi når hver linje har en hensikt.

### Automatisk Runtime Management

Backend administrerer kodeserverprosessen gjennom `VsCodeServerManager`:

- Sjekk prosessstatus
- Start stoppede tjenester automatisk
- Returner øyeblikksbilder av kjøretid som port, prosess-ID og starttid

Denne utformingen lar systemet automatisk håndtere kodeserverens livssyklus, slik at brukere ikke trenger å administrere tjenesteprosesser manuelt. Livet er allerede komplisert nok; alt som kan automatiseres bør være det.

### Språksynkronisering

HagiCode støtter et flerspråklig grensesnitt, og kodeserveren må følge den innstillingen. Systemet støtter tre språkmoduser:

- `follow`: Følg gjeldende grensesnittspråk
- `zh-CN`: fast til kinesisk
- `en-US`: fast til engelsk

Innstillingen sendes til kodeserveren gjennom `vscode-lang` URL-parameter slik at redigeringsspråket forblir konsistent med HagiCode-grensesnittet. Språket føles best når det er enhetlig.

### MonoSpecs Multi-Repository Workspace

For MonoSpecs-prosjekter, som inneholder flere deldepoter i én monorepo, oppretter systemet automatisk en `.code-workspace` fil:

```csharp
private async Task<string> CreateWorkspaceFileAsync(Project project, Guid projectId)
{
    var folders = await ResolveWorkspaceFoldersAsync(project.Path);
    var workspaceDocument = new {
        folders = folders.Select(path => new { path }).ToArray(),
    };
    // Generate workspace file...
}
```

Dette gjør det mulig å redigere flere sub-repositories i samme kodeserverforekomst, noe som er spesielt praktisk for store monorepo-prosjekter. Flere depoter i ett vindu kan føles som flere historier samlet i samme bok.

## Frontend-integrasjon

HagiCode-grensesnittet bruker React + TypeScript, og integrering av kodeserver er ikke spesielt komplisert.

### Hurtighandlingsknapp

Legg til en kodeserverknapp på prosjektkortet:

```tsx
// QuickActionsZone.tsx
<Button
  size="sm"
  variant="default"
  onClick={() => onAction({ type: 'open-code-server' })}
>
  <Globe className="h-3 w-3 mr-1" />
  <span className="text-xs">{t('project.openCodeServer')}</span>
</Button>
```

Denne knappen utløser den åpne handlingen og kaller opp backend-APIet for å få lanserings-URLen. Én knapp, én handling, direkte og enkelt.

### Håndtering av den åpne handlingen

```tsx
const handleAction = async (action: ProjectAction) => {
  if (action.type === 'open-code-server') {
    const response = await openProjectInCodeServer(project.id, i18n.language);
    window.open(response.launchUrl, '_blank', 'noopener,noreferrer');
  }
};
```

Bruk `window.open` for å åpne kodeserver i en ny fane. Den `noopener,noreferrer` parametere gir ekstra sikkerhet. Når det kommer til sikkerhet, er det ikke noe som heter å være for forsiktig.

### Vault Editing Entry

Legg til en lignende redigeringsknapp i Vault-listen:

```tsx
const handleEditVault = async (vault: VaultItemDto) => {
  const response = await openVaultInCodeServer(vault.id);
  window.open(response.launchUrl, '_blank', 'noopener,noreferrer');
};
```

Prosjekter og hvelv bruker den samme åpne mekanismen, som holder samhandlingen konsistent. Konsistens betyr nesten like mye som selve funksjonen.

## URL-konstruksjonslogikk

URL-formatet for kodeserver har noen få detaljer som er verdt å merke seg.

**Mappemodus**:
```
http://{host}:{port}/?folder={path}&tkn={token}&vscode-lang={lang}
```

**Arbeidsområdemodus**:
```
http://{host}:{port}/?workspace={workspacePath}&tkn={token}&vscode-lang={lang}
```

Her, `tkn` er tilkoblingssymbolet. Den genereres automatisk hver gang kodeserveren starter, noe som sikrer sikker tilgang. Den `vscode-lang` parameter kontrollerer språket for redigeringsgrensesnittet. Hver og en av disse parameterne har en rolle å spille.

## Sikkerhetshensyn

Når du implementerer kode-server-integrasjon, fortjener sikkerhet spesiell oppmerksomhet. Hvis sikkerheten går galt, merker du alltid for sent.

### Tilkoblingstoken

Den `connection-token` genereres tilfeldig og skal ikke eksponeres. Det brukes best under HTTPS for å forhindre at tokenet blir fanget opp av en mann-i-midten. Sensitiv informasjon er verdt å beskytte ordentlig.

### Filbanesikkerhet

Backend implementerer banegjennomgangssjekker:

```csharp
private static string ResolveLaunchDirectory(VaultRegistryEntry vault, string? relativePath)
{
    var vaultRoot = EnsureTrailingSeparator(Path.GetFullPath(vault.PhysicalPath));
    var combinedPath = Path.GetFullPath(Path.Combine(vaultRoot, relativePath ?? "."));
    if (!combinedPath.StartsWith(vaultRoot, StringComparison.OrdinalIgnoreCase))
    {
        throw new BusinessException(VaultRelativePathTraversalCode, "Relative path traversal detected.");
    }
    return combinedPath;
}
```

Denne koden sikrer at brukere ikke kan bruke `../` eller lignende mønstre for å få tilgang til filer utenfor Vault-katalogen. Grensekontroll er alltid bedre utført enn å hoppe over.

### Tillatelseskontroll

Kodeserverprosessen bør kjøres med passende brukertillatelser slik at den ikke får tilgang til sensitive systemfiler. Det er best å kjøre kodeservertjenesten under en dedikert bruker. Tillatelseskontroll er en av de grunnleggende tingene du alltid bør ha på plass.

## Ytelsesoptimalisering

kodeserver bruker serverressurser, så her er noen optimaliseringsforslag:

- Overvåk CPU- og minnebruk, og juster ressursgrensene når det er nødvendig
- Store prosjekter kan kreve lengre tidsavbrudd
- Implementer automatisk timeout-opprydding for økter for å frigjøre ressurser
- Vurder hurtigbufring for å redusere gjentatt beregning

HagiCode gir en kjøretidsstatusovervåkings-API, og frontend kan ringe `getVsCodeServerSettings()` for å hente gjeldende tilstand:

```typescript
const { settings, runtime } = await getVsCodeServerSettings();
// runtime.status: 'disabled' | 'stopped' | 'starting' | 'running' | 'unhealthy'
// runtime.baseUrl: "http://localhost:8080"
// runtime.processId: 12345
```

Denne utformingen lar brukere tydelig forstå helsestatusen til kodeserveren og raskt finne problemer når noe går galt. Når statusen er synlig, føler folk seg mer i kontroll.

## Detaljer om brukeropplevelse

Under implementeringen oppdaget vi noen få detaljer som merkbart påvirker brukeropplevelsen og fortjener ekstra oppmerksomhet.

Å åpne kodeserver for første gang kan kreve å vente på oppstart, og denne forsinkelsen kan variere fra noen få sekunder til et halvt minutt. Det er en god idé å vise en lastestatus i frontend slik at brukerne vet at systemet fortsatt fungerer. Å vente er lettere når det er tilbakemelding.

Nettlesere kan blokkere popup-vinduet, så brukere bør bli bedt om å tillate det manuelt. Ved første lansering viser HagiCode veiledning som forklarer hvordan du gir de nødvendige nettlesertillatelsene. Brukeropplevelsen bor ofte i akkurat disse små detaljene.

Det er også en god idé å vise kjøretidsstatus som start, kjører eller feil, slik at når det oppstår problemer, kan brukere raskt se om problemet er på serversiden eller i deres egen drift. Å vite hvor problemet er gir deg i det minste et sted å begynne.

## Konfigurasjonseksempel

Konfigurasjonen for kodeserver er ikke komplisert:

```json
{
  "vscodeServer": {
    "enabled": true,
    "host": "0.0.0.0",
    "port": 8080,
    "language": "follow"
  }
}
```

`enabled` kontrollerer om funksjonen er slått på, `host` og `port` definere lytteadressen, og `language` stiller inn språkmodus. Disse innstillingene kan endres gjennom brukergrensesnittet og trer i kraft umiddelbart. Enkle ting er ofte de enkleste å bruke.
