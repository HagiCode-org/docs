## Hvordan automatisere Steam-utgivelser med GitHub-handlinger

> Denne artikkelen deler den komplette løsningen vi implementerte for automatiserte Steam-utgivelser i HagiCode Desktop-prosjektet, som dekker ende-til-ende-automatiseringsflyten fra GitHub Release til Steam-plattformen, inkludert viktige tekniske detaljer som Steam Guard-autentisering og multi-plattform Depot-opplastinger.

## Bakgrunn

Utgivelsesarbeidsflyten på Steam er faktisk ganske forskjellig fra tradisjonell applikasjonsdistribusjon. Steam har sitt eget komplette leveringssystem for oppdateringer. Utviklere må bruke SteamCMD for å laste opp byggeartefakter til Steams CDN-nettverk, i stedet for å bare slippe inn en nedlastingslenke som på andre plattformer.

HagiCode Desktop-prosjektet forbereder seg på en Steam-utgivelse, som introduserte noen nye utfordringer for utgivelsesarbeidsflyten vår:

1. Vi trengte å konvertere eksisterende byggeartefakter til et Steam-kompatibelt format.
2. Vi måtte laste dem opp til Steam gjennom SteamCMD.
3. Vi måtte også håndtere Steam Guard-autentisering.
4. Vi trengte å støtte depotopplastinger med flere plattformer for Linux, Windows og macOS.
5. Vi ønsket en helautomatisert flyt fra GitHub Release til Steam.

Prosjektet hadde allerede implementert "bærbar versjonsmodus", som lar applikasjonen oppdage faste nyttelaster pakket i `extra` katalog. Målet vårt var å integrere den bærbare versjonsmodusen sømløst med Steam-distribusjon.

## Om HagiCode

Løsningen som er delt i denne artikkelen kommer fra vår praktiske erfaring i [HagiCode](https://hagicode.com) prosjekt. HagiCode er en AI-kodingsassistent som støtter skrivebordsbruk. Siden vi jobber aktivt mot lansering på Steam, trengte vi å etablere en pålitelig automatisert utgivelsesarbeidsflyt.

## Arkitektur design

Kjernen i hele Steam-utgivelsesprosessen er en GitHub Actions-arbeidsflyt som deler prosessen inn i tre hovedtrinn:

```
┌─────────────────────────────────────────────────────────────┐
│ GitHub Actions Workflow (Steam Release)                    │
├─────────────────────────────────────────────────────────────┤
│ 1. Preparation Stage:                                      │
│    - Check out portable-version code                       │
│    - Download build artifacts from GitHub Release          │
│    - Extract and prepare the Steam content directory       │
│                                                             │
│ 2. SteamCMD Setup:                                         │
│    - Install or reuse SteamCMD                             │
│    - Authenticate with Steam Guard                         │
│                                                             │
│ 3. Release Stage:                                          │
│    - Generate Depot VDF configuration files                │
│    - Generate App Build VDF configuration files            │
│    - Invoke SteamCMD to upload to Steam                    │
└─────────────────────────────────────────────────────────────┘
```

Denne designen gir flere fordeler:
- Den gjenbruker eksisterende GitHub Release-artefakter og unngår å gjenoppbygge de samme utgangene.
- Den bruker selvhostede løpere for sikker isolasjon.
- Den støtter veksling mellom forhåndsvisningsmodus og formelle utgivelsesgrener.
- Den inkluderer fullstendig feilhåndtering og logging, noe som gjør feil lettere å diagnostisere.

## Arbeidsflytimplementering

### Utløserparameterdesign

Arbeidsflyten vår støtter følgende nøkkelparametere:

```yaml
inputs:
  release:           # Portable Version release tag
    description: 'Version tag to release (for example v1.0.0)'
    required: true
  steam_preview:     # Whether to generate a preview build
    description: 'Whether to use preview mode'
    required: false
    default: 'false'
  steam_branch:      # Steam branch to set live
    description: 'Target Steam branch'
    required: false
    default: 'preview'
  steam_description: # Build description override
    description: 'Build description'
    required: false
```

### Self-hosted Runner Configuration

Av sikkerhetsmessige årsaker bruker vi en selvhostet løper med `steam` etikett:

```yaml
runs-on:
  - self-hosted
  - Linux
  - X64
  - steam
```

Dette sikrer at Steam-utgivelsene kjører på en dedikert løper og holder sensitive legitimasjoner trygt isolert.

### Samtidig kontroll

For å forhindre at utgivelser av samme versjon forstyrrer hverandre, konfigurerte vi samtidighetskontroll:

```yaml
concurrency:
  group: portable-version-steam-${{ github.event.inputs.release }}
  cancel-in-progress: false
```

Legg merke til det `cancel-in-progress: false` er satt her fordi Steam-utgivelser kan ta en stund, og vi vil ikke at en nylig utløst kjøring skal avbryte en som allerede lastes opp.

## Kjerneskriptimplementering

### Forbereder utgivelsesinnganger

Den `prepare-steam-release-input.mjs` script er ansvarlig for å forberede inndataene som kreves for utgivelse:

```javascript
// Download the GitHub Release build manifest and artifact inventory
const buildManifest = await downloadBuildManifest(releaseTag);
const artifactInventory = await downloadArtifactInventory(releaseTag);

// Download compressed archives for each platform
for (const platform of ['linux-x64', 'win-x64', 'osx-universal']) {
  const artifactUrl = getArtifactUrl(artifactInventory, platform);
  await downloadArtifact(artifactUrl, platform);
}

// Extract into the Steam content directory structure
await extractToSteamContent(sources, contentRoot);
```

### Steam Guard-autentisering

Steam krever at kontoer beskyttes med Steam Guard, så vi implementerte en delt-hemmelighetsbasert kodegenereringsalgoritme:

```javascript
function generateSteamGuardCode(sharedSecret, timestamp = Date.now()) {
  const secret = decodeSharedSecret(sharedSecret);
  const time = Math.floor(timestamp / 1000 / 30);

  const timeBuffer = Buffer.alloc(8);
  timeBuffer.writeBigUInt64BE(BigInt(time));

  // Use HMAC-SHA1 to generate a time-based one-time code
  const hash = crypto.createHmac('sha1', secret)
    .update(timeBuffer)
    .digest();

  // Convert it into a 5-character Steam Guard code
  const code = steamGuardCode(hash);
  return code;
}
```

Denne implementeringen er basert på Steam Guards TOTP (Time-based One-Time Password)-mekanisme, som genererer en ny bekreftelseskode hvert 30. sekund.

### VDF-konfigurasjonsgenerering

VDF (Valve Data Format) er konfigurasjonsformatet som brukes av Steam. Vi må generere to typer VDF-filer:

**Depot VDF** brukes til å konfigurere innhold for hver plattform:

```javascript
function buildDepotVdf(depotId, contentRoot) {
  return [
    '"DepotBuildConfig"',
    '{',
    `  "DepotID" "${escapeVdf(depotId)}"`,
    `  "ContentRoot" "${escapeVdf(contentRoot)}"`,
    '  "FileMapping"',
    '  {',
    '    "LocalPath" "*"',
    '    "DepotPath" "."',
    '    "recursive" "1"',
    '  }',
    '}'
  ].join('\n');
}
```

**App Build VDF** brukes til å konfigurere hele applikasjonsbygget:

```javascript
function buildAppBuildVdf(appId, depotBuilds, description, setLive) {
  const vdf = [
    '"appbuild"',
    '{',
    `  "appid" "${appId}"`,
    `  "desc" "${escapeVdf(description)}"`,
    `  "contentroot" "${escapeVdf(contentRoot)}"`,
    '  "buildoutput" "build_output"',
    '  "depots"',
    '  {'
  ];

  for (const [depotId, depotVdfPath] of Object.entries(depotBuilds)) {
    vdf.push(`    "${depotId}" "${depotVdfPath}"`);
  }

  if (setLive) {
    vdf.push(`  }`);
    vdf.push(`  "setlive" "${setLive}"`);
  }

  vdf.push('}');
  return vdf.join('\n');
}
```

### SteamCMD-påkallelse

Til slutt utføres opplastingen ved å påkalle SteamCMD:

```javascript
await runCommand(steamcmdPath, [
  '+login', steamUsername, steamPassword, steamGuardCode,
  '+run_app_build', appBuildPath,
  '+quit'
]);
```

Dette er det siste hoppet i hele arbeidsflyten. Når det lykkes, er utgivelsen ferdig.

## Flerplattformdepothåndtering

Steam bruker Depot-systemet til å administrere innhold for forskjellige plattformer. Vi støtter tre hoveddepoter:

| Plattform | Depotidentifikator | Arkitekturstøtte |
|-----|-----------|---------|
| Linux | `linux-x64` | x64_64 |
| Windows | `win-x64` | x64_64 |
| macOS | `osx-universal` | universal, x64_64, arm64 |

Hvert depot har sin egen innholdskatalog og VDF-konfigurasjonsfil. Dette sikrer at brukere på ulike plattformer kun laster ned innholdet de faktisk trenger.

## Slipp arbeidsflyt

### Trinn 1: Forbered GitHub-utgivelsen

Lag først en GitHub-utgivelse i `portable-version` depot som inkluderer:
- Komprimerte arkiver for hver plattform
- Bygg manifest (`{tag}.build-manifest.json`)
- Artefaktbeholdning (`{tag}.artifact-inventory.json`)

### Trinn 2: Utløs Steam Release Workflow

Utløs arbeidsflyten manuelt gjennom GitHub Actions og fyll inn de nødvendige parameterne:
- `release`: versjonskoden som skal publiseres, for eksempel `v1.0.0`
- `steam_branch`: målgrenen, som f.eks `preview` eller `public`
- `steam_preview`: om du skal bruke forhåndsvisningsmodus

### Trinn 3: Utfør utgivelsesflyten automatisk

Arbeidsflyten utfører automatisk følgende trinn:

1. Last ned og pakk ut GitHub Release-artefakter.
2. Installer eller oppdater SteamCMD.
3. Generer Steam VDF-konfigurasjonsfiler.
4. Autentiser med Steam Guard.
5. Last opp innhold til Steam CDN.
6. Sett den angitte grenen live.

Når denne sekvensen er fullført, dekkes hele utgivelsesbanen.

### Nødvendig konfigurasjon av hemmeligheter

Konfigurer følgende hemmeligheter i GitHub-depotinnstillingene:

| Hemmelig navn | Beskrivelse |
|-----------|------|
| `STEAM_USERNAME` | Steam-konto brukernavn |

| `STEAM_DEPOT_ID_WINDOWS` | Windows Depot ID |
| `STEAM_DEPOT_ID_MACOS` | macOS Depot ID |

Det er ingenting spesielt uvanlig med disse innstillingene. Du trenger ganske enkelt alle de forventede verdiene på plass.

## Beste praksis

### Steam Guard Authentication Management

På den første kjøringen må du skrive inn Steam Guard-koden manuelt. Etter det anbefales det å konfigurere den delte hemmeligheten slik at koden kan genereres automatisk. Dette unngår manuell inngripen ved hver utgivelse.

SteamCMD lagrer påloggingstokenet og kan gjenbruke det ved senere kjøringer. Du må fortsatt holde et øye med tokens utløp, fordi re-autentisering kreves etter at tokenet utløper.

### Innholdskatalogstruktur

Sørg for at Steam-innholdskatalogstrukturen er riktig:

```
steam-content/
├── linux-x64/     # Linux platform content
├── win-x64/       # Windows platform content
└── osx-universal/ # macOS universal binary content
```

Hver katalog skal inneholde de fullstendige applikasjonsfilene for den tilhørende plattformen.

### Bruk av forhåndsvisningsmodus

Forhåndsvisningsmodus setter ingen gren live, så den er egnet for testing og validering:

```yaml
if [ "$STEAM_PREVIEW_INPUT" = 'true' ]; then
  cmd+=(--preview)
fi
```

Dette lar deg laste opp til Steam først for verifisering, og deretter bytte til den formelle grenen etter at alt er sjekket ut.

### Feilhåndtering og logging

Skriptene inkluderer fullstendig feilhåndtering og logging:
- Bekreft at GitHub-utgivelsen eksisterer.
- Sjekk nødvendige metadatafiler.
- Sørg for at plattforminnhold er tilstede.
- Generer en GitHub Actions sammendragsrapport.

Denne informasjonen er svært verdifull for både feilsøking og revisjon.

### Artefakthåndtering

Arbeidsflyten genererer to typer artefakter:
- `portable-steam-release-preparation-{tag}`: metadata for utgivelsesforberedelse
- `portable-steam-build-metadata-{tag}`: Steam bygge metadata

Disse artefaktene kan brukes til senere revisjon og feilsøking. En oppbevaringsperiode på 30 dager er en praktisk standard.

## Bruk i den virkelige verden

I HagiCode-prosjektet har denne automatiserte utgivelsesarbeidsflyten allerede kjørt på tvers av flere versjoner. Hele banen fra GitHub Release til Steam-plattformen er helautomatisert og krever ingen manuell intervensjon.

Dette forbedret både utgivelseseffektiviteten og påliteligheten betydelig. Tidligere tok det mer enn 30 minutter å publisere én versjon manuelt. Nå er hele prosessen ferdig på bare noen få minutter.

Enda viktigere, den automatiserte arbeidsflyten reduserer sjansen for menneskelige feil. Hver utgivelse følger den samme standardiserte prosessen, noe som gjør resultatene mer forutsigbare.
