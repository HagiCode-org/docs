## Bygge elegante nye brukere ombord i React-prosjekter: HagiCodes driver.js-praksis

> Når brukere åpner produktet ditt for første gang, vet de virkelig hvor de skal begynne? I denne artikkelen vil jeg snakke litt om hvordan vi brukte driver.js for onboarding av nye brukere i HagiCode-prosjektet. Betrakt det som et lite praktisk eksempel for å få i gang samtalen.

## Bakgrunn

Har du noen gang havnet i denne situasjonen? En ny bruker registrerer seg for produktet ditt, åpner siden og ser umiddelbart tapt ut. De skanner rundt, usikre på hva de skal klikke eller hva de skal gjøre videre. Som utviklere antar vi ofte at brukerne bare vil "utforske på egenhånd" fordi, tross alt, menneskelig nysgjerrighet er grenseløs. Men virkeligheten er annerledes: de fleste brukere går stille i løpet av minutter hvis de ikke finner det rette inngangspunktet, som om historien begynner plutselig og slutter like naturlig.

Ny bruker onboarding er en viktig måte å løse dette problemet på, men å bygge det godt er ikke så enkelt. Et godt onboarding-system må:

- Finn sideelementer nøyaktig og marker dem
- Støtt flertrinns onboarding-flyter
- Husk brukerens valg (fullfør eller hopp over)
- Unngå å påvirke sideytelsen og normal interaksjon
- Hold kodestrukturen klar og enkel å vedlikeholde

Mens vi bygde HagiCode, møtte vi den samme utfordringen. HagiCode er en AI-kodingsassistent, og kjernearbeidsflyten er en OpenSpec-arbeidsflyt som ser slik ut: "brukeren oppretter et forslag -> AI-en genererer en plan -> brukeren vurderer den -> AI-en utfører den." For brukere som møter dette konseptet for første gang, er arbeidsflyten helt ny, så de trenger solid onboarding for å komme raskt i gang. Nye ting tar alltid litt tid å venne seg til.

## Hvorfor vi valgte driver.js

I løpet av den tekniske evalueringsfasen så vi på flere mainstream onboarding-biblioteker. Hver av dem hadde sine egne styrker:

- **Intro.js**: Kraftig, men relativt stor, og stiltilpasning er noe kompleks
- **Shepherd.js**: Godt utformet API, men litt for "tung" for vår brukssituasjon
- **driver.js**: Lett, kortfattet, intuitiv API og fungerer godt i React-økosystemet

Til slutt valgte vi driver.js. Det var ingen spesielt dramatisk grunn. Valget kom hovedsakelig ned til disse hensynene:

1. **Lett**: Kjernebiblioteket er lite og øker ikke buntstørrelsen vesentlig
2. **Enkel API**: Konfigurasjonen er oversiktlig og intuitiv, så den er lett å plukke opp
3. **Fleksibel**: Støtter tilpasset posisjonering, styling og interaksjonsatferd
4. **Dynamisk import**: Kan lastes på forespørsel uten å påvirke ytelsen på første skjerm

Med teknologivalg finnes det sjelden et universelt best svar. Vanligvis er det bare det alternativet som passer best.

## Teknisk implementering

### Kjernekonfigurasjon

driver.js har en veldig intuitiv konfigurasjonsmodell. Her er kjernekonfigurasjonen vi bruker i HagiCode-prosjektet:

```typescript
import { driver } from 'driver.js';
import 'driver.js/dist/driver.css';

const newConversationDriver = driver({
  allowClose: true,           // Allow users to close the guide
  animate: true,              // Enable animations
  overlayClickBehavior: 'close', // Close the guide when the overlay is clicked
  disableActiveInteraction: false, // Keep elements interactive
  showProgress: false,        // Do not show the progress bar (we manage progress ourselves)
  steps: guideSteps           // Array of guide steps
});
```

Begrunnelsen bak disse innstillingene er:

- `allowClose: true` - Respekter brukerens valg og ikke tving dem til å fullføre veiledningen
- `disableActiveInteraction: false` – Noen trinn krever reelle brukerhandlinger, for eksempel å skrive inn, så interaksjon kan ikke deaktiveres
- `overlayClickBehavior: 'close'` - Gi brukerne en rask måte å avslutte

### Statens ledelse

Vedvarende ombordstigningsstatus er kritisk. Vi ønsker ikke å starte guiden på nytt hver gang siden oppdateres, fordi det blir irriterende fort. HagiCode bruker `localStorage` for å administrere guidestatus:

```typescript
export type GuideState = 'pending' | 'dismissed' | 'completed';

export interface UserGuideState {
  session: GuideState;
  detailGuides: Record<string, GuideState>;
}

// Read state
export const getUserGuideState = (): UserGuideState => {
  const state = localStorage.getItem('userGuideState');
  return state ? JSON.parse(state) : { session: 'pending', detailGuides: {} };
};

// Update state
export const setUserGuideState = (state: UserGuideState) => {
  localStorage.setItem('userGuideState', JSON.stringify(state));
};
```

Vi definerer tre tilstander:

- `pending`: Veiledningen pågår fortsatt, og brukeren har ikke fullført eller hoppet over den
- `dismissed`: Brukeren lukket veiledningen proaktivt
- `completed`: Brukeren fullførte alle trinnene

For introduksjon av forslagsdetaljside støtter vi også mer finmasket tilstandssporing gjennom `detailGuides` kart, fordi ett forslag kan gå gjennom flere stadier - utkast, gjennomgang og utførelse fullført - og hvert trinn trenger forskjellig veiledning. Tingenes tilstand er alltid i endring, og onboarding bør gjenspeile det.

### Valg av målelement

driver.js bruker CSS-velgere for å finne målelementer. HagiCode følger en enkel konvensjon: bruk en egendefinert `data-guide` attributt for å markere onboarding-mål:

```typescript
const steps = [
  {
    element: '[data-guide="launch"]',
    popover: {
      title: 'Start a New Conversation',
      description: 'Click here to create a new conversation session...'
    }
  }
];
```

I komponenter ser det slik ut:

```typescript
<button data-guide="launch" onClick={handleLaunch}>
  New Conversation
</button>
```

Fordelene med denne tilnærmingen er:

- Unngå konflikter med navn på forretningsstylingklassene
- Tydelig semantikk, slik at du umiddelbart kan se at elementet er relatert til onboarding
- Enklere å administrere og vedlikeholde konsekvent

### Dynamisk importoptimalisering

Fordi onboarding-funksjonen bare er nødvendig i spesifikke scenarier, for eksempel en brukers første besøk, bruker vi dynamisk import for å optimalisere den første lasteytelsen:

```typescript
const initNewUserGuide = async () => {
  // Dynamically import driver.js
  const { driver } = await import('driver.js');
  await import('driver.js/dist/driver.css');

  // Initialize the guide
  const newConversationDriver = driver({
    // ...configuration
  });

  newConversationDriver.drive();
};
```

På denne måten lastes driver.js og stilarket bare inn når det er nødvendig og påvirker ikke ytelsen på første skjerm. Det er ikke mange som liker å vente på noe de ikke engang trenger ennå.

## Onboarding Flow Design

HagiCode implementerer to onboarding-stier som dekker brukerens kjernescenarier.

### Økt onboarding (10 trinn)

Denne ombordstigningsstien hjelper brukere med å fullføre hele flyten fra å opprette en samtale til å sende inn sitt første komplette forslag:

1. **lansering** - Start veiledningen og introduser "Ny samtale"-knappen
2. **skriv** - Veiled brukeren til å skrive inn en forespørsel i inndataboksen
3. **send** - Veiled brukeren til å klikke på send-knappen
4. **proposal-launch-readme** - Veiled brukeren til å lage et README-forslag
5. **proposal-compose-readme** - Veiled brukeren til å redigere README-forespørselsinnholdet
6. **proposal-submit-readme** - Veiled brukeren til å sende inn README-forslaget
7. **proposal-launch-agents** - Veiled brukeren til å opprette en `AGENTS.md` Forslag
8. **proposal-compose-agents** - Veiled brukeren til å redigere `AGENTS.md` forespørsel
9. **proposal-submit-agents** - Veiled brukeren til å sende inn `AGENTS.md` Forslag
10. **proposal-wait** - Forklar at AI-en behandler og brukeren bør vente et øyeblikk

Ideen bak denne banen er å la brukere oppleve HagiCodes kjernearbeidsflyt på egenhånd gjennom to reelle forslagsopprettingsoppgaver, en for `README` og en for `AGENTS.md`. Det er stor forskjell på å høre om en arbeidsflyt og å gå gjennom den selv.

### Introduksjon av forslagsdetaljer (3 trinn)

Når brukere går inn på Forslagets detaljside, utløser HagiCode den tilsvarende veiledningen basert på Forslagets nåværende tilstand:

1. **utkast** (utkaststadium) - Veiled brukeren til å se gjennom den AI-genererte planen
2. **gjennomgang** (gjennomgangsfase) - Veiled brukeren til å utføre planen
3. **executionCompleted** (fullført stadium) - Veiled brukeren til å arkivere planen

Den definerende egenskapen til denne veiledningen er at den er statsdrevet: den bestemmer dynamisk hvilket innføringstrinn som skal vises basert på forslagets faktiske tilstand. Ting endres, og onboarding bør endres med dem.

### Element Gjengivelse Prøvemekanisme på nytt

I React-applikasjoner kan det hende at mål-onboarding-elementet ikke er ferdig gjengitt ennå, for eksempel fordi asynkrone data fortsatt lastes inn. For å håndtere dette implementerer HagiCode en prøvemekanisme:

```typescript
const waitForElement = (selector: string, maxRetries = 10, interval = 100) => {
  let retries = 0;
  return new Promise<HTMLElement>((resolve, reject) => {
    const checkElement = () => {
      const element = document.querySelector(selector) as HTMLElement;
      if (element) {
        resolve(element);
      } else if (retries < maxRetries) {
        retries++;
        setTimeout(checkElement, interval);
      } else {
        reject(new Error(`Element not found: ${selector}`));
      }
    };
    checkElement();
  });
};
```

Kall denne funksjonen før du initialiserer guiden for å sikre at målelementet allerede eksisterer. Noen ganger er det verdt det å vente litt lenger.

## Sammendrag av beste praksis

Basert på HagiCodes praktiske erfaring, her er noen viktige beste fremgangsmåter:

### 1. Onboarding skal være enkelt å avslutte

Ikke tving brukere til å fullføre onboarding. Noen brukere er oppdagelsesreisende av natur og foretrekker å finne ut av ting på egenhånd. Gi en tydelig "Hopp over"-knapp, husk valget deres, og ikke avbryt dem igjen neste gang.

### 2. Hold ombordinnholdet kort og skarpt

Hvert ombordstigningstrinn bør fokusere på ett enkelt mål:

- **Tittel**: Kort og tydelig, helst ikke mer enn noen få ord
- **Beskrivelse**: Gå rett på sak og fortell brukeren hva dette er og hvorfor det betyr noe

Unngå langdrykkede forklaringer. Brukeroppmerksomheten er begrenset under onboarding, og jo mer du sier, jo mindre sannsynlig er det at de leser det.

### 3. Hold velgerne stabile

Bruk stabile elementmarkører som ikke endres ofte. Skikken `data-guide` attributt er et godt valg. Unngå avhengig av klassenavn eller DOM-struktur, fordi de er enkle å endre under refaktorer. Koden endres hele tiden, men noen ankere bør holde seg stabile når det er mulig.

### 4. Test onboarding

HagiCode inkluderer komplette testtilfeller for onboarding-funksjonen:

```typescript
describe('NewUserConversationGuide', () => {
  it('should initialize guide state correctly', () => {
    const state = getUserGuideState();
    expect(state.session).toBe('pending');
  });

  it('should update guide state correctly', () => {
    setUserGuideState({ session: 'completed', detailGuides: {} });
    const state = getUserGuideState();
    expect(state.session).toBe('completed');
  });
});
```

Tester bidrar til å sikre at refaktorering ikke ved et uhell bryter ombordatferd. Ingen vil ha en liten kodeendring for å skade tidligere fungerende funksjonalitet.

### 5. Ytelsesoptimalisering

- Bruk dynamisk import for å la det laste inn onboarding-biblioteket
- Unngå å initialisere onboarding-logikk etter at brukeren allerede har fullført veiledningen
- Vurder ytelseseffekten av animasjoner, og deaktiver dem på lavere enheter om nødvendig

Ytelse, som mange ting i livet, fortjener litt nøye budsjettering.

## Konklusjon

Introduksjon av nye brukere er en viktig del av å forbedre produktbrukeropplevelsen. I HagiCode-prosjektet brukte vi driver.js til å bygge et komplett onboarding-system som dekker hele arbeidsflyten fra øktoppretting til utførelse av forslag.

Kjernepunktene vi håper å dele gjennom denne artikkelen er:

1. **Tekniske valg bør samsvare med faktiske behov**: driver.js er ikke det kraftigste alternativet, men det passer best for oss
2. **Statens ledelse er kritisk**: Bruk `localStorage` for å opprettholde onboarding-tilstanden og unngå å avbryte brukere gjentatte ganger
3. **Onboarding-design bør holde fokus**: Hvert trinn skal løse ett problem, og ikke mer
4. **Kodestrukturen bør være klar**: Separat onboarding-konfigurasjon, tilstandsadministrasjon og brukergrensesnittlogikk for å gjøre vedlikehold enklere

Hvis du legger til nye brukeronboarding i ditt eget prosjekt, håper jeg den praktiske erfaringen i denne artikkelen hjelper. Det er ikke noe spesielt mystisk med denne typen teknologi. Fortsett å prøve, fortsett å oppsummere det du lærer, og ting blir gradvis lettere.
