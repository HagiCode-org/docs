## Fra 120 sekunder til 45 sekunder: En praktisk veiledning for å optimalisere ytelsen til Vite Build med arbeidertråder

> Når du jobber med store frontend-prosjekter, kan produksjonsbygg føles smertefullt sakte. Denne artikkelen deler hvordan vi brukte Node.js Worker Threads for å redusere obfuskeringsstadiet i en Vite-bygg fra 120 sekunder til 45 sekunder, sammen med implementeringsdetaljene og lærdommene i HagiCode-prosjektet.

## Bakgrunn

I vår frontend-ingeniørpraksis ble problemer med byggeeffektivitet stadig mer fremtredende etter hvert som prosjektet vokste. Spesielt under produksjonsprosessen introduserer vi vanligvis JavaScript-obfuskeringsverktøy som f.eks `javascript-obfuscator` for å beskytte kildekodelogikken. Dette trinnet er nødvendig, men det er også beregningsmessig dyrt og tungt CPU-bundet.

Under det tidlige utviklingsstadiet av **HagiCode** møtte vi en veldig vanskelig ytelsesflaskehals: produksjonsbyggetidene ble raskt dårligere etter hvert som kodebasen vokste.

**De spesifikke smertepunktene var**:
- Obfuskeringsoppgaver kjørte serielt på en enkelt tråd, og makserte én CPU-kjerne mens de andre satt inaktive
- Byggetiden økte fra de opprinnelige 30 sekunder til 110–120 sekunder
- Byggverifiseringsløkken etter endring ble ekstremt lang, noe som bremset utviklingsiterasjonen alvorlig
- I CI/CD-pipelinen ble byggefasen den mest tidkrevende delen

**Hvorfor trengte HagiCode dette?**
HagiCode er en AI-drevet kodeassistent hvis frontend-arkitektur inkluderer kompleks forretningslogikk og AI-interaksjonsmoduler. For å sikre sikkerheten til kjernekoden vår, håndhevet vi høyintensitetsforvirring i produksjonsutgivelser. Overfor byggeventer som nærmer seg to minutter, bestemte vi oss for å utføre en dyp ytelsesoptimalisering av byggesystemet.

## Analyse: Finne gjennombruddspunktet i ytelsesflaskehalsen

Før vi løser ytelsesproblemet, trengte vi først å avklare tankegangen vår og identifisere den beste tekniske løsningen.

### Kjernebeslutning: Hvorfor velge arbeidstråder?

Det er tre hovedmåter for å oppnå parallell beregning i Node.js:
1. **barneprosess**: lag uavhengige underordnede prosesser
2. **Web Workers**: brukes hovedsakelig på nettlesersiden
3. **worker_threads**: innebygd multithreading-støtte i Node.js

Etter å ha sammenlignet alternativene, valgte HagiCode til slutt **Worker Threads** av følgende grunner:
- **Null serialiseringskostnader**: Arbeidstråder kjøres i samme prosess og kan dele minne gjennom `SharedArrayBuffer` eller overføre eierskap, unngå de store serialiseringskostnadene ved kommunikasjon mellom prosesser.
- **Native support**: innebygd i Node.js 12+ uten behov for ekstra tungvektsavhengigheter.
- **Enhetlig kontekst**: feilsøking og logging er mer praktisk enn med underordnede prosesser.

### Oppgavegranularitet: Hvordan bør obfuskeringsoppgaver deles?

Det er vanskelig å parallellisere tilsløringen av en enorm JS-buntfil fordi koden har avhengigheter, men Vite-byggutdata er sammensatt av flere **biter**. Det gir oss en naturlig parallell grense:

- **Uavhengighet**: Etter Vite-emballasje er avhengigheter mellom forskjellige deler allerede frakoblet, slik at de kan behandles trygt parallelt.
- **Passelig granularitet**: prosjekter har vanligvis 10-30 biter, som er en utmerket skala for parallell planlegging.
- **Enkel integrasjon**: den `generateBundle` Hook in Vite-plugins lar oss fange opp og behandle disse bitene før filene sendes ut.

### Arkitektur design

Vi designet et parallelt prosesseringssystem med fire kjernekomponenter:

1. **Task Splitter**: itererer over Vites buntobjekt, filtrerer ut filer som ikke trenger tilsløring, for eksempel leverandørbiter, og genererer en oppgavekø.
2. **Worker Pool Manager**: administrerer Workers livssyklus og håndterer oppgavedistribusjon, resirkulering og forsøk på nytt ved feil.
3. **Progress Reporter**: utdata bygger fremgang i sanntid for å redusere venteangst.
4. **ObfuscationWorker**: Worker-tråden som faktisk utfører obfuscation-logikken.

## Løsning: Praktisk implementering

Basert på analysen ovenfor begynte vi å implementere dette parallelle obfuskasjonssystemet.

### 1. Konfigurer Vite-plugin-modulen

Først integrerte vi parallell obfuscation-plugin i `vite.config.ts`. Konfigurasjonen er grei. Du trenger bare å spesifisere antall arbeidere og obfuskeringsreglene.

```typescript
import { defineConfig } from 'vite'
import { parallelJavascriptObfuscator } from './buildTools/plugin'

export default defineConfig(({ mode }) => {
  const isProduction = mode === 'production'
  
  return {
    build: {
      rollupOptions: {
        ...(isProduction
          ? {
              plugins: [
                parallelJavascriptObfuscator({
                  enabled: true,
                  // Automatically adjust based on CPU core count; leave one core for the main thread
                  workerCount: 4, 
                  retryAttempts: 3,
                  fallbackToMainThread: true, // Automatically degrade to single-thread mode on failure
                  // Filter out vendor chunks; third-party libraries usually do not need obfuscation
                  isVendorChunk: (fileName: string) => fileName.includes('vendor-'),
                  obfuscationConfig: {
                    compact: true,
                    controlFlowFlattening: true,
                    deadCodeInjection: true,
                    disableConsoleOutput: true,
                    // ... more obfuscation options
                  },
                }),
              ],
            }
          : {}),
      },
    },
  }
})
```

### 2. Implementer Worker Logic

En arbeider er enheten som utfører oppgaver. Vi må definere input- og outputdatastrukturene tydelig.

**Merk**: selv om koden her er enkel, er det flere fallgruver å passe på, for eksempel å sjekke om `parentPort` er null og håndterer feil riktig. I HagiCodes implementering fant vi ut at visse spesielle ES6-syntaksmønstre kunne få obfuscatoren til å krasje, så vi la til `try-catch` beskyttelse.

```typescript
import { parentPort } from 'worker_threads'
import javascriptObfuscator from 'javascript-obfuscator'

export interface ObfuscationTask {
  chunkId: string
  code: string
  config: any
}

export interface ObfuscationResult {
  chunkId: string
  obfuscatedCode: string
  error?: string
}

// Listen for tasks sent from the main thread
if (parentPort) {
  parentPort.on('message', async (task: ObfuscationTask) => {
    try {
      // Perform obfuscation
      const obfuscated = javascriptObfuscator.obfuscate(task.code, task.config)
      const result: ObfuscationResult = {
        chunkId: task.chunkId,
        obfuscatedCode: obfuscated.getObfuscatedCode(),
      }
      // Send the result back to the main thread
      parentPort?.postMessage(result)
    } catch (error) {
      // Handle exceptions so one Worker crash does not block the whole build
      const result: ObfuscationResult = {
        chunkId: task.chunkId,
        obfuscatedCode: '',
        error: error instanceof Error ? error.message : 'Unknown error',
      }
      parentPort?.postMessage(result)
    }
  })
}
```

### 3. Worker Pool Manager

Dette er kjernen i hele løsningen. Vi må opprettholde en Worker-pool med fast størrelse og planlegge oppgaver ved å bruke en **FIFO (først inn, først ut)**-strategi.

```typescript
import { Worker } from 'worker_threads'
import os from 'os'

export class WorkerPool {
  private workers: Worker[] = []
  private taskQueue: Array<{
    task: ObfuscationTask
    resolve: (result: ObfuscationResult) => void
    reject: (error: Error) => void
  }> = []
  
  constructor(options: WorkerPoolOptions = {}) {
    // Default to core count - 1 so the main thread still has some breathing room
    const workerCount = options.workerCount ?? Math.max(1, (os.cpus().length || 4) - 1)
    
    for (let i = 0; i < workerCount; i++) {
      this.createWorker()
    }
  }

  private createWorker() {
    const worker = new Worker('./worker.ts')
    
    worker.on('message', (result) => {
      // After one task finishes, take the next task from the queue
      const nextTask = this.taskQueue.shift()
      if (nextTask) {
        this.dispatchTask(worker, nextTask)
      } else {
        // If there are no pending tasks, mark the Worker as idle
        this.activeWorkers.delete(worker)
      }
    })
    
    this.workers.push(worker)
  }

  // Submit a task to the pool
  public runTask(task: ObfuscationTask): Promise<ObfuscationResult> {
    return new Promise((resolve, reject) => {
      const job = { task, resolve, reject }
      const idleWorker = this.workers.find(w => !this.activeWorkers.has(w))
      
      if (idleWorker) {
        this.dispatchTask(idleWorker, job)
      } else {
        this.taskQueue.push(job)
      }
    })
  }

  private dispatchTask(worker: Worker, job: any) {
    this.activeWorkers.set(worker, job.task)
    worker.postMessage(job.task)
  }
}
```

### 4. Fremdriftsrapportering

Å vente er smertefullt, spesielt når du ikke aner hvor mye lengre tid det vil ta. Så vi la til en enkel fremdriftsrapporter for å gi sanntids tilbakemelding om gjeldende status.

```typescript
export class ProgressReporter {
  private completed = 0
  private readonly total: number
  private readonly startTime: number

  constructor(total: number) {
    this.total = total
    this.startTime = Date.now()
  }

  increment(): void {
    this.completed++
    this.report()
  }

  private report(): void {
    const now = Date.now()
    const elapsed = now - this.startTime
    const percentage = (this.completed / this.total) * 100
    
    // Simple ETA estimate
    const avgTimePerChunk = elapsed / this.completed
    const remaining = (this.total - this.completed) * avgTimePerChunk

    console.log(
      `[Parallel Obfuscation] ${this.completed}/${this.total} chunks completed (${percentage.toFixed(1)}%) | ETA: ${(remaining / 1000).toFixed(1)}s`
    )
  }
}
```

## Praksis: Resultater og lærdom

Etter å ha implementert denne løsningen, forbedret byggeytelsen til HagiCode-prosjektet umiddelbart.

### Ytelsesreferansedata

Vi testet i følgende miljø:
- CPU: Intel Core i7-12700K (12 kjerner / 20 tråder)
- RAM: 32 GB DDR4
- Node.js: v18.17.0
- OS: Ubuntu 22.04

**Resultatsammenligning**:
- **Entrådet (før optimalisering)**: 118 sekunder
- **4 arbeidere**: 55 sekunder (**53 %** forbedring)
- **8 arbeidere**: 48 sekunder (**60 %** forbedring)
- **12 arbeidere**: 45 sekunder (**62 %** forbedring)

Som du kan se, var ikke gevinstene lineære. Når antallet arbeidere oversteg 8, ble forbedringen mindre. Dette var hovedsakelig begrenset av jevnheten i oppgavedistribusjonen og flaskehalser for minnebåndbredde.

### Vanlige problemer og løsninger

I HagiCodes bruk i den virkelige verden har vi også støtt på flere fallgruver, så her er de for referanse:

**Q1: Byggetiden gikk ikke mye ned og ble til og med tregere?**
- **Årsak**: å opprette Workers har sin egen overhead, eller for mange Workers ble konfigurert, noe som forårsaker hyppige kontekstbytter.
- **Løsning**: vi anbefaler å sette antall arbeidere til `CPU core count - 1`. Sjekk også om en enkelt del er spesielt stor, for eksempel `> 5MB`. Den typen "monster"-fil vil bli flaskehalsen, så du må kanskje optimalisere kodedelingsstrategien.

**Spørsmål 2: Arbeidere krasjer av og til og forårsaker byggefeil?**
- **Årsak**: noen spesielle kodesyntaksmønstre kan forårsake interne feil inne i obfuskatoren.
- **Løsning**: vi implementerte en **automatisk degraderingsmekanisme**. Når en arbeider når feilterskelen, faller plugin-modulen automatisk tilbake til enkelttrådsmodus for å sikre at byggingen ikke stopper. Samtidig registrerer den filnavnet som forårsaket feilen, slik at den kan fikses senere.

**Q3: Minnebruken er for høy (OOM)?**
- **Årsak**: hver arbeider trenger sin egen minneplass for å laste obfuscatoren og analysere AST.
- **Løsning**:
  - Reduser antall arbeidere.
  - Øk Node.js-minnegrensen: `NODE_OPTIONS="--max-old-space-size=4096" npm run build`.
  - Pass på at arbeiderne ikke oppbevarer unødvendige referanser til store gjenstander.

## Konklusjon

Ved å introdusere Node.js Worker Threads har vi redusert produksjonsbyggetiden til HagiCode-prosjektet fra 120 sekunder til rundt 45 sekunder, noe som forbedret utviklingsopplevelsen og CI/CD-effektiviteten betraktelig.

Kjernen i denne løsningen er:
1. **Del opp oppgaver riktig**: bruk Vite-biter som parallell enhet.
2. **Kontrollressurser**: bruk en Worker-pool for å unngå ressursbruk.
3. **Design for feiltoleranse**: en automatisk degraderingsmekanisme sikrer byggestabilitet.
