## Fra Docusaurus 3.x til Astro 5.x: A Retrospective on the HagiCode Site Migration

> Denne artikkelen ser tilbake på vår fulle migrering av HagiCodes offisielle nettsted fra Docusaurus 3.x til Astro 5.x. Vi vil ta et dypdykk i hvordan Astro's Islands Architecture hjalp oss med å løse ytelsesflaskehalser samtidig som vi bevarte våre eksisterende React-komponentressurser, og levere forbedringer i både byggehastighet og lasteytelse.

## Bakgrunn

I januar 2026 utførte vi en "hjertetransplantasjon" på HagiCodes offisielle nettsted ved å fullstendig migrere kjernerammeverket fra Docusaurus 3.x til Astro 5.x. Dette var ikke en impulsiv omskrivning, men en nøye overveid teknisk beslutning.

Før migreringen var nettstedet vårt funksjonelt komplett, men det hadde begynt å vise noen klassiske "luksusproblemer": oppsvulmede byggeartefakter, overdreven JavaScript-nyttelast og mindre enn ideell sideinnlastingshastighet på komplekse dokumentasjonssider. Som et AI-kodeassistentprosjekt trenger HagiCode hyppig dokumentasjon og funksjonsoppdateringer, så byggeeffektivitet påvirker utgivelseshastigheten direkte. Samtidig ønsket vi at siden skulle være mer søkemotorvennlig (SEO) slik at flere utviklere kunne oppdage prosjektet.

For å løse disse smertepunktene tok vi en dristig beslutning: gjenoppbygg hele systemet på Astro. Virkningen av den avgjørelsen kan være enda større enn du forventer. Jeg vil komme inn på detaljene snart.

## Om HagiCode

Nettstedmigreringstilnærmingen som deles i denne artikkelen kommer fra vår praktiske erfaring i [HagiCode](https://github.com/HagiCode-org/site) prosjekt.

HagiCode er en AI-kodingsassistent fokusert på å forbedre utviklingseffektiviteten. Vi bryr oss ikke bare om å gjenta kjernefunksjonene, men også om utvikleropplevelsen. Denne nettstedsrefaktoren var også ment å gi brukerne den raskeste mulige opplevelsen når de surfer på dokumentene våre og den offisielle nettsiden.

## Hvorfor forlate det modne Docusaurus-økosystemet?

Innenfor React-økosystemet har Docusaurus lenge vært "standardsvaret" for dokumentasjonssteder. Den fungerer ut av boksen, tilbyr et rikt plugin-økosystem og har et aktivt fellesskap. Men etter hvert som HagiCode fikk flere funksjoner, følte vi også begrensningene:

1.  **Ytelsesflaskehalser**: Docusaurus er i utgangspunktet en React SPA (ensidig applikasjon). Selv om du bare skriver statiske sider, må klienten fortsatt laste inn React-kjøretiden og hydrere siden, noe som er unødvendig tungt for enkle dokumentsider.
2.  **Stor ressursstørrelse**: Selv når en side inneholder svært lite innhold, forblir den medfølgende JS-størrelsen relativt fast. Det er ikke ideelt for mobilbrukere eller dårlige nettverksforhold.
3.  **Begrenset fleksibilitet**: Selv om den er utvidbar, ønsket vi mer kontroll på lavt nivå over byggerørledningen.

Astro kom til akkurat rett tid for å løse disse problemene. Den introduserte en ny "Islands Architecture": Som standard genererer Astro statisk HTML med null JavaScript, og bare komponenter som krever interaktivitet blir "aktivert" og laster JS. Det betyr at det meste av nettstedet vårt blir ren HTML og laster ekstremt raskt.

## Core Migration Strategy: A Smooth Architectural Transition

Migrering var ikke bare kopier og lim. Det krevde et tankesettskifte. Vi gikk fra Docusaurus sin «all React»-modell til Astros «Core + Islands»-modell.

### 1. Gjenoppbygge konfigurasjonssystemet

Først måtte vi flytte fra `docusaurus.config.ts` til `astro.config.mjs`. Dette var ikke bare et filskifte, men en omskriving av ruting og byggelogikk.

I Docusaurus er alt en plugin. I Astro er alt en integrasjon. Vi trengte å redefinere nettstedets basisbane, bygge utgangsmodus (statisk vs SSR) og strategi for aktivaoptimalisering.

**Før migrering:**
```typescript
// docusaurus.config.ts
export default {
  title: 'HagiCode',
  url: 'https://hagicode.com',
  baseUrl: '/',
  // ... more configuration
};
```

**Etter migrering:**
```javascript
// astro.config.mjs
import { defineConfig } from 'astro/config';
import react from '@astrojs/react';

export default defineConfig({
  integrations: [react()],
  site: 'https://hagicode.com',
  base: '/',
  // Optimization settings for static assets
  build: {
    inlineStylesheets: 'auto',
  },
});
```

### 2. Hva du skal beholde og hva du skal refaktorere i React-komponenter

Dette var den mest smertefulle delen av migrasjonen. Vår eksisterende side hadde mange React-komponenter, for eksempel faner, kodeutheving, tilbakemeldingsknapper og mer. Å kaste dem ville være bortkastet, men å beholde alt ville gjøre JavaScript-nyttelasten for tung.

HagiCode tok i bruk en **progressiv hydrering**-strategi:
*   **Rene statiske komponenter**: For presentasjonsinnhold som topptekster, bunntekster og ren tekstdokumentasjon, skrev vi dem om som Astro-komponenter (`.astro` filer) og gjengitt dem direkte til HTML på byggetidspunktet.
*   **Interaktive øyer**: For komponenter som må forbli interaktive, som temabyttere, tabulatorbytte og kodeblokkkopiknapper, beholdt vi React-implementeringen og la til `client:load` eller `client:visible` direktiver.

For eksempel vår ofte brukte Tabs-komponent i dokumentasjonen:

```jsx
// src/components/Tabs.jsx
import { useState } from 'react';
import './Tabs.css'; // Import styles

export default function Tabs({ items }) {
  const [activeIndex, setActiveIndex] = useState(0);
  // ... state logic
  return (
    <div className="tabs-wrapper">
      {/* Rendering logic */}
    </div>
  );
}
```

Når den brukes i Markdown, sier vi eksplisitt til Astro: "Denne komponenten trenger JS."

```markdown
// src/content/docs/example.mdx
import Tabs from '../../components/Tabs.jsx';

<!-- Load JS only when the component enters the viewport -->
<Tabs client:visible items={...} />
```

På denne måten konkurrerer ikke interaktive komponenter utenfor visningsporten om båndbredde, noe som forbedrer lastehastigheten på første skjerm betydelig.

### 3. Tilpasning av stylingsystemet: Fra CSS-moduler til scoped CSS

Docusaurus støtter CSS-moduler som standard, mens Astro oppmuntrer Scoped CSS gjennom `<style>` tag. Kjerneideen bak begge er stilisolasjon, men syntaksen er forskjellig.

Under HagiCode-migreringen konverterte vi de fleste komplekse CSS-moduler til Astros scoped-stiler. Dette viste seg faktisk å være en god ting, fordi i `.astro` filer stilene og malene lever i samme fil, noe som gjør vedlikeholdet mer intuitivt.

**Før refaktorisering:**
```css
/* Tabs.module.css */
.wrapper { background: var(--ifm-background-color); }
```

**Etter refaktorering (Astro Scoped):**
```html
<!-- Tabs.astro -->
<div class="tabs-wrapper">
  <slot />
</div>

<style>
  .tabs-wrapper {
    /* Use CSS variables directly to adapt to the theme */
    background: var(--bg-color);
    padding: 1rem;
  }
</style>
```

Samtidig forenet vi det globale CSS-variabelsystemet og brukte Astros miljøbevisste muligheter for å sikre at mørk modus veksler jevnt på tvers av sider.

## Fallgruver vi traff i praksis og hvordan vi løste dem

Under selve HagiCode-migreringen møtte vi ganske mange problemer. Her er flere av de mest typiske.

### 1. Bane og miljø Variable smertepunkter

HagiCode støtter distribusjon av underbane, for eksempel distribusjon under en GitHub Pages-underkatalog. I Docusaurus, `baseUrl` håndteres automatisk. I Astro må vi imidlertid være mer forsiktige når vi håndterer bildelenker og API-forespørsler.

Vi introduserte en miljøvariabelmekanisme for å administrere dette konsekvent:

```javascript
// Handle paths in the build script
const getBasePath = () => import.meta.env.VITE_SITE_BASE || '/';
```

Pass på at du ikke hardkode baner som begynner med `/` i koden din. I utvikling versus produksjon, eller etter konfigurering av en `base` bane, kan dette føre til 404s for eiendeler.

### 2. CommonJS-skriptkompatibilitet

Vår gamle side hadde noen Node.js-skript brukt til oppgaver som automatisk henting av metrikkdata og oppdatering av områdekartet, og de ble skrevet i CommonJS (`require`). Astro og moderne byggeverktøy har fullt ut omfavnet ES-moduler (`import`/`export`).

Hvis du også har lignende skript, husk å refaktorisere dem alle til ES-moduler. Det er retningen økosystemet beveger seg, og jo før du gjør endringen, jo mindre problemer vil du ha senere.

```javascript
// Old way
const fs = require('fs');

// New way
import fs from 'fs';
```

### 3. Ikke glem SEO og omdirigeringer

Søkemotorer har allerede indeksert HagiCodes gamle Docusaurus-sider. Hvis du bytter direkte til Astro og URL-strukturen endres, kan du ende opp med et stort antall 404-er og et stort fall i søkerangeringen.

Vi konfigurerte omdirigeringsregler i Astro:

```javascript
// astro.config.mjs
export default defineConfig({
  redirects: {
    '/docs/old-path': '/docs/new-path',
    // Map old links to new links in bulk
  }
});
```

Eller du kan håndtere dette på serverkonfigurasjonslaget. Sørg for at gamle lenker kan 301 omdirigeres til de nye adressene, fordi dette er avgjørende for SEO.

## Sammendrag

For HagiCode var migrering fra Docusaurus til Astro ikke bare en oppgradering av rammeverket. Det var også en praktisk implementering av en "ytelse først"-filosofi.

**Hva vi fikk:**
*   **Enestående Lighthouse-score**: Etter migreringen nærmet HagiCode-nettstedets ytelsespoeng seg lett til en perfekt poengsum.
*   **Raskere byggehastighet**: Astros inkrementelle byggefunksjoner halverer utgivelsestiden for dokumentasjonsoppdateringer.
*   **Bevart fleksibilitet**: Med Islands Architecture ofret vi ingen interaktive funksjoner og kunne fortsatt bruke React der det var nødvendig.

Hvis du også vedlikeholder et dokumentasjonsorientert nettsted og sliter med buntstørrelse eller lastehastighet, er Astro vel verdt å prøve. Selv om migreringsprosessen krever noe kirurgi, som å gi nytt navn til PCode til HagiCode og flytte komponenter over én etter én, gjør den silkemyke brukeropplevelsen du får tilbake det absolutt verdt det.

Byggesystemet som deles i denne artikkelen er den nøyaktige tilnærmingen vi utviklet gjennom ekte prøving og feiling mens vi bygde HagiCode. Hvis du finner denne tilnærmingen verdifull, sier det noe om vår tekniske styrke, og HagiCode i seg selv er sannsynligvis også verdt en nærmere titt.
