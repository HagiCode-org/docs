## Bygge en GitHub-problemintegrering fra bunnen av: HagiCodes frontend-direkte tilkoblingspraksis

> Denne artikkelen dokumenterer hele prosessen med å integrere GitHub-problemer i HagiCode-plattformen. Vi vil utforske hvordan du bruker en "frontend direkte tilkobling + minimal backend"-arkitektur for å oppnå sikker OAuth-autentisering og effektiv problemsynkronisering samtidig som backend-en holdes lett.

## Bakgrunn: Hvorfor integrere GitHub?

Som en AI-assistert utviklingsplattform ligger HagiCodes kjerneverdi i å koble ideer med implementering. Men i faktisk bruk fant vi ut at etter at brukere har fullført et forslag i HagiCode, må de ofte manuelt kopiere innholdet inn i GitHub Issues for prosjektsporing.

Dette skaper flere åpenbare smertepunkter:

1.  **Fragmentert arbeidsflyt**: Brukere må bytte frem og tilbake mellom to systemer. Opplevelsen er ikke jevn, og nøkkelinformasjon kan lett gå tapt under kopiering og liming.
2.  **Upraktisk samarbeid**: Andre teammedlemmer er vant til å sjekke oppgaver på GitHub og kan ikke direkte se forslagsfremdriften inne i HagiCode.
3.  **Gjentatt manuelt arbeid**: Hver gang et forslag oppdateres, må noen manuelt oppdatere det tilsvarende problemet på GitHub, og legge til unødvendige vedlikeholdskostnader.

For å løse dette problemet bestemte vi oss for å introdusere funksjonen **GitHub Issues Integration**, som kobler HagiCode-økter med GitHub-lagre for å aktivere "ett-klikks-synkronisering."

## Teknisk valg: Frontend Direct Connection vs Backend Proxy

Da vi utformet integrasjonstilnærmingen, hadde vi to alternativer foran oss: den tradisjonelle "backend proxy-modellen" og den mer aggressive "frontend-direkte tilkoblingsmodellen."

### Sammenligning av løsninger

I den tradisjonelle **backend proxy-modellen** må hver forespørsel fra frontend først gå gjennom backend vår, som deretter kaller GitHub API. Dette sentraliserer logikken, men det legger også en betydelig byrde på backend:

1.  **Bloated backend**: Vi må skrive en dedikert GitHub API-klientinnpakning og også håndtere den komplekse OAuth-tilstandsmaskinen.
2.  **Tokenrisiko**: Brukerens GitHub-token må lagres i backend-databasen. Selv med kryptering øker dette fortsatt sikkerhetsoverflaten.
3.  **Utviklingskostnad**: Vi trenger databasemigrering for å lagre tokens og en ekstra synkroniseringstjeneste for å vedlikeholde.

**Modellen med direkte tilkobling for frontend** er mye lettere. I denne tilnærmingen bruker vi backend kun for det mest sensitive trinnet "hemmelig utveksling" (OAuth-tilbakekallingen). Etter å ha fått tokenet lagrer vi det direkte i nettleserens `localStorage`. Senere operasjoner som å opprette problemer og oppdatere kommentarer sendes direkte fra frontend til GitHub over HTTP.

| Sammenligningsdimensjon | Backend Proxy-modell | Frontend Direct Connection Model |
| :--- | :--- | :--- |
| **Kompleksitet på baksiden** | Krever en full OAuth-tjeneste og GitHub API-klient | Trenger bare et OAuth-tilbakeringingsendepunkt |
| **Tokenbehandling** | Må være kryptert og lagret i databasen, med lekkasjerisiko | Lagret i nettleseren og kun synlig for brukeren |
| **Implementeringskostnad** | Krever databasemigrering og utvikling av flere tjenester | Primært frontend arbeid |
| **Brukeropplevelse** | Sentralisert logikk, men serverforsinkelsen kan være litt høyere | Ekstremt rask respons med direkte GitHub-interaksjon |

Fordi vi ønsket rask integrasjon og minimale endringer i backend, **valgte vi til slutt "frontend-direkte tilkoblingsmodellen"**. Det er som å gi nettleseren et «midlertidig pass». Når den har fått passet, kan nettleseren håndtere ting på GitHub av seg selv uten å spørre backend-administratoren om godkjenning hver gang.

## Kjernedesign: Dataflyt og sikkerhet

Etter å ha bestemt oss for arkitekturen, trengte vi å designe den spesifikke dataflyten. Kjernen i synkroniseringsprosessen er hvordan man kan skaffe tokenet sikkert og bruke det effektivt.

### Overordnet arkitekturdiagram

Hele systemet kan abstraheres i tre roller: nettleseren (frontend), HagiCode-backend og GitHub.

```text
+--------------+        +--------------+        +--------------+
| Frontend     |        |   Backend    |        |   GitHub     |
| React        |        |   ASP.NET    |        |   REST API   |
|              |        |              |        |              |
|  +--------+  |        |              |        |              |
|  | OAuth  |--+--------> /callback    |        |              |
|  | Flow   |  |        |              |        |              |
|  +--------+  |        |              |        |              |
|              |        |              |        |              |
|  +--------+  |        |  +--------+  |        |  +--------+  |
|  | GitHub |  +------------> Session |  +----------> Issues |  |
|  | API    |  |        |  |Metadata|  |        |  |        |  |
|  | Direct |  |        |  +--------+  |        |  +--------+  |
|  +--------+  |        |              |        |              |
+--------------+        +--------------+        +--------------+
```

**Nøkkelpunktet er**: bare ett lite trinn i OAuth (bytte koden mot et token) trenger å gå gjennom backend. Etter det håndteres de tunge løftene (skaper problemer) direkte mellom frontend og GitHub.

### Synkroniseringsdataflyt i detalj

Når brukeren klikker på "Synkroniser med GitHub"-knappen i HagiCode-grensesnittet, skjer en rekke komplekse handlinger:

```text
User clicks "Sync to GitHub"
         │
         ▼
1. Frontend checks localStorage for the GitHub token
         │
         ▼
2. Format issue content (convert the Proposal into Markdown)
         │
         ▼
3. Frontend directly calls the GitHub API to create/update the issue
         │
         ▼
4. Call the HagiCode backend API to update Session.metadata (store issue URL and other info)
         │
         ▼
5. Backend broadcasts the SessionUpdated event via SignalR
         │
         ▼
6. Frontend receives the event and updates the UI to show the "Synced" state
```

### Sikkerhetsdesign

Sikkerhet er alltid toppprioritet ved integrering av tredjepartstjenester. Vi gjorde følgende betraktninger:

1.  **Forsvar mot CSRF-angrep**: Generer en tilfeldig `state` parameter under OAuth-viderekoblingen og lagre den i `sessionStorage`. Valider tilstanden i tilbakeringingen strengt for å forhindre falske forespørsler.
2.  **Isolert tokenlagring**: Tokenet lagres kun i nettleserens `localStorage`. Ved å bruke Same-Origin Policy, kan bare HagiCode-skript lese den, og unngår risikoen for at en databaselekkasje på serversiden påvirker brukerne.
3.  **Feilgrenser**: Vi utviklet dedikert håndtering for vanlige GitHub API-feil (som 401 utløpt token, 422 valideringsfeil og 429 ratebegrensning), slik at brukerne får vennlige tilbakemeldinger.

### 2. Frontend OAuth-flyt

Dette er inngangspunktet for forbindelsen. Vi bruker standard autorisasjonskodeflyt.

```typescript
// src/HagiCode.Client/src/services/githubOAuth.ts

// Generate the authorization URL and redirect
export async function generateAuthUrl(): Promise<string> {
  const state = generateRandomString(); // Generate a random string for CSRF protection
  sessionStorage.setItem('hagicode_github_state', state);
  
  const params = new URLSearchParams({
    client_id: clientId,
    redirect_uri: window.location.origin + '/settings?tab=github&oauth=callback',
    scope: ['repo', 'public_repo'].join(' '),
    state: state,
  });
  
  return `https://github.com/login/oauth/authorize?${params.toString()}`;
}

// Handle the code-to-token exchange on the callback page
export async function exchangeCodeForToken(code: string, state: string): Promise<GitHubToken> {
  // 1. Validate state to prevent CSRF
  const savedState = sessionStorage.getItem('hagicode_github_state');
  if (state !== savedState) throw new Error('Invalid state parameter');

  // 2. Call the backend API to exchange the token
  // Note: this must go through the backend because ClientSecret cannot be exposed to the frontend
  const response = await fetch('/api/GitHubOAuth/callback', {
    method: 'POST',
    headers: { 'Content-Type': 'application/json' },
    body: JSON.stringify({ code, state, redirectUri: window.location.origin + '/settings?tab=github&oauth=callback' }),
  });

  if (!response.ok) throw new Error('Failed to exchange token');
  
  const token = await response.json();
  
  // 3. Save into LocalStorage
  saveToken(token);
  return token;
}
```

### 3. GitHub API Client Wrapper

Når vi har tokenet, trenger vi et solid verktøy for å kalle GitHub API.

```typescript
// src/HagiCode.Client/src/services/githubApiClient.ts

const GITHUB_API_BASE = 'https://api.github.com';

// Core request wrapper
async function githubApi<T>(endpoint: string, options: RequestInit = {}): Promise<T> {
  const token = localStorage.getItem('gh_token');
  if (!token) throw new Error('Not connected to GitHub');
  
  const response = await fetch(`${GITHUB_API_BASE}${endpoint}`, {
    ...options,
    headers: {
      ...options.headers,
      Authorization: `Bearer ${token}`,
      Accept: 'application/vnd.github.v3+json', // Specify the API version
    },
  });
  
  // Error handling logic
  if (!response.ok) {
    if (response.status === 401) throw new Error('GitHub token expired, please reconnect');
    if (response.status === 403) throw new Error('No permission to access this repository or rate limit exceeded');
    if (response.status === 422) throw new Error('Issue validation failed, the title may be duplicated');
    throw new Error(`GitHub API Error: ${response.statusText}`);
  }
  
  return response.json();
}

// Create issue
export async function createIssue(owner: string, repo: string, data: { title: string, body: string, labels: string[] }) {
  return githubApi(`/repos/${owner}/${repo}/issues`, {
    method: 'POST',
    body: JSON.stringify(data),
  });
}
```

### 4. Innholdsformatering og synkronisering

Det siste trinnet er å konvertere HagiCode-sesjonsdata til formatet til et GitHub-problem. Det er litt som oversettelsesarbeid.

```typescript
// Convert a Session object into a Markdown string
function formatIssueForSession(session: Session): string {
  let content = `# ${session.title}\n\n`;
  content += `**> HagiCode Session:** #${session.code}\n`;
  content += `**> Status:** ${session.status}\n\n`;
  content += `## Description\n\n${session.description || 'No description provided.'}\n\n`;
  
  // If this is a Proposal session, add extra fields
  if (session.type === 'proposal') {
    content += `## Chief Complaint\n\n${session.chiefComplaint || ''}\n\n`;
    // Add a deep link so users can jump back from GitHub to HagiCode
    content += `---\n\n**[View in HagiCode](hagicode://sessions/${session.id})**\n`;
  }
  
  return content;
}

// Main logic when clicking the sync button
const handleSync = async (session: Session) => {
  try {
    const repoInfo = parseRepositoryFromUrl(session.repoUrl); // Parse the repository URL
    if (!repoInfo) throw new Error('Invalid repository URL');

    toast.loading('Syncing to GitHub...');
    
    // 1. Format content
    const issueBody = formatIssueForSession(session);
    
    // 2. Call API
    const issue = await githubApiClient.createIssue(repoInfo.owner, repoInfo.repo, {
      title: `[HagiCode] ${session.title}`,
      body: issueBody,
      labels: ['hagicode', 'proposal', `status:${session.status}`],
    });
    
    // 3. Update Session Metadata (save the issue link)
    await SessionsService.patchApiSessionsSessionId(session.id, {
      metadata: {
        githubIssue: {
          owner: repoInfo.owner,
          repo: repoInfo.repo,
          issueNumber: issue.number,
          issueUrl: issue.html_url,
          syncedAt: new Date().toISOString(),
        }
      }
    });

    toast.success('Synced successfully!');
  } catch (err) {
    console.error(err);
    toast.error('Sync failed, please check your token or network');
  }
};
```

## Sammendrag og Outlook

Med denne "frontend direct connection"-tilnærmingen oppnådde vi sømløs GitHub Issues-integrasjon med minst mulig backend-kode.

### Viktige gevinster
1.  **Høy utviklingseffektivitet**: Backend-endringer er minimale, hovedsakelig ett ekstra databasefelt og et enkelt OAuth-tilbakeringingsendepunkt. Det meste av logikken er fullført på frontend.
2.  **Sterk sikkerhet**: Tokenet går ikke gjennom serverdatabasen, noe som reduserer lekkasjerisikoen.
3.  **God brukeropplevelse**: Forespørsler initieres direkte fra frontend, så responshastigheten er rask og det er ikke behov for backend-videresending.

### Ting å passe på
Det er noen fallgruver å huske på under reell distribusjon:
- **OAuth App-innstillinger**: Husk å angi riktig `Authorization callback URL` i GitHub OAuth App-innstillingene (vanligvis `http://localhost:3000/settings?tab=github&oauth=callback`).
- **Prisgrenser**: GitHub API begrenser uautentiserte forespørsler ganske strengt, men med et token er kvoten vanligvis tilstrekkelig (5000 forespørsler/time).
- **Parsing av nettadresser**: Brukere skriver inn alle typer depot-URL-er, så sørg for at det regulære uttrykket ditt kan matche `.git` suffikser, SSH-formater og lignende tilfeller.

### Fremtidige forbedringer
Den nåværende funksjonen er fortsatt enveissynkronisering (HagiCode -> GitHub). I fremtiden planlegger vi å implementere toveis synkronisering gjennom GitHub Webhooks. For eksempel, hvis et problem er lukket på GitHub, kan øktstatusen på HagiCode-siden også oppdateres automatisk. Det vil kreve at vi avslører et webhook-endepunkt på backend, noe som vil være et interessant neste steg.
