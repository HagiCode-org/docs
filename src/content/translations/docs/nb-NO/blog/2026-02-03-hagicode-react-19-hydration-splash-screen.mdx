## Designe 12 eksepsjonelle oppstartsopplevelser for HagiCode: Fra minimalisme til cyberpunk

> Det korte gapet mellom å laste ned en React 19-app og fullføre Hydration er en gylden mulighet for brukere til å føle merkevaren din. I denne artikkelen deler vi et komplett oppstartsstilsystem vi bygget for HagiCode-prosjektet ved hjelp av HTML/CSS/JS.

## Bakgrunn

Som en moderne applikasjon bygget med ASP.NET Core 10 og React 19 (Vite), bruker HagiCode en frontend-backend-separert distribusjonsarkitektur. Frontend build-utdata er pakket inn i backend `wwwroot/` katalog og vert av ASP.NET Core.

Imidlertid introduserer denne arkitekturen et klassisk UX-smertepunkt: når brukere besøker siden, laster nettleseren først HTML-en, laster deretter ned den store JS-pakken og lar til slutt React utføre Hydration. I løpet av denne «vakuumperioden» som varer fra noen hundre millisekunder til flere sekunder, ser brukerne enten en tom skjerm eller en livløs statisk side.

For å fylle dette gapet og injisere HagiCodes merkepersonlighet, trengte vi å designe et oppstartsstilsystem implementert utelukkende med innebygd kode inne i `index.html`.

## Om HagiCode

Tilnærmingen til sprutskjermdesign som deles i denne artikkelen kommer fra vår praktiske erfaring i [HagiCode](https://github.com/HagiCode-org/site) prosjekt. Som en AI-kodingsassistent bryr HagiCode seg ikke bare om kodegenereringseffektivitet, men også om utviklerens visuelle opplevelse. Dette oppstartssystemet er et av resultatene av vår streben etter ultimate frontend-ytelse.

## Kjerneutfordringer og arkitekturdesign

Før vi begynte å designe, måtte vi først avklare de tekniske begrensningene. Siden alt måtte implementeres inline i `index.html`, vi kunne ikke laste inn noen eksterne CSS- eller JS-filer annet enn Reacts egen pakke.

### Teknisk begrensningsanalyse

1.  **Nullavhengighetsprinsipp**: Alle stiler må leve i en `<style>` tag, og all logikk må leve i en `<script>` tag.
2.  **Defensiv CSS**: For å forhindre at globale stiler forurenser splash-skjermen etter at React-appen er montert, bestemte vi oss for å pakke alle oppstartsstiler med et høyprioritert ID-prefiks som f.eks. `#boot-screen`.
3.  **Ytelse først**: Animasjoner bør bruke CSS `transform` og `opacity` når det er mulig for å unngå reflow og sikre at hovedtråden forblir ublokkert.
4.  **Visuell konsistens**: Farger og fonter må være på linje med HagiCodes Tailwind-konfigurasjon.

### Arkitekturmønster: Shell & Injector

Vi tok i bruk et **variantmønster**. Kjernelogikken er innkapslet i et umiddelbart påkalt funksjonsuttrykk (IIFE), mens den spesifikke gjengivelseslogikken injiseres gjennom konfigurasjon. Dette lar oss bytte mellom forskjellige stiler gjennom enkel konfigurasjon i stedet for å omskrive DOM-manipulasjonslogikk gjentatte ganger.

Her er kjernearkitekturkoden:

```html
<!-- Inline in index.html -->
<div id="boot-root"></div>

<script>
(function() {
  const BootSequence = {
    config: {
      theme: 'terminal', // Configurable as 'minimal', 'skeleton', 'code-rain', etc.
      color: '#3b82f6'   // Brand color
    },
    
    // Core lifecycle
    init() {
      this.render();
      this.listenForMount();
    },

    // Render the currently selected style
    render() {
      const root = document.getElementById('boot-root');
      if (this.variants[this.config.theme]) {
        root.innerHTML = this.variants[this.config.theme].render();
      }
    },

    // Listen for successful React mount and exit gracefully
    listenForMount() {
      window.addEventListener('hagicode:ready', () => {
        const screen = document.getElementById('boot-root');
        // Fade out first, then remove the DOM to avoid flicker
        screen.style.opacity = '0';
        screen.style.transition = 'opacity 0.3s ease';
        setTimeout(() => screen.remove(), 300);
      });
    },

    // The implementation logic for all 12 styles lives here
    variants: {
      // ...see details below
    }
  };

  BootSequence.init();
})();
</script>
```

## En sjekkliste med 12 oppstartsstildesign

Vi grupperte disse 12 stilene i seks hovedkategorier for å tilfredsstille ulike scenarier og estetiske preferanser.

### A. Minimalisme

> "Mindre er mer." For scenarier som forfølger den ultimate lastehastigheten, tilbyr vi de lettest mulige alternativene.

#### 1. Minimalistisk prikk
En enkel prikk sitter i midten av skjermen, sammen med en pusteanimasjon.
*   **Implementering**: CSS `@keyframes` kontrollerer skala og opasitet.
*   **Best for**: Alle tilfeller der siden må forbli helt ren.

#### 2. Brand Reveal
Bruker SVG `stroke-dasharray` animasjon, simulerer den en håndtegnet avsløring av HagiCode-logolinjene, etterfulgt av en tekstfade-in.
*   **Teknikk**: SVG-baneanimasjon med et svært polert preg.

### B. Skjelettskjermmimikk

> "Kunsten å lure øyet." Ved å simulere en ekte UI-layout, føler brukerne at siden allerede er halvt lastet.

#### 3. Chatskjelett på sidepanelet
Dette kan være det mest praktiske alternativet. Vi bygde manuelt en layout i HTML som speiler React `Sidebar` og `ChatInput` komponentene nøyaktig, og deretter overlappet den med en grå skimmer-animasjon.
*   **Verdi**: Når React-hydreringen er fullført, blir skjelettet øyeblikkelig den virkelige komponenten, og brukerne kan knapt oppfatte bryteren.

#### 4. Kortstabelskjelett
Den simulerer stablet bevegelse av forslagskort mens du laster inn, ved å bruke 3D-transformasjoner for å få kortene til å flyte subtilt.

### C. Abstraksjon og kunst

> Vis frem HagiCodes geek-DNA.

#### 5. Geometrisk morf
En geometrisk form (en firkant) gjengis i midten av skjermen, og forvandles deretter jevnt over tid til en sirkel, en trekant og til slutt logoen.
*   **Teknologi**: Myke overganger med CSS `border-radius`.

#### 6. Code Rain
En hyllest til *The Matrix*. Ved å bruke JetBrains Mono-fonten faller svake strømmer av tegn i bakgrunnen.
*   **Merk**: For ytelse må karakterstrømmene holde seg innenfor et mindre område eller bruke en lavere oppdateringsfrekvens.

#### 7. Neonpuls
En glødende ring i cyberpunk-stil som bruker flere `box-shadow` lag for å skape en kraftig neonglød.

### D. Merkevarebygging og temaer

> Få systemet til å føles levende.

#### 8. Sesongtema
Dette er en dynamisk laster. Den sjekker gjeldende dato for høytider som månenyttår eller jul og laster inn den tilsvarende SVG-animasjonen.
*   **Eksempel**: I løpet av Lunar New Year svaier røde lanterner forsiktig nederst på skjermen.

#### 9. Gradientflyt
Bakgrunnen bruker en flytende gradient basert på HagiCode-merkefarger. Kombinert med animert `background-size` og `background-position`, skaper det en nordlys-lignende følelse av bevegelse.

### E. Teknisk følelse

> En honnør til utviklerne.

#### 10. Terminal Boot
Den simulerer konsollutgang. Kodelinjer ruller raskt forbi:
```text
> Initializing HagiCode Core...
> Loading models...
> Connecting to neural network...
```
Det føles umiddelbart kjent for alle utviklere.

#### 11. Fremdriftslinje
En tynn fremdriftslinje vises øverst på skjermen, med en prosentandel vist til høyre. Selv om vi ikke får tilgang til den virkelige nedlastingsfremgangen, kan vi bruke en tidtaker til å simulere en "troverdig" lasteprosess: rask de første 80 %, deretter gradvis langsommere de siste 20 %.

### F. Kreativitet

#### 12. Pikselmontering
Dette er en veldig interessant idé. Små firkanter er spredt over skjermen, for så å konvergere mot midten og gradvis settes sammen til HagiCode-logoikonet. Det symboliserer prosessen med å bygge kode.

## Beste praksis og fallgruver

I HagiCodes reelle utviklingsarbeid oppsummerte vi flere kritisk viktige implementeringsdetaljer.

### 1. Defensiv CSS er obligatorisk
Bli aldri lat og hopp over prefikset. En gang glemte vi å omfange splash-skjermstilene med en ID, og ​​global `div` stiler etter at React var montert, påvirket uventet splash-skjermen og brøt oppsettet.
**Lærdom**: Sett hver CSS-velger under `#boot-screen`, og bruk `!important` for å heve prioritet når det er nødvendig, men kun inne i splash-skjermens CSS.

### 2. Grasiøse overganger
Etter at React er montert, ikke gjør det direkte `remove()` splash-skjermen DOM.
**Riktig tilnærming**:
1.  Reager triggere `window.dispatchEvent(new Event('hagicode:ready'))`.
2.  Splash-skjermen lytter etter arrangementet og første sett `opacity: 0`.
3.  Vent 300 ms, som samsvarer med CSS-overgangen, og ring `.remove()` først etter at skjermen er helt usynlig.

### 3. Tema Variabel synkronisering
Splash-skjermens fargeverdier er hardkodet inn `index.html`. Hvis vi endrer Tailwinds primærfarge, må vi også oppdatere splash-skjermen.
**Optimalisering**: Skriv en enkel plugin i Vite build-skriptet for å lese `tailwind.config.js` og injiser fargevariabler i `index.html` malvariabler, og skaper én enkelt kilde til sannhet.

### 4. Forhåndsinnlasting av skrift
Splash-skjermer må ofte bruke en merkeskrift, men hvis skriften laster sakte, kan FOUT (Flash of Unstyled Text) vises.
**Løsning**: Legg til `<link rel="preload" href="/fonts/JetBrainsMono.woff2" as="font" type="font/woff2" crossorigin>` inne `<head>`. Dette er en rimelig måte å forbedre opplevelsen på med høy avkastning.

### 5. Ytelsesovervåking
Vi injiserte `performance.mark('boot-start')` nederst på `index.html`, og merket `boot-end` når React ble montert.
**Hvorfor er det viktig**: Ved å samle inn disse dataene gjennom Application Insights kan vi direkte måle hvor mye splash-skjermen forkorter opplevd ventetid. Dataene viser at en utmerket skjelettskjerm kan forbedre brukernes toleranse for et "tregt nettverk" med mer enn 50 %.
