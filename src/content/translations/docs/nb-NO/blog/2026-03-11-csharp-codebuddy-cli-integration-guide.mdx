## Praktisk veiledning for å integrere CodeBuddy CLI i en C#-backend

> Denne artikkelen går gjennom en komplett tilnærming til å integrere CodeBuddy CLI i et C#-backend-prosjekt, slik at du kan levere AI-kodingsassistentfunksjoner fra ende til annen.

:::notat[Tidskontekst]
Denne artikkelen diskuterer CodeBuddy-integrasjonsdesignet og leverandørstrukturen slik de eksisterte på publiseringstidspunktet. For gjeldende permanent støttet omfang, se [AI Agent CLI installasjon](/nb-NO/related-software-installation/ai-agent-cli/) og den [Installasjonsveiledning](/nb-NO/installation); `IFlowCli` er nå bare beholdt som en historisk kompatibilitetsverdi og er ikke en del av det gjeldende aktive støtteomfanget.
:::

## Bakgrunn

I moderne AI-kodingsassistentutvikling kan en enkelt AI-leverandør ofte ikke tilfredsstille komplekse og skiftende utviklingsscenarier. HagiCode, som en multifunksjonell AI-kodingsassistent, må støtte flere AI-leverandører for å levere en bedre brukeropplevelse. Brukere bør ha nok frihet til å velge. Tidlig i 2026 sto prosjektet overfor en nøkkelbeslutning: hvordan gjenopprette CodeBuddy ACP (Agent Communication Protocol) integrasjonsevner i C#-backend.

Prosjektet hadde tidligere implementert CodeBuddy-integrasjon, men den relaterte koden ble fjernet under en refaktor. Det er ikke mye å klage på der; under iterativ utvikling blir det alltid noe igjen. Målet med denne tekniske løsningen var å fullt ut gjenopprette denne evnen og forbedre arkitekturen slik at den ble mer robust og vedlikeholdbar.

Hvis du også vurderer å koble flere AI-kodingsassistenter til ditt eget prosjekt, kan tilnærmingen nedenfor gi deg noen ideer. Det gjenspeiler leksjoner vi oppsummerte etter å ha gått inn i mange fallgruver, og kanskje det kan hjelpe deg med å unngå noen omveier.

## Arkitektur design

### Lagdelt arkitekturoversikt

Integrasjonen mellom C# og CodeBuddy bruker en tydelig lagdelt arkitektur. Denne designen gjør ansvar eksplisitt og gjør langsiktig vedlikehold mye enklere:

```
┌─────────────────────────────────────────────┐
│           Provider Contract Layer           │
│   AIProviderType enum + extension methods   │
├─────────────────────────────────────────────┤
│           Provider Factory Layer            │
│   AIProviderFactory dependency injection factory │
├─────────────────────────────────────────────┤
│           Provider Implementation Layer     │
│   CodebuddyCliProvider concrete implementation │
├─────────────────────────────────────────────┤
│           ACP Infrastructure Layer          │
│  ACPSessionManager / StdioAcpTransport      │
│  AcpRpcClient / AcpAgentClient              │
└─────────────────────────────────────────────┘
```

Hva er fordelene med denne lagdelingen? Enkelt sagt, hvert lag holder seg unna de andres vei. Hvis vi senere ønsker å endre kommunikasjonsmekanismen, for eksempel fra `stdio` til WebSocket trenger vi bare å modifisere bunnlaget, og forretningslogikken over det forblir urørt. Ingen vil at en kommunikasjonsendring skal bølge gjennom hele kodebasen.

### Kjernekomponentsammenbrudd

**Leverandørkontraktslaget** er grunnlaget for hele arkitekturen. Vi definerer `AIProviderType` enum, hvor `CodebuddyCli = 3` brukes som enum-verdi, og implementerer toveis kartlegging mellom strenger og enums gjennom utvidelsesmetoder. Det gjør at strenger i konfigurasjonsfiler enkelt kan konverteres til enums, og enums kan konverteres tilbake til strenger for feilsøking av utdata.

**Leverandørfabrikklaget** er ansvarlig for å lage den tilsvarende leverandørforekomsten basert på konfigurasjon. Den bruker .NET avhengighetsinjeksjon sammen med `ActivatorUtilities.CreateInstance` for dynamisk skapelse. Fordelen med fabrikkmønsteret er at når du legger til en ny leverandør, trenger du bare å legge til opprettelseslogikken i stedet for å endre eksisterende kode.

**Leverandørimplementeringslaget** er der selve arbeidet skjer. `CodebuddyCliProvider` implementerer `IAIProvider` grensesnitt og gir to påkallingsmoduser: `ExecuteAsync` for ikke-streamende samtaler og `StreamAsync` for streaming av samtaler.

**ACP-infrastrukturlaget** gir kommunikasjonsgrunnlaget under. Dette laget håndterer alle protokolldetaljer, inkludert prosessadministrasjon, meldingsserialisering og svarparsing. Det er grunnlaget som holder alt over det stabilt.

## Kommunikasjonsmekanisme

### Stdio transportmodus

CodeBuddy bruker **Stdio (standard input/output)** for å kommunisere med eksterne prosesser. Oppstartskommandoen er enkel:

```bash
codebuddy --acp
```

Etter det blir JSON-RPC-meldinger utvekslet gjennom standard input og output. Denne tilnærmingen har flere fordeler:

1. **Rask oppstart**: lokal prosesskommunikasjon unngår nettverksforsinkelse
2. **Enkel konfigurasjon**: du trenger bare å spesifisere den kjørbare banen
3. **Miljøisolering**: hver økt kjører i en uavhengig prosess, slik at de ikke påvirker hverandre

Miljøvariabelinjeksjon støttes under kommunikasjon. Vanlige eksempler inkluderer:

- `CODEBUDDY_API_KEY`: API-nøkkelautentisering
- `CODEBUDDY_INTERNET_ENVIRONMENT`: nettverksmiljøkonfigurasjon

Som med kommunikasjon mellom mennesker, hjelper det å velge en praktisk kanal først.

### Meldingsprotokoll

ACP er basert på JSON-RPC 2.0. Meldingsformatet ser omtrent slik ut:

```json
// Request message
{
  "jsonrpc": "2.0",
  "id": 1,
  "method": "agent/prompt",
  "params": {
    "prompt": "Help me write a sorting algorithm",
    "sessionId": "session-123"
  }
}

// Response message
{
  "jsonrpc": "2.0",
  "id": 1,
  "result": {
    "content": "Here is the AI response..."
  }
}
```

I den virkelige implementeringen kapsler vi inn alle disse protokolldetaljene, slik at det øvre forretningslaget bare trenger å bry seg om spørsmålet og svaret.

### 2. Integrer Provider Factory

Legg til en CodeBuddy-opprettingsgren i fabrikkklassen:

```csharp
// AIProviderFactory.cs
private IAIProvider? CreateProvider(AIProviderType providerType, ProviderConfiguration config)
{
    return providerType switch
    {
        AIProviderType.CodebuddyCli =>
            ActivatorUtilities.CreateInstance<CodebuddyCliProvider>(
                _serviceProvider,
                Options.Create(config)),
        // ... Other providers
        _ => throw new NotSupportedException($"Provider {providerType} not supported")
    };
}
```

Dette bruker avhengighetsinjeksjon gjennom `ActivatorUtilities`, som automatisk håndterer konstruktørparameterinjeksjon og er veldig praktisk.

### 3. Fullfør leverandørimplementering

Nedenfor er kjerneimplementeringen av `CodebuddyCliProvider`, som dekker både streaming og ikke-streaming påkallingsmodus:

```csharp
public class CodebuddyCliProvider : IAIProvider
{
    private readonly ILogger<CodebuddyCliProvider> _logger;
    private readonly IACPSessionManager _sessionManager;
    private readonly ProviderConfiguration _config;

    public string Name => "CodebuddyCli";
    public bool SupportsStreaming => true;
    public ProviderCapabilities Capabilities { get; }

    public CodebuddyCliProvider(
        ILogger<CodebuddyCliProvider> logger,
        IACPSessionManager sessionManager,
        IOptions<ProviderConfiguration> config)
    {
        _logger = logger;
        _sessionManager = sessionManager;
        _config = config.Value;

        // Define the capabilities of the current Provider
        Capabilities = new ProviderCapabilities
        {
            SupportsStreaming = true,
            SupportsTools = true,
            SupportsSystemMessages = true,
            SupportsArtifacts = false,
            MaxTokens = 8192
        };
    }

    // Non-streaming call: return all results together after completion
    public async Task<AIResponse> ExecuteAsync(
        AIRequest request,
        CancellationToken cancellationToken = default)
    {
        // Create an independent session for the request
        var session = await _sessionManager.CreateSessionAsync(
            "CodebuddyCli",
            request.WorkingDirectory,
            cancellationToken,
            request.SessionId);

        try
        {
            var fullPrompt = BuildPrompt(request);
            await session.SendPromptAsync(fullPrompt, cancellationToken);

            var responseBuilder = new StringBuilder();
            var toolCalls = new List<AIToolCall>();

            // Collect all response chunks
            await foreach (var chunk in StreamFromSession(session, cancellationToken))
            {
                if (!string.IsNullOrEmpty(chunk.Content))
                {
                    responseBuilder.Append(chunk.Content);
                }
                // Handle tool calls...
            }

            return new AIResponse
            {
                Content = AIResultContentSanitizer.SanitizeResultContent(
                    responseBuilder.ToString()),
                ToolCalls = toolCalls,
                Provider = Name,
                Model = string.Empty
            };
        }
        finally
        {
            // Release session resources
            await session.DisposeAsync();
        }
    }

    // Streaming call: return response chunks in real time
    public async IAsyncEnumerable<AIStreamingChunk> StreamAsync(
        AIRequest request,
        [EnumeratorCancellation] CancellationToken cancellationToken = default)
    {
        var session = await _sessionManager.CreateSessionAsync(
            "CodebuddyCli",
            request.WorkingDirectory,
            cancellationToken);

        try
        {
            var fullPrompt = BuildPrompt(request);
            await session.SendPromptAsync(fullPrompt, cancellationToken);

            await foreach (var chunk in StreamFromSession(session, cancellationToken))
            {
                yield return chunk;
            }
        }
        finally
        {
            await session.DisposeAsync();
        }
    }

    private async IAsyncEnumerable<AIStreamingChunk> StreamFromSession(
        IACPSession session,
        [EnumeratorCancellation] CancellationToken cancellationToken)
    {
        // Iterate through all updates in the session
        await foreach (var notification in session.ReceiveUpdatesAsync(cancellationToken))
        {
            switch (notification.Update)
            {
                case AgentMessageChunkSessionUpdate agentMessage:
                    // Handle text content chunks
                    if (agentMessage.Content is AcpImp.TextContentBlock textContent)
                    {
                        yield return new AIStreamingChunk
                        {
                            Content = textContent.Text,
                            Type = StreamingChunkType.ContentDelta,
                            IsComplete = false
                        };
                    }
                    break;

                case ToolCallSessionUpdate toolCall:
                    // Handle tool calls
                    yield return new AIStreamingChunk
                    {
                        Content = string.Empty,
                        Type = StreamingChunkType.ToolCallDelta,
                        ToolCallDelta = new AIToolCallDelta
                        {
                            Id = toolCall.ToolCallId,
                            Name = toolCall.Kind.ToString(),
                            Arguments = toolCall.RawInput?.ToString()
                        }
                    };
                    break;

                case AcpImp.PromptCompletedSessionUpdate:
                    // Response complete
                    yield break;
            }
        }
    }

    // Build the full prompt
    private string BuildPrompt(AIRequest request, string? embeddedCommandPrompt = null)
    {
        var sb = new StringBuilder();

        // Embedded command prompt, if present
        if (!string.IsNullOrEmpty(embeddedCommandPrompt))
        {
            sb.AppendLine(embeddedCommandPrompt);
            sb.AppendLine();
        }

        // System message
        if (!string.IsNullOrEmpty(request.SystemMessage))
        {
            sb.AppendLine(request.SystemMessage);
            sb.AppendLine();
        }

        // User prompt
        sb.Append(request.Prompt);
        return sb.ToString();
    }
}
```

Det er flere nøkkelpunkter i denne koden:

1. **Session management**: hver forespørsel oppretter en uavhengig økt og frigjør ressurser etter at forespørselen er fullført. Dette er en leksjon lært gjennom prøving og feiling. Hvis gjenbruk av økter ikke håndteres godt, oppstår det lett forurensning fra staten.

2. **Strømmebehandling**: `IAsyncEnumerable` lar svaret returneres mens det fortsatt genereres, i stedet for å vente på at alt innhold er ferdig. Dette er spesielt viktig for scenarier med lang tekst og forbedrer brukeropplevelsen betydelig.

3. **Verktøyanrop**: CodeBuddy støtter verktøyanrop (Function Calling), håndtert gjennom `ToolCallSessionUpdate`. Denne egenskapen er avgjørende for komplekse koderedigeringsoppgaver.

4. **Innholdsfiltrering**: `AIResultContentSanitizer` brukes til å filtrere Think-blokkinnhold og holde utdataene rene.

// PCodeClaudeHelperModule.cs
public void ConfigureModule(IServiceCollection context)

## Konfigurasjonseksempel

### Konfigurasjonsfil

Legg til CodeBuddy-relatert konfigurasjon til `appsettings.json`:

```yaml
AI:
  # Default Provider to use
  DefaultProvider: "CodebuddyCli"

  # Provider configuration
  Providers:
    CodebuddyCli:
      Type: "CodebuddyCli"
      WorkingDirectory: "C:/projects/my-app"
      ExecutablePath: "C:/tools/codebuddy.cmd"

  # Platform-specific configuration
  PlatformConfigurations:
    CodebuddyCli:
      ExecutablePath: "C:/tools/codebuddy.cmd"
      Arguments: "--acp"
      StartupTimeoutMs: 5000
      EnvironmentVariables:
        CODEBUDDY_API_KEY: "${CODEBUDDY_API_KEY}"
        CODEBUDDY_INTERNET_ENVIRONMENT: "production"
```

## Praktiske erfaringer

### Fallgruvelogg

Vi møtte flere typiske fallgruver under implementeringen, og å dele dem her kan hjelpe andre å unngå de samme omveiene:

1. **Problem med øktlekkasje**: til å begynne med ble ikke økter utgitt på riktig måte, noe som tømte prosessressursene. Løsningen var å bruke `try-finally` for å sikre at ressurser frigjøres for hver forespørsel.

2. **Miljøvariabeloverføring**: Windows og Linux bruker forskjellig miljøvariabelsyntaks, så vi standardiserte senere på `Dictionary<string, string?>` å håndtere dette.

3. **Tidsavbruddskonfigurasjon**: CLI-oppstart tar tid, så vi angir en 5-sekunders oppstartstid for å unngå raske forespørselsfeil.

4. **Kodingsproblemer**: på Windows kan standardkodingen forårsake forvrengt kinesisk tekst, så UTF-8-koding er eksplisitt spesifisert når prosessen startes.

### Ytelsesoptimalisering

1. **Session pool**: for hyppige korte forespørsler, vurder å implementere en sesjonspool for å gjenbruke prosesser
2. **Tilkoblingsbuffer**: Fabrikkklassen støtter allerede caching Provider-forekomster
3. **Asynkron først**: bruk asynkron programmering hele veien for å unngå blokkering av tråder

Ytelse er alltid verdt å optimalisere. Jo lenger brukere venter, jo dårligere blir opplevelsen.
