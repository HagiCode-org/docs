## Legger til støtte for automatisk distribusjon av GitHub-sider til HagiCode

> Dette prosjektet fikk opprinnelig kodenavnet PCode og har nå offisielt fått nytt navn til HagiCode. Denne artikkelen dokumenterer hvordan vi introduserte automatisert statisk nettsteddistribusjon, slik at publisering av innhold blir like enkelt som å drikke vann.

## Bakgrunn / Introduksjon

Under utviklingen av HagiCode møtte vi et veldig praktisk problem: ettersom dokumentasjonen og forslagene våre fortsatte å vokse, ble det stadig mer presserende å administrere og presentere innholdet på en effektiv måte. Vi bestemte oss for å bruke GitHub-sider til å være vert for vårt statiske nettsted, men å bygge og distribuere det manuelt var bare for tungvint. Hver endring krevde en lokal konstruksjon, pakking og deretter et manuelt trykk til `gh-pages` gren. Det var ikke bare ineffektivt, men også lett å ta feil.

For å løse dette problemet, hovedsakelig fordi vi ønsket å spare oss selv, trengte vi en automatisert distribusjonsarbeidsflyt. Denne artikkelen registrerer i detalj hvordan vi la til GitHub Actions-basert støtte for automatisk distribusjon til HagiCode-prosjektet, slik at vi kan holde fokus på å skrive innhold og la automatisering håndtere resten.

## Målanalyse

Før vi går inn i implementeringen, må vi først avklare hva denne oppgaven faktisk skal oppnå. Tross alt, skjerping av øksen forsinker ikke huggingen av ved.

### Kjernekrav

1.  **Automatisk bygg**: Utløs automatisk byggearbeidsflyten når koden skyves til `main` gren.
2.  **Automatisk distribusjon**: Etter en vellykket bygging, distribuer de genererte statiske filene automatisk til GitHub-sider.
3.  **Miljøkonsistens**: Sørg for at CI-miljøet samsvarer med det lokale byggemiljøet for å unngå den klassiske forlegenheten "det fungerer lokalt, men mislykkes på nettet."

### Teknisk valg

Siden HagiCode er bygget på Docusaurus, en veldig populær React statisk nettstedgenerator, kan vi bruke GitHub Actions for å oppnå dette målet.

## Konfigurere GitHub Actions arbeidsflyt

GitHub Actions er CI/CD-tjenesten levert av GitHub. Ved å definere arbeidsflytfiler i YAML-format inne i depotet, kan vi tilpasse alle slags automatiseringsoppgaver.

### Opprette arbeidsflytfilen

Vi må opprette en ny konfigurasjonsfil i `.github/workflows` katalog ved prosjektroten, for eksempel `deploy.yml`. Hvis katalogen ikke eksisterer ennå, opprett den først.

Kjernelogikken til denne konfigurasjonsfilen er som følger:

1.  **Triggertilstand**: Lytt etter `push` hendelser på `main` gren.
2.  **Runtime-miljø**: Den nyeste Ubuntu-utgivelsen.
3.  **Byggetrinn**:
    -   Sjekk ut koden
    -   Installer Node.js
    -   Installasjonsavhengigheter (`npm install`)
    -   Bygg de statiske filene (`npm run build`)
4.  **Implementeringstrinn**: Bruk den offisielle `action-gh-pages` handling for å skyve byggeutgangen til `gh-pages` gren.

### Nøkkelkonfigurasjonskode

Nedenfor er konfigurasjonsmalen vi til slutt tok i bruk:

```yaml
name: Deploy to GitHub Pages

# Trigger condition: run when code is pushed to the main branch
on:
  push:
    branches:
      - main
    # Add path filters if needed, for example only build when docs change
    # paths:
    #   - 'docs/**'
    #   - 'package.json'

# Set permissions, which is important for GitHub Pages deployment
permissions:
  contents: read
  pages: write
  id-token: write

# Concurrency control: cancel older builds for the same branch
concurrency:
  group: "pages"
  cancel-in-progress: false

jobs:
  build:
    runs-on: ubuntu-latest
    steps:
      - name: Checkout
        uses: actions/checkout@v4
        # Note: fetch-depth: 0 is required, otherwise the build version may be inaccurate
        with:
          fetch-depth: 0

      - name: Setup Node
        uses: actions/setup-node@v4
        with:
          node-version: 20 # Keep this consistent with the local development environment
          cache: 'npm'     # Enabling cache can speed up the build process

      - name: Install dependencies
        run: npm ci
        # Use npm ci instead of npm install because it is faster and stricter, which makes it better for CI

      - name: Build website
        run: npm run build
        env:
          # Configure environment variables here if your site build requires them
          # NODE_ENV: production
          # PUBLIC_URL: /your-repo-name
          
      - name: Upload artifact
        uses: actions/upload-pages-artifact@v3
        with:
          path: ./build # Default Docusaurus output directory

  deploy:
    environment:
      name: github-pages
      url: ${{ steps.deployment.outputs.page_url }}
    runs-on: ubuntu-latest
    needs: build
    steps:
      - name: Deploy to GitHub Pages
        id: deployment
        uses: actions/deploy-pages@v4
```

## Vanlige fallgruver under implementering

I praksis møtte vi flere problemer. Jeg deler dem her i håp om at andre kan unngå dem, eller i det minste forberede løsninger på forhånd.

### 1. Problemer med tillatelse til GitHub Token

Da vi først satte opp ting, fortsatte distribusjonen å mislykkes med en 403 (forbudt). Etter mye graving fant vi ut at GitHub er standard `GITHUB_TOKEN` hadde ikke tillatelse til å skrive til Pages.

**Løsning**: I depotets `Settings` -> `Actions` -> `General` -> `Workflow permissions`, sørg for å velge **«Lese- og skrivetillatelser»**.

### 2. Feil byggekatalogbane

Docusaurus plasserer bygde statiske filer i `build` katalog som standard. Noen prosjekter kan imidlertid være konfigurert annerledes. For eksempel, Create React App er standard til `build`, mens Vite er standard til `dist`. Hvis Actions rapporterer at den ikke finner utdatafilene, sjekk utdatabanen inn `docusaurus.config.js`.

### 3. Underbaneproblemer

Hvis depotet ditt ikke er en brukerhjemmeside, betyr det at det ikke er det `username.github.io`, men i stedet en prosjektside som f.eks `username.github.io/project-name`, så må du konfigurere `baseUrl`.

i `docusaurus.config.js`:

```javascript
module.exports = {
  // ...
  url: 'https://hagicode.com', // Your Hagicode URL
  baseUrl: '/', // Deploy from the root path
  // ...
};
```
Denne detaljen er lett å overse. Hvis den er konfigurert feil, kan siden lastes inn som en tom skjerm fordi aktivabanene ikke kan løses.

## Verifisering av resultatet

Etter å ha konfigurert alt og presset koden, kan vi gå til **Handlinger**-fanen i GitHub-depotet og se showet.

Du vil se en gul sirkel mens arbeidsflyten kjører, og når den blir grønn, betyr det suksess. Hvis den blir rød, klikk inn og inspiser loggene. I de fleste tilfeller kan du spore problemet der nede, vanligvis en skrivefeil eller en feil banekonfigurasjon.

Når byggingen er vellykket, besøk `https://<your-username>.github.io/<repository-name>/` og du vil se det nylig distribuerte nettstedet.

## Sammendrag

Ved å introdusere GitHub Actions, automatiserte vi vellykket distribusjon for HagiCode-dokumentasjonssiden. Dette sparer ikke bare tiden tidligere brukt på manuelle operasjoner, men enda viktigere standardiserer utgivelsesarbeidsflyten. Nå, uansett hvilket teammedlem som oppdaterer dokumentasjonen, når endringen er slått sammen til `main`, vil det nyeste innholdet være live noen minutter senere.

**Kjernefordeler**:
- **Høyere effektivitet**: Vi gikk fra «manuell pakking og manuell opplasting» til «kode er lik utgivelse».
- **Færre feil**: Vi eliminerte muligheten for menneskelige feil under manuelle operasjoner.
- **Bedre utvikleropplevelse**: Utviklere kan fokusere mer på innholdskvalitet i stedet for å bli distrahert av kjedelige implementeringstrinn.

Selv om konfigurering av CI/CD er litt plagsomt i begynnelsen, spesielt med alle tillatelser og banedetaljer, er det en engangsinvestering med stor langsiktig avkastning. Jeg anbefaler på det sterkeste at hvert statisk nettstedsprosjekt tar i bruk en lignende automatisert arbeidsflyt.
