## Løse nettleser WebSocket-autentiseringsutfordringer: et praktisk proxy-mønster for Doubao talegjenkjenning

> Nettleseren WebSocket API støtter ikke tilpassede HTTP-hoder, noe som skaper en utfordring for talegjenkjenningstjenester som krever autentiseringsdata i overskrifter. Denne artikkelen deler hvordan vi løste det problemet i HagiCode-prosjektet med et backend proxy-mønster, og hvordan tilnærmingen utviklet seg fra lekeplasseksperimenter til produksjonsbruk.


## Bakgrunn

Da vi begynte å bygge talegjenkjenning for HagiCode-prosjektet, valgte vi selvsikkert ByteDance sin Doubao talegjenkjenningstjeneste. Den opprinnelige utformingen var enkel: la frontend koble direkte til Doubaos WebSocket-tjeneste. Hvor vanskelig kan det være? Bare åpne en tilkobling og send litt data, ikke sant?

Så kom overraskelsen: Doubaos API krever at autentiseringsinformasjon sendes gjennom HTTP-hoder, inkludert ting som `accessToken` og `secretKey`. Det ble umiddelbart vanskelig, fordi nettleseren WebSocket API rett og slett ikke støtter innstilling av egendefinerte overskrifter.

Så hva gjør du når nettleseren ikke lar deg sende dem?

På det tidspunktet vurderte vi to alternativer:

1. Legg inn legitimasjonen i URL-søkeparametere – enkelt og sløvt
2. Legg til et proxy-lag på backend - mer arbeid ved første øyekast

Det første alternativet viser legitimasjon direkte i grensesnittkode og lokal lagring. Er det virkelig trygt? Jeg var ikke komfortabel med det. På toppen av det krever noen API-er hodebasert verifisering, så denne tilnærmingen er ikke engang levedyktig.

Til slutt valgte vi det andre alternativet: implementer en WebSocket-proxy på backend. Tilfeldigvis ble dette mønsteret først validert i vårt lekeplassmiljø, og først etter at vi bekreftet stabiliteten, flyttet vi det til produksjon. Tross alt vil ingen at produksjonen skal dobles som et laboratorieeksperiment.

## Teknisk utfordringsanalyse

### Nettleser WebSocket-begrensninger

Standard WebSocket API ser fantastisk enkelt ut:

```javascript
const ws = new WebSocket('wss://example.com/ws');
```

Men den enkelheten er akkurat der problemet ligger - den sender bare parametere i URL-en, og den kan ikke sette overskrifter slik en HTTP-forespørsel kan:

```javascript
// This is not supported in the WebSocket API
const ws = new WebSocket('wss://example.com/ws', {
  headers: {
    'Authorization': 'Bearer token'
  }
});
```

Og det er kjernespørsmålet. For tjenester som Doubao talegjenkjenning som er avhengig av hodebasert autentisering, er denne begrensningen en hard blokkering.

Når du aksepterer den begrensningen, må arkitekturen endres.

### Beslutninger om arkitektonisk design

Ved utformingen av løsningen sammenlignet vi avveiningene nøye.

**Beslutning 1: Velge proxy-mønsteret**

Vi sammenlignet to tilnærminger:

| Alternativ | Fordeler | Ulemper | Beslutning |
|------|------|------|------|
| Native WebSocket | Lett, enkel, direkte videresending | Tilkoblingshåndtering må håndteres manuelt | Valgt |
| SignalR | Automatisk gjentilkobling, sterk skriving | Altfor komplekse, ekstra avhengigheter | Avvist |

Vi valgte til slutt native WebSocket. For å være ærlig var det det letteste alternativet og passet bedre for enkel toveis videresending av binær strøm. Å trekke inn SignalR ville ha føltes som overengineering for denne brukstilfellet, og det kan legge til ekstra ventetid.

**Beslutning 2: Tilkoblingsstyringsstrategi**

Vi tok i bruk en "én tilkobling, en økt"-modell - hver frontend WebSocket-tilkobling tilordnes sin egen uavhengige Doubao-backend-tilkobling.

Fordelene er enkle:
- Enkel å implementere og tilpasset det vanlige bruksmønsteret
- Lettere å feilsøke og feilsøke
- God ressursisolasjon, forhindrer interferens mellom økter

Enkelt sagt er den direkte løsningen noen ganger den beste. Kompleksitet gjør ikke automatisk et design bedre.

**Beslutning 3: Lagring av autentiseringsdata**

Legitimasjon lagres i konfigurasjonsfiler for backend (`appsettings.yml` eller miljøvariabler) og lastet gjennom avhengighetsinjeksjon:

- Enkel konfigurasjonsmodell som matcher eksisterende backend-konvensjoner
- Sensitive data når aldri frontend
- Støtter multimiljøoppsett for utvikling, testing og produksjon

Det nivået av separasjon er viktig. Ingen vil ha legitimasjon flytende rundt på steder de ikke burde være.

### Dataflytdesign

Den generelle dataflyten ser slik ut:

```
Frontend (browser)
  │
  │ ws://backend/api/voice/ws
  │ WebSocket (binary)
  ▼
Backend (proxy)
  │
  │ wss://openspeech.bytedance.com/
  │ (with authentication headers)
  ▼
Doubao API
```

Selve flyten er ikke komplisert:
1. Frontend kobles til backend-proxyen via WebSocket
2. Backend-proxyen mottar lyddata og kobler til Doubao API med autentiserte overskrifter
3. Doubao API returnerer gjenkjenningsresultater, og proxyen videresender dem til frontend
4. Hele prosessen forblir helt asynkron med toveis streaming

Når ansvaret er delt klart, blir designet ganske naturlig.

## Implementering av kjernekomponenter

### 1. WebSocket-endepunktkonfigurasjon

```csharp
app.Map("/ws", async context =>
{
    if (context.WebSockets.IsWebSocketRequest)
    {
        // Read configuration from query parameters
        var appId = context.Request.Query["appId"];
        var accessToken = context.Request.Query["accessToken"];

        // Validate required parameters
        if (string.IsNullOrEmpty(appId) || string.IsNullOrEmpty(accessToken))
        {
            context.Response.StatusCode = 400;
            return;
        }

        // Accept the WebSocket connection
        using var webSocket = await context.WebSockets.AcceptWebSocketAsync();

        // Message handling loop
        var buffer = new byte[4096];
        while (!webSocket.CloseStatus.HasValue)
        {
            var result = await webSocket.ReceiveAsync(buffer, CancellationToken.None);

            if (result.MessageType == WebSocketMessageType.Close)
            {
                await webSocket.CloseAsync(
                    result.CloseStatus.Value,
                    result.CloseStatusDescription,
                    CancellationToken.None);
                break;
            }

            // Process audio data
            await HandleAudioDataAsync(buffer, result.Count);
        }
    }
});
```

### 2. Sesjonsledelse

```csharp
public class DoubaoSessionManager : IDoubaoSessionManager
{
    private readonly ConcurrentDictionary<string, DoubaoSession> _sessions = new();

    public DoubaoSession CreateSession(string connectionId)
    {
        var session = new DoubaoSession(connectionId);
        _sessions[connectionId] = session;
        return session;
    }

    public async Task SendAudioAsync(string connectionId, byte[] audioData)
    {
        if (_sessions.TryGetValue(connectionId, out var session))
        {
            await session.SendAudioAsync(audioData);
        }
    }

    public void RemoveSession(string connectionId)
    {
        if (_sessions.TryRemove(connectionId, out var session))
        {
            session.Dispose();
        }
    }
}
```

Bruker `ConcurrentDictionary` for øktstyring betyr at trådsikkerhet i stor grad håndteres for oss. Hver innkommende tilkobling får sin egen økt, og opprydding skjer automatisk ved frakobling.

## Utforming av meldingsprotokoll

Frontend og backend bruker JSON-tekstmeldinger for kontroll, mens binære meldinger har lyddata.

Eksempel på kontrollmelding:

```json
{
    "type": "control",
    "messageId": "msg_123",
    "timestamp": "2026-03-03T10:00:00Z",
    "payload": {
        "command": "StartRecognition",
        "parameters": {
            "hotwordId": "hotword1",
            "boosting_table_id": "table123"
        }
    }

```

Eksempel på gjenkjenningsresultat:

```json
{
    "type": "result",
    "timestamp": "2026-03-03T10:00:03Z",
    "payload": {
        "text": "Hello world",
        "confidence": 0.95,
        "duration": 1500,
        "isFinal": true,

            {
```

Denne designen skiller kontrollsignaler fra lydnyttelaster, noe som gjør systemet lettere å resonnere rundt og implementere. Å dele ansvar rent er ofte den rette oppfordringen.

## Frontend-integrasjon i praksis

### WebSocket-tilkobling

```javascript
class DoubaoVoiceClient {
    constructor(config) {
        this.config = config;
        this.ws = null;
    }

    async connect() {
        const url = new URL(this.config.wsUrl);
        // Add query parameters
        Object.entries(this.config.params).forEach(([key, value]) => {
            url.searchParams.set(key, value);
        });

        this.ws = new WebSocket(url);

        return new Promise((resolve, reject) => {
            this.ws.onopen = () => {
                console.log('[DoubaoVoice] Connected');
                resolve();
            };

            this.ws.onmessage = (event) => {
                this._handleMessage(JSON.parse(event.data));
            };

            this.ws.onerror = reject;
        });
    }

    _handleMessage(message) {
        switch (message.type) {
            case 'status':
                this._handleStatus(message.payload);
                break;
            case 'result':
                this.onResult?.(message.payload);
                break;
            case 'error':
                console.error('[DoubaoVoice] Error:', message.payload);
                break;
        }
    }
}

// Usage example
const client = new DoubaoVoiceClient({
    wsUrl: 'ws://localhost:5000/ws',
    params: {
        appId: 'your-app-id',
        accessToken: 'your-access-token',
        sampleRate: 16000,
        bitsPerSample: 16,
        channels: 1
    }
});
```

### Lydopptak og overføring

Bruker `AudioWorkletNode` for lydbehandling gir bedre ytelse:

```javascript
// audio-worklet.js
class AudioProcessorWorklet extends AudioWorkletProcessor {
    process(inputs, outputs, parameters) {
        const input = inputs[0]?.[0];
        if (!input) return true;

        // Convert to 16-bit PCM
        const pcm = new Int16Array(input.length);
        for (let i = 0; i < input.length; i++) {
            pcm[i] = Math.max(-32768, Math.min(32767, input[i] * 32767));
        }

        this.port.postMessage({
            type: 'audioData',
            data: pcm.buffer
        }, [pcm.buffer]);

        return true;
    }
}

registerProcessor('audio-processor', AudioProcessorWorklet);

// Main thread code
async function startAudioRecording() {
    const stream = await navigator.mediaDevices.getUserMedia({
        audio: {
            echoCancellation: true,
            noiseSuppression: true,
            autoGainControl: true,
            sampleRate: 48000
        }
    });

    const audioContext = new AudioContext();
    const audioSource = audioContext.createMediaStreamSource(stream);

    await audioContext.audioWorklet.addModule('/audio-worklet.js');
    const audioWorkletNode = new AudioWorkletNode(audioContext, 'audio-processor');

    audioWorkletNode.port.onmessage = (event) => {
        if (event.data.type === 'audioData' && ws?.readyState === WebSocket.OPEN) {
            ws.send(event.data.data); // Send binary data directly
        }
    };

    audioSource.connect(audioWorkletNode);
}
```

`AudioWorklet` presterer langt bedre enn `ScriptProcessorNode` og unngår lydstammeproblemene som eldre prosesseringsbaner ofte introduserer.

## Notater og beste fremgangsmåter

### 1. Tilkoblingsovervåking

- Logg regelmessig sesjonstilstand for å spore hele tilkoblingens livssyklus
- Overvåk antall og varighet av lydsegmenter for å oppdage unormale tilkoblinger
- Registrer tilkoblingsstatus og gjentilkoblingsadferd for Doubao-tjenesten

Dette er grunnleggende operasjonelle praksiser, men de utgjør en reell forskjell i produksjonen.

### 2. Feilhåndtering

- Registrer og logg alle WebSocket-unntak
- Bruk `IAsyncDisposable` for å sikre at ressursene blir ryddet opp
- Implementer grasiøs tilkoblingsavslutning og tidsavbruddshåndtering

Kort sagt, favoriser robusthet.

### 3. Krav til lydformat

- Samplingsfrekvens: 16000 Hz (anbefalt) eller 8000 Hz
- Bitdybde: 16-bit
- Kanaler: mono
- Koding: PCM (rå)

Hvis formatet er feil, kan gjenkjenningen svikte eller kvaliteten kan forringes betydelig. Disse kravene har betydning.

### 4. Sikkerhetshensyn

- Behold sensitiv legitimasjon bare i backend-konfigurasjon
- Håndhev tilkoblingsgrenser for å forhindre ressursbruk
- Bruk HTTPS/WSS i produksjonen

Sikkerhet er aldri en liten bekymring.

### 5. Ytelsesoptimalisering

- Bruk asynkrone operasjoner for å unngå blokkering
- Juster bufferstørrelser etter behov (standard: 4096 byte)
- Vurder tilkoblingspooling og gjenbruksstrategier

Bruk disse optimaliseringene der de gir mening for arbeidsmengden din.

## Implementeringsanbefalinger

1. **Dokker-distribusjon**: Pakk proxy-tjenesten som en beholder for enklere skalering og administrasjon
2. **Belastningsbalansering**: Bruk Nginx eller Envoy som en omvendt proxy for WebSocket-trafikk
3. **Helsesjekker**: Implementer hjerteslagbaserte kontroller for å overvåke tjenestetilgjengeligheten
4. **Loggaggregering**: Send logger til et sentralisert loggsystem som ELK eller Loki

Implementeringen kan være enkel eller kompleks avhengig av teamet og miljøet, så juster deretter.

## Sammendrag

WebSocket proxy-mønsteret løser kjerneproblemet at nettleserens WebSocket API ikke støtter egendefinerte overskrifter. I HagiCode-prosjektet viste dette mønsteret seg både gjennomførbart og stabilt da det gikk fra lekeplassvalidering til produksjonsdistribusjon.

Viktige takeaways:
- En backend-proxy kan sende autentiseringsinformasjon sikkert
- Native WebSocket er lett og effektiv for enkle scenarier
- Modellen "én tilkobling, en økt" forenkler både implementering og feilsøking
- Frontend-backend-protokollen skal skille kontrollsignaler fra lyddata

Hvis du bygger en funksjon som også avhenger av WebSocket-autentisering, håper jeg dette mønsteret gir deg et nyttig utgangspunkt.

Hvis du har spørsmål, kan du gjerne diskutere dem med oss. Teknisk fremgang skjer raskere når folk sammenligner notater.
