## Full GLM-5.1-støtte og Gemini CLI-integrasjon: HagiCode's Path of Multi-Model Evolution

> Denne artikkelen introduserer to store nyere oppdateringer til HagiCode-plattformen: full støtte for Zhipu AI GLM-5.1-modellen og den vellykkede integrasjonen av Gemini CLI som den tiende Agent CLI. Sammen styrker disse oppdateringene ytterligere plattformens multi-modell evner og multi-CLI økosystem.

:::note[Tidskontekstvarsel]
Denne artikkelen registrerer multi-CLI-utvidelsen slik den var på utgivelsestidspunktet. For gjeldende alltid støttet omfang, se [AI Agent CLI installasjon](/nb-NO/related-software-installation/ai-agent-cli/) og den [Installasjonsveiledning](/nb-NO/installation); den `IFlowCli` nevnt her representerer kun en historisk konfigurasjon på det tidspunktet og er ikke en del av det nåværende aktive støtteomfanget.
:::

## Bakgrunn

Tiden går virkelig fort. Utviklingen av store språkmodeller har økt som bambus om våren. For ikke lenge siden heiet vi fortsatt på «en AI som kan skrive kode». Nå er vi allerede inne i en tid med multi-modell samarbeid og multi-verktøy integrering. Er det spennende? Kanskje. Det utviklere trenger har tross alt aldri bare vært selve verktøyet, men det enkle å tilpasse seg ulike scenarier og bytte fleksibelt når det trengs.

Som en AI-assistert kodeplattform har HagiCode nylig ønsket to viktige utviklinger velkommen: For det første den fulle integrasjonen av Zhipu AI sin GLM-5.1-modell; for det andre, det offisielle tillegget av Gemini CLI som den tiende støttede Agent CLI. Disse to oppdateringene høres kanskje ikke jordskjelvende ut, men de er utvilsomt gode nyheter for plattformens fortsatte modning.

GLM-5.1 er Zhipu AIs nyeste flaggskipmodell. Sammenlignet med GLM-5.0 tilbyr den sterkere resonnement, dypere kodeforståelse og jevnere verktøykalling. Enda viktigere, det er den første GLM-modellen som støtter bildeinndata. Hva betyr det? Det betyr at brukere kan la AI-en se direkte på et skjermbilde i stedet for å slite med å beskrive problemet med ord. Når du har brukt denne bekvemmeligheten, forstår du umiddelbart verdien.

Samtidig, gjennom `HagiCode.Libs.Providers` arkitektur, integrerte HagiCode Gemini CLI med suksess i plattformen. Dette er nå den tiende Agent CLI. For å være ærlig, å komme til dette punktet gir en beskjeden følelse av prestasjon.

Det er også verdt å nevne at HagiCodes bildeopplastingsfunksjon lar brukere kommunisere med AI direkte gjennom skjermbilder. Selv når du kjører GLM 4.7, fungerer plattformen fortsatt bra og har allerede hjulpet med å fullføre mange viktige byggeoppgaver. Når det gjelder GLM-5.1, går det naturligvis ett skritt videre.

## Om HagiCode

Tilnærmingen som deles i denne artikkelen kommer fra vår praktiske erfaring i [HagiCode](https://hagicode.com) prosjekt. HagiCode er en åpen kildekode AI-assistert kodeplattform designet for å gi utviklere en fleksibel og kraftig AI-programmeringsassistent gjennom en multi-modell, multi-CLI-arkitektur. Prosjektlager: [github.com/HagiCode-org/site](https://github.com/HagiCode-org/site)

## Multi-CLI arkitekturdesign

En av HagiCodes kjernestyrker er støtten for flere AI-programmerings-CLI-verktøy gjennom et enhetlig abstraksjonslag. Fordelen med denne designen er faktisk ganske enkel: nye verktøy kan komme inn, gamle verktøy kan bli, og kodebasen blir ikke til kaos. For å være rettferdig, det er slik alle vil at livet skal fungere.

### AIProviderType enum

Plattformen definerer støttede CLI-leverandørtyper gjennom `AIProviderType` enum:

```csharp
public enum AIProviderType
{
    ClaudeCodeCli = 0,    // Claude Code CLI
    CodexCli = 1,          // GitHub Copilot Codex
    GitHubCopilot = 2,     // GitHub Copilot
    CodebuddyCli = 3,     // Codebuddy CLI
    OpenCodeCli = 4,      // OpenCode CLI
    IFlowCli = 5,         // IFlow CLI
    HermesCli = 6,        // Hermes CLI
    QoderCli = 7,         // Qoder CLI
    KiroCli = 8,          // Kiro CLI
    KimiCli = 9,          // Kimi CLI
    GeminiCli = 10,       // Gemini CLI (new)
}
```

Som du kan se, slutter Gemini CLI seg til denne familien som det tiende medlemmet. Hver CLI har sine egne distinkte egenskaper og bruksscenarier, slik at brukere kan velge fleksibelt basert på deres behov. Tross alt fører mange veier til Roma; noen er rett og slett enklere enn andre.

### Leverandørarkitektur

`HagiCode.Libs.Providers` gir et enhetlig leverandørgrensesnitt som gjør hver CLI-integrasjon standardisert og konsis. Ta Gemini CLI som et eksempel:

```csharp
public class GeminiProvider : ICliProvider<GeminiOptions>
{
    private static readonly string[] DefaultExecutableCandidates = ["gemini", "gemini-cli"];
    private const string ManagedBootstrapArgument = "--acp";

    public string Name => "gemini";
    public bool IsAvailable => _executableResolver.ResolveFirstAvailablePath(DefaultExecutableCandidates) is not null;
}
```

Fordelene med dette designet er:
- Integrering av en ny CLI krever bare implementering av én leverandørklasse
- Samlet livssyklusadministrasjon og samling av økter
- Automatisert aliasoppløsning og kjørbar oppdagelse

Enkelt sagt, denne designen gjør kompliserte ting til enklere og gjør livet litt enklere.

### Leverandørregister

Leverandørregisteret håndterer automatisk aliaskartlegging og registrering:

```csharp
if (provider is GeminiProvider)
{
    registry.Register(provider.Name, provider, ["gemini-cli"]);
    continue;
}
```

Dette betyr at brukere kan påkalle Gemini CLI med begge `gemini` eller `gemini-cli`, og systemet vil gjenkjenne den automatisk. Det er som en venn med både et formelt navn og et kallenavn - uansett, folk vet hvem du mener.

## GLM-5.1 modellstøtte

GLM-5.1 er Zhipu AIs nyeste flaggskipmodell, og HagiCode har fullført full støtte for den.

### Katalog for sekundære yrker

HagiCode administrerer alle støttede modeller gjennom Secondary Professionals Catalog. Her er konfigurasjonen for GLM-serien:

| Modell ID | Navn | Støtter Bilde | Kompatible CLI-familier |
|----------|------|----------------|-------------------------|
| `glm-4.7` | GLM 4.7 | - | claude, codebuddy, hermes, qoder, kiro |
| `glm-5` | GLM 5 | - | claude, codebuddy, hermes, qoder, kiro |
| `glm-5-turbo` | GLM 5 Turbo | - | claude, codebuddy, hermes, qoder, kiro |
| `glm-5.0` | GLM 5.0 (eldre) | - | claude, codebuddy, hermes, qoder, kiro |
| `glm-5.1` | GLM 5.1 | **sant** | claude, codebuddy, hermes, qoder, kiro |

Nøkkelegenskapene til GLM-5.1 kan oppsummeres som følger:
- En frittstående versjonsidentifikator uten eldre bagasje
- Den første GLM-modellen som støtter bildeinndata
- Sterkere resonnement og kodeforståelse
- Bred multi-CLI-kompatibilitet

### GLM-5.1 vs GLM-5.0

På kodenivå vises hovedforskjellen mellom GLM-5.1 og GLM-5.0 her:

```csharp
// GLM-5.0 (Legacy) - contains special retention logic
private const string Glm50CodebuddySecondaryProfessionId = "secondary-glm-5-codebuddy";
private const string Glm50CodebuddyModelValue = "glm-5.0";

// GLM-5.1 - standalone new model identifier
private const string Glm51SecondaryProfessionId = "secondary-glm-5-1";
private const string Glm51ModelValue = "glm-5.1";
```

GLM-5.0 bærer "Legacy"-etiketten fordi det er en gammel versjonsidentifikator som er beholdt for bakoverkompatibilitet. GLM-5.1, derimot, er en helt ny frittstående versjon uten historisk byrde. Noen ting forblir i fortiden; andre reiser lettere og beveger seg raskere.

### Konfigurer GLM-5.1

Her er et konfigurasjonseksempel for bruk av GLM-5.1 i HagiCode:

```json
{
  "primaryProfessionId": "profession-claude-code",
  "secondaryProfessionId": "secondary-glm-5-1",
  "model": "glm-5.1",
  "reasoning": "high"
}
```

## Bildeopplastingsfunksjon

HagiCodes bildestøtte implementeres gjennom `SupportsImage` eiendom på `SecondaryProfession`:

```csharp
public class HeroSecondaryProfessionSettingDto
{
    public bool SupportsImage { get; set; }
}
```

I Secondary Professions Catalog ser GLM-5.1-konfigurasjonen slik ut:

```json
{
  "id": "secondary-glm-5-1",
  "supportsImage": true
}
```

Dette betyr at brukere kan laste opp skjermbilder direkte for AI-analyse, for eksempel:
- Skjermbilder av feilmeldinger
- Problemer i en UI-skjerm
- Datavisualiseringsdiagrammer
- Kodeutførelsesresultater

Det er ikke lenger nødvendig å beskrive alt manuelt - bare last opp skjermbildet. Bekvemmeligheten med denne funksjonen er åpenbar når du har brukt den. Noen ganger sier ett blikk mer enn en lang forklaring.

## Gemini CLI-integrasjon

Som den tiende Agent CLI er Gemini CLI integrert i HagiCode gjennom standard Provider-arkitektur.

### Konfigurasjonsalternativer

Gemini CLI støtter et rikt sett med konfigurasjonsalternativer:

```csharp
public class GeminiOptions
{
    public string? ExecutablePath { get; set; }
    public string? WorkingDirectory { get; set; }
    public string? SessionId { get; set; }
    public string? Model { get; set; }
    public string? AuthenticationMethod { get; set; }
    public string? AuthenticationToken { get; set; }
    public Dictionary<string, string?> AuthenticationInfo { get; set; }
    public Dictionary<string, string?> EnvironmentVariables { get; set; }
    public string[] ExtraArguments { get; set; }
    public TimeSpan? StartupTimeout { get; set; }
    public CliPoolSettings? PoolSettings { get; set; }
}
```

Disse alternativene dekker alt fra grunnleggende oppsett til avanserte funksjoner, og gir brukerne fleksibiliteten til å konfigurere CLI etter deres egne behov. Alles arbeidsflyt er forskjellig, så litt fleksibilitet er alltid velkommen.

### ACP kommunikasjonsprotokoll

Gemini CLI støtter ACP (Agent Communication Protocol), som er HagiCodes enhetlige CLI-kommunikasjonsstandard. Gjennom ACP kan forskjellige CLI-er samhandle med plattformen på en konsistent måte, noe som i stor grad forenkler integreringsarbeidet. Kort sagt, det standardiserer de kompliserte delene slik at alle kan jobbe lettere.

## HagiCodes egen byggepraksis

Når vi snakker om praksis i den virkelige verden, er det beste eksemplet HagiCode-plattformens egen byggearbeidsflyt. HagiCodes utviklingsprosess har allerede gjort full bruk av AI-evner.

### Fungerer bra selv med GLM 4.7

HagiCodes plattformdesign er godt optimalisert, så den kan fortsatt gi en god utviklingsopplevelse selv med GLM 4.7. Plattformen har allerede hjulpet med å fullføre flere viktige byggeprosjekter, inkludert:
- Integrasjon av flere CLI-leverandører
- Implementering av bildeopplastingsfunksjonen
- Dokumentasjonsgenerering og innholdspublisering

Det er faktisk en god ting. Ikke alle trenger det nyeste hele tiden. Det som passer deg best er ofte det som betyr mest.

### GLM-5.1 gir mer med mindre innsats

Etter oppgradering til GLM-5.1 blir disse egenskapene enda sterkere:
- Sterkere kodeforståelse, reduserer frem og tilbake kommunikasjon
- Mer nøyaktig avhengighetsanalyse, peker i riktig retning umiddelbart
- Mer effektiv feildiagnose, lokalisering av problemer raskere
- Støtte for bildeinndata, akselererende problembeskrivelser

Det er som å bytte fra sykkel til bil. Du kan fortsatt nå samme destinasjon, men hastigheten og komforten er ikke den samme.

## Beste praksis for multi-CLI-integrasjon

`HagiCode.Libs.Providers` gir en enhetlig mekanisme for registrering og bruk:

```csharp
services.AddHagiCodeLibs();

var gemini = serviceProvider.GetRequiredService<ICliProvider<GeminiOptions>>();
var codebuddy = serviceProvider.GetRequiredService<ICliProvider<CodebuddyOptions>>();
var hermes = serviceProvider.GetRequiredService<ICliProvider<HermesOptions>>();
```

Denne avhengighetsinjeksjonsdesignen holder bruk på tvers av forskjellige CLI-er svært kortfattet og gjør også enhetstesting og hån mer praktisk. Ren kode er en måte å være ansvarlig overfor deg selv.

### Notater

Det er et par ting å huske på ved faktisk bruk:

1. **API-nøkkelkonfigurasjon**: Sørg for `ANTHROPIC_AUTH_TOKEN` er riktig innstilt, eller modellen kan ikke kalles
2. **Modelltilgjengelighet**: GLM-5.1 må være aktivert av den tilsvarende modellleverandøren
3. **Bildefunksjon**: Kun modeller med `supportsImage: true` kan bruke bildeopplasting
4. **CLI-installasjon**: Kontroller før du bruker Gemini CLI `gemini` eller `gemini-cli` er i systemet PATH

Dette kan være små detaljer, men små detaljer som håndteres dårlig kan bli til store problemer, så de er verdt å ta hensyn til.

## Konklusjon

Med full støtte for GLM-5.1 og den vellykkede integrasjonen av Gemini CLI, styrker HagiCode sine evner ytterligere som en multi-modell, multi-CLI AI programmeringsplattform. Disse oppdateringene gir ikke bare brukerne flere valgmuligheter, men demonstrerer også HagiCodes fremtidsrettede arkitektur og skalerbarhet.

GLM-5.1s bildestøtte, kombinert med HagiCodes opplastingsfunksjon for skjermbilder, gjør det mulig å la AI «forstå fra bildet» og reduserer kostnadene ved å beskrive problemer betraktelig. Og med støtte for ti CLI-er kan brukere fleksibelt velge AI-programmeringsassistenten som passer best til deres preferanser og scenarier. Flere valgmuligheter er nesten alltid en god ting.

Viktigst av alt, HagiCodes egen byggepraksis beviser at plattformen allerede kan kjøre godt og fullføre komplekse oppgaver selv med GLM 4.7, mens oppgradering til GLM-5.1 kan forbedre utviklingseffektiviteten ytterligere. Livet er ofte også slik: du trenger ikke alltid det absolutt beste, bare det som passer deg. Selvfølgelig, hvis det som passer deg kan bli enda bedre, så desto bedre.

Hvis du er interessert i en multi-modell, multi-CLI AI programmeringsplattform, prøv HagiCode - åpen kildekode, gratis og i stadig utvikling. Å prøve det koster ingenting, og det kan vise seg å være akkurat det du trenger.
