## Dyp integrasjon og praktisk bruk av StreamJsonRpc i HagiCode

> Denne artikkelen forklarer i detalj hvordan HagiCode-prosjektet (tidligere PCode) vellykket integrerte Microsofts StreamJsonRpc-kommunikasjonsbibliotek for å erstatte den originale tilpassede JSON-RPC-implementeringen, samtidig som den løste de tekniske smertepunktene og arkitektoniske utfordringene som oppstår under integrasjonsprosessen.

## Bakgrunn

StreamJsonRpc er Microsofts offisielt vedlikeholdte JSON-RPC-kommunikasjonsbibliotek for .NET og TypeScript, kjent for sin sterke typesikkerhet, automatiske proxy-generering og modne unntakshåndteringsmekanismer. I HagiCode-prosjektet bestemte teamet seg for å integrere StreamJsonRpc for å kommunisere med eksterne AI-verktøy som iflow CLI og OpenCode CLI gjennom ACP (Agent Communication Protocol), samtidig som de eliminerte vedlikeholdskostnadene og potensielle feil introdusert av den tidligere tilpassede JSON-RPC-implementeringen. Integrasjonsprosessen møtte imidlertid utfordringer som var spesifikke for streaming av JSON-RPC, spesielt ved håndtering av proxy-målbinding og generisk parametergjenkjenning.

For å løse disse smertepunktene tok vi en dristig beslutning: gjenoppbygg hele byggesystemet fra grunnen av. Virkningen av den avgjørelsen kan være enda større enn du forventer, og jeg vil forklare det i detalj snart.

## Analyse

Det nåværende prosjektet er i et kritisk stadium av ACP-protokollintegrasjon og står overfor følgende tekniske smertepunkter og arkitektoniske utfordringer:

### 1. Begrensninger for den tilpassede implementeringen
Den originale JSON-RPC-implementeringen er plassert i `src/HagiCode.ClaudeHelper/AcpImp/` og inkluderer komponenter som `JsonRpcEndpoint` og `ClientSideConnection`. Å vedlikeholde denne tilpassede kodebasen er kostbart, og den mangler de avanserte egenskapene til et modent bibliotek, for eksempel fremdriftsrapportering og kanselleringsstøtte.

### 2. StreamJsonRpc-integreringshindringer
Når du prøver å migrere den eksisterende `CallbackProxyTarget` mønster til StreamJsonRpc, fant vi ut at `_rpc.AddLocalRpcTarget(target)` metoden kunne ikke gjenkjenne mål opprettet gjennom proxy-mønsteret. Nærmere bestemt kunne ikke StreamJsonRpc automatisk dele egenskaper av den generiske typen `T` inn i RPC-metodeparametere, noe som får serversiden til å mislykkes ved behandling av metodeanrop initiert av klienten.

### 3. Forvirret arkitektonisk lagdeling
Det eksisterende `ClientSideConnection` blander transportlaget (WebSocket/Stdio), protokolllaget (JSON-RPC) og forretningslaget (ACP Agent-grensesnitt), noe som fører til uklare ansvarsforhold. Den lider også av manglende metodebindinger `AcpAgentCallbackRpcAdapter`.

### 4. Manglende logger
WebSocket-transportlaget mangler rå JSON-innholdslogging, noe som gjør det vanskelig å avgjøre under RPC-feilsøking om et problem stammer fra serialisering eller fra nettverket.

## Løsning

For å løse problemene ovenfor, tok vi i bruk følgende systematiske løsning, optimalisert fra tre dimensjoner: arkitektonisk refactoring, bibliotekintegrasjon og forbedret feilsøking.

### 1. Migrer fullstendig til StreamJsonRpc

#### Fjern eldre kode
Slett `JsonRpcEndpoint.cs`, `AgentSideConnection.cs`, og relaterte tilpassede serialiseringsomformere som f.eks `JsonRpcMessageJsonConverter`.

#### Integrer det offisielle biblioteket
Introduser `StreamJsonRpc` NuGet-pakken og bruk dens `JsonRpc` klasse for å håndtere kjernekommunikasjonslogikken.

#### Abstrahere transportlaget
Definer `IAcpTransport` grensesnitt for å håndtere begge deler `WebSocket` og `Stdio` transportmoduser på en enhetlig måte, og sikrer at protokolllaget er koblet fra transportlaget.

```csharp
// Definition of the IAcpTransport interface
public interface IAcpTransport
{
    Task SendAsync(string message, CancellationToken cancellationToken = default);
    Task<string> ReceiveAsync(CancellationToken cancellationToken = default);
    Task CloseAsync(CancellationToken cancellationToken = default);
}

// WebSocket transport implementation
public class WebSocketTransport : IAcpTransport
{
    private readonly WebSocket _webSocket;

    public WebSocketTransport(WebSocket webSocket)
    {
        _webSocket = webSocket;
    }

    // Implement send and receive methods
    // ...
}

// Stdio transport implementation
public class StdioTransport : IAcpTransport
{
    private readonly StreamReader _reader;
    private readonly StreamWriter _writer;

    public StdioTransport(StreamReader reader, StreamWriter writer)
    {
        _reader = reader;
        _writer = writer;
    }

    // Implement send and receive methods
    // ...
}
```

### 2. Løs problemer med gjenkjenning av proxy-mål

#### Analyser `CallbackProxyTarget`
Inspiser den eksisterende dynamiske proxy-genereringslogikken og identifiser årsaken til hvorfor StreamJsonRpc ikke kan gjenkjenne den. I de fleste tilfeller skjer dette fordi proxy-objektet ikke viser de faktiske metodesignaturene offentlig, eller det bruker parametertyper som ikke støttes av StreamJsonRpc.

#### Refaktorparameter passerer
Del generiske egenskaper inn i eksplisitte RPC-metodeparametere. I stedet for å stole på dynamiske egenskaper, definer konkrete Request/Response DTOer (dataoverføringsobjekter) slik at StreamJsonRpc kan gjenkjenne metodesignaturer korrekt gjennom refleksjon.

```csharp
// Original generic property approach
public class CallbackProxyTarget<T>
{
    public Func<T, Task> Callback { get; set; }
}

// Refactored concrete method approach
public class ReadTextFileRequest
{
    public string FilePath { get; set; }
}

public class ReadTextFileResponse
{
    public string Content { get; set; }
}

public interface IAcpAgentCallback
{
    Task<ReadTextFileResponse> ReadTextFileAsync(ReadTextFileRequest request);
    // Other methods...
}
```

#### Bruk `Attach` I stedet for `AddLocalRpcTarget`
I noen komplekse scenarier, manuell proxying av `JsonRpc` gjenstand og håndtering `RpcConnection` kan være mer fleksibel enn å legge til et lokalt mål direkte.

### 3. Implementer metodebinding og sterkere logging

#### Implementere `AcpAgentCallbackRpcAdapter`
Sørg for at denne komponenten eksplisitt implementerer StreamJsonRpc proxy-grensesnittet og kartlegger metoder definert av ACP-protokollen, som f.eks. `ReadTextFileAsync`, til StreamJsonRpc tilbakeringingsbehandlere.

#### Integrer logging
Avskjær og ta opp råteksten til JSON-RPC-forespørsler og svar i WebSocket- eller Stdio-meldingsbehandlingspipelinen. Bruk `ILogger` å sende ut den rå nyttelasten før parsing og etter serialisering, slik at formateringsproblemer kan diagnostiseres lettere.

```csharp
// Transport wrapper with enhanced logging
public class LoggingAcpTransport : IAcpTransport
{
    private readonly IAcpTransport _innerTransport;
    private readonly ILogger<LoggingAcpTransport> _logger;

    public LoggingAcpTransport(IAcpTransport innerTransport, ILogger<LoggingAcpTransport> logger)
    {
        _innerTransport = innerTransport;
        _logger = logger;
    }

    public async Task SendAsync(string message, CancellationToken cancellationToken = default)
    {
        _logger.LogTrace("Sending message: {Message}", message);
        await _innerTransport.SendAsync(message, cancellationToken);
    }

    public async Task<string> ReceiveAsync(CancellationToken cancellationToken = default)
    {
        var message = await _innerTransport.ReceiveAsync(cancellationToken);
        _logger.LogTrace("Received message: {Message}", message);
        return message;
    }

    public async Task CloseAsync(CancellationToken cancellationToken = default)
    {
        _logger.LogDebug("Closing connection");
        await _innerTransport.CloseAsync(cancellationToken);
    }
}
```

### 4. Refaktor arkitektonisk lagdeling

#### Transportlag (`AcpRpcClient`)
Pakk inn StreamJsonRpc-tilkoblingen og gjør den ansvarlig for `InvokeAsync` anrop og administrasjon av tilkoblingslivssyklus.

```csharp
public class AcpRpcClient : IDisposable
{
    private readonly JsonRpc _rpc;
    private readonly IAcpTransport _transport;

    public AcpRpcClient(IAcpTransport transport)
    {
        _transport = transport;
        _rpc = new JsonRpc(new StreamRpcTransport(transport));
        _rpc.StartListening();
    }

    public async Task<TResponse> InvokeAsync<TResponse>(string methodName, object parameters)
    {
        return await _rpc.InvokeAsync<TResponse>(methodName, parameters);
    }

    public void Dispose()
    {
        _rpc.Dispose();
        _transport.Dispose();
    }

    // StreamRpcTransport is the StreamJsonRpc adapter for IAcpTransport
    private class StreamRpcTransport : IDuplexPipe
    {
        // Implement the IDuplexPipe interface
        // ...
    }
}
```

#### Protokolllag (`IAcpAgentClient` / `IAcpAgentCallback`)
Definer klare klient-til-agent og agent-til-klient-grensesnitt. Fjern det sykliske fabrikkmønsteret `Func<IAcpAgent, IAcpClient>` og erstatte den med avhengighetsinjeksjon eller direkte tilbakeringingsregistrering.

## Øv

Basert på StreamJsonRpc beste praksis og prosjekterfaring, er følgende de viktigste anbefalingene for implementering:

### 1. Sterkt skrevet DTO-er er bedre enn dynamiske objekter
Kjernefordelen med StreamJsonRpc ligger i sterk skriving. Ikke bruk `dynamic` eller `JObject` å sende parametere. Definer i stedet eksplisitte C# POCO-klasser som parametere for hver RPC-metode. Dette løser ikke bare problemet med gjenkjenning av proxy-mål, men fanger også opp typefeil ved kompilering.

Eksempel: erstatt de generiske egenskapene i `CallbackProxyTarget` med betongklasser som f.eks `ReadTextFileRequest` og `WriteTextFileRequest`.

### 2. Deklarer eksplisitt metodenavn
Bruk `[JsonRpcMethod]` attributt for å eksplisitt spesifisere RPC-metodenavn i stedet for å stole på standard metodenavn-tilordning. Dette forhindrer invokasjonsfeil forårsaket av navnestilsforskjeller som PascalCase versus camelCase.

```csharp
public interface IAcpAgentCallback
{
    [JsonRpcMethod("readTextFile")]
    Task<ReadTextFileResponse> ReadTextFileAsync(ReadTextFileRequest request);
    
    [JsonRpcMethod("writeTextFile")]
    Task WriteTextFileAsync(WriteTextFileRequest request);
}
```

### 3. Dra nytte av tilbakeringing av tilkoblingstilstand
StreamJsonRpc gir `JsonRpc.ConnectionLost` begivenhet. Du bør absolutt lytte etter denne hendelsen for å håndtere uventede prosessavslutninger eller nettverksfrakoblinger, noe som er mer betimelig enn å bare stole på oppdagelse av Orleans Grain-feil.

```csharp
_rpc.ConnectionLost += (sender, e) =>
{
    _logger.LogError("RPC connection lost: {Reason}", e.ToString());
    // Handle reconnection logic or notify the user
};
```

### 4. Registrer logger etter lag
- **Sporingsnivå**: Registrer hele rå JSON-forespørsel/svar nyttelast.
- **Feilsøkingsnivå**: Registrer metodeanropsstabler og parametersammendrag.
- **Merk**: Sørg for at loggene ikke inkluderer sensitive autorisasjonstokener eller Base64-kodet innhold med store filer.

### 5. Håndter den spesielle karakteren til streamingtransport
StreamJsonRpc støtter naturlig `IAsyncEnumerable`. Når du implementerer direktesendingssvar for ACP, bruk `IAsyncEnumerable` direkte i stedet for tilpasset pagineringslogikk. Dette kan i stor grad forenkle mengden kode som trengs for strømmebehandling.

```csharp
public interface IAcpAgentCallback
{
    [JsonRpcMethod("streamText")]
    IAsyncEnumerable<string> StreamTextAsync(StreamTextRequest request);
}
```

### 6. Adaptermønster
Behold `ACPSession` og `ClientSideConnection` skille. `ACPSession` bør fokusere på orleans delstatsledelse og forretningslogikk, som for eksempel meldingskø, og bør bruke StreamJsonRpc-tilkoblingsobjektet gjennom komposisjon i stedet for arv.

## Konklusjon

Ved omfattende integrering av StreamJsonRpc, tok HagiCode-prosjektet opp de høye vedlikeholdskostnadene, funksjonsbegrensningene og den arkitektoniske forvirringen av den originale tilpassede implementeringen. De viktigste forbedringene inkluderer:

1. Erstatter dynamiske egenskaper med sterkt innskrevne DTO-er, forbedrer vedlikehold og pålitelighet
2. Implementere transportlagabstraksjon og protokolllagseparasjon, forbedre arkitektonisk klarhet
3. Styrke loggingsevner for å gjøre kommunikasjonsproblemer lettere å diagnostisere
4. Introduserer streamingstøtte for å forenkle streamingimplementering

Disse forbedringene gir HagiCode et mer stabilt og mer effektivt kommunikasjonsgrunnlag, som lar den samhandle bedre med eksterne AI-verktøy og legger et solid grunnlag for fremtidig utvidelse av funksjoner.
