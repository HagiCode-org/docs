## Design.md: En løsning for konsistent AI-drevet grensesnittdesign

> I en tid med AI-assistert frontend-utvikling, hvordan kan du holde AI-genererte brukergrensesnitt konsistente? Denne artikkelen deler vår praktiske erfaring med å bygge et designgallerinettsted basert på awesome-design-md, sammen med hvordan du lager en strukturert `design.md` for å veilede AI mot standardisert UI-design.

## Bakgrunn

Alle som har brukt AI til å skrive frontend-kode har sannsynligvis hatt en lignende opplevelse: be AI om å generere den samme siden flere ganger, og hvert resultat kommer ut i en annen stil. Noen ganger er hjørner avrundet, noen ganger er de skarpe. Noen ganger er avstanden 8px, og andre ganger blir den 16px. Selv den samme knappen kan se annerledes ut på tvers av forskjellige samtaler.

Dette er ikke et isolert problem. Etter hvert som AI-assistert utvikling blir mer vanlig, har **manglen på konsistens i AI-generert frontend-grensesnitt** blitt et utbredt problem. Ulike AI-assistenter, forskjellige spørsmål og til og med den samme assistenten på tvers av forskjellige samtaler kan produsere dramatisk forskjellige grensesnittdesign. Det skaper en enorm vedlikeholdskostnad under produktiterasjon.

Grunnårsaken er faktisk enkel: det finnes ikke noe **autoritativt designreferansedokument**. Tradisjonelle CSS-stilark kan fortelle utviklere "hvordan de skal implementere" noe, men de kan ikke fullt ut kommunisere "hvorfor det er designet på denne måten" eller "hvilket designmønster som skal brukes i hvilket scenario." For AI er en klar og strukturert beskrivelse enda viktigere for å forstå designkonvensjoner.

Samtidig tilbyr åpen kildekode-fellesskapet allerede noen utmerkede ressurser. Den [VoltAgent/awesome-design-md](https://github.com/VoltAgent/awesome-design-md) prosjekt samler designsystemdokumentasjon fra mange kjente selskaper. Hver katalog inneholder en `README.md`, a `DESIGN.md`, og forhåndsvise HTML. Alt dette er imidlertid spredt over oppstrømsdepotet, noe som gjør det vanskelig å bla gjennom og sammenligne raskt.

Så kan vi konsolidere disse ressursene i et designgalleri som er lett å bla gjennom, samtidig som vi destillerer et strukturert `design.md` for AI å bruke?

Svaret er ja. Deretter, la meg gå gjennom tilnærmingen vår.

## Analyse

Før du skriver kode, la oss bryte ned de tekniske utfordringene bak dette problemet.

### Source Content Management: Hvordan forener du spredte designressurser?

Oppstrøms awesome-design-md-depotet inneholder et stort antall designdokumenter, men vi trengte en måte å bringe dem inn i vårt eget prosjekt.

**Løsning: bruk git undermodul**

```
awesome-design-md-site
└── vendor/awesome-design-md          # Upstream resources (git submodule)
```

Dette gir oss flere fordeler:
- Versjonskontroll: vi kan feste en spesifikk oppstrømsversjon
- Offline builds: ingen grunn til å be om eksterne APIer under byggingen
- Innholdsgjennomgang: spesifikke endringer er synlige i PR-er

### Statisk nettstedsarkitektur: Hvordan gir du dynamisk søk mens du holder deg helt statisk?

Siden dette er et designgalleri, er søk et must. Men Astro er en statisk nettstedsgenerator, så hvordan implementerer du sanntidssøk?

**Løsning: Synkronisering av React Island + URL-søkeparameter**


### Designdokumentasjon: Hvordan hjelper du AI med å forstå og følge designstandarder?

Dette er kjernen i hele løsningen. Vi må skape en strukturert `design.md` som AI kan forstå og anvende.

**Løsning: lån strukturen til ClickHouse `DESIGN.md`**

[ClickHouse's `DESIGN.md`](https://github.com/ClickHouse/ClickHouse/blob/master/DESIGN.md) er en utmerket referanse. Det inkluderer:
- Visuelt tema og atmosfære
- Fargepalett og roller
- Typografi regler
- Komponentstylinger
- Oppsettprinsipper
- Dybde og høyde
- Gjør og ikke gjør
- Responsiv oppførsel
- Agentveiledning

Vår tilnærming er: **gjenbruk strukturen, omskriv innholdet**. Vi beholder seksjonsstrukturen til ClickHouse `DESIGN.md`, men erstatt innholdet med de faktiske designtokenene og komponentkonvensjonene som brukes i vårt eget prosjekt.

## Løsning

Basert på analysen ovenfor består løsningen vår av fire kjernemoduler.

### 1. Innholdssvelgingsrørledning

Dette er grunnlaget for hele systemet, ansvarlig for å trekke ut og normalisere innhold fra oppstrømsressurser.

```typescript
// src/lib/content/awesomeDesignCatalog.ts

export async function scanSourceEntries(): Promise<DesignEntry[]> {
  const designDir = 'vendor/awesome-design-md/design-md';
  const entries: DesignEntry[] = [];

  for (const dir of await fs.readdir(designDir)) {
    const entryPath = path.join(designDir, dir);
    if (await isValidDesignEntry(entryPath)) {
      const entry = await normalizeDesignEntry(entryPath);
      entries.push(entry);
    }
  }

  return entries;
}

async function isValidDesignEntry(dir: string): Promise<boolean> {
  const requiredFiles = ['README.md', 'DESIGN.md'];
  for (const file of requiredFiles) {
    if (!(await fileExists(path.join(dir, file)))) {
      return false;
    }
  }
  return true;
}
```

### 3. Søkefunksjonalitet

Søkefunksjonen er basert på klientsidefiltrering, med status bevart gjennom URL-søkeparametere:

```typescript
// src/components/gallery/SearchToolbar.tsx

function SearchToolbar({ entries }: { entries: DesignEntry[] }) {
  const [query, setQuery] = useState('');
  const [results, setResults] = useState(entries);

  useEffect(() => {
    const params = new URLSearchParams(window.location.search);
    const initialQuery = params.get('q') || '';
    setQuery(initialQuery);
    filterEntries(initialQuery);
  }, []);

  const filterEntries = (searchQuery: string) => {
    const filtered = entries.filter(entry =>
      entry.searchText.toLowerCase().includes(searchQuery.toLowerCase())
    );
    setResults(filtered);
  };

  const handleChange = (e: React.ChangeEvent<HTMLInputElement>) => {
    const value = e.target.value;
    setQuery(value);
    filterEntries(value);

    // Update the URL without triggering a page refresh
    const newUrl = value
      ? `${window.location.pathname}?q=${encodeURIComponent(value)}`
      : window.location.pathname;
    window.history.replaceState({}, '', newUrl);
  };

  return (
    <div className="search-toolbar">
      <input
        type="text"
        value={query}
        onChange={handleChange}
        placeholder="Search design entries..."
      />
      <span className="result-count">{results.length} results</span>
    </div>
  );
}
```

## Øv

Nå som vi har dekket løsningen, hvordan implementerer du den egentlig?

### Implementeringstrinn

**Trinn 1: Initialiser undermodulen**

```bash
# Add the upstream repository as a submodule
git submodule add https://github.com/VoltAgent/awesome-design-md.git vendor/awesome-design-md

# Initialize and update the submodule
git submodule update --init --recursive
```

**Trinn 2: Lag innholdspipeline**

Implementere `awesomeDesignCatalog.ts`, inkludert:
- Filskanning og valideringslogikk
- Markdown-gjengivelse ved hjelp av Astros innebygde gjengivelse
- Uttrekk av oppføringsdata

**Trinn 3: Bygg galleriets brukergrensesnitt**

Bruk Astro + React Islands for å lage:
- Hjemmesidegallerilayout (kortrutenett)
- Designkortkomponenter
- Søkeverktøylinje
- Detaljsideoppsett

**Trinn 4: Skriv designdokumentet**

Basert på strukturen til ClickHouse `DESIGN.md`, fyll ut de faktiske designtokenene fra ditt eget prosjekt. Oppdater `README.md` og legg til en lenke til `design.md`.

### Notater

**Sikkerhet**: Markdown-gjengivelse krever filtrering av usikker HTML. Astros innebygde gjengivelsesfiltre `script` tagger som standard, men du må fortsatt se etter XSS-risikoer.

**Ytelse**: Et stort antall iframe-forhåndsvisninger kan påvirke ytelsen ved første maling. Det anbefales å bruke `loading="lazy"` for å lade inn forhåndsvisningsinnhold.

**Vedlikehold**: `design.md` må være synkronisert med kodeimplementeringen. Det anbefales å legge til CI-sjekker for å sikre at CSS-variabler forblir konsistente mellom dokumentasjon og kode.

**Tilgjengelighet**: Sørg for at fargekontrasten oppfyller WCAG AA-standarden (minst 4,5:1).

### AI-bruksveiledning

Etter å ha opprettet `design.md`, hvordan får du AI til å faktisk bruke det? Her er noen praktiske tips:

**Tips 1: Henvis det eksplisitt i ledeteksten**

```
Please refer to the design.md file in the project root and use the design conventions defined there to implement the following components:
- Buttons: use the primary color with an 8px border radius
- Cards: use the elevation-2 shadow level
```

**Tips 2: Krev AI for å referere til spesifikke CSS-variabler**

```
Implement a navigation bar with the following requirements:
- Use --color-bg-primary for the background color
- Use --color-border-subtle for borders
- Use --text-color-primary for text
```

**Tips 3: Ta med `design.md` innhold i systemmeldingen**

Hvis AI-verktøyet ditt støtter tilpassede systemforespørsler, kan du legge til kjerneinnholdet i `design.md` direkte til den.

### Teststrategi

**Innholdsrørledningstesting**:
- Manglende fil-scenarier (mangler `README.md` eller `DESIGN.md`)
- Formatfeilscenarier (Markdown-parsing-feil)
- Scenarier for tomme kataloger

**Testing av søkefunksjoner**:
- Tom resultathåndtering
- Spesialtegn som kinesisk og emoji
- URL-synkroniseringsbekreftelse

**UI-komponenttesting**:
- Bytte av lys/mørkt tema
- Responsive layout
- Forhåndsvis lastetilstander

## Konklusjon

Inkonsekvensen i AI-genererte brukergrensesnitt som HagiCode møtte under utviklingen, var i sin kjerne forårsaket av mangelen på et strukturert designreferansedokument. Ved å bygge en designgalleriside og lage en standardisert `design.md`, vi løste dette problemet.

Kjerneverdien av denne løsningen ligger i:
- **Samlede ressurser**: konsolidering av spredt designsystemdokumentasjon
- **Strukturerte standarder**: uttrykker designkonvensjoner i et format AI kan forstå
- **Kontinuerlig vedlikehold**: holde innholdet oppdatert gjennom `git submodule`

Hvis du også bruker AI for frontend-utvikling, er denne tilnærmingen verdt å prøve. Å lage en strukturert `design.md` forbedrer ikke bare konsistensen til AI-generert kode, men hjelper også teamet ditt med å opprettholde enhetlige designstandarder internt.
