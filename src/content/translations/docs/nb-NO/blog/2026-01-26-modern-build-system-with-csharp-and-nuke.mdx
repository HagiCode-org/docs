## Goodbye Script Hell: Why We Chose C# for a Modern Build System

> Et dypdykk i hvordan HagiCode-prosjektet bruker Nuke til å bygge en typesikker, tverrplattforms- og svært utvidbar automatisert byggepipeline, som fullt ut adresserer vedlikeholdsproblemene til tradisjonelle byggeskripter.

## Bakgrunn

Gjennom den lange reisen med programvareutvikling har ordet «bygge» en tendens til å inspirere til både kjærlighet og frustrasjon. Kjærlighet, for med ett klikk blir koden til et produkt, og det er et av de mest tilfredsstillende øyeblikkene innen programmering. Frustrasjon, fordi å opprettholde en haug med rotete byggeskript kan være et mareritt.

I mange prosjekter er vi vant til å skrive skript i Python eller stole på XML-konfigurasjonsfiler. Tenk deg frykten for å bli styrt av `<property>` blokker. Men etter hvert som prosjektkompleksiteten vokser, spesielt i et prosjekt som HagiCode som spenner over frontend- og backend-arbeid, flere plattformer og flere språk, begynner tradisjonelle byggemetoder å vise sine grenser. Spredt skriptlogikk, ingen typekontroll, svak IDE-støtte ... hver av disse problemene blir en liten fallgruve som snubler opp teamet fra tid til annen.

For å løse disse smertepunktene introduserte HagiCode-prosjektet **Nuke**, et moderne byggesystem basert på C#. Det er ikke bare et verktøy. Det er en annen måte å tenke på å bygge arbeidsflyter. I dag skal vi se på hvorfor vi valgte det og hvordan det har forbedret utviklingsopplevelsen vår dramatisk.

## Kjerneanalyse: Hvorfor Nuke?

Du lurer kanskje på: "Det er allerede så mange byggesystemer, som Make, Gradle eller til og med vanlige Shell-skript. Hvorfor gå av veien for å bruke et C#-basert?"

Det er et rettferdig spørsmål. Nukes kjerneappell er at den bringer programmeringsspråkfunksjonene vi allerede kjenner godt inn i byggeskriptverdenen.

### 1. Modularizing Build Workflows: The Art of Targets

Nuke har en veldig klar designfilosofi: **alt er et mål**.

I tradisjonelle skript er det lett å ende opp med hundrevis av linjer med lineær utførelseslogikk viklet sammen. I Nuke bryter vi byggeprosessen til uavhengige `Target`s. Hvert mål gjør nøyaktig én ting, for eksempel:

- `Clean`: rengjør utdatakatalogen
- `Restore`: gjenopprett avhengighetspakker
- `Compile`: kompiler koden
- `Test`: kjøre enhetstester

Denne utformingen stemmer godt overens med enkeltansvarsprinsippet. I likhet med byggeklosser kan disse målene kombineres fritt. Enda viktigere, Nuke lar oss definere avhengigheter mellom mål. For eksempel hvis du vil løpe `Test`, vil systemet automatisk sjekke om `Compile` er allerede utført. Hvis du vil `Compile`, da `Restore` kommer naturlig først.

Denne avhengighetsgrafen gjør logikken mye klarere og forbedrer utførelseseffektiviteten betydelig, fordi Nuke automatisk kan analysere den optimale utførelsesveien.

### 2. Typesikkerhet: Farvel til marerittet med skrivefeil

Alle som har skrevet byggeskript i Python har sannsynligvis havnet i denne vanskelige situasjonen: skriptet kjører i fem minutter og mislykkes deretter fordi `Confi.guration` ble feilstavet, eller fordi en streng ble sendt der en numerisk parameter var forventet.

Den største fordelen med å skrive byggeskript i C# er **typesikkerhet**. Det betyr:

- **Kompileringstidskontroll**: IDE kan fortelle deg hva som er galt mens du skriver i stedet for å la deg vente til kjøretid.
- **Sikker refactoring**: hvis du ønsker å gi nytt navn til en variabel eller metode, kan IDE håndtere det i ett trinn uten å tvinge deg til å gjøre et risikabelt globalt søk og erstatte.
- **Intelligent fullføring**: kraftig IntelliSense kan fullføre kode for deg automatisk, slik at du ikke trenger å fortsette å bla i dokumentasjonen for å huske obskure APIer.

### 3. Støtte på tvers av plattformer: Én konsistent byggeopplevelse

Tidligere skrev vi `.bat` på Windows og `.sh` på Linux, og deretter kanskje lagt til et Python-skript for å bygge bro over gapet. Nå, uansett hvor .NET Core, eller .NET 5+ i dag, kan kjøre, kan Nuke også kjøre.

Det betyr at teammedlemmer kan bruke Windows, Linux eller macOS, og enten de foretrekker Visual Studio, VS Code eller Rider, kjører de alle nøyaktig samme logikk. Dette reduserer problemer forårsaket av miljøforskjeller, spesielt det klassiske problemet "det fungerer på maskinen min".

### 4. Parametere og konfigurasjonsadministrasjon

Nuke gir en veldig elegant parameter-parsing-mekanisme. Du trenger ikke å analysere manuelt `string[] args`. Du definerer ganske enkelt en egenskap og merker den med `[Parameter]`, og Nuke håndterer automatisk tilordningen fra kommandolinjeargumenter og konfigurasjonsfiler.

For eksempel kan vi definere byggekonfigurasjonen veldig enkelt:

```csharp
[Parameter("Configuration to build - Default is 'Debug'")]
readonly Configuration BuildConfiguration = IsLocalBuild ? Configuration.Debug : Configuration.Release;

Target Compile => _ => _
    .DependsOn(Restore)
    .Executes(() =>
    {
        // Use BuildConfiguration here; it is type-safe
        DotNetBuild(s => s
            .SetConfiguration(BuildConfiguration)
            .SetProjectFile(SolutionFile));
    });
```

Denne stilen er både intuitiv og motstandsdyktig mot feil.

## Praktisk veiledning: Hvordan sette det i praksis i et prosjekt

Teori alene bringer oss ingen vei. La oss se på hvordan vi faktisk brukte denne tilnærmingen i HagiCode-prosjektet.

### 1. Planlegg prosjektstrukturen

Vi ønsket ikke å bygge skript som rotet opp i prosjektroten, og vi ønsket heller ikke en katalogstruktur så dyp at den lignet visse Java-prosjekter. Så vi plasserte alle Nuke-relaterte byggefiler i en enkelt `nukeBuild/` katalog.

Fordelene er enkle:

- Prosjektroten forblir ren.
- Bygglogikk forblir sammenhengende og enklere å administrere.
- Nye teammedlemmer kan umiddelbart se: "Det er her den byggerelaterte logikken bor."

### 2. Design en klar målavhengighetskjede

Når vi utformet mål, fulgte vi ett prinsipp: **atomisitet pluss avhengighetsflyt**.

Hvert mål skal være lite nok til å gjøre nøyaktig én ting. For eksempel `Clean` skal bare fjerne filer og skal ikke håndtere emballasje på siden i det stille.

En anbefalt avhengighetsflyt ser omtrent slik ut:

`Clean` -> `Restore` -> `Compile` -> `Test` -> `Pack`

Dette er selvfølgelig ikke absolutt. For eksempel, hvis du bare vil kjøre tester og ikke vil pakke noe, lar Nuke deg utføre `nuke Test` direkte, og den vil automatisk håndtere det nødvendige `Restore` og `Compile` trinn først.

### 3. Robust feilhåndtering og logging

Hva er det mest frustrerende med å bygge skript? Uklare feilmeldinger. En mislykket konstruksjon som bare logger "Feil: 1" er nok til å gjøre noen gale.

I Nuke, fordi vi direkte kan bruke C#-unntakshåndtering, kan vi fange opp og rapportere feil med mye større presisjon.

```csharp
Target Publish => _ => _
    .DependsOn(Test)
    .Executes(() =>
    {
        try 
        {
            // Try publishing to NuGet
            DotNetNuGetPush(s => s
                .SetTargetPath(ArtifactPath)
                .SetSource("https://api.nuget.org/v3/index.json")
                .SetApiKey(ApiKey));
        }
        catch (Exception ex)
        {
            Log.Error($"Publishing failed. Team, please check whether the key is correct: {ex.Message}");
            throw; // Make sure the build process exits with a non-zero exit code
        }
    });
```

### 4. Bruk tester for å beskytte kvaliteten

Byggskript er også kode, noe som betyr at de også trenger tester. Nuke lar oss skrive tester for byggearbeidsflyter, slik at vi ikke bryter eksisterende utgivelsesprosesser ved et uhell når vi endrer byggelogikken. Dette er spesielt viktig i kontinuerlige integrasjonsrørledninger.

## Sammendrag

Ved å introdusere Nuke ble HagiCodes byggeprosess jevnere enn noen gang før. Dette var ikke bare en verktøyerstatning. Det var en oppgradering i ingeniørtankegangen.

**Hva fikk vi?**

- **Vedlikehold**: konfigurasjon som kode, klarere logikk og en raskere innføringsvei for nye bidragsytere.
- **Stabilitet**: Sterk skriving eliminerer mer enn 90 % av feilene på lavt nivå.
- **Konsistens**: en enhetlig opplevelse på tvers av plattformer som fjerner miljødrift.

Hvis det å skrive byggeskript pleide å føles som å "føle deg gjennom mørket", så føles det å bruke Nuke som å "gå om natten med lysene på." Hvis du er lei av å vedlikeholde skript som er vanskelige å feilsøke, bør du vurdere å flytte byggelogikken inn i C#-verdenen. Du vil kanskje oppdage at konstruksjoner også kan være elegante.
