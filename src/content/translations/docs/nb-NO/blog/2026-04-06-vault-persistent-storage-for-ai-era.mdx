## Hvordan reprodusere prosjekter i AI-æraen: Vault, et Cross-Project Persistent Storage System

> Hvordan kan vi hjelpe AI-assistenter til å forstå læringsressursene våre bedre i en tid med AI-assistert utvikling? HagiCode-prosjektet bygde Vault-systemet som et enhetlig kunnskapslagerabstraksjonslag som AI kan forstå, og forbedrer effektiviteten av læring gjennom prosjektreproduksjon.

## Bakgrunn

I AI-tiden er måten utviklere lærer nye teknologier og arkitekturer på i dyp endring. "Reprodusere prosjekter" - det vil si å studere og lære av koden, arkitekturen og designmønstrene til utmerkede åpen kildekode-prosjekter - har blitt en effektiv måte å lære på. Sammenlignet med tradisjonelle metoder som å lese bøker eller se videoer, hjelper direkte lesing og kjøring av åpen kildekode-prosjekter av høy kvalitet deg å forstå den virkelige ingeniørpraksisen mye raskere.

Likevel kommer denne læringsmetoden med ganske mange utfordringer.

Læremateriell er for spredt. Notatene dine kan ligge i Obsidian, kodelagre kan være spredt over forskjellige mapper, og AI-assistentens samtalehistorikk blir nok en isolert dataøy. Når du vil at AI skal hjelpe til med å analysere et prosjekt, må du manuelt kopiere kodebiter og organisere konteksten, noe som er ganske kjedelig.

Det som er enda mer plagsomt er den ødelagte konteksten. AI-assistenter kan ikke få direkte tilgang til de lokale læringsressursene dine, så du må gi bakgrunnsinformasjon på nytt i hver samtale. På toppen av det oppdateres reproduserte kodelagre raskt, manuell synkronisering er utsatt for feil, og kunnskap er vanskelig å dele på tvers av flere læringsprosjekter.

Ved roten kommer alle disse problemene fra «dataøyer». Hvis det fantes et enhetlig lagringsabstraksjonslag som gjorde det mulig for AI-assistenter å forstå og få tilgang til alle læringsressursene dine, ville problemet være løst.

For å hjelpe AI-assistenter bedre å forstå disse læringsressursene, utviklet vi Vault, et vedvarende lagringssystem på tvers av prosjekter.

## Vault system design filosofi

Kjerneideen til Vault-systemet er enkel: lag et enhetlig kunnskapslagerabstraksjonslag som AI kan forstå. Fra et implementeringsperspektiv har systemet flere sentrale egenskaper.

### Multi-type støtte

Systemet støtter fire hvelvtyper, som hver tilsvarer et annet bruksscenario:

```typescript
// folder: general-purpose folder type
export const DEFAULT_VAULT_TYPE = 'folder';

// coderef: a type specifically for reproduced code projects
export const CODEREF_VAULT_TYPE = 'coderef';

// obsidian: integrated with Obsidian note-taking software
export const OBSIDIAN_VAULT_TYPE = 'obsidian';

// system-managed: vault automatically managed by the system
export const SYSTEM_MANAGED_VAULT_TYPE = 'system-managed';
```

Blant dem er **coderef**-typen den mest brukte i HagiCode. Den er spesielt designet for reproduserte kodeprosjekter, og gir en standardisert katalogstruktur og AI-lesbare metadatabeskrivelser.

### Vedvarende lagringsmekanisme

Vault-registeret lagres vedvarende i JSON-format, og sikrer at konfigurasjonen forblir tilgjengelig etter at applikasjonen starter på nytt:

```csharp
public class VaultRegistryStore : IVaultRegistryStore
{
    private readonly string _registryFilePath;

    public VaultRegistryStore(IConfiguration configuration, ILogger<VaultRegistryStore> logger)
    {
        var dataDir = configuration["DataDir"] ?? "./data";
        var absoluteDataDir = Path.IsPathRooted(dataDir)
            ? dataDir
            : Path.GetFullPath(Path.Combine(Directory.GetCurrentDirectory(), dataDir));

        _registryFilePath = Path.Combine(absoluteDataDir, "personal-data", "vaults", "registry.json");
    }
}
```

Fordelen med denne designen er at den er enkel og pålitelig. JSON er lesbar for mennesker, noe som gjør feilsøking og manuell redigering enklere; filsystemlagring unngår kompleksiteten til en database og reduserer systemavhengigheter. Tross alt, noen ganger er det enkleste alternativet virkelig det beste.

### AI-kontekstintegrasjon

Det viktigste er at systemet automatisk kan injisere hvelvinformasjon i sammenheng med AI-forslag:

```typescript
export function buildTargetVaultsText(
  vaults: VaultForText[],
  template: VaultPromptTemplate = DEFAULT_VAULT_PROMPT_TEMPLATE,
): string {
  const readOnlyVaults = vaults.filter((vault) => vault.accessType === 'read');
  const editableVaults = vaults.filter((vault) => vault.accessType === 'write');

  if (readOnlyVaults.length === 0 && editableVaults.length === 0) {
    return '';
  }

  const sections = [
    buildVaultSection(readOnlyVaults, template.reference),
    buildVaultSection(editableVaults, template.editable),
  ].filter(Boolean);

  return `\n\n### ${template.heading}\n\n${sections.join('\n')}`;
}
```

Dette muliggjør en viktig funksjon: AI-assistenter kan automatisk forstå de tilgjengelige læringsressursene uten at brukerne manuelt gir kontekst. Du kan si det teller som en slags stilltiende forståelse.

## Den standardiserte strukturen til CodeRef Vault

For hvelvtypen coderef gir HagiCode en standardisert katalogstruktur:

```
my-coderef-vault/
├── index.yaml          # vault metadata description
├── AGENTS.md           # operating guide for AI assistants
├── docs/               # stores study notes and documents
└── repos/              # manages reproduced code repositories through Git submodules
```

Når du oppretter et hvelv, initialiserer systemet automatisk denne strukturen:

```csharp
private async Task EnsureCodeRefStructureAsync(
    string vaultName,
    string physicalPath,
    ICollection<VaultBootstrapDiagnosticDto> diagnostics,
    CancellationToken cancellationToken)
{
    Directory.CreateDirectory(physicalPath);

    var indexPath = Path.Combine(physicalPath, CodeRefIndexFileName);
    var docsPath = Path.Combine(physicalPath, CodeRefDocsDirectoryName);
    var reposPath = Path.Combine(physicalPath, CodeRefReposDirectoryName);

    // Create the standard directory structure
    if (!Directory.Exists(docsPath))
    {
        Directory.CreateDirectory(docsPath);
    }

    if (!Directory.Exists(reposPath))
    {
        Directory.CreateDirectory(reposPath);
    }

    // Create the AGENTS.md guide
    await EnsureCodeRefAgentsDocumentAsync(physicalPath, cancellationToken);

    // Create the index.yaml metadata
    await WriteCodeRefIndexDocumentAsync(indexPath, mergedDocument, cancellationToken);
}
```

Denne strukturen er også nøye utformet:

- **docs/** lagrer studienotatene dine, der du kan registrere din forståelse av koden, arkitekturanalyse, lærdom og så videre i Markdown
- **repos/** administrerer reproduserte arkiver gjennom Git-undermoduler i stedet for å kopiere kode direkte, noe som holder koden synkronisert og sparer plass
- **index.yaml** inneholder hvelvets metadata slik at AI-assistenter raskt kan forstå formålet med og innholdet til hvelvet
- **AGENTS.md** er en veiledning skrevet spesielt for AI-assistenter, som forklarer hvordan du håndterer innholdet i hvelvet

Organisert på denne måten kan AI kanskje forstå hva du har i tankene litt lettere.

## Automatisk initialisering for systemadministrerte hvelv

I tillegg til manuelt opprettede hvelv, støtter HagiCode også systemstyrte hvelv:

```csharp
public async Task<IReadOnlyList<VaultRegistryEntry>> EnsureAllSystemManagedVaultsAsync(
    CancellationToken cancellationToken = default)
{
    var definitions = GetAllResolvedDefinitions();
    var entries = new List<VaultRegistryEntry>(definitions.Count);

    foreach (var definition in definitions)
    {
        entries.Add(await EnsureResolvedSystemManagedVaultAsync(definition, cancellationToken));
    }

    return entries;
}
```

Systemet oppretter og administrerer automatisk følgende hvelv:

- **hagiprojectdata**: lagring av prosjektdata som brukes til å lagre prosjektkonfigurasjon og -status
- **personlige data**: personlig datalagring som brukes til å lagre brukerpreferanser
- **hbsprompt**: et forespørselsmalbibliotek som brukes til å administrere vanlige AI-meldinger

Disse hvelvene initialiseres automatisk når systemet starter, slik at brukerne ikke trenger å konfigurere dem manuelt. Noen ting er rett og slett bedre overlatt til systemet i stedet for at mennesker bekymrer seg for dem.

## Adgangskontrollmekanisme

En viktig del av designet er adgangskontroll. Systemet deler hvelv i to tilgangstyper:

```typescript
export interface VaultForText {
  id: string;
  name: string;
  type: string;
  physicalPath: string;
  accessType: 'read' | 'write';  // Key: distinguish read-only from editable
}
```

- **referanse (skrivebeskyttet)**: AI brukes kun til analyse og forståelse og kan ikke endre innhold. Egnet for refererte åpen kildekode-prosjekter, dokumenter og lignende materialer
- **redigerbar (redigerbar)**: AI kan endre innhold etter behov for oppgaven. Egnet for notater, utkast og lignende materialer

Dette skillet er viktig. Den forteller AI hvilket innhold som er "skrivebeskyttet referanse" og hvilket innhold som er "trygt å redigere", noe som reduserer risikoen for tilfeldige endringer. Tross alt vil ingen at deres harde arbeid skal forsvinne på grunn av en utilsiktet redigering.

## I praksis: opprette og bruke Vault

Nå som vi har dekket ideene, la oss se på hvordan det fungerer i praksis.

### Opprett et CodeRef Vault

Her er et fullstendig frontend-anropseksempel:

```typescript
const createCodeRefVault = async () => {
  const response = await VaultService.postApiVaults({
    requestBody: {
      name: "React Learning Vault",
      type: "coderef",
      physicalPath: "/Users/developer/vaults/react-learning",
      gitUrl: "https://github.com/facebook/react.git"
    }
  });

  // The system will automatically:
  // 1. Clone the React repository into vault/repos/react
  // 2. Create the docs/ directory for notes
  // 3. Generate the index.yaml metadata
  // 4. Create the AGENTS.md guide file

  return response;
};
```

Dette API-kallet fullfører en rekke handlinger: lage katalogstrukturen, initialisere Git-undermoduler, generere metadatafiler og mer. Du trenger bare å oppgi den grunnleggende informasjonen og la systemet håndtere resten. Det er ærlig talt en ganske bekymringsfri tilnærming.

### Bruk Vault i et AI-forslag

Etter å ha opprettet hvelvet, kan du referere til det i et AI-forslag:

const proposal = composeProposalChiefComplaint({

Systemet injiserer automatisk hvelvinformasjon i AI-konteksten, og lar AI vite hvilke læringsressurser som er tilgjengelige. Når AI kan forstå hva du har i tankene, er den typen stilltiende forståelse vanskelig å få til.

## Beste praksis og ting å se etter

Mens vi bruker Vault-systemet, har vi oppsummert noen lærdommer.

### Veisikkerhet

Systemet validerer stier strengt for å forhindre angrep på veier:

```csharp
private static string ResolveFilePath(string vaultRoot, string relativePath)
{
    var rootPath = EnsureTrailingSeparator(Path.GetFullPath(vaultRoot));
    var combinedPath = Path.GetFullPath(Path.Combine(rootPath, relativePath));
    if (!combinedPath.StartsWith(rootPath, StringComparison.OrdinalIgnoreCase))
    {
        throw new BusinessException(VaultRelativePathTraversalCode,
            "Vault file paths must stay inside the registered vault root.");
    }
    return combinedPath;
}
```

Dette er viktig. Hvis du tilpasser en hvelvbane, sørg for at den holder seg innenfor det tillatte området, ellers vil systemet avvise operasjonen. Du kan virkelig ikke overvurdere sikkerhet.

### Git-undermoduladministrasjon

CodeRef Vault anbefaler Git-undermoduler i stedet for å kopiere kode direkte:

```csharp
private static string BuildCodeRefAgentsContent()
{
    return """
    # CodeRef Vault Guide

    Repositories under `repos/` should be maintained through Git submodules
    rather than copied directly into the vault root.

    Keep this structure stable so assistants and tools can understand the vault quickly.
    """ + Environment.NewLine;
}
```

Dette gir flere fordeler: holde koden synkronisert med oppstrøms, spare diskplass og gjøre det enklere å administrere flere versjoner av koden. Tross alt, hvem vil laste ned det samme igjen og igjen?

### Begrensninger for forhåndsvisning av filer

For å forhindre ytelsesproblemer begrenser systemet filstørrelse og type:

```csharp
private const int FileEnumerationLimit = 500;
private const int PreviewByteLimit = 256 * 1024;  // 256KB
```

Hvis hvelvet ditt inneholder et stort antall filer eller svært store filer, kan forhåndsvisningsytelsen bli påvirket. I så fall kan du vurdere å behandle filer i grupper eller bruke spesialiserte søkeverktøy. Noen ganger når noe blir for stort, blir det vanskeligere å håndtere, ikke lettere.

### Diagnostisk informasjon

Når du oppretter et hvelv, returnerer systemet diagnostisk informasjon for å hjelpe med feilsøking:

```csharp
List<VaultBootstrapDiagnosticDto> bootstrapDiagnostics = [];

if (IsCodeRefVaultType(normalizedType))
{
    bootstrapDiagnostics = await EnsureCodeRefBootstrapAsync(
        normalizedName,
        normalizedPhysicalPath,
        normalizedGitUrl,
        cancellationToken);
}
```

Hvis opprettelsen mislykkes, kan du inspisere diagnoseinformasjonen for å forstå den spesifikke årsaken. Når noe går galt, er sjekk av diagnostikken ofte den mest direkte veien videre.

## Sammendrag

Gjennom et enhetlig lagringsabstraksjonslag løser Vault-systemet flere kjernepunkter ved reprodusering av prosjekter i AI-æraen:

- **Sentralisert kunnskapsledelse**: alle læringsressurser er samlet på ett sted i stedet for spredt overalt
- **Automatisk AI-kontekstinjeksjon**: AI-assistenter kan automatisk forstå de tilgjengelige læringsressursene uten manuell kontekstoppsett
- **Gjenbruk av kunnskap på tvers av prosjekter**: kunnskap kan deles og gjenbrukes på tvers av flere læringsprosjekter
- **Standardisert katalogstruktur**: et konsistent katalogoppsett senker læringskurven

Denne løsningen er allerede validert i HagiCode-prosjektet. Hvis du også bygger verktøy relatert til AI-assistert utvikling, eller står overfor lignende kunnskapshåndteringsproblemer, håper jeg disse erfaringene kan tjene som en nyttig referanse.
