HagiCode Desktop leveres med en Desktop-administrert Node-verktøykjede. Den tar ikke over den globale Node-installasjonen på datamaskinen din. Formålet er å gi Desktop-oppstart og kjøretidsrelaterte flyter et stabilt og verifiserbart kjøretidsgrunnlag.

Les denne FAQ hvis du ser `portable-fixed/toolchain` i diagnostikk, trenger å vite hvilken `node` Desktop bruker, eller lurer på hvorfor systemet Node-versjonen ikke direkte definerer Desktop kjøretidsatferd.

## Hvorfor Desktop pakker Node

Desktop må kjøre de samme støtteflytene på tvers av operativsystemer, installasjonskanaler og brukermiljøer. Hvis det bare var avhengig av brukerens eksisterende nodeinstallasjon, ville støtten måtte håndtere utdaterte versjoner, manglende npm, feil banebestilling, forurensede globale pakker og organisasjonsspesifikke registerpolicyer på en gang.

Desktop beskriver dette mer spesifikt som en isolert Node/npm-kjøretid for administrerte CLI-er i stedet for å gjenbruke det systemomfattende miljøet. Denne designen har disse hovedfordelene:

- **Konsekvente versjoner**: Desktop bruker en festet Node 22 kjøretidskontrakt i stedet for å gjette brukerens lokale hovedversjon.
- **Kontrollerte oppstartsflyter**: Desktop-administrerte kommandoer kan foretrekke den pakkede `node` og `npm` oppføringer.
- **Tøm konfigurasjonsgrenser**: CLI-relatert konfigurasjon holdes atskilt fra vertsomfattende global konfigurasjon, så det er mindre sannsynlig at oppdateringer, veksling og gjenoppretting blir forstyrret av systemmiljøforskjeller.
- **Reduserte konflikter**: Bunlede baner gjelder for kommandomiljøer opprettet av Desktop og erstatter ikke brukerens globale node i terminalen.
- **Ikke-påtrengende vertsatferd**: Denne isolasjonen unngår å skrive verktøykjedetilstand inn i brukerens eksisterende Node/npm-miljø, og avinstallering av Desktop forstyrrer ikke noden, npm eller konfigurasjonen som allerede finnes på maskinen.

## Hvordan Desktop velger nodebanen

Når du kjører Desktop normalt, foretrekker Desktop Node-verktøykjeden installert med appen i stedet for først å lese Node-installasjonen fra systemet ditt. Dette holder Node-versjonen i terminalen din atskilt fra Node-versjonen Desktop forventer mens den kjører.

Når den medfølgende verktøykjeden er aktivert, legger kommandomiljøer opprettet av Desktop sammenlagte verktøykjedebaner foran på `PATH`. Som et resultat, selv om systemet ditt har en annen Node-versjon installert, løses Desktop-administrerte flyter til Desktops medfølgende Node og npm først.

Dette påvirker kun kjøretidsmiljøer opprettet av Desktop. Hvis du løper `node`, `npm`, eller andre kommandoer direkte i systemterminalen, fortsetter de å bruke ditt vanlige systemmiljø og skallkonfigurasjon.

I diagnostikk- eller barneprosessmiljøer startet av Desktop, kan du se en variabel som dette:

```bash
HAGICODE_PORTABLE_TOOLCHAIN_ROOT=/path/to/portable-fixed/toolchain
```

Denne variabelen betyr at det skrivebordsadministrerte kjøretidsmiljøet gjenkjente den medfølgende verktøykjederoten. Å se denne banen betyr ikke at Desktop endret systemnode; det betyr bare at Desktop bruker Node-miljøet som ble levert med appen innenfor sitt eget kjøretidsområde.

## Hvor Desktop installerer administrerte npm-pakker

Desktop holder kjøretidsfiler og mutbare npm-pakkefiler på forskjellige steder. Den medfølgende verktøykjedebanen som f.eks `portable-fixed/toolchain/node` er den uforanderlige kilden for skrivebordsadministrerte `node` og `npm` kommandoer. Administrerte globale pakker er installert under Electron-brukerdatakatalogen i stedet:

```text
userData/node22/npmGlobal
```

Hvis Desktop senere flytter til Node 24, bruker den et eget pakketre som f.eks `userData/node24/npmGlobal`. Pakker installert for Node 22 forblir på disken, men behandles ikke som installert for Node 24 før de er installert på nytt i det nye Node-major-prefikset.

Skrivebordsavhengighetsstatus sjekker bare den aktive `userData/node<major>/npmGlobal` prefiks. Pakkefiler utenfor dette prefikset ignoreres av avhengighetsadministrasjon.
