# Introduksjon til OpenSpec

OpenSpec er et spesifikasjonsdrevet utviklingsverktøy som hjelper team med å administrere programvareutviklingsprosessen gjennom strukturerte forslag og spesifikasjoner. Denne artikkelen gir en omfattende introduksjon til OpenSpecs kjernebegreper, arbeidsflyter og bruksmetoder.

## Hva er OpenSpec?

OpenSpec er et kommandolinjeverktøy som brukes til å administrere forslag (Proposals), spesifikasjoner (Specifications) og endringer (Changes) i programvareutviklingsprosessen. Det standardiserer utviklingsprosessen og sørger for at hver funksjon går gjennom grundig design og gjennomgang fra idé til implementering.

### Kjernen i prinsippene

OpenSpec bygger på følgende kjerneprinsipper:

- **Spesifikasjon først**: Definer klare spesifikasjoner før du skriver kode
- **Strukturert samarbeid**: Fremme teamsamarbeid gjennom standardiserte dokumentformater
- **Sporbarhet**: Registrer hele prosessen fra forslag til implementering
- **Kvalitetssikring**: Sikre at koden samsvarer med designintensjonen ved å validere spesifikasjonene

### Hovedfunksjoner

OpenSpec tilbyr følgende hovedfunksjoner:

1. **Forslagsstyring**: Opprett, gjennomgå og spore funksjonsforslag
2. **Spesifikasjonsdefinisjon**: Skriv og vedlikehold tekniske spesifikasjonsdokumenter
3. **Endringssporing**: Registrer hele historikken til gjennomgåtte endringer
4. **Spesifikasjonsvalidering**: Sikre at kodeendringer samsvarer med forhåndsdefinerte spesifikasjoner
5. **Teamsamarbeid**: Støtte utviklings- og gjennomgangsprosesser med flere deltakere

## OpenSpec arbeidsflyt

Den typiske OpenSpec arbeidsflyten inkluderer følgende faser:

### 1. Opprett forslag (Proposal)

Når det er en ny funksjon eller et krav om betydelig endring, opprett først et forslag:

```bash
openspec proposal create
```

Forslaget inkluderer:
- **Problemstilling**: Problemet eller kravet som må løses
- **Løsning**: Oversikt over den foreslåtte løsningen
- **Omfang av påvirkning**: Påvirkning på det eksisterende systemet
- **Alternative løsninger**: Andre vurderte løsninger

### 2. Skriv spesifikasjon (Specification)

Etter at forslaget er godkjent, skriv en detaljert teknisk spesifikasjon:

```bash
openspec spec create
```

Spesifikasjonen inkluderer:
- **Teknisk design**: Detaljert plan for teknisk implementering
- **Grensesnittdefinisjon**: Design av grensesnitt som API, datastrukturer osv.
- **Implementasjonsdetaljer**: Nøkkellogikk og algoritmer for implementasjonen
- **Testplan**: Hvordan verifisere korrektheten av implementasjonen

### 3. Opprett endring (Change)

Når du starter implementeringen, opprett en endring for å spore utviklingsprosessen:

```bash
openspec change create
```

Endringen inkluderer:
- **Tilknyttet spesifikasjon**: Spesifikasjonen som implementeres av denne endringen
- **Implementeringsoppgaver**: Liste over konkrete utviklingsoppgaver
- **Fremdriftssporing**: Status for oppgavefullføring
- **Kodeendringer**: De faktiske kodeendringene

### 4. Validering og arkivering

Når utviklingen er fullført, valider om endringen samsvarer med spesifikasjonen, og arkiver deretter:

```bash
openspec change validate
openspec change archive
```

## OpenSpec katalogstruktur

OpenSpec oppretter en `openspec/` katalog i prosjektets rotmappe med følgende struktur:

```
openspec/
├── proposals/           # Forslagsmappe
│   ├── PROPOSAL-001.md
│   └── PROPOSAL-002.md
├── specs/              # Spesifikasjonsmappe
│   ├── SPEC-001.md
│   └── SPEC-002.md
├── changes/            # Endringsmappe
│   ├── feature-name/
│   │   ├── proposal.md
│   │   ├── design.md
│   │   ├── specs.md
│   │   └── tasks.md
│   └── another-feature/
└── archived/           # Arkiverte endringer
    └── feature-name/
```

### Katalogforklaring

- **proposals/**: Lagrer alle forslagsdokumenter, hvert forslag i en egen Markdown-fil
- **specs/**: Lagrer tekniske spesifikasjonsdokumenter som tilsvarer forslagene
- **changes/**: Lagrer pågående endringer, hver endring i et underkatalog
- **archived/**: Lagrer fullførte og arkiverte endringer

## Kjernekommandoer

### Forslagsrelaterte kommandoer

```bash
# Opprett nytt forslag
openspec proposal create

# Liste over alle forslag
openspec proposal list

# Vis forslagsdetaljer
openspec proposal view <proposal-id>

# Godkjenn forslag
openspec proposal approve <proposal-id>
```

### Spesifikasjonsrelaterte kommandoer

```bash
# Opprett ny spesifikasjon
openspec spec create

# Liste over alle spesifikasjoner
openspec spec list

# Vis spesifikasjonsdetaljer
openspec spec view <spec-id>

# Oppdater spesifikasjon
openspec spec update <spec-id>
```

### Endringsrelaterte kommandoer

```bash
# Opprett ny endring
openspec change create

# Liste over alle endringer
openspec change list

# Vis endringsdetaljer
openspec change view <change-name>

# Legg til oppgave
openspec change add-task <change-name>

# Valider endring
openspec change validate <change-name>

# Arkiver endring
openspec change archive <change-name>
```

### Generelle kommandoer

```bash
# Vis versjon
openspec --version

# Vis hjelp
openspec --help

# Vis hjelp for en spesifikk kommando
openspec <command> --help
```

## Bruke OpenSpec i HagiCode-prosjektet

### Prosjektintegrasjon

I HagiCode-prosjektet er OpenSpec integrert i utviklingsarbeidsflyten:

1. **Initieringskontroll**: Oppstartsguiden sjekker om OpenSpec er installert
2. **Forslagsdrevet utvikling**: Alle nye funksjoner må gå gjennom OpenSpec-forlgsprosessen
3. **Kodegjennomgang**: Endringer må samsvare med OpenSpec-spesifikasjonene for å kunne merges
4. **Dokumentasjonssynkronisering**: OpenSpec-dokumentasjonen holdes synkronisert med kodeimplementasjonen

### Versjonskrav

HagiCode-prosjektet krever bruk av versjonen **OpenSpec >=1.0.0 <2.0.0**.

Installkommando:

```bash
npm install -g @fission-ai/openspec@1
```

### Arbeidsflyteksempel

Her er et komplett eksempel på bruk av OpenSpec i HagiCode-prosjektet:

#### Trinn 1: Opprett forslag

```bash
cd /path/to/hagicode-project
openspec proposal create
```

Skriv inn i den interaktive prompten:
- Forslagstittel: Legg til brukerautentiseringsfunksjon
- Problemstilling: Systemet trenger brukerinnlogging og tilgangsstyring
- Løsning: Implementer JWT-basert autentiseringssystem

#### Trinn 2: Skriv spesifikasjon

Etter godkjenning av forslaget, opprett en detaljert spesifikasjon:

```bash
openspec spec create
```

Definer i spesifikasjonen:
- Design av autentiseringsflyt
- API-grensesnittdefinisjon
- Database tabellstruktur
- Sikkerhetshensyn

#### Trinn 3: Opprett endring

Start implementeringen:

```bash
openspec change create
```

Skriv inn endringsnavn: `add-user-authentication`

#### Trinn 4: Legg til oppgaver

```bash
openspec change add-task add-user-authentication
```

Legg til konkrete oppgaver:
- Implementer brukerregistreringsendepunkt
- Implementer innloggingsendepunkt
- Implementer JWT middleware
- Skriv enhetstester

#### Trinn 5: Implementering og validering

Etter at koden er fullført, valider endringen:

```bash
openspec change validate add-user-authentication
```

#### Trinn 6: Arkivering

Etter vellykket validering, arkiver endringen:

```bash
openspec change archive add-user-authentication
```

## Beste praksis

### Skriving av forslag

1. **Klar problemstilling**: Definer klart hvilket problem som skal løses, unngå vage beskrivelser
2. **Vurder alternative løsninger**: List og sammenlign forskjellige løsninger
3. **Vurder påvirkning**: Analyser påvirkning og risiko for det eksisterende systemet
4. **Hold det kortfattet**: Forslag bør være korte og tydelige; la detaljene være i spesifikasjonen

### Skriving av spesifikasjoner

1. **Detaljert, men ikke overflødig**: Gi nok detaljer, men unngå overdesign
2. **Bruk diagrammer**: Bruk flytskjemaer, arkitekturdiagrammer osv. for å hjelpe forklaringen
3. **Definer grensesnitt**: Definer klart alle offentlige grensesnitt og datastrukturer
4. **Vurder kanttilfeller**: List unntakssituasjoner og feilbehandling

### Endringsstyring

1. **Små iterasjoner**: Del store funksjoner i flere små endringer
2. **Oppdater i tide**: Hold oppgavestatus og dokumentasjon synkronisert
3. **Kodegjennomgang**: Gjennomfør kodegjennomgang etter at endringen er fullført
4. **Arkiver i tide**: Arkiver fullførte endringer i tide, hold arbeidsområdet ryddig

### Teamsamarbeid

1. **Regelmessige gjennomganger**: Gjennomgå forslag og spesifikasjoner regelmessig for å sikre teamkonsensus
2. **Dokumentasjon først**: Fullfør dokumentasjonen før du begynner å kode
3. **Oppretthold kommunikasjon**: Registrer diskusjoner og beslutninger i forslag og spesifikasjoner
4. **Versjonskontroll**: Inkluder OpenSpec-dokumentasjon i versjonskontrollen

## Vanlige bruksscenarier

### Scenario 1: Legge til ny funksjon

1. Opprett et forslag som beskriver kravene til den nye funksjonen
2. Teamet diskuterer og godkjenner forslaget
3. Skriv en detaljert teknisk spesifikasjon
4. Opprett en endring og implementer funksjonen
5. Valider og arkiver endringen

### Scenario 2: Refaktorere eksisterende kode

1. Opprett et forslag som forklarer årsaker og mål for refaktoreringen
2. Definer refaktorplan og trinn i spesifikasjonen
3. Opprett en endring og refaktorer gradvis
4. Sikre testdekning, valider refaktorresultatene
5. Arkiver endringen

### Scenario 3: Feilretting

For enkle feil:
- Kan rettes direkte, trenger ikke hele OpenSpec-prosessen

For komplekse feil:
1. Opprett et forslag for å analysere rotårsaken
2. Definer rettelsesløsningen i spesifikasjonen
3. Opprett en endring og implementer rettelsen
4. Legg til tester for å forhindre regresjon
5. Arkiver endringen

### Scenario 4: Håndtering av teknisk gjeld

1. Opprett et forslag som lister tekniske gjeldsposter
2. Definer en forbedringsplan i spesifikasjonen
3. Opprett en endring for gradvis nedbetaling av teknisk gjeld
4. Spor fremdrift og effekt
5. Arkiver fullførte forbedringer

## Integrasjon med andre verktøy

### Git-integrasjon

OpenSpec integreres sømløst med Git:

```bash
# OpenSpec-dokumentasjon bør inkluderes i versjonskontrollen
git add openspec/
git commit -m "Add OpenSpec proposal for feature X"
```

### CI/CD-integrasjon

Valider OpenSpec-spesifikasjoner i CI/CD-pipeline:

```yaml
# .github/workflows/openspec.yml
name: OpenSpec Validation
on: [pull_request]
jobs:
  validate:
    runs-on: ubuntu-latest
    steps:
      - uses: actions/checkout@v2
      - name: Install OpenSpec
        run: npm install -g @fission-ai/openspec@1
      - name: Validate changes
        run: openspec change validate
```

### IDE-integrasjon

Det anbefales å bruke en editor som støtter Markdown for å redigere OpenSpec-dokumenter:
- VS Code + Markdown-plugin
- JetBrains IDEs + Markdown-støtte
- Vim/Neovim + Markdown-plugin

## Feilsøking

### Vanlige problemer

#### Problem: OpenSpec-kommandoen ble ikke funnet

**Løsning**:
1. Bekreft at OpenSpec er installert: `npm list -g @fission-ai/openspec`
2. Kontroller at PATH miljøvariabelen inkluderer npm sin globale bin-katalog
3. Installer på nytt: `npm install -g @fission-ai/openspec@1`

#### Problem: Kan ikke opprette forslag eller endring

**Løsning**:
1. Bekreft at gjeldende katalog er prosjektets rot
2. Kontroller om `openspec/` katalogen eksisterer; hvis ikke, opprett den
3. Bekreft at du har skrivetillatelse

#### Problem: Validering mislyktes

**Løsning**:
1. Sjekk at dokumentasjonen i endringskatalogen er komplett
2. Bekreft at alle nødvendige oppgaver er fullført
3. Se detaljert feilmelding: `openspec change validate <change-name> --verbose`

#### Problem: Arkivering mislyktes

**Løsning**:
1. Forsikre deg om at endringen har bestått validering
2. Kontroller at `openspec/archived/` katalogen finnes
3. Bekreft at det ikke er ucommitted endringer i koden

## Avansert bruk

### Tilpassede maler

OpenSpec støtter tilpassede maler for forslag og spesifikasjoner:

```bash
# Opprett en mal-katalog i prosjektets rot
mkdir -p .openspec/templates

# Opprett en tilpasset forslagmal
cat > .openspec/templates/proposal.md << 'EOF'
# Proposal: {{title}}

## Problem Statement
{{problem}}

## Proposed Solution
{{solution}}

## Custom Section
Your custom content here
EOF
```

### Konfigurasjonsfil

Opprett en `.openspec.json` konfigurasjonsfil i prosjektets rot:

```json
{
  "version": "1.0",
  "templates": {
    "proposal": ".openspec/templates/proposal.md",
    "spec": ".openspec/templates/spec.md"
  },
  "validation": {
    "requireTests": true,
    "requireDocs": true
  },
  "hooks": {
    "pre-archive": "npm test"
  }
}
```

### Hooks

OpenSpec støtter kjøring av egendefinerte skript før og etter bestemte operasjoner:

```json
{
  "hooks": {
    "pre-create": "scripts/check-permissions.sh",
    "post-archive": "scripts/notify-team.sh"
  }
}
```
