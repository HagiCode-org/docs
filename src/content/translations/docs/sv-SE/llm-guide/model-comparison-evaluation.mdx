## Omfattning och metod

- **Mål**: Ge vägledning för modellval baserad på verklig integrationserfarenhet i HagiCode.
- **Uppgiftstyper**: Frontend-komponentimplementering, backend API refactoring, testslutförande och dokumentationsgenerering.
- **Utvärderingsaxlar**: leveranseffektivitet (kan det på ett tillförlitligt sätt slutföra verkliga uppgifter) och kostnadseffektivitet (kostnad + inhemsk tillgänglighet).

## Anteckningar om testtid och scenario

- **Senaste testdatum**: 2026-03-08
- **Testperiod**: 2026-03-01 till 2026-03-08
- **Samplebas**: Subjektiv utvärdering från verkliga HagiCode-arbetsflöden, inte leverantörsbenchmarksiffror.
- **Tillämpning**: Slutsatserna är anpassade till detta projekts nuvarande arbetsflöde och begränsningar.

### Nyckelfynd

- Förutom Qwen kan alla andra testade modeller nå våra mål till viss del.
- GLM 4.7+ (GLM 4.7 och GLM 5) uppfyller i allmänhet våra krav smidigt.
- GPT 5.3 Codex och GPT 5.4 kompletterar inte bara kraven utan ger också bättre ingenjörspraxis och implementeringskvalitet.
- Minimax M2.5 har en återkommande svaghet: kodstängningsfel (t.ex. ofullständig parentes/blockstängning), så extra granskning behövs.

> Notera: den här rankningen skiljer sig avsiktligt från rankningen av leveranseffektivitet.

## Valvägledning

- **Kvalitet först**: välj GPT 5.4 / GPT 5.3 Codex.
- **Balanserad strategi**: välj GLM 5 / GLM 4.7.
- **Kostnad först**: välj Minimax M2.5 (med strängare kodstängningskontroller).
- **Praktisk routing**: använd premiummodeller för kritiska uppgifter och kostnadseffektiva modeller för rutinuppgifter.

## Otestad modellutlåtande

För modeller som inte är listade här har vi för närvarande **inga testdata** och **ingen praktisk erfarenhet**, så vi tillhandahåller inga utvärderingar.

Om sponsorer ger tillgång till ytterligare modeller kommer vi att köra erfarenhetsbaserade utvärderingar i vårt verkliga arbetsflöde och uppdatera den här sidan.
