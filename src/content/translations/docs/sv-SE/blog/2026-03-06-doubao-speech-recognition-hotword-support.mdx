## Guide till implementering av hotword-stöd för Doubao taligenkänning

> Den här artikeln förklarar i detalj hur man implementerar kommandoordsstöd för Doubao taligenkänning i HagiCode-projektet. Genom att använda både anpassade kommandoord och plattformskommandotabeller kan du avsevärt förbättra igenkänningsnoggrannheten för domänspecifikt ordförråd.

## Bakgrund

Taligenkänningstekniken har utvecklats i många år, men ett problem har konsekvent besvärat utvecklare. Allmänna modeller för taligenkänning kan täcka vardagligt språk, men de kommer ofta till korta när det kommer till professionell terminologi, produktnamn och personnamn. Tänk på det: en röstassistent inom det medicinska området måste noggrant känna igen termer som "hypertoni", "diabetes" och "koronar hjärtsjukdom"; ett rättssystem behöver exakt fånga termer som "ansökan", "försvar" och "bevisbörda". I dessa scenarier gör en generell modell sitt bästa, men det räcker ofta inte.

Vi stötte på samma utmaning i HagiCode-projektet. Som en multifunktionell AI-kodningsassistent behöver HagiCode hantera taligenkänning för ett brett spektrum av teknisk terminologi. Doubao taligenkännings-API, i sin standardkonfiguration, kunde dock inte helt uppfylla våra noggrannhetskrav för specialiserade termer. Det är inte så att Doubao inte är tillräckligt bra; snarare har varje domän sitt eget terminologisystem. Efter lite forskning och teknisk utforskning fann vi att Doubao taligenkännings-API faktiskt ger stöd för kommandoord. Med en enkel konfiguration kan den avsevärt förbättra igenkänningsnoggrannheten för ett specifikt ordförråd. På sätt och vis, när du väl berättar vilka ord den ska vara uppmärksam på, lyssnar den efter dem mer noggrant.

Vad den här artikeln delar är den kompletta lösningen som vi använde i HagiCode-projektet för att implementera Doubao taligenkänningskomord. Båda lägena, anpassade kommandoord och plattformskommandotabeller, är tillgängliga, och de kan också kombineras. Med den här lösningen kan utvecklare flexibelt konfigurera kommandoord baserat på affärsscenarier så att taligenkänningssystemet bättre kan "känna igen" professionellt, ovanligt men ändå kritiskt ordförråd.

## Kärnimplementering

### Förstå de två hotword-lägena

Doubao taligenkännings-API ger två sätt att konfigurera kommandoord, och vart och ett har sina egna idealiska användningsfall och fördelar.

**Anpassat kommandoordsläge** låter oss skicka kommandoordstext direkt genom `corpus.context` fältet. Detta tillvägagångssätt är särskilt lämpligt för scenarier där du snabbt behöver konfigurera ett litet antal kommandoord, som att tillfälligt känna igen ett produktnamn eller en persons namn. I HagiCodes implementering analyserar vi den flerradiga kommandoordstexten som användaren matat in i en lista med strängar och formaterar den sedan till `context_data` array som krävs av Doubao API. Det här tillvägagångssättet är väldigt direkt: du berättar helt enkelt för systemet vilka ord du ska uppmärksamma, och det gör precis det.

**Plattforms kommandoordstabellläge** använder `corpus.boosting_table_id` för att referera till en förkonfigurerad kommandoordstabell i Doubaos självlärande plattform. Detta tillvägagångssätt är lämpligt för scenarier där du behöver hantera ett stort antal kommandoord. Vi kan skapa och underhålla kommandoordstabeller på Doubaos självlärande plattform och sedan referera till dem med ID. För ett projekt som HagiCode, där specialiserade termer måste uppdateras och underhållas kontinuerligt, erbjuder detta läge mycket bättre hanterbarhet. När antalet kommandoord växer är det mycket bättre att ha en centraliserad plats för att hantera dem än att ange dem manuellt varje gång.

Intressant nog kan dessa två lägen också användas tillsammans. Doubao API stöder inkludering av både anpassade kommandoord och ett ID för plattformskommandotabeller i samma begäran, med kombinationsstrategin kontrollerad av `combine_mode` parameter. Denna flexibilitet gör att HagiCode kan hantera ett brett utbud av komplexa behov av professionell terminologiigenkänning. Ibland ger en kombination av flera tillvägagångssätt bättre resultat.

### Gränssnittstypdefinitioner och validering

I HagiCodes frontend-implementering definierade vi en komplett uppsättning kommandoordskonfigurationstyper och valideringslogik. Den första delen är typdefinitionen:

```typescript
export interface HotwordConfig {
  contextText: string;           // Multi-line hotword text
  boostingTableId: string;      // Doubao platform hotword table ID
  combineMode: boolean;          // Whether to use both together
}
```

Detta enkla gränssnitt innehåller alla konfigurationsobjekt för kommandoordsfunktionen. Bland dem, `contextText` är den del som användarna interagerar med mest direkt: vi tillåter användare att ange en kommandoordsfras per rad, vilket är väldigt intuitivt. Att be användarna att ange en term per rad är mycket lättare än att få dem att förstå ett komplicerat konfigurationsformat.

Därefter kommer valideringsfunktionen. Baserat på Doubaos API-begränsningar definierade vi strikta valideringsregler: högst 100 rader med kommandoordstext, upp till 50 tecken per rad och inte mer än 5000 tecken totalt; `boosting_table_id` kan vara högst 200 tecken och får endast innehålla bokstäver, siffror, understreck och bindestreck. Dessa gränser är inte godtyckliga; de kommer direkt från den officiella Doubao-dokumentationen. API-gränser är API-gränser, och vi måste följa dem.

```typescript
export function validateContextText(contextText: string): HotwordValidationResult {
  if (!contextText || contextText.trim().length === 0) {
    return { isValid: true, errors: [] };
  }

  const lines = contextText.split('\n').filter(line => line.trim().length > 0);
  const errors: string[] = [];

  if (lines.length > 100) {
    errors.push(`Hotword line count cannot exceed 100 lines; current count is ${lines.length}`);
  }

  const totalChars = contextText.length;
  if (totalChars > 5000) {
    errors.push(`Total hotword character count cannot exceed 5000; current count is ${totalChars}`);
  }

  for (let i = 0; i < lines.length; i++) {
    if (lines[i].length > 50) {
      errors.push(`Hotword on line ${i + 1} exceeds the 50-character limit`);
    }
  }

  return { isValid: errors.length === 0, errors };
}

export function validateBoostingTableId(boostingTableId: string): HotwordValidationResult {
  if (!boostingTableId || boostingTableId.trim().length === 0) {
    return { isValid: true, errors: [] };
  }

  const errors: string[] = [];

  if (boostingTableId.length > 200) {
    errors.push(`boosting_table_id cannot exceed 200 characters; current count is ${boostingTableId.length}`);
  }

  if (!/^[a-zA-Z0-9_-]+$/.test(boostingTableId)) {
    errors.push('boosting_table_id can contain only letters, numbers, underscores, and hyphens');
  }

  return { isValid: errors.length === 0, errors };
}
```

Dessa valideringsfunktioner körs omedelbart när användaren konfigurerar kommandoord, vilket säkerställer att problem upptäcks så tidigt som möjligt. Ur ett användarupplevelseperspektiv är denna typ av omedelbar feedback mycket viktig. Det är alltid bättre för användare att veta vad som är fel medan de skriver i stället för efter att de har skickat in.

### Frontend Configuration Persistence

I HagiCodes frontend-implementering valde vi att använda webbläsarens `localStorage` för att lagra kommandoordskonfigurationen. Det fanns flera överväganden bakom detta designbeslut. För det första är kommandoordskonfigurationen mycket personlig, och olika användare kan ha olika domänspecifika behov. För det andra förenklar denna metod implementeringen av backend eftersom den inte kräver extra databastabeller eller API-slutpunkter. Slutligen, efter att användare har konfigurerat det en gång i webbläsaren, kan inställningarna laddas automatiskt vid efterföljande användningar, vilket är mycket bekvämt. Enkelt uttryckt är det det enklaste tillvägagångssättet.

```typescript
const HOTWORD_STORAGE_KEYS = {
  contextText: 'hotword-context-text',
  boostingTableId: 'hotword-boosting-table-id',
  combineMode: 'hotword-combine-mode',
} as const;

export const DEFAULT_HOTWORD_CONFIG: HotwordConfig = {
  contextText: '',
  boostingTableId: '',
  combineMode: false,
};

// Load hotword configuration
export function loadHotwordConfig(): HotwordConfig {
  const contextText = localStorage.getItem(HOTWORD_STORAGE_KEYS.contextText) || '';
  const boostingTableId = localStorage.getItem(HOTWORD_STORAGE_KEYS.boostingTableId) || '';
  const combineMode = localStorage.getItem(HOTWORD_STORAGE_KEYS.combineMode) === 'true';

  return { contextText, boostingTableId, combineMode };
}

// Save hotword configuration
export function saveHotwordConfig(config: HotwordConfig): void {
  localStorage.setItem(HOTWORD_STORAGE_KEYS.contextText, config.contextText);
  localStorage.setItem(HOTWORD_STORAGE_KEYS.boostingTableId, config.boostingTableId);
  localStorage.setItem(HOTWORD_STORAGE_KEYS.combineMode, String(config.combineMode));
}
```

Logiken i denna kod är enkel och tydlig. Vi läser från `localStorage` när du laddar konfigurationen och skriv till `localStorage` när du sparar den. Vi tillhandahåller också en standardkonfiguration så att systemet fortfarande kan fungera korrekt när ingen konfiguration finns ännu. Det måste trots allt finnas en förnuftig standard.

### Backend SDK Configuration Extension

I HagiCodes backend-implementering behövde vi lägga till hotword-relaterade egenskaper till SDK-konfigurationsklassen. Med hänsyn till C#-språkets egenskaper och användningsmönster använde vi `List<string>` för att lagra anpassade kommandoordskontexter:

```csharp
public class DoubaoVoiceConfig
{
    /// <summary>
    /// App ID
    /// </summary>
    public string AppId { get; set; } = string.Empty;

    /// <summary>
    /// Access token
    /// </summary>
    public string AccessToken { get; set; } = string.Empty;

    /// <summary>
    /// Service URL
    /// </summary>
    public string ServiceUrl { get; set; } = string.Empty;

    /// <summary>
    /// Custom hotword context list
    /// </summary>
    public List<string>? HotwordContexts { get; set; }

    /// <summary>
    /// Doubao platform hotword table ID
    /// </summary>
    public string? BoostingTableId { get; set; }
}
```

Designen av denna konfigurationsklass följer HagiCodes vanliga koncisa stil. `HotwordContexts` är en nollbar listtyp och `BoostingTableId` är en nullbar sträng, så när det inte finns någon kommandoordskonfiguration har dessa egenskaper ingen effekt på begäran alls. Om du inte använder funktionen bör den hållas ur vägen.

### Nyttolastkonstruktionslogik

Nyttolastkonstruktion är kärnan i hela kommandoordsfunktionen. När vi väl har konfigurerat kommandoord måste vi formatera det till JSON-strukturen som krävs av Doubao API. Denna process sker innan SDK:n skickar begäran:

```csharp
private void AddCorpusToRequest(Dictionary<string, object> request)
{
    var corpus = new Dictionary<string, object>();

    // Add custom hotwords
    if (Config.HotwordContexts != null && Config.HotwordContexts.Count > 0)
    {
        corpus["context"] = new Dictionary<string, object>
        {
            ["context_type"] = "dialog_ctx",
            ["context_data"] = Config.HotwordContexts
                .Select(text => new Dictionary<string, object> { ["text"] = text })
                .ToList()
        };
    }

    // Add platform hotword table ID
    if (!string.IsNullOrEmpty(Config.BoostingTableId))
    {
        corpus["boosting_table_id"] = Config.BoostingTableId;
    }

    // Add corpus to the request only when it is not empty
    if (corpus.Count > 0)
    {
        request["corpus"] = corpus;
    }
}
```

Den här koden visar hur man dynamiskt konstruerar `corpus` fält baserat på konfiguration. Det viktigaste är att vi lägger till `corpus` endast när kommandoordskonfigurationen faktiskt finns. Denna design säkerställer bakåtkompatibilitet: när inga kommandoord är konfigurerade förblir begäranstrukturen exakt densamma som tidigare. Bakåtkompatibilitet spelar roll; att lägga till en funktion bör inte störa befintlig logik.

### WebSocket Parameter Passing

Mellan frontend och backend skickas kommandoordsparametrar genom WebSocket-kontrollmeddelanden. HagiCode är utformad så att när frontend börjar spela in, laddar den kommandoordskonfigurationen från `localStorage` och skickar den till backend via ett WebSocket-meddelande.

```typescript
const controlMessage = {
  type: 'control',
  payload: {
    command: 'StartRecognition',
    contextText: '高血压\n糖尿病\n冠心病',
    boosting_table_id: 'medical_table',
    combineMode: false
  }
};
```

Det finns en detalj att notera här: frontend skickar flerradstext separerad av nyradstecken, och backend måste analysera den. Backend WebSocket-hanteraren analyserar dessa parametrar och skickar dem till SDK:n:

```csharp
private async Task HandleControlMessageAsync(
    string connectionId,
    DoubaoSession session,
    ControlMessage message)
{
    if (message.Payload is SessionControlRequest controlRequest)
    {
        // Parse hotword parameters
        string? contextText = controlRequest.ContextText;
        string? boostingTableId = controlRequest.BoostingTableId;
        bool? combineMode = controlRequest.CombineMode;

        // Parse multi-line text into a hotword list
        if (!string.IsNullOrEmpty(contextText))
        {
            var hotwords = contextText
                .Split('\n', StringSplitOptions.RemoveEmptyEntries)
                .Select(s => s.Trim())
                .Where(s => s.Length > 0)
                .ToList();

            session.HotwordContexts = hotwords;
        }

        session.BoostingTableId = boostingTableId;
    }
}
```

Med denna design blir det tydligt och effektivt att överföra kommandoordskonfiguration från frontend till backend. Det finns inget särskilt mystiskt med det; data förs helt enkelt igenom lager för lager.

## Praktisk guide

### Konfigurera anpassade kommandoord

I verklig användning är det mycket enkelt att konfigurera anpassade kommandoord. Öppna sidan för taligenkänningsinställningar i HagiCode och hitta avsnittet "Hotword Configuration". I inmatningsrutan "Custom Hotword Text" anger du en hotword-fras per rad.

Om du till exempel utvecklar en medicinskt relaterad applikation kan du konfigurera den så här:

```
高血压
糖尿病
冠心病
心绞痛
心肌梗死
心力衰竭
```

När du har sparat konfigurationen skickas dessa kommandoord automatiskt till Doubao API varje gång taligenkänning startar. I våra tester förbättrades igenkänningsnoggrannheten för relaterade professionella termer märkbart när kommandoorden väl konfigurerats. Förbättringen är verklig, och klart bättre än tidigare.

### Konfigurera en plattformskommandotabell

Om du behöver hantera ett stort antal hotwords, eller om hotwords behöver frekventa uppdateringar, passar plattformens hotword-tabellläge bättre. Skapa först en kommandoordstabell på Doubaos självlärande plattform och skaffa den genererade `boosting_table_id`, ange sedan detta ID på inställningssidan för HagiCode.

Doubaos självlärande plattform tillhandahåller funktioner som massimport och kategoriserad hantering för kommandoord, vilket är mycket praktiskt för team som behöver hantera stora uppsättningar av specialiserad terminologi. Genom att hantera kommandoord på plattformen kan du underhålla dem centralt och rulla ut uppdateringar konsekvent. När kommandoordslistan blir stor är det mycket mer praktiskt att ha en enda plats att hantera den än manuellt varje gång.

### Använder kombinationsläge

I vissa komplexa scenarier kan du behöva använda både anpassade kommandoord och en plattformskommandotabell samtidigt. I så fall konfigurerar du helt enkelt båda i HagiCode och aktiverar omkopplaren "Kombinationsläge".

I kombinationsläge tar Doubao API hänsyn till båda kommandoordskällorna samtidigt, så igenkänningsnoggrannheten är vanligtvis högre än att använda endera källan ensam. Det är dock värt att notera att kombinationsläget ökar begärans komplexitet, så det är bäst att bestämma om det ska aktiveras efter praktiska tester. Mer komplexitet är bara värt det om de verkliga resultaten motiverar det.

### Exempel på kodintegrering

Att integrera hotword-funktionen i HagiCode-projektet är mycket enkelt. Här är några vanliga kodavsnitt:

```typescript
import {
  loadHotwordConfig,
  saveHotwordConfig,
  validateHotwordConfig,
  parseContextText,
  getEffectiveHotwordMode,
  type HotwordConfig
} from '@/types/hotword';

// Load and validate configuration
const config = loadHotwordConfig();
const validation = validateHotwordConfig(config);

if (!validation.isValid) {
  console.error('Hotword configuration validation failed:', validation.errors);
  return;
}

// Parse hotword text
const hotwords = parseContextText(config.contextText);
console.log('Parsed hotwords:', hotwords);

// Get effective hotword mode
const mode = getEffectiveHotwordMode(config);
console.log('Current hotword mode:', mode);
```

## Saker att tänka på

Det finns flera punkter som förtjänar särskild uppmärksamhet när du implementerar och använder kommandoordsfunktionen.

Först är teckenbegränsningen. Doubao API har strikta begränsningar för kommandoord, inklusive radantal, tecken per rad och totalt antal tecken. Om någon gräns överskrids returnerar API:et ett fel. I HagiCodes frontend-implementering kontrollerar vi dessa begränsningar under användarinmatning genom valideringsfunktioner, vilket förhindrar att ogiltiga konfigurationer skickas till backend. Att fånga problem tidigt är alltid bättre än att vänta på att API:et ska misslyckas.

För det andra är formatet på `boosting_table_id`. Det här fältet tillåter endast bokstäver, siffror, understreck och bindestreck, och det får inte innehålla mellanslag eller andra specialtecken. När du skapar en kommandoordstabell på Doubaos självlärande plattform, se till att följa namnreglerna. Den typen av strikt formatvalidering är vanligt för API:er.

För det tredje är bakåtkompatibilitet. Hotword-parametrar är helt valfria. Om inga kommandoord är konfigurerade, fungerar systemet exakt som det gjorde tidigare. Denna design säkerställer att befintliga användare inte påverkas på något sätt, och den gör också gradvis migrering och uppgraderingar enklare. Att lägga till en funktion bör inte störa den tidigare logiken.

Slutligen finns det felhantering. När kommandoordskonfigurationen är ogiltig returnerar Doubao API motsvarande felmeddelanden. HagiCodes implementering registrerar detaljerade loggar för att hjälpa utvecklare att felsöka problem. Samtidigt visar frontend valideringsfel i användargränssnittet för att hjälpa användare att korrigera konfigurationen. Bra felhantering leder naturligtvis till en bättre användarupplevelse.
