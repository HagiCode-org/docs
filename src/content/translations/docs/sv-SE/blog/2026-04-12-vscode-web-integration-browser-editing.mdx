## Snabb kodredigering i webbläsaren: VSCode webbintegration i praktiken

> När AI är klar med att analysera kod, hur öppnar du omedelbart en redigerare i webbläsaren och börjar göra ändringar? Den här artikeln delar vår praktiska erfarenhet av att integrera kodserver i HagiCode-projektet för att skapa en sömlös bro mellan AI-assistenten och kodredigeringsupplevelsen.

## Bakgrund

I en tidevarv med AI-assisterad programmering behöver utvecklare ofta inspektera och redigera kod snabbt. Det traditionella arbetsflödet är enkelt: öppna projektet i en stationär IDE, leta reda på filen, redigera den och spara. Men i vissa situationer känns det flödet alltid något av.

Scenario ett: fjärrutveckling. När du använder en AI-assistent som HagiCode kan backend köras på en fjärrserver eller inuti en behållare, och lokala maskiner kan inte direkt komma åt projektfilerna. Varje gång du behöver inspektera eller modifiera kod måste du ansluta via SSH eller annan metod, och upplevelsen känns splittrad. Det är som att vilja träffa någon genom en tjock glasruta: du kan se dem, men du kan inte nå dem.

Scenario två: snabba förhandsvisningar. Efter att AI-assistenten analyserat koden kanske användaren bara vill snabbt bläddra i en fil eller göra en liten ändring. Att lansera en komplett skrivbords-IDE känns tung, medan en lätt redigerare i webbläsaren bättre passar behovet av en "snabb look". När allt kommer omkring, vem vill mobilisera en hel verktygskedja bara för att ta en blick?

Scenario tre: samarbete över enheter. När du arbetar på olika enheter tillhandahåller en webbläsarbaserad redigerare en enhetlig åtkomstpunkt utan att varje maskin behöver konfigureras med en utvecklingsmiljö. Bara det sparar mycket besvär. Livet är kort; varför upprepa samma installationsarbete om och om igen?

För att lösa dessa smärtpunkter integrerade vi VSCode Web i HagiCode-projektet. Detta låter AI-assistenten och kodredigeringsupplevelsen ansluta sömlöst: efter att AI analyserat koden kan användare omedelbart öppna en redigerare och göra ändringar i samma webbläsarsession, utan att byta applikation. Det är den typen av upplevelse där den bara finns där när du behöver den.

## Tekniskt val: Varför kodserver?

Bland de många VSCode webblösningar som finns, valde vi kodserver. Det fanns några konkreta skäl bakom det beslutet.

Funktionens fullständighet. kodserver är webbversionen av VSCode och stöder de flesta skrivbordsfunktioner, inklusive tilläggssystemet, intelligenta förslag, felsökning och mer. Det betyder att användare kan få en redigeringsupplevelse i webbläsaren som är mycket nära skrivbordsversionen. När allt kommer omkring, vem vill egentligen kompromissa med funktionalitet?

Flexibel distribution. kodserver kan köras som en oberoende tjänst och stöder även Docker-baserad distribution, vilket passar bra med HagiCodes arkitektur. Vår backend är skriven i C#, frontend använder React, och de två kommunicerar med kodservertjänsten via REST API:er. Det är som att bygga med block: varje del har sin plats.

Säker autentisering. kodserver innehåller en inbyggd `connection-token` mekanism för att förhindra obehörig åtkomst. Varje session har en unik token så att endast behöriga användare kan öppna redigeraren. Säkerhet är en av de saker du bara uppskattar till fullo när du väl har den.

## Arkitektur design

HagiCodes VSCode Web-integration använder en front-end/back-end-separerad arkitektur.

### Frontend Service Layer

Frontend omsluter interaktioner med backend genom `vscodeServerService.ts`:

```typescript
// Open project
export async function openProjectInCodeServer(
  id: string,
  currentInterfaceLanguage?: string,
): Promise<VsCodeServerLaunchResponseDto>

// Open vault
export async function openVaultInCodeServer(
  id: string,
  path?: string,
  currentInterfaceLanguage?: string,
): Promise<VsCodeServerLaunchResponseDto>
```

Skillnaden mellan dessa två metoder är enkel: `openProjectInCodeServer` öppnar hela projektet, medan `openVaultInCodeServer` öppnar en specifik väg inuti ett valv. För MonoSpecs multi-repository-projekt skapar systemet automatiskt en arbetsytefil. Tydliga ansvarsområden räcker ofta när varje del gör sitt eget jobb bra.

### Backend Service Layer

Backend `VaultAppService.cs` implementerar kärnlogiken:

```csharp
public async Task<VsCodeServerLaunchResponseDto> OpenInCodeServerAsync(
    string id,
    string? relativePath = null,
    string? currentInterfaceLanguage = null,
    CancellationToken cancellationToken = default)
{
    // 1. Get settings and check whether the feature is enabled
    var settings = await _vsCodeServerSettingsService.GetResolvedSettingsAsync(cancellationToken);
    if (!settings.Enabled) {
        throw new BusinessException(VsCodeServerErrorCodes.Disabled, "VSCode Server is disabled.");
    }

    // 2. Get vault and resolve the launch directory
    var vault = await RequireVaultAsync(id, cancellationToken);
    var launchDirectory = ResolveLaunchDirectory(vault, relativePath);

    // 3. Ensure code-server is running and get runtime info
    var runtime = await _vsCodeServerManager.EnsureStartedAsync(settings, cancellationToken);

    // 4. Resolve language settings
    var language = _vsCodeServerSettingsService.ResolveLaunchLanguage(
        settings.Language,
        currentInterfaceLanguage);

    // 5. Build launch URL
    return new VsCodeServerLaunchResponseDto {
        LaunchUrl = AppendQueryString(runtime.BaseUrl, new Dictionary<string, string?> {
            ["folder"] = launchDirectory,
            ["tkn"] = runtime.ConnectionToken,
            ["vscode-lang"] = language,
        }),
        ConnectionToken = runtime.ConnectionToken,
        OpenMode = "folder",
        Runtime = VsCodeServerSettingsService.MapRuntime(
            await _vsCodeServerManager.GetRuntimeSnapshotAsync(cancellationToken)),
    };
}
```

Den här metoden har ett mycket tydligt ansvar: kontrollera inställningar, lös sökvägar, starta tjänsten och bygg URL:en. Bland dem, den `ResolveLaunchDirectory` metod utför sökvägssäkerhetskontroller för att förhindra vägövergångsattacker. Kod kan kännas lite som poesi när varje rad har ett syfte.

### Automatisk körtidshantering

Backend hanterar kodserverprocessen genom `VsCodeServerManager`:

- Kontrollera processstatus
- Starta stoppade tjänster automatiskt
- Returnera körtidsbilder som port, process-ID och starttid

Denna design låter systemet automatiskt hantera kodserverns livscykel, så att användarna inte behöver hantera tjänsteprocesser manuellt. Livet är redan tillräckligt komplicerat; allt som kan automatiseras borde vara det.

### Språksynkronisering

HagiCode stöder ett flerspråkigt gränssnitt, och kodservern måste följa den inställningen. Systemet stöder tre språklägen:

- `follow`: Följ det aktuella gränssnittsspråket
- `zh-CN`: fast till kinesiska
- `en-US`: fast till engelska

Inställningen skickas till kodservern via `vscode-lang` URL-parameter så att redigeringsspråket förblir konsekvent med HagiCode-gränssnittet. Språket mår bäst när det är enhetligt.

### MonoSpecs arbetsyta för flera arkiv

För MonoSpecs-projekt, som innehåller flera underlager i en monorepo, skapar systemet automatiskt en `.code-workspace` fil:

```csharp
private async Task<string> CreateWorkspaceFileAsync(Project project, Guid projectId)
{
    var folders = await ResolveWorkspaceFoldersAsync(project.Path);
    var workspaceDocument = new {
        folders = folders.Select(path => new { path }).ToArray(),
    };
    // Generate workspace file...
}
```

Detta gör det möjligt att redigera flera sub-repositories i samma kodserverinstans, vilket är särskilt praktiskt för stora monorepo-projekt. Flera arkiv i ett fönster kan kännas som flera berättelser samlade i samma bok.

## Frontend-integration

HagiCode-gränssnittet använder React + TypeScript, och det är inte särskilt komplicerat att integrera kodservern.

### Snabbåtgärdsknapp

Lägg till en kodserverknapp till projektkortet:

```tsx
// QuickActionsZone.tsx
<Button
  size="sm"
  variant="default"
  onClick={() => onAction({ type: 'open-code-server' })}
>
  <Globe className="h-3 w-3 mr-1" />
  <span className="text-xs">{t('project.openCodeServer')}</span>
</Button>
```

Den här knappen utlöser den öppna åtgärden och anropar backend-API:et för att få start-URL. En knapp, en åtgärd, direkt och enkelt.

### Hantera den öppna åtgärden

```tsx
const handleAction = async (action: ProjectAction) => {
  if (action.type === 'open-code-server') {
    const response = await openProjectInCodeServer(project.id, i18n.language);
    window.open(response.launchUrl, '_blank', 'noopener,noreferrer');
  }
};
```

Använd `window.open` för att öppna kodservern på en ny flik. Den `noopener,noreferrer` parametrar ger extra säkerhet. När det kommer till säkerhet finns det inget som heter att vara för försiktig.

### Valvredigeringspost

Lägg till en liknande redigeringsknapp i valvlistan:

```tsx
const handleEditVault = async (vault: VaultItemDto) => {
  const response = await openVaultInCodeServer(vault.id);
  window.open(response.launchUrl, '_blank', 'noopener,noreferrer');
};
```

Projekt och valv använder samma öppna mekanism, vilket håller interaktionen konsekvent. Konsistens betyder nästan lika mycket som själva funktionen.

## URL-konstruktionslogik

URL-formatet för kodserver har några detaljer värda att notera.

**Mappläge**:
```
http://{host}:{port}/?folder={path}&tkn={token}&vscode-lang={lang}
```

**Arbetsutrymmesläge**:
```
http://{host}:{port}/?workspace={workspacePath}&tkn={token}&vscode-lang={lang}
```

Här, `tkn` är anslutningstoken. Den genereras automatiskt varje gång kodservern startar, vilket säkerställer säker åtkomst. Den `vscode-lang` parametern styr språket för redigeringsgränssnittet. Var och en av dessa parametrar har en roll att spela.

## Säkerhetsöverväganden

När du implementerar kod-serverintegration förtjänar säkerheten särskild uppmärksamhet. Om säkerheten går fel märker du alltid för sent.

### Anslutningstoken

Den `connection-token` genereras slumpmässigt och bör inte exponeras. Det är bäst att använda under HTTPS för att förhindra att token fångas upp av en man-i-mitten. Känslig information är värd att skydda ordentligt.

### Filsökvägssäkerhet

Backend implementerar sökvägskontroller:

```csharp
private static string ResolveLaunchDirectory(VaultRegistryEntry vault, string? relativePath)
{
    var vaultRoot = EnsureTrailingSeparator(Path.GetFullPath(vault.PhysicalPath));
    var combinedPath = Path.GetFullPath(Path.Combine(vaultRoot, relativePath ?? "."));
    if (!combinedPath.StartsWith(vaultRoot, StringComparison.OrdinalIgnoreCase))
    {
        throw new BusinessException(VaultRelativePathTraversalCode, "Relative path traversal detected.");
    }
    return combinedPath;
}
```

Denna kod säkerställer att användare inte kan använda `../` eller liknande mönster för att komma åt filer utanför Vault-katalogen. Gränskontroller är alltid bättre att göra än att hoppa över.

### Tillståndskontroll

Kodserverprocessen bör köras med lämpliga användarbehörigheter så att den inte kan komma åt känsliga systemfiler. Det är bäst att köra kodservertjänsten under en dedikerad användare. Behörighetskontroll är en av de grundläggande sakerna du alltid bör hålla på plats.

## Prestandaoptimering

kodservern förbrukar serverresurser, så här är några optimeringsförslag:

- Övervaka CPU- och minnesanvändning och justera resursgränserna vid behov
- Stora projekt kan kräva längre timeouts
- Implementera automatisk rensning av sessionstimeout för att frigöra resurser
- Överväg cachning för att minska upprepad beräkning

HagiCode tillhandahåller ett körtidsstatusövervaknings-API, och frontend kan anropa `getVsCodeServerSettings()` för att hämta det aktuella läget:

```typescript
const { settings, runtime } = await getVsCodeServerSettings();
// runtime.status: 'disabled' | 'stopped' | 'starting' | 'running' | 'unhealthy'
// runtime.baseUrl: "http://localhost:8080"
// runtime.processId: 12345
```

Denna design tillåter användare att tydligt förstå hälsostatusen för kodservern och snabbt lokalisera problem när något går fel. När statusen är synlig känner folk sig mer i kontroll.

## Information om användarupplevelse

Under implementeringen upptäckte vi några detaljer som märkbart påverkar användarupplevelsen och förtjänar extra uppmärksamhet.

Att öppna kodservern för första gången kan kräva att man väntar på start, och den fördröjningen kan variera från några sekunder till en halv minut. Det är en bra idé att visa ett laddningstillstånd i frontend så att användarna vet att systemet fortfarande fungerar. Det är lättare att vänta när det finns feedback.

Webbläsare kan blockera popup-fönstret, så användare bör uppmanas att tillåta det manuellt. Vid första lanseringen visar HagiCode vägledning som förklarar hur man beviljar nödvändiga webbläsarbehörigheter. Användarupplevelsen bor ofta i just dessa små detaljer.

Det är också en bra idé att visa körtidsstatus som start, körning eller fel, så att när problem uppstår kan användare snabbt se om problemet är på serversidan eller i deras egen drift. Att veta var problemet finns ger dig åtminstone en plats att börja.

## Konfigurationsexempel

Konfigurationen för kodserver är inte komplicerad:

```json
{
  "vscodeServer": {
    "enabled": true,
    "host": "0.0.0.0",
    "port": 8080,
    "language": "follow"
  }
}
```

`enabled` styr om funktionen är påslagen, `host` och `port` definiera avlyssningsadressen, och `language` ställer in språkläget. Dessa inställningar kan ändras via användargränssnittet och träder i kraft omedelbart. Enkla saker är ofta de enklaste att använda.
