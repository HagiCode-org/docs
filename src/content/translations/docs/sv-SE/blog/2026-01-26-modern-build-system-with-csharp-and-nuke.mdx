## Goodbye Script Hell: Varför vi valde C# för ett modernt byggsystem

> En djupdykning i hur HagiCode-projektet använder Nuke för att bygga en typsäker, plattformsoberoende och mycket utbyggbar automatiserad byggpipeline, som helt tar itu med underhållsproblemen med traditionella byggskript.

## Bakgrund

Under hela mjukvaruutvecklingens långa resa tenderar ordet "bygga" att inspirera till både kärlek och frustration. Kärlek, för med ett klick förvandlas koden till en produkt, och det är ett av de mest tillfredsställande ögonblicken i programmering. Frustration, eftersom det kan vara en mardröm att behålla en hög med stökiga skript.

I många projekt är vi vana vid att skriva skript i Python eller förlita oss på XML-konfigurationsfiler. Föreställ dig bara rädslan för att bli styrd av `<property>` block. Men när projektkomplexiteten växer, särskilt i ett projekt som HagiCode som spänner över frontend- och backendarbete, flera plattformar och flera språk, börjar traditionella byggmetoder att visa sina gränser. Utspridda skriptlogik, ingen typkontroll, svagt IDE-stöd... var och en av dessa problem blir en liten fallgrop som gör att teamet blir smutsigt då och då.

För att lösa dessa smärtpunkter introducerade HagiCode-projektet **Nuke**, ett modernt byggsystem baserat på C#. Det är inte bara ett verktyg. Det är ett annat sätt att tänka på att bygga arbetsflöden. Idag ska vi titta på varför vi valde det och hur det dramatiskt har förbättrat vår utvecklingsupplevelse.

## Kärnanalys: Varför Nuke?

Du kanske undrar: "Det finns redan så många byggsystem, som Make, Gradle eller till och med vanliga Shell-skript. Varför gå ur vägen för att använda ett C#-baserat?"

Det är en rättvis fråga. Nukes kärna överklagande är att det tar med de programmeringsspråksfunktioner vi redan känner väl till en värld av byggskript.

### 1. Modularizing Build Workflows: The Art of Targets

Nuke har en mycket tydlig designfilosofi: **allt är ett mål**.

I traditionella skript är det lätt att sluta med hundratals rader av linjär exekveringslogik trasslade ihop. I Nuke bryter vi byggprocessen till oberoende `Target`s. Varje mål gör exakt en sak, till exempel:

- `Clean`: rensa utdatakatalogen
- `Restore`: återställ beroendepaket
- `Compile`: kompilera koden
- `Test`: kör enhetstester

Denna design överensstämmer mycket väl med principen om ett ansvar. Liksom byggstenar kan dessa mål kombineras fritt. Ännu viktigare, Nuke tillåter oss att definiera beroenden mellan mål. Till exempel om du vill springa `Test`, kommer systemet automatiskt att kontrollera om `Compile` har redan avrättats. Om du vill `Compile`, då `Restore` kommer naturligtvis först.

Denna beroendegraf gör logiken mycket tydligare och förbättrar exekveringseffektiviteten avsevärt, eftersom Nuke automatiskt kan analysera den optimala exekveringsvägen.

### 2. Typsäkerhet: Adjö till skrivfelens mardröm

Alla som har skrivit byggskript i Python har förmodligen hamnat i den här besvärliga situationen: skriptet körs i fem minuter och misslyckas sedan pga. `Confi.guration` stavades fel eller för att en sträng skickades där en numerisk parameter förväntades.

Den största fördelen med att skriva byggskript i C# är **typsäkerhet**. Det betyder:

- **Kompileringstidskontroller**: IDE kan berätta för dig vad som är fel medan du skriver istället för att få dig att vänta till körning.
- **Säker refactoring**: om du vill byta namn på en variabel eller metod kan IDE hantera det i ett steg utan att tvinga dig att göra en riskabel global sökning och ersättning.
- **Intelligent komplettering**: kraftfull IntelliSense kan komplettera koden åt dig automatiskt, så du behöver inte fortsätta bläddra i dokumentationen för att komma ihåg obskyra API:er.

### 3. Stöd över plattformar: En konsekvent byggupplevelse

Förr skrev vi `.bat` på Windows och `.sh` på Linux, och sedan kanske lagt till ett Python-skript för att överbrygga klyftan. Nu, varhelst .NET Core, eller .NET 5+ idag, kan köras, kan Nuke också köras.

Det betyder att teammedlemmar kan använda Windows, Linux eller macOS, och oavsett om de föredrar Visual Studio, VS Code eller Rider, kör de alla exakt samma logik. Detta minskar avsevärt problem som orsakas av miljöskillnader, särskilt det klassiska problemet "det fungerar på min maskin".

### 4. Parametrar och konfigurationshantering

Nuke tillhandahåller en mycket elegant parameteranalysmekanism. Du behöver inte analysera manuellt `string[] args`. Du definierar helt enkelt en egenskap och kommenterar den med `[Parameter]`, och Nuke hanterar automatiskt mappningen från kommandoradsargument och konfigurationsfiler.

Till exempel kan vi definiera byggkonfigurationen mycket enkelt:

```csharp
[Parameter("Configuration to build - Default is 'Debug'")]
readonly Configuration BuildConfiguration = IsLocalBuild ? Configuration.Debug : Configuration.Release;

Target Compile => _ => _
    .DependsOn(Restore)
    .Executes(() =>
    {
        // Use BuildConfiguration here; it is type-safe
        DotNetBuild(s => s
            .SetConfiguration(BuildConfiguration)
            .SetProjectFile(SolutionFile));
    });
```

Denna stil är både intuitiv och motståndskraftig mot misstag.

## Praktisk guide: Hur man omsätter det i praktiken i ett projekt

Bara teorin tar oss ingen vart. Låt oss titta på hur vi faktiskt tillämpade detta tillvägagångssätt i HagiCode-projektet.

### 1. Planera projektstrukturen

Vi ville inte bygga skript som belamrade projektroten, och vi ville inte heller ha en katalogstruktur så djup att den liknade vissa Java-projekt. Så vi placerade alla Nuke-relaterade byggfiler i en singel `nukeBuild/` katalog.

Fördelarna är enkla:

- Projektroten förblir ren.
- Bygglogik förblir sammanhållen och lättare att hantera.
- Nya teammedlemmar kan omedelbart se, "Det är här den byggrelaterade logiken bor."

### 2. Designa en tydlig målberoendekedja

När vi utformade Targets följde vi en princip: **atomicitet plus beroendeflöde**.

Varje mål ska vara tillräckligt litet för att göra exakt en sak. Till exempel, `Clean` ska bara ta bort filer och ska inte tyst hantera förpackningar på sidan.

Ett rekommenderat beroendeflöde ser ungefär ut så här:

`Clean` -> `Restore` -> `Compile` -> `Test` -> `Pack`

Naturligtvis är detta inte absolut. Till exempel, om du bara vill köra tester och inte vill paketera något, låter Nuke dig köra `nuke Test` direkt, och den kommer automatiskt att hantera det som krävs `Restore` och `Compile` steg först.

### 3. Robust felhantering och loggning

Vad är det mest frustrerande med att bygga skript? Otydliga felmeddelanden. Ett misslyckat bygge som bara loggar "Error: 1" är tillräckligt för att göra vem som helst galen.

I Nuke, eftersom vi direkt kan använda C# undantagshantering, kan vi fånga och rapportera fel med mycket större precision.

```csharp
Target Publish => _ => _
    .DependsOn(Test)
    .Executes(() =>
    {
        try 
        {
            // Try publishing to NuGet
            DotNetNuGetPush(s => s
                .SetTargetPath(ArtifactPath)
                .SetSource("https://api.nuget.org/v3/index.json")
                .SetApiKey(ApiKey));
        }
        catch (Exception ex)
        {
            Log.Error($"Publishing failed. Team, please check whether the key is correct: {ex.Message}");
            throw; // Make sure the build process exits with a non-zero exit code
        }
    });
```

### 4. Använd tester för att skydda kvaliteten

Byggskript är också kod, vilket betyder att de också behöver tester. Nuke tillåter oss att skriva tester för byggarbetsflöden så att vi inte av misstag bryter befintliga releaseprocesser när vi ändrar bygglogiken. Detta är särskilt viktigt i pipelines för kontinuerlig integration.

## Sammanfattning

Genom att introducera Nuke blev HagiCodes byggprocess smidigare än någonsin tidigare. Detta var inte bara ett verktygsersättning. Det var en uppgradering av det tekniska tankesättet.

**Vad fick vi?**

- **Underhållbarhet**: konfiguration som kod, tydligare logik och en snabbare introduktionsväg för nya bidragsgivare.
- **Stabilitet**: stark skrivning eliminerar mer än 90 % av lågnivåfel.
- **Konsistens**: en enhetlig plattformsoberoende upplevelse som tar bort miljöavvikelser.

Om att skriva byggmanus brukade kännas som att "känna dig igenom mörkret", så känns det som att använda Nuke som "att gå på natten med ljusen på." Om du är trött på att underhålla svårfelsökta skript, överväg att flytta din bygglogik till C#-världen. Du kanske upptäcker att konstruktioner också kan vara eleganta.
