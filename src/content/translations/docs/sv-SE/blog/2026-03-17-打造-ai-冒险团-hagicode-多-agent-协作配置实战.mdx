## Bygga ett AI-äventyrsfest: HagiCode Multi-Agent Collaboration Configuration i praktiken

> I modern mjukvaruutveckling räcker det inte längre med en enda AI-agent för att möta komplexa krav. Hur kan flera AI-assistenter från olika företag samarbeta inom samma projekt? Den här artikeln delar samarbetskonfigurationen för flera agenter som HagiCode-projektet utvecklade genom praktik i verkligheten.

:::not[Time Context Note]
Den här artikeln registrerar konfigurationen av flera agenter **som släpptes 2026-03-17**. För den aktuella långtidsstödda omfattningen, se [AI Agent CLI-installation](/sv-SE/related-software-installation/ai-agent-cli/) och den [Installationsguide](/sv-SE/installation); `IFlowCli` behålls nu endast som ett äldre-kompatibelt värde och är inte längre en del av den aktivt stödda omfattningen.
:::

## Bakgrund

Många utvecklare har förmodligen haft denna erfarenhet: efter att ha introducerat en AI-assistent i ett projekt, förbättras produktiviteten verkligen. Men när kraven blir mer och mer komplexa, börjar en AI-agent kännas otillräcklig. Du vill att den ska hantera kodgranskning, dokumentationsgenerering, enhetstestning och andra uppgifter samtidigt, men resultatet blir ofta att det inte kan hålla allt balanserat och utdatakvaliteten blir inkonsekvent.

Vad som är ännu mer frustrerande är att när du väl försöker ta in flera AI-assistenter blir problemet mer komplicerat. Varje agent har sin egen konfigurationsmetod, API-gränssnitt och exekveringslogik, och de kan till och med komma i konflikt med varandra. Det är som ett idrottslag där alla spelare är talangfulla, men ingen vet hur man samarbetar, så matchen blir en enda röra.

HagiCode-projektet stötte på samma utmaning under utvecklingen. Som ett komplext projekt som involverade en frontend VSCode-tillägg, backend AI-tjänster och en plattformsoberoende skrivbordsklient, behövde vi koppla ihop flera AI-assistenter från olika företag samtidigt: Claude Code, Codex, CodeBuddy, iFlow och mer. Hur man låter dem samexistera harmoniskt i samma projekt och få ut det mesta av deras styrkor blev ett nyckelproblem vi var tvungna att lösa.

Bara det skulle redan vara tillräckligt med besvär. När allt kommer omkring, vem vill ta itu med ett gäng kämpande AI:er varje dag?

Tillvägagångssättet som delas i den här artikeln är samarbetskonfigurationen med flera agenter som vi utvecklade i HagiCode-projektet genom verkligt försök och fel och upprepad optimering. Om du också kämpar med flera AI-assistenter som arbetar tillsammans, kan den här artikeln ge dig lite inspiration. Kanske. Varje projekt är trots allt olika.

## Om HagiCode

[HagiCode](https://hagicode.com) är ett AI-kodningsassistentprojekt som använder en "äventyrsfest"-modell där flera AI-motorer arbetar tillsammans. Projektförråd: [github.com/HagiCode-org/site](https://github.com/HagiCode-org/site).

Multi-Agent-konfigurationsmetoden som delas i den här artikeln är en av kärnteknologierna som gör att HagiCode kan upprätthålla effektiv utveckling i komplexa projekt. Det finns inget speciellt magiskt med det; det förvandlar helt enkelt en grupp AI:er till en äventyrsfest som faktiskt kan koordinera.

## HagiCodes Multi-Agent Architecture Design

### Från "Going Solo" till "Team Collaboration"

I början av HagiCode-projektet försökte vi också använda en enda AI-agent för att hantera varje uppgift. Vi upptäckte snart en tydlig flaskhals i det förhållningssättet: olika uppgifter kräver olika styrkor. Vissa uppgifter behöver bättre kontextuell förståelse, medan andra behöver mer exakta kodmodifieringsmöjligheter. One Agent har svårt att briljera med allt.

Det fick oss att inse att flera agenter var tvungna att arbeta tillsammans. Men problemet var detta: hur låter man AI-produkter från olika företag samexistera fredligt i samma projekt? Vi behövde lösa flera kärnfrågor:

1. **Komplexitet för konfigurationshantering**: varje agent har olika konfigurationsmetoder, API-gränssnitt och exekveringslägen
2. **Enhetligt kommunikationsprotokoll**: vi behöver ett standardiserat sätt för olika agenter att utbyta data
3. **Uppgiftskoordinering och arbetsfördelning**: hur fördelar vi arbete på ett rimligt sätt så att varje agent kan spela efter sina styrkor

Med dessa frågor i åtanke började vi designa HagiCodes multi-Agent-arkitektur. Det var faktiskt inte så komplicerat; vi var bara tvungna att tänka igenom det tydligt.

### Övergripande arkitektur i ett ögonkast

Efter flera iterationer är det här arkitekturen vi satte oss för:

```
┌─────────────────────────────────────────────────────────────────┐
│                    AIProviderFactory                             │
│  (Factory pattern for unified management of all AI Providers)    │
├─────────────────────────────────────────────────────────────────┤
│  ClaudeCodeCli  │  CodexCli  │  CodebuddyCli  │  IFlowCli    │
│  (Anthropic)   │  (OpenAI)  │  (Zhipu GLM)    │  (Zhipu)     │
└─────────────────────────────────────────────────────────────────┘
```

Kärnidén är att låta olika AI-agenter hanteras av samma uppsättning kod genom ett enhetligt leverantörsgränssnitt. Samtidigt används fabriksmönstret för att dynamiskt skapa och konfigurera dessa leverantörer, vilket säkerställer skalbarhet och flexibilitet över hela systemet.

Det är som arbetsfördelning i vardagen. Alla har sin egen roll; här gjorde vi helt enkelt den idén till kodarkitektur.

### Agenttyper och ansvarsfördelning

Baserat på HagiCodes verkliga erfarenhet tilldelade vi olika ansvarsområden till varje agent:

| Agent | Leverantör | Modell | Primär användning |
|-------|----------|-------|-------------|
| ClaudeCodeCli | Antropisk | glm-5-turbo | Generera tekniska lösningar och förslag |
| CodexCli | OpenAI/Zed | gpt-5.4 | Utför exakta kodändringar |
| CodebuddyCli | Zhipu | glm-4,7 | Förfina förslagsbeskrivningar och dokumentation |
| IFlowCli | Zhipu | glm-4,7 | Arkivförslag och historiska dokument |
| OpenCodeCli | - | - | Allmän kodredigering |
| GitHubCopilot | Microsoft | - | Assisterad programmering och kodkomplettering |

Logiken bakom denna arbetsfördelning är enkel: varje agent har sitt eget styrkeområde. Claude Code presterar bra på att förstå och analysera komplexa krav, så den hanterar tidig lösningsdesign. Codex är mer exakt vid modifiering av kod, så det är bättre lämpat för konkret implementeringsarbete. CodeBuddy erbjuder hög kostnadsprestanda, vilket gör den idealisk för att förfina förslagstext och dokumentation.

När allt kommer omkring är rätt verktyg för rätt jobb det bästa valet. Det finns många vägar till Rom; vissa är helt enkelt lättare att gå än andra.

## Kärnkonfigurationsmekanismer

### Unified Provider Interface Design

För att hantera olika AI-agenter på ett enhetligt sätt måste vi först definiera ett gemensamt gränssnitt. I HagiCode ser det gränssnittet ut så här:

```csharp
public interface IAIProvider
{
    // Unified Provider interface
    Task<IAIProvider?> GetProviderAsync(AIProviderType providerType);
    Task<IAIProvider?> GetProviderAsync(string providerName, CancellationToken cancellationToken);
}
```

Gränssnittet ser enkelt ut, men det är grunden för hela multi-Agent-systemet. Med ett enhetligt gränssnitt kan vi kalla AI-produkter från olika företag på samma sätt oavsett vilket företag som står bakom dem.

Det här handlar egentligen bara om att göra komplexa saker enkla. Enkelt är vackert, trots allt.

### Implementering av leverantörsfabriksmönster

När gränssnittet är förenat är nästa fråga hur man skapar dessa Provider-instanser. HagiCode använder fabriksmönstret:

```csharp
private IAIProvider? CreateProvider(AIProviderType providerType, ProviderConfiguration config)
{
    return providerType switch
    {
        AIProviderType.ClaudeCodeCli =>
            ActivatorUtilities.CreateInstance<ClaudeCodeCliProvider>(_serviceProvider, Options.Create(config)),
        AIProviderType.CodebuddyCli =>
            ActivatorUtilities.CreateInstance<CodebuddyCliProvider>(_serviceProvider, Options.Create(config)),
        AIProviderType.CodexCli =>
            ActivatorUtilities.CreateInstance<CodexCliProvider>(_serviceProvider, Options.Create(config)),
        AIProviderType.IFlowCli =>
            ActivatorUtilities.CreateInstance<IFlowCliProvider>(_serviceProvider, Options.Create(config)),
        _ => null
    };
}
```

Detta använder beroendeinjektion genom `ActivatorUtilities.CreateInstance`, som dynamiskt kan skapa Provider-instanser under körning samtidigt som beroenden automatiskt injiceras. Fördelen med denna design är att när en ny agenttyp läggs till behöver du bara lägga till motsvarande leverantörsklass och sedan lägga till en till `case` gren i fabriksmetoden. Det finns inget behov av att modifiera den befintliga koden alls.

Det är skäl nog. Vem vill skriva om en hög med gammal kod varje gång en ny funktion läggs till?

### Dynamisk konfigurationsupplösning

För att göra konfigurationen mer flexibel implementerade vi också en typkartläggningsmekanism:

```csharp
public static AIProviderTypeExtensions
{
    private static readonly Dictionary<string, AIProviderType> _typeMap = new(
        StringComparer.OrdinalIgnoreCase)
    {
        ["ClaudeCodeCli"] = AIProviderType.ClaudeCodeCli,
        ["CodebuddyCli"] = AIProviderType.CodebuddyCli,
        ["CodexCli"] = AIProviderType.CodexCli,
        ["IFlowCli"] = AIProviderType.IFlowCli,
        // ...more type mappings
    };
}
```

Syftet med den här mappningstabellen är att konvertera leverantörsnamn i strängform till enumtyper. Detta tillåter konfigurationsfiler att använda intuitiva strängnamn, medan den interna koden använder typsäkra enums för bearbetning.

Konfigurationen bör vara så intuitiv som möjligt. Ingen vill memorera en hög med komplicerade kodnamn.

### Exempel på konfigurationsfil

I praktiken kan allt konfigureras i `appsettings.json`:

```yaml
AI:
  Providers:
    Providers:
      ClaudeCodeCli:
        Enabled: true
        Model: glm-5-turbo
        WorkingDirectory: /path/to/project
      CodebuddyCli:
        Enabled: true
        Model: glm-4.7
      CodexCli:
        Enabled: true
        Model: gpt-5.4
      IFlowCli:
        Enabled: true
        Model: glm-4.7
```

Varje leverantör kan oberoende konfigurera parametrar som aktivering, modellversion och arbetskatalog. Denna design bevarar flexibiliteten samtidigt som den är enkel att hantera och underhålla.

Konfigurationsfiler är lite som livets alternativ: du kan välja att aktivera eller inaktivera vissa saker. Den enda skillnaden är att kodval är lättare att ångra senare.

## Äventyrsfestuppgiftsflöde

### Uppgiftskonsten

Med den enhetliga tekniska arkitekturen på plats är nästa steg att få flera agenter att arbeta tillsammans. HagiCode designade en uppgiftsflödesmekanism så att olika agenter kan hantera olika stadier av arbetet:

```
Proposal creation (user)
    │
    ▼
[Claude Code] ──generate proposal──▶ Proposal document
    │                               │
    │                               ▼
    │                      [Codebuddy] ──refine description──▶ Refined proposal
    │                               │
    │                               ▼
    │                      [Codex] ──execute changes──▶ Code changes
    │                               │
    │                               ▼
    └──────────────────────▶ [iFlow] ──archive──▶ Historical records
```

Fördelen med denna arbetsfördelning är att varje agent bara behöver fokusera på de uppgifter den gör bäst, snarare än att försöka göra allt. Claude Code är ansvarig för att ta fram förslag från grunden. Codebuddy gör förslagsbeskrivningarna tydligare. Codex förvandlar förslag till faktiska kodändringar. iFlow arkiverar och bevarar dessa ändringar.

Det här är egentligen bara lagarbete, ungefär som i vardagen. Alla har sin egen roll, och bara tillsammans kan något stort bli gjort. Den enda skillnaden är att teammedlemmarna här råkar vara AI:er.

### Viktiga praktiska takeaways

I själva verket sammanfattade vi följande lärdomar:

**1. Agenturvalsstrategi spelar roll**

Uppgifter bör inte tilldelas slentrianmässigt; de bör matchas till varje agents styrkor:
- Förslagsgenerering: använd Claude Code, eftersom den har en starkare kontextuell förståelse
- Kodexekvering: använd Codex, eftersom det är mer exakt för kodändring
- Förfining av förslag: använd Codebuddy, eftersom det erbjuder en stark kostnadsprestanda
- Arkivlagring: använd iFlow, eftersom det är stabilt och pålitligt

Att sätta rätt person på rätt uppgift är trots allt en tidlös princip.

**2. Konfigurationsisolering säkerställer stabilitet**

Varje agents konfiguration hanteras oberoende, stöder miljövariable åsidosättningar och använder separata arbetskataloger. Som ett resultat påverkar ett konfigurationsfel i en agent inte de andra.

Det här är som personliga gränser i livet. Alla behöver sitt eget utrymme; icke-interferens gör harmonisk samexistens möjlig.

**3. Felhanteringsmekanism**

Ett fel i en enskild agent bör inte påverka det övergripande arbetsflödet. Vi implementerade en reservstrategi: när en agent misslyckas kan systemet automatiskt byta till en backupplan eller hoppa över det steget och fortsätta med senare uppgifter. Samtidigt gör komplett loggning felsökning enklare efteråt.

Ingen kan garantera att fel aldrig kommer att inträffa. Nyckeln är hur du hanterar dem. Livet fungerar ungefär på samma sätt.

**4. Övervakning och observerbarhet**

Genom ACP-protokollet (vårt anpassade kommunikationsprotokoll baserat på JSON-RPC 2.0) kan vi spåra exekveringsstatusen för varje agent. Sessionsisolering säkerställer samtidighetssäkerhet, medan dynamisk cachelagring förbättrar prestandan.

De saker du inte kan se är ofta de som är mest benägna att gå fel. Viss sikt är alltid bättre än att flyga blind.

## Verkliga resultat och fördelar

Efter att ha antagit denna multi-Agent-samarbetskonfiguration förbättrades HagiCode-projektets utvecklingseffektivitet avsevärt. Specifikt:

1. **Uppgiftshanteringskapaciteten fördubblades**: tidigare var en agent tvungen att hantera många typer av uppgifter samtidigt; nu kan uppgifter bearbetas parallellt och genomströmningen har ökat dramatiskt
2. **Stabilare utskriftskvalitet**: varje agent fokuserar bara på det den är bäst på, så både konsekvens och kvalitet förbättras
3. **Lägre underhållskostnad**: enhetliga gränssnitt och konfigurationshantering gör hela systemet lättare att underhålla och utöka
4. **Det är enkelt att lägga till nya agenter**: för att integrera en ny AI-produkt behöver du bara implementera gränssnittet och lägga till konfiguration, utan att ändra kärnlogiken

Detta tillvägagångssätt löste inte bara HagiCodes egna problem, utan bevisade också att samarbete med flera agenter är ett hållbart arkitektoniskt val.

Vinsterna var ganska märkbara. Processen var bara lite av en krångel.

## Slutsats

Den här artikeln delade HagiCode-projektets praktiska erfarenhet av samarbetskonfiguration med flera agenter. De viktigaste takeaways inkluderar:

1. **Standardiserade gränssnitt**: `IAIProvider` förenar olika agenters beteende, vilket gör att koden ignorerar vilket företags produkt som finns under
2. **Fabriksmönster**: `ActivatorUtilities.CreateInstance` skapar dynamiskt Provider-instanser, stödjer runtime-konfiguration och beroendeinjektion
3. **Protokollförening**: ACP-protokollet tillhandahåller standardiserad kommunikation mellan agenter genom en dubbelriktad mekanism baserad på JSON-RPC 2.0
4. **Uppgiftsdirigering**: fördela arbete på ett rimligt sätt mellan olika agenter så att var och en kan spela efter sina styrkor, istället för att förvänta sig att en agent ska göra allt

Den här designen löser inte bara problemet med att "flera agenter slåss mot varandra", utan använder också uppgiftsflödesmekanismen för äventyrsfest för att göra utvecklingsprocessen mer automatiserad och specialiserad.

Om du också funderar på att introducera flera AI-assistenter hoppas jag att den här artikeln ger dig några användbara referenspunkter. Naturligtvis är varje projekt olika, och det specifika tillvägagångssättet måste fortfarande anpassas till den faktiska situationen. Det finns ingen lösning som passar alla; den bästa lösningen är den som passar dig.

Vackra saker eller människor behöver inte vara besatta. Så länge de förblir vackra räcker det att bara uppskatta att skönheten. Tekniska lösningar är desamma: den som passar dig är den bästa...
