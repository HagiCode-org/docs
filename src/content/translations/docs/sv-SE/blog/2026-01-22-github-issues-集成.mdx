## Att bygga en GitHub-problemintegrering från grunden: HagiCodes frontend-direktanslutningspraxis

> Den här artikeln dokumenterar hela processen för att integrera GitHub-problem i HagiCode-plattformen. Vi kommer att utforska hur man använder en "frontend-direktanslutning + minimal backend"-arkitektur för att uppnå säker OAuth-autentisering och effektiv problemsynkronisering samtidigt som backend-delen är lätt.

## Bakgrund: Varför integrera GitHub?

Som en AI-stödd utvecklingsplattform ligger HagiCodes kärnvärde i att koppla ihop idéer med implementering. Men vid faktisk användning upptäckte vi att efter att användare har slutfört ett förslag i HagiCode, måste de ofta manuellt kopiera innehållet till GitHub Issues för projektspårning.

Detta skapar flera uppenbara smärtpunkter:

1.  **Fragmenterat arbetsflöde**: Användare måste växla fram och tillbaka mellan två system. Upplevelsen är inte smidig och viktig information kan lätt gå förlorad under kopiering och inklistring.
2.  **Obekvämt samarbete**: Andra teammedlemmar är vana vid att kontrollera uppgifter på GitHub och kan inte direkt se förslagets framsteg i HagiCode.
3.  **Upprepad manuellt arbete**: Varje gång ett förslag uppdateras måste någon manuellt uppdatera motsvarande problem på GitHub, vilket tillför onödiga underhållskostnader.

För att lösa detta problem bestämde vi oss för att introducera funktionen **GitHub Issues Integration**, som kopplar HagiCode-sessioner med GitHub-arkiv för att möjliggöra "en-klickssynkronisering."

## Tekniskt val: Frontend Direct Connection vs Backend Proxy

När vi utformade integrationsmetoden hade vi två alternativ framför oss: den traditionella "backend proxy-modellen" och den mer aggressiva "frontend-direktanslutningsmodellen."

### Jämförelse av lösningar

I den traditionella **backend-proxymodellen** måste varje begäran från frontend först gå via vår backend, som sedan anropar GitHub API. Detta centraliserar logiken, men det lägger också en betydande börda på backend:

1.  **Uppblåst backend**: Vi skulle behöva skriva ett dedikerat GitHub API-klientomslag och även hantera den komplexa OAuth-tillståndsmaskinen.
2.  **Tokenrisk**: Användarens GitHub-token måste lagras i backend-databasen. Även med kryptering ökar detta fortfarande säkerhetsytan.
3.  **Utvecklingskostnad**: Vi skulle behöva databasmigreringar för att lagra tokens och en extra synkroniseringstjänst att underhålla.

**Modellen för frontend direktanslutning** är mycket lättare. I detta tillvägagångssätt använder vi backend endast för det mest känsliga "hemliga utbytet"-steget (OAuth-återuppringningen). Efter att ha erhållit token lagrar vi den direkt i webbläsarens `localStorage`. Senare operationer som att skapa problem och uppdatera kommentarer skickas direkt från frontend till GitHub över HTTP.

| Jämförelsedimension | Backend-proxymodell | Frontend direktanslutningsmodell |
| :--- | :--- | :--- |
| **Backend komplexitet** | Kräver en fullständig OAuth-tjänst och GitHub API-klient | Behöver bara en OAuth-återuppringningsslutpunkt |
| **Tokenhantering** | Måste vara krypterad och lagrad i databasen, med risk för läckage | Lagras i webbläsaren och endast synlig för användaren |
| **Implementeringskostnad** | Kräver databasmigrering och utveckling av flera tjänster | Främst frontend-arbete |
| **Användarupplevelse** | Centraliserad logik, men serverfördröjningen kan vara något högre | Extremt snabb respons med direkt GitHub-interaktion |

Eftersom vi ville ha snabb integration och minimala backend-ändringar, valde vi slutligen "frontend-direktanslutningsmodellen"**. Det är som att ge webbläsaren ett "tillfälligt pass". När den väl har fått passet kan webbläsaren hantera saker på GitHub själv utan att be backend-administratören om godkännande varje gång.

## Kärndesign: Dataflöde och säkerhet

Efter att ha bestämt oss för arkitekturen behövde vi designa det specifika dataflödet. Kärnan i synkroniseringsprocessen är hur man skaffar token säkert och använder det effektivt.

### Övergripande arkitekturdiagram

Hela systemet kan abstraheras i tre roller: webbläsaren (frontend), HagiCode backend och GitHub.

```text
+--------------+        +--------------+        +--------------+
| Frontend     |        |   Backend    |        |   GitHub     |
| React        |        |   ASP.NET    |        |   REST API   |
|              |        |              |        |              |
|  +--------+  |        |              |        |              |
|  | OAuth  |--+--------> /callback    |        |              |
|  | Flow   |  |        |              |        |              |
|  +--------+  |        |              |        |              |
|              |        |              |        |              |
|  +--------+  |        |  +--------+  |        |  +--------+  |
|  | GitHub |  +------------> Session |  +----------> Issues |  |
|  | API    |  |        |  |Metadata|  |        |  |        |  |
|  | Direct |  |        |  +--------+  |        |  +--------+  |
|  +--------+  |        |              |        |              |
+--------------+        +--------------+        +--------------+
```

**Nyckelpunkten är**: endast ett litet steg i OAuth (byte av koden mot en token) behöver gå genom backend. Därefter hanteras de tunga lyften (skapande problem) direkt mellan frontend och GitHub.

### Synkroniseringsdataflöde i detalj

När användaren klickar på knappen "Synkronisera med GitHub" i HagiCode-gränssnittet, sker en serie komplexa åtgärder:

```text
User clicks "Sync to GitHub"
         │
         ▼
1. Frontend checks localStorage for the GitHub token
         │
         ▼
2. Format issue content (convert the Proposal into Markdown)
         │
         ▼
3. Frontend directly calls the GitHub API to create/update the issue
         │
         ▼
4. Call the HagiCode backend API to update Session.metadata (store issue URL and other info)
         │
         ▼
5. Backend broadcasts the SessionUpdated event via SignalR
         │
         ▼
6. Frontend receives the event and updates the UI to show the "Synced" state
```

### Säkerhetsdesign

Säkerhet är alltid högsta prioritet när man integrerar tredjepartstjänster. Vi gjorde följande överväganden:

1.  **Försvara sig mot CSRF-attacker**: Skapa en slumpmässig `state` parametern under OAuth-omdirigeringen och lagra den i `sessionStorage`. Validera strikt tillståndet i återuppringningen för att förhindra falska förfrågningar.
2.  **Isolerad tokenlagring**: Token lagras endast i webbläsarens `localStorage`. Genom att använda Same-Origin Policy kan endast HagiCode-skript läsa den, vilket undviker risken för en databasläcka på serversidan som påverkar användarna.
3.  **Felgränser**: Vi utformade dedikerad hantering för vanliga GitHub API-fel (som 401 utgången token, 422 valideringsfel och 429 hastighetsbegränsning), så att användarna får vänlig feedback.

## I praktiken: Implementeringsdetaljer i kod

Teorin går bara så långt. Låt oss titta på hur koden faktiskt fungerar.

### 1. Minimala Backend-ändringar

Backend behöver bara göra två saker: lagra synkroniseringsinformation och hantera OAuth-återuppringningen.

### 2. Frontend OAuth-flöde

Detta är ingångspunkten för anslutningen. Vi använder standardauktoriseringskodflödet.

```typescript
// src/HagiCode.Client/src/services/githubOAuth.ts

// Generate the authorization URL and redirect
export async function generateAuthUrl(): Promise<string> {
  const state = generateRandomString(); // Generate a random string for CSRF protection
  sessionStorage.setItem('hagicode_github_state', state);
  
  const params = new URLSearchParams({
    client_id: clientId,
    redirect_uri: window.location.origin + '/settings?tab=github&oauth=callback',
    scope: ['repo', 'public_repo'].join(' '),
    state: state,
  });
  
  return `https://github.com/login/oauth/authorize?${params.toString()}`;
}

// Handle the code-to-token exchange on the callback page
export async function exchangeCodeForToken(code: string, state: string): Promise<GitHubToken> {
  // 1. Validate state to prevent CSRF
  const savedState = sessionStorage.getItem('hagicode_github_state');
  if (state !== savedState) throw new Error('Invalid state parameter');

  // 2. Call the backend API to exchange the token
  // Note: this must go through the backend because ClientSecret cannot be exposed to the frontend
  const response = await fetch('/api/GitHubOAuth/callback', {
    method: 'POST',
    headers: { 'Content-Type': 'application/json' },
    body: JSON.stringify({ code, state, redirectUri: window.location.origin + '/settings?tab=github&oauth=callback' }),
  });

  if (!response.ok) throw new Error('Failed to exchange token');
  
  const token = await response.json();
  
  // 3. Save into LocalStorage
  saveToken(token);
  return token;
}
```

### 3. GitHub API Client Wrapper

När vi väl har token behöver vi ett gediget verktyg för att anropa GitHub API.

```typescript
// src/HagiCode.Client/src/services/githubApiClient.ts

const GITHUB_API_BASE = 'https://api.github.com';

// Core request wrapper
async function githubApi<T>(endpoint: string, options: RequestInit = {}): Promise<T> {
  const token = localStorage.getItem('gh_token');
  if (!token) throw new Error('Not connected to GitHub');
  
  const response = await fetch(`${GITHUB_API_BASE}${endpoint}`, {
    ...options,
    headers: {
      ...options.headers,
      Authorization: `Bearer ${token}`,
      Accept: 'application/vnd.github.v3+json', // Specify the API version
    },
  });
  
  // Error handling logic
  if (!response.ok) {
    if (response.status === 401) throw new Error('GitHub token expired, please reconnect');
    if (response.status === 403) throw new Error('No permission to access this repository or rate limit exceeded');
    if (response.status === 422) throw new Error('Issue validation failed, the title may be duplicated');
    throw new Error(`GitHub API Error: ${response.statusText}`);
  }
  
  return response.json();
}

// Create issue
export async function createIssue(owner: string, repo: string, data: { title: string, body: string, labels: string[] }) {
  return githubApi(`/repos/${owner}/${repo}/issues`, {
    method: 'POST',
    body: JSON.stringify(data),
  });
}
```

### 4. Innehållsformatering och synkronisering

Det sista steget är att konvertera HagiCode-sessionsdata till formatet för ett GitHub-problem. Det är lite som översättningsarbete.

```typescript
// Convert a Session object into a Markdown string
function formatIssueForSession(session: Session): string {
  let content = `# ${session.title}\n\n`;
  content += `**> HagiCode Session:** #${session.code}\n`;
  content += `**> Status:** ${session.status}\n\n`;
  content += `## Description\n\n${session.description || 'No description provided.'}\n\n`;
  
  // If this is a Proposal session, add extra fields
  if (session.type === 'proposal') {
    content += `## Chief Complaint\n\n${session.chiefComplaint || ''}\n\n`;
    // Add a deep link so users can jump back from GitHub to HagiCode
    content += `---\n\n**[View in HagiCode](hagicode://sessions/${session.id})**\n`;
  }
  
  return content;
}

// Main logic when clicking the sync button
const handleSync = async (session: Session) => {
  try {
    const repoInfo = parseRepositoryFromUrl(session.repoUrl); // Parse the repository URL
    if (!repoInfo) throw new Error('Invalid repository URL');

    toast.loading('Syncing to GitHub...');
    
    // 1. Format content
    const issueBody = formatIssueForSession(session);
    
    // 2. Call API
    const issue = await githubApiClient.createIssue(repoInfo.owner, repoInfo.repo, {
      title: `[HagiCode] ${session.title}`,
      body: issueBody,
      labels: ['hagicode', 'proposal', `status:${session.status}`],
    });
    
    // 3. Update Session Metadata (save the issue link)
    await SessionsService.patchApiSessionsSessionId(session.id, {
      metadata: {
        githubIssue: {
          owner: repoInfo.owner,
          repo: repoInfo.repo,
          issueNumber: issue.number,
          issueUrl: issue.html_url,
          syncedAt: new Date().toISOString(),
        }
      }
    });

    toast.success('Synced successfully!');
  } catch (err) {
    console.error(err);
    toast.error('Sync failed, please check your token or network');
  }
};
```

## Sammanfattning och Outlook

Med denna "frontend-direktanslutning"-metoden uppnådde vi sömlös GitHub Issues-integration med minsta möjliga backend-kod.

### Viktiga vinster
1.  **Hög utvecklingseffektivitet**: Backend-ändringar är minimala, främst ett extra databasfält och en enkel OAuth-återuppringningsslutpunkt. Det mesta av logiken är klar på frontend.
2.  **Stark säkerhet**: Token passerar inte genom serverdatabasen, vilket minskar risken för läckage.
3.  **Bra användarupplevelse**: Förfrågningar initieras direkt från frontend, så svarshastigheten är snabb och det finns inget behov av backend-vidarebefordran.

### Saker att se upp för
Det finns några fallgropar att tänka på under verklig implementering:
- **OAuth-appinställningar**: Kom ihåg att ange rätt `Authorization callback URL` i dina GitHub OAuth-appinställningar (vanligtvis `http://localhost:3000/settings?tab=github&oauth=callback`).
- **Prisgränser**: GitHub API begränsar oautentiserade förfrågningar ganska strikt, men med en token är kvoten vanligtvis tillräcklig (5000 förfrågningar/timme).
- **URL parsing**: Användare anger alla typer av repository URLs, så se till att ditt regex kan matcha `.git` suffix, SSH-format och liknande fall.

### Framtida förbättringar
Den nuvarande funktionen är fortfarande envägssynkronisering (HagiCode -> GitHub). I framtiden planerar vi att implementera tvåvägssynkronisering genom GitHub Webhooks. Till exempel, om ett problem stängs på GitHub, kan sessionstillståndet på HagiCode-sidan också uppdateras automatiskt. Det kommer att kräva att vi exponerar en webhook-slutpunkt på backend, vilket kommer att vara ett intressant nästa steg.
