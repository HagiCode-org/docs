## ImgBin CLI-verktygsdesign: HagiCodes tillvägagångssätt för Image Asset Management

> Den här artikeln förklarar hur man bygger en automatiserbar bildtillgångspipeline från grunden, och täcker CLI-verktygsdesign, en Provider Adapter-arkitektur och metadatahanteringsstrategier.

## Bakgrund

Ärligt talat, jag förväntade mig inte att imagetillgångsförvaltning skulle hålla oss trassliga så länge.

Under utvecklingen av HagiCode stötte vi på ett problem som såg enkelt ut på ytan men som var förvånansvärt besvärligt i praktiken: att generera och hantera bildtillgångar. På ett sätt var det som tonårens dramer – lugnt på utsidan, turbulent undertill.

Eftersom projektet samlade på sig mer dokumentation och marknadsföringsmaterial behövde vi ett stort antal stödjande bilder. Vissa måste vara AI-genererade, vissa måste väljas från ett befintligt tillgångsbibliotek och andra behövde AI-igenkänning plus automatisk märkning. Problemet var att allt detta länge hade hanterats genom spridda skript och manuella steg. Varje gång vi genererade en bild var vi tvungna att köra ett skript för hand, organisera metadata för hand och skapa miniatyrer för hand. Bara det var irriterande nog, men det större problemet var att allt var utspritt överallt. När vi ville hitta något kunde vi inte. När vi behövde återanvända något kunde vi inte.

Smärtpunkterna var konkreta:

1. **Ingen enhetlig ingångspunkt**: logiken för bildgenerering var spridd över olika skript, så batchkörning var i princip omöjlig.
2. **Metadata saknas**: genererade bilder hade ingen enhetlig `metadata.json`, vilket innebar ingen pålitlig sökbarhet eller spårbarhet.
3. **Hög kostnad för manuell organisation**: titlar och taggar måste sorteras ut en efter en för hand, vilket var ineffektivt.
4. **Ingen automatisering**: genererar visuella tillgångar automatiskt i en CI/CD-pipeline? Inte en chans.

Vi funderade på att bara lämna det ifred. Men projekten måste fortfarande gå framåt. Eftersom vi inte kunde undvika problemet tänkte vi att vi lika gärna kunde lösa det. Så vi bestämde oss för att uppgradera ImgBin från en uppsättning spridda skript till en bildtillgångspipeline som kan köras automatiskt. Vissa problem försvinner trots allt inte bara för att man tittar bort.

## Om HagiCode

Tillvägagångssättet som delas i den här artikeln kommer från vår praktiska erfarenhet av [HagiCode](https://hagicode.com) projekt. HagiCode är ett AI-kodningsassistentprojekt som samtidigt underhåller flera komponenter, inklusive en VSCode-tillägg, backend AI-tjänster och en plattformsoberoende skrivbordsklient. I en komplex, flerspråkig, plattformsoberoende miljö som denna, blir standardiserad bildtillgångshantering en nyckeldel för att förbättra utvecklingseffektiviteten.

Man kan säga att detta var en av de där små växtvärkarna på HagiCodes resa. Varje projekt har sådana ögonblick: ett mindre problem som ser obetydligt ut, men som ändå på något sätt lyckas ta upp halva dagen.

HagiCodes byggsystem är baserat på ekosystemet TypeScript + Node.js, så ImgBin antog naturligtvis samma tekniska stack för att hålla projektet tekniskt konsekvent. När du väl är van vid en stack känns det bara som ett onödigt problem att byta till något annat.

## Kärndesign

### Övergripande arkitektur

ImgBin använder en skiktad arkitektur som rent separerar CLI-kommandon, applikationstjänster, API-adaptrar från tredje part och infrastrukturlagret:

```
Component hierarchy
├── CLI Entry (cli.ts)              Global argument parsing, command routing
├── Commands (commands/*)           generate | batch | annotate | thumbnail
├── Application Services            job-runner | metadata | thumbnail | asset-writer
├── Provider Adapters               image-api-provider | vision-api-provider
└── Infrastructure Layer            config | logger | paths | schema
```

Fördelen med denna skiktade design är tydliga ansvarsgränser. Det gör också testning enklare eftersom externa beroenden kan hånas rent. I praktiken betyder det bara att varje lager gör sitt eget jobb utan att komma i vägen för de andra, så när något går sönder är det lättare att ta reda på varför.

### Katalogmodell med en tillgång

ImgBin använder en modell av "en tillgång, en katalog." Varje gång en bild genereras skapar den en struktur så här:

```
library/
└── 2026-03/
    └── orange-dashboard/
        ├── original.png      # Original image
        ├── thumbnail.webp    # 512x512 thumbnail
        └── metadata.json     # Structured metadata
```

Fördelarna med denna modell är:

1. **Självständig**: alla filer för en enskild tillgång finns i samma katalog, vilket gör migrering och säkerhetskopiering bekvämt.
2. **Spårbar**: `metadata.json` gör det möjligt att spåra generationstid, prompt, modell och andra detaljer.
3. **Utökningsbar**: om fler varianter behövs senare, till exempel miniatyrbilder i flera storlekar, kan vi helt enkelt lägga till nya filer i samma katalog.

Vackra saker behöver inte alltid ägas. Ibland räcker det med att de förblir vackra, och att man tyst kan uppskatta dem. Det kanske låter lite långt borta, men logiken håller fortfarande här: när bilderna hålls ihop är de trevligare att titta på och mycket lättare att hitta.

### Lagring av lager med metadata

`metadata.json` är kärnan i hela systemet. Den använder en lagerlagringsstrategi som separerar fält i tre kategorier:

```json
{
  "schemaVersion": 2,
  "assetId": "orange-dashboard",
  "slug": "orange-dashboard",
  "title": "Orange Dashboard",
  "tags": ["dashboard", "hero", "orange"],

  "source": { "type": "generated" },

  "paths": {
    "assetDir": "library/2026-03/orange-dashboard",
    "original": "original.png",
    "thumbnail": "thumbnail.webp"
  },

  "generated": {
    "prompt": "orange dashboard for docs hero",
    "provider": "azure-openai-image-api",
    "model": "gpt-image-1.5"
  },

  "recognized": {
    "title": "Orange Dashboard",
    "tags": ["dashboard", "ui", "orange"],
    "description": "A modern orange dashboard with charts and metrics"
  },

  "status": {
    "generation": "succeeded",
    "recognition": "succeeded",
    "thumbnail": "succeeded"
  },

  "timestamps": {
    "createdAt": "2026-03-11T04:01:19.570Z",
    "updatedAt": "2026-03-11T04:02:09.132Z"
  }
}
```

- **genererad**: registrerar den ursprungliga informationen från bildgenereringen, såsom uppmaning, leverantör och modell.
- **igenkänd**: lagrar AI-igenkänningsresultat, som automatiskt genererade titlar, taggar och beskrivningar.
- **manuell**: lagrar manuellt utvalda resultat. Data inom detta område har högsta prioritet och kommer inte att skrivas över av AI-igenkänning.

Denna skiktade strategi löser en av våra tidigare kärnkonflikter: när AI-igenkänning och manuell kuration inte överensstämmer, vilken ska vinna? Svaret är manuell inmatning. AI-igenkänning är till för att hjälpa, inte för att bestämma. Den frågan blev också tydligare med tiden – maskiner är fortfarande maskiner, och i slutändan behöver folk fortfarande ringa.

## Leverantörsadaptermönster

En annan central del av ImgBin är Provider Adapter-mönstret. Vi abstraherar externa API:er bakom ett enhetligt gränssnitt så att även om vi byter AI-tjänsteleverantör behöver vi inte ändra affärslogiken.

På ett sätt är det lite som relationer – det yttre kan förändras, men det som spelar roll är att den inre strukturen förblir densamma. När gränssnittet är fixat kan den interna implementeringen variera fritt.

### Bildgenereringsleverantör

```typescript
interface ImageGenerationProvider {
  // Generate an image and return its Buffer
  generate(options: GenerateOptions): Promise<Buffer>;

  // Get the list of supported models
  getSupportedModels(): Promise<string[]>;
}

interface GenerateOptions {
  prompt: string;
  model?: string;
  size?: '1024x1024' | '1792x1024' | '1024x1792';
  quality?: 'standard' | 'hd';
  format?: 'png' | 'webp' | 'jpeg';
}
```

### Leverantör av synigenkänning

```typescript
interface VisionRecognitionProvider {
  // Recognize image content and return structured metadata
  recognize(imageBuffer: Buffer): Promise<RecognitionResult>;

  // Get the list of supported models
  getSupportedModels(): Promise<string[]>;
}

interface RecognitionResult {
  title?: string;
  tags: string[];
  description?: string;
  confidence: number;
}
```

Fördelarna med denna gränssnittsdesign är:

1. **Testbar**: i enhetstester kan vi skicka in skenleverantörer istället för att göra riktiga externa API-anrop.
2. **Utökningsbar**: att lägga till en ny leverantör kräver bara implementering av gränssnittet; samtalskoden behöver inte ändras.
3. **Utbytbar**: produktion kan använda Azure OpenAI medan testning kan använda en lokal modell, med konfiguration som det enda som ändras.

Ibland känns projektarbete också så. På ytan ser det ut som att vi precis har bytt ett API, men den interna logiken förblir exakt densamma, och det gör det hela mycket mindre skrämmande.

## CLI Command Design

ImgBin tillhandahåller fyra kärnkommandon för att täcka olika användningsscenarier:

### generera: enbildsgenerering

```bash
# Simplest usage
imgbin generate --prompt "orange dashboard for docs hero"

# Generate a thumbnail and AI annotations at the same time
imgbin generate --prompt "orange dashboard" --annotate --thumbnail

# Specify an output directory
imgbin generate --prompt "orange dashboard" --output ./library
```

### batch: batchjobb

Batchjobb definieras genom YAML- eller JSON-manifestfiler, vilket gör dem lämpliga för CI/CD-arbetsflöden:

```yaml
# assets/jobs/launch.yaml
defaults:
  annotate: true
  thumbnail: true
  libraryRoot: ./library

jobs:
  - prompt: "orange dashboard hero"
    slug: orange-dashboard
    tags: [dashboard, hero, orange]

  - prompt: "pricing grid for docs"
    slug: pricing-grid
    tags: [pricing, grid, docs]
```

Kör kommandot:

```bash
imgbin batch assets/jobs/launch.yaml
```

Batchjobbdesignen stöder felisolering: objekt i manifestet bearbetas en efter en, och ett fel i en artikel påverkar inte de andra. Du kan också förhandsgranska jobbet med `--dry-run` utan att faktiskt utföra det.

Och det bästa är att den berättar exakt vad som lyckades och vad som misslyckades. Till skillnad från vissa saker i livet, där misslyckanden händer och du inte ens vet hur det hände.

### annotera: AI-anteckning

Kör AI-igenkänning på befintliga bilder för att automatiskt generera titlar, taggar och beskrivningar:

```bash
# Annotate a single image
imgbin annotate ./library/2026-03/orange-dashboard

# Annotate an entire directory in batch
imgbin annotate ./library/2026-03/
```

### thumbnail: thumbnail generation

Generera miniatyrer för befintliga bilder:

```bash
# Generate a thumbnail
imgbin thumbnail ./library/2026-03/orange-dashboard
```

## Batch Job Manifest Design

Manifestformatet för batchjobb stöder flexibel konfiguration. Standardvärden kan ställas in globalt och enskilda jobb kan åsidosätta dem:

```yaml
# Global defaults
defaults:
  annotate: true        # Enable AI annotation by default
  thumbnail: true       # Generate thumbnails by default
  libraryRoot: ./library
  model: gpt-image-1.5

jobs:
  # Minimal configuration: only provide a prompt
  - prompt: "first image"

  # Full configuration
  - prompt: "second image"
    slug: custom-slug
    tags: [tag1, tag2]
    annotate: false     # Do not run AI annotation for this job
    model: dall-e-3    # Use a different model for this job
```

När det körs, bearbetar ImgBin jobb en efter en. Resultatet av varje jobb skrivs till dess motsvarande `metadata.json`. Även om ett jobb misslyckas påverkas de andra opåverkade. När alla jobb är klara, skickar CLI en sammanfattande rapport:

```
✓ orange-dashboard (succeeded)
✓ pricing-grid (succeeded)
✗ hero-banner (failed: API rate limit exceeded)

2/3 succeeded, 1 failed
```

Vissa saker går inte att skynda på. Att ta dem en i taget är ofta den stadigare vägen. Kanske är det filosofin bakom batchjobb.

## Konfiguration av miljövariabel

ImgBin stöder flexibel konfiguration genom miljövariabler:

```bash
# ImgBin working directory
IMGBIN_WORKDIR=/path/to/imgbin

# Executable path (for invocation inside scripts)
IMGBIN_EXECUTABLE=/path/to/imgbin/dist/cli.js

# Asset library root
IMGBIN_LIBRARY_ROOT=./.imgbin-library

# Azure OpenAI configuration (if using the Azure provider)
AZURE_OPENAI_ENDPOINT=https://your-resource.openai.azure.com/
AZURE_OPENAI_API_KEY=***
AZURE_OPENAI_IMAGE_DEPLOYMENT=gpt-image-1
```

Konfiguration är en av de saker som kan kännas både viktiga och inte så viktiga på samma gång. I slutändan är det som känns bekvämt och passar ditt arbetsflöde oftast det rätta valet.

## Anmärkningar om genomförande

Under implementeringen sammanfattade vi några nyckelpunkter:

### Utformning av gränssnitt för leverantörer

Gränssnittsdefinitioner bör vara tydliga och fullständiga, inklusive indataparametrar, returvärden och felhantering. Det är också en bra idé att tillhandahålla både synkrona och asynkrona anropsstilar för olika scenarier.

Det är en liten bit av hårt förvärvad erfarenhet. När ett gränssnitt väl har ställts in vill ingen fortsätta att ändra det senare.

### Felhanteringsstrategi

När ett objekt misslyckas i ett batchjobb bör CLI:n:

1. Skriv detaljerad felinformation till en separat loggfil.
2. Fortsätt att utföra andra jobb istället för att avbryta hela processen.
3. Returnera en utgångskod som inte är noll i slutet för att indikera att vissa jobb misslyckades.
4. Visa tydligt exekveringsresultatet för varje jobb i sammanfattningsrapporten.

Vissa misslyckanden är bara misslyckanden. Det är ingen idé att låtsas något annat. Det är bättre att erkänna dem öppet och sedan komma på hur man löser dem. Samma logik gäller för projekt och för livet.

### Strategi för sammanslagning av metadata

Erkännanderesultat skrivs till `recognized` avsnitt som standard, medan manuellt redigerade fält är markerade i `manual`. Metadatauppdateringar följer en append-only-strategi: om inte `--force` är uttryckligen godkänd, skrivs inte befintliga manuellt kurerade resultat över.

Den punkten blev också tydlig - vissa saker, när de väl har skrivits över, är bara borta. Det är ofta bättre att bevara dem, eftersom själva skivan har ett värde.

### Katalogskapande Atomicity

Använd `fs.mkdir({ recursive: true })` för att säkerställa att katalogskapandet förblir atomärt och för att undvika rasförhållanden i samtidiga scenarier.

Kanske är det så säkerhet känns - att vara stabil när stabilitet är viktig, att röra sig snabbt när hastigheten spelar roll, och aldrig fastna i andra skull.

## Slutsats

Som kärnverktyget för bildtillgångshantering i HagiCode-projektet löser ImgBin våra problem genom följande designval:

1. **Enhetlig ingångspunkt**: CLI täcker generering, anteckningar, miniatyrer och alla andra kärnoperationer.
2. **Metadatadriven**: varje tillgång har en komplett `metadata.json`, vilket möjliggör sökning och spårbarhet.
3. **Provider Adapter**: flexibel abstraktion för externa API:er, vilket gör testning och förlängning enklare.
4. **Stöd för batchjobb**: batchbildgenerering kan automatiseras inom CI/CD-arbetsflöden.

Allt annat kan ha bleknat, men det här tillvägagångssättet visade sig verkligen vara användbart.

Denna lösning förbättrar inte bara HagiCodes egen utvecklingseffektivitet, utan bildar också ett återanvändbart ramverk för bildtillgångshantering. Om du bygger ett liknande flerkomponentprojekt tror jag att ImgBins designidéer kan ge dig lite inspiration.

Ungdom handlar om att pröva saker och göra lite oreda. Om du aldrig utsätter dig för det, hur skulle du då veta vad du verkligen är kapabel till?
