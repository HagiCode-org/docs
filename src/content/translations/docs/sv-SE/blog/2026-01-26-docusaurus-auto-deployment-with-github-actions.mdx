## Lägga till stöd för GitHub Pages Auto-Deployment till HagiCode

> Detta projekt fick ursprungligen kodnamnet PCode och har nu officiellt bytt namn till HagiCode. Den här artikeln dokumenterar hur vi introducerade automatisk statisk webbplatsdistribution så att publicera innehåll blir lika enkelt som att dricka vatten.

## Bakgrund / Introduktion

Under utvecklingen av HagiCode stötte vi på ett mycket praktiskt problem: eftersom vår dokumentation och våra förslag fortsatte att växa, blev hanteringen och presentationen av innehållet allt mer brådskande. Vi bestämde oss för att använda GitHub-sidor för att vara värd för vår statiska webbplats, men att bygga och distribuera den manuellt var alldeles för besvärligt. Varje förändring krävde en lokal konstruktion, förpackning och sedan en manuell push till `gh-pages` gren. Det var inte bara ineffektivt, utan också lätt att få fel.

För att lösa detta problem, främst för att vi ville spara lite ansträngning, behövde vi ett automatiserat arbetsflöde för distribution. Den här artikeln registrerar i detalj hur vi lade till GitHub Actions-baserat stöd för automatisk distribution till HagiCode-projektet, så att vi kan fokusera på att skriva innehåll och låta automatisering hantera resten.

## Målanalys

Innan vi går in i implementeringen måste vi först klargöra vad den här uppgiften faktiskt är tänkt att åstadkomma. Att slipa yxan försenar trots allt inte huggningen av ved.

### Kärnkrav

1.  **Automatiskt bygge**: Utlöser automatiskt byggarbetsflödet när koden skjuts till `main` gren.
2.  **Automatisk distribution**: Efter en framgångsrik konstruktion distribuerar du automatiskt de genererade statiska filerna till GitHub-sidor.
3.  **Konsistent miljö**: Se till att CI-miljön matchar den lokala byggmiljön för att undvika den klassiska pinsamheten med "det fungerar lokalt, men misslyckas online."

### Tekniskt val

Eftersom HagiCode är byggd på Docusaurus, en mycket populär React statisk webbplatsgenerator, kan vi använda GitHub Actions för att uppnå detta mål.

## Konfigurera GitHub Actions arbetsflöde

GitHub Actions är CI/CD-tjänsten som tillhandahålls av GitHub. Genom att definiera arbetsflödesfiler i YAML-format inuti arkivet kan vi anpassa alla typer av automatiseringsuppgifter.

### Skapa arbetsflödesfilen

Vi måste skapa en ny konfigurationsfil i `.github/workflows` katalog vid projektroten, till exempel `deploy.yml`. Om katalogen inte finns ännu, skapa den först.

Kärnan i denna konfigurationsfil är följande:

1.  **Triggertillstånd**: Lyssna efter `push` händelser på `main` gren.
2.  **Runtime-miljö**: Den senaste Ubuntu-versionen.
3.  **Byggsteg**:
    -   Kolla in koden
    -   Installera Node.js
    -   Installationsberoenden (`npm install`)
    -   Bygg de statiska filerna (`npm run build`)
4.  **Implementeringssteg**: Använd den officiella `action-gh-pages` åtgärd för att skjuta byggutgången till `gh-pages` gren.

### Nyckelkonfigurationskod

Nedan är konfigurationsmallen som vi till slut antog:

```yaml
name: Deploy to GitHub Pages

# Trigger condition: run when code is pushed to the main branch
on:
  push:
    branches:
      - main
    # Add path filters if needed, for example only build when docs change
    # paths:
    #   - 'docs/**'
    #   - 'package.json'

# Set permissions, which is important for GitHub Pages deployment
permissions:
  contents: read
  pages: write
  id-token: write

# Concurrency control: cancel older builds for the same branch
concurrency:
  group: "pages"
  cancel-in-progress: false

jobs:
  build:
    runs-on: ubuntu-latest
    steps:
      - name: Checkout
        uses: actions/checkout@v4
        # Note: fetch-depth: 0 is required, otherwise the build version may be inaccurate
        with:
          fetch-depth: 0

      - name: Setup Node
        uses: actions/setup-node@v4
        with:
          node-version: 20 # Keep this consistent with the local development environment
          cache: 'npm'     # Enabling cache can speed up the build process

      - name: Install dependencies
        run: npm ci
        # Use npm ci instead of npm install because it is faster and stricter, which makes it better for CI

      - name: Build website
        run: npm run build
        env:
          # Configure environment variables here if your site build requires them
          # NODE_ENV: production
          # PUBLIC_URL: /your-repo-name
          
      - name: Upload artifact
        uses: actions/upload-pages-artifact@v3
        with:
          path: ./build # Default Docusaurus output directory

  deploy:
    environment:
      name: github-pages
      url: ${{ steps.deployment.outputs.page_url }}
    runs-on: ubuntu-latest
    needs: build
    steps:
      - name: Deploy to GitHub Pages
        id: deployment
        uses: actions/deploy-pages@v4
```

## Vanliga fallgropar under implementering

I praktiken stötte vi på flera problem. Jag delar dem här i hopp om att andra kan undvika dem, eller åtminstone förbereda lösningar i förväg.

### 1. Tillståndsproblem för GitHub Token

När vi först ställde in saker och ting, fortsatte driftsättningen att misslyckas med en 403 (förbjuden). Efter mycket grävande upptäckte vi att GitHub är standard `GITHUB_TOKEN` hade inte behörighet att skriva till Pages.

**Lösning**: I förvarets `Settings` -> `Actions` -> `General` -> `Workflow permissions`, se till att välja **"Läs- och skrivbehörigheter"**.

### 2. Fel byggkatalogsökväg

Docusaurus placerar byggda statiska filer i `build` katalog som standard. Vissa projekt kan dock konfigureras annorlunda. Till exempel, Create React App är som standard `build`, medan Vite har som standard `dist`. Om Actions rapporterar att den inte kan hitta utdatafilerna, kontrollera utdatasökvägen i `docusaurus.config.js`.

### 3. Subpath-problem

Om ditt förråd inte är en användarhemsida, vilket betyder att det inte är det `username.github.io`, utan istället en projektsida som t.ex `username.github.io/project-name`, då måste du konfigurera `baseUrl`.

In `docusaurus.config.js`:

```javascript
module.exports = {
  // ...
  url: 'https://hagicode.com', // Your Hagicode URL
  baseUrl: '/', // Deploy from the root path
  // ...
};
```
Denna detalj är lätt att förbise. Om den är felaktigt konfigurerad kan sidan laddas som en tom skärm eftersom tillgångsvägarna inte kan lösas.

## Verifierar resultatet

Efter att ha konfigurerat allt och tryckt på koden kan vi gå till fliken **Actions** i GitHub-förvaret och titta på showen.

Du kommer att se en gul cirkel medan arbetsflödet körs, och när det väl blir grönt betyder det framgång. Om den blir röd, klicka in och inspektera loggarna. I de flesta fall kan du spåra problemet där nere, vanligtvis ett stavfel eller en felaktig sökvägskonfiguration.

När bygget har lyckats, besök `https://<your-username>.github.io/<repository-name>/` och du kommer att se den nyinstallerade webbplatsen.

## Sammanfattning

Genom att introducera GitHub Actions har vi framgångsrikt automatiserat distributionen för HagiCodes dokumentationswebbplats. Detta sparar inte bara den tid som tidigare spenderats på manuella operationer, utan ännu viktigare standardiserar release-arbetsflödet. Nu, oavsett vilken gruppmedlem som uppdaterar dokumentationen, när ändringen är sammanfogad till `main`, kommer det senaste innehållet att finnas live några minuter senare.

**Kärnfördelar**:
- **Högre effektivitet**: Vi gick från "manuell förpackning och manuell uppladdning" till "kod är lika med release."
- **Färre misstag**: Vi eliminerade risken för mänskliga fel under manuella operationer.
- **Bättre utvecklarupplevelse**: Utvecklare kan fokusera mer på innehållskvalitet istället för att distraheras av tråkiga implementeringssteg.

Även om det är lite besvärligt att konfigurera CI/CD till en början, särskilt med alla behörigheter och sökvägsdetaljer, är det en engångsinvestering med stor långsiktig avkastning. Jag rekommenderar starkt att varje statiskt webbplatsprojekt använder ett liknande automatiserat arbetsflöde.
