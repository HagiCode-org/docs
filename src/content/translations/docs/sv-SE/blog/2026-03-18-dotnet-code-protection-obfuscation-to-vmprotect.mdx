## .NET-kodskydd i praktiken: från obfuskation till skydd av virtuella maskiner

> Den här artikeln förklarar hur man implementerar en kodskyddsstrategi med flera lager i .NET-projekt, som täcker hela vägen från grundläggande förvirring till professionellt skydd av virtuella maskiner.

## Bakgrund

Inom .NET-applikationsutveckling är skyddet av kärnkod som licensvalidering, affärslogik och känslig konfiguration från dekompilering och reverse engineering ett ämne som du inte kan undvika. I takt med att .NET-ekosystemet har mognat har utvecklare fått tillgång till en rad skyddsalternativ, från inbyggda förvirringsattribut till professionella virtualiseringsbaserade skyddsverktyg.

Som ett komplext flerspråkigt monorepo-projekt inkluderar HagiCode skrivbordsapplikationer, byggsystem och licenshanteringsfunktioner. Koden innehåller oundvikligen licensvalideringslogik, känslig konfiguration som API-nycklar och produkt-ID:n och affärskritisk logik. De delarna behöver seriöst skydd, eftersom ingen vill att deras hårda arbete ska exponeras så lätt.

Den här artikeln delar den kodskyddsstrategi som vi faktiskt använde i HagiCode-projektet och sammanfattar hela resan från tidiga fallgropar till senare optimering. Förhoppningsvis ger det dig några användbara idéer.

## Om HagiCode

[HagiCode](https://hagicode.com) är ett AI-kodningsassistentprojekt med öppen källkod tillägnad att ge utvecklare en intelligent programmeringsupplevelse. Projektet använder en monorepo-arkitektur och upprätthåller samtidigt en VSCode-tillägg, backend AI-tjänster, en plattformsoberoende skrivbordsklient och mer. Denna komplexitet på flera språk och flera plattformar gör kodskydd till en teknisk utmaning som vi måste möta direkt.

Tillvägagångssättet som delas i den här artikeln är resultatet av verkliga försök och misstag under utvecklingen av HagiCode. Om du vill se hur vi löste dessa tekniska problem, fortsätt läsa. Du kan hitta några oväntade takeaways.

## Kärninnehåll

### 1. Microsofts inbyggda obfuskeringsattribut

.NET Framework har en inbyggd `[ObfuscationAttribute]`, som är den mest grundläggande och vanligaste kodobfuskeringsmarkören. Detta attribut lever i `System.Reflection` namnutrymme och låter dig tillämpa baslinjeskydd på kod utan att introducera verktyg från tredje part.

**Kärnfunktioner**:
- `Feature` egenskap: Anger obfuskeringsfunktionen, t.ex `"ultra"` (hög obfuskation) eller `"all"` (fullständig förvirring)
- `Exclude` egendom: `true` betyder utesluta från förvirring, och `false` medel tillämpa fördunkling
- Kan appliceras på klasser, metoder, egenskaper och andra typmedlemmar

I HagiCode-projektet kan du se det användas så här:

```csharp
[Obfuscation(Feature = "ultra", Exclude = false)]
public async Task<LicenseValidationResult?> ValidateLicenseAsync(...)
```

**fördelarna** med detta tillvägagångssätt är ganska uppenbara:
- Inga extra beroenden; den är inbyggd i .NET Framework direkt
- Kan kännas igen och bearbetas av tredjeparts obfuskeringsverktyg
- Ökar inte storleken på den kompilerade sammansättningen avsevärt

Som sagt, det har också begränsningar. Det är bara en markör, och det faktiska förvirringsresultatet beror på verktygsimplementeringen. Det kan inte tillhandahålla säkerhet på skyddsnivå för virtuella maskiner.

### 2. VMP (Virtual Machine Protection)

VMP är ett professionellt kodskyddsverktyg som ger skydd på hög nivå genom att kompilera kod till instruktioner för virtuella maskiner. Till skillnad från enkel namnobfuskering omvandlar VMP faktiskt kodlogik till en form som konventionella dekompilatorer inte kan rekonstruera.

**Klassificering av skyddsnivå**:

| Nivå | Virtualisering | Mutation | Anti-debugging | Strängkryptering | Användningsfall |
|------|--------|------|--------|------------|----------|
| HÖG | full | hög | aktiverat | aktiverat | Licensvalidering, samtidighet av sessioner, känsliga konstanter |
| MEDEL | partiell | medium | aktiverat | aktiverat | Affärslogik, domänmodeller |
| LÅG | ingen | låg | funktionshindrade | funktionshindrade | Verktygsklasser, icke-kritisk kod |

HagiCode-projektet definierar ett deklarativt attributsystem för märkning av kod som behöver skydd:

```csharp
// High-priority protection
[VmProtect(VmProtectionPriority.High, Reason = "Contains license verification logic")]
public class KeygenClient { ... }

// Exclude from protection
[VmExclude(Reason = "Public API that must remain unchanged")]
public class PublicApi { ... }

// Inherited protection
[VmProtect(Priority.High, ProtectDerived = true)]
public class BaseLicenseValidator { ... }
```

### 3. Byggtidsskyddsstrategi

VMP-skydd spelar inte bara roll under körning. Det måste också automatiseras som en del av byggpipelinen, för att göra det manuellt skulle vara alldeles för tråkigt. HagiCodes byggsystem stöder flera lägen:

- **Native Windows-läge**: Anropa VMProtect-verktyget direkt
- **Linux Docker-behållarläge**: Kör VMP i en behållare för att lösa plattformsoberoende kompatibilitetsproblem
- **Attributskanning**: Upptäck automatiskt skyddsmarkörer i kod
- **Verifieringsmekanism**: Bekräfta att skyddet har tillämpats framgångsrikt

Tillsammans gör dessa funktioner processen mycket lättare att hantera.

## Lösning

### 1. Använda Microsofts inbyggda obfuskeringsattribut

Ansök `ObfuscationAttribute` direkt i koden:

```csharp
using System.Reflection;

[Obfuscation(Feature = "ultra", Exclude = false)]
public class LicenseService
{
    [Obfuscation(Feature = "ultra", Exclude = false)]
    public async Task<bool> ValidateLicenseAsync(string key)
    {
        // License validation logic
    }

    [Obfuscation(Feature = "flow", Exclude = false)]
    private string DecryptToken(string encrypted)
    {
        // Decryption logic
    }
}
```

Ibland behöver du låta testsammansättningar komma åt interna medlemmar samtidigt som du håller produktionskoden säker:

```csharp
// AssemblyInfo.cs
[assembly: InternalsVisibleTo("HagiCode.Application.Tests")]
[assembly: InternalsVisibleTo("DynamicProxyGenAssembly2")] // for Moq
```

Detta gör testningen mycket bekvämare, eftersom koden fortfarande måste testas ordentligt.

### 2. Anpassade attributdefinitioner för VMP-skydd

Skapa anpassade skyddsattribut för att kontrollera VMP-beteende:

```csharp
using System;

[AttributeUsage(AttributeTargets.Class | AttributeTargets.Method | AttributeTargets.Property)]
public class VmProtectAttribute : Attribute
{
    public VmProtectionPriority Priority { get; set; }
    public string? Reason { get; set; }
    public bool ProtectDerived { get; set; }
}

[AttributeUsage(AttributeTargets.Class | AttributeTargets.Method | AttributeTargets.Property)]
public class VmExcludeAttribute : Attribute
{
    public string? Reason { get; set; }
}

public enum VmProtectionPriority
{
    None = 0,
    Low = 1,
    Medium = 2,
    High = 3
}
```

Anpassade attribut är ofta lättare att arbeta med eftersom de speglar dina egna skyddskrav direkt.

### 3. VMP-konfigurationsfil

```yaml
# vmp_config.yml
protection:
  priority_mode: "attribute"  # Attribute-based priority
  default_level: "medium"

tools:
  - name: "vmprotect"
    path: "C:\\Program Files\\VMProtect Ultimate\\VMProtect.exe"

protection_levels:
  high:
    virtualization: "full"
    mutation: "high"
    anti_debug: true
    anti_dump: true
    encrypt_strings: true
    encrypt_resources: true

  medium:
    virtualization: "partial"
    mutation: "medium"
    anti_debug: true
    encrypt_strings: true

  low:
    virtualization: "none"
    mutation: "low"
    anti_debug: false
```

En tydligare konfiguration gör systemet mycket lättare att underhålla senare.

## Praktisk guide

### 1. Skyddspraxis för kritiska komponenter

Enligt HagiCodes kodskyddsspecifikation måste följande komponenter använda HÖG-prioritetsskydd:

```csharp
// Production constants - must be encrypted and protected by VMP
[VmProtect(VmProtectionPriority.High, Reason = "Production constants")]
public static class ProductionConstants
{
    // Encrypted string accessor, protected by VMP
    [VmProtect(VmProtectionPriority.High)]
    public static string GetLicenseServerUrl(IOptions<LicenseOptions> options) => ...;
}

// License validation logic
[VmProtect(VmProtectionPriority.High, Reason = "License verification logic")]
public class KeygenClient : IKeygenClient
{
    [Obfuscation(Feature = "ultra", Exclude = false)]
    public async Task<LicenseValidationResult?> ValidateLicenseAsync(...) { ... }
}

// Machine fingerprint service
[VmProtect(VmProtectionPriority.High)]
public class MachineFingerprintService : IMachineFingerprintService { ... }
```

Kritisk kod förtjänar ett starkare skydd, eftersom att exponera kärnlogik skulle orsaka verkliga problem.

### 2. Strängkryptering och körtidsdekryptering

Kryptera strängar vid byggtid och dekryptera dem vid körning:

```csharp
public static class StringDecryption
{
    [VmProtect(VmProtectionPriority.High, Reason = "CRITICAL SECURITY")]
    public static string DecryptString(byte[] encryptedData, byte[] key, byte[] iv)
    {
        using var aes = Aes.Create();
        aes.Key = key;
        aes.IV = iv;

        using var decryptor = aes.CreateDecryptor();
        using var ms = new MemoryStream(encryptedData);
        using var cs = new CryptoStream(ms, decryptor, CryptoStreamMode.Read);
        using var reader = new StreamReader(cs);

        return reader.ReadToEnd();
    }
}

// Production constant accessor (lazy loading + caching)
public static class ProductionConstants
{
    private static string? _cachedLicenseServerUrl;

    public static string GetLicenseServerUrl(IOptions<LicenseOptions> options)
    {
        if (_cachedLicenseServerUrl == null)
        {
            var encrypted = GetEncryptedLicenseServerUrl();
#if DEBUG
            _cachedLicenseServerUrl = options.Value.PrimaryServer.Url;
#else
            _cachedLicenseServerUrl = StringDecryption.DecryptString(
                encrypted,
                GetEncryptionKey(),
                GetEncryptionIV());
#endif
        }
        return _cachedLicenseServerUrl;
    }
}
```

Detta steg är viktigt eftersom känslig information aldrig bör lämnas i klartext.

### 3. VMP-skyddsverifiering

Efter bygget måste du verifiera om skyddet har tillämpats framgångsrikt; annars kan du inte vara säker på att det faktiskt fungerar:

```csharp
// Example verification script
public bool VerifyProtection(string assemblyPath)
{
    // 1. Check the VMP signature
    var bytes = File.ReadAllBytes(assemblyPath);
    var vmpSignature = Encoding.ASCII.GetBytes("VMProtect");
    if (bytes.Any(b => vmpSignature.Contains(b)))
    {
        return true;
    }

    // 2. Check for file size changes (the protected file is usually larger)
    var originalInfo = new FileInfo(assemblyPath.Replace(".dll", ".bak"));
    if (originalInfo.Exists)
    {
        var sizeRatio = (double)new FileInfo(assemblyPath).Length / originalInfo.Length;
        return sizeRatio > 1.1;
    }

    return false;
}
```

Verifiering är alltid värt att göra, för annars kan problem glida fram obemärkt.

### 4. Anteckningar och varningar

Det finns flera fallgropar här som förtjänar särskild uppmärksamhet:

1. **Obfuskera inte all kod**: Offentliga API:er, gränssnittsdefinitioner och DTO-klasser behöver vanligtvis inte skydd. Överdriven förvirring kan skada prestanda och felsökning. HagiCode-projektet lärde sig detta på den hårda vägen.

2. **Skydda nycklar**: Metoder som hämtar krypteringsnycklar måste få samma eller högre skyddsnivå än själva krypterade data; annars förlorar hela installationen sitt värde.

3. **Balanstestning och produktion**: DEBUG-byggen bör hoppa över kryptering för att göra utveckling och felsökning enklare, medan RELEASE-byggen bör möjliggöra fullt skydd. Kom ihåg att separera dem med villkorlig sammanställning som t.ex `#if DEBUG`.

4. **Tänk på Docker-miljön**: Att köra VMP på Linux kräver en containerbaserad metod för att säkerställa verktygskompatibilitet. HagiCode använder en Wine + VMP-containerlösning för att lösa plattformsoberoende problem.

5. **Verifiering är obligatorisk**: När bygget är klart måste du verifiera att skyddet har tillämpats. Annars kan känslig kod fortfarande exponeras, och verifieringskoden som visats tidigare finns för just detta ändamål.

## Slutsats

Med denna flerskiktsskyddsstrategi byggde HagiCode ett omfattande kodsäkerhetssystem som sträcker sig från baslinjeförvirring till skydd av virtuella maskiner:

- **Lager 1**: Använd `ObfuscationAttribute` för baslinjemärkning och ge tips till tredjepartsverktyg
- **Lager 2**: Använd anpassad `VmProtectAttribute` förklaringar för att uttrycka skyddets avsikt och prioritet
- **Layer 3**: Använd VMP virtuell maskinskydd för att omvandla kritisk kod till oåterkalleliga instruktioner för virtuella datorer
- **Layer 4**: Skanna och applicera skydd automatiskt under bygget och verifiera sedan resultatet

Detta tillvägagångssätt kan motstå vanliga dekompileringsverktyg samtidigt som det står upp bättre mot avancerade reverse engineering-attacker. Om du bygger en .NET-applikation som behöver kodskydd hoppas jag att detta ger dig åtminstone en användbar referenspunkt.
