## Design.md: En lösning för konsekvent AI-driven gränssnittsdesign

> Hur kan du hålla AI-genererade användargränssnitt konsekventa i en tidevarv med AI-assisterad frontend-utveckling? Den här artikeln delar vår praktiska erfarenhet av att bygga en designgallerisajt baserad på awesome-design-md, tillsammans med hur man skapar en strukturerad `design.md` för att vägleda AI mot standardiserad UI-design.

## Bakgrund

Alla som har använt AI för att skriva frontend-kod har förmodligen haft en liknande upplevelse: be AI att generera samma sida flera gånger, och varje resultat kommer ut i en annan stil. Ibland är hörnen rundade, ibland är de vassa. Ibland är avståndet 8px, och andra gånger blir det 16px. Till och med samma knapp kan se olika ut i olika konversationer.

Detta är inte en isolerad fråga. När AI-assisterad utveckling blir vanligare har **bristen på konsekvens i AI-genererat frontend-gränssnitt** blivit ett utbrett problem. Olika AI-assistenter, olika uppmaningar och till och med samma assistent över olika konversationer kan producera dramatiskt olika gränssnittsdesigner. Det skapar en enorm underhållskostnad under produktiteration.

Grundorsaken är faktiskt enkel: det finns inget **auktoritativt designreferensdokument**. Traditionella CSS-stilmallar kan berätta för utvecklare "hur man implementerar" något, men de kan inte helt kommunicera "varför det är utformat på det här sättet" eller "vilket designmönster som ska användas i vilket scenario." För AI är en tydlig och strukturerad beskrivning ännu viktigare för att förstå designkonventioner.

Samtidigt erbjuder open source-gemenskapen redan några utmärkta resurser. Den [VoltAgent/awesome-design-md](https://github.com/VoltAgent/awesome-design-md) Project samlar in designsystemdokumentation från många välkända företag. Varje katalog innehåller en `README.md`, a `DESIGN.md`, och förhandsgranska HTML. Men allt detta är utspridda över uppströmsförvaret, vilket gör det svårt att bläddra och jämföra snabbt.

Så kan vi konsolidera dessa resurser till ett lättbläddrat designgalleri, samtidigt som vi destillerar ett strukturerat `design.md` för AI att använda?

Svaret är ja. Låt mig sedan gå igenom vårt tillvägagångssätt.

## Analys

Innan du skriver kod, låt oss dela upp de tekniska utmaningarna bakom detta problem.

### Källa Content Management: Hur förenar du spridda designresurser?

Uppströms awesome-design-md-förvaret innehåller ett stort antal designdokument, men vi behövde ett sätt att föra in dem i vårt eget projekt.

**Lösning: använd git undermodul**

```
awesome-design-md-site
└── vendor/awesome-design-md          # Upstream resources (git submodule)
```

Detta ger oss flera fördelar:
- Versionskontroll: vi kan fästa en specifik uppströmsversion
- Offlinebyggen: du behöver inte begära externa API:er under bygget
- Innehållsgranskning: specifika förändringar är synliga i PR

### Statisk webbplatsarkitektur: Hur tillhandahåller du dynamisk sökning samtidigt som du håller dig helt statisk?

Eftersom detta är ett designgalleri är sökning ett måste. Men Astro är en statisk webbplatsgenerator, så hur implementerar du sökning i realtid?

**Lösning: Synkronisering av reaktionsö + URL-frågeparameter**

```typescript
// src/components/gallery/SearchToolbar.tsx

export function SearchToolbar() {
```

Fördelen med det här tillvägagångssättet är att det behåller distributionsbarheten för en statisk webbplats intakt, vilket innebär att den kan distribueras till vilken statisk värdtjänst som helst, samtidigt som den ger en omedelbar filtreringsupplevelse.

### Designdokumentation: Hur hjälper du AI att förstå och följa designstandarder?

Detta är kärnan i hela lösningen. Vi måste skapa en strukturerad `design.md` som AI kan förstå och tillämpa.

**Lösning: låna strukturen i ClickHouse `DESIGN.md`**

[ClickHouse's `DESIGN.md`](https://github.com/ClickHouse/ClickHouse/blob/master/DESIGN.md) är en utmärkt referens. Det inkluderar:
- Visuellt tema och atmosfär
- Färgpalett & roller
- Typografiregler
- Komponentstylingar
- Layoutprinciper
- Djup & Höjd
- Göra och inte göra
- Responsivt beteende
- Agentpromptguide

Vårt tillvägagångssätt är: **återanvänd strukturen, skriv om innehållet**. Vi behåller sektionsstrukturen för ClickHouse `DESIGN.md`, men ersätt innehållet med de faktiska designtokens och komponentkonventioner som används i vårt eget projekt.

## Lösning

Baserat på analysen ovan består vår lösning av fyra kärnmoduler.

### 1. Innehållsförtäring Pipeline

Detta är grunden för hela systemet, ansvarigt för att extrahera och normalisera innehåll från uppströmsresurser.

```typescript
// src/lib/content/awesomeDesignCatalog.ts

export async function scanSourceEntries(): Promise<DesignEntry[]> {
  const designDir = 'vendor/awesome-design-md/design-md';
  const entries: DesignEntry[] = [];

  for (const dir of await fs.readdir(designDir)) {
    const entryPath = path.join(designDir, dir);
    if (await isValidDesignEntry(entryPath)) {
      const entry = await normalizeDesignEntry(entryPath);
      entries.push(entry);
    }
  }

  return entries;
}

async function isValidDesignEntry(dir: string): Promise<boolean> {
  const requiredFiles = ['README.md', 'DESIGN.md'];
  for (const file of requiredFiles) {
    if (!(await fileExists(path.join(dir, file)))) {
      return false;
    }
  }
  return true;
}
```

### 2. Galleris webbläsargränssnitt

Galleriets gränssnitt innehåller tre huvuddelar:

**Hemsida**: visar ett kortrutnät med alla designposter, och varje kort innehåller:
- Designbidragstitel och sammanfattning
- Förhandsgranska bild, om tillgänglig
- Snabbsökningsmarkering

**Detaljsida**: samlar ihop all information för en enda designpost:
- `README` dokument
- `DESIGN` dokument
- Förhandsgranska med växling av ljus/mörkt tema
- Navigering till intilliggande poster

**Navigering**: stöder återgång till galleriet och bläddra i angränsande poster

Hemsidans galleri använder en kortlayout med hög densitet, tillplattad `design.md` inlägg från olika källor till ett enhetligt visuellt ramverk så att team snabbt kan jämföra varumärkesstilar, knappmönster och typografisk rytm.

Efter att ha öppnat en specifik post placerar detaljsidan designsammanfattningen och liveförhandsgranskningen på samma sida, vilket minskar kostnaderna för att byta fram och tillbaka mellan dokumentation, förhandsgranskningar och källkod.

### 3. Sökfunktioner

Sökfunktionen är baserad på filtrering på klientsidan, med status bevarad genom URL-frågeparametrar:

```typescript
// src/components/gallery/SearchToolbar.tsx

function SearchToolbar({ entries }: { entries: DesignEntry[] }) {
  const [query, setQuery] = useState('');
  const [results, setResults] = useState(entries);

  useEffect(() => {
    const params = new URLSearchParams(window.location.search);
    const initialQuery = params.get('q') || '';
    setQuery(initialQuery);
    filterEntries(initialQuery);
  }, []);

  const filterEntries = (searchQuery: string) => {
    const filtered = entries.filter(entry =>
      entry.searchText.toLowerCase().includes(searchQuery.toLowerCase())
    );
    setResults(filtered);
  };

  const handleChange = (e: React.ChangeEvent<HTMLInputElement>) => {
    const value = e.target.value;
    setQuery(value);
    filterEntries(value);

    // Update the URL without triggering a page refresh
    const newUrl = value
      ? `${window.location.pathname}?q=${encodeURIComponent(value)}`
      : window.location.pathname;
    window.history.replaceState({}, '', newUrl);
  };

  return (
    <div className="search-toolbar">
      <input
        type="text"
        value={query}
        onChange={handleChange}
        placeholder="Search design entries..."
      />
      <span className="result-count">{results.length} results</span>
    </div>
  );
}
```

## Öva

Nu när vi har täckt lösningen, hur implementerar du den egentligen?

### Implementeringssteg

**Steg 1: Initiera undermodulen**

```bash
# Add the upstream repository as a submodule
git submodule add https://github.com/VoltAgent/awesome-design-md.git vendor/awesome-design-md

# Initialize and update the submodule
git submodule update --init --recursive
```

**Steg 2: Skapa innehållspipeline**

Implementera `awesomeDesignCatalog.ts`, inklusive:
- Filskanning och valideringslogik
- Markdown-rendering med Astros inbyggda renderer
- Utvinning av inmatningsdata

**Steg 3: Bygg galleriets användargränssnitt**

Använd Astro + React Islands för att skapa:
- Hemsidas gallerilayout (kortrutnät)
- Designa kortkomponenter
- Sök verktygsfält
- Detalj sidlayout

**Steg 4: Skriv designdokumentet**

Baserat på strukturen i ClickHouse `DESIGN.md`, fyll i själva designpolletterna från ditt eget projekt. Uppdatering `README.md` och lägg till en länk till `design.md`.

### Anteckningar

**Säkerhet**: Markdown-rendering kräver filtrering av osäker HTML. Astros inbyggda renderingsfilter `script` taggar som standard, men du måste fortfarande se upp för XSS-risker.

**Prestanda**: Ett stort antal iframe-förhandsvisningar kan påverka prestandan för första målningen. Det rekommenderas att använda `loading="lazy"` för att lata ladda förhandsgranskningsinnehåll.

**Underhållsbarhet**: `design.md` måste vara synkroniserad med kodimplementeringen. Det rekommenderas att lägga till CI-kontroller för att säkerställa att CSS-variabler förblir konsekventa mellan dokumentation och kod.

**Tillgänglighet**: Se till att färgkontrasten uppfyller WCAG AA-standarden (minst 4,5:1).

### AI-användningsguide

Efter att ha skapat `design.md`, hur får man AI att faktiskt använda det? Här är några praktiska tips:

**Tips 1: Hänvisa till det uttryckligen i prompten**

```
Please refer to the design.md file in the project root and use the design conventions defined there to implement the following components:
- Buttons: use the primary color with an 8px border radius
- Cards: use the elevation-2 shadow level
```

**Tips 2: Kräv AI för att referera till specifika CSS-variabler**

```
Implement a navigation bar with the following requirements:
- Use --color-bg-primary for the background color
- Use --color-border-subtle for borders
- Use --text-color-primary for text
```

**Tips 3: Inkludera `design.md` innehåll i systemprompten**

Om ditt AI-verktyg stöder anpassade systemuppmaningar kan du lägga till kärninnehållet i `design.md` direkt till den.

### Teststrategi

**Tester av innehållspipeline**:
- Scenarier för saknad fil (saknas `README.md` eller `DESIGN.md`)
- Formatfelscenarier (Markdown-parsningsfel)
- Scenarier för tomma kataloger

**Test av sökfunktioner**:
- Tom resultathantering
- Specialtecken som kinesiska och emoji
- URL-synkroniseringsverifiering

**Testning av UI-komponenter**:
- Växla mellan ljus/mörkt tema
- Responsiv layout
- Förhandsgranska laddningstillstånd

## Slutsats

Inkonsekvensen i AI-genererade användargränssnitt som HagiCode stötte på under utvecklingen var i sin kärna orsakad av avsaknaden av ett strukturerat designreferensdokument. Genom att bygga en designgallerisajt och skapa en standardiserad `design.md`, vi löste det här problemet framgångsrikt.

Kärnvärdet av denna lösning ligger i:
- **Enade resurser**: konsoliderar spridd designsystemdokumentation
- **Strukturerade standarder**: uttrycker designkonventioner i ett format som AI kan förstå
- **Kontinuerligt underhåll**: hålla innehållet uppdaterat genom `git submodule`

Om du också använder AI för frontend-utveckling är detta tillvägagångssätt värt att prova. Att skapa en strukturerad `design.md` förbättrar inte bara konsistensen i AI-genererad kod, utan hjälper också ditt team att upprätthålla enhetliga designstandarder internt.
