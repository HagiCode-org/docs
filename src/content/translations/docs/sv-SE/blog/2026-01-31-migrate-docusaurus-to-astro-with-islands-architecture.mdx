## Från Docusaurus 3.x till Astro 5.x: A Retrospective on the HagiCode Site Migration

> Den här artikeln ser tillbaka på vår fullständiga migrering av HagiCodes officiella webbplats från Docusaurus 3.x till Astro 5.x. Vi kommer att ta en djupdykning i hur Astro's Islands Architecture hjälpte oss att lösa prestandaflaskhalsar samtidigt som vi bevarade våra befintliga React-komponenttillgångar och levererade förbättringar i både bygghastighet och laddningsprestanda.

## Bakgrund

I januari 2026 utförde vi en "hjärttransplantation" på HagiCodes officiella webbplats genom att helt migrera dess kärna ramverk från Docusaurus 3.x till Astro 5.x. Detta var ingen impulsiv omskrivning, utan ett noga övervägt tekniskt beslut.

Innan migreringen var vår sida funktionellt klar, men den hade börjat visa några klassiska "lyxproblem": uppsvällda byggartefakter, överdriven JavaScript-nyttolast och mindre än idealisk sidladdningshastighet på komplexa dokumentationssidor. Som ett AI-kodningsassistentprojekt behöver HagiCode frekvent dokumentation och funktionsuppdateringar, så byggeffektiviteten påverkar direkt releasehastigheten. Samtidigt ville vi att sajten skulle vara mer sökmotorvänlig (SEO) så att fler utvecklare kunde upptäcka projektet.

För att lösa dessa smärtpunkter tog vi ett djärvt beslut: bygg om hela systemet på Astro. Effekten av det beslutet kan bli ännu större än du förväntar dig. Jag kommer att gå in på detaljerna inom kort.

## Om HagiCode

Tillvägagångssättet för webbplatsmigrering som delas i den här artikeln kommer från vår praktiska erfarenhet av [HagiCode](https://github.com/HagiCode-org/site) projekt.

HagiCode är en AI-kodningsassistent fokuserad på att förbättra utvecklingseffektiviteten. Vi bryr oss inte bara om att iterera på kärnfunktioner, utan också om utvecklarupplevelsen. Denna webbplatsrefaktor var också tänkt att ge användarna den snabbaste möjliga upplevelsen när de surfar på våra dokument och officiella webbplats.

## Varför lämna det mogna ekosystemet Docusaurus?

Inom Reacts ekosystem har Docusaurus länge varit "standardsvaret" för dokumentationsplatser. Det fungerar direkt, erbjuder ett rikt plugin-ekosystem och har en aktiv community. Men när HagiCode fick fler funktioner kände vi också dess begränsningar:

1.  **Prestandaflaskhalsar**: Docusaurus är i grunden en React SPA (ensidig applikation). Även om du bara skriver statiska sidor behöver klienten fortfarande ladda React-körtiden och hydratisera sidan, vilket är onödigt tungt för enkla dokumentsidor.
2.  **Stor tillgångsstorlek**: Även när en sida innehåller väldigt lite innehåll förblir den buntade JS-storleken relativt fast. Det är inte idealiskt för mobilanvändare eller dåliga nätverksförhållanden.
3.  **Begränsad flexibilitet**: Även om det är utbyggbart ville vi ha mer kontroll på låg nivå över byggpipelinen.

Astro kom vid exakt rätt tidpunkt för att lösa dessa problem. Den introducerade en ny "öararkitektur": som standard genererar Astro statisk HTML med noll JavaScript, och endast komponenter som kräver interaktivitet "aktiveras" och laddar JS. Det betyder att större delen av vår webbplats blir ren HTML och laddas extremt snabbt.

## Core Migration Strategy: A Smooth Architectural Transition

Migration var inte bara kopiera och klistra. Det krävde en förändring i tankesättet. Vi gick från Docusaurus "all React"-modell till Astros "Core + Islands"-modell.

### 1. Bygga om konfigurationssystemet

Först var vi tvungna att flytta från `docusaurus.config.ts` till `astro.config.mjs`. Detta var inte bara ett filbyte, utan en omskrivning av routing- och bygglogik.

I Docusaurus är allt ett plugin. I Astro är allt en integration. Vi behövde omdefiniera webbplatsens basväg, bygga utdataläge (statisk vs SSR) och tillgångsoptimeringsstrategi.

**Före migrering:**
```typescript
// docusaurus.config.ts
export default {
  title: 'HagiCode',
  url: 'https://hagicode.com',
  baseUrl: '/',
  // ... more configuration
};
```

**Efter migrering:**
```javascript
// astro.config.mjs
import { defineConfig } from 'astro/config';
import react from '@astrojs/react';

export default defineConfig({
  integrations: [react()],
  site: 'https://hagicode.com',
  base: '/',
  // Optimization settings for static assets
  build: {
    inlineStylesheets: 'auto',
  },
});
```

### 2. Vad som ska behållas och vad som ska återställas i React-komponenter

Detta var den mest smärtsamma delen av migrationen. Vår befintliga webbplats hade många React-komponenter, såsom flikar, kodmarkering, feedbackknappar och mer. Att slänga dem skulle vara slösaktigt, men att behålla allt skulle göra JavaScript-nyttolasten för tung.

HagiCode antog en strategi för **progressiv hydrering**:
*   **Rena statiska komponenter**: För presentationsinnehåll som sidhuvuden, sidfötter och vanlig textdokumentation skrev vi om dem som Astro-komponenter (`.astro` filer) och renderade dem direkt till HTML vid byggtiden.
*   **Interaktiva öar**: För komponenter som måste förbli interaktiva, såsom temaväxlare, flikbyte och kodblockskopieringsknappar, behöll vi React-implementeringen och lade till `client:load` eller `client:visible` direktiv.

Till exempel vår vanligaste Tabs-komponent i dokumentationen:

```jsx
// src/components/Tabs.jsx
import { useState } from 'react';
import './Tabs.css'; // Import styles

export default function Tabs({ items }) {
  const [activeIndex, setActiveIndex] = useState(0);
  // ... state logic
  return (
    <div className="tabs-wrapper">
      {/* Rendering logic */}
    </div>
  );
}
```

När den används i Markdown säger vi uttryckligen till Astro: "Denna komponent behöver JS."

```markdown
// src/content/docs/example.mdx
import Tabs from '../../components/Tabs.jsx';

<!-- Load JS only when the component enters the viewport -->
<Tabs client:visible items={...} />
```

På så sätt konkurrerar inte interaktiva komponenter utanför visningsporten om bandbredd, vilket avsevärt förbättrar laddningshastigheten på första skärmen.

### 3. Anpassning av stylingsystemet: Från CSS-moduler till scoped CSS

Docusaurus stöder CSS-moduler som standard, medan Astro uppmuntrar Scoped CSS genom `<style>` tagga. Kärnan bakom båda är stilisolering, men syntaxen är annorlunda.

Under HagiCode-migreringen konverterade vi de flesta komplexa CSS-moduler till Astros scoped styles. Detta visade sig faktiskt vara en bra sak, för i `.astro` filer stilarna och mallarna finns i samma fil, vilket gör underhållet mer intuitivt.

**Före omfaktorn:**
```css
/* Tabs.module.css */
.wrapper { background: var(--ifm-background-color); }
```

**Efter refaktorering (Astro Scoped):**
```html
<!-- Tabs.astro -->
<div class="tabs-wrapper">
  <slot />
</div>

<style>
  .tabs-wrapper {
    /* Use CSS variables directly to adapt to the theme */
    background: var(--bg-color);
    padding: 1rem;
  }
</style>
```

Samtidigt förenade vi det globala CSS-variabelsystemet och använde Astros miljömedvetna funktioner för att säkerställa att mörkt läge växlar smidigt över sidorna.

## Fallgropar vi träffade i praktiken och hur vi löste dem

Under själva HagiCode-migreringen stötte vi på en hel del problem. Här är några av de mest typiska.

### 1. Väg och miljö Variabla smärtpunkter

HagiCode stöder distribution av undersökvägar, till exempel distribution under en underkatalog för GitHub Pages. I Docusaurus, `baseUrl` hanteras automatiskt. I Astro måste vi dock vara mer försiktiga när vi hanterar bildlänkar och API-förfrågningar.

Vi introducerade en miljövariabelmekanism för att hantera detta konsekvent:

```javascript
// Handle paths in the build script
const getBasePath = () => import.meta.env.VITE_SITE_BASE || '/';
```

Se till att inte hårdkoda sökvägar som börjar med `/` i din kod. I utveckling kontra produktion, eller efter konfiguration av en `base` sökväg kan det orsaka 404:or för tillgångar.

### 2. CommonJS-skriptkompatibilitet

Vår gamla webbplats hade några Node.js-skript som användes för uppgifter som att automatiskt hämta mätdata och uppdatera webbplatskartan, och de skrevs i CommonJS (`require`). Astro och moderna byggverktyg har fullt ut omfattat ES-moduler (`import`/`export`).

Om du också har liknande skript, kom ihåg att omfaktorera dem alla till ES-moduler. Det är den riktning som ekosystemet rör sig, och ju tidigare du gör förändringen, desto mindre problem kommer du att ha senare.

```javascript
// Old way
const fs = require('fs');

// New way
import fs from 'fs';
```

### 3. Glöm inte SEO och omdirigeringar

Sökmotorer har redan indexerat HagiCodes gamla Docusaurus-sidor. Om du byter direkt till Astro och URL-strukturen ändras kan du sluta med ett stort antal 404:or och en rejäl nedgång i sökrankningen.

Vi konfigurerade omdirigeringsregler i Astro:

```javascript
// astro.config.mjs
export default defineConfig({
  redirects: {
    '/docs/old-path': '/docs/new-path',
    // Map old links to new links in bulk
  }
});
```

Eller så kan du hantera detta i serverkonfigurationslagret. Se till att gamla länkar kan 301 omdirigeras till de nya adresserna, eftersom detta är avgörande för SEO.

## Sammanfattning

För HagiCode var migreringen från Docusaurus till Astro inte bara en uppgradering av ramverket. Det var också en praktisk implementering av en "performance first"-filosofi.

**Vad vi fick:**
*   **Enastående Lighthouse-resultat**: Efter migreringen närmade sig HagiCode-webbplatsens prestandapoäng lätt ett perfekt resultat.
*   **Snabbare bygghastighet**: Astros inkrementella byggmöjligheter halverar releasetiden för dokumentationsuppdateringar.
*   **Bevarad flexibilitet**: Med Islands Architecture offrade vi inte några interaktiva funktioner och kunde fortfarande använda React där det behövdes.

Om du också har en dokumentationsorienterad webbplats och kämpar med paketstorlek eller laddningshastighet är Astro väl värt att testa. Även om migreringsprocessen kräver en del kirurgi, som att byta namn på PCode till HagiCode och flytta komponenter över en efter en, gör den silkeslena användarupplevelsen du får i gengäld det absolut värt besväret.

Byggsystemet som delas i den här artikeln är det exakta tillvägagångssättet som vi utvecklade genom riktiga försök och fel när vi byggde HagiCode. Om du tycker att det här tillvägagångssättet är värdefullt säger det en del om vår tekniska styrka, och HagiCode i sig är förmodligen värt en närmare titt också.
