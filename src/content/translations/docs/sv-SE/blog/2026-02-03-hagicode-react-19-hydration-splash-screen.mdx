## Designa 12 exceptionella startupplevelser för HagiCode: Från minimalism till cyberpunk

> Det korta gapet mellan att ladda ner en React 19-app och att slutföra Hydration är ett gyllene tillfälle för användare att känna din varumärkespersonlighet. I den här artikeln delar vi ett komplett startstilssystem som vi byggde för HagiCode-projektet med HTML/CSS/JS.

## Bakgrund

Som en modern applikation byggd med ASP.NET Core 10 och React 19 (Vite), använder HagiCode en frontend-backend-separerad implementeringsarkitektur. Frontend build-utgången är paketerad i backend `wwwroot/` katalog och värd av ASP.NET Core.

Den här arkitekturen introducerar dock en klassisk UX-smärta: när användare besöker sidan laddar webbläsaren först HTML, laddar sedan ner det stora JS-paketet och låter slutligen React utföra Hydration. Under denna "vakuumperiod" som varar från några hundra millisekunder till flera sekunder ser användarna antingen en tom skärm eller en livlös statisk sida.

För att fylla det gapet och injicera HagiCodes varumärkespersonlighet behövde vi designa ett startstilssystem implementerat helt med inline-kod inuti `index.html`.

## Om HagiCode

Tillvägagångssättet för stänkskärmsdesign som delas i den här artikeln kommer från vår praktiska erfarenhet av [HagiCode](https://github.com/HagiCode-org/site) projekt. Som AI-kodningsassistent bryr sig HagiCode inte bara om kodgenereringseffektivitet, utan också om utvecklarens visuella upplevelse. Detta startsystem är ett av resultaten av vår strävan efter ultimat frontend-prestanda.

## Kärnutmaningar och arkitekturdesign

Innan vi började designa var vi först tvungna att klargöra de tekniska begränsningarna. Eftersom allt måste implementeras inline i `index.html`, vi kunde inte ladda några externa CSS- eller JS-filer förutom Reacts eget paket.

### Teknisk begränsningsanalys

1.  **Nollberoendeprincip**: Alla stilar måste leva inuti en `<style>` tagg, och all logik måste leva inuti en `<script>` tagga.
2.  **Defensiv CSS**: För att förhindra att globala stilar förorenar startskärmen efter att React-appen har monterats, bestämde vi oss för att slå in alla startstilar med ett högprioriterat ID-prefix som t.ex. `#boot-screen`.
3.  **Prestandan först**: Animationer bör använda CSS `transform` och `opacity` där det är möjligt för att undvika återflöde och säkerställa att huvudtråden förblir oblockerad.
4.  **Visuell konsistens**: Färger och typsnitt måste vara i linje med HagiCodes Tailwind-konfiguration.

### Arkitekturmönster: Skal och injektor

Vi antog ett **variantmönster**. Kärnlogiken är inkapslad i ett omedelbart anropat funktionsuttryck (IIFE), medan den specifika renderingslogiken injiceras genom konfiguration. Detta låter oss växla mellan olika stilar genom enkel konfiguration istället för att skriva om DOM-manipulationslogik upprepade gånger.

Här är kärnarkitekturkoden:

```html
<!-- Inline in index.html -->
<div id="boot-root"></div>

<script>
(function() {
  const BootSequence = {
    config: {
      theme: 'terminal', // Configurable as 'minimal', 'skeleton', 'code-rain', etc.
      color: '#3b82f6'   // Brand color
    },
    
    // Core lifecycle
    init() {
      this.render();
      this.listenForMount();
    },

    // Render the currently selected style
    render() {
      const root = document.getElementById('boot-root');
      if (this.variants[this.config.theme]) {
        root.innerHTML = this.variants[this.config.theme].render();
      }
    },

    // Listen for successful React mount and exit gracefully
    listenForMount() {
      window.addEventListener('hagicode:ready', () => {
        const screen = document.getElementById('boot-root');
        // Fade out first, then remove the DOM to avoid flicker
        screen.style.opacity = '0';
        screen.style.transition = 'opacity 0.3s ease';
        setTimeout(() => screen.remove(), 300);
      });
    },

    // The implementation logic for all 12 styles lives here
    variants: {
      // ...see details below
    }
  };

  BootSequence.init();
})();
</script>
```

## En checklista med 12 startstilsdesigner

Vi grupperade dessa 12 stilar i sex huvudkategorier för att tillfredsställa olika scenarier och estetiska preferenser.

### A. Minimalism

> "Mindre är mer." För scenarier som eftersträvar den ultimata lastningshastigheten erbjuder vi de lättaste möjliga alternativen.

#### 1. Minimalistisk prick
En enkel prick sitter i mitten av skärmen, ihopkopplad med en andningsanimation.
*   **Implementering**: CSS `@keyframes` kontrollerar skala och opacitet.
*   **Bäst för**: Alla fall där sidan måste förbli helt ren.

#### 2. Brand Reveal
Använder SVG `stroke-dasharray` animering, simulerar den en handritad avslöjande av HagiCode-logotypens rader, följt av en tona in text.
*   **Teknik**: SVG-banaanimering med en mycket polerad känsla.

### B. Skeleton Screen Mimicry

> "Konsten att lura ögat." Genom att simulera en riktig UI-layout känns det som att sidan redan är halvladdad.

#### 3. Chattskelett i sidofältet
Detta kan vara det mest praktiska alternativet. Vi byggde manuellt en layout i HTML som speglar React `Sidebar` och `ChatInput` komponenter exakt, sedan överlagrade den med en grå skimmeranimation.
*   **Värde**: När React-hydreringen är klar blir skelettet omedelbart den verkliga komponenten och användarna kan knappt uppfatta omkopplaren.

#### 4. Kortstapelskelett
Den simulerar den staplade rörelsen av förslagskort under laddning, med hjälp av 3D-transformationer för att få korten att flyta subtilt.

### C. Abstraktion och konst

> Visa upp HagiCodes nördiga DNA.

#### 5. Geometrisk morf
En geometrisk form (en kvadrat) återges i mitten av skärmen och förvandlas sedan smidigt över tiden till en cirkel, en triangel och slutligen logotypen.
*   **Teknik**: Smidiga övergångar med CSS `border-radius`.

#### 6. Code Rain
En hyllning till *The Matrix*. Med JetBrains Mono-fonten faller svaga strömmar av tecken i bakgrunden.
*   **Obs**: För prestanda måste karaktärsströmmarna stanna inom ett mindre område eller använda en lägre uppdateringsfrekvens.

#### 7. Neonpuls
En glödande ring i cyberpunk-stil som använder flera `box-shadow` lager för att skapa en kraftfull neonglöd.

### D. Varumärke och teman

> Få systemet att kännas levande.

#### 8. Säsongstema
Detta är en dynamisk lastare. Den kontrollerar det aktuella datumet för helgdagar som nyår eller jul och laddar motsvarande SVG-animation.
*   **Exempel**: Under månnyåret svajar röda lyktor försiktigt längst ner på skärmen.

#### 9. Gradientflöde
Bakgrunden använder en flytande gradient baserad på HagiCodes varumärkesfärger. Kombinerat med animerad `background-size` och `background-position`, skapar det en norrskensliknande rörelsekänsla.

### E. Teknisk känsla

> En hälsning till utvecklarna.

#### 10. Terminalstart
Den simulerar konsolutgång. Kodrader rullar snabbt förbi:
```text
> Initializing HagiCode Core...
> Loading models...
> Connecting to neural network...
```
Det känns direkt bekant för varje utvecklare.

#### 11. Förloppsindikator
En tunn förloppsindikator visas högst upp på skärmen, med en procentandel som visas till höger. Även om vi inte kan komma åt det verkliga nedladdningsförloppet kan vi använda en timer för att simulera en "trovärdig" laddningsprocess: snabb under de första 80 %, sedan gradvis långsammare under de sista 20 %.

### F. Kreativitet

#### 12. Pixelmontering
Detta är en mycket intressant idé. Små rutor är utspridda över skärmen, konvergerar sedan mot mitten och samlas gradvis till HagiCode-logotypen. Det symboliserar processen med att bygga kod.

## Bästa praxis och fallgropar

I HagiCodes verkliga utvecklingsarbete sammanfattade vi flera kritiskt viktiga implementeringsdetaljer.

### 1. Defensiv CSS är obligatorisk
Bli aldrig lat och hoppa över prefixet. En gång glömde vi att omfånga stilarna på startskärmen med ett ID och globalt `div` stilar efter att React monterats påverkade oväntat startskärmen och bröt layouten.
**Lärdom**: Placera varje CSS-väljare under `#boot-screen`, och använda `!important` för att höja prioritet när det behövs, men bara inne i startskärmens CSS.

### 2. Graciösa övergångar
Efter att React har monterats framgångsrikt, gör det inte direkt `remove()` startskärmen DOM.
**Rätt tillvägagångssätt**:
1.  Reagera triggers `window.dispatchEvent(new Event('hagicode:ready'))`.
2.  Startskärmen lyssnar efter händelsen och första set `opacity: 0`.
3.  Vänta 300ms, vilket matchar CSS-övergångens varaktighet, och ring `.remove()` först efter att skärmen är helt osynlig.

### 3. Tema Variabel Synkronisering
Färgvärdena för startskärmen är hårdkodade `index.html`. Om vi ​​ändrar Tailwinds primärfärg måste vi också uppdatera startskärmen.
**Optimering**: Skriv ett enkelt plugin i Vite build-skriptet för att läsa `tailwind.config.js` och injicera färgvariabler i `index.html` mallvariabler, vilket skapar en enda källa till sanning.

### 4. Förladdning av teckensnitt
Startskärmar behöver ofta använda ett märkestypsnitt, men om teckensnittet laddas långsamt kan FOUT (Flash of Unstyled Text) dyka upp.
**Lösning**: Lägg till `<link rel="preload" href="/fonts/JetBrainsMono.woff2" as="font" type="font/woff2" crossorigin>` inuti `<head>`. Det här är ett billigt sätt att förbättra upplevelsen med hög avkastning.

### 5. Prestandaövervakning
Vi injicerade `performance.mark('boot-start')` längst ner på `index.html`, och markerad `boot-end` när React har monterats framgångsrikt.
**Varför det spelar roll**: Genom att samla in denna data genom Application Insights kan vi direkt mäta hur mycket startskärmen förkortar upplevd väntetid. Data visar att en utmärkt skelettskärm kan förbättra användarnas tolerans för ett "långsamt nätverk" med mer än 50%.
