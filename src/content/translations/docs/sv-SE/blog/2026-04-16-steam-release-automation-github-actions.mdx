## Hur man automatiserar Steam-släpp med GitHub-åtgärder

> Den här artikeln delar den kompletta lösningen som vi implementerade för automatiserade Steam-utgåvor i HagiCode Desktop-projektet, som täcker automatiseringsflödet från GitHub Release till Steam-plattformen, inklusive viktiga tekniska detaljer som Steam Guard-autentisering och depåuppladdningar på flera plattformar.

## Bakgrund

Utgivningsarbetsflödet på Steam skiljer sig faktiskt ganska mycket från traditionell applikationsdistribution. Steam har sitt eget kompletta uppdateringssystem. Utvecklare måste använda SteamCMD för att ladda upp byggartefakter till Steams CDN-nätverk, snarare än att bara släppa in en nedladdningslänk som på andra plattformar.

HagiCode Desktop-projektet förbereder sig för en Steam-release, som introducerade några nya utmaningar för vårt release-arbetsflöde:

1. Vi behövde konvertera befintliga byggartefakter till ett Steam-kompatibelt format.
2. Vi var tvungna att ladda upp dem till Steam genom SteamCMD.
3. Vi var också tvungna att hantera Steam Guard-autentisering.
4. Vi behövde stödja Depot-uppladdningar med flera plattformar för Linux, Windows och macOS.
5. Vi ville ha ett helt automatiserat flöde från GitHub Release till Steam.

Projektet hade redan implementerat "portabelt versionsläge", vilket gör att applikationen kan upptäcka fasta tjänstnyttolaster paketerade i `extra` katalog. Vårt mål var att integrera det bärbara versionsläget sömlöst med Steam-distribution.

## Om HagiCode

Lösningen som delas i den här artikeln kommer från vår praktiska erfarenhet av [HagiCode](https://hagicode.com) projekt. HagiCode är en AI-kodningsassistent som stöder skrivbordsanvändning. Eftersom vi aktivt arbetar mot lansering på Steam behövde vi upprätta ett tillförlitligt arbetsflöde för automatiserad release.

## Arkitektur design

Kärnan i hela Steam-releaseprocessen är ett GitHub Actions-arbetsflöde som delar upp processen i tre huvudsteg:

```
┌─────────────────────────────────────────────────────────────┐
│ GitHub Actions Workflow (Steam Release)                    │
├─────────────────────────────────────────────────────────────┤
│ 1. Preparation Stage:                                      │
│    - Check out portable-version code                       │
│    - Download build artifacts from GitHub Release          │
│    - Extract and prepare the Steam content directory       │
│                                                             │
│ 2. SteamCMD Setup:                                         │
│    - Install or reuse SteamCMD                             │
│    - Authenticate with Steam Guard                         │
│                                                             │
│ 3. Release Stage:                                          │
│    - Generate Depot VDF configuration files                │
│    - Generate App Build VDF configuration files            │
│    - Invoke SteamCMD to upload to Steam                    │
└─────────────────────────────────────────────────────────────┘
```

Denna design erbjuder flera fördelar:
- Den återanvänder befintliga GitHub Release-artefakter och undviker att bygga om samma utdata.
- Den använder löpare med egen värd för säker isolering.
- Det stöder växling mellan förhandsgranskningsläge och formella utgivningsgrenar.
- Det inkluderar fullständig felhantering och loggning, vilket gör fel lättare att diagnostisera.

## Arbetsflödesimplementering

### Utlösarparameterdesign

Vårt arbetsflöde stöder följande nyckelparametrar:

```yaml
inputs:
  release:           # Portable Version release tag
    description: 'Version tag to release (for example v1.0.0)'
    required: true
  steam_preview:     # Whether to generate a preview build
    description: 'Whether to use preview mode'
    required: false
    default: 'false'
  steam_branch:      # Steam branch to set live
    description: 'Target Steam branch'
    required: false
    default: 'preview'
  steam_description: # Build description override
    description: 'Build description'
    required: false
```

### Self-hosted Runner Configuration

Av säkerhetsskäl använder vi en självvärd löpare med `steam` etikett:

```yaml
runs-on:
  - self-hosted
  - Linux
  - X64
  - steam
```

Detta säkerställer att Steam-släpp körs på en dedikerad löpare och håller känsliga uppgifter säkert isolerade.

### Samtidighetskontroll

För att förhindra att versioner av samma version stör varandra konfigurerade vi samtidighetskontroll:

```yaml
concurrency:
  group: portable-version-steam-${{ github.event.inputs.release }}
  cancel-in-progress: false
```

Lägg märke till det `cancel-in-progress: false` är inställd här eftersom Steam-släpp kan ta ett tag och vi vill inte att en nyutlöst körning avbryter en som redan laddas upp.

## Core Script Implementation

### Förbereda release-ingångar

Den `prepare-steam-release-input.mjs` script är ansvarigt för att förbereda de indata som krävs för release:

```javascript
// Download the GitHub Release build manifest and artifact inventory
const buildManifest = await downloadBuildManifest(releaseTag);
const artifactInventory = await downloadArtifactInventory(releaseTag);

// Download compressed archives for each platform
for (const platform of ['linux-x64', 'win-x64', 'osx-universal']) {
  const artifactUrl = getArtifactUrl(artifactInventory, platform);
  await downloadArtifact(artifactUrl, platform);
}

// Extract into the Steam content directory structure
await extractToSteamContent(sources, contentRoot);
```

### Steam Guard-autentisering

Steam kräver att konton skyddas med Steam Guard, så vi implementerade en delad hemlighetsbaserad kodgenereringsalgoritm:

```javascript
function generateSteamGuardCode(sharedSecret, timestamp = Date.now()) {
  const secret = decodeSharedSecret(sharedSecret);
  const time = Math.floor(timestamp / 1000 / 30);

  const timeBuffer = Buffer.alloc(8);
  timeBuffer.writeBigUInt64BE(BigInt(time));

  // Use HMAC-SHA1 to generate a time-based one-time code
  const hash = crypto.createHmac('sha1', secret)
    .update(timeBuffer)
    .digest();

  // Convert it into a 5-character Steam Guard code
  const code = steamGuardCode(hash);
  return code;
}
```

Denna implementering är baserad på Steam Guards TOTP-mekanism (Time-based One-Time Password) som genererar en ny verifieringskod var 30:e sekund.

### VDF-konfigurationsgenerering

VDF (Valve Data Format) är det konfigurationsformat som används av Steam. Vi behöver generera två typer av VDF-filer:

**Depot VDF** används för att konfigurera innehåll för varje plattform:

```javascript
function buildDepotVdf(depotId, contentRoot) {
  return [
    '"DepotBuildConfig"',
    '{',
    `  "DepotID" "${escapeVdf(depotId)}"`,
    `  "ContentRoot" "${escapeVdf(contentRoot)}"`,
    '  "FileMapping"',
    '  {',
    '    "LocalPath" "*"',
    '    "DepotPath" "."',
    '    "recursive" "1"',
    '  }',
    '}'
  ].join('\n');
}
```

**App Build VDF** används för att konfigurera hela applikationsbygget:

```javascript
function buildAppBuildVdf(appId, depotBuilds, description, setLive) {
  const vdf = [
    '"appbuild"',
    '{',
    `  "appid" "${appId}"`,
    `  "desc" "${escapeVdf(description)}"`,
    `  "contentroot" "${escapeVdf(contentRoot)}"`,
    '  "buildoutput" "build_output"',
    '  "depots"',
    '  {'
  ];

  for (const [depotId, depotVdfPath] of Object.entries(depotBuilds)) {
    vdf.push(`    "${depotId}" "${depotVdfPath}"`);
  }

  if (setLive) {
    vdf.push(`  }`);
    vdf.push(`  "setlive" "${setLive}"`);
  }

  vdf.push('}');
  return vdf.join('\n');
}
```

### SteamCMD-anrop

Slutligen utförs uppladdningen genom att anropa SteamCMD:

```javascript
await runCommand(steamcmdPath, [
  '+login', steamUsername, steamPassword, steamGuardCode,
  '+run_app_build', appBuildPath,
  '+quit'
]);
```

Detta är det sista steget i hela arbetsflödet. När det väl lyckas är släppet klart.

## Flerplattformsdepåhantering

Steam använder Depot-systemet för att hantera innehåll för olika plattformar. Vi stöder tre huvuddepåer:

| Plattform | Depåidentifierare | Arkitekturstöd |
|-----|-----------|---------|
| Linux | `linux-x64` | x64_64 |
| Windows | `win-x64` | x64_64 |
| macOS | `osx-universal` | universell, x64_64, arm64 |

Varje depå har sin egen innehållskatalog och VDF-konfigurationsfil. Detta säkerställer att användare på olika plattformar bara laddar ner det innehåll de faktiskt behöver.

## Släpp arbetsflöde

### Steg 1: Förbered GitHub-versionen

Skapa först en GitHub-version i `portable-version` arkiv som inkluderar:
- Komprimerade arkiv för varje plattform
- Bygg manifest (`{tag}.build-manifest.json`)
- Artefaktinventering (`{tag}.artifact-inventory.json`)

### Steg 2: Aktivera Steam Release Workflow

Utlösa arbetsflödet manuellt genom GitHub Actions och fyll i de nödvändiga parametrarna:
- `release`: versionstaggen som ska publiceras, t.ex `v1.0.0`
- `steam_branch`: målgrenen, såsom `preview` eller `public`
- `steam_preview`: om förhandsgranskningsläget ska användas

### Steg 3: Utför releaseflödet automatiskt

Arbetsflödet utför automatiskt följande steg:

1. Ladda ner och extrahera GitHub Release-artefakter.
2. Installera eller uppdatera SteamCMD.
3. Generera Steam VDF-konfigurationsfiler.
4. Autentisera med Steam Guard.
5. Ladda upp innehåll till Steam CDN.
6. Ställ in den angivna grenen live.

När denna sekvens är klar täcks hela frigöringsvägen.

## Konfigurationsguide

### Obligatorisk hemlighetskonfiguration

Konfigurera följande hemligheter i GitHub-förvarets inställningar:

| Hemligt namn | Beskrivning |
|-----------|------|
| `STEAM_USERNAME` | Användarnamn för Steam-konto |
| `STEAM_PASSWORD` | Lösenord för Steam-konto |
| `STEAM_SHARED_SECRET` | Steam Guard delad hemlighet (valfritt) |
| `STEAM_GUARD_CODE` | Steam Guard-kod (valfritt) |
| `STEAM_APP_ID` | Steam-applikations-ID |
| `STEAM_DEPOT_ID_LINUX` | Linux Depot ID |
| `STEAM_DEPOT_ID_WINDOWS` | Windows Depå ID |
| `STEAM_DEPOT_ID_MACOS` | macOS Depå ID |

Det är inget speciellt ovanligt med dessa inställningar. Du behöver helt enkelt alla förväntade värden på plats.

### Konfiguration av miljövariabel

| Variabelnamn | Beskrivning | Standardvärde |
|---------|------|--------|
| `PORTABLE_VERSION_STEAMCMD_ROOT` | SteamCMD installationskatalog | `~/.local/share/portable-version/steamcmd` |

## Bästa metoder

### Steam Guard Authentication Management

Vid första körningen måste du ange Steam Guard-koden manuellt. Efter det rekommenderas det att konfigurera den delade hemligheten så att koden kan genereras automatiskt. Detta undviker manuella ingrepp vid varje release.

SteamCMD sparar inloggningstoken och kan återanvända den vid senare körningar. Du måste fortfarande hålla ett öga på att token löper ut, eftersom omautentisering krävs efter att token har gått ut.

### Innehållskatalogstruktur

Se till att Steam-innehållskatalogstrukturen är korrekt:

```
steam-content/
├── linux-x64/     # Linux platform content
├── win-x64/       # Windows platform content
└── osx-universal/ # macOS universal binary content
```

Varje katalog bör innehålla de fullständiga programfilerna för dess motsvarande plattform.

### Användning av förhandsgranskningsläge

Förhandsgranskningsläget ställer inte in någon gren live, så det är lämpligt för testning och validering:

```yaml
if [ "$STEAM_PREVIEW_INPUT" = 'true' ]; then
  cmd+=(--preview)
fi
```

Detta låter dig ladda upp till Steam först för verifiering och sedan byta till den formella grenen efter att allt har checkats ut.

### Felhantering och loggning

Skripten inkluderar fullständig felhantering och loggning:
- Bekräfta att GitHub-versionen finns.
- Kontrollera nödvändiga metadatafiler.
- Se till att plattformsinnehåll finns.
- Skapa en sammanfattningsrapport för GitHub Actions.

Denna information är mycket värdefull för både felsökning och granskning.

### Artefakthantering

Arbetsflödet genererar två typer av artefakter:
- `portable-steam-release-preparation-{tag}`: metadata för släppförberedelser
- `portable-steam-build-metadata-{tag}`: Steam build metadata

Dessa artefakter kan användas för senare granskning och felsökning. En lagringstid på 30 dagar är en praktisk standard.
