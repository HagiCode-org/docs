## Bygga en elegant ny användarintroduktion i React-projekt: HagiCodes driver.js-övning

> När användare öppnar din produkt för första gången, vet de verkligen var de ska börja? I den här artikeln vill jag prata lite om hur vi använde driver.js för introduktion av nya användare i HagiCode-projektet. Se det som ett litet praktiskt exempel för att få igång samtalet.

## Bakgrund

Har du någonsin hamnat i den här situationen? En ny användare registrerar sig för din produkt, öppnar sidan och ser omedelbart förlorad ut. De söker runt, osäkra på vad de ska klicka på eller vad de ska göra härnäst. Som utvecklare antar vi ofta att användarna bara kommer att "utforska på egen hand" eftersom, trots allt, mänsklig nyfikenhet är obegränsad. Men verkligheten är annorlunda: de flesta användare lämnar tyst inom några minuter om de inte kan hitta rätt ingångspunkt, som om historien börjar plötsligt och slutar lika naturligt.

Introduktion av nya användare är ett viktigt sätt att lösa detta problem, men att bygga det väl är inte så enkelt. Ett bra onboarding-system behöver:

- Lokalisera sidelement exakt och markera dem
- Stöd introduktionsflöden i flera steg
- Kom ihåg användarens val (slutför eller hoppa över)
- Undvik att påverka sidans prestanda och normal interaktion
- Håll kodstrukturen tydlig och enkel att underhålla

När vi byggde HagiCode stötte vi på samma utmaning. HagiCode är en AI-kodningsassistent, och dess kärnarbetsflöde är ett OpenSpec-arbetsflöde som ser ut så här: "användaren skapar ett förslag -> AI:n genererar en plan -> användaren granskar det -> AI:n exekverar det." För användare som stöter på detta koncept för första gången är arbetsflödet helt nytt, så de behöver gedigen introduktion för att komma igång snabbt. Nya saker tar alltid lite tid att vänja sig vid.

## Om HagiCode

Tillvägagångssättet som delas i den här artikeln kommer från vår praktiska erfarenhet av [HagiCode](https://hagicode.com) projekt. HagiCode är en Claude-baserad AI-kodningsassistent som hjälper utvecklare att slutföra kodningsuppgifter mer effektivt genom OpenSpec-arbetsflödet. Du kan se vår öppen källkod på [GitHub](https://github.com/HagiCode-org/site).

## Varför vi valde driver.js

Under den tekniska utvärderingsfasen tittade vi på flera vanliga onboarding-bibliotek. Var och en hade sina egna styrkor:

- **Intro.js**: Kraftfull, men relativt stor, och stilanpassning är något komplex
- **Shepherd.js**: Väldesignat API, men lite för "tungt" för vårt användningsfall
- **driver.js**: Lätt, kortfattad, intuitiv API och fungerar bra i React-ekosystemet

Till slut valde vi driver.js. Det fanns ingen särskilt dramatisk anledning. Valet berodde främst på dessa överväganden:

1. **Lättvikt**: Kärnbiblioteket är litet och ökar inte paketstorleken nämnvärt
2. **Enkelt API**: Konfigurationen är tydlig och intuitiv, så den är lätt att plocka upp
3. **Flexibel**: Stöder anpassad positionering, styling och interaktionsbeteende
4. **Dynamisk import**: Kan laddas på begäran utan att det påverkar prestandan på första skärmen

Med teknikval finns det sällan ett universellt bästa svar. Vanligtvis finns det bara det alternativ som passar bäst.

## Tekniskt genomförande

### Kärnkonfiguration

driver.js har en mycket intuitiv konfigurationsmodell. Här är kärnkonfigurationen vi använder i HagiCode-projektet:

```typescript
import { driver } from 'driver.js';
import 'driver.js/dist/driver.css';

const newConversationDriver = driver({
  allowClose: true,           // Allow users to close the guide
  animate: true,              // Enable animations
  overlayClickBehavior: 'close', // Close the guide when the overlay is clicked
  disableActiveInteraction: false, // Keep elements interactive
  showProgress: false,        // Do not show the progress bar (we manage progress ourselves)
  steps: guideSteps           // Array of guide steps
});
```

Anledningen bakom dessa inställningar är:

- `allowClose: true` - Respektera användarens val och tvinga inte dem att avsluta guiden
- `disableActiveInteraction: false` - Vissa steg kräver riktiga användaråtgärder, som att skriva in, så interaktion kan inte inaktiveras
- `overlayClickBehavior: 'close'` - Ge användarna ett snabbt sätt att avsluta

### Statsledning

Det är kritiskt att ombordstigningen kvarstår. Vi vill inte starta om guiden varje gång sidan uppdateras, eftersom det blir irriterande snabbt. HagiCode använder `localStorage` för att hantera guidestatus:

```typescript
export type GuideState = 'pending' | 'dismissed' | 'completed';

export interface UserGuideState {
  session: GuideState;
  detailGuides: Record<string, GuideState>;
}

// Read state
export const getUserGuideState = (): UserGuideState => {
  const state = localStorage.getItem('userGuideState');
  return state ? JSON.parse(state) : { session: 'pending', detailGuides: {} };
};

// Update state
export const setUserGuideState = (state: UserGuideState) => {
  localStorage.setItem('userGuideState', JSON.stringify(state));
};
```

Vi definierar tre tillstånd:

- `pending`: Guiden pågår fortfarande och användaren har inte slutfört eller hoppat över den
- `dismissed`: Användaren stängde guiden proaktivt
- `completed`: Användaren slutförde alla steg

För introduktion av förslagsdetaljsidor stöder vi också mer finkornig tillståndsspårning genom `detailGuides` karta, eftersom ett förslag kan gå igenom flera stadier - utkast, granskning och genomförande komplett - och varje steg behöver olika vägledning. Sakernas tillstånd förändras alltid, och onboarding bör återspegla det.

### Val av målelement

driver.js använder CSS-väljare för att hitta målelement. HagiCode följer en enkel konvention: använd en anpassad `data-guide` attribut för att markera onboarding-mål:

```typescript
const steps = [
  {
    element: '[data-guide="launch"]',
    popover: {
      title: 'Start a New Conversation',
      description: 'Click here to create a new conversation session...'
    }
  }
];
```

I komponenter ser det ut så här:

```typescript
<button data-guide="launch" onClick={handleLaunch}>
  New Conversation
</button>
```

Fördelarna med detta tillvägagångssätt är:

- Undvik konflikter med företagsstylingklassnamn
- Tydlig semantik, så att du omedelbart kan se att elementet är relaterat till onboarding
- Lättare att hantera och underhålla konsekvent

### Dynamisk importoptimering

Eftersom introduktionsfunktionen endast behövs i specifika scenarier, till exempel en användares första besök, använder vi dynamiska importer för att optimera den initiala laddningsprestandan:

```typescript
const initNewUserGuide = async () => {
  // Dynamically import driver.js
  const { driver } = await import('driver.js');
  await import('driver.js/dist/driver.css');

  // Initialize the guide
  const newConversationDriver = driver({
    // ...configuration
  });

  newConversationDriver.drive();
};
```

På så sätt laddas driver.js och dess stilmall bara när det behövs och påverkar inte prestandan på första skärmen. Det är inte många som tycker om att vänta på något de inte ens behöver ännu.

## Onboarding Flow Design

HagiCode implementerar två onboarding-vägar som täcker användarens kärnscenarier.

### Session Onboarding (10 steg)

Denna introduktionsväg hjälper användare att slutföra hela flödet från att skapa en konversation till att skicka in sitt första fullständiga förslag:

1. **lansering** - Starta guiden och introducera knappen "Ny konversation".
2. **skriv** - Guide användaren att skriva en begäran i inmatningsrutan
3. **skicka** - Guide användaren att klicka på skicka-knappen
4. **proposal-launch-readme** - Guide användaren att skapa ett README-förslag
5. **proposal-compose-readme** - Guide användaren att redigera README-förfrågans innehåll
6. **proposal-submit-readme** - Guide användaren att skicka README-förslaget
7. **proposal-launch-agents** - Guide användaren att skapa en `AGENTS.md` Förslag
8. **proposal-compose-agents** - Guide användaren att redigera `AGENTS.md` begäran
9. **proposal-submit-agents** - Guide användaren att skicka in `AGENTS.md` Förslag
10. **proposal-wait** - Förklara att AI:n bearbetas och att användaren bör vänta ett ögonblick

Tanken bakom den här vägen är att låta användare uppleva HagiCodes kärnarbetsflöde på egen hand genom två verkliga uppgifter för att skapa förslag, en för `README` och en för `AGENTS.md`. Det är stor skillnad på att höra om ett arbetsflöde och att gå igenom det själv.

### Introduktion av förslagsdetaljer (3 steg)

När användare går in på förslagsdetaljsidan utlöser HagiCode motsvarande guide baserat på förslagets aktuella status:

1. **utkast** (utkaststadium) - Guide användaren att granska den AI-genererade planen
2. **granskning** (granskningsstadiet) - Guide användaren att verkställa planen
3. **executionCompleted** (slutfört stadium) - Guide användaren att arkivera planen

Det avgörande kännetecknet för denna guide är att den är tillståndsstyrd: den bestämmer dynamiskt vilket introduktionssteg som ska visas baserat på förslagets faktiska tillstånd. Saker och ting förändras, och onboarding borde förändras med dem.

### Element Render Försök Mekanism igen

I React-applikationer kan det hända att målonboarding-elementet inte har slutförts renderingen ännu, till exempel eftersom asynkron data fortfarande laddas. För att hantera detta implementerar HagiCode en försöksmekanism:

```typescript
const waitForElement = (selector: string, maxRetries = 10, interval = 100) => {
  let retries = 0;
  return new Promise<HTMLElement>((resolve, reject) => {
    const checkElement = () => {
      const element = document.querySelector(selector) as HTMLElement;
      if (element) {
        resolve(element);
      } else if (retries < maxRetries) {
        retries++;
        setTimeout(checkElement, interval);
      } else {
        reject(new Error(`Element not found: ${selector}`));
      }
    };
    checkElement();
  });
};
```

Anropa den här funktionen innan du initierar guiden för att se till att målelementet redan finns. Ibland är det värt det att vänta lite längre.

## Sammanfattning av bästa praxis

Baserat på HagiCodes praktiska erfarenhet, här är några viktiga bästa praxis:

### 1. Onboarding bör vara lätt att avsluta

Tvinga inte användare att slutföra onboarding. Vissa användare är upptäcktsresande till sin natur och föredrar att lista ut saker på egen hand. Ange en tydlig "Hoppa över"-knapp, kom ihåg deras val och avbryt dem inte igen nästa gång.

### 2. Håll introduktionsinnehållet kort och skarpt

Varje introduktionssteg bör fokusera på ett enda mål:

- **Titel**: Kort och tydlig, helst inte mer än några få ord
- **Beskrivning**: Gå direkt till saken och berätta för användaren vad detta är och varför det är viktigt

Undvik långrandiga förklaringar. Användarens uppmärksamhet är begränsad under onboarding, och ju mer du säger, desto mindre sannolikt är det att de läser det.

### 3. Håll väljare stabila

Använd stabila elementmarkörer som inte ändras ofta. Seden `data-guide` attribut är ett bra val. Undvik beroende på klassnamn eller DOM-struktur, eftersom de är lätta att ändra under refactors. Koden ändras hela tiden, men vissa ankare bör förbli stabila när det är möjligt.

### 4. Testa din onboarding

HagiCode inkluderar kompletta testfall för onboarding-funktionen:

```typescript
describe('NewUserConversationGuide', () => {
  it('should initialize guide state correctly', () => {
    const state = getUserGuideState();
    expect(state.session).toBe('pending');
  });

  it('should update guide state correctly', () => {
    setUserGuideState({ session: 'completed', detailGuides: {} });
    const state = getUserGuideState();
    expect(state.session).toBe('completed');
  });
});
```

Tester hjälper till att säkerställa att refaktorering inte av misstag bryter onboardingbeteendet. Ingen vill ha en liten kodändring för att tyst skada tidigare fungerande funktionalitet.

### 5. Prestandaoptimering

- Använd dynamisk import för att lata ladda introduktionsbiblioteket
- Undvik att initiera onboarding-logik efter att användaren redan har slutfört guiden
- Tänk på prestandaeffekten av animationer och inaktivera dem på lägre enheter om det behövs

Prestanda, som många saker i livet, förtjänar lite noggrann budgetering.

## Slutsats

Introduktion av nya användare är en viktig del av att förbättra produktens användarupplevelse. I HagiCode-projektet använde vi driver.js för att bygga ett komplett onboarding-system som täcker hela arbetsflödet från skapande av sessioner till exekvering av förslag.

Kärnpunkterna vi hoppas kunna dela med oss av i den här artikeln är:

1. **Tekniska val bör matcha faktiska behov**: driver.js är inte det mest kraftfulla alternativet, men det passar oss bäst
2. **Statsledning är avgörande**: Användning `localStorage` för att bevara onboarding-tillståndet och undvika att upprepade gånger avbryta användare
3. **Onboarding-design bör vara fokuserad**: Varje steg ska lösa ett problem och inte mer
4. **Kodstrukturen bör förbli tydlig**: Separat onboarding-konfiguration, tillståndshantering och UI-logik för att göra underhållet enklare

Om du lägger till nya användare i ditt eget projekt hoppas jag att den praktiska erfarenheten i den här artikeln hjälper. Det finns inget särskilt mystiskt med denna typ av teknik. Fortsätt att försöka, fortsätt sammanfatta vad du lär dig, så blir saker gradvis lättare.
