## Fullständigt GLM-5.1-stöd och Gemini CLI-integration: HagiCodes väg för multimodellutveckling

> Den här artikeln introducerar två stora senaste uppdateringar av HagiCode-plattformen: fullt stöd för Zhipu AI GLM-5.1-modellen och den framgångsrika integrationen av Gemini CLI som den tionde Agent CLI. Tillsammans stärker dessa uppdateringar ytterligare plattformens multi-modell kapacitet och multi-CLI ekosystem.

:::note[Time Context Notice]
Den här artikeln registrerar utvecklingen av multi-CLI-expansionen som den såg ut vid tidpunkten för utgivningen. För den aktuella omfattningen som alltid stöds, se [AI Agent CLI-installation](/sv-SE/related-software-installation/ai-agent-cli/) och den [Installationsguide](/sv-SE/installation); den `IFlowCli` som nämns här representerar endast en historisk konfiguration vid den tidpunkten och är inte en del av den nuvarande aktiva supporten.
:::

## Bakgrund

Tiden går verkligen fort. Utvecklingen av stora språkmodeller har stigit som bambu under våren. För inte så länge sedan hejade vi fortfarande på "en AI som kan skriva kod." Nu är vi redan inne i en era av multimodellsamarbete och multiverktygsintegration. Är det spännande? Kanske. När allt kommer omkring, vad utvecklare behöver har aldrig bara varit själva verktyget, utan lättheten att anpassa sig till olika scenarier och växla flexibelt när det behövs.

Som en AI-assisterad kodningsplattform har HagiCode nyligen välkomnat två viktiga utvecklingar: för det första den fullständiga integrationen av Zhipu AI:s GLM-5.1-modell; för det andra, det officiella tillägget av Gemini CLI som den tionde agent CLI som stöds. Dessa två uppdateringar låter kanske inte omvälvande, men de är utan tvekan goda nyheter för plattformens fortsatta mognad.

GLM-5.1 är Zhipu AI:s senaste flaggskeppsmodell. Jämfört med GLM-5.0 erbjuder den starkare resonemang, djupare kodförståelse och smidigare verktygsanrop. Ännu viktigare är att det är den första GLM-modellen som stöder bildinmatning. Vad betyder det? Det betyder att användare kan låta AI titta direkt på en skärmdump istället för att kämpa för att beskriva problemet med ord. När du har använt den bekvämligheten förstår du omedelbart dess värde.

Samtidigt, genom `HagiCode.Libs.Providers` arkitektur, integrerade HagiCode framgångsrikt Gemini CLI i plattformen. Detta är nu den tionde Agent CLI. För att vara ärlig, att komma till denna punkt ger en blygsam känsla av prestation.

Det är också värt att nämna att HagiCodes bilduppladdningsfunktion låter användare kommunicera med AI direkt genom skärmdumpar. Även när man kör GLM 4.7 fungerar plattformen fortfarande bra och har redan hjälpt till att slutföra många viktiga bygguppgifter. När det gäller GLM-5.1 går det naturligtvis ett steg längre.

## Om HagiCode

Tillvägagångssättet som delas i den här artikeln kommer från vår praktiska erfarenhet av [HagiCode](https://hagicode.com) projekt. HagiCode är en AI-assisterad kodningsplattform med öppen källkod designad för att förse utvecklare med en flexibel och kraftfull AI-programmeringsassistent genom en multi-modell, multi-CLI-arkitektur. Projektförråd: [github.com/HagiCode-org/site](https://github.com/HagiCode-org/site)

## Multi-CLI-arkitekturdesign

En av HagiCodes kärnstyrkor är dess stöd för flera AI-programmerings-CLI-verktyg genom ett enhetligt abstraktionslager. Fördelen med denna design är faktiskt ganska enkel: nya verktyg kan komma in, gamla verktyg kan stanna och kodbasen förvandlas inte till kaos. För att vara rättvis, det är så alla vill att livet ska fungera.

### AIProviderType enum

Plattformen definierar CLI-leverantörstyper som stöds genom `AIProviderType` uppräkning:

```csharp
public enum AIProviderType
{
    ClaudeCodeCli = 0,    // Claude Code CLI
    CodexCli = 1,          // GitHub Copilot Codex
    GitHubCopilot = 2,     // GitHub Copilot
    CodebuddyCli = 3,     // Codebuddy CLI
    OpenCodeCli = 4,      // OpenCode CLI
    IFlowCli = 5,         // IFlow CLI
    HermesCli = 6,        // Hermes CLI
    QoderCli = 7,         // Qoder CLI
    KiroCli = 8,          // Kiro CLI
    KimiCli = 9,          // Kimi CLI
    GeminiCli = 10,       // Gemini CLI (new)
}
```

Som du kan se ansluter sig Gemini CLI till denna familj som den tionde medlemmen. Varje CLI har sina egna distinkta egenskaper och användningsscenarier, så att användare kan välja flexibelt utifrån sina behov. Trots allt leder många vägar till Rom; vissa är helt enkelt lättare än andra.

### Leverantörsarkitektur

`HagiCode.Libs.Providers` tillhandahåller ett enhetligt leverantörsgränssnitt som gör varje CLI-integration standardiserad och koncis. Ta Gemini CLI som ett exempel:

```csharp
public class GeminiProvider : ICliProvider<GeminiOptions>
{
    private static readonly string[] DefaultExecutableCandidates = ["gemini", "gemini-cli"];
    private const string ManagedBootstrapArgument = "--acp";

    public string Name => "gemini";
    public bool IsAvailable => _executableResolver.ResolveFirstAvailablePath(DefaultExecutableCandidates) is not null;
}
```

Fördelarna med denna design är:
- Att integrera en ny CLI kräver bara implementering av en leverantörsklass
- Unified lifecycle management och session pooling
- Automatiserad aliasupplösning och körbar upptäckt

Enkelt uttryckt, denna design förvandlar komplicerade saker till enklare och gör livet lite enklare.

### Leverantörsregistret

Providerregistret hanterar automatiskt aliasmapping och registrering:

```csharp
if (provider is GeminiProvider)
{
    registry.Register(provider.Name, provider, ["gemini-cli"]);
    continue;
}
```

Detta innebär att användare kan anropa Gemini CLI med endera `gemini` eller `gemini-cli`, och systemet kommer att känna igen det automatiskt. Det är som en vän med både ett formellt namn och ett smeknamn - hur som helst, folk vet vem du menar.

## GLM-5.1 modellstöd

GLM-5.1 är Zhipu AI:s senaste flaggskeppsmodell, och HagiCode har fullbordat fullt stöd för den.

### Katalog för sekundära yrken

HagiCode hanterar alla modeller som stöds genom Secondary Professions Catalog. Här är konfigurationen för GLM-serien:

| Modell ID | Namn | SupportsImage | Kompatibla CLI-familjer |
|----------|------|----------------|-------------------------|
| `glm-4.7` | GLM 4.7 | - | claude, codebuddy, hermes, qoder, kiro |
| `glm-5` | GLM 5 | - | claude, codebuddy, hermes, qoder, kiro |
| `glm-5-turbo` | GLM 5 Turbo | - | claude, codebuddy, hermes, qoder, kiro |
| `glm-5.0` | GLM 5.0 (äldre) | - | claude, codebuddy, hermes, qoder, kiro |
| `glm-5.1` | GLM 5.1 | **sant** | claude, codebuddy, hermes, qoder, kiro |

De viktigaste egenskaperna hos GLM-5.1 kan sammanfattas enligt följande:
- En fristående versionsidentifierare utan äldre bagage
- Den första GLM-modellen som stöder bildinmatning
- Starkare resonemang och kodförståelse
- Bred multi-CLI-kompatibilitet

### GLM-5.1 vs GLM-5.0

På kodnivån visas nyckelskillnaden mellan GLM-5.1 och GLM-5.0 här:

```csharp
// GLM-5.0 (Legacy) - contains special retention logic
private const string Glm50CodebuddySecondaryProfessionId = "secondary-glm-5-codebuddy";
private const string Glm50CodebuddyModelValue = "glm-5.0";

// GLM-5.1 - standalone new model identifier
private const string Glm51SecondaryProfessionId = "secondary-glm-5-1";
private const string Glm51ModelValue = "glm-5.1";
```

GLM-5.0 bär etiketten "Legacy" eftersom det är en gammal versionsidentifierare som behålls för bakåtkompatibilitet. GLM-5.1, däremot, är en helt ny fristående version utan historisk börda. Vissa saker stannar i det förflutna; andra reser lättare och rör sig snabbare.

### Konfigurera GLM-5.1

Här är ett konfigurationsexempel för att använda GLM-5.1 i HagiCode:

```json
{
  "primaryProfessionId": "profession-claude-code",
  "secondaryProfessionId": "secondary-glm-5-1",
  "model": "glm-5.1",
  "reasoning": "high"
}
```

## Bilduppladdningsfunktion

HagiCodes bildstöd implementeras genom `SupportsImage` egendom på `SecondaryProfession`:

```csharp
public class HeroSecondaryProfessionSettingDto
{
    public bool SupportsImage { get; set; }
}
```

I Secondary Professions Catalog ser GLM-5.1-konfigurationen ut så här:

```json
{
  "id": "secondary-glm-5-1",
  "supportsImage": true
}
```

Detta innebär att användare kan ladda upp skärmdumpar direkt för AI-analys, till exempel:
- Skärmdumpar av felmeddelanden
- Problem i en UI-skärm
- Datavisualiseringsdiagram
- Kodexekveringsresultat

Det finns inte längre något behov av att beskriva allt manuellt - ladda bara upp skärmdumpen. Bekvämligheten med denna funktion är uppenbar när du har använt den. Ibland säger en blick mer än en lång förklaring.

## Gemini CLI-integration

Som den tionde Agent CLI är Gemini CLI integrerad i HagiCode genom standardleverantörsarkitekturen.

### Konfigurationsalternativ

Gemini CLI stöder en rik uppsättning konfigurationsalternativ:

```csharp
public class GeminiOptions
{
    public string? ExecutablePath { get; set; }
    public string? WorkingDirectory { get; set; }
    public string? SessionId { get; set; }
    public string? Model { get; set; }
    public string? AuthenticationMethod { get; set; }
    public string? AuthenticationToken { get; set; }
    public Dictionary<string, string?> AuthenticationInfo { get; set; }
    public Dictionary<string, string?> EnvironmentVariables { get; set; }
    public string[] ExtraArguments { get; set; }
    public TimeSpan? StartupTimeout { get; set; }
    public CliPoolSettings? PoolSettings { get; set; }
}
```

Dessa alternativ täcker allt från grundläggande inställningar till avancerade funktioner, vilket ger användarna flexibiliteten att konfigurera CLI efter sina egna behov. Allas arbetsflöde är olika, så lite flexibilitet är alltid välkommet.

### ACP kommunikationsprotokoll

Gemini CLI stöder ACP (Agent Communication Protocol), som är HagiCodes enhetliga CLI-kommunikationsstandard. Genom ACP kan olika CLI:er interagera med plattformen på ett konsekvent sätt, vilket avsevärt förenklar integrationsarbetet. Kort sagt, det standardiserar de komplicerade delarna så att alla kan arbeta lättare.

## HagiCodes egen byggpraxis

På tal om verklig praxis, det bästa exemplet är HagiCode-plattformens eget byggarbetsflöde. HagiCodes utvecklingsprocess har redan utnyttjat AI-kapaciteten fullt ut.

### Fungerar bra även med GLM 4.7

HagiCodes plattformsdesign är väl optimerad, så den kan fortfarande ge en bra utvecklingsupplevelse även med GLM 4.7. Plattformen har redan hjälpt till att slutföra flera viktiga byggprojekt, inklusive:
- Integration av flera CLI-leverantörer
- Implementering av bilduppladdningsfunktionen
- Dokumentationsgenerering och innehållspublicering

Det är faktiskt en bra sak. Alla behöver inte det nyaste hela tiden. Det som passar dig bäst är ofta det som är viktigast.

### GLM-5.1 ger mer med mindre ansträngning

Efter att ha uppgraderat till GLM-5.1 blir dessa funktioner ännu starkare:
- Starkare kodförståelse, vilket minskar kommunikation fram och tillbaka
- Mer exakt beroendeanalys, pekar i rätt riktning direkt
- Effektivare feldiagnos, lokalisering av problem snabbare
- Stöd för bildinmatning, snabbare problembeskrivningar

Det är som att byta från en cykel till en bil. Du kan fortfarande nå samma destination, men hastigheten och komforten är inte densamma.

## Bästa metoder för integration med flera CLI

`HagiCode.Libs.Providers` tillhandahåller en enhetlig mekanism för registrering och användning:

```csharp
services.AddHagiCodeLibs();

var gemini = serviceProvider.GetRequiredService<ICliProvider<GeminiOptions>>();
var codebuddy = serviceProvider.GetRequiredService<ICliProvider<CodebuddyOptions>>();
var hermes = serviceProvider.GetRequiredService<ICliProvider<HermesOptions>>();
```

Denna beroendeinjektionsdesign håller användningen över olika CLI:er mycket kortfattad och gör också enhetstester och hån mer bekvämt. Ren kod är ett sätt att ta ansvar för sig själv.

### Anteckningar

Det finns några saker att tänka på vid faktisk användning:

1. **API-nyckelkonfiguration**: Se till `ANTHROPIC_AUTH_TOKEN` är korrekt inställd eller så kan modellen inte anropas
2. **Modelltillgänglighet**: GLM-5.1 måste aktiveras av motsvarande modellleverantör
3. **Bildfunktion**: Endast modeller med `supportsImage: true` kan använda bilduppladdning
4. **CLI-installation**: Innan du använder Gemini CLI, se till `gemini` eller `gemini-cli` finns i systemet PATH

Det kan vara små detaljer, men små detaljer som hanteras dåligt kan bli stora problem, så de är värda att uppmärksamma.

## Slutsats

Med fullt stöd för GLM-5.1 och den framgångsrika integrationen av Gemini CLI, stärker HagiCode ytterligare sina möjligheter som en multi-modell, multi-CLI AI-programmeringsplattform. Dessa uppdateringar ger inte bara användarna fler valmöjligheter, utan visar också HagiCodes framtidsinriktade arkitektur och skalbarhet.

GLM-5.1:s bildstöd, i kombination med HagiCodes uppladdningsfunktion för skärmdumpar, gör det möjligt att låta AI:n "förstå från bilden" och minskar kostnaden avsevärt för att beskriva problem. Och med stöd för tio CLI:er kan användare flexibelt välja den AI-programmeringsassistent som bäst passar deras preferenser och scenarier. Fler valmöjligheter är nästan alltid bra.

Viktigast av allt, HagiCodes egen byggpraxis bevisar att plattformen redan kan köra bra och slutföra komplexa uppgifter även med GLM 4.7, medan uppgradering till GLM-5.1 kan ytterligare förbättra utvecklingseffektiviteten. Livet är ofta också så: man behöver inte alltid det absolut bästa, bara det som passar en själv. Självklart, om det som passar dig kan bli ännu bättre, så desto bättre.

Om du är intresserad av en multi-modell, multi-CLI AI programmeringsplattform, ge HagiCode ett försök - öppen källkod, gratis och fortfarande under utveckling. Att prova det kostar ingenting, och det kan visa sig vara precis vad du behöver.
