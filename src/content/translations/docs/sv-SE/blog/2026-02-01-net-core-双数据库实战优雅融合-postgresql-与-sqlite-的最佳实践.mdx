## .NET Core Dual-Database i praktiken: Låt PostgreSQL och SQLite samexistera fredligt

> När vi bygger moderna applikationer står vi ofta inför denna avvägning: utvecklingsmiljöer vill ha något lätt och bekvämt, medan produktionsmiljöer kräver hög samtidighet och hög tillgänglighet. Den här artikeln delar med sig av hur man elegant stödjer både PostgreSQL och SQLite i ett .NET Core-projekt och implementerar bästa praxis för "SQLite för utveckling, PostgreSQL för produktion."


## Bakgrund

Inom mjukvaruutveckling har skillnader mellan miljöer alltid varit ett av de svåraste problemen för ingenjörsteam. Ta **HagiCode**-plattformen som vi bygger som ett exempel: det är ett AI-assisterat utvecklingssystem baserat på ASP.NET Core 10 och React, med Orleans integrerad internt för distribuerad tillståndshantering. Stacken är modern och ganska sofistikerad.

Tidigt i projektet stötte vi på en klassisk teknisk smärtpunkt: utvecklarna ville att den lokala miljön skulle fungera direkt utan att behöva installera och konfigurera en tung PostgreSQL-databas. Men i produktionen behövde vi hantera skrivningar med hög samtidighet och komplexa JSON-frågor, och det är precis där lättvikts SQLite börjar visa sina gränser.

Hur kan vi behålla en enda kodbas samtidigt som vi låter applikationen dra nytta av SQLites portabilitet som en stationär app, och även utnyttja PostgreSQL:s kraftfulla prestanda som en tjänst av företagsklass? Det är kärnfrågan som denna artikel undersöker.

## Kärnämne 1: Arkitekturdesign och enhetlig abstraktion

För att stödja två databaser i .NET Core är nyckelidén att vara beroende av abstraktioner snarare än konkreta implementeringar. Vi måste separera databasval från affärskod och låta konfigurationslagret bestämma.

### Design tillvägagångssätt

1.  **Enhetligt gränssnitt**: All affärslogik bör bero på `DbContext` basklass eller anpassade gränssnitt, snarare än ett specifikt `PostgreSqlDbContext`.
2.  **Konfigurationsdriven**: Använd konfigurationsobjekt i `appsettings.json` för att dynamiskt bestämma vilken databasleverantör som ska laddas vid applikationsstart.
3.  **Funktionsisolering**: Lägg till anpassningslogik för PostgreSQL-specifika funktioner, såsom JSONB, så att applikationen fortfarande kan försämras på ett elegant sätt på SQLite.

### Kodimplementering: Dynamisk kontextkonfiguration

I ASP.NET Core's `Program.cs`, vi bör inte hårdkoda `UseNpgsql` eller `UseSqlite`. Istället bör vi läsa konfigurationen och bestämma oss dynamiskt.

Definiera först konfigurationsklassen:

```csharp
public class DatabaseSettings
{
    public const string SectionName = "Database";
    
    // Database type: PostgreSQL or SQLite
    public string DbType { get; set; } = "PostgreSQL"; 
    
    // Connection string
    public string ConnectionString { get; set; } = string.Empty;
}
```

Registrera sedan tjänsten i `Program.cs` baserat på konfiguration:

```csharp
// Read configuration
var databaseSettings = builder.Configuration.GetSection(DatabaseSettings.SectionName).Get<DatabaseSettings>();

// Register DbContext
builder.Services.AddDbContext<ApplicationDbContext>(options =>
{
    if (databaseSettings?.DbType?.ToLower() == "sqlite")
    {
        // SQLite configuration
        options.UseSqlite(databaseSettings.ConnectionString);
        
        // Handling SQLite's concurrent write limitations
        // Note: in production, enabling WAL mode is recommended to improve concurrency performance
    }
    else
    {
        // PostgreSQL configuration (default)
        options.UseNpgsql(databaseSettings.ConnectionString, npgsqlOptions =>
        {
            // Enable JSONB support, which is very useful when handling AI conversation records
            npgsqlOptions.UseJsonNet(); 
        });
        
        // Configure connection pool retry policy
        options.EnableRetryOnFailure(3);
    }
});
```

## Kärnämne 2: Hantering av skillnader och migrationsstrategi

Även om PostgreSQL och SQLite båda stöder SQL-standarden, skiljer de sig markant i specifika möjligheter och beteende. Om dessa skillnader inte hanteras försiktigt kan du lätt hamna i den besvärliga situationen där allt fungerar lokalt men misslyckas efter utplacering.

### 1. Hantera JSON-typer

I HagiCode behöver vi lagra en stor mängd prompter och AI-metadata, vilket vanligtvis involverar JSON-kolumner.
- **PostgreSQL**: Har en infödd `JSONB` typ med utmärkt frågeprestanda.
- **SQLite**: Har inte en inbyggd JSON-typ (nyare versioner inkluderar tillägget JSON1, men objektmappningen skiljer sig fortfarande), så data lagras vanligtvis som TEXT.

**Lösning**:
I EF Core-entitetsmapping konfigurerar vi den som en konvertibel typ.

```csharp
protected override void OnModelCreating(ModelBuilder modelBuilder)
{
    base.OnModelCreating(modelBuilder);

    // Configure entity
    modelBuilder.Entity<PromptTemplate>(entity => 
    {
        entity.Property(e => e.Metadata)
              .HasColumnType("jsonb") // PG uses jsonb
              .HasConversion(
                  v => JsonSerializer.Serialize(v, (JsonSerializerOptions)null),
                  v => JsonSerializer.Deserialize<Dictionary<string, object>>(v, (JsonSerializerOptions)null)
              ); 
    });
}
```

När SQLite används, `HasColumnType("jsonb")` kan ignoreras eller utlösa en varning. Men eftersom `HasConversion` är konfigurerad, är data fortfarande serialiserad och deserialiserad korrekt som strängar lagrade i ett TEXT-fält, vilket säkerställer kompatibilitet.

### 2. Separera migrationsstrategier

Försök aldrig att få en uppsättning migreringsskript att fungera för både PostgreSQL och SQLite samtidigt. Skillnader i primärnyckelgenereringsstrategier, indexsyntax och andra databasdetaljer kommer oundvikligen att orsaka misslyckanden.

**Rekommenderad praxis**:
Underhåll två migrationsgrenar eller projekt. I HagiCodes utvecklingsarbetsflöde är det så här vi hanterar det:

1.  **Utvecklingsskede**: Arbeta främst med SQLite. Använd `Add-Migration Init_Sqlite -OutputDir Migrations/Sqlite`.
2.  **Adaptationsfas**: Efter att ha utvecklat en funktion byter du anslutningssträngen till PostgreSQL och kör `Add-Migration Init_Postgres -OutputDir Migrations/Postgres`.
3.  **Automatisk skript**: Skriv ett enkelt PowerShell- eller Bash-skript för att automatiskt tillämpa rätt migrering baserat på den aktuella miljövariabeln.

```bash
# Pseudocode for simple deployment logic
if [ "$DATABASE_PROVIDER" = "PostgreSQL" ]; then
    dotnet ef database update --project Migrations.Postgres
else
    dotnet ef database update --project Migrations.Sqlite
fi
```

## Kärnämne 3: Lärdomar från HagiCode i produktion

Medan vi ändrade **HagiCode** från en enkeldatabasmodell till stöd för dubbla databaser träffade vi några fallgropar och samlade några viktiga lärdomar som kan hjälpa dig att undvika samma misstag.

### 1. Skillnader i samtidighet och transaktioner

PostgreSQL använder en server-klientarkitektur och stöder skrivningar med hög samtidighet med kraftfulla transaktionsisoleringsnivåer. SQLite använder fillåsning, så skrivoperationer låser hela databasfilen om inte WAL-läget är aktiverat.

**Rekommendation**:
När du skriver affärslogik som involverar frekventa skrivningar, såsom realtidssparande av en användares redigeringstillstånd, måste du ta hänsyn till SQLites låsningsmodell. När vi designade OpenSpec-samarbetsmodulen i **HagiCode**, introducerade vi en "merge before write"-mekanism för att minska frekvensen av direkta databasskrivningar, vilket gör att vi kan bibehålla god prestanda på båda databaserna.

### 2. Livscykelhantering av anslutningssträngar

Att etablera en PostgreSQL-anslutning är relativt dyrt och beror på anslutningspooling. SQLite-anslutningar är mycket lätta, men om de inte släpps omedelbart kan fillåsning göra att senare operationer tar timeout.

In `Program.cs`, kan vi finjustera beteendet för varje databas:

```csharp
if (databaseSettings?.DbType?.ToLower() == "sqlite")
{
    // SQLite: keeping connections open can improve performance, but watch out for file locks
    options.UseSqlite(connectionString, sqliteOptions =>
    {
        // Set command timeout
        sqliteOptions.CommandTimeout(30);
    });
}
else
{
    // PG: make full use of connection pooling
    options.UseNpgsql(connectionString, npgsqlOptions =>
    {
        npgsqlOptions.MaxBatchSize(100);
        npgsqlOptions.CommandTimeout(30);
    });
}
```

### 3. Vikten av testtäckning

Många utvecklare, inklusive några tidiga medlemmar i vårt team, tenderar att göra ett vanligt misstag: köra enhetstester endast i utvecklingsmiljön, som vanligtvis är SQLite.

I HagiCodes CI/CD-pipeline genomförde vi ett GitHub Actions-steg för att säkerställa att varje pull-förfrågan körs PostgreSQL-integreringstest.

```yaml
# Example snippet from .github/workflows/test.yml
- name: Run Integration Tests (PostgreSQL)
  run: |
    docker-compose up -d db_postgres
    dotnet test --filter "Category=Integration"
```

Detta hjälpte oss att fånga otaliga buggar relaterade till SQL-syntaxskillnader och skiftlägeskänslighet.

## Slutsats

Genom att introducera ett abstraktionslager och konfigurationsdriven beroendeinjektion, implementerade vi framgångsrikt en dubbelspårig PostgreSQL- och SQLite-setup i **HagiCode**-projektet. Detta sänkte inte bara introduktionsbarriären avsevärt för nya utvecklare genom att ta bort behovet av att installera PostgreSQL, utan gav också starka prestandagarantier för produktionen.

För att sammanfatta nyckelpunkterna:
1.  **Abstraktion först**: Affärskod bör inte bero på konkreta databasimplementeringar.
2.  **Separat konfiguration**: Använd olika `appsettings.json` filer för utveckling och produktion.
3.  **Separata migreringar**: Försök inte att få en migreringsuppsättning att fungera överallt.
4.  **Funktionsförsämring**: Prioritera kompatibilitet i SQLite och prestanda i PostgreSQL.

Detta arkitektoniska mönster är inte bara lämpligt för HagiCode, utan för alla .NET-projekt som behöver hitta en balans mellan lättviktsutveckling och tungviktsproduktion.
