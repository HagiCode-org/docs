## Hur man bygger en distributionsstation för snabb nedladdning med lågkostnadsmolnservrar

> Molnlagringsbandbredd är absurt dyrt, gränsöverskridande åtkomst är smärtsamt långsam, och CDN-prissättning är tillräckligt för att skrämma bort vem som helst... Om du hanterar fildistribution känner du förmodligen till dessa problem väl. I det här inlägget vill jag dela med mig av ett billigt tillvägagångssätt som vi arbetade fram medan vi byggde [HagiCode](https://hagicode.com): en molnserver plus ett Nginx-cachelager. Kostnaden sjönk med ungefär hälften, medan nedladdningshastigheten förbättrades ganska mycket, vilket åtminstone var lite tröstande.

## Bakgrund

När det kommer till internet är nedladdningshastighet och stabilitet verkligen en del av användarupplevelsen. Oavsett om du driver ett projekt med öppen källkod eller en kommersiell produkt, måste du fortfarande ge användarna ett tillförlitligt sätt att ladda ner filer.

Att ladda ner filer direkt från molnlagring, som Azure Blob Storage eller AWS S3, ser enkelt ut, men det kommer med en hel del praktiska problem:

**Nätverkslatens**: Åtkomst över gränser och över regioner kan vara tillräckligt långsam för att du ska vilja krossa tangentbordet. Om användare måste vänta i evighet kommer upplevelsen uppenbarligen inte att bli fantastisk.

**Bandbreddskostnad**: Utgående trafik för molnlagring är smärtsamt dyrt. Att få tillgång till Azure Blob Storage från Kina kostar cirka 0,5 CNY per GB, vilket innebär att 1 TB per månad blir ungefär 500 CNY. För ett litet team är det inte en obetydlig summa. När allt kommer omkring kommer ingens pengar från vinden.

**Åtkomstbegränsningar**: I vissa regioner är tillgången till utländska molntjänster instabil, och ibland är den helt enkelt inte tillgänglig. Användare vill ladda ner filerna men kan inte, vilket är frustrerande för alla.

**CDN-kostnad**: Kommersiella CDN kan lösa dessa problem, men priset är lika verkligt. De flesta små team kan helt enkelt inte motivera det.

Så finns det en lösning som är både prisvärd och praktisk? Ja. Använd en molnserver, en omvänd proxy och ett cachinglager. Det är ett enkelt tillvägagångssätt, men det fungerar. Kostnaden sjunker med ungefär hälften, och hastigheten förbättras också, vilket är en anständig kompromiss.

## Arkitektur design

### Övergripande arkitektur

Låt oss börja med hela arkitekturen:

```
User request
    ↓
DNS resolution
    ↓
┌─────────────────────────────────────┐
│   Reverse proxy layer (Traefik/Bunker Web) │ ← SSL termination, routing, security protection
├─────────────────────────────────────┤
│   Ports: 80/443                     │
│   Features: Automatic Let's Encrypt certificates │
│             Host routing            │
└─────────────────────────────────────┘
    ↓
┌─────────────────────────────────────┐
│   Cache layer (Nginx)               │ ← File caching, Gzip compression
├─────────────────────────────────────┤
│   Ports: 8080(server) / 8081(desktop) │
│   Cache strategy:                   │
│   - index.json: 1 hour              │
│   - other files: 7 days             │
│   Cache size: 1GB                   │
└─────────────────────────────────────┘
    ↓
┌─────────────────────────────────────┐
│   Origin (Azure Blob Storage)       │ ← File storage
└─────────────────────────────────────┘
```

Kärnan i den här arkitekturen är enkel: **lägg en cache mellan användare och molnlagring**.

Användarförfrågningar kommer först till det omvända proxylagret på molnservern, och sedan tar Nginx cachelagret över. Om den begärda filen redan är cachad, returneras den omedelbart. Om inte, hämtar Nginx den från molnlagring och lagrar en lokal kopia samtidigt. Nästa gång någon begär samma fil behöver molnlagring inte vara inblandad igen.

### Varför välja denna arkitektur?

**Fördelar med molnservrar**:
- Förutsägbar kostnad: leverantörer som Alibaba Cloud erbjuder lågkostnadsmolnservrar, med 1-2 vCPU och 2 GB RAM-instanser till ett pris på 50-100 CNY per månad
- Flexibel distribution: du kan fritt konfigurera regler för omvänd proxy och cachelagring
- Flexibel geografi: du kan välja serverregioner närmare dina användare
- Bra skalbarhet: du kan uppgradera serverspecifikationen när trafiken växer

**Omvänd proxy + cache-arkitektur**:
- Minska ursprungstrycket: cachelagra heta filer för att minska direktåtkomst till molnlagring
- Lägre kostnad: molnservertrafik är mycket billigare än utgående molnlagring
- Förbättra hastigheten: åtkomst i närheten och serverbandbredd är vanligtvis bättre än direkt leverans av molnlagring

**Varför välja Nginx som cachelagret?**

Detta var inget slumpmässigt val. Nginx har flera verkliga fördelar här:

1. Hög prestanda: Nginx är allmänt känt för utmärkt prestanda för omvänd proxy
2. Mogen caching: den inbyggda `proxy_cache` funktionen är stabil och pålitlig
3. Låg resursanvändning: den kan köras med så lite som 256 MB minne
4. Flexibel konfiguration: du kan tillämpa olika cache-policyer på olika filtyper

## Reverse Proxy Layer: Traefik vs Bunker Web

HagiCodes distributionslösning stöder två alternativ för omvänd proxy, och var och en har sina egna styrkor:

| Alternativ | Egenskaper | Lämpliga scenarier |
|------|------|----------|
| **Traefik** | Lätt, automatisk SSL, enkel konfiguration | Grundläggande utbyggnad, scenarier med låg trafik |
| **Bunker Web** | Inbyggt WAF, DDoS-skydd, anti-bot-skydd | Scenarier med hög säkerhet och hög trafik |

### Traefik: Det lätta förstahandsvalet

Traefik är en modern HTTP omvänd proxy och lastbalanserare. Dess största fördel är att konfigurationen är enkel och den kan erhålla Let's Encrypt-certifikat automatiskt.

För initiala installationer eller scenarier med låg trafik är Traefik ofta ett mycket bra val:
- Den använder relativt få resurser; 1,5 CPU och 512 MB minne räcker
- SSL-certifikat konfigureras automatiskt, så du behöver inte hantera dem själv
- Routing konfigureras genom Docker-etiketter, vilket är tillräckligt bekvämt

### Bunker Web: För scenarier med hög säkerhet

Bunker Web är en Nginx-baserad webbapplikationsbrandvägg med mer omfattande säkerhetsskydd.

När bör du överväga att byta till Bunker Web? Vanligtvis i fall som dessa:
- Du är under DDoS-attack
- Du behöver ModSecurity-skydd
- Du vill ha antibot-skydd
- Du har strängare säkerhetskrav

HagiCode tillhandahåller `switch-deployment.sh` skript så att du snabbt kan växla mellan de två alternativen:

```bash
# Switch to Bunker Web
./switch-deployment.sh bunkerweb

# Switch back to Traefik
./switch-deployment.sh traefik

# Check current status
./switch-deployment.sh status
```

Skriptet utför förkontroller, hälsokontroller och automatisk återställning, så bytesprocessen är ganska säker och pålitlig.

## Nginx Cache Layer Configuration

Cachelagret är kärnan i hela arkitekturen, så Nginx-konfiguration gör en enorm skillnad i cacheprestanda.

### Cache-sökvägskonfiguration

```nginx
# Cache path configuration
proxy_cache_path /var/cache/nginx levels=1:2 keys_zone=azure_cache:10m
                   max_size=1g inactive=7d use_temp_path=off;
```

Parameterdetaljer:
- `levels=1:2`: cachekataloghierarki med två nivåer för att förbättra filåtkomsteffektiviteten
- `keys_zone=azure_cache:10m`: cache-nyckellagringszon; 10 MB räcker för ett stort antal nycklar
- `max_size=1g`: maximal cachestorlek är 1 GB
- `inactive=7d`: radera cachade filer om de inte har nåtts på 7 dagar
- `use_temp_path=off`: skriv direkt i cachekatalogen för bättre prestanda

### Tiered Cache-strategi

Olika filtyper behöver olika cachestrategier:

```nginx
# Server download service
server {
    listen 8080;

    # Short-term cache for index.json (to allow timely updates)
    location /index.json {
        proxy_cache azure_cache;
        proxy_cache_valid 200 1h;
        proxy_cache_key "$scheme$server_port$request_uri";
        add_header X-Cache-Status $upstream_cache_status;
        add_header Cache-Control "public, max-age=3600";

        # Reverse proxy to Azure OSS
        proxy_pass https://${SERVER_DL_HOST}/${SERVER_DL_CONTAINER}$uri?${SERVER_DL_SAS_TOKEN};
        proxy_ssl_server_name on;
        proxy_ssl_protocols TLSv1.2 TLSv1.3;
    }

    # Long-term cache for static files such as installation packages
    location / {
        proxy_cache azure_cache;
        proxy_cache_valid 200 7d;
        proxy_cache_key "$scheme$server_port$request_uri";
        add_header X-Cache-Status $upstream_cache_status;
        add_header Cache-Control "public, max-age=604800";

        proxy_pass https://${SERVER_DL_HOST}/${SERVER_DL_CONTAINER}$uri?${SERVER_DL_SAS_TOKEN};
        proxy_ssl_server_name on;
        proxy_ssl_protocols TLSv1.2 TLSv1.3;
    }
}
```

**Varför är den utformad så här?**

`index.json` är versionskontrollfilen, så den måste uppdateras omedelbart. Med ett 1-timmes cachefönster kan användare upptäcka en ny version inom högst en timme efter publicering.

Statiska filer som installationspaket ändras sällan, så att cachelagra dem i 7 dagar minskar ursprungsåtkomsten avsevärt. När en uppdatering behövs kan du bara rensa cachen manuellt.

**`X-Cache-Status` svarshuvud**:

Den här rubriken hjälper dig att inspektera cacheträffbeteendet:
- `HIT`: cacheträff
- `MISS`: cache miss, hämtad från ursprung
- `EXPIRED`: cachen har gått ut, hämtat från ursprunget igen
- `BYPASS`: cache förbigås

Så här kontrollerar du det:
```bash
curl -I https://server.dl.hagicode.com/app.zip
```

## Implementeringspraxis

### Miljöförberedelse

Konfigurera först miljövariablerna:

```bash
cd /path/to/hagicode_aliyun_deployment/docker
cp .env.example .env
vi .env  # Fill in the Azure OSS SAS URL and Lark Webhook URL
```

**Viktigt**: den `.env` filen innehåller känslig information som SAS Token och Webhook URL. Överlåt det aldrig till versionskontroll.

### DNS-konfiguration

Lägg till följande DNS A-poster:
- `server.dl.hagicode.com` → server IP
- `desktop.dl.hagicode.com` → server IP

### Initiera servern

Använd Ansible för att initiera servern automatiskt:

```bash
cd /path/to/hagicode_aliyun_deployment
ansible-playbook -i ./ansible/inventory/hosts.yml ./ansible/playbooks/init.yml
```

Den här spelboken hanterar följande uppgifter automatiskt:
- Skapa distributionsanvändaren
- Installera Docker och Docker Compose
- Konfigurera SSH-nycklar
- Ställ in brandväggsregler

Det är det huvudsakliga installationsarbetet, och automatisering sparar mycket tid.

### Distribuera tjänsterna

```bash
./deploy.sh
```

Implementeringsskriptet hjälper dig att göra följande:
- Kontrollera miljökonfigurationen
- Dra den senaste koden
- Starta Docker-containrar
- Kör hälsokontroller
- Skicka implementeringsmeddelanden (Lark)

Ett kommando räcker, vilket gör processen bekväm.

### Verifiera distributionen


## Drifttips

### Cachehantering

Cachar behöver också underhåll från tid till annan:

**Kontrollera cachediskanvändning**:
```bash
docker volume inspect docker_nginx-cache
du -sh /var/lib/docker/volumes/docker_nginx-cache/_data
```

**Rensa cachen manuellt**:
```bash
./clear-cache.sh
```

Eller kör de manuella kommandona direkt om det behövs:
```bash
docker exec nginx sh -c "rm -rf /var/cache/nginx/*"
docker restart nginx
```

### Resursgränser

På en 1-kärnig server på 2 GB ser resursgränskonfigurationen ut så här:

```yaml
services:
  traefik:
    deploy:
      resources:
        limits:
          cpus: '1.50'
          memory: 512M

  nginx:
    deploy:
      resources:
        limits:
          cpus: '0.50'
          memory: 256M
```

För att övervaka resursanvändning kan du ibland köra:
```bash
docker stats
```

### Säkerhetsrutiner för SAS Token

SAS-token är den referens som används för att komma åt Azure Blob Storage, så att läcka det skulle vara allvarligt:

- Begå inte `.env` fil till versionskontroll; den är redan inne `.gitignore`
- Ställ in en lämplig utgångstid för SAS Token, med 1 år rekommenderat
- Begränsa SAS Token-behörigheter till skrivskyddad
- Rotera SAS-tokens regelbundet

### Övervakning och varningar

HagiCode integrerar Lark/Feishu Webhook-aviseringar, som kan skicka varningar för följande händelser:
- Implementeringen lyckades eller misslyckades
- Cacherensningsstatus
- Tjänsteundantag

Aviseringar inkluderar serverinformation, tidsstämplar och feldetaljer, vilket gör felsökningen mycket snabbare.

### Tillägg med hög tillgänglighet

När en server inte längre räcker till kan du överväga följande:

1. **Horisontell skalning**: distribuera flera noder och distribuera trafik med DNS round-robin eller en lastbalanserare
2. **CDN framför**: lägg ett CDN framför molnservrarna för ännu snabbare åtkomst
3. **Cacheuppvärmning**: använd skript för att förladda heta filer i cachen

## Anteckningar

Det finns några saker som är värda att tänka på:

1. **SSL-certifikat**: Let's Encrypt har hastighetsgränser, så byt inte distributioner för ofta, annars kan certifikatutfärdandet misslyckas
2. **Cacherensning**: efter att ha uppdaterat viktiga filer, kom ihåg att rensa cacheminnet annars kan användare fortfarande ladda ner den gamla versionen
3. **Logghantering**: rensa upp Docker-loggar regelbundet, annars kan disken bli full
4. **Säkerhetskopieringsstrategi**: säkerhetskopiera filer som t.ex `Traefik acme.json` och Bunker Web-konfiguration
5. **Övervakning och varningar**: konfigurera Feishu-aviseringar så att du kan spåra distributionsstatus och svara snabbt på problem

En molnserver plus ett Nginx-cachinglager är allt som behövs. HagiCode använder denna lösning med en ganska låg månadskostnad, runt 60-100 CNY för servern, och resultaten har varit mycket solida. De viktigaste fördelarna är:

- **Förutsägbar kostnad**: ungefär 50 % billigare än att använda molnlagring direkt eller betala för ett kommersiellt CDN
- **Flexibel distribution**: välj Traefik eller Bunker Web beroende på dina behov
- **Stark skalbarhet**: du kan skala horisontellt eller lägga till ett CDN senare om det behövs
- **Enkla åtgärder**: Skalskript plus Ansible gör automatiserad distribution enkel
