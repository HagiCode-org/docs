## Automatisera bildsynkronisering från Docker Hub till Azure ACR

> Den här artikeln förklarar hur du använder GitHub Actions och bildsynkroniseringsverktyget för att automatisera bildsynkronisering från Docker Hub till Azure Container Registry, och åtgärdar den långsamma åtkomsten till Docker Hub i Kina och vissa Azure-regioner samtidigt som bildtillgänglighet och distributionseffektivitet i Azure-miljöer förbättras.

## Bakgrund / Introduktion

HagiCode-projektet använder Docker-avbildningar som sina kärnkomponenter, med de primära bilderna värd på Docker Hub. När projektet har vuxit och Azure-distributionsbehoven har ökat, stötte vi på följande smärtpunkter:

- Långsamma bilddraghastigheter, med Docker Hub-åtkomst begränsad i Kina och vissa Azure-regioner
- En enda bildkälla skapar en enda risk för fel
- Azure Container Registry ger bättre nätverksprestanda och integration när de körs i Azure

För att lösa dessa problem behövde vi en automatisk bildsynkroniseringsmekanism för att regelbundet synkronisera bilder från Docker Hub till Azure ACR, vilket säkerställer att användare kan få snabbare bilddraghastigheter och högre tillgänglighet i Azure-miljöer.

## Jämförelse av tekniska lösningar

När vi definierade lösningen jämförde vi flera tekniska alternativ:

### 1. bildsynkronisering (slutval)
- Inkrementell synkronisering: synkroniserar endast ändrade bildlager, vilket avsevärt minskar nätverksöverföringen
- Återuppta support: synkroniseringen kan fortsätta efter nätverksavbrott
- Samtidighetskontroll: konfigurerbara arbetstrådar förbättrar effektiviteten vid synkronisering av stora bilder
- Robust felhantering: inbyggd försöksmekanism för fel (3 försök som standard)
- Lättviktsdistribution: en enda binär fil utan beroenden
- Stöd för flera register: kompatibel med Docker Hub, Azure ACR, Harbor och mer

### 2. Docker CLI
- Ingen inkrementell synkronisering: hela bildinnehållet måste hämtas varje gång
- Lägre effektivitet: stor nätverksöverföringsvolym och längre exekveringstid
- Enkel att använda: baserat på bekant `docker pull` och `docker push` kommandon

### 3. Azure CLI
- Högre komplexitet: Azure CLI-autentisering måste konfigureras
- Begränsad funktionalitet: `az acr import` är relativt snäv i omfattning
- Inbyggd integration: integreras väl med Azure-tjänster

## Implementeringssteg

### 1. Förberedelse
- Skapa eller bekräfta ett Azure Container Registry i Azure Portal
- Skapa ACR-åtkomstuppgifter (användarnamn och lösenord)
- Bekräfta åtkomstbehörigheter för Docker Hub-bildarkivet

### 2. Konfigurera GitHub Secrets
Lägg till följande hemligheter i GitHub-förvarets inställningar:
- AZURE_ACR_USERNAME: Azure ACR-användarnamn
- AZURE_ACR_PASSWORD: Azure ACR-lösenord

### 3. Skapa arbetsflödet för GitHub Actions
Konfigurera arbetsflödet i `.github/workflows/sync-docker-acr.yml`:
- Schemalagd trigger: varje dag kl. 00:00 UTC
- Manuell avtryckare: stödjer `workflow_dispatch`
- Ytterligare trigger: körs på tryck till `publish` gren för snabb synkronisering

### 4. Arbetsflödesexekveringsflöde

| Steg | Deltagare | Åtgärd | Beskrivning |
| --- | --- | --- | --- |
| 1 | GitHub-åtgärder | Utlösa arbetsflödet | Utlöses av schema, manuell exekvering eller push till `publish` gren |
| 2 | GitHub-åtgärder → bildsynkronisering | Ladda ner och kör synkroniseringsverktyget | Går in i den faktiska synkroniseringsfasen |
| 3 | bildsynkronisering → Docker Hub | Hämta bildmanifest och tagglista | Läser metadata från källförrådet |
| 4 | bildsynkronisering → Azure ACR | Hämta befintlig bilddata från målförrådet | Bestämmer det aktuella tillståndet för målsidan |
| 5 | bildsynkronisering | Jämför käll- och målskillnader | Identifierar de bildlager som behöver synkroniseras |
| 6 | bildsynkronisering → Docker Hub | Dra ändrade bildlager | Överför endast det innehåll som behöver uppdateras |
| 7 | bildsynkronisering → Azure ACR | Tryck på ändrade bildlager | Slutför inkrementell synkronisering |
| 8 | image-syncer → GitHub-åtgärder | Returnera synkroniseringsstatistik | Inkluderar resultat, skillnader och felinformation |
| 9 | GitHub-åtgärder | Spela in loggar och ladda upp artefakter | Stöder senare revision och felsökning |

## GitHub Actions Workflow Implementation

Nedan är den faktiska arbetsflödeskonfigurationen som används (`.github/workflows/sync-docker-acr.yml`):

```yaml
name: Sync Docker Image to Azure ACR

on:
  schedule:
    - cron: "0 0 * * *" # Every day at 00:00 UTC
  workflow_dispatch: # Manual trigger
  push:
    branches: [publish]

permissions:
  contents: read

jobs:
  sync:
    runs-on: ubuntu-latest

    steps:
      - name: Checkout code
        uses: actions/checkout@v4

      - name: Download image-syncer
        run: |
          # Download the image-syncer binary
          wget https://github.com/AliyunContainerService/image-syncer/releases/download/v1.5.5/image-syncer-v1.5.5-linux-amd64.tar.gz
          tar -zxvf image-syncer-v1.5.5-linux-amd64.tar.gz
          chmod +x image-syncer

      - name: Create auth config
        run: |
          # Generate the authentication config file (YAML format)
          cat > auth.yaml <<EOF
          hagicode.azurecr.io:
            username: "${{ secrets.AZURE_ACR_USERNAME }}"
            password: "${{ secrets.AZURE_ACR_PASSWORD }}"
          EOF

      - name: Create images config
        run: |
          # Generate the image sync config file (YAML format)
          cat > images.yaml <<EOF
          docker.io/newbe36524/hagicode: hagicode.azurecr.io/hagicode
          EOF

      - name: Run image-syncer
        run: |
          # Run synchronization (using the newer --auth and --images parameters)
          ./image-syncer --auth=./auth.yaml --images=./images.yaml --proc=10 --retries=3

      - name: Upload logs
        if: always()
        uses: actions/upload-artifact@v4
        with:
          name: sync-logs
          path: image-syncer-*.log
          retention-days: 7
```

## Konfigurationsdetaljer

### 1. Utlösningsvillkor
- Schemalagd trigger: `cron: "0 0 * * *"` körs varje dag kl 00:00 UTC
- Manuell trigger: `workflow_dispatch` tillåter användare att köra det manuellt från GitHub UI
- Tryck avtryckaren: `push: branches: [publish]` triggers vid tryck till releasegrenen för snabb synkronisering

### 2. Autentiseringskonfiguration (`auth.yaml`)
```yaml
hagicode.azurecr.io:
  username: "${{ secrets.AZURE_ACR_USERNAME }}"
  password: "${{ secrets.AZURE_ACR_PASSWORD }}"
```

### 3. Konfiguration av bildsynkronisering
```yaml
docker.io/newbe36524/hagicode: hagicode.azurecr.io/hagicode
```
Denna konfiguration innebär att alla taggar från `docker.io/newbe36524/hagicode` kommer att synkroniseras till `hagicode.azurecr.io/hagicode`.

### 4. bildsynkroniseringsparametrar
- `--auth=./auth.yaml`: sökväg till autentiseringskonfigurationsfilen
- `--images=./images.yaml`: sökväg till bildsynkroniseringskonfigurationsfilen
- `--proc=10`: använda `10` samtidiga arbetare
- `--retries=3`: försök igen misslyckades `3` gånger

## Anteckningar och bästa praxis

### 1. Säkerhetsrekommendationer
- Rotera Azure ACR-lösenord regelbundet (rekommenderas var 90:e dag)
- Använd ett dedikerat ACR-tjänstkonto med push-only-behörigheter
- Övervaka ACR-åtkomstloggar för att upptäcka onormal åtkomst i tid
- Mata inte ut autentiseringsuppgifter i loggar
- Överlåt inte autentiseringsuppgifter till kodförrådet

### 2. Prestandaoptimering
- Justera `--proc` parameter: ställ in samtidighet baserat på nätverkets bandbredd (rekommenderas `5-20`)
- Övervaka synkroniseringens varaktighet: om synkroniseringen tar för lång tid, överväg att minska samtidigheten
- Rensa loggar regelbundet: ställ in en rimlig `retention-days` värde (`7` för närvarande)

### 3. Felsökning

#### Problem 1: Autentiseringsfel
```
Error: failed to authenticate to hagicode.azurecr.io
```
Lösning:
1. Kontrollera om GitHub Secrets är korrekt konfigurerade
2. Kontrollera om Azure ACR-lösenordet har upphört att gälla
3. Bekräfta att behörigheterna för ACR-tjänstkontot är korrekta

#### Problem 2: Nätverkstimeout
```
Error: timeout waiting for response
```
Lösning:
1. Kontrollera nätverksanslutning
2. Minska antalet samtidiga arbetare (den `--proc` parameter)
3. Vänta tills nätverket återställs och kör arbetsflödet igen

#### Problem 3: Ofullständig bildsynkronisering
```
Warning: some tags failed to sync
```
Lösning:
1. Kontrollera synkroniseringsloggarna för att identifiera misslyckade taggar
2. Kör arbetsflödet igen manuellt för att synkronisera igen
3. Kontrollera att källbilden på Docker Hub är sund

### 4. Övervakning och larm
- Kontrollera regelbundet sidan Åtgärder för att bekräfta arbetsflödesstatus
- Aktivera GitHub-aviseringar så att arbetsflödesfel rapporteras omgående
- Övervaka Azure ACR-lagringsanvändning
- Verifiera regelbundet taggkonsistens i alla register

## Vanliga frågor och lösningar

### F1: Hur synkroniserar jag specifika taggar istället för alla taggar?

Ändra `images.yaml` konfigurationsfil:
```yaml
# Sync only the latest and v1.0 tags
docker.io/newbe36524/hagicode:latest: hagicode.azurecr.io/hagicode:latest
docker.io/newbe36524/hagicode:v1.0: hagicode.azurecr.io/hagicode:v1.0
```

### F2: Hur synkroniserar jag flera bildarkiv?

Lägg till flera rader till `images.yaml`:
```yaml
docker.io/newbe36524/hagicode: hagicode.azurecr.io/hagicode
docker.io/newbe36524/another-image: hagicode.azurecr.io/another-image
```

### F3: Hur försöker jag igen efter att synkroniseringen misslyckats?
- Automatiskt försök igen: bildsynkroniseringen inkluderar en inbyggd försöksmekanism (`3` försöker igen som standard)
- Manuellt försök igen: klicka `Re-run all jobs` på sidan GitHub Actions

### F4: Hur kan jag se detaljerade synkroniseringsförlopp?
- Se loggar i realtid på sidan Åtgärder
- Ladda ner `sync-logs` artefakt för att inspektera de fullständiga loggfilerna
- Loggfilerna inkluderar synkroniseringsstatus och överföringshastighet för varje tagg

### F5: Hur lång tid tar synkroniseringen?
- Initial full synkronisering: vanligtvis `10-30` minuter beroende på bildstorlek
- Inkrementell synkronisering: vanligtvis `2-5` minuter om bildförändringarna är små
- Tidpunkten beror på nätverkets bandbredd, bildstorlek och samtidighetsinställningar

## Föreslagna tillägg

### 1. Lägg till synkroniseringsmeddelanden
Lägg till ett aviseringssteg i arbetsflödet:
```yaml
- name: Notify on success
  if: success()
  run: |
    echo "Docker images synced successfully to Azure ACR"
```

### 2. Implementera bildtaggfiltrering
Lägg till logik för taggfiltrering i arbetsflödet:
```yaml
- name: Filter tags
  run: |
    # Sync only tags prefixed with v
    echo "docker.io/newbe36524/hagicode:v* : hagicode.azurecr.io/hagicode:v*" > images.yaml
```

### 3. Lägg till en synkroniseringsstatistikrapport
```yaml
- name: Generate report
  if: always()
  run: |
    echo "## Sync Report" >> $GITHUB_STEP_SUMMARY
    echo "- Total tags: $(grep -c 'synced' image-syncer-*.log)" >> $GITHUB_STEP_SUMMARY
    echo "- Sync time: ${{ steps.sync.outputs.duration }}" >> $GITHUB_STEP_SUMMARY
```
