## Progressiv avslöjande: Förbättring av människa-datorinteraktion i AI-produkter med "Less Is More"-filosofin

> I AI-produktdesign avgör ofta kvaliteten på användarinmatningen kvaliteten på resultatet. Den här artikeln delar en "progressiv avslöjande" interaktionsmetod som vi praktiserade i HagiCode-projektet. Genom steg-för-steg-vägledning, intelligent komplettering och omedelbar feedback, omvandlar det användarnas korta och vaga input till strukturerade tekniska förslag, vilket avsevärt förbättrar effektiviteten i interaktion mellan människa och dator.

## Bakgrund

Alla som bygger AI-produkter har förmodligen sett den här situationen: en användare öppnar din app och skriver entusiastiskt in en begäran på en rad, bara för att AI:n ska returnera något helt utanför målet. Det är inte så att AI inte är tillräckligt smart. Användaren gav helt enkelt inte tillräckligt med information. Tankeläsning är svårt för alla.

Detta problem blev särskilt uppenbart när vi byggde HagiCode. HagiCode är en AI-driven kodningsassistent där användare beskriver krav på naturligt språk för att skapa tekniska förslag och sessioner. Vid faktisk användning fann vi att användarinmatning ofta har dessa problem:

- **Inkonsekvent indatakvalitet**: vissa användare skriver bara några få ord, som "optimera inloggning" eller "fixa bugg", utan det nödvändiga sammanhanget
- **Inkonsekvent teknisk terminologi**: olika användare använder olika termer för samma sak; vissa säger "frontend" medan andra säger "FE"
- **Saknar strukturerad information**: det finns ingen projektbakgrund, förvarsomfattning eller påverkan, även om dessa är kritiska detaljer
- **Upprepade problem**: samma typer av förfrågningar dyker upp om och om igen, och varje gång måste de förklaras från början

Det direkta resultatet är förutsägbart: AI:n har svårare att förstå begäran, förslagskvaliteten blir instabil och användarupplevelsen blir lidande. Användare tänker, "Denna AI är inte särskilt bra", medan vi känner oss orättvist klandrade. Om du bara ger mig en mening, hur ska jag kunna gissa vad du verkligen vill?

I sanning är detta förståeligt. Även människor behöver tid för att förstå varandra, och maskiner är inget undantag.

För att lösa dessa smärtpunkter tog vi ett djärvt beslut: introducera designprincipen "progressive disclosure" för att förbättra interaktionen mellan människa och dator. Förändringarna detta medförde var förmodligen större än du kan föreställa dig. För att vara ärlig, förväntade vi oss inte att det skulle vara så effektivt vid den tiden.

## Vad är progressiv avslöjande

"Progressive disclosure" är en designprincip från området HCI (human-computer interaction). Dess kärnidé är enkel: **visa inte användarna all information och alternativ på en gång. Avslöja istället bara vad som är nödvändigt steg för steg, baserat på användarens handlingar och behov**.

Denna princip är särskilt lämplig för AI-produkter eftersom AI-interaktion är naturligt progressiv. Användaren säger lite, AI:n förstår lite, sedan lägger användaren till mer och AI:n förstår mer. Det är väldigt likt hur människor kommunicerar med varandra: förståelsen utvecklas vanligtvis gradvis.

I HagiCodes scenario tillämpade vi progressiv avslöjande inom fyra områden:

### 1. Beskrivningsoptimeringsmekanism: låt AI hjälpa dig att säga saker tydligare

När en användare anger en kort beskrivning skickar vi den inte direkt till AI:n för tolkning. Istället utlöser vi först ett "beskrivningsoptimeringsflöde". Kärnan i detta flöde är "strukturerad utdata": att konvertera användarens fria text till ett standardformat. Det är som att stränga in lösa pärlor i ett halsband så allt blir lättare att förstå.

Den optimerade beskrivningen måste innehålla följande standardavsnitt:

- **Bakgrund**: problembakgrunden och sammanhanget
- **Analys**: teknisk analys och resonemang
- **Lösning**: lösningen och implementeringsstegen
- **Övning**: konkreta kodexempel och anteckningar

Samtidigt genererar vi automatiskt en Markdown-tabell som visar information som målförvaret, sökvägar och redigeringsbehörigheter, vilket gör efterföljande AI-operationer enklare. En tydlig katalog gör det alltid lättare att hitta.

Här är den faktiska implementeringen:

```csharp
// Core method in ProposalDescriptionMemoryService.cs
public async Task<string> OptimizeDescriptionAsync(
    string title,
    string description,
    string locale = "zh-CN",
    DescriptionOptimizationMemoryContext? memoryContext = null,
    CancellationToken cancellationToken = default)
{
    // Build query parameters
    var queryContext = BuildQueryContext(title, description);

    // Retrieve historical context
    var memoryContext = await RetrieveHistoricalContextAsync(queryContext, cancellationToken);

    // Generate a structured prompt
    var prompt = await BuildOptimizationPromptAsync(
        title,
        description,
        memoryContext,
        cancellationToken);

    // Call AI for optimization
    return await _aiService.CompleteAsync(prompt, cancellationToken);
}
```

Nyckeln till detta flöde är "minnesinjektion". Vi injicerar historiska sammanhang som projektkonventioner, liknande fall och negativa mönster i prompten, vilket gör att AI:n kan referera till tidigare erfarenheter under optimering. Erfarenhet ska inte gå till spillo.

**Anmärkningar:**
- Se till att den aktuella inmatningen har prioritet över historiskt minne, så att explicit specificerad användarinformation inte åsidosätts
- HagIndex-referenser ska behandlas som faktakällor och får inte ändras av historiska fall
- Korrigeringsförslag med låg förtroende bör inte injiceras som starka begränsningar

### 2. Röstinmatningsförmåga: tala är mer naturligt än att skriva

Förutom textinmatning stöder vi även röstinmatning. Detta är särskilt användbart för att beskriva komplexa krav. Att skriva en teknisk förfrågan kan ta minuter, medan att säga det högt kan bara ta några dussin sekunder.

Det viktigaste designfokuset för röstinmatning är "state management". Användare måste tydligt veta vilket tillstånd systemet för närvarande är i. Vi definierade följande tillstånd:

- **Idle**: systemet är klart och inspelningen kan starta
- **Väntar uppströms**: systemet ansluter till backend-tjänsten
- **Inspelning**: användarens röst spelas in
- **Bearbetar**: tal konverteras till text
- **Fel**: ett fel inträffade och kräver uppmärksamhet från användaren

Frontend-tillståndsmodellen ser ungefär ut så här:

```typescript
interface VoiceInputState {
  status: 'idle' | 'waiting-upstream' | 'recording' | 'processing' | 'error';
  duration: number;
  error?: string;
  deletedSet: Set<string>; // Fingerprint set of deleted results
}

// State transition when recording starts
const handleVoiceInputStart = async () => {
  // Enter waiting state first and show a loading animation
  setState({ status: 'waiting-upstream' });

  // Wait for backend readiness confirmation
  const isReady = await waitForBackendReady();
  if (!isReady) {
    setState({ status: 'error', error: 'Backend service is not ready' });
    return;
  }

  // Start recording
  setState({ status: 'recording', startTime: Date.now() });
};

// Handle recognition results
const handleRecognitionResult = (result: RecognitionResult) => {
  const fingerprint = normalizeFingerprint(result.text);

  // Check whether it has already been deleted
  if (state.deletedSet.has(fingerprint)) {
    return; // Skip deleted content
  }

  // Merge the result into the text box
  appendResult(result);
};
```

Det finns en viktig detalj här: vi använder en "fingerprint set" för att hantera raderingssynkronisering. När taligenkänning returnerar flera resultat kan användare radera några av dem. Vi lagrar fingeravtrycken från raderat innehåll så att om samma innehåll dyker upp igen senare, hoppas det över automatiskt. Det är i huvudsak ett sätt att komma ihåg vad användaren redan har avvisat.

### 3. Snabbhanteringssystem: externisera AI:s "hjärna"

HagiCode har ett flexibelt prompthanteringssystem där alla prompter lagras som filer:

```
prompts/
├── metadata/
│   ├── optimize-description.zh-CN.json
│   └── optimize-description.en-US.json
└── templates/
    ├── optimize-description.zh-CN.hbs
    └── optimize-description.en-US.hbs
```

Varje prompt består av två delar:
- **Metadatafil (`.json`)**: definierar information såsom promptscenario, version och parametrar
- **Mallfil (`.hbs`)**: det faktiska promptinnehållet, skrivet med Handlebars syntax

Metadatafilformatet ser ut så här:

```json
{
  "scenario": "optimize-description",
  "locale": "zh-CN",
  "version": "1.0.0",
  "syntax": "handlebars",
  "syntaxVersion": "1.0",
  "parameters": [
    {
      "name": "title",
      "type": "string",
      "required": true,
      "description": "Proposal title"
    },
    {
      "name": "description",
      "type": "string",
      "required": true,
      "description": "Original description"
    }
  ],
  "author": "HagiCode Team",
  "description": "Optimize the user's technical proposal description",
  "lastModified": "2026-04-05",
  "tags": ["optimization", "nlp"]
}
```

Mallfilen använder Handlebars-syntax och stöder parameterinjektion:

```handlebars
You are a technical proposal expert.

<task>
Generate a structured technical proposal description based on the following information.
</task>

<input>
<title>{{title}}</title>
<description>{{description}}</description>
{{#if memoryContext}}
<memory_context>
{{memoryContext}}
</memory_context>
{{/if}}
</input>

<output_format>
## Background
[Describe the problem background and context, including project information, repository scope, and so on]

## Analysis
[Provide the technical analysis and reasoning process, and explain why this change is needed]

## Solution
[Provide the solution and implementation steps, listing the key code locations]

## Practice
[Provide concrete code examples and notes]
</output_format>
```

Fördelarna med denna design är tydliga:
- prompter kan versionskontrolleras precis som kod
- flera språk stöds och kan bytas automatiskt baserat på användarens preferenser
- den parametriserade designen gör att kontexten kan injiceras dynamiskt
- fullständighet kan valideras vid start, vilket undviker körtidsfel

Om kunskap bara stannar i någons huvud är det lätt att förlora. Att spela in det på ett strukturerat sätt från början är mycket säkrare.

### 4. Progressiv guide: dela upp komplexa uppgifter i små steg

För komplexa uppgifter, såsom förstagångsinstallation och konfiguration, använder vi en guide med flera steg. Varje steg kräver endast nödvändig information och ger tydliga framstegsindikatorer. Stora uppgifter blir mycket mer hanterbara när de hanteras ett steg i taget.

Guidens tillståndsmodell:

```typescript
interface WizardState {
  currentStep: number;           // 0-3, corresponding to 4 steps
  steps: WizardStep[];
  canGoNext: boolean;
  canGoBack: boolean;
  isLoading: boolean;
  error: string | null;
}

interface WizardStep {
  id: number;
  title: string;
  description: string;
  completed: boolean;
}

// Step navigation logic
const goToNextStep = () => {
  if (wizardState.currentStep < wizardState.steps.length - 1) {
    // Validate input for the current step
    if (validateCurrentStep()) {
      wizardState.currentStep++;
      wizardState.steps[wizardState.currentStep - 1].completed = true;
    }
  }
};

const goToPreviousStep = () => {
  if (wizardState.currentStep > 0) {
    wizardState.currentStep--;
  }
};
```

Varje steg har sin egen valideringslogik, och genomförda steg får tydliga visuella markörer. Genom att avbryta öppnas en bekräftelsedialogruta för att förhindra att användare tappar framsteg genom oavsiktliga åtgärder.

## Slutsats

När vi ser tillbaka på HagiCodes progressiva avslöjandepraxis kan vi sammanfatta flera kärnprinciper:

1. **Steg-för-steg-vägledning**: dela upp komplexa uppgifter i mindre steg och begär endast nödvändig information i varje steg
2. **Intelligent komplettering**: använd historisk kontext och projektkunskap för att fylla i information automatiskt
3. **Omedelbar feedback**: ge varje åtgärd tydlig visuell feedback och statustips
4. **Feltoleransmekanismer**: tillåt användare att ångra och återställa så att misstag inte leder till oåterkallelig förlust
5. **Mångfald indata**: stöder flera inmatningsmetoder som text och röst

I HagiCode var det praktiska resultatet av detta tillvägagångssätt tydligt: den genomsnittliga längden på användarinmatning ökade från färre än 20 tecken till strukturerade beskrivningar på 200-300 tecken, kvaliteten på AI-genererade förslag förbättrades avsevärt, och samtidigt ökade användarnas tillfredsställelse.

Detta är inte förvånande. Ju mer information användarna tillhandahåller, desto mer exakt kan AI:n förstå dem, och desto bättre resultat kan den ge. I den meningen skiljer det sig inte så mycket från kommunikation mellan människor.

Om du också bygger AI-relaterade produkter hoppas jag att dessa erfarenheter ger lite användbar inspiration. Kom ihåg: användare vägrar inte nödvändigtvis att lämna information. Oftare har produkten ännu inte ställt rätt frågor på rätt sätt. Kärnan i progressiv avslöjande är att hitta den bästa timingen och formen för dessa frågor.
