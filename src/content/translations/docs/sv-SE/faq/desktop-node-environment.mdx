HagiCode Desktop levereras med en Desktop-hanterad Node-verktygskedja. Den tar inte över den globala Node-installationen på din dator. Dess syfte är att ge Desktop-start och runtime-relaterade flöden en stabil och verifierbar runtime-grund.

Läs denna FAQ om du ser `portable-fixed/toolchain` i diagnostik, behöver veta vilken `node` Desktop använder, eller undrar varför din system Node-version inte direkt definierar Desktop runtime beteende.

## Varför Desktop paketerar Node

Desktop måste köra samma stödflöden över operativsystem, installationskanaler och användarmiljöer. Om det bara berodde på användarens befintliga nodinstallation, skulle supporten behöva hantera föråldrade versioner, saknade npm, felaktig sökvägsordning, förorenade globala paket och organisationsspecifika registerpolicyer på en gång.

Desktop beskriver detta mer specifikt som en isolerad Node/npm-körtid för hanterade CLI:er istället för att återanvända den systemomfattande miljön. Denna design har dessa huvudsakliga fördelar:

- **Konsekventa versioner**: Desktop använder ett fast Node 22 runtime-kontrakt istället för att gissa användarens lokala huvudversion.
- **Kontrollerade startflöden**: Desktophanterade kommandon kan föredra de paketerade `node` och `npm` poster.
- **Rensa konfigurationsgränser**: CLI-relaterad konfiguration hålls åtskild från värdomfattande global konfiguration, så det är mindre sannolikt att uppdateringar, byte och återställning störs av skillnader i systemmiljön.
- **Minskade konflikter**: Buntade sökvägar gäller för kommandomiljöer skapade av Desktop och ersätter inte användarens globala nod i terminalen.
- **Icke-påträngande värdbeteende**: Denna isolering undviker att skriva verktygskedjetillstånd till användarens befintliga Node/npm-miljö, och avinstallation av Desktop stör inte noden, npm eller konfigurationen som redan finns på maskinen.

## Hur Desktop väljer nodsökväg

När du kör Desktop normalt föredrar Desktop verktygskedjan Node installerad med appen istället för att först läsa nodinstallationen från ditt system. Detta håller nodversionen i din terminal åtskild från nodversionen som Desktop förväntar sig medan den körs.

När den medföljande verktygskedjan är aktiverad, placerar kommandomiljöer som skapats av Desktop paketerade verktygskedjevägar längst fram i `PATH`. Som ett resultat, även om ditt system har en annan Node-version installerad, löser Desktop-hanterade flöden till Desktops medföljande Node och npm först.

Detta påverkar endast körtidsmiljöer som skapats av Desktop. Om du springer `node`, `npm`, eller andra kommandon direkt i din systemterminal, fortsätter de att använda din normala systemmiljö och skalkonfiguration.

I diagnostik- eller barnprocessmiljöer som startas av Desktop, kan du se en variabel så här:

```bash
HAGICODE_PORTABLE_TOOLCHAIN_ROOT=/path/to/portable-fixed/toolchain
```

Den här variabeln betyder att den skrivbordshanterade runtime-miljön kände igen den medföljande verktygskedjans rot. Att se den här sökvägen betyder inte att Desktop har ändrat din systemnod; det betyder bara att Desktop använder Node-miljön som levereras med appen inom sitt eget runtime-omfång.

## Där Desktop installerar hanterade npm-paket

Desktop håller runtime-filer och föränderliga npm-paketfiler på olika platser. Den medföljande verktygskedjans väg som t.ex `portable-fixed/toolchain/node` är den oföränderliga källan för Desktop-hanterade `node` och `npm` kommandon. Hanterade globala paket installeras under Electrons användardatakatalog istället:

```text
userData/node22/npmGlobal
```

Om Desktop senare flyttar till Nod 24 använder det ett separat paketträd som t.ex `userData/node24/npmGlobal`. Paket som är installerade för Nod 22 finns kvar på disken men behandlas inte som installerade för Nod 24 förrän de installeras om i det nya Node-major-prefixet.

Skrivbordsberoendestatus kontrollerar bara det aktiva `userData/node<major>/npmGlobal` prefix. Paketfiler utanför det prefixet ignoreras av beroendehantering.
