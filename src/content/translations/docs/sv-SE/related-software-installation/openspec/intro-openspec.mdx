# Introduktion till OpenSpec

OpenSpec är ett specifikationsdrivet utvecklingsverktyg som hjälper team att hantera mjukvaruutvecklingsprocessen genom strukturerade förslag och specifikationer. Den här artikeln ger en omfattande introduktion till OpenSpecs kärnkoncept, arbetsflöden och användningsmetoder.

## Vad är OpenSpec?

OpenSpec är ett kommandoradsverktyg som används för att hantera förslag (Proposals), specifikationer (Specifications) och ändringar (Changes) i mjukvaruutvecklingsprocessen. Det standardiserar utvecklingsprocessen och säkerställer att varje funktion går igenom grundlig design och granskning från idé till implementering.

### Kärnprinciper

OpenSpec bygger på följande kärnprinciper:

- **Specifikationer först**: Definiera tydliga specifikationer innan du skriver kod
- **Strukturerat samarbete**: Främja teamsamarbete genom standardiserade dokumentformat
- **Spårbarhet**: Registrera hela processen från förslag till implementering
- **Kvalitetssäkring**: Säkerställ att koden överensstämmer med designavsikten genom validering av specifikationer

### Huvudfunktioner

OpenSpec erbjuder följande huvudfunktioner:

1. **Förslagshantering**: Skapa, granska och spåra funktionsförslag
2. **Specifikationsdefinition**: Skriv och underhåll tekniska specifikationsdokument
3. **Ändringsspårning**: Registrera hela historiken för granskade ändringar
4. **Specifikationsvalidering**: Säkerställ att kodändringar överensstämmer med fördefinierade specifikationer
5. **Teamsamarbete**: Stöd utvecklings- och granskningsprocesser med flera deltagare

## OpenSpec arbetsflöde

Det typiska OpenSpec-arbetsflödet omfattar följande faser:

### 1. Skapa ett förslag (Proposal)

När det finns en ny funktion eller ett krav på betydande ändring, skapa först ett förslag:

```bash
openspec proposal create
```

Förslaget innehåller:
- **Problembeskrivning**: Problemet eller kravet som ska lösas
- **Lösning**: Översikt över den föreslagna lösningen
- **Påverkansomfång**: Påverkan på det befintliga systemet
- **Alternativa lösningar**: Andra övervägda lösningar

### 2. Skriv specifikationen (Specification)

Efter att förslaget godkänts, skriv en detaljerad teknisk specifikation:

```bash
openspec spec create
```

Specifikationen innehåller:
- **Teknisk design**: Detaljerad plan för teknisk implementering
- **Gränssnittsdefinition**: Design av gränssnitt som API:er, datastrukturer etc.
- **Implementeringsdetaljer**: Nyckelimplementeringslogik och algoritmer
- **Testplan**: Hur man verifierar implementeringens korrekthet

### 3. Skapa en ändring (Change)

När du börjar implementeringen, skapa en ändring för att spåra utvecklingsprocessen:

```bash
openspec change create
```

Ändringen innehåller:
- **Associerad specifikation**: Specifikationen som implementeras av denna ändring
- **Implementeringsuppgifter**: Lista över konkreta utvecklingsuppgifter
- **Framstegsspårning**: Status för uppgifternas slutförande
- **Kodändringar**: De faktiska kodmodifikationerna

### 4. Validering och arkivering

Efter att utvecklingen är klar, validera om ändringen överensstämmer med specifikationen och arkivera sedan:

```bash
openspec change validate
openspec change archive
```

## OpenSpec katalogstruktur

OpenSpec skapar en `openspec/`-katalog i projektets rotkatalog med följande struktur:

```
openspec/
├── proposals/           # Förslagskatalog
│   ├── PROPOSAL-001.md
│   └── PROPOSAL-002.md
├── specs/              # Specifikationskatalog
│   ├── SPEC-001.md
│   └── SPEC-002.md
├── changes/            # Ändringskatalog
│   ├── feature-name/
│   │   ├── proposal.md
│   │   ├── design.md
│   │   ├── specs.md
│   │   └── tasks.md
│   └── another-feature/
└── archived/           # Arkiverade ändringar
    └── feature-name/
```

### Katalogförklaring

- **proposals/**: Lagrar alla förslagsdokument, varje förslag i en separat Markdown-fil
- **specs/**: Lagrar tekniska specifikationsdokument som motsvarar förslagen
- **changes/**: Lagrar pågående ändringar, varje ändring i en underkatalog
- **archived/**: Lagrar slutförda och arkiverade ändringar

## Kärnkommandon

### Förslagsrelaterade kommandon

```bash
# Skapa nytt förslag
openspec proposal create

# Lista alla förslag
openspec proposal list

# Visa förslagsdetaljer
openspec proposal view <proposal-id>

# Godkänn förslag
openspec proposal approve <proposal-id>
```

### Specifikationsrelaterade kommandon

```bash
# Skapa ny specifikation
openspec spec create

# Lista alla specifikationer
openspec spec list

# Visa specifikationsdetaljer
openspec spec view <spec-id>

# Uppdatera specifikation
openspec spec update <spec-id>
```

### Ändringsrelaterade kommandon

```bash
# Skapa ny ändring
openspec change create

# Lista alla ändringar
openspec change list

# Visa ändringsdetaljer
openspec change view <change-name>

# Lägg till uppgift
openspec change add-task <change-name>

# Validera ändring
openspec change validate <change-name>

# Arkivera ändring
openspec change archive <change-name>
```

### Allmänna kommandon

```bash
# Visa version
openspec --version

# Visa hjälp
openspec --help

# Visa hjälp för ett specifikt kommando
openspec <command> --help
```

## Använda OpenSpec i HagiCode-projektet

### Projektintegration

I HagiCode-projektet är OpenSpec integrerat i utvecklingsarbetsflödet:

1. **Initialiseringskontroll**: Startguiden kontrollerar om OpenSpec är installerat
2. **Förslagsdriven utveckling**: Alla nya funktioner måste gå igenom OpenSpec-förslagsprocessen
3. **Kodgranskning**: Ändringar måste överensstämma med OpenSpec-specifikationer för att kunna slås samman
4. **Dokumentationssynkronisering**: OpenSpec-dokumentation hålls synkroniserad med kodimplementering

### Versionskrav

HagiCode-projektet kräver användning av version **OpenSpec >=1.0.0 {'<'}2.0.0**.

Installationskommando:

```bash
npm install -g @fission-ai/openspec@1
```

### Arbetsflödesexempel

Här är ett komplett exempel på användning av OpenSpec i HagiCode-projektet:

#### Steg 1: Skapa förslag

```bash
cd /path/to/hagicode-project
openspec proposal create
```

Ange i den interaktiva prompten:
- Förslagstitel: Lägg till användarautentiseringsfunktion
- Problembeskrivning: Systemet behöver användarinloggning och behörighetshantering
- Lösning: Implementera JWT-baserat autentiseringssystem

#### Steg 2: Skriv specifikationen

Efter godkännande av förslaget, skapa en detaljerad specifikation:

```bash
openspec spec create
```

Definiera i specifikationen:
- Design av autentiseringsflöde
- API-gränssnittsdefinition
- Databasstruktur
- Säkerhetsöverväganden

#### Steg 3: Skapa ändring

Börja implementeringen:

```bash
openspec change create
```

Ange ändringsnamn: `add-user-authentication`

#### Steg 4: Lägg till uppgifter

```bash
openspec change add-task add-user-authentication
```

Lägg till konkreta uppgifter:
- Implementera användarregistreringsendpoint
- Implementera inloggningsendpoint
- Implementera JWT middleware
- Skriv enhetstester

#### Steg 5: Implementering och validering

Efter att koden är klar, validera ändringen:

```bash
openspec change validate add-user-authentication
```

#### Steg 6: Arkivering

Efter lyckad validering, arkivera ändringen:

```bash
openspec change archive add-user-authentication
```

## Bästa praxis

### Skriva förslag

1. **Tydlig problemformulering**: Definiera tydligt problemet som ska lösas; undvik vaga beskrivningar
2. **Överväg alternativ**: Lista och jämför olika lösningar
3. **Utvärdera påverkan**: Analysera påverkan och risker för det befintliga systemet
4. **Håll det kortfattat**: Förslag bör vara kortfattade och tydliga; lämna detaljer till specifikationen

### Skriva specifikationer

1. **Detaljerad men inte överflödig**: Ge tillräckligt med detaljer men undvik överdesign
2. **Använd diagram**: Använd flödesscheman, arkitekturdiagram etc. för att hjälpa till med förklaringen
3. **Definiera gränssnitt**: Definiera tydligt alla publika gränssnitt och datastrukturer
4. **Överväg kantfall**: Lista undantagssituationer och felhantering

### Ändringshantering

1. **Små iterationer**: Dela upp stora funktioner i flera små ändringar
2. **Uppdatera i tid**: Håll uppgiftsstatus och dokumentation synkroniserade
3. **Kodgranskning**: Utför kodgranskning efter att ändringen är klar
4. **Arkivera i tid**: Arkivera slutförda ändringar i tid och håll arbetskatalogen ren

### Teamsamarbete

1. **Regelbundna granskningar**: Granska regelbundet förslag och specifikationer för att säkerställa teamkonsensus
2. **Dokumentation först**: Slutför dokumentationen innan du börjar koda
3. **Upprätthåll kommunikation**: Registrera diskussioner och beslut i förslag och specifikationer
4. **Versionskontroll**: Inkludera OpenSpec-dokumentation i versionskontrollen

## Vanliga användningsscenarier

### Scenario 1: Lägga till ny funktion

1. Skapa ett förslag som beskriver kraven för den nya funktionen
2. Teamet diskuterar och godkänner förslaget
3. Skriv en detaljerad teknisk specifikation
4. Skapa en ändring och implementera funktionen
5. Validera och arkivera ändringen

### Scenario 2: Refaktorisera befintlig kod

1. Skapa ett förslag som förklarar skälen och målen för refaktoriseringen
2. Definiera refaktoriseringsplanen och stegen i specifikationen
3. Skapa en ändring och refaktorisera gradvis
4. Säkerställ testtäckning och validera refaktoriseringsresultaten
5. Arkivera ändringen

### Scenario 3: Buggfixning

För enkla buggar:
- Kan fixas direkt utan fullständig OpenSpec-process

För komplexa buggar:
1. Skapa ett förslag för att analysera buggans grundorsak
2. Definiera fixlösningen i specifikationen
3. Skapa en ändring och implementera fixet
4. Lägg till tester för att förhindra regression
5. Arkivera ändringen

### Scenario 4: Hantering av teknisk skuld

1. Skapa ett förslag som listar tekniska skuldposter
2. Definiera en förbättringsplan i specifikationen
3. Skapa en ändring för att gradvis betala av teknisk skuld
4. Spåra framsteg och effekt
5. Arkivera slutförda förbättringar

## Integration med andra verktyg

### Git-integration

OpenSpec integreras sömlöst med Git:

```bash
# OpenSpec-dokumentation bör inkluderas i versionskontrollen
git add openspec/
git commit -m "Add OpenSpec proposal for feature X"
```

### CI/CD-integration

Validera OpenSpec-specifikationer i CI/CD-processen:

```yaml
# .github/workflows/openspec.yml
name: OpenSpec Validation
on: [pull_request]
jobs:
  validate:
    runs-on: ubuntu-latest
    steps:
      - uses: actions/checkout@v2
      - name: Install OpenSpec
        run: npm install -g @fission-ai/openspec@1
      - name: Validate changes
        run: openspec change validate
```

### IDE-integration

Det rekommenderas att använda en editor som stöder Markdown för att redigera OpenSpec-dokument:
- VS Code + Markdown-plugin
- JetBrains IDEs + Markdown-stöd
- Vim/Neovim + Markdown-plugin

## Felsökning

### Vanliga problem

#### Problem: OpenSpec-kommandot hittades inte

**Lösning**:
1. Bekräfta att OpenSpec är installerat: `npm list -g @fission-ai/openspec`
2. Kontrollera att PATH-miljövariabeln inkluderar npm:s globala bin-katalog
3. Installera om: `npm install -g @fission-ai/openspec@1`

#### Problem: Kan inte skapa förslag eller ändring

**Lösning**:
1. Bekräfta att den aktuella katalogen är projektets rotkatalog
2. Kontrollera om `openspec/`-katalogen finns; om inte, skapa den
3. Bekräfta att du har skrivbehörighet

#### Problem: Validering misslyckades

**Lösning**:
1. Kontrollera om dokumentationen i ändringskatalogen är komplett
2. Bekräfta att alla nödvändiga uppgifter har slutförts
3. Granska detaljerad felinformation: `openspec change validate <change-name> --verbose`

#### Problem: Arkivering misslyckades

**Lösning**:
1. Säkerställ att ändringen har klarat valideringen
2. Kontrollera om `openspec/archived/`-katalogen finns
3. Bekräfta att det inte finns några ocommittade kodändringar

## Avancerad användning

### Anpassade mallar

OpenSpec stöder anpassade mallar för förslag och specifikationer:

```bash
# Skapa en mallkatalog i projektets rotkatalog
mkdir -p .openspec/templates

# Skapa en anpassad förslagsmall
cat > .openspec/templates/proposal.md << 'EOF'
# Proposal: {{title}}

## Problem Statement
{{problem}}

## Proposed Solution
{{solution}}

## Custom Section
Your custom content here
EOF
```

### Konfigurationsfil

Skapa en `.openspec.json`-konfigurationsfil i projektets rotkatalog:

```json
{
  "version": "1.0",
  "templates": {
    "proposal": ".openspec/templates/proposal.md",
    "spec": ".openspec/templates/spec.md"
  },
  "validation": {
    "requireTests": true,
    "requireDocs": true
  },
  "hooks": {
    "pre-archive": "npm test"
  }
}
```

### Hooks

OpenSpec stöder körning av anpassade skript före och efter specifika operationer:

```json
{
  "hooks": {
    "pre-create": "scripts/check-permissions.sh",
    "post-archive": "scripts/notify-team.sh"
  }
}
```
