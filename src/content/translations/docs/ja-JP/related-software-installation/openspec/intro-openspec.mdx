# OpenSpec の紹介

OpenSpec は、構造化された提案や仕様を通じてソフトウェア開発プロセスを管理するための仕様駆動の開発ツールです。本ガイドでは、OpenSpec の主要な概念、ワークフロー、および利用方法を包括的に紹介します。

## OpenSpec とは？

OpenSpec は、ソフトウェア開発プロセス全体で提案（Proposals）、仕様（Specifications）、および変更（Changes）を管理するためのコマンドラインツールです。開発ワークフローを標準化し、アイデアから実装まで各機能が十分な設計とレビューを経ることを保証します。

### コア理念

OpenSpec は以下のコア理念に基づいています：

- **仕様を優先**：コードを書く前に明確な仕様を定義する
- **構造化された協力**：標準化されたドキュメント形式でチームの協力を促進する
- **トレーサビリティ**：提案から実装までのプロセスを記録する
- **品質保証**：仕様の検証を通じてコードが設計意図に沿っていることを保証する

### 主な機能

OpenSpec は以下の主要な機能を提供します：

1. **提案管理**：機能提案の作成、レビュー、および追跡
2. **仕様定義**：技術仕様ドキュメントの作成と保守
3. **変更追跡**：レビュー済みの変更履歴を記録
4. **仕様の検証**：コード変更が事前に定義された仕様に準拠しているかを確認
5. **チームコラボレーション**：複数人での共同開発とレビューのプロセスをサポート

## OpenSpec のワークフロー

OpenSpec の典型的なワークフローは以下の段階を含みます：

### 1. 提案（Proposal）の作成

新しい機能や大きな変更要求がある場合、まず提案を作成します：

```bash
openspec proposal create
```

提案には以下が含まれます：
- **問題の記述**：解決すべき問題や要求
- **提案する解決策**：提案された解決策の概要
- **影響範囲**：既存システムへの影響
- **代替案**：検討した他の案

### 2. 仕様（Specification）の作成

提案が承認されたら、詳細な技術仕様を作成します：

```bash
openspec spec create
```

仕様には以下が含まれます：
- **技術設計**：詳細な技術的実装計画
- **インターフェース定義**：API、データ構造などのインターフェース設計
- **実装の詳細**：主要な実装ロジックやアルゴリズム
- **テスト計画**：実装の正しさを検証する方法

### 3. 変更（Change）の作成

実装を開始する際には、開発プロセスを追跡するために変更を作成します：

```bash
openspec change create
```

変更には以下が含まれます：
- **関連する仕様**：この変更が実装する仕様
- **実装タスク**：具体的な開発タスクリスト
- **進捗追跡**：タスクの完了状況
- **コード変更**：実際のコード修正

### 4. 検証とアーカイブ

開発が完了したら、変更が仕様に準拠しているかを検証し、その後アーカイブします：

```bash
openspec change validate
openspec change archive
```

## OpenSpec のディレクトリ構成

OpenSpec はプロジェクトのルートに `openspec/` ディレクトリを作成し、以下のような構成になります：

```
openspec/
├── proposals/           # 提案ディレクトリ
│   ├── PROPOSAL-001.md
│   └── PROPOSAL-002.md
├── specs/              # 仕様ディレクトリ
│   ├── SPEC-001.md
│   └── SPEC-002.md
├── changes/            # 変更ディレクトリ
│   ├── feature-name/
│   │   ├── proposal.md
│   │   ├── design.md
│   │   ├── specs.md
│   │   └── tasks.md
│   └── another-feature/
└── archived/           # アーカイブ済みの変更
    └── feature-name/
```

### ディレクトリの説明

- **proposals/**：すべての提案ドキュメントを格納、各提案は独立した Markdown ファイル
- **specs/**：提案に対応する技術仕様ドキュメントを格納
- **changes/**：進行中の変更を格納、各変更はサブディレクトリ
- **archived/**：完了してアーカイブされた変更を格納

## コアコマンド

### 提案に関するコマンド

```bash
# 新しい提案を作成
openspec proposal create

# すべての提案を一覧表示
openspec proposal list

# 提案の詳細を表示
openspec proposal view <proposal-id>

# 提案を承認
openspec proposal approve <proposal-id>
```

### 仕様に関するコマンド

```bash
# 新しい仕様を作成
openspec spec create

# すべての仕様を一覧表示
openspec spec list

# 仕様の詳細を表示
openspec spec view <spec-id>

# 仕様を更新
openspec spec update <spec-id>
```

### 変更に関するコマンド

```bash
# 新しい変更を作成
openspec change create

# すべての変更を一覧表示
openspec change list

# 変更の詳細を表示
openspec change view <change-name>

# タスクを追加
openspec change add-task <change-name>

# 変更を検証
openspec change validate <change-name>

# 変更をアーカイブ
openspec change archive <change-name>
```

### 共通コマンド

```bash
# バージョンを表示
openspec --version

# ヘルプを表示
openspec --help

# 特定コマンドのヘルプを表示
openspec <command> --help
```

## HagiCode プロジェクトでの OpenSpec の利用

### プロジェクト統合

HagiCode プロジェクトでは、OpenSpec が開発ワークフローに統合されています：

1. **初期化チェック**：起動ウィザードが OpenSpec のインストールをチェック
2. **提案駆動開発**：すべての新機能は OpenSpec の提案プロセスを通す必要がある
3. **コードレビュー**：変更は OpenSpec の仕様に準拠している必要があり、マージされる
4. **ドキュメント同期**：OpenSpec のドキュメントはコードの実装と同期される

### バージョン要件

HagiCode プロジェクトでは **OpenSpec >=1.0.0 {'<'}2.0.0** のバージョン範囲を要求します。

インストールコマンド：

```bash
npm install -g @fission-ai/openspec@1
```

### ワークフローの例

以下は HagiCode プロジェクトで OpenSpec を使用する際の完全な例です：

#### ステップ 1：提案の作成

```bash
cd /path/to/hagicode-project
openspec proposal create
```

対話式プロンプトに入力します：
- 提案タイトル：ユーザー認証機能を追加
- 問題の記述：システムにログインと権限管理が必要
- 解決策：JWT ベースの認証システムを実装

#### ステップ 2：仕様の作成

提案が承認されたら、詳細な仕様を作成します：

```bash
openspec spec create
```

仕様に定義する内容：
- 認証フローの設計
- API インターフェース定義
- データベースのテーブル構造
- セキュリティに関する考慮点

#### ステップ 3：変更の作成

実装を開始します：

```bash
openspec change create
```

変更名を入力：`add-user-authentication`

#### ステップ 4：タスクの追加

```bash
openspec change add-task add-user-authentication
```

具体的なタスクを追加します：
- ユーザー登録エンドポイントを実装
- ログインエンドポイントを実装
- JWT ミドルウェアを実装
- 単体テストを作成

#### ステップ 5：実装と検証

コードを完成させたら、変更を検証します：

```bash
openspec change validate add-user-authentication
```

#### ステップ 6：アーカイブ

検証が通ったら、変更をアーカイブします：

```bash
openspec change archive add-user-authentication
```

## ベストプラクティス

### 提案の作成

1. **問題を明確に記述**：解決すべき問題を明確に定義し、あいまいな記述を避ける
2. **代替案を検討**：異なる解決策を列挙して比較する
3. **影響を評価**：既存システムへの影響とリスクを分析する
4. **簡潔に保つ**：提案は簡潔にまとめ、詳細は仕様に任せる

### 仕様の作成

1. **詳細に、しかし冗長にならない**：十分な詳細を提供しつつ、過度な設計を避ける
2. **図を使用する**：フローチャートやアーキテクチャ図などで説明を補助する
3. **インターフェースを定義する**：すべての公開インターフェースとデータ構造を明確に定義する
4. **境界ケースを考慮する**：例外的な状況やエラー処理を列挙する

### 変更管理

1. **小さな反復**：大きな機能は複数の小さな変更に分割する
2. **タイムリーな更新**：タスク状況とドキュメントを同期して保つ
3. **コードレビュー**：変更完了後にコードレビューを行う
4. **迅速なアーカイブ**：完了した変更は速やかにアーカイブして作業ディレクトリを整理する

### チームコラボレーション

1. **定期的なレビュー**：提案と仕様を定期的にレビューしてチームの合意を確認する
2. **ドキュメント優先**：コーディングを始める前にドキュメントを完成させる
3. **コミュニケーションを維持**：提案や仕様に議論と決定を記録する
4. **バージョン管理**：OpenSpec ドキュメントをバージョン管理に含める

## よくある利用シナリオ

### シナリオ 1：新機能の追加

1. 新機能要求を説明する提案を作成
2. チームで提案を議論し承認
3. 詳細な技術仕様を作成
4. 変更を作成して機能を実装
5. 変更を検証してアーカイブ

### シナリオ 2：既存コードのリファクタリング

1. リファクタリングの理由と目標を説明する提案を作成
2. 仕様でリファクタリング案と手順を定義
3. 変更を作成して段階的にリファクタリング
4. テストカバレッジを確保し、リファクタリング結果を検証
5. 変更をアーカイブ

### シナリオ 3：バグ修正

簡単なバグについては：
- 直接修正でき、完全な OpenSpec フローを必要としない場合がある

複雑なバグについては：
1. バグの根本原因を分析する提案を作成
2. 仕様で修正手順を定義
3. 変更を作成して修正を実装
4. 回帰を防ぐためのテストを追加
5. 変更をアーカイブ

### シナリオ 4：技術的負債の管理

1. 技術的負債の項目を列挙する提案を作成
2. 仕様で改善計画を定義
3. 変更を作成して段階的に負債を返済
4. 進捗と効果を追跡
5. 完了した改善をアーカイブ

## 他ツールとの統合

### Git 統合

OpenSpec は Git とシームレスに統合します：

```bash
# OpenSpec ドキュメントはバージョン管理に含める
git add openspec/
git commit -m "Add OpenSpec proposal for feature X"
```

### CI/CD 統合

CI/CD パイプラインで OpenSpec の仕様を検証します：

```yaml
# .github/workflows/openspec.yml
name: OpenSpec Validation
on: [pull_request]
jobs:
  validate:
    runs-on: ubuntu-latest
    steps:
      - uses: actions/checkout@v2
      - name: Install OpenSpec
        run: npm install -g @fission-ai/openspec@1
      - name: Validate changes
        run: openspec change validate
```

### IDE 統合

OpenSpec ドキュメントを編集するために Markdown をサポートするエディタを推奨します：
- VS Code + Markdown プラグイン
- JetBrains 系 IDE + Markdown サポート
- Vim/Neovim + Markdown プラグイン

## トラブルシューティング

### よくある問題

#### 問題：OpenSpec コマンドが見つからない

**解決方法**：
1. OpenSpec がインストールされていることを確認：`npm list -g @fission-ai/openspec`
2. PATH 環境変数に npm のグローバル bin ディレクトリが含まれているか確認
3. 再インストール：`npm install -g @fission-ai/openspec@1`

#### 問題：提案や変更が作成できない

**解決方法**：
1. カレントディレクトリがプロジェクトのルートであることを確認
2. `openspec/` ディレクトリが存在するか確認し、なければ作成
3. 書き込み権限を確認

#### 問題：検証が失敗する

**解決方法**：
1. 変更ディレクトリ内のドキュメントが完全か確認
2. 必要なタスクがすべて完了しているか確認
3. 詳細なエラー情報を確認：`openspec change validate <change-name> --verbose`

#### 問題：アーカイブが失敗する

**解決方法**：
1. 変更が検証に合格していることを確認
2. `openspec/archived/` ディレクトリが存在するか確認
3. 未コミットのコード変更がないことを確認

## 上級者向けの使い方

### カスタムテンプレート

OpenSpec はカスタムの提案および仕様テンプレートをサポートします：

```bash
# プロジェクトルートにテンプレートディレクトリを作成
mkdir -p .openspec/templates

# カスタム提案テンプレートを作成
cat > .openspec/templates/proposal.md << 'EOF'
# Proposal: {{title}}

## Problem Statement
{{problem}}

## Proposed Solution
{{solution}}

## Custom Section
Your custom content here
EOF
```

### 設定ファイル

プロジェクトルートに `.openspec.json` 設定ファイルを作成します：

```json
{
  "version": "1.0",
  "templates": {
    "proposal": ".openspec/templates/proposal.md",
    "spec": ".openspec/templates/spec.md"
  },
  "validation": {
    "requireTests": true,
    "requireDocs": true
  },
  "hooks": {
    "pre-archive": "npm test"
  }
}
```

### フック（Hooks）

OpenSpec は特定の操作の前後にカスタムスクリプトを実行することをサポートします：

```json
{
  "hooks": {
    "pre-create": "scripts/check-permissions.sh",
    "post-archive": "scripts/notify-team.sh"
  }
}
```
