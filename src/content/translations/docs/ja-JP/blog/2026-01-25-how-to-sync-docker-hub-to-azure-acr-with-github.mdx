## Docker Hub から Azure ACR への自動イメージ同期

> 本記事では、GitHub Actions と image-syncer ツールを使用して、Docker Hub から Azure Container Registry へのイメージ自動同期を実現する方法を紹介します。これにより、中国本土および一部の Azure リージョンでの Docker Hub アクセス速度の遅さという問題を解決し、Azure 環境でのイメージの可用性とデプロイ効率を向上させます。

## 背景 / はじめに

HagiCode プロジェクトは Docker イメージをコアランタイムコンポーネントとして使用し、メインイメージは Docker Hub にホストされています。プロジェクトの発展と Azure デプロイニーズの増大に伴い、以下のペインポイントに直面しました：
- イメージのプル速度が遅い：Docker Hub へのアクセスが中国本土および一部の Azure リージョンで制限されているため
- 単一のイメージソースへの依存は単一障害点のリスクを生む
- Azure 環境では Azure Container Registry を使用することで、より良いネットワークパフォーマンスと統合体験が得られる

これらの問題を解決するため、Docker Hub から Azure ACR への自動イメージ同期メカニズムを構築し、Azure 環境でのイメージのプル速度と可用性を向上させる必要があります。

## 技術ソリューションの比較

ソリューションを決定する際、複数の技術アプローチを比較しました：

### 1. image-syncer（最終選定）
- 増分同期：変更されたイメージレイヤーのみを同期し、ネットワーク転送を大幅に削減
- レジュームサポート：ネットワーク中断後も同期を再開可能
- 並行性制御：設定可能な並行スレッド数で大規模イメージの同期効率を向上
- 堅牢なエラー処理：内蔵の失敗時リトライメカニズム（デフォルト3回）
- 軽量デプロイ：依存関係のない単一バイナリ
- マルチレジストリ対応：Docker Hub、Azure ACR、Harbor などに対応

### 2. Docker CLI
- 増分同期非対応：毎回完全なイメージコンテンツをプルする必要がある
- 効率が低い：ネットワーク転送量が多く、実行時間が長い
- シンプルで使いやすい：使い慣れた `docker pull` / `docker push` コマンドを使用

### 3. Azure CLI
- 複雑度が高い：Azure CLI 認証のセットアップが必要
- 機能の制限：`az acr import` の機能がやや限定的
- ネイティブ統合：Azure サービスとの統合が良好

## アーキテクチャ設計の決定

### 決定 1：同期頻度を毎日 UTC 00:00 に設定
- イメージの鮮度とリソース消費のバランスを取る
- ビジネスピーク時を避け、他の操作への影響を最小限にする
- Docker Hub イメージは通常、毎日のビルド後に更新される

### 決定 2：すべてのイメージタグを同期
- Docker Hub との完全な一貫性を維持
- ユーザーに柔軟なバージョン選択を提供
- 複雑なタグフィルタリングルールを避け、同期ロジックをシンプルにする

### 決定 3：GitHub Secrets で認証情報を保存
- GitHub Actions のネイティブサポートで安全性が高い
- 設定が簡単で、管理とメンテナンスが容易
- リポジトリレベルのアクセス制御をサポート

## コアソリューション

## 実装手順

### 1. 準備段階
- Azure Portal で Azure Container Registry を作成または確認
- ACR アクセス認証情報（ユーザー名とパスワード）を作成
- Docker Hub イメージリポジトリへのアクセス権限を確認

### 2. GitHub Secrets の設定
GitHub リポジトリ設定に以下の Secrets を追加：
- AZURE_ACR_USERNAME: Azure ACR ユーザー名
- AZURE_ACR_PASSWORD: Azure ACR パスワード

### 3. GitHub Actions ワークフローの作成
`.github/workflows/sync-docker-acr.yml` でワークフローを設定：
- スケジュールトリガー：毎日 UTC 00:00
- 手動トリガー：`workflow_dispatch` をサポート
- 追加トリガー：`publish` ブランチへのプッシュ時に実行（高速同期用）

### 4. ワークフロー実行フロー

| 順序 | 参加者 | アクション | 説明 |
| --- | --- | --- | --- |
| 1 | GitHub Actions | ワークフローをトリガー | スケジュール、手動実行、または `publish` ブランチへのプッシュでトリガー |
| 2 | GitHub Actions → image-syncer | 同期ツールをダウンロードして実行 | 実際の同期フェーズに入る |
| 3 | image-syncer → Docker Hub | イメージマニフェストとタグリストを取得 | ソースリポジトリのメタデータを読み取る |
| 4 | image-syncer → Azure ACR | ターゲットリポジトリの既存イメージ情報を取得 | ターゲット側の現在の状態を判断 |
| 5 | image-syncer | ソースとターゲットの差分を比較 | 同期が必要なイメージレイヤーを特定 |
| 6 | image-syncer → Docker Hub | 変更されたイメージレイヤーをプル | 更新が必要なコンテンツのみを転送 |
| 7 | image-syncer → Azure ACR | 変更されたイメージレイヤーをプッシュ | 増分同期を完了 |
| 8 | image-syncer → GitHub Actions | 同期統計情報を返す | 結果、差分、エラー情報を含む |
| 9 | GitHub Actions | ログを記録してアーティファクトをアップロード | その後の監査とトラブルシューティングに役立つ |

## GitHub Actions ワークフローの実装

以下は実際に使用中のワークフロー設定（`.github/workflows/sync-docker-acr.yml`）です：

```yaml
name: Sync Docker Image to Azure ACR

on:
  schedule:
    - cron: "0 0 * * *" # 毎日 UTC 00:00
  workflow_dispatch: # 手動トリガー
  push:
    branches: [publish]

permissions:
  contents: read

jobs:
  sync:
    runs-on: ubuntu-latest

    steps:
      - name: Checkout code
        uses: actions/checkout@v4

      - name: Download image-syncer
        run: |
          # image-syncer バイナリをダウンロード
          wget https://github.com/AliyunContainerService/image-syncer/releases/download/v1.5.5/image-syncer-v1.5.5-linux-amd64.tar.gz
          tar -zxvf image-syncer-v1.5.5-linux-amd64.tar.gz
          chmod +x image-syncer

      - name: Create auth config
        run: |
          # 認証設定ファイルを生成 (YAML 形式)
          cat > auth.yaml <<EOF
          hagicode.azurecr.io:
            username: "${{ secrets.AZURE_ACR_USERNAME }}"
            password: "${{ secrets.AZURE_ACR_PASSWORD }}"
          EOF

      - name: Create images config
        run: |
          # イメージ同期設定ファイルを生成 (YAML 形式)
          cat > images.yaml <<EOF
          docker.io/newbe36524/hagicode: hagicode.azurecr.io/hagicode
          EOF

      - name: Run image-syncer
        run: |
          # 同期を実行 (新版の --auth と --images パラメータを使用)
          ./image-syncer --auth=./auth.yaml --images=./images.yaml --proc=10 --retries=3

      - name: Upload logs
        if: always()
        uses: actions/upload-artifact@v4
        with:
          name: sync-logs
          path: image-syncer-*.log
          retention-days: 7
```

## 設定の詳細

### 1. トリガー条件
- スケジュールトリガー：`cron: "0 0 * * *"` - 毎日 UTC 00:00 に実行
- 手動トリガー：`workflow_dispatch` - GitHub UI で手動実行を許可
- プッシュトリガー：`push: branches: [publish]` - publish ブランチへのプッシュ時にトリガー（高速同期用）

### 2. 認証設定（`auth.yaml`）
```yaml
hagicode.azurecr.io:
  username: "${{ secrets.AZURE_ACR_USERNAME }}"
  password: "${{ secrets.AZURE_ACR_PASSWORD }}"
```

### 3. イメージ同期設定
```yaml
docker.io/newbe36524/hagicode: hagicode.azurecr.io/hagicode
```
この設定は、`docker.io/newbe36524/hagicode` のすべてのタグを `hagicode.azurecr.io/hagicode` に同期することを意味します。

### 4. image-syncer パラメータ
- `--auth=./auth.yaml`: 認証設定ファイルのパス
- `--images=./images.yaml`: イメージ同期設定ファイルのパス
- `--proc=10`: 並行スレッド数を10に設定
- `--retries=3`: 失敗時に3回リトライ

## 注意事項とベストプラクティス

### 1. セキュリティの推奨事項
- Azure ACR パスワードを定期的にローテーション（90日ごとを推奨）
- プッシュのみの権限を持つ専用 ACR サービスアカウントを使用
- ACR のアクセスログを監視し、異常なアクセスを検出
- ログに認証情報を出力しない
- 認証情報をコードリポジトリにコミットしない

### 2. パフォーマンスの最適化
- `--proc` パラメータの調整：ネットワーク帯域幅に基づいて並行数を調整（5〜20を推奨）
- 同期時間の監視：同期時間が長すぎる場合は、並行数の減少を検討
- ログの定期クリーンアップ：適切な `retention-days` 値を設定（現在は7日）

### 3. トラブルシューティング

#### 問題 1：認証に失敗した
```
Error: failed to authenticate to hagicode.azurecr.io
```
解決策：
1. GitHub Secrets が正しく設定されているか確認
2. Azure ACR パスワードが期限切れでないか確認
3. ACR サービスアカウントの権限が正しいか確認

#### 問題 2：ネットワークタイムアウト
```
Error: timeout waiting for response
```
解決策：
1. ネットワーク接続を確認
2. 並行スレッド数を減らす（`--proc` パラメータ）
3. ネットワークが回復するまで待ってからワークフローを再トリガー

#### 問題 3：イメージの同期が不完全
```
Warning: some tags failed to sync
```
解決策：
1. 同期ログを確認し、失敗したタグを特定
2. 手動でワークフローをトリガーして再同期
3. Docker Hub のソースイメージが正常に動作しているか確認

### 4. 監視とアラート
- 定期的に `Actions` ページをチェックし、ワークフローの実行状態を確認
- GitHub 通知を設定し、ワークフロー失敗の通知を迅速に受信
- Azure ACR のストレージ使用量を監視
- 定期的にタグの一貫性を検証

## よくある質問と解決策

### Q1: すべてのタグではなく特定のタグを同期するには？

`images.yaml` 設定ファイルを変更：
```yaml
# latest と v1.0 タグのみ同期
docker.io/newbe36524/hagicode:latest: hagicode.azurecr.io/hagicode:latest
docker.io/newbe36524/hagicode:v1.0: hagicode.azurecr.io/hagicode:v1.0
```

### Q2: 複数のイメージリポジトリを同期するには？

`images.yaml` に複数行を追加：
```yaml
docker.io/newbe36524/hagicode: hagicode.azurecr.io/hagicode
docker.io/newbe36524/another-image: hagicode.azurecr.io/another-image
```

### Q3: 同期失敗後にリトライするには？
- 自動リトライ：image-syncer 内蔵のリトライメカニズム（デフォルト3回）
- 手動リトライ：GitHub Actions ページで `Re-run all jobs` をクリック

### Q4: 同期の詳細な進捗を確認するには？
- `Actions` ページでリアルタイムログを確認
- `sync-logs` アーティファクトをダウンロードして完全なログファイルを確認
- ログファイルには各タグの同期ステータスと転送速度が含まれます

### Q5: 同期にはどのくらい時間がかかりますか？
- 初回のフル同期：イメージのサイズにより通常10〜30分
- 増分同期：イメージの変更が小さい場合、通常2〜5分
- 時間はネットワーク帯域幅、イメージのサイズ、並行設定に依存します


### 3. 同期統計レポートの追加
```yaml
- name: Generate report
  if: always()
  run: |
    echo "## Sync Report" >> $GITHUB_STEP_SUMMARY

    echo "- Sync time: ${{ steps.sync.outputs.duration }}" >> $GITHUB_STEP_SUMMARY
```
