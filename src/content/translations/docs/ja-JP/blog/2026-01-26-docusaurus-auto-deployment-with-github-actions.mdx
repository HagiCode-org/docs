## HagiCode に GitHub Pages 自動デプロイサポートを追加

> プロジェクトの初期コードネームは PCode でしたが、現在は正式に HagiCode に改名されました。本記事では、プロジェクトに自動静的サイトデプロイ機能を導入し、コンテンツの公開を水を飲むほど簡単にする方法を記録します。

## 背景 / はじめに

HagiCode の開発過程で、非常に現実的な問題に直面しました：ドキュメントや提案が増え続けるにつれ、これらのコンテンツを効率的に管理・表示することが急務となりました。GitHub Pages を利用して静的サイトをホストすることにしましたが、手動でのビルドとデプロイは非常に面倒でした——毎回の変更ごとにローカルでビルド、パッケージングを行い、その後 `gh-pages` ブランチに手動でプッシュする必要がありました。これは効率が悪いだけでなく、エラーも発生しやすいものでした。

この問題を解決するため（主に怠けたいから）、自動化されたデプロイフローが必要でした。本記事では、HagiCode プロジェクトに GitHub Actions ベースの自動デプロイサポートを追加する方法を詳しく記録し、コンテンツの作成に集中できるようにします。

## 目標分析

手を動かす前に、まずこのタスクで何を達成すべきかを明確にする必要があります。斧を研ぐのに時間をかけることは、木を切る作業を遅らせるものではありませんから。

### コア要件

1.  **自動ビルド**：コードが `main` ブランチにプッシュされたとき、自動的にビルドプロセスをトリガーする。
2.  **自動デプロイ**：ビルド成功後、生成された静的ファイルを自動的に GitHub Pages にデプロイする。
3.  **環境の一貫性**：CI 環境とローカルビルド環境が一致していることを確保し、「ローカルでは動くが本番ではエラー」という厄介な状況を避ける。

### 技術選定

HagiCode は Docusaurus（非常に人気のある React 静的サイトジェネレーター）上に構築されているため、GitHub Actions を利用してこの目標を達成できます。

## GitHub Actions ワークフローの設定

GitHub Actions は GitHub が提供する CI/CD サービスです。リポジトリ内に YAML 形式のワークフローファイルを定義することで、さまざまな自動化タスクをカスタマイズできます。

### ワークフローファイルの作成

プロジェクトルートの `.github/workflows` フォルダに新しい設定ファイルを作成します。例えば `deploy.yml` のようにします。フォルダが存在しない場合は、先に手動で作成してください。

この設定ファイルのコアロジックは以下の通りです：

1.  **トリガー条件**：`main` ブランチの `push` イベントを監視。
2.  **実行環境**：最新の Ubuntu。
3.  **ビルドステップ**：
    -   コードをチェックアウト
    -   Node.js をインストール
    -   依存関係をインストール (`npm install`)
    -   静的ファイルをビルド (`npm run build`)
4.  **デプロイステップ**：公式の `action-gh-pages` を使用してビルド成果物を `gh-pages` ブランチにプッシュ。

### 主要な設定コード

以下は最終的に採用した設定テンプレートです：

```yaml
name: Deploy to GitHub Pages

# トリガー条件：main ブランチへのプッシュ時
on:
  push:
    branches:
      - main
    # 必要に応じてパスフィルターを追加できます。例えばドキュメントの変更時のみビルド
    # paths:
    #   - 'docs/**'
    #   - 'package.json'

# 権限の設定。GitHub Pages へのデプロイに重要
permissions:
  contents: read
  pages: write
  id-token: write

# 並行性制御：同じブランチの古いビルドをキャンセル
concurrency:
  group: "pages"
  cancel-in-progress: false

jobs:
  build:
    runs-on: ubuntu-latest
    steps:
      - name: Checkout
        uses: actions/checkout@v4
        # 注意：fetch-depth: 0 を設定する必要があります。そうしないとビルドバージョンが不正確になる可能性があります
        with:
          fetch-depth: 0

      - name: Setup Node
        uses: actions/setup-node@v4
        with:
          node-version: 20 # ローカル開発環境と一致させることを推奨
          cache: 'npm'     # キャッシュを有効にするとビルドプロセスが高速化

      - name: Install dependencies
        run: npm ci
        # npm install の代わりに npm ci を使用。より高速で厳格、CI 環境に適しています

      - name: Build website
        run: npm run build
        env:
          # サイトビルドに環境変数が必要な場合はここで設定
          # NODE_ENV: production
          # PUBLIC_URL: /your-repo-name

      - name: Upload artifact
        uses: actions/upload-pages-artifact@v3
        with:
          path: ./build # Docusaurus のデフォルト出力ディレクトリ

  deploy:
    environment:
      name: github-pages
      url: ${{ steps.deployment.outputs.page_url }}
    runs-on: ubuntu-latest
    needs: build
    steps:
      - name: Deploy to GitHub Pages
        id: deployment
        uses: actions/deploy-pages@v4
```

## 実装中の落とし穴

実際の運用でいくつかの問題に遭遇しました。皆さんがこれらを回避できるよう（あるいは事前に対策を準備できるよう）共有します。

### 1. GitHub Token の権限問題

最初の設定時、デプロイが常に 403 (Forbidden) エラーで失敗しました。長時間の調査の結果、GitHub のデフォルトの `GITHUB_TOKEN` には Pages への書き込み権限がないことが判明しました。

**解決策**：リポジトリの `Settings` → `Actions` → `General` → `Workflow permissions` で、**「Read and write permissions」**を選択してください。

### 2. ビルドディレクトリパスの誤り

Docusaurus はデフォルトでビルド済みの静的ファイルを `build` ディレクトリに配置します。ただし、一部のプロジェクト（Create React App はデフォルトで `build`、Vite はデフォルトで `dist`）では異なる設定になっている場合があります。Actions でファイルが見つからないエラーが報告された場合は、`docusaurus.config.js` の出力パス設定を確認してください。

### 3. サブパス問題

リポジトリがユーザーホームページ（`username.github.io`）ではなく、プロジェクトページ（`username.github.io/project-name` など）の場合、`baseUrl` を設定する必要があります。

`docusaurus.config.js` で：

```javascript
module.exports = {
  // ...
  url: 'https://hagicode.com', // HagiCode の URL
  baseUrl: '/', // ルートパスにデプロイ
  // ...
};
```
この詳細は見落とされやすいです。設定が間違っていると、リソースパスが解決できず、ページが真っ白になることがあります。

## 結果の確認

すべての設定を完了してコードをプッシュしたら、GitHub リポジトリの **Actions** タブで確認できます。

ワークフロー実行中は黄色の円が表示されます。緑色になれば成功です。赤色になった場合は、ログをクリックして問題を確認してください。通常、そこで追跡でき、ほとんどの場合はタイポやパス設定の間違いです。

ビルドが成功したら、`https://<ユーザー名>.github.io/<リポジトリ名>/` にアクセスすると、新しいサイトが表示されます。

## まとめ

GitHub Actions を導入することで、HagiCode ドキュメントサイトの自動デプロイを成功裏に実現しました。これは手動操作の時間を節約するだけでなく、より重要なことにリリースプロセスの標準化を保証します。現在、どのチームメイトがドキュメントを更新しても、`main` ブランチにマージすれば、数分後に最新のコンテンツがオンラインに反映されます。

**コアメリット**：
- **効率の向上**：「手動パッケージング、手動アップロード」から「コードがリリース」へ。
- **エラーの削減**：人為的な操作ミスの可能性を排除。
- **体験の向上**：開発者を退屈なデプロイ手順から解放し、コンテンツの品質に集中できるように。

CI/CD の設定は最初は少し面倒（特に各種権限やパスの問題）ですが、一度の投資で長期的なリターンは大きいです。すべての静的サイトプロジェクトに同様の自動化フローを導入することを強くお勧めします。
