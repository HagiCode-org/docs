## Docusaurus 3.x から Astro 5.x へ：HagiCode サイト移行実践の振り返り

> 本記事では、HagiCode 公式サイトを Docusaurus 3.x から Astro 5.x へ全面移行した全プロセスを振り返ります。Astro の Islands アーキテクチャによるパフォーマンスボトルネックの解決方法と、既存の React コンポーネント資産の活用について深く掘り下げ、ビルド速度と読み込みパフォーマンスの両面での向上を実現する方法を紹介します。


## 背景

2026 年 1 月、HagiCode の公式サイトに「心臓移植手術」を行いました。コアフレームワークを Docusaurus 3.x から Astro 5.x へ全面移行しました。これは衝動的な大規模リファクタリングではなく、熟慮の上での技術的決断でした。

移行前、当サイトは機能面では充実していましたが、徐々に「豊かすぎる病」が露呈していました。ビルド成果物の肥大化、JavaScript のペイロード過多、複雑なドキュメントページでの読み込み速度の低下などです。AI コードアシスタントプロジェクトとして、HagiCode はドキュメントと機能紹介の頻繁な更新が必要で、ビルド効率がリリース速度に直結します。同時に、より多くの開発者にプロジェクトを発見してもらうため、検索エンジン最適化（SEO）をさらに強化したいと考えていました。

これらの課題を解決するため、大胆な決断を下しました。ビルドシステム全体をゼロから構築し直し、Astro へ移行することです。この決断がもたらした変化は、皆さんが想像するよりもはるかに大きいものでした。

## HagiCode について

本記事で紹介するサイト移行の取り組みは、[HagiCode](https://github.com/HagiCode-org/site) プロジェクトでの実践経験に基づいています。

HagiCode は開発効率の向上に注力する AI コードアシスタントです。コア機能の反復改善だけでなく、開発者体験も重視しています。今回のサイト再構築も、ユーザーがドキュメントや公式サイトを閲覧する際に極限の読み込み速度を得られるようにするための取り組みです。

## なぜ成熟した Docusaurus を捨てたのか

React エコシステムにおいて、Docusaurus はドキュメントサイトの「標準解答」であり続けています。使い始めるのが簡単で、プラグインも豊富で、コミュニティも活発です。しかし、HagiCode の機能増加に伴い、その限界も感じるようになりました。

1.  **パフォーマンスのボトルネック**：Docusaurus は本質的に React SPA（シングルページアプリケーション）です。純粋な静的ページを書いていても、クライアント側で React ランタイムを読み込み、ハイドレーションを行う必要があり、シンプルなドキュメントページには重すぎます。
2.  **リソースの肥大化**：ページコンテンツが少なくても、バンドル後の JS サイズは比較的固定されており、モバイルユーザーやネットワーク環境が良くない状況には不向きです。
3.  **柔軟性の不足**：拡張は可能ですが、ビルドプロセスのカスタマイズにおいて、より低レベルの制御を求めていました。

Astro の登場はこれらの問題を解決するものです。全新的な「アイランドアーキテクチャ」（Islands Architecture）を提供します。デフォルトでは、Astro はゼロ JavaScript の静的 HTML を生成し、インタラクションが必要なコンポーネントのみが「アクティブ化」されて JS を読み込みます。つまり、サイトの大部分が純粋な HTML となり、極めて高速です。

## 移行のコア戦略：アーキテクチャのスムーズな移行

移行は単なるコピペではなく、思考パターンの転換です。Docusaurus の「全 React モード」から Astro の「Core + Islands」モードへの切り替えです。

### 1. 設定システムの再構築

まず、`docusaurus.config.ts` から `astro.config.mjs` への移行が必要です。これは単なるファイル名の変更ではなく、ルーティングとビルドロジックの書き直しです。

Docusaurus ではすべてがプラグイン、Astro ではすべてがインテグレーションです。サイトのベースパス、ビルド出力モード（静的 vs SSR）、リソース圧縮戦略を再定義する必要があります。

**移行前：**
```typescript
// docusaurus.config.ts
export default {
  title: 'HagiCode',
  url: 'https://hagicode.com',
  baseUrl: '/',
  // ... その他設定
};
```

**移行後：**
```javascript
// astro.config.mjs
import { defineConfig } from 'astro/config';
import react from '@astrojs/react';

export default defineConfig({
  integrations: [react()],
  site: 'https://hagicode.com',
  base: '/',
  // 静的アセットの最適化設定
  build: {
    inlineStylesheets: 'auto',
  },
});
```

### 2. React コンポーネントの取捨選択と改造

これが移行で最も頭を悩ませる部分です。既存のサイトには多くの React コンポーネント（Tabs コンポーネント、コードハイライト、フィードバックボタンなど）がありました。すべて捨てるのは惜しいですし、すべて残すと JS ペイロードが重くなります。

HagiCode では**段階的ハイドレーション**戦略を採用しました。
*   **純粋な静的コンポーネント**：表示系コンテンツ（ヘッダー、フッター、プレーンテキストドキュメント）については、Astro コンポーネント（`.astro` ファイル）に書き直し、ビルド時に直接 HTML としてレンダリングします。
*   **インタラクティブなアイランド**：インタラクションを維持する必要があるコンポーネント（テーマ切り替え、Tabs 切り替え、コードブロックのコピーボタン）については、React 実装を残し、`client:load` または `client:visible` ディレクティブを追加します。

例えば、ドキュメントでよく使用する Tabs コンポーネント：

```jsx
// src/components/Tabs.jsx
import { useState } from 'react';
import './Tabs.css'; // スタイルのインポート

export default function Tabs({ items }) {
  const [activeIndex, setActiveIndex] = useState(0);
  // ... 状態ロジック
  return (
    <div className="tabs-wrapper">
      {/* レンダリングロジック */}
    </div>
  );
}
```

Markdown で使用する際、Astro に「このコンポーネントは JS が必要」であることを明示します。

```markdown
// src/content/docs/example.mdx
import Tabs from '../../components/Tabs.jsx';

<!-- ビューポートに入った時だけ JS を読み込む -->
<Tabs client:visible items={...} />
```

これにより、ビューポート外のインタラクティブコンポーネントが帯域を占有せず、ファーストビューの読み込み速度が大幅に最適化されます。

### 3. スタイルシステムの適応：CSS Modules から Scoped へ

Docusaurus はデフォルトで CSS Modules をサポートし、Astro は Scoped CSS（`<style>` タグ経由）を推奨しています。両者の核となる考え方はスタイルの分離ですが、構文が異なります。

HagiCode の移行では、大部分の複雑な CSS Modules を Astro の Scoped スタイルに分解しました。これは実は良いことで、`.astro` ファイルではスタイルとテンプレートが同じファイルに書かれ、メンテナンスがより直感的になります。

**改造前：**
```css
/* Tabs.module.css */
.wrapper { background: var(--ifm-background-color); }
```

**改造後 (Astro Scoped):**
```html
<!-- Tabs.astro -->
<div class="tabs-wrapper">
  <slot />
</div>

<style>
  .tabs-wrapper {
    /* CSS 変数を直接使用し、テーマに適応 */
    background: var(--bg-color);
    padding: 1rem;
  }
</style>
```

同時に、グローバル CSS 変数システムを統一し、Astro の環境認識能力を活用して、ダークモードのページ間切替がシームレスに行われるようにしました。

## 実践での落とし穴と解決策

HagiCode の実際の移行プロセスで直面した落とし穴の中から、最も典型的なものをいくつか紹介します。

### 1. パスと環境変数の課題

HagiCode はサブパスデプロイ（例えば GitHub Pages のサブディレクトリへのデプロイ）をサポートしています。Docusaurus では `baseUrl` を自動処理していましたが、Astro では画像リンクや API リクエストの処理により注意が必要です。

環境変数メカニズムを導入して統一管理しました。

```javascript
// ビルドスクリプトでパスを処理
const getBasePath = () => import.meta.env.VITE_SITE_BASE || '/';
```

コード内に `/` で始まるパスをハードコードしないでください。開発環境と本番環境、または `base` パスを設定した後にリソースが 404 になる原因になります。

### 2. CommonJS スクリプトの互換性

旧サイトには Node.js スクリプト（Metrics データの自動取得、sitemap の更新など用）があり、それらは CommonJS (`require`) で書かれていました。Astro とモダンビルドツールは ES Modules (`import`/`export`) を全面的に採用しています。

同様のスクリプトがある場合は、すべて ES Modules にリファクタリングしてください。これは時代の流れであり、早めに改めておく方が安心です。

```javascript
// 旧方式
const fs = require('fs');

// 新方式
import fs from 'fs';
```

### 3. SEO とリダイレクトを忘れずに

検索エンジンはすでに HagiCode の旧 Docusaurus ページをインデックスしています。Astro に直接切り替えて URL 構造が変わると、大量の 404 が発生し、権重が大幅に低下します。

Astro でリダイレクトルールを設定しました。

```javascript
// astro.config.mjs
export default defineConfig({
  redirects: {
    '/docs/old-path': '/docs/new-path',
    // 旧リンクから新リンクへの一括マッピング
  }
});
```

またはサーバー設定レベルで処理します。旧リンクが 301 リダイレクトで新しいアドレスに転送されることを確認してください。これは SEO に不可欠です。

## まとめ

Docusaurus から Astro への移行は、HagiCode にとって単なるフレームワークアップグレードではなく、「パフォーマンスファースト」の理念の実践でした。

**得られた成果：**
*   **極限の Lighthouse スコア**：移行後、HagiCode サイトのパフォーマンススコアは容易にほぼ満点に近づきました。
*   **高速なビルド**：Astro のインクリメンタルビルドにより、ドキュメント更新の公開時間が半分に短縮されました。
*   **柔軟性の維持**：Islands アーキテクチャにより、インタラクティブ機能を一切犠牲にすることなく、必要な場所で React を使い続けることができます。

ドキュメント型サイトを保守していて、バンドルサイズや読み込み速度に悩まされているなら、Astro を試してみてください。移行には手術が必要ですが（例えばコンポーネントを一つずつ移植したり）、得られるのはシルクのような滑らかなユーザー体験であり、絶対に価値があります。

本記事で紹介したビルドシステムは、HagiCode の開発過程で実際に直面した課題を解決し、最適化したものです。この取り組みが価値あるものであれば、HagiCode 自体も注目に値するはずです。
