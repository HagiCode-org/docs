## コア分析：なぜ Nuke なのか

「ビルドシステムなんて Make や Gradle、あるいは単純に Shell スクリプトで十分じゃない？なぜわざわざ C# のやつを使う必要があるの？」と思うかもしれません。

これは実は良い質問です。Nuke のコアな魅力は、私たちが最もよく知っているプログラミング言語の機能をビルドスクリプトの世界にもたらした点にあります。

### 1. ビルドフローのモジュール化：Target の芸術

Nuke の設計理念は非常に明確です：**すべては Target である**。

従来のスクリプトでは、何百行もの線形実行コードを書き、複雑に入り組んだロジックになることがよくあります。Nuke では、ビルドフローを独立した `Target`（目標）に分解します。各 Target は一つのことだけを担当します：

- `Clean`：出力ディレクトリをクリーンアップ
- `Restore`：依存パッケージを復元
- `Compile`：コードをコンパイル
- `Test`：ユニットテストを実行

この設計は単一責任の原則に非常によく合致しています。ブロックを組み立てるように、これらの Target を自由に組み合わせることができます。さらに重要なのは、Nuke は Target 間の依存関係を定義できることです。例えば、`Test` を実行したい場合、システムは自動的に `Compile` が既に実行されているか確認し、`Compile` を実行したい場合は、当然 `Restore` が先に完了している必要があります。

この依存関係グラフはロジックを明確にするだけでなく、実行効率も大幅に向上させます。Nuke は最適な実行パスを自動的に分析するからです。

### 2. 型安全性：タイプミスの悪夢にさよなら

Python でビルドスクリプトを書いたことがある方なら、こんな経験があるはずです：スクリプトが5分間実行された後、`Confi.guration` のスペルミスか、数値を期待するパラメータに文字列を渡したことが原因でエラーになる。

C# でビルドスクリプトを書く最大の利点は **型安全性** です。これは以下を意味します：

- **コンパイル時チェック**：コードを書いている最中に IDE が間違いを教えてくれるので、実行時まで待つ必要がありません。
- **安全なリファクタリング**：変数名やメソッド名を変更したい場合、IDE のリファクタリング機能でワンクリック解決。不安なグローバル検索置換は不要です。
- **インテリジェント補完**：強力な IntelliSense が自動的にコードを補完するので、難解な API をドキュメントで調べる必要がありません。

### 3. クロスプラットフォーム：統一されたビルド体験

以前は Windows で `.bat` を書き、Linux で `.sh` を書き、両方を互換させるために Python スクリプトを書くということがありました。今は、.NET Core（現在の .NET 5+）が動く場所ならどこでも Nuke が動きます。

つまり、チームメンバーが Windows、Linux、macOS のどれを使っていても、Visual Studio、VS Code、Rider のどれを使っていても、全員が同じロジックを実行します。これは「自分の環境では動く」という環境差異による問題を大幅に減らします。

### 4. パラメータと設定管理

Nuke は非常にエレガントなパラメータ解析メカニズムを提供しています。`string[] args` を手動で解析する必要はありません。プロパティを定義して `[Parameter]` 属性を追加するだけで、Nuke がコマンドライン引数と設定ファイルのマッピングを自動的に処理します。

例えば、ビルド設定を簡単に定義できます：

```csharp
[Parameter("Configuration to build - Default is 'Debug'")]
readonly Configuration BuildConfiguration = IsLocalBuild ? Configuration.Debug : Configuration.Release;

Target Compile => _ => _
    .DependsOn(Restore)
    .Executes(() =>
    {
        // ここで BuildConfiguration を使用。型安全
        DotNetBuild(s => s
            .SetConfiguration(BuildConfiguration)
            .SetProjectFile(SolutionFile));
    });
```

この書き方は直感的でエラーが起きにくいです。

## 実践ガイド：プロジェクトでの適用方法

理論は簡単、実践が大事。HagiCode プロジェクトで具体的にどう適用したか見てみましょう。

### 1. プロジェクト構造の計画

ビルドスクリプトがプロジェクトルートを汚染したくはありませんし、一部の Java プロジェクトのように深すぎるディレクトリ構造にもしたくありません。そのため、Nuke 関連のビルドファイルはすべて `nukeBuild/` ディレクトリに統一して配置しました。

この利点は明確です：
- プロジェクトルートがすっきりする
- ビルドロジックが凝集し、管理しやすい
- 新しいメンバーが参加したとき、「ああ、ここがビルド関連のロジックだ」と一目でわかる

### 2. 明確な Target 依存チェーンの設計

Target を設計する際、私たちは一つの原則に従いました：**原子化 + 依存フロー**。

各 Target は十分に小さく、一つのことだけを行うべきです。例えば、`Clean` はファイルの削除だけを行い、その中でパッケージングも一緒に行うべきではありません。

推奨される依存フローは概ね以下のようになります：

`Clean` -> `Restore` -> `Compile` -> `Test` -> `Pack`

もちろん、これは絶対ではありません。例えばテストだけを実行してパッケージングはしたくない場合、Nuke では `nuke Test` を直接実行でき、必要な `Restore` と `Compile` ステップを自動的に処理します。

### 3. 堅牢なエラー処理とロギング

ビルドスクリプトで最も嫌なのは何でしょうか？それは不明確なエラーメッセージです。例えば、ビルドが失敗したのにログに "Error: 1" としか表示されないと、誰でも発狂しそうになります。

Nuke では、C# の例外処理メカニズムを直接使用できるため、エラーを非常に正確にキャッチして報告できます。

```csharp
Target Publish => _ => _
    .DependsOn(Test)
    .Executes(() =>
    {
        try
        {
            // NuGet への公開を試みる
            DotNetNuGetPush(s => s
                .SetTargetPath(ArtifactPath)
                .SetSource("https://api.nuget.org/v3/index.json")
                .SetApiKey(ApiKey));
        }
        catch (Exception ex)
        {
            Log.Error($"公開に失敗しました。キーが正しいか確認してください: {ex.Message}");
            throw; // ビルドプロセスが非ゼロの終了コードで終了するようにする
        }
    });
```

### 4. テストで品質を保護

ビルドスクリプトもコードであり、テストが必要です。Nuke ではビルドフローに対するテストを書くことができ、ビルドロジックを変更したときに既存のリリースプロセスが壊れないことを確保できます。これは継続的インテグレーション（CI）パイプラインで特に重要です。

## まとめ

Nuke を導入することで、HagiCode のビルドプロセスはかつてないほどスムーズになりました。これは単なるツールの置き換えではなく、エンジニアリング思考のアップグレードです。

**何を得たのか？**
- **保守性**：コードが設定になり、ロジックが明確で、新メンバーもすぐに使いこなせる。
- **安定性**：強い型チェックにより、90%以上の初歩的なミスが削減された。
- **一貫性**：クロスプラットフォームの統一体験により、環境差異が排除された。

以前のビルドスクリプトの作成が「暗闇の中を手探り」なら、Nuke の使用は「電気をつけて夜道を歩く」ようなものです。デバッグの難しいスクリプト言語のメンテナンスにうんざりしているなら、ビルドロジックも C# の世界に移行してみてください。ビルドシステムがこんなにエレガントになれることに気づくかもしれません。
