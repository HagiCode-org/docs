## De 120 Segundos para 45 Segundos: Otimizando o Desempenho de Build do Vite com Worker Threads

> Ao lidar com projetos frontend grandes, o build de código para produção muitas vezes faz esperar muito tempo. Este artigo compartilha como usar Node.js Worker Threads para reduzir o tempo de ofuscação de código no build do Vite de 120 segundos para 45 segundos, com detalhes da implementação e lições aprendidas no projeto HagiCode.

## Contexto

Em nossa prática de engenharia frontend, à medida que a escala do projeto cresce, os problemas de eficiência de build se tornam cada vez mais evidentes. Especialmente no fluxo de build para produção, para proteger a lógica do código-fonte, geralmente introduzimos ferramentas de ofuscação JavaScript (como `javascript-obfuscator`). Essa etapa, embora necessária, exige muito processamento e consome intensamente os recursos da CPU.

No estágio inicial de desenvolvimento do projeto **HagiCode**, encontramos um gargalo de desempenho muito problemático: o tempo de build de produção deteriorava-se rapidamente com o aumento da quantidade de código.

**Pontos de dor específicos:**
- Execução de tarefa de ofuscação em thread único serial, com um único núcleo da CPU a 100% e os outros núcleos ociosos
- O tempo de build saltou de 30 segundos inicial para 110-120 segundos
- O fluxo de verificação de build após cada modificação de código era extremamente longo, prejudicando severamente a eficiência de iteração de desenvolvimento
- No pipeline de CI/CD, a etapa de build se tornou a parte mais demorada

**Por que o HagiCode tinha essa necessidade?**
O HagiCode é um assistente de codificação com IA, cuja arquitetura frontend contém lógica de negócios complexa e módulos de interação com IA. Para garantir a segurança do código central, ativamos ofuscação de alta intensidade obrigatoriamente nas releases de produção. Diante de uma espera de build de dois minutos, decidimos realizar uma otimização profunda de desempenho no sistema de build.

## Análise: Buscando o Ponto de Partida para Resolver o Gargalo de Desempenho

Antes de começar a resolver o problema de desempenho, precisamos organizar as ideias e determinar a melhor solução técnica.

### Decisão Central: Por que Escolher Worker Threads?

Existem principalmente três abordagens para implementar computação paralela no ambiente Node.js:
1. **child_process**: criar processos filhos independentes
2. **Web Workers**: principalmente usados no lado do navegador
3. **worker_threads**: suporte nativo a multithreading do Node.js

Após análise comparativa, o HagiCode optou pelo **Worker Threads** pelos seguintes motivos:
- **Custo zero de serialização**: Worker Threads estão no mesmo processo e podem compartilhar memória por meio de `SharedArrayBuffer` ou transferência de controle, evitando o grande custo de serialização na comunicação entre processos.
- **Suporte nativo**: suporte integrado no Node.js 12+, sem necessidade de introduzir dependências pesadas adicionais.
- **Contexto unificado**: depuração e registro de logs são mais convenientes do que processos filhos.

### Granularidade de Tarefa: Como Dividir a Tarefa de Ofuscação?

Ofuscar um único arquivo JS Bundle enorme é difícil de paralelizar (porque o código tem dependências), mas os artefatos de build do Vite são compostos por múltiplos **Chunks**. Isso nos dá uma fronteira de paralelismo natural:

- **Independência**: os diferentes Chunks após o empacotamento do Vite têm suas dependências desacopladas e podem ser processados com segurança em paralelo.
- **Granularidade adequada**: tipicamente um projeto tem 10-30 Chunks, uma quantidade ideal para escalonamento paralelo.
- **Fácil integração**: o hook `generateBundle` do plugin Vite permite interceptar e processar esses Chunks antes da geração dos arquivos.

### Design de Arquitetura

Projetamos um sistema de processamento paralelo com quatro componentes centrais:

1. **Task Splitter**: percorre o objeto bundle do Vite, filtra arquivos que não precisam de ofuscação (como vendor), e gera uma fila de tarefas.
2. **Worker Pool Manager**: gerencia o ciclo de vida dos Workers, responsável pela distribuição, recuperação e retry de erros das tarefas.
3. **Progress Reporter**: gera saída do progresso de build em tempo real, eliminando a ansiedade de espera do usuário.
4. **ObfuscationWorker**: thread de trabalho que executa efetivamente a lógica de ofuscação.

## Solução: Implementação Prática do Código

Com base na análise acima, começamos a implementar este sistema de ofuscação paralela.

### 1. Configuração do Plugin Vite

Primeiro, integramos o plugin de ofuscação paralela em `vite.config.ts`. A configuração é muito intuitiva, basta especificar o número de Workers e as regras de ofuscação.

```typescript
import { defineConfig } from 'vite'
import { parallelJavascriptObfuscator } from './buildTools/plugin'

export default defineConfig(({ mode }) => {
  const isProduction = mode === 'production'

  return {
    build: {
      rollupOptions: {
        ...(isProduction
          ? {
              plugins: [
                parallelJavascriptObfuscator({
                  enabled: true,
                  // Ajustado automaticamente com base no número de núcleos da CPU, recomenda-se deixar um núcleo para a thread principal
                  workerCount: 4,
                  retryAttempts: 3,
                  fallbackToMainThread: true, // Degrada automaticamente para single-thread em caso de erro
                  // Filtrar vendor chunks, geralmente não é necessário ofuscar bibliotecas de terceiros
                  isVendorChunk: (fileName: string) => fileName.includes('vendor-'),
                  obfuscationConfig: {
                    compact: true,
                    controlFlowFlattening: true,
                    deadCodeInjection: true,
                    disableConsoleOutput: true,
                    // ... mais opções de ofuscação
                  },
                }),
              ],
            }
          : {}),
      },
    },
  }
})
```

### 2. Implementação da Lógica do Worker

O Worker é a unidade de execução de tarefas. Precisamos definir bem as estruturas de dados de entrada e saída.

**Atenção**: embora o código aqui seja simples, há algumas armadilhas a observar. Por exemplo, a verificação de nulo do `parentPort` e o tratamento de erros. Na prática do HagiCode, descobrimos que algumas sintaxes ES6 específicas podem causar falhas no ofuscador, então adicionamos proteção com `try-catch`.

```typescript
import { parentPort } from 'worker_threads'
import javascriptObfuscator from 'javascript-obfuscator'

export interface ObfuscationTask {
  chunkId: string
  code: string
  config: any
}

export interface ObfuscationResult {
  chunkId: string
  obfuscatedCode: string
  error?: string
}

// Ouvir tarefas enviadas pela thread principal
if (parentPort) {
  parentPort.on('message', async (task: ObfuscationTask) => {
    try {
      // Executar ofuscação
      const obfuscated = javascriptObfuscator.obfuscate(task.code, task.config)
      const result: ObfuscationResult = {
        chunkId: task.chunkId,
        obfuscatedCode: obfuscated.getObfuscatedCode(),
      }
      // Enviar resultado de volta para a thread principal
      parentPort?.postMessage(result)
    } catch (error) {
      // Tratar exceção, garantindo que a falha de um único Worker não bloqueie todo o build
      const result: ObfuscationResult = {
        chunkId: task.chunkId,
        obfuscatedCode: '',
        error: error instanceof Error ? error.message : 'Unknown error',
      }
      parentPort?.postMessage(result)
    }
  })
}
```

### 3. Gerenciador de Pool de Workers

Este é o núcleo de toda a solução. Precisamos manter um pool de Workers de tamanho fixo, adotando a estratégia **FIFO (First In, First Out)** para escalonamento de tarefas.

```typescript
import { Worker } from 'worker_threads'
import os from 'os'

export class WorkerPool {
  private workers: Worker[] = []
  private taskQueue: Array<{
    task: ObfuscationTask
    resolve: (result: ObfuscationResult) => void
    reject: (error: Error) => void
  }> = []

  constructor(options: WorkerPoolOptions = {}) {
    // Padrão é número de núcleos - 1, deixando algum espaço para a thread principal
    const workerCount = options.workerCount ?? Math.max(1, (os.cpus().length || 4) - 1)

    for (let i = 0; i < workerCount; i++) {
      this.createWorker()
    }
  }

  private createWorker() {
    const worker = new Worker('./worker.ts')

    worker.on('message', (result) => {
      // Após a conclusão da tarefa, retirar a próxima tarefa da fila
      const nextTask = this.taskQueue.shift()
      if (nextTask) {
        this.dispatchTask(worker, nextTask)
      } else {
        // Se não houver tarefas pendentes, marcar o Worker como ocioso
        this.activeWorkers.delete(worker)
      }
    })

    this.workers.push(worker)
  }

  // Submeter tarefa ao pool
  public runTask(task: ObfuscationTask): Promise<ObfuscationResult> {
    return new Promise((resolve, reject) => {
      const job = { task, resolve, reject }
      const idleWorker = this.workers.find(w => !this.activeWorkers.has(w))

      if (idleWorker) {
        this.dispatchTask(idleWorker, job)
      } else {
        this.taskQueue.push(job)
      }
    })
  }

  private dispatchTask(worker: Worker, job: any) {
    this.activeWorkers.set(worker, job.task)
    worker.postMessage(job.task)
  }
}
```

### 4. Relatório de Progresso

Esperar é doloroso, especialmente quando não se sabe quanto tempo falta. Adicionamos um simples relator de progresso que fornece feedback em tempo real sobre o estado atual.

```typescript
export class ProgressReporter {
  private completed = 0
  private readonly total: number
  private readonly startTime: number

  constructor(total: number) {
    this.total = total
    this.startTime = Date.now()
  }

  increment(): void {
    this.completed++
    this.report()
  }

  private report(): void {
    const now = Date.now()
    const elapsed = now - this.startTime
    const percentage = (this.completed / this.total) * 100

    // Estimativa simples de ETA
    const avgTimePerChunk = elapsed / this.completed
    const remaining = (this.total - this.completed) * avgTimePerChunk

    console.log(
      `[Parallel Obfuscation] ${this.completed}/${this.total} chunks completed (${percentage.toFixed(1)}%) | ETA: ${(remaining / 1000).toFixed(1)}s`
    )
  }
}
```

## Prática: Resultados e Lições Aprendidas

Após implantar esta solução, o desempenho de build do projeto HagiCode teve uma melhoria imediata.

### Dados de Benchmark de Desempenho

Testamos no seguinte ambiente:
- CPU: Intel Core i7-12700K (12 núcleos / 20 threads)
- RAM: 32GB DDR4
- Node.js: v18.17.0
- SO: Ubuntu 22.04

**Comparação de resultados:**
- **Single-thread (antes da otimização)**: 118 segundos
- **4 Workers**: 55 segundos (melhoria de **53%**)
- **8 Workers**: 48 segundos (melhoria de **60%**)
- **12 Workers**: 45 segundos (melhoria de **62%**)

Pode-se notar que os ganhos não são lineares. Quando o número de Workers ultrapassa 8, a margem de melhoria diminui. Isso se deve principalmente à uniformidade da distribuição de tarefas e ao gargalo de largura de banda da memória.

### Problemas Comuns e Soluções

No uso real do HagiCode, também encontramos algumas armadilhas, que compartilhamos aqui:

**P1: O tempo de build não diminuiu significativamente, e até ficou mais lento?**
- **Causa**: A criação do Worker em si tem custo, ou o número excessivo de Workers causa troca de contexto frequente.
- **Solução**: Recomenda-se definir o número de Workers como `número de núcleos da CPU - 1`. Além disso, verifique se há algum Chunk individual particularmente grande (por exemplo, > 5MB), pois esse arquivo "gigante" se torna o gargalo e você pode considerar otimizar a estratégia de divisão de código.

**P2: Ocasionalmente o Worker falha e o build dá erro?**
- **Causa**: Algumas sintaxes de código específicas podem causar erros internos no ofuscador.
- **Solução**: Implementamos um **mecanismo de degradação automática**. Quando o número de falhas consecutivas do Worker atinge um limite, o plugin automaticamente reverte para o modo single-thread, garantindo que o build não seja interrompido. Ao mesmo tempo, registra o nome do arquivo com erro para correção direcionada posterior.

**P3: Consumo excessivo de memória (OOM)?**
- **Causa**: Cada Worker precisa de espaço de memória independente para carregar o ofuscador e analisar a AST.
- **Solução**:
  - Reduzir o número de Workers.
  - Aumentar o limite de memória do Node.js: `NODE_OPTIONS="--max-old-space-size=4096" npm run build`.
  - Garantir que não haja referências a objetos grandes desnecessários mantidos dentro do Worker.

## Conclusão

Com a introdução do Node.js Worker Threads, reduzimos com sucesso o tempo de build de produção do projeto HagiCode de 120 segundos para cerca de 45 segundos, melhorando enormemente a experiência de desenvolvimento e a eficiência do CI/CD.

O núcleo desta solução está em:
1. **Divisão adequada de tarefas**: usar os Chunks do Vite como unidades de paralelismo.
2. **Controle de recursos**: usar um pool de Workers para evitar esgotamento de recursos.
3. **Design tolerante a falhas**: mecanismo de degradação automática garante a estabilidade do build.
