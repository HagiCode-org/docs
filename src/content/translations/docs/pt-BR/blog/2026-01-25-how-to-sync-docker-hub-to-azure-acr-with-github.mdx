## Automatizando a Sincronização de Imagens do Docker Hub para o Azure ACR

> Este artigo explica como usar GitHub Actions e a ferramenta image-syncer para automatizar a sincronização de imagens do Docker Hub para o Azure Container Registry, resolvendo o problema de acesso lento ao Docker Hub na China continental e em algumas regiões do Azure, melhorando a disponibilidade de imagens e a eficiência de implantação em ambientes Azure.

## Contexto / Introdução

O projeto HagiCode usa imagens Docker como componentes centrais de runtime, com as principais imagens hospedadas no Docker Hub. Conforme o projeto evoluiu e as necessidades de implantação no Azure cresceram, encontramos os seguintes pontos de dor:
- Pull de imagens lento, porque o acesso ao Docker Hub é limitado na China continental e em algumas regiões do Azure
- Depender de uma única fonte de imagens cria um risco de ponto único de falha
- Usar o Azure Container Registry em ambientes Azure oferece melhor desempenho de rede e experiência de integração

Para resolver esses problemas, precisamos estabelecer um mecanismo automatizado de sincronização de imagens que sincronize regularmente as imagens do Docker Hub para o Azure ACR, garantindo que os usuários obtenham velocidades de pull mais rápidas e maior disponibilidade em ambientes Azure.

## Comparação de Soluções Técnicas

Ao definir a solução, comparamos múltiplas abordagens técnicas:

### 1. image-syncer (escolha final)
- Sincronização incremental: sincroniza apenas as camadas de imagem alteradas, reduzindo significativamente a transferência de rede
- Suporte a retomada: a sincronização pode ser retomada após interrupções de rede
- Controle de concorrência: suporta configuração de concorrência para melhorar a eficiência de sincronização de imagens grandes
- Tratamento robusto de erros: mecanismo integrado de retry para falhas (3 vezes por padrão)
- Implantação leve: binário único sem dependências
- Suporte a múltiplos repositórios: compatível com Docker Hub, Azure ACR, Harbor e outros

### 2. Docker CLI
- Sem suporte a sincronização incremental: cada execução requer o pull do conteúdo completo da imagem
- Menor eficiência: grande volume de transferência de rede e tempo de execução mais longo
- Simples e fácil de usar: baseado nos comandos familiares `docker pull` / `docker push`

### 3. Azure CLI
- Maior complexidade: requer configuração de autenticação do Azure CLI
- Limitações funcionais: `az acr import` é relativamente limitado
- Integração nativa: integra-se bem com os serviços do Azure

## Decisões de Design de Arquitetura

### Decisão 1: Definir a frequência de sincronização para diariamente às 00:00 UTC
- Equilibra a atualidade das imagens com o consumo de recursos
- Evita horários de pico de negócios e reduz o impacto em outras operações
- Imagens do Docker Hub geralmente são atualizadas após as builds diárias

### Decisão 2: Sincronizar todas as tags de imagem
- Mantém consistência total com o Docker Hub
- Fornece escolhas de versão flexíveis para os usuários
- Simplifica a lógica de sincronização evitando regras complexas de filtragem de tags

### Decisão 3: Armazenar credenciais no GitHub Secrets
- Suportado nativamente pelo GitHub Actions com forte segurança
- Simples de configurar e fácil de gerenciar e manter
- Suporta controle de acesso em nível de repositório

## Avaliação de Riscos e Mitigação

### Risco 1: Vazamento de credenciais do Azure ACR
- Usar GitHub Secrets para armazenamento criptografado
- Rotacionar senhas do ACR regularmente
- Limitar permissões da conta do ACR para apenas push
- Monitorar logs de acesso do ACR

### Risco 2: Falhas de sincronização causando inconsistência de imagens
- image-syncer inclui mecanismo integrado de sincronização incremental
- Retry automático em caso de falha (3 vezes por padrão)
- Logs de erro detalhados e notificações de falha
- Suporte a retomada

### Risco 3: Consumo excessivo de recursos
- Sincronização incremental reduz a transferência de rede
- Concorrência configurável (`10` na configuração atual)
- Monitorar o número e tamanho das imagens sincronizadas
- Executar a sincronização fora dos horários de pico

## Solução Central

Usamos uma solução automatizada com GitHub Actions + image-syncer para sincronizar imagens do Docker Hub para o Azure ACR.

## Etapas de Implementação

### 1. Preparação
- Criar ou confirmar um Azure Container Registry no Azure Portal
- Criar credenciais de acesso ao ACR (nome de usuário e senha)
- Confirmar permissões de acesso ao repositório de imagens do Docker Hub

### 2. Configurar GitHub Secrets
Adicione os seguintes secrets nas configurações do repositório GitHub:
- AZURE_ACR_USERNAME: nome de usuário do Azure ACR
- AZURE_ACR_PASSWORD: senha do Azure ACR

### 3. Criar o workflow do GitHub Actions
Configure o workflow em `.github/workflows/sync-docker-acr.yml`:
- Gatilho agendado: todo dia às 00:00 UTC
- Gatilho manual: suporta `workflow_dispatch`
- Gatilho extra: executar quando o branch `publish` receber um push (para sincronização rápida)

### 4. Fluxo de execução do workflow

| Sequência | Participante | Ação | Descrição |
| --- | --- | --- | --- |
| 1 | GitHub Actions | Disparar workflow | Disparado por agendamento, execução manual ou push no branch `publish` |
| 2 | GitHub Actions → image-syncer | Baixar e executar a ferramenta de sincronização | Entrar na fase real de sincronização |
| 3 | image-syncer → Docker Hub | Buscar manifests e lista de tags | Ler metadados do repositório de origem |
| 4 | image-syncer → Azure ACR | Buscar informações de imagens existentes no repositório de destino | Determinar o estado atual do lado de destino |
| 5 | image-syncer | Comparar diferenças entre origem e destino | Identificar camadas de imagem que precisam ser sincronizadas |
| 6 | image-syncer → Docker Hub | Fazer o pull das camadas de imagem alteradas | Transferir apenas o conteúdo que precisa ser atualizado |
| 7 | image-syncer → Azure ACR | Fazer o push das camadas de imagem alteradas | Completar a sincronização incremental |
| 8 | image-syncer → GitHub Actions | Retornar estatísticas de sincronização | Inclui resultados, diferenças e informações de erro |
| 9 | GitHub Actions | Registrar logs e fazer upload de artefatos | Útil para auditoria e resolução de problemas posteriores |

## Implementação do Workflow do GitHub Actions

Aqui está a configuração real do workflow em uso (`.github/workflows/sync-docker-acr.yml`):

```yaml
name: Sync Docker Image to Azure ACR

on:
  schedule:
    - cron: "0 0 * * *" # Todo dia às 00:00 UTC
  workflow_dispatch: # Gatilho manual
  push:
    branches: [publish]

permissions:
  contents: read

jobs:
  sync:
    runs-on: ubuntu-latest

    steps:
      - name: Checkout code
        uses: actions/checkout@v4

      - name: Download image-syncer
        run: |
          # Baixar o binário do image-syncer
          wget https://github.com/AliyunContainerService/image-syncer/releases/download/v1.5.5/image-syncer-v1.5.5-linux-amd64.tar.gz
          tar -zxvf image-syncer-v1.5.5-linux-amd64.tar.gz
          chmod +x image-syncer

      - name: Create auth config
        run: |
          # Gerar o arquivo de configuração de autenticação (formato YAML)
          cat > auth.yaml <<EOF
          hagicode.azurecr.io:
            username: "${{ secrets.AZURE_ACR_USERNAME }}"
            password: "${{ secrets.AZURE_ACR_PASSWORD }}"
          EOF

      - name: Create images config
        run: |
          # Gerar o arquivo de configuração de sincronização de imagens (formato YAML)
          cat > images.yaml <<EOF
          docker.io/newbe36524/hagicode: hagicode.azurecr.io/hagicode
          EOF

      - name: Run image-syncer
        run: |
          # Executar a sincronização (usando os parâmetros mais recentes --auth e --images)
          ./image-syncer --auth=./auth.yaml --images=./images.yaml --proc=10 --retries=3

      - name: Upload logs
        if: always()
        uses: actions/upload-artifact@v4
        with:
          name: sync-logs
          path: image-syncer-*.log
          retention-days: 7
```

## Detalhes da Configuração

### 1. Condições de gatilho
- Gatilho agendado: `cron: "0 0 * * *"` - executar todo dia às 00:00 UTC
- Gatilho manual: `workflow_dispatch` - permite que os usuários executem manualmente na interface do GitHub
- Gatilho por push: `push: branches: [publish]` - disparado quando o branch de publicação recebe um push (para sincronização rápida)

### 2. Configuração de autenticação (`auth.yaml`)
```yaml
hagicode.azurecr.io:
  username: "${{ secrets.AZURE_ACR_USERNAME }}"
  password: "${{ secrets.AZURE_ACR_PASSWORD }}"
```

### 3. Configuração de sincronização de imagens
```yaml
docker.io/newbe36524/hagicode: hagicode.azurecr.io/hagicode
```
Esta configuração significa sincronizar todas as tags de `docker.io/newbe36524/hagicode` para `hagicode.azurecr.io/hagicode`

### 4. Parâmetros do image-syncer
- `--auth=./auth.yaml`: caminho para o arquivo de configuração de autenticação
- `--images=./images.yaml`: caminho para o arquivo de configuração de sincronização de imagens
- `--proc=10`: definir concorrência para `10`
- `--retries=3`: tentar novamente em caso de falha `3` vezes

## Observações e Melhores Práticas

### 1. Recomendações de segurança
- Rotacionar senhas do Azure ACR regularmente (recomendado a cada 90 dias)
- Usar uma conta de serviço dedicada do ACR com permissões apenas de push
- Monitorar logs de acesso do ACR para detectar acessos anormais em tempo hábil
- Não exibir credenciais nos logs
- Não fazer commit de credenciais no repositório de código

### 2. Otimização de desempenho
- Ajustar o parâmetro `--proc`: ajustar a concorrência com base na largura de banda da rede (recomendado `5-20`)
- Monitorar o tempo de sincronização: se demorar muito, considere reduzir a concorrência
- Limpar logs regularmente: definir um valor razoável para `retention-days` (`7` dias na configuração atual)

### 3. Resolução de problemas

#### Problema 1: Falha de autenticação
```
Error: failed to authenticate to hagicode.azurecr.io
```
Solução:
1. Verificar se os GitHub Secrets estão configurados corretamente
2. Verificar se a senha do Azure ACR expirou
3. Confirmar se as permissões da conta de serviço do ACR estão corretas

#### Problema 2: Timeout de rede
```
Error: timeout waiting for response
```
Solução:
1. Verificar a conectividade de rede
2. Reduzir a concorrência (parâmetro `--proc`)
3. Aguardar a recuperação da rede e acionar o workflow novamente

#### Problema 3: Sincronização de imagem incompleta
```
Warning: some tags failed to sync
```
Solução:
1. Verificar os logs de sincronização para identificar tags com falha
2. Acionar manualmente o workflow para sincronizar novamente
3. Verificar se a imagem de origem no Docker Hub está funcionando corretamente

### 4. Monitoramento e alertas
- Verificar regularmente a página `Actions` para confirmar o status de execução dos workflows
- Configurar notificações do GitHub para receber alertas de falha de workflow prontamente
- Monitorar o uso de armazenamento do Azure ACR
- Verificar regularmente a consistência das tags

## Perguntas Frequentes e Soluções

### P1: Como sincronizar tags específicas em vez de todas as tags?

Modifique o arquivo de configuração `images.yaml`:
```yaml
# Sincronizar apenas as tags latest e v1.0
docker.io/newbe36524/hagicode:latest: hagicode.azurecr.io/hagicode:latest
docker.io/newbe36524/hagicode:v1.0: hagicode.azurecr.io/hagicode:v1.0
```

### P2: Como sincronizar múltiplos repositórios de imagens?

Adicione múltiplas linhas em `images.yaml`:
```yaml
docker.io/newbe36524/hagicode: hagicode.azurecr.io/hagicode
docker.io/newbe36524/another-image: hagicode.azurecr.io/another-image
```

### P3: Como tentar novamente após uma falha de sincronização?
- Retry automático: image-syncer inclui um mecanismo integrado de retry (`3` vezes por padrão)
- Retry manual: clique em `Re-run all jobs` na página do GitHub Actions

### P4: Como visualizar o progresso detalhado da sincronização?
- Visualizar logs em tempo real na página `Actions`
- Baixar o artefato `sync-logs` para ver o arquivo de log completo
- O arquivo de log inclui o status de sincronização e a velocidade de transferência para cada tag

### P5: Quanto tempo leva a sincronização?
- Sincronização completa inicial: tipicamente leva de `10-30` minutos dependendo do tamanho da imagem
- Sincronização incremental: geralmente de `2-5` minutos se as mudanças na imagem forem pequenas
- O tempo depende da largura de banda da rede, tamanho da imagem e configurações de concorrência

### 2. Implementar filtragem de tags de imagem
Adicione lógica de filtragem de tags ao workflow:
```yaml
- name: Filter tags
  run: |
    # Sincronizar apenas tags que começam com v
    echo "docker.io/newbe36524/hagicode:v* : hagicode.azurecr.io/hagicode:v*" > images.yaml
```

### 3. Adicionar um relatório estatístico de sincronização
```yaml
- name: Generate report
  if: always()
  run: |
    echo "## Sync Report" >> $GITHUB_STEP_SUMMARY
    echo "- Total tags: $(grep -c 'synced' image-syncer-*.log)" >> $GITHUB_STEP_SUMMARY
    echo "- Sync time: ${{ steps.sync.outputs.duration }}" >> $GITHUB_STEP_SUMMARY
```
