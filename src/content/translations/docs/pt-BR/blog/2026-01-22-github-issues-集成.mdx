## Construindo uma Integração com GitHub Issues do Zero: A Prática de Conexão Direta do Frontend no HagiCode

> Este artigo documenta o processo completo de integração do GitHub Issues na plataforma HagiCode. Vamos explorar como utilizar uma arquitetura de "conexão direta do frontend + backend mínimo" para obter autenticação OAuth segura e sincronização eficiente de issues, mantendo o backend enxuto.

## Contexto: Por que Integrar com o GitHub?

Como uma plataforma de desenvolvimento assistido por IA, o valor central do HagiCode está em conectar ideias à implementação. Mas na prática, percebemos que, após os usuários completarem uma Proposal no HagiCode, frequentemente precisam copiar manualmente o conteúdo para o GitHub Issues para o acompanhamento do projeto.

Isso cria vários pontos de dor evidentes:

1.  **Fluxo de trabalho fragmentado**: Os usuários precisam alternar entre dois sistemas. A experiência não é fluida, e informações importantes podem se perder facilmente durante o copiar e colar.
2.  **Colaboração inconveniente**: Outros membros da equipe estão acostumados a verificar tarefas no GitHub e não conseguem ver diretamente o progresso das propostas dentro do HagiCode.
3.  **Trabalho manual repetitivo**: Toda vez que uma proposta é atualizada, alguém precisa atualizar manualmente a issue correspondente no GitHub, adicionando custo desnecessário de manutenção.

Para resolver esse problema, decidimos introduzir o recurso de **Integração com GitHub Issues**, conectando as sessões do HagiCode com os repositórios do GitHub para habilitar a "sincronização com um clique".

## Escolha Técnica: Conexão Direta do Frontend vs Proxy do Backend

Ao projetar a abordagem de integração, tínhamos duas opções à nossa frente: o modelo tradicional de "proxy do backend" e o modelo mais ousado de "conexão direta do frontend".

### Comparação de Soluções

No modelo tradicional de **proxy do backend**, toda requisição do frontend precisa passar primeiro pelo nosso backend, que então chama a API do GitHub. Isso centraliza a lógica, mas também coloca uma carga significativa no backend:

1.  **Backend inchado**: Precisaríamos escrever um wrapper dedicado de cliente da API do GitHub e também lidar com a máquina de estados complexa do OAuth.
2.  **Risco de token**: O token do GitHub do usuário teria que ser armazenado no banco de dados do backend. Mesmo com criptografia, isso ainda aumenta a superfície de segurança.
3.  **Custo de desenvolvimento**: Precisaríamos de migrações de banco de dados para armazenar tokens e um serviço de sincronização adicional para manter.

O modelo de **conexão direta do frontend** é muito mais leve. Nessa abordagem, usamos o backend apenas para a etapa mais sensível de "troca de segredos" (o callback do OAuth). Após obter o token, o armazenamos diretamente no `localStorage` do navegador. Operações posteriores como criação de issues e atualização de comentários são enviadas diretamente do frontend para o GitHub via HTTP.

| Dimensão de Comparação | Modelo de Proxy do Backend | Modelo de Conexão Direta do Frontend |
| :--- | :--- | :--- |
| **Complexidade do backend** | Requer um serviço OAuth completo e cliente da API do GitHub | Precisa apenas de um endpoint de callback OAuth |
| **Gerenciamento de token** | Deve ser criptografado e armazenado no banco de dados, com risco de vazamento | Armazenado no navegador, visível apenas para o usuário |
| **Custo de implementação** | Requer migrações de banco de dados e desenvolvimento multi-serviço | Principalmente trabalho de frontend |
| **Experiência do usuário** | Lógica centralizada, mas a latência do servidor pode ser ligeiramente maior | Resposta extremamente rápida com interação direta com o GitHub |

Como queríamos uma integração rápida e mudanças mínimas no backend, **optamos pelo "modelo de conexão direta do frontend"**. É como dar ao navegador um "passe temporário". Uma vez com o passe em mãos, o navegador pode ir resolver coisas no GitHub por conta própria, sem precisar pedir permissão ao administrador do backend toda vez.

## Design Central: Fluxo de Dados e Segurança

Após definir a arquitetura, precisávamos projetar o fluxo de dados específico. O núcleo do processo de sincronização está em como obter o token com segurança e utilizá-lo de forma eficiente.

### Diagrama de Arquitetura Geral

Todo o sistema pode ser abstraído em três papéis: o navegador (frontend), o backend do HagiCode e o GitHub.

```text
+--------------+        +--------------+        +--------------+
| Frontend     |        |   Backend    |        |   GitHub     |
| React        |        |   ASP.NET    |        |   REST API   |
|              |        |              |        |              |
|  +--------+  |        |              |        |              |
|  | OAuth  |--+--------> /callback    |        |              |
|  | Fluxo  |  |        |              |        |              |
|  +--------+  |        |              |        |              |
|              |        |              |        |              |
|  +--------+  |        |  +--------+  |        |  +--------+  |
|  | GitHub |  +------------> Session |  +----------> Issues |  |
|  | API    |  |        |  |Metadata|  |        |  |        |  |
|  | Direta |  |        |  +--------+  |        |  +--------+  |
|  +--------+  |        |              |        |              |
+--------------+        +--------------+        +--------------+
```

**O ponto chave é**: apenas uma pequena etapa do OAuth (trocar o code pelo token) precisa passar pelo backend. Depois disso, o trabalho pesado (criar issues) é tratado diretamente entre o frontend e o GitHub.

### Fluxo de Dados de Sincronização em Detalhes

Quando o usuário clica no botão "Sync to GitHub" na interface do HagiCode, uma série de ações complexas ocorre:

```text
Usuário clica em "Sync to GitHub"
         │
         ▼
1. Frontend verifica o localStorage em busca do token do GitHub
         │
         ▼
2. Formata o conteúdo da issue (converte a Proposal em Markdown)
         │
         ▼
3. Frontend chama a API do GitHub diretamente para criar/atualizar a issue
         │
         ▼
4. Chama a API do backend do HagiCode para atualizar Session.metadata (armazena URL da issue e outras informações)
         │
         ▼
5. Backend transmite o evento SessionUpdated via SignalR
         │
         ▼
6. Frontend recebe o evento e atualiza a UI para exibir o estado "Sincronizado"
```

### Design de Segurança

A segurança é sempre a prioridade máxima ao integrar serviços de terceiros. Fizemos as seguintes considerações:

1.  **Defesa contra ataques CSRF**: Geramos um parâmetro `state` aleatório durante o redirecionamento OAuth e o armazenamos no `sessionStorage`. Validamos rigorosamente o state no callback para evitar requisições forjadas.
2.  **Armazenamento isolado de token**: O token é armazenado apenas no `localStorage` do navegador. Utilizando a política de mesma origem (Same-Origin Policy), apenas scripts do HagiCode podem lê-lo, evitando o risco de vazamento de banco de dados do servidor afetar os usuários.
3.  **Limites de erro**: Projetamos um tratamento dedicado para erros comuns da API do GitHub (como token expirado 401, falha de validação 422 e limitação de taxa 429), para que os usuários recebam feedback amigável.

### 2. Fluxo OAuth do Frontend

Este é o ponto de entrada da conexão. Utilizamos o fluxo padrão de Authorization Code.

```typescript
// src/HagiCode.Client/src/services/githubOAuth.ts

// Gerar a URL de autorização e redirecionar
export async function generateAuthUrl(): Promise<string> {
  const state = generateRandomString(); // Gerar uma string aleatória para proteção contra CSRF
  sessionStorage.setItem('hagicode_github_state', state);

  const params = new URLSearchParams({
    client_id: clientId,
    redirect_uri: window.location.origin + '/settings?tab=github&oauth=callback',
    scope: ['repo', 'public_repo'].join(' '),
    state: state,
  });

  return `https://github.com/login/oauth/authorize?${params.toString()}`;
}

// Tratar a troca de code por token na página de callback
export async function exchangeCodeForToken(code: string, state: string): Promise<GitHubToken> {
  // 1. Validar o state para prevenir CSRF
  const savedState = sessionStorage.getItem('hagicode_github_state');
  if (state !== savedState) throw new Error('Invalid state parameter');

  // 2. Chamar a API do backend para trocar o token
  // Nota: isso precisa passar pelo backend porque o ClientSecret não pode ser exposto ao frontend
  const response = await fetch('/api/GitHubOAuth/callback', {
    method: 'POST',
    headers: { 'Content-Type': 'application/json' },
    body: JSON.stringify({ code, state, redirectUri: window.location.origin + '/settings?tab=github&oauth=callback' }),
  });

  if (!response.ok) throw new Error('Failed to exchange token');

  const token = await response.json();

  // 3. Salvar no LocalStorage
  saveToken(token);
  return token;
}
```

### 3. Wrapper do Cliente da API do GitHub

Com o token em mãos, precisamos de uma ferramenta sólida para chamar a API do GitHub.

```typescript
// src/HagiCode.Client/src/services/githubApiClient.ts

const GITHUB_API_BASE = 'https://api.github.com';

// Wrapper central de requisições
async function githubApi<T>(endpoint: string, options: RequestInit = {}): Promise<T> {
  const token = localStorage.getItem('gh_token');
  if (!token) throw new Error('Not connected to GitHub');

  const response = await fetch(`${GITHUB_API_BASE}${endpoint}`, {
    ...options,
    headers: {
      ...options.headers,
      Authorization: `Bearer ${token}`,
      Accept: 'application/vnd.github.v3+json', // Especificar a versão da API
    },
  });

  // Lógica de tratamento de erros
  if (!response.ok) {
    if (response.status === 401) throw new Error('GitHub token expirado, por favor reconecte');
    if (response.status === 403) throw new Error('Sem permissão para acessar este repositório ou limite de taxa excedido');
    if (response.status === 422) throw new Error('Validação da issue falhou, o título pode estar duplicado');
    throw new Error(`GitHub API Error: ${response.statusText}`);
  }

  return response.json();
}

// Criar issue
export async function createIssue(owner: string, repo: string, data: { title: string, body: string, labels: string[] }) {
  return githubApi(`/repos/${owner}/${repo}/issues`, {
    method: 'POST',
    body: JSON.stringify(data),
  });
}
```

### 4. Formatação de Conteúdo e Sincronização

O passo final é converter os dados da sessão do HagiCode no formato de uma issue do GitHub. É um trabalho um pouco semelhante a tradução.

```typescript
// Converter um objeto Session em uma string Markdown
function formatIssueForSession(session: Session): string {
  let content = `# ${session.title}\n\n`;
  content += `**> HagiCode Session:** #${session.code}\n`;
  content += `**> Status:** ${session.status}\n\n`;
  content += `## Description\n\n${session.description || 'No description provided.'}\n\n`;

  // Se for uma sessão do tipo Proposal, adicionar campos extras
  if (session.type === 'proposal') {
    content += `## Chief Complaint\n\n${session.chiefComplaint || ''}\n\n`;
    // Adicionar um deep link para que os usuários possam voltar do GitHub para o HagiCode
    content += `---\n\n**[Ver no HagiCode](hagicode://sessions/${session.id})**\n`;
  }

  return content;
}

// Lógica principal ao clicar no botão de sincronização
const handleSync = async (session: Session) => {
  try {
    const repoInfo = parseRepositoryFromUrl(session.repoUrl); // Analisar a URL do repositório
    if (!repoInfo) throw new Error('Invalid repository URL');

    toast.loading('Sincronizando com o GitHub...');

    // 1. Formatar o conteúdo
    const issueBody = formatIssueForSession(session);

    // 2. Chamar a API
    const issue = await githubApiClient.createIssue(repoInfo.owner, repoInfo.repo, {
      title: `[HagiCode] ${session.title}`,
      body: issueBody,
      labels: ['hagicode', 'proposal', `status:${session.status}`],
    });

    // 3. Atualizar o Metadata da Session (salvar o link da issue)
    await SessionsService.patchApiSessionsSessionId(session.id, {
      metadata: {
        githubIssue: {
          owner: repoInfo.owner,
          repo: repoInfo.repo,
          issueNumber: issue.number,
          issueUrl: issue.html_url,
          syncedAt: new Date().toISOString(),
        }
      }
    });

    toast.success('Sincronizado com sucesso!');
  } catch (err) {
    console.error(err);
    toast.error('Falha na sincronização, por favor verifique o token ou a rede');
  }
};
```

## Resumo e Perspectivas

Com essa abordagem de "conexão direta do frontend", alcançamos uma integração perfeita do GitHub Issues com a menor quantidade possível de código no backend.

### Principais ganhos
1.  **Alta eficiência de desenvolvimento**: As mudanças no backend são mínimas, principalmente um campo extra no banco de dados e um endpoint simples de callback OAuth. A maior parte da lógica é completada no frontend.
2.  **Segurança robusta**: O token não passa pelo banco de dados do servidor, reduzindo o risco de vazamento.
3.  **Ótima experiência do usuário**: As requisições são iniciadas diretamente do frontend, então a velocidade de resposta é rápida e não há necessidade de encaminhamento pelo backend.

### Pontos de atenção
Existem algumas armadilhas para manter em mente durante a implantação real:
- **Configuração do OAuth App**: Lembre-se de inserir a `Authorization callback URL` correta nas configurações do seu GitHub OAuth App (geralmente `http://localhost:3000/settings?tab=github&oauth=callback`).
- **Limites de taxa**: A API do GitHub limita bastante as requisições não autenticadas, mas com um token a cota geralmente é suficiente (5000 requisições/hora).
- **Análise de URL**: Os usuários inserem todos os tipos de URLs de repositório, então certifique-se de que sua regex consiga corresponder sufixos `.git`, formatos SSH e casos similares.

### Melhorias futuras
O recurso atual ainda é de sincronização unidirecional (HagiCode -> GitHub). No futuro, planejamos implementar a sincronização bidirecional por meio de GitHub Webhooks. Por exemplo, se uma issue for fechada no GitHub, o estado da sessão no lado do HagiCode também poderia ser atualizado automaticamente. Isso exigirá que exponhamos um endpoint de webhook no backend, o que será um próximo passo interessante.
