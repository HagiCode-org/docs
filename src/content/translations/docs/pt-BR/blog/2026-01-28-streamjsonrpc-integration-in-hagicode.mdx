## Integração Profunda e Uso Prático do StreamJsonRpc no HagiCode

> Este artigo apresenta em detalhes como o projeto HagiCode (anteriormente PCode) integrou com sucesso a biblioteca de comunicação StreamJsonRpc da Microsoft, substituindo a implementação JSON-RPC personalizada anterior e resolvendo os desafios técnicos e arquiteturais do processo de integração.

## Contexto

StreamJsonRpc é uma biblioteca de comunicação JSON-RPC para .NET e TypeScript mantida oficialmente pela Microsoft, conhecida por sua forte segurança de tipos, geração automática de proxies e mecanismo maduro de tratamento de exceções. No projeto HagiCode, para se comunicar com ferramentas de IA externas (como iflow CLI, OpenCode CLI) através do ACP (Agent Communication Protocol) e eliminar os custos de manutenção e bugs potenciais da implementação JSON-RPC personalizada anterior, o projeto decidiu integrar o StreamJsonRpc. No entanto, durante o processo de integração, foram encontrados desafios específicos do JSON-RPC em streaming, especialmente no tratamento da vinculação de targets de proxy e na identificação de parâmetros genéricos.

Para resolver esses problemas, tomamos uma decisão ousada: reconstruir todo o sistema de build do zero. As mudanças que essa decisão trouxe podem ser maiores do que você imagina — detalharemos isso mais adiante.

## Análise

O projeto está atualmente em uma fase crítica de integração do protocolo ACP, enfrentando os seguintes desafios técnicos e arquiteturais:

### 1. Limitações da Implementação Personalizada
A implementação JSON-RPC anterior estava localizada em `src/HagiCode.ClaudeHelper/AcpImp/`, contendo componentes como `JsonRpcEndpoint` e `ClientSideConnection`. Manter esse código personalizado tinha um custo alto e carecia de funcionalidades avançadas de bibliotecas maduras (como relatório de progresso, suporte a cancelamento).

### 2. Obstáculos na Integração do StreamJsonRpc
Ao tentar migrar o padrão `CallbackProxyTarget` existente para o StreamJsonRpc, descobrimos que o método `_rpc.AddLocalRpcTarget(target)` não conseguia reconhecer targets criados por meio de padrão de proxy. Especificamente, o StreamJsonRpc não conseguia automaticamente decompor as propriedades do tipo genérico `T` em parâmetros de método RPC, fazendo com que o servidor não pudesse processar corretamente as chamadas de método iniciadas pelo cliente.

### 3. Camadas Arquiteturais Confusas
O `ClientSideConnection` existente misturava a camada de transporte (WebSocket/Stdio), a camada de protocolo (JSON-RPC) e a camada de negócios (interface do ACP Agent), resultando em responsabilidades indefinidas e problemas de vinculação de métodos ausentes no `AcpAgentCallbackRpcAdapter`.

### 4. Ausência de Logs
A camada de transporte WebSocket carecia de saída de logs do conteúdo JSON bruto, dificultando a identificação em problemas de depuração de comunicação RPC se o problema era de serialização ou de rede.

## Solução

Para os problemas acima, adotamos a seguinte solução sistemática, otimizando em três dimensões: refatoração arquitetural, integração de biblioteca e aprimoramento de depuração:

### 1. Migração Completa para o StreamJsonRpc

#### Remoção do Código Antigo
Exclusão de `JsonRpcEndpoint.cs`, `AgentSideConnection.cs` e conversores de serialização personalizados relacionados (`JsonRpcMessageJsonConverter`, etc.).

#### Integração da Biblioteca Oficial
Introdução do pacote NuGet `StreamJsonRpc`, utilizando sua classe `JsonRpc` para processar a lógica central de comunicação.

#### Abstração da Camada de Transporte
Definição da interface `IAcpTransport` para tratar uniformemente os dois modos de transporte `WebSocket` e `Stdio`, garantindo o desacoplamento entre a camada de protocolo e a camada de transporte.

```csharp
// Definição da interface IAcpTransport
public interface IAcpTransport
{
    Task SendAsync(string message, CancellationToken cancellationToken = default);
    Task<string> ReceiveAsync(CancellationToken cancellationToken = default);
    Task CloseAsync(CancellationToken cancellationToken = default);
}

// Implementação de transporte WebSocket
public class WebSocketTransport : IAcpTransport
{
    private readonly WebSocket _webSocket;

    public WebSocketTransport(WebSocket webSocket)
    {
        _webSocket = webSocket;
    }

    // Implementar métodos de envio e recebimento
    // ...
}

// Implementação de transporte Stdio
public class StdioTransport : IAcpTransport
{
    private readonly StreamReader _reader;
    private readonly StreamWriter _writer;

    public StdioTransport(StreamReader reader, StreamWriter writer)
    {
        _reader = reader;
        _writer = writer;
    }

    // Implementar métodos de envio e recebimento
    // ...
}
```

### 2. Correção do Problema de Identificação de Target de Proxy

#### Análise do `CallbackProxyTarget`
Verificação da lógica de geração de proxy dinâmico existente para determinar a causa raiz de o StreamJsonRpc não conseguir reconhecer (geralmente porque o objeto proxy não expõe as assinaturas reais dos métodos, ou usa tipos de parâmetros não suportados pelo StreamJsonRpc).

#### Refatoração da Passagem de Parâmetros
Decomposição das propriedades genéricas em parâmetros de método RPC explícitos. Em vez de depender de propriedades dinâmicas, foram definidos DTOs (Data Transfer Objects) de Request/Response específicos, garantindo que o StreamJsonRpc possa identificar corretamente as assinaturas dos métodos por meio de reflexão.

```csharp
// Forma anterior com propriedades genéricas
public class CallbackProxyTarget<T>
{
    public Func<T, Task> Callback { get; set; }
}

// Forma refatorada com métodos específicos
public class ReadTextFileRequest
{
    public string FilePath { get; set; }
}

public class ReadTextFileResponse
{
    public string Content { get; set; }
}

public interface IAcpAgentCallback
{
    Task<ReadTextFileResponse> ReadTextFileAsync(ReadTextFileRequest request);
    // Outros métodos...
}
```

#### Usar `Attach` em vez de `AddLocalRpcTarget`
Em alguns cenários complexos, gerenciar manualmente o objeto `JsonRpc` e tratar a `RpcConnection` pode ser mais flexível do que adicionar diretamente um target.

### 3. Implementação de Vinculação de Métodos e Aprimoramento de Logs

#### Implementação do `AcpAgentCallbackRpcAdapter`
Garantir que este componente implemente explicitamente a interface de proxy do StreamJsonRpc, mapeando os métodos definidos pelo protocolo ACP (como `ReadTextFileAsync`) para os manipuladores de callback do StreamJsonRpc.

#### Integração de Registro de Logs
No pipeline de processamento de mensagens do WebSocket ou Stdio, interceptar e registrar o texto bruto das requisições e respostas JSON-RPC. Utilizar o `ILogger` para gerar a saída do payload bruto antes da análise e após a serialização, facilitando a identificação de erros de formato.

```csharp
// Wrapper de transporte com registro de logs
public class LoggingAcpTransport : IAcpTransport
{
    private readonly IAcpTransport _innerTransport;
    private readonly ILogger<LoggingAcpTransport> _logger;

    public LoggingAcpTransport(IAcpTransport innerTransport, ILogger<LoggingAcpTransport> logger)
    {
        _innerTransport = innerTransport;
        _logger = logger;
    }

    public async Task SendAsync(string message, CancellationToken cancellationToken = default)
    {
        _logger.LogTrace("Sending message: {Message}", message);
        await _innerTransport.SendAsync(message, cancellationToken);
    }

    public async Task<string> ReceiveAsync(CancellationToken cancellationToken = default)
    {
        var message = await _innerTransport.ReceiveAsync(cancellationToken);
        _logger.LogTrace("Received message: {Message}", message);
        return message;
    }

    public async Task CloseAsync(CancellationToken cancellationToken = default)
    {
        _logger.LogDebug("Closing connection");
        await _innerTransport.CloseAsync(cancellationToken);
    }
}
```

### 4. Refatoração da Camada Arquitetural

#### Camada de Transporte (`AcpRpcClient`)
Encapsulamento da conexão StreamJsonRpc, responsável por `InvokeAsync` e gerenciamento do ciclo de vida da conexão.

```csharp
public class AcpRpcClient : IDisposable
{
    private readonly JsonRpc _rpc;
    private readonly IAcpTransport _transport;

    public AcpRpcClient(IAcpTransport transport)
    {
        _transport = transport;
        _rpc = new JsonRpc(new StreamRpcTransport(transport));
        _rpc.StartListening();
    }

    public async Task<TResponse> InvokeAsync<TResponse>(string methodName, object parameters)
    {
        return await _rpc.InvokeAsync<TResponse>(methodName, parameters);
    }

    public void Dispose()
    {
        _rpc.Dispose();
        _transport.Dispose();
    }

    // StreamRpcTransport é um adaptador do StreamJsonRpc para IAcpTransport
    private class StreamRpcTransport : IDuplexPipe
    {
        // Implementação da interface IDuplexPipe
        // ...
    }
}
```

#### Camada de Protocolo (`IAcpAgentClient` / `IAcpAgentCallback`)
Definição de interfaces claras de client-to-agent e agent-to-client, removendo o padrão de fábrica com dependência circular `Func<IAcpAgent, IAcpClient>` e adotando injeção de dependência ou registro direto de callbacks.

## Prática

Com base nas melhores práticas do StreamJsonRpc e na experiência do projeto, seguem as recomendações-chave para o processo de implementação:

### 1. DTOs Fortemente Tipados são Melhores que Objetos Dinâmicos
A vantagem central do StreamJsonRpc está na tipagem forte. Não use `dynamic` ou `JObject` para passar parâmetros. Defina classes C# POCO específicas para cada método RPC. Isso não apenas resolve o problema de identificação de targets de proxy, mas também permite descobrir erros de tipo em tempo de compilação.

Exemplo: substituir as propriedades genéricas em `CallbackProxyTarget` por classes específicas como `ReadTextFileRequest` e `WriteTextFileRequest`.

### 2. Declarar Explicitamente o Nome do Método
Use o atributo `[JsonRpcMethod]` para especificar explicitamente o nome do método RPC, sem depender do mapeamento padrão de nomes de métodos. Isso evita falhas de chamada causadas por diferenças de estilo de nomenclatura (como PascalCase vs camelCase).

```csharp
public interface IAcpAgentCallback
{
    [JsonRpcMethod("readTextFile")]
    Task<ReadTextFileResponse> ReadTextFileAsync(ReadTextFileRequest request);

    [JsonRpcMethod("writeTextFile")]
    Task WriteTextFileAsync(WriteTextFileRequest request);
}
```

### 3. Aproveitar os Callbacks de Estado da Conexão
O StreamJsonRpc fornece o evento `JsonRpc.ConnectionLost`. Monitore este evento obrigatoriamente para tratar casos de saída inesperada de processo ou desconexão de rede, o que é mais tempestivo do que depender apenas da detecção de falha de Grain do Orleans.

```csharp
_rpc.ConnectionLost += (sender, e) =>
{
    _logger.LogError("RPC connection lost: {Reason}", e.ToString());
    // Tratar lógica de reconexão ou notificar o usuário
};
```

### 4. Registro de Logs em Camadas
- **Nível Trace**: registrar o texto completo das requisições/respostas JSON.
- **Nível Debug**: registrar a pilha de chamadas de métodos e resumo dos parâmetros.
- **Atenção**: certifique-se de que os logs não contenham Authorization Tokens sensíveis ou codificações Base64 de conteúdo de arquivos grandes.

### 5. Tratamento das Peculiaridades da Transmissão em Streaming
O StreamJsonRpc suporta nativamente `IAsyncEnumerable`. Ao implementar respostas de Prompt em streaming do ACP, use diretamente `IAsyncEnumerable` em vez de lógica de paginação personalizada. Isso simplifica enormemente a quantidade de código de processamento em streaming.

```csharp
public interface IAcpAgentCallback
{
    [JsonRpcMethod("streamText")]
    IAsyncEnumerable<string> StreamTextAsync(StreamTextRequest request);
}
```

### 6. Padrão Adapter (Padrão Adaptador)
Mantenha a separação entre `ACPSession` e `ClientSideConnection`. O `ACPSession` deve se concentrar no gerenciamento de estado do Orleans e na lógica de negócios (como enfileiramento de mensagens), utilizando o objeto de conexão StreamJsonRpc por meio de composição em vez de herança.

## Conclusão

Com a integração completa do StreamJsonRpc, o projeto HagiCode resolveu com sucesso os problemas de alto custo de manutenção, limitações funcionais e camadas arquiteturais confusas da implementação personalizada anterior. As melhorias principais incluem:

1. Adoção de DTOs fortemente tipados em vez de propriedades dinâmicas, melhorando a manutenibilidade e confiabilidade do código
2. Implementação da abstração da camada de transporte e separação da camada de protocolo, melhorando a clareza da arquitetura
3. Aprimoramento da funcionalidade de registro de logs, facilitando a identificação de problemas de comunicação
4. Introdução de suporte a transmissão em streaming, simplificando a implementação do processamento em streaming

Essas melhorias forneceram ao HagiCode uma base de comunicação mais estável e eficiente, permitindo melhor interação com ferramentas de IA externas e estabelecendo uma base sólida para expansão futura de funcionalidades.
