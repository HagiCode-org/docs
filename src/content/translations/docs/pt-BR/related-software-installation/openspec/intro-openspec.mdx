# Introdução ao OpenSpec

OpenSpec é uma ferramenta de desenvolvimento orientada por especificações que ajuda equipes a gerenciar o processo de desenvolvimento de software através de propostas e especificações estruturadas. Este guia fornece uma introdução abrangente aos conceitos principais, fluxos de trabalho e métodos de uso do OpenSpec.

## O que é OpenSpec?

OpenSpec é uma ferramenta de linha de comando para gerenciar Proposals, Specifications e Changes ao longo do processo de desenvolvimento de software. Ele padroniza o fluxo de trabalho de desenvolvimento, garantindo que cada recurso passe por design e revisão completos, da ideia à implementação.

### Filosofia Central

OpenSpec é baseado nos seguintes princípios fundamentais:

- **Especificação Primeiro**: Definir especificações claras antes de escrever código
- **Colaboração Estruturada**: Promover a colaboração da equipe através de formatos de documentos padronizados
- **Rastreabilidade**: Registrar o processo completo da proposta à implementação
- **Garantia de Qualidade**: Garantir que o código esteja em conformidade com a intenção do design através da validação de especificações

### Recursos Principais

OpenSpec fornece os seguintes recursos principais:

1. **Gerenciamento de Propostas**: Criar, revisar e rastrear propostas de recursos
2. **Definição de Especificações**: Escrever e manter documentos de especificação técnica
3. **Rastreamento de Mudanças**: Registrar todo o histórico de mudanças revisadas
4. **Validação de Especificações**: Garantir que as mudanças de código estejam em conformidade com as especificações predefinidas
5. **Colaboração em Equipe**: Suportar processos de desenvolvimento e revisão colaborativos

## Fluxo de Trabalho do OpenSpec

O fluxo de trabalho típico do OpenSpec inclui as seguintes etapas:

### 1. Criar uma Proposta

Quando há um novo recurso ou mudança importante, primeiro crie uma proposta:

```bash
openspec proposal create
```

Uma proposta contém:
- **Descrição do Problema**: O problema ou requisito a ser resolvido
- **Solução**: Visão geral da solução proposta
- **Escopo de Impacto**: Impacto nos sistemas existentes
- **Alternativas**: Outras soluções consideradas

### 2. Escrever uma Especificação

Após a aprovação da proposta, escreva uma especificação técnica detalhada:

```bash
openspec spec create
```

Uma especificação contém:
- **Design Técnico**: Plano de implementação técnica detalhado
- **Definição de Interface**: API, estruturas de dados e outras definições de interface
- **Detalhes de Implementação**: Lógica de implementação chave e algoritmos
- **Plano de Testes**: Como verificar a correção da implementação

### 3. Criar uma Mudança

Ao iniciar a implementação, crie um change para rastrear o processo de desenvolvimento:

```bash
openspec change create
```

Um change contém:
- **Especificação Associada**: A especificação que este change implementa
- **Tarefas de Implementação**: Lista específica de tarefas de desenvolvimento
- **Rastreamento de Progresso**: Status de conclusão das tarefas
- **Mudanças de Código**: Modificações reais do código

### 4. Validar e Arquivar

Após concluir o desenvolvimento, valide se o change está em conformidade com a especificação e, em seguida, arquive:

```bash
openspec change validate
openspec change archive
```

## Estrutura de Diretórios do OpenSpec

OpenSpec cria um diretório `openspec/` na raiz do projeto com a seguinte estrutura:

```
openspec/
├── proposals/           # Diretório de propostas
│   ├── PROPOSAL-001.md
│   └── PROPOSAL-002.md
├── specs/              # Diretório de especificações
│   ├── SPEC-001.md
│   └── SPEC-002.md
├── changes/            # Diretório de mudanças
│   ├── feature-name/
│   │   ├── proposal.md
│   │   ├── design.md
│   │   ├── specs.md
│   │   └── tasks.md
│   └── another-feature/
└── archived/           # Mudanças arquivadas
    └── feature-name/
```

### Descrição dos Diretórios

- **proposals/**: Armazena todos os documentos de proposta, um arquivo Markdown independente por proposta
- **specs/**: Armazena documentos de especificação técnica correspondentes às propostas
- **changes/**: Armazena mudanças em andamento, um subdiretório por change
- **archived/**: Armazena mudanças concluídas e arquivadas

## Comandos Principais

### Comandos de Proposta

```bash
# Criar uma nova proposta
openspec proposal create

# Listar todas as propostas
openspec proposal list

# Ver detalhes de uma proposta
openspec proposal view <proposal-id>

# Aprovar uma proposta
openspec proposal approve <proposal-id>
```

### Comandos de Especificação

```bash
# Criar uma nova especificação
openspec spec create

# Listar todas as especificações
openspec spec list

# Ver detalhes de uma especificação
openspec spec view <spec-id>

# Atualizar uma especificação
openspec spec update <spec-id>
```

### Comandos de Change

```bash
# Criar um novo change
openspec change create

# Listar todos os changes
openspec change list

# Ver detalhes de um change
openspec change view <change-name>

# Adicionar uma tarefa
openspec change add-task <change-name>

# Validar um change
openspec change validate <change-name>

# Arquivar um change
openspec change archive <change-name>
```

### Comandos Gerais

```bash
# Ver versão
openspec --version

# Ver ajuda
openspec --help

# Ver ajuda de um comando específico
openspec <command> --help
```

## Usando OpenSpec em Projetos HagiCode

### Integração do Projeto

Em projetos HagiCode, OpenSpec está integrado ao fluxo de trabalho de desenvolvimento:

1. **Verificação de Inicialização**: O assistente de inicialização verifica se o OpenSpec está instalado
2. **Desenvolvimento Orientado por Proposta**: Todos os novos recursos devem passar pelo processo de propostas do OpenSpec
3. **Revisão de Código**: As mudanças devem estar em conformidade com as especificações do OpenSpec para serem mescladas
4. **Sincronização de Documentação**: A documentação do OpenSpec permanece sincronizada com a implementação do código

### Requisitos de Versão

Projetos HagiCode requerem a versão **OpenSpec >=1.0.0 <2.0.0**.

Comando de instalação:

```bash
npm install -g @fission-ai/openspec@1
```

### Exemplo de Fluxo de Trabalho

Aqui está um exemplo completo de uso do OpenSpec em um projeto HagiCode:

#### Passo 1: Criar uma Proposta

```bash
cd /path/to/hagicode-project
openspec proposal create
```

Digite no prompt interativo:
- Título da proposta: Adicionar recurso de autenticação de usuário
- Descrição do problema: O sistema precisa de login e gerenciamento de permissões
- Solução: Implementar um sistema de autenticação baseado em JWT

#### Passo 2: Escrever uma Especificação

Após a aprovação da proposta, crie uma especificação detalhada:

```bash
openspec spec create
```

Defina na especificação:
- Design do fluxo de autenticação
- Definição da interface API
- Estrutura das tabelas do banco de dados
- Considerações de segurança

#### Passo 3: Criar um Change

Inicie a implementação:

```bash
openspec change create
```

Digite o nome do change: `add-user-authentication`

#### Passo 4: Adicionar Tarefas

```bash
openspec change add-task add-user-authentication
```

Adicione tarefas específicas:
- Implementar endpoint de registro de usuário
- Implementar endpoint de login
- Implementar middleware JWT
- Escrever testes unitários

#### Passo 5: Implementar e Validar

Após concluir o código, valide o change:

```bash
openspec change validate add-user-authentication
```

#### Passo 6: Arquivar

Após a validação, arquive o change:

```bash
openspec change archive add-user-authentication
```

## Melhores Práticas

### Escrevendo Propostas

1. **Declaração Clara do Problema**: Defina claramente o problema a ser resolvido e evite descrições vagas
2. **Considerar Alternativas**: Liste e compare diferentes soluções
3. **Avaliar Impacto**: Analise o impacto e os riscos nos sistemas existentes
4. **Manter Conciso**: As propostas devem ser breves; os detalhes pertencem à especificação

### Escrevendo Especificações

1. **Detalhado mas Não Redundante**: Forneça detalhes suficientes, evite over-engineering
2. **Usar Diagramas**: Use fluxogramas, diagramas de arquitetura, etc. para auxiliar na explicação
3. **Definir Interfaces**: Defina claramente todas as interfaces públicas e estruturas de dados
4. **Considerar Casos Extremos**: Liste situações excepcionais e tratamento de erros

### Gerenciamento de Mudanças

1. **Pequenas Iterações**: Divida grandes recursos em várias pequenas mudanças
2. **Atualizações Oportunas**: Mantenha o status das tarefas e a documentação sincronizados
3. **Revisão de Código**: Realize revisões de código após a conclusão das mudanças
4. **Arquivamento Oportuno**: Arquive mudanças concluídas para manter o diretório de trabalho limpo

### Colaboração em Equipe

1. **Revisões Regulares**: Revise regularmente propostas e especificações para garantir o consenso da equipe
2. **Documentação Primeiro**: Complete a documentação antes de começar a codificar
3. **Manter Comunicação**: Registre discussões e decisões nas propostas e especificações
4. **Controle de Versão**: Inclua documentos OpenSpec no controle de versão

## Casos de Uso Comuns

### Cenário 1: Adicionar Novos Recursos

1. Crie uma proposta descrevendo os requisitos do novo recurso
2. A equipe discute e aprova a proposta
3. Escreva uma especificação técnica detalhada
4. Crie um change e implemente o recurso
5. Valide e arquive o change

### Cenário 2: Refatoração de Código Existente

1. Crie uma proposta explicando as razões e objetivos da refatoração
2. Defina o plano de refatoração e as etapas na especificação
3. Crie um change e refatore gradualmente
4. Garanta a cobertura de testes e valide os resultados da refatoração
5. Arquive o change

### Cenário 3: Correção de Bugs

Para bugs simples:
- Podem ser corrigidos diretamente sem o processo completo do OpenSpec

Para bugs complexos:
1. Crie uma proposta analisando a causa raiz do bug
2. Defina a correção na especificação
3. Crie um change e implemente a correção
4. Adicione testes para prevenir regressões
5. Arquive o change

### Cenário 4: Gerenciamento de Dívida Técnica

1. Crie uma proposta listando itens de dívida técnica
2. Defina um plano de melhoria na especificação
3. Crie changes para reduzir gradualmente a dívida
4. Rastreie o progresso e a eficácia
5. Arquive as melhorias concluídas

## Integração com Outras Ferramentas

### Integração com Git

OpenSpec se integra perfeitamente ao Git:

```bash
# Documentos OpenSpec devem ser incluídos no controle de versão
git add openspec/
git commit -m "Add OpenSpec proposal for feature X"
```

### Integração CI/CD

Valide especificações OpenSpec em pipelines CI/CD:

```yaml
# .github/workflows/openspec.yml
name: OpenSpec Validation
on: [pull_request]
jobs:
  validate:
    runs-on: ubuntu-latest
    steps:
      - uses: actions/checkout@v2
      - name: Install OpenSpec
        run: npm install -g @fission-ai/openspec@1
      - name: Validate changes
        run: openspec change validate
```

### Integração com IDE

Editores recomendados com suporte Markdown para editar documentos OpenSpec:
- VS Code + plugins Markdown
- IDEs JetBrains + suporte Markdown
- Vim/Neovim + plugins Markdown

## Solução de Problemas

### Problemas Comuns

#### Problema: comando OpenSpec não encontrado

**Solução**:
1. Verifique se o OpenSpec está instalado: `npm list -g @fission-ai/openspec`
2. Verifique se a variável PATH inclui o diretório bin global do npm
3. Reinstale: `npm install -g @fission-ai/openspec@1`

#### Problema: não é possível criar propostas ou changes

**Solução**:
1. Certifique-se de que o diretório atual seja a raiz do projeto
2. Verifique se o diretório `openspec/` existe; se não, crie-o
3. Verifique as permissões de escrita

#### Problema: a validação falha

**Solução**:
1. Verifique se os documentos no diretório do change estão completos
2. Certifique-se de que todas as tarefas necessárias estejam concluídas
3. Veja os erros detalhados: `openspec change validate <change-name> --verbose`

#### Problema: o arquivamento falha

**Solução**:
1. Certifique-se de que o change passou na validação
2. Verifique se o diretório `openspec/archived/` existe
3. Certifique-se de que não há mudanças de código não commitadas

## Uso Avançado

### Templates Personalizados

OpenSpec suporta templates personalizados para propostas e especificações:

```bash
# Crie um diretório de templates na raiz do projeto
mkdir -p .openspec/templates

# Crie um template de proposta personalizado
cat > .openspec/templates/proposal.md << 'EOF'
# Proposal: {{title}}

## Problem Statement
{{problem}}

## Proposed Solution
{{solution}}

## Custom Section
Your custom content here
EOF
```

### Arquivo de Configuração

Crie um arquivo de configuração `.openspec.json` na raiz do projeto:

```json
{
  "version": "1.0",
  "templates": {
    "proposal": ".openspec/templates/proposal.md",
    "spec": ".openspec/templates/spec.md"
  },
  "validation": {
    "requireTests": true,
    "requireDocs": true
  },
  "hooks": {
    "pre-archive": "npm test"
  }
}
```

### Hooks

OpenSpec suporta a execução de scripts personalizados antes e depois de operações específicas:

```json
{
  "hooks": {
    "pre-create": "scripts/check-permissions.sh",
    "post-archive": "scripts/notify-team.sh"
  }
}
```
