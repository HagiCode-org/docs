## Cum să reproducem proiecte în era AI: Vault, un sistem de stocare persistentă cross-project

> În era dezvoltării asistate de AI, cum putem ajuta asistenții AI să înțeleagă mai bine resursele noastre de învățare? Proiectul HagiCode a construit sistemul Vault ca un strat de abstractizare unificat de stocare a cunoștințelor pe care AI îl poate înțelege, îmbunătățind semnificativ eficiența învățării prin reproducerea proiectelor.

## Context

În era AI, modul în care dezvoltatorii învață tehnologii noi și arhitecturi se schimbă profund. „Reproducerea proiectelor" — adică studiul aprofundat și învățarea din codul, arhitectura și tiparele de design ale proiectelor open source excelente — a devenit o modalitate eficientă de a învăța. Comparativ cu metode tradiționale precum cititul cărților sau vizionarea videoclipurilor, citirea și rularea directă a proiectelor open source de înaltă calitate vă ajută să înțelegeți practicile de inginerie din lumea reală mult mai rapid.

Totuși, această metodă de învățare vine cu destul de multe provocări.

Materialele de învățare sunt prea împrăștiate. Notele dumneavoastră pot fi în Obsidian, depozitele de cod pot fi răspândite în foldere diferite, iar istoricul conversațiilor asistentului dumneavoastră AI devine încă o insulă de date izolată. Când vreți ca AI să vă ajute să analizați un proiect, trebuie să copiați manual fragmente de cod și să organizați contextul, ceea ce este destul de tediabil.

Ceea ce este chiar mai deranjant este contextul fragmentat. Asistenții AI nu pot accesa direct resursele dumneavoastră locale de învățare, deci trebuie să furnizați informații de fundal din nou în fiecare conversație. Pe deasupra, depozitele de cod reproduse se actualizează rapid, sincronizarea manuală este predispusă la erori, iar cunoștințele sunt greu de partajat între multiple proiecte de învățare.

La rădăcină, toate aceste probleme provin de la „insule de date". Dacă ar exista un strat de abstractizare de stocare unificat care ar permite asistenților AI să înțeleagă și să acceseze toate resursele dumneavoastră de învățare, problema ar fi rezolvată elegant.

## Despre HagiCode

Sistemul Vault împărtășit în acest articol este exact soluția pe care am dezvoltat-o în timp ce construia [HagiCode](https://hagicode.com). HagiCode este un proiect de asistent de codare AI, iar în munca noastră zilnică de dezvoltare avem adesea nevoie să studiăm și să referim multe proiecte open source diferite. Pentru a ajuta asistenții AI să înțeleagă mai bine aceste resurse de învățare, am proiectat Vault, un sistem de stocare persistentă cross-project.

Această soluție a fost deja validată în utilizarea reală în HagiCode. Dacă vă confruntați cu provocări similare de management al cunoștințelor, sper că aceste experiențe vă pot oferi o inspirație. La urma urmei, odată ce ați căzut în câteva capcane, ar trebui să lăsați ceva în urmă pentru următoarea persoană.

## Filosofia de design a sistemului Vault

Ideea de bază a sistemului Vault este simplă: creați un strat de abstractizare unificat de stocare a cunoștințelor pe care AI îl poate înțelege. Din perspectiva implementării, sistemul are mai multe caracteristici cheie.

### Suport multi-tip

Sistemul suportă patru tipuri de vault, fiecare corespunzând unui scenariu de utilizare diferit:

```typescript
// folder: tip folder de uz general
export const DEFAULT_VAULT_TYPE = 'folder';

// coderef: un tip special pentru proiecte de cod reproduse
export const CODEREF_VAULT_TYPE = 'coderef';

// obsidian: integrat cu software-ul de notițe Obsidian
export const OBSIDIAN_VAULT_TYPE = 'obsidian';

// system-managed: vault gestionat automat de sistem
export const SYSTEM_MANAGED_VAULT_TYPE = 'system-managed';
```

Dintre acestea, tipul **coderef** este cel mai frecvent utilizat în HagiCode. Este conceput special pentru proiectele de cod reproduse, oferind o structură de directoare standardizată și descrieri de metadate lizibile de AI.

### Mecanismul de stocare persistentă

Registrul Vault este stocat persistent în format JSON, asigurând că configurația rămâne disponibilă după repornirea aplicației:

```csharp
public class VaultRegistryStore : IVaultRegistryStore
{
    private readonly string _registryFilePath;

    public VaultRegistryStore(IConfiguration configuration, ILogger<VaultRegistryStore> logger)
    {
        var dataDir = configuration["DataDir"] ?? "./data";
        var absoluteDataDir = Path.IsPathRooted(dataDir)
            ? dataDir
            : Path.GetFullPath(Path.Combine(Directory.GetCurrentDirectory(), dataDir));

        _registryFilePath = Path.Combine(absoluteDataDir, "personal-data", "vaults", "registry.json");
    }
}
```

Avantajul acestui design este că este simplu și fiabil. JSON este lizibil de om, ceea ce face depanarea și editarea manuală mai ușoare; stocarea pe sistemul de fișiere evită complexitatea unei baze de date și reduce dependențele sistemului. La urma urmei, uneori cea mai simplă opțiune este într-adevăr cea mai bună.

### Integrarea contextului AI

Cel mai important, sistemul poate injecta automat informații despre vault în contextul propunerilor AI:

```typescript
export function buildTargetVaultsText(
  vaults: VaultForText[],
  template: VaultPromptTemplate = DEFAULT_VAULT_PROMPT_TEMPLATE,
): string {
  const readOnlyVaults = vaults.filter((vault) => vault.accessType === 'read');
  const editableVaults = vaults.filter((vault) => vault.accessType === 'write');

  if (readOnlyVaults.length === 0 && editableVaults.length === 0) {
    return '';
  }

  const sections = [
    buildVaultSection(readOnlyVaults, template.reference),
    buildVaultSection(editableVaults, template.editable),
  ].filter(Boolean);

  return `\n\n### ${template.heading}\n\n${sections.join('\n')}`;
}
```

Aceasta permite o capabilitate importantă: asistenții AI pot înțelege automat resursele de învățare disponibile fără ca utilizatorii să furnizeze manual context. S-ar putea spune că acesta este un fel de înțelegere tacită.

## Structura standardizată a CodeRef Vault

Pentru tipul coderef de vault, HagiCode oferă o structură de directoare standardizată:

```
my-coderef-vault/
├── index.yaml          # descrierea metadatelor vault
├── AGENTS.md           # ghid de operare pentru asistenții AI
├── docs/               # stochează note de studiu și documente
└── repos/              # gestionează depozite de cod reproduse prin submoduluri Git
```

Când creați un vault, sistemul inițializează automat această structură:

```csharp
private async Task EnsureCodeRefStructureAsync(
    string vaultName,
    string physicalPath,
    ICollection<VaultBootstrapDiagnosticDto> diagnostics,
    CancellationToken cancellationToken)
{
    Directory.CreateDirectory(physicalPath);

    var indexPath = Path.Combine(physicalPath, CodeRefIndexFileName);
    var docsPath = Path.Combine(physicalPath, CodeRefDocsDirectoryName);
    var reposPath = Path.Combine(physicalPath, CodeRefReposDirectoryName);

    // Creare structură standard de directoare
    if (!Directory.Exists(docsPath))
    {
        Directory.CreateDirectory(docsPath);
    }

    if (!Directory.Exists(reposPath))
    {
        Directory.CreateDirectory(reposPath);
    }

    // Creare ghid AGENTS.md
    await EnsureCodeRefAgentsDocumentAsync(physicalPath, cancellationToken);

    // Creare metadate index.yaml
    await WriteCodeRefIndexDocumentAsync(indexPath, mergedDocument, cancellationToken);
}
```

Această structură este de asemenea atent proiectată:

- **docs/** stochează notele dumneavoastră de studiu, unde puteți înregistra în Markdown înțelegerea codului, analiza arhitecturii, lecții învățate, și așa mai departe
- **repos/** gestionează depozitele reproduse prin submoduluri Git în loc de copiere directă a codului, ceea ce menține codul sincronizat și economisește spațiu
- **index.yaml** conține metadatele vault astfel încât asistenții AI pot înțelege rapid scopul și conținutul vault-ului
- **AGENTS.md** este un ghid scris special pentru asistenții AI, explicând cum să gestioneze conținutul vault-ului

Organizat în acest mod, poate că AI-ul vă poate înțelege intențiile puțin mai ușor.

## Inițializarea automată pentru vault-uri gestionate de sistem

Pe lângă vault-urile create manual, HagiCode suportă și vault-uri gestionate de sistem:

```csharp
public async Task<IReadOnlyList<VaultRegistryEntry>> EnsureAllSystemManagedVaultsAsync(
    CancellationToken cancellationToken = default)
{
    var definitions = GetAllResolvedDefinitions();
    var entries = new List<VaultRegistryEntry>(definitions.Count);

    foreach (var definition in definitions)
    {
        entries.Add(await EnsureResolvedSystemManagedVaultAsync(definition, cancellationToken));
    }

    return entries;
}
```

Sistemul creează și gestionează automat următoarele vault-uri:

- **hagiprojectdata**: stocare date de proiect utilizată pentru salvarea configurației și stării proiectului
- **personaldata**: stocare date personale utilizată pentru salvarea preferințelor utilizatorului
- **hbsprompt**: o bibliotecă de șabloane de prompturi utilizată pentru gestionarea prompturilor AI frecvent utilizate

Aceste vault-uri sunt inițializate automat la pornirea sistemului, deci utilizatorii nu trebuie să le configureze manual. Unele lucruri sunt pur și simplu mai bine lăsate în seama sistemului în loc ca oamenii să se îngrijoreze de ele.

## Mecanismul de control al accesului

O parte importantă a designului este controlul accesului. Sistemul împarte vault-urile în două tipuri de acces:

```typescript
export interface VaultForText {
  id: string;
  name: string;
  type: string;
  physicalPath: string;
  accessType: 'read' | 'write';  // Cheie: distingerea între doar-citire și editabil
}
```

- **reference (doar-citire)**: AI este utilizat doar pentru analiză și înțelegere și nu poate modifica conținutul. Potrivit pentru proiecte open source referențiate, documente și materiale similare
- **editable (editabil)**: AI poate modifica conținutul după cum este necesar pentru sarcină. Potrivit pentru notele proprii, schițe și materiale similare

Această distincție contează. Spune AI-ului ce conținut este „referință doar-citire" și ce conținut este „sigur de editat", reducând riscul modificărilor accidentale. La urma urmei, nimeni nu vrea ca munca sa grea să dispară din cauza unei editări neintenționate.

## În practică: crearea și utilizarea Vault

Acum că am acoperit ideile, să ne uităm cum funcționează în practică.

### Crearea unui CodeRef Vault

Iată un exemplu complet de apelare frontend:

```typescript
const createCodeRefVault = async () => {
  const response = await VaultService.postApiVaults({
    requestBody: {
      name: "React Learning Vault",
      type: "coderef",
      physicalPath: "/Users/developer/vaults/react-learning",
      gitUrl: "https://github.com/facebook/react.git"
    }
  });

  // Sistemul va automat:
  // 1. Clona depozitul React în vault/repos/react
  // 2. Crea directorul docs/ pentru note
  // 3. Genera metadatele index.yaml
  // 4. Crea fișierul ghid AGENTS.md

  return response;
};
```

Acest apel API finalizează o serie de acțiuni: crearea structurii de directoare, inițializarea submodulurilor Git, generarea fișierelor de metadate, și altele. Trebuie doar să furnizați informațiile de bază și să lăsați sistemul să gestioneze restul. Este sincer o abordare destul de lipsită de griji.

### Utilizarea Vault într-o propunere AI

După crearea vault-ului, îl puteți referenția într-o propunere AI:

```typescript
const proposal = composeProposalChiefComplaint({
  chiefComplaint: "Help me analyze React's concurrent rendering mechanism",
  repositories: [
    { id: "react", gitUrl: "https://github.com/facebook/react.git" }

    {

```

Sistemul injectează automat informațiile despre vault în contextul AI, lăsând AI-ul să știe ce resurse de învățare sunt disponibile. Când AI-ul poate înțelege ce aveți în minte, acel tip de înțelegere tacită este greu de găsit.

## Cele mai bune practici și lucruri de urmărit

În timpul utilizării sistemului Vault, am rezumat câteva lecții învățate.

### Securitatea căilor

Sistemul validează strict căile pentru a preveni atacurile de traversare a căilor:

```csharp
private static string ResolveFilePath(string vaultRoot, string relativePath)
{
    var rootPath = EnsureTrailingSeparator(Path.GetFullPath(vaultRoot));
    var combinedPath = Path.GetFullPath(Path.Combine(rootPath, relativePath));
    if (!combinedPath.StartsWith(rootPath, StringComparison.OrdinalIgnoreCase))
    {
        throw new BusinessException(VaultRelativePathTraversalCode,
            "Vault file paths must stay inside the registered vault root.");
    }
    return combinedPath;
}
```

Acest lucru este important. Dacă personalizați o cale de vault, asigurați-vă că rămâne în domeniul permis, altfel sistemul va respinge operațiunea. Securitatea nu poate fi supraaccentuată niciodată.

### Managementul submodulurilor Git

CodeRef Vault recomandă submoduluri Git în loc de copiere directă a codului:

```csharp
private static string BuildCodeRefAgentsContent()
{
    return """
    # CodeRef Vault Guide

    Repositories under `repos/` should be maintained through Git submodules
    rather than copied directly into the vault root.

    Keep this structure stable so assistants and tools can understand the vault quickly.
    """ + Environment.NewLine;
}
```

Aceasta aduce mai multe avantaje: menținerea codului sincronizat cu upstream, economisirea spațiului pe disc și facilitarea gestionării mai multor versiuni ale codului. La urma urmei, cine vrea să descarce același lucru din nou și din nou?

### Limite de previzualizare a fișierelor

Pentru a preveni probleme de performanță, sistemul limitează dimensiunea și tipul fișierelor:

```csharp
private const int FileEnumerationLimit = 500;
private const int PreviewByteLimit = 256 * 1024;  // 256KB
```

Dacă vault-ul dumneavoastră conține un număr mare de fișiere sau fișiere foarte mari, performanța de previzualizare poate fi afectată. În acest caz, puteți lua în considerare procesarea fișierelor în loturi sau utilizarea instrumente de căutare specializate. Uneori când ceva devine prea mare, devine mai greu de gestionat, nu mai ușor.

### Informații de diagnosticare

La crearea unui vault, sistemul returnează informații de diagnosticare pentru a ajuta la depanare:

```csharp
List<VaultBootstrapDiagnosticDto> bootstrapDiagnostics = [];

if (IsCodeRefVaultType(normalizedType))
{
    bootstrapDiagnostics = await EnsureCodeRefBootstrapAsync(
        normalizedName,
        normalizedPhysicalPath,
        normalizedGitUrl,
        cancellationToken);
}
```

Dacă crearea eșuează, puteți inspecta informațiile de diagnosticare pentru a înțelege cauza specifică. Când ceva merge prost, verificarea diagnosticelor este adesea cea mai directă cale înainte.

## Rezumat

Printr-un strat de abstractizare de stocare unificat, sistemul Vault rezolvă mai multe puncte dureroase de bază ale reproducerii proiectelor în era AI:

- **Management centralizat al cunoștințelor**: toate resursele de învățare sunt adunate într-un singur loc în loc să fie împrăștiate peste tot
- **Injecție automată de context AI**: asistenții AI pot înțelege automat resursele de învățare disponibile fără configurare manuală a contextului
- **Reutilizare cross-project a cunoștințelor**: cunoștințele pot fi partajate și reutilizate între multiple proiecte de învățare
- **Structură standardizată de directoare**: un layout de directoare consistent reduce curba de învățare

Această soluție a fost deja validată în proiectul HagiCode. Dacă construiți și instrumente legate de dezvoltare asistată de AI, sau vă confruntați cu probleme similare de management al cunoștințelor, sper că aceste experiențe vă pot servi ca referință utilă.

Într-adevăr, valoarea unei soluții tehnice nu stă în cât de complicată este, ci dacă rezolvă probleme reale. Ideea de bază a sistemului Vault este foarte simplă: construiți un strat de stocare unificat de cunoștințe pe care AI îl poate înțelege. Totuși, tocmai această abstractizare simplă ne-a îmbunătățit eficiența dezvoltării destul de mult.

Uneori abordarea simplă este într-adevăr cea mai bună. La urma urmei, lucrurile complicate ascund adesea și mai multe capcane...
