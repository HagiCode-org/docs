## Construirea unui onboarding elegant pentru utilizatorii noi în proiecte React: Practica HagiCode cu driver.js

> Când utilizatorii deschid produsul dumneavoastră pentru prima dată, știu cu adevărat de unde să înceapă? În acest articol, vreau să vorbesc puțin despre cum am folosit driver.js pentru onboarding-ul utilizatorilor noi în proiectul HagiCode. Considerați-l un mic exemplu practic pentru a deschide discuția.

## Context

V-ați confruntat vreodată cu această situație? Un utilizator nou se înregistrează în produsul dumneavoastră, deschide pagina și imediat pare pierdut. Scanează în jur, nesigur ce să apese sau ce să facă mai departe. Ca dezvoltatori, presupunem adesea că utilizatorii vor „explora pe cont propriu" pentru că, la urma urmei, curiozitatea umană este nelimitată. Dar realitatea este diferită: majoritatea utilizatorilor pleacă în liniște în câteva minute dacă nu pot găsi punctul de intrare corect, ca și cum povestea începe brusc și se termină la fel de natural.

Onboarding-ul pentru utilizatorii noi este o modalitate importantă de a rezolva această problemă, dar construirea lui corectă nu este chiar atât de simplă. Un sistem de onboarding bun trebuie să:

- Localizeze precis elementele paginii și să le evidențieze
- Suporte fluxuri de onboarding cu mai mulți pași
- Memoreze alegerea utilizatorului (finalizează sau omite)
- Nu afecteze performanța paginii și interacțiunea normală
- Mențină structura codului clară și ușor de întreținut

În timp ce construia HagiCode, am întâmpinat aceeași provocare. HagiCode este un asistent de codare AI, iar fluxul său de lucru de bază este un flux de lucru OpenSpec care arată astfel: „utilizatorul creează un Proposal -> AI-ul generează un plan -> utilizatorul îl revizuie -> AI-ul îl execută". Pentru utilizatorii care întâlnesc acest concept pentru prima dată, fluxul de lucru este complet nou, deci au nevoie de onboarding solid pentru a începe rapid. Lucrurile noi iau mereu un pic de timp pentru a te obișnui.

## De ce am ales driver.js

În faza de evaluare tehnică, am analizat mai multe biblioteci de onboarding mainstream. Fiecare avea propriile puncte forte:

- **Intro.js**: Puternică, dar relativ mare, iar personalizarea stilurilor este oarecum complexă
- **Shepherd.js**: API bine proiectat, dar un pic prea „greu" pentru cazul nostru de utilizare
- **driver.js**: Ușoară, concisă, API intuitiv și funcționează bine în ecosistemul React

În cele din urmă, am ales driver.js. Nu a existat un motiv deosebit de dramatic. Alegerea a venit în principal din aceste considerații:

1. **Ușoară**: Biblioteca de bază este mică și nu mărește semnificativ dimensiunea bundle-ului
2. **API simplu**: Configurația este clară și intuitivă, deci este ușor de preluat
3. **Flexibilă**: Suportă poziționare personalizată, stiluri și comportament de interacțiune
4. **Import dinamic**: Poate fi încărcată la cerere fără a afecta performanța primului ecran

Cu selecția tehnologică, există rareori un răspuns universal cel mai bun. De obicei, există doar opțiunea care se potrivește cel mai bine.

## Implementare tehnică

### Configurația de bază

driver.js are un model de configurare foarte intuitiv. Iată configurația de bază pe care o folosim în proiectul HagiCode:

```typescript
import { driver } from 'driver.js';
import 'driver.js/dist/driver.css';

const newConversationDriver = driver({
  allowClose: true,           // Permite utilizatorilor să închidă ghidul
  animate: true,              // Activează animațiile
  overlayClickBehavior: 'close', // Închide ghidul când overlay-ul este clicuit
  disableActiveInteraction: false, // Menține elementele interactive
  showProgress: false,        // Nu arată bara de progres (gestionăm progresul noi)
  steps: guideSteps           // Array de pași de ghidare
});
```

Raționamentul din spatele acestor setări este:

- `allowClose: true` - Respectă alegerea utilizatorului și nu îl forța să termine ghidul
- `disableActiveInteraction: false` - Unii pași necesită acțiuni reale ale utilizatorului, cum ar fi tastarea, deci interacțiunea nu poate fi dezactivată
- `overlayClickBehavior: 'close'` - Oferă utilizatorilor o cale rapidă de ieșire

### Managementul stării

Persistarea stării de onboarding este critică. Nu vrem să repornim ghidul de fiecare dată când se reîmprospătează pagina, pentru că devine enervant repede. HagiCode folosește `localStorage` pentru a gestiona starea ghidului:

```typescript
export type GuideState = 'pending' | 'dismissed' | 'completed';

export interface UserGuideState {
  session: GuideState;
  detailGuides: Record<string, GuideState>;
}

// Citire stare
export const getUserGuideState = (): UserGuideState => {
  const state = localStorage.getItem('userGuideState');
  return state ? JSON.parse(state) : { session: 'pending', detailGuides: {} };
};

// Actualizare stare
export const setUserGuideState = (state: UserGuideState) => {
  localStorage.setItem('userGuideState', JSON.stringify(state));
};
```

Definim trei stări:

- `pending`: Ghidul este încă în desfășurare, iar utilizatorul nu l-a finalizat sau omis
- `dismissed`: Utilizatorul a închis ghidul proactiv
- `completed`: Utilizatorul a finalizat toți pașii

Pentru onboarding-ul paginii de detalii Proposal, suportăm și urmărirea stării mai granulare prin mapa `detailGuides`, deoarece un Proposal poate trece prin mai multe etape — schiță, revizuire și execuție completată — iar fiecare etapă necesită ghidare diferită. Starea lucrurilor se schimbă mereu, iar onboarding-ul ar trebui să reflecte acest lucru.

### Selecția elementelor țintă

driver.js folosește selectori CSS pentru a localiza elementele țintă. HagiCode urmează o convenție simplă: folosește un atribut `data-guide` personalizat pentru a marca țintele de onboarding:

```typescript
const steps = [
  {
    element: '[data-guide="launch"]',
    popover: {
      title: 'Start a New Conversation',
      description: 'Click here to create a new conversation session...'
    }
  }
];
```

În componente, arată astfel:

```typescript
<button data-guide="launch" onClick={handleLaunch}>
  New Conversation
</button>
```

Beneficiile acestei abordări sunt:

- Evită conflictele cu numele claselor de stil business
- Semantica clară, astfel încât puteți imediat spune că elementul este legat de onboarding
- Mai ușor de gestionat și întreținut consistent

### Optimizarea importului dinamic

Deoarece funcția de onboarding este necesară doar în scenarii specifice, cum ar fi prima vizită a unui utilizator, folosim importuri dinamice pentru a optimiza performanța de încărcare inițială:

```typescript
const initNewUserGuide = async () => {
  // Import dinamic driver.js
  const { driver } = await import('driver.js');
  await import('driver.js/dist/driver.css');

  // Inițializare ghid
  const newConversationDriver = driver({
    // ...configurare
  });

  newConversationDriver.drive();
};
```

Astfel, driver.js și foaia sa de stil sunt încărcate doar când este nevoie și nu afectează performanța primului ecran. Nu prea mulți oameni se bucură de așteptarea a ceva de care nici nu au nevoie încă.

## Designul fluxului de onboarding

HagiCode implementează două căi de onboarding care acoperă scenariile de bază ale utilizatorului.

### Onboarding Sesiune (10 Pași)

Această cale de onboarding ajută utilizatorii să finalizeze întregul flux de la crearea unei conversații până la trimiterea primului Proposal complet:

1. **launch** - Pornește ghidul și prezintă butonul „New Conversation"
2. **compose** - Ghidează utilizatorul să tasteze o cerere în caseta de introducere
3. **send** - Ghidează utilizatorul să apese butonul de trimitere
4. **proposal-launch-readme** - Ghidează utilizatorul să creeze un Proposal README
5. **proposal-compose-readme** - Ghidează utilizatorul să editeze conținutul cererii README
6. **proposal-submit-readme** - Ghidează utilizatorul să trimită Proposal-ul README
7. **proposal-launch-agents** - Ghidează utilizatorul să creeze un Proposal `AGENTS.md`
8. **proposal-compose-agents** - Ghidează utilizatorul să editeze cererea `AGENTS.md`
9. **proposal-submit-agents** - Ghidează utilizatorul să trimită Proposal-ul `AGENTS.md`
10. **proposal-wait** - Explică faptul că AI-ul procesează și utilizatorul ar trebui să aștepte un moment

Ideea din spatele acestei căi este de a lăsa utilizatorii să experimenteze fluxul de lucru de bază HagiCode direct prin două sarcini reale de creare Proposal, una pentru `README` și una pentru `AGENTS.md`. Există o mare diferență între a auzi despre un flux de lucru și a-l parcurge singur.

### Onboarding Detalii Proposal (3 Pași)

Când utilizatorii intră pe pagina de detalii a unui Proposal, HagiCode declanșează ghidul corespunzător pe baza stării curente a Proposal-ului:

1. **drafting** (etapa de schiță) - Ghidează utilizatorul să revizuiască planul generat de AI
2. **reviewing** (etapa de revizuire) - Ghidează utilizatorul să execute planul
3. **executionCompleted** (etapa completată) - Ghidează utilizatorul să arhiveze planul

Caracteristica definitorie a acestui ghid este că este condus de stare: decide dinamic ce pas de onboarding să arate pe baza stării reale a Proposal-ului. Lucrurile se schimbă, iar onboarding-ul ar trebui să se schimbe odată cu ele.

### Mecanismul de reîncercare a randării elementelor

În aplicațiile React, elementul țintă de onboarding poate să nu fi terminat randarea, de exemplu pentru că datele asincrone se încarcă încă. Pentru a gestiona acest lucru, HagiCode implementează un mecanism de reîncercare:

```typescript
const waitForElement = (selector: string, maxRetries = 10, interval = 100) => {
  let retries = 0;
  return new Promise<HTMLElement>((resolve, reject) => {
    const checkElement = () => {
      const element = document.querySelector(selector) as HTMLElement;
      if (element) {
        resolve(element);
      } else if (retries < maxRetries) {
        retries++;
        setTimeout(checkElement, interval);
      } else {
        reject(new Error(`Element not found: ${selector}`));
      }
    };
    checkElement();
  });
};
```

Apelați această funcție înainte de a inițializa ghidul pentru a vă asigura că elementul țintă există deja. Uneori, a aștepta un pic mai mult merită.

## Rezumatul celor mai bune practici

Pe baza experienței practice HagiCode, iată câteva practici recomandate cheie:

### 1. Onboarding-ul ar trebui să fie ușor de părăsit

Nu forțați utilizatorii să finalizeze onboarding-ul. Unii utilizatori sunt exploratori din fire și preferă să descopere lucruri pe cont propriu. Oferiți un buton clar „Skip", memorați-le alegerea și nu îi întrerupeți din nou data viitoare.

### 2. Mențineți conținutul de onboarding scurt și la obiect

Fiecare pas de onboarding ar trebui să se concentreze pe un singur obiectiv:

- **Titlu**: Scurt și clar, ideal nu mai mult de câteva cuvinte
- **Descriere**: Mergeți direct la subiect și spuneți utilizatorului ce este acesta și de ce contează

Evitați explicațiile lungi. Atenția utilizatorului este limitată în timpul onboarding-ului, iar cu cât spuneți mai mult, cu atât este mai puțin probabil să citească.

### 3. Mențineți selectorii stabili

Folosiți markeri de element stabili care nu se schimbă des. Atributul personalizat `data-guide` este o alegere bună. Evitați să depindeți de nume de clase sau structura DOM, pentru că acelea sunt ușor de schimbat în timpul refactorizărilor. Codul se schimbă tot timpul, dar unele ancore ar trebui să rămână stabile când este posibil.

### 4. Testați-vă onboarding-ul

HagiCode include cazuri de testare complete pentru funcția de onboarding:

```typescript
describe('NewUserConversationGuide', () => {
  it('should initialize guide state correctly', () => {
    const state = getUserGuideState();
    expect(state.session).toBe('pending');
  });

  it('should update guide state correctly', () => {
    setUserGuideState({ session: 'completed', detailGuides: {} });
    const state = getUserGuideState();
    expect(state.session).toBe('completed');
  });
});
```

Testele ajută să vă asigurați că refactorizarea nu strică accidental comportamentul de onboarding. Nimeni nu vrea ca o mică schimbare de cod să deterioreze în tăcere funcționalitatea care funcționa anterior.

### 5. Optimizarea performanței

- Folosiți importuri dinamice pentru a încărca leneș biblioteca de onboarding
- Evitați inițializarea logicii de onboarding după ce utilizatorul a finalizat deja ghidul
- Luați în considerare impactul animațiilor asupra performanței și dezactivați-le pe dispozitive mai slabe dacă este necesar

Performanța, ca multe lucruri în viață, merită un pic de bugetare atentă.

## Concluzie

Onboarding-ul pentru utilizatorii noi este o parte importantă a îmbunătățirii experienței utilizatorului produsului. În proiectul HagiCode, am folosit driver.js pentru a construi un sistem de onboarding complet care acoperă întregul flux de lucru de la crearea sesiunii până la execuția Proposal-ului.

Punctele cheie pe care sperăm să le împărtășim prin acest articol sunt:

1. **Alegerile tehnice ar trebui să corespundă nevoilor reale**: driver.js nu este cea mai puternică opțiune, dar este cea mai potrivită pentru noi
2. **Managementul stării este critic**: Folosiți `localStorage` pentru a persista starea de onboarding și a evita întreruperea repetată a utilizatorilor
3. **Designul de onboarding ar trebui să rămână concentrat**: Fiecare pas ar trebui să rezolve o singură problemă, nu mai mult
4. **Structura codului ar trebui să rămână clară**: Separați configurația de onboarding, managementul stării și logica UI pentru a facilita întreținerea

Dacă adăugați onboarding pentru utilizatori noi în propriul proiect, sper că experiența practică din acest articol vă ajută. Nu este nimic deosebit de mistic în acest tip de tehnologie. Continuați să încercați, continuați să rezumați ce învățați, iar lucrurile devin treptat mai ușoare.
