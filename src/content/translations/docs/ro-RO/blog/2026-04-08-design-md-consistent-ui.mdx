## Design.md: O soluție pentru design UI frontend consistent condus de AI

> În era dezvoltării frontend asistate de AI, cum puteți menține UI-urile generate de AI consistente? Acest articol împărtășește experiența noastră practică în construirea unui site galerie de design bazat pe awesome-design-md, împreună cu modul de a crea un `design.md` structurat pentru a ghida AI-ul spre design UI standardizat.

## Context

Oricine a folosit AI pentru a scrie cod frontend a probabil avut o experiență similară: cereți AI-ului să genereze aceeași pagină de mai multe ori, iar fiecare rezultat iese într-un stil diferit. Uneori colțurile sunt rotunjite, alteori sunt ascuțite. Uneori spațierea este 8px, iar alteori devine 16px. Chiar și același buton poate arăta diferit în conversații diferite.

Aceasta nu este o problemă izolată. Pe măsură ce dezvoltarea asistată de AI devine mai comună, **lipsa de consistență în UI-ul frontend generat de AI** a devenit o problemă răspândită. Asistenți AI diferiți, prompturi diferite, și chiar același asistent în conversații diferite pot produce designuri de interfață dramatic diferite. Asta creează un cost uriaș de mentenanță în timpul iterației produsului.

Cauza de bază este de fapt simplă: nu există un **document de referință de design autoritativ**. Foile de stil CSS tradiționale pot spune dezvoltatorilor „cum să implementeze" ceva, dar nu pot comunica complet „de ce este conceput așa" sau „ce tipar de design ar trebui folosit în ce scenariu". Pentru AI, o descriere clară și structurată este și mai importantă pentru înțelegerea convențiilor de design.

În același timp, comunitatea open source oferă deja unele resurse excelente. Proiectul [VoltAgent/awesome-design-md](https://github.com/VoltAgent/awesome-design-md) colectează documentație de sisteme de design de la multe companii cunoscute. Fiecare director conține un `README.md`, un `DESIGN.md` și HTML de previzualizare. Totuși, toate acestea sunt împrăștiate în depozitul upstream, făcând dificilă navigarea și compararea rapidă.

Deci, putem consolida acele resurse într-o galerie de design ușor de navigat, extrăgând în același timp un `design.md` structurat pentru utilizarea AI-ului?

Răspunsul este da. În continuare, să parcurgem abordarea noastră.

## Analiză

Înainte de a scrie cod, să descompunem provocările tehnice din spatele acestei probleme.

### Managementul conținutului sursă: Cum unificăm resursele de design împrăștiate?

Depozitul upstream awesome-design-md conține un număr mare de documente de design, dar aveam nevoie de o modalitate de a le aduce în propriul nostru proiect.

**Soluție: utilizare submodul git**

```
awesome-design-md-site
└── vendor/awesome-design-md          # Resurse upstream (submodul git)
```

Aceasta ne oferă mai multe beneficii:
- Controlul versiunilor: putem fixa o versiune upstream specifică
- Build-uri offline: nu este nevoie să solicităm API-uri externe în timpul build-ului
- Revizuirea conținutului: modificările specifice sunt vizibile în PR-uri

### Normalizarea datelor: Cum standardizăm structuri de documente diferite?

Companii diferite își structurează documentele de design diferit. Unele lipsesc fișiere de previzualizare, iar unele folosesc denumiri inconsistente. Trebuie să le normalizăm în timpul procesului de build.

**Soluție: scanare și generare de intrări normalizate la momentul build-ului**

Modulul de bază este `awesomeDesignCatalog.ts`, responsabil pentru:
1. Scanarea directorului `vendor/awesome-design-md/design-md/*`
2. Validarea dacă fiecare intrare conține fișierele necesare (`README.md`, `DESIGN.md` și cel puțin un fișier de previzualizare)
3. Extragerea și randarea conținutului Markdown în HTML
4. Generarea datelor de intrări normalizate

### Arhitectura site-ului static: Cum oferim căutare dinamică rămânând complet static?

Deoarece aceasta este o galerie de design, căutarea este o funcție necesară. Dar Astro este un generator de site-uri statice, deci cum implementați căutare în timp real?

**Soluție: React island + sincronizare parametri URL de interogare**

```typescript
// src/components/gallery/SearchToolbar.tsx

export function SearchToolbar() {
  const [query, setQuery] = useState('');

  // Sincronizare cu URL-ul
  useEffect(() => {
    const params = new URLSearchParams(window.location.search);
    setQuery(params.get('q') || '');
  }, []);

  // Filtrare în timp real
  const filtered = entries.filter(entry =>
    entry.searchText.includes(query)
  );

  return <input value={query} onChange={e => {
    setQuery(e.target.value);
    updateURL(e.target.value);
  }} />;
}
```

Avantajul acestei abordări este că menține capacitățile de implementare ale unui site static intacte, adică poate fi implementat pe orice serviciu de găzduire statică, oferind în același timp o experiență de filtrare instantă.

### Documentație de design: Cum ajutăm AI-ul să înțeleagă și să urmeze standardele de design?

Aceasta este nucleul întregii soluții. Trebuie să creăm un `design.md` structurat pe care AI îl poate înțelege și aplica.

**Soluție: împrumutarea structurii ClickHouse `DESIGN.md`**

[`DESIGN.md`-ul ClickHouse](https://github.com/ClickHouse/ClickHouse/blob/master/DESIGN.md) este o referință excelentă. Include:
- Visual Theme & Atmosphere
- Color Palette & Roles
- Typography Rules
- Component Stylings
- Layout Principles
- Depth & Elevation
- Do's and Don'ts
- Responsive Behavior
- Agent Prompt Guide

Abordarea noastră este: **reutilizați structura, rescrieți conținutul**. Păstrăm structura secțiunilor din ClickHouse `DESIGN.md`, dar înlocuim conținutul cu token-urile de design reale și convențiile de componente utilizate în propriul nostru proiect.

## Soluție

Pe baza analizei de mai sus, soluția noastră constă din patru module de bază.

### 1. Pipeline de ingestie a conținutului

Aceasta este fundația întregului sistem, responsabilă pentru extragerea și normalizarea conținutului din resursele upstream.

```typescript
// src/lib/content/awesomeDesignCatalog.ts

export async function scanSourceEntries(): Promise<DesignEntry[]> {
  const designDir = 'vendor/awesome-design-md/design-md';
  const entries: DesignEntry[] = [];

  for (const dir of await fs.readdir(designDir)) {
    const entryPath = path.join(designDir, dir);
    if (await isValidDesignEntry(entryPath)) {
      const entry = await normalizeDesignEntry(entryPath);
      entries.push(entry);
    }
  }

  return entries;
}

async function isValidDesignEntry(dir: string): Promise<boolean> {
  const requiredFiles = ['README.md', 'DESIGN.md'];
  for (const file of requiredFiles) {
    if (!(await fileExists(path.join(dir, file)))) {
      return false;
    }
  }
  return true;
}
```

### 2. Interfața de navigare a galeriei

Interfața galeriei include trei părți principale:

**Pagina principală**: afișează o grilă de carduri a tuturor intrărilor de design, iar fiecare card include:
- Titlul și rezumatul intrării de design
- Imagine de previzualizare, dacă este disponibilă
- Evidențiere căutare rapidă

**Pagina de detalii**: agreghează informațiile complete pentru o singură intrare de design:
- Document `README`
- Document `DESIGN`
- Previzualizare cu comutare temă clară/întunecată
- Navigare către intrări adiacente

**Navigare**: suportă revenirea la galerie și navigarea intrărilor adiacente

Galeria paginii principale folosește un layout de carduri cu densitate ridicată, aplatizând intrările `design.md` din surse diferite într-un framework vizual unificat astfel încât echipele pot compara rapid stiluri de brand, tipare de butoane și ritm tipografic.

![Grilă carduri de design Awesome Design MD Gallery](/article-images/design-md-consistent-ui/1775613199000-e652f7155dd5.png)

După deschiderea unei intrări specifice, pagina de detalii plasează rezumatul de design și previzualizarea live pe aceeași pagină, reducând costul comutării între documentație, previzualizări și cod sursă.

### 3. Funcționalitatea de căutare

Funcția de căutare este bazată pe filtrare client-side, cu starea păstrată prin parametri URL de interogare:

```typescript
// src/components/gallery/SearchToolbar.tsx

function SearchToolbar({ entries }: { entries: DesignEntry[] }) {
  const [query, setQuery] = useState('');
  const [results, setResults] = useState(entries);

  useEffect(() => {
    const params = new URLSearchParams(window.location.search);
    const initialQuery = params.get('q') || '';
    setQuery(initialQuery);
    filterEntries(initialQuery);
  }, []);

  const filterEntries = (searchQuery: string) => {
    const filtered = entries.filter(entry =>
      entry.searchText.toLowerCase().includes(searchQuery.toLowerCase())
    );
    setResults(filtered);
  };

  const handleChange = (e: React.ChangeEvent<HTMLInputElement>) => {
    const value = e.target.value;
    setQuery(value);
    filterEntries(value);

    // Actualizare URL fără declanșarea unei reîmprospătări a paginii
    const newUrl = value
      ? `${window.location.pathname}?q=${encodeURIComponent(value)}`
      : window.location.pathname;
    window.history.replaceState({}, '', newUrl);
  };

  return (
    <div className="search-toolbar">
      <input
        type="text"
        value={query}
        onChange={handleChange}
        placeholder="Search design entries..."
      />
      <span className="result-count">{results.length} results</span>
    </div>
  );
}
```

### 4. Documentul de referință de design (`design.md`)

Aceasta este livrabilul de bază al întregii soluții. Creăm un `design.md` în rădăcina proiectului cu următoarea structură:

Pe lângă conținutul brut `design.md` consumat de AI, plasăm de asemenea ambele documente `README` și `DESIGN` în aceeași interfață de citire, facilitând oamenilor corectura, copierea fragmentelor și compararea lor cu rezultatele previzualizării.

## Practică

Acum că am acoperit soluția, cum implementați de fapt?

### Pași de implementare

**Pasul 1: Inițializarea submodulului**

```bash
# Adăugare depozit upstream ca submodul
git submodule add https://github.com/VoltAgent/awesome-design-md.git vendor/awesome-design-md

# Inițializare și actualizare submodul
git submodule update --init --recursive
```

**Pasul 2: Crearea pipeline-ului de conținut**

Implementați `awesomeDesignCatalog.ts`, inclusiv:
- Logică de scanare și validare fișiere
- Randare Markdown folosind renderer-ul integrat Astro
- Extragere date intrări

**Pasul 3: Construirea UI-ului galeriei**

Folosiți Astro + React Islands pentru a crea:
- Layout galerie pagină principală (grilă carduri)
- Componente card de design
- Bară de instrumente căutare
- Layout pagină detalii

**Pasul 4: Scrierea documentului de design**

Pe baza structurii ClickHouse `DESIGN.md`, completați token-urile de design reale din propriul proiect. Actualizați `README.md` și adăugați un link către `design.md`.

### Note

**Securitate**: Randarea Markdown necesită filtrarea HTML-ului nesigur. Renderer-ul integrat Astro filtrează tag-urile `script` implicit, dar tot trebuie să fiți atenți la riscurile XSS.

**Performanță**: Un număr mare de previzualizări iframe poate afecta performanța primei vopsiri. Este recomandat să folosiți `loading="lazy"` pentru a încărca leneș conținutul de previzualizare.

**Mentenabilitate**: `design.md` trebuie să rămână sincronizat cu implementarea codului. Este recomandat să adăugați verificări CI pentru a asigura că variabilele CSS rămân consistente între documentație și cod.

**Accesibilitate**: Asigurați-vă că contrastul culorilor îndeplinește standardul WCAG AA (cel puțin 4.5:1).

### Ghid de utilizare AI

După crearea `design.md`, cum faceți AI-ul să îl folosească efectiv? Iată câteva sfaturi practice:

**Sfat 1: Referențiați-l explicit în prompt**

```
Please refer to the design.md file in the project root and use the design conventions defined there to implement the following components:
- Buttons: use the primary color with an 8px border radius
- Cards: use the elevation-2 shadow level
```

**Sfat 2: Solicitați AI-ului să referențieze variabile CSS specifice**

```
Implement a navigation bar with the following requirements:
- Use --color-bg-primary for the background color
- Use --color-border-subtle for borders
- Use --text-color-primary for text
```

**Sfat 3: Includeți conținutul `design.md` în promptul de sistem**

Dacă instrumentul dumneavoastră AI suportă prompturi de sistem personalizate, puteți adăuga conținutul de bază al `design.md` direct în ele.

### Strategia de testare

**Testare pipeline conținut**:
- Scenarii cu fișiere lipsă (lipsă `README.md` sau `DESIGN.md`)
- Scenarii cu erori de format (eșec parsare Markdown)
- Scenarii cu directoare goale

**Testare funcționalitate căutare**:
- Gestionare rezultate goale
- Caractere speciale precum chineză și emoji
- Verificare sincronizare URL

**Testare componente UI**:
- Comutare temă clară/întunecată
- Layout responsiv
- Stări de încărcare previzualizare

### Flux de lucru de implementare

```bash
# 1. Actualizare submodul la cea mai recentă versiune
git submodule update --remote

# 2. Rebuild site
npm run build

# 3. Implementare active statice
npm run deploy
```

Este recomandat să automatizați actualizările submodulurilor plus build și implementare, astfel încât CI să fie declanșat automat ori de câte ori depozitul upstream este actualizat.

## Concluzie

Inconsistența UI-ului generat de AI pe care HagiCode a întâmpinat-o în timpul dezvoltării a fost, la bază, cauzată de lipsa unui document de referință de design structurat. Prin construirea unui site galerie de design și crearea unui `design.md` standardizat, am rezolvat cu succes această problemă.

Valoarea de bază a acestei soluții constă în:
- **Resurse unificate**: consolidarea documentației de sisteme de design împrăștiate
- **Standarde structurate**: exprimarea convențiilor de design într-un format pe care AI îl poate înțelege
- **Mentenanță continuă**: menținerea conținutului actualizat prin `git submodule`

Dacă folosiți și AI pentru dezvoltare frontend, această abordare merită încercată. Crearea unui `design.md` structurat nu doar îmbunătățește consistența codului generat de AI, ci ajută și echipa dumneavoastră să mențină standarde de design unificate intern.
