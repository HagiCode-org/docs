## Arhitectură Practică Multi-Furnizor AI în Platforma HagiCode

> Acest articol prezintă abordarea tehnică pe care am folosit-o sub arhitectura Orleans Grain pentru a integra două instrumente AI, iflow și OpenCode, printr-o interfață unificată `IAIProvider` și compară în detaliu diferențele de implementare între comunicarea WebSocket și HTTP.

:::note[Context Temporal]
Acest articol înregistrează practica arhitecturii furnizorilor la momentul publicării. Pentru domeniul activ susținut în prezent, consultați [Instalare AI Agent CLI](/en-US/related-software-installation/ai-agent-cli/) și [Ghidul de Instalare](/en-US/installation); `IFlowCli` este acum păstrat doar ca valoare de compatibilitate istorică și nu face parte din domeniul de suport activ curent.
:::

## Context

Nu este nimic deosebit de misterios în acest lucru. La construirea HagiCode, ne-am confruntat cu o problemă foarte practică: utilizatorii doreau să lucreze cu diferite instrumente AI. Asta nu este surprinzător, deoarece fiecare are propriile obiceiuri. Unii preferă Claude Code, unii iubesc GitHub Copilot, iar unele echipe folosesc instrumente dezvoltate de ele.

Soluția noastră inițială a fost simplă și directă: scrie cod de integrare dedicat pentru fiecare instrument AI. Dar dezavantajele au apărut rapid. Baza de cod s-a umplut de ramuri `if-else`, fiecare modificare necesita testare în mai multe locuri și fiecare instrument nou însemna scrierea unei grămezi noi de logică de la zero.

Mai târziu, mi-am dat seama că ar fi mai bine să creez o interfață unificată `IAIProvider` și să abstractizez capacitățile partajate de toți furnizorii AI. În acest fel, indiferent ce instrument este folosit dedesubt, straturile superioare îl pot apela în același mod.

Recent, proiectul a trebuit să integreze două instrumente noi: iflow și OpenCode. Ambele suportă protocolul ACP, dar stilurile lor de comunicare sunt diferite. iflow folosește WebSocket, în timp ce OpenCode folosește un API HTTP. Asta a devenit un test arhitectural util: adaptează două moduri de transport diferite în spatele unei singure interfețe unificate.

## Despre HagiCode

Abordarea împărtășită în acest articol provine din experiența noastră practică în proiectul [HagiCode](https://hagicode.com). HagiCode este o platformă de dezvoltare asistată de AI construită pe arhitectura Orleans Grain. Se integrează cu diferiți furnizori AI printr-o interfață unificată `IAIProvider`, permițând utilizatorilor să aleagă flexibil instrumentele AI pe care le preferă.

## Design Arhitectural

### Abstracția Interfeței Unificate

Mai întâi, am definit interfața `IAIProvider` și am abstractizat capacitățile pe care fiecare furnizor AI trebuie să le implementeze:

```csharp
public interface IAIProvider
{
    string Name { get; }
    bool SupportsStreaming { get; }
    ProviderCapabilities Capabilities { get; }

    Task<AIResponse> ExecuteAsync(AIRequest request, CancellationToken cancellationToken = default);
    IAsyncEnumerable<AIStreamingChunk> StreamAsync(AIRequest request, CancellationToken cancellationToken = default);
    Task<ProviderTestResult> PingAsync(CancellationToken cancellationToken = default);
    IAsyncEnumerable<AIStreamingChunk> SendMessageAsync(AIRequest request, string? embeddedCommandPrompt = null, CancellationToken cancellationToken = default);
}
```

Această interfață include mai multe metode cheie:
- `ExecuteAsync`: execută o cerere AI unică
- `StreamAsync`: obține răspunsuri în flux pentru afișare în timp real
- `PingAsync`: efectuează o verificare de sănătate pentru a verifica dacă furnizorul este disponibil
- `SendMessageAsync`: trimite un mesaj cu suport pentru comenzi integrate

### IFlowCliProvider: O Implementare Bazată pe WebSocket

iflow folosește WebSocket pentru comunicarea ACP. Arhitectura generală arată astfel:

```
IFlowCliProvider → ACPSessionManager → WebSocketAcpTransport → iflow CLI
                ↓
         Dynamic port allocation + process management
```

Fluxul de bază este de asemenea destul de simplu:
1. `ACPSessionManager` creează și gestionează sesiuni ACP.
2. `WebSocketAcpTransport` gestionează comunicarea WebSocket.
3. Un port este alocat dinamic, iar procesul iflow este pornit cu `iflow --experimental-acp --port`.
4. `IAIRequestToAcpMapper` și `IAcpToAIResponseMapper` convertesc cererile și răspunsurile.

Iată codul de bază:

```csharp
private async IAsyncEnumerable<AIStreamingChunk> StreamCoreAsync(
    AIRequest request,
    string? embeddedCommandPrompt,
    [EnumeratorCancellation] CancellationToken cancellationToken)
{
    // Resolve working directory
    var resolvedWorkingDirectory = ResolveWorkingDirectory(request);
    var effectiveRequest = ApplyEmbeddedCommandPrompt(request, embeddedCommandPrompt);

    // Create ACP session
    await using var session = await _sessionManager.CreateSessionAsync(
        Name,
        resolvedWorkingDirectory,
        cancellationToken,
        request.SessionId);

    // Send prompt
    var prompt = _requestMapper.ToPromptString(effectiveRequest);
    var promptResponse = await session.SendPromptAsync(prompt, cancellationToken);

    // Receive streaming response
    await foreach (var notification in session.ReceiveUpdatesAsync(cancellationToken))
    {
        if (_responseMapper.TryConvertToStreamingChunk(notification, out var chunk))
        {
            if (chunk.Type == StreamingChunkType.Metadata && chunk.IsComplete)
            {
                yield return chunk;
                yield break;
            }
            yield return chunk;
        }
    }
}
```

Există câteva puncte de design care merită menționate aici:
- Folosește `await using` pentru a asigura eliberarea corectă a sesiunii și evitarea scurgerilor de resurse.
- Returnează răspunsuri în flux prin `IAsyncEnumerable`, care suportă natural fluxuri asincrone.
- Folosește fragmente `Metadata` pentru a determina finalizarea și a asigura primirea răspunsului complet.

### OpenCodeCliProvider: O Implementare Bazată pe API HTTP

OpenCode își oferă serviciul printr-un API HTTP, deci arhitectura este ușor diferită:

```plaintext
OpenCodeCliProvider → OpenCodeRuntimeManager → OpenCodeClient → OpenCode HTTP API
                      ↓
                OpenCodeProcessManager → opencode process management
```

O caracteristică notabilă a OpenCode este că folosește o bază de date SQLite pentru a persista asocierile de sesiune. Asta face posibilă recuperarea sesiunilor și recuperarea prompt-răspuns:

```csharp
private async Task<OpenCodePromptExecutionResult> ExecutePromptAsync(
    AIRequest request,
    string? embeddedCommandPrompt,
    CancellationToken cancellationToken)
{
    var prompt = BuildPrompt(request, embeddedCommandPrompt);
    var resolvedWorkingDirectory = ResolveWorkingDirectory(request.WorkingDirectory);
    var client = await _runtimeManager.GetClientAsync(resolvedWorkingDirectory, cancellationToken);
    var bindingSessionId = request.SessionId;
    var boundSession = TryGetBinding(bindingSessionId, resolvedWorkingDirectory);

    // Try to use the already bound session
    if (boundSession is not null)
    {
        try
        {
            return await PromptSessionAsync(
                client,
                boundSession,
                BuildPromptRequest(request, prompt, CreatePromptMessageId()),
                request.Model ?? _settings.Model,
                cancellationToken);
        }
        catch (OpenCodeApiException ex) when (IsStaleBinding(ex))
        {
            // The session has expired, remove the binding
            RemoveBinding(bindingSessionId);
        }
    }

    // Create a new session
    var session = await client.Session.CreateAsync(new OpenCodeSessionCreateRequest
    {
        Title = BuildSessionTitle(request)
    }, cancellationToken);

    BindSession(bindingSessionId, session.Id, resolvedWorkingDirectory);
    return await PromptSessionAsync(client, session.Id, ...);
}
```

Această implementare are câteva aspecte interesante:
- Mecanism de asociere a sesiunii: același `SessionId` reutilizează aceeași sesiune OpenCode, evitând crearea repetată a sesiunilor.
- Gestionarea expirării: când o sesiune este găsită expirată, asocierea este curățată automat.
- Persistență în baza de date: asocierile sunt stocate în SQLite și rămân eficiente după repornire.

### Compararea celor Două Abordări

| Aspect | IFlowCliProvider | OpenCodeCliProvider |
|------|-----------------|---------------------|
| Comunicare | WebSocket (ACP) | API HTTP |
| Gestionarea proceselor | ACPSessionManager | OpenCodeProcessManager |
| Alocarea porturilor | Port dinamic | Fără port (folosește HTTP) |
| Gestionarea sesiunilor | ACPSession | OpenCodeSession |
| Persistență | Cache în memorie | Bază de date SQLite |
| Comandă de pornire | `iflow --experimental-acp --port` | `opencode` |
| Latență | Mai mică (conexiune de lungă durată) | Relativ mai mare (cereri HTTP) |

Care abordare alegi depinde în principal de nevoile tale. WebSocket este mai bun pentru scenariile cu cerințe ridicate de timp real, în timp ce un API HTTP este mai simplu și mai ușor de depanat.

## Ghid Practic

### Configurarea Furnizorilor

Mai întâi, activează cei doi furnizori în fișierul de configurare:

```yaml
AI:
  Providers:
    IFlowCli:
      Type: "IFlowCli"
      Enabled: true
      ExecutablePath: "iflow"
      Model: null
      WorkingDirectory: null
    OpenCodeCli:
      Type: "OpenCodeCli"
      Enabled: true
      ExecutablePath: "opencode"
      Model: "anthropic/claude-sonnet-4"
      WorkingDirectory: null

OpenCode:
  Enabled: true
  BaseUrl: "http://localhost:38376"
  ExecutablePath: "opencode"
  StartupTimeoutSeconds: 30
  RequestTimeoutSeconds: 120
```

### Utilizarea IFlowCliProvider

```csharp
// Get provider through the factory
var provider = await _providerFactory.GetProviderAsync(AIProviderType.IFlowCli);

// Execute an AI request
var request = new AIRequest
{
    Prompt = "请帮我重构这个函数",
    WorkingDirectory = "/path/to/project",
    Model = "claude-sonnet-4"
};

// Get the complete response
var response = await provider.ExecuteAsync(request, cancellationToken);
Console.WriteLine(response.Content);

// Or use streaming responses
await foreach (var chunk in provider.StreamAsync(request, cancellationToken))
{
    if (chunk.Type == StreamingChunkType.ContentDelta)
    {
        Console.Write(chunk.Content);
    }
}
```

### Utilizarea OpenCodeCliProvider

```csharp
// Get provider through the factory
var provider = await _providerFactory.GetProviderAsync(AIProviderType.OpenCodeCli);

var request = new AIRequest
{
    Prompt = "请帮我分析这个错误",
    WorkingDirectory = "/path/to/project",
    Model = "anthropic/claude-sonnet-4"
};

var response = await provider.ExecuteAsync(request, cancellationToken);
Console.WriteLine(response.Content);
```

### Verificări de Sănătate

Înainte de pornire sau înainte de utilizare, poți verifica dacă furnizorul este disponibil:

```csharp
var iflowResult = await iflowProvider.PingAsync(cancellationToken);
if (!iflowResult.Success)
{
    Console.WriteLine($"IFlow is unavailable: {iflowResult.ErrorMessage}");
    return;
}

var openCodeResult = await openCodeProvider.PingAsync(cancellationToken);
if (!openCodeResult.Success)
{
    Console.WriteLine($"OpenCode is unavailable: {openCodeResult.ErrorMessage}");
    return;
}
```

### Suport pentru Comenzi Integrate

Ambii furnizori suportă comenzi integrate, cum ar fi `/file:xxx`:

```csharp
var request = new AIRequest
{
    Prompt = "分析这个文件的问题",
    SystemMessage = "你是一个代码分析专家"
};

await foreach (var chunk in provider.SendMessageAsync(
    request,
    embeddedCommandPrompt: "/file:src/main.cs",
    cancellationToken))
{
    Console.Write(chunk.Content);
}
```

## Note și Bune Practici

### Gestionarea Resurselor

IFlow folosește conexiuni WebSocket de lungă durată, deci gestionarea resurselor merită atenție specială:
- Folosește `await using` pentru a asigura eliberarea corectă a sesiunilor.
- Anularea declanșează curățarea proceselor.
- `ACPSessionManager` suportă o limită maximă de număr de sesiuni.

Gestionarea proceselor OpenCode este relativ mai simplă, iar `OpenCodeRuntimeManager` o gestionează automat.

### Gestionarea Erorilor

Ambii furnizori au o gestionare completă a erorilor:
- Erorile IFlow sunt propagate prin actualizări de sesiune ACP.
- Erorile OpenCode sunt aruncate prin `OpenCodeApiException`.
- Este recomandat ca apelantul să captureze și să gestioneze aceste excepții.

### Considerații de Performanță

- Comunicarea WebSocket IFlow are latență mai mică decât HTTP.
- Reutilizarea sesiunii OpenCode poate reduce suprasarcina cererilor HTTP.
- Mecanismul de cache al fabricii evită crearea repetată a furnizorilor.
- În scenarii de concurență ridicată, acordă atenție atentă limitelor de număr de procese și conexiuni.

### Validarea Configurației

Calea executabilă este validată la pornire, dar problemele la execuție pot apărea totuși. `PingAsync` este un instrument util pentru a verifica dacă configurația este corectă:

```csharp
// Check at startup
var provider = await _providerFactory.GetProviderAsync(providerType);
var result = await provider.PingAsync(cancellationToken);
if (!result.Success)
{
    _logger.LogError("Provider {ProviderType} is unavailable: {Error}", providerType, result.ErrorMessage);
}
```

## Rezumat

Acest articol prezintă abordarea tehnică folosită de platforma HagiCode la integrarea celor două instrumente AI iflow și OpenCode. Printr-o interfață unificată `IAIProvider`, am adaptat stiluri de comunicare diferite, WebSocket și HTTP, menținând în același timp consistent modelul de apelare de la nivelul superior.

Ideea de bază este de fapt destul de simplă:
1. Definește o abstracție de interfață unificată.
2. Construiește straturi adaptor pentru implementări diferite.
3. Gestionează totul uniform prin modelul fabricii.

Asta oferă sistemului o extensibilitate bună. Când un instrument AI nou trebuie integrat ulterior, tot ce trebuie să facem este să implementăm interfața `IAIProvider` fără a schimba prea mult cod existent.

Dacă lucrezi și tu la integrarea multi-instrumente AI, sper că acest articol îți este de ajutor.
