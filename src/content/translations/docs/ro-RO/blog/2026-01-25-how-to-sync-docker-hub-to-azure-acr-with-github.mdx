## Automatizarea sincronizării imaginilor de la Docker Hub la Azure ACR

> Acest articol explică modul de utilizare a GitHub Actions și a instrumentului image-syncer pentru a automatiza sincronizarea imaginilor de la Docker Hub la Azure Container Registry, abordând accesul lent la Docker Hub din China și unele regiuni Azure, îmbunătățind în același timp disponibilitatea imaginilor și eficiența implementării în mediile Azure.

## Context / Introducere

Proiectul HagiCode folosește imagini Docker ca componente de execuție de bază, cu imaginile principale găzduite pe Docker Hub. Pe măsură ce proiectul a crescut și nevoile de implementare Azure au crescut, am întâmpinat următoarele puncte dureroase:

- Viteze de tragere a imaginilor lente, cu accesul la Docker Hub limitat în China și în unele regiuni Azure
- O singură sursă de imagini creează un risc de punct unic de eșec
- Azure Container Registry oferă performanță de rețea și integrare mai bune când rulează în Azure

Pentru a rezolva aceste probleme, aveam nevoie de un mecanism automat de sincronizare a imaginilor pentru a sincroniza regulat imaginile de la Docker Hub la Azure ACR, asigurându-ne că utilizatorii pot obține viteze mai rapide de tragere a imaginilor și o disponibilitate mai mare în mediile Azure.

## Compararea soluțiilor tehnice

La definirea soluției, am comparat mai multe opțiuni tehnice:

### 1. image-syncer (alegerea finală)
- Sincronizare incrementală: sincronizează doar straturile de imagine modificate, reducând semnificativ transferul de rețea
- Suport pentru reluare: sincronizarea poate continua după întreruperi de rețea
- Controlul concurenței: fire de execuție worker configurabile care îmbunătățesc eficiența sincronizării imaginilor mari
- Gestionare robustă a erorilor: mecanism de reîncercare integrat pentru eșecuri (3 reîncercări implicit)
- Implementare ușoară: un singur binar fără dependențe
- Suport multi-registru: compatibil cu Docker Hub, Azure ACR, Harbor și altele

### 2. Docker CLI
- Fără sincronizare incrementală: conținutul complet al imaginii trebuie tras de fiecare dată
- Eficiență mai scăzută: volum mare de transfer de rețea și timp de execuție mai lung
- Simplu de utilizat: bazat pe comenzile familiare `docker pull` și `docker push`

### 3. Azure CLI
- Complexitate mai mare: autentificarea Azure CLI trebuie configurată
- Funcționalitate limitată: `az acr import` are un domeniu de aplicare relativ restrâns
- Integrare nativă: se integrează bine cu serviciile Azure

## Decizii de proiectare arhitecturală

### Decizia 1: Stabilirea frecvenței de sincronizare la zilnic, ora 00:00 UTC
- Echilibrează prospețimea imaginilor și consumul de resurse
- Evită orele de vârf de afaceri și reduce impactul asupra altor operațiuni
- Imaginile Docker Hub sunt de obicei actualizate după construirile zilnice

### Decizia 2: Sincronizarea tuturor etichetelor de imagine
- Menține o concordanță completă cu Docker Hub
- Oferă utilizatorilor opțiuni flexibile de versiune
- Simplifică logica de sincronizare și evită reguli complexe de filtrare a etichetelor

### Decizia 3: Utilizarea GitHub Secrets pentru stocarea acreditărilor
- Suport nativ GitHub Actions cu securitate puternică
- Configurare simplă și ușor de întreținut
- Suportă controlul accesului la nivel de depozit

## Evaluarea și atenuarea riscurilor

### Riscul 1: Scurgerea acreditărilor Azure ACR
- Stocarea acreditărilor criptate în GitHub Secrets
- Rotirea regulată a parolelor ACR
- Restricționarea contului ACR la permisiuni doar de push
- Monitorizarea jurnalelor de acces ACR

### Riscul 2: Eșecuri de sincronizare cauzând inconsistența imaginilor
- image-syncer include un mecanism integrat de sincronizare incrementală
- Reîncercări automate în caz de eșec (3 implicit)
- Jurnale de eroare detaliate și notificări de eșec
- Capacitate de reluare

### Riscul 3: Consum excesiv de resurse
- Sincronizarea incrementală reduce transferul de rețea
- Concurență configurabilă (`10` în configurația actuală)
- Monitorizarea numărului și dimensiunii imaginilor sincronizate
- Rularea sincronizării în afara orelor de vârf

## Soluția de bază

Folosim o soluție automatizată bazată pe GitHub Actions + image-syncer pentru a sincroniza imaginile de la Docker Hub la Azure ACR.

## Pași de implementare

### 1. Pregătire
- Creați sau confirmați un Azure Container Registry în Azure Portal
- Creați acreditări de acces ACR (nume de utilizator și parolă)
- Confirmați permisiunile de acces pentru depozitul de imagini Docker Hub

### 2. Configurarea GitHub Secrets
Adăugați următoarele Secrets în setările depozitului GitHub:
- AZURE_ACR_USERNAME: Numele de utilizator Azure ACR
- AZURE_ACR_PASSWORD: Parola Azure ACR

### 3. Crearea fluxului de lucru GitHub Actions
Configurați fluxul de lucru în `.github/workflows/sync-docker-acr.yml`:
- Declanșare programată: în fiecare zi la 00:00 UTC
- Declanșare manuală: suportă `workflow_dispatch`
- Declanșare suplimentară: rulează la push-uri pe ramura `publish` pentru sincronizare rapidă

### 4. Fluxul de execuție al fluxului de lucru

| Pas | Participant | Acțiune | Descriere |
| --- | --- | --- | --- |
| 1 | GitHub Actions | Declanșarea fluxului de lucru | Declanșat de programare, execuție manuală sau push-uri pe ramura `publish` |
| 2 | GitHub Actions -> image-syncer | Descărcarea și executarea instrumentului de sincronizare | Intră în faza reală de sincronizare |
| 3 | image-syncer -> Docker Hub | Obținerea manifestelor imaginii și a listei de etichete | Citește metadate din depozitul sursă |
| 4 | image-syncer -> Azure ACR | Obținerea datelor de imagine existente din depozitul țintă | Determină starea curentă a părții țintă |
| 5 | image-syncer | Compararea diferențelor dintre sursă și țintă | Identifică straturile de imagine care necesită sincronizare |
| 6 | image-syncer -> Docker Hub | Tragerea straturilor de imagine modificate | Transferă doar conținutul care necesită actualizare |
| 7 | image-syncer -> Azure ACR | Încărcarea straturilor de imagine modificate | Completează sincronizarea incrementală |
| 8 | image-syncer -> GitHub Actions | Returnarea statisticilor de sincronizare | Include rezultate, diferențe și informații despre erori |
| 9 | GitHub Actions | Înregistrarea jurnalelor și încărcarea artifactelor | Suportă auditare ulterioară și depanare |

## Implementarea fluxului de lucru GitHub Actions

Mai jos este configurația reală a fluxului de lucru în uz (`.github/workflows/sync-docker-acr.yml`):

```yaml
name: Sync Docker Image to Azure ACR

on:
  schedule:
    - cron: "0 0 * * *" # Every day at 00:00 UTC
  workflow_dispatch: # Manual trigger
  push:
    branches: [publish]

permissions:
  contents: read

jobs:
  sync:
    runs-on: ubuntu-latest

    steps:
      - name: Checkout code
        uses: actions/checkout@v4

      - name: Download image-syncer
        run: |
          # Download the image-syncer binary
          wget https://github.com/AliyunContainerService/image-syncer/releases/download/v1.5.5/image-syncer-v1.5.5-linux-amd64.tar.gz
          tar -zxvf image-syncer-v1.5.5-linux-amd64.tar.gz
          chmod +x image-syncer

      - name: Create auth config
        run: |
          # Generate the authentication config file (YAML format)
          cat > auth.yaml <<EOF
          hagicode.azurecr.io:
            username: "${{ secrets.AZURE_ACR_USERNAME }}"
            password: "${{ secrets.AZURE_ACR_PASSWORD }}"
          EOF

      - name: Create images config
        run: |
          # Generate the image sync config file (YAML format)
          cat > images.yaml <<EOF
          docker.io/newbe36524/hagicode: hagicode.azurecr.io/hagicode
          EOF

      - name: Run image-syncer
        run: |
          # Run synchronization (using the newer --auth and --images parameters)
          ./image-syncer --auth=./auth.yaml --images=./images.yaml --proc=10 --retries=3

      - name: Upload logs
        if: always()
        uses: actions/upload-artifact@v4
        with:
          name: sync-logs
          path: image-syncer-*.log
          retention-days: 7
```

## Detalii de configurare

### 1. Condiții de declanșare
- Declanșare programată: `cron: "0 0 * * *"` rulează în fiecare zi la 00:00 UTC
- Declanșare manuală: `workflow_dispatch` permite utilizatorilor să ruleze manual din interfața GitHub
- Declanșare la push: `push: branches: [publish]` declanșează la push-uri pe ramura de lansare pentru sincronizare rapidă

### 2. Configurarea autentificării (`auth.yaml`)
```yaml
hagicode.azurecr.io:
  username: "${{ secrets.AZURE_ACR_USERNAME }}"
  password: "${{ secrets.AZURE_ACR_PASSWORD }}"
```

### 3. Configurarea sincronizării imaginilor
```yaml
docker.io/newbe36524/hagicode: hagicode.azurecr.io/hagicode
```
Această configurație înseamnă că toate etichetele de la `docker.io/newbe36524/hagicode` vor fi sincronizate la `hagicode.azurecr.io/hagicode`.

### 4. Parametrii image-syncer
- `--auth=./auth.yaml`: calea către fișierul de configurare a autentificării
- `--images=./images.yaml`: calea către fișierul de configurare a sincronizării imaginilor
- `--proc=10`: utilizează `10` workeri concurenți
- `--retries=3`: reîncearcă eșecurile de `3` ori

## Note și bune practici

### 1. Recomandări de securitate
- Rotiți parolele Azure ACR regulat (recomandat la fiecare 90 de zile)
- Utilizați un cont de serviciu ACR dedicat cu permisiuni doar de push
- Monitorizați jurnalele de acces ACR pentru a detecta acces anormal în timp util
- Nu afișa acreditările în jurnale
- Nu comiteți acreditările în depozitul de cod

### 2. Optimizarea performanței
- Ajustați parametrul `--proc`: reglați concurența în funcție de lățimea de bandă a rețelei (recomandat `5-20`)
- Monitorizați durata sincronizării: dacă sincronizarea durează prea mult, luați în considerare reducerea concurenței
- Curățați jurnalele regulat: setați o valoare rezonabilă pentru `retention-days` (actual `7`)

### 3. Depanare

#### Problema 1: Eșec de autentificare
```
Error: failed to authenticate to hagicode.azurecr.io
```
Soluție:
1. Verificați dacă GitHub Secrets sunt configurate corect
2. Verificați dacă parola Azure ACR a expirat
3. Confirmați că permisiunile contului de serviciu ACR sunt corecte

#### Problema 2: Timeout de rețea
```
Error: timeout waiting for response
```
Soluție:
1. Verificați conectivitatea de rețea
2. Reduceți numărul de workeri concurenți (parametrul `--proc`)
3. Așteptați ca rețeaua să se recupereze și rulați din nou fluxul de lucru

#### Problema 3: Sincronizare incompletă a imaginilor
```
Warning: some tags failed to sync
```
Soluție:
1. Verificați jurnalele de sincronizare pentru a identifica etichetele eșuate
2. Rulați manual din nou fluxul de lucru pentru a sincroniza din nou
3. Verificați că imaginea sursă de pe Docker Hub este sănătoasă

### 4. Monitorizare și avertizare
- Verificați regulat pagina Actions pentru a confirma starea fluxului de lucru
- Activați notificările GitHub astfel încât eșecurile fluxului de lucru să fie raportate prompt
- Monitorizați utilizarea spațiului de stocare Azure ACR
- Verificați regulat consistența etichetelor între registre

## Întrebări frecvente și soluții

### Î1: Cum sincronizez etichete specifice în loc de toate etichetele?

Modificați fișierul de configurare `images.yaml`:
```yaml
# Sync only the latest and v1.0 tags
docker.io/newbe36524/hagicode:latest: hagicode.azurecr.io/hagicode:latest
docker.io/newbe36524/hagicode:v1.0: hagicode.azurecr.io/hagicode:v1.0
```

### Î2: Cum sincronizez mai multe depozite de imagini?

Adăugați mai multe linii în `images.yaml`:
```yaml
docker.io/newbe36524/hagicode: hagicode.azurecr.io/hagicode
docker.io/newbe36524/another-image: hagicode.azurecr.io/another-image
```

### Î3: Cum reîncerc după ce sincronizarea eșuează?
- Reîncercare automată: image-syncer include un mecanism de reîncercare integrat (`3` reîncercări implicit)
- Reîncercare manuală: faceți clic pe `Re-run all jobs` pe pagina GitHub Actions

### Î4: Cum pot vedea progresul detaliat al sincronizării?
- Vizualizați jurnalele în timp real pe pagina Actions
- Descărcați artifactul `sync-logs` pentru a inspecta fișierele de jurnal complete
- Fișierele de jurnal includ starea sincronizării și viteza de transfer pentru fiecare etichetă

### Î5: Cât durează sincronizarea?
- Sincronizare completă inițială: de obicei `10-30` minute în funcție de dimensiunea imaginii
- Sincronizare incrementală: de obicei `2-5` minute dacă modificările imaginii sunt mici
- Timpul depinde de lățimea de bandă a rețelei, dimensiunea imaginii și setările de concurență

### 2. Implementarea filtrării etichetelor de imagine
Adăugați logica de filtrare a etichetelor în fluxul de lucru:
```yaml
- name: Filter tags
  run: |
```

### 3. Adăugarea unui raport de statistici de sincronizare
```yaml
- name: Generate report
  if: always()
  run: |
    echo "## Sync Report" >> $GITHUB_STEP_SUMMARY
    echo "- Total tags: $(grep -c 'synced' image-syncer-*.log)" >> $GITHUB_STEP_SUMMARY
    echo "- Sync time: ${{ steps.sync.outputs.duration }}" >> $GITHUB_STEP_SUMMARY
```
