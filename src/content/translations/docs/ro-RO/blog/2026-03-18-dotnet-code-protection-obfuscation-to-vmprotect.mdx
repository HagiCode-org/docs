## Protecția codului .NET în practică: De la ofuscare la protecția prin mașină virtuală

> Acest articol explică modul de implementare a unei strategii de protecție a codului pe mai multe niveluri în proiectele .NET, acoperind parcursul complet de la ofuscarea de bază la protecția profesională prin mașină virtuală.

## Context

În dezvoltarea aplicațiilor .NET, protejarea codului de bază, cum ar fi validarea licențelor, logica de business și configurațiile sensibile, împotriva decompilării și ingineriei inverse este, sincer, un subiect pe care nu îl poți evita. Pe măsură ce ecosistemul .NET a maturizat, dezvoltatorii au acces la o gamă de opțiuni de protecție, de la atribute de ofuscare integrate la instrumente profesionale de protecție bazată pe virtualizare.

Ca proiect complex de monorepo multilingv, HagiCode include aplicații desktop, sisteme de build și capacități de gestionare a licențelor. Codul conține inevitabil logica de validare a licențelor, configurații sensibile precum chei API și ID-uri de produs, și logică critică de business. Aceste părți necesită protecție serioasă, pentru că nimeni nu vrea ca munca sa grea să fie expusă atât de ușor.

Acest articol prezintă abordarea de protecție a codului pe care am adoptat-o efectiv în proiectul HagiCode și rezumă întregul parcurs de la primele probleme întâmpinate până la optimizările ulterioare. Sperăm să vă oferă câteva idei utile.

## Conținut de bază

### 1. Atributul de ofuscare integrat de la Microsoft

.NET Framework oferă un atribut integrat `[ObfuscationAttribute]`, care este cel mai de bază și frecvent utilizat marcator de ofuscare a codului. Acest atribut se află în namespace-ul `System.Reflection` și vă permite să aplicați protecția de bază codului fără a introduce instrumente terțe.

**Caracteristici de bază**:
- Proprietatea `Feature`: Specifică funcționalitatea de ofuscare, cum ar fi `"ultra"` (ofuscare ridicată) sau `"all"` (ofuscare completă)
- Proprietatea `Exclude`: `true` înseamnă excludere din ofuscare, iar `false` înseamnă aplicare a ofuscării
- Poate fi aplicat la clase, metode, proprietăți și alți membri de tip

În proiectul HagiCode, îl puteți vedea utilizat astfel:

```csharp
[Obfuscation(Feature = "ultra", Exclude = false)]
public async Task<LicenseValidationResult?> ValidateLicenseAsync(...)
```

**Avantajele** acestei abordări sunt destul de evidente:
- Fără dependențe suplimentare; este integrat în .NET Framework
- Poate fi recunoscut și procesat de instrumente terțe de ofuscare
- Nu mărește semnificativ dimensiunea assembly-ului compilat

Acestea fiind spuse, are și limitări. Este doar un marcator, iar rezultatul real al ofuscării depinde de implementarea instrumentului. Nu poate oferi securitate la nivelul protecției prin mașină virtuală.

### 2. VMP (Protecție prin Mașină Virtuală)

VMP este un instrument profesional de protecție a codului care oferă protecție de nivel înalt compilând codul în instrucțiuni de mașină virtuală. Spre deosebire de simpla ofuscare a numelor, VMP transformă efectiv logica codului într-o formă pe care decompilatoarele convenționale nu o pot reconstrui.

**Clasificarea nivelurilor de protecție**:

| Nivel | Virtualizare | Mutație | Anti-depanare | Criptare șiruri | Cazuri de utilizare |
|------|--------|------|--------|------------|----------|
| HIGH | completă | ridicată | activată | activată | Validare licență, concurență sesiuni, constante sensibile |
| MEDIUM | parțială | medie | activată | activată | Logică de business, modele de domeniu |
| LOW | niciuna | scăzută | dezactivată | dezactivată | Clase utilitare, cod necritic |

Proiectul HagiCode definește un sistem de atribute declarative pentru marcarea codului care necesită protecție:

```csharp
// High-priority protection
[VmProtect(VmProtectionPriority.High, Reason = "Contains license verification logic")]
public class KeygenClient { ... }

// Exclude from protection
[VmExclude(Reason = "Public API that must remain unchanged")]
public class PublicApi { ... }

// Inherited protection
[VmProtect(Priority.High, ProtectDerived = true)]
public class BaseLicenseValidator { ... }
```

### 3. Strategia de protecție la momentul build-ului

Protecția VMP nu contează doar la execuție. De asemenea, trebuie automatizată ca parte a pipeline-ului de build, deoarece realizarea manuală ar fi mult prea obositoare. Sistemul de build HagiCode suportă mai multe moduri:

- **Mod nativ Windows**: Invocă direct instrumentul VMProtect
- **Mod container Linux Docker**: Rulează VMP într-un container pentru a rezolva problemele de compatibilitate cross-platform
- **Scanare a atributelor**: Descoperă automat marcatorii de protecție din cod
- **Mecanism de validare**: Confirmă că protecția a fost aplicată cu succes

Luate împreună, aceste capabilități fac procesul mult mai ușor de gestionat.

## Soluție

### 1. Utilizarea atributului de ofuscare integrat de la Microsoft

Aplicați `ObfuscationAttribute` direct în cod:

```csharp
using System.Reflection;

[Obfuscation(Feature = "ultra", Exclude = false)]
public class LicenseService
{
    [Obfuscation(Feature = "ultra", Exclude = false)]
    public async Task<bool> ValidateLicenseAsync(string key)
    {
        // License validation logic
    }

    [Obfuscation(Feature = "flow", Exclude = false)]
    private string DecryptToken(string encrypted)
    {
        // Decryption logic
    }
}
```

Uneori aveți nevoie să permiteți assembly-urilor de test să acceseze membrii interni, menținând în același timp codul de producție securizat:

```csharp
// AssemblyInfo.cs
[assembly: InternalsVisibleTo("HagiCode.Application.Tests")]
[assembly: InternalsVisibleTo("DynamicProxyGenAssembly2")] // for Moq
```

Acest lucru face testarea mult mai convenabilă, deoarece codul trebuie încă testat corespunzător.

### 2. Definiții de atribute personalizate pentru protecția VMP

Creați atribute de protecție personalizate pentru a controla comportamentul VMP:

```csharp
using System;

[AttributeUsage(AttributeTargets.Class | AttributeTargets.Method | AttributeTargets.Property)]
public class VmProtectAttribute : Attribute
{
    public VmProtectionPriority Priority { get; set; }
    public string? Reason { get; set; }
    public bool ProtectDerived { get; set; }
}

[AttributeUsage(AttributeTargets.Class | AttributeTargets.Method | AttributeTargets.Property)]
public class VmExcludeAttribute : Attribute
{
    public string? Reason { get; set; }
}

public enum VmProtectionPriority
{
    None = 0,
    Low = 1,
    Medium = 2,
    High = 3
}
```

Atributele personalizate sunt adesea mai ușor de lucrat deoarece reflectă direct propriile cerințe de protecție.

### 3. Fișierul de configurare VMP

```yaml
# vmp_config.yml
protection:
  priority_mode: "attribute"  # Attribute-based priority
  default_level: "medium"

tools:
  - name: "vmprotect"
    path: "C:\\Program Files\\VMProtect Ultimate\\VMProtect.exe"

protection_levels:
  high:
    virtualization: "full"
    mutation: "high"
    anti_debug: true
    anti_dump: true
    encrypt_strings: true
    encrypt_resources: true

  medium:
    virtualization: "partial"
    mutation: "medium"
    anti_debug: true
    encrypt_strings: true

  low:
    virtualization: "none"
    mutation: "low"
    anti_debug: false
```

O configurație mai clară face sistemul mult mai ușor de întreținut ulterior.

## Ghid practic

### 1. Practici de protecție pentru componente critice

Conform specificației de protecție a codului HagiCode, următoarele componente trebuie să folosească protecție cu prioritate HIGH:

```csharp
// Production constants - must be encrypted and protected by VMP
[VmProtect(VmProtectionPriority.High, Reason = "Production constants")]
public static class ProductionConstants
{
    // Encrypted string accessor, protected by VMP
    [VmProtect(VmProtectionPriority.High)]
    public static string GetLicenseServerUrl(IOptions<LicenseOptions> options) => ...;
}

// License validation logic
[VmProtect(VmProtectionPriority.High, Reason = "License verification logic")]
public class KeygenClient : IKeygenClient
{
    [Obfuscation(Feature = "ultra", Exclude = false)]
    public async Task<LicenseValidationResult?> ValidateLicenseAsync(...) { ... }
}

// Machine fingerprint service
[VmProtect(VmProtectionPriority.High)]
public class MachineFingerprintService : IMachineFingerprintService { ... }
```

Codul critic merită protecție mai puternică, deoarece expunerea logicii de bază ar cauza probleme reale.

### 2. Criptarea șirurilor și decriptarea la execuție

Criptați șirurile la momentul build-ului și decriptați-le la execuție:

```csharp
public static class StringDecryption
{
    [VmProtect(VmProtectionPriority.High, Reason = "CRITICAL SECURITY")]
    public static string DecryptString(byte[] encryptedData, byte[] key, byte[] iv)
    {
        using var aes = Aes.Create();
        aes.Key = key;
        aes.IV = iv;

        using var decryptor = aes.CreateDecryptor();
        using var ms = new MemoryStream(encryptedData);
        using var cs = new CryptoStream(ms, decryptor, CryptoStreamMode.Read);
        using var reader = new StreamReader(cs);

        return reader.ReadToEnd();
    }
}

// Production constant accessor (lazy loading + caching)
public static class ProductionConstants
{
    private static string? _cachedLicenseServerUrl;

    public static string GetLicenseServerUrl(IOptions<LicenseOptions> options)
    {
        if (_cachedLicenseServerUrl == null)
        {
            var encrypted = GetEncryptedLicenseServerUrl();
#if DEBUG
            _cachedLicenseServerUrl = options.Value.PrimaryServer.Url;
#else
            _cachedLicenseServerUrl = StringDecryption.DecryptString(
                encrypted,
                GetEncryptionKey(),
                GetEncryptionIV());
#endif
        }
        return _cachedLicenseServerUrl;
    }
}
```

Acest pas contează deoarece informațiile sensibile nu ar trebui niciodată lăsate în text clar.

### 3. Verificarea protecției VMP

După build, trebuie să verificați dacă protecția a fost aplicată cu succes; altfel, nu puteți fi siguri că funcționează efectiv:

```csharp
// Example verification script
public bool VerifyProtection(string assemblyPath)
{
    // 1. Check the VMP signature
    var bytes = File.ReadAllBytes(assemblyPath);
    var vmpSignature = Encoding.ASCII.GetBytes("VMProtect");
    if (bytes.Any(b => vmpSignature.Contains(b)))
    {
        return true;
    }

    // 2. Check for file size changes (the protected file is usually larger)
    var originalInfo = new FileInfo(assemblyPath.Replace(".dll", ".bak"));
    if (originalInfo.Exists)
    {
        var sizeRatio = (double)new FileInfo(assemblyPath).Length / originalInfo.Length;
        return sizeRatio > 1.1;
    }

    return false;
}
```

Verificarea merită întotdeauna făcută, deoarece altfel problemele pot trece neobservate.

### 4. Note și avertismente

Există câteva capcane aici care merită atenție specială:

1. **Nu ofuscați tot codul**: API-ile publice, definițiile interfețelor și clasele DTO de obicei nu au nevoie de protecție. Ofuscarea excesivă poate afecta performanța și depanarea. Proiectul HagiCode a învățat asta pe pielea sa.

2. **Protejați accessorii cheilor**: Metodele care recuperează cheile de criptare trebuie să primească același nivel de protecție sau superior datelor criptate; altfel întreaga configurare își pierde valoarea.

3. **Echilibrați testarea și producția**: Build-urile DEBUG ar trebui să sară criptarea pentru a face dezvoltarea și depanarea mai ușoare, în timp ce build-urile RELEASE ar trebui să activeze protecția completă. Nu uitați să le separați cu compilare condiționată precum `#if DEBUG`.

4. **Luați în considerare mediul Docker**: Rularea VMP pe Linux necesită o abordare containerizată pentru a asigura compatibilitatea instrumentelor. HagiCode folosește o soluție de container Wine + VMP pentru a rezolva problema cross-platform.

5. **Verificarea este obligatorie**: După ce build-ul se termină, trebuie să verificați că protecția a fost aplicată cu succes. Altfel codul sensibil poate fi încă expus, iar codul de verificare prezentat mai devreme există exact în acest scop.

## Concluzie

Cu această strategie de protecție pe mai multe niveluri, HagiCode a construit un sistem comprehensiv de securitate a codului care se întinde de la ofuscarea de bază la protecția prin mașină virtuală:

- **Nivelul 1**: Folosiți `ObfuscationAttribute` pentru marcarea de bază și oferiți indicii instrumentelor terțe
- **Nivelul 2**: Folosiți declarații `VmProtectAttribute` personalizate pentru a exprima intenția și prioritatea de protecție
- **Nivelul 3**: Folosiți protecția prin mașină virtuală VMP pentru a transforma codul critic în instrucțiuni de mașină virtuală ireversibile
- **Nivelul 4**: Scanarea și aplicarea automată a protecției în timpul build-ului, apoi verificarea rezultatului

Această abordare poate rezista instrumentelor obișnuite de decompilare și, de asemenea, să facă față mai bine atacurilor avansate de inginerie inversă. Dacă construiți o aplicație .NET care necesită protecția codului, sper că acest articol vă oferă cel puțin un punct de referință util.
