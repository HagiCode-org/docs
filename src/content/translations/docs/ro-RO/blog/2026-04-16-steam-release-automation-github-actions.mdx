## Cum să automatizați lansările Steam cu GitHub Actions

> Acest articol împărtășește soluția completă pe care am implementat-o pentru lansări Steam automate în proiectul HagiCode Desktop, acoperind fluxul de automatizare end-to-end de la GitHub Release la platforma Steam, inclusiv detalii tehnice cheie cum ar fi autentificarea Steam Guard și încărcarea multi-platformă Depot.

## Context

Fluxul de lansare pe Steam este de fapt destul de diferit de distribuția tradițională a aplicațiilor. Steam are propriul sistem complet de livrare a actualizărilor. Dezvoltatorii trebuie să folosească SteamCMD pentru a încărca artefactele de build pe rețeaua CDN a Steam, mai degrabă decât să adauge pur și simplu un link de descărcare ca pe alte platforme.

Proiectul HagiCode Desktop se pregătește pentru o lansare pe Steam, ceea ce a introdus câteva provocări noi în fluxul nostru de lansare:

1. Trebuia să convertim artefactele de build existente într-un format compatibil cu Steam.
2. Trebuia să le încărcăm pe Steam prin SteamCMD.
3. De asemenea, trebuia să gestionăm autentificarea Steam Guard.
4. Aveam nevoie de suport pentru încărcări Depot multi-platformă pentru Linux, Windows și macOS.
5. Doream un flux complet automatizat de la GitHub Release la Steam.

Proiectul implementase deja „modul versiune portabilă", care permite aplicației să detecteze payloads de serviciu fixe ambalate în directorul `extra`. Scopul nostru era să integrăm acel mod versiune portabilă fără întreruperi cu distribuția Steam.

## Despre HagiCode

Soluția împărtășită în acest articol provine din experiența noastră practică în proiectul [HagiCode](https://hagicode.com). HagiCode este un asistent de codare AI care suportă utilizare desktop. Deoarece lucrăm activ la lansarea pe Steam, a trebuit să stabilim un flux de lucru de lansare automatizat fiabil.

## Designul arhitecturii

Nucleul întregului proces de lansare Steam este un flux de lucru GitHub Actions care împarte procesul în trei etape principale:

```
┌─────────────────────────────────────────────────────────────┐
│ Flux de lucru GitHub Actions (Lansare Steam)                    │
├─────────────────────────────────────────────────────────────┤
│ 1. Etapa de pregătire:                                      │
│    - Checkout cod versiune-portabilă                       │
│    - Descărcare artefacte de build de la GitHub Release          │
│    - Extragere și pregătire a directorului de conținut Steam       │
│                                                             │
│ 2. Configurare SteamCMD:                                         │
│    - Instalare sau reutilizare SteamCMD                             │
│    - Autentificare cu Steam Guard                         │
│                                                             │
│ 3. Etapa de lansare:                                          │
│    - Generare fișiere de configurație Depot VDF                │
│    - Generare fișiere de configurație App Build VDF            │
│    - Invocare SteamCMD pentru încărcare pe Steam                    │
└─────────────────────────────────────────────────────────────┘
```

Acest design oferă mai multe avantaje:
- Reutilizează artefactele GitHub Release existente și evită reconstruirea acelorași rezultate.
- Folosește runner-uri self-hosted pentru izolare securizată.
- Suportă comutarea între modul preview și ramurile de lansare formală.
- Include gestionare completă a erorilor și logging, ceea ce face eșecurile mai ușor de diagnosticat.

## Implementarea fluxului de lucru

### Designul parametrilor de declanșare

Fluxul nostru de lucru suportă următorii parametri cheie:

```yaml
inputs:
  release:           # Tag-ul de lansare al versiunii portabile
    description: 'Version tag to release (for example v1.0.0)'
    required: true
  steam_preview:     # Dacă se generează un build de previzualizare
    description: 'Whether to use preview mode'
    required: false
    default: 'false'
  steam_branch:      # Ramura Steam de setat live
    description: 'Target Steam branch'
    required: false
    default: 'preview'
  steam_description: # Suprascrierea descrierii build-ului
    description: 'Build description'
    required: false
```

### Configurarea runner-ului self-hosted

Din motive de securitate, folosim un runner self-hosted cu eticheta `steam`:

```yaml
runs-on:
  - self-hosted
  - Linux
  - X64
  - steam
```

Acest lucru asigură că lansările Steam rulează pe un runner dedicat și menține credențialele sensibile izolate în siguranță.

### Controlul concurenței

Pentru a preveni interferența între lansările aceleiași versiuni, am configurat controlul concurenței:

```yaml
concurrency:
  group: portable-version-steam-${{ github.event.inputs.release }}
  cancel-in-progress: false
```

Observați că `cancel-in-progress: false` este setat aici deoarece lansările Steam pot dura ceva timp și nu dorim ca o rulare nou declanșată să anuleze una care încarcă deja.

## Implementarea scriptului de bază

### Pregătirea input-urilor de lansare

Scriptul `prepare-steam-release-input.mjs` este responsabil pentru pregătirea input-urilor necesare pentru lansare:

```javascript
// Descarcă manifestul de build GitHub Release și inventarul de artefacte
const buildManifest = await downloadBuildManifest(releaseTag);
const artifactInventory = await downloadArtifactInventory(releaseTag);

// Descarcă arhive comprimate pentru fiecare platformă
for (const platform of ['linux-x64', 'win-x64', 'osx-universal']) {
  const artifactUrl = getArtifactUrl(artifactInventory, platform);
  await downloadArtifact(artifactUrl, platform);
}

// Extrage în structura directorului de conținut Steam
await extractToSteamContent(sources, contentRoot);
```

### Autentificarea Steam Guard

Steam necesită ca conturile să fie protejate cu Steam Guard, deci am implementat un algoritm de generare de coduri bazat pe secret partajat:

```javascript
function generateSteamGuardCode(sharedSecret, timestamp = Date.now()) {
  const secret = decodeSharedSecret(sharedSecret);
  const time = Math.floor(timestamp / 1000 / 30);

  const timeBuffer = Buffer.alloc(8);
  timeBuffer.writeBigUInt64BE(BigInt(time));

  // Folosește HMAC-SHA1 pentru a genera un cod one-time bazat pe timp
  const hash = crypto.createHmac('sha1', secret)
    .update(timeBuffer)
    .digest();

  // Convertește-l într-un cod Steam Guard de 5 caractere
  const code = steamGuardCode(hash);
  return code;
}
```

Această implementare se bazează pe mecanismul TOTP (Time-based One-Time Password) al Steam Guard, generând un nou cod de verificare la fiecare 30 de secunde.

### Generarea configurației VDF

VDF (Valve Data Format) este formatul de configurație folosit de Steam. Trebuie să generăm două tipuri de fișiere VDF:

**Depot VDF** este folosit pentru a configura conținutul pentru fiecare platformă:

```javascript
function buildDepotVdf(depotId, contentRoot) {
  return [
    '"DepotBuildConfig"',
    '{',
    `  "DepotID" "${escapeVdf(depotId)}"`,
    `  "ContentRoot" "${escapeVdf(contentRoot)}"`,
    '  "FileMapping"',
    '  {',
    '    "LocalPath" "*"',
    '    "DepotPath" "."',
    '    "recursive" "1"',
    '  }',
    '}'
  ].join('\n');
}
```

**App Build VDF** este folosit pentru a configura întregul build de aplicație:

```javascript
function buildAppBuildVdf(appId, depotBuilds, description, setLive) {
  const vdf = [
    '"appbuild"',
    '{',
    `  "appid" "${appId}"`,
    `  "desc" "${escapeVdf(description)}"`,
    `  "contentroot" "${escapeVdf(contentRoot)}"`,
    '  "buildoutput" "build_output"',
    '  "depots"',
    '  {'
  ];

  for (const [depotId, depotVdfPath] of Object.entries(depotBuilds)) {
    vdf.push(`    "${depotId}" "${depotVdfPath}"`);
  }

  if (setLive) {
    vdf.push(`  }`);
    vdf.push(`  "setlive" "${setLive}"`);
  }

  vdf.push('}');
  return vdf.join('\n');
}
```

### Invocarea SteamCMD

În final, încărcarea este efectuată prin invocarea SteamCMD:

```javascript
await runCommand(steamcmdPath, [
  '+login', steamUsername, steamPassword, steamGuardCode,
  '+run_app_build', appBuildPath,
  '+quit'
]);
```

Acesta este saltul final din întregul flux de lucru. Odată ce reușește, lansarea este completă.

## Gestionarea Depot multi-platformă

Steam folosește sistemul Depot pentru a gestiona conținutul pentru platforme diferite. Suportăm trei Depots principale:

| Platformă | Identificator Depot | Suport arhitectură |
|-----|-----------|---------|
| Linux | `linux-x64` | x64_64 |
| Windows | `win-x64` | x64_64 |
| macOS | `osx-universal` | universal, x64_64, arm64 |

Fiecare Depot are propriul director de conținut și fișier de configurație VDF. Acest lucru asigură că utilizatorii pe platforme diferite descarcă doar conținutul de care au efectiv nevoie.

## Fluxul de lansare

### Pasul 1: Pregătirea GitHub Release

Mai întâi, creați un GitHub Release în repository-ul `portable-version` care include:
- Arhive comprimate pentru fiecare platformă
- Manifestul de build (`{tag}.build-manifest.json`)
- Inventarul de artefacte (`{tag}.artifact-inventory.json`)

### Pasul 2: Declanșarea fluxului de lansare Steam

Declanșați fluxul de lucru manual prin GitHub Actions și completați parametrii necesari:
- `release`: tag-ul de versiune de publicat, cum ar fi `v1.0.0`
- `steam_branch`: ramura țintă, cum ar fi `preview` sau `public`
- `steam_preview`: dacă se folosește modul preview

### Pasul 3: Executarea automată a fluxului de lansare

Fluxul de lucru efectuează automat următorii pași:

1. Descarcă și extrage artefactele GitHub Release.
2. Instalează sau actualizează SteamCMD.
3. Generează fișierele de configurație Steam VDF.
4. Autentificare cu Steam Guard.
5. Încarcă conținut pe CDN-ul Steam.
6. Setează ramura specificată ca live.

Odată ce această secvență se completează, întregul parcurs de lansare este acoperit.

## Ghid de configurare

### Configurarea Secrets necesare

Configurați următoarele secrets în setările repository-ului GitHub:

| Nume Secret | Descriere |
|-----------|------|
| `STEAM_USERNAME` | Numele de utilizator al contului Steam |
| `STEAM_PASSWORD` | Parola contului Steam |
| `STEAM_SHARED_SECRET` | Secretul partajat Steam Guard (opțional) |
| `STEAM_GUARD_CODE` | Codul Steam Guard (opțional) |
| `STEAM_APP_ID` | ID-ul aplicației Steam |
| `STEAM_DEPOT_ID_LINUX` | ID-ul Depot Linux |
| `STEAM_DEPOT_ID_WINDOWS` | ID-ul Depot Windows |
| `STEAM_DEPOT_ID_MACOS` | ID-ul Depot macOS |

Nu este nimic deosebit de neobișnuit în aceste setări. Pur și simplu trebuie să aveți toate valorile așteptate la locul lor.

### Configurarea variabilelor de mediu

| Nume variabilă | Descriere | Valoare implicită |
|---------|------|--------|
| `PORTABLE_VERSION_STEAMCMD_ROOT` | Directorul de instalare SteamCMD | `~/.local/share/portable-version/steamcmd` |

## Bune practici

### Gestionarea autentificării Steam Guard

La prima rulare, trebuie să introduceți manual codul Steam Guard. După aceea, este recomandat să configurați secretul partajat astfel încât codul să poată fi generat automat. Acest lucru evită intervenția manuală la fiecare lansare.

SteamCMD salvează token-ul de conectare și îl poate reutiliza la rulările ulterioare. Tot trebuie să aveți un ochi pe expirarea token-ului, deoarece re-autentificarea este necesară după ce token-ul expiră.

### Structura directorului de conținut

Asigurați-vă că structura directorului de conținut Steam este corectă:

```
steam-content/
├── linux-x64/     # Conținut platformă Linux
├── win-x64/       # Conținut platformă Windows
└── osx-universal/ # Conținut binar universal macOS
```

Fiecare director ar trebui să conțină fișierele complete ale aplicației pentru platforma corespunzătoare.

### Utilizarea modului preview

Modul preview nu setează nicio ramură ca live, deci este potrivit pentru testare și validare:

```yaml
if [ "$STEAM_PREVIEW_INPUT" = 'true' ]; then
  cmd+=(--preview)
fi
```

Acest lucru vă permite să încărcați mai întâi pe Steam pentru verificare, apoi să comutați la ramura formală după ce totul este în regulă.

### Gestionarea erorilor și logging

Scripturile includ gestionare completă a erorilor și logging:
- Validați că GitHub Release există.
- Verificați fișierele de metadate necesare.
- Asigurați-vă că conținutul platformelor este prezent.
- Generați un raport sumar GitHub Actions.

Aceste informații sunt extrem de valoroase atât pentru depanare, cât și pentru auditare.

### Gestionarea artefactelor

Fluxul de lucru generează două tipuri de artefacte:
- `portable-steam-release-preparation-{tag}`: metadate de pregătire a lansării
- `portable-steam-build-metadata-{tag}`: metadate de build Steam

Aceste artefacte pot fi folosite pentru auditare și depanare ulterioară. O perioadă de retenție de 30 de zile este un implicit practic.

## Utilizare în lumea reală

În proiectul HagiCode, acest flux de lansare automatizat a rulat deja cu succes pe mai multe versiuni. Tot parcursul de la GitHub Release la platforma Steam este complet automatizat și nu necesită intervenție manuală.

Acest lucru a îmbunătățit semnificativ atât eficiența, cât și fiabilitatea lansării. În trecut, publicarea manuală a unei versiuni dura mai mult de 30 de minute. Acum întregul proces se finalizează în doar câteva minute.

Mai important, fluxul de lucru automatizat reduce șansa de eroare umană. Fiecare lansare urmează același proces standardizat, ceea ce face rezultatele mai predictibile.
