## Tastarea este mai lentă decât vorbirea, iar vorbirea este mai lentă decât o captură de ecran - Practici de intrare multimodală pentru asistenții de codare AI

> Scrierea codului are o limită de viteză, indiferent cât de repede tastați. Uneori, ceva ce ați putea spune într-o propoziție durează o veșnicie să tastați; uneori, o singură captură de ecran explică totul, dar tot trebuie să o descrieți cu un munte de text. Acest articol vorbește despre ceea ce am întâlnit în timp ce construia [HagiCode](https://hagicode.com), de la recunoașterea vocală la încărcarea imaginilor. În cele din urmă, am vrut doar să facem un asistent de codare AI un pic mai ușor de folosit.

## Context

În timp ce construia HagiCode, am observat o problemă — sau mai degrabă, o problemă care a apărut în mod natural odată ce oamenii au început să-l folosească intensiv: bazarea doar pe tastare poate fi obositoare.

Gândiți-vă: interacțiunea dintre utilizatori și Agent este un scenariu de bază. Dar dacă fiecare schimb necesită tastare continuă la tastatură, eficiența nu este grozavă:

1. **Tastarea este prea lentă**: Pentru probleme complicate, cum ar fi mesaje de eroare sau probleme UI, tastarea tuturor poate dura o jumătate de minut, în timp ce a le spune cu voce tare poate dura zece secunde. Această diferență este reală.

2. **Imaginile sunt mai directe**: Uneori UI-ul aruncă o eroare, uneori doriți să comparați un șablon de design, iar uneori trebuie să arătați o structură de cod. „O imagine face cât o mie de cuvinte" poate fi o zicală veche, dar încă rezistă. Permiteți AI-ului să „vadă" direct problema este mult mai clar decât a o descrie la nesfârșit.

3. **Interacțiunea ar trebui să se simtă naturală**: Asistenții AI moderni ar trebui să suporte text, voce și imagini. Utilizatorii ar trebui să poată alege orice metodă de intrare li se pare cea mai naturală.

Așa că am decis să adăugăm recunoaștere vocală și suport pentru încărcare de imagini în HagiCode pentru a face interacțiunile cu Agent mai convenabile. Dacă utilizatorii pot tasta un pic mai puțin, asta este deja o victorie.

## Despre HagiCode

Soluțiile împărtășite în acest articol provin din munca noastră practică în proiectul [HagiCode](https://hagicode.com) — sau, mai exact, din lecțiile învățate în timp ce am căzut în destul de multe capcane.

HagiCode este un proiect de asistent de codare AI cu sursă deschisă cu un obiectiv simplu: folosi AI pentru a îmbunătăți eficiența dezvoltării. Pe măsură ce am continuat să construim, a devenit clar că utilizatorii își doreau cu tărie intrare multimodală. Uneori, a spune o propoziție este mai rapid decât a tasta un paragraf lung, iar uneori o captură de ecran este mult mai clară decât o lungă explicație.

Aceste nevoi ne-au împins înainte, și așa au luat forma funcții precum recunoașterea vocală și încărcarea imaginilor. Utilizatorii pot acum interacționa cu AI-ul în cel mai natural mod disponibil lor, și asta se simte bine.

## Analiză

### Provocări tehnice în recunoașterea vocală

Când am construit recunoașterea vocală, am întâlnit o problemă complicată: **API-ul WebSocket al browserului nu suportă headere HTTP personalizate**.

Serviciul de recunoaștere vocală pe care l-am ales a fost API-ul de Recunoaștere Vocală Doubao de la ByteDance. Din păcate, acest API necesită informații de autentificare precum `accessToken` și `secretKey` să fie transmise prin headere HTTP. Asta a creat un conflict tehnic imediat:

```javascript
// API-ul WebSocket al browserului nu suportă această abordare
const ws = new WebSocket('wss://api.com/ws', {
  headers: {
    'Authorization': 'Bearer token'  // Nu este suportat
  }
});
```

Practic, aveam două opțiuni:

1. **Abordarea cu parametri în URL-ul de interogare**: pune informațiile de autentificare în URL
   - Avantajul este că este simplu de implementat
   - Dezavantajul este că credențialele sunt expuse în frontend, ceea ce este nesigur; unele API-uri necesită și autentificare prin headere

2. **Abordarea cu proxy pe backend**: implementează un proxy WebSocket pe backend
   - Avantajul este că credențialele rămân stocate în siguranță pe backend, iar soluția este complet compatibilă cu cerințele API-ului
   - Dezavantajul este că implementarea este un pic mai complexă

În cele din urmă, am ales **abordarea cu proxy pe backend**. Securitatea nu este ceva cu care faci compromisuri.

### Cerințe funcționale pentru încărcarea imaginilor

Cerințele noastre pentru încărcarea imaginilor erau de f destul de simple:

1. **Metode multiple de încărcare**: clic pentru selectare fișier, tragere și plasare, și lipire din clipboard
2. **Validare fișiere**: restricții de tip (PNG, JPG, WebP, GIF) și limite de dimensiune (5-10 MB) sunt cerințe de bază
3. **Experiența utilizatorului**: progres de încărcare, previzualizări și mesaje de eroare pentru ca utilizatorii să știe mereu ce se întâmplă
4. **Securitate**: validare pe server și protecție împotriva încărcării de fișiere malițioase sunt esențiale

## Soluție

### Recunoaștere vocală: Arhitectura cu proxy WebSocket

Am proiectat o arhitectură cu trei straturi pentru recunoașterea vocală și am găsit o cale care a funcționat:

```
Browser WebSocket
       |
       | ws://backend/api/voice/ws
       | (audio binar)
       v
Backend Proxy
       |
       | wss://openspeech.bytedance.com/ (cu header de autentificare)
       v
Doubao API
```

**Implementări ale componentelor de bază**:

1. **Procesor Frontend AudioWorklet**:

```javascript
class AudioProcessorWorklet extends AudioWorkletProcessor {
  process(inputs, outputs, parameters) {
    const input = inputs[0]?.[0];
    if (!input) return true;

    // Reeșantionare la 16 kHz (necesar de API-ul Doubao)
    const samples = this.resampleAudio(input, 48000, 16000);

    // Acumulare de eșantioane în bucăți de 500 ms
    this.accumulatedSamples.push(...samples);

    if (this.accumulatedSamples.length >= 8000) {
      // Convertire în PCM pe 16 biți și trimitere
      const pcm = this.floatToPcm16(this.accumulatedSamples);
      this.port.postMessage({ type: 'audioData', data: pcm.buffer }, [pcm.buffer]);
      this.accumulatedSamples = [];
    }
    return true;
  }
}
```

2. **Handler WebSocket pe backend** (C#):

```csharp
[HttpGet("ws")]
public async Task GetWebSocket()
{
    if (HttpContext.WebSockets.IsWebSocketRequest)
    {
        await _webSocketHandler.HandleAsync(HttpContext);
    }
}
```

3. **Componenta frontend `VoiceTextArea`**:

```tsx
export const VoiceTextArea = forwardRef<HTMLTextAreaElement, VoiceTextAreaProps>(
  ({ value, onChange, onTextRecognized, maxDuration }, ref) => {
    const { isRecording, interimText, volume, duration, startRecording, stopRecording } =
      useVoiceRecording({ onTextRecognized, maxDuration });

    return (
      <div className="flex gap-2">
        {/* Buton vocal */}
        <button onClick={handleButtonClick}>
          {isRecording ? <VolumeWaveform volume={volume} /> : <Mic />}
        </button>
        {/* Zona de introducere text */}
        <textarea value={displayValue} onChange={handleChange} />
      </div>
    );
  }
);
```

### Încărcare imagini: Componentă de încărcare cu metode multiple

Am construit o componentă de încărcare de imagini complet funcțională cu suport pentru toate cele trei metode de încărcare, acoperind cele mai comune scenarii pe care le întâlnesc utilizatorii.

**Funcții de bază**:

1. **Trei metode de încărcare**:

```tsx
// Încărcare prin clic
const handleClick = () => fileInputRef.current?.click();

// Încărcare prin tragere și plasare
const handleDrop = (e: React.DragEvent) => {
  const file = e.dataTransfer.files?.[0];
  if (file) uploadFile(file);
};

// Lipire din clipboard
const handlePaste = (e: ClipboardEvent) => {
  for (const item of Array.from(e.clipboardData?.items || [])) {
    if (item.type.startsWith('image/')) {
      const file = item.getAsFile();
      if (file) uploadFile(file);
    }
  }
};
```

2. **Validare frontend**:

```tsx
const validateFile = (file: File): { valid: boolean; error?: string } => {
  if (!acceptedTypes.includes(file.type)) {
    return { valid: false, error: 'Only PNG, JPG, JPEG, WebP, and GIF images are allowed' };
  }
  if (file.size > maxSize) {
    return { valid: false, error: `Maximum file size is ${(maxSize / 1024 / 1024).toFixed(1)}MB` };
  }
  return { valid: true };
};
```

3. **Handler de încărcare pe backend** (TypeScript):

```typescript
export const Route = createFileRoute('/api/upload')({
  server: {
    handlers: {
      POST: async ({ request }) => {
        const formData = await request.formData();
        const file = formData.get('file') as File;

        // Validare
        const validation = validateFile(file);
        if (!validation.isValid) {
          return Response.json({ error: validation.error }, { status: 400 });
        }

        // Salvare fișier
        const uuid = uuidv4();
        const filePath = join(uploadDir, `${uuid}${extension}`);
        await writeFile(filePath, buffer);

        return Response.json({ url: `/uploaded/${today}/${uuid}${extension}` });
      }
    }
  }
});
```

## Ghid practic

### Cum să utilizați recunoașterea vocală

1. **Configurați serviciul de recunoaștere vocală**:
   - Deschideți pagina de setări pentru recunoaștere vocală
   - Configurați `AppId` și `AccessToken` Doubao Speech
   - Opțional, configurați cuvinte cheie pentru a îmbunătăți acuratețea recunoașterii pentru termeni specifici domeniului

2. **Folosiți-o în caseta de introducere**:
   - Faceți clic pe pictograma microfon din partea stângă a casetei de introducere
   - Începeți să vorbiți după ce apare animația formei de undă
   - Faceți clic din nou pe pictogramă pentru a opri înregistrarea
   - Textul recunoscut este inserat automat la poziția cursorului

3. **Exemplu de configurare a cuvintelor cheie**:

```text
TypeScript
React
useState
useEffect
```

### Cum să utilizați încărcarea imaginilor

1. **Metode de încărcare**:
   - Faceți clic pe butonul de încărcare pentru a alege un fișier
   - Trageți o imagine direct în zona de încărcare
   - Folosiți `Ctrl+V` pentru a lipi o captură de ecran din clipboard

2. **Formate susținute**: PNG, JPG, JPEG, WebP, GIF
3. **Limita de dimensiune**: 5 MB în mod implicit (configurabil)

### Note

1. **Recunoaștere vocală**:
   - Este necesară permisiunea microfonului
   - Folosiți într-un mediu liniștit când este posibil
   - Durata maximă de înregistrare susținută este de 300 de secunde în mod implicit (configurabil)

2. **Încărcare imagini**:
   - Sunt susținute doar formate comune de imagini
   - Acordați atenție limitelor de dimensiune a fișierelor
   - Imaginile încărcate primesc automat un URL de previzualizare

3. **Considerații de securitate**:
   - Credențialele de recunoaștere vocală sunt stocate pe backend
   - Încărcarea imaginilor trece prin validare strictă pe server
   - HTTPS/WSS este recomandat în mediile de producție

## Concluzie

După adăugarea recunoașterii vocale și a încărcării imaginilor, experiența utilizatorului HagiCode s-a îmbunătățit vizibil. Utilizatorii pot acum interacționa cu AI-ul într-un mod mai natural — vorbind în loc să tasteze și partajând capturi de ecran în loc să descrie totul manual. Se simte ca și cum am găsit în sfârșit o modalitate mai confortabilă de a comunica.

În timp ce construia această funcție, am întâmpinat problema că API-urile WebSocket ale browserului nu suportă headere personalizate. În cele din urmă, am rezolvat-o cu o abordare de proxy pe backend. Acea soluție nu doar a păstrat securitatea, ci a pus și bazele pentru integrarea ulterioară a altor servicii WebSocket autentificate.

Componenta de încărcare a imaginilor beneficiază și ea de suportul mai multor metode de încărcare, permițând utilizatorilor să aleagă ce este cel mai convenabil în acel moment. Click, tragere sau lipire — toate funcționează, iar fiecare cale finalizează treaba rapid.

„Tastarea este mai lentă decât vorbirea, iar vorbirea este mai lentă decât o captură de ecran" se potrivește temei destul de bine aici. Dacă construiți un produs de asistent AI similar, sper că aceste experiențe vă ajută, chiar dacă doar un pic.
