## Editarea DESIGN.md direct în interfața web: de la idee la implementare

> În sistemul de gestionare a proiectelor MonoSpecs, `DESIGN.md` conține designul arhitectural și deciziile tehnice ale unui proiect. Dar fluxul tradițional de editare obligă utilizatorii să sară la un editor extern. Acea experiență fragmentată este ca și cum ai fi întrerupt în mijlocul citirii unui poem: inspirația dispare, iar dispoziția la fel. Acest articol prezintă soluția pe care am pus-o în practică în proiectul HagiCode: editarea `DESIGN.md` direct în interfața web, cu suport pentru importul de șabloane de pe un site de design online. La urma urmei, cărui nu-i place sentimentul de a finaliza totul într-un singur flux?

## Context

Ca suport central al documentelor de design al proiectului, `DESIGN.md` deține informații cheie precum designul arhitectural, deciziile tehnice și ghidurile de implementare. Cu toate acestea, abordarea tradițională de editare necesită ca utilizatorii să comute la un editor extern precum VS Code, să localizeze manual calea fizică și apoi să editeze fișierul. Nu este deosebit de complicat, dar după ce repeti procesul de câteva ori, devine obositor.

Problemele se manifestă în principal în următoarele moduri:

- **Flux de lucru fragmentat**: utilizatorii trebuie să comute constant între interfața de gestionare web și un editor local, întrerupând continuitatea fluxului de lucru, la fel ca atunci când muzica se oprește în mijlocul unui cântec.
- **Dificil de reutilizat**: site-ul de design publică deja o bibliotecă bogată de șabloane de design, dar acestea nu pot fi integrate direct în fluxul de editare al proiectului. Lucrurile bune există, dar tot nu le poți folosi unde ai nevoie.
- **Lipsa buclei de experiență**: nu există o buclă închisă pentru „previzualizare-selecție-import”, deci utilizatorii trebuie să copieze și să lipească manual, ceea ce crește riscul de greșeli.
- **Fricțiune în colaborare**: menținerea sincronizării documentelor de design cu implementarea codului devine un proces cu multă frecare, ceea ce afectează eficiența echipei.

Pentru a rezolva aceste puncte dureroase, am decis să adăugăm suport de editare directă pentru `DESIGN.md` în interfața web și să permitem importul de șabloane cu un singur clic de pe un site de design online. Nu a fost o decizie care să zguduie pământul. Pur și simplu am vrut să facem experiența de dezvoltare mai fluidă.

## Despre HagiCode

Soluția prezentată în acest articol provine din experiența noastră practică în proiectul [HagiCode](https://hagicode.com). HagiCode este un proiect de asistent de codare bazat pe AI, iar în timpul dezvoltării avem frecvent nevoie să menținem documente de design ale proiectului. Pentru a ajuta echipa să colaboreze mai eficient, am explorat și implementat această soluție de editare online și import. Nu este nimic misterios în asta. Ne-am confruntat cu o problemă și am găsit o modalitate de a o rezolva.

## Soluția tehnică

### Arhitectura generală

Această soluție folosește o arhitectură separată frontend-backend cu un proxy de aceeași origine, compusă în principal din următoarele straturi. În practică, designul poate fi rezumat ca „fiecare parte își face propria treabă”:

**1. Stratul editorului frontend**
```typescript
// Componentă centrală: DesignMdManagementDrawer
// Locație: repos/web/src/components/project/DesignMdManagementDrawer.tsx
// Responsabilitate: gestionarea editării, salvării, detectării conflictelor de versiune și a fluxului de import
```

**2. Stratul serviciului backend**
```csharp
// Serviciu central: ProjectAppService.DesignMd
// Locație: repos/hagicode-core
// Responsabilitate: citirea fișierelor, salvarea, detectarea conflictelor, generarea diff-urilor
```

**3. Stratul proxy de aceeași origine**
```csharp
// Configurare proxy: Program.cs
// Responsabilitate: direcționarea cererilor API către backend-ul local, rezolvarea problemelor CORS
```

**4. Stratul de date**
```
// DESIGN.md ca sursă unică de adevăr
// Stocare: depozitul de fișiere al proiectului
// Formate suportate: Markdown + MonoSpecs
```

### Implementarea detaliată

#### 1. Componenta editor frontend

Componenta centrală este `DesignMdManagementDrawer`, un panou lateral care oferă funcții complete de editare:

```tsx
// Funcționalități principale:
// - Editor de text bogat (bazat pe Monaco Editor)
// - Previzualizare în timp real
// - Detectarea conflictelor de versiune
// - Import șabloane cu un singur clic
```

#### 2. Serviciul backend

Backend-ul oferă următoarele API-uri cheie:

```csharp
// GET  /api/project/{id}/design-md      - Obține conținutul DESIGN.md
// PUT  /api/project/{id}/design-md      - Salvează conținutul DESIGN.md
// POST /api/project/{id}/design-md/diff - Generează diff pentru detectarea conflictelor
// POST /api/project/{id}/design-md/import - Importă șablon extern
```

#### 3. Detectarea conflictelor de versiune

Când doi utilizatori editează același `DESIGN.md` în același timp, este posibil să apară conflicte de versiune. Soluția noastră:

```typescript
// Înainte de salvare, obține versiunea curentă de pe server
// Compară cu versiunea din momentul încărcării
// Dacă există diferențe, arată dialogul de conflict
// Utilizatorul alege: suprascriere, îmbinare sau abandonare
```

#### 4. Importul de șabloane

Funcția de import de șabloane permite utilizatorilor să importe direct șabloane de design de pe site-ul online:

```typescript
// Fluxul de import:
// 1. Navighează biblioteca de șabloane în panoul lateral
// 2. Previzualizează conținutul șablonului
// 3. Confirmă importul
// 4. Conținutul este inserat în editor
```

### Considerații de securitate

- Toate cererile API trec prin proxy-ul de aceeași origine, evitând problemele CORS
- Operațiunile de scriere necesită autentificare și autorizare
- Fiecare salvare generează un diff pentru trasabilitate
- Sistemul suportă revenirea la versiuni anterioare

## Rezumat

Această soluție poate părea simplă, dar adresează mai multe puncte dureroase reale:

1. **Flux de lucru unificat**: nu mai este nevoie să comuți între diferite instrumente, editarea și previzualizarea se fac în același loc
2. **Reutilizare îmbunătățită**: șabloanele de design pot fi importate direct, fără copiere-lipire manuală
3. **Colaborare mai bună**: detectarea conflictelor de versiune reduce riscul de suprascriere accidentală
4. **Experiență mai fluidă**: de la idee la implementare, totul se întâmplă într-un singur flux continuu

Designul tehnic nu este complicat, dar detaliile contează. Sperăm că această experiență de partajare va fi utilă celor care întâmpină probleme similare.
