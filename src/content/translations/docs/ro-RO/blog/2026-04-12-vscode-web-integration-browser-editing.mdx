## Editare rapidă a codului în browser: Integrarea VSCode Web în practică

> După ce AI-ul termină analiza codului, cum deschideți imediat un editor în browser și începeți să faceți modificări? Acest articol împărtășește experiența noastră practică de integrare a code-server în proiectul HagiCode pentru a crea o punte fără întreruperi între asistentul AI și experiența de editare a codului.

## Context

În era programării asistate de AI, dezvoltatorii au adesea nevoie să inspecteze și să editeze codul rapid. Fluxul de lucru tradițional este simplu: deschideți proiectul într-un IDE desktop, localizați fișierul, editați-l și salvați. Dar în unele situații, acest flux pare întotdeauna ușor incomod.

Scenariul unu: dezvoltare la distanță. Când folosiți un asistent AI precum HagiCode, backend-ul poate rula pe un server la distanță sau într-un container, iar mașinile locale nu pot accesa direct fișierele proiectului. De fiecare dată când trebuie să inspectați sau să modificați cod, trebuie să vă conectați prin SSH sau o altă metodă, iar experiența se simte fragmentată. Este ca și cum ați vrea să întâlniți pe cineva printr-un geam gros: îi puteți vedea, dar nu îi puteți atinge.

Scenariul doi: previzualizări rapide. După ce asistentul AI analizează codul, utilizatorul poate dori doar să răsfoiască rapid un fișier sau să facă o mică modificare. Lansarea unui IDE desktop complet pare greoaie, în timp ce un editor ușor în browser se potrivește mai bine nevoii pentru „o privire rapidă". La urma urmei, cine vrea să mobilizeze un întreg lanț de instrumente doar pentru o aruncătură de ochi?

Scenariul trei: colaborare cross-dispozitiv. Când lucrați pe dispozitive diferite, un editor bazat pe browser oferă un punct de acces unificat fără a necesita ca fiecare mașină să fie configurată cu un mediu de dezvoltare. Asta singur economisește multă bătaie de cap. Viața este scurtă; de ce să repetați aceeași muncă de configurare iar și iar?

Pentru a rezolva aceste puncte dureroase, am integrat VSCode Web în proiectul HagiCode. Acest lucru permite asistentului AI și experienței de editare a codului să se conecteze fără întreruperi: după ce AI-ul analizează codul, utilizatorii pot deschide imediat un editor și pot face modificări în aceeași sesiune de browser, fără a schimba aplicațiile. Este genul de experiență în care, atunci când aveți nevoie de ea, pur și simplu este acolo.

## Despre HagiCode

Soluția împărtășită în acest articol provine din experiența noastră practică în proiectul [HagiCode](https://hagicode.com). HagiCode este un asistent de codare bazat pe AI conceput pentru a îmbunătăți eficiența dezvoltării prin interacțiune în limbaj natural. În timpul dezvoltării, am descoperit că utilizatorii au adesea nevoie să comute rapid între analiza AI și editarea codului, ceea ce ne-a determinat să explorăm cum să integrăm editorul direct în browser.

Repository-ul proiectului: [github.com/HagiCode-org/site](https://github.com/HagiCode-org/site)

## Alegerea tehnică: De ce code-server?

Printre multele soluții VSCode Web disponibile, am ales code-server. Au existat câteva motive concrete în spatele acestei decizii.

Completitudinea funcțională. code-server este versiunea web a VSCode și suportă majoritatea funcțiilor desktop, inclusiv sistemul de extensii, sugestiile inteligente, depanarea și multe altele. Asta înseamnă că utilizatorii pot obține o experiență de editare în browser foarte apropiată de versiunea desktop. La urma urmei, cine vrea cu adevărat să facă compromisuri în privința funcționalității?

Implementare flexibilă. code-server poate rula ca serviciu independent și suportă, de asemenea, implementare bazată pe Docker, ceea ce se potrivește bine cu arhitectura HagiCode. Backend-ul nostru este scris în C#, frontend-ul folosește React, iar cele două comunică cu serviciul code-server prin API-uri REST. Este ca și cum ați construi cu blocuri: fiecare piesă are locul ei.

Autentificare securizată. code-server include un mecanism `connection-token` integrat pentru a preveni accesul neautorizat. Fiecare sesiune are un token unic astfel încât doar utilizatorii autorizați pot deschide editorul. Securitatea este unul dintre acele lucruri pe care le apreciați pe deplin doar după ce o aveți.

## Designul arhitecturii

Integrarea VSCode Web a HagiCode folosește o arhitectură separată frontend/backend.

### Stratul de servicii frontend

Frontend-ul împachetează interacțiunile cu backend-ul prin `vscodeServerService.ts`:

```typescript
// Deschide proiectul
export async function openProjectInCodeServer(
  id: string,
  currentInterfaceLanguage?: string,
): Promise<VsCodeServerLaunchResponseDto>

// Deschide vault-ul
export async function openVaultInCodeServer(
  id: string,
  path?: string,
  currentInterfaceLanguage?: string,
): Promise<VsCodeServerLaunchResponseDto>
```

Diferența dintre aceste două metode este simplă: `openProjectInCodeServer` deschide întregul proiect, în timp ce `openVaultInCodeServer` deschide o cale specifică într-un Vault. Pentru proiectele MonoSpecs cu multiple repository-uri, sistemul creează automat un fișier workspace. Responsabilități clare sunt adesea suficiente atunci când fiecare parte își face bine treaba ei.

### Stratul de servicii backend

Backend-ul `VaultAppService.cs` implementează logica de bază:

```csharp
public async Task<VsCodeServerLaunchResponseDto> OpenInCodeServerAsync(
    string id,
    string? relativePath = null,
    string? currentInterfaceLanguage = null,
    CancellationToken cancellationToken = default)
{
    // 1. Obține setările și verifică dacă funcția este activată
    var settings = await _vsCodeServerSettingsService.GetResolvedSettingsAsync(cancellationToken);
    if (!settings.Enabled) {
        throw new BusinessException(VsCodeServerErrorCodes.Disabled, "VSCode Server is disabled.");
    }

    // 2. Obține vault-ul și rezolvă directorul de lansare
    var vault = await RequireVaultAsync(id, cancellationToken);
    var launchDirectory = ResolveLaunchDirectory(vault, relativePath);

    // 3. Asigură-te că code-server rulează și obține informații de runtime
    var runtime = await _vsCodeServerManager.EnsureStartedAsync(settings, cancellationToken);

    // 4. Rezolvă setările de limbă
    var language = _vsCodeServerSettingsService.ResolveLaunchLanguage(
        settings.Language,
        currentInterfaceLanguage);

    // 5. Construiește URL-ul de lansare
    return new VsCodeServerLaunchResponseDto {
        LaunchUrl = AppendQueryString(runtime.BaseUrl, new Dictionary<string, string?> {
            ["folder"] = launchDirectory,
            ["tkn"] = runtime.ConnectionToken,
            ["vscode-lang"] = language,
        }),
        ConnectionToken = runtime.ConnectionToken,
        OpenMode = "folder",
        Runtime = VsCodeServerSettingsService.MapRuntime(
            await _vsCodeServerManager.GetRuntimeSnapshotAsync(cancellationToken)),
    };
}
```

Această metodă are o responsabilitate foarte clară: verifică setările, rezolvă căile, pornește serviciul și construiește URL-ul. Printre acestea, metoda `ResolveLaunchDirectory` efectuează verificări de securitate a căii pentru a preveni atacurile de tip path traversal. Codul poate părea puțin ca poezie atunci când fiecare linie are un scop.

### Gestionarea automată a runtime-ului

Backend-ul gestionează procesul code-server prin `VsCodeServerManager`:

- Verifică starea procesului
- Pornește automat serviciile oprite
- Returnează snapshot-uri de runtime precum port, ID proces și timp de pornire

Acest design permite sistemului să gestioneze automat ciclul de viață code-server, astfel încât utilizatorii nu trebuie să gestioneze manual procesele de serviciu. Viața este deja suficient de complicată; orice poate fi automatizat ar trebui să fie.

### Sincronizarea limbii

HagiCode suportă o interfață multilingvă, iar code-server trebuie să urmeze acea setare. Sistemul suportă trei moduri de limbă:

- `follow`: urmează limba curentă a interfeței
- `zh-CN`: fixat pe chineză
- `en-US`: fixat pe engleză

Setarea este transmisă către code-server prin parametrul URL `vscode-lang` astfel încât limba editorului să rămână consistentă cu interfața HagiCode. Limba se simte cel mai bine atunci când este unificată.

### Workspace MonoSpecs pentru multiple repository-uri

Pentru proiectele MonoSpecs, care conțin multiple sub-repository-uri într-un monorepo, sistemul creează automat un fișier `.code-workspace`:

```csharp
private async Task<string> CreateWorkspaceFileAsync(Project project, Guid projectId)
{
    var folders = await ResolveWorkspaceFoldersAsync(project.Path);
    var workspaceDocument = new {
        folders = folders.Select(path => new { path }).ToArray(),
    };
    // Generează fișierul workspace...
}
```

Acest lucru face posibilă editarea mai multor sub-repository-uri în aceeași instanță code-server, ceea ce este deosebit de practic pentru proiecte monorepo mari. Multiple repository-uri într-o fereastră pot părea ca multiple povești adunate în aceeași carte.

## Integrarea frontend-ului

Frontend-ul HagiCode folosește React + TypeScript, iar integrarea code-server nu este deosebit de complicată.

### Buton de acțiune rapidă

Adăugați un buton Code Server pe cardul proiectului:

```tsx
// QuickActionsZone.tsx
<Button
  size="sm"
  variant="default"
  onClick={() => onAction({ type: 'open-code-server' })}
>
  <Globe className="h-3 w-3 mr-1" />
  <span className="text-xs">{t('project.openCodeServer')}</span>
</Button>
```

Acest buton declanșează acțiunea de deschidere și apelează API-ul backend-ului pentru a obține URL-ul de lansare. Un buton, o acțiune, direct și simplu.

### Gestionarea acțiunii de deschidere

```tsx
const handleAction = async (action: ProjectAction) => {
  if (action.type === 'open-code-server') {
    const response = await openProjectInCodeServer(project.id, i18n.language);
    window.open(response.launchUrl, '_blank', 'noopener,noreferrer');
  }
};
```

Folosiți `window.open` pentru a deschide code-server într-un tab nou. Parametrii `noopener,noreferrer` oferă securitate suplimentară. Când vine vorba de securitate, nu există așa ceva ca „prea atent".

### Punct de intrare pentru editarea Vault-ului

Adăugați un buton de editare similar în lista de Vault-uri:

```tsx
const handleEditVault = async (vault: VaultItemDto) => {
  const response = await openVaultInCodeServer(vault.id);
  window.open(response.launchUrl, '_blank', 'noopener,noreferrer');
};
```

Proiectele și Vault-urile folosesc același mecanism de deschidere, ceea ce menține interacțiunea consistentă. Consistența contează aproape la fel de mult ca funcționalitatea în sine.

## Logica de construcție a URL-ului

Formatul URL-ului pentru code-server are câteva detalii demne de reținut.

**Modul folder:**
```
http://{host}:{port}/?folder={path}&tkn={token}&vscode-lang={lang}
```

**Modul workspace:**
```
http://{host}:{port}/?workspace={workspacePath}&tkn={token}&vscode-lang={lang}
```

Aici, `tkn` este token-ul de conexiune. Este generat automat de fiecare dată când code-server pornește, asigurând acces securizat. Parametrul `vscode-lang` controlează limba interfeței editorului. Fiecare dintre acești parametri are un rol de jucat.

## Scenarii de utilizare

### Scenariul 1: Revizuire a codului asistată de AI

Utilizatorul discută cu HagiCode, AI-ul analizează codul proiectului și găsește o problemă potențială, apoi utilizatorul face clic pe butonul „Deschide în Code Server" pentru a deschide editorul direct în browser, inspectează fișierul afectat, îl corectează și revine la HagiCode pentru a continua conversația. Tot fluxul are loc în browser fără a schimba aplicațiile. Se simte fluid în felul în care apa curgătoare se simte fluidă.

### Scenariul 2: Editarea materialelor de studiu într-un Vault

Un utilizator creează un Vault pentru a studia un proiect open source și dorește să adauge notițe de studiu sub directorul `docs/`. Cu code-server, poate edita fișiere Markdown direct în browser, le poate salva și lăsa HagiCode să citească imediat notițele actualizate. Acest lucru este deosebit de util pentru construirea unei baze de cunoștințe personale. Cunoștințele devin doar mai valoroase cu cât le acumulați mai mult.

### Scenariul 3: Dezvoltare MonoSpecs cu multiple repository-uri

Un proiect MonoSpecs conține multiple sub-repository-uri, iar code-server creează automat un workspace cu multiple foldere. În browser, utilizatorii pot edita cod din mai multe repository-uri simultan și apoi pot trimite modificările înapoi la repository-urile Git respective. Acest flux de lucru este deosebit de potrivit pentru modificări care trebuie să acopere multiple repository-uri. Editarea mai multor repository-uri împreună necesită puțină tehnică, la fel ca gestionarea mai multor sarcini în același timp.

## Considerații de securitate

Când implementați integrarea code-server, securitatea merită o atenție specială. Dacă securitatea merge prost, observați întotdeauna prea târziu.

### Token-ul de conexiune

`connection-token` este generat aleatoriu și nu ar trebui expus. Este cel mai bine folosit sub HTTPS pentru a preveni interceptarea token-ului de către un atac man-in-the-middle. Informațiile sensibile merită protejate corespunzător.

### Securitatea căii fișierelor

Backend-ul implementează verificări de tip path traversal:

```csharp
private static string ResolveLaunchDirectory(VaultRegistryEntry vault, string? relativePath)
{
    var vaultRoot = EnsureTrailingSeparator(Path.GetFullPath(vault.PhysicalPath));
    var combinedPath = Path.GetFullPath(Path.Combine(vaultRoot, relativePath ?? "."));
    if (!combinedPath.StartsWith(vaultRoot, StringComparison.OrdinalIgnoreCase))
    {
        throw new BusinessException(VaultRelativePathTraversalCode, "Relative path traversal detected.");
    }
    return combinedPath;
}
```

Acest cod se asigură că utilizatorii nu pot folosi `../` sau modele similare pentru a accesa fișiere în afara directorului Vault. Verificările de limite sunt întotdeauna mai bine făcute decât sărate.

### Controlul permisiunilor

Procesul code-server ar trebui să ruleze cu permisiuni de utilizator adecvate astfel încât să nu poată accesa fișiere de sistem sensibile. Este recomandat să rulați serviciul code-server sub un utilizator dedicat. Controlul permisiunilor este unul dintre acele fundamentale pe care ar trebui să le mențineți întotdeauna.

## Optimizarea performanței

code-server consumă resurse de server, deci iată câteva sugestii de optimizare:

- Monitorizați utilizarea CPU și memorie și ajustați limitele de resurse când este necesar
- Proiectele mari pot necesita timeout-uri mai lungi
- Implementați curățarea automată a sesiunilor expirate pentru a elibera resurse
- Luați în considerare utilizarea cache-ului pentru a reduce calculul repetat

HagiCode oferă un API de monitorizare a stării runtime-ului, iar frontend-ul poate apela `getVsCodeServerSettings()` pentru a obține starea curentă:

```typescript
const { settings, runtime } = await getVsCodeServerSettings();
// runtime.status: 'disabled' | 'stopped' | 'starting' | 'running' | 'unhealthy'
// runtime.baseUrl: "http://localhost:8080"
// runtime.processId: 12345
```

Acest design permite utilizatorilor să înțeleagă clar starea de sănătate a code-server-ului și să localizeze rapid problemele când ceva merge prost. Când starea este vizibilă, oamenii se simt mai în control.

## Detalii ale experienței utilizatorului

În timpul implementării, am descoperit câteva detalii care afectează vizibil experiența utilizatorului și merită atenție suplimentară.

Deschiderea code-server pentru prima dată poate necesita așteptarea pornirii, iar acea întârziere poate varia de la câteva secunde la jumătate de minut. Este o idee bună să afișați o stare de încărcare în frontend astfel încât utilizatorii să știe că sistemul încă funcționează. Așteptarea este mai ușoară când există feedback.

Browserul poate bloca popup-ul, deci utilizatorii ar trebui solicitați să permită manual. La prima lansare, HagiCode afișează ghidare care explică cum să acordați permisiunile necesare ale browserului. Experiența utilizatorului locuiește adesea exact în aceste mici detalii.

De asemenea, este o idee bună să afișați starea runtime-ului cum ar fi pornire, funcționare sau eroare, astfel încât atunci când apar probleme, utilizatorii să poată spune rapid dacă problema este pe partea serverului sau în propria operațiune. A ști unde este problema vă oferă cel puțin un punct de plecare.

## Exemplu de configurare

Configurarea pentru code-server nu este complicată:

```json
{
  "vscodeServer": {
    "enabled": true,
    "host": "0.0.0.0",
    "port": 8080,
    "language": "follow"
  }
}
```

`enabled` controlează dacă funcția este activată, `host` și `port` definesc adresa de ascultare, iar `language` setează modul de limbă. Aceste setări pot fi modificate prin UI și intră în vigoare imediat. Lucrurile simple sunt adesea cele mai ușor de folosit.

## Concluzie

Integrarea VSCode Web a HagiCode oferă o soluție elegantă: permite asistentului AI și experienței de editare a codului să se conecteze fără întreruperi. Prin integrarea code-server în browser, utilizatorii pot acționa rapid pe rezultatele analizei AI și pot completa fluxul complet de la analiză la editare în aceeași sesiune de browser.

Această soluție aduce câteva avantaje cheie: o experiență unificată, deoarece proiectele și Vault-urile folosesc același mecanism de deschidere; suport pentru multiple repository-uri, deoarece proiectele MonoSpecs creează automat workspace-uri; și securitate controlabilă, datorită monitorizării stării runtime-ului și verificărilor de siguranță a căilor.

Dacă găsiți această soluție valoroasă, asta sugerează că practica noastră de inginerie face ceva drept, iar HagiCode în sine poate merita o privire mai atentă.
