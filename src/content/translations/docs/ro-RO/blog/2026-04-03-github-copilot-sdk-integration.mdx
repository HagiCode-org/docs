## De la apeluri CLI la integrare SDK: Cele mai bune practici pentru GitHub Copilot în proiecte .NET

> Drumul de actualizare de la invocarea prin linie de comandă la integrarea oficială SDK a fost o călătorie destul de lungă. Astăzi vreau să împărtășesc capcanele în care am căzut și ceea ce am învățat în timp ce construia HagiCode.

## Context

După ce GitHub Copilot SDK a fost lansat oficial în 2025, am început să îl integrăm în stratul nostru de capabilități AI. Înainte de aceasta, proiectul folosea în principal GitHub Copilot invocând direct instrumentul de linie de comandă Copilot CLI, dar acea abordare avea mai multe probleme evidente:

- **Management complex al proceselor**: Trebuia să gestionăm manual ciclul de viață al procesului CLI, timeout-urile de pornire și curățarea proceselor. Procesele se pot prăbuși fără avertisment.
- **Gestionare incompletă a evenimentelor**: Invocarea brută a CLI face dificilă capturarea evenimentelor granulare din raționamentul modelului și execuția instrumentelor. Este ca și cum ai vedea doar rezultatul fără procesul de gândire.
- **Management dificil al sesiunilor**: Nu exista un mecanism eficient pentru refolosirea și recuperarea sesiunilor, deci fiecare interacțiune trebuia să reînceapă de la zero.
- **Probleme de compatibilitate**: Argumentele CLI se schimbau frecvent, ceea ce însemna că trebuia să menținem constant logica de compatibilitate pentru acei parametri.

Aceste probleme au devenit din ce în ce mai evidente în dezvoltarea de zi cu zi, mai ales când trebuia să urmărim raționamentul modelului (`thinking`) și starea execuției instrumentelor în timp real. În acel moment, a devenit clar că aveam nevoie de o abordare de integrare mai de bază și mai completă.

Aceste cerințe din lumea reală ne-au determinat să trecem de la invocarea CLI la SDK-ul oficial.

## Designul arhitecturii

Proiectul folosește o arhitectură pe straturi pentru a aborda limitările invocării CLI:

```
┌─────────────────────────────────────────────────────────┐
│  hagicode-core (Orleans Grains + AI Provider Layer)    │
│  - CopilotAIProvider: Converts AIRequest to CopilotOptions │
│  - GitHubCopilotGrain: Orleans distributed execution interface │
└─────────────────────────────────────────────────────────┘
                          ↓
┌─────────────────────────────────────────────────────────┐
│  HagiCode.Libs (Shared Provider Layer)                 │
│  - CopilotProvider: CLI Provider interface implementation │
│  - ICopilotSdkGateway: SDK invocation abstraction      │
│  - GitHubCopilotSdkGateway: SDK session management and event dispatch │
└─────────────────────────────────────────────────────────┘
                          ↓
┌─────────────────────────────────────────────────────────┐
│  GitHub Copilot SDK (Official .NET SDK)                │
│  - CopilotClient: SDK client                           │
│  - CopilotSession: Session management                  │
│  - SessionEvent: Event stream                          │
└─────────────────────────────────────────────────────────┘
```

Acest design pe straturi aduce mai multe avantaje tehnice practice:

1. **Separarea preocupărilor**: Logica de business de bază este decuplată de detaliile de implementare SDK.
2. **Testabilitate**: Interfața `ICopilotSdkGateway` face testarea unitară straightforward.
3. **Reutilizabilitate**: `HagiCode.Libs` poate fi referențiat de multiple proiecte.
4. **Mentenabilitate**: Actualizările SDK necesită modificări doar în stratul Gateway, în timp ce straturile superioare rămân neatinger.

## Implementarea de bază

### Fluxul de autentificare

Autentificarea este primul și cel mai important pas al integrării SDK. Dacă autentificarea eșuează, nimic altceva nu contează. Am proiectat o configurație de autentificare flexibilă care suportă multiple surse de autentificare:

```csharp
// CopilotProvider.cs - Configurarea sursei de autentificare
public class CopilotOptions
{
    public bool UseLoggedInUser { get; set; } = true;
    public string? GitHubToken { get; set; }
    public string? CliUrl { get; set; }
}

// Convertire în cerere SDK
return new CopilotSdkRequest(
    GitHubToken: options.AuthSource == CopilotAuthSource.GitHubToken
        ? options.GitHubToken
        : null,
    UseLoggedInUser: options.AuthSource != CopilotAuthSource.GitHubToken
);
```

Beneficiile acestui design sunt destul de clare:

- Suportă modul utilizatorului conectat fără a necesita un token, ceea ce se potrivește bine scenariilor desktop.
- Suportă modul GitHub Token, care este potrivit pentru implementări pe server și gestionare centralizată.
- Suportă suprascrierea URL-ului Copilot CLI, ceea ce ajută cu configurarea proxy pentru întreprinderi.

În practică, acest model flexibil de autentificare a simplificat semnificativ configurarea în diferite scenarii de implementare. Clientul desktop poate folosi starea proprie de conectare Copilot a fiecărui utilizator, în timp ce partea de server poate gestiona autentificarea centralizat prin token-uri.

### Gestionarea fluxului de evenimente

Una dintre cele mai puternice capabilități ale SDK-ului este capturarea completă a fluxului de evenimente. Am implementat un sistem de expediere a evenimentelor care poate gestiona tot felul de evenimente SDK în timp real:

```csharp
// GitHubCopilotSdkGateway.cs - Logica de bază pentru expedierea evenimentelor
internal static SessionEventDispatchResult DispatchSessionEvent(
    SessionEvent evt, bool sawDelta)
{
    switch (evt)
    {
        case AssistantReasoningEvent reasoningEvent:
            // Capturați procesul de raționament al modelului
            events.Add(new CopilotSdkStreamEvent(
                CopilotSdkStreamEventType.ReasoningDelta,
                Content: reasoningEvent.Data.Content));
            break;

        case ToolExecutionStartEvent toolStartEvent:
            // Capturați începerea unei invocări de instrument
            events.Add(new CopilotSdkStreamEvent(
                CopilotSdkStreamEventType.ToolExecutionStart,
                ToolName: toolStartEvent.Data.ToolName,
                ToolCallId: toolStartEvent.Data.ToolCallId));
            break;

        case ToolExecutionCompleteEvent toolCompleteEvent:
            // Capturați finalizarea instrumentului și rezultatul său
            events.Add(new CopilotSdkStreamEvent(
                CopilotSdkStreamEventType.ToolExecutionEnd,
                Content: ExtractToolExecutionContent(toolCompleteEvent)));
            break;

        default:
            // Păstrați evenimentele negestionate ca RawEvent
            events.Add(new CopilotSdkStreamEvent(
                CopilotSdkStreamEventType.RawEvent,
                RawEventType: evt.GetType().Name));
            break;
    }
}
```

Valoarea acestei implementări este semnificativă:

- **Capturarea completă a procesului de raționament al modelului** (`thinking`): utilizatorii pot vedea cum raționează AI-ul, nu doar răspunsul final.
- **Urmărirea în timp real a stării execuției instrumentelor**: știm ce instrumente rulează, când se termină și ce returnează.
- **Pierdere zero de evenimente**: prin revenirea la `RawEvent`, asigurăm că fiecare eveniment este înregistrat.

În HagiCode, aceste evenimente granulare ajută utilizatorii să înțeleagă cum funcționează AI-ul intern, mai ales la depanarea sarcinilor complexe.

### Gestionarea compatibilității CLI

După migrarea de la invocarea CLI la SDK, am descoperit că unii parametri CLI existenți nu mai erau aplicați în SDK. Pentru a păstra compatibilitatea inversă, am implementat un sistem de filtrare a parametrilor:

```csharp
// CopilotCliCompatibility.cs - Filtrarea argumentelor
private static readonly Dictionary<string, string> RejectedFlags = new()
{
    ["--headless"] = "Unsupported startup argument",
    ["--model"] = "Passed through an SDK-native field",
    ["--prompt"] = "Passed through an SDK-native field",
    ["--interactive"] = "Interaction is managed by the provider",
};

public static CopilotCliArgumentBuildResult BuildCliArgs(CopilotOptions options)
{
    // Filtrați argumentele nesuportate și păstrați-le pe cele compatibile
    // Generați informații de diagnosticare
}
```

Aceasta ne oferă mai multe beneficii:

- Filtrează automat argumentele CLI incompatibile pentru a preveni erorile la execuție.
- Generează mesaje de diagnosticare clare pentru a ajuta dezvoltatorii să localizeze problemele rapid.
- Menține SDK-ul stabil și izolat de schimbările parametrilor CLI.

În timpul actualizării, acest mecanism de compatibilitate ne-a ajutat să tranziționăm fără probleme. Fișierele de configurare existente puteau fi încă folosite și necesitau doar ajustări graduale pe baza ieșirii de diagnosticare.

### Pooling la execuție

Crearea sesiunilor Copilot SDK este relativ costisitoare, iar crearea și distrugerea sesiunilor prea frecvent afectează performanța. Pentru a rezolva asta, am implementat un sistem de gestionare a pool-ului de sesiuni:

```csharp
// CopilotProvider.cs - Managementul pool-ului de sesiuni
await using var lease = await _poolCoordinator.AcquireCopilotRuntimeAsync(
    request,
    async ct => await _gateway.CreateRuntimeAsync(sdkRequest, ct),
    cancellationToken);

if (lease.IsWarmLease)
{
    // Reutilizați o sesiune existentă
    yield return CreateSessionReusedMessage();
}

await foreach (var eventData in lease.Entry.Resource.SendPromptAsync(...))
{
    yield return MapEvent(eventData);
}
```

Beneficiile pooling-ului de sesiuni includ:

- **Reutilizarea sesiunilor**: cererile cu același `sessionId` pot refolosi sesiunile existente și reduc overhead-ul de pornire.
- **Suport pentru recuperarea sesiunilor**: după o întrerupere de rețea, starea sesiunii anterioare poate fi restaurată.
- **Management automat al pool-ului**: sesiunile expirate sunt curățate automat pentru a evita scurgerile de resurse.

În HagiCode, pooling-ul de sesiuni a îmbunătățit vizibil timpul de răspuns, mai ales pentru conversațiile continue.

### Integrarea Orleans

HagiCode folosește Orleans ca framework distribuit, iar am integrat Copilot SDK în Orleans Grains:

```csharp
// GitHubCopilotGrain.cs - Execuție distribuită
public async IAsyncEnumerable<GitHubCopilotResponse> ExecuteCommandStreamAsync(
    string command,
    CancellationToken token = default)
{
    var provider = await aiProviderFactory.GetProviderAsync(AIProviderType.GitHubCopilot);

    await foreach (var chunk in provider.SendMessageAsync(request, null, token))
    {
        // Mapare la formatul de răspuns unificat
        yield return BuildChunkResponse(chunk, startedAt);
    }
}
```

Avantajele integrării Orleans sunt substanțiale:

- **Abstractizare unificată AI Provider**: devine ușor să comutați între diferiți furnizori AI.
- **Izolare multi-tenant**: sesiunile Copilot pentru utilizatori diferiți rămân izolate între ele.
- **Stare persistentă a sesiunii**: starea sesiunii poate fi restaurată chiar și după repornirea serverului.

Pentru scenariile care necesită gestionarea unui număr mare de cereri concurente, Orleans oferă scalabilitate puternică.

## Ghid practic

### Exemplu de configurare

Iată un exemplu complet de configurare:

```json
{
  "AI": {
    "Providers": {
      "Providers": {
        "GitHubCopilot": {
          "Enabled": true,
          "ExecutablePath": "copilot",
          "Model": "gpt-5",
          "WorkingDirectory": "/path/to/project",
          "Timeout": 7200,
          "StartupTimeout": 30,
          "UseLoggedInUser": true,
          "NoAskUser": true,
          "Permissions": {
            "AllowAllTools": false,
            "AllowedTools": ["Read", "Bash", "Grep"],
            "DeniedTools": ["Edit"]
          }
        }
      }
    }
  }
}
```

### Note de utilizare

În utilizarea din lumea reală, am rezumat câteva puncte care merită atenție:

**Configurarea timeout-ului de pornire**: Prima pornire a Copilot CLI poate dura relativ mult, deci recomandăm setarea `StartupTimeout` la cel puțin 30 de secunde. Dacă este prima conectare, poate dura chiar și mai mult.

**Managementul permisiunilor**: În mediile de producție, evitați folosirea `AllowAllTools: true`. Folosiți lista de permisiuni `AllowedTools` pentru a controla ce instrumente sunt disponibile și folosiți lista de interzicere `DeniedTools` pentru a bloca operațiile periculoase. Aceasta previne eficient AI-ul de la a executa comenzi riscante.

**Managementul sesiunilor**: Cererile cu același `sessionId` refolosesc automat sesiunile. Starea sesiunii este persistentă prin `ProviderSessionId`. Anularea este propagată prin `CancellationTokenSource`.

**Ieșirea de diagnosticare**: Argumentele CLI incompatibile generează mesaje de tip `diagnostic`. Evenimentele SDK brute sunt păstrate ca `event.raw`. Mesajele de eroare includ categorii precum timeout de pornire și incompatibilitate de argumente pentru a facilita depanarea.

### Cele mai bune practici

Pe baza experienței noastre practice, iată câteva practici recomandate:

**1. Folosiți o listă de permisiuni pentru instrumente**

```csharp
var request = new AIRequest
{
    Prompt = "Analyze this file",
    AllowedTools = new[] { "Read", "Grep", "Bash(git:*)" }
};
```

Specificarea explicită a instrumentelor permise printr-o listă de permisiuni ajută la prevenirea acțiunilor AI neașteptate. Aceasta este deosebit de importantă pentru instrumentele cu permisiuni de scriere, cum ar fi `Edit`, care ar trebui tratate cu grijă suplimentară.

**2. Setați timeout-uri rezonabile**

```csharp
options.Timeout = 3600;  // 1 oră
options.StartupTimeout = 60;  // 1 minut
```

Setați valori de timeout adecvate pe baza complexității sarcinii. Dacă valoarea este prea scurtă, sarcinile pot fi întrerupte. Dacă este prea lungă, resursele pot fi irosite așteptând cereri care nu răspund.

**3. Activați reutilizarea sesiunilor**

```csharp
options.SessionId = "my-session-123";
```

Folosirea aceluiași `sessionId` pentru sarcinile legate vă permite să refolosiți contextul sesiunii anterioare și să îmbunătățiți viteza de răspuns.

**4. Gestionarea răspunsurilor de streaming**

```csharp
await foreach (var chunk in provider.StreamAsync(request))
{
    switch (chunk.Type)
    {
        case StreamingChunkType.ThinkingDelta:
            // Gestionare proces de raționament
            break;
        case StreamingChunkType.ToolCallDelta:
            // Gestionare invocare instrument
            break;
        case StreamingChunkType.ContentDelta:
            // Gestionare ieșire text
            break;
    }
}
```

Răspunsurile de streaming vă permit să afișați progresul procesării AI în timp real, ceea ce îmbunătățește experiența utilizatorului. Aceasta este deosebit de valoroasă pentru sarcinile de lungă durată.

**5. Gestionarea erorilor și reîncercările**

```csharp
try
{
    await foreach (var chunk in provider.StreamAsync(request))
    {
        // Gestionare răspuns
    }
}
catch (CopilotSessionException ex)
{
    // Gestionare excepții de sesiune
    logger.LogError(ex, "Copilot session failed");
    // Decideți dacă să reîncercați pe baza tipului de excepție
}
```

Gestionarea corectă a erorilor și logica de reîncercare îmbunătățesc stabilitatea generală a sistemului.

## Concluzie

Actualizarea de la invocarea CLI la integrarea SDK a adus o valoare substanțială proiectului HagiCode:

- **Stabilitate îmbunătățită**: SDK-ul oferă o interfață mai stabilă care nu este afectată de schimbările de versiune CLI.
- **Funcționalitate mai completă**: capturează fluxul complet de evenimente, inclusiv raționamentul și starea execuției instrumentelor.
- **Eficiență de dezvoltare mai mare**: interfața SDK type-safe face dezvoltarea mai eficientă și reduce erorile la execuție.
- **Experiență utilizator mai bună**: feedback-ul în timp real al evenimentelor oferă utilizatorilor o înțelegere mai clară a ceea ce face AI-ul.

Această actualizare nu a fost doar o înlocuire de implementare tehnică. A fost și o optimizare arhitecturală a întregului strat de capabilități AI. Prin design pe straturi și interfețe de abstractizare, am câștigat o mai bună mentenabilitate și extensibilitate.

Dacă luați în considerare integrarea GitHub Copilot în propriul proiect .NET, sper că aceste lecții practice vă ajută să evitați ocoluri inutile. SDK-ul oficial este într-adevăr mai stabil și mai complet decât invocarea CLI, iar merită timpul pentru a-l înțelege și adopta.
