## De la Docusaurus 3.x la Astro 5.x: O retrospectivă asupra migrării site-ului HagiCode

> Acest articol revizuiește migrarea noastră completă a site-ului oficial HagiCode de la Docusaurus 3.x la Astro 5.x. Vom analiza aprofundat modul în care arhitectura Islands a Astro ne-a ajutat să rezolvăm blocajele de performanță păstrând în același timp activele componente React existente, livrând îmbunătățiri atât în viteza de construcție cât și în performanța de încărcare.

## Context

În ianuarie 2026, am efectuat un „transplant de inimă" pe site-ul oficial HagiCode migrând complet framework-ul său de bază de la Docusaurus 3.x la Astro 5.x. Aceasta nu a fost o rescriere impulsivă, ci o decizie tehnică atent calculată.

Înainte de migrare, site-ul nostru era funcțional complet, dar începuse să arate unele clasice „probleme de lux": artifacte de construcție umflate, payload-uri JavaScript excesive și viteză de încărcare a paginilor sub ideală pe paginile de documentație complexe. Ca proiect de asistent de programare AI, HagiCode are nevoie de actualizări frecvente de documentație și funcționalități, deci eficiența construcției afectează direct viteza de lansare. În același timp, doream ca site-ul să fie mai prietenos cu motoarele de căutare (SEO) pentru ca mai mulți dezvoltatori să poată descoperi proiectul.

Pentru a rezolva aceste puncte dureroase, am luat o decizie curajoasă: reconstruirea întregului sistem pe Astro. Impactul acelei decizii poate fi chiar mai mare decât vă așteptați. Voi intra în detalii în curând.

## Despre HagiCode

Abordarea de migrare a site-ului partajată în acest articol provine din experiența noastră practică în proiectul [HagiCode](https://github.com/HagiCode-org/site).

HagiCode este un asistent de programare AI axat pe îmbunătățirea eficienței dezvoltării. Ne pasă nu doar de iterarea funcționalităților de bază, ci și de experiența dezvoltatorului. Această refactorizare a site-ului a fost de asemenea menită să ofere utilizatorilor cea mai rapidă experiență posibilă la navigarea documentației noastre și a site-ului oficial.

## De ce să părăsim ecosistemul matur Docusaurus?

În ecosistemul React, Docusaurus a fost mult timp „răspunsul standard" pentru site-urile de documentație. Funcționează din cutie, oferă un ecosistem bogat de plugin-uri și are o comunitate activă. Dar pe măsură ce HagiCode a căpătat mai multe funcționalități, am simțit și limitările sale:

1.  **Blocaje de performanță**: Docusaurus este fundamental o aplicație React SPA (single-page application). Chiar dacă scrieți doar pagini statice, clientul trebuie totuși să încarce runtime-ul React și să hidrateze pagina, ceea ce este inutil de greu pentru pagini simple de documentație.
2.  **Dimensiune mare a activelor**: Chiar și când o pagină conține foarte puțin conținut, dimensiunea JS-ului grupat rămâne relativ fixă. Asta nu este ideal pentru utilizatorii mobili sau conexiuni slabe.
3.  **Flexibilitate limitată**: Deși este extensibil, doream mai mult control la nivel inferior asupra pipeline-ului de construcție.

Astro a sosit exact la momentul potrivit pentru a rezolva aceste probleme. A introdus o nouă „Arhitectură Islands": implicit, Astro generează HTML static cu zero JavaScript, și doar componentele care necesită interactivitate sunt „activate" și încarcă JS. Asta înseamnă că cea mai mare parte a site-ului nostru devine HTML pur și se încarcă extrem de rapid.

## Strategia de migrare de bază: O tranziție arhitecturală lină

Migrarea nu a fost doar copiere și lipire. A necesitat o schimbare de mentalitate. Ne-am mutat de la modelul „tot React" al Docusaurus la modelul „Nucleu + Islands" al Astro.

### 1. Reconstruirea sistemului de configurare

Mai întâi, a trebuit să ne mutăm de la `docusaurus.config.ts` la `astro.config.mjs`. Aceasta nu a fost doar o redenumire de fișier, ci o rescriere a rutării și logicii de construcție.

În Docusaurus, totul este un plugin. În Astro, totul este o integrare. A trebuit să redefinim calea de bază a site-ului, modul de ieșire al construcției (static vs SSR) și strategia de optimizare a activelor.

**Înainte de migrare:**
```typescript
// docusaurus.config.ts
export default {
  title: 'HagiCode',
  url: 'https://hagicode.com',
  baseUrl: '/',
  // ... more configuration
};
```

**După migrare:**
```javascript
// astro.config.mjs
import { defineConfig } from 'astro/config';
import react from '@astrojs/react';

export default defineConfig({
  integrations: [react()],
  site: 'https://hagicode.com',
  base: '/',
  // Optimization settings for static assets
  build: {
    inlineStylesheets: 'auto',
  },
});
```

### 2. Ce să păstrăm și ce să refactorizăm din componentele React

Aceasta a fost cea mai dureroasă parte a migrării. Site-ul nostru existent avea multe componente React, cum ar fi Tabs, evidențierea codului, butoanele de feedback și multe altele. Aruncarea lor ar fi fost irositoare, dar păstrarea tuturor ar fi făcut payload-ul JavaScript prea greu.

HagiCode a adoptat o strategie de **hidratare progresivă**:
*   **Componente pur statice**: Pentru conținut de prezentare precum headere, footer-uri și documentație text simplu, le-am rescris ca componente Astro (fișiere `.astro`) și le-am randat direct la HTML în timpul construcției.
*   **Insule interactive**: Pentru componentele care trebuie să rămână interactive, cum ar fi comutatoarele de temă, comutarea tab-urilor și butoanele de copiere a blocurilor de cod, am păstrat implementarea React și am adăugat directivele `client:load` sau `client:visible`.

De exemplu, componenta noastră frecvent utilizată Tabs în documentație:

```jsx
// src/components/Tabs.jsx
import { useState } from 'react';
import './Tabs.css'; // Import styles

export default function Tabs({ items }) {
  const [activeIndex, setActiveIndex] = useState(0);
  // ... state logic
  return (
    <div className="tabs-wrapper">
      {/* Rendering logic */}
    </div>
  );
}
```

Când este utilizată în Markdown, îi spunem explicit Astro: „Această componentă are nevoie de JS."

```markdown
// src/content/docs/example.mdx
import Tabs from '../../components/Tabs.jsx';

<!-- Load JS only when the component enters the viewport -->
<Tabs client:visible items={...} />
```

În acest fel, componentele interactive în afara viewport-ului nu concurează pentru lățimea de bandă, ceea ce îmbunătățește considerabil viteza de încărcare a primei pagini.

### 3. Adaptarea sistemului de stilizare: De la CSS Modules la CSS cu domeniu

Docusaurus suportă CSS Modules implicit, în timp ce Astro încurajează CSS cu domeniu prin tag-ul `<style>`. Ideea de bază din spatele ambelor este izolarea stilurilor, dar sintaxa este diferită.

În timpul migrării HagiCode, am convertit majoritatea CSS Modules complexe în stiluri cu domeniu Astro. Asta s-a dovedit a fi de fapt un lucru bun, deoarece în fișierele `.astro` stilurile și șabloanele trăiesc în același fișier, ceea ce face mentenanța mai intuitivă.

**Înainte de refactorizare:**
```css
/* Tabs.module.css */
.wrapper { background: var(--ifm-background-color); }
```

**După refactorizare (Astro Scoped):**
```html
<!-- Tabs.astro -->
<div class="tabs-wrapper">
  <slot />
</div>

<style>
  .tabs-wrapper {
    /* Use CSS variables directly to adapt to the theme */
    background: var(--bg-color);
    padding: 1rem;
  }
</style>
```

În același timp, am unificat sistemul de variabile CSS globale și am folosit capabilitățile Astro de conștientizare a mediului pentru a asigura că modul întunecat comută fluid între pagini.

## Capcane pe care le-am întâmpinat în practică și cum le-am rezolvat

În timpul migrării reale HagiCode, am întâmpinat destul de multe probleme. Iată câteva dintre cele mai tipice.

### 1. Probleme de căi și variabile de mediu

HagiCode suportă implementare pe subpath, cum ar fi implementare sub un subdirector GitHub Pages. În Docusaurus, `baseUrl` este gestionat automat. În Astro, totuși, trebuie să fim mai atenți la gestionarea link-urilor de imagini și cererilor API.

Am introdus un mecanism de variabile de mediu pentru a gestiona acest lucru în mod consistent:

```javascript
// Handle paths in the build script
const getBasePath = () => import.meta.env.VITE_SITE_BASE || '/';
```

Asigurați-vă că nu hardcodați căi care încep cu `/` în codul dvs. În dezvoltare versus producție, sau după configurarea unei căi `base`, acest lucru poate cauza erori 404 pentru active.

### 2. Compatibilitate scripturi CommonJS

Vechiul nostru site avea câteva scripturi Node.js folosite pentru sarcini precum obținerea automată a datelor Metrics și actualizarea sitemap-ului, și erau scrise în CommonJS (`require`). Astro și instrumentele de construcție moderne au adoptat complet Modulele ES (`import`/`export`).

Dacă aveți și scripturi similare, nu uitați să le refactorizați pe toate la Module ES. Aceasta este direcția în care se îndreaptă ecosistemul, iar cu cât faceți schimbarea mai devreme, cu atât veți avea mai puține probleme mai târziu.

```javascript
// Old way
const fs = require('fs');

// New way
import fs from 'fs';
```

### 3. Nu uitați de SEO și redirecționări

Motoarele de căutare au deja indexat paginile vechi Docusaurus ale HagiCode. Dacă treceți direct la Astro și structura URL se schimbă, puteți ajunge cu un număr mare de erori 404 și o scădere majoră a clasamentului de căutare.

Am configurat reguli de redirecționare în Astro:

```javascript
// astro.config.mjs
export default defineConfig({
  redirects: {
    '/docs/old-path': '/docs/new-path',
    // Map old links to new links in bulk
  }
});
```

Sau puteți gestiona acest lucru la nivelul configurației serverului. Asigurați-vă că link-urile vechi pot fi redirecționate 301 la noile adrese, deoarece acest lucru este critic pentru SEO.

## Rezumat

Pentru HagiCode, migrarea de la Docusaurus la Astro nu a fost doar o actualizare de framework. A fost de asemenea o implementare practică a unei filosofii „performanță pe primul loc."

**Ce am câștigat:**
*   **Scoruri Lighthouse remarcabile**: După migrare, scorul de performanță al site-ului HagiCode s-a apropiat ușor de un scor perfect.
*   **Viteză de construcție mai rapidă**: Capabilitățile de construcție incrementală ale Astro au redus la jumătate timpul de lansare pentru actualizările de documentație.
*   **Flexibilitate păstrată**: Cu arhitectura Islands, nu am sacrificat nicio funcționalitate interactivă și am putut folosi în continuare React unde era necesar.

Dacă mențineți și un site orientat spre documentație și vă confruntați cu dimensiunea bundle-ului sau viteza de încărcare, Astro merită cu siguranță încercat. Deși procesul de migrare necesită într-adevăr o intervenție chirurgicală, cum ar fi redenumirea PCode în HagiCode și mutarea componentelor una câte una, experiența utilizatorului mătăsoasă pe care o obțineți în schimb face totul absolut meritor.

Sistemul de construcție partajat în acest articol este abordarea exactă pe care am dezvoltat-o prin încercări și erori reale în timp ce construim HagiCode. Dacă considerați că această abordare este valoroasă, asta spune ceva despre forța noastră de inginerie, iar HagiCode în sine merită probabil o privire mai atentă.
