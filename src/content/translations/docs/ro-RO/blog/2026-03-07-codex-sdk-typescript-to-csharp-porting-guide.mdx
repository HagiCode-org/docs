## De la TypeScript la C#: Practica de Portare Cross-Limbaj pentru SDK-ul Codex

> Simplu spus, acest articol este oarecum și un copil al nostru: înregistrează procesul complet de portare a SDK-ului oficial TypeScript Codex la C#. A-l numi "portare" aproape că îl face să sune prea ușor - a fost mai degrabă o aventură lungă, deoarece aceste două limbaje au personalități foarte diferite și a trebuit să găsim o modalitate de a le face să coopereze.

## Context

Codex este instrumentul CLI AI Agent lansat de OpenAI și este cu adevărat puternic. Echipa oficială oferă un SDK TypeScript în pachetul `@openai/codex`. Acesta interacționează cu CLI-ul Codex apelând comanda `codex exec --experimental-json` și analizând un flux de evenimente JSONL.

Problema este că în proiectul HagiCode, trebuie să îl folosim într-un mediu pur .NET - în mod specific în servicii backend C# și aplicații desktop. Nu puteam introduce rezonabil un runtime Node.js într-un proiect .NET doar pentru a apela un instrument CLI. Asta ar fi fost mult prea greoi.

Deci ne-au rămas două opțiuni: să menținem un strat de punte Node.js complex sau să construim noi înșine un SDK C# nativ.

Am ales-o pe a doua.

## Conținut de Bază

### Comparația Designului Arhitectural

Înainte de a traduce codul unu-la-unu, a trebuit mai întâi să înțelegem designul arhitectural al ambelor SDK-uri. Trebuie să înțelegi ambele părți înainte de a le putea portă bine.

Arhitectura de bază a SDK-ului TypeScript arată astfel:

```
Codex (entry class)
  └── CodexExec (executor, manages child processes)
      └── Thread (conversation thread)
          ├── run() / runStreamed() (synchronous/asynchronous execution)
          └── event stream parsing
```

SDK-ul C# păstrează aceeași stratificare arhitecturală, dar adaptează detaliile de implementare. Ideea generală este simplă: păstrează consistența API-ului valorificând complet caracteristicile limbajului C# în implementare.

### Conversia Sistemului de Tipuri

Aceasta este cea mai fundamentală și totodată cea mai importantă parte a muncii. Dacă fundația este slabă, tot ce urmează devine mai greu.

Sistemul de tipuri TypeScript este mai flexibil decât cel al C#, și acesta este pur și simplu un fapt. A trebuit să găsim o strategie de mapare adecvată:

| TypeScript | C# | Note |
|------------|-----|------|
| `interface` / `type` | `record` | C# folosește `record` pentru structuri de date imuabile |
| `string \| null` | `string?` | Tip referință nulabil |
| `boolean \| undefined` | `bool?` | Boolean nulabil |
| `AsyncGenerator` | `IAsyncEnumerable` | Iterator asincron |

**Sistemul de tipuri de evenimente** este un exemplu tipic. TypeScript folosește tipuri de uniune pentru a defini evenimentele:

```typescript
export type ThreadEvent =
  | ThreadStartedEvent
  | TurnStartedEvent
  | TurnCompletedEvent
  | ...
```

În C#, folosim o ierarhie de moștenire și potrivire de pattern-uri pentru a obține un efect similar:

```csharp
public abstract record ThreadEvent(string Type);

public sealed record ThreadStartedEvent(string ThreadId) : ThreadEvent("thread.started");
public sealed record TurnStartedEvent() : ThreadEvent("turn.started");
public sealed record TurnCompletedEvent(Usage Usage) : ThreadEvent("turn.completed");
// ...
```

Am ales `record` în loc de `class` deoarece obiectele de eveniment ar trebui să fie imuabile, ceea ce se potrivește cu intenția din spatele utilizării obiectelor simple în TypeScript. Cuvântul cheie `sealed` previne de asemenea moștenirea suplimentară și oferă compilatorului spațiu de optimizare.

### Puncte Cheie de Portare

#### 1. Analizatorul de evenimente

Analizarea evenimentelor este nucleul întregului SDK, deoarece determină dacă putem înțelege corect fiecare mesaj returnat de CLI-ul Codex. Dacă analizarea este greșită, tot ce urmează este efort irosit.

Versiunea TypeScript folosește `JSON.parse()` pentru a analiza fiecare linie JSON:

```typescript
export function parseEvent(line: string): ThreadEvent {
  const data = JSON.parse(line);
  // Handle different event types...
}
```

Versiunea C# folosește `System.Text.Json.JsonDocument` în schimb:

```csharp
public static ThreadEvent Parse(string line)
{
    using var document = JsonDocument.Parse(line);
    var root = document.RootElement;
    var type = GetRequiredString(root, "type", "event.type");

    return type switch
    {
        "thread.started" => new ThreadStartedEvent(GetRequiredString(root, "thread_id", ...)),
        "turn.started" => new TurnStartedEvent(),
        "turn.completed" => new TurnCompletedEvent(ParseUsage(...)),
        // ...
        _ => new UnknownThreadEvent(type, root.Clone()),
    };
}
```

Există un truc mic dar important aici: `root.Clone()` este necesar deoarece elementele din `JsonDocument` devin invalide după ce documentul este eliminat. Trebuie să păstrăm o copie pentru tipurile de evenimente necunoscute. Acesta este pur și simplu unul dintre diferențele dintre gestionarea JSON în C# și JavaScript.

#### 2. Diferențe în gestionarea proceselor

Aici cele două SDK-uri diferă cel mai mult. Node.js și .NET au convenții de runtime diferite, deci implementarea trebuie să se adapteze.

TypeScript folosește funcția `spawn()` din Node.js:

```typescript
const child = spawn(this.executablePath, commandArgs, { env, signal });
```

C# folosește `System.Diagnostics.Process` din .NET:

```csharp
using var process = new Process { StartInfo = startInfo };
process.Start();

// stdin/stdout/stderr must be managed manually
```

Mai specific, versiunea C# trebuie să configureze procesul astfel:

```csharp
var startInfo = new ProcessStartInfo
{
    FileName = _executablePath,
    RedirectStandardInput = true,
    RedirectStandardOutput = true,
    RedirectStandardError = true,
    UseShellExecute = false,
    CreateNoWindow = true,
};
```

Cea mai mare diferență este mecanismul de anulare. TypeScript folosește `AbortSignal`, care face parte din API-ul Web și este foarte convenabil de utilizat:

```typescript
const child = spawn(cmd, args, { signal: cancellationSignal });
```

C# folosește `CancellationToken` în schimb:

```csharp
public async IAsyncEnumerable<string> RunAsync(
    CodexExecArgs args,
    [EnumeratorCancellation] CancellationToken cancellationToken = default)
{
    // Check cancellation status inside the loop
    while (!cancellationToken.IsCancellationRequested)
    {
        // Process output...
    }

    // Terminate the process when cancellation is requested
    if (cancellationToken.IsCancellationRequested)
    {
        try { process.Kill(entireProcessTree: true); } catch { }
    }
}
```

La un nivel înalt, acesta este doar un alt exemplu al diferenței dintre ecosistemul API Web și ecosistemul .NET.

#### 3. Păstrarea serializării configurației

Ambele SDK-uri implementează logica care convertește configurația JSON în configurație TOML, deoarece CLI-ul Codex acceptă suprascrieri de configurație în format TOML. Această parte trebuie să rămână complet consecventă, altfel aceeași configurație se va comporta diferit în cele două SDK-uri.

Acesta este genul de detaliu pe care nu îl poți compromite. Succesul sau eșecul depinde adesea de detalii precum aceasta.

### Detalii de Implementare

#### Structura Proiectului

Am creat următoarea structură de proiect:

```
CodexSdk/
├── CodexSdk.csproj
├── Codex.cs           # Entry class
├── CodexThread.cs     # Conversation thread
├── CodexExec.cs       # Executor
├── Events.cs          # Event type definitions
├── Items.cs           # Item type definitions
├── EventParser.cs     # Event parser
├── OutputSchemaTempFile.cs  # Temporary file management
└── ...
```

Este o structură destul de curată, ceea ce a ajutat mult în timpul portării.

#### Exemplu de Utilizare

Utilizarea de bază rămâne consecventă cu SDK-ul TypeScript:

```csharp
using CodexSdk;

// Create a Codex instance
var codex = new Codex();
var thread = codex.StartThread();

// Execute a query
var result = await thread.RunAsync("Summarize this repository.");
Console.WriteLine(result.FinalResponse);
```

Gestionarea evenimentelor în streaming valorifică potrivirea de pattern-uri din C#:

```csharp
await foreach (var @event in thread.RunStreamedAsync("Analyze the code."))
{
    switch (@event)
    {
        case ItemCompletedEvent itemCompleted
            when itemCompleted.Item is AgentMessageItem msg:
            Console.WriteLine($"Assistant: {msg.Text}");
            break;
        case TurnCompletedEvent completed:
            Console.WriteLine($"Tokens: in={completed.Usage.InputTokens}");
            break;
        case CommandExecutionItem command:
            Console.WriteLine($"Command: {command.Command}");
            break;
    }
}
```

### Note

În timpul implementării, am colectat câteva lecții practice:

1. **Gestionarea proceselor**: Versiunea C# trebuie să gestioneze manual ciclul complet de viață al procesului, inclusiv terminarea procesului în timpul anulării. Folosește `Kill(entireProcessTree: true)` pentru a te asigura că procesele copil sunt de asemenea curățate.

2. **Gestionarea erorilor**: Folosim `InvalidOperationException` pentru a arunca erori de analizare, menținând stilul de gestionare a erorilor similar cu cel al SDK-ului TypeScript.

3. **Curățarea resurselor**: `OutputSchemaTempFile` implementează `IAsyncDisposable` pentru a asigura curățarea corectă a fișierelor temporare.

4. **Variabile de mediu**: Versiunea C# suportă suprascrierea completă a variabilelor de mediu ale procesului prin `CodexOptions.Env`. Este o funcționalitate mică, dar foarte practică.

5. **Diferențe de platformă**: Versiunea C# nu include logica versiunii TypeScript pentru localizarea automată a binarelor în pachetele npm. Deoarece proiectele .NET nu depind de obicei de npm, calea către executabilul `codex` trebuie specificată prin variabila de mediu `CODEX_EXECUTABLE` sau `CodexPathOverride`.

## Concluzie

Portarea unui SDK TypeScript matur la C# nu este doar o chestiune de conversie de sintaxă - necesită și înțelegerea filosofiilor de design ale ambelor limbaje. Flexibilitatea TypeScript și caracteristicile ecosistemului JavaScript precum `AbortSignal` au nevoie de omologhi adecvați în C#.

Concluzia cheie este aceasta: **menținerea consistenței API contează mai mult decât menținerea consistenței la nivel de implementare**. Utilizatorii sunt interesați dacă interfața este ușor de folosit, nu dacă implementarea internă este identică. Asta sună simplu, dar a face aceste compromisuri necesită judecată.

Dacă lucrezi la o portare cross-limbaj similară, experiența noastră este să înțelegi mai întâi complet arhitectura SDK-ului original, apoi să îl traduci modul cu modul și în cele din urmă să folosești o suită completă de teste pentru a asigura consistența comportamentală. Acest gen de muncă nu poate fi grăbită.

Totul se va rezolva în cele din urmă.
