## Rămas bun iadului scripturilor: De ce am ales C# pentru un sistem de construcție modern

> O analiză aprofundată a modului în care proiectul HagiCode folosește Nuke pentru a construi un pipeline de construcție automatizat, type-safe, cross-platform și extrem de extensibil, abordând complet punctele dureroase de întreținere ale scripturilor de construcție tradiționale.

## Context

De-a lungul lungii călătorii a dezvoltării software, cuvântul „construcție" tinde să inspire atât iubire cât și frustrare. Iubire, pentru că cu un singur clic, codul se transformă într-un produs, și acesta este unul dintre cele mai satisfăcătoare momente în programare. Frustrare, pentru că menținerea unei grămezi de scripturi de construcție dezordonate poate fi un coșmar.

În multe proiecte, suntem obișnuiți să scriem scripturi în Python sau să ne bazăm pe fișiere de configurare XML. Imaginați-vă frica de a fi dominați de blocuri `<property>`. Dar pe măsură ce complexitatea proiectului crește, în special într-un proiect precum HagiCode care acoperă lucrul pe frontend și backend, platforme multiple și limbaje multiple, abordările tradiționale de construcție încep să își arate limitele. Logică de script dispersată, fără verificare de tip, suport IDE slab... fiecare dintre aceste probleme devine o mică capcană care împiedică echipa din când în când.

Pentru a rezolva acele puncte dureroase, proiectul HagiCode a introdus **Nuke**, un sistem de construcție modern bazat pe C#. Nu este doar un instrument. Este un mod diferit de a gândi despre fluxurile de lucru de construcție. Astăzi, vom analiza de ce l-am ales și cum a îmbunătățit dramatic experiența noastră de dezvoltare.

## Despre HagiCode

> O scurtă introducere la ceea ce construim

Construim **HagiCode**, un asistent de programare bazat pe AI care face experiența de dezvoltare mai inteligentă, mai convenabilă și mai plăcută.

**Inteligent**: Asistența AI este disponibilă pe tot parcursul procesului, de la idei la cod, multiplicând eficiența dezvoltării. **Convenabil**: Operațiile concurente multi-thread utilizează complet resursele și mențin fluxurile de lucru fluide. **Distractiv**: Gamificarea și un sistem de realizări fac programarea mai puțin monotonă și mult mai satisfăcătoare.

Proiectul evoluează rapid. Dacă sunteți interesați de redactarea tehnică, managementul cunoștințelor sau dezvoltarea asistată de AI, vă invităm să vizitați [GitHub](https://github.com/HagiCode-org/site).

## Analiză de bază: De ce Nuke?

Vă puteți întreba: „Există deja atât de multe sisteme de construcție, cum ar fi Make, Gradle sau chiar scripturi Shell simple. De ce să folosim unul bazat pe C#?"

Aceasta este o întrebare corectă. Atracția de bază a lui Nuke este că aduce funcționalitățile limbajului de programare pe care le cunoaștem deja bine în lumea scripturilor de construcție.

### 1. Modularizarea fluxurilor de lucru de construcție: Arta Target-urilor

Nuke are o filosofie de design foarte clară: **totul este un target**.

În scripturile tradiționale, este ușor să ajungi la sute de linii de logică de execuție liniară încurcate împreună. În Nuke, descompunem procesul de construcție în `Target`-uri independente. Fiecare target face exact un singur lucru, de exemplu:

- `Clean`: curățarea directorului de ieșire
- `Restore`: restaurarea pachetelor de dependențe
- `Compile`: compilarea codului
- `Test`: rularea testelor unitare

Acest design se aliniază foarte bine cu principiul responsabilității unice. Ca și piesele de construcție, aceste Target-uri pot fi combinate liber. Mai important, Nuke ne permite să definim dependențe între Target-uri. De exemplu, dacă doriți să rulați `Test`, sistemul va verifica automat dacă `Compile` a fost deja executat. Dacă doriți `Compile`, atunci `Restore` vine natural înainte.

Acest graf de dependențe face logica mult mai clară și îmbunătățește semnificativ eficiența execuției, deoarece Nuke poate analiza automat calea optimă de execuție.

### 2. Siguranța tipurilor: Rămas bun coșmarului greșelilor de tastare

Oricine a scris scripturi de construcție în Python s-a probabil confruntat cu această situație stânjenitoare: scriptul rulează cinci minute și apoi eșuează pentru că `Confi.guration` a fost scris greșit sau pentru că un string a fost transmis unde se aștepta un parametru numeric.

Cel mai mare avantaj al scrierii scripturilor de construcție în C# este **siguranța tipurilor**. Asta înseamnă:

- **Verificări la compilare**: IDE-ul vă poate spune ce este în neregulă în timp ce tastați, în loc să vă facă să așteptați până la execuție.
- **Refactorizare sigură**: dacă doriți să redenumiți o variabilă sau metodă, IDE-ul o poate gestiona într-un singur pas fără să vă forțeze să faceți o căutare și înlocuire globală riscantă.
- **Completare inteligentă**: IntelliSense puternic vă poate completa codul automat, deci nu trebuie să răsfoiți constant documentația pentru a vă aminti API-uri obscure.

### 3. Suport cross-platform: O experiență de construcție consistentă

În trecut, scriam `.bat` pe Windows și `.sh` pe Linux, și apoi poate adăugam un script Python pentru a acoperi diferența. Acum, oriunde poate rula .NET Core, sau .NET 5+ astăzi, Nuke poate rula de asemenea.

Asta înseamnă că membrii echipei pot folosi Windows, Linux sau macOS, și indiferent dacă preferă Visual Studio, VS Code sau Rider, toți execută exact aceeași logică. Acest lucru reduce mare problemele cauzate de diferențele de mediu, în special problema clasică „funcționează pe mașina mea."

### 4. Managementul parametrilor și configurației

Nuke oferă un mecanism de parsare a parametrilor foarte elegant. Nu trebuie să parsați manual `string[] args`. Pur și simplu definiți o proprietate și o annotați cu `[Parameter]`, iar Nuke gestionează automat maparea din argumentele de linie de comandă și fișierele de configurare.

De exemplu, putem defini configurația de construcție foarte ușor:

```csharp
[Parameter("Configuration to build - Default is 'Debug'")]
readonly Configuration BuildConfiguration = IsLocalBuild ? Configuration.Debug : Configuration.Release;

Target Compile => _ => _
    .DependsOn(Restore)
    .Executes(() =>
    {
        // Use BuildConfiguration here; it is type-safe
        DotNetBuild(s => s
            .SetConfiguration(BuildConfiguration)
            .SetProjectFile(SolutionFile));
    });
```

Acest stil este atât intuitiv cât și rezistent la greșeli.

## Ghid practic: Cum să punem în practică într-un proiect

Doar teoria nu ne duce nicăieri. Să vedem cum am aplicat efectiv această abordare în proiectul HagiCode.

### 1. Planificarea structurii proiectului

Nu voiam ca scripturile de construcție să aglomereze rădăcina proiectului, și nici nu voiam o structură de directoare atât de adâncă încât să semene cu anumite proiecte Java. Așa că am plasat toate fișierele de construcție legate de Nuke într-un singur director `nukeBuild/`.

Beneficiile sunt directe:

- Rădăcina proiectului rămâne curată.
- Logica de construcție rămâne coerentă și mai ușor de gestionat.
- Noii membri ai echipei pot vedea imediat: „Aici se află logica legată de construcție."

### 2. Proiectarea unui lanț de dependențe clar de Target-uri

La proiectarea Target-urilor, am urmat un principiu: **atomicitate plus flux de dependențe**.

Fiecare Target ar trebui să fie suficient de mic pentru a face exact un singur lucru. De exemplu, `Clean` ar trebui doar să șteargă fișiere și să nu se ocupe și de ambalare pe ascuns.

Un flux de dependențe recomandat arată cam așa:

`Clean` -> `Restore` -> `Compile` -> `Test` -> `Pack`

Desigur, acest lucru nu este absolut. De exemplu, dacă doriți doar să rulați teste și nu doriți să ambalați nimic, Nuke vă permite să executați `nuke Test` direct, și va gestiona automat pașii necesari de `Restore` și `Compile` mai întâi.

### 3. Gestionarea robustă a erorilor și logare

Care este cel mai frustrant lucru despre scripturile de construcție? Mesajele de eroare neclare. O construcție eșuată care loghează doar „Error: 1" este suficientă pentru a înnebuni pe oricine.

În Nuke, deoarece putem folosi direct gestionarea excepțiilor din C#, putem capta și raporta erori cu mult mai mare precizie.

```csharp
Target Publish => _ => _
    .DependsOn(Test)
    .Executes(() =>
    {
        try 
        {
            // Try publishing to NuGet
            DotNetNuGetPush(s => s
                .SetTargetPath(ArtifactPath)
                .SetSource("https://api.nuget.org/v3/index.json")
                .SetApiKey(ApiKey));
        }
        catch (Exception ex)
        {
            Log.Error($"Publishing failed. Team, please check whether the key is correct: {ex.Message}");
            throw; // Make sure the build process exits with a non-zero exit code
        }
    });
```

### 4. Utilizarea testelor pentru a proteja calitatea

Scripturile de construcție sunt și ele cod, ceea ce înseamnă că au nevoie și de teste. Nuke ne permite să scriem teste pentru fluxurile de lucru de construcție, astfel încât atunci când schimbăm logica de construcție, să nu stricăm accidental procesele de lansare existente. Acest lucru este deosebit de important în pipeline-urile de integrare continuă.

## Rezumat

Prin introducerea Nuke, procesul de construcție al HagiCode a devenit mai fluid ca niciodată. Aceasta nu a fost doar o înlocuire de instrument. A fost o actualizare a mentalității de inginerie.

**Ce am câștigat?**

- **Mentenabilitate**: configurație ca și cod, logică mai clară și o cale de integrare mai rapidă pentru noi contribuitori.
- **Stabilitate**: tipizarea puternică elimină peste 90% din greșelile de nivel inferior.
- **Consecvență**: o experiență unificată cross-platform care elimină deviațiile de mediu.

Dacă scrierea scripturilor de construcție se simțea înainte ca și cum „te-ai mișca prin întuneric", atunci folosirea Nuke se simte ca și cum „ai merge noaptea cu luminile aprinse." Dacă sunteți obosit de menținerea unor scripturi greu de depanat, luați în considerare mutarea logicii de construcție în lumea C#. S-ar putea să descoperiți că și construcțiile pot fi elegante.
