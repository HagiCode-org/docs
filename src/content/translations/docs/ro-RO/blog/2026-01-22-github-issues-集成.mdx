## Construirea unei integrări GitHub Issues de la zero: Practica conexiunii directe din frontend în HagiCode

> Acest articol documentează procesul complet de integrare a GitHub Issues în platforma HagiCode. Vom explora cum să folosim o arhitectură „conexiune directă din frontend + backend minimal" pentru a realiza autentificare OAuth securizată și sincronizare eficientă a problemelor, menținând în același timp backend-ul ușor.

## Context: De ce să integrăm GitHub?

Ca platformă de dezvoltare asistată de AI, valoarea de bază a HagiCode constă în conectarea ideilor cu implementarea. Dar în utilizarea reală, am descoperit că, după ce utilizatorii finalizează un Proposal în HagiCode, aceștia trebuie adesea să copieze manual conținutul în GitHub Issues pentru urmărirea proiectului.

Acest lucru creează câteva puncte dureroase evidente:

1.  **Flux de lucru fragmentat**: Utilizatorii trebuie să comute între două sisteme. Experiența nu este fluidă, iar informațiile cheie se pot pierde cu ușurință în timpul copierii și lipirii.
2.  **Colaborare dificilă**: Alți membri ai echipei sunt obișnuiți să verifice sarcinile pe GitHub și nu pot vedea direct progresul propunerilor din HagiCode.
3.  **Muncă manuală repetitivă**: De fiecare dată când o propunere este actualizată, cineva trebuie să actualizeze manual problema corespunzătoare pe GitHub, adăugând costuri de întreținere inutile.

Pentru a rezolva această problemă, am decis să introducem funcția **Integrare GitHub Issues**, conectând sesiunile HagiCode cu depozitele GitHub pentru a permite „sincronizare cu un singur clic."

## Alegere tehnică: Conexiune directă din frontend vs. Proxy prin backend

La proiectarea abordării de integrare, aveam două opțiuni în fața noastră: „modelul tradițional cu proxy prin backend" și „modelul mai agresiv cu conexiune directă din frontend."

### Compararea soluțiilor

În **modelul tradițional cu proxy prin backend**, fiecare cerere de la frontend trebuie mai întâi să treacă prin backend-ul nostru, care apoi apelează API-ul GitHub. Acest lucru centralizează logica, dar pune și o povară semnificativă pe backend:

1.  **Backend umflat**: Ar trebui să scriem un wrapper dedicat pentru clientul GitHub API și, de asemenea, să gestionăm mașina de stări OAuth complexă.
2.  **Risc pentru token**: Token-ul GitHub al utilizatorului ar trebui stocat în baza de date a backend-ului. Chiar și cu criptare, acest lucru crește suprafața de atac.
3.  **Costuri de dezvoltare**: Ar fi nevoie de migrări ale bazei de date pentru a stoca token-uri și un serviciu de sincronizare suplimentar de întreținut.

**Modelul cu conexiune directă din frontend** este mult mai ușor. În această abordare, folosim backend-ul doar pentru cel mai sensibil pas de „schimb de secret" (callback-ul OAuth). După obținerea token-ului, îl stocăm direct în `localStorage`-ul browserului. Operațiunile ulterioare, cum ar fi crearea problemelor și actualizarea comentariilor, sunt trimise direct din frontend către GitHub prin HTTP.

| Dimensiune de comparare | Model cu proxy prin backend | Model cu conexiune directă din frontend |
| :--- | :--- | :--- |
| **Complexitate backend** | Necesită un serviciu OAuth complet și un client GitHub API | Are nevoie doar de un endpoint de callback OAuth |
| **Managementul token-urilor** | Trebuie criptate și stocate în baza de date, cu risc de scurgere | Stocate în browser și vizibile doar utilizatorului |
| **Cost de implementare** | Necesită migrări de bază de date și dezvoltare multi-serviciu | În principal lucru pe frontend |
| **Experiența utilizatorului** | Logică centralizată, dar latența serverului poate fi ușor mai mare | Răspuns extrem de rapid cu interacțiune directă cu GitHub |

Deoarece am dorit o integrare rapidă și modificări minime ale backend-ului, **în cele din urmă am ales „modelul cu conexiune directă din frontend"**. Este ca și cum i-ai da browserului un „ permis temporar." Odată ce primește permisul, browserul poate merge să se ocupe de lucruri pe GitHub de unul singur, fără a cere aprobarea administratorului backend de fiecare dată.

## Design de bază: Fluxul de date și securitatea

După stabilirea arhitecturii, trebuia să proiectăm fluxul specific de date. Nucleul procesului de sincronizare este cum să obținem token-ul în mod securizat și să-l folosim eficient.

### Diagrama arhitecturii generale

Întregul sistem poate fi abstractizat în trei roluri: browser-ul (frontend), backend-ul HagiCode și GitHub.

```text
+--------------+        +--------------+        +--------------+
| Frontend     |        |   Backend    |        |   GitHub     |
| React        |        |   ASP.NET    |        |   REST API   |
|              |        |              |        |              |
|  +--------+  |        |              |        |              |
|  | OAuth  |--+--------> /callback    |        |              |
|  | Flow   |  |        |              |        |              |
|  +--------+  |        |              |        |              |
|              |        |              |        |              |
|  +--------+  |        |  +--------+  |        |  +--------+  |
|  | GitHub |  +------------> Session |  +----------> Issues |  |
|  | API    |  |        |  |Metadata|  |        |  |        |  |
|  | Direct |  |        |  +--------+  |        |  +--------+  |
|  +--------+  |        |              |        |              |
+--------------+        +--------------+        +--------------+
```

**Punctul cheie**: doar un singur pas mic în OAuth (schimbarea codului pentru un token) trebuie să treacă prin backend. După aceea, munca grea (crearea problemelor) este gestionată direct între frontend și GitHub.

### Fluxul de date de sincronizare în detaliu

Când utilizatorul face clic pe butonul „Sincronizare cu GitHub" în interfața HagiCode, are loc o serie de acțiuni complexe:

```text
Utilizatorul face clic pe „Sincronizare cu GitHub"
         │
         ▼
1. Frontend verifică localStorage pentru token-ul GitHub
         │
         ▼
2. Formatează conținutul problemei (convertește Proposal în Markdown)
         │
         ▼
3. Frontend apelează direct API-ul GitHub pentru a crea/actualiza problema
         │
         ▼
4. Apelează API-ul backend HagiCode pentru a actualiza Session.metadata (stochează URL-ul problemei și alte informații)
         │
         ▼
5. Backend difuzează evenimentul SessionUpdated prin SignalR
         │
         ▼
6. Frontend primește evenimentul și actualizează interfața pentru a afișa starea „Sincronizat"
```

### Design de securitate

Securitatea este întotdeauna prioritatea numărul unu la integrarea serviciilor terțe. Am făcut următoarele considerații:

1.  **Apărare împotriva atacurilor CSRF**: Generăm un parametru `state` aleatoriu în timpul redirecționării OAuth și îl stocăm în `sessionStorage`. Validăm strict starea în callback pentru a preveni cererile falsificate.
2.  **Stocare izolată a token-ului**: Token-ul este stocat doar în `localStorage`-ul browserului. Folosind Politica de Aceeași Origine, doar scripturile HagiCode îl pot citi, evitând riscul ca o scurgere din baza de date de pe server să afecteze utilizatorii.
3.  **Limite de eroare**: Am proiectat gestionare dedicate pentru erorile comune ale API-ului GitHub (cum ar fi 401 token expirat, 422 eșec de validare și 429 limitare de rată), astfel încât utilizatorii primească feedback prietenos.

## În practică: Detalii de implementare în cod

Teoria merge doar până într-un punct. Să vedem cum funcționează codul efectiv.

### 1. Modificări minime ale backend-ului

Backend-ul trebuie să facă doar două lucruri: să stocheze informațiile de sincronizare și să gestioneze callback-ul OAuth.

### 2. Fluxul OAuth pe frontend

Acesta este punctul de intrare al conexiunii. Folosim fluxul standard Authorization Code Flow.

```typescript
// src/HagiCode.Client/src/services/githubOAuth.ts

// Generate the authorization URL and redirect
export async function generateAuthUrl(): Promise<string> {
  const state = generateRandomString(); // Generate a random string for CSRF protection
  sessionStorage.setItem('hagicode_github_state', state);
  
  const params = new URLSearchParams({
    client_id: clientId,
    redirect_uri: window.location.origin + '/settings?tab=github&oauth=callback',
    scope: ['repo', 'public_repo'].join(' '),
    state: state,
  });
  
  return `https://github.com/login/oauth/authorize?${params.toString()}`;
}

// Handle the code-to-token exchange on the callback page
export async function exchangeCodeForToken(code: string, state: string): Promise<GitHubToken> {
  // 1. Validate state to prevent CSRF
  const savedState = sessionStorage.getItem('hagicode_github_state');
  if (state !== savedState) throw new Error('Invalid state parameter');

  // 2. Call the backend API to exchange the token
  // Note: this must go through the backend because ClientSecret cannot be exposed to the frontend
  const response = await fetch('/api/GitHubOAuth/callback', {
    method: 'POST',
    headers: { 'Content-Type': 'application/json' },
    body: JSON.stringify({ code, state, redirectUri: window.location.origin + '/settings?tab=github&oauth=callback' }),
  });

  if (!response.ok) throw new Error('Failed to exchange token');
  
  const token = await response.json();
  
  // 3. Save into LocalStorage
  saveToken(token);
  return token;
}
```

### 3. Wrapper pentru clientul GitHub API

Odată ce avem token-ul, avem nevoie de un instrument solid pentru a apela API-ul GitHub.

```typescript
// src/HagiCode.Client/src/services/githubApiClient.ts

const GITHUB_API_BASE = 'https://api.github.com';

// Core request wrapper
async function githubApi<T>(endpoint: string, options: RequestInit = {}): Promise<T> {
  const token = localStorage.getItem('gh_token');
  if (!token) throw new Error('Not connected to GitHub');
  
  const response = await fetch(`${GITHUB_API_BASE}${endpoint}`, {
    ...options,
    headers: {
      ...options.headers,
      Authorization: `Bearer ${token}`,
      Accept: 'application/vnd.github.v3+json', // Specify the API version
    },
  });
  
  // Error handling logic
  if (!response.ok) {
    if (response.status === 401) throw new Error('GitHub token expired, please reconnect');
    if (response.status === 403) throw new Error('No permission to access this repository or rate limit exceeded');
    if (response.status === 422) throw new Error('Issue validation failed, the title may be duplicated');
    throw new Error(`GitHub API Error: ${response.statusText}`);
  }
  
  return response.json();
}

// Create issue
export async function createIssue(owner: string, repo: string, data: { title: string, body: string, labels: string[] }) {
  return githubApi(`/repos/${owner}/${repo}/issues`, {
    method: 'POST',
    body: JSON.stringify(data),
  });
}
```

### 4. Formatarea conținutului și sincronizarea

Ultimul pas este convertirea datelor sesiunii HagiCode în formatul unei probleme GitHub. Este oarecum similar cu o muncă de traducere.

```typescript
// Convert a Session object into a Markdown string
function formatIssueForSession(session: Session): string {
  let content = `# ${session.title}\n\n`;
  content += `**> HagiCode Session:** #${session.code}\n`;
  content += `**> Status:** ${session.status}\n\n`;
  content += `## Description\n\n${session.description || 'No description provided.'}\n\n`;
  
  // If this is a Proposal session, add extra fields
  if (session.type === 'proposal') {
    content += `## Chief Complaint\n\n${session.chiefComplaint || ''}\n\n`;
    // Add a deep link so users can jump back from GitHub to HagiCode
    content += `---\n\n**[View in HagiCode](hagicode://sessions/${session.id})**\n`;
  }
  
  return content;
}

// Main logic when clicking the sync button
const handleSync = async (session: Session) => {
  try {
    const repoInfo = parseRepositoryFromUrl(session.repoUrl); // Parse the repository URL
    if (!repoInfo) throw new Error('Invalid repository URL');

    toast.loading('Syncing to GitHub...');
    
    // 1. Format content
    const issueBody = formatIssueForSession(session);
    
    // 2. Call API
    const issue = await githubApiClient.createIssue(repoInfo.owner, repoInfo.repo, {
      title: `[HagiCode] ${session.title}`,
      body: issueBody,
      labels: ['hagicode', 'proposal', `status:${session.status}`],
    });
    
    // 3. Update Session Metadata (save the issue link)
    await SessionsService.patchApiSessionsSessionId(session.id, {
      metadata: {
        githubIssue: {
          owner: repoInfo.owner,
          repo: repoInfo.repo,
          issueNumber: issue.number,
          issueUrl: issue.html_url,
          syncedAt: new Date().toISOString(),
        }
      }
    });

    toast.success('Synced successfully!');
  } catch (err) {
    console.error(err);
    toast.error('Sync failed, please check your token or network');
  }
};
```

## Rezumat și perspective

Cu această abordare de „conexiune directă din frontend", am realizat o integrare perfectă a GitHub Issues cu cel mai puțin cod de backend posibil.

### Câștiguri cheie
1.  **Eficiență ridicată de dezvoltare**: Modificările la backend sunt minime, în principal un câmp suplimentar în baza de date și un endpoint simplu de callback OAuth. Cea mai mare parte a logicii este finalizată pe frontend.
2.  **Securitate puternică**: Token-ul nu trece prin baza de date a serverului, reducând riscul de scurgere.
3.  **Experiență excelentă pentru utilizator**: Cererile sunt inițiate direct din frontend, deci viteza de răspuns este rapidă și nu este nevoie de redirecționare prin backend.

### Lucruri de care să țineți cont
Există câteva capcane de care trebuie să țineți cont în timpul implementării reale:
- **Setări OAuth App**: Nu uitați să introduceți `Authorization callback URL` corect în setările GitHub OAuth App (de obicei `http://localhost:3000/settings?tab=github&oauth=callback`).
- **Limite de rată**: API-ul GitHub limitează destul de strict cererile neautentificate, dar cu un token cota este de obicei suficientă (5000 cereri/oră).
- **Parsarea URL-urilor**: Utilizatorii introduc tot felul de URL-uri de depozite, asigurați-vă că expresiile regulate pot potrivi sufixele `.git`, formatele SSH și cazuri similare.

### Îmbunătățiri viitoare
Funcția actuală este încă o sincronizare unidirecțională (HagiCode -> GitHub). În viitor, planificăm să implementăm sincronizarea bidirecțională prin GitHub Webhooks. De exemplu, dacă o problemă este închisă pe GitHub, starea sesiunii pe partea HagiCode s-ar putea actualiza automat. Asta va necesita să expunem un endpoint webhook pe backend, ceea ce va fi un pas interesant următor.
