## Cum să construiți o stație de distribuire a descărcărilor rapide cu servere cloud cu cost redus

> Lățimea de bandă a stocării cloud este absurd de scumpă, accesul cross-border este dureros de lent, iar prețurile CDN sunt suficiente să sperie pe oricine... Dacă vă ocupați de distribuirea fișierelor, probabil cunoașteți bine aceste probleme. În această postare, vreau să împărtășesc o abordare cu cost redus pe care am elaborat-o în timpul construirii [HagiCode](https://hagicode.com): un server cloud plus un strat de cache Nginx. Costul a scăzut cu aproximativ jumătate, în timp ce viteza de descărcare s-a îmbunătățit destul de mult, ceea ce a fost cel puțin puțin consolant.

## Context

Când vine vorba de internet, viteza de descărcare și stabilitatea sunt într-adevăr parte a experienței utilizatorului. Fie că rulați un proiect open source sau un produs comercial, aveți încă nevoie să oferiți utilizatorilor o modalitate fiabilă de a descărca fișiere.

Descărcarea fișierelor direct din stocarea cloud, cum ar fi Azure Blob Storage sau AWS S3, pare simplă, dar vine cu destul de multe probleme practice:

**Latența rețelei**: Accesul cross-border și cross-region poate fi suficient de lent încât să vă facă să vreți să spargeți tastatura. Dacă utilizatorii trebuie să aștepte la nesfârșit, experiența nu va fi evident prea bună.

**Costul lățimii de bandă**: Traficul de ieșire al stocării cloud este dureros de scump. Accesarea Azure Blob Storage din China continentală costă aproximativ 0,5 CNY per GB, ceea ce înseamnă că 1 TB pe lună se adună la aproximativ 500 CNY. Pentru o echipă mică, aceasta nu este o sumă nesemnificativă. La urma urmei, banii nimănui nu vin din vânt.

**Restricții de acces**: În unele regiuni, accesul la servicii cloud de peste mări este instabil și uneori pur și simplu indisponibil. Utilizatorii vor să descarce fișierele dar nu pot, ceea ce este frustrant pentru toată lumea.

**Costul CDN**: CDN-urile comerciale pot rezolva aceste probleme, dar prețul este la fel de real. Majoritatea echipelor mici pur și simplu nu îl pot justifica.

Deci există o soluție care este atât accesibilă, cât și practică? Da. Folosiți un server cloud, un proxy invers și un strat de cache. Este o abordare directă, dar funcționează. Costul scade cu aproximativ jumătate, iar viteza se îmbunătățește de asemenea, ceea ce reprezintă un compromis decent.

## Designul arhitecturii

### Arhitectura generală

Să începem cu arhitectura completă:

```
Cererea utilizatorului
    ↓
Rezoluție DNS
    ↓
┌─────────────────────────────────────┐
│   Stratul proxy invers (Traefik/Bunker Web) │ ← Terminare SSL, rutare, protecție de securitate
├─────────────────────────────────────┤
│   Porturi: 80/443                     │
│   Funcții: Certificate Let's Encrypt automate │
│             Rutare gazdă            │
└─────────────────────────────────────┘
    ↓
┌─────────────────────────────────────┐
│   Stratul cache (Nginx)               │ ← Cache fișiere, compresie Gzip
├─────────────────────────────────────┤
│   Porturi: 8080(server) / 8081(desktop) │
│   Strategie cache:                   │
│   - index.json: 1 oră              │
│   - alte fișiere: 7 zile             │
│   Dimensiune cache: 1GB                   │
└─────────────────────────────────────┘
    ↓
┌─────────────────────────────────────┐
│   Origine (Azure Blob Storage)       │ ← Stocare fișiere
└─────────────────────────────────────┘
```

Ideea de bază a acestei arhitecturi este simplă: **plasați un cache între utilizatori și stocarea cloud**.

Cererea utilizatorilor ajunge mai întâi la stratul proxy invers pe serverul cloud, apoi stratul cache Nginx preia controlul. Dacă fișierul solicitat este deja în cache, este returnat imediat. Dacă nu, Nginx îl preia din stocarea cloud și stochează o copie locală în același timp. Data viitoare când cineva solicită același fișier, stocarea cloud nu mai trebuie să fie implicată.

### De ce să alegeți această arhitectură?

**Avantajele serverelor cloud**:
- Cost predictibil: furnizori precum Alibaba Cloud oferă servere cloud cu cost redus, cu instanțe 1-2 vCPU și 2 GB RAM la prețuri de aproximativ 50-100 CNY pe lună
- Implementare flexibilă: puteți configura reguli proxy invers și politici de cache liber
- Geografie flexibilă: puteți alege regiuni de server mai apropiate de utilizatorii dvs.
- Scalabilitate bună: puteți actualiza specificațiile serverului pe măsură ce traficul crește

**Arhitectura proxy invers + cache**:
- Reducerea presiunii pe origine: cache fișierele populare pentru a reduce accesul direct la stocarea cloud
- Cost mai mic: traficul serverului cloud este mult mai ieftin decât ieșirea din stocarea cloud
- Îmbunătățirea vitezei: accesul apropiat și lățimea de bandă a serverului sunt de obicei mai bune decât livrarea directă din stocarea cloud

**De ce să alegeți Nginx ca strat de cache?**

Aceasta nu a fost o alegere aleatoare. Nginx are câteva avantaje reale aici:

1. Performanță ridicată: Nginx este recunoscut pe scară largă pentru performanță excelentă de proxy invers
2. Cache matur: funcția integrată `proxy_cache` este stabilă și fiabilă
3. Utilizare redusă a resurselor: poate rula cu doar 256 MB de memorie
4. Configurare flexibilă: puteți aplica politici de cache diferite pentru tipuri diferite de fișiere

## Stratul proxy invers: Traefik vs Bunker Web

Soluția de implementare HagiCode suportă două opțiuni de proxy invers și fiecare are propriile puncte forte:

| Opțiune | Caracteristici | Scenarii potrivite |
|------|------|----------|
| **Traefik** | Ușor, SSL automat, configurare simplă | Implementare de bază, scenarii cu trafic redus |
| **Bunker Web** | WAF integrat, protecție DDoS, protecție anti-bot | Securitate ridicată, scenarii cu trafic intens |

### Traefik: Prima alegere ușoară

Traefik este un proxy invers HTTP modern și un load balancer. Cel mai mare avantaj al său este că configurarea este simplă și poate obține certificate Let's Encrypt automat.

Pentru implementările inițiale sau scenariile cu trafic redus, Traefik este adesea o alegere foarte bună:
- Folosește relativ puține resurse; 1,5 CPU și 512 MB memorie sunt suficiente
- Certificatele SSL sunt configurate automat, deci nu trebuie să le gestionați singuri
- Rutarea este configurată prin etichete Docker, ceea ce este suficient de convenabil

### Bunker Web: Pentru scenarii cu securitate ridicată

Bunker Web este un firewall de aplicație web bazat pe Nginx cu protecție de securitate mai cuprinzătoare.

Când ar trebui să luați în considerare trecerea la Bunker Web? De obicei în cazuri precum acestea:
- Sunteți sub atac DDoS
- Aveți nevoie de protecție ModSecurity
- Doriți protecție anti-bot
- Aveți cerințe de securitate mai stricte

HagiCode oferă scriptul `switch-deployment.sh` astfel încât să puteți comuta rapid între cele două opțiuni:

```bash
# Comutați la Bunker Web
./switch-deployment.sh bunkerweb

# Comutați înapoi la Traefik
./switch-deployment.sh traefik

# Verificați starea curentă
./switch-deployment.sh status
```

Scriptul efectuează verificări preliminare, verificări de sănătate și rollback automat, deci procesul de comutare este destul de sigur și fiabil.

## Configurarea stratului cache Nginx

Stratul cache este nucleul întregii arhitecturi, deci configurația Nginx face o diferență uriașă în performanța cache-ului.

### Configurarea căii cache

```nginx
# Configurarea căii cache
proxy_cache_path /var/cache/nginx levels=1:2 keys_zone=azure_cache:10m
                   max_size=1g inactive=7d use_temp_path=off;
```

Detalii parametri:
- `levels=1:2`: ierarhia directoarelor cache cu două niveluri pentru a îmbunătăți eficiența accesului la fișiere
- `keys_zone=azure_cache:10m`: zona de stocare a cheilor cache; 10 MB sunt suficiente pentru un număr mare de chei
- `max_size=1g`: dimensiunea maximă a cache-ului este 1 GB
- `inactive=7d`: șterge fișierele în cache dacă nu au fost accesate timp de 7 zile
- `use_temp_path=off`: scrie direct în directorul cache pentru performanță mai bună

### Strategie de cache pe niveluri

Diferite tipuri de fișiere au nevoie de strategii de cache diferite:

```nginx
# Serviciu de descărcare server
server {
    listen 8080;

    # Cache pe termen scurt pentru index.json (pentru a permite actualizări la timp)
    location /index.json {
        proxy_cache azure_cache;
        proxy_cache_valid 200 1h;
        proxy_cache_key "$scheme$server_port$request_uri";
        add_header X-Cache-Status $upstream_cache_status;
        add_header Cache-Control "public, max-age=3600";

        # Proxy invers către Azure OSS
        proxy_pass https://${SERVER_DL_HOST}/${SERVER_DL_CONTAINER}$uri?${SERVER_DL_SAS_TOKEN};
        proxy_ssl_server_name on;
        proxy_ssl_protocols TLSv1.2 TLSv1.3;
    }

    # Cache pe termen lung pentru fișiere statice cum ar fi pachetele de instalare
    location / {
        proxy_cache azure_cache;
        proxy_cache_valid 200 7d;
        proxy_cache_key "$scheme$server_port$request_uri";
        add_header X-Cache-Status $upstream_cache_status;
        add_header Cache-Control "public, max-age=604800";

        proxy_pass https://${SERVER_DL_HOST}/${SERVER_DL_CONTAINER}$uri?${SERVER_DL_SAS_TOKEN};
        proxy_ssl_server_name on;
        proxy_ssl_protocols TLSv1.2 TLSv1.3;
    }
}
```

**De ce este proiectat astfel?**

`index.json` este fișierul de verificare a versiunii, deci trebuie să se actualizeze prompt. Cu o fereastră de cache de 1 oră, utilizatorii pot detecta o nouă versiune în cel mult o oră de la publicare.

Fișierele statice cum ar fi pachetele de instalare se schimbă rar, deci cache-ul lor timp de 7 zile reduce considerabil accesul la origine. Când este nevoie de o actualizare, puteți pur și simplu să goliți cache-ul manual.

**Header-ul de răspuns `X-Cache-Status`**:

Acest header vă ajută să inspectați comportamentul de hit cache:
- `HIT`: hit cache
- `MISS`: miss cache, preluat de la origine
- `EXPIRED`: cache expirat, preluat din nou de la origine
- `BYPASS`: cache ocolit

Cum să verificați:
```bash
curl -I https://server.dl.hagicode.com/app.zip
```

## Analiza costurilor

Presupunând 1 TB de trafic de descărcare pe lună. Să facem calculele:

| Opțiune | Cost trafic | Cost server | Total |
|------|----------|------------|------|
| Azure OSS direct | Aproximativ 500 CNY | 0 CNY | 500 CNY |
| Server cloud + OSS (raport hit cache 80%) | 100 CNY + 80 CNY | 60 CNY | 240 CNY |
| CDN comercial | 300-500 CNY | 0 CNY | 300-500 CNY |

**Concluzie**: adăugarea unui strat cache poate reduce costul de distribuire cu aproximativ 50%.

Această estimare presupune un raport de hit cache de 80%. În practică, dacă fișierele nu se schimbă des, raportul de hit poate fi chiar mai mare.

## Practica de implementare

### Pregătirea mediului

Mai întâi, configurați variabilele de mediu:

```bash
cd /path/to/hagicode_aliyun_deployment/docker
cp .env.example .env
vi .env  # Completați URL-ul SAS Azure OSS și URL-ul Lark Webhook
```

**Important**: fișierul `.env` conține informații sensibile cum ar fi SAS Token și URL-ul Webhook. Nu îl introduceți niciodată în controlul versiunilor.

### Configurarea DNS

Adăugați următoarele înregistrări DNS A:
- `server.dl.hagicode.com` → IP server
- `desktop.dl.hagicode.com` → IP server

### Inițializarea serverului

Folosiți Ansible pentru a inițializa serverul automat:

```bash
cd /path/to/hagicode_aliyun_deployment
ansible-playbook -i ./ansible/inventory/hosts.yml ./ansible/playbooks/init.yml
```

Acest playbook gestionează automat următoarele sarcini:
- Crearea utilizatorului de implementare
- Instalarea Docker și Docker Compose
- Configurarea cheilor SSH
- Setarea regulilor de firewall

Aceasta este principala muncă de configurare, iar automatizarea economisește mult timp.

### Implementarea serviciilor

```bash
./deploy.sh
```

Scriptul de implementare vă ajută să faceți următoarele:
- Verificați configurația de mediu
- Trageți cel mai recent cod
- Porniți containerele Docker
- Efectuați verificări de sănătate
- Trimiteți notificări de implementare (Lark)

O singură comandă este suficientă, ceea ce menține procesul convenabil.

### Verificarea implementării

```bash
# Verificați starea containerelor
docker ps

# Testați domeniile de descărcare
curl -I https://server.dl.hagicode.com/index.json
curl -I https://desktop.dl.hagicode.com/index.json
```

## Sfaturi de operare

### Gestionarea cache-ului

Cache-urile au de asemenea nevoie de mentenanță din când în când:

**Verificați utilizarea discului cache**:
```bash
docker volume inspect docker_nginx-cache
du -sh /var/lib/docker/volumes/docker_nginx-cache/_data
```

**Goliți manual cache-ul**:
```bash
./clear-cache.sh
```

Sau rulați comenzile manuale direct dacă este necesar:
```bash
docker exec nginx sh -c "rm -rf /var/cache/nginx/*"
docker restart nginx
```

### Limite de resurse

Pe un server de 1 nucleu, 2 GB, configurația limitelor de resurse arată astfel:

```yaml
services:
  traefik:
    deploy:
      resources:
        limits:
          cpus: '1.50'
          memory: 512M

  nginx:
    deploy:
      resources:
        limits:
          cpus: '0.50'
          memory: 256M
```

Pentru a monitoriza utilizarea resurselor, puteți rula ocazional:
```bash
docker stats
```

### Practici de securitate SAS Token

SAS Token-ul este credențialul folosit pentru a accesa Azure Blob Storage, deci divulgarea sa ar fi gravă:

- Nu introduceți fișierul `.env` în controlul versiunilor; este deja în `.gitignore`
- Setați un timp de expirare adecvat pentru SAS Token, cu 1 an recomandat
- Limitați permisiunile SAS Token la doar citire
- Rotiți SAS Token-urile regulat

### Monitorizare și alerte

HagiCode integrează notificări Lark/Feishu Webhook, care pot trimite alerte pentru următoarele evenimente:
- Succesul sau eșecul implementării
- Starea golirii cache-ului
- Excepții de serviciu

Notificările includ informații despre server, timestamp-uri și detalii de eroare, ceea ce face depanarea mult mai rapidă.

### Extensii de înaltă disponibilitate

Când un server nu mai este suficient, puteți lua în considerare următoarele:

1. **Scalare orizontală**: implementați multiple noduri și distribuiți traficul cu DNS round-robin sau un load balancer
2. **CDN în față**: plasați un CDN în fața serverelor cloud pentru acces și mai rapid
3. **Încălzirea cache-ului**: folosiți scripturi pentru a preîncărca fișierele populare în cache

## Note

Există câteva lucruri demne de reținut:

1. **Certificate SSL**: Let's Encrypt are limite de rată, deci nu comutați implementările prea frecvent sau emiterea certificatelor poate eșua
2. **Golirea cache-ului**: după actualizarea fișierelor importante, nu uitați să goliți cache-ul sau utilizatorii ar putea descărca încă versiunea veche
3. **Gestionarea logurilor**: curățați logurile Docker regulat sau discul se poate umple
4. **Strategie de backup**: faceți backup la fișiere precum `Traefik acme.json` și configurația Bunker Web
5. **Monitorizare și alerte**: configurați notificări Feishu pentru a urmări starea implementării și a răspunde rapid la probleme

## Concluzie

Un server cloud plus un strat de cache Nginx este tot ce este nevoie. HagiCode folosește această soluție cu un cost lunar destul de redus, aproximativ 60-100 CNY pentru server, iar rezultatele au fost foarte solide. Principalele avantaje sunt:

- **Cost predictibil**: cu aproximativ 50% mai ieftin decât utilizarea directă a stocării cloud sau plata unui CDN comercial
- **Implementare flexibilă**: alegeți Traefik sau Bunker Web în funcție de nevoile dvs.
- **Scalabilitate puternică**: puteți scala orizontal sau adăuga un CDN ulterior dacă este necesar
- **Operațiuni simple**: scripturi Shell plus Ansible fac implementarea automată simplă

Pentru echipe mici și dezvoltatori independenți care au nevoie de distribuire de fișiere, aceasta este cu siguranță o opțiune practică care merită încercată.

HagiCode rulează această arhitectură stabil în producție de ceva timp, iar descărcările utilizatorilor globali au rămas fiabile. Dacă căutați o soluție similară, merită cu siguranță o încercare.
