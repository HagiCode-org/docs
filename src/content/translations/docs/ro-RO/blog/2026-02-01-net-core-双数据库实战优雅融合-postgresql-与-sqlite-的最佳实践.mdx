## Bază de date duală .NET Core în practică: Făcând PostgreSQL și SQLite să coexiste pașnic

> La construirea aplicațiilor moderne, ne confruntăm adesea cu acest compromis: mediile de dezvoltare doresc ceva ușor și convenabil, în timp ce mediile de producție cer concurență ridicată și disponibilitate mare. Acest articol prezintă modul de a suporta elegant atât PostgreSQL cât și SQLite într-un proiect .NET Core și de a implementa cea mai bună practică de „SQLite pentru dezvoltare, PostgreSQL pentru producție."


## Context

În dezvoltarea software, diferențele dintre medii au fost întotdeauna una dintre cele mai grele probleme pentru echipele de inginerie. Luând platforma **HagiCode** pe care o construim ca exemplu: este un sistem de dezvoltare asistat de AI bazat pe ASP.NET Core 10 și React, cu Orleans integrat intern pentru managementul distribuit al stării. Stack-ul este modern și destul de sofisticat.

Dar în producție, trebuia să gestionăm scrieri cu concurență ridicată și interogări JSON complexe, și exact acolo începe SQLite ușor să își arate limitele.

Cum putem menține o singură bază de cod permițând în același timp aplicației să beneficieze de portabilitatea SQLite ca o aplicație desktop, și totodată să profite de performanța puternică a PostgreSQL ca un serviciu de nivel enterprise? Aceasta este întrebarea de bază pe care acest articol o explorează.

## Subiect de bază 1: Design arhitectural și abstractizare unificată

Pentru a suporta două baze de date în .NET Core, ideea cheie este să depindeți de abstractizări mai degrabă decât de implementări concrete. Trebuie să separăm selecția bazei de date de codul de business și să lăsăm stratul de configurație să decidă.

### Abordare de design

1.  **Interfață unificată**: Toată logica de business ar trebui să depindă de clasa de bază `DbContext` sau interfețe personalizate, mai degrabă decât de un `PostgreSqlDbContext` specific.
2.  **Condus de configurație**: Utilizarea elementelor de configurație din `appsettings.json` pentru a decide dinamic ce provider de bază de date să încarce la pornirea aplicației.
3.  **Izolarea funcționalităților**: Adăugarea logicii de adaptare pentru capabilități specifice PostgreSQL, cum ar fi JSONB, astfel încât aplicația încă se poate degrada grațios pe SQLite.

### Implementare cod: Configurarea dinamică a contextului

În `Program.cs` al ASP.NET Core, nu ar trebui să hardcodăm `UseNpgsql` sau `UseSqlite`. În schimb, ar trebui să citim configurația și să decidem dinamic.

Mai întâi, definiți clasa de configurație:

```csharp
public class DatabaseSettings
{
    public const string SectionName = "Database";
    
    // Database type: PostgreSQL or SQLite
    public string DbType { get; set; } = "PostgreSQL"; 
    
    // Connection string
    public string ConnectionString { get; set; } = string.Empty;
}
```

Apoi înregistrați serviciul în `Program.cs` pe baza configurației:

```csharp
// Read configuration
var databaseSettings = builder.Configuration.GetSection(DatabaseSettings.SectionName).Get<DatabaseSettings>();

// Register DbContext
builder.Services.AddDbContext<ApplicationDbContext>(options =>
{
    if (databaseSettings?.DbType?.ToLower() == "sqlite")
    {
        // SQLite configuration
        options.UseSqlite(databaseSettings.ConnectionString);
        
        // Handling SQLite's concurrent write limitations
        // Note: in production, enabling WAL mode is recommended to improve concurrency performance
    }
    else
    {
        // PostgreSQL configuration (default)
        options.UseNpgsql(databaseSettings.ConnectionString, npgsqlOptions =>
        {
            // Enable JSONB support, which is very useful when handling AI conversation records
            npgsqlOptions.UseJsonNet(); 
        });
        
        // Configure connection pool retry policy
        options.EnableRetryOnFailure(3);
    }
});
```

## Subiect de bază 2: Gestionarea diferențelor și strategia de migrare

Deși atât PostgreSQL cât și SQLite suportă standardul SQL, ele diferă semnificativ în capabilități și comportament specifice. Dacă aceste diferențe nu sunt gestionate cu atenție, puteți ajunge ușor în situația stânjenitoare în care totul funcționează local dar eșuează după implementare.

### 1. Gestionarea tipurilor JSON

În HagiCode, trebuie să stocăm o cantitate mare de prompt-uri și metadate AI, ceea ce implică de obicei coloane JSON.
- **PostgreSQL**: Are un tip `JSONB` nativ cu performanță excelentă de interogare.
- **SQLite**: Nu are un tip JSON nativ (versiunile mai noi includ extensia JSON1, dar maparea obiectelor diferă încă), deci datele sunt de obicei stocate ca TEXT.

**Soluție**:
În maparea entităților EF Core, o configurăm ca tip convertibil.

```csharp
protected override void OnModelCreating(ModelBuilder modelBuilder)
{
    base.OnModelCreating(modelBuilder);

    // Configure entity
    modelBuilder.Entity<PromptTemplate>(entity => 
    {
        entity.Property(e => e.Metadata)
              .HasColumnType("jsonb") // PG uses jsonb
              .HasConversion(
                  v => JsonSerializer.Serialize(v, (JsonSerializerOptions)null),
                  v => JsonSerializer.Deserialize<Dictionary<string, object>>(v, (JsonSerializerOptions)null)
              ); 
    });
}
```

Când se folosește SQLite, `HasColumnType("jsonb")` poate fi ignorat sau declanșa un avertisment. Totuși, deoarece `HasConversion` este configurat, datele sunt încă serializate și deserializate corect ca string-uri stocate într-un câmp TEXT, asigurând compatibilitatea.

### 2. Separarea strategiei de migrare

Nu încercați niciodată să faceți un set de scripturi Migration să funcționeze pentru atât PostgreSQL cât și SQLite în același timp. Diferențele în strategiile de generare a cheilor primare, sintaxa indexelor și alte detalii de bază de date vor cauza inevitabil eșecuri.

**Practica recomandată**:
Mențineți două ramuri sau proiecte de migrare. În fluxul de lucru de dezvoltare HagiCode, gestionăm astfel:

1.  **Etapa de dezvoltare**: Lucrați în principal cu SQLite. Folosiți `Add-Migration Init_Sqlite -OutputDir Migrations/Sqlite`.
2.  **Etapa de adaptare**: După dezvoltarea unei funcționalități, schimbați string-ul de conexiune la PostgreSQL și rulați `Add-Migration Init_Postgres -OutputDir Migrations/Postgres`.
3.  **Scripturi de automatizare**: Scrieți un script simplu PowerShell sau Bash pentru a aplica automat migrarea corectă pe baza variabilei de mediu curente.

```bash
# Pseudocode for simple deployment logic
if [ "$DATABASE_PROVIDER" = "PostgreSQL" ]; then
    dotnet ef database update --project Migrations.Postgres
else
    dotnet ef database update --project Migrations.Sqlite
fi
```

## Subiect de bază 3: Lecții învățate din HagiCode în producție

În timpul refactorizării **HagiCode** de la un model cu o singură bază de date la suport cu bază de date duală, am lovit câteva capcane și am adunat câteva lecții importante care vă pot ajuta să evitați aceleași greșeli.

### 1. Diferențe în concurență și tranzacții

PostgreSQL folosește o arhitectură server-client și suportă scrieri cu concurență ridicată cu niveluri puternice de izolare a tranzacțiilor. SQLite folosește blocare la nivel de fișier, deci operațiunile de scriere blochează întregul fișier de bază de date cu excepția cazului în care este activat modul WAL.

**Recomandare**:
Când scrieți logică de business care implică scrieri frecvente, cum ar fi salvarea în timp real a stării de editare a unui utilizator, trebuie să luați în considerare modelul de blocare al SQLite. La proiectarea modulului de colaborare OpenSpec în **HagiCode**, am introdus un mecanism de „îmbinare înainte de scriere" pentru a reduce frecvența scrierilor directe în baza de date, permițându-ne să menținem performanță bună pe ambele baze de date.

### 2. Managementul ciclului de viață al string-urilor de conexiune

Stabilirea unei conexiuni PostgreSQL este relativ costisitoare și depinde de pool-ul de conexiuni. Conexiunile SQLite sunt foarte ușoare, dar dacă nu sunt eliberate prompt, blocările de fișiere pot cauza timeout-uri pentru operațiunile ulterioare.

În `Program.cs`, putem ajusta fin comportamentul pentru fiecare bază de date:

```csharp
if (databaseSettings?.DbType?.ToLower() == "sqlite")
{
    // SQLite: keeping connections open can improve performance, but watch out for file locks
    options.UseSqlite(connectionString, sqliteOptions =>
    {
        // Set command timeout
        sqliteOptions.CommandTimeout(30);
    });
}
else
{
    // PG: make full use of connection pooling
    options.UseNpgsql(connectionString, npgsqlOptions =>
    {
        npgsqlOptions.MaxBatchSize(100);
        npgsqlOptions.CommandTimeout(30);
    });
}
```

### 3. Importanța acoperirii cu teste

Mulți dezvoltatori, inclusiv unii membri timpurii ai echipei noastre, tind să facă o greșeală comună: rularea testelor unitare doar în mediul de dezvoltare, care este de obicei SQLite.

În pipeline-ul CI/CD al HagiCode, am impus un pas GitHub Actions pentru a ne asigura că fiecare pull request rulează teste de integrare PostgreSQL.

```yaml
# Example snippet from .github/workflows/test.yml
- name: Run Integration Tests (PostgreSQL)
  run: |
    docker-compose up -d db_postgres
    dotnet test --filter "Category=Integration"
```

Acest lucru ne-a ajutat să prindem nenumărate bug-uri legate de diferențele de sintaxă SQL și sensibilitatea la majuscule/minuscule.

## Concluzie

Prin introducerea unui strat de abstractizare și injecție de dependențe condusă de configurație, am implementat cu succes o configurație duală PostgreSQL și SQLite în proiectul **HagiCode**. Acest lucru nu numai că a redus considerabil bariera de integrare pentru noii dezvoltatori eliminând necesitatea de a instala PostgreSQL, dar a oferit de asemenea garanții puternice de performanță pentru producție.

Pentru a rezuma punctele cheie:
1.  **Abstractizare pe primul loc**: Codul de business nu ar trebui să depindă de implementări concrete de baze de date.
2.  **Configurație separată**: Utilizați fișiere `appsettings.json` diferite pentru dezvoltare și producție.
3.  **Migrări separate**: Nu încercați să faceți un set de Migration să funcționeze peste tot.
4.  **Degradare funcțională**: Prioritizați compatibilitatea pe SQLite și performanța pe PostgreSQL.

Acest model arhitectural nu este potrivit doar pentru HagiCode, ci pentru orice proiect .NET care trebuie să găsească un echilibru între dezvoltare ușoară și producție grea.
