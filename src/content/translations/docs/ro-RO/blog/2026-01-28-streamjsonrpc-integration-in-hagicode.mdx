## Integrarea profundă și utilizarea practică a StreamJsonRpc în HagiCode

> Acest articol explică în detaliu modul în care proiectul HagiCode (anterior PCode) a integrat cu succes biblioteca de comunicare StreamJsonRpc de la Microsoft pentru a înlocui implementarea sa personalizată de JSON-RPC, abordând în același timp punctele dureroase tehnice și provocările arhitecturale întâmpinate în timpul procesului de integrare.

## Context

StreamJsonRpc este biblioteca de comunicare JSON-RPC întreținută oficial de Microsoft pentru .NET și TypeScript, cunoscută pentru siguranța puternică a tipurilor, generarea automată de proxy-uri și mecanismele mature de gestionare a excepțiilor. În proiectul HagiCode, echipa a decis să integreze StreamJsonRpc pentru a comunica cu instrumente AI externe precum iflow CLI și OpenCode CLI prin ACP (Agent Communication Protocol), eliminând în același timp costul de întreținere și bug-urile potențiale introduse de implementarea anterioară personalizată de JSON-RPC. Cu toate acestea, procesul de integrare a întâmpinat provocări specifice streaming-ului JSON-RPC, în special la gestionarea legării țintelor proxy și recunoașterii parametrilor generici.

Pentru a aborda aceste puncte dureroase, am luat o decizie curajoasă: reconstruirea întregului sistem de construcție de la zero. Impactul acelei decizii poate fi chiar mai mare decât vă așteptați, și voi explica în detaliu în curând.

## Analiză

Proiectul curent se află într-un stadiu critic de integrare a protocolului ACP și se confruntă cu următoarele puncte dureroase tehnice și provocări arhitecturale:

### 1. Limitările implementării personalizate
Implementarea JSON-RPC originală se află în `src/HagiCode.ClaudeHelper/AcpImp/` și include componente precum `JsonRpcEndpoint` și `ClientSideConnection`. Menținerea acestei baze de cod personalizate este costisitoare, și îi lipsesc capabilitățile avansate ale unei biblioteci mature, cum ar fi raportarea progresului și suportul pentru anulare.

### 2. Obstacole în integrarea StreamJsonRpc
La încercarea de a migra modelul existent `CallbackProxyTarget` la StreamJsonRpc, am descoperit că metoda `_rpc.AddLocalRpcTarget(target)` nu putea recunoaște țintele create prin modelul proxy. În mod specific, StreamJsonRpc nu putea împărți automat proprietățile tipului generic `T` în parametri de metodă RPC, cauzând eșecul pe partea serverului la procesarea apelurilor de metodă inițiate de client.

### 3. Stratificare arhitecturală confuză
`ClientSideConnection` existentă amestecă stratul de transport (WebSocket/Stdio), stratul de protocol (JSON-RPC) și stratul de business (interfața ACP Agent), ducând la responsabilități neclare. De asemenea, suferă de legături de metodă lipsă în `AcpAgentCallbackRpcAdapter`.

### 4. Jurnale lipsă
Stratul de transport WebSocket nu are logare de conținut JSON brut, făcând dificilă determinarea în timpul depanării RPC dacă o problemă provine de la serializare sau de la rețea.

## Soluție

Pentru a aborda problemele de mai sus, am adoptat următoarea soluție sistemică, optimizând din trei dimensiuni: refactorizare arhitecturală, integrare de biblioteci și depanare îmbunătățită.

### 1. Migrarea completă la StreamJsonRpc

#### Eliminarea codului moștenit
Ștergeți `JsonRpcEndpoint.cs`, `AgentSideConnection.cs` și convertoarele de serializare personalizate asociate precum `JsonRpcMessageJsonConverter`.

#### Integrarea bibliotecii oficiale
Introduceți pachetul NuGet `StreamJsonRpc` și folosiți clasa sa `JsonRpc` pentru a gestiona logica de comunicare de bază.

#### Abstractizarea stratului de transport
Definiți interfața `IAcpTransport` pentru a gestiona atât modurile de transport `WebSocket` cât și `Stdio` într-un mod unificat, asigurându-se că stratul de protocol este decuplat de stratul de transport.

```csharp
// Definition of the IAcpTransport interface
public interface IAcpTransport
{
    Task SendAsync(string message, CancellationToken cancellationToken = default);
    Task<string> ReceiveAsync(CancellationToken cancellationToken = default);
    Task CloseAsync(CancellationToken cancellationToken = default);
}

// WebSocket transport implementation
public class WebSocketTransport : IAcpTransport
{
    private readonly WebSocket _webSocket;

    public WebSocketTransport(WebSocket webSocket)
    {
        _webSocket = webSocket;
    }

    // Implement send and receive methods
    // ...
}

// Stdio transport implementation
public class StdioTransport : IAcpTransport
{
    private readonly StreamReader _reader;
    private readonly StreamWriter _writer;

    public StdioTransport(StreamReader reader, StreamWriter writer)
    {
        _reader = reader;
        _writer = writer;
    }

    // Implement send and receive methods
    // ...
}
```

### 2. Remedierea problemelor de recunoaștere a țintelor proxy

#### Analizarea `CallbackProxyTarget`
Inspectați logica existentă de generare a proxy-urilor dinamice și identificați cauza principală pentru care StreamJsonRpc nu le poate recunoaște. În majoritatea cazurilor, acest lucru se întâmplă deoarece obiectul proxy nu expune public semnăturile reale ale metodelor sau folosește tipuri de parametri nesuportate de StreamJsonRpc.

#### Refactorizarea transmiterii parametrilor
Împărțiți proprietățile generice în parametri expliciți de metodă RPC. În loc să vă bazați pe proprietăți dinamice, definiți DTO-uri (obiecte de transfer de date) Request/Response concrete astfel încât StreamJsonRpc să poată recunoaște corect semnăturile metodelor prin reflecție.

```csharp
// Original generic property approach
public class CallbackProxyTarget<T>
{
    public Func<T, Task> Callback { get; set; }
}

// Refactored concrete method approach
public class ReadTextFileRequest
{
    public string FilePath { get; set; }
}

public class ReadTextFileResponse
{
    public string Content { get; set; }
}

public interface IAcpAgentCallback
{
    Task<ReadTextFileResponse> ReadTextFileAsync(ReadTextFileRequest request);
    // Other methods...
}
```

#### Utilizarea `Attach` în loc de `AddLocalRpcTarget`
În unele scenarii complexe, proxy-area manuală a obiectului `JsonRpc` și gestionarea `RpcConnection` poate fi mai flexibilă decât adăugarea directă a unei ținte locale.

### 3. Implementarea legăturii de metode și a logării mai puternice

#### Implementarea `AcpAgentCallbackRpcAdapter`
Asigurați-vă că această componentă implementează explicit interfața proxy StreamJsonRpc și mapează metodele definite de protocolul ACP, cum ar fi `ReadTextFileAsync`, la handlere de callback StreamJsonRpc.

#### Integrarea logării
Interceptați și înregistrați textul brut al cererilor și răspunsurilor JSON-RPC în pipeline-ul de procesare a mesajelor WebSocket sau Stdio. Folosiți `ILogger` pentru a emite payload-ul brut înainte de parsare și după serializare astfel încât problemele de formatare pot fi diagnosticate mai ușor.

```csharp
// Transport wrapper with enhanced logging
public class LoggingAcpTransport : IAcpTransport
{
    private readonly IAcpTransport _innerTransport;
    private readonly ILogger<LoggingAcpTransport> _logger;

    public LoggingAcpTransport(IAcpTransport innerTransport, ILogger<LoggingAcpTransport> logger)
    {
        _innerTransport = innerTransport;
        _logger = logger;
    }

    public async Task SendAsync(string message, CancellationToken cancellationToken = default)
    {
        _logger.LogTrace("Sending message: {Message}", message);
        await _innerTransport.SendAsync(message, cancellationToken);
    }

    public async Task<string> ReceiveAsync(CancellationToken cancellationToken = default)
    {
        var message = await _innerTransport.ReceiveAsync(cancellationToken);
        _logger.LogTrace("Received message: {Message}", message);
        return message;
    }

    public async Task CloseAsync(CancellationToken cancellationToken = default)
    {
        _logger.LogDebug("Closing connection");
        await _innerTransport.CloseAsync(cancellationToken);
    }
}
```

### 4. Refactorizarea stratificării arhitecturale

#### Stratul de transport (`AcpRpcClient`)
Înfășurați conexiunea StreamJsonRpc și faceți-o responsabilă pentru apelurile `InvokeAsync` și managementul ciclului de viață al conexiunii.

```csharp
public class AcpRpcClient : IDisposable
{
    private readonly JsonRpc _rpc;
    private readonly IAcpTransport _transport;

    public AcpRpcClient(IAcpTransport transport)
    {
        _transport = transport;
        _rpc = new JsonRpc(new StreamRpcTransport(transport));
        _rpc.StartListening();
    }

    public async Task<TResponse> InvokeAsync<TResponse>(string methodName, object parameters)
    {
        return await _rpc.InvokeAsync<TResponse>(methodName, parameters);
    }

    public void Dispose()
    {
        _rpc.Dispose();
        _transport.Dispose();
    }

    // StreamRpcTransport is the StreamJsonRpc adapter for IAcpTransport
    private class StreamRpcTransport : IDuplexPipe
    {
        // Implement the IDuplexPipe interface
        // ...
    }
}
```

#### Stratul de protocol (`IAcpAgentClient` / `IAcpAgentCallback`)
Definiți interfețe clare client-agenți și agenți-client. Eliminați modelul de fabrică ciclică `Func<IAcpAgent, IAcpClient>` și înlocuiți-l cu injecție de dependențe sau înregistrare directă de callback-uri.

## Practică

Pe baza celor mai bune practici StreamJsonRpc și a experienței de proiect, următoarele sunt recomandările cheie pentru implementare:

### 1. DTO-uri cu tipizare puternică sunt mai bune decât obiecte dinamice
Avantajul de bază al StreamJsonRpc constă în tipizarea puternică. Nu folosiți `dynamic` sau `JObject` pentru a transmite parametri. În schimb, definiți clase POCO C# explicite ca parametri pentru fiecare metodă RPC. Acest lucru nu numai că rezolvă problema recunoașterii țintelor proxy, dar prinde și erorile de tip la compilare.

Exemplu: înlocuiți proprietățile generice din `CallbackProxyTarget` cu clase concrete precum `ReadTextFileRequest` și `WriteTextFileRequest`.

### 2. Declarați explicit numele metodelor
Folosiți atributul `[JsonRpcMethod]` pentru a specifica explicit numele metodelor RPC în loc să vă bazați pe maparea implicită a numelor de metode. Acest lucru previne eșecurile de invocare cauzate de diferențe de stil de denumire precum PascalCase versus camelCase.

```csharp
public interface IAcpAgentCallback
{
    [JsonRpcMethod("readTextFile")]
    Task<ReadTextFileResponse> ReadTextFileAsync(ReadTextFileRequest request);
    
    [JsonRpcMethod("writeTextFile")]
    Task WriteTextFileAsync(WriteTextFileRequest request);
}
```

### 3. Profitati de callback-urile de stare a conexiunii
StreamJsonRpc oferă evenimentul `JsonRpc.ConnectionLost`. Ar trebui absolut să ascultați acest eveniment pentru a gestiona ieșirile neașteptate ale proceselor sau deconectările de rețea, ceea ce este mai oportun decât să vă bazați doar pe detectarea eșecurilor Orleans Grain.

```csharp
_rpc.ConnectionLost += (sender, e) =>
{
    _logger.LogError("RPC connection lost: {Reason}", e.ToString());
    // Handle reconnection logic or notify the user
};
```

### 4. Înregistrarea jurnalelor pe straturi
- **Nivel Trace**: Înregistrați payload-ul complet de cerere/răspuns JSON brut.
- **Nivel Debug**: Înregistrați stivele de apeluri de metodă și rezumate de parametri.
- **Notă**: Asigurați-vă că jurnalele nu includ token-uri de autorizare sensibile sau conținut de fișiere mari codat Base64.

### 5. Gestionarea specificului transportului de streaming
StreamJsonRpc suportă nativ `IAsyncEnumerable`. La implementarea răspunsurilor de prompt-uri streaming pentru ACP, folosiți `IAsyncEnumerable` direct în loc de logică de paginare personalizată. Acest lucru poate simplifica considerabil cantitatea de cod necesară pentru procesarea streaming-ului.

```csharp
public interface IAcpAgentCallback
{
    [JsonRpcMethod("streamText")]
    IAsyncEnumerable<string> StreamTextAsync(StreamTextRequest request);
}
```

### 6. Modelul adaptor
Mențineți `ACPSession` și `ClientSideConnection` separate. `ACPSession` ar trebui să se concentreze pe managementul stării Orleans și logica de business, cum ar fi încadrarea mesajelor în coadă, și ar trebui să folosească obiectul de conexiune StreamJsonRpc prin compoziție mai degrabă decât prin moștenire.

## Concluzie

Prin integrarea comprehensivă a StreamJsonRpc, proiectul HagiCode a abordat cu succes costul ridicat de întreținere, limitările funcționale și confuzia de stratificare arhitecturală ale implementării personalizate originale. Îmbunătățirile cheie includ:

1. Înlocuirea proprietăților dinamice cu DTO-uri cu tipizare puternică, îmbunătățind mentenabilitatea și fiabilitatea
2. Implementarea abstractizării stratului de transport și separării stratului de protocol, îmbunătățind claritatea arhitecturală
3. Consolidarea capabilităților de logare pentru a face problemele de comunicare mai ușor de diagnosticat
4. Introducerea suportului de streaming pentru a simplifica implementarea streaming-ului

Aceste îmbunătățiri oferă HagiCode o fundație de comunicare mai stabilă și mai eficientă, permițându-i să interacționeze mai bine cu instrumentele AI externe și stabilind o bază solidă pentru extinderea viitoare a funcționalităților.
