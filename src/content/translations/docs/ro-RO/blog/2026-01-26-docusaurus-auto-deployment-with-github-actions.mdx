## Adăugarea suportului pentru implementare automată GitHub Pages la HagiCode

> Acest proiect a fost inițial codificat PCode și acum a fost redenumit oficial HagiCode. Acest articol documentează cum am introdus implementarea automată a site-urilor statice, astfel încât publicarea conținutului devine la fel de ușoară ca a bea apă.

## Context / Introducere

În timpul dezvoltării HagiCode, am întâmpinat o problemă foarte practică: pe măsură ce documentația și propunerile noastre creșteau continuu, gestionarea și prezentarea eficientă a acelui conținut a devenit din ce în ce mai urgentă. Am decis să folosim GitHub Pages pentru a găzdui site-ul nostru static, dar construirea și implementarea manuală era pur și simplu prea greoaie. Fiecare modificare necesita o construcție locală, ambalare și apoi un push manual pe ramura `gh-pages`. Asta nu era doar ineficient, ci și ușor de greșit.

Pentru a rezolva această problemă, în principal pentru că voiam să ne economisim efortul, aveam nevoie de un flux de lucru de implementare automată. Acest articol înregistrează în detaliu cum am adăugat suport pentru implementare automată bazată pe GitHub Actions la proiectul HagiCode, astfel încât să ne putem concentra pe scrierea conținutului și să lăsăm automatizarea să se ocupe de restul.

## Analiza obiectivelor

Înainte de a trece la implementare, trebuie mai întâi să clarificăm ce ar trebui să realizeze efectiv această sarcină. La urma urmei, ascuțirea topoarei nu întârzie tăierea lemnelor.

### Cerințe de bază

1.  **Construcție automată**: Declanșarea automată a fluxului de lucru de construcție când codul este push-uit pe ramura `main`.
2.  **Implementare automată**: După o construcție reușită, implementarea automată a fișierelor statice generate pe GitHub Pages.
3.  **Consecvența mediului**: Asigurarea că mediul CI corespunde mediului de construcție local pentru a evita situația clasică de „funcționează local, dar eșuează online."

### Alegere tehnică

Deoarece HagiCode este construit pe Docusaurus, un generator de site-uri statice React foarte popular, putem folosi GitHub Actions pentru a atinge acest obiectiv.

## Configurarea fluxului de lucru GitHub Actions

GitHub Actions este serviciul CI/CD oferit de GitHub. Prin definirea fișierelor de flux de lucru în format YAML în interiorul depozitului, putem personaliza tot felul de sarcini de automatizare.

### Crearea fișierului de flux de lucru

Trebuie să creăm un nou fișier de configurare în directorul `.github/workflows` la rădăcina proiectului, de exemplu `deploy.yml`. Dacă directorul nu există încă, creați-l mai întâi.

Logica de bază a acestui fișier de configurare este următoarea:

1.  **Condiție de declanșare**: Ascultare evenimente `push` pe ramura `main`.
2.  **Mediu de execuție**: Cea mai recentă versiune Ubuntu.
3.  **Pași de construcție**:
    -   Checkout codul
    -   Instalare Node.js
    -   Instalare dependențe (`npm install`)
    -   Construirea fișierelor statice (`npm run build`)
4.  **Pasul de implementare**: Utilizarea acțiunii oficiale `action-gh-pages` pentru a push-ui rezultatul construcției pe ramura `gh-pages`.

### Cod de configurare cheie

Mai jos este șablonul de configurare pe care l-am adoptat în final:

```yaml
name: Deploy to GitHub Pages

# Trigger condition: run when code is pushed to the main branch
on:
  push:
    branches:
      - main
    # Add path filters if needed, for example only build when docs change
    # paths:
    #   - 'docs/**'
    #   - 'package.json'

# Set permissions, which is important for GitHub Pages deployment
permissions:
  contents: read
  pages: write
  id-token: write

# Concurrency control: cancel older builds for the same branch
concurrency:
  group: "pages"
  cancel-in-progress: false

jobs:
  build:
    runs-on: ubuntu-latest
    steps:
      - name: Checkout
        uses: actions/checkout@v4
        # Note: fetch-depth: 0 is required, otherwise the build version may be inaccurate
        with:
          fetch-depth: 0

      - name: Setup Node
        uses: actions/setup-node@v4
        with:
          node-version: 20 # Keep this consistent with the local development environment
          cache: 'npm'     # Enabling cache can speed up the build process

      - name: Install dependencies
        run: npm ci
        # Use npm ci instead of npm install because it is faster and stricter, which makes it better for CI

      - name: Build website
        run: npm run build
        env:
          # Configure environment variables here if your site build requires them
          # NODE_ENV: production
          # PUBLIC_URL: /your-repo-name
          
      - name: Upload artifact
        uses: actions/upload-pages-artifact@v3
        with:
          path: ./build # Default Docusaurus output directory

  deploy:
    environment:
      name: github-pages
      url: ${{ steps.deployment.outputs.page_url }}
    runs-on: ubuntu-latest
    needs: build
    steps:
      - name: Deploy to GitHub Pages
        id: deployment
        uses: actions/deploy-pages@v4
```

## Capcane comune în timpul implementării

În practică, am întâmpinat mai multe probleme. Le împărtășesc aici în speranța că alții le pot evita sau cel puțin pregăti soluții în avans.

### 1. Probleme de permisiuni GitHub Token

La prima configurare, implementarea eșua continuu cu un 403 (Forbidden). După multe investigații, am descoperit că `GITHUB_TOKEN` implicit de la GitHub nu avea permisiunea de a scrie pe Pages.

**Soluție**: În `Settings` -> `Actions` -> `General` -> `Workflow permissions` ale depozitului, asigurați-vă că selectați **„Read and write permissions"**.

### 2. Cale greșită a directorului de construcție

Docusaurus plasează fișierele statice construite în directorul `build` implicit. Totuși, unele proiecte pot fi configurate diferit. De exemplu, Create React App folosește implicit `build`, în timp ce Vite folosește implicit `dist`. Dacă Actions raportează că nu găsește fișierele de ieșire, verificați calea de ieșire în `docusaurus.config.js`.

### 3. Probleme de subpath

Dacă depozitul dvs. nu este o pagină de utilizator, adică nu este `username.github.io`, ci o pagină de proiect cum ar fi `username.github.io/project-name`, atunci trebuie să configurați `baseUrl`.

În `docusaurus.config.js`:

```javascript
module.exports = {
  // ...
  url: 'https://hagicode.com', // Your Hagicode URL
  baseUrl: '/', // Deploy from the root path
  // ...
};
```
Acest detaliu este ușor de trecut cu vederea. Dacă este configurat incorect, pagina se poate încărca ca ecran alb deoarece căile resurselor nu pot fi rezolvate.

## Verificarea rezultatului

După configurarea tuturor și push-uirea codului, putem merge la tab-ul **Actions** al depozitului GitHub și să urmărim spectacolul.

Veți vedea un cerc galben în timp ce fluxul de lucru rulează, iar când devine verde, înseamnă succes. Dacă devine roșu, faceți clic și inspectați jurnalele. În majoritatea cazurilor, puteți urmări problema acolo, de obicei o greșeală de tastare sau o configurare incorectă a căii.

Odată ce construcția reușește, vizitați `https://<your-username>.github.io/<repository-name>/` și veți vedea site-ul proaspăt implementat.

## Rezumat

Prin introducerea GitHub Actions, am automatizat cu succes implementarea site-ului de documentație HagiCode. Asta nu numai că economisește timpul petrecut anterior pe operațiuni manuale, dar mai ales standardizează fluxul de lucru de lansare. Acum, indiferent care membru al echipei actualizează documentația, odată ce modificarea este fuzionată în `main`, cel mai recent conținut va fi live în câteva minute.

**Beneficii de bază**:
- **Eficiență mai mare**: Am trecut de la „ambalare manuală și încărcare manuală" la „cod înseamnă lansare."
- **Mai puține greșeli**: Am eliminat posibilitatea erorii umane în timpul operațiunilor manuale.
- **Experiență mai bună pentru dezvoltatori**: Dezvoltatorii se pot concentra mai mult pe calitatea conținutului în loc să fie distrași de pași de implementare tedioși.

Deși configurarea CI/CD este oarecum dificilă la început, mai ales cu toate detaliile de permisiuni și căi, este o investiție unică cu randamente majore pe termen lung. Recomand cu tărie ca fiecare proiect de site static să adopte un flux de lucru automat similar.
