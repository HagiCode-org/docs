## Prezentare generală

Monospecs este abordarea HagiCode de gestionare a mai multor depozite. Urmărește și gestionează sub-depozitele prin intermediul `monospecs.yaml`, și funcționează pentru orice proiect care are nevoie de un singur loc pentru a coordona mai multe depozite independente.

### Concepte de bază

**Fișier de configurare**

`monospecs.yaml` este nucleul întregii soluții de management, definind:
- Lista și locația sub-depozitelor
- Adresa URL Git pentru fiecare depozit
- Numele afișat și pictograma
- Comportament de integrare cu OpenSpec

**Diferențe față de arhivele obișnuite**

| Caracteristică | Depozitul obișnuit | Proiect gestionat Monospecs |
|----------|-------------------|---------------------------|
| **Locație OpenSpec** | Directorul rădăcină `openspec/` | Directorul rădăcină al depozitului principal `openspec/` |
| **Schimbați domeniul de aplicare** | Limitat la depozitul curent | Poate implica mai multe sub-repozitive |
| **Puritatea sub-depozitarului** | Specificații amestecate cu cod | Specificațiile stabilite în depozitul principal, sub-depozitele sunt mai curate |
| **Arhivați auto-comitare** | Necesita comitere manuale ale specificațiilor | `commit_when_archive: true` comite automat specificațiile în depozitul principal |

### De ce să folosiți Monospecs?

- **Gestionare unificată**: gestionați toate sub-depozitele într-un singur fișier de configurare
- **Asistență pentru automatizare**: oferă metadate de depozit pentru scripturi și instrumente
- **Utilitatea AI**: configurația structurată ajută agentul AI să înțeleagă relațiile dintre depozit
- **Integrare OpenSpec**: Integrare perfectă cu fluxul de lucru de management al schimbărilor
- **Sub-depozite mai curate**: specificațiile sunt păstrate în depozitul principal, sub-depozitele conțin doar cod real

### Scenarii aplicabile

Utilizați mai întâi Monospecs în scenarii ca acestea:

**1. Dezvoltare colaborativă cu mai multe depozite**
Când un proiect este împărțit în mai multe depozite independente:
- Aplicația front-end, serviciul backend și aplicația desktop dezvoltate separat
- Trebuie să coordonați versiunile și dependențele între depozite
- Documentația și proiectele de ajutor separate de codul principal

**2. Management de proiect auxiliar**
Utilizați Monospecs pentru a gestiona:
- Proiect de documentare
- Site oficial
- Construiți instrumente și scripturi
- Fiecare depozit își menține propriul control al versiunii

**3. Dezvoltare de caracteristici între depozite**
O caracteristică care necesită modificări în mai multe depozite:
- Trebuie să urmăriți modificările în care depozite
- Trimiteți automat specificațiile pentru a corecta depozitul țintă
- Coordonează actualizările versiunii și dependențele

## Pornire rapidă

### Inițializare

Urmați acești pași pentru a finaliza inițializarea monospecilor:

#### 1. Creați un depozit git gol

Creați un gol `git` folderul din depozitul dvs. principal (notă: acesta este `git`, nu `repos/` - aceasta este convenția monospecilor):

```bash
mkdir git
cd git
git init
cd ..
```

#### 2. Configurați fișierul monospecs

Creați `monospecs.yaml` fișierul de configurare din directorul principal al depozitului:

```yaml
version: "1.0"
commit_when_archive: true

repositories:
  - path: "repos/frontend"
    url: "https://github.com/your-org/frontend.git"
    displayName: "Frontend App"
    icon: "🌐"

  - path: "repos/backend"
    url: "https://github.com/your-org/backend.git"
    displayName: "Backend Service"
    icon: "⚙️"
```

**Elemente de configurare**:
| Element de configurare | Tip | Necesar | Descriere |
|----------------|------|----------|-------------|
| `version` | şir | Da | Versiunea fișierului de configurare |
| `commit_when_archive` | boolean | Nu | Dacă se comite automat specificațiile la arhivare (nu se comite codul sub-repo) |
| `repositories` | matrice | Da | Lista de configurare a depozitului |
| `repositories[].path` | şir | Da | Calea depozitului (față de rădăcina proiectului) |
| `repositories[].url` | şir | Da | Adresa URL a depozitului Git |
| `repositories[].displayName` | şir | Nu | Numele afișat al depozitului |
| `repositories[].icon` | şir | Nu | Pictograma depozit (emoji) |

### Adăugați sub-arhive la depozitul principal

Adăugați toate sub-depozitele la `repos/` director. Acest director trebuie **exclus din controlul versiunii Git al depozitului principal** pentru a evita urmărirea conținutului subdepozitului:

```plaintext
# Exclude all sub-repositories
repos/
```

#### 3. Adăugați depozitul părinte la HagiCode

Dacă utilizați aplicația desktop HagiCode, deschideți aplicația și adăugați depozitul principal ca proiect. Faceți clic pe butonul „Adăugați proiect” sau „+” de pe ecranul de pornire, selectați folderul principal de depozit și veți vedea toate sub-depozitele și informațiile despre starea acestora.

#### 4. Clonare sub-arhive

Utilizați `monospecs.yaml` configurație pentru a clona toate sub-depozitele configurate în `repos/` director:

Aceasta va clona toate sub-depozitele pe baza URL-urilor specificate în fișierul de configurare.

## Structura fișierului de configurare

### Configurare la nivel de rădăcină

```yaml
version: "1.0"
commit_when_archive: true

repositories:
  # ... repository configurations
```

### Configurare la nivel de depozit

Fiecare sub-repozitiv poate avea:
- Dosarul de modificare OpenSpec (`openspec/`)
- Independent `.git/` director
- Opțional `AGENTS.md` pentru configurarea AI Agent

## Operațiuni de gestionare a depozitelor

### Adăugați un nou depozit

Când adăugați un nou sub-repozitiv la `monospecs.yaml` `repositories` matrice:

```yaml
repositories:
  - path: "repos/new-service"
    url: "https://github.com/HagiCode-org/new-service.git"
    displayName: "New Service"
    icon: "🆕"
```

### Actualizați configurația depozitului

Când adresele URL ale depozitului sau metadatele se modifică:
1. Edita `monospecs.yaml` pentru a actualiza intrările corespunzătoare
2. Verificați că sintaxa YAML este corectă
3. Dacă sincronizarea se modifică, actualizați manual configurația depozitului local

**Comportamente specifice proiectului**

### Beneficii AI

Folosind `AGENTS.md` permite agentului AI să:
- Înțelegeți relațiile și dependențele din depozit
- Generați cod și refactorizați în mai multe depozite
- Mențineți consecvența în controlul versiunilor și configurații

## Integrare OpenSpec

### Cum funcționează Monospecs cu OpenSpec

Indiferent dacă gestionați un proiect gestionat monospecs sau un depozit obișnuit, puteți utiliza fluxul de lucru OpenSpec al HagiCode:

**Documentul obișnuit:**
- OpenSpec în rădăcină `openspec/` director
- Modificările afectează numai depozitul curent

**Proiect gestionat Monospecs:**
- Citește OpenSpec `monospecs.yaml` configurație
- Modificări urmărite de sub-repozitiv
- Specificațiile sunt angajate automat pentru ținte corecte

### Comportamentul arhivării

Când `commit_when_archive: true`:
- Specificațiile sunt trimise automat în depozitul principal
- Codul sub-depozitelor nu este comis (rămâne separat)
- Simplifica gestionarea specificațiilor - nu sunt necesare comiteri manuale de specificații pentru sub-depozite

### Comite manuale sub-depozitare

Sub-depozitele mențin un control independent al versiunilor:
- Trimiteți modificările reale ale codului în propriile depozite
- Specificațiile sunt trimise separat (din depozitul principal la arhivare)

## Cele mai bune practici

### Convenții de denumire a depozitelor

- Folosiți kebab-case (litere mici și cratime) pentru căile de depozit
- Folosiți nume afișate simple și descriptive în chineză
- Folosiți emoji-urile relevante ca pictograme

### Instrucțiuni privind numele afișat

- Păstrați-l concis și descriptiv
- Utilizați o denumire consecventă în depozite similare
- Selectați pictogramele care reprezintă scopul depozitului

### Întreținerea configurației

**Sincronizare configurație cu starea reală**
- Verificați în mod regulat `monospecs.yaml` reflectă structura actuală a depozitului
- Actualizați sau eliminați intrările atunci când depozitele sunt adăugate sau eliminate
- Testați modificările pentru a asigura validitatea configurației

**Control versiuni pentru configurare**
- Urmăriți modificările fișierului de configurare cu git
- Documentați motivele modificărilor majore ale configurației
- Etichetați versiunile fișierelor de configurare pentru o derulare ușoară

## Depanare

### Fișierul de configurare nu a fost găsit

Dacă `monospecs.yaml` fișierul nu este găsit în rădăcina proiectului:
1. Verificați dacă fișierul există în locația corectă
2. Verificați că vă aflați în directorul de lucru corect
3. Verificați greșelile de scriere în numele fișierului de configurare

### Erori de sintaxă YAML

Probleme comune de sintaxă YAML:
- Indentare incorectă (utilizați spații, nu file)
- Ghilimele lipsă sau suplimentare în jurul șirurilor de caractere
- Tipuri de date nevalide (șirurile necesită ghilimele, numerele nu)
- Lipsesc câmpuri obligatorii (cum ar fi `path` sau `url`)

### Depozitul nu a fost detectat

Dacă un depozit nou adăugat nu apare:
1. Verificați `path` este corect în `monospecs.yaml`
2. Verificați că depozitul a fost clonat cu succes
3. Verificați dacă depozitul există în locația așteptată

### Eșecuri ale clonelor

Dacă clonarea depozitului eșuează:
1. Verificați conexiunea la rețea
2. Verificați că adresa URL a depozitului Git este corectă
3. Verificați dacă există probleme de autentificare
4. Examinați starea containerului Docker
5. Verificați disponibilitatea spațiului pe disc
