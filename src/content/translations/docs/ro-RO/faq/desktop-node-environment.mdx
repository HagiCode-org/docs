HagiCode Desktop este livrat cu un lanț de instrumente Node gestionat de desktop. Nu preia instalarea globală Node pe computer. Scopul său este de a oferi pornirii desktopului și fluxurilor legate de runtime o bază stabilă și verificabilă.

Citiți aceste întrebări frecvente dacă vedeți `portable-fixed/toolchain` în diagnosticare, trebuie să știi care `node` Desktop-ul folosește sau mă întreb de ce versiunea nodului dvs. de sistem nu definește în mod direct comportamentul de rulare a desktopului.

## De ce Desktop bundle Node

Desktopul trebuie să ruleze aceleași fluxuri de asistență între sistemele de operare, canalele de instalare și mediile de utilizator. Dacă ar depinde doar de instalarea Node existentă a utilizatorului, suportul ar trebui să gestioneze simultan versiunile învechite, npm lipsă, ordonarea incorectă a căilor, pachetele globale poluate și politicile de registru specifice organizației.

Desktop descrie acest lucru mai precis ca un timp de execuție Node/npm izolat pentru CLI gestionate, în loc să reutilizeze mediul la nivelul întregului sistem. Acest design are următoarele avantaje principale:

- **Versiuni consistente**: Desktop utilizează un contract de rulare Node 22 fixat în loc să ghicească versiunea majoră locală a utilizatorului.
- **Fluxuri de pornire controlate**: comenzile gestionate de desktop pot prefera cele împachetate `node` şi `npm` intrări.
- **Ștergeți limitele configurației**: configurația legată de CLI este păstrată separat de configurația globală la nivel de gazdă, astfel încât actualizările, comutarea și recuperarea sunt mai puțin probabil să fie perturbate de diferențele de mediu ale sistemului.
- **Conflicte reduse**: căile grupate se aplică mediilor de comandă create de Desktop și nu înlocuiesc nodul global al utilizatorului în terminalul lor.
- **Comportament non-intruziv al gazdei**: această izolare evită scrierea stării lanțului de instrumente în mediul existent Node/npm al utilizatorului, iar dezinstalarea Desktop nu deranjează Nodul, npm sau configurația deja prezentă pe computer.

## Cum selectează Desktop calea Nodului

Când rulați Desktop în mod normal, Desktop preferă lanțul de instrumente Node instalat cu aplicația în loc să citească mai întâi instalarea Node din sistemul dvs. Acest lucru păstrează versiunea Node din terminalul dvs. separată de versiunea Node pe care Desktop o așteaptă în timp ce rulează.

Când lanțul de instrumente grupat este activat, mediile de comandă create de Desktop pun căile lanțului de instrumente grupate în fața `PATH`. În consecință, chiar dacă sistemul dvs. are instalată o altă versiune Node, fluxurile gestionate de Desktop se rezolvă mai întâi la Node și npm incluse în Desktop.

Acest lucru afectează numai mediile de rulare create de Desktop. Dacă fugi `node`, `npm`, sau alte comenzi direct în terminalul de sistem, acestea continuă să utilizeze mediul de sistem normal și configurația shell.

În mediile de diagnosticare sau de proces copil începute de Desktop, este posibil să vedeți o variabilă ca aceasta:

```bash
HAGICODE_PORTABLE_TOOLCHAIN_ROOT=/path/to/portable-fixed/toolchain
```

Această variabilă înseamnă că mediul de rulare gestionat de desktop a recunoscut rădăcina lanțului de instrumente inclus. Vederea acestei căi nu înseamnă că Desktop v-a schimbat Nodul sistemului; înseamnă doar că Desktop utilizează mediul Node livrat cu aplicația în propriul său domeniu de aplicare.

## Unde Desktop instalează pachete gestionate npm

Desktop păstrează fișierele de rulare și fișierele de pachet npm mutabile în diferite locații. Calea pachetului de instrumente, cum ar fi `portable-fixed/toolchain/node` este sursa imuabilă pentru desktop-ul gestionat `node` şi `npm` comenzi. Pachetele globale gestionate sunt instalate în directorul de date utilizator Electron:

```text
userData/node22/npmGlobal
```

Dacă Desktop se mută ulterior la Nodul 24, acesta folosește un arbore de pachete separat, cum ar fi `userData/node24/npmGlobal`. Pachetele instalate pentru Nodul 22 rămân pe disc, dar nu sunt tratate ca instalate pentru Nodul 24 până când nu sunt reinstalate în noul prefix Node-major.

Starea dependenței de desktop verifică doar cea activă `userData/node<major>/npmGlobal` prefix. Fișierele pachetelor din afara acestui prefix sunt ignorate de managementul dependențelor.
