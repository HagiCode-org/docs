# Introducere în OpenSpec

OpenSpec este un instrument de dezvoltare bazat pe specificații care ajută echipele să gestioneze procesul de dezvoltare software prin propuneri și specificații structurate. Acest articol oferă o introducere cuprinzătoare în conceptele de bază, fluxurile de lucru și metodele de utilizare ale OpenSpec.

## Ce este OpenSpec?

OpenSpec este un instrument de linie de comandă utilizat pentru gestionarea propunerilor (Proposals), specificațiilor (Specifications) și modificărilor (Changes) în procesul de dezvoltare software. Acesta standardizează procesul de dezvoltare și asigură că fiecare funcționalitate trece printr-un design și o revizuire amănunțite, de la idee până la implementare.

### Principii fundamentale

OpenSpec se bazează pe următoarele principii fundamentale:

- **Specificații mai întâi**: Definiți specificații clare înainte de a scrie cod
- **Colaborare structurată**: Promovați colaborarea în echipă prin formate de documentație standardizate
- **Trasabilitate**: Înregistrați întregul proces de la propunere până la implementare
- **Asigurarea calității**: Asigurați-vă că codul corespunde intenției de design prin validarea specificațiilor

### Funcționalități principale

OpenSpec oferă următoarele funcționalități principale:

1. **Gestionarea propunerilor**: Creați, revizuiți și urmăriți propunerile de funcționalități
2. **Definirea specificațiilor**: Scrieți și mențineți documente de specificații tehnice
3. **Urmărirea modificărilor**: Înregistrați întregul istoric al modificărilor revizuite
4. **Validarea specificațiilor**: Asigurați-vă că modificările de cod respectă specificațiile predefinite
5. **Colaborarea în echipă**: Susțineți procesele de dezvoltare și revizuire cu mai mulți participanți

## Fluxul de lucru OpenSpec

Fluxul de lucru tipic OpenSpec include următoarele etape:

### 1. Crearea unei propuneri (Proposal)

Când există o funcționalitate nouă sau o cerință de modificare semnificativă, creați mai întâi o propunere:

```bash
openspec proposal create
```

Propunerea include:
- **Descrierea problemei**: Problema sau cerința care trebuie rezolvată
- **Soluția**: Prezentarea generală a soluției propuse
- **Domeniul de impact**: Impactul asupra sistemului existent
- **Soluții alternative**: Alte soluții luate în considerare

### 2. Scrierea specificației (Specification)

După aprobarea propunerii, scrieți o specificație tehnică detaliată:

```bash
openspec spec create
```

Specificația include:
- **Design tehnic**: Plan detaliat pentru implementarea tehnică
- **Definirea interfețelor**: Design-ul interfețelor precum API-uri, structuri de date etc.
- **Detalii de implementare**: Logica cheie de implementare și algoritmii
- **Plan de testare**: Cum să verificați corectitudinea implementării

### 3. Crearea unei modificări (Change)

Când începeți implementarea, creați o modificare pentru a urmări procesul de dezvoltare:

```bash
openspec change create
```

Modificarea include:
- **Specificația asociată**: Specificația implementată de această modificare
- **Sarcini de implementare**: Lista sarcinilor concrete de dezvoltare
- **Urmărirea progresului**: Starea de finalizare a sarcinilor
- **Modificări de cod**: Modificările efective ale codului

### 4. Validare și arhivare

După finalizarea dezvoltării, validați dacă modificarea respectă specificația, apoi arhivați-o:

```bash
openspec change validate
openspec change archive
```

## Structura directoarelor OpenSpec

OpenSpec creează un director `openspec/` în directorul rădăcină al proiectului cu următoarea structură:

```
openspec/
├── proposals/           # Director propuneri
│   ├── PROPOSAL-001.md
│   └── PROPOSAL-002.md
├── specs/              # Director specificații
│   ├── SPEC-001.md
│   └── SPEC-002.md
├── changes/            # Director modificări
│   ├── feature-name/
│   │   ├── proposal.md
│   │   ├── design.md
│   │   ├── specs.md
│   │   └── tasks.md
│   └── another-feature/
└── archived/           # Modificări arhivate
    └── feature-name/
```

### Explicarea directoarelor

- **proposals/**: Stochează toate documentele de propuneri, fiecare propunere într-un fișier Markdown separat
- **specs/**: Stochează documentele de specificații tehnice corespunzătoare propunerilor
- **changes/**: Stochează modificările în curs, fiecare modificare într-un subdirector
- **archived/**: Stochează modificările finalizate și arhivate

## Comenzi principale

### Comenzi legate de propuneri

```bash
# Creați o propunere nouă
openspec proposal create

# Listați toate propunerile
openspec proposal list

# Vizualizați detaliile propunerii
openspec proposal view <proposal-id>

# Aprobați propunerea
openspec proposal approve <proposal-id>
```

### Comenzi legate de specificații

```bash
# Creați o specificație nouă
openspec spec create

# Listați toate specificațiile
openspec spec list

# Vizualizați detaliile specificației
openspec spec view <spec-id>

# Actualizați specificația
openspec spec update <spec-id>
```

### Comenzi legate de modificări

```bash
# Creați o modificare nouă
openspec change create

# Listați toate modificările
openspec change list

# Vizualizați detaliile modificării
openspec change view <change-name>

# Adăugați o sarcină
openspec change add-task <change-name>

# Validați modificarea
openspec change validate <change-name>

# Arhivați modificarea
openspec change archive <change-name>
```

### Comenzi generale

```bash
# Vizualizați versiunea
openspec --version

# Vizualizați ajutorul
openspec --help

# Vizualizați ajutorul pentru o comandă specifică
openspec <command> --help
```

## Utilizarea OpenSpec în proiectul HagiCode

### Integrarea proiectului

În proiectul HagiCode, OpenSpec este integrat în fluxul de lucru de dezvoltare:

1. **Verificare la inițializare**: Asistentul de pornire verifică dacă OpenSpec este instalat
2. **Dezvoltare bazată pe propuneri**: Toate funcționalitățile noi trebuie să treacă prin procesul de propuneri OpenSpec
3. **Revizuirea codului**: Modificările trebuie să respecte specificațiile OpenSpec pentru a putea fi integrate
4. **Sincronizarea documentației**: Documentația OpenSpec este menținută sincronizată cu implementarea codului

### Cerințe de versiune

Proiectul HagiCode necesită utilizarea versiunii **OpenSpec >=1.0.0 <2.0.0**.

Comandă de instalare:

```bash
npm install -g @fission-ai/openspec@1
```

### Exemplu de flux de lucru

Iată un exemplu complet de utilizare a OpenSpec în proiectul HagiCode:

#### Pasul 1: Creați propunerea

```bash
cd /path/to/hagicode-project
openspec proposal create
```

Introduceți în promptul interactiv:
- Titlul propunerii: Adăugarea funcționalității de autentificare a utilizatorilor
- Descrierea problemei: Sistemul necesită autentificare utilizatori și gestionarea permisiunilor
- Soluția: Implementarea unui sistem de autentificare bazat pe JWT

#### Pasul 2: Scrieți specificația

După aprobarea propunerii, creați o specificație detaliată:

```bash
openspec spec create
```

Definiți în specificație:
- Design-ul fluxului de autentificare
- Definirea interfeței API
- Structura tabelelor din baza de date
- Considerații de securitate

#### Pasul 3: Creați modificarea

Începeți implementarea:

```bash
openspec change create
```

Introduceți numele modificării: `add-user-authentication`

#### Pasul 4: Adăugați sarcini

```bash
openspec change add-task add-user-authentication
```

Adăugați sarcini concrete:
- Implementați endpoint-ul de înregistrare utilizatori
- Implementați endpoint-ul de autentificare
- Implementați middleware-ul JWT
- Scrieți teste unitare

#### Pasul 5: Implementare și validare

După finalizarea codului, validați modificarea:

```bash
openspec change validate add-user-authentication
```

#### Pasul 6: Arhivare

După validarea cu succes, arhivați modificarea:

```bash
openspec change archive add-user-authentication
```

## Cele mai bune practici

### Scrierea propunerilor

1. **Declarație clară a problemei**: Definiți clar problema de rezolvat; evitați descrierile vagi
2. **Luați în considerare alternative**: Enumerați și comparați diferite soluții
3. **Evaluați impactul**: Analizați impactul și riscurile pentru sistemul existent
4. **Fiți concis**: Propunerile ar trebui să fie scurte și clare; lăsați detaliile pentru specificație

### Scrierea specificațiilor

1. **Detaliat, dar nu redundant**: Oferiți suficiente detalii, dar evitați supraproiectarea
2. **Utilizați diagrame**: Folosiți diagrame de flux, diagrame de arhitectură etc. pentru a ajuta la explicare
3. **Definiți interfețele**: Definiți clar toate interfețele publice și structurile de date
4. **Luați în considerare cazurile limită**: Enumerați situațiile excepționale și gestionarea erorilor

### Gestionarea modificărilor

1. **Iterații mici**: Împărțiți funcționalitățile mari în mai multe modificări mici
2. **Actualizați la timp**: Mențineți starea sarcinilor și documentația sincronizate
3. **Revizuirea codului**: Efectuați revizuirea codului după finalizarea modificării
4. **Arhivați la timp**: Arhivați modificările finalizate la timp și mențineți directorul de lucru curat

### Colaborarea în echipă

1. **Revizuiri regulate**: Revizuiți regulat propunerile și specificațiile pentru a asigura consensul în echipă
2. **Documentație mai întâi**: Finalizați documentația înainte de a începe codificarea
3. **Mențineți comunicarea**: Înregistrați discuțiile și deciziile în propuneri și specificații
4. **Control de versiune**: Includeți documentația OpenSpec în controlul versiunilor

## Scenarii comune de utilizare

### Scenariul 1: Adăugarea unei funcționalități noi

1. Creați o propunere care descrie cerințele noii funcționalități
2. Echipa discută și aprobă propunerea
3. Scrieți o specificație tehnică detaliată
4. Creați o modificare și implementați funcționalitatea
5. Validați și arhivați modificarea

### Scenariul 2: Refactorizarea codului existent

1. Creați o propunere care explică motivele și obiectivele refactorizării
2. Definiți planul și pașii de refactorizare în specificație
3. Creați o modificare și refactorizați treptat
4. Asigurați acoperirea cu teste și validați rezultatele refactorizării
5. Arhivați modificarea

### Scenariul 3: Corectarea bug-urilor

Pentru bug-uri simple:
- Pot fi corectate direct, fără procesul complet OpenSpec

Pentru bug-uri complexe:
1. Creați o propunere pentru a analiza cauza principală a bug-ului
2. Definiți soluția de corectare în specificație
3. Creați o modificare și implementați corectarea
4. Adăugați teste pentru a preveni regresiile
5. Arhivați modificarea

### Scenariul 4: Gestionarea datoriei tehnice

1. Creați o propunere care enumeră elementele datoriei tehnice
2. Definiți un plan de îmbunătățire în specificație
3. Creați o modificare pentru a plăti treptat datoria tehnică
4. Urmăriți progresul și efectul
5. Arhivați îmbunătățirile finalizate

## Integrare cu alte instrumente

### Integrare cu Git

OpenSpec se integrează fără probleme cu Git:

```bash
# Documentația OpenSpec ar trebui inclusă în controlul versiunilor
git add openspec/
git commit -m "Add OpenSpec proposal for feature X"
```

### Integrare cu CI/CD

Validați specificațiile OpenSpec în procesul CI/CD:

```yaml
# .github/workflows/openspec.yml
name: OpenSpec Validation
on: [pull_request]
jobs:
  validate:
    runs-on: ubuntu-latest
    steps:
      - uses: actions/checkout@v2
      - name: Install OpenSpec
        run: npm install -g @fission-ai/openspec@1
      - name: Validate changes
        run: openspec change validate
```

### Integrare cu IDE

Se recomandă utilizarea unui editor care suportă Markdown pentru editarea documentelor OpenSpec:
- VS Code + plugin Markdown
- JetBrains IDEs + suport Markdown
- Vim/Neovim + plugin Markdown

## Depanare

### Probleme comune

#### Problemă: Comanda OpenSpec nu a fost găsită

**Soluție**:
1. Confirmați că OpenSpec este instalat: `npm list -g @fission-ai/openspec`
2. Verificați dacă variabila de mediu PATH include directorul bin global npm
3. Reinstalați: `npm install -g @fission-ai/openspec@1`

#### Problemă: Nu se poate crea propunerea sau modificarea

**Soluție**:
1. Confirmați că directorul curent este directorul rădăcină al proiectului
2. Verificați dacă există directorul `openspec/`; dacă nu, creați-l
3. Confirmați că aveți permisiuni de scriere

#### Problemă: Validarea a eșuat

**Soluție**:
1. Verificați dacă documentația din directorul de modificări este completă
2. Confirmați că toate sarcinile necesare au fost finalizate
3. Examinați informațiile detaliate despre eroare: `openspec change validate <change-name> --verbose`

#### Problemă: Arhivarea a eșuat

**Soluție**:
1. Asigurați-vă că modificarea a trecut validarea
2. Verificați dacă directorul `openspec/archived/` există
3. Confirmați că nu există modificări de cod necomise

## Utilizare avansată

### Șabloane personalizate

OpenSpec suportă șabloane personalizate pentru propuneri și specificații:

```bash
# Creați un director de șabloane în directorul rădăcină al proiectului
mkdir -p .openspec/templates

# Creați un șablon personalizat de propunere
cat > .openspec/templates/proposal.md << 'EOF'
# Proposal: {{title}}

## Problem Statement
{{problem}}

## Proposed Solution
{{solution}}

## Custom Section
Your custom content here
EOF
```

### Fișier de configurare

Creați un fișier de configurare `.openspec.json` în directorul rădăcină al proiectului:

```json
{
  "version": "1.0",
  "templates": {
    "proposal": ".openspec/templates/proposal.md",
    "spec": ".openspec/templates/spec.md"
  },
  "validation": {
    "requireTests": true,
    "requireDocs": true
  },
  "hooks": {
    "pre-archive": "npm test"
  }
}
```

### Hook-uri

OpenSpec suportă executarea de scripturi personalizate înainte și după operații specifice:

```json
{
  "hooks": {
    "pre-create": "scripts/check-permissions.sh",
    "post-archive": "scripts/notify-team.sh"
  }
}
```
