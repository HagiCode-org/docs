# Introduction to OpenSpec

OpenSpec is a specification-driven development tool that helps teams manage software development processes through structured proposals and specifications. This guide provides a comprehensive introduction to OpenSpec's core concepts, workflows, and usage methods.

## What is OpenSpec?

OpenSpec is a command-line tool for managing Proposals, Specifications, and Changes throughout the software development process. It standardizes the development workflow, ensuring that every feature goes through thorough design and review from idea to implementation.

### Core Philosophy

OpenSpec is built on the following core principles:

- **Specification-First**: Define clear specifications before writing code
- **Structured Collaboration**: Promote team collaboration through standardized document formats
- **Traceability**: Record the complete process from proposal to implementation
- **Quality Assurance**: Ensure code conforms to design intent through specification validation

### Key Features

OpenSpec provides the following key features:

1. **Proposal Management**: Create, review, and track feature proposals
2. **Specification Definition**: Write and maintain technical specification documents
3. **Change Tracking**: Record all reviewed change history
4. **Specification Validation**: Ensure code changes comply with predefined specifications
5. **Team Collaboration**: Support multi-person collaborative development and review processes

## OpenSpec Workflow

The typical OpenSpec workflow includes the following stages:

### 1. Create a Proposal

When there's a new feature or major change requirement, first create a proposal:

```bash
openspec proposal create
```

A proposal contains:
- **Problem Description**: The problem or requirement to be solved
- **Solution**: Overview of the proposed solution
- **Impact Scope**: Impact on existing systems
- **Alternatives**: Other solutions considered

### 2. Write a Specification

After the proposal is approved, write a detailed technical specification:

```bash
openspec spec create
```

A specification contains:
- **Technical Design**: Detailed technical implementation plan
- **Interface Definition**: API, data structure, and other interface designs
- **Implementation Details**: Key implementation logic and algorithms
- **Test Plan**: How to verify the correctness of the implementation

### 3. Create a Change

When starting implementation, create a change to track the development process:

```bash
openspec change create
```

A change contains:
- **Associated Specification**: The specification this change implements
- **Implementation Tasks**: Specific development task list
- **Progress Tracking**: Task completion status
- **Code Changes**: Actual code modifications

### 4. Validate and Archive

After completing development, validate whether the change conforms to the specification, then archive:

```bash
openspec change validate
openspec change archive
```

## OpenSpec Directory Structure

OpenSpec creates an `openspec/` directory in the project root with the following structure:

```
openspec/
├── proposals/           # Proposals directory
│   ├── PROPOSAL-001.md
│   └── PROPOSAL-002.md
├── specs/              # Specifications directory
│   ├── SPEC-001.md
│   └── SPEC-002.md
├── changes/            # Changes directory
│   ├── feature-name/
│   │   ├── proposal.md
│   │   ├── design.md
│   │   ├── specs.md
│   │   └── tasks.md
│   └── another-feature/
└── archived/           # Archived changes
    └── feature-name/
```

### Directory Descriptions

- **proposals/**: Stores all proposal documents, one independent Markdown file per proposal
- **specs/**: Stores technical specification documents, corresponding to proposals
- **changes/**: Stores ongoing changes, one subdirectory per change
- **archived/**: Stores completed and archived changes

## Core Commands

### Proposal Commands

```bash
# Create a new proposal
openspec proposal create

# List all proposals
openspec proposal list

# View proposal details
openspec proposal view <proposal-id>

# Approve a proposal
openspec proposal approve <proposal-id>
```

### Specification Commands

```bash
# Create a new specification
openspec spec create

# List all specifications
openspec spec list

# View specification details
openspec spec view <spec-id>

# Update a specification
openspec spec update <spec-id>
```

### Change Commands

```bash
# Create a new change
openspec change create

# List all changes
openspec change list

# View change details
openspec change view <change-name>

# Add a task
openspec change add-task <change-name>

# Validate a change
openspec change validate <change-name>

# Archive a change
openspec change archive <change-name>
```

### General Commands

```bash
# View version
openspec --version

# View help
openspec --help

# View help for a specific command
openspec <command> --help
```

## Using OpenSpec in HagiCode Projects

### Project Integration

In HagiCode projects, OpenSpec is integrated into the development workflow:

1. **Initialization Check**: The startup wizard checks whether OpenSpec is installed
2. **Proposal-Driven Development**: All new features must go through the OpenSpec proposal process
3. **Code Review**: Changes must comply with OpenSpec specifications to be merged
4. **Documentation Sync**: OpenSpec documentation stays synchronized with code implementation

### Version Requirements

HagiCode projects require **OpenSpec >=1.0.0 <2.0.0** version range.

Installation command:

```bash
npm install -g @fission-ai/openspec@1
```

### Workflow Example

Here's a complete example of using OpenSpec in a HagiCode project:

#### Step 1: Create a Proposal

```bash
cd /path/to/hagicode-project
openspec proposal create
```

Enter in the interactive prompt:
- Proposal title: Add user authentication feature
- Problem description: System needs user login and permission management
- Solution: Implement JWT-based authentication system

#### Step 2: Write a Specification

After the proposal is approved, create a detailed specification:

```bash
openspec spec create
```

Define in the specification:
- Authentication flow design
- API interface definition
- Database table structure
- Security considerations

#### Step 3: Create a Change

Start implementation:

```bash
openspec change create
```

Enter change name: `add-user-authentication`

#### Step 4: Add Tasks

```bash
openspec change add-task add-user-authentication
```

Add specific tasks:
- Implement user registration endpoint
- Implement login endpoint
- Implement JWT middleware
- Write unit tests

#### Step 5: Implement and Validate

After completing the code, validate the change:

```bash
openspec change validate add-user-authentication
```

#### Step 6: Archive

After validation passes, archive the change:

```bash
openspec change archive add-user-authentication
```

## Best Practices

### Writing Proposals

1. **Clear Problem Statement**: Clearly define the problem to be solved, avoid vague descriptions
2. **Consider Alternatives**: List and compare different solutions
3. **Assess Impact**: Analyze the impact and risks on existing systems
4. **Keep It Concise**: Proposals should be brief and to the point, leave details for specifications

### Writing Specifications

1. **Detailed but Not Redundant**: Provide sufficient detail, but avoid over-engineering
2. **Use Diagrams**: Use flowcharts, architecture diagrams, etc. to aid explanation
3. **Define Interfaces**: Clearly define all public interfaces and data structures
4. **Consider Edge Cases**: List exceptional situations and error handling

### Change Management

1. **Small Iterations**: Break large features into multiple small changes
2. **Timely Updates**: Keep task status and documentation synchronized
3. **Code Review**: Conduct code reviews after changes are complete
4. **Timely Archiving**: Archive completed changes promptly to keep the working directory clean

### Team Collaboration

1. **Regular Reviews**: Regularly review proposals and specifications to ensure team consensus
2. **Documentation First**: Complete documentation before starting to code
3. **Maintain Communication**: Record discussions and decisions in proposals and specifications
4. **Version Control**: Include OpenSpec documents in version control

## Common Use Cases

### Scenario 1: Adding New Features

1. Create a proposal describing the new feature requirements
2. Team discusses and approves the proposal
3. Write detailed technical specifications
4. Create a change and implement the feature
5. Validate and archive the change

### Scenario 2: Refactoring Existing Code

1. Create a proposal explaining the refactoring reasons and goals
2. Define the refactoring approach and steps in the specification
3. Create a change and refactor incrementally
4. Ensure test coverage, validate refactoring results
5. Archive the change

### Scenario 3: Fixing Bugs

For simple bugs:
- Can be fixed directly, no need for the complete OpenSpec process

For complex bugs:
1. Create a proposal analyzing the bug root cause
2. Define the fix approach in the specification
3. Create a change and implement the fix
4. Add tests to prevent regression
5. Archive the change

### Scenario 4: Technical Debt Management

1. Create a proposal listing technical debt items
2. Define an improvement plan in the specification
3. Create changes to gradually pay down technical debt
4. Track progress and effectiveness
5. Archive completed improvements

## Integration with Other Tools

### Git Integration

OpenSpec integrates seamlessly with Git:

```bash
# OpenSpec documents should be included in version control
git add openspec/
git commit -m "Add OpenSpec proposal for feature X"
```

### CI/CD Integration

Validate OpenSpec specifications in CI/CD pipelines:

```yaml
# .github/workflows/openspec.yml
name: OpenSpec Validation
on: [pull_request]
jobs:
  validate:
    runs-on: ubuntu-latest
    steps:
      - uses: actions/checkout@v2
      - name: Install OpenSpec
        run: npm install -g @fission-ai/openspec@1
      - name: Validate changes
        run: openspec change validate
```

### IDE Integration

Recommended editors with Markdown support for editing OpenSpec documents:
- VS Code + Markdown plugins
- JetBrains IDEs + Markdown support
- Vim/Neovim + Markdown plugins

## Troubleshooting

### Common Issues

#### Issue: OpenSpec command not found

**Solution**:
1. Confirm OpenSpec is installed: `npm list -g @fission-ai/openspec`
2. Check if PATH environment variable includes npm global bin directory
3. Reinstall: `npm install -g @fission-ai/openspec@1`

#### Issue: Cannot create proposals or changes

**Solution**:
1. Confirm current directory is the project root
2. Check if `openspec/` directory exists, create it if not
3. Confirm write permissions

#### Issue: Validation fails

**Solution**:
1. Check if documents in the change directory are complete
2. Confirm all required tasks are completed
3. View detailed error information: `openspec change validate <change-name> --verbose`

#### Issue: Archive fails

**Solution**:
1. Ensure the change has passed validation
2. Check if `openspec/archived/` directory exists
3. Confirm there are no uncommitted code changes

## Advanced Usage

### Custom Templates

OpenSpec supports custom proposal and specification templates:

```bash
# Create template directory in project root
mkdir -p .openspec/templates

# Create custom proposal template
cat > .openspec/templates/proposal.md << 'EOF'
# Proposal: {{title}}

## Problem Statement
{{problem}}

## Proposed Solution
{{solution}}

## Custom Section
Your custom content here
EOF
```

### Configuration File

Create a `.openspec.json` configuration file in the project root:

```json
{
  "version": "1.0",
  "templates": {
    "proposal": ".openspec/templates/proposal.md",
    "spec": ".openspec/templates/spec.md"
  },
  "validation": {
    "requireTests": true,
    "requireDocs": true
  },
  "hooks": {
    "pre-archive": "npm test"
  }
}
```

### Hooks

OpenSpec supports executing custom scripts before and after specific operations:

```json
{
  "hooks": {
    "pre-create": "scripts/check-permissions.sh",
    "post-archive": "scripts/notify-team.sh"
  }
}
```
