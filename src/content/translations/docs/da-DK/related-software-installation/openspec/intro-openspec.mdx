# Introduktion til OpenSpec

OpenSpec er et specifikationsdrevet udviklingsværktøj, der hjælper teams med at styre softwareudviklingsprocessen gennem strukturerede forslag og specifikationer. Denne artikel giver en omfattende introduktion til OpenSpecs kernekoncept, arbejdsgange og anvendelsesmetoder.

## Hvad er OpenSpec?

OpenSpec er et kommandolinjeværktøj, der bruges til at styre forslag (Proposals), specifikationer (Specifications) og ændringer (Changes) i softwareudviklingsprocessen. Det standardiserer udviklingsprocessen og sikrer, at hver funktion gennemgår grundig design og gennemgang fra idé til implementering.

### Kerneprincip

OpenSpec er baseret på følgende kerneprincip:

- **Specifikation først**: Definer klare specifikationer før du skriver kode
- **Struktureret samarbejde**: Fremme teamsamarbejde gennem standardiserede dokumentformater
- **Sporbarhed**: Registrer hele processen fra forslag til implementering
- **Kvalitetssikring**: Sikr at koden matcher designhensigten gennem specifikationsvalidering

### Hovedfunktioner

OpenSpec tilbyder følgende hovedfunktioner:

1. **Forslagsstyring**: Opret, gennemgå og spor funktionsforslag
2. **Specifikationsdefinition**: Skriv og vedligehold tekniske specifikationsdokumenter
3. **Ændringssporing**: Registrer hele historikken af gennemgåede ændringer
4. **Specifikationsvalidering**: Sikr at kodeændringer matcher foruddefinerede specifikationer
5. **Teamsamarbejde**: Understøt udviklings- og gennemgangsprocesser med flere deltagere

## OpenSpec arbejdsgang

Den typiske OpenSpec arbejdsgang omfatter følgende faser:

### 1. Opret forslag (Proposal)

Når der er en ny funktion eller krav om væsentlig ændring, skal du først oprette et forslag:

```bash
openspec proposal create
```

Forslaget omfatter:
- **Problembeskrivelse**: Problemet eller kravet, der skal løses
- **Løsning**: Oversigt over den foreslåede løsning
- **Påvirkningsomfang**: Påvirkning på det eksisterende system
- **Alternative løsninger**: Andre overvejede løsninger

### 2. Skriv specifikation (Specification)

Efter godkendelse af forslaget skal du skrive en detaljeret teknisk specifikation:

```bash
openspec spec create
```

Specifikationen omfatter:
- **Teknisk design**: Detaljeret teknisk implementeringsplan
- **Grænsefladedefinition**: Design af grænseflader som API, datastrukturer osv.
- **Implementeringsdetaljer**: Nøgleimplementeringslogik og algoritmer
- **Testplan**: Hvordan man verificerer implementeringens korrekthed

### 3. Opret ændring (Change)

Når du starter implementeringen, skal du oprette en ændring for at spore udviklingsprocessen:

```bash
openspec change create
```

Ændringen omfatter:
- **Tilknyttet specifikation**: Specifikationen implementeret af denne ændring
- **Implementeringsopgaver**: Liste over specifikke udviklingsopgaver
- **Fremgangssporing**: Status for opgavefuldførelse
- **Kodeændringer**: De faktiske kodemodifikationer

### 4. Validering og arkivering

Efter afslutning af udviklingen skal du validere, om ændringen matcher specifikationen, og derefter arkivere den:

```bash
openspec change validate
openspec change archive
```

## OpenSpec mappestruktur

OpenSpec opretter en `openspec/` mappe i projektets rodmappe med følgende struktur:

```
openspec/
├── proposals/           # Forslag mappe
│   ├── PROPOSAL-001.md
│   └── PROPOSAL-002.md
├── specs/              # Specifikations mappe
│   ├── SPEC-001.md
│   └── SPEC-002.md
├── changes/            # Ændrings mappe
│   ├── feature-name/
│   │   ├── proposal.md
│   │   ├── design.md
│   │   ├── specs.md
│   │   └── tasks.md
│   └── another-feature/
└── archived/           # Arkiverede ændringer
    └── feature-name/
```

### Mappebeskrivelse

- **proposals/**: Gemmer alle forslagsdokumenter, hvert forslag i en separat Markdown-fil
- **specs/**: Gemmer tekniske specifikationsdokumenter, der svarer til forslag
- **changes/**: Gemmer igangværende ændringer, hver ændring i en undermappe
- **archived/**: Gemmer afsluttede og arkiverede ændringer

## Kernekommandoer

### Forslag-relaterede kommandoer

```bash
# Opret nyt forslag
openspec proposal create

# Liste over alle forslag
openspec proposal list

# Vis forslagsdetaljer
openspec proposal view <proposal-id>

# Godkend forslag
openspec proposal approve <proposal-id>
```

### Specifikations-relaterede kommandoer

```bash
# Opret ny specifikation
openspec spec create

# Liste over alle specifikationer
openspec spec list

# Vis specifikationsdetaljer
openspec spec view <spec-id>

# Opdater specifikation
openspec spec update <spec-id>
```

### Ændrings-relaterede kommandoer

```bash
# Opret ny ændring
openspec change create

# Liste over alle ændringer
openspec change list

# Vis ændringsdetaljer
openspec change view <change-name>

# Tilføj opgave
openspec change add-task <change-name>

# Valider ændring
openspec change validate <change-name>

# Arkiver ændring
openspec change archive <change-name>
```

### Generelle kommandoer

```bash
# Vis version
openspec --version

# Vis hjælp
openspec --help

# Vis hjælp til specifik kommando
openspec <command> --help
```

## Brug af OpenSpec i HagiCode-projektet

### Projektintegration

I HagiCode-projektet er OpenSpec integreret i udviklingsarbejdsgangen:

1. **Initialiseringskontrol**: Startguiden kontrollerer, om OpenSpec er installeret
2. **Forslagsdrevet udvikling**: Alle nye funktioner skal gennemgå OpenSpec forslag-processen
3. **Kodegennemgang**: Ændringer skal matche OpenSpec-specifikationer for at kunne flettes
4. **Dokumentationssynkronisering**: OpenSpec-dokumentation holdes synkroniseret med kodeimplementering

### Versionskrav

HagiCode-projektet kræver brug af **OpenSpec >=1.0.0 {'<'}2.0.0** versionsområde.

Installationskommando:

```bash
npm install -g @fission-ai/openspec@1
```

### Arbejdsgangeksempel

Her er et komplet eksempel på brug af OpenSpec i HagiCode-projektet:

#### Trin 1: Opret forslag

```bash
cd /path/to/hagicode-project
openspec proposal create
```

Indtast i den interaktive prompt:
- Forslagstitel: Tilføj brugerautentificeringsfunktion
- Problembeskrivelse: Systemet har brug for brugerlogin og rettighedsstyring
- Løsning: Implementer JWT-baseret autentificeringssystem

#### Trin 2: Skriv specifikation

Efter godkendelse af forslaget skal du oprette en detaljeret specifikation:

```bash
openspec spec create
```

Definer i specifikationen:
- Design af autentificeringsproces
- API-grænsefladedefinition
- Databasetabelstruktur
- Sikkerhedsovervejelser

#### Trin 3: Opret ændring

Start implementeringen:

```bash
openspec change create
```

Indtast ændringsnavn: `add-user-authentication`

#### Trin 4: Tilføj opgaver

```bash
openspec change add-task add-user-authentication
```

Tilføj specifikke opgaver:
- Implementer brugerregistreringsgrænseflade
- Implementer login-grænseflade
- Implementer JWT middleware
- Skriv enhedstests

#### Trin 5: Implementering og validering

Efter afslutning af koden skal du validere ændringen:

```bash
openspec change validate add-user-authentication
```

#### Trin 6: Arkivering

Efter vellykket validering skal du arkivere ændringen:

```bash
openspec change archive add-user-authentication
```

## Bedste praksis

### Skrivning af forslag

1. **Klar problemformulering**: Definer klart problemet, der skal løses, undgå vage beskrivelser
2. **Overvej alternative løsninger**: Angiv og sammenlign forskellige løsninger
3. **Vurder påvirkning**: Analyser påvirkning og risici for det eksisterende system
4. **Vær kortfattet**: Forslag skal være kortfattede og klare, efterlad detaljer til specifikationen

### Skrivning af specifikationer

1. **Detaljeret men ikke overflødig**: Giv tilstrækkelige detaljer, men undgå overdesign
2. **Brug diagrammer**: Brug flowdiagrammer, arkitekturdiagrammer osv. til illustration
3. **Definer grænseflader**: Definer klart alle offentlige grænseflader og datastrukturer
4. **Overvej grænsecases**: Angiv undtagelsessituationer og fejlhåndtering

### Ændringsstyring

1. **Iterer i små trin**: Opdel store funktioner i flere små ændringer
2. **Opdater rettidigt**: Hold opgavestatus og dokumentation synkroniseret
3. **Kodegennemgang**: Udfør kodegennemgang efter afslutning af ændring
4. **Arkiver rettidigt**: Arkiver afsluttede ændringer rettidigt, hold arbejdsmappen ren

### Teamsamarbejde

1. **Regelmæssige gennemgange**: Gennemgå regelmæssigt forslag og specifikationer for at sikre teamkonsensus
2. **Dokumentation først**: Afslut dokumentation før du starter kodning
3. **Oprethold kommunikation**: Registrer diskussioner og beslutninger i forslag og specifikationer
4. **Versionskontrol**: Inkluder OpenSpec-dokumentation i versionskontrol

## Almindelige anvendelsesscenarier

### Scenarie 1: Tilføj ny funktion

1. Opret forslag, der beskriver nye funktionskrav
2. Teamet diskuterer og godkender forslaget
3. Skriv detaljeret teknisk specifikation
4. Opret ændring og implementer funktion
5. Valider og arkiver ændring

### Scenarie 2: Refaktorering af eksisterende kode

1. Opret forslag, der forklarer årsager og mål for refaktorering
2. Definer refaktoreringsplan og trin i specifikationen
3. Opret ændring og refaktorer gradvist
4. Sikr testdækning, valider refaktoreringsresultater
5. Arkiver ændring

### Scenarie 3: Ret fejl

For simple fejl:
- Kan rettes direkte, kræver ikke fuld OpenSpec-proces

For komplekse fejl:
1. Opret forslag til analyse af fejlens grundårsag
2. Definer rettelsesløsning i specifikationen
3. Opret ændring og implementer rettelse
4. Tilføj tests for at forhindre regression
5. Arkiver ændring

### Scenarie 4: Teknisk gældsstyring

1. Opret forslag, der angiver tekniske gældsposter
2. Definer forbedringsplan i specifikationen
3. Opret ændring for gradvis afbetaling af teknisk gæld
4. Spor fremgang og effekt
5. Arkiver afsluttede forbedringer

## Integration med andre værktøjer

### Git-integration

OpenSpec integreres problemfrit med Git:

```bash
# OpenSpec-dokumentation skal inkluderes i versionskontrol
git add openspec/
git commit -m "Add OpenSpec proposal for feature X"
```

### CI/CD-integration

Valider OpenSpec-specifikationer i CI/CD-processen:

```yaml
# .github/workflows/openspec.yml
name: OpenSpec Validation
on: [pull_request]
jobs:
  validate:
    runs-on: ubuntu-latest
    steps:
      - uses: actions/checkout@v2
      - name: Install OpenSpec
        run: npm install -g @fission-ai/openspec@1
      - name: Validate changes
        run: openspec change validate
```

### IDE-integration

Det anbefales at bruge en editor, der understøtter Markdown, til redigering af OpenSpec-dokumenter:
- VS Code + Markdown-plugin
- JetBrains IDEs + Markdown-understøttelse
- Vim/Neovim + Markdown-plugin

## Fejlfinding

### Almindelige problemer

#### Problem: OpenSpec-kommando ikke fundet

**Løsning**:
1. Bekræft at OpenSpec er installeret: `npm list -g @fission-ai/openspec`
2. Kontroller om PATH-miljøvariablen inkluderer npm global bin-mappe
3. Geninstaller: `npm install -g @fission-ai/openspec@1`

#### Problem: Kan ikke oprette forslag eller ændring

**Løsning**:
1. Bekræft at den aktuelle mappe er projektets rodmappe
2. Kontroller om der er en `openspec/` mappe, hvis ikke, opret den
3. Bekræft at du har skrivetilladelser

#### Problem: Validering mislykkedes

**Løsning**:
1. Kontroller om dokumentationen i ændringsmappen er komplet
2. Bekræft at alle nødvendige opgaver er afsluttet
3. Gennemgå detaljeret fejlinformation: `openspec change validate <change-name> --verbose`

#### Problem: Arkivering mislykkedes

**Løsning**:
1. Sikr at ændringen har bestået validering
2. Kontroller om `openspec/archived/` mappen eksisterer
3. Bekræft at der ikke er nogen ubekræftede kodeændringer

## Avanceret brug

### Brugerdefinerede skabeloner

OpenSpec understøtter brugerdefinerede skabeloner til forslag og specifikationer:

```bash
# Opret skabelonmappe i projektets rodmappe
mkdir -p .openspec/templates

# Opret brugerdefineret forslagsskabelon
cat > .openspec/templates/proposal.md << 'EOF'
# Proposal: {{title}}

## Problem Statement
{{problem}}

## Proposed Solution
{{solution}}

## Custom Section
Your custom content here
EOF
```

### Konfigurationsfil

Opret en `.openspec.json` konfigurationsfil i projektets rodmappe:

```json
{
  "version": "1.0",
  "templates": {
    "proposal": ".openspec/templates/proposal.md",
    "spec": ".openspec/templates/spec.md"
  },
  "validation": {
    "requireTests": true,
    "requireDocs": true
  },
  "hooks": {
    "pre-archive": "npm test"
  }
}
```

### Hooks

OpenSpec understøtter kørsel af brugerdefinerede scripts før og efter specifikke operationer:

```json
{
  "hooks": {
    "pre-create": "scripts/check-permissions.sh",
    "post-archive": "scripts/notify-team.sh"
  }
}
```
