HagiCode Desktop leveres med en Desktop-administreret Node-værktøjskæde. Det overtager ikke den globale Node-installation på din computer. Dens formål er at give Desktop-start og runtime-relaterede flows et stabilt og verificerbart runtime-fundament.

Læs denne FAQ, hvis du ser `portable-fixed/toolchain` i diagnostik, skal vide hvilken `node` Desktop bruger, eller undrer dig over, hvorfor dit system Node-version ikke direkte definerer Desktop runtime-adfærd.

## Hvorfor Desktop bundter Node

Desktop skal køre de samme understøttende flows på tværs af operativsystemer, installationskanaler og brugermiljøer. Hvis det kun afhang af brugerens eksisterende Node-installation, ville support skulle håndtere forældede versioner, manglende npm, forkert stibestilling, forurenede globale pakker og organisationsspecifikke registreringspolitikker på én gang.

Desktop beskriver dette mere specifikt som en isoleret Node/npm-runtime for administrerede CLI'er i stedet for at genbruge det systemomspændende miljø. Dette design har disse vigtigste fordele:

- **Konsistente versioner**: Desktop bruger en fastgjort Node 22 runtime-kontrakt i stedet for at gætte brugerens lokale hovedversion.
- **Kontrollerede opstartsforløb**: Desktop-administrerede kommandoer kan foretrække det pakkede `node` og `npm` indgange.
- **Ryd konfigurationsgrænser**: CLI-relateret konfiguration holdes adskilt fra host-dækkende globale konfigurationer, så opdateringer, skift og gendannelse er mindre tilbøjelige til at blive forstyrret af systemmiljøforskelle.
- **Reducerede konflikter**: Bundlede stier gælder for kommandomiljøer oprettet af Desktop og erstatter ikke brugerens globale node i deres terminal.
- **Ikke-påtrængende værtsadfærd**: Denne isolation undgår at skrive værktøjskædetilstand ind i brugerens eksisterende Node/npm-miljø, og afinstallation af Desktop forstyrrer ikke den Node, npm eller konfiguration, der allerede findes på maskinen.

## Hvordan Desktop vælger nodestien

Når du kører Desktop normalt, foretrækker Desktop Node-værktøjskæden installeret med appen i stedet for først at læse Node-installationen fra dit system. Dette holder Node-versionen i din terminal adskilt fra den Node-version, Desktop forventer, mens den kører.

Når den medfølgende værktøjskæde er aktiveret, placerer kommandomiljøer oprettet af Desktop bundtede værktøjskædestier foran på `PATH`. Som et resultat, selvom dit system har en anden Node-version installeret, løses Desktop-administrerede flows til Desktops bundtede Node og npm først.

Dette påvirker kun runtime-miljøer, der er oprettet af Desktop. Hvis du løber `node`, `npm`, eller andre kommandoer direkte i din systemterminal, fortsætter de med at bruge dit normale systemmiljø og skalkonfiguration.

I diagnosticerings- eller børneprocesmiljøer startet af Desktop, kan du muligvis se en variabel som denne:

```bash
HAGICODE_PORTABLE_TOOLCHAIN_ROOT=/path/to/portable-fixed/toolchain
```

Denne variabel betyder, at det desktop-administrerede runtime-miljø genkendte den medfølgende værktøjskæderod. At se denne sti betyder ikke, at Desktop ændrede din systemnode; det betyder kun, at Desktop bruger Node-miljøet, der blev leveret med appen inden for sit eget runtime-omfang.

## Hvor Desktop installerer administrerede npm-pakker

Desktop opbevarer runtime-filer og mutable npm-pakkefiler på forskellige steder. Den medfølgende værktøjskædesti som f.eks `portable-fixed/toolchain/node` er den uforanderlige kilde til den Desktop-administrerede `node` og `npm` kommandoer. Administrerede globale pakker installeres i stedet for Electron-brugerdatabiblioteket:

```text
userData/node22/npmGlobal
```

Hvis Desktop senere flytter til Node 24, bruger det et separat pakketræ som f.eks `userData/node24/npmGlobal`. Pakker installeret til Node 22 forbliver på disken, men behandles ikke som installeret for Node 24, før de geninstalleres i det nye Node-major præfiks.

Desktop afhængighedsstatus kontrollerer kun den aktive `userData/node<major>/npmGlobal` præfiks. Pakkefiler uden for dette præfiks ignoreres af afhængighedsstyring.
