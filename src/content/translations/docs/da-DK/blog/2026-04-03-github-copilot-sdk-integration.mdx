## Fra CLI-opkald til SDK-integration: bedste praksis for GitHub Copilot i .NET-projekter

> Opgraderingsvejen fra kommandolinjekald til officiel SDK-integration har været noget af en rejse. I dag vil jeg dele de faldgruber, vi løb ind i, og hvad vi lærte, mens vi byggede HagiCode.

## Baggrund

Efter GitHub Copilot SDK blev officielt frigivet i 2025, begyndte vi at integrere det i vores AI-kapacitetslag. Før det brugte projektet hovedsageligt GitHub Copilot ved direkte at påkalde Copilot CLI-kommandolinjeværktøjet, men den tilgang havde flere åbenlyse problemer:

- **Kompleks processtyring**: Vi var nødt til manuelt at administrere CLI-processens livscyklus, opstartstimeouts og procesoprydning. Processer kan gå ned uden varsel.
- **Ufuldstændig hændelseshåndtering**: Rå CLI-ankaldelse gør det svært at fange finkornede hændelser fra modelbegrundelse og værktøjsudførelse. Det er som kun at se resultatet uden tænkeprocessen.
- **Svær sessionsstyring**: Der var ingen effektiv mekanisme til genbrug og gendannelse af sessioner, så enhver interaktion måtte starte forfra.
- **Kompatibilitetsproblemer**: CLI-argumenter ændrede sig ofte, hvilket betød, at vi konstant skulle opretholde kompatibilitetslogik for disse parametre.

Disse problemer blev mere og mere tydelige i den daglige udvikling, især når vi havde brug for at spore modelræsonnementer (`thinking`) og værktøjsudførelsesstatus i realtid. På det tidspunkt blev det klart, at vi havde brug for en mere komplet integrationstilgang på et lavere niveau.

## Om HagiCode

Den tilgang, der er delt i denne artikel, kommer fra vores praktiske erfaring i [HagiCode](https://hagicode.com) projekt. HagiCode er et open source AI-kodningsassistentprojekt, og under udviklingen havde vi brug for dyb integration med GitHub Copilot-funktioner, fra grundlæggende kodefuldførelse til komplekse multi-turn-samtaler og værktøjsopkald. Disse krav fra den virkelige verden skubbede os til at flytte fra CLI-indkaldelse til det officielle SDK.

Hvis denne implementering lyder nyttig for dig, er der en god chance for, at vores ingeniørerfaring kan hjælpe. I så fald kan selve HagiCode-projektet også være værd at tjekke ud. Du kan endda finde flere projektoplysninger og links i slutningen af ​​denne artikel.

## Arkitektur design

Projektet bruger en lagdelt arkitektur til at adressere begrænsningerne ved CLI-påkaldelse:

```
┌─────────────────────────────────────────────────────────┐
│  hagicode-core (Orleans Grains + AI Provider Layer)    │
│  - CopilotAIProvider: Converts AIRequest to CopilotOptions │
│  - GitHubCopilotGrain: Orleans distributed execution interface │
└─────────────────────────────────────────────────────────┘
                          ↓
┌─────────────────────────────────────────────────────────┐
│  HagiCode.Libs (Shared Provider Layer)                 │
│  - CopilotProvider: CLI Provider interface implementation │
│  - ICopilotSdkGateway: SDK invocation abstraction      │
│  - GitHubCopilotSdkGateway: SDK session management and event dispatch │
└─────────────────────────────────────────────────────────┘
                          ↓
┌─────────────────────────────────────────────────────────┐
│  GitHub Copilot SDK (Official .NET SDK)                │
│  - CopilotClient: SDK client                           │
│  - CopilotSession: Session management                  │
│  - SessionEvent: Event stream                          │
└─────────────────────────────────────────────────────────┘
```

Dette lagdelte design giver flere praktiske tekniske fordele:

1. **Adskillelse af bekymringer**: Kerneforretningslogik er afkoblet fra SDK-implementeringsdetaljer.
2. **Testbarhed**: Den `ICopilotSdkGateway` interface gør enhedstest ligetil.
3. **Genbrugbarhed**: `HagiCode.Libs` kan refereres af flere projekter.
4. **Vedligeholdelse**: SDK-opgraderinger kræver kun ændringer i Gateway-laget, mens de øverste lag forbliver uberørte.

## Kerneimplementering

### Autentificeringsflow

Autentificering er det første og vigtigste trin i SDK-integration. Hvis godkendelsen mislykkes, betyder intet andet noget. Vi har designet en fleksibel godkendelseskonfiguration, der understøtter flere godkendelseskilder:

```csharp
// CopilotProvider.cs - Authentication source configuration
public class CopilotOptions
{
    public bool UseLoggedInUser { get; set; } = true;
    public string? GitHubToken { get; set; }
    public string? CliUrl { get; set; }
}

// Convert to SDK request
return new CopilotSdkRequest(
    GitHubToken: options.AuthSource == CopilotAuthSource.GitHubToken
        ? options.GitHubToken
        : null,
    UseLoggedInUser: options.AuthSource != CopilotAuthSource.GitHubToken
);
```

Fordelene ved dette design er ret klare:

- Det understøtter logget ind brugertilstand uden at kræve et token, hvilket passer godt til skrivebordsscenarier.
- Den understøtter GitHub Token-tilstand, som er velegnet til serverside-implementeringer og centraliseret styring.
- Det understøtter tilsidesættelse af Copilot CLI URL, hvilket hjælper med konfiguration af virksomhedens proxy.

I praksis forenklede denne fleksible godkendelsesmodel konfigurationen i høj grad på tværs af forskellige implementeringsscenarier. Desktop-klienten kan bruge hver brugers egen Copilot login-tilstand, mens serversiden kan administrere godkendelse centralt gennem tokens.

### Håndtering af hændelsesstrømme

En af de mest kraftfulde muligheder i SDK er komplet hændelsesstream-optagelse. Vi implementerede et hændelsesafsendelsessystem, der kan håndtere alle slags SDK-begivenheder i realtid:

```csharp
// GitHubCopilotSdkGateway.cs - Core logic for event dispatch
internal static SessionEventDispatchResult DispatchSessionEvent(
    SessionEvent evt, bool sawDelta)
{
    switch (evt)
    {
        case AssistantReasoningEvent reasoningEvent:
            // Capture the model reasoning process
            events.Add(new CopilotSdkStreamEvent(
                CopilotSdkStreamEventType.ReasoningDelta,
                Content: reasoningEvent.Data.Content));
            break;

        case ToolExecutionStartEvent toolStartEvent:
            // Capture the start of a tool invocation
            events.Add(new CopilotSdkStreamEvent(
                CopilotSdkStreamEventType.ToolExecutionStart,
                ToolName: toolStartEvent.Data.ToolName,
                ToolCallId: toolStartEvent.Data.ToolCallId));
            break;

        case ToolExecutionCompleteEvent toolCompleteEvent:
            // Capture tool completion and its result
            events.Add(new CopilotSdkStreamEvent(
                CopilotSdkStreamEventType.ToolExecutionEnd,
                Content: ExtractToolExecutionContent(toolCompleteEvent)));
            break;

        default:
            // Preserve unhandled events as RawEvent
            events.Add(new CopilotSdkStreamEvent(
                CopilotSdkStreamEventType.RawEvent,
                RawEventType: evt.GetType().Name));
            break;
    }
}
```

Værdien af denne implementering er betydelig:

- **Fuldstændig indfangning af modelbegrundelsesprocessen** (`thinking`): brugere kan se, hvordan AI'en ræsonnerer, ikke kun det endelige svar.
- **Sporing i realtid af værktøjsudførelsesstatus**: vi ved, hvilke værktøjer der kører, hvornår de er færdige, og hvad de returnerer.
- **Nul tab af begivenhed**: ved at falde tilbage til `RawEvent`, sikrer vi, at hver begivenhed bliver optaget.

I HagiCode hjælper disse finkornede hændelser brugere med at forstå, hvordan AI fungerer internt, især når de fejlfinder komplekse opgaver.

### CLI-kompatibilitetshåndtering

Efter migrering fra CLI-indkaldelse til SDK'et, fandt vi ud af, at nogle eksisterende CLI-parametre ikke længere var gældende i SDK'et. For at bevare bagudkompatibilitet implementerede vi et parameterfiltreringssystem:

```csharp
// CopilotCliCompatibility.cs - Argument filtering
private static readonly Dictionary<string, string> RejectedFlags = new()
{
    ["--headless"] = "Unsupported startup argument",
    ["--model"] = "Passed through an SDK-native field",
    ["--prompt"] = "Passed through an SDK-native field",
    ["--interactive"] = "Interaction is managed by the provider",
};

public static CopilotCliArgumentBuildResult BuildCliArgs(CopilotOptions options)
{
    // Filter out unsupported arguments and keep compatible ones
    // Generate diagnostic information
}
```

Dette giver os flere fordele:

- Det filtrerer automatisk inkompatible CLI-argumenter for at forhindre runtime-fejl.
- Det genererer klare diagnostiske meddelelser for at hjælpe udviklere med at lokalisere problemer hurtigt.
- Det holder SDK'et stabilt og isoleret mod ændringer i CLI-parametre.

Under opgraderingen hjalp denne kompatibilitetsmekanisme os med at skifte problemfrit. Eksisterende konfigurationsfiler kunne stadig bruges og behøvede kun gradvise justeringer baseret på det diagnostiske output.

### Runtime Pooling

Det er relativt dyrt at oprette Copilot SDK-sessioner, og at oprette og ødelægge sessioner for ofte skader ydeevnen. For at løse det implementerede vi et sessionspuljestyringssystem:

```csharp
// CopilotProvider.cs - Session pool management
await using var lease = await _poolCoordinator.AcquireCopilotRuntimeAsync(
    request,
    async ct => await _gateway.CreateRuntimeAsync(sdkRequest, ct),
    cancellationToken);

if (lease.IsWarmLease)
{
    // Reuse an existing session
    yield return CreateSessionReusedMessage();
}

await foreach (var eventData in lease.Entry.Resource.SendPromptAsync(...))
{
    yield return MapEvent(eventData);
}
```

Fordelene ved session pooling inkluderer:

- **Genbrug af session**: anmodninger med samme `sessionId` kan genbruge eksisterende sessioner og reducere opstartsomkostninger.
- **Sessionsgendannelsesunderstøttelse**: Efter en netværksafbrydelse kan tidligere sessionstilstand gendannes.
- **Automatisk poolingstyring**: Udløbne sessioner ryddes automatisk op for at undgå ressourcelækage.

I HagiCode forbedrede sessionspooling mærkbart reaktionsevnen, især for kontinuerlige samtaler.

### Orleans integration

HagiCode bruger Orleans som sin distribuerede ramme, og vi integrerede Copilot SDK i Orleans Grains:

```csharp
// GitHubCopilotGrain.cs - Distributed execution
public async IAsyncEnumerable<GitHubCopilotResponse> ExecuteCommandStreamAsync(
    string command,
    CancellationToken token = default)
{
    var provider = await aiProviderFactory.GetProviderAsync(AIProviderType.GitHubCopilot);

    await foreach (var chunk in provider.SendMessageAsync(request, null, token))
    {
        // Map to the unified response format
        yield return BuildChunkResponse(chunk, startedAt);
    }
}
```

Fordelene ved Orleans integration er betydelige:

- **Unified AI Provider abstraktion**: det bliver nemt at skifte mellem forskellige AI-udbydere.
- **Isolering af flere lejere**: Copilotsessioner for forskellige brugere forbliver isolerede fra hinanden.
- **Vedvarende sessionstilstand**: Sessionstilstand kan gendannes, selv efter serveren genstartes.

Til scenarier, der skal håndtere et stort antal samtidige anmodninger, giver Orleans stærk skalerbarhed.

## Praktisk vejledning

### Konfigurationseksempel

Her er et komplet konfigurationseksempel:

```json
{
  "AI": {
    "Providers": {
      "Providers": {
        "GitHubCopilot": {
          "Enabled": true,
          "ExecutablePath": "copilot",
          "Model": "gpt-5",
          "WorkingDirectory": "/path/to/project",
          "Timeout": 7200,
          "StartupTimeout": 30,
          "UseLoggedInUser": true,
          "NoAskUser": true,
          "Permissions": {
            "AllowAllTools": false,
            "AllowedTools": ["Read", "Bash", "Grep"],
            "DeniedTools": ["Edit"]
          }
        }
      }
    }
  }
}
```

### Noter om brug

I den virkelige verden har vi opsummeret flere punkter, der er værd at være opmærksomme på:

**Opstartstimeoutkonfiguration**: Den første opstart af Copilot CLI kan tage relativt lang tid, så vi anbefaler at indstille `StartupTimeout` til mindst 30 sekunder. Hvis dette er første login, kan det tage endnu længere tid.

**Tilladelsesstyring**: Undgå at bruge i produktionsmiljøer `AllowAllTools: true`. Brug `AllowedTools` tilladelsesliste for at kontrollere, hvilke værktøjer der er tilgængelige, og bruge `DeniedTools` afvisningsliste for at blokere farlige operationer. Dette forhindrer effektivt AI i at udføre risikable kommandoer.

**Sessionsstyring**: Forespørgsler med samme `sessionId` automatisk genbrug sessioner. Sessionstilstand er vedvarende igennem `ProviderSessionId`. Annullering udbredes via `CancellationTokenSource`.

**Diagnostisk output**: Inkompatible CLI-argumenter genererer meddelelser af typen `diagnostic`. Raw SDK-begivenheder bevares som `event.raw`. Fejlmeddelelser inkluderer kategorier som starttimeout og argumentinkompatibilitet for at gøre fejlfinding nemmere.

### Bedste praksis

Baseret på vores praktiske erfaring er her et par bedste fremgangsmåder:

**1. Brug en værktøjstilladelsesliste**

```csharp
var request = new AIRequest
{
    Prompt = "Analyze this file",
    AllowedTools = new[] { "Read", "Grep", "Bash(git:*)" }
};
```

Eksplicit angivelse af de tilladte værktøjer gennem en tilladelsesliste hjælper med at forhindre uventede AI-handlinger. Dette er især vigtigt for værktøjer med skrivetilladelser, som f.eks `Edit`, som skal håndteres med ekstra forsigtighed.

**2. Indstil rimelige timeouts**

```csharp
options.Timeout = 3600;  // 1 hour
options.StartupTimeout = 60;  // 1 minute
```

Indstil passende timeoutværdier baseret på opgavens kompleksitet. Hvis værdien er for kort, kan opgaver blive afbrudt. Hvis det er for lang tid, kan ressourcer blive spildt på at vente på henvendelser, der ikke reagerer.

**3. Aktiver genbrug af session**

```csharp
options.SessionId = "my-session-123";
```

Bruger det samme `sessionId` til relaterede opgaver lader dig genbruge tidligere sessionskontekst og forbedre responshastigheden.

**4. Håndter streamingsvar**

```csharp
await foreach (var chunk in provider.StreamAsync(request))
{
    switch (chunk.Type)
    {
        case StreamingChunkType.ThinkingDelta:
            // Handle the reasoning process
            break;
        case StreamingChunkType.ToolCallDelta:
            // Handle tool invocation
            break;
        case StreamingChunkType.ContentDelta:
            // Handle text output
            break;
    }
}
```

Streamingsvar giver dig mulighed for at vise AI-behandlingsfremskridt i realtid, hvilket forbedrer brugeroplevelsen. Dette er især værdifuldt til langvarige opgaver.

**5. Fejlhåndtering og genforsøg**

```csharp
try
{
    await foreach (var chunk in provider.StreamAsync(request))
    {
        // Handle the response
    }
}
catch (CopilotSessionException ex)
{
    // Handle session exceptions
    logger.LogError(ex, "Copilot session failed");
    // Decide whether to retry based on the exception type
}
```

Korrekt fejlhåndtering og genforsøgslogik forbedrer den overordnede systemstabilitet.

## Konklusion

Opgradering fra CLI-indkaldelse til SDK-integration gav betydelig værdi til HagiCode-projektet:

- **Forbedret stabilitet**: SDK'en giver en mere stabil grænseflade, der ikke påvirkes af CLI-versionsændringer.
- **Mere komplet funktionalitet**: den fanger hele hændelsesstrømmen, inklusive ræsonnement og værktøjsudførelsesstatus.
- **Højere udviklingseffektivitet**: den typesikre SDK-grænseflade gør udviklingen mere effektiv og reducerer runtime-fejl.
- **Bedre brugeroplevelse**: Hændelsesfeedback i realtid giver brugerne en klarere forståelse af, hvad AI laver.

Denne opgradering var ikke kun en erstatning for den tekniske implementering. Det var også en arkitektonisk optimering af hele AI-kapacitetslaget. Gennem lagdelte design- og abstraktionsgrænseflader fik vi bedre vedligeholdelsesmuligheder og udvidelsesmuligheder.

Hvis du overvejer at integrere GitHub Copilot i dit eget .NET-projekt, håber jeg, at disse praktiske lektioner hjælper dig med at undgå unødvendige omveje. Det officielle SDK er faktisk mere stabilt og komplet end CLI-invokation, og det er tiden værd at forstå og adoptere det.
