## Design af 12 exceptionelle opstartsoplevelser til HagiCode: Fra minimalisme til cyberpunk

> Det korte mellemrum mellem at downloade en React 19-app og fuldføre Hydration er en gylden mulighed for brugere til at mærke din brandpersonlighed. I denne artikel deler vi et komplet opstartsstilsystem, vi byggede til HagiCode-projektet ved hjælp af HTML/CSS/JS.

## Baggrund

Som en moderne applikation bygget med ASP.NET Core 10 og React 19 (Vite), bruger HagiCode en frontend-backend-separeret implementeringsarkitektur. Frontend build-outputtet er pakket ind i backend `wwwroot/` bibliotek og hostet af ASP.NET Core.

Denne arkitektur introducerer dog et klassisk UX-smertepunkt: Når brugere besøger siden, indlæser browseren først HTML'en, downloader derefter den store JS-pakke og lader til sidst React udføre Hydration. I løbet af denne "vakuumperiode", der varer fra et par hundrede millisekunder til flere sekunder, ser brugerne enten en tom skærm eller en livløs statisk side.

For at udfylde dette hul og injicere HagiCodes brandpersonlighed, var vi nødt til at designe et startup-stilsystem, der var implementeret udelukkende med inline-kode indeni `index.html`.

## Om HagiCode

Splash screen-designtilgangen, der er delt i denne artikel, kommer fra vores praktiske erfaring i [HagiCode](https://github.com/HagiCode-org/site) projekt. Som AI-kodningsassistent bekymrer HagiCode sig ikke kun om kodegenereringseffektivitet, men også om udviklerens visuelle oplevelse. Dette opstartssystem er et af resultaterne af vores stræben efter ultimativ frontend-ydeevne.

## Kerneudfordringer og arkitekturdesign

Inden vi begyndte at designe, skulle vi først afklare de tekniske begrænsninger. Da alt skulle implementeres inline i `index.html`, kunne vi ikke indlæse andre eksterne CSS- eller JS-filer end Reacts eget bundt.

### Teknisk begrænsningsanalyse

1.  **Nul-afhængighedsprincippet**: Alle stilarter skal leve inde i en `<style>` tag, og al logik skal leve inde i en `<script>` tag.
2.  **Defensiv CSS**: For at forhindre globale stilarter i at forurene splash-skærmen, efter at React-appen er monteret, besluttede vi at omslutte alle opstartsstile med et højt prioriteret ID-præfiks som f.eks. `#boot-screen`.
3.  **Performance først**: Animationer skal bruge CSS `transform` og `opacity` hvor det er muligt for at undgå tilbagestrømning og sikre, at hovedtråden forbliver ublokeret.
4.  **Visuel konsistens**: Farver og skrifttyper skal forblive på linje med HagiCodes Tailwind-konfiguration.

### Arkitekturmønster: Skal og injektor

Vi adopterede et **variantmønster**. Kernelogikken er indkapslet i et umiddelbart påkaldt funktionsudtryk (IIFE), mens den specifikke gengivelseslogik injiceres gennem konfiguration. Dette lader os skifte mellem forskellige stilarter gennem simpel konfiguration i stedet for at omskrive DOM-manipulationslogik gentagne gange.

Her er kernearkitekturkoden:

```html
<!-- Inline in index.html -->
<div id="boot-root"></div>

<script>
(function() {
  const BootSequence = {
    config: {
      theme: 'terminal', // Configurable as 'minimal', 'skeleton', 'code-rain', etc.
      color: '#3b82f6'   // Brand color
    },
    
    // Core lifecycle
    init() {
      this.render();
      this.listenForMount();
    },

    // Render the currently selected style
    render() {
      const root = document.getElementById('boot-root');
      if (this.variants[this.config.theme]) {
        root.innerHTML = this.variants[this.config.theme].render();
      }
    },

    // Listen for successful React mount and exit gracefully
    listenForMount() {
      window.addEventListener('hagicode:ready', () => {
        const screen = document.getElementById('boot-root');
        // Fade out first, then remove the DOM to avoid flicker
        screen.style.opacity = '0';
        screen.style.transition = 'opacity 0.3s ease';
        setTimeout(() => screen.remove(), 300);
      });
    },

    // The implementation logic for all 12 styles lives here
    variants: {
      // ...see details below
    }
  };

  BootSequence.init();
})();
</script>
```

## En tjekliste med 12 opstartsstildesign

Vi grupperede disse 12 stilarter i seks hovedkategorier for at tilfredsstille forskellige scenarier og æstetiske præferencer.

### A. Minimalisme

> "Mindre er mere." For scenarier, der forfølger ultimativ læssehastighed, tilbyder vi de lettest mulige muligheder.

#### 1. Minimalistisk Prik
En simpel prik sidder i midten af skærmen, parret med en vejrtrækningsanimation.
*   **Implementering**: CSS `@keyframes` styrer skala og opacitet.
*   **Bedst til**: Ethvert tilfælde, hvor siden skal forblive absolut ren.

#### 2. Brand Reveal
Bruger SVG `stroke-dasharray` animation, simulerer den en håndtegnet afsløring af HagiCode-logoets linjer, efterfulgt af en tekstfade-in.
*   **Teknik**: SVG-stianimation med en meget poleret fornemmelse.

### B. Skeleton Screen Mimicry

> "Kunsten at bedrage øjet." Ved at simulere et rigtigt UI-layout føler brugerne, at siden allerede er halvt indlæst.

#### 3. Sidebar Chat Skelet
Dette kan være den mest praktiske mulighed. Vi byggede manuelt et layout i HTML, der afspejler React `Sidebar` og `ChatInput` komponenter nøjagtigt, og derefter overlejret det med en grå shimmer-animation.
*   **Værdi**: Når React-hydreringen er fuldført, bliver skelettet øjeblikkeligt den virkelige komponent, og brugerne kan næsten ikke opfatte kontakten.

#### 4. Kortstakskelet
Den simulerer den stablede bevægelse af forslagskort under indlæsning, ved hjælp af 3D-transformationer for at få kortene til at flyde subtilt.

### C. Abstraktion og kunst

> Vis HagiCodes nørde-DNA frem.

#### 5. Geometrisk Morf
En geometrisk form (en firkant) gengives i midten af skærmen og forvandles derefter jævnt over tid til en cirkel, en trekant og til sidst logoet.
*   **Teknologi**: Glatte overgange med CSS `border-radius`.

#### 6. Code Rain
En hyldest til *The Matrix*. Ved at bruge JetBrains Mono-skrifttypen falder svage strømme af tegn i baggrunden.
*   **Bemærk**: For ydeevne skal karakterstrømmene forblive inden for et mindre område eller bruge en lavere opdateringshastighed.

#### 7. Neonpuls
En glødende ring i cyberpunk-stil, der bruger flere `box-shadow` lag for at skabe en kraftig neonglød.

### D. Branding og temaer

> Få systemet til at føles levende.

#### 8. Sæsonbestemt tema
Dette er en dynamisk læsser. Den tjekker den aktuelle dato for helligdage såsom månenytår eller jul og indlæser den tilsvarende SVG-animation.
*   **Eksempel**: I løbet af Lunar New Year svajer røde lanterner forsigtigt nederst på skærmen.

#### 9. Gradientflow
Baggrunden bruger en flydende gradient baseret på HagiCode-mærkets farver. Kombineret med animeret `background-size` og `background-position`, det skaber en nordlys-lignende følelse af bevægelse.

### E. Teknisk fornemmelse

> En hilsen til udviklerne.

#### 10. Terminal Boot
Det simulerer konsoloutput. Kodelinjer ruller hurtigt forbi:
```text
> Initializing HagiCode Core...
> Loading models...
> Connecting to neural network...
```
Det føles øjeblikkeligt bekendt for enhver udvikler.

#### 11. Fremskridtslinje
En tynd statuslinje vises øverst på skærmen, med en procentdel vist til højre. Selvom vi ikke kan få adgang til den reelle downloadfremgang, kan vi bruge en timer til at simulere en "troværdig" indlæsningsproces: hurtig for de første 80 %, derefter gradvist langsommere for de sidste 20 %.

### F. Kreativitet

#### 12. Pixel samling
Dette er en meget interessant idé. Små firkanter er spredt ud over skærmen, konvergerer derefter mod midten og samles gradvist til HagiCode-logo-ikonet. Det symboliserer processen med at bygge kode.

## Bedste praksis og faldgruber

I HagiCodes reelle udviklingsarbejde opsummerede vi flere kritisk vigtige implementeringsdetaljer.

### 1. Defensiv CSS er obligatorisk
Bliv aldrig doven og spring over præfikset. Engang glemte vi at omfange splash screen-stilene med et ID og globalt `div` stilarter efter React monteret påvirkede uventet splash-skærmen og brød layoutet.
**Lektion lært**: Sæt hver CSS-vælger under `#boot-screen`, og brug `!important` for at hæve prioritet, når det er nødvendigt, men kun inde i splash-skærmens CSS.

### 2. Yndefulde overgange
Efter at React er monteret med succes, skal du ikke gøre det direkte `remove()` splash-skærmen DOM.
**Korrekt tilgang**:
1.  Reager triggere `window.dispatchEvent(new Event('hagicode:ready'))`.
2.  Splash-skærmen lytter efter begivenheden og de første sæt `opacity: 0`.
3.  Vent 300 ms, hvilket svarer til CSS-overgangens varighed, og ring `.remove()` først efter at skærmen er helt usynlig.

### 3. Tema Variabel synkronisering
Splash-skærmens farveværdier er hårdkodet ind `index.html`. Hvis vi ændrer Tailwinds primære farve, skal vi også opdatere splash-skærmen.
**Optimering**: Skriv et simpelt plugin i Vite build-scriptet til at læse `tailwind.config.js` og injicer farvevariabler i `index.html` skabelonvariabler, hvilket skaber en enkelt kilde til sandhed.

### 4. Skriftforudindlæsning
Splash-skærme skal ofte bruge en mærkeskrifttype, men hvis skrifttypen indlæses langsomt, kan FOUT (Flash of Unstyled Text) vises.
**Løsning**: Tilføj `<link rel="preload" href="/fonts/JetBrainsMono.woff2" as="font" type="font/woff2" crossorigin>` indeni `<head>`. Dette er en billig måde at forbedre oplevelsen på.

### 5. Ydelsesovervågning
Vi injicerede `performance.mark('boot-start')` i bunden af `index.html`, og markeret `boot-end` når React er monteret med succes.
**Hvorfor det betyder noget**: Ved at indsamle disse data gennem Application Insights kan vi direkte måle, hvor meget startskærmen forkorter den opfattede ventetid. Dataene viser, at en fremragende skeletskærm kan forbedre brugernes tolerance over for et "langsomt netværk" med mere end 50%.
