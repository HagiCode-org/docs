## Fra Docusaurus 3.x til Astro 5.x: A Retrospective on the HagiCode Site Migration

> Denne artikel ser tilbage på vores fulde migrering af HagiCodes officielle hjemmeside fra Docusaurus 3.x til Astro 5.x. Vi vil tage et dybt dyk ned i, hvordan Astro's Islands Architecture hjalp os med at løse ydeevneflaskehalse, mens vi bevarede vores eksisterende React-komponentaktiver og leverede forbedringer i både byggehastighed og indlæsningsydelse.

## Baggrund

I januar 2026 udførte vi en "hjertetransplantation" på det officielle HagiCode-websted ved fuldt ud at migrere dets kerneramme fra Docusaurus 3.x til Astro 5.x. Dette var ikke en impulsiv omskrivning, men en nøje overvejet teknisk beslutning.

Før migreringen var vores side funktionelt færdig, men den var begyndt at vise nogle klassiske "luksusproblemer": oppustede byggeartefakter, overdreven JavaScript-nyttelast og mindre end ideel sideindlæsningshastighed på komplekse dokumentationssider. Som et AI-kodningsassistentprojekt har HagiCode brug for hyppig dokumentation og funktionsopdateringer, så byggeeffektiviteten påvirker udgivelseshastigheden direkte. Samtidig ønskede vi, at siden skulle være mere søgemaskinevenlig (SEO), så flere udviklere kunne opdage projektet.

For at løse disse smertepunkter tog vi en dristig beslutning: genopbygg hele systemet på Astro. Virkningen af ​​denne beslutning kan være endnu større, end du forventer. Jeg vil komme ind i detaljerne snart.

## Om HagiCode

Webstedsmigreringstilgangen, der er delt i denne artikel, kommer fra vores praktiske erfaring i [HagiCode](https://github.com/HagiCode-org/site) projekt.

HagiCode er en AI-kodningsassistent med fokus på at forbedre udviklingseffektiviteten. Vi bekymrer os ikke kun om at gentage kernefunktioner, men også om udvikleroplevelsen. Denne siderefaktor var også beregnet til at give brugerne den hurtigst mulige oplevelse, når de gennemser vores dokumenter og officielle hjemmeside.

## Hvorfor forlade det modne Docusaurus-økosystem?

Inden for React-økosystemet har Docusaurus længe været "standardsvaret" for dokumentationssteder. Det fungerer ud af boksen, tilbyder et rigt plugin-økosystem og har et aktivt fællesskab. Men da HagiCode fik flere funktioner, følte vi også dets begrænsninger:

1.  **Ydeevneflaskehalse**: Docusaurus er grundlæggende en React SPA (enkeltsidet applikation). Selvom du kun skriver statiske sider, skal klienten stadig indlæse React-runtiden og hydrere siden, hvilket er unødvendigt tungt for simple docs-sider.
2.  **Stor aktivstørrelse**: Selv når en side indeholder meget lidt indhold, forbliver den medfølgende JS-størrelse relativt fast. Det er ikke ideelt for mobile brugere eller dårlige netværksforhold.
3.  **Begrænset fleksibilitet**: Selvom det kan udvides, ønskede vi mere kontrol på lavt niveau over byggepipelinen.

Astro ankom på præcis det rigtige tidspunkt til at løse disse problemer. Det introducerede en ny "Islands Architecture": Som standard genererer Astro statisk HTML med nul JavaScript, og kun komponenter, der kræver interaktivitet, "aktiveres" og indlæser JS. Det betyder, at det meste af vores websted bliver ren HTML og indlæses ekstremt hurtigt.

## Kernemigreringsstrategi: En glat arkitektonisk overgang

Migration var ikke bare kopier og indsæt. Det krævede et tankesæt. Vi flyttede fra Docusaurus's "all React"-model til Astros "Core + Islands"-model.

### 1. Genopbygning af konfigurationssystemet

Først skulle vi flytte fra `docusaurus.config.ts` til `astro.config.mjs`. Dette var ikke bare et filomdøb, men en omskrivning af routing og build-logik.

I Docusaurus er alt et plugin. I Astro er alt en integration. Vi havde brug for at omdefinere webstedets basissti, opbygge outputtilstand (statisk vs SSR) og aktivoptimeringsstrategi.

**Før migrering:**
```typescript
// docusaurus.config.ts
export default {
  title: 'HagiCode',
  url: 'https://hagicode.com',
  baseUrl: '/',
  // ... more configuration
};
```

**Efter migrering:**
```javascript
// astro.config.mjs
import { defineConfig } from 'astro/config';
import react from '@astrojs/react';

export default defineConfig({
  integrations: [react()],
  site: 'https://hagicode.com',
  base: '/',
  // Optimization settings for static assets
  build: {
    inlineStylesheets: 'auto',
  },
});
```

### 2. Hvad skal opbevares, og hvad der skal refaktoreres i React-komponenter

Dette var den mest smertefulde del af migrationen. Vores eksisterende websted havde mange React-komponenter, såsom faner, kodefremhævning, feedbackknapper og mere. At smide dem væk ville være spild, men at beholde alt ville gøre JavaScript-nyttelasten for tung.

HagiCode vedtog en **progressiv hydreringsstrategi**:
*   **Rene statiske komponenter**: Til præsentationsindhold såsom sidehoveder, sidefødder og almindelig tekstdokumentation omskrev vi dem som Astro-komponenter (`.astro` filer) og gengivet dem direkte til HTML på byggetidspunktet.
*   **Interaktive øer**: For komponenter, der skal forblive interaktive, såsom temaskiftere, faneskift og kodeblokkopieringsknapper, beholdt vi React-implementeringen og tilføjede `client:load` eller `client:visible` direktiver.

For eksempel vores almindeligt anvendte Tabs-komponent i dokumentationen:

```jsx
// src/components/Tabs.jsx
import { useState } from 'react';
import './Tabs.css'; // Import styles

export default function Tabs({ items }) {
  const [activeIndex, setActiveIndex] = useState(0);
  // ... state logic
  return (
    <div className="tabs-wrapper">
      {/* Rendering logic */}
    </div>
  );
}
```

Når det bruges i Markdown, fortæller vi eksplicit til Astro: "Denne komponent har brug for JS."

```markdown
// src/content/docs/example.mdx
import Tabs from '../../components/Tabs.jsx';

<!-- Load JS only when the component enters the viewport -->
<Tabs client:visible items={...} />
```

På denne måde konkurrerer interaktive komponenter uden for viewporten ikke om båndbredden, hvilket i høj grad forbedrer indlæsningshastigheden på første skærm.

### 3. Tilpasning af stylingsystemet: Fra CSS-moduler til Scoped CSS

Docusaurus understøtter CSS-moduler som standard, mens Astro opfordrer til Scoped CSS gennem `<style>` tag. Kerneideen bag begge er stilisolering, men syntaksen er anderledes.

Under HagiCode-migreringen konverterede vi de fleste komplekse CSS-moduler til Astro's scoped styles. Dette viste sig faktisk at være en god ting, fordi i `.astro` filer stilene og skabelonerne lever i den samme fil, hvilket gør vedligeholdelsen mere intuitiv.

**Før refaktorisering:**
```css
/* Tabs.module.css */
.wrapper { background: var(--ifm-background-color); }
```

**Efter refactoring (Astro Scoped):**
```html
<!-- Tabs.astro -->
<div class="tabs-wrapper">
  <slot />
</div>

<style>
  .tabs-wrapper {
    /* Use CSS variables directly to adapt to the theme */
    background: var(--bg-color);
    padding: 1rem;
  }
</style>
```

Samtidig forenede vi det globale CSS-variabelsystem og brugte Astros miljøbevidste funktioner til at sikre, at mørk tilstand skifter jævnt på tværs af sider.

## Faldgruber vi ramte i praksis, og hvordan vi løste dem

Under selve HagiCode-migreringen stødte vi ind i en del problemer. Her er flere af de mest typiske.

### 1. Sti og miljø Variable smertepunkter

HagiCode understøtter implementering af understier, såsom implementering under en GitHub Pages-undermappe. I Docusaurus, `baseUrl` håndteres automatisk. I Astro skal vi dog være mere forsigtige, når vi håndterer billedlinks og API-anmodninger.

Vi introducerede en miljøvariabelmekanisme til at styre dette konsekvent:

```javascript
// Handle paths in the build script
const getBasePath = () => import.meta.env.VITE_SITE_BASE || '/';
```

Sørg for ikke at hardkode stier, der begynder med `/` i din kode. I udvikling versus produktion, eller efter konfiguration af en `base` sti, kan det forårsage 404s for aktiver.

### 2. CommonJS Script-kompatibilitet

Vores gamle websted havde nogle Node.js-scripts, der blev brugt til opgaver såsom automatisk hentning af Metrics-data og opdatering af sitemap, og de blev skrevet i CommonJS (`require`). Astro og moderne byggeværktøjer har fuldt ud omfavnet ES-moduler (`import`/`export`).

Hvis du også har lignende scripts, skal du huske at omfaktorere dem alle til ES-moduler. Det er den retning, økosystemet bevæger sig, og jo før du foretager ændringen, jo mindre problemer vil du have senere.

```javascript
// Old way
const fs = require('fs');

// New way
import fs from 'fs';
```

### 3. Glem ikke SEO og omdirigeringer

Søgemaskiner har allerede indekseret HagiCodes gamle Docusaurus-sider. Hvis du skifter direkte til Astro, og URL-strukturen ændres, kan du ende med et stort antal 404'er og et stort fald i søgerangeringen.

Vi konfigurerede omdirigeringsregler i Astro:

```javascript
// astro.config.mjs
export default defineConfig({
  redirects: {
    '/docs/old-path': '/docs/new-path',
    // Map old links to new links in bulk
  }
});
```

Eller du kan håndtere dette på serverkonfigurationslaget. Sørg for, at gamle links kan 301 omdirigeres til de nye adresser, for dette er afgørende for SEO.

## Resumé

For HagiCode var migreringen fra Docusaurus til Astro ikke kun en rammeopgradering. Det var også en praktisk implementering af en "performance first"-filosofi.

**Hvad vi fik:**
*   **Enestående Lighthouse-resultater**: Efter migreringen nærmede HagiCode-webstedets præstationsscore sig let til en perfekt score.
*   **Hurtigere byggehastighed**: Astros trinvise byggefunktioner halverer udgivelsestiden for dokumentationsopdateringer.
*   **Bevaret fleksibilitet**: Med Islands Architecture ofrede vi ikke nogen interaktive funktioner og kunne stadig bruge React, hvor det var nødvendigt.

Hvis du også vedligeholder et dokumentationsorienteret websted og kæmper med bundtstørrelse eller indlæsningshastighed, er Astro værd at prøve. Selvom migreringsprocessen kræver en del operation, såsom at omdøbe PCode til HagiCode og flytte komponenter over én efter én, gør den silkebløde brugeroplevelse, du får til gengæld, det absolut umagen værd.

Byggesystemet, der deles i denne artikel, er den nøjagtige tilgang, vi udviklede gennem ægte forsøg og fejl, mens vi byggede HagiCode. Hvis du finder denne tilgang værdifuld, siger det noget om vores tekniske styrke, og HagiCode i sig selv er nok også værd at se nærmere på.
