## Komplet vejledning til Codex SDK Console Message Parsing

> Denne artikel forklarer Codex SDK-hændelsesstrømmekanismen, meddelelsestypeparsing og bedste praksis i rigtige projekter, hvilket hjælper udviklere med hurtigt at mestre kernefærdighederne bag AI-udførelsestjenester.

## Baggrund

Når vi bygger en AI-udførelsestjeneste baseret på Codex SDK, støder vi uundgåeligt ind i et praktisk spørgsmål: hvordan skal vi håndtere de streamede hændelsesmeddelelser, der returneres af Codex? Disse meddelelser indeholder vigtige oplysninger såsom udførelsesstatus, outputindhold og fejldetaljer, så de fortjener omhyggelig håndtering.

Som en del af HagiCode-projektet havde vi brug for en pålidelig eksekutør til AI-kodningsassistentscenarier. Det er præcis derfor, vi besluttede at studere Codex SDK-hændelsesstrømme-mekanismen i dybden. Når alt kommer til alt, kun ved at forstå, hvordan de underliggende meddelelser fungerer, kan vi opbygge en virkelig virksomhedskvalitet AI-udførelsesplatform.

Codex SDK er en programmeringsassistance SDK udgivet af OpenAI. Det returnerer eksekveringsresultater gennem en hændelsesstrøm. I modsætning til den traditionelle anmodning-svar-model bruger Codex streamede begivenheder, så vi kan:

- Få eksekveringsfremskridt i realtid
- Håndter fejl omgående
- Få detaljerede tokenbrugsstatistikker
- Understøtte langvarige komplekse opgaver

Forståelse af disse hændelsestyper og parsing af dem korrekt er afgørende for at implementere en fuldt egnet AI-eksekutor. I sidste ende er der ingen, der ønsker at arbejde med en sort boks.

## Event Stream Mekanisme

### Grundlæggende koncepter

Codex SDK bruger `thread.runStreamed()` metode til at returnere en asynkron hændelseiterator:

```typescript
import { Codex } from '@openai/codex-sdk';

const client = new Codex({
  apiKey: process.env.CODEX_API_KEY,
  baseUrl: process.env.CODEX_BASE_URL,
});

const thread = client.startThread({
  workingDirectory: '/path/to/project',
  skipGitRepoCheck: false,
});

const { events } = await thread.runStreamed('your prompt here', {
  outputSchema: {
    type: 'object',
    properties: {
      output: { type: 'string' },
      status: { type: 'string', enum: ['ok', 'action_required'] },
    },
    required: ['output', 'status'],
  },
});

for await (const event of events) {
  // Handle each event
}
```

### Detaljeret forklaring af begivenhedstyper

| Begivenhedstype | Beskrivelse | Nøgledata |
|---------|------|----------|
| `thread.started` | Tråden startede med succes | `thread_id` |
| `item.updated` | Beskedindhold opdateret | `item.text` |
| `item.completed` | Besked afsluttet | `item.text` |
| `turn.completed` | Udførelse afsluttet | `usage` (tokenbrug) |
| `turn.failed` | Udførelse mislykkedes | `error.message` |
| `error` | Fejlhændelse | `message` |

I rigtige projekter er HagiCodes eksekveringskomponent bygget oven på disse hændelsestyper. Vi er nødt til at håndtere hver slags begivenhed omhyggeligt for at sikre en smidig brugeroplevelse. Gode ​​systemer bygges ved at tage detaljer alvorligt.

## Implementering af beskedparsing

### Udpakning af beskedindhold

Beskedindhold udtrækkes gennem en hændelseshandler:

```typescript
private handleThreadEvent(event: ThreadEvent, onMessage: (content: string) => void): void {
  // Only handle message update and completion events
  if (event.type !== 'item.updated' && event.type !== 'item.completed') {
    return;
  }

  // Only handle agent message content
  if (event.item.type !== 'agent_message') {
    return;
  }

  // Extract text content
  onMessage(event.item.text);
}
```

Nøglepunkter:
- Kun håndtag `item.updated` og `item.completed` begivenheder
- Håndter kun indhold af type `agent_message`
- Beskedens indhold er i `event.item.text` felt

### Struktureret outputparsing

Codex understøtter JSON-struktureret output. Du kan angive returformatet gennem `outputSchema` parameter:

```typescript
const DEFAULT_OUTPUT_SCHEMA = {
  type: 'object',
  properties: {
    output: { type: 'string' },
    status: { type: 'string', enum: ['ok', 'action_required'] },
  },
  required: ['output', 'status'],
  additionalProperties: false,
} as const;
```

Parsing-funktionen forsøger at parse JSON, og hvis det mislykkes, falder den tilbage til den rå tekst.

```typescript
function toStructuredOutput(raw: string): StructuredOutput {
  try {
    const parsed = JSON.parse(raw) as Partial<StructuredOutput>;
    if (typeof parsed.output === 'string') {
      return {
        output: parsed.output,
        status: parsed.status === 'action_required' ? 'action_required' : 'ok',
      };
    }
  } catch {
    // JSON parsing failed, fall back to the raw text
  }

  return {
    output: raw,
    status: 'ok',
  };
}
```

### Fuldfør hændelseshåndteringsflow

```typescript
private async runWithStreaming(
  thread: Thread,
  input: CodexStageExecutionInput
): Promise<{ output: string; usage: Usage | null }> {
  const abortController = new AbortController();
  const timeoutHandle = setTimeout(() => {
    abortController.abort();
  }, Math.max(1000, input.timeoutMs));

  let latestMessage = '';
  let usage: Usage | null = null;
  let emittedLength = 0;

  try {
    const { events } = await thread.runStreamed(input.prompt, {
      outputSchema: DEFAULT_OUTPUT_SCHEMA,
      signal: abortController.signal,
    });

    for await (const event of events) {
      // Handle message content
      this.handleThreadEvent(event, (nextContent) => {
        const delta = nextContent.slice(emittedLength);
        if (delta.length > 0) {
          emittedLength = nextContent.length;
          input.callbacks?.onChunk?.(delta);  // Streaming callback
        }
        latestMessage = nextContent;
      });

      // Process different data based on the event type
      if (event.type === 'thread.started') {
        this.threadId = event.thread_id;
      } else if (event.type === 'turn.completed') {
        usage = event.usage;
      } else if (event.type === 'turn.failed') {
        throw new CodexExecutorError('gateway_unavailable', event.error.message, true);
      } else if (event.type === 'error') {
        throw new CodexExecutorError('gateway_unavailable', event.message, true);
      }
    }
  } catch (error) {
    if (abortController.signal.aborted) {
      throw new CodexExecutorError(
        'upstream_timeout',
        `Codex stage timed out after ${input.timeoutMs}ms`,
        true
      );
    }
    throw error;
  } finally {
    clearTimeout(timeoutHandle);
  }

  const structured = toStructuredOutput(latestMessage);
  return { output: structured.output, usage };
}
```

## Fejlhåndteringsstrategi

### Fejlkodekortlægning

Kortlæg specifikke fejlmønstre til konkrete fejlkoder, så de øverste lag kan håndtere dem lettere:

```typescript
function mapError(error: unknown): CodexExecutorError {
  if (error instanceof CodexExecutorError) {
    return error;
  }

  const message = error instanceof Error ? error.message : String(error);
  const normalized = message.toLowerCase();

  // Authentication errors - not retryable
  if (normalized.includes('401') ||
      normalized.includes('403') ||
      normalized.includes('api key') ||
      normalized.includes('auth')) {
    return new CodexExecutorError('auth_invalid', message, false);
  }

  // Rate limit errors - retryable
  if (normalized.includes('429') || normalized.includes('rate limit')) {
    return new CodexExecutorError('rate_limited', message, true);
  }

  // Timeout errors - retryable
  if (normalized.includes('timeout') || normalized.includes('aborted')) {
    return new CodexExecutorError('upstream_timeout', message, true);
  }

  // Default error
  return new CodexExecutorError('gateway_unavailable', message, true);
}
```

### Definitioner af fejltype

```typescript
export type CodexErrorCode =
  | 'auth_invalid'      // Authentication failure
  | 'upstream_timeout'  // Upstream timeout
  | 'rate_limited'      // Rate limited
  | 'gateway_unavailable'; // Gateway unavailable

export class CodexExecutorError extends Error {
  readonly code: CodexErrorCode;
  readonly retryable: boolean;

  constructor(code: CodexErrorCode, message: string, retryable: boolean) {
    super(message);
    this.name = 'CodexExecutorError';
    this.code = code;
    this.retryable = retryable;
  }
}
```

## Konfiguration af arbejdsmappe og miljø

### Working Directory Validering

Codex SDK kræver, at arbejdsbiblioteket er et gyldigt Git-lager.

```typescript
export function validateWorkingDirectory(
  workingDirectory: string,
  skipGitRepoCheck: boolean
): void {
  const resolvedWorkingDirectory = path.resolve(workingDirectory);

  if (!existsSync(resolvedWorkingDirectory)) {
    throw new CodexExecutorError(
      'gateway_unavailable',
      'Working directory does not exist.',
      false
    );
  }

  if (!statSync(resolvedWorkingDirectory).isDirectory()) {
    throw new CodexExecutorError(
      'gateway_unavailable',
      'Working directory is not a directory.',
      false
    );
  }

  if (skipGitRepoCheck) {
    return;
  }

  const gitDir = path.join(resolvedWorkingDirectory, '.git');
  if (!existsSync(gitDir)) {
    throw new CodexExecutorError(
      'gateway_unavailable',
      'Working directory is not a git repository.',
      false
    );
  }
}
```

### Indlæser miljøvariabler

Codex SDK skal indlæse miljøvariabler fra login-skallen, så AI-agenten kan få adgang til systemkommandoer:

```typescript
function parseEnvironmentOutput(output: Buffer): Record<string, string> {
  const parsed: Record<string, string> = {};

  for (const entry of output.toString('utf8').split('\0')) {
    if (!entry) continue;

    const separatorIndex = entry.indexOf('=');
    if (separatorIndex <= 0) continue;

    const key = entry.slice(0, separatorIndex);
    const value = entry.slice(separatorIndex + 1);
    if (key.length > 0) {
      parsed[key] = value;
    }
  }

  return parsed;
}

function tryLoadEnvironmentFromShell(shellPath: string): Record<string, string> | null {
  const result = spawnSync(shellPath, ['-ilc', 'env -0'], {
    env: process.env,
    stdio: ['ignore', 'pipe', 'pipe'],
    timeout: 5000,
  });

  if (result.error || result.status !== 0) {
    return null;
  }

  return parseEnvironmentOutput(result.stdout);
}

export function createExecutorEnvironment(
  envOverrides: Record<string, string> = {}
): Record<string, string> {
  // Load environment variables from the login shell
  const consoleEnv = loadConsoleEnvironmentFromShell();

  return {
    ...process.env,
    ...consoleEnv,
    ...envOverrides,
  };
}
```

## Komplet brugseksempel

### Grundlæggende brug

I HagiCode-projektet bruger vi følgende tilgang til at initialisere Codex-klienten og udføre opgaver:

```typescript
import { Codex } from '@openai/codex-sdk';

async function executeWithCodex(prompt: string, workingDir: string) {
  const client = new Codex({
    apiKey: process.env.CODEX_API_KEY,
    env: { PATH: process.env.PATH },
  });

  const thread = client.startThread({
    workingDirectory: workingDir,
  });

  const { events } = await thread.runStreamed(prompt);

  let result = '';
  for await (const event of events) {
    if (event.type === 'item.updated' && event.item.type === 'agent_message') {
      result = event.item.text;
    }
    if (event.type === 'turn.completed') {
      console.log('Token usage:', event.usage);
    }
  }

  // Try to parse JSON output
  try {
    const parsed = JSON.parse(result);
    return parsed.output;
  } catch {
    return result;
  }
}
```

### Fuldfør implementering med genforsøg

```typescript
export class CodexSdkExecutor {
  private readonly config: CodexRuntimeConfig;
  private readonly client: Codex;
  private threadId: string | null = null;

  async executeStage(input: CodexStageExecutionInput): Promise<CodexStageExecutionResult> {
    const maxAttempts = Math.max(1, this.config.retryCount + 1);
    let attempt = 0;
    let lastError: CodexExecutorError | null = null;

    while (attempt < maxAttempts) {
      attempt += 1;

      try {
        const thread = this.getThread(input.workingDirectory);
        const { output, usage } = await this.runWithStreaming(thread, input);

        return {
          output,
          usage,
          threadId: this.threadId!,
          attempts: attempt,
          latencyMs: Date.now() - startedAt,
        };
      } catch (error) {
        const mappedError = mapError(error);
        lastError = mappedError;

        // Non-retryable error or max retry attempts reached
        if (!mappedError.retryable || attempt >= maxAttempts) {
          throw mappedError;
        }

        // Wait before retrying
        await new Promise(resolve => setTimeout(resolve, 1000 * attempt));
      }
    }

    throw lastError!;
  }
}
```

## Bedste praksis

### 1. Krav til arbejdskatalog

- Sørg for, at arbejdsbiblioteket er et gyldigt Git-lager
- Brug `PROJECT_ROOT` miljøvariabel for at specificere den eksplicit
- Under debugging kan du indstille `CODEX_SKIP_GIT_REPO_CHECK=true` at springe kontrollen over

### 2. Konfiguration af miljøvariabel

- Send kun de nødvendige miljøvariabler gennem en hvidlistemekanisme
- Brug login-skallen til at indlæse hele miljøet
- Undgå at videregive følsomme oplysninger

### 3. Timeout og strategi for genforsøg

- Indstil rimelige timeouts baseret på opgavens kompleksitet
- Implementer eksponentiel backoff for fejl, der kan prøves igen
- Registrer antallet af genforsøg og årsager

### 4. Fejlhåndtering

- Skelne mellem fejl, der kan prøves igen og ikke-gentages
- Giv klare fejlmeddelelser og forslag
- Brug forenede fejlkoder, så de øverste lag kan håndtere dem konsekvent

### 5. Streaming output

- Implementer trinvise output-tilbagekald for at forbedre brugeroplevelsen
- Håndter trinvise meddelelsesopdateringer korrekt
- Registrer tokenbrug til omkostningsanalyse

I det faktiske produktionsmiljø for HagiCode-projektet har vi allerede verificeret effektiviteten af de bedste praksisser ovenfor. Denne tilgang har hjulpet os med at opbygge en stabil og pålidelig AI-udførelsestjeneste. I sidste ende betyder praktisk validering mere end teori alene.
