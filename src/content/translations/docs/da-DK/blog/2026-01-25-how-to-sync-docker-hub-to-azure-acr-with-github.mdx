## Automatisering af billedsynkronisering fra Docker Hub til Azure ACR

> Denne artikel forklarer, hvordan du bruger GitHub Actions og billedsynkroniseringsværktøjet til at automatisere billedsynkronisering fra Docker Hub til Azure Container Registry, hvilket adresserer den langsomme adgang til Docker Hub i Kina og nogle Azure-regioner, samtidig med at billedtilgængeligheden og implementeringseffektiviteten forbedres i Azure-miljøer.

## Baggrund / Introduktion

HagiCode-projektet bruger Docker-billeder som dets kerne-runtime-komponenter, med de primære billeder hostet på Docker Hub. Efterhånden som projektet er vokset, og Azure-implementeringsbehovet er steget, stødte vi på følgende smertepunkter:

- Langsomme billedtrækhastigheder med Docker Hub-adgang begrænset i Kina og nogle Azure-regioner
- En enkelt billedkilde skaber en enkelt fejlrisiko
- Azure Container Registry giver bedre netværksydelse og integration, når de kører i Azure

For at løse disse problemer havde vi brug for en automatiseret billedsynkroniseringsmekanisme til regelmæssigt at synkronisere billeder fra Docker Hub til Azure ACR, hvilket sikrer, at brugere kan få hurtigere billedudtrækningshastigheder og højere tilgængelighed i Azure-miljøer.

## Sammenligning af tekniske løsninger

Da vi definerede løsningen, sammenlignede vi flere tekniske muligheder:

### 1. billedsynkronisering (endelig valg)
- Inkrementel synkronisering: synkroniserer kun ændrede billedlag, hvilket reducerer netværksoverførslen markant
- Genoptag support: synkroniseringen kan fortsætte efter netværksafbrydelser
- Samtidig kontrol: Konfigurerbare arbejdstråde forbedrer effektiviteten ved synkronisering af store billeder
- Robust fejlhåndtering: indbygget genforsøgsmekanisme for fejl (3 genforsøg som standard)
- Letvægtsimplementering: en enkelt binær uden afhængigheder
- Multi-registry support: kompatibel med Docker Hub, Azure ACR, Harbor og mere

### 2. Docker CLI
- Ingen trinvis synkronisering: det fulde billedindhold skal trækkes hver gang
- Lavere effektivitet: stor netværksoverførselsvolumen og længere udførelsestid
- Enkel at bruge: baseret på velkendte `docker pull` og `docker push` kommandoer

### 3. Azure CLI
- Højere kompleksitet: Azure CLI-godkendelse skal konfigureres
- Begrænset funktionalitet: `az acr import` er relativt snævert i omfang
- Indbygget integration: integreres godt med Azure-tjenester

## Implementeringstrin

### 1. Forberedelse
- Opret eller bekræft et Azure Container Registry i Azure Portal
- Opret ACR-adgangsoplysninger (brugernavn og adgangskode)
- Bekræft adgangstilladelser til Docker Hub-billedlageret

### 2. Konfigurer GitHub Secrets
Tilføj følgende hemmeligheder i indstillingerne for GitHub-lageret:
- AZURE_ACR_USERNAME: Azure ACR-brugernavn
- AZURE_ACR_PASSWORD: Azure ACR-adgangskode

### 3. Opret GitHub Actions-arbejdsgangen
Konfigurer arbejdsgangen i `.github/workflows/sync-docker-acr.yml`:
- Planlagt trigger: hver dag kl. 00:00 UTC
- Manuel udløser: understøtter `workflow_dispatch`
- Ekstra trigger: kører på skub til `publish` gren til hurtig synkronisering

### 4. Workflow eksekveringsflow

| Trin | Deltager | Handling | Beskrivelse |
| --- | --- | --- | --- |
| 1 | GitHub-handlinger | Udløs arbejdsgangen | Udløst af tidsplan, manuel udførelse eller push til `publish` gren |
| 2 | GitHub-handlinger → billedsynkronisering | Download og kør synkroniseringsværktøjet | Går ind i den egentlige synkroniseringsfase |
| 3 | billedsynkronisering → Docker Hub | Hent billedmanifester og tagliste | Læser metadata fra kildelageret |
| 4 | billedsynkronisering → Azure ACR | Hent eksisterende billeddata fra mållageret | Bestemmer den aktuelle tilstand for målsiden |
| 5 | billedsynkronisering | Sammenlign kilde- og målforskelle | Identificerer de billedlag, der skal synkroniseres |
| 6 | billedsynkronisering → Docker Hub | Træk ændrede billedlag | Overfører kun det indhold, der skal opdateres |
| 7 | billedsynkronisering → Azure ACR | Skub ændrede billedlag | Fuldfører trinvis synkronisering |
| 8 | billedsynkronisering → GitHub-handlinger | Returner synkroniseringsstatistikker | Indeholder resultater, forskelle og fejlinformation |
| 9 | GitHub-handlinger | Optag logfiler og upload artefakter | Understøtter senere revision og fejlfinding |

## Implementering af GitHub Actions Workflow

Nedenfor er den faktiske workflow-konfiguration, der er i brug (`.github/workflows/sync-docker-acr.yml`):

```yaml
name: Sync Docker Image to Azure ACR

on:
  schedule:
    - cron: "0 0 * * *" # Every day at 00:00 UTC
  workflow_dispatch: # Manual trigger
  push:
    branches: [publish]

permissions:
  contents: read

jobs:
  sync:
    runs-on: ubuntu-latest

    steps:
      - name: Checkout code
        uses: actions/checkout@v4

      - name: Download image-syncer
        run: |
          # Download the image-syncer binary
          wget https://github.com/AliyunContainerService/image-syncer/releases/download/v1.5.5/image-syncer-v1.5.5-linux-amd64.tar.gz
          tar -zxvf image-syncer-v1.5.5-linux-amd64.tar.gz
          chmod +x image-syncer

      - name: Create auth config
        run: |
          # Generate the authentication config file (YAML format)
          cat > auth.yaml <<EOF
          hagicode.azurecr.io:
            username: "${{ secrets.AZURE_ACR_USERNAME }}"
            password: "${{ secrets.AZURE_ACR_PASSWORD }}"
          EOF

      - name: Create images config
        run: |
          # Generate the image sync config file (YAML format)
          cat > images.yaml <<EOF
          docker.io/newbe36524/hagicode: hagicode.azurecr.io/hagicode
          EOF

      - name: Run image-syncer
        run: |
          # Run synchronization (using the newer --auth and --images parameters)
          ./image-syncer --auth=./auth.yaml --images=./images.yaml --proc=10 --retries=3

      - name: Upload logs
        if: always()
        uses: actions/upload-artifact@v4
        with:
          name: sync-logs
          path: image-syncer-*.log
          retention-days: 7
```

## Konfigurationsdetaljer

### 1. Triggerbetingelser
- Planlagt trigger: `cron: "0 0 * * *"` kører hver dag kl. 00:00 UTC
- Manuel udløser: `workflow_dispatch` giver brugerne mulighed for at køre det manuelt fra GitHub UI
- Tryk på aftrækkeren: `push: branches: [publish]` udløses ved tryk til udløsergrenen for hurtig synkronisering

### 2. Godkendelseskonfiguration (`auth.yaml`)
```yaml
hagicode.azurecr.io:
  username: "${{ secrets.AZURE_ACR_USERNAME }}"
  password: "${{ secrets.AZURE_ACR_PASSWORD }}"
```

### 3. Billedsynkroniseringskonfiguration
```yaml
docker.io/newbe36524/hagicode: hagicode.azurecr.io/hagicode
```
Denne konfiguration betyder, at alle tags fra `docker.io/newbe36524/hagicode` vil blive synkroniseret til `hagicode.azurecr.io/hagicode`.

### 4. billedsynkroniseringsparametre
- `--auth=./auth.yaml`: sti til godkendelseskonfigurationsfilen
- `--images=./images.yaml`: sti til billedsynkroniseringskonfigurationsfilen
- `--proc=10`: brug `10` samtidige arbejdere
- `--retries=3`: Prøv igen fejl `3` gange

## Noter og bedste praksis

### 1. Sikkerhedsanbefalinger
- Roter Azure ACR-adgangskoder regelmæssigt (anbefales hver 90. dag)
- Brug en dedikeret ACR-tjenestekonto med kun push-tilladelser
- Overvåg ACR-adgangslogfiler for at opdage unormal adgang i tide
- Udfør ikke legitimationsoplysninger i logfiler
- Overgiv ikke legitimationsoplysninger til kodelageret

### 2. Performance optimering
- Juster `--proc` parameter: tune samtidighed baseret på netværksbåndbredde (anbefales `5-20`)
- Overvåg synkroniseringsvarighed: Hvis synkroniseringen tager for lang tid, kan du overveje at sænke samtidighed
- Ryd op logs regelmæssigt: Indstil en rimelig `retention-days` værdi (`7` i øjeblikket)

### 3. Fejlfinding

#### Problem 1: Godkendelsesfejl
```
Error: failed to authenticate to hagicode.azurecr.io
```
Løsning:
1. Tjek om GitHub Secrets er konfigureret korrekt
2. Bekræft, om Azure ACR-adgangskoden er udløbet
3. Bekræft, at tilladelserne til ACR-tjenestekontoen er korrekte

#### Problem 2: Netværkstimeout
```
Error: timeout waiting for response
```
Løsning:
1. Tjek netværksforbindelsen
2. Reducer antallet af samtidige arbejdere (den `--proc` parameter)
3. Vent på, at netværket gendannes, og kør arbejdsgangen igen

#### Problem 3: Ufuldstændig billedsynkronisering
```
Warning: some tags failed to sync
```
Løsning:
1. Tjek synkroniseringslogfilerne for at identificere fejlbehæftede tags
2. Kør arbejdsgangen manuelt igen for at synkronisere igen
3. Bekræft, at kildebilledet på Docker Hub er sundt

### 4. Overvågning og alarmering
- Tjek jævnligt siden Handlinger for at bekræfte status for arbejdsgangen
- Aktiver GitHub-meddelelser, så workflowfejl rapporteres omgående
- Overvåg Azure ACR-lagerbrug
- Kontroller regelmæssigt tagkonsistens på tværs af registre

## Ofte stillede spørgsmål og løsninger

### Q1: Hvordan synkroniserer jeg specifikke tags i stedet for alle tags?

Rediger `images.yaml` konfigurationsfil:
```yaml
# Sync only the latest and v1.0 tags
docker.io/newbe36524/hagicode:latest: hagicode.azurecr.io/hagicode:latest
docker.io/newbe36524/hagicode:v1.0: hagicode.azurecr.io/hagicode:v1.0
```

### Q2: Hvordan synkroniserer jeg flere billedarkiver?

Tilføj flere linjer til `images.yaml`:
```yaml
docker.io/newbe36524/hagicode: hagicode.azurecr.io/hagicode
docker.io/newbe36524/another-image: hagicode.azurecr.io/another-image
```

### Q3: Hvordan prøver jeg igen, efter at synkroniseringen mislykkes?
- Automatisk genforsøg: billedsynkronisering inkluderer en indbygget genforsøgsmekanisme (`3` genforsøg som standard)
- Manuelt forsøg igen: klik `Re-run all jobs` på siden GitHub Actions

### Q4: Hvordan kan jeg se detaljerede synkroniseringsfremskridt?
- Se logfiler i realtid på siden Handlinger
- Download `sync-logs` artefakt for at inspicere de fulde logfiler
- Logfilerne inkluderer synkroniseringsstatus og overførselshastighed for hvert tag

### Q5: Hvor lang tid tager synkronisering?
- Indledende fuld synkronisering: normalt `10-30` minutter afhængig af billedstørrelse
- Inkrementel synkronisering: normalt `2-5` minutter, hvis billedændringerne er små
- Timing afhænger af netværkets båndbredde, billedstørrelse og samtidighedsindstillinger
