## Opbygning af elegant ny brugeronboarding i React-projekter: HagiCodes driver.js-praksis

> Når brugerne åbner dit produkt for første gang, ved de så virkelig, hvor de skal starte? I denne artikel vil jeg fortælle lidt om, hvordan vi brugte driver.js til onboarding af nye brugere i HagiCode-projektet. Betragt det som et lille praktisk eksempel for at få gang i samtalen.

## Baggrund

Er du nogensinde løbet ind i denne situation? En ny bruger tilmelder sig dit produkt, åbner siden og ser straks tabt ud. De scanner rundt, usikre på, hvad de skal klikke på, eller hvad de skal gøre næste gang. Som udviklere antager vi ofte, at brugerne bare vil "udforske på egen hånd", fordi menneskelig nysgerrighed trods alt er ubegrænset. Men virkeligheden er anderledes: De fleste brugere går stille og roligt inden for få minutter, hvis de ikke kan finde det rigtige indgangspunkt, som om historien begynder pludseligt og slutter lige så naturligt.

Onboarding af nye brugere er en vigtig måde at løse dette problem på, men at bygge det godt er ikke så enkelt. Et godt onboarding-system skal:

- Find sideelementer præcist og fremhæv dem
- Understøttelse af flertrins onboarding-flows
- Husk brugerens valg (fuldfør eller spring over)
- Undgå at påvirke sidens ydeevne og normal interaktion
- Hold kodestrukturen klar og nem at vedligeholde

Mens vi byggede HagiCode, løb vi ind i den samme udfordring. HagiCode er en AI-kodningsassistent, og dens kernearbejdsgang er en OpenSpec-arbejdsgang, der ser sådan ud: "brugeren opretter et forslag -> AI'en genererer en plan -> brugeren gennemgår den -> AI'en udfører den." For brugere, der møder dette koncept for første gang, er arbejdsgangen helt ny, så de har brug for solid onboarding for at komme hurtigt i gang. Nye ting tager altid lidt tid at vænne sig til.

HagiCode er en Claude-baseret AI-kodningsassistent, der hjælper udviklere med at fuldføre kodningsopgaver mere effektivt gennem OpenSpec-workflowet. ## Hvorfor vi valgte driver.js

Under den tekniske evalueringsfase så vi på flere almindelige onboarding-biblioteker. Hver af dem havde sine egne styrker:

- **Intro.js**: Kraftig, men relativt stor, og stiltilpasning er noget kompleks
- **Shepherd.js**: Veldesignet API, men en smule for "tung" til vores brug
- **driver.js**: Let, kortfattet, intuitiv API og fungerer godt i React-økosystemet

Til sidst valgte vi driver.js. Der var ingen særlig dramatisk grund. Valget kom hovedsageligt til grund af disse overvejelser:

1. **Letvægt**: Kernebiblioteket er lille og øger ikke bundtstørrelsen markant
2. **Simpel API**: Konfigurationen er klar og intuitiv, så den er nem at samle op
3. **Fleksibel**: Understøtter tilpasset positionering, styling og interaktionsadfærd
4. **Dynamisk import**: Kan indlæses efter behov uden at påvirke ydeevnen på første skærm

Med teknologivalg er der sjældent et universelt bedste svar. Normalt er der kun den mulighed, der passer bedst.

## Teknisk implementering

### Kernekonfiguration

driver.js har en meget intuitiv konfigurationsmodel. Her er den kernekonfiguration, vi bruger i HagiCode-projektet:

```typescript
import { driver } from 'driver.js';
import 'driver.js/dist/driver.css';

const newConversationDriver = driver({
  allowClose: true,           // Allow users to close the guide
  animate: true,              // Enable animations
  overlayClickBehavior: 'close', // Close the guide when the overlay is clicked
  disableActiveInteraction: false, // Keep elements interactive
  showProgress: false,        // Do not show the progress bar (we manage progress ourselves)
  steps: guideSteps           // Array of guide steps
});
```

Begrundelsen bag disse indstillinger er:

- `allowClose: true` - Respekter brugerens valg og tving dem ikke til at afslutte vejledningen
- `disableActiveInteraction: false` - Nogle trin kræver rigtige brugerhandlinger, såsom indtastning, så interaktion kan ikke deaktiveres
- `overlayClickBehavior: 'close'` - Giv brugerne en hurtig måde at afslutte

### Statsledelse

Vedvarende onboarding-tilstand er kritisk. Vi ønsker ikke at genstarte guiden hver gang siden opdateres, for det bliver hurtigt irriterende. HagiCode bruger `localStorage` for at administrere guidetilstand:

```typescript
export type GuideState = 'pending' | 'dismissed' | 'completed';

export interface UserGuideState {
  session: GuideState;
  detailGuides: Record<string, GuideState>;
}

// Read state
export const getUserGuideState = (): UserGuideState => {
  const state = localStorage.getItem('userGuideState');
  return state ? JSON.parse(state) : { session: 'pending', detailGuides: {} };
};

// Update state
export const setUserGuideState = (state: UserGuideState) => {
  localStorage.setItem('userGuideState', JSON.stringify(state));
};
```

Vi definerer tre tilstande:

- `pending`: Vejledningen er stadig i gang, og brugeren har ikke fuldført eller sprunget over den
- `dismissed`: Brugeren lukkede vejledningen proaktivt
- `completed`: Brugeren fuldførte alle trin

For forslags detaljeside onboarding understøtter vi også mere finkornet tilstandssporing gennem `detailGuides` kort, fordi et forslag kan gennemgå flere stadier - udkast, gennemgang og udførelse færdig - og hver fase har brug for forskellig vejledning. Tingenes tilstand ændrer sig altid, og onboarding bør afspejle det.

### Valg af målelement

driver.js bruger CSS-vælgere til at finde målelementer. HagiCode følger en simpel konvention: brug en brugerdefineret `data-guide` egenskab for at markere onboarding-mål:

```typescript
const steps = [
  {
    element: '[data-guide="launch"]',
    popover: {
      title: 'Start a New Conversation',
      description: 'Click here to create a new conversation session...'
    }
  }
];
```

I komponenter ser det sådan ud:

```typescript
<button data-guide="launch" onClick={handleLaunch}>
  New Conversation
</button>
```

Fordelene ved denne tilgang er:

- Undgå konflikter med business styling klassenavne
- Tydelig semantik, så du med det samme kan se, at elementet er relateret til onboarding
- Lettere at administrere og vedligeholde konsekvent

### Dynamisk importoptimering

Fordi onboarding-funktionen kun er nødvendig i specifikke scenarier, såsom en brugers første besøg, bruger vi dynamiske importer til at optimere den indledende indlæsningsydelse:

```typescript
const initNewUserGuide = async () => {
  // Dynamically import driver.js
  const { driver } = await import('driver.js');
  await import('driver.js/dist/driver.css');

  // Initialize the guide
  const newConversationDriver = driver({
    // ...configuration
  });

  newConversationDriver.drive();
};
```

På denne måde indlæses driver.js og dets stylesheet kun, når det er nødvendigt og påvirker ikke ydeevnen på første skærm. Ikke mange mennesker nyder at vente på noget, de ikke engang har brug for endnu.

## Onboarding Flow Design

HagiCode implementerer to onboarding-stier, der dækker brugerens kernescenarier.

### Session onboarding (10 trin)

Denne onboarding-sti hjælper brugere med at fuldføre hele flowet fra oprettelse af en samtale til indsendelse af deres første komplette forslag:

1. **lancering** - Start guiden og introducer knappen "Ny samtale".
2. **skriv** - Guide brugeren til at skrive en anmodning i inputfeltet
3. **send** - Guide brugeren til at klikke på send-knappen
4. **proposal-launch-readme** - Guide brugeren til at oprette et README-forslag
5. **proposal-compose-readme** - Guide brugeren til at redigere README-anmodningsindholdet
6. **proposal-submit-readme** - Guide brugeren til at indsende README-forslaget
7. **proposal-launch-agents** - Guide brugeren til at oprette en `AGENTS.md` Forslag
8. **proposal-compose-agents** - Guide brugeren til at redigere `AGENTS.md` anmodning
9. **proposal-submit-agents** - Guide brugeren til at indsende `AGENTS.md` Forslag
10. **proposal-wait** - Forklar, at AI'en behandler, og brugeren skal vente et øjeblik

Ideen bag denne vej er at lade brugerne opleve HagiCodes kernearbejdsgang på egen hånd gennem to reelle forslagsoprettelsesopgaver, en for `README` og en til `AGENTS.md`. Der er stor forskel på at høre om en arbejdsgang og selv at gennemgå den.

### Onboarding af forslagsdetaljer (3 trin)

Når brugere går ind på Forslagets detaljeside, udløser HagiCode den tilsvarende vejledning baseret på Forslagets aktuelle tilstand:

1. **drafting** (draft stage) - Guide brugeren til at gennemgå den AI-genererede plan
2. **gennemgang** (gennemgangsfase) - Guide brugeren til at udføre planen
3. **executionCompleted** (afsluttet fase) - Guide brugeren til at arkivere planen

Det afgørende kendetegn ved denne vejledning er, at den er statsstyret: den bestemmer dynamisk, hvilket onboarding-trin, der skal vises, baseret på forslagets faktiske tilstand. Ting ændrer sig, og onboarding bør ændre sig med dem.

### Element Render Retry Mechanism

I React-applikationer er mål-onboarding-elementet muligvis ikke færdiggjort endnu, for eksempel fordi asynkrone data stadig indlæses. For at håndtere dette implementerer HagiCode en genforsøgsmekanisme:

```typescript
const waitForElement = (selector: string, maxRetries = 10, interval = 100) => {
  let retries = 0;
  return new Promise<HTMLElement>((resolve, reject) => {
    const checkElement = () => {
      const element = document.querySelector(selector) as HTMLElement;
      if (element) {
        resolve(element);
      } else if (retries < maxRetries) {
        retries++;
        setTimeout(checkElement, interval);
      } else {
        reject(new Error(`Element not found: ${selector}`));
      }
    };
    checkElement();
  });
};
```

Kald denne funktion før initialisering af guiden for at sikre, at målelementet allerede eksisterer. Nogle gange er det det værd at vente lidt længere.

## Oversigt over bedste praksis

Baseret på HagiCodes praktiske erfaring er her et par vigtige bedste praksisser:

### 1. Onboarding skal være let at afslutte

Tving ikke brugere til at fuldføre onboarding. Nogle brugere er opdagelsesrejsende af natur og foretrækker at finde ud af tingene på egen hånd. Angiv en tydelig "Spring"-knap, husk deres valg, og afbryd dem ikke igen næste gang.

### 2. Hold onboarding-indhold kort og skarpt

Hvert onboarding-trin skal fokusere på et enkelt mål:

- **Titel**: Kort og tydeligt, ideelt set ikke mere end et par ord
- **Beskrivelse**: Gå direkte til sagen og fortæl brugeren, hvad dette er, og hvorfor det betyder noget

Undgå lange forklaringer. Brugernes opmærksomhed er begrænset under onboarding, og jo mere du siger, desto mindre er sandsynligheden for, at de læser det.

### 3. Hold vælgere stabile

Brug stabile elementmarkører, der ikke ændres ofte. Skikken `data-guide` attribut er et godt valg. Undgå afhængighed af klassenavne eller DOM-struktur, fordi de er nemme at ændre under refactors. Koden ændres hele tiden, men nogle ankre bør forblive stabile, når det er muligt.

### 4. Test din onboarding

HagiCode inkluderer komplette testcases for onboarding-funktionen:

```typescript
describe('NewUserConversationGuide', () => {
  it('should initialize guide state correctly', () => {
    const state = getUserGuideState();
    expect(state.session).toBe('pending');
  });

  it('should update guide state correctly', () => {
    setUserGuideState({ session: 'completed', detailGuides: {} });
    const state = getUserGuideState();
    expect(state.session).toBe('completed');
  });
});
```

Tests hjælper med at sikre, at refactoring ikke ved et uheld bryder onboarding-adfærden. Ingen ønsker en lille kodeændring for stille og roligt at skade tidligere fungerende funktionalitet.

### 5. Ydeevneoptimering

- Brug dynamisk import til at doven indlæse onboarding-biblioteket
- Undgå at initialisere onboarding-logik, efter at brugeren allerede har gennemført vejledningen
- Overvej virkningen af animationers ydeevne, og deaktiver dem på lavere enheder, hvis det er nødvendigt

Ydeevne, som mange ting i livet, fortjener en smule omhyggelig budgettering.

## Konklusion

Onboarding af nye brugere er en vigtig del af en forbedring af produktets brugeroplevelse. I HagiCode-projektet brugte vi driver.js til at bygge et komplet onboarding-system, der dækker hele arbejdsgangen fra oprettelse af session til udførelse af forslag.

De kernepunkter, vi håber at dele gennem denne artikel, er:

1. **Tekniske valg bør matche de faktiske behov**: driver.js er ikke den mest effektive løsning, men den passer bedst til os
2. **Statsledelse er kritisk**: Brug `localStorage` for at fortsætte onboarding-tilstand og undgå gentagne gange at afbryde brugere
3. **Onboarding-design skal forblive fokuseret**: Hvert trin skal løse ét problem og ikke mere
4. **Kodestrukturen bør forblive klar**: Separat onboarding-konfiguration, tilstandsadministration og UI-logik for at gøre vedligeholdelse lettere

Hvis du tilføjer ny brugeronboarding til dit eget projekt, håber jeg, at den praktiske erfaring i denne artikel hjælper. Der er ikke noget særligt mystisk ved denne form for teknologi. Bliv ved med at prøve, fortsæt med at opsummere, hvad du lærer, og tingene bliver gradvist nemmere.
