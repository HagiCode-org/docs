## Fuld GLM-5.1-understøttelse og Gemini CLI-integration: HagiCode's Path of Multi-Model Evolution

> Denne artikel introducerer to store seneste opdateringer til HagiCode-platformen: fuld understøttelse af Zhipu AI GLM-5.1-modellen og den vellykkede integration af Gemini CLI som den tiende Agent CLI. Tilsammen styrker disse opdateringer platformens multi-model-kapacitet og multi-CLI-økosystem yderligere.

:::note[Bemærkning om tidskontekst]
Denne artikel registrerer multi-CLI-udvidelsesfremskridtet, som det var på udgivelsestidspunktet. For det nuværende altid understøttede omfang henvises til [AI Agent CLI installation](/da-DK/related-software-installation/ai-agent-cli/) og den [Installationsvejledning](/da-DK/installation); den `IFlowCli` nævnt her repræsenterer kun en historisk konfiguration på det tidspunkt og er ikke en del af det nuværende aktive supportomfang.
:::

## Baggrund

Tiden går virkelig hurtigt. Udviklingen af ​​store sprogmodeller har været stigende som bambus i foråret. For ikke længe siden jublede vi stadig over "en AI, der kan skrive kode." Nu er vi allerede i en æra med multi-model samarbejde og multi-værktøj integration. Er det spændende? Måske. Det, som udviklere har brug for, har jo aldrig kun været selve værktøjet, men letheden ved at tilpasse sig forskellige scenarier og skifte fleksibelt, når det er nødvendigt.

Som en AI-assisteret kodningsplatform har HagiCode for nylig hilst to vigtige udviklinger velkommen: For det første den fulde integration af Zhipu AI's GLM-5.1-model; for det andet den officielle tilføjelse af Gemini CLI som den tiende understøttede Agent CLI. Disse to opdateringer lyder måske ikke jordrystende, men de er utvivlsomt gode nyheder for platformens fortsatte modning.

GLM-5.1 er Zhipu AIs seneste flagskibsmodel. Sammenlignet med GLM-5.0 tilbyder den stærkere ræsonnement, dybere kodeforståelse og mere jævne værktøjsopkald. Endnu vigtigere er det den første GLM-model, der understøtter billedinput. Hvad betyder det? Det betyder, at brugere kan lade AI se direkte på et skærmbillede i stedet for at kæmpe for at beskrive problemet med ord. Når du først har brugt den bekvemmelighed, forstår du straks dens værdi.

Samtidig gennem `HagiCode.Libs.Providers` arkitektur integrerede HagiCode med succes Gemini CLI i platformen. Dette er nu den tiende Agent CLI. For at være ærlig giver det at komme til dette punkt en beskeden følelse af præstation.

Det er også værd at nævne, at HagiCodes billedoverførselsfunktion lader brugere kommunikere med AI direkte gennem skærmbilleder. Selv når man kører GLM 4.7, fungerer platformen stadig godt og har allerede hjulpet med at udføre mange vigtige byggeopgaver. Hvad angår GLM-5.1, går det naturligvis et skridt videre.

## Om HagiCode

Den tilgang, der er delt i denne artikel, kommer fra vores praktiske erfaring i [HagiCode](https://hagicode.com) projekt. HagiCode er en open source AI-assisteret kodningsplatform designet til at give udviklere en fleksibel og kraftfuld AI-programmeringsassistent gennem en multi-model, multi-CLI-arkitektur. Projektlager: [github.com/HagiCode-org/site](https://github.com/HagiCode-org/site)

## Multi-CLI arkitektur design

En af HagiCodes kernestyrker er dens understøttelse af flere AI-programmerings-CLI-værktøjer gennem et samlet abstraktionslag. Fordelen ved dette design er faktisk ret simpel: nye værktøjer kan komme ind, gamle værktøjer kan blive, og kodebasen bliver ikke til kaos. For at være retfærdig er det sådan, alle gerne vil have livet til at fungere.

### AIProviderType enum

Platformen definerer understøttede CLI-udbydertyper gennem `AIProviderType` enum:

```csharp
public enum AIProviderType
{
    ClaudeCodeCli = 0,    // Claude Code CLI
    CodexCli = 1,          // GitHub Copilot Codex
    GitHubCopilot = 2,     // GitHub Copilot
    CodebuddyCli = 3,     // Codebuddy CLI
    OpenCodeCli = 4,      // OpenCode CLI
    IFlowCli = 5,         // IFlow CLI
    HermesCli = 6,        // Hermes CLI
    QoderCli = 7,         // Qoder CLI
    KiroCli = 8,          // Kiro CLI
    KimiCli = 9,          // Kimi CLI
    GeminiCli = 10,       // Gemini CLI (new)
}
```

Som du kan se, slutter Gemini CLI sig til denne familie som det tiende medlem. Hver CLI har sine egne særskilte karakteristika og brugsscenarier, så brugerne kan vælge fleksibelt baseret på deres behov. Mange veje fører jo til Rom; nogle er simpelthen nemmere end andre.

### Udbyder arkitektur

`HagiCode.Libs.Providers` giver en samlet udbydergrænseflade, der gør hver CLI-integration standardiseret og kortfattet. Tager Gemini CLI som et eksempel:

```csharp
public class GeminiProvider : ICliProvider<GeminiOptions>
{
    private static readonly string[] DefaultExecutableCandidates = ["gemini", "gemini-cli"];
    private const string ManagedBootstrapArgument = "--acp";

    public string Name => "gemini";
    public bool IsAvailable => _executableResolver.ResolveFirstAvailablePath(DefaultExecutableCandidates) is not null;
}
```

Fordelene ved dette design er:
- Integrering af en ny CLI kræver kun implementering af én udbyderklasse
- Ensartet livscyklusstyring og samling af sessioner
- Automatiseret alias-opløsning og eksekverbar opdagelse

Kort sagt gør dette design komplicerede ting til enklere ting og gør livet en smule lettere.

### Udbyderregister

Udbyderregistret håndterer automatisk alias-kortlægning og registrering:

```csharp
if (provider is GeminiProvider)
{
    registry.Register(provider.Name, provider, ["gemini-cli"]);
    continue;
}
```

Dette betyder, at brugere kan påberåbe sig Gemini CLI med enten `gemini` eller `gemini-cli`, og systemet genkender det automatisk. Det er som en ven med både et formelt navn og et kaldenavn - uanset hvad, så ved folk, hvem du mener.

## GLM-5.1 model support

GLM-5.1 er Zhipu AIs seneste flagskibsmodel, og HagiCode har fuldført fuld support til den.

### Sekundære erhverv Katalog

HagiCode administrerer alle understøttede modeller gennem Secondary Professions Catalogue. Her er konfigurationen for GLM-serien:

| Model ID | Navn | Understøtter billede | Kompatible CLI-familier |
|----------|------|----------------|-------------------------|
| `glm-4.7` | GLM 4.7 | - | claude, codebuddy, hermes, qoder, kiro |
| `glm-5` | GLM 5 | - | claude, codebuddy, hermes, qoder, kiro |
| `glm-5-turbo` | GLM 5 Turbo | - | claude, codebuddy, hermes, qoder, kiro |
| `glm-5.0` | GLM 5.0 (ældre) | - | claude, codebuddy, hermes, qoder, kiro |
| `glm-5.1` | GLM 5.1 | **sandt** | claude, codebuddy, hermes, qoder, kiro |

De vigtigste egenskaber ved GLM-5.1 kan opsummeres som følger:
- En selvstændig version-id uden ældre bagage
- Den første GLM-model, der understøtter billedinput
- Stærkere ræsonnement og kodeforståelse
- Bred multi-CLI-kompatibilitet

### GLM-5.1 vs GLM-5.0

På kodeniveau er den vigtigste forskel mellem GLM-5.1 og GLM-5.0 vist her:

```csharp
// GLM-5.0 (Legacy) - contains special retention logic
private const string Glm50CodebuddySecondaryProfessionId = "secondary-glm-5-codebuddy";
private const string Glm50CodebuddyModelValue = "glm-5.0";

// GLM-5.1 - standalone new model identifier
private const string Glm51SecondaryProfessionId = "secondary-glm-5-1";
private const string Glm51ModelValue = "glm-5.1";
```

GLM-5.0 bærer etiketten "Legacy", fordi det er en gammel versions-id, der er bevaret for bagudkompatibilitet. GLM-5.1 er derimod en helt ny selvstændig version uden historisk byrde. Nogle ting bliver i fortiden; andre rejser lettere og bevæger sig hurtigere.

### Konfigurer GLM-5.1

Her er et konfigurationseksempel til brug af GLM-5.1 i HagiCode:

```json
{
  "primaryProfessionId": "profession-claude-code",
  "secondaryProfessionId": "secondary-glm-5-1",
  "model": "glm-5.1",
  "reasoning": "high"
}
```

## Billed upload funktion

HagiCodes billedstøtte implementeres gennem `SupportsImage` ejendom på `SecondaryProfession`:

```csharp
public class HeroSecondaryProfessionSettingDto
{
    public bool SupportsImage { get; set; }
}
```

I Secondary Professions Catalogue ser GLM-5.1-konfigurationen sådan ud:

```json
{
  "id": "secondary-glm-5-1",
  "supportsImage": true
}
```

Dette betyder, at brugere kan uploade skærmbilleder direkte til AI-analyse, såsom:
- Skærmbilleder af fejlmeddelelser
- Problemer i en UI-skærm
- Datavisualiseringsdiagrammer
- Kodeeksekveringsresultater

Der er ikke længere behov for at beskrive alt manuelt - bare upload skærmbilledet. Bekvemmeligheden ved denne funktion er indlysende, når du har brugt den. Nogle gange siger et blik mere end en lang forklaring.

## Gemini CLI integration

Som den tiende Agent CLI er Gemini CLI integreret i HagiCode gennem standard Provider-arkitektur.

### Konfigurationsmuligheder

Gemini CLI understøtter et rigt sæt af konfigurationsmuligheder:

```csharp
public class GeminiOptions
{
    public string? ExecutablePath { get; set; }
    public string? WorkingDirectory { get; set; }
    public string? SessionId { get; set; }
    public string? Model { get; set; }
    public string? AuthenticationMethod { get; set; }
    public string? AuthenticationToken { get; set; }
    public Dictionary<string, string?> AuthenticationInfo { get; set; }
    public Dictionary<string, string?> EnvironmentVariables { get; set; }
    public string[] ExtraArguments { get; set; }
    public TimeSpan? StartupTimeout { get; set; }
    public CliPoolSettings? PoolSettings { get; set; }
}
```

Disse muligheder dækker alt fra grundlæggende opsætning til avancerede funktioner, hvilket giver brugerne fleksibiliteten til at konfigurere CLI'en efter deres egne behov. Alles arbejdsgang er forskellig, så lidt fleksibilitet er altid velkommen.

### ACP kommunikationsprotokol

Gemini CLI understøtter ACP (Agent Communication Protocol), som er HagiCodes unified CLI kommunikationsstandard. Gennem ACP kan forskellige CLI'er interagere med platformen på en ensartet måde, hvilket i høj grad forenkler integrationsarbejdet. Kort sagt standardiserer det de komplicerede dele, så alle kan arbejde lettere.

## HagiCodes egen byggepraksis

Når vi taler om praksis i den virkelige verden, er det bedste eksempel HagiCode-platformens egen bygge-workflow. HagiCodes udviklingsproces har allerede gjort fuld brug af AI-kapaciteter.

### Fungerer godt selv med GLM 4.7

HagiCodes platformdesign er godt optimeret, så det kan stadig give en god udviklingsoplevelse selv med GLM 4.7. Platformen har allerede hjulpet med at fuldføre flere vigtige byggeprojekter, herunder:
- Integration af flere CLI-udbydere
- Implementering af billedoverførselsfunktionen
- Dokumentationsgenerering og indholdsudgivelse

Det er faktisk en god ting. Ikke alle har brug for det nyeste hele tiden. Det, der passer dig bedst, er ofte det, der betyder mest.

### GLM-5.1 leverer mere med mindre indsats

Efter opgradering til GLM-5.1 bliver disse funktioner endnu stærkere:
- Stærkere kodeforståelse, hvilket reducerer frem og tilbage kommunikation
- Mere præcis afhængighedsanalyse, der peger i den rigtige retning med det samme
- Mere effektiv fejldiagnose, lokalisering af problemer hurtigere
- Understøttelse af billedinput, accelererende problembeskrivelser

Det er som at skifte fra en cykel til en bil. Du kan stadig nå den samme destination, men hastigheden og komforten er ikke den samme.

## Bedste praksis for multi-CLI-integration

`HagiCode.Libs.Providers` giver en samlet mekanisme til registrering og brug:

```csharp
services.AddHagiCodeLibs();

var gemini = serviceProvider.GetRequiredService<ICliProvider<GeminiOptions>>();
var codebuddy = serviceProvider.GetRequiredService<ICliProvider<CodebuddyOptions>>();
var hermes = serviceProvider.GetRequiredService<ICliProvider<HermesOptions>>();
```

Dette afhængighedsinjektionsdesign holder brugen på tværs af forskellige CLI'er meget kortfattet og gør også enhedstestning og hån mere bekvem. Ren kode er en måde at være ansvarlig over for dig selv.

### Noter

Der er et par ting at huske på ved faktisk brug:

1. **API-nøglekonfiguration**: Sørg for `ANTHROPIC_AUTH_TOKEN` er indstillet korrekt, eller modellen kan ikke kaldes
2. **Modeltilgængelighed**: GLM-5.1 skal aktiveres af den tilsvarende modeludbyder
3. **Billedfunktion**: Kun modeller med `supportsImage: true` kan bruge billedoverførsel
4. **CLI-installation**: Før du bruger Gemini CLI, skal du sørge for `gemini` eller `gemini-cli` er i systemet PATH

Det kan være små detaljer, men små detaljer, der håndteres dårligt, kan blive til store problemer, så de er værd at være opmærksomme på.

## Konklusion

Med fuld understøttelse af GLM-5.1 og den succesfulde integration af Gemini CLI, styrker HagiCode yderligere sine muligheder som en multi-model, multi-CLI AI programmeringsplatform. Disse opdateringer giver ikke kun brugerne flere valgmuligheder, men demonstrerer også HagiCodes fremadrettede arkitektur og skalerbarhed.

GLM-5.1's billedunderstøttelse, kombineret med HagiCodes skærmbillede upload-funktion, gør det muligt at lade AI'en "forstå fra billedet" og reducerer i høj grad omkostningerne ved at beskrive problemer. Og med understøttelse af ti CLI'er kan brugere fleksibelt vælge den AI-programmeringsassistent, der passer bedst til deres præferencer og scenarier. Flere valgmuligheder er næsten altid en god ting.

Vigtigst er det, at HagiCodes egen byggepraksis beviser, at platformen allerede kan køre godt og fuldføre komplekse opgaver selv med GLM 4.7, mens opgradering til GLM-5.1 kan forbedre udviklingseffektiviteten yderligere. Livet er ofte også sådan: man har ikke altid brug for det absolut bedste, kun det der passer en selv. Selvfølgelig, hvis det, der passer dig, kan blive endnu bedre, så er det så meget desto bedre.

Hvis du er interesseret i en multi-model multi-CLI AI programmeringsplatform, så prøv HagiCode - open source, gratis og stadig under udvikling. At prøve det koster ingenting, og det kan vise sig at være præcis, hvad du har brug for.
