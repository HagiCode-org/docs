## Tilføjelse af GitHub Pages Auto-Deployment Support til HagiCode

> Dette projekt blev oprindeligt kodenavnet PCode og er nu officielt omdøbt til HagiCode. Denne artikel dokumenterer, hvordan vi introducerede automatiseret statisk webstedsimplementering, så udgivelse af indhold bliver lige så let som at drikke vand.

## Baggrund / Introduktion

Under udviklingen af HagiCode løb vi ind i et meget praktisk problem: Da vores dokumentation og forslag blev ved med at vokse, blev det mere og mere presserende at administrere og præsentere indholdet effektivt. Vi besluttede at bruge GitHub Pages til at være vært for vores statiske websted, men det var bare for besværligt at bygge og implementere det manuelt. Hver ændring krævede en lokal opbygning, pakning og derefter et manuelt skub til `gh-pages` gren. Det var ikke kun ineffektivt, men det var også nemt at tage fejl.

For at løse dette problem, hovedsagelig fordi vi ville spare os selv for nogle kræfter, havde vi brug for en automatiseret implementeringsarbejdsgang. Denne artikel beskriver i detaljer, hvordan vi tilføjede GitHub Actions-baseret auto-deployment support til HagiCode-projektet, så vi kan forblive fokuseret på at skrive indhold og lade automatisering klare resten.

## Målanalyse

Inden vi går i gang med implementeringen, skal vi først afklare, hvad denne opgave faktisk skal udrette. Når alt kommer til alt, forsinker slibning af øksen ikke huggningen af ​​brænde.

### Kernekrav

1.  **Automatiseret build**: Udløser automatisk byggeworkflowet, når koden skubbes til `main` gren.
2.  **Automatisk implementering**: Efter en vellykket build skal du automatisk implementere de genererede statiske filer til GitHub-sider.
3.  **Miljøkonsistens**: Sørg for, at CI-miljøet matcher det lokale byggemiljø for at undgå den klassiske forlegenhed med "det fungerer lokalt, men fejler online."

### Teknisk valg

Da HagiCode er bygget på Docusaurus, en meget populær React statisk sitegenerator, kan vi bruge GitHub Actions til at nå dette mål.

## Konfiguration af GitHub Actions Workflow

GitHub Actions er CI/CD-tjenesten leveret af GitHub. Ved at definere workflow-filer i YAML-format inde i depotet, kan vi tilpasse alle slags automatiseringsopgaver.

### Oprettelse af Workflow-filen

Vi skal oprette en ny konfigurationsfil i `.github/workflows` bibliotek ved projektets rod, for eksempel `deploy.yml`. Hvis mappen ikke eksisterer endnu, skal du oprette den først.

Kernelogikken i denne konfigurationsfil er som følger:

1.  **Triggertilstand**: Lyt efter `push` begivenheder på `main` gren.
2.  **Runtime-miljø**: Den seneste Ubuntu-udgivelse.
3.  **Byggetrin**:
    -   Tjek koden
    -   Installer Node.js
    -   Installationsafhængigheder (`npm install`)
    -   Byg de statiske filer (`npm run build`)
4.  **Implementeringstrin**: Brug den officielle `action-gh-pages` handling for at skubbe byggeoutputtet til `gh-pages` gren.

### Nøglekonfigurationskode

Nedenfor er den konfigurationsskabelon, vi i sidste ende har vedtaget:

```yaml
name: Deploy to GitHub Pages

# Trigger condition: run when code is pushed to the main branch
on:
  push:
    branches:
      - main
    # Add path filters if needed, for example only build when docs change
    # paths:
    #   - 'docs/**'
    #   - 'package.json'

# Set permissions, which is important for GitHub Pages deployment
permissions:
  contents: read
  pages: write
  id-token: write

# Concurrency control: cancel older builds for the same branch
concurrency:
  group: "pages"
  cancel-in-progress: false

jobs:
  build:
    runs-on: ubuntu-latest
    steps:
      - name: Checkout
        uses: actions/checkout@v4
        # Note: fetch-depth: 0 is required, otherwise the build version may be inaccurate
        with:
          fetch-depth: 0

      - name: Setup Node
        uses: actions/setup-node@v4
        with:
          node-version: 20 # Keep this consistent with the local development environment
          cache: 'npm'     # Enabling cache can speed up the build process

      - name: Install dependencies
        run: npm ci
        # Use npm ci instead of npm install because it is faster and stricter, which makes it better for CI

      - name: Build website
        run: npm run build
        env:
          # Configure environment variables here if your site build requires them
          # NODE_ENV: production
          # PUBLIC_URL: /your-repo-name
          
      - name: Upload artifact
        uses: actions/upload-pages-artifact@v3
        with:
          path: ./build # Default Docusaurus output directory

  deploy:
    environment:
      name: github-pages
      url: ${{ steps.deployment.outputs.page_url }}
    runs-on: ubuntu-latest
    needs: build
    steps:
      - name: Deploy to GitHub Pages
        id: deployment
        uses: actions/deploy-pages@v4
```

## Almindelige faldgruber under implementering

I praksis løb vi ind i flere problemer. Jeg deler dem her i håbet om, at andre kan undgå dem, eller i det mindste forberede løsninger på forhånd.

### 1. Problemer med tilladelse til GitHub Token

Da vi først satte tingene op, blev implementeringen ved med at fejle med en 403 (forbudt). Efter en masse gravearbejde fandt vi ud af, at GitHub er standard `GITHUB_TOKEN` havde ikke tilladelse til at skrive til Pages.

**Løsning**: I depotets `Settings` -> `Actions` -> `General` -> `Workflow permissions`, skal du sørge for at vælge **"Læse- og skrivetilladelser"**.

### 2. Forkert Build Directory Path

Docusaurus placerer indbyggede statiske filer i `build` mappe som standard. Nogle projekter kan dog være konfigureret anderledes. For eksempel er Create React App standard til `build`, mens Vite som standard er til `dist`. Hvis Actions rapporterer, at den ikke kan finde outputfilerne, skal du tjekke outputstien ind `docusaurus.config.js`.

### 3. Substi-problemer

Hvis dit lager ikke er en brugerhjemmeside, betyder det, at det ikke er det `username.github.io`, men i stedet en projektside som f.eks `username.github.io/project-name`, så skal du konfigurere `baseUrl`.

I `docusaurus.config.js`:

```javascript
module.exports = {
  // ...
  url: 'https://hagicode.com', // Your Hagicode URL
  baseUrl: '/', // Deploy from the root path
  // ...
};
```
Denne detalje er let at overse. Hvis den er konfigureret forkert, kan siden indlæses som en tom skærm, fordi aktivstierne ikke kan løses.

## Bekræftelse af resultatet

Efter at have konfigureret alt og skubbet koden, kan vi gå til fanen **Handlinger** i GitHub-lageret og se showet.

Du vil se en gul cirkel, mens arbejdsgangen kører, og når den først bliver grøn, betyder det succes. Hvis det bliver rødt, skal du klikke ind og inspicere logfilerne. I de fleste tilfælde kan du spore problemet dernede, normalt en tastefejl eller en forkert stikonfiguration.

Når opbygningen lykkes, besøg `https://<your-username>.github.io/<repository-name>/` og du vil se det nyligt installerede websted.

## Resumé

Ved at introducere GitHub Actions automatiserede vi med succes implementeringen af HagiCode-dokumentationssiden. Dette sparer ikke kun den tid, der tidligere er brugt på manuelle operationer, men endnu vigtigere standardiserer udgivelsesarbejdsgangen. Nu, uanset hvilket teammedlem der opdaterer dokumentationen, når ændringen er flettet ind i `main`, vil det seneste indhold være live et par minutter senere.

**Kernefordele**:
- **Større effektivitet**: Vi gik fra "manuel emballering og manuel upload" til "kode er lig med frigivelse."
- **Færre fejl**: Vi eliminerede muligheden for menneskelige fejl under manuelle operationer.
- **Bedre udvikleroplevelse**: Udviklere kan fokusere mere på indholdskvalitet i stedet for at blive distraheret af kedelige implementeringstrin.

Selvom konfiguration af CI/CD er lidt besværlig i starten, især med alle tilladelser og stidetaljer, er det en engangsinvestering med store langsigtede afkast. Jeg anbefaler kraftigt, at hvert statisk webstedsprojekt anvender en lignende automatiseret arbejdsgang.
