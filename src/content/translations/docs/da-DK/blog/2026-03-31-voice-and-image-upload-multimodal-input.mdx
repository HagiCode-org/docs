## At skrive er langsommere end at tale, og at tale er langsommere end et skærmbillede - Multimodal input-praksis for AI-kodningsassistenter

> At skrive kode har en hastighedsgrænse, uanset hvor hurtigt du skriver. Nogle gange tager noget, man kan sige i én sætning, evigheder at skrive ud; nogle gange forklarer ét skærmbillede alt, men du skal stadig beskrive det med en bunke tekst. Denne artikel fortæller om, hvad vi løb ind i, mens vi byggede [HagiCode](https://hagicode.com), fra talegenkendelse til upload af billeder. Til sidst ville vi bare gøre en AI-kodningsassistent lidt nemmere at bruge.

## Baggrund

Mens vi byggede HagiCode, bemærkede vi et problem - eller rettere, et problem, der naturligt dukkede op, når folk begyndte at bruge det i høj grad: At stole på at skrive alene kan være trættende.

Tænk over det: interaktion mellem brugere og agenten er et kernescenarie. Men hvis hver udveksling kræver nonstop-tastning på tastaturet, er effektiviteten ikke stor:

1. **Indtastning er for langsom**: For komplicerede problemer, såsom fejlmeddelelser eller problemer med brugergrænsefladen, kan det tage et halvt minut at skrive alt, mens det kan tage ti sekunder at sige det højt. Den kløft er reel.

2. **Billeder er mere direkte**: Nogle gange giver brugergrænsefladen en fejl, nogle gange vil du sammenligne et designudkast, og nogle gange skal du vise en kodestruktur. "Et billede siger mere end tusind ord" er måske et gammelt ordsprog, men det holder stadig. At lade AI direkte "se" problemet er meget klarere end at beskrive det i evigheder.

3. **Interaktion skal føles naturligt**: Moderne AI-assistenter skal understøtte tekst, stemme og billeder. Brugere skal kunne vælge den inputmetode, der føles mest naturlig.

Så vi besluttede at tilføje understøttelse af talegenkendelse og billedoverførsel til HagiCode for at gøre agentinteraktioner mere bekvemme. Hvis brugerne kan skrive lidt mindre, er det allerede en gevinst.

## Om HagiCode

Løsningerne, der er delt i denne artikel, kommer fra vores praktiske arbejde i [HagiCode](https://hagicode.com) projekt - eller mere præcist, fra erfaringer, mens du snublede gennem en del faldgruber.

HagiCode er et open source AI-kodningsassistentprojekt med et simpelt mål: Brug AI til at forbedre udviklingseffektiviteten. Mens vi blev ved med at bygge, blev det klart, at brugerne stærkt ønskede multimodale input. Nogle gange er det hurtigere at sige en sætning end at skrive et langt afsnit, og nogle gange er et skærmbillede langt klarere end en lang forklaring.

Disse behov skubbede os fremad, og det var sådan funktioner som talegenkendelse og billedupload til sidst tog form. Brugere kan nu interagere med AI på den mest naturlige måde, som er tilgængelig for dem, og det føles godt.

## Analyse

### Tekniske udfordringer i talegenkendelse

Da vi byggede talegenkendelse, stødte vi på et vanskeligt problem: **browserens WebSocket API understøtter ikke tilpassede HTTP-headere**.

Den talegenkendelsestjeneste, vi valgte, var ByteDances Doubao Speech Recognition API. Desværre kræver denne API godkendelsesoplysninger som f.eks `accessToken` og `secretKey` skal sendes gennem HTTP-headere. Det skabte en øjeblikkelig teknisk konflikt:

```javascript
// The browser WebSocket API does not support this approach
const ws = new WebSocket('wss://api.com/ws', {
  headers: {
    'Authorization': 'Bearer token'  // Not supported
  }
});
```

Vi havde grundlæggende to muligheder:

1. **URL-forespørgselsparametertilgang**: Indsæt godkendelsesoplysningerne i URL'en
   - Fordelen er, at den er enkel at implementere
   - Ulempen er, at legitimationsoplysninger er udsat for frontend, hvilket er usikkert; nogle API'er kræver også header-baseret godkendelse

2. **Backend proxy tilgang**: implementer en WebSocket proxy på backend
   - Fordelen er, at legitimationsoplysninger forbliver sikkert gemt på backend, og løsningen er fuldt ud kompatibel med API-krav
   - Ulempen er, at implementeringen er lidt mere kompleks

Til sidst valgte vi **backend proxy-tilgangen**. Sikkerhed er ikke noget, man går på kompromis med.

### Funktionelle krav til billedupload

Vores krav til billedupload var faktisk ret ligetil:

1. **Flere uploadmetoder**: Klik for at vælge en fil, træk og slip og indsæt fra udklipsholderen
2. **Filvalidering**: Typebegrænsninger (PNG, JPG, WebP, GIF) og størrelsesbegrænsninger (5-10 MB) er grundlæggende krav
3. **Brugeroplevelse**: upload fremskridt, forhåndsvisninger og fejlmeddelelser, så brugerne altid ved, hvad der sker
4. **Sikkerhed**: Server-side validering og beskyttelse mod ondsindede filuploads er afgørende

## Løsning

### Talegenkendelse: WebSocket Proxy Architecture

Vi designede en tre-lags arkitektur til talegenkendelse og fandt en vej, der fungerede:

```
Browser WebSocket
       |
       | ws://backend/api/voice/ws
       | (binary audio)
       v
Backend Proxy
       |
       | wss://openspeech.bytedance.com/ (with auth header)
       v
Doubao API
```

**Kernekomponentimplementeringer**:

1. **Frontend AudioWorklet-processor**:

```javascript
class AudioProcessorWorklet extends AudioWorkletProcessor {
  process(inputs, outputs, parameters) {
    const input = inputs[0]?.[0];
    if (!input) return true;

    // Resample to 16 kHz (required by the Doubao API)
    const samples = this.resampleAudio(input, 48000, 16000);

    // Accumulate samples into 500 ms chunks
    this.accumulatedSamples.push(...samples);

    if (this.accumulatedSamples.length >= 8000) {
      // Convert to 16-bit PCM and send
      const pcm = this.floatToPcm16(this.accumulatedSamples);
      this.port.postMessage({ type: 'audioData', data: pcm.buffer }, [pcm.buffer]);
      this.accumulatedSamples = [];
    }
    return true;
  }
}
```

2. **Backend WebSocket-handler** (C#):

```csharp
[HttpGet("ws")]
public async Task GetWebSocket()
{
    if (HttpContext.WebSockets.IsWebSocketRequest)
    {
        await _webSocketHandler.HandleAsync(HttpContext);
    }
}
```

3. ** Frontend `VoiceTextArea` komponent**:

```tsx
export const VoiceTextArea = forwardRef<HTMLTextAreaElement, VoiceTextAreaProps>(
  ({ value, onChange, onTextRecognized, maxDuration }, ref) => {
    const { isRecording, interimText, volume, duration, startRecording, stopRecording } =
      useVoiceRecording({ onTextRecognized, maxDuration });

    return (
      <div className="flex gap-2">
        {/* Voice button */}

        </button>
        {/* Text input area */}
        <textarea value={displayValue} onChange={handleChange} />
      </div>
    );
  }
);
```

### Billeduploads: Multi-Method Upload Component

Vi byggede en komplet billedupload-komponent med understøttelse af alle tre uploadmetoder, der dækker de mest almindelige scenarier, som brugere løber ind i.

**Kernefunktioner**:

1. **Tre uploadmetoder**:

```tsx
// Click to upload
const handleClick = () => fileInputRef.current?.click();

// Drag-and-drop upload
const handleDrop = (e: React.DragEvent) => {
  const file = e.dataTransfer.files?.[0];
  if (file) uploadFile(file);
};

// Clipboard paste
const handlePaste = (e: ClipboardEvent) => {
  for (const item of Array.from(e.clipboardData?.items || [])) {
    if (item.type.startsWith('image/')) {
      const file = item.getAsFile();
      if (file) uploadFile(file);
    }
  }
};
```

2. **Frontend-validering**:

```tsx
const validateFile = (file: File): { valid: boolean; error?: string } => {
  if (!acceptedTypes.includes(file.type)) {
    return { valid: false, error: 'Only PNG, JPG, JPEG, WebP, and GIF images are allowed' };
  }
  if (file.size > maxSize) {
    return { valid: false, error: `Maximum file size is ${(maxSize / 1024 / 1024).toFixed(1)}MB` };
  }
  return { valid: true };
};
```

3. **Backend upload handler** (TypeScript):

```typescript
export const Route = createFileRoute('/api/upload')({
  server: {
    handlers: {
      POST: async ({ request }) => {
        const formData = await request.formData();
        const file = formData.get('file') as File;

        // Validation
        const validation = validateFile(file);
        if (!validation.isValid) {
          return Response.json({ error: validation.error }, { status: 400 });
        }

        // Save file
        const uuid = uuidv4();
        const filePath = join(uploadDir, `${uuid}${extension}`);
        await writeFile(filePath, buffer);

        return Response.json({ url: `/uploaded/${today}/${uuid}${extension}` });
      }
    }
  }
});
```

## Praktisk vejledning

### Sådan bruger du talegenkendelse

1. **Konfigurer talegenkendelsestjenesten**:
   - Åbn siden med indstillinger for talegenkendelse
   - Konfigurer Doubao-talen `AppId` og `AccessToken`
   - Konfigurer valgfrit hotwords for at forbedre genkendelsesnøjagtigheden for domænespecifikke termer

2. **Brug det i indtastningsfeltet**:
   - Klik på mikrofonikonet i venstre side af inputfeltet
   - Begynd at tale, efter at bølgeformsanimationen vises
   - Klik på ikonet igen for at stoppe optagelsen
   - Den genkendte tekst indsættes automatisk ved markørens position

3. **Eksempel på hotword-konfiguration**:

```text

useEffect
```

### Sådan bruger du billedoverførsel

1. **Uploadmetoder**:
   - Klik på upload-knappen for at vælge en fil
   - Træk et billede direkte ind i uploadområdet
   - Brug `Ctrl+V` for at indsætte et skærmbillede fra udklipsholderen

2. **Understøttede formater**: PNG, JPG, JPEG, WebP, GIF
3. **Størrelsesgrænse**: 5 MB som standard (konfigurerbar)

### Noter

1. **Talegenkendelse**:
   - Mikrofontilladelse er påkrævet
   - Brug i rolige omgivelser, når det er muligt
   - Den maksimale understøttede optagelsesvarighed er 300 sekunder som standard (kan konfigureres)

2. **Billeduploads**:
   - Kun almindelige billedformater understøttes
   - Vær opmærksom på filstørrelsesgrænser
   - Uploadede billeder modtager automatisk en forhåndsvisnings-URL

3. **Sikkerhedshensyn**:
   - Talegenkendelseslegitimationsoplysninger gemmes på backend
   - Billeduploads gennemgår streng validering på serversiden
   - HTTPS/WSS anbefales i produktionsmiljøer

## Konklusion

Efter tilføjelse af talegenkendelse og billedupload blev HagiCode-brugeroplevelsen mærkbart forbedret. Brugere kan nu interagere med AI på en mere naturlig måde - tale i stedet for at skrive og dele skærmbilleder i stedet for at beskrive alt manuelt. Det føles som endelig at finde en mere behagelig måde at kommunikere på.

Mens vi opbyggede denne funktion, stødte vi på det problem, at browser WebSocket API'er ikke understøtter tilpassede overskrifter. I sidste ende løste vi det med en backend proxy-tilgang. Denne løsning bevarede ikke kun sikkerheden, men lagde også grundlaget for senere integration af andre autentificerede WebSocket-tjenester.

Billedoverførselskomponenten drager også fordel af at understøtte flere uploadmetoder, så brugerne kan vælge, hvad der er mest praktisk i øjeblikket. Hvis du klikker, trækker eller indsætter alt arbejde, og hver sti får jobbet gjort hurtigt.

"At skrive er langsommere end at tale, og at tale er langsommere end et skærmbillede" passer ganske godt til temaet her. Hvis du bygger et lignende AI-assistentprodukt, håber jeg, at disse erfaringer hjælper, selvom det kun er lidt.
