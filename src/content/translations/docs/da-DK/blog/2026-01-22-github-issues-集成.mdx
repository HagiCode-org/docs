## Opbygning af en GitHub-problemintegration fra bunden: HagiCodes frontend-direkte forbindelsespraksis

> Denne artikel dokumenterer hele processen med at integrere GitHub-problemer i HagiCode-platformen. Vi vil undersøge, hvordan man bruger en "frontend-direkte forbindelse + minimal backend"-arkitektur for at opnå sikker OAuth-godkendelse og effektiv problemsynkronisering, mens backend'en holdes let.

## Baggrund: Hvorfor integrere GitHub?

Som en AI-assisteret udviklingsplatform ligger HagiCodes kerneværdi i at forbinde ideer med implementering. Men i faktisk brug fandt vi ud af, at efter at brugere har fuldført et forslag i HagiCode, er de ofte nødt til manuelt at kopiere indholdet til GitHub Issues til projektsporing.

Dette skaber flere tydelige smertepunkter:

1.  **Fragmenteret arbejdsgang**: Brugere skal skifte frem og tilbage mellem to systemer. Oplevelsen er ikke glat, og nøgleoplysninger kan let gå tabt under kopiering og indsæt.
2.  **Besværligt samarbejde**: Andre teammedlemmer er vant til at tjekke opgaver på GitHub og kan ikke direkte se forslags fremskridt inde i HagiCode.
3.  **Gentaget manuelt arbejde**: Hver gang et forslag opdateres, skal nogen manuelt opdatere det tilsvarende problem på GitHub, hvilket tilføjer unødvendige vedligeholdelsesomkostninger.

For at løse dette problem besluttede vi at introducere funktionen **GitHub Issues Integration**, der forbinder HagiCode-sessioner med GitHub-lagre for at aktivere "et-klik-synkronisering."

## Teknisk valg: Frontend Direct Connection vs Backend Proxy

Da vi designede integrationstilgangen, havde vi to muligheder foran os: den traditionelle "backend proxy model" og den mere aggressive "frontend direkte forbindelsesmodel."

### Sammenligning af løsninger

I den traditionelle **backend-proxymodel** skal hver anmodning fra frontend først gå gennem vores backend, som derefter kalder GitHub API. Dette centraliserer logikken, men det lægger også en betydelig byrde på backend:

1.  **Opsvulmet backend**: Vi bliver nødt til at skrive en dedikeret GitHub API-klientindpakning og også håndtere den komplekse OAuth-tilstandsmaskine.
2.  **Tokenrisiko**: Brugerens GitHub-token skal gemmes i backend-databasen. Selv med kryptering øger dette stadig sikkerhedsoverfladen.
3.  **Udviklingsomkostninger**: Vi har brug for databasemigreringer for at gemme tokens og en ekstra synkroniseringstjeneste for at vedligeholde.

**frontend-modellen med direkte forbindelse** er meget lettere. I denne tilgang bruger vi kun backend til det mest følsomme "hemmelige udveksling"-trin (OAuth-tilbagekaldet). Efter at have fået tokenet gemmer vi det direkte i browserens `localStorage`. Senere operationer såsom oprettelse af problemer og opdatering af kommentarer sendes direkte fra frontend til GitHub over HTTP.

| Sammenligningsdimension | Backend Proxy Model | Frontend Direct Connection Model |
| :--- | :--- | :--- |
| **Backend kompleksitet** | Kræver en fuld OAuth-tjeneste og GitHub API-klient | Behøver kun et OAuth-tilbagekaldsslutpunkt |
| **Tokenhåndtering** | Skal være krypteret og gemt i databasen, med risiko for lækage | Gemt i browseren og kun synlig for brugeren |
| **Implementeringsomkostninger** | Kræver databasemigrering og multi-service udvikling | Primært frontend arbejde |
| **Brugeroplevelse** | Centraliseret logik, men serverforsinkelse kan være lidt højere | Ekstremt hurtig respons med direkte GitHub-interaktion |

Fordi vi ønskede hurtig integration og minimale backend-ændringer, **valgte vi i sidste ende "frontend-direkte forbindelsesmodellen"**. Det er som at give browseren et "midlertidigt pass". Når først den får passet, kan browseren selv håndtere tingene på GitHub uden at bede backend-administratoren om godkendelse hver gang.

## Kernedesign: Dataflow og sikkerhed

Efter at have lagt os fast på arkitekturen, var vi nødt til at designe det specifikke dataflow. Kernen i synkroniseringsprocessen er, hvordan man opnår tokenet sikkert og bruger det effektivt.

### Overordnet arkitekturdiagram

Hele systemet kan abstraheres i tre roller: browseren (frontend), HagiCode-backend og GitHub.

```text
+--------------+        +--------------+        +--------------+
| Frontend     |        |   Backend    |        |   GitHub     |
| React        |        |   ASP.NET    |        |   REST API   |
|              |        |              |        |              |
|  +--------+  |        |              |        |              |
|  | OAuth  |--+--------> /callback    |        |              |
|  | Flow   |  |        |              |        |              |
|  +--------+  |        |              |        |              |
|              |        |              |        |              |
|  +--------+  |        |  +--------+  |        |  +--------+  |
|  | GitHub |  +------------> Session |  +----------> Issues |  |
|  | API    |  |        |  |Metadata|  |        |  |        |  |
|  | Direct |  |        |  +--------+  |        |  +--------+  |
|  +--------+  |        |              |        |              |
+--------------+        +--------------+        +--------------+
```

**Nøglepunktet er**: kun et lille trin i OAuth (udskiftning af koden til et token) skal gå gennem backend. Derefter håndteres de tunge løft (skabende problemer) direkte mellem frontend og GitHub.

### Synkroniseringsdataflow i detaljer

Når brugeren klikker på knappen "Synkroniser med GitHub" i HagiCode UI, finder en række komplekse handlinger sted:

```text
User clicks "Sync to GitHub"
         │
         ▼
1. Frontend checks localStorage for the GitHub token
         │
         ▼
2. Format issue content (convert the Proposal into Markdown)
         │
         ▼
3. Frontend directly calls the GitHub API to create/update the issue
         │
         ▼
4. Call the HagiCode backend API to update Session.metadata (store issue URL and other info)
         │
         ▼
5. Backend broadcasts the SessionUpdated event via SignalR
         │
         ▼
6. Frontend receives the event and updates the UI to show the "Synced" state
```

### Sikkerhedsdesign

Sikkerhed er altid topprioritet, når der integreres tredjepartstjenester. Vi gjorde os følgende overvejelser:

1.  **Forsvar mod CSRF-angreb**: Generer et tilfældigt `state` parameter under OAuth-omdirigeringen og gem den i `sessionStorage`. Validerer strengt tilstanden i tilbagekaldet for at forhindre forfalskede anmodninger.
2.  **Isoleret token-lagring**: Tokenet gemmes kun i browserens `localStorage`. Ved at bruge Same-Origin-politikken er det kun HagiCode-scripts, der kan læse den, hvilket undgår risikoen for, at en databaselæk på serversiden påvirker brugerne.
3.  **Fejlgrænser**: Vi har designet dedikeret håndtering af almindelige GitHub API-fejl (såsom 401 udløbet token, 422 valideringsfejl og 429 hastighedsbegrænsning), så brugerne modtager venlig feedback.

### 2. Frontend OAuth Flow

Dette er indgangspunktet for forbindelsen. Vi bruger standardautorisationskodeflowet.

```typescript
// src/HagiCode.Client/src/services/githubOAuth.ts

// Generate the authorization URL and redirect
export async function generateAuthUrl(): Promise<string> {
  const state = generateRandomString(); // Generate a random string for CSRF protection
  sessionStorage.setItem('hagicode_github_state', state);
  
  const params = new URLSearchParams({
    client_id: clientId,
    redirect_uri: window.location.origin + '/settings?tab=github&oauth=callback',
    scope: ['repo', 'public_repo'].join(' '),
    state: state,
  });
  
  return `https://github.com/login/oauth/authorize?${params.toString()}`;
}

// Handle the code-to-token exchange on the callback page
export async function exchangeCodeForToken(code: string, state: string): Promise<GitHubToken> {
  // 1. Validate state to prevent CSRF
  const savedState = sessionStorage.getItem('hagicode_github_state');
  if (state !== savedState) throw new Error('Invalid state parameter');

  // 2. Call the backend API to exchange the token
  // Note: this must go through the backend because ClientSecret cannot be exposed to the frontend
  const response = await fetch('/api/GitHubOAuth/callback', {
    method: 'POST',
    headers: { 'Content-Type': 'application/json' },
    body: JSON.stringify({ code, state, redirectUri: window.location.origin + '/settings?tab=github&oauth=callback' }),
  });

  if (!response.ok) throw new Error('Failed to exchange token');
  
  const token = await response.json();
  
  // 3. Save into LocalStorage
  saveToken(token);
  return token;
}
```

### 3. GitHub API Client Wrapper

Når vi har tokenet, har vi brug for et solidt værktøj til at kalde GitHub API.

```typescript
// src/HagiCode.Client/src/services/githubApiClient.ts

const GITHUB_API_BASE = 'https://api.github.com';

// Core request wrapper
async function githubApi<T>(endpoint: string, options: RequestInit = {}): Promise<T> {
  const token = localStorage.getItem('gh_token');
  if (!token) throw new Error('Not connected to GitHub');
  
  const response = await fetch(`${GITHUB_API_BASE}${endpoint}`, {
    ...options,
    headers: {
      ...options.headers,
      Authorization: `Bearer ${token}`,
      Accept: 'application/vnd.github.v3+json', // Specify the API version
    },
  });
  
  // Error handling logic
  if (!response.ok) {
    if (response.status === 401) throw new Error('GitHub token expired, please reconnect');
    if (response.status === 403) throw new Error('No permission to access this repository or rate limit exceeded');
    if (response.status === 422) throw new Error('Issue validation failed, the title may be duplicated');
    throw new Error(`GitHub API Error: ${response.statusText}`);
  }
  
  return response.json();
}

// Create issue
export async function createIssue(owner: string, repo: string, data: { title: string, body: string, labels: string[] }) {
  return githubApi(`/repos/${owner}/${repo}/issues`, {
    method: 'POST',
    body: JSON.stringify(data),
  });
}
```

### 4. Indholdsformatering og synkronisering

Det sidste trin er at konvertere HagiCode-sessionsdata til formatet af et GitHub-problem. Det er lidt ligesom oversættelsesarbejde.

```typescript
// Convert a Session object into a Markdown string
function formatIssueForSession(session: Session): string {
  let content = `# ${session.title}\n\n`;
  content += `**> HagiCode Session:** #${session.code}\n`;
  content += `**> Status:** ${session.status}\n\n`;
  content += `## Description\n\n${session.description || 'No description provided.'}\n\n`;
  
  // If this is a Proposal session, add extra fields
  if (session.type === 'proposal') {
    content += `## Chief Complaint\n\n${session.chiefComplaint || ''}\n\n`;
    // Add a deep link so users can jump back from GitHub to HagiCode
    content += `---\n\n**[View in HagiCode](hagicode://sessions/${session.id})**\n`;
  }
  
  return content;
}

// Main logic when clicking the sync button
const handleSync = async (session: Session) => {
  try {
    const repoInfo = parseRepositoryFromUrl(session.repoUrl); // Parse the repository URL
    if (!repoInfo) throw new Error('Invalid repository URL');

    toast.loading('Syncing to GitHub...');
    
    // 1. Format content
    const issueBody = formatIssueForSession(session);
    
    // 2. Call API
    const issue = await githubApiClient.createIssue(repoInfo.owner, repoInfo.repo, {
      title: `[HagiCode] ${session.title}`,
      body: issueBody,
      labels: ['hagicode', 'proposal', `status:${session.status}`],
    });
    
    // 3. Update Session Metadata (save the issue link)
    await SessionsService.patchApiSessionsSessionId(session.id, {
      metadata: {
        githubIssue: {
          owner: repoInfo.owner,
          repo: repoInfo.repo,
          issueNumber: issue.number,
          issueUrl: issue.html_url,
          syncedAt: new Date().toISOString(),
        }
      }
    });

    toast.success('Synced successfully!');
  } catch (err) {
    console.error(err);
    toast.error('Sync failed, please check your token or network');
  }
};
```

## Resumé og Outlook

Med denne "frontend direkte forbindelse" tilgang opnåede vi problemfri GitHub Issues integration med den mindst mulige backend kode.

### Vigtige gevinster
1.  **Høj udviklingseffektivitet**: Backend-ændringer er minimale, primært et ekstra databasefelt og et simpelt OAuth-tilbagekaldsslutpunkt. Det meste af logikken er gennemført på frontend.
2.  **Stærk sikkerhed**: Tokenet passerer ikke gennem serverdatabasen, hvilket reducerer risikoen for lækage.
3.  **Fantastisk brugeroplevelse**: Forespørgsler indledes direkte fra frontend, så responshastigheden er hurtig, og der er ikke behov for backend-videresendelse.

### Ting at passe på
Der er et par faldgruber at huske på under reel implementering:
- **OAuth App-indstillinger**: Husk at indtaste de rigtige `Authorization callback URL` i dine GitHub OAuth-appindstillinger (normalt `http://localhost:3000/settings?tab=github&oauth=callback`).
- **Satsgrænser**: GitHub API begrænser ikke-godkendte anmodninger ganske strengt, men med et token er kvoten normalt tilstrækkelig (5000 anmodninger/time).
- **URL-parsing**: Brugere indtaster alle slags depot-URL'er, så sørg for, at dit regex kan matche `.git` suffikser, SSH-formater og lignende tilfælde.

### Fremtidige forbedringer
Den nuværende funktion er stadig envejssynkronisering (HagiCode -> GitHub). I fremtiden planlægger vi at implementere to-vejs synkronisering gennem GitHub Webhooks. For eksempel, hvis et problem er lukket på GitHub, kan sessionstilstanden på HagiCode-siden også opdateres automatisk. Det vil kræve, at vi afslører et webhook-endepunkt på backend, hvilket vil være et interessant næste skridt.
