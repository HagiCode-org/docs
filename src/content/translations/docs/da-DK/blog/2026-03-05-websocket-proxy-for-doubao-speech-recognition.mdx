## Løsning af browser WebSocket-godkendelsesudfordringer: Et praktisk proxy-mønster til Doubao-talegenkendelse

> Browserens WebSocket API understøtter ikke tilpassede HTTP-headere, hvilket skaber en udfordring for talegenkendelsestjenester, der kræver godkendelsesdata i headere. Denne artikel fortæller, hvordan vi løste det problem i HagiCode-projektet med et backend-proxymønster, og hvordan tilgangen udviklede sig fra legepladseksperimenter til produktionsbrug.


## Baggrund

Da vi begyndte at bygge talegenkendelse til HagiCode-projektet, valgte vi trygt ByteDances Doubao talegenkendelsestjeneste. Det oprindelige design var ligetil: Lad frontenden oprette forbindelse direkte til Doubaos WebSocket-tjeneste. Hvor svært kan det være? Du skal bare åbne en forbindelse og sende nogle data, ikke?

Så kom overraskelsen: Doubaos API kræver, at godkendelsesoplysninger sendes gennem HTTP-headere, herunder ting som `accessToken` og `secretKey`. Det blev straks akavet, fordi browserens WebSocket API simpelthen ikke understøtter indstilling af brugerdefinerede overskrifter.

Så hvad gør du, når browseren ikke vil lade dig sende dem?

På det tidspunkt overvejede vi to muligheder:

1. Indsæt legitimationsoplysningerne i URL-forespørgselsparametre - enkelt og sløvt
2. Tilføj et proxy-lag på backend - mere arbejde ved første øjekast

Den første mulighed afslører legitimationsoplysninger direkte i frontend-kode og lokal lagring. Er det virkelig sikkert? Jeg var ikke tryg ved det. Oven i det kræver nogle API'er header-baseret verifikation, så denne tilgang er ikke engang levedygtig.

Til sidst valgte vi den anden mulighed: implementer en WebSocket-proxy på backend. Tilfældigvis blev dette mønster først valideret i vores legepladsmiljø, og først efter at vi havde bekræftet dets stabilitet, flyttede vi det i produktion. Der er jo ingen, der ønsker, at produktionen fordobles som et laboratorieeksperiment.

Nogle gange er disse tekniske vejspærringer frustrerende, men de tvinger dig også til at lære mønstre, som senere viser sig at være nyttige.

## Teknisk udfordringsanalyse

### Browser WebSocket-begrænsninger

Standard WebSocket API ser vidunderligt simpelt ud:

```javascript
const ws = new WebSocket('wss://example.com/ws');
```

Men den enkelhed er præcis, hvor problemet ligger - den sender kun parametre i URL'en, og den kan ikke indstille overskrifter, som en HTTP-anmodning kan:

```javascript
// This is not supported in the WebSocket API
const ws = new WebSocket('wss://example.com/ws', {
  headers: {
    'Authorization': 'Bearer token'
  }
});
```

Og det er kernespørgsmålet. For tjenester som Doubao-talegenkendelse, der afhænger af header-baseret godkendelse, er denne begrænsning en hård blokering.

Når du accepterer denne begrænsning, skal arkitekturen ændres.

### Beslutninger om arkitektonisk design

Ved udformningen af løsningen sammenlignede vi omhyggeligt afvejningerne.

**Beslutning 1: Valg af proxy-mønster**

Vi sammenlignede to tilgange:

| Mulighed | Fordele | Ulemper | Beslutning |
|------|------|------|------|
| Native WebSocket | Let, enkel, direkte videresendelse | Forbindelsesstyring skal håndteres manuelt | Udvalgt |
| SignalR | Automatisk genforbindelse, stærk indtastning | Alt for komplekse, ekstra afhængigheder | Afvist |

Vi valgte i sidste ende native WebSocket. For at være ærlig var det den letteste mulighed og en bedre egnethed til simpel bidirektionel binær stream-videresendelse. At trække i SignalR ville have føltes som overengineering for denne use case, og det kunne tilføje ekstra latency.

**Beslutning 2: Forbindelsesstyringsstrategi**

Vi har vedtaget en "én forbindelse, en session"-model - hver frontend WebSocket-forbindelse er knyttet til sin egen uafhængige Doubao-backend-forbindelse.

Fordelene er ligetil:
- Enkel at implementere og tilpasset det almindelige brugsmønster
- Lettere at fejlfinde og fejlfinde
- God ressourceisolering, der forhindrer interferens mellem sessioner

Enkelt sagt er den direkte løsning nogle gange den bedste. Kompleksitet gør ikke automatisk et design bedre.

**Beslutning 3: Lagring af godkendelsesdata**

Legitimationsoplysninger gemmes i backend-konfigurationsfiler (`appsettings.yml` eller miljøvariabler) og indlæst gennem afhængighedsinjektion:

- Enkel konfigurationsmodel, der matcher eksisterende backend-konventioner
- Følsomme data når aldrig frontend
- Understøtter multi-miljø opsætning til udvikling, test og produktion

Det niveau af adskillelse betyder noget. Ingen ønsker, at legitimationsoplysninger flyder rundt på steder, de ikke burde være.

### Dataflow design

Det overordnede dataflow ser således ud:

```
Frontend (browser)
  │
  │ ws://backend/api/voice/ws
  │ WebSocket (binary)
  ▼
Backend (proxy)
  │
  │ wss://openspeech.bytedance.com/
  │ (with authentication headers)
  ▼
Doubao API
```

Selve flowet er ikke kompliceret:
1. Frontenden opretter forbindelse til backend-proxyen via WebSocket
2. Backend-proxyen modtager lyddata og forbinder til Doubao API'et med autentificerede overskrifter
3. Doubao API returnerer genkendelsesresultater, og proxyen videresender dem til frontend
4. Hele processen forbliver fuldstændig asynkron med tovejs streaming

Når først ansvaret er opdelt klart, bliver designet helt naturligt.

## Implementering af kernekomponenter

### 1. WebSocket-endepunktskonfiguration

```csharp
app.Map("/ws", async context =>
{
    if (context.WebSockets.IsWebSocketRequest)
    {
        // Read configuration from query parameters
        var appId = context.Request.Query["appId"];
        var accessToken = context.Request.Query["accessToken"];

        // Validate required parameters
        if (string.IsNullOrEmpty(appId) || string.IsNullOrEmpty(accessToken))
        {
            context.Response.StatusCode = 400;
            return;
        }

        // Accept the WebSocket connection
        using var webSocket = await context.WebSockets.AcceptWebSocketAsync();

        // Message handling loop
        var buffer = new byte[4096];
        while (!webSocket.CloseStatus.HasValue)
        {
            var result = await webSocket.ReceiveAsync(buffer, CancellationToken.None);

            if (result.MessageType == WebSocketMessageType.Close)
            {
                await webSocket.CloseAsync(
                    result.CloseStatus.Value,
                    result.CloseStatusDescription,
                    CancellationToken.None);
                break;
            }

            // Process audio data
            await HandleAudioDataAsync(buffer, result.Count);
        }
    }
});
```

### 2. Sessionsstyring

```csharp
public class DoubaoSessionManager : IDoubaoSessionManager
{
    private readonly ConcurrentDictionary<string, DoubaoSession> _sessions = new();

    public DoubaoSession CreateSession(string connectionId)
    {
        var session = new DoubaoSession(connectionId);
        _sessions[connectionId] = session;
        return session;
    }

    public async Task SendAudioAsync(string connectionId, byte[] audioData)
    {
        if (_sessions.TryGetValue(connectionId, out var session))
        {
            await session.SendAudioAsync(audioData);
        }
    }

    public void RemoveSession(string connectionId)
    {
        if (_sessions.TryRemove(connectionId, out var session))
        {
            session.Dispose();
        }
    }
}
```

Bruger `ConcurrentDictionary` for sessionsstyring betyder, at trådsikkerhed i høj grad håndteres for os. Hver indgående forbindelse får sin egen session, og oprydning sker automatisk ved afbrydelse.

Frontend og backend bruger JSON-tekstbeskeder til kontrol, mens binære beskeder bærer lyddata.

Eksempel på kontrolmeddelelse:

## Frontend integration i praksis

### WebSocket forbindelse

```javascript
class DoubaoVoiceClient {
    constructor(config) {
        this.config = config;
        this.ws = null;
    }

    async connect() {
        const url = new URL(this.config.wsUrl);
        // Add query parameters
        Object.entries(this.config.params).forEach(([key, value]) => {
            url.searchParams.set(key, value);
        });

        this.ws = new WebSocket(url);

        return new Promise((resolve, reject) => {
            this.ws.onopen = () => {
                console.log('[DoubaoVoice] Connected');
                resolve();
            };

            this.ws.onmessage = (event) => {
                this._handleMessage(JSON.parse(event.data));
            };

            this.ws.onerror = reject;
        });
    }

    _handleMessage(message) {
        switch (message.type) {
            case 'status':
                this._handleStatus(message.payload);
                break;
            case 'result':
                this.onResult?.(message.payload);
                break;
            case 'error':
                console.error('[DoubaoVoice] Error:', message.payload);
                break;
        }
    }
}

// Usage example
const client = new DoubaoVoiceClient({
    wsUrl: 'ws://localhost:5000/ws',
    params: {
        appId: 'your-app-id',
        accessToken: 'your-access-token',
        sampleRate: 16000,
        bitsPerSample: 16,
        channels: 1
    }
});
```

### Lydoptagelse og transmission

Bruger `AudioWorkletNode` for lydbehandling giver bedre ydeevne:

```javascript
// audio-worklet.js
class AudioProcessorWorklet extends AudioWorkletProcessor {
    process(inputs, outputs, parameters) {
        const input = inputs[0]?.[0];
        if (!input) return true;

        // Convert to 16-bit PCM
        const pcm = new Int16Array(input.length);
        for (let i = 0; i < input.length; i++) {
            pcm[i] = Math.max(-32768, Math.min(32767, input[i] * 32767));
        }

        this.port.postMessage({
            type: 'audioData',
            data: pcm.buffer
        }, [pcm.buffer]);

        return true;
    }
}

registerProcessor('audio-processor', AudioProcessorWorklet);

// Main thread code
async function startAudioRecording() {
    const stream = await navigator.mediaDevices.getUserMedia({
        audio: {
            echoCancellation: true,
            noiseSuppression: true,
            autoGainControl: true,
            sampleRate: 48000
        }
    });

    const audioContext = new AudioContext();
    const audioSource = audioContext.createMediaStreamSource(stream);

    await audioContext.audioWorklet.addModule('/audio-worklet.js');
    const audioWorkletNode = new AudioWorkletNode(audioContext, 'audio-processor');

    audioWorkletNode.port.onmessage = (event) => {
        if (event.data.type === 'audioData' && ws?.readyState === WebSocket.OPEN) {
            ws.send(event.data.data); // Send binary data directly
        }
    };

    audioSource.connect(audioWorkletNode);
}
```

`AudioWorklet` præsterer langt bedre end `ScriptProcessorNode` og undgår lydstammeproblemer, som ældre behandlingsveje ofte introducerer.

## Noter og bedste praksis

### 1. Tilslutningsovervågning

- Log regelmæssigt sessionstilstand for at spore forbindelsens fulde livscyklus
- Overvåg antallet og varigheden af lydsegmenter for at opdage unormale forbindelser
- Registrer forbindelsesstatus og genforbindelsesadfærd for Doubao-tjenesten

Disse er grundlæggende operationelle praksisser, men de gør en reel forskel i produktionen.

### 2. Fejlhåndtering

- Indfang og log alle WebSocket-undtagelser
- Brug `IAsyncDisposable` for at sikre, at der bliver ryddet op i ressourcerne
- Implementer yndefuld forbindelseslukning og timeout-håndtering

Kort sagt, favoriser robusthed.

### 3. Krav til lydformat

- Sample rate: 16000 Hz (anbefalet) eller 8000 Hz
- Bitdybde: 16-bit
- Kanaler: mono
- Kodning: PCM (rå)

Hvis formatet er forkert, kan genkendelsen mislykkes, eller kvaliteten kan forringes betydeligt. Disse krav har betydning.

### 4. Sikkerhedshensyn

- Bevar kun følsomme legitimationsoplysninger i backend-konfiguration
- Håndhæv forbindelsesgrænser for at forhindre ressourceudtømning
- Brug HTTPS/WSS i produktionen

Sikkerhed er aldrig en mindre bekymring.

### 5. Performance optimering

- Brug asynkrone operationer for at undgå blokering
- Juster bufferstørrelser efter behov (standard: 4096 bytes)
- Overvej forbindelsespooling og genbrugsstrategier

Anvend disse optimeringer, hvor de giver mening for din arbejdsbyrde.

## Implementeringsanbefalinger

1. **Docker-implementering**: Pak proxytjenesten som en beholder for lettere skalering og administration
2. **Load balancing**: Brug Nginx eller Envoy som en omvendt proxy for WebSocket-trafik
3. **Sundhedstjek**: Implementer hjerteslagsbaserede kontroller for at overvåge servicetilgængelighed
4. **Logaggregation**: Send logfiler til et centraliseret logningssystem såsom ELK eller Loki

Implementeringen kan være enkel eller kompleks afhængigt af teamet og miljøet, så juster derefter.

## Resumé

WebSocket proxy-mønsteret løser det kerneproblem, at browserens WebSocket API ikke understøtter brugerdefinerede headere. I HagiCode-projektet viste dette mønster sig både gennemførligt og stabilt, da det gik fra legepladsvalidering til produktionsimplementering.

Nøgle takeaways:
- En backend-proxy kan videregive autentificeringsoplysninger sikkert
- Native WebSocket er let og effektiv til simple scenarier
- Modellen "én forbindelse, én session" forenkler både implementering og fejlfinding
- Frontend-backend-protokollen skal adskille kontrolsignaler fra lyddata

Hvis du bygger en funktion, der også afhænger af WebSocket-godkendelse, håber jeg, at dette mønster giver dig et nyttigt udgangspunkt.

Hvis du har spørgsmål, er du velkommen til at diskutere dem med os. Tekniske fremskridt sker hurtigere, når folk sammenligner noter.
