## Design.md: En løsning til konsistent AI-drevet frontend UI-design

> I en tid med AI-assisteret frontend-udvikling, hvordan kan du holde AI-genererede brugergrænseflader konsekvente? Denne artikel deler vores praktiske erfaring med at bygge et designgalleri-websted baseret på awesome-design-md, sammen med hvordan man opretter en struktureret `design.md` at guide AI mod standardiseret UI-design.

## Baggrund

Enhver, der har brugt AI til at skrive frontend-kode, har sandsynligvis haft en lignende oplevelse: Bed AI om at generere den samme side flere gange, og hvert resultat kommer ud i en anden stil. Nogle gange er hjørner afrundede, nogle gange er de skarpe. Nogle gange er mellemrummet 8px, og andre gange bliver det 16px. Selv den samme knap kan se anderledes ud på tværs af forskellige samtaler.

Dette er ikke et isoleret problem. Efterhånden som AI-assisteret udvikling bliver mere almindelig, er **manglen på konsistens i AI-genereret frontend UI** blevet et udbredt problem. Forskellige AI-assistenter, forskellige prompter og endda den samme assistent på tværs af forskellige samtaler kan producere dramatisk forskellige interfacedesigns. Det skaber en enorm vedligeholdelsesomkostning under produktiteration.

Grundårsagen er faktisk enkel: Der er ikke noget **autoritativt designreferencedokument**. Traditionelle CSS-stylesheets kan fortælle udviklere "hvordan man implementerer" noget, men de kan ikke fuldt ud kommunikere "hvorfor det er designet på denne måde" eller "hvilket designmønster der skal bruges i hvilket scenarie." For AI er en klar og struktureret beskrivelse endnu vigtigere for at forstå designkonventioner.

Samtidig tilbyder open source-fællesskabet allerede nogle fremragende ressourcer. Den [VoltAgent/awesome-design-md](https://github.com/VoltAgent/awesome-design-md) projekt indsamler designsystemdokumentation fra mange kendte virksomheder. Hver mappe indeholder en `README.md`, a `DESIGN.md`, og få vist HTML. Alt dette er dog spredt ud over opstrømsdepotet, hvilket gør det svært at gennemse og sammenligne hurtigt.

Så kan vi konsolidere disse ressourcer i et designgalleri, der er let at gennemse, samtidig med at vi destillerer et struktureret `design.md` til AI at bruge?

Svaret er ja. Lad mig derefter gennemgå vores tilgang.

## Analyse

Før du skriver kode, lad os nedbryde de tekniske udfordringer bag dette problem.

### Kildeindholdsstyring: Hvordan forener du spredte designressourcer?

Upstream awesome-design-md-lageret indeholder et stort antal designdokumenter, men vi havde brug for en måde at bringe dem ind i vores eget projekt.

**Løsning: brug git undermodul**

```
awesome-design-md-site
└── vendor/awesome-design-md          # Upstream resources (git submodule)
```

Dette giver os flere fordele:
- Versionskontrol: vi kan fastgøre en specifik upstream-version
- Offline builds: ingen grund til at anmode om eksterne API'er under build
- Indholdsgennemgang: specifikke ændringer er synlige i PR'er

### Statisk webstedsarkitektur: Hvordan giver du dynamisk søgning, mens du forbliver helt statisk?

Da dette er et designgalleri, er søgning et must-have. Men Astro er en statisk webstedsgenerator, så hvordan implementerer du søgning i realtid?

**Løsning: Synkronisering af React Island + URL-forespørgselsparameter**

```typescript
// src/components/gallery/SearchToolbar.tsx

export function SearchToolbar() {
  const [query, setQuery] = useState('');

  // Sync with the URL
```

Fordelen ved denne tilgang er, at den bevarer implementeringsmulighederne for et statisk websted intakt, hvilket betyder, at det kan implementeres til enhver statisk hostingtjeneste, mens den stadig leverer en øjeblikkelig filtreringsoplevelse.

### Designdokumentation: Hvordan hjælper du AI med at forstå og følge designstandarder?

Dette er kernen i hele løsningen. Vi skal skabe en struktureret `design.md` at AI kan forstå og anvende.

**Løsning: lån strukturen i ClickHouse `DESIGN.md`**

[ClickHouse's `DESIGN.md`](https://github.com/ClickHouse/ClickHouse/blob/master/DESIGN.md) er en fremragende reference. Det omfatter:
- Visuelt tema og atmosfære
- Farvepalet og roller
- Typografi regler
- Komponent Stylings
- Layoutprincipper
- Dybde & Højde
- Gør og lad være
- Responsiv adfærd
- Agentpromptvejledning

Vores tilgang er: **genbrug strukturen, omskriv indholdet**. Vi beholder sektionsstrukturen i ClickHouse `DESIGN.md`, men udskift indholdet med de faktiske designtokens og komponentkonventioner, der bruges i vores eget projekt.

## Løsning

Baseret på analysen ovenfor består vores løsning af fire kernemoduler.

### 1. Indtagelsesrørledning for indhold

Dette er grundlaget for hele systemet, ansvarligt for at udtrække og normalisere indhold fra upstream-ressourcer.

```typescript
// src/lib/content/awesomeDesignCatalog.ts

export async function scanSourceEntries(): Promise<DesignEntry[]> {
  const designDir = 'vendor/awesome-design-md/design-md';
  const entries: DesignEntry[] = [];

  for (const dir of await fs.readdir(designDir)) {
    const entryPath = path.join(designDir, dir);
    if (await isValidDesignEntry(entryPath)) {
      const entry = await normalizeDesignEntry(entryPath);
      entries.push(entry);
    }
  }

  return entries;
}

async function isValidDesignEntry(dir: string): Promise<boolean> {
  const requiredFiles = ['README.md', 'DESIGN.md'];
  for (const file of requiredFiles) {
    if (!(await fileExists(path.join(dir, file)))) {
      return false;
    }
  }
  return true;
}
```

### 3. Søgefunktion

Søgefunktionen er baseret på klientsidefiltrering, med tilstand bevaret gennem URL-forespørgselsparametre:

```typescript
// src/components/gallery/SearchToolbar.tsx

function SearchToolbar({ entries }: { entries: DesignEntry[] }) {
  const [query, setQuery] = useState('');
  const [results, setResults] = useState(entries);

  useEffect(() => {
    const params = new URLSearchParams(window.location.search);
    const initialQuery = params.get('q') || '';
    setQuery(initialQuery);
    filterEntries(initialQuery);
  }, []);

  const filterEntries = (searchQuery: string) => {
    const filtered = entries.filter(entry =>
      entry.searchText.toLowerCase().includes(searchQuery.toLowerCase())
    );
    setResults(filtered);
  };

  const handleChange = (e: React.ChangeEvent<HTMLInputElement>) => {
    const value = e.target.value;
    setQuery(value);
    filterEntries(value);

    // Update the URL without triggering a page refresh
    const newUrl = value
      ? `${window.location.pathname}?q=${encodeURIComponent(value)}`
      : window.location.pathname;
    window.history.replaceState({}, '', newUrl);
  };

  return (
    <div className="search-toolbar">
      <input
        type="text"
        value={query}
        onChange={handleChange}
        placeholder="Search design entries..."
      />
      <span className="result-count">{results.length} results</span>
    </div>
  );
}
```

## Øv

Now that we have covered the solution, how do you actually implement it?

### Implementeringstrin

**Trin 1: Initialiser undermodulet**

```bash
# Add the upstream repository as a submodule
git submodule add https://github.com/VoltAgent/awesome-design-md.git vendor/awesome-design-md

# Initialize and update the submodule
git submodule update --init --recursive
```

**Trin 2: Opret indholdspipeline**

Implementere `awesomeDesignCatalog.ts`, herunder:
- Filscanning og valideringslogik
- Markdown rendering using Astro's built-in renderer
- Udtræk af indtastningsdata

**Trin 3: Byg galleriets brugerflade**

Brug Astro + React Islands til at oprette:
- Hjemmesidegallerilayout (kortgitter)
- Designkortkomponenter
- Søg værktøjslinje
- Detalje sidelayout

**Trin 4: Skriv designdokumentet**

Baseret på strukturen af ClickHouse `DESIGN.md`, fill in the actual design tokens from your own project. Opdatering `README.md` og tilføje et link til `design.md`.

### Noter

**Sikkerhed**: Markdown-gengivelse kræver filtrering af usikker HTML. Astros indbyggede renderer-filtre `script` tags som standard, men du skal stadig holde øje med XSS-risici.

**Ydeevne**: Et stort antal iframe-forhåndsvisninger kan påvirke ydeevnen for første maling. Det anbefales at bruge `loading="lazy"` for at doven indlæse forhåndsvisningsindhold.

**Vedligeholdelse**: `design.md` skal forblive synkroniseret med kodeimplementeringen. Det anbefales at tilføje CI-tjek for at sikre, at CSS-variabler forbliver konsistente mellem dokumentation og kode.

**Tilgængelighed**: Sørg for, at farvekontrasten opfylder WCAG AA-standarden (mindst 4,5:1).

### AI-brugsvejledning

Efter oprettelse `design.md`, hvordan får du AI til rent faktisk at bruge det? Her er et par praktiske tips:

**Tip 1: Henvis det eksplicit i prompten**

```
Please refer to the design.md file in the project root and use the design conventions defined there to implement the following components:
- Buttons: use the primary color with an 8px border radius
- Cards: use the elevation-2 shadow level
```

**Tip 2: Kræv AI for at referere til specifikke CSS-variabler**

```
Implement a navigation bar with the following requirements:
- Use --color-bg-primary for the background color
- Use --color-border-subtle for borders
- Use --text-color-primary for text
```

**Tip 3: Inkluder `design.md` indhold i systemprompten**

Hvis dit AI-værktøj understøtter brugerdefinerede systemprompter, kan du tilføje kerneindholdet i `design.md` direkte til det.

### Teststrategi

**Indholdspipelinetest**:
- Manglende fil-scenarier (manglende `README.md` eller `DESIGN.md`)
- Formateringsfejlscenarier (Markdown-parsingsfejl)
- Scenarier for tomme biblioteker

**Søgefunktionstest**:

**UI-komponenttest**:

## Konklusion

Inkonsistensen i AI-genererede UI'er, som HagiCode stødte på under udviklingen, var i sin kerne forårsaget af manglen på et struktureret designreferencedokument. Ved at bygge et designgalleri site og skabe et standardiseret `design.md`, har vi løst dette problem.

Kerneværdien af denne løsning ligger i:
- **Samlede ressourcer**: konsolidering af spredt designsystemdokumentation
- **Strukturerede standarder**: udtrykker designkonventioner i et format, AI kan forstå
- **Kontinuerlig vedligeholdelse**: Hold indhold opdateret igennem `git submodule`

Hvis du også bruger AI til frontend-udvikling, er denne tilgang værd at prøve. At skabe en struktureret `design.md` forbedrer ikke kun konsistensen af AI-genereret kode, men hjælper også dit team med at opretholde ensartede designstandarder internt.
