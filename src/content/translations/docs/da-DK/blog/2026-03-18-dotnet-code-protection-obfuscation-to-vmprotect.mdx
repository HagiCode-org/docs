## .NET-kodebeskyttelse i praksis: Fra sløring til beskyttelse af virtuel maskine

> Denne artikel forklarer, hvordan man implementerer en flerlags kodebeskyttelsesstrategi i .NET-projekter, der dækker hele vejen fra grundlæggende sløring til professionel beskyttelse af virtuel maskine.

## Baggrund

I .NET-applikationsudvikling er beskyttelse af kernekode såsom licensvalidering, forretningslogik og følsom konfiguration fra dekompilering og reverse engineering et emne, du ikke kan undgå. Efterhånden som .NET-økosystemet er blevet modnet, har udviklere fået adgang til en række beskyttelsesmuligheder, fra indbyggede sløringsattributter til professionelle virtualiseringsbaserede beskyttelsesværktøjer.

Som et komplekst flersproget monorepo-projekt inkluderer HagiCode desktop-applikationer, byggesystemer og licensstyringsfunktioner. Koden indeholder uundgåeligt licensvalideringslogik, følsom konfiguration såsom API-nøgler og produkt-id'er og forretningskritisk logik. Disse dele har brug for seriøs beskyttelse, fordi ingen ønsker, at deres hårde arbejde skal afsløres så let.

Denne artikel deler den kodebeskyttelsestilgang, vi faktisk brugte i HagiCode-projektet, og opsummerer hele rejsen fra tidlige faldgruber til senere optimering. Forhåbentlig giver det dig nogle brugbare ideer.

## Om HagiCode

[HagiCode](https://hagicode.com) er et open source AI-kodningsassistent-projekt dedikeret til at give udviklere en intelligent programmeringsoplevelse. Projektet bruger en monorepo-arkitektur og vedligeholder samtidig en VSCode-udvidelse, backend AI-tjenester, en cross-platform desktop-klient og mere. Denne flersprogede, multi-platform kompleksitet gør kodebeskyttelse til en teknisk udfordring, vi skal stå over for.

Den tilgang, der er delt i denne artikel, er resultatet af ægte forsøg og fejl under udviklingen af HagiCode. Hvis du vil se, hvordan vi løste disse tekniske problemer, så fortsæt med at læse. Du kan finde et par uventede takeaways.

## Kerneindhold

### 1. Microsofts indbyggede obfuscation-attribut

.NET Framework har en indbygget `[ObfuscationAttribute]`, som er den mest grundlæggende og almindeligt anvendte kodeobfuskationsmarkør. Denne egenskab lever i `System.Reflection` navneområde og giver dig mulighed for at anvende baselinebeskyttelse til kode uden at introducere tredjepartsværktøjer.

**Kernefunktioner**:
- `Feature` egenskab: Specificerer sløringsfunktionen, som f.eks `"ultra"` (høj sløring) eller `"all"` (fuld uklarhed)
- `Exclude` ejendom: `true` betyder udelukke fra sløring, og `false` betyder anvende sløring
- Kan anvendes på klasser, metoder, egenskaber og andre typemedlemmer

I HagiCode-projektet kan du se det brugt sådan her:

```csharp
[Obfuscation(Feature = "ultra", Exclude = false)]
public async Task<LicenseValidationResult?> ValidateLicenseAsync(...)
```

**Fordelene** ved denne fremgangsmåde er ret indlysende:
- Ingen ekstra afhængigheder; det er indbygget i .NET Framework ud af æsken
- Kan genkendes og behandles af tredjeparts obfuskationsværktøjer
- Øger ikke størrelsen af den kompilerede samling væsentligt

Når det er sagt, har det også begrænsninger. Det er kun en markør, og det faktiske sløringsresultat afhænger af værktøjsimplementeringen. Det kan ikke give sikkerhed på beskyttelsesniveau for virtuel maskine.

### 2. VMP (Virtual Machine Protection)

VMP er et professionelt kodebeskyttelsesværktøj, der giver beskyttelse på højt niveau ved at kompilere kode i instruktioner til virtuelle maskiner. I modsætning til simpel navnsløring omdanner VMP faktisk kodelogik til en form, som konventionelle decompilere ikke kan rekonstruere.

**Klassificering af beskyttelsesniveau**:

| Niveau | Virtualisering | Mutation | Anti-debugging | Strengkryptering | Brug Cases |
|------|--------|------|--------|------------|----------|
| HØJ | fuld | høj | aktiveret | aktiveret | Licensvalidering, session samtidighed, følsomme konstanter |
| MIDDEL | delvis | medium | aktiveret | aktiveret | Forretningslogik, domænemodeller |
| LAV | ingen | lav | deaktiveret | deaktiveret | Hjælpeklasser, ikke-kritisk kode |

HagiCode-projektet definerer et deklarativt attributsystem til mærkning af kode, der kræver beskyttelse:

```csharp
// High-priority protection
[VmProtect(VmProtectionPriority.High, Reason = "Contains license verification logic")]
public class KeygenClient { ... }

// Exclude from protection
[VmExclude(Reason = "Public API that must remain unchanged")]
public class PublicApi { ... }

// Inherited protection
[VmProtect(Priority.High, ProtectDerived = true)]
public class BaseLicenseValidator { ... }
```

### 3. Byggetidsbeskyttelsesstrategi

VMP-beskyttelse har ikke kun betydning under kørsel. Det skal også automatiseres som en del af byggepipelinen, fordi det ville være alt for kedeligt at gøre det manuelt. HagiCodes byggesystem understøtter flere tilstande:

- **Native Windows-tilstand**: Kald VMProtect-værktøjet direkte
- **Linux Docker containertilstand**: Kør VMP inde i en container for at løse kompatibilitetsproblemer på tværs af platforme
- **Attributscanning**: Opdag automatisk beskyttelsesmarkører i kode
- **Valideringsmekanisme**: Bekræft, at beskyttelsen er blevet anvendt korrekt

Tilsammen gør disse muligheder processen meget nemmere at styre.

## Løsning

### 1. Brug af Microsofts indbyggede obfuscation-attribut

Ansøg `ObfuscationAttribute` direkte i kode:

```csharp
using System.Reflection;

[Obfuscation(Feature = "ultra", Exclude = false)]
public class LicenseService
{
    [Obfuscation(Feature = "ultra", Exclude = false)]
    public async Task<bool> ValidateLicenseAsync(string key)
    {
        // License validation logic
    }

    [Obfuscation(Feature = "flow", Exclude = false)]
    private string DecryptToken(string encrypted)
    {
        // Decryption logic
    }
}
```

Nogle gange er du nødt til at lade testsamlinger få adgang til interne medlemmer, mens du stadig holder produktionskoden sikker:

```csharp
// AssemblyInfo.cs
[assembly: InternalsVisibleTo("HagiCode.Application.Tests")]
[assembly: InternalsVisibleTo("DynamicProxyGenAssembly2")] // for Moq
```

Dette gør test meget mere bekvemt, fordi koden stadig skal testes ordentligt.

### 2. Brugerdefinerede attributdefinitioner for VMP-beskyttelse

Opret tilpassede beskyttelsesattributter for at kontrollere VMP-adfærd:

```csharp
using System;

[AttributeUsage(AttributeTargets.Class | AttributeTargets.Method | AttributeTargets.Property)]
public class VmProtectAttribute : Attribute
{
    public VmProtectionPriority Priority { get; set; }
    public string? Reason { get; set; }
    public bool ProtectDerived { get; set; }
}

[AttributeUsage(AttributeTargets.Class | AttributeTargets.Method | AttributeTargets.Property)]
public class VmExcludeAttribute : Attribute
{
    public string? Reason { get; set; }
}

public enum VmProtectionPriority
{
    None = 0,
    Low = 1,
    Medium = 2,
    High = 3
}
```

Tilpassede attributter er ofte nemmere at arbejde med, fordi de direkte afspejler dine egne beskyttelseskrav.

### 3. VMP-konfigurationsfil

```yaml
# vmp_config.yml
protection:

  default_level: "medium"

  high:
    virtualization: "full"

    anti_dump: true
    encrypt_strings: true
    encrypt_resources: true

  medium:
    virtualization: "partial"
    mutation: "medium"
    anti_debug: true
    encrypt_strings: true

  low:
    virtualization: "none"
    mutation: "low"
    anti_debug: false
```

En klarere konfiguration gør systemet meget nemmere at vedligeholde senere.

## Praktisk vejledning

### 1. Beskyttelsespraksis for kritiske komponenter

Ifølge HagiCodes kodebeskyttelsesspecifikation skal følgende komponenter bruge HØJ-prioritet beskyttelse:

```csharp
// Production constants - must be encrypted and protected by VMP
[VmProtect(VmProtectionPriority.High, Reason = "Production constants")]
public static class ProductionConstants
{
    // Encrypted string accessor, protected by VMP
    [VmProtect(VmProtectionPriority.High)]
    public static string GetLicenseServerUrl(IOptions<LicenseOptions> options) => ...;
}

// License validation logic
[VmProtect(VmProtectionPriority.High, Reason = "License verification logic")]
public class KeygenClient : IKeygenClient
{
    [Obfuscation(Feature = "ultra", Exclude = false)]
    public async Task<LicenseValidationResult?> ValidateLicenseAsync(...) { ... }
}

// Machine fingerprint service
[VmProtect(VmProtectionPriority.High)]
public class MachineFingerprintService : IMachineFingerprintService { ... }
```

Kritisk kode fortjener stærkere beskyttelse, fordi blotlæggelse af kernelogik ville forårsage reelle problemer.

### 2. String Encryption og Runtime Decryption

Krypter strenge på byggetidspunktet og dekrypter dem under kørsel:

```csharp
public static class StringDecryption
{
    [VmProtect(VmProtectionPriority.High, Reason = "CRITICAL SECURITY")]
    public static string DecryptString(byte[] encryptedData, byte[] key, byte[] iv)
    {
        using var aes = Aes.Create();
        aes.Key = key;
        aes.IV = iv;

        using var decryptor = aes.CreateDecryptor();
        using var ms = new MemoryStream(encryptedData);
        using var cs = new CryptoStream(ms, decryptor, CryptoStreamMode.Read);
        using var reader = new StreamReader(cs);

        return reader.ReadToEnd();
    }
}

// Production constant accessor (lazy loading + caching)
public static class ProductionConstants
{
    private static string? _cachedLicenseServerUrl;

    public static string GetLicenseServerUrl(IOptions<LicenseOptions> options)
    {
        if (_cachedLicenseServerUrl == null)
        {
            var encrypted = GetEncryptedLicenseServerUrl();
#if DEBUG
            _cachedLicenseServerUrl = options.Value.PrimaryServer.Url;
#else
            _cachedLicenseServerUrl = StringDecryption.DecryptString(
                encrypted,
                GetEncryptionKey(),
                GetEncryptionIV());
#endif
        }
        return _cachedLicenseServerUrl;
    }
}
```

Dette trin er vigtigt, fordi følsomme oplysninger aldrig bør efterlades i almindelig tekst.

### 3. Verifikation af VMP-beskyttelse

Efter opbygningen skal du kontrollere, om beskyttelsen blev anvendt med succes; ellers kan du ikke være sikker på, at det rent faktisk virker:

```csharp
// Example verification script
public bool VerifyProtection(string assemblyPath)
{
    // 1. Check the VMP signature
    var bytes = File.ReadAllBytes(assemblyPath);
    var vmpSignature = Encoding.ASCII.GetBytes("VMProtect");
    if (bytes.Any(b => vmpSignature.Contains(b)))
    {
        return true;
    }

    // 2. Check for file size changes (the protected file is usually larger)
    var originalInfo = new FileInfo(assemblyPath.Replace(".dll", ".bak"));
    if (originalInfo.Exists)
    {
        var sizeRatio = (double)new FileInfo(assemblyPath).Length / originalInfo.Length;
        return sizeRatio > 1.1;
    }

    return false;
}
```

Verifikation er altid værd at gøre, for ellers kan problemer glide ubemærket igennem.

### 4. Bemærkninger og forbehold

Der er flere faldgruber her, som fortjener særlig opmærksomhed:

1. **Undgå at sløre al kode**: Offentlige API'er, grænsefladedefinitioner og DTO-klasser behøver normalt ikke beskyttelse. Overdreven sløring kan skade ydeevne og fejlfinding. HagiCode-projektet lærte dette på den hårde måde.

2. **Beskyt nøgleaccessorer**: Metoder, der henter krypteringsnøgler, skal have samme eller et højere beskyttelsesniveau end selve de krypterede data; ellers mister hele opsætningen sin værdi.

3. **Balancetest og produktion**: DEBUG-builds bør springe kryptering over for at gøre udvikling og debugging nemmere, mens RELEASE-builds bør muliggøre fuld beskyttelse. Husk at adskille dem med betinget kompilering som f.eks `#if DEBUG`.

4. **Tænk på Docker-miljøet**: At køre VMP på Linux kræver en containeriseret tilgang for at sikre værktøjskompatibilitet. HagiCode bruger en Wine + VMP container løsning til at løse problemet på tværs af platforme.

5. **Bekræftelse er obligatorisk**: Når konstruktionen er færdig, skal du bekræfte, at beskyttelsen blev anvendt korrekt. Ellers kan følsom kode stadig blive afsløret, og den tidligere viste bekræftelseskode eksisterer netop til dette formål.

## Konklusion

Med denne flerlagsbeskyttelsesstrategi byggede HagiCode et omfattende kodesikkerhedssystem, der spænder fra basislinjeformidling til beskyttelse af virtuel maskine:

- **Lag 1**: Brug `ObfuscationAttribute` til basislinjemarkering og give tip til tredjepartsværktøjer
- **Layer 2**: Brug brugerdefineret `VmProtectAttribute` erklæringer til at udtrykke beskyttelseshensigt og -prioritet
- **Layer 3**: Brug VMP virtuel maskinbeskyttelse til at transformere kritisk kode til irreversible virtuelle maskininstruktioner
- **Layer 4**: Scan og anvend automatisk beskyttelse under opbygningen, og bekræft derefter resultatet

Denne tilgang kan modstå almindelige dekompileringsværktøjer og samtidig stå bedre op mod avancerede reverse engineering-angreb. Hvis du bygger en .NET-applikation, der har brug for kodebeskyttelse, håber jeg, at dette i det mindste giver dig et nyttigt referencepunkt.
