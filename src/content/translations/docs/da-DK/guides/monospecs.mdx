## Oversigt

Monospecs er HagiCodes multi-repository management tilgang. Det sporer og administrerer underlager igennem `monospecs.yaml`, og det fungerer for ethvert projekt, der har brug for ét sted til at koordinere flere uafhængige depoter.

### Kernekoncepter

**Konfigurationsfil**

`monospecs.yaml` er kernen i hele administrationsløsningen, der definerer:
- Liste og placering af underdepoter
- Git URL for hvert lager
- Vis navn og ikon
- Integrationsadfærd med OpenSpec

**Forskelle fra almindelige arkiver**

| Feature | Almindelig Depot | Monospecs Managed Project |
|----------|-------------------|---------------------------|
| **OpenSpec-placering** | Rodbiblioteker `openspec/` | Hovedlagerets rodmapper `openspec/` |
| **Skift omfang** | Begrænset til nuværende lager | Kan involvere flere underdepoter |
| **Underlagerets renhed** | Specifikationer blandet med kode | Specifikationer etableret i hovedlageret, underdepoter er renere |
| **Arkiv automatisk forpligtelse** | Har brug for manuelle spec commits | `commit_when_archive: true` forpligter automatisk specifikationer til hovedlageret |

### Hvorfor bruge monospecs?

- **Unified Management**: Administrer alle underlagre i én konfigurationsfil
- **Automationsunderstøttelse**: Giver lagermetadata til scripts og værktøjer
- **AI-venlighed**: Struktureret konfiguration hjælper AI Agent med at forstå repository-relationer
- **OpenSpec-integration**: Sømløs integration med workflowet til forandringsstyring
- **Renere underlagre**: Specifikationer opbevares i hovedrepo, underlagre indeholder kun faktisk kode

### Gældende scenarier

Brug Monospecs først i scenarier som disse:

**1. Multi-Repository Collaborative Development**
Når et projekt er opdelt i flere uafhængige arkiver:
- Frontend-applikation, backend-tjeneste og desktop-app udviklet separat
- Behov for at koordinere versioner og afhængigheder på tværs af repositories
- Dokumentation og hjælpeprojekter adskilt fra hovedkoden

**2. Hjælpeprojektledelse**
Brug Monospecs til at administrere:
- Dokumentationsprojekt
- Officiel hjemmeside
- Byg værktøjer og scripts
- Hvert lager bevarer sin egen versionskontrol

**3. Funktionsudvikling på tværs af lager**
En funktion, der kræver ændringer på tværs af flere lagre:
- Skal spore ændringer i hvilke lagre
- Bevis automatisk specifikationer til korrekt mållager
- Koordiner versionsopdateringer og afhængigheder

## Hurtig start

### Initialisering

Følg disse trin for at fuldføre monospecs initialisering:

#### 1. Opret et tomt git-lager

Opret en tom `git` mappe i dit hovedlager (bemærk: dette er `git`, ikke `repos/` - dette er monospecs-konventionen):

```bash
mkdir git
cd git
git init
cd ..
```

#### 2. Konfigurer monospecs-fil

Opret `monospecs.yaml` konfigurationsfil i dit hovedlagers rodmappe:

```yaml
version: "1.0"
commit_when_archive: true

repositories:
  - path: "repos/frontend"
    url: "https://github.com/your-org/frontend.git"
    displayName: "Frontend App"
    icon: "🌐"

  - path: "repos/backend"
    url: "https://github.com/your-org/backend.git"
    displayName: "Backend Service"
    icon: "⚙️"
```

**Konfigurationselementer**:
| Konfigurationselement | Type | Påkrævet | Beskrivelse |
|----------------|------|----------|-------------|
| `version` | streng | Ja | Konfigurationsfil version |
| `commit_when_archive` | boolesk | Nej | Om specifikationer skal overføres automatisk ved arkivering (ikke begå sub-repo-kode) |
| `repositories` | array | Ja | Repository konfigurationsliste |
| `repositories[].path` | streng | Ja | Depotsti (i forhold til projektroden) |
| `repositories[].url` | streng | Ja | Git repository URL |
| `repositories[].displayName` | streng | Nej | Visningsnavn på lageret |
| `repositories[].icon` | streng | Nej | Lagerikon (emoji) |

### Føj underlager til hovedlager

Tilføj alle underlagre til `repos/` bibliotek. Denne mappe skal **udelukkes fra hovedlagerets Git-versionskontrol** for at undgå at spore indhold fra underlager:

```plaintext
# Exclude all sub-repositories
repos/
```

#### 3. Tilføj Parent Repository til HagiCode

Hvis du bruger HagiCode desktop-applikationen, skal du åbne appen og tilføje dit hovedlager som et projekt. Klik på knappen "Tilføj projekt" eller "+" på startskærmen, vælg din hoveddepotmappe, og du vil se alle underlagre og deres statusoplysninger.

#### 4. Klon underdepoter

Brug `monospecs.yaml` konfiguration til at klone alle konfigurerede underlagre til `repos/` mappe:

Dette vil klone alle underlagre baseret på de URL'er, der er angivet i konfigurationsfilen.

## Konfigurationsfilstruktur

### Konfiguration af rodniveau

```yaml
version: "1.0"
commit_when_archive: true

repositories:
  # ... repository configurations
```

### Konfiguration af lagerniveau

Hvert underlager kan have:
- OpenSpec ændringsmappe (`openspec/`)
- Uafhængig `.git/` bibliotek
- Valgfrit `AGENTS.md` til AI Agent-konfiguration

## Drift af lagerstyring

### Tilføj nyt lager

Når du tilføjer et nyt underlager til `monospecs.yaml` `repositories` array:

```yaml
repositories:
  - path: "repos/new-service"
    url: "https://github.com/HagiCode-org/new-service.git"
    displayName: "New Service"
    icon: "🆕"
```

### Opdater lagerkonfiguration

Når lagerwebadresser eller metadata ændres:
1. Rediger `monospecs.yaml` for at opdatere tilsvarende poster
2. Kontroller, at YAML-syntaksen er korrekt
3. Hvis du synkroniserer ændringer, skal du manuelt opdatere konfigurationen af det lokale lager

## AI Agent Konfiguration Support

### AGENTER.md

Den `AGENTS.md` fil giver lagerspecifik vejledning til AI-agenten:

**Tekniske stakoplysninger**
- Rammer og byg værktøjer
- Kodekonventioner og navngivningspraksis
- Versionsstyring og implementeringskonfiguration

**Projektspecifik adfærd**
- Konfigurationsudvidelser og specielle krav
- OpenSpec integrationsmønstre
- Behov for koordinering på tværs af depoter

### AI fordele

Bruger `AGENTS.md` gør det muligt for AI-agenten at:
- Forstå lagerforhold og afhængigheder
- Generer kode og refactor på tværs af flere repositories
- Oprethold konsistens i versionskontrol og konfigurationer

## OpenSpec-integration

### Sådan fungerer Monospecs med OpenSpec

Uanset om du administrerer et monospecs-styret projekt eller et almindeligt lager, kan du bruge HagiCodes OpenSpec-arbejdsgang:

**Almindelig lager:**
- OpenSpec i root `openspec/` bibliotek
- Ændringer påvirker kun det nuværende lager

**Monospecs Managed Project:**
- OpenSpec læser `monospecs.yaml` konfiguration
- Ændringer spores af underlager
- Specifikationer forpligtes automatisk til korrekte mål

### Arkivadfærd

Hvornår `commit_when_archive: true`:
- Specifikationer er automatisk forpligtet til hovedlageret
- Underlagrenes kode er ikke forpligtet (forbliver adskilt)
- Forenkler specifikationsadministration - ingen manuelle spec-commits er nødvendige for sub-repositories

### Manuelle underlagerforpligtelser

Underlagre opretholder uafhængig versionskontrol:
- Send faktiske kodeændringer til deres egne lagre
- Specifikationer er forpligtet separat (fra hovedlageret ved arkivering)

## Bedste praksis

### Depotnavnekonventioner

- Brug kebab-bogstaver (små bogstaver og bindestreger) til depotstier
- Brug enkle, beskrivende visningsnavne på kinesisk
- Brug relevante emojis som ikoner

### Retningslinjer for vist navn

- Hold det kortfattet og beskrivende
- Brug ensartet navngivning på tværs af lignende lagre
- Vælg ikoner, der repræsenterer depotets formål

### Konfigurationsvedligeholdelse

**Synkroniser konfiguration med faktisk status**
- Kontroller regelmæssigt `monospecs.yaml` afspejler den faktiske depotstruktur
- Opdater eller fjern poster, når depoter tilføjes eller fjernes
- Test ændringer for at sikre konfigurationsvaliditet

**Versionskontrol til konfiguration**
- Spor ændringer i konfigurationsfilen med git
- Dokumenter årsager til større konfigurationsændringer
- Tag-konfigurationsfilversioner for nem rollback

## Fejlfinding

### Konfigurationsfil ikke fundet

Hvis `monospecs.yaml` filen findes ikke i projektets rod:
1. Tjek, om filen findes på den rigtige placering
2. Bekræft, at du er i den korrekte arbejdsmappe
3. Tjek for stavefejl i konfigurationsfilnavnet

### YAML-syntaksfejl

Almindelige YAML-syntaksproblemer:
- Forkert indrykning (brug mellemrum, ikke tabulatorer)
- Manglende eller ekstra anførselstegn omkring strenge
- Ugyldige datatyper (strenge kræver anførselstegn, tal ikke)
- Manglende obligatoriske felter (f.eks `path` eller `url`)

### Depot ikke fundet

Hvis et nyligt tilføjet lager ikke vises:
1. Tjek `path` er korrekt i `monospecs.yaml`
2. Bekræft, at lageret blev klonet
3. Tjek, om depotet findes på den forventede placering

### Kloningsfejl

Hvis lagerkloning mislykkes:
1. Tjek netværksforbindelsen
2. Bekræft, at URL-adressen til Git-lageret er korrekt
3. Tjek for godkendelsesproblemer
4. Gennemgå Docker-containerstatus
5. Tjek tilgængeligheden af diskplads
