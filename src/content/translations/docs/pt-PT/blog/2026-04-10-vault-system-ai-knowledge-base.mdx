## Construindo uma base de conhecimento entre projetos para a era da IA com o sistema Vault

> Aprender através do estudo e da reprodução de projetos reais está se tornando popular, mas materiais de aprendizagem dispersos e contextos fragmentados dificultam que os assistentes de IA forneçam todo o seu valor. Este artigo apresenta o design do sistema Vault no projeto HagiCode: por meio de uma camada de abstração de armazenamento unificada, os assistentes de IA podem compreender e acessar todos os recursos de aprendizagem, permitindo a verdadeira reutilização do conhecimento entre projetos.

## Plano de fundo

Na verdade, na era da IA, a forma como aprendemos novas tecnologias está a mudar silenciosamente. As abordagens tradicionais, como ler livros e assistir a vídeos, ainda são importantes, mas “estudar e reproduzir projetos” – pesquisar profundamente e aprender com o código, a arquitetura e os padrões de design de excelentes projetos de código aberto – está claramente se tornando mais eficiente. Executar e modificar diretamente projetos de código aberto de alta qualidade é uma das maneiras mais rápidas de compreender a prática de engenharia do mundo real.

Mas esta abordagem também traz novos desafios.

**Os materiais de aprendizagem estão muito dispersos.** As notas podem estar no Obsidian, os repositórios de código podem estar espalhados em pastas diferentes e o histórico de conversas de um assistente de IA se torna uma ilha de dados separada. Cada vez que você precisa da ajuda da IA ​​para analisar um projeto, é necessário copiar manualmente trechos de código e organizar o contexto, o que é bastante tedioso.

**O contexto continua se perdendo.** Os assistentes de IA não podem acessar diretamente os recursos de aprendizagem locais, portanto, cada conversa começa com a reexplicação de informações básicas. Os repositórios de código que você estuda são atualizados rapidamente e a sincronização manual é propensa a erros. Pior ainda, é difícil compartilhar conhecimento entre vários projetos de aprendizagem – os padrões de design aprendidos no projeto A são completamente desconhecidos para a IA quando ela trabalha no projeto B.

No fundo, estas questões são todas formas de “ilhas de dados”. Se houvesse uma camada de abstração de armazenamento unificada que permitisse aos assistentes de IA compreender e acessar todos os recursos de aprendizagem, o problema estaria resolvido.

Para resolver esses pontos problemáticos, tomamos uma decisão importante de design ao desenvolver o HagiCode: construir um sistema Vault como uma camada unificada de abstração de armazenamento de conhecimento. O impacto dessa decisão pode ser ainda maior do que você espera – falaremos mais sobre isso em breve.

## Sobre HagiCode

A abordagem compartilhada neste artigo vem da experiência prática no [HagiCode](https://hagicode.com) projeto. HagiCode é um assistente de codificação de IA baseado no fluxo de trabalho OpenSpec. Sua ideia central é que a IA não deve apenas ser capaz de “falar”, mas também de “fazer” – operar diretamente em repositórios de código, executar comandos e executar testes. Github: [github.com/HagiCode-org/site](https://github.com/HagiCode-org/site)

Durante o desenvolvimento, descobrimos que os assistentes de IA precisam de acesso frequente a vários tipos de recursos de aprendizagem do usuário: repositórios de código, notas, arquivos de configuração e muito mais. Se os usuários tivessem que fornecer tudo manualmente todas as vezes, a experiência seria terrível. Isso nos levou a projetar o sistema Vault.

## Projeto Central

### Suporte multitipo

O sistema Vault do HagiCode suporta quatro tipos, cada um correspondendo a diferentes cenários de uso:

| Tipo | Objetivo | Cenário Típico |
|------|------|----------|
| `folder` | Tipo de pasta de uso geral | Materiais de aprendizagem temporários, rascunhos |
| `coderef` | Projetado especificamente para estudar projetos de código | Aprendendo sistematicamente um projeto de código aberto |
| `obsidian` | Integra-se com o software de anotações Obsidian | Reutilizando uma biblioteca de notas existente |
| `system-managed` | Gerenciado automaticamente pelo sistema | Configuração do projeto, modelos de prompt e muito mais |

Entre eles, o `coderef` type é o mais comumente usado no HagiCode. Ele fornece uma estrutura de diretório padronizada e descrições de metadados legíveis por IA para projetos de estudo de código. Por que projetar esse tipo especificamente? Porque estudar um projeto de código aberto não é tão simples quanto “baixar código”. Você também precisa gerenciar o próprio código, notas de aprendizagem, arquivos de configuração e outros conteúdos ao mesmo tempo, e `coderef` padroniza tudo isso.

### Mecanismo de armazenamento persistente

O registro do Vault é persistido no sistema de arquivos como JSON:

```csharp
_registryFilePath = Path.Combine(absoluteDataDir, "personal-data", "vaults", "registry.json");
```

Este design pode parecer simples, mas foi cuidadosamente considerado:

**Simples e confiável.** JSON é legível por humanos, facilitando a depuração e modificação manual. Quando algo dá errado, você pode abrir o arquivo diretamente para inspecionar o estado ou até mesmo repará-lo manualmente – especialmente útil durante o desenvolvimento.

**Dependências reduzidas.** O armazenamento do sistema de arquivos evita a complexidade de um banco de dados. Não há necessidade de instalar e configurar um serviço de banco de dados extra, o que reduz a complexidade do sistema e o custo de manutenção.

**Simultaneidade segura.** `SemaphoreSlim` é usado para garantir a segurança do thread. Em um cenário de assistente de codificação de IA, várias operações podem acessar o registro do Vault ao mesmo tempo, portanto, o controle de simultaneidade é necessário.

### Integração de contexto de IA

A principal capacidade do sistema é poder injetar automaticamente informações do Vault no contexto de propostas de IA:

```typescript
export function buildTargetVaultsText(
  vaults: VaultForText[],
  template: VaultPromptTemplate = DEFAULT_VAULT_PROMPT_TEMPLATE,
): string {
  const readOnlyVaults = vaults.filter((vault) => vault.accessType === 'read');
  const editableVaults = vaults.filter((vault) => vault.accessType === 'write');

  const sections = [
    buildVaultSection(readOnlyVaults, template.reference),
    buildVaultSection(editableVaults, template.editable),
  ].filter(Boolean);

  return `\n\n### ${template.heading}\n\n${sections.join('\n')}`;
}
```

Isso permite que o assistente de IA entenda automaticamente quais recursos de aprendizagem estão disponíveis, sem exigir que o usuário forneça contexto manualmente todas as vezes. Isso faz com que a experiência do HagiCode pareça especialmente natural - diga à IA: "Ajude-me a analisar a renderização simultânea do React" e ela pode encontrar automaticamente o React Learning Vault registrado anteriormente, em vez de solicitar que você cole o código repetidamente.

### Mecanismo de controle de acesso

O sistema divide os Vaults em dois tipos de acesso:

- **referência (somente leitura)**: a IA só pode usar o conteúdo para análise e compreensão, sem modificá-lo
- **editável (modificável)**: a IA pode modificar o conteúdo conforme necessário para a tarefa

Esta distinção informa à IA qual conteúdo é “referência somente leitura” e qual conteúdo ela pode modificar, reduzindo o risco de alterações acidentais. Por exemplo, se você registrar o Vault de um projeto de código aberto como material de aprendizagem, você definitivamente não deseja que a IA edite casualmente o código dentro dele - então marque-o como `reference`. Mas se for o Vault do seu próprio projeto, você pode marcá-lo como `editable` e deixe a IA ajudar a modificar o código.

## Guia Prático

### Estrutura padronizada para um cofre CodeRef

Para `coderef` Vaults, o sistema fornece uma estrutura de diretório padronizada:

```
my-coderef-vault/
├── index.yaml          # vault metadata description
├── AGENTS.md           # operating guide for AI assistants
├── docs/               # stores learning notes and documentation
└── repos/              # manages referenced code repositories through Git submodules
```

Qual é a filosofia de design por trás dessa estrutura?

**docs/** armazena notas de aprendizagem, usando Markdown para registrar sua compreensão do código, análise de arquitetura e lições de depuração. Essas notas não são apenas para você - a IA também pode entendê-las e irá referenciá-las automaticamente ao lidar com tarefas relacionadas.

**repos/** gerencia os repositórios estudados por meio de submódulos Git em vez de copiar o código diretamente. Isso tem dois benefícios: primeiro, permanece sincronizado com o upstream e um único `git submodule update` busca o código mais recente; segundo, economiza espaço, porque vários Vaults podem fazer referência a diferentes versões do mesmo repositório.

**index.yaml** contém metadados do Vault para que o assistente de IA possa entender rapidamente sua finalidade e conteúdo. É essencialmente uma “autoapresentação” para o Vault, então a IA sabe o que é na primeira vez que o vê.

**AGENTS.md** é um guia escrito especificamente para assistentes de IA, que explica como lidar com o conteúdo dentro do Vault. Você pode dizer à IA coisas como: "Ao analisar este projeto, concentre-se no código relacionado à otimização de desempenho" ou "Não modifique os arquivos de teste".

### Criando e usando um cofre

Criar um CodeRef Vault é simples:

```typescript
const createCodeRefVault = async () => {
  const response = await VaultService.postApiVaults({
    requestBody: {
      name: "React Learning Vault",
      type: "coderef",
      physicalPath: "/Users/developer/vaults/react-learning",
      gitUrl: "https://github.com/facebook/react.git"
    }
  });

  // The system will automatically:
  // 1. Clone the React repository to vault/repos/react
  // 2. Create the docs/ directory for notes
  // 3. Generate index.yaml metadata
  // 4. Create the AGENTS.md guide file

  return response;
};
```

Em seguida, faça referência a este Vault em uma proposta de IA:

```typescript
const proposal = composeProposalChiefComplaint({
  chiefComplaint: "Help me analyze React's concurrent rendering mechanism",
  repositories: [
    { id: "react", gitUrl: "https://github.com/facebook/react.git" }
  ],
  vaults: [
    {
      id: "react-learning",
      name: "React Learning Vault",
      type: "coderef",
      physicalPath: "/vaults/react-learning",
      accessType: "read"  // AI can only read, not modify
    }
  ],
  quickRequestText: "Pay special attention to the Fiber architecture and scheduler implementation"
});
```

### Cenários típicos de uso

**Cenário 1: Estudar sistematicamente projetos de código aberto**

Crie um CodeRef Vault, gerencie o repositório de destino por meio de submódulos Git e registre notas de aprendizagem no `docs/` diretório. A IA pode acessar o código e as notas ao mesmo tempo, proporcionando uma análise mais precisa. As notas escritas durante o estudo de um módulo são referenciadas automaticamente pela IA quando ela analisa posteriormente o código relacionado - como ter um "assistente" que se lembra do seu pensamento anterior.

**Cenário 2: Reutilizando uma biblioteca de notas Obsidian**

Se você já usa o Obsidian para gerenciar notas, basta registrar seu Vault existente diretamente no HagiCode. A IA pode acessar sua base de conhecimento sem copiar e colar manualmente. Esse recurso é especialmente prático porque muitas pessoas têm anos de anotações acumuladas e, uma vez conectadas, a IA pode “ler” e compreender esse sistema de conhecimento.

**Cenário 3: Reutilização de conhecimento entre projetos**

Várias propostas de IA podem fazer referência ao mesmo Vault, permitindo a reutilização de conhecimento em projetos. Por exemplo, você pode criar um "Vault de aprendizagem de padrões de design" que contém notas e exemplos de código para muitos padrões de design. Não importa qual projeto a IA esteja analisando, ela pode se referir ao conteúdo daquele Vault – o conhecimento não precisa ser acumulado repetidamente.

### Mecanismo de segurança de caminho

O sistema valida estritamente os caminhos para evitar ataques de passagem de caminho:

```csharp
private static string ResolveFilePath(string vaultRoot, string relativePath)
{
    var rootPath = EnsureTrailingSeparator(Path.GetFullPath(vaultRoot));
    var combinedPath = Path.GetFullPath(Path.Combine(rootPath, relativePath));
    if (!combinedPath.StartsWith(rootPath, StringComparison.OrdinalIgnoreCase))
    {
        throw new BusinessException(VaultRelativePathTraversalCode,
            "Vault file paths must stay inside the registered vault root.");
    }
    return combinedPath;
}
```

Isso garante que todas as operações de arquivo permaneçam no diretório raiz do Vault e evita acesso a caminhos maliciosos. Segurança não é algo para se encarar levianamente. Se um assistente de IA for operar no sistema de arquivos, os limites deverão ser claramente definidos.

## Notas

Ao usar o sistema HagiCode Vault, há várias coisas às quais prestar atenção especial:

1. **Segurança de caminho**: certifique-se de que os caminhos personalizados permaneçam dentro do escopo permitido, caso contrário o sistema rejeitará a operação. Isso evita o uso indevido acidental e possíveis riscos de segurança.

2. **Gerenciamento de submódulos Git**: CodeRef Vaults são melhor gerenciados com submódulos Git em vez de copiar código diretamente. Os benefícios foram abordados anteriormente - manter a sincronia e economizar espaço. Dito isso, os submódulos têm seu próprio fluxo de trabalho, portanto, os usuários iniciantes podem precisar de um pouco de tempo para se familiarizarem com eles.

3. **Limites de visualização de arquivos**: o sistema limita o tamanho do arquivo (256 KB) e a quantidade (500 arquivos), portanto, arquivos grandes precisam ser manipulados em lotes. Este limite existe por motivos de desempenho. Se você encontrar arquivos muito grandes, poderá dividi-los manualmente ou processá-los de outra maneira.

4. **Informações de diagnóstico**: a criação de um Vault retorna informações de diagnóstico que podem ser usadas para depuração em caso de falha. Verifique primeiro o diagnóstico quando tiver problemas - na maioria dos casos, é aí que você encontrará a pista.

## Conclusão

O sistema HagiCode Vault está resolvendo fundamentalmente um problema simples, mas profundo: como permitir que os assistentes de IA entendam e usem os recursos de conhecimento local.

Por meio de uma camada de abstração de armazenamento unificada, uma estrutura de diretório padronizada e injeção automatizada de contexto, ele oferece um modelo de gerenciamento de conhecimento de “registre uma vez, reutilize em qualquer lugar”. Depois que um Vault é criado, a IA pode acessar e compreender automaticamente notas de aprendizagem, repositórios de código e recursos de documentação.

A melhoria da experiência com este design é óbvia. Não há mais necessidade de copiar manualmente trechos de código ou explicar repetidamente informações básicas – o assistente de IA se torna mais como um colega de equipe que realmente entende o projeto e pode fornecer ajuda mais valiosa com base no conhecimento existente.

O sistema Vault compartilhado neste artigo é uma solução moldada por tentativa e erro reais e otimização real durante o desenvolvimento do HagiCode. Se você acha que esse design é valioso, isso diz algo sobre a engenharia por trás dele - e vale a pena conferir o próprio HagiCode também.
