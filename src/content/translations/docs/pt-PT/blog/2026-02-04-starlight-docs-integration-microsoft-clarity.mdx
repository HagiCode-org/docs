## Seleção Técnica e Exploração

Inicialmente, discutimos diversas abordagens de integração durante a fase de Proposta. Como estávamos usando o Starlight, um framework de documentação construído no Astro, a ideia mais óbvia era usar o Astro Hooks.

Primeiro tentamos modificar `astro.config.mjs` e planejou injetar o script do Clarity durante a construção. Embora essa abordagem pudesse garantir cobertura global, faltava-lhe flexibilidade: não podíamos carregar ou descarregar dinamicamente o script com base na preferência do utilizador.

Levando em consideração a experiência do usuário e o controle de privacidade, escolhemos uma abordagem de **substituição de componentes**. Starlight permite que os desenvolvedores substituam seus componentes internos, o que significa que poderíamos assumir a lógica de renderização para `<footer>` ou `<head>` e controle com precisão quando o Clarity é carregado.

Houve também um pequeno desvio aqui: originalmente, queríamos criar um wrapper de layout chamado `StarlightWrapper.astro`. Mas durante a depuração real, descobrimos que o mecanismo de roteamento do Starlight não invoca automaticamente esse wrapper personalizado, fazendo com que o script falhe em algumas páginas. Foi uma armadilha clássica de "suposição óbvia" e nos lembrou que **devemos compreender profundamente o fluxo de renderização do framework em vez de aplicar cegamente padrões genéricos de framework**.

## Solução principal: substituindo o componente rodapé

Para garantir que o script do Clarity carregue em todas as páginas, incluindo documentos e postagens de blog, sem quebrar a estrutura original da página, optamos por substituir o script do Starlight `Footer` componente.

### Por que o rodapé?

1.  **Presença global**: o Rodapé aparece em quase todas as páginas padrão.
2.  **Não intrusivo**: colocar o script na área Rodapé, que na verdade é renderizado próximo à parte inferior do corpo, não bloqueia o caminho crítico de renderização (LCP) da página, portanto, o impacto no desempenho é mínimo.
3.  **Lógica centralizada**: a lógica de consentimento do cookie pode ser tratada em um local dentro do componente.

Crie um arquivo chamado `StarlightFooter.astro` sob `src/components/`. Starlight irá reconhecê-lo automaticamente e substituir o rodapé padrão.

A lógica do código principal é a seguinte:

**Pontos principais explicados**:
- `is:inline`: diz ao Astro para não processar o conteúdo desta tag de script e enviá-lo diretamente para o HTML. Isso é fundamental para scripts analíticos de terceiros; caso contrário, a otimização do pacote do Astro poderá fazer com que o script pare de funcionar.
- `define:vars`: um recurso Astro 3+ que permite que variáveis sejam injetadas com segurança dentro do escopo.
- `import.meta.env.PROD`: garante que nenhuma análise sem sentido seja gerada durante o desenvolvimento local, a menos que seja explicitamente necessário para depuração, mantendo seus dados limpos.

### Tópico Avançado: Conformidade com Privacidade e Controle de Cookies

## Lições aprendidas e armadilhas

À medida que implementamos esta solução no HagiCode, resumimos vários detalhes que são fáceis de ignorar:

1.  **`StarlightWrapper.astro` é uma armadilha **:
    Conforme mencionado anteriormente, não tente criar um Wrapper global para injetar o script. Essa abordagem não funciona no Starlight. A solução correta é substituir um componente específico, como `StarlightFooter.astro` ou `StarlightHead.astro`.

2.  **Considerações de desempenho para posicionamento de script**:
    Embora o Clarity recomende colocar o script em `<head>` para garantir a máxima precisão dos dados, para um site de documentação, a velocidade de carregamento da primeira tela (LCP) afeta diretamente o SEO e a retenção de usuários. Optamos por colocá-lo no Rodapé, próximo à parte inferior do corpo. Isso pode perder um pouco de dados do usuário de "rejeição em segundos", mas nos proporciona uma experiência de carregamento de página mais rápida, o que é uma compensação que vale a pena.

3.  **Interferência do ambiente de desenvolvimento**:
    Certifique-se de adicionar o `import.meta.env.PROD` verifique. No modo de desenvolvimento, você atualizará as páginas com frequência, o que, de outra forma, geraria uma grande quantidade de dados de teste sem sentido e poluiria seu painel do Clarity.

## Verificando o resultado

Após a implantação, você poderá visualizar dados em tempo real no console do Clarity. Normalmente, em alguns minutos, você começará a ver mapas de calor e gravações do usuário.

Para o HagiCode, esses dados nos ajudaram a descobrir que:
- muitos usuários revisitam repetidamente a seção "Início rápido", sugerindo que nossas orientações de instalação ainda podem não ser intuitivas o suficiente.
- a página "Referência da API" tem o tempo de permanência mais longo, confirmando as necessidades do nosso grupo principal de usuários.

## Conclusão

A integração do Microsoft Clarity não requer alterações complexas no servidor nem a instalação de um SDK pesado.

Aproveitando o mecanismo de substituição de componentes do Starlight, alcançamos análises em todo o site com nada mais do que um leve `StarlightFooter.astro` componente. Esse tipo de “microintegração” mantém a base de código limpa, ao mesmo tempo que nos dá a capacidade de entender o comportamento do usuário.

Se você também estiver operando um projeto técnico, especialmente um como o **HagiCode**, onde a documentação precisa continuar evoluindo, recomendo fortemente tentar o Clarity. Os dados dirão onde os usuários estão realmente enfrentando dificuldades.
