## Adicionando suporte de implantação automática de páginas do GitHub ao HagiCode

> Este projeto foi originalmente denominado PCode e agora foi oficialmente renomeado como HagiCode. Este artigo documenta como introduzimos a implantação automatizada de sites estáticos para que a publicação de conteúdo se torne tão fácil quanto beber água.

## Antecedentes/Introdução

Durante o desenvolvimento do HagiCode, nos deparamos com um problema muito prático: à medida que nossa documentação e propostas continuavam crescendo, o gerenciamento e a apresentação eficiente desse conteúdo tornavam-se cada vez mais urgentes. Decidimos usar GitHub Pages para hospedar nosso site estático, mas criá-lo e implantá-lo manualmente era muito complicado. Cada mudança exigia uma construção local, um empacotamento e, em seguida, um envio manual para o `gh-pages` filial. Isso não era apenas ineficiente, mas também fácil de errar.

Para resolver esse problema, principalmente porque queríamos economizar algum esforço, precisávamos de um fluxo de trabalho de implantação automatizado. Este artigo registra em detalhes como adicionamos suporte de implantação automática baseado em GitHub Actions ao projeto HagiCode, para que possamos manter o foco na escrita de conteúdo e deixar a automação cuidar do resto.

## Análise de Metas

Antes de avançarmos para a implementação, primeiro precisamos esclarecer o que esta tarefa realmente pretende realizar. Afinal, afiar o machado não atrasa o corte da lenha.

### Requisitos Básicos

1.  **Compilação automatizada**: acione automaticamente o fluxo de trabalho de compilação quando o código for enviado para o `main` filial.
2.  **Implantação automatizada**: após uma compilação bem-sucedida, implante automaticamente os arquivos estáticos gerados nas páginas do GitHub.
3.  **Consistência do ambiente**: certifique-se de que o ambiente de CI corresponda ao ambiente de compilação local para evitar o constrangimento clássico de "funciona localmente, mas falha online".

### Escolha Técnica

Como o HagiCode é construído no Docusaurus, um gerador de sites estáticos React muito popular, podemos usar o GitHub Actions para atingir esse objetivo.

## Configurando o fluxo de trabalho de ações do GitHub

GitHub Actions é o serviço CI/CD fornecido pelo GitHub. Ao definir arquivos de fluxo de trabalho no formato YAML dentro do repositório, podemos personalizar todos os tipos de tarefas de automação.

### Criando o arquivo de fluxo de trabalho

Precisamos criar um novo arquivo de configuração no `.github/workflows` diretório na raiz do projeto, por exemplo `deploy.yml`. Se o diretório ainda não existir, crie-o primeiro.

A lógica central deste arquivo de configuração é a seguinte:

1.  **Condição de gatilho**: Ouça `push` eventos no `main` filial.
2.  **Ambiente de execução**: a versão mais recente do Ubuntu.
3.  **Etapas de construção**:
    -   Confira o código
    -   Instale o Node.js.
    -   Instalar dependências (`npm install`)
    -   Construa os arquivos estáticos (`npm run build`)
4.  **Etapa de implantação**: use o oficial `action-gh-pages` ação para enviar a saída do build para o `gh-pages` filial.

### Código de configuração chave

Abaixo está o modelo de configuração que adotamos:

```yaml
name: Deploy to GitHub Pages

# Trigger condition: run when code is pushed to the main branch
on:
  push:
    branches:
      - main
    # Add path filters if needed, for example only build when docs change
    # paths:
    #   - 'docs/**'
    #   - 'package.json'

# Set permissions, which is important for GitHub Pages deployment
permissions:
  contents: read
  pages: write
  id-token: write

# Concurrency control: cancel older builds for the same branch
concurrency:
  group: "pages"
  cancel-in-progress: false

jobs:
  build:
    runs-on: ubuntu-latest
    steps:
      - name: Checkout
        uses: actions/checkout@v4
        # Note: fetch-depth: 0 is required, otherwise the build version may be inaccurate
        with:
          fetch-depth: 0

      - name: Setup Node
        uses: actions/setup-node@v4
        with:
          node-version: 20 # Keep this consistent with the local development environment
          cache: 'npm'     # Enabling cache can speed up the build process

      - name: Install dependencies
        run: npm ci
        # Use npm ci instead of npm install because it is faster and stricter, which makes it better for CI

      - name: Build website
        run: npm run build
        env:
          # Configure environment variables here if your site build requires them
          # NODE_ENV: production
          # PUBLIC_URL: /your-repo-name
          
      - name: Upload artifact
        uses: actions/upload-pages-artifact@v3
        with:
          path: ./build # Default Docusaurus output directory

  deploy:
    environment:
      name: github-pages
      url: ${{ steps.deployment.outputs.page_url }}
    runs-on: ubuntu-latest
    needs: build
    steps:
      - name: Deploy to GitHub Pages
        id: deployment
        uses: actions/deploy-pages@v4
```

## Armadilhas comuns durante a implementação

Na prática, nos deparamos com vários problemas. Estou compartilhando-os aqui na esperança de que outros possam evitá-los, ou pelo menos preparar soluções com antecedência.

### 1. Problemas de permissão de token GitHub

Quando configuramos as coisas pela primeira vez, a implantação continuou falhando com um 403 (Proibido). Depois de muita pesquisa, descobrimos que o padrão do GitHub `GITHUB_TOKEN` não tinha permissão para escrever no Pages.

**Solução**: No repositório `Settings` -> `Actions` -> `General` -> `Workflow permissions`, certifique-se de selecionar **"Permissões de leitura e gravação"**.

### 2. Caminho de diretório de construção errado

O Docusaurus coloca arquivos estáticos construídos no `build` diretório por padrão. No entanto, alguns projetos podem ser configurados de forma diferente. Por exemplo, o padrão Create React App é `build`, enquanto o padrão do Vite é `dist`. Se o Actions informar que não consegue encontrar os arquivos de saída, verifique o caminho de saída em `docusaurus.config.js`.

### 3. Problemas de subcaminho

Se o seu repositório não for uma página inicial do usuário, o que significa que não é `username.github.io`, mas em vez disso uma página do projeto como `username.github.io/project-name`, então você precisa configurar `baseUrl`.

Em `docusaurus.config.js`:

```javascript
module.exports = {
  // ...
  url: 'https://hagicode.com', // Your Hagicode URL
  baseUrl: '/', // Deploy from the root path
  // ...
};
```
Esse detalhe é fácil de ignorar. Se estiver configurado incorretamente, a página poderá carregar como uma tela em branco porque os caminhos dos ativos não podem ser resolvidos.

## Verificando o resultado

Depois de configurar tudo e enviar o código, podemos ir para a aba **Ações** do repositório GitHub e assistir ao show.

Você verá um círculo amarelo enquanto o fluxo de trabalho estiver em execução e, quando ficar verde, significa sucesso. Se ficar vermelho, clique e inspecione os registros. Na maioria dos casos, você pode rastrear o problema ali, geralmente um erro de digitação ou uma configuração de caminho incorreta.

Assim que a compilação for bem-sucedida, visite `https://<your-username>.github.io/<repository-name>/` e você verá o site recém-implantado.

## Resumo

Ao apresentar o GitHub Actions, automatizamos com sucesso a implantação do site de documentação do HagiCode. Isso não apenas economiza o tempo gasto anteriormente em operações manuais, mas, mais importante ainda, padroniza o fluxo de trabalho de liberação. Agora, não importa qual membro da equipe atualize a documentação, uma vez que a mudança seja incorporada `main`, o conteúdo mais recente estará disponível alguns minutos depois.

**Benefícios principais**:
- **Maior eficiência**: passamos de "empacotamento manual e upload manual" para "código igual a lançamento".
- **Menos erros**: eliminamos a possibilidade de erro humano durante operações manuais.
- **Melhor experiência do desenvolvedor**: os desenvolvedores podem se concentrar mais na qualidade do conteúdo em vez de se distrair com etapas tediosas de implantação.

Embora configurar o CI/CD seja um pouco problemático no início, especialmente com todas as permissões e detalhes de caminho, é um investimento único com grandes retornos a longo prazo. Eu recomendo fortemente que todo projeto de site estático adote um fluxo de trabalho automatizado semelhante.
