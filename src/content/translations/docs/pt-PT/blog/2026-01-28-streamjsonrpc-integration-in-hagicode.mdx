## Integração profunda e uso prático de StreamJsonRpc em HagiCode

> Este artigo explica em detalhes como o projeto HagiCode (anteriormente PCode) integrou com sucesso a biblioteca de comunicação StreamJsonRpc da Microsoft para substituir sua implementação JSON-RPC personalizada original, ao mesmo tempo que resolveu os problemas técnicos e desafios arquitetônicos encontrados durante o processo de integração.

## Plano de fundo

StreamJsonRpc é a biblioteca de comunicação JSON-RPC mantida oficialmente pela Microsoft para .NET e TypeScript, conhecida por sua forte segurança de tipo, geração automática de proxy e mecanismos maduros de tratamento de exceções. No projeto HagiCode, a equipe decidiu integrar StreamJsonRpc para se comunicar com ferramentas externas de IA, como iflow CLI e OpenCode CLI por meio de ACP (Agent Communication Protocol), ao mesmo tempo em que eliminava o custo de manutenção e possíveis bugs introduzidos pela implementação anterior de JSON-RPC personalizado. No entanto, o processo de integração enfrentou desafios específicos para streaming JSON-RPC, especialmente ao lidar com vinculação de destino de proxy e reconhecimento de parâmetros genéricos.

Para resolver esses problemas, tomamos uma decisão ousada: reconstruir todo o sistema de construção do zero. O impacto dessa decisão pode ser ainda maior do que você espera, e explicarei isso em detalhes em breve.

## Análise

O projeto atual está em um estágio crítico de integração do protocolo ACP e enfrenta os seguintes pontos técnicos e desafios arquitetônicos:

### 1. Limitações da Implementação Personalizada
A implementação JSON-RPC original está localizada em `src/HagiCode.ClaudeHelper/AcpImp/` e inclui componentes como `JsonRpcEndpoint` e `ClientSideConnection`. Manter essa base de código personalizada é caro e carece dos recursos avançados de uma biblioteca madura, como relatórios de progresso e suporte para cancelamento.

### 2. Obstáculos de integração StreamJsonRpc
Ao tentar migrar o existente `CallbackProxyTarget` padrão para StreamJsonRpc, descobrimos que o `_rpc.AddLocalRpcTarget(target)` O método não pôde reconhecer destinos criados por meio do padrão de proxy. Especificamente, StreamJsonRpc não pôde dividir automaticamente propriedades do tipo genérico `T` nos parâmetros do método RPC, fazendo com que o lado do servidor falhe ao processar chamadas de método iniciadas pelo cliente.

### 3. Camadas arquitetônicas confusas
O existente `ClientSideConnection` mistura a camada de transporte (WebSocket/Stdio), a camada de protocolo (JSON-RPC) e a camada de negócios (interface do agente ACP), levando a responsabilidades pouco claras. Ele também sofre com a falta de ligações de métodos em `AcpAgentCallbackRpcAdapter`.

### 4. Registros ausentes
A camada de transporte WebSocket não possui registro de conteúdo JSON bruto, tornando difícil determinar durante a depuração RPC se um problema se origina da serialização ou da rede.

## Solução

Para resolver os problemas acima, adotamos a seguinte solução sistemática, otimizando a partir de três dimensões: refatoração arquitetônica, integração de bibliotecas e depuração aprimorada.

### 1. Migrar totalmente para StreamJsonRpc

#### Remover código legado
Excluir `JsonRpcEndpoint.cs`, `AgentSideConnection.cs`e conversores de serialização personalizados relacionados, como `JsonRpcMessageJsonConverter`.

#### Integrar a Biblioteca Oficial
Apresente o `StreamJsonRpc` pacote NuGet e usar seu `JsonRpc` classe para lidar com a lógica de comunicação principal.

#### Abstrair a camada de transporte
Defina o `IAcpTransport` interface para lidar com ambos `WebSocket` e `Stdio` modos de transporte de forma unificada, garantindo que a camada de protocolo seja desacoplada da camada de transporte.

```csharp
// Definition of the IAcpTransport interface
public interface IAcpTransport
{
    Task SendAsync(string message, CancellationToken cancellationToken = default);
    Task<string> ReceiveAsync(CancellationToken cancellationToken = default);
    Task CloseAsync(CancellationToken cancellationToken = default);
}

// WebSocket transport implementation
public class WebSocketTransport : IAcpTransport
{
    private readonly WebSocket _webSocket;

    public WebSocketTransport(WebSocket webSocket)
    {
        _webSocket = webSocket;
    }

    // Implement send and receive methods
    // ...
}

// Stdio transport implementation
public class StdioTransport : IAcpTransport
{
    private readonly StreamReader _reader;
    private readonly StreamWriter _writer;

    public StdioTransport(StreamReader reader, StreamWriter writer)
    {
        _reader = reader;
        _writer = writer;
    }

    // Implement send and receive methods
    // ...
}
```

### 2. Corrigir problemas de reconhecimento de alvo de proxy

#### Analisar `CallbackProxyTarget`
Inspecione a lógica de geração de proxy dinâmico existente e identifique a causa raiz do motivo pelo qual StreamJsonRpc não consegue reconhecê-la. Na maioria dos casos, isso acontece porque o objeto proxy não expõe publicamente as assinaturas reais do método ou usa tipos de parâmetros não suportados pelo StreamJsonRpc.

#### Refatorar passagem de parâmetro
Divida as propriedades genéricas em parâmetros de método RPC explícitos. Em vez de confiar em propriedades dinâmicas, defina DTOs de solicitação/resposta (objetos de transferência de dados) concretos para que StreamJsonRpc possa reconhecer corretamente assinaturas de método por meio de reflexão.

```csharp
// Original generic property approach
public class CallbackProxyTarget<T>
{
    public Func<T, Task> Callback { get; set; }
}

// Refactored concrete method approach
public class ReadTextFileRequest
{
    public string FilePath { get; set; }
}

public class ReadTextFileResponse
{
    public string Content { get; set; }
}

public interface IAcpAgentCallback
{
    Task<ReadTextFileResponse> ReadTextFileAsync(ReadTextFileRequest request);
    // Other methods...
}
```

#### Usar `Attach` Em vez de `AddLocalRpcTarget`
Em alguns cenários complexos, o proxy manual do `JsonRpc` objeto e manipulação `RpcConnection` pode ser mais flexível do que adicionar diretamente um alvo local.

### 3. Implementar vinculação de método e registro mais forte

#### Implementar `AcpAgentCallbackRpcAdapter`
Certifique-se de que este componente implemente explicitamente a interface de proxy StreamJsonRpc e mapeie métodos definidos pelo protocolo ACP, como `ReadTextFileAsync`, para manipuladores de retorno de chamada StreamJsonRpc.

#### Integrar registro
Intercepte e registre o texto bruto de solicitações e respostas JSON-RPC no pipeline de processamento de mensagens WebSocket ou Stdio. Usar `ILogger` para gerar a carga bruta antes da análise e após a serialização, para que problemas de formatação possam ser diagnosticados com mais facilidade.

```csharp
// Transport wrapper with enhanced logging
public class LoggingAcpTransport : IAcpTransport
{
    private readonly IAcpTransport _innerTransport;
    private readonly ILogger<LoggingAcpTransport> _logger;

    public LoggingAcpTransport(IAcpTransport innerTransport, ILogger<LoggingAcpTransport> logger)
    {
        _innerTransport = innerTransport;
        _logger = logger;
    }

    public async Task SendAsync(string message, CancellationToken cancellationToken = default)
    {
        _logger.LogTrace("Sending message: {Message}", message);
        await _innerTransport.SendAsync(message, cancellationToken);
    }

    public async Task<string> ReceiveAsync(CancellationToken cancellationToken = default)
    {
        var message = await _innerTransport.ReceiveAsync(cancellationToken);
        _logger.LogTrace("Received message: {Message}", message);
        return message;
    }

    public async Task CloseAsync(CancellationToken cancellationToken = default)
    {
        _logger.LogDebug("Closing connection");
        await _innerTransport.CloseAsync(cancellationToken);
    }
}
```

### 4. Refatorar camadas arquitetônicas

#### Camada de Transporte (`AcpRpcClient`)
Envolva a conexão StreamJsonRpc e torne-a responsável por `InvokeAsync` chamadas e gerenciamento do ciclo de vida da conexão.

```csharp
public class AcpRpcClient : IDisposable
{
    private readonly JsonRpc _rpc;
    private readonly IAcpTransport _transport;

    public AcpRpcClient(IAcpTransport transport)
    {
        _transport = transport;
        _rpc = new JsonRpc(new StreamRpcTransport(transport));
        _rpc.StartListening();
    }

    public async Task<TResponse> InvokeAsync<TResponse>(string methodName, object parameters)
    {
        return await _rpc.InvokeAsync<TResponse>(methodName, parameters);
    }

    public void Dispose()
    {
        _rpc.Dispose();
        _transport.Dispose();
    }

    // StreamRpcTransport is the StreamJsonRpc adapter for IAcpTransport
    private class StreamRpcTransport : IDuplexPipe
    {
        // Implement the IDuplexPipe interface
        // ...
    }
}
```

#### Camada de Protocolo (`IAcpAgentClient` / `IAcpAgentCallback`)
Defina interfaces claras de cliente para agente e de agente para cliente. Remova o padrão cíclico de fábrica `Func<IAcpAgent, IAcpClient>` e substitua-o por injeção de dependência ou registro de retorno de chamada direto.

## Pratique

Com base nas melhores práticas do StreamJsonRpc e na experiência do projeto, a seguir estão as principais recomendações para implementação:

### 1. DTOs fortemente tipados são melhores que objetos dinâmicos
A principal vantagem do StreamJsonRpc reside na digitação forte. Não use `dynamic` ou `JObject` para passar parâmetros. Em vez disso, defina classes C# POCO explícitas como parâmetros para cada método RPC. Isso não apenas resolve o problema de reconhecimento do destino do proxy, mas também detecta erros de tipo em tempo de compilação.

Exemplo: substitua as propriedades genéricas em `CallbackProxyTarget` com classes concretas como `ReadTextFileRequest` e `WriteTextFileRequest`.

### 2. Declarar explicitamente nomes de métodos
Use o `[JsonRpcMethod]` atributo para especificar explicitamente nomes de métodos RPC em vez de depender do mapeamento de nome de método padrão. Isso evita falhas de invocação causadas por diferenças de estilo de nomenclatura, como PascalCase versus camelCase.

```csharp
public interface IAcpAgentCallback
{
    [JsonRpcMethod("readTextFile")]
    Task<ReadTextFileResponse> ReadTextFileAsync(ReadTextFileRequest request);
    
    [JsonRpcMethod("writeTextFile")]
    Task WriteTextFileAsync(WriteTextFileRequest request);
}
```

### 3. Aproveite as vantagens dos retornos de chamada do estado da conexão
StreamJsonRpc fornece o `JsonRpc.ConnectionLost` evento. Você deve absolutamente ouvir esse evento para lidar com saídas inesperadas de processos ou desconexões de rede, o que é mais oportuno do que depender apenas da detecção de falhas do Orleans Grain.

```csharp
_rpc.ConnectionLost += (sender, e) =>
{
    _logger.LogError("RPC connection lost: {Reason}", e.ToString());
    // Handle reconnection logic or notify the user
};
```

### 4. Registrar logs por camada
- **Nível de rastreamento**: registre a carga completa de solicitação/resposta JSON bruta.
- **Nível de depuração**: registra pilhas de chamadas de método e resumos de parâmetros.
- **Observação**: certifique-se de que os logs não incluam tokens de autorização confidenciais ou conteúdo de arquivo grande codificado em Base64.

### 5. Lide com a natureza especial do transporte de streaming
StreamJsonRpc suporta nativamente `IAsyncEnumerable`. Ao implementar respostas de prompt de streaming para ACP, use `IAsyncEnumerable` diretamente em vez da lógica de paginação personalizada. Isso pode simplificar bastante a quantidade de código necessária para o processamento de streaming.

```csharp
public interface IAcpAgentCallback
{
    [JsonRpcMethod("streamText")]
    IAsyncEnumerable<string> StreamTextAsync(StreamTextRequest request);
}
```

### 6. Padrão do adaptador
Manter `ACPSession` e `ClientSideConnection` separado. `ACPSession` deve se concentrar no gerenciamento do estado de Orleans e na lógica de negócios, como enfileiramento de mensagens, e deve usar o objeto de conexão StreamJsonRpc por meio de composição em vez de herança.

## Conclusão

Ao integrar de forma abrangente o StreamJsonRpc, o projeto HagiCode abordou com sucesso o alto custo de manutenção, as limitações funcionais e a confusão de camadas arquitetônicas da implementação personalizada original. As principais melhorias incluem:

1. Substituindo propriedades dinâmicas por DTOs fortemente tipados, melhorando a capacidade de manutenção e a confiabilidade
2. Implementando abstração da camada de transporte e separação da camada de protocolo, melhorando a clareza arquitetônica
3. Fortalecendo os recursos de registro para facilitar o diagnóstico de problemas de comunicação
4. Apresentando suporte de streaming para simplificar a implementação de streaming

Essas melhorias fornecem ao HagiCode uma base de comunicação mais estável e eficiente, permitindo que ele interaja melhor com ferramentas externas de IA e estabelecendo uma base sólida para futura expansão de recursos.
