## Proteção de código .NET na prática: da ofuscação à proteção de máquina virtual

> Este artigo explica como implementar uma estratégia de proteção de código em várias camadas em projetos .NET, abrangendo todo o caminho desde o ofuscamento básico até a proteção profissional de máquinas virtuais.

## Plano de fundo

No desenvolvimento de aplicativos .NET, proteger o código principal, como validação de licença, lógica de negócios e configuração confidencial, contra descompilação e engenharia reversa é, francamente, um tópico que você não pode evitar. À medida que o ecossistema .NET amadureceu, os desenvolvedores obtiveram acesso a uma variedade de opções de proteção, desde atributos de ofuscação integrados até ferramentas de proteção profissionais baseadas em virtualização.

Como um projeto monorepo multilíngue complexo, o HagiCode inclui aplicativos de desktop, sistemas de construção e recursos de gerenciamento de licenças. O código inevitavelmente contém lógica de validação de licença, configurações confidenciais, como chaves de API e IDs de produtos, e lógica crítica para os negócios. Essas partes precisam de proteção séria, porque ninguém quer que seu trabalho duro seja exposto tão facilmente.

Este artigo compartilha a abordagem de proteção de código que adotamos no projeto HagiCode e resume toda a jornada, desde as armadilhas iniciais até a otimização posterior. Esperamos que isso lhe dê algumas idéias úteis.

## Sobre HagiCode

[HagiCode](https://hagicode.com) é um projeto de assistente de codificação de IA de código aberto dedicado a fornecer aos desenvolvedores uma experiência de programação inteligente. O projeto usa uma arquitetura monorepo e mantém simultaneamente uma extensão VSCode, serviços de back-end de IA, um cliente de desktop multiplataforma e muito mais. Essa complexidade multilíngue e multiplataforma torna a proteção de código um desafio de engenharia que temos que enfrentar de frente.

A abordagem compartilhada neste artigo é o resultado de tentativa e erro reais durante o desenvolvimento do HagiCode. Se você quiser ver como resolvemos esses problemas técnicos, continue lendo. Você pode encontrar algumas conclusões inesperadas.

## Conteúdo principal

### 1. Atributo de ofuscação integrado da Microsoft

O .NET Framework fornece um recurso integrado `[ObfuscationAttribute]`, que é o marcador de ofuscação de código mais básico e comumente usado. Este atributo reside no `System.Reflection` namespace e permite aplicar proteção de linha de base ao código sem introduzir ferramentas de terceiros.

**Recursos principais**:
- `Feature` propriedade: especifica o recurso de ofuscação, como `"ultra"` (alta ofuscação) ou `"all"` (ofuscação total)
- `Exclude` propriedade: `true` significa excluir da ofuscação, e `false` significa aplicar ofuscação
- Pode ser aplicado a classes, métodos, propriedades e outros membros de tipo

No projeto HagiCode, você pode vê-lo usado assim:

```csharp
[Obfuscation(Feature = "ultra", Exclude = false)]
public async Task<LicenseValidationResult?> ValidateLicenseAsync(...)
```

As **vantagens** desta abordagem são bastante óbvias:
- Sem dependências extras; ele é integrado ao .NET Framework pronto para uso
- Pode ser reconhecido e processado por ferramentas de ofuscação de terceiros
- Não aumenta significativamente o tamanho do assembly compilado

Dito isto, também tem limitações. É apenas um marcador e o resultado real do ofuscamento depende da implementação da ferramenta. Ele não pode fornecer segurança em nível de proteção de máquina virtual.

### 2. VMP (Proteção de Máquina Virtual)

VMP é uma ferramenta profissional de proteção de código que fornece proteção de alto nível compilando código em instruções de máquina virtual. Ao contrário da simples ofuscação de nomes, o VMP na verdade transforma a lógica do código em uma forma que os descompiladores convencionais não conseguem reconstruir.

**Classificação do nível de proteção**:

| Nível | Virtualização | Mutação | Anti-depuração | Criptografia de string | Casos de uso |
|------|--------|------|--------|------------|----------|
| ALTO | cheio | alto | habilitado | habilitado | Validação de licença, simultaneidade de sessão, constantes confidenciais |
| MÉDIO | parcial | médio | habilitado | habilitado | Lógica de negócios, modelos de domínio |
| BAIXO | nenhum | baixo | desativado | desativado | Classes utilitárias, código não crítico |

O projeto HagiCode define um sistema de atributos declarativos para marcação de código que precisa de proteção:

```csharp
// High-priority protection
[VmProtect(VmProtectionPriority.High, Reason = "Contains license verification logic")]
public class KeygenClient { ... }

// Exclude from protection
[VmExclude(Reason = "Public API that must remain unchanged")]
public class PublicApi { ... }

// Inherited protection
[VmProtect(Priority.High, ProtectDerived = true)]
public class BaseLicenseValidator { ... }
```

### 3. Estratégia de proteção em tempo de construção

A proteção VMP não importa apenas em tempo de execução. Ele também precisa ser automatizado como parte do pipeline de construção, porque fazê-lo manualmente seria muito tedioso. O sistema de construção do HagiCode suporta vários modos:

- **Modo nativo do Windows**: invoque a ferramenta VMProtect diretamente
- **Modo contêiner Linux Docker**: execute o VMP dentro de um contêiner para resolver problemas de compatibilidade entre plataformas
- **Verificação de atributos**: descubra automaticamente marcadores de proteção no código
- **Mecanismo de validação**: confirme se a proteção foi aplicada com sucesso

Juntos, esses recursos tornam o processo muito mais fácil de gerenciar.

## Solução

### 1. Usando o atributo de ofuscação integrado da Microsoft

Aplicar `ObfuscationAttribute` diretamente no código:

```csharp
using System.Reflection;

[Obfuscation(Feature = "ultra", Exclude = false)]
public class LicenseService
{
    [Obfuscation(Feature = "ultra", Exclude = false)]
    public async Task<bool> ValidateLicenseAsync(string key)
    {
        // License validation logic
    }

    [Obfuscation(Feature = "flow", Exclude = false)]
    private string DecryptToken(string encrypted)
    {
        // Decryption logic
    }
}
```

Às vezes, você precisa permitir que os assemblies de teste acessem membros internos e, ao mesmo tempo, manter o código de produção seguro:

```csharp
// AssemblyInfo.cs
[assembly: InternalsVisibleTo("HagiCode.Application.Tests")]
[assembly: InternalsVisibleTo("DynamicProxyGenAssembly2")] // for Moq
```

Isso torna o teste muito mais conveniente, porque o código ainda precisa ser testado adequadamente.

### 2. Definições de atributos personalizados para proteção VMP

Crie atributos de proteção personalizados para controlar o comportamento do VMP:

```csharp
using System;

[AttributeUsage(AttributeTargets.Class | AttributeTargets.Method | AttributeTargets.Property)]
public class VmProtectAttribute : Attribute
{
    public VmProtectionPriority Priority { get; set; }
    public string? Reason { get; set; }
    public bool ProtectDerived { get; set; }
}

[AttributeUsage(AttributeTargets.Class | AttributeTargets.Method | AttributeTargets.Property)]
public class VmExcludeAttribute : Attribute
{
    public string? Reason { get; set; }
}

public enum VmProtectionPriority
{
    None = 0,
    Low = 1,
    Medium = 2,
    High = 3
}
```

Os atributos personalizados costumam ser mais fáceis de trabalhar porque refletem diretamente seus próprios requisitos de proteção.

### 3. Arquivo de configuração VMP

```yaml
# vmp_config.yml
protection:
  priority_mode: "attribute"  # Attribute-based priority
  default_level: "medium"

tools:
  - name: "vmprotect"
    path: "C:\\Program Files\\VMProtect Ultimate\\VMProtect.exe"

protection_levels:
  high:
    virtualization: "full"
    mutation: "high"
    anti_debug: true
    anti_dump: true
    encrypt_strings: true
    encrypt_resources: true

  medium:
    virtualization: "partial"
    mutation: "medium"
    anti_debug: true
    encrypt_strings: true

  low:
    virtualization: "none"
    mutation: "low"
    anti_debug: false
```

Uma configuração mais clara torna o sistema muito mais fácil de manter posteriormente.

## Guia Prático

### 1. Práticas de proteção para componentes críticos

De acordo com a especificação de proteção de código do HagiCode, os seguintes componentes devem usar proteção de ALTA prioridade:

```csharp
// Production constants - must be encrypted and protected by VMP
[VmProtect(VmProtectionPriority.High, Reason = "Production constants")]
public static class ProductionConstants
{
    // Encrypted string accessor, protected by VMP
    [VmProtect(VmProtectionPriority.High)]
    public static string GetLicenseServerUrl(IOptions<LicenseOptions> options) => ...;
}

// License validation logic
[VmProtect(VmProtectionPriority.High, Reason = "License verification logic")]
public class KeygenClient : IKeygenClient
{
    [Obfuscation(Feature = "ultra", Exclude = false)]
    public async Task<LicenseValidationResult?> ValidateLicenseAsync(...) { ... }
}

// Machine fingerprint service
[VmProtect(VmProtectionPriority.High)]
public class MachineFingerprintService : IMachineFingerprintService { ... }
```

O código crítico merece proteção mais forte, porque expor a lógica central causaria problemas reais.

### 2. Criptografia de string e descriptografia em tempo de execução

Criptografe strings em tempo de construção e descriptografe-as em tempo de execução:

```csharp
public static class StringDecryption
{
    [VmProtect(VmProtectionPriority.High, Reason = "CRITICAL SECURITY")]
    public static string DecryptString(byte[] encryptedData, byte[] key, byte[] iv)
    {
        using var aes = Aes.Create();
        aes.Key = key;
        aes.IV = iv;

        using var decryptor = aes.CreateDecryptor();
        using var ms = new MemoryStream(encryptedData);
        using var cs = new CryptoStream(ms, decryptor, CryptoStreamMode.Read);
        using var reader = new StreamReader(cs);

        return reader.ReadToEnd();
    }
}

// Production constant accessor (lazy loading + caching)
public static class ProductionConstants
{
    private static string? _cachedLicenseServerUrl;

    public static string GetLicenseServerUrl(IOptions<LicenseOptions> options)
    {
        if (_cachedLicenseServerUrl == null)
        {
            var encrypted = GetEncryptedLicenseServerUrl();
#if DEBUG
            _cachedLicenseServerUrl = options.Value.PrimaryServer.Url;
#else
            _cachedLicenseServerUrl = StringDecryption.DecryptString(
                encrypted,
                GetEncryptionKey(),
                GetEncryptionIV());
#endif
        }
        return _cachedLicenseServerUrl;
    }
}
```

Esta etapa é importante porque informações confidenciais nunca devem ser deixadas em texto simples.

### 3. Verificação de proteção VMP

Após a construção, você deve verificar se a proteção foi aplicada com sucesso; caso contrário, você não poderá ter certeza de que está realmente funcionando:

```csharp
// Example verification script
public bool VerifyProtection(string assemblyPath)
{
    // 1. Check the VMP signature
    var bytes = File.ReadAllBytes(assemblyPath);
    var vmpSignature = Encoding.ASCII.GetBytes("VMProtect");
    if (bytes.Any(b => vmpSignature.Contains(b)))
    {
        return true;
    }

    // 2. Check for file size changes (the protected file is usually larger)
    var originalInfo = new FileInfo(assemblyPath.Replace(".dll", ".bak"));
    if (originalInfo.Exists)
    {
        var sizeRatio = (double)new FileInfo(assemblyPath).Length / originalInfo.Length;
        return sizeRatio > 1.1;
    }

    return false;
}
```

Sempre vale a pena fazer a verificação, caso contrário os problemas podem passar despercebidos.

### 4. Notas e advertências

Existem várias armadilhas aqui que merecem atenção especial:

1. **Não ofusque todo o código**: APIs públicas, definições de interface e classes DTO geralmente não precisam de proteção. A ofuscação excessiva pode prejudicar o desempenho e a depuração. O projeto HagiCode aprendeu isso da maneira mais difícil.

2. **Proteger acessores de chave**: os métodos que recuperam chaves de criptografia devem receber o mesmo nível de proteção ou um nível de proteção superior ao dos próprios dados criptografados; caso contrário, toda a configuração perde o seu valor.

3. **Equilibrar teste e produção**: as compilações DEBUG devem ignorar a criptografia para facilitar o desenvolvimento e a depuração, enquanto as compilações RELEASE devem permitir proteção total. Lembre-se de separá-los com compilação condicional, como `#if DEBUG`.

4. **Considere o ambiente Docker**: a execução do VMP no Linux requer uma abordagem em contêiner para garantir a compatibilidade da ferramenta. HagiCode usa uma solução de contêiner Wine + VMP para resolver o problema de plataforma cruzada.

5. **A verificação é obrigatória**: após a conclusão da compilação, você deverá verificar se a proteção foi aplicada com êxito. Caso contrário, o código confidencial ainda poderá ser exposto, e o código de verificação mostrado anteriormente existe exatamente para esse propósito.

## Conclusão

Com esta estratégia de proteção multicamadas, a HagiCode construiu um sistema abrangente de segurança de código que abrange desde a ofuscação da linha de base até a proteção da máquina virtual:

- **Camada 1**: Usar `ObfuscationAttribute` para marcação de linha de base e fornecer dicas para ferramentas de terceiros
- **Camada 2**: Use personalizado `VmProtectAttribute` declarações para expressar intenção e prioridade de proteção
- **Camada 3**: Use a proteção de máquina virtual VMP para transformar código crítico em instruções irreversíveis de máquina virtual
- **Camada 4**: verifica e aplica proteção automaticamente durante a compilação e, em seguida, verifica o resultado

Essa abordagem pode resistir a ferramentas comuns de descompilação e, ao mesmo tempo, resistir melhor a ataques avançados de engenharia reversa. Se você estiver criando um aplicativo .NET que precisa de proteção de código, espero que isso forneça pelo menos um ponto de referência útil.
