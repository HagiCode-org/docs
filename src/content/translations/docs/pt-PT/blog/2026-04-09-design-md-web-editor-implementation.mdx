## Edite DESIGN.md diretamente na interface da Web: da ideia à implementação

> No sistema de gerenciamento de projetos MonoSpecs, `DESIGN.md` carrega o projeto arquitetônico e as decisões técnicas de um projeto. Mas o fluxo de trabalho de edição tradicional força os usuários a recorrer a um editor externo. Essa experiência fragmentada é como ser interrompido no meio da leitura de um poema: a inspiração se foi, e o clima também. Este artigo compartilha a solução que colocamos em prática no projeto HagiCode: editar `DESIGN.md` diretamente na interface web, com suporte para importação de templates de um site de design online. Afinal, quem não gosta da sensação de completar tudo em um só fluxo?

## Plano de fundo

Como principal portador dos documentos de concepção do projeto, `DESIGN.md` contém informações importantes, como projeto de arquitetura, decisões técnicas e orientações de implementação. No entanto, a abordagem de edição tradicional exige que os usuários mudem para um editor externo, como o VS Code, localizem manualmente o caminho físico e editem o arquivo. Não é especialmente complicado, mas depois de repetir o processo algumas vezes torna-se cansativo.

Os problemas aparecem principalmente das seguintes maneiras:

- **Fluxo de trabalho fragmentado**: os usuários precisam alternar constantemente entre a interface de gerenciamento da Web e um editor local, quebrando a continuidade do fluxo de trabalho, como se a música fosse cortada no meio de uma música.
- **Difícil de reutilizar**: o site de design já publica uma rica biblioteca de modelos de design, mas eles não podem ser integrados diretamente no fluxo de trabalho de edição do projeto. As coisas boas existem, mas você ainda não pode usá-las onde precisa.
- **Loop de experiência ausente**: não há loop fechado para "visualizar-selecionar-importar", portanto os usuários devem copiar e colar manualmente, o que aumenta o risco de erros.
- **Atrito na colaboração**: manter os documentos de design e a implementação do código sincronizados torna-se um processo de alto atrito, o que prejudica a eficiência da equipe.

Para resolver esses problemas, decidimos adicionar suporte de edição direta para `DESIGN.md` na interface da web e permite a importação de modelos com um clique de um site de design online. Não foi uma decisão que abalou a terra. Queríamos simplesmente tornar a experiência de desenvolvimento mais tranquila.

## Sobre HagiCode

A solução compartilhada neste artigo vem de nossa experiência prática no [HagiCode](https://hagicode.com) projeto. HagiCode é um projeto assistente de codificação baseado em IA e, durante o desenvolvimento, frequentemente precisamos manter os documentos de design do projeto. Para ajudar a equipe a colaborar de forma mais eficiente, exploramos e implementamos esta solução online de edição e importação. Não há nada de misterioso nisso. Encontramos um problema e descobrimos uma maneira de resolvê-lo.

## Solução Técnica

### Arquitetura Geral

Esta solução usa uma arquitetura separada frontend-backend com um proxy de mesma origem, composto principalmente pelas seguintes camadas. Na prática, o design pode ser resumido como “cada parte fazendo seu próprio trabalho”:

**1. Camada do editor de front-end**
```typescript
// Core component: DesignMdManagementDrawer
// Location: repos/web/src/components/project/DesignMdManagementDrawer.tsx
// Responsibility: handle editing, saving, version conflict detection, and import flow
```

**2. Camada de serviço de back-end**
```csharp
// Core service: ProjectAppService.DesignMd
// Location: repos/hagicode-core/src/PCode.Application/ProjectAppService.DesignMd.cs
// Responsibility: path resolution, file read/write, and version management
```

**3. Camada proxy de mesma origem**
```csharp
// Proxy service: ProjectAppService.DesignMdSiteIndex
// Location: repos/hagicode-core/src/PCode.Application/ProjectAppService.DesignMdSiteIndex.cs
// Responsibility: proxy design site resources, preview image caching, and security validation
```

### Principais decisões técnicas

#### Decisão 1: Padrão Global de Gaveta

Usamos uma única gaveta global em vez de camadas pop-up locais, com estado gerenciado por meio de `layoutSlice`, que oferece aos usuários uma experiência consistente em todas as visualizações (`classic` e `kanban`). Não importa de qual visualização o usuário abre o editor, eles obtêm o mesmo modelo de interação. Uma experiência consistente faz com que as pessoas se sintam mais à vontade, em vez de ficarem desorientadas ao mudar de opinião.

#### Decisão 2: API com escopo de projeto

Nós montamos `DESIGN.md`endpoints relacionados em `ProjectController`, reutilizando o limite de permissão do projeto existente e evitando a complexidade de adicionar um controlador separado. Isso torna o tratamento de permissões mais claro e também se alinha com a organização de recursos RESTful. Às vezes, a reutilização é mais significativa do que criar algo novo do zero.

#### Decisão 3: Detecção de Conflitos de Versão

Derivamos uma versão opaca do sistema de arquivos `LastWriteTimeUtc`, o que nos dá um controle de simultaneidade otimista e leve. Quando vários usuários editam o mesmo arquivo ao mesmo tempo, o sistema pode detectar conflitos e solicitar que o usuário atualize. Este design não bloqueia a edição, ao mesmo tempo que protege a consistência dos dados.

#### Decisão 4: Padrão de proxy da mesma origem

Nós usamos `IHttpClientFactory` para proxy de recursos externos do local de projeto, evitando problemas de origem cruzada e riscos de SSRF. Isso mantém o sistema seguro e ao mesmo tempo simplifica as chamadas de front-end. Você dificilmente pode ser muito cuidadoso com a segurança.

## Implementação principal

### 1. Edite DESIGN.md diretamente

#### Implementação de back-end

O back-end é o principal responsável pela resolução de caminhos, leitura/gravação de arquivos e gerenciamento de versões. Estas tarefas são básicas, mas indispensáveis, como a fundação de uma casa:

```csharp
// Path resolution and security validation
private Task<string> ResolveDesignDocumentDirectoryAsync(string projectPath, string? repositoryPath)
{
    if (string.IsNullOrWhiteSpace(repositoryPath))
    {
        return Task.FromResult(Path.GetFullPath(projectPath));
    }
    return ValidateSubPathAsync(projectPath, repositoryPath);
}

// Version generation (based on file system timestamp and size)
private static string BuildDesignDocumentVersion(string path)
{
    var fileInfo = new FileInfo(path);
    fileInfo.Refresh();
    return string.Create(
        CultureInfo.InvariantCulture,
        $"{fileInfo.LastWriteTimeUtc.Ticks:x}-{fileInfo.Length:x}");
}
```

O design da versão é interessante por sua simplicidade: usamos a hora e o tamanho da última modificação do arquivo para gerar um identificador de versão exclusivo. É leve e confiável, sem nenhum banco de dados de versão extra para manter. Soluções simples são muitas vezes as mais eficazes.

#### Implementação de front-end

No frontend, implementamos a detecção de estado sujo e salvamos a lógica. Esse design ajuda os usuários a entender se suas alterações foram salvas e reduz a ansiedade de “e se eu perdê-las?”:

```typescript
// Dirty-state detection and save logic
const [draft, setDraft] = useState('');
const [savedDraft, setSavedDraft] = useState('');
const isDirty = draft !== savedDraft;

const handleSave = useCallback(async () => {
    const result = await saveProjectDesignMdDocument({
        ...activeTarget,
        content: draft,
        expectedVersion: document.version, // optimistic concurrency control
    });
    setSavedDraft(draft); // update saved state
}, [activeTarget, document, draft]);
```

Nesta implementação, mantemos dois estados: `draft` é o conteúdo que está sendo editado no momento, enquanto `savedDraft` é o conteúdo salvo. Compará-los nos diz se há alterações não salvas. O design é simples, mas dá tranquilidade às pessoas. Ninguém quer que aquilo em que trabalharam arduamente desapareça.

#### Formato de dados de índice

O arquivo de índice no site de design define todos os modelos disponíveis. Com este índice, os usuários podem escolher o modelo que desejam com a mesma facilidade com que fazem o pedido em um menu:


Cada entrada inclui as informações básicas do modelo e links para download. O backend lê a lista de modelos disponíveis neste índice e os apresenta para o usuário escolher. Isso torna a seleção intuitiva, em vez de forçar as pessoas a se orientarem no escuro.

#### Implementação de proxy de mesma origem

Para manter tudo seguro, o back-end realiza uma validação rigorosa no acesso ao site de design. Você não pode ser muito cauteloso com a segurança:

```csharp
// Safe slug validation
private static readonly Regex SafeDesignSiteSlugRegex =
    new("^[A-Za-z0-9](?:[A-Za-z0-9._-]{0,127})$", RegexOptions.Compiled);

private static string NormalizeDesignSiteSlug(string slug)
{
    var normalizedSlug = slug?.Trim() ?? string.Empty;
    if (!IsSafeDesignSiteSlug(normalizedSlug))
    {
        throw new BusinessException(
            ProjectDesignSiteIndexErrorCodes.InvalidSlug,
            "Design site slug must be a single safe path segment.");
    }
    return normalizedSlug;
}

// Preview image caching (OS temp directory)
private static string ComputePreviewCacheKey(string slug, string theme, string previewUrl)
{
    var raw = $"{slug}|{theme}|{previewUrl}";
    var bytes = SHA256.HashData(Encoding.UTF8.GetBytes(raw));
    return Convert.ToHexString(bytes).ToLowerInvariant();
}
```

Fazemos duas coisas aqui: primeiro, validamos o formato do slug estritamente com uma expressão regular para evitar ataques de passagem de caminho; segundo, armazenamos imagens de visualização em cache para reduzir a pressão no site externo. A primeira é proteção, a segunda é otimização e ambas são importantes.

### 3. Fluxo completo de importação

```typescript
// 1. Open the import drawer
const handleRequestImportDrawer = useCallback(() => {
    setIsImportDrawerOpen(true);
}, []);

// 2. Select and import
const handleImportRequest = useCallback((entry) => {
    if (isDirty) {
        setPendingImportEntry(entry);
        setConfirmMode('import'); // overwrite confirmation
        return;
    }
    void executeImport(entry);
}, [isDirty]);

// 3. Execute import
const executeImport = useCallback(async (entry) => {
    const result = await getProjectDesignMdSiteImportDocument(
        activeTarget.projectId,
        entry.slug
    );
    setDraft(result.content); // replace editor text only, do not save automatically
    setIsImportDrawerOpen(false);
}, [activeTarget?.projectId]);
```

O fluxo de importação segue o princípio de “confirmação do usuário”: após a importação, apenas o conteúdo do editor é atualizado e nada é salvo automaticamente. Os usuários podem inspecionar o conteúdo importado e salvá-lo manualmente somente após confirmarem que está correto. A decisão final deve ficar nas mãos do usuário.

## Exemplos práticos

### Cenário 1: Criando DESIGN.md na raiz do projeto

Quando `DESIGN.md` não existe, o back-end retorna um estado de documento virtual. Isso permite que o frontend evite tratamento especial para o caso "arquivo não existe" e uma API unificada simplifica a lógica do código:

```csharp
return new ProjectDesignDocumentDto
{
    Path = targetPath,
    Exists = false,  // virtual document state
    Content = string.Empty,
    Version = null
};

// Automatically create the file on first save
public async Task<SaveProjectDesignDocumentResultDto> SaveDesignDocumentAsync(...)
{
    Directory.CreateDirectory(targetDirectory);
    await File.WriteAllTextAsync(targetPath, input.Content);
    return new SaveProjectDesignDocumentResultDto { Created = !exists };
}
```

A vantagem desse design é que o frontend não precisa de lógica de casos especiais para arquivos ausentes. Ao ocultar essa complexidade no backend, o frontend pode se concentrar mais facilmente na experiência do usuário.

### Cenário 2: importar um modelo do site de design

Após o usuário selecionar o modelo de design "Linear" na gaveta de importação, o sistema busca o `DESIGN.md` conteúdo por meio do proxy de back-end. Todo o processo é transparente para o usuário: ele escolhe apenas um modelo e o sistema trata automaticamente das solicitações de rede e da transformação dos dados.

```typescript
// 1. The system fetches DESIGN.md content through the backend proxy
GET /api/project/{id}/design-md/site-index/linear.app

// 2. The backend validates the slug and fetches content from upstream
var entry = FindDesignSiteEntry(catalog, "linear.app");
using var upstreamResponse = await httpClient.SendAsync(request);
var content = await upstreamResponse.Content.ReadAsStringAsync();

// 3. The frontend replaces the editor text
setDraft(result.content);
// The user reviews it and then saves it manually to disk
```

Todo o fluxo permanece transparente para o usuário. Eles apenas escolhem um modelo e o sistema cuida da rede e da transformação nos bastidores. Essa é a experiência que queremos: simples, mas poderosa.

### Cenário 3: Lidando com Conflitos de Versão

Quando vários usuários editam o mesmo `DESIGN.md` ao mesmo tempo, o sistema detecta conflitos de versão. Este mecanismo otimista de controle de simultaneidade preserva a consistência dos dados sem bloquear as edições do usuário:

```csharp
if (!string.Equals(currentVersion, expectedVersion, StringComparison.Ordinal))
{
    throw new BusinessException(
        ProjectDesignDocumentErrorCodes.VersionConflict,
        $"DESIGN.md at '{targetPath}' changed on disk.");
}
```

O frontend detecta esse erro e avisa ao usuário:

```typescript
// Frontend prompts the user to refresh and retry
<Alert>
    <AlertTitle>Version conflict</AlertTitle>
    <AlertDescription>
        The file was modified by another process. Please refresh to get the latest version and try again.
    </AlertDescription>
</Alert>
```

Esse mecanismo otimista de controle de simultaneidade mantém os dados consistentes sem bloquear os usuários enquanto eles trabalham. Os conflitos são inevitáveis, mas pelo menos os usuários devem saber o que aconteceu, em vez de perderem silenciosamente as alterações.

## Notas e melhores práticas

### 1. Segurança do caminho

Sempre validar `repositoryPath` para evitar ataques de passagem de caminho. Você nunca pode fazer muito quando se trata de segurança:

```csharp
// Always validate repositoryPath to prevent path traversal attacks
return ValidateSubPathAsync(projectPath, repositoryPath);
// Reject dangerous inputs such as "../" and absolute paths
```

### 2. Estratégia de Cache

Imagens de visualização em cache por 24 horas, com no máximo 160 arquivos. O cache moderado melhora o desempenho, mas o equilíbrio ainda é importante:

```csharp
// Cache preview images for 24 hours, with a maximum of 160 files
private static readonly TimeSpan PreviewCacheTtl = TimeSpan.FromHours(24);
private const int PreviewCacheMaxFiles = 160;
// Periodically clean up expired cache
```

### 3. Tratamento de erros

Degradação normal quando o site upstream não está disponível. Esse design garante que, mesmo que uma dependência externa falhe, a funcionalidade principal de edição ainda funcione normalmente:

```typescript
// Gracefully degrade when the upstream site is unavailable
try {
    const catalog = await getProjectDesignMdSiteImportCatalog(projectId);
} catch (error) {
    toast.error(t('project.designMd.siteImport.feedback.catalogLoadFailed'));
    // The main editing drawer remains available
}
```

Essa degradação elegante garante que, mesmo quando as dependências externas não estiverem disponíveis, a função principal de edição continue funcionando. Um sistema deve ser resiliente em vez de entrar em colapso no momento em que algo dá errado.

### 4. Otimização da experiência do usuário

Confirme as substituições antes de importar e não salve automaticamente após a importação. Os usuários devem manter o controle de suas próprias ações:

```typescript
// Confirm overwrite before import
if (isDirty) {
    setConfirmMode('import');
    return;
}

// Do not save automatically after import; let the user confirm
setDraft(result.content); // update draft only
// The content is written to disk only after the user reviews it and clicks Save
```

### 5. Considerações de desempenho

Use uma fábrica de cliente HTTP para evitar a criação de muitas conexões. A gestão de recursos pode parecer pequena, mas fazê-la bem pode fazer uma grande diferença:

```csharp
// Use an HTTP client factory to avoid creating too many connections
private const string DesignSiteProxyClientName = "ProjectDesignSiteProxy";
private static readonly TimeSpan DesignSiteProxyTimeout = TimeSpan.FromSeconds(8);
```

## Extensões sugeridas

1. **Aprimoramento de Markdown**: atualmente usamos um básico `Textarea`, mas poderíamos atualizar para CodeMirror para realce de sintaxe e atalhos de teclado. Quando o editor se sente melhor, escrever a documentação também é melhor.
2. **Modo de visualização**: adicione visualização do Markdown em tempo real para melhorar a experiência de edição. O que você vê é o que você obtém sempre dá mais confiança às pessoas.
3. **Mesclagem de diferenças**: implemente um algoritmo de mesclagem inteligente em vez de uma simples substituição de texto completo. Os conflitos são inevitáveis, mas o processo de resolução de conflitos não tem de ser doloroso.
4. **Cache local**: cache `design.json` no banco de dados para reduzir a dependência do site externo. Quanto menos dependências um sistema tiver, mais estável ele tende a ser.

## Resumo

No projeto HagiCode implementamos uma solução completa de edição e importação online para `DESIGN.md` por meio da colaboração front-end-backend. O valor central desta solução reside nos seguintes pontos:

- **Maior eficiência**: não há necessidade de trocar de ferramenta; a edição e importação de documentos de design podem acontecer em uma interface web unificada.
- **Menor barreira de entrada**: a importação de modelos de design com um clique ajuda novos projetos a serem iniciados rapidamente.
- **Seguro e confiável**: validação de caminho, detecção de conflitos de versão e mecanismos de degradação elegantes mantêm o sistema estável.
- **Melhor experiência do usuário**: a gaveta global, a detecção de estado sujo e as caixas de diálogo de confirmação refinam a experiência geral de interação.

Esta solução já está em execução no projeto HagiCode e resolveu os problemas da equipe em relação ao gerenciamento de documentos de design. Se você estiver enfrentando problemas semelhantes, espero que este artigo lhe dê algumas ideias úteis.
