## Construindo uma integração de problemas do GitHub do zero: prática de conexão direta front-end do HagiCode

> Este artigo documenta o processo completo de integração do GitHub Issues na plataforma HagiCode. Exploraremos como usar uma arquitetura de "conexão direta de front-end + back-end mínimo" para obter autenticação OAuth segura e sincronização eficiente de problemas, mantendo o back-end leve.

## Antecedentes: Por que integrar o GitHub?

Como plataforma de desenvolvimento assistida por IA, o valor central do HagiCode reside em conectar ideias com implementação. Mas, no uso real, descobrimos que depois que os usuários concluem uma proposta no HagiCode, eles geralmente precisam copiar manualmente o conteúdo para o GitHub Issues para rastrear o projeto.

Isso cria vários pontos problemáticos óbvios:

1.  **Fluxo de trabalho fragmentado**: os usuários precisam alternar entre dois sistemas. A experiência não é tranquila e informações importantes podem ser facilmente perdidas durante a cópia e colagem.
2.  **Colaboração inconveniente**: outros membros da equipe estão acostumados a verificar tarefas no GitHub e não podem ver diretamente o progresso da proposta dentro do HagiCode.
3.  **Trabalho manual repetido**: sempre que uma proposta é atualizada, alguém precisa atualizar manualmente o problema correspondente no GitHub, adicionando custos de manutenção desnecessários.

Para resolver esse problema, decidimos introduzir o recurso **GitHub Issues Integration**, conectando sessões HagiCode com repositórios GitHub para permitir a "sincronização com um clique".

## Escolha técnica: conexão direta de front-end versus proxy de back-end

Ao projetar a abordagem de integração, tínhamos duas opções diante de nós: o tradicional "modelo de proxy de back-end" e o mais agressivo "modelo de conexão direta de front-end".

### Comparação de soluções

No **modelo de proxy de back-end** tradicional, cada solicitação do front-end deve primeiro passar pelo nosso back-end, que então chama a API do GitHub. Isso centraliza a lógica, mas também sobrecarrega significativamente o back-end:

1.  **Back-end inchado**: precisaríamos escrever um wrapper de cliente GitHub API dedicado e também lidar com a complexa máquina de estado OAuth.
2.  **Risco de token**: o token GitHub do usuário teria que ser armazenado no banco de dados de back-end. Mesmo com criptografia, isso ainda aumenta a superfície de segurança.
3.  **Custo de desenvolvimento**: precisaríamos de migrações de banco de dados para armazenar tokens e de um serviço de sincronização adicional para manter.

O **modelo de conexão direta frontend** é muito mais leve. Nessa abordagem, usamos o back-end apenas para a etapa mais sensível de “troca secreta” (o retorno de chamada do OAuth). Após obter o token, armazenamos ele diretamente no navegador `localStorage`. Operações posteriores, como criação de problemas e atualização de comentários, são enviadas diretamente do frontend para o GitHub por HTTP.

| Dimensão de comparação | Modelo de proxy de back-end | Modelo de conexão direta de front-end |
| :--- | :--- | :--- |
| **Complexidade de back-end** | Requer um serviço OAuth completo e um cliente GitHub API | Precisa apenas de um endpoint de retorno de chamada OAuth |
| **Gerenciamento de tokens** | Deve ser criptografado e armazenado em banco de dados, com risco de vazamento | Armazenado no navegador e visível apenas para o usuário |
| **Custo de implementação** | Requer migrações de banco de dados e desenvolvimento multisserviço | Principalmente trabalho de front-end |
| **Experiência do usuário** | Lógica centralizada, mas a latência do servidor pode ser um pouco maior | Resposta extremamente rápida com interação direta com GitHub |

Como queríamos uma integração rápida e alterações mínimas de back-end, **escolhemos o "modelo de conexão direta de front-end"**. É como dar um “passe temporário” ao navegador. Depois de aprovado, o navegador pode cuidar das coisas no GitHub sozinho, sem pedir aprovação ao administrador de back-end todas as vezes.

## Design Principal: Fluxo de Dados e Segurança

Depois de definir a arquitetura, precisávamos projetar o fluxo de dados específico. O núcleo do processo de sincronização é como obter o token com segurança e usá-lo de forma eficiente.

### Diagrama geral da arquitetura

Todo o sistema pode ser abstraído em três funções: o navegador (frontend), o backend do HagiCode e o GitHub.

```text
+--------------+        +--------------+        +--------------+
| Frontend     |        |   Backend    |        |   GitHub     |
| React        |        |   ASP.NET    |        |   REST API   |
|              |        |              |        |              |
|  +--------+  |        |              |        |              |
|  | OAuth  |--+--------> /callback    |        |              |
|  | Flow   |  |        |              |        |              |
|  +--------+  |        |              |        |              |
|              |        |              |        |              |
|  +--------+  |        |  +--------+  |        |  +--------+  |
|  | GitHub |  +------------> Session |  +----------> Issues |  |
|  | API    |  |        |  |Metadata|  |        |  |        |  |
|  | Direct |  |        |  +--------+  |        |  +--------+  |
|  +--------+  |        |              |        |              |
+--------------+        +--------------+        +--------------+
```

**O ponto principal é**: apenas uma pequena etapa no OAuth (troca do código por um token) precisa passar pelo backend. Depois disso, o trabalho pesado (criar problemas) é feito diretamente entre o frontend e o GitHub.

### Fluxo de dados de sincronização em detalhes

Quando o usuário clica no botão "Sincronizar com GitHub" na UI do HagiCode, uma série de ações complexas ocorre:

```text
User clicks "Sync to GitHub"
         │
         ▼
1. Frontend checks localStorage for the GitHub token
         │
         ▼
2. Format issue content (convert the Proposal into Markdown)
         │
         ▼
3. Frontend directly calls the GitHub API to create/update the issue
         │
         ▼
4. Call the HagiCode backend API to update Session.metadata (store issue URL and other info)
         │
         ▼
5. Backend broadcasts the SessionUpdated event via SignalR
         │
         ▼
6. Frontend receives the event and updates the UI to show the "Synced" state
```

### Projeto de segurança

A segurança é sempre a principal prioridade na integração de serviços de terceiros. Fizemos as seguintes considerações:

1.  **Defenda-se contra ataques CSRF**: Gere um `state` parâmetro durante o redirecionamento OAuth e armazená-lo em `sessionStorage`. Valide estritamente o estado no retorno de chamada para evitar solicitações forjadas.
2.  **Armazenamento de token isolado**: O token é armazenado apenas no navegador `localStorage`. Usando a Política de Mesma Origem, apenas scripts HagiCode podem lê-lo, evitando o risco de um vazamento de banco de dados do lado do servidor afetar os usuários.
3.  **Limites de erro**: Projetamos um tratamento dedicado para erros comuns da API do GitHub (como token expirado 401, falha de validação 422 e limitação de taxa 429), para que os usuários recebam feedback amigável.

### 1. Mudanças mínimas de back-end

O backend só precisa fazer duas coisas: armazenar informações de sincronização e lidar com o retorno de chamada do OAuth.

### 2. Fluxo OAuth de front-end

Este é o ponto de entrada da conexão. Usamos o fluxo de código de autorização padrão.

```typescript
// src/HagiCode.Client/src/services/githubOAuth.ts

// Generate the authorization URL and redirect
export async function generateAuthUrl(): Promise<string> {
  const state = generateRandomString(); // Generate a random string for CSRF protection
  sessionStorage.setItem('hagicode_github_state', state);
  
  const params = new URLSearchParams({
    client_id: clientId,
    redirect_uri: window.location.origin + '/settings?tab=github&oauth=callback',
    scope: ['repo', 'public_repo'].join(' '),
    state: state,
  });
  
  return `https://github.com/login/oauth/authorize?${params.toString()}`;
}

// Handle the code-to-token exchange on the callback page
export async function exchangeCodeForToken(code: string, state: string): Promise<GitHubToken> {
  // 1. Validate state to prevent CSRF
  const savedState = sessionStorage.getItem('hagicode_github_state');
  if (state !== savedState) throw new Error('Invalid state parameter');

  // 2. Call the backend API to exchange the token
  // Note: this must go through the backend because ClientSecret cannot be exposed to the frontend
  const response = await fetch('/api/GitHubOAuth/callback', {
    method: 'POST',
    headers: { 'Content-Type': 'application/json' },
    body: JSON.stringify({ code, state, redirectUri: window.location.origin + '/settings?tab=github&oauth=callback' }),
  });

  if (!response.ok) throw new Error('Failed to exchange token');
  
  const token = await response.json();
  
  // 3. Save into LocalStorage
  saveToken(token);
  return token;
}
```

### 3. Wrapper do cliente da API GitHub

Assim que tivermos o token, precisaremos de uma ferramenta sólida para chamar a API do GitHub.

```typescript
// src/HagiCode.Client/src/services/githubApiClient.ts

const GITHUB_API_BASE = 'https://api.github.com';

// Core request wrapper
async function githubApi<T>(endpoint: string, options: RequestInit = {}): Promise<T> {
  const token = localStorage.getItem('gh_token');
  if (!token) throw new Error('Not connected to GitHub');
  
  const response = await fetch(`${GITHUB_API_BASE}${endpoint}`, {
    ...options,
    headers: {
      ...options.headers,
      Authorization: `Bearer ${token}`,
      Accept: 'application/vnd.github.v3+json', // Specify the API version
    },
  });
  
  // Error handling logic
  if (!response.ok) {
    if (response.status === 401) throw new Error('GitHub token expired, please reconnect');
    if (response.status === 403) throw new Error('No permission to access this repository or rate limit exceeded');
    if (response.status === 422) throw new Error('Issue validation failed, the title may be duplicated');
    throw new Error(`GitHub API Error: ${response.statusText}`);
  }
  
  return response.json();
}

// Create issue
export async function createIssue(owner: string, repo: string, data: { title: string, body: string, labels: string[] }) {
  return githubApi(`/repos/${owner}/${repo}/issues`, {
    method: 'POST',
    body: JSON.stringify(data),
  });
}
```

### 4. Formatação e sincronização de conteúdo

A etapa final é converter os dados da sessão HagiCode no formato de um problema do GitHub. É um pouco como um trabalho de tradução.

```typescript
// Convert a Session object into a Markdown string
function formatIssueForSession(session: Session): string {
  let content = `# ${session.title}\n\n`;
  content += `**> HagiCode Session:** #${session.code}\n`;
  content += `**> Status:** ${session.status}\n\n`;
  content += `## Description\n\n${session.description || 'No description provided.'}\n\n`;
  
  // If this is a Proposal session, add extra fields
  if (session.type === 'proposal') {
    content += `## Chief Complaint\n\n${session.chiefComplaint || ''}\n\n`;
    // Add a deep link so users can jump back from GitHub to HagiCode
    content += `---\n\n**[View in HagiCode](hagicode://sessions/${session.id})**\n`;
  }
  
  return content;
}

// Main logic when clicking the sync button
const handleSync = async (session: Session) => {
  try {
    const repoInfo = parseRepositoryFromUrl(session.repoUrl); // Parse the repository URL
    if (!repoInfo) throw new Error('Invalid repository URL');

    toast.loading('Syncing to GitHub...');
    
    // 1. Format content
    const issueBody = formatIssueForSession(session);
    
    // 2. Call API
    const issue = await githubApiClient.createIssue(repoInfo.owner, repoInfo.repo, {
      title: `[HagiCode] ${session.title}`,
      body: issueBody,
      labels: ['hagicode', 'proposal', `status:${session.status}`],
    });
    
    // 3. Update Session Metadata (save the issue link)
    await SessionsService.patchApiSessionsSessionId(session.id, {
      metadata: {
        githubIssue: {
          owner: repoInfo.owner,
          repo: repoInfo.repo,
          issueNumber: issue.number,
          issueUrl: issue.html_url,
          syncedAt: new Date().toISOString(),
        }
      }
    });

    toast.success('Synced successfully!');
  } catch (err) {
    console.error(err);
    toast.error('Sync failed, please check your token or network');
  }
};
```

## Resumo e Perspectiva

Com essa abordagem de “conexão direta de front-end”, alcançamos uma integração perfeita do GitHub Issues com o mínimo de código de back-end possível.

### Principais ganhos
1.  **Alta eficiência de desenvolvimento**: As alterações de back-end são mínimas, principalmente um campo de banco de dados extra e um endpoint de retorno de chamada OAuth simples. A maior parte da lógica é concluída no frontend.
2.  **Forte segurança**: O token não passa pelo banco de dados do servidor, reduzindo o risco de vazamento.
3.  **Ótima experiência do usuário**: as solicitações são iniciadas diretamente no front-end, portanto, a velocidade de resposta é rápida e não há necessidade de encaminhamento no back-end.

### Coisas a observar
Existem algumas armadilhas a serem lembradas durante a implantação real:
- **Configurações do aplicativo OAuth**: lembre-se de inserir o valor correto `Authorization callback URL` nas configurações do aplicativo GitHub OAuth (geralmente `http://localhost:3000/settings?tab=github&oauth=callback`).
- **Limites de taxa**: a API do GitHub limita as solicitações não autenticadas de forma bastante estrita, mas com um token a cota geralmente é suficiente (5.000 solicitações/hora).
- **Análise de URL**: os usuários inserem todos os tipos de URLs de repositório, portanto, certifique-se de que sua regex corresponda `.git` sufixos, formatos SSH e casos semelhantes.

### Melhorias futuras
O recurso atual ainda é a sincronização unilateral (HagiCode -> GitHub). No futuro, planejamos implementar a sincronização bidirecional por meio de GitHub Webhooks. Por exemplo, se um problema for encerrado no GitHub, o estado da sessão no lado do HagiCode também poderá ser atualizado automaticamente. Isso exigirá que exponhamos um endpoint de webhook no back-end, o que será uma próxima etapa interessante.
