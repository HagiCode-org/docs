## Guia prático para integração do CodeBuddy CLI em um back-end C#

> Este artigo apresenta uma abordagem completa para integrar o CodeBuddy CLI em um projeto de back-end C# para que você possa fornecer recursos de assistente de codificação de IA de ponta a ponta.

:::nota[Contexto de tempo]
Este artigo discute o design de integração do CodeBuddy e a estrutura do provedor conforme existiam no momento da publicação. Para o escopo atual com suporte permanente, consulte [Instalação da CLI do Agente AI](/pt-PT/related-software-installation/ai-agent-cli/) e o [Guia de instalação](/pt-PT/installation); `IFlowCli` agora é retido apenas como um valor histórico de compatibilidade e não faz parte do escopo de suporte ativo atual.
:::

## Plano de fundo

No desenvolvimento moderno de assistentes de codificação de IA, um único provedor de IA muitas vezes não consegue satisfazer cenários de desenvolvimento complexos e mutáveis. HagiCode, como assistente multifuncional de codificação de IA, precisa oferecer suporte a vários provedores de IA para oferecer uma melhor experiência ao usuário. Os usuários devem ter liberdade suficiente para escolher. No início de 2026, o projeto enfrentou uma decisão importante: como restaurar os recursos de integração CodeBuddy ACP (Agent Communication Protocol) no backend C#.

O projeto já havia implementado a integração do CodeBuddy, mas o código relacionado foi removido durante uma refatoração. Não há muito do que reclamar aí; durante o desenvolvimento iterativo, algo sempre fica para trás. O objetivo desta solução técnica era restaurar totalmente essa capacidade e melhorar a arquitetura para que fosse mais robusta e de fácil manutenção.

Se você também está pensando em conectar vários assistentes de codificação de IA ao seu próprio projeto, a abordagem abaixo pode lhe dar algumas ideias. Ele reflete as lições que resumimos depois de passar por muitas armadilhas e talvez possa ajudá-lo a evitar alguns desvios.

## Sobre HagiCode

A abordagem compartilhada neste artigo vem de nossa experiência prática no [HagiCode](https://hagicode.com) projeto. HagiCode usa um design modular, com provedores de IA implementados como componentes conectáveis. ## Projeto de Arquitetura

### Visão geral da arquitetura em camadas

A integração entre C# e CodeBuddy usa uma arquitetura em camadas clara. Este design torna as responsabilidades explícitas e torna a manutenção a longo prazo muito mais fácil:

```
┌─────────────────────────────────────────────┐
│           Provider Contract Layer           │
│   AIProviderType enum + extension methods   │
├─────────────────────────────────────────────┤
│           Provider Factory Layer            │
│   AIProviderFactory dependency injection factory │
├─────────────────────────────────────────────┤
│           Provider Implementation Layer     │
│   CodebuddyCliProvider concrete implementation │
├─────────────────────────────────────────────┤
│           ACP Infrastructure Layer          │
│  ACPSessionManager / StdioAcpTransport      │
│  AcpRpcClient / AcpAgentClient              │
└─────────────────────────────────────────────┘
```

Quais são os benefícios dessa estratificação? Simplificando, cada camada fica fora do caminho das outras. Se mais tarde quisermos alterar o mecanismo de comunicação, por exemplo de `stdio` para o WebSocket, precisamos apenas modificar a camada inferior e a lógica de negócios acima dela permanece intacta. Ninguém quer que uma mudança na comunicação se espalhe por toda a base de código.

### Detalhamento dos componentes principais

A **camada de contrato do provedor** é a base de toda a arquitetura. Nós definimos o `AIProviderType` enum, onde `CodebuddyCli = 3` é usado como o valor enum e implementa o mapeamento bidirecional entre strings e enums por meio de métodos de extensão. Isso permite que strings em arquivos de configuração sejam convertidas convenientemente em enums, e enums sejam convertidas novamente em strings para depuração de saída.

A **camada de fábrica do Provedor** é responsável por criar a instância do Provedor correspondente com base na configuração. Ele usa injeção de dependência .NET junto com `ActivatorUtilities.CreateInstance` para criação dinâmica. A vantagem do padrão de fábrica é que ao adicionar um novo Provedor, você só precisa adicionar a lógica de criação em vez de modificar o código existente.

A **camada de implementação do provedor** é onde o trabalho real acontece. `CodebuddyCliProvider` implementa o `IAIProvider` interface e fornece dois modos de invocação: `ExecuteAsync` para chamadas sem streaming e `StreamAsync` para chamadas de streaming.

A **camada de infraestrutura ACP** fornece a base de comunicação subjacente. Esta camada lida com todos os detalhes do protocolo, incluindo gerenciamento de processos, serialização de mensagens e análise de respostas. É a base que mantém estável tudo acima dela.

## Mecanismo de Comunicação

### Modo de transporte estúdio

CodeBuddy usa **Stdio (entrada/saída padrão)** para se comunicar com processos externos. O comando de inicialização é simples:

```bash
codebuddy --acp
```

Depois disso, as mensagens JSON-RPC são trocadas por meio de entrada e saída padrão. Esta abordagem tem várias vantagens:

1. **Inicialização rápida**: a comunicação do processo local evita a latência da rede
2. **Configuração simples**: você só precisa especificar o caminho do executável
3. **Isolamento de ambiente**: cada sessão é executada em um processo independente, para que não afetem uma à outra

A injeção de variáveis de ambiente é suportada durante a comunicação. Exemplos comuns incluem:

- `CODEBUDDY_API_KEY`: Autenticação de chave API
- `CODEBUDDY_INTERNET_ENVIRONMENT`: configuração do ambiente de rede

Tal como acontece com a comunicação entre pessoas, é útil escolher primeiro um canal conveniente.

### Protocolo de mensagem

ACP é baseado em JSON-RPC 2.0. O formato da mensagem é mais ou menos assim:

```json
// Request message
{
  "jsonrpc": "2.0",
  "id": 1,
  "method": "agent/prompt",
  "params": {
    "prompt": "Help me write a sorting algorithm",
    "sessionId": "session-123"
  }
}

// Response message
{
  "jsonrpc": "2.0",
  "id": 1,
  "result": {
    "content": "Here is the AI response..."
  }
}
```

Na implementação real, encapsulamos todos esses detalhes do protocolo para que a camada superior de negócios só precise se preocupar com o prompt e a resposta.

## Implementação principal

### 1. Restaure o contrato do fornecedor

Primeiro, restaure o tipo CodeBuddy no arquivo enum:

```csharp

public enum AIProviderType
{

    CodexCli = 1,
    GitHubCopilot = 2,

    IFlowCli = 5,

```

Em seguida, adicione mapeamento de string nos métodos de extensão para que o arquivo de configuração possa especificar o Provedor por string:

```csharp

private static readonly Dictionary<string, AIProviderType> _typeMap = new(
    StringComparer.OrdinalIgnoreCase)
{
    ["CodebuddyCli"] = AIProviderType.CodebuddyCli,
    ["Codebuddy"] = AIProviderType.CodebuddyCli,
    ["codebuddy"] = AIProviderType.CodebuddyCli,
    // ... Mappings for other providers
};
```

### 2. Integrar a Fábrica de Provedores

Adicione uma ramificação de criação do CodeBuddy na classe de fábrica:

```csharp
// AIProviderFactory.cs
private IAIProvider? CreateProvider(AIProviderType providerType, ProviderConfiguration config)
{
    return providerType switch
    {
        AIProviderType.CodebuddyCli =>
            ActivatorUtilities.CreateInstance<CodebuddyCliProvider>(
                _serviceProvider,
                Options.Create(config)),
        // ... Other providers
        _ => throw new NotSupportedException($"Provider {providerType} not supported")
    };
}
```

Isso usa injeção de dependência por meio de `ActivatorUtilities`, que lida automaticamente com a injeção de parâmetros do construtor e é muito conveniente.

### 3. Implementação completa do provedor

Abaixo está a implementação principal do `CodebuddyCliProvider`, abrangendo modos de invocação de streaming e não streaming:

```csharp
public class CodebuddyCliProvider : IAIProvider
{
    private readonly ILogger<CodebuddyCliProvider> _logger;
    private readonly IACPSessionManager _sessionManager;
    private readonly ProviderConfiguration _config;

    public string Name => "CodebuddyCli";
    public bool SupportsStreaming => true;
    public ProviderCapabilities Capabilities { get; }

    public CodebuddyCliProvider(
        ILogger<CodebuddyCliProvider> logger,
        IACPSessionManager sessionManager,
        IOptions<ProviderConfiguration> config)
    {
        _logger = logger;
        _sessionManager = sessionManager;
        _config = config.Value;

        // Define the capabilities of the current Provider
        Capabilities = new ProviderCapabilities
        {
            SupportsStreaming = true,
            SupportsTools = true,
            SupportsSystemMessages = true,
            SupportsArtifacts = false,
            MaxTokens = 8192
        };
    }

    // Non-streaming call: return all results together after completion
    public async Task<AIResponse> ExecuteAsync(
        AIRequest request,
        CancellationToken cancellationToken = default)
    {
        // Create an independent session for the request
        var session = await _sessionManager.CreateSessionAsync(
            "CodebuddyCli",
            request.WorkingDirectory,
            cancellationToken,
            request.SessionId);

        try
        {
            var fullPrompt = BuildPrompt(request);
            await session.SendPromptAsync(fullPrompt, cancellationToken);

            var responseBuilder = new StringBuilder();
            var toolCalls = new List<AIToolCall>();

            // Collect all response chunks
            await foreach (var chunk in StreamFromSession(session, cancellationToken))
            {
                if (!string.IsNullOrEmpty(chunk.Content))
                {
                    responseBuilder.Append(chunk.Content);
                }
                // Handle tool calls...
            }

            return new AIResponse
            {
                Content = AIResultContentSanitizer.SanitizeResultContent(
                    responseBuilder.ToString()),
                ToolCalls = toolCalls,
                Provider = Name,
                Model = string.Empty
            };
        }
        finally
        {
            // Release session resources
            await session.DisposeAsync();
        }
    }

    // Streaming call: return response chunks in real time
    public async IAsyncEnumerable<AIStreamingChunk> StreamAsync(
        AIRequest request,
        [EnumeratorCancellation] CancellationToken cancellationToken = default)
    {
        var session = await _sessionManager.CreateSessionAsync(
            "CodebuddyCli",
            request.WorkingDirectory,
            cancellationToken);

        try
        {
            var fullPrompt = BuildPrompt(request);
            await session.SendPromptAsync(fullPrompt, cancellationToken);

            await foreach (var chunk in StreamFromSession(session, cancellationToken))
            {
                yield return chunk;
            }
        }
        finally
        {
            await session.DisposeAsync();
        }
    }

    private async IAsyncEnumerable<AIStreamingChunk> StreamFromSession(
        IACPSession session,
        [EnumeratorCancellation] CancellationToken cancellationToken)
    {
        // Iterate through all updates in the session
        await foreach (var notification in session.ReceiveUpdatesAsync(cancellationToken))
        {
            switch (notification.Update)
            {
                case AgentMessageChunkSessionUpdate agentMessage:
                    // Handle text content chunks
                    if (agentMessage.Content is AcpImp.TextContentBlock textContent)
                    {
                        yield return new AIStreamingChunk
                        {
                            Content = textContent.Text,
                            Type = StreamingChunkType.ContentDelta,
                            IsComplete = false
                        };
                    }
                    break;

                case ToolCallSessionUpdate toolCall:
                    // Handle tool calls
                    yield return new AIStreamingChunk
                    {
                        Content = string.Empty,
                        Type = StreamingChunkType.ToolCallDelta,
                        ToolCallDelta = new AIToolCallDelta
                        {
                            Id = toolCall.ToolCallId,
                            Name = toolCall.Kind.ToString(),
                            Arguments = toolCall.RawInput?.ToString()
                        }
                    };
                    break;

                case AcpImp.PromptCompletedSessionUpdate:
                    // Response complete
                    yield break;
            }
        }
    }

    // Build the full prompt
    private string BuildPrompt(AIRequest request, string? embeddedCommandPrompt = null)
    {
        var sb = new StringBuilder();

        // Embedded command prompt, if present
        if (!string.IsNullOrEmpty(embeddedCommandPrompt))
        {
            sb.AppendLine(embeddedCommandPrompt);
            sb.AppendLine();
        }

        // System message
        if (!string.IsNullOrEmpty(request.SystemMessage))
        {
            sb.AppendLine(request.SystemMessage);
            sb.AppendLine();
        }

        // User prompt
        sb.Append(request.Prompt);
        return sb.ToString();
    }
}
```

Existem vários pontos-chave neste código:

1. **Gerenciamento de sessões**: cada solicitação cria uma sessão independente e libera recursos após a conclusão da solicitação. Esta é uma lição aprendida por tentativa e erro. Se a reutilização da sessão não for bem tratada, a poluição do estado aparecerá facilmente.

2. **Processamento de streaming**: `IAsyncEnumerable` permite que a resposta seja retornada enquanto ainda está sendo gerada, em vez de esperar que todo o conteúdo termine. Isto é especialmente importante para cenários de texto longo e melhora significativamente a experiência do usuário.

3. **Chamadas de ferramentas**: CodeBuddy suporta chamadas de ferramentas (chamadas de função), tratadas por meio de `ToolCallSessionUpdate`. Esse recurso é fundamental para tarefas complexas de edição de código.

4. **Filtragem de conteúdo**: `AIResultContentSanitizer` é usado para filtrar o conteúdo do bloco Think e manter a saída limpa.

### 4. Configuração de injeção de dependência

Adicione os serviços relacionados durante o registro do módulo:

```csharp
// PCodeClaudeHelperModule.cs
public void ConfigureModule(IServiceCollection context)
{
    // Register Provider
    context.Services.AddTransient<CodebuddyCliProvider>();

    // Register ACP infrastructure
    context.Services.AddSingleton<IACPSessionManager, ACPSessionManager>();
    context.Services.AddSingleton<IAcpPlatformConfigurationResolver, AcpPlatformConfigurationResolver>();
    context.Services.AddSingleton<IAIRequestToAcpMapper, AIRequestToAcpMapper>();
    context.Services.AddSingleton<IAcpToAIResponseMapper, AcpToAIResponseMapper>();
}
```

## Exemplo de configuração

### Arquivo de configuração

Adicione configuração relacionada ao CodeBuddy para `appsettings.json`:

```yaml
AI:
  # Default Provider to use
  DefaultProvider: "CodebuddyCli"

  # Provider configuration
  Providers:
    CodebuddyCli:
      Type: "CodebuddyCli"
      WorkingDirectory: "C:/projects/my-app"
      ExecutablePath: "C:/tools/codebuddy.cmd"

  # Platform-specific configuration
  PlatformConfigurations:
    CodebuddyCli:
      ExecutablePath: "C:/tools/codebuddy.cmd"
      Arguments: "--acp"
      StartupTimeoutMs: 5000
      EnvironmentVariables:
        CODEBUDDY_API_KEY: "${CODEBUDDY_API_KEY}"
        CODEBUDDY_INTERNET_ENVIRONMENT: "production"
```

A definição do modelo de configuração correspondente:

    public int StartupTimeoutMs { get; set; } = 5000;

    public Dictionary<string, string?>? EnvironmentVariables { get; set; }
}

## Lições práticas aprendidas

### Registro de armadilha

Encontramos várias armadilhas típicas durante a implementação, e compartilhá-las aqui pode ajudar outras pessoas a evitar os mesmos desvios:

1. **Problema de vazamento de sessão**: a princípio as sessões não eram liberadas corretamente, o que esgotava os recursos do processo. A solução foi usar `try-finally` para garantir que os recursos sejam liberados para cada solicitação.

2. **Passagem de variável de ambiente**: Windows e Linux usam sintaxe de variável de ambiente diferente, por isso padronizamos posteriormente `Dictionary<string, string?>` para lidar com isso.

3. **Configuração de tempo limite**: a inicialização da CLI leva tempo, por isso definimos um tempo limite de inicialização de 5 segundos para evitar falhas rápidas de solicitação.

4. **Problemas de codificação**: no Windows, a codificação padrão pode causar texto em chinês distorcido, portanto a codificação UTF-8 é especificada explicitamente ao iniciar o processo.

### Otimização de desempenho

1. **Pool de sessões**: para solicitações curtas frequentes, considere implementar um pool de sessões para reutilizar processos
2. **Cache de conexão**: a classe de fábrica já suporta cache de instâncias do Provedor
3. **Assíncrono primeiro**: use programação assíncrona para evitar o bloqueio de threads

Sempre vale a pena otimizar o desempenho. Quanto mais os usuários esperam, pior se torna a experiência.

## Conclusão

Este artigo apresentou uma solução completa para integração do CodeBuddy CLI em um back-end C#, cobrindo todo o processo, desde o design da arquitetura até a implementação concreta. Através de uma arquitetura em camadas, separamos os detalhes do protocolo da lógica de negócio, tornando o código mais claro e fácil de manter.

Principais conclusões:

- Use uma arquitetura em camadas com uma camada de contrato de provedor, uma camada de fábrica, uma camada de implementação e uma camada de infraestrutura
- Use JSON-RPC sobre Stdio para comunicação entre processos
- Implemente configuração flexível e extensibilidade por meio de injeção de dependência
- Fornece modos de invocação de streaming e não streaming

Essa abordagem não é adequada apenas para CodeBuddy; adicionar novos provedores de IA pode seguir o mesmo padrão. Se você também estiver construindo uma integração semelhante de vários provedores de IA, espero que este artigo forneça uma referência útil.
