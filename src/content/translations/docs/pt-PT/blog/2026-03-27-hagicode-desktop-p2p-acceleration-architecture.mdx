## Arquitetura de distribuição híbrida HagiCode Desktop explicada: como o P2P acelera downloads de arquivos grandes

## Plano de fundo

As equipes que criam aplicativos de desktop enfrentarão a mesma dor de cabeça, mais cedo ou mais tarde: como distribuir arquivos grandes?

É um problema estranho. Os downloads diretos HTTP/HTTPS tradicionais ainda podem funcionar quando os arquivos são pequenos e o número de usuários é limitado. Mas o tempo raramente é gentil. À medida que um projeto continua crescendo, os pacotes de instalação também crescem: pacotes ZIP para desktop, pacotes portáteis, arquivos de implantação da web e muito mais. Então os problemas começam a surgir:

- **A velocidade de download é limitada pela largura de banda de origem**: não importa quanta largura de banda um único servidor tenha, ele ainda terá dificuldades quando todos fizerem downloads ao mesmo tempo.
- **O suporte para retomada é quase inexistente**: se um download HTTP for interrompido, muitas vezes você terá que recomeçar desde o início. Isso desperdiça tempo e largura de banda.
- **O servidor de origem assume toda a pressão**: todo o tráfego flui de volta para um servidor central, os custos de largura de banda continuam aumentando e a escalabilidade se torna um problema real.

O projeto HagiCode Desktop não foi exceção. Quando projetamos o sistema de distribuição, nos perguntamos: podemos introduzir uma abordagem de distribuição híbrida sem alterar o sistema existente? `index.json` plano de controle? Em outras palavras, podemos usar a natureza distribuída das redes P2P para acelerar downloads e ao mesmo tempo manter o recurso de origem HTTP para que o sistema permaneça utilizável em ambientes restritos, como redes corporativas?

O impacto dessa decisão acabou sendo maior do que você poderia esperar. Vamos percorrer isso passo a passo.

## Filosofia de Design Central: P2P Primeiro, HTTP Fallback

Em sua essência, o modelo de distribuição híbrida pode ser resumido em uma única frase: **P2P primeiro, HTTP substituto**.

A chave está na palavra “híbrido”. Não se trata simplesmente de adicionar o BitTorrent e encerrar o dia. O objetivo é fazer com que os dois métodos de entrega funcionem juntos e se complementem:

- **A rede P2P** oferece aceleração distribuída. Quanto mais pessoas baixam, mais peers ingressam e mais rápida se torna a transferência.
- **O substituto WebSeed/HTTP** garante a disponibilidade, para que os downloads ainda possam funcionar em firewalls corporativos e ambientes de rede internos.
- **O plano de controle permanece simples**. Não mudamos a lógica central do `index.json`; adicionamos apenas alguns campos de metadados opcionais.

O benefício real é simples: os usuários sentem que “os downloads são mais rápidos”, enquanto a equipe de engenharia não precisa arcar com muita complexidade extra. Afinal, o protocolo BT já está maduro e há poucos motivos para reinventar a roda.

## Projeto de Arquitetura

### Visão geral da arquitetura em camadas

Vamos começar com o diagrama geral da arquitetura para construir um modelo mental de alto nível:

```
┌─────────────────────────────────────┐
│     Renderer (UI layer)             │
├─────────────────────────────────────┤
│     IPC/Preload (bridge layer)      │
├─────────────────────────────────────┤
│   VersionManager (version manager)  │
├─────────────────────────────────────┤
│ HybridDownloadCoordinator (coord.)  │
│  ├── DistributionPolicyEvaluator    │
│  ├── DownloadEngineAdapter          │
│  ├── CacheRetentionManager          │
│  └── SHA256 Verifier                │
├─────────────────────────────────────┤
│   WebTorrent (download engine)      │
└─────────────────────────────────────┘
```

Como mostra o diagrama, o sistema usa um design em camadas. A razão para separar responsabilidades desta forma é claramente simples: **testabilidade e substituibilidade**.

- **A camada UI** é responsável por exibir o progresso do download e alternar a aceleração de compartilhamento. É a superfície.
- **A camada de coordenação** é o núcleo. Ele contém avaliação de políticas, adaptação de mecanismo, gerenciamento de cache e verificação de integridade.
- **A camada do mecanismo** encapsula a implementação concreta do download. No momento, utiliza WebTorrent.

A camada do motor é abstraída por trás do `DownloadEngineAdapter` interface. Se quisermos trocar um mecanismo BT diferente posteriormente ou mover a implementação para um processo secundário, isso se tornará muito mais fácil.

### Separação do Plano de Controle e Plano de Dados

HagiCode Desktop mantém `index.json` como o único plano de controle, e esse design é crítico. O plano de controle é responsável pela descoberta de versão, seleção de canal e política centralizada, enquanto o plano de dados é onde ocorre a transferência real de arquivos.

Os novos campos adicionados a `index.json` são opcionais:

```json
{
  "asset": {
    "torrentUrl": "https://cdn.example.com/app.torrent",
    "infoHash": "abc123...",
    "webSeeds": [
      "https://cdn.example.com/app.zip",
      "https://backup.example.com/app.zip"
    ],
    "sha256": "def456...",
    "directUrl": "https://cdn.example.com/app.zip"
  }
}
```

Todos esses campos são opcionais. Se estiverem faltando, o cliente volta ao modo de download HTTP tradicional. A vantagem desse design é a **compatibilidade com versões anteriores**: clientes mais antigos não são afetados.

### Decisões Orientadas por Políticas

Nem todo arquivo vale a pena ser distribuído via P2P.

`DistributionPolicyEvaluator` é responsável pela avaliação da política. Somente arquivos que atendam a todas as condições a seguir usarão o download híbrido:

1. **O tipo de origem deve ser um índice HTTP**: downloads diretos do GitHub ou fontes de pastas locais não usam esse caminho.
2. **O tamanho do arquivo deve ser de pelo menos 100 MB**: para arquivos menores, a sobrecarga do P2P supera o benefício.
3. **Os metadados híbridos completos devem estar presentes**: `torrentUrl`, `webSeeds`e `sha256` são todos obrigatórios.
4. **Apenas o pacote de desktop e o pacote de implantação na Web mais recentes são elegíveis**: as versões históricas continuam a usar o caminho de distribuição tradicional.

```typescript
class DistributionPolicyEvaluator {
  evaluate(version: Version, settings: SharingAccelerationSettings): HybridDownloadPolicy {
    // Check source type
    if (version.sourceType !== 'http-index') {
      return { useHybrid: false, reason: 'not-http-index' };
    }

    // Check metadata completeness
    if (!version.hybrid) {
      return { useHybrid: false, reason: 'not-eligible' };
    }

    // Check whether the feature is enabled
    if (!settings.enabled) {
      return { useHybrid: false, reason: 'shared-disabled' };
    }

    // Check asset type (latest desktop/web packages only)
    if (!version.hybrid.isLatestDesktopAsset && !version.hybrid.isLatestWebAsset) {
      return { useHybrid: false, reason: 'latest-only' };
    }

    return { useHybrid: true, reason: 'shared-enabled' };
  }
}
```

Isso dá ao sistema **comportamento previsível**. Tanto os desenvolvedores quanto os usuários podem entender claramente quais arquivos usarão P2P e quais não.

## Implementação principal

### Sistema de definição de tipo

Comecemos com as definições de tipo, pois elas constituem a base de todo o sistema.

```typescript
// Hybrid distribution metadata
interface HybridDistributionMetadata {
  torrentUrl?: string;      // Torrent file URL
  infoHash?: string;        // InfoHash
  webSeeds: string[];       // WebSeed list
  sha256?: string;          // File hash
  directUrl?: string;       // HTTP direct link (for origin fallback)
  eligible: boolean;        // Whether hybrid distribution is applicable
  thresholdBytes: number;   // Threshold in bytes
  assetKind: VersionAssetKind;
  isLatestDesktopAsset: boolean;
  isLatestWebAsset: boolean;
}

// Sharing acceleration settings
interface SharingAccelerationSettings {
  enabled: boolean;           // Master switch
  uploadLimitMbps: number;    // Upload bandwidth limit
  cacheLimitGb: number;       // Cache limit
  retentionDays: number;      // Retention period
  hybridThresholdMb: number;  // Hybrid distribution threshold
  onboardingChoiceRecorded: boolean;
}

// Download progress
interface VersionDownloadProgress {
  current: number;
  total: number;
  percentage: number;
  stage: VersionInstallStage;  // queued, downloading, backfilling, verifying, extracting, completed, error
  mode: VersionDownloadMode;   // http-direct, shared-acceleration, source-fallback
  peers?: number;              // Number of connected peers
  p2pBytes?: number;           // Bytes received from P2P
  fallbackBytes?: number;      // Bytes received from fallback
  verified?: boolean;          // Whether verification has completed
}
```

Uma vez que o sistema de tipos esteja claro, o resto da implementação segue naturalmente.

### Coordenador do Núcleo

`HybridDownloadCoordinator` orquestra todo o fluxo de trabalho de download. Ele coordena a avaliação de políticas, a execução do mecanismo, a verificação SHA256 e o ​​gerenciamento de cache.

```typescript
class HybridDownloadCoordinator {
  async download(
    version: Version,
    cachePath: string,
    packageSource: PackageSource,
    onProgress?: DownloadProgressCallback,
  ): Promise<HybridDownloadResult> {
    // 1. Evaluate the policy: should hybrid download be used?
    const policy = this.policyEvaluator.evaluate(version, settings);

    // 2. Execute the download
    if (policy.useHybrid) {
      await this.engine.download(version, cachePath, settings, onProgress);
    } else {
      await packageSource.downloadPackage(version, cachePath, onProgress);
    }

    // 3. SHA256 verification (hard gate)
    const verified = await this.verify(version, cachePath, onProgress);
    if (!verified) {
      await this.cacheRetentionManager.discard(version.id, cachePath);
      throw new Error(`sha256 verification failed for ${version.id}`);
    }

    // 4. Mark as trusted cache and begin controlled seeding
    await this.cacheRetentionManager.markTrusted({
      versionId: version.id,
      cachePath,
      cacheSize,
    }, settings);

    return { cachePath, policy, verified };
  }
}
```

Há um ponto especialmente importante aqui: **A verificação SHA256 é uma porta rígida**. Um arquivo baixado deve passar pela verificação antes de poder entrar no fluxo de instalação. Se a verificação falhar, o cache será descartado para garantir que um arquivo incorreto nunca cause problemas de instalação.

### Baixar abstração do mecanismo

`DownloadEngineAdapter` é uma interface abstrata que define os métodos que cada mecanismo deve implementar:

```typescript
interface DownloadEngineAdapter {
  download(
    version: Version,
    destinationPath: string,
    settings: SharingAccelerationSettings,
    onProgress?: (progress: VersionDownloadProgress) => void,
  ): Promise<void>;

  stopAll(): Promise<void>;
}
```

A implementação V1 é baseada em WebTorrent e está envolvida em `InProcessTorrentEngineAdapter`:

```typescript
class InProcessTorrentEngineAdapter implements DownloadEngineAdapter {
  async download(...) {
    const client = this.getClient(settings);  // Apply upload rate limiting
    const torrent = client.add(torrentId, {
      path: path.dirname(destinationPath),
      destroyStoreOnDestroy: false,
      maxWebConns: 8,
    });

    // Add WebSeed sources
    torrent.on('ready', () => {
      for (const seed of hybrid.webSeeds) {
        torrent.addWebSeed(seed);
      }
      if (hybrid.directUrl) {
        torrent.addWebSeed(hybrid.directUrl);
      }
    });

    // Progress reporting - distinguish P2P from origin fallback
    torrent.on('download', () => {
      const hasP2PPeer = torrent.wires.some(w => w.type !== 'webSeed');
      const mode = hasP2PPeer ? 'shared-acceleration' : 'source-fallback';
      // ... report progress
    });
  }
}
```

Um design de motor conectável torna a otimização futura muito mais fácil. Por exemplo, V2 poderia executar o mecanismo em um processo auxiliar para evitar a interrupção do processo principal se o mecanismo travar.

### Distinguindo modos de relatório de progresso

Na camada da interface do usuário, o que mais preocupa os usuários é simples: "estou baixando atualmente por meio de P2P ou de substituto HTTP?" `InProcessTorrentEngineAdapter` determina isso verificando os tipos dentro `torrent.wires`:

```typescript
const hasP2PPeer = torrent.wires.some((wire) => wire.type !== 'webSeed');
const hasFallbackWire = torrent.wires.some((wire) => wire.type === 'webSeed');

const mode = hasP2PPeer ? 'shared-acceleration'
         : hasFallbackWire ? 'source-fallback'
         : 'shared-acceleration';

const stage = hasP2PPeer ? 'downloading'
           : hasFallbackWire ? 'backfilling'
           : 'downloading';
```

A lógica parece simples, mas é uma parte fundamental da experiência do usuário. Os usuários podem ver claramente se o estado atual é “aceleração de compartilhamento” ou “preenchimento de origem”, o que torna o comportamento mais fácil de entender.

### Verificação de streaming SHA256

A verificação de integridade usa Node.js `crypto` módulo para calcular o hash de maneira streaming, o que evita carregar o arquivo inteiro na memória:

```typescript
private async computeSha256(filePath: string): Promise<string> {
  const hash = createHash('sha256');
  await new Promise<void>((resolve, reject) => {
    const stream = fs.createReadStream(filePath);
    stream.on('data', (chunk) => hash.update(chunk));
    stream.on('error', reject);
    stream.on('end', resolve);
  });
  return hash.digest('hex').toLowerCase();
}
```

Esta implementação é especialmente amigável para arquivos grandes. Imagine baixar um pacote de instalação de 2 GB e tentar carregar tudo na memória apenas para verificar. O streaming resolve isso de forma limpa.

## Produtização

Mesmo o melhor design técnico fracassará se a experiência do usuário for ruim. O HagiCode Desktop investiu bastante esforço na produção desse recurso.

### Ocultar terminologia BT

A maioria dos usuários não sabe o que significa BitTorrent ou InfoHash. Assim, no nível do produto, apresentamos o recurso usando a frase “aceleração de compartilhamento”:

- O recurso é chamado de “aceleração de compartilhamento”, e não de download P2P.
- A configuração é chamada de “limite de upload”, e não de propagação.
- O rótulo de progresso diz “preenchimento de origem”, não substituto do WebSeed.

Isto reduz a carga cognitiva da terminologia e torna o recurso mais fácil de aceitar.

### Ativado por padrão no assistente de primeira execução

Quando novos usuários iniciam o aplicativo de desktop pela primeira vez, eles veem uma página do assistente apresentando a aceleração de compartilhamento:

> Para melhorar a velocidade de download, compartilhamos as partes que você já baixou com outros usuários enquanto o seu download está em andamento. Isso é totalmente opcional e você pode desativá-lo a qualquer momento nas Configurações.

Ele está habilitado por padrão, mas os usuários têm uma maneira clara de cancelar. Se os usuários corporativos não quiserem, eles podem simplesmente desativá-lo durante a integração.

### Parâmetros controlados pelo usuário

A página de configurações expõe três parâmetros ajustáveis:

| Parâmetro | Padrão | Descrição |
|------|--------|------|
| Limite de upload | 2MB/s | Evita o uso excessivo de largura de banda upstream |
| Limite de cache | 10GB | Controla o consumo de espaço em disco |
| Dias de retenção | 7 dias | Limpa automaticamente o cache antigo após este período |

Todos esses parâmetros têm padrões razoáveis. A maioria dos usuários nunca precisa alterá-los, enquanto os usuários avançados podem ajustá-los com base em seu próprio ambiente de rede.

## Principais decisões de design

Olhando para trás, para a solução geral, vale a pena destacar várias decisões de design.

### O motor funciona no processo principal (V1)

Por que não começar imediatamente com um processo secundário ou auxiliar? A razão é simples: **envie rapidamente**. Um design em processo tem um ciclo de desenvolvimento mais curto e é mais fácil de depurar. A primeira prioridade é colocar o recurso em execução e depois melhorar a estabilidade.

É claro que esta decisão tem um custo: se o motor falhar, pode afetar o processo principal. Reduzimos esse risco por meio de limites de adaptadores e controles de tempo limite, e também mantemos um caminho de migração aberto para que a V2 possa passar para um processo separado com mais facilidade.

### SHA256 como verificação de integridade

Usamos SHA256 em vez de MD5 ou CRC32 porque SHA256 é mais seguro. O custo de colisão para MD5 e CRC32 é muito baixo. Se alguém criasse maliciosamente um pacote de instalação falso, as consequências poderiam ser graves. O SHA256 custa mais para ser computado, mas o ganho de segurança vale a pena.

### Habilitado somente para fontes de índice HTTP

Cenários como downloads do GitHub e fontes de pastas locais não usam distribuição híbrida. Esta não é uma limitação técnica; trata-se de **evitar complexidade desnecessária**. Os protocolos BT agregam valor limitado em cenários de redes privadas e apenas aumentariam a complexidade do código.

## Notas Práticas

### Normalização de configurações

Dentro `SharingAccelerationSettingsStore`, todo valor numérico deve passar pela verificação e normalização de limites:

```typescript
private normalize(settings: SharingAccelerationSettings): SharingAccelerationSettings {
  return {
    enabled: Boolean(settings.enabled),
    uploadLimitMbps: this.clampNumber(settings.uploadLimitMbps, 1, 200, DEFAULT_SETTINGS.uploadLimitMbps),
    cacheLimitGb: this.clampNumber(settings.cacheLimitGb, 1, 500, DEFAULT_SETTINGS.cacheLimitGb),
    retentionDays: this.clampNumber(settings.retentionDays, 1, 90, DEFAULT_SETTINGS.retentionDays),
    hybridThresholdMb: DEFAULT_SETTINGS.hybridThresholdMb,  // Fixed value, not user-configurable
    onboardingChoiceRecorded: Boolean(settings.onboardingChoiceRecorded),
  };
}

private clampNumber(value: number, min: number, max: number, fallback: number): number {
  if (!Number.isFinite(value)) {
    return fallback;
  }
  return Math.min(max, Math.max(min, Math.round(value)));
}
```

Isso evita que os usuários editem manualmente o arquivo de configuração em valores inválidos.

### Limpeza de cache LRU

`CacheRetentionManager.prune()` é responsável por limpar entradas de cache expiradas ou superdimensionadas. A estratégia de limpeza usa LRU (usado menos recentemente):

```typescript
const records = [...this.listRecords()]
  .sort((left, right) =>
    new Date(left.lastUsedAt).getTime() - new Date(right.lastUsedAt).getTime()
  );

// When over the limit, evict the least recently used entries first
while (totalBytes > maxBytes && retainedEntries.length > 0) {
  const evicted = records.find((record) => retainedEntries.includes(record.versionId));
  retainedEntries.splice(retainedEntries.indexOf(evicted.versionId), 1);
  removedEntries.push(evicted.versionId);
  totalBytes -= evicted.cacheSize;
  await fs.rm(evicted.cachePath, { force: true });
}
```

Essa lógica garante que o espaço em disco seja usado de forma eficiente, preservando versões históricas que o usuário ainda possa precisar.

### Comportamento imediato de interrupção da propagação

Quando o usuário desativa a aceleração de compartilhamento, o aplicativo deve interromper imediatamente a propagação e destruir o cliente de torrent:

```typescript
async disableSharingAcceleration(): Promise<void> {
  this.settingsStore.updateSettings({ enabled: false });
  await this.cacheRetentionManager.stopAllSeeding();  // Stop seeding
  await this.engine.stopAll();  // Destroy the torrent client
}
```

Se um usuário desabilitar o recurso, o produto não deverá mais consumir recursos P2P. Essa é a etiqueta básica do produto.

## Riscos e compensações

Não existe uma solução perfeita e a distribuição híbrida não é exceção. Estas são as principais compensações:

**O isolamento de colisão é mais fraco que um sidecar**: V1 usa um mecanismo em processo, portanto, uma falha no motor pode afetar o processo principal. Os limites do adaptador e os controles de tempo limite reduzem o risco, mas não são uma solução fundamental. V2 inclui um caminho de migração planejado para um processo auxiliar.

**Uso de recursos habilitado por padrão**: as configurações padrão de upload de 2 MB/s, cache de 10 GB e retenção de 7 dias consomem alguns recursos da máquina. As expectativas do usuário são gerenciadas por meio de cópia de integração e configurações transparentes.

**Compatibilidade de redes corporativas**: o fallback automático de WebSeed/HTTPS preserva a usabilidade em redes corporativas, mas pode reduzir os ganhos de aceleração do P2P. Esta é uma troca intencional que prioriza a disponibilidade.

**Metadados compatíveis com versões anteriores**: todos os novos campos são opcionais. Se estiverem faltando, o sistema volta ao modo HTTP. Os clientes mais antigos não são afetados, facilitando as atualizações.

## Conclusão

Este artigo percorreu a arquitetura de distribuição híbrida usada no projeto HagiCode Desktop. As principais conclusões são:

1. **Arquitetura em camadas**: o plano de controle e o plano de dados são separados, e o mecanismo é abstraído por trás de uma interface conectável para testes e extensões mais fáceis.

2. **Comportamento orientado por políticas**: nem todo arquivo usa P2P. A distribuição híbrida é habilitada apenas para arquivos grandes que atendam às condições exigidas.

3. **Verificação de integridade**: SHA256 serve como uma porta rígida e a verificação de streaming evita pressão de memória.

4. **Apresentação produzida**: a terminologia BT está oculta atrás da frase "aceleração de compartilhamento" e o recurso é ativado por padrão durante a integração.

5. **Controle do usuário**: limites de upload, limites de cache, dias de retenção e outros parâmetros permanecem ajustáveis pelo usuário.

Esta arquitetura já foi implementada no projeto HagiCode Desktop. Se você experimentar, adoraríamos ouvir seus comentários após a instalação e uso no mundo real.
