## Executando ferramentas AI CLI em contêineres Docker: um guia prático para isolamento de usuários e volumes persistentes

> Integrar ferramentas de codificação de IA como Claude Code, Codex e OpenCode em ambientes conteinerizados parece simples, mas existem complexidades ocultas em todos os lugares. Este artigo se aprofunda em como o projeto HagiCode resolve os principais desafios nas implantações do Docker, incluindo permissões de usuário, persistência de configuração e gerenciamento de versão, para que você possa evitar as armadilhas comuns.

## Plano de fundo

Quando decidimos executar ferramentas CLI de codificação de IA dentro de contêineres Docker, o pensamento mais intuitivo foi provavelmente: "Os contêineres não são apenas root? Por que não instalar tudo diretamente e encerrar?" Na realidade, esta ideia aparentemente simples esconde vários problemas fundamentais que devem ser resolvidos.

Primeiro, **as restrições de segurança são o primeiro obstáculo**. Tomemos como exemplo o Claude CLI: ele proíbe explicitamente a execução como usuário root. Esta é uma verificação de segurança obrigatória e, se o root for detectado, ele se recusa a iniciar. Você pode pensar: não posso simplesmente trocar de usuário com o `USER` diretiva? Não é tão simples. Ainda há um problema de mapeamento entre o usuário não root dentro do contêiner e as permissões do usuário na máquina host.

Em segundo lugar, **a persistência do estado é a segunda armadilha**. Claude Code requer login, Codex possui configuração própria e OpenCode também possui um diretório de cache. Se você tiver que reconfigurar tudo toda vez que o contêiner for reiniciado, toda a ideia de “automação” perde o sentido. Precisamos que essas configurações persistam além do ciclo de vida do contêiner.

O terceiro problema é a **consistência de permissão**. Os processos dentro do contêiner podem acessar arquivos de configuração criados pelo usuário host? As incompatibilidades de UID/GID geralmente causam erros de permissão de arquivo, e isso é extremamente comum em implantações reais.

Estes problemas podem parecer independentes, mas na prática estão intimamente ligados. Durante o desenvolvimento do HagiCode, gradualmente elaboramos uma solução prática. A seguir, compartilharei os detalhes técnicos e as lições aprendidas com essas armadilhas.

## Por que não podemos simplesmente usar root?

Há um mal-entendido comum aqui: os contêineres do Docker são executados como root por padrão, então por que não apenas instalar as ferramentas como root? Se você pensa assim, Claude CLI rapidamente lhe ensinará o contrário.

```bash
# Run Claude CLI directly as root? No.
docker run --rm -it --user root myimage claude
# Output: Error: This command cannot be run as root user
```

Esta é uma restrição de segurança rígida no Claude CLI. O motivo é simples: essas ferramentas CLI leem e gravam configurações confidenciais do usuário, incluindo tokens de API, caches locais e até mesmo scripts escritos pelo usuário. Executá-los com privilégios de root apresenta muitos riscos.

Portanto, a questão é: como podemos satisfazer os requisitos de segurança da CLI e, ao mesmo tempo, manter o gerenciamento de contêineres flexível? Precisamos mudar a forma como pensamos sobre isso: **em vez de trocar de usuário em tempo de execução, crie um usuário dedicado durante a fase de construção da imagem**.

## Criando um usuário dedicado: mais do que apenas mudar um nome

Você pode pensar que adicionar um único `USER` linha para o Dockerfile é suficiente. Esta é, de facto, a abordagem mais simples, mas não é suficientemente robusta.

### Criação estática versus mapeamento dinâmico

A abordagem do HagiCode é criar um `hagicode` usuário com UID 1000, que geralmente corresponde ao usuário padrão na maioria das máquinas host:

```dockerfile
RUN groupadd -o -g 1000 hagicode && \
    useradd -o -u 1000 -g 1000 -s /bin/bash -m hagicode && \
    mkdir -p /home/hagicode/.claude && \
    chown -R hagicode:hagicode /home/hagicode
```

Mas isso só resolve o usuário embutido dentro da imagem. E se o usuário host for UID 1001? Você ainda precisa oferecer suporte ao mapeamento dinâmico quando o contêiner for iniciado.

docker-entrypoint.sh contém a lógica principal:

```bash
if [ -n "$PUID" ] && [ -n "$PGID" ]; then
    if ! id hagicode >/dev/null 2>&1; then
        groupadd -g "$PGID" hagicode
        useradd -u "$PUID" -g "$PGID" -s /bin/bash -m hagicode
    fi
fi
```

A vantagem deste design é clara: **use o UID 1000 padrão no momento da construção da imagem e, em seguida, ajuste dinamicamente no tempo de execução através do `PUID` e `PGID` variáveis de ambiente**. Não importa qual UID o usuário host possua, a propriedade dos arquivos de configuração permanece correta.

## A filosofia de design de volumes persistentes

Cada ferramenta AI CLI tem seu próprio diretório de configuração preferencial, portanto, eles precisam ser mapeados um por um:

| Ferramenta CLI | Caminho no contêiner | Volume nomeado |
|---------|-----------|--------|
| Cláudio | `/home/hagicode/.claude` | `claude-data` |
| Códice | `/home/hagicode/.codex` | `codex-data` |
| Código aberto | `/home/hagicode/.config/opencode` | `opencode-config-data` |

Por que usar volumes nomeados em vez de montagens vinculadas? Três razões:

1. **Gerenciamento mais simples**: os volumes nomeados são gerenciados automaticamente pelo Docker, portanto, você não precisa criar diretórios de host manualmente.
2. **Isolamento de permissão**: O conteúdo inicial dos volumes é criado pelo usuário dentro do container, evitando conflitos de permissão com o host.
3. **Migração independente**: os volumes podem existir independentemente dos contêineres, para que os dados não sejam perdidos quando as imagens forem atualizadas.

docker-compose-builder-web gera automaticamente a configuração de volume correspondente:

```yaml
volumes:
  claude-data:
  codex-data:
  opencode-config-data:

services:
  hagicode:
    volumes:
      - claude-data:/home/hagicode/.claude
      - codex-data:/home/hagicode/.codex
      - opencode-config-data:/home/hagicode/.config/opencode
    user: "${PUID:-1000}:${PGID:-1000}"
```

Preste atenção ao `user` campo aqui: `PUID` e `PGID` são injetados por meio de variáveis de ambiente para garantir que os processos dentro do contêiner sejam executados com uma identidade que corresponda ao usuário do host. Esse detalhe é importante porque os problemas de permissão são difíceis de depurar quando aparecem.

## Gerenciamento de versões: versões integradas com substituições de tempo de execução

Fixar versões de imagem do Docker é essencial para a reprodutibilidade. Mas no desenvolvimento real, muitas vezes precisamos testar uma versão mais recente ou corrigir um bug com urgência. Se tivéssemos que reconstruir a imagem todas as vezes, o fluxo de trabalho seria muito ineficiente.

A estratégia do HagiCode é **versões fixas como padrão, com substituições de tempo de execução como mecanismo de extensão**. É um compromisso pragmático de engenharia entre estabilidade e flexibilidade.

Dockerfile.template fixa versões aqui:

```dockerfile
USER hagicode
WORKDIR /home/hagicode

# Configure the global npm install path
RUN mkdir -p /home/hagicode/.npm-global && \
    npm config set prefix '/home/hagicode/.npm-global'

# Install CLI tools using pinned versions
RUN npm install -g @anthropic-ai/claude-code@2.1.71 && \
    npm install -g @openai/codex@0.112.0 && \
    npm install -g opencode-ai@1.2.25 && \
    npm cache clean --force
```

docker-entrypoint.sh oferece suporte a substituições de tempo de execução:

```bash
install_cli_override_if_needed() {
    local package_name="$2"
    local override_version="$5"

    if [ -n "$override_version" ]; then
        gosu hagicode npm install -g "${package_name}@${override_version}"
    fi
}

# Example usage
install_cli_override_if_needed "" "@anthropic-ai/claude-code" "" "" "${CLAUDE_CODE_CLI_VERSION}"
```

Isso permite testar uma nova versão por meio de uma variável de ambiente sem reconstruir a imagem:

```bash
docker run -e CLAUDE_CODE_CLI_VERSION=2.2.0 myimage
```

Esse design é prático porque ninguém quer reconstruir uma imagem toda vez que testa um novo recurso.

## Injeção automática de configuração

Além de configurar as ferramentas CLI manualmente, alguns cenários exigem injeção automática de configuração. O exemplo mais típico é um token de API.

```bash
if [ -n "$ANTHROPIC_AUTH_TOKEN" ]; then
    mkdir -p /home/hagicode/.claude
    cat > /home/hagicode/.claude/settings.json <<EOF
{
  "env": {
    "ANTHROPIC_AUTH_TOKEN": "${ANTHROPIC_AUTH_TOKEN}"
  }
}
EOF
    chown -R hagicode:hagicode /home/hagicode/.claude
fi
```

Duas coisas são importantes aqui: **passar informações confidenciais por meio de variáveis de ambiente em vez de codificá-las na imagem** e certificar-se de que a propriedade dos arquivos de configuração esteja definida corretamente, caso contrário, as ferramentas CLI não serão capazes de lê-los.

## Melhores práticas e uma lista de verificação de armadilhas

### Problemas de incompatibilidade de permissão

Esta é a armadilha mais fácil de cair. O usuário host possui UID 1001, enquanto o contêiner usa 1000, portanto, os arquivos criados de um lado não podem ser acessados ​​do outro.

```bash
# Correct approach: make the container match the host user
docker run \
    -e PUID=$(id -u) \
    -e PGID=$(id -g) \
    myimage
```

Esse problema é muito comum e pode ser frustrante na primeira vez que você o encontra.

### A configuração desaparece após a reinicialização do contêiner

Se você fizer login novamente após cada reinicialização, verifique se esqueceu de montar um volume persistente:

```yaml
volumes:
  - claude-data:/home/hagicode/.claude
```

Nada é mais frustrante do que definir cuidadosamente uma configuração apenas para vê-la desaparecer.

### A maneira certa de atualizar versões

Não corra `npm install -g` diretamente dentro de um contêiner em execução. As abordagens corretas são:

1. Defina uma variável de ambiente para acionar a instalação de substituição.
2. Ou reconstrua a imagem.

```bash
# Option 1: runtime override
docker run -e CLAUDE_CODE_CLI_VERSION=2.2.0 myimage

# Option 2: rebuild the image
docker build -t myimage:v2 .
```

Há mais de uma estrada para Roma, mas algumas estradas são mais suaves que outras.

### Lista de verificação de reforço de segurança

- Passe tokens de API por meio de variáveis de ambiente em vez de gravá-los na imagem.
- Defina as permissões do arquivo de configuração para `600`.
- Sempre execute o aplicativo como um usuário não root.
- Atualize as versões da CLI regularmente para corrigir vulnerabilidades de segurança.

A segurança é sempre importante, mas o verdadeiro desafio é aplicá-la de forma consistente na prática.

## Estendendo o suporte para novas ferramentas CLI

Se você quiser oferecer suporte a uma nova ferramenta CLI no futuro, existem apenas três etapas:

1. **Dockerfile.template**: adicione a etapa de instalação.
2. **docker-entrypoint.sh**: adicione a lógica de substituição de versão.
3. **docker-compose-builder-web**: adicione o mapeamento de volume persistente.

Esse design baseado em modelo simplifica a extensão sem alterar a lógica central.

## Conclusão

A execução de ferramentas AI CLI em contêineres Docker envolve três desafios principais: **permissões de usuário, persistência de configuração e gerenciamento de versão**. Ao combinar usuários dedicados, isolamento de volume nomeado e substituições baseadas em variáveis ​​de ambiente, o projeto HagiCode construiu uma arquitetura de implantação que é segura e flexível.

Principais pontos de design:

- **Isolamento de usuário**: crie um usuário dedicado durante o estágio de criação da imagem, com suporte em tempo de execução para dinâmica `PUID`/`PGID` mapeamento.
- **Estratégia de persistência**: cada ferramenta CLI recebe seu próprio volume nomeado, portanto, as reinicializações não afetam a configuração.
- **Flexibilidade de versão**: padrões fixos garantem reprodutibilidade, enquanto substituições de tempo de execução oferecem espaço para testes.
- **Configuração automatizada**: a configuração sensível pode ser injetada automaticamente por meio de variáveis ​​de ambiente.

Esta solução está funcionando de forma estável no projeto HagiCode há algum tempo e espero que ofereça pontos de referência úteis para desenvolvedores com necessidades semelhantes.
