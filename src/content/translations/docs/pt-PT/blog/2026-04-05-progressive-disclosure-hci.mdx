## Divulgação Progressiva: Melhorando a Interação Humano-Computador em Produtos de IA com a Filosofia "Menos é Mais"

> No design de produtos de IA, a qualidade da entrada do usuário geralmente determina a qualidade do resultado. Este artigo compartilha uma abordagem de interação de “divulgação progressiva” que praticamos no projeto HagiCode. Através de orientação passo a passo, preenchimento inteligente e feedback instantâneo, ele transforma as contribuições breves e vagas dos usuários em propostas técnicas estruturadas, melhorando significativamente a eficiência da interação humano-computador.

## Plano de fundo

Qualquer pessoa que crie produtos de IA provavelmente já viu esta situação: um usuário abre seu aplicativo e digita com entusiasmo uma solicitação de uma linha, apenas para que a IA retorne algo completamente errado. Não é que a IA não seja inteligente o suficiente. O usuário simplesmente não forneceu informações suficientes. Ler mentes é difícil para qualquer um.

Esse problema tornou-se especialmente óbvio enquanto estávamos construindo o HagiCode. HagiCode é um assistente de codificação baseado em IA onde os usuários descrevem requisitos em linguagem natural para criar propostas e sessões técnicas. No uso real, descobrimos que a entrada do usuário geralmente apresenta estes problemas:

- **Qualidade de entrada inconsistente**: alguns usuários digitam apenas algumas palavras, como "otimizar login" ou "corrigir bug", sem o contexto necessário
- **Terminologia técnica inconsistente**: usuários diferentes usam termos diferentes para a mesma coisa; alguns dizem "frontend" enquanto outros dizem "FE"
- **Informações estruturadas ausentes**: não há histórico do projeto, escopo do repositório ou escopo de impacto, mesmo que esses detalhes sejam críticos
- **Problemas repetidos**: os mesmos tipos de solicitações aparecem repetidamente e sempre precisam ser explicados do zero

O resultado direto é previsível: a IA tem mais dificuldade em compreender a solicitação, a qualidade da proposta torna-se instável e a experiência do usuário é prejudicada. Os usuários pensam: “Esta IA não é muito boa”, enquanto nos sentimos injustamente culpados. Se você me der apenas uma frase, como vou adivinhar o que você realmente quer?

Na verdade, isso é compreensível. Até as pessoas precisam de tempo para se entenderem, e as máquinas não são exceção.

Para resolver esses problemas, tomamos uma decisão ousada: introduzir o princípio de design de “divulgação progressiva” para melhorar a interação humano-computador. As mudanças que isso trouxe foram provavelmente maiores do que você imagina. Para ser honesto, não esperávamos que fosse tão eficaz na época.

## O que é divulgação progressiva

"Divulgação progressiva" é um princípio de design da área de HCI (interação humano-computador). Sua ideia central é simples: **não mostre aos usuários todas as informações e opções de uma só vez. Em vez disso, revele apenas o que é necessário passo a passo, com base nas ações e necessidades do usuário**.

Este princípio é especialmente adequado para produtos de IA porque a interação com IA é naturalmente progressiva. O usuário fala um pouco, a IA entende um pouco, depois o usuário acrescenta mais e a IA entende mais. É muito semelhante à forma como as pessoas se comunicam: a compreensão geralmente se desenvolve gradualmente.

No cenário do HagiCode, aplicamos a divulgação progressiva em quatro áreas:

### 1. Mecanismo de otimização da descrição: deixe a IA ajudá-lo a dizer as coisas com mais clareza

Quando um usuário insere uma breve descrição, não a enviamos diretamente à IA para interpretação. Em vez disso, primeiro acionamos um fluxo de “otimização da descrição”. O núcleo deste fluxo é a “saída estruturada”: converter o texto livre do usuário em um formato padrão. É como enfiar pérolas soltas em um colar para que tudo fique mais fácil de entender.

A descrição otimizada deve incluir as seguintes seções padrão:

- **Antecedentes**: o histórico e o contexto do problema
- **Análise**: análise técnica e raciocínio
- **Solução**: as etapas de solução e implementação
- **Prática**: exemplos de códigos concretos e notas

Ao mesmo tempo, geramos automaticamente uma tabela Markdown mostrando informações como repositório de destino, caminhos e permissões de edição, facilitando as operações subsequentes de IA. Um diretório claro sempre torna as coisas mais fáceis de encontrar.

Aqui está a implementação real:

```csharp
// Core method in ProposalDescriptionMemoryService.cs
public async Task<string> OptimizeDescriptionAsync(
    string title,
    string description,
    string locale = "zh-CN",
    DescriptionOptimizationMemoryContext? memoryContext = null,
    CancellationToken cancellationToken = default)
{
    // Build query parameters
    var queryContext = BuildQueryContext(title, description);

    // Retrieve historical context
    var memoryContext = await RetrieveHistoricalContextAsync(queryContext, cancellationToken);

    // Generate a structured prompt
    var prompt = await BuildOptimizationPromptAsync(
        title,
        description,
        memoryContext,
        cancellationToken);

    // Call AI for optimization
    return await _aiService.CompleteAsync(prompt, cancellationToken);
}
```

A chave para esse fluxo é a “injeção de memória”. Injetamos contexto histórico, como convenções de projeto, casos semelhantes e padrões negativos, no prompt, permitindo que a IA faça referência a experiências anteriores durante a otimização. A experiência não deve ser desperdiçada.

**Notas:**
- Certifique-se de que a entrada atual tenha prioridade sobre a memória histórica, para que as informações do usuário especificadas explicitamente não sejam substituídas
- As referências do HagIndex devem ser tratadas como fontes factuais e não devem ser alteradas por casos históricos
- Sugestões de correção de baixa confiança não devem ser injetadas como fortes restrições

### 2. Capacidade de entrada de voz: falar é mais natural do que digitar

Além da entrada de texto, também oferecemos suporte para entrada de voz. Isto é especialmente útil para descrever requisitos complexos. Digitar uma solicitação técnica pode levar alguns minutos, enquanto dizê-la em voz alta pode levar apenas algumas dezenas de segundos.

O foco principal do design para entrada de voz é o "gerenciamento de estado". Os usuários devem saber claramente em que estado o sistema se encontra. Definimos os seguintes estados:

- **Inativo**: o sistema está pronto e a gravação pode começar
- **Aguardando upstream**: o sistema está se conectando ao serviço de back-end
- **Gravação**: a voz do usuário está sendo gravada
- **Processando**: a fala está sendo convertida em texto
- **Erro**: ocorreu um erro e precisa da atenção do usuário

O modelo de estado de front-end é mais ou menos assim:

```typescript
interface VoiceInputState {
  status: 'idle' | 'waiting-upstream' | 'recording' | 'processing' | 'error';
  duration: number;
  error?: string;
  deletedSet: Set<string>; // Fingerprint set of deleted results
}

// State transition when recording starts
const handleVoiceInputStart = async () => {
  // Enter waiting state first and show a loading animation
  setState({ status: 'waiting-upstream' });

  // Wait for backend readiness confirmation
  const isReady = await waitForBackendReady();
  if (!isReady) {
    setState({ status: 'error', error: 'Backend service is not ready' });
    return;
  }

  // Start recording
  setState({ status: 'recording', startTime: Date.now() });
};

// Handle recognition results
const handleRecognitionResult = (result: RecognitionResult) => {
  const fingerprint = normalizeFingerprint(result.text);

  // Check whether it has already been deleted
  if (state.deletedSet.has(fingerprint)) {
    return; // Skip deleted content
  }

  // Merge the result into the text box
  appendResult(result);
};
```

Há um detalhe importante aqui: usamos um “conjunto de impressões digitais” para gerenciar a sincronização de exclusões. Quando o reconhecimento de fala retorna vários resultados, os usuários podem excluir alguns deles. Armazenamos as impressões digitais do conteúdo excluído para que, se o mesmo conteúdo aparecer novamente mais tarde, ele seja ignorado automaticamente. É essencialmente uma forma de lembrar o que o usuário já rejeitou.

### 3. Sistema de gerenciamento imediato: externalize o "cérebro" da IA

HagiCode possui um sistema flexível de gerenciamento de prompts no qual todos os prompts são armazenados como arquivos:

```
prompts/
├── metadata/
│   ├── optimize-description.zh-CN.json
│   └── optimize-description.en-US.json
└── templates/
    ├── optimize-description.zh-CN.hbs
    └── optimize-description.en-US.hbs
```

Cada prompt consiste em duas partes:
- **Arquivo de metadados (`.json`)**: define informações como cenário de prompt, versão e parâmetros
- **Arquivo modelo (`.hbs`)**: o conteúdo real do prompt, escrito com a sintaxe do Handlebars

O formato do arquivo de metadados é semelhante a este:

```json
{
  "scenario": "optimize-description",
  "locale": "zh-CN",
  "version": "1.0.0",
  "syntax": "handlebars",
  "syntaxVersion": "1.0",
  "parameters": [
    {
      "name": "title",
      "type": "string",
      "required": true,
      "description": "Proposal title"
    },
    {
      "name": "description",
      "type": "string",
      "required": true,
      "description": "Original description"
    }
  ],
  "author": "HagiCode Team",
  "description": "Optimize the user's technical proposal description",
  "lastModified": "2026-04-05",
  "tags": ["optimization", "nlp"]
}
```

O arquivo de modelo usa a sintaxe do Handlebars e suporta injeção de parâmetros:

```handlebars
You are a technical proposal expert.

<task>
Generate a structured technical proposal description based on the following information.
</task>

<input>
<title>{{title}}</title>
<description>{{description}}</description>
{{#if memoryContext}}
<memory_context>
{{memoryContext}}
</memory_context>
{{/if}}
</input>

<output_format>
## Background
[Describe the problem background and context, including project information, repository scope, and so on]

## Analysis
[Provide the technical analysis and reasoning process, and explain why this change is needed]

## Solution
[Provide the solution and implementation steps, listing the key code locations]

## Practice
[Provide concrete code examples and notes]
</output_format>
```

Os benefícios deste design são claros:
- prompts podem ser controlados por versão, assim como o código
- vários idiomas são suportados e podem ser alternados automaticamente com base na preferência do usuário
- o design parametrizado permite que o contexto seja injetado dinamicamente
- a integridade pode ser validada na inicialização, evitando erros de tempo de execução

Se o conhecimento ficar apenas na cabeça de alguém, é fácil perdê-lo. Gravar de forma estruturada desde o início é muito mais seguro.

### 4. Assistente progressivo: divida tarefas complexas em pequenas etapas

Para tarefas complexas, como instalação e configuração pela primeira vez, usamos um design de assistente de várias etapas. Cada etapa solicita apenas as informações necessárias e fornece indicadores de progresso claros. Tarefas grandes tornam-se muito mais gerenciáveis ​​quando realizadas passo a passo.

O modelo de estado do assistente:

```typescript
interface WizardState {
  currentStep: number;           // 0-3, corresponding to 4 steps
  steps: WizardStep[];
  canGoNext: boolean;
  canGoBack: boolean;
  isLoading: boolean;
  error: string | null;
}

interface WizardStep {
  id: number;
  title: string;
  description: string;
  completed: boolean;
}

// Step navigation logic
const goToNextStep = () => {
  if (wizardState.currentStep < wizardState.steps.length - 1) {
    // Validate input for the current step
    if (validateCurrentStep()) {
      wizardState.currentStep++;
      wizardState.steps[wizardState.currentStep - 1].completed = true;
    }
  }
};

const goToPreviousStep = () => {
  if (wizardState.currentStep > 0) {
    wizardState.currentStep--;
  }
};
```

Cada etapa tem sua própria lógica de validação e as etapas concluídas recebem marcadores visuais claros. O cancelamento abre uma caixa de diálogo de confirmação para evitar que os usuários percam o progresso devido a ações acidentais.

## Conclusão

Olhando para trás, para a prática de divulgação progressiva do HagiCode, podemos resumir vários princípios fundamentais:

1. **Orientação passo a passo**: divida tarefas complexas em etapas menores e solicite apenas as informações necessárias em cada etapa
2. **Preenchimento inteligente**: use o contexto histórico e o conhecimento do projeto para preencher informações automaticamente
3. **Feedback instantâneo**: forneça feedback visual claro e dicas de status para cada ação
4. **Mecanismos de tolerância a falhas**: permitem aos usuários desfazer e redefinir para que erros não levem a perdas irreversíveis
5. **Diversidade de entrada**: suporte a vários métodos de entrada, como texto e voz

No HagiCode, o resultado prático desta abordagem foi claro: o comprimento médio da entrada do usuário aumentou de menos de 20 caracteres para descrições estruturadas de 200 a 300 caracteres, a qualidade das propostas geradas por IA melhorou significativamente e a satisfação do usuário aumentou junto com isso.

Isto não é surpreendente. Quanto mais informações os usuários fornecerem, mais precisamente a IA poderá entendê-los e melhores serão os resultados que ela poderá retornar. Nesse sentido, não é muito diferente da comunicação entre pessoas.

Se você também está construindo produtos relacionados à IA, espero que essas experiências ofereçam alguma inspiração útil. Lembre-se: os usuários não necessariamente se recusam a fornecer informações. Mais frequentemente, o produto ainda não fez as perguntas certas da maneira certa. O cerne da divulgação progressiva é encontrar o melhor momento e forma para essas questões.
