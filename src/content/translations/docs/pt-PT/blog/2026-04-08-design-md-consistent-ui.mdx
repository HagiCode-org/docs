## Design.md: uma solução para design de interface de usuário front-end consistente baseado em IA

> Na era do desenvolvimento de front-end assistido por IA, como você pode manter consistentes as UIs geradas por IA? Este artigo compartilha nossa experiência prática na construção de um site de galeria de design baseado em awesome-design-md, além de como criar um site estruturado. `design.md` para orientar a IA em direção ao design de UI padronizado.

## Plano de fundo

Qualquer pessoa que tenha usado IA para escrever código frontend provavelmente já teve uma experiência semelhante: peça à IA para gerar a mesma página várias vezes e cada resultado sairá em um estilo diferente. Às vezes os cantos são arredondados, às vezes são pontiagudos. Às vezes o espaçamento é de 8px e outras vezes chega a 16px. Até o mesmo botão pode parecer diferente em conversas diferentes.

Esta não é uma questão isolada. À medida que o desenvolvimento assistido por IA se torna mais comum, a **falta de consistência na UI frontend gerada por IA** tornou-se um problema generalizado. Diferentes assistentes de IA, diferentes prompts e até mesmo o mesmo assistente em diferentes conversas podem produzir designs de interface dramaticamente diferentes. Isso cria um enorme custo de manutenção durante a iteração do produto.

A causa raiz é realmente simples: não existe um **documento oficial de referência de design**. As folhas de estilo CSS tradicionais podem dizer aos desenvolvedores “como implementar” algo, mas não podem comunicar totalmente “por que foi projetado dessa maneira” ou “qual padrão de design deve ser usado em qual cenário”. Para a IA, uma descrição clara e estruturada é ainda mais importante para a compreensão das convenções de design.

Ao mesmo tempo, a comunidade de código aberto já oferece excelentes recursos. O [VoltAgent/awesome-design-md](https://github.com/VoltAgent/awesome-design-md) O projeto coleta documentação de sistemas de design de muitas empresas conhecidas. Cada diretório contém um `README.md`, um `DESIGN.md`e visualize o HTML. No entanto, tudo isso está espalhado pelo repositório upstream, dificultando a navegação e comparação rápida.

Então, podemos consolidar esses recursos em uma galeria de design fácil de navegar, ao mesmo tempo que destilamos uma estrutura estruturada? `design.md` para a IA usar?

A resposta é sim. A seguir, deixe-me explicar nossa abordagem.

## Análise

Antes de escrever o código, vamos analisar os desafios técnicos por trás desse problema.

### Gerenciamento de conteúdo de origem: como unificar recursos de design dispersos?

O repositório upstream awesome-design-md contém um grande número de documentos de design, mas precisávamos de uma maneira de trazê-los para nosso próprio projeto.

**Solução: use o submódulo git**

```
awesome-design-md-site
└── vendor/awesome-design-md          # Upstream resources (git submodule)
```

Isso nos traz vários benefícios:
- Controle de versão: podemos fixar uma versão upstream específica
- Compilações offline: não há necessidade de solicitar APIs externas durante a compilação
- Revisão de conteúdo: mudanças específicas são visíveis nos PRs

### Normalização de dados: como padronizar diferentes estruturas de documentos?

Diferentes empresas estruturam seus documentos de design de maneira diferente. Alguns estão faltando arquivos de visualização e alguns usam nomes inconsistentes. Precisamos normalizá-los durante o processo de construção.

**Solução: verifique e gere entradas normalizadas no momento da compilação**

O módulo principal é `awesomeDesignCatalog.ts`, responsável por:
1. Digitalizando o `vendor/awesome-design-md/design-md/*` diretório
2. Validando se cada entrada contém os arquivos necessários (`README.md`, `DESIGN.md`e pelo menos um arquivo de visualização)
3. Extraindo e renderizando conteúdo Markdown em HTML
4. Gerando dados de entrada normalizados

### Arquitetura de site estático: como fornecer pesquisa dinâmica enquanto permanece totalmente estático?

Por se tratar de uma galeria de design, a pesquisa é obrigatória. Mas o Astro é um gerador de sites estáticos, então como implementar a pesquisa em tempo real?

**Solução: ilha React + sincronização de parâmetros de consulta de URL**

```typescript
// src/components/gallery/SearchToolbar.tsx
```

A vantagem dessa abordagem é que ela mantém intacta a capacidade de implantação de um site estático, o que significa que ele pode ser implantado em qualquer serviço de hospedagem estática, ao mesmo tempo em que oferece uma experiência de filtragem instantânea.

### Documentação de design: como você ajuda a IA a compreender e seguir os padrões de design?

Este é o núcleo de toda a solução. Precisamos criar uma estrutura `design.md` que a IA pode compreender e aplicar.

**Solução: emprestar a estrutura do ClickHouse `DESIGN.md`**

[ClickHouse `DESIGN.md`](https://github.com/ClickHouse/ClickHouse/blob/master/DESIGN.md) é uma excelente referência. Inclui:
- Tema visual e atmosfera
- Paleta de cores e funções
- Regras de tipografia
- Estilos de componentes
- Princípios de layout
- Profundidade e elevação
- O que fazer e o que não fazer
- Comportamento Responsivo
- Guia de prompt do agente

Nossa abordagem é: **reutilizar a estrutura, reescrever o conteúdo**. Mantemos a estrutura de seções do ClickHouse `DESIGN.md`, mas substitua o conteúdo pelos tokens de design e convenções de componentes reais usados em nosso próprio projeto.

## Solução

Com base na análise acima, nossa solução consiste em quatro módulos principais.

### 1. Pipeline de ingestão de conteúdo

Esta é a base de todo o sistema, responsável por extrair e normalizar o conteúdo dos recursos upstream.

```typescript
// src/lib/content/awesomeDesignCatalog.ts

export async function scanSourceEntries(): Promise<DesignEntry[]> {
  const designDir = 'vendor/awesome-design-md/design-md';
  const entries: DesignEntry[] = [];

  for (const dir of await fs.readdir(designDir)) {
    const entryPath = path.join(designDir, dir);
    if (await isValidDesignEntry(entryPath)) {
      const entry = await normalizeDesignEntry(entryPath);
      entries.push(entry);
    }
  }

  return entries;
}

async function isValidDesignEntry(dir: string): Promise<boolean> {
  const requiredFiles = ['README.md', 'DESIGN.md'];
  for (const file of requiredFiles) {
    if (!(await fileExists(path.join(dir, file)))) {
      return false;
    }
  }
  return true;
}
```

### 2. Interface de navegação na galeria

A interface da galeria inclui três partes principais:

**Página inicial**: exibe uma grade de cartões com todas as entradas de design, e cada cartão inclui:
- Título e resumo da entrada do design
- Visualizar imagem, se disponível
- Destaque de pesquisa rápida

**Página de detalhes**: agrega todas as informações para uma única entrada de design:
- `README` documento
- `DESIGN` documento
- Visualização com alternância de tema claro/escuro
- Navegação para entradas adjacentes

**Navegação**: suporta retornar à galeria e navegar pelas entradas adjacentes

A galeria da página inicial usa um layout de cartão de alta densidade, achatando `design.md` entradas de diferentes fontes em uma estrutura visual unificada para que as equipes possam comparar rapidamente estilos de marca, padrões de botões e ritmo tipográfico.

Depois de abrir uma entrada específica, a página de detalhes coloca o resumo do design e a visualização ao vivo na mesma página, reduzindo o custo de alternar entre documentação, visualizações e código-fonte.

### 3. Funcionalidade de pesquisa

O recurso de pesquisa é baseado na filtragem do lado do cliente, com estado preservado por meio de parâmetros de consulta de URL:

```typescript
// src/components/gallery/SearchToolbar.tsx

function SearchToolbar({ entries }: { entries: DesignEntry[] }) {
  const [query, setQuery] = useState('');
  const [results, setResults] = useState(entries);

  useEffect(() => {
    const params = new URLSearchParams(window.location.search);
    const initialQuery = params.get('q') || '';
    setQuery(initialQuery);
    filterEntries(initialQuery);
  }, []);

  const filterEntries = (searchQuery: string) => {
    const filtered = entries.filter(entry =>
      entry.searchText.toLowerCase().includes(searchQuery.toLowerCase())
    );
    setResults(filtered);
  };

  const handleChange = (e: React.ChangeEvent<HTMLInputElement>) => {
    const value = e.target.value;
    setQuery(value);
    filterEntries(value);

    // Update the URL without triggering a page refresh
    const newUrl = value
      ? `${window.location.pathname}?q=${encodeURIComponent(value)}`
      : window.location.pathname;
    window.history.replaceState({}, '', newUrl);
  };

  return (
    <div className="search-toolbar">
      <input
        type="text"
        value={query}
        onChange={handleChange}
        placeholder="Search design entries..."
      />
      <span className="result-count">{results.length} results</span>
    </div>
  );
}
```

## Pratique

Agora que cobrimos a solução, como você realmente a implementa?

### Etapas de implementação

**Etapa 1: inicializar o submódulo**

```bash
# Add the upstream repository as a submodule
git submodule add https://github.com/VoltAgent/awesome-design-md.git vendor/awesome-design-md

# Initialize and update the submodule
git submodule update --init --recursive
```

**Etapa 2: Crie o pipeline de conteúdo**

Implementar `awesomeDesignCatalog.ts`, incluindo:
- Lógica de verificação e validação de arquivos
- Renderização Markdown usando o renderizador integrado do Astro
- Extração de dados de entrada

**Etapa 3: Crie a IU da galeria**

Use Ilhas Astro + React para criar:
- Layout da galeria da página inicial (grade de cartões)
- Componentes do cartão de design
- Barra de ferramentas de pesquisa
- Layout da página de detalhes

**Etapa 4: Escreva o documento de design**

Com base na estrutura do ClickHouse `DESIGN.md`, preencha os tokens de design reais do seu próprio projeto. Atualizar `README.md` e adicione um link para `design.md`.

### Notas

**Segurança**: a renderização de Markdown requer filtragem de HTML inseguro. Filtros de renderização integrados do Astro `script` tags por padrão, mas você ainda precisa ficar atento aos riscos de XSS.

**Desempenho**: um grande número de visualizações de iframe pode afetar o desempenho da primeira pintura. É recomendado usar `loading="lazy"` para carregar lentamente o conteúdo de visualização.

**Manutenção**: `design.md` precisa ficar em sincronia com a implementação do código. Recomenda-se adicionar verificações de CI para garantir que as variáveis ​​CSS permaneçam consistentes entre a documentação e o código.

**Acessibilidade**: certifique-se de que o contraste das cores atenda ao padrão WCAG AA (pelo menos 4,5:1).

### Guia de uso de IA

Depois de criar `design.md`, como você faz com que a IA realmente o use? Aqui estão algumas dicas práticas:

**Dica 1: faça referência explícita no prompt**

```
Please refer to the design.md file in the project root and use the design conventions defined there to implement the following components:
- Buttons: use the primary color with an 8px border radius
- Cards: use the elevation-2 shadow level
```

**Dica 2: exija que a IA faça referência a variáveis CSS específicas**

```
Implement a navigation bar with the following requirements:
- Use --color-bg-primary for the background color
- Use --color-border-subtle for borders
- Use --text-color-primary for text
```

**Dica 3: inclua `design.md` conteúdo no prompt do sistema**

Se sua ferramenta de IA oferecer suporte a prompts de sistema personalizados, você poderá adicionar o conteúdo principal de `design.md` diretamente para ele.

### Estratégia de teste

**Teste de pipeline de conteúdo**:
- Cenários de arquivos ausentes (ausentes `README.md` ou `DESIGN.md`)
- Cenários de erro de formato (falha na análise de Markdown)
- Cenários de diretório vazio

**Teste de recursos de pesquisa**:
- Tratamento de resultados vazios
- Caracteres especiais, como chinês e emoji
- Verificação de sincronização de URL

**Teste de componentes de IU**:
- Troca de tema claro/escuro
- Layout responsivo
- Visualizar estados de carregamento

### Fluxo de trabalho de implantação

```bash

git submodule update --remote

# 2. Rebuild the site

# 3. Deploy static assets
npm run deploy
```

Recomenda-se automatizar as atualizações de submódulos, além de construção e implantação, para que o CI possa ser acionado automaticamente sempre que o repositório upstream for atualizado.

## Conclusão

A inconsistência nas UIs geradas por IA que o HagiCode encontrou durante o desenvolvimento foi, em sua essência, causada pela falta de um documento de referência de design estruturado. Ao construir um site de galeria de design e criar um site padronizado `design.md`, resolvemos esse problema com sucesso.

O valor central desta solução reside em:
- **Recursos unificados**: consolidando a documentação dispersa do sistema de design
- **Padrões estruturados**: expressando convenções de design em um formato que a IA possa entender
- **Manutenção contínua**: manter o conteúdo atualizado através `git submodule`

Se você também estiver usando IA para desenvolvimento de front-end, vale a pena tentar essa abordagem. Criando uma estrutura `design.md` não apenas melhora a consistência do código gerado por IA, mas também ajuda sua equipe a manter padrões de design unificados internamente.
