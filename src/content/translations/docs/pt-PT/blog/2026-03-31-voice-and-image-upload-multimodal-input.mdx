## Digitar é mais lento que falar e falar é mais lento que uma captura de tela - Práticas de entrada multimodal para assistentes de codificação de IA

> Escrever código tem um limite de velocidade, não importa quão rápido você digite. Às vezes, algo que você poderia dizer em uma frase leva uma eternidade para ser digitado; às vezes, uma captura de tela explica tudo, mas você ainda precisa descrevê-la com uma pilha de texto. Este artigo fala sobre o que encontramos durante a construção [HagiCode](https://hagicode.com), desde reconhecimento de fala até upload de imagens. No final, queríamos apenas tornar um assistente de codificação de IA um pouco mais fácil de usar.

## Plano de fundo

Ao construir o HagiCode, notamos um problema - ou melhor, um problema que surgiu naturalmente quando as pessoas começaram a usá-lo intensamente: depender apenas da digitação pode ser cansativo.

Pense nisso: a interação entre os usuários e o Agente é um cenário central. Mas se toda troca exige digitação ininterrupta no teclado, a eficiência não é grande:

1. **A digitação é muito lenta**: para questões complicadas, como mensagens de erro ou problemas de interface do usuário, digitar tudo pode levar meio minuto, enquanto dizer em voz alta pode levar dez segundos. Essa lacuna é real.

2. **As imagens são mais diretas**: às vezes a UI gera um erro, às vezes você deseja comparar um rascunho de design e às vezes você precisa mostrar uma estrutura de código. “Uma imagem vale mais que mil palavras” pode ser um ditado antigo, mas ainda é válido. Deixar a IA “ver” diretamente o problema é muito mais claro do que descrevê-lo por muito tempo.

3. **A interação deve parecer natural**: Os assistentes modernos de IA devem oferecer suporte a texto, voz e imagens. Os usuários devem ser capazes de escolher qualquer método de entrada que pareça mais natural.

Portanto, decidimos adicionar reconhecimento de fala e suporte para upload de imagens ao HagiCode para tornar as interações dos agentes mais convenientes. Se os usuários conseguirem digitar um pouco menos, já é uma vitória.

## Sobre HagiCode

As soluções compartilhadas neste artigo vêm de nosso trabalho prático no [HagiCode](https://hagicode.com) projeto - ou, mais precisamente, das lições aprendidas enquanto tropeçamos em algumas armadilhas.

HagiCode é um projeto assistente de codificação de IA de código aberto com um objetivo simples: usar IA para melhorar a eficiência do desenvolvimento. À medida que continuávamos construindo, ficou claro que os usuários queriam fortemente informações multimodais. Às vezes, falar uma frase é mais rápido do que digitar um parágrafo longo e, às vezes, uma captura de tela é muito mais clara do que uma explicação longa.

Essas necessidades nos impulsionaram e foi assim que recursos como reconhecimento de voz e upload de imagens finalmente tomaram forma. Os usuários agora podem interagir com a IA da maneira mais natural disponível, e isso é bom.

## Análise

### Desafios técnicos no reconhecimento de fala

Ao construir o reconhecimento de fala, nos deparamos com um problema complicado: **a API WebSocket do navegador não suporta cabeçalhos HTTP personalizados**.

O serviço de reconhecimento de fala que escolhemos foi a API Doubao Speech Recognition da ByteDance. Infelizmente, esta API requer informações de autenticação, como `accessToken` e `secretKey` para ser passado através de cabeçalhos HTTP. Isso criou um conflito técnico imediato:

```javascript
// The browser WebSocket API does not support this approach
const ws = new WebSocket('wss://api.com/ws', {
  headers: {
    'Authorization': 'Bearer token'  // Not supported
  }
});
```

Basicamente tínhamos duas opções:

1. **Abordagem de parâmetro de consulta de URL**: coloque as informações de autenticação no URL
   - A vantagem é que é simples de implementar
   - A desvantagem é que as credenciais ficam expostas ao frontend, que é inseguro; algumas APIs também exigem autenticação baseada em cabeçalho

2. **Abordagem de proxy de back-end**: implemente um proxy WebSocket no back-end
   - A vantagem é que as credenciais permanecem armazenadas com segurança no backend e a solução é totalmente compatível com os requisitos da API
   - A desvantagem é que a implementação é um pouco mais complexa

No final, escolhemos a **abordagem de proxy de backend**. Segurança não é algo que você compromete.

### Requisitos funcionais para uploads de imagens

Nossos requisitos para uploads de imagens eram bastante simples:

1. **Vários métodos de upload**: clique para selecionar um arquivo, arraste e solte e cole da área de transferência
2. **Validação de arquivo**: restrições de tipo (PNG, JPG, WebP, GIF) e limites de tamanho (5-10 MB) são requisitos básicos
3. **Experiência do usuário**: faça upload do progresso, das visualizações e das mensagens de erro para que os usuários sempre saibam o que está acontecendo
4. **Segurança**: a validação do lado do servidor e a proteção contra uploads de arquivos maliciosos são essenciais

## Solução

### Reconhecimento de fala: arquitetura de proxy WebSocket

Projetamos uma arquitetura de três camadas para reconhecimento de fala e encontramos um caminho que funcionou:

```
Browser WebSocket
       |
       | ws://backend/api/voice/ws
       | (binary audio)
       v
Backend Proxy
       |
       | wss://openspeech.bytedance.com/ (with auth header)
       v
Doubao API
```

**Implementações de componentes principais**:

1. **Processador Front-end AudioWorklet**:

```javascript
class AudioProcessorWorklet extends AudioWorkletProcessor {
  process(inputs, outputs, parameters) {
    const input = inputs[0]?.[0];
    if (!input) return true;

    // Resample to 16 kHz (required by the Doubao API)
    const samples = this.resampleAudio(input, 48000, 16000);

    // Accumulate samples into 500 ms chunks
    this.accumulatedSamples.push(...samples);

    if (this.accumulatedSamples.length >= 8000) {
      // Convert to 16-bit PCM and send
      const pcm = this.floatToPcm16(this.accumulatedSamples);
      this.port.postMessage({ type: 'audioData', data: pcm.buffer }, [pcm.buffer]);
      this.accumulatedSamples = [];
    }
    return true;
  }
}
```

2. **Manipulador WebSocket de back-end** (C#):

```csharp
[HttpGet("ws")]
public async Task GetWebSocket()
{
    if (HttpContext.WebSockets.IsWebSocketRequest)
    {
        await _webSocketHandler.HandleAsync(HttpContext);
    }
}
```

3. ** Front-end `VoiceTextArea` componente**:

```tsx
export const VoiceTextArea = forwardRef<HTMLTextAreaElement, VoiceTextAreaProps>(
  ({ value, onChange, onTextRecognized, maxDuration }, ref) => {
```

### Uploads de imagens: componente de upload de vários métodos

Construímos um componente de upload de imagens completo com suporte para todos os três métodos de upload, cobrindo os cenários mais comuns que os usuários enfrentam.

**Recursos principais**:

1. **Três métodos de upload**:

```tsx
// Click to upload
const handleClick = () => fileInputRef.current?.click();

// Drag-and-drop upload
const handleDrop = (e: React.DragEvent) => {
  const file = e.dataTransfer.files?.[0];
  if (file) uploadFile(file);
};

// Clipboard paste
const handlePaste = (e: ClipboardEvent) => {
  for (const item of Array.from(e.clipboardData?.items || [])) {
    if (item.type.startsWith('image/')) {
      const file = item.getAsFile();
      if (file) uploadFile(file);
    }
  }
};
```

2. **Validação de front-end**:

```tsx
const validateFile = (file: File): { valid: boolean; error?: string } => {
  if (!acceptedTypes.includes(file.type)) {
    return { valid: false, error: 'Only PNG, JPG, JPEG, WebP, and GIF images are allowed' };
  }
  if (file.size > maxSize) {
    return { valid: false, error: `Maximum file size is ${(maxSize / 1024 / 1024).toFixed(1)}MB` };
  }
  return { valid: true };
};
```

3. **Manipulador de upload de back-end** (TypeScript):

```typescript
export const Route = createFileRoute('/api/upload')({
  server: {
    handlers: {
      POST: async ({ request }) => {
        const formData = await request.formData();
        const file = formData.get('file') as File;

        // Validation
        const validation = validateFile(file);
        if (!validation.isValid) {
          return Response.json({ error: validation.error }, { status: 400 });
        }

        // Save file
        const uuid = uuidv4();
        const filePath = join(uploadDir, `${uuid}${extension}`);
        await writeFile(filePath, buffer);

        return Response.json({ url: `/uploaded/${today}/${uuid}${extension}` });
      }
    }
  }
});
```

## Guia Prático

### Como usar o reconhecimento de fala

1. **Configure o serviço de reconhecimento de fala**:
   - Abra a página de configurações de reconhecimento de fala
   - Configurar o discurso Doubao `AppId` e `AccessToken`
   - Opcionalmente, configure hotwords para melhorar a precisão do reconhecimento de termos específicos do domínio

2. **Use-o na caixa de entrada**:
   - Clique no ícone do microfone no lado esquerdo da caixa de entrada
   - Comece a falar depois que a animação da forma de onda aparecer
   - Clique no ícone novamente para interromper a gravação
   - O texto reconhecido é inserido automaticamente na posição do cursor

### Como usar uploads de imagens

1. **Métodos de upload**:
   - Clique no botão de upload para escolher um arquivo
   - Arraste uma imagem diretamente para a área de upload
   - Usar `Ctrl+V` para colar uma captura de tela da área de transferência

2. **Formatos suportados**: PNG, JPG, JPEG, WebP, GIF
3. **Limite de tamanho**: 5 MB por padrão (configurável)

### Notas

1. **Reconhecimento de fala**:
   - A permissão do microfone é necessária
   - Use em um ambiente silencioso quando possível
   - A duração máxima de gravação suportada é de 300 segundos por padrão (configurável)

2. **Uploads de imagens**:
   - Apenas formatos de imagem comuns são suportados
   - Preste atenção aos limites de tamanho de arquivo
   - As imagens enviadas recebem automaticamente um URL de visualização

3. **Considerações de segurança**:
   - As credenciais de reconhecimento de fala são armazenadas no back-end
   - Os uploads de imagens passam por uma validação rigorosa do lado do servidor
   - HTTPS/WSS é recomendado em ambientes de produção

## Conclusão

Depois de adicionar reconhecimento de fala e upload de imagens, a experiência do usuário do HagiCode melhorou visivelmente. Os usuários agora podem interagir com a IA de uma forma mais natural – falando em vez de digitar e compartilhando capturas de tela em vez de descrever tudo manualmente. É como finalmente encontrar uma maneira mais confortável de se comunicar.

Ao construir esse recurso, encontramos o problema de que as APIs WebSocket do navegador não suportam cabeçalhos personalizados. No final, resolvemos isso com uma abordagem de proxy de back-end. Essa solução não apenas preservou a segurança, mas também lançou as bases para a integração posterior de outros serviços WebSocket autenticados.

O componente de upload de imagens também se beneficia do suporte a vários métodos de upload, permitindo que os usuários escolham o que for mais conveniente no momento. Clicar, arrastar ou colar todo o trabalho e cada caminho realiza o trabalho rapidamente.

“Digitar é mais lento do que falar, e falar é mais lento do que uma captura de tela” se encaixa muito bem no tema aqui. Se você estiver construindo um produto assistente de IA semelhante, espero que essas experiências ajudem, mesmo que apenas um pouco.
