## Construindo uma integração elegante de novos usuários em projetos React: prática driver.js do HagiCode

> Quando os usuários abrem seu produto pela primeira vez, eles realmente sabem por onde começar? Neste artigo, quero falar um pouco sobre como usamos driver.js para integração de novos usuários no projeto HagiCode. Considere isso um pequeno exemplo prático para iniciar a conversa.

## Plano de fundo

Você já se deparou com essa situação? Um novo usuário se inscreve no seu produto, abre a página e imediatamente parece perdido. Eles examinam, sem saber o que clicar ou o que fazer a seguir. Como desenvolvedores, muitas vezes presumimos que os usuários irão apenas “explorar por conta própria” porque, afinal, a curiosidade humana é ilimitada. Mas a realidade é diferente: a maioria dos usuários sai silenciosamente em poucos minutos se não conseguir encontrar o ponto de entrada certo, como se a história começasse de repente e terminasse com a mesma naturalidade.

A integração de novos usuários é uma forma importante de resolver esse problema, mas construí-la bem não é tão simples. Um bom sistema de integração precisa:

- Localize com precisão os elementos da página e destaque-os
- Apoie fluxos de integração em várias etapas
- Lembre-se da escolha do usuário (concluir ou pular)
- Evite afetar o desempenho da página e a interação normal
- Mantenha a estrutura do código clara e fácil de manter

Ao construir o HagiCode, enfrentamos o mesmo desafio. HagiCode é um assistente de codificação de IA e seu fluxo de trabalho principal é um fluxo de trabalho OpenSpec semelhante a este: "o usuário cria uma proposta -> a IA gera um plano -> o usuário o analisa -> a IA o executa." Para usuários que encontram esse conceito pela primeira vez, o fluxo de trabalho é completamente novo, portanto, eles precisam de uma integração sólida para começar rapidamente. Coisas novas sempre demoram um pouco para se acostumar.

A abordagem compartilhada neste artigo vem de nossa experiência prática no [HagiCode](https://hagicode.com) projeto. HagiCode é um assistente de codificação de IA baseado em Claude que ajuda os desenvolvedores a concluir tarefas de codificação com mais eficiência por meio do fluxo de trabalho OpenSpec. Você pode visualizar nosso código-fonte aberto em [GitHub](https://github.com/HagiCode-org/site).

## Por que escolhemos driver.js

Durante a fase de avaliação técnica, analisamos várias bibliotecas de integração convencionais. Cada um tinha seus próprios pontos fortes:

- **Intro.js**: poderoso, mas relativamente grande, e a personalização de estilo é um tanto complexa
- **Shepherd.js**: API bem projetada, mas um pouco "pesada" para nosso caso de uso
- **driver.js**: API leve, concisa e intuitiva e funciona bem no ecossistema React

No final, escolhemos driver.js. Não houve nenhuma razão especialmente dramática. A escolha se resumiu principalmente a estas considerações:

1. **Leve**: a biblioteca principal é pequena e não aumenta significativamente o tamanho do pacote
2. **API simples**: a configuração é clara e intuitiva, por isso é fácil de aprender
3. **Flexível**: suporta posicionamento, estilo e comportamento de interação personalizados
4. **Importação dinâmica**: pode ser carregado sob demanda sem afetar o desempenho da primeira tela

Com a seleção de tecnologia, raramente existe uma resposta universalmente melhor. Normalmente, existe apenas a opção que melhor se adapta.

## Implementação Técnica

### Configuração principal

driver.js possui um modelo de configuração muito intuitivo. Aqui está a configuração principal que usamos no projeto HagiCode:

```typescript
import { driver } from 'driver.js';
import 'driver.js/dist/driver.css';

const newConversationDriver = driver({
  allowClose: true,           // Allow users to close the guide
  animate: true,              // Enable animations
  overlayClickBehavior: 'close', // Close the guide when the overlay is clicked
  disableActiveInteraction: false, // Keep elements interactive
  showProgress: false,        // Do not show the progress bar (we manage progress ourselves)
  steps: guideSteps           // Array of guide steps
});
```

O raciocínio por trás dessas configurações é:

- `allowClose: true` - Respeite a escolha do usuário e não o force a terminar o guia
- `disableActiveInteraction: false` - Algumas etapas exigem ações reais do usuário, como digitar entradas, portanto a interação não pode ser desativada
- `overlayClickBehavior: 'close'` - Dê aos usuários uma maneira rápida de sair

### Gestão Estadual

Persistindo o estado de integração é fundamental. Não queremos reiniciar o guia toda vez que a página for atualizada, porque isso se torna irritante rapidamente. HagiCode usa `localStorage` para gerenciar o estado do guia:

```typescript
export type GuideState = 'pending' | 'dismissed' | 'completed';

export interface UserGuideState {
  session: GuideState;
  detailGuides: Record<string, GuideState>;
}

// Read state
export const getUserGuideState = (): UserGuideState => {
  const state = localStorage.getItem('userGuideState');
  return state ? JSON.parse(state) : { session: 'pending', detailGuides: {} };
};

// Update state
export const setUserGuideState = (state: UserGuideState) => {
  localStorage.setItem('userGuideState', JSON.stringify(state));
};
```

Definimos três estados:

- `pending`: o guia ainda está em andamento e o usuário não o concluiu ou o ignorou
- `dismissed`: O usuário fechou o guia proativamente
- `completed`: O usuário concluiu todas as etapas

Para a integração da página de detalhes da proposta, também oferecemos suporte a um rastreamento de estado mais refinado por meio do `detailGuides` mapa, porque uma proposta pode passar por vários estágios - elaboração, revisão e execução completa - e cada estágio precisa de orientação diferente. A situação está sempre mudando e a integração deve refletir isso.

### Seleção do Elemento Alvo

driver.js usa seletores CSS para localizar os elementos de destino. HagiCode segue uma convenção simples: use um código personalizado `data-guide` atributo para marcar alvos de integração:

```typescript
const steps = [
  {
    element: '[data-guide="launch"]',
    popover: {
      title: 'Start a New Conversation',
      description: 'Click here to create a new conversation session...'
    }
  }
];
```

Em componentes, fica assim:

```typescript
<button data-guide="launch" onClick={handleLaunch}>
  New Conversation
</button>
```

Os benefícios desta abordagem são:

- Evite conflitos com nomes de classes de estilo empresarial
- Semântica clara, para que você possa saber imediatamente se o elemento está relacionado à integração
- Mais fácil de gerenciar e manter de forma consistente

### Otimização de importação dinâmica

Como o recurso de integração só é necessário em cenários específicos, como na primeira visita de um usuário, usamos importações dinâmicas para otimizar o desempenho do carregamento inicial:

```typescript
const initNewUserGuide = async () => {
  // Dynamically import driver.js
  const { driver } = await import('driver.js');
  await import('driver.js/dist/driver.css');

  // Initialize the guide
  const newConversationDriver = driver({
    // ...configuration
  });

  newConversationDriver.drive();
};
```

Dessa forma, driver.js e sua folha de estilo só são carregados quando necessário e não afetam o desempenho da primeira tela. Poucas pessoas gostam de esperar por algo de que ainda nem precisam.

## Design de fluxo de integração

HagiCode implementa dois caminhos de integração que cobrem os principais cenários do usuário.

### Integração da sessão (10 etapas)

Este caminho de integração ajuda os usuários a completar todo o fluxo, desde a criação de uma conversa até o envio da primeira proposta completa:

1. **lançamento** - Inicie o guia e introduza o botão "Nova Conversa"
2. **compose** - Oriente o usuário a digitar uma solicitação na caixa de entrada
3. **enviar** - Oriente o usuário a clicar no botão enviar
4. **proposal-launch-readme** - Oriente o usuário a criar uma proposta README
5. **proposal-compose-readme** - Oriente o usuário a editar o conteúdo da solicitação README
6. **proposal-submit-readme** - Oriente o usuário a enviar a proposta README
7. **proposal-launch-agents** - Oriente o usuário a criar um `AGENTS.md` Proposta
8. **proposal-compose-agents** - Oriente o usuário a editar o `AGENTS.md` solicitação
9. **proposal-submit-agents** - Oriente o usuário a enviar o `AGENTS.md` Proposta
10. **proposal-wait** – Explique que a IA está processando e o usuário deve esperar um momento

A ideia por trás desse caminho é permitir que os usuários experimentem o fluxo de trabalho principal do HagiCode em primeira mão por meio de duas tarefas reais de criação de propostas, uma para `README` e um para `AGENTS.md`. Há uma grande diferença entre ouvir falar de um fluxo de trabalho e executá-lo você mesmo.

### Integração de detalhes da proposta (3 etapas)

Quando os usuários acessam a página de detalhes da Proposta, o HagiCode aciona o guia correspondente com base no estado atual da Proposta:

1. **Esboço** (fase de rascunho) - Oriente o usuário a revisar o plano gerado pela IA
2. **revisão** (etapa de revisão) - Orientar o usuário na execução do plano
3. **executionCompleted** (etapa concluída) – Orientar o usuário a arquivar o plano

A característica definidora deste guia é que ele é orientado pelo estado: ele decide dinamicamente qual etapa de integração mostrar com base no estado real da proposta. As coisas mudam e a integração deve mudar com elas.

### Mecanismo de nova tentativa de renderização de elemento

Em aplicativos React, o elemento de integração de destino pode ainda não ter concluído a renderização, por exemplo, porque os dados assíncronos ainda estão sendo carregados. Para lidar com isso, o HagiCode implementa um mecanismo de nova tentativa:

```typescript
const waitForElement = (selector: string, maxRetries = 10, interval = 100) => {
  let retries = 0;
  return new Promise<HTMLElement>((resolve, reject) => {
    const checkElement = () => {
      const element = document.querySelector(selector) as HTMLElement;
      if (element) {
        resolve(element);
      } else if (retries < maxRetries) {
        retries++;
        setTimeout(checkElement, interval);
      } else {
        reject(new Error(`Element not found: ${selector}`));
      }
    };
    checkElement();
  });
};
```

Chame esta função antes de inicializar o guia para garantir que o elemento de destino já exista. Às vezes vale a pena esperar um pouco mais.

## Resumo das melhores práticas

Com base na experiência prática do HagiCode, aqui estão algumas práticas recomendadas importantes:

### 1. A integração deve ser fácil de sair

Não force os usuários a concluir a integração. Alguns usuários são exploradores por natureza e preferem descobrir as coisas por conta própria. Forneça um botão "Pular" claro, lembre-se da escolha e não os interrompa novamente na próxima vez.

### 2. Mantenha o conteúdo de integração curto e nítido

Cada etapa de integração deve se concentrar em um único objetivo:

- **Título**: curto e claro, de preferência não mais do que algumas palavras
- **Descrição**: vá direto ao ponto e diga ao usuário o que é isso e por que é importante

Evite explicações prolixas. A atenção do usuário é limitada durante a integração e, quanto mais você disser, menor será a probabilidade de ele ler.

### 3. Mantenha os seletores estáveis

Use marcadores de elementos estáveis que não mudam com frequência. O costume `data-guide` atributo é uma boa escolha. Evite depender de nomes de classes ou estruturas DOM, pois são fáceis de alterar durante refatorações. O código muda o tempo todo, mas algumas âncoras devem permanecer estáveis ​​quando possível.

### 4. Teste sua integração

HagiCode inclui casos de teste completos para o recurso de integração:

```typescript
describe('NewUserConversationGuide', () => {
  it('should initialize guide state correctly', () => {
    const state = getUserGuideState();
    expect(state.session).toBe('pending');
  });

  it('should update guide state correctly', () => {
    setUserGuideState({ session: 'completed', detailGuides: {} });
    const state = getUserGuideState();
    expect(state.session).toBe('completed');
  });
});
```

Os testes ajudam a garantir que a refatoração não interrompa acidentalmente o comportamento de integração. Ninguém quer que uma pequena alteração no código danifique silenciosamente a funcionalidade que funcionava anteriormente.

### 5. Otimização de desempenho

- Use importações dinâmicas para carregar lentamente a biblioteca de integração
- Evite inicializar a lógica de integração depois que o usuário já tiver concluído o guia
- Considere o impacto das animações no desempenho e desative-as em dispositivos de baixo custo, se necessário

O desempenho, como muitas coisas na vida, merece um orçamento cuidadoso.

## Conclusão

A integração de novos usuários é uma parte importante para melhorar a experiência do usuário do produto. No projeto HagiCode, usamos driver.js para construir um sistema de integração completo que cobre todo o fluxo de trabalho, desde a criação da sessão até a execução da proposta.

Os pontos principais que esperamos compartilhar neste artigo são:

1. **As escolhas técnicas devem corresponder às necessidades reais**: driver.js não é a opção mais poderosa, mas é a mais adequada para nós
2. **A gestão do estado é crítica**: Use `localStorage` para persistir o estado de integração e evitar interromper repetidamente os usuários
3. **O design de integração deve permanecer focado**: cada etapa deve resolver um problema e nada mais
4. **A estrutura do código deve permanecer clara**: configuração de integração, gerenciamento de estado e lógica de UI separados para facilitar a manutenção

Se você estiver adicionando integração de novos usuários ao seu próprio projeto, espero que a experiência prática neste artigo ajude. Não há nada especialmente místico nesse tipo de tecnologia. Continue tentando, resumindo o que você aprendeu e as coisas gradualmente ficarão mais fáceis.
