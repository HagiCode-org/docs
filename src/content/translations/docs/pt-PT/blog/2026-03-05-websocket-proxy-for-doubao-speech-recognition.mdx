## Resolvendo desafios de autenticação WebSocket do navegador: um padrão de proxy prático para reconhecimento de fala Doubao

> A API WebSocket do navegador não oferece suporte a cabeçalhos HTTP personalizados, o que cria um desafio para serviços de reconhecimento de fala que exigem dados de autenticação em cabeçalhos. Este artigo compartilha como resolvemos esse problema no projeto HagiCode com um padrão de proxy de backend e como a abordagem evoluiu de experimentos de playground para uso em produção.


## Plano de fundo

Quando começamos a construir o reconhecimento de fala para o projeto HagiCode, escolhemos com confiança o serviço de reconhecimento de fala Doubao da ByteDance. O design inicial era simples: permitir que o frontend se conectasse diretamente ao serviço WebSocket da Doubao. Quão difícil isso poderia ser? Basta abrir uma conexão e enviar alguns dados, certo?

Então veio a surpresa: a API do Doubao exige que as informações de autenticação sejam passadas através de cabeçalhos HTTP, incluindo coisas como `accessToken` e `secretKey`. Isso imediatamente se tornou estranho, porque a API WebSocket do navegador simplesmente não oferece suporte à configuração de cabeçalhos personalizados.

Então, o que você faz quando o navegador não permite enviá-los?

Na época, pesamos duas opções:

1. Coloque as credenciais em parâmetros de consulta de URL - simples e direto
2. Adicione uma camada de proxy no back-end – mais trabalho à primeira vista

A primeira opção expõe credenciais diretamente no código frontend e no armazenamento local. Isso é realmente seguro? Eu não estava confortável com isso. Além disso, algumas APIs exigem verificação baseada em cabeçalho, portanto essa abordagem nem sequer é viável.

No final, escolhemos a segunda opção: implementar um proxy WebSocket no backend. Coincidentemente, esse padrão foi validado pela primeira vez em nosso ambiente de playground e somente depois de confirmarmos sua estabilidade é que o colocamos em produção. Afinal, ninguém quer que a produção funcione como um experimento de laboratório.

## Sobre HagiCode

HagiCode é um projeto de assistente de codificação de IA com suporte para interação de voz. Às vezes, esses obstáculos técnicos são frustrantes, mas também forçam você a aprender padrões que serão úteis mais tarde.

## Análise de Desafio Técnico

### Limitações do navegador WebSocket

A API WebSocket padrão parece maravilhosamente simples:

```javascript
const ws = new WebSocket('wss://example.com/ws');
```

Mas é exatamente nessa simplicidade que reside o problema - ela apenas passa parâmetros na URL e não pode definir cabeçalhos da mesma forma que uma solicitação HTTP:

```javascript
// This is not supported in the WebSocket API
const ws = new WebSocket('wss://example.com/ws', {
  headers: {
    'Authorization': 'Bearer token'
  }
});
```

E essa é a questão central. Para serviços como o reconhecimento de fala Doubao, que dependem de autenticação baseada em cabeçalho, essa limitação é um grande bloqueador.

Depois de aceitar essa restrição, a arquitetura terá que mudar.

### Decisões de projeto arquitetônico

Ao projetar a solução, comparamos cuidadosamente as compensações.

**Decisão 1: Escolhendo o padrão de proxy**

Comparamos duas abordagens:

| Opção | Prós | Contras | Decisão |
|------|------|------|------|
| WebSocket nativo | Encaminhamento leve, simples e direto | O gerenciamento de conexão deve ser feito manualmente | Escolhido |
| SinalR | Reconexão automática, digitação forte | Dependências extras e excessivamente complexas | Rejeitado |

No final das contas, escolhemos o WebSocket nativo. Para ser honesto, era a opção mais leve e mais adequada para o encaminhamento simples de fluxo binário bidirecional. Extrair o SignalR teria parecido um excesso de engenharia para este caso de uso e poderia adicionar latência extra.

**Decisão 2: Estratégia de gerenciamento de conexão**

Adotamos um modelo "uma conexão, uma sessão" - cada conexão WebSocket de front-end é mapeada para sua própria conexão de back-end Doubao independente.

Os benefícios são diretos:
- Simples de implementar e alinhado com o padrão de uso comum
- Mais fácil de depurar e solucionar problemas
- Bom isolamento de recursos, evitando interferência entre sessões

Simplificando, a solução direta às vezes é a melhor. A complexidade não torna automaticamente um design melhor.

**Decisão 3: Armazenar dados de autenticação**

As credenciais são armazenadas em arquivos de configuração de back-end (`appsettings.yml` ou variáveis de ambiente) e carregados por meio de injeção de dependência:

- Modelo de configuração simples que corresponde às convenções de back-end existentes
- Dados confidenciais nunca chegam ao frontend
- Suporta configuração de vários ambientes para desenvolvimento, teste e produção

Esse nível de separação é importante. Ninguém quer credenciais flutuando em lugares onde não deveriam estar.

### Projeto de fluxo de dados

O fluxo geral de dados é assim:

```
Frontend (browser)
  │
  │ ws://backend/api/voice/ws
  │ WebSocket (binary)
  ▼
Backend (proxy)
  │
  │ wss://openspeech.bytedance.com/
  │ (with authentication headers)
  ▼
Doubao API
```

O fluxo em si não é complicado:
1. O frontend se conecta ao proxy backend através do WebSocket
2. O proxy backend recebe dados de áudio e se conecta à API Doubao com cabeçalhos autenticados
3. A API Doubao retorna resultados de reconhecimento e o proxy os encaminha para o frontend
4. Todo o processo permanece totalmente assíncrono com streaming bidirecional

Uma vez que as responsabilidades são claramente divididas, o design torna-se bastante natural.

## Implementação de componentes principais

### 1. Configuração do terminal WebSocket

```csharp
app.Map("/ws", async context =>
{
    if (context.WebSockets.IsWebSocketRequest)
    {
        // Read configuration from query parameters
        var appId = context.Request.Query["appId"];
        var accessToken = context.Request.Query["accessToken"];

        // Validate required parameters
        if (string.IsNullOrEmpty(appId) || string.IsNullOrEmpty(accessToken))
        {
            context.Response.StatusCode = 400;
            return;
        }

        // Accept the WebSocket connection
        using var webSocket = await context.WebSockets.AcceptWebSocketAsync();

        // Message handling loop
        var buffer = new byte[4096];
        while (!webSocket.CloseStatus.HasValue)
        {
            var result = await webSocket.ReceiveAsync(buffer, CancellationToken.None);

            if (result.MessageType == WebSocketMessageType.Close)
            {
                await webSocket.CloseAsync(
                    result.CloseStatus.Value,
                    result.CloseStatusDescription,
                    CancellationToken.None);
                break;
            }

            // Process audio data
            await HandleAudioDataAsync(buffer, result.Count);
        }
    }
});
```

### 2. Gerenciamento de sessão

```csharp
public class DoubaoSessionManager : IDoubaoSessionManager
{
    private readonly ConcurrentDictionary<string, DoubaoSession> _sessions = new();

    public DoubaoSession CreateSession(string connectionId)
    {
        var session = new DoubaoSession(connectionId);
        _sessions[connectionId] = session;
        return session;
    }

    public async Task SendAudioAsync(string connectionId, byte[] audioData)
    {
        if (_sessions.TryGetValue(connectionId, out var session))
        {
            await session.SendAudioAsync(audioData);
        }
    }

    public void RemoveSession(string connectionId)
    {
        if (_sessions.TryRemove(connectionId, out var session))
        {
            session.Dispose();
        }
    }
}
```

Usando `ConcurrentDictionary` para gerenciamento de sessão significa que a segurança do thread é amplamente tratada para nós. Cada conexão recebida obtém sua própria sessão e a limpeza acontece automaticamente ao ser desconectada.

## Design de protocolo de mensagem

O frontend e o backend usam mensagens de texto JSON para controle, enquanto as mensagens binárias transportam dados de áudio.

Exemplo de mensagem de controle:

```json
{
    "type": "control",
    "messageId": "msg_123",
    "timestamp": "2026-03-03T10:00:00Z",
    "payload": {
        "command": "StartRecognition",
        "parameters": {
            "hotwordId": "hotword1",
            "boosting_table_id": "table123"
        }
    }
}
```

Exemplo de resultado de reconhecimento:

```json
{
    "type": "result",
    "timestamp": "2026-03-03T10:00:03Z",
    "payload": {
        "text": "Hello world",
        "confidence": 0.95,
        "duration": 1500,
        "isFinal": true,
        "utterances": [
            {
                "text": "Hello",
                "startTime": 0,
                "endTime": 800,
                "definite": true
            }
        ]
    }
}
```

Esse design separa os sinais de controle das cargas de áudio, o que torna o sistema mais fácil de raciocinar e implementar. Dividir responsabilidades de forma clara costuma ser a decisão certa.

## Integração frontend na prática

### Conexão WebSocket

```javascript
class DoubaoVoiceClient {
    constructor(config) {
        this.config = config;
        this.ws = null;
    }

    async connect() {
        const url = new URL(this.config.wsUrl);
        // Add query parameters
        Object.entries(this.config.params).forEach(([key, value]) => {
            url.searchParams.set(key, value);
        });

        this.ws = new WebSocket(url);

        return new Promise((resolve, reject) => {
            this.ws.onopen = () => {
                console.log('[DoubaoVoice] Connected');
                resolve();
            };

            this.ws.onmessage = (event) => {
                this._handleMessage(JSON.parse(event.data));
            };

            this.ws.onerror = reject;
        });
    }

    _handleMessage(message) {
        switch (message.type) {
            case 'status':
                this._handleStatus(message.payload);
                break;
            case 'result':
                this.onResult?.(message.payload);
                break;
            case 'error':
                console.error('[DoubaoVoice] Error:', message.payload);
                break;
        }
    }
}

// Usage example
const client = new DoubaoVoiceClient({
    wsUrl: 'ws://localhost:5000/ws',
    params: {
        appId: 'your-app-id',
        accessToken: 'your-access-token',
        sampleRate: 16000,
        bitsPerSample: 16,
        channels: 1
    }
});
```

### Captura e transmissão de áudio

Usando `AudioWorkletNode` para processamento de áudio oferece melhor desempenho:

```javascript
// audio-worklet.js
class AudioProcessorWorklet extends AudioWorkletProcessor {
    process(inputs, outputs, parameters) {
        const input = inputs[0]?.[0];
        if (!input) return true;

        // Convert to 16-bit PCM
        const pcm = new Int16Array(input.length);
        for (let i = 0; i < input.length; i++) {
            pcm[i] = Math.max(-32768, Math.min(32767, input[i] * 32767));
        }

        this.port.postMessage({
            type: 'audioData',
            data: pcm.buffer
        }, [pcm.buffer]);

        return true;
    }
}

registerProcessor('audio-processor', AudioProcessorWorklet);

// Main thread code
async function startAudioRecording() {
    const stream = await navigator.mediaDevices.getUserMedia({
        audio: {
            echoCancellation: true,
            noiseSuppression: true,
            autoGainControl: true,
            sampleRate: 48000
        }
    });

    const audioContext = new AudioContext();
    const audioSource = audioContext.createMediaStreamSource(stream);

    await audioContext.audioWorklet.addModule('/audio-worklet.js');
    const audioWorkletNode = new AudioWorkletNode(audioContext, 'audio-processor');

    audioWorkletNode.port.onmessage = (event) => {
        if (event.data.type === 'audioData' && ws?.readyState === WebSocket.OPEN) {
            ws.send(event.data.data); // Send binary data directly
        }
    };

    audioSource.connect(audioWorkletNode);
}
```

`AudioWorklet` tem um desempenho muito melhor do que `ScriptProcessorNode` e evita os problemas de interrupção de áudio que os caminhos de processamento mais antigos costumam apresentar.

## Notas e práticas recomendadas

### 1. Monitoramento de conexão

- Registrar regularmente o estado da sessão para rastrear o ciclo de vida completo da conexão
- Monitore o número e a duração dos segmentos de áudio para detectar conexões anormais
- Registre o status da conexão e o comportamento de reconexão para o serviço Doubao

Estas são práticas operacionais básicas, mas que fazem uma diferença real na produção.

### 2. Tratamento de erros

- Capture e registre todas as exceções do WebSocket
- Usar `IAsyncDisposable` para garantir que os recursos sejam limpos
- Implementar o desligamento elegante da conexão e o tratamento de tempo limite

Em suma, privilegie a robustez.

### 3. Requisitos de formato de áudio

- Taxa de amostragem: 16.000 Hz (recomendado) ou 8.000 Hz
- Profundidade de bits: 16 bits
- Canais: mono
- Codificação: PCM (bruto)

Se o formato estiver errado, o reconhecimento poderá falhar ou a qualidade poderá diminuir significativamente. Esses requisitos são importantes.

### 4. Considerações de segurança

- Mantenha credenciais confidenciais apenas na configuração de back-end
- Imponha limites de conexão para evitar o esgotamento de recursos
- Use HTTPS/WSS na produção

A segurança nunca é uma preocupação menor.

### 5. Otimização de desempenho

- Use operações assíncronas para evitar bloqueios
- Ajuste os tamanhos dos buffers conforme necessário (padrão: 4.096 bytes)
- Considere estratégias de pooling e reutilização de conexões

Aplique essas otimizações onde elas fizerem sentido para sua carga de trabalho.

## Recomendações de implantação

1. **Implantação do Docker**: Empacote o serviço de proxy como um contêiner para facilitar o dimensionamento e o gerenciamento
2. **Balanceamento de carga**: use Nginx ou Envoy como proxy reverso para tráfego WebSocket
3. **Verificações de integridade**: implemente verificações baseadas em pulsação para monitorar a disponibilidade do serviço
4. **Agregação de logs**: envie logs para um sistema de log centralizado, como ELK ou Loki

A implantação pode ser simples ou complexa dependendo da equipe e do ambiente, portanto, ajuste de acordo.

## Resumo

O padrão de proxy WebSocket resolve o problema principal de que a API WebSocket do navegador não oferece suporte a cabeçalhos personalizados. No projeto HagiCode, esse padrão se mostrou viável e estável à medida que passou da validação do playground para a implantação na produção.

Principais conclusões:
- Um proxy backend pode passar informações de autenticação com segurança
- Native WebSocket é leve e eficiente para cenários simples
- O modelo “uma conexão, uma sessão” simplifica a implementação e a depuração
- O protocolo frontend-backend deve separar os sinais de controle dos dados de áudio

Se você estiver criando um recurso que também depende da autenticação WebSocket, espero que esse padrão forneça um ponto de partida útil.

Se você tiver dúvidas, sinta-se à vontade para discuti-las conosco. O progresso técnico acontece mais rápido quando as pessoas comparam notas.
