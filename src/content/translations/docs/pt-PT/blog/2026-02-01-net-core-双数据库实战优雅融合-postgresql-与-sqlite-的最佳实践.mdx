## Banco de dados duplo .NET Core na prática: deixe o PostgreSQL e o SQLite coexistirem pacificamente

> Ao construir aplicações modernas, muitas vezes enfrentamos esta compensação: os ambientes de desenvolvimento querem algo leve e conveniente, enquanto os ambientes de produção exigem alta simultaneidade e alta disponibilidade. Este artigo compartilha como oferecer suporte elegante ao PostgreSQL e ao SQLite em um projeto .NET Core e implementar as práticas recomendadas de "SQLite para desenvolvimento, PostgreSQL para produção".


## Plano de fundo

No desenvolvimento de software, as diferenças entre ambientes sempre foram um dos problemas mais difíceis para as equipes de engenharia. Tomemos como exemplo a plataforma **HagiCode** que estamos construindo: é um sistema de desenvolvimento assistido por IA baseado em ASP.NET Core 10 e React, com Orleans integrado internamente para gerenciamento de estado distribuído. A pilha é moderna e bastante sofisticada.

No início do projeto, nos deparamos com um problema clássico de engenharia: os desenvolvedores queriam que o ambiente local funcionasse imediatamente, sem a necessidade de instalar e configurar um banco de dados PostgreSQL pesado. Mas na produção, precisávamos lidar com gravações de alta simultaneidade e consultas JSON complexas, e é exatamente aí que o SQLite leve começa a mostrar seus limites.

Como podemos manter uma única base de código e ao mesmo tempo permitir que o aplicativo se beneficie da portabilidade do SQLite, como um aplicativo de desktop, e também aproveitar o poderoso desempenho do PostgreSQL como um serviço de nível empresarial? Essa é a questão central que este artigo explora.

## Tópico Central 1: Projeto de Arquitetura e Abstração Unificada

Para oferecer suporte a dois bancos de dados no .NET Core, a ideia principal é depender de abstrações em vez de implementações concretas. Precisamos separar a seleção do banco de dados do código comercial e deixar a camada de configuração decidir.

### Abordagem de projeto

1.  **Interface unificada**: toda lógica de negócios deve depender do `DbContext` classe base ou interfaces personalizadas, em vez de um específico `PostgreSqlDbContext`.
2.  **Orientado pela configuração**: use itens de configuração em `appsettings.json` para decidir dinamicamente qual provedor de banco de dados carregar na inicialização do aplicativo.
3.  **Isolamento de recursos**: adicione lógica de adaptação para recursos específicos do PostgreSQL, como JSONB, para que o aplicativo ainda possa ser degradado normalmente no SQLite.

### Implementação de código: configuração de contexto dinâmico

No ASP.NET Core `Program.cs`, não devemos codificar `UseNpgsql` ou `UseSqlite`. Em vez disso, devemos ler a configuração e decidir dinamicamente.

Primeiro, defina a classe de configuração:

```csharp
public class DatabaseSettings
{
    public const string SectionName = "Database";
    
    // Database type: PostgreSQL or SQLite
    public string DbType { get; set; } = "PostgreSQL"; 
    
    // Connection string
    public string ConnectionString { get; set; } = string.Empty;
}
```

Em seguida, registre o serviço em `Program.cs` com base na configuração:

```csharp
// Read configuration
var databaseSettings = builder.Configuration.GetSection(DatabaseSettings.SectionName).Get<DatabaseSettings>();

// Register DbContext
builder.Services.AddDbContext<ApplicationDbContext>(options =>
{
    if (databaseSettings?.DbType?.ToLower() == "sqlite")
    {
        // SQLite configuration
        options.UseSqlite(databaseSettings.ConnectionString);
        
        // Handling SQLite's concurrent write limitations
        // Note: in production, enabling WAL mode is recommended to improve concurrency performance
    }
    else
    {
        // PostgreSQL configuration (default)
        options.UseNpgsql(databaseSettings.ConnectionString, npgsqlOptions =>
        {
            // Enable JSONB support, which is very useful when handling AI conversation records
            npgsqlOptions.UseJsonNet(); 
        });
        
        // Configure connection pool retry policy
        options.EnableRetryOnFailure(3);
    }
});
```

## Tópico Central 2: Lidando com Diferenças e Estratégia de Migração

Embora PostgreSQL e SQLite suportem o padrão SQL, eles diferem significativamente em recursos e comportamento específicos. Se essas diferenças não forem tratadas com cuidado, você poderá facilmente acabar na situação embaraçosa em que tudo funciona localmente, mas falha após a implantação.

### 1. Tratamento de tipos JSON

No HagiCode, precisamos armazenar uma grande quantidade de prompts e metadados de IA, que geralmente envolvem colunas JSON.
- **PostgreSQL**: Possui um recurso nativo `JSONB` digite com excelente desempenho de consulta.
- **SQLite**: não possui um tipo JSON nativo (as versões mais recentes incluem a extensão JSON1, mas o mapeamento de objetos ainda é diferente), portanto os dados geralmente são armazenados como TEXT.

**Solução**:
No mapeamento de entidade do EF Core, configuramos como um tipo conversível.

```csharp
protected override void OnModelCreating(ModelBuilder modelBuilder)
{
    base.OnModelCreating(modelBuilder);

    // Configure entity
    modelBuilder.Entity<PromptTemplate>(entity => 
    {
        entity.Property(e => e.Metadata)
              .HasColumnType("jsonb") // PG uses jsonb
              .HasConversion(
                  v => JsonSerializer.Serialize(v, (JsonSerializerOptions)null),
                  v => JsonSerializer.Deserialize<Dictionary<string, object>>(v, (JsonSerializerOptions)null)
              ); 
    });
}
```

Quando SQLite é usado, `HasColumnType("jsonb")` pode ser ignorado ou disparar um aviso. No entanto, porque `HasConversion` estiver configurado, os dados ainda serão serializados e desserializados corretamente como strings armazenadas em um campo TEXT, garantindo compatibilidade.

### 2. Separando estratégias de migração

Nunca tente fazer um conjunto de scripts de migração funcionar para PostgreSQL e SQLite ao mesmo tempo. Diferenças nas estratégias de geração de chave primária, sintaxe de índice e outros detalhes do banco de dados inevitavelmente causarão falhas.

**Prática recomendada**:
Mantenha duas filiais ou projetos de migração. No fluxo de trabalho de desenvolvimento do HagiCode, é assim que lidamos com isso:

1.  **Estágio de desenvolvimento**: Trabalhar principalmente com SQLite. Usar `Add-Migration Init_Sqlite -OutputDir Migrations/Sqlite`.
2.  **Estágio de adaptação**: Após desenvolver um recurso, mude a string de conexão para PostgreSQL e execute `Add-Migration Init_Postgres -OutputDir Migrations/Postgres`.
3.  **Scripts de automação**: escreva um script simples do PowerShell ou Bash para aplicar automaticamente a migração correta com base na variável de ambiente atual.

```bash
# Pseudocode for simple deployment logic
if [ "$DATABASE_PROVIDER" = "PostgreSQL" ]; then
    dotnet ef database update --project Migrations.Postgres
else
    dotnet ef database update --project Migrations.Sqlite
fi
```

## Tópico principal 3: Lições aprendidas com HagiCode em produção

Ao refatorar o **HagiCode** de um modelo de banco de dados único para suporte a banco de dados duplo, encontramos algumas armadilhas e reunimos algumas lições importantes que podem ajudá-lo a evitar os mesmos erros.

### 1. Diferenças em simultaneidade e transações

O PostgreSQL usa uma arquitetura servidor-cliente e suporta gravações de alta simultaneidade com poderosos níveis de isolamento de transações. SQLite usa bloqueio de arquivo, portanto, as operações de gravação bloqueiam todo o arquivo do banco de dados, a menos que o modo WAL esteja habilitado.

**Recomendação**:
Ao escrever lógica de negócios que envolve gravações frequentes, como salvar em tempo real o estado de edição de um usuário, você deve levar em consideração o modelo de bloqueio do SQLite. Ao projetar o módulo de colaboração OpenSpec no **HagiCode**, introduzimos um mecanismo de "mesclagem antes de gravação" para reduzir a frequência de gravações diretas no banco de dados, permitindo-nos manter um bom desempenho em ambos os bancos de dados.

### 2. Gerenciamento do ciclo de vida de strings de conexão

Estabelecer uma conexão PostgreSQL é relativamente caro e depende do pool de conexões. As conexões SQLite são muito leves, mas se não forem liberadas imediatamente, os bloqueios de arquivos podem causar o tempo limite das operações posteriores.

Em `Program.cs`, podemos ajustar o comportamento de cada banco de dados:

```csharp
if (databaseSettings?.DbType?.ToLower() == "sqlite")
{
    // SQLite: keeping connections open can improve performance, but watch out for file locks
    options.UseSqlite(connectionString, sqliteOptions =>
    {
        // Set command timeout
        sqliteOptions.CommandTimeout(30);
    });
}
else
{
    // PG: make full use of connection pooling
    options.UseNpgsql(connectionString, npgsqlOptions =>
    {
        npgsqlOptions.MaxBatchSize(100);
        npgsqlOptions.CommandTimeout(30);
    });
}
```

### 3. A importância da cobertura de testes

Muitos desenvolvedores, incluindo alguns dos primeiros membros de nossa equipe, tendem a cometer um erro comum: executar testes unitários apenas no ambiente de desenvolvimento, que geralmente é SQLite.

No pipeline CI/CD do HagiCode, aplicamos uma etapa GitHub Actions para garantir que cada solicitação pull execute testes de integração PostgreSQL.

```yaml
# Example snippet from .github/workflows/test.yml
- name: Run Integration Tests (PostgreSQL)
  run: |
    docker-compose up -d db_postgres
    dotnet test --filter "Category=Integration"
```

Isso nos ajudou a detectar inúmeros bugs relacionados às diferenças de sintaxe SQL e à diferenciação de maiúsculas e minúsculas.

## Conclusão

Ao introduzir uma camada de abstração e injeção de dependência orientada por configuração, implementamos com sucesso uma configuração PostgreSQL e SQLite de via dupla no projeto **HagiCode**. Isso não apenas reduziu bastante a barreira de integração para novos desenvolvedores, eliminando a necessidade de instalar o PostgreSQL, mas também forneceu fortes garantias de desempenho para produção.

Para recapitular os pontos principais:
1.  **Abstração primeiro**: O código comercial não deve depender de implementações concretas de banco de dados.
2.  **Configuração separada**: Use diferentes `appsettings.json` arquivos para desenvolvimento e produção.
3.  **Migrações separadas**: não tente fazer com que um conjunto de migração funcione em todos os lugares.
4.  **Degradação de recursos**: Priorize a compatibilidade no SQLite e o desempenho no PostgreSQL.

Este padrão de arquitetura não é adequado apenas para HagiCode, mas para qualquer projeto .NET que precise encontrar um equilíbrio entre desenvolvimento leve e produção pesada.
