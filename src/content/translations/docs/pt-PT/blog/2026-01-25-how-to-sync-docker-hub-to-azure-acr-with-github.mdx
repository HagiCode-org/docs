## Automatizando a sincronização de imagens do Docker Hub para o Azure ACR

> Este artigo explica como usar GitHub Actions e a ferramenta de sincronização de imagens para automatizar a sincronização de imagens do Docker Hub para o Azure Container Registry, abordando o acesso lento ao Docker Hub na China e em algumas regiões do Azure, melhorando ao mesmo tempo a disponibilidade de imagens e a eficiência de implementação em ambientes Azure.

## Antecedentes/Introdução

O projeto HagiCode usa imagens Docker como seus principais componentes de tempo de execução, com as imagens primárias hospedadas no Docker Hub. À medida que o projeto cresceu e as necessidades de implantação do Azure aumentaram, encontramos os seguintes pontos problemáticos:

- Velocidades lentas de extração de imagens, com acesso ao Docker Hub restrito na China e em algumas regiões do Azure
- Uma única fonte de imagem cria um único ponto de risco de falha
- O Registro de Contêiner do Azure oferece melhor desempenho e integração de rede quando executado no Azure

Para resolver estes problemas, precisávamos de um mecanismo automatizado de sincronização de imagens para sincronizar regularmente imagens do Docker Hub para o Azure ACR, garantindo que os utilizadores pudessem obter velocidades de extração de imagens mais rápidas e maior disponibilidade em ambientes Azure.

## Comparação de soluções técnicas

Ao definir a solução, comparamos diversas opções técnicas:

### 1. sincronizador de imagens (escolha final)
- Sincronização incremental: sincroniza apenas camadas de imagem alteradas, reduzindo significativamente a transferência de rede
- Retomar suporte: a sincronização pode continuar após interrupções na rede
- Controle de simultaneidade: threads de trabalho configuráveis melhoram a eficiência da sincronização de imagens grandes
- Tratamento robusto de erros: mecanismo de repetição integrado para falhas (3 tentativas por padrão)
- Implantação leve: um único binário sem dependências
- Suporte a vários registros: compatível com Docker Hub, Azure ACR, Harbor e muito mais

### 2. CLI do Docker
- Sem sincronização incremental: o conteúdo completo da imagem deve ser extraído sempre
- Menor eficiência: grande volume de transferência de rede e maior tempo de execução
- Simples de usar: baseado em informações familiares `docker pull` e `docker push` comandos

### 3. CLI do Azure
- Maior complexidade: a autenticação CLI do Azure deve ser configurada
- Funcionalidade limitada: `az acr import` tem um escopo relativamente restrito
- Integração nativa: integra-se bem aos serviços do Azure

## Decisões de projeto de arquitetura

### Decisão 1: Definir frequência de sincronização diariamente às 00:00 UTC
- Equilibra a atualização da imagem e o consumo de recursos
- Evita horários de pico comercial e reduz o impacto em outras operações
- As imagens do Docker Hub normalmente são atualizadas após compilações diárias

### Decisão 2: sincronizar todas as tags de imagem
- Mantém consistência total com Docker Hub
- Oferece aos usuários opções de versão flexíveis
- Simplifica a lógica de sincronização e evita regras complexas de filtragem de tags

### Decisão 3: Use GitHub Secrets para armazenar credenciais
- Suporte nativo ao GitHub Actions com forte segurança
- Configuração simples e fácil manutenção
- Suporta controle de acesso em nível de repositório

## Avaliação e Mitigação de Riscos

### Risco 1: vazamento de credenciais do Azure ACR
- Armazene credenciais criptografadas em GitHub Secrets
- Alterne as senhas ACR regularmente
- Restringir a conta ACR para permissões somente push
- Monitore logs de acesso ACR

### Risco 2: Falhas de sincronização causando inconsistência de imagem
- image-syncer inclui um mecanismo de sincronização incremental integrado
- Novas tentativas automáticas em caso de falha (3 por padrão)
- Logs de erros detalhados e notificações de falhas
- Retomar capacidade

### Risco 3: Consumo excessivo de recursos
- A sincronização incremental reduz a transferência de rede
- Simultaneidade configurável (`10` na configuração atual)
- Monitore o número e o tamanho das imagens sincronizadas
- Execute a sincronização fora dos horários de pico

## Solução Principal

Usamos uma solução automatizada baseada em GitHub Actions + image-syncer para sincronizar imagens do Docker Hub para o Azure ACR.

## Etapas de implementação

### 1. Preparação
- Crie ou confirme um Registro de Contêiner do Azure no Portal do Azure
- Crie credenciais de acesso ACR (nome de usuário e senha)
- Confirme as permissões de acesso para o repositório de imagens Docker Hub

### 2. Configurar segredos do GitHub
Adicione os seguintes segredos nas configurações do repositório GitHub:
- AZURE_ACR_USERNAME: nome de usuário do Azure ACR
- AZURE_ACR_PASSWORD: senha do Azure ACR

### 3. Crie o fluxo de trabalho do GitHub Actions
Configure o fluxo de trabalho em `.github/workflows/sync-docker-acr.yml`:
- Gatilho programado: todos os dias às 00:00 UTC
- Gatilho manual: suporta `workflow_dispatch`
- Gatilho adicional: é executado em pushes para o `publish` branch para sincronização rápida

### 4. Fluxo de execução do fluxo de trabalho

| Passo | Participante | Ação | Descrição |
| --- | --- | --- | --- |
| 1 | Ações do GitHub | Acione o fluxo de trabalho | Acionado por agendamento, execução manual ou push para o `publish` filial |
| 2 | Ações do GitHub → sincronizador de imagens | Baixe e execute a ferramenta de sincronização | Entra na fase de sincronização real |
| 3 | sincronizador de imagens → Docker Hub | Buscar manifestos de imagens e lista de tags | Lê metadados do repositório de origem |
| 4 | sincronizador de imagens → Azure ACR | Busque dados de imagem existentes do repositório de destino | Determina o estado atual do lado alvo |
| 5 | sincronizador de imagens | Compare as diferenças de origem e destino | Identifica as camadas da imagem que precisam de sincronização |
| 6 | sincronizador de imagens → Docker Hub | Extrair camadas de imagem alteradas | Transfere apenas o conteúdo que precisa ser atualizado |
| 7 | sincronizador de imagens → Azure ACR | Enviar camadas de imagem alteradas | Conclui a sincronização incremental |
| 8 | sincronizador de imagens → Ações do GitHub | Estatísticas de sincronização de retorno | Inclui resultados, diferenças e informações de erro |
| 9 | Ações do GitHub | Registrar logs e fazer upload de artefatos | Suporta auditoria e solução de problemas posteriores |

## Implementação do fluxo de trabalho de ações do GitHub

Abaixo está a configuração real do fluxo de trabalho em uso (`.github/workflows/sync-docker-acr.yml`):

```yaml
name: Sync Docker Image to Azure ACR

on:
  schedule:
    - cron: "0 0 * * *" # Every day at 00:00 UTC
  workflow_dispatch: # Manual trigger
  push:
    branches: [publish]

permissions:
  contents: read

jobs:
  sync:
    runs-on: ubuntu-latest

    steps:
      - name: Checkout code
        uses: actions/checkout@v4

      - name: Download image-syncer
        run: |
          # Download the image-syncer binary
          wget https://github.com/AliyunContainerService/image-syncer/releases/download/v1.5.5/image-syncer-v1.5.5-linux-amd64.tar.gz
          tar -zxvf image-syncer-v1.5.5-linux-amd64.tar.gz
          chmod +x image-syncer

      - name: Create auth config
        run: |
          # Generate the authentication config file (YAML format)
          cat > auth.yaml <<EOF
          hagicode.azurecr.io:
            username: "${{ secrets.AZURE_ACR_USERNAME }}"
            password: "${{ secrets.AZURE_ACR_PASSWORD }}"
          EOF

      - name: Create images config
        run: |
          # Generate the image sync config file (YAML format)
          cat > images.yaml <<EOF
          docker.io/newbe36524/hagicode: hagicode.azurecr.io/hagicode
          EOF

      - name: Run image-syncer
        run: |
          # Run synchronization (using the newer --auth and --images parameters)
          ./image-syncer --auth=./auth.yaml --images=./images.yaml --proc=10 --retries=3

      - name: Upload logs
        if: always()
        uses: actions/upload-artifact@v4
        with:
          name: sync-logs
          path: image-syncer-*.log
          retention-days: 7
```

## Detalhes de configuração

### 1. Condições de gatilho
- Gatilho programado: `cron: "0 0 * * *"` funciona todos os dias às 00:00 UTC
- Gatilho manual: `workflow_dispatch` permite que os usuários o executem manualmente a partir da UI do GitHub
- Empurre o gatilho: `push: branches: [publish]` dispara em pushes para o branch de lançamento para sincronização rápida

### 2. Configuração de autenticação (`auth.yaml`)
```yaml
hagicode.azurecr.io:
  username: "${{ secrets.AZURE_ACR_USERNAME }}"
  password: "${{ secrets.AZURE_ACR_PASSWORD }}"
```

### 3. Configuração de sincronização de imagem
```yaml
docker.io/newbe36524/hagicode: hagicode.azurecr.io/hagicode
```
Esta configuração significa que todas as tags de `docker.io/newbe36524/hagicode` será sincronizado para `hagicode.azurecr.io/hagicode`.

### 4. Parâmetros do sincronizador de imagem
- `--auth=./auth.yaml`: caminho para o arquivo de configuração de autenticação
- `--images=./images.yaml`: caminho para o arquivo de configuração de sincronização de imagem
- `--proc=10`: usar `10` trabalhadores simultâneos
- `--retries=3`: repetir falhas `3` vezes

## Notas e melhores práticas

### 1. Recomendações de segurança
- Alterne as senhas do Azure ACR regularmente (recomendado a cada 90 dias)
- Use uma conta de serviço ACR dedicada com permissões somente push
- Monitore os logs de acesso ACR para detectar acessos anormais a tempo
- Não gere credenciais em logs
- Não confirme credenciais no repositório de código

### 2. Otimização de desempenho
- Ajuste o `--proc` parâmetro: ajuste a simultaneidade com base na largura de banda da rede (recomendado `5-20`)
- Monitore a duração da sincronização: se a sincronização demorar muito, considere reduzir a simultaneidade
- Limpe os registros regularmente: defina um valor razoável `retention-days` valor (`7` atualmente)

### 3. Solução de problemas

#### Problema 1: falha de autenticação
```
Error: failed to authenticate to hagicode.azurecr.io
```
Solução:
1. Verifique se os segredos do GitHub estão configurados corretamente
2. Verifique se a senha do Azure ACR expirou
3. Confirme se as permissões da conta de serviço ACR estão corretas

#### Problema 2: tempo limite da rede
```
Error: timeout waiting for response
```
Solução:
1. Verifique a conectividade da rede
2. Reduzir o número de trabalhadores simultâneos (o `--proc` parâmetro)
3. Aguarde a recuperação da rede e execute novamente o fluxo de trabalho

#### Problema 3: sincronização de imagem incompleta
```
Warning: some tags failed to sync
```
Solução:
1. Verifique os logs de sincronização para identificar tags com falha
2. Execute novamente manualmente o fluxo de trabalho para sincronizar novamente
3. Verifique se a imagem de origem no Docker Hub está íntegra

### 4. Monitoramento e alertas
- Verifique regularmente a página Ações para confirmar o status do fluxo de trabalho
- Habilite notificações do GitHub para que falhas no fluxo de trabalho sejam relatadas imediatamente
- Monitorar o uso do armazenamento do Azure ACR
- Verifique regularmente a consistência das tags nos registros

## Perguntas frequentes e soluções

### P1: Como sincronizo tags específicas em vez de todas as tags?

Modifique o `images.yaml` arquivo de configuração:
```yaml
# Sync only the latest and v1.0 tags
docker.io/newbe36524/hagicode:latest: hagicode.azurecr.io/hagicode:latest
docker.io/newbe36524/hagicode:v1.0: hagicode.azurecr.io/hagicode:v1.0
```

### P2: Como sincronizo vários repositórios de imagens?

Adicione várias linhas a `images.yaml`:
```yaml
docker.io/newbe36524/hagicode: hagicode.azurecr.io/hagicode
docker.io/newbe36524/another-image: hagicode.azurecr.io/another-image
```

### P3: Como posso tentar novamente após falha na sincronização?
- Nova tentativa automática: o image-syncer inclui um mecanismo de nova tentativa integrado (`3` novas tentativas por padrão)
- Nova tentativa manual: clique `Re-run all jobs` na página de ações do GitHub

### P4: Como posso visualizar o progresso detalhado da sincronização?
- Veja registros em tempo real na página Ações
- Baixe o `sync-logs` artefato para inspecionar os arquivos de log completos
- Os arquivos de log incluem status de sincronização e velocidade de transferência para cada tag

### Q5: Quanto tempo leva a sincronização?
- Sincronização completa inicial: geralmente `10-30` minutos dependendo do tamanho da imagem
- Sincronização incremental: geralmente `2-5` minutos se as alterações na imagem forem pequenas
- O tempo depende da largura de banda da rede, do tamanho da imagem e das configurações de simultaneidade

### 2. Implementar filtragem de tags de imagem
Adicione lógica de filtragem de tags ao fluxo de trabalho:
```yaml
- name: Filter tags
  run: |
    # Sync only tags prefixed with v
    echo "docker.io/newbe36524/hagicode:v* : hagicode.azurecr.io/hagicode:v*" > images.yaml
```

### 3. Adicione um relatório de estatísticas de sincronização
```yaml
- name: Generate report
  if: always()
  run: |
    echo "## Sync Report" >> $GITHUB_STEP_SUMMARY
    echo "- Total tags: $(grep -c 'synced' image-syncer-*.log)" >> $GITHUB_STEP_SUMMARY
    echo "- Sync time: ${{ steps.sync.outputs.duration }}" >> $GITHUB_STEP_SUMMARY
```
