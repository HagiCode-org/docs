## Hagicode.Libs: prática de engenharia para integração unificada de múltiplas CLIs do AI Coding Assistant

> Durante o desenvolvimento do projeto HagiCode, precisávamos integrar vários CLIs assistentes de codificação de IA ao mesmo tempo, incluindo Claude Code, Codex e CodeBuddy. Cada CLI possui diferentes interfaces, parâmetros e formatos de saída, e o código de integração repetido tornou o projeto cada vez mais difícil de manter. Neste artigo, compartilhamos como construímos uma camada de abstração unificada com HagiCode.Libs para resolver esse problema de engenharia. Você também poderia dizer que é simplesmente uma experiência suada, adquirida com as armadilhas que já enfrentamos.

## Plano de fundo

O mercado de assistentes de codificação de IA está bastante animado agora. Além do Claude Code, há também o Codex da OpenAI, o CodeBuddy da Zhipu e muito mais. Como um projeto de assistente de codificação de IA, o HagiCode precisa integrar essas diferentes ferramentas CLI em vários subprojetos, incluindo desktop, back-end e web.

No início, o problema era administrável. A integração de uma CLI exigiu apenas algumas centenas de linhas de código. Mas à medida que o número de CLIs que precisávamos para oferecer suporte continuava crescendo, as coisas começaram a ficar complicadas.

Cada CLI tem seu próprio formato de argumento de linha de comando, diferentes requisitos de variáveis de ambiente e uma ampla variedade de formatos de saída. Alguns produzem JSON, alguns produzem JSON de streaming e alguns produzem texto simples. Além disso, existem problemas de compatibilidade entre plataformas. A descoberta de executáveis ​​e o gerenciamento de processos funcionam de maneira muito diferente entre os sistemas Windows e Unix, portanto, a duplicação de código continuou aumentando. Na verdade, era só um pouco mais de Ctrl+C e Ctrl+V, mas a manutenção rapidamente se tornou dolorosa.

A parte mais frustrante foi que toda vez que queríamos adicionar suporte para um novo recurso CLI, tínhamos que alterar o mesmo código em vários projetos. Essa abordagem claramente não era sustentável no longo prazo. Code também tem temperamento; duplique-o muitas vezes e isso começará a causar problemas.

## Sobre HagiCode

A abordagem compartilhada neste artigo vem de nossa experiência prática no [HagiCode](https://hagicode.com) projeto. HagiCode é um projeto de assistente de codificação de IA de código aberto que precisa manter vários subprojetos ao mesmo tempo, incluindo uma extensão VSCode de front-end, serviços de IA de back-end e um cliente de desktop multiplataforma. De certa forma, foi exatamente esse ambiente complexo, multilíngue e multiplataforma que levou ao nascimento do HagiCode.Libs. Você poderia dizer que fomos forçados a isso, e que assim seja.

## Análise: Encontrando um terreno comum

Embora cada uma dessas CLIs assistentes de codificação de IA tenha suas próprias características, do ponto de vista técnico elas compartilham várias características óbvias:

**Padrões de interação semelhantes**: todos iniciam um processo CLI, enviam um prompt, recebem respostas de streaming, analisam mensagens e, em seguida, encerram ou continuam a sessão. No final das contas, todo o fluxo segue o mesmo molde básico.

**Necessidades de configuração semelhantes**: todos eles precisam de autenticação de chave de API, configuração de diretório de trabalho, seleção de modelo, controle de permissão de ferramenta e gerenciamento de sessão. Afinal, todo mundo ganha a vida com APIs; as diferenças são principalmente uma questão de sabor.

**Os mesmos desafios de plataforma cruzada**: todos eles precisam resolver a resolução de caminho executável (`claude` contra `claude.exe` contra `/usr/local/bin/claude`), inicialização de processos e manipulação de variáveis ​​de ambiente, escape de comandos shell e construção de argumentos. O trabalho entre plataformas é doloroso, não importa como você o descreva. Somente as pessoas que caíram nas armadilhas realmente entendem a diferença entre o Windows e o Unix.

Com base nessa análise, precisávamos de uma camada de abstração unificada que pudesse fornecer uma interface consistente, encapsular a lógica de descoberta CLI de plataforma cruzada, lidar com a análise de saída de streaming e oferecer suporte a cenários de injeção de dependência e não-DI. É o tipo de problema que dá dor de cabeça só de pensar, mas você ainda tem que enfrentá-lo. Afinal, é um projeto nosso, então temos que terminá-lo, mesmo que tenhamos que chorar.

## Solução: HagiCode.Libs

Criamos **HagiCode.Libs**, um espaço de trabalho leve de biblioteca .NET 10 lançado sob a licença MIT e agora publicado em [GitHub](https://github.com/HagiCode-org/Hagicode.Libs). Pode não ser uma obra-prima que abale o mundo, mas é genuinamente útil para resolver problemas reais.

### Estrutura do projeto

```
HagiCode.Libs/
├── src/
│   ├── HagiCode.Libs.Core/           # Core capabilities
│   │   ├── Discovery/                 # CLI executable discovery
│   │   ├── Process/                   # Cross-platform process management
│   │   ├── Transport/                 # Streaming message transport
│   │   └── Environment/               # Runtime environment resolution
│   ├── HagiCode.Libs.Providers/       # Provider implementations
│   │   ├── ClaudeCode/                # Claude Code provider
│   │   ├── Codex/                     # Codex provider
│   │   └── Codebuddy/                 # CodeBuddy provider
│   ├── HagiCode.Libs.ConsoleTesting/  # Testing framework
│   ├── HagiCode.Libs.ClaudeCode.Console/
│   ├── HagiCode.Libs.Codex.Console/
│   └── HagiCode.Libs.Codebuddy.Console/
└── tests/                             # xUnit tests
```

### Objetivos de design

Ao projetar o HagiCode.Libs, seguimos alguns princípios. Todos eles vieram de lições aprendidas da maneira mais difícil:

**Zero dependências pesadas de framework**: não depende de ABP ou de qualquer outro framework grande, o que o mantém leve. Hoje em dia, quanto menos dependências você tiver, menos dores de cabeça terá. A maioria das pessoas já foi espancada pelo inferno da dependência pelo menos uma vez.

**Suporte multiplataforma**: suporte nativo para Windows, macOS e Linux, sem escrever código separado para plataformas diferentes. Uma base de código que roda em qualquer lugar é uma coisa muito boa.

**Processamento de streaming**: a saída da CLI é tratada com fluxos assíncronos, o que se adapta muito melhor aos padrões de programação .NET modernos. Os tempos mudam e o assíncrono é rei.

**Integração flexível**: suporta cenários de injeção de dependência e também permite instanciação direta. Pessoas diferentes têm preferências diferentes, por isso queríamos que fosse conveniente de qualquer maneira.

## Como usar

### Através de injeção de dependência

Se o seu projeto já usa injeção de dependência, como ASP.NET Core ou host genérico, você pode integrá-lo diretamente. É uma coisa pequena, mas bem comportada:

```csharp
using HagiCode.Libs.Providers;
using Microsoft.Extensions.DependencyInjection;

var services = new ServiceCollection();
services.AddHagiCodeLibs();

await using var provider = services.BuildServiceProvider();
var claude = provider.GetRequiredService<ICliProvider<ClaudeCodeOptions>>();

var options = new ClaudeCodeOptions
{
    ApiKey = "your-api-key",
    Model = "claude-sonnet-4-20250514"
};

await foreach (var message in claude.ExecuteAsync(options, "Hello, Claude!"))
{
    Console.WriteLine($"{message.Type}: {message.Content}");
}
```

### Instanciação direta

Se você estiver escrevendo um script simples ou trabalhando em um cenário sem DI, criar uma instância diretamente também funciona. Simplificando, depende da sua preferência pessoal:

```csharp
var claude = new ClaudeCodeProvider();
var options = new ClaudeCodeOptions
{
    ApiKey = "sk-ant-xxx",
    Model = "claude-sonnet-4-20250514"
};

await foreach (var message in claude.ExecuteAsync(options, "Help me write a quicksort"))
{
    // Handle messages
}
```

Ambas as abordagens usam a mesma implementação subjacente, para que você possa escolher o estilo de integração que melhor se adapta ao seu projeto. Não existe uma resposta certa universal neste mundo. O que combina com você é a melhor opção. Pode parecer clichê, mas é verdade.

## Experiência prática

### 1. Consoles de teste dedicados

Cada provedor possui seu próprio projeto de console de testes dedicado, facilitando a validação da integração de forma independente. O teste é uma daquelas coisas em que, se você quiser fazê-lo, deverá fazê-lo corretamente:

```bash
# Claude Code tests
dotnet run --project src/HagiCode.Libs.ClaudeCode.Console -- --test-provider
dotnet run --project src/HagiCode.Libs.ClaudeCode.Console -- --test-all claude

# CodeBuddy tests
dotnet run --project src/HagiCode.Libs.Codebuddy.Console -- --test-provider codebuddy-cli

# Codex tests
dotnet run --project src/HagiCode.Libs.Codex.Console -- --test-provider codex-cli
```

Os cenários de teste cobrem vários casos importantes:
- **Ping**: verificação de integridade para confirmar se a CLI está disponível
- **Prompt Simples**: teste básico de prompt
- **Prompt Complexo**: teste de conversação multivoltas
- **Restauração/Retomada de Sessão**: teste de recuperação de sessão
- **Análise de Repositório**: teste de análise de repositório

Esse design de console de teste independente é especialmente útil durante a depuração porque nos permite identificar rapidamente se o problema está na camada HagiCode.Libs ou na própria CLI. A depuração consiste apenas em descobrir onde está o problema. Quando a direção estiver correta, você já estará na metade do caminho.

### 2. Validação CI/CD multiplataforma

A compatibilidade entre plataformas é um dos principais objetivos do HagiCode.Libs. Configuramos o fluxo de trabalho do GitHub Actions `.github/workflows/cli-discovery-cross-platform.yml` para executar validação de descoberta CLI real em `ubuntu-latest`, `macos-latest`e `windows-latest`.

Isso garante que cada alteração de código não prejudique a compatibilidade entre plataformas. Durante o desenvolvimento local, você também pode reproduzi-lo com os seguintes comandos. Afinal, você não pode pedir à CI que assuma a culpa por tudo. Seu ambiente local também deve ser capaz de executá-lo:

```bash
npm install --global @anthropic-ai/claude-code@2.1.79
HAGICODE_REAL_CLI_TESTS=1 dotnet test --filter "Category=RealCli"
```

### 3. Processamento de fluxo de mensagens

HagiCode.Libs usa fluxos assíncronos para processar a saída CLI. Em comparação com o retorno de chamada tradicional ou abordagens baseadas em eventos, isso se adapta muito melhor ao estilo de programação assíncrona do .NET moderno. No final das contas, é simplesmente assim que a tecnologia avança, quer alguém goste ou não:

```csharp
public async IAsyncEnumerable<CliMessage> ExecuteAsync(
    TOptions options,
    string prompt,
    [EnumeratorCancellation] CancellationToken cancellationToken = default)
{
    // Start the CLI process
    // Parse streaming JSON output
    // Yield the CliMessage sequence
}
```

Os tipos de mensagens incluem:
- `user`: mensagem do usuário
- `assistant`: resposta do assistente
- `tool_use`: invocação de ferramenta
- `result`: fim da sessão

Esse design permite que os chamadores lidem com a saída de streaming de maneira flexível, seja para exibição em tempo real, pós-processamento em buffer ou encaminhamento para outros serviços. Por que se preocupar se o céu está ensolarado ou nublado? O que importa é que, uma vez que a ideia se abra, você possa usá-la como quiser.

### 4. Exploração do repositório Git

O módulo HagiCode.Libs.Exploration fornece descoberta de repositório Git e verificação de status, o que é especialmente útil em cenários de análise de repositório. Esse recurso também nasceu por necessidade, pois o HagiCode precisa analisar repositórios:

```csharp
// Discover Git repositories
var repositories = await GitRepositoryDiscovery.DiscoverAsync("/path/to/search");

// Get repository information
var info = await GitRepository.GetInfoAsync(repoPath);
Console.WriteLine($"Branch: {info.Branch}, Remote: {info.RemoteUrl}");
Console.WriteLine($"Has uncommitted changes: {info.HasUncommittedChanges}");
```

Os recursos de análise de código do HagiCode usam este módulo para identificar a estrutura do projeto e o status do Git. É um bom exemplo de como aproveitar ao máximo o que construímos.

## Coisas a serem observadas

Com base na nossa prática no projeto HagiCode, existem vários pontos que merecem atenção especial. Todas são questões reais que precisam ser tratadas com cuidado:

**Segurança de chave de API**: não codifique chaves de API em seu código. Use variáveis ​​de ambiente ou gerenciamento de configuração. HagiCode.Libs suporta a passagem de configuração por meio de objetos Options, facilitando a integração com diferentes fontes de configuração. Quando se trata de segurança, não existe cuidado demais.

**Fixação de versão CLI**: em CI/CD, fixamos versões específicas, como `@anthropic-ai/claude-code@2.1.79`, para reduzir a incerteza causada pelo desvio de versão. Também é uma boa ideia usar versões fixas no desenvolvimento local. O controle de versão pode ser doloroso. Se você não fixar as versões, o problema lhe ensinará uma lição muito rapidamente.

**Categorização de testes**: os testes padrão usam provedores falsos para mantê-los determinísticos e rápidos, enquanto os testes CLI reais devem ser habilitados explicitamente. Isso fornece feedback rápido ao CI e ainda permite a validação do ambiente real quando necessário. Alcançar esse equilíbrio nunca é fácil. Velocidade e estabilidade sempre exigem compensações.

**Gerenciamento de sessões**: CLIs diferentes possuem mecanismos de recuperação de sessão diferentes. Código Claude usa o `.claude/` diretório para armazenar sessões, enquanto Codex e CodeBuddy têm suas próprias abordagens. Ao utilizá-los, certifique-se de verificar sua respectiva documentação e compreender os detalhes de seus mecanismos de persistência de sessão. Não há mal nenhum em entendê-lo claramente.
