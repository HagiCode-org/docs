## Edição rápida de código no navegador: integração Web VSCode na prática

> Depois que a IA termina de analisar o código, como você abre imediatamente um editor no navegador e começa a fazer alterações? Este artigo compartilha nossa experiência prática na integração do servidor de código no projeto HagiCode para criar uma ponte perfeita entre o assistente de IA e a experiência de edição de código.

## Plano de fundo

Na era da programação assistida por IA, os desenvolvedores muitas vezes precisam inspecionar e editar o código rapidamente. O fluxo de trabalho tradicional é simples: abra o projeto em um IDE de desktop, localize o arquivo, edite-o e salve. Mas em algumas situações, esse fluxo sempre parece um pouco estranho.

Cenário um: desenvolvimento remoto. Ao usar um assistente de IA como o HagiCode, o backend pode estar sendo executado em um servidor remoto ou dentro de um contêiner, e as máquinas locais não podem acessar diretamente os arquivos do projeto. Cada vez que você precisa inspecionar ou modificar código, é necessário conectar-se por meio de SSH ou outro método, e a experiência parece fragmentada. É como querer conhecer alguém através de um vidro grosso: você pode vê-lo, mas não pode alcançá-lo.

Cenário dois: visualizações rápidas. Depois que o assistente de IA analisa o código, o usuário pode querer apenas navegar rapidamente em um arquivo ou fazer uma pequena alteração. Lançar um IDE de desktop completo parece pesado, enquanto um editor leve no navegador atende melhor à necessidade de uma “visualização rápida”. Afinal, quem quer mobilizar todo um conjunto de ferramentas só para dar uma olhada?

Cenário três: colaboração entre dispositivos. Ao trabalhar em diferentes dispositivos, um editor baseado em navegador fornece um ponto de acesso unificado sem exigir que cada máquina seja configurada com um ambiente de desenvolvimento. Só isso já evita muitos problemas. A vida é curta; por que repetir o mesmo trabalho de configuração indefinidamente?

Para resolver esses problemas, integramos o VSCode Web ao projeto HagiCode. Isso permite que o assistente de IA e a experiência de edição de código se conectem perfeitamente: depois que a IA analisa o código, os usuários podem abrir imediatamente um editor e fazer alterações na mesma sessão do navegador, sem trocar de aplicativo. É o tipo de experiência onde, quando você precisa, ela simplesmente está lá.

## Sobre HagiCode

A solução compartilhada neste artigo vem de nossa experiência prática no [HagiCode](https://hagicode.com) projeto. HagiCode é um assistente de codificação baseado em IA, projetado para melhorar a eficiência do desenvolvimento por meio da interação em linguagem natural. Durante o desenvolvimento, descobrimos que os usuários muitas vezes precisam alternar rapidamente entre a análise de IA e a edição de código, o que nos levou a explorar como integrar o editor diretamente ao navegador.

Repositório do projeto: [github.com/HagiCode-org/site](https://github.com/HagiCode-org/site)

## Escolha Técnica: Por que servidor de código?

Entre as diversas soluções VSCode Web disponíveis, escolhemos o servidor de código. Houve algumas razões concretas por trás dessa decisão.

Completude de recursos. code-server é a versão web do VSCode e oferece suporte à maioria dos recursos de desktop, incluindo sistema de extensão, sugestões inteligentes, depuração e muito mais. Isso significa que os usuários podem obter uma experiência de edição no navegador muito próxima da versão para desktop. Afinal, quem realmente quer comprometer a funcionalidade?

Implantação flexível. code-server pode ser executado como um serviço independente e também suporta implantação baseada em Docker, que se adapta bem à arquitetura do HagiCode. Nosso backend é escrito em C#, o frontend usa React e os dois se comunicam com o serviço do servidor de código por meio de APIs REST. É como construir com blocos: cada peça tem o seu lugar.

Autenticação segura. servidor de código inclui um built-in `connection-token` mecanismo para impedir o acesso não autorizado. Cada sessão possui um token exclusivo para que apenas usuários autorizados possam abrir o editor. Segurança é uma daquelas coisas que você só aprecia totalmente quando a possui.

## Projeto de Arquitetura

A integração VSCode Web do HagiCode usa uma arquitetura separada front-end/back-end.

### Camada de serviço front-end

O front-end envolve interações com o back-end por meio de `vscodeServerService.ts`:

```typescript
// Open project
export async function openProjectInCodeServer(
  id: string,
  currentInterfaceLanguage?: string,
): Promise<VsCodeServerLaunchResponseDto>

// Open vault
export async function openVaultInCodeServer(
  id: string,
  path?: string,
  currentInterfaceLanguage?: string,
): Promise<VsCodeServerLaunchResponseDto>
```

A diferença entre esses dois métodos é direta: `openProjectInCodeServer` abre o projeto inteiro, enquanto `openVaultInCodeServer` abre um caminho específico dentro de um Vault. Para projetos de vários repositórios MonoSpecs, o sistema cria automaticamente um arquivo de espaço de trabalho. Muitas vezes, responsabilidades claras são suficientes quando cada parte faz bem o seu trabalho.

### Camada de serviço de back-end

O back-end `VaultAppService.cs` implementa a lógica central:

```csharp
public async Task<VsCodeServerLaunchResponseDto> OpenInCodeServerAsync(
    string id,
    string? relativePath = null,
    string? currentInterfaceLanguage = null,
    CancellationToken cancellationToken = default)
{
    // 1. Get settings and check whether the feature is enabled
    var settings = await _vsCodeServerSettingsService.GetResolvedSettingsAsync(cancellationToken);
    if (!settings.Enabled) {
        throw new BusinessException(VsCodeServerErrorCodes.Disabled, "VSCode Server is disabled.");
    }

    // 2. Get vault and resolve the launch directory
    var vault = await RequireVaultAsync(id, cancellationToken);
    var launchDirectory = ResolveLaunchDirectory(vault, relativePath);

    // 3. Ensure code-server is running and get runtime info
    var runtime = await _vsCodeServerManager.EnsureStartedAsync(settings, cancellationToken);

    // 4. Resolve language settings
    var language = _vsCodeServerSettingsService.ResolveLaunchLanguage(
        settings.Language,
        currentInterfaceLanguage);

    // 5. Build launch URL
    return new VsCodeServerLaunchResponseDto {
        LaunchUrl = AppendQueryString(runtime.BaseUrl, new Dictionary<string, string?> {
            ["folder"] = launchDirectory,
            ["tkn"] = runtime.ConnectionToken,
            ["vscode-lang"] = language,
        }),
        ConnectionToken = runtime.ConnectionToken,
        OpenMode = "folder",
        Runtime = VsCodeServerSettingsService.MapRuntime(
            await _vsCodeServerManager.GetRuntimeSnapshotAsync(cancellationToken)),
    };
}
```

Este método tem uma responsabilidade muito clara: verificar configurações, resolver caminhos, iniciar o serviço e construir a URL. Entre eles, o `ResolveLaunchDirectory` O método executa verificações de segurança de caminho para evitar ataques de passagem de caminho. O código pode parecer um pouco com poesia quando cada linha tem um propósito.

### Gerenciamento Automático de Tempo de Execução

O back-end gerencia o processo do servidor de código por meio de `VsCodeServerManager`:

- Verifique o status do processo
- Iniciar automaticamente serviços interrompidos
- Retornar instantâneos de tempo de execução, como porta, ID do processo e hora de início

Esse design permite que o sistema lide automaticamente com o ciclo de vida do servidor de código, para que os usuários não precisem gerenciar processos de serviço manualmente. A vida já é bastante complicada; qualquer coisa que possa ser automatizada deveria ser.

### Sincronização de idiomas

HagiCode suporta uma interface multilíngue e o servidor de código precisa seguir essa configuração. O sistema suporta três modos de idioma:

- `follow`: siga o idioma da interface atual
- `zh-CN`: corrigido para chinês
- `en-US`: corrigido para inglês

A configuração é passada para o servidor de código através do `vscode-lang` Parâmetro de URL para que a linguagem do editor fique consistente com a interface do HagiCode. A linguagem parece melhor quando é unificada.

### Espaço de trabalho multi-repositório MonoSpecs

Para projetos MonoSpecs, que contêm vários subrepositórios dentro de um monorepo, o sistema cria automaticamente um `.code-workspace` arquivo:

```csharp
private async Task<string> CreateWorkspaceFileAsync(Project project, Guid projectId)
{
    var folders = await ResolveWorkspaceFoldersAsync(project.Path);
    var workspaceDocument = new {
        folders = folders.Select(path => new { path }).ToArray(),
    };
    // Generate workspace file...
}
```

Isso torna possível editar vários subrepositórios na mesma instância do servidor de código, o que é especialmente prático para grandes projetos monorepo. Vários repositórios em uma janela podem parecer várias histórias reunidas no mesmo livro.

## Integração de front-end

O frontend HagiCode usa React + TypeScript, e a integração do servidor de código não é especialmente complicada.

### Botão de ação rápida

Adicione um botão Code Server ao cartão do projeto:

```tsx
// QuickActionsZone.tsx
<Button
  size="sm"
  variant="default"
  onClick={() => onAction({ type: 'open-code-server' })}
>
  <Globe className="h-3 w-3 mr-1" />
  <span className="text-xs">{t('project.openCodeServer')}</span>
</Button>
```

Este botão aciona a ação de abertura e chama a API de backend para obter o URL de lançamento. Um botão, uma ação, direta e simples.

### Lidando com a ação aberta

```tsx
const handleAction = async (action: ProjectAction) => {
  if (action.type === 'open-code-server') {
    const response = await openProjectInCodeServer(project.id, i18n.language);
    window.open(response.launchUrl, '_blank', 'noopener,noreferrer');
  }
};
```

Usar `window.open` para abrir o servidor de código em uma nova guia. O `noopener,noreferrer` parâmetros fornecem segurança extra. Quando se trata de segurança, não existe cuidado demais.

### Entrada de edição do Vault

Adicione um botão de edição semelhante na lista do Vault:

```tsx
const handleEditVault = async (vault: VaultItemDto) => {
  const response = await openVaultInCodeServer(vault.id);
  window.open(response.launchUrl, '_blank', 'noopener,noreferrer');
};
```

Projetos e Vaults usam o mesmo mecanismo aberto, o que mantém a interação consistente. A consistência é quase tão importante quanto o próprio recurso.

## Lógica de construção de URL

O formato da URL para code-server possui alguns detalhes dignos de nota.

**Modo pasta**:
```
http://{host}:{port}/?folder={path}&tkn={token}&vscode-lang={lang}
```

**Modo de espaço de trabalho**:
```
http://{host}:{port}/?workspace={workspacePath}&tkn={token}&vscode-lang={lang}
```

Aqui, `tkn` é o token de conexão. Ele é gerado automaticamente sempre que o servidor de código é iniciado, garantindo acesso seguro. O `vscode-lang` O parâmetro controla o idioma da UI do editor. Cada um desses parâmetros tem um papel a desempenhar.

## Cenários de uso

### Cenário Um: Revisão de Código Assistida por IA

O usuário conversa com o HagiCode, a IA analisa o código do projeto e encontra um possível problema, e então o usuário clica no botão "Abrir no servidor de código" para abrir o editor diretamente no navegador, inspecionar o arquivo afetado, corrigi-lo e retornar ao HagiCode para continuar a conversa. Todo o fluxo acontece no navegador sem trocar de aplicativo. É suave da mesma forma que a água corrente é suave.

### Cenário Dois: Editando Materiais de Estudo em um Vault

Um usuário cria um Vault para estudar um projeto de código aberto e deseja adicionar notas de estudo no `docs/` diretório. Com o code-server, eles podem editar arquivos Markdown diretamente no navegador, salvá-los e deixar o HagiCode ler imediatamente as notas atualizadas. Isto é especialmente útil para construir uma base de conhecimento pessoal. O conhecimento só se torna mais valioso quanto mais você o acumula.

### Cenário Três: Desenvolvimento de Multi-Repositórios MonoSpecs

Um projeto MonoSpecs contém vários subrepositórios e o servidor de código cria automaticamente um espaço de trabalho com várias pastas. No navegador, os usuários podem editar o código em vários repositórios de uma só vez e, em seguida, enviar as alterações de volta para seus respectivos repositórios Git. Este fluxo de trabalho é particularmente adequado para alterações que precisam abranger vários repositórios. Editar vários repositórios juntos exige um pouco de técnica, assim como lidar com várias tarefas ao mesmo tempo.

## Considerações de segurança

Ao implementar a integração código-servidor, a segurança merece atenção especial. Se a segurança der errado, você sempre perceberá tarde demais.

### Token de conexão

O `connection-token` é gerado aleatoriamente e não deve ser exposto. É melhor usado em HTTPS para evitar que o token seja interceptado por um intermediário. Vale a pena proteger adequadamente as informações confidenciais.

### Segurança do caminho de arquivo

O back-end implementa verificações de passagem de caminho:

```csharp
private static string ResolveLaunchDirectory(VaultRegistryEntry vault, string? relativePath)
{
    var vaultRoot = EnsureTrailingSeparator(Path.GetFullPath(vault.PhysicalPath));
    var combinedPath = Path.GetFullPath(Path.Combine(vaultRoot, relativePath ?? "."));
    if (!combinedPath.StartsWith(vaultRoot, StringComparison.OrdinalIgnoreCase))
    {
        throw new BusinessException(VaultRelativePathTraversalCode, "Relative path traversal detected.");
    }
    return combinedPath;
}
```

Este código garante que os usuários não possam usar `../` ou padrões semelhantes para acessar arquivos fora do diretório do Vault. As verificações de limites são sempre melhor realizadas do que ignoradas.

### Controle de permissão

O processo do servidor de código deve ser executado com permissões de usuário apropriadas para que não possa acessar arquivos confidenciais do sistema. É melhor executar o serviço do servidor de código com um usuário dedicado. O controle de permissão é um daqueles fundamentos que você deve sempre manter.

## Otimização de desempenho

code-server consome recursos do servidor, então aqui estão algumas sugestões de otimização:

- Monitore o uso de CPU e memória e ajuste os limites de recursos quando necessário
- Grandes projetos podem exigir tempos limite mais longos
- Implemente a limpeza automática do tempo limite da sessão para liberar recursos
- Considere o armazenamento em cache para reduzir cálculos repetidos

HagiCode fornece uma API de monitoramento de status de tempo de execução e o frontend pode chamar `getVsCodeServerSettings()` para recuperar o estado atual:

```typescript
const { settings, runtime } = await getVsCodeServerSettings();
// runtime.status: 'disabled' | 'stopped' | 'starting' | 'running' | 'unhealthy'
// runtime.baseUrl: "http://localhost:8080"
// runtime.processId: 12345
```

Esse design permite que os usuários entendam claramente o status de integridade do servidor de código e localizem rapidamente os problemas quando algo dá errado. Quando o status é visível, as pessoas se sentem mais no controle.

## Detalhes da experiência do usuário

Durante a implementação, descobrimos alguns detalhes que afetam visivelmente a experiência do usuário e merecem atenção extra.

Abrir o servidor de código pela primeira vez pode exigir a espera pela inicialização, e esse atraso pode variar de alguns segundos a meio minuto. É uma boa ideia mostrar o estado de carregamento no frontend para que os usuários saibam que o sistema ainda está funcionando. Esperar é mais fácil quando há feedback.

Os navegadores podem bloquear o pop-up, portanto, os usuários devem ser solicitados a permiti-lo manualmente. Na primeira inicialização, o HagiCode exibe orientações que explicam como conceder as permissões necessárias do navegador. A experiência do usuário muitas vezes reside exatamente nesses pequenos detalhes.

Também é uma boa ideia exibir o status do tempo de execução, como inicialização, execução ou erro, para que, quando ocorrerem problemas, os usuários possam saber rapidamente se o problema está no lado do servidor ou em sua própria operação. Saber onde está o problema pelo menos lhe dá um ponto de partida.

## Exemplo de configuração

A configuração do code-server não é complicada:

```json
{
  "vscodeServer": {
    "enabled": true,
    "host": "0.0.0.0",
    "port": 8080,
    "language": "follow"
  }
}
```

`enabled` controla se o recurso está ativado, `host` e `port` definir o endereço de escuta e `language` define o modo de idioma. Essas configurações podem ser modificadas por meio da IU e entram em vigor imediatamente. Coisas simples costumam ser as mais fáceis de usar.

## Conclusão

A integração VSCode Web do HagiCode oferece uma solução elegante: permite que o assistente de IA e a experiência de edição de código se conectem perfeitamente. Ao integrar o servidor de código ao navegador, os usuários podem agir rapidamente com base nos resultados da análise de IA e concluir o fluxo completo, desde a análise até a edição, na mesma sessão do navegador.

Esta solução traz diversas vantagens importantes: uma experiência unificada, pois projetos e Vaults utilizam o mesmo mecanismo aberto; suporte a vários repositórios, porque os projetos MonoSpecs criam automaticamente espaços de trabalho; e segurança controlável, graças ao monitoramento do status do tempo de execução e às verificações de segurança do caminho.

A abordagem compartilhada neste artigo é algo que o HagiCode destilou de um trabalho de desenvolvimento real. Se você achar esta solução valiosa, isso sugere que nossa prática de engenharia está fazendo algo certo, e o próprio HagiCode pode valer uma olhada mais de perto. Boas ferramentas merecem ser vistas por mais pessoas.
