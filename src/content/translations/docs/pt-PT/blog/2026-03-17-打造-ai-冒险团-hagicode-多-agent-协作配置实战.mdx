## Construindo uma festa de aventura com IA: configuração de colaboração multiagente HagiCode na prática

> No desenvolvimento de software moderno, um único agente de IA não é mais suficiente para atender a requisitos complexos. Como vários assistentes de IA de diferentes empresas podem colaborar no mesmo projeto? Este artigo compartilha a abordagem de configuração de colaboração multiagente que o projeto HagiCode desenvolveu por meio da prática do mundo real.

:::note[Nota de contexto de tempo]
Este artigo registra a prática de configuração multiagente **conforme lançada em 17/03/2026**. Para o escopo atual com suporte de longo prazo, consulte [Instalação da CLI do Agente AI](/pt-PT/related-software-installation/ai-agent-cli/) e o [Guia de instalação](/pt-PT/installation); `IFlowCli` agora é retido apenas como um valor compatível com o legado e não faz mais parte do escopo com suporte ativo.
:::

## Plano de fundo

Muitos desenvolvedores provavelmente já tiveram esta experiência: depois de introduzir um assistente de IA em um projeto, a produtividade realmente melhora. Mas à medida que os requisitos se tornam cada vez mais complexos, um Agente de IA começa a sentir-se insuficiente. Você deseja que ele lide com revisão de código, geração de documentação, testes de unidade e outras tarefas ao mesmo tempo, mas o resultado geralmente é que ele não consegue manter tudo equilibrado e a qualidade da saída se torna inconsistente.

O que é ainda mais frustrante é que, quando você tenta contratar vários assistentes de IA, o problema se torna mais complicado. Cada Agente possui seu próprio método de configuração, interface API e lógica de execução, podendo até entrar em conflito entre si. É como uma equipe esportiva em que todos os jogadores são talentosos, mas ninguém sabe trabalhar em conjunto, então a partida vira uma bagunça.

O projeto HagiCode enfrentou o mesmo desafio durante o desenvolvimento. Como um projeto complexo envolvendo uma extensão frontend VSCode, serviços backend de IA e um cliente de desktop multiplataforma, precisávamos conectar vários assistentes de IA de diferentes empresas ao mesmo tempo: Claude Code, Codex, CodeBuddy, iFlow e muito mais. Como deixá-los coexistir harmoniosamente no mesmo projeto e aproveitar ao máximo seus pontos fortes tornou-se um problema fundamental que tínhamos que resolver.

Só isso já seria problema suficiente. Afinal, quem quer lidar com um monte de IAs combatentes todos os dias?

A abordagem compartilhada neste artigo é a prática de configuração de colaboração multiagente que desenvolvemos no projeto HagiCode por meio de tentativa e erro real e otimização repetida. Se você também está enfrentando problemas com vários assistentes de IA trabalhando juntos, este artigo pode lhe dar alguma inspiração. Talvez. Afinal, cada projeto é diferente.

## Sobre HagiCode

[HagiCode](https://hagicode.com) é um projeto de assistente de codificação de IA que adota um modelo de “festa de aventura” no qual vários mecanismos de IA trabalham juntos. Repositório do projeto: [github.com/HagiCode-org/site](https://github.com/HagiCode-org/site).

A abordagem de configuração multiagente compartilhada neste artigo é uma das principais tecnologias que permite ao HagiCode manter o desenvolvimento eficiente em projetos complexos. Não há nada de especialmente mágico nisso; simplesmente transforma um grupo de IAs em um grupo de aventura que pode realmente coordenar.

## Design de arquitetura multiagente do HagiCode

### De "ir sozinho" a "colaboração em equipe"

Nos primeiros dias do projeto HagiCode, também tentamos usar um único agente de IA para lidar com todas as tarefas. Logo descobrimos um gargalo claro nessa abordagem: tarefas diferentes exigem forças diferentes. Algumas tarefas necessitam de uma compreensão contextual mais forte, enquanto outras necessitam de capacidades de modificação de código mais precisas. Um agente tem dificuldade em se destacar em tudo.

Isso nos fez perceber que vários Agentes precisavam trabalhar juntos. Mas o problema era este: como permitir que produtos de IA de diferentes empresas coexistissem pacificamente no mesmo projeto? Precisávamos resolver vários problemas principais:

1. **Complexidade de gerenciamento de configuração**: cada Agente possui diferentes métodos de configuração, interfaces API e modos de execução
2. **Protocolo de comunicação unificada**: precisamos de uma forma padronizada para diferentes Agentes trocarem dados
3. **Coordenação de tarefas e divisão de trabalho**: como atribuímos o trabalho de maneira razoável para que cada agente possa aproveitar seus pontos fortes

Com essas questões em mente, começamos a projetar a arquitetura multiagente do HagiCode. Na verdade não foi tão complicado; nós apenas tivemos que pensar sobre isso com clareza.

### Visão geral da arquitetura geral

Após várias iterações, esta é a arquitetura que escolhemos:

```
┌─────────────────────────────────────────────────────────────────┐
│                    AIProviderFactory                             │
│  (Factory pattern for unified management of all AI Providers)    │
├─────────────────────────────────────────────────────────────────┤
│  ClaudeCodeCli  │  CodexCli  │  CodebuddyCli  │  IFlowCli    │
│  (Anthropic)   │  (OpenAI)  │  (Zhipu GLM)    │  (Zhipu)     │
└─────────────────────────────────────────────────────────────────┘
```

A ideia central é permitir que diferentes agentes de IA sejam gerenciados pelo mesmo conjunto de códigos por meio de uma interface unificada do provedor. Ao mesmo tempo, o padrão de fábrica é usado para criar e configurar dinamicamente esses Provedores, garantindo escalabilidade e flexibilidade em todo o sistema.

É como a divisão do trabalho na vida cotidiana. Cada um tem o seu papel; aqui simplesmente transformamos essa ideia em arquitetura de código.

### Tipos de agentes e divisão de responsabilidades

Com base na experiência real do HagiCode, atribuímos diferentes responsabilidades a cada Agente:

| Agente | Provedor | Modelo | Uso primário |
|-------|----------|-------|-------------|
| ClaudeCodeCli | Antrópico | glm-5-turbo | Gerar soluções técnicas e propostas |
| CodexCli | OpenAI/Zed | gpt-5.4 | Execute alterações precisas no código |
| CodebuddyCli | Zhipu | glm-4.7 | Refinar as descrições e a documentação da proposta |
| IFlowCli | Zhipu | glm-4.7 | Arquivar propostas e registros históricos |
| OpenCodeCli | - | - | Edição de código de uso geral |
| GitHubCopilot | Microsoft | - | Programação assistida e conclusão de código |

A lógica por trás desta divisão de trabalho é simples: cada Agente tem a sua própria área de força. Claude Code tem um bom desempenho na compreensão e análise de requisitos complexos, por isso lida com o design inicial da solução. O Codex é mais preciso ao modificar o código, por isso é mais adequado para trabalhos de implementação concretos. CodeBuddy oferece forte desempenho de custo, o que o torna ideal para refinar textos e documentação de propostas.

Afinal, a ferramenta certa para o trabalho certo é a melhor escolha. Existem muitas estradas para Roma; alguns são simplesmente mais fáceis de andar do que outros.

## Mecanismos de configuração central

### Design de interface de provedor unificado

Para gerenciar diferentes Agentes de IA de forma unificada, primeiro precisamos definir uma interface comum. No HagiCode, essa interface se parece com isto:

```csharp
public interface IAIProvider
{
    // Unified Provider interface
    Task<IAIProvider?> GetProviderAsync(AIProviderType providerType);
    Task<IAIProvider?> GetProviderAsync(string providerName, CancellationToken cancellationToken);
}
```

A interface parece simples, mas é a base de todo o sistema multiagente. Com uma interface unificada, podemos chamar produtos de IA de diferentes empresas da mesma forma, independentemente de qual empresa está por trás deles.

Na verdade, trata-se apenas de simplificar coisas complexas. Afinal, o simples é lindo.

### Implementação do padrão de fábrica do provedor

Depois que a interface estiver unificada, a próxima questão é como criar essas instâncias do Provedor. HagiCode usa o padrão de fábrica:

```csharp
private IAIProvider? CreateProvider(AIProviderType providerType, ProviderConfiguration config)
{
    return providerType switch
    {
        AIProviderType.ClaudeCodeCli =>
            ActivatorUtilities.CreateInstance<ClaudeCodeCliProvider>(_serviceProvider, Options.Create(config)),
        AIProviderType.CodebuddyCli =>
            ActivatorUtilities.CreateInstance<CodebuddyCliProvider>(_serviceProvider, Options.Create(config)),
        AIProviderType.CodexCli =>
            ActivatorUtilities.CreateInstance<CodexCliProvider>(_serviceProvider, Options.Create(config)),
        AIProviderType.IFlowCli =>
            ActivatorUtilities.CreateInstance<IFlowCliProvider>(_serviceProvider, Options.Create(config)),
        _ => null
    };
}
```

Isso usa injeção de dependência por meio de `ActivatorUtilities.CreateInstance`, que pode criar instâncias de Provider dinamicamente em tempo de execução enquanto injeta dependências automaticamente. A vantagem deste design é que quando um novo tipo de Agente é adicionado, você só precisa adicionar a classe de Provedor correspondente e depois adicionar mais uma. `case` ramificação no método de fábrica. Não há necessidade de modificar o código existente.

Isso é razão suficiente. Quem quer reescrever uma pilha de código antigo sempre que um novo recurso é adicionado?

### Resolução de configuração dinâmica

Para tornar a configuração mais flexível, também implementamos um mecanismo de mapeamento de tipos:

```csharp
public static AIProviderTypeExtensions
{
    private static readonly Dictionary<string, AIProviderType> _typeMap = new(
        StringComparer.OrdinalIgnoreCase)
    {
        ["ClaudeCodeCli"] = AIProviderType.ClaudeCodeCli,
        ["CodebuddyCli"] = AIProviderType.CodebuddyCli,
        ["CodexCli"] = AIProviderType.CodexCli,
        ["IFlowCli"] = AIProviderType.IFlowCli,
        // ...more type mappings
    };
}
```

O objetivo desta tabela de mapeamento é converter nomes de provedores em formato de string em tipos enum. Isso permite que os arquivos de configuração usem nomes de string intuitivos, enquanto o código interno usa enums com segurança de tipo para processamento.

A configuração deve ser o mais intuitiva possível. Ninguém quer memorizar uma pilha de codinomes complicados.

### Exemplo de arquivo de configuração

Na prática, tudo pode ser configurado em `appsettings.json`:

```yaml
AI:
  Providers:
    Providers:
      ClaudeCodeCli:
        Enabled: true
        Model: glm-5-turbo
        WorkingDirectory: /path/to/project
      CodebuddyCli:
        Enabled: true
        Model: glm-4.7
      CodexCli:
        Enabled: true
        Model: gpt-5.4
      IFlowCli:
        Enabled: true
        Model: glm-4.7
```

Cada Provedor pode configurar parâmetros de forma independente, como ativação, versão do modelo e diretório de trabalho. Este design preserva a flexibilidade e ao mesmo tempo é fácil de gerenciar e manter.

Os arquivos de configuração são um pouco como as opções da vida: você pode optar por ativar ou desativar certas coisas. A única diferença é que as escolhas de código são mais fáceis de se arrepender mais tarde.

## Fluxo de tarefas da festa de aventura

### A Arte da Divisão de Tarefas

Com a arquitetura técnica unificada implementada, o próximo passo é fazer com que vários Agentes trabalhem juntos. HagiCode projetou um mecanismo de fluxo de tarefas para que diferentes Agentes possam lidar com diferentes estágios do trabalho:

```
Proposal creation (user)
    │
    ▼
[Claude Code] ──generate proposal──▶ Proposal document
    │                               │
    │                               ▼
    │                      [Codebuddy] ──refine description──▶ Refined proposal
    │                               │
    │                               ▼
    │                      [Codex] ──execute changes──▶ Code changes
    │                               │
    │                               ▼
    └──────────────────────▶ [iFlow] ──archive──▶ Historical records
```

O benefício desta divisão de trabalho é que cada Agente só precisa se concentrar nas tarefas que realiza melhor, em vez de tentar fazer tudo. Claude Code é responsável por gerar propostas do zero. Codebuddy torna as descrições das propostas mais claras. Codex transforma propostas em mudanças reais de código. O iFlow arquiva e preserva essas alterações.

Na verdade, isso é apenas trabalho em equipe, assim como na vida cotidiana. Cada um tem o seu papel e só juntos é que algo grande pode ser feito. A única diferença é que os membros da equipe aqui são IAs.

### Principais conclusões práticas

Na operação real, resumimos as seguintes lições:

**1. A estratégia de seleção de agentes é importante**

As tarefas não devem ser atribuídas casualmente; eles devem corresponder aos pontos fortes de cada agente:
- Geração de propostas: use o Código Claude, porque tem uma compreensão contextual mais forte
- Execução de código: use Codex, pois é mais preciso para modificação de código
- Refinamento da proposta: use Codebuddy, pois oferece forte desempenho de custo
- Armazenamento de arquivos: use o iFlow, porque é estável e confiável

Afinal, colocar a pessoa certa na tarefa certa é um princípio atemporal.

**2. O isolamento da configuração garante estabilidade**

A configuração de cada agente é gerenciada de forma independente, oferece suporte a substituições de variáveis de ambiente e usa diretórios de trabalho separados. Como resultado, um erro de configuração em um Agente não afeta os outros.

Isso é como limites pessoais na vida. Cada um precisa do seu espaço; a não interferência torna possível a coexistência harmoniosa.

**3. Mecanismo de tratamento de erros**

Uma falha em um único Agente não deverá afetar o fluxo de trabalho geral. Implementamos uma estratégia alternativa: quando um Agente falha, o sistema pode mudar automaticamente para um plano de backup ou pular essa etapa e continuar com tarefas posteriores. Ao mesmo tempo, o registro completo facilita a solução de problemas posteriormente.

Ninguém pode garantir que erros nunca acontecerão. A chave é como você lida com eles. A vida funciona da mesma maneira.

**4. Monitoramento e observabilidade**

Através do protocolo ACP (nosso protocolo de comunicação personalizado baseado em JSON-RPC 2.0), podemos acompanhar o status de execução de cada Agente. O isolamento de sessão garante segurança de simultaneidade, enquanto o cache dinâmico melhora o desempenho.

As coisas que você não consegue ver são muitas vezes as que têm maior probabilidade de dar errado. Alguma visibilidade é sempre melhor do que voar às cegas.

## Resultados e benefícios do mundo real

Após adotar esta configuração de colaboração multiagente, a eficiência de desenvolvimento do projeto HagiCode melhorou significativamente. Especificamente:

1. **A capacidade de gerenciamento de tarefas dobrou**: no passado, um Agente tinha que lidar com vários tipos de tarefas ao mesmo tempo; agora as tarefas podem ser processadas em paralelo e o rendimento aumentou dramaticamente
2. **Qualidade de saída mais estável**: cada agente se concentra apenas no que faz melhor, de modo que a consistência e a qualidade melhoram
3. **Menor custo de manutenção**: interfaces unificadas e gerenciamento de configuração tornam todo o sistema mais fácil de manter e ampliar
4. **Adicionar novos Agentes é simples**: para integrar um novo produto de IA, você só precisa implementar a interface e adicionar configuração, sem alterar a lógica central

Esta abordagem não apenas resolveu os problemas do HagiCode, mas também provou que a colaboração multiagente é uma escolha arquitetônica viável.

Os ganhos foram bastante perceptíveis. O processo foi um pouco complicado.

## Conclusão

Este artigo compartilhou a experiência prática do projeto HagiCode com configuração de colaboração multiagente. As principais conclusões incluem:

1. **Interfaces padronizadas**: `IAIProvider` unifica o comportamento de diferentes Agentes, permitindo que o código ignore qual produto da empresa está abaixo
2. **Padrão de fábrica**: `ActivatorUtilities.CreateInstance` cria dinamicamente instâncias de provedor, suportando configuração de tempo de execução e injeção de dependência
3. **Unificação de protocolos**: o protocolo ACP fornece comunicação padronizada entre Agentes através de um mecanismo bidirecional baseado em JSON-RPC 2.0
4. **Roteamento de tarefas**: atribua trabalho de maneira razoável entre diferentes Agentes para que cada um possa aproveitar seus pontos fortes, em vez de esperar que um Agente faça tudo

Este design não apenas resolve o problema de “múltiplos agentes lutando entre si”, mas também usa o mecanismo de fluxo de tarefas do grupo de aventura para tornar o processo de desenvolvimento mais automatizado e especializado.

Se você também está pensando em apresentar vários assistentes de IA, espero que este artigo forneça alguns pontos de referência úteis. É claro que cada projeto é diferente e a abordagem específica ainda precisa de ser ajustada à situação real. Não existe uma solução única para todos; a melhor solução é aquela que combina com você.

Coisas ou pessoas bonitas não precisam ser possuídas. Desde que permaneçam belos, basta apreciar essa beleza. As soluções técnicas são as mesmas: a que mais lhe convém é a melhor...
