## Do Docusaurus 3.x ao Astro 5.x: uma retrospectiva sobre a migração do site HagiCode

> Este artigo analisa nossa migração completa do site oficial do HagiCode do Docusaurus 3.x para o Astro 5.x. Iremos nos aprofundar em como a arquitetura de ilhas do Astro nos ajudou a resolver gargalos de desempenho, preservando nossos ativos de componentes React existentes, proporcionando melhorias na velocidade de construção e no desempenho de carregamento.

## Plano de fundo

Em janeiro de 2026, realizamos um “transplante de coração” no site oficial do HagiCode, migrando totalmente sua estrutura principal do Docusaurus 3.x para o Astro 5.x. Esta não foi uma reescrita impulsiva, mas uma decisão técnica cuidadosamente considerada.

Antes da migração, nosso site estava funcionalmente completo, mas começou a apresentar alguns "problemas de luxo" clássicos: artefatos de construção inchados, cargas excessivas de JavaScript e velocidade de carregamento de página abaixo do ideal em páginas de documentação complexa. Como um projeto de assistente de codificação de IA, o HagiCode precisa de documentação frequente e atualizações de recursos, portanto, a eficiência da construção afeta diretamente a velocidade de lançamento. Ao mesmo tempo, queríamos que o site fosse mais amigável aos mecanismos de pesquisa (SEO) para que mais desenvolvedores pudessem descobrir o projeto.

Para resolver esses problemas, tomamos uma decisão ousada: reconstruir todo o sistema no Astro. O impacto dessa decisão pode ser ainda maior do que você espera. Entrarei em detalhes em breve.

## Sobre HagiCode

A abordagem de migração de site compartilhada neste artigo vem de nossa experiência prática no [HagiCode](https://github.com/HagiCode-org/site) projeto.

HagiCode é um assistente de codificação de IA focado em melhorar a eficiência do desenvolvimento. Nós nos preocupamos não apenas com a iteração dos recursos principais, mas também com a experiência do desenvolvedor. Este refatorador de site também teve como objetivo oferecer aos usuários a experiência mais rápida possível ao navegar em nossos documentos e no site oficial.

## Por que deixar o ecossistema Docusaurus maduro?

Dentro do ecossistema React, o Docusaurus é há muito tempo a “resposta padrão” para sites de documentação. Ele funciona imediatamente, oferece um rico ecossistema de plug-ins e possui uma comunidade ativa. Mas à medida que o HagiCode ganhou mais recursos, também sentimos suas limitações:

1.  **Gargalos de desempenho**: Docusaurus é fundamentalmente um React SPA (aplicativo de página única). Mesmo se você escrever apenas páginas estáticas, o cliente ainda precisará carregar o tempo de execução do React e hidratar a página, o que é desnecessariamente pesado para páginas de documentos simples.
2.  **Tamanho grande do ativo**: mesmo quando uma página contém muito pouco conteúdo, o tamanho do JS incluído permanece relativamente fixo. Isso não é ideal para usuários móveis ou condições de rede ruins.
3.  **Flexibilidade limitada**: embora seja extensível, queríamos mais controle de baixo nível sobre o pipeline de construção.

O Astro chegou exatamente na hora certa para resolver esses problemas. Ele introduziu uma nova "Arquitetura de Ilhas": por padrão, o Astro gera HTML estático com zero JavaScript, e apenas os componentes que requerem interatividade são "ativados" e carregam JS. Isso significa que a maior parte do nosso site se torna HTML puro e carrega extremamente rápido.

## Estratégia de migração central: uma transição arquitetônica suave

A migração não foi apenas copiar e colar. Foi necessária uma mudança de mentalidade. Passamos do modelo “all React” do Docusaurus para o modelo “Core + Islands” do Astro.

### 1. Reconstruindo o Sistema de Configuração

Primeiro, tivemos que passar de `docusaurus.config.ts` para `astro.config.mjs`. Isso não foi apenas uma renomeação de arquivo, mas uma reescrita do roteamento e da lógica de construção.

No Docusaurus, tudo é um plugin. No Astro tudo é uma integração. Precisávamos redefinir o caminho base do site, o modo de saída de construção (estático vs SSR) e a estratégia de otimização de ativos.

**Antes da migração:**
```typescript
// docusaurus.config.ts
export default {
  title: 'HagiCode',
  url: 'https://hagicode.com',
  baseUrl: '/',
  // ... more configuration
};
```

**Após a migração:**
```javascript
// astro.config.mjs
import { defineConfig } from 'astro/config';
import react from '@astrojs/react';

export default defineConfig({
  integrations: [react()],
  site: 'https://hagicode.com',
  base: '/',
  // Optimization settings for static assets
  build: {
    inlineStylesheets: 'auto',
  },
});
```

### 2. O que manter e o que refatorar nos componentes do React

Esta foi a parte mais dolorosa da migração. Nosso site existente tinha muitos componentes React, como guias, destaque de código, botões de feedback e muito mais. Jogá-los fora seria um desperdício, mas manter tudo tornaria a carga útil do JavaScript muito pesada.

HagiCode adotou uma estratégia de **hidratação progressiva**:
*   **Componentes estáticos puros**: para conteúdo de apresentação, como cabeçalhos, rodapés e documentação de texto simples, nós os reescrevemos como componentes Astro (`.astro` arquivos) e os renderizou diretamente em HTML no momento da construção.
*   **Ilhas interativas**: para componentes que devem permanecer interativos, como alternadores de tema, alternância de guias e botões de cópia de bloco de código, mantivemos a implementação do React e adicionamos `client:load` ou `client:visible` diretivas.

Por exemplo, nosso componente Tabs comumente usado na documentação:

```jsx
// src/components/Tabs.jsx
import { useState } from 'react';
import './Tabs.css'; // Import styles

export default function Tabs({ items }) {
  const [activeIndex, setActiveIndex] = useState(0);
  // ... state logic
  return (
    <div className="tabs-wrapper">
      {/* Rendering logic */}
    </div>
  );
}
```

Quando usado no Markdown, dizemos explicitamente ao Astro: "Este componente precisa de JS."

```markdown
// src/content/docs/example.mdx
import Tabs from '../../components/Tabs.jsx';

<!-- Load JS only when the component enters the viewport -->
<Tabs client:visible items={...} />
```

Dessa forma, os componentes interativos fora da janela de visualização não competem pela largura de banda, o que melhora muito a velocidade de carregamento da primeira tela.

### 3. Adaptando o sistema de estilo: dos módulos CSS ao CSS com escopo

O Docusaurus oferece suporte a módulos CSS por padrão, enquanto o Astro incentiva CSS com escopo por meio do `<style>` etiqueta. A ideia central por trás de ambos é o isolamento de estilo, mas a sintaxe é diferente.

Durante a migração do HagiCode, convertemos os módulos CSS mais complexos nos estilos com escopo do Astro. Na verdade, isso acabou sendo uma coisa boa, porque em `.astro` arquivos os estilos e modelos ficam no mesmo arquivo, o que torna a manutenção mais intuitiva.

**Antes de refatorar:**
```css
/* Tabs.module.css */
.wrapper { background: var(--ifm-background-color); }
```

**Após a refatoração (Astro Scoped):**
```html
<!-- Tabs.astro -->
<div class="tabs-wrapper">
  <slot />
</div>

<style>
  .tabs-wrapper {
    /* Use CSS variables directly to adapt to the theme */
    background: var(--bg-color);
    padding: 1rem;
  }
</style>
```

Ao mesmo tempo, unificamos o sistema global de variáveis CSS e usamos os recursos de reconhecimento de ambiente do Astro para garantir que o modo escuro alterne suavemente entre as páginas.

## Armadilhas que encontramos na prática e como as resolvemos

Durante a migração real do HagiCode, encontramos alguns problemas. Aqui estão alguns dos mais típicos.

### 1. Pontos problemáticos variáveis do caminho e do ambiente

HagiCode suporta implantação de subcaminho, como implantação em um subdiretório GitHub Pages. No Docusaurus, `baseUrl` é tratado automaticamente. No Astro, entretanto, precisamos ter mais cuidado ao lidar com links de imagens e solicitações de API.

Introduzimos um mecanismo de variável de ambiente para gerenciar isso de forma consistente:

```javascript
// Handle paths in the build script
const getBasePath = () => import.meta.env.VITE_SITE_BASE || '/';
```

Certifique-se de não codificar caminhos começando com `/` no seu código. No desenvolvimento versus produção, ou após configurar um `base` caminho, isso pode causar erros 404 para ativos.

### 2. Compatibilidade de script CommonJS

Nosso site antigo tinha alguns scripts Node.js usados para tarefas como busca automática de dados de métricas e atualização do mapa do site, e eles foram escritos em CommonJS (`require`). O Astro e as ferramentas de construção modernas adotaram totalmente os Módulos ES (`import`/`export`).

Se você também tiver scripts semelhantes, lembre-se de refatorá-los todos para Módulos ES. Essa é a direção em que o ecossistema está se movendo, e quanto mais cedo você fizer a mudança, menos problemas terá mais tarde.

```javascript
// Old way
const fs = require('fs');

// New way
import fs from 'fs';
```

### 3. Não se esqueça de SEO e redirecionamentos

Os motores de busca já indexaram as antigas páginas do Docusaurus do HagiCode. Se você mudar diretamente para o Astro e a estrutura do URL mudar, você poderá acabar com um grande número de erros 404 e uma grande queda na classificação de pesquisa.

Configuramos regras de redirecionamento no Astro:

```javascript
// astro.config.mjs
export default defineConfig({
  redirects: {
    '/docs/old-path': '/docs/new-path',
    // Map old links to new links in bulk
  }
});
```

Ou você pode lidar com isso na camada de configuração do servidor. Certifique-se de que links antigos possam ser redirecionados 301 para os novos endereços, porque isso é fundamental para o SEO.

## Resumo

Para o HagiCode, a migração do Docusaurus para o Astro não foi apenas uma atualização de framework. Foi também uma implementação prática de uma filosofia de “desempenho em primeiro lugar”.

**O que ganhamos:**
*   **Pontuações excepcionais do Lighthouse**: Após a migração, a pontuação de desempenho do site HagiCode facilmente se aproximou de uma pontuação perfeita.
*   **Velocidade de compilação mais rápida**: os recursos de compilação incremental do Astro reduzem pela metade o tempo de lançamento para atualizações de documentação.
*   **Flexibilidade preservada**: Com o Islands Architecture, não sacrificamos nenhum recurso interativo e ainda podemos usar o React quando necessário.

Se você também mantém um site orientado para documentação e está lutando com o tamanho do pacote ou velocidade de carregamento, vale a pena tentar o Astro. Embora o processo de migração exija alguma cirurgia, como renomear PCode para HagiCode e mover componentes um por um, a experiência de usuário suave que você obtém em troca faz com que valha a pena.

O sistema de construção compartilhado neste artigo é a abordagem exata que desenvolvemos por meio de tentativa e erro real durante a construção do HagiCode. Se você acha essa abordagem valiosa, isso diz algo sobre a força de nossa engenharia, e o próprio HagiCode provavelmente também merece uma olhada mais de perto.
