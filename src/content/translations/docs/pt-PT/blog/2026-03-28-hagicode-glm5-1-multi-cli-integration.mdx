## Guia de integração multi-CLI Hagicode e GLM-5.1

:::nota[Aviso de Contexto de Tempo]

Para o escopo atual sempre suportado, consulte [Instalação da CLI do Agente AI](/pt-PT/related-software-installation/ai-agent-cli/) e o [Guia de instalação](/pt-PT/installation); o `IFlowCli` mencionado aqui representa apenas uma configuração histórica da época e não faz parte do escopo atual de suporte ativo.

## Plano de fundo

No projeto Hagicode, os usuários podem escolher entre várias ferramentas CLI para conduzir assistentes de programação de IA, incluindo Claude Code CLI, GitHub Copilot, OpenCode CLI, Codebuddy CLI, Hermes CLI e muito mais. Essas ferramentas CLI são ferramentas de programação de IA de uso geral por si só, mas por meio da camada de abstração do Hagicode, elas podem se conectar de maneira flexível a diferentes provedores de modelos de IA.

Zhipu AI (ZAI) fornece uma interface compatível com a API Anthropic Claude, permitindo que essas ferramentas CLI usem diretamente modelos domésticos da série GLM. Entre eles, o GLM-5.1 é o mais recente lançamento de modelo de linguagem grande da Zhipu, com melhorias significativas em relação ao GLM-5.0.

## Arquitetura de abstração CLI do Hagicode

Hagicode define 11 tipos de provedores CLI através do `AIProviderType` enum, cobrindo as principais ferramentas CLI de programação de IA:

```csharp
public enum AIProviderType
{
    ClaudeCodeCli = 0,    // Claude Code CLI
    CodexCli = 1,          // GitHub Copilot Codex
    GitHubCopilot = 2,     // GitHub Copilot
    CodebuddyCli = 3,     // Codebuddy CLI
    OpenCodeCli = 4,      // OpenCode CLI
    IFlowCli = 5,         // IFlow CLI
    HermesCli = 6,        // Hermes CLI
    QoderCli = 7,         // Qoder CLI
    KiroCli = 8,          // Kiro CLI
    KimiCli = 9,          // Kimi CLI
    GeminiCli = 10,       // Gemini CLI
}
```

Cada CLI tem configuração de parâmetro de modelo correspondente e suporta o `model` e `reasoning` parâmetros:

```csharp
private static readonly IReadOnlyDictionary<AIProviderType, IReadOnlyList<string>> ManagedModelParameterKeysByProvider =
    new Dictionary<AIProviderType, IReadOnlyList<string>>
    {
        [AIProviderType.ClaudeCodeCli] = ["model", "reasoning"],
        [AIProviderType.CodexCli] = ["model", "reasoning"],
        [AIProviderType.OpenCodeCli] = ["model", "reasoning"],
        [AIProviderType.HermesCli] = ["model", "reasoning"],
        [AIProviderType.CodebuddyCli] = ["model", "reasoning"],
        [AIProviderType.QoderCli] = ["model", "reasoning"],
        [AIProviderType.KiroCli] = ["model", "reasoning"],
        [AIProviderType.GeminiCli] = ["model"],  // Gemini does not support the reasoning parameter
        // ...
    };
```

## Usando GLM-5.1 em CLIs diferentes

### 1. Código Claude CLI + GLM-5.1

Claude Code CLI é um dos principais CLIs do Hagicode e é configurado através do sistema de configuração Hero:

```json
{
  "primaryProfessionId": "profession-claude-code",
  "secondaryProfessionId": "secondary-glm-5-1",
  "model": "glm-5.1",
  "reasoning": "high"
}
```

Correspondente `HeroEquipmentCatalogItem` configuração:

```typescript
{
  id: 'secondary-glm-5-1',
  name: 'GLM 5.1',
  family: 'anthropic',
  kind: 'model',
  primaryFamily: 'claude',
  compatiblePrimaryFamilies: ['claude', 'codebuddy', 'hermes', 'qoder', 'kiro'],
  defaultParameters: {
    model: 'glm-5.1',
    reasoning: 'high'
  }
}
```

### 2. CLI OpenCode + GLM-5.1

OpenCode CLI é a CLI mais flexível e suporta a especificação de qualquer modelo no `provider/model` formato:

**Método 1: use o prefixo do provedor ZAI**
```json
{
  "primaryProfessionId": "profession-opencode",
  "model": "zai/glm-5.1",
  "reasoning": "high"
}
```

**Método 2: use o ID do modelo diretamente**
```json
{
  "model": "glm-5.1"
}
```

**Método 3: IU de configuração de front-end**

Em `HeroModelEquipmentForm.tsx`, o OpenCode CLI tem uma dica de espaço reservado dedicada:

```tsx
const OPEN_CODE_MODEL_PLACEHOLDER = 'myprovider/glm-4.7';

const modelPlaceholder = primaryProviderType === PCode_Models_AIProviderType.OPEN_CODE_CLI
  ? OPEN_CODE_MODEL_PLACEHOLDER
  : 'gpt-5.4';
```

Os usuários podem inserir:
```
zai/glm-5.1
glm-5.1
```

**Lógica de análise do modelo OpenCode CLI**:

```csharp
internal OpenCodeModelSelection? ResolveModelSelection(string? rawModel)
{
    var normalized = NormalizeOptionalValue(rawModel);
    if (normalized == null) return null;

    var slashIndex = normalized.IndexOf('/');
    if (slashIndex < 0)
    {
        // No slash: use the model ID directly
        return new OpenCodeModelSelection {
            ProviderId = string.Empty,
            ModelId = normalized,
        };
    }

    // Slash exists: parse the provider/model format
    var providerId = normalized[..slashIndex].Trim();
    var modelId = normalized[(slashIndex + 1)..].Trim();

    return new OpenCodeModelSelection {
        ProviderId = providerId,
        ModelId = modelId,
    };
}
```

### 3. Codebuddy CLI + GLM-5.1

Codebuddy CLI possui lógica de manipulação legada dedicada:

```json
{
  "primaryProfessionId": "profession-codebuddy",
  "model": "glm-5.1",
  "reasoning": "high"
}
```

**Nota**: Codebuddy mantém tratamento especial para GLM-5.0 e não usa normalização legada:

```csharp
return !string.Equals(providerName, "CodebuddyCli", StringComparison.OrdinalIgnoreCase)
       && string.Equals(normalizedModel, LegacyGlm5TurboModelValue, StringComparison.OrdinalIgnoreCase)
    ? Glm5TurboModelValue
    : normalizedModel;
// For CodebuddyCli, glm-5.0 is not normalized to glm-5-turbo
```

## Vantagens de melhoria do GLM-5.1

Comparado com o GLM-5.0, o GLM-5.1 traz as seguintes melhorias significativas:

### 1. Melhor capacidade de raciocínio

De acordo com as informações oficiais de lançamento da Zhipu, as melhorias no GLM-5.1 incluem:

- **Maior compreensão do código**: análise mais precisa de estruturas de código complexas
- **Maior compreensão do contexto**: Suporta contexto de conversação mais longo
- **Chamadas de ferramentas aprimoradas**: maior taxa de sucesso para chamadas de ferramentas MCP
- **Estabilidade de saída**: Reduz aleatoriedade e alucinações

### 2. Compatibilidade abrangente com vários CLI

GLM-5.1 cobre todas as CLIs convencionais suportadas pelo Hagicode:

```
compatiblePrimaryFamilies: [
  "claude",      // Claude Code CLI
  "codebuddy",   // Codebuddy CLI
  "hermes",      // Hermes CLI
  "qoder",       // Qoder CLI
  "kiro"         // Kiro CLI
]
```

## Notas

### 1. Configuração da chave API

Certifique-se de que `ANTHROPIC_AUTH_TOKEN` variável de ambiente está definida corretamente. É a credencial necessária para cada CLI se conectar ao modelo.

### 2. Disponibilidade do modelo

O GLM-5.1 precisa ser habilitado pelo provedor de modelo correspondente:
- A plataforma Zhipu AI ZAI oferece suporte por padrão
- Alibaba Cloud DashScope pode exigir um aplicativo separado

### 3. Formato CLI OpenCode

Ao usar o `provider/model` formato, certifique-se de que o ID do provedor esteja correto:
- IA Zhipu: `zai` ou `zhipuai`
- Nuvem Alibaba: `aliyun` ou `dashscope`

### 4. Parâmetro de raciocínio

- `high` é recomendado para obter os melhores resultados de geração de código
- Gemini CLI não suporta o parâmetro de raciocínio e irá ignorar esta configuração automaticamente

## Conclusão

Através de uma camada de abstração unificada, o Hagicode permite integração flexível entre GLM-5.1 e múltiplas CLIs. Os desenvolvedores podem escolher a ferramenta CLI que melhor se adapta às suas preferências e cenários de uso e, em seguida, usar o modelo GLM-5.1 mais recente por meio de configuração simples.

Como a versão mais recente do modelo Zhipu, o GLM-5.1 oferece melhorias claras em relação ao GLM-5.0:
- Um identificador de versão independente sem sobrecarga herdada
- Raciocínio mais forte e compreensão do código
- Ampla compatibilidade multi-CLI
- Configuração de nível de raciocínio flexível

Com as variáveis de ambiente corretas e o equipamento Hero configurado, os usuários podem desbloquear totalmente o poder do GLM-5.1 em diferentes ambientes CLI.
