## Adeus Script Hell: Por que escolhemos C# para um sistema de compilação moderno

> Um mergulho profundo em como o projeto HagiCode usa Nuke para construir um pipeline de construção automatizado de tipo seguro, multiplataforma e altamente extensível, abordando totalmente os pontos problemáticos de manutenção dos scripts de construção tradicionais.

## Plano de fundo

Ao longo da longa jornada de desenvolvimento de software, a palavra “construir” tende a inspirar amor e frustração. Amor, porque com um clique o código vira produto, e esse é um dos momentos mais gratificantes da programação. Frustração, porque manter uma pilha de scripts de construção confusos pode ser um pesadelo.

Em muitos projetos, estamos acostumados a escrever scripts em Python ou a confiar em arquivos de configuração XML. Imagine o medo de ser governado por `<property>` blocos. Mas à medida que a complexidade do projeto aumenta, especialmente em um projeto como o HagiCode, que abrange trabalho de front-end e back-end, múltiplas plataformas e vários idiomas, as abordagens tradicionais de construção começam a mostrar seus limites. Lógica de script dispersa, sem verificação de tipo, suporte IDE fraco... cada um desses problemas se torna uma pequena armadilha que confunde a equipe de vez em quando.

Para resolver esses problemas, o projeto HagiCode introduziu o **Nuke**, um sistema de compilação moderno baseado em C#. Não é apenas uma ferramenta. É uma maneira diferente de pensar sobre fluxos de trabalho de construção. Hoje veremos por que o escolhemos e como ele melhorou drasticamente nossa experiência de desenvolvimento.

## Análise central: Por que Nuke?

Você deve estar se perguntando: "Já existem tantos sistemas de compilação, como Make, Gradle ou até mesmo scripts Shell simples. Por que se esforçar para usar um sistema baseado em C#?"

Essa é uma pergunta justa. O principal apelo do Nuke é que ele traz os recursos da linguagem de programação que já conhecemos bem para o mundo dos scripts de construção.

### 1. Modularizando Fluxos de Trabalho de Construção: A Arte dos Alvos

Nuke tem uma filosofia de design muito clara: **tudo é um alvo**.

Em scripts tradicionais, é fácil acabar com centenas de linhas de lógica de execução linear emaranhadas. No Nuke, dividimos o processo de construção em processos independentes `Target`S. Cada alvo faz exatamente uma coisa, por exemplo:

- `Clean`: limpe o diretório de saída
- `Restore`: restaurar pacotes de dependência
- `Compile`: compilar o código
- `Test`: execute testes de unidade

Este design se alinha muito bem com o princípio da responsabilidade única. Tal como os blocos de construção, estes alvos podem ser combinados livremente. Mais importante ainda, o Nuke nos permite definir dependências entre alvos. Por exemplo, se você quiser executar `Test`, o sistema verificará automaticamente se `Compile` já foi executado. Se você quiser `Compile`, então `Restore` naturalmente vem primeiro.

Este gráfico de dependência torna a lógica muito mais clara e melhora significativamente a eficiência de execução, porque o Nuke pode analisar automaticamente o caminho de execução ideal.

### 2. Digite Segurança: Adeus ao pesadelo dos erros de digitação

Qualquer pessoa que tenha escrito scripts de construção em Python provavelmente já se deparou com esta situação embaraçosa: o script é executado por cinco minutos e depois falha porque `Confi.guration` foi digitado incorretamente ou porque uma string foi passada onde um parâmetro numérico era esperado.

A maior vantagem de escrever scripts de construção em C# é a **segurança de tipo**. Isso significa:

- **Verificações em tempo de compilação**: o IDE pode dizer o que há de errado enquanto você digita, em vez de fazer você esperar até o tempo de execução.
- **Refatoração segura**: se você quiser renomear uma variável ou método, o IDE pode lidar com isso em uma única etapa, sem forçá-lo a fazer uma pesquisa e substituição global arriscada.
- **Conclusão inteligente**: o poderoso IntelliSense pode completar o código automaticamente para você, para que você não precise ficar folheando a documentação para lembrar APIs obscuras.

### 3. Suporte multiplataforma: uma experiência de construção consistente

No passado, escrevemos `.bat` no Windows e `.sh` no Linux e talvez tenha adicionado um script Python para preencher a lacuna. Agora, onde quer que o .NET Core, ou .NET 5+ hoje, possa ser executado, o Nuke também pode ser executado.

Isso significa que os membros da equipe podem usar Windows, Linux ou macOS e, quer prefiram Visual Studio, VS Code ou Rider, todos executam exatamente a mesma lógica. Isso reduz bastante os problemas causados ​​por diferenças de ambiente, especialmente o clássico problema “funciona na minha máquina”.

### 4. Parâmetros e gerenciamento de configuração

Nuke fornece um mecanismo de análise de parâmetros muito elegante. Você não precisa analisar manualmente `string[] args`. Você simplesmente define uma propriedade e anota-a com `[Parameter]`, e o Nuke lida automaticamente com o mapeamento de argumentos de linha de comando e arquivos de configuração.

Por exemplo, podemos definir a configuração do build com muita facilidade:

```csharp
[Parameter("Configuration to build - Default is 'Debug'")]
readonly Configuration BuildConfiguration = IsLocalBuild ? Configuration.Debug : Configuration.Release;

Target Compile => _ => _
    .DependsOn(Restore)
    .Executes(() =>
    {
        // Use BuildConfiguration here; it is type-safe
        DotNetBuild(s => s
            .SetConfiguration(BuildConfiguration)
            .SetProjectFile(SolutionFile));
    });
```

Este estilo é intuitivo e resistente a erros.

## Guia prático: como colocar em prática em um projeto

A teoria por si só não nos leva a lugar nenhum. Vejamos como realmente aplicamos essa abordagem no projeto HagiCode.

### 1. Planeje a Estrutura do Projeto

Não queríamos scripts de construção sobrecarregando a raiz do projeto e também não queríamos uma estrutura de diretórios tão profunda que se assemelhasse a certos projetos Java. Então colocamos todos os arquivos de construção relacionados ao Nuke em um único `nukeBuild/` diretório.

Os benefícios são diretos:

- A raiz do projeto permanece limpa.
- A lógica de construção permanece coesa e mais fácil de gerenciar.
- Novos membros da equipe podem ver imediatamente: “É aqui que reside a lógica relacionada à construção”.

### 2. Projete uma cadeia de dependência de destino clara

Ao projetar Targets, seguimos um princípio: **atomicidade mais fluxo de dependência**.

Cada alvo deve ser pequeno o suficiente para fazer exatamente uma coisa. Por exemplo, `Clean` deve apenas remover arquivos e não deve manusear silenciosamente as embalagens nas laterais.

Um fluxo de dependência recomendado é mais ou menos assim:

`Clean` -> `Restore` -> `Compile` -> `Test` -> `Pack`

Claro, isso não é absoluto. Por exemplo, se você deseja apenas executar testes e não deseja empacotar nada, o Nuke permite executar `nuke Test` diretamente, e ele irá lidar automaticamente com o necessário `Restore` e `Compile` passos primeiro.

### 3. Tratamento e registro robusto de erros

Qual é a coisa mais frustrante na construção de scripts? Mensagens de erro pouco claras. Uma compilação com falha que registra apenas “Erro: 1” é suficiente para deixar qualquer um louco.

No Nuke, como podemos usar diretamente o tratamento de exceções C#, podemos capturar e relatar erros com muito maior precisão.

```csharp
Target Publish => _ => _
    .DependsOn(Test)
    .Executes(() =>
    {
        try 
        {
            // Try publishing to NuGet
            DotNetNuGetPush(s => s
                .SetTargetPath(ArtifactPath)
                .SetSource("https://api.nuget.org/v3/index.json")
                .SetApiKey(ApiKey));
        }
        catch (Exception ex)
        {
            Log.Error($"Publishing failed. Team, please check whether the key is correct: {ex.Message}");
            throw; // Make sure the build process exits with a non-zero exit code
        }
    });
```

### 4. Use testes para proteger a qualidade

Os scripts de construção também são códigos, o que significa que também precisam de testes. Nuke nos permite escrever testes para fluxos de trabalho de construção para que, quando alterarmos a lógica de construção, não interrompamos acidentalmente os processos de lançamento existentes. Isto é especialmente importante em pipelines de integração contínua.

## Resumo

Ao apresentar o Nuke, o processo de construção do HagiCode tornou-se mais suave do que nunca. Esta não foi apenas uma substituição de ferramenta. Foi uma atualização na mentalidade da engenharia.

**O que ganhamos?**

- **Manutenção**: configuração como código, lógica mais clara e um caminho de integração mais rápido para novos colaboradores.
- **Estabilidade**: digitação forte elimina mais de 90% dos erros de baixo nível.
- **Consistência**: uma experiência unificada entre plataformas que elimina desvios de ambiente.

Se escrever scripts de construção costumava ser como "tatear o caminho no escuro", então usar o Nuke é como "caminhar à noite com as luzes acesas". Se você está cansado de manter scripts difíceis de depurar, considere migrar sua lógica de construção para o mundo do C#. Você pode descobrir que as construções também podem ser elegantes.
