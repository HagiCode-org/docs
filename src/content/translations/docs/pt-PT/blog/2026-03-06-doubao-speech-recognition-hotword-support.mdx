## Guia para implementar suporte a hotword para reconhecimento de fala Doubao

> Este artigo explica em detalhes como implementar suporte a hotword para reconhecimento de fala Doubao no projeto HagiCode. Ao usar hotwords personalizados e tabelas de hotwords da plataforma, você pode melhorar significativamente a precisão do reconhecimento de vocabulário específico do domínio.

## Plano de fundo

A tecnologia de reconhecimento de fala vem se desenvolvendo há muitos anos, mas um problema tem incomodado consistentemente os desenvolvedores. Os modelos de reconhecimento de fala de uso geral podem abranger a linguagem cotidiana, mas muitas vezes ficam aquém quando se trata de terminologia profissional, nomes de produtos e nomes pessoais. Pense nisso: um assistente de voz na área médica precisa reconhecer com precisão termos como “hipertensão”, “diabetes” e “doença coronariana”; um sistema jurídico precisa capturar com precisão termos como “causa da ação”, “defesa” e “ônus da prova”. Nesses cenários, um modelo de uso geral tenta o seu melhor, mas muitas vezes isso não é suficiente.

Enfrentamos o mesmo desafio no projeto HagiCode. Como assistente multifuncional de codificação de IA, o HagiCode precisa lidar com o reconhecimento de fala para uma ampla gama de terminologia técnica. No entanto, a API de reconhecimento de fala Doubao, em sua configuração padrão, não atendeu totalmente aos nossos requisitos de precisão para termos especializados. Não é que Doubao não seja bom o suficiente; em vez disso, cada domínio tem seu próprio sistema terminológico. Após alguma pesquisa e exploração técnica, descobrimos que a API de reconhecimento de fala Doubao, na verdade, fornece suporte a hotwords. Com uma configuração simples, pode melhorar significativamente a precisão do reconhecimento de vocabulário específico. De certa forma, uma vez que você lhe diz em quais palavras prestar atenção, ele as escuta com mais atenção.

O que este artigo compartilha é a solução completa que usamos no projeto HagiCode para implementar hotwords de reconhecimento de fala Doubao. Ambos os modos, hotwords personalizados e tabelas de hotwords de plataforma, estão disponíveis e também podem ser combinados. Com esta solução, os desenvolvedores podem configurar hotwords de maneira flexível com base em cenários de negócios para que o sistema de reconhecimento de fala possa “reconhecer” melhor vocabulário profissional, incomum, mas crítico.

## Implementação principal

### Compreendendo os dois modos de hotword

A API de reconhecimento de fala Doubao oferece duas maneiras de configurar hotwords, e cada uma tem seus próprios casos de uso e vantagens ideais.

**Modo hotword personalizado** nos permite passar o texto da hotword diretamente pelo `corpus.context` campo. Essa abordagem é especialmente adequada para cenários em que você precisa configurar rapidamente um pequeno número de hotwords, como reconhecer temporariamente o nome de um produto ou o nome de uma pessoa. Na implementação do HagiCode, analisamos o texto da hotword multilinha inserido pelo usuário em uma lista de strings e, em seguida, formatamos no formato `context_data` matriz exigida pela API Doubao. Essa abordagem é muito direta: basta dizer ao sistema em quais palavras prestar atenção, e ele faz exatamente isso.

**Modo de tabela de hotword da plataforma** usa o `corpus.boosting_table_id` campo para fazer referência a uma tabela de hotword pré-configurada na plataforma de autoaprendizagem Doubao. Essa abordagem é adequada para cenários em que você precisa gerenciar um grande número de hotwords. Podemos criar e manter tabelas de hotwords na plataforma de autoaprendizagem Doubao e depois referenciá-las por ID. Para um projeto como o HagiCode, onde termos especializados precisam ser continuamente atualizados e mantidos, este modo oferece uma capacidade de gerenciamento muito melhor. À medida que o número de hotwords aumenta, ter um local centralizado para gerenciá-las é muito melhor do que inseri-las manualmente todas as vezes.

Curiosamente, estes dois modos também podem ser usados em conjunto. A API Doubao suporta a inclusão de hotwords personalizadas e um ID de tabela de hotwords de plataforma na mesma solicitação, com a estratégia de combinação controlada pelo `combine_mode` parâmetro. Essa flexibilidade permite que o HagiCode lide com uma ampla gama de necessidades complexas de reconhecimento de terminologia profissional. Às vezes, combinar múltiplas abordagens produz melhores resultados.

### Definições e validação de tipo de front-end

Na implementação frontend do HagiCode, definimos um conjunto completo de tipos de configuração de hotword e lógica de validação. A primeira parte é a definição do tipo:

```typescript
export interface HotwordConfig {
  contextText: string;           // Multi-line hotword text
  boostingTableId: string;      // Doubao platform hotword table ID
  combineMode: boolean;          // Whether to use both together
}
```

Esta interface simples contém todos os itens de configuração do recurso hotword. Entre eles, `contextText` é a parte com a qual os usuários interagem mais diretamente: permitimos que os usuários insiram uma frase de hotword por linha, o que é muito intuitivo. Pedir aos usuários que insiram um termo por linha é muito mais fácil do que fazê-los entender um formato de configuração complicado.

Em seguida vem a função de validação. Com base nas limitações da API Doubao, definimos regras de validação rígidas: no máximo 100 linhas de texto de hotword, até 50 caracteres por linha e não mais que 5.000 caracteres no total; `boosting_table_id` pode ter no máximo 200 caracteres e conter apenas letras, números, sublinhados e hifens. Estes limites não são arbitrários; eles vêm diretamente da documentação oficial do Doubao. Os limites da API são limites da API e temos que segui-los.

```typescript
export function validateContextText(contextText: string): HotwordValidationResult {
  if (!contextText || contextText.trim().length === 0) {
    return { isValid: true, errors: [] };
  }

  const lines = contextText.split('\n').filter(line => line.trim().length > 0);
  const errors: string[] = [];

  if (lines.length > 100) {
    errors.push(`Hotword line count cannot exceed 100 lines; current count is ${lines.length}`);
  }

  const totalChars = contextText.length;
  if (totalChars > 5000) {
    errors.push(`Total hotword character count cannot exceed 5000; current count is ${totalChars}`);
  }

  for (let i = 0; i < lines.length; i++) {
    if (lines[i].length > 50) {
      errors.push(`Hotword on line ${i + 1} exceeds the 50-character limit`);
    }
  }

  return { isValid: errors.length === 0, errors };
}

export function validateBoostingTableId(boostingTableId: string): HotwordValidationResult {
  if (!boostingTableId || boostingTableId.trim().length === 0) {
    return { isValid: true, errors: [] };
  }

  const errors: string[] = [];

  if (boostingTableId.length > 200) {
    errors.push(`boosting_table_id cannot exceed 200 characters; current count is ${boostingTableId.length}`);
  }

  if (!/^[a-zA-Z0-9_-]+$/.test(boostingTableId)) {
    errors.push('boosting_table_id can contain only letters, numbers, underscores, and hyphens');
  }

  return { isValid: errors.length === 0, errors };
}
```

Essas funções de validação são executadas imediatamente quando o usuário configura hotwords, garantindo que os problemas sejam detectados o mais cedo possível. Do ponto de vista da experiência do usuário, esse tipo de feedback instantâneo é muito importante. É sempre melhor que os usuários saibam o que está errado enquanto digitam, e não depois de enviar.

### Persistência de configuração de front-end

Na implementação frontend do HagiCode, optamos por utilizar o navegador `localStorage` para armazenar a configuração do hotword. Houve várias considerações por trás dessa decisão de design. Primeiro, a configuração do hotword é altamente personalizada e diferentes usuários podem ter diferentes necessidades específicas de domínio. Em segundo lugar, esta abordagem simplifica a implementação de back-end porque não requer tabelas de banco de dados extras ou pontos de extremidade de API. Finalmente, depois que os usuários o configuram uma vez no navegador, as configurações podem ser carregadas automaticamente nos usos subsequentes, o que é muito conveniente. Simplificando, é a abordagem mais fácil.

```typescript
const HOTWORD_STORAGE_KEYS = {
  contextText: 'hotword-context-text',
  boostingTableId: 'hotword-boosting-table-id',
  combineMode: 'hotword-combine-mode',
} as const;

export const DEFAULT_HOTWORD_CONFIG: HotwordConfig = {
  contextText: '',
  boostingTableId: '',
  combineMode: false,
};

// Load hotword configuration
export function loadHotwordConfig(): HotwordConfig {
  const contextText = localStorage.getItem(HOTWORD_STORAGE_KEYS.contextText) || '';
  const boostingTableId = localStorage.getItem(HOTWORD_STORAGE_KEYS.boostingTableId) || '';
  const combineMode = localStorage.getItem(HOTWORD_STORAGE_KEYS.combineMode) === 'true';

  return { contextText, boostingTableId, combineMode };
}

// Save hotword configuration
export function saveHotwordConfig(config: HotwordConfig): void {
  localStorage.setItem(HOTWORD_STORAGE_KEYS.contextText, config.contextText);
  localStorage.setItem(HOTWORD_STORAGE_KEYS.boostingTableId, config.boostingTableId);
  localStorage.setItem(HOTWORD_STORAGE_KEYS.combineMode, String(config.combineMode));
}
```

A lógica neste código é direta e clara. Nós lemos de `localStorage` ao carregar a configuração e escrever para `localStorage` ao salvá-lo. Também fornecemos uma configuração padrão para que o sistema ainda possa funcionar corretamente quando ainda não existir nenhuma configuração. Afinal, tem que haver um padrão sensato.

### Extensão de configuração do SDK de back-end

Na implementação de back-end do HagiCode, precisávamos adicionar propriedades relacionadas ao hotword à classe de configuração do SDK. Levando em consideração as características da linguagem C# e os padrões de uso, usamos `List<string>` para armazenar contextos de hotword personalizados:

```csharp
public class DoubaoVoiceConfig
{
    /// <summary>
    /// App ID
    /// </summary>
    public string AppId { get; set; } = string.Empty;

    /// <summary>
    /// Access token
    /// </summary>
    public string AccessToken { get; set; } = string.Empty;

    /// <summary>
    /// Service URL
    /// </summary>
    public string ServiceUrl { get; set; } = string.Empty;

    /// <summary>
    /// Custom hotword context list
    /// </summary>
    public List<string>? HotwordContexts { get; set; }

    /// <summary>
    /// Doubao platform hotword table ID
    /// </summary>
    public string? BoostingTableId { get; set; }
}
```

O design desta classe de configuração segue o estilo conciso usual do HagiCode. `HotwordContexts` é um tipo de lista anulável e `BoostingTableId` é uma sequência anulável, portanto, quando não há configuração de hotword, essas propriedades não têm nenhum efeito na solicitação. Se você não estiver usando o recurso, ele deve ficar fora do caminho.

### Lógica de construção de carga útil

A construção da carga útil é o núcleo de todo o recurso de hotword. Assim que tivermos a configuração do hotword, precisamos formatá-lo na estrutura JSON exigida pela API Doubao. Este processo acontece antes do SDK enviar a solicitação:

```csharp
private void AddCorpusToRequest(Dictionary<string, object> request)
{
    var corpus = new Dictionary<string, object>();

    // Add custom hotwords
    if (Config.HotwordContexts != null && Config.HotwordContexts.Count > 0)
    {
        corpus["context"] = new Dictionary<string, object>
        {
            ["context_type"] = "dialog_ctx",
            ["context_data"] = Config.HotwordContexts
                .Select(text => new Dictionary<string, object> { ["text"] = text })
                .ToList()
        };
    }

    // Add platform hotword table ID
    if (!string.IsNullOrEmpty(Config.BoostingTableId))
    {
        corpus["boosting_table_id"] = Config.BoostingTableId;
    }

    // Add corpus to the request only when it is not empty
    if (corpus.Count > 0)
    {
        request["corpus"] = corpus;
    }
}
```

Este código mostra como construir dinamicamente o `corpus` campo com base na configuração. O ponto chave é que adicionamos o `corpus` campo somente quando a configuração do hotword realmente existir. Este design garante compatibilidade com versões anteriores: quando nenhuma hotword é configurada, a estrutura da solicitação permanece exatamente a mesma de antes. A compatibilidade com versões anteriores é importante; adicionar um recurso não deve interromper a lógica existente.

### Passagem de parâmetros WebSocket

Entre o frontend e o backend, os parâmetros de hotword são passados através de mensagens de controle WebSocket. O HagiCode foi projetado para que quando o frontend iniciar a gravação, ele carregue a configuração do hotword do `localStorage` e envia para o backend por meio de uma mensagem WebSocket.

```typescript
const controlMessage = {
  type: 'control',
  payload: {
    command: 'StartRecognition',
    contextText: '高血压\n糖尿病\n冠心病',
    boosting_table_id: 'medical_table',
    combineMode: false
  }
};
```

Há um detalhe a ser observado aqui: o frontend passa texto de várias linhas separadas por caracteres de nova linha e o backend precisa analisá-lo. O manipulador WebSocket de back-end analisa esses parâmetros e os passa para o SDK:

```csharp
private async Task HandleControlMessageAsync(
    string connectionId,
    DoubaoSession session,
    ControlMessage message)
{
    if (message.Payload is SessionControlRequest controlRequest)
    {
        // Parse hotword parameters
        string? contextText = controlRequest.ContextText;
        string? boostingTableId = controlRequest.BoostingTableId;
        bool? combineMode = controlRequest.CombineMode;

        // Parse multi-line text into a hotword list
        if (!string.IsNullOrEmpty(contextText))
        {
            var hotwords = contextText
                .Split('\n', StringSplitOptions.RemoveEmptyEntries)
                .Select(s => s.Trim())
                .Where(s => s.Length > 0)
                .ToList();

            session.HotwordContexts = hotwords;
        }

        session.BoostingTableId = boostingTableId;
    }
}
```

Com esse design, passar a configuração do hotword do frontend para o backend torna-se claro e eficiente. Não há nada de especialmente misterioso nisso; os dados são simplesmente passados ​​camada por camada.

## Guia Prático

### Configurar palavras-chave personalizadas

No uso real, configurar hotwords personalizados é muito simples. Abra a página de configurações de reconhecimento de fala no HagiCode e encontre a seção "Configuração de Hotword". Na caixa de entrada "Texto de hotword personalizado", insira uma frase de hotword por linha.

Por exemplo, se você estiver desenvolvendo um aplicativo médico, poderá configurá-lo assim:

```
高血压
糖尿病
冠心病
心绞痛
心肌梗死
心力衰竭
```

Depois de salvar a configuração, essas hotwords são passadas automaticamente para a API Doubao sempre que o reconhecimento de fala é iniciado. Em nossos testes, depois que as hotwords foram configuradas, a precisão do reconhecimento de termos profissionais relacionados melhorou visivelmente. A melhoria é real e claramente melhor do que antes.

### Configurar uma tabela de hotwords de plataforma

Se você precisar gerenciar um grande número de hotwords ou se as hotwords precisarem de atualizações frequentes, o modo de tabela de hotwords da plataforma é mais adequado. Primeiro, crie uma tabela de hotwords na plataforma de autoaprendizagem Doubao e obtenha o gerado `boosting_table_id`e insira esse ID na página de configurações do HagiCode.

A plataforma de autoaprendizagem Doubao oferece recursos como importação em massa e gerenciamento categorizado de hotwords, o que é muito prático para equipes que precisam gerenciar grandes conjuntos de terminologia especializada. Ao gerenciar hotwords na plataforma, você pode mantê-las centralmente e implementar atualizações de forma consistente. Uma vez que a lista de hotwords se torna grande, ter um único local para gerenciá-la é muito mais prático do que inseri-la manualmente sempre.

### Usando o modo de combinação

Em alguns cenários complexos, pode ser necessário usar hotwords personalizados e uma tabela de hotwords de plataforma ao mesmo tempo. Nesse caso, basta configurar ambos no HagiCode e habilitar a chave "Modo Combinação".

No modo de combinação, a API Doubao considera ambas as fontes de hotword ao mesmo tempo, portanto, a precisão do reconhecimento é geralmente maior do que usar qualquer uma das fontes sozinha. No entanto, vale a pena notar que o modo combinado aumenta a complexidade da solicitação, por isso é melhor decidir se deve ser habilitado após testes práticos. Mais complexidade só vale a pena se os resultados do mundo real a justificarem.

### Exemplos de integração de código

Integrar o recurso hotword ao projeto HagiCode é muito simples. Aqui estão alguns trechos de código comumente usados:

```typescript
import {
  loadHotwordConfig,
  saveHotwordConfig,
  validateHotwordConfig,
  parseContextText,
  getEffectiveHotwordMode,
  type HotwordConfig
} from '@/types/hotword';

// Load and validate configuration
const config = loadHotwordConfig();
const validation = validateHotwordConfig(config);

if (!validation.isValid) {
  console.error('Hotword configuration validation failed:', validation.errors);
  return;
}

// Parse hotword text
const hotwords = parseContextText(config.contextText);
console.log('Parsed hotwords:', hotwords);

// Get effective hotword mode
const mode = getEffectiveHotwordMode(config);
console.log('Current hotword mode:', mode);
```

## Coisas para manter em mente

Existem vários pontos que merecem atenção especial na implementação e utilização do recurso hotword.

O primeiro é o limite de caracteres. A API Doubao tem restrições estritas sobre hotwords, incluindo contagem de linhas, caracteres por linha e contagem total de caracteres. Se algum limite for excedido, a API retornará um erro. Na implementação frontend do HagiCode, verificamos essas restrições durante a entrada do usuário por meio de funções de validação, o que evita que configurações inválidas sejam enviadas para o backend. Detectar problemas antecipadamente é sempre melhor do que esperar que a API falhe.

Em segundo lugar está o formato de `boosting_table_id`. Este campo permite apenas letras, números, sublinhados e hifens e não pode conter espaços ou outros caracteres especiais. Ao criar uma tabela de hotwords na plataforma de autoaprendizagem Doubao, certifique-se de seguir as regras de nomenclatura. Esse tipo de validação estrita de formato é comum para APIs.

O terceiro é a compatibilidade com versões anteriores. Os parâmetros de hotword são totalmente opcionais. Se nenhuma hotword estiver configurada, o sistema se comportará exatamente como antes. Esse design garante que os usuários existentes não sejam afetados de forma alguma e também facilita a migração e as atualizações graduais. Adicionar um recurso não deve atrapalhar a lógica anterior.

Finalmente, há tratamento de erros. Quando a configuração do hotword é inválida, a API Doubao retorna mensagens de erro correspondentes. A implementação do HagiCode registra logs detalhados para ajudar os desenvolvedores a solucionar problemas. Ao mesmo tempo, o frontend exibe erros de validação na IU para ajudar os usuários a corrigir a configuração. Um bom tratamento de erros leva naturalmente a uma melhor experiência do usuário.
