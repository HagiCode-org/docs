## Guia completo para análise de mensagens do Codex SDK Console

> Este artigo explica o mecanismo de fluxo de eventos do Codex SDK, análise de tipo de mensagem e práticas recomendadas em projetos reais, ajudando os desenvolvedores a dominar rapidamente as habilidades básicas por trás dos serviços de execução de IA.

## Plano de fundo

Ao construir um serviço de execução de IA baseado no Codex SDK, inevitavelmente nos deparamos com uma questão prática: como devemos lidar com as mensagens de eventos transmitidas retornadas pelo Codex? Essas mensagens contêm informações importantes, como status de execução, conteúdo de saída e detalhes de erros, portanto merecem um tratamento cuidadoso.

Como parte do projeto HagiCode, precisávamos de um executor confiável para cenários de assistente de codificação de IA. É exatamente por isso que decidimos estudar a fundo o mecanismo de fluxo de eventos do Codex SDK. Afinal, somente compreendendo como funcionam as mensagens subjacentes poderemos construir uma plataforma de execução de IA verdadeiramente empresarial.

O Codex SDK é um SDK de assistência de programação lançado pela OpenAI. Ele retorna resultados de execução por meio de um Event Stream. Ao contrário do modelo tradicional de solicitação-resposta, o Codex usa eventos transmitidos para que possamos:

- Obtenha o progresso da execução em tempo real
- Lidar com erros prontamente
- Obtenha estatísticas detalhadas de uso de token
- Suporta tarefas complexas de longa duração

Compreender esses tipos de eventos e analisá-los corretamente é essencial para implementar um executor de IA totalmente capaz. Afinal, ninguém quer trabalhar com caixa preta.

## Mecanismo de fluxo de eventos

### Conceitos Básicos

O Codex SDK usa o `thread.runStreamed()` método para retornar um iterador de evento assíncrono:

```typescript
import { Codex } from '@openai/codex-sdk';

const client = new Codex({
  apiKey: process.env.CODEX_API_KEY,
  baseUrl: process.env.CODEX_BASE_URL,
});

const thread = client.startThread({
  workingDirectory: '/path/to/project',
  skipGitRepoCheck: false,
});

const { events } = await thread.runStreamed('your prompt here', {
  outputSchema: {
    type: 'object',
    properties: {
      output: { type: 'string' },
      status: { type: 'string', enum: ['ok', 'action_required'] },
    },
    required: ['output', 'status'],
  },
});

for await (const event of events) {
  // Handle each event
}
```

### Explicação detalhada dos tipos de eventos

| Tipo de Evento | Descrição | Dados principais |
|---------|------|----------|
| `thread.started` | Tópico iniciado com sucesso | `thread_id` |
| `item.updated` | Conteúdo da mensagem atualizado | `item.text` |
| `item.completed` | Mensagem concluída | `item.text` |
| `turn.completed` | Execução concluída | `usage` (uso de token) |
| `turn.failed` | Falha na execução | `error.message` |
| `error` | Evento de erro | `message` |

Em projetos reais, o componente executor do HagiCode é construído sobre esses tipos de eventos. Precisamos lidar com cada tipo de evento com cuidado para garantir uma experiência tranquila do usuário. Bons sistemas são construídos levando-se os detalhes a sério.

## Implementação de análise de mensagens

### Extraindo o conteúdo da mensagem

O conteúdo da mensagem é extraído por meio de um manipulador de eventos:

```typescript
private handleThreadEvent(event: ThreadEvent, onMessage: (content: string) => void): void {
  // Only handle message update and completion events
  if (event.type !== 'item.updated' && event.type !== 'item.completed') {
    return;
  }

  // Only handle agent message content
  if (event.item.type !== 'agent_message') {
    return;
  }

  // Extract text content
  onMessage(event.item.text);
}
```

Pontos principais:
- Manuseie apenas `item.updated` e `item.completed` eventos
- Lidar apenas com conteúdo do tipo `agent_message`
- O conteúdo da mensagem está no `event.item.text` campo

### Análise de saída estruturada

Codex suporta saída estruturada JSON. Você pode especificar o formato de retorno através do `outputSchema` parâmetro:

```typescript
const DEFAULT_OUTPUT_SCHEMA = {
  type: 'object',
  properties: {
    output: { type: 'string' },
    status: { type: 'string', enum: ['ok', 'action_required'] },
  },
  required: ['output', 'status'],
  additionalProperties: false,
} as const;
```

A função de análise tenta analisar o JSON e, se isso falhar, ela retorna ao texto bruto.

```typescript
function toStructuredOutput(raw: string): StructuredOutput {
  try {
    const parsed = JSON.parse(raw) as Partial<StructuredOutput>;
    if (typeof parsed.output === 'string') {
      return {
        output: parsed.output,
        status: parsed.status === 'action_required' ? 'action_required' : 'ok',
      };
    }
  } catch {
    // JSON parsing failed, fall back to the raw text
  }

  return {
    output: raw,
    status: 'ok',
  };
}
```

### Fluxo completo de tratamento de eventos

```typescript
private async runWithStreaming(
  thread: Thread,
  input: CodexStageExecutionInput
): Promise<{ output: string; usage: Usage | null }> {
  const abortController = new AbortController();
  const timeoutHandle = setTimeout(() => {
    abortController.abort();
  }, Math.max(1000, input.timeoutMs));

  let latestMessage = '';
  let usage: Usage | null = null;
  let emittedLength = 0;

  try {
    const { events } = await thread.runStreamed(input.prompt, {
      outputSchema: DEFAULT_OUTPUT_SCHEMA,
      signal: abortController.signal,
    });

    for await (const event of events) {
      // Handle message content
      this.handleThreadEvent(event, (nextContent) => {
        const delta = nextContent.slice(emittedLength);
        if (delta.length > 0) {
          emittedLength = nextContent.length;
          input.callbacks?.onChunk?.(delta);  // Streaming callback
        }
        latestMessage = nextContent;
      });

      // Process different data based on the event type
      if (event.type === 'thread.started') {
        this.threadId = event.thread_id;
      } else if (event.type === 'turn.completed') {
        usage = event.usage;
      } else if (event.type === 'turn.failed') {
        throw new CodexExecutorError('gateway_unavailable', event.error.message, true);
      } else if (event.type === 'error') {
        throw new CodexExecutorError('gateway_unavailable', event.message, true);
      }
    }
  } catch (error) {
    if (abortController.signal.aborted) {
      throw new CodexExecutorError(
        'upstream_timeout',
        `Codex stage timed out after ${input.timeoutMs}ms`,
        true
      );
    }
    throw error;
  } finally {
    clearTimeout(timeoutHandle);
  }

  const structured = toStructuredOutput(latestMessage);
  return { output: structured.output, usage };
}
```

## Estratégia de tratamento de erros

### Mapeamento de código de erro

Mapeie padrões de erro específicos para códigos de erro concretos para que as camadas superiores possam lidar com eles mais facilmente:

```typescript
function mapError(error: unknown): CodexExecutorError {
  if (error instanceof CodexExecutorError) {
    return error;
  }

  const message = error instanceof Error ? error.message : String(error);
  const normalized = message.toLowerCase();

  // Authentication errors - not retryable
  if (normalized.includes('401') ||
      normalized.includes('403') ||
      normalized.includes('api key') ||
      normalized.includes('auth')) {
    return new CodexExecutorError('auth_invalid', message, false);
  }

  // Rate limit errors - retryable
  if (normalized.includes('429') || normalized.includes('rate limit')) {
    return new CodexExecutorError('rate_limited', message, true);
  }

  // Timeout errors - retryable
  if (normalized.includes('timeout') || normalized.includes('aborted')) {
    return new CodexExecutorError('upstream_timeout', message, true);
  }

  // Default error
  return new CodexExecutorError('gateway_unavailable', message, true);
}
```

### Definições de tipo de erro

```typescript
export type CodexErrorCode =
  | 'auth_invalid'      // Authentication failure
  | 'upstream_timeout'  // Upstream timeout
  | 'rate_limited'      // Rate limited
  | 'gateway_unavailable'; // Gateway unavailable

export class CodexExecutorError extends Error {
  readonly code: CodexErrorCode;
  readonly retryable: boolean;

  constructor(code: CodexErrorCode, message: string, retryable: boolean) {
    super(message);
    this.name = 'CodexExecutorError';
    this.code = code;
    this.retryable = retryable;
  }
}
```

## Diretório de trabalho e configuração do ambiente

### Validação do diretório de trabalho

O Codex SDK exige que o diretório de trabalho seja um repositório Git válido.

```typescript
export function validateWorkingDirectory(
  workingDirectory: string,
  skipGitRepoCheck: boolean
): void {
  const resolvedWorkingDirectory = path.resolve(workingDirectory);

  if (!existsSync(resolvedWorkingDirectory)) {
    throw new CodexExecutorError(
      'gateway_unavailable',
      'Working directory does not exist.',
      false
    );
  }

  if (!statSync(resolvedWorkingDirectory).isDirectory()) {
    throw new CodexExecutorError(
      'gateway_unavailable',
      'Working directory is not a directory.',
      false
    );
  }

  if (skipGitRepoCheck) {
    return;
  }

  const gitDir = path.join(resolvedWorkingDirectory, '.git');
  if (!existsSync(gitDir)) {
    throw new CodexExecutorError(
      'gateway_unavailable',
      'Working directory is not a git repository.',
      false
    );
  }
}
```

### Carregando variáveis de ambiente

O Codex SDK precisa carregar variáveis de ambiente do shell de login para que o Agente AI possa acessar os comandos do sistema:

```typescript
function parseEnvironmentOutput(output: Buffer): Record<string, string> {
  const parsed: Record<string, string> = {};

  for (const entry of output.toString('utf8').split('\0')) {
    if (!entry) continue;

    const separatorIndex = entry.indexOf('=');
    if (separatorIndex <= 0) continue;

    const key = entry.slice(0, separatorIndex);
    const value = entry.slice(separatorIndex + 1);
    if (key.length > 0) {
      parsed[key] = value;
    }
  }

  return parsed;
}

function tryLoadEnvironmentFromShell(shellPath: string): Record<string, string> | null {
  const result = spawnSync(shellPath, ['-ilc', 'env -0'], {
    env: process.env,
    stdio: ['ignore', 'pipe', 'pipe'],
    timeout: 5000,
  });

  if (result.error || result.status !== 0) {
    return null;
  }

  return parseEnvironmentOutput(result.stdout);
}

export function createExecutorEnvironment(
  envOverrides: Record<string, string> = {}
): Record<string, string> {
  // Load environment variables from the login shell
  const consoleEnv = loadConsoleEnvironmentFromShell();

  return {
    ...process.env,
    ...consoleEnv,
    ...envOverrides,
  };
}
```

## Exemplo de uso completo

### Uso Básico

No projeto HagiCode, usamos a seguinte abordagem para inicializar o cliente Codex e executar tarefas:

```typescript
import { Codex } from '@openai/codex-sdk';

async function executeWithCodex(prompt: string, workingDir: string) {
  const client = new Codex({
    apiKey: process.env.CODEX_API_KEY,
    env: { PATH: process.env.PATH },
  });

  const thread = client.startThread({
    workingDirectory: workingDir,
  });

  const { events } = await thread.runStreamed(prompt);

  let result = '';
  for await (const event of events) {
    if (event.type === 'item.updated' && event.item.type === 'agent_message') {
      result = event.item.text;
    }
    if (event.type === 'turn.completed') {
      console.log('Token usage:', event.usage);
    }
  }

  // Try to parse JSON output
  try {
    const parsed = JSON.parse(result);
    return parsed.output;
  } catch {
    return result;
  }
}
```

### Implementação completa com novas tentativas

```typescript
export class CodexSdkExecutor {
  private readonly config: CodexRuntimeConfig;
  private readonly client: Codex;
  private threadId: string | null = null;

  async executeStage(input: CodexStageExecutionInput): Promise<CodexStageExecutionResult> {
    const maxAttempts = Math.max(1, this.config.retryCount + 1);
    let attempt = 0;
    let lastError: CodexExecutorError | null = null;

    while (attempt < maxAttempts) {
      attempt += 1;

      try {
        const thread = this.getThread(input.workingDirectory);
        const { output, usage } = await this.runWithStreaming(thread, input);

        return {
          output,
          usage,
          threadId: this.threadId!,
          attempts: attempt,
          latencyMs: Date.now() - startedAt,
        };
      } catch (error) {
        const mappedError = mapError(error);
        lastError = mappedError;

        // Non-retryable error or max retry attempts reached
        if (!mappedError.retryable || attempt >= maxAttempts) {
          throw mappedError;
        }

        // Wait before retrying
        await new Promise(resolve => setTimeout(resolve, 1000 * attempt));
      }
    }

    throw lastError!;
  }
}
```

## Melhores práticas

### 1. Requisitos do diretório de trabalho

- Certifique-se de que o diretório de trabalho seja um repositório Git válido
- Use o `PROJECT_ROOT` variável de ambiente para especificá-la explicitamente
- Durante a depuração, você pode definir `CODEX_SKIP_GIT_REPO_CHECK=true` pular a verificação

### 2. Configuração de variável de ambiente

- Passe apenas as variáveis de ambiente necessárias por meio de um mecanismo de lista de permissões
- Use o shell de login para carregar o ambiente completo
- Evite passar informações sensíveis

### 3. Estratégia de tempo limite e nova tentativa

- Defina tempos limite razoáveis com base na complexidade da tarefa
- Implementar espera exponencial para erros repetíveis
- Registrar contagens e motivos de novas tentativas

### 4. Tratamento de erros

- Distinguir entre erros repetíveis e não repetíveis
- Forneça mensagens de erro e sugestões claras
- Use códigos de erro unificados para que as camadas superiores possam lidar com eles de forma consistente

### 5. Saída de streaming

- Implemente retornos de chamada de saída incrementais para melhorar a experiência do usuário
- Lidar corretamente com atualizações incrementais de mensagens
- Registre o uso de token para análise de custos
