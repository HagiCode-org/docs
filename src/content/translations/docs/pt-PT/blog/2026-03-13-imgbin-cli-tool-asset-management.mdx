## Design da ferramenta ImgBin CLI: abordagem de gerenciamento de ativos de imagem da HagiCode

> Este artigo explica como construir um pipeline de ativos de imagem automatizável do zero, abrangendo o design da ferramenta CLI, uma arquitetura de adaptador de provedor e estratégias de gerenciamento de metadados.

## Plano de fundo

Honestamente, eu não esperava que o gerenciamento de ativos de imagem nos mantivesse confusos por tanto tempo.

Durante o desenvolvimento do HagiCode, nos deparamos com um problema que parecia simples na superfície, mas era surpreendentemente espinhoso na prática: gerar e gerenciar ativos de imagem. De certa forma, era como os dramas da adolescência – calmos por fora, turbulentos por baixo.

À medida que o projeto acumulava mais documentação e materiais de marketing, precisávamos de um grande número de imagens de apoio. Alguns tiveram que ser gerados por IA, alguns tiveram que ser selecionados de uma biblioteca de ativos existente e outros precisavam de reconhecimento de IA e rotulagem automática. O problema era que tudo isso era tratado há muito tempo por meio de scripts dispersos e etapas manuais. Cada vez que geramos uma imagem, tínhamos que executar um script manualmente, organizar metadados manualmente e criar miniaturas manualmente. Só isso já era bastante irritante, mas o maior problema era que tudo estava espalhado por toda parte. Quando queríamos encontrar algo, não conseguíamos. Quando precisávamos reutilizar algo, não podíamos.

Os pontos problemáticos eram concretos:

1. **Sem ponto de entrada unificado**: a lógica para geração de imagens estava espalhada por diferentes scripts, então a execução em lote era basicamente impossível.
2. **Metadados ausentes**: as imagens geradas não tinham unificação `metadata.json`, o que significava que não havia capacidade de pesquisa ou rastreabilidade confiável.
3. **Alto custo de organização manual**: títulos e tags precisavam ser separados manualmente, o que era ineficiente.
4. **Sem automação**: gerando automaticamente ativos visuais em um pipeline de CI/CD? Não é uma chance.

Nós pensamos em simplesmente deixar isso como está. Mas os projetos ainda precisam avançar. Como não podíamos evitar o problema, achamos que poderíamos resolvê-lo. Portanto, decidimos atualizar o ImgBin de um conjunto de scripts dispersos para um pipeline de ativos de imagem que pode ser executado automaticamente. Afinal, alguns problemas não desaparecem só porque você desvia o olhar.

## Sobre HagiCode

A abordagem compartilhada neste artigo vem de nossa experiência prática no [HagiCode](https://hagicode.com) projeto. HagiCode é um projeto de assistente de codificação de IA que mantém simultaneamente vários componentes, incluindo uma extensão VSCode, serviços de back-end de IA e um cliente de desktop multiplataforma. Em um ambiente complexo, multilíngue e multiplataforma como este, o gerenciamento padronizado de ativos de imagem torna-se uma parte fundamental para melhorar a eficiência do desenvolvimento.

Você poderia dizer que esta foi uma daquelas pequenas dores de crescimento na jornada do HagiCode. Todo projeto tem momentos assim: um problema menor que parece insignificante, mas que de alguma forma consegue ocupar metade do dia.

O sistema de construção do HagiCode é baseado no ecossistema TypeScript + Node.js, então o ImgBin adotou naturalmente a mesma pilha de tecnologia para manter o projeto tecnicamente consistente. Depois que você estiver acostumado com uma pilha, mudar para outra parecerá um problema desnecessário.

## Projeto Central

### Arquitetura Geral

ImgBin usa uma arquitetura em camadas que separa claramente comandos CLI, serviços de aplicativos, adaptadores API de terceiros e a camada de infraestrutura:

```
Component hierarchy
├── CLI Entry (cli.ts)              Global argument parsing, command routing
├── Commands (commands/*)           generate | batch | annotate | thumbnail
├── Application Services            job-runner | metadata | thumbnail | asset-writer
├── Provider Adapters               image-api-provider | vision-api-provider
└── Infrastructure Layer            config | logger | paths | schema
```

O benefício desse design em camadas são limites claros de responsabilidade. Também torna os testes mais fáceis porque as dependências externas podem ser simuladas de forma limpa. Na prática, significa apenas que cada camada faz seu próprio trabalho sem atrapalhar as outras, portanto, quando algo quebra, é mais fácil descobrir o porquê.

### Modelo de diretório de ativo único

ImgBin usa um modelo de “um ativo, um diretório”. Cada vez que uma imagem é gerada, ela cria uma estrutura como esta:

```
library/
└── 2026-03/
    └── orange-dashboard/
        ├── original.png      # Original image
        ├── thumbnail.webp    # 512x512 thumbnail
        └── metadata.json     # Structured metadata
```

As vantagens deste modelo são:

1. **Independente**: todos os arquivos de um único ativo ficam no mesmo diretório, tornando a migração e o backup convenientes.
2. **Rastreável**: `metadata.json` torna possível rastrear o tempo de geração, prompt, modelo e outros detalhes.
3. **Extensível**: se mais variantes forem necessárias posteriormente, como miniaturas em vários tamanhos, podemos simplesmente adicionar novos arquivos no mesmo diretório.

Coisas bonitas nem sempre precisam ser possuídas. Às vezes basta que permaneçam bonitos e que você possa apreciá-los com tranquilidade. Isso pode parecer um pouco distante, mas a lógica ainda se mantém aqui: uma vez que as imagens são mantidas juntas, elas são mais agradáveis ​​de se olhar e muito mais fáceis de encontrar.

### Armazenamento de metadados em camadas

`metadata.json` é o núcleo de todo o sistema. Ele usa uma estratégia de armazenamento em camadas que separa os campos em três categorias:

```json
{
  "schemaVersion": 2,
  "assetId": "orange-dashboard",
  "slug": "orange-dashboard",
  "title": "Orange Dashboard",
  "tags": ["dashboard", "hero", "orange"],

  "source": { "type": "generated" },

  "paths": {
    "assetDir": "library/2026-03/orange-dashboard",
    "original": "original.png",
    "thumbnail": "thumbnail.webp"
  },

  "generated": {
    "prompt": "orange dashboard for docs hero",
    "provider": "azure-openai-image-api",
    "model": "gpt-image-1.5"
  },

  "recognized": {
    "title": "Orange Dashboard",
    "tags": ["dashboard", "ui", "orange"],
    "description": "A modern orange dashboard with charts and metrics"
  },

  "status": {
    "generation": "succeeded",
    "recognition": "succeeded",
    "thumbnail": "succeeded"
  },

  "timestamps": {
    "createdAt": "2026-03-11T04:01:19.570Z",
    "updatedAt": "2026-03-11T04:02:09.132Z"
  }
}
```

- **gerado**: registra as informações originais da geração da imagem, como prompt, provedor e modelo.
- **reconhecido**: armazena resultados de reconhecimento de IA, como títulos, tags e descrições gerados automaticamente.
- **manual**: armazena resultados selecionados manualmente. Os dados nesta área têm a mais alta prioridade e não serão substituídos pelo reconhecimento da IA.

Esta estratégia em camadas resolve um dos nossos conflitos centrais anteriores: quando o reconhecimento da IA e a curadoria manual discordam, qual deles deve vencer? A resposta é entrada manual. O reconhecimento da IA ​​existe para ajudar, não para decidir. Essa questão também ficou mais clara com o tempo – as máquinas ainda são máquinas e, no final, as pessoas ainda precisam fazer a ligação.

## Padrão de adaptador de provedor

Outra parte central do ImgBin é o padrão Provider Adapter. Abstraímos APIs externas por trás de uma interface unificada para que, mesmo que troquemos de provedor de serviços de IA, não precisemos alterar a lógica de negócios.

De certa forma, é um pouco como os relacionamentos – as aparências externas podem mudar, mas o que importa é que a estrutura interna permanece a mesma. Uma vez corrigida a interface, a implementação interna pode variar livremente.

### Provedor de geração de imagens

```typescript
interface ImageGenerationProvider {
  // Generate an image and return its Buffer
  generate(options: GenerateOptions): Promise<Buffer>;

  // Get the list of supported models
  getSupportedModels(): Promise<string[]>;
}

interface GenerateOptions {
  prompt: string;
  model?: string;
  size?: '1024x1024' | '1792x1024' | '1024x1792';
  quality?: 'standard' | 'hd';
  format?: 'png' | 'webp' | 'jpeg';
}
```

### Provedor de reconhecimento de visão

```typescript
interface VisionRecognitionProvider {
  // Recognize image content and return structured metadata
  recognize(imageBuffer: Buffer): Promise<RecognitionResult>;

  // Get the list of supported models
  getSupportedModels(): Promise<string[]>;
}

interface RecognitionResult {
  title?: string;
  tags: string[];
  description?: string;
  confidence: number;
}
```

As vantagens deste design de interface são:

1. **Testável**: em testes unitários, podemos passar provedores simulados em vez de fazer chamadas de API externas reais.
2. **Extensível**: adicionar um novo provedor requer apenas a implementação da interface; o código do chamador não precisa ser alterado.
3. **Substituível**: a produção pode usar o Azure OpenAI, enquanto o teste pode usar um modelo local, sendo a configuração a única coisa que muda.

Às vezes, o trabalho do projeto também parece assim. Superficialmente, parece que acabamos de trocar uma API, mas a lógica interna permanece exatamente a mesma, e isso torna tudo muito menos assustador.

## Projeto de comando CLI

ImgBin fornece quatro comandos principais para cobrir diferentes cenários de uso:

### gerar: geração de imagem única

```bash
# Simplest usage
imgbin generate --prompt "orange dashboard for docs hero"

# Generate a thumbnail and AI annotations at the same time
imgbin generate --prompt "orange dashboard" --annotate --thumbnail

# Specify an output directory
imgbin generate --prompt "orange dashboard" --output ./library
```

### lote: trabalhos em lote

Os trabalhos em lote são definidos por meio de arquivos de manifesto YAML ou JSON, o que os torna adequados para fluxos de trabalho de CI/CD:

```yaml
# assets/jobs/launch.yaml
defaults:
  annotate: true
  thumbnail: true
  libraryRoot: ./library

jobs:
  - prompt: "orange dashboard hero"
    slug: orange-dashboard
    tags: [dashboard, hero, orange]

  - prompt: "pricing grid for docs"
    slug: pricing-grid
    tags: [pricing, grid, docs]
```

Execute o comando:

```bash
imgbin batch assets/jobs/launch.yaml
```

O design do trabalho em lote oferece suporte ao isolamento de falhas: os itens no manifesto são processados um por um e uma falha em um item não afeta os outros. Você também pode visualizar o trabalho com `--dry-run` sem realmente executá-lo.

E a melhor parte é que ele informa exatamente o que deu certo e o que falhou. Ao contrário de algumas coisas na vida, onde o fracasso acontece e você fica sem saber como aconteceu.

### anotar: anotação AI

Execute o reconhecimento de IA em imagens existentes para gerar títulos, tags e descrições automaticamente:

```bash
# Annotate a single image
imgbin annotate ./library/2026-03/orange-dashboard

# Annotate an entire directory in batch
imgbin annotate ./library/2026-03/
```

### miniatura: geração de miniaturas

Gere miniaturas para imagens existentes:

```bash
# Generate a thumbnail
imgbin thumbnail ./library/2026-03/orange-dashboard
```

## Design de manifesto de trabalho em lote

O formato de manifesto para trabalhos em lote oferece suporte à configuração flexível. Os padrões podem ser definidos globalmente e trabalhos individuais podem substituí-los:

```yaml
# Global defaults
defaults:
  annotate: true        # Enable AI annotation by default
  thumbnail: true       # Generate thumbnails by default
  libraryRoot: ./library
  model: gpt-image-1.5

jobs:
  # Minimal configuration: only provide a prompt
  - prompt: "first image"

  # Full configuration
  - prompt: "second image"
    slug: custom-slug
    tags: [tag1, tag2]
    annotate: false     # Do not run AI annotation for this job
    model: dall-e-3    # Use a different model for this job
```

Quando executado, o ImgBin processa os trabalhos um por um. O resultado de cada trabalho é escrito em seu correspondente `metadata.json`. Mesmo que um trabalho falhe, os outros não serão afetados. Após a conclusão de todos os trabalhos, a CLI gera um relatório resumido:

```
✓ orange-dashboard (succeeded)
✓ pricing-grid (succeeded)
✗ hero-banner (failed: API rate limit exceeded)

2/3 succeeded, 1 failed
```

Algumas coisas não podem ser apressadas. Seguir um de cada vez costuma ser o caminho mais estável. Talvez essa seja a filosofia por trás dos trabalhos em lote.

## Configuração de variável de ambiente

ImgBin suporta configuração flexível através de variáveis de ambiente:

```bash
# ImgBin working directory
IMGBIN_WORKDIR=/path/to/imgbin

# Executable path (for invocation inside scripts)
IMGBIN_EXECUTABLE=/path/to/imgbin/dist/cli.js

# Asset library root
IMGBIN_LIBRARY_ROOT=./.imgbin-library

# Azure OpenAI configuration (if using the Azure provider)
AZURE_OPENAI_ENDPOINT=https://your-resource.openai.azure.com/
AZURE_OPENAI_API_KEY=***
AZURE_OPENAI_IMAGE_DEPLOYMENT=gpt-image-1
```

A configuração é uma daquelas coisas que pode parecer importante e não tão importante ao mesmo tempo. No final das contas, o que for confortável e se adequar melhor ao seu fluxo de trabalho geralmente é a escolha certa.

## Notas de implementação

Durante a implementação, resumimos alguns pontos-chave:

### Design de interface do provedor

As definições de interface devem ser claras e completas, incluindo parâmetros de entrada, valores de retorno e tratamento de erros. Também é uma boa ideia fornecer estilos de invocação síncrona e assíncrona para diferentes cenários.

Esse é um pequeno pedaço de experiência conquistada com dificuldade. Depois que uma interface é definida, ninguém quer alterá-la posteriormente.

### Estratégia de Tratamento de Falhas

Quando um item falha em um trabalho em lote, a CLI deve:

1. Grave informações detalhadas sobre erros em um arquivo de log separado.
2. Continue executando outros trabalhos em vez de interromper todo o processo.
3. Retorne um código de saída diferente de zero no final para indicar que alguns trabalhos falharam.
4. Exiba claramente o resultado da execução de cada trabalho no relatório resumido.

Algumas falhas são apenas falhas. Não adianta fingir o contrário. É melhor reconhecê-los abertamente e depois descobrir como resolvê-los. A mesma lógica se aplica aos projetos e à vida.

### Estratégia de mesclagem de metadados

Os resultados do reconhecimento são gravados no `recognized` seção por padrão, enquanto os campos editados manualmente são marcados em `manual`. As atualizações de metadados seguem uma estratégia apenas de acréscimo: a menos que `--force` for explicitamente passado, os resultados existentes selecionados manualmente não serão substituídos.

Esse ponto também ficou claro – algumas coisas, uma vez substituídas, simplesmente desaparecem. Muitas vezes é melhor preservá-los, porque o próprio documento tem valor.

### Atomicidade de criação de diretório

Usar `fs.mkdir({ recursive: true })` para garantir que a criação do diretório permaneça atômica e para evitar condições de corrida em cenários simultâneos.

Talvez seja assim que se sente a segurança - ser estável quando a estabilidade é importante, mover-se rapidamente quando a velocidade é importante e nunca ficar preso a dúvidas.

## Conclusão

Como ferramenta central para gerenciamento de ativos de imagem no projeto HagiCode, o ImgBin resolve nossos problemas por meio das seguintes opções de design:

1. **Ponto de entrada unificado**: a CLI abrange geração, anotação, miniaturas e todas as outras operações principais.
2. **Orientado por metadados**: cada ativo tem um `metadata.json`, permitindo pesquisa e rastreabilidade.
3. **Provedor Adaptador**: abstração flexível para APIs externas, facilitando testes e extensões.
4. **Suporte a trabalhos em lote**: a geração de imagens em lote pode ser automatizada em fluxos de trabalho de CI/CD.

Todo o resto pode ter desaparecido, mas esta abordagem acabou por se revelar útil.

Esta solução não apenas melhora a eficiência de desenvolvimento do HagiCode, mas também forma uma estrutura reutilizável para gerenciamento de ativos de imagem. Se você estiver construindo um projeto semelhante com vários componentes, acredito que as ideias de design do ImgBin podem lhe dar alguma inspiração.

A juventude tem tudo a ver com tentar coisas e fazer uma bagunça. Se você nunca passasse por isso, como saberia do que realmente é capaz?
