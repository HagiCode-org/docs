## Prática de integração de agentes Hermes: do protocolo à produção

> Compartilhando a experiência completa do HagiCode de integração do Agente Hermes, incluindo lições básicas sobre adaptação do protocolo ACP, gerenciamento de pool de sessões e sincronização de contratos front-end/back-end.

## Plano de fundo

Ao construir o HagiCode, uma plataforma de codificação assistida por IA, nossa equipe precisava integrar uma estrutura de agente que pudesse ser executada localmente e também escalonada para a nuvem. Após pesquisa, o Agente Hermes da Nous Research foi escolhido como o mecanismo subjacente para nossos recursos de agente de uso geral.

Na verdade, a seleção de tecnologia não é especialmente difícil nem especialmente fácil. Existem muitas estruturas de agentes fortes no mercado, mas a Hermes se destacou porque seu protocolo ACP e sistema de ferramentas atendem particularmente bem aos exigentes requisitos do HagiCode: desenvolvimento local, colaboração em equipe e expansão na nuvem. Ainda assim, trazer o Hermes para um sistema de produção real significou resolver uma longa lista de problemas de engenharia. Esta parte foi tudo menos trivial.

A pilha do HagiCode usa Orleans para construir um sistema distribuído, enquanto o front end é construído com React + TypeScript. Integrar o Hermes significou preservar a consistência arquitetônica e ao mesmo tempo tornar o Hermes um executor de primeira classe ao lado de ClaudeCode e OpenCode. Parece bastante simples, mas a implementação sempre conta a história real.

Este artigo compartilha nossa experiência prática na integração do Agente Hermes ao HagiCode e esperamos que ele ofereça material de referência útil para equipes que enfrentam necessidades semelhantes. Afinal, uma vez que você caiu em um buraco, não há razão para deixar outra pessoa cair no mesmo.

## Sobre HagiCode

A solução descrita neste artigo vem do nosso trabalho prático no [HagiCode](https://hagicode.com) projeto. HagiCode é uma plataforma de assistência de codificação orientada por IA que oferece suporte ao acesso unificado e ao gerenciamento de vários provedores de IA. Durante a integração do Hermes Agent, projetamos uma camada genérica de abstração de Provedor para que novos tipos de Agentes pudessem se conectar perfeitamente ao sistema existente.

Se você estiver interessado no HagiCode, sinta-se à vontade para visitar [GitHub](https://github.com/HagiCode-org/site) para saber mais. Quanto mais pessoas prestarem atenção, mais forte será o impulso.

## Projeto de Arquitetura

### Abordagem de design em camadas

A integração Hermes do HagiCode usa uma arquitetura clara em camadas, com cada camada focada em suas próprias responsabilidades:

**Camada principal de back-end**
- `HermesCliProvider`: implementa o `IAIProvider` interface como ponto de entrada unificado do provedor de IA
- `HermesPlatformConfiguration`: gerencia o caminho executável do Hermes, argumentos, autenticação e configurações relacionadas
- `ICliProvider<HermesOptions>`: a abstração CLI de baixo nível fornecida por HagiCode.Libs para lidar com ciclos de vida de subprocessos

**Camada de transporte**
- `StdioAcpTransport`: comunica-se com o subprocesso Hermes ACP através de entrada e saída padrão
- Métodos de protocolo ACP: `initialize`, `authenticate`, `session/new`, `session/prompt`

**Camada de tempo de execução**
- `HermesGrain`: Implementação Orleans Grain que lida com a execução de sessões distribuídas
- `CliAcpSessionPool`: pool de sessões que reutiliza subprocessos ACP para evitar sobrecarga frequente de inicialização

**Camada frontal**
- `ExecutorAvatar`: Identidade visual e ícone Hermes
- `executorTypeAdapter`: Lógica de mapeamento de tipo de provedor
- Mensagens em tempo real SignalR: mantém a consistência da identidade Hermes em todo o fluxo de mensagens

Este design em camadas permite que cada camada evolua de forma independente. Por exemplo, se quisermos adicionar um novo mecanismo de transporte no futuro, como o WebSocket, só precisaremos modificar a camada de transporte. Não há necessidade de entregar todo o sistema só porque um transporte muda.

### Abstração de interface unificada

Todos os provedores de IA implementam o `IAIProvider` interface, que é uma das principais opções de design na arquitetura do HagiCode:

```csharp
public interface IAIProvider
{
    string Name { get; }
    ProviderCapabilities Capabilities { get; }

    IAsyncEnumerable<AIStreamingChunk> StreamAsync(
        AIRequest request,
        CancellationToken cancellationToken = default);

    Task<AIResponse> ExecuteAsync(
        AIRequest request,
        CancellationToken cancellationToken = default);
}
```

`HermesCliProvider` implementa esta interface e está em pé de igualdade com `ClaudeCodeProvider`, `OpenCodeProvider`e outros. Os benefícios deste design incluem:

1. **Capacidade de substituição**: a troca de provedores não afeta a lógica de negócios da camada superior
2. **Testabilidade**: os provedores podem ser facilmente ridicularizados para testes unitários
3. **Extensibilidade**: adicionar um novo Provedor requer apenas a implementação da interface

No final das contas, as interfaces são muito parecidas com regras. Uma vez estabelecidas as regras, todos poderão coexistir harmoniosamente, aproveitar seus pontos fortes e evitar pisar uns nos outros. Há uma certa elegância nisso.

## Implementação principal

### Implementação da camada de provedor

`HermesCliProvider` é o núcleo de toda a integração. Ele coordena os vários componentes necessários para completar uma única invocação de IA:

```csharp
public sealed class HermesCliProvider : IAIProvider, IVersionedAIProvider
{
    private readonly ICliProvider<LibsHermesOptions> _provider;
    private readonly ConcurrentDictionary<string, string> _sessionBindings;

    public ProviderCapabilities Capabilities { get; } = new()
    {
        SupportsStreaming = true,
        SupportsTools = true,
        SupportsSystemMessages = true,
        SupportsArtifacts = false
    };

    public async IAsyncEnumerable<AIStreamingChunk> StreamAsync(
        AIRequest request,
        [EnumeratorCancellation] CancellationToken cancellationToken = default)
    {
        // 1. Resolve the session binding key
        var bindingKey = ResolveBindingKey(request.CessionId);

        // 2. Get or create a Hermes session through the session pool
        var options = new HermesOptions
        {
            ExecutablePath = _platformConfiguration.ExecutablePath,
            Arguments = _platformConfiguration.Arguments,
            SessionId = _sessionBindings.TryGetValue(bindingKey, out var sessionId) ? sessionId : null,
            WorkingDirectory = request.WorkingDirectory,
            Model = request.Model
        };

        // 3. Execute and collect the streaming response
        await foreach (var message in _provider.ExecuteAsync(options, request.Prompt, cancellationToken))
        {
            // 4. Map ACP messages to AIStreamingChunk
            if (_responseMapper.TryConvertToStreamingChunk(message, out var chunk))
            {
                yield return chunk;
            }
        }
    }
}
```

Vários pontos de design são especialmente importantes aqui:

1. **Vinculação de sessão**: usos `CessionId` para vincular várias solicitações ao mesmo subprocesso Hermes, preservando a continuidade do contexto em conversas multivoltas
2. **Mapeamento de resposta**: converte o formato de mensagem Hermes ACP no formato unificado `AIStreamingChunk` formato
3. **Suporte para streaming**: usos `IAsyncEnumerable` para suportar respostas de streaming verdadeiras

A vinculação de sessão é um pouco como os relacionamentos humanos. Depois que uma conexão é estabelecida, a comunicação futura tem contexto, portanto você não precisa começar do zero todas as vezes. É claro que essa relação ainda precisa ser mantida.

### Adaptação do protocolo ACP

Hermes usa ACP (Agent Communication Protocol), que difere de uma API HTTP tradicional. ACP é um protocolo baseado em entrada e saída padrão e possui diversas características:

1. **Marcador de inicialização**: após o início do processo Hermes, ele gera o `//ready` marcador
2. **Autenticação dinâmica**: os métodos de autenticação não são fixos e devem ser negociados através do protocolo
3. **Reutilização de sessão**: sessões estabelecidas são reutilizadas por meio `SessionId`
4. **Respostas fragmentadas**: uma resposta completa pode ser dividida em vários `session/update` notificações

HagiCode lida com essas características através `StdioAcpTransport`:

```csharp
public class StdioAcpTransport
{
    public async Task InitializeAsync(CancellationToken cancellationToken)
    {
        // Wait for the //ready marker
        var readyLine = await _outputReader.ReadLineAsync(cancellationToken);
        if (readyLine != "//ready")
        {
            throw new InvalidOperationException("Hermes did not send ready signal");
        }

        // Send the initialize request
        await SendRequestAsync(new
        {
            jsonrpc = "2.0",
            id = 1,
            method = "initialize",
            @params = new
            {
                protocolVersion = "2024-11-05",
                capabilities = new { },
                clientInfo = new { name = "HagiCode", version = "1.0.0" }
            }
        }, cancellationToken);
    }
}
```

Os protocolos são um pouco como o entendimento mútuo entre as pessoas. Uma vez que essa compreensão existe, a comunicação flui muito mais suavemente. Construí-lo leva tempo.

### Gerenciamento de pool de sessões

Iniciar subprocessos Hermes com frequência é caro, então implementamos um mecanismo de pool de sessões:

```csharp
services.AddSingleton(static _ =>
{
    var registry = new CliProviderPoolConfigurationRegistry();
    registry.Register("hermes", new CliPoolSettings
    {
        MaxActiveSessions = 50,
        IdleTimeout = TimeSpan.FromMinutes(10)
    });
    return registry;
});
```

Parâmetros principais do conjunto de sessões:
- `MaxActiveSessions`: controla o limite de simultaneidade para evitar o esgotamento de recursos
- `IdleTimeout`: tempo limite de inatividade que equilibra o custo de inicialização em relação ao uso de memória

Na prática, descobrimos que:
1. Se o tempo limite de inatividade for muito curto, as sessões serão reiniciadas com frequência; se for muito longo, a memória permanece ocupada
2. O limite de simultaneidade deve ser ajustado de acordo com a carga real, pois configurá-lo muito alto pode tornar o sistema lento
3. A utilização do pool de sessões precisa de monitoramento para que os parâmetros possam ser ajustados a tempo

Isto é muito parecido com muitas escolhas na vida: ser demasiado agressivo cria problemas, enquanto ser demasiado conservador perde oportunidades. O objetivo é simplesmente encontrar o equilíbrio certo.

## Integração front-end

O front end precisa identificar corretamente o Provedor Hermes e exibir os elementos visuais correspondentes:


### Sincronização de contrato

O front-end e o back-end mantêm seus contratos alinhados por meio da geração OpenAPI. O back-end define o `AIProviderType` enum:

```csharp
public enum AIProviderType
{
    Unknown,
    ClaudeCode,
    OpenCode,
    HermesCli  // Newly added
}
```

O front-end gera o tipo TypeScript correspondente por meio do OpenAPI, garantindo que os valores enum permaneçam consistentes. Esta é a chave para evitar a exibição do front-end `Unknown`.

Um contrato é muito parecido com uma promessa. Uma vez acordado, tem que ser honrado, caso contrário você acabará em situações embaraçosas como `Unknown`.

## Gerenciamento de configuração

A configuração do Hermes é gerenciada através `appsettings.json`:

```json
{
  "Providers": {
    "HermesCli": {
      "ExecutablePath": "hermes",
      "Arguments": "acp",
      "StartupTimeoutMs": 10000,
      "ClientName": "HagiCode",
      "Authentication": {
        "PreferredMethodId": "api-key",
        "MethodInfo": {
          "api-key": "your-api-key-here"
        }
      },
      "SessionDefaults": {
        "Model": "claude-sonnet-4-20250514",
        "ModeId": "default"
      }
    }
  }
}
```

Este design baseado em configuração traz flexibilidade:
- caminhos executáveis podem ser substituídos, o que é conveniente para desenvolvimento e teste
- argumentos de inicialização podem ser personalizados para corresponder a diferentes versões do Hermes
- as informações de autenticação podem ser configuradas para suportar vários métodos de autenticação

A configuração é um pouco como questões de múltipla escolha na vida. Se houver opções suficientes disponíveis, geralmente há uma que se adapta. Dito isso, muitas opções podem criar seu próprio cansaço de decisão.

## Experiência Prática

### Verificações de saúde

Construir um provedor confiável requer verificações de saúde abrangentes:

```csharp
public async Task<ProviderTestResult> PingAsync(CancellationToken cancellationToken = default)
{
    var response = await ExecuteAsync(new AIRequest
    {
        Prompt = "Reply with exactly PONG.",
        CessionId = null,
        AllowedTools = Array.Empty<string>(),
        WorkingDirectory = ResolveWorkingDirectory(null)
    }, cancellationToken);

    var success = string.Equals(response.Content.Trim(), "PONG", StringComparison.OrdinalIgnoreCase);
    return new ProviderTestResult
    {
        ProviderName = Name,
        Success = success,
        ResponseTimeMs = stopwatch.ElapsedMilliseconds,
        ErrorMessage = success ? null : $"Unexpected Hermes ping response: '{response.Content}'."
    };
}
```

Pontos a serem observados nas verificações de saúde:
1. Use casos de teste simples e evite cenários excessivamente complexos
2. Defina valores de tempo limite razoáveis
3. Registre o tempo de resposta para apoiar a análise de desempenho

Assim como as pessoas precisam de exames físicos, os sistemas também precisam de exames de saúde. Quanto mais cedo os problemas forem encontrados, mais fácil será resolvê-los.

HagiCode fornece um console dedicado para validar a integração do Hermes:


Esta ferramenta é extremamente útil durante o desenvolvimento porque permite verificar rapidamente se a integração está correta. Afinal, ninguém quer esperar até que surja um problema antes de se lembrar de testar.

### Lidando com problemas comuns

**Falha na autenticação**
- Verifique se `Authentication.PreferredMethodId` corresponde ao método de autenticação que Hermes realmente suporta
- Confirme se o formato das informações de autenticação está correto, como chave de API ou token de portador

**Tempo limite da sessão**
- Aumente o `StartupTimeoutMs` valor
- Verifique a acessibilidade do servidor MCP
- Revise a utilização de recursos do sistema

**Resposta incompleta**
- Garantir `session/update` notificações e o resultado final são agregados corretamente
- Verifique a lógica de cancelamento no tratamento de streaming
- Verifique se o tratamento de erros foi concluído

**O front-end exibe Desconhecido**
- Confirme se a geração do OpenAPI já inclui o `HermesCli` valor de enumeração
- Verifique se o mapeamento de tipo está correto
- Limpe o cache do navegador e gere novamente os tipos

Problemas sempre existirão. Quando aparecem, o importante é não entrar em pânico. Rastreie a causa passo a passo e, no final, a maioria delas poderá ser resolvida.

### Sugestões de otimização de desempenho

1. **Use o pool de sessões**: reutilize subprocessos ACP para reduzir a sobrecarga de inicialização
2. **Defina os tempos limite adequadamente**: equilibre o uso da memória em relação ao custo de inicialização
3. **Reutilizar IDs de sessão**: use os mesmos `CessionId` para tarefas em lote
4. **Configure o MCP sob demanda**: evite invocações desnecessárias de ferramentas

O desempenho é muito parecido com a eficiência na vida diária. Quando você acerta, consegue mais com menos; quando você erra, o esforço se multiplica enquanto os resultados diminuem. Encontrar o ponto “certo” exige experiência e sorte.

## Conclusão

A integração do Hermes Agent em um sistema de produção requer a consideração de problemas em múltiplas dimensões:

1. **Arquitetura**: projete uma interface unificada do provedor e implemente uma arquitetura de componentes substituíveis
2. **Protocolo**: lide corretamente com comportamentos específicos do ACP, como marcadores de inicialização e autenticação dinâmica
3. **Desempenho**: reutilize recursos por meio do pool de sessões e equilibre o custo de inicialização em relação ao uso de memória
4. **Front-end**: garanta a sincronização do contrato e forneça uma experiência visual consistente

A experiência da HagiCode mostra que com um bom design em camadas e implementação orientada à configuração, um sistema de agente complexo pode ser integrado perfeitamente a uma arquitetura existente.

Estes princípios parecem simples quando descritos em palavras, mas a implementação real introduz sempre muitos tipos diferentes de problemas. Tudo bem. Se um problema for resolvido, ele se tornará experiência. Se isso não acontecer, torna-se uma lição. De qualquer forma, ainda tem valor.

Coisas ou pessoas bonitas não precisam ser possuídas; enquanto permanecerem belos, basta apreciar silenciosamente essa beleza. A tecnologia é praticamente a mesma. Se ajudar a melhorar o sistema, então a estrutura ou protocolo específico importa muito menos do que as pessoas às vezes pensam.
