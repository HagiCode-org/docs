# Introdução ao OpenSpec

OpenSpec é uma ferramenta de desenvolvimento orientada por especificações que ajuda as equipas a gerir o processo de desenvolvimento de software através de propostas e especificações estruturadas. Este artigo apresenta uma introdução completa aos conceitos essenciais, fluxos de trabalho e formas de utilização do OpenSpec.

## O que é o OpenSpec?

OpenSpec é uma ferramenta de linha de comandos usada para gerir propostas (Proposals), especificações (Specifications) e alterações (Changes) no processo de desenvolvimento de software. Ele standardiza o fluxo de desenvolvimento, garantindo que cada funcionalidade passe por um planeamento e revisão detalhados, desde a ideia até à implementação.

### Princípios fundamentais

O OpenSpec baseia-se nos seguintes princípios fundamentais:

- **Especificações em primeiro lugar**: Defina especificações claras antes de escrever código
- **Colaboração estruturada**: Promova a colaboração da equipa através de formatos de documentação padronizados
- **Rastreabilidade**: Registe todo o processo desde a proposta até à implementação
- **Garantia de qualidade**: Assegure que o código cumpre a intenção do design através da validação das especificações

### Principais funcionalidades

O OpenSpec fornece as seguintes funcionalidades principais:

1. **Gestão de propostas**: Criar, rever e acompanhar propostas de funcionalidades
2. **Definição de especificações**: Escrever e manter documentos de especificações técnicas
3. **Rastreio de alterações**: Registar todo o histórico de alterações revista
4. **Validação de especificações**: Garantir que as alterações de código cumprem as especificações pré-definidas
5. **Colaboração de equipa**: Suportar processos de desenvolvimento e revisão com múltiplos participantes

## Fluxo de trabalho do OpenSpec

O fluxo de trabalho típico do OpenSpec inclui as seguintes fases:

### 1. Criar uma proposta (Proposal)

Quando existe uma nova funcionalidade ou um requisito de mudança significativa, comece por criar uma proposta:

```bash
openspec proposal create
```

A proposta inclui:
- **Descrição do problema**: O problema ou requisito a resolver
- **Solução**: Resumo da solução proposta
- **Âmbito do impacto**: Impacto no sistema existente
- **Soluções alternativas**: Outras opções consideradas

### 2. Escrever a especificação (Specification)

Após aprovada a proposta, escreva uma especificação técnica detalhada:

```bash
openspec spec create
```

A especificação inclui:
- **Design técnico**: Plano detalhado para a implementação técnica
- **Definição de interfaces**: Design de interfaces como APIs, estruturas de dados, etc.
- **Detalhes de implementação**: Lógica e algoritmos principais da implementação
- **Plano de testes**: Como verificar a corretude da implementação

### 3. Criar uma alteração (Change)

Ao iniciar a implementação, crie uma alteração para acompanhar o processo de desenvolvimento:

```bash
openspec change create
```

A alteração inclui:
- **Especificação associada**: A especificação que esta alteração implementa
- **Tarefas de implementação**: Lista de tarefas de desenvolvimento concretas
- **Rastreio de progresso**: Estado de conclusão das tarefas
- **Alterações de código**: As modificações reais no código

### 4. Validar e arquivar

Depois de concluído o desenvolvimento, valide se a alteração cumpre a especificação e, em seguida, archive-a:

```bash
openspec change validate
openspec change archive
```

## Estrutura de diretórios do OpenSpec

O OpenSpec cria um diretório `openspec/` na raiz do projeto com a seguinte estrutura:

```
openspec/
├── proposals/           # Diretório de propostas
│   ├── PROPOSAL-001.md
│   └── PROPOSAL-002.md
├── specs/              # Diretório de especificações
│   ├── SPEC-001.md
│   └── SPEC-002.md
├── changes/            # Diretório de alterações
│   ├── feature-name/
│   │   ├── proposal.md
│   │   ├── design.md
│   │   ├── specs.md
│   │   └── tasks.md
│   └── another-feature/
└── archived/           # Alterações arquivadas
    └── feature-name/
```

### Explicação dos diretórios

- **proposals/**: Armazena todos os documentos de proposta, cada proposta num ficheiro Markdown separado
- **specs/**: Armazena documentos de especificação técnica correspondentes às propostas
- **changes/**: Armazena as alterações em curso, cada alteração num subdiretório
- **archived/**: Armazena alterações concluídas e arquivadas

## Comandos principais

### Comandos relacionados com propostas

```bash
# Criar nova proposta
openspec proposal create

# Listar todas as propostas
openspec proposal list

# Ver detalhes da proposta
openspec proposal view <proposal-id>

# Aprovar proposta
openspec proposal approve <proposal-id>
```

### Comandos relacionados com especificações

```bash
# Criar nova especificação
openspec spec create

# Listar todas as especificações
openspec spec list

# Ver detalhes da especificação
openspec spec view <spec-id>

# Atualizar especificação
openspec spec update <spec-id>
```

### Comandos relacionados com alterações

```bash
# Criar nova alteração
openspec change create

# Listar todas as alterações
openspec change list

# Ver detalhes da alteração
openspec change view <change-name>

# Adicionar tarefa
openspec change add-task <change-name>

# Validar alteração
openspec change validate <change-name>

# Arquivar alteração
openspec change archive <change-name>
```

### Comandos gerais

```bash
# Ver versão
openspec --version

# Ver ajuda
openspec --help

# Ver ajuda de um comando específico
openspec <command> --help
```

## Usar o OpenSpec no projeto HagiCode

### Integração no projeto

No projeto HagiCode, o OpenSpec está integrado no fluxo de trabalho de desenvolvimento:

1. **Verificação na inicialização**: O assistente de arranque verifica se o OpenSpec está instalado
2. **Desenvolvimento orientado por propostas**: Todas as novas funcionalidades devem passar pelo processo de propostas do OpenSpec
3. **Revisão de código**: As alterações devem cumprir as especificações do OpenSpec para serem integradas
4. **Sincronização da documentação**: A documentação do OpenSpec é mantida sincronizada com a implementação do código

### Requisitos de versão

O projeto HagiCode requer o uso da versão **OpenSpec >=1.0.0 {'<'}2.0.0**.

Comando de instalação:

```bash
npm install -g @fission-ai/openspec@1
```

### Exemplo de fluxo de trabalho

Segue um exemplo completo de utilização do OpenSpec no projeto HagiCode:

#### Passo 1: Criar proposta

```bash
cd /path/to/hagicode-project
openspec proposal create
```

No prompt interativo, introduza:
- Título da proposta: Adicionar funcionalidade de autenticação de utilizadores
- Descrição do problema: O sistema necessita de login de utilizadores e gestão de permissões
- Solução: Implementar um sistema de autenticação baseado em JWT

#### Passo 2: Escrever a especificação

Após aprovação da proposta, crie uma especificação detalhada:

```bash
openspec spec create
```

Defina na especificação:
- Design do fluxo de autenticação
- Definição da API
- Estrutura das tabelas da base de dados
- Considerações de segurança

#### Passo 3: Criar alteração

Inicie a implementação:

```bash
openspec change create
```

Introduza o nome da alteração: `add-user-authentication`

#### Passo 4: Adicionar tarefas

```bash
openspec change add-task add-user-authentication
```

Adicione tarefas concretas:
- Implementar endpoint de registo de utilizadores
- Implementar endpoint de login
- Implementar middleware JWT
- Escrever testes unitários

#### Passo 5: Implementação e validação

Após concluir o código, valide a alteração:

```bash
openspec change validate add-user-authentication
```

#### Passo 6: Arquivar

Após validação bem-sucedida, archive a alteração:

```bash
openspec change archive add-user-authentication
```

## Melhores práticas

### Escrever propostas

1. **Declaração clara do problema**: Defina claramente o problema a resolver; evite descrições vagas
2. **Considere alternativas**: Liste e compare diferentes soluções
3. **Avalie o impacto**: Analise o impacto e os riscos no sistema existente
4. **Seja conciso**: As propostas devem ser sucintas; deixe os detalhes para a especificação

### Escrever especificações

1. **Detalhado, mas não redundante**: Forneça detalhes suficientes, evitando sobreengenharia
2. **Use diagramas**: Utilize fluxogramas, diagramas de arquitetura, etc., para ajudar a explicar
3. **Defina interfaces**: Defina claramente todas as interfaces públicas e estruturas de dados
4. **Considere casos limite**: Liste condições excecionais e tratamento de erros

### Gestão de alterações

1. **Itere em passos pequenos**: Divida grandes funcionalidades em várias alterações pequenas
2. **Atualize atempadamente**: Mantenha o estado das tarefas e a documentação sincronizados
3. **Revisão de código**: Faça revisão de código quando a alteração estiver concluída
4. **Archive atempadamente**: Archive alterações concluídas para manter o diretório de trabalho limpo

### Colaboração de equipa

1. **Revisões regulares**: Reveja propostas e especificações regularmente para garantir consenso na equipa
2. **Documentação primeiro**: Termine a documentação antes de iniciar a codificação
3. **Mantenha comunicação**: Registe discussões e decisões nas propostas e especificações
4. **Controlo de versões**: Inclua a documentação OpenSpec no controlo de versões

## Cenários de utilização comuns

### Cenário 1: Adicionar nova funcionalidade

1. Crie uma proposta que descreva os requisitos da nova funcionalidade
2. A equipa discute e aprova a proposta
3. Escreva uma especificação técnica detalhada
4. Crie uma alteração e implemente a funcionalidade
5. Valide e archive a alteração

### Cenário 2: Refatoração de código existente

1. Crie uma proposta explicando as razões e objetivos da refatoração
2. Defina o plano e os passos na especificação
3. Crie uma alteração e refatore gradualmente
4. Garanta cobertura por testes e valide os resultados da refatoração
5. Archive a alteração

### Cenário 3: Correção de bug

Para bugs simples:
- Podem ser corrigidos diretamente, sem o processo completo do OpenSpec

Para bugs complexos:
1. Crie uma proposta para analisar a causa raiz do bug
2. Defina a solução de correção na especificação
3. Crie uma alteração e implemente a correção
4. Adicione testes para prevenir regressões
5. Archive a alteração

### Cenário 4: Gestão de dívida técnica

1. Crie uma proposta listando itens de dívida técnica
2. Defina um plano de melhoria na especificação
3. Crie uma alteração para pagar gradualmente a dívida técnica
4. Acompanhe o progresso e o impacto
5. Archive as melhorias concluídas

## Integração com outras ferramentas

### Integração com Git

O OpenSpec integra-se sem problemas com o Git:

```bash
# A documentação OpenSpec deve ser incluída no controlo de versões
git add openspec/
git commit -m "Add OpenSpec proposal for feature X"
```

### Integração com CI/CD

Valide as especificações OpenSpec no pipeline CI/CD:

```yaml
# .github/workflows/openspec.yml
name: OpenSpec Validation
on: [pull_request]
jobs:
  validate:
    runs-on: ubuntu-latest
    steps:
      - uses: actions/checkout@v2
      - name: Install OpenSpec
        run: npm install -g @fission-ai/openspec@1
      - name: Validate changes
        run: openspec change validate
```

### Integração com IDE

Recomenda-se o uso de um editor com suporte a Markdown para editar documentos OpenSpec:
- VS Code + plugin Markdown
- JetBrains IDEs + suporte a Markdown
- Vim/Neovim + plugin Markdown

## Solução de problemas

### Problemas comuns

#### Problema: Comando OpenSpec não encontrado

**Solução**:
1. Confirme se o OpenSpec está instalado: `npm list -g @fission-ai/openspec`
2. Verifique se a variável de ambiente PATH inclui o diretório bin global do npm
3. Reinstale: `npm install -g @fission-ai/openspec@1`

#### Problema: Não é possível criar proposta ou alteração

**Solução**:
1. Confirme que o diretório atual é a raiz do projeto
2. Verifique se existe o diretório `openspec/`; se não, crie-o
3. Confirme que tem permissões de escrita

#### Problema: Validação falhou

**Solução**:
1. Verifique se a documentação no diretório de alteração está completa
2. Confirme que todas as tarefas necessárias foram concluídas
3. Revise a informação detalhada do erro: `openspec change validate <change-name> --verbose`

#### Problema: Arquivamento falhou

**Solução**:
1. Certifique-se de que a alteração passou na validação
2. Verifique se o diretório `openspec/archived/` existe
3. Confirme que não existem alterações de código por commitar

## Uso avançado

### Modelos personalizados

O OpenSpec suporta modelos personalizados para propostas e especificações:

```bash
# Crie um diretório de modelos na raiz do projeto
mkdir -p .openspec/templates

# Crie um modelo de proposta personalizado
cat > .openspec/templates/proposal.md << 'EOF'
# Proposal: {{title}}

## Problem Statement
{{problem}}

## Proposed Solution
{{solution}}

## Custom Section
Your custom content here
EOF
```

### Ficheiro de configuração

Crie um ficheiro de configuração `.openspec.json` na raiz do projeto:

```json
{
  "version": "1.0",
  "templates": {
    "proposal": ".openspec/templates/proposal.md",
    "spec": ".openspec/templates/spec.md"
  },
  "validation": {
    "requireTests": true,
    "requireDocs": true
  },
  "hooks": {
    "pre-archive": "npm test"
  }
}
```

### Hooks

O OpenSpec suporta a execução de scripts personalizados antes e depois de operações específicas:

```json
{
  "hooks": {
    "pre-create": "scripts/check-permissions.sh",
    "post-archive": "scripts/notify-team.sh"
  }
}
```
