## Menambahkan Dukungan Penerapan Otomatis Halaman GitHub ke HagiCode

> Proyek ini awalnya diberi nama kode PCode dan kini telah resmi berganti nama menjadi HagiCode. Artikel ini mendokumentasikan bagaimana kami memperkenalkan penerapan situs statis otomatis sehingga memublikasikan konten semudah meminum air.

## Latar Belakang / Pendahuluan

Selama pengembangan HagiCode, kami menghadapi masalah yang sangat praktis: ketika dokumentasi dan proposal kami terus berkembang, pengelolaan dan penyajian konten secara efisien menjadi semakin mendesak. Kami memutuskan untuk menggunakan Halaman GitHub untuk menghosting situs statis kami, namun membangun dan menerapkannya secara manual terlalu rumit. Setiap perubahan memerlukan pembuatan lokal, pengemasan, dan kemudian dorongan manual ke `gh-pages` cabang. Hal ini tidak hanya tidak efisien, tetapi juga mudah untuk melakukan kesalahan.

Untuk mengatasi masalah ini, terutama karena kami ingin menghemat upaya, kami memerlukan alur kerja penerapan otomatis. Artikel ini mencatat secara detail bagaimana kami menambahkan dukungan penerapan otomatis berbasis GitHub Actions ke proyek HagiCode, sehingga kami dapat tetap fokus pada penulisan konten dan membiarkan otomatisasi menangani sisanya.

## Analisis Tujuan

Sebelum terjun ke implementasi, pertama-tama kita perlu mengklarifikasi apa yang sebenarnya ingin dicapai oleh tugas ini. Toh, mengasah kapak tidak menunda pemotongan kayu bakar.

### Persyaratan Inti

1.  **Pembangunan otomatis**: Secara otomatis memicu alur kerja pembangunan ketika kode didorong ke `main` cabang.
2.  **Penerapan otomatis**: Setelah build berhasil, sebarkan file statis yang dihasilkan secara otomatis ke Halaman GitHub.
3.  **Konsistensi lingkungan**: Pastikan lingkungan CI cocok dengan lingkungan build lokal untuk menghindari rasa malu yang klasik karena "berfungsi secara lokal, tetapi gagal secara online".

### Pilihan Teknis

Karena HagiCode dibangun di Docusaurus, generator situs statis React yang sangat populer, kita dapat menggunakan GitHub Actions untuk mencapai tujuan ini.

## Mengonfigurasi Alur Kerja Tindakan GitHub

GitHub Actions adalah layanan CI/CD yang disediakan oleh GitHub. Dengan mendefinisikan file alur kerja dalam format YAML di dalam repositori, kita dapat menyesuaikan semua jenis tugas otomatisasi.

### Membuat File Alur Kerja

Kita perlu membuat file konfigurasi baru di `.github/workflows` direktori di root proyek, misalnya `deploy.yml`. Jika direktori belum ada, buatlah terlebih dahulu.

Logika inti dari file konfigurasi ini adalah sebagai berikut:

1.  **Kondisi pemicu**: Dengarkan `push` acara di `main` cabang.
2.  **Lingkungan runtime**: Rilis Ubuntu terbaru.
3.  **Langkah-langkah pembuatan**:
    -   Lihat kodenya
    -   Instal Node.js
    -   Instal dependensi (`npm install`)
    -   Bangun file statis (`npm run build`)
4.  **Langkah penerapan**: Gunakan yang resmi `action-gh-pages` tindakan untuk mendorong output build ke `gh-pages` cabang.

### Kode Konfigurasi Kunci

Di bawah ini adalah template konfigurasi yang akhirnya kami adopsi:

```yaml
name: Deploy to GitHub Pages

# Trigger condition: run when code is pushed to the main branch
on:
  push:
    branches:
      - main
    # Add path filters if needed, for example only build when docs change
    # paths:
    #   - 'docs/**'
    #   - 'package.json'

# Set permissions, which is important for GitHub Pages deployment
permissions:
  contents: read
  pages: write
  id-token: write

# Concurrency control: cancel older builds for the same branch
concurrency:
  group: "pages"
  cancel-in-progress: false

jobs:
  build:
    runs-on: ubuntu-latest
    steps:
      - name: Checkout
        uses: actions/checkout@v4
        # Note: fetch-depth: 0 is required, otherwise the build version may be inaccurate
        with:
          fetch-depth: 0

      - name: Setup Node
        uses: actions/setup-node@v4
        with:
          node-version: 20 # Keep this consistent with the local development environment
          cache: 'npm'     # Enabling cache can speed up the build process

      - name: Install dependencies
        run: npm ci
        # Use npm ci instead of npm install because it is faster and stricter, which makes it better for CI

      - name: Build website
        run: npm run build
        env:
          # Configure environment variables here if your site build requires them
          # NODE_ENV: production
          # PUBLIC_URL: /your-repo-name
          
      - name: Upload artifact
        uses: actions/upload-pages-artifact@v3
        with:
          path: ./build # Default Docusaurus output directory

  deploy:
    environment:
      name: github-pages
      url: ${{ steps.deployment.outputs.page_url }}
    runs-on: ubuntu-latest
    needs: build
    steps:
      - name: Deploy to GitHub Pages
        id: deployment
        uses: actions/deploy-pages@v4
```

## Kesalahan Umum Selama Implementasi

Dalam praktiknya, kami mengalami beberapa masalah. Saya membagikannya di sini dengan harapan orang lain dapat menghindarinya, atau setidaknya mempersiapkan solusinya terlebih dahulu.

### 1. Masalah Izin Token GitHub

Saat kami pertama kali menyiapkannya, penerapan terus gagal dengan 403 (Terlarang). Setelah banyak menggali, kami menemukan bahwa GitHub default `GITHUB_TOKEN` tidak memiliki izin untuk menulis ke Pages.

**Solusi**: Di repositori `Settings` -> `Actions` -> `General` -> `Workflow permissions`, pastikan untuk memilih **"Izin baca dan tulis"**.

### 2. Jalur Direktori Pembuatan yang Salah

Tempat Docusaurus membuat file statis di `build` direktori secara default. Namun, beberapa proyek mungkin dikonfigurasi secara berbeda. Misalnya, Buat Aplikasi React secara default `build`, sedangkan Vite defaultnya adalah `dist`. Jika Tindakan melaporkan bahwa ia tidak dapat menemukan file keluaran, periksa jalur keluaran `docusaurus.config.js`.

### 3. Masalah Subjalur

Jika repositori Anda bukan beranda pengguna, berarti bukan `username.github.io`, melainkan halaman proyek seperti `username.github.io/project-name`, maka Anda perlu mengkonfigurasi `baseUrl`.

Di `docusaurus.config.js`:

```javascript
module.exports = {
  // ...
  url: 'https://hagicode.com', // Your Hagicode URL
  baseUrl: '/', // Deploy from the root path
  // ...
};
```
Detail ini mudah untuk diabaikan. Jika dikonfigurasi dengan tidak benar, halaman mungkin dimuat sebagai layar kosong karena jalur aset tidak dapat diselesaikan.

## Memverifikasi Hasilnya

Setelah mengonfigurasi semuanya dan memasukkan kode, kita dapat menuju ke tab **Actions** di repositori GitHub dan menonton pertunjukannya.

Anda akan melihat lingkaran kuning saat alur kerja sedang berjalan, dan jika berubah menjadi hijau, berarti berhasil. Jika berubah menjadi merah, klik dan periksa log. Dalam kebanyakan kasus, Anda dapat melacak masalahnya di sana, biasanya kesalahan ketik atau konfigurasi jalur yang salah.

Setelah pembangunan berhasil, kunjungi `https://<your-username>.github.io/<repository-name>/` dan Anda akan melihat situs yang baru dikerahkan.

## Ringkasan

Dengan memperkenalkan GitHub Actions, kami berhasil mengotomatiskan penerapan untuk situs dokumentasi HagiCode. Hal ini tidak hanya menghemat waktu yang sebelumnya dihabiskan untuk operasi manual, namun yang lebih penting lagi adalah menstandarkan alur kerja rilis. Sekarang, tidak peduli anggota tim mana yang memperbarui dokumentasi, setelah perubahan digabungkan `main`, konten terbaru akan ditayangkan beberapa menit kemudian.

**Manfaat inti**:
- **Efisiensi lebih tinggi**: Kami beralih dari "pengemasan manual dan pengunggahan manual" ke "kode sama dengan rilis".
- **Lebih sedikit kesalahan**: Kami menghilangkan kemungkinan kesalahan manusia selama pengoperasian manual.
- **Pengalaman pengembang yang lebih baik**: Pengembang dapat lebih fokus pada kualitas konten daripada terganggu oleh langkah-langkah penerapan yang membosankan.

Meskipun mengonfigurasi CI/CD agak merepotkan pada awalnya, terutama dengan semua izin dan detail jalur, ini merupakan investasi satu kali dengan keuntungan jangka panjang yang besar. Saya sangat menyarankan agar setiap proyek situs statis mengadopsi alur kerja otomatis yang serupa.
