## Desain Alat ImgBin CLI: Pendekatan Manajemen Aset Gambar HagiCode

> Artikel ini menjelaskan cara membuat pipeline aset gambar otomatis dari awal, yang mencakup desain alat CLI, arsitektur Adaptor Penyedia, dan strategi manajemen metadata.

## Latar Belakang

Sejujurnya, saya tidak menyangka manajemen aset gambar akan membuat kita terjebak selama ini.

Selama pengembangan HagiCode, kami menemui masalah yang tampak sederhana di permukaan namun ternyata sulit dalam praktiknya: membuat dan mengelola aset gambar. Di satu sisi, ini seperti drama masa remaja – tenang di luar, penuh gejolak di bawahnya.

Ketika proyek mengumpulkan lebih banyak dokumentasi dan materi pemasaran, kami memerlukan sejumlah besar gambar pendukung. Beberapa diantaranya harus dihasilkan oleh AI, beberapa harus dipilih dari perpustakaan aset yang ada, dan lainnya memerlukan pengenalan AI ditambah pelabelan otomatis. Masalahnya semua itu sudah lama ditangani melalui skrip yang tersebar dan langkah manual. Setiap kali kami membuat gambar, kami harus menjalankan skrip dengan tangan, mengatur metadata dengan tangan, dan membuat thumbnail dengan tangan. Itu saja sudah cukup menjengkelkan, tapi masalah yang lebih besar adalah semuanya tersebar dimana-mana. Ketika kami ingin menemukan sesuatu, kami tidak bisa. Ketika kami perlu menggunakan kembali sesuatu, kami tidak bisa.

Poin-poin menyakitkannya nyata:

1. **Tidak ada titik masuk terpadu**: logika pembuatan gambar tersebar di berbagai skrip, sehingga eksekusi batch pada dasarnya tidak mungkin dilakukan.
2. **Metadata tidak ada**: gambar yang dihasilkan tidak disatukan `metadata.json`, yang berarti tidak ada kemampuan penelusuran atau penelusuran yang dapat diandalkan.
3. **Biaya pengorganisasian manual yang tinggi**: judul dan tag harus disortir satu per satu dengan tangan, dan hal ini tidak efisien.
4. **Tanpa otomatisasi**: secara otomatis menghasilkan aset visual dalam pipeline CI/CD? Tidak mungkin.

Kami sempat berpikir untuk membiarkannya saja. Namun proyek masih perlu dilanjutkan. Karena kami tidak dapat menghindari masalah tersebut, kami pikir sebaiknya kami menyelesaikannya. Jadi kami memutuskan untuk memutakhirkan ImgBin dari sekumpulan skrip yang tersebar menjadi saluran aset gambar yang dapat dijalankan secara otomatis. Beberapa masalah, bagaimanapun juga, tidak hilang hanya karena Anda memalingkan muka.

## Tentang HagiCode

Pendekatan yang dibagikan dalam artikel ini berasal dari pengalaman langsung kami di [Kode Hagi](https://hagicode.com) proyek. HagiCode adalah proyek asisten pengkodean AI yang secara bersamaan memelihara beberapa komponen, termasuk ekstensi VSCode, layanan AI backend, dan klien desktop lintas platform. Dalam lingkungan lintas platform yang kompleks dan multibahasa seperti ini, manajemen aset gambar terstandarisasi menjadi bagian penting dalam meningkatkan efisiensi pengembangan.

Bisa dibilang ini adalah salah satu kesulitan kecil dalam perjalanan HagiCode. Setiap proyek mempunyai momen-momen seperti itu: masalah kecil yang kelihatannya tidak penting, namun entah bagaimana berhasil menyita waktu setengah hari.

Sistem pembangunan HagiCode didasarkan pada ekosistem TypeScript + Node.js, jadi ImgBin secara alami mengadopsi tumpukan teknologi yang sama untuk menjaga proyek tetap konsisten secara teknis. Setelah Anda terbiasa dengan satu tumpukan, beralih ke tumpukan lain terasa seperti masalah yang tidak perlu.

## Desain Inti

### Arsitektur Keseluruhan

ImgBin menggunakan arsitektur berlapis yang memisahkan perintah CLI, layanan aplikasi, adaptor API pihak ketiga, dan lapisan infrastruktur dengan rapi:

```
Component hierarchy
├── CLI Entry (cli.ts)              Global argument parsing, command routing
├── Commands (commands/*)           generate | batch | annotate | thumbnail
├── Application Services            job-runner | metadata | thumbnail | asset-writer
├── Provider Adapters               image-api-provider | vision-api-provider
└── Infrastructure Layer            config | logger | paths | schema
```

Keuntungan dari desain berlapis ini adalah batasan tanggung jawab yang jelas. Ini juga membuat pengujian lebih mudah karena dependensi eksternal dapat ditiru dengan mudah. Dalam praktiknya, ini hanya berarti setiap lapisan melakukan tugasnya sendiri tanpa mengganggu lapisan lainnya, jadi ketika ada sesuatu yang rusak, akan lebih mudah untuk mengetahui alasannya.

### Model Direktori Aset Tunggal

ImgBin menggunakan model "satu aset, satu direktori". Setiap kali gambar dibuat, struktur seperti ini akan dibuat:

```
library/
└── 2026-03/
    └── orange-dashboard/
        ├── original.png      # Original image
        ├── thumbnail.webp    # 512x512 thumbnail
        └── metadata.json     # Structured metadata
```

Kelebihan model ini adalah:

1. **Mandiri**: semua file untuk satu aset berada di direktori yang sama, sehingga memudahkan migrasi dan pencadangan.
2. **Dapat Dilacak**: `metadata.json` memungkinkan untuk melacak waktu pembuatan, prompt, model, dan detail lainnya.
3. **Dapat diperluas**: jika nantinya diperlukan lebih banyak varian, seperti thumbnail dalam berbagai ukuran, kita cukup menambahkan file baru di direktori yang sama.

Hal-hal indah tidak selalu perlu dimiliki. Terkadang cukuplah mereka tetap cantik, dan Anda bisa menghargainya dengan tenang. Mungkin kedengarannya agak berlebihan, namun logikanya tetap berlaku: setelah gambar disatukan, gambar akan lebih enak dilihat dan lebih mudah ditemukan.

### Penyimpanan Metadata Berlapis

`metadata.json` merupakan inti dari keseluruhan sistem. Ini menggunakan strategi penyimpanan berlapis yang memisahkan bidang menjadi tiga kategori:

```json
{
  "schemaVersion": 2,
  "assetId": "orange-dashboard",
  "slug": "orange-dashboard",
  "title": "Orange Dashboard",
  "tags": ["dashboard", "hero", "orange"],

  "source": { "type": "generated" },

  "paths": {
    "assetDir": "library/2026-03/orange-dashboard",
    "original": "original.png",
    "thumbnail": "thumbnail.webp"
  },

  "generated": {
    "prompt": "orange dashboard for docs hero",
    "provider": "azure-openai-image-api",
    "model": "gpt-image-1.5"
  },

  "recognized": {
    "title": "Orange Dashboard",
    "tags": ["dashboard", "ui", "orange"],
    "description": "A modern orange dashboard with charts and metrics"
  },

  "status": {
    "generation": "succeeded",
    "recognition": "succeeded",
    "thumbnail": "succeeded"
  },

  "timestamps": {
    "createdAt": "2026-03-11T04:01:19.570Z",
    "updatedAt": "2026-03-11T04:02:09.132Z"
  }
}
```

- **generated**: mencatat informasi asli dari pembuatan gambar, seperti prompt, penyedia, dan model.
- **dikenali**: menyimpan hasil pengenalan AI, seperti judul, tag, dan deskripsi yang dibuat secara otomatis.
- **manual**: menyimpan hasil yang dikurasi secara manual. Data di area ini memiliki prioritas tertinggi dan tidak akan ditimpa oleh pengenalan AI.

Strategi berlapis ini menyelesaikan salah satu konflik inti kami sebelumnya: ketika pengakuan AI dan kurasi manual tidak sejalan, mana yang harus menang? Jawabannya adalah input manual. Pengenalan AI ada untuk membantu, bukan untuk memutuskan. Pertanyaan tersebut juga menjadi lebih jelas seiring berjalannya waktu - mesin tetaplah mesin, dan pada akhirnya, manusia masih perlu mengambil keputusan.

## Pola Adaptor Penyedia

Bagian inti lain dari ImgBin adalah pola Adaptor Penyedia. Kami mengabstraksi API eksternal di balik antarmuka terpadu sehingga meskipun kami berpindah penyedia layanan AI, kami tidak perlu mengubah logika bisnis.

Di satu sisi, ini seperti hubungan - penampilan luar bisa berubah, tapi yang penting adalah struktur batinnya tetap sama. Setelah antarmuka diperbaiki, implementasi internal dapat bervariasi secara bebas.

### Penyedia Pembuatan Gambar

```typescript
interface ImageGenerationProvider {
  // Generate an image and return its Buffer
  generate(options: GenerateOptions): Promise<Buffer>;

  // Get the list of supported models
  getSupportedModels(): Promise<string[]>;
}

interface GenerateOptions {
  prompt: string;
  model?: string;
  size?: '1024x1024' | '1792x1024' | '1024x1792';
  quality?: 'standard' | 'hd';
  format?: 'png' | 'webp' | 'jpeg';
}
```

### Penyedia Pengenalan Visi

```typescript
interface VisionRecognitionProvider {
  // Recognize image content and return structured metadata
  recognize(imageBuffer: Buffer): Promise<RecognitionResult>;

  // Get the list of supported models
  getSupportedModels(): Promise<string[]>;
}

interface RecognitionResult {
  title?: string;
  tags: string[];
  description?: string;
  confidence: number;
}
```

Keuntungan dari desain antarmuka ini adalah:

1. **Dapat diuji**: dalam pengujian unit, kita dapat meneruskan penyedia tiruan alih-alih membuat panggilan API eksternal yang sebenarnya.
2. **Dapat diperluas**: menambahkan penyedia baru hanya memerlukan penerapan antarmuka; kode penelepon tidak perlu diubah.
3. **Dapat diganti**: produksi dapat menggunakan Azure OpenAI sementara pengujian dapat menggunakan model lokal, dengan konfigurasi menjadi satu-satunya hal yang berubah.

Terkadang pekerjaan proyek juga terasa seperti itu. Di permukaan sepertinya kita baru saja menukar API, namun logika internalnya tetap sama, dan itu membuat semuanya menjadi tidak terlalu menakutkan.

## Desain Perintah CLI

ImgBin menyediakan empat perintah inti untuk mencakup skenario penggunaan yang berbeda:

### menghasilkan: pembuatan gambar tunggal

```bash
# Simplest usage
imgbin generate --prompt "orange dashboard for docs hero"

# Generate a thumbnail and AI annotations at the same time
imgbin generate --prompt "orange dashboard" --annotate --thumbnail

# Specify an output directory
imgbin generate --prompt "orange dashboard" --output ./library
```

### batch: pekerjaan batch

Pekerjaan batch ditentukan melalui file manifes YAML atau JSON, sehingga cocok untuk alur kerja CI/CD:

```yaml
# assets/jobs/launch.yaml
defaults:
  annotate: true
  thumbnail: true
  libraryRoot: ./library

jobs:
  - prompt: "orange dashboard hero"
    slug: orange-dashboard
    tags: [dashboard, hero, orange]

  - prompt: "pricing grid for docs"
    slug: pricing-grid
    tags: [pricing, grid, docs]
```

Jalankan perintah:

```bash
imgbin batch assets/jobs/launch.yaml
```

Desain tugas batch mendukung isolasi kegagalan: item dalam manifes diproses satu per satu, dan kegagalan pada satu item tidak memengaruhi item lainnya. Anda juga dapat melihat pratinjau pekerjaan dengan `--dry-run` tanpa benar-benar mengeksekusinya.

Dan bagian terbaiknya adalah ia memberi tahu Anda apa yang berhasil dan apa yang gagal. Berbeda dengan beberapa hal dalam hidup, di mana kegagalan terjadi dan Anda bahkan tidak mengetahui bagaimana hal itu terjadi.

### anotasi: anotasi AI

Jalankan pengenalan AI pada gambar yang ada untuk menghasilkan judul, tag, dan deskripsi secara otomatis:

```bash
# Annotate a single image
imgbin annotate ./library/2026-03/orange-dashboard

# Annotate an entire directory in batch
imgbin annotate ./library/2026-03/
```

### thumbnail: pembuatan thumbnail

Hasilkan thumbnail untuk gambar yang ada:

```bash
# Generate a thumbnail
imgbin thumbnail ./library/2026-03/orange-dashboard
```

## Desain Manifes Pekerjaan Batch

Format manifes untuk pekerjaan batch mendukung konfigurasi yang fleksibel. Defaultnya dapat diatur secara global, dan tugas individual dapat menimpanya:

```yaml
# Global defaults
defaults:
  annotate: true        # Enable AI annotation by default
  thumbnail: true       # Generate thumbnails by default
  libraryRoot: ./library
  model: gpt-image-1.5

jobs:
  # Minimal configuration: only provide a prompt
  - prompt: "first image"

  # Full configuration
  - prompt: "second image"
    slug: custom-slug
    tags: [tag1, tag2]
    annotate: false     # Do not run AI annotation for this job
    model: dall-e-3    # Use a different model for this job
```

Saat dijalankan, ImgBin memproses pekerjaan satu per satu. Hasil dari setiap pekerjaan ditulis ke yang sesuai `metadata.json`. Sekalipun satu pekerjaan gagal, pekerjaan lainnya tidak terpengaruh. Setelah semua pekerjaan selesai, CLI mengeluarkan laporan ringkasan:

```
✓ orange-dashboard (succeeded)
✓ pricing-grid (succeeded)
✗ hero-banner (failed: API rate limit exceeded)

2/3 succeeded, 1 failed
```

Beberapa hal tidak bisa diburu-buru. Mengambilnya satu per satu sering kali merupakan jalan yang lebih mantap. Mungkin itulah filosofi di balik pekerjaan batch.

## Konfigurasi Variabel Lingkungan

ImgBin mendukung konfigurasi fleksibel melalui variabel lingkungan:

```bash
# ImgBin working directory
IMGBIN_WORKDIR=/path/to/imgbin

# Executable path (for invocation inside scripts)
IMGBIN_EXECUTABLE=/path/to/imgbin/dist/cli.js

# Asset library root
IMGBIN_LIBRARY_ROOT=./.imgbin-library

# Azure OpenAI configuration (if using the Azure provider)
AZURE_OPENAI_ENDPOINT=https://your-resource.openai.azure.com/
AZURE_OPENAI_API_KEY=***
AZURE_OPENAI_IMAGE_DEPLOYMENT=gpt-image-1
```

Konfigurasi adalah salah satu hal yang terasa penting dan tidak penting pada saat yang bersamaan. Pada akhirnya, apa pun yang terasa nyaman dan paling sesuai dengan alur kerja Anda biasanya merupakan pilihan yang tepat.

## Catatan Implementasi

Selama implementasi, kami merangkum beberapa poin penting:

### Desain Antarmuka Penyedia

Definisi antarmuka harus jelas dan lengkap, termasuk parameter input, nilai kembalian, dan penanganan kesalahan. Sebaiknya sediakan gaya pemanggilan sinkron dan asinkron untuk skenario yang berbeda.

Itu adalah satu bagian kecil dari pengalaman yang diperoleh dengan susah payah. Setelah antarmuka disetel, tidak ada yang ingin terus mengubahnya nanti.

### Strategi Penanganan Kegagalan

Ketika satu item gagal dalam tugas batch, CLI harus:

1. Tulis informasi kesalahan terperinci ke file log terpisah.
2. Lanjutkan menjalankan pekerjaan lain alih-alih mengganggu keseluruhan proses.
3. Kembalikan kode keluar bukan nol di bagian akhir untuk menunjukkan bahwa beberapa pekerjaan gagal.
4. Tampilkan dengan jelas hasil eksekusi setiap pekerjaan dalam laporan ringkasan.

Beberapa kegagalan hanyalah kegagalan. Tidak ada gunanya berpura-pura sebaliknya. Lebih baik mengakuinya secara terbuka dan kemudian mencari cara untuk menyelesaikannya. Logika yang sama berlaku untuk proyek dan kehidupan.

### Strategi Penggabungan Metadata

Hasil pengakuan ditulis ke `recognized` bagian secara default, sedangkan bidang yang diedit secara manual ditandai `manual`. Pembaruan metadata mengikuti strategi tambahan saja: kecuali `--force` diteruskan secara eksplisit, hasil yang dikurasi secara manual tidak ditimpa.

Poin tersebut menjadi jelas juga - beberapa hal, setelah ditimpa, akan hilang begitu saja. Seringkali lebih baik melestarikannya, karena catatan itu sendiri mempunyai nilai.

### Atomisitas Pembuatan Direktori

Gunakan `fs.mkdir({ recursive: true })` untuk memastikan pembuatan direktori tetap bersifat atomik dan untuk menghindari kondisi balapan dalam skenario yang bersamaan.

Mungkin seperti itulah rasanya keamanan - menjadi stabil ketika stabilitas penting, bergerak cepat ketika kecepatan penting, dan tidak pernah terjebak dalam menebak-nebak.

## Kesimpulan

Sebagai alat inti untuk manajemen aset gambar dalam proyek HagiCode, ImgBin memecahkan masalah kami melalui pilihan desain berikut:

1. **Titik masuk terpadu**: CLI mencakup pembuatan, anotasi, gambar mini, dan semua operasi inti lainnya.
2. **Berbasis metadata**: setiap aset memiliki kelengkapan `metadata.json`, memungkinkan pencarian dan ketertelusuran.
3. **Adaptor Penyedia**: abstraksi fleksibel untuk API eksternal, membuat pengujian dan ekstensi menjadi lebih mudah.
4. **Dukungan tugas batch**: pembuatan gambar batch dapat diotomatisasi dalam alur kerja CI/CD.

Segala hal lainnya mungkin sudah memudar, namun pendekatan ini benar-benar terbukti bermanfaat.

Solusi ini tidak hanya meningkatkan efisiensi pengembangan HagiCode, namun juga membentuk kerangka kerja yang dapat digunakan kembali untuk manajemen aset gambar. Jika Anda sedang membangun proyek multi-komponen serupa, saya yakin ide desain ImgBin dapat memberi Anda inspirasi.

Masa muda adalah tentang mencoba berbagai hal dan membuat sedikit kekacauan. Jika Anda tidak pernah mengalami hal itu, bagaimana Anda tahu apa yang sebenarnya mampu Anda lakukan?
