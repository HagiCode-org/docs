## Menjalankan Alat AI CLI di Kontainer Docker: Panduan Praktis untuk Isolasi Pengguna dan Volume Persisten

> Mengintegrasikan alat pengkodean AI seperti Claude Code, Codex, dan OpenCode ke dalam lingkungan terkontainer terdengar sederhana, namun terdapat kompleksitas tersembunyi di mana-mana. Artikel ini membahas secara mendalam bagaimana proyek HagiCode memecahkan tantangan inti dalam penerapan Docker, termasuk izin pengguna, persistensi konfigurasi, dan manajemen versi, sehingga Anda dapat menghindari kesalahan umum.

## Latar Belakang

Saat kami memutuskan untuk menjalankan alat CLI pengkodean AI di dalam container Docker, pemikiran yang paling intuitif mungkin adalah: "Bukankah container hanya melakukan root? Mengapa tidak menginstal semuanya secara langsung dan menyelesaikannya?" Kenyataannya, ide yang tampak sederhana itu menyembunyikan beberapa permasalahan inti yang harus diselesaikan.

Pertama, **pembatasan keamanan adalah rintangan pertama**. Ambil Claude CLI sebagai contoh: secara eksplisit melarang dijalankan sebagai pengguna root. Ini adalah pemeriksaan keamanan wajib, dan jika root terdeteksi, ia menolak untuk memulai. Anda mungkin berpikir, tidak bisakah saya mengganti pengguna saja dengan `USER` arahan? Hal ini tidak sesederhana itu. Masih ada masalah pemetaan antara pengguna non-root di dalam penampung dan izin pengguna di mesin host.

Kedua, **kegigihan negara adalah jebakan kedua**. Claude Code memerlukan login, Codex memiliki konfigurasinya sendiri, dan OpenCode juga memiliki direktori cache. Jika Anda harus mengkonfigurasi ulang semuanya setiap kali wadah dimulai ulang, gagasan "otomatisasi" kehilangan maknanya. Kita memerlukan konfigurasi ini untuk bertahan melampaui siklus hidup container.

Masalah ketiga adalah **konsistensi izin**. Bisakah proses di dalam kontainer mengakses file konfigurasi yang dibuat oleh pengguna host? Ketidakcocokan UID/GID sering kali menyebabkan kesalahan izin file, dan ini sangat umum terjadi dalam penerapan nyata.

Permasalahan-permasalahan ini mungkin terlihat independen, namun dalam praktiknya permasalahan-permasalahan tersebut berkaitan erat. Selama pengembangan HagiCode, kami secara bertahap menemukan solusi praktis. Selanjutnya, saya akan membagikan detail teknis dan pembelajaran dari kendala tersebut.

## Mengapa kita tidak bisa menggunakan root saja?

Ada kesalahpahaman umum di sini: Kontainer Docker dijalankan sebagai root secara default, jadi mengapa tidak menginstal alat sebagai root saja? Jika Anda berpikir seperti itu, Claude CLI akan segera mengajari Anda sebaliknya.

```bash
# Run Claude CLI directly as root? No.
docker run --rm -it --user root myimage claude
# Output: Error: This command cannot be run as root user
```

Ini adalah batasan keamanan yang ketat di Claude CLI. Alasannya sederhana: alat CLI ini membaca dan menulis konfigurasi pengguna yang sensitif, termasuk token API, cache lokal, dan bahkan skrip yang ditulis oleh pengguna. Menjalankannya dengan hak akses root menimbulkan terlalu banyak risiko.

Jadi pertanyaannya menjadi: bagaimana kita dapat memenuhi persyaratan keamanan CLI sekaligus menjaga fleksibilitas manajemen container? Kita perlu mengubah cara berpikir kita tentang hal ini: **daripada berpindah pengguna saat runtime, buatlah pengguna khusus selama tahap pembuatan image**.

## Membuat pengguna khusus: lebih dari sekedar mengubah nama

Anda mungkin berpikir bahwa menambahkan satu `USER` baris ke Dockerfile sudah cukup. Ini memang merupakan pendekatan yang paling sederhana, namun tidak cukup kuat.

### Pembuatan statis vs. pemetaan dinamis

Pendekatan HagiCode adalah membuat a `hagicode` pengguna dengan UID 1000, yang biasanya cocok dengan pengguna default di sebagian besar mesin host:

```dockerfile
RUN groupadd -o -g 1000 hagicode && \
    useradd -o -u 1000 -g 1000 -s /bin/bash -m hagicode && \
    mkdir -p /home/hagicode/.claude && \
    chown -R hagicode:hagicode /home/hagicode
```

Tapi ini hanya menyelesaikan pengguna bawaan di dalam gambar. Bagaimana jika pengguna hostnya adalah UID 1001? Anda masih perlu mendukung pemetaan dinamis saat penampung dimulai.

docker-entrypoint.sh berisi logika kunci:

```bash
if [ -n "$PUID" ] && [ -n "$PGID" ]; then
    if ! id hagicode >/dev/null 2>&1; then
        groupadd -g "$PGID" hagicode
        useradd -u "$PUID" -g "$PGID" -s /bin/bash -m hagicode
    fi
fi
```

Keuntungan dari desain ini jelas: **gunakan UID 1000 default pada waktu pembuatan gambar, lalu sesuaikan secara dinamis saat runtime melalui `PUID` dan `PGID` variabel lingkungan**. Apa pun UID yang dimiliki pengguna host, kepemilikan file konfigurasi tetap benar.

## Filosofi desain volume yang persisten

Setiap alat AI CLI memiliki direktori konfigurasi pilihannya sendiri, sehingga perlu dipetakan satu per satu:

| Alat CLI | Jalur dalam Kontainer | Bernama Volume |
|---------|-----------|--------|
| Claude | `/home/hagicode/.claude` | `claude-data` |
| Kodeks | `/home/hagicode/.codex` | `codex-data` |
| Kode Terbuka | `/home/hagicode/.config/opencode` | `opencode-config-data` |

Mengapa menggunakan volume bernama alih-alih mengikat mount? Tiga alasan:

1. **Manajemen yang lebih sederhana**: Volume bernama dikelola secara otomatis oleh Docker, sehingga Anda tidak perlu membuat direktori host secara manual.
2. **Isolasi izin**: Konten awal volume dibuat oleh pengguna di dalam penampung, menghindari konflik izin dengan host.
3. **Migrasi independen**: Volume dapat ada secara independen dari container, sehingga data tidak hilang saat gambar diupgrade.

docker-compose-builder-web secara otomatis menghasilkan konfigurasi volume yang sesuai:

```yaml
volumes:
  claude-data:
  codex-data:
  opencode-config-data:

services:
  hagicode:
    volumes:
      - claude-data:/home/hagicode/.claude
      - codex-data:/home/hagicode/.codex
      - opencode-config-data:/home/hagicode/.config/opencode
    user: "${PUID:-1000}:${PGID:-1000}"
```

Perhatikan `user` bidang di sini: `PUID` dan `PGID` disuntikkan melalui variabel lingkungan untuk memastikan bahwa proses di dalam wadah berjalan dengan identitas yang cocok dengan pengguna host. Detail ini penting karena masalah izin sulit untuk di-debug begitu muncul.

## Manajemen versi: versi bawaan dengan penggantian waktu proses

Menyematkan versi image Docker sangat penting agar dapat direproduksi. Namun dalam perkembangan sebenarnya, sering kali kita perlu menguji versi yang lebih baru atau segera memperbaiki bug. Jika kami harus membangun kembali gambar setiap saat, alur kerjanya akan menjadi sangat tidak efisien.

Strategi HagiCode adalah **versi tetap sebagai default, dengan penggantian waktu proses sebagai mekanisme ekstensi**. Ini adalah kompromi rekayasa pragmatis antara stabilitas dan fleksibilitas.

Versi pin Dockerfile.template di sini:

```dockerfile
USER hagicode
WORKDIR /home/hagicode

# Configure the global npm install path
RUN mkdir -p /home/hagicode/.npm-global && \
    npm config set prefix '/home/hagicode/.npm-global'

# Install CLI tools using pinned versions
RUN npm install -g @anthropic-ai/claude-code@2.1.71 && \
    npm install -g @openai/codex@0.112.0 && \
    npm install -g opencode-ai@1.2.25 && \
    npm cache clean --force
```

docker-entrypoint.sh mendukung penggantian waktu proses:

```bash
install_cli_override_if_needed() {
    local package_name="$2"
    local override_version="$5"

    if [ -n "$override_version" ]; then
        gosu hagicode npm install -g "${package_name}@${override_version}"
    fi
}

# Example usage
install_cli_override_if_needed "" "@anthropic-ai/claude-code" "" "" "${CLAUDE_CODE_CLI_VERSION}"
```

Ini memungkinkan Anda menguji versi baru melalui variabel lingkungan tanpa membuat ulang image:

```bash
docker run -e CLAUDE_CODE_CLI_VERSION=2.2.0 myimage
```

Desain ini praktis karena tidak ada seorang pun yang ingin membuat ulang suatu gambar setiap kali mereka menguji fitur baru.

## Injeksi konfigurasi otomatis

Selain mengonfigurasi alat CLI secara manual, beberapa skenario memerlukan injeksi konfigurasi otomatis. Contoh paling umum adalah token API.

```bash
if [ -n "$ANTHROPIC_AUTH_TOKEN" ]; then
    mkdir -p /home/hagicode/.claude
    cat > /home/hagicode/.claude/settings.json <<EOF
{
  "env": {
    "ANTHROPIC_AUTH_TOKEN": "${ANTHROPIC_AUTH_TOKEN}"
  }
}
EOF
    chown -R hagicode:hagicode /home/hagicode/.claude
fi
```

Ada dua hal yang penting di sini: **meneruskan informasi sensitif melalui variabel lingkungan alih-alih mengkodekannya ke dalam gambar**, dan pastikan kepemilikan file konfigurasi disetel dengan benar, jika tidak, alat CLI tidak akan dapat membacanya.

## Praktik terbaik dan daftar periksa jebakan

### Masalah ketidakcocokan izin

Ini adalah jebakan yang paling mudah untuk dijebak. Pengguna host memiliki UID 1001, sedangkan container menggunakan 1000, sehingga file yang dibuat di satu sisi tidak dapat diakses di sisi lain.

```bash
# Correct approach: make the container match the host user
docker run \
    -e PUID=$(id -u) \
    -e PGID=$(id -g) \
    myimage
```

Masalah ini sangat umum terjadi, dan dapat membuat frustasi saat pertama kali Anda mengalaminya.

### Konfigurasi hilang setelah kontainer dimulai ulang

Jika Anda masuk lagi setelah setiap restart, periksa apakah Anda lupa memasang volume persisten:

```yaml
volumes:
  - claude-data:/home/hagicode/.claude
```

Tidak ada yang lebih membuat frustrasi daripada menyiapkan konfigurasi secara hati-hati hanya untuk melihatnya menghilang.

### Cara yang tepat untuk mengupgrade versi

Jangan lari `npm install -g` langsung di dalam wadah yang sedang berjalan. Pendekatan yang benar adalah:

1. Tetapkan variabel lingkungan untuk memicu pemasangan override.
2. Atau bangun kembali gambarnya.

```bash
# Option 1: runtime override
docker run -e CLAUDE_CODE_CLI_VERSION=2.2.0 myimage

# Option 2: rebuild the image
docker build -t myimage:v2 .
```

Ada lebih dari satu jalan menuju Roma, namun beberapa jalan lebih mulus dibandingkan yang lain.

### Daftar periksa penguatan keamanan

- Meneruskan token API melalui variabel lingkungan alih-alih menuliskannya ke dalam gambar.
- Setel izin file konfigurasi ke `600`.
- Selalu jalankan aplikasi sebagai pengguna non-root.
- Perbarui versi CLI secara berkala untuk memperbaiki kerentanan keamanan.

Keamanan selalu penting, namun tantangan sebenarnya adalah menerapkannya secara konsisten dalam praktik.

## Memperluas dukungan untuk alat CLI baru

Jika Anda ingin mendukung alat CLI baru di masa mendatang, hanya ada tiga langkah:

1. **Dockerfile.template**: tambahkan langkah instalasi.
2. **docker-entrypoint.sh**: tambahkan logika penggantian versi.
3. **docker-compose-builder-web**: menambahkan pemetaan volume persisten.

Desain berbasis template ini membuat ekstensi menjadi sederhana tanpa mengubah logika inti.

## Kesimpulan

Menjalankan alat AI CLI di container Docker melibatkan tiga tantangan inti: **izin pengguna, persistensi konfigurasi, dan manajemen versi**. Dengan menggabungkan pengguna khusus, isolasi volume bernama, dan penggantian berbasis variabel lingkungan, proyek HagiCode membangun arsitektur penerapan yang aman dan fleksibel.

Poin desain utama:

- **Isolasi pengguna**: Membuat pengguna khusus selama tahap pembuatan gambar, dengan dukungan waktu proses untuk dinamis `PUID`/`PGID` pemetaan.
- **Strategi persistensi**: Setiap alat CLI mendapatkan volume bernama sendiri, sehingga memulai ulang tidak memengaruhi konfigurasi.
- **Fleksibilitas versi**: Default yang diperbaiki memastikan reproduktifitas, sementara penggantian waktu proses memberikan ruang untuk pengujian.
- **Konfigurasi otomatis**: Konfigurasi sensitif dapat dimasukkan secara otomatis melalui variabel lingkungan.

Solusi ini telah berjalan secara stabil di proyek HagiCode selama beberapa waktu, dan saya berharap solusi ini dapat memberikan titik referensi yang berguna bagi pengembang dengan kebutuhan serupa.
