## Dari Docusaurus 3.x hingga Astro 5.x: Retrospektif Migrasi Situs HagiCode

> Artikel ini mengulas kembali migrasi lengkap situs resmi HagiCode dari Docusaurus 3.x ke Astro 5.x. Kami akan mempelajari lebih dalam bagaimana Arsitektur Kepulauan Astro membantu kami mengatasi hambatan kinerja sekaligus mempertahankan aset komponen React yang ada, memberikan peningkatan dalam kecepatan pembuatan dan kinerja pemuatan.

## Latar Belakang

Pada bulan Januari 2026, kami melakukan "transplantasi jantung" di situs resmi HagiCode dengan memigrasikan sepenuhnya kerangka intinya dari Docusaurus 3.x ke Astro 5.x. Ini bukanlah penulisan ulang yang impulsif, namun keputusan teknis yang dipertimbangkan dengan cermat.

Sebelum migrasi, situs kami telah selesai secara fungsional, namun mulai menunjukkan beberapa "masalah kemewahan" klasik: artefak bangunan yang membengkak, muatan JavaScript yang berlebihan, dan kecepatan memuat halaman yang kurang ideal pada halaman dokumentasi yang rumit. Sebagai proyek asisten pengkodean AI, HagiCode memerlukan dokumentasi dan pembaruan fitur secara berkala, sehingga efisiensi pembangunan secara langsung memengaruhi kecepatan rilis. Pada saat yang sama, kami ingin situs tersebut lebih ramah mesin pencari (SEO) sehingga lebih banyak pengembang dapat menemukan proyek tersebut.

Untuk mengatasi permasalahan ini, kami mengambil keputusan yang berani: membangun kembali seluruh sistem di Astro. Dampak dari keputusan tersebut mungkin lebih besar dari yang Anda perkirakan. Saya akan segera membahas detailnya.

## Tentang HagiCode

Pendekatan migrasi situs yang dibagikan dalam artikel ini berasal dari pengalaman langsung kami di [Kode Hagi](https://github.com/HagiCode-org/site) proyek.

HagiCode adalah asisten pengkodean AI yang berfokus pada peningkatan efisiensi pengembangan. Kami tidak hanya peduli pada iterasi fitur inti, namun juga pada pengalaman pengembang. Pemfaktoran ulang situs ini juga dimaksudkan untuk memberikan pengalaman tercepat kepada pengguna saat menelusuri dokumen dan situs web resmi kami.

## Mengapa Meninggalkan Ekosistem Docusaurus yang Sudah Dewasa?

Dalam ekosistem React, Docusaurus telah lama menjadi "jawaban standar" untuk situs dokumentasi. Ini berfungsi dengan baik, menawarkan ekosistem plugin yang kaya, dan memiliki komunitas yang aktif. Namun seiring HagiCode memperoleh lebih banyak fitur, kami juga merasakan keterbatasannya:

1.  **Kemacetan kinerja**: Docusaurus pada dasarnya adalah React SPA (aplikasi satu halaman). Bahkan jika Anda hanya menulis halaman statis, klien masih perlu memuat runtime React dan menghidrasi halaman, yang tidak terlalu berat untuk halaman dokumen sederhana.
2.  **Ukuran aset besar**: Meskipun halaman berisi konten yang sangat sedikit, ukuran JS yang dibundel tetap relatif tetap. Itu tidak ideal untuk pengguna seluler atau kondisi jaringan yang buruk.
3.  **Fleksibilitas terbatas**: Meskipun dapat diperluas, kami menginginkan kontrol tingkat rendah yang lebih besar terhadap pipeline build.

Astro tiba pada waktu yang tepat untuk menyelesaikan masalah ini. Ini memperkenalkan "Arsitektur Pulau" baru: secara default, Astro menghasilkan HTML statis tanpa JavaScript, dan hanya komponen yang memerlukan interaktivitas yang "diaktifkan" dan memuat JS. Itu berarti sebagian besar situs kami menjadi HTML murni dan memuat dengan sangat cepat.

## Strategi Migrasi Inti: Transisi Arsitektur yang Lancar

Migrasi bukan sekadar salin dan tempel. Untuk itu diperlukan perubahan pola pikir. Kami beralih dari model "semua React" Docusaurus ke model "Core + Islands" Astro.

### 1. Membangun Kembali Sistem Konfigurasi

Pertama, kami harus pindah `docusaurus.config.ts` untuk `astro.config.mjs`. Ini bukan hanya penggantian nama file, tetapi penulisan ulang perutean dan logika pembangunan.

Di Docusaurus, semuanya adalah sebuah plugin. Di Astro, semuanya merupakan integrasi. Kami perlu mendefinisikan ulang jalur dasar situs, membangun mode keluaran (statis vs SSR), dan strategi pengoptimalan aset.

**Sebelum migrasi:**
```typescript
// docusaurus.config.ts
export default {
  title: 'HagiCode',
  url: 'https://hagicode.com',
  baseUrl: '/',
  // ... more configuration
};
```

**Setelah migrasi:**
```javascript
// astro.config.mjs
import { defineConfig } from 'astro/config';
import react from '@astrojs/react';

export default defineConfig({
  integrations: [react()],
  site: 'https://hagicode.com',
  base: '/',
  // Optimization settings for static assets
  build: {
    inlineStylesheets: 'auto',
  },
});
```

### 2. Apa yang Harus Dipertahankan dan Apa yang Harus Difaktorkan Ulang pada Komponen React

Ini adalah bagian migrasi yang paling menyakitkan. Situs kami saat ini memiliki banyak komponen React, seperti Tab, penyorotan kode, tombol umpan balik, dan banyak lagi. Membuangnya akan sia-sia, namun menyimpan semuanya akan membuat muatan JavaScript menjadi terlalu berat.

HagiCode mengadopsi strategi **hidrasi progresif**:
*   **Komponen statis murni**: Untuk konten presentasi seperti header, footer, dan dokumentasi teks biasa, kami menulis ulangnya sebagai komponen Astro (`.astro` file) dan merendernya langsung ke HTML pada waktu pembuatan.
*   **Pulau interaktif**: Untuk komponen yang harus tetap interaktif, seperti pengalih tema, peralihan tab, dan tombol penyalinan blok kode, kami mempertahankan implementasi React dan menambahkan `client:load` atau `client:visible` arahan.

Misalnya, komponen Tab yang biasa kami gunakan dalam dokumentasi:

```jsx
// src/components/Tabs.jsx
import { useState } from 'react';
import './Tabs.css'; // Import styles

export default function Tabs({ items }) {
  const [activeIndex, setActiveIndex] = useState(0);
  // ... state logic
  return (
    <div className="tabs-wrapper">
      {/* Rendering logic */}
    </div>
  );
}
```

Saat digunakan dalam Markdown, kami secara eksplisit memberi tahu Astro: "Komponen ini memerlukan JS."

```markdown
// src/content/docs/example.mdx
import Tabs from '../../components/Tabs.jsx';

<!-- Load JS only when the component enters the viewport -->
<Tabs client:visible items={...} />
```

Dengan cara ini, komponen interaktif di luar area pandang tidak bersaing untuk mendapatkan bandwidth, sehingga sangat meningkatkan kecepatan pemuatan layar pertama.

### 3. Mengadaptasi Sistem Styling: Dari Modul CSS ke Scoped CSS

Docusaurus mendukung Modul CSS secara default, sementara Astro mendorong Scoped CSS melalui `<style>` menandai. Ide inti di balik keduanya adalah isolasi gaya, tetapi sintaksisnya berbeda.

Selama migrasi HagiCode, kami mengubah Modul CSS paling kompleks menjadi gaya cakupan Astro. Ini sebenarnya merupakan hal yang baik, karena di `.astro` file gaya dan templat berada dalam file yang sama, sehingga pemeliharaan menjadi lebih intuitif.

**Sebelum melakukan pemfaktoran ulang:**
```css
/* Tabs.module.css */
.wrapper { background: var(--ifm-background-color); }
```

**Setelah pemfaktoran ulang (Astro Scoped):**
```html
<!-- Tabs.astro -->
<div class="tabs-wrapper">
  <slot />
</div>

<style>
  .tabs-wrapper {
    /* Use CSS variables directly to adapt to the theme */
    background: var(--bg-color);
    padding: 1rem;
  }
</style>
```

Pada saat yang sama, kami menyatukan sistem variabel CSS global dan menggunakan kemampuan Astro yang sadar lingkungan untuk memastikan mode gelap beralih dengan lancar di seluruh halaman.

## Kendala yang Kami Hadapi dalam Praktek dan Cara Kami Menyelesaikannya

Selama migrasi HagiCode sebenarnya, kami mengalami beberapa masalah. Berikut beberapa yang paling umum.

### 1. Titik Sakit Variabel Jalur dan Lingkungan

HagiCode mendukung penerapan subjalur, seperti penerapan di bawah subdirektori Halaman GitHub. Di Dokusaurus, `baseUrl` ditangani secara otomatis. Namun di Astro, kami harus lebih berhati-hati saat menangani tautan gambar dan permintaan API.

Kami memperkenalkan mekanisme variabel lingkungan untuk mengelolanya secara konsisten:

```javascript
// Handle paths in the build script
const getBasePath = () => import.meta.env.VITE_SITE_BASE || '/';
```

Pastikan untuk tidak melakukan hardcode jalur yang dimulai dengan `/` dalam kode Anda. Dalam pengembangan versus produksi, atau setelah konfigurasi a `base` jalur, hal itu dapat menyebabkan 404 pada aset.

### 2. Kompatibilitas Skrip CommonJS

Situs lama kami memiliki beberapa skrip Node.js yang digunakan untuk tugas-tugas seperti mengambil data Metrik secara otomatis dan memperbarui peta situs, dan skrip tersebut ditulis dalam CommonJS (`require`). Astro dan alat pembangunan modern telah sepenuhnya menggunakan Modul ES (`import`/`export`).

Jika Anda juga memiliki skrip serupa, ingatlah untuk memfaktorkan ulang semuanya ke Modul ES. Ini adalah arah pergerakan ekosistem, dan semakin cepat Anda melakukan perubahan, semakin sedikit masalah yang akan Anda alami nantinya.

```javascript
// Old way
const fs = require('fs');

// New way
import fs from 'fs';
```

### 3. Jangan Lupakan SEO dan Redirect

Mesin pencari telah mengindeks halaman Docusaurus lama HagiCode. Jika Anda beralih langsung ke Astro dan struktur URL berubah, Anda mungkin mendapatkan banyak 404 dan penurunan besar dalam peringkat pencarian.

Kami mengonfigurasi aturan pengalihan di Astro:

```javascript
// astro.config.mjs
export default defineConfig({
  redirects: {
    '/docs/old-path': '/docs/new-path',
    // Map old links to new links in bulk
  }
});
```

Atau Anda dapat menanganinya di lapisan konfigurasi server. Pastikan link lama dapat dialihkan 301 ke alamat baru, karena ini penting untuk SEO.

## Ringkasan

Bagi HagiCode, migrasi dari Docusaurus ke Astro bukan sekadar peningkatan kerangka kerja. Ini juga merupakan implementasi praktis dari filosofi "kinerja yang utama".

**Apa yang kami peroleh:**
*   **Skor Lighthouse yang Luar Biasa**: Setelah migrasi, skor kinerja situs HagiCode dengan mudah mendekati skor sempurna.
*   **Kecepatan build lebih cepat**: Kemampuan build inkremental Astro memangkas waktu rilis untuk pembaruan dokumentasi menjadi setengahnya.
*   **Fleksibilitas yang dipertahankan**: Dengan Arsitektur Kepulauan, kami tidak mengorbankan fitur interaktif apa pun dan masih dapat menggunakan React jika diperlukan.

Jika Anda juga mengelola situs yang berorientasi pada dokumentasi dan kesulitan dengan ukuran paket atau kecepatan memuat, Astro layak untuk dicoba. Meskipun proses migrasi memerlukan beberapa operasi, seperti mengganti nama PCode menjadi HagiCode dan memindahkan komponen satu per satu, pengalaman pengguna sehalus sutra yang Anda dapatkan sebagai imbalannya menjadikannya sangat berharga.

Sistem pembangunan yang dibagikan dalam artikel ini adalah pendekatan persis yang kami kembangkan melalui uji coba nyata saat membangun HagiCode. Jika menurut Anda pendekatan ini berharga, hal ini mencerminkan kekuatan teknis kami, dan HagiCode sendiri mungkin layak untuk dicermati juga.
