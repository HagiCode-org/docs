## Selamat tinggal Script Hell: Mengapa Kami Memilih C# untuk Sistem Pembangunan Modern

> Penjelasan mendalam tentang bagaimana proyek HagiCode menggunakan Nuke untuk membangun pipeline build otomatis yang aman untuk tipe, lintas platform, dan sangat dapat diperluas, yang sepenuhnya mengatasi masalah pemeliharaan skrip build tradisional.

## Latar Belakang

Sepanjang perjalanan panjang pengembangan perangkat lunak, kata "membangun" cenderung menginspirasi rasa cinta dan frustrasi. Senang sekali, karena dengan satu klik, kode berubah menjadi produk, dan itu adalah salah satu momen paling memuaskan dalam pemrograman. Frustrasi, karena menyimpan tumpukan skrip build yang berantakan bisa menjadi mimpi buruk.

Di banyak proyek, kami terbiasa menulis skrip dengan Python atau mengandalkan file konfigurasi XML. Bayangkan saja rasa takut diperintah `<property>` blok. Namun seiring dengan meningkatnya kompleksitas proyek, terutama dalam proyek seperti HagiCode yang mencakup pekerjaan frontend dan backend, berbagai platform, dan berbagai bahasa, pendekatan pembangunan tradisional mulai menunjukkan batasnya. Logika skrip yang tersebar, tidak ada pemeriksaan jenis, dukungan IDE yang lemah... masing-masing masalah ini menjadi jebakan kecil yang membuat tim tersandung dari waktu ke waktu.

Untuk mengatasi masalah tersebut, proyek HagiCode memperkenalkan **Nuke**, sistem pembangunan modern berdasarkan C#. Ini bukan sekedar alat. Ini adalah cara berpikir yang berbeda tentang alur kerja build. Hari ini, kita akan melihat alasan kami memilihnya dan bagaimana hal ini telah meningkatkan pengalaman pengembangan kami secara signifikan.

## Analisis Inti: Mengapa Nuke?

Anda mungkin bertanya-tanya: "Sudah ada begitu banyak sistem pembangunan, seperti Make, Gradle, atau bahkan skrip Shell biasa. Mengapa harus menggunakan sistem berbasis C#?"

Itu adalah pertanyaan yang wajar. Daya tarik inti Nuke adalah ia menghadirkan fitur-fitur bahasa pemrograman yang sudah kita kenal dengan baik ke dalam dunia skrip build.

### 1. Modularisasi Alur Kerja Pembangunan: Seni Menargetkan

Nuke memiliki filosofi desain yang sangat jelas: **semuanya adalah target**.

Dalam skrip tradisional, mudah untuk mendapatkan ratusan baris logika eksekusi linier yang saling terkait. Di Nuke, kami memecah proses pembangunan menjadi proses independen `Target`s. Setiap target melakukan satu hal, misalnya:

- `Clean`: membersihkan direktori keluaran
- `Restore`: memulihkan paket ketergantungan
- `Compile`: kompilasi kodenya
- `Test`: menjalankan pengujian unit

Desain ini sangat selaras dengan prinsip tanggung jawab tunggal. Seperti halnya elemen penyusun, Target ini dapat digabungkan secara bebas. Lebih penting lagi, Nuke memungkinkan kita untuk menentukan ketergantungan antar Target. Misalnya saja jika Anda ingin lari `Test`, sistem akan secara otomatis memeriksa apakah `Compile` sudah dieksekusi. Jika kamu mau `Compile`, lalu `Restore` secara alami didahulukan.

Grafik ketergantungan ini membuat logika lebih jelas dan secara signifikan meningkatkan efisiensi eksekusi, karena Nuke dapat secara otomatis menganalisis jalur eksekusi optimal.

### 2. Keamanan Ketik: Selamat Tinggal Mimpi Buruk Kesalahan Ketik

Siapa pun yang pernah menulis skrip build dengan Python mungkin pernah mengalami situasi canggung ini: skrip berjalan selama lima menit dan kemudian gagal karena `Confi.guration` salah eja, atau karena string diteruskan ke tempat yang diharapkan parameter numeriknya.

Keuntungan terbesar menulis skrip build di C# adalah **keamanan tipe**. Artinya:

- **Pemeriksaan waktu kompilasi**: IDE dapat memberi tahu Anda apa yang salah saat Anda mengetik alih-alih membuat Anda menunggu hingga waktu proses.
- **Pemfaktoran ulang yang aman**: jika Anda ingin mengganti nama variabel atau metode, IDE dapat menanganinya dalam satu langkah tanpa memaksa Anda melakukan pencarian dan penggantian global yang berisiko.
- **Penyelesaian cerdas**: IntelliSense yang canggih dapat menyelesaikan kode untuk Anda secara otomatis, sehingga Anda tidak perlu terus-terusan membuka dokumentasi untuk mengingat API yang tidak jelas.

### 3. Dukungan Lintas Platform: Satu Pengalaman Pembangunan yang Konsisten

Di masa lalu, kami menulis `.bat` pada Windows dan `.sh` di Linux, dan mungkin menambahkan skrip Python untuk menjembatani kesenjangan tersebut. Sekarang, di mana pun .NET Core, atau .NET 5+ saat ini, dapat dijalankan, Nuke juga dapat dijalankan.

Itu berarti anggota tim dapat menggunakan Windows, Linux, atau macOS, dan apakah mereka lebih suka Visual Studio, VS Code, atau Rider, semuanya menjalankan logika yang sama persis. Hal ini sangat mengurangi masalah yang disebabkan oleh perbedaan lingkungan, terutama masalah klasik "berfungsi di mesin saya".

### 4. Parameter dan Manajemen Konfigurasi

Nuke menyediakan mekanisme penguraian parameter yang sangat elegan. Anda tidak perlu menguraikannya secara manual `string[] args`. Anda cukup mendefinisikan properti dan membubuhi keterangannya `[Parameter]`, dan Nuke secara otomatis menangani pemetaan dari argumen baris perintah dan file konfigurasi.

Misalnya, kita dapat mendefinisikan konfigurasi build dengan sangat mudah:

```csharp
[Parameter("Configuration to build - Default is 'Debug'")]
readonly Configuration BuildConfiguration = IsLocalBuild ? Configuration.Debug : Configuration.Release;

Target Compile => _ => _
    .DependsOn(Restore)
    .Executes(() =>
    {
        // Use BuildConfiguration here; it is type-safe
        DotNetBuild(s => s
            .SetConfiguration(BuildConfiguration)
            .SetProjectFile(SolutionFile));
    });
```

Gaya ini intuitif dan tahan terhadap kesalahan.

## Panduan Praktis: Cara Mempraktikkannya dalam Sebuah Proyek

Teori saja tidak membawa kita kemana-mana. Mari kita lihat bagaimana kami menerapkan pendekatan ini dalam proyek HagiCode.

### 1. Rencanakan Struktur Proyek

Kami tidak ingin membuat skrip yang mengacaukan akar proyek, dan kami juga tidak ingin struktur direktori terlalu dalam sehingga menyerupai proyek Java tertentu. Jadi kami menempatkan semua file build terkait Nuke dalam satu file `nukeBuild/` direktori.

Manfaatnya sangat jelas:

- Akar proyek tetap bersih.
- Logika build tetap kohesif dan lebih mudah dikelola.
- Anggota tim baru dapat langsung melihat, "Di sinilah logika terkait pembangunan berada."

### 2. Rancang Rantai Ketergantungan Target yang Jelas

Saat merancang Target, kami mengikuti satu prinsip: **atomisitas plus aliran ketergantungan**.

Setiap Target harus cukup kecil untuk melakukan satu hal. Misalnya, `Clean` sebaiknya hanya menghapus file dan tidak menangani kemasan di sampingnya secara diam-diam.

Alur ketergantungan yang direkomendasikan kira-kira seperti ini:

`Clean` -> `Restore` -> `Compile` -> `Test` -> `Pack`

Tentu saja hal ini tidak mutlak. Misalnya, jika Anda hanya ingin menjalankan pengujian dan tidak ingin mengemas apa pun, Nuke mengizinkan Anda untuk mengeksekusinya `nuke Test` secara langsung, dan secara otomatis akan menangani yang diperlukan `Restore` dan `Compile` langkah pertama.

### 3. Penanganan Kesalahan dan Pencatatan yang Kuat

Apa hal yang paling membuat frustrasi tentang skrip pembuatan? Pesan kesalahan tidak jelas. Build gagal yang hanya mencatat "Kesalahan: 1" sudah cukup untuk membuat siapa pun menjadi gila.

Di Nuke, karena kita bisa langsung menggunakan penanganan pengecualian C#, kita bisa menangkap dan melaporkan kesalahan dengan lebih presisi.

```csharp
Target Publish => _ => _
    .DependsOn(Test)
    .Executes(() =>
    {
        try 
        {
            // Try publishing to NuGet
            DotNetNuGetPush(s => s
                .SetTargetPath(ArtifactPath)
                .SetSource("https://api.nuget.org/v3/index.json")
                .SetApiKey(ApiKey));
        }
        catch (Exception ex)
        {
            Log.Error($"Publishing failed. Team, please check whether the key is correct: {ex.Message}");
            throw; // Make sure the build process exits with a non-zero exit code
        }
    });
```

### 4. Gunakan Pengujian untuk Melindungi Kualitas

Skrip pembuatan juga merupakan kode, yang berarti skrip tersebut juga memerlukan pengujian. Nuke memungkinkan kita menulis pengujian untuk alur kerja build sehingga ketika kita mengubah logika build, kita tidak merusak proses rilis yang ada secara tidak sengaja. Hal ini sangat penting dalam jalur integrasi berkelanjutan.

## Ringkasan

Dengan memperkenalkan Nuke, proses pembuatan HagiCode menjadi lebih lancar dari sebelumnya. Ini bukan sekedar penggantian alat. Ini merupakan peningkatan dalam pola pikir teknik.

**Apa yang kami peroleh?**

- **Kemampuan pemeliharaan**: konfigurasi sebagai kode, logika yang lebih jelas, dan jalur orientasi yang lebih cepat untuk kontributor baru.
- **Stabilitas**: pengetikan yang kuat menghilangkan lebih dari 90% kesalahan tingkat rendah.
- **Konsistensi**: pengalaman lintas platform terpadu yang menghilangkan penyimpangan lingkungan.

Jika menulis skrip build biasanya terasa seperti "merasakan jalan menembus kegelapan", maka menggunakan Nuke terasa seperti "berjalan di malam hari dengan lampu menyala". Jika Anda lelah mempertahankan skrip yang sulit di-debug, pertimbangkan untuk memindahkan logika build Anda ke dunia C#. Anda mungkin menemukan bahwa build juga bisa menjadi elegan.
