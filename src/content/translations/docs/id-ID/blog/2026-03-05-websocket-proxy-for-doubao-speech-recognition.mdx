## Memecahkan Tantangan Otentikasi WebSocket Browser: Pola Proxy Praktis untuk Pengenalan Ucapan Doubao

> API WebSocket browser tidak mendukung header HTTP khusus, sehingga menimbulkan tantangan bagi layanan pengenalan suara yang memerlukan data autentikasi di header. Artikel ini membagikan cara kami memecahkan masalah tersebut di proyek HagiCode dengan pola proksi backend, dan bagaimana pendekatan tersebut berkembang dari eksperimen taman bermain hingga penggunaan produksi.


## Latar Belakang

Saat kami mulai membangun pengenalan ucapan untuk proyek HagiCode, kami dengan percaya diri memilih layanan pengenalan ucapan Doubao dari ByteDance. Desain awalnya sederhana: biarkan frontend terhubung langsung ke layanan WebSocket Doubao. Seberapa sulitkah hal itu? Buka saja koneksi dan kirim beberapa data, bukan?

Kemudian muncul kejutan: API Doubao memerlukan informasi autentikasi untuk diteruskan melalui header HTTP, termasuk hal-hal seperti `accessToken` dan `secretKey`. Hal ini langsung menjadi canggung, karena browser WebSocket API tidak mendukung pengaturan header khusus.

Jadi apa yang Anda lakukan ketika browser tidak mengizinkan Anda mengirimkannya?

Saat itu, kami mempertimbangkan dua opsi:

1. Masukkan kredensial ke dalam parameter kueri URL - sederhana dan blak-blakan
2. Tambahkan lapisan proxy di backend - sekilas lebih banyak pekerjaan

Opsi pertama memperlihatkan kredensial secara langsung dalam kode frontend dan penyimpanan lokal. Apakah itu benar-benar aman? Saya merasa tidak nyaman dengan hal itu. Selain itu, beberapa API memerlukan verifikasi berbasis header, sehingga pendekatan ini bahkan tidak dapat dilakukan.

Pada akhirnya, kami memilih opsi kedua: menerapkan proxy WebSocket di backend. Secara kebetulan, pola ini pertama kali divalidasi di lingkungan taman bermain kami, dan hanya setelah kami memastikan stabilitasnya barulah kami memindahkannya ke produksi. Lagi pula, tidak ada seorang pun yang ingin produksinya berfungsi ganda sebagai eksperimen laboratorium.

## Analisis Tantangan Teknis

### Batasan WebSocket peramban

API WebSocket standar terlihat sangat sederhana:

```javascript
const ws = new WebSocket('wss://example.com/ws');
```

Namun kesederhanaan itulah yang menjadi letak masalahnya - ia hanya meneruskan parameter di URL, dan tidak dapat menyetel header seperti yang dapat dilakukan permintaan HTTP:

```javascript
// This is not supported in the WebSocket API
const ws = new WebSocket('wss://example.com/ws', {
  headers: {
    'Authorization': 'Bearer token'
  }
});
```

Dan itulah persoalan intinya. Untuk layanan seperti pengenalan ucapan Doubao yang bergantung pada autentikasi berbasis header, batasan ini merupakan pemblokiran yang sulit.

Begitu Anda menerima batasan itu, arsitekturnya harus berubah.

### Keputusan desain arsitektur

Saat merancang solusi, kami membandingkan trade-off dengan hati-hati.

**Keputusan 1: Memilih pola proxy**

Kami membandingkan dua pendekatan:

| Pilihan | Kelebihan | Kontra | Keputusan |
|------|------|------|------|
| WebSocket Asli | Ringan, sederhana, penerusan langsung | Manajemen koneksi harus ditangani secara manual | Terpilih |
| SinyalR | Koneksi ulang otomatis, pengetikan kuat | Terlalu rumit, ketergantungan ekstra | Ditolak |

Kami akhirnya memilih WebSocket asli. Sejujurnya, ini adalah opsi paling ringan dan lebih cocok untuk penerusan aliran biner dua arah yang sederhana. Menggunakan SignalR akan terasa seperti rekayasa berlebihan untuk kasus penggunaan ini, dan dapat menambah latensi ekstra.

**Keputusan 2: Strategi pengelolaan koneksi**

Kami mengadopsi model "satu koneksi, satu sesi" - setiap koneksi WebSocket frontend dipetakan ke koneksi backend Doubao independennya sendiri.

Manfaatnya sangat jelas:
- Mudah diterapkan dan selaras dengan pola penggunaan umum
- Lebih mudah untuk melakukan debug dan memecahkan masalah
- Isolasi sumber daya yang baik, mencegah interferensi antar sesi

Sederhananya, solusi langsung terkadang merupakan solusi terbaik. Kompleksitas tidak secara otomatis membuat suatu desain menjadi lebih baik.

**Keputusan 3: Menyimpan data autentikasi**

Kredensial disimpan dalam file konfigurasi backend (`appsettings.yml` atau variabel lingkungan) dan dimuat melalui injeksi ketergantungan:

- Model konfigurasi sederhana yang sesuai dengan konvensi backend yang ada
- Data sensitif tidak pernah mencapai frontend
- Mendukung pengaturan multi-lingkungan untuk pengembangan, pengujian, dan produksi

Tingkat pemisahan itu penting. Tidak seorang pun ingin kredensial beredar di tempat yang tidak seharusnya.

### Desain aliran data

Aliran data keseluruhan terlihat seperti ini:

```
Frontend (browser)
  │
  │ ws://backend/api/voice/ws
  │ WebSocket (binary)
  ▼
Backend (proxy)
  │
  │ wss://openspeech.bytedance.com/
  │ (with authentication headers)
  ▼
Doubao API
```

Alurnya sendiri tidak rumit:
1. Frontend terhubung ke proksi backend melalui WebSocket
2. Proksi backend menerima data audio dan terhubung ke Doubao API dengan header yang diautentikasi
3. Doubao API mengembalikan hasil pengenalan, dan proksi meneruskannya ke frontend
4. Seluruh proses tetap sepenuhnya asinkron dengan streaming dua arah

Setelah tanggung jawab dibagi dengan jelas, desainnya menjadi sangat alami.

## Implementasi komponen inti

### 1. Konfigurasi titik akhir WebSocket

```csharp
app.Map("/ws", async context =>
{
    if (context.WebSockets.IsWebSocketRequest)
    {
        // Read configuration from query parameters
        var appId = context.Request.Query["appId"];
        var accessToken = context.Request.Query["accessToken"];

        // Validate required parameters
        if (string.IsNullOrEmpty(appId) || string.IsNullOrEmpty(accessToken))
        {
            context.Response.StatusCode = 400;
            return;
        }

        // Accept the WebSocket connection
        using var webSocket = await context.WebSockets.AcceptWebSocketAsync();

        // Message handling loop
        var buffer = new byte[4096];
        while (!webSocket.CloseStatus.HasValue)
        {
            var result = await webSocket.ReceiveAsync(buffer, CancellationToken.None);

            if (result.MessageType == WebSocketMessageType.Close)
            {
                await webSocket.CloseAsync(
                    result.CloseStatus.Value,
                    result.CloseStatusDescription,
                    CancellationToken.None);
                break;
            }

            // Process audio data
            await HandleAudioDataAsync(buffer, result.Count);
        }
    }
});
```

### 2. Manajemen sesi

```csharp
public class DoubaoSessionManager : IDoubaoSessionManager
{
    private readonly ConcurrentDictionary<string, DoubaoSession> _sessions = new();

    public DoubaoSession CreateSession(string connectionId)
    {
        var session = new DoubaoSession(connectionId);
        _sessions[connectionId] = session;
        return session;
    }

    public async Task SendAudioAsync(string connectionId, byte[] audioData)
    {
        if (_sessions.TryGetValue(connectionId, out var session))
        {
            await session.SendAudioAsync(audioData);
        }
    }

    public void RemoveSession(string connectionId)
    {
        if (_sessions.TryRemove(connectionId, out var session))
        {
            session.Dispose();
        }
    }
}
```

Menggunakan `ConcurrentDictionary` karena manajemen sesi berarti keamanan thread sebagian besar ditangani untuk kami. Setiap sambungan masuk mendapatkan sesinya sendiri, dan pembersihan terjadi secara otomatis saat sambungan terputus.

## Integrasi frontend dalam praktiknya

### Koneksi WebSocket

```javascript
class DoubaoVoiceClient {
    constructor(config) {
        this.config = config;
        this.ws = null;
    }

    async connect() {
        const url = new URL(this.config.wsUrl);
        // Add query parameters
        Object.entries(this.config.params).forEach(([key, value]) => {
            url.searchParams.set(key, value);
        });

        this.ws = new WebSocket(url);

        return new Promise((resolve, reject) => {
            this.ws.onopen = () => {
                console.log('[DoubaoVoice] Connected');
                resolve();
            };

            this.ws.onmessage = (event) => {
                this._handleMessage(JSON.parse(event.data));
            };

            this.ws.onerror = reject;
        });
    }

    _handleMessage(message) {
        switch (message.type) {
            case 'status':
                this._handleStatus(message.payload);
                break;
            case 'result':
                this.onResult?.(message.payload);
                break;
            case 'error':
                console.error('[DoubaoVoice] Error:', message.payload);
                break;
        }
    }
}

// Usage example
const client = new DoubaoVoiceClient({
    wsUrl: 'ws://localhost:5000/ws',
    params: {
        appId: 'your-app-id',
        accessToken: 'your-access-token',
        sampleRate: 16000,
        bitsPerSample: 16,
        channels: 1
    }
});
```

### Pengambilan dan transmisi audio

Menggunakan `AudioWorkletNode` untuk pemrosesan audio memberikan kinerja yang lebih baik:

```javascript
// audio-worklet.js
class AudioProcessorWorklet extends AudioWorkletProcessor {
    process(inputs, outputs, parameters) {
        const input = inputs[0]?.[0];
        if (!input) return true;

        // Convert to 16-bit PCM
        const pcm = new Int16Array(input.length);
        for (let i = 0; i < input.length; i++) {
            pcm[i] = Math.max(-32768, Math.min(32767, input[i] * 32767));
        }

        this.port.postMessage({
            type: 'audioData',
            data: pcm.buffer
        }, [pcm.buffer]);

        return true;
    }
}

registerProcessor('audio-processor', AudioProcessorWorklet);

// Main thread code
async function startAudioRecording() {
    const stream = await navigator.mediaDevices.getUserMedia({
        audio: {
            echoCancellation: true,
            noiseSuppression: true,
            autoGainControl: true,
            sampleRate: 48000
        }
    });

    const audioContext = new AudioContext();
    const audioSource = audioContext.createMediaStreamSource(stream);

    await audioContext.audioWorklet.addModule('/audio-worklet.js');
    const audioWorkletNode = new AudioWorkletNode(audioContext, 'audio-processor');

    audioWorkletNode.port.onmessage = (event) => {
        if (event.data.type === 'audioData' && ws?.readyState === WebSocket.OPEN) {
            ws.send(event.data.data); // Send binary data directly
        }
    };

    audioSource.connect(audioWorkletNode);
}
```

`AudioWorklet` berkinerja jauh lebih baik daripada `ScriptProcessorNode` dan menghindari masalah kegagapan audio yang sering ditimbulkan oleh jalur pemrosesan lama.

## Catatan dan praktik terbaik

### 1. Pemantauan koneksi

- Catat status sesi secara teratur untuk melacak siklus hidup koneksi penuh
- Pantau jumlah dan durasi segmen audio untuk mendeteksi koneksi abnormal
- Catat status koneksi dan perilaku koneksi ulang untuk layanan Doubao

Ini adalah praktik operasional dasar, namun memberikan perbedaan nyata dalam produksi.

### 2. Penanganan kesalahan

- Tangkap dan catat semua pengecualian WebSocket
- Gunakan `IAsyncDisposable` untuk memastikan sumber daya dibersihkan
- Terapkan penghentian koneksi dengan baik dan penanganan batas waktu

Singkatnya, dukung ketahanan.

### 3. Persyaratan format audio

- Sample rate: 16000 Hz (disarankan) atau 8000 Hz
- Kedalaman bit: 16-bit
- Saluran: mono
- Pengkodean: PCM (mentah)

Jika formatnya salah, pengenalan mungkin gagal atau kualitasnya menurun secara signifikan. Persyaratan ini penting.

### 4. Pertimbangan keamanan

- Simpan kredensial sensitif hanya dalam konfigurasi backend
- Terapkan batas koneksi untuk mencegah kehabisan sumber daya
- Gunakan HTTPS/WSS dalam produksi

Keamanan tidak pernah menjadi perhatian kecil.

### 5. Optimalisasi kinerja

- Gunakan operasi asinkron untuk menghindari pemblokiran
- Sesuaikan ukuran buffer sesuai kebutuhan (default: 4096 byte)
- Pertimbangkan pengumpulan koneksi dan strategi penggunaan kembali

Terapkan pengoptimalan ini jika sesuai dengan beban kerja Anda.

## Rekomendasi penerapan

1. **Penerapan Docker**: Kemas layanan proksi sebagai wadah untuk penskalaan dan pengelolaan yang lebih mudah
2. **Penyeimbangan beban**: Gunakan Nginx atau Envoy sebagai proksi terbalik untuk lalu lintas WebSocket
3. **Pemeriksaan kesehatan**: Menerapkan pemeriksaan berbasis detak jantung untuk memantau ketersediaan layanan
4. **Agregasi log**: Mengirim log ke sistem logging terpusat seperti ELK atau Loki

Penerapannya bisa sederhana atau rumit bergantung pada tim dan lingkungan, jadi sesuaikanlah.

## Ringkasan

Pola proksi WebSocket memecahkan masalah inti yaitu browser WebSocket API tidak mendukung header khusus. Dalam proyek HagiCode, pola ini terbukti layak dan stabil saat beralih dari validasi taman bermain ke penerapan produksi.

Kesimpulan utama:
- Proksi backend dapat meneruskan informasi autentikasi dengan aman
- WebSocket Asli ringan dan efisien untuk skenario sederhana
- Model "satu koneksi, satu sesi" menyederhanakan implementasi dan debugging
- Protokol frontend-backend harus memisahkan sinyal kontrol dari data audio

Jika Anda membuat fitur yang juga bergantung pada autentikasi WebSocket, saya harap pola ini memberi Anda titik awal yang berguna.
