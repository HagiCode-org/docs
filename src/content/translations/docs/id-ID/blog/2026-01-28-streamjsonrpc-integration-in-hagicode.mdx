## Integrasi Mendalam dan Penggunaan Praktis StreamJsonRpc di HagiCode

> Artikel ini menjelaskan secara rinci bagaimana proyek HagiCode (sebelumnya PCode) berhasil mengintegrasikan pustaka komunikasi StreamJsonRpc Microsoft untuk menggantikan implementasi JSON-RPC kustom aslinya, sekaligus menyelesaikan masalah teknis dan tantangan arsitektur yang dihadapi selama proses integrasi.

## Latar Belakang

StreamJsonRpc adalah pustaka komunikasi JSON-RPC yang dikelola secara resmi oleh Microsoft untuk .NET dan TypeScript, yang dikenal dengan keamanan tipenya yang kuat, pembuatan proksi otomatis, dan mekanisme penanganan pengecualian yang matang. Dalam proyek HagiCode, tim memutuskan untuk mengintegrasikan StreamJsonRpc untuk berkomunikasi dengan alat AI eksternal seperti iflow CLI dan OpenCode CLI melalui ACP (Agent Communication Protocol), sekaligus menghilangkan biaya pemeliharaan dan potensi bug yang disebabkan oleh implementasi JSON-RPC khusus sebelumnya. Namun, proses integrasi menghadapi tantangan khusus untuk streaming JSON-RPC, terutama saat menangani pengikatan target proxy dan pengenalan parameter umum.

Untuk mengatasi permasalahan ini, kami mengambil keputusan yang berani: membangun kembali seluruh sistem pembangunan dari awal. Dampak dari keputusan tersebut mungkin lebih besar dari yang Anda harapkan, dan saya akan segera menjelaskannya secara detail.

## Analisis

Proyek saat ini berada dalam tahap kritis integrasi protokol ACP dan menghadapi kendala teknis dan tantangan arsitektur berikut:

### 1. Batasan Pelaksanaan Adat
Implementasi JSON-RPC asli terletak di `src/HagiCode.ClaudeHelper/AcpImp/` dan termasuk komponen seperti `JsonRpcEndpoint` dan `ClientSideConnection`. Mempertahankan basis kode khusus ini membutuhkan biaya yang besar, dan tidak memiliki kemampuan tingkat lanjut dari perpustakaan yang matang, seperti pelaporan kemajuan dan dukungan pembatalan.

### 2. Hambatan Integrasi StreamJsonRpc
Saat mencoba memigrasikan yang sudah ada `CallbackProxyTarget` pola ke StreamJsonRpc, kami menemukan bahwa `_rpc.AddLocalRpcTarget(target)` metode tidak dapat mengenali target yang dibuat melalui pola proksi. Secara khusus, StreamJsonRpc tidak dapat secara otomatis membagi properti tipe generik `T` ke dalam parameter metode RPC, menyebabkan sisi server gagal saat memproses panggilan metode yang dimulai oleh klien.

### 3. Lapisan Arsitektur yang Bingung
Yang ada `ClientSideConnection` mencampurkan lapisan transport (WebSocket/Stdio), lapisan protokol (JSON-RPC), dan lapisan bisnis (antarmuka Agen ACP), sehingga menyebabkan tanggung jawab yang tidak jelas. Itu juga menderita karena tidak adanya pengikatan metode `AcpAgentCallbackRpcAdapter`.

### 4. Log Hilang
Lapisan transport WebSocket tidak memiliki pencatatan konten JSON mentah, sehingga sulit untuk menentukan selama proses debug RPC apakah masalah berasal dari serialisasi atau dari jaringan.

## Solusi

Untuk mengatasi masalah di atas, kami mengadopsi solusi sistematis berikut, mengoptimalkan dari tiga dimensi: refactoring arsitektur, integrasi perpustakaan, dan peningkatan debugging.

### 1. Bermigrasi Sepenuhnya ke StreamJsonRpc

#### Hapus Kode Lama
Hapus `JsonRpcEndpoint.cs`, `AgentSideConnection.cs`, dan konverter serialisasi khusus terkait seperti `JsonRpcMessageJsonConverter`.

#### Integrasikan Perpustakaan Resmi
Perkenalkan `StreamJsonRpc` Paket NuGet dan gunakan `JsonRpc` kelas untuk menangani logika komunikasi inti.

#### Abstrak Lapisan Transportasi
Tentukan `IAcpTransport` antarmuka untuk menangani keduanya `WebSocket` dan `Stdio` mode transport dengan cara yang terpadu, memastikan lapisan protokol dipisahkan dari lapisan transport.

```csharp
// Definition of the IAcpTransport interface
public interface IAcpTransport
{
    Task SendAsync(string message, CancellationToken cancellationToken = default);
    Task<string> ReceiveAsync(CancellationToken cancellationToken = default);
    Task CloseAsync(CancellationToken cancellationToken = default);
}

// WebSocket transport implementation
public class WebSocketTransport : IAcpTransport
{
    private readonly WebSocket _webSocket;

    public WebSocketTransport(WebSocket webSocket)
    {
        _webSocket = webSocket;
    }

    // Implement send and receive methods
    // ...
}

// Stdio transport implementation
public class StdioTransport : IAcpTransport
{
    private readonly StreamReader _reader;
    private readonly StreamWriter _writer;

    public StdioTransport(StreamReader reader, StreamWriter writer)
    {
        _reader = reader;
        _writer = writer;
    }

    // Implement send and receive methods
    // ...
}
```

### 2. Perbaiki Masalah Pengenalan Target Proxy

#### Analisis `CallbackProxyTarget`
Periksa logika pembuatan proksi dinamis yang ada dan identifikasi penyebab utama mengapa StreamJsonRpc tidak dapat mengenalinya. Dalam kebanyakan kasus, hal ini terjadi karena objek proksi tidak mengekspos tanda tangan metode sebenarnya secara publik, atau menggunakan tipe parameter yang tidak didukung oleh StreamJsonRpc.

#### Pengoperan Parameter Refaktor
Pisahkan properti generik menjadi parameter metode RPC eksplisit. Daripada mengandalkan properti dinamis, tentukan DTO Permintaan/Respons (objek transfer data) yang konkret sehingga StreamJsonRpc dapat mengenali tanda tangan metode dengan benar melalui refleksi.

```csharp
// Original generic property approach
public class CallbackProxyTarget<T>
{
    public Func<T, Task> Callback { get; set; }
}

// Refactored concrete method approach
public class ReadTextFileRequest
{
    public string FilePath { get; set; }
}

public class ReadTextFileResponse
{
    public string Content { get; set; }
}

public interface IAcpAgentCallback
{
    Task<ReadTextFileResponse> ReadTextFileAsync(ReadTextFileRequest request);
    // Other methods...
}
```

#### Gunakan `Attach` Dari pada `AddLocalRpcTarget`
Dalam beberapa skenario kompleks, melakukan proksi secara manual `JsonRpc` objek dan penanganannya `RpcConnection` bisa lebih fleksibel daripada menambahkan target lokal secara langsung.

### 3. Menerapkan Metode Binding dan Stronger Logging

#### Menerapkan `AcpAgentCallbackRpcAdapter`
Pastikan komponen ini secara eksplisit mengimplementasikan antarmuka proksi StreamJsonRpc dan memetakan metode yang ditentukan oleh protokol ACP, seperti `ReadTextFileAsync`, ke penangan panggilan balik StreamJsonRpc.

#### Integrasikan Pencatatan
Mencegat dan mencatat teks mentah permintaan dan respons JSON-RPC di jalur pemrosesan pesan WebSocket atau Stdio. Gunakan `ILogger` untuk mengeluarkan payload mentah sebelum penguraian dan setelah serialisasi sehingga masalah pemformatan dapat didiagnosis dengan lebih mudah.

```csharp
// Transport wrapper with enhanced logging
public class LoggingAcpTransport : IAcpTransport
{
    private readonly IAcpTransport _innerTransport;
    private readonly ILogger<LoggingAcpTransport> _logger;

    public LoggingAcpTransport(IAcpTransport innerTransport, ILogger<LoggingAcpTransport> logger)
    {
        _innerTransport = innerTransport;
        _logger = logger;
    }

    public async Task SendAsync(string message, CancellationToken cancellationToken = default)
    {
        _logger.LogTrace("Sending message: {Message}", message);
        await _innerTransport.SendAsync(message, cancellationToken);
    }

    public async Task<string> ReceiveAsync(CancellationToken cancellationToken = default)
    {
        var message = await _innerTransport.ReceiveAsync(cancellationToken);
        _logger.LogTrace("Received message: {Message}", message);
        return message;
    }

    public async Task CloseAsync(CancellationToken cancellationToken = default)
    {
        _logger.LogDebug("Closing connection");
        await _innerTransport.CloseAsync(cancellationToken);
    }
}
```

### 4. Refaktor Pelapisan Arsitektur

#### Lapisan Transportasi (`AcpRpcClient`)
Bungkus koneksi StreamJsonRpc dan jadikan itu bertanggung jawab `InvokeAsync` panggilan dan manajemen siklus hidup koneksi.

```csharp
public class AcpRpcClient : IDisposable
{
    private readonly JsonRpc _rpc;
    private readonly IAcpTransport _transport;

    public AcpRpcClient(IAcpTransport transport)
    {
        _transport = transport;
        _rpc = new JsonRpc(new StreamRpcTransport(transport));
        _rpc.StartListening();
    }

    public async Task<TResponse> InvokeAsync<TResponse>(string methodName, object parameters)
    {
        return await _rpc.InvokeAsync<TResponse>(methodName, parameters);
    }

    public void Dispose()
    {
        _rpc.Dispose();
        _transport.Dispose();
    }

    // StreamRpcTransport is the StreamJsonRpc adapter for IAcpTransport
    private class StreamRpcTransport : IDuplexPipe
    {
        // Implement the IDuplexPipe interface
        // ...
    }
}
```

#### Lapisan Protokol (`IAcpAgentClient` / `IAcpAgentCallback`)
Tentukan antarmuka klien-ke-agen dan agen-ke-klien yang jelas. Hapus pola pabrik siklik `Func<IAcpAgent, IAcpClient>` dan menggantinya dengan injeksi ketergantungan atau registrasi panggilan balik langsung.

## Praktek

Berdasarkan praktik terbaik dan pengalaman proyek StreamJsonRpc, berikut adalah rekomendasi utama untuk implementasi:

### 1. DTO yang Diketik dengan Kuat Lebih Baik Daripada Objek Dinamis
Keunggulan inti StreamJsonRpc terletak pada pengetikan yang kuat. Jangan gunakan `dynamic` atau `JObject` untuk melewati parameter. Sebagai gantinya, tentukan kelas C# POCO eksplisit sebagai parameter untuk setiap metode RPC. Ini tidak hanya memecahkan masalah pengenalan target proxy, tetapi juga menangkap kesalahan tipe pada waktu kompilasi.

Contoh: ganti properti generik di `CallbackProxyTarget` dengan kelas konkrit seperti `ReadTextFileRequest` dan `WriteTextFileRequest`.

### 2. Deklarasikan Nama Metode Secara Eksplisit
Gunakan `[JsonRpcMethod]` atribut untuk secara eksplisit menentukan nama metode RPC daripada mengandalkan pemetaan nama metode default. Hal ini mencegah kegagalan pemanggilan yang disebabkan oleh perbedaan gaya penamaan seperti PascalCase versus camelCase.

```csharp
public interface IAcpAgentCallback
{
    [JsonRpcMethod("readTextFile")]
    Task<ReadTextFileResponse> ReadTextFileAsync(ReadTextFileRequest request);
    
    [JsonRpcMethod("writeTextFile")]
    Task WriteTextFileAsync(WriteTextFileRequest request);
}
```

### 3. Manfaatkan Panggilan Balik Status Koneksi
StreamJsonRpc menyediakan `JsonRpc.ConnectionLost` acara. Anda harus benar-benar mendengarkan kejadian ini untuk menangani proses keluar yang tidak terduga atau pemutusan jaringan, yang lebih tepat waktu daripada hanya mengandalkan deteksi kegagalan Orleans Grain.

```csharp
_rpc.ConnectionLost += (sender, e) =>
{
    _logger.LogError("RPC connection lost: {Reason}", e.ToString());
    // Handle reconnection logic or notify the user
};
```

### 4. Rekam Log berdasarkan Lapisan
- **Tingkat pelacakan**: Catat payload permintaan/respons JSON mentah secara lengkap.
- **Tingkat debug**: Merekam tumpukan panggilan metode dan ringkasan parameter.
- **Catatan**: Pastikan log tidak menyertakan token Otorisasi sensitif atau konten file besar yang dikodekan Base64.

### 5. Menangani Sifat Khusus Transportasi Streaming
StreamJsonRpc mendukung secara asli `IAsyncEnumerable`. Saat menerapkan respons cepat streaming untuk ACP, gunakan `IAsyncEnumerable` secara langsung, bukan logika penomoran halaman khusus. Hal ini dapat sangat menyederhanakan jumlah kode yang diperlukan untuk pemrosesan streaming.

```csharp
public interface IAcpAgentCallback
{
    [JsonRpcMethod("streamText")]
    IAsyncEnumerable<string> StreamTextAsync(StreamTextRequest request);
}
```

### 6. Pola Adaptor
Simpan `ACPSession` dan `ClientSideConnection` terpisah. `ACPSession` harus fokus pada manajemen negara bagian Orleans dan logika bisnis, seperti antrean pesan, dan harus menggunakan objek koneksi StreamJsonRpc melalui komposisi, bukan pewarisan.

## Kesimpulan

Dengan mengintegrasikan StreamJsonRpc secara komprehensif, proyek HagiCode berhasil mengatasi tingginya biaya pemeliharaan, keterbatasan fungsional, dan kebingungan lapisan arsitektur dari implementasi kustom asli. Peningkatan utama meliputi:

1. Mengganti properti dinamis dengan DTO yang diketik dengan kuat, meningkatkan kemudahan pemeliharaan dan keandalan
2. Menerapkan abstraksi lapisan transport dan pemisahan lapisan protokol, meningkatkan kejelasan arsitektur
3. Memperkuat kemampuan logging untuk membuat masalah komunikasi lebih mudah didiagnosis
4. Memperkenalkan dukungan streaming untuk menyederhanakan implementasi streaming

Peningkatan ini memberi HagiCode landasan komunikasi yang lebih stabil dan efisien, memungkinkannya berinteraksi lebih baik dengan alat AI eksternal dan meletakkan landasan yang kuat untuk perluasan fitur di masa depan.
