## Penjelasan Arsitektur Distribusi Hibrid Desktop HagiCode: Bagaimana P2P Mempercepat Pengunduhan File Besar

> Saya menahan artikel ini cukup lama sebelum akhirnya menulisnya, dan saya masih tidak yakin apakah artikel ini dapat dibaca dengan baik. Penulisan teknis cukup mudah untuk dibuat, tetapi sulit untuk dibuat benar-benar menarik. Lagi pula, saya bukan ahli sastra yang hebat, jadi sebaiknya saya berikan penjelasan sederhana ini.

## Latar Belakang

Cepat atau lambat, tim yang membangun aplikasi desktop akan mengalami sakit kepala yang sama: bagaimana Anda mendistribusikan file besar?

Ini adalah masalah yang aneh. Pengunduhan langsung HTTP/HTTPS tradisional masih dapat bertahan ketika file berukuran kecil dan jumlah pengguna terbatas. Tapi waktu jarang sekali baik. Ketika sebuah proyek terus berkembang, paket instalasi pun ikut berkembang: paket ZIP Desktop, paket portabel, arsip penerapan web, dan banyak lagi. Kemudian permasalahan mulai muncul ke permukaan:

- **Kecepatan pengunduhan dibatasi oleh bandwidth asal**: tidak peduli berapa banyak bandwidth yang dimiliki satu server, server tetap kesulitan saat semua orang mengunduh sekaligus.
- **Dukungan melanjutkan hampir tidak ada**: jika pengunduhan HTTP terganggu, Anda sering kali harus memulai dari awal. Itu membuang-buang waktu dan bandwidth.
- **Server asal menanggung semua tekanan**: semua lalu lintas mengalir kembali ke server pusat, biaya bandwidth terus meningkat, dan skalabilitas menjadi masalah nyata.

Proyek HagiCode Desktop tidak terkecuali. Saat kami merancang sistem distribusi, kami terus bertanya pada diri sendiri: bisakah kami memperkenalkan pendekatan distribusi hybrid tanpa mengubah sistem yang sudah ada `index.json` pesawat kendali? Dengan kata lain, dapatkah kita menggunakan sifat jaringan P2P yang terdistribusi untuk mempercepat pengunduhan sambil tetap mempertahankan fallback asal HTTP sehingga sistem tetap dapat digunakan di lingkungan terbatas seperti jaringan perusahaan?

Dampak dari keputusan tersebut ternyata lebih besar dari yang Anda duga. Mari kita bahas langkah demi langkah.

## Filosofi Desain Inti: P2P Pertama, HTTP Fallback

Intinya, model distribusi hibrid dapat diringkas dalam satu kalimat: **P2P dulu, HTTP fallback**.

Kuncinya terletak pada kata 'hibrida'. Ini bukan tentang sekadar menambahkan BitTorrent dan menghentikannya. Intinya adalah membuat kedua metode penyampaian tersebut bekerja sama dan saling melengkapi:

- **Jaringan P2P** memberikan akselerasi terdistribusi. Semakin banyak orang mengunduh, semakin banyak rekan yang bergabung, dan semakin cepat transfernya.
- **Penggantian WebSeed/HTTP** menjamin ketersediaan, sehingga unduhan tetap dapat berfungsi di firewall perusahaan dan lingkungan jaringan internal.
- **Bidang kendalinya tetap sederhana**. Kami tidak mengubah logika inti `index.json`; kami hanya menambahkan beberapa bidang metadata opsional.

Manfaat sebenarnya sangat jelas: pengguna merasa bahwa "pengunduhan lebih cepat", sementara tim teknis tidak perlu memikul terlalu banyak kerumitan tambahan. Bagaimanapun, protokol BT sudah matang, dan hanya ada sedikit alasan untuk menemukan kembali rodanya.

## Desain Arsitektur

│  └── SHA256 Verifier                │

└─────────────────────────────────────┘

Seperti yang ditunjukkan diagram, sistem menggunakan desain berlapis. Alasan untuk memisahkan tanggung jawab ini jelas sederhana: **dapat diuji dan dapat diganti**.

- **Lapisan UI** bertanggung jawab untuk menampilkan kemajuan pengunduhan dan tombol akselerasi berbagi. Itu adalah permukaannya.
- **Lapisan koordinasi** adalah intinya. Ini berisi evaluasi kebijakan, adaptasi mesin, manajemen cache, dan verifikasi integritas.

Lapisan mesin diabstraksi di belakang `DownloadEngineAdapter` antarmuka. Jika nanti kita ingin menukar mesin BT lain, atau memindahkan implementasinya ke proses sespan, itu menjadi lebih mudah.

### Pemisahan Control Plane dan Data Plane

HagiCode Desktop tetap `index.json` sebagai satu-satunya bidang kendali, dan desain itu sangat penting. Bidang kontrol bertanggung jawab atas penemuan versi, pemilihan saluran, dan kebijakan terpusat, sedangkan bidang data adalah tempat terjadinya transfer file sebenarnya.

Bidang baru ditambahkan ke `index.json` bersifat opsional:

```json
{
  "asset": {
    "torrentUrl": "https://cdn.example.com/app.torrent",
    "infoHash": "abc123...",
    "webSeeds": [
      "https://cdn.example.com/app.zip",
      "https://backup.example.com/app.zip"
    ],
    "sha256": "def456...",
    "directUrl": "https://cdn.example.com/app.zip"
  }
}
```

Semua bidang ini bersifat opsional. Jika tidak ada, klien kembali ke mode pengunduhan HTTP tradisional. Keuntungan desain ini adalah **kompatibilitas mundur**: klien lama sama sekali tidak terpengaruh.

### Keputusan Berdasarkan Kebijakan

Tidak semua file layak didistribusikan melalui P2P.

`DistributionPolicyEvaluator` bertanggung jawab untuk mengevaluasi kebijakan tersebut. Hanya file yang memenuhi semua ketentuan berikut yang akan menggunakan pengunduhan hibrid:

1. **Jenis sumber harus berupa indeks HTTP**: unduhan GitHub langsung atau sumber folder lokal tidak menggunakan jalur ini.
2. **Ukuran file minimal harus 100 MB**: untuk file yang lebih kecil, biaya P2P lebih besar daripada manfaatnya.
3. **Metadata hibrid lengkap harus ada**: `torrentUrl`, `webSeeds`, dan `sha256` semuanya diperlukan.
4. **Hanya paket desktop dan paket penerapan web terbaru yang memenuhi syarat**: versi historis tetap menggunakan jalur distribusi tradisional.

```typescript
class DistributionPolicyEvaluator {
  evaluate(version: Version, settings: SharingAccelerationSettings): HybridDownloadPolicy {
    // Check source type
    if (version.sourceType !== 'http-index') {
      return { useHybrid: false, reason: 'not-http-index' };
    }

    // Check metadata completeness
    if (!version.hybrid) {
      return { useHybrid: false, reason: 'not-eligible' };
    }

    // Check whether the feature is enabled
    if (!settings.enabled) {
      return { useHybrid: false, reason: 'shared-disabled' };
    }

    // Check asset type (latest desktop/web packages only)
    if (!version.hybrid.isLatestDesktopAsset && !version.hybrid.isLatestWebAsset) {
      return { useHybrid: false, reason: 'latest-only' };
    }

    return { useHybrid: true, reason: 'shared-enabled' };
  }
}
```

Hal ini memberikan sistem **perilaku yang dapat diprediksi**. Baik pengembang maupun pengguna dapat memahami dengan jelas file mana yang akan menggunakan P2P dan mana yang tidak.

## Implementasi Inti

Mari kita mulai dengan definisi tipe, karena definisi tipe membentuk fondasi keseluruhan sistem.

Setelah sistem tipenya jelas, implementasi selanjutnya akan mengikuti secara alami.

### Koordinator Inti

`HybridDownloadCoordinator` mengatur seluruh alur kerja pengunduhan. Ini mengoordinasikan evaluasi kebijakan, eksekusi mesin, verifikasi SHA256, dan manajemen cache.

```typescript
class HybridDownloadCoordinator {
  async download(
    version: Version,
    cachePath: string,
    packageSource: PackageSource,
    onProgress?: DownloadProgressCallback,
  ): Promise<HybridDownloadResult> {
    // 1. Evaluate the policy: should hybrid download be used?
    const policy = this.policyEvaluator.evaluate(version, settings);

    // 2. Execute the download
    if (policy.useHybrid) {
      await this.engine.download(version, cachePath, settings, onProgress);
    } else {
      await packageSource.downloadPackage(version, cachePath, onProgress);
    }

    // 3. SHA256 verification (hard gate)
    const verified = await this.verify(version, cachePath, onProgress);
    if (!verified) {
      await this.cacheRetentionManager.discard(version.id, cachePath);
      throw new Error(`sha256 verification failed for ${version.id}`);
    }

    // 4. Mark as trusted cache and begin controlled seeding
    await this.cacheRetentionManager.markTrusted({
      versionId: version.id,
      cachePath,
      cacheSize,
    }, settings);

    return { cachePath, policy, verified };
  }
}
```

Ada satu poin yang sangat penting di sini: **Verifikasi SHA256 adalah sebuah gerbang yang sulit**. File yang diunduh harus lolos verifikasi sebelum dapat masuk ke alur instalasi. Jika verifikasi gagal, cache akan dibuang untuk memastikan bahwa file yang salah tidak akan menyebabkan masalah instalasi.

### Unduh Abstraksi Mesin

`DownloadEngineAdapter` adalah antarmuka abstrak yang mendefinisikan metode yang harus diterapkan setiap mesin:

Implementasi V1 didasarkan pada WebTorrent dan disertakan `InProcessTorrentEngineAdapter`:

```typescript
class InProcessTorrentEngineAdapter implements DownloadEngineAdapter {
  async download(...) {
    const client = this.getClient(settings);  // Apply upload rate limiting
    const torrent = client.add(torrentId, {
      path: path.dirname(destinationPath),
      destroyStoreOnDestroy: false,
      maxWebConns: 8,
    });

    // Add WebSeed sources
    torrent.on('ready', () => {
      for (const seed of hybrid.webSeeds) {
        torrent.addWebSeed(seed);
      }
      if (hybrid.directUrl) {
        torrent.addWebSeed(hybrid.directUrl);
      }
    });

    // Progress reporting - distinguish P2P from origin fallback
    torrent.on('download', () => {
      const hasP2PPeer = torrent.wires.some(w => w.type !== 'webSeed');
      const mode = hasP2PPeer ? 'shared-acceleration' : 'source-fallback';
      // ... report progress
    });
  }
}
```

Desain mesin yang dapat dicolokkan membuat pengoptimalan di masa depan menjadi lebih mudah. Misalnya, V2 dapat menjalankan mesin dalam proses pembantu untuk menghindari terhentinya proses utama jika mesin mengalami kerusakan.

### Membedakan Mode Pelaporan Kemajuan

Pada lapisan UI, hal yang paling dipedulikan pengguna adalah sederhana: "apakah saat ini saya mengunduh melalui P2P atau melalui HTTP fallback?" `InProcessTorrentEngineAdapter` menentukan itu dengan memeriksa tipe di dalamnya `torrent.wires`:

```typescript
const hasP2PPeer = torrent.wires.some((wire) => wire.type !== 'webSeed');
const hasFallbackWire = torrent.wires.some((wire) => wire.type === 'webSeed');

const mode = hasP2PPeer ? 'shared-acceleration'
         : hasFallbackWire ? 'source-fallback'
         : 'shared-acceleration';

const stage = hasP2PPeer ? 'downloading'
           : hasFallbackWire ? 'backfilling'
           : 'downloading';
```

Logikanya terlihat sederhana, namun merupakan bagian penting dari pengalaman pengguna. Pengguna dapat melihat dengan jelas apakah status saat ini adalah "akselerasi berbagi" atau "pengisian ulang asal", yang membuat perilaku tersebut lebih mudah dipahami.

### Verifikasi Streaming SHA256

Verifikasi integritas menggunakan Node.js `crypto` modul untuk menghitung hash secara streaming, yang menghindari memuat seluruh file ke dalam memori:

```typescript
private async computeSha256(filePath: string): Promise<string> {
  const hash = createHash('sha256');
  await new Promise<void>((resolve, reject) => {
    const stream = fs.createReadStream(filePath);
    stream.on('data', (chunk) => hash.update(chunk));
    stream.on('error', reject);
    stream.on('end', resolve);
  });
  return hash.digest('hex').toLowerCase();
}
```

Implementasi ini sangat ramah untuk file besar. Bayangkan mengunduh paket instalasi 2 GB dan kemudian mencoba memuat semuanya ke dalam memori hanya untuk memverifikasinya. Streaming menyelesaikannya dengan bersih.

## Keputusan Desain Utama

Melihat kembali solusi keseluruhan, beberapa keputusan desain layak untuk dipertimbangkan.

### Mesin Berjalan di Proses Utama (V1)

Mengapa tidak segera memulai dengan proses sespan atau pembantu? Alasannya sederhana: **kirim dengan cepat**. Desain dalam proses memiliki siklus pengembangan yang lebih pendek dan lebih mudah untuk di-debug. Prioritas pertama adalah menjalankan fitur tersebut, kemudian meningkatkan stabilitas setelahnya.

Tentu saja, keputusan ini memerlukan biaya: jika mesin mengalami kerusakan, hal ini dapat memengaruhi proses utama. Kami mengurangi risiko tersebut melalui batasan adaptor dan kontrol batas waktu, dan kami juga menjaga jalur migrasi tetap terbuka sehingga V2 dapat berpindah ke proses terpisah dengan lebih mudah.

### SHA256 sebagai Pemeriksaan Integritas

Kami menggunakan SHA256 daripada MD5 atau CRC32 karena SHA256 lebih aman. Biaya tabrakan untuk MD5 dan CRC32 terlalu rendah. Jika seseorang dengan jahat membuat paket instalasi palsu, konsekuensinya bisa sangat parah. SHA256 membutuhkan biaya komputasi yang lebih besar, namun keuntungan keamanannya sepadan.

### Hanya Diaktifkan untuk Sumber Indeks HTTP

Skenario seperti unduhan GitHub dan sumber folder lokal tidak menggunakan distribusi hibrid. Ini bukanlah batasan teknis; ini tentang **menghindari kerumitan yang tidak perlu**. Protokol BT menambahkan nilai terbatas dalam skenario jaringan pribadi dan hanya akan meningkatkan kompleksitas kode.

## Catatan Praktis

### Pengaturan Normalisasi

Di dalam `SharingAccelerationSettingsStore`, setiap nilai numerik harus melalui pemeriksaan batas dan normalisasi:

```typescript
private normalize(settings: SharingAccelerationSettings): SharingAccelerationSettings {
  return {
    enabled: Boolean(settings.enabled),
    uploadLimitMbps: this.clampNumber(settings.uploadLimitMbps, 1, 200, DEFAULT_SETTINGS.uploadLimitMbps),
    cacheLimitGb: this.clampNumber(settings.cacheLimitGb, 1, 500, DEFAULT_SETTINGS.cacheLimitGb),
    retentionDays: this.clampNumber(settings.retentionDays, 1, 90, DEFAULT_SETTINGS.retentionDays),
    hybridThresholdMb: DEFAULT_SETTINGS.hybridThresholdMb,  // Fixed value, not user-configurable
    onboardingChoiceRecorded: Boolean(settings.onboardingChoiceRecorded),
  };
}

private clampNumber(value: number, min: number, max: number, fallback: number): number {
  if (!Number.isFinite(value)) {
    return fallback;
  }
  return Math.min(max, Math.max(min, Math.round(value)));
}
```

Hal ini mencegah pengguna mengedit file konfigurasi secara manual menjadi nilai yang tidak valid.

### Pembersihan Cache LRU

`CacheRetentionManager.prune()` bertanggung jawab untuk membersihkan entri cache yang kedaluwarsa atau terlalu besar. Strategi pembersihan menggunakan LRU (paling jarang digunakan):

```typescript
const records = [...this.listRecords()]
  .sort((left, right) =>
    new Date(left.lastUsedAt).getTime() - new Date(right.lastUsedAt).getTime()
  );

// When over the limit, evict the least recently used entries first
while (totalBytes > maxBytes && retainedEntries.length > 0) {
  const evicted = records.find((record) => retainedEntries.includes(record.versionId));
  retainedEntries.splice(retainedEntries.indexOf(evicted.versionId), 1);
  removedEntries.push(evicted.versionId);
  totalBytes -= evicted.cacheSize;
  await fs.rm(evicted.cachePath, { force: true });
}
```

Logika ini memastikan ruang disk digunakan secara efisien sambil mempertahankan versi historis yang mungkin masih dibutuhkan pengguna.

### Perilaku Menghentikan Penyemaian Segera

Saat pengguna mematikan akselerasi berbagi, aplikasi harus segera menghentikan penyemaian dan menghancurkan klien torrent:

```typescript
async disableSharingAcceleration(): Promise<void> {
  this.settingsStore.updateSettings({ enabled: false });
  await this.cacheRetentionManager.stopAllSeeding();  // Stop seeding
  await this.engine.stopAll();  // Destroy the torrent client
}
```

Jika pengguna menonaktifkan fitur tersebut, produk tidak akan lagi menggunakan sumber daya P2P apa pun. Itu adalah etika produk dasar.

## Risiko dan Trade-Off

Tidak ada solusi yang sempurna, tidak terkecuali distribusi hybrid. Inilah trade-off utamanya:

**Isolasi kerusakan lebih lemah dibandingkan sespan**: V1 menggunakan mesin yang sedang dalam proses, sehingga kerusakan mesin dapat memengaruhi proses utama. Batasan adaptor dan kontrol batas waktu mengurangi risiko, namun hal tersebut bukanlah solusi yang mendasar. V2 menyertakan jalur migrasi terencana ke proses pembantu.

**Penggunaan sumber daya yang diaktifkan secara default**: pengaturan default unggahan 2 MB/dtk, cache 10 GB, dan retensi 7 hari memang menghabiskan sebagian sumber daya mesin. Harapan pengguna dikelola melalui salinan orientasi dan pengaturan transparan.

**Kompatibilitas jaringan perusahaan**: penggantian WebSeed/HTTPS otomatis mempertahankan kegunaan di jaringan perusahaan, namun dapat mengurangi peningkatan akselerasi dari P2P. Ini adalah trade-off yang disengaja dan memprioritaskan ketersediaan.

**Metadata yang kompatibel dengan versi sebelumnya**: semua kolom baru bersifat opsional. Jika tidak ada, sistem akan kembali ke mode HTTP. Klien lama sama sekali tidak terpengaruh, membuat peningkatan menjadi lancar.

## Kesimpulan

Artikel ini membahas arsitektur distribusi hybrid yang digunakan dalam proyek HagiCode Desktop. Kesimpulan utamanya adalah:

1. **Arsitektur berlapis**: bidang kontrol dan bidang data dipisahkan, dan mesin diabstraksi di balik antarmuka yang dapat dicolokkan untuk pengujian dan perluasan yang lebih mudah.

2. **Perilaku berdasarkan kebijakan**: tidak semua file menggunakan P2P. Distribusi hibrid hanya diaktifkan untuk file besar yang memenuhi ketentuan yang diperlukan.

3. **Verifikasi integritas**: SHA256 berfungsi sebagai gerbang keras, dan verifikasi streaming menghindari tekanan memori.

4. **Presentasi yang diproduksi**: Terminologi BT tersembunyi di balik frasa "akselerasi berbagi", dan fitur ini diaktifkan secara default selama orientasi.

5. **Kontrol pengguna**: batas unggahan, batas cache, hari penyimpanan, dan parameter lainnya tetap dapat disesuaikan oleh pengguna.

Arsitektur ini telah diimplementasikan dalam proyek HagiCode Desktop. Jika Anda mencobanya, kami akan senang mendengar tanggapan Anda setelah instalasi dan penggunaan di dunia nyata.
