## VSCode dan server kode: Memilih Solusi Pengeditan Kode Berbasis Browser

> Saat membangun kemampuan pengeditan kode berbasis browser, pengembang menghadapi pilihan utama: menggunakan versi resmi VSCode `code serve-web` fitur, atau mengadopsi berbasis komunitas `code-server` solusi? Keputusan ini tidak hanya memengaruhi arsitektur teknis, namun juga kepatuhan lisensi dan fleksibilitas penerapan.

## Latar Belakang

Seleksi teknis sangat mirip dengan memilih jalan hidup. Setelah Anda memilih satu, biasanya Anda harus terus berjalan, dan berpindah di kemudian hari bisa menjadi sangat mahal.

Di era pemrograman berbantuan AI, pengeditan kode berbasis browser menjadi semakin penting. Pengguna berharap setelah asisten AI selesai menganalisis kode, mereka dapat segera membuka editor di sesi browser yang sama dan melakukan perubahan tanpa berpindah aplikasi. Pengalaman mulus seperti itu seharusnya ada saat Anda membutuhkannya.

Namun, saat mengimplementasikan fitur ini, pengembang menghadapi pilihan teknis yang penting: apakah mereka harus menggunakan versi resmi VSCode `code serve-web` fitur, atau berbasis komunitas `code-server` solusi?

Setiap pilihan mempunyai kelebihan dan kekurangannya masing-masing, dan memilih yang buruk dapat menimbulkan banyak masalah di kemudian hari. Pemberian lisensi adalah salah satu contohnya: jika setelah peluncuran Anda baru mengetahui bahwa produk Anda tidak mematuhi lisensi, maka hal tersebut sudah terlambat. Penerapan adalah hal lain: sebuah solusi mungkin berfungsi sempurna dalam pengembangan, kemudian mengalami berbagai macam masalah setelah dipindahkan ke dalam container. Ini adalah jenis jebakan yang ingin dihindari oleh tim.

## Tentang HagiCode

Saat menerapkan pengeditan kode berbasis browser, kami mempelajari kedua solusi secara mendalam dan pada akhirnya merancang arsitektur kami untuk mendukung keduanya, sambil memilih server kode sebagai default.

Repositori proyek: [github.com/HagiCode-org/site](https://github.com/HagiCode-org/site)

## Perbedaan Lisensi (Paling Penting)

Inilah perbedaan paling mendasar antara kedua solusi tersebut, dan faktor pertama yang kami pertimbangkan selama evaluasi. Saat membuat pilihan teknis, penting untuk memahami risiko hukum di awal.

**server kode**
- Lisensi MIT, sepenuhnya open source
- Dikelola oleh Coder.com dengan komunitas aktif
- Bebas untuk digunakan secara komersial, dimodifikasi, dan didistribusikan
- Tidak ada batasan pada skenario penggunaan

**Kode VSC `code serve-web`**
- Bagian dari produk Microsoft VSCode
- Menggunakan lisensi Microsoft (lisensi VS Code mencakup pembatasan penggunaan komersial)
- Terutama ditujukan untuk pengembang individu
- Penerapan di perusahaan mungkin memerlukan peninjauan lisensi komersial tambahan

Dari perspektif perizinan, server kode lebih ramah terhadap proyek komersial. Ini adalah sesuatu yang perlu Anda pikirkan selama perencanaan produk, karena migrasi nanti bisa menjadi sangat mahal.

## Perbedaan Penerapan

Setelah perizinan diselesaikan, masalah berikutnya adalah penerapan. Hal ini secara langsung memengaruhi biaya operasional dan desain arsitektur Anda.

**server kode**
- Aplikasi Node.js mandiri yang dapat diterapkan secara mandiri
- Mendukung berbagai sumber runtime:
  - Secara langsung menentukan jalur yang dapat dieksekusi
  - Mencarinya melalui sistem PATH
  - Deteksi otomatis lingkungan NVM Node.js 22.x
- Tidak perlu menginstal aplikasi desktop VSCode di server
- Lebih mudah untuk diterapkan dalam container

**Kode VSC `code serve-web`**
- Harus bergantung pada VSCode CLI yang diinstal secara lokal
- Membutuhkan tersedia `code` perintah pada mesin host
- Sistem menyaring wrapper VS Code Remote CLI
- Terutama dirancang untuk skenario pembangunan lokal

code-server lebih cocok untuk skenario penerapan server dan container. Jika produk Anda perlu dijalankan di Docker, atau pengguna Anda belum menginstal VSCode, server kode biasanya merupakan pilihan yang tepat.

## Perbedaan Parameter Fitur

Kedua solusi tersebut juga berbeda dalam beberapa parameter fitur. Perbedaannya tidak terlalu besar, namun dapat menimbulkan gesekan integrasi dalam penggunaan di dunia nyata.

| Fitur | server kode | kode melayani-web |
|------|-------------|----------------|
| Jalur pangkalan umum | `/` (dapat dikonfigurasi) | `/vscode-server` (tetap) |
| Otentikasi | `--auth` parameter dengan beberapa mode | `--connection-token` / `--without-connection-token` |
| Direktori data | `{DataDir}/code-server` | `{DataDir}/vscode-serve-web` |
| Telemetri | Dinonaktifkan secara default dengan `--disable-telemetry` | Tergantung pada pengaturan VSCode |
| Perbarui pemeriksaan | Dapat dinonaktifkan dengan `--disable-update-check` | Tergantung pada pengaturan VSCode |

Perbedaan-perbedaan ini memerlukan perhatian khusus pada saat integrasi. Misalnya, jalur URL yang berbeda berarti kode frontend Anda memerlukan penanganan khusus.

## Perbedaan Deteksi Ketersediaan

Saat menerapkan peralihan editor, logika deteksi ketersediaan juga berbeda.

**server kode**
- Selalu dikembalikan sebagai implementasi yang terlihat
- Masih ditampilkan meskipun tidak tersedia, dengan `install-required` status
- Mendukung deteksi otomatis lingkungan NVM Node.js 22.x

**layanan kode-web**
- Hanya terlihat ketika ada warga lokal `code` CLI terdeteksi
- Jika tidak tersedia, frontend secara otomatis menyembunyikan opsi ini
- Tergantung pada status instalasi VSCode lokal

Perbedaan ini secara langsung mempengaruhi pengalaman pengguna. server kode lebih transparan: pengguna dapat melihat opsi dan memahami bahwa instalasi masih diperlukan. kode serve-web lebih tersembunyi: pengguna mungkin tidak menyadari bahwa opsi tersebut ada. Pendekatan mana yang lebih baik bergantung pada positioning produk.

## Arsitektur Implementasi Ganda HagiCode

Setelah analisis mendalam, proyek HagiCode mengadopsi arsitektur implementasi ganda yang mendukung kedua solusi pada tingkat arsitektur.

### Defaultnya adalah server kode

```csharp
// The default active implementation is code-server
// If an explicit activeImplementation is saved, try that implementation first
// If the requested implementation is unavailable, the resolver tries the other one
// If a fallback occurs, return fallbackReason
```

Kami menggunakan server kode secara default terutama karena fleksibilitas lisensi dan penerapan. Namun, bagi pengguna yang sudah memiliki lingkungan VSCode lokal, code serve-web juga merupakan pilihan yang tepat.

### Pemilih Implementasi

`CodeServerImplementationResolver` bertanggung jawab untuk menyatukan:
- Pemilihan implementasi selama pemanasan startup
- Pilihan implementasi saat membaca status
- Pemilihan implementasi saat membuka proyek
- Pilihan implementasi saat membuka Vault

Desain ini memungkinkan sistem merespons berbagai skenario secara fleksibel, dan pengguna dapat memilih implementasi yang paling sesuai dengan lingkungan mereka.

### Aturan Adaptasi Frontend

```typescript
// When localCodeAvailable=false, do not show code serve-web
// When localCodeAvailable=true, show the code serve-web configuration
```

Frontend secara otomatis menampilkan opsi yang tersedia berdasarkan lingkungan, sehingga pengguna tidak bingung dengan fitur yang tidak dapat mereka gunakan.

## Panduan Praktis

Setelah semua teori tersebut, apa yang harus Anda perhatikan selama penerapan sebenarnya? Pada akhirnya, implementasi adalah hal yang terpenting.

### Rekomendasi Penerapan Docker

Untuk penerapan dalam container, server kode adalah pilihan yang lebih baik:

```dockerfile
# Use the official code-server image directly
FROM codercom/code-server:latest

# Or install through npm
RUN npm install -g code-server
```

Ini memecahkan masalah dalam satu lapisan tanpa memerlukan instalasi VSCode tambahan.

### Contoh Konfigurasi

**konfigurasi server kode**
```json
{
  "vscodeServer": {
    "enabled": true,
    "activeImplementation": "code-server",
    "codeServer": {
      "host": "0.0.0.0",
      "port": 8080,
      "executablePath": "",
      "authMode": "none"
    }
  }
}
```

**konfigurasi web layanan kode**
```json
{
  "vscodeServer": {
    "enabled": true,
    "activeImplementation": "serve-web",
    "serveWeb": {
      "host": "0.0.0.0",
      "port": 8080,
      "executablePath": "/usr/local/bin/code"
    }
  }
}
```

Konfigurasi mungkin sedikit membosankan pada kali pertama, namun setelah diterapkan, segalanya menjadi lebih mudah untuk dipertahankan.

### Perbedaan Konstruksi URL

**server kode**
```
http://localhost:8080/?folder=/path/to/project&vscode-lang=zh-CN
```

**layanan kode-web**
```
http://localhost:8080/vscode-server/?folder=/path/to/project&tkn=xxx&vscode-lang=zh-CN
```

Perhatikan perbedaan jalur dan parameter. Anda perlu menanganinya secara terpisah selama integrasi.

### Peralihan Implementasi

Sistem mendukung peralihan waktu proses dan secara otomatis menghentikan implementasi sebelumnya saat peralihan:

```csharp
// VsCodeServerManager automatically handles mutual exclusion
// When switching activeImplementation, the old implementation will not keep running in the background
```

Desain ini memungkinkan pengguna mencoba penerapan yang berbeda kapan saja dan menemukan opsi yang paling sesuai bagi mereka.

### Pemantauan Status

```typescript
const { settings, runtime } = await getVsCodeServerSettings();

// runtime.activeImplementation: "code-server" | "serve-web"
// runtime.fallbackReason: reason for switching
// runtime.status: "running" | "starting" | "stopped" | "unhealthy"
```

Ketika status terlihat, pengguna dapat dengan cepat menentukan apakah suatu masalah berasal dari sisi server atau dari operasi mereka sendiri.

## Kesimpulan

| Dimensi Perbandingan | server kode | kode melayani-web | Rekomendasi |
|---------|-------------|----------------|------|
| Lisensi | MIT (ramah komersial) | Microsoft (terbatas) | server kode |
| Fleksibilitas penerapan | Penyebaran independen | Tergantung pada VSCode lokal | server kode |
| Kesesuaian server | Dirancang untuk server | Terutama untuk pembangunan lokal | server kode |
| Kontainerisasi | Dukungan asli | Memerlukan instalasi VSCode | server kode |
| Kelengkapan fitur | Dekat dengan edisi desktop | Versi lengkap resmi | kode melayani-web |
| Aktivitas pemeliharaan | Komunitas aktif | Secara resmi dikelola oleh Microsoft | Keduanya memiliki kekuatan |

**Strategi yang disarankan**: Gunakan server kode terlebih dahulu, dan pertimbangkan web server kode saat Anda memerlukan fungsionalitas resmi penuh dan sudah memiliki lingkungan VSCode lokal.
