## Membangun Orientasi Pengguna Baru yang Elegan di Proyek React: Latihan driver.js HagiCode

> Saat pengguna membuka produk Anda untuk pertama kalinya, apakah mereka benar-benar tahu harus mulai dari mana? Pada artikel ini, saya ingin berbicara sedikit tentang bagaimana kami menggunakan driver.js untuk orientasi pengguna baru di proyek HagiCode. Anggap saja ini sebagai contoh praktis kecil untuk memulai percakapan.

## Latar Belakang

Pernahkah Anda mengalami situasi ini? Pengguna baru mendaftar ke produk Anda, membuka halaman, dan langsung terlihat bingung. Mereka mengamati sekeliling, tidak yakin apa yang harus diklik atau apa yang harus dilakukan selanjutnya. Sebagai pengembang, kami sering berasumsi pengguna hanya akan "mengeksplorasi sendiri" karena keingintahuan manusia tidak terbatas. Namun kenyataannya berbeda: sebagian besar pengguna diam-diam keluar dalam beberapa menit jika mereka tidak dapat menemukan titik masuk yang tepat, seolah-olah cerita dimulai secara tiba-tiba dan berakhir secara alami.

Orientasi pengguna baru adalah cara penting untuk mengatasi masalah ini, namun membangunnya dengan baik tidaklah sesederhana itu. Sistem orientasi yang baik perlu:

- Temukan elemen halaman dengan tepat dan sorot elemen tersebut
- Mendukung alur orientasi multi-langkah
- Ingat pilihan pengguna (selesaikan atau lewati)
- Hindari memengaruhi kinerja halaman dan interaksi normal
- Jaga agar struktur kode tetap jelas dan mudah dipelihara

Saat membuat HagiCode, kami menghadapi tantangan yang sama. HagiCode adalah asisten pengkodean AI, dan alur kerja intinya adalah alur kerja OpenSpec yang terlihat seperti ini: "pengguna membuat Proposal -> AI membuat rencana -> pengguna meninjaunya -> AI mengeksekusinya." Bagi pengguna yang baru pertama kali menemukan konsep ini, alur kerjanya benar-benar baru, sehingga mereka memerlukan orientasi yang solid untuk memulai dengan cepat. Hal-hal baru selalu membutuhkan sedikit waktu untuk membiasakan diri.

## Tentang HagiCode

Pendekatan yang dibagikan dalam artikel ini berasal dari pengalaman langsung kami di [Kode Hagi](https://hagicode.com) proyek. HagiCode adalah asisten pengkodean AI berbasis Claude yang membantu pengembang menyelesaikan tugas pengkodean dengan lebih efisien melalui alur kerja OpenSpec. Anda dapat melihat kode sumber terbuka kami di [GitHub](https://github.com/HagiCode-org/site).

## Mengapa Kami Memilih driver.js

Selama fase evaluasi teknis, kami melihat beberapa perpustakaan orientasi utama. Masing-masing memiliki kekuatannya sendiri:

- **Intro.js**: Kuat, namun relatif besar, dan penyesuaian gaya agak rumit
- **Shepherd.js**: API yang dirancang dengan baik, tetapi agak terlalu "berat" untuk kasus penggunaan kami
- **driver.js**: API yang ringan, ringkas, intuitif, dan berfungsi dengan baik di ekosistem React

Pada akhirnya, kami memilih driver.js. Tidak ada alasan yang sangat dramatis. Pilihannya terutama didasarkan pada pertimbangan berikut:

1. **Ringan**: Pustaka inti berukuran kecil dan tidak meningkatkan ukuran paket secara signifikan
2. **API Sederhana**: Konfigurasinya jelas dan intuitif, sehingga mudah untuk digunakan
3. **Fleksibel**: Mendukung pemosisian, gaya, dan perilaku interaksi khusus
4. **Impor dinamis**: Dapat dimuat sesuai permintaan tanpa memengaruhi performa layar pertama

Dalam pemilihan teknologi, jarang sekali ada jawaban yang terbaik secara universal. Biasanya hanya ada pilihan yang paling sesuai.

## Teknis Implementasi

### Konfigurasi Inti

driver.js memiliki model konfigurasi yang sangat intuitif. Berikut adalah konfigurasi inti yang kami gunakan dalam proyek HagiCode:

```typescript
import { driver } from 'driver.js';
import 'driver.js/dist/driver.css';

const newConversationDriver = driver({
  allowClose: true,           // Allow users to close the guide
  animate: true,              // Enable animations
  overlayClickBehavior: 'close', // Close the guide when the overlay is clicked
  disableActiveInteraction: false, // Keep elements interactive
  showProgress: false,        // Do not show the progress bar (we manage progress ourselves)
  steps: guideSteps           // Array of guide steps
});
```

Alasan di balik pengaturan ini adalah:

- `allowClose: true` - Hormati pilihan pengguna dan jangan memaksa mereka untuk menyelesaikan panduan ini
- `disableActiveInteraction: false` - Beberapa langkah memerlukan tindakan pengguna nyata, seperti mengetik masukan, sehingga interaksi tidak dapat dinonaktifkan
- `overlayClickBehavior: 'close'` - Berikan pengguna cara cepat untuk keluar

### Manajemen Negara

Mempertahankan status orientasi sangatlah penting. Kami tidak ingin memulai ulang panduan ini setiap kali halaman disegarkan, karena hal itu akan cepat mengganggu. HagiCode menggunakan `localStorage` untuk mengelola status panduan:

```typescript
export type GuideState = 'pending' | 'dismissed' | 'completed';

export interface UserGuideState {
  session: GuideState;
  detailGuides: Record<string, GuideState>;
}

// Read state
export const getUserGuideState = (): UserGuideState => {
  const state = localStorage.getItem('userGuideState');
  return state ? JSON.parse(state) : { session: 'pending', detailGuides: {} };
};

// Update state
export const setUserGuideState = (state: UserGuideState) => {
  localStorage.setItem('userGuideState', JSON.stringify(state));
};
```

Kami mendefinisikan tiga negara bagian:

- `pending`: Panduan ini masih dalam proses, dan pengguna belum menyelesaikan atau melewatkannya
- `dismissed`: Pengguna menutup panduan secara proaktif
- `completed`: Pengguna menyelesaikan semua langkah

Untuk orientasi halaman detail Proposal, kami juga mendukung pelacakan status yang lebih terperinci melalui `detailGuides` peta, karena satu Proposal dapat melalui beberapa tahap – rancangan, peninjauan, dan pelaksanaan selesai – dan setiap tahap memerlukan panduan yang berbeda. Keadaan selalu berubah, dan orientasi harus mencerminkan hal tersebut.

### Pemilihan Elemen Sasaran

driver.js menggunakan pemilih CSS untuk menemukan elemen target. HagiCode mengikuti konvensi sederhana: gunakan custom `data-guide` atribut untuk menandai target orientasi:

```typescript
const steps = [
  {
    element: '[data-guide="launch"]',
    popover: {
      title: 'Start a New Conversation',
      description: 'Click here to create a new conversation session...'
    }
  }
];
```

Secara komponen, tampilannya seperti ini:

```typescript
<button data-guide="launch" onClick={handleLaunch}>
  New Conversation
</button>
```

Manfaat dari pendekatan ini adalah:

- Hindari konflik dengan nama kelas gaya bisnis
- Semantik yang jelas, sehingga Anda dapat langsung mengetahui bahwa elemen tersebut terkait dengan orientasi
- Lebih mudah untuk mengelola dan memelihara secara konsisten

### Optimasi Impor Dinamis

Karena fitur orientasi hanya diperlukan dalam skenario tertentu, seperti kunjungan pertama pengguna, kami menggunakan impor dinamis untuk mengoptimalkan kinerja pemuatan awal:

```typescript
const initNewUserGuide = async () => {
  // Dynamically import driver.js
  const { driver } = await import('driver.js');
  await import('driver.js/dist/driver.css');

  // Initialize the guide
  const newConversationDriver = driver({
    // ...configuration
  });

  newConversationDriver.drive();
};
```

Dengan cara ini, driver.js dan stylesheet-nya hanya dimuat saat diperlukan dan tidak memengaruhi performa layar pertama. Tidak banyak orang yang senang menunggu sesuatu yang bahkan belum mereka perlukan.

## Desain Alur Orientasi

HagiCode mengimplementasikan dua jalur orientasi yang mencakup skenario inti pengguna.

### Orientasi Sesi (10 Langkah)

Jalur orientasi ini membantu pengguna menyelesaikan seluruh alur mulai dari membuat percakapan hingga mengirimkan Proposal lengkap pertama mereka:

1. **peluncuran** - Mulai panduan dan perkenalkan tombol "Percakapan Baru".
2. **compose** - Memandu pengguna untuk mengetikkan permintaan di kotak input
3. **kirim** - Memandu pengguna untuk mengklik tombol kirim
4. **proposal-launch-readme** - Memandu pengguna untuk membuat Proposal README
5. **proposal-compose-readme** - Memandu pengguna untuk mengedit konten permintaan README
6. **proposal-submit-readme** - Memandu pengguna untuk mengirimkan Proposal README
7. **proposal-launch-agents** - Memandu pengguna untuk membuat `AGENTS.md` Usulan
8. **proposal-compose-agents** - Memandu pengguna untuk mengedit `AGENTS.md` permintaan
9. **proposal-submit-agents** - Memandu pengguna untuk mengirimkan `AGENTS.md` Usulan
10. **proposal-tunggu** - Jelaskan bahwa AI sedang memproses dan pengguna harus menunggu sebentar

Ide di balik jalur ini adalah untuk memungkinkan pengguna merasakan alur kerja inti HagiCode secara langsung melalui dua tugas pembuatan Proposal nyata, satu untuk `README` dan satu untuk `AGENTS.md`. Ada perbedaan besar antara mendengar tentang alur kerja dan menjalaninya sendiri.

### Orientasi Detail Proposal (3 Langkah)

Saat pengguna memasuki halaman detail Proposal, HagiCode memicu panduan terkait berdasarkan status Proposal saat ini:

1. **drafting** (tahap draf) - Memandu pengguna untuk meninjau rencana yang dihasilkan AI
2. **meninjau** (tahap peninjauan) - Memandu pengguna untuk melaksanakan rencana
3. **executionCompleted** (tahap selesai) - Memandu pengguna untuk mengarsipkan rencana

Karakteristik yang menentukan dari panduan ini adalah bahwa panduan ini bersifat state-driven: panduan ini secara dinamis memutuskan langkah orientasi mana yang akan ditampilkan berdasarkan status Proposal yang sebenarnya. Banyak hal berubah, dan orientasi harus berubah seiring dengan perubahan tersebut.

### Mekanisme Coba Ulang Render Elemen

Dalam aplikasi React, elemen orientasi target mungkin belum selesai dirender, misalnya karena data asinkron masih dimuat. Untuk menangani hal ini, HagiCode mengimplementasikan mekanisme percobaan ulang:

```typescript
const waitForElement = (selector: string, maxRetries = 10, interval = 100) => {
  let retries = 0;
  return new Promise<HTMLElement>((resolve, reject) => {
    const checkElement = () => {
      const element = document.querySelector(selector) as HTMLElement;
      if (element) {
        resolve(element);
      } else if (retries < maxRetries) {
        retries++;
        setTimeout(checkElement, interval);
      } else {
        reject(new Error(`Element not found: ${selector}`));
      }
    };
    checkElement();
  });
};
```

Panggil fungsi ini sebelum menginisialisasi panduan untuk memastikan elemen target sudah ada. Terkadang menunggu lebih lama tidak ada gunanya.

## Ringkasan Praktik Terbaik

Berdasarkan pengalaman praktis HagiCode, berikut beberapa praktik terbaik utama:

### 1. Orientasi Harus Mudah untuk Keluar

Jangan memaksa pengguna untuk menyelesaikan orientasi. Beberapa pengguna pada dasarnya adalah penjelajah dan lebih suka mencari tahu sendiri. Berikan tombol "Lewati" yang jelas, ingat pilihannya, dan jangan ganggu lagi di lain waktu.

### 2. Jaga agar Konten Orientasi Singkat dan Tajam

Setiap langkah orientasi harus fokus pada satu tujuan:

- **Judul**: Singkat dan jelas, idealnya tidak lebih dari beberapa kata
- **Deskripsi**: Langsung ke pokok persoalan dan beri tahu pengguna apa ini dan mengapa ini penting

Hindari penjelasan yang bertele-tele. Perhatian pengguna terbatas selama orientasi, dan semakin banyak Anda mengucapkannya, semakin kecil kemungkinan mereka untuk membacanya.

### 3. Jaga agar Selector Tetap Stabil

Gunakan penanda elemen stabil yang tidak sering berubah. Kebiasaan `data-guide` atribut adalah pilihan yang baik. Hindari bergantung pada nama kelas atau struktur DOM, karena hal tersebut mudah diubah selama pemfaktoran ulang. Kode berubah setiap saat, namun beberapa jangkar harus tetap stabil bila memungkinkan.

### 4. Uji Orientasi Anda

HagiCode menyertakan kasus pengujian lengkap untuk fitur orientasi:

```typescript
describe('NewUserConversationGuide', () => {
  it('should initialize guide state correctly', () => {
    const state = getUserGuideState();
    expect(state.session).toBe('pending');
  });

  it('should update guide state correctly', () => {
    setUserGuideState({ session: 'completed', detailGuides: {} });
    const state = getUserGuideState();
    expect(state.session).toBe('completed');
  });
});
```

Pengujian membantu memastikan bahwa pemfaktoran ulang tidak merusak perilaku orientasi secara tidak sengaja. Tidak seorang pun ingin perubahan kecil pada kode secara diam-diam merusak fungsi yang sebelumnya berfungsi.

### 5. Optimalisasi Kinerja

- Gunakan impor dinamis untuk memuat lambat perpustakaan orientasi
- Hindari menginisialisasi logika orientasi setelah pengguna menyelesaikan panduan ini
- Pertimbangkan dampak animasi terhadap performa, dan nonaktifkan animasi tersebut di perangkat kelas bawah jika diperlukan

Kinerja, seperti banyak hal dalam hidup, memerlukan penganggaran yang cermat.

## Kesimpulan

Orientasi pengguna baru adalah bagian penting dalam meningkatkan pengalaman pengguna produk. Dalam proyek HagiCode, kami menggunakan driver.js untuk membangun sistem orientasi lengkap yang mencakup seluruh alur kerja mulai dari pembuatan sesi hingga eksekusi Proposal.

Poin inti yang ingin kami bagikan melalui artikel ini adalah:

1. **Pilihan teknis harus sesuai dengan kebutuhan aktual**: driver.js bukanlah opsi yang paling ampuh, namun paling cocok bagi kami
2. **Manajemen negara sangat penting**: Penggunaan `localStorage` untuk mempertahankan status orientasi dan menghindari interupsi pengguna berulang kali
3. **Desain orientasi harus tetap fokus**: Setiap langkah harus menyelesaikan satu masalah, dan tidak lebih
4. **Struktur kode harus tetap jelas**: Pisahkan konfigurasi orientasi, pengelolaan status, dan logika UI untuk mempermudah pemeliharaan

Jika Anda menambahkan pengguna baru ke proyek Anda sendiri, saya harap pengalaman praktis dalam artikel ini membantu. Tidak ada sesuatu pun yang mistis dalam teknologi semacam ini. Teruslah mencoba, terus rangkum apa yang Anda pelajari, dan segala sesuatunya secara bertahap menjadi lebih mudah.
