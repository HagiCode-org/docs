## Edit DESIGN.md Langsung di Antarmuka Web: Dari Ide hingga Implementasi

> Dalam sistem manajemen proyek MonoSpecs, `DESIGN.md` membawa desain arsitektur dan keputusan teknis suatu proyek. Namun alur kerja pengeditan tradisional memaksa pengguna untuk beralih ke editor eksternal. Pengalaman yang terpecah-pecah itu ibarat terputus di tengah pembacaan puisi: inspirasi hilang, begitu pula suasana hati. Artikel ini membagikan solusi yang kami praktikkan dalam proyek HagiCode: pengeditan `DESIGN.md` langsung di antarmuka web, dengan dukungan untuk mengimpor template dari situs desain online. Lagi pula, siapa yang tidak menikmati perasaan menyelesaikan semuanya dalam satu aliran?

## Latar Belakang

Sebagai pembawa inti dokumen desain proyek, `DESIGN.md` menyimpan informasi penting seperti desain arsitektur, keputusan teknis, dan panduan implementasi. Namun, pendekatan pengeditan tradisional mengharuskan pengguna untuk beralih ke editor eksternal seperti VS Code, mencari jalur fisik secara manual, dan kemudian mengedit file. Ini tidak terlalu rumit, tetapi setelah mengulangi prosesnya beberapa kali, menjadi melelahkan.

Masalahnya terutama muncul dalam bentuk berikut:

- **Alur kerja terfragmentasi**: pengguna harus terus-menerus beralih antara antarmuka manajemen web dan editor lokal, sehingga mengganggu kelangsungan alur kerja mereka, seperti halnya musik dipotong di tengah-tengah lagu.
- **Sulit untuk digunakan kembali**: situs desain telah menerbitkan perpustakaan templat desain yang kaya, namun tidak dapat diintegrasikan secara langsung ke dalam alur kerja pengeditan proyek. Barang bagus memang ada, tetapi Anda tetap tidak dapat menggunakannya di tempat yang Anda butuhkan.
- **Perulangan pengalaman tidak ada**: tidak ada pengulangan tertutup untuk "pratinjau-pilih-impor", sehingga pengguna harus menyalin dan menempelkan secara manual, sehingga meningkatkan risiko kesalahan.
- **Gesekan kolaborasi**: menjaga dokumen desain dan implementasi kode tetap sinkron menjadi proses yang penuh gesekan, yang merugikan efisiensi tim.

Untuk mengatasi masalah ini, kami memutuskan untuk menambahkan dukungan pengeditan langsung `DESIGN.md` di antarmuka web dan izinkan impor templat sekali klik dari situs desain online. Itu bukanlah keputusan yang mengejutkan. Kami hanya ingin membuat pengalaman pengembangan lebih lancar.

## Tentang HagiCode

Solusi yang dibagikan dalam artikel ini berasal dari pengalaman langsung kami dalam [Kode Hagi](https://hagicode.com) proyek. HagiCode adalah proyek asisten pengkodean berbasis AI, dan selama pengembangan kami sering kali perlu memelihara dokumen desain proyek. Untuk membantu tim berkolaborasi dengan lebih efisien, kami mengeksplorasi dan menerapkan solusi pengeditan dan impor online ini. Tidak ada yang misterius tentang hal itu. Kami mengalami masalah dan mencari cara untuk menyelesaikannya.

## Solusi Teknis

### Arsitektur Keseluruhan

Solusi ini menggunakan arsitektur terpisah frontend-backend dengan proxy asal yang sama, terutama terdiri dari lapisan berikut. Dalam praktiknya, desain dapat disimpulkan sebagai “setiap bagian melakukan tugasnya sendiri”:

**1. Lapisan editor frontend**
```typescript
// Core component: DesignMdManagementDrawer
// Location: repos/web/src/components/project/DesignMdManagementDrawer.tsx
// Responsibility: handle editing, saving, version conflict detection, and import flow
```

**2. Lapisan layanan backend**
```csharp
// Core service: ProjectAppService.DesignMd
// Location: repos/hagicode-core/src/PCode.Application/ProjectAppService.DesignMd.cs
// Responsibility: path resolution, file read/write, and version management
```

**3. Lapisan proxy asal yang sama**
```csharp
// Proxy service: ProjectAppService.DesignMdSiteIndex
// Location: repos/hagicode-core/src/PCode.Application/ProjectAppService.DesignMdSiteIndex.cs
// Responsibility: proxy design site resources, preview image caching, and security validation
```

### Keputusan Teknis Utama

#### Keputusan 1: Pola Laci Global

Kami menggunakan laci global tunggal, bukan lapisan pop-up lokal, dengan status dikelola `layoutSlice`, yang memberi pengguna pengalaman yang konsisten di seluruh tampilan (`classic` dan `kanban`). Terlepas dari tampilan mana pengguna membuka editor, mereka mendapatkan model interaksi yang sama. Pengalaman yang konsisten membuat orang merasa lebih nyaman daripada mengalami disorientasi saat berpindah pandangan.

#### Keputusan 2: API Cakupan Proyek

Kami menaikinya `DESIGN.md`-titik akhir terkait di bawah `ProjectController`, menggunakan kembali batasan izin proyek yang ada dan menghindari kerumitan dalam menambahkan pengontrol terpisah. Hal ini membuat penanganan izin menjadi lebih jelas dan juga selaras dengan organisasi sumber daya RESTful. Terkadang penggunaan kembali lebih bermakna daripada menciptakan sesuatu yang baru dari awal.

#### Keputusan 3: Deteksi Konflik Versi

Kami memperoleh versi buram dari sistem file `LastWriteTimeUtc`, yang memberi kita kontrol konkurensi optimis yang ringan. Ketika beberapa pengguna mengedit file yang sama sekaligus, sistem dapat mendeteksi konflik dan meminta pengguna untuk menyegarkan. Desain ini tidak menghalangi pengeditan, namun tetap melindungi konsistensi data.

#### Keputusan 4: Pola Proxy Asal yang Sama

Kami menggunakan `IHttpClientFactory` untuk mewakili sumber daya lokasi desain eksternal, menghindari masalah lintas asal dan risiko SSRF. Hal ini menjaga keamanan sistem sekaligus menyederhanakan panggilan frontend. Anda tidak bisa terlalu berhati-hati dengan keamanan.

## Implementasi Inti

### 1. Edit DESIGN.md Secara Langsung

#### Implementasi Backend

Backend terutama bertanggung jawab atas resolusi jalur, baca/tulis file, dan manajemen versi. Tugas-tugas ini mendasar, tetapi sangat diperlukan, seperti fondasi sebuah rumah:

```csharp
// Path resolution and security validation
private Task<string> ResolveDesignDocumentDirectoryAsync(string projectPath, string? repositoryPath)
{
    if (string.IsNullOrWhiteSpace(repositoryPath))
    {
        return Task.FromResult(Path.GetFullPath(projectPath));
    }
    return ValidateSubPathAsync(projectPath, repositoryPath);
}

// Version generation (based on file system timestamp and size)
private static string BuildDesignDocumentVersion(string path)
{
    var fileInfo = new FileInfo(path);
    fileInfo.Refresh();
    return string.Create(
        CultureInfo.InvariantCulture,
        $"{fileInfo.LastWriteTimeUtc.Ticks:x}-{fileInfo.Length:x}");
}
```

Desain versinya menarik karena kesederhanaannya: kami menggunakan waktu dan ukuran file yang terakhir diubah untuk menghasilkan pengidentifikasi versi unik. Ini ringan dan dapat diandalkan, tanpa database versi tambahan yang harus dipelihara. Solusi sederhana seringkali merupakan solusi yang paling efektif.

#### Implementasi Bagian Depan

Di frontend, kami menerapkan deteksi keadaan kotor dan logika penyimpanan. Desain ini membantu pengguna memahami apakah perubahan mereka telah disimpan dan mengurangi kekhawatiran "bagaimana jika saya kehilangannya?":

```typescript
// Dirty-state detection and save logic
const [draft, setDraft] = useState('');
const [savedDraft, setSavedDraft] = useState('');
const isDirty = draft !== savedDraft;

const handleSave = useCallback(async () => {
    const result = await saveProjectDesignMdDocument({
        ...activeTarget,
        content: draft,
        expectedVersion: document.version, // optimistic concurrency control
    });
    setSavedDraft(draft); // update saved state
}, [activeTarget, document, draft]);
```

Dalam implementasi ini, kami mempertahankan dua bagian negara: `draft` adalah konten yang sedang diedit, sementara `savedDraft` adalah konten yang disimpan. Membandingkannya akan memberi tahu kita apakah ada perubahan yang belum disimpan. Desainnya sederhana, namun memberikan ketenangan pikiran bagi orang-orang. Tak seorang pun ingin hal yang telah mereka kerjakan dengan keras menghilang.

### 2. Impor File Desain dari Sumber Online

#### Struktur Direktori

```
repos/index/

repos/awesome-design-md-site/

│       │   └── DESIGN.md
│       └── ...
└── src/lib/content/

```

#### Format Data Indeks

File indeks di situs desain mendefinisikan semua templat yang tersedia. Dengan indeks ini, pengguna dapat memilih template yang diinginkan semudah memesan dari menu:

```json
{
  "entries": [
    {
      "slug": "linear.app",
      "title": "Linear Inspired Design System",
      "summary": "AI Product / Dark Feel",
      "detailUrl": "/designs/linear.app/",
      "designDownloadUrl": "/designs/linear.app/DESIGN.md",
      "previewLightImageUrl": "...",
      "previewDarkImageUrl": "..."
    }
  ]
}
```

Setiap entri menyertakan informasi dasar templat dan tautan unduhan. Backend membaca daftar templat yang tersedia dari indeks ini dan menyajikannya kepada pengguna untuk dipilih. Hal ini membuat pemilihan menjadi intuitif dan bukannya memaksa orang untuk menentukan arah dalam kegelapan.

#### Implementasi Proxy Asal yang Sama

Untuk menjaga keamanan, backend melakukan validasi ketat pada akses ke situs desain. Anda tidak bisa terlalu berhati-hati tentang keamanan:

```csharp
// Safe slug validation
private static readonly Regex SafeDesignSiteSlugRegex =
    new("^[A-Za-z0-9](?:[A-Za-z0-9._-]{0,127})$", RegexOptions.Compiled);

private static string NormalizeDesignSiteSlug(string slug)
{
    var normalizedSlug = slug?.Trim() ?? string.Empty;
    if (!IsSafeDesignSiteSlug(normalizedSlug))
    {
        throw new BusinessException(
            ProjectDesignSiteIndexErrorCodes.InvalidSlug,
            "Design site slug must be a single safe path segment.");
    }
    return normalizedSlug;
}

// Preview image caching (OS temp directory)
private static string ComputePreviewCacheKey(string slug, string theme, string previewUrl)
{
    var raw = $"{slug}|{theme}|{previewUrl}";
    var bytes = SHA256.HashData(Encoding.UTF8.GetBytes(raw));
    return Convert.ToHexString(bytes).ToLowerInvariant();
}
```

Kami melakukan dua hal di sini: pertama, kami memvalidasi format slug secara ketat dengan ekspresi reguler untuk mencegah serangan traversal jalur; kedua, kami menyimpan gambar pratinjau dalam cache untuk mengurangi tekanan pada situs eksternal. Yang pertama adalah perlindungan, yang kedua adalah optimasi, dan keduanya penting.

### 3. Arus Impor Penuh

```typescript
// 1. Open the import drawer
const handleRequestImportDrawer = useCallback(() => {
    setIsImportDrawerOpen(true);
}, []);

// 2. Select and import
const handleImportRequest = useCallback((entry) => {
    if (isDirty) {
        setPendingImportEntry(entry);
        setConfirmMode('import'); // overwrite confirmation
        return;
    }
    void executeImport(entry);
}, [isDirty]);

// 3. Execute import
const executeImport = useCallback(async (entry) => {
    const result = await getProjectDesignMdSiteImportDocument(
        activeTarget.projectId,
        entry.slug
    );
    setDraft(result.content); // replace editor text only, do not save automatically
    setIsImportDrawerOpen(false);
}, [activeTarget?.projectId]);
```

Alur impor mengikuti prinsip "konfirmasi pengguna": setelah impor, hanya konten editor yang diperbarui, dan tidak ada yang disimpan secara otomatis. Pengguna dapat memeriksa konten yang diimpor dan menyimpannya secara manual hanya setelah memastikan konten tersebut terlihat benar. Keputusan akhir harus tetap berada di tangan pengguna.

## Contoh Praktis

### Skenario 1: Membuat DESIGN.md di Project Root

Kapan `DESIGN.md` tidak ada, backend mengembalikan status dokumen virtual. Hal ini memungkinkan frontend menghindari penanganan khusus untuk kasus "file tidak ada", dan API terpadu menyederhanakan logika kode:

```csharp
return new ProjectDesignDocumentDto
{
    Path = targetPath,
    Exists = false,  // virtual document state
    Content = string.Empty,
    Version = null
};

// Automatically create the file on first save
public async Task<SaveProjectDesignDocumentResultDto> SaveDesignDocumentAsync(...)
{
    Directory.CreateDirectory(targetDirectory);
    await File.WriteAllTextAsync(targetPath, input.Content);
    return new SaveProjectDesignDocumentResultDto { Created = !exists };
}
```

Keuntungan dari desain ini adalah frontend tidak memerlukan logika kasus khusus untuk file yang hilang. Dengan menyembunyikan kompleksitas tersebut di backend, frontend dapat lebih mudah fokus pada pengalaman pengguna.

### Skenario 2: Impor Templat dari Situs Desain

Setelah pengguna memilih templat desain "Linear" di laci impor, sistem akan mengambil `DESIGN.md` konten melalui proxy backend. Seluruh proses transparan bagi pengguna: mereka hanya memilih template, dan sistem menangani permintaan jaringan dan transformasi data secara otomatis.

```typescript
// 1. The system fetches DESIGN.md content through the backend proxy
GET /api/project/{id}/design-md/site-index/linear.app

// 2. The backend validates the slug and fetches content from upstream
var entry = FindDesignSiteEntry(catalog, "linear.app");
using var upstreamResponse = await httpClient.SendAsync(request);
var content = await upstreamResponse.Content.ReadAsStringAsync();

// 3. The frontend replaces the editor text
setDraft(result.content);
// The user reviews it and then saves it manually to disk
```

Seluruh aliran tetap transparan bagi pengguna. Mereka tinggal memilih template, dan sistem akan menangani jaringan dan transformasi di balik layar. Itulah pengalaman yang kami inginkan: sederhana, namun kuat.

### Skenario 3: Menangani Konflik Versi

Ketika beberapa pengguna mengedit hal yang sama `DESIGN.md` pada saat yang sama, sistem mendeteksi konflik versi. Mekanisme kontrol konkurensi optimis ini menjaga konsistensi data tanpa menghalangi pengeditan pengguna:

```csharp
if (!string.Equals(currentVersion, expectedVersion, StringComparison.Ordinal))
{
    throw new BusinessException(
        ProjectDesignDocumentErrorCodes.VersionConflict,
        $"DESIGN.md at '{targetPath}' changed on disk.");
}
```

Frontend menangkap kesalahan ini dan meminta pengguna:

```typescript
// Frontend prompts the user to refresh and retry
<Alert>
    <AlertTitle>Version conflict</AlertTitle>
    <AlertDescription>
        The file was modified by another process. Please refresh to get the latest version and try again.
    </AlertDescription>
</Alert>
```

Mekanisme kontrol konkurensi yang optimis ini menjaga konsistensi data tanpa menghalangi pengguna saat mereka bekerja. Konflik tidak dapat dihindari, namun setidaknya pengguna harus mengetahui apa yang terjadi daripada diam-diam kehilangan perubahannya.

## Catatan dan Praktik Terbaik

### 1. Keamanan Jalur

Selalu validasi `repositoryPath` untuk mencegah serangan traversal jalur. Anda tidak akan pernah bisa berbuat banyak dalam hal keamanan:

```csharp
// Always validate repositoryPath to prevent path traversal attacks
return ValidateSubPathAsync(projectPath, repositoryPath);
// Reject dangerous inputs such as "../" and absolute paths
```

### 2. Strategi Tembolok

Gambar pratinjau cache selama 24 jam, dengan maksimal 160 file. Caching moderat meningkatkan kinerja, namun keseimbangan tetap penting:

```csharp
// Cache preview images for 24 hours, with a maximum of 160 files
private static readonly TimeSpan PreviewCacheTtl = TimeSpan.FromHours(24);
private const int PreviewCacheMaxFiles = 160;
// Periodically clean up expired cache
```

### 3. Penanganan Kesalahan

Turunkan dengan baik ketika situs upstream tidak tersedia. Desain ini memastikan bahwa meskipun ketergantungan eksternal gagal, fungsi pengeditan inti masih berfungsi normal:

```typescript
// Gracefully degrade when the upstream site is unavailable
try {
    const catalog = await getProjectDesignMdSiteImportCatalog(projectId);
} catch (error) {
    toast.error(t('project.designMd.siteImport.feedback.catalogLoadFailed'));
    // The main editing drawer remains available
}
```

Degradasi yang baik ini memastikan bahwa meskipun dependensi eksternal tidak tersedia, fungsi pengeditan inti tetap berfungsi. Sebuah sistem harus memiliki ketahanan, bukannya runtuh saat terjadi kesalahan.

### 4. Optimasi Pengalaman Pengguna

Konfirmasikan penimpaan sebelum mengimpor, dan jangan menyimpan secara otomatis setelah mengimpor. Pengguna harus tetap mengendalikan tindakan mereka sendiri:

```typescript
// Confirm overwrite before import
if (isDirty) {
    setConfirmMode('import');
    return;
}

// Do not save automatically after import; let the user confirm
setDraft(result.content); // update draft only
// The content is written to disk only after the user reviews it and clicks Save
```

### 5. Pertimbangan Kinerja

Gunakan pabrik klien HTTP untuk menghindari pembuatan terlalu banyak koneksi. Pengelolaan sumber daya mungkin tampak kecil, namun melakukannya dengan baik dapat membawa perbedaan besar:

```csharp
// Use an HTTP client factory to avoid creating too many connections
private const string DesignSiteProxyClientName = "ProjectDesignSiteProxy";
private static readonly TimeSpan DesignSiteProxyTimeout = TimeSpan.FromSeconds(8);
```

## Ekstensi yang Disarankan

1. **Peningkatan penurunan harga**: saat ini kami menggunakan dasar `Textarea`, tapi kami dapat meningkatkan ke CodeMirror untuk penyorotan sintaksis dan pintasan keyboard. Ketika editor merasa lebih baik, menulis dokumentasi pun terasa lebih baik.
2. **Mode pratinjau**: tambahkan pratinjau penurunan harga secara real-time untuk meningkatkan pengalaman pengeditan. Apa yang Anda lihat adalah apa yang Anda dapatkan selalu membuat orang lebih percaya diri.
3. **Penggabungan diff**: menerapkan algoritme penggabungan cerdas alih-alih penggantian teks lengkap sederhana. Konflik tidak bisa dihindari, namun proses penyelesaian konflik tidak harus menyakitkan.
4. **Caching lokal**: cache `design.json` dalam database untuk mengurangi ketergantungan pada situs eksternal. Semakin sedikit ketergantungan yang dimiliki suatu sistem, cenderung semakin stabil sistem tersebut.
