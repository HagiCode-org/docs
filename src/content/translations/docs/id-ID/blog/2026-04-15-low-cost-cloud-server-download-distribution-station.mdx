## Cara Membangun Stasiun Distribusi Unduhan Cepat dengan Server Cloud Berbiaya Rendah

> Bandwidth penyimpanan cloud sangat mahal, akses lintas batas sangat lambat, dan harga CDN cukup untuk menakut-nakuti siapa pun... Jika Anda menangani distribusi file, Anda mungkin mengetahui masalah ini dengan baik. Dalam postingan ini, saya ingin berbagi pendekatan berbiaya rendah yang kami lakukan saat membangun [Kode Hagi](https://hagicode.com): server cloud ditambah lapisan caching Nginx. Biayanya turun sekitar setengahnya, sementara kecepatan pengunduhan sedikit meningkat, yang setidaknya sedikit melegakan.

## Latar Belakang

Ketika berbicara tentang internet, kecepatan dan stabilitas pengunduhan benar-benar merupakan bagian dari pengalaman pengguna. Baik Anda menjalankan proyek sumber terbuka atau produk komersial, Anda tetap perlu menyediakan cara yang andal kepada pengguna untuk mengunduh file.

Mengunduh file langsung dari penyimpanan cloud, seperti Azure Blob Storage atau AWS S3, terlihat sederhana, namun memiliki beberapa masalah praktis:

**Latensi jaringan**: Akses lintas negara dan lintas wilayah bisa jadi cukup lambat hingga membuat Anda ingin menghancurkan keyboard Anda. Jika pengguna harus menunggu selamanya, pengalamannya jelas tidak akan bagus.

**Biaya bandwidth**: Lalu lintas keluar penyimpanan cloud sangat mahal. Mengakses Azure Blob Storage dari Tiongkok daratan memerlukan biaya sekitar CNY 0,5 per GB, yang berarti 1 TB per bulan berarti sekitar CNY 500. Untuk tim kecil, jumlah tersebut bukanlah jumlah yang kecil. Lagi pula, uang tidak ada yang datang dari angin.

**Pembatasan akses**: Di beberapa wilayah, akses ke layanan cloud luar negeri tidak stabil, dan terkadang tidak tersedia. Pengguna ingin mengunduh file tetapi tidak bisa, yang membuat frustrasi semua orang.

**Biaya CDN**: CDN komersial dapat mengatasi masalah ini, namun harganya juga sama nyatanya. Kebanyakan tim kecil tidak bisa membenarkan hal ini.

Jadi apakah ada solusi yang terjangkau dan praktis? Ya. Gunakan server cloud, proxy terbalik, dan lapisan cache. Ini adalah pendekatan yang mudah, tetapi berhasil. Biayanya turun sekitar setengahnya, dan kecepatannya juga meningkat, yang merupakan trade-off yang layak.

## Desain Arsitektur

### Arsitektur Keseluruhan

Mari kita mulai dengan arsitektur lengkapnya:

```
User request
    ↓
DNS resolution
    ↓
┌─────────────────────────────────────┐
│   Reverse proxy layer (Traefik/Bunker Web) │ ← SSL termination, routing, security protection
├─────────────────────────────────────┤
│   Ports: 80/443                     │
│   Features: Automatic Let's Encrypt certificates │
│             Host routing            │
└─────────────────────────────────────┘
    ↓
┌─────────────────────────────────────┐
│   Cache layer (Nginx)               │ ← File caching, Gzip compression
├─────────────────────────────────────┤
│   Ports: 8080(server) / 8081(desktop) │
│   Cache strategy:                   │
│   - index.json: 1 hour              │
│   - other files: 7 days             │
│   Cache size: 1GB                   │
└─────────────────────────────────────┘
    ↓
┌─────────────────────────────────────┐
│   Origin (Azure Blob Storage)       │ ← File storage
└─────────────────────────────────────┘
```

Ide inti arsitektur ini sederhana: **letakkan cache antara pengguna dan penyimpanan cloud**.

Permintaan pengguna pertama-tama tiba di lapisan proxy terbalik di server cloud, dan kemudian lapisan cache Nginx mengambil alih. Jika file yang diminta sudah di-cache, file tersebut akan segera dikembalikan. Jika tidak, Nginx akan mengambilnya dari penyimpanan cloud dan menyimpan salinan lokal pada saat yang bersamaan. Saat berikutnya seseorang meminta file yang sama, penyimpanan cloud tidak perlu dilibatkan lagi.

### Mengapa Memilih Arsitektur Ini?

**Kelebihan server cloud**:
- Biaya yang dapat diprediksi: penyedia seperti Alibaba Cloud menawarkan server cloud berbiaya rendah, dengan 1-2 vCPU dan 2 GB RAM dengan harga sekitar CNY 50-100 per bulan
- Penerapan yang fleksibel: Anda dapat mengonfigurasi aturan proksi terbalik dan kebijakan cache dengan bebas
- Geografi yang fleksibel: Anda dapat memilih wilayah server yang lebih dekat dengan pengguna Anda
- Skalabilitas yang baik: Anda dapat meningkatkan spesifikasi server seiring pertumbuhan lalu lintas

**Proxy terbalik + arsitektur cache**:
- Kurangi tekanan asal: cache file panas untuk mengurangi akses langsung ke penyimpanan cloud
- Biaya lebih rendah: lalu lintas server cloud jauh lebih murah daripada jalan keluar penyimpanan cloud
- Tingkatkan kecepatan: akses terdekat dan bandwidth server biasanya lebih baik daripada pengiriman penyimpanan cloud langsung

**Mengapa memilih Nginx sebagai lapisan cache?**

Ini bukanlah pilihan yang acak. Nginx memiliki beberapa keunggulan nyata di sini:

1. Performa tinggi: Nginx dikenal luas karena performa reverse proxy yang luar biasa
2. Caching matang: bawaan `proxy_cache` fitur stabil dan dapat diandalkan
3. Penggunaan sumber daya rendah: dapat dijalankan hanya dengan memori 256 MB
4. Konfigurasi fleksibel: Anda dapat menerapkan kebijakan cache yang berbeda untuk jenis file yang berbeda

## Lapisan Proksi Terbalik: Traefik vs Bunker Web

Solusi penerapan HagiCode mendukung dua opsi proxy terbalik, dan masing-masing memiliki kelebihannya sendiri:

| Pilihan | Karakteristik | Skenario yang Cocok |
|------|------|----------|
| **Traefik** | SSL ringan, otomatis, konfigurasi sederhana | Penerapan dasar, skenario lalu lintas rendah |
| **Jaring Bunker** | WAF bawaan, perlindungan DDoS, perlindungan anti-bot | Skenario keamanan tinggi dan lalu lintas tinggi |

### Traefik: Pilihan Pertama Kelas Ringan

Traefik adalah proxy balik HTTP modern dan penyeimbang beban. Keuntungan terbesarnya adalah konfigurasinya sederhana, dan dapat memperoleh sertifikat Let's Encrypt secara otomatis.

Untuk penerapan awal atau skenario lalu lintas rendah, Traefik sering kali merupakan pilihan yang sangat baik:
- Ia menggunakan sumber daya yang relatif sedikit; 1,5 CPU dan memori 512 MB sudah cukup
- Sertifikat SSL dikonfigurasi secara otomatis, jadi Anda tidak perlu mengelolanya sendiri
- Perutean dikonfigurasi melalui label Docker, yang cukup nyaman

### Web Bunker: Untuk Skenario Keamanan Tinggi

Bunker Web adalah firewall aplikasi web berbasis Nginx dengan perlindungan keamanan yang lebih komprehensif.

Kapan sebaiknya Anda mempertimbangkan untuk beralih ke Bunker Web? Biasanya dalam kasus seperti ini:
- Anda berada di bawah serangan DDoS
- Anda memerlukan perlindungan ModSecurity
- Anda menginginkan perlindungan anti-bot
- Anda memiliki persyaratan keamanan yang lebih ketat

HagiCode menyediakan `switch-deployment.sh` skrip sehingga Anda dapat beralih dengan cepat di antara dua opsi:

```bash
# Switch to Bunker Web
./switch-deployment.sh bunkerweb

# Switch back to Traefik
./switch-deployment.sh traefik

# Check current status
./switch-deployment.sh status
```

Skrip melakukan pemeriksaan awal, pemeriksaan kesehatan, dan rollback otomatis, sehingga proses peralihan cukup aman dan andal.

## Konfigurasi Lapisan Cache Nginx

Lapisan cache adalah inti dari keseluruhan arsitektur, sehingga konfigurasi Nginx membuat perbedaan besar dalam kinerja cache.

### Konfigurasi Jalur Cache

```nginx
# Cache path configuration
proxy_cache_path /var/cache/nginx levels=1:2 keys_zone=azure_cache:10m
                   max_size=1g inactive=7d use_temp_path=off;
```

Detail parameter:
- `levels=1:2`: hierarki direktori cache dengan dua tingkat untuk meningkatkan efisiensi akses file
- `keys_zone=azure_cache:10m`: zona penyimpanan kunci cache; 10 MB cukup untuk sejumlah besar kunci
- `max_size=1g`: ukuran cache maksimum adalah 1 GB
- `inactive=7d`: menghapus file cache jika belum diakses selama 7 hari
- `use_temp_path=off`: menulis langsung ke direktori cache untuk kinerja yang lebih baik

### Strategi Cache Berjenjang

Jenis file yang berbeda memerlukan strategi cache yang berbeda:

```nginx
# Server download service
server {
    listen 8080;

    # Short-term cache for index.json (to allow timely updates)
    location /index.json {
        proxy_cache azure_cache;
        proxy_cache_valid 200 1h;
        proxy_cache_key "$scheme$server_port$request_uri";
        add_header X-Cache-Status $upstream_cache_status;
        add_header Cache-Control "public, max-age=3600";

        # Reverse proxy to Azure OSS
        proxy_pass https://${SERVER_DL_HOST}/${SERVER_DL_CONTAINER}$uri?${SERVER_DL_SAS_TOKEN};
        proxy_ssl_server_name on;
        proxy_ssl_protocols TLSv1.2 TLSv1.3;
    }

    # Long-term cache for static files such as installation packages
    location / {
        proxy_cache azure_cache;
        proxy_cache_valid 200 7d;
        proxy_cache_key "$scheme$server_port$request_uri";
        add_header X-Cache-Status $upstream_cache_status;
        add_header Cache-Control "public, max-age=604800";

        proxy_pass https://${SERVER_DL_HOST}/${SERVER_DL_CONTAINER}$uri?${SERVER_DL_SAS_TOKEN};
        proxy_ssl_server_name on;
        proxy_ssl_protocols TLSv1.2 TLSv1.3;
    }
}
```

**Mengapa dirancang seperti ini?**

`index.json` adalah file pemeriksaan versi, sehingga perlu segera diperbarui. Dengan jendela cache 1 jam, pengguna dapat mendeteksi rilis baru paling lama satu jam setelah publikasi.

File statis seperti paket instalasi jarang berubah, jadi menyimpannya dalam cache selama 7 hari sangat mengurangi akses asal. Ketika pembaruan diperlukan, Anda cukup menghapus cache secara manual.

**`X-Cache-Status` tajuk tanggapan**:

Header ini membantu Anda memeriksa perilaku cache hit:
- `HIT`: cache terkena
- `MISS`: cache miss, diambil dari asal
- `EXPIRED`: cache kedaluwarsa, diambil dari asal lagi
- `BYPASS`: cache dilewati

Cara memeriksanya:
```bash
curl -I https://server.dl.hagicode.com/app.zip
```

Asumsikan lalu lintas unduhan 1 TB per bulan. Mari kita hitung:

| Pilihan | Biaya Lalu Lintas | Biaya Server | Jumlah |
|------|----------|------------|------|
| Azure OSS langsung | Sekitar CNY 500 | CNY 0 | CNY 500 |
| Server cloud + OSS (rasio cache hit 80%) | CNY 100 + CNY 80 | CNY 60 | CNY240 |

## Praktek Penerapan

### Persiapan Lingkungan

Pertama, konfigurasikan variabel lingkungan:

```bash
cd /path/to/hagicode_aliyun_deployment/docker
cp .env.example .env
vi .env  # Fill in the Azure OSS SAS URL and Lark Webhook URL
```

**Penting**: itu `.env` file berisi informasi sensitif seperti Token SAS dan URL Webhook. Jangan pernah mengkomitnya ke kontrol versi.

### Konfigurasi DNS

Tambahkan catatan DNS A berikut:
- `server.dl.hagicode.com` → IP server
- `desktop.dl.hagicode.com` → IP server

### Inisialisasi Server

Gunakan Ansible untuk menginisialisasi server secara otomatis:

```bash
cd /path/to/hagicode_aliyun_deployment
ansible-playbook -i ./ansible/inventory/hosts.yml ./ansible/playbooks/init.yml
```

Buku pedoman ini menangani tugas-tugas berikut secara otomatis:
- Buat pengguna penerapan
- Instal Docker dan Docker Compose
- Konfigurasikan kunci SSH
- Tetapkan aturan firewall

Itu adalah pekerjaan penyiapan utama, dan otomatisasi menghemat banyak waktu.

## Kiat Operasi

### Manajemen Tembolok

Cache juga memerlukan pemeliharaan dari waktu ke waktu:

**Periksa penggunaan cache disk**:
```bash
docker volume inspect docker_nginx-cache
du -sh /var/lib/docker/volumes/docker_nginx-cache/_data
```

**Hapus cache secara manual**:
```bash
./clear-cache.sh
```

Atau jalankan perintah manual secara langsung jika diperlukan:
```bash
docker exec nginx sh -c "rm -rf /var/cache/nginx/*"
docker restart nginx
```

### Batasan Sumber Daya

Pada server 1-core, 2 GB, konfigurasi batas sumber daya terlihat seperti ini:

```yaml
services:
  traefik:
    deploy:
      resources:
        limits:
          cpus: '1.50'
          memory: 512M

  nginx:
    deploy:
      resources:
        limits:
          cpus: '0.50'
          memory: 256M
```

Untuk memantau penggunaan sumber daya, Anda kadang-kadang dapat menjalankan:
```bash
docker stats
```

### Praktik Keamanan Token SAS

Token SAS adalah kredensial yang digunakan untuk mengakses Azure Blob Storage, jadi membocorkannya akan berakibat serius:

- Jangan melakukan `.env` file ke kontrol versi; itu sudah masuk `.gitignore`
- Tetapkan waktu kedaluwarsa Token SAS yang sesuai, dengan direkomendasikan 1 tahun
- Batasi izin Token SAS menjadi hanya baca
- Rotasi Token SAS secara teratur

### Pemantauan dan Peringatan

HagiCode mengintegrasikan notifikasi Lark/Feishu Webhook, yang dapat mengirimkan peringatan untuk kejadian berikut:
- Keberhasilan atau kegagalan penerapan
- Status pembersihan cache

Pemberitahuan mencakup informasi server, stempel waktu, dan detail kesalahan, membuat pemecahan masalah menjadi lebih cepat.

### Ekstensi Ketersediaan Tinggi

Ketika satu server tidak lagi cukup, Anda dapat mempertimbangkan hal berikut:

1. **Penskalaan horizontal**: menerapkan beberapa node dan mendistribusikan lalu lintas dengan DNS round-robin atau penyeimbang beban
2. **CDN di depan**: letakkan CDN di depan server cloud untuk akses yang lebih cepat
3. **Pemanasan cache**: gunakan skrip untuk memuat file panas terlebih dahulu ke dalam cache

## Catatan

Ada beberapa hal yang perlu diingat:

1. **Sertifikat SSL**: Let's Encrypt memiliki batas kecepatan, jadi jangan terlalu sering berpindah penerapan atau penerbitan sertifikat mungkin gagal
2. **Pembersihan cache**: setelah memperbarui file penting, ingatlah untuk menghapus cache atau pengguna masih dapat mengunduh versi lama
3. **Manajemen log**: bersihkan log Docker secara teratur, atau disk akan terisi
4. **Strategi pencadangan**: mencadangkan file seperti `Traefik acme.json` dan konfigurasi Web Bunker
5. **Pemantauan dan peringatan**: konfigurasikan notifikasi Feishu sehingga Anda dapat melacak status penerapan dan merespons masalah dengan cepat
