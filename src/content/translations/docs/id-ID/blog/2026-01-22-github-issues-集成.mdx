## Membangun Integrasi Masalah GitHub dari Awal: Praktik Koneksi Langsung Frontend HagiCode

> Artikel ini mendokumentasikan proses penuh pengintegrasian Masalah GitHub ke dalam platform HagiCode. Kita akan mempelajari cara menggunakan arsitektur "koneksi langsung frontend + backend minimal" untuk mencapai autentikasi OAuth yang aman dan sinkronisasi masalah yang efisien sekaligus menjaga backend tetap ringan.

## Latar Belakang: Mengapa Mengintegrasikan GitHub?

Sebagai platform pengembangan yang dibantu AI, nilai inti HagiCode terletak pada menghubungkan ide dengan implementasi. Namun dalam penggunaan sebenarnya, kami menemukan bahwa setelah pengguna menyelesaikan Proposal di HagiCode, mereka sering kali perlu menyalin konten secara manual ke Masalah GitHub untuk pelacakan proyek.

Hal ini menciptakan beberapa masalah yang jelas:

1.  **Alur kerja terfragmentasi**: Pengguna perlu beralih antara dua sistem. Pengalamannya tidak mulus, dan informasi penting dapat dengan mudah hilang saat disalin dan ditempel.
2.  **Kolaborasi yang tidak nyaman**: Anggota tim lain terbiasa memeriksa tugas di GitHub dan tidak dapat melihat kemajuan proposal secara langsung di dalam HagiCode.
3.  **Pekerjaan manual yang berulang**: Setiap kali proposal diperbarui, seseorang harus memperbarui masalah terkait di GitHub secara manual, sehingga menambah biaya pemeliharaan yang tidak diperlukan.

Untuk mengatasi masalah ini, kami memutuskan untuk memperkenalkan fitur **GitHub Issues Integration**, yang menghubungkan sesi HagiCode dengan repositori GitHub untuk mengaktifkan "sinkronisasi satu klik".

## Pilihan Teknis: Koneksi Langsung Frontend vs Proxy Backend

Saat merancang pendekatan integrasi, kami memiliki dua opsi: "model proxy backend" tradisional dan "model koneksi langsung frontend" yang lebih agresif.

### Perbandingan Solusi

Dalam **model proksi backend** tradisional, setiap permintaan dari frontend harus terlebih dahulu melalui backend kami, yang kemudian memanggil GitHub API. Hal ini memusatkan logika, tetapi juga memberikan beban yang signifikan pada backend:

1.  **Backend yang membengkak**: Kita perlu menulis wrapper klien GitHub API khusus dan juga menangani mesin status OAuth yang kompleks.
2.  **Risiko token**: Token GitHub pengguna harus disimpan di database backend. Bahkan dengan enkripsi, hal ini masih meningkatkan permukaan keamanan.
3.  **Biaya pengembangan**: Kami memerlukan migrasi database untuk menyimpan token dan layanan sinkronisasi tambahan untuk memeliharanya.

**model koneksi langsung frontend** jauh lebih ringan. Dalam pendekatan ini, kami menggunakan backend hanya untuk langkah "pertukaran rahasia" yang paling sensitif (callback OAuth). Setelah mendapatkan token, kami menyimpannya langsung di browser `localStorage`. Operasi selanjutnya seperti membuat masalah dan memperbarui komentar dikirim langsung dari frontend ke GitHub melalui HTTP.

| Dimensi Perbandingan | Model Proksi Ujung Belakang | Model Koneksi Langsung Frontend |
| :--- | :--- | :--- |
| **Kompleksitas latar belakang** | Memerlukan layanan OAuth lengkap dan klien API GitHub | Hanya memerlukan titik akhir panggilan balik OAuth |
| **Manajemen token** | Harus dienkripsi dan disimpan dalam database, dengan risiko kebocoran | Disimpan di browser dan hanya terlihat oleh pengguna |
| **Biaya implementasi** | Memerlukan migrasi database dan pengembangan multi-layanan | Terutama pekerjaan frontend |
| **Pengalaman pengguna** | Logikanya terpusat, tetapi latensi server mungkin sedikit lebih tinggi | Respons sangat cepat dengan interaksi GitHub langsung |

Karena kami menginginkan integrasi yang cepat dan perubahan backend yang minimal, **kami akhirnya memilih "model koneksi langsung frontend"**. Ini seperti memberi browser "izin sementara". Setelah mendapat izin, browser dapat menangani sendiri berbagai hal di GitHub tanpa meminta persetujuan administrator backend setiap saat.

## Desain Inti: Aliran Data dan Keamanan

Setelah menentukan arsitekturnya, kami perlu merancang aliran data tertentu. Inti dari proses sinkronisasi adalah bagaimana mendapatkan token dengan aman dan menggunakannya secara efisien.

### Diagram Arsitektur Keseluruhan

Keseluruhan sistem dapat diringkas menjadi tiga peran: browser (frontend), backend HagiCode, dan GitHub.

```text
+--------------+        +--------------+        +--------------+
| Frontend     |        |   Backend    |        |   GitHub     |
| React        |        |   ASP.NET    |        |   REST API   |
|              |        |              |        |              |
|  +--------+  |        |              |        |              |
|  | OAuth  |--+--------> /callback    |        |              |
|  | Flow   |  |        |              |        |              |
|  +--------+  |        |              |        |              |
|              |        |              |        |              |
|  +--------+  |        |  +--------+  |        |  +--------+  |
|  | GitHub |  +------------> Session |  +----------> Issues |  |
|  | API    |  |        |  |Metadata|  |        |  |        |  |
|  | Direct |  |        |  +--------+  |        |  +--------+  |
|  +--------+  |        |              |        |              |
+--------------+        +--------------+        +--------------+
```

**Poin kuncinya adalah**: hanya satu langkah kecil di OAuth (menukar kode dengan token) yang perlu melalui backend. Setelah itu, pekerjaan berat (yang menimbulkan masalah) ditangani langsung antara frontend dan GitHub.

### Sinkronisasi Aliran Data Secara Detail

Saat pengguna mengklik tombol "Sinkronkan ke GitHub" di UI HagiCode, serangkaian tindakan kompleks terjadi:

```text
User clicks "Sync to GitHub"
         │
         ▼
1. Frontend checks localStorage for the GitHub token
         │
         ▼
2. Format issue content (convert the Proposal into Markdown)
         │
         ▼
3. Frontend directly calls the GitHub API to create/update the issue
         │
         ▼
4. Call the HagiCode backend API to update Session.metadata (store issue URL and other info)
         │
         ▼
5. Backend broadcasts the SessionUpdated event via SignalR
         │
         ▼
6. Frontend receives the event and updates the UI to show the "Synced" state
```

### Desain Keamanan

Keamanan selalu menjadi prioritas utama saat mengintegrasikan layanan pihak ketiga. Kami membuat pertimbangan sebagai berikut:

1.  **Pertahankan dari serangan CSRF**: Hasilkan secara acak `state` parameter selama pengalihan OAuth dan simpan di `sessionStorage`. Validasi secara ketat status dalam panggilan balik untuk mencegah permintaan palsu.
2.  **Penyimpanan token terisolasi**: Token hanya disimpan di browser `localStorage`. Dengan menggunakan Kebijakan Same-Origin, hanya skrip HagiCode yang dapat membacanya, sehingga menghindari risiko kebocoran database sisi server yang memengaruhi pengguna.
3.  **Batas kesalahan**: Kami merancang penanganan khusus untuk kesalahan umum GitHub API (seperti token 401 yang kedaluwarsa, kegagalan validasi 422, dan pembatasan laju 429), sehingga pengguna menerima masukan yang ramah.

### 2. Alur OAuth Frontend

Ini adalah titik masuk koneksi. Kami menggunakan Alur Kode Otorisasi standar.

```typescript
// src/HagiCode.Client/src/services/githubOAuth.ts

// Generate the authorization URL and redirect
export async function generateAuthUrl(): Promise<string> {
  const state = generateRandomString(); // Generate a random string for CSRF protection
  sessionStorage.setItem('hagicode_github_state', state);
  
  const params = new URLSearchParams({
    client_id: clientId,
    redirect_uri: window.location.origin + '/settings?tab=github&oauth=callback',
    scope: ['repo', 'public_repo'].join(' '),
    state: state,
  });
  
  return `https://github.com/login/oauth/authorize?${params.toString()}`;
}

// Handle the code-to-token exchange on the callback page
export async function exchangeCodeForToken(code: string, state: string): Promise<GitHubToken> {
  // 1. Validate state to prevent CSRF
  const savedState = sessionStorage.getItem('hagicode_github_state');
  if (state !== savedState) throw new Error('Invalid state parameter');

  // 2. Call the backend API to exchange the token
  // Note: this must go through the backend because ClientSecret cannot be exposed to the frontend
  const response = await fetch('/api/GitHubOAuth/callback', {
    method: 'POST',
    headers: { 'Content-Type': 'application/json' },
    body: JSON.stringify({ code, state, redirectUri: window.location.origin + '/settings?tab=github&oauth=callback' }),
  });

  if (!response.ok) throw new Error('Failed to exchange token');
  
  const token = await response.json();
  
  // 3. Save into LocalStorage
  saveToken(token);
  return token;
}
```

### 3. Pembungkus Klien API GitHub

Setelah kita memiliki tokennya, kita memerlukan alat yang solid untuk memanggil API GitHub.

```typescript
// src/HagiCode.Client/src/services/githubApiClient.ts

const GITHUB_API_BASE = 'https://api.github.com';

// Core request wrapper
async function githubApi<T>(endpoint: string, options: RequestInit = {}): Promise<T> {
  const token = localStorage.getItem('gh_token');
  if (!token) throw new Error('Not connected to GitHub');
  
  const response = await fetch(`${GITHUB_API_BASE}${endpoint}`, {
    ...options,
    headers: {
      ...options.headers,
      Authorization: `Bearer ${token}`,
      Accept: 'application/vnd.github.v3+json', // Specify the API version
    },
  });
  
  // Error handling logic
  if (!response.ok) {
    if (response.status === 401) throw new Error('GitHub token expired, please reconnect');
    if (response.status === 403) throw new Error('No permission to access this repository or rate limit exceeded');
    if (response.status === 422) throw new Error('Issue validation failed, the title may be duplicated');
    throw new Error(`GitHub API Error: ${response.statusText}`);
  }
  
  return response.json();
}

// Create issue
export async function createIssue(owner: string, repo: string, data: { title: string, body: string, labels: string[] }) {
  return githubApi(`/repos/${owner}/${repo}/issues`, {
    method: 'POST',
    body: JSON.stringify(data),
  });
}
```

### 4. Pemformatan dan Sinkronisasi Konten

Langkah terakhir adalah mengonversi data sesi HagiCode ke dalam format masalah GitHub. Ini seperti pekerjaan penerjemahan.

```typescript
// Convert a Session object into a Markdown string
function formatIssueForSession(session: Session): string {
  let content = `# ${session.title}\n\n`;
  content += `**> HagiCode Session:** #${session.code}\n`;
  content += `**> Status:** ${session.status}\n\n`;
  content += `## Description\n\n${session.description || 'No description provided.'}\n\n`;
  
  // If this is a Proposal session, add extra fields
  if (session.type === 'proposal') {
    content += `## Chief Complaint\n\n${session.chiefComplaint || ''}\n\n`;
    // Add a deep link so users can jump back from GitHub to HagiCode
    content += `---\n\n**[View in HagiCode](hagicode://sessions/${session.id})**\n`;
  }
  
  return content;
}

// Main logic when clicking the sync button
const handleSync = async (session: Session) => {
  try {
    const repoInfo = parseRepositoryFromUrl(session.repoUrl); // Parse the repository URL
    if (!repoInfo) throw new Error('Invalid repository URL');

    toast.loading('Syncing to GitHub...');
    
    // 1. Format content
    const issueBody = formatIssueForSession(session);
    
    // 2. Call API
    const issue = await githubApiClient.createIssue(repoInfo.owner, repoInfo.repo, {
      title: `[HagiCode] ${session.title}`,
      body: issueBody,
      labels: ['hagicode', 'proposal', `status:${session.status}`],
    });
    
    // 3. Update Session Metadata (save the issue link)
    await SessionsService.patchApiSessionsSessionId(session.id, {
      metadata: {
        githubIssue: {
          owner: repoInfo.owner,
          repo: repoInfo.repo,
          issueNumber: issue.number,
          issueUrl: issue.html_url,
          syncedAt: new Date().toISOString(),
        }
      }
    });

    toast.success('Synced successfully!');
  } catch (err) {
    console.error(err);
    toast.error('Sync failed, please check your token or network');
  }
};
```

## Ringkasan dan Pandangan

Dengan pendekatan "koneksi langsung frontend" ini, kami mencapai integrasi Masalah GitHub yang lancar dengan kode backend sesedikit mungkin.

### Keuntungan utama
1.  **Efisiensi pengembangan tinggi**: Perubahan backend minimal, terutama satu kolom database tambahan dan titik akhir panggilan balik OAuth sederhana. Sebagian besar logika diselesaikan di frontend.
2.  **Keamanan yang kuat**: Token tidak melewati database server, sehingga mengurangi risiko kebocoran.
3.  **Pengalaman pengguna yang luar biasa**: Permintaan dimulai langsung dari frontend, sehingga kecepatan responsnya cepat dan tidak perlu penerusan backend.

### Hal-hal yang harus diperhatikan
Ada beberapa kendala yang perlu diingat selama penerapan sebenarnya:
- **Pengaturan Aplikasi OAuth**: Ingatlah untuk memasukkan yang benar `Authorization callback URL` di pengaturan Aplikasi GitHub OAuth Anda (biasanya `http://localhost:3000/settings?tab=github&oauth=callback`).
- **Batas kapasitas**: API GitHub membatasi permintaan yang tidak diautentikasi dengan cukup ketat, namun dengan token, kuota biasanya mencukupi (5000 permintaan/jam).
- **Penguraian URL**: Pengguna memasukkan semua jenis URL repositori, jadi pastikan ekspresi reguler Anda cocok `.git` akhiran, format SSH, dan kasus serupa.

### Peningkatan di masa depan
Fitur saat ini masih sinkronisasi satu arah (HagiCode -> GitHub). Di masa mendatang, kami berencana menerapkan sinkronisasi dua arah melalui GitHub Webhook. Misalnya, jika masalah ditutup di GitHub, status sesi di sisi HagiCode juga dapat diperbarui secara otomatis. Hal ini mengharuskan kita untuk mengekspos titik akhir webhook di backend, yang akan menjadi langkah menarik berikutnya.
