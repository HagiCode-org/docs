## Cara Mengotomatiskan Rilis Steam dengan Tindakan GitHub

> Artikel ini membagikan solusi lengkap yang kami terapkan untuk rilis Steam otomatis di proyek HagiCode Desktop, yang mencakup alur otomatisasi end-to-end dari GitHub Release ke platform Steam, termasuk detail teknis utama seperti autentikasi Steam Guard dan unggahan Depot multi-platform.

## Latar Belakang

Alur kerja rilis di Steam sebenarnya cukup berbeda dengan distribusi aplikasi tradisional. Steam memiliki sistem pengiriman pembaruan lengkapnya sendiri. Pengembang perlu menggunakan SteamCMD untuk mengunggah artefak bangunan ke jaringan CDN Steam, daripada hanya memasukkan tautan unduhan seperti pada platform lain.

Proyek HagiCode Desktop sedang mempersiapkan rilis Steam, yang memperkenalkan beberapa tantangan baru pada alur kerja rilis kami:

1. Kami perlu mengonversi artefak bangunan yang ada ke dalam format yang kompatibel dengan Steam.
2. Kami harus mengunggahnya ke Steam melalui SteamCMD.
3. Kami juga harus menangani otentikasi Steam Guard.
4. Kami perlu mendukung unggahan Depot multi-platform untuk Linux, Windows, dan macOS.
5. Kami menginginkan aliran yang sepenuhnya otomatis dari GitHub Release ke Steam.

Proyek ini telah menerapkan "mode versi portabel", yang memungkinkan aplikasi mendeteksi muatan layanan tetap yang dikemas dalam `extra` direktori. Tujuan kami adalah mengintegrasikan mode versi portabel tersebut secara mulus dengan distribusi Steam.

## Tentang HagiCode

Solusi yang dibagikan dalam artikel ini berasal dari pengalaman praktis kami di [Kode Hagi](https://hagicode.com) proyek. HagiCode adalah asisten pengkodean AI yang mendukung penggunaan desktop. Karena kami secara aktif berupaya meluncurkannya di Steam, kami perlu membuat alur kerja rilis otomatis yang andal.

## Desain Arsitektur

Inti dari keseluruhan proses rilis Steam adalah alur kerja GitHub Actions yang membagi proses menjadi tiga tahap utama:

```
┌─────────────────────────────────────────────────────────────┐
│ GitHub Actions Workflow (Steam Release)                    │
├─────────────────────────────────────────────────────────────┤
│ 1. Preparation Stage:                                      │
│    - Check out portable-version code                       │
│    - Download build artifacts from GitHub Release          │
│    - Extract and prepare the Steam content directory       │
│                                                             │
│ 2. SteamCMD Setup:                                         │
│    - Install or reuse SteamCMD                             │
│    - Authenticate with Steam Guard                         │
│                                                             │
│ 3. Release Stage:                                          │
│    - Generate Depot VDF configuration files                │
│    - Generate App Build VDF configuration files            │
│    - Invoke SteamCMD to upload to Steam                    │
└─────────────────────────────────────────────────────────────┘
```

Desain ini menawarkan beberapa keunggulan:
- Ini menggunakan kembali artefak Rilis GitHub yang ada dan menghindari pembuatan kembali keluaran yang sama.
- Ia menggunakan runner yang dihosting sendiri untuk isolasi yang aman.
- Ini mendukung peralihan antara mode pratinjau dan cabang rilis formal.
- Ini mencakup penanganan kesalahan dan pencatatan log yang lengkap, yang membuat kegagalan lebih mudah didiagnosis.

## Implementasi Alur Kerja

### Desain Parameter Pemicu

Alur kerja kami mendukung parameter utama berikut:

```yaml
inputs:
  release:           # Portable Version release tag
    description: 'Version tag to release (for example v1.0.0)'
    required: true
  steam_preview:     # Whether to generate a preview build
    description: 'Whether to use preview mode'
    required: false
    default: 'false'
  steam_branch:      # Steam branch to set live
    description: 'Target Steam branch'
    required: false
    default: 'preview'
  steam_description: # Build description override
    description: 'Build description'
    required: false
```

### Konfigurasi Pelari yang dihosting sendiri

Untuk alasan keamanan, kami menggunakan runner yang dihosting sendiri dengan `steam` label:

```yaml
runs-on:
  - self-hosted
  - Linux
  - X64
  - steam
```

Hal ini memastikan bahwa rilis Steam berjalan pada runner khusus dan menjaga kredensial sensitif tetap terisolasi dengan aman.

### Kontrol Konkurensi

Untuk mencegah rilis versi yang sama saling mengganggu, kami mengonfigurasi kontrol konkurensi:

```yaml
concurrency:
  group: portable-version-steam-${{ github.event.inputs.release }}
  cancel-in-progress: false
```

Perhatikan itu `cancel-in-progress: false` diatur di sini karena rilis Steam dapat memakan waktu cukup lama, dan kami tidak ingin proses yang baru dipicu membatalkan proses yang sudah diunggah.

## Implementasi Skrip Inti

### Mempersiapkan Input Rilis

Itu `prepare-steam-release-input.mjs` skrip bertanggung jawab untuk menyiapkan masukan yang diperlukan untuk rilis:

```javascript
// Download the GitHub Release build manifest and artifact inventory
const buildManifest = await downloadBuildManifest(releaseTag);
const artifactInventory = await downloadArtifactInventory(releaseTag);

// Download compressed archives for each platform
for (const platform of ['linux-x64', 'win-x64', 'osx-universal']) {
  const artifactUrl = getArtifactUrl(artifactInventory, platform);
  await downloadArtifact(artifactUrl, platform);
}

// Extract into the Steam content directory structure
await extractToSteamContent(sources, contentRoot);
```

### Otentikasi Penjaga Uap

Steam mengharuskan akun dilindungi dengan Steam Guard, jadi kami menerapkan algoritma pembuatan kode berbasis rahasia bersama:

```javascript
function generateSteamGuardCode(sharedSecret, timestamp = Date.now()) {
  const secret = decodeSharedSecret(sharedSecret);
  const time = Math.floor(timestamp / 1000 / 30);

  const timeBuffer = Buffer.alloc(8);
  timeBuffer.writeBigUInt64BE(BigInt(time));

  // Use HMAC-SHA1 to generate a time-based one-time code
  const hash = crypto.createHmac('sha1', secret)
    .update(timeBuffer)
    .digest();

  // Convert it into a 5-character Steam Guard code
  const code = steamGuardCode(hash);
  return code;
}
```

Implementasi ini didasarkan pada mekanisme TOTP (Time-based One-Time Password) Steam Guard, yang menghasilkan kode verifikasi baru setiap 30 detik.

### Pembuatan Konfigurasi VDF

VDF (Valve Data Format) adalah format konfigurasi yang digunakan oleh Steam. Kita perlu membuat dua jenis file VDF:

**Depot VDF** digunakan untuk mengonfigurasi konten untuk setiap platform:

```javascript
function buildDepotVdf(depotId, contentRoot) {
  return [
    '"DepotBuildConfig"',
    '{',
    `  "DepotID" "${escapeVdf(depotId)}"`,
    `  "ContentRoot" "${escapeVdf(contentRoot)}"`,
    '  "FileMapping"',
    '  {',
    '    "LocalPath" "*"',
    '    "DepotPath" "."',
    '    "recursive" "1"',
    '  }',
    '}'
  ].join('\n');
}
```

**App Build VDF** digunakan untuk mengonfigurasi seluruh build aplikasi:

```javascript
function buildAppBuildVdf(appId, depotBuilds, description, setLive) {
  const vdf = [
    '"appbuild"',
    '{',
    `  "appid" "${appId}"`,
    `  "desc" "${escapeVdf(description)}"`,
    `  "contentroot" "${escapeVdf(contentRoot)}"`,
    '  "buildoutput" "build_output"',
    '  "depots"',
    '  {'
  ];

  for (const [depotId, depotVdfPath] of Object.entries(depotBuilds)) {
    vdf.push(`    "${depotId}" "${depotVdfPath}"`);
  }

  if (setLive) {
    vdf.push(`  }`);
    vdf.push(`  "setlive" "${setLive}"`);
  }

  vdf.push('}');
  return vdf.join('\n');
}
```

### Doa SteamCMD

Terakhir, pengunggahan dilakukan dengan menjalankan SteamCMD:

```javascript
await runCommand(steamcmdPath, [
  '+login', steamUsername, steamPassword, steamGuardCode,
  '+run_app_build', appBuildPath,
  '+quit'
]);
```

Ini adalah lompatan terakhir dalam keseluruhan alur kerja. Setelah berhasil, rilis selesai.

## Penanganan Depot Multi-platform

Steam menggunakan sistem Depot untuk mengelola konten untuk berbagai platform. Kami mendukung tiga Depot utama:

| Peron | Pengenal Depo | Dukungan Arsitektur |
|-----|-----------|---------|
| Linux | `linux-x64` | x64_64 |
| jendela | `win-x64` | x64_64 |
| macOS | `osx-universal` | universal, x64_64, arm64 |

Setiap Depot memiliki direktori konten dan file konfigurasi VDF sendiri. Hal ini memastikan bahwa pengguna di platform berbeda hanya mengunduh konten yang benar-benar mereka perlukan.

## Rilis Alur Kerja

### Langkah 1: Siapkan Rilis GitHub

Pertama, buat Rilis GitHub di `portable-version` repositori yang meliputi:
- Arsip terkompresi untuk setiap platform
- Bangun manifes (`{tag}.build-manifest.json`)
- Inventaris artefak (`{tag}.artifact-inventory.json`)

### Langkah 2: Memicu Alur Kerja Rilis Steam

Picu alur kerja secara manual melalui GitHub Actions dan isi parameter yang diperlukan:
- `release`: tag versi yang akan dipublikasikan, seperti `v1.0.0`
- `steam_branch`: cabang target, seperti `preview` atau `public`
- `steam_preview`: apakah akan menggunakan mode pratinjau

### Langkah 3: Jalankan Aliran Rilis Secara Otomatis

Alur kerja secara otomatis melakukan langkah-langkah berikut:

1. Unduh dan ekstrak artefak Rilis GitHub.
2. Instal atau perbarui SteamCMD.
3. Hasilkan file konfigurasi Steam VDF.
4. Otentikasi dengan Steam Guard.
5. Unggah konten ke Steam CDN.
6. Aktifkan cabang yang ditentukan.

Setelah urutan ini selesai, seluruh jalur rilis tercakup.

## Panduan Konfigurasi

### Konfigurasi Rahasia yang Diperlukan

Konfigurasikan rahasia berikut dalam pengaturan repositori GitHub:

| Nama Rahasia | Deskripsi |
|-----------|------|
| `STEAM_USERNAME` | Nama pengguna akun Steam |
| `STEAM_PASSWORD` | Kata sandi akun uap |
| `STEAM_SHARED_SECRET` | Rahasia bersama Steam Guard (opsional) |
| `STEAM_GUARD_CODE` | Kode Steam Guard (opsional) |
| `STEAM_APP_ID` | ID aplikasi uap |
| `STEAM_DEPOT_ID_LINUX` | ID Depot Linux |
| `STEAM_DEPOT_ID_WINDOWS` | ID Depot Windows |
| `STEAM_DEPOT_ID_MACOS` | ID Depot macOS |

Tidak ada yang aneh dengan pengaturan ini. Anda hanya perlu semua nilai yang diharapkan ada.

### Konfigurasi Variabel Lingkungan

| Nama Variabel | Deskripsi | Nilai Bawaan |
|---------|------|--------|
| `PORTABLE_VERSION_STEAMCMD_ROOT` | Direktori instalasi SteamCMD | `~/.local/share/portable-version/steamcmd` |

## Praktik Terbaik

### Manajemen Otentikasi Steam Guard

Saat pertama kali dijalankan, Anda harus memasukkan kode Steam Guard secara manual. Setelah itu, disarankan untuk mengkonfigurasi rahasia bersama sehingga kode dapat dibuat secara otomatis. Hal ini menghindari intervensi manual pada setiap rilis.

SteamCMD menyimpan token login dan dapat menggunakannya kembali saat dijalankan nanti. Anda tetap perlu memperhatikan masa berlaku token, karena autentikasi ulang diperlukan setelah masa berlaku token habis.

### Struktur Direktori Konten

Pastikan struktur direktori konten Steam sudah benar:

```
steam-content/
├── linux-x64/     # Linux platform content
├── win-x64/       # Windows platform content
└── osx-universal/ # macOS universal binary content
```

Setiap direktori harus berisi file aplikasi lengkap untuk platform terkait.

### Penggunaan Mode Pratinjau

Mode pratinjau tidak mengaktifkan cabang apa pun, sehingga cocok untuk pengujian dan validasi:

```yaml
if [ "$STEAM_PREVIEW_INPUT" = 'true' ]; then
  cmd+=(--preview)
fi
```

Ini memungkinkan Anda mengunggah ke Steam terlebih dahulu untuk verifikasi, kemudian beralih ke cabang formal setelah semuanya diperiksa.

### Penanganan Kesalahan dan Pencatatan

Skripnya mencakup penanganan kesalahan dan pencatatan log yang lengkap:
- Validasi bahwa Rilis GitHub ada.
- Periksa file metadata yang diperlukan.
- Pastikan konten platform ada.
- Buat laporan ringkasan Tindakan GitHub.

Informasi ini sangat berharga untuk debugging dan audit.

### Manajemen Artefak

Alur kerja menghasilkan dua jenis artefak:
- `portable-steam-release-preparation-{tag}`: merilis metadata persiapan
- `portable-steam-build-metadata-{tag}`: Metadata pembuatan Steam

Artefak ini dapat digunakan untuk audit dan debugging nanti. Periode retensi selama 30 hari merupakan standar praktis.

Dalam proyek HagiCode, alur kerja rilis otomatis ini telah berhasil dijalankan di beberapa versi. Seluruh jalur dari Rilis GitHub ke platform Steam sepenuhnya otomatis dan tidak memerlukan intervensi manual.
