## Dari Panggilan CLI hingga Integrasi SDK: Praktik Terbaik untuk GitHub Copilot di Proyek .NET

> Jalur pemutakhiran dari pemanggilan baris perintah ke integrasi SDK resmi merupakan perjalanan yang cukup panjang. Hari ini, saya ingin berbagi kendala yang kami hadapi dan apa yang kami pelajari saat membuat HagiCode.

## Latar Belakang

Setelah GitHub Copilot SDK dirilis secara resmi pada tahun 2025, kami mulai mengintegrasikannya ke dalam lapisan kemampuan AI kami. Sebelumnya, proyek ini terutama menggunakan GitHub Copilot dengan langsung menggunakan alat baris perintah Copilot CLI, namun pendekatan tersebut memiliki beberapa masalah yang jelas:

- **Manajemen proses yang kompleks**: Kami harus mengelola siklus hidup proses CLI, batas waktu startup, dan pembersihan proses secara manual. Proses dapat terhenti tanpa peringatan.
- **Penanganan peristiwa yang tidak lengkap**: Pemanggilan CLI mentah mempersulit pengambilan peristiwa terperinci dari penalaran model dan eksekusi alat. Ibaratnya hanya melihat hasil tanpa proses berpikir.
- **Manajemen sesi yang sulit**: Tidak ada mekanisme efektif untuk penggunaan kembali dan pemulihan sesi, sehingga setiap interaksi harus dimulai dari awal.
- **Masalah kompatibilitas**: Argumen CLI sering berubah, yang berarti kami harus terus-menerus mempertahankan logika kompatibilitas untuk parameter tersebut.

Masalah-masalah ini menjadi semakin jelas dalam perkembangan sehari-hari, terutama ketika kita perlu melacak penalaran model (`thinking`) dan status eksekusi alat secara real time. Pada saat itu, menjadi jelas bahwa kita memerlukan pendekatan integrasi tingkat rendah dan lebih lengkap.

## Tentang HagiCode

Pendekatan yang dibagikan dalam artikel ini berasal dari pengalaman praktis kami di [Kode Hagi](https://hagicode.com) proyek. HagiCode adalah proyek asisten pengkodean AI sumber terbuka, dan selama pengembangan kami memerlukan integrasi mendalam dengan kemampuan GitHub Copilot, mulai dari penyelesaian kode dasar hingga percakapan multi-putaran yang kompleks dan pemanggilan alat. Persyaratan dunia nyata tersebut mendorong kami untuk beralih dari pemanggilan CLI ke SDK resmi.

Jika penerapan ini terdengar berguna bagi Anda, kemungkinan besar pengalaman teknik kami dapat membantu. Dalam hal ini, proyek HagiCode itu sendiri mungkin layak untuk dicoba. Anda bahkan mungkin menemukan lebih banyak informasi proyek dan tautan di akhir artikel ini.

Desain berlapis ini memberikan beberapa keuntungan teknis praktis:

1. **Pemisahan masalah**: Logika bisnis inti dipisahkan dari detail implementasi SDK.

3. **Dapat digunakan kembali**: `HagiCode.Libs` dapat direferensikan oleh banyak proyek.
4. **Kemampuan Pemeliharaan**: Peningkatan SDK hanya memerlukan perubahan pada lapisan Gateway, sedangkan lapisan atas tetap tidak tersentuh.

## Implementasi Inti

### Alur Otentikasi

Otentikasi adalah langkah pertama dan terpenting dalam integrasi SDK. Jika otentikasi gagal, tidak ada hal lain yang penting. Kami merancang konfigurasi autentikasi fleksibel yang mendukung berbagai sumber autentikasi:

```csharp
// CopilotProvider.cs - Authentication source configuration
public class CopilotOptions
{
    public bool UseLoggedInUser { get; set; } = true;
    public string? GitHubToken { get; set; }
    public string? CliUrl { get; set; }
}

// Convert to SDK request
return new CopilotSdkRequest(
    GitHubToken: options.AuthSource == CopilotAuthSource.GitHubToken
        ? options.GitHubToken
        : null,
    UseLoggedInUser: options.AuthSource != CopilotAuthSource.GitHubToken
);
```

Manfaat dari desain ini cukup jelas:

- Ini mendukung mode pengguna masuk tanpa memerlukan token, yang cocok dengan skenario desktop.
- Ini mendukung mode Token GitHub, yang cocok untuk penerapan sisi server dan manajemen terpusat.
- Ini mendukung penggantian URL Copilot CLI, yang membantu konfigurasi proxy perusahaan.

Dalam praktiknya, model autentikasi fleksibel ini sangat menyederhanakan konfigurasi di berbagai skenario penerapan. Klien desktop dapat menggunakan status login Copilot masing-masing pengguna, sedangkan sisi server dapat mengelola otentikasi secara terpusat melalui token.

### Penanganan Aliran Acara

Salah satu kemampuan paling kuat dari SDK adalah penangkapan aliran peristiwa secara lengkap. Kami menerapkan sistem pengiriman peristiwa yang dapat menangani semua jenis peristiwa SDK secara real time:

```csharp
// GitHubCopilotSdkGateway.cs - Core logic for event dispatch
internal static SessionEventDispatchResult DispatchSessionEvent(
    SessionEvent evt, bool sawDelta)
{
    switch (evt)
    {
        case AssistantReasoningEvent reasoningEvent:
            // Capture the model reasoning process
            events.Add(new CopilotSdkStreamEvent(
                CopilotSdkStreamEventType.ReasoningDelta,
                Content: reasoningEvent.Data.Content));
            break;

        case ToolExecutionStartEvent toolStartEvent:
            // Capture the start of a tool invocation
            events.Add(new CopilotSdkStreamEvent(
                CopilotSdkStreamEventType.ToolExecutionStart,
                ToolName: toolStartEvent.Data.ToolName,
                ToolCallId: toolStartEvent.Data.ToolCallId));
            break;

        case ToolExecutionCompleteEvent toolCompleteEvent:
            // Capture tool completion and its result
            events.Add(new CopilotSdkStreamEvent(
                CopilotSdkStreamEventType.ToolExecutionEnd,
                Content: ExtractToolExecutionContent(toolCompleteEvent)));
            break;

        default:
            // Preserve unhandled events as RawEvent
            events.Add(new CopilotSdkStreamEvent(
                CopilotSdkStreamEventType.RawEvent,
                RawEventType: evt.GetType().Name));
            break;
    }
}
```

Nilai dari implementasi ini sangat penting:

- **Rekaman lengkap proses penalaran model** (`thinking`): pengguna dapat melihat bagaimana AI berpikir, bukan hanya jawaban akhir.
- **Pelacakan status eksekusi alat secara real-time**: kami mengetahui alat mana yang sedang berjalan, kapan selesai, dan apa yang dikembalikan.
- **Kerugian peristiwa nol**: dengan melakukan fall back ke `RawEvent`, kami memastikan setiap peristiwa dicatat.

Di HagiCode, peristiwa terperinci ini membantu pengguna memahami cara kerja AI secara internal, terutama saat melakukan debug pada tugas-tugas kompleks.

### Penanganan Kompatibilitas CLI

Setelah bermigrasi dari pemanggilan CLI ke SDK, kami menemukan bahwa beberapa parameter CLI yang ada tidak lagi diterapkan di SDK. Untuk menjaga kompatibilitas ke belakang, kami menerapkan sistem pemfilteran parameter:

```csharp
// CopilotCliCompatibility.cs - Argument filtering
private static readonly Dictionary<string, string> RejectedFlags = new()
{
    ["--headless"] = "Unsupported startup argument",
    ["--model"] = "Passed through an SDK-native field",
    ["--prompt"] = "Passed through an SDK-native field",
    ["--interactive"] = "Interaction is managed by the provider",
};

public static CopilotCliArgumentBuildResult BuildCliArgs(CopilotOptions options)
{
    // Filter out unsupported arguments and keep compatible ones
    // Generate diagnostic information
}
```

Ini memberi kita beberapa manfaat:

- Secara otomatis memfilter argumen CLI yang tidak kompatibel untuk mencegah kesalahan runtime.
- Ini menghasilkan pesan diagnostik yang jelas untuk membantu pengembang menemukan masalah dengan cepat.
- Itu membuat SDK tetap stabil dan terisolasi dari perubahan parameter CLI.

Selama peningkatan, mekanisme kompatibilitas ini membantu kami melakukan transisi dengan lancar. File konfigurasi yang ada masih dapat digunakan dan hanya memerlukan penyesuaian bertahap berdasarkan keluaran diagnostik.

### Penggabungan Waktu Proses

Membuat sesi Copilot SDK relatif mahal, dan membuat serta menghancurkan sesi terlalu sering akan mengganggu kinerja. Untuk mengatasinya, kami menerapkan sistem manajemen kumpulan sesi:

```csharp
// CopilotProvider.cs - Session pool management
await using var lease = await _poolCoordinator.AcquireCopilotRuntimeAsync(
    request,
    async ct => await _gateway.CreateRuntimeAsync(sdkRequest, ct),
    cancellationToken);

if (lease.IsWarmLease)
{
    // Reuse an existing session
    yield return CreateSessionReusedMessage();
}

await foreach (var eventData in lease.Entry.Resource.SendPromptAsync(...))
{
    yield return MapEvent(eventData);
}
```

Manfaat pengumpulan sesi meliputi:

- **Sesi penggunaan kembali**: permintaan dengan hal yang sama `sessionId` dapat menggunakan kembali sesi yang ada dan mengurangi overhead startup.
- **Dukungan pemulihan sesi**: setelah gangguan jaringan, status sesi sebelumnya dapat dipulihkan.
- **Manajemen pengumpulan otomatis**: sesi yang kedaluwarsa dibersihkan secara otomatis untuk menghindari kebocoran sumber daya.

Di HagiCode, pengumpulan sesi secara nyata meningkatkan respons, terutama untuk percakapan berkelanjutan.

### Integrasi Orleans

HagiCode menggunakan Orleans sebagai kerangka terdistribusinya, dan kami mengintegrasikan Copilot SDK ke dalam Orleans Grains:

```csharp
// GitHubCopilotGrain.cs - Distributed execution
public async IAsyncEnumerable<GitHubCopilotResponse> ExecuteCommandStreamAsync(
    string command,
    CancellationToken token = default)
{
    var provider = await aiProviderFactory.GetProviderAsync(AIProviderType.GitHubCopilot);

    await foreach (var chunk in provider.SendMessageAsync(request, null, token))
    {
        // Map to the unified response format
        yield return BuildChunkResponse(chunk, startedAt);
    }
}
```

Keuntungan integrasi Orleans sangat besar:

- **Abstraksi Penyedia AI Terpadu**: beralih antar penyedia AI yang berbeda menjadi mudah.
- **Isolasi multi-penyewa**: Sesi kopilot untuk pengguna yang berbeda tetap terisolasi satu sama lain.
- **Status sesi persisten**: status sesi dapat dipulihkan bahkan setelah server dimulai ulang.

Untuk skenario yang perlu menangani sejumlah besar permintaan bersamaan, Orleans memberikan skalabilitas yang kuat.

## Panduan Praktis

### Contoh Konfigurasi

Berikut contoh konfigurasi lengkapnya:

```json
{
  "AI": {
    "Providers": {
      "Providers": {
        "GitHubCopilot": {
          "Enabled": true,
          "ExecutablePath": "copilot",
          "Model": "gpt-5",
          "WorkingDirectory": "/path/to/project",
          "Timeout": 7200,
          "StartupTimeout": 30,
          "UseLoggedInUser": true,
          "NoAskUser": true,
          "Permissions": {
            "AllowAllTools": false,
            "AllowedTools": ["Read", "Bash", "Grep"],
            "DeniedTools": ["Edit"]
          }
        }
      }
    }
  }
}
```

### Catatan Penggunaan

Dalam penggunaan di dunia nyata, kami merangkum beberapa poin yang perlu diperhatikan:

**Konfigurasi batas waktu pengaktifan**: Pengaktifan pertama Copilot CLI dapat memerlukan waktu yang relatif lama, jadi sebaiknya lakukan pengaturan `StartupTimeout` hingga setidaknya 30 detik. Jika ini adalah login pertama, mungkin memerlukan waktu lebih lama.

**Manajemen izin**: Di lingkungan produksi, hindari penggunaan `AllowAllTools: true`. Gunakan `AllowedTools` daftar yang diizinkan untuk mengontrol alat mana yang tersedia, dan menggunakan `DeniedTools` daftar penolakan untuk memblokir operasi berbahaya. Ini secara efektif mencegah AI menjalankan perintah yang berisiko.

**Manajemen sesi**: Permintaan dengan hal yang sama `sessionId` secara otomatis menggunakan kembali sesi. Status sesi dipertahankan `ProviderSessionId`. Pembatalan disebarkan melalui `CancellationTokenSource`.

**Output diagnostik**: Argumen CLI yang tidak kompatibel menghasilkan pesan bertipe `diagnostic`. Peristiwa SDK mentah dipertahankan sebagai `event.raw`. Pesan kesalahan mencakup kategori seperti waktu tunggu startup dan ketidakcocokan argumen untuk mempermudah pemecahan masalah.

### Praktik Terbaik

Berdasarkan pengalaman praktis kami, berikut beberapa praktik terbaik:

**1. Gunakan daftar alat yang diizinkan**

```csharp
var request = new AIRequest
{
    Prompt = "Analyze this file",
    AllowedTools = new[] { "Read", "Grep", "Bash(git:*)" }
};
```

Menentukan alat yang diizinkan secara eksplisit melalui daftar yang diizinkan membantu mencegah tindakan AI yang tidak terduga. Hal ini sangat penting untuk alat dengan izin menulis, seperti `Edit`, yang harus ditangani dengan ekstra hati-hati.

**2. Tetapkan batas waktu yang wajar**

```csharp
options.Timeout = 3600;  // 1 hour
options.StartupTimeout = 60;  // 1 minute
```

Tetapkan nilai batas waktu yang sesuai berdasarkan kompleksitas tugas. Jika nilainya terlalu pendek, tugas mungkin terganggu. Jika terlalu lama, sumber daya mungkin terbuang percuma karena menunggu permintaan yang tidak responsif.

**3. Aktifkan penggunaan kembali sesi**

```csharp
options.SessionId = "my-session-123";
```

Menggunakan hal yang sama `sessionId` untuk tugas terkait memungkinkan Anda menggunakan kembali konteks sesi sebelumnya dan meningkatkan kecepatan respons.

**4. Menangani respons streaming**

```csharp
await foreach (var chunk in provider.StreamAsync(request))
{
    switch (chunk.Type)
    {
        case StreamingChunkType.ThinkingDelta:
            // Handle the reasoning process
            break;
        case StreamingChunkType.ToolCallDelta:
            // Handle tool invocation
            break;
        case StreamingChunkType.ContentDelta:
            // Handle text output
            break;
    }
}
```

Respons streaming memungkinkan Anda menampilkan kemajuan pemrosesan AI secara real-time, sehingga meningkatkan pengalaman pengguna. Hal ini sangat berharga untuk tugas-tugas jangka panjang.

**5. Penanganan kesalahan dan percobaan ulang**

```csharp
try
{
    await foreach (var chunk in provider.StreamAsync(request))
    {
        // Handle the response
    }
}
catch (CopilotSessionException ex)
{
    // Handle session exceptions
    logger.LogError(ex, "Copilot session failed");
    // Decide whether to retry based on the exception type
}
```

Penanganan kesalahan yang tepat dan logika percobaan ulang meningkatkan stabilitas sistem secara keseluruhan.
