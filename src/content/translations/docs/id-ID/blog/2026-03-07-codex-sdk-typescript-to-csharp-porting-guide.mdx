## Dari TypeScript ke C#: Praktik Porting Lintas Bahasa untuk Codex SDK

> Sederhananya, artikel ini juga merupakan karya kecil kami: artikel ini mencatat proses lengkap porting TypeScript Codex SDK resmi ke C#. Menyebutnya sebagai "port" hampir membuatnya terdengar terlalu mudah - ini lebih seperti sebuah petualangan panjang, karena kedua bahasa ini memiliki kepribadian yang sangat berbeda, dan kami harus menemukan cara untuk membuat keduanya bekerja sama.

## Latar Belakang

Codex adalah alat AI Agent CLI yang dirilis oleh OpenAI, dan ini benar-benar hebat. Tim resmi menyediakan TypeScript SDK di `@openai/codex` paket. Ia berinteraksi dengan Codex CLI dengan memanggil `codex exec --experimental-json` perintah dan parsing aliran acara JSONL.

Masalahnya adalah dalam proyek HagiCode, kita perlu menggunakannya di lingkungan .NET murni - khususnya di layanan backend C# dan aplikasi desktop. Kami tidak dapat memperkenalkan runtime Node.js ke dalam proyek .NET hanya untuk memanggil alat CLI. Itu terlalu rumit.

Jadi kita mempunyai dua pilihan: mempertahankan lapisan jembatan Node.js yang kompleks, atau membangun sendiri C# SDK asli.

Kami memilih yang terakhir.

## Konten Inti

### Perbandingan Desain Arsitektur

Sebelum menerjemahkan kode satu-ke-satu, pertama-tama kita harus memahami desain arsitektur kedua SDK. Anda harus memahami kedua sisi sebelum Anda dapat mem-portingnya dengan baik.

Arsitektur inti dari TypeScript SDK terlihat seperti ini:

```
Codex (entry class)
  └── CodexExec (executor, manages child processes)
      └── Thread (conversation thread)
          ├── run() / runStreamed() (synchronous/asynchronous execution)
          └── event stream parsing
```

C# SDK mempertahankan lapisan arsitektur yang sama, tetapi menyesuaikan detail implementasi. Ide keseluruhannya sederhana: pertahankan konsistensi API sambil memanfaatkan sepenuhnya fitur bahasa C# dalam implementasinya.

### Ketik Konversi Sistem

Ini adalah bagian pekerjaan yang paling mendasar dan juga paling penting. Jika fondasinya lemah, segala sesuatu yang terjadi selanjutnya akan menjadi lebih sulit.

Sistem tipe TypeScript lebih fleksibel daripada C#, dan itu adalah sebuah fakta. Kami perlu menemukan strategi pemetaan yang tepat:

| skrip ketikan | C# | Catatan |
|------------|-----|------|
| `interface` / `type` | `record` | C# menggunakan `record` untuk struktur data yang tidak dapat diubah |
| `tali \| batal` | `string?` | Jenis referensi yang tidak dapat dibatalkan |
| `boolean \| tidak terdefinisi` | `bool?` | Boolean yang tidak dapat dibatalkan |
| `AsyncGenerator` | `IAsyncEnumerable` | Iterator asinkron |

**Sistem tipe peristiwa ** adalah contoh umum. TypeScript menggunakan tipe gabungan untuk mendefinisikan peristiwa:

```typescript
export type ThreadEvent =
  | ThreadStartedEvent
  | TurnStartedEvent
  | TurnCompletedEvent
  | ...
```

Di C#, kami menggunakan hierarki warisan dan pencocokan pola untuk mencapai efek serupa:

```csharp
public abstract record ThreadEvent(string Type);

public sealed record ThreadStartedEvent(string ThreadId) : ThreadEvent("thread.started");
public sealed record TurnStartedEvent() : ThreadEvent("turn.started");
public sealed record TurnCompletedEvent(Usage Usage) : ThreadEvent("turn.completed");
// ...
```

Kami memilih `record` sebagai gantinya `class` karena objek acara harus tidak dapat diubah, yang sesuai dengan maksud di balik penggunaan objek biasa di TypeScript. Itu `sealed` kata kunci juga mencegah pewarisan tambahan dan memberikan ruang kompiler untuk mengoptimalkan.

### Poin Porting Utama

#### 1. Pengurai acara

Penguraian peristiwa adalah inti dari keseluruhan SDK, karena ini menentukan apakah kita dapat memahami dengan benar setiap pesan yang dikembalikan oleh Codex CLI. Jika penguraian salah, semua upaya setelahnya akan sia-sia.

Versi TypeScript menggunakan `JSON.parse()` untuk mengurai setiap baris JSON:

```typescript
export function parseEvent(line: string): ThreadEvent {
  const data = JSON.parse(line);
  // Handle different event types...
}
```

Versi C# menggunakan `System.Text.Json.JsonDocument` sebagai gantinya:

```csharp
public static ThreadEvent Parse(string line)
{
    using var document = JsonDocument.Parse(line);
    var root = document.RootElement;
    var type = GetRequiredString(root, "type", "event.type");

    return type switch
    {
        "thread.started" => new ThreadStartedEvent(GetRequiredString(root, "thread_id", ...)),
        "turn.started" => new TurnStartedEvent(),
        "turn.completed" => new TurnCompletedEvent(ParseUsage(...)),
        // ...
        _ => new UnknownThreadEvent(type, root.Clone()),
    };
}
```

Ada satu trik kecil namun penting di sini: `root.Clone()` diperlukan, karena elemen dari `JsonDocument` menjadi tidak sah setelah dokumen tersebut dibuang. Kita perlu menyimpan salinan untuk jenis acara yang tidak diketahui. Itu hanyalah salah satu perbedaan antara penanganan C# JSON dan JavaScript.

#### 2. Perbedaan manajemen proses

Di sinilah kedua SDK tersebut paling berbeda. Node.js dan .NET memiliki konvensi runtime yang berbeda, sehingga implementasinya harus beradaptasi.

TypeScript menggunakan Node.js `spawn()` fungsi:

```typescript
const child = spawn(this.executablePath, commandArgs, { env, signal });
```

C# menggunakan .NET `System.Diagnostics.Process`:

```csharp
using var process = new Process { StartInfo = startInfo };
process.Start();

// stdin/stdout/stderr must be managed manually
```

Lebih khusus lagi, versi C# perlu mengkonfigurasi proses seperti ini:

```csharp
var startInfo = new ProcessStartInfo
{
    FileName = _executablePath,
    RedirectStandardInput = true,
    RedirectStandardOutput = true,
    RedirectStandardError = true,
    UseShellExecute = false,
    CreateNoWindow = true,
};
```

Perbedaan terbesar adalah mekanisme pembatalannya. Penggunaan TypeScript `AbortSignal`, yang merupakan bagian dari Web API dan sangat nyaman untuk digunakan:

```typescript
const child = spawn(cmd, args, { signal: cancellationSignal });
```

C# menggunakan `CancellationToken` sebagai gantinya:

```csharp
public async IAsyncEnumerable<string> RunAsync(
    CodexExecArgs args,
    [EnumeratorCancellation] CancellationToken cancellationToken = default)
{
    // Check cancellation status inside the loop
    while (!cancellationToken.IsCancellationRequested)
    {
        // Process output...
    }

    // Terminate the process when cancellation is requested
    if (cancellationToken.IsCancellationRequested)
    {
        try { process.Kill(entireProcessTree: true); } catch { }
    }
}
```

Pada tingkat tinggi, ini hanyalah contoh lain perbedaan antara ekosistem Web API dan ekosistem .NET.

#### 3. Mempertahankan serialisasi konfigurasi

Kedua SDK menerapkan logika yang mengubah konfigurasi JSON menjadi konfigurasi TOML, karena Codex CLI menerima penggantian konfigurasi dalam format TOML. Bagian ini harus tetap konsisten sepenuhnya, jika tidak, konfigurasi yang sama akan berperilaku berbeda di kedua SDK.

Detail seperti itulah yang tidak dapat Anda kompromikan. Keberhasilan atau kegagalan sering kali disebabkan oleh detail seperti ini.

### Detail Implementasi

#### Struktur proyek

Kami membuat struktur proyek berikut:

```
CodexSdk/
├── CodexSdk.csproj
├── Codex.cs           # Entry class
├── CodexThread.cs     # Conversation thread
├── CodexExec.cs       # Executor
├── Events.cs          # Event type definitions
├── Items.cs           # Item type definitions
├── EventParser.cs     # Event parser
├── OutputSchemaTempFile.cs  # Temporary file management
└── ...
```

Ini adalah struktur yang cukup bersih, dan itu banyak membantu selama pelabuhan.

#### Contoh penggunaan

Penggunaan dasar tetap konsisten dengan TypeScript SDK:

```csharp
using CodexSdk;

// Create a Codex instance
var codex = new Codex();
var thread = codex.StartThread();

// Execute a query
var result = await thread.RunAsync("Summarize this repository.");
Console.WriteLine(result.FinalResponse);
```

Penanganan acara streaming memanfaatkan pencocokan pola C#:

```csharp
await foreach (var @event in thread.RunStreamedAsync("Analyze the code."))
{
    switch (@event)
    {
        case ItemCompletedEvent itemCompleted
            when itemCompleted.Item is AgentMessageItem msg:
            Console.WriteLine($"Assistant: {msg.Text}");
            break;
        case TurnCompletedEvent completed:
            Console.WriteLine($"Tokens: in={completed.Usage.InputTokens}");
            break;
        case CommandExecutionItem command:
            Console.WriteLine($"Command: {command.Command}");
            break;
    }
}
```

### Catatan

Selama implementasi, kami mengumpulkan beberapa pelajaran praktis:

1. **Manajemen proses**: Versi C# harus mengelola seluruh siklus hidup proses secara manual, termasuk penghentian proses selama pembatalan. Gunakan `Kill(entireProcessTree: true)` untuk memastikan proses anak juga dibersihkan.

2. **Penanganan kesalahan**: Kami menggunakan `InvalidOperationException` untuk memunculkan kesalahan penguraian, menjaga gaya penanganan kesalahan serupa dengan TypeScript SDK.

3. **Pembersihan sumber daya**: `OutputSchemaTempFile` mengimplementasikan `IAsyncDisposable` untuk memastikan file-file sementara dibersihkan dengan benar.

4. **Variabel lingkungan**: Versi C# mendukung sepenuhnya penggantian variabel lingkungan proses `CodexOptions.Env`. Ini adalah fitur kecil, tapi sangat praktis.

5. **Perbedaan platform**: Versi C# tidak menyertakan logika versi TypeScript untuk secara otomatis menemukan biner di dalam paket npm. Karena proyek .NET biasanya tidak bergantung pada npm, jalur ke `codex` dapat dieksekusi harus ditentukan melalui `CODEX_EXECUTABLE` variabel lingkungan atau `CodexPathOverride`.

## Kesimpulan

Memindahkan SDK TypeScript yang matang ke C# bukan hanya masalah konversi sintaksis - hal ini juga memerlukan pemahaman filosofi desain kedua bahasa. Fleksibilitas TypeScript dan fitur ekosistem JavaScript seperti `AbortSignal` membutuhkan rekanan yang sesuai di C#.

Kesimpulan utamanya adalah: **menjaga konsistensi API lebih penting daripada menjaga konsistensi tingkat implementasi**. Pengguna peduli apakah antarmukanya mudah digunakan, bukan apakah implementasi internalnya identik. Kedengarannya sederhana, namun melakukan trade-off tersebut membutuhkan pertimbangan.

Jika Anda mengerjakan port lintas bahasa yang serupa, pengalaman kami adalah memahami sepenuhnya arsitektur SDK asli terlebih dahulu, lalu menerjemahkannya modul demi modul, dan terakhir menggunakan rangkaian pengujian lengkap untuk memastikan konsistensi perilaku. Pekerjaan seperti ini tidak bisa dilakukan dengan terburu-buru.

Semuanya akan berhasil pada akhirnya.
