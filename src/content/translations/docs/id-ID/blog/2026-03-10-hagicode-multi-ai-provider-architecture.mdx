## Arsitektur Penyedia Multi-AI Praktis di Platform HagiCode

> Artikel ini membagikan pendekatan teknis yang kami gunakan dalam arsitektur Orleans Grain untuk mengintegrasikan dua alat AI, iflow dan OpenCode, melalui integrasi yang terpadu. `IAIProvider` antarmuka, dan membandingkan perbedaan implementasi antara komunikasi WebSocket dan HTTP secara detail.

:::catatan[Konteks Waktu]
Artikel ini mencatat praktik arsitektur penyedia pada saat publikasi. Untuk cakupan yang saat ini didukung secara aktif, lihat [Instalasi CLI Agen AI](/id-ID/related-software-installation/ai-agent-cli/) dan itu [Panduan Instalasi](/id-ID/installation); `IFlowCli` kini dipertahankan hanya sebagai nilai kompatibilitas historis dan bukan merupakan bagian dari cakupan dukungan aktif saat ini.

## Latar Belakang

Tidak ada sesuatu pun yang misterius tentang hal itu. Saat membuat HagiCode, kami menemui masalah praktis: pengguna ingin bekerja dengan alat AI yang berbeda. Hal ini tidak mengherankan, karena setiap orang memiliki kebiasaannya masing-masing. Beberapa lebih menyukai Kode Claude, beberapa menyukai GitHub Copilot, dan beberapa tim menggunakan alat yang mereka kembangkan sendiri.

Solusi awal kami sederhana dan langsung: tulis kode integrasi khusus untuk setiap alat AI. Namun kelemahannya muncul dengan cepat. Basis kode terisi `if-else` cabang, setiap perubahan memerlukan pengujian di banyak tempat, dan setiap alat baru berarti menulis setumpuk logika lagi dari awal.

Belakangan, saya menyadari akan lebih baik jika dibuat terpadu `IAIProvider` antarmuka dan abstrak kemampuan yang dimiliki oleh semua penyedia AI. Dengan begitu, alat apa pun yang digunakan di bawahnya, lapisan atas dapat menyebutnya dengan cara yang sama.

Baru-baru ini, proyek tersebut perlu mengintegrasikan dua alat baru: iflow dan OpenCode. Keduanya mendukung protokol ACP, namun gaya komunikasinya berbeda. iflow menggunakan WebSocket, sedangkan OpenCode menggunakan HTTP API. Hal ini menjadi ujian arsitektur yang berguna: mengadaptasi dua mode transportasi berbeda di balik satu antarmuka terpadu.

## Tentang HagiCode

Pendekatan yang dibagikan dalam artikel ini berasal dari pengalaman praktis kami di [Kode Hagi](https://hagicode.com) proyek. HagiCode adalah platform pengembangan berbantuan AI yang dibangun berdasarkan arsitektur Orleans Grain. Ini terintegrasi dengan penyedia AI yang berbeda melalui kesatuan `IAIProvider` antarmuka, memungkinkan pengguna untuk secara fleksibel memilih alat AI yang mereka sukai.

## Desain Arsitektur

### Abstraksi Antarmuka Terpadu

Pertama, kami mendefinisikan `IAIProvider` antarmuka dan mengabstraksi kemampuan yang perlu diterapkan oleh setiap penyedia AI:

```csharp
public interface IAIProvider
{
    string Name { get; }
    bool SupportsStreaming { get; }
    ProviderCapabilities Capabilities { get; }

    Task<AIResponse> ExecuteAsync(AIRequest request, CancellationToken cancellationToken = default);
    IAsyncEnumerable<AIStreamingChunk> StreamAsync(AIRequest request, CancellationToken cancellationToken = default);
    Task<ProviderTestResult> PingAsync(CancellationToken cancellationToken = default);
    IAsyncEnumerable<AIStreamingChunk> SendMessageAsync(AIRequest request, string? embeddedCommandPrompt = null, CancellationToken cancellationToken = default);
}
```

Antarmuka ini mencakup beberapa metode utama:
- `ExecuteAsync`: menjalankan permintaan AI sekali pakai
- `StreamAsync`: dapatkan tanggapan streaming untuk tampilan waktu nyata
- `PingAsync`: melakukan pemeriksaan kesehatan untuk memverifikasi apakah penyedia tersedia
- `SendMessageAsync`: mengirim pesan dengan dukungan untuk perintah yang disematkan

### IFlowCliProvider: Implementasi Berbasis WebSocket

iflow menggunakan WebSocket untuk komunikasi ACP. Arsitektur keseluruhannya terlihat seperti ini:

```
IFlowCliProvider → ACPSessionManager → WebSocketAcpTransport → iflow CLI
                ↓
         Dynamic port allocation + process management
```

Alur intinya juga cukup mudah:
1. `ACPSessionManager` membuat dan mengelola sesi ACP.
2. `WebSocketAcpTransport` menangani komunikasi WebSocket.
3. Sebuah port dialokasikan secara dinamis, dan proses iflow dimulai `iflow --experimental-acp --port`.
4. `IAIRequestToAcpMapper` dan `IAcpToAIResponseMapper` mengkonversi permintaan dan tanggapan.

Berikut kode intinya:

```csharp
private async IAsyncEnumerable<AIStreamingChunk> StreamCoreAsync(
    AIRequest request,
    string? embeddedCommandPrompt,
    [EnumeratorCancellation] CancellationToken cancellationToken)
{
    // Resolve working directory
    var resolvedWorkingDirectory = ResolveWorkingDirectory(request);
    var effectiveRequest = ApplyEmbeddedCommandPrompt(request, embeddedCommandPrompt);

    // Create ACP session
    await using var session = await _sessionManager.CreateSessionAsync(
        Name,
        resolvedWorkingDirectory,
        cancellationToken,
        request.SessionId);

    // Send prompt
    var prompt = _requestMapper.ToPromptString(effectiveRequest);
    var promptResponse = await session.SendPromptAsync(prompt, cancellationToken);

    // Receive streaming response
    await foreach (var notification in session.ReceiveUpdatesAsync(cancellationToken))
    {
        if (_responseMapper.TryConvertToStreamingChunk(notification, out var chunk))
        {
            if (chunk.Type == StreamingChunkType.Metadata && chunk.IsComplete)
            {
                yield return chunk;
                yield break;
            }
            yield return chunk;
        }
    }
}
```

Ada beberapa poin desain yang patut disebutkan di sini:
- Gunakan `await using` untuk memastikan sesi dirilis dengan benar dan menghindari kebocoran sumber daya.
- Kembalikan tanggapan streaming melalui `IAsyncEnumerable`, yang secara alami mendukung aliran asinkron.
- Gunakan `Metadata` potongan untuk menentukan penyelesaian dan memastikan respons penuh telah diterima.

### OpenCodeCliProvider: Implementasi Berbasis API HTTP

OpenCode menyediakan layanannya melalui HTTP API, sehingga arsitekturnya sedikit berbeda:

```plaintext
OpenCodeCliProvider → OpenCodeRuntimeManager → OpenCodeClient → OpenCode HTTP API
                      ↓
                OpenCodeProcessManager → opencode process management
```

Fitur penting dari OpenCode adalah ia menggunakan database SQLite untuk mempertahankan pengikatan sesi. Hal ini memungkinkan pemulihan sesi dan pemulihan respons cepat:

```csharp
private async Task<OpenCodePromptExecutionResult> ExecutePromptAsync(
    AIRequest request,
    string? embeddedCommandPrompt,
    CancellationToken cancellationToken)
{
    var prompt = BuildPrompt(request, embeddedCommandPrompt);
    var resolvedWorkingDirectory = ResolveWorkingDirectory(request.WorkingDirectory);
    var client = await _runtimeManager.GetClientAsync(resolvedWorkingDirectory, cancellationToken);
    var bindingSessionId = request.SessionId;
    var boundSession = TryGetBinding(bindingSessionId, resolvedWorkingDirectory);

    // Try to use the already bound session
    if (boundSession is not null)
    {
        try
        {
            return await PromptSessionAsync(
                client,
                boundSession,
                BuildPromptRequest(request, prompt, CreatePromptMessageId()),
                request.Model ?? _settings.Model,
                cancellationToken);
        }
        catch (OpenCodeApiException ex) when (IsStaleBinding(ex))
        {
            // The session has expired, remove the binding
            RemoveBinding(bindingSessionId);
        }
    }

    // Create a new session
    var session = await client.Session.CreateAsync(new OpenCodeSessionCreateRequest
    {
        Title = BuildSessionTitle(request)
    }, cancellationToken);

    BindSession(bindingSessionId, session.Id, resolvedWorkingDirectory);
    return await PromptSessionAsync(client, session.Id, ...);
}
```

Implementasi ini memiliki beberapa hal menarik:
- Mekanisme pengikatan sesi: sama `SessionId` menggunakan kembali sesi OpenCode yang sama, menghindari pembuatan sesi berulang.
- Penanganan kedaluwarsa: ketika suatu sesi ditemukan telah kedaluwarsa, pengikatan secara otomatis dibersihkan.
- Persistensi basis data: pengikatan disimpan dalam SQLite dan tetap efektif setelah dimulai ulang.

### Membandingkan Dua Pendekatan

| Aspek | Penyedia IFlowCli | Penyedia OpenCodeCli |
|------|-----------------|---------------------|
| Komunikasi | Soket Web (ACP) | API HTTP |
| Manajemen proses | Manajer Sesi ACPS | OpenCodeProcessManager |
| Alokasi pelabuhan | Pelabuhan dinamis | Tidak ada port (menggunakan HTTP) |
| Manajemen sesi | Sesi ACPS | Sesi OpenCode |
| Kegigihan | Cache dalam memori | basis data SQLite |
| Perintah permulaan | `iflow --experimental-acp --port` | `opencode` |
| Latensi | Lebih rendah (koneksi berumur panjang) | Relatif lebih tinggi (permintaan HTTP) |

Pendekatan mana yang Anda pilih bergantung terutama pada kebutuhan Anda. WebSocket lebih baik untuk skenario dengan persyaratan real-time yang tinggi, sedangkan API HTTP lebih sederhana dan mudah untuk di-debug.

## Panduan Praktis

### Konfigurasikan Penyedia

Pertama, aktifkan kedua penyedia di file konfigurasi:

```yaml
AI:
  Providers:
    IFlowCli:
      Type: "IFlowCli"
      Enabled: true
      ExecutablePath: "iflow"
      Model: null
      WorkingDirectory: null
    OpenCodeCli:
      Type: "OpenCodeCli"
      Enabled: true
      ExecutablePath: "opencode"
      Model: "anthropic/claude-sonnet-4"
      WorkingDirectory: null

OpenCode:
  Enabled: true
  BaseUrl: "http://localhost:38376"
  ExecutablePath: "opencode"
  StartupTimeoutSeconds: 30
  RequestTimeoutSeconds: 120
```

### Gunakan IFlowCliProvider

```csharp
// Get provider through the factory
var provider = await _providerFactory.GetProviderAsync(AIProviderType.IFlowCli);

// Execute an AI request
var request = new AIRequest
{
    Prompt = "请帮我重构这个函数",
    WorkingDirectory = "/path/to/project",
    Model = "claude-sonnet-4"
};

// Get the complete response
var response = await provider.ExecuteAsync(request, cancellationToken);
Console.WriteLine(response.Content);

// Or use streaming responses
await foreach (var chunk in provider.StreamAsync(request, cancellationToken))
{
    if (chunk.Type == StreamingChunkType.ContentDelta)
    {
        Console.Write(chunk.Content);
    }
}
```

### Gunakan OpenCodeCliProvider

```csharp
// Get provider through the factory
var provider = await _providerFactory.GetProviderAsync(AIProviderType.OpenCodeCli);

var request = new AIRequest
{
    Prompt = "请帮我分析这个错误",
    WorkingDirectory = "/path/to/project",
    Model = "anthropic/claude-sonnet-4"
};

var response = await provider.ExecuteAsync(request, cancellationToken);
Console.WriteLine(response.Content);
```

### Pemeriksaan Kesehatan

Sebelum memulai atau sebelum digunakan, Anda dapat memeriksa apakah penyedia tersedia:

```csharp
var iflowResult = await iflowProvider.PingAsync(cancellationToken);
if (!iflowResult.Success)
{
    Console.WriteLine($"IFlow is unavailable: {iflowResult.ErrorMessage}");
    return;
}

var openCodeResult = await openCodeProvider.PingAsync(cancellationToken);
if (!openCodeResult.Success)
{
    Console.WriteLine($"OpenCode is unavailable: {openCodeResult.ErrorMessage}");
    return;
}
```

### Dukungan Perintah Tertanam

Kedua penyedia mendukung perintah tertanam, seperti `/file:xxx`:

```csharp
var request = new AIRequest
{
    Prompt = "分析这个文件的问题",
    SystemMessage = "你是一个代码分析专家"
};

await foreach (var chunk in provider.SendMessageAsync(
    request,
    embeddedCommandPrompt: "/file:src/main.cs",
    cancellationToken))
{
    Console.Write(chunk.Content);
}
```

## Catatan dan Praktik Terbaik

### Manajemen Sumber Daya

IFlow menggunakan koneksi WebSocket yang berumur panjang, sehingga manajemen sumber daya perlu mendapat perhatian khusus:
- Gunakan `await using` untuk memastikan sesi dirilis dengan benar.
- Pembatalan memicu pembersihan proses.
- `ACPSessionManager` mendukung batas jumlah sesi maksimum.

Manajemen proses OpenCode relatif lebih sederhana, dan `OpenCodeRuntimeManager` menanganinya secara otomatis.

### Penanganan Kesalahan

Kedua penyedia memiliki penanganan kesalahan yang lengkap:
- Kesalahan IFlow disebarkan melalui pembaruan sesi ACP.
- Kesalahan OpenCode terjadi `OpenCodeApiException`.
- Disarankan agar penelepon menangkap dan menangani pengecualian ini.

### Pertimbangan Kinerja

- Komunikasi IFlow WebSocket memiliki latensi lebih rendah dibandingkan HTTP.
- Penggunaan kembali sesi OpenCode dapat mengurangi overhead permintaan HTTP.
- Mekanisme cache pabrik menghindari pembuatan penyedia berulang kali.
- Dalam skenario konkurensi tinggi, perhatikan baik-baik batasan jumlah proses dan jumlah koneksi.

### Validasi Konfigurasi

Jalur yang dapat dieksekusi divalidasi saat startup, namun masalah runtime masih bisa terjadi. `PingAsync` adalah alat yang berguna untuk memverifikasi apakah konfigurasi sudah benar:

```csharp
// Check at startup
var provider = await _providerFactory.GetProviderAsync(providerType);
var result = await provider.PingAsync(cancellationToken);
if (!result.Success)
{
    _logger.LogError("Provider {ProviderType} is unavailable: {Error}", providerType, result.ErrorMessage);
}
```

## Ringkasan

Artikel ini membagikan pendekatan teknis yang digunakan oleh platform HagiCode saat mengintegrasikan dua alat AI iflow dan OpenCode. Melalui kesatuan `IAIProvider` antarmuka, kami mengadaptasi gaya komunikasi yang berbeda, WebSocket dan HTTP, sekaligus menjaga pola panggilan lapisan atas tetap konsisten.

Ide intinya sebenarnya cukup sederhana:
1. Tentukan abstraksi antarmuka terpadu.
2. Bangun lapisan adaptor untuk implementasi yang berbeda.
3. Kelola semuanya secara seragam melalui pola pabrik.

Hal ini memberikan sistem fleksibilitas yang baik. Ketika alat AI baru perlu diintegrasikan nanti, yang perlu kita lakukan hanyalah mengimplementasikannya `IAIProvider` antarmuka tanpa mengubah terlalu banyak kode yang ada.

Jika Anda juga sedang mengerjakan integrasi multi-AI-tool, saya harap artikel ini bermanfaat.
