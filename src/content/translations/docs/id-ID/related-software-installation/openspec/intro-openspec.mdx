# Pengenalan OpenSpec

OpenSpec adalah alat pengembangan berbasis spesifikasi yang membantu tim mengelola proses pengembangan perangkat lunak melalui proposal dan spesifikasi terstruktur. Artikel ini memberikan pengenalan komprehensif tentang konsep inti, alur kerja, dan metode penggunaan OpenSpec.

## Apa itu OpenSpec?

OpenSpec adalah alat baris perintah yang digunakan untuk mengelola proposal (Proposals), spesifikasi (Specifications), dan perubahan (Changes) dalam proses pengembangan perangkat lunak. Ini menstandarkan proses pengembangan dan memastikan bahwa setiap fitur melalui desain dan tinjauan yang menyeluruh dari ide hingga implementasi.

### Prinsip Inti

OpenSpec didasarkan pada prinsip-prinsip inti berikut:

- **Spesifikasi terlebih dahulu**: Definisikan spesifikasi yang jelas sebelum menulis kode
- **Kolaborasi terstruktur**: Promosikan kolaborasi tim melalui format dokumen yang terstandarisasi
- **Ketertelusuran**: Catat seluruh proses dari proposal hingga implementasi
- **Jaminan kualitas**: Pastikan kode sesuai dengan maksud desain melalui validasi spesifikasi

### Fitur Utama

OpenSpec menyediakan fitur-fitur utama berikut:

1. **Manajemen proposal**: Buat, tinjau, dan lacak proposal fitur
2. **Definisi spesifikasi**: Tulis dan pertahankan dokumen spesifikasi teknis
3. **Pelacakan perubahan**: Catat seluruh riwayat perubahan yang telah ditinjau
4. **Validasi spesifikasi**: Pastikan perubahan kode sesuai dengan spesifikasi yang telah ditentukan
5. **Kolaborasi tim**: Dukung proses pengembangan dan tinjauan dengan banyak peserta

## Alur Kerja OpenSpec

Alur kerja tipikal OpenSpec mencakup tahap-tahap berikut:

### 1. Membuat Proposal (Proposal)

Ketika ada fitur baru atau persyaratan perubahan signifikan, pertama-tama buat proposal:

```bash
openspec proposal create
```

Proposal mencakup:
- **Deskripsi masalah**: Masalah atau persyaratan yang perlu diselesaikan
- **Solusi**: Ikhtisar solusi yang diusulkan
- **Cakupan dampak**: Dampak pada sistem yang ada
- **Solusi alternatif**: Solusi lain yang dipertimbangkan

### 2. Menulis Spesifikasi (Specification)

Setelah proposal disetujui, tulis spesifikasi teknis yang terperinci:

```bash
openspec spec create
```

Spesifikasi mencakup:
- **Desain teknis**: Rencana implementasi teknis yang terperinci
- **Definisi antarmuka**: Desain antarmuka seperti API, struktur data, dll.
- **Detail implementasi**: Logika implementasi kunci dan algoritma
- **Rencana pengujian**: Cara memverifikasi kebenaran implementasi

### 3. Membuat Perubahan (Change)

Saat memulai implementasi, buat perubahan untuk melacak proses pengembangan:

```bash
openspec change create
```

Perubahan mencakup:
- **Spesifikasi terkait**: Spesifikasi yang diimplementasikan oleh perubahan ini
- **Tugas implementasi**: Daftar tugas pengembangan spesifik
- **Pelacakan kemajuan**: Status penyelesaian tugas
- **Perubahan kode**: Modifikasi kode yang sebenarnya

### 4. Validasi dan Pengarsipan

Setelah pengembangan selesai, validasi apakah perubahan sesuai dengan spesifikasi, lalu arsipkan:

```bash
openspec change validate
openspec change archive
```

## Struktur Direktori OpenSpec

OpenSpec membuat direktori `openspec/` di direktori root proyek dengan struktur berikut:

```
openspec/
├── proposals/           # Direktori proposal
│   ├── PROPOSAL-001.md
│   └── PROPOSAL-002.md
├── specs/              # Direktori spesifikasi
│   ├── SPEC-001.md
│   └── SPEC-002.md
├── changes/            # Direktori perubahan
│   ├── feature-name/
│   │   ├── proposal.md
│   │   ├── design.md
│   │   ├── specs.md
│   │   └── tasks.md
│   └── another-feature/
└── archived/           # Perubahan yang diarsipkan
    └── feature-name/
```

### Penjelasan Direktori

- **proposals/**: Menyimpan semua dokumen proposal, setiap proposal dalam file Markdown terpisah
- **specs/**: Menyimpan dokumen spesifikasi teknis yang sesuai dengan proposal
- **changes/**: Menyimpan perubahan yang sedang berlangsung, setiap perubahan dalam subdirektori
- **archived/**: Menyimpan perubahan yang telah selesai dan diarsipkan

## Perintah Inti

### Perintah Terkait Proposal

```bash
# Membuat proposal baru
openspec proposal create

# Daftar semua proposal
openspec proposal list

# Melihat detail proposal
openspec proposal view <proposal-id>

# Menyetujui proposal
openspec proposal approve <proposal-id>
```

### Perintah Terkait Spesifikasi

```bash
# Membuat spesifikasi baru
openspec spec create

# Daftar semua spesifikasi
openspec spec list

# Melihat detail spesifikasi
openspec spec view <spec-id>

# Memperbarui spesifikasi
openspec spec update <spec-id>
```

### Perintah Terkait Perubahan

```bash
# Membuat perubahan baru
openspec change create

# Daftar semua perubahan
openspec change list

# Melihat detail perubahan
openspec change view <change-name>

# Menambahkan tugas
openspec change add-task <change-name>

# Memvalidasi perubahan
openspec change validate <change-name>

# Mengarsipkan perubahan
openspec change archive <change-name>
```

### Perintah Umum

```bash
# Melihat versi
openspec --version

# Melihat bantuan
openspec --help

# Melihat bantuan untuk perintah tertentu
openspec <command> --help
```

## Menggunakan OpenSpec dalam Proyek HagiCode

### Integrasi Proyek

Dalam proyek HagiCode, OpenSpec terintegrasi ke dalam alur kerja pengembangan:

1. **Pemeriksaan inisialisasi**: Wizard startup memeriksa apakah OpenSpec terinstal
2. **Pengembangan berbasis proposal**: Semua fitur baru harus melalui proses proposal OpenSpec
3. **Tinjauan kode**: Perubahan harus sesuai dengan spesifikasi OpenSpec untuk dapat digabungkan
4. **Sinkronisasi dokumentasi**: Dokumentasi OpenSpec dijaga tetap sinkron dengan implementasi kode

### Persyaratan Versi

Proyek HagiCode memerlukan penggunaan versi **OpenSpec >=1.0.0 <2.0.0**.

Perintah instalasi:

```bash
npm install -g @fission-ai/openspec@1
```

### Contoh Alur Kerja

Berikut adalah contoh lengkap penggunaan OpenSpec dalam proyek HagiCode:

#### Langkah 1: Membuat Proposal

```bash
cd /path/to/hagicode-project
openspec proposal create
```

Masukkan dalam prompt interaktif:
- Judul proposal: Menambahkan fitur autentikasi pengguna
- Deskripsi masalah: Sistem memerlukan login pengguna dan manajemen izin
- Solusi: Implementasi sistem autentikasi berbasis JWT

#### Langkah 2: Menulis Spesifikasi

Setelah proposal disetujui, buat spesifikasi terperinci:

```bash
openspec spec create
```

Definisikan dalam spesifikasi:
- Desain proses autentikasi
- Definisi antarmuka API
- Struktur tabel database
- Pertimbangan keamanan

#### Langkah 3: Membuat Perubahan

Mulai implementasi:

```bash
openspec change create
```

Masukkan nama perubahan: `add-user-authentication`

#### Langkah 4: Menambahkan Tugas

```bash
openspec change add-task add-user-authentication
```

Tambahkan tugas spesifik:
- Implementasi antarmuka pendaftaran pengguna
- Implementasi antarmuka login
- Implementasi JWT middleware
- Menulis unit test

#### Langkah 5: Implementasi dan Validasi

Setelah kode selesai, validasi perubahan:

```bash
openspec change validate add-user-authentication
```

#### Langkah 6: Pengarsipan

Setelah validasi berhasil, arsipkan perubahan:

```bash
openspec change archive add-user-authentication
```

## Praktik Terbaik

### Menulis Proposal

1. **Pernyataan masalah yang jelas**: Definisikan dengan jelas masalah yang perlu diselesaikan, hindari deskripsi yang samar
2. **Pertimbangkan solusi alternatif**: Daftarkan dan bandingkan solusi yang berbeda
3. **Evaluasi dampak**: Analisis dampak dan risiko pada sistem yang ada
4. **Tetap ringkas**: Proposal harus singkat dan jelas; tinggalkan detail untuk spesifikasi

### Menulis Spesifikasi

1. **Terperinci tetapi tidak berlebihan**: Berikan detail yang cukup, tetapi hindari desain berlebihan
2. **Gunakan diagram**: Gunakan diagram alur, diagram arsitektur, dll. untuk membantu penjelasan
3. **Definisikan antarmuka**: Definisikan dengan jelas semua antarmuka publik dan struktur data
4. **Pertimbangkan kasus tepi**: Daftarkan situasi pengecualian dan penanganan kesalahan

### Manajemen Perubahan

1. **Iterasi kecil**: Bagi fitur besar menjadi beberapa perubahan kecil
2. **Perbarui tepat waktu**: Jaga status tugas dan dokumentasi tetap sinkron
3. **Tinjauan kode**: Lakukan tinjauan kode setelah perubahan selesai
4. **Arsipkan tepat waktu**: Arsipkan perubahan yang selesai tepat waktu, jaga direktori kerja tetap bersih

### Kolaborasi Tim

1. **Tinjauan berkala**: Tinjau proposal dan spesifikasi secara berkala untuk memastikan konsensus tim
2. **Dokumentasi terlebih dahulu**: Selesaikan dokumentasi sebelum memulai pengkodean
3. **Jaga komunikasi**: Catat diskusi dan keputusan dalam proposal dan spesifikasi
4. **Kontrol versi**: Sertakan dokumentasi OpenSpec dalam kontrol versi

## Skenario Penggunaan Umum

### Skenario 1: Menambahkan Fitur Baru

1. Buat proposal yang menjelaskan persyaratan fitur baru
2. Tim mendiskusikan dan menyetujui proposal
3. Tulis spesifikasi teknis terperinci
4. Buat perubahan dan implementasikan fitur
5. Validasi dan arsipkan perubahan

### Skenario 2: Refaktorisasi Kode yang Ada

1. Buat proposal yang menjelaskan alasan dan tujuan refaktorisasi
2. Definisikan rencana dan langkah refaktorisasi dalam spesifikasi
3. Buat perubahan dan refaktorisasi secara bertahap
4. Pastikan cakupan pengujian, validasi hasil refaktorisasi
5. Arsipkan perubahan

### Skenario 3: Memperbaiki Bug

Untuk bug sederhana:
- Dapat diperbaiki langsung, tidak memerlukan proses OpenSpec lengkap

Untuk bug kompleks:
1. Buat proposal untuk menganalisis akar penyebab bug
2. Definisikan solusi perbaikan dalam spesifikasi
3. Buat perubahan dan implementasikan perbaikan
4. Tambahkan pengujian untuk mencegah regresi
5. Arsipkan perubahan

### Skenario 4: Manajemen Utang Teknis

1. Buat proposal yang mendaftarkan item utang teknis
2. Definisikan rencana perbaikan dalam spesifikasi
3. Buat perubahan untuk membayar utang teknis secara bertahap
4. Lacak kemajuan dan efek
5. Arsipkan perbaikan yang selesai

## Integrasi dengan Alat Lain

### Integrasi Git

OpenSpec terintegrasi dengan mulus dengan Git:

```bash
# Dokumentasi OpenSpec harus disertakan dalam kontrol versi
git add openspec/
git commit -m "Add OpenSpec proposal for feature X"
```

### Integrasi CI/CD

Validasi spesifikasi OpenSpec dalam proses CI/CD:

```yaml
# .github/workflows/openspec.yml
name: OpenSpec Validation
on: [pull_request]
jobs:
  validate:
    runs-on: ubuntu-latest
    steps:
      - uses: actions/checkout@v2
      - name: Install OpenSpec
        run: npm install -g @fission-ai/openspec@1
      - name: Validate changes
        run: openspec change validate
```

### Integrasi IDE

Disarankan menggunakan editor yang mendukung Markdown untuk mengedit dokumen OpenSpec:
- VS Code + plugin Markdown
- JetBrains IDEs + dukungan Markdown
- Vim/Neovim + plugin Markdown

## Pemecahan Masalah

### Masalah Umum

#### Masalah: Perintah OpenSpec tidak ditemukan

**Solusi**:
1. Konfirmasi bahwa OpenSpec terinstal: `npm list -g @fission-ai/openspec`
2. Periksa apakah variabel lingkungan PATH mencakup direktori bin global npm
3. Instal ulang: `npm install -g @fission-ai/openspec@1`

#### Masalah: Tidak dapat membuat proposal atau perubahan

**Solusi**:
1. Konfirmasi bahwa direktori saat ini adalah direktori root proyek
2. Periksa apakah ada direktori `openspec/`; jika tidak, buat
3. Konfirmasi bahwa Anda memiliki izin menulis

#### Masalah: Validasi gagal

**Solusi**:
1. Periksa apakah dokumentasi dalam direktori perubahan lengkap
2. Konfirmasi bahwa semua tugas yang diperlukan telah selesai
3. Tinjau informasi kesalahan terperinci: `openspec change validate <change-name> --verbose`

#### Masalah: Pengarsipan gagal

**Solusi**:
1. Pastikan perubahan telah lulus validasi
2. Periksa apakah direktori `openspec/archived/` ada
3. Konfirmasi bahwa tidak ada perubahan kode yang belum di-commit

## Penggunaan Lanjutan

### Template Kustom

OpenSpec mendukung template kustom untuk proposal dan spesifikasi:

```bash
# Buat direktori template di direktori root proyek
mkdir -p .openspec/templates

# Buat template proposal kustom
cat > .openspec/templates/proposal.md << 'EOF'
# Proposal: {{title}}

## Problem Statement
{{problem}}

## Proposed Solution
{{solution}}

## Custom Section
Your custom content here
EOF
```

### File Konfigurasi

Buat file konfigurasi `.openspec.json` di direktori root proyek:

```json
{
  "version": "1.0",
  "templates": {
    "proposal": ".openspec/templates/proposal.md",
    "spec": ".openspec/templates/spec.md"
  },
  "validation": {
    "requireTests": true,
    "requireDocs": true
  },
  "hooks": {
    "pre-archive": "npm test"
  }
}
```

### Hooks

OpenSpec mendukung eksekusi skrip kustom sebelum dan sesudah operasi tertentu:

```json
{
  "hooks": {
    "pre-create": "scripts/check-permissions.sh",
    "post-archive": "scripts/notify-team.sh"
  }
}
```
