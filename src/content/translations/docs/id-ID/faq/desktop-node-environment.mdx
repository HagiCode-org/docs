HagiCode Desktop dikirimkan dengan toolchain Node yang dikelola Desktop. Itu tidak mengambil alih instalasi Node global di komputer Anda. Tujuannya adalah untuk memberikan landasan runtime yang stabil dan dapat diverifikasi pada startup Desktop dan alur terkait runtime.

Baca FAQ ini jika Anda melihatnya `portable-fixed/toolchain` dalam diagnostik, perlu tahu yang mana `node` Desktop menggunakan, atau bertanya-tanya mengapa versi Node sistem Anda tidak secara langsung menentukan perilaku runtime Desktop.

## Mengapa Desktop menggabungkan Node.js?

Desktop perlu menjalankan alur pendukung yang sama di seluruh sistem operasi, saluran instalasi, dan lingkungan pengguna. Jika hanya bergantung pada instalasi Node pengguna yang ada, dukungan harus menangani versi yang ketinggalan jaman, npm yang hilang, urutan jalur yang salah, paket global yang tercemar, dan kebijakan registri khusus organisasi sekaligus.

Desktop menjelaskan hal ini secara lebih spesifik sebagai runtime Node/npm yang terisolasi untuk CLI terkelola alih-alih menggunakan kembali lingkungan seluruh sistem. Desain ini memiliki manfaat utama sebagai berikut:

- **Versi yang konsisten**: Desktop menggunakan kontrak runtime Node 22 yang disematkan alih-alih menebak versi utama lokal pengguna.
- **Alur startup terkontrol**: Perintah yang dikelola desktop dapat memilih perintah yang dikemas `node` dan `npm` entri.
- **Hapus batasan konfigurasi**: Konfigurasi terkait CLI disimpan terpisah dari konfigurasi global seluruh host, sehingga pembaruan, peralihan, dan pemulihan cenderung tidak terganggu oleh perbedaan lingkungan sistem.
- **Mengurangi konflik**: Jalur yang dibundel berlaku untuk lingkungan perintah yang dibuat oleh Desktop dan tidak menggantikan Node global pengguna di terminal mereka.
- **Perilaku host non-intrusif**: Isolasi ini menghindari penulisan status rantai alat ke dalam lingkungan Node/npm pengguna yang sudah ada, dan menghapus instalasi Desktop tidak akan mengganggu Node, npm, atau konfigurasi yang sudah ada di mesin.

## Bagaimana Desktop memilih jalur Node

Saat Anda menjalankan Desktop secara normal, Desktop lebih memilih rantai alat Node yang diinstal dengan aplikasi daripada membaca instalasi Node terlebih dahulu dari sistem Anda. Ini membuat versi Node di terminal Anda terpisah dari versi Node yang diharapkan Desktop saat dijalankan.

Ketika rantai alat yang dibundel diaktifkan, lingkungan perintah yang dibuat oleh Desktop menempatkan jalur rantai alat yang dibundel di bagian depan `PATH`. Akibatnya, meskipun sistem Anda memiliki versi Node lain yang terinstal, alur yang dikelola Desktop memutuskan untuk menggunakan Node dan npm yang dibundel Desktop terlebih dahulu.

Ini hanya mempengaruhi lingkungan runtime yang dibuat oleh Desktop. Jika kamu lari `node`, `npm`, atau perintah lain langsung di terminal sistem Anda, perintah tersebut terus menggunakan lingkungan sistem normal dan konfigurasi shell Anda.

Di lingkungan diagnostik atau proses anak yang dimulai oleh Desktop, Anda mungkin melihat variabel seperti ini:

```bash
HAGICODE_PORTABLE_TOOLCHAIN_ROOT=/path/to/portable-fixed/toolchain
```

Variabel ini berarti lingkungan runtime yang dikelola Desktop mengenali akar rantai alat yang dibundel. Melihat jalur ini tidak berarti Desktop mengubah Node sistem Anda; ini hanya berarti Desktop menggunakan lingkungan Node yang dikirimkan bersama aplikasi di dalam cakupan runtime-nya sendiri.

## Tempat Desktop menginstal paket npm yang dikelola

Desktop menyimpan file runtime dan file paket npm yang dapat diubah di lokasi berbeda. Jalur rantai alat yang dibundel seperti `portable-fixed/toolchain/node` adalah sumber yang tidak dapat diubah untuk file yang dikelola Desktop `node` dan `npm` perintah. Paket global terkelola dipasang di bawah direktori data pengguna Electron:

```text
userData/node22/npmGlobal
```

Jika Desktop kemudian berpindah ke Node 24, ia menggunakan pohon paket terpisah seperti `userData/node24/npmGlobal`. Paket yang diinstal untuk Node 22 tetap ada di disk tetapi tidak diperlakukan sebagai diinstal untuk Node 24 hingga paket tersebut diinstal ulang ke awalan Node-major yang baru.

Status ketergantungan desktop hanya memeriksa yang aktif `userData/node<major>/npmGlobal` awalan. File paket di luar awalan tersebut diabaikan oleh manajemen ketergantungan.
