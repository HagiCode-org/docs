當您第一次聽到 **HagiCode** 時，通常會想到幾個問題。

**HagiCode 是人工智慧編碼工具嗎？ **

**HagiCode是遊戲嗎？ **

也許所有人的答案都是肯定的。

HagiCode 從來就不是另一個只能談論程式碼的聊天框。它想做的是將人工智慧引入整個軟體開發過程。您可以使用它來了解儲存庫、編寫提案、分解任務、修改程式碼、組織提交、管理多個儲存庫以及建立可重複使用的知識庫。在同一工作區中，您還可以查看成就、每日報告、效率乘數、代幣吞吐量和主題介面。

因此，如果您確實想要一個簡短的定義，那麼它更接近：

> **HagiCode 是一款將人工智慧編碼工具、遊戲化回饋系統和完整的開發工作區結合到一個平台中的產品。 **

這張截圖已經說明很多了。 HagiCode 不會讓「對話」滯留在頁面中間。它將會話、狀態、工作流程、指標和操作帶入同一工作區。你打開它不僅僅是為了問：「你能為我寫一些程式碼嗎？」打開它即可推進整個開發工作。

## 為什麼 HagiCode 看起來不像傳統的 AI 編碼工具

傳統的人工智慧編碼工具通常專注於生成。 HagiCode 更關心的是減少漂移、可靠地運輸以及事後可審查。

這意味著它的設計傾向於真正的軟體開發工作流程，而不是一次性的問答互動：

- 在更改程式碼之前了解儲存庫
- 執行開始前明確目標
- 在人工智慧行動之前定義邊界
- 不僅保留結果，也保留過程和推理

這也是HagiCode隨後的三個身分的基礎。它是一個人工智慧編碼工具，一個遊戲化的工作空間，一個匯集多種開發能力的平台。

## 1.HagiCode作為AI編碼工具

如果只看「AI編碼」這一層，HagiCode並不是想讓AI寫出更華麗的代碼。它正在努力讓人工智慧的書寫更加可靠。

### 1.它在生成程式碼之前組織工作

HagiCode 內建了 **OpenSpec** 工作流程。對於稍微複雜的事情，AI 不會直接跳到編輯文件。它首先將請求轉化為提案、任務、影響範圍和驗證步驟。

這一點很重要。許多人工智慧編碼工具感到有風險，不是因為它們無法產生程式碼，而是因為當上下文不完整時，它們開始太快改變事物。 HagiCode 試圖扭轉這一局面：

- 先把問題弄清楚
- 確認哪些模組將受到影響
- 分解任務和驗收標準
- 然後進入實施階段

直接結果是人工智慧不太可能在複雜的專案中進行隨機的、直覺驅動的編輯。換句話說，HagiCode 並不追求最短路徑。它正在尋找更可靠的道路。

### 2.強調專案層面的理解，而非僅完成當前的任務

許多 IDE 已經可以編輯多個文件，有些甚至可以在單一會話中更改多個目錄。所以HagiCode的優勢已經不能簡單地概括為「它不是單檔自動完成」。

我想強調的是 HagiCode 的目標是**整個專案的視角**。

它不僅僅關心"此任務需要更改哪些文件"。它也關心更高層次的問題：

- 整個專案正在解決什麼問題
- 目前儲存庫與其他儲存庫如何關聯
- 此變更是否還涉及前端、後端、文件、部署或腳本
- 以前做過哪些類似的決定，為什麼
- 稍後應如何重複使用今天的提案、提交和知識

換句話說，HagiCode 不僅僅是試圖為您完成一項任務。它試圖將人工智慧納入長期參與專案的角度。

從這個角度來看，單一任務只是可見的表面。更重要的是以下能力可以自然連結：

- 跨多個項目的切換和協調
- 跨多個儲存庫建立共識
- 隨著時間的推移保留歷史提案、承諾和知識
- 將個人對話轉化為持久的專案背景

這就是為什麼我將 HagiCode 設計為工作區而不是簡單的聊天視窗。我希望人工智慧看到的不是一個孤立的請求，而是整個專案的進展。

![HagiCode 會話板視圖顯示「待處理」、「進行中」和「已歸檔」列](../img/screenshots/shared/kanban-view/original.png)

從這個角度來看，HagiCode 給人的感覺更像是"一個從整個專案角度思考的 AI"，而不是"一個幫助你完成一次臨時編輯的 AI"。

### 3.支援多種主流Agent CLI，將CLI與模型徹底分離

HagiCode目前的主動支援範圍涵蓋多個主流Agent CLI，包括：

- Codex
- Claude Code
- GitHub Copilot
- OpenCode
- Hermes
- QoderCLI
- Kiro
- Kimi
- Gemini
- DeepAgents
- Codebuddy

我想在這裡澄清一個重要的觀點：**CLI 和模型並不是彼此硬綁定的。 **

許多產品將「您正在使用哪個 CLI」和「您正在使用哪個型號訂閱」視為同一決定。 HagiCode 不想這樣做。

![AI 監控彈出視窗顯示目前代理 CLI 版本、回應時間和運作狀況](../img/screenshots/shared/status-of-agent-clis/original.png)

### 4. OmniRoute將模型層與CLI分離，讓路由更有彈性

HagiCode 整合 **OmniRoute**，使模型存取成為其自己的基礎設施層。這樣，CLI 可以處理您喜歡的互動方式，同時可以透過統一的路由層選擇模型和訂閱。

其價值很簡單：

- 您可以繼續使用您喜歡的 CLI
- 您不會被迫訂閱 CLI 預設的模型
- 您可以在一個統一的層中管理模型選擇、模型目錄和端點訪問
- 多個 CLI 可以重複使用相同的模型路由策略

換句話說，即使您想使用 **Claude Code** 作為 CLI，您仍然可以透過 OmniRoute 將其連接到其他模型來源和訂閱。例如，如果您想使用 **GitHub Copilot** 的訂閱功能，而不是將 CLI 硬綁定到其預設訂閱路徑，則可以在 HagiCode 中使用。

我想要的很簡單：您應該選擇 CLI，因為您喜歡它的使用感覺，並選擇模型或訂閱，因為您信任它的成本、功能和可用性。這些不應該被迫成為單一的捆綁選擇。

## 2.HagiCode作為綜合性AI開發平台

如果第一部分回答"它可以處理編碼嗎？"，那麼本部分回答一個不同的問題：**為什麼它感覺像一個 IDE，並且在某些方面比傳統 IDE 更像一個完整的平台？ **

答案是，HagiCode 不會止於聊天，也不會止於提案。它將通常分散在不同工具中的功能匯集在一起，並將它們轉變為一個連續的系統。

### 1. MonoSpecs 阻止跨儲存庫開發變得拼湊

對於真正的團隊來說，一項需求很少會只出現在一個儲存庫中。前端、後端、文件、腳本和部署配置可能都需要一起變更。

HagiCode 引入 **MonoSpecs** 將此跨儲存庫協作帶回一個視圖下。在單一專案中，您可以維護儲存庫清單、提案範圍和歸檔策略。您還可以讓 AI 更清楚地了解更改真正跨越了哪些邊界。

![MonoSpecs 多重儲存庫狀態概述顯示跨多個儲存庫的分支和變更狀態](../img/screenshots/monospecs/hover-project-show-repository-status/original.png)

對於單一儲存庫使用者來說，這可能不是他們接觸的第一個功能。但是，一旦您開始處理前後端協調、保持文件與產品一致或維護多個子項目，它的價值就變得顯而易見。

### 2.技能係統讓平台不斷成長

許多人工智慧產品的擴展方式很粗糙：要么等待官方發布功能，要么讓用戶自己在終端上修改。 HagiCode 將 **技能** 轉變為正式的產品模組。

在 HagiCode 中，您可以：

- 查看本地已經安裝的技能
- 搜尋技能目錄
- 根據目前專案獲取技能建議
- 檢查技能詳細資訊、安裝命令和信任狀態
- 大量更新本地技能

![HagiCode 技能庫顯示搜尋結果、來源過濾器和安裝計數](../img/screenshots/shared/skills-gallary/original.png)

![HagiCode本機技能頁面顯示已安裝的技能清單和批次操作](../img/screenshots/shared/local-skills-view/original.png)

這意味著 HagiCode 不是密封產品。它更像是一個外殼，可以不斷吸收新的技能、功能和工作流程。

### 3.Vault系統讓知識庫不至於分散各處

您可以將 **Vault** 視為 HagiCode 的知識儲存層。

它支援將不同類型的材料引入平台，包括：

- 程式碼參考儲存庫
- 普通資料夾
- 黑曜石金庫
- 系統管理的目錄

這樣，在一個專案中收集的分析筆記、參考程式碼和設計記錄就不會被困在單一會話中。它們可以再次被引用、進一步組織，並在未來的工作中作為上下文重複使用。

對於許多團隊來說，這非常重要。人工智慧變得真正有價值，不是因為它"回答了一次"，而是因為它可以根據已經組織好的知識體系繼續工作。

![顯示使用者 Vault、系統 Vault 以及開啟代碼伺服器等操作的 Vault 工作區](../img/screenshots/shared/vault-view/original.png)

### 4. AI Compose Commit 將"完成的編碼"擴展為"清晰的提交編寫"

對許多團隊來說，真正的痛點不是編碼本身，而是最後一步：程式碼完成了，但沒有人願意仔細寫提交訊息。

HagiCode 提供 **AI Compose Commit**，它將提交訊息產生也引入到工作流程中。

- 你不需要逐行記住每一個變化
- 您不需要在最後一刻即興創作倉促的提交描述
- 你可以讓AI根據實際的diff組織出更清晰的提交信息
- 在Turbo Engine工作流程中，您也可以自訂AI `Co-Authored-By` 簽名並讓儲存庫層級的配置覆蓋您的全域預設值

![HagiCode 內的 AI Compose Commit 入口點](../img/screenshots/ai-compose-commit/trigger-ai-compose-commit-button/original.png)

它的價值不僅僅是節省幾十秒。正是「提交」最終不再與上下文的其餘部分分離。對團隊來說，這也意味著人工智慧產生的提交可以繼續使用他們已經依賴的機器人名稱、公司電子郵件網域和儲存庫約定，而不是被鎖定到一個固定簽名。

### 5.程式碼伺服器集成，本地、遠端編輯更流暢

HagiCode 也透過**程式碼伺服器**整合了基於瀏覽器的編輯。無論您的專案位於本機、伺服器上、容器中或遠端執行時間中，您都可以更輕鬆地開啟專案或 Vault 並直接進行編輯。

這使得 HagiCode 感覺更像是一個真正的開發平台，而不僅僅是一個分析程式碼的前端介面。很多時候，人工智慧已經將問題追溯到特定文件。如果您仍然需要跳回另一個工具並自行重新定位所有內容，那麼工作流程就會失去動力。代碼伺服器整合解決了這個問題。

![顯示最近啟動和重新開啟入口點的程式碼伺服器執行時間摘要](../img/screenshots/shared/code-server-status/original.png)

除了提案、執行、技能和知識管理之外，HagiCode 還包括許多真正影響日常體驗的功能：

- GitHub 集成
- 語音辨識

- 主題和介面個性化
- 報告和統計入口點

這些可能看起來像"小功能"，但它們決定了一個平台是否是人們希望長期保持開放的平台。 HagiCode 不會在邊緣隱藏它們。它試圖使它們成為產品的可見、完整和可配置的部分。

## 3.HagiCode作為遊戲

HagiCode 內部的遊戲化並不是裝飾。它的存在是為了讓長期使用人工智慧開發平台感覺更靈敏、更有節奏、更容易堅持。

### 1.你可以看到你的進度，而不僅僅是看到聊天記錄

在HagiCode中，許多操作轉化為明確的進度回饋。建立會話、傳送訊息、執行計畫、切換項目和提交註解不再作為一次性操作消失。它們累積成**每日成就**、里程碑進度和完成記錄。

這個設計的重點不僅僅是「有趣」。而是讓你更容易感受到一天中你實際上取得了什麼進展。對許多長期開發者來說，令人疲憊的部分並不是工作量本身。這是缺乏反饋。 HagiCode 正在努力填補這一空白。

![成就大廳顯示每日進度、全球成就和里程碑指標卡](../img/screenshots/shared/achivement-view/original.png)

### 2. 不只停留在成就上。它還為您提供每日報告

除了成就之外，HagiCode 還使用每日報告來顯示您昨天實際完成的工作、積分從何而來以及您的連勝進展如何。

這意味著該平台不僅僅記錄您所做的事情。它將這些動作重新組織成具有實際節奏的審查表面。您可以更輕鬆地判斷您是否在會話進度、工具使用、程式碼執行或只是活動時間和任務連續性方面受到阻礙。

### 3.它也將生產力轉化為可見的回饋

許多產品告訴你"人工智慧讓你更有生產力"，但無法解釋生產力提高了多少。 HagiCode 更願意用可見的數據來表達這一點。

在這些生產力報告中，您可以看到運行時間、AI 花費的時間、效率提升和並發分佈。這並不是神話化人工智慧。它正試圖將「生產力」從口號變成具體的回饋。

### 4.它甚至將代幣的使用變成了你可以即時感受到的東西

如果你是重度用戶，這種設計的價值就變得顯而易見。在許多情況下，人工智慧的成本和效能問題並不會在月底顯現出來。他們在會議正在進行時出現。

HagiCode 直接在產品中顯示輸入令牌、輸出令牌、令牌總數和吞吐量層。這讓您可以更直接地了解會話的實際負載量、當前模型是否處於高負載狀態以及會話是否變得過於臃腫。

### 5.英雄、職業、等級不是噱頭。他們繪製了工作流程

HagiCode 包括一個圍繞英雄、職業、負載和等級進程構建的完整表示層。這不僅僅是表面上的重命名。它將不同的代理、職責和工作狀態映射為更易於理解和管理的介面語言。

這使得多代理協作、角色切換和多模型管理變得不那麼抽象。你看到的不僅僅是"一個配置"，而是"這個英雄現在在做什麼，主副職業是什麼，狀態進展如何"。

## 讓我再回答一次開頭問題

**HagiCode 是人工智慧編碼工具嗎？ **

是的，而且它更強調減少幻覺、避免隨波逐流、產生真正落地的改變。

**HagiCode是遊戲嗎？ **

也是的，因為它在工作空間中認真對待成就、每日報告、乘數、英雄、職業和反饋循環。

**HagiCode 是 IDE 嗎？ **

在某些方面，它更接近一個平台。它不僅僅覆蓋編輯器表面。它將提案、會議、技能、知識庫、跨儲存庫協作、提交組織和基於瀏覽器的編輯連接到一個完整的流程。

所以HagiCode最終想要推廣的不是一個孤立的功能，而是一種新的工作方式：

> **將人工智慧開發從「問一次、回答一次」升級為理解、規劃、執行、知識捕獲和回饋的全鏈條。 **
