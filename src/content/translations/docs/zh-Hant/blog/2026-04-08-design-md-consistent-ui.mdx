## Design.md：讓 AI 一致性進行前端 UI 設計的解決方案

> 在 AI 輔助前端開發時代，如何讓 AI 產生的 UI 保持一致性？本文分享了我們基於 awesome-design-md 建置設計畫廊站點的實踐經驗，以及如何建立結構化的 design.md 來指導 AI 進行規範化的 UI 設計。

## 背景

用過 AI 寫前端程式碼的朋友應該都有過類似的經歷：同一個頁面，讓 AI 多產生幾次，每次的風格都不一樣。有時候是圓角有時候是方角，有時候間距是 8px 有時候又變成 16px，甚至同一個按鈕在不同對話里長得都不一樣。

這不僅僅是個別現象。隨着 AI 輔助開發的普及，**AI 產生的前端 UI 缺乏一致性**已經成爲一個普遍問題。不同的 AI 助手、不同的提示詞，甚至同一助手在不同對話中，都會產生風格迥異的界面設計。這給產品疊代帶來了巨大的維護成本。

問題的根源其實很簡單：缺少一份**權威的設計參考文件**。傳統的 CSS 樣式檔案只能告訴開發人員"怎麼實作"，卻無法完整傳達"爲什麼這樣設計"以及"在什麼場景下使用什麼設計模式"。而對於 AI 來說，它更需要一個清晰的結構化描述來理解設計規範。

與此同時，開放原始碼社區已經有了一些很好的資源。[VoltAgent/awesome-design-md](https://github.com/VoltAgent/awesome-design-md) 專案收集了大量知名公司的設計系統文件，每個目錄包含 README.md、DESIGN.md 和預覽 HTML。但這些都分散在上游存放庫中，難以快速查閱和比較。

那能不能把這些資源整合起來，做成一個方便查閱的設計畫廊，同時沉澱出一份結構化的 design.md 給 AI 用呢？

答案是肯定的。接下來分享一下我們的方案。

## 關於 HagiCode

本文分享的方案來自我們在 [HagiCode](https://hagicode.com) 專案中的實踐經驗。HagiCode 是一個 AI 輔助開發平台，在開發程序中，我們也遇到了 AI 產生 UI 不一致的問題。爲了解決這個問題，我們建置設計畫廊站點並建立規範化的 design.md，本文就是這套方案的總結。

## 分析

在動手寫程式碼之前，我們先來拆解一下這個問題的幾個技術挑戰。

### 內容源管理：如何統一分散的設計資源？

上游的 awesome-design-md 存放庫包含了大量設計文件，但我們需要一種方式把它納入到我們的專案中。

**方案：使用 git submodule**

```
awesome-design-md-site
└── vendor/awesome-design-md          # 上游资源（git submodule）
```

這樣做有幾個好處：
- 版本可控：可以鎖定定特定的上游版本
- 離線建置：不需要在建置時請求外部 API
- 內容審閱：可以在 PR 中看到具體變更

### 資料標準化：不同文件結構怎麼統一？

不同公司的設計文件結構可能不同，有些缺少預覽檔案，有些命名不統一。我們需要在建置期進行標準化處理。

3. 提取並渲染 Markdown 內容爲 HTML

### 靜態站點架構：怎麼在保持靜態部署的同時提供動態搜索？

既然是設計畫廊，搜索功能是必須的。但 Astro 是靜態站點產生器，怎麼實作實時搜索呢？

**方案：React island + URL 查詢參數同步**

```typescript
// src/components/gallery/SearchToolbar.tsx

export function SearchToolbar() {

  // URL 同步
  useEffect(() => {
    const params = new URLSearchParams(window.location.search);

```

這樣做的好處是保留了靜態站點的可部署性（可以部署到任何靜態代管服務），同時提供了即時過濾的使用者體驗。

### 設計文件化：怎麼讓 AI 理解並遵守設計規範？

這是整個方案的核心。我們需要建立一份結構化的 design.md，讓 AI 能夠理解並應用我們的設計規範。

**方案：借鑑 ClickHouse DESIGN.md 的結構**

[ClickHouse 的 DESIGN.md](https://github.com/ClickHouse/ClickHouse/blob/master/DESIGN.md) 是一個很好的參考，它包含了：
- Visual Theme & Atmosphere
- Color Palette & Roles
- Typography Rules
- Component Stylings
- Layout Principles
- Depth & Elevation
- Do's and Don'ts
- Responsive Behavior
- Agent Prompt Guide

我們的做法是：**結構參考，內容覆寫**。保留 ClickHouse DESIGN.md 的章節結構，但把內容取代成我們自己專案實際使用的設計 token 和元件規範。

## 解決

基於上述分析，我們的解決方案包含四個核心模組。

### 1. 內容攝取管線

這是整個系統的基礎，負責從上游資源中提取和標準化內容。

```typescript
// src/lib/content/awesomeDesignCatalog.ts

export async function scanSourceEntries(): Promise<DesignEntry[]> {
  const designDir = 'vendor/awesome-design-md/design-md';
  const entries: DesignEntry[] = [];

  for (const dir of await fs.readdir(designDir)) {
    const entryPath = path.join(designDir, dir);
    if (await isValidDesignEntry(entryPath)) {
      const entry = await normalizeDesignEntry(entryPath);
      entries.push(entry);
    }
  }

  return entries;
}

async function isValidDesignEntry(dir: string): Promise<boolean> {
  const requiredFiles = ['README.md', 'DESIGN.md'];
  for (const file of requiredFiles) {
    if (!(await fileExists(path.join(dir, file)))) {
      return false;
    }
  }
  return true;
}
```

### 2. 畫廊瀏覽界面

畫廊界面包括三個主要部分：

**首頁**：展示所有設計條目的卡片網格，每個卡片包含：
- 設計條目標題和簡介
- 預覽圖（如果有）
- 快速搜索高亮

**詳情頁**：聚合展示單個設計條目的完整資訊：
- README 文件
- DESIGN 文件
- 預覽（支援明/暗主題切換）
- 相鄰條目導航

**導航**：支援傳回畫廊、瀏覽相鄰條目

首頁畫廊使用高密度卡片佈局，把不同來源的 design.md 條目平鋪在一個統一的視覺框架裏，方便快速對比品牌風格、按鈕模式和排版節奏。

![Awesome Design MD Gallery 設計卡片網格](/article-images/design-md-consistent-ui/1775613199000-e652f7155dd5.png)

進入具體條目後，詳情頁會把設計摘要和實時預覽放在同一個頁面中，減少在文件、預覽和源碼之間來回切換的成本。

### 3. 搜索功能

搜索功能基於用戶端過濾，使用 URL 查詢參數保持狀態：

```typescript
// src/components/gallery/SearchToolbar.tsx

function SearchToolbar({ entries }: { entries: DesignEntry[] }) {
  const [query, setQuery] = useState('');
  const [results, setResults] = useState(entries);

  useEffect(() => {
    const params = new URLSearchParams(window.location.search);
    const initialQuery = params.get('q') || '';
    setQuery(initialQuery);
    filterEntries(initialQuery);
  }, []);

  const filterEntries = (searchQuery: string) => {
    const filtered = entries.filter(entry =>
      entry.searchText.toLowerCase().includes(searchQuery.toLowerCase())
    );
    setResults(filtered);
  };

  const handleChange = (e: React.ChangeEvent<HTMLInputElement>) => {
    const value = e.target.value;
    setQuery(value);
    filterEntries(value);

    // 更新 URL（不触发页面刷新）
    const newUrl = value
      ? `${window.location.pathname}?q=${encodeURIComponent(value)}`
      : window.location.pathname;
    window.history.replaceState({}, '', newUrl);
  };

  return (
    <div className="search-toolbar">
      <input
        type="text"
        value={query}
        onChange={handleChange}
        placeholder="搜索设计条目..."
      />
      <span className="result-count">{results.length} 个结果</span>
    </div>
  );
}
```

## 實踐

瞭解了方案之後，具體怎麼實施呢？

### 實施步驟

**第一步：初始化子模組**

```bash
# 添加上游仓库为子模块
git submodule add https://github.com/VoltAgent/awesome-design-md.git vendor/awesome-design-md

# 初始化并更新子模块
git submodule update --init --recursive
```

**第二步：建立內容管線**

實作 `awesomeDesignCatalog.ts`，包括：
- 檔案掃描和校驗邏輯
- Markdown 渲染（使用 Astro 的內建渲染器）
- 條目資料提取

**第三步：建置畫廊 UI**

使用 Astro + React Islands 建立：
- 首頁畫廊佈局（卡片網格）
- 設計卡片元件
- 搜索工具列
- 詳情頁佈局

**第四步：編寫設計文件**

基於 ClickHouse DESIGN.md 結構，填充自己專案的實際設計 token。更新 README.md，新增指向 design.md 的連結。

### 注意事項

**安全性**：Markdown 渲染需要過濾不安全的 HTML。Astro 的內建渲染器預設會過濾 script 標籤，但仍需注意 XSS 風險。

**效能**：大量 iframe 預覽可能影響首屏載入。建議使用 `loading="lazy"` 延遲載入預覽內容。

**維護性**：design.md 需要與程式碼實作保持同步。建議在 CI 中新增檢查，確保 CSS 變數在文件和程式碼中一致。

**可存取性**：確保顏色對比度符合 WCAG AA 標準（至少 4.5:1）。

### AI 使用指南

建立 design.md 之後，怎麼讓 AI 真正用它呢？這裏有幾個實用技巧：

**技巧一：在提示詞中明確參考**

```
请参考项目根目录的 design.md 文件，使用其中定义的设计规范来实现以下组件：
- 按钮：使用 primary 色调，圆角 8px
- 卡片：使用 elevation-2 阴影层级
```

**技巧二：要求 AI 參考具體的 CSS 變數**

```
实现一个导航栏，要求：
- 背景色使用 --color-bg-primary
- 边框使用 --color-border-subtle
- 文字使用 --text-color-primary
```

**技巧三：在系統提示詞中包含 design.md 內容**

如果你的 AI 工具支援自定義系統提示詞，可以將 design.md 的核心內容直接新增進去。

### 測試策略

**內容管線測試**：
- 檔案缺失場景（缺少 README.md 或 DESIGN.md）
- 格式錯誤場景（Markdown 剖析失敗）
- 空目錄場景

**搜索功能測試**：
- 空結果處理
- 特殊字元（如中文、emoji）
- URL 同步驗證

**UI 元件測試**：
- 明/暗主題切換
- 回應式佈局
- 預覽載入狀態

### 部署流程

```bash
# 1. 更新子模块到最新版本
git submodule update --remote

# 2. 重新构建站点
npm run build

# 3. 部署静态资源
npm run deploy
```

建議將子模組更新和建置部署自動化，可以在上游存放庫更新時自動觸發 CI 流程。

## 總結

HagiCode 在開發程序中遇到的 AI 產生 UI 不一致問題，本質上是缺少結構化的設計參考文件。透過建置設計畫廊站點和建立規範化的 design.md，我們成功解決了這個問題。

這套方案的核心價值在於：
- **統一資源**：整合分散的設計系統文件
- **結構化規範**：將設計規範以 AI 可理解的形式呈現
- **持續維護**：透過 git submodule 保持內容更新

如果你也在使用 AI 輔助前端開發，建議嘗試一下這個方案。建立一份結構化的 design.md，不僅能提升 AI 產生程式碼的一致性，也能說明團隊內部保持設計規範的統一。
