## C# 後端集成 CodeBuddy CLI 實戰指南

> 本文將詳細介紹如何在 C# 後端專案中集成 CodeBuddy CLI，實作 AI 程式設計助手能力的完整方案。

:::note[時間語境說明]
本文討論的是發佈當時的 CodeBuddy 集成設計與 provider 結構。當前常駐支援範圍請以 [AI Agent CLI 安裝](/related-software-installation/ai-agent-cli/) 與 [安裝指南](/installation) 爲準；`IFlowCli` 現僅保留爲歷史相容值，不屬於當前活躍支援範圍。
:::

## 背景

在現代 AI 程式碼助手開發中，單一 AI Provider 往往無法滿足複雜多變的開發場景。這就像，人生路遠，總不能只認一個方向吧？HagiCode 作爲一款多功能 AI 程式設計助手，需要支援多種 AI Provider 以提供更好的使用者體驗。畢竟，使用者的選擇權還是要給夠的。在 2026 年初，專案面臨一個關鍵決策：如何在 C# 後端中復原 CodeBuddy 的 ACP（Agent Communication Protocol）集成能力。

此前專案中曾實作過 CodeBuddy 對接，但相關程式碼在一次重構中被移除了。其實也沒什麼好抱怨的，程式碼疊代嘛，總有東西要被遺忘。本次技術方案的目標是完整復原這一能力，並優化架構使其更加穩健和可維護。

如果你也在考慮爲自己的專案接入多種 AI 程式設計助手，下面的方案或許能給你一些啓發——這可是我們踩了無數坑之後總結出來的經驗。或許能讓你少走點彎路，也算是我做過的一點好事吧。

## 關於 HagiCode

本文分享的方案來自我們在 [HagiCode](https://hagicode.com) 專案中的實踐經驗。HagiCode 是一個開放原始碼的 AI 程式碼助手專案，支援多種 AI Provider 和跨平台執行。爲了滿足不同使用者的偏好設定，我們需要能夠靈活切換各種 AI 程式設計助手，這就有了本文要介紹的 CodeBuddy 集成方案。

HagiCode 採用模組化設計，AI Provider 作爲可插拔的元件，這種架構讓我們可以輕鬆新增新的 AI 支援，而不影響現有功能。這也罷了，設計這種東西，當初做得好，後面省心不少。如果你對我們的技術架構感興趣，可以在 GitHub 上查看完整源碼。

## 架構設計

### 分層架構概覽

C# 與 CodeBuddy 的對接採用清晰的分層架構，這種設計讓程式碼職責分明，後期維護起來也更加方便：

```
┌─────────────────────────────────────────────┐
│           Provider 契约层                    │
│   AIProviderType 枚举 + 扩展方法             │
├─────────────────────────────────────────────┤
│           Provider 工厂层                    │
│   AIProviderFactory 依赖注入工厂             │
├─────────────────────────────────────────────┤
│           Provider 实现层                    │
│   CodebuddyCliProvider 具体实现              │
├─────────────────────────────────────────────┤
│           ACP 基础设施层                     │
│  ACPSessionManager / StdioAcpTransport      │
│  AcpRpcClient / AcpAgentClient              │
└─────────────────────────────────────────────┘
```

這種分層的好處是什麼呢？簡單說就是各層之間互不打擾。假設以後要換一種通信方式（比如從 stdio 改成 WebSocket），你只需要改最下面那一層，上面的業務程式碼完全不用動。畢竟，誰也不想牽一髮而動全身，改個通信方式還要改半天業務程式碼，那也太慘了。

### 核心元件剖析

**Provider 契約層** 是整個架構的基石。我們定義了 `AIProviderType` 列舉，其中 `CodebuddyCli = 3` 作爲列舉值，透過擴充方法實作字串與列舉的雙向映射。這樣設定檔案中的字串可以很方便地轉成列舉，調試時列舉也能轉成字串輸出。這也罷了，其實就是個映射關係，但做好了就是省心。

**Provider 工廠層** 負責根據設定建立對應的 Provider 執行個體。這裏使用了 .NET 的相依注入機制，配合 `ActivatorUtilities.CreateInstance` 實作動態建立。工廠模式的好處在於，新增一個 Provider 時只需要新增建立邏輯，不用修改已有的程式碼。這和寫文章差不多，想加個新章節，就加個新章節，不用把前面的都覆寫一遍。

**Provider 實作層** 是真正幹活的地方。`CodebuddyCliProvider` 實作了 `IAIProvider` 介面，提供 `ExecuteAsync`（非流式）和 `StreamAsync`（流式）兩種呼叫方式。

**ACP 基礎設施層** 則是通信的底層支撐。這一層處理所有的通訊協定細節，包括處理序管理、訊息序列化、回應剖析等。就像房子的地基，上面蓋得再漂亮，底下的東西得穩纔行。

## 通信機制

### Stdio 傳輸模式

CodeBuddy 使用 **Stdio（標準輸入輸出）** 方式與外部處理序通信。啓動命令很簡單：

```bash
codebuddy --acp
```

然後透過標準輸入輸出進行 JSON-RPC 訊息交換。這種方式的優勢在於：

1. **啓動迅速**：本地處理序通信沒有網路延遲
2. **設定簡單**：只需要指定可執行檔案路徑
3. **環境隔離**：每個會話獨立處理序，互不影響

通信程序中支援環境變數注入，常用的包括：

- `CODEBUDDY_API_KEY`：API 密鑰認證
- `CODEBUDDY_INTERNET_ENVIRONMENT`：網路環境設定

這就像，人與人之間的溝通，找個方便的方式，才能說得上話。

### 訊息通訊協定

ACP 基於 JSON-RPC 2.0 通訊協定，訊息格式大概是醬紫的：

```json
// 请求消息
{
  "jsonrpc": "2.0",
  "id": 1,
  "method": "agent/prompt",
  "params": {
    "prompt": "帮我写一个排序算法",
    "sessionId": "session-123"
  }
}

// 响应消息
{
  "jsonrpc": "2.0",
  "id": 1,
  "result": {
    "content": "这里是 AI 的回复..."
  }
}
```

實際實作中，我們把這些通訊協定細節都封裝好了，上層業務程式碼只需要關注 prompt 和 response 就行。這也罷了，封裝得好，後面的人用起來就舒服點。

## 核心實作

### 1. Provider 契約復原

首先在列舉檔案中復原 CodeBuddy 型別：

然後在擴充方法中新增字串映射，這樣設定檔案就可以用字串指定 Provider：

```csharp

private static readonly Dictionary<string, AIProviderType> _typeMap = new(
    StringComparer.OrdinalIgnoreCase)
{
    ["CodebuddyCli"] = AIProviderType.CodebuddyCli,
    ["Codebuddy"] = AIProviderType.CodebuddyCli,
    ["codebuddy"] = AIProviderType.CodebuddyCli,
    // ... 其他 provider 的映射

```

### 2. Provider 工廠集成

在工廠類中新增 CodeBuddy 的建立分支：

```csharp
// AIProviderFactory.cs
private IAIProvider? CreateProvider(AIProviderType providerType, ProviderConfiguration config)
{
    return providerType switch
    {
        AIProviderType.CodebuddyCli =>
            ActivatorUtilities.CreateInstance<CodebuddyCliProvider>(
                _serviceProvider,
                Options.Create(config)),
        // ... 其他 provider
        _ => throw new NotSupportedException($"Provider {providerType} not supported")
    };
}
```

這裏用了相依注入的 `ActivatorUtilities`，它會自動處理建構函式的參數注入，非常方便。這也罷了，.NET 的東西，用對了就是省心。

### 3. 完整的 Provider 實作

下面是 `CodebuddyCliProvider` 的核心實作，包含了流式和非流式兩種呼叫方式：

```csharp
public class CodebuddyCliProvider : IAIProvider
{
    private readonly ILogger<CodebuddyCliProvider> _logger;
    private readonly IACPSessionManager _sessionManager;
    private readonly ProviderConfiguration _config;

    public string Name => "CodebuddyCli";
    public bool SupportsStreaming => true;
    public ProviderCapabilities Capabilities { get; }

    public CodebuddyCliProvider(
        ILogger<CodebuddyCliProvider> logger,
        IACPSessionManager sessionManager,
        IOptions<ProviderConfiguration> config)
    {
        _logger = logger;
        _sessionManager = sessionManager;
        _config = config.Value;

        // 定义当前 Provider 的能力
        Capabilities = new ProviderCapabilities
        {
            SupportsStreaming = true,
            SupportsTools = true,
            SupportsSystemMessages = true,
            SupportsArtifacts = false,
            MaxTokens = 8192
        };
    }

    // 非流式调用：等所有结果一起返回
    public async Task<AIResponse> ExecuteAsync(
        AIRequest request,
        CancellationToken cancellationToken = default)
    {
        // 为请求创建独立会话
        var session = await _sessionManager.CreateSessionAsync(
            "CodebuddyCli",
            request.WorkingDirectory,
            cancellationToken,
            request.SessionId);

        try
        {
            var fullPrompt = BuildPrompt(request);
            await session.SendPromptAsync(fullPrompt, cancellationToken);

            var responseBuilder = new StringBuilder();
            var toolCalls = new List<AIToolCall>();

            // 收集所有响应块
            await foreach (var chunk in StreamFromSession(session, cancellationToken))
            {
                if (!string.IsNullOrEmpty(chunk.Content))
                {
                    responseBuilder.Append(chunk.Content);
                }
                // 处理工具调用...
            }

            return new AIResponse
            {
                Content = AIResultContentSanitizer.SanitizeResultContent(
                    responseBuilder.ToString()),
                ToolCalls = toolCalls,
                Provider = Name,
                Model = string.Empty
            };
        }
        finally
        {
            // 释放会话资源
            await session.DisposeAsync();
        }
    }

    // 流式调用：实时返回响应块
    public async IAsyncEnumerable<AIStreamingChunk> StreamAsync(
        AIRequest request,
        [EnumeratorCancellation] CancellationToken cancellationToken = default)
    {
        var session = await _sessionManager.CreateSessionAsync(
            "CodebuddyCli",
            request.WorkingDirectory,
            cancellationToken);

        try
        {
            var fullPrompt = BuildPrompt(request);
            await session.SendPromptAsync(fullPrompt, cancellationToken);

            await foreach (var chunk in StreamFromSession(session, cancellationToken))
            {
                yield return chunk;
            }
        }
        finally
        {
            await session.DisposeAsync();
        }
    }

    private async IAsyncEnumerable<AIStreamingChunk> StreamFromSession(
        IACPSession session,
        [EnumeratorCancellation] CancellationToken cancellationToken)
    {
        // 遍历会话中的所有更新
        await foreach (var notification in session.ReceiveUpdatesAsync(cancellationToken))
        {
            switch (notification.Update)
            {
                case AgentMessageChunkSessionUpdate agentMessage:
                    // 处理文本内容块
                    if (agentMessage.Content is AcpImp.TextContentBlock textContent)
                    {
                        yield return new AIStreamingChunk
                        {
                            Content = textContent.Text,
                            Type = StreamingChunkType.ContentDelta,
                            IsComplete = false
                        };
                    }
                    break;

                case ToolCallSessionUpdate toolCall:
                    // 处理工具调用
                    yield return new AIStreamingChunk
                    {
                        Content = string.Empty,
                        Type = StreamingChunkType.ToolCallDelta,
                        ToolCallDelta = new AIToolCallDelta
                        {
                            Id = toolCall.ToolCallId,
                            Name = toolCall.Kind.ToString(),
                            Arguments = toolCall.RawInput?.ToString()
                        }
                    };
                    break;

                case AcpImp.PromptCompletedSessionUpdate:
                    // 响应完成
                    yield break;
            }
        }
    }

    // 构建完整的提示词
    private string BuildPrompt(AIRequest request, string? embeddedCommandPrompt = null)
    {
        var sb = new StringBuilder();

        // 嵌入命令提示词（如果有）
        if (!string.IsNullOrEmpty(embeddedCommandPrompt))
        {
            sb.AppendLine(embeddedCommandPrompt);
            sb.AppendLine();
        }

        // 系统消息
        if (!string.IsNullOrEmpty(request.SystemMessage))
        {
            sb.AppendLine(request.SystemMessage);
            sb.AppendLine();
        }

        // 用户 prompt
        sb.Append(request.Prompt);
        return sb.ToString();
    }
}
```

這段程式碼有幾個關鍵點：

1. **會話管理**：每個請求建立獨立會話，請求完成後釋放資源。這是坑踩出來的經驗——如果會話複用做得不好，很容易出現狀態污染的問題。畢竟，用過就得收拾乾淨，不然下次用的人就麻煩了。

2. **流式處理**：`IAsyncEnumerable` 讓回應可以邊產生邊傳回，不用等全部內容產生完。這對於長文字場景特別重要，使用者體驗會好很多。就像，等結果的人也不想一直乾等着不是。

3. **工具呼叫**：CodeBuddy 支援工具呼叫（Function Calling），透過 `ToolCallSessionUpdate` 處理。這個能力對於複雜的程式碼編輯任務很關鍵。

4. **內容過濾**：使用 `AIResultContentSanitizer` 過濾 Think 塊內容，保持輸出乾淨。

### 4. 相依注入設定

在模組註冊中新增相關服務：

```csharp
// PCodeClaudeHelperModule.cs
public void ConfigureModule(IServiceCollection context)
{
    // 注册 Provider
    context.Services.AddTransient<CodebuddyCliProvider>();

    // 注册 ACP 基础设施
    context.Services.AddSingleton<IACPSessionManager, ACPSessionManager>();
    context.Services.AddSingleton<IAcpPlatformConfigurationResolver, AcpPlatformConfigurationResolver>();
    context.Services.AddSingleton<IAIRequestToAcpMapper, AIRequestToAcpMapper>();
    context.Services.AddSingleton<IAcpToAIResponseMapper, AcpToAIResponseMapper>();
}
```

## 設定示例

### 設定檔案

在 `appsettings.json` 中新增 CodeBuddy 相關設定：

```yaml
AI:
  # 默认使用的 Provider
  DefaultProvider: "CodebuddyCli"

  # Provider 配置
  Providers:
    CodebuddyCli:
      Type: "CodebuddyCli"
      WorkingDirectory: "C:/projects/my-app"
      ExecutablePath: "C:/tools/codebuddy.cmd"

  # 平台相关配置
  PlatformConfigurations:
    CodebuddyCli:
      ExecutablePath: "C:/tools/codebuddy.cmd"
      Arguments: "--acp"
      StartupTimeoutMs: 5000
      EnvironmentVariables:
        CODEBUDDY_API_KEY: "${CODEBUDDY_API_KEY}"
        CODEBUDDY_INTERNET_ENVIRONMENT: "production"
```

### 設定模型

對應的設定模型定義：

```csharp
public class CodebuddyPlatformConfiguration : IAcpPlatformConfiguration
{
    public string ProviderName => "CodebuddyCli";
    public AcpTransportType TransportType => AcpTransportType.Stdio;

    public string ExecutablePath { get; set; } = "codebuddy";
    public string Arguments { get; set; } = "--acp";
    public int StartupTimeoutMs { get; set; } = 5000;

    public Dictionary<string, string?>? EnvironmentVariables { get; set; }
}
```

## 實踐經驗總結

### 踩坑記錄

我們在實作程序中遇到了幾個典型的坑，分享出來讓大家少走彎路。畢竟，別人的坑，自己能避開就是好事：

1. **會話泄漏問題**：一開始沒有正確釋放會話，導致處理序資源耗盡。解決方法是使用 `try-finally` 確保每次請求都會釋放資源。這也罷了，用過的東西得放回去，不然後面的人用什麼。

2. **環境變數傳遞**：Windows 和 Linux 的環境變數語法不同，後來統一使用 `Dictionary<string, string?>` 來處理。跨平台這種事，一開始就統一規範，後面就省心。

3. **超時設定**：CLI 啓動需要時間，設定了 5 秒的啓動超時，避免快速請求失敗。凡事都得有個度，太急了反而辦不成事。

4. **編碼問題**：Windows 上預設編碼可能導致中文亂碼，在啓動處理序時顯式指定 UTF-8 編碼。中文顯示不出來，那多難受。

### 效能優化

1. **會話池**：對於頻繁的短請求，可以考慮實作會話池來複用處理序
2. **連接快取**：工廠類已經支援 Provider 執行個體快取
3. **非同步優先**：全程使用非同步程式設計，避免封鎖執行緒

效能這種事，能優化就優化，畢竟使用者等的越久，體驗就越差。
