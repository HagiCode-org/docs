## 打造 AI 冒險團：HagiCode 多 Agent 協作設定實戰

> 在現代軟體開發中，單一 AI Agent 已經難以滿足複雜需求。如何讓來自不同公司的多個 AI 助手在同一專案中協同工作？本文將分享 HagiCode 專案在實際開發中探索出的多 Agent 協作設定方案。

:::note[時間語境說明]
本文記錄的是 **2026-03-17 發佈時** 的多 Agent 設定實踐。當前常駐支援範圍已經以後端現行設定爲準：Claude Code、Codex、GitHub Copilot、OpenCode、Hermes、QoderCLI、Kiro、Kimi、Gemini、DeepAgents、Codebuddy；其中 `IFlowCli` 現僅保留爲歷史相容值，不屬於當前活躍支援範圍。若你想看當前安裝與支援口徑，請改看 [AI Agent CLI 安裝](/related-software-installation/ai-agent-cli/) 與 [安裝指南](/installation)。

## 背景

相信很多開發人員都有過這樣的經歷：專案中引入了 AI 助手輔助程式設計，效率確實提高了。但隨着需求越來越複雜，一個 AI Agent 開始不夠用了——你想讓它同時處理程式碼審查、文件產生、單元測試等多個任務，結果往往是顧此失彼，輸出質量參差不齊。

更頭疼的是，當你嘗試引入多個 AI 助手時，問題就變得更復雜了。每個 Agent 有自己的設定方式、API 介面和執行邏輯，彼此之間甚至會產生衝突。這就像一支球隊，每個球員都很厲害，但沒有人知道該怎麼配合，結果踢得亂七八糟。

HagiCode 專案在開發程序中也遇到了同樣的困擾。作爲一個涉及前端 VSCode 擴充、後端 AI 服務、跨平台桌面用戶端的複雜專案，我們在 **2026-03 的當時版本** 裏需要同時對接來自不同公司的多個 AI 助手：Claude Code、Codex、CodeBuddy、iFlow 等等。如何讓它們在同一專案中和諧共處、發揮各自特長，成了必須解決的關鍵問題。

其實這也罷了，畢竟誰願意每天跟一羣打架的 AI 打交道呢。

本文分享的方案，正是我們在 HagiCode 專案中實際踩坑、實際優化出來的多 Agent 協作設定實踐。如果你也在爲多 AI 助手協作而頭疼，相信這篇文章會給你一些啓發。或許吧，畢竟每個人的情況都不一樣。

## 關於 HagiCode

[HagiCode](https://hagicode.com) 是一個 AI 程式碼助手專案，採用多 AI 引擎協同工作的"冒險團"模式。專案地址：[github.com/HagiCode-org/site](https://github.com/HagiCode-org/site)。

本文分享的多 Agent 設定方案，正是 HagiCode 能夠在複雜專案中保持高效開發的核心技術之一。也沒什麼特別的，就是把一羣 AI 變成一支能打配合的冒險團而已。

## HagiCode 的多 Agent 架構設計

### 從"單打獨鬥"到"團隊協作"

在 HagiCode 專案早期，我們也嘗試過只用一個 AI Agent 來處理所有任務。很快我們就發現，這種方式存在明顯的瓶頸：不同的任務需要不同的能力側重點，有的任務需要更強的上下文理解能力，有的則需要更精準的程式碼修改能力。一個 Agent 很難在所有方面都表現出色。

這讓我們意識到，必須讓多個 Agent 協同工作。但問題是，如何讓不同公司的 AI 產品在同一個專案中和平共處？我們需要解決幾個核心問題：

1. **設定管理複雜性**：每個 Agent 有不同的設定方式、API 介面和執行模式
2. **通信通訊協定統一**：需要一種標準化的方式讓不同 Agent 之間進行資料交換
3. **任務分工協調**：如何合理分配任務，讓每個 Agent 發揮特長

帶着這些問題，我們開始設計 HagiCode 的多 Agent 架構。其實也沒那麼複雜，只是想明白了而已。

### 整體架構一覽

經過多次疊代，我們最終確定的架構是這樣的：

```
┌─────────────────────────────────────────────────────────────────┐
│                    AIProviderFactory                             │
│  (工厂模式统一管理所有 AI Provider)                               │
├─────────────────────────────────────────────────────────────────┤
│  ClaudeCodeCli  │  CodexCli  │  CodebuddyCli  │  IFlowCli    │
│  (Anthropic)   │  (OpenAI)  │  (智谱 GLM)     │  (智谱)      │
└─────────────────────────────────────────────────────────────────┘
```

核心思路是：透過統一的 Provider 介面，讓不同的 AI Agent 可以被同一套程式碼管理。同時使用工廠模式動態建立和設定這些 Provider，確保系統的擴充性和靈活性。

這就像生活中的分工，每個人都有自己的角色，只是這裏把這種分工變成了程式碼架構而已。

### Agent 型別與職責分工

根據 HagiCode 專案的實際使用經驗，我們爲每個 Agent 分配了不同的職責：

| Agent | 提供商 | 模型 | 主要用途 |
|-------|--------|------|---------|
| ClaudeCodeCli | Anthropic | glm-5-turbo | 產生技術方案和Proposal |
| CodexCli | OpenAI/Zed | gpt-5.4 | 執行精準的程式碼修改 |
| CodebuddyCli | 智譜 | glm-4.7 | 優化提案描述和文件 |
| IFlowCli | 智譜 | glm-4.7 | 歸檔提案和瀏覽記錄（當時設定；當前僅歷史相容） |
| OpenCodeCli | - | - | 通用程式碼編輯 |
| GitHubCopilot | Microsoft | - | 輔助程式設計和程式碼補全 |

這種分工的背後邏輯是：每個 Agent 都有自己擅長的領域。Claude Code 在理解和分析複雜需求方面表現出色，所以讓它負責前期的方案設計；Codex 在程式碼修改方面更精準，適合處理具體的實作任務；CodeBuddy 性價比高，用來優化文件再合適不過。

畢竟適合自己的纔是最好的，條條大路通羅馬，只是有的路好走一點，有的路稍微曲折一點罷了。

## 核心設定機制

### 統一的 Provider 介面設計

要讓不同的 AI Agent 能夠被統一管理，首先需要定義一套統一的介面。HagiCode 中定義了這個介面：

```csharp
public interface IAIProvider
{
    // 统一的 Provider 接口
    Task<IAIProvider?> GetProviderAsync(AIProviderType providerType);
    Task<IAIProvider?> GetProviderAsync(string providerName, CancellationToken cancellationToken);
}
```

這個介面看起來很簡單，但它是整個多 Agent 系統的基石。透過統一的介面，我們可以無視底層是哪個公司的 AI 產品，都以相同的方式進行呼叫。

其實這就是把複雜的事情簡單化了，畢竟簡單纔是美。

### Provider 工廠模式實作

有了統一的介面，接下來就是如何建立這些 Provider 執行個體。HagiCode 使用了工廠模式：

```csharp
private IAIProvider? CreateProvider(AIProviderType providerType, ProviderConfiguration config)
{
    return providerType switch
    {
        AIProviderType.ClaudeCodeCli =>
            ActivatorUtilities.CreateInstance<ClaudeCodeCliProvider>(_serviceProvider, Options.Create(config)),
        AIProviderType.CodebuddyCli =>
            ActivatorUtilities.CreateInstance<CodebuddyCliProvider>(_serviceProvider, Options.Create(config)),
        AIProviderType.CodexCli =>
            ActivatorUtilities.CreateInstance<CodexCliProvider>(_serviceProvider, Options.Create(config)),
        AIProviderType.IFlowCli =>
            ActivatorUtilities.CreateInstance<IFlowCliProvider>(_serviceProvider, Options.Create(config)),
        _ => null
    };
}
```

這裏用到了相依注入的 `ActivatorUtilities.CreateInstance`，它可以在執行階段動態建立 Provider 執行個體，並且自動注入相依項。這種設計的好處是：新增一個 Agent 型別時，只需要新增對應的 Provider 類，然後在工廠方法中加一個 case 分支即可，完全不需要修改現有程式碼。

這也罷了，畢竟誰願意每次加新功能都要改一堆舊程式碼呢。

### 動態設定剖析

爲了讓設定更靈活，我們還實作了型別映射機制：

```csharp
public static AIProviderTypeExtensions
{
    private static readonly Dictionary<string, AIProviderType> _typeMap = new(
        StringComparer.OrdinalIgnoreCase)
    {
        ["ClaudeCodeCli"] = AIProviderType.ClaudeCodeCli,
        ["CodebuddyCli"] = AIProviderType.CodebuddyCli,
        ["CodexCli"] = AIProviderType.CodexCli,
        ["IFlowCli"] = AIProviderType.IFlowCli,
        // ...更多类型映射
    };
}
```

這個映射表的作用是將字串形式的 Provider 名稱轉換爲列舉型別。這樣一來，設定檔案就可以使用直觀的字串名稱，而程式碼內部則使用型別安全的列舉進行處理。

畢竟設定這東西，越直觀越好，誰願意記一堆複雜的程式碼呢。

### 設定檔案示例

實際使用時，只需要在 `appsettings.json` 中設定即可：

```yaml
AI:
  Providers:
    Providers:
      ClaudeCodeCli:
        Enabled: true
        Model: glm-5-turbo
        WorkingDirectory: /path/to/project
      CodebuddyCli:
        Enabled: true
        Model: glm-4.7
      CodexCli:
        Enabled: true
        Model: gpt-5.4
      IFlowCli:
        Enabled: true
        Model: glm-4.7
```

每個 Provider 都可以獨立設定開關、模型版本、工作目錄等參數。這種設計既保證了靈活性，又便於管理和維護。

其實設定檔案就像人生的選項，你可以選擇開啓或關閉某些功能，只是程式碼裏的選擇更容易後悔罷了。

## 冒險團任務流轉

### 任務分工的藝術

有了統一的技術架構，接下來就是如何讓多個 Agent 協同工作了。HagiCode 設計了一套任務流轉機制，讓不同的 Agent 處理不同階段的任務：

```
提案创建 (用户)
    │
    ▼
[Claude Code] ──生成提案──▶ 提案文档
    │                        │
    │                        ▼
    │               [Codebuddy] ──优化描述──▶ 优化后提案
    │                        │
    │                        ▼
    │               [Codex] ──执行修改──▶ 代码变更
    │                        │
    │                        ▼
    └───────────────▶ [iFlow] ──归档──▶ 历史记录
```

這種分工的好處是：每個 Agent 只需要專注於自己擅長的任務，不需要"什麼都會"。Claude Code 負責從無到有產生提案，Codebuddy 負責把提案描述得更清晰，Codex 負責把提案變成實際的程式碼變更，iFlow 則負責把這些變更歸檔保存。

其實這就像生活中的團隊合作，每個人都有自己的角色，合起來才能完成一件大事。只是這裏的團隊成員是 AI 而已。

### 關鍵實踐要點

在實際執行中，我們總結了以下幾點經驗：

**1. Agent 選擇策略很重要**

不是隨便分配任務，而是要根據每個 Agent 的特長來分配：
- 提案產生：使用 Claude Code，因爲它有更強的上下文理解能力
- 程式碼執行：使用 Codex，因爲它在程式碼修改方面更精準
- 提案優化：使用 Codebuddy，因爲它的性價比高
- 歸檔存儲：使用 iFlow，因爲它穩定可靠

畢竟讓合適的人做合適的事，這是千古不變的道理。

**2. 設定隔離確保穩定性**

每個 Agent 的設定獨立管理，支援環境變數覆寫，工作目錄也相互獨立。這樣一來，一個 Agent 的設定出錯不會影響到其他 Agent。

這就像生活中的界限，每個人都有自己的空間，互不干擾才能和諧共處。

**3. 錯誤處理機制**

單個 Agent 失敗不應該影響整體流程。我們實作了降級策略：當某個 Agent 執行失敗時，系統可以自動切換到備用方案，或者直接跳過該步驟繼續執行後續任務。同時，完整的日誌記錄也便於事後排查問題。

畢竟誰也不能保證永遠不會出錯，關鍵是怎麼處理錯誤。這就像人生，總會遇到挫折，重要的是怎麼走出來。

**4. 監控與可觀測性**

透過 ACP 通訊協定（我們自定義的通信通訊協定，基於 JSON-RPC 2.0），可以追蹤每個 Agent 的執行狀態。會話隔離確保了並行安全，動態快取則優化了效能表現。

畢竟看不見的東西最容易出問題，有點監控總好過兩眼一抹黑。

## 實際效果與收益

採用這套多 Agent 協作設定後，HagiCode 專案的開發效率有了明顯提升。具體表現在：

1. **任務處理能力翻倍**：以前一個 Agent 需要同時處理多種任務，現在可以平行處理，吞吐量翻倍不止
2. **輸出質量更穩定**：每個 Agent 只專注於自己擅長的任務，輸出結果的一致性和質量都更高
3. **維護成本降低**：統一的介面和設定管理，讓整個系統更容易維護和擴充
4. **新增 Agent 簡單**：如果要接入新的 AI 產品，只需要實作介面、新增設定，不需要修改核心邏輯

這套方案不僅解決了 HagiCode 自身的問題，也證明了多 Agent 協作確實是一種可行的架構選擇。

其實效果還挺明顯的，只是程序有點折騰罷了。

## 總結

本文分享了 HagiCode 專案在多 Agent 協作設定方面的實踐經驗。核心要點包括：

1. **標準化介面**：透過 `IAIProvider` 統一不同 Agent 的行爲，讓程式碼可以無視底層是哪個公司的產品
2. **工廠模式**：使用 `ActivatorUtilities.CreateInstance` 動態建立 Provider 執行個體，支援執行階段設定和相依注入
3. **通訊協定統一**：ACP 通訊協定實作 Agent 間的標準化通信，基於 JSON-RPC 2.0 的雙向通信機制
4. **任務分流**：合理分配任務給不同的 Agent，讓它們各展所長，而不是試圖讓一個 Agent 做所有事情

這種設計不僅解決了"多 Agent 打架"的問題，還透過冒險團的任務流轉機制，實作了開發流程的自動化和專業化。

如果你也在考慮引入多個 AI 助手，希望本文能給你一些參考。當然，每個專案的情況不同，具體方案還需要根據實際情況調整。畢竟沒有放之四海而皆準的方案，適合自己的纔是最好的。

美的事物或人，不一定要佔有，只要她還是美的，自己好好看着她的美就好了。技術方案也是如此，適合自己的，就是最好的......
