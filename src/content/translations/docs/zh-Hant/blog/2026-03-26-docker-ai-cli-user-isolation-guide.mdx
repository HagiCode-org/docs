## Docker 容器中執行 AI CLI 工具：使用者隔離與永續性卷實戰指南

> 在容器化環境中集成 Claude Code、Codex、OpenCode 等 AI 程式設計工具，聽起來簡單，實則暗藏玄機。本文將深入剖析 HagiCode 專案在 Docker 部署中如何解決使用者權限、設定永續性、版本管理等核心挑戰，帶你避坑避雷。

## 背景

當我們決定在 Docker 容器內執行 AI 程式設計 CLI 工具時，最直覺的想法可能是："容器不就是 root 嗎，直接裝不就完事了？"其實啊，這想法看似簡單，背後卻藏着幾個必須解決的核心問題。

首先，**安全限制是第一道坎**。以 Claude CLI 爲例，它明確禁止以 root 使用者執行——這是強制性的安全檢查，檢測到 root 直接拒絕啓動。你可能會想，那我用 `USER` 指令切換一下不就行了？事情沒那麼簡單，容器的非 root 使用者和宿主機的使用者權限之間還存在映射問題。畢竟，這世間的事，哪有那麼簡單的呢？

其次，**狀態永續性是第二個坑**。Claude Code 需要登入，Codex 有自己的設定，OpenCode 也有快取目錄。如果每次容器重啓都重新設定，那這個"自動化"就毫無意義了。我們需要讓這些設定在容器生命週期之外持久存在。設定這東西，就像記憶一樣，說沒就沒，那也挺讓人鬱悶的。

第三個問題就是**權限一致性**。宿主機使用者建立的設定檔案，容器內的處理序能不能存取？UID/GID 不匹配會導致檔案權限報錯，這在實際部署中非常常見。這問題說起來也挺無奈的，可是沒轍。

這些問題看似獨立，實際上環環相扣。HagiCode 專案在開發程序中逐步摸索出了一套可行的解決方案，接下來我會詳細分享其中的技術細節和踩坑經歷。

## 爲什麼不能簡單用 root？

這裏有個常見的誤解：Docker 容器預設以 root 執行，那我就直接用 root 裝工具唄。這麼想的話，Claude CLI 會毫不客氣地給你一個下馬威。

```bash
# 直接以 root 运行 Claude CLI？不行
docker run --rm -it --user root myimage claude
# 输出: Error: This command cannot be run as root user
```

這是 Claude CLI 的硬性安全限制。原因很簡單：這些 CLI 工具會讀寫使用者的敏感設定，包括 API Token、本地快取、甚至可能執行使用者編寫的腳本。以 root 權限執行這些工具，潛在風險太大。畢竟，安全這東西，怎麼謹慎都不爲過。

那麼問題來了：怎麼才能既滿足 CLI 的安全要求，又保持容器管理的靈活性？我們需要換個思路——**不是在執行階段切換使用者，而是從映像建置階段就建立專用使用者**。有時候啊，換個角度看問題，答案就自然浮現了。

## 建立專用使用者：不止是換個名字

你可能會想，那我直接在 Dockerfile 里加一行 `USER` 指令不就得了？這確實是最簡單的方案，但不夠穩健。簡單的東西往往不夠優雅，不是嗎？

### 靜態建立 vs 動態映射

HagiCode 的方案是建立一個 UID 1000 的 `hagicode` 使用者，這個 UID 通常匹配大多數宿主機的預設使用者：

```dockerfile
RUN groupadd -o -g 1000 hagicode && \
    useradd -o -u 1000 -g 1000 -s /bin/bash -m hagicode && \
    mkdir -p /home/hagicode/.claude && \
    chown -R hagicode:hagicode /home/hagicode
```

但這隻解決了映像內建使用者的問題。如果宿主機使用者是 UID 1001 呢？容器啓動時還需要支援動態映射。

docker-entrypoint.sh 中的關鍵邏輯：

```bash
if [ -n "$PUID" ] && [ -n "$PGID" ]; then
    if ! id hagicode >/dev/null 2>&1; then
        groupadd -g "$PGID" hagicode
        useradd -u "$PUID" -g "$PGID" -s /bin/bash -m hagicode
    fi
fi
```

這樣設計的好處是：**映像建置時使用預設的 UID 1000，執行階段可以透過環境變數 PUID/PGID 動態調整**。無論宿主機使用者是什麼 UID，設定檔案的所有權都不會出問題。這設計說起來也挺自然的，畢竟，靈活性和預設值之間需要找到一個平衡點罷了。

## 永續性卷的設計哲學

每個 AI CLI 工具都有自己偏好設定的設定目錄，這需要一一對應：

| CLI 工具 | 容器內路徑 | 命名卷 |
|---------|-----------|--------|
| Claude | `/home/hagicode/.claude` | `claude-data` |
| Codex | `/home/hagicode/.codex` | `codex-data` |
| OpenCode | `/home/hagicode/.config/opencode` | `opencode-config-data` |

爲什麼用命名卷而不是綁定掛載？三個原因：

1. **簡化管理**：命名卷由 Docker 自動管理生命週期，不需要手動建立宿主機目錄
2. **權限隔離**：卷的初始內容由容器內使用者建立，避免宿主機權限衝突
3. **獨立遷移**：卷可以獨立於容器存在，升級映像時資料不會丟失

docker-compose-builder-web 會自動產生對應的卷設定：

```yaml
volumes:
  claude-data:
  codex-data:
  opencode-config-data:

services:
  hagicode:
    volumes:
      - claude-data:/home/hagicode/.claude
      - codex-data:/home/hagicode/.codex
      - opencode-config-data:/home/hagicode/.config/opencode
    user: "${PUID:-1000}:${PGID:-1000}"
```

注意這裏的 `user` 字段：透過環境變數注入 PUID/PGID，確保容器處理序以匹配宿主機的使用者身份執行。這細節說起來挺重要的，畢竟，權限問題一旦出現，排查起來也挺讓人頭疼的。

## 版本管理：烘焙版本與執行階段覆寫

Docker 映像的版本固定是保證可重現性的關鍵。但在實際開發中，我們經常需要測試新版本，或者緊急修復一個 bug。如果每次都要重新建置映像，那效率也太低了。

HagiCode 的策略是**固定版本作爲預設值，執行階段覆寫作爲擴充能力**。這也算是工程實踐中的一種妥協吧，穩定性和靈活性之間總要有個取捨。

Dockerfile.template 中固定版本：

```dockerfile
USER hagicode
WORKDIR /home/hagicode

# 配置 npm 全局安装路径
RUN mkdir -p /home/hagicode/.npm-global && \
    npm config set prefix '/home/hagicode/.npm-global'

# 安装 CLI 工具（使用固定版本）
RUN npm install -g @anthropic-ai/claude-code@2.1.71 && \
    npm install -g @openai/codex@0.112.0 && \
    npm install -g opencode-ai@1.2.25 && \
    npm cache clean --force
```

docker-entrypoint.sh 中支援執行階段覆寫：

```bash
install_cli_override_if_needed() {
    local package_name="$2"
    local override_version="$5"

    if [ -n "$override_version" ]; then
        gosu hagicode npm install -g "${package_name}@${override_version}"
    fi
}

# 使用示例
install_cli_override_if_needed "" "@anthropic-ai/claude-code" "" "" "${CLAUDE_CODE_CLI_VERSION}"
```

這樣，在不重新建置映像的情況下，可以透過環境變數測試新版本：

```bash
docker run -e CLAUDE_CODE_CLI_VERSION=2.2.0 myimage
```

這設計說起來也挺實用的，畢竟，誰願意每次測試新功能都要重新建置映像呢？

## 自動設定注入

除了手動設定 CLI 工具，有些場景下還需要自動注入設定。最典型的就是 API Token。

```bash
if [ -n "$ANTHROPIC_AUTH_TOKEN" ]; then
    mkdir -p /home/hagicode/.claude
    cat > /home/hagicode/.claude/settings.json <<EOF
{
  "env": {
    "ANTHROPIC_AUTH_TOKEN": "${ANTHROPIC_AUTH_TOKEN}"
  }
}
EOF
    chown -R hagicode:hagicode /home/hagicode/.claude
fi
```

這裏需要注意兩點：**敏感資訊透過環境變數傳入，不要硬編碼到映像中**；設定檔案的所有權要正確設定，否則 CLI 工具無法讀取。這事兒說起來挺基礎的，可是做錯的人還真不少。

## 最佳實踐與避坑指南

### 權限不匹配問題

這是最容易踩的坑。宿主機使用者的 UID 是 1001，容器內是 1000，建立的檔案互相存取不了。

```bash
# 正确做法：让容器匹配宿主机用户
docker run \
    -e PUID=$(id -u) \
    -e PGID=$(id -g) \
    myimage
```

這問題說起來也挺常見的，可是第一次遇到的時候，還是挺讓人鬱悶的。

### 容器重啓後設定丟失

如果你發現每次重啓都要重新登入，檢查一下是不是忘記掛載永續性捲了：

```yaml
volumes:
  - claude-data:/home/hagicode/.claude
```

設定這東西，辛辛苦苦設定好了，說沒就沒了，那感覺，怎麼說呢，挺讓人崩潰的。

### 版本升級的正確姿勢

不要直接在執行的容器裏執行 `npm install -g`。正確做法是：

1. 設定環境變數觸發覆寫安裝
2. 或者重新建置映像

```bash
# 方式一：运行时覆盖
docker run -e CLAUDE_CODE_CLI_VERSION=2.2.0 myimage

# 方式二：重新构建
docker build -t myimage:v2 .
```

條條大路通羅馬，只是有的路好走一點，有的路稍微曲折一點罷了。

### 安全加固清單

- API Token 透過環境變數傳入，不寫入映像
- 設定檔案設定 600 權限
- 始終以非 root 使用者執行應用
- 定期更新 CLI 版本，修復安全漏洞

安全這東西，說起來挺重要，可是真正落實的時候，又有多少人能做得好呢？

## 擴充新 CLI 工具

如果以後要支援新的 CLI 工具，只需要三步：

1. **Dockerfile.template**：新增安裝步驟
2. **docker-entrypoint.sh**：新增版本覆寫邏輯
3. **docker-compose-builder-web**：新增永續性卷映射

範本化的設計讓擴充變得簡單，不需要改動核心邏輯。這也算是過來人的一點心得，不是什麼大道理，只是踩過的坑罷了。

## 總結

Docker 容器中執行 AI CLI 工具，核心挑戰在於**使用者權限、設定永續性、版本管理**三個維度。HagiCode 專案透過建立專用使用者、命名卷隔離、環境變數覆寫的組合方案，實作了既安全又靈活的部署架構。

關鍵設計要點：

- **使用者隔離**：從映像建置階段建立專用使用者，執行階段支援 PUID/PGID 動態映射
- **永續性策略**：每個 CLI 工具對應獨立的命名卷，容器重啓不影響設定
- **版本靈活性**：固定預設值確保可重現性，執行階段覆寫提供測試能力
- **自動化設定**：支援透過環境變數自動注入敏感設定

這套方案在 HagiCode 專案中已經穩定執行了一段時間，希望能給有類似需求的開發人員一些參考。其實也沒那麼複雜，不過是些工程實踐罷了。
