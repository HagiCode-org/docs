在運營 **HagiCode** 的程序中，我們一直面臨一個"盲盒"問題：我們產出內容，但不清楚使用者是如何閱讀的。雖然 GitHub 能看到 Star 數，但這太滯後了。我們需要知道：

- 使用者到底有沒有看完我們的教程？
- 那些複雜的設定文件，是在哪一步勸退了使用者的？

市面上有很多分析工具，比如 Google Analytics（GA）和 Microsoft Clarity。GA 功能強大但設定複雜，且受到隱私法規（如 GDPR）的嚴格限制。而 Clarity 作爲微軟推出的免費熱力圖工具，不僅功能直觀，而且在隱私合規上相對寬鬆，非常適合技術文件站點。

**我們的目標很明確**：在 **HagiCode** 的文件站點中無縫集成 Clarity，既要在所有頁面生效，又要給使用者留有"退出"的權利（隱私合規）。

## 技術選型與探索

起初，我們在 Proposal 階段討論了多種集成方式。既然我們使用的是 Starlight（基於 Astro 的文件框架），最直觀的想法是利用 Astro 的 Hooks。

我們首先嚐試了修改 `astro.config.mjs`，計劃在建置時注入 Clarity 腳本。雖然這種方式能保證全局覆寫，但缺乏靈活性——我們無法根據使用者的偏好設定動態載入或卸載腳本。

考慮到使用者體驗和隱私控制，我們最終決定採用 **元件覆寫** 的方案。Starlight 允許開發人員覆寫其內部元件，這意味着我們可以接管 `<footer>` 或 `<head>` 的渲染邏輯，從而精細控制 Clarity 的載入時機。

這裏有一個小插曲：原本我們想建立一個名爲 `StarlightWrapper.astro` 的佈局包裝器。但在實際調試中發現，Starlight 的路由機制並不會自動呼叫這個自定義 Wrapper，這導致腳本在部分頁面失效。這算是一個典型的"想當然"踩坑經歷，提醒我們**必須深入理解框架的渲染流程，而不是盲目套用通用框架模式**。

## 核心方案：Footer 元件覆寫

爲了確保 Clarity 腳本在所有頁面（包括文件和博客）載入，並且不破壞原有的頁面結構，我們選擇了覆寫 Starlight 的 `Footer` 元件。

### 爲什麼是 Footer？

1.  **全局性**：Footer 幾乎在所有標準頁面都會出現。
2.  **非侵入性**：將腳本放在 Footer 區域（實際渲染在 body 底部）不會封鎖頁面的關鍵渲染路徑（LCP），對效能影響最小。
3.  **邏輯集中**：可以在元件內部統一處理 Cookie 同意邏輯。

在 `src/components/` 目錄下建立檔案 `StarlightFooter.astro`。Starlight 會自動識別這個檔案並覆寫預設的 Footer。

**關鍵點剖析**：
- `is:inline`：告訴 Astro 不要處理這個 script 標籤內的內容，直接輸出到 HTML。這對第三方統計腳本至關重要，否則 Astro 的打包優化可能會導致腳本失效。
- `define:vars`：這是 Astro 3+ 的特性，允許在作用域內安全地注入變數。
- `import.meta.env.PROD`：確保在本地開發時（除非爲了調試）不產生無效統計，保持資料純淨。

## 經驗總結與坑點

在將這套方案落地到 HagiCode 的程序中，我們總結了幾個容易被忽視的細節：

1.  **`StarlightWrapper.astro` 是個陷阱**：
    如前所述，不要試圖去建立一個全局 Wrapper 來注入腳本，這在 Starlight 中行不通。老老實實覆寫特定元件（如 `StarlightFooter.astro` 或 `StarlightHead.astro`）纔是正解。

2.  **腳本位置的效能考量**：
    雖然 Clarity 建議放在 `<head>` 中以確保資料準確性，但對於文件站點，首屏載入速度（LCP）直接影響了 SEO 和使用者留存。我們選擇了放在 Footer（Body 底部），這會輕微丟失極少量"秒退"使用者的資料，但換來了更快的頁面載入體驗，這是一個值得的權衡。

3.  **開發環境的干擾**：
    一定要加上 `import.meta.env.PROD` 判斷。在開發模式下，你會頻繁重新整理頁面，這會產生大量無意義的測試資料，污染你的 Clarity 儀表盤。

## 效果驗證

部署完成後，你可以在 Clarity 主控台查看實時資料。通常在幾分鐘內，你就能看到使用者的heatmap（熱力圖）和 recordings（錄屏）。

對於 HagiCode 來說，透過這些資料我們發現：
- 很多使用者會反覆查看"快速開始"章節，說明我們的安裝指引可能還不夠直觀。
- "API 參考"頁面的停留時間最長，證實了我們核心使用者羣體的需求。

## 總結

接入 Microsoft Clarity 並不需要複雜的服務端改造，也不需要引入沉重的 SDK。

利用 Starlight 的元件覆寫機制，我們僅透過一個輕量級的 `StarlightFooter.astro` 元件，就實作了全局資料統計。這種"微集成"的方式，既保證了程式碼的整潔，又賦予了我們洞察使用者行爲的能力。

如果你也在運營技術類專案，特別是像 **HagiCode** 這樣需要不斷疊代文件的專案，強烈建議嘗試接入 Clarity。資料會告訴你，使用者真正的痛點在哪裏。
