## 在 React 專案中優雅實作新使用者引導：HagiCode 的 driver.js 實踐

> 當使用者第一次開啟你的產品時，他們真的知道該從哪裏開始嗎？這篇文章聊聊我們在 HagiCode 專案裏用 driver.js 做新使用者引導的那些事兒，也算是拋磚引玉罷了。

## 背景

你有沒有遇到過這樣的場景：新使用者註冊了你的產品，開啟頁面後一臉茫然，東張西望，不知道該點哪裏、該做什麼。作爲開發人員，我們總以爲使用者會"自己探索"，畢竟人的好奇心是無限的嘛。可現實是——大部分使用者會在幾分鐘內因爲找不到入口而悄悄離開，就像故事開始得突然，結束得也自然。

新使用者引導是解決這個問題的重要手段，只是實作起來也不那麼簡單。一個好的引導系統需要：

- 能夠精準定位頁面元素並高亮顯示
- 支援多步驟引導流程
- 能夠記住使用者的選擇（完成/跳過）
- 不影響頁面效能和正常互動
- 程式碼結構清晰，易於維護

在開發 HagiCode 的程序中，我們也遇到了同樣的挑戰。HagiCode 是一個 AI 程式碼助手專案，核心工作流是"使用者建立提案 → AI 產生計劃 → 使用者審覈 → AI 執行"這樣一套 OpenSpec 流程。對於第一次接觸這個概念的使用者來說，這套流程是全新的，必須有一個好的引導來說明他們快速上手。畢竟，新事物總是需要一點時間的。

## 關於 HagiCode

本文分享的方案來自我們在 [HagiCode](https://hagicode.com) 專案中的實踐經驗。HagiCode 是一個基於 Claude 的 AI 程式碼助手，透過 OpenSpec 工作流說明開發人員更高效地完成程式碼任務。你可以在 [GitHub](https://github.com/HagiCode-org/site) 上查看我們的開放原始碼程式碼。

## 爲什麼選擇 driver.js

在技術選型階段，我們評估了幾個主流的引導程式庫，怎麼說呢，每個都有自己的特點：

- **Intro.js**：功能強大但體積較大，樣式定製相對複雜
- **Shepherd.js**：API 設計很好，但對於我們的場景來說有點"重"
- **driver.js**：輕量、簡潔、API 直觀，且支援 React 生態

最終我們選擇了 driver.js，其實也沒什麼特別的理由，主要基於以下幾點考慮：

1. **輕量級**：核心程式庫體積小，不會顯著增加打包體積
2. **API 簡潔**：設定項清晰直觀，上手快
3. **靈活性**：支援自定義定位、樣式和互動行爲
4. **動態匯入**：可以按需載入，不影響首屏效能

選型這件事，其實沒有最好的，只有最合適的罷了。

## 技術實作

### 核心設定

driver.js 的設定非常直觀，以下是 HagiCode 專案中的核心設定：

```typescript
import { driver } from 'driver.js';
import 'driver.js/dist/driver.css';

const newConversationDriver = driver({
  allowClose: true,           // 允许用户关闭引导
  animate: true,              // 启用动画效果
  overlayClickBehavior: 'close', // 点击遮罩层关闭引导
  disableActiveInteraction: false, // 保持元素可交互
  showProgress: false,        // 不显示进度条（我们有自定义进度管理）
  steps: guideSteps           // 引导步骤数组
});
```

這些設定背後的考慮是：

- `allowClose: true` - 尊重使用者選擇，不強制完成引導，畢竟強扭的瓜不甜
- `disableActiveInteraction: false` - 某些步驟需要使用者實際操作（如輸入文字），所以不能停用互動
- `overlayClickBehavior: 'close'` - 給使用者一個快速的退出方式

### 狀態管理

引導狀態的永續性是關鍵——我們不希望每次重新整理頁面都重新引導，那樣挺煩人的。HagiCode 使用 localStorage 來管理引導狀態：

```typescript
export type GuideState = 'pending' | 'dismissed' | 'completed';

export interface UserGuideState {
  session: GuideState;
  detailGuides: Record<string, GuideState>;
}

// 读取状态
export const getUserGuideState = (): UserGuideState => {
  const state = localStorage.getItem('userGuideState');
  return state ? JSON.parse(state) : { session: 'pending', detailGuides: {} };
};

// 更新状态
export const setUserGuideState = (state: UserGuideState) => {
  localStorage.setItem('userGuideState', JSON.stringify(state));
};
```

我們定義了三種狀態：

- `pending`：引導進行中，使用者還未完成或跳過
- `dismissed`：使用者主動關閉了引導
- `completed`：使用者完成了所有步驟

對於提案詳情頁的引導，我們還支援更細粒度的狀態追蹤（透過 `detailGuides` 字典），因爲一個提案可能會經歷多個階段（草稿、審覈、執行完成），每個階段都需要不同的引導。畢竟，事情的狀態總是在變化的。

### 目標元素定位

driver.js 使用 CSS 選擇器來定位目標元素。HagiCode 採用了一個約定：使用 `data-guide` 自定義屬性來標記引導目標：

```typescript
const steps = [
  {
    element: '[data-guide="launch"]',
    popover: {
      title: '开始新对话',
      description: '点击这里创建一个新的对话会话...'
    }
  }
];
```

在元件中這樣使用：

```typescript
<button data-guide="launch" onClick={handleLaunch}>
  新建对话
</button>
```

這種做法的好處是：

- 避免與業務樣式類名衝突
- 語義清晰，一眼就能看出這個元素與引導相關
- 便於統一管理和維護

### 動態匯入優化

因爲引導功能只在特定場景下才需要（比如新使用者第一次存取），我們採用動態匯入來優化初始載入效能：

```typescript
const initNewUserGuide = async () => {
  // 动态导入 driver.js
  const { driver } = await import('driver.js');
  await import('driver.js/dist/driver.css');

  // 初始化引导
  const newConversationDriver = driver({
    // ...配置
  });

  newConversationDriver.drive();
};
```

這樣 driver.js 及其樣式檔案只會在需要時才載入，不會影響首屏效能。畢竟，誰願意爲暫時用不到的東西付出等待的代價呢？

## 引導流程設計

HagiCode 實作了兩條引導路徑，覆寫了使用者的核心使用場景。

### 會話引導（10步）

這條引導說明使用者完成從建立對話到提交第一個完整提案的整個流程：

1. **launch** - 啓動引導，介紹"新建對話"按鈕
2. **compose** - 引導使用者在輸入框中輸入請求
3. **send** - 引導點擊發送按鈕
4. **proposal-launch-readme** - 引導建立 README 提案
5. **proposal-compose-readme** - 引導編輯 README 請求內容
6. **proposal-submit-readme** - 引導提交 README 提案
7. **proposal-launch-agents** - 引導建立 AGENTS.md 提案
8. **proposal-compose-agents** - 引導編輯 AGENTS.md 請求
9. **proposal-submit-agents** - 引導提交 AGENTS.md 提案
10. **proposal-wait** - 說明 AI 正在處理，請稍候

這條引導的設計思路是：透過兩個實際的提案建立任務（README 和 AGENTS.md），讓使用者親手體驗 HagiCode 的核心工作流。畢竟，紙上得來終覺淺，絕知此事要躬行。

### 提案詳情引導（3步）

當使用者進入提案詳情頁時，根據提案的當前狀態觸發對應的引導：

1. **drafting**（草稿階段）- 引導使用者查看 AI 產生的計劃
2. **reviewing**（審覈階段）- 引導使用者執行計劃
3. **executionCompleted**（完成階段）- 引導使用者歸檔計劃

這條引導的特點是狀態驅動程式——根據提案的實際狀態動態決定顯示哪個引導步驟。事物總是在變化，引導也應該跟着變化纔是。

在這個階段，引導會把使用者注意力聚焦到“產生規劃”這個關鍵動作上，避免第一次進入詳情頁時不知道該先做什麼。

### 元素渲染重試機制

在 React 應用中，引導目標元素可能還沒渲染完成（比如等待非同步資料載入）。爲了處理這種情況，HagiCode 實作了一個重試機制：

```typescript
const waitForElement = (selector: string, maxRetries = 10, interval = 100) => {
  let retries = 0;
  return new Promise<HTMLElement>((resolve, reject) => {
    const checkElement = () => {
      const element = document.querySelector(selector) as HTMLElement;
      if (element) {
        resolve(element);
      } else if (retries < maxRetries) {
        retries++;
        setTimeout(checkElement, interval);
      } else {
        reject(new Error(`Element not found: ${selector}`));
      }
    };
    checkElement();
  });
};
```

在初始化引導前呼叫這個函式，確保目標元素已經存在。有時候，多等待一下也是值得的。

## 最佳實踐總結

基於 HagiCode 的實踐經驗，這裏分享幾個關鍵的最佳實踐：

### 1. 引導應該是"可逃離的"

不要強制使用者完成引導。有些使用者是探索型的，他們更喜歡自己摸索。提供清晰的"跳過"按鈕，並記住使用者的選擇，下次不再打擾。畢竟，美的事物或人，不一定要佔有，只要她還是美的，自己好好看着她的美就好了。

### 2. 引導內容要簡潔有力

每個引導步驟應該聚焦於單一目標：

- **Title**：簡短清晰，不超過 10 個字
- **Description**：直擊要點，告訴使用者"這是啥"和"爲啥要用"

避免長篇大論的說明——使用者在引導階段的注意力是很有限的。話說多了，反而沒人願意看。

### 3. 選擇器要穩定

使用穩定的、不頻繁變化的元素標記方式。`data-guide` 自定義屬性是一個好選擇，避免相依 class 名或 DOM 結構，因爲這些很容易在重構中變化。程式碼總是在變化的，但有些東西應該儘量保持穩定。

### 4. 測試你的引導

HagiCode 爲引導功能編寫了完整的測試用例：

```typescript
describe('NewUserConversationGuide', () => {
  it('应该正确初始化引导状态', () => {
    const state = getUserGuideState();
    expect(state.session).toBe('pending');
  });

  it('应该正确更新引导状态', () => {
    setUserGuideState({ session: 'completed', detailGuides: {} });
    const state = getUserGuideState();
    expect(state.session).toBe('completed');
  });
});
```

測試可以確保在重構程式碼時不會不小心破壞引導功能。畢竟，誰也不希望改點程式碼就把之前的功能搞壞了。

### 5. 效能優化

- 使用動態匯入延遲載入引導程式庫
- 避免在使用者已經完成引導後仍然初始化引導邏輯
- 考慮引導動畫的效能影響，低端設備上可以關閉動畫

效能這東西，就像生活一樣，該省的地方還是要省的。

## 總結

新使用者引導是提升產品使用者體驗的重要環節。在 HagiCode 專案中，我們使用 driver.js 建置了一套完整的引導系統，覆寫了從會話建立到提案執行的整個工作流。

透過本文的分享，我們希望傳達的核心觀點是：

1. **技術選型要匹配需求**：driver.js 不是最強的，但對我們來說是最合適的
2. **狀態管理很關鍵**：用 localStorage 永續性引導狀態，避免重複打擾使用者
3. **引導設計要聚焦**：每個步驟解決一個問題，不要貪多
4. **程式碼結構要清晰**：分離引導設定、狀態管理和 UI 邏輯，便於維護

如果你正在爲自己的專案新增新使用者引導功能，希望本文的實踐經驗能對你有所說明。其實技術這東西，也沒什麼神祕的，多嘗試，多總結，慢慢就好了......
