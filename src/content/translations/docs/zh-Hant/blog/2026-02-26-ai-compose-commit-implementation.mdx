## AI Compose Commit：用 AI 智能重構 Git 提交工作流

## Introduction

在軟體開發程序中,提交程式碼是程式員每天都要面對的日常工作。可是你有沒有經歷過這樣的場景:一天工作結束後,開啟 Git 看到幾十個未暫存的修改檔案,卻不知道該如何將它們組織成合理的提交?

傳統的方式是手動將檔案分批暫存、逐個提交、撰寫提交資訊,這個程序既耗時又容易出錯。咱們就常常在這上面浪費了不少時間,畢竟誰也不想在已經疲憊的晚上還要爲這些瑣事煩心。

我們在 HagiCode 專案中推出了一項新功能——AI Compose Commit,旨在徹底改變這個工作流程。它透過 AI 智能分析工作區中的所有未提交變更,自動將它們分組爲多個邏輯提交,並執行符合規範的提交操作。本文將深入探討這個功能的實作原理、技術架構以及我們在實踐中遇到的挑戰與解決方案。

## About HagiCode

本文分享的方案來自我們在 [HagiCode](https://hagicode.com) 專案中的實踐經驗。

## Background

### 傳統 Git 提交的痛點

Git 作爲版本控制系統,爲開發人員提供了強大的程式碼管理能力。但在實際使用中,提交操作往往成爲開發流程中的瓶頸:

1. **手動分組耗時**: 當有大量檔案變更時,開發人員需要逐個檢查檔案內容,判斷哪些屬於同一個功能,這需要耗費大量腦力
2. **提交資訊質量參差**: 撰寫符合 Conventional Commits 規範的提交資訊需要經驗和技巧,新手常常寫出不規範的提交
3. **多存放庫管理複雜**: 在 monorepo 環境中,需要在不同存放庫間切換,增加了操作複雜度
4. **工作流被打斷**: 提交程式碼會打斷開發思路,影響編碼效率

這些問題在大型專案和團隊協作環境中尤爲明顯。一個優秀的開發工具應該讓開發人員專注於核心的編碼工作,而不是被繁瑣的提交流程所困擾。

### AI 輔助開發的趨勢

近年來,AI 技術在軟體開發領域的應用日益廣泛。從程式碼補全、錯誤檢測到自動產生文件,AI 正在逐步滲透到開發的各個環節。在 Git 工作流方面,雖然已有一些工具提供提交資訊產生的功能,但大多侷限於單次提交的場景,缺乏對整個工作區變更的智能分析和分組能力。

其實 HagiCode 在開發程序中也遇到了這些痛點,我們曾嘗試過多種工具,但都或多或少存在一些侷限性。要麼是功能不夠完善,要麼是使用者體驗不夠好。這也是爲什麼我們最終決定自己實作 AI Compose Commit 功能的原因。

HagiCode 的 AI Compose Commit 功能正是爲了填補這一空白而生,它不僅是產生提交資訊,而是完整接管從檔案分析到執行提交的整個流程。

## Problem

### 技術挑戰

在實作 AI Compose Commit 功能的程序中,我們面臨了多個技術挑戰:

1. **檔案語義理解**: AI 需要理解檔案變更的語義關係,判斷哪些檔案屬於同一個功能模組。這需要深入分析檔案內容、目錄結構以及變更的上下文。

2. **提交分組策略**: 如何定義合理的分組標準?是按功能、按模組,還是按檔案型別?不同的專案可能適用不同的策略。

3. **實時反饋與非同步處理**: Git 操作可能需要較長時間,特別是處理大量檔案時。如何在保證使用者體驗的同時完成複雜操作?

4. **多存放庫支援**: 在 monorepo 架構下,需要在主存放庫和子存放庫之間正確路由操作。

5. **錯誤處理與回復**: 如果某個提交失敗,如何處理已執行的提交?是否需要回復已暫存的檔案?

6. **提交資訊一致性**: 產生的提交資訊需要符合專案現有的風格,保持歷史提交的格式一致。

### 效能考量

AI 處理大量檔案變更會消耗顯著的時間和計算資源。我們需要在以下方面進行優化:

- 減少不必要的 AI 呼叫
- 優化檔案上下文的建置方式
- 實作高效的 Git 操作批處理

這些問題在 HagiCode 的實際使用中都真實出現過,我們透過不斷的疊代和優化才找到了相對完美的解決方案。如果你也在開發類似的工具,希望我們的經驗能給你一些啓發。

## Solution

### 整體架構設計

我們採用了分層架構來實作 AI Compose Commit 功能,確保系統具有良好的可擴充性和可維護性:

#### 1. API 層(Web 層)

**GitController** 提供了 `POST /api/git/auto-compose-commit` 端點,作爲功能入口。爲了優化使用者體驗,我們採用了 Fire-and-Forget 非同步模式:

- 用戶端發起請求後,伺服器立即傳回 HTTP 202 Accepted
- 實際的 AI 處理在後臺非同步執行
- 處理完成後透過 SignalR 通知用戶端

這種設計確保了即使 AI 處理需要幾分鐘,使用者也能立即得到回應,不會感覺系統卡頓。

#### 2. 應用服務層(Application 層)

**GitAppService** 負責核心業務邏輯:

- 存放庫檢測:支援 monorepo 中的多存放庫管理
- 鎖定管理:防止並行操作導致的衝突
- 檔案暫存協調:與 AI 處理流程的互動
- 錯誤回復:處理失敗場景下的狀態復原

#### 3. 分佈式計算層(Orleans Grains)

**AIGrain** 作爲 AI 操作的核心執行單元,實作了 `IAIGrain` 介面中的 `AutoComposeCommitAsync` 方法:

```csharp
// 定义 AI 自动组合提交的接口方法
// 参数说明:
// - projectId: 项目唯一标识符
// - unstagedFiles: 未暂存文件列表,包含文件路径和状态信息
// - projectPath: 项目根目录路径(可选),用于访问项目上下文
// 返回值: 包含执行结果的响应对象,包括成功/失败状态和详细信息
[Alias("AutoComposeCommitAsync")]
[ResponseTimeout("00:20:00")] // 20 分钟超时,适用于处理大型变更集
Task<AutoComposeCommitResponseDto> AutoComposeCommitAsync(
    string projectId,
    GitFileStatusDto[] unstagedFiles,
    string? projectPath = null);
```

這個方法設定了 20 分鐘的超時時間,以處理大型變更集。HagiCode 在實際使用中發現,有些專案的單次變更可能涉及上百個檔案,需要更長的處理時間。

#### 4. AI 服務層

透過抽象的 `IAIService` 介面,我們實作了 AI 服務的可插拔架構。目前使用 Claude Helper 服務,但可以輕鬆切換到其他 AI 提供商。

### 核心實作邏輯

#### 檔案上下文建置

AI 需要了解每個檔案的狀態才能做出智能決策。我們透過 `BuildFileChangesXml` 方法建置檔案上下文:

```csharp
/// <summary>
/// 构建文件变更的 XML 表示形式,用于为 AI 提供完整的文件上下文信息
/// </summary>
/// <param name="stagedFiles">已暂存的文件列表,包含文件路径、状态和旧路径(针对重命名操作)</param>
/// <returns>格式化的 XML 字符串,包含所有文件的元数据信息</returns>
private static string BuildFileChangesXml(GitFileStatusDto[] stagedFiles)
{
    var sb = new StringBuilder();
    sb.AppendLine("<files>");

    foreach (var file in stagedFiles)
    {
        sb.AppendLine("  <file>");
        // 使用 XML 转义确保特殊字符不会破坏 XML 结构
        sb.AppendLine($"    <path>{System.Security.SecurityElement.Escape(file.Path)}</path>");
        sb.AppendLine($"    <status>{System.Security.SecurityElement.Escape(file.Status)}</status>");

        // 处理文件重命名场景,记录旧路径以便 AI 理解变更关系
        if (!string.IsNullOrEmpty(file.OldPath))
        {
            sb.AppendLine($"    <oldPath>{System.Security.SecurityElement.Escape(file.OldPath)}</oldPath>");
        }

        sb.AppendLine("  </file>");
    }

    sb.AppendLine("</files>");
    return sb.ToString();
}
```

這個 XML 格式的上下文包含檔案路徑、狀態和舊路徑(針對重命名操作),爲 AI 提供了完整的元資料。透過結構化的 XML 格式,我們確保了 AI 能夠準確理解每個檔案的狀態和變更型別。

#### AI 權限管理

爲了讓 AI 能夠直接執行 Git 操作,我們設定了全面的工具權限:

```csharp
// 定义 AI 可以使用的工具集合,包括文件操作和 Git 命令执行权限
// Read/Write/Edit: 文件读写和编辑能力
// Bash(git:*): 执行所有 Git 命令的权限
// 其他 Bash 命令: 用于查看文件内容和目录结构,辅助 AI 理解上下文
var allowedTools = new[]
{
    "Read", "Write", "Edit",
    "Bash(git:*)", "Bash(cat:*)", "Bash(ls:*)", "Bash(find:*)",
    "Bash(grep:*)", "Bash(head:*)", "Bash(tail:*)", "Bash(wc:*)"
};

// 构建完整的 AI 请求对象
var request = new AIRequest
{
    Prompt = prompt,                          // 完整的 Prompt 模板,包含任务指令和约束条件
    WorkingDirectory = projectPath ?? GetTempDirectory(), // 工作目录,确保 AI 在正确的项目上下文中执行
    AllowedTools = allowedTools,               // 允许使用的工具集合
    PermissionMode = PermissionMode.bypassPermissions, // 绕过权限检查,允许直接执行 Git 操作
    LanguagePreference = languagePreference         // 语言偏好设置,确保生成符合用户期望的提交信息
};
```

這裏使用了 `PermissionMode.bypassPermissions` 模式,允許 AI 直接執行 Git 命令而無需使用者確認。這是功能設計的核心,但同時也需要嚴格的輸入驗證來防止濫用。HagiCode 在實際部署中,透過後端的參數驗證和日誌監控,確保了這個機制的安全性。

#### 提交結果剖析

AI 執行完成後,會傳回結構化的結果。我們實作了雙重剖析策略以確保相容性:

```csharp
/// <summary>
/// 解析 AI 返回的提交执行结果,支持分隔符格式和正则表达式格式
/// </summary>
/// <param name="aiResponse">AI 返回的原始响应内容</param>
/// <returns>解析后的提交结果列表,每个结果包含提交哈希和执行状态</returns>
private List<CommitResultDto> ParseCommitExecutionResults(string aiResponse)
{
    var results = new List<CommitResultDto>();

    // 优先使用分隔符解析(新格式),这种格式更加明确和可靠
    if (aiResponse.Contains("---"))
    {
        logger.LogDebug("Using delimiter-based parsing for AI response");
        results = ParseDelimitedFormat(aiResponse);

        if (results.Count > 0)
        {
            return results; // 成功解析,直接返回结果
        }

        logger.LogWarning("Delimiter-based parsing produced no results, falling back to regex");
    }
    else
    {
        logger.LogDebug("No delimiter found, using legacy regex-based parsing");
    }

    // 回退到正则表达式解析(旧格式),确保向后兼容性
    return ParseLegacyFormat(aiResponse);
}
```

分隔符號格式使用 `---` 作爲提交之間的分隔,格式清晰且易於剖析:

```
---
Commit 1: abc123def456
feat(auth): add user login functionality

Implement JWT-based authentication with login form and API endpoints.

Co-Authored-By: Hagicode <noreply@hagicode.com>
---
Commit 2: 789ghi012jkl
docs(readme): update installation instructions

Add new setup steps for Docker environment.

Co-Authored-By: Hagicode <noreply@hagicode.com>
---
```

這種格式設計讓剖析變得簡單可靠,同時人類閱讀也很清晰。

### 鎖定管理機制

爲了防止並行操作導致的狀態衝突,我們實作了存放庫鎖定機制:

```csharp
// 获取仓库锁,防止并发操作
// 参数说明:
// - fullPath: 仓库的完整路径,用于标识不同的仓库实例
// - requestedBy: 请求者标识,用于追踪和日志记录
await _autoComposeLockService.AcquireLockAsync(fullPath, requestedBy);

try
{
    // 执行 AI Compose Commit 操作
    // 这部分代码会调用 Orleans Grain 的方法,执行实际的 AI 处理和 Git 操作
    await aiGrain.AutoComposeCommitAsync(projectId, unstagedFiles, projectPath);
}
finally
{
    // 确保锁被释放,无论操作成功或失败
    // 使用 finally 块可以保证异常情况下也能释放锁,避免死锁
    await _autoComposeLockService.ReleaseLockAsync(fullPath);
}
```

鎖定具有 20 分鐘的超時時間,與 AI 操作的超時設定保持一致。如果操作失敗或超時,系統會自動釋放鎖定,避免永久封鎖。HagiCode 在實際使用中發現,這個鎖定機制非常重要,特別是在團隊協作環境中,多個開發人員可能同時觸發 AI Compose Commit 操作。

### SignalR 實時通知

處理完成後,系統透過 SignalR 向前端發送通知:

```csharp
/// <summary>
/// 发送自动组合提交完成的通知
/// </summary>
/// <param name="projectId">项目标识符,用于路由通知到正确的客户端</param>
/// <param name="totalCount">总提交数量,包括成功和失败</param>
/// <param name="successCount">成功提交的数量</param>
/// <param name="failureCount">失败提交的数量</param>
/// <param name="success">整体操作是否成功标志</param>
/// <param name="error">错误信息(如果操作失败)</param>
private async Task SendAutoComposeCommitNotificationAsync(
    string projectId,
    int totalCount,
    int successCount,
    int failureCount,
    bool success,
    string? error)
{
    try
    {
        // 构建通知数据传输对象,包含详细的执行结果
        var notification = new AutoComposeCommitCompletedDto
        {
            ProjectId = projectId,
            TotalCount = totalCount,
            SuccessCount = successCount,
            FailureCount = failureCount,
            Success = success,
            Error = error
        };

        // 通过 SignalR Hub 广播通知到所有连接的客户端
        await messageService.SendAutoComposeCommitCompletedAsync(notification);

        logger.LogInformation(
            "Auto compose commit notification sent for project {ProjectId}: {SuccessCount}/{TotalCount} succeeded",
            projectId, successCount, totalCount);
    }
    catch (Exception ex)
    {
        // 记录通知错误但不影响主操作流程
        // 通知失败不应该导致整个操作失败
        logger.LogError(ex, "Failed to send auto compose commit notification for project {ProjectId}", projectId);
    }
}
```

前端收到通知後可以更新 UI,顯示提交成功或失敗的狀態,提升使用者體驗。這種實時反饋機制在 HagiCode 的使用中獲得了很好的使用者反饋,使用者可以清楚地知道操作何時完成以及結果如何。

## Implementation

### Prompt 工程設計

AI 的行爲完全由 Prompt 決定,我們精心設計了 Auto Compose Commit 的 Prompt 範本。以中文版本爲例(`auto-compose-commit.zh-CN.hbs`):

#### 非互動式模式支援

Prompt 開頭明確宣告支援非互動式執行模式,這是 CI/CD 和自動化腳本的關鍵需求:

```handlebars
**重要提示**:此提示词可能在非交互式环境中运行(如 CI/CD、自动化脚本)。

**非交互式模式**:
- 禁止使用 AskUserQuestion 或任何交互式工具
- 当需要用户输入时:
  - 使用合理的默认值(如提交类型使用 feat)
  - 跳过可选的确认步骤
  - 记录所做的假设
```

這個設計確保了 AI Compose Commit 功能不僅能在互動式 IDE 環境中使用,也能集成到 CI/CD 流程中,實作完全自動化的提交流程。

#### 分支保護機制

爲了防止 AI 執行危險操作,我們在 Prompt 中新增了嚴格的分支保護規則:

```handlebars
**分支保护**:
- 禁止执行任何分支切换操作(git checkout、git switch)
- 所有 git commit 命令必须在当前分支上执行
- 不得创建、删除或重命名分支
- 不得修改未跟踪文件或未暂存变更
- 如果需要分支切换才能完成操作,应返回错误而非执行
```

這些規則透過約束 AI 的工具使用範圍,確保操作的安全性。HagiCode 在實際測試中驗證了這些約束的有效性,AI 在遇到需要分支切換的場景時會安全地傳回錯誤,而不是執行危險操作。

#### 智能分組決策樹

Prompt 中詳細定義了檔案分組的決策邏輯:

```handlebars
**文件分组决策树**:
├── 是否为配置文件(package.json、tsconfig.json、.env 等)?
│   ├── 是 → 独立提交(类型:chore 或 build)
│   └── 否 → 继续
├── 是否为文档文件(README.md、*.md、docs/**)?
│   ├── 是 → 独立提交(类型:docs)
│   └── 否 → 继续
├── 是否与同一功能相关?
│   ├── 是 → 合并到同一提交
│   └── 否 → 分别提交
└── 是否为跨模块变更?
    ├── 是 → 按模块分组
    └── 否 → 按功能分组
```

這個決策樹爲 AI 提供了清晰的分組邏輯,確保產生的提交符合語義合理性。HagiCode 在實際使用中發現,這個決策樹能夠處理絕大多數常見場景,產生的分組結果符合開發人員預期。

#### 歷史格式一致性分析

爲了讓提交資訊與專案歷史保持一致,Prompt 要求 AI 在產生前分析最近的提交歷史:

```handlebars
**历史格式一致性**:在生成提交信息之前,你**必须**分析当前仓库的提交历史以匹配现有风格

1. 使用 git log -n 15 --pretty=format:"%H|%s|%b%n---%n" 获取最近的提交历史
2. 分析提交以识别:
   - 结构模式:项目是否使用多段落?是否有 "Changes:" 或 "Capabilities:" 部分?
   - 语言模式:提交信息是英文、中文还是混合?
   - 常用类型:最常使用哪些提交类型(feat、fix、docs 等)?
   - 特殊格式:是否有 Co-Authored-By 行?其他项目特定的约定?
3. 生成遵循检测到的模式的提交信息
```

這個分析確保了 AI 產生的提交資訊不會顯得突兀,而是與專案的提交歷史保持風格一致。在 HagiCode 的多語言專案中,這個功能特別重要,它能夠根據專案的提交歷史自動選擇合適的語言和格式。

#### Co-Authored-By 要求

每個提交必須包含 Co-Authored-By 資訊:

```handlebars
**重要**:每个提交必须添加 Co-Authored-By 信息
- 使用以下格式:git commit -m "type(scope): subject" -m "" -m "Co-Authored-By: Hagicode <noreply@hagicode.com>"
- 或者直接在提交信息中包含 Co-Authored-By 行
```

這不僅是爲了貢獻規範,也是爲了追蹤 AI 輔助的提交歷史。HagiCode 將這個要求作爲強制規則,確保所有 AI 產生的提交都帶有明確的來源標識。

### 工作流程詳解

完整的 AI Compose Commit 工作流程如下:

1. **使用者觸發**: 使用者在 Git Status 面板或 Quick Actions Zone 點擊"AI Auto Compose Commit"按鈕
2. **API 請求**: 前端發送 POST 請求到 `/api/git/auto-compose-commit` 端點
3. **立即回應**: 伺服器傳回 HTTP 202 Accepted,不等待處理完成
4. **後臺處理**:
   - GitAppService 取得存放庫鎖定
   - 呼叫 AIGrain 的 AutoComposeCommitAsync 方法
   - 建置檔案上下文 XML
   - 執行 AI Prompt,讓 AI 分析並執行提交
5. **AI 執行**:
   - 使用 Git 命令取得所有未暫存變更
   - 讀取檔案內容理解變更性質
   - 按語義關係對檔案分組
   - 對每組執行 `git add` 和 `git commit` 操作
6. **結果剖析**: 剖析 AI 傳回的執行結果
7. **通知發送**: 透過 SignalR 通知前端
8. **鎖定釋放**: 無論成功或失敗,都釋放存放庫鎖定

這個流程的設計確保了使用者可以在發起操作後立即繼續其他工作,而不需要等待 AI 處理完成。HagiCode 的使用者反饋表明,這種非同步處理方式大大提升了工作流體驗。

### 錯誤處理機制

我們實作了多層級的錯誤處理:

#### 1. 輸入驗證

```csharp
// 验证请求参数的有效性,防止无效请求到达后端处理逻辑
if (request.UnstagedFiles == null || request.UnstagedFiles.Count == 0)
```

#### 2. 錯誤回復

如果 AI 處理程序中出現錯誤,系統會執行回復操作,將已暫存的檔案取消暫存,避免留下不一致的狀態。這個機制在 HagiCode 的實際使用中挽救了多次意外中斷,確保了存放庫狀態的完整性。

#### 3. 超時處理

20 分鐘的超時設定確保了長時間執行的操作不會無限期封鎖資源。超時後,系統會釋放鎖定並通知使用者操作失敗。HagiCode 在實際使用中發現,大部分操作能夠在 2-5 分鐘內完成,只有處理超大型變更集時纔會接近超時限制。

## Best Practices

### 使用 AI Compose Commit 的最佳實踐

#### 1. 合理使用時機

AI Compose Commit 最適合以下場景:

- 一天工作結束後,批量處理多個檔案的變更
- 重構操作後,多個相關檔案需要分別提交
- 功能開發完成,需要將相關變更分組提交

不適合以下場景:

- 單個檔案的快速提交(直接使用普通提交更快)
- 需要精確控制提交內容的場景
- 包含敏感資訊的提交(需要人工審覈)

#### 2. 審查 AI 產生的提交

雖然 AI 智能分組很強大,但開發人員仍應審查產生的提交:

- 檢查提交的分組是否符合預期
- 驗證提交資訊的準確性
- 確認沒有遺漏或錯誤包含檔案

如果發現不合理的分組,可以使用 `git reset --soft HEAD~N` 撤銷後重新分組。HagiCode 的經驗表明,即使 AI 分組很智能,人工審查仍然是有價值的,特別是在重要的功能提交時。

#### 3. 配合專案規範

確保專案的 Git 設定支援 Conventional Commits:

```bash
# 安装 commitlint
npm install -g @commitlint/cli @commitlint/config-conventional

# 配置 commitlint
echo "module.exports = {extends: ['@commitlint/config-conventional']}" > commitlint.config.js
```

這樣可以在 CI/CD 流程中驗證提交資訊格式,與 AI Compose Commit 產生的格式保持一致。

### 實作類似功能的建議

如果你想在專案中實作類似的 AI 輔助提交功能,以下是我們的建議:

#### 1. 從小規模開始

先實作單次提交資訊產生,再逐步擴充到多提交分組功能。這樣更容易驗證和疊代。HagiCode 也是按照這個路徑逐步完善功能的,早期版本只支援單次提交,後來才擴充到多提交智能分組。

#### 2. 使用成熟的 AI SDK

不要自己實作 AI 呼叫邏輯,使用現有的 SDK 可以減少開發時間和潛在 bug。我們使用了 Claude Helper 服務,它提供了穩定的介面和完善的錯誤處理。

#### 3. 重視 Prompt 設計

Prompt 的質量直接決定了 AI 輸出的質量。投入時間設計詳細的 Prompt,包括:

- 明確的任務描述
- 具體的輸出格式要求
- 邊界情況的處理規則
- 示例說明

HagiCode 在 Prompt 設計上投入了大量時間,這是功能成功的關鍵因素之一。

#### 4. 實作全面的錯誤處理

AI 操作可能因爲各種原因失敗(網路問題、API 限流、內容審查等)。確保你的系統能夠優雅地處理這些錯誤,並提供有意義的錯誤資訊。

#### 5. 提供手動干預機制

不要完全自動化,給使用者保留控制權。提供查看分組結果、調整分組、手動編輯提交資訊等選項,平衡自動化與靈活性。HagiCode 雖然實作了自動執行,但仍然保留了預覽和調整的能力。

### 效能優化技巧

#### 1. 檔案過濾

在建置檔案上下文時,過濾掉不需要 AI 分析的檔案:

```csharp
// 过滤掉自动生成的文件和过大的文件,减少 AI 处理负担
var relevantFiles = stagedFiles
    .Where(f => !IsGeneratedFile(f.Path))
    .Where(f => !IsLargeFile(f.Path))
    .ToArray();
```

#### 2. 平行處理

如果支援多個獨立存放庫,可以平行處理不同存放庫的提交,提高整體效率。

#### 3. 快取優化

快取專案提交歷史分析結果,避免每次都重新分析。可以在設定檔案中存儲歷史格式偏好設定,減少 AI 呼叫次數。

## Conclusion

AI Compose Commit 功能代表了 AI 技術在軟體開發工具中的深度應用。透過智能分析檔案變更、自動分組提交、產生規範的提交資訊,它顯著提升了 Git 工作流的效率,讓開發人員能夠更專注於核心的編碼工作。

在實作程序中,我們學到了幾個重要的經驗:

1. **使用者反饋是關鍵**: 早期版本採用同步等待方式,使用者反饋體驗不佳,改爲 Fire-and-Forget 模式後滿意度大幅提升
2. **Prompt 設計決定質量**: 一個精心設計的 Prompt 比複雜的演算法更能保證 AI 輸出的質量
3. **安全永遠是第一位的**: 雖然賦予 AI 直接執行 Git 命令的權限帶來了效率提升,但必須配合嚴格的約束和驗證
4. **漸進式改進**: 從簡單場景開始,逐步增加複雜度,比一次性實作所有功能更容易成功

未來,我們計劃進一步優化 AI Compose Commit 功能,包括:

- 支援更多提交分組策略(按時間、按開發人員等)
- 集成程式碼審查流程,在提交前自動觸發審查
- 支援自定義提交資訊範本,滿足不同專案的個性化需求
