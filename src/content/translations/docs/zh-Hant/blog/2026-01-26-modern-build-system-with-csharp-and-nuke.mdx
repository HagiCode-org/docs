## 告別腳本地獄：爲什麼我們選擇用 C# 打造現代化建置系統

> 揭祕 HagiCode 專案如何利用 Nuke 實作型別安全、跨平台且高度可擴充的自動化建置流程，徹底解決傳統建置腳本的維護痛點。

## 背景

在軟體開發的漫長旅途中，"建置"這個詞往往讓人又愛又恨。愛的是，一鍵點擊，程式碼變成產品，那是程式員最迷人的時刻；恨的是，維護那一堆亂糟糟的建置腳本，簡直是噩夢。

在很多專案中，我們習慣了用 Python 寫腳本，或者用 XML 設定檔案（想象一下那段被 `<property>` 支配的恐懼）。但隨着專案複雜度的提升，尤其是像 HagiCode 這樣涉及前後端、多平台、多語言混合開發的專案，傳統的建置方式開始顯得力不從心。腳本邏輯分散、缺乏型別檢查、IDE 支援弱……這些問題像一個個小坑，時不時就讓開發團隊絆個跟頭。

爲了解決這些痛點，在 HagiCode 專案中，我們決定引入 **Nuke** —— 一個基於 C# 的現代化建置系統。它不僅僅是一個工具，更像是一種對建置流程的重新思考。今天，我們就來聊聊爲什麼選擇它，以及它是如何讓我們的開發體驗"起飛"的。

## 關於 HagiCode

> 嘿，介紹一下我們正在做的東西

我們正在開發 **HagiCode** —— 一款 AI 驅動程式的程式碼智能助手，讓開發體驗變得更智能、更便捷、更有趣。

**智能** —— AI 全程輔助，從想法到程式碼，讓編碼效率提升數倍。**便捷** —— 多執行緒並行操作，充分利用資源，開發流程順暢無阻。**有趣** —— 遊戲化機制和成就係統，讓編碼不再枯燥，充滿成就感。

專案正在快速疊代中，如果你對技術寫作、知識管理或者 AI 輔助開發感興趣，歡迎來 [GitHub](https://github.com/HagiCode-org/site) 看看～

## 核心剖析：爲什麼是 Nuke？

你可能心裏會犯嘀咕："哎呀，建置系統那麼多，比如 Make、Gradle，甚至直接用 Shell 腳本不行嗎？爲啥非得整一個 C# 的？"

這其實是個好問題。Nuke 的核心魅力在於它把我們最熟悉的程式設計語言特性帶進了建置腳本的世界。

### 1. 將建置流程模組化：Target 的藝術

Nuke 的設計理念非常清晰：**一切皆爲目標**。

在傳統的腳本里，我們可能會寫出幾百行線性執行的程式碼，邏輯錯綜複雜。而在 Nuke 中，我們將建置流程分解爲獨立的 `Target`（目標）。每個目標只負責一件事，比如：

- `Clean`: 清理輸出目錄
- `Restore`: 還原相依包
- `Compile`: 編譯程式碼
- `Test`: 執行單元測試

這種設計非常符合單一職責原則。就像搭積木一樣，我們可以隨意組合這些 Target。更重要的是，Nuke 允許我們定義 Target 之間的相依關係。比如，你想要 `Test`，那系統會自動檢查你是否先執行了 `Compile`；想要 `Compile`，自然得先 `Restore`。

這種相依關係圖不僅讓邏輯更清晰，還極大地提高了執行效率，Nuke 會自動分析最優執行路徑。

### 2. 型別安全：告別拼寫錯誤的噩夢

用過 Python 寫建置腳本的朋友肯定遇到過這種尷尬：腳本跑了五分鐘，最後報錯說 `Confi.guration` 拼寫錯了，或者傳了一個字串給了一個本該是數字的參數。

使用 C# 編寫建置腳本最大的優勢就是 **型別安全**。這意味着：

- **編譯時期檢查**：你在敲程式碼的時候，IDE 就會告訴你哪裏錯了，不用等到執行階段才發現。
- **重構無憂**：如果你想改個變數名或者方法名，IDE 的重構功能一鍵搞定，不用全局搜索取代提心吊膽。
- **智能提示**：強大的 IntelliSense 會自動補全程式碼，你不需要去翻文件記那些生僻的 API。

### 3. 跨平台：統一的建置體驗

以前在 Windows 上寫 `.bat`，在 Linux 上寫 `.sh`，爲了相容兩者，還得寫個 Python 腳本。現在，只要是 .NET Core（現 .NET 5+）能跑的地方，Nuke 就能跑。

這意味着無論團隊成員是使用 Windows、Linux 還是 macOS，無論是用 Visual Studio、VS Code 還是 Rider，大家執行的都是同一套邏輯。這就極大地消除了"在我機器上能跑"這類環境差異導致的問題。

### 4. 參數與設定管理

Nuke 提供了一套非常優雅的參數剖析機制。你不需要手動去剖析 `string[] args`，只需要定義一個屬性，加上 `[Parameter]` 特性，Nuke 就會自動處理命令列參數和設定檔案的映射。

比如，我們可以輕鬆定義建置設定：

```csharp
[Parameter("Configuration to build - Default is 'Debug'")]
readonly Configuration BuildConfiguration = IsLocalBuild ? Configuration.Debug : Configuration.Release;

Target Compile => _ => _
    .DependsOn(Restore)
    .Executes(() =>
    {
        // 在这里使用 BuildConfiguration，它是类型安全的
        DotNetBuild(s => s
            .SetConfiguration(BuildConfiguration)
            .SetProjectFile(SolutionFile));
    });
```

這種寫法既直觀又不容易出錯。

## 實踐指南：如何在專案中落地

空談誤國，實幹興邦。讓我們看看在 HagiCode 專案中，具體是怎麼落地這套方案的。

### 1. 規劃專案結構

我們不想讓建置腳本污染專案根目錄，也不想搞得像某些 Java 專案那樣目錄結構深不見底。所以，我們將所有與 Nuke 相關的建置檔案統一放置在 `nukeBuild/` 檔案夾中。

這樣做的好處是：
- 專案根目錄保持清爽。
- 建置邏輯內聚，方便管理。
- 新成員加入時，一眼就能看到"哦，這是建置相關的邏輯"。

### 2. 設計清晰的 Target 相依鏈

在設計 Target 時，我們遵循了一個原則：**原子化 + 相依流**。

每個 Target 應該足夠小，只做一件事。比如 `Clean` 就只管刪檔案，不要在裏面順便做打包。

推薦的相依流大概是這個樣子的：

`Clean` -> `Restore` -> `Compile` -> `Test` -> `Pack`

當然，這不是絕對的。比如如果你只想跑個測試，不想打包，Nuke 允許你直接執行 `nuke Test`，它會自動處理好前置的 Restore 和 Compile 步驟。

### 3. 完善的錯誤處理與日誌

建置腳本最怕的是什麼？是報錯資訊不明確。比如建置失敗了，日誌只顯示 "Error: 1"，這就讓人很抓狂。

在 Nuke 中，由於我們可以直接使用 C# 的例外處理機制，因此可以非常精確地捕獲和報告錯誤。

```csharp
Target Publish => _ => _
    .DependsOn(Test)
    .Executes(() =>
    {
        try 
        {
            // 尝试发布到 NuGet
            DotNetNuGetPush(s => s
                .SetTargetPath(ArtifactPath)
                .SetSource("https://api.nuget.org/v3/index.json")
                .SetApiKey(ApiKey));
        }
        catch (Exception ex)
        {
            Log.Error($"发布失败了，兄弟们检查一下 Key 对不对: {ex.Message}");
            throw; // 确保构建进程以非零退出码结束
        }
    });
```

### 4. 集成測試保障質量

建置腳本本身也是程式碼，也需要測試。Nuke 允許我們爲建置流程編寫測試，確保當我們修改了建置邏輯後，不會破壞現有的發佈流程。這在持續集成（CI）流水線中尤爲重要。

## 總結

透過引入 Nuke，HagiCode 的建置流程變得前所未有的順暢。它不僅僅是一個工具的取代，更是工程化思維的提升。

**我們收穫了什麼？**
- **可維護性**：程式碼即設定，邏輯清晰，新人也能快速上手。
- **穩定性**：強型別檢查減少了 90% 以上的低級錯誤。
- **一致性**：跨平台的統一體驗，消除了環境差異。

如果說以前寫建置腳本是"在黑暗中摸索"，那麼使用 Nuke 就像是"開着燈走夜路"。如果你受夠了維護那些難以調試的腳本語言，不妨試試把建置邏輯也搬到 C# 的世界裏來，也許你會發現，原來建置也可以這麼優雅。
