## Hagicode.Libs：統一集成多個 AI 程式設計助手 CLI 的工程實踐

> 在開發 HagiCode 專案的程序中，我們需要同時集成 Claude Code、Codex、CodeBuddy 等多個 AI 程式設計助手 CLI。每個 CLI 的介面、參數、輸出格式都不一樣，重複的集成程式碼讓專案越來越難以維護。本文分享我們如何透過 HagiCode.Libs 程式庫建置統一抽象層，解決這個工程痛點——也算是我們踩過的坑，積累下來的一點經驗罷了。

## 背景

現在的 AI 程式設計助手市場挺熱鬧的，除了 Claude Code，還有 OpenAI 的 Codex、智譜的 CodeBuddy 等等。作爲一個 AI 程式碼助手專案，HagiCode 需要在多個子專案（桌面端、後端、Web 等）中集成這些不同的 CLI 工具。

一開始問題還不大，集成一個 CLI 也就是幾百行程式碼的事兒。只是隨着要支援的 CLI 越來越多，事情就開始變得麻煩了——

每個 CLI 有不同的命令列參數格式，環境變數的要求也不一樣，輸出格式更是五花八門——有的輸出 JSON，有的流式 JSON，有的就是純文字。再加上跨平台相容性問題，Windows 和 Unix 系統的可執行檔案發現和處理序管理機制完全不同，程式碼重複度越來越高。其實這也無非就是 Ctrl+C、Ctrl+V 多了一點，但維護起來可就頭疼了。

最頭疼的是，每次要新增一個 CLI 功能支援，就得在好幾個專案裏改同樣的程式碼。這種方式顯然不是長久之計——程式碼也是有脾氣的，重複太多它也會鬧彆扭。

## 關於 HagiCode

本文分享的方案來自我們在 [HagiCode](https://hagicode.com) 專案中的實踐經驗。HagiCode 是一個開放原始碼的 AI 程式碼助手專案，需要同時維護前端 VSCode 擴充、後端 AI 服務、跨平台桌面用戶端等多個子專案。怎麼說呢，正是這種多語言、多平台的複雜場景，促成了 HagiCode.Libs 的誕生——算是被逼出來的，也罷。

## 分析：尋找共同點

雖然這些 AI 程式設計助手 CLI 各有特點，但從技術層面來看，它們存在明顯的共同特徵：

**相似的互動模式**：都是啓動 CLI 處理序，發送提示詞，接收流式回應，剖析訊息，最後會話結束或繼續——這一套流程，說到底都是一個模子刻出來的。

**相似的設定需求**：都需要 API 密鑰認證、工作目錄設定、模型選擇、工具權限控制、會話管理。畢竟大家都是喫 API 這碗飯的，差別無非是口味不同。

**跨平台挑戰一致**：都需要解決可執行檔案路徑剖析（`claude` vs `claude.exe` vs `/usr/local/bin/claude`）、處理序啓動和環境變數處理、Shell 命令跳脫和參數建置等問題。這跨平台的事兒，說多了都是淚——Windows 和 Unix 的差異，只有踩過坑的人才知道。

基於這些分析，我們需要一個統一的抽象層來提供一致的介面，封裝跨平台的 CLI 發現邏輯，處理流式輸出的剖析，同時支援相依注入和非 DI 場景使用。這事兒想想就頭大，但還是要面對——畢竟是自己的專案，哭着也要做完。

## 解決方案：HagiCode.Libs

我們建立了 **HagiCode.Libs** —— 一個輕量級的 .NET 10 程式庫工作空間，採用 MIT 開放原始碼通訊協定，現已發佈在 [GitHub](https://github.com/HagiCode-org/Hagicode.Libs)。雖然不是什麼驚天地泣鬼神的大作，但解決實際問題，還是挺香的。

### 專案結構

```
HagiCode.Libs/
├── src/
│   ├── HagiCode.Libs.Core/           # 核心功能
│   │   ├── Discovery/                 # CLI 可执行文件发现
│   │   ├── Process/                   # 跨平台进程管理
│   │   ├── Transport/                 # 流式消息传输
│   │   └── Environment/               # 运行时环境解析
│   ├── HagiCode.Libs.Providers/       # 提供者实现
│   │   ├── ClaudeCode/                # Claude Code 提供者
│   │   ├── Codex/                     # Codex 提供者
│   │   └── Codebuddy/                 # CodeBuddy 提供者
│   ├── HagiCode.Libs.ConsoleTesting/  # 测试框架
│   ├── HagiCode.Libs.ClaudeCode.Console/
│   ├── HagiCode.Libs.Codex.Console/
│   └── HagiCode.Libs.Codebuddy.Console/
└── tests/                             # xUnit 测试
```

### 設計目標

在設計 HagiCode.Libs 時，我們遵循了幾個原則——畢竟也都是踩過的坑，總結出來的經驗：

**零重型框架相依**：不相依 ABP 或其他大型框架，保持輕量級。這年頭，相依越少，麻煩越少——誰還沒被相依地獄毒打過呢。

**跨平台支援**：Windows、macOS、Linux 原生支援，不需要針對不同平台寫不同的程式碼。一套程式碼走天下，也挺好的。

**流式處理**：使用非同步流處理 CLI 輸出，更符合現代 .NET 的程式設計模式。畢竟時代在變，非同步纔是王道。

**靈活集成**：既支援相依注入場景，也允許直接執行個體化使用。蘿蔔青菜，各有所愛，怎麼方便怎麼來。

## 使用方式

### 透過相依注入

如果你的專案已經在使用相依注入（比如 ASP.NET Core 或通用主機），可以直接集成——這也算是個孩子，雖然不大，但挺乖的：

```csharp
using HagiCode.Libs.Providers;
using Microsoft.Extensions.DependencyInjection;

var services = new ServiceCollection();
services.AddHagiCodeLibs();

await using var provider = services.BuildServiceProvider();
var claude = provider.GetRequiredService<ICliProvider<ClaudeCodeOptions>>();

var options = new ClaudeCodeOptions
{
    ApiKey = "your-api-key",
    Model = "claude-sonnet-4-20250514"
};

await foreach (var message in claude.ExecuteAsync(options, "Hello, Claude!"))
{
    Console.WriteLine($"{message.Type}: {message.Content}");
}
```

### 直接執行個體化

如果是簡單的腳本或者不使用 DI 的場景，直接建立執行個體也行——說白了就是看個人喜好：

```csharp
var claude = new ClaudeCodeProvider();
var options = new ClaudeCodeOptions
{
    ApiKey = "sk-ant-xxx",
    Model = "claude-sonnet-4-20250514"
};

await foreach (var message in claude.ExecuteAsync(options, "帮我写一个快速排序"))
{
    // 处理消息
}
```

兩種方式使用的是同一套底層實作，你可以根據專案實際情況選擇合適的集成方式。這世界本就沒有標準答案，適合自己的纔是最好的——雖然這話有點老套，但確實是這麼個理兒。

## 實踐經驗分享

### 1. 專用測試主控台

每個提供者都有專用的測試主控台專案，方便獨立驗證集成效果——怎麼說呢，測試這件事，要麼不做，要做就做到位：

```bash
# Claude Code 测试
dotnet run --project src/HagiCode.Libs.ClaudeCode.Console -- --test-provider
dotnet run --project src/HagiCode.Libs.ClaudeCode.Console -- --test-all claude

# CodeBuddy 测试
dotnet run --project src/HagiCode.Libs.Codebuddy.Console -- --test-provider codebuddy-cli

# Codex 测试
dotnet run --project src/HagiCode.Libs.Codex.Console -- --test-provider codex-cli
```

測試場景覆寫了幾個關鍵場景：
- **Ping**：健康檢查，確認 CLI 可用
- **Simple Prompt**：基本提示測試
- **Complex Prompt**：多輪對話測試
- **Session Restore/Resume**：會話復原測試
- **Repository Analysis**：程式碼程式庫分析測試

這種獨立的測試主控台設計在調試時特別有用，可以快速定位問題是在 HagiCode.Libs 層還是 CLI 本身。這調試嘛，說白了就是看問題出在哪兒——方向對了，就成功了一半。

### 2. 跨平台 CI/CD 驗證

跨平台相容性是 HagiCode.Libs 的核心目標之一。我們設定了 GitHub Actions 工作流 `.github/workflows/cli-discovery-cross-platform.yml`，在 `ubuntu-latest`、`macos-latest`、`windows-latest` 三個平台上執行真實的 CLI 發現驗證。

這確保了每次程式碼變更都不會破壞跨平台相容性。本地開發時也可以透過以下命令復現——總不能讓 CI 幫你背所有鍋，自己本地也要能跑起來：

```bash
npm install --global @anthropic-ai/claude-code@2.1.79
HAGICODE_REAL_CLI_TESTS=1 dotnet test --filter "Category=RealCli"
```

### 3. 訊息流處理

HagiCode.Libs 使用非同步流處理 CLI 輸出，這種方式比傳統的回呼或事件模式更符合現代 .NET 的非同步程式設計風格——說到底，這就是技術的進步，誰也擋不住：

```csharp
public async IAsyncEnumerable<CliMessage> ExecuteAsync(
    TOptions options,
    string prompt,
    [EnumeratorCancellation] CancellationToken cancellationToken = default)
{
    // 启动 CLI 进程
    // 解析流式 JSON 输出
    // 生成 CliMessage 序列
}
```

訊息型別包括：
- `user`：使用者訊息
- `assistant`：助手回應
- `tool_use`：工具呼叫
- `result`：會話結束

這種設計讓呼叫方可以靈活地處理流式輸出，比如實時顯示、緩衝後處理、或者轉發到其他服務。美又何必在乎天晴陰呢？重要的是思路開啟了，怎麼用都成。

### 4. Git 存放庫探索

HagiCode.Libs.Exploration 模組提供了 Git 存放庫發現和狀態檢查功能，這在分析程式碼程式庫場景特別有用——這也是被逼出來的功能，誰讓 HagiCode 需要分析程式碼程式庫呢：

```csharp
// 发现 Git 仓库
var repositories = await GitRepositoryDiscovery.DiscoverAsync("/path/to/search");

// 获取仓库信息
var info = await GitRepository.GetInfoAsync(repoPath);
Console.WriteLine($"Branch: {info.Branch}, Remote: {info.RemoteUrl}");
Console.WriteLine($"Has uncommitted changes: {info.HasUncommittedChanges}");
```

HagiCode 的程式碼分析功能就用到了這個模組來識別專案結構和 Git 狀態。算是物盡其用，也算是個孩子，沒白養。

## 注意事項

基於我們在 HagiCode 專案中的實踐，有幾個地方需要特別注意——都是事兒，事兒，事兒：

**API 密鑰安全**：不要將 API 密鑰硬編碼到程式碼中，使用環境變數或設定管理。HagiCode.Libs 支援透過 Options 物件傳遞設定，方便集成各種設定源。畢竟安全這件事，怎麼小心都不爲過。

**CLI 版本鎖定定**：CI/CD 中我們鎖定定了特定版本（如 `@anthropic-ai/claude-code@2.1.79`）以減少版本漂移帶來的不確定性。本地開發時建議也使用固定版本。這版本的事兒，說多了都是淚——不固定版本，分分鐘教你做人。

**測試分類**：預設測試使用假提供者保持確定性和速度，真實 CLI 測試需要顯式啓用。這樣既保證了 CI 的快速反饋，又能在需要時進行真實環境驗證。這平衡木走起來，也不容易——快和穩，總是需要取捨。

**會話管理**：不同 CLI 的會話復原機制不同，Claude Code 使用 `.claude/` 目錄存儲會話，Codex 和 CodeBuddy 各有自己的方式。使用時要注意查看各自文件，瞭解會話永續性的具體機制。這也罷了，瞭解清楚總沒壞處。

## 總結

HagiCode.Libs 是我們在開發 HagiCode 程序中，爲了解決多 CLI 集成的重複工程問題而建置的統一抽象層。它透過提供一致的介面、封裝跨平台細節、支援靈活的集成方式，大大降低了集成多個 AI 程式設計助手的工程複雜度。一切都淡了——但經驗留下了。
