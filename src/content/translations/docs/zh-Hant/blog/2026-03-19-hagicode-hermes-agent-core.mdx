## HagiCode 爲什麼選擇 Hermes 作爲綜合 Agent 核心

> 在建置 AI 輔助編碼平台時，選擇合適的 Agent 核心直接決定了系統能力的天花板。畢竟有些事情，勉強不來——選錯了框架，怎麼折騰都不得勁。本文分享 HagiCode 在技術選型中的思考，以及 Hermes Agent 的集成實踐。

## 背景

做 AI 輔助編碼這事兒，最頭疼的莫過於選擇底層 Agent 框架了。其實市面上可選的方案也挺多的，只是吧——有的功能太簡單，有的部署太複雜，有的擴充性又不夠看。我們要的是一個既能跑在 5 美元 VPS 上，又能接入 GPU 集羣的方案，這要求說高也不高，說低吧，也不少人被勸退了。

但實際情況是，很多所謂的"全能 Agent" 要麼只能跑在雲端，要麼本地部署要求高得離譜。花了兩週時間調研各種方案後，我們做了一個大膽的決定：整個 Agent 核心推倒重來，採用 Hermes 作爲綜合 Agent 的底層引擎。

這決定帶來的一切，或許都是冥冥之中罷。

## 關於 HagiCode

本文分享的方案來自 [HagiCode](https://hagicode.com) 專案中的實踐經驗。HagiCode 是一個 AI 輔助編碼平台，透過 VSCode 擴充、桌面用戶端和 Web 服務，爲開發人員提供智能編碼助手。或許你也用過類似的工具，只是總覺得差了那麼一口氣——這我們也理解。

- GitHub: [github.com/HagiCode-org/site](https://github.com/HagiCode-org/site)
- 官網: [hagicode.com](https://hagicode.com)

## 爲什麼 HagiCode 需要 Hermes

在詳細介紹 Hermes 之前，先說說 HagiCode 爲什麼會有這樣的需求。這世上的事情啊，往往不是你想怎麼樣就能怎麼樣的，總得找個合適的由頭。

作爲一個 AI 程式碼助手，HagiCode 需要同時支援多種使用場景：
- 本地開發環境：開發人員希望在自gu電腦上執行，資料不出本地——這年頭，資料安全這事說大不大，說小也不小
- 團隊協作環境：小團隊可以共享部署在伺服器上的 Agent——省錢嘛，大家都不容易
- 雲端彈力擴充：處理複雜任務時，能自動擴充到 GPU 集羣——有備無患

這種"既要又要"的需求，讓我們把目光投向了 Hermes。這選擇對不對我不知道，只是當時也沒別的更好的辦法了。

## 什麼是 Hermes Agent

Hermes Agent 是由 **Nous Research** 建立的自主 AI Agent。可能有人對 Nous Research 不熟悉——他們就是開發了 Hermes、Nomos 和 Psyché 等開放原始碼大模型的實驗室。說起來他們也挺不容易的，做了這麼多好東西，知道的人卻不多。

跟傳統的 IDE 程式設計助手或者簡單的 API 聊天包裝器不同，Hermes 有一個特點：執行階段間越長，能力越強。它不是一次性完成任務就完事，而是能在長時間執行中持續學習和積累經驗。這點也挺像人的，是不是？

### 核心特性

Hermes 的幾個核心特性，正好契合了 HagiCode 的需求。你說巧不巧？

這意味着 HagiCode 可以根據使用者場景，選擇最合適的部署方式。個人使用者本地跑，團隊使用者伺服器部署，複雜任務上 GPU——一套程式碼搞定。這世道，能省一事算一事罷。

**多平台訊息網關**
Hermes 原生支援 Telegram、Discord、Slack、WhatsApp 等平台。對 HagiCode 來說，這意味着未來可以輕鬆支援這些渠道的 AI 助手。畢竟誰不想多幾條路呢？

**豐富的工具系統**
40+ 內建工具，加上 MCP（Model Context Protocol）擴充能力。這對於程式碼助手來說太重要了——執行 shell 命令、操作檔案系統、呼叫 Git，這些都需要工具支援。沒有工具的 Agent，就像沒有翅膀的鳥——想飛也飛不起來。

**跨會話記憶**
Hermes 有持久記憶系統，用 FTS5 全文檢索召回歷史對話。這讓 Agent 能記住之前的上下文，不會每次都"失憶"。有時候我也想失憶一下，什麼都不想，可就是做不到。

## HagiCode 如何集成 Hermes

說完了"爲什麼"，接下來看看"怎麼做"。有些事情想明白了，就得動手，光想不做也不是個事兒。

### Provider 層抽象

在 HagiCode 的架構中，所有 AI Provider 都實作統一的 `IAIProvider` 介面：

```csharp
public sealed class HermesCliProvider : IAIProvider, IVersionedAIProvider
{
    public ProviderCapabilities Capabilities { get; } = new ProviderCapabilities
    {
        SupportsStreaming = true,   // 支持流式输出
        SupportsTools = true,       // 支持工具调用
        SupportsSystemMessages = true, // 支持系统提示
        SupportsArtifacts = false
    };
}
```

這個抽象層讓 HagiCode 可以無縫切換不同的 AI Provider，無論是 OpenAI、Claude 還是 Hermes，上層呼叫方式完全一致。說白了，就是省事兒。

### ACP 通信通訊協定

Hermes 使用 **ACP (Agent Communication Protocol)** 進行通信。這是一個專門爲 Agent 通信設計的通訊協定，主要方法包括：

| 方法 | 說明 |
|------|------|
| `initialize` | 初始化連接，取得通訊協定版本和用戶端能力 |
| `authenticate` | 處理認證，支援多種認證方法 |
| `session/new` | 建立新會話，設定工作目錄和 MCP 伺服器 |
| `session/prompt` | 發送提示並取得回應 |

HagiCode 透過 `StdioAcpTransport` 實作 ACP 傳輸層，啓動 Hermes 子處理序並透過標準輸入輸出進行通信。這事兒聽起來複雜，做起來也還行——主要是要有耐心。

### 設定管理

透過 `HermesPlatformConfiguration` 類管理設定：

```csharp
public sealed class HermesPlatformConfiguration : IAcpPlatformConfiguration
{
    public string ExecutablePath { get; set; } = "hermes";
    public string Arguments { get; set; } = "acp";
    public int StartupTimeoutMs { get; set; } = 5000;
    public string ClientName { get; set; } = "HagiCode";
    public HermesAuthenticationConfiguration Authentication { get; set; }
    public HermesSessionDefaultsConfiguration SessionDefaults { get; set; }
}
```

在 `appsettings.json` 中設定 Hermes：

```json
{
  "Providers": {
    "HermesCli": {
      "ExecutablePath": "hermes",
      "Arguments": "acp",
      "StartupTimeoutMs": 10000,
      "ClientName": "HagiCode",
      "Authentication": {
        "PreferredMethodId": "api-key",
        "MethodInfo": {
          "api-key": "your-api-key-here"
        }
      },
      "SessionDefaults": {
        "Model": "claude-sonnet-4-20250514",
        "ModeId": "default"
      }
    }
  }
}
```

設定這東西吧，看着簡單，真要調對了也得費些功夫。

### Orleans 分佈式架構

HagiCode 使用 Orleans 建置分佈式系統，Hermes 集成透過以下元件實作：

- **HermesGrain**：Orleans Grain 實作，處理會話執行
- **HermesPlatformConfiguration**：平台特定設定
- **HermesAcpSessionAdapter**：ACP 會話配接器
- **HermesConsole**：專用的驗證主控台

 Orleans 這名字起得挺好聽的，傳說中的阿里巴巴——雖然此 Orleans 非彼 Orleans，但名字好聽總是加分的。

### 完整執行流程

以下是 Hermes Provider 的核心執行邏輯：

```csharp
private async IAsyncEnumerable<AIStreamingChunk> StreamCoreAsync(
    AIRequest request,
    string? embeddedCommandPrompt,
    [EnumeratorCancellation] CancellationToken cancellationToken)
{
    // 1. 创建传输层，启动 Hermes 子进程
    await using var transport = new StdioAcpTransport(
        platformConfiguration.GetExecutablePath(),
        platformConfiguration.GetArguments(),
        platformConfiguration.GetEnvironmentVariables(),
        platformConfiguration.GetStartupTimeout(),
        _loggerFactory.CreateLogger<StdioAcpTransport>());
    await transport.ConnectAsync(cancellationToken);

    // 2. 初始化，获取协议版本和认证方法
    var initializeResult = await SendHermesRequestAsync(
        transport, nextRequestId++, "initialize",
        BuildInitializeParameters(platformConfiguration), cancellationToken);

    // 3. 处理认证
    var authMethods = ParseAuthMethods(initializeResult);
    if (!isAuthenticated)
    {
        var methodId = platformConfiguration.Authentication.ResolveMethodId(authMethods);
        await SendHermesRequestAsync(transport, nextRequestId++, "authenticate", ...);
    }

    // 4. 创建会话
    var newSessionResult = await SendHermesRequestAsync(
        transport, nextRequestId++, "session/new",
        BuildNewSessionParameters(platformConfiguration, workingDirectory, model), cancellationToken);
    var sessionId = ParseSessionId(newSessionResult);

    // 5. 执行提示并收集流式响应
    await foreach (var payload in transport.ReceiveMessagesAsync(cancellationToken))
    {
        // 处理 session/update 通知，转换为流式块
        if (TryParseSessionNotification(root, out var notification))
        {
            if (_responseMapper.TryConvertToStreamingChunk(notification, out var chunk))
            {
                yield return chunk;
            }
        }
    }
}
```

程式碼嘛，看多了也就那麼回事。重要的是思路，對吧？

### 健康檢查

爲了保證 Hermes 服務的可用性，HagiCode 實作了健康檢查機制：

```csharp
public async Task<ProviderTestResult> PingAsync(CancellationToken cancellationToken = default)
{
    var response = await ExecuteAsync(
        new AIRequest
        {
            Prompt = "Reply with exactly PONG.",
            CessionId = null,
            AllowedTools = Array.Empty<string>(),
            WorkingDirectory = ResolveWorkingDirectory(null)
        },
        cancellationToken);

    var success = string.Equals(response.Content.Trim(), "PONG", StringComparison.OrdinalIgnoreCase);
    return new ProviderTestResult
    {
        ProviderName = Name,
        Success = success,
        ResponseTimeMs = stopwatch.ElapsedMilliseconds,
        ErrorMessage = success ? null : $"Unexpected Hermes ping response: '{response.Content}'."
    };
}
```

這大概就是所謂的"健康檢查"了罷。其實人也一樣，總要時不時檢查一下自己——只是通常沒人告訴我們應該檢查什麼。

## 實踐中的注意事項

集成 Hermes 程序中，有一些坑值得提前瞭解。這年頭，誰還沒踩過幾個坑呢？

### 認證方法設定

Hermes 支援多種認證方法（API Key、Token 等），需要根據實際部署情況選擇。設定錯誤會導致連接失敗，但錯誤資訊可能不夠直觀。有時候報錯資訊跟實際原因差了十萬八千里，得慢慢排查。

### MCP 伺服器設定

建立會話時可以設定 MCP 伺服器清單，讓 Hermes 呼叫外部工具。但要注意：
- MCP 伺服器地址必須可存取
- 超時時間要合理設定
- 伺服器不可用時的降級處理

這世道，防不勝防啊。

### 工作目錄管理

每個會話都需要指定工作目錄，確保 Hermes 能正確存取專案檔案。對於多專案場景，需要動態切換工作目錄。說起來簡單，做起來要考慮的情況也挺多的。

### 回應聚合處理

Hermes 的回應可能分散在 `session/update` 通知和最終結果中，需要正確合併處理，否則會出現內容丟失。這事兒我也沒少喫虧，慢慢就好了。

### 錯誤處理策略

執行階段錯誤應該明確傳回，而不是靜默回退到其他 Provider。這樣使用者才知道是 Hermes 出了問題，而不是莫名其妙換了別的模型。畢竟糊弄事兒也不是這麼個糊弄法。

## 總結

HagiCode 選擇 Hermes 作爲綜合 Agent 核心，不是拍腦袋的決定，而是基於實際需求和技術特點的慎重選擇。這選擇對不對，現在說也爲時過早，只是目前用起來還算順手。

Hermes 提供的靈活部署能力，讓 HagiCode 可以適應各種使用場景；強大的工具系統和 MCP 支援，讓 AI 助手能真正幹實事；而 ACP 通訊協定和 Provider 抽象層，則讓整個集成程序清晰可控。
