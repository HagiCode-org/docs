## Hermes Agent 集成實踐：從通訊協定到生產

> 分享 HagiCode 集成 Hermes Agent 的完整實踐，包括 ACP 通訊協定適配、會話池管理、前後端契約同步等核心經驗。

## 背景

在建置 AI 輔助編碼平台 HagiCode 的程序中，團隊需要集成一個既能在本地執行又能擴充到雲端的 Agent 框架。經過調研，Nous Research 的 Hermes Agent 被選爲綜合 Agent 的底層引擎。

其實選型這事兒，說難也不難，說簡單也不簡單。畢竟市面上能打的 Agent 框架也不少，只是 Hermes 那個 ACP 通訊協定和工具系統確實有點東西，剛好契合 HagiCode "既要又要"的需求場景——本地開發、團隊協作和雲端擴充。但要把 Hermes 真正融入生產系統，還需要解決一系列工程問題，這可不是鬧着玩的。

HagiCode 的技術棧基於 Orleans 建置分佈式系統，前端使用 React + TypeScript。集成 Hermes 需要在保持現有架構統一性的前提下，讓 Hermes 成爲與 ClaudeCode、OpenCode 等平行的"一等公民"執行器。說起來容易，做起來嘛，也就那樣吧。

本文分享我們在 HagiCode 專案中集成 Hermes Agent 的實踐經驗，希望能給面臨類似需求的團隊提供參考。畢竟，踩過的坑，沒必要讓別人再踩一遍。

## 關於 HagiCode

本文分享的方案來自我們在 [HagiCode](https://hagicode.com) 專案中的實踐經驗。HagiCode 是一個 AI 驅動程式的編碼輔助平台，支援多種 AI Provider 的統一接入和管理。在集成 Hermes Agent 的程序中，我們設計了一套通用的 Provider 抽象層，使得新的 Agent 型別可以無縫接入現有系統。

如果你對 HagiCode 感興趣，歡迎存取 [GitHub](https://github.com/HagiCode-org/site) 瞭解更多。多個人看，多份力量罷了。

## 架構設計

### 分層設計思路

HagiCode 的 Hermes 集成採用了清晰的分層架構，每層各司其職：

**後端核心層**
- `HermesCliProvider`: 實作 `IAIProvider` 介面，作爲統一的 AI Provider 入口
- `HermesPlatformConfiguration`: 管理 Hermes 可執行檔案路徑、參數、認證等設定
- `ICliProvider<HermesOptions>`: HagiCode.Libs 提供的底層 CLI 抽象，處理子處理序生命週期

**傳輸層**
- `StdioAcpTransport`: 透過標準輸入輸出與 Hermes ACP 子處理序通信
- ACP 通訊協定方法：`initialize`、`authenticate`、`session/new`、`session/prompt`

**執行階段層**
- `HermesGrain`: Orleans Grain 實作，處理分佈式會話執行
- `CliAcpSessionPool`: 會話池，複用 ACP 子處理序，避免頻繁啓動開銷

**前端層**
- `ExecutorAvatar`: Hermes 視覺標識和圖標
- `executorTypeAdapter`: Provider 型別映射邏輯
- SignalR 實時訊息傳遞：保持 Hermes 身份在訊息流中的一致性

這種分層設計使得各層可以獨立演進，比如未來要新增新的傳輸方式（如 WebSocket），只需修改傳輸層即可。畢竟，誰願意因爲改一個傳輸方式就把整個系統都翻一遍呢？多累啊。

### 統一介面抽象

所有 AI Provider 都實作 `IAIProvider` 介面，這是 HagiCode 架構的核心設計：

```csharp
public interface IAIProvider
{
    string Name { get; }
    ProviderCapabilities Capabilities { get; }

    IAsyncEnumerable<AIStreamingChunk> StreamAsync(
        AIRequest request,
        CancellationToken cancellationToken = default);

    Task<AIResponse> ExecuteAsync(
        AIRequest request,
        CancellationToken cancellationToken = default);
}
```

`HermesCliProvider` 實作了這個介面，與 `ClaudeCodeProvider`、`OpenCodeProvider` 等處於平等地位。這種設計帶來的好處：

1. **可取代性**: 切換 Provider 不影響上層業務邏輯
2. **可測試性**: 可以輕鬆 Mock Provider 進行單元測試
3. **可擴充性**: 新增 Provider 只需實作介面即可

說到底，介面這東西，就像規矩一樣。有了規矩，大家才能和諧共處，各自發揮所長，互不干擾。這難道不是一種美嗎？

## 核心實作

### Provider 層實作

`HermesCliProvider` 是整個集成的核心，它負責協調各個元件完成一次 AI 呼叫：

```csharp
public sealed class HermesCliProvider : IAIProvider, IVersionedAIProvider
{
    private readonly ICliProvider<LibsHermesOptions> _provider;
    private readonly ConcurrentDictionary<string, string> _sessionBindings;

    public ProviderCapabilities Capabilities { get; } = new()
    {
        SupportsStreaming = true,
        SupportsTools = true,
        SupportsSystemMessages = true,
        SupportsArtifacts = false
    };

    public async IAsyncEnumerable<AIStreamingChunk> StreamAsync(
        AIRequest request,
        [EnumeratorCancellation] CancellationToken cancellationToken = default)
    {
        // 1. 解析会话绑定 key
        var bindingKey = ResolveBindingKey(request.CessionId);

        // 2. 通过会话池获取或创建 Hermes 会话
        var options = new HermesOptions
        {
            ExecutablePath = _platformConfiguration.ExecutablePath,
            Arguments = _platformConfiguration.Arguments,
            SessionId = _sessionBindings.TryGetValue(bindingKey, out var sessionId) ? sessionId : null,
            WorkingDirectory = request.WorkingDirectory,
            Model = request.Model
        };

        // 3. 执行并收集流式响应
        await foreach (var message in _provider.ExecuteAsync(options, request.Prompt, cancellationToken))
        {
            // 4. 映射 ACP 消息到 AIStreamingChunk
            if (_responseMapper.TryConvertToStreamingChunk(message, out var chunk))
            {
                yield return chunk;
            }
        }
    }
}
```

這裏有幾個關鍵設計點：

1. **會話綁定**: 透過 `CessionId` 將多個請求綁定到同一個 Hermes 子處理序，實作多輪對話的上下文連續性
2. **回應映射**: 將 Hermes ACP 訊息格式轉換爲統一的 `AIStreamingChunk` 格式
3. **流式處理**: 使用 `IAsyncEnumerable` 支援真正的流式回應

其實會話綁定這事兒，就像人和人之間的關係一樣。一旦建立了聯繫，後續的交流就有了上下文，不用每次都從頭開始。只是這關係要維護好，不然斷了就斷了。

### ACP 通訊協定適配

Hermes 使用 ACP（Agent Communication Protocol）通訊協定，與傳統的 HTTP API 不同。ACP 是基於標準輸入輸出的通訊協定，有幾個特點：

1. **啓動標記**: Hermes 處理序啓動後會輸出 `//ready` 標記
2. **動態認證**: 認證方法不是固定的，需要透過通訊協定協商
3. **會話複用**: 透過 `SessionId` 複用已建立的會話
4. **回應分散**: 完整回應可能分散在多個 `session/update` 通知中

HagiCode 透過 `StdioAcpTransport` 處理這些特性：

```csharp
public class StdioAcpTransport
{
    public async Task InitializeAsync(CancellationToken cancellationToken)
    {
        // 等待 //ready 标记
        var readyLine = await _outputReader.ReadLineAsync(cancellationToken);
        if (readyLine != "//ready")
        {
            throw new InvalidOperationException("Hermes did not send ready signal");
        }

        // 发送 initialize 请求
        await SendRequestAsync(new
        {
            jsonrpc = "2.0",
            id = 1,
            method = "initialize",
            @params = new
            {
                protocolVersion = "2024-11-05",
                capabilities = new { },
                clientInfo = new { name = "HagiCode", version = "1.0.0" }
            }
        }, cancellationToken);
    }
}
```

通訊協定這東西，就像人與人之間的默契。有了默契，交流就會順暢很多。只是建立默契需要時間，磨合嘛，誰都免不了。

### 會話池管理

頻繁啓動 Hermes 子處理序的開銷很大，因此我們實作了會話池機制：

```csharp
services.AddSingleton(static _ =>
{
    var registry = new CliProviderPoolConfigurationRegistry();
    registry.Register("hermes", new CliPoolSettings
    {
        MaxActiveSessions = 50,
        IdleTimeout = TimeSpan.FromMinutes(10)
    });
    return registry;
});
```

會話池的關鍵參數：
- `MaxActiveSessions`: 控制並行上限，避免資源耗盡
- `IdleTimeout`: 空閒超時，平衡啓動成本和記憶體佔用

實踐中我們發現：
1. 空閒超時設定太短會導致頻繁重啓，設定太長會佔用記憶體
2. 並行上限需要根據實際負載調整，過大可能導致系統卡頓
3. 需要監控會話池的使用情況，以便及時調整參數

這就好比人生中的許多選擇，太激進容易出問題，太保守又錯失機會。找個平衡點罷了。

### 契約同步

前後端透過 OpenAPI 產生保持契約一致。後端定義了 `AIProviderType` 列舉：

```csharp
public enum AIProviderType
{
    Unknown,
    ClaudeCode,
    OpenCode,
    HermesCli  // 新增
}
```

前端透過 OpenAPI 產生對應的 TypeScript 型別，確保列舉值一致。這是避免前端顯示 "Unknown" 的關鍵。

契約這東西，就像承諾一樣。說好了就要做到，不然就會出現"Unknown"這種尷尬的局面。

## 設定管理

Hermes 的設定透過 `appsettings.json` 管理：

```json
{
  "Providers": {
    "HermesCli": {
      "ExecutablePath": "hermes",
      "Arguments": "acp",
      "StartupTimeoutMs": 10000,
      "ClientName": "HagiCode",
      "Authentication": {
        "PreferredMethodId": "api-key",
        "MethodInfo": {
          "api-key": "your-api-key-here"
        }
      },
      "SessionDefaults": {
        "Model": "claude-sonnet-4-20250514",
        "ModeId": "default"
      }
    }
  }
}
```

這種設定驅動程式的設計帶來了靈活性：
- 可以覆寫可執行檔案路徑，方便開發測試
- 可以自定義啓動參數，適配不同版本的 Hermes
- 可以設定認證資訊，支援多種認證方式

設定這東西，就像人生的選擇題。給足了選項，總能找到適合自己的那一個。只是有時候選項太多，也會讓人犯選擇困難症。

## 實踐經驗

### 健康檢查

實作一個可靠的 Provider 需要完善的健康檢查：

```csharp
public async Task<ProviderTestResult> PingAsync(CancellationToken cancellationToken = default)
{
    var response = await ExecuteAsync(new AIRequest
    {
        Prompt = "Reply with exactly PONG.",
        CessionId = null,
        AllowedTools = Array.Empty<string>(),
        WorkingDirectory = ResolveWorkingDirectory(null)
    }, cancellationToken);

    var success = string.Equals(response.Content.Trim(), "PONG", StringComparison.OrdinalIgnoreCase);
    return new ProviderTestResult
    {
        ProviderName = Name,
        Success = success,
        ResponseTimeMs = stopwatch.ElapsedMilliseconds,
        ErrorMessage = success ? null : $"Unexpected Hermes ping response: '{response.Content}'."
    };
}
```

健康檢查需要注意：
1. 使用簡單的測試用例，避免複雜場景
2. 設定合理的超時時間
3. 記錄回應時間，便於效能分析

就像人需要體檢一樣，系統也需要健康檢查。早發現早治療，省得到時候出大問題。

### 常見問題處理

**認證失敗**
- 檢查 `Authentication.PreferredMethodId` 與 Hermes 實際支援的認證方法是否匹配
- 確認認證資訊格式正確（API Key、Bearer Token 等）

**會話超時**
- 增加 `StartupTimeoutMs` 值
- 檢查 MCP 伺服器可達性
- 查看系統資源使用情況

**回應不完整**
- 確保正確聚合 `session/update` 通知和最終結果
- 檢查流式處理的取消邏輯
- 驗證錯誤處理是否完整

**前端顯示 Unknown**
- 確認 OpenAPI 產生已包含 `HermesCli` 列舉值
- 檢查型別映射是否正確
- 清除瀏覽器快取重新產生型別

問題嘛，總會有的。只是遇到問題的時候，別慌，慢慢找原因，總能解決的。畢竟，辦法總比困難多。

### 效能優化建議

1. **使用會話池**: 複用 ACP 子處理序，減少啓動開銷
2. **合理設定超時**: 平衡記憶體和啓動成本
3. **複用會話 ID**: 批量任務使用同一個 `CessionId`
4. **按需設定 MCP**: 避免不必要的工具呼叫

效能這東西，就像生活中的效率。做對了，事半功倍；做錯了，事倍功半。只是找到那個"對"的點，需要經驗和運氣。

## 總結

集成 Hermes Agent 到生產系統需要考慮多個層面的問題：

1. **架構層面**: 設計統一的 Provider 介面，實作可取代的元件架構
2. **通訊協定層面**: 正確處理 ACP 通訊協定的特殊性，如啓動標記、動態認證等
3. **效能層面**: 透過會話池複用資源，平衡啓動成本和記憶體佔用
4. **前端層面**: 確保契約同步，提供一致的視覺體驗

HagiCode 的實踐表明，透過良好的分層設計和設定驅動程式，可以將複雜的 Agent 系統無縫集成到現有架構中。

其實這些道理說起來都挺簡單的，只是真正做起來的時候，總會遇到各種各樣的問題。不過沒關係，問題解決了就是經驗，解決不了就是教訓，都是有價值的東西。

美的事物或人，不一定要佔有，只要她還是美的，自己好好看着她的美就好了。技術也是如此，只要能讓系統變得更好，用什麼框架、什麼通訊協定，其實都沒那麼重要......
