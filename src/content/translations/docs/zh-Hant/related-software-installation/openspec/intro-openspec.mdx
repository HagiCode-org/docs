# OpenSpec 介紹

OpenSpec 是一個透過結構化的提案與規範來協助團隊管理軟體開發流程的規範驅動開發工具。本指南將全面介紹 OpenSpec 的核心概念、工作流程與使用方式。

## 什麼是 OpenSpec？

OpenSpec 是一個用來管理軟體開發過程中提案（Proposals）、規範（Specifications）與變更（Changes）的命令列工具。它將開發工作流程標準化，確保每項功能從構想到實作都經過完整的設計與審核程序。

### 核心理念

OpenSpec 建基於下列核心理念：

- **規範優先**：在撰寫程式碼之前先定義明確的規範
- **結構化協作**：透過標準化的文件格式促進團隊協作
- **可追溯性**：記錄從提案到實作的完整流程
- **品質保證**：透過規範驗證確保程式碼符合設計意圖

### 主要功能

OpenSpec 提供以下主要功能：

1. **提案管理**：建立、審查與追蹤功能提案
2. **規範定義**：撰寫與維護技術規範文件
3. **變更追蹤**：記錄所有經過審查的變更歷史
4. **規範驗證**：確保程式碼變更符合預先定義的規範
5. **團隊協作**：支援多人協作開發與審查流程

## OpenSpec 工作流程

OpenSpec 的典型工作流程包含以下階段：

### 1. 建立提案（Proposal）

當有新功能或重大變更需求時，首先建立一個提案：

```bash
openspec proposal create
```

提案包含：
- **問題描述**：要解決的問題或需求
- **解決方案**：建議的解決方案概述
- **影響範圍**：對現有系統的影響
- **替代方案**：曾考慮的其他方案

### 2. 撰寫規範（Specification）

提案通過後，撰寫詳細的技術規範：

```bash
openspec spec create
```

規範包含：
- **技術設計**：詳細的技術實作方案
- **介面定義**：API、資料結構等介面設計
- **實作細節**：關鍵實作邏輯與演算法
- **測試計畫**：如何驗證實作的正確性

### 3. 建立變更（Change）

開始實作時，建立一個變更用來追蹤開發過程：

```bash
openspec change create
```

變更包含：
- **關聯規範**：此變更實作的規範
- **實作任務**：具體的開發任務清單
- **進度追蹤**：任務完成狀態
- **程式碼變更**：實際的程式碼修改

### 4. 驗證與封存

完成開發後，驗證變更是否符合規範，然後進行封存：

```bash
openspec change validate
openspec change archive
```

## OpenSpec 目錄結構

OpenSpec 會在專案根目錄建立 `openspec/` 目錄，結構如下：

```
openspec/
├── proposals/           # 提案目錄
│   ├── PROPOSAL-001.md
│   └── PROPOSAL-002.md
├── specs/              # 規範目錄
│   ├── SPEC-001.md
│   └── SPEC-002.md
├── changes/            # 變更目錄
│   ├── feature-name/
│   │   ├── proposal.md
│   │   ├── design.md
│   │   ├── specs.md
│   │   └── tasks.md
│   └── another-feature/
└── archived/           # 已封存的變更
    └── feature-name/
```

### 目錄說明

- **proposals/**：儲存所有提案文件，每個提案一個獨立的 Markdown 檔案
- **specs/**：儲存與提案對應的技術規範文件
- **changes/**：儲存進行中的變更，每個變更一個子目錄
- **archived/**：儲存已完成並封存的變更

## 核心命令

### 提案相關命令

```bash
# 建立新提案
openspec proposal create

# 列出所有提案
openspec proposal list

# 查看提案詳細
openspec proposal view <proposal-id>

# 批准提案
openspec proposal approve <proposal-id>
```

### 規範相關命令

```bash
# 建立新規範
openspec spec create

# 列出所有規範
openspec spec list

# 查看規範詳細
openspec spec view <spec-id>

# 更新規範
openspec spec update <spec-id>
```

### 變更相關命令

```bash
# 建立新變更
openspec change create

# 列出所有變更
openspec change list

# 查看變更詳細
openspec change view <change-name>

# 新增任務
openspec change add-task <change-name>

# 驗證變更
openspec change validate <change-name>

# 封存變更
openspec change archive <change-name>
```

### 通用命令

```bash
# 查看版本
openspec --version

# 查看說明
openspec --help

# 查看特定命令說明
openspec <command> --help
```

## 在 HagiCode 專案中使用 OpenSpec

### 專案整合

在 HagiCode 專案中，OpenSpec 已整合到開發工作流程中：

1. **初始化檢查**：啟動精靈會檢查 OpenSpec 是否已安裝
2. **提案驅動開發**：所有新功能都需要通過 OpenSpec 的提案流程
3. **程式碼審查**：變更必須符合 OpenSpec 規範才能合併
4. **文件同步**：OpenSpec 文件與程式碼實作保持同步

### 版本需求

HagiCode 專案要求使用 **OpenSpec >=1.0.0 {'<'}2.0.0** 的版本範圍。

安裝命令：

```bash
npm install -g @fission-ai/openspec@1
```

### 工作流程範例

以下是在 HagiCode 專案中使用 OpenSpec 的完整範例：

#### 步驟 1：建立提案

```bash
cd /path/to/hagicode-project
openspec proposal create
```

在互動式提示中輸入：
- 提案標題：新增使用者驗證功能
- 問題描述：系統需要登入與權限管理
- 解決方案：實作基於 JWT 的驗證系統

#### 步驟 2：撰寫規範

提案獲批後，建立詳細的規範：

```bash
openspec spec create
```

在規範中定義：
- 驗證流程設計
- API 介面定義
- 資料庫表結構
- 安全性考量

#### 步驟 3：建立變更

開始實作：

```bash
openspec change create
```

輸入變更名稱：`add-user-authentication`

#### 步驟 4：新增任務

```bash
openspec change add-task add-user-authentication
```

新增具體任務：
- 實作使用者註冊端點
- 實作登入端點
- 實作 JWT 中介軟體
- 撰寫單元測試

#### 步驟 5：實作與驗證

完成程式碼後，驗證變更：

```bash
openspec change validate add-user-authentication
```

#### 步驟 6：封存

驗證通過後，封存變更：

```bash
openspec change archive add-user-authentication
```

## 最佳實踐

### 撰寫提案

1. **清楚的問題陳述**：明確定義要解決的問題，避免模糊描述
2. **考慮替代方案**：列出並比較不同的解決方案
3. **評估影響**：分析對現有系統的影響與風險
4. **保持簡潔**：提案應簡明扼要，詳細內容留給規範

### 撰寫規範

1. **詳盡但不冗贅**：提供足夠的細節，但避免過度設計
2. **使用圖表**：用流程圖、架構圖等輔助說明
3. **定義介面**：清楚定義所有公共介面與資料結構
4. **考慮邊界情況**：列出例外情況與錯誤處理

### 變更管理

1. **小步迭代**：將大型功能拆分為多個小變更
2. **即時更新**：保持任務狀態與文件同步
3. **程式碼審查**：變更完成後進行程式碼審查
4. **即時封存**：完成的變更應及時封存以保持工作目錄整潔

### 團隊協作

1. **定期審查**：定期審查提案與規範以確保團隊共識
2. **文件為先**：先完成文件再開始編碼
3. **保持溝通**：在提案與規範中記錄討論與決策
4. **版本控制**：將 OpenSpec 文件納入版本控制

## 常見使用情境

### 情境 1：新增功能

1. 建立描述新功能需求的提案
2. 團隊討論並批准提案
3. 撰寫詳細技術規範
4. 建立變更並實作功能
5. 驗證並封存變更

### 情境 2：重構既有程式碼

1. 建立說明重構原因與目標的提案
2. 在規範中定義重構方案與步驟
3. 建立變更並逐步重構
4. 確保測試涵蓋並驗證重構結果
5. 封存變更

### 情境 3：修復 Bug

對於簡單的 Bug：
- 可以直接修復，無需完整的 OpenSpec 流程

對於複雜的 Bug：
1. 建立分析 Bug 根本原因的提案
2. 在規範中定義修復方案
3. 建立變更並實作修復
4. 新增測試以防止回歸
5. 封存變更

### 情境 4：技術債務管理

1. 建立列出技術債務項目的提案
2. 在規範中定義改進計畫
3. 建立變更逐步償還技術債務
4. 追蹤進度與效果
5. 封存完成的改進

## 與其他工具整合

### Git 整合

OpenSpec 與 Git 無縫整合：

```bash
# OpenSpec 文件應納入版本控制
git add openspec/
git commit -m "Add OpenSpec proposal for feature X"
```

### CI/CD 整合

在 CI/CD 流程中驗證 OpenSpec 規範：

```yaml
# .github/workflows/openspec.yml
name: OpenSpec Validation
on: [pull_request]
jobs:
  validate:
    runs-on: ubuntu-latest
    steps:
      - uses: actions/checkout@v2
      - name: Install OpenSpec
        run: npm install -g @fission-ai/openspec@1
      - name: Validate changes
        run: openspec change validate
```

### IDE 整合

建議使用支援 Markdown 的編輯器編輯 OpenSpec 文件：
- VS Code + Markdown 外掛
- JetBrains 系列 IDE + Markdown 支援
- Vim/Neovim + Markdown 外掛

## 疑難排解

### 常見問題

#### 問題：找不到 OpenSpec 命令

**解決方案**：
1. 確認 OpenSpec 已安裝：`npm list -g @fission-ai/openspec`
2. 檢查 PATH 環境變數是否包含 npm 全域 bin 目錄
3. 重新安裝：`npm install -g @fission-ai/openspec@1`

#### 問題：無法建立提案或變更

**解決方案**：
1. 確認目前目錄為專案根目錄
2. 檢查是否存在 `openspec/` 目錄，如無則建立
3. 確認具有寫入權限

#### 問題：驗證失敗

**解決方案**：
1. 檢查變更目錄下的文件是否完整
2. 確認所有必要的任務都已完成
3. 查看詳細錯誤資訊：`openspec change validate <change-name> --verbose`

#### 問題：封存失敗

**解決方案**：
1. 確保變更已通過驗證
2. 檢查 `openspec/archived/` 目錄是否存在
3. 確認沒有未提交的程式碼變更

## 進階用法

### 自訂模板

OpenSpec 支援自訂提案與規範模板：

```bash
# 在專案根目錄建立模板目錄
mkdir -p .openspec/templates

# 建立自訂提案模板
cat > .openspec/templates/proposal.md << 'EOF'
# Proposal: {{title}}

## Problem Statement
{{problem}}

## Proposed Solution
{{solution}}

## Custom Section
Your custom content here
EOF
```

### 設定檔

在專案根目錄建立 `.openspec.json` 設定檔：

```json
{
  "version": "1.0",
  "templates": {
    "proposal": ".openspec/templates/proposal.md",
    "spec": ".openspec/templates/spec.md"
  },
  "validation": {
    "requireTests": true,
    "requireDocs": true
  },
  "hooks": {
    "pre-archive": "npm test"
  }
}
```

### 鉤子（Hooks）

OpenSpec 支援在特定操作前後執行自訂腳本：

```json
{
  "hooks": {
    "pre-create": "scripts/check-permissions.sh",
    "post-archive": "scripts/notify-team.sh"
  }
}
```
