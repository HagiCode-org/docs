## Panoramica

Monospecs è l'approccio di gestione multi-repository di HagiCode. Tiene traccia e gestisce i sottorepository tramite `monospecs.yaml`e funziona per qualsiasi progetto che necessiti di un unico posto per coordinare più repository indipendenti.

### Concetti fondamentali

**File di configurazione**

`monospecs.yaml` costituisce il cuore dell’intera soluzione gestionale, definendo:
- Elenco e posizione dei sottorepository
- URL Git per ogni repository
- Visualizza nome e icona
- Comportamento di integrazione con OpenSpec

**Differenze rispetto ai repository normali**

| Caratteristica | Deposito regolare | Progetto gestito da Monospecs |
|----------|-------------------|---------------------------|
| **Posizione OpenSpec** | Directory principale `openspec/` | Directory root del repository principale `openspec/` |
| **Cambia ambito** | Limitato al repository corrente | Può coinvolgere più sottorepository |
| **Purezza del sottorepository** | Specifiche mescolate con codice | Le specifiche stabilite nel repository principale, i sottorepository sono più puliti |
| **Archivio automatico dell'archivio** | Richiede commit delle specifiche manuali | `commit_when_archive: true` impegna automaticamente le specifiche nel repository principale |

### Perché utilizzare Monospec?

- **Gestione unificata**: gestisci tutti i sottorepository in un unico file di configurazione
- **Supporto per l'automazione**: fornisce metadati del repository per script e strumenti
- **Funzionalità AI**: la configurazione strutturata aiuta l'agente AI a comprendere le relazioni tra repository
- **Integrazione OpenSpec**: integrazione perfetta con il flusso di lavoro di gestione delle modifiche
- **Sottorepository più puliti**: le specifiche vengono conservate nel repository principale, i sottorepository contengono solo il codice effettivo

### Scenari applicabili

Utilizza prima Monospecs in scenari come questi:

**1. Sviluppo collaborativo multi-repository**
Quando un progetto è suddiviso in più repository indipendenti:
- Applicazione frontend, servizio backend e app desktop sviluppati separatamente
- Necessità di coordinare versioni e dipendenze tra repository
- Documentazione e progetti di supporto separati dal codice principale

**2. Gestione progetti ausiliari**
Utilizza Monospecs per gestire:
- Progetto di documentazione
- Sito ufficiale
- Costruisci strumenti e script
- Ogni repository mantiene il proprio controllo della versione

**3. Sviluppo di funzionalità tra repository**
Una funzionalità che richiede modifiche su più repository:
- È necessario tenere traccia delle modifiche in quali repository
- Esegui automaticamente il commit delle specifiche nel repository di destinazione corretto
- Coordinare gli aggiornamenti delle versioni e le dipendenze

## Avvio rapido

### Inizializzazione

Seguire questi passaggi per completare l'inizializzazione di monospec:

#### 1. Crea un repository git vuoto

Crea un vuoto `git` cartella nel tuo repository principale (nota: questo è `git`, no `repos/` - questa è la convenzione monospecifica):

```bash
mkdir git
cd git
git init
cd ..
```

#### 2. Configurare il file monospecs

Crea il `monospecs.yaml` file di configurazione nella directory principale del repository principale:

```yaml
version: "1.0"
commit_when_archive: true

repositories:
  - path: "repos/frontend"
    url: "https://github.com/your-org/frontend.git"
    displayName: "Frontend App"
    icon: "🌐"

  - path: "repos/backend"
    url: "https://github.com/your-org/backend.git"
    displayName: "Backend Service"
    icon: "⚙️"
```

**Elementi di configurazione**:
| Elemento di configurazione | Digitare | Obbligatorio | Descrizione |
|----------------|------|----------|-------------|
| `version` | stringa | Sì | Versione del file di configurazione |
| `commit_when_archive` | booleano | No | Se eseguire il commit automatico delle specifiche durante l'archiviazione (non impegnare il codice del sub-repo) |
| `repositories` | matrice | Sì | Elenco di configurazione del repository |
| `repositories[].path` | stringa | Sì | Percorso del repository (relativo alla root del progetto) |
| `repositories[].url` | stringa | Sì | URL del repository Git |
| `repositories[].displayName` | stringa | No | Nome visualizzato del repository |
| `repositories[].icon` | stringa | No | Icona del repository (emoji) |

### Aggiungi sottorepository al repository principale

Aggiungi tutti i sottorepository al file `repos/` directory. Questa directory deve essere **esclusa dal controllo della versione Git del repository principale** per evitare di tracciare il contenuto del sottorepository:

```plaintext
# Exclude all sub-repositories
repos/
```

#### 3. Aggiungi il repository principale a HagiCode

Se utilizzi l'applicazione desktop HagiCode, apri l'app e aggiungi il tuo repository principale come progetto. Fai clic sul pulsante "Aggiungi progetto" o "+" nella schermata principale, seleziona la cartella del repository principale e vedrai tutti i sottorepository e le relative informazioni sullo stato.

#### 4. Clonare sottorepository

Usa il `monospecs.yaml` configurazione per clonare tutti i sottorepository configurati nel file `repos/` rubrica:

Ciò clonerà tutti i sottorepository in base agli URL specificati nel file di configurazione.

## Struttura del file di configurazione

### Configurazione a livello di root

```yaml
version: "1.0"
commit_when_archive: true

repositories:
  # ... repository configurations
```

### Configurazione a livello di repository

Ogni sottorepository può avere:
- OpenSpec cambia cartella (`openspec/`)
- Indipendente `.git/` directory
- Facoltativo `AGENTS.md` per la configurazione dell'agente AI

## Operazioni di gestione del repository

### Aggiungi nuovo archivio

Quando si aggiunge un nuovo sottorepository al file `monospecs.yaml` `repositories` matrice:

```yaml
repositories:
  - path: "repos/new-service"
    url: "https://github.com/HagiCode-org/new-service.git"
    displayName: "New Service"
    icon: "🆕"
```

### Aggiorna la configurazione del repository

Quando gli URL o i metadati del repository cambiano:
1. Modifica `monospecs.yaml` per aggiornare le voci corrispondenti
2. Verifica che la sintassi YAML sia corretta
3. Se si sincronizzano le modifiche, aggiornare manualmente la configurazione del repository locale

## Supporto per la configurazione dell'agente AI

### AGENTI.md

Il `AGENTS.md` il file fornisce indicazioni specifiche per il repository per l'agente AI:

**Informazioni sullo stack tecnico**
- Framework e strumenti di creazione
- Convenzioni del codice e pratiche di denominazione
- Gestione delle versioni e configurazione della distribuzione

**Comportamenti specifici del progetto**
- Estensioni della configurazione e requisiti speciali
- Modelli di integrazione OpenSpec
- Esigenze di coordinamento tra repository

### Vantaggi dell'intelligenza artificiale

Utilizzando `AGENTS.md` consente all'agente AI di:
- Comprendere le relazioni e le dipendenze del repository
- Genera codice e refactoring su più repository
- Mantenere la coerenza nel controllo della versione e nelle configurazioni

## Integrazione OpenSpec

### Come funziona Monospec con OpenSpec

Che tu gestisca un progetto gestito da monospecs o un repository regolare, puoi utilizzare il flusso di lavoro OpenSpec di HagiCode:

**Archivio normale:**
- OpenSpec nella radice `openspec/` directory
- Le modifiche riguardano solo il repository corrente

**Progetto gestito Monospecs:**
- OpenSpec legge `monospecs.yaml` configurazione
- Modifiche tracciate dal sottorepository
- Le specifiche vengono automaticamente impegnate sugli obiettivi corretti

### Comportamento di archiviazione

Quando `commit_when_archive: true`:
- Le specifiche vengono automaticamente impegnate nel repository principale
- Il codice dei sottorepository non è impegnato (rimane separato)
- Semplifica la gestione delle specifiche: non è necessario alcun commit manuale delle specifiche per i sottorepository

### Commit manuali del sottorepository

I sottorepository mantengono un controllo della versione indipendente:
- Effettua il commit delle modifiche effettive al codice nei propri repository
- Le specifiche vengono impegnate separatamente (dal repository principale durante l'archiviazione)

## Migliori pratiche

### Convenzioni di denominazione dei repository

- Utilizza kebab-case (lettere minuscole e trattini) per i percorsi dei repository
- Utilizza nomi visualizzati semplici e descrittivi in cinese
- Usa emoji pertinenti come icone

### Linee guida per il nome visualizzato

- Mantienilo conciso e descrittivo
- Utilizza nomi coerenti tra repository simili
- Seleziona le icone che rappresentano lo scopo del repository

### Manutenzione della configurazione

**Sincronizza la configurazione con lo stato effettivo**
- Verificare regolarmente `monospecs.yaml` riflette la struttura effettiva del repository
- Aggiorna o rimuovi le voci quando i repository vengono aggiunti o rimossi
- Testare le modifiche per garantire la validità della configurazione

**Controllo della versione per la configurazione**
- Tieni traccia delle modifiche ai file di configurazione con git
- Documentare le ragioni delle principali modifiche alla configurazione
- Tagga le versioni dei file di configurazione per un facile rollback

## Risoluzione dei problemi

### File di configurazione non trovato

Se il `monospecs.yaml` il file non è stato trovato nella radice del progetto:
1. Controlla se il file esiste nella posizione corretta
2. Verifica di essere nella directory di lavoro corretta
3. Verificare la presenza di errori di battitura nel nome del file di configurazione

### Errori di sintassi YAML

Problemi comuni di sintassi YAML:
- Rientro errato (usa spazi, non tabulazioni)
- Virgolette mancanti o aggiuntive attorno alle stringhe
- Tipi di dati non validi (le stringhe necessitano di virgolette, i numeri no)
- Campi obbligatori mancanti (come `path` o `url`)

### Repository non rilevato

Se un repository appena aggiunto non viene visualizzato:
1. Controlla il `path` è corretto `monospecs.yaml`
2. Verificare che il repository sia stato clonato correttamente
3. Controlla se il repository esiste nella posizione prevista

### Clonazioni fallite

Se la clonazione del repository fallisce:
1. Controlla la connessione di rete
2. Verifica che l'URL del repository Git sia corretto
3. Verifica eventuali problemi di autenticazione
4. Esaminare lo stato del contenitore Docker
5. Controlla la disponibilità dello spazio su disco
