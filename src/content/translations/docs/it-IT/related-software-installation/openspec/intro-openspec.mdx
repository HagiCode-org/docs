# Introduzione a OpenSpec

OpenSpec è uno strumento di sviluppo guidato dalle specifiche che aiuta i team a gestire il processo di sviluppo software attraverso proposte e specifiche strutturate. Questa guida fornisce un'introduzione completa ai concetti fondamentali, ai flussi di lavoro e all'utilizzo di OpenSpec.

## Cos'è OpenSpec?

OpenSpec è uno strumento da riga di comando per gestire Proposals, Specifications e Changes durante l'intero processo di sviluppo software. Standardizza il flusso di lavoro di sviluppo, garantendo che ogni funzionalità passi attraverso una progettazione e revisione approfondite dall'idea all'implementazione.

### Filosofia centrale

OpenSpec si basa sui seguenti principi fondamentali:

- **Specifica prima**: Definire specifiche chiare prima di scrivere codice
- **Collaborazione strutturata**: Promuovere la collaborazione del team attraverso formati di documenti standardizzati
- **Tracciabilità**: Registrare il processo completo dalla proposta all'implementazione
- **Garanzia di qualità**: Assicurare che il codice sia conforme all'intento progettuale attraverso la validazione delle specifiche

### Funzionalità principali

OpenSpec fornisce le seguenti funzionalità principali:

1. **Gestione delle proposte**: Creare, rivedere e tracciare le proposte di funzionalità
2. **Definizione delle specifiche**: Scrivere e mantenere documenti di specifica tecnica
3. **Tracciamento delle modifiche**: Registrare tutta la cronologia delle modifiche revisionate
4. **Validazione delle specifiche**: Assicurare che le modifiche al codice rispettino le specifiche predefinite
5. **Collaborazione del team**: Supportare processi di sviluppo e revisione collaborativi

## Flusso di lavoro di OpenSpec

Il tipico flusso di lavoro di OpenSpec include le seguenti fasi:

### 1. Creare una proposta

Quando c'è una nuova funzionalità o un cambiamento importante, crea prima una proposta:

```bash
openspec proposal create
```

Una proposta contiene:
- **Descrizione del problema**: Il problema o il requisito da risolvere
- **Soluzione**: Panoramica della soluzione proposta
- **Ambito di impatto**: Impatto sui sistemi esistenti
- **Alternative**: Altre soluzioni considerate

### 2. Scrivere una specifica

Dopo l'approvazione della proposta, scrivi una specifica tecnica dettagliata:

```bash
openspec spec create
```

Una specifica contiene:
- **Progettazione tecnica**: Piano di implementazione tecnica dettagliato
- **Definizione delle interfacce**: API, strutture dati e altre definizioni di interfaccia
- **Dettagli di implementazione**: Logica di implementazione chiave e algoritmi
- **Piano di test**: Come verificare la correttezza dell'implementazione

### 3. Creare una modifica

All'inizio dell'implementazione, crea un change per tracciare il processo di sviluppo:

```bash
openspec change create
```

Un change contiene:
- **Specifica associata**: La specifica che questo change implementa
- **Attività di implementazione**: Elenco specifico delle attività di sviluppo
- **Tracciamento dei progressi**: Stato di completamento delle attività
- **Modifiche al codice**: Modifiche effettive al codice

### 4. Validare e archiviare

Dopo aver completato lo sviluppo, valida se il change è conforme alla specifica, quindi archivialo:

```bash
openspec change validate
openspec change archive
```

## Struttura delle directory di OpenSpec

OpenSpec crea una directory `openspec/` nella radice del progetto con la seguente struttura:

```
openspec/
├── proposals/           # Directory delle proposte
│   ├── PROPOSAL-001.md
│   └── PROPOSAL-002.md
├── specs/              # Directory delle specifiche
│   ├── SPEC-001.md
│   └── SPEC-002.md
├── changes/            # Directory delle modifiche
│   ├── feature-name/
│   │   ├── proposal.md
│   │   ├── design.md
│   │   ├── specs.md
│   │   └── tasks.md
│   └── another-feature/
└── archived/           # Modifiche archiviate
    └── feature-name/
```

### Descrizione delle directory

- **proposals/**: Memorizza tutti i documenti di proposta, un file Markdown indipendente per proposta
- **specs/**: Memorizza i documenti di specifica tecnica corrispondenti alle proposte
- **changes/**: Memorizza le modifiche in corso, una sottodirectory per change
- **archived/**: Memorizza le modifiche completate e archiviate

## Comandi principali

### Comandi per le proposte

```bash
# Creare una nuova proposta
openspec proposal create

# Elencare tutte le proposte
openspec proposal list

# Visualizzare i dettagli di una proposta
openspec proposal view <proposal-id>

# Approvare una proposta
openspec proposal approve <proposal-id>
```

### Comandi per le specifiche

```bash
# Creare una nuova specifica
openspec spec create

# Elencare tutte le specifiche
openspec spec list

# Visualizzare i dettagli di una specifica
openspec spec view <spec-id>

# Aggiornare una specifica
openspec spec update <spec-id>
```

### Comandi per le modifiche

```bash
# Creare un nuovo change
openspec change create

# Elencare tutti i change
openspec change list

# Visualizzare i dettagli di un change
openspec change view <change-name>

# Aggiungere un'attività
openspec change add-task <change-name>

# Validare un change
openspec change validate <change-name>

# Archiviare un change
openspec change archive <change-name>
```

### Comandi generali

```bash
# Visualizzare la versione
openspec --version

# Visualizzare l'aiuto
openspec --help

# Visualizzare l'aiuto per un comando specifico
openspec <command> --help
```

## Utilizzo di OpenSpec nei progetti HagiCode

### Integrazione del progetto

Nei progetti HagiCode, OpenSpec è integrato nel flusso di lavoro di sviluppo:

1. **Controllo di inizializzazione**: La procedura guidata di avvio verifica se OpenSpec è installato
2. **Sviluppo guidato dalle proposte**: Tutte le nuove funzionalità devono passare attraverso il processo di proposte di OpenSpec
3. **Revisione del codice**: Le modifiche devono rispettare le specifiche OpenSpec per essere unite
4. **Sincronizzazione della documentazione**: La documentazione OpenSpec rimane sincronizzata con l'implementazione del codice

### Requisiti di versione

I progetti HagiCode richiedono la versione **OpenSpec >=1.0.0 {'<'}2.0.0**.

Comando di installazione:

```bash
npm install -g @fission-ai/openspec@1
```

### Esempio di flusso di lavoro

Ecco un esempio completo di utilizzo di OpenSpec in un progetto HagiCode:

#### Passaggio 1: Creare una proposta

```bash
cd /path/to/hagicode-project
openspec proposal create
```

Inserisci nel prompt interattivo:
- Titolo della proposta: Aggiungere la funzionalità di autenticazione utente
- Descrizione del problema: Il sistema necessita di login e gestione dei permessi
- Soluzione: Implementare un sistema di autenticazione basato su JWT

#### Passaggio 2: Scrivere una specifica

Dopo l'approvazione della proposta, crea una specifica dettagliata:

```bash
openspec spec create
```

Definisci nella specifica:
- Progettazione del flusso di autenticazione
- Definizione dell'interfaccia API
- Struttura delle tabelle del database
- Considerazioni sulla sicurezza

#### Passaggio 3: Creare un change

Inizia l'implementazione:

```bash
openspec change create
```

Inserisci il nome del change: `add-user-authentication`

#### Passaggio 4: Aggiungere attività

```bash
openspec change add-task add-user-authentication
```

Aggiungi attività specifiche:
- Implementare l'endpoint di registrazione utente
- Implementare l'endpoint di login
- Implementare il middleware JWT
- Scrivere test unitari

#### Passaggio 5: Implementare e validare

Dopo aver completato il codice, valida il change:

```bash
openspec change validate add-user-authentication
```

#### Passaggio 6: Archiviare

Dopo la validazione, archivia il change:

```bash
openspec change archive add-user-authentication
```

## Best practice

### Scrivere proposte

1. **Dichiarazione chiara del problema**: Definisci chiaramente il problema da risolvere ed evita descrizioni vaghe
2. **Considerare le alternative**: Elenca e confronta diverse soluzioni
3. **Valutare l'impatto**: Analizza l'impatto e i rischi sui sistemi esistenti
4. **Mantieni la concisione**: Le proposte dovrebbero essere brevi; i dettagli appartengono alla specifica

### Scrivere specifiche

1. **Dettagliato ma non ridondante**: Fornisci dettagli sufficienti, evita l'over-engineering
2. **Usare diagrammi**: Utilizza diagrammi di flusso, diagrammi di architettura, ecc. per aiutare la spiegazione
3. **Definire le interfacce**: Definisci chiaramente tutte le interfacce pubbliche e le strutture dati
4. **Considerare i casi limite**: Elenca situazioni eccezionali e gestione degli errori

### Gestione delle modifiche

1. **Piccole iterazioni**: Suddividi le grandi funzionalità in più piccole modifiche
2. **Aggiornamenti tempestivi**: Mantieni sincronizzati lo stato delle attività e la documentazione
3. **Revisione del codice**: Esegui revisioni del codice dopo il completamento delle modifiche
4. **Archiviazione tempestiva**: Archivia le modifiche completate per mantenere pulita la directory di lavoro

### Collaborazione del team

1. **Revisioni regolari**: Rivedi regolarmente le proposte e le specifiche per garantire il consenso del team
2. **Documentazione prima**: Completa la documentazione prima di iniziare a codificare
3. **Mantenere la comunicazione**: Registra discussioni e decisioni nelle proposte e nelle specifiche
4. **Controllo di versione**: Includi i documenti OpenSpec nel controllo di versione

## Casi d'uso comuni

### Scenario 1: Aggiungere nuove funzionalità

1. Crea una proposta che descriva i requisiti della nuova funzionalità
2. Il team discute e approva la proposta
3. Scrivi una specifica tecnica dettagliata
4. Crea un change e implementa la funzionalità
5. Valida e archivia il change

### Scenario 2: Refactoring del codice esistente

1. Crea una proposta che spieghi le ragioni e gli obiettivi del refactoring
2. Definisci il piano di refactoring e i passaggi nella specifica
3. Crea un change e refactorizza gradualmente
4. Assicura la copertura dei test e valida i risultati del refactoring
5. Archivia il change

### Scenario 3: Correzione di bug

Per bug semplici:
- Possono essere corretti direttamente senza il processo completo di OpenSpec

Per bug complessi:
1. Crea una proposta che analizzi la causa principale del bug
2. Definisci la correzione nella specifica
3. Crea un change e implementa la correzione
4. Aggiungi test per prevenire regressioni
5. Archivia il change

### Scenario 4: Gestione del debito tecnico

1. Crea una proposta che elenchi gli elementi del debito tecnico
2. Definisci un piano di miglioramento nella specifica
3. Crea change per ridurre gradualmente il debito
4. Traccia i progressi e l'efficacia
5. Archivia i miglioramenti completati

## Integrazione con altri strumenti

### Integrazione con Git

OpenSpec si integra perfettamente con Git:

```bash
# I documenti OpenSpec dovrebbero essere inclusi nel controllo di versione
git add openspec/
git commit -m "Add OpenSpec proposal for feature X"
```

### Integrazione CI/CD

Valida le specifiche OpenSpec nelle pipeline CI/CD:

```yaml
# .github/workflows/openspec.yml
name: OpenSpec Validation
on: [pull_request]
jobs:
  validate:
    runs-on: ubuntu-latest
    steps:
      - uses: actions/checkout@v2
      - name: Install OpenSpec
        run: npm install -g @fission-ai/openspec@1
      - name: Validate changes
        run: openspec change validate
```

### Integrazione con IDE

Editor consigliati con supporto Markdown per modificare i documenti OpenSpec:
- VS Code + plugin Markdown
- IDE JetBrains + supporto Markdown
- Vim/Neovim + plugin Markdown

## Risoluzione dei problemi

### Problemi comuni

#### Problema: comando OpenSpec non trovato

**Soluzione**:
1. Verifica che OpenSpec sia installato: `npm list -g @fission-ai/openspec`
2. Controlla che la variabile PATH includa la directory bin globale di npm
3. Reinstalla: `npm install -g @fission-ai/openspec@1`

#### Problema: impossibile creare proposte o change

**Soluzione**:
1. Assicurati che la directory corrente sia la radice del progetto
2. Verifica se esiste la directory `openspec/`; se non esiste, creala
3. Verifica i permessi di scrittura

#### Problema: la validazione fallisce

**Soluzione**:
1. Verifica che i documenti nella directory del change siano completi
2. Assicurati che tutte le attività richieste siano completate
3. Visualizza gli errori dettagliati: `openspec change validate <change-name> --verbose`

#### Problema: l'archiviazione fallisce

**Soluzione**:
1. Assicurati che il change abbia superato la validazione
2. Verifica se esiste la directory `openspec/archived/`
3. Assicurati che non ci siano modifiche al codice non committate

## Utilizzo avanzato

### Template personalizzati

OpenSpec supporta template personalizzati per proposte e specifiche:

```bash
# Crea una directory di template nella radice del progetto
mkdir -p .openspec/templates

# Crea un template di proposta personalizzato
cat > .openspec/templates/proposal.md << 'EOF'
# Proposal: {{title}}

## Problem Statement
{{problem}}

## Proposed Solution
{{solution}}

## Custom Section
Your custom content here
EOF
```

### File di configurazione

Crea un file di configurazione `.openspec.json` nella radice del progetto:

```json
{
  "version": "1.0",
  "templates": {
    "proposal": ".openspec/templates/proposal.md",
    "spec": ".openspec/templates/spec.md"
  },
  "validation": {
    "requireTests": true,
    "requireDocs": true
  },
  "hooks": {
    "pre-archive": "npm test"
  }
}
```

### Hook

OpenSpec supporta l'esecuzione di script personalizzati prima e dopo operazioni specifiche:

```json
{
  "hooks": {
    "pre-create": "scripts/check-permissions.sh",
    "post-archive": "scripts/notify-team.sh"
  }
}
```
