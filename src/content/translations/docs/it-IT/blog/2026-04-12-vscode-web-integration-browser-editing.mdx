## Modifica rapida del codice nel browser: integrazione Web VSCode in pratica

> Dopo che l'intelligenza artificiale ha finito di analizzare il codice, come puoi aprire immediatamente un editor nel browser e iniziare ad apportare modifiche? Questo articolo condivide la nostra esperienza pratica nell'integrazione del server di codice nel progetto HagiCode per creare un ponte senza soluzione di continuità tra l'assistente AI e l'esperienza di modifica del codice.

## Sfondo

Nell'era della programmazione assistita dall'intelligenza artificiale, gli sviluppatori spesso hanno bisogno di ispezionare e modificare rapidamente il codice. Il flusso di lavoro tradizionale è semplice: apri il progetto in un IDE desktop, individua il file, modificalo e salva. Ma in alcune situazioni, quel flusso sembra sempre leggermente fuori posto.

Scenario uno: sviluppo remoto. Quando si utilizza un assistente AI come HagiCode, il backend potrebbe essere in esecuzione su un server remoto o all'interno di un contenitore e le macchine locali non possono accedere direttamente ai file di progetto. Ogni volta che devi ispezionare o modificare il codice, devi connetterti tramite SSH o un altro metodo e l'esperienza sembra frammentata. È come voler incontrare qualcuno attraverso una spessa lastra di vetro: puoi vederlo, ma non puoi raggiungerlo.

Scenario due: anteprime rapide. Dopo che l'assistente AI ha analizzato il codice, l'utente potrebbe voler solo sfogliare rapidamente un file o apportare una piccola modifica. Il lancio di un IDE desktop completo sembra pesante, mentre un editor leggero nel browser si adatta meglio alla necessità di una "rapida occhiata". Dopotutto, chi vorrebbe mobilitare un'intera catena di strumenti solo per dare un'occhiata?

Scenario tre: collaborazione tra dispositivi. Quando si lavora su dispositivi diversi, un editor basato su browser fornisce un punto di accesso unificato senza richiedere che ogni macchina sia configurata con un ambiente di sviluppo. Solo questo risparmia un sacco di problemi. La vita è breve; perché ripetere più e più volte lo stesso lavoro di configurazione?

Per risolvere questi punti critici, abbiamo integrato VSCode Web nel progetto HagiCode. Ciò consente all’assistente AI e all’esperienza di modifica del codice di connettersi perfettamente: dopo che l’AI ha analizzato il codice, gli utenti possono aprire immediatamente un editor e apportare modifiche nella stessa sessione del browser, senza cambiare applicazione. È il tipo di esperienza in cui, quando ne hai bisogno, è semplicemente lì.

## A proposito di HagiCode

La soluzione condivisa in questo articolo deriva dalla nostra esperienza pratica nel [HagiCode](https://hagicode.com) progetto. HagiCode è un assistente di codifica basato sull'intelligenza artificiale progettato per migliorare l'efficienza dello sviluppo attraverso l'interazione del linguaggio naturale. Durante lo sviluppo, abbiamo scoperto che gli utenti spesso hanno bisogno di passare rapidamente dall'analisi dell'intelligenza artificiale alla modifica del codice, il che ci ha spinto a esplorare come integrare l'editor direttamente nel browser.

Archivio del progetto: [github.com/HagiCode-org/site](https://github.com/HagiCode-org/site)

## Scelta tecnica: perché il code-server?

Tra le tante soluzioni Web VSCode disponibili, abbiamo scelto code-server. C’erano alcune ragioni concrete dietro quella decisione.

Completezza delle funzionalità. code-server è la versione web di VSCode e supporta la maggior parte delle funzionalità desktop, incluso il sistema di estensione, suggerimenti intelligenti, debug e altro. Ciò significa che gli utenti possono ottenere un'esperienza di modifica nel browser molto simile alla versione desktop. Dopo tutto, chi vuole davvero scendere a compromessi sulla funzionalità?

Distribuzione flessibile. code-server può essere eseguito come servizio indipendente e supporta anche la distribuzione basata su Docker, che si adatta bene all'architettura di HagiCode. Il nostro backend è scritto in C#, il frontend utilizza React e i due comunicano con il servizio code-server tramite API REST. È come costruire con i mattoncini: ogni pezzo ha il suo posto.

Autenticazione sicura. code-server include un file built-in `connection-token` meccanismo per impedire l’accesso non autorizzato. Ogni sessione ha un token univoco in modo che solo gli utenti autorizzati possano aprire l'editor. La sicurezza è una di quelle cose che apprezzi appieno solo quando ce l’hai.

## Progettazione dell'architettura

L'integrazione Web VSCode di HagiCode utilizza un'architettura separata front-end/back-end.

### Livello di servizio frontend

Il frontend avvolge le interazioni con il backend `vscodeServerService.ts`:

```typescript
// Open project
export async function openProjectInCodeServer(
  id: string,
  currentInterfaceLanguage?: string,
): Promise<VsCodeServerLaunchResponseDto>

// Open vault
export async function openVaultInCodeServer(
  id: string,
  path?: string,
  currentInterfaceLanguage?: string,
): Promise<VsCodeServerLaunchResponseDto>
```

La differenza tra questi due metodi è semplice: `openProjectInCodeServer` apre l'intero progetto, mentre `openVaultInCodeServer` apre un percorso specifico all'interno di un Vault. Per i progetti multi-repository MonoSpecs, il sistema crea automaticamente un file dell'area di lavoro. Spesso sono sufficienti responsabilità chiare quando ciascuna parte svolge bene il proprio lavoro.

### Livello di servizio backend

Il back-end `VaultAppService.cs` implementa la logica di base:

```csharp
public async Task<VsCodeServerLaunchResponseDto> OpenInCodeServerAsync(
    string id,
    string? relativePath = null,
    string? currentInterfaceLanguage = null,
    CancellationToken cancellationToken = default)
{
    // 1. Get settings and check whether the feature is enabled
    var settings = await _vsCodeServerSettingsService.GetResolvedSettingsAsync(cancellationToken);
    if (!settings.Enabled) {
        throw new BusinessException(VsCodeServerErrorCodes.Disabled, "VSCode Server is disabled.");
    }

    // 2. Get vault and resolve the launch directory
    var vault = await RequireVaultAsync(id, cancellationToken);
    var launchDirectory = ResolveLaunchDirectory(vault, relativePath);

    // 3. Ensure code-server is running and get runtime info
    var runtime = await _vsCodeServerManager.EnsureStartedAsync(settings, cancellationToken);

    // 4. Resolve language settings
    var language = _vsCodeServerSettingsService.ResolveLaunchLanguage(
        settings.Language,
        currentInterfaceLanguage);

    // 5. Build launch URL
    return new VsCodeServerLaunchResponseDto {
        LaunchUrl = AppendQueryString(runtime.BaseUrl, new Dictionary<string, string?> {
            ["folder"] = launchDirectory,
            ["tkn"] = runtime.ConnectionToken,
            ["vscode-lang"] = language,
        }),
        ConnectionToken = runtime.ConnectionToken,
        OpenMode = "folder",
        Runtime = VsCodeServerSettingsService.MapRuntime(
            await _vsCodeServerManager.GetRuntimeSnapshotAsync(cancellationToken)),
    };
}
```

Questo metodo ha una responsabilità molto chiara: controllare le impostazioni, risolvere i percorsi, avviare il servizio e creare l'URL. Tra questi, il `ResolveLaunchDirectory` Il metodo esegue controlli di sicurezza del percorso per prevenire attacchi attraversanti il percorso. Il codice può sembrare un po' poesia quando ogni riga ha uno scopo.

### Gestione automatica del tempo di esecuzione

Il backend gestisce il processo del code-server `VsCodeServerManager`:

- Controlla lo stato del processo
- Avvia automaticamente i servizi interrotti
- Restituisce snapshot di runtime come porta, ID processo e ora di inizio

Questa progettazione consente al sistema di gestire automaticamente il ciclo di vita del code-server, in modo che gli utenti non debbano gestire manualmente i processi del servizio. La vita è già abbastanza complicata; tutto ciò che può essere automatizzato dovrebbe esserlo.

### Sincronizzazione della lingua

HagiCode supporta un'interfaccia multilingue e il code-server deve seguire tale impostazione. Il sistema supporta tre modalità linguistiche:

- `follow`: segue la lingua dell'interfaccia corrente
- `zh-CN`: fisso in cinese
- `en-US`: fisso in inglese

L'impostazione viene passata al code-server tramite il file `vscode-lang` Parametro URL in modo che la lingua dell'editor rimanga coerente con l'interfaccia HagiCode. La lingua si sente meglio quando è unificata.

### Area di lavoro multi-repository MonoSpecs

Per i progetti MonoSpecs, che contengono più sottorepository all'interno di un monorepo, il sistema crea automaticamente un file `.code-workspace` cartella:

```csharp

{

    };
    // Generate workspace file...

```

Ciò rende possibile modificare più sottorepository nella stessa istanza del server di codice, il che è particolarmente pratico per progetti monorepo di grandi dimensioni. Più archivi in ​​una finestra possono sembrare più storie raccolte nello stesso libro.

## Integrazione frontend

Il frontend HagiCode utilizza React + TypeScript e l'integrazione del code-server non è particolarmente complicata.

### Pulsante di azione rapida

Aggiungi un pulsante Code Server alla scheda progetto:

```tsx
// QuickActionsZone.tsx
<Button
  size="sm"
  variant="default"
  onClick={() => onAction({ type: 'open-code-server' })}
>
  <Globe className="h-3 w-3 mr-1" />
  <span className="text-xs">{t('project.openCodeServer')}</span>
</Button>
```

Questo pulsante attiva l'azione di apertura e chiama l'API di backend per ottenere l'URL di avvio. Un pulsante, un'azione, diretta e semplice.

### Gestire l'azione aperta

```tsx
const handleAction = async (action: ProjectAction) => {
  if (action.type === 'open-code-server') {
    const response = await openProjectInCodeServer(project.id, i18n.language);
    window.open(response.launchUrl, '_blank', 'noopener,noreferrer');
  }
};
```

Utilizzare `window.open` per aprire il code-server in una nuova scheda. Il `noopener,noreferrer` i parametri forniscono una maggiore sicurezza. Quando si parla di sicurezza, non esiste la prudenza.

### Voce di modifica del vault

Aggiungi un pulsante di modifica simile nell'elenco del Vault:

```tsx
const handleEditVault = async (vault: VaultItemDto) => {
  const response = await openVaultInCodeServer(vault.id);
  window.open(response.launchUrl, '_blank', 'noopener,noreferrer');
};
```

Progetti e Vault utilizzano lo stesso meccanismo aperto, che mantiene coerente l'interazione. La coerenza conta quasi quanto la funzionalità stessa.

## Logica di costruzione dell'URL

Il formato dell'URL per il code-server presenta alcuni dettagli degni di nota.

**Modalità cartella**:
```
http://{host}:{port}/?folder={path}&tkn={token}&vscode-lang={lang}
```

**Modalità area di lavoro**:
```
http://{host}:{port}/?workspace={workspacePath}&tkn={token}&vscode-lang={lang}
```

qui, `tkn` è il token di connessione. Viene generato automaticamente ogni volta che si avvia il code-server, garantendo un accesso sicuro. Il `vscode-lang` Il parametro controlla la lingua dell'interfaccia utente dell'editor. Ognuno di questi parametri ha un ruolo da svolgere.

Un utente crea un Vault per studiare un progetto open source e desidera aggiungere note di studio sotto il file `docs/` directory. ## Considerazioni sulla sicurezza

Quando si implementa l'integrazione del code-server, la sicurezza merita un'attenzione speciale. Se la sicurezza va male, te ne accorgi sempre troppo tardi.

### Gettone di connessione

Il `connection-token` viene generato in modo casuale e non deve essere esposto. È meglio utilizzarlo in HTTPS per impedire che il token venga intercettato da un man-in-the-middle. Vale la pena proteggere adeguatamente le informazioni sensibili.

### Sicurezza del percorso dei file

Il backend implementa i controlli di attraversamento del percorso:

```csharp
private static string ResolveLaunchDirectory(VaultRegistryEntry vault, string? relativePath)
{
    var vaultRoot = EnsureTrailingSeparator(Path.GetFullPath(vault.PhysicalPath));
    var combinedPath = Path.GetFullPath(Path.Combine(vaultRoot, relativePath ?? "."));
    if (!combinedPath.StartsWith(vaultRoot, StringComparison.OrdinalIgnoreCase))
    {
        throw new BusinessException(VaultRelativePathTraversalCode, "Relative path traversal detected.");
    }
    return combinedPath;
}
```

Questo codice garantisce che gli utenti non possano utilizzarlo `../` o modelli simili per accedere ai file all'esterno della directory del Vault. I controlli sui confini sono sempre meglio fatti che saltati.

### Controllo dei permessi

Il processo del server di codice deve essere eseguito con le autorizzazioni utente appropriate in modo che non possa accedere a file di sistema sensibili. È meglio eseguire il servizio code-server con un utente dedicato. Il controllo delle autorizzazioni è uno di quegli aspetti fondamentali che dovresti sempre tenere a posto.

## Ottimizzazione delle prestazioni

code-server consuma risorse del server, quindi ecco alcuni suggerimenti di ottimizzazione:

- Monitora l'utilizzo della CPU e della memoria e regola i limiti delle risorse quando necessario
- I progetti di grandi dimensioni potrebbero richiedere timeout più lunghi
- Implementare la pulizia automatica del timeout della sessione per rilasciare le risorse
- Prendi in considerazione la memorizzazione nella cache per ridurre i calcoli ripetuti

HagiCode fornisce un'API di monitoraggio dello stato di runtime e il frontend può chiamare `getVsCodeServerSettings()` per recuperare lo stato corrente:

```typescript
const { settings, runtime } = await getVsCodeServerSettings();
// runtime.status: 'disabled' | 'stopped' | 'starting' | 'running' | 'unhealthy'
// runtime.baseUrl: "http://localhost:8080"
// runtime.processId: 12345
```

Questo design consente agli utenti di comprendere chiaramente lo stato di integrità del code-server e di individuare rapidamente i problemi quando qualcosa va storto. Quando lo stato è visibile, le persone si sentono più in controllo.

## Dettagli sull'esperienza dell'utente

Durante l'implementazione, abbiamo scoperto alcuni dettagli che influenzano notevolmente l'esperienza dell'utente e meritano particolare attenzione.

L'apertura del code-server per la prima volta potrebbe richiedere l'attesa dell'avvio e il ritardo può variare da pochi secondi a mezzo minuto. È una buona idea mostrare uno stato di caricamento nel frontend in modo che gli utenti sappiano che il sistema sta ancora funzionando. L'attesa è più facile quando c'è un feedback.

I browser potrebbero bloccare il popup, quindi agli utenti dovrebbe essere richiesto di consentirlo manualmente. Al primo avvio, HagiCode visualizza una guida che spiega come concedere le autorizzazioni del browser necessarie. L'esperienza dell'utente spesso vive proprio in questi piccoli dettagli.

È anche una buona idea visualizzare lo stato del runtime, ad esempio avvio, esecuzione o errore, in modo che quando si verificano problemi, gli utenti possano capire rapidamente se il problema è sul lato server o nelle proprie operazioni. Sapere dove si trova il problema almeno ti dà un punto di partenza.

## Esempio di configurazione

La configurazione per code-server non è complicata:

```json
{
  "vscodeServer": {
    "enabled": true,
    "host": "0.0.0.0",
    "port": 8080,
    "language": "follow"
  }
}
```

`enabled` controlla se la funzione è attivata, `host` e `port` definire l'indirizzo di ascolto e `language` imposta la modalità lingua. Queste impostazioni possono essere modificate tramite l'interfaccia utente e hanno effetto immediato. Le cose semplici sono spesso le più facili da usare.
