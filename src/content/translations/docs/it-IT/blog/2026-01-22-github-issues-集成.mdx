## Costruire un'integrazione dei problemi GitHub da zero: pratica di connessione diretta del frontend di HagiCode

> Questo articolo documenta l'intero processo di integrazione di GitHub Issues nella piattaforma HagiCode. Esploreremo come utilizzare un'architettura "connessione diretta frontend + backend minimo" per ottenere un'autenticazione OAuth sicura e un'efficiente sincronizzazione dei problemi mantenendo il backend leggero.

## Background: perché integrare GitHub?

In quanto piattaforma di sviluppo assistita dall'intelligenza artificiale, il valore fondamentale di HagiCode risiede nel collegare le idee con l'implementazione. Ma nell'uso reale, abbiamo scoperto che dopo che gli utenti hanno completato una proposta in HagiCode, spesso devono copiare manualmente il contenuto in GitHub Issues per il monitoraggio del progetto.

Ciò crea diversi evidenti punti dolenti:

1.  **Flusso di lavoro frammentato**: gli utenti devono alternare tra due sistemi. L'esperienza non è fluida e le informazioni chiave possono essere facilmente perse durante il copia e incolla.
2.  **Collaborazione scomoda**: gli altri membri del team sono abituati a controllare le attività su GitHub e non possono vedere direttamente lo stato di avanzamento della proposta all'interno di HagiCode.
3.  **Lavoro manuale ripetuto**: ogni volta che una proposta viene aggiornata, qualcuno deve aggiornare manualmente il problema corrispondente su GitHub, aggiungendo costi di manutenzione non necessari.

Per risolvere questo problema, abbiamo deciso di introdurre la funzionalità **GitHub Issues Integration**, collegando le sessioni HagiCode con i repository GitHub per abilitare la "sincronizzazione con un clic".

## Scelta tecnica: connessione diretta frontend vs proxy backend

Durante la progettazione dell'approccio di integrazione, avevamo due opzioni davanti a noi: il tradizionale "modello proxy backend" e il più aggressivo "modello di connessione diretta frontend".

### Confronto delle soluzioni

Nel tradizionale **modello proxy backend**, ogni richiesta dal frontend deve prima passare attraverso il nostro backend, che poi chiama l'API GitHub. Ciò centralizza la logica, ma pone anche un onere significativo sul backend:

1.  **Backend gonfio**: dovremmo scrivere un wrapper client API GitHub dedicato e gestire anche la complessa macchina a stati OAuth.
2.  **Rischio token**: il token GitHub dell'utente dovrebbe essere archiviato nel database back-end. Anche con la crittografia, ciò aumenta comunque la superficie di sicurezza.
3.  **Costi di sviluppo**: Avremmo bisogno di migrazioni di database per archiviare i token e di un servizio di sincronizzazione aggiuntivo da mantenere.

Il **modello di connessione diretta del frontend** è molto più leggero. In questo approccio, utilizziamo il backend solo per il passaggio più delicato dello "scambio segreto" (il callback OAuth). Dopo aver ottenuto il token, lo memorizziamo direttamente nel browser `localStorage`. Le operazioni successive come la creazione di problemi e l'aggiornamento dei commenti vengono inviate direttamente dal frontend a GitHub tramite HTTP.

| Dimensione di confronto | Modello proxy back-end | Modello di connessione diretta frontend |
| :--- | :--- | :--- |
| **Complessità del backend** | Richiede un servizio OAuth completo e un client API GitHub | È necessario solo un endpoint di callback OAuth |
| **Gestione dei token** | Deve essere crittografato e archiviato nel database, con rischio di fuga di dati | Memorizzato nel browser e visibile solo all'utente |
| **Costo di implementazione** | Richiede migrazioni di database e sviluppo multiservizio | Principalmente lavoro di frontend |
| **Esperienza utente** | Logica centralizzata, ma la latenza del server potrebbe essere leggermente superiore | Risposta estremamente rapida con interazione diretta su GitHub |

Poiché volevamo un'integrazione rapida e modifiche minime al backend, **alla fine abbiamo scelto il "modello di connessione diretta frontend"**. È come dare al browser un "passaggio temporaneo". Una volta ottenuto il pass, il browser può gestire le cose su GitHub da solo senza chiedere ogni volta l'approvazione all'amministratore del backend.

## Progettazione principale: flusso di dati e sicurezza

Dopo aver definito l'architettura, dovevamo progettare il flusso di dati specifico. Il fulcro del processo di sincronizzazione è come ottenere il token in modo sicuro e utilizzarlo in modo efficiente.

### Diagramma dell'architettura generale

L'intero sistema può essere astratto in tre ruoli: il browser (frontend), il backend HagiCode e GitHub.

```text
+--------------+        +--------------+        +--------------+
| Frontend     |        |   Backend    |        |   GitHub     |
| React        |        |   ASP.NET    |        |   REST API   |
|              |        |              |        |              |
|  +--------+  |        |              |        |              |
|  | OAuth  |--+--------> /callback    |        |              |
|  | Flow   |  |        |              |        |              |
|  +--------+  |        |              |        |              |
|              |        |              |        |              |
|  +--------+  |        |  +--------+  |        |  +--------+  |
|  | GitHub |  +------------> Session |  +----------> Issues |  |
|  | API    |  |        |  |Metadata|  |        |  |        |  |
|  | Direct |  |        |  +--------+  |        |  +--------+  |
|  +--------+  |        |              |        |              |
+--------------+        +--------------+        +--------------+
```

**Il punto chiave è**: solo un piccolo passaggio in OAuth (lo scambio del codice con un token) deve passare attraverso il backend. Successivamente, il lavoro pesante (la creazione di problemi) viene gestito direttamente tra il frontend e GitHub.

### Flusso di dati di sincronizzazione in dettaglio

Quando l'utente fa clic sul pulsante "Sincronizza su GitHub" nell'interfaccia utente di HagiCode, hanno luogo una serie di azioni complesse:

```text
User clicks "Sync to GitHub"
         │
         ▼
1. Frontend checks localStorage for the GitHub token
         │
         ▼
2. Format issue content (convert the Proposal into Markdown)
         │
         ▼
3. Frontend directly calls the GitHub API to create/update the issue
         │
         ▼
4. Call the HagiCode backend API to update Session.metadata (store issue URL and other info)
         │
         ▼
5. Backend broadcasts the SessionUpdated event via SignalR
         │
         ▼
6. Frontend receives the event and updates the UI to show the "Synced" state
```

### Progettazione della sicurezza

La sicurezza è sempre la massima priorità quando si integrano servizi di terze parti. Abbiamo fatto le seguenti considerazioni:

1.  **Difenditi dagli attacchi CSRF**: genera un attacco casuale `state` parametro durante il reindirizzamento OAuth e memorizzarlo in `sessionStorage`. Convalida rigorosamente lo stato nel callback per evitare richieste contraffatte.
2.  **Archiviazione isolata di token**: il token viene archiviato solo nel browser `localStorage`. Utilizzando la politica della stessa origine, solo gli script HagiCode possono leggerlo, evitando il rischio di una perdita di database lato server che colpisca gli utenti.
3.  **Limiti di errore**: abbiamo progettato una gestione dedicata per gli errori comuni dell'API GitHub (come token scaduto 401, errore di convalida 422 e limitazione della velocità 429), in modo che gli utenti ricevano un feedback amichevole.

## In pratica: dettagli di implementazione nel codice

La teoria arriva solo fino a un certo punto. Vediamo come funziona effettivamente il codice.

### 1. Modifiche minime al backend

Il backend deve fare solo due cose: archiviare le informazioni di sincronizzazione e gestire il callback OAuth.

**Modifiche al database**
Dobbiamo solo aggiungere a `Metadata` colonna al `Sessions` tabella per archiviare i dati dell'estensione in formato JSON.

### 2. Flusso OAuth frontend

Questo è il punto di ingresso della connessione. Utilizziamo il flusso del codice di autorizzazione standard.

```typescript
// src/HagiCode.Client/src/services/githubOAuth.ts

// Generate the authorization URL and redirect
export async function generateAuthUrl(): Promise<string> {
  const state = generateRandomString(); // Generate a random string for CSRF protection
  sessionStorage.setItem('hagicode_github_state', state);
  
  const params = new URLSearchParams({
    client_id: clientId,
    redirect_uri: window.location.origin + '/settings?tab=github&oauth=callback',
    scope: ['repo', 'public_repo'].join(' '),
    state: state,
  });
  
  return `https://github.com/login/oauth/authorize?${params.toString()}`;
}

// Handle the code-to-token exchange on the callback page
export async function exchangeCodeForToken(code: string, state: string): Promise<GitHubToken> {
  // 1. Validate state to prevent CSRF
  const savedState = sessionStorage.getItem('hagicode_github_state');
  if (state !== savedState) throw new Error('Invalid state parameter');

  // 2. Call the backend API to exchange the token
  // Note: this must go through the backend because ClientSecret cannot be exposed to the frontend
  const response = await fetch('/api/GitHubOAuth/callback', {
    method: 'POST',
    headers: { 'Content-Type': 'application/json' },
    body: JSON.stringify({ code, state, redirectUri: window.location.origin + '/settings?tab=github&oauth=callback' }),
  });

  if (!response.ok) throw new Error('Failed to exchange token');
  
  const token = await response.json();
  
  // 3. Save into LocalStorage
  saveToken(token);
  return token;
}
```

### 3. Wrapper client API GitHub

Una volta ottenuto il token, abbiamo bisogno di uno strumento solido per chiamare l'API GitHub.

```typescript
// src/HagiCode.Client/src/services/githubApiClient.ts

const GITHUB_API_BASE = 'https://api.github.com';

// Core request wrapper
async function githubApi<T>(endpoint: string, options: RequestInit = {}): Promise<T> {
  const token = localStorage.getItem('gh_token');
  if (!token) throw new Error('Not connected to GitHub');
  
  const response = await fetch(`${GITHUB_API_BASE}${endpoint}`, {
    ...options,
    headers: {
      ...options.headers,
      Authorization: `Bearer ${token}`,
      Accept: 'application/vnd.github.v3+json', // Specify the API version
    },
  });
  
  // Error handling logic
  if (!response.ok) {
    if (response.status === 401) throw new Error('GitHub token expired, please reconnect');
    if (response.status === 403) throw new Error('No permission to access this repository or rate limit exceeded');
    if (response.status === 422) throw new Error('Issue validation failed, the title may be duplicated');
    throw new Error(`GitHub API Error: ${response.statusText}`);
  }
  
  return response.json();
}

// Create issue
export async function createIssue(owner: string, repo: string, data: { title: string, body: string, labels: string[] }) {
  return githubApi(`/repos/${owner}/${repo}/issues`, {
    method: 'POST',
    body: JSON.stringify(data),
  });
}
```

### 4. Formattazione e sincronizzazione dei contenuti

Il passaggio finale è convertire i dati della sessione HagiCode nel formato di un problema GitHub. È un po’ come il lavoro di traduzione.

```typescript
// Convert a Session object into a Markdown string
function formatIssueForSession(session: Session): string {
  let content = `# ${session.title}\n\n`;
  content += `**> HagiCode Session:** #${session.code}\n`;
  content += `**> Status:** ${session.status}\n\n`;
  content += `## Description\n\n${session.description || 'No description provided.'}\n\n`;
  
  // If this is a Proposal session, add extra fields
  if (session.type === 'proposal') {
    content += `## Chief Complaint\n\n${session.chiefComplaint || ''}\n\n`;
    // Add a deep link so users can jump back from GitHub to HagiCode
    content += `---\n\n**[View in HagiCode](hagicode://sessions/${session.id})**\n`;
  }
  
  return content;
}

// Main logic when clicking the sync button
const handleSync = async (session: Session) => {
  try {
    const repoInfo = parseRepositoryFromUrl(session.repoUrl); // Parse the repository URL
    if (!repoInfo) throw new Error('Invalid repository URL');

    toast.loading('Syncing to GitHub...');
    
    // 1. Format content
    const issueBody = formatIssueForSession(session);
    
    // 2. Call API
    const issue = await githubApiClient.createIssue(repoInfo.owner, repoInfo.repo, {
      title: `[HagiCode] ${session.title}`,
      body: issueBody,
      labels: ['hagicode', 'proposal', `status:${session.status}`],
    });
    
    // 3. Update Session Metadata (save the issue link)
    await SessionsService.patchApiSessionsSessionId(session.id, {
      metadata: {
        githubIssue: {
          owner: repoInfo.owner,
          repo: repoInfo.repo,
          issueNumber: issue.number,
          issueUrl: issue.html_url,
          syncedAt: new Date().toISOString(),
        }
      }
    });

    toast.success('Synced successfully!');
  } catch (err) {
    console.error(err);
    toast.error('Sync failed, please check your token or network');
  }
};
```

## Riepilogo e prospettive

Con questo approccio di "connessione diretta frontend", abbiamo ottenuto un'integrazione perfetta di GitHub Issues con il minor codice backend possibile.

### Guadagni chiave
1.  **Elevata efficienza di sviluppo**: le modifiche al backend sono minime, principalmente un campo di database aggiuntivo e un semplice endpoint di callback OAuth. La maggior parte della logica viene completata sul frontend.
2.  **Forte sicurezza**: il token non passa attraverso il database del server, riducendo il rischio di perdite.
3.  **Ottima esperienza utente**: le richieste vengono avviate direttamente dal frontend, quindi la velocità di risposta è elevata e non è necessario l'inoltro al backend.

### Cose a cui prestare attenzione
Ci sono alcune insidie ​​da tenere a mente durante la distribuzione reale:
- **Impostazioni app OAuth**: ricordati di inserire i dati corretti `Authorization callback URL` nelle impostazioni dell'app GitHub OAuth (solitamente `http://localhost:3000/settings?tab=github&oauth=callback`).
- **Limiti di velocità**: l'API GitHub limita le richieste non autenticate in modo abbastanza rigoroso, ma con un token la quota è generalmente sufficiente (5000 richieste/ora).
- **Analisi URL**: gli utenti inseriscono tutti i tipi di URL del repository, quindi assicurati che la tua espressione regolare possa corrispondere `.git` suffissi, formati SSH e casi simili.

### Miglioramenti futuri
La funzionalità attuale è ancora la sincronizzazione unidirezionale (HagiCode -> GitHub). In futuro, prevediamo di implementare la sincronizzazione bidirezionale tramite GitHub Webhooks. Ad esempio, se un problema viene chiuso su GitHub, anche lo stato della sessione sul lato HagiCode potrebbe aggiornarsi automaticamente. Ciò richiederà di esporre un endpoint webhook sul backend, che sarà un passo successivo interessante.
