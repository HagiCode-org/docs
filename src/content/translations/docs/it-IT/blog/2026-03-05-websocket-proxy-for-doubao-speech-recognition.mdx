## Risoluzione delle sfide di autenticazione WebSocket del browser: un modello proxy pratico per il riconoscimento vocale Doubao

> L'API WebSocket del browser non supporta le intestazioni HTTP personalizzate, il che crea una sfida per i servizi di riconoscimento vocale che richiedono dati di autenticazione nelle intestazioni. Questo articolo spiega come abbiamo risolto il problema nel progetto HagiCode con un modello proxy backend e come l'approccio si è evoluto dagli esperimenti nel parco giochi all'uso in produzione.


## Sfondo

Quando abbiamo iniziato a creare il riconoscimento vocale per il progetto HagiCode, abbiamo scelto con fiducia il servizio di riconoscimento vocale Doubao di ByteDance. Il progetto iniziale era semplice: lasciare che il frontend si connettesse direttamente al servizio WebSocket di Doubao. Quanto potrebbe essere difficile? Basta aprire una connessione e inviare alcuni dati, giusto?

Poi è arrivata la sorpresa: l'API di Doubao richiede che le informazioni di autenticazione vengano passate attraverso le intestazioni HTTP, incluse cose come `accessToken` e `secretKey`. Ciò è diventato immediatamente imbarazzante, perché l'API WebSocket del browser semplicemente non supporta l'impostazione di intestazioni personalizzate.

Allora cosa fai quando il browser non ti consente di inviarli?

All’epoca valutavamo due opzioni:

1. Inserisci le credenziali nei parametri di query dell'URL: semplice e schietto
2. Aggiungi un livello proxy sul backend: più lavoro a prima vista

La prima opzione espone le credenziali direttamente nel codice frontend e nell'archiviazione locale. È davvero sicuro? Non mi sentivo a mio agio. Oltre a ciò, alcune API richiedono la verifica basata sull'intestazione, quindi questo approccio non è nemmeno praticabile.

Alla fine abbiamo scelto la seconda opzione: implementare un proxy WebSocket sul backend. Per coincidenza, questo modello è stato convalidato per la prima volta nel nostro ambiente di gioco e solo dopo averne confermato la stabilità lo abbiamo messo in produzione. Dopotutto, nessuno vuole che la produzione serva anche da esperimento di laboratorio.

## A proposito di HagiCode

L'approccio condiviso in questo articolo deriva dalla nostra esperienza pratica nel [HagiCode](https://github.com/HagiCode-org/site) progetto.

HagiCode è un progetto di assistente alla codifica AI con supporto dell'interazione vocale. Poiché avevamo bisogno di chiamare un servizio di riconoscimento vocale dal frontend, ci siamo imbattuti direttamente in questo problema di autenticazione WebSocket, che ci ha portato alla soluzione qui descritta. A volte questi ostacoli tecnici sono frustranti, ma ti costringono anche ad apprendere schemi che si rivelano utili in seguito.

## Analisi delle sfide tecniche

### Limitazioni del WebSocket del browser

L'API WebSocket standard sembra meravigliosamente semplice:

```javascript
const ws = new WebSocket('wss://example.com/ws');
```

Ma questa semplicità è proprio il problema: passa solo i parametri nell'URL e non può impostare le intestazioni come fa una richiesta HTTP:

```javascript
// This is not supported in the WebSocket API
const ws = new WebSocket('wss://example.com/ws', {
  headers: {
    'Authorization': 'Bearer token'
  }
});
```

E questa è la questione centrale. Per servizi come il riconoscimento vocale Doubao che dipendono dall'autenticazione basata sull'intestazione, questa limitazione è un grosso ostacolo.

Una volta accettato questo vincolo, l’architettura deve cambiare.

### Decisioni di progettazione architettonica

Durante la progettazione della soluzione, abbiamo confrontato attentamente i compromessi.

**Decisione 1: scelta del modello proxy**

Abbiamo confrontato due approcci:

| Opzione | Pro | Contro | Decisione |
|------|------|------|------|
| WebSocket nativo | Inoltro leggero, semplice e diretto | La gestione della connessione deve essere gestita manualmente | Scelto |
| SegnaleR | Riconnessione automatica, digitazione forte | Dipendenze extra e eccessivamente complesse | Rifiutato |

Alla fine abbiamo scelto WebSocket nativo. Ad essere onesti, era l'opzione più leggera e si adattava meglio al semplice inoltro di flussi binari bidirezionali. L'inserimento di SignalR sarebbe stato considerato un intervento di ingegneria eccessiva per questo caso d'uso e avrebbe potuto aggiungere ulteriore latenza.

**Decisione 2: strategia di gestione della connessione**

Abbiamo adottato un modello "una connessione, una sessione": ciascuna connessione WebSocket frontend viene mappata alla propria connessione backend Doubao indipendente.

I vantaggi sono semplici:
- Semplice da implementare e allineato al modello di utilizzo comune
- Più facile eseguire il debug e risolvere i problemi
- Buon isolamento delle risorse, prevenendo interferenze tra le sessioni

In parole povere, la soluzione diretta a volte è la migliore. La complessità non rende automaticamente migliore un progetto.

**Decisione 3: archiviazione dei dati di autenticazione**

Le credenziali vengono archiviate nei file di configurazione del backend (`appsettings.yml` o variabili d'ambiente) e caricati tramite l'inserimento delle dipendenze:

- Modello di configurazione semplice che corrisponde alle convenzioni di backend esistenti
- I dati sensibili non raggiungono mai il frontend
- Supporta la configurazione multiambiente per sviluppo, test e produzione

Quel livello di separazione è importante. Nessuno vuole che le credenziali vadano in giro in posti dove non dovrebbero essere.

### Progettazione del flusso di dati

Il flusso di dati complessivo è simile al seguente:

```
Frontend (browser)
  │
  │ ws://backend/api/voice/ws
  │ WebSocket (binary)
  ▼
Backend (proxy)
  │
  │ wss://openspeech.bytedance.com/
  │ (with authentication headers)
  ▼
Doubao API
```

Il flusso in sé non è complicato:
1. Il frontend si connette al proxy backend tramite WebSocket
2. Il proxy backend riceve dati audio e si connette all'API Doubao con intestazioni autenticate
3. L'API Doubao restituisce i risultati del riconoscimento e il proxy li inoltra al frontend
4. L'intero processo rimane completamente asincrono con lo streaming bidirezionale

Una volta suddivise chiaramente le responsabilità, la progettazione diventa del tutto naturale.

## Implementazione dei componenti principali

### 1. Configurazione dell'endpoint WebSocket

```csharp
app.Map("/ws", async context =>
{
    if (context.WebSockets.IsWebSocketRequest)
    {
        // Read configuration from query parameters
        var appId = context.Request.Query["appId"];
        var accessToken = context.Request.Query["accessToken"];

        // Validate required parameters
        if (string.IsNullOrEmpty(appId) || string.IsNullOrEmpty(accessToken))
        {
            context.Response.StatusCode = 400;
            return;
        }

        // Accept the WebSocket connection
        using var webSocket = await context.WebSockets.AcceptWebSocketAsync();

        // Message handling loop
        var buffer = new byte[4096];
        while (!webSocket.CloseStatus.HasValue)
        {
            var result = await webSocket.ReceiveAsync(buffer, CancellationToken.None);

            if (result.MessageType == WebSocketMessageType.Close)
            {
                await webSocket.CloseAsync(
                    result.CloseStatus.Value,
                    result.CloseStatusDescription,
                    CancellationToken.None);
                break;
            }

            // Process audio data
            await HandleAudioDataAsync(buffer, result.Count);
        }
    }
});
```

### 2. Gestione della sessione

```csharp
public class DoubaoSessionManager : IDoubaoSessionManager
{
    private readonly ConcurrentDictionary<string, DoubaoSession> _sessions = new();

    public DoubaoSession CreateSession(string connectionId)
    {
        var session = new DoubaoSession(connectionId);
        _sessions[connectionId] = session;
        return session;
    }

    public async Task SendAudioAsync(string connectionId, byte[] audioData)
    {
        if (_sessions.TryGetValue(connectionId, out var session))
        {
            await session.SendAudioAsync(audioData);
        }
    }

    public void RemoveSession(string connectionId)
    {
        if (_sessions.TryRemove(connectionId, out var session))
        {
            session.Dispose();
        }
    }
}
```

Utilizzando `ConcurrentDictionary` per la gestione della sessione significa che la sicurezza dei thread è in gran parte gestita per noi. Ogni connessione in entrata ottiene la propria sessione e la pulizia avviene automaticamente alla disconnessione.

## Progettazione del protocollo di messaggio

Il frontend e il backend utilizzano messaggi di testo JSON per il controllo, mentre i messaggi binari trasportano dati audio.

Esempio di messaggio di controllo:

```json
{
    "type": "control",
    "messageId": "msg_123",
    "timestamp": "2026-03-03T10:00:00Z",
    "payload": {
        "command": "StartRecognition",
        "parameters": {
            "hotwordId": "hotword1",
            "boosting_table_id": "table123"
        }
    }
}
```

Risultato del riconoscimento di esempio:

```json
{
    "type": "result",
    "timestamp": "2026-03-03T10:00:03Z",
    "payload": {
        "text": "Hello world",
        "confidence": 0.95,
        "duration": 1500,
        "isFinal": true,
        "utterances": [
            {
                "text": "Hello",
                "startTime": 0,
                "endTime": 800,
                "definite": true
            }
        ]
    }
}
```

Questo design separa i segnali di controllo dai payload audio, il che rende il sistema più facile da ragionare e implementare. Suddividere nettamente le responsabilità è spesso la scelta giusta.

## Integrazione del frontend nella pratica

### Connessione WebSocket

```javascript
class DoubaoVoiceClient {
    constructor(config) {
        this.config = config;
        this.ws = null;
    }

    async connect() {
        const url = new URL(this.config.wsUrl);
        // Add query parameters
        Object.entries(this.config.params).forEach(([key, value]) => {
            url.searchParams.set(key, value);
        });

        this.ws = new WebSocket(url);

        return new Promise((resolve, reject) => {
            this.ws.onopen = () => {
                console.log('[DoubaoVoice] Connected');
                resolve();
            };

            this.ws.onmessage = (event) => {
                this._handleMessage(JSON.parse(event.data));
            };

            this.ws.onerror = reject;
        });
    }

    _handleMessage(message) {
        switch (message.type) {
            case 'status':
                this._handleStatus(message.payload);
                break;
            case 'result':
                this.onResult?.(message.payload);
                break;
            case 'error':
                console.error('[DoubaoVoice] Error:', message.payload);
                break;
        }
    }
}

// Usage example
const client = new DoubaoVoiceClient({
    wsUrl: 'ws://localhost:5000/ws',
    params: {
        appId: 'your-app-id',
        accessToken: 'your-access-token',
        sampleRate: 16000,
        bitsPerSample: 16,
        channels: 1
    }
});
```

### Acquisizione e trasmissione dell'audio

Utilizzando `AudioWorkletNode` per l'elaborazione audio offre prestazioni migliori:

```javascript
// audio-worklet.js
class AudioProcessorWorklet extends AudioWorkletProcessor {
    process(inputs, outputs, parameters) {
        const input = inputs[0]?.[0];
        if (!input) return true;

        // Convert to 16-bit PCM
        const pcm = new Int16Array(input.length);
        for (let i = 0; i < input.length; i++) {
            pcm[i] = Math.max(-32768, Math.min(32767, input[i] * 32767));
        }

        this.port.postMessage({
            type: 'audioData',
            data: pcm.buffer
        }, [pcm.buffer]);

        return true;
    }
}

registerProcessor('audio-processor', AudioProcessorWorklet);

// Main thread code
async function startAudioRecording() {
    const stream = await navigator.mediaDevices.getUserMedia({
        audio: {
            echoCancellation: true,
            noiseSuppression: true,
            autoGainControl: true,
            sampleRate: 48000
        }
    });

    const audioContext = new AudioContext();
    const audioSource = audioContext.createMediaStreamSource(stream);

    await audioContext.audioWorklet.addModule('/audio-worklet.js');
    const audioWorkletNode = new AudioWorkletNode(audioContext, 'audio-processor');

    audioWorkletNode.port.onmessage = (event) => {
        if (event.data.type === 'audioData' && ws?.readyState === WebSocket.OPEN) {
            ws.send(event.data.data); // Send binary data directly
        }
    };

    audioSource.connect(audioWorkletNode);
}
```

`AudioWorklet` funziona molto meglio di `ScriptProcessorNode` ed evita i problemi di stuttering audio che spesso introducono i percorsi di elaborazione meno recenti.

## Note e buone pratiche

### 1. Monitoraggio della connessione

- Registra regolarmente lo stato della sessione per tracciare l'intero ciclo di vita della connessione
- Monitora il numero e la durata dei segmenti audio per rilevare connessioni anomale
- Registra lo stato della connessione e il comportamento di riconnessione per il servizio Doubao

Queste sono pratiche operative di base, ma fanno davvero la differenza nella produzione.

### 2. Gestione degli errori

- Cattura e registra tutte le eccezioni WebSocket
- Utilizzare `IAsyncDisposable` per garantire che le risorse vengano ripulite
- Implementare l'arresto regolare della connessione e la gestione del timeout

In breve, privilegiate la robustezza.

### 3. Requisiti del formato audio

- Frequenza di campionamento: 16000 Hz (consigliata) o 8000 Hz
- Profondità di bit: 16 bit
- Canali: mono
- Codifica: PCM (grezzo)

Se il formato è sbagliato, il riconoscimento potrebbe non riuscire o la qualità potrebbe peggiorare in modo significativo. Questi requisiti contano.

### 4. Considerazioni sulla sicurezza

- Mantieni le credenziali sensibili solo nella configurazione del backend
- Applica i limiti di connessione per prevenire l'esaurimento delle risorse
- Utilizza HTTPS/WSS in produzione

La sicurezza non è mai una preoccupazione minore.

### 5. Ottimizzazione delle prestazioni

- Utilizzare operazioni asincrone per evitare blocchi
- Ottimizza le dimensioni del buffer secondo necessità (impostazione predefinita: 4096 byte)
- Prendere in considerazione il pooling delle connessioni e le strategie di riutilizzo

Applica queste ottimizzazioni dove hanno senso per il tuo carico di lavoro.

## Raccomandazioni per la distribuzione

1. **Distribuzione Docker**: crea il pacchetto del servizio proxy come contenitore per semplificare la scalabilità e la gestione
2. **Bilanciamento del carico**: utilizza Nginx o Envoy come proxy inverso per il traffico WebSocket
3. **Controlli dello stato**: implementa controlli basati sull'heartbeat per monitorare la disponibilità del servizio
4. **Aggregazione dei registri**: invia i registri a un sistema di registrazione centralizzato come ELK o Loki

La distribuzione può essere semplice o complessa a seconda del team e dell'ambiente, quindi adeguarsi di conseguenza.

## Riepilogo

Il modello proxy WebSocket risolve il problema principale relativo al fatto che l'API WebSocket del browser non supporta le intestazioni personalizzate. Nel progetto HagiCode, questo modello si è rivelato fattibile e stabile nel passaggio dalla convalida del parco giochi all’implementazione della produzione.

Punti chiave:
- Un proxy back-end può trasmettere le informazioni di autenticazione in modo sicuro
- WebSocket nativo è leggero ed efficiente per scenari semplici
- Il modello "una connessione, una sessione" semplifica sia l'implementazione che il debug
- Il protocollo frontend-backend dovrebbe separare i segnali di controllo dai dati audio

Se stai creando una funzionalità che dipende anche dall'autenticazione WebSocket, spero che questo modello ti offra un utile punto di partenza.

Se avete domande, sentitevi liberi di discuterne con noi. Il progresso tecnico avviene più velocemente quando le persone confrontano le note.
