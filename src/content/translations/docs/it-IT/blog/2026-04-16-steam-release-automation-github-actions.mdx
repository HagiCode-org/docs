## Come automatizzare i rilasci di Steam con le azioni GitHub

> Questo articolo condivide la soluzione completa che abbiamo implementato per i rilasci automatizzati di Steam nel progetto HagiCode Desktop, coprendo il flusso di automazione end-to-end dal rilascio di GitHub alla piattaforma Steam, inclusi dettagli tecnici chiave come l'autenticazione di Steam Guard e i caricamenti di Depot multipiattaforma.

## Sfondo

Il flusso di lavoro di rilascio su Steam è in realtà molto diverso dalla distribuzione tradizionale delle applicazioni. Steam ha il proprio sistema completo di consegna degli aggiornamenti. Gli sviluppatori devono utilizzare SteamCMD per caricare artefatti di build sulla rete CDN di Steam, anziché semplicemente inserire un collegamento per il download come su altre piattaforme.

Il progetto HagiCode Desktop si sta preparando per un rilascio su Steam, che ha introdotto alcune nuove sfide al nostro flusso di lavoro di rilascio:

1. Avevamo bisogno di convertire gli artefatti di build esistenti in un formato compatibile con Steam.
2. Abbiamo dovuto caricarli su Steam tramite SteamCMD.
3. Dovevamo anche gestire l'autenticazione di Steam Guard.
4. Avevamo bisogno di supportare i caricamenti Depot multipiattaforma per Linux, Windows e macOS.
5. Volevamo un flusso completamente automatizzato dal rilascio di GitHub a Steam.

Il progetto aveva già implementato la "modalità versione portatile", che consente all'applicazione di rilevare i payload di servizi fissi inseriti nel pacchetto `extra` directory. Il nostro obiettivo era integrare perfettamente la modalità della versione portatile con la distribuzione Steam.

## Progettazione dell'architettura

Il nucleo dell'intero processo di rilascio di Steam è un flusso di lavoro GitHub Actions che divide il processo in tre fasi principali:

```
┌─────────────────────────────────────────────────────────────┐
│ GitHub Actions Workflow (Steam Release)                    │
├─────────────────────────────────────────────────────────────┤
│ 1. Preparation Stage:                                      │
│    - Check out portable-version code                       │
│    - Download build artifacts from GitHub Release          │
│    - Extract and prepare the Steam content directory       │
│                                                             │
│ 2. SteamCMD Setup:                                         │
│    - Install or reuse SteamCMD                             │
│    - Authenticate with Steam Guard                         │
│                                                             │
│ 3. Release Stage:                                          │
│    - Generate Depot VDF configuration files                │
│    - Generate App Build VDF configuration files            │
│    - Invoke SteamCMD to upload to Steam                    │
└─────────────────────────────────────────────────────────────┘
```

Questo design offre diversi vantaggi:
- Riutilizza gli artefatti della versione GitHub esistenti ed evita di ricostruire gli stessi output.
- Utilizza corridori self-hosted per un isolamento sicuro.
- Supporta il passaggio dalla modalità di anteprima ai rami di rilascio formale.
- Include la gestione e la registrazione complete degli errori, che semplificano la diagnosi degli errori.

## Implementazione del flusso di lavoro

### Progettazione dei parametri di attivazione

Il nostro flusso di lavoro supporta i seguenti parametri chiave:

```yaml
inputs:
  release:           # Portable Version release tag
    description: 'Version tag to release (for example v1.0.0)'
    required: true
  steam_preview:     # Whether to generate a preview build
    description: 'Whether to use preview mode'
    required: false
    default: 'false'
  steam_branch:      # Steam branch to set live
    description: 'Target Steam branch'
    required: false
    default: 'preview'
  steam_description: # Build description override
    description: 'Build description'
    required: false
```

### Configurazione del runner self-hosted

Per motivi di sicurezza, utilizziamo un corridore self-hosted con il file `steam` etichetta:

```yaml
runs-on:
  - self-hosted
  - Linux
  - X64
  - steam
```

Ciò garantisce che le versioni di Steam vengano eseguite su un runner dedicato e mantenga le credenziali sensibili isolate in modo sicuro.

### Controllo della concorrenza

Per evitare che rilasci della stessa versione interferiscano tra loro, abbiamo configurato il controllo della concorrenza:

```yaml
concurrency:
  group: portable-version-steam-${{ github.event.inputs.release }}
  cancel-in-progress: false
```

Notalo `cancel-in-progress: false` è impostato qui perché i rilasci di Steam possono richiedere del tempo e non vogliamo che una nuova esecuzione avviata annulli quella già in caricamento.

## Implementazione dello script principale

### Preparazione degli input di rilascio

Il `prepare-steam-release-input.mjs` lo script è responsabile della preparazione degli input richiesti per il rilascio:

```javascript
// Download the GitHub Release build manifest and artifact inventory
const buildManifest = await downloadBuildManifest(releaseTag);
const artifactInventory = await downloadArtifactInventory(releaseTag);

// Download compressed archives for each platform
for (const platform of ['linux-x64', 'win-x64', 'osx-universal']) {
  const artifactUrl = getArtifactUrl(artifactInventory, platform);
  await downloadArtifact(artifactUrl, platform);
}

// Extract into the Steam content directory structure
await extractToSteamContent(sources, contentRoot);
```

### Autenticazione di Steam Guard

Steam richiede che gli account siano protetti con Steam Guard, quindi abbiamo implementato un algoritmo di generazione del codice basato su segreti condivisi:

```javascript
function generateSteamGuardCode(sharedSecret, timestamp = Date.now()) {
  const secret = decodeSharedSecret(sharedSecret);
  const time = Math.floor(timestamp / 1000 / 30);

  const timeBuffer = Buffer.alloc(8);
  timeBuffer.writeBigUInt64BE(BigInt(time));

  // Use HMAC-SHA1 to generate a time-based one-time code
  const hash = crypto.createHmac('sha1', secret)
    .update(timeBuffer)
    .digest();

  // Convert it into a 5-character Steam Guard code
  const code = steamGuardCode(hash);
  return code;
}
```

Questa implementazione si basa sul meccanismo TOTP (Time-based One-Time Password) di Steam Guard, che genera un nuovo codice di verifica ogni 30 secondi.

### Generazione della configurazione VDF

VDF (Valve Data Format) è il formato di configurazione utilizzato da Steam. Dobbiamo generare due tipi di file VDF:

**Depot VDF** viene utilizzato per configurare il contenuto per ciascuna piattaforma:

```javascript
function buildDepotVdf(depotId, contentRoot) {
  return [
    '"DepotBuildConfig"',
    '{',
    `  "DepotID" "${escapeVdf(depotId)}"`,
    `  "ContentRoot" "${escapeVdf(contentRoot)}"`,
    '  "FileMapping"',
    '  {',
    '    "LocalPath" "*"',
    '    "DepotPath" "."',
    '    "recursive" "1"',
    '  }',
    '}'
  ].join('\n');
}
```

**App Build VDF** viene utilizzato per configurare l'intera build dell'applicazione:

```javascript
function buildAppBuildVdf(appId, depotBuilds, description, setLive) {
  const vdf = [
    '"appbuild"',
    '{',
    `  "appid" "${appId}"`,
    `  "desc" "${escapeVdf(description)}"`,
    `  "contentroot" "${escapeVdf(contentRoot)}"`,
    '  "buildoutput" "build_output"',
    '  "depots"',
    '  {'
  ];

  for (const [depotId, depotVdfPath] of Object.entries(depotBuilds)) {
    vdf.push(`    "${depotId}" "${depotVdfPath}"`);
  }

  if (setLive) {
    vdf.push(`  }`);
    vdf.push(`  "setlive" "${setLive}"`);
  }

  vdf.push('}');
  return vdf.join('\n');
}
```

### Invocazione di SteamCMD

Infine, il caricamento viene eseguito invocando SteamCMD:

```javascript
await runCommand(steamcmdPath, [
  '+login', steamUsername, steamPassword, steamGuardCode,
  '+run_app_build', appBuildPath,
  '+quit'
]);
```

Questo è il passaggio finale dell'intero flusso di lavoro. Una volta riuscito, il rilascio è terminato.

## Movimentazione deposito multipiattaforma

Steam utilizza il sistema Depot per gestire i contenuti per diverse piattaforme. Supportiamo tre depositi principali:

| Piattaforma | Identificatore del deposito | Supporto all'architettura |
|-----|-----------|---------|
| Linux | `linux-x64` | x64_64 |
| Finestre | `win-x64` | x64_64 |
| macOS | `osx-universal` | universale, x64_64, arm64 |

Ogni depot ha la propria directory dei contenuti e un file di configurazione VDF. Ciò garantisce che gli utenti di piattaforme diverse scarichino solo i contenuti di cui hanno effettivamente bisogno.

## Rilascio del flusso di lavoro

### Passaggio 1: preparare la versione GitHub

Innanzitutto, crea una versione GitHub nel file `portable-version` repository che include:
- Archivi compressi per ogni piattaforma
- Crea manifesto (`{tag}.build-manifest.json`)
- Inventario degli artefatti (`{tag}.artifact-inventory.json`)

### Passaggio 2: attiva il flusso di lavoro di rilascio di Steam

Attiva manualmente il flusso di lavoro tramite GitHub Actions e inserisci i parametri richiesti:
- `release`: il tag della versione da pubblicare, ad esempio `v1.0.0`
- `steam_branch`: il ramo di destinazione, ad esempio `preview` o `public`
- `steam_preview`: se utilizzare la modalità di anteprima

### Passaggio 3: eseguire automaticamente il flusso di rilascio

Il flusso di lavoro esegue automaticamente i seguenti passaggi:

1. Scaricare ed estrarre gli artefatti della versione GitHub.
2. Installa o aggiorna SteamCMD.
3. Genera file di configurazione VDF di Steam.
4. Autenticarsi con Steam Guard.
5. Carica contenuti sul CDN di Steam.
6. Imposta il ramo specificato come attivo.

Una volta completata questa sequenza, l'intero percorso di rilascio è coperto.

## Guida alla configurazione

### Configurazione dei segreti richiesta

Configura i seguenti segreti nelle impostazioni del repository GitHub:

| Nome segreto | Descrizione |
|-----------|------|
| `STEAM_USERNAME` | Nome utente dell'account Steam |
| `STEAM_PASSWORD` | Password dell'account Steam |
| `STEAM_SHARED_SECRET` | Segreto condiviso di Steam Guard (facoltativo) |
| `STEAM_GUARD_CODE` | Codice Steam Guard (opzionale) |
| `STEAM_APP_ID` | ID dell'applicazione Steam |
| `STEAM_DEPOT_ID_LINUX` | ID deposito Linux |
| `STEAM_DEPOT_ID_WINDOWS` | ID del deposito di Windows |
| `STEAM_DEPOT_ID_MACOS` | ID deposito macOS |

Non c'è nulla di particolarmente insolito in queste impostazioni. Hai semplicemente bisogno di tutti i valori attesi.

| Nome variabile | Descrizione | Valore predefinito |
|---------|------|--------|

## Migliori pratiche

### Gestione dell'autenticazione di Steam Guard

Al primo avvio, devi inserire manualmente il codice Steam Guard. Successivamente, si consiglia di configurare il segreto condiviso in modo che il codice possa essere generato automaticamente. Ciò evita l'intervento manuale su ogni rilascio.

SteamCMD salva il token di accesso e può riutilizzarlo nelle esecuzioni successive. È comunque necessario tenere d'occhio la scadenza del token, poiché è necessaria la riautenticazione dopo la scadenza del token.

### Struttura della directory dei contenuti

Assicurati che la struttura della directory dei contenuti di Steam sia corretta:

```
steam-content/
├── linux-x64/     # Linux platform content
├── win-x64/       # Windows platform content
└── osx-universal/ # macOS universal binary content
```

Ciascuna directory dovrebbe contenere i file dell'applicazione completi per la piattaforma corrispondente.

### Utilizzo della modalità di anteprima

La modalità di anteprima non imposta alcun ramo attivo, quindi è adatta per test e validazione:

```yaml
if [ "$STEAM_PREVIEW_INPUT" = 'true' ]; then
  cmd+=(--preview)
fi
```

Ciò ti consente di caricare prima su Steam per la verifica, quindi passare al ramo formale dopo che tutto è stato verificato.

### Gestione e registrazione degli errori

Gli script includono la gestione e la registrazione complete degli errori:
- Convalidare l'esistenza della versione GitHub.
- Controlla i file di metadati richiesti.
- Assicurarsi che il contenuto della piattaforma sia presente.
- Genera un report di riepilogo delle azioni GitHub.

Queste informazioni sono estremamente preziose sia per il debug che per il controllo.

### Gestione degli artefatti

Il flusso di lavoro genera due tipi di artefatti:
- `portable-steam-release-preparation-{tag}`: metadati di preparazione del rilascio
- `portable-steam-build-metadata-{tag}`: metadati della build di Steam

Questi artefatti possono essere utilizzati per successivi controlli e debug. Un periodo di conservazione di 30 giorni è un valore predefinito pratico.

## Utilizzo nel mondo reale

Nel progetto HagiCode, questo flusso di lavoro di rilascio automatizzato è già stato eseguito con successo su più versioni. L'intero percorso dal rilascio di GitHub alla piattaforma Steam è completamente automatizzato e non richiede alcun intervento manuale.

Ciò ha migliorato significativamente sia l'efficienza che l'affidabilità del rilascio. In passato, la pubblicazione manuale di una versione richiedeva più di 30 minuti. Ora l'intero processo termina in pochi minuti.

Ancora più importante, il flusso di lavoro automatizzato riduce la possibilità di errore umano. Ogni versione segue lo stesso processo standardizzato, che rende i risultati più prevedibili.
