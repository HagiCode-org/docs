## Da TypeScript a C#: pratica di porting multilingue per l'SDK del Codex

> In parole povere, anche questo articolo è un po' nostro: registra l'intero processo di porting dell'SDK ufficiale TypeScript Codex in C#. Chiamarlo "porto" sembra quasi troppo facile: è stata più una lunga avventura, perché queste due lingue hanno personalità molto diverse e bisognava trovare il modo di farle collaborare.

## Sfondo

Codex è lo strumento CLI dell'agente AI rilasciato da OpenAI ed è davvero potente. Il team ufficiale fornisce un SDK TypeScript nel file `@openai/codex` pacchetto. Interagisce con la CLI del Codex chiamando il file `codex exec --experimental-json` comando e analisi di un flusso di eventi JSONL.

Il problema è che nel progetto HagiCode dobbiamo usarlo in un ambiente .NET puro, in particolare nei servizi backend C# e nelle applicazioni desktop. Non potremmo ragionevolmente introdurre un runtime Node.js in un progetto .NET solo per chiamare uno strumento CLI. Sarebbe troppo complicato.

Quindi ci sono rimaste due scelte: mantenere un livello bridge Node.js complesso o creare noi stessi un SDK C# nativo.

Abbiamo scelto quest'ultimo.

## A proposito di HagiCode

Anche questo articolo deriva direttamente dalla nostra esperienza pratica nel [HagiCode](https://hagicode.com) progetto. HagiCode è un progetto di assistente alla codifica AI open source. Questa complessità multilingue e multipiattaforma è esattamente il motivo per cui avevamo bisogno di un SDK C# nativo: non volevamo eseguire Node.js all'interno di un progetto .NET.

## Contenuto principale

### Confronto di progetti architettonici

Prima di tradurre il codice uno a uno, abbiamo dovuto comprendere la progettazione dell'architettura di entrambi gli SDK. Devi comprendere entrambi i lati prima di poterli adattare bene.

L'architettura principale dell'SDK TypeScript è simile alla seguente:

```
Codex (entry class)
  └── CodexExec (executor, manages child processes)
      └── Thread (conversation thread)
          ├── run() / runStreamed() (synchronous/asynchronous execution)
          └── event stream parsing
```

L'SDK C# mantiene la stessa stratificazione dell'architettura, ma adatta i dettagli di implementazione. L'idea generale è semplice: preservare la coerenza dell'API sfruttando appieno le funzionalità del linguaggio C# nell'implementazione.

### Digitare Conversione del sistema

Questa è la parte più fondamentale e anche la più importante del lavoro. Se le fondamenta sono deboli, tutto ciò che segue diventa più difficile.

Il sistema di tipi di TypeScript è più flessibile di quello di C# e questo è semplicemente un dato di fatto. Avevamo bisogno di trovare una strategia di mappatura adeguata:

| TypeScript | C# | Note |
|------------|-----|------|
| `interface` / `type` | `record` | C# utilizza `record` per strutture dati immutabili |
| `stringa \| nullo` | `string?` | Tipo di riferimento nullable |
| `booleano \| indefinito». | `bool?` | Booleano nullable |
| `AsyncGenerator` | `IAsyncEnumerable` | Iteratore asincrono |

Il **sistema del tipo di evento** è un tipico esempio. TypeScript utilizza i tipi di unione per definire gli eventi:

```typescript
export type ThreadEvent =
  | ThreadStartedEvent
  | TurnStartedEvent
  | TurnCompletedEvent
  | ...
```

In C# utilizziamo una gerarchia di ereditarietà e una corrispondenza di modelli per ottenere un effetto simile:

```csharp
public abstract record ThreadEvent(string Type);

public sealed record ThreadStartedEvent(string ThreadId) : ThreadEvent("thread.started");
public sealed record TurnStartedEvent() : ThreadEvent("turn.started");
public sealed record TurnCompletedEvent(Usage Usage) : ThreadEvent("turn.completed");
// ...
```

Abbiamo scelto `record` invece di `class` perché gli oggetti evento dovrebbero essere immutabili, il che corrisponde all'intento dietro l'utilizzo di oggetti semplici in TypeScript. Il `sealed` La parola chiave impedisce inoltre un'ulteriore ereditarietà e offre al compilatore spazio per l'ottimizzazione.

### Punti chiave del trasferimento

#### 1. Analizzatore di eventi

L'analisi degli eventi è il nucleo dell'intero SDK, poiché determina se possiamo comprendere correttamente ogni messaggio restituito dalla CLI del Codex. Se l'analisi è sbagliata, tutto ciò che segue è uno sforzo sprecato.

La versione TypeScript utilizza `JSON.parse()` per analizzare ogni riga di JSON:

```typescript
export function parseEvent(line: string): ThreadEvent {
  const data = JSON.parse(line);
  // Handle different event types...
}
```

La versione C# utilizza `System.Text.Json.JsonDocument` invece:

```csharp
public static ThreadEvent Parse(string line)
{
    using var document = JsonDocument.Parse(line);
    var root = document.RootElement;
    var type = GetRequiredString(root, "type", "event.type");

    return type switch
    {
        "thread.started" => new ThreadStartedEvent(GetRequiredString(root, "thread_id", ...)),
        "turn.started" => new TurnStartedEvent(),
        "turn.completed" => new TurnCompletedEvent(ParseUsage(...)),
        // ...
        _ => new UnknownThreadEvent(type, root.Clone()),
    };
}
```

C'è un piccolo ma importante trucco qui: `root.Clone()` è obbligatorio, perché gli elementi from `JsonDocument` diventano nulli dopo che il documento è stato smaltito. Dobbiamo conservare una copia per i tipi di eventi sconosciuti. Questa è semplicemente una delle differenze tra la gestione di C# JSON e JavaScript.

#### 2. Differenze nella gestione dei processi

È qui che i due SDK differiscono maggiormente. Node.js e .NET hanno convenzioni di runtime diverse, quindi l'implementazione deve adattarsi.

TypeScript utilizza Node.js `spawn()` funzione:

```typescript
const child = spawn(this.executablePath, commandArgs, { env, signal });
```

C# utilizza .NET `System.Diagnostics.Process`:

```csharp
using var process = new Process { StartInfo = startInfo };
process.Start();

// stdin/stdout/stderr must be managed manually
```

Più specificamente, la versione C# deve configurare il processo in questo modo:

```csharp
var startInfo = new ProcessStartInfo
{
    FileName = _executablePath,
    RedirectStandardInput = true,
    RedirectStandardOutput = true,
    RedirectStandardError = true,
    UseShellExecute = false,
    CreateNoWindow = true,
};
```

La differenza più grande è il meccanismo di cancellazione. TypeScript utilizza `AbortSignal`, che fa parte dell'API Web ed è molto comodo con cui lavorare:

```typescript
const child = spawn(cmd, args, { signal: cancellationSignal });
```

C# utilizza `CancellationToken` invece:

```csharp
public async IAsyncEnumerable<string> RunAsync(
    CodexExecArgs args,
    [EnumeratorCancellation] CancellationToken cancellationToken = default)
{
    // Check cancellation status inside the loop
    while (!cancellationToken.IsCancellationRequested)
    {
        // Process output...
    }

    // Terminate the process when cancellation is requested
    if (cancellationToken.IsCancellationRequested)
    {
        try { process.Kill(entireProcessTree: true); } catch { }
    }
}
```

Ad alto livello, questo è solo un altro esempio della differenza tra l'ecosistema API Web e l'ecosistema .NET.

#### 3. Conservazione della serializzazione della configurazione

Entrambi gli SDK implementano la logica che converte la configurazione JSON in configurazione TOML, poiché la CLI del Codex accetta le sostituzioni della configurazione in formato TOML. Questa parte deve rimanere completamente coerente, altrimenti la stessa configurazione si comporterà diversamente nei due SDK.

Questo è il tipo di dettaglio su cui non puoi scendere a compromessi. Il successo o il fallimento spesso dipendono da dettagli come questo.

### Dettagli di implementazione

#### Struttura del progetto

Abbiamo creato la seguente struttura del progetto:

```
CodexSdk/
├── CodexSdk.csproj
├── Codex.cs           # Entry class
├── CodexThread.cs     # Conversation thread
├── CodexExec.cs       # Executor
├── Events.cs          # Event type definitions
├── Items.cs           # Item type definitions
├── EventParser.cs     # Event parser
├── OutputSchemaTempFile.cs  # Temporary file management
└── ...
```

È una struttura abbastanza pulita, e questo ha aiutato molto durante il porto.

#### Esempio di utilizzo

L'utilizzo di base rimane coerente con l'SDK TypeScript:

```csharp
using CodexSdk;

// Create a Codex instance
var codex = new Codex();
var thread = codex.StartThread();

// Execute a query
var result = await thread.RunAsync("Summarize this repository.");
Console.WriteLine(result.FinalResponse);
```

La gestione degli eventi di streaming sfrutta la corrispondenza dei modelli C#:

```csharp
await foreach (var @event in thread.RunStreamedAsync("Analyze the code."))
{
    switch (@event)
    {
        case ItemCompletedEvent itemCompleted
            when itemCompleted.Item is AgentMessageItem msg:
            Console.WriteLine($"Assistant: {msg.Text}");
            break;
        case TurnCompletedEvent completed:
            Console.WriteLine($"Tokens: in={completed.Usage.InputTokens}");
            break;
        case CommandExecutionItem command:
            Console.WriteLine($"Command: {command.Command}");
            break;
    }
}
```

### Note

Durante l'implementazione, abbiamo raccolto diverse lezioni pratiche:

1. **Gestione del processo**: la versione C# deve gestire manualmente l'intero ciclo di vita del processo, inclusa la terminazione del processo durante l'annullamento. Utilizzare `Kill(entireProcessTree: true)` per assicurarsi che anche i processi figli vengano ripuliti.

2. **Gestione degli errori**: Usiamo `InvalidOperationException` per generare errori di analisi, mantenendo lo stile di gestione degli errori simile a TypeScript SDK.

3. **Pulizia delle risorse**: `OutputSchemaTempFile` implementa `IAsyncDisposable` per garantire che i file temporanei vengano ripuliti correttamente.

4. **Variabili di ambiente**: la versione C# supporta l'override completo delle variabili di ambiente del processo tramite `CodexOptions.Env`. È una caratteristica piccola, ma molto pratica.

5. **Differenze di piattaforma**: la versione C# non include la logica della versione TypeScript per l'individuazione automatica dei file binari all'interno dei pacchetti npm. Poiché i progetti .NET in genere non dipendono da npm, il percorso del file `codex` l'eseguibile deve essere specificato tramite il file `CODEX_EXECUTABLE` variabile d'ambiente o `CodexPathOverride`.

## Conclusione

Il porting di un SDK TypeScript maturo in C# non è solo una questione di conversione della sintassi, ma richiede anche la comprensione delle filosofie di progettazione di entrambi i linguaggi. La flessibilità di TypeScript e le funzionalità dell'ecosistema JavaScript come `AbortSignal` necessitano di controparti appropriate in C#.

Il punto fondamentale è questo: **mantenere la coerenza dell'API è più importante che mantenere la coerenza a livello di implementazione**. Agli utenti interessa se l'interfaccia è facile da usare, non se l'implementazione interna è identica. Sembra semplice, ma per fare questi compromessi è necessario giudizio.

Se stai lavorando su un porting multilingue simile, la nostra esperienza consiste nel comprendere prima l'architettura originale dell'SDK, quindi tradurla modulo per modulo e infine utilizzare una suite di test completa per garantire la coerenza comportamentale. Questo tipo di lavoro non può essere affrettato.

Tutto funzionerà alla fine.
