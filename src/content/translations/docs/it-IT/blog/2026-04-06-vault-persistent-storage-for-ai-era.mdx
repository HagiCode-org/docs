## Come riprodurre progetti nell'era dell'intelligenza artificiale: Vault, un sistema di archiviazione persistente tra progetti

> Nell’era dello sviluppo assistito dall’intelligenza artificiale, come possiamo aiutare gli assistenti AI a comprendere meglio le nostre risorse di apprendimento? Il progetto HagiCode ha creato il sistema Vault come uno strato unificato di astrazione per l’archiviazione della conoscenza che l’intelligenza artificiale può comprendere, migliorando notevolmente l’efficienza dell’apprendimento attraverso la riproduzione del progetto.

## Sfondo

Nell’era dell’intelligenza artificiale, il modo in cui gli sviluppatori apprendono nuove tecnologie e architetture sta cambiando profondamente. La "riproduzione di progetti", ovvero lo studio approfondito e l'apprendimento dal codice, dall'architettura e dai modelli di progettazione di eccellenti progetti open source, è diventato un modo efficace per apprendere. Rispetto ai metodi tradizionali come leggere libri o guardare video, leggere ed eseguire direttamente progetti open source di alta qualità ti aiuta a comprendere le pratiche ingegneristiche del mondo reale molto più velocemente.

Tuttavia, questo metodo di apprendimento presenta alcune sfide.

I materiali didattici sono troppo sparsi. I tuoi appunti potrebbero vivere in Obsidian, i repository di codice potrebbero essere sparsi in cartelle diverse e la cronologia delle conversazioni del tuo assistente AI diventa ancora un'altra isola di dati isolata. Quando vuoi che l'intelligenza artificiale aiuti ad analizzare un progetto, devi copiare manualmente frammenti di codice e organizzare il contesto, il che è piuttosto noioso.

Ciò che è ancora più problematico è il contesto frammentato. Gli assistenti AI non possono accedere direttamente alle tue risorse di apprendimento locali, quindi devi fornire nuovamente informazioni di base in ogni conversazione. Oltre a ciò, i repository di codice riprodotto si aggiornano rapidamente, la sincronizzazione manuale è soggetta a errori e la conoscenza è difficile da condividere tra più progetti di apprendimento.

Alla radice, tutti questi problemi derivano dalle “isole di dati”. Se esistesse un livello di astrazione dello storage unificato che consentisse agli assistenti AI di comprendere e accedere a tutte le risorse di apprendimento, il problema sarebbe risolto perfettamente.

Il sistema Vault condiviso in questo articolo è esattamente la soluzione che abbiamo sviluppato durante la creazione [HagiCode](https://hagicode.com). Per aiutare gli assistenti IA a comprendere meglio queste risorse di apprendimento, abbiamo progettato Vault, un sistema di archiviazione persistente interprogetto.

Questa soluzione è già stata validata in HagiCode nell'uso reale. Se stai affrontando sfide simili nella gestione della conoscenza, spero che queste esperienze possano offrire qualche ispirazione. Dopotutto, una volta che anche tu sei caduto in qualche fossa, dovresti lasciare qualcosa dietro di te per la persona successiva.

## Filosofia di progettazione del sistema di volte

L’idea centrale del sistema Vault è semplice: creare un livello di astrazione unificato per l’archiviazione della conoscenza che l’intelligenza artificiale possa comprendere. Dal punto di vista dell’implementazione, il sistema presenta diverse caratteristiche chiave.

### Supporto multitipo

Il sistema supporta quattro tipi di depositi, ciascuno corrispondente a uno scenario di utilizzo diverso:

```typescript
// folder: general-purpose folder type
export const DEFAULT_VAULT_TYPE = 'folder';

// coderef: a type specifically for reproduced code projects
export const CODEREF_VAULT_TYPE = 'coderef';

// obsidian: integrated with Obsidian note-taking software
export const OBSIDIAN_VAULT_TYPE = 'obsidian';

// system-managed: vault automatically managed by the system
export const SYSTEM_MANAGED_VAULT_TYPE = 'system-managed';
```

Tra questi, il tipo **coderef** è quello più comunemente utilizzato in HagiCode. È progettato specificamente per progetti di codice riprodotto, fornendo una struttura di directory standardizzata e descrizioni di metadati leggibili dall'intelligenza artificiale.

### Meccanismo di archiviazione persistente

Il registro del Vault viene archiviato in modo persistente in formato JSON, garantendo che la configurazione rimanga disponibile dopo il riavvio dell'applicazione:

```csharp
public class VaultRegistryStore : IVaultRegistryStore
{
    private readonly string _registryFilePath;

    public VaultRegistryStore(IConfiguration configuration, ILogger<VaultRegistryStore> logger)
    {
        var dataDir = configuration["DataDir"] ?? "./data";
        var absoluteDataDir = Path.IsPathRooted(dataDir)
            ? dataDir
            : Path.GetFullPath(Path.Combine(Directory.GetCurrentDirectory(), dataDir));

        _registryFilePath = Path.Combine(absoluteDataDir, "personal-data", "vaults", "registry.json");
    }
}
```

Il vantaggio di questo design è che è semplice e affidabile. JSON è leggibile dall'uomo, il che semplifica il debug e la modifica manuale; L'archiviazione del file system evita la complessità di un database e riduce le dipendenze del sistema. Dopotutto, a volte l’opzione più semplice è davvero quella migliore.

### Integrazione del contesto AI

Ancora più importante, il sistema può inserire automaticamente le informazioni del deposito nel contesto delle proposte di intelligenza artificiale:

```typescript
export function buildTargetVaultsText(
  vaults: VaultForText[],
  template: VaultPromptTemplate = DEFAULT_VAULT_PROMPT_TEMPLATE,
): string {
  const readOnlyVaults = vaults.filter((vault) => vault.accessType === 'read');
  const editableVaults = vaults.filter((vault) => vault.accessType === 'write');

  if (readOnlyVaults.length === 0 && editableVaults.length === 0) {
    return '';
  }

  const sections = [
    buildVaultSection(readOnlyVaults, template.reference),
    buildVaultSection(editableVaults, template.editable),
  ].filter(Boolean);

  return `\n\n### ${template.heading}\n\n${sections.join('\n')}`;
}
```

Ciò consente una funzionalità importante: gli assistenti AI possono comprendere automaticamente le risorse di apprendimento disponibili senza che gli utenti forniscano manualmente il contesto. Si potrebbe dire che conta come una sorta di tacita intesa.

## La struttura standardizzata di CodeRef Vault

Per il tipo di vault coderef, HagiCode fornisce una struttura di directory standardizzata:

```
my-coderef-vault/
├── index.yaml          # vault metadata description
├── AGENTS.md           # operating guide for AI assistants
├── docs/               # stores study notes and documents
└── repos/              # manages reproduced code repositories through Git submodules
```

Quando si crea un deposito, il sistema inizializza automaticamente questa struttura:

```csharp
private async Task EnsureCodeRefStructureAsync(
    string vaultName,
    string physicalPath,
    ICollection<VaultBootstrapDiagnosticDto> diagnostics,
    CancellationToken cancellationToken)
{
    Directory.CreateDirectory(physicalPath);

    var indexPath = Path.Combine(physicalPath, CodeRefIndexFileName);
    var docsPath = Path.Combine(physicalPath, CodeRefDocsDirectoryName);
    var reposPath = Path.Combine(physicalPath, CodeRefReposDirectoryName);

    // Create the standard directory structure
    if (!Directory.Exists(docsPath))
    {
        Directory.CreateDirectory(docsPath);
    }

    if (!Directory.Exists(reposPath))
    {
        Directory.CreateDirectory(reposPath);
    }

    // Create the AGENTS.md guide
    await EnsureCodeRefAgentsDocumentAsync(physicalPath, cancellationToken);

    // Create the index.yaml metadata
    await WriteCodeRefIndexDocumentAsync(indexPath, mergedDocument, cancellationToken);
}
```

Anche questa struttura è attentamente progettata:

- **docs/** memorizza i tuoi appunti di studio, dove puoi registrare la tua comprensione del codice, l'analisi dell'architettura, le lezioni apprese e così via in Markdown
- **repos/** gestisce i repository riprodotti tramite i sottomoduli Git invece di copiare direttamente il codice, mantenendo il codice sincronizzato e risparmiando spazio
- **index.yaml** contiene i metadati del vault in modo che gli assistenti IA possano comprendere rapidamente lo scopo e i contenuti del vault
- **AGENTS.md** è una guida scritta appositamente per gli assistenti IA, che spiega come gestire il contenuto del vault

Organizzata in questo modo, forse l’intelligenza artificiale può capire quello che hai in mente un po’ più facilmente.

## Inizializzazione automatica per depositi gestiti dal sistema

Oltre ai depositi creati manualmente, HagiCode supporta anche depositi gestiti dal sistema:

```csharp
public async Task<IReadOnlyList<VaultRegistryEntry>> EnsureAllSystemManagedVaultsAsync(
    CancellationToken cancellationToken = default)
{
    var definitions = GetAllResolvedDefinitions();
    var entries = new List<VaultRegistryEntry>(definitions.Count);

    foreach (var definition in definitions)
    {
        entries.Add(await EnsureResolvedSystemManagedVaultAsync(definition, cancellationToken));
    }

    return entries;
}
```

Il sistema crea e gestisce automaticamente i seguenti depositi:

- **hagiprojectdata**: archiviazione dei dati di progetto utilizzata per salvare la configurazione e lo stato del progetto
- **datipersonali**: memorizzazione dei dati personali utilizzata per salvare le preferenze dell'utente
- **hbsprompt**: una libreria di modelli di prompt utilizzata per gestire i prompt AI di uso comune

Questi depositi vengono inizializzati automaticamente all'avvio del sistema, quindi gli utenti non devono configurarli manualmente. Alcune cose è semplicemente meglio lasciarle al sistema invece che gli umani se ne preoccupino.

## Meccanismo di controllo degli accessi

Una parte importante del progetto è il controllo degli accessi. Il sistema divide i depositi in due tipologie di accesso:

```typescript
export interface VaultForText {
  id: string;
  name: string;
  type: string;
  physicalPath: string;
  accessType: 'read' | 'write';  // Key: distinguish read-only from editable
}
```

- **riferimento (sola lettura)**: l'intelligenza artificiale viene utilizzata solo per l'analisi e la comprensione e non può modificare i contenuti. Adatto per progetti open source referenziati, documenti e materiali simili
- **modificabile (modificabile)**: l'intelligenza artificiale può modificare il contenuto secondo necessità per l'attività. Adatto per appunti, bozze e materiali simili

Questa distinzione è importante. Indica all'IA quale contenuto è "riferimento di sola lettura" e quale contenuto è "sicuro da modificare", riducendo il rischio di modifiche accidentali. Dopotutto, nessuno vuole che il proprio duro lavoro scompaia a causa di una modifica involontaria.

## In pratica: creazione e utilizzo di Vault

Ora che abbiamo trattato le idee, diamo un'occhiata a come funziona nella pratica.

### Crea un archivio CodeRef

Ecco un esempio di chiamata frontend completa:

```typescript
const createCodeRefVault = async () => {
  const response = await VaultService.postApiVaults({
    requestBody: {
      name: "React Learning Vault",
      type: "coderef",
      physicalPath: "/Users/developer/vaults/react-learning",
      gitUrl: "https://github.com/facebook/react.git"
    }
  });

  // The system will automatically:
  // 1. Clone the React repository into vault/repos/react
  // 2. Create the docs/ directory for notes
  // 3. Generate the index.yaml metadata
  // 4. Create the AGENTS.md guide file

  return response;
};
```

Questa chiamata API completa una serie di azioni: creazione della struttura di directory, inizializzazione dei sottomoduli Git, generazione di file di metadati e altro. Devi solo fornire le informazioni di base e lasciare che il sistema gestisca il resto. Onestamente è un approccio abbastanza senza preoccupazioni.

### Utilizza Vault in una proposta AI

Dopo aver creato il vault, puoi farvi riferimento in una proposta AI:

## Migliori pratiche e cose a cui prestare attenzione

Durante l'utilizzo del sistema Vault, abbiamo riassunto alcune lezioni apprese.

### Sicurezza del percorso

Il sistema convalida rigorosamente i percorsi per prevenire attacchi trasversali:

```csharp
private static string ResolveFilePath(string vaultRoot, string relativePath)
{
    var rootPath = EnsureTrailingSeparator(Path.GetFullPath(vaultRoot));
    var combinedPath = Path.GetFullPath(Path.Combine(rootPath, relativePath));
    if (!combinedPath.StartsWith(rootPath, StringComparison.OrdinalIgnoreCase))
    {
        throw new BusinessException(VaultRelativePathTraversalCode,
            "Vault file paths must stay inside the registered vault root.");
    }
    return combinedPath;
}
```

Questo è importante. Se personalizzi un percorso del deposito, assicurati che rimanga entro l'intervallo consentito, altrimenti il ​​sistema rifiuterà l'operazione. Non puoi davvero enfatizzare abbastanza la sicurezza.

### Gestione dei sottomoduli Git

CodeRef Vault consiglia i sottomoduli Git invece di copiare direttamente il codice:

```csharp
private static string BuildCodeRefAgentsContent()
{
    return """
    # CodeRef Vault Guide

    Repositories under `repos/` should be maintained through Git submodules
    rather than copied directly into the vault root.

    Keep this structure stable so assistants and tools can understand the vault quickly.
    """ + Environment.NewLine;
}
```

Ciò comporta numerosi vantaggi: mantenere il codice sincronizzato con l'upstream, risparmiare spazio su disco e semplificare la gestione di più versioni del codice. Dopo tutto, chi vorrebbe scaricare sempre la stessa cosa?

### Limiti dell'anteprima del file

Per evitare problemi di prestazioni, il sistema limita la dimensione e il tipo di file:

```csharp
private const int FileEnumerationLimit = 500;
private const int PreviewByteLimit = 256 * 1024;  // 256KB
```

Se il tuo vault contiene un numero elevato di file o file molto grandi, le prestazioni dell'anteprima potrebbero risentirne. In tal caso, puoi prendere in considerazione l'elaborazione dei file in batch o l'utilizzo di strumenti di ricerca specializzati. A volte, quando qualcosa diventa troppo grande, diventa più difficile da gestire, non più facile.

### Informazioni diagnostiche

Quando si crea un deposito, il sistema restituisce informazioni diagnostiche per facilitare il debug:

```csharp
List<VaultBootstrapDiagnosticDto> bootstrapDiagnostics = [];

if (IsCodeRefVaultType(normalizedType))
{
    bootstrapDiagnostics = await EnsureCodeRefBootstrapAsync(
        normalizedName,
        normalizedPhysicalPath,
        normalizedGitUrl,
        cancellationToken);
}
```

Se la creazione non riesce, è possibile esaminare le informazioni diagnostiche per comprendere la causa specifica. Quando qualcosa va storto, controllare la diagnostica è spesso la soluzione più diretta.
