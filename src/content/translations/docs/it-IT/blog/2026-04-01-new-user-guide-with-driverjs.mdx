## Creazione di un onboarding elegante di nuovi utenti nei progetti React: pratica driver.js di HagiCode

> Quando gli utenti aprono il tuo prodotto per la prima volta, sanno davvero da dove cominciare? In questo articolo, voglio parlare un po' di come abbiamo utilizzato driver.js per l'onboarding di nuovi utenti nel progetto HagiCode. Consideralo un piccolo esempio pratico per avviare la conversazione.

## Sfondo

Ti è mai capitato di imbatterti in questa situazione? Un nuovo utente si iscrive al tuo prodotto, apre la pagina e sembra immediatamente perso. Si guardano intorno, incerti su cosa fare clic o cosa fare dopo. Come sviluppatori, spesso diamo per scontato che gli utenti semplicemente "esploreranno per conto proprio" perché, dopo tutto, la curiosità umana è illimitata. Ma la realtà è diversa: la maggior parte degli utenti se ne va silenziosamente nel giro di pochi minuti se non riesce a trovare il punto di ingresso giusto, come se la storia iniziasse all'improvviso e finisse in modo altrettanto naturale.

L'onboarding di nuovi utenti è un modo importante per risolvere questo problema, ma costruirlo bene non è così semplice. Un buon sistema di onboarding deve:

- Individua con precisione gli elementi della pagina ed evidenziali
- Supporta flussi di onboarding in più fasi
- Ricorda la scelta dell'utente (completa o salta)
- Evita di influenzare le prestazioni della pagina e la normale interazione
- Mantieni la struttura del codice chiara e di facile manutenzione

Durante la creazione di HagiCode, ci siamo imbattuti nella stessa sfida. HagiCode è un assistente di codifica basato sull'intelligenza artificiale e il suo flusso di lavoro principale è un flusso di lavoro OpenSpec simile a questo: "l'utente crea una proposta -> l'intelligenza artificiale genera un piano -> l'utente lo rivede -> l'intelligenza artificiale lo esegue". Per gli utenti che incontrano questo concetto per la prima volta, il flusso di lavoro è completamente nuovo, quindi hanno bisogno di un solido onboarding per iniziare rapidamente. Le cose nuove richiedono sempre un po’ di tempo per abituarsi.

## A proposito di HagiCode

HagiCode è un assistente di codifica AI basato su Claude che aiuta gli sviluppatori a completare le attività di codifica in modo più efficiente attraverso il flusso di lavoro OpenSpec. Puoi visualizzare il nostro codice open source su [GitHub](https://github.com/HagiCode-org/site).

## Perché abbiamo scelto driver.js

Durante la fase di valutazione tecnica, abbiamo esaminato diverse librerie di onboarding tradizionali. Ognuno aveva i suoi punti di forza:

- **Intro.js**: potente, ma relativamente grande e la personalizzazione dello stile è alquanto complessa
- **Shepherd.js**: API ben progettata, ma un po' troppo "pesante" per il nostro caso d'uso
- **driver.js**: API leggera, concisa, intuitiva e funziona bene nell'ecosistema React

Alla fine, abbiamo scelto driver.js. Non c'era una ragione particolarmente drammatica. La scelta è ricaduta principalmente su queste considerazioni:

1. **Leggero**: la libreria principale è piccola e non aumenta in modo significativo le dimensioni del bundle
2. **API semplice**: la configurazione è chiara e intuitiva, quindi è facile da imparare
3. **Flessibile**: supporta il posizionamento, lo stile e il comportamento di interazione personalizzati
4. **Importazione dinamica**: può essere caricato su richiesta senza influire sulle prestazioni della prima schermata

Con la selezione della tecnologia, raramente esiste una risposta universalmente migliore. Di solito c'è solo l'opzione più adatta.

## Implementazione tecnica

### Configurazione principale

driver.js ha un modello di configurazione molto intuitivo. Ecco la configurazione principale che utilizziamo nel progetto HagiCode:

```typescript
import { driver } from 'driver.js';
import 'driver.js/dist/driver.css';

const newConversationDriver = driver({
  allowClose: true,           // Allow users to close the guide
  animate: true,              // Enable animations
  overlayClickBehavior: 'close', // Close the guide when the overlay is clicked
  disableActiveInteraction: false, // Keep elements interactive
  showProgress: false,        // Do not show the progress bar (we manage progress ourselves)
  steps: guideSteps           // Array of guide steps
});
```

Il ragionamento alla base di queste impostazioni è:

- `allowClose: true` - Rispettare la scelta dell'utente e non obbligarlo a finire la guida
- `disableActiveInteraction: false` - Alcuni passaggi richiedono azioni reali da parte dell'utente, come la digitazione di input, quindi l'interazione non può essere disabilitata
- `overlayClickBehavior: 'close'` - Fornire agli utenti un modo rapido per uscire

### Gestione statale

La persistenza dello stato di onboarding è fondamentale. Non vogliamo riavviare la guida ogni volta che la pagina si aggiorna, perché diventa rapidamente fastidioso. HagiCode utilizza `localStorage` per gestire lo stato guida:

```typescript
export type GuideState = 'pending' | 'dismissed' | 'completed';

export interface UserGuideState {
  session: GuideState;
  detailGuides: Record<string, GuideState>;
}

// Read state
export const getUserGuideState = (): UserGuideState => {
  const state = localStorage.getItem('userGuideState');
  return state ? JSON.parse(state) : { session: 'pending', detailGuides: {} };
};

// Update state
export const setUserGuideState = (state: UserGuideState) => {
  localStorage.setItem('userGuideState', JSON.stringify(state));
};
```

Definiamo tre stati:

- `pending`: la guida è ancora in corso e l'utente non l'ha completata né saltata
- `dismissed`: l'utente ha chiuso la guida in modo proattivo
- `completed`: l'utente ha completato tutti i passaggi

Per l'onboarding della pagina dei dettagli della proposta, supportiamo anche un monitoraggio dello stato più dettagliato tramite `detailGuides` mappa, perché una proposta può passare attraverso più fasi (bozza, revisione ed esecuzione completata) e ciascuna fase necessita di indicazioni diverse. Lo stato delle cose è in continua evoluzione e l’onboarding dovrebbe rifletterlo.

### Selezione dell'elemento di destinazione

driver.js utilizza selettori CSS per individuare gli elementi di destinazione. HagiCode segue una semplice convenzione: utilizzare un codice personalizzato `data-guide` attributo per contrassegnare gli obiettivi di onboarding:

```typescript
const steps = [
  {
    element: '[data-guide="launch"]',
    popover: {
      title: 'Start a New Conversation',
      description: 'Click here to create a new conversation session...'
    }
  }
];
```

Nei componenti, assomiglia a questo:

```typescript
<button data-guide="launch" onClick={handleLaunch}>
  New Conversation
</button>
```

I vantaggi di questo approccio sono:

- Evita conflitti con i nomi delle classi di stile aziendale
- Semantica chiara, così puoi immediatamente capire che l'elemento è correlato all'onboarding
- Più facile da gestire e mantenere in modo coerente

### Ottimizzazione dinamica delle importazioni

Poiché la funzionalità di onboarding è necessaria solo in scenari specifici, come la prima visita di un utente, utilizziamo le importazioni dinamiche per ottimizzare le prestazioni di caricamento iniziale:

```typescript
const initNewUserGuide = async () => {
  // Dynamically import driver.js
  const { driver } = await import('driver.js');
  await import('driver.js/dist/driver.css');

  // Initialize the guide
  const newConversationDriver = driver({
    // ...configuration
  });

  newConversationDriver.drive();
};
```

In questo modo, driver.js e il suo foglio di stile vengono caricati solo quando necessario e non influiscono sulle prestazioni della prima schermata. Non molte persone amano aspettare qualcosa di cui non hanno ancora bisogno.

## Progettazione del flusso di onboarding

HagiCode implementa due percorsi di onboarding che coprono gli scenari principali dell'utente.

### Onboarding della sessione (10 passaggi)

Questo percorso di onboarding aiuta gli utenti a completare l'intero flusso dalla creazione di una conversazione all'invio della prima proposta completa:

1. **avvia** - Avvia la guida e introduce il pulsante "Nuova conversazione".
2. **componi**: guida l'utente a digitare una richiesta nella casella di input
3. **invia**: guida l'utente a fare clic sul pulsante di invio
4. **proposal-launch-readme** - Guida l'utente nella creazione di una proposta README
5. **proposal-compose-readme** - Guida l'utente nella modifica del contenuto della richiesta README
6. **proposta-submit-readme** - Guida l'utente a inviare la proposta README
7. **proposal-launch-agents** - Guida l'utente nella creazione di un file `AGENTS.md` Proposta
8. **proposal-compose-agents** - Guida l'utente nella modifica del file `AGENTS.md` richiesta
9. **proposal-submit-agents** - Guida l'utente a inviare il file `AGENTS.md` Proposta
10. **proposal-wait** - Spiega che l'intelligenza artificiale sta elaborando e che l'utente deve attendere un momento

L'idea alla base di questo percorso è consentire agli utenti di sperimentare in prima persona il flusso di lavoro principale di HagiCode attraverso due attività reali di creazione della proposta, una per `README` e uno per `AGENTS.md`. C'è una grande differenza tra sentire parlare di un flusso di lavoro ed affrontarlo da soli.

### Onboarding dei dettagli della proposta (3 passaggi)

Quando gli utenti accedono alla pagina dei dettagli della proposta, HagiCode attiva la guida corrispondente in base allo stato attuale della proposta:

1. **redazione** (fase di bozza): guida l'utente a rivedere il piano generato dall'intelligenza artificiale
2. **revisione** (fase di revisione): guida l'utente nell'esecuzione del piano
3. **executionCompleted** (fase completata): guida l'utente all'archiviazione del piano

La caratteristica distintiva di questa guida è che è guidata dallo stato: decide dinamicamente quale passaggio di onboarding mostrare in base allo stato attuale della proposta. Le cose cambiano e l’onboarding dovrebbe cambiare con loro.

In questa fase, la guida focalizza l'attenzione dell'utente sull'azione chiave di generazione di un piano, in modo che non si chieda cosa fare prima quando entra per la prima volta nella pagina dei dettagli.

### Meccanismo di ripetizione del rendering degli elementi

Nelle applicazioni React, l'elemento di onboarding di destinazione potrebbe non aver ancora terminato il rendering, ad esempio perché i dati asincroni sono ancora in fase di caricamento. Per gestire questo, HagiCode implementa un meccanismo di ripetizione:

```typescript
const waitForElement = (selector: string, maxRetries = 10, interval = 100) => {
  let retries = 0;
  return new Promise<HTMLElement>((resolve, reject) => {
    const checkElement = () => {
      const element = document.querySelector(selector) as HTMLElement;
      if (element) {
        resolve(element);
      } else if (retries < maxRetries) {
        retries++;
        setTimeout(checkElement, interval);
      } else {
        reject(new Error(`Element not found: ${selector}`));
      }
    };
    checkElement();
  });
};
```

Chiama questa funzione prima di inizializzare la guida per assicurarti che l'elemento di destinazione esista già. A volte vale la pena aspettare ancora un po'.

## Riepilogo delle migliori pratiche

Sulla base dell'esperienza pratica di HagiCode, ecco alcune best practice chiave:

### 1. L’onboarding dovrebbe essere facile da uscire

Non forzare gli utenti a completare l'onboarding. Alcuni utenti sono esploratori per natura e preferiscono capire le cose da soli. Fornisci un pulsante "Salta" chiaro, ricorda la loro scelta e non interromperli nuovamente la prossima volta.

### 2. Mantieni i contenuti di onboarding brevi e nitidi

Ogni fase di onboarding dovrebbe concentrarsi su un unico obiettivo:

- **Titolo**: breve e chiaro, idealmente non più di poche parole
- **Descrizione**: Vai dritto al punto e spiega all'utente di cosa si tratta e perché è importante

Evita spiegazioni prolisse. L'attenzione dell'utente è limitata durante l'onboarding e più dici, meno è probabile che lo leggano.

### 3. Mantenere stabili i selettori

Utilizzare segnalini di elementi stabili che non cambino spesso. L'usanza `data-guide` l'attributo è una buona scelta. Evita di dipendere dai nomi delle classi o dalla struttura DOM, perché sono facili da modificare durante i refactoring. Il codice cambia continuamente, ma alcuni ancoraggi dovrebbero rimanere stabili quando possibile.

### 4. Metti alla prova il tuo onboarding

HagiCode include casi di test completi per la funzionalità di onboarding:

```typescript
describe('NewUserConversationGuide', () => {
  it('should initialize guide state correctly', () => {
    const state = getUserGuideState();
    expect(state.session).toBe('pending');
  });

  it('should update guide state correctly', () => {
    setUserGuideState({ session: 'completed', detailGuides: {} });
    const state = getUserGuideState();
    expect(state.session).toBe('completed');
  });
});
```

I test aiutano a garantire che il refactoring non interrompa accidentalmente il comportamento di onboarding. Nessuno vuole che una piccola modifica al codice danneggi silenziosamente funzionalità precedentemente funzionanti.

### 5. Ottimizzazione delle prestazioni

- Utilizza le importazioni dinamiche per eseguire il caricamento lento della libreria di onboarding
- Evitare di inizializzare la logica di onboarding dopo che l'utente ha già completato la guida
- Considera l'impatto sulle prestazioni delle animazioni e, se necessario, disabilitale sui dispositivi di fascia bassa

Le prestazioni, come molte cose nella vita, meritano un po’ di attenzione al budget.

## Conclusione

L'onboarding di nuovi utenti è una parte importante del miglioramento dell'esperienza utente del prodotto. Nel progetto HagiCode, abbiamo utilizzato driver.js per creare un sistema di onboarding completo che copra l'intero flusso di lavoro, dalla creazione della sessione all'esecuzione della proposta.

I punti fondamentali che speriamo di condividere attraverso questo articolo sono:

1. **Le scelte tecniche dovrebbero corrispondere alle esigenze reali**: driver.js non è l'opzione più potente, ma è la soluzione migliore per noi
2. **La gestione dello stato è fondamentale**: utilizzare `localStorage` per mantenere lo stato di onboarding ed evitare di interrompere ripetutamente gli utenti
3. **La progettazione dell'onboarding deve rimanere focalizzata**: ogni passaggio deve risolvere un problema e non di più
4. **La struttura del codice deve rimanere chiara**: configurazione di onboarding, gestione dello stato e logica dell'interfaccia utente separate per semplificare la manutenzione

Se stai aggiungendo nuovi utenti al tuo progetto, spero che l'esperienza pratica contenuta in questo articolo ti sia d'aiuto. Non c’è nulla di particolarmente mistico in questo tipo di tecnologia. Continua a provare, continua a riassumere ciò che impari e le cose gradualmente diventeranno più facili.
