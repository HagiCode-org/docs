## Esecuzione degli strumenti AI CLI nei contenitori Docker: una guida pratica all'isolamento degli utenti e ai volumi persistenti

> L'integrazione di strumenti di codifica AI come Claude Code, Codex e OpenCode in ambienti containerizzati sembra semplice, ma ci sono complessità nascoste ovunque. Questo articolo approfondisce il modo in cui il progetto HagiCode risolve le sfide principali nelle distribuzioni Docker, comprese le autorizzazioni utente, la persistenza della configurazione e la gestione delle versioni, in modo da poter evitare le insidie ​​​​comuni.

## Sfondo

Quando abbiamo deciso di eseguire gli strumenti CLI di codifica AI all'interno dei contenitori Docker, il pensiero più intuitivo è stato probabilmente: "I contenitori non sono solo root? Perché non installare tutto direttamente e dire tutto?" In realtà, quell’idea apparentemente semplice nasconde diversi problemi fondamentali che devono essere risolti.

Innanzitutto, le **restrizioni di sicurezza sono il primo ostacolo**. Prendiamo ad esempio la CLI di Claude: vieta esplicitamente l'esecuzione come utente root. Questo è un controllo di sicurezza obbligatorio e, se viene rilevato root, si rifiuta di avviarsi. Potresti pensare, non posso semplicemente cambiare utente con il file `USER` direttiva? Non è così semplice. Esiste ancora un problema di mappatura tra l'utente non root all'interno del contenitore e le autorizzazioni dell'utente sul computer host.

In secondo luogo, la **persistenza dello stato è la seconda trappola**. Claude Code richiede l'accesso, Codex ha una propria configurazione e OpenCode ha anche una directory della cache. Se devi riconfigurare tutto ogni volta che il contenitore si riavvia, l'intera idea di "automazione" perde il suo significato. Abbiamo bisogno che queste configurazioni persistano oltre il ciclo di vita del contenitore.

Il terzo problema è la **coerenza dei permessi**. I processi all'interno del contenitore possono accedere ai file di configurazione creati dall'utente host? Le mancate corrispondenze UID/GID spesso causano errori di autorizzazione dei file e questo è estremamente comune nelle distribuzioni reali.

Questi problemi possono sembrare indipendenti, ma in realtà sono strettamente connessi. Durante lo sviluppo di HagiCode abbiamo gradualmente elaborato una soluzione pratica. Successivamente, condividerò i dettagli tecnici e le lezioni apprese da queste insidie.

## Perché non possiamo semplicemente usare root?

C'è un malinteso comune qui: i contenitori Docker vengono eseguiti come root per impostazione predefinita, quindi perché non installare semplicemente gli strumenti come root? Se la pensi in questo modo, Claude CLI ti insegnerà rapidamente il contrario.

```bash
# Run Claude CLI directly as root? No.
docker run --rm -it --user root myimage claude
# Output: Error: This command cannot be run as root user
```

Questa è una restrizione di sicurezza rigida nella CLI di Claude. Il motivo è semplice: questi strumenti CLI leggono e scrivono configurazioni utente sensibili, inclusi token API, cache locali e persino script scritti dall'utente. Eseguirli con i privilegi di root introduce troppi rischi.

Quindi la domanda diventa: come possiamo soddisfare i requisiti di sicurezza della CLI mantenendo flessibile la gestione dei container? Dobbiamo cambiare il modo in cui la pensiamo: **invece di cambiare utente in fase di runtime, crea un utente dedicato durante la fase di creazione dell'immagine**.

## Creare un utente dedicato: molto più che cambiare semplicemente un nome

Potresti pensare che aggiungendo un singolo `USER` la linea al Dockerfile è sufficiente. Questo è sicuramente l’approccio più semplice, ma non è sufficientemente robusto.

### Creazione statica e mappatura dinamica

L'approccio di HagiCode è quello di creare un file `hagicode` utente con UID 1000, che di solito corrisponde all'utente predefinito sulla maggior parte delle macchine host:

```dockerfile
RUN groupadd -o -g 1000 hagicode && \
    useradd -o -u 1000 -g 1000 -s /bin/bash -m hagicode && \
    mkdir -p /home/hagicode/.claude && \
    chown -R hagicode:hagicode /home/hagicode
```

Ma questo risolve solo l'utente integrato all'interno dell'immagine. Cosa succede se l'utente host è UID 1001? È comunque necessario supportare la mappatura dinamica all'avvio del contenitore.

docker-entrypoint.sh contiene la chiave logica:

```bash
if [ -n "$PUID" ] && [ -n "$PGID" ]; then
    if ! id hagicode >/dev/null 2>&1; then
        groupadd -g "$PGID" hagicode
        useradd -u "$PUID" -g "$PGID" -s /bin/bash -m hagicode
    fi
fi
```

Il vantaggio di questo design è chiaro: **utilizza l'UID predefinito 1000 al momento della creazione dell'immagine, quindi regolalo dinamicamente in fase di runtime tramite il `PUID` e `PGID` variabili d'ambiente**. Indipendentemente dall'UID dell'utente host, la proprietà dei file di configurazione rimane corretta.

## La filosofia progettuale dei volumi persistenti

Ciascuno strumento AI CLI ha la propria directory di configurazione preferita, quindi devono essere mappati uno per uno:

| Strumento CLI | Percorso nel contenitore | Volume denominato |
|---------|-----------|--------|
| Claudio | `/home/hagicode/.claude` | `claude-data` |
| Codice | `/home/hagicode/.codex` | `codex-data` |
| OpenCode | `/home/hagicode/.config/opencode` | `opencode-config-data` |

Perché utilizzare volumi denominati invece di montaggi di associazione? Tre ragioni:

1. **Gestione più semplice**: i volumi denominati vengono gestiti automaticamente da Docker, quindi non è necessario creare manualmente le directory host.
2. **Isolamento dei permessi**: il contenuto iniziale dei volumi viene creato dall'utente all'interno del contenitore, evitando conflitti di permessi con l'host.
3. **Migrazione indipendente**: i volumi possono esistere indipendentemente dai contenitori, quindi i dati non vengono persi quando le immagini vengono aggiornate.

docker-compose-builder-web genera automaticamente la configurazione del volume corrispondente:

```yaml
volumes:
  claude-data:
  codex-data:
  opencode-config-data:

services:
  hagicode:
    volumes:
      - claude-data:/home/hagicode/.claude
      - codex-data:/home/hagicode/.codex
      - opencode-config-data:/home/hagicode/.config/opencode
    user: "${PUID:-1000}:${PGID:-1000}"
```

Presta attenzione a `user` campo qui: `PUID` e `PGID` vengono inseriti tramite variabili di ambiente per garantire che i processi all'interno del contenitore vengano eseguiti con un'identità che corrisponda all'utente host. Questo dettaglio è importante perché è difficile eseguire il debug dei problemi di autorizzazione una volta visualizzati.

## Gestione delle versioni: versioni integrate con override di runtime

Il blocco delle versioni delle immagini Docker è essenziale per la riproducibilità. Ma nello sviluppo reale, spesso abbiamo bisogno di testare una versione più recente o correggere urgentemente un bug. Se dovessimo ricostruire l’immagine ogni volta, il flusso di lavoro sarebbe decisamente troppo inefficiente.

La strategia di HagiCode è **versioni fisse come impostazione predefinita, con override di runtime come meccanismo di estensione**. È un compromesso ingegneristico pragmatico tra stabilità e flessibilità.

Dockerfile.template aggiunge le versioni dei pin qui:

```dockerfile
USER hagicode
WORKDIR /home/hagicode

# Configure the global npm install path
RUN mkdir -p /home/hagicode/.npm-global && \
    npm config set prefix '/home/hagicode/.npm-global'

# Install CLI tools using pinned versions
RUN npm install -g @anthropic-ai/claude-code@2.1.71 && \
    npm install -g @openai/codex@0.112.0 && \
    npm install -g opencode-ai@1.2.25 && \
    npm cache clean --force
```

docker-entrypoint.sh supporta gli override di runtime:

```bash
install_cli_override_if_needed() {
    local package_name="$2"
    local override_version="$5"

    if [ -n "$override_version" ]; then
        gosu hagicode npm install -g "${package_name}@${override_version}"
    fi
}

# Example usage
install_cli_override_if_needed "" "@anthropic-ai/claude-code" "" "" "${CLAUDE_CODE_CLI_VERSION}"
```

Ciò ti consente di testare una nuova versione tramite una variabile di ambiente senza ricostruire l'immagine:

```bash
docker run -e CLAUDE_CODE_CLI_VERSION=2.2.0 myimage
```

Questo design è pratico perché nessuno vuole ricostruire un'immagine ogni volta che testa una nuova funzionalità.

## Iniezione automatica della configurazione

Oltre a configurare manualmente gli strumenti CLI, alcuni scenari richiedono l'inserimento automatico della configurazione. L'esempio più tipico è un token API.

```bash
if [ -n "$ANTHROPIC_AUTH_TOKEN" ]; then
    mkdir -p /home/hagicode/.claude
    cat > /home/hagicode/.claude/settings.json <<EOF
{
  "env": {
    "ANTHROPIC_AUTH_TOKEN": "${ANTHROPIC_AUTH_TOKEN}"
  }
}
EOF
    chown -R hagicode:hagicode /home/hagicode/.claude
fi
```

Due cose contano qui: **passare informazioni sensibili attraverso variabili di ambiente invece di codificarle nell'immagine** e assicurarsi che la proprietà dei file di configurazione sia impostata correttamente, altrimenti gli strumenti CLI non saranno in grado di leggerli.

## Migliori pratiche e una lista di controllo delle trappole

### Problemi di mancata corrispondenza dei permessi

Questa è la trappola più facile in cui cadere. L'utente host ha UID 1001, mentre il contenitore utilizza 1000, quindi non è possibile accedere ai file creati su un lato dall'altro.

```bash
# Correct approach: make the container match the host user
docker run \
    -e PUID=$(id -u) \
    -e PGID=$(id -g) \
    myimage
```

Questo problema è molto comune e può essere frustrante la prima volta che lo incontri.

### La configurazione scompare dopo il riavvio del contenitore

Se ti ritrovi ad accedere nuovamente dopo ogni riavvio, controlla se hai dimenticato di montare un volume persistente:

```yaml
volumes:
  - claude-data:/home/hagicode/.claude
```

Niente è più frustrante che impostare attentamente una configurazione solo per vederla scomparire.

### Il modo giusto per aggiornare le versioni

Non correre `npm install -g` direttamente all'interno di un contenitore in esecuzione. Gli approcci corretti sono:

1. Imposta una variabile di ambiente per attivare l'installazione sostitutiva.
2. Oppure ricostruisci l'immagine.

```bash
# Option 1: runtime override
docker run -e CLAUDE_CODE_CLI_VERSION=2.2.0 myimage

# Option 2: rebuild the image
docker build -t myimage:v2 .
```

C'è più di una strada per Roma, ma alcune strade sono più agevoli di altre.

### Elenco di controllo per il rafforzamento della sicurezza

- Passa i token API attraverso le variabili di ambiente invece di scriverli nell'immagine.
- Imposta le autorizzazioni del file di configurazione su `600`.
- Esegui sempre l'applicazione come utente non root.
- Aggiorna regolarmente le versioni della CLI per correggere le vulnerabilità della sicurezza.

La sicurezza è sempre importante, ma la vera sfida è applicarla costantemente nella pratica.

## Estensione del supporto per i nuovi strumenti CLI

Se desideri supportare un nuovo strumento CLI in futuro, sono necessari solo tre passaggi:

1. **Dockerfile.template**: aggiunge il passaggio di installazione.
2. **docker-entrypoint.sh**: aggiunge la logica di override della versione.
3. **docker-compose-builder-web**: aggiunge la mappatura del volume persistente.

Questo design basato su modelli semplifica l'estensione senza modificare la logica di base.

## Conclusione

L'esecuzione degli strumenti AI CLI nei contenitori Docker comporta tre sfide principali: **autorizzazioni utente, persistenza della configurazione e gestione delle versioni**. Combinando utenti dedicati, isolamento dei volumi con nome e override basati su variabili di ambiente, il progetto HagiCode ha creato un'architettura di distribuzione sicura e flessibile.

Punti chiave della progettazione:

- **Isolamento utente**: crea un utente dedicato durante la fase di creazione dell'immagine, con supporto runtime per la dinamica `PUID`/`PGID` mappatura.
- **Strategia di persistenza**: ogni strumento CLI ottiene il proprio volume denominato, quindi i riavvii non influiscono sulla configurazione.
- **Flessibilità della versione**: le impostazioni predefinite fisse garantiscono la riproducibilità, mentre le sostituzioni di runtime offrono spazio per i test.
- **Configurazione automatizzata**: la configurazione sensibile può essere inserita automaticamente tramite variabili di ambiente.

Questa soluzione gira stabilmente nel progetto HagiCode da qualche tempo, e spero che offra utili punti di riferimento per sviluppatori con esigenze simili.
