## Aggiunta del supporto per la distribuzione automatica di GitHub Pages a HagiCode

> Questo progetto aveva originariamente il nome in codice PCode e ora è stato ufficialmente ribattezzato HagiCode. Questo articolo documenta come abbiamo introdotto la distribuzione automatizzata dei siti statici in modo che la pubblicazione di contenuti diventi facile come bere acqua.

## Contesto/Introduzione

Durante lo sviluppo di HagiCode, ci siamo imbattuti in un problema molto pratico: man mano che la nostra documentazione e le nostre proposte continuavano a crescere, gestire e presentare in modo efficiente tali contenuti diventava sempre più urgente. Abbiamo deciso di utilizzare GitHub Pages per ospitare il nostro sito statico, ma crearlo e distribuirlo manualmente era semplicemente troppo complicato. Ogni modifica richiedeva una compilazione locale, un confezionamento e quindi un push manuale al file `gh-pages` ramo. Non solo era inefficiente, ma era anche facile sbagliare.

Per risolvere questo problema, principalmente perché volevamo risparmiare qualche sforzo, avevamo bisogno di un flusso di lavoro di distribuzione automatizzato. Questo articolo descrive in dettaglio come abbiamo aggiunto il supporto per la distribuzione automatica basato su GitHub Actions al progetto HagiCode, così possiamo concentrarci sulla scrittura dei contenuti e lasciare che l'automazione gestisca il resto.

## Analisi degli obiettivi

Prima di passare all'implementazione, dobbiamo innanzitutto chiarire cosa dovrebbe effettivamente realizzare questo compito. Dopotutto, affilare l'ascia non ritarda il taglio della legna da ardere.

### Requisiti fondamentali

1.  **Build automatizzata**: attiva automaticamente il flusso di lavoro di build quando il codice viene inviato a `main` ramo.
2.  **Distribuzione automatizzata**: dopo una compilazione riuscita, distribuisci automaticamente i file statici generati su GitHub Pages.
3.  **Coerenza dell'ambiente**: assicurati che l'ambiente CI corrisponda all'ambiente di compilazione locale per evitare il classico imbarazzo di "funziona localmente, ma fallisce online".

### Scelta tecnica

Poiché HagiCode è basato su Docusaurus, un generatore di siti statici React molto popolare, possiamo utilizzare GitHub Actions per raggiungere questo obiettivo.

## Configurazione del flusso di lavoro delle azioni GitHub

GitHub Actions è il servizio CI/CD fornito da GitHub. Definendo i file del flusso di lavoro in formato YAML all'interno del repository, possiamo personalizzare tutti i tipi di attività di automazione.

### Creazione del file del flusso di lavoro

Dobbiamo creare un nuovo file di configurazione nel file `.github/workflows` directory nella radice del progetto, ad esempio `deploy.yml`. Se la directory non esiste ancora, creala prima.

La logica principale di questo file di configurazione è la seguente:

1.  **Condizione di attivazione**: Ascolta `push` eventi sul `main` ramo.
2.  **Ambiente runtime**: l'ultima versione di Ubuntu.
3.  **Crea passaggi**:
    -   Controlla il codice
    -   Installa Node.js
    -   Installa le dipendenze (`npm install`)
    -   Costruisci i file statici (`npm run build`)
4.  **Passaggio di distribuzione**: utilizzare il file official `action-gh-pages` azione per inviare l'output di compilazione al file `gh-pages` ramo.

### Codice di configurazione chiave

Di seguito è riportato il modello di configurazione che alla fine abbiamo adottato:

```yaml
name: Deploy to GitHub Pages

# Trigger condition: run when code is pushed to the main branch
on:
  push:
    branches:
      - main
    # Add path filters if needed, for example only build when docs change
    # paths:
    #   - 'docs/**'
    #   - 'package.json'

# Set permissions, which is important for GitHub Pages deployment
permissions:
  contents: read
  pages: write
  id-token: write

# Concurrency control: cancel older builds for the same branch
concurrency:
  group: "pages"
  cancel-in-progress: false

jobs:
  build:
    runs-on: ubuntu-latest
    steps:
      - name: Checkout
        uses: actions/checkout@v4
        # Note: fetch-depth: 0 is required, otherwise the build version may be inaccurate
        with:
          fetch-depth: 0

      - name: Setup Node
        uses: actions/setup-node@v4
        with:
          node-version: 20 # Keep this consistent with the local development environment
          cache: 'npm'     # Enabling cache can speed up the build process

      - name: Install dependencies
        run: npm ci
        # Use npm ci instead of npm install because it is faster and stricter, which makes it better for CI

      - name: Build website
        run: npm run build
        env:
          # Configure environment variables here if your site build requires them
          # NODE_ENV: production
          # PUBLIC_URL: /your-repo-name
          
      - name: Upload artifact
        uses: actions/upload-pages-artifact@v3
        with:
          path: ./build # Default Docusaurus output directory

  deploy:
    environment:
      name: github-pages
      url: ${{ steps.deployment.outputs.page_url }}
    runs-on: ubuntu-latest
    needs: build
    steps:
      - name: Deploy to GitHub Pages
        id: deployment
        uses: actions/deploy-pages@v4
```

## Insidie ​​comuni durante l'implementazione

In pratica, abbiamo riscontrato diversi problemi. Li condivido qui nella speranza che altri possano evitarli, o almeno preparare soluzioni in anticipo.

### 1. Problemi di autorizzazione del token GitHub

Quando abbiamo impostato le cose per la prima volta, la distribuzione continuava a fallire con un 403 (Proibito). Dopo molte ricerche, abbiamo scoperto che l'impostazione predefinita di GitHub `GITHUB_TOKEN` non aveva il permesso di scrivere su Pages.

**Soluzione**: nel repository `Settings` -> `Actions` -> `General` -> `Workflow permissions`, assicurati di selezionare **"Autorizzazioni di lettura e scrittura"**.

### 2. Percorso della directory di build errato

Docusaurus inserisce file statici creati nel file `build` directory per impostazione predefinita. Tuttavia, alcuni progetti potrebbero essere configurati in modo diverso. Ad esempio, per impostazione predefinita Crea app React è `build`, mentre Vite per impostazione predefinita è `dist`. Se Actions segnala che non è possibile trovare i file di output, controlla il percorso di output `docusaurus.config.js`.

### 3. Problemi di sottopercorso

Se il tuo repository non è una home page utente, significa che non lo è `username.github.io`, ma invece una pagina di progetto come `username.github.io/project-name`, quindi è necessario configurare `baseUrl`.

Dentro `docusaurus.config.js`:

```javascript
module.exports = {
  // ...
  url: 'https://hagicode.com', // Your Hagicode URL
  baseUrl: '/', // Deploy from the root path
  // ...
};
```
Questo dettaglio è facile da trascurare. Se è configurato in modo errato, la pagina potrebbe caricarsi come una schermata vuota perché i percorsi delle risorse non possono essere risolti.

## Verifica del risultato

Dopo aver configurato tutto e aver inviato il codice, possiamo andare alla scheda **Azioni** del repository GitHub e guardare lo spettacolo.

Vedrai un cerchio giallo mentre il flusso di lavoro è in esecuzione e una volta che diventa verde significa successo. Se diventa rosso, fai clic e controlla i registri. Nella maggior parte dei casi, puoi rintracciare il problema lì, solitamente un errore di battitura o una configurazione errata del percorso.

Una volta completata la compilazione, visita `https://<your-username>.github.io/<repository-name>/` e vedrai il sito appena distribuito.

## Riepilogo

Introducendo GitHub Actions, abbiamo automatizzato con successo la distribuzione per il sito di documentazione di HagiCode. Ciò non solo consente di risparmiare il tempo precedentemente dedicato alle operazioni manuali, ma, cosa ancora più importante, standardizza il flusso di lavoro di rilascio. Ora, indipendentemente da quale membro del team aggiorni la documentazione, una volta incorporata la modifica `main`, i contenuti più recenti saranno pubblicati pochi minuti dopo.

**Vantaggi principali**:
- **Maggiore efficienza**: siamo passati da "confezionamento manuale e caricamento manuale" a "codice uguale a rilascio".
- **Meno errori**: abbiamo eliminato la possibilità di errore umano durante le operazioni manuali.
- **Migliore esperienza per gli sviluppatori**: gli sviluppatori possono concentrarsi maggiormente sulla qualità dei contenuti invece di farsi distrarre da noiose fasi di distribuzione.

Sebbene la configurazione CI/CD sia inizialmente un po' problematica, soprattutto con tutte le autorizzazioni e i dettagli del percorso, si tratta di un investimento una tantum con importanti ritorni a lungo termine. Consiglio vivamente che ogni progetto di sito statico adotti un flusso di lavoro automatizzato simile.
