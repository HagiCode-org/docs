## Dalle chiamate CLI all'integrazione dell'SDK: best practice per GitHub Copilot nei progetti .NET

> Il percorso di aggiornamento dall'invocazione della riga di comando all'integrazione ufficiale dell'SDK è stato un viaggio piuttosto impegnativo. Oggi voglio condividere le trappole in cui ci siamo imbattuti e ciò che abbiamo imparato durante la creazione di HagiCode.

## Sfondo

Dopo il rilascio ufficiale dell'SDK GitHub Copilot nel 2025, abbiamo iniziato a integrarlo nel nostro livello di capacità AI. Prima di ciò, il progetto utilizzava principalmente GitHub Copilot invocando direttamente lo strumento da riga di comando Copilot CLI, ma tale approccio presentava diversi ovvi problemi:

- **Gestione complessa dei processi**: abbiamo dovuto gestire manualmente il ciclo di vita del processo CLI, i timeout di avvio e la pulizia del processo. I processi possono bloccarsi senza preavviso.
- **Gestione degli eventi incompleta**: l'invocazione della CLI non elaborata rende difficile acquisire eventi a grana fine dal ragionamento del modello e dall'esecuzione dello strumento. È come vedere solo il risultato senza il processo di pensiero.
- **Gestione difficile delle sessioni**: non esisteva un meccanismo efficace per il riutilizzo e il ripristino delle sessioni, quindi ogni interazione doveva ricominciare da capo.
- **Problemi di compatibilità**: gli argomenti della CLI cambiavano frequentemente, il che significava che dovevamo mantenere costantemente la logica di compatibilità per tali parametri.

Questi problemi sono diventati sempre più evidenti nello sviluppo quotidiano, soprattutto quando avevamo bisogno di tenere traccia del ragionamento del modello (`thinking`) e stato di esecuzione dello strumento in tempo reale. A quel punto è diventato chiaro che avevamo bisogno di un approccio di integrazione di livello inferiore e più completo.

## A proposito di HagiCode

L'approccio condiviso in questo articolo deriva dalla nostra esperienza pratica nel [HagiCode](https://hagicode.com) progetto. HagiCode è un progetto di assistente alla codifica AI open source e durante lo sviluppo avevamo bisogno di una profonda integrazione con le funzionalità di GitHub Copilot, dal completamento del codice di base alle complesse conversazioni multi-turno e alle chiamate di strumenti. Questi requisiti del mondo reale ci hanno spinto a passare dall'invocazione della CLI all'SDK ufficiale.

Se questa implementazione ti sembra utile, ci sono buone probabilità che la nostra esperienza ingegneristica possa aiutarti. In tal caso, potrebbe valere la pena dare un’occhiata anche al progetto HagiCode stesso. Potresti anche trovare ulteriori informazioni e collegamenti sul progetto alla fine di questo articolo.

## Progettazione dell'architettura

Il progetto utilizza un'architettura a più livelli per affrontare le limitazioni dell'invocazione della CLI:

```
┌─────────────────────────────────────────────────────────┐
│  hagicode-core (Orleans Grains + AI Provider Layer)    │
│  - CopilotAIProvider: Converts AIRequest to CopilotOptions │
│  - GitHubCopilotGrain: Orleans distributed execution interface │
└─────────────────────────────────────────────────────────┘
                          ↓
┌─────────────────────────────────────────────────────────┐
│  HagiCode.Libs (Shared Provider Layer)                 │
│  - CopilotProvider: CLI Provider interface implementation │
│  - ICopilotSdkGateway: SDK invocation abstraction      │
│  - GitHubCopilotSdkGateway: SDK session management and event dispatch │
└─────────────────────────────────────────────────────────┘
                          ↓
┌─────────────────────────────────────────────────────────┐
│  GitHub Copilot SDK (Official .NET SDK)                │
│  - CopilotClient: SDK client                           │
│  - CopilotSession: Session management                  │
│  - SessionEvent: Event stream                          │
└─────────────────────────────────────────────────────────┘
```

Questo design a strati offre numerosi vantaggi tecnici pratici:

1. **Separazione degli aspetti**: la logica aziendale principale è disaccoppiata dai dettagli di implementazione dell'SDK.
2. **Testabilità**: Il `ICopilotSdkGateway` l'interfaccia semplifica i test unitari.
3. **Riutilizzabilità**: `HagiCode.Libs` può essere referenziato da più progetti.
4. **Manutenibilità**: gli aggiornamenti dell'SDK richiedono solo modifiche al livello Gateway, mentre i livelli superiori rimangono intatti.

## Implementazione principale

### Flusso di autenticazione

L'autenticazione è il primo e più importante passaggio dell'integrazione dell'SDK. Se l'autenticazione fallisce, nient'altro conta. Abbiamo progettato una configurazione di autenticazione flessibile che supporta più origini di autenticazione:

```csharp
// CopilotProvider.cs - Authentication source configuration
public class CopilotOptions
{
    public bool UseLoggedInUser { get; set; } = true;
    public string? GitHubToken { get; set; }
    public string? CliUrl { get; set; }
}

// Convert to SDK request
return new CopilotSdkRequest(
    GitHubToken: options.AuthSource == CopilotAuthSource.GitHubToken
        ? options.GitHubToken
        : null,
    UseLoggedInUser: options.AuthSource != CopilotAuthSource.GitHubToken
);
```

I vantaggi di questo design sono abbastanza chiari:

- Supporta la modalità utente connesso senza richiedere un token, che si adatta bene agli scenari desktop.
- Supporta la modalità GitHub Token, adatta per distribuzioni lato server e gestione centralizzata.
- Supporta l'override dell'URL della CLI di Copilot, che aiuta con la configurazione del proxy aziendale.

In pratica, questo modello di autenticazione flessibile ha notevolmente semplificato la configurazione in diversi scenari di distribuzione. Il client desktop può utilizzare lo stato di accesso Copilot di ciascun utente, mentre il lato server può gestire l'autenticazione centralmente tramite token.

### Gestione del flusso di eventi

Una delle funzionalità più potenti dell'SDK è l'acquisizione completa del flusso di eventi. Abbiamo implementato un sistema di invio eventi in grado di gestire tutti i tipi di eventi SDK in tempo reale:

```csharp
// GitHubCopilotSdkGateway.cs - Core logic for event dispatch
internal static SessionEventDispatchResult DispatchSessionEvent(
    SessionEvent evt, bool sawDelta)
{
    switch (evt)
    {
        case AssistantReasoningEvent reasoningEvent:
            // Capture the model reasoning process
            events.Add(new CopilotSdkStreamEvent(
                CopilotSdkStreamEventType.ReasoningDelta,
                Content: reasoningEvent.Data.Content));
            break;

        case ToolExecutionStartEvent toolStartEvent:
            // Capture the start of a tool invocation
            events.Add(new CopilotSdkStreamEvent(
                CopilotSdkStreamEventType.ToolExecutionStart,
                ToolName: toolStartEvent.Data.ToolName,
                ToolCallId: toolStartEvent.Data.ToolCallId));
            break;

        case ToolExecutionCompleteEvent toolCompleteEvent:
            // Capture tool completion and its result
            events.Add(new CopilotSdkStreamEvent(
                CopilotSdkStreamEventType.ToolExecutionEnd,
                Content: ExtractToolExecutionContent(toolCompleteEvent)));
            break;

        default:
            // Preserve unhandled events as RawEvent
            events.Add(new CopilotSdkStreamEvent(
                CopilotSdkStreamEventType.RawEvent,
                RawEventType: evt.GetType().Name));
            break;
    }
}
```

Il valore di questa implementazione è significativo:

- **Cattura completa del processo di ragionamento del modello** (`thinking`): gli utenti possono vedere come ragiona l'intelligenza artificiale, non solo la risposta finale.
- **Tracciamento in tempo reale dello stato di esecuzione degli strumenti**: sappiamo quali strumenti sono in esecuzione, quando finiscono e cosa restituiscono.
- **Zero perdite per eventi**: ricorrendo a `RawEvent`, ci assicuriamo che ogni evento venga registrato.

In HagiCode, questi eventi a grana fine aiutano gli utenti a capire come funziona internamente l'intelligenza artificiale, soprattutto durante il debug di attività complesse.

### Gestione della compatibilità CLI

Dopo la migrazione dalla chiamata della CLI all'SDK, abbiamo riscontrato che alcuni parametri della CLI esistenti non venivano più applicati nell'SDK. Per preservare la compatibilità con le versioni precedenti, abbiamo implementato un sistema di filtraggio dei parametri:

```csharp
// CopilotCliCompatibility.cs - Argument filtering
private static readonly Dictionary<string, string> RejectedFlags = new()
{
    ["--headless"] = "Unsupported startup argument",
    ["--model"] = "Passed through an SDK-native field",
    ["--prompt"] = "Passed through an SDK-native field",
    ["--interactive"] = "Interaction is managed by the provider",
};

public static CopilotCliArgumentBuildResult BuildCliArgs(CopilotOptions options)
{
    // Filter out unsupported arguments and keep compatible ones
    // Generate diagnostic information
}
```

Questo ci offre diversi vantaggi:

- Filtra automaticamente gli argomenti CLI incompatibili per prevenire errori di runtime.
- Genera messaggi diagnostici chiari per aiutare gli sviluppatori a individuare rapidamente i problemi.
- Mantiene l'SDK stabile e isolato dalle modifiche dei parametri CLI.

Durante l'aggiornamento, questo meccanismo di compatibilità ci ha aiutato a effettuare una transizione senza intoppi. I file di configurazione esistenti potevano ancora essere utilizzati e necessitavano solo di aggiustamenti graduali in base all'output diagnostico.

### Raggruppamento di runtime

La creazione di sessioni dell'SDK di Copilot è relativamente costosa e la creazione e l'eliminazione di sessioni troppo spesso influiscono negativamente sulle prestazioni. Per risolvere questo problema, abbiamo implementato un sistema di gestione del pool di sessioni:

```csharp
// CopilotProvider.cs - Session pool management
await using var lease = await _poolCoordinator.AcquireCopilotRuntimeAsync(
    request,
    async ct => await _gateway.CreateRuntimeAsync(sdkRequest, ct),
    cancellationToken);

if (lease.IsWarmLease)
{
    // Reuse an existing session
    yield return CreateSessionReusedMessage();
}

await foreach (var eventData in lease.Entry.Resource.SendPromptAsync(...))
{
    yield return MapEvent(eventData);
}
```

I vantaggi del pooling delle sessioni includono:

- **Riutilizzo della sessione**: richieste con lo stesso `sessionId` può riutilizzare le sessioni esistenti e ridurre il sovraccarico di avvio.
- **Supporto per il recupero della sessione**: dopo un'interruzione della rete, è possibile ripristinare lo stato della sessione precedente.
- **Gestione automatica del pooling**: le sessioni scadute vengono pulite automaticamente per evitare perdite di risorse.

In HagiCode, il pooling delle sessioni ha notevolmente migliorato la reattività, soprattutto per le conversazioni continue.

### Integrazione di Orléans

HagiCode utilizza Orleans come framework distribuito e abbiamo integrato Copilot SDK in Orleans Grains:

```csharp
// GitHubCopilotGrain.cs - Distributed execution
public async IAsyncEnumerable<GitHubCopilotResponse> ExecuteCommandStreamAsync(
    string command,
    CancellationToken token = default)
{
    var provider = await aiProviderFactory.GetProviderAsync(AIProviderType.GitHubCopilot);

    await foreach (var chunk in provider.SendMessageAsync(request, null, token))
    {
        // Map to the unified response format
        yield return BuildChunkResponse(chunk, startedAt);
    }
}
```

I vantaggi dell’integrazione di Orleans sono sostanziali:

- **Astrazione unificata del fornitore di intelligenza artificiale**: diventa facile passare da un fornitore di intelligenza artificiale all'altro.
- **Isolamento multi-tenant**: le sessioni Copilot per utenti diversi rimangono isolate l'una dall'altra.
- **Stato sessione persistente**: lo stato della sessione può essere ripristinato anche dopo il riavvio del server.

Per gli scenari che devono gestire un numero elevato di richieste simultanee, Orleans offre una forte scalabilità.

Ecco un esempio di configurazione completa:

```json
{

          "UseLoggedInUser": true,

            "AllowAllTools": false,

            "DeniedTools": ["Edit"]
          }
        }
      }
    }
  }
}
```

### Note sull'utilizzo

Nell'utilizzo nel mondo reale, abbiamo riassunto diversi punti a cui vale la pena prestare attenzione:

**Configurazione del timeout di avvio**: il primo avvio di Copilot CLI può richiedere un tempo relativamente lungo, pertanto si consiglia di impostare `StartupTimeout` ad almeno 30 secondi. Se questo è il primo accesso, potrebbe essere necessario più tempo.

**Gestione delle autorizzazioni**: negli ambienti di produzione, evitare l'utilizzo `AllowAllTools: true`. Usa il `AllowedTools` lista consentita per controllare quali strumenti sono disponibili e utilizzare il file `DeniedTools` lista negata per bloccare operazioni pericolose. Ciò impedisce efficacemente all'IA di eseguire comandi rischiosi.

**Gestione sessioni**: Richieste con lo stesso `sessionId` riutilizzare automaticamente le sessioni. Lo stato della sessione viene mantenuto `ProviderSessionId`. La cancellazione viene propagata tramite `CancellationTokenSource`.

**Output diagnostico**: argomenti CLI incompatibili generano messaggi di tipo `diagnostic`. Gli eventi SDK non elaborati vengono conservati come `event.raw`. I messaggi di errore includono categorie come timeout di avvio e incompatibilità degli argomenti per semplificare la risoluzione dei problemi.

### Migliori pratiche

Sulla base della nostra esperienza pratica, ecco alcune migliori pratiche:

**1. Utilizza una lista consentita di strumenti**

```csharp
var request = new AIRequest
{
    Prompt = "Analyze this file",
    AllowedTools = new[] { "Read", "Grep", "Bash(git:*)" }
};
```

Specificare esplicitamente gli strumenti consentiti tramite una lista consentita aiuta a prevenire azioni IA impreviste. Ciò è particolarmente importante per gli strumenti con autorizzazioni di scrittura, come `Edit`, che dovrebbe essere maneggiato con particolare attenzione.

**2. Imposta timeout ragionevoli**

```csharp
options.Timeout = 3600;  // 1 hour
options.StartupTimeout = 60;  // 1 minute
```

Impostare valori di timeout appropriati in base alla complessità dell'attività. Se il valore è troppo basso, le attività potrebbero essere interrotte. Se è troppo lungo, le risorse potrebbero essere sprecate in attesa di richieste che non rispondono.

**3. Abilita riutilizzo sessione**

```csharp
options.SessionId = "my-session-123";
```

Usando lo stesso `sessionId` per le attività correlate consente di riutilizzare il contesto della sessione precedente e migliorare la velocità di risposta.

**4. Gestisci le risposte in streaming**

```csharp
await foreach (var chunk in provider.StreamAsync(request))
{
    switch (chunk.Type)
    {
        case StreamingChunkType.ThinkingDelta:
            // Handle the reasoning process
            break;
        case StreamingChunkType.ToolCallDelta:
            // Handle tool invocation
            break;
        case StreamingChunkType.ContentDelta:
            // Handle text output
            break;
    }
}
```

Le risposte in streaming ti consentono di mostrare i progressi dell'elaborazione dell'intelligenza artificiale in tempo reale, migliorando l'esperienza dell'utente. Ciò è particolarmente utile per le attività di lunga durata.

**5. Gestione degli errori e tentativi**

```csharp
try
{
    await foreach (var chunk in provider.StreamAsync(request))
    {
        // Handle the response
    }
}
catch (CopilotSessionException ex)
{
    // Handle session exceptions
    logger.LogError(ex, "Copilot session failed");
    // Decide whether to retry based on the exception type
}
```

La corretta gestione degli errori e la logica dei tentativi migliorano la stabilità complessiva del sistema.

## Conclusione

L'aggiornamento dall'invocazione della CLI all'integrazione dell'SDK ha apportato un valore sostanziale al progetto HagiCode:

- **Stabilità migliorata**: l'SDK fornisce un'interfaccia più stabile che non è influenzata dalle modifiche alla versione della CLI.
- **Funzionalità più completa**: acquisisce l'intero flusso di eventi, inclusi il ragionamento e lo stato di esecuzione dello strumento.
- **Maggiore efficienza di sviluppo**: l'interfaccia SDK indipendente dai tipi rende lo sviluppo più efficiente e riduce gli errori di runtime.
- **Migliore esperienza utente**: il feedback sugli eventi in tempo reale offre agli utenti una comprensione più chiara di ciò che sta facendo l'intelligenza artificiale.

Questo aggiornamento non è stato solo una sostituzione dell'implementazione tecnica. Si è trattato anche di un'ottimizzazione dell'architettura dell'intero livello di capacità dell'intelligenza artificiale. Attraverso la progettazione a più livelli e le interfacce di astrazione, abbiamo ottenuto una migliore manutenibilità ed estensibilità.

Se stai pensando di integrare GitHub Copilot nel tuo progetto .NET, spero che queste lezioni pratiche ti aiutino a evitare deviazioni inutili. L'SDK ufficiale è infatti più stabile e completo dell'invocazione della CLI e vale la pena dedicare del tempo a comprenderlo e adottarlo.
