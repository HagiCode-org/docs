## Progettazione dell'architettura

### Panoramica dell'architettura a più livelli

L'integrazione tra C# e CodeBuddy utilizza un'architettura a livelli chiara. Questo design rende esplicite le responsabilità e rende molto più semplice la manutenzione a lungo termine:

```
┌─────────────────────────────────────────────┐
│           Provider Contract Layer           │
│   AIProviderType enum + extension methods   │
├─────────────────────────────────────────────┤
│           Provider Factory Layer            │
│   AIProviderFactory dependency injection factory │
├─────────────────────────────────────────────┤
│           Provider Implementation Layer     │
│   CodebuddyCliProvider concrete implementation │
├─────────────────────────────────────────────┤
│           ACP Infrastructure Layer          │
│  ACPSessionManager / StdioAcpTransport      │
│  AcpRpcClient / AcpAgentClient              │
└─────────────────────────────────────────────┘
```

Quali sono i vantaggi di questa stratificazione? In parole povere, ogni strato rimane lontano dagli altri. Se in seguito vogliamo cambiare il meccanismo di comunicazione, ad esempio from `stdio` a WebSocket, dobbiamo solo modificare il livello inferiore e la logica aziendale sopra di esso rimane intatta. Nessuno vuole che un cambiamento nella comunicazione si ripercuota sull'intero codice base.

### Ripartizione dei componenti principali

Il **livello del contratto del fornitore** è il fondamento dell'intera architettura. Definiamo il `AIProviderType` enum, dove `CodebuddyCli = 3` viene utilizzato come valore enum e implementa la mappatura bidirezionale tra stringhe ed enumerazioni tramite metodi di estensione. Ciò consente di convertire comodamente le stringhe nei file di configurazione in enumerazioni e di riconvertire le enumerazioni in stringhe per il debug dell'output.

Il **livello factory del provider** è responsabile della creazione dell'istanza del provider corrispondente in base alla configurazione. Utilizza l'inserimento delle dipendenze .NET insieme a `ActivatorUtilities.CreateInstance` per la creazione dinamica. Il vantaggio del modello factory è che quando si aggiunge un nuovo Provider, è necessario aggiungere solo la logica di creazione invece di modificare il codice esistente.

Il **livello di implementazione del provider** è il luogo in cui avviene il lavoro vero e proprio. `CodebuddyCliProvider` implementa il `IAIProvider` interfaccia e fornisce due modalità di invocazione: `ExecuteAsync` per chiamate non in streaming e `StreamAsync` per le chiamate in streaming.

Il **livello dell'infrastruttura ACP** fornisce la base di comunicazione sottostante. Questo livello gestisce tutti i dettagli del protocollo, inclusa la gestione dei processi, la serializzazione dei messaggi e l'analisi delle risposte. È il fondamento che mantiene stabile tutto ciò che sta sopra di esso.

## Meccanismo di comunicazione

### Modalità di trasporto Stdio

CodeBuddy utilizza **Stdio (input/output standard)** per comunicare con processi esterni. Il comando di avvio è semplice:

```bash
codebuddy --acp
```

Successivamente, i messaggi JSON-RPC vengono scambiati tramite input e output standard. Questo approccio presenta diversi vantaggi:

1. **Avvio rapido**: la comunicazione del processo locale evita la latenza della rete
2. **Configurazione semplice**: devi solo specificare il percorso dell'eseguibile
3. **Isolamento dell'ambiente**: ogni sessione viene eseguita in un processo indipendente, quindi non si influenzano a vicenda

L'inserimento di variabili d'ambiente è supportato durante la comunicazione. Esempi comuni includono:

- `CODEBUDDY_API_KEY`: autenticazione della chiave API
- `CODEBUDDY_INTERNET_ENVIRONMENT`: configurazione dell'ambiente di rete

Come per la comunicazione tra le persone, è utile scegliere prima un canale conveniente.

### Protocollo di messaggio

ACP è basato su JSON-RPC 2.0. Il formato del messaggio è più o meno questo:

```json
// Request message
{
  "jsonrpc": "2.0",
  "id": 1,
  "method": "agent/prompt",
  "params": {
    "prompt": "Help me write a sorting algorithm",
    "sessionId": "session-123"
  }
}

// Response message
{
  "jsonrpc": "2.0",
  "id": 1,
  "result": {
    "content": "Here is the AI response..."
  }
}
```

Nell'implementazione reale, incapsuliamo tutti questi dettagli del protocollo in modo che il livello aziendale superiore debba preoccuparsi solo della tempestività e della risposta.

## Implementazione principale

### 1. Ripristina il contratto del fornitore

Innanzitutto, ripristina il tipo CodeBuddy nel file enum:

```csharp
// PCode.Models/AIProviderType.cs
public enum AIProviderType
{
    ClaudeCodeCli = 0,
    CodexCli = 1,
    GitHubCopilot = 2,
    CodebuddyCli = 3,  // Restore this enum value
    OpenCodeCli = 4,
    IFlowCli = 5,
}
```

Quindi aggiungi la mappatura delle stringhe nei metodi di estensione in modo che il file di configurazione possa specificare il provider tramite stringa:

```csharp
// AIProviderTypeExtensions.cs
private static readonly Dictionary<string, AIProviderType> _typeMap = new(
    StringComparer.OrdinalIgnoreCase)
{
    ["CodebuddyCli"] = AIProviderType.CodebuddyCli,
    ["Codebuddy"] = AIProviderType.CodebuddyCli,
    ["codebuddy"] = AIProviderType.CodebuddyCli,
    // ... Mappings for other providers
};
```

### 2. Integrare la Provider Factory

Aggiungi un ramo di creazione CodeBuddy nella classe factory:

```csharp
// AIProviderFactory.cs
private IAIProvider? CreateProvider(AIProviderType providerType, ProviderConfiguration config)
{
    return providerType switch
    {
        AIProviderType.CodebuddyCli =>
            ActivatorUtilities.CreateInstance<CodebuddyCliProvider>(
                _serviceProvider,
                Options.Create(config)),
        // ... Other providers
        _ => throw new NotSupportedException($"Provider {providerType} not supported")
    };
}
```

Questo utilizza l'iniezione delle dipendenze `ActivatorUtilities`, che gestisce automaticamente l'inserimento dei parametri del costruttore ed è molto conveniente.

### 3. Completare l'implementazione del provider

Di seguito è riportata l'implementazione principale di `CodebuddyCliProvider`, che copre sia le modalità di chiamata in streaming che quelle non in streaming:

```csharp
public class CodebuddyCliProvider : IAIProvider
{
    private readonly ILogger<CodebuddyCliProvider> _logger;
    private readonly IACPSessionManager _sessionManager;
    private readonly ProviderConfiguration _config;

    public string Name => "CodebuddyCli";
    public bool SupportsStreaming => true;
    public ProviderCapabilities Capabilities { get; }

    public CodebuddyCliProvider(
        ILogger<CodebuddyCliProvider> logger,
        IACPSessionManager sessionManager,
        IOptions<ProviderConfiguration> config)
    {
        _logger = logger;
        _sessionManager = sessionManager;
        _config = config.Value;

        // Define the capabilities of the current Provider
        Capabilities = new ProviderCapabilities
        {
            SupportsStreaming = true,
            SupportsTools = true,
            SupportsSystemMessages = true,
            SupportsArtifacts = false,
            MaxTokens = 8192
        };
    }

    // Non-streaming call: return all results together after completion
    public async Task<AIResponse> ExecuteAsync(
        AIRequest request,
        CancellationToken cancellationToken = default)
    {
        // Create an independent session for the request
        var session = await _sessionManager.CreateSessionAsync(
            "CodebuddyCli",
            request.WorkingDirectory,
            cancellationToken,
            request.SessionId);

        try
        {
            var fullPrompt = BuildPrompt(request);
            await session.SendPromptAsync(fullPrompt, cancellationToken);

            var responseBuilder = new StringBuilder();
            var toolCalls = new List<AIToolCall>();

            // Collect all response chunks
            await foreach (var chunk in StreamFromSession(session, cancellationToken))
            {
                if (!string.IsNullOrEmpty(chunk.Content))
                {
                    responseBuilder.Append(chunk.Content);
                }
                // Handle tool calls...
            }

            return new AIResponse
            {
                Content = AIResultContentSanitizer.SanitizeResultContent(
                    responseBuilder.ToString()),
                ToolCalls = toolCalls,
                Provider = Name,
                Model = string.Empty
            };
        }
        finally
        {
            // Release session resources
            await session.DisposeAsync();
        }
    }

    // Streaming call: return response chunks in real time
    public async IAsyncEnumerable<AIStreamingChunk> StreamAsync(
        AIRequest request,
        [EnumeratorCancellation] CancellationToken cancellationToken = default)
    {
        var session = await _sessionManager.CreateSessionAsync(
            "CodebuddyCli",
            request.WorkingDirectory,
            cancellationToken);

        try
        {
            var fullPrompt = BuildPrompt(request);
            await session.SendPromptAsync(fullPrompt, cancellationToken);

            await foreach (var chunk in StreamFromSession(session, cancellationToken))
            {
                yield return chunk;
            }
        }
        finally
        {
            await session.DisposeAsync();
        }
    }

    private async IAsyncEnumerable<AIStreamingChunk> StreamFromSession(
        IACPSession session,
        [EnumeratorCancellation] CancellationToken cancellationToken)
    {
        // Iterate through all updates in the session
        await foreach (var notification in session.ReceiveUpdatesAsync(cancellationToken))
        {
            switch (notification.Update)
            {
                case AgentMessageChunkSessionUpdate agentMessage:
                    // Handle text content chunks
                    if (agentMessage.Content is AcpImp.TextContentBlock textContent)
                    {
                        yield return new AIStreamingChunk
                        {
                            Content = textContent.Text,
                            Type = StreamingChunkType.ContentDelta,
                            IsComplete = false
                        };
                    }
                    break;

                case ToolCallSessionUpdate toolCall:
                    // Handle tool calls
                    yield return new AIStreamingChunk
                    {
                        Content = string.Empty,
                        Type = StreamingChunkType.ToolCallDelta,
                        ToolCallDelta = new AIToolCallDelta
                        {
                            Id = toolCall.ToolCallId,
                            Name = toolCall.Kind.ToString(),
                            Arguments = toolCall.RawInput?.ToString()
                        }
                    };
                    break;

                case AcpImp.PromptCompletedSessionUpdate:
                    // Response complete
                    yield break;
            }
        }
    }

    // Build the full prompt
    private string BuildPrompt(AIRequest request, string? embeddedCommandPrompt = null)
    {
        var sb = new StringBuilder();

        // Embedded command prompt, if present
        if (!string.IsNullOrEmpty(embeddedCommandPrompt))
        {
            sb.AppendLine(embeddedCommandPrompt);
            sb.AppendLine();
        }

        // System message
        if (!string.IsNullOrEmpty(request.SystemMessage))
        {
            sb.AppendLine(request.SystemMessage);
            sb.AppendLine();
        }

        // User prompt
        sb.Append(request.Prompt);
        return sb.ToString();
    }
}
```

Ci sono diversi punti chiave in questo codice:

1. **Gestione delle sessioni**: ogni richiesta crea una sessione indipendente e rilascia risorse al termine della richiesta. Questa è una lezione appresa attraverso tentativi ed errori. Se il riutilizzo delle sessioni non viene gestito bene, l’inquinamento dello stato appare facilmente.

2. **Elaborazione dello streaming**: `IAsyncEnumerable` consente di restituire la risposta mentre è ancora in fase di generazione, invece di attendere il completamento di tutto il contenuto. Ciò è particolarmente importante per gli scenari con testo lungo e migliora significativamente l'esperienza dell'utente.

3. **Chiamate degli strumenti**: CodeBuddy supporta le chiamate degli strumenti (chiamate di funzioni), gestite tramite `ToolCallSessionUpdate`. Questa funzionalità è fondamentale per attività complesse di modifica del codice.

4. **Filtro contenuti**: `AIResultContentSanitizer` viene utilizzato per filtrare il contenuto del blocco Think e mantenere pulito l'output.

### 4. Configurazione dell'inserimento delle dipendenze

Aggiungi i servizi correlati durante la registrazione del modulo:

```csharp
// PCodeClaudeHelperModule.cs
public void ConfigureModule(IServiceCollection context)
{
    // Register Provider
    context.Services.AddTransient<CodebuddyCliProvider>();

    // Register ACP infrastructure
    context.Services.AddSingleton<IACPSessionManager, ACPSessionManager>();
    context.Services.AddSingleton<IAcpPlatformConfigurationResolver, AcpPlatformConfigurationResolver>();
    context.Services.AddSingleton<IAIRequestToAcpMapper, AIRequestToAcpMapper>();
    context.Services.AddSingleton<IAcpToAIResponseMapper, AcpToAIResponseMapper>();
}
```

## Esempio di configurazione

### File di configurazione

Aggiungi la configurazione relativa a CodeBuddy a `appsettings.json`:

```yaml
AI:
  # Default Provider to use
  DefaultProvider: "CodebuddyCli"

  # Provider configuration
  Providers:
    CodebuddyCli:
      Type: "CodebuddyCli"
      WorkingDirectory: "C:/projects/my-app"
      ExecutablePath: "C:/tools/codebuddy.cmd"

  # Platform-specific configuration
  PlatformConfigurations:
    CodebuddyCli:
      ExecutablePath: "C:/tools/codebuddy.cmd"
      Arguments: "--acp"
      StartupTimeoutMs: 5000
      EnvironmentVariables:
        CODEBUDDY_API_KEY: "${CODEBUDDY_API_KEY}"
        CODEBUDDY_INTERNET_ENVIRONMENT: "production"
```

### Modello di configurazione

La definizione del modello di configurazione corrispondente:

```csharp
public class CodebuddyPlatformConfiguration : IAcpPlatformConfiguration

    public string Arguments { get; set; } = "--acp";
    public int StartupTimeoutMs { get; set; } = 5000;

    public Dictionary<string, string?>? EnvironmentVariables { get; set; }
}
```

## Lezioni pratiche apprese

### Registro delle insidie

Durante l'implementazione ci siamo imbattuti in diverse trappole tipiche e condividerle qui potrebbe aiutare altri a evitare le stesse deviazioni:

1. **Problema di perdita di sessione**: inizialmente, le sessioni non venivano rilasciate correttamente, il che esauriva le risorse del processo. La soluzione era usare `try-finally` per garantire che le risorse vengano rilasciate per ogni richiesta.

2. **Passaggio delle variabili d'ambiente**: Windows e Linux utilizzano una sintassi diversa per le variabili d'ambiente, quindi in seguito l'abbiamo standardizzata `Dictionary<string, string?>` per gestire questa cosa.

3. **Configurazione timeout**: l'avvio della CLI richiede tempo, quindi impostiamo un timeout di avvio di 5 secondi per evitare errori rapidi delle richieste.

4. **Problemi di codifica**: su Windows, la codifica predefinita può causare testo cinese confuso, quindi la codifica UTF-8 viene specificata esplicitamente all'avvio del processo.

### Ottimizzazione delle prestazioni

1. **Pool di sessioni**: per richieste brevi e frequenti, valuta la possibilità di implementare un pool di sessioni per riutilizzare i processi
2. **Cache di connessione**: la classe factory supporta già la memorizzazione nella cache delle istanze del provider
3. **Prima l'asincrono**: utilizza sempre la programmazione asincrona per evitare di bloccare i thread

Vale sempre la pena ottimizzare le prestazioni. Più a lungo gli utenti aspettano, peggiore diventa l’esperienza.
