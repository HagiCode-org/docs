## Pratica architettura multi-AI provider nella piattaforma HagiCode

> Questo articolo condivide l'approccio tecnico che abbiamo utilizzato nell'architettura Orleans Grain per integrare due strumenti di intelligenza artificiale, iflow e OpenCode, attraverso un sistema unificato `IAIProvider` interfaccia e confronta in dettaglio le differenze di implementazione tra la comunicazione WebSocket e HTTP.

:::nota[Contesto temporale]
Questo articolo registra la pratica dell'architettura del provider al momento della pubblicazione. Per l'ambito attualmente supportato attivamente, fare riferimento a [Installazione della CLI dell'agente AI](/it-IT/related-software-installation/ai-agent-cli/) e il [Guida all'installazione](/it-IT/installation); `IFlowCli` viene ora mantenuto solo come valore di compatibilità storica e non fa parte dell'attuale ambito di supporto attivo.
:::

## Sfondo

Non c'è niente di particolarmente misterioso in questo. Durante la creazione di HagiCode, ci siamo imbattuti in un problema molto pratico: gli utenti volevano lavorare con diversi strumenti di intelligenza artificiale. Ciò non sorprende, poiché ognuno ha le proprie abitudini. Alcuni preferiscono Claude Code, altri amano GitHub Copilot e alcuni team utilizzano strumenti sviluppati da loro stessi.

La nostra soluzione iniziale era semplice e diretta: scrivere un codice di integrazione dedicato per ogni strumento AI. Ma gli inconvenienti si sono manifestati rapidamente. La base di codice si è riempita di `if-else` rami, ogni modifica richiedeva test in più posti e ogni nuovo strumento significava scrivere un'altra pila di logica da zero.

Successivamente ho capito che sarebbe stato meglio creare un sistema unificato `IAIProvider` interfaccia e astraggono le capacità condivise da tutti i fornitori di intelligenza artificiale. In questo modo, indipendentemente dallo strumento utilizzato sotto, gli strati superiori possono richiamarlo allo stesso modo.

Recentemente, il progetto ha dovuto integrare due nuovi strumenti: iflow e OpenCode. Entrambi supportano il protocollo ACP, ma i loro stili di comunicazione sono diversi. iflow utilizza WebSocket, mentre OpenCode utilizza un'API HTTP. Questo è diventato un utile test architetturale: adattare due diverse modalità di trasporto dietro un'unica interfaccia unificata.

## A proposito di HagiCode

L'approccio condiviso in questo articolo deriva dalla nostra esperienza pratica nel [HagiCode](https://hagicode.com) progetto. HagiCode è una piattaforma di sviluppo assistita dall'intelligenza artificiale costruita sull'architettura Orleans Grain. Si integra con diversi fornitori di intelligenza artificiale attraverso un sistema unificato `IAIProvider` interfaccia, consentendo agli utenti di scegliere in modo flessibile gli strumenti di intelligenza artificiale che preferiscono.

## Progettazione dell'architettura

### Astrazione dell'interfaccia unificata

Innanzitutto, abbiamo definito il `IAIProvider` interfaccia e ha estratto le funzionalità di cui ogni fornitore di intelligenza artificiale deve implementare:

```csharp
public interface IAIProvider
{
    string Name { get; }
    bool SupportsStreaming { get; }
    ProviderCapabilities Capabilities { get; }

    Task<AIResponse> ExecuteAsync(AIRequest request, CancellationToken cancellationToken = default);
    IAsyncEnumerable<AIStreamingChunk> StreamAsync(AIRequest request, CancellationToken cancellationToken = default);
    Task<ProviderTestResult> PingAsync(CancellationToken cancellationToken = default);
    IAsyncEnumerable<AIStreamingChunk> SendMessageAsync(AIRequest request, string? embeddedCommandPrompt = null, CancellationToken cancellationToken = default);
}
```

Questa interfaccia include diversi metodi chiave:
- `ExecuteAsync`: esegue una richiesta AI one-shot
- `StreamAsync`: ottieni risposte in streaming per la visualizzazione in tempo reale
- `PingAsync`: esegue un controllo dello stato per verificare se il fornitore è disponibile
- `SendMessageAsync`: invia un messaggio con supporto per comandi incorporati

### IFlowCliProvider: un'implementazione basata su WebSocket

iflow utilizza WebSocket per la comunicazione ACP. L'architettura complessiva è simile alla seguente:

```
IFlowCliProvider → ACPSessionManager → WebSocketAcpTransport → iflow CLI
                ↓
         Dynamic port allocation + process management
```

Anche il flusso principale è abbastanza semplice:
1. `ACPSessionManager` crea e gestisce sessioni ACP.
2. `WebSocketAcpTransport` gestisce la comunicazione WebSocket.
3. Una porta viene allocata dinamicamente e viene avviato il processo iflow `iflow --experimental-acp --port`.
4. `IAIRequestToAcpMapper` e `IAcpToAIResponseMapper` convertire richieste e risposte.

Ecco il codice principale:

```csharp
private async IAsyncEnumerable<AIStreamingChunk> StreamCoreAsync(
    AIRequest request,
    string? embeddedCommandPrompt,
    [EnumeratorCancellation] CancellationToken cancellationToken)
{
    // Resolve working directory
    var resolvedWorkingDirectory = ResolveWorkingDirectory(request);
    var effectiveRequest = ApplyEmbeddedCommandPrompt(request, embeddedCommandPrompt);

    // Create ACP session
    await using var session = await _sessionManager.CreateSessionAsync(
        Name,
        resolvedWorkingDirectory,
        cancellationToken,
        request.SessionId);

    // Send prompt
    var prompt = _requestMapper.ToPromptString(effectiveRequest);
    var promptResponse = await session.SendPromptAsync(prompt, cancellationToken);

    // Receive streaming response
    await foreach (var notification in session.ReceiveUpdatesAsync(cancellationToken))
    {
        if (_responseMapper.TryConvertToStreamingChunk(notification, out var chunk))
        {
            if (chunk.Type == StreamingChunkType.Metadata && chunk.IsComplete)
            {
                yield return chunk;
                yield break;
            }
            yield return chunk;
        }
    }
}
```

Ci sono alcuni punti di progettazione che vale la pena richiamare qui:
- Utilizzare `await using` per garantire che la sessione venga rilasciata correttamente ed evitare perdite di risorse.
- Restituisci le risposte in streaming `IAsyncEnumerable`, che supporta naturalmente i flussi asincroni.
- Utilizzare `Metadata` blocchi per determinare il completamento e garantire che la risposta completa sia stata ricevuta.

### OpenCodeCliProvider: un'implementazione basata su API HTTP

OpenCode fornisce il suo servizio tramite un'API HTTP, quindi l'architettura è leggermente diversa:

```plaintext
OpenCodeCliProvider → OpenCodeRuntimeManager → OpenCodeClient → OpenCode HTTP API
                      ↓
                OpenCodeProcessManager → opencode process management
```

Una caratteristica notevole di OpenCode è che utilizza un database SQLite per rendere persistenti i collegamenti di sessione. Ciò rende possibile il ripristino della sessione e il ripristino con risposta rapida:

```csharp
private async Task<OpenCodePromptExecutionResult> ExecutePromptAsync(
    AIRequest request,
    string? embeddedCommandPrompt,
    CancellationToken cancellationToken)
{
    var prompt = BuildPrompt(request, embeddedCommandPrompt);
    var resolvedWorkingDirectory = ResolveWorkingDirectory(request.WorkingDirectory);
    var client = await _runtimeManager.GetClientAsync(resolvedWorkingDirectory, cancellationToken);
    var bindingSessionId = request.SessionId;
    var boundSession = TryGetBinding(bindingSessionId, resolvedWorkingDirectory);

    // Try to use the already bound session
    if (boundSession is not null)
    {
        try
        {
            return await PromptSessionAsync(
                client,
                boundSession,
                BuildPromptRequest(request, prompt, CreatePromptMessageId()),
                request.Model ?? _settings.Model,
                cancellationToken);
        }
        catch (OpenCodeApiException ex) when (IsStaleBinding(ex))
        {
            // The session has expired, remove the binding
            RemoveBinding(bindingSessionId);
        }
    }

    // Create a new session
    var session = await client.Session.CreateAsync(new OpenCodeSessionCreateRequest
    {
        Title = BuildSessionTitle(request)
    }, cancellationToken);

    BindSession(bindingSessionId, session.Id, resolvedWorkingDirectory);
    return await PromptSessionAsync(client, session.Id, ...);
}
```

Questa implementazione presenta diversi punti salienti interessanti:
- Meccanismo di associazione della sessione: lo stesso `SessionId` riutilizza la stessa sessione OpenCode, evitando la creazione di sessioni ripetute.
- Gestione della scadenza: quando una sessione risulta scaduta, l'associazione viene automaticamente ripulita.
- Persistenza del database: i collegamenti vengono archiviati in SQLite e rimangono efficaci dopo il riavvio.

### Confronto tra i due approcci

| Aspetto | IFlowCliProvider | OpenCodeCliProvider |
|------|-----------------|---------------------|
| Comunicazione | WebSocket (ACP) | API HTTP |
| Gestione dei processi | ACPSessionManager | OpenCodeProcessManager |
| Assegnazione dei porti | Porto dinamico | Nessuna porta (utilizza HTTP) |
| Gestione della sessione | Sessione ACP | OpenCodeSession |
| Persistenza | Cache in memoria | Base di dati SQLite |
| Comando di avvio | `iflow --experimental-acp --port` | `opencode` |
| Latenza | Inferiore (connessione di lunga durata) | Relativamente più alto (richieste HTTP) |

L'approccio che sceglierai dipende principalmente dalle tue esigenze. WebSocket è migliore per scenari con elevati requisiti in tempo reale, mentre un'API HTTP è più semplice e facile da eseguire il debug.

## Guida pratica

### Configura provider

Per prima cosa abilita i due provider nel file di configurazione:

```yaml
AI:
  Providers:
    IFlowCli:
      Type: "IFlowCli"
      Enabled: true
      ExecutablePath: "iflow"
      Model: null
      WorkingDirectory: null
    OpenCodeCli:
      Type: "OpenCodeCli"
      Enabled: true
      ExecutablePath: "opencode"
      Model: "anthropic/claude-sonnet-4"
      WorkingDirectory: null

OpenCode:
  Enabled: true
  BaseUrl: "http://localhost:38376"
  ExecutablePath: "opencode"
  StartupTimeoutSeconds: 30
  RequestTimeoutSeconds: 120
```

### Utilizzare IFlowCliProvider

```csharp
// Get provider through the factory
var provider = await _providerFactory.GetProviderAsync(AIProviderType.IFlowCli);

// Execute an AI request
var request = new AIRequest
{
    Prompt = "请帮我重构这个函数",
    WorkingDirectory = "/path/to/project",
    Model = "claude-sonnet-4"
};

// Get the complete response
var response = await provider.ExecuteAsync(request, cancellationToken);
Console.WriteLine(response.Content);

// Or use streaming responses
await foreach (var chunk in provider.StreamAsync(request, cancellationToken))
{
    if (chunk.Type == StreamingChunkType.ContentDelta)
    {
        Console.Write(chunk.Content);
    }
}
```

### Utilizzare OpenCodeCliProvider

```csharp
// Get provider through the factory
var provider = await _providerFactory.GetProviderAsync(AIProviderType.OpenCodeCli);

var request = new AIRequest
{
    Prompt = "请帮我分析这个错误",
    WorkingDirectory = "/path/to/project",
    Model = "anthropic/claude-sonnet-4"
};

var response = await provider.ExecuteAsync(request, cancellationToken);
Console.WriteLine(response.Content);
```

### Controlli sanitari

Prima dell'avvio o prima dell'uso è possibile verificare se il provider è disponibile:

```csharp
var iflowResult = await iflowProvider.PingAsync(cancellationToken);
if (!iflowResult.Success)
{
    Console.WriteLine($"IFlow is unavailable: {iflowResult.ErrorMessage}");
    return;
}

var openCodeResult = await openCodeProvider.PingAsync(cancellationToken);
if (!openCodeResult.Success)
{
    Console.WriteLine($"OpenCode is unavailable: {openCodeResult.ErrorMessage}");
    return;
}
```

### Supporto comandi integrato

Entrambi i provider supportano comandi incorporati, come `/file:xxx`:

```csharp
var request = new AIRequest
{
    Prompt = "分析这个文件的问题",
    SystemMessage = "你是一个代码分析专家"
};

await foreach (var chunk in provider.SendMessageAsync(
    request,
    embeddedCommandPrompt: "/file:src/main.cs",
    cancellationToken))
{
    Console.Write(chunk.Content);
}
```

## Note e migliori pratiche

### Gestione delle risorse

IFlow utilizza connessioni WebSocket di lunga durata, quindi la gestione delle risorse merita un'attenzione speciale:
- Utilizzare `await using` per garantire che le sessioni vengano rilasciate correttamente.
- L'annullamento attiva la pulizia del processo.
- `ACPSessionManager` supporta un limite massimo di numero di sessioni.

La gestione del processo OpenCode è relativamente più semplice e `OpenCodeRuntimeManager` lo gestisce automaticamente.

### Gestione degli errori

Entrambi i provider hanno una gestione completa degli errori:
- Gli errori IFlow vengono propagati tramite gli aggiornamenti della sessione ACP.
- Vengono generati errori OpenCode `OpenCodeApiException`.
- Si consiglia al chiamante di rilevare e gestire queste eccezioni.

### Considerazioni sulle prestazioni

- La comunicazione IFlow WebSocket ha una latenza inferiore rispetto a HTTP.
- Il riutilizzo della sessione OpenCode può ridurre il sovraccarico delle richieste HTTP.
- Il meccanismo della cache di fabbrica evita la creazione ripetuta di provider.
- Negli scenari ad alta concorrenza, prestare particolare attenzione ai limiti sul conteggio dei processi e sul conteggio delle connessioni.

### Convalida della configurazione

Il percorso dell'eseguibile viene convalidato all'avvio, ma possono comunque verificarsi problemi di runtime. `PingAsync` è uno strumento utile per verificare se la configurazione è corretta:

```csharp
// Check at startup
var provider = await _providerFactory.GetProviderAsync(providerType);
var result = await provider.PingAsync(cancellationToken);
if (!result.Success)
{
    _logger.LogError("Provider {ProviderType} is unavailable: {Error}", providerType, result.ErrorMessage);
}
```

## Riepilogo

Questo articolo condivide l'approccio tecnico utilizzato dalla piattaforma HagiCode quando integra i due strumenti AI iflow e OpenCode. Attraverso un unificato `IAIProvider` interfaccia, abbiamo adattato diversi stili di comunicazione, WebSocket e HTTP, mantenendo coerente il modello di chiamata del livello superiore.

L’idea di base è in realtà abbastanza semplice:
1. Definire un'astrazione dell'interfaccia unificata.
2. Crea livelli di adattatori per diverse implementazioni.
3. Gestisci tutto in modo uniforme attraverso il modello di fabbrica.

Ciò conferisce al sistema una buona estensibilità. Quando in un secondo momento è necessario integrare un nuovo strumento di intelligenza artificiale, tutto ciò che dobbiamo fare è implementare il `IAIProvider` interfaccia senza modificare troppo il codice esistente.

Se anche tu stai lavorando all'integrazione di più strumenti AI, spero che questo articolo ti sia utile.
