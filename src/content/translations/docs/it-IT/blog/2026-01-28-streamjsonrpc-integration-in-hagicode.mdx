## Profonda integrazione e uso pratico di StreamJsonRpc in HagiCode

> Questo articolo spiega in dettaglio come il progetto HagiCode (in precedenza PCode) ha integrato con successo la libreria di comunicazione StreamJsonRpc di Microsoft per sostituire la sua implementazione JSON-RPC personalizzata originale, risolvendo al tempo stesso i punti critici tecnici e le sfide architettoniche incontrate durante il processo di integrazione.

## Sfondo

StreamJsonRpc è la libreria di comunicazione JSON-RPC gestita ufficialmente da Microsoft per .NET e TypeScript, nota per la sua forte sicurezza dei tipi, la generazione automatica di proxy e i meccanismi maturi di gestione delle eccezioni. Nel progetto HagiCode, il team ha deciso di integrare StreamJsonRpc per comunicare con strumenti AI esterni come iflow CLI e OpenCode CLI tramite ACP (Agent Communication Protocol), eliminando al tempo stesso i costi di manutenzione e i potenziali bug introdotti dalla precedente implementazione personalizzata JSON-RPC. Tuttavia, il processo di integrazione ha dovuto affrontare sfide specifiche per lo streaming JSON-RPC, in particolare durante la gestione dell'associazione del target proxy e del riconoscimento dei parametri generici.

Per affrontare questi punti critici, abbiamo preso una decisione coraggiosa: ricostruire l'intero sistema di costruzione da zero. L’impatto di questa decisione potrebbe essere ancora più grande di quanto ti aspetti e lo spiegherò in dettaglio a breve.

## Analisi

L'attuale progetto si trova in una fase critica dell'integrazione del protocollo ACP e sta affrontando i seguenti punti critici tecnici e sfide architetturali:

### 1. Limitazioni dell'implementazione personalizzata
L'implementazione JSON-RPC originale si trova in `src/HagiCode.ClaudeHelper/AcpImp/` e include componenti come `JsonRpcEndpoint` e `ClientSideConnection`. Mantenere questa codebase personalizzata è costoso e manca delle funzionalità avanzate di una libreria matura, come la segnalazione dei progressi e il supporto per l'annullamento.

### 2. Ostacoli all'integrazione di StreamJsonRpc
Quando si tenta di migrare il file esistente `CallbackProxyTarget` pattern su StreamJsonRpc, abbiamo scoperto che il `_rpc.AddLocalRpcTarget(target)` Il metodo non è in grado di riconoscere le destinazioni create tramite il modello proxy. In particolare, StreamJsonRpc non è in grado di dividere automaticamente le proprietà di tipo generico `T` nei parametri del metodo RPC, causando un errore sul lato server durante l'elaborazione delle chiamate al metodo avviate dal client.

### 3. Stratificazione architettonica confusa
L'esistente `ClientSideConnection` mescola il livello di trasporto (WebSocket/Stdio), il livello di protocollo (JSON-RPC) e il livello aziendale (interfaccia dell'agente ACP), portando a responsabilità poco chiare. Soffre anche della mancanza di collegamenti al metodo `AcpAgentCallbackRpcAdapter`.

### 4. Registri mancanti
Il livello di trasporto WebSocket è privo di registrazione del contenuto JSON non elaborato, rendendo difficile determinare durante il debug RPC se un problema ha origine dalla serializzazione o dalla rete.

## Soluzione

Per risolvere i problemi di cui sopra, abbiamo adottato la seguente soluzione sistematica, ottimizzando da tre dimensioni: refactoring dell'architettura, integrazione della libreria e debug avanzato.

### 1. Migrazione completa a StreamJsonRpc

#### Rimuovi il codice legacy
Elimina `JsonRpcEndpoint.cs`, `AgentSideConnection.cs`e i relativi convertitori di serializzazione personalizzata come `JsonRpcMessageJsonConverter`.

#### Integrare la Biblioteca ufficiale
Presentare il `StreamJsonRpc` Pacchetto NuGet e usalo `JsonRpc` classe per gestire la logica di comunicazione principale.

#### Astrarre il livello di trasporto
Definire il `IAcpTransport` interfaccia per gestire entrambi `WebSocket` e `Stdio` modalità di trasporto in modo unificato, garantendo che il livello del protocollo sia disaccoppiato dal livello di trasporto.

```csharp
// Definition of the IAcpTransport interface
public interface IAcpTransport
{
    Task SendAsync(string message, CancellationToken cancellationToken = default);
    Task<string> ReceiveAsync(CancellationToken cancellationToken = default);
    Task CloseAsync(CancellationToken cancellationToken = default);
}

// WebSocket transport implementation
public class WebSocketTransport : IAcpTransport
{
    private readonly WebSocket _webSocket;

    public WebSocketTransport(WebSocket webSocket)
    {
        _webSocket = webSocket;
    }

    // Implement send and receive methods
    // ...
}

// Stdio transport implementation
public class StdioTransport : IAcpTransport
{
    private readonly StreamReader _reader;
    private readonly StreamWriter _writer;

    public StdioTransport(StreamReader reader, StreamWriter writer)
    {
        _reader = reader;
        _writer = writer;
    }

    // Implement send and receive methods
    // ...
}
```

### 2. Risolvere i problemi di riconoscimento del target proxy

#### Analizzare `CallbackProxyTarget`
Esaminare la logica di generazione del proxy dinamico esistente e identificare la causa principale del motivo per cui StreamJsonRpc non è in grado di riconoscerlo. Nella maggior parte dei casi, ciò accade perché l'oggetto proxy non espone pubblicamente le firme effettive del metodo oppure utilizza tipi di parametri non supportati da StreamJsonRpc.

#### Passaggio dei parametri di refactoring
Dividere le proprietà generiche in parametri espliciti del metodo RPC. Invece di fare affidamento su proprietà dinamiche, definire DTO di richiesta/risposta concreti (oggetti di trasferimento dati) in modo che StreamJsonRpc possa riconoscere correttamente le firme dei metodi attraverso la riflessione.

```csharp
// Original generic property approach
public class CallbackProxyTarget<T>
{
    public Func<T, Task> Callback { get; set; }
}

// Refactored concrete method approach
public class ReadTextFileRequest
{
    public string FilePath { get; set; }
}

public class ReadTextFileResponse
{
    public string Content { get; set; }
}

public interface IAcpAgentCallback
{
    Task<ReadTextFileResponse> ReadTextFileAsync(ReadTextFileRequest request);
    // Other methods...
}
```

#### Utilizzare `Attach` Invece di `AddLocalRpcTarget`
In alcuni scenari complessi, l'inoltro manuale del file `JsonRpc` oggetto e manipolazione `RpcConnection` può essere più flessibile rispetto all'aggiunta diretta di un target locale.

### 3. Implementare l'associazione dei metodi e una registrazione più efficace

#### Implementare `AcpAgentCallbackRpcAdapter`
Assicurarsi che questo componente implementi esplicitamente l'interfaccia proxy StreamJsonRpc e mappi i metodi definiti dal protocollo ACP, come ad esempio `ReadTextFileAsync`, ai gestori di callback StreamJsonRpc.

#### Integra la registrazione
Intercetta e registra il testo non elaborato delle richieste e delle risposte JSON-RPC nella pipeline di elaborazione dei messaggi WebSocket o Stdio. Utilizzare `ILogger` per produrre il payload non elaborato prima dell'analisi e dopo la serializzazione in modo che i problemi di formattazione possano essere diagnosticati più facilmente.

```csharp
// Transport wrapper with enhanced logging
public class LoggingAcpTransport : IAcpTransport
{
    private readonly IAcpTransport _innerTransport;
    private readonly ILogger<LoggingAcpTransport> _logger;

    public LoggingAcpTransport(IAcpTransport innerTransport, ILogger<LoggingAcpTransport> logger)
    {
        _innerTransport = innerTransport;
        _logger = logger;
    }

    public async Task SendAsync(string message, CancellationToken cancellationToken = default)
    {
        _logger.LogTrace("Sending message: {Message}", message);
        await _innerTransport.SendAsync(message, cancellationToken);
    }

    public async Task<string> ReceiveAsync(CancellationToken cancellationToken = default)
    {
        var message = await _innerTransport.ReceiveAsync(cancellationToken);
        _logger.LogTrace("Received message: {Message}", message);
        return message;
    }

    public async Task CloseAsync(CancellationToken cancellationToken = default)
    {
        _logger.LogDebug("Closing connection");
        await _innerTransport.CloseAsync(cancellationToken);
    }
}
```

### 4. Stratificazione architettonica del refactoring

#### Livello di trasporto (`AcpRpcClient`)
Avvolgi la connessione StreamJsonRpc e rendila responsabile `InvokeAsync` gestione del ciclo di vita delle chiamate e delle connessioni.

```csharp
public class AcpRpcClient : IDisposable
{
    private readonly JsonRpc _rpc;
    private readonly IAcpTransport _transport;

    public AcpRpcClient(IAcpTransport transport)
    {
        _transport = transport;
        _rpc = new JsonRpc(new StreamRpcTransport(transport));
        _rpc.StartListening();
    }

    public async Task<TResponse> InvokeAsync<TResponse>(string methodName, object parameters)
    {
        return await _rpc.InvokeAsync<TResponse>(methodName, parameters);
    }

    public void Dispose()
    {
        _rpc.Dispose();
        _transport.Dispose();
    }

    // StreamRpcTransport is the StreamJsonRpc adapter for IAcpTransport
    private class StreamRpcTransport : IDuplexPipe
    {
        // Implement the IDuplexPipe interface
        // ...
    }
}
```

#### Livello protocollo (`IAcpAgentClient` / `IAcpAgentCallback`)
Definire chiare interfacce client-agente e agente-cliente. Rimuovere il modello ciclico di fabbrica `Func<IAcpAgent, IAcpClient>` e sostituirlo con l'inserimento delle dipendenze o la registrazione diretta del callback.

## Pratica

Sulla base delle migliori pratiche e dell'esperienza del progetto StreamJsonRpc, di seguito sono riportate le raccomandazioni chiave per l'implementazione:

### 1. I DTO fortemente tipizzati sono migliori degli oggetti dinamici
Il vantaggio principale di StreamJsonRpc risiede nella tipizzazione forte. Non utilizzare `dynamic` o `JObject` per passare i parametri. Definire invece classi POCO C# esplicite come parametri per ogni metodo RPC. Ciò non solo risolve il problema del riconoscimento della destinazione proxy, ma rileva anche gli errori di tipo in fase di compilazione.

Esempio: sostituire le proprietà generiche in `CallbackProxyTarget` con classi concrete come `ReadTextFileRequest` e `WriteTextFileRequest`.

### 2. Dichiarare esplicitamente i nomi dei metodi
Usa il `[JsonRpcMethod]` attributo per specificare in modo esplicito i nomi dei metodi RPC invece di fare affidamento sulla mappatura predefinita dei nomi dei metodi. Ciò impedisce errori di invocazione causati da differenze di stile di denominazione come PascalCase rispetto a camelCase.

```csharp
public interface IAcpAgentCallback
{
    [JsonRpcMethod("readTextFile")]
    Task<ReadTextFileResponse> ReadTextFileAsync(ReadTextFileRequest request);
    
    [JsonRpcMethod("writeTextFile")]
    Task WriteTextFileAsync(WriteTextFileRequest request);
}
```

### 3. Sfrutta i vantaggi delle richiamate dello stato della connessione
StreamJsonRpc fornisce il file `JsonRpc.ConnectionLost` evento. Dovresti assolutamente prestare attenzione a questo evento per gestire le uscite impreviste del processo o le disconnessioni di rete, il che è più tempestivo che fare affidamento solo sul rilevamento degli errori di Orleans Grain.

```csharp
_rpc.ConnectionLost += (sender, e) =>
{
    _logger.LogError("RPC connection lost: {Reason}", e.ToString());
    // Handle reconnection logic or notify the user
};
```

### 4. Registra i registri per livello
- **Livello di traccia**: registra l'intero payload di richiesta/risposta JSON non elaborato.
- **Livello di debug**: registra stack di chiamate di metodi e riepiloghi dei parametri.
- **Nota**: assicurati che i log non includano token di autorizzazione sensibili o contenuti di file di grandi dimensioni con codifica Base64.

### 5. Gestire la natura speciale del trasporto in streaming
StreamJsonRpc supporta nativamente `IAsyncEnumerable`. Quando si implementano le risposte ai prompt di streaming per ACP, utilizzare `IAsyncEnumerable` direttamente invece della logica di impaginazione personalizzata. Ciò può semplificare notevolmente la quantità di codice necessaria per l'elaborazione dello streaming.

```csharp
public interface IAcpAgentCallback
{
    [JsonRpcMethod("streamText")]
    IAsyncEnumerable<string> StreamTextAsync(StreamTextRequest request);
}
```

### 6. Modello adattatore
Tieni `ACPSession` e `ClientSideConnection` separato. `ACPSession` dovrebbe concentrarsi sulla gestione dello stato di Orleans e sulla logica aziendale, come l'accodamento dei messaggi, e dovrebbe utilizzare l'oggetto di connessione StreamJsonRpc attraverso la composizione anziché l'ereditarietà.

## Conclusione

Integrando in modo completo StreamJsonRpc, il progetto HagiCode ha affrontato con successo gli elevati costi di manutenzione, le limitazioni funzionali e la confusione della stratificazione architettonica dell'implementazione personalizzata originale. I miglioramenti principali includono:

1. Sostituzione delle proprietà dinamiche con DTO fortemente tipizzati, miglioramento della manutenibilità e dell'affidabilità
2. Implementazione dell'astrazione del livello di trasporto e della separazione del livello di protocollo, migliorando la chiarezza dell'architettura
3. Rafforzamento delle capacità di registrazione per facilitare la diagnosi dei problemi di comunicazione
4. Presentazione del supporto dello streaming per semplificare l'implementazione dello streaming

Questi miglioramenti forniscono a HagiCode una base di comunicazione più stabile ed efficiente, consentendogli di interagire meglio con strumenti di intelligenza artificiale esterni e gettando solide basi per la futura espansione delle funzionalità.
