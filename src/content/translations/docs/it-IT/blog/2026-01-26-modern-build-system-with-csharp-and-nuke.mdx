## Addio inferno degli script: perché abbiamo scelto C# per un sistema di build moderno

> Un approfondimento sul modo in cui il progetto HagiCode utilizza Nuke per creare una pipeline di build automatizzata indipendente dai tipi, multipiattaforma e altamente estensibile, affrontando completamente i punti critici di manutenzione degli script di build tradizionali.

## Sfondo

Durante il lungo viaggio dello sviluppo del software, la parola "costruire" tende a ispirare sia amore che frustrazione. Amore, perché con un clic il codice si trasforma in un prodotto, ed è uno dei momenti più soddisfacenti della programmazione. Frustrazione, perché mantenere una pila di script di compilazione disordinati può essere un incubo.

In molti progetti siamo abituati a scrivere script in Python o a fare affidamento su file di configurazione XML. Immagina solo la paura di essere governato da `<property>` blocchi. Ma man mano che la complessità del progetto aumenta, soprattutto in un progetto come HagiCode che abbraccia il lavoro di frontend e backend, più piattaforme e più linguaggi, gli approcci tradizionali alla creazione iniziano a mostrare i loro limiti. Logica di script sparpagliata, nessun controllo del tipo, supporto IDE debole... ognuno di questi problemi diventa una piccola trappola che di tanto in tanto fa inciampare il team.

Per risolvere questi punti critici, il progetto HagiCode ha introdotto **Nuke**, un moderno sistema di build basato su C#. Non è solo uno strumento. È un modo diverso di pensare ai flussi di lavoro di creazione. Oggi vedremo perché lo abbiamo scelto e come ha migliorato notevolmente la nostra esperienza di sviluppo.

## Analisi principale: perché il nucleare?

Forse ti starai chiedendo: "Esistono già così tanti sistemi di compilazione, come Make, Gradle o anche semplici script di Shell. Perché fare di tutto per usarne uno basato su C#?"

Questa è una domanda giusta. L'attrattiva principale di Nuke è che porta le funzionalità del linguaggio di programmazione che già conosciamo bene nel mondo degli script di build.

### 1. Modularizzare i flussi di lavoro di costruzione: l'arte dei bersagli

Nuke ha una filosofia di design molto chiara: **tutto è un obiettivo**.

Negli script tradizionali, è facile ritrovarsi con centinaia di linee di logica di esecuzione lineare aggrovigliate insieme. In Nuke, suddividiamo il processo di costruzione in indipendenti `Target`s. Ogni target fa esattamente una cosa, ad esempio:

- `Clean`: pulisce la directory di output
- `Restore`: ripristina i pacchetti di dipendenze
- `Compile`: compila il codice
- `Test`: esegue test unitari

Questo disegno si allinea molto bene con il principio di responsabilità unica. Come i mattoni, questi obiettivi possono essere combinati liberamente. Ancora più importante, Nuke ci consente di definire le dipendenze tra Target. Ad esempio, se vuoi correre `Test`, il sistema controllerà automaticamente se `Compile` è già stato eseguito. Se vuoi `Compile`, quindi `Restore` naturalmente viene prima.

Questo grafico delle dipendenze rende la logica molto più chiara e migliora significativamente l'efficienza di esecuzione, perché Nuke può analizzare automaticamente il percorso di esecuzione ottimale.

### 2. Sicurezza dei tipi: addio all'incubo degli errori di battitura

Chiunque abbia scritto script di build in Python probabilmente si è imbattuto in questa situazione imbarazzante: lo script viene eseguito per cinque minuti e poi fallisce perché `Confi.guration` è stato scritto in modo errato o perché è stata passata una stringa dove era previsto un parametro numerico.

Il più grande vantaggio di scrivere script di build in C# è l'**indipendenza dai tipi**. Ciò significa:

- **Controlli in fase di compilazione**: l'IDE può dirti cosa c'è che non va mentre stai digitando invece di farti aspettare fino al runtime.
- **Refactoring sicuro**: se vuoi rinominare una variabile o un metodo, l'IDE può gestirlo in un solo passaggio senza costringerti a fare una rischiosa ricerca e sostituzione globale.
- **Completamento intelligente**: il potente IntelliSense può completare il codice automaticamente, quindi non è necessario continuare a sfogliare la documentazione per ricordare API oscure.

### 3. Supporto multipiattaforma: un'esperienza di creazione coerente

In passato, abbiamo scritto `.bat` su Windows e `.sh` su Linux, e poi magari aggiunto uno script Python per colmare il divario. Ora, ovunque sia possibile eseguire .NET Core o .NET 5+ oggi, è possibile eseguire anche Nuke.

Ciò significa che i membri del team possono utilizzare Windows, Linux o macOS e, indipendentemente dal fatto che preferiscano Visual Studio, VS Code o Rider, eseguono tutti esattamente la stessa logica. Ciò riduce notevolmente i problemi causati dalle differenze ambientali, in particolare il classico problema "funziona sulla mia macchina".

### 4. Gestione dei parametri e della configurazione

Nuke fornisce un meccanismo di analisi dei parametri molto elegante. Non è necessario analizzare manualmente `string[] args`. Devi semplicemente definire una proprietà e annotarla con `[Parameter]`e Nuke gestisce automaticamente la mappatura dagli argomenti della riga di comando e dai file di configurazione.

Ad esempio, possiamo definire la configurazione della build molto facilmente:

```csharp
[Parameter("Configuration to build - Default is 'Debug'")]
readonly Configuration BuildConfiguration = IsLocalBuild ? Configuration.Debug : Configuration.Release;

Target Compile => _ => _
    .DependsOn(Restore)
    .Executes(() =>
    {
        // Use BuildConfiguration here; it is type-safe
        DotNetBuild(s => s
            .SetConfiguration(BuildConfiguration)
            .SetProjectFile(SolutionFile));
    });
```

Questo stile è intuitivo e resistente agli errori.

## Guida pratica: come metterla in pratica in un progetto

La sola teoria non porta da nessuna parte. Vediamo come abbiamo effettivamente applicato questo approccio nel progetto HagiCode.

### 1. Pianificare la struttura del progetto

Non volevamo creare script che ingombrassero la root del progetto, e non volevamo nemmeno una struttura di directory così profonda da assomigliare ad alcuni progetti Java. Quindi abbiamo inserito tutti i file di build relativi a Nuke in un unico file `nukeBuild/` directory.

I vantaggi sono semplici:

- La radice del progetto rimane pulita.
- La logica di creazione rimane coesa e più facile da gestire.
- I nuovi membri del team possono immediatamente vedere: "Qui è dove risiede la logica relativa alla creazione".

### 2. Progettare una catena di dipendenze di destinazione chiara

Durante la progettazione dei target, abbiamo seguito un principio: **atomicità più flusso di dipendenza**.

Ogni Target dovrebbe essere abbastanza piccolo da fare esattamente una cosa. Ad esempio, `Clean` dovrebbe rimuovere solo i file e non dovrebbe maneggiare silenziosamente l'imballaggio sul lato.

Un flusso di dipendenza consigliato è più o meno simile al seguente:

`Clean` -> `Restore` -> `Compile` -> `Test` -> `Pack`

Naturalmente questo non è assoluto. Ad esempio, se vuoi solo eseguire test e non vuoi impacchettare nulla, Nuke ti consente di eseguire `nuke Test` direttamente e gestirà automaticamente quanto richiesto `Restore` e `Compile` passi prima.

### 3. Robusta gestione e registrazione degli errori

Qual è la cosa più frustrante degli script di compilazione? Messaggi di errore poco chiari. Una build fallita che registra solo "Errore: 1" è sufficiente per far impazzire chiunque.

In Nuke, poiché possiamo utilizzare direttamente la gestione delle eccezioni C#, possiamo acquisire e segnalare errori con una precisione molto maggiore.

```csharp
Target Publish => _ => _
    .DependsOn(Test)
    .Executes(() =>
    {
        try 
        {
            // Try publishing to NuGet
            DotNetNuGetPush(s => s
                .SetTargetPath(ArtifactPath)
                .SetSource("https://api.nuget.org/v3/index.json")
                .SetApiKey(ApiKey));
        }
        catch (Exception ex)
        {
            Log.Error($"Publishing failed. Team, please check whether the key is correct: {ex.Message}");
            throw; // Make sure the build process exits with a non-zero exit code
        }
    });
```

### 4. Utilizzare i test per proteggere la qualità

Anche gli script di compilazione sono codice, il che significa che necessitano anche di test. Nuke ci consente di scrivere test per i flussi di lavoro di build in modo che quando cambiamo la logica di build, non interrompiamo accidentalmente i processi di rilascio esistenti. Ciò è particolarmente importante nelle pipeline di integrazione continua.

## Riepilogo

Con l'introduzione di Nuke, il processo di creazione di HagiCode è diventato più fluido che mai. Non si trattava solo di una sostituzione dello strumento. È stato un aggiornamento nella mentalità ingegneristica.

**Cosa abbiamo guadagnato?**

- **Manutenibilità**: configurazione come codice, logica più chiara e percorso di onboarding più rapido per i nuovi contributori.
- **Stabilità**: la digitazione forte elimina oltre il 90% degli errori di basso livello.
- **Coerenza**: un'esperienza multipiattaforma unificata che elimina la deviazione dall'ambiente.

Se scrivere script di build era come "camminare a tentoni nel buio", allora usare Nuke è come "camminare di notte con le luci accese". Se sei stanco di gestire script difficili da eseguire il debug, valuta la possibilità di spostare la logica di compilazione nel mondo di C#. Potresti scoprire che anche le build possono essere eleganti.
