## Design.md: una soluzione per una progettazione coerente dell'interfaccia utente frontend basata sull'intelligenza artificiale

> Nell'era dello sviluppo frontend assistito dall'intelligenza artificiale, come puoi mantenere coerenti le UI generate dall'intelligenza artificiale? Questo articolo condivide la nostra esperienza pratica nella creazione di un sito di gallerie di design basato su awesome-design-md, insieme a come creare un sito strutturato `design.md` per guidare l’intelligenza artificiale verso una progettazione dell’interfaccia utente standardizzata.

## Sfondo

Chiunque abbia utilizzato l’intelligenza artificiale per scrivere codice frontend ha probabilmente avuto un’esperienza simile: chiedi all’intelligenza artificiale di generare la stessa pagina più volte e ogni risultato verrà fuori con uno stile diverso. A volte gli angoli sono arrotondati, a volte sono acuti. A volte la spaziatura è 8px e altre volte diventa 16px. Anche lo stesso pulsante può avere un aspetto diverso nelle diverse conversazioni.

Questo non è un problema isolato. Man mano che lo sviluppo assistito dall'intelligenza artificiale diventa più comune, la **mancanza di coerenza nell'interfaccia utente frontend generata dall'intelligenza artificiale** è diventata un problema diffuso. Assistenti IA diversi, istruzioni diverse e persino lo stesso assistente in conversazioni diverse possono produrre progetti di interfaccia notevolmente diversi. Ciò crea un enorme costo di manutenzione durante l’iterazione del prodotto.

La causa principale è in realtà semplice: non esiste un **documento autorevole di riferimento per la progettazione**. I fogli di stile CSS tradizionali possono dire agli sviluppatori "come implementare" qualcosa, ma non possono comunicare completamente "perché è progettato in questo modo" o "quale modello di progettazione dovrebbe essere utilizzato in quale scenario". Per l’intelligenza artificiale, una descrizione chiara e strutturata è ancora più importante per comprendere le convenzioni di progettazione.

Allo stesso tempo, la comunità open source offre già alcune risorse eccellenti. Il [VoltAgent/awesome-design-md](https://github.com/VoltAgent/awesome-design-md) project raccoglie la documentazione del sistema di progettazione di molte aziende rinomate. Ogni directory contiene a `README.md`, un `DESIGN.md`e visualizzare l'anteprima HTML. Tuttavia, tutto ciò è sparso nel repository upstream, rendendo difficile la navigazione e il confronto rapido.

Quindi, possiamo consolidare queste risorse in una galleria di design facile da sfogliare, distillando allo stesso tempo un approccio strutturato `design.md` per l'intelligenza artificiale da utilizzare?

La risposta è sì. Successivamente, lasciatemi illustrare il nostro approccio.

## Analisi

Prima di scrivere il codice, analizziamo le sfide tecniche alla base di questo problema.

### Gestione del contenuto sorgente: come unificare le risorse di progettazione sparse?

Il repository awesome-design-md originale contiene un gran numero di documenti di progettazione, ma avevamo bisogno di un modo per inserirli nel nostro progetto.

**Soluzione: utilizzare il sottomodulo git**

```
awesome-design-md-site
└── vendor/awesome-design-md          # Upstream resources (git submodule)
```

Questo ci offre diversi vantaggi:
- Controllo della versione: possiamo bloccare una versione upstream specifica
- Build offline: non è necessario richiedere API esterne durante la build
- Revisione dei contenuti: modifiche specifiche sono visibili nei PR

### Normalizzazione dei dati: come standardizzare le diverse strutture dei documenti?

Diverse aziende strutturano i propri documenti di progettazione in modo diverso. Ad alcuni mancano i file di anteprima e alcuni utilizzano nomi incoerenti. Dobbiamo normalizzarli durante il processo di creazione.

**Soluzione: scansiona e genera voci normalizzate in fase di creazione**

Il modulo principale è `awesomeDesignCatalog.ts`, responsabile di:
1. Scansione del `vendor/awesome-design-md/design-md/*` directory
2. Verifica se ciascuna voce contiene i file richiesti (`README.md`, `DESIGN.md`e almeno un file di anteprima)
3. Estrazione e rendering del contenuto Markdown in HTML
4. Generazione di dati di ingresso normalizzati

### Architettura del sito statico: come fornire una ricerca dinamica rimanendo completamente statico?

Poiché si tratta di una galleria di design, la ricerca è un must. Ma Astro è un generatore di siti statici, quindi come si implementa la ricerca in tempo reale?

**Soluzione: isola React + sincronizzazione dei parametri di query URL**

```typescript
// src/components/gallery/SearchToolbar.tsx

export function SearchToolbar() {

```

Il vantaggio di questo approccio è che mantiene intatta la distribuibilità di un sito statico, il che significa che può essere distribuito a qualsiasi servizio di hosting statico, offrendo comunque un'esperienza di filtraggio istantanea.

### Documentazione di progettazione: come aiutare l'intelligenza artificiale a comprendere e seguire gli standard di progettazione?

Questo è il fulcro dell'intera soluzione. Dobbiamo creare un sistema strutturato `design.md` che l’intelligenza artificiale può comprendere e applicare.

**Soluzione: prendere in prestito la struttura di ClickHouse `DESIGN.md`**

[ClickHouse `DESIGN.md`](https://github.com/ClickHouse/ClickHouse/blob/master/DESIGN.md) è un eccellente riferimento. Include:
- Tema visivo e atmosfera
- Tavolozza dei colori e ruoli
- Regole tipografiche
- Stili dei componenti
- Principi di disposizione
- Profondità ed elevazione
- Cosa fare e cosa non fare
- Comportamento reattivo
- Guida ai suggerimenti dell'agente

Il nostro approccio è: **riutilizzare la struttura, riscrivere il contenuto**. Manteniamo la struttura delle sezioni di ClickHouse `DESIGN.md`, ma sostituiamo il contenuto con i token di progettazione e le convenzioni dei componenti effettivi utilizzati nel nostro progetto.

## Soluzione

Sulla base dell'analisi di cui sopra, la nostra soluzione è composta da quattro moduli principali.

### 1. Pipeline di acquisizione dei contenuti

Questo è il fondamento dell'intero sistema, responsabile dell'estrazione e della normalizzazione dei contenuti dalle risorse a monte.

```typescript
// src/lib/content/awesomeDesignCatalog.ts

export async function scanSourceEntries(): Promise<DesignEntry[]> {
  const designDir = 'vendor/awesome-design-md/design-md';
  const entries: DesignEntry[] = [];

  for (const dir of await fs.readdir(designDir)) {
    const entryPath = path.join(designDir, dir);
    if (await isValidDesignEntry(entryPath)) {
      const entry = await normalizeDesignEntry(entryPath);
      entries.push(entry);
    }
  }

  return entries;
}

async function isValidDesignEntry(dir: string): Promise<boolean> {
  const requiredFiles = ['README.md', 'DESIGN.md'];
  for (const file of requiredFiles) {
    if (!(await fileExists(path.join(dir, file)))) {
      return false;
    }
  }
  return true;
}
```

### 2. Interfaccia di navigazione nella Galleria

L'interfaccia della galleria comprende tre parti principali:

**Homepage**: visualizza una griglia di schede con tutte le voci di progettazione e ciascuna scheda include:
- Titolo e riepilogo della proposta di progetto
- Immagine di anteprima, se disponibile
- Evidenziazione della ricerca rapida

**Pagina dettagli**: aggrega le informazioni complete per un singolo progetto:
- `README` documento
- `DESIGN` documento
- Anteprima con commutazione tema chiaro/scuro
- Navigazione alle voci adiacenti

**Navigazione**: supporta il ritorno alla galleria e la navigazione delle voci adiacenti

La galleria della home page utilizza un layout di carte ad alta densità, appiattito `design.md` voci provenienti da diverse fonti in un quadro visivo unificato in modo che i team possano confrontare rapidamente gli stili del marchio, i modelli dei pulsanti e il ritmo tipografico.

### 3. Funzionalità di ricerca

La funzionalità di ricerca si basa sul filtraggio lato client, con lo stato conservato tramite i parametri di query dell'URL:

```typescript
// src/components/gallery/SearchToolbar.tsx

function SearchToolbar({ entries }: { entries: DesignEntry[] }) {
  const [query, setQuery] = useState('');
  const [results, setResults] = useState(entries);

  useEffect(() => {
    const params = new URLSearchParams(window.location.search);
    const initialQuery = params.get('q') || '';
    setQuery(initialQuery);
    filterEntries(initialQuery);
  }, []);

  const filterEntries = (searchQuery: string) => {
    const filtered = entries.filter(entry =>
      entry.searchText.toLowerCase().includes(searchQuery.toLowerCase())
    );
    setResults(filtered);
  };

  const handleChange = (e: React.ChangeEvent<HTMLInputElement>) => {
    const value = e.target.value;
    setQuery(value);
    filterEntries(value);

    // Update the URL without triggering a page refresh
    const newUrl = value
      ? `${window.location.pathname}?q=${encodeURIComponent(value)}`
      : window.location.pathname;
    window.history.replaceState({}, '', newUrl);
  };

  return (
    <div className="search-toolbar">
      <input
        type="text"
        value={query}
        onChange={handleChange}
        placeholder="Search design entries..."
      />
      <span className="result-count">{results.length} results</span>
    </div>
  );
}
```

## Pratica

Ora che abbiamo trattato la soluzione, come la implementi effettivamente?

### Fasi di implementazione

**Passaggio 1: inizializzare il sottomodulo**

```bash
# Add the upstream repository as a submodule
git submodule add https://github.com/VoltAgent/awesome-design-md.git vendor/awesome-design-md

# Initialize and update the submodule
git submodule update --init --recursive
```

**Passaggio 2: crea la pipeline di contenuti**

Implementare `awesomeDesignCatalog.ts`, tra cui:
- Logica di scansione e convalida dei file
- Rendering Markdown utilizzando il renderer integrato di Astro
- Estrazione dei dati di ingresso

**Passaggio 3: crea l'interfaccia utente della galleria**

Usa Astro + React Islands per creare:
- Layout della galleria home page (griglia delle carte)
- Componenti della scheda di progettazione
- Barra degli strumenti di ricerca
- Layout della pagina dei dettagli

**Passaggio 4: scrivere il documento di progettazione**

Basato sulla struttura di ClickHouse `DESIGN.md`, compila i token di progettazione effettivi dal tuo progetto. Aggiorna `README.md` e aggiungi un collegamento a `design.md`.

### Note

**Sicurezza**: il rendering Markdown richiede il filtraggio di HTML non sicuro. Filtri di rendering integrati di Astro `script` tag per impostazione predefinita, ma è comunque necessario prestare attenzione ai rischi XSS.

**Prestazioni**: un numero elevato di anteprime iframe può influire sulle prestazioni della prima verniciatura. Si consiglia di utilizzare `loading="lazy"` per caricare lentamente il contenuto dell'anteprima.

**Manutenibilità**: `design.md` deve rimanere sincronizzato con l'implementazione del codice. Si consiglia di aggiungere controlli CI per garantire che le variabili CSS rimangano coerenti tra documentazione e codice.

**Accessibilità**: assicurati che il contrasto dei colori soddisfi lo standard WCAG AA (almeno 4,5:1).

### Guida all'utilizzo dell'intelligenza artificiale

Dopo aver creato `design.md`, come si fa a far sì che l'intelligenza artificiale lo utilizzi effettivamente? Ecco alcuni consigli pratici:

**Suggerimento 1: fare riferimento esplicitamente nel prompt**

```
Please refer to the design.md file in the project root and use the design conventions defined there to implement the following components:
- Buttons: use the primary color with an 8px border radius
- Cards: use the elevation-2 shadow level
```

**Suggerimento 2: richiedi all'intelligenza artificiale di fare riferimento a variabili CSS specifiche**

```
Implement a navigation bar with the following requirements:
- Use --color-bg-primary for the background color
- Use --color-border-subtle for borders
- Use --text-color-primary for text
```

**Suggerimento 3: includi `design.md` contenuto nel prompt del sistema**

Se il tuo strumento AI supporta prompt di sistema personalizzati, puoi aggiungere il contenuto principale di `design.md` direttamente ad esso.

### Strategia di test

**Test della pipeline di contenuti**:
- Scenari con file mancanti (missing `README.md` o `DESIGN.md`)
- Scenari di errore di formato (errore di analisi Markdown)
- Scenari di directory vuote

**Test della funzionalità di ricerca**:
- Gestione dei risultati vuoti
- Caratteri speciali come cinese ed emoji
- Verifica della sincronizzazione dell'URL

**Test dei componenti dell'interfaccia utente**:
- Commutazione tema chiaro/scuro
- Disposizione reattiva
- Anteprima degli stati di caricamento

### Flusso di lavoro di distribuzione

```bash
# 1. Update the submodule to the latest version
git submodule update --remote

# 2. Rebuild the site
npm run build

# 3. Deploy static assets
npm run deploy
```

Si consiglia di automatizzare gli aggiornamenti dei sottomoduli oltre alla creazione e alla distribuzione, in modo che l'elemento CI possa essere attivato automaticamente ogni volta che viene aggiornato il repository upstream.

## Conclusione

L'incoerenza nelle UI generate dall'intelligenza artificiale riscontrata da HagiCode durante lo sviluppo era, essenzialmente, causata dalla mancanza di un documento di riferimento per la progettazione strutturata. Costruendo un sito di gallerie di design e creando un file standardizzato `design.md`, abbiamo risolto con successo questo problema.

Il valore fondamentale di questa soluzione risiede in:
- **Risorse unificate**: consolidamento della documentazione sparsa del sistema di progettazione
- **Standard strutturati**: esprimere convenzioni di progettazione in un formato comprensibile dall'intelligenza artificiale
- **Manutenzione continua**: mantenere i contenuti aggiornati `git submodule`

Se utilizzi l’intelligenza artificiale anche per lo sviluppo frontend, vale la pena provare questo approccio. Creare una struttura strutturata `design.md` non solo migliora la coerenza del codice generato dall'intelligenza artificiale, ma aiuta anche il tuo team a mantenere internamente standard di progettazione unificati.
