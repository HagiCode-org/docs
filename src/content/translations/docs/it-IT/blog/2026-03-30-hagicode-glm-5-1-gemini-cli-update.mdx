## Supporto completo GLM-5.1 e integrazione Gemini CLI: il percorso di evoluzione multi-modello di HagiCode

> Questo articolo introduce due importanti aggiornamenti recenti alla piattaforma HagiCode: il supporto completo per il modello Zhipu AI GLM-5.1 e la riuscita integrazione di Gemini CLI come decimo Agent CLI. Insieme, questi aggiornamenti rafforzano ulteriormente le capacità multi-modello della piattaforma e l'ecosistema multi-CLI.

:::nota[Avviso sul contesto temporale]
Questo articolo registra lo stato di avanzamento dell'espansione multi-CLI al momento del rilascio. Per l'ambito corrente sempre supportato, fare riferimento a [Installazione della CLI dell'agente AI](/it-IT/related-software-installation/ai-agent-cli/) e il [Guida all'installazione](/it-IT/installation); il `IFlowCli` qui menzionato rappresenta solo una configurazione storica in quel momento e non fa parte dell'attuale ambito di supporto attivo.

## Sfondo

Il tempo scorre davvero velocemente. Lo sviluppo di grandi modelli linguistici è cresciuto come il bambù in primavera. Non molto tempo fa, facevamo ancora il tifo per "un'intelligenza artificiale in grado di scrivere codice". Ora siamo già in un’era di collaborazione multi-modello e di integrazione multi-strumento. È emozionante? Forse. Dopotutto, ciò di cui gli sviluppatori hanno bisogno non è mai stato solo lo strumento in sé, ma la facilità di adattarsi a diversi scenari e di cambiare in modo flessibile quando necessario.

In quanto piattaforma di codifica assistita dall'intelligenza artificiale, HagiCode ha recentemente accolto con favore due importanti sviluppi: in primo luogo, la piena integrazione del modello GLM-5.1 di Zhipu AI; in secondo luogo, l'aggiunta ufficiale di Gemini CLI come decimo agente CLI supportato. Questi due aggiornamenti potrebbero non sembrare sconvolgenti, ma sono senza dubbio una buona notizia per la continua maturazione della piattaforma.

GLM-5.1 è l'ultimo modello di punta di Zhipu AI. Rispetto a GLM-5.0, offre un ragionamento più forte, una comprensione più profonda del codice e una chiamata degli strumenti più fluida. Ancora più importante, è il primo modello GLM a supportare l'input di immagini. Che cosa significa? Ciò significa che gli utenti possono lasciare che l’intelligenza artificiale guardi direttamente uno screenshot invece di faticare a descrivere il problema a parole. Una volta utilizzata questa comodità, ne capisci immediatamente il valore.

Allo stesso tempo, attraverso il `HagiCode.Libs.Providers` architettura, HagiCode ha integrato con successo Gemini CLI nella piattaforma. Questa è ora la decima CLI dell'agente. Ad essere onesti, arrivare a questo punto porta un modesto senso di realizzazione.

Vale anche la pena ricordare che la funzione di caricamento delle immagini di HagiCode consente agli utenti di comunicare con l'intelligenza artificiale direttamente tramite screenshot. Anche quando si esegue GLM 4.7, la piattaforma funziona ancora bene e ha già contribuito a completare molte importanti attività di creazione. Per quanto riguarda GLM-5.1, naturalmente, fa un ulteriore passo avanti.

L'approccio condiviso in questo articolo deriva dalla nostra esperienza pratica nel [HagiCode](https://hagicode.com) progetto. ## Progettazione di architetture multi-CLI

Uno dei punti di forza principali di HagiCode è il supporto per più strumenti CLI di programmazione AI attraverso un livello di astrazione unificato. Il vantaggio di questo design è in realtà abbastanza semplice: possono entrare nuovi strumenti, i vecchi strumenti possono rimanere e la base di codice non si trasforma nel caos. A dire il vero, tutti vorrebbero che la vita funzionasse così.

### Enumerazione AIProviderType

La piattaforma definisce i tipi di provider CLI supportati tramite `AIProviderType` enumerazione:

```csharp
public enum AIProviderType
{
    ClaudeCodeCli = 0,    // Claude Code CLI
    CodexCli = 1,          // GitHub Copilot Codex
    GitHubCopilot = 2,     // GitHub Copilot
    CodebuddyCli = 3,     // Codebuddy CLI
    OpenCodeCli = 4,      // OpenCode CLI
    IFlowCli = 5,         // IFlow CLI
    HermesCli = 6,        // Hermes CLI
    QoderCli = 7,         // Qoder CLI
    KiroCli = 8,          // Kiro CLI
    KimiCli = 9,          // Kimi CLI
    GeminiCli = 10,       // Gemini CLI (new)
}
```

Come puoi vedere, Gemini CLI si unisce a questa famiglia come decimo membro. Ciascuna CLI ha caratteristiche e scenari di utilizzo distinti, quindi gli utenti possono scegliere in modo flessibile in base alle proprie esigenze. Dopotutto, molte strade portano a Roma; alcuni sono semplicemente più facili di altri.

### Architettura del fornitore

`HagiCode.Libs.Providers` fornisce un'interfaccia provider unificata che rende ogni integrazione CLI standardizzata e concisa. Prendendo Gemini CLI come esempio:

```csharp
public class GeminiProvider : ICliProvider<GeminiOptions>
{
    private static readonly string[] DefaultExecutableCandidates = ["gemini", "gemini-cli"];
    private const string ManagedBootstrapArgument = "--acp";

    public string Name => "gemini";
    public bool IsAvailable => _executableResolver.ResolveFirstAvailablePath(DefaultExecutableCandidates) is not null;
}
```

I vantaggi di questo design sono:
- L'integrazione di una nuova CLI richiede solo l'implementazione di una classe Provider
- Gestione unificata del ciclo di vita e pooling delle sessioni
- Risoluzione automatica degli alias e rilevamento degli eseguibili

In parole povere, questo design trasforma le cose complicate in cose più semplici e rende la vita un po’ più semplice.

### Registro dei fornitori

Il Registro dei provider gestisce automaticamente la mappatura e la registrazione degli alias:

```csharp
if (provider is GeminiProvider)
{
    registry.Register(provider.Name, provider, ["gemini-cli"]);
    continue;
}
```

Ciò significa che gli utenti possono richiamare Gemini CLI con entrambi `gemini` o `gemini-cli`, e il sistema lo riconoscerà automaticamente. È come un amico con sia un nome formale che un soprannome: in ogni caso, le persone sanno a chi ti riferisci.

## Supporto del modello GLM-5.1

GLM-5.1 è l'ultimo modello di punta di Zhipu AI e HagiCode ne ha completato il pieno supporto.

### Catalogo delle professioni secondarie

HagiCode gestisce tutti i modelli supportati attraverso il Catalogo delle Professioni Secondarie. Ecco la configurazione per la serie GLM:

| ID modello | Nome | Supporta l'immagine | Famiglie CLI compatibili |
|----------|------|----------------|-------------------------|
| `glm-4.7` | GLM4.7 | - | claude, codebuddy, hermes, qoder, kiro |
| `glm-5` | GLM5 | - | claude, codebuddy, hermes, qoder, kiro |
| `glm-5-turbo` | GLM5 Turbo | - | claude, codebuddy, hermes, qoder, kiro |
| `glm-5.0` | GLM 5.0 (precedente) | - | claude, codebuddy, hermes, qoder, kiro |
| `glm-5.1` | GLM5.1 | **vero** | claude, codebuddy, hermes, qoder, kiro |

Le caratteristiche principali di GLM-5.1 possono essere riassunte come segue:
- Un identificatore di versione autonomo senza bagaglio legacy
- Il primo modello GLM a supportare l'input di immagini
- Ragionamento e comprensione del codice più forti
- Ampia compatibilità multi-CLI

### GLM-5.1 contro GLM-5.0

A livello di codice, la differenza chiave tra GLM-5.1 e GLM-5.0 è mostrata qui:

```csharp
// GLM-5.0 (Legacy) - contains special retention logic
private const string Glm50CodebuddySecondaryProfessionId = "secondary-glm-5-codebuddy";
private const string Glm50CodebuddyModelValue = "glm-5.0";

// GLM-5.1 - standalone new model identifier
private const string Glm51SecondaryProfessionId = "secondary-glm-5-1";
private const string Glm51ModelValue = "glm-5.1";
```

GLM-5.0 porta l'etichetta "Legacy" perché è un identificatore di versione precedente mantenuto per compatibilità con le versioni precedenti. GLM-5.1, al contrario, è una nuovissima versione standalone senza oneri storici. Alcune cose restano nel passato; altri viaggiano più leggeri e si muovono più velocemente.

### Configura GLM-5.1

Ecco un esempio di configurazione per l'utilizzo di GLM-5.1 in HagiCode:

```json
{
  "primaryProfessionId": "profession-claude-code",
  "secondaryProfessionId": "secondary-glm-5-1",
  "model": "glm-5.1",
  "reasoning": "high"
}
```

## Funzionalità di caricamento delle immagini

Il supporto delle immagini di HagiCode è implementato tramite il file `SupportsImage` proprietà su `SecondaryProfession`:

```csharp
public class HeroSecondaryProfessionSettingDto
{
    public bool SupportsImage { get; set; }
}
```

Nel Catalogo delle professioni secondarie, la configurazione GLM-5.1 si presenta così:

```json
{
  "id": "secondary-glm-5-1",
  "supportsImage": true
}
```

Ciò significa che gli utenti possono caricare screenshot direttamente per l'analisi AI, come ad esempio:
- Schermate dei messaggi di errore
- Problemi in una schermata dell'interfaccia utente
- Grafici di visualizzazione dei dati
- Risultati dell'esecuzione del codice

Non è più necessario descrivere tutto manualmente: basta caricare lo screenshot. La comodità di questa funzione è evidente una volta utilizzata. A volte uno sguardo dice più di una lunga spiegazione.

## Integrazione della CLI Gemini

Essendo il decimo Agent CLI, Gemini CLI è integrato in HagiCode tramite l'architettura standard del Provider.

### Opzioni di configurazione

Gemini CLI supporta un ricco set di opzioni di configurazione:

```csharp
public class GeminiOptions
{
    public string? ExecutablePath { get; set; }
    public string? WorkingDirectory { get; set; }
    public string? SessionId { get; set; }
    public string? Model { get; set; }
    public string? AuthenticationMethod { get; set; }
    public string? AuthenticationToken { get; set; }
    public Dictionary<string, string?> AuthenticationInfo { get; set; }
    public Dictionary<string, string?> EnvironmentVariables { get; set; }
    public string[] ExtraArguments { get; set; }
    public TimeSpan? StartupTimeout { get; set; }
    public CliPoolSettings? PoolSettings { get; set; }
}
```

Queste opzioni coprono tutto, dalla configurazione di base alle funzionalità avanzate, offrendo agli utenti la flessibilità di configurare la CLI in base alle proprie esigenze. Il flusso di lavoro di ognuno è diverso, quindi un po' di flessibilità è sempre benvenuta.

### Protocollo di comunicazione ACP

Gemini CLI supporta ACP (Agent Communication Protocol), che è lo standard di comunicazione CLI unificato di HagiCode. Attraverso ACP diverse CLI possono interagire con la piattaforma in modo coerente, semplificando notevolmente il lavoro di integrazione. In breve, standardizza le parti complicate in modo che tutti possano lavorare più facilmente.

## Best practice per l'integrazione multi-CLI

`HagiCode.Libs.Providers` fornisce un meccanismo unificato per la registrazione e l'utilizzo:

```csharp
services.AddHagiCodeLibs();

var gemini = serviceProvider.GetRequiredService<ICliProvider<GeminiOptions>>();
var codebuddy = serviceProvider.GetRequiredService<ICliProvider<CodebuddyOptions>>();
var hermes = serviceProvider.GetRequiredService<ICliProvider<HermesOptions>>();
```

Questo design di inserimento delle dipendenze mantiene l'utilizzo tra diverse CLI molto conciso e rende anche più convenienti i test unitari e il mocking. Il codice pulito è un modo di essere responsabile verso te stesso.

### Note

Ci sono alcune cose da tenere a mente nell'uso reale:

1. **Configurazione chiave API**: assicurati `ANTHROPIC_AUTH_TOKEN` è impostato correttamente oppure non è possibile richiamare il modello
2. **Disponibilità del modello**: GLM-5.1 deve essere abilitato dal fornitore del modello corrispondente
3. **Caratteristica immagine**: Solo modelli con `supportsImage: true` può utilizzare il caricamento di immagini
4. **Installazione CLI**: prima di utilizzare Gemini CLI, assicurati `gemini` o `gemini-cli` si trova nel PATH del sistema

Potrebbero essere piccoli dettagli, ma piccoli dettagli gestiti male possono trasformarsi in grossi problemi, quindi vale la pena prestare attenzione.
