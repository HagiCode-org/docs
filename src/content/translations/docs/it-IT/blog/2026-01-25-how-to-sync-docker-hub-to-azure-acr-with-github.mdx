## Automatizzazione della sincronizzazione delle immagini da Docker Hub ad Azure ACR

> Questo articolo spiega come utilizzare GitHub Actions e lo strumento di sincronizzazione delle immagini per automatizzare la sincronizzazione delle immagini da Docker Hub al Registro contenitori di Azure, risolvendo il problema dell'accesso lento a Docker Hub in Cina e in alcune aree di Azure e migliorando al tempo stesso la disponibilità delle immagini e l'efficienza della distribuzione negli ambienti Azure.

## Contesto/Introduzione

Il progetto HagiCode utilizza le immagini Docker come componenti runtime principali, con le immagini primarie ospitate su Docker Hub. Con la crescita del progetto e l'aumento delle esigenze di distribuzione di Azure, abbiamo riscontrato i seguenti punti critici:

- Velocità di pull delle immagini lente, con accesso Docker Hub limitato in Cina e in alcune aree di Azure
- Una singola origine immagine crea un singolo punto di rischio di errore
- Registro contenitori di Azure offre prestazioni di rete e integrazione migliori durante l'esecuzione in Azure

Per risolvere questi problemi, avevamo bisogno di un meccanismo di sincronizzazione automatizzata delle immagini per sincronizzare regolarmente le immagini da Docker Hub ad Azure ACR, garantendo che gli utenti possano ottenere velocità di pull delle immagini più elevate e maggiore disponibilità negli ambienti Azure.

## Confronto delle soluzioni tecniche

Nel definire la soluzione, abbiamo confrontato più opzioni tecniche:

### 1. sincronizzatore di immagini (scelta finale)
- Sincronizzazione incrementale: sincronizza solo i livelli di immagine modificati, riducendo significativamente il trasferimento di rete
- Riprendi il supporto: la sincronizzazione può continuare dopo le interruzioni della rete
- Controllo della concorrenza: i thread di lavoro configurabili migliorano l'efficienza della sincronizzazione di immagini di grandi dimensioni
- Gestione efficace degli errori: meccanismo di ripetizione integrato per gli errori (3 tentativi per impostazione predefinita)
- Distribuzione leggera: un singolo binario senza dipendenze
- Supporto multi-registro: compatibile con Docker Hub, Azure ACR, Harbour e altro

### 2. CLI Docker
- Nessuna sincronizzazione incrementale: l'intero contenuto dell'immagine deve essere estratto ogni volta
- Efficienza inferiore: volume di trasferimento di rete elevato e tempo di esecuzione più lungo
- Semplice da usare: basato su familiarità `docker pull` e `docker push` comandi

### 3. interfaccia della riga di comando di Azure
- Complessità maggiore: è necessario configurare l'autenticazione dell'interfaccia della riga di comando di Azure
- Funzionalità limitate: `az acr import` ha una portata relativamente ristretta
- Integrazione nativa: si integra bene con i servizi Azure

## Soluzione fondamentale

Utilizziamo una soluzione automatizzata basata su GitHub Actions + image-syncer per sincronizzare le immagini da Docker Hub ad Azure ACR.

## Fasi di implementazione

### 1. Preparazione
- Creare o confermare un Registro contenitori di Azure nel portale di Azure
- Creare credenziali di accesso ACR (nome utente e password)
- Conferma le autorizzazioni di accesso per il repository di immagini Docker Hub

### 2. Configura i segreti GitHub
Aggiungi i seguenti segreti nelle impostazioni del repository GitHub:
- AZURE_ACR_USERNAME: nome utente ACR di Azure
- AZURE_ACR_PASSWORD: password ACR di Azure

### 3. Creare il flusso di lavoro Azioni GitHub
Configura il flusso di lavoro in `.github/workflows/sync-docker-acr.yml`:
- Trigger programmato: ogni giorno alle 00:00 UTC
- Trigger manuale: supporta `workflow_dispatch`
- Trigger aggiuntivo: viene eseguito tramite push su `publish` ramo per la sincronizzazione veloce

### 4. Flusso di esecuzione del flusso di lavoro

| Passo | Partecipante | Azione | Descrizione |
| --- | --- | --- | --- |
| 1 | Azioni GitHub | Attiva il flusso di lavoro | Attivato dalla pianificazione, dall'esecuzione manuale o dai push al `publish` ramo |
| 2 | Azioni GitHub → sincronizzatore di immagini | Scarica ed esegui lo strumento di sincronizzazione | Entra nella fase di sincronizzazione vera e propria |
| 3 | sincronizzatore di immagini → Docker Hub | Recupera i manifest delle immagini e l'elenco dei tag | Legge i metadati dal repository di origine |
| 4 | sincronizzatore di immagini → Azure ACR | Recupera i dati dell'immagine esistente dal repository di destinazione | Determina lo stato corrente del lato di destinazione |
| 5 | sincronizzatore di immagini | Confrontare le differenze di origine e di destinazione | Identifica i livelli dell'immagine che necessitano di sincronizzazione |
| 6 | sincronizzatore di immagini → Docker Hub | Estrai i livelli di immagine modificati | Trasferisce solo il contenuto che necessita di aggiornamento |
| 7 | sincronizzatore di immagini → Azure ACR | Spingi i livelli di immagine modificati | Completa la sincronizzazione incrementale |
| 8 | sincronizzatore di immagini → Azioni GitHub | Restituisce le statistiche di sincronizzazione | Include risultati, differenze e informazioni sugli errori |
| 9 | Azioni GitHub | Registra i log e carica gli artefatti | Supporta il controllo e la risoluzione dei problemi successivi |

## Implementazione del flusso di lavoro delle azioni GitHub

Di seguito è riportata la configurazione effettiva del flusso di lavoro in uso (`.github/workflows/sync-docker-acr.yml`):

```yaml
name: Sync Docker Image to Azure ACR

on:
  schedule:
    - cron: "0 0 * * *" # Every day at 00:00 UTC
  workflow_dispatch: # Manual trigger
  push:
    branches: [publish]

permissions:
  contents: read

jobs:
  sync:
    runs-on: ubuntu-latest

    steps:
      - name: Checkout code
        uses: actions/checkout@v4

      - name: Download image-syncer
        run: |
          # Download the image-syncer binary
          wget https://github.com/AliyunContainerService/image-syncer/releases/download/v1.5.5/image-syncer-v1.5.5-linux-amd64.tar.gz
          tar -zxvf image-syncer-v1.5.5-linux-amd64.tar.gz
          chmod +x image-syncer

      - name: Create auth config
        run: |
          # Generate the authentication config file (YAML format)
          cat > auth.yaml <<EOF
          hagicode.azurecr.io:
            username: "${{ secrets.AZURE_ACR_USERNAME }}"
            password: "${{ secrets.AZURE_ACR_PASSWORD }}"
          EOF

      - name: Create images config
        run: |
          # Generate the image sync config file (YAML format)
          cat > images.yaml <<EOF
          docker.io/newbe36524/hagicode: hagicode.azurecr.io/hagicode
          EOF

      - name: Run image-syncer
        run: |
          # Run synchronization (using the newer --auth and --images parameters)
          ./image-syncer --auth=./auth.yaml --images=./images.yaml --proc=10 --retries=3

      - name: Upload logs
        if: always()
        uses: actions/upload-artifact@v4
        with:
          name: sync-logs
          path: image-syncer-*.log
          retention-days: 7
```

## Dettagli di configurazione

### 1. Condizioni scatenanti
- Trigger pianificato: `cron: "0 0 * * *"` viene eseguito tutti i giorni alle 00:00 UTC
- Attivazione manuale: `workflow_dispatch` consente agli utenti di eseguirlo manualmente dall'interfaccia utente di GitHub
- Premi il grilletto: `push: branches: [publish]` si attiva durante i push al ramo di rilascio per una sincronizzazione rapida

### 2. Configurazione dell'autenticazione (`auth.yaml`)
```yaml
hagicode.azurecr.io:
  username: "${{ secrets.AZURE_ACR_USERNAME }}"
  password: "${{ secrets.AZURE_ACR_PASSWORD }}"
```

### 3. Configurazione della sincronizzazione delle immagini
```yaml
docker.io/newbe36524/hagicode: hagicode.azurecr.io/hagicode
```
Questa configurazione significa che tutti i tag da `docker.io/newbe36524/hagicode` verrà sincronizzato con `hagicode.azurecr.io/hagicode`.

### 4. parametri del sincronizzatore di immagini
- `--auth=./auth.yaml`: percorso del file di configurazione dell'autenticazione
- `--images=./images.yaml`: percorso del file di configurazione della sincronizzazione delle immagini
- `--proc=10`: utilizzare `10` lavoratori contemporanei
- `--retries=3`: riprovare gli errori `3` volte

## Note e migliori pratiche

### 1. Raccomandazioni sulla sicurezza
- Ruota regolarmente le password ACR di Azure (consigliato ogni 90 giorni)
- Utilizza un account di servizio ACR dedicato con autorizzazioni di sola push
- Monitora i log di accesso ACR per rilevare tempestivamente accessi anomali
- Non visualizzare le credenziali nei log
- Non impegnare le credenziali nel repository del codice

### 2. Ottimizzazione delle prestazioni
- Regola il `--proc` parametro: ottimizza la concorrenza in base alla larghezza di banda della rete (consigliato `5-20`)
- Monitorare la durata della sincronizzazione: se la sincronizzazione richiede troppo tempo, valutare la possibilità di ridurre la concorrenza
- Pulisci regolarmente i registri: imposta un valore ragionevole `retention-days` valore (`7` attualmente)

### 3. Risoluzione dei problemi

#### Problema 1: errore di autenticazione
```
Error: failed to authenticate to hagicode.azurecr.io
```
Soluzione:
1. Controlla se i segreti GitHub sono configurati correttamente
2. Verificare se la password di Azure ACR è scaduta
3. Confermare che le autorizzazioni dell'account del servizio ACR siano corrette

#### Problema 2: timeout della rete
```
Error: timeout waiting for response
```
Soluzione:
1. Controlla la connettività di rete
2. Ridurre il numero di lavoratori simultanei (i `--proc` parametro)
3. Attendi il ripristino della rete ed esegui nuovamente il flusso di lavoro

#### Problema 3: sincronizzazione delle immagini incompleta
```
Warning: some tags failed to sync
```
Soluzione:
1. Controlla i log di sincronizzazione per identificare i tag non riusciti
2. Eseguire nuovamente manualmente il flusso di lavoro per eseguire nuovamente la sincronizzazione
3. Verificare che l'immagine di origine su Docker Hub sia integra

### 4. Monitoraggio e allerta
- Controlla regolarmente la pagina Azioni per confermare lo stato del flusso di lavoro
- Abilita le notifiche GitHub in modo che gli errori del flusso di lavoro vengano segnalati tempestivamente
- Monitorare l'utilizzo dello spazio di archiviazione ACR di Azure
- Verificare regolarmente la coerenza dei tag tra i registri

## Domande e soluzioni frequenti

### Q1: Come posso sincronizzare tag specifici invece di tutti i tag?

Modifica il `images.yaml` file di configurazione:
```yaml
# Sync only the latest and v1.0 tags
docker.io/newbe36524/hagicode:latest: hagicode.azurecr.io/hagicode:latest
docker.io/newbe36524/hagicode:v1.0: hagicode.azurecr.io/hagicode:v1.0
```

### Q2: Come posso sincronizzare più repository di immagini?

Aggiungi più righe a `images.yaml`:
```yaml
docker.io/newbe36524/hagicode: hagicode.azurecr.io/hagicode
docker.io/newbe36524/another-image: hagicode.azurecr.io/another-image
```

### Q3: Come posso riprovare dopo che la sincronizzazione non è riuscita?
- Nuovo tentativo automatico: il sincronizzatore di immagini include un meccanismo di nuovo tentativo integrato (`3` riprova per impostazione predefinita)
- Riprova manuale: fare clic `Re-run all jobs` nella pagina Azioni GitHub

### Q4: Come posso visualizzare i progressi dettagliati della sincronizzazione?
- Visualizza i log in tempo reale nella pagina Azioni
- Scarica il `sync-logs` artefatto per esaminare i file di registro completi
- I file di registro includono lo stato di sincronizzazione e la velocità di trasferimento per ciascun tag

### Q5: Quanto tempo richiede la sincronizzazione?
- Sincronizzazione completa iniziale: di solito `10-30` minuti a seconda delle dimensioni dell'immagine
- Sincronizzazione incrementale: solitamente `2-5` minuti se i cambiamenti dell'immagine sono piccoli
- La tempistica dipende dalla larghezza di banda della rete, dalle dimensioni dell'immagine e dalle impostazioni di concorrenza
