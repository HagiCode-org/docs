HagiCode Desktop viene fornito con una toolchain Node gestita da desktop. Non sostituisce l'installazione globale del nodo sul tuo computer. Il suo scopo è fornire all'avvio del desktop e ai flussi correlati al runtime una base di runtime stabile e verificabile.

Leggi queste FAQ se vedi `portable-fixed/toolchain` in diagnostica, è necessario sapere quale `node` Desktop sta utilizzando o ti chiedi perché la versione del nodo del tuo sistema non definisce direttamente il comportamento di runtime del desktop.

## Perché Desktop raggruppa Node

Il desktop deve eseguire gli stessi flussi di supporto tra sistemi operativi, canali di installazione e ambienti utente. Se dipendesse solo dall'installazione del nodo esistente dell'utente, il supporto dovrebbe gestire contemporaneamente versioni obsolete, npm mancante, ordinamento errato dei percorsi, pacchetti globali inquinati e policy di registro specifiche dell'organizzazione.

Desktop lo descrive più specificamente come runtime Node/npm isolato per CLI gestite invece di riutilizzare l'ambiente a livello di sistema. Questo design presenta questi vantaggi principali:

- **Versioni coerenti**: Desktop utilizza un contratto di runtime Nodo 22 bloccato invece di indovinare la versione principale locale dell'utente.
- **Flussi di avvio controllati**: i comandi gestiti dal desktop possono preferire quelli in pacchetto `node` e `npm` voci.
- **Chiari limiti di configurazione**: la configurazione relativa alla CLI viene mantenuta separata dalla configurazione globale a livello di host, pertanto è meno probabile che aggiornamenti, passaggi e ripristini vengano interrotti dalle differenze dell'ambiente di sistema.
- **Conflitti ridotti**: i percorsi in bundle si applicano agli ambienti di comando creati da Desktop e non sostituiscono il nodo globale dell'utente nel suo terminale.
- **Comportamento host non intrusivo**: questo isolamento evita di scrivere lo stato della toolchain nell'ambiente Node/npm esistente dell'utente e la disinstallazione di Desktop non disturba il nodo, npm o la configurazione già presente sulla macchina.

## Come Desktop seleziona il percorso del nodo

Quando esegui Desktop normalmente, Desktop preferisce la toolchain Node installata con l'app invece di leggere prima l'installazione di Node dal tuo sistema. Ciò mantiene la versione del nodo nel terminale separata dalla versione del nodo prevista dal desktop durante l'esecuzione.

Quando la toolchain in bundle è abilitata, gli ambienti di comando creati da Desktop mettono in primo piano i percorsi della toolchain in bundle `PATH`. Di conseguenza, anche se nel sistema è installata un'altra versione di Node, i flussi gestiti da Desktop si risolvono prima nel nodo in bundle di Desktop e in npm.

Ciò riguarda solo gli ambienti runtime creati da Desktop. Se corri `node`, `npm`o altri comandi direttamente nel terminale di sistema, continuano a utilizzare il normale ambiente di sistema e la configurazione della shell.

Negli ambienti di diagnostica o di processi secondari avviati da Desktop, potresti visualizzare una variabile come questa:

```bash
HAGICODE_PORTABLE_TOOLCHAIN_ROOT=/path/to/portable-fixed/toolchain
```

Questa variabile indica che l'ambiente runtime gestito da desktop ha riconosciuto la root della toolchain in bundle. Vedere questo percorso non significa che Desktop abbia cambiato il nodo del tuo sistema; significa solo che Desktop utilizza l'ambiente Node fornito con l'app all'interno del proprio ambito di runtime.

## Dove Desktop installa i pacchetti npm gestiti

Il desktop conserva i file di runtime e i file del pacchetto npm modificabili in posizioni diverse. Il percorso della toolchain in bundle come `portable-fixed/toolchain/node` è la fonte immutabile per i file gestiti da desktop `node` e `npm` comandi. I pacchetti globali gestiti vengono invece installati nella directory dei dati utente di Electron:

```text
userData/node22/npmGlobal
```

Se Desktop successivamente si sposta sul Nodo 24, utilizza un albero dei pacchetti separato come `userData/node24/npmGlobal`. I pacchetti installati per il Nodo 22 rimangono sul disco ma non vengono considerati come installati per il Nodo 24 finché non vengono reinstallati nel nuovo prefisso del Nodo principale.

Lo stato delle dipendenze desktop controlla solo l'attivo `userData/node<major>/npmGlobal` prefisso. I file del pacchetto esterni a tale prefisso vengono ignorati dalla gestione delle dipendenze.
