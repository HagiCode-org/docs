# Introducción a OpenSpec

OpenSpec es una herramienta de desarrollo impulsada por especificaciones que ayuda a los equipos a gestionar el proceso de desarrollo de software mediante propuestas y especificaciones estructuradas. Esta guía ofrece una introducción completa a los conceptos principales, flujos de trabajo y el uso de OpenSpec.

## ¿Qué es OpenSpec?

OpenSpec es una herramienta de línea de comandos para gestionar Proposals, Specifications y Changes a lo largo del proceso de desarrollo de software. Estandariza el flujo de trabajo de desarrollo, asegurando que cada característica pase por un diseño y revisión exhaustivos desde la idea hasta la implementación.

### Filosofía central

OpenSpec se basa en los siguientes principios fundamentales:

- **Especificación primero**: Definir especificaciones claras antes de escribir código
- **Colaboración estructurada**: Promover la colaboración del equipo mediante formatos de documentos estandarizados
- **Trazabilidad**: Registrar el proceso completo desde la propuesta hasta la implementación
- **Aseguramiento de calidad**: Garantizar que el código cumpla con la intención de diseño mediante la validación de especificaciones

### Funcionalidades clave

OpenSpec proporciona las siguientes funcionalidades clave:

1. **Gestión de propuestas**: Crear, revisar y hacer seguimiento de propuestas de características
2. **Definición de especificaciones**: Redactar y mantener documentos de especificación técnica
3. **Seguimiento de cambios**: Registrar todo el historial de cambios revisados
4. **Validación de especificaciones**: Asegurar que los cambios de código cumplan con las especificaciones predefinidas
5. **Colaboración en equipo**: Soportar procesos de desarrollo y revisión colaborativos

## Flujo de trabajo de OpenSpec

El flujo de trabajo típico de OpenSpec incluye las siguientes etapas:

### 1. Crear una propuesta

Cuando hay una nueva característica o un cambio importante, primero crea una propuesta:

```bash
openspec proposal create
```

Una propuesta contiene:
- **Descripción del problema**: El problema o requisito a resolver
- **Solución**: Resumen de la solución propuesta
- **Alcance de impacto**: Impacto en los sistemas existentes
- **Alternativas**: Otras soluciones consideradas

### 2. Escribir una especificación

Después de que la propuesta sea aprobada, escriba una especificación técnica detallada:

```bash
openspec spec create
```

Una especificación contiene:
- **Diseño técnico**: Plan de implementación técnica detallado
- **Definición de interfaces**: API, estructuras de datos y otras definiciones de interfaz
- **Detalles de implementación**: Lógica de implementación clave y algoritmos
- **Plan de pruebas**: Cómo verificar la corrección de la implementación

### 3. Crear un cambio

Al comenzar la implementación, cree un change para rastrear el proceso de desarrollo:

```bash
openspec change create
```

Un change contiene:
- **Especificación asociada**: La especificación que este change implementa
- **Tareas de implementación**: Lista de tareas de desarrollo específicas
- **Seguimiento del progreso**: Estado de finalización de las tareas
- **Cambios de código**: Modificaciones reales del código

### 4. Validar y archivar

Después de completar el desarrollo, valide si el change cumple con la especificación y luego archívelo:

```bash
openspec change validate
openspec change archive
```

## Estructura de directorios de OpenSpec

OpenSpec crea un directorio `openspec/` en la raíz del proyecto con la siguiente estructura:

```
openspec/
├── proposals/           # Directorio de propuestas
│   ├── PROPOSAL-001.md
│   └── PROPOSAL-002.md
├── specs/              # Directorio de especificaciones
│   ├── SPEC-001.md
│   └── SPEC-002.md
├── changes/            # Directorio de cambios
│   ├── feature-name/
│   │   ├── proposal.md
│   │   ├── design.md
│   │   ├── specs.md
│   │   └── tasks.md
│   └── another-feature/
└── archived/           # Cambios archivados
    └── feature-name/
```

### Descripción de directorios

- **proposals/**: Almacena todos los documentos de propuesta, un archivo Markdown independiente por propuesta
- **specs/**: Almacena documentos de especificación técnica que corresponden a las propuestas
- **changes/**: Almacena cambios en curso, un subdirectorio por change
- **archived/**: Almacena los cambios completados y archivados

## Comandos principales

### Comandos de propuesta

```bash
# Crear una nueva propuesta
openspec proposal create

# Listar todas las propuestas
openspec proposal list

# Ver detalles de una propuesta
openspec proposal view <proposal-id>

# Aprobar una propuesta
openspec proposal approve <proposal-id>
```

### Comandos de especificación

```bash
# Crear una nueva especificación
openspec spec create

# Listar todas las especificaciones
openspec spec list

# Ver detalles de una especificación
openspec spec view <spec-id>

# Actualizar una especificación
openspec spec update <spec-id>
```

### Comandos de change

```bash
# Crear un nuevo change
openspec change create

# Listar todos los changes
openspec change list

# Ver detalles de un change
openspec change view <change-name>

# Añadir una tarea
openspec change add-task <change-name>

# Validar un change
openspec change validate <change-name>

# Archivar un change
openspec change archive <change-name>
```

### Comandos generales

```bash
# Ver versión
openspec --version

# Ver ayuda
openspec --help

# Ver ayuda de un comando específico
openspec <command> --help
```

## Uso de OpenSpec en proyectos HagiCode

### Integración del proyecto

En proyectos HagiCode, OpenSpec está integrado en el flujo de trabajo de desarrollo:

1. **Comprobación de inicialización**: El asistente de inicio comprueba si OpenSpec está instalado
2. **Desarrollo impulsado por propuestas**: Todas las nuevas características deben pasar por el proceso de propuestas de OpenSpec
3. **Revisión de código**: Los cambios deben cumplir con las especificaciones OpenSpec para ser fusionados
4. **Sincronización de documentación**: La documentación de OpenSpec se mantiene sincronizada con la implementación del código

### Requisitos de versión

Los proyectos HagiCode requieren la versión **OpenSpec >=1.0.0 {'<'}2.0.0**.

Comando de instalación:

```bash
npm install -g @fission-ai/openspec@1
```

### Ejemplo de flujo de trabajo

A continuación se muestra un ejemplo completo de uso de OpenSpec en un proyecto HagiCode:

#### Paso 1: Crear una propuesta

```bash
cd /path/to/hagicode-project
openspec proposal create
```

Introduzca en el indicador interactivo:
- Título de la propuesta: Añadir la función de autenticación de usuarios
- Descripción del problema: El sistema necesita inicio de sesión y gestión de permisos
- Solución: Implementar un sistema de autenticación basado en JWT

#### Paso 2: Escribir una especificación

Después de aprobar la propuesta, cree una especificación detallada:

```bash
openspec spec create
```

Defina en la especificación:
- Diseño del flujo de autenticación
- Definición de la interfaz API
- Estructura de tablas de la base de datos
- Consideraciones de seguridad

#### Paso 3: Crear un change

Comience la implementación:

```bash
openspec change create
```

Introduzca el nombre del change: `add-user-authentication`

#### Paso 4: Añadir tareas

```bash
openspec change add-task add-user-authentication
```

Añada tareas específicas:
- Implementar el endpoint de registro de usuarios
- Implementar el endpoint de inicio de sesión
- Implementar middleware JWT
- Escribir pruebas unitarias

#### Paso 5: Implementar y validar

Tras completar el código, valide el change:

```bash
openspec change validate add-user-authentication
```

#### Paso 6: Archivar

Una vez validado, archive el change:

```bash
openspec change archive add-user-authentication
```

## Buenas prácticas

### Redacción de propuestas

1. **Enunciado claro del problema**: Defina claramente el problema a resolver y evite descripciones vagas
2. **Considerar alternativas**: Enumere y compare diferentes soluciones
3. **Evaluar el impacto**: Analice el impacto y los riesgos en los sistemas existentes
4. **Mantener la concisión**: Las propuestas deben ser breves; los detalles pertenecerán a la especificación

### Redacción de especificaciones

1. **Detallado pero no redundante**: Proporcione suficiente detalle, evite el sobreingeniería
2. **Usar diagramas**: Utilice diagramas de flujo, diagramas de arquitectura, etc., para ayudar a la explicación
3. **Definir interfaces**: Defina claramente todas las interfaces públicas y estructuras de datos
4. **Considerar los casos límite**: Enumere situaciones excepcionales y el manejo de errores

### Gestión de cambios

1. **Pequeñas iteraciones**: Divida las grandes características en varios changes pequeños
2. **Actualizaciones oportunas**: Mantenga sincronizados el estado de las tareas y la documentación
3. **Revisión de código**: Realice revisiones de código tras la finalización de los changes
4. **Archivado oportuno**: Archive los changes completados para mantener limpio el directorio de trabajo

### Colaboración del equipo

1. **Revisiones periódicas**: Revise regularmente las propuestas y las especificaciones para asegurar el consenso del equipo
2. **Documentación primero**: Complete la documentación antes de comenzar a codificar
3. **Mantener la comunicación**: Documente discusiones y decisiones en las propuestas y especificaciones
4. **Control de versiones**: Incluya los documentos OpenSpec en el control de versiones

## Casos de uso comunes

### Escenario 1: Añadir nuevas funciones

1. Cree una propuesta que describa los requisitos de la nueva función
2. El equipo discute y aprueba la propuesta
3. Redacte una especificación técnica detallada
4. Cree un change e implemente la función
5. Valide y archive el change

### Escenario 2: Refactorizar código existente

1. Cree una propuesta que explique las razones y objetivos de la refactorización
2. Defina el plan de refactorización y los pasos en la especificación
3. Cree un change y refactorice por etapas
4. Asegure la cobertura de pruebas y valide los resultados de la refactorización
5. Archive el change

### Escenario 3: Corregir errores

Para errores sencillos:
- Pueden corregirse directamente sin el proceso completo de OpenSpec

Para errores complejos:
1. Cree una propuesta que analice la causa raíz del error
2. Defina la corrección en la especificación
3. Cree un change e implemente la corrección
4. Añada pruebas para evitar regresiones
5. Archive el change

### Escenario 4: Gestión de deuda técnica

1. Cree una propuesta que enumere los elementos de deuda técnica
2. Defina un plan de mejora en la especificación
3. Cree changes para pagar la deuda progresivamente
4. Realice seguimiento del progreso y la efectividad
5. Archive las mejoras completadas

## Integración con otras herramientas

### Integración con Git

OpenSpec se integra sin problemas con Git:

```bash
# Los documentos OpenSpec deben incluirse en el control de versiones
git add openspec/
git commit -m "Add OpenSpec proposal for feature X"
```

### Integración CI/CD

Valide las especificaciones OpenSpec en los pipelines CI/CD:

```yaml
# .github/workflows/openspec.yml
name: OpenSpec Validation
on: [pull_request]
jobs:
  validate:
    runs-on: ubuntu-latest
    steps:
      - uses: actions/checkout@v2
      - name: Install OpenSpec
        run: npm install -g @fission-ai/openspec@1
      - name: Validate changes
        run: openspec change validate
```

### Integración con IDE

Editores recomendados con soporte Markdown para editar documentos OpenSpec:
- VS Code + plugins de Markdown
- JetBrains IDEs + soporte Markdown
- Vim/Neovim + plugins de Markdown

## Solución de problemas

### Problemas comunes

#### Problema: comando OpenSpec no encontrado

**Solución**:
1. Compruebe si OpenSpec está instalado: `npm list -g @fission-ai/openspec`
2. Verifique que la variable PATH incluya el directorio bin global de npm
3. Reinstale: `npm install -g @fission-ai/openspec@1`

#### Problema: No se pueden crear propuestas o changes

**Solución**:
1. Asegúrese de que el directorio actual sea la raíz del proyecto
2. Verifique si existe el directorio `openspec/`; si no, créelo
3. Compruebe los permisos de escritura

#### Problema: la validación falla

**Solución**:
1. Verifique que los documentos en el directorio del change estén completos
2. Asegúrese de que todas las tareas requeridas estén completas
3. Vea los errores detallados: `openspec change validate <change-name> --verbose`

#### Problema: el archivado falla

**Solución**:
1. Asegúrese de que el change haya pasado la validación
2. Verifique si existe el directorio `openspec/archived/`
3. Asegúrese de que no haya cambios de código sin commitear

## Uso avanzado

### Plantillas personalizadas

OpenSpec admite plantillas personalizadas para proposals y especificaciones:

```bash
# Crear directorio de plantillas en la raíz del proyecto
mkdir -p .openspec/templates

# Crear plantilla personalizada para proposals
cat > .openspec/templates/proposal.md << 'EOF'
# Proposal: {{title}}

## Problem Statement
{{problem}}

## Proposed Solution
{{solution}}

## Custom Section
Your custom content here
EOF
```

### Archivo de configuración

Cree un archivo de configuración `.openspec.json` en la raíz del proyecto:

```json
{
  "version": "1.0",
  "templates": {
    "proposal": ".openspec/templates/proposal.md",
    "spec": ".openspec/templates/spec.md"
  },
  "validation": {
    "requireTests": true,
    "requireDocs": true
  },
  "hooks": {
    "pre-archive": "npm test"
  }
}
```

### Hooks

OpenSpec admite la ejecución de scripts personalizados antes y después de operaciones específicas:

```json
{
  "hooks": {
    "pre-create": "scripts/check-permissions.sh",
    "post-archive": "scripts/notify-team.sh"
  }
}
```
