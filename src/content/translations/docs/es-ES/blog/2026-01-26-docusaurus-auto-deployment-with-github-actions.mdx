## Añadir soporte de despliegue automatizado a GitHub Pages para HagiCode

> El nombre en clave inicial de este proyecto era PCode, ahora renombrado oficialmente como HagiCode. Este artículo documenta cómo se incorporó la capacidad de despliegue automatizado de sitios estáticos al proyecto, haciendo que la publicación de contenidos sea tan sencilla como beber agua.

## Contexto/Introducción

Durante el desarrollo de HagiCode, nos encontramos con un problema muy práctico: a medida que aumentaban los documentos y las propuestas, cómo gestionar y mostrar eficientemente estos contenidos se convirtió en una prioridad. Decidimos utilizar GitHub Pages para alojar nuestro sitio estático, pero la construcción y el despliegue manuales eran demasiado tediosos — cada cambio requería una compilación y empaquetado local, y luego la subida manual a la rama `gh-pages`. Esto no solo era ineficiente, sino también propenso a errores.

Para resolver este problema (principalmente por pereza), necesitábamos un flujo de despliegue automatizado. Este artículo documenta en detalle cómo se añadió el soporte de despliegue automatizado con GitHub Actions al proyecto HagiCode, permitiéndonos concentrarnos únicamente en la creación de contenido y dejando el resto al proceso de automatización.

## Análisis de objetivos

Antes de actuar, debemos primero aclarar qué queremos lograr exactamente. Después de todo, afilar el hacha no estropea el corte.

### Requisitos principales

1.  **Construcción automatizada**: cuando se envíe código a la rama `main`, activar automáticamente el proceso de compilación.
2.  **Despliegue automático**: después de una compilación exitosa, desplegar automáticamente los archivos estáticos generados en GitHub Pages.
3.  **Consistencia del entorno**: asegurar que el entorno de CI y el entorno de compilación local sean consistentes, evitando la vergüenza de "funciona en local pero falla en producción".

### Selección técnica

Dado que HagiCode está construido con Docusaurus (un generador de sitios estáticos muy popular basado en React), podemos utilizar GitHub Actions para lograr este objetivo.

## Configuración del flujo de trabajo de GitHub Actions

GitHub Actions es el servicio de CI/CD proporcionado por GitHub. A través de la definición de archivos de flujo de trabajo en formato YAML en el repositorio de código, podemos personalizar varias tareas automatizadas.

### Crear el archivo de flujo de trabajo

Necesitamos crear un nuevo archivo de configuración en la carpeta `.github/workflows` del directorio raíz del proyecto, por ejemplo llamado `deploy.yml`. Si la carpeta no existe, recuerde crearla manualmente primero.

La lógica central de este archivo de configuración es la siguiente:

1.  **Condición de activación**: escuchar el evento `push` en la rama `main`.
2.  **Entorno de ejecución**: la última versión de Ubuntu.
3.  **Pasos de compilación**:
    -   Extraer el código
    -   Instalar Node.js
    -   Instalar dependencias (`npm install`)
    -   Construir los archivos estáticos (`npm run build`)
4.  **Pasos de despliegue**: utilizar el `action-gh-pages` oficial para enviar los artefactos de compilación a la rama `gh-pages`.

### Código de configuración clave

A continuación se muestra la plantilla de configuración que finalmente adoptamos:

```yaml
name: Deploy to GitHub Pages

# Condición de activación: al hacer push a la rama main
on:
  push:
    branches:
      - main
    # Se puede añadir filtrado de ruta según sea necesario, por ejemplo compilar solo cuando haya cambios en la documentación
    # paths:
    #   - 'docs/**'
    #   - 'package.json'

# Establecer permisos, esto es importante para el despliegue en GitHub Pages
permissions:
  contents: read
  pages: write
  id-token: write

# Control de concurrencia: cancelar compilaciones antiguas de la misma rama
concurrency:
  group: "pages"
  cancel-in-progress: false

jobs:
  build:
    runs-on: ubuntu-latest
    steps:
      - name: Checkout
        uses: actions/checkout@v4
        # Nota: es necesario establecer fetch-depth: 0, de lo contrario puede provocar que el número de versión de compilación sea inexacto
        with:
          fetch-depth: 0

      - name: Setup Node
        uses: actions/setup-node@v4
        with:
          node-version: 20 # Se recomienda mantener la consistencia con el entorno de desarrollo local
          cache: 'npm'     # Habilitar la caché puede acelerar el proceso de compilación

      - name: Install dependencies
        run: npm ci
        # Usar npm ci en lugar de npm install porque es más rápido y más estricto, adecuado para entornos de CI

      - name: Build website
        run: npm run build
        env:
          # Si la compilación de su sitio requiere variables de entorno, configúrelas aquí
          # NODE_ENV: production
          # PUBLIC_URL: /your-repo-name
          
      - name: Upload artifact
        uses: actions/upload-pages-artifact@v3
        with:
          path: ./build # Directorio de salida predeterminado de Docusaurus

  deploy:
    environment:
      name: github-pages
      url: ${{ steps.deployment.outputs.page_url }}
    runs-on: ubuntu-latest
    needs: build
    steps:
      - name: Deploy to GitHub Pages
        id: deployment
        uses: actions/deploy-pages@v4
```

## Problemas encontrados durante la implementación

En la práctica, nos encontramos con algunos problemas que compartimos aquí con la esperanza de que puedan evitarlos (o prepararse con antelación con las soluciones).

### 1. Problema de permisos del GitHub Token

Al principio de la configuración, el despliegue siempre fallaba con el error 403 (Forbidden). Después de buscar mucho, descubrimos que el `GITHUB_TOKEN` predeterminado de GitHub no tenía permisos de escritura en Pages.

**Solución**: en `Settings` -> `Actions` -> `General` -> `Workflow permissions` del repositorio, asegúrese de seleccionar **"Read and write permissions"**.

### 2. Ruta incorrecta del directorio de compilación

Docusaurus por defecto coloca los archivos estáticos compilados en el directorio `build`. Pero algunos proyectos (por ejemplo, Create React App usa `build` por defecto, Vite usa `dist` por defecto) pueden tener una configuración diferente. Si en Actions aparece un error de archivo no encontrado, recuerde verificar la configuración de la ruta de salida en `docusaurus.config.js`.

### 3. Problema de subruta

Si su repositorio no es una página de usuario (es decir, no es `username.github.io`), sino una página de proyecto (por ejemplo `username.github.io/project-name`), necesita configurar `baseUrl`.

En `docusaurus.config.js`:

```javascript
module.exports = {
  // ...
  url: 'https://hagicode.com', // Su URL de HagiCode
  baseUrl: '/', // Despliegue en la ruta raíz
  // ...
};
```

Este punto se pasa por alto fácilmente; una configuración incorrecta hará que la página se abra con una pantalla blanca porque las rutas de los recursos no se pueden cargar.

## Verificación de resultados

Después de configurar todo y enviar el código, podemos ir a la pestaña **Actions** del repositorio de GitHub a observar.

Verá un círculo amarillo (el flujo de trabajo está en ejecución), cuando se ponga verde significará que tuvo éxito. Si se pone rojo, haga clic para ver los registros; normalmente se puede diagnosticar el problema (la mayoría de las veces es un error ortográfico o una configuración de ruta incorrecta).

Después de una compilación exitosa, visite `https://<su-usuario>.github.io/<nombre-del-repositorio>/` para ver el sitio recién desplegado.

## Conclusión

A través de la incorporación de GitHub Actions, hemos logrado con éxito el despliegue automatizado del sitio de documentación de HagiCode. Esto no solo ahorra el tiempo de operación manual, sino que lo más importante es que garantiza la estandarización del proceso de publicación. Ahora, sin importar qué compañero actualice la documentación, siempre que se fusione en la rama `main`, en unos minutos se podrá ver el contenido más reciente en línea.

**Beneficios principales**:
- **Mejora de eficiencia**: de "empaquetado manual, subida manual" a "el código es la publicación".
- **Reducción de errores**: elimina la posibilidad de errores de operación humana.
- **Optimización de la experiencia**: permite a los desarrolladores concentrarse más en la calidad del contenido en lugar de verse perturbados por tediosos procesos de despliegue.

Aunque configurar CI/CD al principio es un poco complicado (especialmente con los diversos problemas de permisos y rutas), es una inversión única con rendimientos a largo plazo enormes. Recomendamos encarecidamente que todos los proyectos de sitios estáticos incorporen flujos de automatización similares.
