## Integración profunda y uso práctico de StreamJsonRpc en HagiCode

> Este artículo describe en detalle cómo el proyecto HagiCode (antes PCode) integró con éxito la biblioteca de comunicación StreamJsonRpc de Microsoft, para reemplazar la implementación personalizada original de JSON-RPC y resolver los desafíos técnicos y de arquitectura encontrados durante el proceso de integración.

## Contexto

StreamJsonRpc es una biblioteca de comunicación JSON-RPC mantenida oficialmente por Microsoft para .NET y TypeScript, conocida por su potente seguridad de tipos, generación automática de proxies y mecanismo maduro de manejo de excepciones. En el proyecto HagiCode, para comunicarse con herramientas de IA externas (como iflow CLI, OpenCode CLI) a través de ACP (Agent Communication Protocol), y para eliminar los costos de mantenimiento y los posibles errores de la implementación personalizada temprana de JSON-RPC, el proyecto decidió integrar StreamJsonRpc. Sin embargo, durante la integración se encontraron desafíos específicos del JSON-RPC streaming, especialmente en el manejo de la vinculación de objetivos de proxy y la identificación de parámetros genéricos.

Para resolver estos problemas, tomamos una decisión audaz: reconstruir todo el sistema de compilación desde cero. El cambio que trajo esta decisión pudo ser mayor de lo que imaginará — lo detallaré más adelante.

## Análisis

El proyecto se encuentra actualmente en una fase clave de integración del protocolo ACP, enfrentando los siguientes desafíos técnicos y de arquitectura:

### 1. Limitaciones de la implementación personalizada
La implementación original de JSON-RPC se encuentra en `src/HagiCode.ClaudeHelper/AcpImp/`, incluyendo componentes como `JsonRpcEndpoint` y `ClientSideConnection`. Mantener este código personalizado tiene un alto costo y carece de las funciones avanzadas de una biblioteca madura (como reporte de progreso, soporte de cancelación).

### 2. Obstáculos para la integración de StreamJsonRpc
Al intentar migrar el patrón existente de `CallbackProxyTarget` a StreamJsonRpc, se descubrió que el método `_rpc.AddLocalRpcTarget(target)` no podía reconocer los objetivos creados mediante el patrón de proxy. Específicamente, StreamJsonRpc no podía dividir automáticamente las propiedades del tipo genérico `T` en parámetros de métodos RPC, lo que impedía que el servidor manejara correctamente las invocaciones de métodos iniciadas por el cliente.

### 3. Confusión en la estratificación de la arquitectura
El `ClientSideConnection` existente mezclaba la capa de transporte (WebSocket/Stdio), la capa de protocolo (JSON-RPC) y la capa de negocio (interfaz ACP Agent), lo que resultaba en responsabilidades poco claras y problemas de vinculación de métodos faltantes en `AcpAgentCallbackRpcAdapter`.

### 4. Ausencia de registros
La capa de transporte WebSocket carecía de salida de registros del contenido JSON original, lo que dificultaba diagnosticar si los problemas de comunicación RPC eran de serialización o de red.

## Solución

Para los problemas anteriores, adoptamos la siguiente solución sistemática, optimizando desde tres dimensiones: refactorización de arquitectura, integración de bibliotecas y mejora de la depuración:

### 1. Migración completa a StreamJsonRpc

#### Eliminación del código antiguo
Se eliminaron `JsonRpcEndpoint.cs`, `AgentSideConnection.cs` y los convertidores de serialización personalizados relacionados (`JsonRpcMessageJsonConverter`, etc.).

#### Integración de la biblioteca oficial
Se incorporó el paquete NuGet `StreamJsonRpc`, utilizando su clase `JsonRpc` para manejar la lógica de comunicación central.

#### Abstracción de la capa de transporte
Se definió la interfaz `IAcpTransport` para unificar el manejo de los dos modos de transporte: `WebSocket` y `Stdio`, asegurando el desacoplamiento entre la capa de protocolo y la capa de transporte.

```csharp
// Definición de la interfaz IAcpTransport
public interface IAcpTransport
{
    Task SendAsync(string message, CancellationToken cancellationToken = default);
    Task<string> ReceiveAsync(CancellationToken cancellationToken = default);
    Task CloseAsync(CancellationToken cancellationToken = default);
}

// Implementación de transporte WebSocket
public class WebSocketTransport : IAcpTransport
{
    private readonly WebSocket _webSocket;

    public WebSocketTransport(WebSocket webSocket)
    {
        _webSocket = webSocket;
    }

    // Implementación de métodos de envío y recepción
    // ...
}

// Implementación de transporte Stdio
public class StdioTransport : IAcpTransport
{
    private readonly StreamReader _reader;
    private readonly StreamWriter _writer;

    public StdioTransport(StreamReader reader, StreamWriter writer)
    {
        _reader = reader;
        _writer = writer;
    }

    // Implementación de métodos de envío y recepción
    // ...
}
```

### 2. Reparación del problema de reconocimiento de objetivos de proxy

#### Análisis de `CallbackProxyTarget`
Se verificó la lógica existente de generación de proxies dinámicos, determinando la causa raíz por la que StreamJsonRpc no podía reconocerlos (generalmente porque el objeto proxy no exponía las firmas de métodos reales, o utilizaba tipos de parámetros no soportados por StreamJsonRpc).

#### Refactorización del paso de parámetros
Se dividieron las propiedades genéricas en parámetros de métodos RPC explícitos. En lugar de depender de propiedades dinámicas, se definieron DTOs (Objetos de Transferencia de Datos) específicos de Request/Response, asegurando que StreamJsonRpc pudiera identificar correctamente las firmas de métodos mediante reflexión.

```csharp
// Forma original con propiedades genéricas
public class CallbackProxyTarget<T>
{
    public Func<T, Task> Callback { get; set; }
}

// Forma refactorizada con métodos específicos
public class ReadTextFileRequest
{
    public string FilePath { get; set; }
}

public class ReadTextFileResponse
{
    public string Content { get; set; }
}

public interface IAcpAgentCallback
{
    Task<ReadTextFileResponse> ReadTextFileAsync(ReadTextFileRequest request);
    // Otros métodos...
}
```

#### Uso de `Attach` en lugar de `AddLocalRpcTarget`
En algunos escenarios complejos, hacer proxy manualmente del objeto `JsonRpc` y manejar `RpcConnection` puede ser más flexible que añadir directamente el objetivo.

### 3. Implementación de vinculación de métodos y mejora de registros

#### Implementación de `AcpAgentCallbackRpcAdapter`
Se aseguró de que este componente implemente explícitamente la interfaz de proxy de StreamJsonRpc, mapeando los métodos definidos por el protocolo ACP (como `ReadTextFileAsync`) a los manejadores de callback de StreamJsonRpc.

#### Integración de registro
En la canalización de procesamiento de mensajes de WebSocket o Stdio, se interceptaron y registraron los textos originales de solicitudes y respuestas JSON-RPC. Se utilizó `ILogger` para generar el payload original antes del análisis y después de la serialización, facilitando la identificación de errores de formato.

```csharp
// Contenedor de transporte con registro mejorado
public class LoggingAcpTransport : IAcpTransport
{
    private readonly IAcpTransport _innerTransport;
    private readonly ILogger<LoggingAcpTransport> _logger;

    public LoggingAcpTransport(IAcpTransport innerTransport, ILogger<LoggingAcpTransport> logger)
    {
        _innerTransport = innerTransport;
        _logger = logger;
    }

    public async Task SendAsync(string message, CancellationToken cancellationToken = default)
    {
        _logger.LogTrace("Sending message: {Message}", message);
        await _innerTransport.SendAsync(message, cancellationToken);
    }

    public async Task<string> ReceiveAsync(CancellationToken cancellationToken = default)
    {
        var message = await _innerTransport.ReceiveAsync(cancellationToken);
        _logger.LogTrace("Received message: {Message}", message);
        return message;
    }

    public async Task CloseAsync(CancellationToken cancellationToken = default)
    {
        _logger.LogDebug("Closing connection");
        await _innerTransport.CloseAsync(cancellationToken);
    }
}
```

### 4. Refactorización de la estratificación de la arquitectura

#### Capa de transporte (`AcpRpcClient`)
Encapsula la conexión de StreamJsonRpc, responsable de `InvokeAsync` y la gestión del ciclo de vida de la conexión.

```csharp
public class AcpRpcClient : IDisposable
{
    private readonly JsonRpc _rpc;
    private readonly IAcpTransport _transport;

    public AcpRpcClient(IAcpTransport transport)
    {
        _transport = transport;
        _rpc = new JsonRpc(new StreamRpcTransport(transport));
        _rpc.StartListening();
    }

    public async Task<TResponse> InvokeAsync<TResponse>(string methodName, object parameters)
    {
        return await _rpc.InvokeAsync<TResponse>(methodName, parameters);
    }

    public void Dispose()
    {
        _rpc.Dispose();
        _transport.Dispose();
    }

    // StreamRpcTransport es un adaptador de StreamJsonRpc para IAcpTransport
    private class StreamRpcTransport : IDuplexPipe
    {
        // Implementación de la interfaz IDuplexPipe
        // ...
    }
}
```

#### Capa de protocolo (`IAcpAgentClient` / `IAcpAgentCallback`)
Define interfaces claras de cliente-a-agente y agente-a-cliente, eliminando el patrón de fábrica con dependencia circular como `Func<IAcpAgent, IAcpClient>`, y cambiando a inyección de dependencias o registro directo de callbacks.

## Práctica

Basándonos en las mejores prácticas de StreamJsonRpc y la experiencia del proyecto, a continuación se presentan las recomendaciones clave durante la implementación:

### 1. DTOs fuertemente tipados优于 objetos dinámicos
La ventaja central de StreamJsonRpc radica en el tipado fuerte. No utilice `dynamic` ni `JObject` para pasar parámetros. Debe definir clases POCO de C# específicas para cada método RPC como parámetros. Esto no solo resuelve el problema de reconocimiento de objetivos de proxy, sino que también permite detectar errores de tipos en tiempo de compilación.

Ejemplo: reemplazar las propiedades genéricas en `CallbackProxyTarget` por clases específicas como `ReadTextFileRequest` y `WriteTextFileRequest`.

### 2. Declaración explícita de nombres de métodos
Utilice el atributo `[JsonRpcMethod]` para especificar explícitamente el nombre del método RPC, sin depender del mapeo predeterminado de nombres de métodos. Esto puede prevenir fallos en las invocaciones debido a diferencias en las convenciones de nomenclatura (como PascalCase vs. camelCase).

```csharp
public interface IAcpAgentCallback
{
    [JsonRpcMethod("readTextFile")]
    Task<ReadTextFileResponse> ReadTextFileAsync(ReadTextFileRequest request);
    
    [JsonRpcMethod("writeTextFile")]
    Task WriteTextFileAsync(WriteTextFileRequest request);
}
```

### 3. Aprovechar los callbacks de estado de conexión
StreamJsonRpc proporciona el evento `JsonRpc.ConnectionLost`. Es imprescindible escuchar este evento para manejar salidas inesperadas de procesos o desconexiones de red, lo cual es más oportuno que depender únicamente de la detección de失效 de Orleans Grain.

```csharp
_rpc.ConnectionLost += (sender, e) =>
{
    _logger.LogError("RPC connection lost: {Reason}", e.ToString());
    // Manejar la lógica de reconexión o notificar al usuario
};
```

### 4. Registro por niveles
- **Nivel Trace**: registrar el texto completo de la solicitud/respuesta JSON.
- **Nivel Debug**: registrar la pila de llamadas a métodos y el resumen de parámetros.
- **Nota**: asegurar que los registros no contengan Authorization Tokens sensibles ni codificaciones Base64 de contenido de archivos grandes.

### 5. Manejo de las particularidades de la transmisión streaming
StreamJsonRpc soporta nativamente `IAsyncEnumerable`. Al implementar respuestas de Prompt streaming en ACP, se debe usar directamente `IAsyncEnumerable` en lugar de lógica personalizada de paginación. Esto simplifica enormemente la cantidad de código para el manejo de streaming.

```csharp
public interface IAcpAgentCallback
{
    [JsonRpcMethod("streamText")]
    IAsyncEnumerable<string> StreamTextAsync(StreamTextRequest request);
}
```

### 6. Patrón adaptador (Adapter Pattern)
Mantener la separación entre `ACPSession` y `ClientSideConnection`. `ACPSession` debe enfocarse en la gestión de estado de Orleans y la lógica de negocio (como la puesta en cola de mensajes), utilizando el objeto de conexión StreamJsonRpc a través de composición en lugar de herencia.

## Conclusión

A través de la integración completa de StreamJsonRpc, el proyecto HagiCode resolvió con éxito los problemas de alto costo de mantenimiento, limitaciones funcionales y confusión en la estratificación de la arquitectura de la implementación personalizada original. Las mejoras clave incluyen:

1. Adopción de DTOs fuertemente tipados en lugar de propiedades dinámicas, mejorando la mantenibilidad y fiabilidad del código
2. Implementación de la abstracción de la capa de transporte y la separación de la capa de protocolo, mejorando la claridad de la arquitectura
3. Mejora de las funciones de registro, facilitando la resolución de problemas de comunicación
4. Introducción de soporte para transmisión streaming, simplificando la implementación del manejo de streams

Estas mejoras proporcionan a HagiCode una base de comunicación más estable y eficiente, permitiéndole interactuar mejor con herramientas de IA externas, y sentando una base sólida para la expansión funcional futura.
