## Construir una integración de GitHub Issues desde cero: la práctica de conexión directa desde el frontend en HagiCode

> Este artículo documenta el proceso completo de integración de GitHub Issues en la plataforma HagiCode. Exploraremos cómo utilizar una arquitectura de "conexión directa desde el frontend + backend mínimo" para lograr una autenticación OAuth segura y una sincronización eficiente de Issues, manteniendo el backend ligero.

## Contexto: por qué integrar GitHub

Como plataforma de desarrollo asistido por IA, el valor central de HagiCode radica en conectar ideas con implementación. Sin embargo, en el uso real, descubrimos que después de que los usuarios completan un Proposal (propuesta) en HagiCode, a menudo necesitan copiar manualmente el contenido a GitHub Issues para el seguimiento del proyecto.

Esto genera varios problemas evidentes:

1.  **Flujo de trabajo fragmentado**: los usuarios necesitan alternar entre dos sistemas. La experiencia no es fluida y es fácil perder información clave durante el proceso de copiar y pegar.
2.  **Colaboración inconveniente**: otros miembros del equipo están acostumbrados a consultar tareas en GitHub y no pueden ver directamente el progreso de las propuestas dentro de HagiCode.
3.  **Trabajo repetitivo**: cada vez que se actualiza una propuesta, alguien debe actualizar manualmente el Issue correspondiente en GitHub, añadiendo un costo de mantenimiento innecesario.

Para resolver este problema, decidimos introducir la función de **GitHub Issues Integration**, conectando las sesiones de HagiCode con los repositorios de GitHub para habilitar la "sincronización con un clic".

## Selección técnica: conexión directa desde el frontend vs. proxy del backend

Al diseñar la solución de integración, teníamos dos caminos frente a nosotros: el tradicional "modelo de proxy del backend" y el más audaz "modelo de conexión directa desde el frontend".

### Comparación de soluciones

En el tradicional **modelo de proxy del backend**, todas las solicitudes del frontend deben pasar primero por nuestro backend, que luego llama a la API de GitHub. Esto centraliza la lógica, pero también impone una carga significativa al backend:

1.  **Backend inflado**: sería necesario escribir un contenedor dedicado de cliente de API de GitHub y también manejar la compleja máquina de estados de OAuth.
2.  **Riesgo con el Token**: el Token de GitHub del usuario tendría que almacenarse en la base de datos del backend. Aunque se pueda cifrar, esto sigue aumentando la superficie de ataque de seguridad.
3.  **Costo de desarrollo**: se necesitarían migraciones de base de datos para almacenar Tokens y un servicio de sincronización adicional para mantener.

El **modelo de conexión directa desde el frontend** es mucho más ligero. En este enfoque, utilizamos el backend solo para el paso más sensible de "intercambio de secretos" (el callback de OAuth). Una vez obtenido el Token, lo almacenamos directamente en el `localStorage` del navegador. Las operaciones posteriores como crear Issues y actualizar comentarios se envían directamente desde el frontend a GitHub a través de HTTP.

| Dimensión de comparación | Modelo de proxy del backend | Modelo de conexión directa desde el frontend |
| :--- | :--- | :--- |
| **Complejidad del backend** | Requiere un servicio OAuth completo y cliente de API de GitHub | Solo necesita un endpoint de callback de OAuth |
| **Gestión de Tokens** | Debe cifrarse y almacenarse en la base de datos, con riesgo de filtración | Se almacena en el navegador, visible solo para el usuario |
| **Costo de implementación** | Requiere migraciones de base de datos y desarrollo de múltiples servicios | Principalmente trabajo del frontend |
| **Experiencia del usuario** | Lógica centralizada, pero la latencia del servidor puede ser ligeramente mayor | Respuesta extremadamente rápida, interacción directa con GitHub |

Dado que buscábamos una integración rápida y cambios mínimos en el backend, **finalmente adoptamos el "modelo de conexión directa desde el frontend"**. Es como darle al navegador un "pase temporal"; una vez que lo obtiene, el navegador puede ir a GitHub a hacer sus cosas por sí mismo, sin necesidad de pedir aprobación al administrador del backend cada vez.

## Diseño central: flujo de datos y seguridad

Una vez definida la arquitectura, necesitamos diseñar el flujo de datos específico. El núcleo del proceso de sincronización radica en cómo obtener el Token de forma segura y utilizarlo de manera eficiente.

### Diagrama de arquitectura general

Todo el sistema puede abstraerse en tres roles: el navegador (frontend), el backend de HagiCode y GitHub.

```text
+--------------+        +--------------+        +--------------+
| Frontend     |        |   Backend    |        |   GitHub     |
| React        |        |   ASP.NET    |        |   REST API   |
|              |        |              |        |              |
|  +--------+  |        |              |        |              |
|  | OAuth  |--+--------> /callback    |        |              |
|  | Flujo  |  |        |              |        |              |
|  +--------+  |        |              |        |              |
|              |        |              |        |              |
|  +--------+  |        |  +--------+  |        |  +--------+  |
|  | GitHub |  +------------> Session |  +----------> Issues |  |
|  | API    |  |        |  |Metadata|  |        |  |        |  |
|  | Directo|  |        |  +--------+  |        |  +--------+  |
|  +--------+  |        |              |        |              |
+--------------+        +--------------+        +--------------+
```

**El punto clave es**: solo un pequeño paso de OAuth (intercambiar el code por el token) necesita pasar por el backend. Después de eso, el trabajo pesado (crear Issues) se maneja directamente entre el frontend y GitHub.

### Flujo de datos de sincronización en detalle

Cuando el usuario hace clic en el botón "Sync to GitHub" en la interfaz de HagiCode, se desencadena una serie de acciones complejas:

```text
El usuario hace clic en "Sync to GitHub"
         │
         ▼
1. El frontend verifica localStorage para obtener el Token de GitHub
         │
         ▼
2. Formatear el contenido del Issue (convertir el Proposal a Markdown)
         │
         ▼
3. El frontend llama directamente a la API de GitHub para crear/actualizar el Issue
         │
         ▼
4. Llamar a la API del backend de HagiCode para actualizar Session.metadata (almacenar la URL del Issue y otra información)
         │
         ▼
5. El backend difunde el evento SessionUpdated a través de SignalR
         │
         ▼
6. El frontend recibe el evento y actualiza la interfaz para mostrar el estado "Sincronizado"
```

### Diseño de seguridad

La seguridad siempre es la máxima prioridad al integrar servicios de terceros. Hicimos las siguientes consideraciones:

1.  **Protección contra ataques CSRF**: generar un parámetro `state` aleatorio durante la redirección de OAuth y almacenarlo en `sessionStorage`. Validar estrictamente el state en el callback para evitar solicitudes falsificadas.
2.  **Aislamiento del almacenamiento de Tokens**: el Token se almacena únicamente en el `localStorage` del navegador. Utilizando la política del mismo origen (Same-Origin Policy), solo los scripts de HagiCode pueden leerlo, evitando que una filtración de la base de datos del servidor afecte a los usuarios.
3.  **Límites de error**: diseñamos un manejo dedicado para los errores comunes de la API de GitHub (como 401 Token expirado, 422 fallo de validación y 429 límite de tasa), para que los usuarios reciban retroalimentación amigable.

## En la práctica: detalles de implementación del código

La teoría tiene sus límites; veamos cómo funciona realmente el código.

### 1. Cambios mínimos en el backend

El backend solo necesita hacer dos cosas: almacenar información de sincronización y manejar el callback de OAuth.

### 2. Flujo de OAuth del frontend

Este es el punto de entrada de la conexión. Utilizamos el flujo estándar de Authorization Code Flow.

```typescript
// src/HagiCode.Client/src/services/githubOAuth.ts

// Generar la URL de autorización y redirigir
export async function generateAuthUrl(): Promise<string> {
  const state = generateRandomString(); // Generar cadena aleatoria para protección CSRF
  sessionStorage.setItem('hagicode_github_state', state);
  
  const params = new URLSearchParams({
    client_id: clientId,
    redirect_uri: window.location.origin + '/settings?tab=github&oauth=callback',
    scope: ['repo', 'public_repo'].join(' '),
    state: state,
  });
  
  return `https://github.com/login/oauth/authorize?${params.toString()}`;
}

// Manejar el intercambio de code por token en la página de callback
export async function exchangeCodeForToken(code: string, state: string): Promise<GitHubToken> {
  // 1. Validar el state para prevenir CSRF
  const savedState = sessionStorage.getItem('hagicode_github_state');
  if (state !== savedState) throw new Error('Invalid state parameter');

  // 2. Llamar a la API del backend para intercambiar el token
  // Nota: esto debe pasar por el backend porque no se puede exponer el ClientSecret al frontend
  const response = await fetch('/api/GitHubOAuth/callback', {
    method: 'POST',
    headers: { 'Content-Type': 'application/json' },
    body: JSON.stringify({ code, state, redirectUri: window.location.origin + '/settings?tab=github&oauth=callback' }),
  });

  if (!response.ok) throw new Error('Failed to exchange token');
  
  const token = await response.json();
  
  // 3. Guardar en LocalStorage
  saveToken(token);
  return token;
}
```

### 3. Contenedor del cliente de API de GitHub

Una vez que tenemos el Token, necesitamos una herramienta robusta para llamar a la API de GitHub.

```typescript
// src/HagiCode.Client/src/services/githubApiClient.ts

const GITHUB_API_BASE = 'https://api.github.com';

// Contenedor principal de solicitudes
async function githubApi<T>(endpoint: string, options: RequestInit = {}): Promise<T> {
  const token = localStorage.getItem('gh_token');
  if (!token) throw new Error('Not connected to GitHub');
  
  const response = await fetch(`${GITHUB_API_BASE}${endpoint}`, {
    ...options,
    headers: {
      ...options.headers,
      Authorization: `Bearer ${token}`,
      Accept: 'application/vnd.github.v3+json', // Especificar la versión de la API
    },
  });
  
  // Lógica de manejo de errores
  if (!response.ok) {
    if (response.status === 401) throw new Error('El Token de GitHub ha expirado, por favor vuelva a conectar');
    if (response.status === 403) throw new Error('Sin permiso para acceder a este repositorio o se excedió el límite de tasa');
    if (response.status === 422) throw new Error('La validación del Issue falló, el título puede estar duplicado');
    throw new Error(`GitHub API Error: ${response.statusText}`);
  }
  
  return response.json();
}

// Crear Issue
export async function createIssue(owner: string, repo: string, data: { title: string, body: string, labels: string[] }) {
  return githubApi(`/repos/${owner}/${repo}/issues`, {
    method: 'POST',
    body: JSON.stringify(data),
  });
}
```

### 4. Formateo de contenido y sincronización

El último paso es convertir los datos de la sesión de HagiCode al formato de un Issue de GitHub. Es un poco como un trabajo de "traducción".

```typescript
// Convertir un objeto Session en una cadena Markdown
function formatIssueForSession(session: Session): string {
  let content = `# ${session.title}\n\n`;
  content += `**> HagiCode Session:** #${session.code}\n`;
  content += `**> Status:** ${session.status}\n\n`;
  content += `## Description\n\n${session.description || 'No description provided.'}\n\n`;
  
  // Si es de tipo Proposal, añadir campos adicionales
  if (session.type === 'proposal') {
    content += `## Chief Complaint\n\n${session.chiefComplaint || ''}\n\n`;
    // Añadir un enlace profundo para que los usuarios puedan volver de GitHub a HagiCode
    content += `---\n\n**[View in HagiCode](hagicode://sessions/${session.id})**\n`;
  }
  
  return content;
}

// Lógica principal al hacer clic en el botón de sincronización
const handleSync = async (session: Session) => {
  try {
    const repoInfo = parseRepositoryFromUrl(session.repoUrl); // Analizar la URL del repositorio
    if (!repoInfo) throw new Error('Invalid repository URL');

    toast.loading('Sincronizando con GitHub...');
    
    // 1. Formatear contenido
    const issueBody = formatIssueForSession(session);
    
    // 2. Llamar a la API
    const issue = await githubApiClient.createIssue(repoInfo.owner, repoInfo.repo, {
      title: `[HagiCode] ${session.title}`,
      body: issueBody,
      labels: ['hagicode', 'proposal', `status:${session.status}`],
    });
    
    // 3. Actualizar Session Metadata (guardar el enlace del Issue)
    await SessionsService.patchApiSessionsSessionId(session.id, {
      metadata: {
        githubIssue: {
          owner: repoInfo.owner,
          repo: repoInfo.repo,
          issueNumber: issue.number,
          issueUrl: issue.html_url,
          syncedAt: new Date().toISOString(),
        }
      }
    });

    toast.success('Sincronizado correctamente');
  } catch (err) {
    console.error(err);
    toast.error('Error en la sincronización, por favor verifique el Token o la red');
  }
};
```

## Resumen y perspectivas

Con este enfoque de "conexión directa desde el frontend", logramos una integración sin fricciones de GitHub Issues con la menor cantidad posible de código en el backend.

### Logros principales
1.  **Alta eficiencia de desarrollo**: los cambios en el backend son mínimos, principalmente un campo adicional en la base de datos y un simple endpoint de callback de OAuth. La mayor parte de la lógica se completa en el frontend.
2.  **Buena seguridad**: el Token no pasa por la base de datos del servidor, reduciendo el riesgo de filtración.
3.  **Excelente experiencia del usuario**: las solicitudes se inician directamente desde el frontend, por lo que la velocidad de respuesta es rápida y no se necesita el reenvío a través del backend.

### Aspectos a tener en cuenta
Hay algunos obstáculos a considerar durante la implementación real:
- **Configuración de OAuth App**: recuerde ingresar la `Authorization callback URL` correcta en la configuración de su GitHub OAuth App (generalmente `http://localhost:3000/settings?tab=github&oauth=callback`).
- **Límites de tasa**: la API de GitHub limita bastante estrictamente las solicitudes no autenticadas, pero con un Token la cuota suele ser suficiente (5000 solicitudes/hora).
- **Análisis de URL**: los usuarios ingresan todo tipo de URLs de repositorio, así que asegúrese de que su expresión regular pueda coincidir con sufijos `.git`, formatos SSH y casos similares.

### Mejoras futuras
La función actual sigue siendo de sincronización unidireccional (HagiCode -> GitHub). En el futuro, planeamos implementar la sincronización bidireccional a través de GitHub Webhooks. Por ejemplo, si un Issue se cierra en GitHub, el estado de la sesión en el lado de HagiCode también podría actualizarse automáticamente. Esto requeriría que expongamos un endpoint de webhook en el backend, lo cual será un interesante siguiente paso.
