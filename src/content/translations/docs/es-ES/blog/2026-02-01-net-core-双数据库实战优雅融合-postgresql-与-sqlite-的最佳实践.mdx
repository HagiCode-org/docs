## .NET Core con doble base de datos en práctica: que PostgreSQL y SQLite coexistan pacíficamente

> Al construir aplicaciones modernas, a menudo nos enfrentamos a esta decisión: el entorno de desarrollo desea ser ligero y conveniente, mientras que el entorno de producción necesita alta concurrencia y alta disponibilidad. Este artículo compartirá cómo soportar elegantemente PostgreSQL y SQLite simultáneamente en un proyecto .NET Core, implementando las mejores prácticas de "SQLite para desarrollo, PostgreSQL para producción".


## Contexto

En el desarrollo de software, la diferenciación de entornos siempre ha sido uno de los problemas que acosan a los equipos de desarrollo. Tomando como ejemplo la plataforma **HagiCode** que estamos construyendo, se trata de un sistema de desarrollo asistido por IA basado en ASP.NET Core 10 y React, que integra Orleans internamente para la gestión de estado distribuido, con un stack tecnológico bastante moderno y complejo.

En las fases tempranas del proyecto, nos encontramos con un problema de ingeniería típico: los desarrolladores deseaban que el entorno local fuera "listo para usar", sin tener que instalar y configurar una pesada base de datos PostgreSQL; pero en el entorno de producción, necesitábamos manejar escrituras concurrentes de alto volumen y consultas JSON complejas, donde la ligera SQLite se quedaba corta.

Cómo mantener la unidad del código base permitiendo que la aplicación aproveche la portabilidad de SQLite como un software de cliente, y al mismo tiempo explotar el formidable rendimiento de PostgreSQL como un servicio empresarial? Este es el problema central que este artículo abordará.

## Contenido central 1: diseño de arquitectura y abstracción unificada

Para implementar soporte de doble base de datos en .NET Core, la idea central es "depender de abstracciones, no de implementaciones concretas". Necesitamos separar la elección de la base de datos del código de negocio y dejar que la capa de configuración decida.

### Enfoque de diseño

1.  **Interfaz unificada**: toda la lógica de negocio debe depender de la clase base `DbContext` o de interfaces personalizadas, no de un `PostgreSqlDbContext` específico.
2.  **Impulsado por configuración**: a través de los elementos de configuración en `appsettings.json`, decidir dinámicamente qué proveedor de base de datos cargar al iniciar la aplicación.
3.  **Aislamiento de características**: adaptar las funcionalidades específicas de PostgreSQL (como JSONB) para asegurar que también puedan funcionar con degradación en SQLite.

### Implementación de código: configuración dinámica del contexto

En `Program.cs` de ASP.NET Core, no deberíamos codificar `UseNpgsql` o `UseSqlite`. En su lugar, deberíamos leer la configuración para decidir dinámicamente.

Primero, definir la clase de configuración:

```csharp
public class DatabaseSettings
{
    public const string SectionName = "Database";
    
    // Tipo de base de datos: PostgreSQL o SQLite
    public string DbType { get; set; } = "PostgreSQL"; 
    
    // Cadena de conexión
    public string ConnectionString { get; set; } = string.Empty;
}
```

Luego, en `Program.cs` registrar el servicio según la configuración:

```csharp
// Leer configuración
var databaseSettings = builder.Configuration.GetSection(DatabaseSettings.SectionName).Get<DatabaseSettings>();

// Registrar DbContext
builder.Services.AddDbContext<ApplicationDbContext>(options =>
{
    if (databaseSettings?.DbType?.ToLower() == "sqlite")
    {
        // Configuración de SQLite
        options.UseSqlite(databaseSettings.ConnectionString);
        
        // Manejo de las limitaciones de escritura concurrente de SQLite
        // Nota: en producción se recomienda habilitar el modo WAL para mejorar el rendimiento concurrente
    }
    else
    {
        // Configuración de PostgreSQL (predeterminada)
        options.UseNpgsql(databaseSettings.ConnectionString, npgsqlOptions =>
        {
            // Habilitar soporte JSONB, muy útil al manejar registros de conversaciones con IA
            npgsqlOptions.UseJsonNet(); 
        });
        
        // Configurar estrategia de reconexión del pool de conexiones
        options.EnableRetryOnFailure(3);
    }
});
```

## Contenido central 2: manejo de diferencias y estrategia de migraciones

Aunque PostgreSQL y SQLite soportan el estándar SQL, existen diferencias significativas en características y comportamientos específicos. Si no se manejan bien estas diferencias, es muy probable que aparezca la vergonzosa situación de "funciona en local pero falla en producción".

### 1. Manejo del tipo JSON

En HagiCode, necesitamos almacenar una gran cantidad de prompts y metadatos de IA, lo que generalmente implica columnas JSON.
- **PostgreSQL**: tiene el tipo nativo `JSONB` con un rendimiento de consulta excelente.
- **SQLite**: no tiene un tipo JSON nativo (las versiones nuevas tienen la extensión JSON1, pero todavía hay diferencias en el mapeo de objetos), generalmente se almacena como TEXT.

**Solución**:
En el mapeo de entidades de EF Core, lo configuramos como un tipo convertible.

```csharp
protected override void OnModelCreating(ModelBuilder modelBuilder)
{
    base.OnModelCreating(modelBuilder);

    // Configurar entidad
    modelBuilder.Entity<PromptTemplate>(entity => 
    {
        entity.Property(e => e.Metadata)
              .HasColumnType("jsonb") // PG usa jsonb
              .HasConversion(
                  v => JsonSerializer.Serialize(v, (JsonSerializerOptions)null),
                  v => JsonSerializer.Deserialize<Dictionary<string, object>>(v, (JsonSerializerOptions)null)
              ); 
    });
}
```

Cuando se usa SQLite, aunque `HasColumnType("jsonb")` sea ignorado o produzca una advertencia, gracias a la configuración de `HasConversion`, los datos se serializarán y deserializarán correctamente como cadenas en un campo TEXT, garantizando así la compatibilidad.

### 2. Separación de estrategias de migración

Nunca intente que el mismo conjunto de scripts de Migration sirva tanto para PG como para SQLite. Debido a las diferencias en estrategias de generación de claves primarias, sintaxis de índices, etc., esto inevitablemente provocará fallos.

**Práctica recomendada**:
Mantener dos ramas o proyectos de migración. En el flujo de desarrollo de HagiCode, lo manejamos así:

1.  **Fase de desarrollo**: se trabaja principalmente con SQLite. Se usa `Add-Migration Init_Sqlite -OutputDir Migrations/Sqlite`.
2.  **Fase de adaptación**: después de completar una funcionalidad, se cambia la cadena de conexión para apuntar a PostgreSQL y se ejecuta `Add-Migration Init_Postgres -OutputDir Migrations/Postgres`.
3.  **Script de automatización**: escribir un script simple de PowerShell o Bash que aplique automáticamente la migración correspondiente según las variables de entorno actuales.

```bash
# Lógica de despliegue simple en pseudocódigo
if [ "$DATABASE_PROVIDER" = "PostgreSQL" ]; then
    dotnet ef database update --project Migrations.Postgres
else
    dotnet ef database update --project Migrations.Sqlite
fi
```

## Contenido central 3: lecciones aprendidas en la práctica de HagiCode

En el proceso de refactorizar **HagiCode** de una única base de datos a soporte de doble base de datos, encontramos algunos obstáculos y resumimos algunas experiencias clave, con la esperanza de ayudarles a evitarlos.

### 1. Diferencias en concurrencia y transacciones

PostgreSQL es de arquitectura cliente-servidor, soporta escrituras de alta concurrencia y tiene niveles de aislamiento de transacciones muy potentes. SQLite usa un mecanismo de bloqueo de archivos, donde las operaciones de escritura bloquean todo el archivo de la base de datos (a menos que se habilite el modo WAL).

**Recomendación**:
Al escribir lógica de negocio que involucra escrituras frecuentes (por ejemplo, guardar en tiempo real el estado de edición del usuario), es imprescindible considerar el mecanismo de bloqueo de SQLite. Al diseñar el módulo de colaboración OpenSpec de **HagiCode**, introdujimos un mecanismo de "combinar antes de escribir" para reducir la frecuencia de escrituras directas a la base de datos, manteniendo así un alto rendimiento en ambos tipos de bases de datos.

### 2. Gestión del ciclo de vida de las cadenas de conexión

El establecimiento de conexiones de PostgreSQL tiene un costo alto y depende del pool de conexiones. Las conexiones de SQLite son muy ligeras, pero si no se liberan a tiempo, el bloqueo de archivos puede causar que las operaciones posteriores agoten el tiempo de espera.

En `Program.cs`, podemos hacer ajustes específicos según la base de datos:

```csharp
if (databaseSettings?.DbType?.ToLower() == "sqlite")
{
    // SQLite: mantener la conexión abierta mejora el rendimiento, pero hay que cuidar el bloqueo de archivos
    options.UseSqlite(connectionString, sqliteOptions =>
    {
        // Establecer tiempo de espera de comandos
        sqliteOptions.CommandTimeout(30);
    });
}
else
{
    // PG: aprovechar el pool de conexiones
    options.UseNpgsql(connectionString, npgsqlOptions =>
    {
        npgsqlOptions.MaxBatchSize(100);
        npgsqlOptions.CommandTimeout(30);
    });
}
```

### 3. Importancia de la cobertura de pruebas

Muchos desarrolladores (incluidos los primeros miembros de nuestro equipo) cometen fácilmente un error: ejecutar pruebas unitarias solo en el entorno de desarrollo (generalmente SQLite).

En la pipeline de CI/CD de HagiCode, añadimos forzosamente un paso de GitHub Action para asegurar que cada Pull Request pase las pruebas de integración de PostgreSQL.

```yaml
# Ejemplo de fragmento de .github/workflows/test.yml
- name: Run Integration Tests (PostgreSQL)
  run: |
    docker-compose up -d db_postgres
    dotnet test --filter "Category=Integration"
```

Esto nos ayudó a interceptar innumerables errores relacionados con diferencias de sintaxis SQL y sensibilidad a mayúsculas y minúsculas.

## Conclusión

A través de la introducción de una capa de abstracción e inyección de dependencias impulsada por configuración, logramos con éxito en el proyecto **HagiCode** la operación de "doble vía" con PostgreSQL y SQLite. Esto no solo redujo enormemente la barrera de entrada para los nuevos desarrolladores (no necesitan instalar PG), sino que también proporcionó una garantía sólida de rendimiento para el entorno de producción.

Repasando los puntos clave:
1.  **Abstracción ante todo**: el código de negocio no depende de implementaciones específicas de base de datos.
2.  **Separación de configuración**: desarrollo y producción usan diferentes `appsettings.json`.
3.  **Separación de migraciones**: no intente que un solo Migration sirva para todo.
4.  **Degradación de características**: en SQLite, priorizar la compatibilidad; en PostgreSQL, priorizar el rendimiento.

Este patrón de arquitectura no solo se aplica a HagiCode, sino también a cualquier proyecto .NET que necesite encontrar un equilibrio entre un desarrollo ligero y una producción robusta.
