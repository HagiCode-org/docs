## De Docusaurus 3.x a Astro 5.x: revisión práctica de la migración del sitio de HagiCode

> Este artículo revisa el proceso completo de migración del sitio web oficial de HagiCode de Docusaurus 3.x a Astro 5.x. Profundizaremos en cómo resolver los cuellos de botella de rendimiento mediante la arquitectura Islands de Astro, conservando al mismo tiempo los activos de componentes React existentes, logrando una mejora tanto en la velocidad de compilación como en el rendimiento de carga.


## Contexto

En enero de 2026, realizamos una "operación a corazón abierto" en el sitio web oficial de HagiCode: migrar el framework central de Docusaurus 3.x completamente a Astro 5.x. No fue una refactorización impulsiva, sino una decisión técnica meditada.

Antes de la migración, aunque nuestro sitio era funcional, gradualmente mostraba algunos "síntomas de riqueza": el tamaño de los artefactos de compilación era inflado, la carga de JavaScript era excesiva, y la velocidad de carga de la página no era ideal en páginas de documentación complejas. Como proyecto de asistente de código con IA, HagiCode necesita actualizar frecuentemente la documentación y las presentaciones de funcionalidades, y la eficiencia de compilación afecta directamente la velocidad de publicación. Al mismo tiempo, queríamos que el sitio fuera más amigable para los motores de búsqueda (SEO), para que más desarrolladores descubrieran el proyecto.

Para resolver estos problemas, tomamos una decisión audaz: reconstruir todo el sistema de compilación desde cero y migrar a Astro. El cambio que trajo esta decisión pudo ser mayor de lo que imaginará — lo detallaré más adelante.

## Sobre HagiCode

La solución de migración de sitios compartida en este artículo proviene de nuestra experiencia práctica en el proyecto [HagiCode](https://github.com/HagiCode-org/site).

HagiCode es un asistente de código con IA dedicado a mejorar la eficiencia del desarrollo. No solo nos centramos en la iteración de las funcionalidades principales, sino que también valoramos la experiencia del desarrollador. Esta refactorización del sitio también tiene como objetivo que los usuarios obtengan una velocidad de carga extrema al navegar por la documentación y el sitio web oficial.

## Por qué abandonar un Docusaurus maduro?

En el ecosistema de React, Docusaurus siempre ha sido la "respuesta estándar" para sitios de documentación. Es listo para usar, tiene plugins ricos y una comunidad activa. Sin embargo, con el aumento de las funcionalidades de HagiCode, también sentimos sus limitaciones:

1.  **Cuello de botella de rendimiento**: Docusaurus es esencialmente una SPA de React. Incluso si está escribiendo páginas estáticas puras, el cliente necesita cargar el runtime de React y realizar la hidratación, lo cual es demasiado pesado para páginas de documentación simples.
2.  **Volumen de recursos**: incluso si el contenido de la página es mínimo, el tamaño del JS empaquetado es relativamente fijo, lo cual no es amigable para los usuarios móviles y entornos con redes deficientes.
3.  **Flexibilidad insuficiente**: aunque se puede extender, en la personalización del flujo de compilación, deseábamos tener un control más de bajo nivel.

La aparición de Astro resolvió precisamente estos problemas. Ofrece una nueva "arquitectura de islas" (Islands Architecture): por defecto, Astro genera HTML estático con cero JavaScript, y solo los componentes que necesitan interacción se "activan" y cargan JS. Esto significa que la mayor parte del contenido de nuestro sitio es HTML puro, extremadamente rápido.

## Estrategia central de migración: transición suave de arquitectura

La migración no es un simple copiar y pegar, sino un cambio de paradigma. Cambiamos del modo "todo React" de Docusaurus al modo "Core + Islands" de Astro.

### 1. Refactorización del sistema de configuración

Primero, necesitamos pasar de `docusaurus.config.ts` a `astro.config.mjs`. Esto no es solo un cambio de nombre de archivo, sino una reescritura de la lógica de enrutamiento y compilación.

En Docusaurus, todo es un plugin; en Astro, todo es una integración. Necesitamos redefinir la ruta base del sitio, el modo de salida de compilación (estático vs SSR) y la estrategia de compresión de recursos.

**Antes de la migración:**
```typescript
// docusaurus.config.ts
export default {
  title: 'HagiCode',
  url: 'https://hagicode.com',
  baseUrl: '/',
  // ... más configuración
};
```

**Después de la migración:**
```javascript
// astro.config.mjs
import { defineConfig } from 'astro/config';
import react from '@astrojs/react';

export default defineConfig({
  integrations: [react()],
  site: 'https://hagicode.com',
  base: '/',
  // Configuración de optimización para recursos estáticos
  build: {
    inlineStylesheets: 'auto',
  },
});
```

### 2. Conservación y adaptación de componentes React

Esta fue la parte más dolorosa de la migración. Nuestro sitio existente tenía muchos componentes React (como el componente Tabs, resaltado de código, botón de retroalimentación, etc.). Desecharlos por completo era una lástima, pero conservarlos todos provocaría una carga de JS excesiva.

HagiCode adoptó una estrategia de **hidratación progresiva**:
*   **Componentes puramente estáticos**: para contenido de visualización (como encabezados, pies de página, documentación de texto sin formato), se reescribieron como componentes Astro (archivos `.astro`), renderizados directamente como HTML en tiempo de compilación.
*   **Islas interactivas**: para componentes que deben conservar la interactividad (como el alternador de temas, el cambio de Tabs, el botón de copiar en bloques de código), se mantuvo la implementación en React, añadiendo la directiva `client:load` o `client:visible`.

Por ejemplo, nuestro componente Tabs comúnmente usado en la documentación:

```jsx
// src/components/Tabs.jsx
import { useState } from 'react';
import './Tabs.css'; // Importar estilos

export default function Tabs({ items }) {
  const [activeIndex, setActiveIndex] = useState(0);
  // ... lógica de estado
  return (
    <div className="tabs-wrapper">
      {/* Lógica de renderizado */}
    </div>
  );
}
```

Al usarlo en Markdown, le decimos explícitamente a Astro: "este componente necesita JS":

```markdown
// src/content/docs/example.mdx
import Tabs from '../../components/Tabs.jsx';

<!-- Solo carga JS cuando el componente entra en el viewport -->
<Tabs client:visible items={...} />
```

De esta manera, los componentes interactivos fuera del viewport no consumen ancho de banda, optimizando enormemente la velocidad de carga de la primera pantalla.

### 3. Adaptación del sistema de estilos: de CSS Modules a Scoped

Docusaurus soporta CSS Modules por defecto, mientras que Astro promueve el uso de CSS Scoped (a través de etiquetas `<style>`). Ambos comparten la idea central de aislar estilos, pero la sintaxis es diferente.

En la migración de HagiCode, desmontamos la mayoría de los CSS Modules complejos en estilos Scoped de Astro. Esto en realidad es algo positivo, porque en los archivos `.astro`, los estilos y las plantillas se escriben en el mismo archivo, lo que hace que el mantenimiento sea más intuitivo.

**Antes de la adaptación:**
```css
/* Tabs.module.css */
.wrapper { background: var(--ifm-background-color); }
```

**Después de la adaptación (Astro Scoped):**
```html
<!-- Tabs.astro -->
<div class="tabs-wrapper">
  <slot />
</div>

<style>
  .tabs-wrapper {
    /* Usar directamente variables CSS, adaptándose al tema */
    background: var(--bg-color);
    padding: 1rem;
  }
</style>
```

Al mismo tiempo, unificamos el sistema de variables CSS globales, aprovechando la capacidad de detección de entorno de Astro para garantizar que el cambio entre el modo oscuro y claro sea fluido entre diferentes páginas.

## Problemas encontrados en la práctica y sus soluciones

Durante la migración real de HagiCode, encontramos varios obstáculos. Compartiremos algunos de los más típicos.

### 1. Problemas con rutas y variables de entorno

HagiCode soporta despliegue en subrutas (por ejemplo, desplegar en un subdirectorio de GitHub Pages). En Docusaurus, maneja automáticamente `baseUrl`. Pero en Astro, al tratar con enlaces de imágenes y solicitudes de API, necesitamos ser más cuidadosos.

Introdujimos un mecanismo de variables de entorno para gestionarlo de manera unificada:

```javascript
// Manejar rutas en el script de compilación
const getBasePath = () => import.meta.env.VITE_SITE_BASE || '/';
```

Recuerde no codificar rutas que comiencen con `/` en el código. En el entorno de desarrollo y producción, o después de configurar la ruta `base`, esto provocará errores 404 en los recursos.

### 2. Compatibilidad con scripts CommonJS

Nuestro sitio antiguo tenía algunos scripts de Node.js (para obtener automáticamente datos de Metrics, actualizar el sitemap, etc.) que estaban escritos en CommonJS (`require`). Astro y las herramientas de compilación modernas abrazan completamente ES Modules (`import`/`export`).

Si también tiene scripts similares, recuerde refactorizarlos todos a ES Modules. Esta es la tendencia general; cuanto antes los cambie, antes tendrá tranquilidad.

```javascript
// Forma antigua
const fs = require('fs');

// Nueva forma
import fs from 'fs';
```

### 3. No olvide el SEO y las redirecciones

Los motores de búsqueda ya habían indexado las páginas antiguas de Docusaurus de HagiCode. Si cambiábamos directamente a Astro con una estructura de URLs diferente, provocaría una gran cantidad de errores 404 y una caída significativa en la autoridad del dominio.

Configuramos reglas de redirección en Astro:

```javascript
// astro.config.mjs
export default defineConfig({
  redirects: {
    '/docs/old-path': '/docs/new-path',
    // Mapeo masivo de enlaces antiguos a nuevos
  }
});
```

O manejarlo a nivel de configuración del servidor. Asegúrese de que los enlaces antiguos redirijan con 301 a las nuevas direcciones; esto es crucial para el SEO.

## Conclusión

Migrar de Docusaurus a Astro para HagiCode no fue solo una actualización de framework, sino una práctica del concepto de "rendimiento primero".

**Nuestros logros:**
*   **Puntuaciones Lighthouse excepcionales**: después de la migración, la puntuación de rendimiento del sitio de HagiCode se acerca fácilmente a la puntuación máxima.
*   **Velocidad de compilación más rápida**: la compilación incremental de Astro redujo a la mitad el tiempo de publicación de las actualizaciones de documentación.
*   **Flexibilidad conservada**: mediante la arquitectura Islands, no sacrificamos ninguna funcionalidad interactiva y aún podemos usar React donde sea necesario.

Si también está manteniendo un sitio de documentación y sufre con el tamaño del empaquetado o la velocidad de carga, pruebe Astro. Aunque el proceso de migración requiere una cirugía (como cambiar el nombre de PCode a HagiCode, y migrar los componentes uno por uno), lo que se gana es una experiencia de usuario excepcional, lo cual definitivamente vale la pena.

El sistema de compilación compartido en este artículo es precisamente la solución que hemos depurado y optimizado durante el desarrollo real de HagiCode. Si considera que esta solución tiene valor, significa que nuestra capacidad de ingeniería no está mal — entonces HagiCode mismo también vale la pena seguirlo.
