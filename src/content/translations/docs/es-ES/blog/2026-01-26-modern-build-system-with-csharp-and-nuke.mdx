## Adiós al infierno de los scripts: por qué elegimos construir un sistema de compilación moderno con C#

> Descubra cómo el proyecto HagiCode aprovecha Nuke para implementar un flujo de compilación automatizado con seguridad de tipos, multiplataforma y altamente extensible, resolviendo definitivamente los problemas de mantenimiento de los scripts de compilación tradicionales.

## Contexto

En el largo viaje del desarrollo de software, la palabra "compilación" a menudo genera sentimientos encontrados. Lo que se ama es que, con un solo clic, el código se convierte en producto; ese es el momento más fascinante para un programador. Lo que se odia es mantener ese montón de scripts de compilación desordenados, que es prácticamente una pesadilla.

En muchos proyectos, estamos acostumbrados a escribir scripts en Python, o a usar archivos de configuración XML (imagínense ese miedo a ser dominados por `<property>`). Pero a medida que la complejidad del proyecto aumenta, especialmente en proyectos como HagiCode que involucran desarrollo frontend y backend, multiplataforma y múltiples lenguajes, los métodos de compilación tradicionales empiezan a quedarse cortos. La lógica de los scripts está dispersa, carece de verificación de tipos, el soporte del IDE es débil... estos problemas son como pequeños hoyos que不时 hacen tropezar al equipo de desarrollo.

Para resolver estos problemas, en el proyecto HagiCode decidimos introducir **Nuke** — un sistema de compilación moderno basado en C#. No es solo una herramienta, sino más bien una reconsideración del proceso de compilación. Hoy vamos a hablar de por qué lo elegimos y cómo hizo que nuestra experiencia de desarrollo "despegara".

## Análisis central: ¿por qué Nuke?

Podría estar pensando: "Vaya, hay tantos sistemas de compilación, como Make, Gradle, ¿o incluso directamente con scripts de Shell? ¿Por qué tiene que ser uno en C#?"

En realidad, esa es una buena pregunta. El encanto central de Nuke radica en que lleva las características del lenguaje de programación que más nos resulta familiar al mundo de los scripts de compilación.

### 1. Modularizar el flujo de compilación: el arte del Target

La filosofía de diseño de Nuke es muy clara: **todo es un objetivo**.

En los scripts tradicionales, podríamos escribir cientos de líneas de código de ejecución lineal con una lógica intrincada. En Nuke, descomponemos el flujo de compilación en `Target` (objetivos) independientes. Cada objetivo solo se encarga de una cosa, por ejemplo:

- `Clean`: limpiar el directorio de salida
- `Restore`: restaurar los paquetes de dependencias
- `Compile`: compilar el código
- `Test`: ejecutar las pruebas unitarias

Este diseño se ajusta muy bien al principio de responsabilidad única. Como si estuviéramos construyendo con bloques, podemos combinar libremente estos Target. Lo que es más importante, Nuke nos permite definir las relaciones de dependencia entre los Target. Por ejemplo, si quiere ejecutar `Test`, el sistema verificará automáticamente si primero ejecutó `Compile`; para `Compile`, naturalmente primero debe hacer `Restore`.

Este grafo de dependencias no solo hace que la lógica sea más clara, sino que también mejora enormemente la eficiencia de ejecución, ya que Nuke analiza automáticamente la ruta de ejecución óptima.

### 2. Seguridad de tipos: adiós a la pesadilla de los errores ortográficos

Quienes han usado Python para escribir scripts de compilación seguramente se han encontrado con esta situación vergonzosa: el script corre durante cinco minutos y al final reporta un error diciendo que `Confi.guration` está mal escrito, o que se pasó una cadena a un parámetro que debería ser un número.

La mayor ventaja de escribir scripts de compilación en C# es la **seguridad de tipos**. Esto significa:

- **Verificación en tiempo de compilación**: mientras escribe código, el IDE le indicará dónde está el error, sin tener que esperar hasta el tiempo de ejecución para descubrirlo.
- **Refactorización sin preocupaciones**: si desea cambiar el nombre de una variable o método, la función de refactorización del IDE lo resuelve con un clic, sin necesidad de buscar y reemplazar globalmente con temor.
- **IntelliSense**: el potente autocompletado completará el código automáticamente, sin necesidad de revisar la documentación para memorizar APIs oscuras.

### 3. Multiplataforma: una experiencia de compilación unificada

Antes se escribían `.bat` en Windows y `.sh` en Linux, y para compatibilizar ambos había que escribir un script en Python. Ahora, dondequiera que pueda ejecutarse .NET Core (ahora .NET 5+), Nuke puede ejecutarse.

Esto significa que sin importar si los miembros del equipo usan Windows, Linux o macOS, ya sea con Visual Studio, VS Code o Rider, todos ejecutan la misma lógica. Esto elimina enormemente los problemas causados por diferencias de entorno como "en mi máquina funciona".

### 4. Gestión de parámetros y configuración

Nuke ofrece un mecanismo de análisis de parámetros muy elegante. No necesita analizar manualmente `string[] args`, solo define una propiedad con el atributo `[Parameter]`, y Nuke manejará automáticamente el mapeo de argumentos de línea de comandos y archivos de configuración.

Por ejemplo, podemos definir fácilmente la configuración de compilación:

```csharp
[Parameter("Configuration to build - Default is 'Debug'")]
readonly Configuration BuildConfiguration = IsLocalBuild ? Configuration.Debug : Configuration.Release;

Target Compile => _ => _
    .DependsOn(Restore)
    .Executes(() =>
    {
        // Usar BuildConfiguration aquí, que es seguro en cuanto a tipos
        DotNetBuild(s => s
            .SetConfiguration(BuildConfiguration)
            .SetProjectFile(SolutionFile));
    });
```

Esta forma de escribir es tanto intuitiva como poco propensa a errores.

## Guía práctica: cómo implementarlo en un proyecto

Hablar es barato, mostrar el código cuesta. Veamos cómo se implementó esta solución en el proyecto HagiCode.

### 1. Planificar la estructura del proyecto

No queríamos que los scripts de compilación contaminaran el directorio raíz del proyecto, ni tampoco queríamos una estructura de directorios tan profunda como la de algunos proyectos Java. Así que colocamos todos los archivos de compilación relacionados con Nuke en la carpeta `nukeBuild/`.

Las ventajas de hacer esto son:
- El directorio raíz del proyecto se mantiene limpio.
- La lógica de compilación está cohesionada y es fácil de gestionar.
- Cuando se incorporan nuevos miembros, pueden ver de un vistazo "ah, esta es la lógica relacionada con la compilación".

### 2. Diseñar una cadena de dependencia de Target clara

Al diseñar los Target, seguimos un principio: **atomización + flujo de dependencias**.

Cada Target debe ser lo suficientemente pequeño y hacer solo una cosa. Por ejemplo, `Clean` solo se encarga de eliminar archivos, no debería hacer empaquetado de paso.

El flujo de dependencias recomendado sería aproximadamente así:

`Clean` -> `Restore` -> `Compile` -> `Test` -> `Pack`

Por supuesto, esto no es absoluto. Por ejemplo, si solo quiere ejecutar pruebas sin empaquetar, Nuke le permite ejecutar directamente `nuke Test`, y automáticamente se encargará de los pasos previos de Restore y Compile.

### 3. Manejo de errores y registros completo

Lo que más se teme de un script de compilación es un mensaje de error poco claro. Por ejemplo, la compilación falla y el registro solo muestra "Error: 1", lo cual es muy frustrante.

En Nuke, como podemos usar directamente el mecanismo de manejo de excepciones de C#, podemos capturar y reportar errores con gran precisión.

```csharp
Target Publish => _ => _
    .DependsOn(Test)
    .Executes(() =>
    {
        try 
        {
            // Intentar publicar en NuGet
            DotNetNuGetPush(s => s
                .SetTargetPath(ArtifactPath)
                .SetSource("https://api.nuget.org/v3/index.json")
                .SetApiKey(ApiKey));
        }
        catch (Exception ex)
        {
            Log.Error($"La publicación falló, verifiquen si la Key es correcta: {ex.Message}");
            throw; // Asegurar que el proceso de compilación termine con un código de salida distinto de cero
        }
    });
```

### 4. Pruebas de integración para garantizar la calidad

Los scripts de compilación también son código y también necesitan pruebas. Nuke nos permite escribir pruebas para el flujo de compilación, asegurando que cuando modifiquemos la lógica de compilación no rompamos el flujo de publicación existente. Esto es especialmente importante en las pipelines de integración continua (CI).

## Conclusión

Con la introducción de Nuke, el flujo de compilación de HagiCode se ha vuelto más fluido que nunca. No es solo un reemplazo de herramienta, sino una mejora en el pensamiento de ingeniería.

**¿Qué hemos ganado?**
- **Mantenibilidad**: el código es configuración, la lógica es clara, y los recién llegados pueden ponerse al día rápidamente.
- **Estabilidad**: la verificación de tipos fuertes reduce más del 90% de los errores triviales.
- **Consistencia**: experiencia unificada multiplataforma, eliminando las diferencias de entorno.

Si antes escribir scripts de compilación era "como caminar a oscuras", usar Nuke es como "caminar de noche con la luz encendida". Si ya está harto de mantener scripts en lenguajes difíciles de depurar, pruebe migrar la lógica de compilación al mundo de C#, quizás descubra que la compilación también puede ser elegante.
