## Diseñar 12 experiencias de inicio extremas para HagiCode: del minimalismo al cyberpunk

> En el breve intervalo entre la descarga y la hidratación de una aplicación React 19, existe una ventana dorada para que los usuarios perciban la personalidad de la marca. Este artículo comparte cómo construimos en el proyecto HagiCode un sistema completo de estilos de inicio basado en HTML/CSS/JS.


## Contexto

HagiCode, como una aplicación moderna basada en ASP.NET Core 10 y React 19 (Vite), adopta una arquitectura de despliegue con separación entre frontend y backend. Los artefactos del frontend se empaquetan y colocan en el directorio `wwwroot/` del backend para ser servidos por ASP.NET Core.

Sin embargo, esta arquitectura trae un problema clásico de experiencia de usuario: cuando el usuario accede a la página web, el navegador necesita primero cargar el HTML, luego descargar el enorme JS Bundle, y finalmente React ejecuta la hidratación. En este "período de vacío" de cientos de milisegundos a varios segundos, el usuario se enfrenta a una pantalla en blanco o a una página estática sin vida.

Para cubrir este intervalo e inyectar la personalidad de marca de HagiCode, necesitábamos diseñar un sistema completo de estilos de inicio implementado enteramente con código inline en `index.html`.

## Sobre HagiCode

El diseño de pantalla de inicio compartido en este artículo proviene de nuestra experiencia práctica en el proyecto [HagiCode](https://github.com/HagiCode-org/site). Como asistente de código con IA, HagiCode no solo se centra en la eficiencia de la generación de código, sino que también valora la experiencia visual del desarrollador. Este sistema de inicio es precisamente el resultado de nuestra búsqueda del rendimiento extremo del frontend.

## Desafío central y diseño de arquitectura

Antes de diseñar, debemos primero aclarar las restricciones técnicas. Dado que la implementación debe ser inline en `index.html`, significa que no podemos cargar ningún archivo CSS o JS externo (excepto el Bundle de React en sí).

### Análisis de restricciones técnicas

1.  **Principio de cero dependencias**: todos los estilos deben escribirse dentro de etiquetas `<style>`, y la lógica dentro de etiquetas `<script>`.
2.  **CSS defensivo**: para evitar que los estilos globales de la aplicación React una vez montada contaminen la pantalla de inicio, decidimos usar un prefijo de ID de alta prioridad (como `#boot-screen`) para envolver todos los estilos de inicio.
3.  **Rendimiento primero**: las animaciones deben usar preferentemente `transform` y `opacity` de CSS, evitando activar reflows y asegurando no bloquear el hilo principal.
4.  **Consistencia visual**: los colores y fuentes deben mantenerse consistentes con la configuración de Tailwind de HagiCode.

### Patrón de arquitectura: Shell e Injector

Adoptamos un **patrón de variantes**. La lógica central se encapsula en una función de ejecución inmediata (IIFE), y la lógica de renderizado específica se inyecta como elemento de configuración. De esta manera podemos cambiar entre diferentes estilos con una simple configuración, sin necesidad de reescribir la lógica de manipulación del DOM.

A continuación se muestra el código de arquitectura central:

```html
<!-- Inline en index.html -->
<div id="boot-root"></div>

<script>
(function() {
  const BootSequence = {
    config: {
      theme: 'terminal', // Configurable como 'minimal', 'skeleton', 'code-rain', etc.
      color: '#3b82f6'   // Color de marca
    },
    
    // Ciclo de vida central
    init() {
      this.render();
      this.listenForMount();
    },

    // Renderizar el estilo seleccionado actualmente
    render() {
      const root = document.getElementById('boot-root');
      if (this.variants[this.config.theme]) {
        root.innerHTML = this.variants[this.config.theme].render();
      }
    },

    // Escuchar el montaje exitoso de React, salida elegante
    listenForMount() {
      window.addEventListener('hagicode:ready', () => {
        const screen = document.getElementById('boot-root');
        // Primero desvanecer, luego eliminar el DOM, evitando parpadeos
        screen.style.opacity = '0';
        screen.style.transition = 'opacity 0.3s ease';
        setTimeout(() => screen.remove(), 300);
      });
    },

    // La lógica de implementación de los 12 estilos se concentra aquí
    variants: {
      // ...implementaciones específicas ver más abajo
    }
  };

  BootSequence.init();
})();
</script>
```

## Inventario de diseño de 12 estilos de inicio

Dividimos estos 12 estilos en seis categorías principales, para satisfacer diferentes escenarios y necesidades estéticas.

### A. Minimalismo

> "Menos es más". Para escenarios que buscan una velocidad de carga extrema, ofrecemos la solución más ligera.

#### 1. Minimalist Dot (respiración minimalista)
Solo un simple punto en el centro de la pantalla, con una animación de respiración.
*   **Implementación**: `@keyframes` de CSS controlando scale y opacity.
*   **Aplicación**: cualquier ocasión donde se necesite mantener la página absolutamente limpia.

#### 2. Brand Reveal (revelación de marca)
A través de la animación SVG `stroke-dasharray`, simula el dibujo a mano de las líneas del Logo de HagiCode, seguido de un desvanecimiento del texto.
*   **Técnica**: animación de rutas SVG, muy texturizada.

### B. Mimicismo de pantalla de esqueleto

> "El arte de engañar a los ojos". Simulando el diseño real de la interfaz de usuario, haciendo que el usuario sienta que la página ya se ha cargado a medias.

#### 3. Sidebar Chat Skeleton (pantalla de esqueleto de barra lateral)
Probablemente la más práctica. Construimos manualmente con HTML un diseño idéntico a los componentes React `Sidebar` y `ChatInput`, cubierto con una animación de rayas grises.
*   **Valor**: cuando React completa la hidratación, la pantalla de esqueleto se transforma instantáneamente en los componentes reales, y el usuario casi no percibe el cambio.

#### 4. Card Stack Skeleton (pila de tarjetas)
Simula el efecto de apilamiento durante la carga de tarjetas de propuestas, usando transformaciones 3D para que las tarjetas floten ligeramente.

### C. Abstracción y arte

> Mostrando los genes geek de HagiCode.

#### 5. Geometric Morph (morfismo geométrico)
Renderiza una forma geométrica (cuadrado) en el centro de la pantalla, que se transforma suavemente en un círculo, luego en un triángulo, y finalmente se convierte en el Logo.
*   **Técnica**: transición suave de `border-radius` de CSS.

#### 6. Code Rain (lluvia de código)
Un homenaje a "The Matrix". Usa la fuente JetBrains Mono, con un flujo tenue de caracteres cayendo en el fondo.
*   **Nota**: por rendimiento, el flujo de caracteres debe limitarse a un área pequeña o reducir la frecuencia de actualización.

#### 7. Neon Pulse (pulso neón)
Anillos luminosos de estilo cyberpunk, utilizando múltiples superposiciones de `box-shadow` para crear una fuerte sensación de resplandor.

### D. Marca y tema

> Hacer que el sistema "cobre vida".

#### 8. Seasonal Theme (tema festivo)
Este es un cargador dinámico. Según la fecha actual, determina la festividad (como el Año Nuevo Lunar, Navidad) y carga la animación SVG correspondiente.
*   **Ejemplo**: durante el Año Nuevo Lunar, en la parte inferior de la pantalla hay linternas rojas balanceándose suavemente.

#### 9. Gradient Flow (flujo degradado)
El fondo usa un degradado fluido con los colores de marca de HagiCode, combinado con animaciones de `background-size` y `background-position`, creando una sensación de fluidez similar a las auroras boreales.

### E. Sensación tecnológica

> Un tributo a los desarrolladores.

#### 10. Terminal Boot (inicio de terminal)
Simula la salida de consola. Líneas de código se desplazan rápidamente:
```text
> Initializing HagiCode Core...
> Loading models...
> Connecting to neural network...
```
Esto hará que cualquier desarrollador se sienta en casa.

#### 11. Progress Bar (barra de progreso minimalista)
Una fina barra de progreso en la parte superior de la pantalla, con un porcentaje a la derecha. Aunque no podemos obtener el progreso real de descarga, podemos usar un temporizador para simular un proceso de carga "creíble" (rápido al 80%, desacelerando en el 20% final).

### F. Creatividad

#### 12. Pixel Assembly (ensamblaje de píxeles)
Una idea muy interesante. Bloques dispersos en la pantalla se congregan hacia el centro, ensamblando gradualmente el icono del Logo de HagiCode. Simboliza el proceso de construcción de código.

## Mejores prácticas y lecciones aprendidas

En el desarrollo real de HagiCode, resumimos algunos detalles de implementación cruciales.

### 1. El CSS defensivo es obligatorio
No sea perezoso al omitir los prefijos. Una vez, no añadimos restricciones de ID a los estilos de la pantalla de inicio, lo que causó que los estilos globales `div` después del montaje de React afectaran inesperadamente la pantalla de inicio, provocando una ruptura del diseño.
**Experiencia**: todos los selectores CSS deben colgarse bajo `#boot-screen`, y usar `!important` para elevar la prioridad (solo en el CSS de la pantalla de inicio).

### 2. Transición elegante
Después de que React monte con éxito, no elimine directamente el DOM de la pantalla de inicio con `remove()`.
**Enfoque correcto**:
1.  React dispara `window.dispatchEvent(new Event('hagicode:ready'))`.
2.  La pantalla de inicio detecta el evento, primero estableciendo `opacity: 0`.
3.  Esperar 300ms (tiempo de transición CSS), asegurando que el usuario no pueda verlo, y luego ejecutar `.remove()`.

### 3. Sincronización de variables de tema
Los códigos de color de la pantalla de inicio están codificados en `index.html`. Si modificamos el color principal de Tailwind, debemos sincronizar el cambio aquí.
**Solución optimizada**: en el script de compilación de Vite, escribir un plugin simple que lea `tailwind.config.js` e inyecte las variables de color en las variables de plantilla de `index.html`, implementando una única fuente de datos.

### 4. Precarga de fuentes
La pantalla de inicio generalmente necesita usar la fuente de marca, pero si la fuente se carga lentamente, aparecerá un FOUT (Flash of Unstyled Text).
**Solución**: añadir en `<head>` un `<link rel="preload" href="/fonts/JetBrainsMono.woff2" as="font" type="font/woff2" crossorigin>`. Este es un medio de bajo costo y alto retorno para mejorar la experiencia.

### 5. Monitoreo de rendimiento
Inyectamos `performance.mark('boot-start')` en la parte inferior de `index.html`, y marcamos `boot-end` cuando React monta con éxito.
**Significado**: a través de Application Insights recopilamos estos datos, pudiendo ver realmente hasta qué punto la pantalla de inicio reduce el tiempo de espera percibido por el usuario (Perceived Loading Time). Los datos muestran que una excelente pantalla de esqueleto puede aumentar la tolerancia de los usuarios a las "redes lentas" en más de un 50%.

## Conclusión

Una buena pantalla de inicio no es solo "decoración mientras se espera", es la señal de apretón de manos de la primera interacción del producto con el usuario. En el proyecto HagiCode, este sistema de inicio basado en el **patrón Variants** nos permite cambiar flexiblemente entre estilos en diferentes festividades y versiones, mejorando enormemente la personalidad y profesionalismo del producto.
