## Xây dựng sự tích hợp các vấn đề GitHub từ đầu: Thực hành kết nối trực tiếp Frontend của HagiCode

> Bài viết này ghi lại toàn bộ quá trình tích hợp các vấn đề GitHub vào nền tảng HagiCode. Chúng ta sẽ khám phá cách sử dụng kiến ​​trúc "kết nối trực tiếp giao diện người dùng + phần phụ trợ tối thiểu" để đạt được xác thực OAuth an toàn và đồng bộ hóa vấn đề hiệu quả trong khi vẫn giữ cho phần phụ trợ nhẹ.

## Bối cảnh: Tại sao nên tích hợp GitHub?

Là nền tảng phát triển được hỗ trợ bởi AI, giá trị cốt lõi của HagiCode nằm ở việc kết nối ý tưởng với việc triển khai. Nhưng trong thực tế sử dụng, chúng tôi nhận thấy rằng sau khi người dùng hoàn thành Đề xuất trong HagiCode, họ thường cần sao chép thủ công nội dung vào GitHub Issues để theo dõi dự án.

Điều này tạo ra một số điểm khó khăn rõ ràng:

1.  **Quy trình làm việc bị phân mảnh**: Người dùng cần chuyển đổi qua lại giữa hai hệ thống. Trải nghiệm không mượt mà và thông tin chính có thể dễ dàng bị mất trong quá trình sao chép và dán.
2.  **Cộng tác không thuận tiện**: Các thành viên khác trong nhóm đã quen với việc kiểm tra các nhiệm vụ trên GitHub và không thể xem trực tiếp tiến trình đề xuất trong HagiCode.
3.  **Công việc thủ công lặp đi lặp lại**: Mỗi khi một đề xuất được cập nhật, ai đó phải cập nhật thủ công vấn đề tương ứng trên GitHub, làm tăng thêm chi phí bảo trì không cần thiết.

Để giải quyết vấn đề này, chúng tôi quyết định giới thiệu tính năng **Tích hợp vấn đề GitHub**, kết nối các phiên HagiCode với kho lưu trữ GitHub để bật "đồng bộ hóa bằng một cú nhấp chuột".

## Giới thiệu về HagiCode

## Lựa chọn kỹ thuật: Kết nối trực tiếp Frontend và Proxy phụ trợ

Khi thiết kế phương pháp tích hợp, chúng tôi có hai lựa chọn trước mắt: "mô hình proxy phụ trợ" truyền thống và "mô hình kết nối trực tiếp giao diện người dùng" tích cực hơn.

### So sánh giải pháp

Trong **mô hình proxy phụ trợ** truyền thống, mọi yêu cầu từ giao diện người dùng trước tiên phải đi qua chương trình phụ trợ của chúng tôi, sau đó gọi API GitHub. Điều này tập trung hóa logic, nhưng nó cũng đặt gánh nặng đáng kể lên phần phụ trợ:

1.  **Phần phụ trợ cồng kềnh**: Chúng tôi cần phải viết trình bao bọc ứng dụng khách API GitHub chuyên dụng và cũng xử lý máy trạng thái OAuth phức tạp.
2.  **Rủi ro về mã thông báo**: Mã thông báo GitHub của người dùng sẽ phải được lưu trữ trong cơ sở dữ liệu phụ trợ. Ngay cả với mã hóa, điều này vẫn làm tăng bề mặt bảo mật.
3.  **Chi phí phát triển**: Chúng tôi sẽ cần di chuyển cơ sở dữ liệu để lưu trữ mã thông báo và dịch vụ đồng bộ hóa bổ sung để duy trì.

**Mô hình kết nối trực tiếp giao diện người dùng** nhẹ hơn nhiều. Trong phương pháp này, chúng tôi chỉ sử dụng phần phụ trợ cho bước "trao đổi bí mật" nhạy cảm nhất (gọi lại OAuth). Sau khi nhận được mã thông báo, chúng tôi lưu trữ trực tiếp trong trình duyệt `localStorage`. Các hoạt động sau này như tạo sự cố và cập nhật nhận xét được gửi trực tiếp từ giao diện người dùng tới GitHub qua HTTP.

| Thứ nguyên so sánh | Mô hình proxy phụ trợ | Mô hình kết nối trực tiếp Frontend |
| :--- | :--- | :--- |
| **Độ phức tạp của phần phụ trợ** | Yêu cầu dịch vụ OAuth đầy đủ và ứng dụng khách API GitHub | Chỉ cần điểm cuối gọi lại OAuth |
| **Quản lý mã thông báo** | Phải được mã hóa và lưu trữ trong cơ sở dữ liệu, có nguy cơ rò rỉ | Được lưu trữ trong trình duyệt và chỉ hiển thị cho người dùng |
| **Chi phí thực hiện** | Yêu cầu di chuyển cơ sở dữ liệu và phát triển đa dịch vụ | Công việc chủ yếu ở frontend |
| **Trải nghiệm người dùng** | Logic tập trung, nhưng độ trễ của máy chủ có thể cao hơn một chút | Phản hồi cực nhanh với tương tác trực tiếp GitHub |

Bởi vì chúng tôi muốn tích hợp nhanh chóng và giảm thiểu những thay đổi phụ trợ nên **cuối cùng chúng tôi đã chọn "mô hình kết nối trực tiếp giao diện người dùng"**. Nó giống như cấp cho trình duyệt một "vượt qua tạm thời". Sau khi vượt qua, trình duyệt có thể tự xử lý mọi việc trên GitHub mà không cần yêu cầu quản trị viên phụ trợ phê duyệt mỗi lần.

## Thiết kế cốt lõi: Luồng dữ liệu và bảo mật

Sau khi giải quyết xong kiến trúc, chúng tôi cần thiết kế luồng dữ liệu cụ thể. Cốt lõi của quá trình đồng bộ hóa là làm thế nào để lấy được mã thông báo một cách an toàn và sử dụng nó một cách hiệu quả.

### Sơ đồ kiến trúc tổng thể

Toàn bộ hệ thống có thể được tóm tắt thành ba vai trò: trình duyệt (giao diện người dùng), phần phụ trợ HagiCode và GitHub.

```text
+--------------+        +--------------+        +--------------+
| Frontend     |        |   Backend    |        |   GitHub     |
| React        |        |   ASP.NET    |        |   REST API   |
|              |        |              |        |              |
|  +--------+  |        |              |        |              |
|  | OAuth  |--+--------> /callback    |        |              |
|  | Flow   |  |        |              |        |              |
|  +--------+  |        |              |        |              |
|              |        |              |        |              |
|  +--------+  |        |  +--------+  |        |  +--------+  |
|  | GitHub |  +------------> Session |  +----------> Issues |  |
|  | API    |  |        |  |Metadata|  |        |  |        |  |
|  | Direct |  |        |  +--------+  |        |  +--------+  |
|  +--------+  |        |              |        |              |
+--------------+        +--------------+        +--------------+
```

**Điểm mấu chốt là**: chỉ một bước nhỏ trong OAuth (trao đổi mã lấy mã thông báo) cần phải trải qua phần phụ trợ. Sau đó, công việc nặng nhọc (tạo ra sự cố) được xử lý trực tiếp giữa giao diện người dùng và GitHub.

### Luồng dữ liệu đồng bộ hóa chi tiết

Khi người dùng nhấp vào nút "Đồng bộ hóa với GitHub" trong Giao diện người dùng HagiCode, một loạt hành động phức tạp sẽ diễn ra:

```text
User clicks "Sync to GitHub"
         │
         ▼
1. Frontend checks localStorage for the GitHub token
         │
         ▼
2. Format issue content (convert the Proposal into Markdown)
         │
         ▼
3. Frontend directly calls the GitHub API to create/update the issue
         │
         ▼
4. Call the HagiCode backend API to update Session.metadata (store issue URL and other info)
         │
         ▼
5. Backend broadcasts the SessionUpdated event via SignalR
         │
         ▼
6. Frontend receives the event and updates the UI to show the "Synced" state
```

### Thiết kế bảo mật

Bảo mật luôn được ưu tiên hàng đầu khi tích hợp dịch vụ của bên thứ ba. Chúng tôi đã đưa ra những cân nhắc sau:

1.  **Bảo vệ chống lại các cuộc tấn công CSRF**: Tạo ngẫu nhiên `state` tham số trong quá trình chuyển hướng OAuth và lưu trữ nó trong `sessionStorage`. Xác thực nghiêm ngặt trạng thái trong lệnh gọi lại để ngăn chặn các yêu cầu giả mạo.
2.  **Lưu trữ mã thông báo riêng biệt**: Mã thông báo chỉ được lưu trữ trong trình duyệt `localStorage`. Sử dụng Chính sách cùng nguồn gốc, chỉ các tập lệnh HagiCode mới có thể đọc được, tránh nguy cơ rò rỉ cơ sở dữ liệu phía máy chủ ảnh hưởng đến người dùng.
3.  **Ranh giới lỗi**: Chúng tôi đã thiết kế cách xử lý dành riêng cho các lỗi API GitHub phổ biến (chẳng hạn như mã thông báo hết hạn 401, lỗi xác thực 422 và giới hạn tốc độ 429), để người dùng nhận được phản hồi thân thiện.

## Trong thực tế: Chi tiết triển khai trong mã

Lý thuyết chỉ đi xa đến thế. Chúng ta hãy xem mã thực sự hoạt động như thế nào.

### 1. Thay đổi phụ trợ tối thiểu

Phần phụ trợ chỉ cần thực hiện hai việc: lưu trữ thông tin đồng bộ hóa và xử lý lệnh gọi lại OAuth.

**Thay đổi cơ sở dữ liệu**
Chúng ta chỉ cần thêm một `Metadata` cột vào `Sessions` bảng để lưu trữ dữ liệu tiện ích mở rộng ở định dạng JSON.

### 2. Luồng OAuth giao diện người dùng

Đây là điểm vào của kết nối. Chúng tôi sử dụng Luồng mã ủy quyền tiêu chuẩn.

```typescript
// src/HagiCode.Client/src/services/githubOAuth.ts

// Generate the authorization URL and redirect
export async function generateAuthUrl(): Promise<string> {
  const state = generateRandomString(); // Generate a random string for CSRF protection
  sessionStorage.setItem('hagicode_github_state', state);
  
  const params = new URLSearchParams({
    client_id: clientId,
    redirect_uri: window.location.origin + '/settings?tab=github&oauth=callback',
    scope: ['repo', 'public_repo'].join(' '),
    state: state,
  });
  
  return `https://github.com/login/oauth/authorize?${params.toString()}`;
}

// Handle the code-to-token exchange on the callback page
export async function exchangeCodeForToken(code: string, state: string): Promise<GitHubToken> {
  // 1. Validate state to prevent CSRF
  const savedState = sessionStorage.getItem('hagicode_github_state');
  if (state !== savedState) throw new Error('Invalid state parameter');

  // 2. Call the backend API to exchange the token
  // Note: this must go through the backend because ClientSecret cannot be exposed to the frontend
  const response = await fetch('/api/GitHubOAuth/callback', {
    method: 'POST',
    headers: { 'Content-Type': 'application/json' },
    body: JSON.stringify({ code, state, redirectUri: window.location.origin + '/settings?tab=github&oauth=callback' }),
  });

  if (!response.ok) throw new Error('Failed to exchange token');
  
  const token = await response.json();
  
  // 3. Save into LocalStorage
  saveToken(token);
  return token;
}
```

### 3. Trình bao bọc ứng dụng khách API GitHub

Sau khi có mã thông báo, chúng tôi cần một công cụ vững chắc để gọi API GitHub.

```typescript
// src/HagiCode.Client/src/services/githubApiClient.ts

const GITHUB_API_BASE = 'https://api.github.com';

// Core request wrapper
async function githubApi<T>(endpoint: string, options: RequestInit = {}): Promise<T> {
  const token = localStorage.getItem('gh_token');
  if (!token) throw new Error('Not connected to GitHub');
  
  const response = await fetch(`${GITHUB_API_BASE}${endpoint}`, {
    ...options,
    headers: {
      ...options.headers,
      Authorization: `Bearer ${token}`,
      Accept: 'application/vnd.github.v3+json', // Specify the API version
    },
  });
  
  // Error handling logic
  if (!response.ok) {
    if (response.status === 401) throw new Error('GitHub token expired, please reconnect');
    if (response.status === 403) throw new Error('No permission to access this repository or rate limit exceeded');
    if (response.status === 422) throw new Error('Issue validation failed, the title may be duplicated');
    throw new Error(`GitHub API Error: ${response.statusText}`);
  }
  
  return response.json();
}

// Create issue
export async function createIssue(owner: string, repo: string, data: { title: string, body: string, labels: string[] }) {
  return githubApi(`/repos/${owner}/${repo}/issues`, {
    method: 'POST',
    body: JSON.stringify(data),
  });
}
```

### 4. Định dạng và đồng bộ hóa nội dung

Bước cuối cùng là chuyển đổi dữ liệu phiên HagiCode sang định dạng của sự cố GitHub. Nó hơi giống công việc dịch thuật.

```typescript
// Convert a Session object into a Markdown string
function formatIssueForSession(session: Session): string {
  let content = `# ${session.title}\n\n`;
  content += `**> HagiCode Session:** #${session.code}\n`;
  content += `**> Status:** ${session.status}\n\n`;
  content += `## Description\n\n${session.description || 'No description provided.'}\n\n`;
  
  // If this is a Proposal session, add extra fields
  if (session.type === 'proposal') {
    content += `## Chief Complaint\n\n${session.chiefComplaint || ''}\n\n`;
    // Add a deep link so users can jump back from GitHub to HagiCode
    content += `---\n\n**[View in HagiCode](hagicode://sessions/${session.id})**\n`;
  }
  
  return content;
}

// Main logic when clicking the sync button
const handleSync = async (session: Session) => {
  try {
    const repoInfo = parseRepositoryFromUrl(session.repoUrl); // Parse the repository URL
    if (!repoInfo) throw new Error('Invalid repository URL');

    toast.loading('Syncing to GitHub...');
    
    // 1. Format content
    const issueBody = formatIssueForSession(session);
    
    // 2. Call API
    const issue = await githubApiClient.createIssue(repoInfo.owner, repoInfo.repo, {
      title: `[HagiCode] ${session.title}`,
      body: issueBody,
      labels: ['hagicode', 'proposal', `status:${session.status}`],
    });
    
    // 3. Update Session Metadata (save the issue link)
    await SessionsService.patchApiSessionsSessionId(session.id, {
      metadata: {
        githubIssue: {
          owner: repoInfo.owner,
          repo: repoInfo.repo,
          issueNumber: issue.number,
          issueUrl: issue.html_url,
          syncedAt: new Date().toISOString(),
        }
      }
    });

    toast.success('Synced successfully!');
  } catch (err) {
    console.error(err);
    toast.error('Sync failed, please check your token or network');
  }
};
```

## Tóm tắt và Triển vọng

Với phương pháp "kết nối trực tiếp giao diện người dùng" này, chúng tôi đã đạt được sự tích hợp liền mạch Các vấn đề GitHub với ít mã phụ trợ nhất có thể.

### Lợi ích chính
1.  **Hiệu quả phát triển cao**: Ít thay đổi ở phần cuối, chủ yếu là một trường cơ sở dữ liệu bổ sung và điểm cuối gọi lại OAuth đơn giản. Hầu hết logic được hoàn thành ở mặt trước.
2.  **Bảo mật mạnh mẽ**: Mã thông báo không đi qua cơ sở dữ liệu máy chủ, giảm rủi ro rò rỉ.
3.  **Trải nghiệm người dùng tuyệt vời**: Yêu cầu được bắt đầu trực tiếp từ giao diện người dùng nên tốc độ phản hồi nhanh và không cần chuyển tiếp phụ trợ.

### Những điều cần chú ý
Có một số cạm bẫy cần lưu ý trong quá trình triển khai thực tế:
- **Cài đặt ứng dụng OAuth**: Hãy nhớ nhập chính xác `Authorization callback URL` trong cài đặt Ứng dụng GitHub OAuth của bạn (thường `http://localhost:3000/settings?tab=github&oauth=callback`).
- **Giới hạn tỷ lệ**: API GitHub giới hạn các yêu cầu không được xác thực khá nghiêm ngặt, nhưng với mã thông báo, hạn ngạch thường là đủ (5000 yêu cầu/giờ).
- **Phân tích cú pháp URL**: Người dùng nhập tất cả các loại URL kho lưu trữ, vì vậy hãy đảm bảo biểu thức chính quy của bạn có thể khớp `.git` hậu tố, định dạng SSH và các trường hợp tương tự.

### Những cải tiến trong tương lai
Tính năng hiện tại vẫn là đồng bộ một chiều (HagiCode -> GitHub). Trong tương lai, chúng tôi dự định triển khai đồng bộ hóa hai chiều thông qua GitHub Webhooks. Ví dụ: nếu sự cố được khắc phục trên GitHub, trạng thái phiên ở phía HagiCode cũng có thể tự động cập nhật. Điều đó sẽ yêu cầu chúng tôi hiển thị điểm cuối webhook trên phần phụ trợ. Đây sẽ là bước thú vị tiếp theo.

Chúng tôi hy vọng bài viết này mang lại cho bạn một chút cảm hứng để phát triển tích hợp bên thứ ba của riêng bạn. Nếu bạn có thắc mắc, vui lòng mở một vấn đề để thảo luận trên [Mã Hagi GitHub](https://github.com/HagiCode-org/site).
