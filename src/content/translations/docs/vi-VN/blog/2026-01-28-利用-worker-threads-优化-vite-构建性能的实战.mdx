## Phân tích: Tìm điểm đột phá trong điểm nghẽn hiệu suất

Trước khi giải quyết vấn đề hiệu suất, trước tiên chúng tôi cần làm rõ suy nghĩ của mình và xác định giải pháp kỹ thuật tốt nhất.

### Quyết định cốt lõi: Tại sao chọn chủ đề công nhân?

Có ba cách chính để đạt được tính toán song song trong Node.js:
1. **child_process**: tạo các tiến trình con độc lập
2. **Web Workers**: chủ yếu được sử dụng trên trình duyệt
3. **worker_threads**: hỗ trợ đa luồng nguyên bản trong Node.js

Sau khi so sánh các lựa chọn, HagiCode cuối cùng đã chọn **Worker Threads** vì những lý do sau:
- **Không tốn chi phí tuần tự hóa**: Chuỗi công nhân chạy trong cùng một quy trình và có thể chia sẻ bộ nhớ thông qua `SharedArrayBuffer` hoặc chuyển quyền sở hữu, tránh chi phí tuần tự hóa nặng nề khi liên lạc giữa các quá trình.
- **Hỗ trợ gốc**: được tích hợp vào Node.js 12+ mà không cần thêm phần phụ thuộc nặng nề.
- **Ngữ cảnh thống nhất**: việc gỡ lỗi và ghi nhật ký thuận tiện hơn so với các tiến trình con.

### Mức độ chi tiết của nhiệm vụ: Nên phân chia các nhiệm vụ làm rối mã nguồn như thế nào?

Thật khó để song song hóa việc làm rối mã nguồn của một tệp gói JS khổng lồ vì mã có các phần phụ thuộc, nhưng đầu ra của bản dựng Vite bao gồm nhiều **khối**. Điều đó cho chúng ta một ranh giới song song tự nhiên:

- **Tính độc lập**: sau khi đóng gói Vite, các phần phụ thuộc giữa các phần khác nhau đã được tách riêng nên chúng có thể được xử lý song song một cách an toàn.
- **Mức độ chi tiết phù hợp**: các dự án thường có 10-30 phần, đây là một quy mô tuyệt vời để lập kế hoạch song song.
- **Tích hợp dễ dàng**: `generateBundle` hook trong plugin Vite cho phép chúng tôi chặn và xử lý các đoạn này trước khi tệp được phát ra.

### Thiết kế kiến trúc

Chúng tôi đã thiết kế một hệ thống xử lý song song với bốn thành phần cốt lõi:

1. **Bộ chia tác vụ**: lặp lại đối tượng gói của Vite, lọc ra các tệp không cần làm xáo trộn chẳng hạn như các khối của nhà cung cấp và tạo hàng đợi tác vụ.
2. **Trình quản lý nhóm công nhân**: quản lý vòng đời của Công nhân và xử lý việc phân phối, tái chế nhiệm vụ và thử lại khi không thành công.
3. **Trình báo cáo tiến độ**: kết quả đầu ra xây dựng tiến độ theo thời gian thực để giảm bớt sự lo lắng khi chờ đợi.
4. **ObfuscationWorker**: luồng Worker thực sự thực hiện logic che giấu.

## Giải pháp: Triển khai thực tế

Dựa trên phân tích ở trên, chúng tôi bắt đầu triển khai hệ thống che giấu song song này.

### 1. Định cấu hình Plugin Vite

Đầu tiên, chúng tôi đã tích hợp plugin làm rối mã nguồn song song trong `vite.config.ts`. Cấu hình rất đơn giản. Bạn chỉ cần chỉ định số lượng Công nhân và quy tắc che giấu.

```typescript
import { defineConfig } from 'vite'
import { parallelJavascriptObfuscator } from './buildTools/plugin'

export default defineConfig(({ mode }) => {
  const isProduction = mode === 'production'
  
  return {
    build: {
      rollupOptions: {
        ...(isProduction
          ? {
              plugins: [
                parallelJavascriptObfuscator({
                  enabled: true,
                  // Automatically adjust based on CPU core count; leave one core for the main thread
                  workerCount: 4, 
                  retryAttempts: 3,
                  fallbackToMainThread: true, // Automatically degrade to single-thread mode on failure
                  // Filter out vendor chunks; third-party libraries usually do not need obfuscation
                  isVendorChunk: (fileName: string) => fileName.includes('vendor-'),
                  obfuscationConfig: {
                    compact: true,
                    controlFlowFlattening: true,
                    deadCodeInjection: true,
                    disableConsoleOutput: true,
                    // ... more obfuscation options
                  },
                }),
              ],
            }
          : {}),
      },
    },
  }
})
```

### 2. Triển khai Logic công nhân

Worker là đơn vị thực hiện các nhiệm vụ. Chúng ta cần xác định rõ ràng cấu trúc dữ liệu đầu vào và đầu ra.

**Lưu ý**: mặc dù mã ở đây đơn giản nhưng có một số cạm bẫy cần lưu ý, chẳng hạn như kiểm tra xem liệu `parentPort` là null và xử lý lỗi đúng cách. Trong quá trình triển khai HagiCode, chúng tôi nhận thấy rằng một số mẫu cú pháp ES6 đặc biệt nhất định có thể khiến bộ mã hóa bị lỗi, vì vậy chúng tôi đã thêm `try-catch` bảo vệ.

```typescript
import { parentPort } from 'worker_threads'
import javascriptObfuscator from 'javascript-obfuscator'

export interface ObfuscationTask {
  chunkId: string
  code: string
  config: any
}

export interface ObfuscationResult {
  chunkId: string
  obfuscatedCode: string
  error?: string
}

// Listen for tasks sent from the main thread
if (parentPort) {
  parentPort.on('message', async (task: ObfuscationTask) => {
    try {
      // Perform obfuscation
      const obfuscated = javascriptObfuscator.obfuscate(task.code, task.config)
      const result: ObfuscationResult = {
        chunkId: task.chunkId,
        obfuscatedCode: obfuscated.getObfuscatedCode(),
      }
      // Send the result back to the main thread
      parentPort?.postMessage(result)
    } catch (error) {
      // Handle exceptions so one Worker crash does not block the whole build
      const result: ObfuscationResult = {
        chunkId: task.chunkId,
        obfuscatedCode: '',
        error: error instanceof Error ? error.message : 'Unknown error',
      }
      parentPort?.postMessage(result)
    }
  })
}
```

### 3. Quản lý nhóm công nhân

Đây là cốt lõi của toàn bộ giải pháp. Chúng ta cần duy trì một nhóm Công nhân có kích thước cố định và lên lịch các tác vụ bằng chiến lược **FIFO (nhập trước, xuất trước)**.

```typescript
import { Worker } from 'worker_threads'
import os from 'os'

export class WorkerPool {
  private workers: Worker[] = []
  private taskQueue: Array<{
    task: ObfuscationTask
    resolve: (result: ObfuscationResult) => void
    reject: (error: Error) => void
  }> = []
  
  constructor(options: WorkerPoolOptions = {}) {
    // Default to core count - 1 so the main thread still has some breathing room
    const workerCount = options.workerCount ?? Math.max(1, (os.cpus().length || 4) - 1)
    
    for (let i = 0; i < workerCount; i++) {
      this.createWorker()
    }
  }

  private createWorker() {
    const worker = new Worker('./worker.ts')
    
    worker.on('message', (result) => {
      // After one task finishes, take the next task from the queue
      const nextTask = this.taskQueue.shift()
      if (nextTask) {
        this.dispatchTask(worker, nextTask)
      } else {
        // If there are no pending tasks, mark the Worker as idle
        this.activeWorkers.delete(worker)
      }
    })
    
    this.workers.push(worker)
  }

  // Submit a task to the pool
  public runTask(task: ObfuscationTask): Promise<ObfuscationResult> {
    return new Promise((resolve, reject) => {
      const job = { task, resolve, reject }
      const idleWorker = this.workers.find(w => !this.activeWorkers.has(w))
      
      if (idleWorker) {
        this.dispatchTask(idleWorker, job)
      } else {
        this.taskQueue.push(job)
      }
    })
  }

  private dispatchTask(worker: Worker, job: any) {
    this.activeWorkers.set(worker, job.task)
    worker.postMessage(job.task)
  }
}
```

### 4. Báo cáo tiến độ

Chờ đợi là điều đau đớn, nhất là khi bạn không biết sẽ mất bao lâu nữa. Vì vậy, chúng tôi đã thêm một trình báo cáo tiến độ đơn giản để cung cấp phản hồi theo thời gian thực về trạng thái hiện tại.

```typescript
export class ProgressReporter {
  private completed = 0
  private readonly total: number
  private readonly startTime: number

  constructor(total: number) {
    this.total = total
    this.startTime = Date.now()
  }

  increment(): void {
    this.completed++
    this.report()
  }

  private report(): void {
    const now = Date.now()
    const elapsed = now - this.startTime
    const percentage = (this.completed / this.total) * 100
    
    // Simple ETA estimate
    const avgTimePerChunk = elapsed / this.completed
    const remaining = (this.total - this.completed) * avgTimePerChunk

    console.log(
      `[Parallel Obfuscation] ${this.completed}/${this.total} chunks completed (${percentage.toFixed(1)}%) | ETA: ${(remaining / 1000).toFixed(1)}s`
    )
  }
}
```

## Thực hành: Kết quả và bài học kinh nghiệm

Sau khi triển khai giải pháp này, hiệu suất xây dựng của dự án HagiCode được cải thiện ngay lập tức.

### Dữ liệu điểm chuẩn hiệu suất

Chúng tôi đã thử nghiệm trong môi trường sau:
- CPU: Intel Core i7-12700K (12 nhân/20 luồng)
- RAM: 32GB DDR4
- Node.js: v18.17.0
- Hệ điều hành: Ubuntu 22.04

**So sánh kết quả**:

- **4 Công nhân**: 55 giây (**cải thiện 53%**)
- **8 Công nhân**: 48 giây (**cải thiện 60%**)
- **12 Công nhân**: 45 giây (**cải thiện 62%**)

Như bạn có thể thấy, mức tăng không tuyến tính. Khi số lượng Công nhân vượt quá 8, sự cải thiện sẽ trở nên nhỏ hơn. Điều này chủ yếu bị hạn chế bởi sự đồng đều trong phân bổ tác vụ và tắc nghẽn băng thông bộ nhớ.

### Các vấn đề và giải pháp chung

Trong quá trình sử dụng HagiCode trong thế giới thực, chúng tôi cũng gặp phải một số cạm bẫy, vì vậy chúng tôi đưa ra đây để tham khảo:

**Q1: Thời gian xây dựng không giảm nhiều và thậm chí còn chậm hơn?**
- **Lý do**: việc tạo Công nhân có chi phí riêng hoặc có quá nhiều Công nhân được định cấu hình, gây ra việc chuyển đổi ngữ cảnh thường xuyên.
- **Giải pháp**: chúng tôi khuyên bạn nên đặt Số lượng công nhân thành `CPU core count - 1`. Ngoài ra, hãy kiểm tra xem có đoạn nào đặc biệt lớn không, ví dụ `> 5MB`. Loại file “quái vật” đó sẽ trở thành nút thắt cổ chai, vì vậy bạn có thể cần phải tối ưu hóa chiến lược chia tách mã của mình.

**Q2: Đôi khi công nhân gặp sự cố và gây ra lỗi xây dựng?**
- **Lý do**: một số mẫu cú pháp mã đặc biệt có thể gây ra lỗi nội bộ bên trong bộ mã hóa.
- **Giải pháp**: chúng tôi đã triển khai **cơ chế phân hủy tự động**. Khi một Worker đạt đến ngưỡng lỗi, plugin sẽ tự động quay trở lại chế độ một luồng để đảm bảo quá trình xây dựng không dừng lại. Đồng thời, nó ghi lại tên file gây ra lỗi để có thể sửa chữa sau này.

**Q3: Mức sử dụng bộ nhớ quá cao (OOM)?**
- **Lý do**: mỗi Công nhân cần có không gian bộ nhớ riêng để tải bộ làm xáo trộn và phân tích cú pháp AST.
- **Giải pháp**:
  - Giảm số lượng công nhân.
  - Tăng giới hạn bộ nhớ Node.js: `NODE_OPTIONS="--max-old-space-size=4096" npm run build`.
  - Đảm bảo Công nhân không giữ các tham chiếu không cần thiết đến các đối tượng lớn.
