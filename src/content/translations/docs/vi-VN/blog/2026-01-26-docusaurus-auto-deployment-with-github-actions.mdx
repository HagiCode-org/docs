## Thêm hỗ trợ triển khai tự động trang GitHub vào HagiCode

> Dự án này ban đầu có tên mã là PCode và hiện tại đã được đổi tên chính thức thành HagiCode. Bài viết này trình bày cách chúng tôi triển khai triển khai trang web tĩnh tự động để việc xuất bản nội dung trở nên dễ dàng như uống nước.

## Bối cảnh / Giới thiệu

Trong quá trình phát triển HagiCode, chúng tôi gặp phải một vấn đề rất thực tế: khi tài liệu và đề xuất của chúng tôi không ngừng phát triển, việc quản lý và trình bày hiệu quả nội dung đó ngày càng trở nên cấp thiết. Chúng tôi đã quyết định sử dụng Trang GitHub để lưu trữ trang web tĩnh của mình, nhưng việc xây dựng và triển khai nó theo cách thủ công quá cồng kềnh. Mọi thay đổi đều yêu cầu xây dựng, đóng gói cục bộ và sau đó đẩy thủ công tới `gh-pages` chi nhánh. Điều đó không chỉ không hiệu quả mà còn dễ mắc sai lầm.

Để giải quyết vấn đề này, chủ yếu là vì chúng tôi muốn tiết kiệm công sức nên chúng tôi cần một quy trình triển khai tự động. Bài viết này ghi lại chi tiết cách chúng tôi thêm hỗ trợ triển khai tự động dựa trên Hành động GitHub vào dự án HagiCode, để chúng tôi có thể tập trung vào việc viết nội dung và để tự động hóa xử lý phần còn lại.

## Giới thiệu về HagiCode

> Này, hãy để chúng tôi giới thiệu những gì chúng tôi đang xây dựng

Chúng tôi đang xây dựng **HagiCode** — một trợ lý mã hóa được hỗ trợ bởi AI giúp việc phát triển trở nên thông minh hơn, dễ dàng hơn và thú vị hơn.

**Thông minh hơn** — AI hỗ trợ trong toàn bộ hành trình, từ ý tưởng đến mã, giúp tăng hiệu quả phát triển lên gấp bội. **Dễ dàng hơn** — Hoạt động đồng thời đa luồng giúp tận dụng tối đa tài nguyên và giữ cho quy trình phát triển diễn ra suôn sẻ. **Thú vị hơn** — Gamification và hệ thống thành tích giúp việc viết mã bớt tẻ nhạt hơn và bổ ích hơn.

Dự án đang được lặp lại nhanh chóng. Nếu bạn quan tâm đến viết kỹ thuật, quản lý kiến thức hoặc phát triển được hỗ trợ bởi AI, vui lòng kiểm tra chúng tôi trên [GitHub](https://github.com/HagiCode-org/site)~

## Phân tích mục tiêu

Trước khi bắt tay vào thực hiện, trước tiên chúng ta cần làm rõ nhiệm vụ này thực sự phải hoàn thành là gì. Rốt cuộc, việc mài rìu không làm trì hoãn việc chặt củi.

### Yêu cầu cốt lõi

1.  **Xây dựng tự động**: Tự động kích hoạt quy trình xây dựng khi mã được đẩy lên `main` chi nhánh.
2.  **Triển khai tự động**: Sau khi xây dựng thành công, hãy tự động triển khai các tệp tĩnh được tạo lên Trang GitHub.
3.  **Tính nhất quán của môi trường**: Đảm bảo môi trường CI phù hợp với môi trường xây dựng cục bộ để tránh tình trạng xấu hổ điển hình là "nó hoạt động cục bộ nhưng không hoạt động trực tuyến".

### Lựa chọn kỹ thuật

Vì HagiCode được xây dựng trên Docusaurus, một trình tạo trang tĩnh React rất phổ biến, nên chúng tôi có thể sử dụng GitHub Actions để đạt được mục tiêu này.

## Định cấu hình quy trình làm việc hành động GitHub

GitHub Actions là dịch vụ CI/CD do GitHub cung cấp. Bằng cách xác định các tệp quy trình công việc ở định dạng YAML bên trong kho lưu trữ, chúng tôi có thể tùy chỉnh tất cả các loại tác vụ tự động hóa.

### Tạo tệp quy trình công việc

Chúng ta cần tạo một tệp cấu hình mới trong `.github/workflows` thư mục ở thư mục gốc của dự án, ví dụ `deploy.yml`. Nếu thư mục chưa tồn tại, hãy tạo nó trước.

Logic cốt lõi của tệp cấu hình này như sau:

1.  **Điều kiện kích hoạt**: Lắng nghe `push` sự kiện trên `main` chi nhánh.
2.  **Môi trường thời gian chạy**: Bản phát hành Ubuntu mới nhất.
3.  **Các bước xây dựng**:
    -   Kiểm tra mã
    -   Cài đặt Node.js
    -   Cài đặt phụ thuộc (`npm install`)
    -   Xây dựng các tập tin tĩnh (`npm run build`)
4.  **Bước triển khai**: Sử dụng bản chính thức `action-gh-pages` hành động để đẩy đầu ra bản dựng lên `gh-pages` chi nhánh.

### Mã cấu hình chính

Dưới đây là mẫu cấu hình cuối cùng chúng tôi đã áp dụng:

```yaml
name: Deploy to GitHub Pages

# Trigger condition: run when code is pushed to the main branch
on:
  push:
    branches:
      - main
    # Add path filters if needed, for example only build when docs change
    # paths:
    #   - 'docs/**'
    #   - 'package.json'

# Set permissions, which is important for GitHub Pages deployment
permissions:
  contents: read
  pages: write
  id-token: write

# Concurrency control: cancel older builds for the same branch
concurrency:
  group: "pages"
  cancel-in-progress: false

jobs:
  build:
    runs-on: ubuntu-latest
    steps:
      - name: Checkout
        uses: actions/checkout@v4
        # Note: fetch-depth: 0 is required, otherwise the build version may be inaccurate
        with:
          fetch-depth: 0

      - name: Setup Node
        uses: actions/setup-node@v4
        with:
          node-version: 20 # Keep this consistent with the local development environment
          cache: 'npm'     # Enabling cache can speed up the build process

      - name: Install dependencies
        run: npm ci
        # Use npm ci instead of npm install because it is faster and stricter, which makes it better for CI

      - name: Build website
        run: npm run build
        env:
          # Configure environment variables here if your site build requires them
          # NODE_ENV: production
          # PUBLIC_URL: /your-repo-name
          
      - name: Upload artifact
        uses: actions/upload-pages-artifact@v3
        with:
          path: ./build # Default Docusaurus output directory

  deploy:
    environment:
      name: github-pages
      url: ${{ steps.deployment.outputs.page_url }}
    runs-on: ubuntu-latest
    needs: build
    steps:
      - name: Deploy to GitHub Pages
        id: deployment
        uses: actions/deploy-pages@v4
```

## Những cạm bẫy thường gặp trong quá trình thực hiện

Trong thực tế, chúng tôi gặp phải một số vấn đề. Tôi chia sẻ chúng ở đây với hy vọng những người khác có thể tránh được hoặc ít nhất là chuẩn bị trước các giải pháp.

### 1. Các vấn đề về quyền của mã thông báo GitHub

Khi chúng tôi thiết lập mọi thứ lần đầu tiên, việc triển khai liên tục không thành công với lỗi 403 (Bị cấm). Sau khi tìm hiểu kỹ, chúng tôi thấy rằng mặc định của GitHub `GITHUB_TOKEN` không có quyền viết lên Trang.

**Giải pháp**: Trong kho lưu trữ `Settings` -> `Actions` -> `General` -> `Workflow permissions`, hãy nhớ chọn **"Quyền đọc và ghi"**.

### 2. Đường dẫn thư mục bản dựng sai

Docusaurus đặt các tệp tĩnh được xây dựng trong `build` thư mục theo mặc định. Tuy nhiên, một số dự án có thể được cấu hình khác nhau. Ví dụ: Tạo ứng dụng React mặc định là `build`, trong khi Vite mặc định là `dist`. Nếu Hành động báo cáo rằng nó không thể tìm thấy tệp đầu ra, hãy kiểm tra đường dẫn đầu ra trong `docusaurus.config.js`.

### 3. Các vấn đề về đường dẫn phụ

Nếu kho lưu trữ của bạn không phải là trang chủ của người dùng, nghĩa là nó không phải `username.github.io`, mà thay vào đó là một trang dự án như `username.github.io/project-name`, thì bạn cần phải cấu hình `baseUrl`.

trong `docusaurus.config.js`:

```javascript
module.exports = {
  // ...
  url: 'https://hagicode.com', // Your Hagicode URL
  baseUrl: '/', // Deploy from the root path
  // ...
};
```
Chi tiết này rất dễ bị bỏ qua. Nếu được định cấu hình không chính xác, trang có thể tải dưới dạng màn hình trống vì không thể giải quyết được đường dẫn nội dung.

## Xác minh kết quả

Sau khi định cấu hình mọi thứ và đẩy mã, chúng ta có thể đi tới tab **Hành động** của kho GitHub và xem chương trình.

Bạn sẽ thấy một vòng tròn màu vàng khi quy trình làm việc đang chạy và khi nó chuyển sang màu xanh lục nghĩa là đã thành công. Nếu nó chuyển sang màu đỏ, hãy nhấp vào và kiểm tra nhật ký. Trong hầu hết các trường hợp, bạn có thể theo dõi sự cố ở dưới đó, thường là lỗi đánh máy hoặc cấu hình đường dẫn không chính xác.

Khi quá trình xây dựng thành công, hãy truy cập `https://<your-username>.github.io/<repository-name>/` và bạn sẽ thấy trang web mới được triển khai.

## Tóm tắt

Bằng cách giới thiệu GitHub Actions, chúng tôi đã tự động triển khai thành công trang tài liệu HagiCode. Điều này không chỉ tiết kiệm thời gian dành cho các thao tác thủ công trước đây mà quan trọng hơn là tiêu chuẩn hóa quy trình phát hành. Bây giờ, bất kể thành viên nào trong nhóm cập nhật tài liệu, sau khi thay đổi được hợp nhất vào `main`, nội dung mới nhất sẽ được phát hành sau đó vài phút.

**Lợi ích cốt lõi**:
- **Hiệu quả cao hơn**: Chúng tôi đã chuyển từ "đóng gói thủ công và tải lên thủ công" sang "phát hành mã tương đương".
- **Ít lỗi hơn**: Chúng tôi đã loại bỏ khả năng xảy ra lỗi do con người trong quá trình vận hành thủ công.
- **Trải nghiệm nhà phát triển tốt hơn**: Nhà phát triển có thể tập trung hơn vào chất lượng nội dung thay vì bị phân tâm bởi các bước triển khai tẻ nhạt.

Mặc dù việc định cấu hình CI/CD lúc đầu hơi rắc rối, đặc biệt là với tất cả các quyền và chi tiết đường dẫn, nhưng đây là khoản đầu tư một lần với lợi nhuận dài hạn lớn. Tôi thực sự khuyên mọi dự án trang web tĩnh nên áp dụng quy trình làm việc tự động tương tự.
