# Giới thiệu về OpenSpec

OpenSpec là một công cụ phát triển hướng đặc tả giúp các nhóm quản lý quy trình phát triển phần mềm thông qua các đề xuất và đặc tả có cấu trúc. Bài viết này cung cấp một giới thiệu toàn diện về các khái niệm cốt lõi, quy trình làm việc và cách sử dụng OpenSpec.

## OpenSpec là gì?

OpenSpec là một công cụ dòng lệnh được sử dụng để quản lý các đề xuất (Proposals), đặc tả (Specifications) và thay đổi (Changes) trong quy trình phát triển phần mềm. Nó chuẩn hóa quy trình phát triển, đảm bảo rằng mỗi tính năng đều trải qua thiết kế và đánh giá kỹ lưỡng từ ý tưởng đến triển khai.

### Nguyên tắc cốt lõi

OpenSpec dựa trên các nguyên tắc sau:

- **Đặc tả là ưu tiên**: Định nghĩa đặc tả rõ ràng trước khi viết mã
- **Hợp tác có cấu trúc**: Thúc đẩy hợp tác nhóm thông qua định dạng tài liệu chuẩn hóa
- **Có thể truy vết**: Ghi lại toàn bộ quy trình từ đề xuất đến triển khai
- **Đảm bảo chất lượng**: Đảm bảo mã phù hợp với ý định thiết kế bằng cách xác thực đặc tả

### Các tính năng chính

OpenSpec cung cấp các tính năng chính sau:

1. **Quản lý đề xuất**: Tạo, xem xét và theo dõi đề xuất tính năng
2. **Định nghĩa đặc tả**: Viết và duy trì tài liệu đặc tả kỹ thuật
3. **Theo dõi thay đổi**: Ghi lại toàn bộ lịch sử các thay đổi đã được rà soát
4. **Xác thực đặc tả**: Đảm bảo các thay đổi mã phù hợp với các đặc tả đã định nghĩa trước
5. **Hợp tác nhóm**: Hỗ trợ quy trình phát triển và rà soát với nhiều người tham gia

## Quy trình làm việc của OpenSpec

Quy trình làm việc tiêu biểu của OpenSpec bao gồm các giai đoạn sau:

### 1. Tạo đề xuất (Proposal)

Khi có tính năng mới hoặc yêu cầu thay đổi đáng kể, trước tiên tạo một đề xuất:

```bash
openspec proposal create
```

Đề xuất bao gồm:
- **Mô tả vấn đề**: Vấn đề hoặc yêu cầu cần giải quyết
- **Giải pháp**: Tổng quan về giải pháp đề xuất
- **Phạm vi ảnh hưởng**: Tác động tới hệ thống hiện tại
- **Giải pháp thay thế**: Các phương án khác đã xem xét

### 2. Viết đặc tả (Specification)

Sau khi đề xuất được chấp thuận, viết đặc tả kỹ thuật chi tiết:

```bash
openspec spec create
```

Đặc tả bao gồm:
- **Thiết kế kỹ thuật**: Kế hoạch chi tiết cho việc triển khai kỹ thuật
- **Định nghĩa giao diện**: Thiết kế giao diện như API, cấu trúc dữ liệu, v.v.
- **Chi tiết thực hiện**: Logic chính và các thuật toán
- **Kế hoạch kiểm thử**: Cách xác minh tính đúng đắn của triển khai

### 3. Tạo thay đổi (Change)

Khi bắt đầu triển khai, tạo một thay đổi để theo dõi quá trình phát triển:

```bash
openspec change create
```

Thay đổi bao gồm:
- **Đặc tả liên quan**: Đặc tả được thực hiện bởi thay đổi này
- **Nhiệm vụ triển khai**: Danh sách các nhiệm vụ phát triển cụ thể
- **Theo dõi tiến độ**: Trạng thái hoàn thành các nhiệm vụ
- **Thay đổi mã**: Các sửa đổi mã thực tế

### 4. Xác thực và lưu trữ

Sau khi phát triển hoàn tất, xác thực xem thay đổi có tuân thủ đặc tả hay không, sau đó lưu trữ:

```bash
openspec change validate
openspec change archive
```

## Cấu trúc thư mục OpenSpec

OpenSpec tạo thư mục `openspec/` trong thư mục gốc của dự án với cấu trúc sau:

```
openspec/
├── proposals/           # Thư mục đề xuất
│   ├── PROPOSAL-001.md
│   └── PROPOSAL-002.md
├── specs/              # Thư mục đặc tả
│   ├── SPEC-001.md
│   └── SPEC-002.md
├── changes/            # Thư mục thay đổi
│   ├── feature-name/
│   │   ├── proposal.md
│   │   ├── design.md
│   │   ├── specs.md
│   │   └── tasks.md
│   └── another-feature/
└── archived/           # Các thay đổi đã lưu trữ
    └── feature-name/
```

### Giải thích các thư mục

- **proposals/**: Lưu tất cả tài liệu đề xuất, mỗi đề xuất là một tệp Markdown riêng
- **specs/**: Lưu tài liệu đặc tả kỹ thuật tương ứng với các đề xuất
- **changes/**: Lưu các thay đổi đang tiến hành, mỗi thay đổi ở một thư mục con riêng
- **archived/**: Lưu các thay đổi đã hoàn thành và được lưu trữ

## Lệnh chính

### Lệnh liên quan tới đề xuất

```bash
# Tạo đề xuất mới
openspec proposal create

# Liệt kê tất cả đề xuất
openspec proposal list

# Xem chi tiết đề xuất
openspec proposal view <proposal-id>

# Phê duyệt đề xuất
openspec proposal approve <proposal-id>
```

### Lệnh liên quan tới đặc tả

```bash
# Tạo đặc tả mới
openspec spec create

# Liệt kê tất cả đặc tả
openspec spec list

# Xem chi tiết đặc tả
openspec spec view <spec-id>

# Cập nhật đặc tả
openspec spec update <spec-id>
```

### Lệnh liên quan tới thay đổi

```bash
# Tạo thay đổi mới
openspec change create

# Liệt kê tất cả thay đổi
openspec change list

# Xem chi tiết thay đổi
openspec change view <change-name>

# Thêm nhiệm vụ
openspec change add-task <change-name>

# Xác thực thay đổi
openspec change validate <change-name>

# Lưu trữ thay đổi
openspec change archive <change-name>
```

### Lệnh chung

```bash
# Hiển thị phiên bản
openspec --version

# Hiển thị trợ giúp
openspec --help

# Hiển thị trợ giúp cho lệnh cụ thể
openspec <command> --help
```

## Sử dụng OpenSpec trong dự án HagiCode

### Tích hợp dự án

Trong dự án HagiCode, OpenSpec được tích hợp vào quy trình phát triển:

1. **Kiểm tra khi khởi tạo**: Trình hướng dẫn khởi tạo kiểm tra xem OpenSpec đã được cài đặt chưa
2. **Phát triển theo đề xuất**: Tất cả tính năng mới phải trải qua quy trình đề xuất của OpenSpec
3. **Đánh giá mã**: Các thay đổi phải tuân thủ đặc tả OpenSpec để được gộp
4. **Đồng bộ hóa tài liệu**: Tài liệu OpenSpec được giữ đồng bộ với việc triển khai mã

### Yêu cầu phiên bản

Dự án HagiCode yêu cầu sử dụng phiên bản **OpenSpec >=1.0.0 <2.0.0**.

Lệnh cài đặt:

```bash
npm install -g @fission-ai/openspec@1
```

### Ví dụ quy trình làm việc

Dưới đây là ví dụ đầy đủ về cách sử dụng OpenSpec trong dự án HagiCode:

#### Bước 1: Tạo đề xuất

```bash
cd /path/to/hagicode-project
openspec proposal create
```

Trong giao diện tương tác, nhập:
- Tiêu đề đề xuất: Thêm tính năng xác thực người dùng
- Mô tả vấn đề: Hệ thống cần đăng nhập người dùng và quản lý quyền
- Giải pháp: Triển khai hệ thống xác thực dựa trên JWT

#### Bước 2: Viết đặc tả

Sau khi đề xuất được chấp thuận, tạo đặc tả chi tiết:

```bash
openspec spec create
```

Định nghĩa trong đặc tả:
- Thiết kế luồng xác thực
- Định nghĩa API
- Cấu trúc bảng cơ sở dữ liệu
- Các cân nhắc về bảo mật

#### Bước 3: Tạo thay đổi

Bắt đầu triển khai:

```bash
openspec change create
```

Nhập tên thay đổi: `add-user-authentication`

#### Bước 4: Thêm nhiệm vụ

```bash
openspec change add-task add-user-authentication
```

Thêm các nhiệm vụ cụ thể:
- Triển khai điểm cuối đăng ký người dùng
- Triển khai điểm cuối đăng nhập
- Triển khai middleware JWT
- Viết unit tests

#### Bước 5: Triển khai và xác thực

Sau khi mã hoàn tất, xác thực thay đổi:

```bash
openspec change validate add-user-authentication
```

#### Bước 6: Lưu trữ

Sau khi xác thực thành công, lưu trữ thay đổi:

```bash
openspec change archive add-user-authentication
```

## Các thực hành tốt nhất

### Viết đề xuất

1. **Xác định vấn đề rõ ràng**: Mô tả rõ vấn đề cần giải quyết; tránh mô tả mơ hồ
2. **Xem xét các phương án khác**: Liệt kê và so sánh các giải pháp khác nhau
3. **Đánh giá tác động**: Phân tích tác động và rủi ro đối với hệ thống hiện có
4. **Ngắn gọn**: Đề xuất nên ngắn gọn và rõ ràng; bỏ chi tiết sang đặc tả

### Viết đặc tả

1. **Chi tiết nhưng không thừa**: Cung cấp đủ chi tiết nhưng tránh over-design
2. **Sử dụng sơ đồ**: Dùng flowchart, sơ đồ kiến trúc, v.v. để minh họa
3. **Định nghĩa giao diện**: Xác định rõ tất cả giao diện công khai và cấu trúc dữ liệu
4. **Xem xét các trường hợp rìa**: Liệt kê tình huống ngoại lệ và xử lý lỗi

### Quản lý thay đổi

1. **Lặp lại bằng các bước nhỏ**: Chia các tính năng lớn thành nhiều thay đổi nhỏ
2. **Cập nhật kịp thời**: Giữ trạng thái nhiệm vụ và tài liệu đồng bộ
3. **Đánh giá mã**: Thực hiện code review sau khi thay đổi hoàn thành
4. **Lưu trữ kịp thời**: Lưu trữ các thay đổi khi hoàn thành và giữ workspace sạch sẽ

### Hợp tác nhóm

1. **Đánh giá định kỳ**: Thường xuyên rà soát đề xuất và đặc tả để đảm bảo sự đồng thuận trong nhóm
2. **Tài liệu trước tiên**: Hoàn thiện tài liệu trước khi bắt đầu mã hóa
3. **Duy trì liên lạc**: Ghi lại các cuộc thảo luận và quyết định trong đề xuất và đặc tả
4. **Quản lý phiên bản**: Bao gồm tài liệu OpenSpec trong hệ thống quản lý phiên bản

## Tích hợp với công cụ khác

### Tích hợp Git

OpenSpec tích hợp mượt mà với Git:

```bash
# Tài liệu OpenSpec nên được bao gồm trong version control
git add openspec/
git commit -m "Add OpenSpec proposal for feature X"
```

### Tích hợp CI/CD

Xác thực các đặc tả OpenSpec trong pipeline CI/CD:

```yaml
# .github/workflows/openspec.yml
name: OpenSpec Validation
on: [pull_request]
jobs:
  validate:
    runs-on: ubuntu-latest
    steps:
      - uses: actions/checkout@v2
      - name: Install OpenSpec
        run: npm install -g @fission-ai/openspec@1
      - name: Validate changes
        run: openspec change validate
```

### Tích hợp IDE

Nên sử dụng trình soạn thảo hỗ trợ Markdown để chỉnh sửa tài liệu OpenSpec:
- VS Code + plugin Markdown
- JetBrains IDEs + hỗ trợ Markdown
- Vim/Neovim + plugin Markdown

## Khắc phục sự cố

### Các sự cố thường gặp

#### Sự cố: Lệnh OpenSpec không tìm thấy

**Giải pháp**:
1. Xác nhận OpenSpec đã được cài đặt: `npm list -g @fission-ai/openspec`
2. Kiểm tra biến môi trường PATH có chứa thư mục bin toàn cục của npm không
3. Cài lại: `npm install -g @fission-ai/openspec@1`

#### Sự cố: Không thể tạo đề xuất hoặc thay đổi

**Giải pháp**:
1. Xác nhận thư mục hiện tại là thư mục gốc của dự án
2. Kiểm tra xem thư mục `openspec/` tồn tại hay không; nếu không, tạo nó
3. Xác nhận bạn có quyền ghi

#### Sự cố: Xác thực thất bại

**Giải pháp**:
1. Kiểm tra xem tài liệu trong thư mục thay đổi đã đầy đủ chưa
2. Xác nhận tất cả nhiệm vụ cần thiết đã hoàn thành
3. Xem thông tin lỗi chi tiết: `openspec change validate <change-name> --verbose`

#### Sự cố: Lưu trữ thất bại

**Giải pháp**:
1. Đảm bảo thay đổi đã vượt qua xác thực
2. Kiểm tra xem thư mục `openspec/archived/` có tồn tại không
3. Xác nhận không có thay đổi mã nào chưa commit

## Sử dụng nâng cao

### Mẫu tùy chỉnh

OpenSpec hỗ trợ mẫu tùy chỉnh cho đề xuất và đặc tả:

```bash
# Tạo thư mục mẫu trong thư mục gốc của dự án
mkdir -p .openspec/templates

# Tạo mẫu đề xuất tùy chỉnh
cat > .openspec/templates/proposal.md << 'EOF'
# Proposal: {{title}}

## Problem Statement
{{problem}}

## Proposed Solution
{{solution}}

## Custom Section
Your custom content here
EOF
```

### Tệp cấu hình

Tạo tệp cấu hình `.openspec.json` trong thư mục gốc của dự án:

```json
{
  "version": "1.0",
  "templates": {
    "proposal": ".openspec/templates/proposal.md",
    "spec": ".openspec/templates/spec.md"
  },
  "validation": {
    "requireTests": true,
    "requireDocs": true
  },
  "hooks": {
    "pre-archive": "npm test"
  }
}
```

### Hooks

OpenSpec hỗ trợ chạy script tùy chỉnh trước và sau các thao tác cụ thể:

```json
{
  "hooks": {
    "pre-create": "scripts/check-permissions.sh",
    "post-archive": "scripts/notify-team.sh"
  }
}
```
