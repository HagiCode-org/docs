# Bevezetés az OpenSpec-be

Az OpenSpec egy specifikációvezérelt fejlesztési eszköz, amely segíti a csapatokat abban, hogy strukturált javaslatok és specifikációk révén kezeljék a szoftverfejlesztési folyamatot. Ez a dokumentum átfogó bevezetést nyújt az OpenSpec alapvető fogalmaiba, munkafolyamataiba és használati módjaiba.

## Mi az OpenSpec?

Az OpenSpec egy parancssori eszköz, amelyet a fejlesztési folyamat során a javaslatok (Proposals), a specifikációk (Specifications) és a változtatások (Changes) kezelésére használnak. Standardizálja a fejlesztési folyamatot, és biztosítja, hogy minden funkció a tervtől a megvalósításig alapos tervezésen és felülvizsgálaton menjen keresztül.

### Alapelvek

Az OpenSpec az alábbi alapelvekre épül:

- **Specifikáció előrejár**: Határozzunk meg világos specifikációkat, mielőtt kódot írunk
- **Strukturált együttműködés**: Támogassa a csapatmunkát szabványos dokumentumformátumokon keresztül
- **Követhetőség**: Rögzítse a teljes folyamatot a javaslattól a megvalósításig
- **Minőségbiztosítás**: A specifikációk érvényesítésével biztosítsa, hogy a kód megfelel a tervezési céloknak

### Főbb funkciók

Az OpenSpec a következő fő funkciókat kínálja:

1. **Javaslatkezelés**: Funkciójavaslatok létrehozása, felülvizsgálata és követése
2. **Specifikációk definiálása**: Műszaki specifikációs dokumentumok írása és karbantartása
3. **Változások nyomon követése**: A felülvizsgált változások teljes előzményeinek rögzítése
4. **Specifikáció-ellenőrzés**: Biztosítsa, hogy a kódváltoztatások megfeleljenek az előre definiált specifikációknak
5. **Csapatmunka támogatása**: Több résztvevős fejlesztési- és felülvizsgálati folyamatok támogatása

## OpenSpec munkafolyamata

Az OpenSpec tipikus munkafolyamata a következő lépésekből áll:

### 1. Javaslat létrehozása (Proposal)

Ha új funkcióra vagy jelentős változtatásra van szükség, először hozzon létre egy javaslatot:

```bash
openspec proposal create
```

A javaslat tartalmazza:
- **A probléma leírása**: A megoldandó probléma vagy követelmény
- **A megoldás**: A javasolt megoldás áttekintése
- **Hatás körfogata**: A meglévő rendszerre gyakorolt hatás
- **Alternatívák**: Egyéb mérlegelt megoldások

### 2. Specifikáció írása (Specification)

A javaslat jóváhagyása után írjon részletes műszaki specifikációt:

```bash
openspec spec create
```

A specifikáció tartalmazza:
- **Műszaki terv**: Részletes technikai megvalósítási terv
- **Interfészdefiníciók**: API-k, adatszerkezetek és egyéb interfészek tervezése
- **Megvalósítási részletek**: A kulcsfontosságú megvalósítási logika és algoritmusok
- **Tesztelési terv**: Hogyan ellenőrizzük a megvalósítás helyességét

### 3. Változtatás létrehozása (Change)

A fejlesztés megkezdésekor hozzon létre egy változtatást a fejlesztési folyamat nyomon követéséhez:

```bash
openspec change create
```

A változtatás tartalmazza:
- **Kapcsolódó specifikáció**: Az a specifikáció, amelyet ez a változtatás valósít meg
- **Megvalósítási feladatok**: A konkrét fejlesztési feladatok listája
- **Előrehaladás követése**: A feladatok készültségi állapota
- **Kódváltoztatások**: A tényleges kódmódosítások

### 4. Érvényesítés és archiválás

A fejlesztés befejezése után érvényesítse, hogy a változtatás megfelel-e a specifikációnak, majd archiválja azt:

```bash
openspec change validate
openspec change archive
```

## OpenSpec könyvtárszerkezete

Az OpenSpec a projekt gyökérkönyvtárában létrehoz egy `openspec/` könyvtárat a következő szerkezettel:

```
openspec/
├── proposals/           # Javaslatok könyvtára
│   ├── PROPOSAL-001.md
│   └── PROPOSAL-002.md
├── specs/              # Specifikációk könyvtára
│   ├── SPEC-001.md
│   └── SPEC-002.md
├── changes/            # Változások könyvtára
│   ├── feature-name/
│   │   ├── proposal.md
│   │   ├── design.md
│   │   ├── specs.md
│   │   └── tasks.md
│   └── another-feature/
└── archived/           # Archivált változások
    └── feature-name/
```

### Könyvtárak magyarázata

- **proposals/**: Az összes javaslatdokumentum tárolására szolgál; minden javaslat külön Markdown fájlban
- **specs/**: A javaslatoknak megfelelő műszaki specifikációs dokumentumok tárolása
- **changes/**: Folyamatban lévő változtatások tárolása; minden változtatás külön alkönyvtárban
- **archived/**: Befejezett és archivált változtatások tárolása

## Alapvető parancsok

### Javaslatokkal kapcsolatos parancsok

```bash
# Új javaslat létrehozása
openspec proposal create

# Az összes javaslat listázása
openspec proposal list

# Javaslat részleteinek megtekintése
openspec proposal view <proposal-id>

# Javaslat jóváhagyása
openspec proposal approve <proposal-id>
```

### Specifikációkkal kapcsolatos parancsok

```bash
# Új specifikáció létrehozása
openspec spec create

# Az összes specifikáció listázása
openspec spec list

# Specifikáció részleteinek megtekintése
openspec spec view <spec-id>

# Specifikáció frissítése
openspec spec update <spec-id>
```

### Változtatásokkal kapcsolatos parancsok

```bash
# Új változtatás létrehozása
openspec change create

# Az összes változtatás listázása
openspec change list

# Változtatás részleteinek megtekintése
openspec change view <change-name>

# Feladat hozzáadása
openspec change add-task <change-name>

# Változtatás érvényesítése
openspec change validate <change-name>

# Változtatás archiválása
openspec change archive <change-name>
```

### Általános parancsok

```bash
# Verzió megtekintése
openspec --version

# Súgó megtekintése
openspec --help

# Egy adott parancs súgójának megtekintése
openspec <command> --help
```

## OpenSpec használata a HagiCode projektben

### Projekt integráció

A HagiCode projektben az OpenSpec beépült a fejlesztési munkafolyamatba:

1. **Inicializálás ellenőrzése**: A varázsló ellenőrzi, hogy az OpenSpec telepítve van-e
2. **Javaslatvezérelt fejlesztés**: Minden új funkciónak át kell mennie az OpenSpec javaslati folyamatán
3. **Kódáttekintés**: A változtatásoknak meg kell felelniük az OpenSpec specifikációinak a merge előtt
4. **Dokumentáció szinkronizálása**: Az OpenSpec dokumentációját összehangolják a kód megvalósításával

### Verziókövetelmény

A HagiCode projekt megköveteli az **OpenSpec >=1.0.0 {'<'}2.0.0** verzió használatát.

Telepítési parancs:

```bash
npm install -g @fission-ai/openspec@1
```

### Munkafolyamat példa

Az alábbiakban egy teljes példa található az OpenSpec használatára a HagiCode projektben:

#### 1. lépés: Javaslat létrehozása

```bash
cd /path/to/hagicode-project
openspec proposal create
```

Adja meg az interaktív promptban:
- Javaslat címe: Felhasználói hitelesítési funkció hozzáadása
- A probléma leírása: A rendszernek szüksége van felhasználói bejelentkezésre és jogosultságkezelésre
- Megoldás: JWT-alapú hitelesítési rendszer megvalósítása

#### 2. lépés: Specifikáció írása

A javaslat jóváhagyása után hozzon létre egy részletes specifikációt:

```bash
openspec spec create
```

Határozza meg a specifikációban:
- A hitelesítési folyamat tervezése
- API interfészek definiálása
- Adatbázis táblaszerkezete
- Biztonsági megfontolások

#### 3. lépés: Változtatás létrehozása

Kezdje el a megvalósítást:

```bash
openspec change create
```

Adja meg a változtatás nevét: `add-user-authentication`

#### 4. lépés: Feladatok hozzáadása

```bash
openspec change add-task add-user-authentication
```

Adjon meg konkrét feladatokat:
- Felhasználói regisztrációs végpont megvalósítása
- Bejelentkezési végpont megvalósítása
- JWT middleware megvalósítása
- Unit tesztek írása

#### 5. lépés: Megvalósítás és érvényesítés

A kód elkészülte után érvényesítse a változtatást:

```bash
openspec change validate add-user-authentication
```

#### 6. lépés: Archiválás

Sikeres érvényesítés után archiválja a változtatást:

```bash
openspec change archive add-user-authentication
```

## Ajánlott gyakorlatok

### Javaslatok írása

1. **Világos problémameghatározás**: Határozza meg világosan a megoldandó problémát; kerülje a homályos leírásokat
2. **Alternatívák mérlegelése**: Sorolja fel és hasonlítsa össze a különböző megoldásokat
3. **Hatás értékelése**: Elemezze a hatást és a kockázatokat a meglévő rendszerre
4. **Tömörség**: A javaslat legyen rövid és világos; a részleteket hagyja a specifikációnak

### Specifikációk írása

1. **Részletes, de nem túlzó**: Adjon elegendő részletet, de kerülje a túltervezést
2. **Használjon ábrákat**: Alkalmazzon folyamatábrákat, architektúra-ábrákat stb. a magyarázathoz
3. **Definiálja az interfészeket**: Pontosan határozza meg az összes publikus interfészt és adatszerkezetet
4. **Vegye figyelembe a szélsőséges eseteket**: Sorolja fel a kivételes eseteket és a hibakezelést

### Változáskezelés

1. **Kis lépésekben iteráljon**: Törje fel a nagy funkciókat több kisebb változtatásra
2. **Időben frissítsen**: Tartsa szinkronban a feladatok állapotát és a dokumentációt
3. **Kódáttekintés**: Végezzen kódáttekintést a változtatás befejezése után
4. **Archiváljon időben**: Archiválja a kész változtatásokat időben, tartsa tisztán a munkakönyvtárat

### Csapatmunka

1. **Rendszeres felülvizsgálatok**: Rendszeresen vizsgálják felül a javaslatokat és specifikációkat a csapat konszenzusának biztosítása érdekében
2. **Dokumentáció először**: Fejezze be a dokumentációt a kódolás megkezdése előtt
3. **Kommunikáció fenntartása**: Rögzítse a vitákat és döntéseket a javaslatokban és specifikációkban
4. **Verziókezelés**: Tartalmazza az OpenSpec dokumentációt a verziókezelésben

## Gyakori használati esetek

### Eset 1: Új funkció hozzáadása

1. Hozzon létre javaslatot, amely leírja az új funkció követelményeit
2. A csapat vitatja és jóváhagyja a javaslatot
3. Írjon részletes műszaki specifikációt
4. Hozzon létre változtatást és valósítsa meg a funkciót
5. Érvényesítse és archiválja a változtatást

### Eset 2: Meglévő kód refaktorálása

1. Hozzon létre javaslatot a refaktorálás okairól és céljairól
2. Határozza meg a refaktorálási tervet és lépéseket a specifikációban
3. Hozzon létre változtatást és refaktoráljon fokozatosan
4. Biztosítson tesztlefedettséget, és validálja a refaktorálás eredményét
5. Archiválja a változtatást

### Eset 3: Hibajavítás

Egyszerű hibák esetén:
- Közvetlenül javíthatók, nem szükséges a teljes OpenSpec folyamat

Összetett hibák esetén:
1. Készítsen javaslatot a hiba gyökérokának elemzésére
2. Határozza meg a javítási megoldást a specifikációban
3. Hozzon létre változtatást és valósítsa meg a javítást
4. Adjon hozzá teszteket a regresszió megelőzésére
5. Archiválja a változtatást

### Eset 4: Technikai adósság kezelése

1. Készítsen javaslatot a technikai adósság tételeiről
2. Határozza meg a javítási tervet a specifikációban
3. Hozzon létre változtatást a technikai adósság fokozatos rendezésére
4. Kövesse nyomon az előrehaladást és a hatást
5. Archiválja a befejezett javításokat

## Integráció más eszközökkel

### Git integráció

Az OpenSpec zökkenőmentesen integrálódik a Gittel:

```bash
# Az OpenSpec dokumentációt tartalmazza a verziókezelés
git add openspec/
git commit -m "Add OpenSpec proposal for feature X"
```

### CI/CD integráció

Érvényesítse az OpenSpec specifikációkat a CI/CD folyamatban:

```yaml
# .github/workflows/openspec.yml
name: OpenSpec Validation
on: [pull_request]
jobs:
  validate:
    runs-on: ubuntu-latest
    steps:
      - uses: actions/checkout@v2
      - name: Install OpenSpec
        run: npm install -g @fission-ai/openspec@1
      - name: Validate changes
        run: openspec change validate
```

### IDE integráció

Ajánlott Markdown-támogató szerkesztő használata az OpenSpec dokumentumok szerkesztéséhez:
- VS Code + Markdown bővítmény
- JetBrains IDEs + Markdown támogatás
- Vim/Neovim + Markdown bővítmény

## Hibakeresés

### Gyakori problémák

#### Probléma: Az OpenSpec parancs nem található

**Megoldás**:
1. Ellenőrizze, hogy az OpenSpec telepítve van-e: `npm list -g @fission-ai/openspec`
2. Ellenőrizze, hogy a PATH környezeti változó tartalmazza-e az npm globális bin könyvtárát
3. Telepítse újra: `npm install -g @fission-ai/openspec@1`

#### Probléma: Nem lehet javaslatot vagy változtatást létrehozni

**Megoldás**:
1. Ellenőrizze, hogy az aktuális könyvtár a projekt gyökérkönyvtára
2. Ellenőrizze, hogy létezik-e `openspec/` könyvtár; ha nem, hozza létre
3. Győződjön meg róla, hogy rendelkezik írási jogosultsággal

#### Probléma: Érvényesítés sikertelen

**Megoldás**:
1. Ellenőrizze, hogy a változtatás könyvtárban található dokumentáció teljes-e
2. Győződjön meg róla, hogy minden szükséges feladat befejeződött
3. Tekintse át a hiba részleteit: `openspec change validate <change-name> --verbose`

#### Probléma: Archiválás sikertelen

**Megoldás**:
1. Győződjön meg arról, hogy a változtatás sikeresen átment az érvényesítésen
2. Ellenőrizze, hogy létezik-e az `openspec/archived/` könyvtár
3. Győződjön meg arról, hogy nincsenek nem commitolt kódváltoztatások

## Haladó használat

### Testreszabott sablonok

Az OpenSpec támogatja a testreszabott javaslat- és specifikációsablonokat:

```bash
# Hozzon létre egy sablonkönyvtárat a projekt gyökerében
mkdir -p .openspec/templates

# Hozzon létre egy egyedi javaslatsablont
cat > .openspec/templates/proposal.md << 'EOF'
# Proposal: {{title}}

## Problem Statement
{{problem}}

## Proposed Solution
{{solution}}

## Custom Section
Your custom content here
EOF
```

### Konfigurációs fájl

Hozzon létre egy `.openspec.json` konfigurációs fájlt a projekt gyökerében:

```json
{
  "version": "1.0",
  "templates": {
    "proposal": ".openspec/templates/proposal.md",
    "spec": ".openspec/templates/spec.md"
  },
  "validation": {
    "requireTests": true,
    "requireDocs": true
  },
  "hooks": {
    "pre-archive": "npm test"
  }
}
```

### Hookok (Hooks)

Az OpenSpec támogatja egyedi script-ek futtatását bizonyos műveletek előtt és után:

```json
{
  "hooks": {
    "pre-create": "scripts/check-permissions.sh",
    "post-archive": "scripts/notify-team.sh"
  }
}
```
