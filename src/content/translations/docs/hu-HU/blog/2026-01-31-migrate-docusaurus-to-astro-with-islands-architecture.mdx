## A Docusaurus 3.x-től az Astro 5.x-ig: Visszatekintés a HagiCode webhely migrációjáról

> Ez a cikk a HagiCode hivatalos webhelyének teljes átállását tekinti át a Docusaurus 3.x-ről az Astro 5.x-re. Mélyen elmerülünk abban, hogy az Astro's Islands Architecture miként segített megoldani a teljesítmény szűk keresztmetszeteit, miközben megőriztük meglévő React-komponenseinket, javítva az építési sebességet és a betöltési teljesítményt.

## Háttér

2026 januárjában „szívtranszplantációt” hajtottunk végre a HagiCode hivatalos oldalán úgy, hogy az alapvető keretrendszert a Docusaurus 3.x-ről az Astro 5.x-re teljesen áttelepítettük. Ez nem egy impulzív újraírás volt, hanem egy alaposan megfontolt technikai döntés.

Az áttelepítés előtt webhelyünk funkcionálisan teljes volt, de kezdett megjelenni néhány klasszikus "luxusprobléma": felduzzadt összeállítási műtermékek, túlzott JavaScript-terhelések és az ideálisnál kisebb oldalbetöltési sebesség összetett dokumentációs oldalakon. Az AI kódolási asszisztens projektjeként a HagiCode-nak gyakori dokumentációra és funkciófrissítésre van szüksége, így a felépítés hatékonysága közvetlenül befolyásolja a kiadás sebességét. Ugyanakkor azt akartuk, hogy az oldal keresőbarátabb legyen (SEO), hogy több fejlesztő fedezze fel a projektet.

E fájdalmas pontok megoldására merész döntést hoztunk: újjáépítjük az egész rendszert az Astro-n. Ennek a döntésnek a hatása még a vártnál is nagyobb lehet. Hamarosan kitérek a részletekre.

## A HagiCode-ról

Az ebben a cikkben megosztott webhely-migrációs megközelítés a webhelyen szerzett gyakorlati tapasztalatainkból származik [HagiCode](https://github.com/HagiCode-org/site) projekt.

A HagiCode egy mesterséges intelligencia kódoló asszisztens, amely a fejlesztési hatékonyság javítására összpontosít. Nemcsak az alapvető funkciók megismétlésével foglalkozunk, hanem a fejlesztői tapasztalattal is. Ennek a webhely-refaktornak az volt a célja, hogy a felhasználók a lehető leggyorsabb élményt nyújtsa a dokumentumok és a hivatalos webhely böngészése során.

## Miért hagyjuk el az érett Docusaurus ökoszisztémát?

A React ökoszisztémán belül a Docusaurus régóta a "szabvány válasz" a dokumentációs oldalak számára. Kifejezetten működik, gazdag plugin-ökoszisztémát kínál, és aktív közösséggel rendelkezik. De ahogy a HagiCode egyre több funkciót kapott, úgy éreztük a korlátait is:

1.  **A teljesítmény szűk keresztmetszete**: A Docusaurus alapvetően egy React SPA (egyoldalas alkalmazás). Még ha csak statikus oldalakat ír is, a kliensnek be kell töltenie a React futási környezetet és hidratálnia kell az oldalt, ami szükségtelenül nehéz az egyszerű dokumentumoldalak esetében.
2.  **Nagy tartalomméret**: Még akkor is, ha egy oldal nagyon kevés tartalmat tartalmaz, a csomagban lévő JS mérete viszonylag állandó marad. Ez nem ideális mobil felhasználók vagy rossz hálózati feltételek számára.
3.  **Korlátozott rugalmasság**: Bár bővíthető, alacsonyabb szintű szabályozást szerettünk volna az összeállítási folyamat felett.

Az Astro pontosan a megfelelő időben érkezett, hogy megoldja ezeket a problémákat. Bevezetett egy új "szigeti architektúrát": alapértelmezés szerint az Astro statikus HTML-kódot generál nulla JavaScript-kóddal, és csak az interaktivitást igénylő összetevők "aktiválódnak" és töltik be a JS-t. Ez azt jelenti, hogy webhelyünk nagy része tiszta HTML-vé válik, és rendkívül gyorsan betöltődik.

## Alapvető migrációs stratégia: Sima építészeti átmenet

A migráció nem csak másolás és beillesztés volt. Szemléletváltásra volt szükség. A Docusaurus „all React” modelljéről áttértünk az Astro „Core + Islands” modelljére.

### 1. A konfigurációs rendszer újraépítése

Először is el kellett költöznünk `docusaurus.config.ts` hogy `astro.config.mjs`. Ez nem csak egy fájl átnevezés volt, hanem az útválasztás és az építési logika átírása.

A Docusaurusban minden plugin. Az Astroban minden egy integráció. Újra kellett definiálnunk a webhely alapútvonalát, a kimeneti módot (statikus vs. SSR) és az eszközoptimalizálási stratégiát.

**Migráció előtt:**
```typescript
// docusaurus.config.ts
export default {
  title: 'HagiCode',
  url: 'https://hagicode.com',
  baseUrl: '/',
  // ... more configuration
};
```

**Áttelepítés után:**
```javascript
// astro.config.mjs
import { defineConfig } from 'astro/config';
import react from '@astrojs/react';

export default defineConfig({
  integrations: [react()],
  site: 'https://hagicode.com',
  base: '/',
  // Optimization settings for static assets
  build: {
    inlineStylesheets: 'auto',
  },
});
```

### 2. Mit kell megtartani és mit kell refaktorálni a React komponensekben

Ez volt a migráció legfájdalmasabb része. Meglévő webhelyünk számos React összetevőt tartalmazott, például lapokat, kódkiemelést, visszajelzési gombokat és még sok mást. Eldobni őket pazarló lenne, de ha mindent megtartana, akkor a JavaScript hasznos terhelése túl nehéz lenne.

A HagiCode **progresszív hidratálási** stratégiát alkalmazott:
*   **Tisztán statikus összetevők**: A prezentációs tartalmakhoz, például fejlécekhez, láblécekhez és egyszerű szöveges dokumentációhoz átírtuk őket Astro összetevőkké (`.astro` fájlokat), és közvetlenül HTML-be renderelte őket összeállításkor.
*   **Interaktív szigetek**: Azon összetevők esetében, amelyeknek interaktívnak kell maradniuk, mint például a témaváltók, a lapváltások és a kódblokk-másoló gombok, megtartottuk a React megvalósítást, és hozzáadtuk `client:load` vagy `client:visible` irányelveket.

Például a dokumentációban gyakran használt Tabs komponensünk:

```jsx
// src/components/Tabs.jsx
import { useState } from 'react';
import './Tabs.css'; // Import styles

export default function Tabs({ items }) {
  const [activeIndex, setActiveIndex] = useState(0);
  // ... state logic
  return (
    <div className="tabs-wrapper">
      {/* Rendering logic */}
    </div>
  );
}
```

A Markdown alkalmazásban kifejezetten azt mondjuk az Astronak: "Ennek az összetevőnek JS-re van szüksége."

```markdown
// src/content/docs/example.mdx
import Tabs from '../../components/Tabs.jsx';

<!-- Load JS only when the component enters the viewport -->
<Tabs client:visible items={...} />
```

Így a nézetablakon kívüli interaktív komponensek nem versenyeznek a sávszélességért, ami nagymértékben javítja az első képernyő betöltési sebességét.

### 3. A stílusrendszer adaptálása: a CSS-moduloktól a kiterjedt CSS-ig

A Docusaurus alapértelmezés szerint támogatja a CSS-modulokat, míg az Astro a Scoped CSS-t támogatja a `<style>` címke. Mindkettő alapötlete a stílusizoláció, de a szintaxis más.

A HagiCode migráció során a legbonyolultabb CSS-modulokat az Astro hatókörű stílusaivá alakítottuk át. Ez valójában jó dolognak bizonyult, mert be `.astro` fájlok esetén a stílusok és a sablonok ugyanabban a fájlban élnek, ami intuitívabbá teszi a karbantartást.

**Refaktorálás előtt:**
```css
/* Tabs.module.css */
.wrapper { background: var(--ifm-background-color); }
```

**Refaktorálás után (Astro Scoped):**
```html
<!-- Tabs.astro -->
<div class="tabs-wrapper">
  <slot />
</div>

<style>
  .tabs-wrapper {
    /* Use CSS variables directly to adapt to the theme */
    background: var(--bg-color);
    padding: 1rem;
  }
</style>
```

Ugyanakkor egységesítettük a globális CSS-változórendszert, és az Astro környezettudatos képességeit használtuk annak biztosítására, hogy a sötét mód zökkenőmentesen váltson az oldalak között.

## A gyakorlatban elszenvedett buktatók és megoldásuk

A tényleges HagiCode migráció során jó néhány problémába ütköztünk. Íme néhány a legjellemzőbbek közül.

### 1. Út és környezet Változó fájdalompontok

A HagiCode támogatja az alútvonal-telepítést, például a GitHub Pages alkönyvtárában történő telepítést. A Docusaurusban `baseUrl` automatikusan kezeli. Az Astro-ban azonban óvatosabbnak kell lennünk a képhivatkozások és az API-kérések kezelésekor.

Ennek következetes kezelésére bevezettünk egy környezeti változó mechanizmust:

```javascript
// Handle paths in the build script
const getBasePath = () => import.meta.env.VITE_SITE_BASE || '/';
```

Ügyeljen arra, hogy ne kódoljon kóddal kezdődő útvonalakat `/` a kódodban. Fejlesztésben a gyártáshoz képest, vagy konfigurálás után a `base` útvonalat, ez 404-eseket okozhat az eszközök számára.

### 2. CommonJS Script kompatibilitás

Régi webhelyünkön volt néhány Node.js szkript olyan feladatokhoz, mint például a Metrics adatok automatikus lekérése és a webhelytérkép frissítése, és CommonJS-ben (`require`). Az Astro és a modern építőeszközök teljes mértékben magukévá teszik az ES modulokat (`import`/`export`).

Ha Önnek is vannak hasonló szkriptjei, ne felejtse el azokat ES-modulokká alakítani. Ebben az irányban mozog az ökoszisztéma, és minél hamarabb hajtja végre a változást, annál kevesebb baja lesz később.

```javascript
// Old way
const fs = require('fs');

// New way
import fs from 'fs';
```

### 3. Ne felejtse el a SEO-t és az átirányításokat

A keresőmotorok már indexelték a HagiCode régi Docusaurus oldalait. Ha közvetlenül az Astro-ra vált, és az URL szerkezete megváltozik, nagyszámú 404-et kaphat, és jelentősen csökkenhet a keresési rangsor.

Átirányítási szabályokat állítottunk be az Astro-ban:

```javascript
// astro.config.mjs
export default defineConfig({
  redirects: {
    '/docs/old-path': '/docs/new-path',
    // Map old links to new links in bulk
  }
});
```

Vagy kezelheti ezt a szerver konfigurációs rétegében. Győződjön meg arról, hogy a régi linkek 301 átirányíthatók az új címekre, mert ez kritikus a SEO szempontjából.

## Összegzés

A HagiCode számára a Docusaurusról az Astro-ra való áttérés nem csupán keretfrissítés volt. Ez egyben az "először a teljesítmény" filozófia gyakorlati megvalósítása volt.

**Amit nyertünk:**
*   **Kiemelkedő Lighthouse-pontszámok**: A migráció után a HagiCode webhely teljesítménypontszáma könnyen megközelítette a tökéletes pontszámot.
*   **Gyorsabb felépítési sebesség**: Az Astro növekményes felépítési képességei felére csökkentik a dokumentációfrissítések kiadási idejét.
*   **Megőrzött rugalmasság**: Az Islands Architecture segítségével nem áldoztunk fel egyetlen interaktív funkciót sem, és továbbra is használhattuk a Reactet, ahol szükséges.

Ha Ön is egy dokumentáció-orientált webhelyet tart fenn, és a csomagmérettel vagy a betöltési sebességgel küszködik, az Astro-t érdemes kipróbálni. Bár a migrációs folyamat bizonyos műveleteket igényel, például a PCode átnevezését HagiCode-ra és az összetevők egyenkénti áthelyezését, a cserébe kapott selymesen sima felhasználói élmény mindenképpen megéri.

Az ebben a cikkben megosztott összeállítási rendszer pontosan az a megközelítés, amelyet valódi próba és hiba útján fejlesztettünk ki a HagiCode felépítése során. Ha értékesnek találja ezt a megközelítést, az elmond valamit a mérnöki erősségünkről, és magát a HagiCode-ot is érdemes közelebbről megvizsgálni.
