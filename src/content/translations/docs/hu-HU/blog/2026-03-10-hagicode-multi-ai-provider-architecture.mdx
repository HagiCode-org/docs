## Gyakorlati multi-AI szolgáltató architektúra a HagiCode platformon

> Ez a cikk megosztja azt a technikai megközelítést, amelyet az Orleans Grain architektúra alatt használtunk két mesterségesintelligencia-eszköz, az iflow és az OpenCode integrálására egy egységes rendszeren keresztül. `IAIProvider` felületet, és részletesen összehasonlítja a WebSocket és a HTTP kommunikáció közötti megvalósítási különbségeket.

:::note[Time Context]
Ez a cikk a közzététel időpontjában érvényes szolgáltatói architektúra gyakorlatát rögzíti. Az aktuálisan aktívan támogatott hatókörről lásd: [AI Agent CLI telepítése](/hu-HU/related-software-installation/ai-agent-cli/) és a [Telepítési útmutató](/hu-HU/installation); `IFlowCli` A rendszer már csak múltbeli kompatibilitási értékként őrzi meg, és nem része a jelenlegi aktív támogatási körnek.
:::

## Háttér

Nincs benne semmi különösebben titokzatos. A HagiCode készítése közben egy nagyon gyakorlati problémába ütköztünk: a felhasználók különböző AI-eszközökkel akartak dolgozni. Ez nem meglepő, hiszen mindenkinek megvannak a saját szokásai. Vannak, akik a Claude Code-ot részesítik előnyben, vannak, akik szeretik a GitHub Copilotot, és néhány csapat saját fejlesztésű eszközöket használ.

Kezdeti megoldásunk egyszerű és közvetlen volt: írjunk dedikált integrációs kódot minden AI-eszközhöz. De a hátrányok gyorsan megmutatkoztak. A kódbázis megtelt `if-else` ágak, minden változtatást több helyen kellett tesztelni, és minden új eszköz egy újabb halom logikát jelentett a semmiből.

Később rájöttem, hogy jobb lenne egy egységeset létrehozni `IAIProvider` interfész, és elvonatkoztatni az összes AI-szolgáltató által megosztott képességeket. Így függetlenül attól, hogy melyik eszközt használják alatta, a felső rétegek ugyanúgy hívhatják.

A közelmúltban a projektbe két új eszközt kellett integrálni: az iflow-t és az OpenCode-ot. Mindkettő támogatja az ACP protokollt, de a kommunikációs stílusuk eltérő. Az iflow a WebSocket, míg az OpenCode HTTP API-t használ. Ez egy hasznos építészeti teszt lett: két különböző szállítási módot egy egységes felület mögé adaptálunk.

## A HagiCode-ról

Az ebben a cikkben megosztott megközelítés a gyakorlati tapasztalatainkból származik [HagiCode](https://hagicode.com) projekt. A HagiCode egy mesterséges intelligencia által támogatott fejlesztői platform, amely az Orleans Grain architektúrára épül. Egységesített módon integrálódik a különböző AI-szolgáltatókkal `IAIProvider` felület, amely lehetővé teszi a felhasználók számára, hogy rugalmasan válasszák ki az általuk preferált AI-eszközöket.

## Építészeti tervezés

### Egységes felület absztrakció

Először is meghatároztuk a `IAIProvider` interfész, és kivonatolta azokat a képességeket, amelyeket minden mesterséges intelligencia-szolgáltatónak megvalósítania kell:

```csharp
public interface IAIProvider
{
    string Name { get; }
    bool SupportsStreaming { get; }
    ProviderCapabilities Capabilities { get; }

    Task<AIResponse> ExecuteAsync(AIRequest request, CancellationToken cancellationToken = default);
    IAsyncEnumerable<AIStreamingChunk> StreamAsync(AIRequest request, CancellationToken cancellationToken = default);
    Task<ProviderTestResult> PingAsync(CancellationToken cancellationToken = default);
    IAsyncEnumerable<AIStreamingChunk> SendMessageAsync(AIRequest request, string? embeddedCommandPrompt = null, CancellationToken cancellationToken = default);
}
```

Ez az interfész számos kulcsfontosságú módszert tartalmaz:
- `ExecuteAsync`: egyszeri AI kérés végrehajtása
- `StreamAsync`: streaming válaszokat kap a valós idejű megjelenítéshez
- `PingAsync`: végezzen állapotfelmérést annak ellenőrzésére, hogy a szolgáltató elérhető-e
- `SendMessageAsync`: üzenet küldése a beágyazott parancsok támogatásával

### IFlowCliProvider: WebSocket alapú megvalósítás

Az iflow a WebSocketet használja az ACP kommunikációhoz. Az általános architektúra így néz ki:

```
IFlowCliProvider → ACPSessionManager → WebSocketAcpTransport → iflow CLI
                ↓
         Dynamic port allocation + process management
```

Az alapfolyamat is meglehetősen egyszerű:
1. `ACPSessionManager` ACP munkameneteket hoz létre és kezel.
2. `WebSocketAcpTransport` kezeli a WebSocket kommunikációt.
3. A port dinamikusan allokálódik, és ezzel elindul az iflow folyamat `iflow --experimental-acp --port`.
4. `IAIRequestToAcpMapper` és `IAcpToAIResponseMapper` kérések és válaszok konvertálása.

Íme az alapkód:

```csharp
private async IAsyncEnumerable<AIStreamingChunk> StreamCoreAsync(
    AIRequest request,
    string? embeddedCommandPrompt,
    [EnumeratorCancellation] CancellationToken cancellationToken)
{
    // Resolve working directory
    var resolvedWorkingDirectory = ResolveWorkingDirectory(request);
    var effectiveRequest = ApplyEmbeddedCommandPrompt(request, embeddedCommandPrompt);

    // Create ACP session
    await using var session = await _sessionManager.CreateSessionAsync(
        Name,
        resolvedWorkingDirectory,
        cancellationToken,
        request.SessionId);

    // Send prompt
    var prompt = _requestMapper.ToPromptString(effectiveRequest);
    var promptResponse = await session.SendPromptAsync(prompt, cancellationToken);

    // Receive streaming response
    await foreach (var notification in session.ReceiveUpdatesAsync(cancellationToken))
    {
        if (_responseMapper.TryConvertToStreamingChunk(notification, out var chunk))
        {
            if (chunk.Type == StreamingChunkType.Metadata && chunk.IsComplete)
            {
                yield return chunk;
                yield break;
            }
            yield return chunk;
        }
    }
}
```

Itt érdemes kiemelni néhány tervezési pontot:
- Használja `await using` hogy biztosítsa a munkamenet helyes feloldását és elkerülje az erőforrásszivárgást.
- A streamelési válaszok visszaküldése `IAsyncEnumerable`, amely természetesen támogatja az aszinkron adatfolyamokat.
- Használja `Metadata` darabokat a befejezés meghatározásához és a teljes válasz megérkezésének biztosításához.

### OpenCodeCliProvider: HTTP API-alapú megvalósítás

Az OpenCode HTTP API-n keresztül nyújtja szolgáltatását, így az architektúra kissé eltér:

```plaintext
OpenCodeCliProvider → OpenCodeRuntimeManager → OpenCodeClient → OpenCode HTTP API
                      ↓
                OpenCodeProcessManager → opencode process management
```

Az OpenCode figyelemre méltó jellemzője, hogy SQLite adatbázist használ a munkamenet-összerendelések fenntartásához. Ez lehetővé teszi a munkamenet-helyreállítást és az azonnali válaszok helyreállítását:

```csharp
private async Task<OpenCodePromptExecutionResult> ExecutePromptAsync(
    AIRequest request,
    string? embeddedCommandPrompt,
    CancellationToken cancellationToken)
{
    var prompt = BuildPrompt(request, embeddedCommandPrompt);
    var resolvedWorkingDirectory = ResolveWorkingDirectory(request.WorkingDirectory);
    var client = await _runtimeManager.GetClientAsync(resolvedWorkingDirectory, cancellationToken);
    var bindingSessionId = request.SessionId;
    var boundSession = TryGetBinding(bindingSessionId, resolvedWorkingDirectory);

    // Try to use the already bound session
    if (boundSession is not null)
    {
        try
        {
            return await PromptSessionAsync(
                client,
                boundSession,
                BuildPromptRequest(request, prompt, CreatePromptMessageId()),
                request.Model ?? _settings.Model,
                cancellationToken);
        }
        catch (OpenCodeApiException ex) when (IsStaleBinding(ex))
        {
            // The session has expired, remove the binding
            RemoveBinding(bindingSessionId);
        }
    }

    // Create a new session
    var session = await client.Session.CreateAsync(new OpenCodeSessionCreateRequest
    {
        Title = BuildSessionTitle(request)
    }, cancellationToken);

    BindSession(bindingSessionId, session.Id, resolvedWorkingDirectory);
    return await PromptSessionAsync(client, session.Id, ...);
}
```

Ennek a megvalósításnak számos érdekessége van:
- Munkamenet-kötési mechanizmus: ugyanaz `SessionId` újra felhasználja ugyanazt az OpenCode munkamenetet, elkerülve az ismételt munkamenet létrehozását.
- Lejáratkezelés: ha egy munkamenet lejártnak találja, a kötés automatikusan törlődik.
- Adatbázis-perzisztencia: az összerendeléseket az SQLite tárolja, és újraindítás után is érvényben marad.

### A két megközelítés összehasonlítása

| Aspect | IFlowCliProvider | OpenCodeCliProvider |
|------|-----------------|---------------------|
| Kommunikáció | WebSocket (ACP) | HTTP API |
| Folyamatmenedzsment | ACPSessionManager | OpenCodeProcessManager |
| Kikötői kiosztás | Dinamikus port | Nincs port (HTTP-t használ) |
| Munkamenet menedzsment | ACPSession | OpenCodeSession |
| Kitartás | Memória gyorsítótár | SQLite adatbázis |
| Indítási parancs | `iflow --experimental-acp --port` | `opencode` |
| Látencia | Alacsonyabb (hosszú élettartamú kapcsolat) | Viszonylag magasabb (HTTP kérések) |

Az, hogy melyik megközelítést választja, elsősorban az Ön igényeitől függ. A WebSocket jobb a magas valós idejű követelményekkel járó forgatókönyvekhez, míg a HTTP API egyszerűbb és könnyebben hibakereshető.

## Gyakorlati útmutató

### Szolgáltatók konfigurálása

Először engedélyezze a két szolgáltatót a konfigurációs fájlban:

```yaml
AI:
  Providers:
    IFlowCli:
      Type: "IFlowCli"
      Enabled: true
      ExecutablePath: "iflow"
      Model: null
      WorkingDirectory: null
    OpenCodeCli:
      Type: "OpenCodeCli"
      Enabled: true
      ExecutablePath: "opencode"
      Model: "anthropic/claude-sonnet-4"
      WorkingDirectory: null

OpenCode:
  Enabled: true
  BaseUrl: "http://localhost:38376"
  ExecutablePath: "opencode"
  StartupTimeoutSeconds: 30
  RequestTimeoutSeconds: 120
```

### Használja az IFlowCliProvider-t

```csharp
// Get provider through the factory
var provider = await _providerFactory.GetProviderAsync(AIProviderType.IFlowCli);

// Execute an AI request
var request = new AIRequest
{
    Prompt = "请帮我重构这个函数",
    WorkingDirectory = "/path/to/project",
    Model = "claude-sonnet-4"
};

// Get the complete response
var response = await provider.ExecuteAsync(request, cancellationToken);
Console.WriteLine(response.Content);

// Or use streaming responses
await foreach (var chunk in provider.StreamAsync(request, cancellationToken))
{
    if (chunk.Type == StreamingChunkType.ContentDelta)
    {
        Console.Write(chunk.Content);
    }
}
```

### Használja az OpenCodeCliProvider-t

```csharp
// Get provider through the factory
var provider = await _providerFactory.GetProviderAsync(AIProviderType.OpenCodeCli);

var request = new AIRequest
{
    Prompt = "请帮我分析这个错误",
    WorkingDirectory = "/path/to/project",
    Model = "anthropic/claude-sonnet-4"
};

var response = await provider.ExecuteAsync(request, cancellationToken);
Console.WriteLine(response.Content);
```

### Egészségügyi ellenőrzések

Indítás vagy használat előtt ellenőrizheti, hogy a szolgáltató elérhető-e:

```csharp
var iflowResult = await iflowProvider.PingAsync(cancellationToken);
if (!iflowResult.Success)
{
    Console.WriteLine($"IFlow is unavailable: {iflowResult.ErrorMessage}");
    return;
}

var openCodeResult = await openCodeProvider.PingAsync(cancellationToken);
if (!openCodeResult.Success)
{
    Console.WriteLine($"OpenCode is unavailable: {openCodeResult.ErrorMessage}");
    return;
}
```

### Beágyazott parancstámogatás

Mindkét szolgáltató támogatja a beágyazott parancsokat, mint pl `/file:xxx`:

```csharp
var request = new AIRequest
{
    Prompt = "分析这个文件的问题",
    SystemMessage = "你是一个代码分析专家"
};

await foreach (var chunk in provider.SendMessageAsync(
    request,
    embeddedCommandPrompt: "/file:src/main.cs",
    cancellationToken))
{
    Console.Write(chunk.Content);
}
```

## Megjegyzések és bevált gyakorlatok

### Erőforrás-kezelés

Az IFlow hosszú élettartamú WebSocket kapcsolatokat használ, így az erőforrás-kezelés külön figyelmet érdemel:
- Használja `await using` hogy a munkamenetek megfelelően megjelenjenek.
- A törlés elindítja a folyamat megtisztítását.
- `ACPSessionManager` támogatja a munkamenetek maximális számának korlátozását.

Az OpenCode folyamatkezelés viszonylag egyszerűbb, és `OpenCodeRuntimeManager` automatikusan kezeli.

### Hibakezelés

Mindkét szolgáltató rendelkezik teljes hibakezeléssel:
- Az IFlow hibákat az ACP munkamenet-frissítések továbbítják.
- Az OpenCode hibákat átdobják `OpenCodeApiException`.
- Javasoljuk, hogy a hívó felfogja és kezelje ezeket a kivételeket.

### Teljesítmény szempontjai

- Az IFlow WebSocket kommunikáció késleltetése alacsonyabb, mint a HTTP.
- Az OpenCode munkamenet újrafelhasználása csökkentheti a HTTP-kérések többletterhelését.
- A gyári gyorsítótár-mechanizmus elkerüli a szolgáltatók ismételt létrehozását.
- Magas egyidejű forgatókönyvek esetén fokozottan ügyeljen a folyamatok és a kapcsolatok számának korlátaira.

### Konfiguráció érvényesítése

A végrehajtható elérési út indításkor érvényesül, de futásidejű problémák továbbra is előfordulhatnak. `PingAsync` hasznos eszköz a konfiguráció helyességének ellenőrzésére:

```csharp
// Check at startup
var provider = await _providerFactory.GetProviderAsync(providerType);
var result = await provider.PingAsync(cancellationToken);
if (!result.Success)
{
    _logger.LogError("Provider {ProviderType} is unavailable: {Error}", providerType, result.ErrorMessage);
}
```

## Összegzés

Ez a cikk megosztja a HagiCode platform által az iflow és az OpenCode két mesterségesintelligencia-eszköz integrálásakor alkalmazott technikai megközelítést. Egy egységesen keresztül `IAIProvider` interfészen a különböző kommunikációs stílusokat, a WebSocketet és a HTTP-t adaptáltuk, miközben a felső réteg hívási mintáját konzisztensnek tartottuk.

Az alapötlet valójában nagyon egyszerű:
1. Határozzon meg egy egységes felület absztrakciót.
2. Adapterrétegek létrehozása a különböző megvalósításokhoz.
3. Kezelje mindent egységesen a gyári mintán keresztül.

Ez jó bővíthetőséget biztosít a rendszernek. Ha később egy új AI-eszközt kell integrálni, akkor csak azt kell megvalósítanunk `IAIProvider` felület túl sok meglévő kód megváltoztatása nélkül.

Ha Ön is több AI-eszköz integrációján dolgozik, remélem, ez a cikk hasznos lesz.
