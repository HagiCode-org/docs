## Böngésző WebSocket hitelesítési kihívások megoldása: Gyakorlati proxy minta a Doubao beszédfelismeréshez

> A böngésző WebSocket API nem támogatja az egyéni HTTP-fejléceket, ami kihívást jelent a beszédfelismerő szolgáltatások számára, amelyek hitelesítési adatokat igényelnek a fejlécekben. Ez a cikk megosztja, hogyan oldottuk meg ezt a problémát a HagiCode projektben egy háttérproxy mintával, és hogyan fejlődött a megközelítés a játszótéri kísérletektől a termelési használatig.


## Háttér

Amikor elkezdtük építeni a HagiCode projekt beszédfelismerését, magabiztosan választottuk a ByteDance Doubao beszédfelismerő szolgáltatását. A kezdeti tervezés egyszerű volt: hagyja, hogy a frontend közvetlenül csatlakozzon a Doubao WebSocket szolgáltatásához. Milyen nehéz lehet? Csak nyisson meg egy kapcsolatot, és küldjön néhány adatot, igaz?

Aztán jött a meglepetés: a Doubao API-ja megköveteli, hogy a hitelesítési információkat HTTP-fejléceken keresztül továbbítsák, beleértve az ilyeneket is `accessToken` és `secretKey`. Ez azonnal kínossá vált, mert a böngésző WebSocket API egyszerűen nem támogatja az egyéni fejlécek beállítását.

Tehát mit csinálsz, ha a böngésző nem engedi elküldeni őket?

Akkoriban két lehetőséget mérlegeltünk:

1. Helyezze be a hitelesítő adatokat az URL-lekérdezési paraméterekbe – egyszerűen és tompa
2. Adjon hozzá egy proxy réteget a háttérhez – első pillantásra több munka

Az első lehetőség a hitelesítő adatokat közvetlenül az előtér-kódban és a helyi tárolóban teszi közzé. Ez tényleg biztonságos? Nem éreztem magam kényelmesen vele. Ráadásul egyes API-k fejléc alapú ellenőrzést igényelnek, így ez a megközelítés nem is életképes.

Végül a második lehetőséget választottuk: WebSocket proxyt valósítunk meg a háttérben. Véletlenül ezt a mintát először a játszótéri környezetünkben érvényesítettük, és csak miután meggyőződtünk a stabilitásáról, helyeztük át a gyártásba. Végtére is, senki sem akarja, hogy a gyártás laboratóriumi kísérletként működjön.

## A HagiCode-ról

Néha ezek a technikai akadályok frusztrálóak, de olyan minták elsajátítására is kényszerítenek, amelyek később hasznosnak bizonyulnak.

## Technikai kihíváselemzés

### A böngésző WebSocket korlátozásai

A szabványos WebSocket API csodálatosan egyszerűnek tűnik:

```javascript
const ws = new WebSocket('wss://example.com/ws');
```

De pontosan ebben az egyszerűségben rejlik a probléma – csak paramétereket ad át az URL-ben, és nem állíthat be fejléceket úgy, ahogyan egy HTTP-kérés:

```javascript
// This is not supported in the WebSocket API
const ws = new WebSocket('wss://example.com/ws', {
  headers: {
    'Authorization': 'Bearer token'
  }
});
```

És ez az alapkérdés. Az olyan szolgáltatások esetében, mint a Doubao beszédfelismerés, amelyek fejléc alapú hitelesítéstől függenek, ez a korlátozás kemény blokkoló.

Ha elfogadja ezt a korlátot, az architektúrának meg kell változnia.

### Építészeti tervezési döntések

A megoldás kialakításakor alaposan összehasonlítottuk a kompromisszumokat.

**1. döntés: A proxy minta kiválasztása**

Két megközelítést hasonlítottunk össze:

| Opció | Profik | Hátrányok | határozat |
|------|------|------|------|
| Natív WebSocket | Könnyű, egyszerű, közvetlen továbbítás | A kapcsolatkezelést manuálisan kell kezelni | Kiválasztva |
| SignalR | Automatikus újracsatlakozás, erős gépelés | Túl bonyolult, extra függőségek | Elutasítva |

Végül a natív WebSocketet választottuk. Hogy őszinte legyek, ez volt a legkönnyebb lehetőség, és jobban illeszkedett az egyszerű kétirányú bináris adatfolyam-továbbításhoz. A SignalR behúzása túlzott tervezésnek tűnt volna ebben a felhasználási esetben, és ez további késleltetést jelenthet.

**2. határozat: Kapcsolatkezelési stratégia**

Elfogadtuk az „egy kapcsolat, egy munkamenet” modellt – minden frontend WebSocket kapcsolat a saját független Doubao háttérkapcsolatára van leképezve.

Az előnyök egyértelműek:
- Egyszerűen kivitelezhető, és igazodik az általános használati mintához
- Könnyebb hibakeresés és hibaelhárítás
- Jó az erőforrás-szigetelés, amely megakadályozza a munkamenetek közötti interferenciát

Egyszerűen fogalmazva, néha a közvetlen megoldás a legjobb. A komplexitás nem teszi automatikusan jobbá a tervezést.

**3. döntés: A hitelesítési adatok tárolása**

A hitelesítési adatokat háttérkonfigurációs fájlok tárolják (`appsettings.yml` vagy környezeti változók) és a függőségi injektálással betöltve:

- Egyszerű konfigurációs modell, amely megfelel a meglévő háttérkonvencióknak
- Az érzékeny adatok soha nem jutnak el a frontendhez
- Támogatja a többkörnyezeti beállítást a fejlesztéshez, teszteléshez és gyártáshoz

Az elkülönülésnek ez a szintje számít. Senki sem akarja, hogy a hitelesítő adatok olyan helyeken lebegjenek, ahol nem lenne szabad.

### Adatfolyam tervezés

A teljes adatfolyam így néz ki:

```
Frontend (browser)
  │
  │ ws://backend/api/voice/ws
  │ WebSocket (binary)
  ▼
Backend (proxy)
  │
  │ wss://openspeech.bytedance.com/
  │ (with authentication headers)
  ▼
Doubao API
```

Maga az áramlás nem bonyolult:
1. Az előtér a WebSocketen keresztül csatlakozik a háttérproxyhoz
2. A háttérproxy hangadatokat fogad, és hitelesített fejlécekkel csatlakozik a Doubao API-hoz
3. A Doubao API felismerési eredményeket ad vissza, a proxy pedig továbbítja azokat a frontendnek
4. Az egész folyamat teljesen aszinkron marad a kétirányú streameléssel

Ha a felelősségek egyértelműen megoszlanak, a tervezés egészen természetessé válik.

## Alapkomponens megvalósítása

### 1. WebSocket végpont konfigurációja

```csharp
app.Map("/ws", async context =>
{
    if (context.WebSockets.IsWebSocketRequest)
    {
        // Read configuration from query parameters
        var appId = context.Request.Query["appId"];
        var accessToken = context.Request.Query["accessToken"];

        // Validate required parameters
        if (string.IsNullOrEmpty(appId) || string.IsNullOrEmpty(accessToken))
        {
            context.Response.StatusCode = 400;
            return;
        }

        // Accept the WebSocket connection
        using var webSocket = await context.WebSockets.AcceptWebSocketAsync();

        // Message handling loop
        var buffer = new byte[4096];
        while (!webSocket.CloseStatus.HasValue)
        {
            var result = await webSocket.ReceiveAsync(buffer, CancellationToken.None);

            if (result.MessageType == WebSocketMessageType.Close)
            {
                await webSocket.CloseAsync(
                    result.CloseStatus.Value,
                    result.CloseStatusDescription,
                    CancellationToken.None);
                break;
            }

            // Process audio data
            await HandleAudioDataAsync(buffer, result.Count);
        }
    }
});
```

### 2. Munkamenet menedzsment

```csharp
public class DoubaoSessionManager : IDoubaoSessionManager
{
    private readonly ConcurrentDictionary<string, DoubaoSession> _sessions = new();

    public DoubaoSession CreateSession(string connectionId)
    {
        var session = new DoubaoSession(connectionId);
        _sessions[connectionId] = session;
        return session;
    }

    public async Task SendAudioAsync(string connectionId, byte[] audioData)
    {
        if (_sessions.TryGetValue(connectionId, out var session))
        {
            await session.SendAudioAsync(audioData);
        }
    }

    public void RemoveSession(string connectionId)
    {
        if (_sessions.TryRemove(connectionId, out var session))
        {
            session.Dispose();
        }
    }
}
```

Használata `ConcurrentDictionary` a munkamenet-kezelés azt jelenti, hogy a szálak biztonságát nagyrészt mi kezeljük. Minden bejövő kapcsolat saját munkamenetet kap, és a tisztítás automatikusan megtörténik a kapcsolat leválasztásakor.

## Frontend integráció a gyakorlatban

### WebSocket csatlakozás

```javascript
class DoubaoVoiceClient {
    constructor(config) {
        this.config = config;
        this.ws = null;
    }

    async connect() {
        const url = new URL(this.config.wsUrl);
        // Add query parameters
        Object.entries(this.config.params).forEach(([key, value]) => {
            url.searchParams.set(key, value);
        });

        this.ws = new WebSocket(url);

        return new Promise((resolve, reject) => {
            this.ws.onopen = () => {
                console.log('[DoubaoVoice] Connected');
                resolve();
            };

            this.ws.onmessage = (event) => {
                this._handleMessage(JSON.parse(event.data));
            };

            this.ws.onerror = reject;
        });
    }

    _handleMessage(message) {
        switch (message.type) {
            case 'status':
                this._handleStatus(message.payload);
                break;
            case 'result':
                this.onResult?.(message.payload);
                break;
            case 'error':
                console.error('[DoubaoVoice] Error:', message.payload);
                break;
        }
    }
}

// Usage example
const client = new DoubaoVoiceClient({
    wsUrl: 'ws://localhost:5000/ws',
    params: {
        appId: 'your-app-id',
        accessToken: 'your-access-token',
        sampleRate: 16000,
        bitsPerSample: 16,
        channels: 1
    }
});
```

### Hangrögzítés és átvitel

Használata `AudioWorkletNode` a hangfeldolgozáshoz jobb teljesítményt nyújt:

```javascript
// audio-worklet.js
class AudioProcessorWorklet extends AudioWorkletProcessor {
    process(inputs, outputs, parameters) {
        const input = inputs[0]?.[0];
        if (!input) return true;

        // Convert to 16-bit PCM
        const pcm = new Int16Array(input.length);
        for (let i = 0; i < input.length; i++) {
            pcm[i] = Math.max(-32768, Math.min(32767, input[i] * 32767));
        }

        this.port.postMessage({
            type: 'audioData',
            data: pcm.buffer
        }, [pcm.buffer]);

        return true;
    }
}

registerProcessor('audio-processor', AudioProcessorWorklet);

// Main thread code
async function startAudioRecording() {
    const stream = await navigator.mediaDevices.getUserMedia({
        audio: {
            echoCancellation: true,
            noiseSuppression: true,
            autoGainControl: true,
            sampleRate: 48000
        }
    });

    const audioContext = new AudioContext();
    const audioSource = audioContext.createMediaStreamSource(stream);

    await audioContext.audioWorklet.addModule('/audio-worklet.js');
    const audioWorkletNode = new AudioWorkletNode(audioContext, 'audio-processor');

    audioWorkletNode.port.onmessage = (event) => {
        if (event.data.type === 'audioData' && ws?.readyState === WebSocket.OPEN) {
            ws.send(event.data.data); // Send binary data directly
        }
    };

    audioSource.connect(audioWorkletNode);
}
```

`AudioWorklet` sokkal jobban teljesít, mint `ScriptProcessorNode` és elkerüli a régebbi feldolgozási útvonalak által gyakran előidézett hang akadozási problémákat.

## Megjegyzések és bevált gyakorlatok

### 1. Csatlakozás figyelése

- Rendszeresen naplózza a munkamenet állapotát a kapcsolat teljes életciklusának nyomon követéséhez
- Figyelje az audioszegmensek számát és időtartamát, hogy észlelje a rendellenes kapcsolatokat
- Rögzítse a kapcsolat állapotát és az újracsatlakozási viselkedést a Doubao szolgáltatáshoz

Ezek alapvető működési gyakorlatok, de valódi különbséget jelentenek a termelésben.

### 2. Hibakezelés

- Rögzítse és naplózza az összes WebSocket kivételt
- Használja `IAsyncDisposable` hogy biztosítsák a források megtisztítását
- Végezzen kecses kapcsolatleállítást és időtúllépés-kezelést

Röviden, előnyben részesítsék a robusztusságot.

### 3. Hangformátum követelményei

- Mintavételi frekvencia: 16000 Hz (ajánlott) vagy 8000 Hz
- Bitmélység: 16 bit
- Csatornák: monó
- Kódolás: PCM (nyers)

Ha a formátum rossz, a felismerés sikertelen lehet, vagy a minőség jelentősen romolhat. Ezek a követelmények számítanak.

### 4. Biztonsági megfontolások

- Az érzékeny hitelesítő adatokat csak a háttérkonfigurációban tartsa meg
- A csatlakozási korlátok érvényesítése az erőforrások kimerülésének megelőzése érdekében
- Használjon HTTPS/WSS-t a termelésben

A biztonság soha nem jelent kisebb gondot.

### 5. Teljesítményoptimalizálás

- A blokkolás elkerülése érdekében használjon aszinkron műveleteket
- Hangolja be a pufferméreteket szükség szerint (alapértelmezett: 4096 bájt)
- Fontolja meg a kapcsolatok összevonását és az újrahasználati stratégiákat

Alkalmazza ezeket az optimalizálásokat, ahol ésszerűek a munkaterheléshez.

## Telepítési ajánlások

1. **Docker üzembe helyezés**: csomagolja be a proxyszolgáltatást tárolóba a könnyebb méretezés és kezelés érdekében
2. **Teherelosztás**: Használja az Nginxet vagy az Envoy-t fordított proxyként a WebSocket forgalomhoz
3. **Egészségügyi ellenőrzések**: Végezzen szívverés-alapú ellenőrzéseket a szolgáltatások elérhetőségének figyelemmel kísérése érdekében
4. **Napló-összesítés**: Küldje el a naplókat egy központi naplózási rendszerbe, mint például az ELK vagy a Loki

A telepítés a csapattól és a környezettől függően lehet egyszerű vagy összetett, ezért ennek megfelelően módosítsa.

## Összegzés

A WebSocket proxy minta megoldja azt az alapvető problémát, hogy a böngésző WebSocket API nem támogatja az egyéni fejléceket. A HagiCode projektben ez a minta megvalósíthatónak és stabilnak is bizonyult, ahogy a játszóterek validálásától az éles üzembe helyezésig jutott.

A legfontosabb elvitelek:
- A háttérproxy biztonságosan továbbíthatja a hitelesítési információkat
- A Native WebSocket könnyű és hatékony az egyszerű forgatókönyvekhez
- Az "egy kapcsolat, egy munkamenet" modell mind a megvalósítást, mind a hibakeresést leegyszerűsíti
- A frontend-backend protokollnak el kell választania a vezérlőjeleket az audioadatoktól

Ha olyan szolgáltatást épít, amely a WebSocket hitelesítéstől is függ, remélem, ez a minta hasznos kiindulópontot ad.

Ha kérdése van, nyugodtan beszélje meg velünk. A technikai fejlődés gyorsabban megy végbe, ha az emberek összehasonlítják a jegyzeteket.
