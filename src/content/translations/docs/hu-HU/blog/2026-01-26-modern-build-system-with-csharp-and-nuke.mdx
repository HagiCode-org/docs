## Alapelemzés: Miért a Nuke?

Elgondolkodhat: "Olyan sok összeállítási rendszer létezik már, mint például a Make, Gradle vagy akár egyszerű Shell-szkriptek. Miért menne ki a C#-alapú rendszer használatából?"

Ez jogos kérdés. A Nuke fő vonzereje az, hogy a már jól ismert programozási nyelvi funkciókat a szkriptek világába hozza.

### 1. Modularizing Build Workflows: The Art of Targets

A Nuke-nak nagyon világos tervezési filozófiája van: **minden egy cél**.

A hagyományos szkriptekben könnyen előfordulhat, hogy a lineáris végrehajtási logika több száz sora összegabalyodik. A Nuke-ban az építési folyamatot függetlenre bontjuk `Target`s. Minden célpont pontosan egy dolgot csinál, például:

- `Clean`: tisztítsa meg a kimeneti könyvtárat
- `Restore`: függőségi csomagok visszaállítása
- `Compile`: fordítsa le a kódot
- `Test`: egységtesztek futtatása

Ez a kialakítás nagyon jól illeszkedik az egyfelelősség elvéhez. Az építőelemekhez hasonlóan ezek a célpontok is szabadon kombinálhatók. Ennél is fontosabb, hogy a Nuke lehetővé teszi a célpontok közötti függőségek meghatározását. Például ha futni akarsz `Test`, a rendszer automatikusan ellenőrzi, hogy `Compile` már kivégezték. Ha akarod `Compile`, akkor `Restore` természetesen az első.

Ez a függőségi gráf sokkal világosabbá teszi a logikát, és jelentősen javítja a végrehajtás hatékonyságát, mivel a Nuke automatikusan elemzi az optimális végrehajtási utat.

### 2. Típusbiztonság: Viszlát a gépelési hibák rémálmától

Bárki, aki Pythonban írt build szkripteket, valószínűleg belefutott ebbe a kínos helyzetbe: a szkript öt percig fut, majd meghiúsul, mert `Confi.guration` hibásan írták le, vagy mert olyan karakterláncot adtak át, ahol numerikus paramétert vártak.

A build szkriptek C# nyelven történő írásának legnagyobb előnye a **típusbiztonság**. Ez azt jelenti:

- **Fordítási idő ellenőrzése**: az IDE gépelés közben meg tudja mondani, hogy mi a hiba, ahelyett, hogy a futásidőig várna.
- **Biztonságos refaktorálás**: ha át szeretne nevezni egy változót vagy metódust, az IDE egy lépésben képes kezelni anélkül, hogy kockázatos globális keresésre és cserére kényszerítené.
- **Intelligens befejezés**: A nagy teljesítményű IntelliSense automatikusan befejezi a kódot, így nem kell folyamatosan lapozgatnia a dokumentációban, hogy emlékezzen a homályos API-kra.

### 3. Platformok közötti támogatás: egyetlen konzisztens építési tapasztalat

A múltkor azt írtuk `.bat` Windowson és `.sh` Linuxon, majd esetleg hozzáadtunk egy Python-szkriptet a szakadék áthidalására. Mostantól bárhol, ahol a .NET Core vagy a .NET 5+ futhat, a Nuke is futhat.

Ez azt jelenti, hogy a csapat tagjai használhatják a Windows, Linux vagy macOS rendszert, és akár a Visual Studio-t, akár a VS Code-ot, akár a Ridert részesítik előnyben, mindegyik pontosan ugyanazt a logikát hajtja végre. Ez nagymértékben csökkenti a környezeti különbségek okozta problémákat, különösen a klasszikus "működik a gépemen" problémát.

### 4. Paraméterek és konfigurációkezelés

A Nuke egy nagyon elegáns paraméterelemző mechanizmust biztosít. Nem szükséges manuálisan elemezni `string[] args`. Egyszerűen definiál egy tulajdonságot, és megjegyzésekkel látja el `[Parameter]`, és a Nuke automatikusan kezeli a leképezést a parancssori argumentumokból és konfigurációs fájlokból.

Például nagyon egyszerűen meghatározhatjuk a felépítési konfigurációt:

```csharp
[Parameter("Configuration to build - Default is 'Debug'")]
readonly Configuration BuildConfiguration = IsLocalBuild ? Configuration.Debug : Configuration.Release;

Target Compile => _ => _
    .DependsOn(Restore)
    .Executes(() =>
    {
        // Use BuildConfiguration here; it is type-safe
        DotNetBuild(s => s
            .SetConfiguration(BuildConfiguration)
            .SetProjectFile(SolutionFile));
    });
```

Ez a stílus intuitív és ellenáll a hibáknak.

## Gyakorlati útmutató: Hogyan ültessük át a gyakorlatba egy projektben

Az elmélet önmagában nem vezet sehova. Nézzük meg, hogyan alkalmaztuk ezt a megközelítést a HagiCode projektben.

### 1. Plan the Project Structure

Nem akartuk, hogy a szkriptek összezavarják a projekt gyökerét, és nem akartunk olyan mély könyvtárstruktúrát sem, amely bizonyos Java projektekre hasonlít. Így az összes Nuke-hoz kapcsolódó build fájlt egyetlen helyen helyeztük el `nukeBuild/` könyvtárat.

Az előnyök egyértelműek:

- A projekt gyökér tiszta marad.
- Az építési logika egységes és könnyebben kezelhető marad.
- Az új csapattagok azonnal láthatják: "Itt él a felépítéssel kapcsolatos logika."

### 2. Tervezze meg a világos célfüggőségi láncot

A Targets tervezésénél egy elvet követtünk: **atomiitás plusz függőségi áramlás**.

Minden célnak elég kicsinek kell lennie ahhoz, hogy pontosan egy dolgot tegyen. Például `Clean` csak a fájlokat távolítsa el, és ne kezelje csendesen az oldalsó csomagolást.

Egy ajánlott függőségi folyamat nagyjából így néz ki:

`Clean` -> `Restore` -> `Compile` -> `Test` -> `Pack`

Természetesen ez nem abszolút. Például, ha csak teszteket szeretne futtatni, és nem akar semmit csomagolni, a Nuke lehetővé teszi a végrehajtást `nuke Test` közvetlenül, és automatikusan kezeli a szükségeset `Restore` és `Compile` lépések először.

### 3. Robusztus hibakezelés és naplózás

Mi a legidegesítőbb a szkriptek létrehozásában? Nem egyértelmű hibaüzenetek. Egy sikertelen összeállítás, amely csak az "Error: 1"-et naplózza, elég ahhoz, hogy bárkit megőrjítsen.

A Nuke-ban, mivel közvetlenül használhatjuk a C# kivételkezelést, sokkal nagyobb pontossággal tudjuk rögzíteni és jelenteni a hibákat.

```csharp
Target Publish => _ => _
    .DependsOn(Test)
    .Executes(() =>
    {
        try 
        {
            // Try publishing to NuGet
            DotNetNuGetPush(s => s
                .SetTargetPath(ArtifactPath)
                .SetSource("https://api.nuget.org/v3/index.json")
                .SetApiKey(ApiKey));
        }
        catch (Exception ex)
        {
            Log.Error($"Publishing failed. Team, please check whether the key is correct: {ex.Message}");
            throw; // Make sure the build process exits with a non-zero exit code
        }
    });
```

### 4. Használjon teszteket a minőség védelmére

A build szkriptek is kódok, ami azt jelenti, hogy tesztekre is szükségük van. A Nuke lehetővé teszi számunkra, hogy teszteket írjunk összeépítési munkafolyamatokhoz, így a felépítési logika megváltoztatásakor véletlenül se törjük meg a meglévő kiadási folyamatokat. Ez különösen fontos a folyamatos integrációs csővezetékeknél.
