## 1. alaptéma: Építészeti tervezés és egységes absztrakció

Két adatbázis támogatásához a .NET Core-ban a legfontosabb ötlet az, hogy absztrakciókra, semmint konkrét megvalósításokra támaszkodjunk. Az adatbázis kiválasztását el kell különítenünk az üzleti kódtól, és hagynunk kell, hogy a konfigurációs réteg döntsön.

### Tervezési megközelítés

1.  **Egységes felület**: Minden üzleti logikának a `DbContext` alaposztály vagy egyéni interfész, nem pedig konkrét `PostgreSqlDbContext`.
2.  **Konfigurációvezérelt**: Használja a konfigurációs elemeket `appsettings.json` dinamikusan eldönteni, hogy melyik adatbázis-szolgáltatót kell betölteni az alkalmazás indításakor.
3.  **Funkciók elkülönítése**: Adja hozzá az adaptációs logikát a PostgreSQL-specifikus képességekhez, például a JSONB-hoz, hogy az alkalmazás továbbra is kecsesen leépülhessen SQLite-on.

### Kód implementáció: Dinamikus kontextus konfiguráció

Az ASP.NET Core-ban `Program.cs`, nem szabad keményen kódolnunk `UseNpgsql` vagy `UseSqlite`. Ehelyett el kell olvasnunk a konfigurációt, és dinamikusan kell döntenünk.

Először határozza meg a konfigurációs osztályt:

```csharp
public class DatabaseSettings
{
    public const string SectionName = "Database";
    
    // Database type: PostgreSQL or SQLite
    public string DbType { get; set; } = "PostgreSQL"; 
    
    // Connection string
    public string ConnectionString { get; set; } = string.Empty;
}
```

Ezután regisztrálja a szolgáltatást `Program.cs` konfiguráció alapján:

```csharp
// Read configuration
var databaseSettings = builder.Configuration.GetSection(DatabaseSettings.SectionName).Get<DatabaseSettings>();

// Register DbContext
builder.Services.AddDbContext<ApplicationDbContext>(options =>
{
    if (databaseSettings?.DbType?.ToLower() == "sqlite")
    {
        // SQLite configuration
        options.UseSqlite(databaseSettings.ConnectionString);
        
        // Handling SQLite's concurrent write limitations
        // Note: in production, enabling WAL mode is recommended to improve concurrency performance
    }
    else
    {
        // PostgreSQL configuration (default)
        options.UseNpgsql(databaseSettings.ConnectionString, npgsqlOptions =>
        {
            // Enable JSONB support, which is very useful when handling AI conversation records
            npgsqlOptions.UseJsonNet(); 
        });
        
        // Configure connection pool retry policy
        options.EnableRetryOnFailure(3);
    }
});
```

## 2. alaptéma: A különbségek kezelése és a migrációs stratégia

Bár a PostgreSQL és az SQLite egyaránt támogatja az SQL-szabványt, jelentősen eltérnek az adott képességeikben és viselkedésükben. Ha ezeket a különbségeket nem kezeli körültekintően, könnyen olyan kínos helyzethez vezethet, amikor minden helyileg működik, de a telepítés után meghiúsul.

### 1. JSON-típusok kezelése

A HagiCode-ban nagy mennyiségű promptot és AI metaadatot kell tárolnunk, amely általában JSON-oszlopokat tartalmaz.
- **PostgreSQL**: Van egy natív `JSONB` típus kiváló lekérdezési teljesítménnyel.
- **SQLite**: Nincs natív JSON-típusa (az újabb verziók tartalmazzák a JSON1 kiterjesztést, de az objektumleképezés továbbra is eltérő), ezért az adatok általában SZÖVEG formátumban kerülnek tárolásra.

**Megoldás**:
Az EF Core entitásleképezésben konvertálható típusként konfiguráljuk.

```csharp
protected override void OnModelCreating(ModelBuilder modelBuilder)
{
    base.OnModelCreating(modelBuilder);

    // Configure entity
    modelBuilder.Entity<PromptTemplate>(entity => 
    {
        entity.Property(e => e.Metadata)
              .HasColumnType("jsonb") // PG uses jsonb
              .HasConversion(
                  v => JsonSerializer.Serialize(v, (JsonSerializerOptions)null),
                  v => JsonSerializer.Deserialize<Dictionary<string, object>>(v, (JsonSerializerOptions)null)
              ); 
    });
}
```

Ha SQLite-t használunk, `HasColumnType("jsonb")` figyelmen kívül hagyható, vagy figyelmeztetést válthat ki. Azonban mivel `HasConversion` konfigurálva van, az adatok továbbra is helyesen vannak sorosítva és deszerializálva, mint a SZÖVEG mezőben tárolt karakterláncok, biztosítva a kompatibilitást.

### 2. A migrációs stratégiák szétválasztása

Soha ne próbálja meg egyetlen migrációs parancsfájl-készletet a PostgreSQL-hez és az SQLite-hoz egyszerre működni. Az elsődleges kulcsgenerálási stratégiákban, az indexszintaxisban és az adatbázis egyéb részleteiben mutatkozó különbségek elkerülhetetlenül hibákat okoznak.

**Ajánlott gyakorlat**:
Két migrációs ág vagy projekt fenntartása. A HagiCode fejlesztési munkafolyamatban a következőképpen kezeljük:

1.  **Fejlesztési szakasz**: Főleg SQLite-tal dolgozik. Használja `Add-Migration Init_Sqlite -OutputDir Migrations/Sqlite`.
2.  **Adaptációs szakasz**: Egy szolgáltatás fejlesztése után állítsa át a kapcsolati karakterláncot PostgreSQL-re, és futtassa `Add-Migration Init_Postgres -OutputDir Migrations/Postgres`.
3.  **Automatizálási szkriptek**: Írjon egy egyszerű PowerShell- vagy Bash-szkriptet, hogy automatikusan alkalmazza a megfelelő áttelepítést az aktuális környezeti változó alapján.

```bash
# Pseudocode for simple deployment logic
if [ "$DATABASE_PROVIDER" = "PostgreSQL" ]; then
    dotnet ef database update --project Migrations.Postgres
else
    dotnet ef database update --project Migrations.Sqlite
fi
```

## 3. alaptéma: A HagiCode-ból levont tanulságok a termelésben

Miközben a **HagiCode-ot** egyadatbázisos modellről kétadatbázisos támogatásra alakítottuk át, találtunk néhány buktatót, és összegyűjtöttünk néhány fontos tanulságot, amelyek segíthetnek elkerülni ugyanazokat a hibákat.

### 1. Az egyidejűség és a tranzakciók különbségei

A PostgreSQL szerver-kliens architektúrát használ, és támogatja a nagy egyidejű írásokat hatékony tranzakció-izolációs szintekkel. Az SQLite fájlzárolást használ, így az írási műveletek zárolják a teljes adatbázisfájlt, hacsak nincs engedélyezve a WAL mód.

**Ajánlás**:
Amikor olyan üzleti logikát ír, amely gyakori írásokat foglal magában, például a felhasználó szerkesztési állapotának valós idejű mentését, figyelembe kell vennie az SQLite zárolási modelljét. Amikor a **HagiCode**-ban az OpenSpec együttműködési modult terveztük, bevezettünk egy "egyesítés írás előtt" mechanizmust, hogy csökkentsük a közvetlen adatbázis-írások gyakoriságát, lehetővé téve számunkra, hogy mindkét adatbázison jó teljesítményt tartsunk fenn.

### 2. A kapcsolati karakterláncok életciklus-kezelése

A PostgreSQL-kapcsolat létrehozása viszonylag költséges, és a kapcsolatkészlettől függ. Az SQLite kapcsolatok nagyon könnyűek, de ha nem oldják fel azonnal, a fájlzárak a későbbi műveletek időtúllépését okozhatják.

In `Program.cs`, finomhangolhatjuk az egyes adatbázisok viselkedését:

```csharp
if (databaseSettings?.DbType?.ToLower() == "sqlite")
{
    // SQLite: keeping connections open can improve performance, but watch out for file locks
    options.UseSqlite(connectionString, sqliteOptions =>
    {
        // Set command timeout
        sqliteOptions.CommandTimeout(30);
    });
}
else
{
    // PG: make full use of connection pooling
    options.UseNpgsql(connectionString, npgsqlOptions =>
    {
        npgsqlOptions.MaxBatchSize(100);
        npgsqlOptions.CommandTimeout(30);
    });
}
```
