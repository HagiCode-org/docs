## A GitHub-problémák integrációjának felépítése a semmiből: a HagiCode közvetlen csatlakozási gyakorlata

> Ez a cikk a GitHub Issues HagiCode platformba való integrálásának teljes folyamatát dokumentálja. Megvizsgáljuk, hogyan használhatunk „közvetlen előtérkapcsolat + minimális háttérrendszer” architektúrát a biztonságos OAuth-hitelesítés és a problémák hatékony szinkronizálása érdekében, miközben a háttérrendszert könnyűvé kell tenni.

## Háttér: Miért érdemes integrálni a GitHubot?

Az AI által támogatott fejlesztési platformként a HagiCode alapvető értéke az ötletek és a megvalósítás összekapcsolásában rejlik. A tényleges használat során azonban azt tapasztaltuk, hogy miután a felhasználók kitöltöttek egy javaslatot a HagiCode-ban, gyakran manuálisan kell átmásolniuk a tartalmat a GitHub Issuesbe a projektkövetés érdekében.

Ez több nyilvánvaló fájdalompontot hoz létre:

1.  **Fragmentált munkafolyamat**: A felhasználóknak oda-vissza kell váltaniuk két rendszer között. Az élmény nem zökkenőmentes, és a legfontosabb információk könnyen elveszhetnek a másolás és beillesztés során.
2.  **Kényelmetlen együttműködés**: A csapat többi tagja hozzászokott a GitHubon lévő feladatok ellenőrzéséhez, és nem láthatja közvetlenül a javaslat előrehaladását a HagiCode-on belül.
3.  **Ismételt kézi munka**: Valahányszor egy ajánlatot frissítenek, valakinek manuálisan frissítenie kell a megfelelő problémát a GitHubon, ami szükségtelen karbantartási költségekkel jár.

A probléma megoldása érdekében úgy döntöttünk, hogy bevezetjük a **GitHub Issues Integration** funkciót, amely összekapcsolja a HagiCode-munkameneteket GitHub-tárolókkal, hogy lehetővé tegye az „egykattintásos szinkronizálást”.

## Technikai választás: Frontend Direct Connection vs Backend Proxy

Az integrációs megközelítés kialakításakor két lehetőség állt előttünk: a hagyományos "backend proxy modell" és az agresszívebb "frontend közvetlen kapcsolati modell".

### Megoldás összehasonlítása

A hagyományos **backend proxy modellben** a frontendtől érkező minden kérésnek először át kell mennie a háttérrendszerünkön, amely ezután meghívja a GitHub API-t. Ez centralizálja a logikát, de jelentős terhet ró a háttérrendszerre is:

1.  **Felduzzadó háttérrendszer**: Dedikált GitHub API-kliens-burkolót kell írnunk, és kezelnünk kell az összetett OAuth állapotú gépet is.
2.  **Token kockázat**: A felhasználó GitHub-tokenjét a háttéradatbázisban kell tárolni. Ez még titkosítás mellett is növeli a biztonsági felületet.
3.  **Fejlesztési költség**: Adatbázisáttelepítésekre lenne szükségünk a tokenek tárolásához, és további szinkronizálási szolgáltatásra lenne szükségünk a karbantartáshoz.

Az **frontend közvetlen csatlakozású modell** sokkal könnyebb. Ebben a megközelítésben a háttérrendszert csak a legérzékenyebb "titkos csere" lépéshez (az OAuth visszahíváshoz) használjuk. A token beszerzése után közvetlenül a böngészőben tároljuk `localStorage`. A későbbi műveletek, például a problémák létrehozása és a megjegyzések frissítése közvetlenül az előtérről a GitHubba HTTP-n keresztül kerülnek elküldésre.

| Összehasonlítási dimenzió | Backend proxy modell | Frontend közvetlen kapcsolati modell |
| :--- | :--- | :--- |
| **Háttér összetettsége** | Teljes OAuth-szolgáltatást és GitHub API-klienst igényel | Csak egy OAuth visszahívási végpontra van szüksége |
| **Token kezelés** | Titkosítani kell és az adatbázisban kell tárolni, szivárgási kockázattal | A böngészőben tárolva és csak a felhasználó számára látható |
| **Megvalósítási költség** | Adatbázis-migrációt és többszolgáltatásos fejlesztést igényel | Elsősorban frontend munka |
| **Felhasználói tapasztalat** | Központosított logika, de a szerver késleltetése valamivel magasabb lehet | Rendkívül gyors válaszadás közvetlen GitHub interakcióval |

Mivel gyors integrációt és minimális háttérmódosítást akartunk, **végül a "frontend közvetlen kapcsolati modellt"** választottuk. Ez olyan, mintha "ideiglenes belépőt" adna a böngészőnek. Miután megkapta az engedélyt, a böngésző képes egyedül kezelni a GitHubon lévő dolgokat anélkül, hogy minden alkalommal jóváhagyást kérne a háttérrendszergazdától.

## Alapvető tervezés: adatáramlás és biztonság

Az architektúra kialakítása után meg kellett terveznünk a konkrét adatfolyamot. A szinkronizálási folyamat lényege a token biztonságos beszerzése és hatékony felhasználása.

### Átfogó építészeti diagram

Az egész rendszer három szerepkörre absztrahálható: a böngésző (frontend), a HagiCode háttérrendszer és a GitHub.

```text
+--------------+        +--------------+        +--------------+
| Frontend     |        |   Backend    |        |   GitHub     |
| React        |        |   ASP.NET    |        |   REST API   |
|              |        |              |        |              |
|  +--------+  |        |              |        |              |
|  | OAuth  |--+--------> /callback    |        |              |
|  | Flow   |  |        |              |        |              |
|  +--------+  |        |              |        |              |
|              |        |              |        |              |
|  +--------+  |        |  +--------+  |        |  +--------+  |
|  | GitHub |  +------------> Session |  +----------> Issues |  |
|  | API    |  |        |  |Metadata|  |        |  |        |  |
|  | Direct |  |        |  +--------+  |        |  +--------+  |
|  +--------+  |        |              |        |              |
+--------------+        +--------------+        +--------------+
```

**A lényeg a**: az OAuth-ban csak egy kis lépésnek (a kód tokenre cseréje) kell keresztülmennie a háttérrendszeren. Ezt követően a nehézségek (problémák előidézése) kezelése közvetlenül a frontend és a GitHub között történik.

### Az adatfolyam szinkronizálása részletesen

Amikor a felhasználó rákattint a "Szinkronizálás a GitHubba" gombra a HagiCode felhasználói felületén, egy sor összetett művelet történik:

```text
User clicks "Sync to GitHub"
         │
         ▼
1. Frontend checks localStorage for the GitHub token
         │
         ▼
2. Format issue content (convert the Proposal into Markdown)
         │
         ▼
3. Frontend directly calls the GitHub API to create/update the issue
         │
         ▼
4. Call the HagiCode backend API to update Session.metadata (store issue URL and other info)
         │
         ▼
5. Backend broadcasts the SessionUpdated event via SignalR
         │
         ▼
6. Frontend receives the event and updates the UI to show the "Synced" state
```

### Biztonsági tervezés

Harmadik féltől származó szolgáltatások integrálásakor mindig a biztonság a legfontosabb. A következő szempontokat vettük figyelembe:

1.  **Védekezzen a CSRF-támadások ellen**: Véletlenszerű véletlen létrehozása `state` paramétert az OAuth átirányítás során, és tárolja `sessionStorage`. Szigorúan ellenőrizze az állapotot a visszahívásban, hogy megakadályozza a hamis kéréseket.
2.  **Izolált tokentároló**: A token csak a böngészőben tárolódik `localStorage`. A Same-Origin Policy használatával csak a HagiCode szkriptek tudják olvasni, elkerülve a szerveroldali adatbázis-szivárgás kockázatát, amely a felhasználókat érintené.
3.  **Hibahatárok**: A gyakori GitHub API-hibák (például a 401 lejárt token, a 422 érvényesítési hiba és a 429-es gyakoriság korlátozása) speciális kezelését terveztük, hogy a felhasználók barátságos visszajelzéseket kapjanak.

### 2. Frontend OAuth Flow

Ez a kapcsolat belépési pontja. A szabványos engedélyezési kódfolyamot használjuk.

```typescript
// src/HagiCode.Client/src/services/githubOAuth.ts

// Generate the authorization URL and redirect
export async function generateAuthUrl(): Promise<string> {
  const state = generateRandomString(); // Generate a random string for CSRF protection
  sessionStorage.setItem('hagicode_github_state', state);
  
  const params = new URLSearchParams({
    client_id: clientId,
    redirect_uri: window.location.origin + '/settings?tab=github&oauth=callback',
    scope: ['repo', 'public_repo'].join(' '),
    state: state,
  });
  
  return `https://github.com/login/oauth/authorize?${params.toString()}`;
}

// Handle the code-to-token exchange on the callback page
export async function exchangeCodeForToken(code: string, state: string): Promise<GitHubToken> {
  // 1. Validate state to prevent CSRF
  const savedState = sessionStorage.getItem('hagicode_github_state');
  if (state !== savedState) throw new Error('Invalid state parameter');

  // 2. Call the backend API to exchange the token
  // Note: this must go through the backend because ClientSecret cannot be exposed to the frontend
  const response = await fetch('/api/GitHubOAuth/callback', {
    method: 'POST',
    headers: { 'Content-Type': 'application/json' },
    body: JSON.stringify({ code, state, redirectUri: window.location.origin + '/settings?tab=github&oauth=callback' }),
  });

  if (!response.ok) throw new Error('Failed to exchange token');
  
  const token = await response.json();
  
  // 3. Save into LocalStorage
  saveToken(token);
  return token;
}
```

### 3. GitHub API Client Wrapper

Ha megvan a token, szükségünk van egy szilárd eszközre a GitHub API meghívásához.

```typescript
// src/HagiCode.Client/src/services/githubApiClient.ts

const GITHUB_API_BASE = 'https://api.github.com';

// Core request wrapper
async function githubApi<T>(endpoint: string, options: RequestInit = {}): Promise<T> {
  const token = localStorage.getItem('gh_token');
  if (!token) throw new Error('Not connected to GitHub');
  
  const response = await fetch(`${GITHUB_API_BASE}${endpoint}`, {
    ...options,
    headers: {
      ...options.headers,
      Authorization: `Bearer ${token}`,
      Accept: 'application/vnd.github.v3+json', // Specify the API version
    },
  });
  
  // Error handling logic
  if (!response.ok) {
    if (response.status === 401) throw new Error('GitHub token expired, please reconnect');
    if (response.status === 403) throw new Error('No permission to access this repository or rate limit exceeded');
    if (response.status === 422) throw new Error('Issue validation failed, the title may be duplicated');
    throw new Error(`GitHub API Error: ${response.statusText}`);
  }
  
  return response.json();
}

// Create issue
export async function createIssue(owner: string, repo: string, data: { title: string, body: string, labels: string[] }) {
  return githubApi(`/repos/${owner}/${repo}/issues`, {
    method: 'POST',
    body: JSON.stringify(data),
  });
}
```

### 4. Tartalom formázása és szinkronizálása

Az utolsó lépés a HagiCode munkamenet adatainak átalakítása egy GitHub-probléma formátumába. Kicsit olyan, mint a fordítói munka.

```typescript
// Convert a Session object into a Markdown string
function formatIssueForSession(session: Session): string {
  let content = `# ${session.title}\n\n`;
  content += `**> HagiCode Session:** #${session.code}\n`;
  content += `**> Status:** ${session.status}\n\n`;
  content += `## Description\n\n${session.description || 'No description provided.'}\n\n`;
  
  // If this is a Proposal session, add extra fields
  if (session.type === 'proposal') {
    content += `## Chief Complaint\n\n${session.chiefComplaint || ''}\n\n`;
    // Add a deep link so users can jump back from GitHub to HagiCode
    content += `---\n\n**[View in HagiCode](hagicode://sessions/${session.id})**\n`;
  }
  
  return content;
}

// Main logic when clicking the sync button
const handleSync = async (session: Session) => {
  try {
    const repoInfo = parseRepositoryFromUrl(session.repoUrl); // Parse the repository URL
    if (!repoInfo) throw new Error('Invalid repository URL');

    toast.loading('Syncing to GitHub...');
    
    // 1. Format content
    const issueBody = formatIssueForSession(session);
    
    // 2. Call API
    const issue = await githubApiClient.createIssue(repoInfo.owner, repoInfo.repo, {
      title: `[HagiCode] ${session.title}`,
      body: issueBody,
      labels: ['hagicode', 'proposal', `status:${session.status}`],
    });
    
    // 3. Update Session Metadata (save the issue link)
    await SessionsService.patchApiSessionsSessionId(session.id, {
      metadata: {
        githubIssue: {
          owner: repoInfo.owner,
          repo: repoInfo.repo,
          issueNumber: issue.number,
          issueUrl: issue.html_url,
          syncedAt: new Date().toISOString(),
        }
      }
    });

    toast.success('Synced successfully!');
  } catch (err) {
    console.error(err);
    toast.error('Sync failed, please check your token or network');
  }
};
```

## Összegzés és kitekintés

Ezzel a „frontend közvetlen kapcsolat” megközelítéssel zökkenőmentesen integráltuk a GitHub Issues-t a lehető legkevesebb háttérkóddal.

### Kulcsnyereségek
1.  **Magas fejlesztési hatékonyság**: A háttérben végzett változtatások minimálisak, főként egy extra adatbázismező és egy egyszerű OAuth visszahívási végpont. A legtöbb logika a frontenden fejeződik be.
2.  **Erős biztonság**: A token nem halad át a kiszolgáló adatbázisán, így csökken a szivárgás kockázata.
3.  **Nagyszerű felhasználói élmény**: A kérések közvetlenül a frontendről indulnak, így a válaszsebesség gyors, és nincs szükség háttértovábbításra.

### Amikre figyelni kell
Néhány buktatót érdemes szem előtt tartani a valós telepítés során:
- **OAuth alkalmazás beállításai**: Ne felejtse el beírni a megfelelőt `Authorization callback URL` a GitHub OAuth alkalmazás beállításaiban (általában `http://localhost:3000/settings?tab=github&oauth=callback`).
- **Drátakorlátok**: A GitHub API meglehetősen szigorúan korlátozza a nem hitelesített kéréseket, de tokennel a kvóta általában elegendő (5000 kérés/óra).
- **URL-elemzés**: A felhasználók mindenféle adattár URL-címet megadnak, ezért győződjön meg arról, hogy a szabályos kifejezés megegyezik `.git` utótagok, SSH formátumok és hasonló esetek.

### Jövőbeli fejlesztések
A jelenlegi szolgáltatás továbbra is egyirányú szinkronizálás (HagiCode -> GitHub). A jövőben kétirányú szinkronizálást tervezünk a GitHub Webhookon keresztül. Például, ha egy probléma le van zárva a GitHubon, a HagiCode oldalon a munkamenet állapota is automatikusan frissülhet. Ehhez meg kell jelenítenünk egy webhook végpontot a háttérben, ami egy érdekes következő lépés lesz.
