## TypeScriptről C#-ra: Cross-Language Porting Practice for the Codex SDK

> Leegyszerűsítve ez a cikk is egy kicsit a miénk: rögzíti a hivatalos TypeScript Codex SDK C#-ra való portolásának teljes folyamatát. Ha „kikötőnek” nevezzük, már-már túl könnyen hangzik – ez inkább egy hosszú kaland volt, mert ennek a két nyelvnek nagyon eltérő a személyisége, és meg kellett találnunk a módját, hogyan tudjunk együttműködni.

## Háttér

A Codex az OpenAI által kiadott AI Agent CLI eszköz, és valóban hatékony. A hivatalos csapat TypeScript SDK-t biztosít a `@openai/codex` csomagot. A Codex CLI-vel úgy lép kapcsolatba, hogy meghívja a `codex exec --experimental-json` parancsot és egy JSONL eseményfolyam elemzését.

A probléma az, hogy a HagiCode projektben tiszta .NET környezetben kell használnunk – különösen a C# háttérszolgáltatásokban és asztali alkalmazásokban. Nem tudtunk ésszerűen bevezetni egy Node.js futtatókörnyezetet egy .NET-projektbe csak azért, hogy meghívjunk egy CLI-eszközt. Ez túlságosan nehézkes lenne.

Így két választásunk maradt: fenntartunk egy komplex Node.js hídréteget, vagy magunk építünk egy natív C# SDK-t.

## Alaptartalom

### Építészeti tervezés összehasonlítása

A kód egy az egyhez lefordítása előtt először meg kellett értenünk mindkét SDK felépítését. Mindkét oldalt meg kell értened, mielőtt jól áthelyezhetnéd őket.

A TypeScript SDK alapvető architektúrája így néz ki:

```
Codex (entry class)
  └── CodexExec (executor, manages child processes)
      └── Thread (conversation thread)
          ├── run() / runStreamed() (synchronous/asynchronous execution)
          └── event stream parsing
```

A C# SDK megtartja ugyanazt az építészeti rétegezést, de igazítja a megvalósítás részleteit. Az általános ötlet egyszerű: őrizze meg az API konzisztenciáját, miközben teljes mértékben kihasználja a C# nyelvi funkciókat a megvalósításban.

### Írja be a Rendszerkonverziót

Ez a munka legalapvetőbb és egyben legfontosabb része. Ha az alap gyenge, minden, ami ezután következik, keményebbé válik.

A TypeScript típusrendszere rugalmasabb, mint a C#, és ez egyszerűen tény. Megfelelő leképezési stratégiát kellett találnunk:

| TypeScript | C# | Megjegyzések |
|------------|-----|------|
| `interface` / `type` | `record` | C# használ `record` megváltoztathatatlan adatstruktúrákhoz |
| `string \| null` | `string?` | Nullálható referencia típus |
| `boolean \| undefined` | `bool?` | Nullable Boolean |
| `AsyncGenerator` | `IAsyncEnumerable` | Aszinkron iterátor |

Tipikus példa az **esemény típusú rendszer**. A TypeScript uniótípusokat használ az események meghatározásához:

```typescript
export type ThreadEvent =
  | ThreadStartedEvent
  | TurnStartedEvent
  | TurnCompletedEvent
  | ...
```

A C#-ban öröklődési hierarchiát és mintaillesztést használunk, hogy hasonló hatást érjünk el:

```csharp
public abstract record ThreadEvent(string Type);

public sealed record ThreadStartedEvent(string ThreadId) : ThreadEvent("thread.started");
public sealed record TurnStartedEvent() : ThreadEvent("turn.started");
public sealed record TurnCompletedEvent(Usage Usage) : ThreadEvent("turn.completed");
// ...
```

választottunk `record` helyett `class` mert az eseményobjektumoknak megváltoztathatatlannak kell lenniük, ami megegyezik a TypeScript egyszerű objektumok használatának szándékával. A `sealed` A kulcsszó megakadályozza a további öröklődést, és teret ad a fordítónak az optimalizálásra.

### Kulcsfontosságú portolási pontok

#### 1. Eseményelemző

Az eseményelemzés a teljes SDK magja, mert ez határozza meg, hogy helyesen értjük-e a Codex CLI által visszaadott minden üzenetet. Ha az elemzés hibás, minden azután elpazarolt erőfeszítés.

A TypeScript verziót használja `JSON.parse()` a JSON egyes sorainak elemzéséhez:

```typescript
export function parseEvent(line: string): ThreadEvent {
  const data = JSON.parse(line);
  // Handle different event types...
}
```

A C# verzió használja `System.Text.Json.JsonDocument` helyette:

```csharp
public static ThreadEvent Parse(string line)
{
    using var document = JsonDocument.Parse(line);
    var root = document.RootElement;
    var type = GetRequiredString(root, "type", "event.type");

    return type switch
    {
        "thread.started" => new ThreadStartedEvent(GetRequiredString(root, "thread_id", ...)),
        "turn.started" => new TurnStartedEvent(),
        "turn.completed" => new TurnCompletedEvent(ParseUsage(...)),
        // ...
        _ => new UnknownThreadEvent(type, root.Clone()),
    };
}
```

Van itt egy apró, de fontos trükk: `root.Clone()` szükséges, mert elemek a `JsonDocument` a dokumentum selejtezése után érvénytelenné válik. Meg kell őriznünk egy példányt az ismeretlen eseménytípusokhoz. Ez egyszerűen az egyik különbség a C# JSON-kezelés és a JavaScript között.

#### 2. Folyamatmenedzsment különbségek

Ebben különbözik a leginkább a két SDK. A Node.js és a .NET eltérő futásidejű konvenciókkal rendelkezik, ezért a megvalósításnak alkalmazkodnia kell.

A TypeScript a Node.js-t használja `spawn()` funkció:

```typescript
const child = spawn(this.executablePath, commandArgs, { env, signal });
```

A C# .NET-et használ `System.Diagnostics.Process`:

```csharp
using var process = new Process { StartInfo = startInfo };
process.Start();

// stdin/stdout/stderr must be managed manually
```

Pontosabban, a C# verziónak a következőképpen kell konfigurálnia a folyamatot:

```csharp
var startInfo = new ProcessStartInfo
{
    FileName = _executablePath,
    RedirectStandardInput = true,
    RedirectStandardOutput = true,
    RedirectStandardError = true,
    UseShellExecute = false,
    CreateNoWindow = true,
};
```

A legnagyobb különbség a törlési mechanizmus. TypeScript használ `AbortSignal`, amely a Web API része, és nagyon kényelmesen használható:

```typescript
const child = spawn(cmd, args, { signal: cancellationSignal });
```

C# használ `CancellationToken` helyette:

```csharp
public async IAsyncEnumerable<string> RunAsync(
    CodexExecArgs args,
    [EnumeratorCancellation] CancellationToken cancellationToken = default)
{
    // Check cancellation status inside the loop
    while (!cancellationToken.IsCancellationRequested)
    {
        // Process output...
    }

    // Terminate the process when cancellation is requested
    if (cancellationToken.IsCancellationRequested)
    {
        try { process.Kill(entireProcessTree: true); } catch { }
    }
}
```

Magas szinten ez csak egy újabb példa a Web API ökoszisztéma és a .NET ökoszisztéma közötti különbségre.

#### 3. A konfiguráció szerializálásának megőrzése

Mindkét SDK megvalósítja azt a logikát, amely a JSON-konfigurációt TOML-konfigurációvá alakítja, mivel a Codex CLI elfogadja a TOML-formátumú konfigurációs felülírásokat. Ennek a résznek teljesen konzisztensnek kell maradnia, különben ugyanaz a konfiguráció eltérően fog viselkedni a két SDK-ban.

Ez az a fajta részlet, amiben nem köthet kompromisszumot. A siker vagy a kudarc gyakran az ehhez hasonló részleteken múlik.

### A megvalósítás részletei

#### Projekt szerkezete

A következő projektstruktúrát hoztuk létre:

```
CodexSdk/
├── CodexSdk.csproj
├── Codex.cs           # Entry class
├── CodexThread.cs     # Conversation thread
├── CodexExec.cs       # Executor
├── Events.cs          # Event type definitions
├── Items.cs           # Item type definitions
├── EventParser.cs     # Event parser
├── OutputSchemaTempFile.cs  # Temporary file management
└── ...
```

Ez egy meglehetősen tiszta szerkezet, és ez sokat segített a kikötőben.

#### Használati példa

Az alapvető használat konzisztens a TypeScript SDK-val:

```csharp
using CodexSdk;

// Create a Codex instance
var codex = new Codex();
var thread = codex.StartThread();

// Execute a query
var result = await thread.RunAsync("Summarize this repository.");
Console.WriteLine(result.FinalResponse);
```

A streaming eseménykezelés kihasználja a C# mintaillesztés előnyeit:

```csharp
await foreach (var @event in thread.RunStreamedAsync("Analyze the code."))
{
    switch (@event)
    {
        case ItemCompletedEvent itemCompleted
            when itemCompleted.Item is AgentMessageItem msg:
            Console.WriteLine($"Assistant: {msg.Text}");
            break;
        case TurnCompletedEvent completed:
            Console.WriteLine($"Tokens: in={completed.Usage.InputTokens}");
            break;
        case CommandExecutionItem command:
            Console.WriteLine($"Command: {command.Command}");
            break;
    }
}
```

### Megjegyzések

A megvalósítás során több gyakorlati leckét gyűjtöttünk össze:

1. **Folyamatkezelés**: A C# verziónak a teljes folyamat életciklust manuálisan kell kezelnie, beleértve a folyamatleállítást a törlés során. Használja `Kill(entireProcessTree: true)` hogy a gyermekfolyamatok is megtisztuljanak.

2. **Hibakezelés**: Használjuk `InvalidOperationException` elemzési hibákat dobhat, a hibakezelési stílust a TypeScript SDK-hoz hasonló módon tartva.

3. **Erőforrás-tisztítás**: `OutputSchemaTempFile` megvalósítja `IAsyncDisposable` hogy az ideiglenes fájlok megfelelően megtisztuljanak.

4. **Környezeti változók**: A C# verzió támogatja a folyamatkörnyezeti változók teljes felülírását `CodexOptions.Env`. Ez egy kicsi funkció, de nagyon praktikus.

5. **Platformkülönbségek**: A C#-verzió nem tartalmazza a TypeScript-verzió logikáját az npm-csomagokon belüli binárisok automatikus megtalálásához. Mivel a .NET-projektek általában nem függnek az npm-től, az elérési út a `codex` végrehajtható fájlt a következőn keresztül kell megadni `CODEX_EXECUTABLE` környezeti változó ill `CodexPathOverride`.

## Következtetés

A kiforrott TypeScript SDK C#-ba történő portolása nem csupán szintaktikai konverzió kérdése, hanem mindkét nyelv tervezési filozófiájának megértését is igényli. A TypeScript rugalmassága és a JavaScript ökoszisztéma funkciói, mint pl `AbortSignal` megfelelő hasonmásokra van szükség a C#-ban.

A legfontosabb dolog a következő: **Az API konzisztencia megőrzése többet számít, mint a megvalósítási szintű konzisztencia**. A felhasználókat az érdekli, hogy a felület könnyen használható-e, és nem az, hogy a belső megvalósítás azonos-e. Ez egyszerűen hangzik, de a kompromisszumok meghozatala ítéletet igényel.

Ha Ön egy hasonló többnyelvű porton dolgozik, akkor az a tapasztalatunk, hogy először teljesen megértjük az eredeti SDK-architektúrát, majd modulonként lefordítjuk, végül pedig teljes tesztcsomagot használunk a viselkedési konzisztencia biztosítására. Ezt a fajta munkát nem lehet siettetni.
