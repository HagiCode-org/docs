## A Docker Hub és az Azure ACR közötti képszinkronizálás automatizálása

> Ez a cikk elmagyarázza, hogyan használhatja a GitHub-műveleteket és a képszinkronizáló eszközt a Docker Hub és az Azure Container Registry közötti képszinkronizálás automatizálására, kezelve a kínai és egyes Azure-régiók Docker Hubhoz való lassú hozzáférését, miközben javítja a képfájlok elérhetőségét és a telepítési hatékonyságot az Azure-környezetekben.

## Háttér / Bevezetés

A HagiCode projekt a Docker képfájlokat használja központi futásidejű összetevőként, az elsődleges lemezképeket pedig a Docker Hub tárolja. A projekt növekedésével és az Azure-telepítési igények növekedésével a következő problémákkal találkoztunk:

- Lassú képletöltési sebesség, a Docker Hub-hozzáférés korlátozott Kínában és néhány Azure-régióban
- Egyetlen képforrás egyetlen pont meghibásodási kockázatát hoz létre
- Az Azure Container Registry jobb hálózati teljesítményt és integrációt biztosít, ha az Azure-ban fut

A problémák megoldásához szükségünk volt egy automatizált képszinkronizálási mechanizmusra, amely rendszeresen szinkronizálja a képeket a Docker Hubból az Azure ACR-be, így biztosítva, hogy a felhasználók gyorsabb képletöltési sebességet és magasabb rendelkezésre állást kapjanak az Azure-környezetekben.

## Műszaki megoldások összehasonlítása

A megoldás meghatározásakor több műszaki lehetőséget hasonlítottunk össze:

### 1. képszinkronizáló (végső választás)
- Növekményes szinkronizálás: csak a megváltozott képrétegeket szinkronizálja, jelentősen csökkentve a hálózati átvitelt
- Támogatás folytatása: a szinkronizálás folytatódhat a hálózati megszakítások után
- Párhuzamosság-vezérlés: a konfigurálható munkaszálak javítják a nagy képszinkronizálás hatékonyságát
- Robusztus hibakezelés: beépített újrapróbálkozási mechanizmus hibák esetén (alapértelmezés szerint 3 újrapróbálkozás)
- Könnyű telepítés: egyetlen bináris függőségek nélkül
- Több rendszerleíró adatbázis támogatása: kompatibilis a Docker Hub, az Azure ACR, a Harbor és más szolgáltatásokkal

### 2. Docker CLI
- Nincs növekményes szinkronizálás: a teljes képtartalmat minden alkalommal le kell húzni
- Alacsonyabb hatékonyság: nagy hálózati átviteli mennyiség és hosszabb végrehajtási idő
- Használata egyszerű: ismerős alapján `docker pull` és `docker push` parancsokat

### 3. Azure CLI
- Nagyobb összetettség: Az Azure CLI-hitelesítést konfigurálni kell
- Korlátozott funkcionalitás: `az acr import` hatóköre viszonylag szűk
- Natív integráció: jól integrálható az Azure-szolgáltatásokkal

## Core Solution

A GitHub Actions + képszinkronizáló alapú automatizált megoldást használjuk a Docker Hub és az Azure ACR közötti képek szinkronizálására.

## Megvalósítás lépései

### 1. Előkészítés
- Hozzon létre vagy erősítsen meg egy Azure Container Registry-t az Azure Portalban
- ACR hozzáférési hitelesítő adatok létrehozása (felhasználónév és jelszó)
- Erősítse meg a Docker Hub képtárhoz tartozó hozzáférési engedélyeket

### 2. Konfigurálja a GitHub Secrets funkciót
Adja hozzá a következő titkokat a GitHub-tárhely beállításaihoz:
- AZURE_ACR_USERNAME: Azure ACR felhasználónév
- AZURE_ACR_PASSWORD: Azure ACR-jelszó

### 3. Hozza létre a GitHub Actions munkafolyamatot
Konfigurálja a munkafolyamatot `.github/workflows/sync-docker-acr.yml`:
- Ütemezett trigger: minden nap 00:00 UTC-kor
- Kézi trigger: támogatja `workflow_dispatch`
- Kiegészítő trigger: fut, amikor lökések a `publish` ág a gyors szinkronizáláshoz

### 4. Munkafolyamat végrehajtási folyamata

| lépés | Résztvevő | Akció | Leírás |
| --- | --- | --- | --- |
| 1 | GitHub-műveletek | Indítsa el a munkafolyamatot | Ütemezés, kézi végrehajtás, vagy az `publish` ág |
| 2 | GitHub Actions → képszinkronizáló | Töltse le és futtassa a szinkronizáló eszközt | Belép a tényleges szinkronizálási fázisba |
| 3 | képszinkronizáló → Docker Hub | Képjegyzékek és címkelista lekérése | Beolvassa a metaadatokat a forrástárból |
| 4 | image-syncer → Azure ACR | A meglévő képadatok lekérése a céltárból | Meghatározza a céloldal aktuális állapotát |
| 5 | képszinkronizáló | Hasonlítsa össze a forrás és a cél közötti különbségeket | Azonosítja a szinkronizálásra szoruló képrétegeket |
| 6 | képszinkronizáló → Docker Hub | Húzza ki a megváltozott képrétegeket | Csak a frissítésre szoruló tartalmat viszi át |
| 7 | image-syncer → Azure ACR | Nyomja meg a megváltozott képrétegeket | Befejezi a növekményes szinkronizálást |
| 8 | image-syncer → GitHub Actions | Szinkronizálási statisztikák visszaküldése | Tartalmazza az eredményeket, a különbségeket és a hibainformációkat |
| 9 | GitHub-műveletek | Rögzítse a naplókat és töltse fel a műtermékeket | Támogatja a későbbi ellenőrzést és hibaelhárítást |

## GitHub Actions munkafolyamat megvalósítása

Az alábbiakban látható a ténylegesen használt munkafolyamat-konfiguráció (`.github/workflows/sync-docker-acr.yml`):

```yaml
name: Sync Docker Image to Azure ACR

on:
  schedule:
    - cron: "0 0 * * *" # Every day at 00:00 UTC
  workflow_dispatch: # Manual trigger
  push:
    branches: [publish]

permissions:
  contents: read

jobs:
  sync:
    runs-on: ubuntu-latest

    steps:
      - name: Checkout code
        uses: actions/checkout@v4

      - name: Download image-syncer
        run: |
          # Download the image-syncer binary
          wget https://github.com/AliyunContainerService/image-syncer/releases/download/v1.5.5/image-syncer-v1.5.5-linux-amd64.tar.gz
          tar -zxvf image-syncer-v1.5.5-linux-amd64.tar.gz
          chmod +x image-syncer

      - name: Create auth config
        run: |
          # Generate the authentication config file (YAML format)
          cat > auth.yaml <<EOF
          hagicode.azurecr.io:
            username: "${{ secrets.AZURE_ACR_USERNAME }}"
            password: "${{ secrets.AZURE_ACR_PASSWORD }}"
          EOF

      - name: Create images config
        run: |
          # Generate the image sync config file (YAML format)
          cat > images.yaml <<EOF
          docker.io/newbe36524/hagicode: hagicode.azurecr.io/hagicode
          EOF

      - name: Run image-syncer
        run: |
          # Run synchronization (using the newer --auth and --images parameters)
          ./image-syncer --auth=./auth.yaml --images=./images.yaml --proc=10 --retries=3

      - name: Upload logs
        if: always()
        uses: actions/upload-artifact@v4
        with:
          name: sync-logs
          path: image-syncer-*.log
          retention-days: 7
```

## Konfiguráció részletei

### 1. Kiváltó feltételek
- Ütemezett trigger: `cron: "0 0 * * *"` minden nap 00:00 UTC-kor indul
- Kézi trigger: `workflow_dispatch` lehetővé teszi a felhasználók számára, hogy manuálisan futtassák a GitHub felhasználói felületről
- Nyomd meg a ravaszt: `push: branches: [publish]` a gyors szinkronizálás érdekében elindítja a kioldó ágra történő nyomást

### 2. Hitelesítési konfiguráció (`auth.yaml`)
```yaml
hagicode.azurecr.io:
  username: "${{ secrets.AZURE_ACR_USERNAME }}"
  password: "${{ secrets.AZURE_ACR_PASSWORD }}"
```

### 3. Képszinkronizálás konfigurációja
```yaml
docker.io/newbe36524/hagicode: hagicode.azurecr.io/hagicode
```
Ez a konfiguráció azt jelenti, hogy az összes címke innen: `docker.io/newbe36524/hagicode` szinkronizálásra kerül `hagicode.azurecr.io/hagicode`.

### 4. képszinkronizáló paraméterek
- `--auth=./auth.yaml`: a hitelesítési konfigurációs fájl elérési útja
- `--images=./images.yaml`: a képszinkronizálási konfigurációs fájl elérési útja
- `--proc=10`: használ `10` párhuzamos dolgozók
- `--retries=3`: újrapróbálkozási hibák `3` alkalommal

## Megjegyzések és bevált gyakorlatok

### 1. Biztonsági ajánlások
- Az Azure ACR jelszavak rendszeres cseréje (ajánlott 90 naponta)
- Használjon dedikált ACR-szolgáltatásfiókot, csak leküldéses engedéllyel
- Figyelje az ACR hozzáférési naplóit, hogy időben észlelje a rendellenes hozzáférést
- Ne adjon ki hitelesítő adatokat a naplókban
- Ne tároljon hitelesítő adatokat a kódtárban

### 2. Teljesítmény optimalizálás
- Állítsa be a `--proc` paraméter: a párhuzamosság hangolása a hálózati sávszélesség alapján (ajánlott `5-20`)
- A szinkronizálás időtartamának figyelése: ha a szinkronizálás túl sokáig tart, fontolja meg a párhuzamosság csökkentését
- Rendszeresen tisztítsa meg a naplókat: állítson be egy ésszerű értéket `retention-days` érték (`7` jelenleg)

### 3. Hibaelhárítás

#### 1. probléma: Hitelesítési hiba
```
Error: failed to authenticate to hagicode.azurecr.io
```
Megoldás:
1. Ellenőrizze, hogy a GitHub Secrets megfelelően van-e konfigurálva
2. Ellenőrizze, hogy az Azure ACR-jelszó lejárt-e
3. Győződjön meg arról, hogy az ACR szolgáltatásfiók engedélyei megfelelőek

#### 2. probléma: Hálózati időtúllépés
```
Error: timeout waiting for response
```
Megoldás:
1. Ellenőrizze a hálózati kapcsolatot
2. Csökkentse az egyidejűleg dolgozók számát (a `--proc` paraméter)
3. Várja meg, amíg a hálózat helyreáll, és futtassa újra a munkafolyamatot

#### 3. probléma: Hiányos képszinkronizálás
```
Warning: some tags failed to sync
```
Megoldás:
1. Ellenőrizze a szinkronizálási naplókat a sikertelen címkék azonosításához
2. Futtassa újra a munkafolyamatot manuálisan az újbóli szinkronizáláshoz
3. Győződjön meg arról, hogy a Docker Hub forrásképe egészséges

### 4. Monitoring és riasztás
- A munkafolyamat állapotának ellenőrzéséhez rendszeresen ellenőrizze a Műveletek oldalt
- Engedélyezze a GitHub értesítéseit, hogy a munkafolyamat-hibákat azonnal jelentse
- Az Azure ACR-tárhelyhasználat figyelése
- Rendszeresen ellenőrizze a címkék konzisztenciáját a nyilvántartásokban

## Gyakran Ismételt Kérdések és megoldások

### 1. kérdés: Hogyan szinkronizálhatok bizonyos címkéket az összes címke helyett?

Módosítsa a `images.yaml` konfigurációs fájl:
```yaml
# Sync only the latest and v1.0 tags
docker.io/newbe36524/hagicode:latest: hagicode.azurecr.io/hagicode:latest
docker.io/newbe36524/hagicode:v1.0: hagicode.azurecr.io/hagicode:v1.0
```

### 2. kérdés: Hogyan szinkronizálhatok több képtárat?

Több sort adjon hozzá `images.yaml`:
```yaml
docker.io/newbe36524/hagicode: hagicode.azurecr.io/hagicode
docker.io/newbe36524/another-image: hagicode.azurecr.io/another-image
```

### 3. kérdés: Hogyan próbálkozhatok újra, ha a szinkronizálás sikertelen?
- Automatikus újrapróbálkozás: a képszinkronizáló beépített újrapróbálkozási mechanizmust tartalmaz (`3` alapértelmezés szerint újra próbálkozik)
- Kézi újrapróbálkozás: kattintson `Re-run all jobs` a GitHub-műveletek oldalon

### 4. kérdés: Hogyan tekinthetem meg a szinkronizálás részletes folyamatát?
- Tekintse meg a valós idejű naplókat a Műveletek oldalon
- Töltse le a `sync-logs` műtermék a teljes naplófájlok ellenőrzéséhez
- A naplófájlok tartalmazzák az egyes címkék szinkronizálási állapotát és átviteli sebességét

### 5. kérdés: Mennyi ideig tart a szinkronizálás?
- Kezdeti teljes szinkronizálás: általában `10-30` perc a kép méretétől függően
- Inkrementális szinkronizálás: általában `2-5` perc, ha a képváltozások kicsik
- Az időzítés a hálózati sávszélességtől, a képmérettől és a párhuzamossági beállításoktól függ

## Javasolt bővítmények

### 1. Szinkronizálási értesítések hozzáadása
Értesítési lépés hozzáadása a munkafolyamathoz:
```yaml
- name: Notify on success
  if: success()
  run: |
    echo "Docker images synced successfully to Azure ACR"
```

### 2. Képcímkeszűrés megvalósítása
Címkeszűrési logika hozzáadása a munkafolyamathoz:
```yaml
- name: Filter tags
  run: |
    # Sync only tags prefixed with v
    echo "docker.io/newbe36524/hagicode:v* : hagicode.azurecr.io/hagicode:v*" > images.yaml
```

### 3. Adjon hozzá egy szinkronizálási statisztikai jelentést
```yaml
- name: Generate report
  if: always()
  run: |
    echo "## Sync Report" >> $GITHUB_STEP_SUMMARY
    echo "- Total tags: $(grep -c 'synced' image-syncer-*.log)" >> $GITHUB_STEP_SUMMARY
    echo "- Sync time: ${{ steps.sync.outputs.duration }}" >> $GITHUB_STEP_SUMMARY
```
