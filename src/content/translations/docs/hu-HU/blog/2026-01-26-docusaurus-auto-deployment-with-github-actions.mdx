## GitHub oldalak automatikus telepítési támogatásának hozzáadása a HagiCode-hoz

> Ennek a projektnek a kódneve eredetileg PCode volt, most pedig hivatalosan is átnevezték HagiCode-ra. Ez a cikk bemutatja, hogyan vezettük be az automatizált statikus webhelytelepítést, így a tartalom közzététele olyan egyszerűvé válik, mint az ivóvíz.

## Háttér / Bevezetés

A HagiCode fejlesztése során egy nagyon gyakorlati problémába ütköztünk: a dokumentációink és javaslataink bővülésével egyre sürgetőbbé vált a tartalom hatékony kezelése és bemutatása. Úgy döntöttünk, hogy a GitHub oldalakat használjuk statikus webhelyünk tárolására, de ennek manuális felépítése és telepítése túlságosan nehézkes volt. Minden változtatás helyi összeállítást, csomagolást, majd kézi indítást igényelt `gh-pages` ág. Ez nemcsak nem volt hatékony, hanem könnyen el is hibázta.

A probléma megoldásához, főként azért, mert meg akartuk magunkat megspórolni, szükségünk volt egy automatizált üzembe helyezési munkafolyamatra. Ez a cikk részletesen leírja, hogyan adtuk hozzá a GitHub Actions-alapú automatikus telepítési támogatást a HagiCode projekthez, így továbbra is a tartalomírásra koncentrálhatunk, a többit pedig az automatizálás intézheti.

## Célelemzés

Mielőtt belevágnánk a megvalósításba, először tisztáznunk kell, hogy ennek a feladatnak valójában mit is kell megvalósítania. Hiszen a fejsze élezése nem késlelteti a tűzifa aprítását.

### Alapkövetelmények

1.  **Automatikus összeállítás**: Automatikusan elindítja az összeállítási munkafolyamatot, amikor a kódot a `main` ág.
2.  **Automatikus üzembe helyezés**: Sikeres összeállítás után automatikusan telepítse a generált statikus fájlokat a GitHub-oldalakra.
3.  **Környezeti konzisztencia**: Győződjön meg arról, hogy a CI-környezet illeszkedik a helyi összeállítási környezethez, hogy elkerülje a klasszikus „helyileg működik, de online meghibásodik” zavarát.

### Technikai választás

Mivel a HagiCode a Docusaurusra, egy nagyon népszerű React statikus webhelygenerátorra épül, a GitHub Actions segítségével elérhetjük ezt a célt.

## A GitHub-műveletek munkafolyamatának konfigurálása

A GitHub Actions a GitHub által biztosított CI/CD szolgáltatás. A munkafolyamat-fájlok YAML formátumú definiálásával az adattáron belül mindenféle automatizálási feladatot testreszabhatunk.

### Munkafolyamat-fájl létrehozása

Létre kell hoznunk egy új konfigurációs fájlt a `.github/workflows` például a projekt gyökérkönyvtárában `deploy.yml`. Ha a könyvtár még nem létezik, először hozza létre.

Ennek a konfigurációs fájlnak a logikája a következő:

1.  **Indítási feltétel**: Hallgassa meg `push` események a `main` ág.
2.  **Futókörnyezet**: A legújabb Ubuntu kiadás.
3.  **Építési lépések**:
    -   Nézze meg a kódot
    -   Telepítse a Node.js-t
    -   Függőségek telepítése (`npm install`)
    -   Statikus fájlok létrehozása (`npm run build`)
4.  **Bevezetési lépés**: Használja a hivatalos `action-gh-pages` művelet a build kimenetének a `gh-pages` ág.

### Kulcs konfigurációs kód

Alább látható a konfigurációs sablon, amelyet végül elfogadtunk:

```yaml
name: Deploy to GitHub Pages

# Trigger condition: run when code is pushed to the main branch
on:
  push:
    branches:
      - main
    # Add path filters if needed, for example only build when docs change
    # paths:
    #   - 'docs/**'
    #   - 'package.json'

# Set permissions, which is important for GitHub Pages deployment
permissions:
  contents: read
  pages: write
  id-token: write

# Concurrency control: cancel older builds for the same branch
concurrency:
  group: "pages"
  cancel-in-progress: false

jobs:
  build:
    runs-on: ubuntu-latest
    steps:
      - name: Checkout
        uses: actions/checkout@v4
        # Note: fetch-depth: 0 is required, otherwise the build version may be inaccurate
        with:
          fetch-depth: 0

      - name: Setup Node
        uses: actions/setup-node@v4
        with:
          node-version: 20 # Keep this consistent with the local development environment
          cache: 'npm'     # Enabling cache can speed up the build process

      - name: Install dependencies
        run: npm ci
        # Use npm ci instead of npm install because it is faster and stricter, which makes it better for CI

      - name: Build website
        run: npm run build
        env:
          # Configure environment variables here if your site build requires them
          # NODE_ENV: production
          # PUBLIC_URL: /your-repo-name
          
      - name: Upload artifact
        uses: actions/upload-pages-artifact@v3
        with:
          path: ./build # Default Docusaurus output directory

  deploy:
    environment:
      name: github-pages
      url: ${{ steps.deployment.outputs.page_url }}
    runs-on: ubuntu-latest
    needs: build
    steps:
      - name: Deploy to GitHub Pages
        id: deployment
        uses: actions/deploy-pages@v4
```

## Gyakori buktatók a megvalósítás során

A gyakorlatban több problémába ütköztünk. Abban a reményben osztom meg itt, hogy mások elkerülhetik, vagy legalább előre elkészíthetik a megoldásokat.

### 1. GitHub token engedélyekkel kapcsolatos problémák

Amikor először beállítottuk a dolgokat, a telepítés folyamatosan meghiúsult egy 403-as (Tiltott) esetén. Sok ásás után azt találtuk, hogy a GitHub alapértelmezett `GITHUB_TOKEN` nem volt jogosultsága írni a Pagesnek.

**Megoldás**: Az adattárban `Settings` -> `Actions` -> `General` -> `Workflow permissions`, feltétlenül válassza ki az **„Írási és olvasási engedélyek”** lehetőséget.

### 2. Rossz Build Directory Path

A Docusaurus beépített statikus fájlokat helyez el a `build` könyvtár alapértelmezés szerint. Előfordulhat azonban, hogy egyes projektek eltérően konfigurálhatók. Például a Create React App alapértelmezett értéke `build`, míg a Vite alapértelmezés szerint `dist`. Ha az Actions azt jelenti, hogy nem találja a kimeneti fájlokat, ellenőrizze a kimeneti útvonalat `docusaurus.config.js`.

### 3. Alútvonal-problémák

Ha a tárhely nem a felhasználó kezdőlapja, akkor nem az `username.github.io`, hanem egy projektoldal, mint pl `username.github.io/project-name`, akkor konfigurálnia kell `baseUrl`.

In `docusaurus.config.js`:

```javascript
module.exports = {
  // ...
  url: 'https://hagicode.com', // Your Hagicode URL
  baseUrl: '/', // Deploy from the root path
  // ...
};
```
Ezt a részletet könnyű figyelmen kívül hagyni. Ha helytelenül van konfigurálva, előfordulhat, hogy az oldal üres képernyőként töltődik be, mert az eszköz elérési útjait nem lehet feloldani.

## Az eredmény ellenőrzése

Minden konfigurálás és a kód lenyomása után a GitHub repository **Actions** lapjára léphetünk, és megnézhetjük a műsort.

A munkafolyamat futása közben sárga kört fog látni, és ha zöldre vált, az sikert jelent. Ha pirosra vált, kattintson rá, és ellenőrizze a naplókat. A legtöbb esetben ott nyomon követheti a problémát, általában elírást vagy helytelen útvonal-konfigurációt.

Ha az építés sikeres, látogassa meg `https://<your-username>.github.io/<repository-name>/` és látni fogja a frissen telepített webhelyet.

## Összegzés

A GitHub Actions bevezetésével sikeresen automatizáltuk a HagiCode dokumentációs webhely telepítését. Ezzel nemcsak a korábban a kézi műveletekre fordított időt takaríthatja meg, de ami még fontosabb, szabványosítja a kiadási munkafolyamatot. Mostantól függetlenül attól, hogy melyik csapattag frissíti a dokumentációt, miután a módosítást egyesítette `main`, a legújabb tartalom néhány perccel később elérhető lesz.

**Alapvető előnyök**:
- **Nagyobb hatékonyság**: A „kézi csomagolás és kézi feltöltés” helyett a „kód egyenlő a kiadással” kategóriára váltottunk.
- **Kevesebb hiba**: Kiküszöböltük az emberi hiba lehetőségét a kézi műveletek során.
- **Jobb fejlesztői élmény**: A fejlesztők jobban összpontosíthatnak a tartalom minőségére, ahelyett, hogy az unalmas telepítési lépések elterelnék őket.

Bár a CI/CD konfigurálása eleinte kissé problémás, különösen az összes jogosultság és elérési út részlete mellett, ez egyszeri befektetés, jelentős hosszú távú megtérüléssel. Erősen ajánlom, hogy minden statikus webhelyprojekt alkalmazzon hasonló automatizált munkafolyamatot.
