## Elegáns új felhasználói beépítés kialakítása a React projektekben: a HagiCode driver.js gyakorlata

> Amikor a felhasználók először nyitják meg a terméket, valóban tudják, hol kezdjék? Ebben a cikkben arról szeretnék beszélni, hogy miként használtuk a driver.js-t az új felhasználók beléptetésére a HagiCode projektben. Tekintsük ezt egy kis gyakorlati példának a beszélgetés elindításához.

## Háttér

Te kerültél már ebbe a helyzetbe? Egy új felhasználó regisztrál a termékére, megnyitja az oldalt, és azonnal elveszettnek tűnik. Körülnéznek, nem tudják, mire kattintsanak, vagy mit tegyenek ezután. Fejlesztőként gyakran azt feltételezzük, hogy a felhasználók csak "önmagukban fedezik fel a felfedezést", mert végül is az emberi kíváncsiság határtalan. A valóság azonban más: a legtöbb felhasználó perceken belül csendben távozik, ha nem találja a megfelelő belépési pontot, mintha a történet hirtelen kezdődne és ugyanolyan természetesen végződne.

A probléma megoldásának fontos módja az új felhasználók beépítése, de a megfelelő felépítés nem olyan egyszerű. Egy jó bevezető rendszernek a következőkre van szüksége:

- Pontosan keresse meg az oldalelemeket, és jelölje ki őket
- Támogatja a többlépcsős bevezetési folyamatokat
- Emlékezzen a felhasználó választására (befejezés vagy kihagyás)
- Ne befolyásolja az oldal teljesítményét és a normál interakciót
- A kódstruktúra legyen világos és könnyen karbantartható

A HagiCode készítése közben ugyanebbe a kihívásba ütköztünk. A HagiCode egy mesterséges intelligencia kódolási asszisztens, és alapvető munkafolyamata egy OpenSpec munkafolyamat, amely így néz ki: "a felhasználó létrehoz egy javaslatot -> az AI létrehoz egy tervet -> a felhasználó felülvizsgálja -> az AI végrehajtja." Azon felhasználók számára, akik először találkoznak ezzel a fogalommal, a munkafolyamat teljesen új, ezért a gyors kezdéshez komoly alapokra van szükségük. Az új dolgokhoz mindig kell egy kis idő, amíg megszokja.

## Miért választottuk a driver.js-t?

A technikai értékelési szakasz során számos főbb beépített könyvtárat vizsgáltunk meg. Mindegyiknek megvolt a maga erőssége:

- **Intro.js**: Erőteljes, de viszonylag nagy, és a stílus testreszabása meglehetősen bonyolult
- **Shepherd.js**: Jól megtervezett API, de a mi használati esetünkhöz kissé túl "nehéz"
- **driver.js**: Könnyű, tömör, intuitív API, és jól működik a React ökoszisztémában

Végül a driver.js-t választottuk. Nem volt különösebben drámai oka. A választást elsősorban a következő szempontok indokolták:

1. **Könnyű**: A központi könyvtár kicsi, és nem növeli jelentősen a köteg méretét
2. **Egyszerű API**: A konfiguráció egyértelmű és intuitív, így könnyen átvehető
3. **Rugalmas**: Támogatja az egyéni pozicionálást, stílust és interakciós viselkedést
4. **Dinamikus import**: Igény szerint betölthető anélkül, hogy ez befolyásolná az első képernyő teljesítményét

A technológia kiválasztásával ritkán van általánosan legjobb válasz. Általában csak a legmegfelelőbb lehetőség van.

## Műszaki megvalósítás

### Alapkonfiguráció

A driver.js nagyon intuitív konfigurációs modellel rendelkezik. Íme a HagiCode projektben használt alapvető konfiguráció:

```typescript
import { driver } from 'driver.js';
import 'driver.js/dist/driver.css';

const newConversationDriver = driver({
  allowClose: true,           // Allow users to close the guide
  animate: true,              // Enable animations
  overlayClickBehavior: 'close', // Close the guide when the overlay is clicked
  disableActiveInteraction: false, // Keep elements interactive
  showProgress: false,        // Do not show the progress bar (we manage progress ourselves)
  steps: guideSteps           // Array of guide steps
});
```

A beállítások mögött a következő indokok állnak:

- `allowClose: true` - Tartsa tiszteletben a felhasználó választását, és ne kényszerítse arra, hogy befejezze az útmutatót
- `disableActiveInteraction: false` - Egyes lépések valódi felhasználói műveleteket igényelnek, például gépelést, így az interakciót nem lehet letiltani
- `overlayClickBehavior: 'close'` - Gyors kilépési lehetőséget biztosít a felhasználóknak

### Államirányítás

A folyamatos belépési állapot kritikus. Nem akarjuk minden alkalommal újraindítani az útmutatót, amikor az oldal frissül, mert az gyorsan bosszantó lesz. HagiCode használ `localStorage` útmutató állapot kezeléséhez:

```typescript
export type GuideState = 'pending' | 'dismissed' | 'completed';

export interface UserGuideState {
  session: GuideState;
  detailGuides: Record<string, GuideState>;
}

// Read state
export const getUserGuideState = (): UserGuideState => {
  const state = localStorage.getItem('userGuideState');
  return state ? JSON.parse(state) : { session: 'pending', detailGuides: {} };
};

// Update state
export const setUserGuideState = (state: UserGuideState) => {
  localStorage.setItem('userGuideState', JSON.stringify(state));
};
```

Három állapotot határozunk meg:

- `pending`: Az útmutató még folyamatban van, és a felhasználó nem fejezte be vagy nem hagyta ki
- `dismissed`: A felhasználó proaktívan bezárta az útmutatót
- `completed`: A felhasználó minden lépést végrehajtott

A Javaslat részletező oldalának bevezetéséhez a részletesebb állapotkövetést is támogatjuk a következőn keresztül `detailGuides` térképen, mert egy Pályázat több szakaszon mehet keresztül – a vázlat, az áttekintés és a végrehajtás befejeződött –, és minden szakaszhoz más-más útmutatás szükséges. A dolgok állása mindig változik, és a bevezetőnek ezt tükröznie kell.

### Célelem kiválasztása

A driver.js CSS-választókat használ a célelemek megtalálásához. A HagiCode egy egyszerű konvenciót követ: használjon egyedit `data-guide` attribútum a belépési célok megjelölésére:

```typescript
const steps = [
  {
    element: '[data-guide="launch"]',
    popover: {
      title: 'Start a New Conversation',
      description: 'Click here to create a new conversation session...'
    }
  }
];
```

Összetevőkben ez így néz ki:

```typescript
<button data-guide="launch" onClick={handleLaunch}>
  New Conversation
</button>
```

Ennek a megközelítésnek az előnyei a következők:

- Kerülje az ütközést az üzleti stílus osztályneveivel
- Tiszta szemantika, így azonnal megtudhatja, hogy az elem kapcsolódik a beépítéshez
- Könnyebb kezelni és következetesen karbantartani

### Dinamikus importoptimalizálás

Mivel a bevezető funkcióra csak bizonyos esetekben van szükség, például a felhasználó első látogatásakor, dinamikus importálást használunk a kezdeti betöltési teljesítmény optimalizálására:

```typescript
const initNewUserGuide = async () => {
  // Dynamically import driver.js
  const { driver } = await import('driver.js');
  await import('driver.js/dist/driver.css');

  // Initialize the guide
  const newConversationDriver = driver({
    // ...configuration
  });

  newConversationDriver.drive();
};
```

Ily módon a driver.js és stíluslapja csak szükség esetén töltődik be, és nincs hatással az első képernyő teljesítményére. Kevesen várnak olyasmire, amire még nincs is szükségük.

### Elemmegjelenítési újrapróbálkozási mechanizmus

A React alkalmazásokban előfordulhat, hogy a bevezető elem még nem fejezte be a megjelenítést, például azért, mert az aszinkron adatok még mindig betöltődnek. Ennek kezelésére a HagiCode egy újrapróbálkozási mechanizmust valósít meg:

```typescript
const waitForElement = (selector: string, maxRetries = 10, interval = 100) => {
  let retries = 0;
  return new Promise<HTMLElement>((resolve, reject) => {
    const checkElement = () => {
      const element = document.querySelector(selector) as HTMLElement;
      if (element) {
        resolve(element);
      } else if (retries < maxRetries) {
        retries++;
        setTimeout(checkElement, interval);
      } else {
        reject(new Error(`Element not found: ${selector}`));
      }
    };
    checkElement();
  });
};
```

Hívja ezt a függvényt az útmutató inicializálása előtt, hogy megbizonyosodjon arról, hogy a célelem már létezik. Néha megéri várni egy kicsit.

## A legjobb gyakorlatok összefoglalása

A HagiCode gyakorlati tapasztalatai alapján az alábbiakban bemutatunk néhány bevált gyakorlatot:

### 1. A beléptetésből könnyen ki kell lépni

Ne kényszerítse a felhasználókat a belépés befejezésére. Egyes felhasználók természetüknél fogva felfedezők, és inkább maguktól találnak rá a dolgokra. Adjon meg egy világos „Kihagyás” gombot, emlékezzen a választásukra, és legközelebb ne szakítsa meg őket.

### 2. A bevezető tartalom legyen rövid és éles

Minden bevezetési lépésnek egyetlen célra kell összpontosítania:

- **Cím**: Rövid és világos, ideális esetben nem több néhány szónál
- **Leírás**: Térjen közvetlenül a lényegre, és mondja el a felhasználónak, hogy mi ez, és miért fontos

Kerülje a hosszadalmas magyarázatokat. A felhasználók figyelme korlátozott a belépés során, és minél többet mond, annál kevésbé valószínű, hogy elolvassa.

### 3. Tartsa stabilan a kiválasztókat

Használjon stabil elemjelzőket, amelyek nem gyakran változnak. A szokás `data-guide` attribútum jó választás. Kerülje az osztálynevektől vagy a DOM-struktúrától függően, mert ezek könnyen megváltoztathatók a feldolgozás során. A kód folyamatosan változik, de néhány horgonynak lehetőség szerint stabilnak kell maradnia.

### 4. Tesztelje a felkészülést

A HagiCode teljes teszteseteket tartalmaz a bevezető funkcióhoz:

```typescript
describe('NewUserConversationGuide', () => {
  it('should initialize guide state correctly', () => {
    const state = getUserGuideState();
    expect(state.session).toBe('pending');
  });

  it('should update guide state correctly', () => {
    setUserGuideState({ session: 'completed', detailGuides: {} });
    const state = getUserGuideState();
    expect(state.session).toBe('completed');
  });
});
```

A tesztek segítenek annak biztosításában, hogy a refaktorálás véletlenül se törje meg a belépési viselkedést. Senki sem akarja, hogy egy kis kódmódosítás csendesen károsítsa a korábban működő funkcionalitást.

### 5. Teljesítményoptimalizálás

- Használjon dinamikus importálást a bevezető könyvtár lusta betöltéséhez
- Kerülje a bevezető logika inicializálását, miután a felhasználó már kitöltötte az útmutatót
- Vegye figyelembe az animációk teljesítményre gyakorolt hatását, és szükség esetén kapcsolja ki őket alacsonyabb kategóriás eszközökön

A teljesítmény, mint sok minden az életben, megérdemel egy kis gondos költségvetés-tervezést.
