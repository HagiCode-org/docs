## A StreamJsonRpc mély integrációja és gyakorlati használata a HagiCode-ban

> Ez a cikk részletesen elmagyarázza, hogy a HagiCode (korábban PCode) projekt hogyan integrálta sikeresen a Microsoft StreamJsonRpc kommunikációs könyvtárát az eredeti egyéni JSON-RPC megvalósítás helyére, miközben megoldotta az integrációs folyamat során felmerülő technikai problémákat és architekturális kihívásokat.

## Háttér

A StreamJsonRpc a Microsoft hivatalosan karbantartott JSON-RPC kommunikációs könyvtára .NET és TypeScript számára, amely erős típusbiztonságáról, automatikus proxygenerálásáról és kiforrott kivételkezelési mechanizmusairól ismert. A HagiCode projektben a csapat úgy döntött, hogy integrálja a StreamJsonRpc-t annak érdekében, hogy az ACP-n (Agent Communication Protocol) keresztül kommunikálhasson olyan külső mesterséges intelligencia-eszközökkel, mint az iflow CLI és az OpenCode CLI, miközben kiküszöböli a karbantartási költségeket és a korábbi egyedi JSON-RPC implementáció által okozott esetleges hibákat. Az integrációs folyamat azonban a JSON-RPC streamingjére jellemző kihívásokba ütközött, különösen a proxycél-kötés és az általános paraméter-felismerés kezelésekor.

Hogy ezeket a fájdalmas pontokat kezeljük, merész döntést hoztunk: újjáépítjük a teljes összeállítási rendszert az alapoktól kezdve. Ennek a döntésnek a hatása még nagyobb lehet, mint várta, és hamarosan részletesen kifejtem.

## Elemzés

A jelenlegi projekt az ACP-protokoll-integráció kritikus szakaszában van, és a következő technikai nehézségekkel és építészeti kihívásokkal néz szembe:

### 1. Az egyéni megvalósítás korlátai
Az eredeti JSON-RPC megvalósítás itt található `src/HagiCode.ClaudeHelper/AcpImp/` és olyan összetevőket tartalmaz, mint pl `JsonRpcEndpoint` és `ClientSideConnection`. Ennek az egyéni kódbázisnak a karbantartása költséges, és hiányoznak belőle a kiforrott könyvtárak fejlett képességei, például a folyamatjelentés és a törlési támogatás.

### 2. StreamJsonRpc integrációs akadályok
Amikor megpróbálja áttelepíteni a meglévőt `CallbackProxyTarget` mintát a StreamJsonRpc-hez, azt találtuk, hogy a `_rpc.AddLocalRpcTarget(target)` metódus nem tudta felismerni a proxy mintán keresztül létrehozott célokat. Pontosabban, a StreamJsonRpc nem tudta automatikusan felosztani az általános típus tulajdonságait `T` RPC metódusparaméterekbe, ami a szerveroldal meghibásodását okozza az ügyfél által kezdeményezett metódushívások feldolgozása során.

### 3. Zavaros építészeti rétegezés
A meglévő `ClientSideConnection` keveri a szállítási réteget (WebSocket/Stdio), a protokollréteget (JSON-RPC) és az üzleti réteget (ACP Agent interfész), ami tisztázatlan felelősségekhez vezet. Ezenkívül hiányzik a metóduskötések hiánya is `AcpAgentCallbackRpcAdapter`.

### 4. Hiányzó naplók
A WebSocket szállítási rétegből hiányzik a nyers JSON tartalomnaplózás, ami megnehezíti annak meghatározását az RPC hibakeresés során, hogy a probléma sorozatosításból vagy hálózatból ered-e.

## Megoldás

A fenti problémák megoldása érdekében a következő szisztematikus megoldást alkalmaztuk, három dimenzióból optimalizálva: architektúra átalakítás, könyvtárintegráció és továbbfejlesztett hibakeresés.

### 1. Teljesen migráljon a StreamJsonRpc-re

#### Távolítsa el a régi kódot
Törölje `JsonRpcEndpoint.cs`, `AgentSideConnection.cs`, és a kapcsolódó egyedi szerializációs konverterek, mint pl `JsonRpcMessageJsonConverter`.

#### Integrálja a Hivatalos Könyvtárat
Mutassa be a `StreamJsonRpc` NuGet csomagot, és használja `JsonRpc` osztály az alapvető kommunikációs logika kezelésére.

#### Absztrakt a szállítási réteg
Határozza meg a `IAcpTransport` interfész mindkettő kezelésére `WebSocket` és `Stdio` szállítási módok egységes módon, biztosítva a protokollréteg leválasztását a szállítási rétegtől.

```csharp
// Definition of the IAcpTransport interface
public interface IAcpTransport
{
    Task SendAsync(string message, CancellationToken cancellationToken = default);
    Task<string> ReceiveAsync(CancellationToken cancellationToken = default);
    Task CloseAsync(CancellationToken cancellationToken = default);
}

// WebSocket transport implementation
public class WebSocketTransport : IAcpTransport
{
    private readonly WebSocket _webSocket;

    public WebSocketTransport(WebSocket webSocket)
    {
        _webSocket = webSocket;
    }

    // Implement send and receive methods
    // ...
}

// Stdio transport implementation
public class StdioTransport : IAcpTransport
{
    private readonly StreamReader _reader;
    private readonly StreamWriter _writer;

    public StdioTransport(StreamReader reader, StreamWriter writer)
    {
        _reader = reader;
        _writer = writer;
    }

    // Implement send and receive methods
    // ...
}
```

### 2. Javítsa ki a proxy célfelismerési problémáit

#### Elemezze `CallbackProxyTarget`
Vizsgálja meg a meglévő dinamikus proxygenerálási logikát, és azonosítsa annak kiváltó okát, hogy a StreamJsonRpc miért nem ismeri fel. A legtöbb esetben ez azért történik, mert a proxyobjektum nem teszi közzé nyilvánosan a tényleges metódus-aláírásokat, vagy a StreamJsonRpc által nem támogatott paramétertípusokat használ.

#### Refaktor paraméter átadása
Az általános tulajdonságok felosztása explicit RPC-módszer-paraméterekre. A dinamikus tulajdonságokra hagyatkozás helyett definiáljon konkrét kérés/válasz DTO-kat (adatátviteli objektumokat), hogy a StreamJsonRpc a reflexión keresztül helyesen tudja felismerni a metódusaláírásokat.

```csharp
// Original generic property approach
public class CallbackProxyTarget<T>
{
    public Func<T, Task> Callback { get; set; }
}

// Refactored concrete method approach
public class ReadTextFileRequest
{
    public string FilePath { get; set; }
}

public class ReadTextFileResponse
{
    public string Content { get; set; }
}

public interface IAcpAgentCallback
{
    Task<ReadTextFileResponse> ReadTextFileAsync(ReadTextFileRequest request);
    // Other methods...
}
```

#### Használja `Attach` Ahelyett `AddLocalRpcTarget`
Egyes összetett forgatókönyvek esetén manuálisan proxyzik a `JsonRpc` tárgy és kezelés `RpcConnection` rugalmasabb lehet, mint egy helyi cél közvetlen hozzáadása.

### 3. Végezze el a Method Binding és erősebb naplózást

#### Megvalósítani `AcpAgentCallbackRpcAdapter`
Győződjön meg arról, hogy ez az összetevő kifejezetten megvalósítja a StreamJsonRpc proxy interfészt és leképezi az ACP protokoll által meghatározott metódusokat, mint pl. `ReadTextFileAsync`, a StreamJsonRpc visszahíváskezelőkhöz.

#### Naplózás integrálása
Fogja el és rögzítse a JSON-RPC kérések és válaszok nyers szövegét a WebSocket vagy Stdio üzenetfeldolgozási folyamatban. Használja `ILogger` a nyers hasznos adat kiadása az elemzés előtt és a szerializálás után, így a formázási problémák könnyebben diagnosztizálhatók.

```csharp
// Transport wrapper with enhanced logging
public class LoggingAcpTransport : IAcpTransport
{
    private readonly IAcpTransport _innerTransport;
    private readonly ILogger<LoggingAcpTransport> _logger;

    public LoggingAcpTransport(IAcpTransport innerTransport, ILogger<LoggingAcpTransport> logger)
    {
        _innerTransport = innerTransport;
        _logger = logger;
    }

    public async Task SendAsync(string message, CancellationToken cancellationToken = default)
    {
        _logger.LogTrace("Sending message: {Message}", message);
        await _innerTransport.SendAsync(message, cancellationToken);
    }

    public async Task<string> ReceiveAsync(CancellationToken cancellationToken = default)
    {
        var message = await _innerTransport.ReceiveAsync(cancellationToken);
        _logger.LogTrace("Received message: {Message}", message);
        return message;
    }

    public async Task CloseAsync(CancellationToken cancellationToken = default)
    {
        _logger.LogDebug("Closing connection");
        await _innerTransport.CloseAsync(cancellationToken);
    }
}
```

### 4. Refaktor építészeti rétegezés

#### szállítási réteg (`AcpRpcClient`)
Csomagolja be a StreamJsonRpc kapcsolatot, és tegye felelőssé `InvokeAsync` hívások és kapcsolatok életciklus-kezelése.

```csharp
public class AcpRpcClient : IDisposable
{
    private readonly JsonRpc _rpc;
    private readonly IAcpTransport _transport;

    public AcpRpcClient(IAcpTransport transport)
    {
        _transport = transport;
        _rpc = new JsonRpc(new StreamRpcTransport(transport));
        _rpc.StartListening();
    }

    public async Task<TResponse> InvokeAsync<TResponse>(string methodName, object parameters)
    {
        return await _rpc.InvokeAsync<TResponse>(methodName, parameters);
    }

    public void Dispose()
    {
        _rpc.Dispose();
        _transport.Dispose();
    }

    // StreamRpcTransport is the StreamJsonRpc adapter for IAcpTransport
    private class StreamRpcTransport : IDuplexPipe
    {
        // Implement the IDuplexPipe interface
        // ...
    }
}
```

#### Protokoll réteg (`IAcpAgentClient` / `IAcpAgentCallback`)
Határozzon meg egyértelmű kliens-ügynök és ügynök-ügyfél interfészt. Távolítsa el a ciklikus gyári mintát `Func<IAcpAgent, IAcpClient>` és cserélje ki függőségi injekcióra vagy közvetlen visszahívás regisztrációra.

## Gyakorlat

A StreamJsonRpc bevált gyakorlatai és a projekt tapasztalatai alapján az alábbiak a legfontosabb ajánlások a megvalósításhoz:

### 1. Az erősen beírt DTO-k jobbak, mint a dinamikus objektumok
A StreamJsonRpc fő előnye az erős gépelésben rejlik. Ne használja `dynamic` vagy `JObject` paraméterek átadására. Ehelyett definiáljon explicit C# POCO osztályokat paraméterként minden RPC metódushoz. Ez nem csak a proxy cél felismerési problémáját oldja meg, hanem a fordítási időben elkapja a típushibákat is.

Példa: cserélje ki az általános tulajdonságokat `CallbackProxyTarget` konkrét osztályokkal, mint pl `ReadTextFileRequest` és `WriteTextFileRequest`.

### 2. A metódusnevek kifejezetten deklarálása
Használja a `[JsonRpcMethod]` attribútumot az RPC metódusnevek explicit megadásához ahelyett, hogy az alapértelmezett metódusnév leképezésre hagyatkozna. Ez megakadályozza az elnevezési stílusbeli eltérések, például a PascalCase és a camelCase által okozott hívási hibákat.

```csharp
public interface IAcpAgentCallback
{
    [JsonRpcMethod("readTextFile")]
    Task<ReadTextFileResponse> ReadTextFileAsync(ReadTextFileRequest request);
    
    [JsonRpcMethod("writeTextFile")]
    Task WriteTextFileAsync(WriteTextFileRequest request);
}
```

### 3. Használja ki a kapcsolódási állapot visszahívásait
A StreamJsonRpc biztosítja a `JsonRpc.ConnectionLost` eseményt. Feltétlenül figyelnie kell erre az eseményre, hogy kezelje a váratlan folyamatkilépéseket vagy a hálózati kapcsolat megszakadásait, ami időszerűbb, mintha csak az Orleans Grain hiba észlelésére hagyatkozna.

```csharp
_rpc.ConnectionLost += (sender, e) =>
{
    _logger.LogError("RPC connection lost: {Reason}", e.ToString());
    // Handle reconnection logic or notify the user
};
```

### 4. Rögzítse a naplókat rétegenként
- **Nyomon követési szint**: Rögzítse a teljes nyers JSON-kérést/választ.
- **Hibakeresési szint**: Metódushívási veremek és paraméterösszegzések rögzítése.
- **Megjegyzés**: Győződjön meg arról, hogy a naplók nem tartalmaznak érzékeny engedélyezési tokeneket vagy Base64-kódolású nagy fájlokat.

### 5. Kezelje a streaming átvitel különleges természetét
A StreamJsonRpc natívan támogatja `IAsyncEnumerable`. A streaming prompt válaszok ACP-hez való implementálásakor használja a `IAsyncEnumerable` közvetlenül az egyéni oldalszámozási logika helyett. Ez nagymértékben leegyszerűsítheti a streaming feldolgozásához szükséges kód mennyiségét.

```csharp
public interface IAcpAgentCallback
{
    [JsonRpcMethod("streamText")]
    IAsyncEnumerable<string> StreamTextAsync(StreamTextRequest request);
}
```

### 6. Adapter minta
Tartsa `ACPSession` és `ClientSideConnection` külön. `ACPSession` az Orleans-i államkezelésre és az üzleti logikára kell összpontosítania, például az üzenetsorba állításra, és a StreamJsonRpc kapcsolatobjektumot összetételen keresztül kell használnia, nem pedig örökléssel.

A StreamJsonRpc átfogó integrálásával a HagiCode projekt sikeresen kezelte az eredeti egyedi megvalósítás magas karbantartási költségeit, funkcionális korlátait és építészeti rétegződési zavarait. A legfontosabb fejlesztések a következők:

1. A dinamikus tulajdonságok lecserélése erősen tipizált DTO-kra, javítva a karbantarthatóságot és a megbízhatóságot
