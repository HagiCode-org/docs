## Elegante Onboarding fuer neue Nutzer in React-Projekten: HagiCodes driver.js-Praxis

> Wenn Nutzer Ihr Produkt zum ersten Mal oeffnen, wissen sie wirklich, wo sie anfangen sollen? In diesem Artikel moechte ich etwas darueber erzaehlen, wie wir driver.js fuer das Onboarding neuer Nutzer im HagiCode-Projekt eingesetzt haben. Betrachten Sie es als ein kleines praktisches Beispiel, um das Gespraech zu beginnen.

## Hintergrund

Sind Ihnen schon einmal folgende Situation begegnet? Ein neuer Nutzer registriert sich fuer Ihr Produkt, oeffnet die Seite und wirkt sofort verloren. Er schaut sich um, unsicher, worauf er klicken oder was er als Naechstes tun soll. Als Entwickler nehmen wir oft an, dass Nutzer einfach „auf eigene Faust erkunden" werden, schliesslich ist die menschliche Neugier grenzenlos. Die Realitaet sieht anders aus: Die meisten Nutzer verlassen die Seite stillschweigend innerhalb von Minuten, wenn sie den richtigen Einstiegspunkt nicht finden.

Das Onboarding neuer Nutzer ist ein wichtiger Weg, dieses Problem zu loesen, aber es gut zu bauen, ist nicht so einfach. Ein gutes Onboarding-System muss:

- Seitenelemente praezise lokalisieren und hervorheben
- Mehrstufige Onboarding-Fluesse unterstuetzen
- Die Wahl des Nutzers merken (abgeschlossen oder uebersprungen)
- Die Seitenleistung und normale Interaktion nicht beeintraechtigen
- Eine klare und leicht wartbare Codestruktur beibehalten

Beim Bau von HagiCode standen wir vor derselben Herausforderung. HagiCode ist ein KI-Programmierassistent, und sein Kern-Workflow ist ein OpenSpec-Workflow, der so aussieht: „Der Nutzer erstellt einen Proposal -> Die KI generiert einen Plan -> Der Nutzer prueft ihn -> Die KI fuehrt ihn aus." Fuer Nutzer, die diesem Konzept zum ersten Mal begegnen, ist der Workflow voellig neu, sodass sie ein solides Onboarding benoetigen, um schnell starten zu koennen. Neue Dinge brauchen immer etwas Gewoehnung.

Der in diesem Artikel vorgestellte Ansatz stammt aus unserer praktischen Erfahrung im [HagiCode](https://hagicode.com)-Projekt. HagiCode ist ein Claude-basierter KI-Programmierassistent, der Entwicklern hilft, Programmiieraufgaben effizienter durch den OpenSpec-Workflow zu erledigen. ## Warum wir driver.js waehlten

Waehrend der technischen Evaluierungsphase prueften wir mehrere gaengige Onboarding-Bibliotheken. Jede hatte ihre eigenen Staerken:

- **Intro.js**: Leistungsfähig, aber relativ umfangreich, und die Stilanpassung ist etwas komplex
- **Shepherd.js**: Gut gestaltete API, aber fuer unseren Anwendungsfall etwas zu „schwer"
- **driver.js**: Leichtgewichtig, praegnante, intuitive API und funktioniert gut im React-Ökosystem

Letztendlich waehlten wir driver.js. Es gab keinen besonders dramatischen Grund. Die Wahl kam hauptsaechlich auf folgende Ueberlegungen hinaus:

1. **Leichtgewichtig**: Die Kernbibliothek ist klein und erhöht die Bundle-Groesse nicht erheblich
2. **Einfache API**: Die Konfiguration ist klar und intuitiv, leicht zu erlernen
3. **Flexibel**: Unterstuetzt benutzerdefinierte Positionierung, Styling und Interaktionsverhalten
4. **Dynamischer Import**: Kann bei Bedarf geladen werden, ohne die Erstbildschirm-Leistung zu beeintraechtigen

Bei der Technologiewahl gibt es selten eine universell beste Antwort. Meistens gibt es nur die Option, die am besten passt.

## Technische Implementierung

### Kernkonfiguration

driver.js hat ein sehr intuitives Konfigurationsmodell. Hier ist die Kernkonfiguration, die wir im HagiCode-Projekt verwenden:

```typescript
import { driver } from 'driver.js';
import 'driver.js/dist/driver.css';

const newConversationDriver = driver({
  allowClose: true,           // Nutzern erlauben, die Fuehrung zu schliessen
  animate: true,              // Animationen aktivieren
  overlayClickBehavior: 'close', // Fuehrung schliessen, wenn auf das Overlay geklickt wird
  disableActiveInteraction: false, // Elemente interaktiv halten
  showProgress: false,        // Fortschrittsbalken nicht anzeigen (wir verwalten den Fortschritt selbst)
  steps: guideSteps           // Array der Fuehrungsschritte
});
```

Die Begründung fuer diese Einstellungen:

- `allowClose: true` - Die Wahl des Nutzers respektieren und nicht zwingen, die Fuehrung zu beenden
- `disableActiveInteraction: false` - Einige Schritte erfordern echte Nutzeraktionen wie die Texteingabe, daher kann die Interaktion nicht deaktiviert werden
- `overlayClickBehavior: 'close'` - Einen schnellen Ausweg bieten

### Zustandsverwaltung

Das Beibehalten des Onboarding-Zustands ist entscheidend. Wir wollen nicht bei jedem Seitenreload die Fuehrung neu starten, da das schnell nervig wird. HagiCode verwendet `localStorage` zur Verwaltung des Fuehrungszustands:

```typescript
export type GuideState = 'pending' | 'dismissed' | 'completed';

export interface UserGuideState {
  session: GuideState;
  detailGuides: Record<string, GuideState>;
}

// Zustand lesen
export const getUserGuideState = (): UserGuideState => {
  const state = localStorage.getItem('userGuideState');
  return state ? JSON.parse(state) : { session: 'pending', detailGuides: {} };
};

// Zustand aktualisieren
export const setUserGuideState = (state: UserGuideState) => {
  localStorage.setItem('userGuideState', JSON.stringify(state));
};
```

Wir definieren drei Zustaende:

- `pending`: Die Fuehrung laeuft noch, der Nutzer hat sie nicht abgeschlossen oder uebersprungen
- `dismissed`: Der Nutzer hat die Fuehrung aktiv geschlossen
- `completed`: Der Nutzer hat alle Schritte abgeschlossen

Fuer das Onboarding der Proposal-Detailseite unterstuetzen wir darueber hinaus eine feingranularere Zustandsverfolgung ueber die `detailGuides`-Map, da ein Proposal mehrere Stufen durchlaufen kann - Entwurf, Pruefung und Ausfuehrung abgeschlossen - und jede Stufe eine andere Fuehrung benoetigt. Der Zustand der Dinge aendert sich immer, und das Onboarding sollte das widerspiegeln.

### Zielelementauswahl

driver.js verwendet CSS-Selektoren, um Zielelemente zu lokalisieren. HagiCode folgt einer einfachen Konvention: ein benutzerdefiniertes `data-guide`-Attribut wird verwendet, um Onboarding-Ziele zu markieren:

```typescript
const steps = [
  {
    element: '[data-guide="launch"]',
    popover: {
      title: 'Neues Gespraech starten',
      description: 'Klicken Sie hier, um eine neue Gespraechssession zu erstellen...'
    }
  }
];
```

In Komponenten sieht das so aus:

```typescript
<button data-guide="launch" onClick={handleLaunch}>
  Neues Gespraech
</button>
```

Die Vorteile dieses Ansatzes sind:

- Vermeidet Konflikte mit geschäftlichen Stil-Klassennamen
- Klare Semantik, sodass sofort erkennbar ist, dass das Element mit dem Onboarding zusammenhaengt
- Leichter zu verwalten und konsistent zu halten

### Dynamischer Import zur Optimierung

Da die Onboarding-Funktion nur in bestimmten Szenarien benoetigt wird, wie beim ersten Besuch eines Nutzers, verwenden wir dynamische Importe, um die anfaengliche Ladeleistung zu optimieren:

```typescript
const initNewUserGuide = async () => {
  // driver.js dynamisch importieren
  const { driver } = await import('driver.js');
  await import('driver.js/dist/driver.css');

  // Fuehrung initialisieren
  const newConversationDriver = driver({
    // ...Konfiguration
  });

  newConversationDriver.drive();
};
```

Auf diese Weise werden driver.js und sein Stylesheet nur bei Bedarf geladen und beeintraechtigen die Erstbildschirm-Leistung nicht. Niemand geniesst es, auf etwas zu warten, das er noch gar nicht braucht.

## Onboarding-Flow-Design

HagiCode implementiert zwei Onboarding-Pfade, die die Kernszenarien der Nutzer abdecken.

### Session-Onboarding (10 Schritte)

Dieser Onboarding-Pfad hilft Nutzern, den gesamten Ablauf von der Erstellung eines Gespraechs bis zur Einreichung ihres ersten vollstaendigen Proposals abzuschliessen:

1. **launch** - Fuehrung starten und den Button „Neues Gespraech" vorstellen
2. **compose** - Den Nutzer anleiten, eine Anfrage im Eingabefeld einzutippen
3. **send** - Den Nutzer anleiten, auf den Senden-Button zu klicken
4. **proposal-launch-readme** - Den Nutzer anleiten, ein README-Proposal zu erstellen
5. **proposal-compose-readme** - Den Nutzer anleiten, die README-Anfrage zu bearbeiten
6. **proposal-submit-readme** - Den Nutzer anleiten, das README-Proposal einzureichen
7. **proposal-launch-agents** - Den Nutzer anleiten, ein `AGENTS.md`-Proposal zu erstellen
8. **proposal-compose-agents** - Den Nutzer anleiten, die `AGENTS.md`-Anfrage zu bearbeiten
9. **proposal-submit-agents** - Den Nutzer anleiten, das `AGENTS.md`-Proposal einzureichen
10. **proposal-wait** - Erklaeren, dass die KI verarbeitet und der Nutzer einen Moment warten sollte

Die Idee hinter diesem Pfad ist, dass Nutzer den Kern-Workflow von HagiCode durch zwei echte Proposal-Erstellungsaufgaben firsthand erleben - eines fuer `README` und eines fuer `AGENTS.md`. Es ist ein grosser Unterschied, von einem Workflow zu hoeren und ihn selbst durchzugehen.

### Proposal-Detail-Onboarding (3 Schritte)

Wenn Nutzer die Proposal-Detailseite betreten, loest HagiCode die entsprechende Fuehrung basierend auf dem aktuellen Zustand des Proposals aus:

1. **drafting** (Entwurfsstadium) - Den Nutzer anleiten, den von der KI generierten Plan zu pruefen
2. **reviewing** (Pruefungsstadium) - Den Nutzer anleiten, den Plan auszufuehren
3. **executionCompleted** (Abgeschlossenes Stadium) - Den Nutzer anleiten, den Plan zu archivieren

Das kennzeichnende Merkmal dieser Fuehrung ist, dass sie zustandsgesteuert ist: sie entscheidet dynamisch, welcher Onboarding-Schritt angezeigt wird, basierend auf dem tatsaechlichen Zustand des Proposals. Dinge aendern sich, und das Onboarding sollte sich mit ihnen aendern.

Der Screenshot unten zeigt die Proposal-Detailseite im Onboarding-Zustand waehrend der Entwurfsphase:

![Proposal-Detail-Onboarding: Zunaechst den Plan waehrend des Entwurfs generieren](/article-images/new-user-guide-with-driverjs/1775022103729-2cb761ccff49.png)

In dieser Phase fokussiert die Fuehrung die Aufmerksamkeit des Nutzers auf die Schlüsselaktion der Planerstellung, sodass er nicht wissen muss, was er beim ersten Betreten der Detailseite tun soll.

### Element-Render-Wiederholungsmechanismus

In React-Anwendungen ist das Onboarding-Zielelement moeglicherweise noch nicht fertig gerendert, beispielsweise weil asynchrone Daten noch laden. Um dies zu handhaben, implementiert HagiCode einen Wiederholungsmechanismus:

```typescript
const waitForElement = (selector: string, maxRetries = 10, interval = 100) => {
  let retries = 0;
  return new Promise<HTMLElement>((resolve, reject) => {
    const checkElement = () => {
      const element = document.querySelector(selector) as HTMLElement;
      if (element) {
        resolve(element);
      } else if (retries < maxRetries) {
        retries++;
        setTimeout(checkElement, interval);
      } else {
        reject(new Error(`Element nicht gefunden: ${selector}`));
      }
    };
    checkElement();
  });
};
```

Rufen Sie diese Funktion vor der Initialisierung der Fuehrung auf, um sicherzustellen, dass das Zielelement bereits existiert. Manchmal lohnt es sich, etwas laenger zu warten.

## Zusammenfassung der bewaehrten Praktiken

Basierend auf HagiCodes praktischer Erfahrung sind hier einige wichtige bewaehrte Praktiken:

### 1. Onboarding sollte einfach zu verlassen sein

Zwingen Sie Nutzer nicht, das Onboarding abzuschliessen. Manche Nutzer sind von Natur aus Entdecker und ziehen es vor, Dinge selbst herauszufinden. Bieten Sie einen klaren „Ueberspringen"-Button an, merken Sie sich ihre Wahl und stoeren Sie sie beim naechsten Mal nicht erneut.

### 2. Onboarding-Inhalte kurz und praegnant halten

Jeder Onboarding-Schritt sollte sich auf ein einzelnes Ziel konzentrieren:

- **Titel**: Kurz und klar, idealerweise nicht laenger als ein paar Worte
- **Beschreibung**: Auf den Punkt kommen und dem Nutzer sagen, was das ist und warum es wichtig ist

Vermeiden Sie langatmige Erklaerungen. Die Aufmerksamkeit der Nutzer waehrend des Onboardings ist begrenzt, und je mehr Sie sagen, desto unwahrscheinlicher ist es, dass sie es lesen.

### 3. Selektoren stabil halten

Verwenden Sie stabile Elementmarkierungen, die sich nicht oft aendern. Das benutzerdefinierte `data-guide`-Attribut ist eine gute Wahl. Vermeiden Sie die Abhaengigkeit von Klassennamen oder DOM-Struktur, da diese sich bei Refactors leicht aendern koennen. Code aendert sich staendig, aber einige Anker sollten moeglichst stabil bleiben.

### 4. Ihr Onboarding testen

HagiCode enthaelt vollstaendige Testfaelle fuer die Onboarding-Funktion:

```typescript
describe('NewUserConversationGuide', () => {
  it('sollte Fuehrungszustand korrekt initialisieren', () => {
    const state = getUserGuideState();
    expect(state.session).toBe('pending');
  });

  it('sollte Fuehrungszustand korrekt aktualisieren', () => {
    setUserGuideState({ session: 'completed', detailGuides: {} });
    const state = getUserGuideState();
    expect(state.session).toBe('completed');
  });
});
```

Tests stellen sicher, dass Refactoring nicht versehentlich das Onboarding-Verhalten beeintraechtigt. Niemand moechte, dass eine kleine Codeaenderung eine zuvor funktionierende Funktion leise zerstoert.

### 5. Leistungsoptimierung

- Verwenden Sie dynamische Importe, um die Onboarding-Bibliothek Lazy zu laden
- Vermeiden Sie die Initialisierung der Onboarding-Logik, nachdem der Nutzer die Fuehrung bereits abgeschlossen hat
- Beruecksichtigen Sie die Leistungsauswirkungen von Animationen und deaktivieren Sie sie bei weniger leistungsfaehigen Geraeten bei Bedarf

Leistung, wie vieles im Leben, verdient ein wenig sorgfaeltige Budgetierung.

## Zusammenfassung

Das Onboarding neuer Nutzer ist ein wichtiger Bestandteil der Verbesserung der Produkt-Benutzererfahrung. Im HagiCode-Projekt verwendeten wir driver.js, um ein vollstaendiges Onboarding-System zu bauen, das den gesamten Workflow von der Session-Erstellung bis zur Proposal-Ausfuehrung abdeckt.

Die Kernpunkte, die wir durch diesen Artikel teilen moechten, sind:

1. **Technologieauswahl sollte den tatsaechlichen Beduerfnissen entsprechen**: driver.js ist nicht die maechtigste Option, aber die beste Passung fuer uns
2. **Zustandsverwaltung ist entscheidend**: Verwenden Sie `localStorage`, um den Onboarding-Zustand beizubehalten und Nutzer nicht wiederholt zu stoeren
3. **Onboarding-Design sollte fokussiert bleiben**: Jeder Schritt sollte ein Problem loesen, nicht mehr
4. **Codestruktur sollte klar bleiben**: Trennen Sie Onboarding-Konfiguration, Zustandsverwaltung und UI-Logik, um die Wartung zu erleichtern

Wenn Sie das Onboarding neuer Nutzer in Ihrem eigenen Projekt hinzufuegen, hoffe ich, dass die praktische Erfahrung in diesem Artikel hilft. An dieser Art von Technologie gibt es nichts besonders Mystisches. Weiter versuchen, weiter lernen und zusammenfassen, und die Dinge werden allmaehlich leichter.
