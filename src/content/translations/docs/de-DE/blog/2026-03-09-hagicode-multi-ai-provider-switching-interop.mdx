## HagiCode Multi-AI-Provider-Umschaltung und Interoperabilität – Umsetzungsplan

Im modernen Entwickler-Ökosystem müssen Entwickler häufig verschiedene KI-gestützte Code-Assistenten einsetzen, um ihre Arbeit zu unterstützen. Anthropic's Claude Code CLI und OpenAI's Codex CLI haben jeweils ihre Staerken: Claude zeichnet sich durch herausragendes Code-Verständnis und die Verarbeitung langer Kontexte aus, während Codex bei der Code-Generierung und Werkzeugnutzung glänzt.

Dieser Artikel beleuchtet eingehend, wie das HagiCode-Projekt einen nahtlosen Wechsel und eine Interoperabilität zwischen mehreren KI-Anbietern erreicht, einschließlich des zentralen Architekturentwurfs, wichtiger Implementierungsdetails und praktischer Überlegungen.

## Hintergrund

### Problemkreis

Die zentrale Herausforderung des HagiCode-Projekts besteht darin, mehrere KI-CLIs auf derselben Plattform zu unterstützen, damit Benutzer:

1. Flexibel zwischen KI-Anbietern je nach Bedarf wechseln können
2. Die Sitzungskontinuität beim Anbieterwechsel aufrechterhalten
3. Die API-Unterschiede verschiedener CLIs hinter einer gemeinsamen Abstraktion vereinheitlichen
4. Erweiterungspunkte für die zukünftige Integration neuer KI-Anbieter bereitstellen

### Technische Herausforderungen

1. **Vereinheitlichung von Schnittstellenunterschieden**: Claude Code CLI wird über Kommandozeilenaufrufe gestartet, während Codex CLI einen JSON-Ereignisstrom verwendet
2. **Verarbeitung von Streaming-Antworten**: Beide Anbieter unterstützen Streaming-Antworten, jedoch mit unterschiedlichen Datenformaten
3. **Werkzeugaufruf-Semantik**: Claude und Codex unterscheiden sich in der Darstellung von Werkzeugaufrufen und deren Lebenszyklusverwaltung
4. **Sitzungslebenszyklus**: Das System muss die Erstellung, Wiederherstellung und Beendigung von Sitzungen für jeden Anbieter korrekt verwalten

## Analyse

### Architekturentwurfsansatz

HagiCode verwendet das **Provider-Muster** in Kombination mit dem **Factory-Muster**, um den KI-Dienstaufruf zu abstrahieren. Die Kernideen dieses Entwurfs sind:

1. **Vereinheitlichte Schnittstellenabstraktion**: Definition der `IAIProvider`-Schnittstelle als gemeinsame Abstraktion für alle KI-Anbieter
2. **Factory-Erstellung von Instanzen**: Verwendung von `AIProviderFactory` zur dynamischen Erstellung der entsprechenden Anbieterinstanz basierend auf dem Typ
3. **Intelligente Auswahlogik**: Verwendung von `AIProviderSelector` zur automatischen Auswahl des am besten geeigneten Anbieters basierend auf Szenario und Konfiguration
4. **Sitzungszustandsverwaltung**: Persistierung der Bindungsbeziehung zwischen Sitzungen und CLI-Threads in der Datenbank

### Schlüsselkomponenten

| Komponente | Verantwortung | Sprache |
|------|--------|------|
| `IAIProvider` | Vereinheitlichte Anbieter-Schnittstelle | C# |

## Lösung

### 1. Kernschnittstellenentwurf

Die **`IAIProvider`-Schnittstelle** definiert die vereinheitlichte Anbieterabstraktion:

```csharp
public interface IAIProvider
{
    /// <summary>
    /// Anzeigename des Anbieters
    /// </summary>
    string Name { get; }

    /// <summary>
    /// Ob Streaming-Antworten unterstützt werden
    /// </summary>
    bool SupportsStreaming { get; }

    /// <summary>
    /// Beschreibung der Anbieterfähigkeiten
    /// </summary>
    ProviderCapabilities Capabilities { get; }

    /// <summary>
    /// Einen einzelnen KI-Anfrage ausführen
    /// </summary>
    Task<AIResponse> ExecuteAsync(AIRequest request, CancellationToken cancellationToken = default);

    /// <summary>
    /// Eine Streaming-KI-Anfrage ausführen
    /// </summary>
    IAsyncEnumerable<AIStreamingChunk> StreamAsync(AIRequest request, CancellationToken cancellationToken = default);

    /// <summary>
    /// Anbieter-Konnektivität und Reaktionsfähigkeit prüfen
    /// </summary>
    Task<ProviderTestResult> PingAsync(CancellationToken cancellationToken = default);

    /// <summary>
    /// Eine Nachricht mit eingebettetem Befehl senden
    /// </summary>
    IAsyncEnumerable<AIStreamingChunk> SendMessageAsync(
        AIRequest request,
        string? embeddedCommandPrompt = null,
        CancellationToken cancellationToken = default);
}
```

Wesentliche Merkmale dieses Schnittstellenentwurfs:
- **Vereinheitlichtes Anfrage-/Antwortmodell**: Alle Anbieter verwenden dieselben Typen `AIRequest` und `AIResponse`
- **Streaming-Unterstützung**: Standardisierung der Streaming-Ausgabe über `IAsyncEnumerable<AIStreamingChunk>`
- **Fähigkeitsbeschreibung**: `ProviderCapabilities` beschreibt die vom Anbieter unterstützten Funktionen (Streaming, Werkzeuge, maximale Tokens usw.)
- **Eingebettete Befehle**: `SendMessageAsync` unterstützt das Einbetten von OpenSpec-Befehlen in Prompts

### 3. Factory-Muster-Implementierung

Die **`AIProviderFactory`** ist für die Erstellung und Verwaltung von Anbieterinstanzen zuständig:

```csharp
public class AIProviderFactory : IAIProviderFactory
{
    private readonly ConcurrentDictionary<AIProviderType, IAIProvider> _cache;
    private readonly IOptions<AIProviderOptions> _options;
    private readonly IServiceProvider _serviceProvider;

    public Task<IAIProvider?> GetProviderAsync(AIProviderType providerType)
    {
        // Caching verwenden, um doppelte Erstellung zu vermeiden
        if (_cache.TryGetValue(providerType, out var cached))
            return Task.FromResult<IAIProvider?>(cached);

        // Anbieterkonfiguration aus den Einstellungen abrufen
        var aiOptions = _options.Value;
        if (!aiOptions.Providers.TryGetValue(providerType, out var config))
        {
            _logger.LogWarning("Anbieter '{ProviderType}' nicht in der Konfiguration gefunden", providerType);
            return Task.FromResult<IAIProvider?>(null);
        }

        // Anbieter nach Typ erstellen
        var provider = providerType switch
        {
            AIProviderType.ClaudeCodeCli =>
                _serviceProvider.GetService(typeof(ClaudeCodeCliProvider)) as IAIProvider,
            AIProviderType.CodexCli =>
                _serviceProvider.GetService(typeof(CodexCliProvider)) as IAIProvider,
            AIProviderType.GitHubCopilot =>
                _serviceProvider.GetService(typeof(CopilotAIProvider)) as IAIProvider,
            _ => null
        };

        if (provider != null)
        {
            _cache[providerType] = provider;
        }

        return Task.FromResult<IAIProvider?>(provider);
    }
}
```

Vorteile des Factory-Musters:
- **Instanz-Caching**: Vermeidet die wiederholte Erstellung desselben Anbietertyps
- **Abhängigkeitsinjektion**: Erstellung von Instanzen über `IServiceProvider` mit Unterstützung für Dependency Injection
- **Konfigurationsgesteuert**: Lesen der Anbieter-Einstellungen aus Konfigurationsdateien
- **Ausnahmebehandlung**: Gibt `null` bei Erstellungsfehlern zurück, was die Fehlerbehandlung in den oberen Schichten erleichtert

### 4. Intelligenter Selektor

Der **`AIProviderSelector`** implementiert Anbieterauswahlstrategien:

```csharp
public class AIProviderSelector : IAIProviderSelector
{
    private readonly BusinessLayerConfiguration _configuration;
    private readonly IAIProviderFactory _providerFactory;
    private readonly IMemoryCache _cache;

    public async Task<AIProviderType> SelectProviderAsync(
        BusinessScenario scenario,
        CancellationToken cancellationToken = default)
    {
        // 1. Versuchen, einen Anbieter aus der Szenario-Zuordnung zu erhalten
        if (_configuration.ScenarioProviderMapping.TryGetValue(scenario, out var providerType))
        {
            if (await IsProviderAvailableAsync(providerType, cancellationToken))
            {
                _logger.LogDebug("Anbieter '{Provider}' für Szenario '{Scenario}' ausgewählt",
                    providerType, scenario);
                return providerType;
            }

            _logger.LogWarning("Konfigurierter Anbieter '{Provider}' für Szenario '{Scenario}' ist nicht verfügbar",
                providerType, scenario);
        }

        // 2. Standardanbieter versuchen
        if (await IsProviderAvailableAsync(_configuration.DefaultProvider, cancellationToken))
        {
            _logger.LogDebug("Standardanbieter '{Provider}' für Szenario '{Scenario}' verwendet",
                _configuration.DefaultProvider, scenario);
            return _configuration.DefaultProvider;
        }

        // 3. Fallback-Kette versuchen
        foreach (var fallbackProvider in _configuration.FallbackChain)
        {
            if (await IsProviderAvailableAsync(fallbackProvider, cancellationToken))
            {
                _logger.LogInformation("Fallback-Anbieter '{Provider}' für Szenario '{Scenario}' verwendet",
                    fallbackProvider, scenario);
                return fallbackProvider;
            }
        }

        // 4. Kein verfügbarer Anbieter gefunden
        throw new InvalidOperationException(
            $"Kein verfügbarer KI-Anbieter für Szenario '{scenario}' gefunden");
    }

    public async Task<bool> IsProviderAvailableAsync(
        AIProviderType providerType,
        CancellationToken cancellationToken = default)
    {
        var cacheKey = $"provider_available_{providerType}";

        // Caching verwenden, um Ping-Aufrufe zu reduzieren
        if (_configuration.EnableCache &&
            _cache.TryGetValue<bool>(cacheKey, out var cached))
        {
            return cached;
        }

        var provider = await _providerFactory.GetProviderAsync(providerType);
        var isAvailable = provider != null;

        if (_configuration.EnableCache && isAvailable)
        {
            _cache.Set(cacheKey, isAvailable,
                TimeSpan.FromSeconds(_configuration.CacheExpirationSeconds));
        }

        return isAvailable;
    }
}
```

Selektor-Strategie:
- **Zuerst Szenario-Zuordnung**: Prüfen, ob für das Geschäftsszenario eine spezifische Anbieterzuordnung vorhanden ist
- **Fallback zum Standardanbieter**: Den Standardanbieter verwenden, wenn die Szenario-Zuordnung fehlschlägt
- **Fallback-Kette als letzter Ausweg**: Anbieter in der Fallback-Kette nacheinander ausprobieren
- **Verfügbarkeits-Caching**: Verfügbarkeitsprüfungen zwischenspeichern, um Ping-Aufrufe zu reduzieren

### 5. Claude Code CLI-Anbieterimplementierung

```csharp
public class ClaudeCodeCliProvider : IAIProvider
{
    private readonly ILogger<ClaudeCodeCliProvider> _logger;
    private readonly IClaudeStreamManager _streamManager;
    private readonly ProviderConfiguration _config;

    public string Name => "ClaudeCodeCli";
    public bool SupportsStreaming => true;

    public ProviderCapabilities Capabilities { get; }

    public async Task<AIResponse> ExecuteAsync(AIRequest request, CancellationToken cancellationToken = default)
    {
        _logger.LogInformation("KI-Anfrage mit Anbieter ausgeführt: {Provider}", Name);

        var sessionOptions = ClaudeRequestMapper.MapToSessionOptions(request, _config);

        var messages = _streamManager.SendMessageAsync(request.Prompt, sessionOptions, cancellationToken);

        var responseBuilder = new StringBuilder();
        ResultMessage? finalResult = null;

        await foreach (var streamMessage in messages)
        {
            switch (streamMessage.Message)
            {
                case ResultMessage result:
                    finalResult = result;
                    responseBuilder.Append(result.Result);
                    break;
            }
        }

        if (finalResult != null)
        {
            return ClaudeResponseMapper.MapToAIResponse(finalResult, Name);
        }

        return new AIResponse
        {
            Content = responseBuilder.ToString(),
            FinishReason = FinishReason.Unknown,
            Provider = Name
        };
    }
}
```

Merkmale des Claude Code CLI-Anbieters:
- **Streaming-Manager-Integration**: Verwendung von `IClaudeStreamManager` zur Kommunikation mit dem Claude CLI
- **`CessionId`-Sitzungsisolierung**: Verwendung von `CessionId` als eindeutige Sitzungskennung, abgegrenzt von der System-`sessionId`
- **Arbeitsverzeichniskonfiguration**: Unterstützung der Konfiguration des Arbeitsverzeichnisses, des Berechtigungsmodus und weiterer Einstellungen
- **Werkzeugunterstützung**: Unterstützung von Werkzeugberechtigungen wie `AllowedTools` und `DisallowedTools`

### 6. Codex CLI-Anbieterimplementierung

```csharp
public class CodexCliProvider : IAIProvider
{
    private readonly ILogger<CodexCliProvider> _logger;
    private readonly CodexSettings _settings;
    private readonly ConcurrentDictionary<string, string> _sessionThreadBindings;

    public string Name => "CodexCli";
    public bool SupportsStreaming => true;

    public ProviderCapabilities Capabilities { get; }

    public async IAsyncEnumerable<AIStreamingChunk> StreamAsync(
        AIRequest request,
        [EnumeratorCancellation] CancellationToken cancellationToken = default)
    {
        _logger.LogInformation("Streaming-KI-Anfrage mit Anbieter ausgeführt: {Provider}", Name);

        var codex = CreateCodexClient();
        var thread = ResolveThread(codex, request);

        var currentTurn = 0;
        var activeToolCalls = new Dictionary<string, AIToolCallDelta>();

        await foreach (var threadEvent in thread.RunStreamedAsync(BuildPrompt(request), cancellationToken))
        {
            if (threadEvent is TurnStartedEvent)
            {
                currentTurn++;
            }

            switch (threadEvent)
            {
                case ItemCompletedEvent { Item: AgentMessageItem message }:
                    var messageText = message.Text ?? string.Empty;
                    yield return new AIStreamingChunk
                    {
                        Content = messageText,
                        Type = StreamingChunkType.ContentDelta,
                        IsComplete = false
                    };
                    break;

                case ItemStartedEvent or ItemUpdatedEvent or ItemCompletedEvent:
                    var toolChunk = BuildToolChunk(threadEvent, currentTurn);
                    if (toolChunk?.ToolCallDelta != null)
                    {
                        yield return toolChunk;
                    }
                    break;

                case TurnCompletedEvent turnCompleted:
                    activeToolCalls.Clear();
                    yield return new AIStreamingChunk
                    {
                        Content = string.Empty,
                        Type = StreamingChunkType.Metadata,
                        IsComplete = true,
                        Usage = MapUsage(turnCompleted.Usage)
                    };
                    break;
            }
        }

        BindSessionThread(request.SessionId, thread.Id);
    }

    private CodexThread ResolveThread(Codex codex, AIRequest request)
    {
        var sessionId = request.SessionId;

        // Prüfen, ob bereits ein gebundener Thread vorhanden ist
        if (!string.IsNullOrWhiteSpace(sessionId) &&
            _sessionThreadBindings.TryGetValue(sessionId, out var threadId) &&
            !string.IsNullOrWhiteSpace(threadId))
        {
            _logger.LogInformation("Codex-Thread {ThreadId} für Sitzung {SessionId} fortgesetzt", threadId, sessionId);
            return codex.ResumeThread(threadId, threadOptions);
        }

        _logger.LogInformation("Neuer Codex-Thread für Sitzung {SessionId} gestartet", sessionId ?? "(keine)");
        return codex.StartThread(threadOptions);
    }
}
```

Merkmale des Codex CLI-Anbieters:
- **JSON-Ereignisstrom-Verarbeitung**: Parsen von Codex-JSON-Ereignisströmen (`TurnStarted`, `ItemStarted`, `TurnCompleted` usw.)
- **Sitzungs-Thread-Bindung**: Persistierung der Bindung zwischen Sitzungen und Threads in einer SQLite-Datenbank
- **Thread-Wiederverwendung**: Unterstützung der Fortsetzung vorhandener Threads zur Aufrechterhaltung der Sitzungskontinuität
- **Werkzeugaufruf-Verfolgung**: Verfolgung des aktiven Werkzeugaufrufzustands und korrekte Behandlung des Werkzeuglebenszyklus

### 7. Sitzungs-Thread-Bindungsmechanismus

Codex CLI verwendet eine SQLite-Datenbank zur Persistierung der Bindung zwischen Sitzungen und Threads:

```csharp
public class CodexCliProvider : IAIProvider
{
    private const int SessionThreadBindingRetentionDays = 30;
    private readonly ConcurrentDictionary<string, string> _sessionThreadBindings;
    private readonly string _sessionThreadBindingDatabaseConnectionString;
    private readonly string _sessionThreadBindingDatabasePath;

    private void BindSessionThread(string? sessionId, string? threadId)
    {
        if (string.IsNullOrWhiteSpace(sessionId) || string.IsNullOrWhiteSpace(threadId))
        {
            return;
        }

        // Zwischenspeicher im Speicher
        _sessionThreadBindings.AddOrUpdate(sessionId, threadId, (_, _) => threadId);

        // Persistierung in SQLite
        PersistSessionThreadBinding(sessionId, threadId);
    }

    private void PersistSessionThreadBinding(string sessionId, string threadId)
    {
        try
        {
            using var connection = new SqliteConnection(_sessionThreadBindingDatabaseConnectionString);
            connection.Open();

            using var upsertCommand = connection.CreateCommand();
            upsertCommand.CommandText =
                """
                INSERT INTO SessionThreadBindings (SessionId, ThreadId, CreatedAtUtc, UpdatedAtUtc)
                VALUES ($sessionId, $threadId, $createdAtUtc, $updatedAtUtc)
                ON CONFLICT(SessionId) DO UPDATE SET
                    ThreadId = excluded.ThreadId,
                    UpdatedAtUtc = excluded.UpdatedAtUtc;
                """;
            var nowUtc = DateTimeOffset.UtcNow.ToString("O");
            upsertCommand.Parameters.AddWithValue("$sessionId", sessionId);
            upsertCommand.Parameters.AddWithValue("$threadId", threadId);
            upsertCommand.Parameters.AddWithValue("$createdAtUtc", nowUtc);
            upsertCommand.Parameters.AddWithValue("$updatedAtUtc", nowUtc);
            upsertCommand.ExecuteNonQuery();
        }
        catch (Exception ex)
        {
            _logger.LogWarning(
                ex,
                "Persistierung der Codex-Sitzungs-Thread-Bindung für Sitzung {SessionId} in {DatabasePath} fehlgeschlagen",
                sessionId,
                _sessionThreadBindingDatabasePath);
        }
    }

    private void LoadPersistedSessionThreadBindings()
    {
        using var connection = new SqliteConnection(_sessionThreadBindingDatabaseConnectionString);
        connection.Open();

        using var loadCommand = connection.CreateCommand();
        loadCommand.CommandText = "SELECT SessionId, ThreadId FROM SessionThreadBindings;";
        using var reader = loadCommand.ExecuteReader();
        while (reader.Read())
        {
            var sessionId = reader.GetString(0);
            var threadId = reader.GetString(1);
            _sessionThreadBindings[sessionId] = threadId;
        }
    }
}
```

Vorteile der Sitzungs-Thread-Bindung:
- **Sitzungswiederherstellung**: Vorherige Sitzungen können nach einem Systemneustart wiederhergestellt werden
- **Thread-Wiederverwendung**: Dieselbe Sitzung kann einen vorhandenen Codex-Thread wiederverwenden
- **Automatische Bereinigung**: Bindungen, die älter als 30 Tage sind, werden automatisch bereinigt

### 8. Desktop-seitige CLI-Verwaltung

`hagicode-desktop` verwaltet die CLI-Auswahl über `AgentCliManager`:

```typescript
export enum AgentCliType {
  ClaudeCode = 'claude-code',
  Codex = 'codex',
  // Zukünftige Erweiterungen: weitere CLIs wie Aider und Cursor
}

export class AgentCliManager {
  private static readonly STORE_KEY = 'agentCliSelection';
  private static readonly EXECUTOR_TYPE_MAP: Record<AgentCliType, string> = {
    [AgentCliType.ClaudeCode]: 'ClaudeCodeCli',
    [AgentCliType.Codex]: 'CodexCli',
  };

  constructor(private store: any) {}

  async saveSelection(cliType: AgentCliType): Promise<void> {
    const selection: StoredAgentCliSelection = {
      cliType,
      isSkipped: false,
      selectedAt: new Date().toISOString(),
    };

    this.store.set(AgentCliManager.STORE_KEY, selection);
  }

  loadSelection(): StoredAgentCliSelection {
    return this.store.get(AgentCliManager.STORE_KEY, {
      cliType: null,
      isSkipped: false,
      selectedAt: null,
    });
  }

  getCommandName(cliType: AgentCliType): string {
    switch (cliType) {
      case AgentCliType.ClaudeCode:
        return 'claude';
      case AgentCliType.Codex:
        return 'codex';
      default:
        return 'claude';
    }
  }

  getExecutorType(cliType: AgentCliType | null): string {
    if (!cliType) return 'ClaudeCodeCli';
    return this.EXECUTOR_TYPE_MAP[cliType] || 'ClaudeCodeCli';
  }
}
```

### Anwendungsbeispiel – Backend-Dienst

```csharp
public class AIOrchestrator
{
    private readonly IAIProviderFactory _providerFactory;
    private readonly IAIProviderSelector _providerSelector;
    private readonly ILogger<AIOrchestrator> _logger;

    public AIOrchestrator(
        IAIProviderFactory providerFactory,
        IAIProviderSelector providerSelector,
        ILogger<AIOrchestrator> logger)
    {
        _providerFactory = providerFactory;
        _providerSelector = providerSelector;
        _logger = logger;
    }

    public async Task<AIResponse> ProcessRequestAsync(
        AIRequest request,
        BusinessScenario scenario)
    {
        _logger.LogInformation("Anfrage für Szenario verarbeitet: {Scenario}", scenario);

        try
        {
            // Anbieter intelligent auswählen
            var providerType = await _providerSelector.SelectProviderAsync(scenario, request.CancellationToken);

            // Anbieterinstanz abrufen
            var provider = await _providerFactory.GetProviderAsync(providerType);
            if (provider == null)
            {
                throw new InvalidOperationException($"Anbieter {providerType} nicht verfügbar");
            }

            _logger.LogInformation("Anbieter verwendet: {Provider} für Anfrage", provider.Name);

            // Anfrage ausführen
            var response = await provider.ExecuteAsync(request, request.CancellationToken);

            _logger.LogInformation("Anfrage mit Anbieter abgeschlossen: {Provider}, verwendete Tokens: {Tokens}",
                provider.Name,
                response.Usage?.TotalTokens ?? 0);

            return response;
        }
        catch (Exception ex)
        {
            _logger.LogError(ex, "Verarbeitung der Anfrage für Szenario fehlgeschlagen: {Scenario}", scenario);
            throw;
        }
    }
}
```

### Anwendungsbeispiel – Streaming-Antworten

```csharp
public async IAsyncEnumerable<AIStreamingChunk> StreamResponseAsync(
    AIRequest request,
    BusinessScenario scenario)
{
    var providerType = await _providerSelector.SelectProviderAsync(scenario);
    var provider = await _providerFactory.GetProviderAsync(providerType);

    if (provider == null)
    {
        throw new InvalidOperationException($"Anbieter {providerType} nicht verfügbar");
    }

    await foreach (var chunk in provider.StreamAsync(request))
    {
        // Streaming-Chunks verarbeiten
        switch (chunk.Type)
        {
            case StreamingChunkType.ContentDelta:
                // Textinhalt in Echtzeit anzeigen
                await SendToClientAsync(chunk.Content);
                break;

            case StreamingChunkType.ToolCallDelta:
                // Werkzeugaufrufe behandeln
                await HandleToolCallAsync(chunk.ToolCallDelta);
                break;

            case StreamingChunkType.Metadata:
                // Abschlussereignisse und Statistiken behandeln
                if (chunk.IsComplete)
                {
                    _logger.LogInformation("Stream abgeschlossen, Nutzung: {@Usage}", chunk.Usage);
                }
                break;

            case StreamingChunkType.Error:
                // Fehler behandeln
                _logger.LogError("Stream-Fehler: {Error}", chunk.ErrorMessage);
                throw new InvalidOperationException(chunk.ErrorMessage);
        }
    }
}
```

### 3. Fehlerbehandlung

Fallback-Mechanismen müssen robust sein, wenn ein Anbieter nicht verfügbar ist, um sicherzustellen, dass mindestens ein Anbieter nutzbar bleibt:

```csharp
public async Task<AIResponse> ExecuteWithFallbackAsync(
    AIRequest request,
    List<AIProviderType> preferredProviders)
{
    Exception? lastException = null;

    foreach (var providerType in preferredProviders)
    {
        try
        {
            var provider = await _providerFactory.GetProviderAsync(providerType);
            if (provider == null) continue;

            // Ausführung versuchen
            return await provider.ExecuteAsync(request);
        }
        catch (Exception ex)
        {
            _logger.LogWarning(ex, "Anbieter {ProviderType} fehlgeschlagen, nächster wird versucht", providerType);
            lastException = ex;
        }
    }

    // Alle Anbieter fehlgeschlagen
    throw new InvalidOperationException(
        "Alle bevorzugten Anbieter fehlgeschlagen. Letzter Fehler: " + lastException?.Message,
        lastException);
}
```

### 4. Konfigurationsvalidierung

```csharp
public void ValidateConfiguration(AIProviderOptions options)
{
    foreach (var (providerType, config) in options.Providers)
    {
        // Ausführbare Pfade validieren (für CLI-basierte Anbieter)
        if (IsCliBasedProvider(providerType))
        {
            if (string.IsNullOrWhiteSpace(config.ExecutablePath))
            {
                throw new ConfigurationException(
                    $"Anbieter {providerType} erfordert ExecutablePath");
            }

            if (!File.Exists(config.ExecutablePath))
            {
                throw new ConfigurationException(
                    $"Ausführbare Datei für {providerType} nicht gefunden: {config.ExecutablePath}");
            }
        }

        // API-Schlüssel validieren (für API-basierte Anbieter)
        if (IsApiBasedProvider(providerType))
        {
            if (string.IsNullOrWhiteSpace(config.ApiKey))
            {
                throw new ConfigurationException(
                    $"Anbieter {providerType} erfordert ApiKey");
            }
        }

        // Modellnamen validieren
        if (string.IsNullOrWhiteSpace(config.Model))
        {
            _logger.LogWarning("Kein Modell für {ProviderType} konfiguriert, Standard wird verwendet", providerType);
        }
    }
```

### 5. Cache-Verwaltung

```csharp
// Cache regelmäßig bereinigen
public void ClearInactiveProviders(TimeSpan inactiveThreshold)
{
    var now = DateTimeOffset.UtcNow;
    var keysToRemove = new List<AIProviderType>();

    foreach (var (type, instance) in _cache)
    {
        // Angenommen, Anbieter haben eine LastUsedTime-Eigenschaft
        if (instance.LastUsedTime.HasValue &&
            now - instance.LastUsedTime.Value > inactiveThreshold)
        {
            keysToRemove.Add(type);
        }
    }

    foreach (var key in keysToRemove)
    {
        _cache.TryRemove(key, out _);
        _logger.LogInformation("Inaktiven Anbieter bereinigt: {Provider}", key);
    }
}
```

### 7. Thread-Sicherheit

Die Verwendung gleichzeitiger Sammlungen wie `ConcurrentDictionary` stellt die Thread-Sicherheit sicher:

```csharp
public class ThreadSafeProviderCache
{
    private readonly ConcurrentDictionary<AIProviderType, IAIProvider> _cache;
    private readonly ReaderWriterLockSlim _lock = new();

    public IAIProvider? GetProvider(AIProviderType type)
    {
        // Leseoperationen erfordern keine Sperre
        if (_cache.TryGetValue(type, out var provider))
            return provider;

        // Erstellung erfordert eine Schreibsperre
        _lock.EnterWriteLock();
        try
        {
            // Doppelprüfung
            if (_cache.TryGetValue(type, out provider))
                return provider;

            var newProvider = CreateProvider(type);
            if (newProvider != null)
            {
                _cache[type] = newProvider;
            }
            return newProvider;
        }
        finally
        {
            _lock.ExitWriteLock();
        }
    }
}
```

### 8. Datenbankmigration

Bei Änderungen am Datenbankschema der Sitzungs-Thread-Bindung muss die Datenmigration berücksichtigt werden:

```csharp
public class SessionThreadMigration
{
    public async Task MigrateAsync(string dbPath)
    {
        var version = await GetSchemaVersionAsync(dbPath);

        if (version >= 2) return; // Bereits die neueste Version

        using var connection = new SqliteConnection(dbPath);
        connection.Open();

        // Migration auf v2: Spalte CreatedAtUtc hinzufügen
        if (version < 2)
        {
            _logger.LogInformation("Migration von SessionThreadBindings auf v2...");

            using var addColumnCommand = connection.CreateCommand();
            addColumnCommand.CommandText = "ALTER TABLE SessionThreadBindings ADD COLUMN CreatedAtUtc TEXT;";
            addColumnCommand.ExecuteNonQuery();

            using var backfillCommand = connection.CreateCommand();
            backfillCommand.CommandText =
                """
                UPDATE SessionThreadBindings
                SET CreatedAtUtc = COALESCE(NULLIF(UpdatedAtUtc, ''), $nowUtc)
                WHERE CreatedAtUtc IS NULL OR CreatedAtUtc = '';
                """;
            backfillCommand.Parameters.AddWithValue("$nowUtc", DateTimeOffset.UtcNow.ToString("O"));
            backfillCommand.ExecuteNonQuery();
        }

        await UpdateSchemaVersionAsync(dbPath, 2);
        _logger.LogInformation("Migration auf v2 abgeschlossen");
    }
}
```

## Zusammenfassung

HagiCode kombiniert das Provider-Muster, das Factory-Muster und das Selektor-Muster, um eine flexible und erweiterbare Multi-AI-Anbieter-Architektur zu implementieren:

- **Vereinheitlichte Schnittstellenabstraktion**: Die `IAIProvider`-Schnittstelle verbirgt die Unterschiede zwischen CLIs
- **Dynamische Instanzerstellung**: `AIProviderFactory` unterstützt die Laufzeiterstellung von Anbieterinstanzen
- **Intelligente Auswahlstrategie**: `AIProviderSelector` implementiert eine szenariogesteuerte Anbietersauswahl
- **Sitzungszustandspersistierung**: Datenbankbindungen stellen die Sitzungskontinuität sicher
- **Desktop-Integration**: `AgentCliManager` unterstützt Benutzerwahl und Konfiguration

Die Vorteile dieser Architektur sind:
1. **Erweiterbarkeit**: Hinzufügen eines neuen KI-Anbieters erfordert nur die Implementierung der `IAIProvider`-Schnittstelle
2. **Testbarkeit**: Anbieter können unabhängig getestet und simuliert werden
3. **Wartbarkeit**: Jede Anbieterimplementierung ist isoliert und hat eine klare Verantwortung
4. **Benutzerfreundlichkeit**: Unterstützung sowohl für szenariobasierte automatische Auswahl als auch für manuellen Wechsel

Mit diesem Entwurf ermöglicht HagiCode einen nahtlosen Wechsel und eine Interoperabilität zwischen Claude Code CLI und Codex CLI und bietet Entwicklern eine flexible und leistungsstarke KI-Code-Assistent-Erfahrung.
