## Praktische Multi-AI-Provider-Architektur in der HagiCode-Plattform

> Dieser Artikel teilt den technischen Ansatz, den wir unter der Orleans Grain-Architektur verwendet haben, um zwei KI-Werkzeuge, iflow und OpenCode, über eine vereinheitlichte `IAIProvider`-Schnittstelle zu integrieren, und vergleicht die Implementierungsunterschiede zwischen WebSocket- und HTTP-Kommunikation im Detail.

:::note[Zeitlicher Kontext]
Dieser Artikel dokumentiert die Provider-Architekturpraxis zum Zeitpunkt der Veröffentlichung. Informationen zum aktuell aktiv unterstützten Bereich finden Sie unter [KI-Agent-CLI-Installation](/en/related-software-installation/ai-agent-cli/) und im [Installationsleitfaden](/en/installation); `IFlowCli` wird jetzt nur noch als historischer Kompatibilitätswert beibehalten und ist nicht Teil des aktuellen aktiven Unterstützungsbereichs.
:::

## Hintergrund

Nichts Besonderes eigentlich. Beim Bau von HagiCode sind wir auf ein sehr praktisches Problem gestoßen: Benutzer wollten mit verschiedenen KI-Werkzeugen arbeiten. Das ist kaum überraschend, da jeder seine eigenen Gewohnheiten hat. Manche bevorzugen Claude Code, andere lieben GitHub Copilot, und manche Teams verwenden selbst entwickelte Werkzeuge.

Unsere anfängliche Lösung war einfach und direkt: für jedes KI-Werkzeug dedizierte Integrationscode zu schreiben. Aber die Nachteile zeigten sich schnell. Die Codebasis füllte sich mit `if-else`-Verzweigungen, jede Änderung erforderte Tests an mehreren Stellen, und jedes neue Werkzeug bedeutete, eine weitere Logik komplett neu zu schreiben.

Später wurde mir klar, dass es besser wäre, eine vereinheitlichte `IAIProvider`-Schnittstelle zu erstellen und die von allen KI-Anbietern geteilten Fähigkeiten zu abstrahieren. So konnten die oberen Schichten unabhängig vom verwendeten Werkzeug auf dieselbe Weise aufrufen.

Kürzlich musste das Projekt zwei neue Werkzeuge integrieren: iflow und OpenCode. Beide unterstützen das ACP-Protokoll, aber ihre Kommunikationsstile sind unterschiedlich. iflow verwendet WebSocket, während OpenCode eine HTTP-API verwendet. Das wurde ein nützlicher architektonischer Test: zwei verschiedene Transportmodi hinter einer vereinheitlichten Schnittstelle anzupassen.

## Ueber HagiCode

Der in diesem Artikel geteilte Ansatz stammt aus unserer praktischen Erfahrung im [HagiCode](https://hagicode.com)-Projekt. HagiCode ist eine KI-gestützte Entwicklungsplattform, die auf der Orleans Grain-Architektur aufbaut. Sie integriert verschiedene KI-Anbieter über eine vereinheitlichte `IAIProvider`-Schnittstelle, sodass Benutzer die KI-Werkzeuge ihrer Wahl flexibel auswählen können.

## Architekturentwurf

### Vereinheitlichte Schnittstellenabstraktion

Zuerst haben wir die `IAIProvider`-Schnittstelle definiert und die Fähigkeiten abstrahiert, die jeder KI-Anbieter implementieren muss:

```csharp
public interface IAIProvider
{
    string Name { get; }
    bool SupportsStreaming { get; }
    ProviderCapabilities Capabilities { get; }

    Task<AIResponse> ExecuteAsync(AIRequest request, CancellationToken cancellationToken = default);
    IAsyncEnumerable<AIStreamingChunk> StreamAsync(AIRequest request, CancellationToken cancellationToken = default);
    Task<ProviderTestResult> PingAsync(CancellationToken cancellationToken = default);
    IAsyncEnumerable<AIStreamingChunk> SendMessageAsync(AIRequest request, string? embeddedCommandPrompt = null, CancellationToken cancellationToken = default);
}
```

Diese Schnittstelle enthält mehrere Schlüsselmethoden:
- `ExecuteAsync`: Eine einmalige KI-Anfrage ausführen
- `StreamAsync`: Streaming-Antworten für Echtzeitanzeige abrufen
- `PingAsync`: Eine Gesundheitsprüfung durchführen, um die Verfügbarkeit des Anbieters zu verifizieren
- `SendMessageAsync`: Eine Nachricht mit Unterstützung für eingebettete Befehle senden

### IFlowCliProvider: Eine WebSocket-basierte Implementierung

iflow verwendet WebSocket für die ACP-Kommunikation. Die Gesamtarchitektur sieht wie folgt aus:

```
IFlowCliProvider → ACPSessionManager → WebSocketAcpTransport → iflow CLI
                ↓
         Dynamische Portzuweisung + Prozessverwaltung
```

Der Kernablauf ist ebenfalls recht unkompliziert:
1. `ACPSessionManager` erstellt und verwaltet ACP-Sitzungen.
2. `WebSocketAcpTransport` behandelt die WebSocket-Kommunikation.
3. Ein Port wird dynamisch zugewiesen, und der iflow-Prozess wird mit `iflow --experimental-acp --port` gestartet.
4. `IAIRequestToAcpMapper` und `IAcpToAIResponseMapper` konvertieren Anforderungen und Antworten.

Hier ist der Kerncode:

```csharp
private async IAsyncEnumerable<AIStreamingChunk> StreamCoreAsync(
    AIRequest request,
    string? embeddedCommandPrompt,
    [EnumeratorCancellation] CancellationToken cancellationToken)
{
    // Arbeitsverzeichnis auflösen
    var resolvedWorkingDirectory = ResolveWorkingDirectory(request);
    var effectiveRequest = ApplyEmbeddedCommandPrompt(request, embeddedCommandPrompt);

    // ACP-Sitzung erstellen
    await using var session = await _sessionManager.CreateSessionAsync(
        Name,
        resolvedWorkingDirectory,
        cancellationToken,
        request.SessionId);

    // Prompt senden
    var prompt = _requestMapper.ToPromptString(effectiveRequest);
    var promptResponse = await session.SendPromptAsync(prompt, cancellationToken);

    // Streaming-Antwort empfangen
    await foreach (var notification in session.ReceiveUpdatesAsync(cancellationToken))
    {
        if (_responseMapper.TryConvertToStreamingChunk(notification, out var chunk))
        {
            if (chunk.Type == StreamingChunkType.Metadata && chunk.IsComplete)
            {
                yield return chunk;
                yield break;
            }
            yield return chunk;
        }
    }
}
```

Einige Entwurfspunkte sind hier hervorzuheben:
- Verwendung von `await using`, um sicherzustellen, dass die Sitzung korrekt freigegeben wird und Ressourcenlecks vermieden werden.
- Rückgabe von Streaming-Antworten über `IAsyncEnumerable`, was asynchrone Ströme natürlich unterstützt.
- Verwendung von `Metadata`-Chunks zur Bestimmung des Abschlusses und um sicherzustellen, dass die vollständige Antwort empfangen wurde.

### OpenCodeCliProvider: Eine HTTP-API-basierte Implementierung

OpenCode stellt seinen Dienst über eine HTTP-API bereit, daher ist die Architektur etwas anders:

```plaintext
OpenCodeCliProvider → OpenCodeRuntimeManager → OpenCodeClient → OpenCode HTTP API
                      ↓
                OpenCodeProcessManager → opencode-Prozessverwaltung
```

Eine bemerkenswerte Eigenschaft von OpenCode ist, dass es eine SQLite-Datenbank zur Persistierung von Sitzungsbindungen verwendet. Dadurch werden Sitzungswiederherstellung und Prompt-Antwort-Wiederherstellung möglich:

```csharp
private async Task<OpenCodePromptExecutionResult> ExecutePromptAsync(
    AIRequest request,
    string? embeddedCommandPrompt,
    CancellationToken cancellationToken)
{
    var prompt = BuildPrompt(request, embeddedCommandPrompt);
    var resolvedWorkingDirectory = ResolveWorkingDirectory(request.WorkingDirectory);
    var client = await _runtimeManager.GetClientAsync(resolvedWorkingDirectory, cancellationToken);
    var bindingSessionId = request.SessionId;
    var boundSession = TryGetBinding(bindingSessionId, resolvedWorkingDirectory);

    // Versuchen, die bereits gebundene Sitzung zu verwenden
    if (boundSession is not null)
    {
        try
        {
            return await PromptSessionAsync(
                client,
                boundSession,
                BuildPromptRequest(request, prompt, CreatePromptMessageId()),
                request.Model ?? _settings.Model,
                cancellationToken);
        }
        catch (OpenCodeApiException ex) when (IsStaleBinding(ex))
        {
            // Die Sitzung ist abgelaufen, Bindung entfernen
            RemoveBinding(bindingSessionId);
        }
    }

    // Neue Sitzung erstellen
    var session = await client.Session.CreateAsync(new OpenCodeSessionCreateRequest
    {
        Title = BuildSessionTitle(request)
    }, cancellationToken);

    BindSession(bindingSessionId, session.Id, resolvedWorkingDirectory);
    return await PromptSessionAsync(client, session.Id, ...);
}
```

Diese Implementierung hat mehrere interessante Highlights:
- Sitzungsbindungsmechanismus: dieselbe `SessionId` verwendet dieselbe OpenCode-Sitzung wieder und vermeidet wiederholte Sitzungserstellung.
- Ablaufbehandlung: wenn eine Sitzung als abgelaufen erkannt wird, wird die Bindung automatisch bereinigt.
- Datenbankpersistierung: Bindungen werden in SQLite gespeichert und bleiben nach einem Neustart wirksam.

### Vergleich der beiden Ansätze

| Aspekt | IFlowCliProvider | OpenCodeCliProvider |
|------|-----------------|---------------------|
| Kommunikation | WebSocket (ACP) | HTTP-API |
| Prozessverwaltung | ACPSessionManager | OpenCodeProcessManager |
| Portzuweisung | Dynamischer Port | Kein Port (verwendet HTTP) |
| Sitzungsverwaltung | ACPSession | OpenCodeSession |
| Persistierung | Speicher-Cache | SQLite-Datenbank |
| Startbefehl | `iflow --experimental-acp --port` | `opencode` |
| Latenz | Niedriger (langlebige Verbindung) | Relativ höher (HTTP-Anfragen) |

Welcher Ansatz gewählt wird, hängt hauptsächlich von den Anforderungen ab. WebSocket ist besser für Szenarien mit hohen Echtzeitanforderungen, während eine HTTP-API einfacher und leichter zu debuggen ist.

## Praktischer Leitfaden

### Provider konfigurieren

Aktivieren Sie zunächst die beiden Anbieter in der Konfigurationsdatei:

```yaml
AI:
  Providers:
    IFlowCli:
      Type: "IFlowCli"
      Enabled: true
      ExecutablePath: "iflow"
      Model: null
      WorkingDirectory: null
    OpenCodeCli:
      Type: "OpenCodeCli"
      Enabled: true
      ExecutablePath: "opencode"
      Model: "anthropic/claude-sonnet-4"
      WorkingDirectory: null

OpenCode:
  Enabled: true
  BaseUrl: "http://localhost:38376"
  ExecutablePath: "opencode"
  StartupTimeoutSeconds: 30
  RequestTimeoutSeconds: 120
```

### IFlowCliProvider verwenden

```csharp
// Provider über die Factory abrufen
var provider = await _providerFactory.GetProviderAsync(AIProviderType.IFlowCli);

// KI-Anfrage ausführen
var request = new AIRequest
{
    Prompt = "Bitte helfen Sie mir, diese Funktion zu refaktorieren",
    WorkingDirectory = "/path/to/project",
    Model = "claude-sonnet-4"
};

// Vollständige Antwort abrufen
var response = await provider.ExecuteAsync(request, cancellationToken);
Console.WriteLine(response.Content);

// Oder Streaming-Antworten verwenden
await foreach (var chunk in provider.StreamAsync(request, cancellationToken))
{
    if (chunk.Type == StreamingChunkType.ContentDelta)
    {
        Console.Write(chunk.Content);
    }
}
```

### OpenCodeCliProvider verwenden

```csharp
// Provider über die Factory abrufen
var provider = await _providerFactory.GetProviderAsync(AIProviderType.OpenCodeCli);

var request = new AIRequest
{
    Prompt = "Bitte helfen Sie mir, diesen Fehler zu analysieren",
    WorkingDirectory = "/path/to/project",
    Model = "anthropic/claude-sonnet-4"
};

var response = await provider.ExecuteAsync(request, cancellationToken);
Console.WriteLine(response.Content);
```

### Gesundheitsprüfungen

Vor dem Start oder vor der Verwendung können Sie prüfen, ob der Anbieter verfügbar ist:

```csharp
var iflowResult = await iflowProvider.PingAsync(cancellationToken);
if (!iflowResult.Success)
{
    Console.WriteLine($"IFlow ist nicht verfügbar: {iflowResult.ErrorMessage}");
    return;
}

var openCodeResult = await openCodeProvider.PingAsync(cancellationToken);
if (!openCodeResult.Success)
{
    Console.WriteLine($"OpenCode ist nicht verfügbar: {openCodeResult.ErrorMessage}");
    return;
}
```

### Unterstützung für eingebettete Befehle

Beide Anbieter unterstützen eingebettete Befehle wie `/file:xxx`:

```csharp
var request = new AIRequest
{
    Prompt = "Probleme in dieser Datei analysieren",
    SystemMessage = "Sie sind ein Code-Analyse-Experte"
};

await foreach (var chunk in provider.SendMessageAsync(
    request,
    embeddedCommandPrompt: "/file:src/main.cs",
    cancellationToken))
{
    Console.Write(chunk.Content);
}
```

## Hinweise und bewährte Praktiken

### Ressourcenverwaltung

IFlow verwendet langlebige WebSocket-Verbindungen, daher verdient die Ressourcenverwaltung besondere Beachtung:
- Verwenden Sie `await using`, um sicherzustellen, dass Sitzungen ordnungsgemäß freigegeben werden.
- Abbrüche lösen eine Prozessbereinigung aus.
- `ACPSessionManager` unterstützt ein Limit für die maximale Sitzungsanzahl.

Die Prozessverwaltung von OpenCode ist relativ einfacher, und `OpenCodeRuntimeManager` erledigt dies automatisch.

### Fehlerbehandlung

Beide Anbieter verfügen über eine vollständige Fehlerbehandlung:
- IFlow-Fehler werden über ACP-Sitzungsaktualisierungen weitergegeben.
- OpenCode-Fehler werden über `OpenCodeApiException` ausgelöst.
- Es wird empfohlen, dass der Aufrufer diese Ausnahmen abfängt und behandelt.

### Leistungsoptimierungen

- IFlow WebSocket-Kommunikation hat eine geringere Latenz als HTTP.
- OpenCode-Sitzungswiederverwendung kann den Aufwand von HTTP-Anforderungen reduzieren.
- Der Factory-Cache-Mechanismus vermeidet die wiederholte Erstellung von Anbietern.
- In Hochparallelitätsszenarien ist auf die Grenzen von Prozessanzahl und Verbindungsanzahl zu achten.

### Konfigurationsvalidierung

Der ausführbare Pfad wird beim Start validiert, aber Laufzeitprobleme können dennoch auftreten. `PingAsync` ist ein nützliches Werkzeug zur Überprüfung, ob die Konfiguration korrekt ist:

```csharp
// Prüfung beim Start
var provider = await _providerFactory.GetProviderAsync(providerType);
var result = await provider.PingAsync(cancellationToken);
if (!result.Success)
{
    _logger.LogError("Anbieter {ProviderType} ist nicht verfügbar: {Error}", providerType, result.ErrorMessage);
}
```

## Zusammenfassung

Dieser Artikel teilt den technischen Ansatz, den die HagiCode-Plattform bei der Integration der beiden KI-Werkzeuge iflow und OpenCode verwendet hat. Durch eine vereinheitlichte `IAIProvider`-Schnittstelle haben wir verschiedene Kommunikationsstile – WebSocket und HTTP – angepasst, während das aufrufende Muster der oberen Schichten konsistent blieb.

Die Kernidee ist eigentlich ganz einfach:
1. Eine vereinheitlichte Schnittstellenabstraktion definieren.
2. Adapter-Schichten für verschiedene Implementierungen aufbauen.
3. Alles einheitlich über das Factory-Muster verwalten.

Dadurch erhält das System eine gute Erweiterbarkeit. Wenn später ein neues KI-Werkzeug integriert werden muss, müssen wir nur die `IAIProvider`-Schnittstelle implementieren, ohne viel vorhandenen Code zu ändern.

Wenn Sie ebenfalls an der Integration mehrerer KI-Werkzeuge arbeiten, hoffe ich, dass dieser Artikel hilfreich ist.
