## DESIGN.md direkt in der Weboberflaeche bearbeiten: Von der Idee zur Implementierung

> Im MonoSpecs-Projektmanagementsystem traegt `DESIGN.md` die Architekturentwuerfe und technischen Entscheidungen eines Projekts. Aber der traditionelle Bearbeitungsworkflow zwingt Nutzer, zu einem externen Editor zu wechseln. Diese fragmentierte Erfahrung ist, als wuerde man mitten im Lesen eines Gedichts unterbrochen: Die Inspiration ist verflogen, und die Stimmung auch. Dieser Artikel teilt die Loesung, die wir im HagiCode-Projekt praktiziert haben: `DESIGN.md` direkt in der Weboberflaeche bearbeiten, mit Unterstuetzung fuer den Import von Vorlagen aus einer Online-Designsite.

## Hintergrund

Als zentrales Traeger von Projektdesigndokumenten enthaelt `DESIGN.md` Schluesselinformationen wie Architekturentwurf, technische Entscheidungen und Implementierungsleitfaeden. Der traditionelle Bearbeitungsansatz erfordert jedoch, dass Nutzer zu einem externen Editor wie VS Code wechseln, den physischen Pfad manuell suchen und dann die Datei bearbeiten. Es ist nicht besonders kompliziert, aber nach mehrmaliger Wiederholung wird es muehsam.

Die Probleme aeussern sich hauptsaechlich wie folgt:

- **Fragmentierter Workflow**: Nutzer muessen staendig zwischen der Web-Verwaltungsoberflaeche und einem lokalen Editor wechseln, was die Kontinuitaet ihres Workflows unterbricht
- **Schwer wiederverwendbar**: Die Designsite veroeffentlicht bereits eine reiche Bibliothek von Designvorlagen, aber sie koennen nicht direkt in den Projektbearbeitungs-Workflow integriert werden
- **Fehlende Erfahrungsschleife**: Es gibt keine geschlossene Schleife fuer „Vorschau - Auswaehlen - Importieren", sodass Nutzer manuell kopieren und einfuegen muessen
- **Zusammenarbeitsreibung**: Design-Dokumente und Code-Implementierung synchron zu halten wird zu einem reibungsreichen Prozess

Um diese Schmerzpunkte zu loesen, beschlossen wir, direkte Bearbeitungsunterstuetzung fuer `DESIGN.md` in der Weboberflaeche hinzuzufuegen und den Ein-Klick-Vorlagenimport aus einer Online-Designsite zu ermoeglichen.

## Ueber HagiCode

Die in diesem Artikel geteilte Loesung stammt aus unserer praktischen Erfahrung im [HagiCode](https://hagicode.com)-Projekt. HagiCode ist ein KI-gesteuertes Programmierassistent-Projekt, und waehrend der Entwicklung muessen wir haeufig Projektdesigndokumente pflegen. Um dem Team zu helfen, effizienter zusammenzuarbeiten, haben wir diese Online-Bearbeitungs- und Import-Loesung erkundet und implementiert.

## Technische Loesung

### Gesamtarchitektur

Diese Loesung verwendet eine Frontend-Backend-getrennte Architektur mit einem Same-Origin-Proxy, hauptsaechlich bestehend aus folgenden Schichten:

**1. Frontend-Editor-Schicht**
```typescript
// Kernkomponente: DesignMdManagementDrawer
// Ort: repos/web/src/components/project/DesignMdManagementDrawer.tsx
// Verantwortung: Bearbeitung, Speicherung, Versionskonflikterkennung und Import-Flow
```

**2. Backend-Service-Schicht**
```csharp
// Kernservice: ProjectAppService.DesignMd
// Ort: repos/hagicode-core/src/PCode.Application/ProjectAppService.DesignMd.cs
// Verantwortung: Pfadaufloesung, Dateilesen/-schreiben und Versionsverwaltung
```

**3. Same-Origin-Proxy-Schicht**
```csharp
// Proxy-Service: ProjectAppService.DesignMdSiteIndex
// Ort: repos/hagicode-core/src/PCode.Application/ProjectAppService.DesignMdSiteIndex.cs
// Verantwortung: Designsite-Ressourcen proxyen, Vorschaubild-Caching und Sicherheitsvalidierung
```

### Wichtige technische Entscheidungen

#### Entscheidung 1: Globales Drawer-Muster

Wir verwenden einen einzigen globalen Drawer statt lokaler Pop-up-Schichten, mit Zustandsverwaltung durch `layoutSlice`, was Nutzern ein konsistentes Erlebnis ueber Ansichten hinweg (`classic` und `kanban`) bietet.

#### Entscheidung 2: Projektbezogene API

Wir hängten `DESIGN.md`-bezogene Endpunkte unter `ProjectController`, wodurch wir die bestehende Projektberechtigungsgrenze wiederverwenden und die Komplexitaet eines separaten Controllers vermeiden.

#### Entscheidung 3: Versionskonflikterkennung

Wir leiten eine opake Version aus dem `LastWriteTimeUtc` des Dateisystems ab, was uns eine leichte optimistische Nebenlaeufigkeitskontrolle gibt. Wenn mehrere Nutzer gleichzeitig dieselbe Datei bearbeiten, kann das System Konflikte erkennen und den Nutzer zum Aktualisieren auffordern.

#### Entscheidung 4: Same-Origin-Proxy-Muster

Wir verwenden `IHttpClientFactory`, um externe Designsite-Ressourcen zu proxyen, wodurch sowohl Cross-Origin-Probleme als auch SSRF-Risiken vermieden werden.

## Kernimplementierung

### 1. DESIGN.md direkt bearbeiten

#### Backend-Implementierung

Das Backend ist hauptsaechlich fuer Pfadaufloesung, Dateilesen/-schreiben und Versionsverwaltung verantwortlich:

```csharp
// Pfadaufloesung und Sicherheitsvalidierung
private Task<string> ResolveDesignDocumentDirectoryAsync(string projectPath, string? repositoryPath)
{
    if (string.IsNullOrWhiteSpace(repositoryPath))
    {
        return Task.FromResult(Path.GetFullPath(projectPath));
    }
    return ValidateSubPathAsync(projectPath, repositoryPath);
}

// Versionsgenerierung (basierend auf Dateisystem-Zeitstempel und Groesse)
private static string BuildDesignDocumentVersion(string path)
{
    var fileInfo = new FileInfo(path);
    fileInfo.Refresh();
    return string.Create(
        CultureInfo.InvariantCulture,
        $"{fileInfo.LastWriteTimeUtc.Ticks:x}-{fileInfo.Length:x}");
}
```

Das Versionsdesign ist interessant in seiner Einfachheit: Wir verwenden die letzte Aenderungszeit und Groesse der Datei, um einen eindeutigen Versionsidentifikator zu generieren. Es ist leichtgewichtig und zuverlaessig, ohne eine zusaetzliche Versionsdatenbank pflegen zu muessen.

#### Frontend-Implementierung

Auf dem Frontend implementieren wir Dirty-State-Erkennung und Speicherlogik:

```typescript
// Dirty-State-Erkennung und Speicherlogik
const [draft, setDraft] = useState('');
const [savedDraft, setSavedDraft] = useState('');
const isDirty = draft !== savedDraft;

const handleSave = useCallback(async () => {
    const result = await saveProjectDesignMdDocument({
        ...activeTarget,
        content: draft,
        expectedVersion: document.version, // optimistische Nebenlaeufigkeitskontrolle
    });
    setSavedDraft(draft); // gespeicherten Zustand aktualisieren
}, [activeTarget, document, draft]);
```

In dieser Implementierung pflegen wir zwei Zustaende: `draft` ist der aktuell bearbeitete Inhalt, waehrend `savedDraft` der gespeicherte Inhalt ist. Durch ihren Vergleich wissen wir, ob ungespeicherte Aenderungen vorliegen.

#### Index-Datenformat

Die Indexdatei auf der Designsite definiert alle verfuegbaren Vorlagen:


#### Same-Origin-Proxy-Implementierung

Um die Sicherheit zu gewaehrleisten, fuehrt das Backend eine strenge Validierung fuer den Zugriff auf die Designsite durch:

```csharp
// Sichere Slug-Validierung
private static readonly Regex SafeDesignSiteSlugRegex =
    new("^[A-Za-z0-9](?:[A-Za-z0-9._-]{0,127})$", RegexOptions.Compiled);

private static string NormalizeDesignSiteSlug(string slug)
{
    var normalizedSlug = slug?.Trim() ?? string.Empty;
    if (!IsSafeDesignSiteSlug(normalizedSlug))
    {
        throw new BusinessException(
            ProjectDesignSiteIndexErrorCodes.InvalidSlug,
            "Design site slug muss ein einzelner sicherer Pfadabschnitt sein.");
    }
    return normalizedSlug;
}

// Vorschaubild-Caching (OS-temp-Verzeichnis)
private static string ComputePreviewCacheKey(string slug, string theme, string previewUrl)
{
    var raw = $"{slug}|{theme}|{previewUrl}";
    var bytes = SHA256.HashData(Encoding.UTF8.GetBytes(raw));
    return Convert.ToHexString(bytes).ToLowerInvariant();
}
```

Wir tun hier zwei Dinge: Erstens validieren wir das Slug-Format streng mit einem regulären Ausdruck, um Pfad-Traversal-Angriffe zu verhindern; zweitens cachen wir Vorschaubilder, um den Druck auf die externe Site zu reduzieren.

### 3. Vollstaendiger Import-Flow

```typescript
// 1. Import-Drawer oeffnen
const handleRequestImportDrawer = useCallback(() => {
    setIsImportDrawerOpen(true);
}, []);

// 2. Auswaehlen und importieren
const handleImportRequest = useCallback((entry) => {
    if (isDirty) {
        setPendingImportEntry(entry);
        setConfirmMode('import'); // Ueberschreiben-Bestaetigung
        return;
    }
    void executeImport(entry);
}, [isDirty]);

// 3. Import ausfuehren
const executeImport = useCallback(async (entry) => {
    const result = await getProjectDesignMdSiteImportDocument(
        activeTarget.projectId,
        entry.slug
    );
    setDraft(result.content); // Nur Editortext ersetzen, nicht automatisch speichern
    setIsImportDrawerOpen(false);
}, [activeTarget?.projectId]);
```

Der Import-Flow folgt dem Prinzip „Nutzer-Bestaetigung": Nach dem Import wird nur der Editor-Inhalt aktualisiert, und es wird nichts automatisch gespeichert. Nutzer koennen den importierten Inhalt pruefen und manuell erst nach Bestaetigung speichern.

## Praktische Beispiele

### Szenario 1: DESIGN.md im Projektstamm erstellen

Wenn `DESIGN.md` nicht existiert, gibt das Backend einen virtuellen Dokumentzustand zurueck. Dies ermoeglicht dem Frontend, keine Sonderbehandlung fuer den Fall „Datei existiert nicht" zu benoetigen:

```csharp
return new ProjectDesignDocumentDto
{
    Path = targetPath,
    Exists = false,  // virtueller Dokumentzustand
    Content = string.Empty,
    Version = null
};

// Beim ersten Speichern automatisch Datei erstellen
public async Task<SaveProjectDesignDocumentResultDto> SaveDesignDocumentAsync(...)
{
    Directory.CreateDirectory(targetDirectory);
    await File.WriteAllTextAsync(targetPath, input.Content);
    return new SaveProjectDesignDocumentResultDto { Created = !exists };
}
```

### Szenario 2: Vorlage aus der Designsite importieren

Nachdem der Nutzer die „Linear"-Designvorlage im Import-Drawer ausgewaehlt hat, ruft das System den `DESIGN.md`-Inhalt ueber den Backend-Proxy ab. Der gesamte Prozess ist fuer den Nutzer transparent.

```typescript
// 1. Das System ruft DESIGN.md-Inhalt ueber den Backend-Proxy ab
GET /api/project/{id}/design-md/site-index/linear.app

// 2. Das Backend validiert den Slug und ruft Inhalte vom Upstream ab
var entry = FindDesignSiteEntry(catalog, "linear.app");
using var upstreamResponse = await httpClient.SendAsync(request);
var content = await upstreamResponse.Content.ReadAsStringAsync();

// 3. Das Frontend ersetzt den Editortext
setDraft(result.content);
// Der Nutzer prueft und speichert dann manuell auf Festplatte
```

### Szenario 3: Versionskonflikte behandeln

Wenn mehrere Nutzer gleichzeitig dieselbe `DESIGN.md` bearbeiten, erkennt das System Versionskonflikte:

```csharp
if (!string.Equals(currentVersion, expectedVersion, StringComparison.Ordinal))
{
    throw new BusinessException(
        ProjectDesignDocumentErrorCodes.VersionConflict,
        $"DESIGN.md at '{targetPath}' wurde auf der Festplatte geaendert.");
}
```

Das Frontend faengt diesen Fehler ab und fordert den Nutzer auf:

```typescript
// Frontend fordert den Nutzer auf, zu aktualisieren und erneut zu versuchen
<Alert>
    <AlertTitle>Versionskonflikt</AlertTitle>
    <AlertDescription>
        Die Datei wurde von einem anderen Prozess geaendert. Bitte aktualisieren Sie, um die neueste Version zu erhalten, und versuchen Sie es erneut.
    </AlertDescription>
</Alert>
```

## Hinweise und bewaehrte Praktiken

### 1. Pfadsicherheit

Validieren Sie `repositoryPath` immer, um Pfad-Traversal-Angriffe zu verhindern:

```csharp
// repositoryPath immer validieren, um Pfad-Traversal-Angriffe zu verhindern
return ValidateSubPathAsync(projectPath, repositoryPath);
```

### 2. Cache-Strategie

Vorschaubilder 24 Stunden cachen, mit maximal 160 Dateien:

```csharp
private static readonly TimeSpan PreviewCacheTtl = TimeSpan.FromHours(24);
private const int PreviewCacheMaxFiles = 160;
```

### 3. Fehlerbehandlung

Bei Unverfuegbarkeit der Upstream-Site geraeushaft degradieren:

```typescript
try {
    const catalog = await getProjectDesignMdSiteImportCatalog(projectId);
} catch (error) {
    toast.error(t('project.designMd.siteImport.feedback.catalogLoadFailed'));
    // Der Haupt-Bearbeitungs-Drawer bleibt verfuegbar
}
```

### 4. Benutzererfahrungsoptimierung

Vor dem Import Ueberschreiben bestaetigen und nach dem Import nicht automatisch speichern. Nutzer sollten die Kontrolle ueber ihre eigenen Aktionen behalten.

### 5. Leistungsueberlegungen

Verwenden Sie eine HTTP-Client-Factory, um zu viele Verbindungen zu vermeiden:

```csharp
private const string DesignSiteProxyClientName = "ProjectDesignSiteProxy";
private static readonly TimeSpan DesignSiteProxyTimeout = TimeSpan.FromSeconds(8);
```

## Vorgeschlagene Erweiterungen

1. **Markdown-Erweiterung**: Wir verwenden derzeit ein einfaches `Textarea`, aber koennten auf CodeMirror fuer Syntaxhervorhebung und Tastenkuerzel upgraden.
2. **Vorschaumodus**: Echtzeit-Markdown-Vorschau hinzufuegen, um die Bearbeitungserfahrung zu verbessern.
3. **Diff-Merge**: Einen intelligenten Merge-Algorithmus statt einfacher Volltextersetzung implementieren.
4. **Lokales Caching**: `design.json` in der Datenbank cachen, um die Abhaengigkeit von der externen Site zu reduzieren.

## Zusammenfassung

Im HagiCode-Projekt implementierten wir eine vollstaendige Online-Bearbeitungs- und Import-Loesung fuer `DESIGN.md` durch Frontend-Backend-Zusammenarbeit. Der Kernwert dieser Loesung liegt in folgenden Punkten:

- **Hoehere Effizienz**: Kein Werkzeugwechsel erforderlich; Bearbeitung und Import von Designdokumenten koennen in einer einheitlichen Weboberflaeche stattfinden.
- **Niedrigere Einstiegshuerde**: Ein-Klick-Designvorlagenimport hilft neuen Projekten, schnell zu starten.
- **Sicher und zuverlaessig**: Pfadvalidierung, Versionskonflikterkennung und geraeushafte Degradierungsmechanismen halten das System stabil.
- **Bessere Benutzererfahrung**: Der globale Drawer, die Dirty-State-Erkennung und Bestaetigungsdialoge verfeinern das gesamte Interaktionserlebnis.
