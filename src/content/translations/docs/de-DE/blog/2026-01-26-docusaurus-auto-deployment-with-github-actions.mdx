## GitHub Pages-Automatische Bereitstellung für HagiCode hinzufügen

> Dieses Projekt trug früher den Codenamen PCode und wurde nun offiziell in HagiCode umbenannt. Dieser Artikel dokumentiert, wie automatische statische Website-Bereitstellung für das Projekt eingeführt wurde, sodass das Veröffentlichen von Inhalten so einfach wie Wasser trinken ist.

## Hintergrund/Einleitung

Während der Entwicklung von HagiCode stießen wir auf ein sehr praktisches Problem: Mit zunehmender Anzahl von Dokumentationen und Proposal wurde das effiziente Verwalten und Präsentieren dieser Inhalte zu einer dringenden Aufgabe. Wir entschieden uns für GitHub Pages zum Hosten unserer statischen Website, aber der manuelle Build- und Bereitstellungsprozess war zu mühsam – bei jeder Änderung mussten wir lokal bauen, paketieren und dann manuell in den `gh-pages`-Zweig pushen. Das war nicht nur ineffizient, sondern auch fehleranfällig.

Um dieses Problem zu lösen (hauptsächlich um uns Arbeit zu sparen), brauchten wir einen automatisierten Bereitstellungsprozess. Dieser Artikel dokumentiert im Detail, wie automatische GitHub Actions-Bereitstellung für das HagiCode-Projekt hinzugefügt wurde, damit wir uns auf die Content-Erstellung konzentrieren können, während der Rest dem automatisierten Prozess überlassen wird.

## Zielanalyse

Bevor wir loslegen, müssen wir klar definieren, was diese Aufgabe bewirken soll. Schließlich heißt es: „Wer gut plant, arbeitet schnell."

### Kernanforderungen

1.  **Automatischer Build**: Wenn Code in den `main`-Zweig gepusht wird, wird der Build-Prozess automatisch ausgelöst.
2.  **Automatische Bereitstellung**: Nach erfolgreichem Build werden die generierten statischen Dateien automatisch auf GitHub Pages bereitgestellt.
3.  **Umgebungskonsistenz**: Sicherstellen, dass die CI-Umgebung und die lokale Build-Umgebung konsistent sind, um „funktioniert lokal, Fehler in Produktion"-Szenarien zu vermeiden.

### Technologische Entscheidung

Da HagiCode auf Docusaurus basiert (einem sehr beliebten React-Static-Site-Generator), können wir GitHub Actions nutzen, um dieses Ziel zu erreichen.

## GitHub Actions-Workflow konfigurieren

GitHub Actions ist der CI/CD-Dienst von GitHub. Durch die Definition von YAML-Workflow-Dateien im Code-Repository können wir verschiedene Automatisierungsaufgaben anpassen.

### Workflow-Datei erstellen

Wir müssen eine neue Konfigurationsdatei im Ordner `.github/workflows` des Projektstamms erstellen, z. B. `deploy.yml`. Wenn der Ordner nicht existiert, erstellen Sie ihn bitte manuell.

Die Kernlogik dieser Konfigurationsdatei lautet wie folgt:

1.  **Auslösebedingungen**: Überwacht `push`-Ereignisse auf dem `main`-Zweig.
2.  **Laufumgebung**: Neueste Ubuntu-Version.
3.  **Build-Schritte**:
    -   Code auschecken
    -   Node.js installieren
    -   Abhängigkeiten installieren (`npm install`)
    -   Statische Dateien bauen (`npm run build`)
4.  **Bereitstellungsschritte**: Verwenden des offiziellen `action-gh-pages`, um die Build-Artefakte in den `gh-pages`-Zweig zu pushen.

### Wichtige Konfigurationscodes

Im Folgenden die von uns verwendete Konfigurationsvorlage:

```yaml
name: Deploy to GitHub Pages

# 触发条件：当推送到 main 分支时
on:
  push:
    branches:
      - main
    # 可以根据需要添加路径过滤，比如只有文档变动才构建
    # paths:
    #   - 'docs/**'
    #   - 'package.json'

# 设置权限，这对于部署到 GitHub Pages 很重要
permissions:
  contents: read
  pages: write
  id-token: write

# 并发控制：取消同一分支的旧构建
concurrency:
  group: "pages"
  cancel-in-progress: false

jobs:
  build:
    runs-on: ubuntu-latest
    steps:
      - name: Checkout
        uses: actions/checkout@v4
        # 注意：必须设置 fetch-depth: 0，否则可能导致构建版本号不准确
        with:
          fetch-depth: 0

      - name: Setup Node
        uses: actions/setup-node@v4
        with:
          node-version: 20 # 建议与本地开发环境保持一致
          cache: 'npm'     # 启用缓存可以加速构建过程

      - name: Install dependencies
        run: npm ci
        # 使用 npm ci 而不是 npm install，因为它更快、更严格，适合 CI 环境

      - name: Build website
        run: npm run build
        env:
          # 如果你的站点构建需要环境变量，在这里配置
          # NODE_ENV: production
          # PUBLIC_URL: /your-repo-name

      - name: Upload artifact
        uses: actions/upload-pages-artifact@v3
        with:
          path: ./build # Docusaurus 默认输出目录

  deploy:
    environment:
      name: github-pages
      url: ${{ steps.deployment.outputs.page_url }}
    runs-on: ubuntu-latest
    needs: build
    steps:
      - name: Deploy to GitHub Pages
        id: deployment
        uses: actions/deploy-pages@v4
```

## Stolpersteine bei der Implementierung

Bei der praktischen Umgebung stießen wir auf einige Probleme, die wir hier teilen, damit andere sie vermeiden können (oder sich rechtzeitig darauf vorbereiten).

### 1. GitHub Token-Berechtigungsproblem

Bei der anfänglichen Konfiguration schlug die Bereitstellung immer mit dem Fehler 403 (Forbidden) fehl. Nach langer Suche stellten wir fest, dass der Standard-`GITHUB_TOKEN` von GitHub keine Schreibberechtigung für Pages hatte.

**Lösung**: Wählen Sie in den Repository-Einstellungen unter `Settings` -> `Actions` -> `General` -> `Workflow permissions` unbedingt **„Read and write permissions"** aus.

### 2. Falscher Build-Verzeichnispfad

Docusaurus speichert die gebauten statischen Dateien standardmäßig im Verzeichnis `build`. Einige Projekte (z. B. Create React App standardmäßig `build`, Vite standardmäßig `dist`) können jedoch eine andere Konfiguration haben. Wenn in Actions ein Fehler auftritt, dass Dateien nicht gefunden werden, überprüfen Sie den Ausgabepfad in der `docusaurus.config.js`.

### 3. Unterpfad-Problem

Wenn Ihr Repository keine Benutzer-Homepage (d. h. nicht `username.github.io`), sondern eine Projekt-Homepage (z. B. `username.github.io/project-name`) ist, müssen Sie `baseUrl` konfigurieren.

In `docusaurus.config.js`:

```javascript
module.exports = {
  // ...
  url: 'https://hagicode.com', // Ihre HagiCode-URL
  baseUrl: '/', // Stammpfad-Bereitstellung
  // ...
};
```
Dies wird leicht übersehen. Eine falsche Konfiguration führt zu einer weißen Seite, da die Ressourcenpfade nicht geladen werden können.

## Ergebnis überprüfen

Nach der Konfiguration aller Einstellungen und dem Push des Codes können wir auf der **Actions**-Seite des GitHub-Repositorys zuschauen.

Sie sehen einen gelben Kreis (Workflow läuft), der bei Erfolg grün wird! Wenn er rot wird, klicken Sie hinein, um die Protokolle zu überprüfen – normalerweise lässt sich das Problem eingrenzen (meistens sind es Tippfehler oder falsche Pfadkonfigurationen).

Nach erfolgreichem Build ist die neue Website unter `https://<Ihr-Benutzername>.github.io/<Repository-Name>/` verfügbar.

## Zusammenfassung

Durch die Einführung von GitHub Actions haben wir die automatische Bereitstellung der HagiCode-Dokumentationswebsite erfolgreich realisiert. Dies spart nicht nur Zeit für manuelle Operationen, sondern stellt vor allem die Standardisierung des Veröffentlichungsprozesses sicher. Jetzt kann jeder, der die Dokumentation aktualisiert, nach dem Merge in den `main`-Zweig die neuesten Inhalte innerhalb weniger Minuten online sehen.

**Kernvorteile**:
- **Effizienzsteigerung**: Von „manuell paketieren und hochladen" zu „Code ist Veröffentlichung".
- **Fehlerreduktion**: Eliminierung der Möglichkeit menschlicher Bedienfehler.
- **Erlebnisoptimierung**: Ermöglicht Entwicklern, sich auf die Inhaltsqualität zu konzentrieren, anstatt von komplizierten Bereitstellungsprozessen aufgehalten zu werden.

Obwohl die Einrichtung von CI/CD anfangs etwas mühsam ist (insbesondere die verschiedenen Berechtigungs- und Pfadprobleme), ist es eine einmalige Investition mit langfristig enormem Nutzen. Es wird dringend empfohlen, alle statischen Website-Projekte mit ähnlichen Automatisierungsprozessen auszustatten.
