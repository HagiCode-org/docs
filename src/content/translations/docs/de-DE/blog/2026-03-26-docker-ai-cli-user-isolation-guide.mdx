## KI-CLI-Werkzeuge in Docker-Containern betreiben: Ein praktischer Leitfaden zur Benutzertrennung und persistenten Volumes

> Die Integration von KI-Codierungswerkzeugen wie Claude Code, Codex und OpenCode in containerisierte Umgebungen klingt einfach, aber überall lauern versteckte Komplexitäten. Dieser Artikel bietet einen tiefen Einblick, wie das HagiCode-Projekt Kernherausforderungen bei Docker-Deployment löst, einschließlich Benutzerberechtigungen, Konfigurationspersistenz und Versionsverwaltung, damit Sie die häufigsten Fallen vermeiden können.

## Hintergrund

Als wir beschlossen, KI-Codierungs-CLI-Werkzeuge in Docker-Containern zu betreiben, war der intuitivste Gedanke wahrscheinlich: "Container laufen doch als root, oder? Warum nicht einfach alles direkt installieren und fertig?" In der Realität verbirgt sich hinter dieser scheinbar einfachen Idee ein paar Kernprobleme, die gelöst werden müssen.

Erstens sind **Sicherheitseinschränkungen die erste Hürde**. Nehmen wir Claude CLI als Beispiel: Es verbietet ausdrücklich die Ausführung als Root-Benutzer. Das ist eine obligatorische Sicherheitsprüfung, und wenn Root erkannt wird, verweigert es den Start. Man könnte denken: Kann ich nicht einfach mit der `USER`-Direktive den Benutzer wechseln? Es ist nicht so einfach. Es gibt immer noch ein Abbildungsproblem zwischen dem Nicht-Root-Benutzer im Container und den Benutzerberechtigungen auf dem Host-System.

Zweitens ist **Zustandspersistenz die zweite Falle**. Claude Code erfordert eine Anmeldung, Codex hat seine eigene Konfiguration, und OpenCode hat ebenfalls ein Cache-Verzeichnis. Wenn Sie bei jedem Neustart des Containers alles neu konfigurieren müssen, verliert die ganze Idee der "Automatisierung" ihren Sinn. Wir brauchen diese Konfigurationen persistent, und sie sollten Container-Neustarts überleben.

Drittens ist **Versionsverwaltung eine dauerhafte Sorge**. CLI-Werkzeuge werden häufig aktualisiert, und die verwaltete Installation und Aktualisierung in einem Container-Setup erfordert einen sorgfältigen Ansatz.

Vierte, **Netzwerkisolierung und Proxy-Konfiguration** müssen ebenfalls berücksichtigt werden, da KI-CLI-Werkzeuge API-Aufrufe an externe Dienste durchführen und die Netzwerkkonfiguration im Container korrekt eingerichtet sein muss.

Wir sind in all diese Fallen getappt, und dieser Artikel teilt unsere Lösungen. Nichts Besonderes eigentlich; es ist einfach ein paar hart verdiente Erfahrung aus der Praxis.

## Kernproblem 1: Sicherheit und Benutzertrennung

### Das Root-Verbot verstehen

Viele KI-CLI-Werkzeuge haben Sicherheitsprüfungen, die die Ausführung als Root verbieten. Das ist kein Zufall, sondern eine bewusste Sicherheitsentscheidung:

```bash
# Claude CLI verweigert den Start als Root
$ docker run --rm claude:latest claude
Error: Claude Code cannot run as root. Please run as a non-root user.
```

Dieses Verbot existiert aus gutem Grund. Ein als Root laufender CLI hat vollen Zugriff auf das Dateisystem, was ein Sicherheitsrisiko darstellt, insbesondere wenn der CLI Werkzeugaufrufe unterstützt, die Befehle auf dem Host-System ausführen können.

### Lösung: Benutzer-Abbildung

Die Kernidee der Lösung besteht darin, einen Nicht-Root-Benutzer im Container zu erstellen, dessen UID/GID mit dem Benutzer auf dem Host-System übereinstimmt. Dadurch werden Dateiberechtigungsprobleme vermieden, wenn Dateien zwischen Container und Host geteilt werden.

```dockerfile
# Dockerfile
FROM node:20-bookworm

# Argumente für Host-Benutzer-UID und GID
ARG HOST_UID=1000
ARG HOST_GID=1000

# Gruppe und Benutzer mit denselben IDs wie der Host erstellen
RUN groupadd -g ${HOST_GID} appgroup && \
    useradd -m -u ${HOST_UID} -g ${HOST_GID} -s /bin/bash appuser

# Arbeitsverzeichnis erstellen und Berechtigungen setzen
RUN mkdir -p /home/appuser/workspace && \
    chown -R appuser:appgroup /home/appuser

# Zum Nicht-Root-Benutzer wechseln
USER appuser
WORKDIR /home/appuser/workspace

# CLI installieren (als Nicht-Root-Benutzer)
RUN npm install -g @anthropic-ai/claude-code@latest

ENTRYPOINT ["claude"]
```

Die Verwendung von Build-Args (`HOST_UID` und `HOST_GID`) ermöglicht eine dynamische Anpassung an die Benutzer-IDs des Host-Systems. Dies ist ein häufiges und bewährtes Muster in containerisierten Setups.

### Benutzer-IDs automatisch ermitteln

Beim `docker compose`-Build können die Benutzer-IDs automatisch ermittelt werden:

```yaml
# docker-compose.yml
services:
  claude:
    build:
      context: .
      args:
        HOST_UID: ${HOST_UID:-1000}
        HOST_GID: ${HOST_GID:-1000}
```

Und der Build-Befehl:

```bash
# Benutzer-IDs des aktuellen Benutzers ermitteln und an Docker übergeben
HOST_UID=$(id -u) HOST_GID=$(id -g) docker compose build
```

Dieser Ansatz stellt sicher, dass Dateien, die vom Container erstellt werden, auf dem Host-System die korrekten Besitzer haben. Ohne diese Übereinstimmung würden Dateien, die vom Container erstellt wurden, Root gehören, was zu Berechtigungsproblemen führen würde.

## Kernproblem 2: Zustandspersistenz

### Warum Persistenz wichtig ist

KI-CLI-Werkzeuge speichern verschiedene Zustandsdaten:

| Werkzeug | Zustandsdaten | Speicherort |
|----------|--------------|-------------|
| Claude Code | Anmeldeinformationen, Sitzungsverlauf, Konfiguration | `~/.claude/` |
| Codex | API-Schlüssel, Sitzungsbindungen, SQLite-Datenbank | `~/.codex/` |
| OpenCode | Konfiguration, Cache, SQLite-Datenbank | `~/.opencode/` |

Ohne Persistenz gehen diese Daten bei jedem Container-Neustart verloren, was bedeutet, dass der Benutzer sich jedes Mal neu anmelden und konfigurieren muss. Das ist keine akzeptable Benutzererfahrung.

### Lösung: Docker-Volumes

Die Lösung besteht darin, Docker-Volumes zu verwenden, um die Zustandsverzeichnisse zwischen Container und Host zu teilen:

```yaml
# docker-compose.yml
services:
  claude:
    image: hagicode/claude:latest
    volumes:
      # Zustandsverzeichnisse einhängen
      - claude-config:/home/appuser/.claude
      - workspace:/home/appuser/workspace
    environment:
      - CLAUDE_API_KEY=${CLAUDE_API_KEY}

volumes:
  claude-config:
    driver: local
  workspace:
    driver: local
```

### Named Volumes vs. Bind Mounts

Es gibt zwei Ansätze für Volumes, und jeder hat seine Vor- und Nachteile:

**Named Volumes** (empfohlen für CLI-Konfiguration):
```yaml
volumes:
  - claude-config:/home/appuser/.claude
```
- Vorteile: Docker verwaltet den Speicherort, keine Pfadprobleme
- Nachteile: Schwerer direkt auf dem Host zuzugreifen

**Bind Mounts** (nützlich für Arbeitsverzeichnisse):
```yaml
volumes:
  - ./workspace:/home/appuser/workspace
```
- Vorteile: Direkter Zugriff auf Host-Dateien, einfach zu debuggen
- Nachteile: Berechtigungsprobleme bei UID/GID-Mismatch

Die Empfehlung ist, Named Volumes für CLI-Konfigurationen zu verwenden (die der Benutzer normalerweise nicht direkt bearbeiten muss) und Bind Mounts für das Arbeitsverzeichnis (das der Benutzer direkt bearbeiten möchte).

### Volume-Initialisierung

Ein häufiges Problem ist, dass ein Volume beim ersten Erstellen leer ist. Wenn der CLI-Werkzeug ein bestimmtes Verzeichnis erwartet, muss es initialisiert werden:

```dockerfile
# Im Dockerfile
RUN mkdir -p /home/appuser/.claude && \
    chown -R appuser:appgroup /home/appuser/.claude

# Eintrittspunkt-Skript für die Initialisierung
COPY entrypoint.sh /home/appuser/entrypoint.sh
RUN chmod +x /home/appuser/entrypoint.sh

ENTRYPOINT ["/home/appuser/entrypoint.sh"]
```

```bash
#!/bin/bash
# entrypoint.sh

# Sicherstellen, dass Konfigurationsverzeichnisse existieren
mkdir -p ~/.claude ~/.codex ~/.opencode

# Claude ausführen
exec "$@"
```

## Kernproblem 3: Netzwerk- und Proxy-Konfiguration

### API-Kommunikation im Container

KI-CLI-Werkzeuge müssen mit externen APIs kommunizieren, was eine korrekte Netzwerkkonfiguration im Container erfordert. In einigen Umgebungen (besonders in China) ist ein Proxy erforderlich:

```yaml
# docker-compose.yml
services:
  claude:
    environment:
      # Proxy-Konfiguration
      - HTTP_PROXY=${HTTP_PROXY:-}
      - HTTPS_PROXY=${HTTPS_PROXY:-}
      - NO_PROXY=${NO_PROXY:-localhost,127.0.0.1}
```

### DNS-Auflösung

In einigen Netzwerkumgebungen kann die DNS-Auflösung im Container problematisch sein. Docker bietet Konfigurationsoptionen dafür:

```yaml
# docker-compose.yml
services:
  claude:
    dns:
      - 8.8.8.8
      - 8.8.4.4
    extra_hosts:
      - "api.anthropic.com:your-proxy-ip"
```

## Kernproblem 4: Mehrere CLIs verwalten

Wenn Sie mehrere KI-CLI-Werkzeuge im selben Container betreiben möchten, muss die Konfiguration sorgfältig gehandhabt werden:

```dockerfile
# Mehrere CLIs installieren
RUN npm install -g @anthropic-ai/claude-code@latest && \
    npm install -g @openai/codex@latest && \
    npm install -g opencode@latest
```

Jeder CLI hat seine eigenen Zustandsverzeichnisse, die alle als separate Volumes gemountet werden müssen:

```yaml
services:
  hagicode:
    volumes:
      - claude-config:/home/appuser/.claude
      - codex-config:/home/appuser/.codex
      - opencode-config:/home/appuser/.opencode
      - workspace:/home/appuser/workspace

volumes:
  claude-config:
  codex-config:
  opencode-config:
  workspace:
```

## Praktische Hinweise

### 1. Gesundheitstests

Fügen Sie einen Gesundheitstest hinzu, um sicherzustellen, dass der CLI im Container korrekt funktioniert:

```dockerfile
HEALTHCHECK --interval=30s --timeout=10s --retries=3 \
  CMD claude --version || exit 1
```

### 2. Multiplattform-Builds

Verwenden Sie `buildx`, um Images für mehrere Plattformen zu erstellen:

```bash
docker buildx build --platform linux/amd64,linux/arm64 -t hagicode/claude:latest .
```

### 3. Image-Größe optimieren

Verwenden Sie Multi-Stage-Builds, um die Image-Größe zu reduzieren:

```dockerfile
# Build-Stage
FROM node:20-bookworm AS builder
RUN npm install -g @anthropic-ai/claude-code@latest

# Runtime-Stage
FROM node:20-bookworm-slim
COPY --from=builder /usr/local/lib/node_modules /usr/local/lib/node_modules
```

### 4. Logging

KI-CLI-Werkzeuge geben oft wichtige Diagnoseinformationen aus. Stellen Sie sicher, dass die Logs korrekt erfasst werden:

```yaml
services:
  claude:
    logging:
      driver: json-file
      options:
        max-size: "10m"
        max-file: "3"
```

## Zusammenfassung

Die Integration von KI-CLI-Werkzeugen in Docker-Container erfordert die Lösung von vier Kernproblemen:

1. **Sicherheit und Benutzertrennung**: Einen Nicht-Root-Benutzer erstellen, dessen UID/GID mit dem Host-Benutzer übereinstimmt
2. **Zustandspersistenz**: Docker-Volumes verwenden, um CLI-Konfigurationen und -Zustände zu persistieren
3. **Netzwerkkonfiguration**: Proxy- und DNS-Einstellungen korrekt konfigurieren
4. **Multi-CLI-Verwaltung**: Mehrere CLIs im selben Container mit separaten Volumes verwalten

Dieser Ansatz bildet die Grundlage für HagiCodes Docker-Deployment-Option und bietet Benutzern eine ein-Klick-Installationserfahrung. Wenn Sie KI-CLI-Werkzeuge in containerisierte Umgebungen integrieren möchten, hoffe ich, dass dieser Artikel einige nützliche Hinweise bietet.
