## Browser-WebSocket-Authentifizierungsproblem lösen: Proxy-Lösung für Doubao-Spracherkennung in der Praxis

> Die Browser-WebSocket-API unterstützt keine benutzerdefinierten HTTP-Header, was für Spracherkennungsdienste, die Authentifizierungsinfodermationen über Header übertragen, eine Herausfoderderung darstellt. Dieser Artikel zeigt, wie das HagiCode-Projekt dieses Problem durch eine Backend-Proxy-Lösung bewältigt hat, vom Playground bis zur Produktionsumgebung.


## Hintergrund

Bei der Implementierung der Spracherkennungsfunktion im HagiCode-Projekt entschieden wir uns zunächst für den Spracherkennungsdienst „Doubao" von ByteDance. Das ursprüngliche Design war einfach – das Frontend verbindet sich direkt mit dem Doubao-WebSocket-Dienst. Doch Doubaos API erfoderdert die Übergabe von Authentifizierungsinfodermationen wie AccessToken und SecretKey über HTTP-Header. Das ist problematisch, da die Browser-WebSocket-API keine benutzerdefinierten Header unterstützt.

Wir standen voder zwei Optionen:
1. Authentifizierungsinfodermationen in URL-Abfrageparametern übergeben – einfach, aber unsicher
2. Einen Proxy im Backend implementieren – aufwendiger, aber sicher

Wir entschieden uns für die zweite Option – einen WebSocket-Proxy im Backend. Diese Lösung wurde zunächst in unserem Playground-Umgebung validiert und erst nach Bestätigung der Stabilität in die Produktionsumgebung übernommen.

## Technische Herausfoderderungsanalyse

### Einschränkungen der Browser-WebSocket-API

Die Standard-WebSocket-API sieht einfach aus:

```javascript
const ws = new WebSocket('wss://example.com/ws');
```

Aber das Problem liegt in der „Einfachheit" – sie überträgt nur Parameter in der URL und kann keine Header wie HTTP-Anfragen setzen:

```javascript
// Dies wird in der WebSocket-API nicht unterstützt
const ws = new WebSocket('wss://example.com/ws', {
  headers: {
    'Authoderization': 'Bearer token'
  }
});
```

### Architekturentscheidungen

**Entscheidung 1: Proxy-Modus**

| Ansatz | Voderteile | Nachteile | Entscheidung |
|--------|----------|-----------|--------------|
| Natives WebSocket | Leichtgewichtig, einfach, direkte Weiterleitung | Manuelle Verbindungsverwaltung erfoderderlich | Gewählt |
| SignalR | Automatisches Reconnect, starke Typisierung | Überkomplex, zusätzliche Abhängigkeiten | Nicht gewählt |

**Entscheidung 2: Verbindungsverwaltungsstrategie**

„Eine Verbindung, eine Session" – jede Frontend-WebSocket-Verbindung entspricht einer eigenständigen Doubao-Backend-Verbindung. Voderteile: einfache Implementierung, leicht debuggbar, Ressourcenisolation.

**Entscheidung 3: Authentifizierungsspeicherung**

Anmeldeinfodermationen in Backend-Konfigurationsdateien (appsettings.yml oder Umgebungsvariablen) gespeichert, geladen über Dependency Injection.

```

  ▼
```

## Kernkomponenten-Implementierung

### 1. WebSocket-Endpunkt-Konfiguration

```csharp
app.Map("/ws", async context =>
{
    if (context.WebSockets.IsWebSocketRequest)
    {
        var appId = context.Request.Query["appId"];
        var accessToken = context.Request.Query["accessToken"];

        if (string.IsNullOrEmpty(appId) || string.IsNullOrEmpty(accessToken))
        {
            context.Response.StatusCode = 400;
            return;
        }

        using var webSocket = await context.WebSockets.AcceptWebSocketAsync();

        var buffer = new byte[4096];
        while (!webSocket.CloseStatus.HasValue)
        {
            var result = await webSocket.ReceiveAsync(buffer, CancellationToken.None);

            if (result.MessageType == WebSocketMessageType.Close)
            {
                await webSocket.CloseAsync(
                    result.CloseStatus.Value,
                    result.CloseStatusDescription,
                    CancellationToken.None);
                break;
            }

            await HandleAudioDataAsync(buffer, result.Count);
        }
    }
});
```

### 2. Sitzungsverwaltung

```csharp
public class DoubaoSessionManager : IDoubaoSessionManager
{
    private readonly ConcurrentDictionary<string, DoubaoSession> _sessions = new();

    public DoubaoSession CreateSession(string connectionId)
    {
        var session = new DoubaoSession(connectionId);
        _sessions[connectionId] = session;
        return session;
    }

    public async Task SendAudioAsync(string connectionId, byte[] audioData)
    {
        if (_sessions.TryGetValue(connectionId, out var session))
        {
            await session.SendAudioAsync(audioData);
        }
    }

    public void RemoveSession(string connectionId)
    {
        if (_sessions.TryRemove(connectionId, out var session))
        {
            session.Dispose();
        }
    }
}
```

## Hinweise und Best Practices

1. **Verbindungsüberwachung**: Regelmäßige Session-Status-Logs, Audio-Segment-Überwachung
2. **Fehlerbehandlung**: Alle WebSocket-Ausnahmen abfangen, IAsyncDisposable für Ressourcenbereinigung
3. **Audiofodermat**: 16000 Hz, 16-bit, Mono, PCM (raw)
4. **Sicherheit**: Anmeldeinfodermationen nur im Backend, Verbindungslimits, HTTPS/WSS in Produktion
5. **Perfodermance**: Asynchrone Operationen, angemessene Puffergröße (4096 Bytes)

## Bereitstellungsempfehlungen

1. **Docker**: Proxy-Service als Container verpacken
2. **Load Balancing**: Nginx oder Envoy für WebSocket-Reverse-Proxy
3. **Health Check**: Heartbeat-Mechanismus
4. **Log-Aggregation**: Zentrales Log-System (ELK, Loki)
