## HagiCode und GLM-5.1 Multi-CLI-Integrationsleitfaden

:::note[Zeitkontext-Hinweis]
Dieser Artikel dokumentiert das Multi-CLI-Integrationsdesign und die Modellzuordnung zum Zeitpunkt der Veröffentlichung. Für den aktuell dauerhaft unterstützten Umfang konsultieren Sie bitte [AI Agent CLI Installation](/en/related-software-installation/ai-agent-cli/) und den [Installationsleitfaden](/en/installation); das hier erwähnte `IFlowCli` stellt lediglich eine historische Konfiguration zu jenem Zeitpunkt dar und gehört nicht zum aktuellen aktiven Unterstützungsumfang.
:::

## Hintergrund

Im HagiCode-Projekt können Nutzer aus mehreren CLI-Tools wählen, um KI-Programmierassistenten anzusteuern, darunter Claude Code CLI, GitHub Copilot, OpenCode CLI, Codebuddy CLI, Hermes CLI und weitere. Diese CLI-Tools sind eigenständige KI-Programmierwerkzeuge, die durch die Abstraktionsschicht von HagiCode jedoch flexibel mit verschiedenen KI-Modellanbietern verbunden werden können.

Zhipu AI (ZAI) stellt eine zur Anthropic Claude API kompatible Schnittstelle bereit, sodass diese CLI-Tools direkt die inländischen GLM-Reihenmodelle verwenden können. Darunter ist GLM-5.1 das neueste große Sprachmodell von Zhipu mit deutlichen Verbesserungen gegenüber GLM-5.0.

## CLI-Abstraktionsarchitektur von HagiCode

HagiCode definiert 11 CLI-Anbietertypen durch den `AIProviderType`-Enum, der gängige KI-Programmier-CLI-Tools abdeckt:

```csharp
public enum AIProviderType
{
    ClaudeCodeCli = 0,    // Claude Code CLI
    CodexCli = 1,          // GitHub Copilot Codex
    GitHubCopilot = 2,     // GitHub Copilot
    CodebuddyCli = 3,     // Codebuddy CLI
    OpenCodeCli = 4,      // OpenCode CLI
    IFlowCli = 5,         // IFlow CLI
    HermesCli = 6,        // Hermes CLI
    QoderCli = 7,         // Qoder CLI
    KiroCli = 8,          // Kiro CLI
    KimiCli = 9,          // Kimi CLI
    GeminiCli = 10,       // Gemini CLI
}
```

Jedes CLI verfügt über eine entsprechende Modellparameterkonfiguration und unterstützt die Parameter `model` und `reasoning`:

```csharp
private static readonly IReadOnlyDictionary<AIProviderType, IReadOnlyList<string>> ManagedModelParameterKeysByProvider =
    new Dictionary<AIProviderType, IReadOnlyList<string>>
    {
        [AIProviderType.ClaudeCodeCli] = ["model", "reasoning"],
        [AIProviderType.CodexCli] = ["model", "reasoning"],
        [AIProviderType.OpenCodeCli] = ["model", "reasoning"],
        [AIProviderType.HermesCli] = ["model", "reasoning"],
        [AIProviderType.CodebuddyCli] = ["model", "reasoning"],
        [AIProviderType.QoderCli] = ["model", "reasoning"],
        [AIProviderType.KiroCli] = ["model", "reasoning"],
        [AIProviderType.GeminiCli] = ["model"],  // Gemini unterstützt den reasoning-Parameter nicht
        // ...
    };
```

| Modell-ID | Name | Standard-Reasoning | Kompatible CLI-Familien |
|----------|------|-------------------|-------------------------|
| `glm-4.7` | GLM 4.7 | hoch | claude, codebuddy, hermes, qoder, kiro |
| `glm-5` | GLM 5 | hoch | claude, codebuddy, hermes, qoder, kiro |
| `glm-5-turbo` | GLM 5 Turbo | hoch | claude, codebuddy, hermes, qoder, kiro |

## GLM-5.1 in verschiedenen CLIs verwenden

### 1. Claude Code CLI + GLM-5.1

Claude Code CLI ist eines der Kern-CLIs von HagiCode und wird über das Hero-Konfigurationssystem konfiguriert:

```json
{
  "primaryProfessionId": "profession-claude-code",
  "secondaryProfessionId": "secondary-glm-5-1",
  "model": "glm-5.1",
  "reasoning": "high"
}
```

Entsprechende `HeroEquipmentCatalogItem`-Konfiguration:

```typescript
{
  id: 'secondary-glm-5-1',
  name: 'GLM 5.1',
  family: 'anthropic',
  kind: 'model',
  primaryFamily: 'claude',
  compatiblePrimaryFamilies: ['claude', 'codebuddy', 'hermes', 'qoder', 'kiro'],
  defaultParameters: {
    model: 'glm-5.1',
    reasoning: 'high'
  }
}
```

### 2. OpenCode CLI + GLM-5.1

OpenCode CLI ist das flexibelste CLI und unterstützt die Angabe eines beliebigen Modells im Format `provider/model`:

**Methode 1: ZAI-Anbieterpräfix verwenden**
```json
{
  "primaryProfessionId": "profession-opencode",
  "model": "zai/glm-5.1",
  "reasoning": "high"
}
```

**Methode 2: Modell-ID direkt verwenden**
```json
{
  "model": "glm-5.1"
}
```

**Methode 3: Frontend-Konfigurationsoberfläche**

In `HeroModelEquipmentForm.tsx` hat OpenCode CLI einen eigenen Platzhalterhinweis:

```tsx
const OPEN_CODE_MODEL_PLACEHOLDER = 'myprovider/glm-4.7';

const modelPlaceholder = primaryProviderType === PCode_Models_AIProviderType.OPEN_CODE_CLI
  ? OPEN_CODE_MODEL_PLACEHOLDER
  : 'gpt-5.4';
```

Nutzer können eingeben:
```
zai/glm-5.1
glm-5.1
```

**OpenCode CLI-Modell-Parsing-Logik**:

```csharp
internal OpenCodeModelSelection? ResolveModelSelection(string? rawModel)
{
    var normalized = NormalizeOptionalValue(rawModel);
    if (normalized == null) return null;

    var slashIndex = normalized.IndexOf('/');
    if (slashIndex < 0)
    {
        // Kein Schrägstrich: Modell-ID direkt verwenden
        return new OpenCodeModelSelection {
            ProviderId = string.Empty,
            ModelId = normalized,
        };
    }

    // Schrägstrich vorhanden: provider/model-Format parsen
    var providerId = normalized[..slashIndex].Trim();
    var modelId = normalized[(slashIndex + 1)..].Trim();

    return new OpenCodeModelSelection {
        ProviderId = providerId,
        ModelId = modelId,
    };
}
```

### 3. Codebuddy CLI + GLM-5.1

Codebuddy CLI verfügt über eine spezielle Legacy-Verarbeitungslogik:

```json
{
  "primaryProfessionId": "profession-codebuddy",
  "model": "glm-5.1",
  "reasoning": "high"
}
```

**Hinweis**: Codebuddy behält eine spezielle Behandlung für GLM-5.0 bei und verwendet keine Legacy-Normierung:

```csharp
return !string.Equals(providerName, "CodebuddyCli", StringComparison.OrdinalIgnoreCase)
       && string.Equals(normalizedModel, LegacyGlm5TurboModelValue, StringComparison.OrdinalIgnoreCase)
    ? Glm5TurboModelValue
    : normalizedModel;
// Für CodebuddyCli wird glm-5.0 nicht zu glm-5-turbo normiert
```

## Verbesserungsvorteile von GLM-5.1

Verglichen mit GLM-5.0 bringt GLM-5.1 die folgenden wesentlichen Verbesserungen:

### 1. Bessere Reasoning-Fähigkeit

Laut den offiziellen Veröffentlichungsinformationen von Zhipu umfassen die Verbesserungen in GLM-5.1:

- **Stärkeres Codeverständnis**: Präzisere Analyse komplexer Codestrukturen
- **Längeres Kontextverständnis**: Unterstützt längere Gesprächskontexte
- **Verbessertes Tool-Aufrufen**: Höhere Erfolgsquote für MCP-Toolaufrufe
- **Ausgabestabilität**: Reduziert Zufälligkeit und Halluzinationen

### 2. Umfassende Multi-CLI-Kompatibilität

GLM-5.1 deckt alle von HagiCode unterstützten gängigen CLIs ab:

```
compatiblePrimaryFamilies: [
  "claude",      // Claude Code CLI
  "codebuddy",   // Codebuddy CLI
  "hermes",      // Hermes CLI
  "qoder",       // Qoder CLI
  "kiro"         // Kiro CLI
]
```

## Hinweise

### 1. API-Schlüsselkonfiguration

Stellen Sie sicher, dass die Umgebungsvariable `ANTHROPIC_AUTH_TOKEN` korrekt gesetzt ist. Sie ist die erforderliche Zugriffsberechtigung für jedes CLI, um sich mit dem Modell zu verbinden.

### 2. Modellverfügbarkeit

GLM-5.1 muss vom entsprechenden Modellanbieter aktiviert werden:
- Die Zhipu AI ZAI-Plattform unterstützt es standardmäßig
- Alibaba Cloud DashScope erfordert möglicherweise eine separate Beantragung

### 3. OpenCode CLI-Format

Bei Verwendung des Formats `provider/model` stellen Sie sicher, dass die Anbieter-ID korrekt ist:
- Zhipu AI: `zai` oder `zhipuai`
- Alibaba Cloud: `aliyun` oder `dashscope`

### 4. Reasoning-Parameter

- `high` wird für die besten Codeerzeugungsergebnisse empfohlen
- Gemini CLI unterstützt den Reasoning-Parameter nicht und ignoriert diese Konfiguration automatisch

## Zusammenfassung

Durch eine einheitliche Abstraktionsschicht ermöglicht HagiCode eine flexible Integration zwischen GLM-5.1 und mehreren CLIs. Entwickler können das CLI-Tool wählen, das am besten zu ihren Präferenzen und Nutzungsszenarien passt, und dann über einfache Konfiguration das neueste GLM-5.1-Modell verwenden.

Als neueste Modellversion von Zhipu bietet GLM-5.1 klare Verbesserungen gegenüber GLM-5.0:
- Ein eigenständiger Versionsidentifikator ohne Legacy-Last
- Stärkeres Reasoning und Codeverständnis
- Breite Multi-CLI-Kompatibilität
- Flexible Reasoning-Stufenkonfiguration

Mit den korrekten Umgebungsvariablen und der Hero-Ausstattung konfiguriert, können Nutzer die volle Leistung von GLM-5.1 in verschiedenen CLI-Umgebungen entfalten.
