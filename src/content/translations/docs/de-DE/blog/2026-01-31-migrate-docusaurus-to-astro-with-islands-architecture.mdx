## Von Docusaurus 3.x zu Astro 5.x: HagiCode-Seitenmigration in der Praxis

> Dieser Artikel blickt auf den vollständigen Migrationsprozess der HagiCode-Website von Docusaurus 3.x zu Astro 5.x zurück. Wir beleuchten, wie die Islands-Architektur von Astro Performance-Engpässe löst, während bestehende React-Komponenten erhalten bleiben, um sowohl Build-Geschwindigkeit als auch Ladeleistung zu verbessern.


## Hintergrund

Im Januar 2026 führten wir an der HagiCode-Website eine „Herztransplantation" durch – die Umstellung des Kernframeworks von Docusaurus 3.x auf Astro 5.x. Dies war keine impulsiventschlossene Restrukturierung, sondern eine wohlüberlegte technische Entscheidung.

Vor der Migration war unsere Website zwar funktional vollständig, zeigte aber zunehmend Anzeichen von „Wohlstandskrankheit": Aufgeblähte Build-Artefakte, hohe JavaScript-Last und unzureichende Seitenladezeit bei komplexen Dokumentationsseiten. Als KI-Code-Assistent-Projekt muss HagiCode Dokumentationen und Funktionsbeschreibungen häufig aktualisieren, weshalb die Build-Effizienz die Veröffentlichungsgeschwindigkeit direkt beeinflusst. Gleichzeitig wollten wir die Website für Suchmaschinen (SEO) freundlicher gestalten, damit mehr Entwickler das Projekt entdecken.

Um diese Probleme zu lösen, trafen wir eine mutige Entscheidung: Das gesamte Build-System wurde neu aufgebaut und zu Astro migriert. Die daraus resultierenden Veränderungen waren größer als man erwarten könnte – dazu später mehr.

## Über HagiCode

Die in diesem Artikel vorgestellte Migrationslösung stammt aus der Praxis im [HagiCode](https://github.com/HagiCode-org/site) Projekt.

HagiCode ist ein KI-Code-Assistent, der sich der Steigerung der Entwicklungseffizienz widmet. Wir achten nicht nur auf die Iteration der Kernfunktionen, sondern legen gleich viel Wert auf die Entwicklererfahrung. Diese Website-Restrukturierung soll sicherstellen, dass Benutzer beim Durchsuchen der Dokumentation und der Website eine extreme Ladegeschwindigkeit erleben.

## Warum das reife Docusaurus aufgeben?

Im React-Ökosystem war Docusaurus lange die „Standardantwort" für Dokumentationswebsites. Es ist sofort einsatzbereit, reich an Plugins und hat eine aktive Community. Mit zunehmender Funktionalität von HagiCode spürten wir jedoch seine Grenzen:

1.  **Performance-Engpass**: Docusaurus ist im Kern eine React SPA (Single-Page-Anwendung). Selbst für reine statische Seiten muss der Client die React-Laufzeit laden und das Hydration durchführen – für einfache Dokumentationsseiten ist das zu schwer.
2.  **Ressourcenvolumen**: Selbst bei wenig Seiteninhalt ist das gepackte JS-Volumen relativ konstant, was für mobile Nutzer und Umgebungen mit schlechter Netzwerkverbindung nicht ideal ist.
3.  **Unzureichende Flexibilität**: Zwar ist es erweiterbar, aber bei der Anpassung des Build-Prozesses wünschten wir uns mehr Kontrolle auf niedriger Ebene.

Astro löst genau diese Probleme. Es bietet eine völlig neue „Islands-Architektur": Standardmäßig generiert Astro statisches HTML ohne JavaScript; nur Komponenten, die Interaktion benötigen, werden „aktiviert" und laden JS. Das bedeutet, dass der Großteil unserer Website aus reinem HTML besteht – extrem schnell.

## Kernmigrationsstrategie: Sanfter Architekturübergang

Migration ist kein einfaches Kopieren und Einfügen, sondern ein Paradigmenwechsel. Wir wechselten von Docusaurus' „Voll-React-Modus" zu Astros „Core + Islands"-Modus.

### 1. Konfigurationssystem-Refactoring

Zuerst mussten wir von `docusaurus.config.ts` zu `astro.config.mjs` wechseln. Das ist nicht nur eine Namensänderung, sondern ein Neu schreiben der Routing- und Build-Logik.

In Docusaurus ist alles ein Plugin; in Astro ist alles eine Integration. Wir mussten den Basispfad der Website, den Build-Ausgabemodus (statisch vs. SSR) und die Ressourcenkomprimierungsstrategie neu definieren.

**Vor der Migration:**
```typescript
// docusaurus.config.ts
export default {
  title: 'HagiCode',
  url: 'https://hagicode.com',
  baseUrl: '/',
  // ... mehr Konfiguration
};
```

**Nach der Migration:**
```javascript
// astro.config.mjs
import { defineConfig } from 'astro/config';
import react from '@astrojs/react';

export default defineConfig({
  integrations: [react()],
  site: 'https://hagicode.com',
  base: '/',
  // Optimierungskonfiguration für statische Ressourcen
  build: {
    inlineStylesheets: 'auto',
  },
});
```

### 2. React-Komponenten: Behalten oder Umschreiben

Dies war der schwierigste Teil der Migration. Unsere bestehende Website hatte viele React-Komponenten (wie Tabs-Komponente, Code-Hervorhebung, Feedback-Button usw.). Einfach alles wegzuwerfen wäre verschwenderisch, alles zu behalten würde zu hoher JS-Last führen.

HagiCode wählte eine **progressive Hydration**-Strategie:
*   **Rein statische Komponenten**: Für Anzeigekomponenten (wie Header, Footer, reine Textdokumentation) wurden sie als Astro-Komponenten (`.astro`-Dateien) neu geschrieben und zur Build-Zeit direkt als HTML gerendert.
*   **Interaktive Inseln**: Für Komponenten, die zwingend Interaktion erfordern (wie Theme-Umschalter, Tabs, Code-Kopier-Button), blieb die React-Implementierung erhalten, ergänzt durch `client:load`- oder `client:visible`-Direktiven.

Zum Beispiel unsere häufig verwendete Tabs-Komponente in der Dokumentation:

```jsx
// src/components/Tabs.jsx
import { useState } from 'react';
import './Tabs.css'; // 样式 importieren

export default function Tabs({ items }) {
  const [activeIndex, setActiveIndex] = useState(0);
  // ... Zustandslogik
  return (
    <div className="tabs-wrapper">
      {/* Renderlogik */}
    </div>
  );
}
```

Bei der Verwendung in Markdown teilen wir Astro explizit mit: „Diese Komponente braucht JS":

```markdown
// src/content/docs/example.mdx
import Tabs from '../../components/Tabs.jsx';

<!-- JS wird nur geladen, wenn die Komponente in den Viewport eintritt -->
<Tabs client:visible items={...} />
```

So beanspruchen interaktive Komponenten außerhalb des Viewports keine Bandbreite, was die First-Paint-Ladezeit erheblich optimiert.

### 3. Stylesystem-Anpassung: Von CSS Modules zu Scoped

Docusaurus unterstützt standardmäßig CSS Modules, während Astro Scoped CSS (über `<style>`-Tags) bevorzugt. Beide Ansätze verfolgen das Ziel der Stilisolation, jedoch mit unterschiedlicher Syntax.

In der HagiCode-Migration zerlegten wir die meisten komplexen CSS Modules in Astros Scoped-Styles. Das ist eigentlich eine gute Sache, da in `.astro`-Dateien Styles und Templates in derselben Datei stehen, was die Wartung intuitiver macht.

**Vor der Umstellung:**
```css
/* Tabs.module.css */
.wrapper { background: var(--ifm-background-color); }
```

**Nach der Umstellung (Astro Scoped):**
```html
<!-- Tabs.astro -->
<div class="tabs-wrapper">
  <slot />
</div>

<style>
  .tabs-wrapper {
    /* Direkte Verwendung von CSS-Variablen, themenkompatibel */
    background: var(--bg-color);
    padding: 1rem;
  }
</style>
```

Gleichzeitig vereinheitlichten wir das globale CSS-Variablensystem und nutzten Astros Umgebungserkennung, um einen nahtlosen Dark-Mode-Wechsel zwischen verschiedenen Seiten sicherzustellen.

## Praktische Probleme und Lösungen

Bei der tatsächlichen Migration von HagiCode stießen wir auf einige Probleme, von denen wir hier die typischsten teilen.

### 1. Pfad- und Umgebungsvariablen-Probleme

HagiCode unterstützt die Bereitstellung in Unterpfaden (z. B. in ein Unterverzeichnis von GitHub Pages). Docusaurus verarbeitet `baseUrl` automatisch. In Astro müssen wir jedoch bei der Handhabung von Bildlinks und API-Anfragen vorsichtiger sein.

Wir führten einen Umgebungsvariablenmechanismus zur einheitlichen Verwaltung ein:

```javascript
// 在构建脚本中处理路径
const getBasePath = () => import.meta.env.VITE_SITE_BASE || '/';
```

Vergessen Sie nicht, hartcodierte Pfade, die mit `/` beginnen, im Code zu vermeiden. In der Entwicklungs- und Produktionsumgebung oder nach Konfiguration eines `base`-Pfads führt dies zu 404-Fehlern für Ressourcen.

### 2. CommonJS-Skriptkompatibilität

Unsere alte Website hatte einige Node.js-Skripte (für automatisches Abrufen von Metriken, Aktualisierung von Sitemaps usw.), die in CommonJS (`require`) geschrieben waren. Astro und moderne Build-Tools setzen vollständig auf ES Modules (`import`/`export`).

Wenn Sie ähnliche Skripte haben, denken Sie daran, sie vollständig auf ES Modules umzustellen. Das ist der Trend – je früher Sie ändern, desto weniger Kopfschmerzen.

```javascript
// 旧方式
const fs = require('fs');

// 新方式
import fs from 'fs';
```

### 3. Vergessen Sie SEO und Weiterleitungen nicht

Suchmaschinen hatten bereits die alten Docusaurus-Seiten von HagiCode indiziert. Wenn Sie direkt zu Astro wechseln und sich die URL-Struktur ändert, führt dies zu massenhaften 404-Fehlern und einem Einbruch des Rankings.

Wir konfigurierten Weiterleitungsregeln in Astro:

```javascript
// astro.config.mjs
export default defineConfig({
  redirects: {
    '/docs/old-path': '/docs/new-path',
    // 批量映射旧链接到新链接
  }
});
```

Oder die Verarbeitung auf Serverkonfigurationsebene. Stellen Sie sicher, dass alte Links per 301-Weiterleitung auf die neuen Adressen weitergeleitet werden – dies ist für SEO entscheidend.

## Zusammenfassung

Die Migration von Docusaurus zu Astro bedeutete für HagiCode nicht nur ein Framework-Upgrade, sondern die praktische Umsetzung einer „Performance-zuerst"-Philosophie.

**Unsere Gewinne:**
*   **Hervorragende Lighthouse-Punktzahlen**: Nach der Migration erreicht die HagiCode-Website bei der Leistungsbewertung fast die volle Punktzahl.
*   **Schnellere Build-Geschwindigkeit**: Astros inkrementeller Build hat die Veröffentlichungszeit für Dokumentationsaktualisierungen halbiert.
*   **Erhaltene Flexibilität**: Durch die Islands-Architektur haben wir keine interaktiven Funktionen aufgegeben und können weiterhin React dort einsetzen, wo es nötig ist.

Wenn Sie ebenfalls eine dokumentationsbasierte Website pflegen und unter zu großem Bundle-Volumen oder langsamer Ladezeit leiden, versuchen Sie Astro. Zwar erfordert die Migration chirurgische Eingriffe (wie das Umbenennen von PCode zu HagiCode und das Übertragen der Komponenten eins nach dem anderen), aber im Gegenzug erhalten Sie ein seidenweiches Benutzererlebnis – absolut lohnenswert.

Das in diesem Artikel vorgestellte Build-System wurde aus der tatsächlichen Praxis bei der Entwicklung von HagiCode abgeleitet – mit echten Stolpersteinen und echter Optimierung. Wenn Sie diese Lösung für wertvoll halten, zeigt das, dass unsere Engineering-Fähigkeiten nicht schlecht sind – dann ist HagiCode selbst ebenfalls einen Blick wert.
