## Von CLI-Aufrufen zur SDK-Integration: Bewaehrte Praktiken fuer GitHub Copilot in .NET-Projekten

> Der Upgrade-Pfad vom Kommandozeilenaufruf zur offiziellen SDK-Integration war eine ziemliche Reise. Heute moechte ich die Fallstricke teilen, auf die wir gestossen sind und was wir beim Bau von HagiCode gelernt haben.

## Hintergrund

Nachdem das GitHub Copilot SDK 2025 offiziell veroeffentlicht wurde, begannen wir, es in unsere KI-Faehigkeitsschicht zu integrieren. Zuvor verwendete das Projekt hauptsaechlich GitHub Copilot durch direkten Aufruf des Copilot CLI-Kommandozeilenwerkzeugs, aber dieser Ansatz hatte mehrere offensichtliche Probleme:

- **Komplexe Prozessverwaltung**: Wir mussten den CLI-Prozesslebenszyklus, Start-Timeouts und Prozessbereinigung manuell verwalten. Prozesse koennen ohne Warnung abstuerzen.
- **Unvollstaendige Ereignisbehandlung**: Roher CLI-Aufruf macht es schwierig, feingranulare Ereignisse aus Modell-Reasoning und Tool-Ausfuehrung zu erfassen.
- **Schwierige Session-Verwaltung**: Es gab keinen effektiven Mechanismus fuer Session-Wiederverwendung und -Wiederherstellung, sodass jede Interaktion von vorne beginnen musste.
- **Kompatibilitaetsprobleme**: CLI-Argumente aenderten sich haeufig, was bedeutete, dass wir staendig Kompatibilitaetslogik fuer diese Parameter pflegen mussten.

Diese Probleme wurden in der taeglichen Entwicklung immer deutlicher, besonders wenn wir den Modell-Reasoning (`thinking`) und den Tool-Ausfuehrungsstatus in Echtzeit verfolgen mussten. An diesem Punkt wurde klar, dass wir einen tieferliegenden und vollstaendigeren Integrationsansatz brauchten.

## Ueber HagiCode

Der in diesem Artikel geteilte Ansatz stammt aus unserer praktischen Erfahrung im [HagiCode](https://hagicode.com)-Projekt. HagiCode ist ein Open-Source-KI-Programmierassistent-Projekt, und waehrend der Entwicklung brauchten wir eine tiefe Integration mit den Faehigkeiten von GitHub Copilot, von der grundlegenden Codevervollstaending bis hin zu komplexen Mehr-Runden-Gespraechen und Tool-Aufrufen. Diese realen Anforderungen trieben uns dazu, vom CLI-Aufruf zum offiziellen SDK zu wechseln.

## Architekturentwurf

Das Projekt verwendet eine geschichtete Architektur, um die Einschraenkungen des CLI-Aufrufs zu beheben:

```
┌─────────────────────────────────────────────────────────┐
│  hagicode-core (Orleans Grains + AI Provider Schicht)   │
│  - CopilotAIProvider: Wandelt AIRequest in CopilotOptions um │
│  - GitHubCopilotGrain: Orleans verteilte Ausfuehrungsschnittstelle │
└─────────────────────────────────────────────────────────┘
                          ↓
┌─────────────────────────────────────────────────────────┐
│  HagiCode.Libs (Gemeinsame Provider-Schicht)           │
│  - CopilotProvider: CLI Provider-Schnittstellen-Implementierung │
│  - ICopilotSdkGateway: SDK-Aufruf-Abstraktion          │
│  - GitHubCopilotSdkGateway: SDK-Session-Verwaltung und Ereignis-Dispatch │
└─────────────────────────────────────────────────────────┘
                          ↓
┌─────────────────────────────────────────────────────────┐
│  GitHub Copilot SDK (Offizielles .NET SDK)             │
│  - CopilotClient: SDK-Client                           │
│  - CopilotSession: Session-Verwaltung                  │
│  - SessionEvent: Ereignisstrom                         │
└─────────────────────────────────────────────────────────┘
```

Dieses geschichtete Design bringt mehrere praktische technische Vorteile:

1. **Trennung der Verantwortlichkeiten**: Kerngeschaeftslogik ist von SDK-Implementierungsdetails entkoppelt.
2. **Testbarkeit**: Die `ICopilotSdkGateway`-Schnittstelle macht Unit-Tests unkompliziert.
3. **Wiederverwendbarkeit**: `HagiCode.Libs` kann von mehreren Projekten referenziert werden.
4. **Wartbarkeit**: SDK-Upgrades erfordern nur Aenderungen in der Gateway-Schicht, waehrend die oberen Schichten unberuehrt bleiben.

## Kernimplementierung

### Authentifizierungsablauf

Authentifizierung ist der erste und wichtigste Schritt der SDK-Integration. Wenn die Authentifizierung fehlschlaegt, ist alles andere bedeutungslos. Wir entwarfen eine flexible Authentifizierungskonfiguration, die mehrere Authentifizierungsquellen unterstuetzt:

```csharp
// CopilotProvider.cs - Authentifizierungsquellenkonfiguration
public class CopilotOptions
{
    public bool UseLoggedInUser { get; set; } = true;
    public string? GitHubToken { get; set; }
    public string? CliUrl { get; set; }
}

// In SDK-Anfrage umwandeln
return new CopilotSdkRequest(
    GitHubToken: options.AuthSource == CopilotAuthSource.GitHubToken
        ? options.GitHubToken
        : null,
    UseLoggedInUser: options.AuthSource != CopilotAuthSource.GitHubToken
);
```

Die Vorteile dieses Designs sind recht klar:

- Es unterstuetzt den Angemeldet-Nutzer-Modus ohne Token-Anforderung, was gut zu Desktop-Szenarien passt.
- Es unterstuetzt den GitHub-Token-Modus, der fuer serverseitige Deployments und zentralisierte Verwaltung geeignet ist.
- Es unterstuetzt das Ueberschreiben der Copilot-CLI-URL, was bei Enterprise-Proxy-Konfigurationen hilft.

### Ereignisstrombehandlung

Eine der maechtigsten Faehigkeiten des SDK ist die vollstaendige Erfassung des Ereignisstroms. Wir implementierten ein Ereignis-Dispatch-System, das alle Arten von SDK-Ereignissen in Echtzeit verarbeiten kann:

```csharp
// GitHubCopilotSdkGateway.cs - Kernlogik fuer Ereignis-Dispatch
internal static SessionEventDispatchResult DispatchSessionEvent(
    SessionEvent evt, bool sawDelta)
{
    switch (evt)
    {
        case AssistantReasoningEvent reasoningEvent:
            // Modell-Reasoning-Prozess erfassen
            events.Add(new CopilotSdkStreamEvent(
                CopilotSdkStreamEventType.ReasoningDelta,
                Content: reasoningEvent.Data.Content));
            break;

        case ToolExecutionStartEvent toolStartEvent:
            // Start eines Tool-Aufrufs erfassen
            events.Add(new CopilotSdkStreamEvent(
                CopilotSdkStreamEventType.ToolExecutionStart,
                ToolName: toolStartEvent.Data.ToolName,
                ToolCallId: toolStartEvent.Data.ToolCallId));
            break;

        case ToolExecutionCompleteEvent toolCompleteEvent:
            // Tool-Abschluss und Ergebnis erfassen
            events.Add(new CopilotSdkStreamEvent(
                CopilotSdkStreamEventType.ToolExecutionEnd,
                Content: ExtractToolExecutionContent(toolCompleteEvent)));
            break;

        default:
            // Unbehandelte Ereignisse als RawEvent beibehalten
            events.Add(new CopilotSdkStreamEvent(
                CopilotSdkStreamEventType.RawEvent,
                RawEventType: evt.GetType().Name));
            break;
    }
}
```

Der Wert dieser Implementierung ist erheblich:

- **Vollstaendige Erfassung des Modell-Reasoning-Prozesses** (`thinking`): Nutzer koennen sehen, wie die KI schliesst, nicht nur die endgueltige Antwort.
- **Echtzeit-Tracking des Tool-Ausfuehrungsstatus**: Wir wissen, welche Tools laufen, wann sie fertig sind und was sie zurueckgeben.
- **Kein Ereignisverlust**: Durch den Fallback auf `RawEvent` stellen wir sicher, dass jedes Ereignis aufgezeichnet wird.

### CLI-Kompatibilitaetsbehandlung

Nach der Migration vom CLI-Aufruf zum SDK stellten wir fest, dass einige bestehende CLI-Parameter im SDK nicht mehr galten. Um Abwaertskompatibilitaet zu gewaehrleisten, implementierten wir ein Parameterfiltersystem:

```csharp
// CopilotCliCompatibility.cs - Argumentfilterung
private static readonly Dictionary<string, string> RejectedFlags = new()
{
    ["--headless"] = "Nicht unterstuetztes Startargument",
    ["--model"] = "Wird durch ein SDK-native Feld uebergeben",
    ["--prompt"] = "Wird durch ein SDK-native Feld uebergeben",
    ["--interactive"] = "Interaktion wird vom Provider verwaltet",
};
```

### Runtime-Pooling

Das Erstellen von Copilot SDK-Sessions ist relativ teuer, und zu haeufiges Erstellen und Zerstoeren von Sessions beeintraechtigt die Leistung. Um das zu loesen, implementierten wir ein Session-Pool-Verwaltungssystem:

```csharp
// CopilotProvider.cs - Session-Pool-Verwaltung
await using var lease = await _poolCoordinator.AcquireCopilotRuntimeAsync(
    request,
    async ct => await _gateway.CreateRuntimeAsync(sdkRequest, ct),
    cancellationToken);

if (lease.IsWarmLease)
{
    // Bestehende Session wiederverwenden
    yield return CreateSessionReusedMessage();
}

await foreach (var eventData in lease.Entry.Resource.SendPromptAsync(...))
{
    yield return MapEvent(eventData);
}
```

### Orleans-Integration

HagiCode verwendet Orleans als verteiltes Framework, und wir integrierten das Copilot SDK in Orleans Grains:

```csharp
// GitHubCopilotGrain.cs - Verteilte Ausfuehrung
public async IAsyncEnumerable<GitHubCopilotResponse> ExecuteCommandStreamAsync(
    string command,
    CancellationToken token = default)
{
    var provider = await aiProviderFactory.GetProviderAsync(AIProviderType.GitHubCopilot);

    await foreach (var chunk in provider.SendMessageAsync(request, null, token))
    {
        // In das einheitliche Antwortformat abbilden
        yield return BuildChunkResponse(chunk, startedAt);
    }
}
```

Die Vorteile der Orleans-Integration sind erheblich:

- **Einheitliche KI-Provider-Abstraktion**: Es wird einfach, zwischen verschiedenen KI-Providern zu wechseln.
- **Multi-Tenant-Isolierung**: Copilot-Sessions verschiedener Nutzer bleiben voneinander isoliert.
- **Persistenter Session-Zustand**: Session-Zustaende koennen sogar nach Serverneustarts wiederhergestellt werden.

## Praktischer Leitfaden

### Konfigurationsbeispiel

Hier ist ein vollstaendiges Konfigurationsbeispiel:

```json
{
  "AI": {
    "Providers": {
      "Providers": {
        "GitHubCopilot": {
          "Enabled": true,
          "ExecutablePath": "copilot",
          "Model": "gpt-5",

          "Timeout": 7200,
          "StartupTimeout": 30,
          "UseLoggedInUser": true,
          "NoAskUser": true,
          "Permissions": {
            "AllowAllTools": false,
            "AllowedTools": ["Read", "Bash", "Grep"],
            "DeniedTools": ["Edit"]
          }
        }
      }
    }
  }
}
```

### Nutzungshinweise

In der praktischen Verwendung haben wir mehrere Punkte zusammengefasst, die Beachtung verdienen:

**Start-Timeout-Konfiguration**: Der erste Start von Copilot CLI kann recht lange dauern, daher empfehlen wir, `StartupTimeout` auf mindestens 30 Sekunden zu setzen.

**Berechtigungsverwaltung**: In Produktionsumgebungen vermeiden Sie `AllowAllTools: true`. Verwenden Sie die `AllowedTools`-Allowlist, um zu steuern, welche Tools verfuegbar sind, und die `DeniedTools`-Denylist, um gefaehrliche Operationen zu blockieren.

**Session-Verwaltung**: Anfragen mit derselben `sessionId` verwenden Sessions automatisch wieder. Der Session-Zustand wird durch `ProviderSessionId` persistiert.

**Diagnostische Ausgabe**: Inkompatible CLI-Argumente erzeugen Meldungen vom Typ `diagnostic`. Rohe SDK-Ereignisse werden als `event.raw` beibehalten.

### Bewaehrte Praktiken

Basierend auf unserer praktischen Erfahrung sind hier einige bewaehrte Praktiken:

**1. Tool-Allowlist verwenden**

var request = new AIRequest

**3. Session-Wiederverwendung aktivieren**

```csharp

```

**4. Streaming-Antworten verarbeiten**

```csharp
await foreach (var chunk in provider.StreamAsync(request))
{
    switch (chunk.Type)
    {
        case StreamingChunkType.ThinkingDelta:
            // Reasoning-Prozess verarbeiten
            break;
        case StreamingChunkType.ToolCallDelta:
            // Tool-Aufruf verarbeiten
            break;
        case StreamingChunkType.ContentDelta:
            // Textausgabe verarbeiten
            break;
    }
}
```

**5. Fehlerbehandlung und Wiederholungen**

```csharp
try
{
    await foreach (var chunk in provider.StreamAsync(request))
    {
        // Antwort verarbeiten
    }
}
catch (CopilotSessionException ex)
{
    // Session-Ausnahme verarbeiten
    logger.LogError(ex, "Copilot-Session fehlgeschlagen");
}
```

## Zusammenfassung

Der Upgrade vom CLI-Aufruf zur SDK-Integration brachte dem HagiCode-Projekt erheblichen Mehrwert:

- **Verbesserte Stabilitaet**: Das SDK bietet eine stabilere Schnittstelle, die nicht von CLI-Versionsaenderungen beeinflusst wird.
- **Vollstaendigere Funktionalitaet**: Es erfasst den vollstaendigen Ereignisstrom, einschliesslich Reasoning und Tool-Ausfuehrungsstatus.
- **Hoehere Entwicklungseffizienz**: Die typsichere SDK-Schnittstelle macht die Entwicklung effizienter und reduziert Laufzeitfehler.
- **Bessere Benutzererfahrung**: Echtzeit-Ereignis-Feedback gibt Nutzern ein klaeres Verstaendnis darueber, was die KI tut.

Dieser Upgrade war nicht nur ein Austausch der technischen Implementierung. Es war auch eine architektonische Optimierung der gesamten KI-Faehigkeitsschicht. Durch geschichtetes Design und Abstraktionsschnittstellen erzielten wir bessere Wartbarkeit und Erweiterbarkeit.

Wenn Sie erwägen, GitHub Copilot in Ihr eigenes .NET-Projekt zu integrieren, hoffe ich, dass diese praktischen Lektionen Ihnen helfen, unnötige Umwege zu vermeiden. Das offizielle SDK ist tatsaechlich stabiler und vollstaendiger als der CLI-Aufruf, und es lohnt sich, die Zeit zu investieren, es zu verstehen und einzusetzen.
