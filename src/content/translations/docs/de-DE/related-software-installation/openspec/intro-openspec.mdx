# Einführung in OpenSpec

OpenSpec ist ein spezifikationsgetriebenes Entwicklungstool, das Teams dabei unterstützt, den Softwareentwicklungsprozess über strukturierte Vorschläge und Spezifikationen zu verwalten. Diese Anleitung bietet eine umfassende Einführung in die Kernkonzepte, Workflows und Anwendung von OpenSpec.

## Was ist OpenSpec?

OpenSpec ist ein Kommandozeilenwerkzeug zur Verwaltung von Proposals, Specifications und Changes während des gesamten Softwareentwicklungsprozesses. Es standardisiert den Entwicklungsworkflow und stellt sicher, dass jedes Feature von der Idee bis zur Implementierung eine gründliche Planung und Überprüfung durchläuft.

### Kernprinzipien

OpenSpec basiert auf folgenden Kernprinzipien:

- **Spezifikation zuerst**: Definieren Sie klare Spezifikationen, bevor Sie Code schreiben
- **Strukturierte Zusammenarbeit**: Fördern Sie die Zusammenarbeit im Team durch standardisierte Dokumentformate
- **Nachvollziehbarkeit**: Zeichnen Sie den vollständigen Prozess von Vorschlag bis Implementierung auf
- **Qualitätssicherung**: Stellen Sie durch Spezifikationsvalidierung sicher, dass der Code der Designabsicht entspricht

### Hauptfunktionen

OpenSpec bietet die folgenden Hauptfunktionen:

1. **Proposal-Management**: Erstellen, prüfen und nachverfolgen von Feature-Vorschlägen
2. **Speziﬁkationsdefinition**: Schreiben und Pflegen technischer Spezifikationsdokumente
3. **Änderungsnachverfolgung**: Aufzeichnen aller überprüften Änderungsverläufe
4. **Spezifikationsvalidierung**: Sicherstellen, dass Codeänderungen vordefinierten Spezifikationen entsprechen
5. **Team-Zusammenarbeit**: Unterstützung für kollaborative Entwicklung und Prüfprozesse

## OpenSpec-Workflow

Der typische OpenSpec-Workflow umfasst die folgenden Phasen:

### 1. Erstellen eines Proposals

Wenn ein neues Feature oder eine größere Änderung ansteht, erstellen Sie zunächst ein Proposal:

```bash
openspec proposal create
```

Ein Proposal enthält:
- **Problemstellung**: Das zu lösende Problem oder die Anforderung
- **Lösung**: Übersicht über die vorgeschlagene Lösung
- **Auswirkungsbereich**: Auswirkungen auf bestehende Systeme
- **Alternativen**: Andere in Betracht gezogene Lösungen

### 2. Schreiben einer Spezifikation

Nach Genehmigung des Proposals schreiben Sie eine detaillierte technische Spezifikation:

```bash
openspec spec create
```

Eine Spezifikation enthält:
- **Technisches Design**: Detaillierter technischer Implementierungsplan
- **Schnittstellendefinition**: API-, Datenstruktur- und sonstige Schnittstellendefinitionen
- **Implementierungsdetails**: Wichtige Implementierungslogik und Algorithmen
- **Testplanung**: Wie die Korrektheit der Implementierung verifiziert wird

### 3. Erstellen einer Change

Beim Beginn der Implementierung erstellen Sie eine Change, um den Entwicklungsprozess nachzuverfolgen:

```bash
openspec change create
```

Eine Change enthält:
- **Zugehörige Spezifikation**: Die Spezifikation, die diese Änderung implementiert
- **Implementierungsaufgaben**: Konkrete Entwicklungsaufgaben
- **Fortschrittsverfolgung**: Status der Aufgabenfertigstellung
- **Codeänderungen**: Tatsächliche Code-Modifikationen

### 4. Validieren und Archivieren

Nach Abschluss der Entwicklung validieren Sie, ob die Änderung der Spezifikation entspricht, und archivieren sie anschließend:

```bash
openspec change validate
openspec change archive
```

## Verzeichnisstruktur von OpenSpec

OpenSpec erstellt im Projektstammverzeichnis ein `openspec/` Verzeichnis mit folgender Struktur:

```
openspec/
├── proposals/           # Proposal-Verzeichnis
│   ├── PROPOSAL-001.md
│   └── PROPOSAL-002.md
├── specs/              # Spezifikationsverzeichnis
│   ├── SPEC-001.md
│   └── SPEC-002.md
├── changes/            # Change-Verzeichnis
│   ├── feature-name/
│   │   ├── proposal.md
│   │   ├── design.md
│   │   ├── specs.md
│   │   └── tasks.md
│   └── another-feature/
└── archived/           # Archivierte Änderungen
    └── feature-name/
```

### Verzeichnisbeschreibung

- **proposals/**: Enthält alle Proposal-Dokumente, jeweils eine eigenständige Markdown-Datei pro Proposal
- **specs/**: Enthält technische Spezifikationsdokumente, die Proposals entsprechen
- **changes/**: Enthält laufende Änderungen, jeweils ein Unterverzeichnis pro Change
- **archived/**: Enthält abgeschlossene und archivierte Änderungen

## Kernbefehle

### Befehle für Proposals

```bash
# Neues Proposal erstellen
openspec proposal create

# Alle Proposals auflisten
openspec proposal list

# Proposal-Details anzeigen
openspec proposal view <proposal-id>

# Proposal genehmigen
openspec proposal approve <proposal-id>
```

### Befehle für Spezifikationen

```bash
# Neue Spezifikation erstellen
openspec spec create

# Alle Spezifikationen auflisten
openspec spec list

# Spezifikationsdetails anzeigen
openspec spec view <spec-id>

# Spezifikation aktualisieren
openspec spec update <spec-id>
```

### Befehle für Changes

```bash
# Neue Change erstellen
openspec change create

# Alle Changes auflisten
openspec change list

# Change-Details anzeigen
openspec change view <change-name>

# Aufgabe hinzufügen
openspec change add-task <change-name>

# Change validieren
openspec change validate <change-name>

# Change archivieren
openspec change archive <change-name>
```

### Allgemeine Befehle

```bash
# Version anzeigen
openspec --version

# Hilfe anzeigen
openspec --help

# Hilfe für einen bestimmten Befehl anzeigen
openspec <command> --help
```

## Verwendung von OpenSpec in HagiCode-Projekten

### Projektintegration

In HagiCode-Projekten ist OpenSpec in den Entwicklungsworkflow integriert:

1. **Initialisierungsprüfung**: Der Startassistent prüft, ob OpenSpec installiert ist
2. **Proposal-gesteuerte Entwicklung**: Alle neuen Features müssen den Proposal-Prozess von OpenSpec durchlaufen
3. **Code-Review**: Änderungen müssen den OpenSpec-Spezifikationen entsprechen, um gemerged zu werden
4. **Dokumentations-Synchronisierung**: OpenSpec-Dokumentation bleibt mit der Code-Implementierung synchron

### Versionsanforderungen

HagiCode-Projekte erfordern die Versionen **OpenSpec >=1.0.0 {'<'}2.0.0**.

Installationsbefehl:

```bash
npm install -g @fission-ai/openspec@1
```

### Workflow-Beispiel

Hier ein komplettes Beispiel zur Verwendung von OpenSpec in einem HagiCode-Projekt:

#### Schritt 1: Proposal erstellen

```bash
cd /path/to/hagicode-project
openspec proposal create
```

Geben Sie im interaktiven Prompt ein:
- Proposal-Titel: Benutzer-Authentifizierungs-Feature hinzufügen
- Problem-Beschreibung: Das System benötigt Login- und Berechtigungsverwaltung
- Lösung: Implementierung eines auf JWT basierenden Authentifizierungssystems

#### Schritt 2: Spezifikation schreiben

Nach Genehmigung des Proposals erstellen Sie eine detaillierte Spezifikation:

```bash
openspec spec create
```

Definieren Sie in der Spezifikation:
- Entwurf des Authentifizierungsablaufs
- API-Schnittstellendefinition
- Datenbank-Tabellenstruktur
- Sicherheitsaspekte

#### Schritt 3: Change erstellen

Starten Sie die Implementierung:

```bash
openspec change create
```

Geben Sie den Namen der Change ein: `add-user-authentication`

#### Schritt 4: Aufgaben hinzufügen

```bash
openspec change add-task add-user-authentication
```

Fügen Sie konkrete Aufgaben hinzu:
- Implementierung des Endpunkts zur Benutzerregistrierung
- Implementierung des Login-Endpunkts
- Implementierung von JWT-Middleware
- Schreiben von Unit-Tests

#### Schritt 5: Implementieren und Validieren

Nachdem Sie den Code abgeschlossen haben, validieren Sie die Change:

```bash
openspec change validate add-user-authentication
```

#### Schritt 6: Archivieren

Nach erfolgreicher Validierung archivieren Sie die Change:

```bash
openspec change archive add-user-authentication
```

## Best Practices

### Proposal schreiben

1. **Klare Problemstellung**: Definieren Sie eindeutig das zu lösendes Problem und vermeiden Sie vage Beschreibungen
2. **Alternativen prüfen**: Listen und vergleichen Sie verschiedene Lösungsansätze
3. **Auswirkungen bewerten**: Analysieren Sie Auswirkungen und Risiken für bestehende Systeme
4. **Kurz und bündig**: Proposals sollten knapp gehalten werden; Details gehören in die Spezifikation

### Spezifikation schreiben

1. **Detailliert, aber nicht redundant**: Bieten Sie ausreichend Details, vermeiden Sie Über-Engineering
2. **Diagramme verwenden**: Nutzen Sie Flussdiagramme, Architekturdiagramme usw., um zu erklären
3. **Schnittstellen definieren**: Definieren Sie alle öffentlichen Schnittstellen und Datenstrukturen klar
4. **Randfälle berücksichtigen**: Listen Sie Ausnahmen und Fehlerbehandlung auf

### Change-Management

1. **Kleine Iterationen**: Zerlegen Sie große Features in mehrere kleine Changes
2. **Zeitnahe Aktualisierungen**: Halten Sie Aufgabenstatus und Dokumentation synchron
3. **Code-Review**: Führen Sie Code-Reviews nach Abschluss der Changes durch
4. **Zeitnahes Archivieren**: Archivieren Sie abgeschlossene Changes, um das Arbeitsverzeichnis ordentlich zu halten

### Team-Zusammenarbeit

1. **Regelmäßige Reviews**: Überprüfen Sie Proposals und Spezifikationen regelmäßig, um Team-Konsens sicherzustellen
2. **Dokumentation zuerst**: Schließen Sie die Dokumentation ab, bevor Sie mit dem Codieren beginnen
3. **Kommunikation aufrechterhalten**: Protokollieren Sie Diskussionen und Entscheidungen in Proposals und Spezifikationen
4. **Versionskontrolle**: Nehmen Sie OpenSpec-Dokumente in die Versionskontrolle auf

## Häufige Anwendungsfälle

### Szenario 1: Neue Features hinzufügen

1. Erstellen Sie ein Proposal, das die Anforderungen für das neue Feature beschreibt
2. Besprechen und genehmigen Sie das Proposal im Team
3. Erstellen Sie eine detaillierte technische Spezifikation
4. Erstellen Sie eine Change und implementieren Sie das Feature
5. Validieren und archivieren Sie die Change

### Szenario 2: Bestehenden Code refaktorieren

1. Erstellen Sie ein Proposal, das die Gründe und Ziele der Refaktorisierung beschreibt
2. Definieren Sie den Refaktorisierungsplan und die Schritte in der Spezifikation
3. Erstellen Sie eine Change und refaktorieren Sie schrittweise
4. Stellen Sie die Testabdeckung sicher und validieren Sie die Refaktorisierungsergebnisse
5. Archivieren Sie die Change

### Szenario 3: Fehler beheben

Für einfache Fehler:
- Können direkt behoben werden, ohne den vollständigen OpenSpec-Prozess zu durchlaufen

Für komplexe Fehler:
1. Erstellen Sie ein Proposal zur Analyse der Ursachen des Fehlers
2. Definieren Sie die Behebung in der Spezifikation
3. Erstellen Sie eine Change und implementieren Sie die Behebung
4. Fügen Sie Tests hinzu, um Regressionen zu verhindern
5. Archivieren Sie die Change

### Szenario 4: Technische Schulden verwalten

1. Erstellen Sie ein Proposal, das die Posten technischer Schulden auflistet
2. Definieren Sie einen Verbesserungsplan in der Spezifikation
3. Erstellen Sie Changes, um die Schulden schrittweise abzubauen
4. Verfolgen Sie Fortschritt und Ergebnisse
5. Archivieren Sie abgeschlossene Verbesserungen

## Integration mit anderen Tools

### Git-Integration

OpenSpec lässt sich nahtlos in Git integrieren:

```bash
# OpenSpec-Dokumente sollten in die Versionskontrolle aufgenommen werden
git add openspec/
git commit -m "Add OpenSpec proposal for feature X"
```

### CI/CD-Integration

Validieren Sie OpenSpec-Spezifikationen in CI/CD-Pipelines:

```yaml
# .github/workflows/openspec.yml
name: OpenSpec Validation
on: [pull_request]
jobs:
  validate:
    runs-on: ubuntu-latest
    steps:
      - uses: actions/checkout@v2
      - name: Install OpenSpec
        run: npm install -g @fission-ai/openspec@1
      - name: Validate changes
        run: openspec change validate
```

### IDE-Integration

Empfohlene Editoren mit Markdown-Unterstützung zum Bearbeiten von OpenSpec-Dokumenten:
- VS Code + Markdown-Plugins
- JetBrains IDEs + Markdown-Unterstützung
- Vim/Neovim + Markdown-Plugins

## Fehlerbehebung

### Häufige Probleme

#### Problem: OpenSpec-Befehl nicht gefunden

**Lösung**:
1. Überprüfen Sie, ob OpenSpec installiert ist: `npm list -g @fission-ai/openspec`
2. Prüfen Sie, ob das PATH-Umgebungsvariable das npm globale bin-Verzeichnis enthält
3. Neuinstallation: `npm install -g @fission-ai/openspec@1`

#### Problem: Proposals oder Changes können nicht erstellt werden

**Lösung**:
1. Stellen Sie sicher, dass das aktuelle Verzeichnis das Projektstammverzeichnis ist
2. Prüfen Sie, ob das Verzeichnis `openspec/` existiert; falls nicht, erstellen Sie es
3. Überprüfen Sie Schreibberechtigungen

#### Problem: Validierung schlägt fehl

**Lösung**:
1. Überprüfen Sie, ob die Dokumente im Change-Verzeichnis vollständig sind
2. Stellen Sie sicher, dass alle erforderlichen Aufgaben abgeschlossen sind
3. Detaillierte Fehlermeldungen anzeigen: `openspec change validate <change-name> --verbose`

#### Problem: Archivierung schlägt fehl

**Lösung**:
1. Stellen Sie sicher, dass die Change die Validierung bestanden hat
2. Prüfen Sie, ob das Verzeichnis `openspec/archived/` existiert
3. Stellen Sie sicher, dass keine uncommitted Code-Änderungen vorhanden sind

## Erweiterte Nutzung

### Anpassbare Templates

OpenSpec unterstützt anpassbare Proposal- und Spezifikationstemplates:

```bash
# Erstellen Sie ein Templates-Verzeichnis im Projektstamm
mkdir -p .openspec/templates

# Erstellen Sie ein benutzerdefiniertes Proposal-Template
cat > .openspec/templates/proposal.md << 'EOF'
# Proposal: {{title}}

## Problem Statement
{{problem}}

## Proposed Solution
{{solution}}

## Custom Section
Your custom content here
EOF
```

### Konfigurationsdatei

Erstellen Sie eine `.openspec.json` Konfigurationsdatei im Projektstamm:

```json
{
  "version": "1.0",
  "templates": {
    "proposal": ".openspec/templates/proposal.md",
    "spec": ".openspec/templates/spec.md"
  },
  "validation": {
    "requireTests": true,
    "requireDocs": true
  },
  "hooks": {
    "pre-archive": "npm test"
  }
}
```

### Hooks

OpenSpec unterstützt das Ausführen benutzerdefinierter Skripte vor und nach bestimmten Vorgängen:

```json
{
  "hooks": {
    "pre-create": "scripts/check-permissions.sh",
    "post-archive": "scripts/notify-team.sh"
  }
}
```
