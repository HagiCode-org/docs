## Yleiskatsaus

Monospecs on HagiCoden usean tietovaraston hallintatapa. Se seuraa ja hallitsee alivarastoja `monospecs.yaml`, ja se toimii kaikissa projekteissa, jotka tarvitsevat yhden paikan useiden itsenäisten tietovarastojen koordinoimiseksi.

### Ydinkäsitteet

**Asetustiedosto**

`monospecs.yaml` on koko hallintaratkaisun ydin, joka määrittelee:
- Luettelo ja alitietovarastojen sijainti
- Anna URL-osoite jokaiselle arkistolle
- Näyttönimi ja kuvake
- Integrointikäyttäytyminen OpenSpecin kanssa

**Erot tavallisiin tietovarastoihin**

| Ominaisuus | Tavallinen arkisto | Monospecin hallinnoima projekti |
|----------|-------------------|---------------------------|
| **OpenSpec-sijainti** | Juurihakemistot `openspec/` | Päävaraston juurihakemistot `openspec/` |
| **Muuta soveltamisalaa** | Rajoitettu nykyiseen arkistoon | Voi sisältää useita alivarastoja |
| **Alivaraston puhtaus** | Tekniset tiedot sekoitettuna koodiin | Päävarastossa määritetyt tiedot, alivarastot ovat puhtaampia |
| **Arkistoi automaattinen vahvistus** | Vaatii manuaalisia teknisiä sitoumuksia | `commit_when_archive: true` sitoo tiedot automaattisesti päävarastoon |

### Miksi käyttää Monospecia?

- **Yhdistetty hallinta**: Hallitse kaikkia alivarastoja yhdessä määritystiedostossa
- **Automaatiotuki**: Tarjoaa arkiston metatiedot komentosarjoille ja työkaluille
- **AI-ystävällisyys**: Strukturoitu kokoonpano auttaa AI-agenttia ymmärtämään arkiston suhteita
- **OpenSpec-integraatio**: Saumaton integrointi muutoksenhallinnan työnkulkuun
- **Puheammat alivarastot**: Tekniset tiedot säilytetään päävarastossa, alivarastot sisältävät vain todellista koodia

### Sovellettavat skenaariot

Käytä Monospecia ensin seuraavissa tilanteissa:

**1. Usean tietovaraston yhteistyökehitys**
Kun projekti on jaettu useisiin itsenäisiin tietovarastoihin:
- Käyttöliittymäsovellus, taustapalvelu ja työpöytäsovellus on kehitetty erikseen
- Versiot ja riippuvuudet on koordinoitava arkistojen välillä
- Dokumentaatio ja apuprojektit erillään pääkoodista

**2. Apuprojektinhallinta**
Käytä Monospecsia hallitsemaan:
- Dokumentaatioprojekti
- Virallinen verkkosivusto
- Rakenna työkaluja ja skriptejä
- Jokainen arkisto ylläpitää omaa versionhallintaansa

**3. Monivaraisten ominaisuuksien kehitys**
Ominaisuus, joka vaatii muutoksia useisiin tietovarastoihin:
- On seurattava muutoksia missä arkistoissa
- Sitouta tiedot automaattisesti oikeaan kohdetietovarastoon
- Koordinoi versiopäivitykset ja riippuvuudet

## Pika-aloitus

### Alustus

Suorita monospecin alustus loppuun seuraavasti:

#### 1. Luo tyhjä git-arkisto

Luo tyhjä `git` kansio päävarastossasi (huomaa: tämä on `git`, ei `repos/` - tämä on monospec-sopimus):

```bash
mkdir git
cd git
git init
cd ..
```

#### 2. Määritä monospecs-tiedosto

Luo `monospecs.yaml` asetustiedosto päävaraston juurihakemistossa:

```yaml
version: "1.0"
commit_when_archive: true

repositories:
  - path: "repos/frontend"
    url: "https://github.com/your-org/frontend.git"
    displayName: "Frontend App"
    icon: "🌐"

  - path: "repos/backend"
    url: "https://github.com/your-org/backend.git"
    displayName: "Backend Service"
    icon: "⚙️"
```

**Konfigurointikohteet**:
| Kokoonpanokohde | Kirjoita | Pakollinen | Kuvaus |
|----------------|------|----------|-------------|
| `version` | merkkijono | Kyllä | Asetustiedoston versio |
| `commit_when_archive` | boolean | Ei | Vahvistetaanko tiedot automaattisesti arkistoinnin aikana (älä sitoa alirepo-koodia) |
| `repositories` | joukko | Kyllä | Arkiston määritysluettelo |
| `repositories[].path` | merkkijono | Kyllä | Arkiston polku (suhteessa projektin juureen) |
| `repositories[].url` | merkkijono | Kyllä | Git arkiston URL-osoite |
| `repositories[].displayName` | merkkijono | Ei | Arkiston näyttönimi |
| `repositories[].icon` | merkkijono | Ei | Arkistokuvake (emoji) |

### Lisää alivarastot päävarastoon

Lisää kaikki alivarastot tiedostoon `repos/` hakemistosta. Tämä hakemisto on **suljettava pois päävaraston Git-version hallinnasta**, jotta vältetään alivaraston sisällön seuranta:

```plaintext
# Exclude all sub-repositories
repos/
```

#### 3. Lisää päätietovarasto HagiCodeen

Jos käytät HagiCode-työpöytäsovellusta, avaa sovellus ja lisää päävarastosi projektiksi. Napsauta "Lisää projekti" tai "+" -painiketta aloitusnäytössä, valitse pääarkistokansiosi ja näet kaikki alivarastot ja niiden tilatiedot.

#### 4. Kloonaa alivarastot

Käytä `monospecs.yaml` kokoonpano kloonataksesi kaikki määritetyt alivarastot tiedostoon `repos/` hakemisto:

Tämä kloonaa kaikki alivarastot määritystiedostossa määritettyjen URL-osoitteiden perusteella.

## Määritystiedostorakenne

### Päätason asetukset

```yaml
version: "1.0"
commit_when_archive: true

repositories:
  # ... repository configurations
```

### Arkiston tason asetukset

Jokaisessa alivarastossa voi olla:
- OpenSpec vaihda kansio (`openspec/`)
- Riippumaton `.git/` hakemistosta
- Valinnainen `AGENTS.md` AI-agentin määritystä varten

## Tietovaraston hallintatoiminnot

### Lisää uusi arkisto

Kun lisäät uuden alivaraston `monospecs.yaml` `repositories` joukko:

```yaml
repositories:
  - path: "repos/new-service"
    url: "https://github.com/HagiCode-org/new-service.git"
    displayName: "New Service"
    icon: "🆕"
```

### Päivitä arkiston asetukset

Kun arkiston URL-osoitteet tai metatiedot muuttuvat:
1. Muokkaa `monospecs.yaml` päivittääksesi vastaavat merkinnät
2. Varmista, että YAML-syntaksi on oikea
3. Jos synkronointi muuttuu, päivitä paikallisen arkiston asetukset manuaalisesti

## AI-agentin määritystuki

### AGENTS.md

The `AGENTS.md` tiedosto tarjoaa arkistokohtaisia ohjeita tekoälyagentille:

**Tekniset tiedot**
- Kehyksen ja rakentamisen työkalut
- Koodikäytännöt ja nimeämiskäytännöt
- Versionhallinta ja käyttöönoton määritykset

**Projektikohtaiset käyttäytymiset**
- Kokoonpanolaajennukset ja erityisvaatimukset
- OpenSpec-integraatiomallit
- Arkiston välinen koordinointitarpeet

### AI edut

Käyttämällä `AGENTS.md` mahdollistaa AI-agentin:
- Ymmärrä arkiston suhteet ja riippuvuudet
- Luo koodia ja uudelleentekijä useissa arkistoissa
- Säilytä yhdenmukaisuus versionhallinnassa ja kokoonpanoissa

## OpenSpec-integraatio

### Kuinka Monospecs toimii OpenSpecin kanssa

Voit käyttää HagiCoden OpenSpec-työnkulkua riippumatta siitä, hallitsetpa monospecs-hallittua projektia tai tavallista arkistoa:

**Tavallinen arkisto:**
- OpenSpec rootissa `openspec/` hakemistosta
- Muutokset vaikuttavat vain nykyiseen tietovarastoon

**Monospecsin hallinnoima projekti:**
- OpenSpec lukee `monospecs.yaml` kokoonpano
- Muutoksia seurataan alitietovaraston mukaan
- Tekniset tiedot sitoutuvat automaattisesti oikeisiin kohteisiin

### Arkiston käyttäytyminen

Milloin `commit_when_archive: true`:
- Tekniset tiedot sitoutuvat automaattisesti päävarastoon
- Alivarastojen koodia ei ole sitoutunut (pysyy erillisenä)
- Yksinkertaistaa teknisten tietojen hallintaa – alivarastoihin ei tarvita manuaalisia teknisiä sitoumuksia

### Manuaaliset alivaraston sitoumukset

Alivarastot ylläpitävät itsenäistä versionhallintaa:
- Sitovat todelliset koodimuutokset omiin arkistoihinsa
- Tekniset tiedot sitovat erikseen (päävarastosta arkistoinnin yhteydessä)

## Parhaat käytännöt

### Tietovaraston nimeämiskäytännöt

- Käytä kebab-koteloa (pieniä kirjaimia ja yhdysmerkkejä) arkiston poluille
- Käytä yksinkertaisia, kuvaavia näyttönimiä kiinaksi
- Käytä asianmukaisia hymiöitä kuvakkeina

### Näyttönimiä koskevat ohjeet

- Pidä se ytimekkäänä ja kuvailevana
- Käytä johdonmukaista nimeämistä samankaltaisissa arkistoissa
- Valitse kuvakkeet, jotka edustavat arkiston tarkoitusta

### Kokoonpanon ylläpito

**Synkronointimääritykset todellisen tilan kanssa**
- Tarkista säännöllisesti `monospecs.yaml` kuvastaa todellista arkiston rakennetta
- Päivitä tai poista merkintöjä, kun arkistoja lisätään tai poistetaan
- Testaa muutokset varmistaaksesi kokoonpanon kelvollisuuden

** Konfiguroinnin versionhallinta**
- Seuraa asetustiedoston muutoksia gitillä
- Dokumentoi suurten kokoonpanomuutosten syyt
- Merkitse määritystiedostoversiot helppoon palautusta varten

## Vianetsintä

### Asetustiedostoa ei löydy

Jos `monospecs.yaml` tiedostoa ei löydy projektin juuresta:
1. Tarkista, onko tiedosto oikeassa paikassa
2. Varmista, että olet oikeassa työhakemistossa
3. Tarkista kirjoitusvirheet asetustiedoston nimessä

### YAML-syntaksivirheet

Yleiset YAML-syntaksiongelmat:
- Väärä sisennys (käytä välilyöntejä, älä sarkaimia)
- Puuttuvat tai ylimääräiset lainausmerkit merkkijonojen ympäriltä
- Virheelliset tietotyypit (merkkijonot tarvitsevat lainausmerkkejä, numerot eivät)
- Pakolliset kentät puuttuvat (esim `path` tai `url`)

### Tietovarastoa ei havaittu

Jos äskettäin lisätty arkisto ei näy:
1. Tarkista `path` on oikeassa `monospecs.yaml`
2. Varmista, että arkisto on kloonattu onnistuneesti
3. Tarkista, onko arkisto olemassa odotetussa paikassa

### Kloonien epäonnistumiset

Jos arkiston kloonaus epäonnistuu:
1. Tarkista verkkoyhteys
2. Varmista, että Git-arkiston URL-osoite on oikea
3. Tarkista todennusongelmat
4. Tarkista Docker-säilön tila
5. Tarkista levytilan saatavuus
