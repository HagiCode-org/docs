# Johdatus OpenSpeciin

OpenSpec on spesifikaatioihin perustuva kehitystyökalu, joka auttaa tiimejä hallitsemaan ohjelmistokehitysprosessia rakenteellisten ehdotusten ja spesifikaatioiden avulla. Tämä artikkeli tarjoaa kattavan johdannon OpenSpecin keskeisiin käsitteisiin, työnkulkuihin ja käyttötapoihin.

## Mikä on OpenSpec?

OpenSpec on komentorivityökalu, jota käytetään ehdotusten (Proposals), spesifikaatioiden (Specifications) ja muutosten (Changes) hallintaan ohjelmistokehitysprosessissa. Se standardisoi kehitysprosessin ja varmistaa, että jokainen ominaisuus käy läpi huolellisen suunnittelun ja arvion ideasta toteutukseen.

### Perusperiaatteet

OpenSpec perustuu seuraaviin perusperiaatteisiin:

- **Spesifikaatiot ensin**: Määrittele selkeät spesifikaatiot ennen koodin kirjoittamista
- **Rakenteellinen yhteistyö**: Edistä tiimiyhteistyötä standardoitujen dokumenttimuotojen avulla
- **Jäljitys**: Kirjaa koko prosessi ehdotuksesta toteutukseen
- **Laadunvarmistus**: Varmista, että koodi vastaa suunnittelun tarkoitusta spesifikaatioiden validoinnin avulla

### Pääominaisuudet

OpenSpec tarjoaa seuraavat pääominaisuudet:

1. **Ehdotusten hallinta**: Luo, tarkista ja seuraa ominaisuusehdotuksia
2. **Spesifikaatioiden määrittely**: Kirjoita ja ylläpidä teknisiä spesifikaatiodokumentteja
3. **Muutosten seuranta**: Kirjaa kaikki tarkastetut muutokset historian muodossa
4. **Spesifikaatioiden validointi**: Varmista, että koodimuutokset vastaavat ennalta määriteltyjä spesifikaatioita
5. **Tiimiyhteistyö**: Tukea monen osallistujan kehitys- ja tarkastusprosesseja

## OpenSpecin työnkulku

Tyypillinen OpenSpec-työnkulku sisältää seuraavat vaiheet:

### 1. Luo ehdotus (Proposal)

Kun on uusi ominaisuus tai vaatimus merkittävään muutokseen, luo ensin ehdotus:

```bash
openspec proposal create
```

Ehdotus sisältää:
- **Ongelman kuvaus**: Ongelma tai vaatimus, joka pitää ratkaista
- **Ratkaisu**: Yleiskatsaus ehdotetusta ratkaisusta
- **Vaikutuksen laajuus**: Vaikutus olemassa olevaan järjestelmään
- **Vaihtoehtoiset ratkaisut**: Muut harkitut ratkaisut

### 2. Kirjoita spesifikaatio (Specification)

Ehdotuksen hyväksymisen jälkeen kirjoita yksityiskohtainen tekninen spesifikaatio:

```bash
openspec spec create
```

Spesifikaatio sisältää:
- **Tekninen suunnittelu**: Yksityiskohtainen suunnitelma teknisestä toteutuksesta
- **Rajapintojen määrittely**: Rajapintasuunnittelu, kuten API:t, tietorakenteet jne.
- **Toteutuksen yksityiskohdat**: Keskeinen toteutuslogiikka ja algoritmit
- **Testausstrategia**: Kuinka toteutuksen oikeellisuus varmistetaan

### 3. Luo muutos (Change)

Kun aloitat toteutuksen, luo muutos kehitysprosessin seuraamista varten:

```bash
openspec change create
```

Muutos sisältää:
- **Liittyvä spesifikaatio**: Spesifikaatio, joka toteutetaan tällä muutoksella
- **Toteutustehtävät**: Luettelo konkreettisista kehitystehtävistä
- **Edistymisen seuranta**: Tehtävien valmistumisen tila
- **Koodimuutokset**: Todelliset koodinmuutokset

### 4. Validointi ja arkistointi

Kun kehitys on valmis, validoi, että muutos vastaa spesifikaatiota, ja arkistoi se:

```bash
openspec change validate
openspec change archive
```

## OpenSpecin hakemistorakenne

OpenSpec luo projektin juurihakemistoon kansion `openspec/`, jonka rakenne on seuraava:

```
openspec/
├── proposals/           # Ehdotuskansio
│   ├── PROPOSAL-001.md
│   └── PROPOSAL-002.md
├── specs/              # Spesifikaatiokansio
│   ├── SPEC-001.md
│   └── SPEC-002.md
├── changes/            # Muutoskansio
│   ├── feature-name/
│   │   ├── proposal.md
│   │   ├── design.md
│   │   ├── specs.md
│   │   └── tasks.md
│   └── another-feature/
└── archived/           # Arkistoidut muutokset
    └── feature-name/
```

### Hakemistojen kuvaus

- **proposals/**: Tallentaa kaikki ehdotusasiakirjat, jokainen ehdotus omassa Markdown-tiedostossaan
- **specs/**: Tallentaa tekniset spesifikaatiodokumentit, jotka vastaavat ehdotuksia
- **changes/**: Tallentaa käynnissä olevat muutokset, jokainen muutos omassa alikansiossaan
- **archived/**: Tallentaa valmistuneet ja arkistoidut muutokset

## Keskeiset komennot

### Ehdotuksiin liittyvät komennot

```bash
# Luo uusi ehdotus
openspec proposal create

# Listaa kaikki ehdotukset
openspec proposal list

# Näytä ehdotuksen tiedot
openspec proposal view <proposal-id>

# Hyväksy ehdotus
openspec proposal approve <proposal-id>
```

### Spesifikaatioihin liittyvät komennot

```bash
# Luo uusi spesifikaatio
openspec spec create

# Listaa kaikki spesifikaatiot
openspec spec list

# Näytä spesifikaation tiedot
openspec spec view <spec-id>

# Päivitä spesifikaatio
openspec spec update <spec-id>
```

### Muutoksiin liittyvät komennot

```bash
# Luo uusi muutos
openspec change create

# Listaa kaikki muutokset
openspec change list

# Näytä muutoksen tiedot
openspec change view <change-name>

# Lisää tehtävä
openspec change add-task <change-name>

# Validointi muutokselle
openspec change validate <change-name>

# Arkistoi muutos
openspec change archive <change-name>
```

### Yleiset komennot

```bash
# Näytä versio
openspec --version

# Näytä ohje
openspec --help

# Näytä ohje tietylle komennolle
openspec <command> --help
```

## OpenSpecin käyttö HagiCode-projektissa

### Projektin integraatio

HagiCode-projektissa OpenSpec on integroitu kehitysprosessiin:

1. **Alustus tarkistus**: Aloitusopas tarkistaa, onko OpenSpec asennettu
2. **Ehdotusvetoiseen kehitykseen**: Kaikkien uusien ominaisuuksien on läpäistävä OpenSpec-ehdotusprosessi
3. **Koodikatselmointi**: Muutosten on noudatettava OpenSpec-spesifikaatioita, jotta ne voidaan yhdistää
4. **Dokumentaatiosynkronointi**: OpenSpec-dokumentaatio pidetään synkronoituna koodin toteutuksen kanssa

### Versiovaatimukset

HagiCode-projekti vaatii OpenSpec-version **OpenSpec >=1.0.0 {'<'}2.0.0**.

Asennuskomento:

```bash
npm install -g @fission-ai/openspec@1
```

### Työnkulun esimerkin käyttöönotto

Alla on täydellinen esimerkki OpenSpecin käytöstä HagiCode-projektissa:

#### Vaihe 1: Luo ehdotus

```bash
cd /path/to/hagicode-project
openspec proposal create
```

Syötä interaktiiviseen kehotteeseen:
- Ehdotuksen otsikko: Lisää käyttäjäautentikointiominaisuus
- Ongelman kuvaus: Järjestelmä tarvitsee käyttäjien kirjautumisen ja oikeuksien hallinnan
- Ratkaisu: Toteuta JWT-pohjainen todennusjärjestelmä

#### Vaihe 2: Kirjoita spesifikaatio

Hyväksytyn ehdotuksen jälkeen luo yksityiskohtainen spesifikaatio:

```bash
openspec spec create
```

Määrittele spesifikaatiossa:
- Todennusprosessin suunnittelu
- API-rajapintojen määrittely
- Tietokantataulujen rakenne
- Turvallisuusnäkökohdat

#### Vaihe 3: Luo muutos

Aloita toteutus:

```bash
openspec change create
```

Syötä muutoksen nimi: `add-user-authentication`

#### Vaihe 4: Lisää tehtäviä

```bash
openspec change add-task add-user-authentication
```

Lisää konkreettisia tehtäviä:
- Toteuta käyttäjärekisteröintirajapinta
- Toteuta kirjautumisrajapinta
- Toteuta JWT-middleware
- Kirjoita yksikkötestit

#### Vaihe 5: Toteutus ja validointi

Kun koodi on valmis, validoi muutos:

```bash
openspec change validate add-user-authentication
```

#### Vaihe 6: Arkistointi

Onnistuneen validoinnin jälkeen arkistoi muutos:

```bash
openspec change archive add-user-authentication
```

## Parhaat käytännöt

### Ehdotusten kirjoittaminen

1. **Selkeä ongelman kuvaus**: Määrittele selkeästi ratkaistava ongelma, vältä epämääräisiä kuvauksia
2. **Harkitse vaihtoehtoisia ratkaisuja**: Luettele ja vertaa eri ratkaisuja
3. **Arvioi vaikutus**: Analysoi vaikutus ja riskit olemassa olevalle järjestelmälle
4. **Pidä tiiviinä**: Ehdotusten tulisi olla ytimekkäitä ja selkeitä; jätä yksityiskohdat spesifikaatioon

### Spesifikaatioiden kirjoittaminen

1. **Yksityiskohtainen mutta ei redundantti**: Tarjoa riittävästi yksityiskohtia, mutta vältä liiallista suunnittelua
2. **Käytä kaavioita**: Käytä kaavioita, arkkitehtuurikaavioita jne. selityksen tukena
3. **Määrittele rajapinnat**: Määrittele selkeästi kaikki julkiset rajapinnat ja tietorakenteet
4. **Harkitse reunatapauksia**: Luettele poikkeustilanteet ja virheenkäsittely

### Muutosten hallinta

1. **Pienet iteroinnit**: Jaa suuret ominaisuudet useiksi pieniksi muutoksiksi
2. **Päivitä ajoissa**: Pidä tehtävien tila ja dokumentaatio synkronoituna
3. **Koodikatselmointi**: Suorita koodikatselmointi muutoksen valmistuttua
4. **Arkistoi ajoissa**: Arkistoi valmiit muutokset ajoissa ja pidä työhakemisto siistinä

### Tiimityöskentely

1. **Säännölliset arvostelut**: Tarkista säännöllisesti ehdotukset ja spesifikaatiot varmistaaksesi tiimin yksimielisyyden
2. **Dokumentaatio ensin**: Viimeistele dokumentaatio ennen koodaamisen aloittamista
3. **Pidä yhteydenpito**: Kirjaa keskustelut ja päätökset ehdotuksiin ja spesifikaatioihin
4. **Versiohallinta**: Sisällytä OpenSpec-dokumentaatio versionhallintaan
