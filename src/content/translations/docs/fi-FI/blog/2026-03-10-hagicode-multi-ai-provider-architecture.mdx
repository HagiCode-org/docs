## Käytännöllinen Multi-AI Provider -arkkitehtuuri HagiCode-alustalla

> Tämä artikkeli jakaa teknisen lähestymistavan, jota käytimme Orleans Grain -arkkitehtuurin alla integroidaksemme kaksi tekoälytyökalua, iflow ja OpenCode, yhtenäisen `IAIProvider` käyttöliittymän ja vertaa yksityiskohtaisesti toteutuseroja WebSocket- ja HTTP-viestinnän välillä.

:::note[Time Context]
Tämä artikkeli tallentaa palveluntarjoajan arkkitehtuurikäytännön julkaisuhetkellä. Katso nykyinen aktiivisesti tuettu laajuus kohdasta [AI Agent CLI -asennus](/fi-FI/related-software-installation/ai-agent-cli/) ja [Asennusopas](/fi-FI/installation); `IFlowCli` on nyt säilytetty vain historiallisena yhteensopivuusarvona, eikä se ole osa nykyistä aktiivista tukialuetta.
:::

## Tausta

Siinä ei ole mitään erityisen mystistä. HagiCodea rakentaessamme törmäsimme hyvin käytännölliseen ongelmaan: käyttäjät halusivat työskennellä erilaisten tekoälytyökalujen kanssa. Se tuskin on yllättävää, sillä jokaisella on omat tapansa. Jotkut pitävät Claude Codesta, jotkut rakastavat GitHub Copilotia, ja jotkut tiimit käyttävät itse kehittämiään työkaluja.

Alkuperäinen ratkaisumme oli yksinkertainen ja suora: kirjoita oma integraatiokoodi jokaiselle tekoälytyökalulle. Mutta haitat paljastuivat nopeasti. Koodikanta täyttyi `if-else` oksat, jokainen muutos vaati testausta useissa paikoissa, ja jokainen uusi työkalu merkitsi uuden kasan logiikkaa kirjoittamista tyhjästä.

Myöhemmin tajusin, että olisi parempi luoda yhtenäinen `IAIProvider` käyttöliittymä ja abstrakti kaikkien tekoälyn tarjoajien yhteiset ominaisuudet. Näin ylemmät kerrokset voivat kutsua sitä samalla tavalla riippumatta siitä, mitä työkalua käytetään alla.

Äskettäin projekti piti integroida kaksi uutta työkalua: iflow ja OpenCode. Molemmat tukevat ACP-protokollaa, mutta niiden viestintätyylit ovat erilaisia. iflow käyttää WebSocketia, kun taas OpenCode käyttää HTTP API:ta. Siitä tuli hyödyllinen arkkitehtoninen testi: sovita kaksi erilaista liikennemuotoa yhden yhtenäisen käyttöliittymän taakse.

## Tietoja HagiCodesta

Tässä artikkelissa jaettu lähestymistapa perustuu käytännön kokemukseemme [HagiCode](https://hagicode.com) hanke. HagiCode on AI-avusteinen kehitysalusta, joka on rakennettu Orleans Grain -arkkitehtuuriin. Se integroituu eri AI-palveluntarjoajien kanssa yhtenäisen järjestelmän kautta `IAIProvider` käyttöliittymä, jonka avulla käyttäjät voivat valita joustavasti haluamansa tekoälytyökalut.

## Arkkitehtuurisuunnittelu

### Unified Interface Abstraktio

Ensin määritimme `IAIProvider` käyttöliittymä ja abstrakti ominaisuudet, jotka jokaisen tekoälyn tarjoajan on otettava käyttöön:

```csharp
public interface IAIProvider
{
    string Name { get; }
    bool SupportsStreaming { get; }
    ProviderCapabilities Capabilities { get; }

    Task<AIResponse> ExecuteAsync(AIRequest request, CancellationToken cancellationToken = default);
    IAsyncEnumerable<AIStreamingChunk> StreamAsync(AIRequest request, CancellationToken cancellationToken = default);
    Task<ProviderTestResult> PingAsync(CancellationToken cancellationToken = default);
    IAsyncEnumerable<AIStreamingChunk> SendMessageAsync(AIRequest request, string? embeddedCommandPrompt = null, CancellationToken cancellationToken = default);
}
```

Tämä käyttöliittymä sisältää useita keskeisiä menetelmiä:
- `ExecuteAsync`: suorita kertaluonteinen AI-pyyntö
- `StreamAsync`: saat suoratoistovastauksia reaaliaikaista näyttöä varten
- `PingAsync`: suorita terveystarkastus varmistaaksesi, onko palveluntarjoaja käytettävissä
- `SendMessageAsync`: lähetä viesti, joka tukee upotettuja komentoja

### IFlowCliProvider: WebSocket-pohjainen toteutus

iflow käyttää WebSocketia ACP-viestintään. Kokonaisarkkitehtuuri näyttää tältä:

```
IFlowCliProvider → ACPSessionManager → WebSocketAcpTransport → iflow CLI
                ↓
         Dynamic port allocation + process management
```

Ydinvirtaus on myös melko suoraviivainen:
1. `ACPSessionManager` luo ja hallitsee ACP-istuntoja.
2. `WebSocketAcpTransport` hoitaa WebSocket-viestinnän.
3. Portti varataan dynaamisesti ja iflow-prosessi käynnistetään `iflow --experimental-acp --port`.
4. `IAIRequestToAcpMapper` ja `IAcpToAIResponseMapper` muuntaa pyynnöt ja vastaukset.

Tässä on ydinkoodi:

```csharp
private async IAsyncEnumerable<AIStreamingChunk> StreamCoreAsync(
    AIRequest request,
    string? embeddedCommandPrompt,
    [EnumeratorCancellation] CancellationToken cancellationToken)
{
    // Resolve working directory
    var resolvedWorkingDirectory = ResolveWorkingDirectory(request);
    var effectiveRequest = ApplyEmbeddedCommandPrompt(request, embeddedCommandPrompt);

    // Create ACP session
    await using var session = await _sessionManager.CreateSessionAsync(
        Name,
        resolvedWorkingDirectory,
        cancellationToken,
        request.SessionId);

    // Send prompt
    var prompt = _requestMapper.ToPromptString(effectiveRequest);
    var promptResponse = await session.SendPromptAsync(prompt, cancellationToken);

    // Receive streaming response
    await foreach (var notification in session.ReceiveUpdatesAsync(cancellationToken))
    {
        if (_responseMapper.TryConvertToStreamingChunk(notification, out var chunk))
        {
            if (chunk.Type == StreamingChunkType.Metadata && chunk.IsComplete)
            {
                yield return chunk;
                yield break;
            }
            yield return chunk;
        }
    }
}
```

Tässä on muutamia huomionarvoisia suunnittelukohtia:
- Käytä `await using` varmistaaksesi, että istunto vapautetaan oikein ja välttääksesi resurssivuodot.
- Palauta suoratoistovastaukset `IAsyncEnumerable`, joka luonnollisesti tukee async-streameja.
- Käytä `Metadata` palasia määrittämään valmistuminen ja varmistamaan, että täydellinen vastaus on vastaanotettu.

### OpenCodeCliProvider: HTTP API-pohjainen toteutus

OpenCode tarjoaa palvelunsa HTTP API:n kautta, joten arkkitehtuuri on hieman erilainen:

```plaintext
OpenCodeCliProvider → OpenCodeRuntimeManager → OpenCodeClient → OpenCode HTTP API
                      ↓
                OpenCodeProcessManager → opencode process management
```

Huomattava OpenCoden ominaisuus on, että se käyttää SQLite-tietokantaa istuntojen sidosten säilyttämiseen. Tämä mahdollistaa istunnon palautuksen ja nopean vastauksen palautuksen:

```csharp
private async Task<OpenCodePromptExecutionResult> ExecutePromptAsync(
    AIRequest request,
    string? embeddedCommandPrompt,
    CancellationToken cancellationToken)
{
    var prompt = BuildPrompt(request, embeddedCommandPrompt);
    var resolvedWorkingDirectory = ResolveWorkingDirectory(request.WorkingDirectory);
    var client = await _runtimeManager.GetClientAsync(resolvedWorkingDirectory, cancellationToken);
    var bindingSessionId = request.SessionId;
    var boundSession = TryGetBinding(bindingSessionId, resolvedWorkingDirectory);

    // Try to use the already bound session
    if (boundSession is not null)
    {
        try
        {
            return await PromptSessionAsync(
                client,
                boundSession,
                BuildPromptRequest(request, prompt, CreatePromptMessageId()),
                request.Model ?? _settings.Model,
                cancellationToken);
        }
        catch (OpenCodeApiException ex) when (IsStaleBinding(ex))
        {
            // The session has expired, remove the binding
            RemoveBinding(bindingSessionId);
        }
    }

    // Create a new session
    var session = await client.Session.CreateAsync(new OpenCodeSessionCreateRequest
    {
        Title = BuildSessionTitle(request)
    }, cancellationToken);

    BindSession(bindingSessionId, session.Id, resolvedWorkingDirectory);
    return await PromptSessionAsync(client, session.Id, ...);
}
```

Tällä toteutuksella on useita mielenkiintoisia kohokohtia:
- Istunnon sitomismekanismi: sama `SessionId` käyttää uudelleen samaa OpenCode-istuntoa välttäen toistuvan istunnon luomisen.
- Vanhenemisen käsittely: kun istunnon todetaan vanhentuneen, sidos siivotaan automaattisesti.
- Tietokannan pysyvyys: sidokset tallennetaan SQLiteen ja pysyvät voimassa uudelleenkäynnistyksen jälkeen.

### Kahden lähestymistavan vertailu

| Aspekti | IFlowCliProvider | OpenCodeCliProvider |
|------|-----------------|---------------------|
| Viestintä | WebSocket (ACP) | HTTP API |
| Prosessien hallinta | ACPSessionManager | OpenCodeProcessManager |
| Satamien jako | Dynaaminen portti | Ei porttia (käyttää HTTP:tä) |
| Istunnon hallinta | ACPS-istunto | OpenCodeSession |
| Pysyvyys | Muistissa oleva välimuisti | SQLite-tietokanta |
| Käynnistyskomento | `iflow --experimental-acp --port` | `opencode` |
| Latenssi | Alempi (pitkäikäinen yhteys) | Suhteellisen korkeampi (HTTP-pyynnöt) |

Valitsemasi lähestymistapa riippuu pääasiassa tarpeistasi. WebSocket on parempi skenaarioissa, joissa on korkeat reaaliaikaiset vaatimukset, kun taas HTTP API on yksinkertaisempi ja helpompi korjata.

## Käytännön opas

### Määritä palveluntarjoajat

Ota ensin käyttöön kaksi palveluntarjoajaa asetustiedostossa:

```yaml
AI:
  Providers:
    IFlowCli:
      Type: "IFlowCli"
      Enabled: true
      ExecutablePath: "iflow"
      Model: null
      WorkingDirectory: null
    OpenCodeCli:
      Type: "OpenCodeCli"
      Enabled: true
      ExecutablePath: "opencode"
      Model: "anthropic/claude-sonnet-4"
      WorkingDirectory: null

OpenCode:
  Enabled: true
  BaseUrl: "http://localhost:38376"
  ExecutablePath: "opencode"
  StartupTimeoutSeconds: 30
  RequestTimeoutSeconds: 120
```

### Käytä IFlowCliProvideria

```csharp
// Get provider through the factory
var provider = await _providerFactory.GetProviderAsync(AIProviderType.IFlowCli);

// Execute an AI request
var request = new AIRequest
{
    Prompt = "请帮我重构这个函数",
    WorkingDirectory = "/path/to/project",
    Model = "claude-sonnet-4"
};

// Get the complete response
var response = await provider.ExecuteAsync(request, cancellationToken);
Console.WriteLine(response.Content);

// Or use streaming responses
await foreach (var chunk in provider.StreamAsync(request, cancellationToken))
{
    if (chunk.Type == StreamingChunkType.ContentDelta)
    {
        Console.Write(chunk.Content);
    }
}
```

### Käytä OpenCodeCliProvideria

```csharp
// Get provider through the factory
var provider = await _providerFactory.GetProviderAsync(AIProviderType.OpenCodeCli);

var request = new AIRequest
{
    Prompt = "请帮我分析这个错误",
    WorkingDirectory = "/path/to/project",
    Model = "anthropic/claude-sonnet-4"
};

var response = await provider.ExecuteAsync(request, cancellationToken);
Console.WriteLine(response.Content);
```

### Terveystarkastukset

Ennen käynnistystä tai ennen käyttöä voit tarkistaa, onko palveluntarjoaja saatavilla:

```csharp
var iflowResult = await iflowProvider.PingAsync(cancellationToken);
if (!iflowResult.Success)
{
    Console.WriteLine($"IFlow is unavailable: {iflowResult.ErrorMessage}");
    return;
}

var openCodeResult = await openCodeProvider.PingAsync(cancellationToken);
if (!openCodeResult.Success)
{
    Console.WriteLine($"OpenCode is unavailable: {openCodeResult.ErrorMessage}");
    return;
}
```

### Sulautettu komentotuki

Molemmat palveluntarjoajat tukevat upotettuja komentoja, kuten `/file:xxx`:

```csharp
var request = new AIRequest
{
    Prompt = "分析这个文件的问题",
    SystemMessage = "你是一个代码分析专家"
};

await foreach (var chunk in provider.SendMessageAsync(
    request,
    embeddedCommandPrompt: "/file:src/main.cs",
    cancellationToken))
{
    Console.Write(chunk.Content);
}
```

## Huomautuksia ja parhaita käytäntöjä

### Resurssienhallinta

IFlow käyttää pitkäikäisiä WebSocket-yhteyksiä, joten resurssien hallinta ansaitsee erityistä huomiota:
- Käytä `await using` varmistaaksesi, että istunnot julkaistaan oikein.
- Peruutus käynnistää prosessin puhdistamisen.
- `ACPSessionManager` tukee istuntojen enimmäismäärää.

OpenCode-prosessinhallinta on suhteellisen yksinkertaisempaa ja `OpenCodeRuntimeManager` hoitaa sen automaattisesti.

### Virheiden käsittely

Molemmilla palveluntarjoajilla on täydellinen virheenkäsittely:
- IFlow-virheet leviävät ACP-istuntopäivitysten kautta.
- OpenCode-virheet heitetään läpi `OpenCodeApiException`.
- On suositeltavaa, että soittaja ottaa kiinni ja käsittelee nämä poikkeukset.

### Suorituskykynäkökohdat

- IFlow WebSocket -viestinnällä on alhaisempi latenssi kuin HTTP:llä.
- OpenCode-istunnon uudelleenkäyttö voi vähentää HTTP-pyyntöjen ylimääräisiä kustannuksia.
- Tehdasvälimuistimekanismi välttää palveluntarjoajien toistuvan luomisen.
- Korkean samanaikaisuuden skenaarioissa kiinnitä erityistä huomiota prosessien ja yhteysmäärän rajoituksiin.

### Kokoonpanon validointi

Suoritettava polku tarkistetaan käynnistyksen yhteydessä, mutta ajonaikaisia ongelmia voi silti tapahtua. `PingAsync` on hyödyllinen työkalu sen tarkistamiseen, onko kokoonpano oikein:

```csharp
// Check at startup
var provider = await _providerFactory.GetProviderAsync(providerType);
var result = await provider.PingAsync(cancellationToken);
if (!result.Success)
{
    _logger.LogError("Provider {ProviderType} is unavailable: {Error}", providerType, result.ErrorMessage);
}
```

## Yhteenveto

Tämä artikkeli jakaa teknisen lähestymistavan, jota HagiCode-alusta käyttää integroidessaan kahta tekoälytyökalua iflow ja OpenCode. Yhtenäisen kautta `IAIProvider` käyttöliittymä, sovitimme erilaisia viestintätyylejä, WebSocketin ja HTTP:n, pitäen samalla ylemmän kerroksen puhelumallin yhtenäisenä.

Perusidea on itse asiassa melko yksinkertainen:
1. Määrittele yhtenäinen käyttöliittymän abstraktio.
2. Rakenna sovitinkerroksia eri toteutuksille.
3. Hallitse kaikkea tasaisesti tehdasmallin avulla.

Tämä antaa järjestelmälle hyvän laajennettavuuden. Kun uusi tekoälytyökalu on integroitava myöhemmin, meidän tarvitsee vain ottaa se käyttöön `IAIProvider` käyttöliittymä muuttamatta liikaa olemassa olevaa koodia.

Jos työskentelet myös monen tekoälytyökalun integroinnin parissa, toivon, että tämä artikkeli on hyödyllinen.
