## AI Compose Commit: Tekoälyn käyttäminen Git Commit -työnkulun älykkääseen muokkaamiseen

## Johdanto

Ohjelmistokehitysprosessissa koodin sitominen on rutiinitehtävä, jota jokainen ohjelmoija kohtaa päivittäin. Mutta oletko koskaan törmännyt tähän tilanteeseen: työpäivän päätteeksi avaat Gitin, näet kymmeniä lavastelemattomia muokattuja tiedostoja, etkä sinulla ole aavistustakaan kuinka järjestää ne järkeviksi toimituksiksi?

Perinteinen lähestymistapa on lavastella tiedostot manuaalisesti erissä, sitoa ne yksitellen ja kirjoittaa toimitusviestejä. Tämä prosessi on sekä aikaa vievä että virhealtis. Käytämme usein tähän paljon aikaa, ja loppujen lopuksi kukaan ei halua murehtia näitä tylsiä askareita myöhään illalla, kun on jo väsynyt.

HagiCode-projektissa esittelimme uuden ominaisuuden - AI Compose Commit - joka on suunniteltu muuttamaan tämä työnkulku täysin. Tässä artikkelissa perehdymme syvällisesti toteutusperiaatteisiin, tekniseen arkkitehtuuriin sekä käytännössä kohtaamiimme haasteisiin ja ratkaisuihin.

## Tausta

### Perinteisten Git-sitoumusten kipupisteet

Versionhallintajärjestelmänä Git tarjoaa kehittäjille tehokkaat koodinhallintaominaisuudet. Mutta tosielämässä sitoutumisesta tulee usein pullonkaula kehitystyön kulussa:

1. **Manuaalinen ryhmittely on aikaa vievää**: Kun tiedostoon tulee useita muutoksia, kehittäjien on tarkastettava jokainen tiedosto yksitellen ja päätettävä, mitkä muutokset kuuluvat samaan ominaisuuteen. Se vaatii paljon henkistä vaivaa.
2. **Epäjohdonmukainen toimitusviestin laatu**: Perinteisten toimitusten määrityksiä noudattavien toimitusviestien kirjoittaminen vaatii kokemusta ja taitoa, ja aloittelijat tekevät usein epätyypillisiä sitoumuksia.
3. **Monimutkainen usean tietovaraston hallinta**: Monorepo-ympäristössä vaihtaminen eri arkiston välillä lisää toiminnan monimutkaisuutta.
4. **Keskeytynyt työnkulku**: Koodin sitominen keskeyttää ajatuskulkusi ja heikentää koodauksen tehokkuutta.

Nämä ongelmat ovat erityisen ilmeisiä suurissa projekteissa ja yhteistyöhön perustuvissa tiimiympäristöissä. Hyvän kehitystyökalun pitäisi antaa kehittäjien keskittyä ydinkoodaustyöhön sen sijaan, että he juututuisivat raskaaseen sitouttamisen työnkulkuun.

## Ongelma

### Tekniset haasteet

AI Compose Commitin käyttöönoton aikana kohtasimme useita teknisiä haasteita:

1. **Tiedoston semanttinen ymmärtäminen**: Tekoälyn on ymmärrettävä tiedostomuutosten väliset semanttiset suhteet ja päätettävä, mitkä tiedostot kuuluvat samaan toiminnalliseen moduuliin. Tämä edellyttää tiedoston sisällön, hakemistorakenteen ja muutoksen kontekstin syvällistä analysointia.

2. **Sitoudu ryhmittelystrategiaan**: Miten järkevä ryhmittelystandardi pitäisi määritellä? Ominaisuuden, moduulin tai tiedostotyypin mukaan? Eri hankkeet saattavat tarvita erilaisia ​​strategioita.

3. **Reaaliaikainen palaute ja asynkroninen käsittely**: Git-toiminnot voivat kestää kauan, varsinkin kun käsitellään suurta määrää tiedostoja. Kuinka voimme suorittaa monimutkaisia ​​toimintoja säilyttäen samalla hyvän käyttökokemuksen?

4. **Useiden tietovarastojen tuki**: Monorepo-arkkitehtuurissa toiminnot täytyy reitittää oikein päävaraston ja alivarastojen välillä.

5. **Virheiden käsittely ja palautus**: Jos yksi toimitus epäonnistuu, miten jo suoritettuja toimituksia tulisi käsitellä? Pitääkö jo lavastettuja tiedostoja palauttaa?

6. **Sitoumusviestien johdonmukaisuus**: Luotujen vahvistusviestien on vastattava projektin nykyistä tyyliä ja oltava yhdenmukaisia historiallisten sitoumusten kanssa.

### Suorituskykynäkökohdat

Tekoälyn käsittely suurella määrällä tiedostomuutoksia kuluttaa huomattavasti aikaa ja laskentaresursseja. Meidän piti optimoida seuraavilla alueilla:

- Vähennä turhia tekoälypuheluita

- Kun asiakas on lähettänyt pyynnön, palvelin palauttaa välittömästi HTTP 202 Accepted

Tämä muotoilu varmistaa, että vaikka tekoälyn käsittely kestäisi useita minuutteja, käyttäjät saavat silti välittömän vastauksen eivätkä koe järjestelmän jäätyneen.

- Tiedoston lavastuksen koordinointi: on vuorovaikutuksessa tekoälyn käsittelyvirran kanssa

### Ydintoteutuslogiikka

#### Tiedoston kontekstin rakentaminen

Tekoälyn on ymmärrettävä kunkin tiedoston tila ennen kuin se voi tehdä älykkäitä päätöksiä. Rakennamme tiedostokontekstin kautta `BuildFileChangesXml` menetelmä:

```csharp
/// <summary>
/// Build an XML representation of file changes to provide the AI with complete file context information
/// </summary>
/// <param name="stagedFiles">List of staged files, including file path, status, and old path (for rename operations)</param>
/// <returns>A formatted XML string containing metadata for all files</returns>
private static string BuildFileChangesXml(GitFileStatusDto[] stagedFiles)
{
    var sb = new StringBuilder();
    sb.AppendLine("<files>");

    foreach (var file in stagedFiles)
    {
        sb.AppendLine("  <file>");
        // Use XML escaping to ensure special characters do not break the XML structure
        sb.AppendLine($"    <path>{System.Security.SecurityElement.Escape(file.Path)}</path>");
        sb.AppendLine($"    <status>{System.Security.SecurityElement.Escape(file.Status)}</status>");

        // Handle file rename scenarios and record the old path so the AI can understand change relationships
        if (!string.IsNullOrEmpty(file.OldPath))
        {
            sb.AppendLine($"    <oldPath>{System.Security.SecurityElement.Escape(file.OldPath)}</oldPath>");
        }

        sb.AppendLine("  </file>");
    }

    sb.AppendLine("</files>");
    return sb.ToString();
}
```

Tämä XML-pohjainen konteksti sisältää tiedostopolut, tilat ja vanhat polut uudelleennimeämistoimintoja varten, mikä antaa tekoälylle täydelliset metatiedot. Strukturoidulla XML-muodolla varmistamme, että tekoäly ymmärtää tarkasti kunkin tiedoston tilan ja muutoksen tyypin.

#### Sitouta tuloksen jäsennys

Kun tekoäly on suorittanut loppuun, se palauttaa jäsennellyt tulokset. Otimme käyttöön kaksoisjäsennysstrategian varmistaaksemme yhteensopivuuden:

```csharp
/// <summary>
/// Parse commit execution results returned by the AI, supporting both delimiter format and regex format
/// </summary>
/// <param name="aiResponse">Raw response content returned by the AI</param>
/// <returns>A parsed list of commit results, where each result includes the commit hash and execution status</returns>
private List<CommitResultDto> ParseCommitExecutionResults(string aiResponse)
{
    var results = new List<CommitResultDto>();

    // Prefer delimiter-based parsing (new format), which is more explicit and reliable
    if (aiResponse.Contains("---"))
    {
        logger.LogDebug("Using delimiter-based parsing for AI response");
        results = ParseDelimitedFormat(aiResponse);

        if (results.Count > 0)
        {
            return results; // Successfully parsed, return the results directly
        }

        logger.LogWarning("Delimiter-based parsing produced no results, falling back to regex");
    }
    else
    {
        logger.LogDebug("No delimiter found, using legacy regex-based parsing");
    }

    // Fall back to regex parsing (old format) to ensure backward compatibility
    return ParseLegacyFormat(aiResponse);
}
```

Erotinmuotoa käyttää `---` erottaa sitoumukset, jolloin rakenne on selkeä ja helppo jäsentää:

```
---
Commit 1: abc123def456
feat(auth): add user login functionality

Implement JWT-based authentication with login form and API endpoints.

Co-Authored-By: Hagicode <noreply@hagicode.com>
---
Commit 2: 789ghi012jkl
docs(readme): update installation instructions

Add new setup steps for Docker environment.

Co-Authored-By: Hagicode <noreply@hagicode.com>
---
```

Tämä muoto tekee jäsentämisestä yksinkertaista ja luotettavaa, mutta on myös helppo lukea.

### Lukkojen hallintamekanismi

Samanaikaisten toimintojen aiheuttamien tilaristiriitojen estämiseksi otimme käyttöön arkiston lukitusmekanismin:

```csharp
// Acquire the repository lock to prevent concurrent operations
```

### SignalR:n reaaliaikaiset ilmoitukset

Kun käsittely on valmis, järjestelmä lähettää ilmoituksen käyttöliittymälle SignalR:n kautta:

```csharp
/// <summary>
/// Send a notification when automatic commit composition is complete
/// </summary>
/// <param name="projectId">Project identifier, used to route the notification to the correct client</param>
/// <param name="totalCount">Total number of commits, including successes and failures</param>
/// <param name="successCount">Number of successful commits</param>
/// <param name="failureCount">Number of failed commits</param>
/// <param name="success">Whether the overall operation succeeded</param>
/// <param name="error">Error message (if the operation failed)</param>
private async Task SendAutoComposeCommitNotificationAsync(
    string projectId,
    int totalCount,
    int successCount,
    int failureCount,
    bool success,
    string? error)
{
    try
    {
        // Build the notification DTO containing detailed execution results
        var notification = new AutoComposeCommitCompletedDto
        {
            ProjectId = projectId,
            TotalCount = totalCount,
            SuccessCount = successCount,
            FailureCount = failureCount,
            Success = success,
            Error = error
        };

        // Broadcast the notification to all connected clients through the SignalR Hub
        await messageService.SendAutoComposeCommitCompletedAsync(notification);

        logger.LogInformation(
            "Auto compose commit notification sent for project {ProjectId}: {SuccessCount}/{TotalCount} succeeded",
            projectId, successCount, totalCount);
    }
    catch (Exception ex)
    {
        // Log notification errors without affecting the main operation flow
        // A notification failure should not cause the entire operation to fail
        logger.LogError(ex, "Failed to send auto compose commit notification for project {ProjectId}", projectId);
    }
}
```

Kun käyttöliittymä on vastaanottanut ilmoituksen, se voi päivittää käyttöliittymän näyttämään, onko toimitus onnistunut vai epäonnistunut, mikä parantaa käyttökokemusta. Tämä reaaliaikainen palautemekanismi sai vahvaa palautetta HagiCode-käyttäjiltä, ​​jotka näkevät selvästi, milloin toimenpide on päättynyt ja mikä on lopputulos.

Kehote määrittää yksityiskohtaisesti tiedostojen ryhmittelyn päätöslogiikan:

```handlebars

├── Is it a configuration file (package.json, tsconfig.json, .env, etc.)?
```

#### Historiallisen muodon johdonmukaisuusanalyysi

Jotta toimitusviestit pysyisivät johdonmukaisina projektihistorian kanssa, kehote vaatii tekoälyä analysoimaan viimeaikaisen toimitushistorian ennen luomista:

```handlebars
**Historical Format Consistency**: Before generating commit messages, you **must** analyze the current repository's commit history to match the existing style.

1. Use `git log -n 15 --pretty=format:"%H|%s|%b%n---%n"` to get the recent commit history
2. Analyze the commits to identify:
   - Structural patterns: does the project use multi-paragraph messages? Are there `Changes:` or `Capabilities:` sections?
   - Language patterns: are commit messages in English, Chinese, or mixed?
   - Common types: which commit types are most often used (`feat`, `fix`, `docs`, etc.)?
   - Special formatting: are there `Co-Authored-By` lines? Any other project-specific conventions?
3. Generate commit messages that follow the detected patterns
```

Tämä analyysi varmistaa, että tekoälyn luomat sitoutumisviestit eivät tunnu oikeilta, vaan ne pysyvät tyylillisesti linjassa projektin historian kanssa. HagiCoden monikielisissä projekteissa tämä ominaisuus on erityisen tärkeä, koska se voi automaattisesti valita sopivan kielen ja muodon toimitushistorian perusteella.

   - Suorittaa `git add` ja `git commit` jokaiselle ryhmälle
6. **Tulosten jäsentäminen**: Jäsentää tekoälyn palauttamat suoritustulokset.
7. **Ilmoitusten toimitus**: Ilmoittaa käyttöliittymälle SignalR:n kautta.
8. **Lukon vapautus**: Vapauttaa arkiston lukituksen riippumatta siitä, onnistuuko toiminto vai epäonnistuu.

Tämä työnkulku on suunniteltu siten, että käyttäjät voivat jatkaa muuta työtä heti toiminnan aloittamisen jälkeen odottamatta tekoälyn valmistumista. HagiCoden käyttäjien palaute osoittaa, että tämä asynkroninen käsittelymalli parantaa huomattavasti työnkulkukokemusta.

### Virheenkäsittelymekanismi

Otimme käyttöön monikerroksisen virheenkäsittelyn:

#### 1. Syötteen vahvistus

```csharp
// Validate request parameters to prevent invalid requests from reaching backend processing logic
if (request.UnstagedFiles == null || request.UnstagedFiles.Count == 0)
{
    return BadRequest(new
    {
        message = "No unstaged files provided. Please make changes in the working directory first.",
        status = "validation_failed"
    });
}
```

#### 2. Virheen palautus

Jos tekoälyn käsittelyn aikana tapahtuu virhe, järjestelmä suorittaa palautustoiminnon ja poistaa jo vaiheistettujen tiedostojen vaiheet, mikä estää epäjohdonmukaisen tilan jäämisen jälkeen. Varsinaisessa HagiCode-käytössä tämä mekanismi säästi meidät useilta odottamattomilta keskeytyksiltä ja varmisti arkiston tilan eheyden.

#### 3. Aikakatkaisun käsittely

20 minuutin aikakatkaisu varmistaa, että pitkäkestoiset toiminnot eivät estä resursseja loputtomiin. Aikakatkaisun jälkeen järjestelmä vapauttaa lukon ja ilmoittaa käyttäjälle, että toiminto epäonnistui. HagiCoden todellisessa käytössä havaitsimme, että useimmat toiminnot valmistuvat 2–5 minuutissa, ja vain erittäin suuret muutosjoukot lähestyvät aikakatkaisurajaa.

## Parhaat käytännöt

### Parhaat käytännöt tekoälyn kirjoittamiseen

#### 1. Valitse oikeat hetket

AI Compose Commit sopii parhaiten seuraaviin tilanteisiin:

- Työpäivän lopussa, kun sinun on käsiteltävä useiden tiedostojen muutokset yhdessä erässä
- Refaktorointitoiminnon jälkeen, kun useita toisiinsa liittyviä tiedostoja on sitova erikseen
- Kun ominaisuus on valmis, kun siihen liittyvät muutokset on ryhmiteltävä sitoumuksiin

Se ei sovellu seuraaviin skenaarioihin:

- Nopeat sitoumukset yhdelle tiedostolle (tavallinen toimitus on nopeampi)
- Skenaariot, jotka edellyttävät tarkkaa hallintaa sitoutumissisällössä
- Sitoumus sisältää arkaluontoisia tietoja, jotka edellyttävät ihmisen tarkastelua

#### 2. Tarkista tekoälyn luomat sitoumukset

Vaikka tekoälyn käyttämä älykäs ryhmittely on tehokasta, kehittäjien tulisi silti tarkistaa luodut sitoumukset:

- Tarkista, vastaako ryhmittely odotuksia
- Tarkista vahvistusviestien tarkkuus
- Varmista, että yhtään tiedostoa ei ole jätetty pois tai sisällytetty väärin

Jos löydät järjettömän ryhmittelyn, voit käyttää `git reset --soft HEAD~N` peruuttaaksesi sen ja ryhmitelläksesi uudelleen. HagiCoden kokemus osoittaa, että vaikka tekoälyn ryhmittely on fiksua, manuaalinen tarkastelu on silti arvokasta varsinkin tärkeiden ominaisuussitoumusten yhteydessä.

#### 3. Yhdenmukaista projektistandardien kanssa

Varmista, että projektisi Git-kokoonpano tukee perinteisiä sitoumuksia:

```bash
# Install commitlint
npm install -g @commitlint/cli @commitlint/config-conventional

# Configure commitlint
echo "module.exports = {extends: ['@commitlint/config-conventional']}" > commitlint.config.js
```

Tämän avulla voit vahvistaa vahvistusviestin muodon CI/CD-työnkuluissa ja pitää sen linjassa AI Compose Commitin luoman muodon kanssa.
