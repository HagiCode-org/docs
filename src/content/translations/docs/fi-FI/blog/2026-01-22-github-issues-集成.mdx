## GitHub-ongelmien integroinnin rakentaminen tyhjästä: HagiCoden käyttöliittymän suorayhteyskäytäntö

> Tämä artikkeli dokumentoi koko prosessin, jolla GitHub Issues integroidaan HagiCode-alustaan. Tutkimme, kuinka käyttää "etupuolen suoraa yhteyttä + minimaalista taustajärjestelmää" -arkkitehtuuria turvallisen OAuth-todennuksen ja tehokkaan ongelmien synkronoinnin saavuttamiseksi samalla, kun taustajärjestelmä pysyy kevyenä.

## Taustaa: Miksi integroida GitHub?

Tekoälyavusteisena kehitysalustana HagiCoden ydinarvo on ideoiden yhdistäminen toteutukseen. Mutta todellisessa käytössä havaitsimme, että kun käyttäjät ovat tehneet ehdotuksen HagiCodessa, heidän on usein kopioitava sisältö manuaalisesti GitHub Issues -ohjelmaan projektin seurantaa varten.

Tämä luo useita ilmeisiä kipupisteitä:

1.  **Hajanainen työnkulku**: Käyttäjien on vaihdettava edestakaisin kahden järjestelmän välillä. Kokemus ei ole sujuvaa, ja avaintiedot voivat helposti kadota kopioinnin ja liittämisen aikana.
2.  **Epämukava yhteistyö**: Muut tiimin jäsenet ovat tottuneet tarkistamaan tehtäviä GitHubissa eivätkä näe ehdotuksen edistymistä suoraan HagiCoden sisällä.
3.  **Toistuva manuaalinen työ**: Aina kun ehdotus päivitetään, jonkun on päivitettävä vastaava ongelma manuaalisesti GitHubissa, mikä lisää tarpeettomia ylläpitokustannuksia.

Tämän ongelman ratkaisemiseksi päätimme ottaa käyttöön **GitHub Issues Integration** -ominaisuuden, joka yhdistää HagiCode-istunnot GitHub-tietovarastoihin "yhden napsautuksen synkronoinnin" mahdollistamiseksi.

## Tekninen valinta: Frontend Direct Connection vs backend Proxy

Integraatiolähestymistapaa suunniteltaessa meillä oli edessämme kaksi vaihtoehtoa: perinteinen "taustavälityspalvelinmalli" ja aggressiivisempi "frontend-suorayhteysmalli".

### Ratkaisun vertailu

Perinteisessä **taustavälityspalvelinmallissa** jokaisen käyttöliittymän pyynnön on ensin läpäistävä taustajärjestelmämme, joka sitten kutsuu GitHub API:ta. Tämä keskittää logiikan, mutta se myös rasittaa taustaa merkittävästi:

1.  **Paisunut taustajärjestelmä**: Meidän on kirjoitettava oma GitHub API -asiakaskääre ja myös käsiteltävä monimutkainen OAuth-tilakone.
2.  **Tokenin riski**: Käyttäjän GitHub-tunnus on tallennettava taustatietokantaan. Jopa salauksella tämä lisää silti turvapintaa.
3.  **Kehityskustannukset**: Tarvitsemme tietokantojen siirtoja tokenien tallentamiseen ja ylimääräisen synkronointipalvelun ylläpitoon.

**Etuosan suoraliitäntämalli** on paljon kevyempi. Tässä lähestymistavassa käytämme taustaa vain herkimmässä "salaisessa vaihdossa" (OAuth-soitto). Kun tunnuksen on saatu, tallennamme sen suoraan selaimeen `localStorage`. Myöhemmät toiminnot, kuten ongelmien luominen ja kommenttien päivittäminen, lähetetään suoraan käyttöliittymästä GitHubiin HTTP:n kautta.

| Vertailumitta | Taustavälityspalvelinmalli | Frontend Direct Connection -malli |
| :--- | :--- | :--- |
| **Taustamonimutkaisuus** | Edellyttää täyden OAuth-palvelun ja GitHub API -asiakkaan | Tarvitsee vain OAuth-backback-päätepisteen |
| **Tokenin hallinta** | On salattava ja tallennettava tietokantaan vuotoriskin kanssa | Tallennetaan selaimeen ja näkyvät vain käyttäjälle |
| **Käyttöönottokustannukset** | Edellyttää tietokantojen siirtoja ja monipalvelukehitystä | Pääasiassa frontend-työtä |
| **Käyttäjäkokemus** | Keskitetty logiikka, mutta palvelimen viive voi olla hieman korkeampi | Erittäin nopea vastaus suoralla GitHub-vuorovaikutuksella |

Koska halusimme nopean integroinnin ja minimaalisia taustamuutoksia, **valitsimme lopulta "etupuolen suoran yhteysmallin"**. Se on kuin antaisi selaimelle "väliaikaisen passin". Kun se on saanut hyväksynnän, selain voi käsitellä asioita GitHubissa itse ilman, että se pyytää taustajärjestelmän järjestelmänvalvojalta hyväksyntää joka kerta.

## Ydinsuunnittelu: tiedonkulku ja tietoturva

Arkkitehtuuriin asettumisen jälkeen meidän piti suunnitella tietty tietovirta. Synkronointiprosessin ydin on tunnuksen hankkiminen turvallisesti ja tehokas käyttö.

### Kokonaisarkkitehtuurikaavio

Koko järjestelmä voidaan tiivistää kolmeen rooliin: selain (käyttöliittymä), HagiCode-taustajärjestelmä ja GitHub.

```text
+--------------+        +--------------+        +--------------+
| Frontend     |        |   Backend    |        |   GitHub     |
| React        |        |   ASP.NET    |        |   REST API   |
|              |        |              |        |              |
|  +--------+  |        |              |        |              |
|  | OAuth  |--+--------> /callback    |        |              |
|  | Flow   |  |        |              |        |              |
|  +--------+  |        |              |        |              |
|              |        |              |        |              |
|  +--------+  |        |  +--------+  |        |  +--------+  |
|  | GitHub |  +------------> Session |  +----------> Issues |  |
|  | API    |  |        |  |Metadata|  |        |  |        |  |
|  | Direct |  |        |  +--------+  |        |  +--------+  |
|  +--------+  |        |              |        |              |
+--------------+        +--------------+        +--------------+
```

**Avainkohta on**: vain yksi pieni vaihe OAuthissa (koodin vaihtaminen tunnukseksi) tarvitsee taustajärjestelmän läpi. Sen jälkeen raskas nostaminen (ongelmien luominen) hoidetaan suoraan käyttöliittymän ja GitHubin välillä.

### Synkronointitietovirta yksityiskohtaisesti

Kun käyttäjä napsauttaa "Synkronoi GitHubiin" -painiketta HagiCode-käyttöliittymässä, tapahtuu sarja monimutkaisia toimintoja:

```text
User clicks "Sync to GitHub"
         │
         ▼
1. Frontend checks localStorage for the GitHub token
         │
         ▼
2. Format issue content (convert the Proposal into Markdown)
         │
         ▼
3. Frontend directly calls the GitHub API to create/update the issue
         │
         ▼
4. Call the HagiCode backend API to update Session.metadata (store issue URL and other info)
         │
         ▼
5. Backend broadcasts the SessionUpdated event via SignalR
         │
         ▼
6. Frontend receives the event and updates the UI to show the "Synced" state
```

### Turvallisuussuunnittelu

Turvallisuus on aina etusijalla integroitaessa kolmannen osapuolen palveluita. Teimme seuraavat huomiot:

1.  **Puolusta CSRF-hyökkäyksiä vastaan**: Luo satunnainen `state` parametri OAuth-uudelleenohjauksen aikana ja tallenna se sisään `sessionStorage`. Vahvista takaisinkutsun tila tiukasti väärennettyjen pyyntöjen estämiseksi.
2.  **Erillinen tunnuksen tallennus**: Tunnus tallennetaan vain selaimen muistiin `localStorage`. Saman alkuperän käytäntöä käytettäessä vain HagiCode-komentosarjat voivat lukea sen välttäen palvelinpuolen tietokantavuodon, joka vaikuttaa käyttäjiin.
3.  **Virherajat**: Suunnittelimme omistetun käsittelyn yleisille GitHub API -virheille (kuten 401 vanhentunut tunnus, 422 vahvistusvirhe ja 429 taajuuden rajoitus), jotta käyttäjät saavat ystävällistä palautetta.

## Käytännössä: Käyttöönoton tiedot koodissa

Teoria menee vain niin pitkälle. Katsotaanpa kuinka koodi todella toimii.

### 1. Minimaaliset taustamuutokset

Taustajärjestelmän tarvitsee tehdä vain kaksi asiaa: tallentaa synkronointitiedot ja käsitellä OAuth-soittoa.

### 2. Käyttöliittymän OAuth Flow

Tämä on yhteyden sisääntulopiste. Käytämme normaalia valtuutuskoodikulkua.

```typescript
// src/HagiCode.Client/src/services/githubOAuth.ts

// Generate the authorization URL and redirect
export async function generateAuthUrl(): Promise<string> {
  const state = generateRandomString(); // Generate a random string for CSRF protection
  sessionStorage.setItem('hagicode_github_state', state);
  
  const params = new URLSearchParams({
    client_id: clientId,
    redirect_uri: window.location.origin + '/settings?tab=github&oauth=callback',
    scope: ['repo', 'public_repo'].join(' '),
    state: state,
  });
  
  return `https://github.com/login/oauth/authorize?${params.toString()}`;
}

// Handle the code-to-token exchange on the callback page
export async function exchangeCodeForToken(code: string, state: string): Promise<GitHubToken> {
  // 1. Validate state to prevent CSRF
  const savedState = sessionStorage.getItem('hagicode_github_state');
  if (state !== savedState) throw new Error('Invalid state parameter');

  // 2. Call the backend API to exchange the token
  // Note: this must go through the backend because ClientSecret cannot be exposed to the frontend
  const response = await fetch('/api/GitHubOAuth/callback', {
    method: 'POST',
    headers: { 'Content-Type': 'application/json' },
    body: JSON.stringify({ code, state, redirectUri: window.location.origin + '/settings?tab=github&oauth=callback' }),
  });

  if (!response.ok) throw new Error('Failed to exchange token');
  
  const token = await response.json();
  
  // 3. Save into LocalStorage
  saveToken(token);
  return token;
}
```

### 3. GitHub API Client Wrapper

Kun meillä on tunnus, tarvitsemme vankan työkalun GitHub API:n kutsumiseen.

```typescript
// src/HagiCode.Client/src/services/githubApiClient.ts

const GITHUB_API_BASE = 'https://api.github.com';

// Core request wrapper
async function githubApi<T>(endpoint: string, options: RequestInit = {}): Promise<T> {
  const token = localStorage.getItem('gh_token');
  if (!token) throw new Error('Not connected to GitHub');
  
  const response = await fetch(`${GITHUB_API_BASE}${endpoint}`, {
    ...options,
    headers: {
      ...options.headers,
      Authorization: `Bearer ${token}`,
      Accept: 'application/vnd.github.v3+json', // Specify the API version
    },
  });
  
  // Error handling logic
  if (!response.ok) {
    if (response.status === 401) throw new Error('GitHub token expired, please reconnect');
    if (response.status === 403) throw new Error('No permission to access this repository or rate limit exceeded');
    if (response.status === 422) throw new Error('Issue validation failed, the title may be duplicated');
    throw new Error(`GitHub API Error: ${response.statusText}`);
  }
  
  return response.json();
}

// Create issue
export async function createIssue(owner: string, repo: string, data: { title: string, body: string, labels: string[] }) {
  return githubApi(`/repos/${owner}/${repo}/issues`, {
    method: 'POST',
    body: JSON.stringify(data),
  });
}
```

### 4. Sisällön muotoilu ja synkronointi

Viimeinen vaihe on muuntaa HagiCode-istuntotiedot GitHub-ongelman muotoon. Se on vähän kuin käännöstyötä.

```typescript
// Convert a Session object into a Markdown string
function formatIssueForSession(session: Session): string {
  let content = `# ${session.title}\n\n`;
  content += `**> HagiCode Session:** #${session.code}\n`;
  content += `**> Status:** ${session.status}\n\n`;
  content += `## Description\n\n${session.description || 'No description provided.'}\n\n`;
  
  // If this is a Proposal session, add extra fields
  if (session.type === 'proposal') {
    content += `## Chief Complaint\n\n${session.chiefComplaint || ''}\n\n`;
    // Add a deep link so users can jump back from GitHub to HagiCode
    content += `---\n\n**[View in HagiCode](hagicode://sessions/${session.id})**\n`;
  }
  
  return content;
}

// Main logic when clicking the sync button
const handleSync = async (session: Session) => {
  try {
    const repoInfo = parseRepositoryFromUrl(session.repoUrl); // Parse the repository URL
    if (!repoInfo) throw new Error('Invalid repository URL');

    toast.loading('Syncing to GitHub...');
    
    // 1. Format content
    const issueBody = formatIssueForSession(session);
    
    // 2. Call API
    const issue = await githubApiClient.createIssue(repoInfo.owner, repoInfo.repo, {
      title: `[HagiCode] ${session.title}`,
      body: issueBody,
      labels: ['hagicode', 'proposal', `status:${session.status}`],
    });
    
    // 3. Update Session Metadata (save the issue link)
    await SessionsService.patchApiSessionsSessionId(session.id, {
      metadata: {
        githubIssue: {
          owner: repoInfo.owner,
          repo: repoInfo.repo,
          issueNumber: issue.number,
          issueUrl: issue.html_url,
          syncedAt: new Date().toISOString(),
        }
      }
    });

    toast.success('Synced successfully!');
  } catch (err) {
    console.error(err);
    toast.error('Sync failed, please check your token or network');
  }
};
```

## Yhteenveto ja näkymät

Tällä "frontend-suora yhteys" -lähestymistavalla saavutimme saumattoman GitHub Issues -integraation mahdollisimman pienellä taustakoodilla.

### Tärkeimmät voitot
1.  **Korkea kehitystehokkuus**: Backend-muutokset ovat minimaalisia, pääasiassa yksi ylimääräinen tietokantakenttä ja yksinkertainen OAuth-backback-päätepiste. Suurin osa logiikasta valmistuu käyttöliittymässä.
2.  **Vahva suojaus**: Tunnus ei kulje palvelintietokannan läpi, mikä vähentää vuotoriskiä.
3.  **Erinomainen käyttökokemus**: Pyynnöt käynnistetään suoraan käyttöliittymästä, joten vastausnopeus on nopea eikä taustan edelleenlähetystä tarvita.

### Varottavia asioita
On olemassa muutamia sudenkuoppia, jotka on pidettävä mielessä todellisen käyttöönoton aikana:
- **OAuth-sovelluksen asetukset**: Muista kirjoittaa oikea `Authorization callback URL` GitHub OAuth -sovelluksen asetuksissa (yleensä `http://localhost:3000/settings?tab=github&oauth=callback`).
- **Nopeusrajoitukset**: GitHub API rajoittaa todentamattomia pyyntöjä melko tiukasti, mutta tunnuksella kiintiö on yleensä riittävä (5000 pyyntöä/tunti).
- **URL-osoitteiden jäsennys**: Käyttäjät syöttävät kaikenlaisia arkiston URL-osoitteita, joten varmista, että säännöllinen lauseke vastaa `.git` jälkiliitteet, SSH-muodot ja vastaavat tapaukset.

### Tulevia parannuksia
Nykyinen ominaisuus on edelleen yksisuuntainen synkronointi (HagiCode -> GitHub). Jatkossa aiomme toteuttaa kaksisuuntaisen synkronoinnin GitHub Webhooksin kautta. Jos ongelma on esimerkiksi suljettu GitHubissa, istunnon tila HagiCode-puolella saattaa myös päivittyä automaattisesti. Tämä edellyttää, että paljastamme webhook-päätepisteen taustalla, mikä on mielenkiintoinen seuraava askel.
