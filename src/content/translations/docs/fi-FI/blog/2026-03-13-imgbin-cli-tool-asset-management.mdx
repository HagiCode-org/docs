## ImgBin CLI Tool Design: HagiCoden Image Asset Management Approach

> Tässä artikkelissa kerrotaan, miten automatisoitava kuvamateriaaliputki rakennetaan tyhjästä. Se kattaa CLI-työkalujen suunnittelun, Provider Adapter -arkkitehtuurin ja metatietojen hallintastrategiat.

## Tausta

Rehellisesti sanottuna en odottanut kuvaomaisuuden hallinnan pitävän meitä sotkussa näin pitkään.

HagiCode-kehityksen aikana törmäsimme ongelmaan, joka näytti pinnalta yksinkertaiselta, mutta oli yllättävän hankala käytännössä: kuvaresurssien luominen ja hallinta. Se oli tavallaan kuin nuoruuden draamoja - ulkoa rauhallista, alla myrskyisää.

Kun projekti keräsi lisää dokumentaatiota ja markkinointimateriaalia, tarvitsimme suuren määrän tukikuvia. Jotkut piti olla tekoälyn luomia, jotkin valittava olemassa olevasta aineistokirjastosta, ja toiset tarvitsivat tekoälytunnistuksen ja automaattisen merkitsemisen. Ongelmana oli, että kaikki tämä oli pitkään käsitelty hajallaan olevien komentosarjojen ja manuaalisten vaiheiden avulla. Joka kerta kun loimme kuvan, meidän täytyi ajaa skripti käsin, järjestää metatiedot käsin ja luoda pikkukuvat käsin. Se yksinään oli tarpeeksi ärsyttävää, mutta suurempi ongelma oli, että kaikki oli hajallaan kaikkialla. Kun halusimme löytää jotain, emme löytäneet. Kun meidän piti käyttää jotain uudelleen, emme voineet.

Kipukohdat olivat konkreettisia:

1. **Ei yhtenäistä aloituspistettä**: kuvien luomisen logiikka oli hajallaan eri skriptien kesken, joten eräsuoritus oli käytännössä mahdotonta.
2. **Puuttuvat metatiedot**: luoduilla kuvilla ei ollut yhtenäistystä `metadata.json`, mikä ei merkinnyt luotettavaa hakua tai jäljitettävyyttä.
3. **Korkeat manuaaliset järjestelykustannukset**: otsikot ja tunnisteet piti lajitella yksitellen käsin, mikä oli tehotonta.
4. **Ei automaatiota**: luodaanko visuaalisia resursseja automaattisesti CI/CD-liukuhihnassa? Ei mahdollisuutta.

Ajattelimme jättää sen rauhaan. Mutta hankkeita on vielä vietävä eteenpäin. Koska emme voineet välttää ongelmaa, ajattelimme, että voisimme myös ratkaista sen. Joten päätimme päivittää ImgBinin hajallaan olevien komentosarjojen joukosta kuvaresurssien putkistoon, joka voidaan suorittaa automaattisesti. Jotkut ongelmat eivät loppujen lopuksi katoa vain siksi, että katsot muualle.

## Ydinsuunnittelu

### Kokonaisarkkitehtuuri

ImgBin käyttää kerrostettua arkkitehtuuria, joka erottaa selkeästi CLI-komennot, sovelluspalvelut, kolmannen osapuolen API-sovittimet ja infrastruktuurikerroksen:

```
Component hierarchy
├── CLI Entry (cli.ts)              Global argument parsing, command routing
├── Commands (commands/*)           generate | batch | annotate | thumbnail
├── Application Services            job-runner | metadata | thumbnail | asset-writer
├── Provider Adapters               image-api-provider | vision-api-provider
└── Infrastructure Layer            config | logger | paths | schema
```

Tämän kerrostetun suunnittelun etuna on selkeät vastuurajat. Se tekee myös testaamisesta helpompaa, koska ulkoisia riippuvuuksia voidaan pilkata puhtaasti. Käytännössä se tarkoittaa vain sitä, että jokainen kerros tekee oman työnsä ilman, että se häiritsee muita, joten kun jokin hajoaa, on helpompi selvittää miksi.

### Yhden omaisuuden hakemistomalli

ImgBin käyttää mallia "yksi omaisuus, yksi hakemisto". Joka kerta kun kuva luodaan, se luo seuraavanlaisen rakenteen:

```
library/
└── 2026-03/
    └── orange-dashboard/
        ├── original.png      # Original image
        ├── thumbnail.webp    # 512x512 thumbnail
        └── metadata.json     # Structured metadata
```

Tämän mallin edut ovat:

1. **Itsenäinen**: kaikki yhden omaisuuden tiedostot sijaitsevat samassa hakemistossa, mikä tekee siirrosta ja varmuuskopioinnista kätevää.
2. **Jäljitettävissä**: `metadata.json` mahdollistaa sukupolven ajan, kehotteen, mallin ja muiden yksityiskohtien jäljittämisen.
3. **Laajennettavissa**: jos myöhemmin tarvitaan lisää muunnelmia, kuten erikokoisia pikkukuvia, voimme yksinkertaisesti lisätä uusia tiedostoja samaan hakemistoon.

Kauniita asioita ei aina tarvitse omistaa. Joskus riittää, että ne pysyvät kauniina ja voit hiljaa arvostaa niitä. Tämä saattaa kuulostaa hieman kaukaiselta, mutta logiikka pätee silti: kun kuvat pidetään yhdessä, niitä on miellyttävämpi katsella ja paljon helpompi löytää.

### Kerrostettu metatietojen tallennus

`metadata.json` on koko järjestelmän ydin. Se käyttää kerrostettua tallennusstrategiaa, joka jakaa kentät kolmeen luokkaan:

```json
{
  "schemaVersion": 2,
  "assetId": "orange-dashboard",
  "slug": "orange-dashboard",
  "title": "Orange Dashboard",
  "tags": ["dashboard", "hero", "orange"],

  "source": { "type": "generated" },

  "paths": {
    "assetDir": "library/2026-03/orange-dashboard",
    "original": "original.png",
    "thumbnail": "thumbnail.webp"
  },

  "generated": {
    "prompt": "orange dashboard for docs hero",
    "provider": "azure-openai-image-api",
    "model": "gpt-image-1.5"
  },

  "recognized": {
    "title": "Orange Dashboard",
    "tags": ["dashboard", "ui", "orange"],
    "description": "A modern orange dashboard with charts and metrics"
  },

  "status": {
    "generation": "succeeded",
    "recognition": "succeeded",
    "thumbnail": "succeeded"
  },

  "timestamps": {
    "createdAt": "2026-03-11T04:01:19.570Z",
    "updatedAt": "2026-03-11T04:02:09.132Z"
  }
}
```

- **generated**: tallentaa kuvan luomisen alkuperäiset tiedot, kuten kehotteen, toimittajan ja mallin.
- **tunnistettu**: tallentaa tekoälyn tunnistustuloksia, kuten automaattisesti luodut otsikot, tunnisteet ja kuvaukset.
- **manuaalinen**: tallentaa manuaalisesti kuratoidut tulokset. Tämän alueen tiedoilla on korkein prioriteetti, eikä tekoälytunnistus korvaa niitä.

Tämä kerrostettu strategia ratkaisee yhden aikaisemmista ydinkonflikteistamme: kun tekoälytunnistus ja manuaalinen kuratointi ovat eri mieltä, kumman pitäisi voittaa? Vastaus on manuaalinen syöttö. AI-tunnistus on tarkoitettu auttamaan, ei päättämään. Tämä kysymys myös selkiytyi ajan myötä - koneet ovat edelleen koneita, ja lopulta ihmisten täytyy silti soittaa.

## Palveluntarjoajan sovitinmalli

Toinen ImgBinin ydinosa on Provider Adapter -kuvio. Abstraktoimme ulkoiset API:t yhtenäisen käyttöliittymän taakse, joten vaikka vaihtaisimme tekoälypalveluntarjoajaa, meidän ei tarvitse muuttaa liiketoimintalogiikkaa.

Se on tavallaan vähän kuin ihmissuhteita – ulkonäkö voi muuttua, mutta tärkeintä on, että sisäinen rakenne pysyy samana. Kun käyttöliittymä on korjattu, sisäinen toteutus voi vaihdella vapaasti.

### Image Generation Provider

```typescript
interface ImageGenerationProvider {
  // Generate an image and return its Buffer
  generate(options: GenerateOptions): Promise<Buffer>;

  // Get the list of supported models
  getSupportedModels(): Promise<string[]>;
}

interface GenerateOptions {
  prompt: string;
  model?: string;
  size?: '1024x1024' | '1792x1024' | '1024x1792';
  quality?: 'standard' | 'hd';
  format?: 'png' | 'webp' | 'jpeg';
}
```

### Näöntunnistuksen tarjoaja

```typescript
interface VisionRecognitionProvider {
  // Recognize image content and return structured metadata
  recognize(imageBuffer: Buffer): Promise<RecognitionResult>;

  // Get the list of supported models
  getSupportedModels(): Promise<string[]>;
}

interface RecognitionResult {
  title?: string;
  tags: string[];
  description?: string;
  confidence: number;
}
```

Tämän käyttöliittymäsuunnittelun edut ovat:

1. **Testattava**: Yksikkötesteissä voimme hyväksyä valepalveluntarjoajat todellisten ulkoisten API-kutsujen sijaan.
2. **Laajennettavissa**: uuden palveluntarjoajan lisääminen edellyttää vain käyttöliittymän toteuttamista; soittajan koodia ei tarvitse muuttaa.
3. **Vaihdettava**: Tuotanto voi käyttää Azure OpenAI:ta, kun taas testaus voi käyttää paikallista mallia, jolloin kokoonpano on ainoa muuttuva asia.

Joskus myös projektityö tuntuu siltä. Pinnalla näyttää siltä, ​​että olemme juuri vaihtaneet API:n, mutta sisäinen logiikka pysyy täsmälleen samana, mikä tekee koko asiasta paljon vähemmän pelottavaa.

## CLI-komentosuunnittelu

ImgBin tarjoaa neljä ydinkomentoa eri käyttöskenaarioiden kattamiseksi:

### generoi: yhden kuvan luominen

```bash
# Simplest usage
imgbin generate --prompt "orange dashboard for docs hero"

# Generate a thumbnail and AI annotations at the same time
imgbin generate --prompt "orange dashboard" --annotate --thumbnail

# Specify an output directory
imgbin generate --prompt "orange dashboard" --output ./library
```

### erä: erätyöt

Erätyöt määritellään YAML- tai JSON-luettelotiedostoilla, mikä tekee niistä sopivia CI/CD-työnkulkuihin:

```yaml
# assets/jobs/launch.yaml
defaults:
  annotate: true
  thumbnail: true
  libraryRoot: ./library

jobs:
  - prompt: "orange dashboard hero"
    slug: orange-dashboard
    tags: [dashboard, hero, orange]

  - prompt: "pricing grid for docs"
    slug: pricing-grid
    tags: [pricing, grid, docs]
```

Suorita komento:

```bash
imgbin batch assets/jobs/launch.yaml
```

Erätyön suunnittelu tukee vikojen eristämistä: luettelon kohteet käsitellään yksitellen, eikä yhden kohteen virhe vaikuta muihin. Voit myös esikatsella työtä `--dry-run` toteuttamatta sitä varsinaisesti.

Ja mikä parasta, se kertoo tarkalleen, mikä onnistui ja mikä epäonnistui. Toisin kuin joissakin elämän asioissa, joissa epäonnistumisia tapahtuu ja et edes tiedä kuinka se tapahtui.

### annotate: AI-merkintä

Suorita AI-tunnistus olemassa oleville kuville luodaksesi automaattisesti otsikot, tunnisteet ja kuvaukset:

```bash
# Annotate a single image
imgbin annotate ./library/2026-03/orange-dashboard

# Annotate an entire directory in batch
imgbin annotate ./library/2026-03/
```

### pikkukuva: pikkukuvien luominen

Luo pikkukuvat olemassa oleville kuville:

```bash
# Generate a thumbnail
imgbin thumbnail ./library/2026-03/orange-dashboard
```

## Erätyön manifestisuunnittelu

Erätöiden luettelomuoto tukee joustavaa määritystä. Oletusasetukset voidaan asettaa maailmanlaajuisesti, ja yksittäiset työt voivat ohittaa ne:

```yaml
# Global defaults
defaults:
  annotate: true        # Enable AI annotation by default
  thumbnail: true       # Generate thumbnails by default
  libraryRoot: ./library
  model: gpt-image-1.5

jobs:
  # Minimal configuration: only provide a prompt
  - prompt: "first image"

  # Full configuration
  - prompt: "second image"
    slug: custom-slug
    tags: [tag1, tag2]
    annotate: false     # Do not run AI annotation for this job
    model: dall-e-3    # Use a different model for this job
```

Kun ImgBin suoritetaan, se käsittelee työt yksitellen. Jokaisen työn tulos kirjoitetaan sitä vastaavaan `metadata.json`. Vaikka yksi työ epäonnistuisi, muut eivät vaikuta. Kun kaikki työt on suoritettu, CLI tulostaa yhteenvetoraportin:

```
✓ orange-dashboard (succeeded)
✓ pricing-grid (succeeded)
✗ hero-banner (failed: API rate limit exceeded)

2/3 succeeded, 1 failed
```

Joitakin asioita ei voi kiirehtiä. Niiden ottaminen yksi kerrallaan on usein vakaampi tie. Ehkä tämä on erätöiden taustalla oleva filosofia.

## Ympäristömuuttujan kokoonpano

ImgBin tukee joustavaa konfigurointia ympäristömuuttujien kautta:

```bash
# ImgBin working directory
IMGBIN_WORKDIR=/path/to/imgbin

# Executable path (for invocation inside scripts)
IMGBIN_EXECUTABLE=/path/to/imgbin/dist/cli.js

# Asset library root
IMGBIN_LIBRARY_ROOT=./.imgbin-library

# Azure OpenAI configuration (if using the Azure provider)
AZURE_OPENAI_ENDPOINT=https://your-resource.openai.azure.com/
AZURE_OPENAI_API_KEY=***
AZURE_OPENAI_IMAGE_DEPLOYMENT=gpt-image-1
```

Kokoonpano on yksi niistä asioista, jotka voivat tuntua samanaikaisesti sekä tärkeiltä että ei niin tärkeiltä. Loppujen lopuksi se, mikä tuntuu mukavalta ja sopii parhaiten työnkulkuusi, on yleensä oikea valinta.

## Toteutusohjeet

Toteutuksen aikana tiivistimme muutamia keskeisiä kohtia:

### Palveluntarjoajan käyttöliittymäsuunnittelu

Liitäntämääritelmien tulee olla selkeitä ja täydellisiä, mukaan lukien syöttöparametrit, palautusarvot ja virheiden käsittely. On myös hyvä idea tarjota sekä synkroniset että asynkroniset kutsutyylit eri skenaarioihin.

Se on pieni pala kovalla työllä ansaittua kokemusta. Kun käyttöliittymä on asetettu, kukaan ei halua muuttaa sitä myöhemmin.

### Viankäsittelystrategia

Kun yksi kohde epäonnistuu erätyössä, CLI:n pitäisi:

1. Kirjoita yksityiskohtaiset virhetiedot erilliseen lokitiedostoon.
2. Jatka muiden töiden suorittamista koko prosessin keskeyttämisen sijaan.
3. Palauta nollasta poikkeava poistumiskoodi lopussa osoittaaksesi, että jotkut työt epäonnistuivat.
4. Näytä jokaisen työn suoritustulos selkeästi yhteenvetoraportissa.

Jotkut epäonnistumiset ovat vain epäonnistumisia. Ei ole mitään järkeä teeskennellä muuta. On parempi tunnustaa ne avoimesti ja sitten selvittää, kuinka ne ratkaistaan. Sama logiikka pätee projekteihin ja elämään.

### Metatietojen yhdistämisstrategia

Tunnustuksen tulokset kirjoitetaan `recognized` oletuksena, kun taas manuaalisesti muokatut kentät on merkitty `manual`. Metatietojen päivitykset noudattavat vain liittämisstrategiaa: ellei `--force` on nimenomaisesti hyväksytty, olemassa olevia manuaalisesti kuratoituja tuloksia ei kirjoiteta päälle.

Sekin seikka tuli selväksi - jotkin asiat, kerran päällekirjoitettuina, ovat vain poissa. Usein on parempi säilyttää ne, koska levyllä itsessään on arvoa.

### Hakemiston luominen Atomicity

Käytä `fs.mkdir({ recursive: true })` varmistaaksesi, että hakemistojen luominen pysyy atomina ja välttää kilpailuolosuhteet samanaikaisissa skenaarioissa.

Ehkä tältä turvallisuus tuntuu - olla vakaa, kun vakaudella on väliä, liikkua nopeasti, kun nopeudella on väliä, eikä koskaan jäädä jumiin arvaamaan.

## Johtopäätös

HagiCode-projektin kuvaomaisuuden hallinnan ydintyökaluna ImgBin ratkaisee ongelmamme seuraavien suunnitteluvaihtoehtojen avulla:

1. **Yhdistetty aloituspiste**: CLI kattaa luomisen, huomautukset, pikkukuvat ja kaikki muut ydintoiminnot.
2. **Metatietopohjainen**: jokaisella sisällöllä on täydellinen `metadata.json`mahdollistaa haun ja jäljitettävyyden.
3. **Provider Adapter**: joustava abstraktio ulkoisille sovellusliittymille, mikä helpottaa testausta ja laajentamista.
4. **Erätyön tuki**: Eräkuvan luominen voidaan automatisoida CI/CD-työnkuluissa.

Kaikki muu voi olla haalistunut, mutta tämä lähestymistapa osoittautui todella hyödylliseksi.

Tämä ratkaisu ei ainoastaan paranna HagiCoden omaa kehitystehokkuutta, vaan muodostaa myös uudelleen käytettävän kehyksen kuvaomaisuuden hallintaan. Jos rakennat samanlaista monikomponenttiprojektia, uskon, että ImgBinin suunnitteluideat voivat antaa sinulle inspiraatiota.

Nuoruudessa on kyse asioiden kokeilemisesta ja sotkuista. Jos et koskaan selviä siitä, mistä tietäisit, mihin todella pystyt?
