## VSCode ja koodipalvelin: Selainpohjaisen koodinmuokkausratkaisun valitseminen

> Rakentaessaan selainpohjaisia koodinmuokkausominaisuuksia kehittäjien edessä on keskeinen valinta: käyttää VSCoden virallista `code serve-web` ominaisuus tai ota käyttöön yhteisövetoinen `code-server` ratkaisu? Tämä päätös ei vaikuta ainoastaan ​​tekniseen arkkitehtuuriin, vaan myös lisenssien noudattamiseen ja käyttöönoton joustavuuteen.

## Tausta

Tekninen valinta on paljon kuin elämänpolun valinta. Kun valitset yhden, sinun on yleensä jatkettava sen kävelemistä, ja myöhemmin vaihtamisesta voi tulla erittäin kallista.

Tekoälyavusteisen ohjelmoinnin aikakaudella selainpohjaisesta koodinmuokkauksesta on tulossa yhä tärkeämpää. Käyttäjät odottavat, että kun tekoälyavustaja on lopettanut koodin analysoinnin, he voivat heti avata editorin samassa selainistunnossa ja tehdä muutoksia vaihtamatta sovellusta. Tällaisen saumattoman kokemuksen pitäisi olla olemassa silloin, kun sitä tarvitset.

Kuitenkin tämän ominaisuuden käyttöönoton yhteydessä kehittäjät kohtaavat kriittisen teknisen valinnan: pitäisikö heidän käyttää VSCoden virallista `code serve-web` ominaisuus tai yhteisölähtöinen `code-server` ratkaisu?

Jokaisella vaihtoehdolla on omat vahvuutensa ja kompromissinsa, ja huono valinta voi aiheuttaa myöhemmin paljon ongelmia. Lisenssi on yksi esimerkki: jos huomaat vasta julkaisun jälkeen, että tuotteesi ei ole lisenssin mukainen, on jo liian myöhäistä. Käyttöönotto on toinen: ratkaisu saattaa toimia täydellisesti kehitysvaiheessa, ja sen jälkeen kohdata kaikenlaisia ​​ongelmia, kun se on siirretty säiliöön. Juuri tällaisia ​​sudenkuoppia joukkueet haluavat välttää.

## Lisenssierot (tärkeimmät)

Tämä on perustavanlaatuisin ero näiden kahden ratkaisun välillä ja ensimmäinen tekijä, jonka huomioimme arvioinnissa. Teknistä valintaa tehtäessä on tärkeää ymmärtää oikeudelliset riskit etukäteen.

**koodipalvelin**
- MIT-lisenssi, täysin avoin lähdekoodi
- Ylläpitää Coder.com aktiivisen yhteisön kanssa
- Vapaasti käytettävä kaupallisesti, muokata ja jakaa
- Ei rajoituksia käyttöskenaarioihin

**VSCode `code serve-web`**
- Osa Microsoft VSCode -tuotetta
- Käyttää Microsoftin lisenssiä (VS Code -lisenssi sisältää rajoituksia kaupalliseen käyttöön)
- Ensisijaisesti tarkoitettu yksittäisille kehittäjille
- Yrityskäyttöönotto voi vaatia kaupallisten lisenssien lisätarkistuksia

Lisensoinnin näkökulmasta koodipalvelin on ystävällisempi kaupallisiin projekteihin. Tämä on asia, jota sinun on harkittava tuotesuunnittelun aikana, koska siirtyminen myöhemmin voi tulla erittäin kalliiksi.

## Käyttöönoton erot

Kun lisensointi on ratkaistu, seuraava ongelma on käyttöönotto. Se vaikuttaa suoraan toimintakustannuksiisi ja arkkitehtoniseen suunnitteluun.

**koodipalvelin**
- Itsenäinen Node.js-sovellus, joka voidaan ottaa käyttöön itsenäisesti
- Tukee useita ajonaikaisia lähteitä:
  - Suoritettavan polun määrittäminen suoraan
  - Sen etsiminen järjestelmän PATH:n kautta
  - NVM Node.js 22.x -ympäristön automaattinen tunnistus
- VSCode-työpöytäsovellusta ei tarvitse asentaa palvelimelle
- Helpompi ottaa käyttöön konteissa

**VSCode `code serve-web`**
- Täytyy riippua paikallisesti asennetusta VSCode CLI:stä
- Edellyttää saatavilla olevaa `code` komento isäntäkoneessa
- Järjestelmä suodattaa pois VS Code Remote CLI -kääreet
- Suunniteltu ensisijaisesti paikallisiin kehitysskenaarioihin

code-server sopii paremmin palvelimen ja säilön käyttöönottoskenaarioihin. Jos tuotteesi on toimittava Dockerissa tai käyttäjilläsi ei ole asennettuna VSCodea, koodipalvelin on yleensä oikea valinta.

## Ominaisuusparametrien erot

Nämä kaksi ratkaisua eroavat myös muutamista ominaisuusparametreista. Erot eivät ole valtavia, mutta ne voivat aiheuttaa integraatiokitkaa todellisessa käytössä.

| Ominaisuus | koodipalvelin | koodi palvele-web |
|------|-------------|----------------|
| Julkinen tukikohta | `/` (konfiguroitavissa) | `/vscode-server` (korjattu) |
| Todennus | `--auth` parametri useilla tiloilla | `--connection-token` / `--without-connection-token` |
| Tietohakemisto | `{DataDir}/code-server` | `{DataDir}/vscode-serve-web` |
| Telemetria | Poistettu oletuksena käytöstä `--disable-telemetry` | Riippuu VSCode-asetuksista |
| Päivitä tarkistukset | Voidaan poistaa käytöstä `--disable-update-check` | Riippuu VSCode-asetuksista |

Nämä erot vaativat erityistä huomiota integraation aikana. Esimerkiksi erilaiset URL-polut tarkoittavat, että käyttöliittymäkoodisi tarvitsee erityistä käsittelyä.

## Saatavuus havaitsemiserot

Editorin vaihtoa toteutettaessa myös saatavuuden havaitsemislogiikka eroaa.

**koodipalvelin**
- Palautettu aina näkyvänä toteutuksena
- Näytetään edelleen, vaikka se ei ole saatavilla, ja an `install-required` tila
- Tukee NVM Node.js 22.x -ympäristön automaattista tunnistamista

**koodipalvelu-verkko**
- Näkyy vain paikallisena `code` CLI on havaittu
- Jos se ei ole käytettävissä, käyttöliittymä piilottaa tämän vaihtoehdon automaattisesti
- Riippuu paikallisesta VSCode-asennustilasta

Tämä ero vaikuttaa suoraan käyttökokemukseen. koodipalvelin on läpinäkyvämpi: käyttäjät näkevät vaihtoehdon ja ymmärtävät, että asennus on edelleen tarpeen. code serve-web on piilotettu: käyttäjät eivät ehkä edes huomaa vaihtoehdon olemassaoloa. Kumpi lähestymistapa on parempi, riippuu tuotteen sijainnista.

## HagiCoden kaksois-toteutusarkkitehtuuri

HagiCode-projekti omaksui perusteellisen analyysin jälkeen kaksoistoteutusarkkitehtuurin, joka tukee molempia ratkaisuja arkkitehtuuritasolla.

### Oletuksena koodipalvelin

```csharp
// The default active implementation is code-server
// If an explicit activeImplementation is saved, try that implementation first
// If the requested implementation is unavailable, the resolver tries the other one
// If a fallback occurs, return fallbackReason
```

Käytämme oletuksena koodipalvelinta pääasiassa lisensoinnin ja käyttöönoton joustavuuden vuoksi. Kuitenkin käyttäjille, joilla on jo paikallinen VSCode-ympäristö, koodiserver-web on myös hyvä vaihtoehto.

### Toteutuksen valitsin

`CodeServerImplementationResolver` on vastuussa yhdistämisestä:
- Toteutusvalinta käynnistyksen lämmittelyn aikana
- Toteutuksen valinta luettaessa tilaa
- Toteutusvalinta projekteja avattaessa
- Toteutusvalinta Vaultsia avattaessa

Tämän suunnittelun ansiosta järjestelmä voi reagoida joustavasti erilaisiin skenaarioihin, ja käyttäjät voivat valita ympäristöään parhaiten sopivan toteutuksen.

### Käyttöliittymän mukautussäännöt

```typescript
// When localCodeAvailable=false, do not show code serve-web
// When localCodeAvailable=true, show the code serve-web configuration
```

Käyttöliittymä näyttää automaattisesti käytettävissä olevat vaihtoehdot ympäristön perusteella, joten käyttäjät eivät hämmentyisi ominaisuuksista, joita he eivät voi käyttää.

## Käytännön opas

Mihin kaiken teorian jälkeen pitäisi kiinnittää huomiota todellisen käyttöönoton aikana? Loppujen lopuksi toteutus ratkaisee.

### Dockerin käyttöönottosuositus

Säiliöityyn käyttöön koodipalvelin on parempi valinta:

```dockerfile
# Use the official code-server image directly
FROM codercom/code-server:latest

# Or install through npm
RUN npm install -g code-server
```

Tämä ratkaisee ongelman yhdessä kerroksessa ilman ylimääräistä VSCode-asennusta.

### Esimerkkejä kokoonpanosta

**koodipalvelimen määritys**
```json
{
  "vscodeServer": {
    "enabled": true,
    "activeImplementation": "code-server",
    "codeServer": {
      "host": "0.0.0.0",
      "port": 8080,
      "executablePath": "",
      "authMode": "none"
    }
  }
}
```

**koodipalvelun verkkomääritys**
```json
{
  "vscodeServer": {
    "enabled": true,
    "activeImplementation": "serve-web",
    "serveWeb": {
      "host": "0.0.0.0",
      "port": 8080,
      "executablePath": "/usr/local/bin/code"
    }
  }
}
```

Kokoonpano voi olla hieman tylsää ensimmäisellä kerralla, mutta kun se on paikallaan, asioita on paljon helpompi ylläpitää.

### URL-rakenteen erot

**koodipalvelin**
```
http://localhost:8080/?folder=/path/to/project&vscode-lang=zh-CN
```

**koodipalvelu-verkko**
```
http://localhost:8080/vscode-server/?folder=/path/to/project&tkn=xxx&vscode-lang=zh-CN
```

Kiinnitä huomiota polkujen ja parametrien eroihin. Sinun on käsiteltävä niitä erikseen integroinnin aikana.

### Toteutusten vaihtaminen

Järjestelmä tukee ajonaikaista vaihtoa ja pysäyttää automaattisesti edellisen toteutuksen vaihdon yhteydessä:

```csharp
// VsCodeServerManager automatically handles mutual exclusion
// When switching activeImplementation, the old implementation will not keep running in the background
```

Tämän suunnittelun avulla käyttäjät voivat kokeilla erilaisia toteutuksia milloin tahansa ja löytää itselleen parhaiten sopivan vaihtoehdon.

### Tilan valvonta

```typescript
const { settings, runtime } = await getVsCodeServerSettings();

// runtime.activeImplementation: "code-server" | "serve-web"
// runtime.fallbackReason: reason for switching
// runtime.status: "running" | "starting" | "stopped" | "unhealthy"
```

Kun tila on näkyvissä, käyttäjät voivat nopeasti määrittää, johtuuko ongelma palvelinpuolelta vai omasta toiminnasta.

## Johtopäätös

| Vertailumitta | koodipalvelin | koodi palvele-web | Suositus |
|---------|-------------|----------------|------|
| Lisenssi | MIT (kaupallinen) | Microsoft (rajoitettu) | koodipalvelin |
| Käyttöönoton joustavuus | Itsenäinen käyttöönotto | Riippuu paikallisesta VSC-koodista | koodipalvelin |
| Palvelimen soveltuvuus | Suunniteltu palvelimille | Lähinnä paikalliseen kehittämiseen | koodipalvelin |
| Säiliöinti | Alkuperäinen tuki | Vaatii VSCode-asennuksen | koodipalvelin |
| Ominaisuuden täydellisyys | Lähes työpöytäversiota | Virallinen täydellinen versio | koodi palvele-web |
| Huoltotoiminta | Aktiivinen yhteisö | Virallisesti ylläpitämä Microsoft | Molemmilla on vahvuuksia |

**Suositeltu strategia**: Käytä ensin koodipalvelinta ja harkitse koodiserveriä, kun tarvitset täyden virallisen toiminnallisuuden ja sinulla on jo paikallinen VSCode-ympäristö.
