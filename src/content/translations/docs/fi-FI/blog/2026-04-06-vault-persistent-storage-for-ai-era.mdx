## Kuinka toistaa projekteja tekoälyn aikakaudella: Holvi, projektien välinen pysyvä tallennusjärjestelmä

> Kuinka voimme auttaa tekoälyassistentteja ymmärtämään oppimisresurssejamme paremmin tekoälyavusteisen kehityksen aikakaudella? HagiCode-projekti rakensi Vault-järjestelmän yhtenäiseksi tiedontallennus-abstraktiokerrokseksi, jota tekoäly voi ymmärtää ja parantaa huomattavasti oppimisen tehokkuutta projektin toistamisen kautta.

## Tausta

Tekoälyn aikakaudella tapa, jolla kehittäjät oppivat uusia teknologioita ja arkkitehtuureja, muuttuu perusteellisesti. "Projektien toistamisesta" - eli erinomaisten avoimen lähdekoodin projektien koodin, arkkitehtuurin ja suunnittelumallien syvällinen opiskelu ja niistä oppiminen - on tullut tehokas tapa oppia. Verrattuna perinteisiin menetelmiin, kuten kirjojen lukemiseen tai videoiden katseluun, korkealaatuisten avoimen lähdekoodin projektien suora lukeminen ja suorittaminen auttaa sinua ymmärtämään todellisia suunnittelukäytäntöjä paljon nopeammin.

Silti tämä oppimismenetelmä tuo mukanaan melkoisia haasteita.

Oppimateriaalit ovat liian hajallaan. Muistiinpanosi voivat olla Obsidianissa, koodivarastot voivat olla hajallaan eri kansioissa, ja tekoälyassistenttisi keskusteluhistoriasta tulee jälleen yksi erillinen tietosaari. Kun haluat tekoälyn auttavan analysoimaan projektia, sinun on kopioitava manuaalisesti koodinpätkät ja järjestettävä konteksti, mikä on melko tylsää.

Vielä ongelmallisempaa on rikkinäinen konteksti. Tekoälyassistentit eivät pääse suoraan paikallisiin oppimisresursseihin, joten sinun on annettava taustatiedot uudelleen jokaisessa keskustelussa. Tämän lisäksi toistetut koodivarastot päivittyvät nopeasti, manuaalinen synkronointi on virhealtista ja tietoa on vaikea jakaa useiden oppimisprojektien kesken.

Pohjimmiltaan kaikki nämä ongelmat tulevat "tietosaarilta". Ongelma ratkeaisi siististi, jos olemassa olisi yhtenäinen tallennusabstraktiokerros, jonka avulla tekoälyassistentit voisivat ymmärtää ja käyttää kaikkia oppimisresurssejasi.

## Tietoja HagiCodesta

Tässä artikkelissa jaettu Vault-järjestelmä on juuri se ratkaisu, jonka kehitimme rakentaessamme [HagiCode](https://hagicode.com). HagiCode on tekoälyn koodausassistenttiprojekti, ja päivittäisessä kehitystyössämme joudumme usein tutkimaan ja referoimaan monia erilaisia ​​avoimen lähdekoodin projekteja. Auttaaksemme tekoälyassistentteja ymmärtämään näitä oppimisresursseja paremmin, suunnittelimme Vaultin, projektien välisen pysyvän tallennusjärjestelmän.

Tämä ratkaisu on jo validoitu HagiCodessa todellisessa käytössä. Jos kohtaat samanlaisia ​​tiedonhallinnan haasteita, toivon, että nämä kokemukset voivat tarjota inspiraatiota. Loppujen lopuksi, kun olet itse pudonnut muutamaan kuoppaan, sinun pitäisi jättää jotain seuraavalle henkilölle.

## Holvijärjestelmän suunnittelufilosofia

Vault-järjestelmän ydinidea on yksinkertainen: luo yhtenäinen tiedon varastointikerros, jota tekoäly voi ymmärtää. Toteutuksen näkökulmasta järjestelmällä on useita keskeisiä ominaisuuksia.

### Monimuotoinen tuki

Järjestelmä tukee neljää varastotyyppiä, joista jokainen vastaa eri käyttöskenaariota:

```typescript
// folder: general-purpose folder type
export const DEFAULT_VAULT_TYPE = 'folder';

// coderef: a type specifically for reproduced code projects
export const CODEREF_VAULT_TYPE = 'coderef';

// obsidian: integrated with Obsidian note-taking software
export const OBSIDIAN_VAULT_TYPE = 'obsidian';

// system-managed: vault automatically managed by the system
export const SYSTEM_MANAGED_VAULT_TYPE = 'system-managed';
```

Niistä **coderef**-tyyppi on yleisimmin käytetty HagiCodessa. Se on suunniteltu erityisesti toistettuja koodiprojekteja varten, ja se tarjoaa standardoidun hakemistorakenteen ja tekoälyn luettavat metatietokuvaukset.

### Pysyvä säilytysmekanismi

Vault-rekisteri tallennetaan jatkuvasti JSON-muodossa, mikä varmistaa, että kokoonpano pysyy saatavilla sovelluksen uudelleenkäynnistyksen jälkeen:

```csharp
public class VaultRegistryStore : IVaultRegistryStore
{
    private readonly string _registryFilePath;

    public VaultRegistryStore(IConfiguration configuration, ILogger<VaultRegistryStore> logger)
    {
        var dataDir = configuration["DataDir"] ?? "./data";
        var absoluteDataDir = Path.IsPathRooted(dataDir)
            ? dataDir
            : Path.GetFullPath(Path.Combine(Directory.GetCurrentDirectory(), dataDir));

        _registryFilePath = Path.Combine(absoluteDataDir, "personal-data", "vaults", "registry.json");
    }
}
```

Tämän suunnittelun etuna on, että se on yksinkertainen ja luotettava. JSON on ihmisten luettavissa, mikä tekee virheenkorjauksesta ja manuaalisesta muokkaamisesta helpompaa. tiedostojärjestelmän tallennus välttää tietokannan monimutkaisuuden ja vähentää järjestelmän riippuvuuksia. Loppujen lopuksi joskus yksinkertaisin vaihtoehto on todellakin paras.

### AI-kontekstin integrointi

Mikä tärkeintä, järjestelmä voi automaattisesti syöttää holvin tiedot tekoälyehdotusten kontekstiin:

```typescript
export function buildTargetVaultsText(
  vaults: VaultForText[],
  template: VaultPromptTemplate = DEFAULT_VAULT_PROMPT_TEMPLATE,
): string {
  const readOnlyVaults = vaults.filter((vault) => vault.accessType === 'read');
  const editableVaults = vaults.filter((vault) => vault.accessType === 'write');

  if (readOnlyVaults.length === 0 && editableVaults.length === 0) {
    return '';
  }

  const sections = [
    buildVaultSection(readOnlyVaults, template.reference),
    buildVaultSection(editableVaults, template.editable),
  ].filter(Boolean);

  return `\n\n### ${template.heading}\n\n${sections.join('\n')}`;
}
```

Tämä mahdollistaa tärkeän ominaisuuden: AI-avustajat voivat automaattisesti ymmärtää käytettävissä olevat oppimisresurssit ilman, että käyttäjät antavat kontekstia manuaalisesti. Voisi sanoa, että se on eräänlainen hiljainen ymmärrys.

## CodeRef Vaultin standardoitu rakenne

Coderef-tyyppiselle varastolle HagiCode tarjoaa standardoidun hakemistorakenteen:

```
my-coderef-vault/
├── index.yaml          # vault metadata description
├── AGENTS.md           # operating guide for AI assistants
├── docs/               # stores study notes and documents
└── repos/              # manages reproduced code repositories through Git submodules
```

Kun luot varaston, järjestelmä alustaa automaattisesti tämän rakenteen:

```csharp
private async Task EnsureCodeRefStructureAsync(
    string vaultName,
    string physicalPath,
    ICollection<VaultBootstrapDiagnosticDto> diagnostics,
    CancellationToken cancellationToken)
{
    Directory.CreateDirectory(physicalPath);

    var indexPath = Path.Combine(physicalPath, CodeRefIndexFileName);
    var docsPath = Path.Combine(physicalPath, CodeRefDocsDirectoryName);
    var reposPath = Path.Combine(physicalPath, CodeRefReposDirectoryName);

    // Create the standard directory structure
    if (!Directory.Exists(docsPath))
    {
        Directory.CreateDirectory(docsPath);
    }

    if (!Directory.Exists(reposPath))
    {
        Directory.CreateDirectory(reposPath);
    }

    // Create the AGENTS.md guide
    await EnsureCodeRefAgentsDocumentAsync(physicalPath, cancellationToken);

    // Create the index.yaml metadata
    await WriteCodeRefIndexDocumentAsync(indexPath, mergedDocument, cancellationToken);
}
```

Tämä rakenne on myös huolellisesti suunniteltu:

- **docs/** tallentaa opiskelumuistiinpanosi, joihin voit tallentaa ymmärryksesi koodista, arkkitehtuurianalyysistä, oppituneista ja niin edelleen Markdownissa
- **repos/** hallitsee toistettuja tietovarastoja Git-alimoduulien kautta sen sijaan, että kopioiisi koodia suoraan, mikä pitää koodin synkronoituna ja säästää tilaa
- **index.yaml** sisältää holvin metatiedot, jotta tekoälyavustajat voivat nopeasti ymmärtää holvin tarkoituksen ja sisällön
- **AGENTS.md** on erityisesti tekoälyassistenteille kirjoitettu opas, joka selittää kuinka käsitellä holvin sisältöä

Näin organisoituna tekoäly ymmärtää ehkä hieman helpommin, mitä ajattelet.

## Automaattinen alustus järjestelmän hallinnoimille varastoille

Manuaalisesti luotujen varastojen lisäksi HagiCode tukee myös järjestelmähallittuja varastoja:

```csharp
public async Task<IReadOnlyList<VaultRegistryEntry>> EnsureAllSystemManagedVaultsAsync(
    CancellationToken cancellationToken = default)
{
    var definitions = GetAllResolvedDefinitions();
    var entries = new List<VaultRegistryEntry>(definitions.Count);

    foreach (var definition in definitions)
    {
        entries.Add(await EnsureResolvedSystemManagedVaultAsync(definition, cancellationToken));
    }

    return entries;
}
```

Järjestelmä luo ja hallitsee automaattisesti seuraavat varastot:

- **hagiprojectdata**: projektitietojen tallennustila, jota käytetään projektin kokoonpanon ja tilan tallentamiseen
- **personaldata**: henkilötietojen tallennustila, jota käytetään käyttäjän asetusten tallentamiseen
- **hbsprompt**: kehotemallikirjasto, jota käytetään yleisesti käytettyjen tekoälykehotteiden hallintaan

Nämä varastot alustetaan automaattisesti järjestelmän käynnistyessä, joten käyttäjien ei tarvitse määrittää niitä manuaalisesti. Jotkut asiat on yksinkertaisesti parempi jättää järjestelmän huoleksi sen sijaan, että ihmiset huolehtisivat niistä.

## Kulunvalvontamekanismi

Tärkeä osa suunnittelua on kulunvalvonta. Järjestelmä jakaa holvit kahteen pääsytyyppiin:

```typescript
export interface VaultForText {
  id: string;
  name: string;
  type: string;
  physicalPath: string;
  accessType: 'read' | 'write';  // Key: distinguish read-only from editable
}
```

- **viite (vain luku)**: tekoälyä käytetään vain analysointiin ja ymmärtämiseen, eikä se voi muokata sisältöä. Soveltuu referoituihin avoimen lähdekoodin projekteihin, asiakirjoihin ja vastaaviin materiaaleihin
- **muokattava (muokattava)**: AI voi muokata sisältöä tehtävän tarpeen mukaan. Sopii muistiinpanoihisi, luonnoksiisi ja vastaaviin materiaaleihin

Tällä erolla on merkitystä. Se kertoo tekoälylle, mikä sisältö on "vain luku -viittaus" ja mikä sisältö on "turvallista muokata", mikä vähentää tahattomien muutosten riskiä. Loppujen lopuksi kukaan ei halua kovan työnsä katoavan tahattoman muokkauksen vuoksi.

## Käytännössä: Vaultin luominen ja käyttö

Nyt kun olemme käsitelleet ideat, katsotaan kuinka se toimii käytännössä.

### Luo CodeRef Vault

Tässä on täydellinen käyttöliittymäkutsuesimerkki:

```typescript
const createCodeRefVault = async () => {
  const response = await VaultService.postApiVaults({
    requestBody: {
      name: "React Learning Vault",
      type: "coderef",
      physicalPath: "/Users/developer/vaults/react-learning",
      gitUrl: "https://github.com/facebook/react.git"
    }
  });

  // The system will automatically:
  // 1. Clone the React repository into vault/repos/react
  // 2. Create the docs/ directory for notes
  // 3. Generate the index.yaml metadata
  // 4. Create the AGENTS.md guide file

  return response;
};
```

Tämä API-kutsu suorittaa sarjan toimintoja: hakemistorakenteen luomisen, Git-alimoduulien alustamisen, metatietotiedostojen luomisen ja paljon muuta. Sinun tarvitsee vain antaa perustiedot ja antaa järjestelmän hoitaa loput. Se on suoraan sanottuna melko huoleton lähestymistapa.

### Käytä Holvia tekoälyehdotuksessa

Kun olet luonut holvin, voit viitata siihen tekoälyehdotuksessa:

```typescript
const proposal = composeProposalChiefComplaint({
  chiefComplaint: "Help me analyze React's concurrent rendering mechanism",
  repositories: [
    { id: "react", gitUrl: "https://github.com/facebook/react.git" }

    {

```

Järjestelmä syöttää automaattisesti holvin tiedot tekoälykontekstiin, jolloin tekoäly tietää, mitkä oppimisresurssit ovat käytettävissä. Kun tekoäly ymmärtää, mitä sinulla on mielessä, tällaista hiljaista ymmärrystä on vaikea saada.

## Parhaat käytännöt ja huomioitavaa

Käytettäessä Vault-järjestelmää olemme koonneet yhteenvedon muutamasta oppimisesta.

### Polun turvallisuus

Järjestelmä vahvistaa tiukasti polut estääkseen polun läpikulkuhyökkäykset:

```csharp
private static string ResolveFilePath(string vaultRoot, string relativePath)
{
    var rootPath = EnsureTrailingSeparator(Path.GetFullPath(vaultRoot));
    var combinedPath = Path.GetFullPath(Path.Combine(rootPath, relativePath));
    if (!combinedPath.StartsWith(rootPath, StringComparison.OrdinalIgnoreCase))
    {
        throw new BusinessException(VaultRelativePathTraversalCode,
            "Vault file paths must stay inside the registered vault root.");
    }
    return combinedPath;
}
```

Tämä on tärkeää. Jos mukautat varastopolkua, varmista, että se pysyy sallitulla alueella, muuten järjestelmä hylkää toiminnon. Turvallisuutta ei todellakaan voi korostaa liikaa.

### Git-alimoduulin hallinta

CodeRef Vault suosittelee Git-alimoduuleja koodin suoran kopioimisen sijaan:

```csharp
private static string BuildCodeRefAgentsContent()
{
    return """
    # CodeRef Vault Guide

    Repositories under `repos/` should be maintained through Git submodules
    rather than copied directly into the vault root.

    Keep this structure stable so assistants and tools can understand the vault quickly.
    """ + Environment.NewLine;
}
```

Tämä tuo useita etuja: koodin pitäminen synkronoituna ylävirran kanssa, levytilan säästäminen ja useiden koodiversioiden hallinnan helpottaminen. Loppujen lopuksi, kuka haluaa ladata saman asian uudestaan ​​ja uudestaan?

### Tiedostojen esikatselun rajoitukset

Suorituskykyongelmien estämiseksi järjestelmä rajoittaa tiedostokokoa ja -tyyppiä:

```csharp
private const int FileEnumerationLimit = 500;
private const int PreviewByteLimit = 256 * 1024;  // 256KB
```

Jos varastosi sisältää suuren määrän tiedostoja tai erittäin suuria tiedostoja, esikatselun suorituskyky saattaa heikentyä. Siinä tapauksessa voit harkita tiedostojen käsittelyä erissä tai erikoishakutyökalujen käyttöä. Joskus kun jokin on liian suuri, sitä on vaikeampi käsitellä, ei helpompi.

### Diagnostiset tiedot

Kun luot varaston, järjestelmä palauttaa diagnostiikkatietoja, jotka auttavat virheenkorjauksessa:

```csharp
List<VaultBootstrapDiagnosticDto> bootstrapDiagnostics = [];

if (IsCodeRefVaultType(normalizedType))
{
    bootstrapDiagnostics = await EnsureCodeRefBootstrapAsync(
        normalizedName,
        normalizedPhysicalPath,
        normalizedGitUrl,
        cancellationToken);
}
```

Jos luominen epäonnistuu, voit tarkastaa diagnostiikkatiedot selvittääksesi tarkan syyn. Kun jokin menee pieleen, diagnoosin tarkistaminen on usein suorin tapa edetä.

## Yhteenveto

Yhtenäisen tallennusabstraktiokerroksen avulla Vault-järjestelmä ratkaisee useita tekoälyn aikakauden toistoprojektien keskeisiä kipukohtia:

- **Keskitetty tiedonhallinta**: kaikki oppimisresurssit on koottu yhteen paikkaan sen sijaan, että ne olisivat hajallaan kaikkialla
- **Automaattinen tekoälykontekstin lisäys**: AI-avustajat voivat automaattisesti ymmärtää käytettävissä olevat oppimisresurssit ilman manuaalista kontekstin määritystä
- **Projektien välinen tiedon uudelleenkäyttö**: tietoa voidaan jakaa ja käyttää uudelleen useissa oppimisprojekteissa
- **Standardioitu hakemistorakenne**: johdonmukainen hakemistoasettelu alentaa oppimiskäyrää

Tämä ratkaisu on jo validoitu HagiCode-projektissa. Jos olet myös rakentamassa tekoälyavusteiseen kehitykseen liittyviä työkaluja tai kohtaat vastaavia tiedonhallintaongelmia, toivon, että nämä kokemukset voivat toimia hyödyllisenä referenssinä.

Todellisuudessa teknisen ratkaisun arvo ei piile siinä, kuinka monimutkainen se on, vaan siinä, ratkaiseeko se todellisia ongelmia. Vault-järjestelmän ydinidea on hyvin yksinkertainen: rakentaa yhtenäinen tietotallennuskerros, jota tekoäly voi ymmärtää. Mutta juuri tämä yksinkertainen abstraktio paransi kehitystehokkuuttamme melkoisesti.

Joskus yksinkertainen lähestymistapa on todella paras. Monimutkaiset asiat kätkevät usein yhä enemmän sudenkuoppia...
