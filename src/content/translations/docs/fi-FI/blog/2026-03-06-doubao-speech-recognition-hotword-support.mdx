## Opas Hotword-tuen käyttöönottoon Doubao-puheentunnistuksessa

> Tässä artikkelissa kerrotaan yksityiskohtaisesti, kuinka Doubao-puheentunnistuksen hotword-tuki otetaan käyttöön HagiCode-projektissa. Käyttämällä sekä mukautettuja toimintosanoja että alustan toimintosanataulukoita voit parantaa merkittävästi verkkotunnuskohtaisen sanaston tunnistustarkkuutta.

## Tausta

Puheentunnistustekniikkaa on kehitetty useita vuosia, mutta yksi ongelma on jatkuvasti vaivannut kehittäjiä. Yleiskäyttöiset puheentunnistusmallit voivat kattaa jokapäiväisen kielen, mutta ne jäävät usein ammattiterminologiaan, tuotenimiin ja henkilönimiin. Ajattele sitä: lääketieteen alan ääniavustajan on tunnistettava tarkasti termit, kuten "hypertensio", "diabetes" ja "sepelvaltimotauti"; Oikeusjärjestelmän on kaapattava tarkasti termit, kuten "toimisyy", "puolustus" ja "todistustaakka". Näissä skenaarioissa yleiskäyttöinen malli yrittää parhaansa, mutta se ei useinkaan riitä.

Törmäsimme samaan haasteeseen HagiCode-projektissa. Monikäyttöisenä AI-koodausavustajana HagiCoden on käsiteltävä puheentunnistusta monenlaisia ​​teknisiä termejä varten. Doubao puheentunnistuksen sovellusliittymä ei kuitenkaan pystynyt oletuskokoonpanossaan täyttämään täysin erikoistermien tarkkuusvaatimuksiamme. Kyse ei ole siitä, että Doubao ei olisi tarpeeksi hyvä; jokaisella verkkotunnuksella on pikemminkin oma terminologiajärjestelmänsä. Tutkimuksen ja teknisen tutkimuksen jälkeen huomasimme, että Doubao puheentunnistuksen sovellusliittymä todella tarjoaa hotword-tuen. Suoraviivaisella kokoonpanolla se voi parantaa merkittävästi tietyn sanaston tunnistustarkkuutta. Eräässä mielessä, kun kerrot sille, mihin sanoihin kiinnittää huomiota, se kuuntelee niitä tarkemmin.

Tämä artikkeli jakaa täydellisen ratkaisun, jota käytimme HagiCode-projektissa Doubao-puheentunnistuksen toimintasanojen toteuttamiseen. Molemmat tilat, mukautetut toimintosanat ja alustan toimintosanataulukot, ovat saatavilla, ja niitä voidaan myös yhdistää. Tämän ratkaisun avulla kehittäjät voivat joustavasti konfiguroida toimintaskenaarioihin perustuvia toimintasanoja, jotta puheentunnistusjärjestelmä voi paremmin "tunnistaa" ammattimaisen, harvinaisen mutta kriittisen sanaston.

## Ydintoteutus

### Kahden toimintosanatilan ymmärtäminen

Doubao puheentunnistuksen API tarjoaa kaksi tapaa määrittää hotwords, ja jokaisella on omat ihanteelliset käyttötapaukset ja etunsa.

**Muokatun toimintosanatilan** avulla voimme välittää toimintosanatekstin suoraan `corpus.context` kenttä. Tämä lähestymistapa sopii erityisen hyvin skenaarioihin, joissa sinun on määritettävä nopeasti pieni määrä toimintasanoja, kuten tuotteen tai henkilön nimen tilapäinen tunnistaminen. HagiCoden toteutuksessa jäsennämme käyttäjän syöttämän monirivisen hotword-tekstin merkkijonoluetteloksi ja muotoilemme sen sitten `context_data` Doubao API:n vaatima taulukko. Tämä lähestymistapa on hyvin suora: kerrot järjestelmälle, mihin sanoihin sinun tulee kiinnittää huomiota, ja se tekee juuri sen.

**Alustan hotword table mode** käyttää `corpus.boosting_table_id` kenttä, joka viittaa esikonfiguroituun hotword-taulukkoon Doubaon itseoppimisalustassa. Tämä lähestymistapa sopii skenaarioihin, joissa sinun on hallittava suurta määrää toimintasanoja. Voimme luoda ja ylläpitää hotword-taulukoita Doubaon itseoppivalla alustalla ja viitata niihin sitten tunnuksella. Hankkeessa, kuten HagiCode, jossa erikoistermejä on jatkuvasti päivitettävä ja ylläpidettävä, tämä tila tarjoaa paljon paremman hallittavuuden. Kun hotwordien määrä kasvaa, keskitetty paikka niiden hallintaan on paljon parempi kuin syöttää ne manuaalisesti joka kerta.

Mielenkiintoista on, että näitä kahta tilaa voidaan käyttää myös yhdessä. Doubao API tukee sekä mukautettujen toimintosanojen että alustan toimintosanataulukkotunnuksen sisällyttämistä samaan pyyntöön, ja yhdistelmästrategiaa hallitsee `combine_mode` parametri. Tämän joustavuuden ansiosta HagiCode pystyy käsittelemään monenlaisia ​​monimutkaisia ​​ammattiterminologian tunnistustarpeita. Joskus useiden lähestymistapojen yhdistäminen tuottaa parempia tuloksia.

### Käyttöliittymän tyypin määritelmät ja validointi

HagiCoden käyttöliittymätoteutuksessa määritimme täydellisen joukon hotword-määritystyyppejä ja validointilogiikka. Ensimmäinen osa on tyypin määritelmä:

```typescript
export interface HotwordConfig {
  contextText: string;           // Multi-line hotword text
  boostingTableId: string;      // Doubao platform hotword table ID
  combineMode: boolean;          // Whether to use both together
}
```

Tämä yksinkertainen käyttöliittymä sisältää kaikki hotword-ominaisuuden asetukset. Heidän joukossaan `contextText` on osa, jonka kanssa käyttäjät ovat vuorovaikutuksessa suorimmin: annamme käyttäjien kirjoittaa yhden hotword-lauseen riviä kohden, mikä on erittäin intuitiivista. On paljon helpompaa pyytää käyttäjiä syöttämään yksi termi riviä kohden kuin saada heidät ymmärtämään monimutkainen määritysmuoto.

Seuraavaksi tulee vahvistustoiminto. Doubao APIn rajoitusten perusteella määritimme tiukat vahvistussäännöt: enintään 100 riviä hotword-tekstiä, enintään 50 merkkiä riviä kohden ja enintään 5000 merkkiä yhteensä; `boosting_table_id` voi olla enintään 200 merkkiä ja sisältää vain kirjaimia, numeroita, alaviivoja ja yhdysmerkkejä. Nämä rajat eivät ole mielivaltaisia; ne tulevat suoraan Doubaon virallisesta dokumentaatiosta. API-rajoitukset ovat API-rajoituksia, ja meidän on noudatettava niitä.

```typescript
export function validateContextText(contextText: string): HotwordValidationResult {
  if (!contextText || contextText.trim().length === 0) {
    return { isValid: true, errors: [] };
  }

  const lines = contextText.split('\n').filter(line => line.trim().length > 0);
  const errors: string[] = [];

  if (lines.length > 100) {
    errors.push(`Hotword line count cannot exceed 100 lines; current count is ${lines.length}`);
  }

  const totalChars = contextText.length;
  if (totalChars > 5000) {
    errors.push(`Total hotword character count cannot exceed 5000; current count is ${totalChars}`);
  }

  for (let i = 0; i < lines.length; i++) {
    if (lines[i].length > 50) {
      errors.push(`Hotword on line ${i + 1} exceeds the 50-character limit`);
    }
  }

  return { isValid: errors.length === 0, errors };
}

export function validateBoostingTableId(boostingTableId: string): HotwordValidationResult {
  if (!boostingTableId || boostingTableId.trim().length === 0) {
    return { isValid: true, errors: [] };
  }

  const errors: string[] = [];

  if (boostingTableId.length > 200) {
    errors.push(`boosting_table_id cannot exceed 200 characters; current count is ${boostingTableId.length}`);
  }

  if (!/^[a-zA-Z0-9_-]+$/.test(boostingTableId)) {
    errors.push('boosting_table_id can contain only letters, numbers, underscores, and hyphens');
  }

  return { isValid: errors.length === 0, errors };
}
```

Nämä vahvistustoiminnot toimivat välittömästi, kun käyttäjä määrittää toimintosanat, mikä varmistaa, että ongelmat havaitaan mahdollisimman varhaisessa vaiheessa. Käyttäjäkokemuksen näkökulmasta tällainen välitön palaute on erittäin tärkeää. Käyttäjien on aina parempi tietää, mikä on vialla, kun he kirjoittavat, kuin lähettämisen jälkeen.

### Käyttöliittymän kokoonpanon pysyvyys

HagiCoden käyttöliittymätoteutuksessa päätimme käyttää selaimen `localStorage` hotword-määrityksen tallentamiseen. Tämän suunnittelupäätöksen taustalla oli useita näkökohtia. Ensinnäkin hotword-määritys on erittäin henkilökohtaista, ja eri käyttäjillä voi olla erilaisia ​​verkkotunnuskohtaisia ​​tarpeita. Toiseksi tämä lähestymistapa yksinkertaistaa taustaohjelman toteutusta, koska se ei vaadi ylimääräisiä tietokantataulukoita tai API-päätepisteitä. Lopuksi, kun käyttäjät ovat määrittäneet sen kerran selaimessa, asetukset voidaan ladata automaattisesti myöhempien käyttökertojen yhteydessä, mikä on erittäin kätevää. Yksinkertaisesti sanottuna se on helpoin tapa.

```typescript
const HOTWORD_STORAGE_KEYS = {
  contextText: 'hotword-context-text',
  boostingTableId: 'hotword-boosting-table-id',
  combineMode: 'hotword-combine-mode',
} as const;

export const DEFAULT_HOTWORD_CONFIG: HotwordConfig = {
  contextText: '',
  boostingTableId: '',
  combineMode: false,
};

// Load hotword configuration
export function loadHotwordConfig(): HotwordConfig {
  const contextText = localStorage.getItem(HOTWORD_STORAGE_KEYS.contextText) || '';
  const boostingTableId = localStorage.getItem(HOTWORD_STORAGE_KEYS.boostingTableId) || '';
  const combineMode = localStorage.getItem(HOTWORD_STORAGE_KEYS.combineMode) === 'true';

  return { contextText, boostingTableId, combineMode };
}

// Save hotword configuration
export function saveHotwordConfig(config: HotwordConfig): void {
  localStorage.setItem(HOTWORD_STORAGE_KEYS.contextText, config.contextText);
  localStorage.setItem(HOTWORD_STORAGE_KEYS.boostingTableId, config.boostingTableId);
  localStorage.setItem(HOTWORD_STORAGE_KEYS.combineMode, String(config.combineMode));
}
```

Tämän koodin logiikka on suoraviivaista ja selkeää. Luimme alkaen `localStorage` asetuksia ladattaessa ja kirjoita kohtaan `localStorage` kun tallennat sen. Tarjoamme myös oletuskokoonpanon, jotta järjestelmä voi toimia oikein, vaikka konfiguraatiota ei ole vielä olemassa. Onhan siinä oltava järkevä oletusarvo.

### Backend SDK -määrityslaajennus

HagiCoden taustatoteutuksessa meidän piti lisätä hotwordiin liittyviä ominaisuuksia SDK-määritysluokkaan. Ottaen huomioon C#-kielen ominaisuudet ja käyttötavat, käytimme `List<string>` tallentaaksesi mukautettuja toimintasanakonteksteja:

```csharp
public class DoubaoVoiceConfig
{
    /// <summary>
    /// App ID
    /// </summary>
    public string AppId { get; set; } = string.Empty;

    /// <summary>
    /// Access token
    /// </summary>
    public string AccessToken { get; set; } = string.Empty;

    /// <summary>
    /// Service URL
    /// </summary>
    public string ServiceUrl { get; set; } = string.Empty;

    /// <summary>
    /// Custom hotword context list
    /// </summary>
    public List<string>? HotwordContexts { get; set; }

    /// <summary>
    /// Doubao platform hotword table ID
    /// </summary>

```

Tämän kokoonpanoluokan suunnittelu noudattaa HagiCoden tavanomaista tiivistä tyyliä. `HotwordContexts` on tyhjä listatyyppi, ja `BoostingTableId` on nollattavissa oleva merkkijono, joten kun hotword-määrityksiä ei ole, nämä ominaisuudet eivät vaikuta pyyntöön ollenkaan. Jos et käytä ominaisuutta, sen pitäisi pysyä poissa tieltä.

### Hyötykuorman rakentamisen logiikka

Hyötykuorman rakentaminen on koko hotword-ominaisuuden ydin. Kun hotword-määritykset on tehty, meidän on muotoiltava se Doubao API:n edellyttämään JSON-rakenteeseen. Tämä prosessi tapahtuu ennen kuin SDK lähettää pyynnön:

```csharp
private void AddCorpusToRequest(Dictionary<string, object> request)
{
    var corpus = new Dictionary<string, object>();

    // Add custom hotwords
    if (Config.HotwordContexts != null && Config.HotwordContexts.Count > 0)
    {
        corpus["context"] = new Dictionary<string, object>
        {
            ["context_type"] = "dialog_ctx",
            ["context_data"] = Config.HotwordContexts
                .Select(text => new Dictionary<string, object> { ["text"] = text })
                .ToList()
        };
    }

    // Add platform hotword table ID
    if (!string.IsNullOrEmpty(Config.BoostingTableId))
    {
        corpus["boosting_table_id"] = Config.BoostingTableId;
    }

    // Add corpus to the request only when it is not empty
    if (corpus.Count > 0)
    {
        request["corpus"] = corpus;
    }
}
```

Tämä koodi näyttää, kuinka dynaamisesti rakennetaan `corpus` kenttä kokoonpanon perusteella. Tärkeintä on, että lisäämme `corpus` kenttään vain, kun toimintosanan määritys on todella olemassa. Tämä rakenne varmistaa taaksepäin yhteensopivuuden: kun toimintosanoja ei ole määritetty, pyyntörakenne pysyy täsmälleen samana kuin ennen. Taaksepäin yhteensopivuus on tärkeää; ominaisuuden lisääminen ei saa häiritä olemassa olevaa logiikkaa.

### WebSocket-parametrin hyväksyminen

Käyttöliittymän ja taustajärjestelmän välillä toimintalauseparametrit välitetään WebSocket-ohjaussanomien kautta. HagiCode on suunniteltu niin, että kun käyttöliittymä aloittaa tallennuksen, se lataa hotword-määritykset alkaen `localStorage` ja lähettää sen taustajärjestelmään WebSocket-viestin kautta.

```typescript
const controlMessage = {
  type: 'control',
  payload: {
    command: 'StartRecognition',
    contextText: '高血压\n糖尿病\n冠心病',
    boosting_table_id: 'medical_table',
    combineMode: false
  }
};
```

Tässä on yksi huomioitava yksityiskohta: käyttöliittymä välittää monirivistä tekstiä rivinvaihtomerkeillä erotettuna, ja taustajärjestelmän on jäsennettävä se. WebSocket-taustakäsittelijä jäsentää nämä parametrit ja välittää ne SDK:lle:

```csharp
private async Task HandleControlMessageAsync(
    string connectionId,
    DoubaoSession session,
    ControlMessage message)
{
    if (message.Payload is SessionControlRequest controlRequest)
    {
        // Parse hotword parameters
        string? contextText = controlRequest.ContextText;
        string? boostingTableId = controlRequest.BoostingTableId;
        bool? combineMode = controlRequest.CombineMode;

        // Parse multi-line text into a hotword list
        if (!string.IsNullOrEmpty(contextText))
        {
            var hotwords = contextText
                .Split('\n', StringSplitOptions.RemoveEmptyEntries)
                .Select(s => s.Trim())
                .Where(s => s.Length > 0)
                .ToList();

            session.HotwordContexts = hotwords;
        }

        session.BoostingTableId = boostingTableId;
    }
}
```

Tämän suunnittelun avulla hotword-määrityksen siirtäminen käyttöliittymästä taustajärjestelmään tulee selkeäksi ja tehokkaaksi. Siinä ei ole mitään erityisen mystistä; tiedot yksinkertaisesti välitetään kerros kerrokselta.

## Pidä mielessä

On useita kohtia, jotka ansaitsevat erityistä huomiota hotword-ominaisuuden käyttöönotossa ja käytössä.

Ensimmäinen on merkkirajoitus. Doubao API:lla on tiukat rajoitukset hotwordille, mukaan lukien rivien määrä, merkit riviä kohden ja merkkien kokonaismäärä. Jos jokin raja ylittyy, API palauttaa virheilmoituksen. HagiCoden käyttöliittymätoteutuksessa tarkistamme nämä rajoitukset käyttäjän syöttämisen aikana vahvistustoimintojen avulla, mikä estää virheellisten konfiguraatioiden lähettämisen taustajärjestelmään. Ongelmien havaitseminen ajoissa on aina parempi kuin odottaa API:n epäonnistumista.

Toinen on muoto `boosting_table_id`. Tämä kenttä sallii vain kirjaimet, numerot, alaviivat ja yhdysmerkit, eikä se voi sisältää välilyöntejä tai muita erikoismerkkejä. Kun luot hotword-taulukkoa Doubao-itseoppivalla alustalla, muista noudattaa nimeämissääntöjä. Tällainen tiukka muototarkistus on yleistä API:ille.

Kolmas on taaksepäin yhteensopivuus. Hotword-parametrit ovat täysin valinnaisia. Jos toimintosanoja ei ole määritetty, järjestelmä toimii täsmälleen kuten ennen. Tämä suunnittelu varmistaa, että nykyiset käyttäjät eivät vaikuta millään tavalla, ja se myös helpottaa asteittaista siirtymistä ja päivityksiä. Ominaisuuden lisääminen ei saa häiritä aiempaa logiikkaa.

Lopuksi on virheiden käsittely. Kun hotword-määritys on virheellinen, Doubao API palauttaa vastaavat virheilmoitukset. HagiCoden toteutus tallentaa yksityiskohtaisia ​​lokeja auttaakseen kehittäjiä vianmäärityksessä. Samanaikaisesti käyttöliittymä näyttää käyttöliittymän vahvistusvirheet auttaakseen käyttäjiä korjaamaan kokoonpanon. Hyvä virheiden käsittely johtaa luonnollisesti parempaan käyttökokemukseen.

Tämä ratkaisu kattaa koko prosessin vaatimusanalyysistä ja teknisestä valinnasta koodin toteutukseen ja antaa kehittäjille käytännön esimerkin, jota he voivat käyttää viitteenä.

Keskeisimmät kohdat voidaan tiivistää seuraavasti. Ensinnäkin Doubao API tukee sekä mukautettuja toimintosanoja että alustan toimintosanataulukoita, ja niitä voidaan käyttää itsenäisesti tai yhdessä. Toiseksi käyttöliittymä käyttää `localStorage` tallentaa kokoonpanot yksinkertaisella ja tehokkaalla tavalla. Kolmanneksi taustaohjelma välittää hotword-parametrit rakentamalla dynaamisesti `corpus` kentässä säilyttäen vahvan taaksepäin yhteensopivuuden. Neljänneksi kattava validointilogiikka varmistaa määritysten oikeellisuuden ja välttää virheelliset pyynnöt. Kaiken kaikkiaan ratkaisu ei ole monimutkainen; se vain noudattaa tarkasti API-vaatimuksia.

Hotword-ominaisuuden käyttöönotto vahvistaa entisestään HagiCoden ominaisuuksia puheentunnistusalueella. Muokkaamalla joustavasti liiketoimintaan liittyviä ammattitermejä kehittäjät voivat auttaa puheentunnistusjärjestelmää ymmärtämään paremmin tiettyjen verkkotunnusten sisältöä ja siten tarjoamaan tarkempia palveluja. Viime kädessä teknologian tulee palvella todellisia liiketoiminnan tarpeita, ja käytännön ongelmien ratkaiseminen on tärkeintä.
