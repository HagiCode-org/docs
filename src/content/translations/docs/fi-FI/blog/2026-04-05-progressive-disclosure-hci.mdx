## Progressiivinen julkistaminen: Ihmisen ja tietokoneen vuorovaikutuksen parantaminen tekoälytuotteissa "Less Is More" -filosofian avulla

> Tekoälyn tuotesuunnittelussa käyttäjän syötteen laatu määrää usein tulosteen laadun. Tämä artikkeli jakaa "progressiivisen paljastamisen" vuorovaikutuslähestymistavan, jota harjoitimme HagiCode-projektissa. Vaiheittaisten ohjeiden, älykkään täydentämisen ja välittömän palautteen avulla se muuttaa käyttäjien lyhyet ja epämääräiset tiedot jäsennellyiksi teknisiksi ehdotuksiksi, mikä parantaa merkittävästi ihmisen ja tietokoneen vuorovaikutuksen tehokkuutta.

## Tausta

Jokainen tekoälytuotteita rakentava on luultavasti nähnyt tämän tilanteen: käyttäjä avaa sovelluksesi ja kirjoittaa innostuneesti yksirivisen pyynnön, jotta tekoäly palauttaa jotain täysin tavoitteen ulkopuolelta. Kyse ei ole siitä, että tekoäly ei olisi tarpeeksi älykäs. Käyttäjä ei yksinkertaisesti antanut tarpeeksi tietoa. Ajatusten lukeminen on vaikeaa kenellekään.

Tämä ongelma tuli erityisen ilmeiseksi HagiCodea rakentaessamme. HagiCode on tekoälyyn perustuva koodausavustaja, jossa käyttäjät kuvaavat vaatimuksia luonnollisella kielellä teknisten ehdotusten ja istuntojen luomiseksi. Varsinaisessa käytössä havaitsimme, että käyttäjän syötteellä on usein seuraavat ongelmat:

- **Epäjohdonmukainen syöttölaatu**: jotkut käyttäjät kirjoittavat vain muutaman sanan, kuten "optimoi kirjautuminen" tai "korjaa virhe", ilman tarvittavaa kontekstia
- **Epäjohdonmukainen tekninen terminologia**: eri käyttäjät käyttävät eri termejä samasta asiasta; toiset sanovat "frontend" kun taas toiset "FE"
- **Puuttuva jäsennelty tieto**: projektin taustaa, tietovaraston laajuutta tai vaikutusalaa ei ole, vaikka nämä ovatkin tärkeitä yksityiskohtia
- **Toistuvat ongelmat**: samantyyppisiä pyyntöjä tulee uudestaan ja uudestaan, ja joka kerta, kun ne täytyy selittää tyhjästä

Suora tulos on ennakoitavissa: tekoälyn on vaikeampi ymmärtää pyyntöä, ehdotuksen laatu muuttuu epävakaaksi ja käyttökokemus kärsii. Käyttäjät ajattelevat: "Tämä tekoäly ei ole kovin hyvä", kun taas meitä syytetään epäoikeudenmukaisesti. Jos annat minulle vain yhden lauseen, kuinka minun pitäisi arvata, mitä todella haluat?

Itse asiassa tämä on ymmärrettävää. Ihmisetkin tarvitsevat aikaa ymmärtääkseen toisiaan, eivätkä koneet ole poikkeus.

Näiden kipukohtien ratkaisemiseksi teimme rohkean päätöksen: otamme käyttöön "progressiivisen paljastamisen" suunnitteluperiaatteen parantaaksemme ihmisen ja tietokoneen välistä vuorovaikutusta. Muutokset, joita tämä toi, olivat luultavasti suurempia kuin voisi kuvitella. Rehellisesti sanottuna emme odottaneet sen olevan näin tehokas tuolloin.

## Mikä on progressiivinen paljastaminen

"Progressive disclosure" on suunnitteluperiaate HCI:n (ihmisen ja tietokoneen välisen vuorovaikutuksen) alalta. Sen ydinidea on yksinkertainen: **älä näytä käyttäjille kaikkia tietoja ja vaihtoehtoja kerralla. Sen sijaan paljasta vain se, mikä on välttämätöntä askel askeleelta, käyttäjän toimien ja tarpeiden mukaan**.

Tämä periaate sopii erityisen hyvin tekoälytuotteisiin, koska tekoälyn vuorovaikutus on luonnollisesti progressiivista. Käyttäjä sanoo vähän, tekoäly ymmärtää vähän, sitten käyttäjä lisää lisää ja tekoäly ymmärtää enemmän. Se on hyvin samanlaista kuin ihmisten kommunikointi keskenään: ymmärrys kehittyy yleensä vähitellen.

HagiCoden skenaariossa sovelsimme progressiivista julkistamista neljällä alueella:

### 1. Kuvauksen optimointimekanismi: anna tekoäly auttaa sinua sanomaan asiat selkeämmin

Kun käyttäjä kirjoittaa lyhyen kuvauksen, emme lähetä sitä suoraan tekoälylle tulkittavaksi. Sen sijaan käynnistämme ensin "kuvauksen optimoinnin". Tämän kulun ydin on "strukturoitu tulos": käyttäjän vapaan tekstin muuntaminen vakiomuotoon. Se on kuin irtonaisten helmien pujottamista kaulakoruun, jolloin kaikki on helpompi ymmärtää.

Optimoidun kuvauksen tulee sisältää seuraavat vakioosiot:

- **Tausta**: ongelman tausta ja konteksti
- **Analyysi**: tekninen analyysi ja perustelut
- **Ratkaisu**: ratkaisu ja toteutusvaiheet
- **Harjoittelu**: konkreettisia koodiesimerkkejä ja huomautuksia

Samanaikaisesti luomme automaattisesti Markdown-taulukon, joka näyttää tiedot, kuten kohdetietovaraston, polut ja muokkausoikeudet, mikä helpottaa myöhempiä tekoälytoimintoja. Selkeä hakemisto helpottaa aina asioiden löytämistä.

Tässä varsinainen toteutus:

```csharp
// Core method in ProposalDescriptionMemoryService.cs
public async Task<string> OptimizeDescriptionAsync(
    string title,
    string description,
    string locale = "zh-CN",
    DescriptionOptimizationMemoryContext? memoryContext = null,
    CancellationToken cancellationToken = default)
{
    // Build query parameters
    var queryContext = BuildQueryContext(title, description);

    // Retrieve historical context
    var memoryContext = await RetrieveHistoricalContextAsync(queryContext, cancellationToken);

    // Generate a structured prompt
    var prompt = await BuildOptimizationPromptAsync(
        title,
        description,
        memoryContext,
        cancellationToken);

    // Call AI for optimization
    return await _aiService.CompleteAsync(prompt, cancellationToken);
}
```

Avain tähän virtaukseen on "muistiinjektio". Lisäämme kehotteeseen historiallisen kontekstin, kuten projektikonventioita, vastaavia tapauksia ja negatiivisia kuvioita, jolloin tekoäly voi viitata aiempaan kokemukseen optimoinnin aikana. Kokemusta ei pidä mennä hukkaan.

**Huomioitavaa:**
- Varmista, että nykyinen syöte on etusijalla historialliseen muistiin nähden, jotta eksplisiittisesti määritettyjä käyttäjätietoja ei ohiteta
- HagIndex-viittauksia on käsiteltävä asialähteinä, eivätkä historialliset tapaukset saa muuttaa niitä
- Alhaisen luotettavuuden korjausehdotuksia ei pidä antaa vahvoina rajoituksina

### 2. Äänisyöttökyky: puhuminen on luonnollisempaa kuin kirjoittaminen

Tekstinsyötön lisäksi tuemme myös äänisyöttöä. Tämä on erityisen hyödyllistä monimutkaisten vaatimusten kuvauksessa. Teknisen pyynnön kirjoittaminen voi kestää minuutteja, kun taas sen ääneen sanominen voi kestää vain muutaman kymmenen sekunnin.

Äänisyötteen suunnittelun tärkein painopiste on "tilanhallinta". Käyttäjien on tiedettävä selvästi, missä tilassa järjestelmä tällä hetkellä on. Määritimme seuraavat tilat:

- **Idle**: järjestelmä on valmis ja tallennus voi alkaa
- **Odottaa ylävirtaa**: järjestelmä muodostaa yhteyttä taustapalveluun
- **Tallennus**: käyttäjän ääntä tallennetaan
- **Käsittelyssä**: puhetta muunnetaan tekstiksi
- **Virhe**: tapahtui virhe, joka vaatii käyttäjän huomiota

Käyttöliittymän tilamalli näyttää suunnilleen tältä:

```typescript
interface VoiceInputState {
  status: 'idle' | 'waiting-upstream' | 'recording' | 'processing' | 'error';
  duration: number;
  error?: string;
  deletedSet: Set<string>; // Fingerprint set of deleted results
}

// State transition when recording starts
const handleVoiceInputStart = async () => {
  // Enter waiting state first and show a loading animation
  setState({ status: 'waiting-upstream' });

  // Wait for backend readiness confirmation
  const isReady = await waitForBackendReady();
  if (!isReady) {
    setState({ status: 'error', error: 'Backend service is not ready' });
    return;
  }

  // Start recording
  setState({ status: 'recording', startTime: Date.now() });
};

// Handle recognition results
const handleRecognitionResult = (result: RecognitionResult) => {
  const fingerprint = normalizeFingerprint(result.text);

  // Check whether it has already been deleted
  if (state.deletedSet.has(fingerprint)) {
    return; // Skip deleted content
  }

  // Merge the result into the text box
  appendResult(result);
};
```

Tässä on tärkeä yksityiskohta: käytämme "sormenjälkijoukkoa" poistojen synkronoinnin hallintaan. Kun puheentunnistus palauttaa useita tuloksia, käyttäjät voivat poistaa osan niistä. Tallennamme poistetun sisällön sormenjäljet ​​niin, että jos sama sisältö ilmestyy myöhemmin uudelleen, se ohitetaan automaattisesti. Se on pohjimmiltaan tapa muistaa, mitä käyttäjä on jo hylännyt.

### 3. Nopea hallintajärjestelmä: ulkoista tekoälyn "aivot"

HagiCodella on joustava kehotteiden hallintajärjestelmä, jossa kaikki kehotteet tallennetaan tiedostoina:

```
prompts/
├── metadata/
│   ├── optimize-description.zh-CN.json
│   └── optimize-description.en-US.json
└── templates/
    ├── optimize-description.zh-CN.hbs
    └── optimize-description.en-US.hbs
```

Jokainen kehote koostuu kahdesta osasta:
- **Metatietotiedosto (`.json`)**: määrittää tiedot, kuten kehotteen skenaarion, version ja parametrit
- **Mallitiedosto (`.hbs`)**: varsinainen kehotesisältö, joka on kirjoitettu Handlebars-syntaksilla

Metatietojen tiedostomuoto näyttää tältä:

```json
{
  "scenario": "optimize-description",
  "locale": "zh-CN",
  "version": "1.0.0",
  "syntax": "handlebars",
  "syntaxVersion": "1.0",
  "parameters": [
    {
      "name": "title",
      "type": "string",
      "required": true,
      "description": "Proposal title"
    },
    {
      "name": "description",
      "type": "string",
      "required": true,
      "description": "Original description"
    }
  ],
  "author": "HagiCode Team",
  "description": "Optimize the user's technical proposal description",
  "lastModified": "2026-04-05",
  "tags": ["optimization", "nlp"]
}
```

Mallitiedosto käyttää Handlebars-syntaksia ja tukee parametrien lisäystä:

```handlebars
You are a technical proposal expert.

<task>
Generate a structured technical proposal description based on the following information.
</task>

<input>
<title>{{title}}</title>
<description>{{description}}</description>
{{#if memoryContext}}
<memory_context>
{{memoryContext}}
</memory_context>
{{/if}}
</input>

<output_format>
## Background
[Describe the problem background and context, including project information, repository scope, and so on]

## Analysis
[Provide the technical analysis and reasoning process, and explain why this change is needed]

## Solution
[Provide the solution and implementation steps, listing the key code locations]

## Practice
[Provide concrete code examples and notes]
</output_format>
```

Tämän suunnittelun edut ovat selvät:
- kehotteita voidaan versioohjata aivan kuten koodia
- useita kieliä tuetaan, ja ne voidaan vaihtaa automaattisesti käyttäjän mieltymysten mukaan
- parametrisoitu suunnittelu mahdollistaa kontekstin lisäämisen dynaamisesti
- täydellisyys voidaan tarkistaa käynnistyksen yhteydessä, jolloin vältetään ajonaikaiset virheet

Jos tieto jää vain jonkun päähän, se on helppo menettää. Sen tallentaminen jäsennellysti alusta alkaen on paljon turvallisempaa.

### 4. Progressiivinen ohjattu toiminto: jaa monimutkaiset tehtävät pieniin vaiheisiin

Monimutkaisiin tehtäviin, kuten ensiasennukseen ja konfigurointiin, käytämme monivaiheista ohjattua suunnittelua. Jokainen vaihe pyytää vain tarvittavat tiedot ja tarjoaa selkeät edistymisindikaattorit. Suurista tehtävistä tulee paljon helpommin hallittavissa, kun niitä käsitellään vaihe kerrallaan.

Ohjatun tilan malli:

```typescript
interface WizardState {
  currentStep: number;           // 0-3, corresponding to 4 steps
  steps: WizardStep[];
  canGoNext: boolean;
  canGoBack: boolean;
  isLoading: boolean;
  error: string | null;
}

interface WizardStep {
  id: number;
  title: string;
  description: string;
  completed: boolean;
}

// Step navigation logic
const goToNextStep = () => {
  if (wizardState.currentStep < wizardState.steps.length - 1) {
    // Validate input for the current step
    if (validateCurrentStep()) {
      wizardState.currentStep++;
      wizardState.steps[wizardState.currentStep - 1].completed = true;
    }
  }
};

const goToPreviousStep = () => {
  if (wizardState.currentStep > 0) {
    wizardState.currentStep--;
  }
};
```

Jokaisella askeleella on oma validointilogiikkansa, ja suoritetut vaiheet saavat selkeät visuaaliset merkit. Peruutus avaa vahvistusikkunan, joka estää käyttäjiä menettämästä edistymistä tahattomien toimien vuoksi.

## Johtopäätös

Kun tarkastellaan HagiCoden progressiivista tiedonantokäytäntöä, voimme tehdä yhteenvedon useista keskeisistä periaatteista:

1. **Vaiheittainen opastus**: jaa monimutkaiset tehtävät pienempiin vaiheisiin ja pyydä vain tarvittavat tiedot kussakin vaiheessa
2. **Älykäs viimeistely**: käytä historiallista kontekstia ja projektitietoa tietojen täyttämiseen automaattisesti
3. **Pikapalaute**: anna jokaisesta toiminnasta selkeää visuaalista palautetta ja tilavinkkejä
4. **Vikasietomekanismit**: antavat käyttäjien kumota ja nollata, jotta virheet eivät johda peruuttamattomiin menetyksiin
5. **Syötteen monimuotoisuus**: tukee useita syöttötapoja, kuten tekstiä ja ääntä

HagiCodessa tämän lähestymistavan käytännön tulos oli selvä: käyttäjän syöttämien keskimääräinen pituus kasvoi alle 20 merkistä 200-300 merkin strukturoituihin kuvauksiin, tekoälyn luomien ehdotusten laatu parani merkittävästi ja käyttäjien tyytyväisyys kasvoi sen mukana.

Tämä ei ole yllättävää. Mitä enemmän tietoa käyttäjät tarjoavat, sitä tarkemmin tekoäly voi ymmärtää ne ja sitä parempia tuloksia se voi tuottaa. Siinä mielessä se ei juuri eroa ihmisten välisestä kommunikaatiosta.

Jos olet myös rakentamassa tekoälyyn liittyviä tuotteita, toivon, että nämä kokemukset tarjoavat hyödyllistä inspiraatiota. Muista: käyttäjät eivät välttämättä kieltäydy antamasta tietoja. Useammin tuote ei ole vielä esittänyt oikeita kysymyksiä oikealla tavalla. Progressiivisen paljastamisen ydin on löytää paras ajoitus ja muoto näille kysymyksille.
