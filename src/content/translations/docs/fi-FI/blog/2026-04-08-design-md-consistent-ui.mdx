## Design.md: Ratkaisu johdonmukaiseen tekoälyyn perustuvaan käyttöliittymäsuunnitteluun

> Kuinka voit pitää tekoälyn luomat käyttöliittymät johdonmukaisina tekoälyn tukeman käyttöliittymän kehityksen aikakaudella? Tämä artikkeli jakaa käytännön kokemuksemme suunnittelugalleriasivuston rakentamisesta awesome-design-md:n pohjalta sekä strukturoidun `design.md` ohjaamaan tekoälyä kohti standardoitua käyttöliittymäsuunnittelua.

## Tausta

Kaikilla, jotka ovat käyttäneet tekoälyä käyttöliittymäkoodin kirjoittamiseen, on todennäköisesti ollut samanlainen kokemus: pyydä tekoälyä luomaan sama sivu useita kertoja, ja jokainen tulos tulee eri tyyliin. Joskus kulmat ovat pyöristettyjä, joskus teräviä. Joskus väli on 8 kuvapistettä ja toisinaan 16 kuvapistettä. Jopa sama painike voi näyttää erilaiselta eri keskusteluissa.

Tämä ei ole yksittäinen ongelma. Tekoälyavusteisen kehityksen yleistyessä **yhdenmukaisuuden puutteesta tekoälyn luomassa käyttöliittymässä** on tullut laajalle levinnyt ongelma. Erilaiset tekoälyavustajat, erilaiset kehotteet ja jopa sama avustaja eri keskusteluissa voivat tuottaa dramaattisesti erilaisia ​​käyttöliittymämalleja. Tämä aiheuttaa valtavia ylläpitokustannuksia tuotteen iteroinnin aikana.

Perimmäinen syy on itse asiassa yksinkertainen: ei ole olemassa **virallista suunnittelun viiteasiakirjaa**. Perinteiset CSS-tyylitaulukot voivat kertoa kehittäjille "miten toteuttaa" jotain, mutta he eivät voi täysin kommunikoida "miksi se on suunniteltu tällä tavalla" tai "mitä suunnittelumallia tulisi käyttää missäkin skenaariossa". Tekoälylle selkeä ja jäsennelty kuvaus on vielä tärkeämpää suunnittelukäytäntöjen ymmärtämisen kannalta.

Samaan aikaan avoimen lähdekoodin yhteisö tarjoaa jo nyt erinomaisia ​​resursseja. The [VoltAgent/awesome-design-md](https://github.com/VoltAgent/awesome-design-md) projekti kerää suunnittelujärjestelmädokumentaatiota monilta tunnetuilta yrityksiltä. Jokainen hakemisto sisältää a `README.md`, a `DESIGN.md`ja esikatsele HTML-koodia. Kaikki tämä on kuitenkin hajallaan ylävirran arkistoon, mikä vaikeuttaa nopeaa selaamista ja vertailua.

Voimmeko siis yhdistää nämä resurssit helposti selaatavaksi suunnittelugalleriaksi ja samalla tislata jäsenneltyä `design.md` tekoälyn käyttöön?

Vastaus on kyllä. Seuraavaksi käyn läpi lähestymistapamme.

## Analyysi

Ennen kuin kirjoitat koodia, selvitämme tämän ongelman taustalla olevat tekniset haasteet.

### Lähdesisällön hallinta: Kuinka yhdistät hajallaan olevat suunnitteluresurssit?

Ylävirran awesome-design-md-arkisto sisältää suuren määrän suunnitteludokumentteja, mutta tarvitsimme tavan tuoda ne omaan projektiimme.

**Ratkaisu: käytä git-alimoduulia**

```
awesome-design-md-site
└── vendor/awesome-design-md          # Upstream resources (git submodule)
```

Tämä antaa meille useita etuja:
- Versionhallinta: voimme kiinnittää tietyn ylävirran version
- Offline-koontiversiot: ulkoisia sovellusliittymiä ei tarvitse pyytää rakentamisen aikana
- Sisällön tarkistus: tietyt muutokset näkyvät PR:ssa

### Tietojen normalisointi: Kuinka standardisoit erilaisia asiakirjarakenteita?

Eri yritykset rakentavat suunnitteluasiakirjansa eri tavalla. Joiltakin puuttuu esikatselutiedostoja, ja joissakin käytetään epäjohdonmukaista nimeämistä. Meidän on normalisoitava ne rakennusprosessin aikana.

**Ratkaisu: skannaa ja luo normalisoituja merkintöjä rakennusaikana**

Ydinmoduuli on `awesomeDesignCatalog.ts`, vastaa:
1. Skannaus `vendor/awesome-design-md/design-md/*` hakemistosta
2. Tarkistaa, sisältääkö jokainen merkintä vaaditut tiedostot (`README.md`, `DESIGN.md`ja vähintään yksi esikatselutiedosto)
3. Markdown-sisällön purkaminen ja muuttaminen HTML-muotoon
4. Normalisoitujen syöttötietojen luominen

### Staattinen sivustoarkkitehtuuri: Miten tarjoat dynaamisen haun pysyen täysin staattisena?

Koska tämä on suunnittelugalleria, haku on pakollinen. Mutta Astro on staattinen sivustogeneraattori, joten miten toteutat reaaliaikaisen haun?

**Ratkaisu: Reaktiosaareke + URL-kyselyparametrin synkronointi**

```typescript
// src/components/gallery/SearchToolbar.tsx

export function SearchToolbar() {
  const [query, setQuery] = useState('');

  // Sync with the URL
  useEffect(() => {
    const params = new URLSearchParams(window.location.search);

```

Tämän lähestymistavan etuna on, että se pitää staattisen sivuston käyttöönoton ennallaan, mikä tarkoittaa, että se voidaan ottaa käyttöön missä tahansa staattisessa isännöintipalvelussa, samalla kun se tarjoaa välittömän suodatuskokemuksen.

### Suunnitteludokumentaatio: Kuinka autat tekoälyä ymmärtämään ja noudattamaan suunnittelustandardeja?

Tämä on koko ratkaisun ydin. Meidän on luotava jäsennelty `design.md` jonka tekoäly voi ymmärtää ja soveltaa.

**Ratkaisu: lainaa ClickHousen rakennetta `DESIGN.md`**

[ClickHouse's `DESIGN.md`](https://github.com/ClickHouse/ClickHouse/blob/master/DESIGN.md) on erinomainen referenssi. Se sisältää:
- Visuaalinen teema ja tunnelma
- Väripaletti ja roolit
- Typografian säännöt
- Komponenttien tyylit
- Asetteluperiaatteet
- Syvyys ja korkeus
- Tee ja kiellot
- Responsiivinen käyttäytyminen
- Agenttikehoteopas

Lähestymistapamme on: **käytä rakennetta, kirjoita sisältö uudelleen**. Säilytämme ClickHousen osiorakenteen `DESIGN.md`, mutta korvaa sisältö todellisilla suunnittelutunnuksilla ja komponenttikonventioilla, joita käytetään omassa projektissamme.

## Ratkaisu

Yllä olevan analyysin perusteella ratkaisumme koostuu neljästä ydinmoduulista.

### 1. Sisällön käsittelyputki

Tämä on koko järjestelmän perusta, joka vastaa sisällön poimimisesta ja normalisoinnista alkupään resursseista.

```typescript
// src/lib/content/awesomeDesignCatalog.ts

export async function scanSourceEntries(): Promise<DesignEntry[]> {
  const designDir = 'vendor/awesome-design-md/design-md';
  const entries: DesignEntry[] = [];

  for (const dir of await fs.readdir(designDir)) {
    const entryPath = path.join(designDir, dir);
    if (await isValidDesignEntry(entryPath)) {
      const entry = await normalizeDesignEntry(entryPath);
      entries.push(entry);
    }
  }

  return entries;
}

async function isValidDesignEntry(dir: string): Promise<boolean> {
  const requiredFiles = ['README.md', 'DESIGN.md'];
  for (const file of requiredFiles) {
    if (!(await fileExists(path.join(dir, file)))) {
      return false;
    }
  }
  return true;
}
```

### 2. Gallerian selausliittymä

Gallerian käyttöliittymä sisältää kolme pääosaa:

**Kotisivu**: näyttää korttiruudukon kaikista suunnittelumerkinnöistä, ja jokainen kortti sisältää:
- Suunnittelutyön otsikko ja yhteenveto
- Esikatselukuva, jos saatavilla
- Pikahaun korostus

**Tietosivu**: kokoaa yhteen suunnittelumerkinnän täydelliset tiedot:
- `README` asiakirja
- `DESIGN` asiakirja
- Esikatselu vaalean/tumman teeman vaihdolla
- Navigointi viereisiin merkintöihin

**Navigointi**: tukee palaamista galleriaan ja viereisten merkintöjen selaamista

Kotisivugalleriassa käytetään tiheää korttiasettelua, litistystä `design.md` merkinnät eri lähteistä yhtenäiseen visuaaliseen kehykseen, jotta tiimit voivat nopeasti vertailla brändityylejä, painikekuvioita ja typografista rytmiä.

Tietyn merkinnän avaamisen jälkeen lisätietosivu sijoittaa suunnittelun yhteenvedon ja live-esikatselun samalle sivulle, mikä vähentää kustannuksia, jotka aiheutuvat vaihtamisesta edestakaisin dokumentaation, esikatselun ja lähdekoodin välillä.

### 3. Hakutoiminnot

Hakuominaisuus perustuu asiakaspuolen suodatukseen, jonka tila säilytetään URL-kyselyparametrien avulla:

```typescript
// src/components/gallery/SearchToolbar.tsx

function SearchToolbar({ entries }: { entries: DesignEntry[] }) {
  const [query, setQuery] = useState('');
  const [results, setResults] = useState(entries);

  useEffect(() => {
    const params = new URLSearchParams(window.location.search);
    const initialQuery = params.get('q') || '';
    setQuery(initialQuery);
    filterEntries(initialQuery);
  }, []);

  const filterEntries = (searchQuery: string) => {
    const filtered = entries.filter(entry =>
      entry.searchText.toLowerCase().includes(searchQuery.toLowerCase())
    );
    setResults(filtered);
  };

  const handleChange = (e: React.ChangeEvent<HTMLInputElement>) => {
    const value = e.target.value;
    setQuery(value);
    filterEntries(value);

    // Update the URL without triggering a page refresh
    const newUrl = value
      ? `${window.location.pathname}?q=${encodeURIComponent(value)}`
      : window.location.pathname;
    window.history.replaceState({}, '', newUrl);
  };

  return (
    <div className="search-toolbar">
      <input
        type="text"
        value={query}
        onChange={handleChange}
        placeholder="Search design entries..."
      />
      <span className="result-count">{results.length} results</span>
    </div>
  );
}
```

## Harjoittele

Nyt kun olemme käsitelleet ratkaisun, miten toteutat sen?

### Toteutusvaiheet

**Vaihe 1: Alusta alimoduuli**

```bash
# Add the upstream repository as a submodule
git submodule add https://github.com/VoltAgent/awesome-design-md.git vendor/awesome-design-md

# Initialize and update the submodule
git submodule update --init --recursive
```

**Vaihe 2: Luo sisältöputki**

Toteuta `awesomeDesignCatalog.ts`, mukaan lukien:
- Tiedostojen skannaus ja validointilogiikka
- Markdown-renderöinti Astron sisäänrakennetulla renderöijalla
- Syöttötietojen poiminta

**Vaihe 3: Luo gallerian käyttöliittymä**

Käytä Astro + React Islandsia luodaksesi:
- Kotisivun gallerian asettelu (korttiruudukko)
- Suunnittelukorttikomponentit
- Hakutyökalupalkki
- Yksityiskohtaisten sivujen asettelu

**Vaihe 4: Kirjoita suunnitteluasiakirja**

Perustuu ClickHousen rakenteeseen `DESIGN.md`, täytä varsinaiset suunnittelutunnukset omasta projektistasi. Päivitä `README.md` ja lisää linkki `design.md`.

### Huomautuksia

**Turvallisuus**: Markdown-renderöinti edellyttää vaarallisen HTML-koodin suodattamista. Astron sisäänrakennetut renderöintisuodattimet `script` tunnisteet oletuksena, mutta sinun on silti varottava XSS-riskejä.

**Suorituskyky**: Suuri määrä iframe-esikatseluita voi vaikuttaa ensimmäisen maalauksen suorituskykyyn. On suositeltavaa käyttää `loading="lazy"` esikatselun sisällön laiskaan lataamiseen.

**Ylläpidettävyys**: `design.md` on pysyttävä synkronoituna koodin toteutuksen kanssa. On suositeltavaa lisätä CI-tarkistuksia sen varmistamiseksi, että CSS-muuttujat pysyvät johdonmukaisina dokumentaation ja koodin välillä.

**Käytettävyys**: Varmista, että värikontrasti täyttää WCAG AA -standardin (vähintään 4,5:1).

### AI-käyttöopas

Luomisen jälkeen `design.md`, miten saat tekoälyn käyttämään sitä? Tässä muutamia käytännön vinkkejä:

**Vinkki 1: viittaa siihen nimenomaisesti kehotteessa**

```
Please refer to the design.md file in the project root and use the design conventions defined there to implement the following components:
- Buttons: use the primary color with an 8px border radius
- Cards: use the elevation-2 shadow level
```

**Vinkki 2: vaadi tekoälyn viittaavan tiettyihin CSS-muuttujiin**

```
Implement a navigation bar with the following requirements:
- Use --color-bg-primary for the background color
- Use --color-border-subtle for borders
- Use --text-color-primary for text
```

**Vinkki 3: Sisällytä `design.md` sisältö järjestelmäkehotteessa**

Jos tekoälytyökalusi tukee mukautettuja järjestelmäkehotteita, voit lisätä sen ydinsisällön `design.md` suoraan siihen.

### Testausstrategia

**Sisällön testaus**:
- Puuttuvien tiedostojen skenaariot (puuttuu `README.md` tai `DESIGN.md`)
- Muotoiluvirheskenaariot (Markdown jäsennysvirhe)
- Tyhjän hakemiston skenaariot

**Hakuominaisuuksien testaus**:
- Tyhjä tuloskäsittely
- Erikoismerkit, kuten kiina ja emoji
- URL-osoitteen synkronoinnin vahvistus

**Käyttöliittymäkomponenttien testaus**:
- Vaalean/tumman teeman vaihto
- Responsiivinen asettelu
- Esikatsele lataustilat

### Käyttöönoton työnkulku

```bash
# 1. Update the submodule to the latest version
git submodule update --remote

# 2. Rebuild the site
npm run build

# 3. Deploy static assets
npm run deploy
```

On suositeltavaa automatisoida alimoduulipäivitykset sekä rakentaminen ja käyttöönotto, jotta CI voidaan käynnistää automaattisesti aina, kun ylävirran arkisto päivitetään.

## Johtopäätös

Tekoälyn luomien käyttöliittymien epäjohdonmukaisuus, jonka HagiCode kohtasi kehitystyön aikana, johtui ennen kaikkea jäsennellyn suunnitteluviiteasiakirjan puutteesta. Rakentamalla suunnittelugalleriasivusto ja luomalla standardoitu `design.md`, ratkaisimme tämän ongelman onnistuneesti.

Tämän ratkaisun ydinarvo on:
- **Yhdistetyt resurssit**: hajallaan olevan suunnittelujärjestelmän dokumentaation yhdistäminen
- **Strukturoidut standardit**: suunnittelun käytäntöjen ilmaiseminen tekoälyn ymmärtämässä muodossa
- **Jatkuva ylläpito**: sisällön pitäminen ajan tasalla `git submodule`

Jos käytät tekoälyä myös käyttöliittymän kehittämiseen, tätä lähestymistapaa kannattaa kokeilla. Rakenteen luominen `design.md` ei vain paranna tekoälyn luoman koodin johdonmukaisuutta, vaan myös auttaa tiimiäsi ylläpitämään yhtenäisiä suunnittelustandardeja sisäisesti.
