## CLI-kutsuista SDK-integraatioon: GitHub Copilotin parhaat käytännöt .NET-projekteissa

> Päivityspolku komentorivin kutsusta viralliseen SDK-integraatioon on ollut melkoinen matka. Tänään haluan jakaa sudenkuopat, joihin törmäsimme ja mitä opimme rakentaessamme HagiCodea.

## Tausta

Kun GitHub Copilot SDK julkaistiin virallisesti vuonna 2025, aloimme integroida sen tekoälykykykerrokseen. Ennen sitä projekti käytti pääasiassa GitHub Copilotia kutsumalla suoraan Copilot CLI -komentorivityökaluun, mutta tällä lähestymistavalla oli useita ilmeisiä ongelmia:

- **Monimutkainen prosessinhallinta**: Meidän piti hallita manuaalisesti CLI-prosessin elinkaarta, käynnistyksen aikakatkaisuja ja prosessien puhdistamista. Prosessit voivat kaatua ilman varoitusta.
- **Epätäydellinen tapahtumien käsittely**: Raaka CLI-kutsu vaikeuttaa hienorakeisten tapahtumien sieppaamista mallin päättelystä ja työkalun suorittamisesta. Se on kuin näkisi vain tuloksen ilman ajatteluprosessia.
- **Vaikea istunnon hallinta**: Istunnon uudelleenkäytölle ja palautukselle ei ollut tehokasta mekanismia, joten jokainen vuorovaikutus oli aloitettava alusta.
- **Yhteensopivuusongelmat**: CLI-argumentit vaihtuivat usein, mikä tarkoitti, että meidän oli jatkuvasti ylläpidettävä näiden parametrien yhteensopivuuslogiikkaa.

Nämä ongelmat tulivat yhä selvemmiksi päivittäisessä kehityksessä, varsinkin kun meidän piti seurata mallin päättelyä (`thinking`) ja työkalun suoritustila reaaliajassa. Siinä vaiheessa kävi selväksi, että tarvitsemme alemman tason ja täydellisemmän integraatiolähestymistavan.

## Tietoja HagiCodesta

Tässä artikkelissa jaettu lähestymistapa perustuu käytännön kokemukseemme [HagiCode](https://hagicode.com) hanke. HagiCode on avoimen lähdekoodin tekoälyn koodausassistenttiprojekti, ja kehitysvaiheessa tarvitsimme syvällistä integraatiota GitHub Copilot -ominaisuuksiin peruskoodin viimeistelystä monimutkaisiin monikäännöksiin keskusteluihin ja työkalukutsuihin. Nämä todelliset vaatimukset saivat meidät siirtymään CLI-kutsusta viralliseen SDK:han.

Jos tämä toteutus kuulostaa hyödylliseltä, on hyvä mahdollisuus, että insinöörikokemuksemme voi auttaa. Siinä tapauksessa itse HagiCode-projekti saattaa olla myös tutustumisen arvoinen. Saatat jopa löytää lisää projektitietoja ja linkkejä tämän artikkelin lopusta.

## Arkkitehtuurisuunnittelu

Projekti käyttää kerrostettua arkkitehtuuria puuttuakseen CLI-kutsujen rajoituksiin:

```
┌─────────────────────────────────────────────────────────┐
│  hagicode-core (Orleans Grains + AI Provider Layer)    │
│  - CopilotAIProvider: Converts AIRequest to CopilotOptions │
│  - GitHubCopilotGrain: Orleans distributed execution interface │
└─────────────────────────────────────────────────────────┘
                          ↓
┌─────────────────────────────────────────────────────────┐
│  HagiCode.Libs (Shared Provider Layer)                 │
│  - CopilotProvider: CLI Provider interface implementation │
│  - ICopilotSdkGateway: SDK invocation abstraction      │
│  - GitHubCopilotSdkGateway: SDK session management and event dispatch │
└─────────────────────────────────────────────────────────┘
                          ↓
┌─────────────────────────────────────────────────────────┐
│  GitHub Copilot SDK (Official .NET SDK)                │
│  - CopilotClient: SDK client                           │
│  - CopilotSession: Session management                  │
│  - SessionEvent: Event stream                          │
└─────────────────────────────────────────────────────────┘
```

Tämä kerrostettu muotoilu tuo useita käytännön teknisiä etuja:

1. **Huolenaiheiden erottaminen**: Ydinliiketoiminnan logiikka on irrotettu SDK:n toteutustiedoista.
2. **Testattavuus**: `ICopilotSdkGateway` käyttöliittymä tekee yksikkötestauksesta suoraviivaista.
3. **Uudelleenkäytettävyys**: `HagiCode.Libs` voidaan viitata useisiin projekteihin.
4. **Ylläpidettävyys**: SDK-päivitykset vaativat muutoksia vain yhdyskäytävätasoon, kun taas ylemmät tasot pysyvät koskemattomina.

## Ydintoteutus

### Todennuskulku

Todennus on ensimmäinen ja tärkein vaihe SDK-integraatiossa. Jos todennus epäonnistuu, millään muulla ei ole väliä. Suunnittelimme joustavan todennuskonfiguraation, joka tukee useita todennuslähteitä:

```csharp
// CopilotProvider.cs - Authentication source configuration
public class CopilotOptions
{
    public bool UseLoggedInUser { get; set; } = true;
    public string? GitHubToken { get; set; }
    public string? CliUrl { get; set; }
}

// Convert to SDK request
return new CopilotSdkRequest(
    GitHubToken: options.AuthSource == CopilotAuthSource.GitHubToken
        ? options.GitHubToken
        : null,
    UseLoggedInUser: options.AuthSource != CopilotAuthSource.GitHubToken
);
```

Tämän suunnittelun edut ovat melko selvät:

- Se tukee kirjautuneen käyttäjän tilaa ilman tunnuksen tarvetta, mikä sopii hyvin työpöytäskenaarioihin.
- Se tukee GitHub Token -tilaa, joka sopii palvelinpuolen käyttöönotuksiin ja keskitettyyn hallintaan.
- Se tukee Copilot CLI URL -osoitteen ohittamista, mikä auttaa yrityksen välityspalvelimen määrityksessä.

Käytännössä tämä joustava todennusmalli yksinkertaisti huomattavasti konfigurointia eri käyttöönottoskenaarioissa. Työpöytäasiakas voi käyttää kunkin käyttäjän omaa Copilot-kirjautumistilaa, kun taas palvelinpuoli voi hallita todentamista keskitetysti tokenien avulla.

### Tapahtumastriimin käsittely

Yksi SDK:n tehokkaimmista ominaisuuksista on täydellinen tapahtumavirran sieppaus. Otimme käyttöön tapahtumanvälitysjärjestelmän, joka pystyy käsittelemään kaikenlaisia SDK-tapahtumia reaaliajassa:

```csharp
// GitHubCopilotSdkGateway.cs - Core logic for event dispatch
internal static SessionEventDispatchResult DispatchSessionEvent(
    SessionEvent evt, bool sawDelta)
{
    switch (evt)
    {
        case AssistantReasoningEvent reasoningEvent:
            // Capture the model reasoning process
            events.Add(new CopilotSdkStreamEvent(
                CopilotSdkStreamEventType.ReasoningDelta,
                Content: reasoningEvent.Data.Content));
            break;

        case ToolExecutionStartEvent toolStartEvent:
            // Capture the start of a tool invocation
            events.Add(new CopilotSdkStreamEvent(
                CopilotSdkStreamEventType.ToolExecutionStart,
                ToolName: toolStartEvent.Data.ToolName,
                ToolCallId: toolStartEvent.Data.ToolCallId));
            break;

        case ToolExecutionCompleteEvent toolCompleteEvent:
            // Capture tool completion and its result
            events.Add(new CopilotSdkStreamEvent(
                CopilotSdkStreamEventType.ToolExecutionEnd,
                Content: ExtractToolExecutionContent(toolCompleteEvent)));
            break;

        default:
            // Preserve unhandled events as RawEvent
            events.Add(new CopilotSdkStreamEvent(
                CopilotSdkStreamEventType.RawEvent,
                RawEventType: evt.GetType().Name));
            break;
    }
}
```

Tämän toteutuksen arvo on merkittävä:

- **Mallin päättelyprosessin täydellinen taltiointi** (`thinking`): käyttäjät näkevät, kuinka tekoäly perustelee, ei vain lopullista vastausta.
- **Työkalun suoritustilan reaaliaikainen seuranta**: tiedämme, mitkä työkalut ovat käynnissä, milloin ne valmistuvat ja mitä ne palauttavat.
- **Nolla tapahtumahäviö**: palaamalla `RawEvent`, varmistamme, että jokainen tapahtuma tallennetaan.

HagiCodessa nämä hienorakeiset tapahtumat auttavat käyttäjiä ymmärtämään, kuinka tekoäly toimii sisäisesti, etenkin kun suoritetaan monimutkaisten tehtävien virheenkorjausta.

### CLI-yhteensopivuuden käsittely

Siirtyessämme CLI-kutsusta SDK:hon havaitsimme, että jotkin nykyiset CLI-parametrit eivät enää olleet voimassa SDK:ssa. Taaksepäin yhteensopivuuden säilyttämiseksi otimme käyttöön parametrien suodatusjärjestelmän:

```csharp
// CopilotCliCompatibility.cs - Argument filtering
private static readonly Dictionary<string, string> RejectedFlags = new()
{
    ["--headless"] = "Unsupported startup argument",
    ["--model"] = "Passed through an SDK-native field",
    ["--prompt"] = "Passed through an SDK-native field",
    ["--interactive"] = "Interaction is managed by the provider",
};

public static CopilotCliArgumentBuildResult BuildCliArgs(CopilotOptions options)
{
    // Filter out unsupported arguments and keep compatible ones
    // Generate diagnostic information
}
```

Tämä antaa meille useita etuja:

- Se suodattaa automaattisesti yhteensopimattomat CLI-argumentit ajonaikaisten virheiden estämiseksi.
- Se luo selkeät diagnostiikkaviestit, jotka auttavat kehittäjiä löytämään ongelmat nopeasti.
- Se pitää SDK:n vakaana ja eristettynä CLI-parametrien muutoksilta.

Päivityksen aikana tämä yhteensopivuusmekanismi auttoi meitä siirtymään sujuvasti. Olemassa olevia konfiguraatiotiedostoja voitiin edelleen käyttää, ja niitä tarvittiin vain asteittain säätämällä diagnostiikkatulosteen perusteella.

### Runtime Pooling

Copilot SDK -istuntojen luominen on suhteellisen kallista, ja istuntojen luominen ja tuhoaminen liian usein heikentää suorituskykyä. Tämän ratkaisemiseksi otimme käyttöön istuntopoolin hallintajärjestelmän:

```csharp
// CopilotProvider.cs - Session pool management
await using var lease = await _poolCoordinator.AcquireCopilotRuntimeAsync(
    request,
    async ct => await _gateway.CreateRuntimeAsync(sdkRequest, ct),
    cancellationToken);

if (lease.IsWarmLease)
{
    // Reuse an existing session
    yield return CreateSessionReusedMessage();
}

await foreach (var eventData in lease.Entry.Resource.SendPromptAsync(...))
{
    yield return MapEvent(eventData);
}
```

Istuntojen yhdistämisen etuja ovat:

- **Istunnon uudelleenkäyttö**: pyynnöt samalla `sessionId` voi käyttää uudelleen olemassa olevia istuntoja ja vähentää käynnistyskustannuksia.
- **Istunnon palautuksen tuki**: verkkokatkoksen jälkeen voidaan palauttaa edellinen istunnon tila.
- **Automaattinen yhdistämisen hallinta**: vanhentuneet istunnot siivotaan automaattisesti resurssien vuotojen välttämiseksi.

HagiCodessa istuntojen yhdistäminen paransi huomattavasti reagointikykyä erityisesti jatkuvissa keskusteluissa.

### Orleansin integraatio

HagiCode käyttää Orleansia hajautettuna kehyksenä, ja integroimme Copilot SDK:n Orleans Grainsiin:

```csharp
// GitHubCopilotGrain.cs - Distributed execution
public async IAsyncEnumerable<GitHubCopilotResponse> ExecuteCommandStreamAsync(
    string command,
    CancellationToken token = default)
{
    var provider = await aiProviderFactory.GetProviderAsync(AIProviderType.GitHubCopilot);

    await foreach (var chunk in provider.SendMessageAsync(request, null, token))
    {
        // Map to the unified response format
        yield return BuildChunkResponse(chunk, startedAt);
    }
}
```

Orleansin integraation edut ovat huomattavia:

- **Unified AI Provider -abstraktio**: on helppoa vaihtaa eri AI-palveluntarjoajien välillä.
- **Useiden vuokralaisten eristäminen**: Eri käyttäjien copilot-istunnot pysyvät eristettyinä toisistaan.
- **Pysyvä istunnon tila**: istunnon tila voidaan palauttaa myös palvelimen uudelleenkäynnistyksen jälkeen.

Skenaarioissa, joissa on käsiteltävä suuri määrä samanaikaisia pyyntöjä, Orleans tarjoaa vahvan skaalautuvuuden.

## Käytännön opas

### Konfigurointiesimerkki

Tässä on täydellinen konfigurointiesimerkki:

```json
{
  "AI": {
    "Providers": {
      "Providers": {
        "GitHubCopilot": {
          "Enabled": true,
          "ExecutablePath": "copilot",
          "Model": "gpt-5",
          "WorkingDirectory": "/path/to/project",
          "Timeout": 7200,
          "StartupTimeout": 30,
          "UseLoggedInUser": true,
          "NoAskUser": true,
          "Permissions": {
            "AllowAllTools": false,
            "AllowedTools": ["Read", "Bash", "Grep"],
            "DeniedTools": ["Edit"]
          }
        }
      }
    }
  }
}
```

### Käyttöhuomautuksia

Todellisessa käytössä teimme yhteenvedon useista huomion arvoisista seikoista:

**Käynnistyksen aikakatkaisun määritys**: Copilot CLI:n ensimmäinen käynnistys voi kestää suhteellisen kauan, joten suosittelemme `StartupTimeout` vähintään 30 sekuntiin. Jos tämä on ensimmäinen kirjautuminen, se voi kestää kauemmin.

**Lupien hallinta**: Vältä käyttöä tuotantoympäristöissä `AllowAllTools: true`. Käytä `AllowedTools` sallittulista hallita käytettävissä olevia työkaluja ja käyttää `DeniedTools` denylist estää vaaralliset toiminnot. Tämä estää tehokkaasti tekoälyä suorittamasta riskialttiita komentoja.

**Istunnon hallinta**: Pyynnöt samalla `sessionId` käyttää istuntoja automaattisesti uudelleen. Istuntotila säilyy läpi `ProviderSessionId`. Peruutus välitetään kautta `CancellationTokenSource`.

**Diagnostiikkalähtö**: Yhteensopimattomat CLI-argumentit luovat tyyppisiä viestejä `diagnostic`. Raw SDK -tapahtumat säilytetään muodossa `event.raw`. Vianmäärityksen helpottamiseksi virheilmoitukset sisältävät luokkia, kuten käynnistyksen aikakatkaisun ja argumenttien yhteensopimattomuuden.

### Parhaat käytännöt

Käytännön kokemuksemme perusteella tässä on muutamia parhaita käytäntöjä:

**1. Käytä työkalujen sallittujen luetteloa**

```csharp
var request = new AIRequest
{
    Prompt = "Analyze this file",
    AllowedTools = new[] { "Read", "Grep", "Bash(git:*)" }
};
```

Sallittujen työkalujen selkeä määrittäminen sallittujen luettelon avulla auttaa estämään odottamattomia tekoälytoimintoja. Tämä on erityisen tärkeää työkaluille, joilla on kirjoitusoikeudet, kuten `Edit`, jota tulee käsitellä erityisen varovasti.

**2. Aseta kohtuulliset aikakatkaisut**

```csharp
options.Timeout = 3600;  // 1 hour
options.StartupTimeout = 60;  // 1 minute
```

Aseta sopivat aikakatkaisuarvot tehtävän monimutkaisuuden perusteella. Jos arvo on liian lyhyt, tehtävät voivat keskeytyä. Jos se on liian pitkä, resursseja saatetaan tuhlata odottamattomiin pyyntöihin.

**3. Ota istunnon uudelleenkäyttö käyttöön**

```csharp
options.SessionId = "my-session-123";
```

Käyttämällä samaa `sessionId` liittyviä tehtäviä varten voit käyttää uudelleen aikaisemman istunnon kontekstia ja parantaa vastausnopeutta.

**4. Käsittele suoratoistovastauksia**

```csharp
await foreach (var chunk in provider.StreamAsync(request))
{
    switch (chunk.Type)
    {
        case StreamingChunkType.ThinkingDelta:
            // Handle the reasoning process
            break;
        case StreamingChunkType.ToolCallDelta:
            // Handle tool invocation
            break;
        case StreamingChunkType.ContentDelta:
            // Handle text output
            break;
    }
}
```

Suoratoistovastausten avulla voit näyttää tekoälyn käsittelyn edistymisen reaaliajassa, mikä parantaa käyttökokemusta. Tämä on erityisen arvokasta pitkäkestoisissa tehtävissä.

**5. Virheiden käsittely ja uudelleenyritykset**

```csharp
try
{
    await foreach (var chunk in provider.StreamAsync(request))
    {
        // Handle the response
    }
}
catch (CopilotSessionException ex)
{
    // Handle session exceptions
    logger.LogError(ex, "Copilot session failed");
    // Decide whether to retry based on the exception type
}
```

Oikea virheenkäsittely ja uudelleenyrityslogiikka parantavat järjestelmän yleistä vakautta.

- **Parempi kehitystehokkuus**: tyyppiturvallinen SDK-käyttöliittymä tekee kehityksestä tehokkaampaa ja vähentää ajonaikaisia virheitä.
- **Parempi käyttökokemus**: reaaliaikainen tapahtumapalaute antaa käyttäjille selkeämmän käsityksen siitä, mitä tekoäly tekee.
