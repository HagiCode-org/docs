## GitHub-sivujen automaattisen käyttöönottotuen lisääminen HagiCodeen

> Tämän projektin koodinimi oli alun perin PCode, ja nyt se on saanut virallisen nimen HagiCode. Tässä artikkelissa kerrotaan, kuinka otimme käyttöön automatisoidun staattisen sivuston käyttöönoton, jotta sisällön julkaiseminen on yhtä helppoa kuin juomavesi.

## Tausta / Johdanto

HagiCode-kehityksen aikana törmäsimme hyvin käytännölliseen ongelmaan: dokumentaatiomme ja ehdotuksemme lisääntyessä sisällön tehokkaasta hallinnasta ja esittämisestä tuli yhä kiireellisempi. Päätimme käyttää GitHub-sivuja staattisen sivustomme isännöimiseen, mutta sen manuaalinen rakentaminen ja käyttöönotto oli aivan liian hankalaa. Jokainen muutos vaati paikallisen rakentamisen, pakkaamisen ja sitten manuaalisen työnnän `gh-pages` haara. Se ei ollut vain tehotonta, vaan myös helppo erehtyä.

Tämän ongelman ratkaisemiseksi tarvitsimme automatisoidun käyttöönoton työnkulun, koska halusimme säästää itseltämme vaivaa. Tässä artikkelissa kerrotaan yksityiskohtaisesti, kuinka lisäsimme GitHub Actions -pohjaisen automaattisen käyttöönottotuen HagiCode-projektiin, jotta voimme keskittyä sisällön kirjoittamiseen ja antaa automaation hoitaa loput.

## Tavoitteen analyysi

Ennen kuin ryhdymme toteuttamaan, meidän on ensin selvitettävä, mitä tällä tehtävällä todella on tarkoitus saavuttaa. Loppujen lopuksi kirveen teroitus ei hidasta polttopuiden pilkkomista.

### Perusvaatimukset

1.  **Automaattinen koontiversio**: Käynnistää automaattisesti koontityönkulun, kun koodi työnnetään tiedostoon `main` haara.
2.  **Automaattinen käyttöönotto**: Ota luodut staattiset tiedostot käyttöön automaattisesti GitHub-sivuille onnistuneen koontiversion jälkeen.
3.  **Ympäristön johdonmukaisuus**: Varmista, että CI-ympäristö vastaa paikallista rakennusympäristöä, jotta vältytään perinteiseltä "toimii paikallisesti, mutta epäonnistuu verkossa".

### Tekninen valinta

Koska HagiCode on rakennettu Docusauruselle, erittäin suositulle Reactin staattisen sivuston generaattorille, voimme käyttää GitHub Actionsia tämän tavoitteen saavuttamiseksi.

## GitHub Actions -työnkulun määrittäminen

GitHub Actions on GitHubin tarjoama CI/CD-palvelu. Määrittämällä arkistossa YAML-muodossa olevat työnkulkutiedostot voimme mukauttaa kaikenlaisia ​​automaatiotehtäviä.

### Työnkulkutiedoston luominen

Meidän on luotava uusi määritystiedosto `.github/workflows` hakemistoon esimerkiksi projektin juurella `deploy.yml`. Jos hakemistoa ei vielä ole, luo se ensin.

Tämän asetustiedoston ydinlogiikka on seuraava:

1.  **Liipaisuehto**: Kuuntele `push` tapahtumia `main` haara.
2.  **Ajonaikainen ympäristö**: Uusin Ubuntu-julkaisu.
3.  ** Rakenna vaiheet**:
    -   Tarkista koodi
    -   Asenna Node.js
    -   Asenna riippuvuudet (`npm install`)
    -   Rakenna staattiset tiedostot (`npm run build`)
4.  **Käyttöönottovaihe**: Käytä virallista `action-gh-pages` toiminto koontitulosteen työntämiseksi `gh-pages` haara.

### Avaimen konfigurointikoodi

Alla on määritysmalli, jonka lopulta hyväksyimme:

```yaml
name: Deploy to GitHub Pages

# Trigger condition: run when code is pushed to the main branch
on:
  push:
    branches:
      - main
    # Add path filters if needed, for example only build when docs change
    # paths:
    #   - 'docs/**'
    #   - 'package.json'

# Set permissions, which is important for GitHub Pages deployment
permissions:
  contents: read
  pages: write
  id-token: write

# Concurrency control: cancel older builds for the same branch
concurrency:
  group: "pages"
  cancel-in-progress: false

jobs:
  build:
    runs-on: ubuntu-latest
    steps:
      - name: Checkout
        uses: actions/checkout@v4
        # Note: fetch-depth: 0 is required, otherwise the build version may be inaccurate
        with:
          fetch-depth: 0

      - name: Setup Node
        uses: actions/setup-node@v4
        with:
          node-version: 20 # Keep this consistent with the local development environment
          cache: 'npm'     # Enabling cache can speed up the build process

      - name: Install dependencies
        run: npm ci
        # Use npm ci instead of npm install because it is faster and stricter, which makes it better for CI

      - name: Build website
        run: npm run build
        env:
          # Configure environment variables here if your site build requires them
          # NODE_ENV: production
          # PUBLIC_URL: /your-repo-name
          
      - name: Upload artifact
        uses: actions/upload-pages-artifact@v3
        with:
          path: ./build # Default Docusaurus output directory

  deploy:
    environment:
      name: github-pages
      url: ${{ steps.deployment.outputs.page_url }}
    runs-on: ubuntu-latest
    needs: build
    steps:
      - name: Deploy to GitHub Pages
        id: deployment
        uses: actions/deploy-pages@v4
```

## Yleisiä sudenkuoppia toteutuksen aikana

Käytännössä törmäsimme useisiin ongelmiin. Jaan ne täällä siinä toivossa, että muut voivat välttää ne tai ainakin valmistella ratkaisuja etukäteen.

### 1. GitHub Token -käyttöoikeusongelmat

Kun määritimme asiat ensimmäisen kerran, käyttöönotto epäonnistui jatkuvasti 403:lla (Forbidden). Pitkän kaivauksen jälkeen huomasimme, että GitHub on oletusarvo `GITHUB_TOKEN` hänellä ei ollut lupaa kirjoittaa Pagesiin.

**Ratkaisu**: Arkistossa `Settings` -> `Actions` -> `General` -> `Workflow permissions`, muista valita **"Luku- ja kirjoitusoikeudet"**.

### 2. Väärä koontihakemistopolku

Docusaurus sijoittaa rakennetut staattiset tiedostot `build` hakemisto oletuksena. Jotkut projektit voivat kuitenkin olla konfiguroituja eri tavalla. Esimerkiksi Luo React App oletusarvo on `build`, kun taas Viten oletusarvo on `dist`. Jos Actions ilmoittaa, että se ei löydä tulostiedostoja, tarkista sisääntulopolku `docusaurus.config.js`.

### 3. Alipolkuongelmat

Jos arkistosi ei ole käyttäjän kotisivu, se ei ole sitä `username.github.io`, vaan sen sijaan projektisivu, kuten `username.github.io/project-name`, sinun on määritettävä `baseUrl`.

sisään `docusaurus.config.js`:

```javascript
module.exports = {
  // ...
  url: 'https://hagicode.com', // Your Hagicode URL
  baseUrl: '/', // Deploy from the root path
  // ...
};
```
Tämä yksityiskohta on helppo jättää huomiotta. Jos se on määritetty väärin, sivu saattaa latautua tyhjänä näyttönä, koska resurssipolkuja ei voida ratkaista.

## Tuloksen tarkistaminen

Kun olet määrittänyt kaiken ja painanut koodin, voimme siirtyä GitHub-arkiston **Toiminnot**-välilehdelle ja katsoa esitystä.

Näet keltaisen ympyrän työnkulun ollessa käynnissä, ja kun se muuttuu vihreäksi, se tarkoittaa menestystä. Jos se muuttuu punaiseksi, napsauta sisään ja tarkista lokit. Useimmissa tapauksissa voit jäljittää ongelman siellä, yleensä kirjoitusvirheen tai virheellisen polun määrityksen.

Kun rakentaminen onnistuu, vieraile `https://<your-username>.github.io/<repository-name>/` ja näet juuri käyttöön otetun sivuston.

## Yhteenveto

Ottamalla käyttöön GitHub Actionsin automatisoimme onnistuneesti HagiCode-dokumentaatiosivuston käyttöönoton. Tämä ei ainoastaan ​​säästä manuaalisiin toimintoihin aiemmin käytettyä aikaa, mutta mikä tärkeintä, standardoi julkaisutyön kulun. Riippumatta siitä, kumpi tiimin jäsen päivittää dokumentaatiota, kun muutos on yhdistetty `main`, uusin sisältö julkaistaan muutaman minuutin kuluttua.

**Ydinedut**:
- **Parempi tehokkuus**: Siirryimme "manuaalisesta pakkaamisesta ja manuaalisesta lataamisesta" koodiin "koodi vastaa vapauttamista".
- **Vähemmän virheitä**: Poistimme inhimillisen virheen mahdollisuuden manuaalisten toimintojen aikana.
- **Parempi kehittäjäkokemus**: Kehittäjät voivat keskittyä enemmän sisällön laatuun sen sijaan, että työläs käyttöönottovaihe häiritsisi heitä.

Vaikka CI/CD:n määrittäminen on aluksi hieman hankalaa, varsinkin kaikkien käyttöoikeuksien ja polun yksityiskohtien kanssa, se on kertaluonteinen investointi, jolla on suuri pitkän aikavälin tuotto. Suosittelen vahvasti, että jokainen staattinen sivustoprojekti ottaa käyttöön samanlaisen automatisoidun työnkulun.
