## TypeScriptistä C#:aan: Cross-Language Porting Practice Codex SDK:lle

> Yksinkertaisesti sanottuna tämä artikkeli on myös pieni vauva: se tallentaa koko prosessin virallisen TypeScript Codex SDK:n siirtämisestä C#:aan. Sanomalla sitä "portiksi" se kuulostaa melkein liian helpolta - se oli enemmän kuin pitkä seikkailu, koska näillä kahdella kielellä on hyvin erilaisia ​​persoonallisuuksia, ja meidän oli löydettävä tapa saada ne tekemään yhteistyötä.

## Tausta

Codex on OpenAI:n julkaisema AI Agent CLI -työkalu, ja se on aidosti tehokas. Virallinen tiimi tarjoaa TypeScript SDK:n `@openai/codex` paketti. Se on vuorovaikutuksessa Codex CLI:n kanssa soittamalla numeroon `codex exec --experimental-json` komento ja JSONL-tapahtumavirran jäsentäminen.

Ongelmana on, että HagiCode-projektissa meidän on käytettävä sitä puhtaassa .NET-ympäristössä - erityisesti C#-taustapalveluissa ja työpöytäsovelluksissa. Emme voineet kohtuudella ottaa Node.js-ajoaikaa .NET-projektiin vain kutsuaksemme CLI-työkalua. Se olisi aivan liian vaivalloista.

Joten meille jäi kaksi vaihtoehtoa: ylläpitää monimutkaista Node.js-siltakerrosta tai rakentaa itse C# SDK.

Valitsimme jälkimmäisen.

## Ydinsisältö

### Arkkitehtisuunnittelun vertailu

Ennen koodin kääntämistä yksitellen meidän oli ensin ymmärrettävä molempien SDK:iden arkkitehtoninen suunnittelu. Sinun on ymmärrettävä molemmat puolet, ennen kuin voit siirtää ne hyvin.

TypeScript SDK:n ydinarkkitehtuuri näyttää tältä:

```
Codex (entry class)
  └── CodexExec (executor, manages child processes)
      └── Thread (conversation thread)
          ├── run() / runStreamed() (synchronous/asynchronous execution)
          └── event stream parsing
```

C# SDK säilyttää saman arkkitehtonisen kerrostuksen, mutta mukauttaa toteutustiedot. Yleisidea on suoraviivainen: Säilytä API johdonmukaisuus samalla kun hyödynnät C#-kielen ominaisuuksia toteutuksessa.

### Kirjoita System Conversion

Tämä on työn perustavanlaatuisin ja myös tärkein osa. Jos perusta on heikko, kaikki seuraava vaikeutuu.

TypeScriptin tyyppijärjestelmä on joustavampi kuin C#:n, ja se on yksinkertaisesti tosiasia. Meidän piti löytää sopiva kartoitusstrategia:

| TypeScript | C# | Huomautuksia |
|------------|-----|------|
| `interface` / `type` | `record` | C# käyttää `record` muuttumattomille tietorakenteille |
| `merkkijono \| nolla` | `string?` | Nollattava viitetyyppi |
| `looginen \| määrittelemätön' | `bool?` | Nullable Boolean |
| `AsyncGenerator` | `IAsyncEnumerable` | Asynkronoitu iteraattori |

**Tapahtumatyyppinen järjestelmä** on tyypillinen esimerkki. TypeScript käyttää liittotyyppejä tapahtumien määrittämiseen:

```typescript
export type ThreadEvent =
  | ThreadStartedEvent
  | TurnStartedEvent
  | TurnCompletedEvent
  | ...
```

C#:ssa käytämme periytymishierarkiaa ja mallisovitusta samanlaisen vaikutuksen saavuttamiseksi:

```csharp
public abstract record ThreadEvent(string Type);

public sealed record ThreadStartedEvent(string ThreadId) : ThreadEvent("thread.started");
public sealed record TurnStartedEvent() : ThreadEvent("turn.started");
public sealed record TurnCompletedEvent(Usage Usage) : ThreadEvent("turn.completed");
// ...
```

Valitsimme `record` sijasta `class` koska tapahtumaobjektien tulee olla muuttumattomia, mikä vastaa TypeScriptin tavallisten objektien käytön tarkoitusta. The `sealed` avainsana estää myös lisäperinnön ja antaa kääntäjälle tilaa optimoida.

### Tärkeimmät siirtokohdat

#### 1. Tapahtuman jäsentäjä

Tapahtuman jäsentäminen on koko SDK:n ydin, koska se määrittää, ymmärrämmekö oikein jokaisen Codex CLI:n palauttaman viestin. Jos jäsentäminen on väärin, kaikki sen jälkeen on turhaa vaivaa.

TypeScript-versio käyttää `JSON.parse()` jäsentääksesi jokaisen JSON-rivin:

```typescript
export function parseEvent(line: string): ThreadEvent {
  const data = JSON.parse(line);
  // Handle different event types...
}
```

C#-versio käyttää `System.Text.Json.JsonDocument` sen sijaan:

```csharp
public static ThreadEvent Parse(string line)
{
    using var document = JsonDocument.Parse(line);
    var root = document.RootElement;
    var type = GetRequiredString(root, "type", "event.type");

    return type switch
    {
        "thread.started" => new ThreadStartedEvent(GetRequiredString(root, "thread_id", ...)),
        "turn.started" => new TurnStartedEvent(),
        "turn.completed" => new TurnCompletedEvent(ParseUsage(...)),
        // ...
        _ => new UnknownThreadEvent(type, root.Clone()),
    };
}
```

Tässä on yksi pieni mutta tärkeä temppu: `root.Clone()` tarvitaan, koska elementtejä `JsonDocument` mitätöidä asiakirjan hävittämisen jälkeen. Meidän on säilytettävä kopio tuntemattomista tapahtumatyypeistä. Tämä on yksinkertaisesti yksi eroista C# JSON-käsittelyn ja JavaScriptin välillä.

#### 2. Prosessinhallinnan erot

Tässä kaksi SDK:ta eroavat eniten. Node.js:llä ja .NET:llä on erilaiset ajonaikaiset käytännöt, joten toteutuksen on mukauduttava.

TypeScript käyttää Node.js:ää `spawn()` toiminto:

```typescript
const child = spawn(this.executablePath, commandArgs, { env, signal });
```

C# käyttää .NET:iä `System.Diagnostics.Process`:

```csharp
using var process = new Process { StartInfo = startInfo };
process.Start();

// stdin/stdout/stderr must be managed manually
```

Tarkemmin sanottuna C#-version on määritettävä prosessi seuraavasti:

```csharp
var startInfo = new ProcessStartInfo
{
    FileName = _executablePath,
    RedirectStandardInput = true,
    RedirectStandardOutput = true,
    RedirectStandardError = true,
    UseShellExecute = false,
    CreateNoWindow = true,
};
```

Suurin ero on peruutusmekanismi. TypeScript käyttää `AbortSignal`, joka on osa Web API:ta ja jonka kanssa on erittäin kätevä työskennellä:

```typescript
const child = spawn(cmd, args, { signal: cancellationSignal });
```

C# käyttää `CancellationToken` sen sijaan:

```csharp
public async IAsyncEnumerable<string> RunAsync(
    CodexExecArgs args,
    [EnumeratorCancellation] CancellationToken cancellationToken = default)
{
    // Check cancellation status inside the loop
    while (!cancellationToken.IsCancellationRequested)
    {
        // Process output...
    }

    // Terminate the process when cancellation is requested
    if (cancellationToken.IsCancellationRequested)
    {
        try { process.Kill(entireProcessTree: true); } catch { }
    }
}
```

Korkealla tasolla tämä on vain yksi esimerkki Web API -ekosysteemin ja .NET-ekosysteemin välisestä erosta.

#### 3. Säilytetään asetusten sarjoittaminen

Molemmat SDK:t toteuttavat logiikan, joka muuntaa JSON-määritykset TOML-kokoonpanoksi, koska Codex CLI hyväksyy konfiguroinnin ohitukset TOML-muodossa. Tämän osan on pysyttävä täysin johdonmukaisena, muuten sama kokoonpano toimii eri tavalla kahdessa SDK:ssa.

Se on sellaisia yksityiskohtia, joista et voi tehdä kompromisseja. Menestys tai epäonnistuminen riippuu usein tällaisista yksityiskohdista.

### Toteutustiedot

#### Hankkeen rakenne

Loimme seuraavan projektirakenteen:

```
CodexSdk/
├── CodexSdk.csproj
├── Codex.cs           # Entry class
├── CodexThread.cs     # Conversation thread
├── CodexExec.cs       # Executor
├── Events.cs          # Event type definitions
├── Items.cs           # Item type definitions
├── EventParser.cs     # Event parser
├── OutputSchemaTempFile.cs  # Temporary file management
└── ...
```

Se on melko puhdas rakenne, ja se auttoi paljon satamassa.

#### Käyttöesimerkki

Peruskäyttö pysyy TypeScript SDK:n mukaisena:

```csharp
using CodexSdk;

// Create a Codex instance
var codex = new Codex();
var thread = codex.StartThread();

// Execute a query
var result = await thread.RunAsync("Summarize this repository.");
Console.WriteLine(result.FinalResponse);
```

Suoratoistotapahtumien käsittelyssä hyödynnetään C#-mallin täsmäämistä:

```csharp
await foreach (var @event in thread.RunStreamedAsync("Analyze the code."))
{
    switch (@event)
    {
        case ItemCompletedEvent itemCompleted
            when itemCompleted.Item is AgentMessageItem msg:
            Console.WriteLine($"Assistant: {msg.Text}");
            break;
        case TurnCompletedEvent completed:
            Console.WriteLine($"Tokens: in={completed.Usage.InputTokens}");
            break;
        case CommandExecutionItem command:
            Console.WriteLine($"Command: {command.Command}");
            break;
    }
}
```

### Huomautuksia

Toteutuksen aikana keräsimme useita käytännön oppitunteja:

1. **Prosessinhallinta**: C#-version on hallittava koko prosessin elinkaarta manuaalisesti, mukaan lukien prosessin lopettaminen peruutuksen aikana. Käytä `Kill(entireProcessTree: true)` varmistaaksesi, että myös lasten prosessit puhdistetaan.

2. **Virheiden käsittely**: Käytämme `InvalidOperationException` heittää jäsennysvirheitä pitäen virheiden käsittelytyylin samanlaisena kuin TypeScript SDK.

3. **Resurssien puhdistus**: `OutputSchemaTempFile` toteuttaa `IAsyncDisposable` varmistaaksesi, että väliaikaiset tiedostot siivotaan oikein.

4. **Ympäristömuuttujat**: C#-versio tukee täysin prosessiympäristömuuttujien ohittamista `CodexOptions.Env`. Se on pieni ominaisuus, mutta erittäin käytännöllinen.

5. **Alustoerot**: C#-versio ei sisällä TypeScript-version logiikkaa binäärien automaattiseen paikallistamiseen npm-pakettien sisällä. Koska .NET-projektit eivät yleensä ole riippuvaisia npm:stä, polku `codex` suoritettava tiedosto on määritettävä `CODEX_EXECUTABLE` ympäristömuuttuja tai `CodexPathOverride`.

## Johtopäätös

Kypsän TypeScript SDK:n siirtäminen C#:hon ei ole vain syntaksin muuntamista - se vaatii myös molempien kielten suunnittelufilosofian ymmärtämistä. TypeScriptin joustavuus ja JavaScript-ekosysteemiominaisuudet, kuten `AbortSignal` tarvitsevat sopivat vastineet C#:ssa.

Keskeinen poiminta on tämä: **API-yhdenmukaisuuden ylläpitäminen on tärkeämpää kuin toteutustason johdonmukaisuuden säilyttäminen**. Käyttäjiä kiinnostaa käyttöliittymän helppokäyttöisyys, ei siitä, onko sisäinen toteutus identtinen. Se kuulostaa yksinkertaiselta, mutta näiden kompromissien tekeminen vaatii harkintaa.

Jos työskentelet samankaltaisen monikielisen portin parissa, kokemuksemme on ymmärtää ensin täysin alkuperäinen SDK-arkkitehtuuri, sitten kääntää se moduuli moduulilta ja käyttää lopuksi täydellistä testipakettia toiminnan johdonmukaisuuden varmistamiseksi. Tällaista työtä ei voi kiirehtiä.

Kaikki järjestyy lopulta.
