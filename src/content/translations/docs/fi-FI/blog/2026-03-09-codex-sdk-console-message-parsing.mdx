## Täydellinen Codex SDK -konsolin viestien jäsennysopas

> Tässä artikkelissa selitetään Codex SDK -tapahtumavirtamekanismi, viestityyppien jäsentäminen ja parhaat käytännöt oikeissa projekteissa, mikä auttaa kehittäjiä hallitsemaan nopeasti tekoälyn suorituspalvelujen ydintaidot.

## Tausta

Kun rakennamme Codex SDK:n pohjalta tekoälyn suorituspalvelua, törmäämme väistämättä käytännön kysymykseen: miten meidän tulisi käsitellä Codexin palauttamia suoratoistettuja tapahtumaviestejä? Nämä viestit sisältävät tärkeitä tietoja, kuten suoritustilan, tulosteen sisällön ja virhetiedot, joten ne ansaitsevat huolellisen käsittelyn.

Osana HagiCode-projektia tarvitsimme luotettavan toteuttajan tekoälyn koodausavustajan skenaarioihin. Juuri tästä syystä päätimme tutkia Codex SDK -tapahtumavirtamekanismia perusteellisesti. Loppujen lopuksi vain ymmärtämällä, kuinka taustalla olevat viestit toimivat, voimme rakentaa todella yritystason tekoälyn suoritusalustan.

Codex SDK on OpenAI:n julkaisema ohjelmointiapu-SDK. Se palauttaa suoritustulokset tapahtumavirran kautta. Toisin kuin perinteinen pyyntö-vastausmalli, Codex käyttää suoratoistettuja tapahtumia, jotta voimme:

- Näet suorituksen edistymisen reaaliajassa
- Käsittele virheet viipymättä
- Hanki yksityiskohtaiset tunnuksen käyttötilastot
- Tukee pitkäkestoisia monimutkaisia tehtäviä

Näiden tapahtumatyyppien ymmärtäminen ja niiden oikea jäsentäminen on olennaista täysin kykenevän tekoälyn toteuttajan toteuttamiseksi. Loppujen lopuksi kukaan ei halua työskennellä mustan laatikon kanssa.

## Tietoja HagiCodesta

Tässä artikkelissa jaettu ratkaisu on peräisin käytännön kokemuksestamme [HagiCode](https://hagicode.com) hanke. HagiCode on avoimen lähdekoodin tekoälyn koodausavustajaprojekti, joka on omistettu tarjoamaan kehittäjille älykästä koodaustukea. Kehityksen aikana meidän täytyi rakentaa luotettava tekoälyn suorituspalvelu käsittelemään käyttäjäkoodin suorituspyyntöjä, mikä on suora syy Codex SDK:n käyttöönotolle.

Tekoälyn koodausassistenttina HagiCoden täytyy käsitellä erilaisia monimutkaisia koodin suoritusskenaarioita: saada suorituksen edistymistä reaaliajassa, käsitellä virheitä nopeasti ja kerätä yksityiskohtaisia tunnuksen käyttötilastoja. Ymmärtämällä syvällisesti Codex SDK -tapahtumavirtamekanismin voimme rakentaa toteuttajan, joka täyttää tuotantoympäristön vaatimukset. Loppujen lopuksi kaikki hyötyy, olipa kyseessä ohjelmisto tai tosielämä, tasaisesta kerääntymisestä ja tarkentamisesta.

## Tapahtumavirtamekanismi

### Peruskäsitteet

Codex SDK käyttää `thread.runStreamed()` tapa palauttaa asynkroninen tapahtumaiteraattori:

```typescript
import { Codex } from '@openai/codex-sdk';

const client = new Codex({
  apiKey: process.env.CODEX_API_KEY,
  baseUrl: process.env.CODEX_BASE_URL,
});

const thread = client.startThread({
  workingDirectory: '/path/to/project',
  skipGitRepoCheck: false,
});

const { events } = await thread.runStreamed('your prompt here', {
  outputSchema: {
    type: 'object',
    properties: {
      output: { type: 'string' },
      status: { type: 'string', enum: ['ok', 'action_required'] },
    },
    required: ['output', 'status'],
  },
});

for await (const event of events) {
  // Handle each event
}
```

### Tapahtumatyyppien yksityiskohtainen selitys

| Tapahtuman tyyppi | Kuvaus | Keskeiset tiedot |
|---------|------|----------|
| `thread.started` | Viesti aloitettiin onnistuneesti | `thread_id` |
| `item.updated` | Viestin sisältö päivitetty | `item.text` |
| `item.completed` | Viesti valmis | `item.text` |
| `turn.completed` | Toteutus suoritettu | `usage` (tunnuksen käyttö) |
| `turn.failed` | Suoritus epäonnistui | `error.message` |
| `error` | Virhetapahtuma | `message` |

Todellisissa projekteissa HagiCoden toteuttajakomponentti on rakennettu näiden tapahtumatyyppien päälle. Meidän on käsiteltävä jokaisen tyyppisiä tapahtumia huolellisesti varmistaaksemme sujuvan käyttökokemuksen. Hyvät järjestelmät rakennetaan ottamalla yksityiskohdat vakavasti.

## Viestien jäsennys toteutus

### Viestin sisällön purkaminen

Viestin sisältö poimitaan tapahtumakäsittelijän kautta:

```typescript
private handleThreadEvent(event: ThreadEvent, onMessage: (content: string) => void): void {
  // Only handle message update and completion events
  if (event.type !== 'item.updated' && event.type !== 'item.completed') {
    return;
  }

  // Only handle agent message content
  if (event.item.type !== 'agent_message') {
    return;
  }

  // Extract text content
  onMessage(event.item.text);
}
```

Pääkohdat:
- Vain kahva `item.updated` ja `item.completed` tapahtumia
- Käsittele vain tyyppistä sisältöä `agent_message`
- Viestin sisältö on `event.item.text` kenttä

### Strukturoidun lähdön jäsentäminen

Codex tukee JSON-strukturoitua tulostusta. Voit määrittää palautusmuodon käyttämällä `outputSchema` parametri:

```typescript
const DEFAULT_OUTPUT_SCHEMA = {
  type: 'object',
  properties: {
    output: { type: 'string' },
    status: { type: 'string', enum: ['ok', 'action_required'] },
  },
  required: ['output', 'status'],
  additionalProperties: false,
} as const;
```

Jäsennystoiminto yrittää jäsentää JSONia, ja jos se epäonnistuu, se palaa raakatekstiin.

```typescript
function toStructuredOutput(raw: string): StructuredOutput {
  try {
    const parsed = JSON.parse(raw) as Partial<StructuredOutput>;
    if (typeof parsed.output === 'string') {
      return {
        output: parsed.output,
        status: parsed.status === 'action_required' ? 'action_required' : 'ok',
      };
    }
  } catch {
    // JSON parsing failed, fall back to the raw text
  }

  return {
    output: raw,
    status: 'ok',
  };
}
```

### Täydellinen tapahtumankäsittelyvirta

```typescript
private async runWithStreaming(
  thread: Thread,
  input: CodexStageExecutionInput
): Promise<{ output: string; usage: Usage | null }> {
  const abortController = new AbortController();
  const timeoutHandle = setTimeout(() => {
    abortController.abort();
  }, Math.max(1000, input.timeoutMs));

  let latestMessage = '';
  let usage: Usage | null = null;
  let emittedLength = 0;

  try {
    const { events } = await thread.runStreamed(input.prompt, {
      outputSchema: DEFAULT_OUTPUT_SCHEMA,
      signal: abortController.signal,
    });

    for await (const event of events) {
      // Handle message content
      this.handleThreadEvent(event, (nextContent) => {
        const delta = nextContent.slice(emittedLength);
        if (delta.length > 0) {
          emittedLength = nextContent.length;
          input.callbacks?.onChunk?.(delta);  // Streaming callback
        }
        latestMessage = nextContent;
      });

      // Process different data based on the event type
      if (event.type === 'thread.started') {
        this.threadId = event.thread_id;
      } else if (event.type === 'turn.completed') {
        usage = event.usage;
      } else if (event.type === 'turn.failed') {
        throw new CodexExecutorError('gateway_unavailable', event.error.message, true);
      } else if (event.type === 'error') {
        throw new CodexExecutorError('gateway_unavailable', event.message, true);
      }
    }
  } catch (error) {
    if (abortController.signal.aborted) {
      throw new CodexExecutorError(
        'upstream_timeout',
        `Codex stage timed out after ${input.timeoutMs}ms`,
        true
      );
    }
    throw error;
  } finally {
    clearTimeout(timeoutHandle);
  }

  const structured = toStructuredOutput(latestMessage);
  return { output: structured.output, usage };
}
```

## Virheenkäsittelystrategia

### Virhekoodikartoitus

Kartoita tietyt virhemallit konkreettisiin virhekoodeihin, jotta ylemmät kerrokset voivat käsitellä niitä helpommin:

```typescript
function mapError(error: unknown): CodexExecutorError {
  if (error instanceof CodexExecutorError) {
    return error;
  }

  const message = error instanceof Error ? error.message : String(error);
  const normalized = message.toLowerCase();

  // Authentication errors - not retryable
  if (normalized.includes('401') ||
      normalized.includes('403') ||
      normalized.includes('api key') ||
      normalized.includes('auth')) {
    return new CodexExecutorError('auth_invalid', message, false);
  }

  // Rate limit errors - retryable
  if (normalized.includes('429') || normalized.includes('rate limit')) {
    return new CodexExecutorError('rate_limited', message, true);
  }

  // Timeout errors - retryable
  if (normalized.includes('timeout') || normalized.includes('aborted')) {
    return new CodexExecutorError('upstream_timeout', message, true);
  }

  // Default error
  return new CodexExecutorError('gateway_unavailable', message, true);
}
```

### Virhetyypin määritelmät

```typescript
export type CodexErrorCode =
  | 'auth_invalid'      // Authentication failure
  | 'upstream_timeout'  // Upstream timeout
  | 'rate_limited'      // Rate limited
  | 'gateway_unavailable'; // Gateway unavailable

export class CodexExecutorError extends Error {
  readonly code: CodexErrorCode;
  readonly retryable: boolean;

  constructor(code: CodexErrorCode, message: string, retryable: boolean) {
    super(message);
    this.name = 'CodexExecutorError';
    this.code = code;
    this.retryable = retryable;
  }
}
```

## Työhakemiston ja ympäristön konfigurointi

### Työhakemiston validointi

Codex SDK edellyttää, että työhakemisto on kelvollinen Git-arkisto.

```typescript
export function validateWorkingDirectory(
  workingDirectory: string,
  skipGitRepoCheck: boolean
): void {
  const resolvedWorkingDirectory = path.resolve(workingDirectory);

  if (!existsSync(resolvedWorkingDirectory)) {
    throw new CodexExecutorError(
      'gateway_unavailable',
      'Working directory does not exist.',
      false
    );
  }

  if (!statSync(resolvedWorkingDirectory).isDirectory()) {
    throw new CodexExecutorError(
      'gateway_unavailable',
      'Working directory is not a directory.',
      false
    );
  }

  if (skipGitRepoCheck) {
    return;
  }

  const gitDir = path.join(resolvedWorkingDirectory, '.git');
  if (!existsSync(gitDir)) {
    throw new CodexExecutorError(
      'gateway_unavailable',
      'Working directory is not a git repository.',
      false
    );
  }
}
```

### Ladataan ympäristömuuttujia

Codex SDK:n on ladattava ympäristömuuttujat kirjautumiskuoresta, jotta AI-agentti voi käyttää järjestelmän komentoja:

```typescript
function parseEnvironmentOutput(output: Buffer): Record<string, string> {
  const parsed: Record<string, string> = {};

  for (const entry of output.toString('utf8').split('\0')) {
    if (!entry) continue;

    const separatorIndex = entry.indexOf('=');
    if (separatorIndex <= 0) continue;

    const key = entry.slice(0, separatorIndex);
    const value = entry.slice(separatorIndex + 1);
    if (key.length > 0) {
      parsed[key] = value;
    }
  }

  return parsed;
}

function tryLoadEnvironmentFromShell(shellPath: string): Record<string, string> | null {
  const result = spawnSync(shellPath, ['-ilc', 'env -0'], {
    env: process.env,
    stdio: ['ignore', 'pipe', 'pipe'],
    timeout: 5000,
  });

  if (result.error || result.status !== 0) {
    return null;
  }

  return parseEnvironmentOutput(result.stdout);
}

export function createExecutorEnvironment(
  envOverrides: Record<string, string> = {}
): Record<string, string> {
  // Load environment variables from the login shell
  const consoleEnv = loadConsoleEnvironmentFromShell();

  return {
    ...process.env,
    ...consoleEnv,
    ...envOverrides,
  };
}
```

## Täydellinen käyttöesimerkki

### Peruskäyttö

HagiCode-projektissa käytämme seuraavaa lähestymistapaa Codex-asiakkaan alustamiseen ja tehtävien suorittamiseen:

```typescript
import { Codex } from '@openai/codex-sdk';

async function executeWithCodex(prompt: string, workingDir: string) {
  const client = new Codex({
    apiKey: process.env.CODEX_API_KEY,
    env: { PATH: process.env.PATH },
  });

  const thread = client.startThread({
    workingDirectory: workingDir,
  });

  const { events } = await thread.runStreamed(prompt);

  let result = '';
  for await (const event of events) {
    if (event.type === 'item.updated' && event.item.type === 'agent_message') {
      result = event.item.text;
    }
    if (event.type === 'turn.completed') {
      console.log('Token usage:', event.usage);
    }
  }

  // Try to parse JSON output
  try {
    const parsed = JSON.parse(result);
    return parsed.output;
  } catch {
    return result;
  }
}
```

### Täydellinen käyttöönotto uudelleenyritysten avulla

```typescript
export class CodexSdkExecutor {
  private readonly config: CodexRuntimeConfig;
  private readonly client: Codex;
  private threadId: string | null = null;

  async executeStage(input: CodexStageExecutionInput): Promise<CodexStageExecutionResult> {
    const maxAttempts = Math.max(1, this.config.retryCount + 1);
    let attempt = 0;
    let lastError: CodexExecutorError | null = null;

    while (attempt < maxAttempts) {
      attempt += 1;

      try {
        const thread = this.getThread(input.workingDirectory);
        const { output, usage } = await this.runWithStreaming(thread, input);

        return {
          output,
          usage,
          threadId: this.threadId!,
          attempts: attempt,
          latencyMs: Date.now() - startedAt,
        };
      } catch (error) {
        const mappedError = mapError(error);
        lastError = mappedError;

        // Non-retryable error or max retry attempts reached
        if (!mappedError.retryable || attempt >= maxAttempts) {
          throw mappedError;
        }

        // Wait before retrying
        await new Promise(resolve => setTimeout(resolve, 1000 * attempt));
      }
    }

    throw lastError!;
  }
}
```

## Parhaat käytännöt

### 1. Työhakemiston vaatimukset

- Varmista, että työhakemisto on kelvollinen Git-arkisto
- Käytä `PROJECT_ROOT` ympäristömuuttuja määrittääksesi sen nimenomaisesti
- Viankorjauksen aikana voit asettaa `CODEX_SKIP_GIT_REPO_CHECK=true` ohittaaksesi sekin

### 2. Ympäristömuuttujan kokoonpano

- Ohjaa vain vaaditut ympäristömuuttujat sallittujen luettelomekanismin kautta
- Käytä kirjautumiskuorta ladataksesi koko ympäristön
- Vältä arkaluonteisten tietojen välittämistä

### 3. Aikakatkaisu- ja uudelleenyritysstrategia

- Aseta kohtuulliset aikakatkaisut tehtävän monimutkaisuuden perusteella
- Toteuta eksponentiaalinen peruutus uudelleen yritettävissä oleville virheille
- Tallenna uudelleenyritysten määrät ja syyt

### 4. Virheiden käsittely

- Erota uudelleen yritettävät ja ei-uudelleenyritettävät virheet
- Anna selkeät virheilmoitukset ja ehdotukset
- Käytä yhtenäisiä virhekoodeja, jotta ylemmät kerrokset voivat käsitellä niitä johdonmukaisesti

### 5. Suoratoistolähtö

- Ota käyttöön inkrementaaliset takaisinkutsut parantaaksesi käyttökokemusta
- Käsittele porrastetut viestipäivitykset oikein
- Tallenna tunnuksen käyttö kustannusanalyysiä varten

Varsinaisessa HagiCode-projektin tuotantoympäristössä olemme jo varmistaneet yllä olevien parhaiden käytäntöjen toimivuuden. Tämä lähestymistapa on auttanut meitä rakentamaan vakaan ja luotettavan tekoälyn suorituspalvelun. Loppujen lopuksi käytännön validointi on tärkeämpää kuin pelkkä teoria.
