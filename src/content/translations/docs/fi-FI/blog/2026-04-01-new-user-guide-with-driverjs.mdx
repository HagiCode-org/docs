## Elegantin uuden käyttäjän perehdyttämisen rakentaminen React-projekteihin: HagiCoden driver.js-käytäntö

> Kun käyttäjät avaavat tuotteesi ensimmäisen kerran, tietävätkö he todella mistä aloittaa? Tässä artikkelissa haluan puhua hieman siitä, kuinka käytimme driver.js:ää uusien käyttäjien perehdyttämiseen HagiCode-projektissa. Pidä sitä pienenä käytännön esimerkkinä keskustelun aloittamiseksi.

## Tausta

Oletko koskaan törmännyt tähän tilanteeseen? Uusi käyttäjä kirjautuu tuotteeseesi, avaa sivun ja näyttää heti eksyneeltä. He selailevat ympäriinsä eivätkä ole varmoja, mitä klikata tai mitä tehdä seuraavaksi. Kehittäjinä oletamme usein, että käyttäjät vain "tutkivat itse", koska ihmisen uteliaisuus on loppujen lopuksi rajaton. Todellisuus on kuitenkin erilainen: useimmat käyttäjät lähtevät hiljaa muutamassa minuutissa, jos he eivät löydä oikeaa sisääntulokohtaa, ikään kuin tarina alkaisi yhtäkkiä ja päättyisi aivan yhtä luonnollisesti.

Uuden käyttäjän käyttöönotto on tärkeä tapa ratkaista tämä ongelma, mutta sen rakentaminen ei ole niin yksinkertaista. Hyvä perehdytysjärjestelmä tarvitsee:

- Paikanna sivun elementit tarkasti ja korosta ne
- Tukee monivaiheisia käyttöönottokulkuja
- Muista käyttäjän valinta (täydennä tai ohita)
- Vältä vaikuttamasta sivun suorituskykyyn ja normaaliin vuorovaikutukseen
- Pidä koodirakenne selkeänä ja helppohoitoisena

HagiCodea rakentaessamme törmäsimme samaan haasteeseen. HagiCode on AI-koodausavustaja, ja sen ydintyönkulku on OpenSpec-työnkulku, joka näyttää tältä: "käyttäjä luo ehdotuksen -> tekoäly luo suunnitelman -> käyttäjä arvioi sen -> tekoäly suorittaa sen." Käyttäjille, jotka kohtaavat tämän konseptin ensimmäistä kertaa, työnkulku on täysin uusi, joten he tarvitsevat vankan perehdytyksen päästäkseen alkuun nopeasti. Uusiin asioihin tottuminen vie aina vähän aikaa.

## Miksi valitsimme driver.js:n

Teknisen arviointivaiheen aikana tarkastelimme useita valtavirtaan perustuvia kirjastoja. Jokaisella oli omat vahvuutensa:

- **Intro.js**: Tehokas, mutta suhteellisen suuri, ja tyylin mukauttaminen on melko monimutkaista
- **Shepherd.js**: Hyvin suunniteltu sovellusliittymä, mutta hieman liian "raskas" käyttötapaukseemme
- **driver.js**: Kevyt, ytimekäs, intuitiivinen API ja toimii hyvin React-ekosysteemissä

Lopulta valitsimme driver.js:n. Mitään erityisen dramaattista syytä ei ollut. Valinta johtui pääasiassa seuraavista näkökohdista:

1. **Kevyt**: Ydinkirjasto on pieni eikä lisää merkittävästi nippukokoa
2. **Yksinkertainen API**: Kokoonpano on selkeä ja intuitiivinen, joten se on helppo poimia
3. **Joustava**: Tukee mukautettua sijoittelua, tyyliä ja vuorovaikutuskäyttäytymistä
4. **Dynaaminen tuonti**: Voidaan ladata pyynnöstä vaikuttamatta ensimmäisen näytön suorituskykyyn

Teknologian valinnassa on harvoin universaalisti parasta vastausta. Yleensä on vain se vaihtoehto, joka sopii parhaiten.

## Tekninen toteutus

### Ydinkokoonpano

driver.js:ssä on erittäin intuitiivinen määritysmalli. Tässä on ydinkokoonpano, jota käytämme HagiCode-projektissa:

```typescript
import { driver } from 'driver.js';
import 'driver.js/dist/driver.css';

const newConversationDriver = driver({
  allowClose: true,           // Allow users to close the guide
  animate: true,              // Enable animations
  overlayClickBehavior: 'close', // Close the guide when the overlay is clicked
  disableActiveInteraction: false, // Keep elements interactive
  showProgress: false,        // Do not show the progress bar (we manage progress ourselves)
  steps: guideSteps           // Array of guide steps
});
```

Syy näiden asetusten taustalla on:

- `allowClose: true` - Kunnioita käyttäjän valintaa äläkä pakota häntä lukemaan opasta loppuun
- `disableActiveInteraction: false` - Jotkut vaiheet edellyttävät todellisia käyttäjän toimia, kuten kirjoittamista, joten vuorovaikutusta ei voi poistaa käytöstä
- `overlayClickBehavior: 'close'` - Anna käyttäjille nopea tapa poistua

### Valtion hallinto

Jatkuva aloitustila on kriittinen. Emme halua käynnistää opasta uudelleen joka kerta, kun sivu päivittyy, koska se ärsyttää nopeasti. HagiCode käyttää `localStorage` ohjata ohjetilaa:

```typescript
export type GuideState = 'pending' | 'dismissed' | 'completed';

export interface UserGuideState {
  session: GuideState;
  detailGuides: Record<string, GuideState>;
}

// Read state
export const getUserGuideState = (): UserGuideState => {
  const state = localStorage.getItem('userGuideState');
  return state ? JSON.parse(state) : { session: 'pending', detailGuides: {} };
};

// Update state
export const setUserGuideState = (state: UserGuideState) => {
  localStorage.setItem('userGuideState', JSON.stringify(state));
};
```

Määrittelemme kolme tilaa:

- `pending`: Opas on edelleen kesken, eikä käyttäjä ole suorittanut tai ohittanut sitä
- `dismissed`: Käyttäjä sulki oppaan ennakoivasti
- `completed`: Käyttäjä suoritti kaikki vaiheet

Ehdotuksen tietosivun käyttöönottoa varten tuemme myös tarkempaa tilanseurantaa `detailGuides` karttaa, koska yksi ehdotus voi käydä läpi useita vaiheita – luonnos, tarkistus ja toteutus valmis – ja jokainen vaihe tarvitsee erilaista ohjausta. Tilanne muuttuu aina, ja perehdyttämisen pitäisi heijastaa sitä.

### Kohdeelementin valinta

driver.js käyttää CSS-valitsimia kohdeelementtien paikantamiseen. HagiCode noudattaa yksinkertaista käytäntöä: käytä mukautettua `data-guide` attribuutti perehdytystavoitteiden merkitsemiseen:

```typescript
const steps = [
  {
    element: '[data-guide="launch"]',
    popover: {
      title: 'Start a New Conversation',
      description: 'Click here to create a new conversation session...'
    }
  }
];
```

Komponenteissa se näyttää tältä:

```typescript
<button data-guide="launch" onClick={handleLaunch}>
  New Conversation
</button>
```

Tämän lähestymistavan edut ovat:

- Vältä ristiriidat bisnestyyliluokkien nimien kanssa
- Selkeä semantiikka, jotta voit heti kertoa, että elementti liittyy perehdyttämiseen
- Helpompi hallita ja ylläpitää johdonmukaisesti

### Dynaaminen tuonnin optimointi

Koska käyttöönotto-ominaisuutta tarvitaan vain tietyissä skenaarioissa, kuten käyttäjän ensimmäisellä vierailulla, käytämme dynaamisia tuontia optimoidaksemme alkulatauksen suorituskyvyn:

```typescript
const initNewUserGuide = async () => {
  // Dynamically import driver.js
  const { driver } = await import('driver.js');
  await import('driver.js/dist/driver.css');

  // Initialize the guide
  const newConversationDriver = driver({
    // ...configuration
  });

  newConversationDriver.drive();
};
```

Tällä tavalla driver.js ja sen tyylitaulukko ladataan vain tarvittaessa, eivätkä ne vaikuta ensimmäisen näytön suorituskykyyn. Harvat ihmiset nauttivat odottamasta jotain, jota he eivät vielä edes tarvitse.

## Flow Designin käyttöönotto

HagiCode toteuttaa kaksi käyttöönottopolkua, jotka kattavat käyttäjän ydinskenaariot.

### Istunnon aloittaminen (10 askelta)

Tämä aloituspolku auttaa käyttäjiä suorittamaan koko kulun keskustelun luomisesta ensimmäisen täydellisen ehdotuksensa lähettämiseen:

1. **käynnistä** - Aloita opas ja ota käyttöön "Uusi keskustelu" -painike
2. **kirjoita** - Ohjaa käyttäjää kirjoittamaan pyyntö syöttöruutuun
3. **lähetä** - Ohjaa käyttäjää napsauttamaan lähetyspainiketta
4. **proposal-launch-readme** - Ohjaa käyttäjää luomaan README-ehdotus
5. **proposal-compose-readme** - Ohjaa käyttäjää muokkaamaan README-pyynnön sisältöä
6. **proposal-submit-readme** - Ohjaa käyttäjää lähettämään README-ehdotus
7. **proposal-launch-agents** - Ohjaa käyttäjää luomaan `AGENTS.md` ehdotus
8. **proposal-compose-agents** - Ohjaa käyttäjää muokkaamaan `AGENTS.md` pyytää
9. **proposal-submit-agents** – Ohjaa käyttäjää lähettämään `AGENTS.md` ehdotus
10. **proposal-wait** - Selitä, että tekoäly käsittelee ja käyttäjän tulee odottaa hetki

Tämän polun ideana on antaa käyttäjien kokea HagiCoden ydintyönkulku omakohtaisesti kahden todellisen ehdotusten luontitehtävän kautta, joista toinen `README` ja yksi varten `AGENTS.md`. On suuri ero sen välillä, että kuulet työnkulusta ja käyt sen läpi itse.

### Ehdotuksen yksityiskohdat perehdyttäminen (3 vaihetta)

Kun käyttäjät tulevat ehdotuksen tietosivulle, HagiCode käynnistää vastaavan oppaan ehdotuksen nykyisen tilan perusteella:

1. **luonnos** (luonnosvaihe) – Ohjaa käyttäjää tarkistamaan tekoälyn luoma suunnitelma
2. **tarkistus** (tarkistusvaihe) – Ohjaa käyttäjää toteuttamaan suunnitelma
3. **executionCompleted** (valmistunut vaihe) - Ohjaa käyttäjää arkistoimaan suunnitelma

Tämän oppaan ominaispiirre on, että se on tilaohjattu: se päättää dynaamisesti, mikä aloitusvaihe näytetään ehdotuksen todellisen tilan perusteella. Asiat muuttuvat, ja perehdyttämisen pitäisi muuttua niiden mukana.

### Elementin renderöinnin uudelleenyritysmekanismi

React-sovelluksissa kohdeonboarding-elementti ei ehkä ole vielä valmis renderöimään, esimerkiksi koska asynkroninen data latautuu edelleen. Tämän käsittelemiseksi HagiCode toteuttaa uudelleenyritysmekanismin:

```typescript
const waitForElement = (selector: string, maxRetries = 10, interval = 100) => {
  let retries = 0;
  return new Promise<HTMLElement>((resolve, reject) => {
    const checkElement = () => {
      const element = document.querySelector(selector) as HTMLElement;
      if (element) {
        resolve(element);
      } else if (retries < maxRetries) {
        retries++;
        setTimeout(checkElement, interval);
      } else {
        reject(new Error(`Element not found: ${selector}`));
      }
    };
    checkElement();
  });
};
```

Kutsu tämä toiminto ennen oppaan alustamista varmistaaksesi, että kohdeelementti on jo olemassa. Joskus kannattaa odottaa vähän kauemmin.

## Yhteenveto parhaista käytännöistä

HagiCoden käytännön kokemuksen perusteella tässä on muutamia keskeisiä parhaita käytäntöjä:

### 1. Onboardingin tulisi olla helppo lopettaa

Älä pakota käyttäjiä suorittamaan käyttöönottoa loppuun. Jotkut käyttäjät ovat luonteeltaan tutkimusmatkailijoita ja haluavat selvittää asioita itse. Anna selkeä "Ohita"-painike, muista heidän valintansa äläkä keskeytä sitä uudelleen seuraavalla kerralla.

### 2. Pidä aloitussisältö lyhyenä ja terävänä

Jokaisen käyttöönottovaiheen tulisi keskittyä yhteen tavoitteeseen:

- **Otsikko**: Lyhyt ja selkeä, mieluiten vain muutama sana
- **Kuvaus**: Siirry suoraan asiaan ja kerro käyttäjälle, mikä tämä on ja miksi sillä on merkitystä

Vältä pitkiä selityksiä. Käyttäjän huomio on rajallista perehdyttämisen aikana, ja mitä enemmän sanot, sitä epätodennäköisemmin he lukevat sen.

### 3. Pidä valitsimet vakaina

Käytä pysyviä elementtimerkkejä, jotka eivät muutu usein. Mukautettu `data-guide` ominaisuus on hyvä valinta. Vältä riippuvuutta luokkien nimistä tai DOM-rakenteesta, koska niitä on helppo muuttaa refaktoreiden aikana. Koodi muuttuu koko ajan, mutta joidenkin ankkurien tulisi pysyä vakaina, kun mahdollista.

### 4. Testaa perehtymistäsi

HagiCode sisältää täydelliset testitapaukset käyttöönottoominaisuudesta:

```typescript
describe('NewUserConversationGuide', () => {
  it('should initialize guide state correctly', () => {
    const state = getUserGuideState();
    expect(state.session).toBe('pending');
  });

  it('should update guide state correctly', () => {
    setUserGuideState({ session: 'completed', detailGuides: {} });
    const state = getUserGuideState();
    expect(state.session).toBe('completed');
  });
});
```

Testit auttavat varmistamaan, että refaktorointi ei vahingossa riko aloituskäyttäytymistä. Kukaan ei halua pienen koodin muutoksen vahingoittavan hiljaa aiemmin toimivia toimintoja.

### 5. Suorituskyvyn optimointi

- Käytä dynaamisia tuontia kirjaston laiskaan lataamiseen
- Vältä aloituslogiikan alustamista sen jälkeen, kun käyttäjä on jo suorittanut oppaan
- Harkitse animaatioiden suorituskykyä ja poista ne käytöstä alemmissa laitteissa tarvittaessa

Suorituskyky, kuten monet asiat elämässä, ansaitsee hieman huolellista budjetointia.

## Johtopäätös

Uusien käyttäjien käyttöönotto on tärkeä osa tuotteen käyttökokemuksen parantamista. HagiCode-projektissa käytimme driver.js:ää rakentaaksemme täydellisen käyttöönottojärjestelmän, joka kattaa koko työnkulun istunnon luomisesta ehdotuksen toteuttamiseen.

Ydinkohdat, jotka toivomme jakavamme tämän artikkelin kautta, ovat:

1. **Teknisten valintojen tulee vastata todellisia tarpeita**: driver.js ei ole tehokkain vaihtoehto, mutta se sopii meille parhaiten
2. **Valtionhallinta on kriittistä**: Käytä `localStorage` pysyä aloitustilassa ja välttää käyttäjien toistuvia keskeytyksiä
3. **Perussuunnittelun tulee pysyä keskittyneenä**: Jokaisen vaiheen tulee ratkaista yksi ongelma, ei enempää
4. **Koodirakenteen tulee pysyä selkeänä**: Erillinen käyttöönottomääritys, tilanhallinta ja käyttöliittymälogiikka ylläpidon helpottamiseksi

Jos olet lisäämässä uutta käyttäjää omaan projektiisi, toivon tämän artikkelin käytännön kokemuksesta apua. Tällaisessa tekniikassa ei ole mitään erityisen mystistä. Jatka yrittämistä, tee yhteenveto oppimastasi, ja asiat helpottuvat vähitellen.
