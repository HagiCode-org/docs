## Hermes-agentin integrointikäytäntö: protokollasta tuotantoon

> Jaamme täydellisen HagiCode-kokemuksen Hermes Agentin integroinnista, mukaan lukien ydinoppitunnit ACP-protokollan mukauttamisesta, istuntovarannon hallinnasta ja käyttöliittymän ja taustan sopimussynkronointista.

## Tausta

Kun rakennamme HagiCodea, tekoälyavusteista koodausalustaa, tiimimme piti integroida Agent-kehys, joka voisi toimia paikallisesti ja myös skaalata pilveen. Tutkimuksen jälkeen Nous Researchin Hermes Agent valittiin yleiskäyttöisten agenttitoimintojemme taustalla olevaksi moottoriksi.

Itse asiassa teknologian valinta ei ole erityisen vaikeaa eikä erityisen helppoa. Markkinoilla on runsaasti vahvoja Agent-kehyksiä, mutta Hermes erottui joukosta, koska sen ACP-protokolla ja työkalujärjestelmä sopivat erityisen hyvin HagiCoden vaativiin vaatimuksiin: paikalliseen kehittämiseen, tiimiyhteistyöhön ja pilvilaajennukseen. Silti Hermeksen saattaminen todelliseen tuotantojärjestelmään merkitsi pitkän listan teknisten ongelmien ratkaisemista. Tämä osa oli kaikkea muuta kuin triviaalia.

HagiCoden pino käyttää Orleansia hajautetun järjestelmän rakentamiseen, kun taas käyttöliittymä on rakennettu React + TypeScriptillä. Hermesin integroiminen tarkoitti arkkitehtonisen johdonmukaisuuden säilyttämistä samalla kun Hermesistä tuli ensiluokkainen toteuttaja ClaudeCoden ja OpenCoden rinnalla. Se kuulostaa riittävän yksinkertaiselta, mutta toteutus kertoo aina todellisen tarinan.

Tämä artikkeli jakaa käytännön kokemuksemme Hermes Agentin integroimisesta HagiCodeen, ja toivomme, että se tarjoaa hyödyllistä viitemateriaalia tiimeille, jotka kohtaavat samanlaisia tarpeita. Loppujen lopuksi, kun olet pudonnut kuoppaan, ei ole mitään syytä antaa jonkun toisen pudota samaan.

## Tietoja HagiCodesta

Tässä artikkelissa kuvattu ratkaisu on peräisin käytännön työstämme [HagiCode](https://hagicode.com) hanke. HagiCode on tekoälyyn perustuva koodausapualusta, joka tukee useiden tekoälypalvelujen tarjoajien yhtenäistä pääsyä ja hallintaa. Hermes Agent -integraation aikana suunnittelimme yleisen Provider-abstraktiokerroksen, jotta uudet agenttityypit voisivat kytkeytyä olemassa olevaan järjestelmään saumattomasti.

Jos olet kiinnostunut HagiCodesta, tule käymään [GitHub](https://github.com/HagiCode-org/site) oppiaksesi lisää. Mitä enemmän ihmisiä kiinnittää huomiota, sitä vahvempi on vauhti.

## Arkkitehtuurisuunnittelu

### Kerrosmainen suunnittelu lähestymistapa

HagiCoden Hermes-integraatio käyttää selkeää kerrostettua arkkitehtuuria, jossa jokainen kerros keskittyy omiin tehtäviinsä:

**Takapään ydinkerros**
- `HermesCliProvider`: toteuttaa `IAIProvider` käyttöliittymä yhtenäisenä AI Provider -tulopisteenä
- `HermesPlatformConfiguration`: hallitsee Hermesin suorituspolkua, argumentteja, todennusta ja niihin liittyviä asetuksia
- `ICliProvider<HermesOptions>`: HagiCode.Libsin tarjoama matalan tason CLI-abstraktio aliprosessien elinkaarien käsittelyyn

**Kuljetuskerros**
- `StdioAcpTransport`: kommunikoi Hermes ACP -aliprosessin kanssa vakiotulon ja -lähdön kautta
- ACP-protokollamenetelmät: `initialize`, `authenticate`, `session/new`, `session/prompt`

**Ajonaikainen kerros**
- `HermesGrain`: Orleans Grain -toteutus, joka hoitaa hajautetun istunnon suorittamisen
- `CliAcpSessionPool`: istuntopooli, joka käyttää uudelleen ACP-aliprosesseja välttääkseen toistuvan käynnistyksen ylimääräisiä kustannuksia

**Etukerros**
- `ExecutorAvatar`: Hermes visuaalinen identiteetti ja ikoni
- `executorTypeAdapter`: Palveluntarjoajan tyypin kartoituslogiikka
- SignalR:n reaaliaikainen viestintä: ylläpitää Hermes-identiteetin johdonmukaisuutta koko viestivirran ajan

Tämä kerrosrakenne mahdollistaa jokaisen kerroksen kehittymisen itsenäisesti. Jos esimerkiksi haluamme lisätä tulevaisuudessa uuden siirtomekanismin, kuten WebSocketin, meidän tarvitsee vain muokata kuljetuskerrosta. Koko järjestelmää ei tarvitse kääntää vain siksi, että yksi kuljetus vaihtuu.

### Yhtenäisen käyttöliittymän abstraktio

Kaikki AI-palveluntarjoajat toteuttavat `IAIProvider` käyttöliittymä, joka on yksi HagiCoden arkkitehtuurin tärkeimmistä suunnitteluvaihtoehdoista:

```csharp
public interface IAIProvider
{
    string Name { get; }
    ProviderCapabilities Capabilities { get; }

    IAsyncEnumerable<AIStreamingChunk> StreamAsync(
        AIRequest request,
        CancellationToken cancellationToken = default);

    Task<AIResponse> ExecuteAsync(
        AIRequest request,
        CancellationToken cancellationToken = default);
}
```

`HermesCliProvider` toteuttaa tämän käyttöliittymän ja on tasavertainen `ClaudeCodeProvider`, `OpenCodeProvider`, ja muut. Tämän suunnittelun etuja ovat:

1. **Vaihdettavuus**: Palveluntarjoajan vaihtaminen ei vaikuta ylemmän tason liiketoimintalogiikkaan
2. **Testattavuus**: Palveluntarjoajia voidaan pilkata helposti yksikkötestauksen vuoksi
3. **Laajennettavuus**: uuden palveluntarjoajan lisääminen edellyttää vain käyttöliittymän käyttöönottoa

Loppujen lopuksi käyttöliittymät ovat paljon kuin säännöt. Kun säännöt ovat paikoillaan, kaikki voivat elää harmonisesti rinnakkain, pelata vahvuuksilla ja välttää astumasta toistensa päälle. Siinä on tiettyä eleganssia.

## Ydintoteutus

### Palveluntarjoajan kerroksen toteutus

`HermesCliProvider` on koko integraation ydin. Se koordinoi eri komponentteja, joita tarvitaan yhden tekoälyn kutsun suorittamiseen:

```csharp
public sealed class HermesCliProvider : IAIProvider, IVersionedAIProvider
{
    private readonly ICliProvider<LibsHermesOptions> _provider;
    private readonly ConcurrentDictionary<string, string> _sessionBindings;

    public ProviderCapabilities Capabilities { get; } = new()
    {
        SupportsStreaming = true,
        SupportsTools = true,
        SupportsSystemMessages = true,
        SupportsArtifacts = false
    };

    public async IAsyncEnumerable<AIStreamingChunk> StreamAsync(
        AIRequest request,
        [EnumeratorCancellation] CancellationToken cancellationToken = default)
    {
        // 1. Resolve the session binding key
        var bindingKey = ResolveBindingKey(request.CessionId);

        // 2. Get or create a Hermes session through the session pool
        var options = new HermesOptions
        {
            ExecutablePath = _platformConfiguration.ExecutablePath,
            Arguments = _platformConfiguration.Arguments,
            SessionId = _sessionBindings.TryGetValue(bindingKey, out var sessionId) ? sessionId : null,
            WorkingDirectory = request.WorkingDirectory,
            Model = request.Model
        };

        // 3. Execute and collect the streaming response
        await foreach (var message in _provider.ExecuteAsync(options, request.Prompt, cancellationToken))
        {
            // 4. Map ACP messages to AIStreamingChunk
            if (_responseMapper.TryConvertToStreamingChunk(message, out var chunk))
            {
                yield return chunk;
            }
        }
    }
}
```

Useat suunnittelukohdat ovat erityisen tärkeitä tässä:

1. **Istunnon sitominen**: käyttötarkoitukset `CessionId` sitoa useita pyyntöjä samaan Hermes-aliprosessiin, mikä säilyttää kontekstin jatkuvuuden monikäänteisissä keskusteluissa
2. **Vastauskartoitus**: muuntaa Hermes ACP -viestimuodon yhtenäiseksi `AIStreamingChunk` muodossa
3. **Suoratoistotuki**: käyttötarkoitukset `IAsyncEnumerable` todellisten suoratoistovastausten tukemiseksi

Istunnon sitominen on vähän kuin ihmissuhteita. Kun yhteys on muodostettu, tulevalla viestinnällä on konteksti, joten sinun ei tarvitse aloittaa nollasta joka kerta. Tietysti tämä suhde on edelleen säilytettävä.

### ACP-protokollan sovitus

Hermes käyttää ACP:tä (Agent Communication Protocol), joka eroaa perinteisestä HTTP API:sta. ACP on standardituloon ja -lähtöön perustuva protokolla, ja sillä on useita ominaisuuksia:

1. **Käynnistysmerkki**: Hermes-prosessin alkamisen jälkeen se tulostaa `//ready` merkki
2. **Dynaaminen todennus**: todennusmenetelmät eivät ole kiinteitä ja niistä on neuvoteltava protokollan kautta
3. **Istunnon uudelleenkäyttö**: vakiintuneita istuntoja käytetään uudelleen `SessionId`
4. **Hajanaiset vastaukset**: täydellinen vastaus voidaan jakaa useisiin `session/update` ilmoituksia

HagiCode käsittelee nämä ominaisuudet `StdioAcpTransport`:

```csharp
public class StdioAcpTransport
{
    public async Task InitializeAsync(CancellationToken cancellationToken)
    {
        // Wait for the //ready marker
        var readyLine = await _outputReader.ReadLineAsync(cancellationToken);
        if (readyLine != "//ready")
        {
            throw new InvalidOperationException("Hermes did not send ready signal");
        }

        // Send the initialize request
        await SendRequestAsync(new
        {
            jsonrpc = "2.0",
            id = 1,
            method = "initialize",
            @params = new
            {
                protocolVersion = "2024-11-05",
                capabilities = new { },
                clientInfo = new { name = "HagiCode", version = "1.0.0" }
            }
        }, cancellationToken);
    }
}
```

Protokollat ovat vähän kuin ihmisten välinen keskinäinen ymmärrys. Kun tämä ymmärrys on olemassa, viestintä sujuu paljon sujuvammin. Sen rakentaminen vie vain aikaa.

### Istuntopoolin hallinta

Hermes-aliprosessien käynnistäminen usein on kallista, joten otimme käyttöön istuntopoolimekanismin:

```csharp
services.AddSingleton(static _ =>
{
    var registry = new CliProviderPoolConfigurationRegistry();
    registry.Register("hermes", new CliPoolSettings
    {
        MaxActiveSessions = 50,
        IdleTimeout = TimeSpan.FromMinutes(10)
    });
    return registry;
});
```

Keskeiset istuntovarannon parametrit:
- `MaxActiveSessions`: hallitsee samanaikaisuusrajaa resurssien kulumisen välttämiseksi
- `IdleTimeout`: tyhjäkäynnin aikakatkaisu, joka tasapainottaa käynnistyskustannukset muistin käytön kanssa

Käytännössä huomasimme, että:
1. Jos tyhjäkäynnin aikakatkaisu on liian lyhyt, istunnot alkavat uudelleen usein; jos se on liian pitkä, muisti pysyy varattuna
2. Samanaikaisuusraja on viritettävä todellisen kuormituksen mukaan, koska sen asettaminen liian korkeaksi voi tehdä järjestelmän hitaaksi
3. Istuntopoolin käyttöä on seurattava, jotta parametreja voidaan säätää ajoissa

Tämä on samanlaista kuin monet valinnat elämässä: liian aggressiivinen oleminen aiheuttaa ongelmia, kun taas liian konservatiivinen menettää mahdollisuuksia. Tavoitteena on vain löytää oikea tasapaino.

## Käyttöliittymän integrointi

### Tyyppikartoitus

Käyttöliittymän on tunnistettava oikein Hermes-palveluntarjoaja ja näytettävä vastaavat visuaaliset elementit:

```typescript
// executorTypeAdapter.ts
export const resolveExecutorVisualTypeFromProviderType = (
  providerType: PCode_Models_AIProviderType | null | undefined
): ExecutorVisualType => {
  switch (providerType) {
    case PCode_Models_AIProviderType.HERMES_CLI:
      return 'Hermes';
    default:
      return 'Unknown';
  }
};
```

Loppujen lopuksi kauniit asiat ansaitsevat kauniin esityksen. Sen varmistaminen, että kauneus on todella näkyvissä, riippuu edelleen käsityön tasosta.

### Sopimuksen synkronointi

Etu- ja takapää pitävät sopimuksensa linjassa OpenAPI-sukupolven ajan. Takapää määrittelee `AIProviderType` luettelo:

```csharp
public enum AIProviderType
{
    Unknown,
    ClaudeCode,
    OpenCode,
    HermesCli  // Newly added
}
```

Käyttöliittymä luo vastaavan TypeScript-tyypin OpenAPI:n kautta, mikä varmistaa, että enum-arvot pysyvät johdonmukaisina. Tämä on avain etupään näyttämisen välttämiseen `Unknown`.

Sopimus on paljon kuin lupaus. Kun kerran on sovittu, sitä on kunnioitettava, muuten päädyt kiusallisiin tilanteisiin, kuten `Unknown`.

## Kokoonpanon hallinta

Hermes-kokoonpanoa hallitaan `appsettings.json`:

```json
{
  "Providers": {
    "HermesCli": {
      "ExecutablePath": "hermes",
      "Arguments": "acp",
      "StartupTimeoutMs": 10000,
      "ClientName": "HagiCode",
      "Authentication": {
        "PreferredMethodId": "api-key",
        "MethodInfo": {
          "api-key": "your-api-key-here"
        }
      },
      "SessionDefaults": {
        "Model": "claude-sonnet-4-20250514",
        "ModeId": "default"
      }
    }
  }
}
```

Tämä konfiguraatiolähtöinen suunnittelu tuo joustavuutta:
- suoritettavat polut voidaan ohittaa, mikä on kätevää kehitystä ja testausta varten
- käynnistysargumentit voidaan mukauttaa vastaamaan eri Hermes-versioita
- todennustiedot voidaan konfiguroida tukemaan useita todennusmenetelmiä

Kokoonpano on vähän kuin monivalintakysymyksiä elämässä. Jos vaihtoehtoja on riittävästi, löytyy yleensä yksi, joka sopii. Liian monet vaihtoehdot voivat kuitenkin aiheuttaa päätöksentekoväsymystä.

## Käytännön kokemus

### Terveystarkastukset

Luotettavan palveluntarjoajan rakentaminen vaatii kattavat terveystarkastukset:

```csharp
public async Task<ProviderTestResult> PingAsync(CancellationToken cancellationToken = default)
{
    var response = await ExecuteAsync(new AIRequest
    {
        Prompt = "Reply with exactly PONG.",
        CessionId = null,
        AllowedTools = Array.Empty<string>(),
        WorkingDirectory = ResolveWorkingDirectory(null)
    }, cancellationToken);

    var success = string.Equals(response.Content.Trim(), "PONG", StringComparison.OrdinalIgnoreCase);
    return new ProviderTestResult
    {
        ProviderName = Name,
        Success = success,
        ResponseTimeMs = stopwatch.ElapsedMilliseconds,
        ErrorMessage = success ? null : $"Unexpected Hermes ping response: '{response.Content}'."
    };
}
```

Terveystarkastuksissa huomioitavaa:
1. Käytä yksinkertaisia testitapauksia ja vältä liian monimutkaisia skenaarioita
2. Aseta kohtuulliset aikakatkaisuarvot
3. Tallenna vasteaika suorituskykyanalyysin tukemiseksi

Aivan kuten ihmiset tarvitsevat fyysisiä tarkastuksia, myös järjestelmät tarvitsevat terveystarkastuksia. Mitä nopeammin ongelmat löydetään, sitä helpompi ne on korjata.

### Validointityökalu

HagiCode tarjoaa erillisen konsolin Hermes-integraation vahvistamiseen:

```bash
# Basic validation
HagiCode.Libs.Hermes.Console --test-provider

# Full suite (including repository analysis)
HagiCode.Libs.Hermes.Console --test-provider-full --repo .

# Custom executable
HagiCode.Libs.Hermes.Console --test-provider-full --executable /path/to/hermes
```

Tämä työkalu on erittäin hyödyllinen kehityksen aikana, koska sen avulla voimme nopeasti tarkistaa, onko integraatio oikein. Loppujen lopuksi kukaan ei halua odottaa, kunnes ongelma tulee esiin ennen kuin muistaa testata.

### Yleisten ongelmien käsittely

**Todennusvirhe**
- Tarkista onko `Authentication.PreferredMethodId` vastaa todennusmenetelmää, jota Hermes todella tukee
- Varmista, että todennustietojen muoto, kuten API-avain tai Bearer Token, on oikea

**Istunnon aikakatkaisu**
- Lisää `StartupTimeoutMs` arvoa
- Tarkista MCP-palvelimen saavutettavuus
- Tarkista järjestelmän resurssien käyttö

**Epätäydellinen vastaus**
- Varmista `session/update` ilmoitukset ja lopputulos on koottu oikein
- Tarkista peruutuslogiikka suoratoiston käsittelyssä
- Varmista, että virheiden käsittely on valmis

**Etupään näytöt Tuntematon**
- Vahvista, että OpenAPI-sukupolvi sisältää jo `HermesCli` enum-arvo
- Tarkista, onko tyyppikartoitus oikea
- Tyhjennä selaimen välimuisti ja luo tyypit uudelleen

Ongelmia tulee aina olemaan. Kun ne ilmestyvät, tärkeintä on olla panikoimatta. Jäljitä syy askel askeleelta, ja lopulta useimmat niistä voidaan ratkaista.

### Suorituskyvyn optimointiehdotuksia

1. **Käytä istuntopoolia**: käytä ACP-aliprosesseja uudelleen vähentääksesi käynnistyskustannuksia
2. **Aseta aikakatkaisut asianmukaisesti**: tasapainota muistin käyttö käynnistyskustannuksia vastaan
3. **Käytä istuntotunnuksia uudelleen**: käytä samaa `CessionId` erätehtäviin
4. **Määritä MCP tarpeen mukaan**: vältä tarpeettomia työkalukutsuja

Suorituskyky on paljon kuin tehokkuus jokapäiväisessä elämässä. Kun saat sen oikein, saavutat enemmän vähemmällä; kun teet sen väärin, vaiva moninkertaistuu, kun taas tulokset heikkenevät. Sen "oikean" pisteen löytäminen vaatii sekä kokemusta että onnea.

## Johtopäätös

Hermes Agentin integroiminen tuotantojärjestelmään edellyttää useiden eri ulottuvuuksien ongelmien huomioon ottamista:

1. **Arkkitehtuuri**: suunnittele yhtenäinen Provider-liittymä ja toteuta vaihdettava komponenttiarkkitehtuuri
2. **Protokolla**: käsittelee oikein ACP-spesifistä toimintaa, kuten käynnistysmerkit ja dynaaminen todennus
3. **Suorituskyky**: käytä resursseja uudelleen istuntopoolin kautta ja tasapainota käynnistyskustannukset muistinkäytön kanssa
4. **Etupää**: varmista sopimussynkronointi ja johdonmukainen visuaalinen kokemus

HagiCoden kokemus osoittaa, että hyvällä kerroksellisella suunnittelulla ja konfiguraatiolähtöisellä toteutuksella monimutkainen agenttijärjestelmä voidaan integroida saumattomasti olemassa olevaan arkkitehtuuriin.

Nämä periaatteet kuulostavat yksinkertaisilta sanoin kuvattuna, mutta varsinaiseen toteutukseen liittyy aina monenlaisia ongelmia. Se on hyvä. Jos ongelma ratkeaa, siitä tulee kokemusta. Jos ei, siitä tulee opetus. Joka tapauksessa sillä on edelleen arvoa.

Kauniita asioita tai ihmisiä ei tarvitse omistaa; niin kauan kuin ne pysyvät kauniina, riittää, että arvostat sitä kauneutta hiljaa. Tekniikka on pitkälti samaa. Jos se auttaa parantamaan järjestelmää, tietyllä viitekehyksellä tai protokollalla on paljon vähemmän merkitystä kuin ihmiset joskus ajattelevat.
