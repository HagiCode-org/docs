## Nopea koodin muokkaus selaimessa: VSCode Web Integration käytännössä

> Kun tekoäly on lopettanut koodin analysoinnin, kuinka voit avata editorin välittömästi selaimessa ja aloittaa muutosten tekemisen? Tämä artikkeli jakaa käytännön kokemuksemme koodipalvelimen integroimisesta HagiCode-projektiin luodaksemme saumattoman sillan tekoälyavustajan ja koodinmuokkauskokemuksen välille.

## Tausta

Tekoälyavusteisen ohjelmoinnin aikakaudella kehittäjien on usein tarkastettava ja muokattava koodia nopeasti. Perinteinen työnkulku on yksinkertainen: avaa projekti työpöydän IDE:ssä, etsi tiedosto, muokkaa sitä ja tallenna. Mutta joissakin tilanteissa tämä virtaus tuntuu aina hieman epätasapainoiselta.

Skenaario yksi: etäkehitys. Käytettäessä AI-avustajaa, kuten HagiCode, taustaohjelma voi olla käynnissä etäpalvelimella tai säilön sisällä, eivätkä paikalliset koneet pääse suoraan projektitiedostoihin. Joka kerta, kun sinun on tarkastettava tai muutettava koodia, sinun on muodostettava yhteys SSH:n tai muun menetelmän kautta, ja kokemus tuntuu pirstoutuneelta. Se on kuin haluaisi tavata jonkun paksun lasin läpi: näet hänet, mutta et tavoita häntä.

Skenaario kaksi: nopeat esikatselut. Kun tekoälyavustaja on analysoinut koodin, käyttäjä saattaa haluta vain selata tiedostoa nopeasti tai tehdä pienen muutoksen. Täydellisen työpöydän IDE:n käynnistäminen tuntuu raskaalta, kun taas kevyt selaimen sisäinen editori sopii paremmin "nopeaan ulkoasun" tarpeeseen. Loppujen lopuksi, kuka haluaa mobilisoida koko työkaluketjun vain katsoakseen?

Kolmas skenaario: laitteiden välinen yhteistyö. Kun työskentelet eri laitteilla, selainpohjainen editori tarjoaa yhtenäisen tukiaseman ilman, että jokaiselle koneelle on määritettävä kehitysympäristö. Pelkästään se säästää monelta vaivalta. Elämä on lyhyt; miksi toistaa samaa asennustyötä uudestaan ​​​​ja uudestaan?

Näiden kipupisteiden ratkaisemiseksi integroimme VSCode Webin HagiCode-projektiin. Näin tekoälyavustaja ja koodinmuokkauskokemus yhdistyvät saumattomasti: kun tekoäly on analysoinut koodin, käyttäjät voivat heti avata editorin ja tehdä muutoksia samassa selainistunnossa vaihtamatta sovelluksia. Se on sellainen kokemus, jossa se on yksinkertaisesti olemassa, kun sitä tarvitset.

## Tietoja HagiCodesta

Tässä artikkelissa jaettu ratkaisu on peräisin käytännön kokemuksestamme [HagiCode](https://hagicode.com) hanke. HagiCode on tekoälyyn perustuva koodausassistentti, joka on suunniteltu parantamaan kehityksen tehokkuutta luonnollisen kielen vuorovaikutuksen avulla. Kehityksen aikana havaitsimme, että käyttäjien on usein vaihdettava nopeasti tekoälyanalyysin ja koodin muokkaamisen välillä, mikä pakotti meidät tutkimaan, kuinka editori voidaan integroida suoraan selaimeen.

Projektin arkisto: [github.com/HagiCode-org/site](https://github.com/HagiCode-org/site)

## Tekninen valinta: Miksi koodipalvelin?

Monien saatavilla olevien VSCode Web -ratkaisujen joukosta valitsimme koodipalvelimen. Päätöksen taustalla oli muutama konkreettinen syy.

Ominaisuuden täydellisyys. code-server on VSCoden verkkoversio ja tukee useimpia työpöytäominaisuuksia, mukaan lukien laajennusjärjestelmä, älykkäät ehdotukset, virheenkorjaus ja paljon muuta. Tämä tarkoittaa, että käyttäjät voivat saada muokkauskokemuksen selaimessa, joka on hyvin lähellä työpöytäversiota. Loppujen lopuksi, kuka todella haluaa tehdä kompromisseja toimivuudesta?

Joustava käyttöönotto. koodipalvelin voi toimia itsenäisenä palveluna ja tukee myös Docker-pohjaista käyttöönottoa, joka sopii hyvin HagiCoden arkkitehtuuriin. Taustajärjestelmämme on kirjoitettu C#:lla, käyttöliittymä käyttää Reactia ja molemmat kommunikoivat koodipalvelinpalvelun kanssa REST API:iden kautta. Se on kuin lohkoista rakentamista: jokaisella palalla on paikkansa.

Turvallinen todennus. koodipalvelin sisältää sisäänrakennetun `connection-token` mekanismi luvattoman pääsyn estämiseksi. Jokaisella istunnolla on yksilöllinen tunnus, joten vain valtuutetut käyttäjät voivat avata editorin. Turvallisuus on yksi niistä asioista, joita arvostat täysin vasta saatuaan sen.

## Arkkitehtuurisuunnittelu

HagiCoden VSCode Web -integraatio käyttää etupäästä/taustasta erotettua arkkitehtuuria.

### Frontend Service Layer

Käyttöliittymä käärii vuorovaikutuksen taustajärjestelmän kanssa läpi `vscodeServerService.ts`:

```typescript
// Open project
export async function openProjectInCodeServer(
  id: string,
  currentInterfaceLanguage?: string,
): Promise<VsCodeServerLaunchResponseDto>

// Open vault
export async function openVaultInCodeServer(
  id: string,
  path?: string,
  currentInterfaceLanguage?: string,
): Promise<VsCodeServerLaunchResponseDto>
```

Ero näiden kahden menetelmän välillä on selvä: `openProjectInCodeServer` avaa koko projektin `openVaultInCodeServer` avaa tietyn polun Holvin sisällä. MonoSpecs-monivarastoprojekteille järjestelmä luo automaattisesti työtilatiedoston. Selkeät vastuut riittävät usein, kun jokainen osa tekee oman työnsä hyvin.

### Backend Service Layer

Taustaosa `VaultAppService.cs` toteuttaa ydinlogiikkaa:

```csharp
public async Task<VsCodeServerLaunchResponseDto> OpenInCodeServerAsync(
    string id,
    string? relativePath = null,
    string? currentInterfaceLanguage = null,
    CancellationToken cancellationToken = default)
{
    // 1. Get settings and check whether the feature is enabled
    var settings = await _vsCodeServerSettingsService.GetResolvedSettingsAsync(cancellationToken);
    if (!settings.Enabled) {
        throw new BusinessException(VsCodeServerErrorCodes.Disabled, "VSCode Server is disabled.");
    }

    // 2. Get vault and resolve the launch directory
    var vault = await RequireVaultAsync(id, cancellationToken);
    var launchDirectory = ResolveLaunchDirectory(vault, relativePath);

    // 3. Ensure code-server is running and get runtime info
    var runtime = await _vsCodeServerManager.EnsureStartedAsync(settings, cancellationToken);

    // 4. Resolve language settings
    var language = _vsCodeServerSettingsService.ResolveLaunchLanguage(
        settings.Language,
        currentInterfaceLanguage);

    // 5. Build launch URL
    return new VsCodeServerLaunchResponseDto {
        LaunchUrl = AppendQueryString(runtime.BaseUrl, new Dictionary<string, string?> {
            ["folder"] = launchDirectory,
            ["tkn"] = runtime.ConnectionToken,
            ["vscode-lang"] = language,
        }),
        ConnectionToken = runtime.ConnectionToken,
        OpenMode = "folder",
        Runtime = VsCodeServerSettingsService.MapRuntime(
            await _vsCodeServerManager.GetRuntimeSnapshotAsync(cancellationToken)),
    };
}
```

Tällä menetelmällä on hyvin selkeä vastuu: tarkista asetukset, ratkaise polut, käynnistä palvelu ja luo URL-osoite. Heidän joukossaan on `ResolveLaunchDirectory` menetelmä suorittaa polun suojaustarkistuksia estääkseen polun läpikulkuhyökkäykset. Koodi voi tuntua vähän runolta, kun jokaisella rivillä on tarkoitus.

### Automaattinen ajonajan hallinta

Taustaosa hallitsee koodipalvelinprosessia `VsCodeServerManager`:

- Tarkista prosessin tila
- Käynnistä pysäytetyt palvelut automaattisesti
- Palauta ajonaikaiset tilannevedokset, kuten portti, prosessitunnus ja aloitusaika

Tämän rakenteen ansiosta järjestelmä pystyy käsittelemään automaattisesti koodipalvelimen elinkaaren, joten käyttäjien ei tarvitse hallita palveluprosesseja manuaalisesti. Elämä on jo tarpeeksi monimutkaista; kaiken automatisoitavan pitäisi olla.

### Kielen synkronointi

HagiCode tukee monikielistä käyttöliittymää, ja koodipalvelimen on noudatettava tätä asetusta. Järjestelmä tukee kolmea kielitilaa:

- `follow`: seuraa nykyistä käyttöliittymän kieltä
- `zh-CN`: kiinteä kiinaksi
- `en-US`: kiinteä englanniksi

Asetus välitetään koodipalvelimelle `vscode-lang` URL-parametri, jotta editorin kieli pysyy yhtenäisenä HagiCode-käyttöliittymän kanssa. Kieli tuntuu parhaalta, kun se on yhtenäinen.

### MonoSpecs Multi-Repository Workspace

MonoSpecs-projekteille, jotka sisältävät useita alivarastoja yhden monorepon sisällä, järjestelmä luo automaattisesti `.code-workspace` tiedosto:

```csharp
private async Task<string> CreateWorkspaceFileAsync(Project project, Guid projectId)
{
    var folders = await ResolveWorkspaceFoldersAsync(project.Path);
    var workspaceDocument = new {
        folders = folders.Select(path => new { path }).ToArray(),
    };
    // Generate workspace file...
}
```

Tämä mahdollistaa useiden alivarastojen muokkaamisen samassa koodipalvelininstanssissa, mikä on erityisen käytännöllistä suurissa monorepo-projekteissa. Useat arkistot yhdessä ikkunassa voivat tuntua useilta tarinoilta, jotka on koottu samaan kirjaan.

## Käyttöliittymän integrointi

HagiCode-käyttöliittymä käyttää React + TypeScriptiä, eikä koodipalvelimen integrointi ole erityisen monimutkaista.

### Pikatoimintopainike

Lisää koodipalvelinpainike projektikorttiin:

```tsx
// QuickActionsZone.tsx
<Button
  size="sm"
  variant="default"
  onClick={() => onAction({ type: 'open-code-server' })}
>
  <Globe className="h-3 w-3 mr-1" />
  <span className="text-xs">{t('project.openCodeServer')}</span>
</Button>
```

Tämä painike käynnistää avoimen toiminnon ja kutsuu taustaohjelman API:ta käynnistys-URL-osoitteen saamiseksi. Yksi painike, yksi toiminto, suora ja yksinkertainen.

### Avoimen toiminnon käsittely

```tsx
const handleAction = async (action: ProjectAction) => {
  if (action.type === 'open-code-server') {
    const response = await openProjectInCodeServer(project.id, i18n.language);
    window.open(response.launchUrl, '_blank', 'noopener,noreferrer');
  }
};
```

Käytä `window.open` avataksesi koodipalvelimen uudelle välilehdelle. The `noopener,noreferrer` parametrit tarjoavat lisäturvaa. Mitä tulee turvallisuuteen, ei ole olemassa sellaista asiaa kuin olla liian varovainen.

### Holvin muokkausmerkintä

Lisää samanlainen muokkauspainike Holvi-luetteloon:

```tsx
const handleEditVault = async (vault: VaultItemDto) => {
  const response = await openVaultInCodeServer(vault.id);
  window.open(response.launchUrl, '_blank', 'noopener,noreferrer');
};
```

Projektit ja holvit käyttävät samaa avointa mekanismia, mikä pitää vuorovaikutuksen johdonmukaisena. Johdonmukaisuudella on melkein yhtä paljon merkitystä kuin itse ominaisuudella.

## URL-rakennuslogiikka

Koodipalvelimen URL-muodossa on muutamia huomionarvoisia yksityiskohtia.

**Kansiotila**:
```
http://{host}:{port}/?folder={path}&tkn={token}&vscode-lang={lang}
```

**Työtilatila**:
```
http://{host}:{port}/?workspace={workspacePath}&tkn={token}&vscode-lang={lang}
```

täällä, `tkn` on yhteystunnus. Se luodaan automaattisesti aina, kun koodipalvelin käynnistyy, mikä varmistaa turvallisen pääsyn. The `vscode-lang` parametri ohjaa editorin käyttöliittymän kieltä. Jokaisella näistä parametreistä on tehtävänsä.

## Käyttöskenaariot

### Ensimmäinen skenaario: AI-avusteinen koodin tarkistus

Käyttäjä keskustelee HagiCoden kanssa, tekoäly analysoi projektikoodin ja löytää mahdollisen ongelman, minkä jälkeen käyttäjä napsauttaa "Avaa koodipalvelimessa" -painiketta avatakseen editorin suoraan selaimessa, tarkastaakseen tiedoston, korjata sen ja palatakseen HagiCodeen jatkaakseen keskustelua. Koko kulku tapahtuu selaimessa ilman sovellusten vaihtamista. Se tuntuu sileältä samalla tavalla kuin juokseva vesi tuntuu sileältä.

### Skenaario kaksi: Oppimateriaalien muokkaaminen holvissa

Käyttäjä luo Holvin avoimen lähdekoodin projektin opiskelua varten ja haluaa lisätä tutkimusmuistiinpanoja sen alle `docs/` hakemistosta. Koodipalvelimen avulla he voivat muokata Markdown-tiedostoja suoraan selaimessa, tallentaa ne ja antaa HagiCoden lukea päivitetyt muistiinpanot välittömästi. Tämä on erityisen hyödyllistä henkilökohtaisen tietopohjan rakentamisessa. Tiedosta tulee vain sitä arvokkaampaa, mitä enemmän sitä kertyy.

### Kolmas skenaario: MonoSpecs Multi-Repository Development

MonoSpecs-projekti sisältää useita alivarastoja, ja koodipalvelin luo automaattisesti monikansioisen työtilan. Selaimessa käyttäjät voivat muokata koodia useissa arkistoissa kerralla ja tehdä sitten muutoksia takaisin vastaaviin Git-tietovarastoihinsa. Tämä työnkulku sopii erityisen hyvin muutoksiin, joiden on katettava useita tietovarastoja. Useiden arkiston muokkaaminen yhdessä vaatii hieman tekniikkaa, aivan kuten useiden tehtävien käsitteleminen samanaikaisesti.

## Turvallisuusnäkökohdat

Koodipalvelinintegraatiota toteutettaessa turvallisuus ansaitsee erityistä huomiota. Jos turvallisuus menee pieleen, huomaat aina liian myöhään.

### Yhteystunnus

The `connection-token` syntyy satunnaisesti, eikä sitä pidä paljastaa. Sitä on parasta käyttää HTTPS:ssä estämään tunnuksen sieppaamisesta välimiestä. Arkaluonteiset tiedot kannattaa suojata kunnolla.

### Tiedostopolun suojaus

Taustaohjelma toteuttaa polun läpikulkutarkistuksia:

```csharp
private static string ResolveLaunchDirectory(VaultRegistryEntry vault, string? relativePath)
{
    var vaultRoot = EnsureTrailingSeparator(Path.GetFullPath(vault.PhysicalPath));
    var combinedPath = Path.GetFullPath(Path.Combine(vaultRoot, relativePath ?? "."));
    if (!combinedPath.StartsWith(vaultRoot, StringComparison.OrdinalIgnoreCase))
    {
        throw new BusinessException(VaultRelativePathTraversalCode, "Relative path traversal detected.");
    }
    return combinedPath;
}
```

Tämä koodi varmistaa, että käyttäjät eivät voi käyttää `../` tai vastaavia malleja päästäksesi käsiksi Vault-hakemiston ulkopuolisiin tiedostoihin. Rajatarkastukset on aina parempi tehdä kuin ohittaa.

### Lupien valvonta

Koodipalvelinprosessin tulee toimia asianmukaisilla käyttöoikeuksilla, jotta se ei pääse käsiksi arkaluontoisiin järjestelmätiedostoihin. On parasta käyttää koodipalvelinpalvelua erillisen käyttäjän alaisuudessa. Lupien valvonta on yksi niistä perustekijöistä, jotka sinun tulee aina pitää paikallaan.

## Suorituskyvyn optimointi

koodipalvelin kuluttaa palvelinresursseja, joten tässä on muutamia optimointiehdotuksia:

- Valvo suorittimen ja muistin käyttöä ja säädä resurssirajoja tarvittaessa
- Suuret projektit saattavat vaatia pidemmän aikakatkaisun
- Ota käyttöön automaattinen istunnon aikakatkaisun puhdistus vapauttaaksesi resursseja
- Harkitse välimuistiin tallentamista vähentääksesi toistuvaa laskentaa

HagiCode tarjoaa ajonaikaisen tilanvalvontasovellusliittymän, ja käyttöliittymä voi kutsua `getVsCodeServerSettings()` nykyisen tilan hakeminen:

```typescript
const { settings, runtime } = await getVsCodeServerSettings();
// runtime.status: 'disabled' | 'stopped' | 'starting' | 'running' | 'unhealthy'
// runtime.baseUrl: "http://localhost:8080"
// runtime.processId: 12345
```

Tämän suunnittelun avulla käyttäjät voivat ymmärtää selkeästi koodipalvelimen terveydentilan ja paikantaa nopeasti ongelmat, kun jokin menee pieleen. Kun tila on näkyvissä, ihmiset tuntevat hallitsevansa paremmin.

## Käyttäjäkokemuksen tiedot

Toteutuksen aikana havaitsimme muutamia yksityiskohtia, jotka vaikuttavat merkittävästi käyttökokemukseen ja ansaitsevat erityistä huomiota.

Koodipalvelimen avaaminen ensimmäisen kerran saattaa edellyttää käynnistyksen odottamista, ja viive voi vaihdella muutamasta sekunnista puoleen minuuttiin. On hyvä idea näyttää lataustila käyttöliittymässä, jotta käyttäjät tietävät, että järjestelmä toimii edelleen. Odottaminen on helpompaa, kun palautetta tulee.

Selaimet voivat estää ponnahdusikkunan, joten käyttäjiä tulee pyytää sallimaan se manuaalisesti. Ensimmäisen käynnistyksen yhteydessä HagiCode näyttää ohjeet, jotka selittävät tarvittavien selainkäyttöoikeuksien myöntämisen. Käyttäjäkokemus elää usein juuri näissä pienissä yksityiskohdissa.

On myös hyvä idea näyttää ajonaikainen tila, kuten käynnistys, käynnissä tai virhe, jotta käyttäjät voivat ongelmien ilmetessä nopeasti kertoa, onko ongelma palvelinpuolella vai omassa toiminnassa. Kun tiedät, missä ongelma on, voit ainakin aloittaa.

## Konfigurointiesimerkki

Koodipalvelimen konfigurointi ei ole monimutkaista:

```json
{
  "vscodeServer": {
    "enabled": true,
    "host": "0.0.0.0",
    "port": 8080,
    "language": "follow"
  }
}
```

`enabled` määrittää, onko ominaisuus käytössä, `host` ja `port` määritä kuunteluosoite ja `language` asettaa kielitilan. Näitä asetuksia voidaan muokata käyttöliittymän kautta ja ne tulevat voimaan välittömästi. Yksinkertaiset asiat ovat usein helpoimpia käyttää.
