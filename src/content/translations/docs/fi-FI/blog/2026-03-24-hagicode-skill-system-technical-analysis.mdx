## HagiCode Skill Systemin tekninen analyysi: Skaalautuvan tekoälyn taitojen hallintaalustan rakentaminen

> Tässä artikkelissa tarkastellaan perusteellisesti HagiCode-projektin Skill Management -järjestelmän arkkitehtuuria ja toteutusta. Se kattaa neljän ydinominaisuuden taustalla olevat tekniset yksityiskohdat: paikallinen globaali hallinta, markkinapaikkahaku, älykkäät suositukset ja luotettavan palveluntarjoajan hallinta.

## Tausta

Tekoälyn koodausassistenttien alalla AI-ominaisuuksien rajojen laajentaminen on aina ollut keskeinen kysymys. Claude Code itsessään on jo vahva koodiavustaja, mutta eri kehitystiimit ja erilaiset teknologiapinot tarvitsevat usein erikoisominaisuuksia tiettyihin skenaarioihin, kuten Docker-asennusten käsittelyyn, tietokannan optimointiin tai käyttöliittymäkomponenttien luomiseen. Juuri siinä Skill-järjestelmästä tulee erityisen tärkeä.

HagiCode-projektia kehitettäessä törmäsimme samanlaiseen haasteeseen: kuinka voimme antaa Claude Coden "oppia" uusia ammatillisia taitoja kuten ihminenkin, samalla kun säilytetään vankka käyttökokemus ja hyvä tekninen ylläpidettävyys? Tämä ongelma on sekä vaikea että yksinkertainen omalla tavallaan. Tämän kysymyksen ympärille suunnittelimme ja otimme käyttöön täydellisen taitojenhallintajärjestelmän.

Tässä artikkelissa käydään läpi järjestelmän tekninen arkkitehtuuri ja ydintoteutus yksityiskohtaisesti. Se on tarkoitettu kehittäjille, jotka ovat kiinnostuneita tekoälyn laajennetavuudesta ja komentorivityökalujen integroinnista. Siitä voi olla sinulle hyötyä tai ei, mutta ainakin se on nyt kirjoitettu ylös.

## Tietoja HagiCodesta

GitHub-arkisto on [HagiCode-org/site](https://github.com/HagiCode-org/site). Jos pidät tämän artikkelin teknistä lähestymistapaa arvokkaana, anna sille tähti. Loppujen lopuksi enemmän tähtiä parantaa mielialaa.

## Järjestelmäarkkitehtuurin yleiskatsaus

Skill-järjestelmä käyttää frontend-backend-eroteltua arkkitehtuuria. Siinä ei ole mitään erityisen mystistä.

**Frontend** käyttää TanStack Start + Reactia käyttöliittymän rakentamiseen Redux Toolkit -hallintatilan kanssa. Neljä päätoimintoa liittyvät suoraan neljään välilehden komponenttiin: Paikalliset taidot, Taitogalleria, Älykkäät suositukset ja Luotetut palveluntarjoajat. Loppujen lopuksi suunnittelussa on lähinnä kyse käyttökokemuksen parantamisesta.

**Backend** perustuu ASP.NET Core + ABP Frameworkiin, jossa käytetään Orleans Grain -ohjelmaa hajautettuun tilanhallintaan. Online API-asiakas käärii `IOnlineApiClient` käyttöliittymä kommunikoidaksesi etätaitoluettelopalvelun kanssa.

Yleisenä arkkitehtonisena periaatteena on erottaa komentojen suorittaminen liiketoimintalogiikasta. Sovitinkuvion kautta npm/npx-komennon toteutuksen yksityiskohdat piilotetaan itsenäisten moduulien sisään. Loppujen lopuksi kukaan ei todellakaan halua komentorivikutsuja hajallaan koko koodikannassa.

## Ydinominaisuus 1: Paikallinen globaali hallinta

Paikallinen globaali hallinta on yksinkertaisin moduuli. Se vastaa asennettujen taitojen luetteloimisesta ja asennuksen poistotoimien tukemisesta. Tässä ei ole mitään liian monimutkaista; kyse on lähinnä siitä, että tehdään perusasiat hyvin.

### Tekninen lähestymistapa

Toteutus elää `LocalSkillsTab.tsx` ja `LocalSkillCommandAdapter.cs`. Ydinideana on kääriä `npx skills` -komento, jäsentää sen JSON-ulostulo ja muuntaa sen sisäisiksi tietorakenteiksi. Kuulostaa yksinkertaiselta ja käytännössä sitä enimmäkseen on.

```csharp
public async Task<IReadOnlyList<LocalSkillInventoryResponseDto>> GetLocalSkillsAsync(
    CancellationToken cancellationToken = default)
{
    var result = await _commandAdapter.ListGlobalSkillsAsync(cancellationToken);
    return result.Skills.Select(skill => new LocalSkillInventoryResponseDto
    {
        Name = skill.Name,
        Version = skill.Version,
        Source = skill.Source,
        InstalledPath = skill.InstalledPath,
        Description = skill.Description
    }).ToList();
}
```

Tiedonkulku on erittäin selkeä: käyttöliittymä lähettää pyynnön -> `SkillGalleryAppService` vastaanottaa sen -> `LocalSkillCommandAdapter` suorittaa npx-komennon -> JSON-tulos jäsennetään -> palautetaan DTO. Jokainen vaihe seuraa luonnollisesti edellisestä.

Asennuksen poisto taitoa käyttää `npx skills remove -g <skillName> -y` komento, ja järjestelmä käsittelee riippuvuudet ja siivouksen automaattisesti. Asennuksen metatiedot on tallennettu `managed-install.json` taitohakemistoon tallentamalla tietoja, kuten asennusaika ja lähdeversio myöhempiä päivityksiä ja tarkastusta varten. Jotkut asiat ovat yksinkertaisesti tallentamisen arvoisia.

### Asennuskulku yksityiskohtaisesti

Taitojen asennus vaatii useita koordinoituja vaiheita. Itse asiassa se ei ole erityisen monimutkaista:

```csharp
public async Task<SkillInstallResultDto> InstallAsync(
    SkillInstallRequestDto request,
    CancellationToken cancellationToken = default)
{
    // 1. Normalize the installation reference
    var normalized = _referenceNormalizer.Normalize(
        request.SkillId,
        request.Source,
        request.SkillSlug,
        request.Version);

    // 2. Check prerequisites
    await _prerequisiteChecker.CheckAsync(cancellationToken);

    // 3. Acquire installation lock
    using var installLock = await _lockProvider.AcquireAsync(normalized.SkillId);

    // 4. Execute installation command
    var result = await _installCommandRunner.ExecuteAsync(
        new SkillInstallCommandExecutionRequest
        {
            Command = $"npx skills add {normalized.FullReference} -g -y",
            Timeout = TimeSpan.FromMinutes(4)
        },
        cancellationToken);

    // 5. Persist installation metadata
    await _metadataStore.WriteAsync(normalized.SkillPath, request);

    return new SkillInstallResultDto { Success = result.Success };
}
```

Tässä käytetään useita keskeisiä suunnittelumalleja: **viittauksen normalisoija** muuntaa erilaisia syöttömuotoja, kuten `tanweai/pua` ja `@opencode/docker-skill`, yhtenäiseksi sisäiseksi edustukseksi; **asennuksen lukitusmekanismi** varmistaa, että vain yksi asennustoiminto voidaan suorittaa samalla taidolla kerrallaan; ja **suoratoistotulos** työntää asennuksen edistymisen käyttöliittymään reaaliajassa palvelimen lähettämien tapahtumien kautta, jotta käyttäjät voivat katsella päätemäisiä lokeja niiden tapahtuessa.

Loppujen lopuksi kaikki nämä mallit ovat olemassa yhtä tarkoitusta varten: järjestelmän pitäminen yksinkertaisempana käyttää ja ylläpitää.

## Core Capability 2: Marketplace Search

Marketplace-haun avulla käyttäjät voivat löytää ja asentaa taitoja yhteisöstä. Yhden henkilön kyky on aina rajallinen; kollektiivinen tieto menee paljon pidemmälle.

### Tekninen lähestymistapa

Hakutoiminto perustuu online-sovellusliittymään `https://api.hagicode.com/v1/skills/search`. Parantaakseen vastausnopeutta järjestelmä toteuttaa välimuistin. Välimuisti on vähän kuin muisti: jos säilytät hyödyllisiä asioita, sinun ei tarvitse ajatella niin lujasti seuraavalla kerralla.

```csharp
private async Task<IReadOnlyList<SkillGallerySkillDto>> SearchCatalogAsync(
    string query,
    CancellationToken cancellationToken,
    IReadOnlySet<string>? allowedSources = null)
{
    var cacheKey = $"skill_search:{query}:{string.Join(",", allowedSources ?? Array.Empty<string>())}";

    if (_memoryCache.TryGetValue(cacheKey, out var cached))
        return (IReadOnlyList<SkillGallerySkillDto>)cached!;

    var response = await _onlineApiClient.SearchAsync(
        new SearchSkillsRequest
        {
            Query = query,
            Limit = _options.LimitPerQuery,
        },
        cancellationToken);

    var results = response.Skills
        .Where(skill => allowedSources is null || allowedSources.Contains(skill.Source))
        .Select(skill => new SkillGallerySkillDto { ... })
        .ToList();

    _memoryCache.Set(cacheKey, results, TimeSpan.FromMinutes(10));
    return results;
}
```

Hakutulokset tukevat suodatusta luotettavien lähteiden mukaan, joten käyttäjät näkevät vain ne taitolähteet, joihin he luottavat. Siemenkyselyt, kuten `popular` ja `recent` Käytetään luettelon alustamiseen, jolloin käyttäjät voivat nähdä suositeltuja suosittuja taitoja, kun he avaavat sen ensimmäisen kerran. Ensivaikutelmalla on silti merkitystä.

## Ydinominaisuus 3: Älykkäät suositukset

Älykkäät suositukset ovat järjestelmän monimutkaisin osa. He voivat automaattisesti suositella sopivimpia taitoja nykyisen projektikontekstin perusteella. Niin monimutkainen kuin se on, se kannattaa silti rakentaa.

### Suosituskulku

Täysi suositusvirta on jaettu viiteen vaiheeseen:

```
1. Build project context
   ↓
2. AI generates search queries
   ↓
3. Search the online catalog in parallel
   ↓
4. AI ranks the candidates
   ↓
5. Return the recommendation list
```

Ensin järjestelmä analysoi ominaisuudet, kuten projektin teknologiapinon, ohjelmointikielet ja verkkoalueen rakenteen rakentaakseen "projektiprofiilin". Tämä profiili on vähän kuin ansioluettelo, joka tallentaa projektin keskeiset piirteet.

Sitten AI Grain -työkalua käytetään kohdistettujen hakukyselyiden luomiseen. Tämä suunnittelu on itse asiassa varsin mielenkiintoinen: sen sijaan, että kysyisit tekoälyltä suoraan: "Mitä taitoja minun pitäisi suositella?", pyydämme sitä ensin miettimään "Millä hakutermeillä todennäköisesti löydetään olennaisia ​​taitoja?" Joskus tapa, jolla esität kysymyksen, on tärkeämpää kuin itse vastaus:

```csharp
var queryGeneration = await aiGrain.GenerateSkillRecommendationQueriesAsync(
    projectContext,      // Project context
    locale,              // User language preference
    maxQueries,           // Maximum number of queries
    effectiveSearchHero); // AI model selection
```

Seuraavaksi nämä hakukyselyt suoritetaan samanaikaisesti ehdokkaiden taitoluettelon keräämiseksi. Rinnakkaiskäsittely on loppujen lopuksi vain tapa säästää aikaa.

Lopuksi toinen AI Grain luokittelee ehdokastaidot. Tässä vaiheessa otetaan huomioon tekijät, kuten taitojen osuvuus projektin kannalta, luottamustila ja käyttäjähistorialliset asetukset:


### Varamekanismi

AI-mallit voivat reagoida hitaasti tai olla tilapäisesti poissa käytöstä. Parhaatkin järjestelmät kompastuvat joskus. Tästä syystä järjestelmä sisältää deterministisen varamekanismin: kun tekoälypalvelu ei ole käytettävissä, se käyttää sääntöpohjaista heuristista algoritmia suositusten luomiseen, kuten todennäköisten tarvittavien taitojen päättelemiseen riippuvuuksista. `package.json`.

Selkeästi sanottuna tämä varamekanismi on yksinkertaisesti järjestelmän varasuunnitelma.

## Ydinominaisuus 4: Luotetun palveluntarjoajan hallinta

Luotetun palveluntarjoajan hallinnan avulla käyttäjät voivat hallita, mitä taitojen lähteitä pidetään luotettavina. Luottamus on edelleen asia, jonka käyttäjien pitäisi pystyä määrittelemään itse.

### Sovitussäännöt

Luotetut palveluntarjoajat tukevat kahta täsmäyssääntöä: **tarkka vastaavuus** (`exact`) ja **etuliite vastaa** (`prefix`).

```csharp
public static TrustedSkillProviderResolutionSnapshot Resolve(
    TrustedSkillProviderSnapshot snapshot,
    string source)
{
    var normalizedSource = Normalize(source);

    foreach (var entry in snapshot.Entries.OrderBy(e => e.SortOrder))
    {
        if (!entry.IsEnabled) continue;

        foreach (var rule in entry.MatchRules)
        {
            bool isMatch = rule.MatchType switch
            {
                TrustedSkillProviderMatchRuleType.Exact
                    => string.Equals(normalizedSource, Normalize(rule.Value),
                        StringComparison.OrdinalIgnoreCase),
                TrustedSkillProviderMatchRuleType.Prefix
                    => normalizedSource.StartsWith(Normalize(rule.Value) + "/",
                        StringComparison.OrdinalIgnoreCase),
                _ => false
            };

            if (isMatch)
                return new TrustedSkillProviderResolutionSnapshot
                {
                    IsTrustedSource = true,
                    ProviderId = entry.ProviderId,
                    DisplayName = entry.DisplayName
                };
        }
    }

    return new TrustedSkillProviderResolutionSnapshot { IsTrustedSource = false };
}
```

Sisäänrakennettuja luotettuja palveluntarjoajia ovat tunnetut organisaatiot ja projektit, kuten Vercel, Azure, anthropics, Microsoft ja selaimen käyttö. Mukautettuja palveluntarjoajia voidaan lisätä määritystiedostojen kautta määrittämällä palveluntarjoajan tunnus, näyttönimi, tunnuksen nimi, vastaavuussäännöt ja paljon muuta. Maailma on tarpeeksi suuri, jotta vain muutamaan sisäänrakennettuun luottaminen ei koskaan riitä.

### Pysyvä toteutus

Luotettu määritys säilyy Orleans Grainilla:

```csharp
public class TrustedSkillProviderGrain : Grain<TrustedSkillProviderState>,
    ITrustedSkillProviderGrain
{
    public async Task UpdateConfigurationAsync(TrustedSkillProviderSnapshot snapshot)
    {
        State.Snapshot = snapshot;
        await WriteStateAsync();
    }

    public Task<TrustedSkillProviderSnapshot> GetConfigurationAsync()
    {
        return Task.FromResult(State.Snapshot);
    }
}
```

Tämän lähestymistavan etuna on, että kokoonpanomuutokset synkronoidaan automaattisesti kaikkien solmujen välillä ilman, että välimuistia tarvitsee päivittää manuaalisesti. Automaatio on viime kädessä ihmisten murehtimisen vähentämistä.

## Keskeinen tekninen suunnittelu

### Komennon suoritussovittimen malli

Skill-järjestelmän on suoritettava erilaisia npx-komentoja. Jos tämä logiikka olisi hajallaan kaikkialla, koodista tulisi nopeasti vaikea ylläpitää. Siksi suunnittelimme sovittimen käyttöliittymän. Suunnittelumallit ovat loppujen lopuksi olemassa koodin ylläpidon helpottamiseksi:

```csharp
public interface ISkillInstallCommandRunner
{
    Task<SkillInstallCommandExecutionResult> ExecuteAsync(
        SkillInstallCommandExecutionRequest request,
        CancellationToken cancellationToken = default);
}
```

Eri komennoilla on erilaiset suorittimen toteutukset, mutta ne kaikki toteuttavat saman käyttöliittymän, mikä tekee testauksesta ja korvaamisesta yksinkertaista.

### SSE-suoratoistolähtö

Asennuksen edistyminen välitetään käyttöliittymään reaaliajassa Server-Sent Events -toiminnon kautta:

```csharp
public async Task InstallWithProgressAsync(
    SkillInstallRequestDto request,
    IServerStreamWriter<SkillInstallProgressEventDto> stream,
    CancellationToken cancellationToken)
{
    var process = new Process
    {
        StartInfo = new ProcessStartInfo
        {
            FileName = "npx",
            Arguments = $"skills add {request.FullReference} -g -y",
            RedirectStandardOutput = true,
            RedirectStandardError = true,
            UseShellExecute = false
        }
    };

    process.OutputDataReceived += async (sender, e) =>
    {
        await stream.WriteAsync(new SkillInstallProgressEventDto
        {
            EventType = "output",
            Data = e.Data ?? string.Empty
        });
    };

    process.Start();
    process.BeginOutputReadLine();
    await process.WaitForExitAsync(cancellationToken);
}
```

Käyttöliittymässä käyttäjät voivat nähdä terminaalin kaltaisen lähdön reaaliajassa, mikä tekee kokemuksesta erittäin intuitiivisen. Reaaliaikainen palaute auttaa ihmisiä tuntemaan olonsa mukavaksi.

## Käytännön opas

### Yhteisötaidon asentaminen

Asenna `pua` taito esimerkkinä (se on suosittu yhteisön taito):

1. Avaa Taidot-laatikko ja vaihda kohtaan `Skill Gallery` -välilehti
2. Enter `pua` hakukentässä
3. Napsauta hakutulosta nähdäksesi taidon tiedot
4. Napsauta `Install` -painiketta
5. Vaihda kohtaan `Local Skills` -välilehteä vahvistaaksesi asennuksen onnistumisen

Asennuskomento on `npx skills add tanweai/pua -g -y`, ja järjestelmä käsittelee kaikki yksityiskohdat automaattisesti. Ei todellakaan ole montaa askelta, kun otat ne yksitellen.

### Mukautetun luotettavan lähteen lisääminen

Jos tiimilläsi on oma taitovarasto, voit lisätä sen luotettavaksi lähteeksi:

```yaml
providerId: "my-team"
displayName: "My Team Skills"
badgeLabel: "MyTeam"
isEnabled: true
sortOrder: 100
matchRules:
  - matchType: "prefix"
    value: "my-team/"
  - matchType: "exact"
    value: "my-team/special-skill"
```

Tällä tavalla kaikki tiimisi taidot näyttävät luotettu-merkin, mikä helpottaa niiden asentamista. Tarrat ja signaalit auttavat ihmisiä tuntemaan olonsa itsevarmemmaksi.

### Taitojen kehittämisen perusteet

Mukautetun taidon luominen vaatii seuraavan rakenteen:

```
my-skill/
├── SKILL.md          # Skill metadata (YAML front matter)
├── index.ts          # Skill entry point
├── agents/           # Supported agent configuration
└── references/       # Reference resources
```

Esimerkki `SKILL.md` muoto:

```yaml
---
name: my-skill
description: A brief description of what this skill does
---

# My Skill

Detailed documentation...
```

## Huomautuksia

1. **Verkkovaatimukset**: taitojen haku ja asennus edellyttävät pääsyä `api.hagicode.com` ja npm-rekisteri
2. **Node.js-versio**: Node.js 18 tai uudempi on suositeltavaa
3. **Lupavaatimukset**: vaaditaan maailmanlaajuiset npm-asennusoikeudet
4. **Samanaikaisuuden hallinta**: vain yksi asennus- tai asennuksen poistotoiminto voidaan suorittaa samalla taidolla kerrallaan
5. **Aikakatkaisuasetukset**: asennuksen oletusaikakatkaisu on 4 minuuttia, mutta monimutkaiset skenaariot saattavat vaatia säätöä

Nämä muistiinpanot ovat olemassa viime kädessä auttaakseen asioita sujumaan sujuvasti.
