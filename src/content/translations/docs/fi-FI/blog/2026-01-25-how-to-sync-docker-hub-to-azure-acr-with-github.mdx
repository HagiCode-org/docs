## Kuvien synkronoinnin automatisointi Docker Hubista Azure ACR:ään

> Tässä artikkelissa kerrotaan, kuinka GitHub Actionsin ja kuvansynkronointityökalun avulla voidaan automatisoida kuvien synkronointi Docker Hubista Azure Container Registry -rekisteriin. Tämä käsittelee Docker Hubin hidasta pääsyä Kiinassa ja joillakin Azure-alueilla ja parantaa kuvien saatavuutta ja käyttöönoton tehokkuutta Azure-ympäristöissä.

## Tausta / Johdanto

HagiCode-projekti käyttää Docker-kuvia ydinajonaikaisina komponentteinaan, ja ensisijaiset kuvat isännöidään Docker Hubissa. Kun projekti on kasvanut ja Azure-käyttöönottotarpeet ovat lisääntyneet, kohtasimme seuraavat ongelmakohdat:

- Hitaat kuvanottonopeudet, ja Docker Hubin pääsy on rajoitettu Kiinassa ja joillakin Azure-alueilla
- Yksittäinen kuvalähde luo yhden virhepisteen riskin
- Azure Container Registry tarjoaa paremman verkon suorituskyvyn ja integraation käytettäessä Azuressa

Näiden ongelmien ratkaisemiseksi tarvitsimme automaattisen kuvien synkronointimekanismin, joka synkronoi kuvat säännöllisesti Docker Hubista Azure ACR:ään. Näin varmistettiin, että käyttäjät voivat saada nopeamman kuvanottonopeuden ja paremman käytettävyyden Azure-ympäristöissä.

## Teknisten ratkaisujen vertailu

Ratkaisua määritellessään vertailimme useita teknisiä vaihtoehtoja:

### 1. kuvan synkronointi (lopullinen valinta)
- Inkrementaalinen synkronointi: synkronoi vain muuttuneet kuvakerrokset, mikä vähentää merkittävästi verkkosiirtoa
- Jatka tukea: synkronointi voi jatkua verkkokatkosten jälkeen
- Samanaikaisuuden hallinta: konfiguroitavat työsäikeet parantavat suurten kuvien synkronoinnin tehokkuutta
- Tehokas virheiden käsittely: sisäänrakennettu uudelleenyritysmekanismi virheiden varalta (oletuksena 3 uudelleenyritystä)
- Kevyt käyttöönotto: yksi binaari ilman riippuvuuksia
- Usean rekisterin tuki: yhteensopiva Docker Hubin, Azure ACR:n, Harborin ja muiden kanssa

### 2. Docker CLI
- Ei inkrementaalista synkronointia: koko kuvasisältö on vedettävä joka kerta
- Alempi tehokkuus: suuri verkon siirtomäärä ja pidempi suoritusaika
- Yksinkertainen käyttää: perustuu tuttuihin `docker pull` ja `docker push` komentoja

### 3. Azure CLI
- Monimutkaisempi: Azure CLI -todennus on määritettävä
- Rajoitettu toiminnallisuus: `az acr import` on suhteellisen kapea
- Natiiviintegraatio: integroituu hyvin Azure-palveluihin

### Päätös 1: Aseta synkronointitaajuus päivittäin klo 00:00 UTC
- Tasapainottaa kuvan tuoreutta ja resurssien kulutusta
- Välttää liiketoiminnan ruuhka-ajat ja vähentää vaikutusta muuhun toimintaan
- Docker Hub -kuvat päivitetään yleensä päivittäisten koontiversioiden jälkeen

### Päätös 2: Synkronoi kaikki kuvatunnisteet
- Säilyttää täyden johdonmukaisuuden Docker Hubin kanssa
- Antaa käyttäjille joustavia versiovaihtoehtoja
- Yksinkertaistaa synkronointilogiikkaa ja välttää monimutkaiset tunnisteiden suodatussäännöt

### Päätös 3: Käytä GitHub Secrets -palvelua kirjautumistietojen tallentamiseen
- Natiivi GitHub Actions -tuki vahvalla suojauksella
- Yksinkertainen konfigurointi ja helppo huolto
- Tukee arkistotason pääsynhallintaa

## Riskien arviointi ja vähentäminen

### Riski 1: Azure ACR -tunnistevuoto
- Tallenna tunnistetiedot salattuna GitHub Secrets -sovelluksessa
- Vaihda ACR-salasanoja säännöllisesti
- Rajoita ACR-tili vain push-käyttöoikeuksiin
- Valvo ACR-käyttölokeja

### Riski 2: Synkronointivirheet aiheuttavat kuvan epäjohdonmukaisuutta
- image-syncer sisältää sisäänrakennetun inkrementaalisen synkronointimekanismin
- Automaattiset uudelleenyritykset epäonnistuessa (oletusarvoisesti 3)
- Yksityiskohtaiset virhelokit ja vikailmoitukset
- Jatkamiskyky

### Riski 3: Liiallinen resurssien kulutus
- Inkrementaalinen synkronointi vähentää verkkosiirtoa
- Konfiguroitava samanaikaisuus (`10` nykyisessä asetuksessa)
- Tarkkaile synkronoitujen kuvien määrää ja kokoa
- Suorita synkronointi ruuhka-aikojen ulkopuolella

## Ydinratkaisu

## Toteutusvaiheet

### 1. Valmistelu
- Luo tai vahvista Azure Container Registry Azure Portalissa
- Luo ACR-käyttöoikeustiedot (käyttäjänimi ja salasana)
- Vahvista Docker Hub -kuvavaraston käyttöoikeudet

### 2. Määritä GitHub Secrets
Lisää seuraavat salaisuudet GitHub-tietovaraston asetuksiin:
- AZURE_ACR_USERNAME: Azure ACR -käyttäjänimi
- AZURE_ACR_PASSWORD: Azure ACR -salasana

### 3. Luo GitHub Actions -työnkulku
Määritä työnkulku sisään `.github/workflows/sync-docker-acr.yml`:
- Ajoitettu laukaisu: joka päivä klo 00.00 UTC
- Manuaalinen liipaisin: tukee `workflow_dispatch`
- Lisäliipaisu: toimii työntöillä `publish` haara nopeaa synkronointia varten

### 4. Työnkulun suorituskulku

| Vaihe | Osallistuja | Toiminta | Kuvaus |
| --- | --- | --- | --- |
| 1 | GitHub-toiminnot | Käynnistä työnkulku | Laukaisee aikataulun, manuaalisen suorituksen tai työnnät `publish` haara |
| 2 | GitHub Actions → image-syncer | Lataa ja suorita synkronointityökalu | Siirtyy varsinaiseen synkronointivaiheeseen |
| 3 | image-syncer → Docker Hub | Hae kuvaluettelot ja tunnisteluettelo | Lukee metatiedot lähdetietovarastosta |
| 4 | image-syncer → Azure ACR | Hae olemassa oleva kuvadata kohdevarastosta | Määrittää kohdepuolen nykyisen tilan |
| 5 | kuvan synkronointi | Vertaa lähteen ja kohteen eroja | Tunnistaa kuvatasot, jotka tarvitsevat synkronoinnin |
| 6 | image-syncer → Docker Hub | Vedä muuttuneet kuvatasot | Siirtää vain päivitettävän sisällön |
| 7 | image-syncer → Azure ACR | Työnnä muutetut kuvatasot | Täyttää asteittaisen synkronoinnin |
| 8 | image-syncer → GitHub Actions | Palauta synkronointitilastot | Sisältää tulokset, erot ja virhetiedot |
| 9 | GitHub-toiminnot | Tallenna lokit ja lataa esineitä | Tukee myöhempää tarkastusta ja vianmääritystä |

## GitHub Actions -työnkulun toteutus

Alla on todellinen käytössä oleva työnkulun konfiguraatio (`.github/workflows/sync-docker-acr.yml`):

```yaml
name: Sync Docker Image to Azure ACR

on:
  schedule:
    - cron: "0 0 * * *" # Every day at 00:00 UTC
  workflow_dispatch: # Manual trigger
  push:
    branches: [publish]

permissions:
  contents: read

jobs:
  sync:
    runs-on: ubuntu-latest

    steps:
      - name: Checkout code
        uses: actions/checkout@v4

      - name: Download image-syncer
        run: |
          # Download the image-syncer binary
          wget https://github.com/AliyunContainerService/image-syncer/releases/download/v1.5.5/image-syncer-v1.5.5-linux-amd64.tar.gz
          tar -zxvf image-syncer-v1.5.5-linux-amd64.tar.gz
          chmod +x image-syncer

      - name: Create auth config
        run: |
          # Generate the authentication config file (YAML format)
          cat > auth.yaml <<EOF
          hagicode.azurecr.io:
            username: "${{ secrets.AZURE_ACR_USERNAME }}"
            password: "${{ secrets.AZURE_ACR_PASSWORD }}"
          EOF

      - name: Create images config
        run: |
          # Generate the image sync config file (YAML format)
          cat > images.yaml <<EOF
          docker.io/newbe36524/hagicode: hagicode.azurecr.io/hagicode
          EOF

      - name: Run image-syncer
        run: |
          # Run synchronization (using the newer --auth and --images parameters)
          ./image-syncer --auth=./auth.yaml --images=./images.yaml --proc=10 --retries=3

      - name: Upload logs
        if: always()
        uses: actions/upload-artifact@v4
        with:
          name: sync-logs
          path: image-syncer-*.log
          retention-days: 7
```

## Kokoonpanon tiedot

### 1. Liipaisuehdot
- Ajoitettu liipaisin: `cron: "0 0 * * *"` kulkee joka päivä klo 00:00 UTC
- Manuaalinen laukaisu: `workflow_dispatch` avulla käyttäjät voivat suorittaa sen manuaalisesti GitHub-käyttöliittymästä
- Paina liipaisinta: `push: branches: [publish]` laukaisee työntöjä vapautushaaraan nopeaa synkronointia varten

### 2. Todennusmääritys (`auth.yaml`)
```yaml
hagicode.azurecr.io:
  username: "${{ secrets.AZURE_ACR_USERNAME }}"
  password: "${{ secrets.AZURE_ACR_PASSWORD }}"
```

### 3. Kuvan synkronoinnin asetukset
```yaml
docker.io/newbe36524/hagicode: hagicode.azurecr.io/hagicode
```
Tämä määritys tarkoittaa, että kaikki tunnisteet osoitteesta `docker.io/newbe36524/hagicode` synkronoidaan `hagicode.azurecr.io/hagicode`.

### 4. image-syncer parametrit
- `--auth=./auth.yaml`: polku todennusasetustiedostoon
- `--images=./images.yaml`: polku kuvan synkronoinnin asetustiedostoon
- `--proc=10`: käytä `10` samanaikaiset työntekijät
- `--retries=3`: uudelleenyritys epäonnistui `3` kertaa

## Huomautuksia ja parhaita käytäntöjä

### 1. Turvallisuussuositukset
- Vaihda Azure ACR -salasanoja säännöllisesti (suositus 90 päivän välein)
- Käytä erillistä ACR-palvelutiliä, jolla on vain push-oikeudet
- Tarkkaile ACR-käyttölokeja havaitaksesi epänormaalit käyttöoikeudet ajoissa
- Älä tulosta kirjautumistietoja lokeihin
- Älä sitoa tunnistetietoja koodisäilöön

### 2. Suorituskyvyn optimointi
- Säädä `--proc` parametri: viritä samanaikaisuus verkon kaistanleveyden perusteella (suositus `5-20`)
- Tarkkaile synkronoinnin kestoa: jos synkronointi kestää liian kauan, harkitse samanaikaisuuden pienentämistä
- Siivoa lokit säännöllisesti: aseta kohtuullinen `retention-days` arvo (`7` tällä hetkellä)

### 3. Vianetsintä

#### Ongelma 1: Todennusvirhe
```
Error: failed to authenticate to hagicode.azurecr.io
```
Ratkaisu:
1. Tarkista, onko GitHub Secrets määritetty oikein
2. Tarkista, onko Azure ACR -salasana vanhentunut
3. Varmista, että ACR-palvelutilin käyttöoikeudet ovat oikeat

#### Ongelma 2: Verkon aikakatkaisu
```
Error: timeout waiting for response
```
Ratkaisu:
1. Tarkista verkkoyhteys
2. Vähennä samanaikaisten työntekijöiden määrää ( `--proc` parametri)
3. Odota, että verkko palautuu ja suorita työnkulku uudelleen

#### Ongelma 3: Epätäydellinen kuvan synkronointi
```
Warning: some tags failed to sync
```
Ratkaisu:
1. Tarkista synkronointilokeista epäonnistuneet tunnisteet
2. Suorita työnkulku uudelleen manuaalisesti synkronoidaksesi uudelleen
3. Varmista, että Docker Hubin lähdekuva on kunnossa

### 4. Valvonta ja hälytykset
- Tarkista säännöllisesti Toiminnot-sivulta työnkulun tila
- Ota GitHub-ilmoitukset käyttöön, jotta työnkulun epäonnistumisista raportoidaan viipymättä
- Valvo Azure ACR -tallennustilan käyttöä
- Tarkista säännöllisesti tunnisteiden yhdenmukaisuus rekistereissä

## Usein kysytyt kysymykset ja ratkaisut

### Kysymys 1: Kuinka synkronoin tietyt tunnisteet kaikkien tunnisteiden sijaan?

Muokkaa `images.yaml` asetustiedosto:
```yaml
# Sync only the latest and v1.0 tags
docker.io/newbe36524/hagicode:latest: hagicode.azurecr.io/hagicode:latest
docker.io/newbe36524/hagicode:v1.0: hagicode.azurecr.io/hagicode:v1.0
```

### Q2: Kuinka synkronoin useita kuvavarastoja?

Lisää useita rivejä `images.yaml`:
```yaml
docker.io/newbe36524/hagicode: hagicode.azurecr.io/hagicode
docker.io/newbe36524/another-image: hagicode.azurecr.io/another-image
```

### Kysymys 3: Kuinka yritän uudelleen, kun synkronointi epäonnistuu?
- Automaattinen uudelleenyritys: image-syncer sisältää sisäänrakennetun uudelleenyritysmekanismin (`3` yrittää uudelleen oletuksena)
- Manuaalinen uudelleenyritys: napsauta `Re-run all jobs` GitHub Actions -sivulla

### Q4: Kuinka voin tarkastella synkronoinnin edistymistä yksityiskohtaisesti?
- Tarkastele reaaliaikaisia lokeja Toiminnot-sivulla
- Lataa `sync-logs` artefakti tarkistaaksesi kaikki lokitiedostot
- Lokitiedostot sisältävät jokaisen tunnisteen synkronoinnin tilan ja siirtonopeuden

### Q5: Kuinka kauan synkronointi kestää?
- Ensimmäinen täydellinen synkronointi: yleensä `10-30` minuuttia kuvan koosta riippuen
- Inkrementaalinen synkronointi: yleensä `2-5` minuuttia, jos kuvan muutokset ovat pieniä
- Ajoitus riippuu verkon kaistanleveydestä, kuvan koosta ja samanaikaisuusasetuksista

## Suositellut laajennukset

### 1. Lisää synkronointiilmoitukset
Lisää ilmoitusvaihe työnkulkuun:
```yaml
- name: Notify on success
  if: success()
  run: |
    echo "Docker images synced successfully to Azure ACR"
```

### 2. Toteuta kuvatunnisteiden suodatus
Lisää tunnisteiden suodatuslogiikka työnkulkuun:
```yaml
- name: Filter tags
  run: |
    # Sync only tags prefixed with v
    echo "docker.io/newbe36524/hagicode:v* : hagicode.azurecr.io/hagicode:v*" > images.yaml
```

### 3. Lisää synkronointitilastoraportti
```yaml
- name: Generate report
  if: always()
  run: |
    echo "## Sync Report" >> $GITHUB_STEP_SUMMARY
    echo "- Total tags: $(grep -c 'synced' image-syncer-*.log)" >> $GITHUB_STEP_SUMMARY
    echo "- Sync time: ${{ steps.sync.outputs.duration }}" >> $GITHUB_STEP_SUMMARY
```
