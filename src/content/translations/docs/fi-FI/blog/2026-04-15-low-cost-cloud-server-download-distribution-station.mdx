## Kuinka rakentaa nopean latauksen jakeluasema edullisilla pilvipalvelimilla

> Pilvitallennuskaistanleveys on järjettömän kallista, rajat ylittävä pääsy on tuskallisen hidasta ja CDN-hinnoittelu riittää pelottamaan kenetkään... Jos käsittelet tiedostojen jakelua, tiedät todennäköisesti nämä ongelmat hyvin. Tässä viestissä haluan jakaa edullisen lähestymistavan, jonka olemme kehittäneet rakentaessamme [HagiCode](https://hagicode.com): pilvipalvelin sekä Nginx-välimuistikerros. Hinta putosi noin puoleen, kun taas latausnopeus parani melkoisesti, mikä oli ainakin hieman lohdullista.

## Tausta

Mitä tulee Internetiin, latausnopeus ja vakaus ovat todella osa käyttökokemusta. Onko sinulla käynnissä avoimen lähdekoodin projekti tai kaupallinen tuote, sinun on silti tarjottava käyttäjille luotettava tapa ladata tiedostoja.

Tiedostojen lataaminen suoraan pilvitallennustilasta, kuten Azure Blob Storagesta tai AWS S3:sta, näyttää yksinkertaiselta, mutta siihen liittyy useita käytännön ongelmia:

**Verkon latenssi**: Rajojen ja alueiden välinen yhteys voi olla tarpeeksi hidasta, jotta näppäimistösi voi murskata. Jos käyttäjien on odotettava ikuisesti, kokemus ei tietenkään ole mahtava.

**Kaistanleveyskustannukset**: Pilvitallennuslähtöliikenne on tuskallisen kallista. Azure Blob Storagen käyttö Manner-Kiinasta maksaa noin 0,5 CNY/GB, mikä tarkoittaa, että 1 Tt kuukaudessa on noin 500 CNY. Pienelle tiimille se ei ole merkityksetön summa. Loppujen lopuksi kenenkään rahat eivät tule tuulesta.

**Pääsyrajoitukset**: Joillakin alueilla pääsy ulkomaisiin pilvipalveluihin on epävakaa, ja joskus se ei yksinkertaisesti ole käytettävissä. Käyttäjät haluavat ladata tiedostoja, mutta eivät voi, mikä on turhauttavaa kaikille.

**CDN-kustannukset**: Kaupalliset CDN-verkot voivat ratkaista nämä ongelmat, mutta hinta on yhtä todellinen. Useimmat pienet joukkueet eivät yksinkertaisesti voi perustella sitä.

Onko siis olemassa ratkaisu, joka on sekä edullinen että käytännöllinen? Kyllä. Käytä pilvipalvelinta, käänteistä välityspalvelinta ja välimuistikerrosta. Se on suoraviivainen lähestymistapa, mutta se toimii. Kustannukset putoavat noin puoleen, ja myös nopeus paranee, mikä on kunnollinen kompromissi.

## Arkkitehtuurisuunnittelu

### Kokonaisarkkitehtuuri

Aloitetaan koko arkkitehtuurista:

```
User request
    ↓
DNS resolution
    ↓
┌─────────────────────────────────────┐
│   Reverse proxy layer (Traefik/Bunker Web) │ ← SSL termination, routing, security protection
├─────────────────────────────────────┤
│   Ports: 80/443                     │
│   Features: Automatic Let's Encrypt certificates │
│             Host routing            │
└─────────────────────────────────────┘
    ↓
┌─────────────────────────────────────┐
│   Cache layer (Nginx)               │ ← File caching, Gzip compression
├─────────────────────────────────────┤
│   Ports: 8080(server) / 8081(desktop) │
│   Cache strategy:                   │
│   - index.json: 1 hour              │
│   - other files: 7 days             │
│   Cache size: 1GB                   │
└─────────────────────────────────────┘
    ↓
┌─────────────────────────────────────┐
│   Origin (Azure Blob Storage)       │ ← File storage
└─────────────────────────────────────┘
```

Tämän arkkitehtuurin ydinidea on yksinkertainen: **aseta välimuisti käyttäjien ja pilvitallennustilan välille**.

Käyttäjäpyynnöt saapuvat ensin pilvipalvelimen käänteiseen välityspalvelimeen, jonka jälkeen Nginx-välimuistikerros ottaa vallan. Jos pyydetty tiedosto on jo välimuistissa, se palautetaan välittömästi. Jos ei, Nginx hakee sen pilvitallennustilasta ja tallentaa samalla paikallisen kopion. Kun seuraavan kerran joku pyytää samaa tiedostoa, pilvitallennustilaa ei tarvitse ottaa uudelleen käyttöön.

### Miksi valita tämä arkkitehtuuri?

**Pilvipalvelinten edut**:
- Arvioitavissa olevat kustannukset: Alibaba Cloudin kaltaiset palveluntarjoajat tarjoavat edullisia pilvipalvelimia, joissa on 1-2 vCPU:ta ja 2 Gt RAM-instanssia, joiden hinta on noin 50-100 CNY kuukaudessa
- Joustava käyttöönotto: voit määrittää käänteisiä välityspalvelinsääntöjä ja välimuistikäytäntöjä vapaasti
- Joustava maantiede: voit valita palvelinalueita lähempänä käyttäjiäsi
- Hyvä skaalautuvuus: voit päivittää palvelinmäärityksiä liikenteen kasvaessa

**Käänteinen välityspalvelin + välimuistin arkkitehtuuri**:
- Vähennä alkuperäpainetta: tallenna kuumat tiedostot välimuistiin vähentääksesi suoraa pääsyä pilvitallennustilaan
- Pienemmät kustannukset: pilvipalvelinliikenne on paljon halvempaa kuin pilvitallennustilan ulospääsy
- Paranna nopeutta: läheinen pääsy ja palvelimen kaistanleveys ovat yleensä parempia kuin suora pilvitallennustoimitus

**Miksi valita Nginx välimuistitasoksi?**

Tämä ei ollut satunnainen valinta. Nginxillä on tässä useita todellisia etuja:

1. Korkea suorituskyky: Nginx on laajalti tunnustettu erinomaisesta käänteisen välityspalvelimen suorituskyvystä
2. Aikuinen välimuisti: sisäänrakennettu `proxy_cache` ominaisuus on vakaa ja luotettava
3. Vähäinen resurssien käyttö: se voi toimia jopa 256 Mt muistilla
4. Joustava määritys: voit käyttää erilaisia välimuistikäytäntöjä eri tiedostotyypeille

## Käänteinen välityspalvelinkerros: Traefik vs Bunker Web

HagiCoden käyttöönottoratkaisu tukee kahta käänteistä välityspalvelinvaihtoehtoa, ja jokaisella on omat vahvuutensa:

| Vaihtoehto | Ominaisuudet | Sopivat skenaariot |
|------|------|----------|
| **Traefik** | Kevyt, automaattinen SSL, yksinkertainen konfigurointi | Peruskäyttöönotto, vähäliikenteen skenaariot |
| **Bunkkeriverkko** | Sisäänrakennettu WAF, DDoS-suojaus, anti-bot-suojaus | Korkean turvallisuuden, vilkkaan liikenteen skenaariot |

### Traefik: Kevyt ensimmäinen valinta

Traefik on moderni HTTP-käänteinen välityspalvelin ja kuormituksen tasapainotin. Sen suurin etu on, että konfigurointi on yksinkertainen ja se voi hankkia Let's Encrypt -sertifikaatit automaattisesti.

Traefik on usein erittäin hyvä valinta alkukäyttöön tai vähäliikenteen skenaarioihin:
- Se käyttää suhteellisen vähän resursseja; 1,5 CPU ja 512 Mt muistia riittää
- SSL-varmenteet määritetään automaattisesti, joten sinun ei tarvitse hallita niitä itse
- Reititys konfiguroidaan Docker-tarrojen kautta, mikä on riittävän kätevää

### Bunkkeriverkko: korkean turvallisuuden skenaarioihin

Bunker Web on Nginx-pohjainen verkkosovelluspalomuuri, jossa on kattavampi suojaus.

Milloin sinun tulisi harkita siirtymistä Bunker Webiin? Yleensä tällaisissa tapauksissa:
- Olet DDoS-hyökkäyksen kohteena
- Tarvitset ModSecurity-suojauksen
- Haluat suojan robotteja vastaan
- Sinulla on tiukemmat turvallisuusvaatimukset

HagiCode tarjoaa `switch-deployment.sh` komentosarja, jotta voit vaihtaa nopeasti kahden vaihtoehdon välillä:

```bash
# Switch to Bunker Web
./switch-deployment.sh bunkerweb

# Switch back to Traefik
./switch-deployment.sh traefik

# Check current status
./switch-deployment.sh status
```

Skripti suorittaa esitarkistuksia, terveystarkastuksia ja automaattisen palautuksen, joten vaihtoprosessi on melko turvallinen ja luotettava.

## Nginx-välimuistikerroksen määritys

Välimuistikerros on koko arkkitehtuurin ydin, joten Nginx-kokoonpanolla on valtava ero välimuistin suorituskyvyssä.

### Välimuistin polun määritys

```nginx
# Cache path configuration
proxy_cache_path /var/cache/nginx levels=1:2 keys_zone=azure_cache:10m
                   max_size=1g inactive=7d use_temp_path=off;
```

Parametrien tiedot:
- `levels=1:2`: välimuistihakemistohierarkia kahdella tasolla tiedostojen käytön tehokkuuden parantamiseksi
- `keys_zone=azure_cache:10m`: välimuistin avainten tallennusalue; 10 Mt riittää suurelle määrälle avaimia
- `max_size=1g`: välimuistin enimmäiskoko on 1 Gt
- `inactive=7d`: poista välimuistissa olevat tiedostot, jos niitä ei ole käytetty 7 päivään
- `use_temp_path=off`: kirjoita suoraan välimuistihakemistoon suorituskyvyn parantamiseksi

### Porrastettu välimuististrategia

Eri tiedostotyypit tarvitsevat erilaisia välimuististrategioita:

```nginx
# Server download service
server {
    listen 8080;

    # Short-term cache for index.json (to allow timely updates)
    location /index.json {
        proxy_cache azure_cache;
        proxy_cache_valid 200 1h;
        proxy_cache_key "$scheme$server_port$request_uri";
        add_header X-Cache-Status $upstream_cache_status;
        add_header Cache-Control "public, max-age=3600";

        # Reverse proxy to Azure OSS
        proxy_pass https://${SERVER_DL_HOST}/${SERVER_DL_CONTAINER}$uri?${SERVER_DL_SAS_TOKEN};
        proxy_ssl_server_name on;
        proxy_ssl_protocols TLSv1.2 TLSv1.3;
    }

    # Long-term cache for static files such as installation packages
    location / {
        proxy_cache azure_cache;
        proxy_cache_valid 200 7d;
        proxy_cache_key "$scheme$server_port$request_uri";
        add_header X-Cache-Status $upstream_cache_status;
        add_header Cache-Control "public, max-age=604800";

        proxy_pass https://${SERVER_DL_HOST}/${SERVER_DL_CONTAINER}$uri?${SERVER_DL_SAS_TOKEN};
        proxy_ssl_server_name on;
        proxy_ssl_protocols TLSv1.2 TLSv1.3;
    }
}
```

**Miksi se on suunniteltu tällä tavalla?**

`index.json` on version tarkistustiedosto, joten se on päivitettävä viipymättä. 1 tunnin välimuistiikkunan ansiosta käyttäjät voivat havaita uuden julkaisun enintään tunnin kuluessa julkaisusta.

Staattiset tiedostot, kuten asennuspaketit, muuttuvat harvoin, joten niiden välimuistiin tallentaminen 7 päivän ajan vähentää huomattavasti pääsyä alkuperään. Kun päivitys tarvitaan, voit tyhjentää välimuistin manuaalisesti.

**`X-Cache-Status` vastauksen otsikko**:

Tämä otsikko auttaa sinua tarkistamaan välimuistin osuman käyttäytymisen:
- `HIT`: välimuisti osui
- `MISS`: välimuisti puuttuu, haettu alkuperästä
- `EXPIRED`: välimuisti vanhentunut, haettu uudelleen alkuperästä
- `BYPASS`: välimuisti ohitettu

Kuinka tarkistaa se:
```bash
curl -I https://server.dl.hagicode.com/app.zip
```

## Kustannusanalyysi

Oletetaan 1 Tt latausliikennettä kuukaudessa. Tehdään laskelma:

| Vaihtoehto | Liikennekustannukset | Palvelimen kustannukset | Yhteensä |
|------|----------|------------|------|
| Suora Azure OSS | Noin 500 CNY | CNY 0 | 500 CNY |
| Pilvipalvelin + OSS (80 % välimuistin osumasuhde) | 100 CNY + 80 CNY | 60 CNY | 240 CNY |
| Kaupallinen CDN | 300-500 CNY | CNY 0 | 300-500 CNY |

**Johtopäätös**: välimuistikerroksen lisääminen voi vähentää jakelukustannuksia noin 50 %.

Tämä arvio olettaa välimuistin osumasuhteeksi 80 %. Käytännössä, jos tiedostot eivät vaihdu usein, osumasuhde voi olla jopa suurempi.

## Toimintavinkkejä

### Välimuistin hallinta

Välimuistit tarvitsevat myös huoltoa ajoittain:

**Tarkista välimuistin käyttö**:
```bash
docker volume inspect docker_nginx-cache
du -sh /var/lib/docker/volumes/docker_nginx-cache/_data
```

**Tyhjennä välimuisti manuaalisesti**:
```bash
./clear-cache.sh
```

Tai suorita manuaaliset komennot tarvittaessa suoraan:
```bash
docker exec nginx sh -c "rm -rf /var/cache/nginx/*"
docker restart nginx
```

### Resurssirajoitukset

1-ytimisessä, 2 Gt:n palvelimessa resurssirajan määritys näyttää tältä:

```yaml
services:
  traefik:
    deploy:
      resources:
        limits:
          cpus: '1.50'
          memory: 512M

  nginx:
    deploy:
      resources:
        limits:
          cpus: '0.50'
          memory: 256M
```

Voit seurata resurssien käyttöä ajoittain suorittamalla:
```bash
docker stats
```

### SAS Token Security Practices

SAS-tunnus on valtuustieto, jota käytetään Azure Blob Storagen käyttöön, joten sen vuotaminen olisi vakavaa:

- Älä sitoudu `.env` tiedostosta versionhallintaan; se on jo sisällä `.gitignore`
- Aseta sopiva SAS-tunnuksen vanhenemisaika, suosituksena yksi vuosi
- Rajoita SAS Token -oikeudet vain luku -käyttöön
- Kierrä SAS-tunnuksia säännöllisesti

### Valvonta ja hälytykset

HagiCode integroi Lark/Feishu Webhook-ilmoitukset, jotka voivat lähettää hälytyksiä seuraavista tapahtumista:
- Käyttöönoton onnistuminen tai epäonnistuminen
- Välimuistin tyhjennystila
- Palvelupoikkeukset

Ilmoitukset sisältävät palvelintietoja, aikaleimoja ja virhetietoja, mikä nopeuttaa vianmääritystä paljon.

### Korkean käytettävyyden laajennukset

Kun yksi palvelin ei enää riitä, voit harkita seuraavaa:

1. **Vaakasuuntainen skaalaus**: ota käyttöön useita solmuja ja jaa liikennettä DNS roundrobinilla tai kuormituksen tasapainottimella
2. **CDN edessä**: aseta CDN pilvipalvelimien eteen vieläkin nopeampaa käyttöä varten
3. **Välimuistin lämpeneminen**: käytä komentosarjoja kuumatiedostojen esilataamiseen välimuistiin

## Huomautuksia

On muutamia asioita, jotka kannattaa pitää mielessä:

1. **SSL-varmenteet**: Let's Encryptillä on nopeusrajoitukset, joten älä vaihda käyttöönottoa liian usein tai varmenteen myöntäminen voi epäonnistua
2. **Välimuistin tyhjennys**: muista tyhjentää välimuisti tärkeiden tiedostojen päivittämisen jälkeen tai käyttäjät voivat silti ladata vanhan version
3. **Lokinhallinta**: puhdista Dockerin lokit säännöllisesti, muuten levy saattaa täyttyä
4. **Varmuuskopiointistrategia**: varmuuskopioi tiedostot, kuten `Traefik acme.json` ja Bunker Web -kokoonpano
5. **Seuranta ja hälytykset**: määritä Feishu-ilmoitukset, jotta voit seurata käyttöönoton tilaa ja vastata nopeasti ongelmiin

- **Ennustettavat kustannukset**: noin 50 % halvempaa kuin pilvitallennustilan käyttäminen suoraan tai kaupallisen CDN:n maksaminen
