## AI CLI -työkalujen käyttäminen Docker-säiliöissä: Käytännön opas käyttäjän eristämiseen ja pysyviin taltioihin

> AI-koodaustyökalujen, kuten Claude Coden, Codexin ja OpenCoden, integroiminen konttiympäristöihin kuulostaa yksinkertaiselta, mutta kaikkialla on piilotettuja monimutkaisia asioita. Tässä artikkelissa perehdytään syvällisesti siihen, kuinka HagiCode-projekti ratkaisee Docker-asennusten ydinhaasteita, mukaan lukien käyttöoikeudet, kokoonpanon pysyvyys ja versionhallinta, jotta voit välttää yleiset sudenkuopat.

## Tausta

Kun päätimme käyttää tekoälykoodauksen CLI-työkaluja Docker-säilöissä, intuitiivisin ajatus oli luultavasti: "Eivätkö säilöt ole vain pääkäyttäjän oikeuksia? Mikset asentaisi kaikkea suoraan ja kutsuisi sitä valmiiksi?" Todellisuudessa tuo näennäisesti yksinkertainen idea kätkee useita ydinongelmia, jotka on ratkaistava.

Ensinnäkin **turvarajoitukset ovat ensimmäinen este**. Otetaan esimerkkinä Claude CLI: se nimenomaisesti kieltää pyörimisen pääkäyttäjänä. Tämä on pakollinen turvatarkastus, ja jos root havaitaan, se kieltäytyy käynnistymästä. Saatat ajatella, enkö voi vain vaihtaa käyttäjää `USER` direktiivi? Se ei ole niin yksinkertaista. Säilön sisällä olevan ei-root-käyttäjän ja isäntäkoneen käyttöoikeuksien välillä on edelleen kartoitusongelma.

Toiseksi **tilan pysyvyys on toinen ansa**. Claude Code vaatii kirjautumisen, Codexilla on omat kokoonpanonsa ja OpenCodella on myös välimuistihakemisto. Jos joudut konfiguroimaan kaikki uudelleen joka kerta, kun kontti käynnistyy uudelleen, koko "automaation" ajatus menettää merkityksensä. Tarvitsemme nämä kokoonpanot säilymään säiliön elinkaaren jälkeen.

Kolmas ongelma on **lupien johdonmukaisuus**. Voivatko säilön sisällä olevat prosessit käyttää isäntäkäyttäjän luomia määritystiedostoja? UID/GID-epäsopivuus aiheuttaa usein tiedostojen käyttöoikeusvirheitä, ja tämä on erittäin yleistä todellisissa käyttöönotoissa.

Nämä ongelmat voivat näyttää itsenäisiltä, mutta käytännössä ne liittyvät tiiviisti toisiinsa. HagiCoden kehityksen aikana kehitimme vähitellen käytännön ratkaisun. Seuraavaksi jaan tekniset yksityiskohdat ja näistä sudenkuoppista saadut opetukset.

## Miksi emme voi käyttää vain rootia?

Tässä on yleinen väärinkäsitys: Docker-säilöt toimivat oletuksena pääkäyttäjänä, joten miksi ei vain asenneta työkaluja pääkäyttäjänä? Jos ajattelet niin, Claude CLI opettaa sinulle nopeasti toisin.

```bash
# Run Claude CLI directly as root? No.
docker run --rm -it --user root myimage claude
# Output: Error: This command cannot be run as root user
```

Tämä on kova tietoturvarajoitus Claude CLI:ssä. Syy on yksinkertainen: nämä CLI-työkalut lukevat ja kirjoittavat arkaluonteisia käyttäjämäärityksiä, mukaan lukien API-tunnukset, paikalliset välimuistit ja jopa käyttäjän kirjoittamat komentosarjat. Niiden suorittaminen pääkäyttäjän oikeuksilla aiheuttaa liikaa riskejä.

Joten kysymys kuuluu: kuinka voimme täyttää CLI:n turvallisuusvaatimukset samalla kun säilytämme kontinhallinnan joustavana? Meidän on muutettava tapaamme ajatella asiasta: **sen sijaan, että vaihtaisit käyttäjää ajon aikana, luo oma käyttäjä kuvan rakennusvaiheessa**.

## Oman käyttäjän luominen: enemmän kuin pelkkä nimen vaihtaminen

Saatat ajatella, että lisäät yhden `USER` rivi Dockerfileen riittää. Se on todellakin yksinkertaisin lähestymistapa, mutta se ei ole tarpeeksi vankka.

### Staattinen luominen vs. dynaaminen kartoitus

HagiCoden lähestymistapa on luoda a `hagicode` käyttäjä, jonka UID 1000, joka yleensä vastaa useimpien isäntäkoneiden oletuskäyttäjää:

```dockerfile
RUN groupadd -o -g 1000 hagicode && \
    useradd -o -u 1000 -g 1000 -s /bin/bash -m hagicode && \
    mkdir -p /home/hagicode/.claude && \
    chown -R hagicode:hagicode /home/hagicode
```

Mutta tämä ratkaisee vain kuvan sisäänrakennetun käyttäjän. Entä jos isäntäkäyttäjä on UID 1001? Sinun on silti tuettava dynaamista kartoitusta, kun säilö käynnistyy.

docker-entrypoint.sh sisältää avainlogiikan:

```bash
if [ -n "$PUID" ] && [ -n "$PGID" ]; then
    if ! id hagicode >/dev/null 2>&1; then
        groupadd -g "$PGID" hagicode
        useradd -u "$PUID" -g "$PGID" -s /bin/bash -m hagicode
    fi
fi
```

Tämän suunnittelun etu on selvä: **käytä oletusarvoista UID 1000:aa kuvan rakennusaikana ja säädä sitten dynaamisesti ajon aikana `PUID` ja `PGID` ympäristömuuttujat**. Riippumatta siitä, mikä UID isäntäkäyttäjällä on, määritystiedostojen omistajuus pysyy oikeana.

## Pysyvien volyymien suunnittelufilosofia

Jokaisella AI CLI -työkalulla on oma ensisijainen määrityshakemisto, joten ne on kartoitettava yksitellen:

| CLI-työkalu | Polku säilössä | nimeltä Volume |
|---------|-----------|--------|
| Claude | `/home/hagicode/.claude` | `claude-data` |
| Codex | `/home/hagicode/.codex` | `codex-data` |
| OpenCode | `/home/hagicode/.config/opencode` | `opencode-config-data` |

Miksi käyttää nimettyjä asemia sidosliitosten sijaan? Kolme syytä:

1. **Yksinkertaisempi hallinta**: Docker hallitsee nimettyjä taltioita automaattisesti, joten sinun ei tarvitse luoda isäntähakemistoja manuaalisesti.
2. **Luvan eristäminen**: Käyttäjä luo taltioiden alkuperäisen sisällön säilön sisällä välttäen käyttöoikeusristiriidat isännän kanssa.
3. **Riippumaton siirto**: Volyymit voivat olla säilöistä riippumatta, joten tietoja ei menetetä, kun kuvia päivitetään.

docker-compose-builder-web luo automaattisesti vastaavan taltiokokoonpanon:

```yaml
volumes:
  claude-data:
  codex-data:
  opencode-config-data:

services:
  hagicode:
    volumes:
      - claude-data:/home/hagicode/.claude
      - codex-data:/home/hagicode/.codex
      - opencode-config-data:/home/hagicode/.config/opencode
    user: "${PUID:-1000}:${PGID:-1000}"
```

Kiinnitä huomiota `user` kenttä täällä: `PUID` ja `PGID` lisätään ympäristömuuttujien kautta, jotta varmistetaan, että säilön sisällä olevat prosessit toimivat identiteetillä, joka vastaa isäntäkäyttäjää. Tällä yksityiskohdalla on merkitystä, koska lupaongelmien korjaaminen on tuskallista, kun ne ilmestyvät.

## Versionhallinta: valmiit versiot ajonaikaisilla ohituksilla

Docker-kuvaversioiden kiinnittäminen on välttämätöntä toistettavuuden kannalta. Mutta todellisessa kehityksessä meidän on usein testattava uudempaa versiota tai kiireellisesti korjattava bugi. Jos meidän pitäisi rakentaa kuva uudelleen joka kerta, työnkulku olisi aivan liian tehoton.

HagiCoden strategia on **kiinteät versiot oletuksena, ajonaikaiset ohitukset laajennusmekanismina**. Se on pragmaattinen tekninen kompromissi vakauden ja joustavuuden välillä.

Dockerfile.template kiinnittää versiot tähän:

```dockerfile
USER hagicode
WORKDIR /home/hagicode

# Configure the global npm install path
RUN mkdir -p /home/hagicode/.npm-global && \
    npm config set prefix '/home/hagicode/.npm-global'

# Install CLI tools using pinned versions
RUN npm install -g @anthropic-ai/claude-code@2.1.71 && \
    npm install -g @openai/codex@0.112.0 && \
    npm install -g opencode-ai@1.2.25 && \
    npm cache clean --force
```

docker-entrypoint.sh tukee ajonaikaisia ohituksia:

```bash
install_cli_override_if_needed() {
    local package_name="$2"
    local override_version="$5"

    if [ -n "$override_version" ]; then
        gosu hagicode npm install -g "${package_name}@${override_version}"
    fi
}

# Example usage
install_cli_override_if_needed "" "@anthropic-ai/claude-code" "" "" "${CLAUDE_CODE_CLI_VERSION}"
```

Tämän avulla voit testata uutta versiota ympäristömuuttujan kautta rakentamatta kuvaa uudelleen:

```bash
docker run -e CLAUDE_CODE_CLI_VERSION=2.2.0 myimage
```

Tämä muotoilu on käytännöllinen, koska kukaan ei halua rakentaa kuvaa uudelleen joka kerta, kun he testaavat uutta ominaisuutta.

## Automaattinen konfigurointiinjektio

CLI-työkalujen manuaalisen määrittämisen lisäksi jotkin skenaariot vaativat automaattisen kokoonpanon lisäämisen. Tyypillisin esimerkki on API-tunnus.

```bash
if [ -n "$ANTHROPIC_AUTH_TOKEN" ]; then
    mkdir -p /home/hagicode/.claude
    cat > /home/hagicode/.claude/settings.json <<EOF
{
  "env": {
    "ANTHROPIC_AUTH_TOKEN": "${ANTHROPIC_AUTH_TOKEN}"
  }
}
EOF
    chown -R hagicode:hagicode /home/hagicode/.claude
fi
```

Tässä on kaksi merkitystä: **välitä arkaluontoiset tiedot ympäristömuuttujien kautta sen sijaan, että koodaat ne kovakoodaamalla ne kuvaan**, ja varmista, että määritystiedostojen omistajuus on asetettu oikein, muuten CLI-työkalut eivät pysty lukemaan niitä.

## Parhaat käytännöt ja sudenkuoppien tarkistuslista

### Lupien yhteensopivuusongelmat

Tähän ansaan on helpoin pudota. Isäntäkäyttäjällä on UID 1001, kun taas säilössä on 1000, joten toiselle puolelle luotuja tiedostoja ei voi käyttää toisella puolella.

```bash
# Correct approach: make the container match the host user
docker run \
    -e PUID=$(id -u) \
    -e PGID=$(id -g) \
    myimage
```

Tämä ongelma on hyvin yleinen, ja se voi olla turhauttavaa, kun törmäät siihen ensimmäisen kerran.

### Määritys katoaa kontin uudelleenkäynnistyksen jälkeen

Jos huomaat kirjautuvasi uudelleen sisään jokaisen uudelleenkäynnistyksen jälkeen, tarkista, oletko unohtanut asentaa jatkuvan levyn:

```yaml
volumes:
  - claude-data:/home/hagicode/.claude
```

Mikään ei ole turhauttavampaa kuin kokoonpanon huolellinen määrittäminen vain nähdäksesi sen katoavan.

### Oikea tapa päivittää versioita

Älä juokse `npm install -g` suoraan käynnissä olevan säiliön sisällä. Oikeat lähestymistavat ovat:

1. Aseta ympäristömuuttuja käynnistämään ohitusasennuksen.
2. Tai rakenna kuva uudelleen.

```bash
# Option 1: runtime override
docker run -e CLAUDE_CODE_CLI_VERSION=2.2.0 myimage

# Option 2: rebuild the image
docker build -t myimage:v2 .
```

Roomaan on enemmän kuin yksi tie, mutta jotkut tiet ovat tasaisempia kuin toiset.

### Turvallisuustarkistuslista

- Ohjaa API-tunnukset ympäristömuuttujien läpi sen sijaan, että kirjoitat ne kuvaan.
- Aseta määritystiedoston käyttöoikeudet `600`.
- Suorita sovellus aina ei-root-käyttäjänä.
- Korjaa tietoturva-aukkoja päivittämällä CLI-versiot säännöllisesti.

Turvallisuus on aina tärkeää, mutta todellinen haaste on sen johdonmukainen toteuttaminen käytännössä.

## Uusien CLI-työkalujen tuen laajentaminen

Jos haluat tukea uutta CLI-työkalua tulevaisuudessa, on vain kolme vaihetta:

1. **Dockerfile.template**: lisää asennusvaihe.
2. **docker-entrypoint.sh**: lisää version ohituslogiikka.
3. **docker-compose-builder-web**: lisää jatkuva volyymikartoitus.

Tämä mallipohjainen suunnittelu tekee laajentamisesta yksinkertaista muuttamatta ydinlogiikkaa.

## Johtopäätös

AI CLI -työkalujen käyttäminen Docker-säiliöissä sisältää kolme ydinhaastetta: **käyttäjäoikeudet, kokoonpanon pysyvyys ja versionhallinta**. Yhdistämällä omistetut käyttäjät, nimetty volyymieristys ja ympäristömuuttujapohjaiset ohitukset, HagiCode-projekti rakensi käyttöönottoarkkitehtuurin, joka on sekä turvallinen että joustava.

Tärkeimmät suunnittelupisteet:

- **Käyttäjien eristäminen**: Luo oma käyttäjä kuvan rakennusvaiheessa, ja ajonaikainen tuki dynaamiselle `PUID`/`PGID` kartoitus.
- **Pysyvyysstrategia**: Jokainen CLI-työkalu saa oman nimensä, joten uudelleenkäynnistykset eivät vaikuta määritykseen.
- **Version joustavuus**: Kiinteät oletusasetukset varmistavat toistettavuuden, kun taas ajonaikaiset ohitukset tarjoavat tilaa testaukselle.
- **Automaattinen määritys**: Herkkä konfiguraatio voidaan syöttää automaattisesti ympäristömuuttujien kautta.

Tämä ratkaisu on toiminut vakaasti HagiCode-projektissa jo jonkin aikaa, ja toivon, että se tarjoaa hyödyllisiä vertailukohtia kehittäjille, joilla on samanlaisia tarpeita.
