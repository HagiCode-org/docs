## Muokkaa DESIGN.md-tiedostoa suoraan verkkoliittymässä: Ideasta toteutukseen

> MonoSpecs-projektinhallintajärjestelmässä `DESIGN.md` kantaa projektin arkkitehtonista suunnittelua ja teknisiä päätöksiä. Mutta perinteinen muokkaustyönkulku pakottaa käyttäjät hyppäämään ulkoiseen editoriin. Tuo pirstaleinen kokemus on kuin keskeytettäisiin kesken runon lukemisen: inspiraatio on poissa, samoin mieliala. Tämä artikkeli jakaa ratkaisun, jonka toteutimme HagiCode-projektissa: editoinnin `DESIGN.md` suoraan verkkokäyttöliittymässä, ja se tukee mallien tuontia online-suunnittelusivustolta. Loppujen lopuksi, kukapa ei nauttisi tunteesta saada kaikki valmiiksi yhdessä virrassa?

## Tausta

Projektisuunnitteluasiakirjojen keskeisenä kantajana `DESIGN.md` sisältää tärkeitä tietoja, kuten arkkitehtuurin suunnittelun, tekniset päätökset ja toteutusohjeet. Perinteinen muokkaustapa edellyttää kuitenkin, että käyttäjät vaihtavat ulkoiseen editoriin, kuten VS-koodiin, etsivät manuaalisesti fyysisen polun ja muokkaavat sitten tiedostoa. Se ei ole erityisen monimutkaista, mutta prosessin muutaman kerran toistamisen jälkeen siitä tulee väsyttävää.

Ongelmat ilmenevät pääasiassa seuraavilla tavoilla:

- **Hajanainen työnkulku**: käyttäjien on jatkuvasti vaihdettava verkkohallintaliittymän ja paikallisen editorin välillä, mikä katkaisee työnkulkunsa jatkuvuuden, aivan kuten jos musiikki katkeaisi kesken kappaleen.
- **Vaikea käyttää uudelleen**: suunnittelusivustolla on jo runsaasti suunnittelumalleja, mutta niitä ei voi integroida suoraan projektin muokkaustyönkulkuun. Hyviä asioita on olemassa, mutta et silti voi käyttää niitä siellä, missä niitä tarvitset.
- **Puuttuva kokemussilmukka**: "Preview-select-import" -toiminnolle ei ole suljettua silmukkaa, joten käyttäjien on kopioitava ja liitettävä manuaalisesti, mikä lisää virheiden riskiä.
- **Yhteistyökitka**: Suunnitteludokumenttien ja koodin toteutuksen synkronoinnista tulee kova prosessi, mikä heikentää tiimin tehokkuutta.

Näiden kipukohtien ratkaisemiseksi päätimme lisätä suoran muokkaustuen kohteelle `DESIGN.md` verkkokäyttöliittymässä ja salli mallin tuonnin yhdellä napsautuksella online-suunnittelusivustolta. Se ei ollut mikään maata järisyttävä päätös. Halusimme vain tehdä kehityskokemuksesta sujuvamman.

## Tietoja HagiCodesta

Tässä artikkelissa jaettu ratkaisu on peräisin käytännön kokemuksistamme [HagiCode](https://hagicode.com) hanke. HagiCode on tekoälyyn perustuva koodausavustajaprojekti, ja kehitystyön aikana meidän on usein ylläpidettävä projektin suunnitteludokumentteja. Auttaaksemme tiimiä tekemään yhteistyötä tehokkaammin tutkimme ja otimme käyttöön tämän online-muokkaus- ja tuontiratkaisun. Siinä ei ole mitään mystistä. Törmäsimme ongelmaan ja keksimme tavan ratkaista se.

## Tekninen ratkaisu

### Kokonaisarkkitehtuuri

Tämä ratkaisu käyttää frontend-backend-eroteltua arkkitehtuuria saman alkuperän välityspalvelimen kanssa, joka koostuu pääasiassa seuraavista kerroksista. Käytännössä suunnittelu voidaan tiivistää seuraavasti: "jokainen osa tekee oman työnsä":

**1. Käyttöliittymän muokkaustaso**
```typescript
// Core component: DesignMdManagementDrawer
// Location: repos/web/src/components/project/DesignMdManagementDrawer.tsx
// Responsibility: handle editing, saving, version conflict detection, and import flow
```

**2. taustapalvelukerros**
```csharp
// Core service: ProjectAppService.DesignMd
// Location: repos/hagicode-core/src/PCode.Application/ProjectAppService.DesignMd.cs
// Responsibility: path resolution, file read/write, and version management
```

**3. Saman alkuperän välityspalvelintaso**
```csharp
// Proxy service: ProjectAppService.DesignMdSiteIndex
// Location: repos/hagicode-core/src/PCode.Application/ProjectAppService.DesignMdSiteIndex.cs
// Responsibility: proxy design site resources, preview image caching, and security validation
```

### Tärkeimmät tekniset päätökset

#### Päätös 1: Yleinen vetolaatikkokuvio

Käytämme yhtä globaalia laatikkoa paikallisten ponnahdusikkunoiden sijasta, ja tilaa hallitaan `layoutSlice`, joka antaa käyttäjille yhtenäisen kokemuksen kaikissa näkymissä (`classic` ja `kanban`). Riippumatta siitä, mistä näkymästä käyttäjä avaa editorin, he saavat saman vuorovaikutusmallin. Jatkuva kokemus saa ihmiset tuntemaan olonsa helpommaksi sen sijaan, että he joutuisivat sekaisin vaihtaessaan näkemystä.

#### Päätös 2: Project-Scoped API

Asensimme `DESIGN.md`-liittyvät päätepisteet alla `ProjectController`, käyttämällä olemassa olevaa projektin käyttöoikeusrajaa uudelleen ja välttäen erillisen ohjaimen lisäämisen monimutkaisuuden. Tämä tekee lupien käsittelystä selkeämpää ja sopii yhteen RESTful-resurssien organisaation kanssa. Joskus uudelleenkäyttö on mielekkäämpää kuin uuden luominen tyhjästä.

#### Päätös 3: Versioristiriitojen tunnistus

Johdamme läpinäkymättömän version tiedostojärjestelmästä `LastWriteTimeUtc`, mikä antaa meille kevyen optimistisen samanaikaisuuden hallinnan. Kun useat käyttäjät muokkaavat samaa tiedostoa kerralla, järjestelmä voi havaita ristiriidat ja kehottaa käyttäjää päivittämään. Tämä muotoilu ei estä muokkaamista, mutta suojaa silti tietojen johdonmukaisuutta.

#### Päätös 4: Saman alkuperän välityspalvelinmalli

Käytämme `IHttpClientFactory` käyttää ulkoisia suunnittelupaikan resursseja välttäen sekä ristiriitaiset ongelmat että SSRF-riskit. Tämä pitää järjestelmän turvassa ja yksinkertaistaa myös käyttöliittymäpuheluita. Turvallisuuden suhteen tuskin voi olla liian varovainen.

## Ydintoteutus

### 1. Muokkaa tiedostoa DESIGN.md suoraan

#### Taustaohjelman toteutus

Backend vastaa pääasiassa polun tarkkuudesta, tiedostojen lukemisesta/kirjoittamisesta ja versionhallinnasta. Nämä ovat perustehtävät, mutta välttämättömiä, kuten talon perusta:

```csharp
// Path resolution and security validation
private Task<string> ResolveDesignDocumentDirectoryAsync(string projectPath, string? repositoryPath)
{
    if (string.IsNullOrWhiteSpace(repositoryPath))
    {
        return Task.FromResult(Path.GetFullPath(projectPath));
    }
    return ValidateSubPathAsync(projectPath, repositoryPath);
}

// Version generation (based on file system timestamp and size)
private static string BuildDesignDocumentVersion(string path)
{
    var fileInfo = new FileInfo(path);
    fileInfo.Refresh();
    return string.Create(
        CultureInfo.InvariantCulture,
        $"{fileInfo.LastWriteTimeUtc.Ticks:x}-{fileInfo.Length:x}");
}
```

Versiosuunnittelu on mielenkiintoinen yksinkertaisuudessaan: luomme yksilöllisen versiotunnisteen tiedoston viimeksi muokatun ajan ja koon perusteella. Se on kevyt ja luotettava, ilman ylimääräistä versiotietokantaa ylläpidettäväksi. Yksinkertaiset ratkaisut ovat usein tehokkaimpia.

#### Käyttöliittymän toteutus

Käyttöliittymässä toteutamme likaisen tilan tunnistuksen ja tallennamme logiikan. Tämä muotoilu auttaa käyttäjiä ymmärtämään, onko heidän tekemänsä muutokset tallennettu, ja vähentää ahdistusta "entä jos menetän sen?":

```typescript
// Dirty-state detection and save logic
const [draft, setDraft] = useState('');
const [savedDraft, setSavedDraft] = useState('');
const isDirty = draft !== savedDraft;

const handleSave = useCallback(async () => {
    const result = await saveProjectDesignMdDocument({
        ...activeTarget,
        content: draft,
        expectedVersion: document.version, // optimistic concurrency control
    });
    setSavedDraft(draft); // update saved state
}, [activeTarget, document, draft]);
```

Tässä toteutuksessa säilytämme kaksi tilaa: `draft` on tällä hetkellä muokattava sisältö, kun `savedDraft` on tallennettu sisältö. Niiden vertailu kertoo, onko tallentamattomia muutoksia. Suunnittelu on yksinkertainen, mutta se antaa ihmisille mielenrauhan. Kukaan ei halua, että asia, jonka parissa he työskentelivät, katoaa.

### 2. Tuo suunnittelutiedostot online-lähteestä

#### Hakemistorakenne

```
repos/index/
└── src/data/public/design.json    # Design template index

repos/awesome-design-md-site/
├── vendor/awesome-design-md/       # Upstream design templates
│   └── design-md/

│       ├── linear/
│       │   └── DESIGN.md
│       └── ...
└── src/lib/content/
    └── awesomeDesignCatalog.ts     # Content pipeline
```

#### Hakemiston tietomuoto

Suunnittelusivuston hakemistotiedosto määrittelee kaikki käytettävissä olevat mallit. Tämän indeksin avulla käyttäjät voivat valita haluamansa mallin yhtä helposti kuin tilata valikosta:

```json
{
  "entries": [
    {
      "slug": "linear.app",
      "title": "Linear Inspired Design System",
      "summary": "AI Product / Dark Feel",
      "detailUrl": "/designs/linear.app/",
      "designDownloadUrl": "/designs/linear.app/DESIGN.md",
      "previewLightImageUrl": "...",
      "previewDarkImageUrl": "..."
    }
  ]
}
```

Jokainen merkintä sisältää mallin perustiedot ja latauslinkit. Taustaohjelma lukee luettelon käytettävissä olevista malleista tästä hakemistosta ja esittää ne käyttäjän valittavaksi. Tämä tekee valinnasta intuitiivista sen sijaan, että se pakottaisi ihmiset tuntemaan tiensä pimeässä.

#### Saman alkuperän välityspalvelimen käyttöönotto

Asioiden turvaamiseksi taustaohjelma suorittaa tiukan validoinnin suunnittelusivustolle pääsylle. Turvallisuuden suhteen ei voi olla liian varovainen:

```csharp
// Safe slug validation
private static readonly Regex SafeDesignSiteSlugRegex =
    new("^[A-Za-z0-9](?:[A-Za-z0-9._-]{0,127})$", RegexOptions.Compiled);

private static string NormalizeDesignSiteSlug(string slug)
{
    var normalizedSlug = slug?.Trim() ?? string.Empty;
    if (!IsSafeDesignSiteSlug(normalizedSlug))
    {
        throw new BusinessException(
            ProjectDesignSiteIndexErrorCodes.InvalidSlug,
            "Design site slug must be a single safe path segment.");
    }
    return normalizedSlug;
}

// Preview image caching (OS temp directory)
private static string ComputePreviewCacheKey(string slug, string theme, string previewUrl)
{
    var raw = $"{slug}|{theme}|{previewUrl}";
    var bytes = SHA256.HashData(Encoding.UTF8.GetBytes(raw));
    return Convert.ToHexString(bytes).ToLowerInvariant();
}
```

Teemme tässä kaksi asiaa: ensin validoimme slug-muodon tiukasti säännöllisellä lausekkeella estääksemme polun läpikulkuhyökkäykset; Toiseksi tallennamme esikatselukuvat välimuistiin vähentääksemme ulkoiseen sivustoon kohdistuvaa painetta. Ensimmäinen on suojaa, jälkimmäinen on optimointi, ja molemmilla on merkitystä.

### 3. Täysi tuontivirta

```typescript
// 1. Open the import drawer
const handleRequestImportDrawer = useCallback(() => {
    setIsImportDrawerOpen(true);
}, []);

// 2. Select and import
const handleImportRequest = useCallback((entry) => {
    if (isDirty) {
        setPendingImportEntry(entry);
        setConfirmMode('import'); // overwrite confirmation
        return;
    }
    void executeImport(entry);
}, [isDirty]);

// 3. Execute import
const executeImport = useCallback(async (entry) => {
    const result = await getProjectDesignMdSiteImportDocument(
        activeTarget.projectId,
        entry.slug
    );
    setDraft(result.content); // replace editor text only, do not save automatically
    setIsImportDrawerOpen(false);
}, [activeTarget?.projectId]);
```

Tuontiprosessi noudattaa "käyttäjän vahvistus" -periaatetta: tuonnin jälkeen vain editorin sisältö päivitetään, eikä mitään tallenneta automaattisesti. Käyttäjät voivat tarkastaa tuodun sisällön ja tallentaa sen manuaalisesti vasta varmistuttuaan, että se näyttää oikealta. Lopullisen päätöksen tulee jäädä käyttäjän käsiin.

## Käytännön esimerkkejä

### Skenaario 1: DESIGN.md:n luominen projektin juurella

Milloin `DESIGN.md` ei ole olemassa, taustaohjelma palauttaa virtuaalisen asiakirjan tilan. Tämä antaa käyttöliittymälle mahdollisuuden välttää "tiedostoa ei ole olemassa" -tapauksen erityiskäsittelyt, ja yhtenäinen API yksinkertaistaa koodilogiikkaa:

```csharp
return new ProjectDesignDocumentDto
{
    Path = targetPath,
    Exists = false,  // virtual document state
    Content = string.Empty,
    Version = null
};

// Automatically create the file on first save
public async Task<SaveProjectDesignDocumentResultDto> SaveDesignDocumentAsync(...)
{
    Directory.CreateDirectory(targetDirectory);
    await File.WriteAllTextAsync(targetPath, input.Content);
    return new SaveProjectDesignDocumentResultDto { Created = !exists };
}
```

Tämän suunnittelun etuna on, että käyttöliittymä ei tarvitse erityistapauslogiikkaa puuttuville tiedostoille. Piilottamalla tämän monimutkaisuuden taustaan, käyttöliittymä voi keskittyä helpommin käyttökokemukseen.

### Skenaario 2: Tuo malli suunnittelusivustosta

Kun käyttäjä on valinnut "Lineaarinen" suunnittelumallin tuontilaatikosta, järjestelmä hakee sen `DESIGN.md` sisältöä taustapalvelimen kautta. Koko prosessi on käyttäjälle läpinäkyvä: hän valitsee vain mallipohjan, ja järjestelmä käsittelee verkkopyynnöt ja tiedonmuunnoksen automaattisesti.

```typescript
// 1. The system fetches DESIGN.md content through the backend proxy
GET /api/project/{id}/design-md/site-index/linear.app

// 2. The backend validates the slug and fetches content from upstream
var entry = FindDesignSiteEntry(catalog, "linear.app");
using var upstreamResponse = await httpClient.SendAsync(request);
var content = await upstreamResponse.Content.ReadAsStringAsync();

// 3. The frontend replaces the editor text
setDraft(result.content);
// The user reviews it and then saves it manually to disk
```

Koko kulku pysyy läpinäkyvänä käyttäjälle. He valitsevat vain mallin, ja järjestelmä hoitaa verkottumisen ja muutoksen kulissien takana. Sellaisen kokemuksen haluamme: yksinkertainen, mutta tehokas.

### Skenaario 3: Versioristiriitojen käsittely

Kun useat käyttäjät muokkaavat samaa `DESIGN.md` samalla järjestelmä havaitsee versioristiriidat. Tämä optimistinen samanaikaisuuden hallintamekanismi säilyttää tietojen johdonmukaisuuden estämättä käyttäjän muokkauksia:

```csharp
if (!string.Equals(currentVersion, expectedVersion, StringComparison.Ordinal))
{
    throw new BusinessException(
        ProjectDesignDocumentErrorCodes.VersionConflict,
        $"DESIGN.md at '{targetPath}' changed on disk.");
}
```

Käyttöliittymä havaitsee tämän virheen ja pyytää käyttäjää:

```typescript
// Frontend prompts the user to refresh and retry
<Alert>
    <AlertTitle>Version conflict</AlertTitle>
    <AlertDescription>
        The file was modified by another process. Please refresh to get the latest version and try again.
    </AlertDescription>
</Alert>
```

Tämä optimistinen samanaikaisuuden hallintamekanismi pitää tiedot johdonmukaisina estämättä käyttäjiä heidän työskennellessään. Konfliktit ovat väistämättömiä, mutta ainakin käyttäjien pitäisi tietää, mitä tapahtui, sen sijaan, että he menettäisivät muutokset hiljaa.

## Huomautuksia ja parhaita käytäntöjä

### 1. Polun suojaus

Vahvista aina `repositoryPath` estämään polun ylityshyökkäykset. Et voi koskaan tehdä liikaa turvallisuuden suhteen:

```csharp
// Always validate repositoryPath to prevent path traversal attacks
return ValidateSubPathAsync(projectPath, repositoryPath);
// Reject dangerous inputs such as "../" and absolute paths
```

### 2. Välimuististrategia

Tallenna esikatselukuvat välimuistiin 24 tunnin ajan, enintään 160 tiedostoa. Kohtuullinen välimuisti parantaa suorituskykyä, mutta tasapainolla on silti merkitystä:

```csharp
// Cache preview images for 24 hours, with a maximum of 160 files
private static readonly TimeSpan PreviewCacheTtl = TimeSpan.FromHours(24);
private const int PreviewCacheMaxFiles = 160;
// Periodically clean up expired cache
```

### 3. Virheiden käsittely

Hajoaa sulavasti, kun ylävirran sivusto ei ole käytettävissä. Tämä suunnittelu varmistaa, että vaikka ulkoinen riippuvuus epäonnistuu, ydinmuokkaustoiminto toimii edelleen normaalisti:

```typescript
// Gracefully degrade when the upstream site is unavailable
try {
    const catalog = await getProjectDesignMdSiteImportCatalog(projectId);
} catch (error) {
    toast.error(t('project.designMd.siteImport.feedback.catalogLoadFailed'));
    // The main editing drawer remains available
}
```

Tämä siro heikentyminen varmistaa, että vaikka ulkoiset riippuvuudet eivät olisi käytettävissä, ydinmuokkaustoiminto jatkaa toimintaansa. Järjestelmän tulee olla kestävä sen sijaan, että se romahtaa heti, kun jokin menee pieleen.

### 4. Käyttökokemuksen optimointi

Vahvista päällekirjoitukset ennen tuontia, äläkä tallenna automaattisesti tuonnin jälkeen. Käyttäjien tulee hallita omia toimiaan:

```typescript
// Confirm overwrite before import
if (isDirty) {
    setConfirmMode('import');
    return;
}

// Do not save automatically after import; let the user confirm
setDraft(result.content); // update draft only
// The content is written to disk only after the user reviews it and clicks Save
```

### 5. Suorituskykyä koskevat näkökohdat

Käytä HTTP-asiakastehdasta välttääksesi liian monien yhteyksien luomisen. Resurssien hallinta saattaa tuntua pieneltä, mutta hyvällä toiminnalla voi olla suuri ero:

```csharp
// Use an HTTP client factory to avoid creating too many connections
private const string DesignSiteProxyClientName = "ProjectDesignSiteProxy";
private static readonly TimeSpan DesignSiteProxyTimeout = TimeSpan.FromSeconds(8);
```

## Suositellut laajennukset

1. **Markdown-parannus**: käytämme tällä hetkellä perustoimintoa `Textarea`, mutta voisimme päivittää CodeMirroriin syntaksin korostamista ja pikanäppäimiä varten. Kun toimittaja voi paremmin, myös dokumentaation kirjoittaminen tuntuu paremmalta.
2. **Esikatselutila**: lisää reaaliaikainen Markdown-esikatselu parantaaksesi muokkauskokemusta. Mitä-näet-on-mitä-saat, antaa ihmisille aina enemmän luottamusta.
3. **Diff merge**: ota käyttöön älykäs yhdistämisalgoritmi yksinkertaisen kokotekstin korvaamisen sijaan. Konfliktit ovat väistämättömiä, mutta konfliktien ratkaisuprosessin ei tarvitse olla tuskallista.
4. **Paikallinen välimuisti**: välimuisti `design.json` tietokannassa vähentääksesi riippuvuutta ulkoisesta sivustosta. Mitä vähemmän riippuvuuksia järjestelmällä on, sitä vakaampi se yleensä on.

## Yhteenveto

HagiCode-projektissa otimme käyttöön täydellisen verkkomuokkaus- ja tuontiratkaisun `DESIGN.md` frontend-backend-yhteistyön kautta. Tämän ratkaisun ydinarvo on seuraavista seikoista:

- **Parempi tehokkuus**: työkaluja ei tarvitse vaihtaa; suunnitteludokumenttien muokkaaminen ja tuonti voi tapahtua yhdessä yhtenäisessä verkkokäyttöliittymässä.
- **Pienempi pääsyn este**: yhden napsautuksen suunnittelumallien tuonti auttaa uusia projekteja pääsemään nopeasti alkuun.
- **Turvallinen ja luotettava**: polun vahvistus, versioristiriitojen tunnistus ja hienot huononemismekanismit pitävät järjestelmän vakaana.
- **Parempi käyttökokemus**: maailmanlaajuinen vetolaatikko, likaisen tilan tunnistus- ja vahvistusikkunat tarkentavat yleistä vuorovaikutuskokemusta.

Tämä ratkaisu on jo käynnissä HagiCode-projektissa ja on ratkaissut tiimin suunnitteluasiakirjojen hallintaan liittyvät ongelmat. Jos kohtaat samanlaisia ​​ongelmia, toivon, että tämä artikkeli antaa sinulle hyödyllisiä ideoita. Tässä ei ole mitään erityisen syvällistä teoriaa, on vain käytännön työ, jossa törmätään ongelmaan ja löydetään tapa ratkaista se.
