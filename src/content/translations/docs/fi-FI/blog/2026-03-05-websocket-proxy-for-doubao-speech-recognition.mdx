## Selaimen WebSocket-todennushaasteiden ratkaiseminen: käytännöllinen välityspalvelinmalli Doubaon puheentunnistukseen

> Selain WebSocket API ei tue mukautettuja HTTP-otsikoita, mikä asettaa haasteen puheentunnistuspalveluille, jotka vaativat todennustietoja otsikoissa. Tässä artikkelissa kerrotaan, kuinka ratkaisimme tämän ongelman HagiCode-projektissa taustavälityspalvelimen mallilla ja kuinka lähestymistapa kehittyi leikkikenttäkokeiluista tuotantokäyttöön.


## Tausta

Kun aloimme rakentaa puheentunnistusta HagiCode-projektiin, valitsimme luottavaisesti ByteDancen Doubao-puheentunnistuspalvelun. Alkuperäinen suunnittelu oli suoraviivainen: anna käyttöliittymän muodostaa yhteys suoraan Doubaon WebSocket-palveluun. Kuinka vaikeaa se voisi olla? Avaa vain yhteys ja lähetä tietoja, eikö niin?

Sitten tuli yllätys: Doubaon API vaatii todennustietojen välittämisen HTTP-otsikoiden kautta, mukaan lukien mm `accessToken` ja `secretKey`. Siitä tuli heti hankala, koska selaimen WebSocket API ei yksinkertaisesti tue mukautettujen otsikoiden asettamista.

Joten mitä teet, kun selain ei anna sinun lähettää niitä?

Punnitsimme silloin kahta vaihtoehtoa:

1. Laita tunnistetiedot URL-kyselyparametreihin – yksinkertaista ja suoraa
2. Lisää välityspalvelintaso taustajärjestelmään - enemmän työtä ensi silmäyksellä

Ensimmäinen vaihtoehto paljastaa kirjautumistiedot suoraan käyttöliittymäkoodissa ja paikallisessa tallennustilassa. Onko se todella turvallista? En viihtynyt siinä. Lisäksi jotkin sovellusliittymät vaativat otsikkopohjaisen vahvistuksen, joten tämä lähestymistapa ei ole edes kannattava.

Lopulta valitsimme toisen vaihtoehdon: ota WebSocket-välityspalvelin käyttöön taustalla. Sattumalta tämä malli validoitiin ensimmäisen kerran leikkipaikkaympäristössämme, ja vasta sen vakauden vahvistamisen jälkeen siirsimme sen tuotantoon. Loppujen lopuksi kukaan ei halua tuotannon kaksinkertaistuvan laboratoriokokeena.

Joskus nämä tekniset tiesulut ovat turhauttavia, mutta ne pakottavat myös oppimaan malleja, jotka osoittautuvat hyödyllisiksi myöhemmin.

## Tekninen haasteanalyysi

### Selaimen WebSocket-rajoitukset

Normaali WebSocket API näyttää ihanan yksinkertaiselta:

```javascript
const ws = new WebSocket('wss://example.com/ws');
```

Mutta juuri tässä yksinkertaisuudessa ongelma piilee – se välittää vain parametrit URL-osoitteessa, eikä se voi asettaa otsikoita HTTP-pyynnön tapaan:

```javascript
// This is not supported in the WebSocket API
const ws = new WebSocket('wss://example.com/ws', {
  headers: {
    'Authorization': 'Bearer token'
  }
});
```

Ja se on ydinkysymys. Palveluille, kuten Doubao-puheentunnistukselle, jotka riippuvat otsikkopohjaisesta todennuksesta, tämä rajoitus on kova esto.

Kun hyväksyt tämän rajoituksen, arkkitehtuurin on muututtava.

### Arkkitehtoniset suunnittelupäätökset

Ratkaisua suunnitellessamme vertailimme kompromisseja huolellisesti.

**Päätös 1: Välityspalvelimen mallin valitseminen**

Vertasimme kahta lähestymistapaa:

| Vaihtoehto | Plussat | Miinukset | Päätös |
|------|------|------|------|
| Alkuperäinen WebSocket | Kevyt, yksinkertainen, suora välitys | Yhteydenhallinta on käsiteltävä manuaalisesti | Valittu |
| SignaaliR | Automaattinen uudelleenkytkentä, vahva kirjoitus | Liian monimutkainen, ylimääräisiä riippuvuuksia | Hylätty |

Valitsimme lopulta alkuperäisen WebSocketin. Ollakseni rehellinen, se oli kevyin vaihtoehto ja sopi paremmin yksinkertaiseen kaksisuuntaiseen binäärivirran edelleenlähetykseen. SignalR:n käyttöönotto olisi tuntunut ylisuunnittelulta tässä käyttötapauksessa, ja se voisi lisätä ylimääräistä latenssia.

**Päätös 2: Yhteydenhallintastrategia**

Otimme käyttöön "yksi yhteys, yksi istunto" -mallin – jokainen käyttöliittymän WebSocket-yhteys liittyy omaan itsenäiseen Doubao-taustayhteyteen.

Edut ovat selvät:
- Yksinkertainen toteuttaa ja yhdenmukaistettu yleisen käyttömallin kanssa
- Helpompi virheenkorjaus ja vianmääritys
- Hyvä resurssieristys, joka estää häiriöt istuntojen välillä

Yksinkertaisesti sanottuna suora ratkaisu on joskus paras. Monimutkaisuus ei automaattisesti tee suunnittelusta parempaa.

**Päätös 3: Todennustietojen tallentaminen**

Tunnistetiedot tallennetaan taustakonfiguraatiotiedostoihin (`appsettings.yml` tai ympäristömuuttujat) ja ladataan riippuvuusinjektion kautta:

- Yksinkertainen konfigurointimalli, joka vastaa olemassa olevia taustakäytäntöjä
- Arkaluonteiset tiedot eivät koskaan pääse käyttöliittymään
- Tukee usean ympäristön asennusta kehitystä, testausta ja tuotantoa varten

Sillä eron tasolla on väliä. Kukaan ei halua valtuustietoja kellua paikoissa, joissa heidän ei pitäisi olla.

### Tietovirran suunnittelu

Tietojen kokonaiskulku näyttää tältä:

```
Frontend (browser)
  │
  │ ws://backend/api/voice/ws
  │ WebSocket (binary)
  ▼
Backend (proxy)
  │
  │ wss://openspeech.bytedance.com/
  │ (with authentication headers)
  ▼
Doubao API
```

Itse virtaus ei ole monimutkainen:
1. Käyttöliittymä muodostaa yhteyden backend-välityspalvelimeen WebSocketin kautta
2. Taustavälityspalvelin vastaanottaa äänidataa ja muodostaa yhteyden Doubao API:hen todennetuilla otsikoilla
3. Doubao API palauttaa tunnistustulokset ja välityspalvelin välittää ne käyttöliittymään
4. Koko prosessi pysyy täysin asynkronisena kaksisuuntaisen suoratoiston kanssa

Kun vastuut on jaettu selkeästi, suunnittelusta tulee melko luonnollista.

## Ydinkomponenttien toteutus

### 1. WebSocket-päätepisteen määritys

```csharp
app.Map("/ws", async context =>
{
    if (context.WebSockets.IsWebSocketRequest)
    {
        // Read configuration from query parameters
        var appId = context.Request.Query["appId"];
        var accessToken = context.Request.Query["accessToken"];

        // Validate required parameters
        if (string.IsNullOrEmpty(appId) || string.IsNullOrEmpty(accessToken))
        {
            context.Response.StatusCode = 400;
            return;
        }

        // Accept the WebSocket connection
        using var webSocket = await context.WebSockets.AcceptWebSocketAsync();

        // Message handling loop
        var buffer = new byte[4096];
        while (!webSocket.CloseStatus.HasValue)
        {
            var result = await webSocket.ReceiveAsync(buffer, CancellationToken.None);

            if (result.MessageType == WebSocketMessageType.Close)
            {
                await webSocket.CloseAsync(
                    result.CloseStatus.Value,
                    result.CloseStatusDescription,
                    CancellationToken.None);
                break;
            }

            // Process audio data
            await HandleAudioDataAsync(buffer, result.Count);
        }
    }
});
```

### 2. Istunnon hallinta

```csharp
public class DoubaoSessionManager : IDoubaoSessionManager
{
    private readonly ConcurrentDictionary<string, DoubaoSession> _sessions = new();

    public DoubaoSession CreateSession(string connectionId)
    {
        var session = new DoubaoSession(connectionId);
        _sessions[connectionId] = session;
        return session;
    }

    public async Task SendAudioAsync(string connectionId, byte[] audioData)
    {
        if (_sessions.TryGetValue(connectionId, out var session))
        {
            await session.SendAudioAsync(audioData);
        }
    }

    public void RemoveSession(string connectionId)
    {
        if (_sessions.TryRemove(connectionId, out var session))
        {
            session.Dispose();
        }
    }
}
```

Käyttämällä `ConcurrentDictionary` istunnon hallinta tarkoittaa, että säikeiden turvallisuus hoidetaan suurelta osin puolestamme. Jokainen saapuva yhteys saa oman istunnon, ja puhdistus tapahtuu automaattisesti, kun yhteys katkeaa.

## Frontend-integraatio käytännössä

### WebSocket-yhteys

```javascript
class DoubaoVoiceClient {
    constructor(config) {
        this.config = config;
        this.ws = null;
    }

    async connect() {
        const url = new URL(this.config.wsUrl);
        // Add query parameters
        Object.entries(this.config.params).forEach(([key, value]) => {
            url.searchParams.set(key, value);
        });

        this.ws = new WebSocket(url);

        return new Promise((resolve, reject) => {
            this.ws.onopen = () => {
                console.log('[DoubaoVoice] Connected');
                resolve();
            };

            this.ws.onmessage = (event) => {
                this._handleMessage(JSON.parse(event.data));
            };

            this.ws.onerror = reject;
        });
    }

    _handleMessage(message) {
        switch (message.type) {
            case 'status':
                this._handleStatus(message.payload);
                break;
            case 'result':
                this.onResult?.(message.payload);
                break;
            case 'error':
                console.error('[DoubaoVoice] Error:', message.payload);
                break;
        }
    }
}

// Usage example
const client = new DoubaoVoiceClient({
    wsUrl: 'ws://localhost:5000/ws',
    params: {
        appId: 'your-app-id',
        accessToken: 'your-access-token',
        sampleRate: 16000,
        bitsPerSample: 16,
        channels: 1
    }
});
```

### Äänen sieppaus ja siirto

Käyttämällä `AudioWorkletNode` äänenkäsittely tarjoaa paremman suorituskyvyn:

```javascript
// audio-worklet.js
class AudioProcessorWorklet extends AudioWorkletProcessor {
    process(inputs, outputs, parameters) {
        const input = inputs[0]?.[0];
        if (!input) return true;

        // Convert to 16-bit PCM
        const pcm = new Int16Array(input.length);
        for (let i = 0; i < input.length; i++) {
            pcm[i] = Math.max(-32768, Math.min(32767, input[i] * 32767));
        }

        this.port.postMessage({
            type: 'audioData',
            data: pcm.buffer
        }, [pcm.buffer]);

        return true;
    }
}

registerProcessor('audio-processor', AudioProcessorWorklet);

// Main thread code
async function startAudioRecording() {
    const stream = await navigator.mediaDevices.getUserMedia({
        audio: {
            echoCancellation: true,
            noiseSuppression: true,
            autoGainControl: true,
            sampleRate: 48000
        }
    });

    const audioContext = new AudioContext();
    const audioSource = audioContext.createMediaStreamSource(stream);

    await audioContext.audioWorklet.addModule('/audio-worklet.js');
    const audioWorkletNode = new AudioWorkletNode(audioContext, 'audio-processor');

    audioWorkletNode.port.onmessage = (event) => {
        if (event.data.type === 'audioData' && ws?.readyState === WebSocket.OPEN) {
            ws.send(event.data.data); // Send binary data directly
        }
    };

    audioSource.connect(audioWorkletNode);
}
```

`AudioWorklet` toimii paljon paremmin kuin `ScriptProcessorNode` ja välttää äänen änkytysongelmat, joita vanhemmat käsittelypolut usein aiheuttavat.

## Huomautuksia ja parhaita käytäntöjä

### 1. Yhteyden valvonta

- Kirjaa istunnon tila säännöllisesti lokiin, jotta voit seurata yhteyden koko elinkaarta
- Tarkkaile äänisegmenttien määrää ja kestoa havaitaksesi epänormaalit yhteydet
- Tallenna Doubao-palvelun yhteyden tila ja uudelleenkytkentäkäyttäytyminen

Nämä ovat perustoiminnallisia käytäntöjä, mutta niillä on todellinen ero tuotantoon.

### 2. Virheiden käsittely

- Tallenna ja kirjaa kaikki WebSocket-poikkeukset
- Käytä `IAsyncDisposable` resurssien siivoamisen varmistamiseksi
- Ota käyttöön siro yhteyden sulkeminen ja aikakatkaisun käsittely

Lyhyesti sanottuna, kannata kestävyyttä.

### 3. Äänimuotovaatimukset

- Näytetaajuus: 16000 Hz (suositus) tai 8000 Hz
- Bittisyvyys: 16-bittinen
- Kanavat: mono
- Koodaus: PCM (raw)

Jos muoto on väärä, tunnistus voi epäonnistua tai laatu voi heikentyä merkittävästi. Näillä vaatimuksilla on merkitystä.

### 4. Turvallisuusnäkökohdat

- Säilytä arkaluontoiset tunnistetiedot vain taustakokoonpanossa
- Varmista yhteysrajoitukset resurssien ehtymisen estämiseksi
- Käytä HTTPS/WSS:ää tuotannossa

Turvallisuus ei ole koskaan vähäinen huolenaihe.

### 5. Suorituskyvyn optimointi

- Käytä asynkronisia toimintoja estääksesi tukkeutumisen
- Säädä puskurin kokoja tarpeen mukaan (oletus: 4096 tavua)
- Harkitse yhteyksien yhdistämistä ja uudelleenkäyttöstrategioita

Käytä näitä optimointeja siellä, missä ne ovat järkeviä työkuormasi kannalta.

## Käyttöönottosuositukset

1. **Dockerin käyttöönotto**: Pakkaa välityspalvelinpalvelu säilöksi helpottamaan skaalausta ja hallintaa
2. **Kuorman tasapainotus**: Käytä Nginxiä tai Envoyta käänteisenä välityspalvelimena WebSocket-liikenteelle
3. **Terveystarkastukset**: Ota käyttöön sykepohjaisia tarkistuksia palvelun saatavuuden seuraamiseksi
4. **Lokin yhdistäminen**: Lähetä lokit keskitettyyn lokijärjestelmään, kuten ELK tai Loki

Käyttöönotto voi olla yksinkertaista tai monimutkaista tiimistä ja ympäristöstä riippuen, joten säädä sen mukaan.

## Yhteenveto

WebSocket-välityspalvelinmalli ratkaisee ydinongelman, että selaimen WebSocket API ei tue mukautettuja otsikoita. HagiCode-projektissa tämä malli osoittautui sekä toteuttamiskelpoiseksi että vakaaksi, kun se siirtyi leikkikentän validoinnista tuotantokäyttöön.

Tärkeimmät takeawayt:
- Taustavälityspalvelin voi välittää todennustiedot turvallisesti
- Native WebSocket on kevyt ja tehokas yksinkertaisiin skenaarioihin
- "Yksi yhteys, yksi istunto" -malli yksinkertaistaa sekä käyttöönottoa että virheenkorjausta
- Frontend-backend-protokollan tulisi erottaa ohjaussignaalit äänidatasta

Jos olet rakentamassa ominaisuutta, joka riippuu myös WebSocket-todennusta, toivon, että tämä malli antaa sinulle hyödyllisen lähtökohdan.

Jos sinulla on kysyttävää, keskustele niistä kanssamme. Tekninen kehitys tapahtuu nopeammin, kun ihmiset vertailevat muistiinpanoja.
