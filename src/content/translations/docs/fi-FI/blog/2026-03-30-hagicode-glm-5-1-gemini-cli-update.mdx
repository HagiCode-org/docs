## Täysi GLM-5.1-tuki ja Gemini CLI -integraatio: HagiCoden polku usean mallin evoluutioon

> Tämä artikkeli esittelee kaksi suurta viimeaikaista HagiCode-alustan päivitystä: täyden tuen Zhipu AI GLM-5.1 -mallille ja Gemini CLI:n onnistuneen integroinnin kymmenenneksi agentti-CLI:ksi. Yhdessä nämä päivitykset vahvistavat edelleen alustan usean mallin ominaisuuksia ja usean CLI:n ekosysteemiä.

:::note[Time Context Notice]
Tämä artikkeli tallentaa usean CLI:n laajennuksen edistymisen sellaisena kuin se oli julkaisuhetkellä. Katso nykyinen aina tuettu laajuus kohdasta [AI Agent CLI -asennus](/fi-FI/related-software-installation/ai-agent-cli/) ja [Asennusopas](/fi-FI/installation); the `IFlowCli` tässä mainittu edustaa vain historiallista kokoonpanoa tuolloin, eikä se ole osa nykyistä aktiivista tukialuetta.
:::

## Tausta

Aika menee todella nopeasti. Suurten kielimallien kehitys on noussut kuin bambu keväällä. Vähän aikaa sitten kannustimme vielä "tekoälyä, joka osaa kirjoittaa koodia". Nyt elämme jo usean mallin yhteistyön ja useiden työkalujen integroinnin aikakautta. Onko se jännittävää? Ehkä. Loppujen lopuksi kehittäjät eivät ole koskaan tarvitseneet vain itse työkalua, vaan eri skenaarioiden mukauttamisen ja joustavan vaihtamisen helppoutta tarvittaessa.

AI-avusteisena koodausalustana HagiCode on hiljattain toivottanut tervetulleeksi kaksi tärkeää kehitystä: ensinnäkin Zhipu AI:n GLM-5.1-mallin täydellinen integrointi; Toiseksi Gemini CLI:n virallinen lisäys kymmenenneksi tuetuksi agentti CLI:ksi. Nämä kaksi päivitystä eivät ehkä kuulosta maata järisyttäviltä, ​​mutta ne ovat kiistatta hyviä uutisia alustan jatkuvalle kypsymiselle.

GLM-5.1 on Zhipu AI:n uusin lippulaivamalli. Verrattuna GLM-5.0:aan, se tarjoaa vahvemman päättelyn, syvemmän koodin ymmärtämisen ja sujuvamman työkalukutsun. Vielä tärkeämpää on, että se on ensimmäinen GLM-malli, joka tukee kuvansyöttöä. Mitä se tarkoittaa? Se tarkoittaa, että käyttäjät voivat antaa tekoälyn katsoa suoraan kuvakaappauksesta sen sijaan, että he yrittäisivät kuvailla ongelmaa sanoin. Kun olet käyttänyt tätä mukavuutta, ymmärrät heti sen arvon.

Samaan aikaan kautta `HagiCode.Libs.Providers` arkkitehtuuri, HagiCode integroi onnistuneesti Gemini CLI:n alustaan. Tämä on nyt kymmenes agentti CLI. Ollakseni rehellinen, tähän pisteeseen pääseminen tuo vaatimattoman onnistumisen tunteen.

On myös syytä mainita, että HagiCoden kuvien latausominaisuuden avulla käyttäjät voivat kommunikoida tekoälyn kanssa suoraan kuvakaappausten kautta. Jopa GLM 4.7:ää käytettäessä alusta toimii edelleen hyvin ja on jo auttanut suorittamaan monia tärkeitä rakennustehtäviä. Mitä tulee GLM-5.1:een, se menee luonnollisesti askeleen pidemmälle.

Projektin arkisto: [github.com/HagiCode-org/site](https://github.com/HagiCode-org/site)

## Multi-CLI-arkkitehtuurin suunnittelu

Yksi HagiCoden ydinvahvuuksista on sen tuki useille AI-ohjelmoinnin CLI-työkaluille yhtenäisen abstraktiokerroksen kautta. Tämän suunnittelun etu on itse asiassa melko yksinkertainen: uusia työkaluja voi tulla sisään, vanhat työkalut voivat jäädä, eikä koodikanta muutu kaaokseksi. Ollakseni rehellinen, niin kaikki haluaisivat elämän toimivan.

### AIProviderType enum

Alusta määrittelee tuetut CLI-palveluntarjoajan tyypit `AIProviderType` luettelo:

```csharp
public enum AIProviderType
{
    ClaudeCodeCli = 0,    // Claude Code CLI
    CodexCli = 1,          // GitHub Copilot Codex
    GitHubCopilot = 2,     // GitHub Copilot
    CodebuddyCli = 3,     // Codebuddy CLI
    OpenCodeCli = 4,      // OpenCode CLI
    IFlowCli = 5,         // IFlow CLI
    HermesCli = 6,        // Hermes CLI
    QoderCli = 7,         // Qoder CLI
    KiroCli = 8,          // Kiro CLI
    KimiCli = 9,          // Kimi CLI
    GeminiCli = 10,       // Gemini CLI (new)
}
```

Kuten näette, Gemini CLI liittyy tähän perheeseen kymmenentenä jäsenenä. Jokaisella CLI:llä on omat erityispiirteensä ja käyttöskenaarionsa, joten käyttäjät voivat valita joustavasti tarpeidensa mukaan. Loppujen lopuksi monet tiet johtavat Roomaan; jotkut ovat yksinkertaisesti helpompia kuin toiset.

### Palveluntarjoajan arkkitehtuuri

`HagiCode.Libs.Providers` tarjoaa yhtenäisen palveluntarjoajan käyttöliittymän, joka tekee jokaisesta CLI-integraatiosta standardoitua ja tiivistä. Otetaan esimerkkinä Gemini CLI:

```csharp
public class GeminiProvider : ICliProvider<GeminiOptions>
{
    private static readonly string[] DefaultExecutableCandidates = ["gemini", "gemini-cli"];
    private const string ManagedBootstrapArgument = "--acp";

    public string Name => "gemini";
    public bool IsAvailable => _executableResolver.ResolveFirstAvailablePath(DefaultExecutableCandidates) is not null;
}
```

Tämän suunnittelun edut ovat:
- Uuden CLI:n integrointi vaatii vain yhden Provider-luokan toteuttamisen
- Yhtenäinen elinkaarihallinta ja istuntojen yhdistäminen
- Automaattinen aliaksen ratkaisu ja suoritettavan tiedoston etsintä

Yksinkertaisesti sanottuna tämä muotoilu muuttaa monimutkaiset asiat yksinkertaisemmiksi ja helpottaa elämää.

### Palveluntarjoajan rekisteri

Palveluntarjoajan rekisteri käsittelee automaattisesti aliaskartoituksen ja rekisteröinnin:

```csharp
if (provider is GeminiProvider)
{
    registry.Register(provider.Name, provider, ["gemini-cli"]);
    continue;
}
```

Tämä tarkoittaa, että käyttäjät voivat kutsua Gemini CLI:tä kummalla tahansa `gemini` tai `gemini-cli`, ja järjestelmä tunnistaa sen automaattisesti. Se on kuin ystävä, jolla on sekä virallinen nimi että lempinimi - joka tapauksessa ihmiset tietävät, ketä tarkoitat.

## GLM-5.1 mallin tuki

GLM-5.1 on Zhipu AI:n uusin lippulaivamalli, ja HagiCode on saanut täyden tuen sille.

### Toissijaisten ammattien luettelo

HagiCode hallitsee kaikkia tuettuja malleja Secondary Professions Catalogin kautta. Tässä on GLM-sarjan kokoonpano:

| Mallin tunnus | Nimi | Tukee kuvaa | Yhteensopivat CLI-perheet |
|----------|------|----------------|-------------------------|
| `glm-4.7` | GLM 4.7 | - | claude, koodikaveri, hermes, qoder, kiro |
| `glm-5` | GLM 5 | - | claude, koodikaveri, hermes, qoder, kiro |
| `glm-5-turbo` | GLM 5 Turbo | - | claude, koodikaveri, hermes, qoder, kiro |
| `glm-5.0` | GLM 5.0 (vanha) | - | claude, koodikaveri, hermes, qoder, kiro |
| `glm-5.1` | GLM 5.1 | **totta** | claude, koodikaveri, hermes, qoder, kiro |

GLM-5.1:n tärkeimmät ominaisuudet voidaan tiivistää seuraavasti:
- Itsenäinen version tunniste, jossa ei ole vanhoja matkatavaroita
- Ensimmäinen GLM-malli, joka tukee kuvansyöttöä
- Vahvempi päättely ja koodin ymmärtäminen
- Laaja usean CLI:n yhteensopivuus

### GLM-5.1 vs GLM-5.0

Kooditasolla keskeinen ero GLM-5.1:n ja GLM-5.0:n välillä näkyy tässä:

```csharp
// GLM-5.0 (Legacy) - contains special retention logic
private const string Glm50CodebuddySecondaryProfessionId = "secondary-glm-5-codebuddy";
private const string Glm50CodebuddyModelValue = "glm-5.0";

// GLM-5.1 - standalone new model identifier
private const string Glm51SecondaryProfessionId = "secondary-glm-5-1";
private const string Glm51ModelValue = "glm-5.1";
```

GLM-5.0 sisältää "Legacy"-merkinnän, koska se on vanha version tunniste, joka on säilytetty taaksepäin yhteensopivuuden vuoksi. GLM-5.1 on sitä vastoin upouusi erillinen versio ilman historiallista taakkaa. Jotkut asiat jäävät menneisyyteen; toiset matkustavat kevyemmin ja nopeammin.

### Määritä GLM-5.1

Tässä on esimerkki GLM-5.1:n käyttämisestä HagiCodessa:

```json
{
  "primaryProfessionId": "profession-claude-code",
  "secondaryProfessionId": "secondary-glm-5-1",
  "model": "glm-5.1",
  "reasoning": "high"
}
```

## Kuvan latausominaisuus

HagiCoden kuvatuki toteutetaan `SupportsImage` omaisuus päällä `SecondaryProfession`:

```csharp
public class HeroSecondaryProfessionSettingDto
{
    public bool SupportsImage { get; set; }
}
```

Secondary Professions Catalog -luettelossa GLM-5.1-kokoonpano näyttää tältä:

```json
{
  "id": "secondary-glm-5-1",
  "supportsImage": true
}
```

Tämä tarkoittaa, että käyttäjät voivat ladata kuvakaappauksia suoraan tekoälyanalyysiä varten, kuten:
- Kuvakaappauksia virheilmoituksista
- Ongelmia käyttöliittymänäytössä
- Datan visualisointikaaviot
- Koodin suorittamisen tulokset

Kaikkea ei enää tarvitse kuvata manuaalisesti - lataa vain kuvakaappaus. Tämän ominaisuuden käyttömukavuus on ilmeinen, kun olet käyttänyt sitä. Joskus yksi katse kertoo enemmän kuin pitkä selitys.

## Gemini CLI -integraatio

Kymmenentenä Agent CLI:nä Gemini CLI on integroitu HagiCodeen tavallisen Provider-arkkitehtuurin kautta.

### Asetusvaihtoehdot

Gemini CLI tukee runsaasti konfigurointivaihtoehtoja:

```csharp
public class GeminiOptions
{
    public string? ExecutablePath { get; set; }
    public string? WorkingDirectory { get; set; }
    public string? SessionId { get; set; }
    public string? Model { get; set; }
    public string? AuthenticationMethod { get; set; }
    public string? AuthenticationToken { get; set; }
    public Dictionary<string, string?> AuthenticationInfo { get; set; }
    public Dictionary<string, string?> EnvironmentVariables { get; set; }
    public string[] ExtraArguments { get; set; }
    public TimeSpan? StartupTimeout { get; set; }
    public CliPoolSettings? PoolSettings { get; set; }
}
```

Nämä vaihtoehdot kattavat kaiken perusasetuksista lisäominaisuuksiin, mikä antaa käyttäjille joustavuuden määrittää CLI omien tarpeidensa mukaan. Jokaisen työnkulku on erilainen, joten pieni joustavuus on aina tervetullutta.

### ACP-viestintäprotokolla

Gemini CLI tukee ACP:tä (Agent Communication Protocol), joka on HagiCoden yhtenäinen CLI-viestintästandardi. ACP:n kautta eri CLI:t voivat olla vuorovaikutuksessa alustan kanssa johdonmukaisella tavalla, mikä yksinkertaistaa huomattavasti integrointityötä. Lyhyesti sanottuna se standardoi monimutkaiset osat, jotta kaikki voivat työskennellä helpommin.

## HagiCoden oma rakennuskäytäntö

Tosielämän käytännöistä puheen ollen, paras esimerkki on HagiCode-alustan oma rakennustyönkulku. HagiCoden kehitysprosessissa on jo käytetty täysimääräisesti AI-valmiuksia.

### Toimii hyvin jopa GLM 4.7:n kanssa

HagiCoden alustasuunnittelu on hyvin optimoitu, joten se voi silti tarjota hyvän kehityskokemuksen jopa GLM 4.7:n kanssa. Alusta on jo auttanut toteuttamaan useita tärkeitä rakennusprojekteja, mukaan lukien:
- Useiden CLI-palveluntarjoajien integrointi
- Kuvien latausominaisuuden toteutus
- Dokumentaation luominen ja sisällön julkaiseminen

Se on itse asiassa hyvä asia. Kaikki eivät tarvitse aina uusinta. Se mikä sopii sinulle parhaiten, on usein tärkeintä.

### GLM-5.1 tuottaa enemmän pienemmällä vaivalla

GLM-5.1:een päivityksen jälkeen nämä ominaisuudet vahvistuvat entisestään:
- Vahvempi koodin ymmärtäminen, mikä vähentää edestakaisin viestintää
- Tarkempi riippuvuusanalyysi, joka osoittaa heti oikeaan suuntaan
- Tehokkaampi virheiden diagnosointi, ongelmien paikantaminen nopeammin
- Tuki kuvan syöttämiselle, nopeuttaa ongelmien kuvauksia

Se on kuin vaihtamista polkupyörästä autoon. Voit silti saavuttaa samaan määränpäähän, mutta nopeus ja mukavuus eivät ole samat.

## Usean CLI:n integroinnin parhaat käytännöt

`HagiCode.Libs.Providers` tarjoaa yhtenäisen mekanismin rekisteröintiä ja käyttöä varten:

```csharp
services.AddHagiCodeLibs();

var gemini = serviceProvider.GetRequiredService<ICliProvider<GeminiOptions>>();
var codebuddy = serviceProvider.GetRequiredService<ICliProvider<CodebuddyOptions>>();
var hermes = serviceProvider.GetRequiredService<ICliProvider<HermesOptions>>();
```

Tämä riippuvuusinjektiorakenne pitää käytön eri CLI:issä erittäin ytimekkäänä ja tekee myös yksiköiden testaamisesta ja pilkkaamisesta helpompaa. Puhdas koodi on tapa olla vastuussa itsestäsi.

### Huomautuksia

Käytännössä on hyvä muistaa muutama seikka:

1. **API-avaimen määritys**: Varmista `ANTHROPIC_AUTH_TOKEN` on asetettu oikein tai mallia ei voida kutsua
2. **Mallin saatavuus**: GLM-5.1 on otettava käyttöön vastaavan mallin toimittajan toimesta
3. **Kuvaominaisuus**: Vain mallit, joissa `supportsImage: true` voi käyttää kuvien latausta
4. **CLI-asennus**: Varmista ennen Gemini CLI:n käyttöä `gemini` tai `gemini-cli` on järjestelmässä PATH

Nämä voivat olla pieniä yksityiskohtia, mutta huonosti käsitellyt pienet yksityiskohdat voivat muuttua suuriksi ongelmiksi, joten niihin kannattaa kiinnittää huomiota.

## Johtopäätös

GLM-5.1:n täyden tuen ja Gemini CLI:n onnistuneen integroinnin ansiosta HagiCode vahvistaa entisestään kykyään usean mallin, usean CLI:n AI-ohjelmointialustana. Nämä päivitykset eivät vain anna käyttäjille enemmän valinnanvaraa, vaan myös osoittavat HagiCoden tulevaisuuteen suuntautuvan arkkitehtuurin ja skaalautuvuuden.

GLM-5.1:n kuvatuki yhdistettynä HagiCoden kuvakaappauslatausominaisuuteen mahdollistaa sen, että tekoäly "ymmärtää kuvasta" ja vähentää huomattavasti ongelmien kuvauksen kustannuksia. Kymmenen CLI:n tuen ansiosta käyttäjät voivat valita joustavasti AI-ohjelmointiavustajan, joka sopii parhaiten heidän mieltymyksiinsä ja skenaarioihinsa. Lisää valinnanvaraa on lähes aina hyvä asia.

Mikä tärkeintä, HagiCoden oma rakennuskäytäntö osoittaa, että alusta voi jo toimia hyvin ja suorittaa monimutkaisia tehtäviä jopa GLM 4.7:n kanssa, kun taas päivittäminen GLM-5.1:een voi parantaa kehitystehokkuutta entisestään. Elämä on usein myös sellaista: aina ei tarvitse ehdottomasti parasta, vain sitä, mikä sinulle sopii. Tietysti, jos mikä sopii sinulle, voi tulla vielä paremmaksi, niin sitä parempi.

Jos olet kiinnostunut usean mallin ja CLI:n AI-ohjelmointialustasta, kokeile HagiCodea – avoimen lähdekoodin, ilmainen ja edelleen kehittyvä. Sen kokeileminen ei maksa mitään, ja se voi osoittautua juuri sitä mitä tarvitset.
