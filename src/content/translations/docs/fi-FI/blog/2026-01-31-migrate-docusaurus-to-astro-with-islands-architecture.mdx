## Docusaurus 3.x:stä Astro 5.x:ään: Retrospektiivi HagiCode-sivuston siirtoon

> Tässä artikkelissa tarkastellaan HagiCoden virallisen verkkosivuston täyttä siirtymistä Docusaurus 3.x:stä Astro 5.x:ään. Sukeltamme syvästi siihen, kuinka Astro's Islands Architecture auttoi meitä ratkaisemaan suorituskyvyn pullonkauloja ja säilyttämään nykyiset React-komponenttivaraisuutemme ja parantamaan sekä rakennusnopeuteen että lataussuorituskykyämme.

## Tausta

Tammikuussa 2026 teimme "sydämensiirron" HagiCoden virallisella sivustolla siirtämällä sen ydinkehyksen kokonaan Docusaurus 3.x:stä Astro 5.x:ään. Tämä ei ollut impulsiivinen uudelleenkirjoitus, vaan huolellisesti harkittu tekninen päätös.

Ennen siirtoa sivustomme oli toiminnallisesti valmis, mutta siinä oli alkanut näkyä klassisia "luksusongelmia": paisuneita rakennusvirheitä, liiallisia JavaScript-hyötykuormia ja sivun latausnopeus monimutkaisilla dokumentaatiosivuilla. AI-koodausavustajana HagiCode tarvitsee usein dokumentaatiota ja ominaisuuspäivityksiä, joten rakennustehokkuus vaikuttaa suoraan julkaisunopeuteen. Samalla halusimme sivuston olevan hakukoneystävällisempi (SEO), jotta useampi kehittäjä voisi löytää projektin.

Ratkaisimme nämä kipukohdat teimme rohkean päätöksen: rakennamme koko järjestelmän uudelleen Astrolla. Päätöksen vaikutus voi olla jopa suurempi kuin odotat. Palaan yksityiskohtiin pian.

## Tietoja HagiCodesta

Tässä artikkelissa jaettu sivuston siirtotapa perustuu käytännön kokemukseemme [HagiCode](https://github.com/HagiCode-org/site) hanke.

HagiCode on tekoälyn koodausassistentti, joka keskittyy parantamaan kehitystehokkuutta. Emme välitä vain ydinominaisuuksien iteroinnista, vaan myös kehittäjäkokemuksesta. Tämän sivuston uudelleentekijän tarkoituksena oli myös tarjota käyttäjille nopein mahdollinen kokemus asiakirjojamme ja virallista verkkosivustoamme selattaessa.

## Miksi jättää kypsä Docusaurus-ekosysteemi?

React-ekosysteemissä Docusaurus on pitkään ollut "vakiovastaus" dokumentaatiosivustoille. Se toimii heti alusta alkaen, tarjoaa rikkaan laajennusekosysteemin ja sillä on aktiivinen yhteisö. Mutta kun HagiCode sai lisää ominaisuuksia, tunsimme myös sen rajoitukset:

1.  **Suorituskyvyn pullonkaulat**: Docusaurus on pohjimmiltaan React SPA (yksisivuinen sovellus). Vaikka kirjoitat vain staattisia sivuja, asiakkaan on silti ladattava React-ajoaika ja hydratoitava sivu, mikä on tarpeettoman raskasta yksinkertaisille dokumenttisivuille.
2.  **Suuri sisältökoko**: Vaikka sivulla olisi hyvin vähän sisältöä, niputettu JS-koko pysyy suhteellisen kiinteänä. Se ei ole ihanteellinen mobiilikäyttäjille tai huonoille verkko-olosuhteille.
3.  **Rajoitettu joustavuus**: Vaikka se on laajennettavissa, halusimme enemmän matalan tason hallintaa rakennusprosessiin.

Astro saapui juuri oikeaan aikaan ratkaisemaan nämä ongelmat. Se esitteli uuden "saarten arkkitehtuurin": oletusarvoisesti Astro luo staattista HTML-koodia ilman JavaScriptiä, ja vain interaktiivisuutta vaativat komponentit "aktivoidaan" ja lataavat JS:n. Tämä tarkoittaa, että suurin osa sivustostamme tulee puhtaaksi HTML:ksi ja latautuu erittäin nopeasti.

## Ydinsiirtostrategia: Sujuva arkkitehtuurimuutos

Siirtyminen ei ollut pelkkää kopiointia ja liittämistä. Se vaati ajattelutavan muutosta. Siirryimme Docusauruksen "all React" -mallista Astron "Core + Islands" -malliin.

### 1. Kokoonpanojärjestelmän uudelleenrakentaminen

Ensin meidän piti muuttaa pois `docusaurus.config.ts` to `astro.config.mjs`. Tämä ei ollut vain tiedoston uudelleennimeäminen, vaan reititys- ja rakennuslogiikan uudelleenkirjoitus.

Docusaurusissa kaikki on laajennusta. Astrossa kaikki on integraatiota. Meidän piti määrittää uudelleen sivuston peruspolku, koontitulostustila (staattinen vs. SSR) ja resurssien optimointistrategia.

**Ennen siirtoa:**
```typescript
// docusaurus.config.ts
export default {
  title: 'HagiCode',
  url: 'https://hagicode.com',
  baseUrl: '/',
  // ... more configuration
};
```

**Siirron jälkeen:**
```javascript
// astro.config.mjs
import { defineConfig } from 'astro/config';
import react from '@astrojs/react';

export default defineConfig({
  integrations: [react()],
  site: 'https://hagicode.com',
  base: '/',
  // Optimization settings for static assets
  build: {
    inlineStylesheets: 'auto',
  },
});
```

### 2. Mitä pitää säilyttää ja mitä refaktoroida reaktiokomponenteissa

Tämä oli muuttoliikkeen tuskallisin osa. Nykyisellä sivustollamme oli monia React-komponentteja, kuten välilehdet, koodin korostukset, palautepainikkeet ja paljon muuta. Niiden heittäminen pois olisi turhaa, mutta kaiken säilyttäminen tekisi JavaScriptistä liian raskaan.

HagiCode otti käyttöön **progressiivisen nesteytysstrategian**:
*   **Puhtaat staattiset komponentit**: Esityssisällön, kuten otsikoiden, alatunnisteiden ja pelkkää tekstidokumentaatiota varten kirjoitimme ne uudelleen Astro-komponenteiksi (`.astro` tiedostot) ja renderöi ne suoraan HTML-muotoon rakennusvaiheessa.
*   **Interaktiiviset saarekkeet**: Säilytimme React-toteutuksen ja lisäsimme komponentteja, joiden on pysyttävä vuorovaikutteisina, kuten teeman vaihtajat, välilehtien vaihto ja koodilohkon kopiointipainikkeet. `client:load` tai `client:visible` direktiivit.

Esimerkiksi dokumentaatiossa yleisesti käytetty Tabs-komponenttimme:

```jsx
// src/components/Tabs.jsx
import { useState } from 'react';
import './Tabs.css'; // Import styles

export default function Tabs({ items }) {
  const [activeIndex, setActiveIndex] = useState(0);
  // ... state logic
  return (
    <div className="tabs-wrapper">
      {/* Rendering logic */}
    </div>
  );
}
```

Kun sitä käytetään Markdownissa, kerromme Astrolle nimenomaisesti: "Tämä komponentti tarvitsee JS:n."

```markdown
// src/content/docs/example.mdx
import Tabs from '../../components/Tabs.jsx';

<!-- Load JS only when the component enters the viewport -->
<Tabs client:visible items={...} />
```

Tällä tavalla interaktiiviset komponentit katseluportin ulkopuolella eivät kilpaile kaistanleveydestä, mikä parantaa huomattavasti ensimmäisen näytön latausnopeutta.

### 3. Tyylijärjestelmän mukauttaminen: CSS-moduuleista laajennettuun CSS:ään

Docusaurus tukee oletusarvoisesti CSS-moduuleja, kun taas Astro kannustaa Scoped CSS:ää `<style>` -tunniste. Molempien taustalla on tyylin eristäminen, mutta syntaksi on erilainen.

HagiCode-migroinnin aikana muunsimme monimutkaisimmat CSS-moduulit Astro-suojattuihin tyyleihin. Tämä osoittautui itse asiassa hyväksi, koska sisään `.astro` tiedostot tyylit ja mallit ovat samassa tiedostossa, mikä tekee ylläpidosta intuitiivisempaa.

**Ennen refaktorointia:**
```css
/* Tabs.module.css */
.wrapper { background: var(--ifm-background-color); }
```

**Refaktoroinnin jälkeen (Astro Scoped):**
```html
<!-- Tabs.astro -->
<div class="tabs-wrapper">
  <slot />
</div>

<style>
  .tabs-wrapper {
    /* Use CSS variables directly to adapt to the theme */
    background: var(--bg-color);
    padding: 1rem;
  }
</style>
```

Samalla yhtenäistimme globaalin CSS-muuttujajärjestelmän ja käytimme Astron ympäristötietoisia ominaisuuksia varmistaaksemme pimeän tilan sujuvan vaihdon sivuilla.

## Käytännössä kohtaamat sudenkuopat ja kuinka ratkaisimme ne

Varsinaisen HagiCode-siirron aikana törmäsimme useisiin ongelmiin. Tässä on muutamia tyypillisimpiä.

### 1. Polku ja ympäristö muuttuvat kipupisteet

HagiCode tukee alipolun käyttöönottoa, kuten käyttöönottoa GitHub Pages -alihakemistossa. Docusaurusessa, `baseUrl` käsitellään automaattisesti. Astrossa meidän on kuitenkin oltava varovaisempia käsitellessämme kuvalinkkejä ja API-pyyntöjä.

Otimme käyttöön ympäristömuuttujamekanismin hallitaksemme tätä johdonmukaisesti:

```javascript
// Handle paths in the build script
const getBasePath = () => import.meta.env.VITE_SITE_BASE || '/';
```

Älä koodaa polkuja, jotka alkavat kirjaimella `/` koodissasi. Kehityksessä verrattuna tuotantoon tai konfiguroinnin jälkeen a `base` polku, se voi aiheuttaa 404:t resursseille.

### 2. CommonJS Script -yhteensopivuus

Vanhalla sivustollamme oli joitakin Node.js-skriptejä, joita käytettiin esimerkiksi metriikkatietojen automaattiseen hakemiseen ja sivustokartan päivittämiseen, ja ne oli kirjoitettu CommonJS:llä (`require`). Astro- ja modernit rakennustyökalut ovat täysin omaksuneet ES-moduulit (`import`/`export`).

Jos sinulla on myös samanlaisia komentosarjoja, muista muokata ne kaikki ES-moduuleiksi. Tähän suuntaan ekosysteemi liikkuu, ja mitä nopeammin teet muutoksen, sitä vähemmän ongelmia sinulla on myöhemmin.

```javascript
// Old way
const fs = require('fs');

// New way
import fs from 'fs';
```

### 3. Älä unohda hakukoneoptimointia ja uudelleenohjauksia

Hakukoneet ovat jo indeksoineet HagiCoden vanhat Docusaurus-sivut. Jos vaihdat suoraan Astroon ja URL-osoitteen rakenne muuttuu, saatat päätyä suureen määrään 404:itä ja haun sijoitusta merkittävästi.

Määritimme uudelleenohjaussäännöt Astrossa:

```javascript
// astro.config.mjs
export default defineConfig({
  redirects: {
    '/docs/old-path': '/docs/new-path',
    // Map old links to new links in bulk
  }
});
```

Tai voit käsitellä tämän palvelimen määritystasolla. Varmista, että vanhat linkit voidaan 301 uudelleenohjata uusiin osoitteisiin, koska tämä on tärkeää hakukoneoptimoinnin kannalta.

## Yhteenveto

HagiCodelle siirtyminen Docusaurusesta Astroon ei ollut vain kehyspäivitys. Se oli myös "suoritus ensin" -filosofian käytännön toteutus.

**Mitä saimme:**
*   **Erinomainen Lighthouse-pisteet**: Siirron jälkeen HagiCode-sivuston suorituskykypisteet lähestyivät helposti täydellistä arvoa.
*   **Nopeampi rakennusnopeus**: Astron inkrementaaliset rakennusominaisuudet lyhentävät dokumentaatiopäivitysten julkaisuajan puoleen.
*   **Säilötty joustavuus**: Islands Architecturessa emme uhrannut mitään interaktiivisia ominaisuuksia ja pystyimme silti käyttämään Reactia tarvittaessa.

Jos ylläpidät myös dokumentaatiokeskeistä sivustoa ja kamppailet paketin koon tai latausnopeuden kanssa, Astroa kannattaa kokeilla. Vaikka siirtoprosessi vaatiikin leikkausta, kuten PCoden uudelleennimeämisen HagiCodeksi ja komponenttien siirtämistä yksitellen, vastineeksi saamasi silkkisen pehmeä käyttökokemus tekee siitä ehdottomasti kannattavan.

Tässä artikkelissa jaettu rakennusjärjestelmä on täsmällinen lähestymistapa, jonka kehitimme todellisen yrityksen ja erehdyksen avulla rakentaessamme HagiCodea. Jos pidät tätä lähestymistapaa arvokkaana, se kertoo jotain teknisistä vahvuuksistamme, ja itse HagiCode on luultavasti myös lähemmän tarkastelun arvoinen.
