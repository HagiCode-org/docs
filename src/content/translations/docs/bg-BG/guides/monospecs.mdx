## Преглед

Monospecs е подходът на HagiCode за управление на множество хранилища. Той проследява и управлява под-хранилища чрез `monospecs.yaml`, и работи за всеки проект, който се нуждае от едно място за координиране на множество независими хранилища.

### Основни концепции

**Конфигурационен файл**

`monospecs.yaml` е сърцевината на цялото решение за управление, определяйки:
- Списък и местоположение на подхранилищата
- Git URL за всяко хранилище
- Екранно име и икона
- Поведение при интегриране с OpenSpec

**Разлики от обикновените хранилища**

| Характеристика | Редовно хранилище | Проект, управляван от Monospecs |
|----------|-------------------|---------------------------|
| **Местоположение на OpenSpec** | Основна директория `openspec/` | Основната директория на хранилището `openspec/` |
| **Промяна на обхвата** | Ограничено до текущото хранилище | Може да включва множество подхранилища |
| **Чистота на подхранилище** | Спецификации, смесени с код | Спецификациите, установени в основното хранилище, подхранилищата са по-чисти |
| **Автоматичен ангажимент за архивиране** | Нуждае се от ръчни ангажименти на спецификациите | `commit_when_archive: true` автоматично записва спецификациите в основното хранилище |

### Защо да използвате Monospecs?

- **Унифицирано управление**: Управлявайте всички под-хранилища в един конфигурационен файл
- **Поддръжка за автоматизация**: Предоставя метаданни за хранилище за скриптове и инструменти
- **Удобство за AI**: Структурираната конфигурация помага на AI Agent да разбере връзките между хранилищата
- **OpenSpec Integration**: Безпроблемна интеграция с работния процес за управление на промените
- **По-чисти подхранилища**: Спецификациите се съхраняват в основното репо, подхранилищата съдържат само действителен код

### Приложими сценарии

Използвайте първо Monospec в сценарии като тези:

**1. Съвместна разработка с множество хранилища**
Когато даден проект е разделен на множество независими хранилища:
- Frontend приложение, backend услуга и настолно приложение, разработени отделно
- Необходимо е да се координират версиите и зависимостите между хранилищата
- Документация и помощни проекти отделно от основния код

**2. Управление на спомагателни проекти**
Използвайте Monospecs за управление на:
- Проект за документация
- Официален сайт
- Инструменти и скриптове за изграждане
- Всяко хранилище поддържа свой собствен контрол на версиите

**3. Разработка на функции за кръстосано хранилище**
Функция, която изисква промени в множество хранилища:
- Необходимо е да се проследят промените в кои хранилища
- Автоматично ангажиране на спецификациите за коригиране на целевото хранилище
- Координирайте актуализациите на версиите и зависимостите

## Бърз старт

### Инициализация

Следвайте тези стъпки, за да завършите инициализацията на monospec:

#### 1. Създайте празно git хранилище

Създайте празен `git` папка във вашето основно хранилище (забележка: това е `git`, не `repos/` - това е конвенцията за моноспектърите):

```bash
mkdir git
cd git
git init
cd ..
```

#### 2. Конфигурирайте файла monospecs

Създайте `monospecs.yaml` конфигурационен файл в основната ви директория на хранилището:

```yaml
version: "1.0"
commit_when_archive: true

repositories:
  - path: "repos/frontend"
    url: "https://github.com/your-org/frontend.git"
    displayName: "Frontend App"
    icon: "🌐"

  - path: "repos/backend"
    url: "https://github.com/your-org/backend.git"
    displayName: "Backend Service"
    icon: "⚙️"
```

**Конфигурационни елементи**:
| Конфигурационен елемент | Тип | Задължително | Описание |
|----------------|------|----------|-------------|
| `version` | низ | да | Версия на конфигурационния файл |
| `commit_when_archive` | булево | не | Дали да се ангажират автоматично спецификациите при архивиране (не ангажирайте подрепо код) |
| `repositories` | масив | да | Списък с конфигурация на хранилище |
| `repositories[].path` | низ | да | Път на хранилището (спрямо корена на проекта) |
| `repositories[].url` | низ | да | URL адрес на Git хранилище |
| `repositories[].displayName` | низ | не | Екранно име на хранилище |
| `repositories[].icon` | низ | не | Икона на хранилище (емоджи) |

### Добавете под-хранилища към основното хранилище

Добавете всички подхранилища към `repos/` указател. Тази директория трябва да бъде **изключена от Git контрола на версиите на главното хранилище**, за да се избегне проследяване на съдържанието на подхранилище:

```plaintext
# Exclude all sub-repositories
repos/
```

#### 3. Добавете родителско хранилище към HagiCode

Ако използвате настолното приложение HagiCode, отворете приложението и добавете основното си хранилище като проект. Щракнете върху бутона „Добавяне на проект“ или „+“ на началния екран, изберете основната папка на хранилището и ще видите всички подхранилища и информация за тяхното състояние.

#### 4. Клониране на под-хранилища

Използвайте `monospecs.yaml` конфигурация за клониране на всички конфигурирани под-хранилища към `repos/` директория:

Това ще клонира всички под-хранилища въз основа на URL адресите, посочени в конфигурационния файл.

## Структура на конфигурационния файл

### Конфигурация на основно ниво

```yaml
version: "1.0"
commit_when_archive: true

repositories:
  # ... repository configurations
```

### Конфигурация на ниво хранилище

Всяко подхранилище може да има:
- Папка за промяна на OpenSpec (`openspec/`)
- Независим `.git/` указател
- Не е задължително `AGENTS.md` за конфигурация на AI Agent

## Операции за управление на хранилище

### Добавете ново хранилище

При добавяне на ново подхранилище към `monospecs.yaml` `repositories` масив:

```yaml
repositories:
  - path: "repos/new-service"
    url: "https://github.com/HagiCode-org/new-service.git"
    displayName: "New Service"
    icon: "🆕"
```

### Актуализиране на конфигурацията на хранилището

Когато URL адресите на хранилището или метаданните се променят:
1. Редактиране `monospecs.yaml` за актуализиране на съответните записи
2. Проверете дали синтаксисът на YAML е правилен
3. Ако синхронизирането се промени, актуализирайте ръчно конфигурацията на локалното хранилище

## Поддръжка за конфигурация на AI Agent

### АГЕНТИ.md

The `AGENTS.md` файл предоставя насоки, специфични за хранилището, за AI Agent:

**Техническа информация за стека**
- Инструменти за рамка и изграждане
- Кодови конвенции и практики за именуване
- Управление на версиите и конфигурация за внедряване

**Специфично за проекта поведение**
- Разширения на конфигурацията и специални изисквания
- OpenSpec интеграционни модели
- Необходимост от координация между хранилища

### Предимства на AI

Използване `AGENTS.md` позволява на AI агента да:
- Разберете връзките и зависимостите на хранилището
- Генерирайте код и преработете в множество хранилища
- Поддържайте последователност в контрола на версиите и конфигурациите

## OpenSpec интеграция

### Как Monospec работи с OpenSpec

Независимо дали управлявате проект, управляван от monospec, или обикновено хранилище, можете да използвате работния процес OpenSpec на HagiCode:

**Редовно хранилище:**
- OpenSpec в root `openspec/` указател
- Промените засягат само текущото хранилище

**Проект, управляван от Monospecs:**
- OpenSpec чете `monospecs.yaml` конфигурация
- Промените се проследяват от подхранилище
- Спецификациите се ангажират автоматично с правилните цели

### Архивно поведение

Кога `commit_when_archive: true`:
- Спецификациите се ангажират автоматично към основното хранилище
- Кодът на подхранилищата не е ангажиран (остава отделен)
- Опростява управлението на спецификациите - не са необходими ръчни ангажименти на спецификации за под-хранилища

### Ръчни ангажименти на подхранилище

Под-хранилищата поддържат независим контрол на версиите:
- Извършете действителни промени в кода в техните собствени хранилища
- Спецификациите се ангажират отделно (от основното хранилище при архивиране)

## Най-добри практики

### Конвенции за именуване на хранилище

- Използвайте kebab-case (малки букви и тирета) за пътища на хранилище
- Използвайте прости, описателни имена на дисплея на китайски
- Използвайте подходящи емотикони като икони

### Указания за показвано име

- Дръжте го кратък и описателен
- Използвайте последователно именуване в подобни хранилища
- Изберете икони, които представляват целта на хранилището

### Поддръжка на конфигурацията

**Синхронизиране на конфигурация с действително състояние**
- Проверявайте редовно `monospecs.yaml` отразява действителната структура на хранилището
- Актуализиране или премахване на записи, когато се добавят или премахват хранилища
- Тествайте промените, за да гарантирате валидността на конфигурацията

**Контрол на версията за конфигурация**
- Проследявайте промените в конфигурационния файл с git
- Документирайте причините за големи промени в конфигурацията
- Маркирайте версиите на конфигурационния файл за лесно връщане назад

## Отстраняване на неизправности

### Конфигурационният файл не е намерен

Ако `monospecs.yaml` файлът не е намерен в корена на проекта:
1. Проверете дали файлът съществува на правилното място
2. Уверете се, че сте в правилната работна директория
3. Проверете за правописни грешки в името на конфигурационния файл

### Синтактични грешки на YAML

Често срещани проблеми със синтаксиса на YAML:
- Неправилен отстъп (използвайте интервали, а не раздели)
- Липсващи или допълнителни кавички около низовете
- Невалидни типове данни (низовете се нуждаят от кавички, числата не)
- Липсващи задължителни полета (като `path` или `url`)

### Хранилището не е открито

Ако ново добавено хранилище не се показва:
1. Проверете `path` е правилно в `monospecs.yaml`
2. Уверете се, че хранилището е клонирано успешно
3. Проверете дали хранилището съществува на очакваното място

### Неуспехи при клониране

Ако клонирането на хранилището е неуспешно:
1. Проверете мрежовата връзка
2. Проверете дали URL адресът на хранилището на Git е правилен
3. Проверете за проблеми с удостоверяването
4. Прегледайте състоянието на контейнера на Docker
5. Проверете наличността на дисково пространство
