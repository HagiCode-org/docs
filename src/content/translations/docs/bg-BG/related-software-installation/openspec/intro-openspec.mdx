# Въведение в OpenSpec

OpenSpec е инструмент за разработка, управляван от спецификации, който помага на екипите да управляват процеса на разработка на софтуер чрез структурирани предложения и спецификации. Тази статия предоставя цялостно въведение в основните концепции, работни процеси и методи за използване на OpenSpec.

## Какво е OpenSpec?

OpenSpec е инструмент за команден ред, използван за управление на предложения (Proposals), спецификации (Specifications) и промени (Changes) в процеса на разработка на софтуер. Той стандартизира процеса на разработка, като гарантира, че всяка функция преминава през задълбочено проектиране и преглед от идеята до реализацията.

### Основни принципи

OpenSpec се основава на следните основни принципи:

- **Спецификация на първо място**: Дефинирайте ясни спецификации преди да пишете код
- **Структурирано сътрудничество**: Насърчавайте екипното сътрудничество чрез стандартизирани формати на документи
- **Проследимост**: Записвайте целия процес от предложение до реализация
- **Осигуряване на качеството**: Гарантирайте, че кодът съответства на проектното намерение чрез валидиране на спецификациите

### Основни функции

OpenSpec предоставя следните основни функции:

1. **Управление на предложения**: Създавайте, преглеждайте и проследявайте предложения за функции
2. **Дефиниране на спецификации**: Пишете и поддържайте документи с технически спецификации
3. **Проследяване на промени**: Записвайте цялата история на прегледани промени
4. **Валидиране на спецификации**: Гарантирайте, че промените в кода съответстват на предварително дефинираните спецификации
5. **Екипно сътрудничество**: Поддържайте процеси на разработка и преглед с участието на много хора

## Работен процес на OpenSpec

Типичният работен процес на OpenSpec включва следните етапи:

### 1. Създаване на предложение (Proposal)

Когато има нова функция или изискване за значителна промяна, първо създайте предложение:

```bash
openspec proposal create
```

Предложението включва:
- **Описание на проблема**: Проблемът или изискването, което трябва да бъде решено
- **Решение**: Преглед на предложеното решение
- **Обхват на въздействие**: Въздействието върху съществуващата система
- **Алтернативни решения**: Други разгледани решения

### 2. Писане на спецификация (Specification)

След одобрение на предложението, напишете подробна техническа спецификация:

```bash
openspec spec create
```

Спецификацията включва:
- **Технически дизайн**: Подробен план за техническа реализация
- **Дефиниране на интерфейс**: Дизайн на интерфейси като API, структури на данни и др.
- **Детайли на реализацията**: Ключова логика на реализация и алгоритми
- **План за тестване**: Как да се провери правилността на реализацията

### 3. Създаване на промяна (Change)

Когато започнете реализацията, създайте промяна за проследяване на процеса на разработка:

```bash
openspec change create
```

Промяната включва:
- **Свързана спецификация**: Спецификацията, реализирана от тази промяна
- **Задачи за реализация**: Списък с конкретни задачи за разработка
- **Проследяване на напредъка**: Състояние на завършване на задачите
- **Промени в кода**: Действителните модификации на кода

### 4. Валидиране и архивиране

След завършване на разработката, валидирайте дали промяната съответства на спецификацията, след което я архивирайте:

```bash
openspec change validate
openspec change archive
```

## Структура на директориите на OpenSpec

OpenSpec създава директория `openspec/` в основната директория на проекта със следната структура:

```
openspec/
├── proposals/           # Директория за предложения
│   ├── PROPOSAL-001.md
│   └── PROPOSAL-002.md
├── specs/              # Директория за спецификации
│   ├── SPEC-001.md
│   └── SPEC-002.md
├── changes/            # Директория за промени
│   ├── feature-name/
│   │   ├── proposal.md
│   │   ├── design.md
│   │   ├── specs.md
│   │   └── tasks.md
│   └── another-feature/
└── archived/           # Архивирани промени
    └── feature-name/
```

### Описание на директориите

- **proposals/**: Съхранява всички документи с предложения, всяко предложение в отделен Markdown файл
- **specs/**: Съхранява документи с технически спецификации, съответстващи на предложенията
- **changes/**: Съхранява текущи промени, всяка промяна в поддиректория
- **archived/**: Съхранява завършени и архивирани промени

## Основни команди

### Команди, свързани с предложения

```bash
# Създаване на нов предложение
openspec proposal create

# Списък на всички предложения
openspec proposal list

# Преглед на детайли на предложение
openspec proposal view <proposal-id>

# Одобряване на предложение
openspec proposal approve <proposal-id>
```

### Команди, свързани със спецификации

```bash
# Създаване на нова спецификация
openspec spec create

# Списък на всички спецификации
openspec spec list

# Преглед на детайли на спецификация
openspec spec view <spec-id>

# Актуализиране на спецификация
openspec spec update <spec-id>
```

### Команди, свързани с промени

```bash
# Създаване на нова промяна
openspec change create

# Списък на всички промени
openspec change list

# Преглед на детайли на промяна
openspec change view <change-name>

# Добавяне на задача
openspec change add-task <change-name>

# Валидиране на промяна
openspec change validate <change-name>

# Архивиране на промяна
openspec change archive <change-name>
```

### Общи команди

```bash
# Преглед на версията
openspec --version

# Преглед на помощ
openspec --help

# Преглед на помощ за конкретна команда
openspec <command> --help
```

## Използване на OpenSpec в проекта HagiCode

### Интеграция на проекта

В проекта HagiCode, OpenSpec е интегриран в работния процес на разработка:

1. **Проверка при инициализация**: Съветникът за стартиране проверява дали OpenSpec е инсталиран
2. **Разработка, управлявана от предложения**: Всички нови функции трябва да преминат през процеса на предложения на OpenSpec
3. **Преглед на код**: Промените трябва да съответстват на спецификациите на OpenSpec, за да бъдат обединени
4. **Синхронизация на документация**: Документацията на OpenSpec се поддържа синхронизирана с реализацията на кода

### Изисквания за версия

Проектът HagiCode изисква използване на версия **OpenSpec >=1.0.0 <2.0.0**.

Команда за инсталиране:

```bash
npm install -g @fission-ai/openspec@1
```

### Пример за работен процес

Ето пълен пример за използване на OpenSpec в проекта HagiCode:

#### Стъпка 1: Създаване на предложение

```bash
cd /path/to/hagicode-project
openspec proposal create
```

Въведете в интерактивната подкана:
- Заглавие на предложението: Добавяне на функция за удостоверяване на потребители
- Описание на проблема: Системата се нуждае от вход на потребители и управление на разрешения
- Решение: Реализиране на система за удостоверяване, базирана на JWT

#### Стъпка 2: Писане на спецификация

След одобрение на предложението, създайте подробна спецификация:

```bash
openspec spec create
```

Дефинирайте в спецификацията:
- Дизайн на процеса на удостоверяване
- Дефиниране на API интерфейс
- Структура на таблиците в базата данни
- Съображения за сигурност

#### Стъпка 3: Създаване на промяна

Започнете реализацията:

```bash
openspec change create
```

Въведете име на промяната: `add-user-authentication`

#### Стъпка 4: Добавяне на задачи

```bash
openspec change add-task add-user-authentication
```

Добавете конкретни задачи:
- Реализиране на интерфейс за регистрация на потребители
- Реализиране на интерфейс за вход
- Реализиране на JWT middleware
- Писане на unit тестове

#### Стъпка 5: Реализация и валидиране

След завършване на кода, валидирайте промяната:

```bash
openspec change validate add-user-authentication
```

#### Стъпка 6: Архивиране

След успешно валидиране, архивирайте промяната:

```bash
openspec change archive add-user-authentication
```

## Най-добри практики

### Писане на предложения

1. **Ясно изложение на проблема**: Ясно дефинирайте проблема, който трябва да бъде решен, избягвайте неясни описания
2. **Разгледайте алтернативни решения**: Изброете и сравнете различни решения
3. **Оценете въздействието**: Анализирайте въздействието и рисковете за съществуващата система
4. **Бъдете кратки**: Предложенията трябва да бъдат кратки и ясни, подробностите оставете за спецификацията

### Писане на спецификации

1. **Подробно, но не излишно**: Предоставете достатъчно детайли, но избягвайте свръхпроектиране
2. **Използвайте диаграми**: Използвайте блок-схеми, архитектурни диаграми и др. за илюстрация
3. **Дефинирайте интерфейси**: Ясно дефинирайте всички публични интерфейси и структури на данни
4. **Разгледайте гранични случаи**: Изброете изключителни ситуации и обработка на грешки

### Управление на промени

1. **Итерирайте на малки стъпки**: Разделете големите функции на множество малки промени
2. **Актуализирайте навременно**: Поддържайте синхронизирани състоянието на задачите и документацията
3. **Преглед на код**: Извършете преглед на кода след завършване на промяната
4. **Архивирайте навременно**: Архивирайте завършените промени навременно, поддържайте работната директория чиста

### Екипно сътрудничество

1. **Редовни прегледи**: Редовно преглеждайте предложения и спецификации, за да осигурите консенсус в екипа
2. **Документация на първо място**: Завършете документацията преди да започнете кодирането
3. **Поддържайте комуникация**: Записвайте дискусии и решения в предложения и спецификации
4. **Контрол на версиите**: Включете документацията на OpenSpec в контрола на версиите

## Общи сценарии за използване

### Сценарий 1: Добавяне на нова функция

1. Създайте предложение, описващо изискванията за новата функция
2. Екипът обсъжда и одобрява предложението
3. Напишете подробна техническа спецификация
4. Създайте промяна и реализирайте функцията
5. Валидирайте и архивирайте промяната

### Сценарий 2: Рефакториране на съществуващ код

1. Създайте предложение, обясняващо причините и целите за рефакториране
2. Дефинирайте плана и стъпките за рефакториране в спецификацията
3. Създайте промяна и рефакторирайте постепенно
4. Осигурете покритие с тестове, валидирайте резултатите от рефакторирането
5. Архивирайте промяната

### Сценарий 3: Поправяне на грешка

За прости грешки:
- Може да се поправят директно, не е необходим пълен процес на OpenSpec

За сложни грешки:
1. Създайте предложение за анализ на основната причина за грешката
2. Дефинирайте решението за поправка в спецификацията
3. Създайте промяна и реализирайте поправката
4. Добавете тестове за предотвратяване на регресия
5. Архивирайте промяната

### Сценарий 4: Управление на технически дълг

1. Създайте предложение, изброяващо елементите на техническия дълг
2. Дефинирайте план за подобрение в спецификацията
3. Създайте промяна за постепенно изплащане на техническия дълг
4. Проследявайте напредъка и ефекта
5. Архивирайте завършените подобрения

## Интеграция с други инструменти

### Интеграция с Git

OpenSpec се интегрира безпроблемно с Git:

```bash
# Документацията на OpenSpec трябва да бъде включена в контрола на версиите
git add openspec/
git commit -m "Add OpenSpec proposal for feature X"
```

### Интеграция с CI/CD

Валидирайте спецификациите на OpenSpec в процеса на CI/CD:

```yaml
# .github/workflows/openspec.yml
name: OpenSpec Validation
on: [pull_request]
jobs:
  validate:
    runs-on: ubuntu-latest
    steps:
      - uses: actions/checkout@v2
      - name: Install OpenSpec
        run: npm install -g @fission-ai/openspec@1
      - name: Validate changes
        run: openspec change validate
```

### Интеграция с IDE

Препоръчително е да използвате редактор, поддържащ Markdown, за редактиране на документи на OpenSpec:
- VS Code + Markdown плъгин
- JetBrains IDEs + поддръжка на Markdown
- Vim/Neovim + Markdown плъгин

## Отстраняване на проблеми

### Често срещани проблеми

#### Проблем: Командата OpenSpec не е намерена

**Решение**:
1. Потвърдете, че OpenSpec е инсталиран: `npm list -g @fission-ai/openspec`
2. Проверете дали променливата на средата PATH включва глобалната bin директория на npm
3. Преинсталирайте: `npm install -g @fission-ai/openspec@1`

#### Проблем: Не може да се създаде предложение или промяна

**Решение**:
1. Потвърдете, че текущата директория е основната директория на проекта
2. Проверете дали има директория `openspec/`, ако няма, създайте я
3. Потвърдете, че имате разрешения за запис

#### Проблем: Валидирането е неуспешно

**Решение**:
1. Проверете дали документацията в директорията за промени е пълна
2. Потвърдете, че всички необходими задачи са завършени
3. Прегледайте подробната информация за грешката: `openspec change validate <change-name> --verbose`

#### Проблем: Архивирането е неуспешно

**Решение**:
1. Уверете се, че промяната е преминала валидиране
2. Проверете дали директорията `openspec/archived/` съществува
3. Потвърдете, че няма непотвърдени промени в кода

## Разширено използване

### Персонализирани шаблони

OpenSpec поддържа персонализирани шаблони за предложения и спецификации:

```bash
# Създайте директория за шаблони в основната директория на проекта
mkdir -p .openspec/templates

# Създайте персонализиран шаблон за предложение
cat > .openspec/templates/proposal.md << 'EOF'
# Proposal: {{title}}

## Problem Statement
{{problem}}

## Proposed Solution
{{solution}}

## Custom Section
Your custom content here
EOF
```

### Конфигурационен файл

Създайте конфигурационен файл `.openspec.json` в основната директория на проекта:

```json
{
  "version": "1.0",
  "templates": {
    "proposal": ".openspec/templates/proposal.md",
    "spec": ".openspec/templates/spec.md"
  },
  "validation": {
    "requireTests": true,
    "requireDocs": true
  },
  "hooks": {
    "pre-archive": "npm test"
  }
}
```

### Куки (Hooks)

OpenSpec поддържа изпълнение на персонализирани скриптове преди и след конкретни операции:

```json
{
  "hooks": {
    "pre-create": "scripts/check-permissions.sh",
    "post-archive": "scripts/notify-team.sh"
  }
}
```
