## Изграждане на проблеми с GitHub Интеграция от нулата: Практиката за директна връзка на предния край на HagiCode

> Тази статия документира пълния процес на интегриране на GitHub Issues в платформата HagiCode. Ще проучим как да използваме архитектура „директна връзка на предния край + минимален бекенд“, за да постигнем сигурно OAuth удостоверяване и ефективна синхронизация на проблемите, като същевременно запазим лекотата на бекенда.

## Предистория: Защо да интегрираме GitHub?

Като платформа за разработка, подпомагана от AI, основната стойност на HagiCode се крие в свързването на идеи с имплементация. Но при реална употреба открихме, че след като потребителите изпълнят предложение в HagiCode, те често трябва ръчно да копират съдържанието в GitHub Issues за проследяване на проекта.

Това създава няколко очевидни болезнени точки:

1.  **Фрагментиран работен процес**: Потребителите трябва да превключват напред и назад между две системи. Изживяването не е гладко и ключова информация може лесно да се загуби по време на копиране и поставяне.
2.  **Неудобно сътрудничество**: Други членове на екипа са свикнали да проверяват задачи в GitHub и не могат директно да видят напредъка на предложението в HagiCode.
3.  **Повтаряща се ръчна работа**: Всеки път, когато дадено предложение се актуализира, някой трябва ръчно да актуализира съответния проблем в GitHub, добавяйки ненужни разходи за поддръжка.

За да разрешим този проблем, решихме да въведем функцията **GitHub Issues Integration**, свързвайки HagiCode сесии с GitHub хранилища, за да активираме „синхронизиране с едно щракване“.

## Технически избор: Frontend Direct Connection срещу Backend Proxy

Когато проектирахме интеграционния подход, имахме две възможности пред себе си: традиционния „backend прокси модел“ и по-агресивния „frontend директен модел на връзка“.

### Сравнение на решението

В традиционния **бекенд прокси модел** всяка заявка от предния край трябва първо да премине през нашия бекенд, който след това извиква GitHub API. Това централизира логиката, но също така поставя значителна тежест върху бекенда:

1.  **Раздут бекенд**: Ще трябва да напишем специална клиентска обвивка на GitHub API и също така да обработваме сложната OAuth машина.
2.  **Риск от токени**: GitHub токенът на потребителя трябва да се съхранява в базата данни на бекенда. Дори и с криптиране, това все още увеличава защитната повърхност.
3.  **Разходи за разработка**: Ще ни трябват миграции на бази данни за съхраняване на токени и допълнителна услуга за синхронизация, която да поддържаме.

**Моделът с директна връзка с интерфейса** е много по-лек. При този подход ние използваме бекенда само за най-чувствителната стъпка за „таен обмен“ (обратното извикване на OAuth). След като получим токена, ние го съхраняваме директно в браузъра `localStorage`. По-късните операции като създаване на проблеми и актуализиране на коментари се изпращат директно от интерфейса към GitHub през HTTP.

| Сравнително измерение | Backend прокси модел | Модел на директно свързване на предния край |
| :--- | :--- | :--- |
| **Сложност на бекенда** | Изисква пълна OAuth услуга и GitHub API клиент | Необходима е само крайна точка за обратно извикване на OAuth |
| **Управление на токени** | Трябва да бъдат криптирани и съхранени в базата данни с риск от изтичане | Съхранява се в браузъра и се вижда само от потребителя |
| **Цена за внедряване** | Изисква миграции на бази данни и разработване на множество услуги | Предимно фронтенд работа |
| **Потребителско изживяване** | Централизирана логика, но латентността на сървъра може да е малко по-висока | Изключително бърза реакция с директно взаимодействие с GitHub |

Тъй като искахме бърза интеграция и минимални промени в бекенда, **в крайна сметка избрахме „модела на директна връзка на предния край“**. Това е като да дадете на браузъра „временен пропуск“. След като получи пропуска, браузърът може сам да се справи с нещата в GitHub, без всеки път да иска одобрение от бекенд администратора.

## Основен дизайн: поток от данни и сигурност

След като се спряхме на архитектурата, трябваше да проектираме конкретния поток от данни. Ядрото на процеса на синхронизация е как да получите сигурно токена и да го използвате ефективно.

### Диаграма на общата архитектура

Цялата система може да се раздели на три роли: браузър (frontend), бекенд HagiCode и GitHub.

```text
+--------------+        +--------------+        +--------------+
| Frontend     |        |   Backend    |        |   GitHub     |
| React        |        |   ASP.NET    |        |   REST API   |
|              |        |              |        |              |
|  +--------+  |        |              |        |              |
|  | OAuth  |--+--------> /callback    |        |              |
|  | Flow   |  |        |              |        |              |
|  +--------+  |        |              |        |              |
|              |        |              |        |              |
|  +--------+  |        |  +--------+  |        |  +--------+  |
|  | GitHub |  +------------> Session |  +----------> Issues |  |
|  | API    |  |        |  |Metadata|  |        |  |        |  |
|  | Direct |  |        |  +--------+  |        |  +--------+  |
|  +--------+  |        |              |        |              |
+--------------+        +--------------+        +--------------+
```

**Ключовият момент е**: само една малка стъпка в OAuth (обмяна на кода за токен) трябва да премине през бекенда. След това тежката работа (създаване на проблеми) се обработва директно между интерфейса и GitHub.

### Поток от данни за синхронизиране в детайли

Когато потребителят щракне върху бутона „Синхронизиране с GitHub“ в потребителския интерфейс на HagiCode, се извършват поредица от сложни действия:

```text
User clicks "Sync to GitHub"
         │
         ▼
1. Frontend checks localStorage for the GitHub token
         │
         ▼
2. Format issue content (convert the Proposal into Markdown)
         │
         ▼
3. Frontend directly calls the GitHub API to create/update the issue
         │
         ▼
4. Call the HagiCode backend API to update Session.metadata (store issue URL and other info)
         │
         ▼
5. Backend broadcasts the SessionUpdated event via SignalR
         │
         ▼
6. Frontend receives the event and updates the UI to show the "Synced" state
```

### Дизайн на сигурността

Сигурността винаги е основен приоритет при интегриране на услуги на трети страни. Взехме следните съображения:

1.  **Защитавайте се срещу CSRF атаки**: Генерирайте произволно `state` параметър по време на OAuth пренасочване и го съхранявайте в `sessionStorage`. Строго валидирайте състоянието в обратното извикване, за да предотвратите фалшиви заявки.
2.  **Изолирано съхранение на токени**: Токенът се съхранява само в браузъра `localStorage`. Използвайки политиката за същия произход, само скриптове на HagiCode могат да го четат, избягвайки риска от изтичане на база данни от страна на сървъра, засягащо потребителите.
3.  **Граници на грешките**: Създадохме специална обработка за често срещани грешки на GitHub API (като 401 изтекъл токен, 422 неуспешно валидиране и 429 ограничаване на скоростта), така че потребителите да получават приятелска обратна връзка.

## На практика: Подробности за внедряването в кода

Теорията стига само дотук. Нека да видим как всъщност работи кодът.

### 1. Минимални промени в бекенда

Бекендът трябва да прави само две неща: да съхранява информация за синхронизиране и да обработва обратното извикване на OAuth.

### 2. Преден OAuth поток

Това е входната точка на връзката. Използваме стандартния поток на кода за оторизация.

```typescript
// src/HagiCode.Client/src/services/githubOAuth.ts

// Generate the authorization URL and redirect
export async function generateAuthUrl(): Promise<string> {
  const state = generateRandomString(); // Generate a random string for CSRF protection
  sessionStorage.setItem('hagicode_github_state', state);
  
  const params = new URLSearchParams({
    client_id: clientId,
    redirect_uri: window.location.origin + '/settings?tab=github&oauth=callback',
    scope: ['repo', 'public_repo'].join(' '),
    state: state,
  });
  
  return `https://github.com/login/oauth/authorize?${params.toString()}`;
}

// Handle the code-to-token exchange on the callback page
export async function exchangeCodeForToken(code: string, state: string): Promise<GitHubToken> {
  // 1. Validate state to prevent CSRF
  const savedState = sessionStorage.getItem('hagicode_github_state');
  if (state !== savedState) throw new Error('Invalid state parameter');

  // 2. Call the backend API to exchange the token
  // Note: this must go through the backend because ClientSecret cannot be exposed to the frontend
  const response = await fetch('/api/GitHubOAuth/callback', {
    method: 'POST',
    headers: { 'Content-Type': 'application/json' },
    body: JSON.stringify({ code, state, redirectUri: window.location.origin + '/settings?tab=github&oauth=callback' }),
  });

  if (!response.ok) throw new Error('Failed to exchange token');
  
  const token = await response.json();
  
  // 3. Save into LocalStorage
  saveToken(token);
  return token;
}
```

### 3. GitHub API Client Wrapper

След като имаме токена, имаме нужда от солиден инструмент за извикване на GitHub API.

```typescript
// src/HagiCode.Client/src/services/githubApiClient.ts

const GITHUB_API_BASE = 'https://api.github.com';

// Core request wrapper
async function githubApi<T>(endpoint: string, options: RequestInit = {}): Promise<T> {
  const token = localStorage.getItem('gh_token');
  if (!token) throw new Error('Not connected to GitHub');
  
  const response = await fetch(`${GITHUB_API_BASE}${endpoint}`, {
    ...options,
    headers: {
      ...options.headers,
      Authorization: `Bearer ${token}`,
      Accept: 'application/vnd.github.v3+json', // Specify the API version
    },
  });
  
  // Error handling logic
  if (!response.ok) {
    if (response.status === 401) throw new Error('GitHub token expired, please reconnect');
    if (response.status === 403) throw new Error('No permission to access this repository or rate limit exceeded');
    if (response.status === 422) throw new Error('Issue validation failed, the title may be duplicated');
    throw new Error(`GitHub API Error: ${response.statusText}`);
  }
  
  return response.json();
}

// Create issue
export async function createIssue(owner: string, repo: string, data: { title: string, body: string, labels: string[] }) {
  return githubApi(`/repos/${owner}/${repo}/issues`, {
    method: 'POST',
    body: JSON.stringify(data),
  });
}
```

### 4. Форматиране и синхронизиране на съдържанието

Последната стъпка е да конвертирате данните от сесията на HagiCode във формата на проблем на GitHub. Това е малко като преводаческа работа.

```typescript
// Convert a Session object into a Markdown string
function formatIssueForSession(session: Session): string {
  let content = `# ${session.title}\n\n`;
  content += `**> HagiCode Session:** #${session.code}\n`;
  content += `**> Status:** ${session.status}\n\n`;
  content += `## Description\n\n${session.description || 'No description provided.'}\n\n`;
  
  // If this is a Proposal session, add extra fields
  if (session.type === 'proposal') {
    content += `## Chief Complaint\n\n${session.chiefComplaint || ''}\n\n`;
    // Add a deep link so users can jump back from GitHub to HagiCode
    content += `---\n\n**[View in HagiCode](hagicode://sessions/${session.id})**\n`;
  }
  
  return content;
}

// Main logic when clicking the sync button
const handleSync = async (session: Session) => {
  try {
    const repoInfo = parseRepositoryFromUrl(session.repoUrl); // Parse the repository URL
    if (!repoInfo) throw new Error('Invalid repository URL');

    toast.loading('Syncing to GitHub...');
    
    // 1. Format content
    const issueBody = formatIssueForSession(session);
    
    // 2. Call API
    const issue = await githubApiClient.createIssue(repoInfo.owner, repoInfo.repo, {
      title: `[HagiCode] ${session.title}`,
      body: issueBody,
      labels: ['hagicode', 'proposal', `status:${session.status}`],
    });
    
    // 3. Update Session Metadata (save the issue link)
    await SessionsService.patchApiSessionsSessionId(session.id, {
      metadata: {
        githubIssue: {
          owner: repoInfo.owner,
          repo: repoInfo.repo,
          issueNumber: issue.number,
          issueUrl: issue.html_url,
          syncedAt: new Date().toISOString(),
        }
      }
    });

    toast.success('Synced successfully!');
  } catch (err) {
    console.error(err);
    toast.error('Sync failed, please check your token or network');
  }
};
```

## Резюме и перспектива

С този подход на „директна връзка на предния край“ постигнахме безпроблемна интеграция на GitHub Issues с възможно най-малко код на задния край.

### Ключови печалби
1.  **Висока ефективност на разработката**: Промените в бекенда са минимални, основно едно допълнително поле на базата данни и проста крайна точка за обратно извикване на OAuth. По-голямата част от логиката е завършена във фронтенда.
2.  **Силна сигурност**: Токенът не преминава през сървърната база данни, което намалява риска от изтичане.
3.  **Страхотно потребителско изживяване**: Заявките се инициират директно от интерфейса, така че скоростта на отговор е бърза и няма нужда от пренасочване от задния край.

### Неща, за които трябва да внимавате
Има няколко клопки, които трябва да имате предвид по време на реалното внедряване:
- **Настройки на приложението OAuth**: Не забравяйте да въведете правилните `Authorization callback URL` в настройките на приложението GitHub OAuth (обикновено `http://localhost:3000/settings?tab=github&oauth=callback`).
- **Ограничения на скоростта**: API на GitHub ограничава неавтентифицираните заявки доста стриктно, но с токен квотата обикновено е достатъчна (5000 заявки/час).
- **URL анализ**: Потребителите въвеждат всички видове URL адреси на хранилища, така че се уверете, че вашият регулярен израз може да съвпадне `.git` суфикси, SSH формати и подобни случаи.

### Бъдещи подобрения
Текущата функция все още е еднопосочна синхронизация (HagiCode -> GitHub). В бъдеще планираме да приложим двупосочна синхронизация чрез GitHub Webhooks. Например, ако даден проблем е затворен в GitHub, състоянието на сесията от страна на HagiCode също може да се актуализира автоматично. Това ще изисква от нас да изложим крайна точка на webhook в бекенда, което ще бъде интересна следваща стъпка.
