## Изграждане на елегантно въвеждане на нови потребители в React проекти: Практиката на driver.js на HagiCode

> Когато потребителите отворят вашия продукт за първи път, наистина ли знаят откъде да започнат? В тази статия искам да поговоря малко за това как използвахме driver.js за включване на нови потребители в проекта HagiCode. Считайте го за малък практически пример, за да започнете разговора.

## Фон

Изпадали ли сте някога в тази ситуация? Нов потребител се регистрира за вашия продукт, отваря страницата и веднага изглежда изгубен. Те сканират наоколо, не са сигурни какво да щракнат или какво да направят след това. Като разработчици често предполагаме, че потребителите просто ще „изследват сами“, защото в крайна сметка човешкото любопитство е безгранично. Но реалността е различна: повечето потребители тихо напускат в рамките на минути, ако не могат да намерят правилната входна точка, сякаш историята започва внезапно и завършва също толкова естествено.

Включването на нов потребител е важен начин за решаване на този проблем, но изграждането му не е толкова просто. Добрата система за включване трябва:

- Точно локализирайте елементите на страницата и ги маркирайте
- Поддържайте многоетапни потоци за включване
- Запомнете избора на потребителя (завършете или пропуснете)
- Избягвайте да влияете на ефективността на страницата и нормалното взаимодействие
- Поддържайте структурата на кода ясна и лесна за поддръжка

Докато създавахме HagiCode, се натъкнахме на същото предизвикателство. HagiCode е асистент за кодиране с изкуствен интелект и неговият основен работен процес е работен поток на OpenSpec, който изглежда така: „потребителят създава предложение -> изкуственият интелект генерира план -> потребителят го преглежда -> изкуственият интелект го изпълнява.“ За потребителите, които се сблъскват с тази концепция за първи път, работният процес е напълно нов, така че те се нуждаят от солидна подготовка, за да започнат бързо. Новите неща винаги отнемат малко време, за да свикнете.

## Защо избрахме driver.js

По време на фазата на техническа оценка разгледахме няколко основни библиотеки за включване. Всеки от тях имаше своите силни страни:

- **Intro.js**: Мощен, но сравнително голям и персонализирането на стила е малко сложно
- **Shepherd.js**: Добре проектиран API, но малко прекалено „тежък“ за нашия случай на употреба
- **driver.js**: Лек, стегнат, интуитивен API и работи добре в екосистемата на React

В крайна сметка избрахме driver.js. Нямаше особено драматична причина. Изборът се свежда главно до следните съображения:

1. **Лек**: Основната библиотека е малка и не увеличава значително размера на пакета
2. **Прост API**: Конфигурацията е ясна и интуитивна, така че е лесна за разбиране
3. **Гъвкав**: Поддържа персонализирано позициониране, стил и поведение на взаимодействие
4. **Динамично импортиране**: Може да се зарежда при поискване, без да се засяга производителността на първия екран

При избора на технология рядко има универсално най-добър отговор. Обикновено има само опцията, която е най-подходяща.

## Техническо изпълнение

### Основна конфигурация

driver.js има много интуитивен конфигурационен модел. Ето основната конфигурация, която използваме в проекта HagiCode:

```typescript
import { driver } from 'driver.js';
import 'driver.js/dist/driver.css';

const newConversationDriver = driver({
  allowClose: true,           // Allow users to close the guide
  animate: true,              // Enable animations
  overlayClickBehavior: 'close', // Close the guide when the overlay is clicked
  disableActiveInteraction: false, // Keep elements interactive
  showProgress: false,        // Do not show the progress bar (we manage progress ourselves)
  steps: guideSteps           // Array of guide steps
});
```

Причината за тези настройки е:

- `allowClose: true` - Уважавайте избора на потребителя и не го принуждавайте да завърши ръководството
- `disableActiveInteraction: false` - Някои стъпки изискват реални потребителски действия, като въвеждане на въвеждане, така че взаимодействието не може да бъде деактивирано
- `overlayClickBehavior: 'close'` - Дайте на потребителите бърз начин за излизане

### Управление на държавата

Продължаващото състояние на включване е критично. Не искаме да рестартираме ръководството всеки път, когато страницата се опреснява, защото това бързо става досадно. HagiCode използва `localStorage` за управление на състоянието на ръководството:

```typescript
export type GuideState = 'pending' | 'dismissed' | 'completed';

export interface UserGuideState {
  session: GuideState;
  detailGuides: Record<string, GuideState>;
}

// Read state
export const getUserGuideState = (): UserGuideState => {
  const state = localStorage.getItem('userGuideState');
  return state ? JSON.parse(state) : { session: 'pending', detailGuides: {} };
};

// Update state
export const setUserGuideState = (state: UserGuideState) => {
  localStorage.setItem('userGuideState', JSON.stringify(state));
};
```

Дефинираме три състояния:

- `pending`: Ръководството все още е в процес и потребителят не го е завършил или пропуснал
- `dismissed`: Потребителят затвори ръководството проактивно
- `completed`: Потребителят е изпълнил всички стъпки

За интегрирането на страницата с подробности за предложението ние също поддържаме по-фино проследяване на състоянието чрез `detailGuides` карта, тъй като едно предложение може да премине през няколко етапа – чернова, преглед и завършено изпълнение – и всеки етап се нуждае от различни насоки. Състоянието на нещата винаги се променя и навлизането трябва да отразява това.

### Избор на целеви елемент

driver.js използва CSS селектори за намиране на целеви елементи. HagiCode следва проста конвенция: използвайте персонализиран `data-guide` атрибут за маркиране на цели за включване:

```typescript
const steps = [
  {
    element: '[data-guide="launch"]',
    popover: {
      title: 'Start a New Conversation',
      description: 'Click here to create a new conversation session...'
    }
  }
];
```

В компонентите изглежда така:

```typescript
<button data-guide="launch" onClick={handleLaunch}>
  New Conversation
</button>
```

Ползите от този подход са:

- Избягвайте конфликти с имена на класове за бизнес стил
- Ясна семантика, така че можете незабавно да разберете, че елементът е свързан с включване
- По-лесен за управление и последователна поддръжка

### Динамично оптимизиране на импортирането

Тъй като функцията за включване е необходима само в конкретни сценарии, като например първото посещение на потребителя, ние използваме динамично импортиране, за да оптимизираме първоначалната производителност на зареждане:

```typescript
const initNewUserGuide = async () => {
  // Dynamically import driver.js
  const { driver } = await import('driver.js');
  await import('driver.js/dist/driver.css');

  // Initialize the guide
  const newConversationDriver = driver({
    // ...configuration
  });

  newConversationDriver.drive();
};
```

По този начин driver.js и неговата таблица със стилове се зареждат само когато е необходимо и не влияят на производителността на първия екран. Не много хора обичат да чакат нещо, от което дори още не се нуждаят.

### Внедряване на подробности за предложението (3 стъпки)

Когато потребителите влязат в страницата с подробности за предложението, HagiCode задейства съответното ръководство въз основа на текущото състояние на предложението:

1. **изготвяне** (етап на чернова) - Насочете потребителя да прегледа генерирания от AI план
2. **преглед** (етап на преглед) - Насочете потребителя към изпълнение на плана

Определящата характеристика на това ръководство е, че е управлявано от състоянието: динамично решава коя стъпка за включване да покаже въз основа на действителното състояние на Предложението. Нещата се променят и адаптирането трябва да се промени с тях.

### Механизъм за повторно изобразяване на елемент

В приложенията на React целевият елемент за включване може все още да не е завършил изобразяването, например защото асинхронните данни все още се зареждат. За да се справи с това, HagiCode прилага механизъм за повторен опит:

```typescript
const waitForElement = (selector: string, maxRetries = 10, interval = 100) => {
  let retries = 0;
  return new Promise<HTMLElement>((resolve, reject) => {
    const checkElement = () => {
      const element = document.querySelector(selector) as HTMLElement;
      if (element) {
        resolve(element);
      } else if (retries < maxRetries) {
        retries++;
        setTimeout(checkElement, interval);
      } else {
        reject(new Error(`Element not found: ${selector}`));
      }
    };
    checkElement();
  });
};
```

Извикайте тази функция, преди да инициализирате ръководството, за да се уверите, че целевият елемент вече съществува. Понякога чакането още малко си струва.

## Резюме на най-добрите практики

Въз основа на практическия опит на HagiCode, ето няколко ключови най-добри практики:

### 1. Включването трябва да е лесно за излизане

Не принуждавайте потребителите да завършат интегрирането. Някои потребители са изследователи по природа и предпочитат да разберат нещата сами. Осигурете ясен бутон „Пропускане“, запомнете техния избор и не ги прекъсвайте отново следващия път.

### 2. Поддържайте съдържанието за включване кратко и остро

Всяка стъпка на борда трябва да се фокусира върху една цел:

- **Заглавие**: Кратко и ясно, в идеалния случай не повече от няколко думи
- **Описание**: Преминете направо към въпроса и кажете на потребителя какво е това и защо има значение

Избягвайте дългите обяснения. Вниманието на потребителите е ограничено по време на въвеждане и колкото повече казвате, толкова по-малка е вероятността те да го прочетат.

### 3. Поддържайте селекторите стабилни

Използвайте стабилни маркери на елементи, които не се променят често. Обичаят `data-guide` атрибут е добър избор. Избягвайте зависимост от имена на класове или DOM структура, защото те лесно се променят по време на рефактори. Кодът се променя през цялото време, но някои котви трябва да останат стабилни, когато е възможно.

### 4. Тествайте вашето включване

HagiCode включва пълни тестови случаи за функцията за включване:

```typescript
describe('NewUserConversationGuide', () => {
  it('should initialize guide state correctly', () => {
    const state = getUserGuideState();
    expect(state.session).toBe('pending');
  });

  it('should update guide state correctly', () => {
    setUserGuideState({ session: 'completed', detailGuides: {} });
    const state = getUserGuideState();
    expect(state.session).toBe('completed');
  });
});
```

Тестовете помагат да се гарантира, че рефакторингът няма случайно да наруши поведението на борда. Никой не иска малка промяна на кода, за да повреди тихо работещата преди това функционалност.

### 5. Оптимизация на производителността

- Използвайте динамично импортиране, за да заредите отложено библиотеката за включване
- Избягвайте инициализиране на логиката за включване, след като потребителят вече е завършил ръководството
- Обмислете въздействието на анимациите върху производителността и ги деактивирайте на устройства от по-нисък клас, ако е необходимо

Изпълнението, както много неща в живота, заслужава малко внимателно бюджетиране.

Включването на нови потребители е важна част от подобряването на потребителското изживяване на продукта. В проекта HagiCode използвахме driver.js, за да изградим пълна система за включване, която покрива пълния работен процес от създаването на сесия до изпълнението на предложението.

Основните точки, които се надяваме да споделим чрез тази статия, са:

1. **Техническият избор трябва да отговаря на действителните нужди**: driver.js не е най-мощната опция, но е най-подходящата за нас
2. **Управлението на състоянието е критично**: Използвайте `localStorage` за да запазите състоянието на включване и да избегнете многократното прекъсване на потребителите
3. **Въвеждащият дизайн трябва да остане фокусиран**: Всяка стъпка трябва да решава един проблем и не повече
4. **Структурата на кода трябва да остане ясна**: Отделна конфигурация за включване, управление на състоянието и логика на потребителския интерфейс, за да улесните поддръжката
