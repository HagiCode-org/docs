## Добавяне на поддръжка за автоматично внедряване на GitHub страници към HagiCode

> Този проект първоначално беше с кодово име PCode и сега официално е преименуван на HagiCode. Тази статия документира как въведохме автоматизирано внедряване на статични сайтове, така че публикуването на съдържание да стане лесно като пиене на вода.

## Предистория / Въведение

По време на разработването на HagiCode се натъкнахме на много практически проблем: тъй като нашата документация и предложения продължаваха да растат, ефективното управление и представяне на това съдържание ставаше все по-спешно. Решихме да използваме GitHub Pages, за да хостваме нашия статичен сайт, но изграждането и внедряването му ръчно беше твърде тромаво. Всяка промяна изискваше локално изграждане, опаковане и след това ръчно натискане на `gh-pages` клон. Това беше не само неефективно, но и лесно да се сбърка.

За да разрешим този проблем, главно защото искахме да си спестим малко усилия, се нуждаехме от автоматизиран работен процес за внедряване. Тази статия описва подробно как добавихме базирана на GitHub Actions поддръжка за автоматично внедряване към проекта HagiCode, така че да можем да останем фокусирани върху писането на съдържание и да оставим автоматизацията да се справи с останалото.

## Анализ на целта

Преди да преминем към изпълнението, първо трябва да изясним какво всъщност трябва да постигне тази задача. В крайна сметка заточването на брадвата не забавя цепенето на дърва за огрев.

### Основни изисквания

1.  **Автоматизирано изграждане**: Автоматично задействане на работния поток за изграждане, когато кодът бъде насочен към `main` клон.
2.  **Автоматизирано внедряване**: След успешно изграждане, автоматично внедряване на генерираните статични файлове в GitHub Pages.
3.  **Съгласуваност на средата**: Уверете се, че CI средата съответства на локалната среда за изграждане, за да избегнете класическото неудобство „работи локално, но се проваля онлайн“.

### Технически избор

Тъй като HagiCode е изграден на Docusaurus, много популярен генератор на статичен сайт на React, можем да използваме GitHub Actions, за да постигнем тази цел.

## Конфигуриране на работния поток на GitHub Actions

GitHub Actions е услугата CI/CD, предоставена от GitHub. Чрез дефиниране на файлове на работен поток във формат YAML вътре в хранилището, можем да персонализираме всички видове задачи за автоматизация.

### Създаване на файла на работния процес

Трябва да създадем нов конфигурационен файл в `.github/workflows` директория в корена на проекта, например `deploy.yml`. Ако директорията все още не съществува, първо я създайте.

Основната логика на този конфигурационен файл е следната:

1.  **Условие на задействане**: Слушайте `push` събития на `main` клон.
2.  **Среда за изпълнение**: Най-новата версия на Ubuntu.
3.  **Стъпки за изграждане**:
    -   Вижте кода
    -   Инсталирайте Node.js
    -   Инсталирайте зависимости (`npm install`)
    -   Създайте статичните файлове (`npm run build`)
4.  **Стъпка на внедряване**: Използвайте официалния `action-gh-pages` действие за насочване на изхода на компилацията към `gh-pages` клон.

### Код за конфигурация на ключ

По-долу е конфигурационният шаблон, който в крайна сметка приехме:

```yaml
name: Deploy to GitHub Pages

# Trigger condition: run when code is pushed to the main branch
on:
  push:
    branches:
      - main
    # Add path filters if needed, for example only build when docs change
    # paths:
    #   - 'docs/**'
    #   - 'package.json'

# Set permissions, which is important for GitHub Pages deployment
permissions:
  contents: read
  pages: write
  id-token: write

# Concurrency control: cancel older builds for the same branch
concurrency:
  group: "pages"
  cancel-in-progress: false

jobs:
  build:
    runs-on: ubuntu-latest
    steps:
      - name: Checkout
        uses: actions/checkout@v4
        # Note: fetch-depth: 0 is required, otherwise the build version may be inaccurate
        with:
          fetch-depth: 0

      - name: Setup Node
        uses: actions/setup-node@v4
        with:
          node-version: 20 # Keep this consistent with the local development environment
          cache: 'npm'     # Enabling cache can speed up the build process

      - name: Install dependencies
        run: npm ci
        # Use npm ci instead of npm install because it is faster and stricter, which makes it better for CI

      - name: Build website
        run: npm run build
        env:
          # Configure environment variables here if your site build requires them
          # NODE_ENV: production
          # PUBLIC_URL: /your-repo-name
          
      - name: Upload artifact
        uses: actions/upload-pages-artifact@v3
        with:
          path: ./build # Default Docusaurus output directory

  deploy:
    environment:
      name: github-pages
      url: ${{ steps.deployment.outputs.page_url }}
    runs-on: ubuntu-latest
    needs: build
    steps:
      - name: Deploy to GitHub Pages
        id: deployment
        uses: actions/deploy-pages@v4
```

## Често срещани клопки по време на внедряването

На практика се натъкнахме на няколко проблема. Споделям ги тук с надеждата, че другите могат да ги избегнат или поне да подготвят решения предварително.

### 1. Проблеми с разрешенията за GitHub Token

Когато за първи път настроихме нещата, внедряването продължи да се проваля с 403 (Забранено). След много ровене открихме, че GitHub е по подразбиране `GITHUB_TOKEN` нямаше разрешение да пише в Pages.

**Решение**: В хранилището `Settings` -> `Actions` -> `General` -> `Workflow permissions`, не забравяйте да изберете **„Разрешения за четене и запис“**.

### 2. Грешен път на директорията за компилация

Docusaurus поставя построени статични файлове в `build` директория по подразбиране. Някои проекти обаче могат да бъдат конфигурирани по различен начин. Например, Create React App е по подразбиране `build`, докато Vite е по подразбиране `dist`. Ако Actions съобщи, че не може да намери изходните файлове, проверете изходния път `docusaurus.config.js`.

### 3. Проблеми с подпът

Ако вашето хранилище не е потребителска начална страница, това означава, че не е `username.github.io`, но вместо това страница на проект като напр `username.github.io/project-name`, тогава трябва да конфигурирате `baseUrl`.

в `docusaurus.config.js`:

```javascript
module.exports = {
  // ...
  url: 'https://hagicode.com', // Your Hagicode URL
  baseUrl: '/', // Deploy from the root path
  // ...
};
```
Тази подробност е лесна за пренебрегване. Ако е конфигуриран неправилно, страницата може да се зареди като празен екран, тъй като пътищата на активите не могат да бъдат разрешени.

## Проверка на резултата

След като конфигурираме всичко и натиснем кода, можем да се отправим към раздела **Действия** на хранилището на GitHub и да гледаме шоуто.

Ще видите жълт кръг, докато работният процес работи, а след като стане зелен, това означава успех. Ако стане червено, щракнете и проверете дневниците. В повечето случаи можете да проследите проблема там, обикновено печатна грешка или неправилна конфигурация на пътя.

След като изграждането успее, посетете `https://<your-username>.github.io/<repository-name>/` и ще видите току-що разгърнатия сайт.

## Резюме

С въвеждането на GitHub Actions ние успешно автоматизирахме внедряването за сайта с документация на HagiCode. Това не само спестява времето, което преди това е било изразходвано за ръчни операции, но което е по-важно, стандартизира работния процес на издаване. Сега, без значение кой член на екипа актуализира документацията, след като промяната бъде обединена в `main`, най-новото съдържание ще бъде на живо няколко минути по-късно.

**Основни предимства**:
- **По-висока ефективност**: Преминахме от „ръчно опаковане и ръчно качване“ към „код е равно на освобождаване“.
- **По-малко грешки**: Премахнахме възможността за човешка грешка по време на ръчни операции.
- **По-добро изживяване за разработчици**: Разработчиците могат да се съсредоточат повече върху качеството на съдържанието, вместо да се разсейват от досадни стъпки по внедряване.

Въпреки че конфигурирането на CI/CD е малко обезпокоително в началото, особено с всички разрешения и подробности за пътя, това е еднократна инвестиция с голяма дългосрочна възвръщаемост. Горещо препоръчвам всеки проект за статичен сайт да приеме подобен автоматизиран работен процес.
