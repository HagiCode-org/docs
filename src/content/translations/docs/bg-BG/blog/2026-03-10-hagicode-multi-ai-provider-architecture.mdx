## Практическа архитектура на мулти-AI доставчик в платформата HagiCode

> Тази статия споделя техническия подход, който използвахме в архитектурата Orleans Grain, за да интегрираме два AI инструмента, iflow и OpenCode, чрез унифициран `IAIProvider` интерфейс и сравнява подробно разликите в изпълнението между WebSocket и HTTP комуникацията.

:::бележка[времеви контекст]
Тази статия записва архитектурната практика на доставчика към момента на публикуване. За текущия активно поддържан обхват вижте [CLI инсталация на AI Agent](/bg-BG/related-software-installation/ai-agent-cli/) и на [Ръководство за инсталиране](/bg-BG/installation); `IFlowCli` сега се запазва само като историческа стойност за съвместимост и не е част от текущия активен обхват на поддръжка.
:::

## Фон

В това няма нищо особено мистериозно. Докато изграждахме HagiCode, се натъкнахме на много практичен проблем: потребителите искаха да работят с различни AI инструменти. Това не е изненадващо, тъй като всеки има свои собствени навици. Някои предпочитат Claude Code, други обичат GitHub Copilot, а някои екипи използват инструменти, които са разработили сами.

Първоначалното ни решение беше просто и директно: напишете специален код за интегриране за всеки инструмент за изкуствен интелект. Но недостатъците се проявиха бързо. Кодовата база се напълни с `if-else` клонове, всяка промяна изискваше тестване на множество места и всеки нов инструмент означаваше писане на нова купчина логика от нулата.

По-късно разбрах, че би било по-добре да създам единна `IAIProvider` интерфейс и абстрахирайте възможностите, споделяни от всички доставчици на AI. По този начин, без значение кой инструмент се използва отдолу, горните слоеве могат да го извикат по същия начин.

Наскоро проектът трябваше да интегрира два нови инструмента: iflow и OpenCode. И двата поддържат ACP протокола, но стиловете им на комуникация са различни. iflow използва WebSocket, докато OpenCode използва HTTP API. Това се превърна в полезен архитектурен тест: адаптирайте два различни транспортни режима зад един унифициран интерфейс.

## Относно HagiCode

Подходът, споделен в тази статия, идва от нашия практически опит в [HagiCode](https://hagicode.com) проект. HagiCode е платформа за разработка, подпомагана от AI, изградена върху архитектурата Orleans Grain. Той се интегрира с различни доставчици на AI чрез унифициран `IAIProvider` интерфейс, позволяващ на потребителите гъвкаво да избират AI инструментите, които предпочитат.

## Архитектурен дизайн

### Абстракция на унифициран интерфейс

Първо дефинирахме `IAIProvider` интерфейс и абстрахира възможностите, които всеки доставчик на AI трябва да внедри:

```csharp
public interface IAIProvider
{
    string Name { get; }
    bool SupportsStreaming { get; }
    ProviderCapabilities Capabilities { get; }

    Task<AIResponse> ExecuteAsync(AIRequest request, CancellationToken cancellationToken = default);
    IAsyncEnumerable<AIStreamingChunk> StreamAsync(AIRequest request, CancellationToken cancellationToken = default);
    Task<ProviderTestResult> PingAsync(CancellationToken cancellationToken = default);
    IAsyncEnumerable<AIStreamingChunk> SendMessageAsync(AIRequest request, string? embeddedCommandPrompt = null, CancellationToken cancellationToken = default);
}
```

Този интерфейс включва няколко ключови метода:
- `ExecuteAsync`: изпълнете еднократна AI заявка
- `StreamAsync`: получавате стрийминг отговори за показване в реално време
- `PingAsync`: извършете проверка на здравето, за да проверите дали доставчикът е наличен
- `SendMessageAsync`: изпратете съобщение с поддръжка за вградени команди

### IFlowCliProvider: Реализация, базирана на WebSocket

iflow използва WebSocket за ACP комуникация. Цялостната архитектура изглежда така:

```
IFlowCliProvider → ACPSessionManager → WebSocketAcpTransport → iflow CLI
                ↓
         Dynamic port allocation + process management
```

Основният поток също е доста ясен:
1. `ACPSessionManager` създава и управлява ACP сесии.
2. `WebSocketAcpTransport` обработва WebSocket комуникация.
3. Портът се разпределя динамично и с него се стартира процесът iflow `iflow --experimental-acp --port`.
4. `IAIRequestToAcpMapper` и `IAcpToAIResponseMapper` конвертиране на заявки и отговори.

Ето основния код:

```csharp
private async IAsyncEnumerable<AIStreamingChunk> StreamCoreAsync(
    AIRequest request,
    string? embeddedCommandPrompt,
    [EnumeratorCancellation] CancellationToken cancellationToken)
{
    // Resolve working directory
    var resolvedWorkingDirectory = ResolveWorkingDirectory(request);
    var effectiveRequest = ApplyEmbeddedCommandPrompt(request, embeddedCommandPrompt);

    // Create ACP session
    await using var session = await _sessionManager.CreateSessionAsync(
        Name,
        resolvedWorkingDirectory,
        cancellationToken,
        request.SessionId);

    // Send prompt
    var prompt = _requestMapper.ToPromptString(effectiveRequest);
    var promptResponse = await session.SendPromptAsync(prompt, cancellationToken);

    // Receive streaming response
    await foreach (var notification in session.ReceiveUpdatesAsync(cancellationToken))
    {
        if (_responseMapper.TryConvertToStreamingChunk(notification, out var chunk))
        {
            if (chunk.Type == StreamingChunkType.Metadata && chunk.IsComplete)
            {
                yield return chunk;
                yield break;
            }
            yield return chunk;
        }
    }
}
```

Има няколко точки на дизайна, които си струва да обърнете внимание тук:
- Използвайте `await using` за да се гарантира, че сесията е освободена правилно и да се избегнат течове на ресурси.
- Връщане на поточно предаване на отговори `IAsyncEnumerable`, който естествено поддържа асинхронни потоци.
- Използвайте `Metadata` парчета, за да се определи завършването и да се гарантира, че е получен пълният отговор.

### OpenCodeCliProvider: HTTP API-базирана реализация

OpenCode предоставя своята услуга чрез HTTP API, така че архитектурата е малко по-различна:

```plaintext
OpenCodeCliProvider → OpenCodeRuntimeManager → OpenCodeClient → OpenCode HTTP API
                      ↓
                OpenCodeProcessManager → opencode process management
```

Забележителна характеристика на OpenCode е, че той използва SQLite база данни, за да поддържа обвързвания на сесии. Това прави възможно възстановяването на сесията и възстановяването с бърз отговор:

```csharp
private async Task<OpenCodePromptExecutionResult> ExecutePromptAsync(
    AIRequest request,
    string? embeddedCommandPrompt,
    CancellationToken cancellationToken)
{
    var prompt = BuildPrompt(request, embeddedCommandPrompt);
    var resolvedWorkingDirectory = ResolveWorkingDirectory(request.WorkingDirectory);
    var client = await _runtimeManager.GetClientAsync(resolvedWorkingDirectory, cancellationToken);
    var bindingSessionId = request.SessionId;
    var boundSession = TryGetBinding(bindingSessionId, resolvedWorkingDirectory);

    // Try to use the already bound session
    if (boundSession is not null)
    {
        try
        {
            return await PromptSessionAsync(
                client,
                boundSession,
                BuildPromptRequest(request, prompt, CreatePromptMessageId()),
                request.Model ?? _settings.Model,
                cancellationToken);
        }
        catch (OpenCodeApiException ex) when (IsStaleBinding(ex))
        {
            // The session has expired, remove the binding
            RemoveBinding(bindingSessionId);
        }
    }

    // Create a new session
    var session = await client.Session.CreateAsync(new OpenCodeSessionCreateRequest
    {
        Title = BuildSessionTitle(request)
    }, cancellationToken);

    BindSession(bindingSessionId, session.Id, resolvedWorkingDirectory);
    return await PromptSessionAsync(client, session.Id, ...);
}
```

Това изпълнение има няколко интересни акцента:
- Механизъм за свързване на сесии: същият `SessionId` използва повторно една и съща OpenCode сесия, като избягва повторното създаване на сесия.
- Обработка на изтичане: когато се установи, че сесията е изтекла, обвързването се почиства автоматично.
- Устойчивост на базата данни: обвързванията се съхраняват в SQLite и остават ефективни след рестартиране.

### Сравнение на двата подхода

| Аспект | IFlowCliProvider | OpenCodeCliProvider |
|------|-----------------|---------------------|
| Комуникация | WebSocket (ACP) | HTTP API |
| Управление на процеси | ACPSessionManager | OpenCodeProcessManager |
| Разпределение на портове | Динамичен порт | Няма порт (използва HTTP) |
| Управление на сесии | ACPSession | OpenCodeSession |
| Упоритост | Кеш в паметта | SQLite база данни |
| Команда за стартиране | `iflow --experimental-acp --port` | `opencode` |
| Латентност | Долна (дълготрайна връзка) | Относително по-висок (HTTP заявки) |

Кой подход ще изберете зависи главно от вашите нужди. WebSocket е по-добър за сценарии с високи изисквания за реално време, докато HTTP API е по-прост и лесен за отстраняване на грешки.

## Практическо ръководство

### Конфигуриране на доставчици

Първо активирайте двата доставчика в конфигурационния файл:

```yaml
AI:
  Providers:
    IFlowCli:
      Type: "IFlowCli"
      Enabled: true
      ExecutablePath: "iflow"
      Model: null
      WorkingDirectory: null
    OpenCodeCli:
      Type: "OpenCodeCli"
      Enabled: true
      ExecutablePath: "opencode"
      Model: "anthropic/claude-sonnet-4"
      WorkingDirectory: null

OpenCode:
  Enabled: true
  BaseUrl: "http://localhost:38376"
  ExecutablePath: "opencode"
  StartupTimeoutSeconds: 30
  RequestTimeoutSeconds: 120
```

### Използвайте IFlowCliProvider

```csharp
// Get provider through the factory
var provider = await _providerFactory.GetProviderAsync(AIProviderType.IFlowCli);

// Execute an AI request
var request = new AIRequest
{
    Prompt = "请帮我重构这个函数",
    WorkingDirectory = "/path/to/project",
    Model = "claude-sonnet-4"
};

// Get the complete response
var response = await provider.ExecuteAsync(request, cancellationToken);
Console.WriteLine(response.Content);

// Or use streaming responses
await foreach (var chunk in provider.StreamAsync(request, cancellationToken))
{
    if (chunk.Type == StreamingChunkType.ContentDelta)
    {
        Console.Write(chunk.Content);
    }
}
```

### Използвайте OpenCodeCliProvider

```csharp
// Get provider through the factory
var provider = await _providerFactory.GetProviderAsync(AIProviderType.OpenCodeCli);

var request = new AIRequest
{
    Prompt = "请帮我分析这个错误",
    WorkingDirectory = "/path/to/project",
    Model = "anthropic/claude-sonnet-4"
};

var response = await provider.ExecuteAsync(request, cancellationToken);
Console.WriteLine(response.Content);
```

### Здравни проверки

Преди стартиране или преди употреба можете да проверите дали доставчикът е наличен:

```csharp
var iflowResult = await iflowProvider.PingAsync(cancellationToken);
if (!iflowResult.Success)
{
    Console.WriteLine($"IFlow is unavailable: {iflowResult.ErrorMessage}");
    return;
}

var openCodeResult = await openCodeProvider.PingAsync(cancellationToken);
if (!openCodeResult.Success)
{
    Console.WriteLine($"OpenCode is unavailable: {openCodeResult.ErrorMessage}");
    return;
}
```

### Поддръжка на вградени команди

И двата доставчика поддържат вградени команди, като напр `/file:xxx`:

```csharp
var request = new AIRequest
{
    Prompt = "分析这个文件的问题",
    SystemMessage = "你是一个代码分析专家"
};

await foreach (var chunk in provider.SendMessageAsync(
    request,
    embeddedCommandPrompt: "/file:src/main.cs",
    cancellationToken))
{
    Console.Write(chunk.Content);
}
```

## Бележки и най-добри практики

### Управление на ресурсите

IFlow използва дълготрайни WebSocket връзки, така че управлението на ресурсите заслужава специално внимание:
- Използвайте `await using` за да се гарантира, че сесиите се освобождават правилно.
- Анулирането задейства почистване на процеса.
- `ACPSessionManager` поддържа ограничение за максимален брой сесии.

Управлението на OpenCode процеси е относително по-просто и `OpenCodeRuntimeManager` обработва го автоматично.

### Обработка на грешки

И двата доставчика имат пълна обработка на грешки:
- IFlow грешките се разпространяват чрез актуализации на ACP сесии.
- Извеждат се грешки в OpenCode `OpenCodeApiException`.
- Препоръчително е повикващият да улови и обработи тези изключения.

### Съображения за производителност

- IFlow WebSocket комуникацията има по-ниска латентност от HTTP.
- Повторното използване на сесията на OpenCode може да намали натоварването на HTTP заявките.
- Фабричният кеш механизъм избягва многократното създаване на доставчици.
- При сценарии с висока паралелност обърнете специално внимание на ограниченията за броя на процесите и броя на връзките.

### Валидиране на конфигурацията

Пътят на изпълнимия файл се проверява при стартиране, но все още могат да възникнат проблеми по време на изпълнение. `PingAsync` е полезен инструмент за проверка дали конфигурацията е правилна:

```csharp
// Check at startup
var provider = await _providerFactory.GetProviderAsync(providerType);
var result = await provider.PingAsync(cancellationToken);
if (!result.Success)
{
    _logger.LogError("Provider {ProviderType} is unavailable: {Error}", providerType, result.ErrorMessage);
}
```

## Резюме

Тази статия споделя техническия подход, използван от платформата HagiCode при интегрирането на двата AI инструмента iflow и OpenCode. Чрез единен `IAIProvider` интерфейс, ние адаптирахме различни стилове на комуникация, WebSocket и HTTP, като същевременно запазихме модела на повикване на горния слой последователен.

Основната идея всъщност е доста проста:
1. Дефинирайте унифицирана абстракция на интерфейса.
2. Изградете адаптерни слоеве за различни реализации.
3. Управлявайте всичко равномерно чрез фабричния модел.

Това дава на системата добра разширяемост. Когато нов AI инструмент трябва да бъде интегриран по-късно, всичко, което трябва да направим, е да внедрим `IAIProvider` интерфейс, без да променя твърде много съществуващ код.

Ако работите и върху интегрирането на мулти-AI-инструменти, надявам се тази статия да е полезна.
