## .NET Core Dual-Database на практика: Нека PostgreSQL и SQLite съжителстват мирно

> Когато създаваме модерни приложения, често се сблъскваме с този компромис: средите за разработка искат нещо леко и удобно, докато производствените среди изискват висока едновременност и висока наличност. Тази статия споделя как елегантно да поддържате както PostgreSQL, така и SQLite в проект на .NET Core и да приложите най-добрата практика на „SQLite за разработка, PostgreSQL за производство“.

Стекът е модерен и доста изискан.

В началото на проекта се сблъскахме с класическа инженерна болна точка: разработчиците искаха локалната среда да работи извън кутията, без да се налага да инсталират и конфигурират тежка PostgreSQL база данни. Но в производството ни трябваше да се справяме с записвания с висока паралелност и сложни JSON заявки и точно тук лекият SQLite започва да показва своите граници.

Как можем да запазим една кодова база, като същевременно позволим на приложението да се възползва от преносимостта на SQLite като приложение за настолни компютри и също така да използваме мощната производителност на PostgreSQL като услуга от корпоративен клас? Това е основният въпрос, който тази статия изследва.

## Основна тема 1: Архитектурен дизайн и унифицирана абстракция

За да се поддържат две бази данни в .NET Core, ключовата идея е да се разчита на абстракции, а не на конкретни реализации. Трябва да отделим избора на база данни от бизнес кода и да оставим конфигурационния слой да реши.

### Дизайнерски подход

1.  **Унифициран интерфейс**: Цялата бизнес логика трябва да зависи от `DbContext` базов клас или персонализирани интерфейси, а не специфичен `PostgreSqlDbContext`.
2.  **Управляван от конфигурация**: Използвайте конфигурационни елементи в `appsettings.json` за динамично решаване кой доставчик на база данни да се зареди при стартиране на приложението.
3.  **Изолиране на функции**: Добавете логика за адаптиране за специфични за PostgreSQL възможности, като JSONB, така че приложението да може да се деградира елегантно на SQLite.

### Внедряване на код: Конфигуриране на динамичен контекст

В ASP.NET Core `Program.cs`, не трябва да кодираме твърдо `UseNpgsql` или `UseSqlite`. Вместо това трябва да четем конфигурацията и да решаваме динамично.

Първо, дефинирайте класа на конфигурацията:

```csharp
public class DatabaseSettings
{
    public const string SectionName = "Database";
    
    // Database type: PostgreSQL or SQLite
    public string DbType { get; set; } = "PostgreSQL"; 
    
    // Connection string
    public string ConnectionString { get; set; } = string.Empty;
}
```

След това регистрирайте услугата в `Program.cs` въз основа на конфигурацията:

```csharp
// Read configuration
var databaseSettings = builder.Configuration.GetSection(DatabaseSettings.SectionName).Get<DatabaseSettings>();

// Register DbContext
builder.Services.AddDbContext<ApplicationDbContext>(options =>
{
    if (databaseSettings?.DbType?.ToLower() == "sqlite")
    {
        // SQLite configuration
        options.UseSqlite(databaseSettings.ConnectionString);
        
        // Handling SQLite's concurrent write limitations
        // Note: in production, enabling WAL mode is recommended to improve concurrency performance
    }
    else
    {
        // PostgreSQL configuration (default)
        options.UseNpgsql(databaseSettings.ConnectionString, npgsqlOptions =>
        {
            // Enable JSONB support, which is very useful when handling AI conversation records
            npgsqlOptions.UseJsonNet(); 
        });
        
        // Configure connection pool retry policy
        options.EnableRetryOnFailure(3);
    }
});
```

## Основна тема 2: Справяне с различията и стратегия за миграция

Въпреки че и PostgreSQL, и SQLite поддържат стандарта SQL, те се различават значително по специфични възможности и поведение. Ако тези разлики не се третират внимателно, лесно можете да се окажете в неудобната ситуация, при която всичко работи локално, но се проваля след внедряването.

### 1. Работа с JSON типове

В HagiCode трябва да съхраняваме голямо количество подкани и AI метаданни, което обикновено включва JSON колони.
- **PostgreSQL**: Има роден `JSONB` тип с отлична производителност на заявките.
- **SQLite**: Няма естествен тип JSON (по-новите версии включват разширението JSON1, но съпоставянето на обекти все още се различава), така че данните обикновено се съхраняват като ТЕКСТ.

**Решение**:
В съпоставянето на обект на EF Core го конфигурираме като конвертируем тип.

```csharp
protected override void OnModelCreating(ModelBuilder modelBuilder)
{
    base.OnModelCreating(modelBuilder);

    // Configure entity
    modelBuilder.Entity<PromptTemplate>(entity => 
    {
        entity.Property(e => e.Metadata)
              .HasColumnType("jsonb") // PG uses jsonb
              .HasConversion(
                  v => JsonSerializer.Serialize(v, (JsonSerializerOptions)null),
                  v => JsonSerializer.Deserialize<Dictionary<string, object>>(v, (JsonSerializerOptions)null)
              ); 
    });
}
```

Когато се използва SQLite, `HasColumnType("jsonb")` може да бъде игнорирано или да задейства предупреждение. Въпреки това, защото `HasConversion` е конфигуриран, данните все още са сериализирани и десериализирани правилно като низове, съхранени в поле TEXT, осигурявайки съвместимост.

### 2. Разделяне на стратегиите за миграция

Никога не се опитвайте да накарате един набор от миграционни скриптове да работят едновременно за PostgreSQL и SQLite. Разликите в стратегиите за генериране на първичен ключ, синтаксиса на индекса и други детайли на базата данни неизбежно ще причинят неуспехи.

**Препоръчителна практика**:
Поддържайте два миграционни клона или проекта. В работния процес на разработка на HagiCode ние го обработваме по следния начин:

1.  **Етап на разработка**: Работете основно с SQLite. Използвайте `Add-Migration Init_Sqlite -OutputDir Migrations/Sqlite`.
2.  **Етап на адаптация**: След разработване на функция, превключете низа за свързване към PostgreSQL и стартирайте `Add-Migration Init_Postgres -OutputDir Migrations/Postgres`.
3.  **Скриптове за автоматизация**: Напишете прост PowerShell или Bash скрипт за автоматично прилагане на правилната миграция въз основа на текущата променлива на средата.

```bash
# Pseudocode for simple deployment logic
if [ "$DATABASE_PROVIDER" = "PostgreSQL" ]; then
    dotnet ef database update --project Migrations.Postgres
else
    dotnet ef database update --project Migrations.Sqlite
fi
```

## Основна тема 3: Уроци, извлечени от HagiCode в производството

Докато рефакторирахме **HagiCode** от модел с една база данни към поддръжка на двойна база данни, срещнахме няколко клопки и събрахме някои важни уроци, които могат да ви помогнат да избегнете същите грешки.

### 1. Разлики в паралелността и транзакциите

PostgreSQL използва архитектура сървър-клиент и поддържа записи с висока едновременност с мощни нива на изолация на транзакциите. SQLite използва заключване на файлове, така че операциите за запис заключват целия файл на базата данни, освен ако режимът WAL не е активиран.

**Препоръка**:
Когато пишете бизнес логика, която включва чести записи, като например запазване в реално време на състоянието на редактиране на потребителя, трябва да вземете под внимание заключващия модел на SQLite. При проектирането на модула за сътрудничество OpenSpec в **HagiCode** въведохме механизъм „сливане преди запис“, за да намалим честотата на директните записи в база данни, което ни позволява да поддържаме добра производителност и на двете бази данни.

### 2. Управление на жизнения цикъл на низовете за свързване

Установяването на PostgreSQL връзка е сравнително скъпо и зависи от групирането на връзки. SQLite връзките са много леки, но ако не бъдат освободени своевременно, заключванията на файлове могат да доведат до изчакване на по-късни операции.

в `Program.cs`, можем да настроим фино поведението за всяка база данни:

```csharp
if (databaseSettings?.DbType?.ToLower() == "sqlite")
{
    // SQLite: keeping connections open can improve performance, but watch out for file locks
    options.UseSqlite(connectionString, sqliteOptions =>
    {
        // Set command timeout
        sqliteOptions.CommandTimeout(30);
    });
}
else
{
    // PG: make full use of connection pooling
    options.UseNpgsql(connectionString, npgsqlOptions =>
    {
        npgsqlOptions.MaxBatchSize(100);
        npgsqlOptions.CommandTimeout(30);
    });
}
```

### 3. Значението на тестовото покритие

Много разработчици, включително някои ранни членове на нашия екип, са склонни да правят една често срещана грешка: провеждане на модулни тестове само в средата за разработка, която обикновено е SQLite.

В CI/CD тръбопровода на HagiCode наложихме стъпка GitHub Actions, за да сме сигурни, че всяка заявка за изтегляне изпълнява тестове за интеграция на PostgreSQL.

```yaml
# Example snippet from .github/workflows/test.yml
- name: Run Integration Tests (PostgreSQL)
  run: |
    docker-compose up -d db_postgres
    dotnet test --filter "Category=Integration"
```

Това ни помогна да хванем безброй грешки, свързани с разликите в синтаксиса на SQL и чувствителността към главни и малки букви.
