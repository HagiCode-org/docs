## Как да автоматизирате изданията на Steam с GitHub Actions

> Тази статия споделя цялостното решение, което внедрихме за автоматизирани издания на Steam в проекта HagiCode Desktop, обхващащо автоматизирания поток от край до край от GitHub Release до платформата Steam, включително ключови технически подробности като удостоверяване на Steam Guard и мултиплатформени качвания в Depot.

## Фон

Работният процес на пускане в Steam всъщност е доста различен от традиционното разпространение на приложения. Steam има своя собствена пълна система за доставка на актуализации. Разработчиците трябва да използват SteamCMD, за да качват компилирани артефакти в CDN мрежата на Steam, вместо просто да пускат връзка за изтегляне, както при други платформи.

Проектът HagiCode Desktop се подготвя за издание на Steam, което въведе няколко нови предизвикателства в нашия работен процес по издаването:

1. Трябваше да преобразуваме съществуващи артефакти за компилация във формат, съвместим със Steam.
2. Трябваше да ги качим в Steam чрез SteamCMD.
3. Трябваше да се справим и с удостоверяването на Steam Guard.
4. Трябваше да поддържаме мултиплатформени качвания в Depot за Linux, Windows и macOS.
5. Искахме напълно автоматизиран поток от GitHub Release към Steam.

Проектът вече е имплементирал „режим на преносима версия“, който позволява на приложението да открива полезни натоварвания на фиксирана услуга, пакетирани в `extra` указател. Нашата цел беше да интегрираме този режим на преносима версия безпроблемно с разпространението на Steam.

## Относно HagiCode

Решението, споделено в тази статия, идва от нашия практически опит в [HagiCode](https://hagicode.com) проект. HagiCode е помощник за кодиране с изкуствен интелект, който поддържа използване на настолен компютър. Тъй като работим активно за пускане в Steam, трябваше да установим надежден автоматизиран работен процес за пускане.

## Архитектурен дизайн

Ядрото на целия процес на пускане на Steam е работен поток GitHub Actions, който разделя процеса на три основни етапа:

```
┌─────────────────────────────────────────────────────────────┐
│ GitHub Actions Workflow (Steam Release)                    │
├─────────────────────────────────────────────────────────────┤
│ 1. Preparation Stage:                                      │
│    - Check out portable-version code                       │
│    - Download build artifacts from GitHub Release          │
│    - Extract and prepare the Steam content directory       │
│                                                             │
│ 2. SteamCMD Setup:                                         │
│    - Install or reuse SteamCMD                             │
│    - Authenticate with Steam Guard                         │
│                                                             │
│ 3. Release Stage:                                          │
│    - Generate Depot VDF configuration files                │
│    - Generate App Build VDF configuration files            │
│    - Invoke SteamCMD to upload to Steam                    │
└─────────────────────────────────────────────────────────────┘
```

Този дизайн предлага няколко предимства:
- Той използва повторно съществуващите артефакти на GitHub Release и избягва повторното изграждане на същите изходи.
- Той използва самостоятелно хоствани бегачи за сигурна изолация.
- Той поддържа превключване между режим на предварителен преглед и официални клонове на изданието.
- Той включва пълно обработване на грешки и регистриране, което прави повредите по-лесни за диагностициране.

## Внедряване на работния процес

### Проектиране на параметри на задействане

Нашият работен процес поддържа следните ключови параметри:

```yaml
inputs:
  release:           # Portable Version release tag
    description: 'Version tag to release (for example v1.0.0)'
    required: true
  steam_preview:     # Whether to generate a preview build
    description: 'Whether to use preview mode'
    required: false
    default: 'false'
  steam_branch:      # Steam branch to set live
    description: 'Target Steam branch'
    required: false
    default: 'preview'
  steam_description: # Build description override
    description: 'Build description'
    required: false
```

### Самостоятелна конфигурация на Runner

От съображения за сигурност ние използваме самостоятелно хостван бегач с `steam` етикет:

```yaml
runs-on:
  - self-hosted
  - Linux
  - X64
  - steam
```

Това гарантира, че изданията на Steam се изпълняват на специален runner и запазва чувствителните идентификационни данни безопасно изолирани.

### Контрол на паралелността

За да предотвратим намесата на изданията на една и съща версия една в друга, ние конфигурирахме контрол на паралелността:

```yaml
concurrency:
  group: portable-version-steam-${{ github.event.inputs.release }}
  cancel-in-progress: false
```

Забележете това `cancel-in-progress: false` е зададено тук, защото изданията на Steam могат да отнемат известно време и не искаме новозадействано изпълнение да анулира такова, което вече се качва.

## Внедряване на основния скрипт

### Подготовка на входове за издание

The `prepare-steam-release-input.mjs` скриптът е отговорен за подготовката на входните данни, необходими за пускане:

```javascript
// Download the GitHub Release build manifest and artifact inventory
const buildManifest = await downloadBuildManifest(releaseTag);
const artifactInventory = await downloadArtifactInventory(releaseTag);

// Download compressed archives for each platform
for (const platform of ['linux-x64', 'win-x64', 'osx-universal']) {
  const artifactUrl = getArtifactUrl(artifactInventory, platform);
  await downloadArtifact(artifactUrl, platform);
}

// Extract into the Steam content directory structure
await extractToSteamContent(sources, contentRoot);
```

### Удостоверяване на Steam Guard

Steam изисква акаунтите да бъдат защитени със Steam Guard, затова внедрихме алгоритъм за генериране на код, базиран на споделена тайна:

```javascript
function generateSteamGuardCode(sharedSecret, timestamp = Date.now()) {
  const secret = decodeSharedSecret(sharedSecret);
  const time = Math.floor(timestamp / 1000 / 30);

  const timeBuffer = Buffer.alloc(8);
  timeBuffer.writeBigUInt64BE(BigInt(time));

  // Use HMAC-SHA1 to generate a time-based one-time code
  const hash = crypto.createHmac('sha1', secret)
    .update(timeBuffer)
    .digest();

  // Convert it into a 5-character Steam Guard code
  const code = steamGuardCode(hash);
  return code;
}
```

Това внедряване се основава на механизма TOTP (базирана на времето еднократна парола) на Steam Guard, генериращ нов код за потвърждение на всеки 30 секунди.

### Генериране на VDF конфигурация

VDF (Valve Data Format) е конфигурационният формат, използван от Steam. Трябва да генерираме два типа VDF файлове:

**Depot VDF** се използва за конфигуриране на съдържание за всяка платформа:

```javascript
function buildDepotVdf(depotId, contentRoot) {
  return [
    '"DepotBuildConfig"',
    '{',
    `  "DepotID" "${escapeVdf(depotId)}"`,
    `  "ContentRoot" "${escapeVdf(contentRoot)}"`,
    '  "FileMapping"',
    '  {',
    '    "LocalPath" "*"',
    '    "DepotPath" "."',
    '    "recursive" "1"',
    '  }',
    '}'
  ].join('\n');
}
```

**App Build VDF** се използва за конфигуриране на цялото приложение:

```javascript
function buildAppBuildVdf(appId, depotBuilds, description, setLive) {
  const vdf = [
    '"appbuild"',
    '{',
    `  "appid" "${appId}"`,
    `  "desc" "${escapeVdf(description)}"`,
    `  "contentroot" "${escapeVdf(contentRoot)}"`,
    '  "buildoutput" "build_output"',
    '  "depots"',
    '  {'
  ];

  for (const [depotId, depotVdfPath] of Object.entries(depotBuilds)) {
    vdf.push(`    "${depotId}" "${depotVdfPath}"`);
  }

  if (setLive) {
    vdf.push(`  }`);
    vdf.push(`  "setlive" "${setLive}"`);
  }

  vdf.push('}');
  return vdf.join('\n');
}
```

### Извикване на SteamCMD

И накрая, качването се извършва чрез извикване на SteamCMD:

```javascript
await runCommand(steamcmdPath, [
  '+login', steamUsername, steamPassword, steamGuardCode,
  '+run_app_build', appBuildPath,
  '+quit'
]);
```

Това е последният скок в целия работен процес. След като успее, освобождаването е направено.

## Мултиплатформено депо за обработка

Steam използва системата Depot за управление на съдържание за различни платформи. Ние поддържаме три основни депа:

| Платформа | Идентификатор на депо | Архитектурна поддръжка |
|-----|-----------|---------|
| Linux | `linux-x64` | x64_64 |
| Windows | `win-x64` | x64_64 |
| macOS | `osx-universal` | универсален, x64_64, arm64 |

Всяко депо има собствена директория със съдържание и VDF конфигурационен файл. Това гарантира, че потребителите на различни платформи изтеглят само съдържанието, от което действително се нуждаят.

## Работен процес на освобождаване

### Стъпка 1: Подгответе изданието на GitHub

Първо създайте GitHub Release в `portable-version` хранилище, което включва:
- Компресирани архиви за всяка платформа
- Изграждане на манифест (`{tag}.build-manifest.json`)
- Инвентаризация на артефакти (`{tag}.artifact-inventory.json`)

### Стъпка 2: Задействайте работния поток за издаване на Steam

Задействайте работния процес ръчно чрез GitHub Actions и попълнете необходимите параметри:
- `release`: етикетът на версията за публикуване, като напр `v1.0.0`
- `steam_branch`: целевият клон, като напр `preview` или `public`
- `steam_preview`: дали да се използва режим на преглед

### Стъпка 3: Автоматично изпълнете потока на освобождаване

Работният процес автоматично изпълнява следните стъпки:

1. Изтеглете и извлечете GitHub Release артефакти.
2. Инсталирайте или актуализирайте SteamCMD.
3. Генерирайте Steam VDF конфигурационни файлове.
4. Удостоверете се със Steam Guard.
5. Качете съдържание в Steam CDN.
6. Задайте посочения клон на живо.

След като тази последователност завърши, целият път на освобождаване е покрит.

## Най-добри практики

### Управление на удостоверяването на Steam Guard

При първото стартиране трябва да въведете ръчно кода на Steam Guard. След това се препоръчва да конфигурирате споделената тайна, така че кодът да може да се генерира автоматично. Това избягва ръчна намеса при всяко освобождаване.

SteamCMD запазва токена за влизане и може да го използва повторно при по-късни изпълнения. Все още трябва да следите изтичането на токена, защото се изисква повторно удостоверяване след изтичане на токена.

### Структура на директория със съдържание

Уверете се, че структурата на директорията със съдържание на Steam е правилна:

```
steam-content/
├── linux-x64/     # Linux platform content
├── win-x64/       # Windows platform content
└── osx-universal/ # macOS universal binary content
```

Всяка директория трябва да съдържа пълните файлове на приложението за съответната платформа.

### Използване на режим на визуализация

Режимът за предварителен преглед не активира нито един клон, така че е подходящ за тестване и валидиране:

```yaml
if [ "$STEAM_PREVIEW_INPUT" = 'true' ]; then
  cmd+=(--preview)
fi
```

Това ви позволява първо да качвате в Steam за проверка, след което да превключите към официалния клон, след като всичко се провери.

### Обработка на грешки и регистриране

Скриптовете включват пълно обработване на грешки и регистриране:
- Уверете се, че GitHub Release съществува.
- Проверете необходимите файлове с метаданни.
- Уверете се, че съдържанието на платформата присъства.
- Генерирайте обобщен отчет за действията на GitHub.

Тази информация е много ценна както за отстраняване на грешки, така и за одит.

### Управление на артефакти

Работният процес генерира два вида артефакти:
- `portable-steam-release-preparation-{tag}`: метаданни за подготовка на изданието
- `portable-steam-build-metadata-{tag}`: Метаданни за изграждане на Steam

Тези артефакти могат да се използват за по-късен одит и отстраняване на грешки. Периодът на задържане от 30 дни е практически стандартен.

## Използване в реалния свят

В проекта HagiCode този работен поток за автоматизирано издаване вече е работил успешно в множество версии. Целият път от GitHub Release до платформата Steam е напълно автоматизиран и не изисква ръчна намеса.

Това значително подобри както ефективността на освобождаване, така и надеждността. В миналото ръчното публикуване на една версия отнемаше повече от 30 минути. Сега целият процес завършва само за няколко минути.

По-важното е, че автоматизираният работен процес намалява вероятността от човешка грешка. Всяко издание следва един и същ стандартизиран процес, което прави резултатите по-предвидими.
