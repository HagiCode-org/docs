## VSCode и кодов сървър: Избор на решение за редактиране на код, базирано на браузър

> Когато създават възможности за редактиране на код, базирани на браузър, разработчиците са изправени пред ключов избор: използвайте официалния VSCode `code serve-web` функция или приемете управляваната от общността `code-server` решение? Това решение засяга не само техническата архитектура, но и съответствието с лиценза и гъвкавостта на внедряване.

## Фон

Техническият подбор е много подобен на избора на път в живота. След като изберете един, обикновено трябва да продължите да го разхождате и смяната по-късно може да стане много скъпа.

В ерата на програмирането, подпомагано от AI, редактирането на код, базирано на браузър, става все по-важно. Потребителите очакват, че след като AI асистент приключи с анализа на кода, те могат незабавно да отворят редактор в същата сесия на браузъра и да направят промени, без да превключват приложения. Този вид безпроблемно изживяване просто трябва да е там, когато имате нужда от него.

Въпреки това, когато прилагат тази функция, разработчиците са изправени пред критичен технически избор: трябва ли да използват официалния VSCode `code serve-web` функция, или управлявано от общността `code-server` решение?

Всяка опция има своите силни страни и компромиси и лошият избор може да създаде много проблеми по-късно. Лицензирането е един пример: ако едва след пускането на пазара откриете, че вашият продукт не е съвместим с лиценза, вече е твърде късно. Внедряването е друго: дадено решение може да работи перфектно в разработката, след което да се натъкне на всякакви проблеми, след като бъде преместено в контейнери. Това са точно видовете капани, които екипите искат да избегнат.

## Разлики в лицензирането (най-важно)

Това е най-фундаменталната разлика между двете решения и първият фактор, който взехме предвид по време на оценката. Когато правите технически избор, важно е предварително да разберете правните рискове.

**код-сървър**
- Лиценз на MIT, напълно отворен код
- Поддържа се от Coder.com с активна общност
- Безплатен за търговска употреба, модифициране и разпространение
- Няма ограничения за сценарии на използване

**VSCode `code serve-web`**
- Част от продукта Microsoft VSCode
- Използва лиценз на Microsoft (лицензът на VS Code включва ограничения за търговска употреба)
- Основно предназначен за индивидуални разработчици
- Корпоративното внедряване може да изисква допълнителен преглед на търговското лицензиране

От гледна точка на лицензирането кодовият сървър е по-приятелски настроен към комерсиалните проекти. Това е нещо, което трябва да обмислите по време на продуктовото планиране, тъй като мигрирането по-късно може да стане много скъпо.

## Разлики в внедряването

След като лицензирането е уредено, следващият проблем е внедряването. Това пряко се отразява на вашите оперативни разходи и архитектурен дизайн.

**код-сървър**
- Самостоятелно приложение Node.js, което може да се внедри независимо
- Поддържа множество източници на време за изпълнение:
  - Директно указване на изпълнимия път
  - Търся го през системата PATH
  - Автоматично откриване на среда на NVM Node.js 22.x
- Няма нужда да инсталирате настолното приложение VSCode на сървъра
- По-лесно за разполагане в контейнери

**VSCode `code serve-web`**
- Трябва да зависи от локално инсталиран VSCode CLI
- Изисква наличен `code` команда на хост машината
- Системата филтрира VS Code Remote CLI обвивки
- Основно предназначен за сценарии за местно развитие

code-server е по-подходящ за сценарии за разгръщане на сървър и контейнер. Ако вашият продукт трябва да работи в Docker или вашите потребители нямат инсталиран VSCode, кодовият сървър обикновено е правилният избор.

## Разлики в параметрите на функциите

Двете решения също се различават по няколко параметъра на характеристиките. Разликите не са огромни, но могат да създадат търкания при интеграция при използване в реалния свят.

| Характеристика | код-сървър | code serve-web |
|------|-------------|----------------|
| Обществен основен път | `/` (може да се конфигурира) | `/vscode-server` (фиксиран) |
| Удостоверяване | `--auth` параметър с множество режими | `--connection-token` / `--without-connection-token` |
| Директория с данни | `{DataDir}/code-server` | `{DataDir}/vscode-serve-web` |
| Телеметрия | Деактивирано по подразбиране с `--disable-telemetry` | Зависи от настройките на VSCode |
| Актуализиране на проверки | Може да се деактивира с `--disable-update-check` | Зависи от настройките на VSCode |

Тези разлики изискват специално внимание по време на интеграцията. Например различните URL пътища означават, че вашият код на интерфейса се нуждае от специална обработка.

## Разлики при откриване на наличност

При прилагане на превключване на редактора логиката за откриване на наличност също се различава.

**код-сървър**
- Винаги се връща като видима реализация
- Все още се показва дори когато не е наличен, с `install-required` състояние
- Поддържа автоматично откриване на NVM среда Node.js 22.x

**code serve-web**
- Вижда се само когато местен `code` CLI е открит
- Ако не е налице, интерфейсът автоматично скрива тази опция
- Зависи от локалното състояние на инсталация на VSCode

Тази разлика пряко влияе върху потребителското изживяване. кодовият сървър е по-прозрачен: потребителите могат да видят опцията и да разберат, че инсталацията все още е необходима. code serve-web е по-скрит: потребителите може дори да не разберат, че опцията съществува. Кой подход е по-добър зависи от позиционирането на продукта.

## Архитектура с двойно внедряване на HagiCode

След задълбочен анализ проектът HagiCode възприе архитектура с двойно внедряване, която поддържа и двете решения на архитектурно ниво.

### По подразбиране кодов сървър

```csharp
// The default active implementation is code-server
// If an explicit activeImplementation is saved, try that implementation first
// If the requested implementation is unavailable, the resolver tries the other one
// If a fallback occurs, return fallbackReason
```

По подразбиране използваме кодов сървър главно поради гъвкавостта на лицензирането и внедряването. Въпреки това, за потребители, които вече имат локална VSCode среда, code serve-web също е солидна опция.

### Селектор на внедряване

`CodeServerImplementationResolver` отговаря за обединяването на:
- Избор на внедряване по време на загряване при стартиране
- Избор на внедряване при четене на статус
- Избор на изпълнение при отваряне на проекти
- Избор на внедряване при отваряне на трезори

Този дизайн позволява на системата да реагира гъвкаво на различни сценарии и потребителите могат да изберат изпълнението, което най-добре отговаря на тяхната среда.

### Правила за адаптиране на предния край

```typescript
// When localCodeAvailable=false, do not show code serve-web
// When localCodeAvailable=true, show the code serve-web configuration
```

Предният интерфейс автоматично показва наличните опции въз основа на средата, така че потребителите да не се объркват от функции, които не могат да използват.

## Практическо ръководство

След цялата тази теория, на какво трябва да обърнете внимание по време на действителното внедряване? В крайна сметка изпълнението е това, което има значение.

### Препоръка за внедряване на Docker

За контейнеризирано внедряване, кодовият сървър е по-добрият избор:

```dockerfile
# Use the official code-server image directly
FROM codercom/code-server:latest

# Or install through npm
RUN npm install -g code-server
```

Това решава проблема на един слой, без да се изисква допълнителна инсталация на VSCode.

### Примери за конфигурация

**конфигурация на кодов сървър**
```json
{
  "vscodeServer": {
    "enabled": true,
    "activeImplementation": "code-server",
    "codeServer": {
      "host": "0.0.0.0",
      "port": 8080,
      "executablePath": "",
      "authMode": "none"
    }
  }
}
```

**конфигуриране на сървъра на кода**
```json
{
  "vscodeServer": {
    "enabled": true,
    "activeImplementation": "serve-web",
    "serveWeb": {
      "host": "0.0.0.0",
      "port": 8080,
      "executablePath": "/usr/local/bin/code"
    }
  }
}
```

Конфигурацията може да бъде малко досадна за първи път, но след като е на мястото си, нещата стават много по-лесни за поддръжка.

### Разлики в изграждането на URL

**код-сървър**
```
http://localhost:8080/?folder=/path/to/project&vscode-lang=zh-CN
```

**code serve-web**
```
http://localhost:8080/vscode-server/?folder=/path/to/project&tkn=xxx&vscode-lang=zh-CN
```

Обърнете внимание на разликите в пътищата и параметрите. Трябва да ги обработвате отделно по време на интеграцията.

### Превключване на реализации

Системата поддържа превключване по време на изпълнение и автоматично спира предишното изпълнение при превключване:

```csharp
// VsCodeServerManager automatically handles mutual exclusion
// When switching activeImplementation, the old implementation will not keep running in the background
```

Този дизайн позволява на потребителите да изпробват различни реализации по всяко време и да намерят опцията, която работи най-добре за тях.

### Мониторинг на състоянието

```typescript
const { settings, runtime } = await getVsCodeServerSettings();

// runtime.activeImplementation: "code-server" | "serve-web"
// runtime.fallbackReason: reason for switching
// runtime.status: "running" | "starting" | "stopped" | "unhealthy"
```

Когато състоянието е видимо, потребителите могат бързо да определят дали проблемът идва от страната на сървъра или от тяхната собствена работа.
