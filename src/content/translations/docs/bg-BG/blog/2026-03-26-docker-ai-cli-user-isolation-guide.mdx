## Изпълнение на AI CLI инструменти в Docker контейнери: Практическо ръководство за изолиране на потребители и постоянни томове

> Интегрирането на инструменти за кодиране на AI като Claude Code, Codex и OpenCode в контейнеризирани среди звучи просто, но навсякъде има скрити сложности. Тази статия се гмурка дълбоко в това как проектът HagiCode решава основните предизвикателства при внедряването на Docker, включително потребителски разрешения, постоянство на конфигурацията и управление на версиите, така че можете да избегнете често срещаните клопки.

## Фон

Когато решихме да стартираме CLI инструменти за кодиране на изкуствен интелект в контейнерите на Docker, най-интуитивната мисъл вероятно беше: „Не са ли контейнерите просто root? Защо не инсталирате всичко директно и го наречете готово?“ В действителност тази на пръв поглед проста идея крие няколко основни проблема, които трябва да бъдат решени.

Първо, **ограниченията за сигурност са първото препятствие**. Вземете Claude CLI като пример: той изрично забранява стартирането като root потребител. Това е задължителна проверка на сигурността и ако бъде открит root, той отказва да стартира. Може би си мислите, не мога ли просто да превключа потребители с `USER` директива? Не е толкова просто. Все още има проблем със съпоставянето между не-root потребителя в контейнера и потребителските разрешения на хост машината.

Второ, **упоритостта на държавата е вторият капан**. Claude Code изисква влизане, Codex има своя собствена конфигурация, а OpenCode също има кеш директория. Ако трябва да преконфигурирате всичко всеки път, когато контейнерът се рестартира, цялата идея за "автоматизация" губи смисъла си. Имаме нужда от тези конфигурации, за да продължат да съществуват след жизнения цикъл на контейнера.

Третият проблем е **последователността на разрешенията**. Могат ли процесите в контейнера да имат достъп до конфигурационни файлове, създадени от хост потребителя? Несъответствията на UID/GID често причиняват грешки при разрешаване на файлове и това е изключително често срещано при реални внедрявания.

Тези проблеми може да изглеждат независими, но на практика са тясно свързани. По време на разработването на HagiCode ние постепенно разработихме практично решение. След това ще споделя техническите подробности и поуките, извлечени от тези капани.

## Защо просто не можем да използваме root?

Тук има често срещано недоразумение: Docker контейнерите се изпълняват като root по подразбиране, така че защо просто не инсталирате инструментите като root? Ако мислите по този начин, Claude CLI бързо ще ви научи на друго.

```bash
# Run Claude CLI directly as root? No.
docker run --rm -it --user root myimage claude
# Output: Error: This command cannot be run as root user
```

Това е твърдо ограничение за сигурност в Claude CLI. Причината е проста: тези CLI инструменти четат и записват чувствителна потребителска конфигурация, включително API токени, локални кешове и дори скриптове, написани от потребителя. Изпълнението им с root привилегии носи твърде голям риск.

Така възниква въпросът: как можем да удовлетворим изискванията за сигурност на CLI, като същевременно поддържаме гъвкаво управление на контейнери? Трябва да променим начина, по който мислим за това: **вместо да превключвате потребители по време на изпълнение, създайте специален потребител по време на етапа на изграждане на изображението**.

## Създаване на специален потребител: повече от просто промяна на име

Може би си мислите, че добавянето на единичен `USER` ред към Dockerfile е достатъчен. Това наистина е най-простият подход, но не е достатъчно стабилен.

### Статично създаване срещу динамично картографиране

Подходът на HagiCode е да създаде a `hagicode` потребител с UID 1000, който обикновено съвпада с потребителя по подразбиране на повечето хост машини:

```dockerfile
RUN groupadd -o -g 1000 hagicode && \
    useradd -o -u 1000 -g 1000 -s /bin/bash -m hagicode && \
    mkdir -p /home/hagicode/.claude && \
    chown -R hagicode:hagicode /home/hagicode
```

Но това решава само вградения потребител в изображението. Ами ако потребителят на хоста е UID 1001? Все още трябва да поддържате динамично картографиране, когато контейнерът стартира.

docker-entrypoint.sh съдържа ключовата логика:

```bash
if [ -n "$PUID" ] && [ -n "$PGID" ]; then
    if ! id hagicode >/dev/null 2>&1; then
        groupadd -g "$PGID" hagicode
        useradd -u "$PUID" -g "$PGID" -s /bin/bash -m hagicode
    fi
fi
```

Предимството на този дизайн е ясно: **използвайте UID по подразбиране 1000 по време на изграждане на изображението, след което коригирайте динамично по време на изпълнение чрез `PUID` и `PGID` променливи на средата**. Без значение какъв UID има потребителят на хоста, собствеността върху конфигурационните файлове остава правилна.

## Философията на дизайна на постоянните обеми

Всеки AI CLI инструмент има своя собствена предпочитана конфигурационна директория, така че те трябва да бъдат картографирани един по един:

| CLI инструмент | Път в контейнера | Именуван том |
|---------|-----------|--------|
| Клод | `/home/hagicode/.claude` | `claude-data` |
| Кодекс | `/home/hagicode/.codex` | `codex-data` |
| OpenCode | `/home/hagicode/.config/opencode` | `opencode-config-data` |

Защо да използвате наименувани томове вместо свързващи монтирания? Три причини:

1. **По-опростено управление**: Именуваните томове се управляват автоматично от Docker, така че не е необходимо да създавате хост директории ръчно.
2. **Изолация на разрешенията**: Първоначалното съдържание на томовете се създава от потребителя вътре в контейнера, като се избягват конфликти на разрешения с хоста.
3. **Независима миграция**: Томовете могат да съществуват независимо от контейнерите, така че данните не се губят, когато изображенията се надграждат.

docker-compose-builder-web автоматично генерира съответната конфигурация на тома:

```yaml
volumes:
  claude-data:
  codex-data:
  opencode-config-data:

services:
  hagicode:
    volumes:
      - claude-data:/home/hagicode/.claude
      - codex-data:/home/hagicode/.codex
      - opencode-config-data:/home/hagicode/.config/opencode
    user: "${PUID:-1000}:${PGID:-1000}"
```

Обърнете внимание на `user` поле тук: `PUID` и `PGID` се инжектират чрез променливи на средата, за да се гарантира, че процесите вътре в контейнера се изпълняват с идентичност, която съответства на потребителя на хоста. Тази подробност има значение, защото проблемите с разрешенията са болезнени за отстраняване на грешки, след като се появят.

## Управление на версиите: вградени версии със замени по време на изпълнение

Фиксирането на версии на изображения на Docker е от съществено значение за възпроизводимостта. Но в реалната разработка често се налага да тестваме по-нова версия или спешно да коригираме грешка. Ако трябваше да изграждаме отново изображението всеки път, работният процес би бил твърде неефективен.

Стратегията на HagiCode е **фиксирани версии по подразбиране, със замени по време на изпълнение като механизъм за разширение**. Това е прагматичен инженерен компромис между стабилност и гъвкавост.

Dockerfile.template фиксира версии тук:

```dockerfile
USER hagicode
WORKDIR /home/hagicode

# Configure the global npm install path
RUN mkdir -p /home/hagicode/.npm-global && \
    npm config set prefix '/home/hagicode/.npm-global'

# Install CLI tools using pinned versions
RUN npm install -g @anthropic-ai/claude-code@2.1.71 && \
    npm install -g @openai/codex@0.112.0 && \
    npm install -g opencode-ai@1.2.25 && \
    npm cache clean --force
```

docker-entrypoint.sh поддържа замени по време на изпълнение:

```bash
install_cli_override_if_needed() {
    local package_name="$2"
    local override_version="$5"

    if [ -n "$override_version" ]; then
        gosu hagicode npm install -g "${package_name}@${override_version}"
    fi
}

# Example usage
install_cli_override_if_needed "" "@anthropic-ai/claude-code" "" "" "${CLAUDE_CODE_CLI_VERSION}"
```

Това ви позволява да тествате нова версия чрез променлива на средата, без да създавате повторно изображението:

```bash
docker run -e CLAUDE_CODE_CLI_VERSION=2.2.0 myimage
```

Този дизайн е практичен, защото никой не иска да възстановява изображение всеки път, когато тества нова функция.

## Автоматично инжектиране на конфигурация

В допълнение към ръчното конфигуриране на CLI инструменти, някои сценарии изискват автоматично инжектиране на конфигурация. Най-типичният пример е API токен.

```bash
if [ -n "$ANTHROPIC_AUTH_TOKEN" ]; then
    mkdir -p /home/hagicode/.claude
    cat > /home/hagicode/.claude/settings.json <<EOF
{
  "env": {
    "ANTHROPIC_AUTH_TOKEN": "${ANTHROPIC_AUTH_TOKEN}"
  }
}
EOF
    chown -R hagicode:hagicode /home/hagicode/.claude
fi
```

Две неща имат значение тук: **предавайте чувствителна информация през променливи на средата, вместо да я кодирате твърдо в изображението** и се уверете, че собствеността върху конфигурационните файлове е зададена правилно, в противен случай CLI инструментите няма да могат да ги прочетат.

## Най-добри практики и списък с клопки

### Проблеми с несъответствие на разрешения

Това е най-лесният капан за попадане. Потребителят на хоста има UID 1001, докато контейнерът използва 1000, така че файловете, създадени от едната страна, не могат да бъдат достъпни от другата.

```bash
# Correct approach: make the container match the host user
docker run \
    -e PUID=$(id -u) \
    -e PGID=$(id -g) \
    myimage
```

Този проблем е много често срещан и може да бъде разочароващ първия път, когато се сблъскате с него.

### Конфигурацията изчезва след рестартиране на контейнера

Ако установите, че влизате отново след всяко рестартиране, проверете дали не сте забравили да монтирате постоянен том:

```yaml
volumes:
  - claude-data:/home/hagicode/.claude
```

Нищо не е по-разочароващо от внимателното настройване на конфигурация само за да видите как тя изчезва.

### Правилният начин за надграждане на версии

Не бягайте `npm install -g` директно в работещ контейнер. Правилните подходи са:

1. Задайте променлива на средата, за да задействате инсталацията за отмяна.
2. Или възстановете изображението.

```bash
# Option 1: runtime override
docker run -e CLAUDE_CODE_CLI_VERSION=2.2.0 myimage

# Option 2: rebuild the image
docker build -t myimage:v2 .
```

Има повече от един път до Рим, но някои пътища са по-гладки от други.

### Контролен списък за втвърдяване на сигурността

- Прекарайте API токени през променливи на средата, вместо да ги записвате в изображението.
- Задайте разрешения за конфигурационен файл на `600`.
- Винаги стартирайте приложението като не-root потребител.
- Актуализирайте редовно CLI версиите, за да коригирате уязвимостите в сигурността.

Сигурността винаги е важна, но истинското предизвикателство е последователното й прилагане на практика.

## Заключение

Изпълнението на AI CLI инструменти в Docker контейнери включва три основни предизвикателства: **потребителски разрешения, постоянство на конфигурацията и управление на версиите**. Чрез комбиниране на специализирани потребители, изолиране на именувани томове и базирани на променливи на средата замени, проектът HagiCode изгради архитектура за внедряване, която е едновременно сигурна и гъвкава.

Основни точки на дизайна:

- **Изолация на потребителя**: Създайте специален потребител по време на етапа на изграждане на изображението, с поддръжка по време на изпълнение за динамика `PUID`/`PGID` картографиране.
- **Стратегия за постоянство**: Всеки CLI инструмент получава свой собствен именуван том, така че рестартирането не засяга конфигурацията.
- **Гъвкавост на версията**: Фиксираните настройки по подразбиране гарантират възпроизводимост, докато отмените по време на изпълнение осигуряват място за тестване.
- **Автоматизирана конфигурация**: Чувствителната конфигурация може да бъде инжектирана автоматично чрез променливи на средата.

Това решение работи стабилно в проекта HagiCode от известно време и се надявам, че предлага полезни отправни точки за разработчици с подобни нужди.
