## От Docusaurus 3.x до Astro 5.x: ретроспекция на миграцията на сайта HagiCode

> Тази статия разглежда нашата пълна миграция на официалния уебсайт на HagiCode от Docusaurus 3.x към Astro 5.x. Ще се потопим дълбоко в това как Islands Architecture на Astro ни помогна да разрешим проблемите с производителността, като същевременно запазихме нашите съществуващи компоненти на React, осигурявайки подобрения както в скоростта на изграждане, така и в производителността на зареждане.

## Фон

През януари 2026 г. извършихме „трансплантация на сърце“ на официалния сайт на HagiCode, като мигрирахме изцяло неговата основна рамка от Docusaurus 3.x към Astro 5.x. Това не беше импулсивно пренаписване, а внимателно обмислено техническо решение.

Преди миграцията нашият сайт беше функционално завършен, но беше започнал да показва някои класически „луксозни проблеми“: раздути артефакти на компилация, прекомерни полезни натоварвания на JavaScript и по-малко от идеалната скорост на зареждане на страниците със сложна документация. Като проект за асистент за кодиране с изкуствен интелект, HagiCode се нуждае от честа документация и актуализации на функциите, така че ефективността на изграждане пряко влияе върху скоростта на издаване. В същото време искахме сайтът да бъде по-удобен за търсачката (SEO), така че повече разработчици да могат да открият проекта.

За да разрешим тези болезнени точки, взехме смело решение: да изградим отново цялата система на Astro. Въздействието на това решение може да бъде дори по-голямо, отколкото очаквате. Скоро ще навляза в подробности.

## Относно HagiCode

Подходът за мигриране на сайт, споделен в тази статия, идва от нашия практически опит в [HagiCode](https://github.com/HagiCode-org/site) проект.

HagiCode е помощник за кодиране с изкуствен интелект, фокусиран върху подобряване на ефективността на разработката. Грижим се не само за повторението на основните функции, но и за опита на разработчиците. Този рефакторинг на сайта също имаше за цел да предостави на потребителите възможно най-бързото изживяване, когато разглеждат нашите документи и официалния уебсайт.

## Защо да напуснете екосистемата на Mature Docusaurus?

В рамките на екосистемата на React Docusaurus отдавна е „стандартният отговор“ за сайтовете за документация. Работи веднага, предлага богата екосистема на плъгини и има активна общност. Но тъй като HagiCode придоби повече функции, ние също усетихме неговите ограничения:

1.  **Тесни места в производителността**: Docusaurus е основно React SPA (приложение от една страница). Дори ако пишете само статични страници, клиентът все още трябва да зареди средата за изпълнение на React и да хидратира страницата, което е ненужно тежко за прости страници с документи.
2.  **Голям размер на актива**: Дори когато една страница съдържа много малко съдържание, пакетният размер на JS остава относително фиксиран. Това не е идеално за мобилни потребители или лоши мрежови условия.
3.  **Ограничена гъвкавост**: Въпреки че е разширяем, искахме повече контрол на ниско ниво върху конвейера за изграждане.

Astro пристигна точно в точното време, за да реши тези проблеми. Той въведе нова „архитектура на островите“: по подразбиране Astro генерира статичен HTML с нулев JavaScript и само компоненти, които изискват интерактивност, се „активират“ и зареждат JS. Това означава, че по-голямата част от нашия сайт става чист HTML и се зарежда изключително бързо.

## Основна миграционна стратегия: плавен архитектурен преход

Миграцията не беше просто копиране и поставяне. Изискваше промяна в мисленето. Преминахме от модела „all React“ на Docusaurus към модела „Core + Islands“ на Astro.

### 1. Възстановяване на системата за конфигурация

Първо трябваше да се преместим от `docusaurus.config.ts` към `astro.config.mjs`. Това не беше просто преименуване на файл, а пренаписване на логиката за маршрутизиране и изграждане.

В Docusaurus всичко е плъгин. В Astro всичко е интеграция. Трябваше да предефинираме основния път на сайта, да изградим изходен режим (статичен срещу SSR) и стратегия за оптимизиране на активи.

**Преди миграцията:**
```typescript
// docusaurus.config.ts
export default {
  title: 'HagiCode',
  url: 'https://hagicode.com',
  baseUrl: '/',
  // ... more configuration
};
```

**След миграция:**
```javascript
// astro.config.mjs
import { defineConfig } from 'astro/config';
import react from '@astrojs/react';

export default defineConfig({
  integrations: [react()],
  site: 'https://hagicode.com',
  base: '/',
  // Optimization settings for static assets
  build: {
    inlineStylesheets: 'auto',
  },
});
```

### 2. Какво да запазите и какво да преработите в компонентите на React

Това беше най-болезнената част от миграцията. Нашият съществуващ сайт имаше много компоненти на React, като раздели, осветяване на код, бутони за обратна връзка и др. Изхвърлянето им би било разточително, но запазването на всичко би направило натоварването на JavaScript твърде тежко.

HagiCode възприе стратегия за **прогресивна хидратация**:
*   **Чисто статични компоненти**: За презентационно съдържание като заглавки, долни колонтитули и документация с обикновен текст, ние ги пренаписахме като Astro компоненти (`.astro` файлове) и ги рендира директно в HTML по време на изграждане.
*   **Интерактивни острови**: За компоненти, които трябва да останат интерактивни, като превключватели на теми, превключване на раздели и бутони за копиране на кодови блокове, запазихме изпълнението на React и добавихме `client:load` или `client:visible` директиви.

Например нашият често използван компонент Tabs в документацията:

```jsx
// src/components/Tabs.jsx
import { useState } from 'react';
import './Tabs.css'; // Import styles

export default function Tabs({ items }) {
  const [activeIndex, setActiveIndex] = useState(0);
  // ... state logic
  return (
    <div className="tabs-wrapper">
      {/* Rendering logic */}
    </div>
  );
}
```

Когато се използва в Markdown, ние изрично казваме на Astro: „Този компонент се нуждае от JS.“

```markdown
// src/content/docs/example.mdx
import Tabs from '../../components/Tabs.jsx';

<!-- Load JS only when the component enters the viewport -->
<Tabs client:visible items={...} />
```

По този начин интерактивните компоненти извън прозореца за изглед не се конкурират за честотна лента, което значително подобрява скоростта на зареждане на първия екран.

### 3. Адаптиране на системата за стилизиране: от CSS модули до CSS с обхват

Docusaurus поддържа CSS модули по подразбиране, докато Astro насърчава Scoped CSS чрез `<style>` етикет. Основната идея зад двете е изолирането на стила, но синтаксисът е различен.

По време на миграцията на HagiCode ние преобразувахме най-сложните CSS модули в стилове с обхват на Astro. Това всъщност се оказа нещо добро, защото в `.astro` файлове, стиловете и шаблоните живеят в един и същи файл, което прави поддръжката по-интуитивна.

**Преди рефакторинг:**
```css
/* Tabs.module.css */
.wrapper { background: var(--ifm-background-color); }
```

**След рефакторинг (Astro Scoped):**
```html
<!-- Tabs.astro -->
<div class="tabs-wrapper">
  <slot />
</div>

<style>
  .tabs-wrapper {
    /* Use CSS variables directly to adapt to the theme */
    background: var(--bg-color);
    padding: 1rem;
  }
</style>
```

В същото време обединихме глобалната система от CSS променливи и използвахме възможностите на Astro, съобразени с околната среда, за да гарантираме плавното превключване на тъмен режим между страниците.

## Клопки, които срещаме на практика и как ги разрешаваме

По време на действителната миграция на HagiCode се натъкнахме на доста проблеми. Ето няколко от най-типичните.

### 1. Болкови точки на променливи на пътя и околната среда

HagiCode поддържа внедряване на подпът, като например внедряване в поддиректория GitHub Pages. В Docusaurus, `baseUrl` се обработва автоматично. В Astro обаче трябва да сме по-внимателни, когато обработваме връзки към изображения и API заявки.

Въведохме механизъм за променлива на средата, за да управляваме това последователно:

```javascript
// Handle paths in the build script
const getBasePath = () => import.meta.env.VITE_SITE_BASE || '/';
```

Уверете се, че не кодирате твърдо пътища, започващи с `/` във вашия код. При разработка срещу производство или след конфигуриране на a `base` път, това може да причини 404 за активи.

### 2. Съвместимост на CommonJS Script

Старият ни сайт имаше някои Node.js скриптове, използвани за задачи като автоматично извличане на данни от Metrics и актуализиране на картата на сайта, и те бяха написани на CommonJS (`require`). Astro и модерните инструменти за изграждане са обхванали напълно ES модулите (`import`/`export`).

Ако и вие имате подобни скриптове, не забравяйте да ги преработите всички в ES модули. Това е посоката, в която се движи екосистемата и колкото по-рано направите промяната, толкова по-малко проблеми ще имате по-късно.

```javascript
// Old way
const fs = require('fs');

// New way
import fs from 'fs';
```

### 3. Не забравяйте SEO и пренасочванията

Търсачките вече са индексирали старите страници на Docusaurus на HagiCode. Ако превключите директно към Astro и URL структурата се промени, може да се окажете с голям брой 404 и сериозен спад в класирането при търсене.

Конфигурирахме правила за пренасочване в Astro:

```javascript
// astro.config.mjs
export default defineConfig({
  redirects: {
    '/docs/old-path': '/docs/new-path',
    // Map old links to new links in bulk
  }
});
```

Или можете да се справите с това на слоя за конфигурация на сървъра. Уверете се, че старите връзки могат да бъдат 301 пренасочени към новите адреси, защото това е критично за SEO.

## Резюме

За HagiCode мигрирането от Docusaurus към Astro не беше просто надграждане на рамка. Това също беше практическо прилагане на философията на „производителността на първо място“.

**Какво спечелихме:**
*   **Изключителни резултати на Lighthouse**: След миграцията резултатът от ефективността на сайта HagiCode лесно се доближи до перфектния резултат.
*   **По-бърза скорост на изграждане**: Възможностите за постепенно изграждане на Astro намаляват наполовина времето за издаване на актуализации на документацията.
*   **Запазена гъвкавост**: С Islands Architecture не жертвахме никакви интерактивни функции и все пак можехме да използваме React, където е необходимо.

Ако също така поддържате ориентиран към документация сайт и се борите с размера на пакета или скоростта на зареждане, Astro си струва да опитате. Въпреки че процесът на миграция изисква известна операция, като преименуване на PCode на HagiCode и преместване на компоненти един по един, копринено гладкото потребителско изживяване, което получавате в замяна, го прави абсолютно полезно.

Системата за изграждане, споделена в тази статия, е точният подход, който разработихме чрез реални проби и грешки, докато изграждахме HagiCode. Ако намирате този подход за ценен, това говори нещо за нашата инженерна сила, а самият HagiCode вероятно също си заслужава да се разгледа по-отблизо.
