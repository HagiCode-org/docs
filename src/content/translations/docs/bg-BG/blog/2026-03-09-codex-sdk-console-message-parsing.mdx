## Пълно ръководство за разбор на съобщения на конзолата на Codex SDK

> Тази статия обяснява механизма за поток на събития на Codex SDK, анализирането на типове съобщения и най-добрите практики в реални проекти, като помага на разработчиците бързо да овладеят основните умения зад услугите за изпълнение на AI.

## Фон

Когато изграждаме услуга за изпълнение на AI, базирана на Codex SDK, неизбежно се сблъскваме с практически въпрос: как трябва да обработваме поточно предаваните съобщения за събития, върнати от Codex? Тези съобщения съдържат важна информация като състояние на изпълнение, изходно съдържание и подробности за грешката, така че заслужават внимателно боравене.

Като част от проекта HagiCode се нуждаехме от надежден изпълнител за сценарии за асистент за кодиране с изкуствен интелект. Точно затова решихме да проучим в дълбочина механизма за поток от събития на Codex SDK. В края на краищата, само като разберем как работят основните съобщения, можем да изградим наистина корпоративна платформа за изпълнение на AI.

Codex SDK е SDK за помощ при програмиране, издаден от OpenAI. Той връща резултатите от изпълнението чрез поток от събития. За разлика от традиционния модел заявка-отговор, Codex използва поточно предавани събития, за да можем:

- Получете напредъка на изпълнението в реално време
- Отстранявайте грешките своевременно
- Получете подробна статистика за използването на токени
- Подкрепете дългосрочни сложни задачи

Разбирането на тези типове събития и правилното им анализиране е от съществено значение за внедряването на напълно способен AI изпълнител. В крайна сметка никой не иска да работи с черна кутия.

По време на разработката трябваше да изградим надеждна услуга за изпълнение на AI, която да обработва заявки за изпълнение на потребителски код, което е пряката причина да въведем Codex SDK.

Като асистент за кодиране с изкуствен интелект, HagiCode трябва да се справи с различни сложни сценарии за изпълнение на код: получаване на напредъка на изпълнението в реално време, своевременно обработване на грешки и събиране на подробна статистика за използването на токени. Чрез задълбочено разбиране на механизма за поток на събития на Codex SDK, можем да изградим изпълнител, който отговаря на изискванията на производствената среда. ## Механизъм за поток на събития

### Основни понятия

Codex SDK използва `thread.runStreamed()` метод за връщане на итератор на асинхронно събитие:

```typescript
import { Codex } from '@openai/codex-sdk';

const client = new Codex({
  apiKey: process.env.CODEX_API_KEY,
  baseUrl: process.env.CODEX_BASE_URL,
});

const thread = client.startThread({
  workingDirectory: '/path/to/project',
  skipGitRepoCheck: false,
});

const { events } = await thread.runStreamed('your prompt here', {
  outputSchema: {
    type: 'object',
    properties: {
      output: { type: 'string' },
      status: { type: 'string', enum: ['ok', 'action_required'] },
    },
    required: ['output', 'status'],
  },
});

for await (const event of events) {
  // Handle each event
}
```

### Подробно обяснение на типовете събития

| Тип събитие | Описание | Ключови данни |
|---------|------|----------|
| `thread.started` | Нишката стартира успешно | `thread_id` |
| `item.updated` | Съдържанието на съобщението е актуализирано | `item.text` |
| `item.completed` | Съобщението е завършено | `item.text` |
| `turn.completed` | Изпълнението е завършено | `usage` (използване на токен) |
| `turn.failed` | Неуспешно изпълнение | `error.message` |
| `error` | Събитие за грешка | `message` |

В реални проекти изпълнителният компонент на HagiCode е изграден върху тези типове събития. Трябва да се отнасяме внимателно към всеки вид събитие, за да осигурим гладко потребителско изживяване. Добрите системи се изграждат, като се вземат на сериозно детайлите.

## Внедряване на анализ на съобщения

### Извличане на съдържанието на съобщението

Съдържанието на съобщението се извлича чрез манипулатор на събития:

```typescript
private handleThreadEvent(event: ThreadEvent, onMessage: (content: string) => void): void {
  // Only handle message update and completion events
  if (event.type !== 'item.updated' && event.type !== 'item.completed') {
    return;
  }

  // Only handle agent message content
  if (event.item.type !== 'agent_message') {
    return;
  }

  // Extract text content
  onMessage(event.item.text);
}
```

Ключови точки:
- Само дръжка `item.updated` и `item.completed` събития
- Обработвайте само съдържание от тип `agent_message`
- Съдържанието на съобщението е в `event.item.text` поле

### Разбор на структуриран изход

Codex поддържа JSON структуриран изход. Можете да посочите формата за връщане чрез `outputSchema` параметър:

```typescript
const DEFAULT_OUTPUT_SCHEMA = {
  type: 'object',
  properties: {
    output: { type: 'string' },
    status: { type: 'string', enum: ['ok', 'action_required'] },
  },
  required: ['output', 'status'],
  additionalProperties: false,
} as const;
```

Функцията за анализиране се опитва да анализира JSON и ако това не успее, се връща към необработения текст.

```typescript
function toStructuredOutput(raw: string): StructuredOutput {
  try {
    const parsed = JSON.parse(raw) as Partial<StructuredOutput>;
    if (typeof parsed.output === 'string') {
      return {
        output: parsed.output,
        status: parsed.status === 'action_required' ? 'action_required' : 'ok',
      };
    }
  } catch {
    // JSON parsing failed, fall back to the raw text
  }

  return {
    output: raw,
    status: 'ok',
  };
}
```

### Пълен поток за обработка на събития

```typescript
private async runWithStreaming(
  thread: Thread,
  input: CodexStageExecutionInput
): Promise<{ output: string; usage: Usage | null }> {
  const abortController = new AbortController();
  const timeoutHandle = setTimeout(() => {
    abortController.abort();
  }, Math.max(1000, input.timeoutMs));

  let latestMessage = '';
  let usage: Usage | null = null;
  let emittedLength = 0;

  try {
    const { events } = await thread.runStreamed(input.prompt, {
      outputSchema: DEFAULT_OUTPUT_SCHEMA,
      signal: abortController.signal,
    });

    for await (const event of events) {
      // Handle message content
      this.handleThreadEvent(event, (nextContent) => {
        const delta = nextContent.slice(emittedLength);
        if (delta.length > 0) {
          emittedLength = nextContent.length;
          input.callbacks?.onChunk?.(delta);  // Streaming callback
        }
        latestMessage = nextContent;
      });

      // Process different data based on the event type
      if (event.type === 'thread.started') {
        this.threadId = event.thread_id;
      } else if (event.type === 'turn.completed') {
        usage = event.usage;
      } else if (event.type === 'turn.failed') {
        throw new CodexExecutorError('gateway_unavailable', event.error.message, true);
      } else if (event.type === 'error') {
        throw new CodexExecutorError('gateway_unavailable', event.message, true);
      }
    }
  } catch (error) {
    if (abortController.signal.aborted) {
      throw new CodexExecutorError(
        'upstream_timeout',
        `Codex stage timed out after ${input.timeoutMs}ms`,
        true
      );
    }
    throw error;
  } finally {
    clearTimeout(timeoutHandle);
  }

  const structured = toStructuredOutput(latestMessage);
  return { output: structured.output, usage };
}
```

## Стратегия за обработка на грешки

### Съпоставяне на код на грешка

Картирайте конкретни модели на грешки към конкретни кодове за грешки, така че горните слоеве да могат да се справят с тях по-лесно:

```typescript
function mapError(error: unknown): CodexExecutorError {
  if (error instanceof CodexExecutorError) {
    return error;
  }

  const message = error instanceof Error ? error.message : String(error);
  const normalized = message.toLowerCase();

  // Authentication errors - not retryable
  if (normalized.includes('401') ||
      normalized.includes('403') ||
      normalized.includes('api key') ||
      normalized.includes('auth')) {
    return new CodexExecutorError('auth_invalid', message, false);
  }

  // Rate limit errors - retryable
  if (normalized.includes('429') || normalized.includes('rate limit')) {
    return new CodexExecutorError('rate_limited', message, true);
  }

  // Timeout errors - retryable
  if (normalized.includes('timeout') || normalized.includes('aborted')) {
    return new CodexExecutorError('upstream_timeout', message, true);
  }

  // Default error
  return new CodexExecutorError('gateway_unavailable', message, true);
}
```

### Дефиниции на типове грешки

```typescript
export type CodexErrorCode =
  | 'auth_invalid'      // Authentication failure
  | 'upstream_timeout'  // Upstream timeout
  | 'rate_limited'      // Rate limited
  | 'gateway_unavailable'; // Gateway unavailable

export class CodexExecutorError extends Error {
  readonly code: CodexErrorCode;
  readonly retryable: boolean;

  constructor(code: CodexErrorCode, message: string, retryable: boolean) {
    super(message);
    this.name = 'CodexExecutorError';
    this.code = code;
    this.retryable = retryable;
  }
}
```

## Конфигурация на работна директория и среда

### Валидиране на работната директория

Codex SDK изисква работната директория да бъде валидно Git хранилище.

```typescript
export function validateWorkingDirectory(
  workingDirectory: string,
  skipGitRepoCheck: boolean
): void {
  const resolvedWorkingDirectory = path.resolve(workingDirectory);

  if (!existsSync(resolvedWorkingDirectory)) {
    throw new CodexExecutorError(
      'gateway_unavailable',
      'Working directory does not exist.',
      false
    );
  }

  if (!statSync(resolvedWorkingDirectory).isDirectory()) {
    throw new CodexExecutorError(
      'gateway_unavailable',
      'Working directory is not a directory.',
      false
    );
  }

  if (skipGitRepoCheck) {
    return;
  }

  const gitDir = path.join(resolvedWorkingDirectory, '.git');
  if (!existsSync(gitDir)) {
    throw new CodexExecutorError(
      'gateway_unavailable',
      'Working directory is not a git repository.',
      false
    );
  }
}
```

### Зареждане на променливи на средата

Codex SDK трябва да зареди променливи на средата от обвивката за вход, така че AI Agent да има достъп до системните команди:

```typescript
function parseEnvironmentOutput(output: Buffer): Record<string, string> {
  const parsed: Record<string, string> = {};

  for (const entry of output.toString('utf8').split('\0')) {
    if (!entry) continue;

    const separatorIndex = entry.indexOf('=');
    if (separatorIndex <= 0) continue;

    const key = entry.slice(0, separatorIndex);
    const value = entry.slice(separatorIndex + 1);
    if (key.length > 0) {
      parsed[key] = value;
    }
  }

  return parsed;
}

function tryLoadEnvironmentFromShell(shellPath: string): Record<string, string> | null {
  const result = spawnSync(shellPath, ['-ilc', 'env -0'], {
    env: process.env,
    stdio: ['ignore', 'pipe', 'pipe'],
    timeout: 5000,
  });

  if (result.error || result.status !== 0) {
    return null;
  }

  return parseEnvironmentOutput(result.stdout);
}

export function createExecutorEnvironment(
  envOverrides: Record<string, string> = {}
): Record<string, string> {
  // Load environment variables from the login shell
  const consoleEnv = loadConsoleEnvironmentFromShell();

  return {
    ...process.env,
    ...consoleEnv,
    ...envOverrides,
  };
}
```

## Пълен пример за използване

### Основна употреба

В проекта HagiCode използваме следния подход за инициализиране на клиента на Codex и изпълнение на задачи:

```typescript
import { Codex } from '@openai/codex-sdk';

async function executeWithCodex(prompt: string, workingDir: string) {
  const client = new Codex({
    apiKey: process.env.CODEX_API_KEY,
    env: { PATH: process.env.PATH },
  });

  const thread = client.startThread({
    workingDirectory: workingDir,
  });

  const { events } = await thread.runStreamed(prompt);

  let result = '';
  for await (const event of events) {
    if (event.type === 'item.updated' && event.item.type === 'agent_message') {
      result = event.item.text;
    }
    if (event.type === 'turn.completed') {
      console.log('Token usage:', event.usage);
    }
  }

  // Try to parse JSON output
  try {
    const parsed = JSON.parse(result);
    return parsed.output;
  } catch {
    return result;
  }
}
```

### Пълно внедряване с повторни опити

```typescript
export class CodexSdkExecutor {
  private readonly config: CodexRuntimeConfig;
  private readonly client: Codex;
  private threadId: string | null = null;

  async executeStage(input: CodexStageExecutionInput): Promise<CodexStageExecutionResult> {
    const maxAttempts = Math.max(1, this.config.retryCount + 1);
    let attempt = 0;
    let lastError: CodexExecutorError | null = null;

    while (attempt < maxAttempts) {
      attempt += 1;

      try {
        const thread = this.getThread(input.workingDirectory);
        const { output, usage } = await this.runWithStreaming(thread, input);

        return {
          output,
          usage,
          threadId: this.threadId!,
          attempts: attempt,
          latencyMs: Date.now() - startedAt,
        };
      } catch (error) {
        const mappedError = mapError(error);
        lastError = mappedError;

        // Non-retryable error or max retry attempts reached
        if (!mappedError.retryable || attempt >= maxAttempts) {
          throw mappedError;
        }

        // Wait before retrying
        await new Promise(resolve => setTimeout(resolve, 1000 * attempt));
      }
    }

    throw lastError!;
  }
}
```

## Най-добри практики

### 1. Изисквания към работната директория

- Уверете се, че работната директория е валидно Git хранилище
- Използвайте `PROJECT_ROOT` променлива на средата, за да я посочите изрично
- По време на отстраняване на грешки можете да зададете `CODEX_SKIP_GIT_REPO_CHECK=true` за да пропуснете проверката

### 2. Конфигурация на променливата на средата

- Предавайте само необходимите променливи на средата през механизъм за бели списъци
- Използвайте обвивката за вход, за да заредите пълната среда
- Избягвайте предаването на чувствителна информация

### 3. Стратегия за изчакване и повторен опит

- Задайте разумни изчаквания въз основа на сложността на задачата
- Приложете експоненциално забавяне за повторни грешки
- Запишете броя на повторните опити и причините

### 4. Обработка на грешки

- Правете разлика между повторни и неповторни грешки
- Предоставете ясни съобщения за грешки и предложения
- Използвайте унифицирани кодове за грешки, така че горните слоеве да могат да ги обработват последователно

### 5. Поточен изход

- Внедрете обратни извиквания на постепенен изход, за да подобрите потребителското изживяване
- Правилно обработвайте постепенните актуализации на съобщенията
- Запишете използването на токени за анализ на разходите

В действителната производствена среда на проекта HagiCode вече проверихме ефективността на най-добрите практики по-горе. Този подход ни помогна да изградим стабилна и надеждна услуга за изпълнение на AI. В крайна сметка практическото валидиране е по-важно от теорията.
