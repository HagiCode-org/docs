## Автоматизиране на синхронизирането на изображения от Docker Hub към Azure ACR

> Тази статия обяснява как да използвате GitHub Actions и инструмента за синхронизиране на изображения за автоматизиране на синхронизирането на изображения от Docker Hub към Azure Container Registry, като се справяте с бавния достъп до Docker Hub в Китай и някои региони на Azure, като същевременно подобрявате наличността на изображения и ефективността на внедряване в среди на Azure.

## Предистория / Въведение

Проектът HagiCode използва Docker изображения като основни компоненти за изпълнение, като основните изображения се хостват в Docker Hub. Тъй като проектът се разрасна и нуждите от внедряване на Azure се увеличиха, се натъкнахме на следните болни точки:

- Бавни скорости на изтегляне на изображения, като достъпът до Docker Hub е ограничен в Китай и някои региони на Azure
- Един източник на изображение създава риск от повреда в една точка
- Azure Container Registry осигурява по-добра мрежова производителност и интеграция, когато работи в Azure

За да разрешим тези проблеми, се нуждаехме от автоматичен механизъм за синхронизиране на изображения, за да синхронизираме редовно изображения от Docker Hub към Azure ACR, като гарантираме, че потребителите могат да получат по-бързи скорости на изтегляне на изображения и по-висока наличност в среди на Azure.

## Сравнение на техническите решения

Когато дефинирахме решението, сравнихме множество технически опции:

### 1. изображение-синхронизатор (окончателен избор)
- Постепенна синхронизация: синхронизира само променените слоеве на изображението, което значително намалява мрежовия трансфер
- Възобновяване на поддръжката: синхронизирането може да продължи след прекъсване на мрежата
- Контрол на паралелността: конфигурируемите работни нишки подобряват ефективността на синхронизиране на големи изображения
- Надеждна обработка на грешки: вграден механизъм за повторен опит за неуспехи (3 опита по подразбиране)
- Леко внедряване: единичен двоичен файл без зависимости
- Поддръжка на множество регистри: съвместим с Docker Hub, Azure ACR, Harbor и др

### 2. Docker CLI
- Без постепенно синхронизиране: пълното съдържание на изображението трябва да се изтегля всеки път
- По-ниска ефективност: голям обем на мрежов трансфер и по-дълго време за изпълнение
- Лесен за използване: базиран на познати `docker pull` и `docker push` команди

### 3. Azure CLI
- По-висока сложност: CLI удостоверяването на Azure трябва да бъде конфигурирано
- Ограничена функционалност: `az acr import` е сравнително тесен по обхват
- Вградена интеграция: интегрира се добре с услугите на Azure

- Стартирайте синхронизирането по време на ненатоварените часове

## Основно решение

Ние използваме автоматизирано решение, базирано на GitHub Actions + image-syncer, за да синхронизираме изображения от Docker Hub към Azure ACR.

## Стъпки на внедряване

### 1. Подготовка
- Създайте или потвърдете регистър на контейнери на Azure в портала на Azure
- Създайте идентификационни данни за достъп до ACR (потребителско име и парола)
- Потвърдете разрешенията за достъп до хранилището на изображения на Docker Hub

### 2. Конфигурирайте GitHub Secrets
Добавете следните тайни в настройките на хранилището на GitHub:
- AZURE_ACR_USERNAME: Azure ACR потребителско име
- AZURE_ACR_PASSWORD: Azure ACR парола

### 3. Създайте работния поток GitHub Actions
Конфигурирайте работния процес в `.github/workflows/sync-docker-acr.yml`:
- Планирано задействане: всеки ден в 00:00 UTC
- Ръчен спусък: поддържа `workflow_dispatch`
- Допълнителен тригер: работи при натискане към `publish` клон за бърза синхронизация

### 4. Поток на изпълнение на работния процес

| стъпка | участник | Действие | Описание |
| --- | --- | --- | --- |
| 1 | GitHub действия | Задействайте работния процес | Задейства се от график, ръчно изпълнение или натискане на `publish` клон |
| 2 | Действия на GitHub → image-syncer | Изтеглете и изпълнете инструмента за синхронизиране | Влиза в действителната фаза на синхронизация |
| 3 | image-syncer → Docker Hub | Извличане на манифести на изображения и списък с етикети | Чете метаданни от изходното хранилище |
| 4 | изображение-синхронизатор → Azure ACR | Извличане на съществуващи данни за изображения от целевото хранилище | Определя текущото състояние на целевата страна |
| 5 | изображение-синхронизатор | Сравнете разликите в източника и целта | Идентифицира слоевете на изображението, които се нуждаят от синхронизиране |
| 6 | image-syncer → Docker Hub | Издърпайте променените слоеве на изображението | Прехвърля само съдържанието, което се нуждае от актуализиране |
| 7 | изображение-синхронизатор → Azure ACR | Избутване на променени слоеве на изображението | Завършва постепенното синхронизиране |
| 8 | image-syncer → Действия на GitHub | Връщане на статистика за синхронизиране | Включва резултати, разлики и информация за грешки |
| 9 | GitHub действия | Записвайте регистрационни файлове и качвайте артефакти | Поддържа по-късен одит и отстраняване на неизправности |

## Внедряване на работния процес на GitHub Actions

По-долу е действителната използвана конфигурация на работния процес (`.github/workflows/sync-docker-acr.yml`):

```yaml
name: Sync Docker Image to Azure ACR

on:
  schedule:
    - cron: "0 0 * * *" # Every day at 00:00 UTC
  workflow_dispatch: # Manual trigger
  push:
    branches: [publish]

permissions:
  contents: read

jobs:
  sync:
    runs-on: ubuntu-latest

    steps:
      - name: Checkout code
        uses: actions/checkout@v4

      - name: Download image-syncer
        run: |
          # Download the image-syncer binary
          wget https://github.com/AliyunContainerService/image-syncer/releases/download/v1.5.5/image-syncer-v1.5.5-linux-amd64.tar.gz
          tar -zxvf image-syncer-v1.5.5-linux-amd64.tar.gz
          chmod +x image-syncer

      - name: Create auth config
        run: |
          # Generate the authentication config file (YAML format)
          cat > auth.yaml <<EOF
          hagicode.azurecr.io:
            username: "${{ secrets.AZURE_ACR_USERNAME }}"
            password: "${{ secrets.AZURE_ACR_PASSWORD }}"
          EOF

      - name: Create images config
        run: |
          # Generate the image sync config file (YAML format)
          cat > images.yaml <<EOF
          docker.io/newbe36524/hagicode: hagicode.azurecr.io/hagicode
          EOF

      - name: Run image-syncer
        run: |
          # Run synchronization (using the newer --auth and --images parameters)
          ./image-syncer --auth=./auth.yaml --images=./images.yaml --proc=10 --retries=3

      - name: Upload logs
        if: always()
        uses: actions/upload-artifact@v4
        with:
          name: sync-logs
          path: image-syncer-*.log
          retention-days: 7
```

## Подробности за конфигурацията

### 1. Условия за задействане
- Планиран тригер: `cron: "0 0 * * *"` работи всеки ден в 00:00 UTC
- Ръчен тригер: `workflow_dispatch` позволява на потребителите да го стартират ръчно от потребителския интерфейс на GitHub
- Натиснете спусъка: `push: branches: [publish]` задейства при натискания към клона за освобождаване за бърза синхронизация

### 2. Конфигурация за удостоверяване (`auth.yaml`)
```yaml
hagicode.azurecr.io:
  username: "${{ secrets.AZURE_ACR_USERNAME }}"
  password: "${{ secrets.AZURE_ACR_PASSWORD }}"
```

### 3. Конфигурация за синхронизиране на изображения
```yaml
docker.io/newbe36524/hagicode: hagicode.azurecr.io/hagicode
```
Тази конфигурация означава, че всички тагове от `docker.io/newbe36524/hagicode` ще се синхронизира с `hagicode.azurecr.io/hagicode`.

### 4. параметри на синхронизатор на изображения
- `--auth=./auth.yaml`: път до конфигурационния файл за удостоверяване
- `--images=./images.yaml`: път до конфигурационния файл за синхронизиране на изображения
- `--proc=10`: използвайте `10` едновременни работници
- `--retries=3`: неуспешни повторни опити `3` пъти

## Бележки и най-добри практики

### 1. Препоръки за сигурност
- Редувайте редовно паролите за Azure ACR (препоръчва се на всеки 90 дни)
- Използвайте специален акаунт за ACR услуга с разрешения само за натискане
- Наблюдавайте регистрационните файлове за достъп до ACR, за да откриете необичаен достъп навреме
- Не извеждайте идентификационни данни в регистрационни файлове
- Не ангажирайте идентификационни данни към хранилището на кодове

### 2. Оптимизиране на производителността
- Коригирайте `--proc` параметър: настройка на паралелността въз основа на честотната лента на мрежата (препоръчително `5-20`)
- Наблюдавайте продължителността на синхронизирането: ако синхронизирането отнема твърде много време, помислете за намаляване на едновременността
- Почиствайте редовно регистрационните файлове: задайте разумно `retention-days` стойност (`7` в момента)

### 3. Отстраняване на неизправности

#### Проблем 1: Грешка при удостоверяване
```
Error: failed to authenticate to hagicode.azurecr.io
```
Решение:
1. Проверете дали GitHub Secrets са конфигурирани правилно
2. Проверете дали паролата за Azure ACR е изтекла
3. Потвърдете, че разрешенията за акаунта на услугата ACR са правилни

#### Проблем 2: Време за изчакване на мрежата
```
Error: timeout waiting for response
```
Решение:
1. Проверете мрежовата свързаност
2. Намалете броя на едновременните работници ( `--proc` параметър)
3. Изчакайте мрежата да се възстанови и повторете работния процес

#### Проблем 3: Непълна синхронизация на изображението
```
Warning: some tags failed to sync
```
Решение:
1. Проверете регистрационните файлове за синхронизиране, за да идентифицирате неуспешни тагове
2. Стартирайте повторно работния процес ръчно, за да синхронизирате отново
3. Проверете дали изходното изображение в Docker Hub е здраво

### 4. Наблюдение и оповестяване
- Проверявайте редовно страницата Действия, за да потвърдите състоянието на работния процес
- Активирайте известията на GitHub, така че грешките в работния процес да се докладват незабавно
- Наблюдавайте използването на хранилището на Azure ACR
- Редовно проверявайте съгласуваността на етикетите в регистрите

## Често задавани въпроси и решения

### В1: Как да синхронизирам определени тагове вместо всички тагове?

Променете `images.yaml` конфигурационен файл:
```yaml
# Sync only the latest and v1.0 tags
docker.io/newbe36524/hagicode:latest: hagicode.azurecr.io/hagicode:latest
docker.io/newbe36524/hagicode:v1.0: hagicode.azurecr.io/hagicode:v1.0
```

### Q2: Как да синхронизирам множество хранилища за изображения?

Добавете няколко реда към `images.yaml`:
```yaml
docker.io/newbe36524/hagicode: hagicode.azurecr.io/hagicode
docker.io/newbe36524/another-image: hagicode.azurecr.io/another-image
```

### Q3: Как да опитам отново след неуспешна синхронизация?
- Автоматичен повторен опит: image-syncer включва вграден механизъм за повторен опит (`3` повторни опити по подразбиране)
- Ръчен повторен опит: щракнете `Re-run all jobs` на страницата GitHub Actions

### Q4: Как мога да видя подробен напредък на синхронизирането?
- Вижте регистрационните файлове в реално време на страницата Действия
- Изтеглете `sync-logs` артефакт за проверка на пълните лог файлове
- Регистрационните файлове включват състояние на синхронизация и скорост на трансфер за всеки етикет

### Q5: Колко време отнема синхронизирането?
- Първоначална пълна синхронизация: обикновено `10-30` минути в зависимост от размера на изображението
- Инкрементална синхронизация: обикновено `2-5` минути, ако промените в изображението са малки
- Времето зависи от честотната лента на мрежата, размера на изображението и настройките за едновременност
