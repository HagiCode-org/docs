## От TypeScript към C#: практика за междуезично пренасяне за Codex SDK

> Казано по-просто, тази статия също е малко наше бебе: тя записва пълния процес на пренасяне на официалния TypeScript Codex SDK към C#. Наричането му „пристанище“ почти го прави твърде лесно – беше по-скоро като дълго приключение, защото тези два езика имат много различни характери и трябваше да намерим начин да ги накараме да си сътрудничат.

## Фон

Codex е CLI инструментът на AI Agent, издаден от OpenAI, и е наистина мощен. Официалният екип предоставя TypeScript SDK в `@openai/codex` пакет. Той взаимодейства с Codex CLI, като извиква `codex exec --experimental-json` команда и анализиране на JSONL поток от събития.

Проблемът е, че в проекта HagiCode трябва да го използваме в чиста .NET среда - по-специално в C# backend услуги и десктоп приложения. Не можехме разумно да въведем среда за изпълнение на Node.js в .NET проект само за да извикаме CLI инструмент. Това би било твърде тромаво.

Така че бяхме оставени с два избора: да поддържаме сложен Node.js мостов слой или сами да изградим собствен C# SDK.

Ние избрахме второто.

Тази многоезична и многоплатформена сложност е точно причината, поради която се нуждаехме от собствен C# SDK - наистина не искахме да изпълняваме Node.js в .NET проект.

## Основно съдържание

### Сравнение на архитектурния дизайн

Преди да преведем кода едно към едно, първо трябваше да разберем архитектурния дизайн на двата SDK. Трябва да разберете и двете страни, преди да можете да ги пренесете добре.

Основната архитектура на TypeScript SDK изглежда така:

```
Codex (entry class)
  └── CodexExec (executor, manages child processes)
      └── Thread (conversation thread)
          ├── run() / runStreamed() (synchronous/asynchronous execution)
          └── event stream parsing
```

C# SDK запазва същото архитектурно наслояване, но адаптира детайлите на изпълнението. Цялостната идея е ясна: запазване на последователността на API, като същевременно се използват напълно функциите на езика C# при внедряването.

### Тип системно преобразуване

Това е най-фундаменталната и също така най-важната част от работата. Ако основата е слаба, всичко, което следва, става по-твърдо.

Типовата система на TypeScript е по-гъвкава от тази на C# и това е просто факт. Трябваше да намерим подходяща стратегия за картографиране:

| TypeScript | C# | Бележки |
|------------|-----|------|
| `interface` / `type` | `record` | C# използва `record` за неизменни структури от данни |
| `низ \| нула` | `string?` | Nullable референтен тип |
| `булев \| неопределен` | `bool?` | Nullable Boolean |
| `AsyncGenerator` | `IAsyncEnumerable` | Асинхронен итератор |

**Системата тип събитие** е типичен пример. TypeScript използва типове обединение, за да дефинира събития:

```typescript
export type ThreadEvent =
  | ThreadStartedEvent
  | TurnStartedEvent
  | TurnCompletedEvent
  | ...
```

В C# използваме йерархия на наследяване и съвпадение на шаблони, за да постигнем подобен ефект:

```csharp
public abstract record ThreadEvent(string Type);

public sealed record ThreadStartedEvent(string ThreadId) : ThreadEvent("thread.started");
public sealed record TurnStartedEvent() : ThreadEvent("turn.started");
public sealed record TurnCompletedEvent(Usage Usage) : ThreadEvent("turn.completed");
// ...
```

Ние избрахме `record` вместо `class` тъй като обектите за събития трябва да бъдат неизменни, което съответства на намерението зад използването на обикновени обекти в TypeScript. The `sealed` ключовата дума също предотвратява допълнително наследяване и дава възможност на компилатора да оптимизира.

### Ключови точки за пренасяне

#### 1. Анализатор на събития

Разборът на събития е ядрото на целия SDK, защото определя дали можем да разберем правилно всяко съобщение, върнато от Codex CLI. Ако анализът е грешен, всичко след това е напразно усилие.

Версията TypeScript използва `JSON.parse()` за анализ на всеки ред от JSON:

```typescript
export function parseEvent(line: string): ThreadEvent {
  const data = JSON.parse(line);
  // Handle different event types...
}
```

C# версията използва `System.Text.Json.JsonDocument` вместо това:

```csharp
public static ThreadEvent Parse(string line)
{
    using var document = JsonDocument.Parse(line);
    var root = document.RootElement;
    var type = GetRequiredString(root, "type", "event.type");

    return type switch
    {
        "thread.started" => new ThreadStartedEvent(GetRequiredString(root, "thread_id", ...)),
        "turn.started" => new TurnStartedEvent(),
        "turn.completed" => new TurnCompletedEvent(ParseUsage(...)),
        // ...
        _ => new UnknownThreadEvent(type, root.Clone()),
    };
}
```

Тук има един малък, но важен трик: `root.Clone()` се изисква, тъй като елементи от `JsonDocument` стават невалидни след изхвърляне на документа. Трябва да запазим копие за неизвестни типове събития. Това е просто една от разликите между обработката на C# JSON и JavaScript.

#### 2. Разлики в управлението на процесите

Това е мястото, където двата SDK се различават най-много. Node.js и .NET имат различни конвенции за изпълнение, така че внедряването трябва да се адаптира.

TypeScript използва Node.js `spawn()` функция:

```typescript
const child = spawn(this.executablePath, commandArgs, { env, signal });
```

C# използва .NET `System.Diagnostics.Process`:

```csharp
using var process = new Process { StartInfo = startInfo };
process.Start();

// stdin/stdout/stderr must be managed manually
```

По-конкретно, C# версията трябва да конфигурира процеса по следния начин:

```csharp
var startInfo = new ProcessStartInfo
{
    FileName = _executablePath,
    RedirectStandardInput = true,
    RedirectStandardOutput = true,
    RedirectStandardError = true,
    UseShellExecute = false,
    CreateNoWindow = true,
};
```

Най-голямата разлика е механизмът за анулиране. TypeScript използва `AbortSignal`, който е част от Web API и много удобен за работа:

```typescript
const child = spawn(cmd, args, { signal: cancellationSignal });
```

C# използва `CancellationToken` вместо това:

```csharp
public async IAsyncEnumerable<string> RunAsync(
    CodexExecArgs args,
    [EnumeratorCancellation] CancellationToken cancellationToken = default)
{
    // Check cancellation status inside the loop
    while (!cancellationToken.IsCancellationRequested)
    {
        // Process output...
    }

    // Terminate the process when cancellation is requested
    if (cancellationToken.IsCancellationRequested)
    {
        try { process.Kill(entireProcessTree: true); } catch { }
    }
}
```

На високо ниво това е просто още един пример за разликата между Web API екосистемата и .NET екосистемата.

#### 3. Запазване на конфигурационната сериализация

И двата SDK прилагат логиката, която преобразува конфигурацията JSON в конфигурация TOML, тъй като CLI на Codex приема замени на конфигурацията във формат TOML. Тази част трябва да остане напълно последователна, в противен случай една и съща конфигурация ще се държи различно в двата SDK.

Това е вид детайл, с който не можете да правите компромис. Успехът или неуспехът често се свежда до подробности като този.

### Подробности за изпълнението

#### Структура на проекта

Създадохме следната структура на проекта:

```
CodexSdk/
├── CodexSdk.csproj
├── Codex.cs           # Entry class
├── CodexThread.cs     # Conversation thread
├── CodexExec.cs       # Executor
├── Events.cs          # Event type definitions
├── Items.cs           # Item type definitions
├── EventParser.cs     # Event parser
├── OutputSchemaTempFile.cs  # Temporary file management
└── ...
```

Това е доста чиста структура и това помогна много по време на пристанището.

#### Пример за употреба

Основното използване остава в съответствие с TypeScript SDK:

```csharp
using CodexSdk;

// Create a Codex instance
var codex = new Codex();
var thread = codex.StartThread();

// Execute a query
var result = await thread.RunAsync("Summarize this repository.");
Console.WriteLine(result.FinalResponse);
```

Обработката на поточно събитие се възползва от съпоставянето на шаблони на C#:

```csharp
await foreach (var @event in thread.RunStreamedAsync("Analyze the code."))
{
    switch (@event)
    {
        case ItemCompletedEvent itemCompleted
            when itemCompleted.Item is AgentMessageItem msg:
            Console.WriteLine($"Assistant: {msg.Text}");
            break;
        case TurnCompletedEvent completed:
            Console.WriteLine($"Tokens: in={completed.Usage.InputTokens}");
            break;
        case CommandExecutionItem command:
            Console.WriteLine($"Command: {command.Command}");
            break;
    }
}
```

### Бележки

По време на изпълнението събрахме няколко практически урока:

1. **Управление на процеси**: C# версията трябва да управлява пълния жизнен цикъл на процеса ръчно, включително прекратяване на процес по време на анулиране. Използвайте `Kill(entireProcessTree: true)` за да сте сигурни, че дъщерните процеси също са изчистени.

2. **Обработка на грешки**: Използваме `InvalidOperationException` за генериране на грешки при анализиране, запазвайки стила на обработка на грешки, подобен на TypeScript SDK.

3. **Почистване на ресурси**: `OutputSchemaTempFile` инструменти `IAsyncDisposable` за да сте сигурни, че временните файлове са почистени правилно.

4. **Променливи на обкръжението**: Версията на C# поддържа пълно замяна на променливи на обкръжението на процеса `CodexOptions.Env`. Това е малка функция, но много практична.

5. **Разлики в платформата**: C# версията не включва логиката на TypeScript версията за автоматично локализиране на двоични файлове в npm пакети. Тъй като .NET проектите обикновено не зависят от npm, пътят към `codex` изпълним файл трябва да бъде указан чрез `CODEX_EXECUTABLE` променлива на средата или `CodexPathOverride`.

## Заключение

Пренасянето на зрял TypeScript SDK към C# не е само въпрос на преобразуване на синтаксиса - то също така изисква разбиране на философията на дизайна на двата езика. Гъвкавостта на TypeScript и функциите на екосистемата на JavaScript, като напр `AbortSignal` се нуждаят от подходящи аналогове в C#.

Ключовият извод е следният: **поддържането на последователност на API е по-важно от поддържането на последователност на ниво внедряване**. Потребителите се интересуват дали интерфейсът е лесен за използване, а не дали вътрешната реализация е идентична. Това звучи просто, но правенето на тези компромиси изисква преценка.

Ако работите върху подобен междуезичен порт, нашият опит е първо да разберем напълно оригиналната SDK архитектура, след това да я преведем модул по модул и накрая да използваме пълен тестов пакет, за да гарантираме поведенческа последователност. Този вид работа не може да се бърза.

Всичко ще се нареди накрая.
