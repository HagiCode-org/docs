## Как да възпроизвеждаме проекти в ерата на изкуствения интелект: Vault, система за постоянно съхранение между проекти

> В ерата на разработката, подпомагана от AI, как можем да помогнем на AI асистентите да разберат по-добре нашите учебни ресурси? Проектът HagiCode изгради системата Vault като унифициран абстрактен слой за съхранение на знания, който AI може да разбере, което значително подобрява ефективността на обучението чрез възпроизвеждане на проекта.

## Фон

В ерата на изкуствения интелект начинът, по който разработчиците изучават нови технологии и архитектури, се променя дълбоко. „Възпроизвеждането на проекти“ – т.е. задълбоченото изучаване и учене от кода, архитектурата и дизайнерските модели на отлични проекти с отворен код – се превърна в ефикасен начин за учене. В сравнение с традиционните методи като четене на книги или гледане на видеоклипове, директното четене и изпълнение на висококачествени проекти с отворен код ви помага да разберете инженерните практики в реалния свят много по-бързо.

Все пак този метод на обучение идва с доста предизвикателства.

Учебните материали са твърде разпръснати. Вашите бележки може да живеят в Obsidian, хранилищата на кодове може да са разпръснати в различни папки, а историята на разговорите на вашия AI асистент се превръща в още един изолиран остров с данни. Когато искате AI да ви помогне да анализирате проект, трябва ръчно да копирате кодови фрагменти и да организирате контекста, което е доста досадно.

Това, което е още по-неприятно, е нарушеният контекст. Асистентите с изкуствен интелект нямат директен достъп до вашите местни учебни ресурси, така че трябва да предоставяте основна информация отново при всеки разговор. Освен това хранилищата на възпроизведен код се актуализират бързо, ръчното синхронизиране е податливо на грешки и знанията се споделят трудно в множество учебни проекти.

В основата си всички тези проблеми идват от „островите на данни“. Ако имаше унифициран абстракционен слой за съхранение, който позволяваше на AI асистентите да разбират и да имат достъп до всичките ви учебни ресурси, проблемът щеше да бъде решен добре.

## Философия на дизайна на системата за трезори

Основната идея на системата Vault е проста: създайте унифициран абстрактен слой за съхранение на знания, който AI може да разбере. От гледна точка на внедряване, системата има няколко ключови характеристики.

### Многотипова поддръжка

Системата поддържа четири типа трезори, всеки от които отговаря на различен сценарий на използване:

```typescript
// folder: general-purpose folder type
export const DEFAULT_VAULT_TYPE = 'folder';

// coderef: a type specifically for reproduced code projects
export const CODEREF_VAULT_TYPE = 'coderef';

// obsidian: integrated with Obsidian note-taking software
export const OBSIDIAN_VAULT_TYPE = 'obsidian';

// system-managed: vault automatically managed by the system
export const SYSTEM_MANAGED_VAULT_TYPE = 'system-managed';
```

Сред тях типът **coderef** е най-често използваният в HagiCode. Той е специално проектиран за проекти с възпроизведен код, като предоставя стандартизирана структура на директория и четими от AI описания на метаданни.

### Механизъм за постоянно съхранение

Регистърът на Vault се съхранява постоянно във формат JSON, което гарантира, че конфигурацията остава налична след рестартиране на приложението:

```csharp
public class VaultRegistryStore : IVaultRegistryStore
{
    private readonly string _registryFilePath;

    public VaultRegistryStore(IConfiguration configuration, ILogger<VaultRegistryStore> logger)
    {
        var dataDir = configuration["DataDir"] ?? "./data";
        var absoluteDataDir = Path.IsPathRooted(dataDir)
            ? dataDir
            : Path.GetFullPath(Path.Combine(Directory.GetCurrentDirectory(), dataDir));

        _registryFilePath = Path.Combine(absoluteDataDir, "personal-data", "vaults", "registry.json");
    }
}
```

Предимството на този дизайн е, че е прост и надежден. JSON е четим от хора, което улеснява отстраняването на грешки и ръчното редактиране; съхранението на файловата система избягва сложността на базата данни и намалява системните зависимости. В крайна сметка понякога най-простият вариант наистина е най-добрият.

### AI контекстна интеграция

Най-важното е, че системата може автоматично да инжектира информация от трезора в контекста на предложенията на AI:

```typescript
export function buildTargetVaultsText(
  vaults: VaultForText[],
  template: VaultPromptTemplate = DEFAULT_VAULT_PROMPT_TEMPLATE,
): string {
  const readOnlyVaults = vaults.filter((vault) => vault.accessType === 'read');
  const editableVaults = vaults.filter((vault) => vault.accessType === 'write');

  if (readOnlyVaults.length === 0 && editableVaults.length === 0) {
    return '';
  }

  const sections = [
    buildVaultSection(readOnlyVaults, template.reference),
    buildVaultSection(editableVaults, template.editable),
  ].filter(Boolean);

  return `\n\n### ${template.heading}\n\n${sections.join('\n')}`;
}
```

Това дава възможност за важна възможност: AI асистентите могат автоматично да разбират наличните учебни ресурси, без потребителите ръчно да предоставят контекст. Може да се каже, че това се счита за вид мълчаливо разбиране.

## Стандартизираната структура на CodeRef Vault

За типа трезор coderef HagiCode предоставя стандартизирана структура на директория:

```
my-coderef-vault/
├── index.yaml          # vault metadata description
├── AGENTS.md           # operating guide for AI assistants
├── docs/               # stores study notes and documents
└── repos/              # manages reproduced code repositories through Git submodules
```

Когато създавате трезор, системата автоматично инициализира тази структура:

```csharp
private async Task EnsureCodeRefStructureAsync(
    string vaultName,
    string physicalPath,
    ICollection<VaultBootstrapDiagnosticDto> diagnostics,
    CancellationToken cancellationToken)
{
    Directory.CreateDirectory(physicalPath);

    var indexPath = Path.Combine(physicalPath, CodeRefIndexFileName);
    var docsPath = Path.Combine(physicalPath, CodeRefDocsDirectoryName);
    var reposPath = Path.Combine(physicalPath, CodeRefReposDirectoryName);

    // Create the standard directory structure
    if (!Directory.Exists(docsPath))
    {
        Directory.CreateDirectory(docsPath);
    }

    if (!Directory.Exists(reposPath))
    {
        Directory.CreateDirectory(reposPath);
    }

    // Create the AGENTS.md guide
    await EnsureCodeRefAgentsDocumentAsync(physicalPath, cancellationToken);

    // Create the index.yaml metadata
    await WriteCodeRefIndexDocumentAsync(indexPath, mergedDocument, cancellationToken);
}
```

Тази структура също е внимателно проектирана:

- **docs/** съхранява вашите учебни бележки, където можете да запишете вашето разбиране на кода, анализ на архитектурата, научени уроци и т.н. в Markdown
- **repos/** управлява възпроизведени хранилища чрез Git подмодули вместо директно копиране на код, което поддържа кода в синхрон и спестява място
- **index.yaml** съдържа метаданните на хранилището, така че AI асистентите могат бързо да разберат целта и съдържанието на хранилището
- **AGENTS.md** е ръководство, написано специално за AI асистенти, обясняващо как да боравите със съдържанието на трезора

Организиран по този начин, може би AI може да разбере какво имате предвид малко по-лесно.

## Автоматична инициализация за управлявани от системата трезори

В допълнение към ръчно създадените трезори, HagiCode поддържа и управлявани от системата трезори:

```csharp
public async Task<IReadOnlyList<VaultRegistryEntry>> EnsureAllSystemManagedVaultsAsync(
    CancellationToken cancellationToken = default)
{
    var definitions = GetAllResolvedDefinitions();
    var entries = new List<VaultRegistryEntry>(definitions.Count);

    foreach (var definition in definitions)
    {
        entries.Add(await EnsureResolvedSystemManagedVaultAsync(definition, cancellationToken));
    }

    return entries;
}
```

Системата автоматично създава и управлява следните трезори:

- **hagiprojectdata**: съхранение на данни за проекта, използвано за запазване на конфигурацията и състоянието на проекта
- **лични данни**: съхранение на лични данни, използвано за запазване на потребителските предпочитания
- **hbsprompt**: библиотека с шаблони за подкани, използвана за управление на често използвани подкани за AI

Тези трезори се инициализират автоматично при стартиране на системата, така че потребителите не трябва да ги конфигурират ръчно. Някои неща просто е по-добре да оставим на системата, вместо хората да се тревожат за тях.

## Механизъм за контрол на достъпа

Важна част от дизайна е контролът на достъпа. Системата разделя трезорите на два типа достъп:

```typescript
export interface VaultForText {
  id: string;
  name: string;
  type: string;
  physicalPath: string;
  accessType: 'read' | 'write';  // Key: distinguish read-only from editable
}
```

- **справка (само за четене)**: AI се използва само за анализ и разбиране и не може да променя съдържанието. Подходящо за реферирани проекти с отворен код, документи и подобни материали
- **с възможност за редактиране (с възможност за редактиране)**: AI може да променя съдържанието според нуждите на задачата. Подходящо за вашите бележки, чернови и подобни материали

Това разграничение има значение. Той казва на AI кое съдържание е „препратка само за четене“ и кое съдържание е „безопасно за редактиране“, намалявайки риска от случайни промени. В края на краищата никой не иска упоритата му работа да изчезне поради непреднамерена редакция.

## На практика: създаване и използване на Vault

Сега, след като разгледахме идеите, нека да видим как работи на практика.

### Създайте CodeRef Vault

Ето пълен пример за обаждане от интерфейса:

```typescript
const createCodeRefVault = async () => {
  const response = await VaultService.postApiVaults({
    requestBody: {
      name: "React Learning Vault",
      type: "coderef",
      physicalPath: "/Users/developer/vaults/react-learning",
      gitUrl: "https://github.com/facebook/react.git"
    }
  });

  // The system will automatically:
  // 1. Clone the React repository into vault/repos/react
  // 2. Create the docs/ directory for notes
  // 3. Generate the index.yaml metadata
  // 4. Create the AGENTS.md guide file

  return response;
};
```

Това извикване на API завършва поредица от действия: създаване на структурата на директорията, инициализиране на Git подмодули, генериране на файлове с метаданни и др. Трябва да предоставите само основната информация и да оставите системата да се справи с останалото. Честно казано, това е доста безгрижен подход.

## Най-добри практики и неща, за които да следите

Докато използвахме системата Vault, ние обобщихме няколко научени урока.

### Безопасност на пътя

Системата стриктно валидира пътищата, за да предотврати атаки при преминаване на пътя:

```csharp
private static string ResolveFilePath(string vaultRoot, string relativePath)
{
    var rootPath = EnsureTrailingSeparator(Path.GetFullPath(vaultRoot));
    var combinedPath = Path.GetFullPath(Path.Combine(rootPath, relativePath));
    if (!combinedPath.StartsWith(rootPath, StringComparison.OrdinalIgnoreCase))
    {
        throw new BusinessException(VaultRelativePathTraversalCode,
            "Vault file paths must stay inside the registered vault root.");
    }
    return combinedPath;
}
```

това е важно Ако персонализирате път на трезор, уверете се, че остава в рамките на разрешения диапазон, в противен случай системата ще отхвърли операцията. Наистина не можете да наблегнете прекалено много на сигурността.

### Управление на подмодули на Git

CodeRef Vault препоръчва Git подмодули вместо директно копиране на код:

```csharp
private static string BuildCodeRefAgentsContent()
{
    return """
    # CodeRef Vault Guide

    Repositories under `repos/` should be maintained through Git submodules
    rather than copied directly into the vault root.

    Keep this structure stable so assistants and tools can understand the vault quickly.
    """ + Environment.NewLine;
}
```

Това носи няколко предимства: поддържане на кода синхронизиран с upstream, спестяване на дисково пространство и улесняване на управлението на множество версии на кода. В крайна сметка, кой иска да изтегля едно и също нещо отново и отново?

### Ограничения за преглед на файлове

За да предотврати проблеми с производителността, системата ограничава размера и типа на файла:

```csharp
private const int FileEnumerationLimit = 500;
private const int PreviewByteLimit = 256 * 1024;  // 256KB
```

Ако вашият трезор съдържа голям брой файлове или много големи файлове, производителността на визуализацията може да бъде засегната. В такъв случай можете да обмислите обработка на файлове на партиди или използване на специализирани инструменти за търсене. Понякога, когато нещо стане твърде голямо, става по-трудно за справяне, а не по-лесно.

### Диагностична информация

Когато създава трезор, системата връща диагностична информация, за да помогне при отстраняването на грешки:

```csharp
List<VaultBootstrapDiagnosticDto> bootstrapDiagnostics = [];

if (IsCodeRefVaultType(normalizedType))
{
    bootstrapDiagnostics = await EnsureCodeRefBootstrapAsync(
        normalizedName,

```

Ако създаването е неуспешно, можете да прегледате диагностичната информация, за да разберете конкретната причина. Когато нещо се обърка, проверката на диагностиката често е най-прекият път напред.

## Резюме

Чрез унифициран абстракционен слой за съхранение, системата Vault разрешава няколко основни проблемни точки при възпроизвеждане на проекти в ерата на AI:

- **Централизирано управление на знанията**: всички учебни ресурси са събрани на едно място, вместо да са разпръснати навсякъде
- **Автоматично инжектиране на AI контекст**: AI асистентите могат автоматично да разбират наличните учебни ресурси без ръчна настройка на контекста
- **Повторно използване на знанието за различни проекти**: знанието може да се споделя и използва повторно в множество учебни проекти
- **Стандартизирана структура на директорията**: последователното оформление на директорията намалява кривата на обучение

Това решение вече е валидирано в проекта HagiCode. Ако също изграждате инструменти, свързани с разработка, подпомагана от AI, или се сблъсквате с подобни проблеми с управлението на знания, надявам се, че тези преживявания могат да послужат като полезна справка.

В интерес на истината стойността на едно техническо решение не се крие в това колко е сложно, а в това дали решава реални проблеми. Основната идея на системата Vault е много проста: изграждане на единен слой за съхранение на знания, който AI може да разбере. И все пак точно тази проста абстракция подобри доста нашата ефективност на разработката.

Понякога простият подход наистина е най-добрият. В крайна сметка сложните неща често крият още повече клопки...
