# OpenSpec'e Giriş

OpenSpec, ekiplerin yapılandırılmış teklif ve spesifikasyonlar aracılığıyla yazılım geliştirme sürecini yönetmelerine yardımcı olan spesifikasyon odaklı bir geliştirme aracıdır. Bu makale, OpenSpec'in temel kavramlarına, iş akışlarına ve kullanımına dair kapsamlı bir giriş sunar.

## OpenSpec nedir?

OpenSpec, yazılım geliştirme sürecinde teklifleri (Proposals), spesifikasyonları (Specifications) ve değişiklikleri (Changes) yönetmek için kullanılan bir komut satırı aracıdır. Geliştirme sürecini standartlaştırır ve her özelliğin fikirden uygulamaya kadar kapsamlı bir tasarım ve inceleme sürecinden geçmesini sağlar.

### Temel ilkeler

OpenSpec aşağıdaki temel ilkelere dayanır:

- **Spesifikasyon önceliklidir**: Kod yazmadan önce net spesifikasyonlar tanımlayın
- **Yapılandırılmış işbirliği**: Standart belgelenme formatları aracılığıyla ekip işbirliğini teşvik edin
- **İzlenebilirlik**: Tekliften uygulamaya kadar tüm süreci kaydedin
- **Kalite güvencesi**: Spesifikasyon doğrulaması ile kodun tasarım amacına uygun olduğundan emin olun

### Öne çıkan özellikler

OpenSpec aşağıdaki ana özellikleri sunar:

1. **Teklif yönetimi**: Özellik teklifleri oluşturma, gözden geçirme ve takip etme
2. **Spesifikasyon tanımı**: Teknik spesifikasyon belgeleri yazma ve bakımını yapma
3. **Değişiklik takibi**: Gözden geçirilmiş değişikliklerin tüm geçmişini kaydetme
4. **Spesifikasyon doğrulama**: Kod değişikliklerinin önceden tanımlanmış spesifikasyonlara uygun olduğunu doğrulama
5. **Ekip işbirliği**: Çok katılımlı geliştirme ve inceleme süreçlerini destekleme

## OpenSpec iş akışı

Tipik OpenSpec iş akışı aşağıdaki aşamaları içerir:

### 1. Teklif oluşturma (Proposal)

Yeni bir özellik veya önemli bir değişiklik talebi olduğunda önce bir teklif oluşturun:

```bash
openspec proposal create
```

Teklif şunları içerir:
- **Sorunun açıklaması**: Çözülmesi gereken sorun veya gereksinim
- **Çözüm**: Önerilen çözümün özeti
- **Etkileşim kapsamı**: Mevcut sistem üzerindeki etkisi
- **Alternatif çözümler**: Düşünülen diğer seçenekler

### 2. Spesifikasyon yazma (Specification)

Teklif onaylandıktan sonra ayrıntılı bir teknik spesifikasyon yazın:

```bash
openspec spec create
```

Spesifikasyon şunları içerir:
- **Teknik tasarım**: Teknik uygulama için ayrıntılı plan
- **Arayüz tanımları**: API'ler, veri yapıları vb. gibi arayüzlerin tasarımı
- **Uygulama detayları**: Önemli uygulama mantığı ve algoritmalar
- **Test planı**: Uygulamanın doğruluğunu nasıl doğrulayacağınız

### 3. Değişiklik oluşturma (Change)

Uygulamaya başlarken geliştirme sürecini takip etmek için bir değişiklik oluşturun:

```bash
openspec change create
```

Değişiklik şunları içerir:
- **İlişkili spesifikasyon**: Bu değişikliğin uyguladığı spesifikasyon
- **Uygulama görevleri**: Belirli geliştirme görevlerinin listesi
- **İlerleme takibi**: Görevlerin tamamlanma durumu
- **Kod değişiklikleri**: Gerçek kod modifikasyonları

### 4. Doğrulama ve arşivleme

Geliştirme tamamlandıktan sonra değişikliğin spesifikasyona uygunluğunu doğrulayın ve ardından arşivleyin:

```bash
openspec change validate
openspec change archive
```

## OpenSpec dizin yapısı

OpenSpec, proje kök dizininde aşağıdaki yapıya sahip `openspec/` dizini oluşturur:

```
openspec/
├── proposals/           # Teklif dizini
│   ├── PROPOSAL-001.md
│   └── PROPOSAL-002.md
├── specs/              # Spesifikasyon dizini
│   ├── SPEC-001.md
│   └── SPEC-002.md
├── changes/            # Değişiklik dizini
│   ├── feature-name/
│   │   ├── proposal.md
│   │   ├── design.md
│   │   ├── specs.md
│   │   └── tasks.md
│   └── another-feature/
└── archived/           # Arşivlenmiş değişiklikler
    └── feature-name/
```

### Dizin açıklaması

- **proposals/**: Tüm teklif belgelerini saklar; her teklif ayrı bir Markdown dosyasıdır
- **specs/**: Teklife karşılık gelen teknik spesifikasyon belgelerini saklar
- **changes/**: Devam eden değişiklikleri saklar; her değişiklik bir alt dizinde bulunur
- **archived/**: Tamamlanmış ve arşivlenmiş değişiklikleri saklar

## Temel komutlar

### Teklif ile ilgili komutlar

```bash
# Yeni teklif oluştur
openspec proposal create

# Tüm teklifleri listele
openspec proposal list

# Teklif detaylarını görüntüle
openspec proposal view <proposal-id>

# Teklifi onayla
openspec proposal approve <proposal-id>
```

### Spesifikasyon ile ilgili komutlar

```bash
# Yeni spesifikasyon oluştur
openspec spec create

# Tüm spesifikasyonları listele
openspec spec list

# Spesifikasyon detaylarını görüntüle
openspec spec view <spec-id>

# Spesifikasyonu güncelle
openspec spec update <spec-id>
```

### Değişiklik ile ilgili komutlar

```bash
# Yeni değişiklik oluştur
openspec change create

# Tüm değişiklikleri listele
openspec change list

# Değişiklik detaylarını görüntüle
openspec change view <change-name>

# Görev ekle
openspec change add-task <change-name>

# Değişikliği doğrula
openspec change validate <change-name>

# Değişikliği arşivle
openspec change archive <change-name>
```

### Genel komutlar

```bash
# Sürümü görüntüle
openspec --version

# Yardımı görüntüle
openspec --help

# Belirli bir komut için yardımı görüntüle
openspec <command> --help
```

## HagiCode projesinde OpenSpec kullanımı

### Proje entegrasyonu

HagiCode projesinde OpenSpec geliştirme iş akışına entegre edilmiştir:

1. **Başlatma denetimi**: Başlangıç sihirbazı OpenSpec'in kurulu olup olmadığını kontrol eder
2. **Teklif odaklı geliştirme**: Tüm yeni özelliklerin OpenSpec teklif sürecinden geçmesi gerekir
3. **Kod incelemesi**: Değişikliklerin merge edilebilmesi için OpenSpec spesifikasyonlarına uygun olması gerekir
4. **Dokümantasyon senkronizasyonu**: OpenSpec dokümantasyonu kod uygulaması ile senkron tutulur

### Sürüm gereksinimleri

HagiCode projesi şu sürümü kullanmayı gerektirir: **OpenSpec >=1.0.0 <2.0.0**.

Kurulum komutu:

```bash
npm install -g @fission-ai/openspec@1
```

### İş akışı örneği

Aşağıda HagiCode projesinde OpenSpec kullanımına dair tam bir örnek bulunmaktadır:

#### Adım 1: Teklif oluşturma

```bash
cd /path/to/hagicode-project
openspec proposal create
```

Etkileşimli istemde şunu girin:
- Teklif başlığı: Kullanıcı kimlik doğrulama özelliği ekle
- Sorun açıklaması: Sistem kullanıcı girişi ve izin yönetimine ihtiyaç duyuyor
- Çözüm: JWT tabanlı kimlik doğrulama sistemi uygulayın

#### Adım 2: Spesifikasyon yazma

Teklif onaylandıktan sonra ayrıntılı bir spesifikasyon oluşturun:

```bash
openspec spec create
```

Spesifikasyonda tanımlayın:
- Kimlik doğrulama akışının tasarımı
- API arayüz tanımları
- Veritabanı tablo yapısı
- Güvenlik hususları

#### Adım 3: Değişiklik oluşturma

Uygulamaya başlayın:

```bash
openspec change create
```

Değişiklik adı girin: `add-user-authentication`

#### Adım 4: Görev ekleme

```bash
openspec change add-task add-user-authentication
```

Belirli görevleri ekleyin:
- Kullanıcı kayıt uç noktasını uygulayın
- Giriş uç noktasını uygulayın
- JWT middleware uygulayın
- Birim testleri yazın

#### Adım 5: Uygulama ve doğrulama

Kod tamamlandıktan sonra değişikliği doğrulayın:

```bash
openspec change validate add-user-authentication
```

#### Adım 6: Arşivleme

Başarılı doğrulamadan sonra değişikliği arşivleyin:

```bash
openspec change archive add-user-authentication
```

## En iyi uygulamalar

### Teklif yazma

1. **Sorunu net bir şekilde ifade edin**: Çözülmesi gereken problemi açıkça tanımlayın; belirsiz açıklamalardan kaçının
2. **Alternatifleri değerlendirin**: Farklı çözümleri listeleyin ve karşılaştırın
3. **Etkisini değerlendirin**: Mevcut sisteme etkileri ve riskleri analiz edin
4. **Kısa ve öz tutun**: Teklifler kısa ve net olmalıdır; detayları spesifikasyona bırakın

### Spesifikasyon yazma

1. **Detaylı ama gereksiz değil**: Yeterli detayı sağlayın, aşırı tasarımdan kaçının
2. **Diyagram kullanın**: Açıklayıcı olması için akış diyagramları, mimari diyagramlar vb. kullanın
3. **Arayüzleri tanımlayın**: Tüm genel arayüzleri ve veri yapıları açıkça tanımlayın
4. **Sınır durumları düşünün**: İstisnai durumları ve hata işleme yöntemlerini listeleyin

### Değişiklik yönetimi

1. **Küçük adımlarla iterasyon**: Büyük özellikleri birden çok küçük değişikliğe bölün
2. **Zamanında güncelleme**: Görev durumu ve dokümantasyonu senkron tutun
3. **Kod incelemesi**: Değişiklik tamamlandıktan sonra kod incelemesi yapın
4. **Zamanında arşivleme**: Tamamlanan değişiklikleri zamanında arşivleyin ve çalışma dizinini temiz tutun

### Ekip işbirliği

1. **Düzenli gözden geçirmeler**: Teklifleri ve spesifikasyonları düzenli olarak gözden geçirerek ekip uzlaşmasını sağlayın
2. **Dokümantasyon önce**: Kodlamaya başlamadan önce dokümantasyonu tamamlayın
3. **İletişimi sürdürün**: Tartışmaları ve kararları tekliflerde ve spesifikasyonlarda kaydedin
4. **Sürüm kontrolü**: OpenSpec dokümantasyonunu sürüm kontrolüne dahil edin

## Yaygın kullanım senaryoları

### Senaryo 1: Yeni özellik ekleme

1. Yeni özelliğin gereksinimlerini açıklayan bir teklif oluşturun
2. Ekip teklifi tartışır ve onaylar
3. Detaylı bir teknik spesifikasyon yazın
4. Bir değişiklik oluşturun ve özelliği uygulayın
5. Değişikliği doğrulayın ve arşivleyin

### Senaryo 2: Mevcut kodu refactor etmek

1. Refactor nedenlerini ve hedeflerini açıklayan bir teklif oluşturun
2. Refactor planını ve adımlarını spesifikasyonda belirtin
3. Bir değişiklik oluşturun ve kademeli olarak refactor edin
4. Test kapsamını sağlayın ve refactor sonuçlarını doğrulayın
5. Değişikliği arşivleyin

### Senaryo 3: Hata düzeltmesi

Basit hatalar için:
- Doğrudan düzeltilebilir, tam OpenSpec süreci gerekmez

Karmaşık hatalar için:
1. Hatanın kök nedenini analiz etmek için bir teklif oluşturun
2. Düzeltme çözümünü spesifikasyonda tanımlayın
3. Bir değişiklik oluşturun ve düzeltmeyi uygulayın
4. Regresyonu önlemek için testler ekleyin
5. Değişikliği arşivleyin

### Senaryo 4: Teknik borcu yönetme

1. Teknik borç öğelerini listeleyen bir teklif oluşturun
2. İyileştirme planını spesifikasyonda tanımlayın
3. Teknik borcu kademeli olarak ödemek için bir değişiklik oluşturun
4. İlerlemeyi ve etkiyi takip edin
5. Tamamlanan iyileştirmeleri arşivleyin

## Diğer araçlarla entegrasyon

### Git entegrasyonu

OpenSpec, Git ile sorunsuz entegrasyona sahiptir:

```bash
# OpenSpec belgeleri sürüm kontrolüne dahil edilmelidir
git add openspec/
git commit -m "Add OpenSpec proposal for feature X"
```

### CI/CD entegrasyonu

OpenSpec spesifikasyonlarını CI/CD boru hattında doğrulayın:

```yaml
# .github/workflows/openspec.yml
name: OpenSpec Validation
on: [pull_request]
jobs:
  validate:
    runs-on: ubuntu-latest
    steps:
      - uses: actions/checkout@v2
      - name: Install OpenSpec
        run: npm install -g @fission-ai/openspec@1
      - name: Validate changes
        run: openspec change validate
```

### IDE entegrasyonu

OpenSpec belgelerini düzenlemek için Markdown destekli bir düzenleyici kullanmanız önerilir:
- VS Code + Markdown eklentisi
- JetBrains IDEs + Markdown desteği
- Vim/Neovim + Markdown eklentisi

## Sorun giderme

### Yaygın sorunlar

#### Sorun: OpenSpec komutu bulunamadı

**Çözüm**:
1. OpenSpec'in kurulu olduğunu doğrulayın: `npm list -g @fission-ai/openspec`
2. PATH ortam değişkeninin npm küresel bin dizinini içerdiğini doğrulayın
3. Yeniden yükleyin: `npm install -g @fission-ai/openspec@1`

#### Sorun: Teklif veya değişiklik oluşturulamıyor

**Çözüm**:
1. Geçerli dizinin proje kök dizini olduğunu doğrulayın
2. `openspec/` dizininin var olup olmadığını kontrol edin; yoksa oluşturun
3. Yazma izinlerine sahip olduğunuzu doğrulayın

#### Sorun: Doğrulama başarısız oldu

**Çözüm**:
1. Değişiklik dizinindeki belgelerin tamam olduğunu kontrol edin
2. Gerekli tüm görevlerin tamamlandığını doğrulayın
3. Ayrıntılı hata bilgilerini inceleyin: `openspec change validate <change-name> --verbose`

#### Sorun: Arşivleme başarısız oldu

**Çözüm**:
1. Değişikliğin doğrulamadan geçtiğinden emin olun
2. `openspec/archived/` dizininin var olduğunu kontrol edin
3. Henüz commit edilmemiş kod değişikliklerinin olmadığını doğrulayın

## İleri düzey kullanım

### Özelleştirilmiş şablonlar

OpenSpec, teklifler ve spesifikasyonlar için özelleştirilmiş şablonları destekler:

```bash
# Proje kökünde bir şablon dizini oluşturun
mkdir -p .openspec/templates

# Özelleştirilmiş bir teklif şablonu oluşturun
cat > .openspec/templates/proposal.md << 'EOF'
# Proposal: {{title}}

## Problem Statement
{{problem}}

## Proposed Solution
{{solution}}

## Custom Section
Your custom content here
EOF
```

### Yapılandırma dosyası

Proje kökünde `.openspec.json` yapılandırma dosyası oluşturun:

```json
{
  "version": "1.0",
  "templates": {
    "proposal": ".openspec/templates/proposal.md",
    "spec": ".openspec/templates/spec.md"
  },
  "validation": {
    "requireTests": true,
    "requireDocs": true
  },
  "hooks": {
    "pre-archive": "npm test"
  }
}
```

### Hooks

OpenSpec, belirli işlemler öncesinde ve sonrasında özel betikler çalıştırmayı destekler:

```json
{
  "hooks": {
    "pre-create": "scripts/check-permissions.sh",
    "post-archive": "scripts/notify-team.sh"
  }
}
```
