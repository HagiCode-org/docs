## HagiCode'da StreamJsonRpc'nin Derin Entegrasyonu ve Pratik Kullanımı

> Bu makale, HagiCode (eski adıyla PCode) projesinin Microsoft'un StreamJsonRpc iletişim kitaplığını orijinal özel JSON-RPC uygulamasının yerine nasıl başarılı bir şekilde entegre ettiğini ve aynı zamanda entegrasyon süreci sırasında karşılaşılan teknik sorunları ve mimari zorlukları nasıl çözdüğünü ayrıntılı olarak açıklamaktadır.

## Arka plan

StreamJsonRpc, Microsoft'un .NET ve TypeScript için resmi olarak tutulan JSON-RPC iletişim kitaplığıdır ve güçlü tür güvenliği, otomatik proxy oluşturma ve olgun istisna işleme mekanizmalarıyla bilinir. HagiCode projesinde ekip, iflow CLI ve OpenCode CLI gibi harici AI araçlarıyla ACP (Ajan İletişim Protokolü) aracılığıyla iletişim kurmak ve aynı zamanda daha önceki özel JSON-RPC uygulamasının getirdiği bakım maliyetini ve olası hataları ortadan kaldırmak için StreamJsonRpc'yi entegre etmeye karar verdi. Ancak entegrasyon sürecinde, özellikle proxy hedef bağlama ve genel parametre tanıma işlenirken, JSON-RPC akışına özgü zorluklarla karşılaşıldı.

Bu sıkıntılı noktaları gidermek için cesur bir karar aldık: tüm yapı sistemini sıfırdan yeniden inşa ettik. Bu kararın etkisi sandığınızdan da büyük olabilir, bunu birazdan detaylı olarak anlatacağım.

## Analiz

Mevcut proje, ACP protokol entegrasyonunun kritik bir aşamasındadır ve aşağıdaki teknik sorunlarla ve mimari zorluklarla karşı karşıyadır:

### 1. Özel Uygulamanın Sınırlamaları
Orijinal JSON-RPC uygulaması şurada bulunur: `src/HagiCode.ClaudeHelper/AcpImp/` ve gibi bileşenleri içerir `JsonRpcEndpoint` ve `ClientSideConnection`. Bu özel kod tabanını korumak maliyetlidir ve olgun bir kitaplığın ilerleme raporlaması ve iptal desteği gibi gelişmiş yeteneklerinden yoksundur.

### 2. StreamJsonRpc Entegrasyon Engelleri
Mevcut olanı taşımaya çalışırken `CallbackProxyTarget` StreamJsonRpc'ye desen yerleştirdiğimizde şunu bulduk: `_rpc.AddLocalRpcTarget(target)` yöntem, proxy modeli aracılığıyla oluşturulan hedefleri tanıyamadı. Özellikle StreamJsonRpc genel türün özelliklerini otomatik olarak bölemedi `T` RPC yöntemi parametrelerine dönüştürülerek istemci tarafından başlatılan yöntem çağrıları işlenirken sunucu tarafının başarısız olmasına neden olur.

### 3. Karışık Mimari Katmanlama
Mevcut `ClientSideConnection` taşıma katmanını (WebSocket/Stdio), protokol katmanını (JSON-RPC) ve iş katmanını (ACP Agent arayüzü) karıştırarak sorumlulukların belirsiz olmasına yol açar. Ayrıca eksik yöntem bağlamalarından da muzdariptir. `AcpAgentCallbackRpcAdapter`.

### 4. Eksik Günlükler
WebSocket aktarım katmanında ham JSON içerik günlüğünün bulunmaması, RPC hata ayıklaması sırasında sorunun serileştirmeden mi yoksa ağdan mı kaynaklandığının belirlenmesini zorlaştırır.

## Çözüm

Yukarıdaki sorunları çözmek için, üç boyuttan optimizasyon yaparak aşağıdaki sistematik çözümü benimsedik: mimari yeniden düzenleme, kitaplık entegrasyonu ve gelişmiş hata ayıklama.

### 1. StreamJsonRpc'ye Tamamen Geçiş Yapın

#### Eski Kodu Kaldır
Sil `JsonRpcEndpoint.cs`, `AgentSideConnection.cs`ve ilgili özel serileştirme dönüştürücüleri gibi `JsonRpcMessageJsonConverter`.

#### Resmi Kütüphaneyi entegre edin
Tanıtın `StreamJsonRpc` NuGet paketini kullanın ve kullanın `JsonRpc` Temel iletişim mantığını ele alacak sınıf.

#### Aktarım Katmanını Soyutla
Tanımla `IAcpTransport` her ikisini de idare edecek arayüz `WebSocket` ve `Stdio` taşıma modlarını birleşik bir şekilde kullanarak protokol katmanının taşıma katmanından ayrılmasını sağlar.

```csharp
// Definition of the IAcpTransport interface
public interface IAcpTransport
{
    Task SendAsync(string message, CancellationToken cancellationToken = default);
    Task<string> ReceiveAsync(CancellationToken cancellationToken = default);
    Task CloseAsync(CancellationToken cancellationToken = default);
}

// WebSocket transport implementation
public class WebSocketTransport : IAcpTransport
{
    private readonly WebSocket _webSocket;

    public WebSocketTransport(WebSocket webSocket)
    {
        _webSocket = webSocket;
    }

    // Implement send and receive methods
    // ...
}

// Stdio transport implementation
public class StdioTransport : IAcpTransport
{
    private readonly StreamReader _reader;
    private readonly StreamWriter _writer;

    public StdioTransport(StreamReader reader, StreamWriter writer)
    {
        _reader = reader;
        _writer = writer;
    }

    // Implement send and receive methods
    // ...
}
```

### 2. Proxy Hedef Tanıma Sorunlarını Düzeltin

#### Analiz et `CallbackProxyTarget`
Mevcut dinamik proxy oluşturma mantığını inceleyin ve StreamJsonRpc'nin bunu tanıyamamasının temel nedenini belirleyin. Çoğu durumda bunun nedeni, proxy nesnesinin gerçek yöntem imzalarını herkese açık olarak göstermemesi veya StreamJsonRpc tarafından desteklenmeyen parametre türlerini kullanmasıdır.

#### Refactor Parametre Geçişi
Genel özellikleri açık RPC yöntemi parametrelerine bölün. Dinamik özelliklere güvenmek yerine somut İstek/Yanıt DTO'ları (veri aktarım nesneleri) tanımlayın, böylece StreamJsonRpc yansıma yoluyla yöntem imzalarını doğru şekilde tanıyabilir.

```csharp
// Original generic property approach
public class CallbackProxyTarget<T>
{
    public Func<T, Task> Callback { get; set; }
}

// Refactored concrete method approach
public class ReadTextFileRequest
{
    public string FilePath { get; set; }
}

public class ReadTextFileResponse
{
    public string Content { get; set; }
}

public interface IAcpAgentCallback
{
    Task<ReadTextFileResponse> ReadTextFileAsync(ReadTextFileRequest request);
    // Other methods...
}
```

#### Kullanım `Attach` Bunun yerine `AddLocalRpcTarget`
Bazı karmaşık senaryolarda, manuel olarak proxy oluşturma `JsonRpc` nesne ve işleme `RpcConnection` doğrudan yerel bir hedef eklemekten daha esnek olabilir.

### 3. Yöntem Bağlama ve Daha Güçlü Günlüğe Kaydetme Uygulayın

#### Uygula `AcpAgentCallbackRpcAdapter`
Bu bileşenin StreamJsonRpc proxy arayüzünü açıkça uyguladığından ve ACP protokolü tarafından tanımlanan aşağıdaki gibi eşleme yöntemlerini uyguladığından emin olun: `ReadTextFileAsync`, StreamJsonRpc geri arama işleyicilerine.

#### Günlük Kaydını Entegre Edin
JSON-RPC isteklerinin ve yanıtlarının ham metnini WebSocket veya Stdio mesaj işleme hattında yakalayın ve kaydedin. Kullanım `ILogger` biçimlendirme sorunlarının daha kolay teşhis edilebilmesi için ayrıştırmadan önce ve serileştirmeden sonra ham yükün çıktısını almak.

```csharp
// Transport wrapper with enhanced logging
public class LoggingAcpTransport : IAcpTransport
{
    private readonly IAcpTransport _innerTransport;
    private readonly ILogger<LoggingAcpTransport> _logger;

    public LoggingAcpTransport(IAcpTransport innerTransport, ILogger<LoggingAcpTransport> logger)
    {
        _innerTransport = innerTransport;
        _logger = logger;
    }

    public async Task SendAsync(string message, CancellationToken cancellationToken = default)
    {
        _logger.LogTrace("Sending message: {Message}", message);
        await _innerTransport.SendAsync(message, cancellationToken);
    }

    public async Task<string> ReceiveAsync(CancellationToken cancellationToken = default)
    {
        var message = await _innerTransport.ReceiveAsync(cancellationToken);
        _logger.LogTrace("Received message: {Message}", message);
        return message;
    }

    public async Task CloseAsync(CancellationToken cancellationToken = default)
    {
        _logger.LogDebug("Closing connection");
        await _innerTransport.CloseAsync(cancellationToken);
    }
}
```

### 4. Mimari Katmanlamayı Yeniden Düzenleyin

#### Taşıma Katmanı (`AcpRpcClient`)
StreamJsonRpc bağlantısını sarın ve sorumluluğu ona verin `InvokeAsync` çağrılar ve bağlantı yaşam döngüsü yönetimi.

```csharp
public class AcpRpcClient : IDisposable
{
    private readonly JsonRpc _rpc;
    private readonly IAcpTransport _transport;

    public AcpRpcClient(IAcpTransport transport)
    {
        _transport = transport;
        _rpc = new JsonRpc(new StreamRpcTransport(transport));
        _rpc.StartListening();
    }

    public async Task<TResponse> InvokeAsync<TResponse>(string methodName, object parameters)
    {
        return await _rpc.InvokeAsync<TResponse>(methodName, parameters);
    }

    public void Dispose()
    {
        _rpc.Dispose();
        _transport.Dispose();
    }

    // StreamRpcTransport is the StreamJsonRpc adapter for IAcpTransport
    private class StreamRpcTransport : IDuplexPipe
    {
        // Implement the IDuplexPipe interface
        // ...
    }
}
```

#### Protokol Katmanı (`IAcpAgentClient` / `IAcpAgentCallback`)
İstemciden aracıya ve aracıdan istemciye anlaşılır arayüzler tanımlayın. Döngüsel fabrika modelini kaldırın `Func<IAcpAgent, IAcpClient>` ve bunu bağımlılık enjeksiyonu veya doğrudan geri arama kaydıyla değiştirin.

## Alıştırma

StreamJsonRpc'nin en iyi uygulamalarına ve proje deneyimine dayanarak, uygulamaya yönelik temel öneriler şunlardır:

### 1. Kesinlikle Yazılmış DTO'lar Dinamik Nesnelerden Daha İyidir
StreamJsonRpc'nin temel avantajı güçlü yazımda yatmaktadır. Kullanma `dynamic` veya `JObject` parametreleri geçirmek için. Bunun yerine, her RPC yöntemi için açık C# POCO sınıflarını parametre olarak tanımlayın. Bu yalnızca proxy hedef tanıma sorununu çözmekle kalmaz, aynı zamanda derleme zamanında tür hatalarını da yakalar.

Örnek: genel özellikleri değiştirin `CallbackProxyTarget` gibi somut sınıflarla `ReadTextFileRequest` ve `WriteTextFileRequest`.

### 2. Yöntem Adlarını Açıkça Bildirin
Şunu kullanın: `[JsonRpcMethod]` Varsayılan yöntem adı eşlemesine güvenmek yerine, RPC yöntem adlarını açıkça belirtme özelliğini kullanın. Bu, PascalCase ve camelCase gibi adlandırma stili farklılıklarından kaynaklanan çağrı hatalarını önler.

```csharp
public interface IAcpAgentCallback
{
    [JsonRpcMethod("readTextFile")]
    Task<ReadTextFileResponse> ReadTextFileAsync(ReadTextFileRequest request);
    
    [JsonRpcMethod("writeTextFile")]
    Task WriteTextFileAsync(WriteTextFileRequest request);
}
```

### 3. Bağlantı Durumu Geri Aramalarından Yararlanın
StreamJsonRpc şunları sağlar: `JsonRpc.ConnectionLost` olay. Beklenmeyen işlem çıkışları veya ağ bağlantı kesintileriyle başa çıkmak için bu olayı mutlaka dinlemelisiniz; bu, yalnızca Orleans Tahıl arıza tespitine güvenmekten daha zamanındadır.

```csharp
_rpc.ConnectionLost += (sender, e) =>
{
    _logger.LogError("RPC connection lost: {Reason}", e.ToString());
    // Handle reconnection logic or notify the user
};
```

### 4. Günlükleri Katmana Göre Kaydedin
- **İzleme düzeyi**: Ham JSON istek/yanıt yükünün tamamını kaydedin.
- **Hata ayıklama düzeyi**: Yöntem çağrısı yığınlarını ve parametre özetlerini kaydedin.
- **Not**: Günlüklerin hassas Yetkilendirme belirteçlerini veya Base64 kodlu büyük dosya içeriğini içermediğinden emin olun.

### 5. Akışlı Taşımacılığın Özel Doğasını Yönetin
StreamJsonRpc yerel olarak destekler `IAsyncEnumerable`. ACP için akış istemi yanıtlarını uygularken şunu kullanın: `IAsyncEnumerable` özel sayfalandırma mantığı yerine doğrudan. Bu, akış işleme için gereken kod miktarını büyük ölçüde basitleştirebilir.

```csharp
public interface IAcpAgentCallback
{
    [JsonRpcMethod("streamText")]
    IAsyncEnumerable<string> StreamTextAsync(StreamTextRequest request);
}
```

### 6. Adaptör Deseni
Tut `ACPSession` ve `ClientSideConnection` ayrı. `ACPSession` Orleans durum yönetimine ve mesaj kuyruğa alma gibi iş mantığına odaklanmalı ve StreamJsonRpc bağlantı nesnesini miras yerine kompozisyon yoluyla kullanmalıdır.

## Sonuç

HagiCode projesi, StreamJsonRpc'yi kapsamlı bir şekilde entegre ederek, orijinal özel uygulamanın yüksek bakım maliyetini, işlevsel sınırlamalarını ve mimari katmanlama karışıklığını başarıyla giderdi. Temel iyileştirmeler şunları içerir:

1. Dinamik özellikleri güçlü bir şekilde yazılan DTO'larla değiştirme, sürdürülebilirliği ve güvenilirliği artırma
2. Taşıma katmanı soyutlaması ve protokol katmanı ayırmanın uygulanması, mimari netliğin iyileştirilmesi
3. İletişim sorunlarının teşhis edilmesini kolaylaştırmak için günlük kaydı yeteneklerinin güçlendirilmesi
4. Yayın uygulamasını basitleştirmek için yayın desteğiyle tanışın

Bu iyileştirmeler, HagiCode'a daha istikrarlı ve daha verimli bir iletişim temeli sağlayarak, onun harici yapay zeka araçlarıyla daha iyi etkileşime girmesine olanak tanıyor ve gelecekteki özellik genişletmeleri için sağlam bir temel oluşturuyor.
