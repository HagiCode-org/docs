## Tarayıcı WebSocket Kimlik Doğrulama Zorluklarını Çözme: Doubao Konuşma Tanıma için Pratik Bir Proxy Modeli

> Tarayıcı WebSocket API'si özel HTTP başlıklarını desteklemez; bu da başlıklarda kimlik doğrulama verileri gerektiren konuşma tanıma hizmetleri için zorluk oluşturur. Bu makale, HagiCode projesinde bu sorunu arka uç proxy modeliyle nasıl çözdüğümüzü ve yaklaşımın oyun alanı deneylerinden üretim kullanımına nasıl evrildiğini paylaşıyor.


## Arka plan

HagiCode projesi için konuşma tanımayı geliştirmeye başladığımızda, güvenle ByteDance'in Doubao konuşma tanıma hizmetini seçtik. İlk tasarım basitti: ön ucun doğrudan Doubao'nun WebSocket hizmetine bağlanmasına izin verin. Bu ne kadar zor olabilir ki? Sadece bir bağlantı açın ve biraz veri gönderin, değil mi?

Sonra bir sürpriz geldi: Doubao'nun API'si, kimlik doğrulama bilgilerinin HTTP başlıklarından iletilmesini gerektiriyor; buna aşağıdakiler de dahil: `accessToken` ve `secretKey`. Tarayıcı WebSocket API'si özel başlıkların ayarlanmasını desteklemediğinden bu durum hemen garipleşti.

Peki tarayıcı bunları göndermenize izin vermediğinde ne yaparsınız?

O zamanlar iki seçeneği tartıyorduk:

1. Kimlik bilgilerini URL sorgu parametrelerine yerleştirin - basit ve net
2. Arka uca bir proxy katmanı ekleyin; ilk bakışta daha fazla iş

İlk seçenek, kimlik bilgilerini doğrudan ön uç kodunda ve yerel depolamada gösterir. Bu gerçekten güvenli mi? Bu konuda rahat değildim. Üstelik bazı API'ler başlık tabanlı doğrulama gerektirir, dolayısıyla bu yaklaşım geçerli bile değildir.

Sonunda ikinci seçeneği seçtik: arka uçta bir WebSocket proxy uygulamak. Tesadüfen, bu model ilk olarak oyun alanı ortamımızda doğrulandı ve ancak stabilitesini doğruladıktan sonra onu üretime taşıdık. Sonuçta hiç kimse üretimin laboratuvar deneyi olarak ikiye katlanmasını istemez.

## Teknik Zorluk Analizi

### Tarayıcı WebSocket sınırlamaları

Standart WebSocket API'si son derece basit görünüyor:

```javascript
const ws = new WebSocket('wss://example.com/ws');
```

Ancak sorunun kaynağı tam olarak bu basitliktir; yalnızca URL'deki parametreleri iletir ve başlıkları bir HTTP isteğinin yapabildiği şekilde ayarlayamaz:

```javascript
// This is not supported in the WebSocket API
const ws = new WebSocket('wss://example.com/ws', {
  headers: {
    'Authorization': 'Bearer token'
  }
});
```

Ve temel mesele de budur. Başlık tabanlı kimlik doğrulamasına bağlı olan Doubao konuşma tanıma gibi hizmetler için bu sınırlama ciddi bir engelleyicidir.

Bu kısıtlamayı kabul ettiğinizde mimarinin değişmesi gerekir.

### Mimari tasarım kararları

Çözümü tasarlarken dengeleri dikkatlice karşılaştırdık.

**Karar 1: Proxy modelini seçme**

İki yaklaşımı karşılaştırdık:

| Seçenek | Artıları | Eksileri | Karar |
|------|------|------|------|
| Yerel WebSocket | Hafif, basit, doğrudan iletme | Bağlantı yönetimi manuel olarak gerçekleştirilmelidir | seçilmiş |
| SinyalR | Otomatik yeniden bağlanma, güçlü yazma | Aşırı karmaşık, ekstra bağımlılıklar | Reddedildi |

Sonuçta yerel WebSocket'i seçtik. Dürüst olmak gerekirse, en hafif seçenekti ve basit çift yönlü ikili akış iletimi için daha uygundu. SignalR'ı kullanmak bu kullanım durumu için aşırı mühendislik gibi gelebilir ve ekstra gecikme yaratabilir.

**Karar 2: Bağlantı yönetimi stratejisi**

"Tek bağlantı, tek oturum" modelini benimsedik; her ön uç WebSocket bağlantısı, kendi bağımsız Doubao arka uç bağlantısıyla eşleşir.

Faydaları basittir:
- Uygulaması basit ve yaygın kullanım modeliyle uyumlu
- Hata ayıklamak ve sorun gidermek daha kolay
- İyi kaynak izolasyonu, oturumlar arasındaki paraziti önleme

Basitçe söylemek gerekirse, doğrudan çözüm bazen en iyisidir. Karmaşıklık bir tasarımı otomatik olarak daha iyi hale getirmez.

**Karar 3: Kimlik doğrulama verilerinin saklanması**

Kimlik bilgileri arka uç yapılandırma dosyalarında (`appsettings.yml` veya ortam değişkenleri) ve bağımlılık enjeksiyonu yoluyla yüklendi:

- Mevcut arka uç kurallarına uyan basit yapılandırma modeli
- Hassas veriler hiçbir zaman ön uca ulaşmaz
- Geliştirme, test ve üretim için çoklu ortam kurulumunu destekler

Bu düzeyde bir ayrım önemlidir. Hiç kimse kimlik bilgilerinin olmaması gereken yerlerde ortalıkta dolaşmasını istemez.

### Veri akışı tasarımı

Genel veri akışı şuna benzer:

```
Frontend (browser)
  │
  │ ws://backend/api/voice/ws
  │ WebSocket (binary)
  ▼
Backend (proxy)
  │
  │ wss://openspeech.bytedance.com/
  │ (with authentication headers)
  ▼
Doubao API
```

Akışın kendisi karmaşık değildir:
1. Ön uç, WebSocket aracılığıyla arka uç proxy'sine bağlanır
2. Arka uç proxy'si ses verilerini alır ve kimliği doğrulanmış başlıklarla Doubao API'sine bağlanır
3. Doubao API, tanıma sonuçlarını döndürür ve proxy bunları ön uca iletir
4. Tüm süreç, çift yönlü akışla tamamen eşzamansız kalır

Sorumluluklar net bir şekilde bölündüğünde tasarım oldukça doğal hale gelir.

## Temel bileşen uygulaması

### 1. WebSocket uç nokta yapılandırması

```csharp
app.Map("/ws", async context =>
{
    if (context.WebSockets.IsWebSocketRequest)
    {
        // Read configuration from query parameters
        var appId = context.Request.Query["appId"];
        var accessToken = context.Request.Query["accessToken"];

        // Validate required parameters
        if (string.IsNullOrEmpty(appId) || string.IsNullOrEmpty(accessToken))
        {
            context.Response.StatusCode = 400;
            return;
        }

        // Accept the WebSocket connection
        using var webSocket = await context.WebSockets.AcceptWebSocketAsync();

        // Message handling loop
        var buffer = new byte[4096];
        while (!webSocket.CloseStatus.HasValue)
        {
            var result = await webSocket.ReceiveAsync(buffer, CancellationToken.None);

            if (result.MessageType == WebSocketMessageType.Close)
            {
                await webSocket.CloseAsync(
                    result.CloseStatus.Value,
                    result.CloseStatusDescription,
                    CancellationToken.None);
                break;
            }

            // Process audio data
            await HandleAudioDataAsync(buffer, result.Count);
        }
    }
});
```

### 2. Oturum yönetimi

```csharp
public class DoubaoSessionManager : IDoubaoSessionManager
{
    private readonly ConcurrentDictionary<string, DoubaoSession> _sessions = new();

    public DoubaoSession CreateSession(string connectionId)
    {
        var session = new DoubaoSession(connectionId);
        _sessions[connectionId] = session;
        return session;
    }

    public async Task SendAudioAsync(string connectionId, byte[] audioData)
    {
        if (_sessions.TryGetValue(connectionId, out var session))
        {
            await session.SendAudioAsync(audioData);
        }
    }

    public void RemoveSession(string connectionId)
    {
        if (_sessions.TryRemove(connectionId, out var session))
        {
            session.Dispose();
        }
    }
}
```

Kullanma `ConcurrentDictionary` oturum yönetimi için iş parçacığı güvenliğinin büyük ölçüde bizim için ele alındığı anlamına gelir. Gelen her bağlantının kendi oturumu vardır ve bağlantı kesildiğinde temizleme işlemi otomatik olarak gerçekleşir.

## Uygulamada ön uç entegrasyonu

### WebSocket bağlantısı

```javascript
class DoubaoVoiceClient {
    constructor(config) {
        this.config = config;
        this.ws = null;
    }

    async connect() {
        const url = new URL(this.config.wsUrl);
        // Add query parameters
        Object.entries(this.config.params).forEach(([key, value]) => {
            url.searchParams.set(key, value);
        });

        this.ws = new WebSocket(url);

        return new Promise((resolve, reject) => {
            this.ws.onopen = () => {
                console.log('[DoubaoVoice] Connected');
                resolve();
            };

            this.ws.onmessage = (event) => {
                this._handleMessage(JSON.parse(event.data));
            };

            this.ws.onerror = reject;
        });
    }

    _handleMessage(message) {
        switch (message.type) {
            case 'status':
                this._handleStatus(message.payload);
                break;
            case 'result':
                this.onResult?.(message.payload);
                break;
            case 'error':
                console.error('[DoubaoVoice] Error:', message.payload);
                break;
        }
    }
}

// Usage example
const client = new DoubaoVoiceClient({
    wsUrl: 'ws://localhost:5000/ws',
    params: {
        appId: 'your-app-id',
        accessToken: 'your-access-token',
        sampleRate: 16000,
        bitsPerSample: 16,
        channels: 1
    }
});
```

### Ses yakalama ve aktarma

Kullanma `AudioWorkletNode` ses işleme için daha iyi performans sağlar:

```javascript
// audio-worklet.js
class AudioProcessorWorklet extends AudioWorkletProcessor {
    process(inputs, outputs, parameters) {
        const input = inputs[0]?.[0];
        if (!input) return true;

        // Convert to 16-bit PCM
        const pcm = new Int16Array(input.length);
        for (let i = 0; i < input.length; i++) {
            pcm[i] = Math.max(-32768, Math.min(32767, input[i] * 32767));
        }

        this.port.postMessage({
            type: 'audioData',
            data: pcm.buffer
        }, [pcm.buffer]);

        return true;
    }
}

registerProcessor('audio-processor', AudioProcessorWorklet);

// Main thread code
async function startAudioRecording() {
    const stream = await navigator.mediaDevices.getUserMedia({
        audio: {
            echoCancellation: true,
            noiseSuppression: true,
            autoGainControl: true,
            sampleRate: 48000
        }
    });

    const audioContext = new AudioContext();
    const audioSource = audioContext.createMediaStreamSource(stream);

    await audioContext.audioWorklet.addModule('/audio-worklet.js');
    const audioWorkletNode = new AudioWorkletNode(audioContext, 'audio-processor');

    audioWorkletNode.port.onmessage = (event) => {
        if (event.data.type === 'audioData' && ws?.readyState === WebSocket.OPEN) {
            ws.send(event.data.data); // Send binary data directly
        }
    };

    audioSource.connect(audioWorkletNode);
}
```

`AudioWorklet` göre çok daha iyi performans gösteriyor `ScriptProcessorNode` ve eski işleme yollarının sıklıkla ortaya çıkardığı ses kesintisi sorunlarını ortadan kaldırır.

## Notlar ve en iyi uygulamalar

### 1. Bağlantı izleme

- Bağlantı yaşam döngüsünün tamamını izlemek için oturum durumunu düzenli olarak günlüğe kaydedin
- Anormal bağlantıları tespit etmek için ses bölümlerinin sayısını ve süresini izleyin
- Doubao hizmeti için bağlantı durumunu ve yeniden bağlanma davranışını kaydedin

Bunlar temel operasyonel uygulamalardır ancak üretimde gerçek bir fark yaratırlar.

### 2. Hata yönetimi

- Tüm WebSocket istisnalarını yakalayın ve günlüğe kaydedin
- Kullanım `IAsyncDisposable` Kaynakların temizlenmesini sağlamak için
- Bağlantının zarif bir şekilde kapatılması ve zaman aşımı yönetimini uygulayın

Kısacası sağlamlığı tercih edin.

### 3. Ses formatı gereksinimleri

- Örnekleme hızı: 16000 Hz (önerilen) veya 8000 Hz
- Bit derinliği: 16 bit
- Kanallar: mono
- Kodlama: PCM (ham)

Biçim yanlışsa tanıma başarısız olabilir veya kalite önemli ölçüde düşebilir. Bu gereksinimler önemlidir.

### 4. Güvenlik hususları

- Hassas kimlik bilgilerini yalnızca arka uç yapılandırmasında tutun
- Kaynak tükenmesini önlemek için bağlantı sınırlarını uygulayın
- Üretimde HTTPS/WSS kullanın

Güvenlik asla küçük bir endişe kaynağı değildir.

### 5. Performans optimizasyonu

- Engellemeyi önlemek için eşzamansız işlemleri kullanın
- Tampon boyutlarını gerektiği gibi ayarlayın (varsayılan: 4096 bayt)
- Bağlantı havuzu oluşturma ve yeniden kullanım stratejilerini göz önünde bulundurun

Bu optimizasyonları iş yükünüz için anlamlı olan yerlere uygulayın.

## Dağıtım önerileri

1. **Docker dağıtımı**: Daha kolay ölçeklendirme ve yönetim için proxy hizmetini kapsayıcı olarak paketleyin
2. **Yük dengeleme**: WebSocket trafiği için Nginx veya Envoy'u ters proxy olarak kullanın
3. **Durum kontrolleri**: Hizmet kullanılabilirliğini izlemek için kalp atışı tabanlı kontroller uygulayın
4. **Günlük toplama**: Günlükleri ELK veya Loki gibi merkezi bir günlük kaydı sistemine gönderin

Dağıtım, ekibe ve ortama bağlı olarak basit veya karmaşık olabilir; dolayısıyla buna göre ayarlayın.

## Özet

WebSocket proxy modeli, tarayıcı WebSocket API'sinin özel başlıkları desteklememesi temel sorununu çözer. HagiCode projesinde bu model, oyun alanı doğrulamasından üretim dağıtımına geçerken hem uygulanabilir hem de istikrarlı olduğunu kanıtladı.

Temel çıkarımlar:
- Bir arka uç proxy'si, kimlik doğrulama bilgilerini güvenli bir şekilde iletebilir
- Yerel WebSocket basit senaryolar için hafif ve etkilidir
- "Tek bağlantı, tek oturum" modeli hem uygulamayı hem de hata ayıklamayı basitleştirir
- Ön uç-arka uç protokolü, kontrol sinyallerini ses verilerinden ayırmalıdır

Aynı zamanda WebSocket kimlik doğrulamasına da bağlı bir özellik oluşturuyorsanız bu modelin size yararlı bir başlangıç noktası vereceğini umuyorum.

Sorularınız varsa bunları bizimle tartışmaktan çekinmeyin. İnsanlar notları karşılaştırdığında teknik ilerleme daha hızlı gerçekleşir.
