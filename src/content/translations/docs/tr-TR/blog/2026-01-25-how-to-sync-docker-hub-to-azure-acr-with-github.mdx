## Docker Hub'dan Azure ACR'ye Görüntü Senkronizasyonunu Otomatikleştirme

> Bu makalede, Docker Hub'dan Azure Container Registry'ye görüntü senkronizasyonunu otomatikleştirmek için GitHub Eylemleri ve görüntü eşitleyici aracının nasıl kullanılacağı açıklanmakta, Çin'deki ve bazı Azure bölgelerindeki Docker Hub'a yavaş erişim sorununa değinilirken Azure ortamlarında görüntü kullanılabilirliği ve dağıtım verimliliği iyileştirilmektedir.

## Arka Plan / Giriş

HagiCode projesi, Docker Hub'da barındırılan birincil görüntülerle birlikte temel çalışma zamanı bileşenleri olarak Docker görüntülerini kullanır. Proje büyüdükçe ve Azure dağıtım ihtiyaçları arttıkça aşağıdaki sıkıntılı noktalarla karşılaştık:

- Docker Hub erişiminin Çin'de ve bazı Azure bölgelerinde kısıtlanması nedeniyle görüntü çekme hızları yavaş
- Tek bir görüntü kaynağı, tek bir hata noktası riski oluşturur
- Azure Container Registry, Azure'da çalışırken daha iyi ağ performansı ve entegrasyon sağlar

Bu sorunları çözmek için, görüntüleri Docker Hub'dan Azure ACR'ye düzenli olarak senkronize edecek ve kullanıcıların Azure ortamlarında daha yüksek görüntü çekme hızlarına ve daha yüksek kullanılabilirliğe sahip olmalarını sağlayacak bir otomatik görüntü senkronizasyon mekanizmasına ihtiyacımız vardı.

## Teknik Çözüm Karşılaştırması

Çözümü tanımlarken birden fazla teknik seçeneği karşılaştırdık:

### 1. görüntü senkronizasyonu (son seçim)
- Artımlı senkronizasyon: yalnızca değiştirilen görüntü katmanlarını senkronize ederek ağ aktarımını önemli ölçüde azaltır
- Desteği sürdürme: senkronizasyon ağ kesintilerinden sonra devam edebilir
- Eşzamanlılık kontrolü: yapılandırılabilir çalışan iş parçacıkları, büyük görüntü senkronizasyon verimliliğini artırır
- Güçlü hata yönetimi: arızalar için yerleşik yeniden deneme mekanizması (varsayılan olarak 3 yeniden deneme)
- Hafif dağıtım: bağımlılığı olmayan tek bir ikili dosya
- Çoklu kayıt defteri desteği: Docker Hub, Azure ACR, Harbor ve daha fazlasıyla uyumlu

### 2. Docker CLI'sı
- Artımlı senkronizasyon yok: her seferinde tam görüntü içeriğinin çekilmesi gerekiyor
- Daha düşük verimlilik: Büyük ağ aktarım hacmi ve daha uzun yürütme süresi
- Kullanımı basit: tanıdıklara dayalı `docker pull` ve `docker push` komutlar

### 3. Azure CLı
- Daha yüksek karmaşıklık: Azure CLI kimlik doğrulamasının yapılandırılması gerekir
- Sınırlı işlevsellik: `az acr import` kapsamı nispeten dardır
- Yerel entegrasyon: Azure hizmetleriyle iyi bir şekilde entegre olur

## Mimari Tasarım Kararları

- İşin yoğun olduğu saatleri önler ve diğer operasyonlar üzerindeki etkiyi azaltır
- Docker Hub görüntüleri genellikle günlük derlemelerden sonra güncellenir

### Karar 2: Tüm resim etiketlerini senkronize edin
- Docker Hub ile tam tutarlılığı korur
- Kullanıcılara esnek sürüm seçenekleri sunar
- Senkronizasyon mantığını basitleştirir ve karmaşık etiket filtreleme kurallarını ortadan kaldırır

### Karar 3: Kimlik bilgilerini depolamak için GitHub Gizli Dizilerini kullanın
- Güçlü güvenlikle Yerel GitHub Eylemleri desteği
- Basit konfigürasyon ve kolay bakım
- Depo düzeyinde erişim kontrolünü destekler

## Risk Değerlendirmesi ve Azaltma

### Risk 1: Azure ACR kimlik bilgisi sızıntısı
- Kimlik bilgilerini GitHub Secrets'ta şifrelenmiş olarak saklayın
- ACR şifrelerini düzenli olarak değiştirin
- ACR hesabını yalnızca push izinleriyle sınırlandırın
- ACR erişim günlüklerini izleyin

### Risk 2: Görüntü tutarsızlığına neden olan senkronizasyon hataları
- image-syncer yerleşik bir artımlı senkronizasyon mekanizması içerir
- Başarısızlık durumunda otomatik yeniden deneme (varsayılan olarak 3)
- Ayrıntılı hata günlükleri ve arıza bildirimleri
- Devam etme özelliği

### Risk 3: Aşırı kaynak tüketimi
- Artımlı senkronizasyon ağ aktarımını azaltır
- Yapılandırılabilir eşzamanlılık (`10` mevcut kurulumda)
- Senkronize edilen görüntülerin sayısını ve boyutunu izleyin
- Senkronizasyonu yoğun olmayan saatlerde çalıştır

## Temel Çözüm

Görüntüleri Docker Hub'dan Azure ACR'ye senkronize etmek için GitHub Actions + image-syncer'ı temel alan otomatik bir çözüm kullanıyoruz.

## Uygulama Adımları

### 1. Hazırlık
- Azure Portal'da bir Azure Container Registry oluşturun veya onaylayın
- ACR erişim kimlik bilgilerini (kullanıcı adı ve şifre) oluşturun
- Docker Hub görüntü deposu için erişim izinlerini onaylayın

### 2. GitHub Sırlarını Yapılandırın
GitHub deposu ayarlarına aşağıdaki Sırları ekleyin:
- AZURE_ACR_USERNAME: Azure ACR kullanıcı adı
- AZURE_ACR_PASSWORD: Azure ACR parolası

### 3. GitHub Eylemleri iş akışını oluşturun
İş akışını şurada yapılandırın `.github/workflows/sync-docker-acr.yml`:
- Planlanmış tetikleme: her gün 00:00 UTC'de
- Manuel tetikleme: destekler `workflow_dispatch`
- Ek tetikleyici: basıldığında çalışır `publish` hızlı senkronizasyon için şube

### 4. İş akışı yürütme akışı

| Adım | Katılımcı | Eylem | Açıklama |
| --- | --- | --- | --- |
| 1 | GitHub Eylemleri | İş akışını tetikleyin | Zamanlama, manuel yürütme veya itmelerle tetiklenir `publish` şube |
| 2 | GitHub Eylemleri → görüntü senkronizasyonu | Senkronizasyon aracını indirin ve çalıştırın | Gerçek senkronizasyon aşamasına girer |
| 3 | görüntü senkronizasyonu → Docker Hub | Resim bildirimlerini ve etiket listesini getir | Kaynak depodan meta verileri okur |
| 4 | görüntü eşitleyici → Azure ACR | Hedef depodan mevcut görüntü verilerini getir | Hedef tarafın mevcut durumunu belirler |
| 5 | görüntü senkronlayıcı | Kaynak ve hedef farklılıklarını karşılaştırın | Senkronizasyon gerektiren görüntü katmanlarını tanımlar |
| 6 | görüntü senkronizasyonu → Docker Hub | Değiştirilen görüntü katmanlarını çek | Yalnızca güncellenmesi gereken içeriği aktarır |
| 7 | görüntü eşitleyici → Azure ACR | Değiştirilen görüntü katmanlarını itin | Artımlı senkronizasyonu tamamlar |
| 8 | görüntü senkronizasyonu → GitHub Eylemleri | Senkronizasyon istatistiklerini döndür | Sonuçları, farklılıkları ve hata bilgilerini içerir |
| 9 | GitHub Eylemleri | Günlükleri kaydedin ve yapıları yükleyin | Daha sonra denetim ve sorun gidermeyi destekler |

## GitHub Eylemleri İş Akışı Uygulaması

Aşağıda kullanımdaki gerçek iş akışı yapılandırması verilmiştir (`.github/workflows/sync-docker-acr.yml`):

```yaml
name: Sync Docker Image to Azure ACR

on:
  schedule:
    - cron: "0 0 * * *" # Every day at 00:00 UTC
  workflow_dispatch: # Manual trigger
  push:
    branches: [publish]

permissions:
  contents: read

jobs:
  sync:
    runs-on: ubuntu-latest

    steps:
      - name: Checkout code
        uses: actions/checkout@v4

      - name: Download image-syncer
        run: |
          # Download the image-syncer binary
          wget https://github.com/AliyunContainerService/image-syncer/releases/download/v1.5.5/image-syncer-v1.5.5-linux-amd64.tar.gz
          tar -zxvf image-syncer-v1.5.5-linux-amd64.tar.gz
          chmod +x image-syncer

      - name: Create auth config
        run: |
          # Generate the authentication config file (YAML format)
          cat > auth.yaml <<EOF
          hagicode.azurecr.io:
            username: "${{ secrets.AZURE_ACR_USERNAME }}"
            password: "${{ secrets.AZURE_ACR_PASSWORD }}"
          EOF

      - name: Create images config
        run: |
          # Generate the image sync config file (YAML format)
          cat > images.yaml <<EOF
          docker.io/newbe36524/hagicode: hagicode.azurecr.io/hagicode
          EOF

      - name: Run image-syncer
        run: |
          # Run synchronization (using the newer --auth and --images parameters)
          ./image-syncer --auth=./auth.yaml --images=./images.yaml --proc=10 --retries=3

      - name: Upload logs
        if: always()
        uses: actions/upload-artifact@v4
        with:
          name: sync-logs
          path: image-syncer-*.log
          retention-days: 7
```

## Yapılandırma Ayrıntıları

### 1. Tetikleme koşulları
- Zamanlanmış tetikleyici: `cron: "0 0 * * *"` her gün 00:00 UTC'de çalışır
- Manuel tetik: `workflow_dispatch` kullanıcıların GitHub kullanıcı arayüzünden manuel olarak çalıştırmasına olanak tanır
- Tetiği itin: `push: branches: [publish]` hızlı senkronizasyon için serbest bırakma dalına basıldığında tetiklenir

### 2. Kimlik doğrulama yapılandırması (`auth.yaml`)
```yaml
hagicode.azurecr.io:
  username: "${{ secrets.AZURE_ACR_USERNAME }}"
  password: "${{ secrets.AZURE_ACR_PASSWORD }}"
```

### 3. Görüntü senkronizasyonu yapılandırması
```yaml
docker.io/newbe36524/hagicode: hagicode.azurecr.io/hagicode
```
Bu yapılandırma, tüm etiketlerin `docker.io/newbe36524/hagicode` senkronize edilecek `hagicode.azurecr.io/hagicode`.

### 4. görüntü senkronizasyon parametreleri
- `--auth=./auth.yaml`: kimlik doğrulama yapılandırma dosyasının yolu
- `--images=./images.yaml`: görüntü senkronizasyonu yapılandırma dosyasının yolu
- `--proc=10`: kullanmak `10` eşzamanlı işçiler
- `--retries=3`: yeniden deneme hataları `3` kez

## Kullanım Talimatları

### 1. Senkronizasyonu manuel olarak tetikleyin
1. GitHub deposunun Eylemler sekmesini ziyaret edin
2. Şunu seçin: `Sync Docker Image to Azure ACR` iş akışı
3. tıklayın `Run workflow` düğme
4. Bir şube seçin ve tıklayın `Run workflow` onaylamak

### 2. Senkronizasyon günlüklerini görüntüleyin
1. Eylemler sayfasında belirli bir iş akışı çalıştırmasına tıklayın
2. Her adıma ilişkin yürütme günlüklerini inceleyin
3. İndir `sync-logs` sayfanın altındaki Yapılar bölümünden yapıt

### 3. Senkronizasyon sonuçlarını doğrulayın
```bash
# Log in to Azure ACR
az acr login --name hagicode

# List images and their tags
az acr repository show-tags --name hagicode --repository hagicode --output table
```

## Notlar ve En İyi Uygulamalar

### 1. Güvenlik önerileri
- Azure ACR şifrelerini düzenli aralıklarla değiştirin (90 günde bir önerilir)
- Yalnızca push izinlerine sahip özel bir ACR hizmet hesabı kullanın
- Zamanında anormal erişimi tespit etmek için ACR erişim günlüklerini izleyin
- Günlüklerde kimlik bilgilerinin çıktısını almayın
- Kimlik bilgilerini kod deposuna teslim etmeyin

### 2. Performans optimizasyonu
- Ayarlayın `--proc` parametre: ağ bant genişliğine göre eşzamanlılığı ayarlayın (önerilir) `5-20`)
- Senkronizasyon süresini izleyin: senkronizasyon çok uzun sürerse eşzamanlılığı azaltmayı düşünün
- Günlükleri düzenli olarak temizleyin: makul bir değer belirleyin `retention-days` değer (`7` şu anda)

### 3. Sorun Giderme

#### Sorun 1: Kimlik doğrulama hatası
```
Error: failed to authenticate to hagicode.azurecr.io
```
Çözüm:
1. GitHub Gizli Anahtarlarının doğru şekilde yapılandırılıp yapılandırılmadığını kontrol edin
2. Azure ACR parolasının süresinin dolup dolmadığını doğrulayın
3. ACR hizmet hesabı izinlerinin doğru olduğunu onaylayın

#### Sorun 2: Ağ zaman aşımı
```
Error: timeout waiting for response
```
Çözüm:
1. Ağ bağlantısını kontrol edin
2. Eş zamanlı çalışan sayısını azaltın ( `--proc` parametre)
3. Ağın kurtarılmasını bekleyin ve iş akışını yeniden çalıştırın

#### Sorun 3: Eksik görüntü senkronizasyonu
```
Warning: some tags failed to sync
```
Çözüm:
1. Başarısız etiketleri belirlemek için senkronizasyon günlüklerini kontrol edin
2. Tekrar senkronize etmek için iş akışını manuel olarak yeniden çalıştırın
3. Docker Hub'daki kaynak görüntünün sağlıklı olduğunu doğrulayın

### 4. İzleme ve uyarı verme
- İş akışı durumunu doğrulamak için Eylemler sayfasını düzenli olarak kontrol edin
- İş akışı hatalarının anında bildirilmesi için GitHub bildirimlerini etkinleştirin
- Azure ACR depolama kullanımını izleyin
- Kayıtlardaki etiket tutarlılığını düzenli olarak doğrulayın

## Sıkça Sorulan Sorular ve Çözümleri

### S1: Tüm etiketler yerine belirli etiketleri nasıl senkronize edebilirim?

Değiştir `images.yaml` yapılandırma dosyası:
```yaml
# Sync only the latest and v1.0 tags
docker.io/newbe36524/hagicode:latest: hagicode.azurecr.io/hagicode:latest
docker.io/newbe36524/hagicode:v1.0: hagicode.azurecr.io/hagicode:v1.0
```

### S2: Birden fazla görüntü deposunu nasıl senkronize edebilirim?

Birden fazla satır ekle `images.yaml`:
```yaml
docker.io/newbe36524/hagicode: hagicode.azurecr.io/hagicode
docker.io/newbe36524/another-image: hagicode.azurecr.io/another-image
```

### S3: Senkronizasyon başarısız olduktan sonra nasıl yeniden deneyebilirim?
- Otomatik yeniden deneme: image-syncer yerleşik bir yeniden deneme mekanizması içerir (`3` varsayılan olarak yeniden dener)
- Manuel yeniden deneme: tıklayın `Re-run all jobs` GitHub Eylemleri sayfasında

### S4: Ayrıntılı senkronizasyon ilerlemesini nasıl görüntüleyebilirim?
- Eylemler sayfasında gerçek zamanlı günlükleri görüntüleyin
- İndir `sync-logs` tam günlük dosyalarını incelemek için yapı
- Günlük dosyaları, her etiket için senkronizasyon durumunu ve aktarım hızını içerir

### S5: Senkronizasyon ne kadar sürer?
- İlk tam senkronizasyon: genellikle `10-30` görüntü boyutuna bağlı olarak dakika
- Artımlı senkronizasyon: genellikle `2-5` görüntü değişiklikleri küçükse dakikalar
- Zamanlama ağ bant genişliğine, görüntü boyutuna ve eşzamanlılık ayarlarına bağlıdır

## Önerilen Uzantılar

### 1. Senkronizasyon bildirimleri ekleyin
İş akışına bir bildirim adımı ekleyin:
```yaml
- name: Notify on success
  if: success()
  run: |
    echo "Docker images synced successfully to Azure ACR"
```

### 2. Resim etiketi filtrelemeyi uygulayın
İş akışına etiket filtreleme mantığını ekleyin:
```yaml
- name: Filter tags
  run: |
    # Sync only tags prefixed with v
    echo "docker.io/newbe36524/hagicode:v* : hagicode.azurecr.io/hagicode:v*" > images.yaml
```

### 3. Senkronizasyon istatistikleri raporu ekleyin
```yaml
- name: Generate report
  if: always()
  run: |
    echo "## Sync Report" >> $GITHUB_STEP_SUMMARY
    echo "- Total tags: $(grep -c 'synced' image-syncer-*.log)" >> $GITHUB_STEP_SUMMARY
    echo "- Sync time: ${{ steps.sync.outputs.duration }}" >> $GITHUB_STEP_SUMMARY
```
