## GitHub Eylemleri İş Akışını Yapılandırma

GitHub Actions, GitHub tarafından sağlanan CI/CD hizmetidir. İş akışı dosyalarını repository içerisinde YAML formatında tanımlayarak her türlü otomasyon görevini özelleştirebiliyoruz.

### İş Akışı Dosyasını Oluşturma

Yeni bir yapılandırma dosyası oluşturmamız gerekiyor. `.github/workflows` proje kökündeki dizin, örneğin `deploy.yml`. Dizin henüz mevcut değilse, önce onu oluşturun.

Bu konfigürasyon dosyasının temel mantığı aşağıdaki gibidir:

1.  **Tetikleme koşulu**: Dinle `push` üzerindeki olaylar `main` şube.
2.  **Çalışma zamanı ortamı**: En son Ubuntu sürümü.
3.  **Adımları oluşturma**:
    -   Kodu inceleyin
    -   Node.js'yi yükleyin
    -   Bağımlılıkları yükleyin (`npm install`)
    -   Statik dosyaları oluşturun (`npm run build`)
4.  **Dağıtım adımı**: Resmi olanı kullanın `action-gh-pages` derleme çıktısını itme eylemi `gh-pages` şube.

### Anahtar Yapılandırma Kodu

Sonunda benimsediğimiz yapılandırma şablonu aşağıdadır:

```yaml
name: Deploy to GitHub Pages

# Trigger condition: run when code is pushed to the main branch
on:
  push:
    branches:
      - main
    # Add path filters if needed, for example only build when docs change
    # paths:
    #   - 'docs/**'
    #   - 'package.json'

# Set permissions, which is important for GitHub Pages deployment
permissions:
  contents: read
  pages: write
  id-token: write

# Concurrency control: cancel older builds for the same branch
concurrency:
  group: "pages"
  cancel-in-progress: false

jobs:
  build:
    runs-on: ubuntu-latest
    steps:
      - name: Checkout
        uses: actions/checkout@v4
        # Note: fetch-depth: 0 is required, otherwise the build version may be inaccurate
        with:
          fetch-depth: 0

      - name: Setup Node
        uses: actions/setup-node@v4
        with:
          node-version: 20 # Keep this consistent with the local development environment
          cache: 'npm'     # Enabling cache can speed up the build process

      - name: Install dependencies
        run: npm ci
        # Use npm ci instead of npm install because it is faster and stricter, which makes it better for CI

      - name: Build website
        run: npm run build
        env:
          # Configure environment variables here if your site build requires them
          # NODE_ENV: production
          # PUBLIC_URL: /your-repo-name
          
      - name: Upload artifact
        uses: actions/upload-pages-artifact@v3
        with:
          path: ./build # Default Docusaurus output directory

  deploy:
    environment:
      name: github-pages
      url: ${{ steps.deployment.outputs.page_url }}
    runs-on: ubuntu-latest
    needs: build
    steps:
      - name: Deploy to GitHub Pages
        id: deployment
        uses: actions/deploy-pages@v4
```

## Uygulama Sırasında Sık Karşılaşılan Tuzaklar

Uygulamada birçok sorunla karşılaştık. Başkalarının bunlardan kaçınabilmesi veya en azından önceden çözüm hazırlayabilmesi umuduyla bunları burada paylaşıyorum.

### 1. GitHub Token İzin Sorunları

İşleri ilk kurduğumuzda dağıtım 403 (Yasak) hatasıyla başarısız olmaya devam ediyordu. Çok fazla araştırmadan sonra GitHub'ın varsayılan ayarını bulduk. `GITHUB_TOKEN` Sayfalara yazma izniniz yoktu.

**Çözüm**: Depoda `Settings` -> `Actions` -> `General` -> `Workflow permissions`**"Okuma ve yazma izinleri"** seçeneğini seçtiğinizden emin olun.

### 2. Yanlış Dizin Yolu Oluşturma

Docusaurus yerleşik statik dosyaları `build` varsayılan olarak dizin. Ancak bazı projeler farklı şekilde yapılandırılabilir. Örneğin, React Uygulaması Oluştur varsayılan olarak şu şekildedir: `build`Vite varsayılan olarak `dist`. Actions çıkış dosyalarını bulamadığını bildirirse, çıkış yolunu kontrol edin. `docusaurus.config.js`.

### 3. Alt Yol Sorunları

Deponuz bir kullanıcı ana sayfası değilse, yani `username.github.io`, bunun yerine şöyle bir proje sayfası `username.github.io/project-name`, ardından yapılandırmanız gerekir `baseUrl`.

içinde `docusaurus.config.js`:

```javascript
module.exports = {
  // ...
  url: 'https://hagicode.com', // Your Hagicode URL
  baseUrl: '/', // Deploy from the root path
  // ...
};
```
Bu ayrıntıyı gözden kaçırmak kolaydır. Yanlış yapılandırılmışsa varlık yolları çözümlenemediğinden sayfa boş bir ekran olarak yüklenebilir.

## Sonucun Doğrulanması

Her şeyi yapılandırıp kodu bastıktan sonra GitHub deposunun **Actions** sekmesine gidip gösteriyi izleyebiliriz.

İş akışı çalışırken sarı bir daire göreceksiniz ve yeşile döndüğünde bu başarı anlamına gelir. Kırmızıya dönerse, tıklayın ve günlükleri inceleyin. Çoğu durumda, genellikle bir yazım hatası veya yanlış yol yapılandırması olmak üzere sorunu buradan takip edebilirsiniz.

Derleme başarılı olduktan sonra şu adresi ziyaret edin: `https://<your-username>.github.io/<repository-name>/` ve yeni konuşlandırılmış siteyi göreceksiniz.

## Özet

GitHub Eylemlerini sunarak, HagiCode dokümantasyon sitesi için dağıtımı başarıyla otomatikleştirdik. Bu yalnızca daha önce manuel işlemlere harcanan zamandan tasarruf etmekle kalmaz, aynı zamanda daha da önemlisi sürüm iş akışını standartlaştırır. Artık, hangi ekip üyesinin belgeleri güncellediği önemli değil, değişiklik bir kez birleştirildikten sonra `main`, en son içerik birkaç dakika sonra yayında olacak.

**Temel avantajlar**:
- **Daha yüksek verimlilik**: "Manuel paketleme ve manuel yükleme"den "kod eşittir yayınlama"ya geçtik.
- **Daha az hata**: Manuel işlemler sırasında insan hatası olasılığını ortadan kaldırdık.
- **Daha iyi geliştirici deneyimi**: Geliştiriciler, sıkıcı dağıtım adımlarıyla dikkatlerinin dağılması yerine içerik kalitesine daha fazla odaklanabilir.

CI/CD'yi yapılandırmak ilk başta özellikle tüm izinler ve yol ayrıntılarıyla biraz zahmetli olsa da, uzun vadede büyük getirisi olan tek seferlik bir yatırımdır. Her statik site projesinin benzer bir otomatik iş akışını benimsemesini şiddetle tavsiye ederim.
