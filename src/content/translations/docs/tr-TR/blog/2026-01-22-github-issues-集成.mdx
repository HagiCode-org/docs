## Sıfırdan GitHub Sorunları Oluşturmak Entegrasyon: HagiCode'un Ön Uç Doğrudan Bağlantı Uygulaması

> Bu makale GitHub Sorunlarını HagiCode platformuna entegre etme sürecinin tamamını belgelemektedir. Arka ucu hafif tutarken güvenli OAuth kimlik doğrulaması ve verimli sorun senkronizasyonu elde etmek için "ön uç doğrudan bağlantı + minimum arka uç" mimarisinin nasıl kullanılacağını keşfedeceğiz.

## Arka Plan: Neden GitHub'u Entegre Etmelisiniz?

Yapay zeka destekli bir geliştirme platformu olarak HagiCode'un temel değeri, fikirleri uygulamayla birleştirmede yatmaktadır. Ancak gerçek kullanımda, kullanıcıların HagiCode'da bir Teklifi tamamladıktan sonra, proje takibi için genellikle içeriği GitHub Sorunlarına manuel olarak kopyalamaları gerektiğini gördük.

Bu, birkaç belirgin sorun noktası yaratır:

1.  **Parçalanmış iş akışı**: Kullanıcıların iki sistem arasında geçiş yapması gerekir. Deneyim sorunsuz değildir ve kopyalama ve yapıştırma sırasında önemli bilgiler kolayca kaybolabilir.
2.  **Uygunsuz işbirliği**: Diğer ekip üyeleri GitHub'daki görevleri kontrol etmeye alışkındır ve HagiCode'daki teklif ilerlemesini doğrudan göremezler.
3.  **Tekrarlanan manuel çalışma**: Bir teklif her güncellendiğinde, birisinin ilgili sorunu GitHub'da manuel olarak güncellemesi gerekir; bu da gereksiz bakım maliyeti anlamına gelir.

Bu sorunu çözmek için, HagiCode oturumlarını GitHub depolarına bağlayarak "tek tıklamayla senkronizasyonu" etkinleştiren **GitHub Sorun Entegrasyonu** özelliğini sunmaya karar verdik.

## Teknik Seçim: Ön Uç Doğrudan Bağlantı ve Arka Uç Proxy Karşılaştırması

Entegrasyon yaklaşımını tasarlarken önümüzde iki seçenek vardı: geleneksel "arka uç proxy modeli" ve daha agresif "ön uç doğrudan bağlantı modeli".

### Çözüm Karşılaştırması

Geleneksel **arka uç proxy modelinde**, ön uçtan gelen her isteğin önce arka uçtan geçmesi gerekir, bu da daha sonra GitHub API'yi çağırır. Bu, mantığı merkezileştirir ancak aynı zamanda arka uca da önemli bir yük getirir:

1.  **Şişmiş arka uç**: Özel bir GitHub API istemci sarmalayıcısı yazmamız ve ayrıca karmaşık OAuth durumu makinesini kullanmamız gerekir.
2.  **Jeton riski**: Kullanıcının GitHub jetonunun arka uç veritabanında saklanması gerekir. Şifrelemeyle bile bu güvenlik yüzeyini artırır.
3.  **Geliştirme maliyeti**: Belirteçleri depolamak için veritabanı geçişlerine ve sürdürmek için ek bir senkronizasyon hizmetine ihtiyacımız olacak.

**Ön uç doğrudan bağlantı modeli** çok daha hafiftir. Bu yaklaşımda, arka ucu yalnızca en hassas "gizli değişim" adımı (OAuth geri çağırma) için kullanırız. Tokenı aldıktan sonra doğrudan tarayıcının belleğinde saklarız. `localStorage`. Sorun oluşturma ve yorumları güncelleme gibi daha sonraki işlemler, HTTP üzerinden doğrudan ön uçtan GitHub'a gönderilir.

| Karşılaştırma Boyutu | Arka Uç Proxy Modeli | Ön Uç Doğrudan Bağlantı Modeli |
| :--- | :--- | :--- |
| **Arka uç karmaşıklığı** | Tam bir OAuth hizmeti ve GitHub API istemcisi gerektirir | Yalnızca bir OAuth geri çağırma uç noktasına ihtiyaç duyar |
| **Jeton yönetimi** | Sızıntı riskiyle birlikte şifrelenip veritabanında saklanmalıdır | Tarayıcıda saklanır ve yalnızca kullanıcı tarafından görülebilir |
| **Uygulama maliyeti** | Veritabanı geçişleri ve çoklu hizmet geliştirme gerektirir | Öncelikle ön uç çalışması |
| **Kullanıcı deneyimi** | Merkezi mantık ancak sunucu gecikmesi biraz daha yüksek olabilir | Doğrudan GitHub etkileşimi ile son derece hızlı yanıt |

Hızlı entegrasyon ve minimum arka uç değişikliği istediğimiz için **sonuçta "ön uç doğrudan bağlantı modelini" seçtik**. Bu, tarayıcıya "geçici geçiş" vermek gibidir. Geçişi aldıktan sonra tarayıcı, her seferinde arka uç yöneticisinin onayını istemeden GitHub'daki işleri kendi başına halledebilir.

## Temel Tasarım: Veri Akışı ve Güvenlik

Mimariye karar verdikten sonra spesifik veri akışını tasarlamamız gerekiyordu. Senkronizasyon sürecinin özü, jetonun güvenli bir şekilde nasıl elde edileceği ve verimli bir şekilde nasıl kullanılacağıdır.

### Genel Mimari Şeması

Sistemin tamamı üç role ayrılabilir: tarayıcı (ön uç), HagiCode arka uç ve GitHub.

```text
+--------------+        +--------------+        +--------------+
| Frontend     |        |   Backend    |        |   GitHub     |
| React        |        |   ASP.NET    |        |   REST API   |
|              |        |              |        |              |
|  +--------+  |        |              |        |              |
|  | OAuth  |--+--------> /callback    |        |              |
|  | Flow   |  |        |              |        |              |
|  +--------+  |        |              |        |              |
|              |        |              |        |              |
|  +--------+  |        |  +--------+  |        |  +--------+  |
|  | GitHub |  +------------> Session |  +----------> Issues |  |
|  | API    |  |        |  |Metadata|  |        |  |        |  |
|  | Direct |  |        |  +--------+  |        |  +--------+  |
|  +--------+  |        |              |        |              |
+--------------+        +--------------+        +--------------+
```

**Önemli nokta şudur**: OAuth'ta yalnızca küçük bir adımın (kodu bir jetonla değiştirme) arka uçtan geçmesi gerekir. Bundan sonra, ağır işler (sorun yaratma) doğrudan ön uç ile GitHub arasında gerçekleştirilir.

### Ayrıntılı Olarak Senkronizasyon Veri Akışı

Kullanıcı HagiCode kullanıcı arayüzünde "GitHub'a Eşitle" düğmesini tıkladığında bir dizi karmaşık eylem gerçekleşir:

```text
User clicks "Sync to GitHub"
         │
         ▼
1. Frontend checks localStorage for the GitHub token
         │
         ▼
2. Format issue content (convert the Proposal into Markdown)
         │
         ▼
3. Frontend directly calls the GitHub API to create/update the issue
         │
         ▼
4. Call the HagiCode backend API to update Session.metadata (store issue URL and other info)
         │
         ▼
5. Backend broadcasts the SessionUpdated event via SignalR
         │
         ▼
6. Frontend receives the event and updates the UI to show the "Synced" state
```

### Güvenlik Tasarımı

Üçüncü taraf hizmetleri entegre ederken güvenlik her zaman en önemli önceliktir. Aşağıdaki değerlendirmeleri yaptık:

1.  **CSRF saldırılarına karşı savunma**: Rastgele bir `state` OAuth yönlendirmesi sırasında parametreyi kullanın ve bunu içinde saklayın. `sessionStorage`. Sahte istekleri önlemek için geri aramadaki durumu kesinlikle doğrulayın.
2.  **İzole jeton depolama**: Jeton yalnızca tarayıcının `localStorage`. Aynı Köken Politikasını kullanarak, yalnızca HagiCode komut dosyaları okuyabilir, böylece kullanıcıları etkileyen sunucu tarafı veritabanı sızıntısı riski ortadan kaldırılır.
3.  **Hata sınırları**: Yaygın GitHub API hataları (401 süresi dolmuş jeton, 422 doğrulama hatası ve 429 hız sınırlaması gibi) için özel bir işleme tasarladık, böylece kullanıcılar kolay geri bildirim alabilirler.

### 2. Ön Uç OAuth Akışı

Bu bağlantının giriş noktasıdır. Standart Yetkilendirme Kodu Akışını kullanıyoruz.

```typescript
// src/HagiCode.Client/src/services/githubOAuth.ts

// Generate the authorization URL and redirect
export async function generateAuthUrl(): Promise<string> {
  const state = generateRandomString(); // Generate a random string for CSRF protection
  sessionStorage.setItem('hagicode_github_state', state);
  
  const params = new URLSearchParams({
    client_id: clientId,
    redirect_uri: window.location.origin + '/settings?tab=github&oauth=callback',
    scope: ['repo', 'public_repo'].join(' '),
    state: state,
  });
  
  return `https://github.com/login/oauth/authorize?${params.toString()}`;
}

// Handle the code-to-token exchange on the callback page
export async function exchangeCodeForToken(code: string, state: string): Promise<GitHubToken> {
  // 1. Validate state to prevent CSRF
  const savedState = sessionStorage.getItem('hagicode_github_state');
  if (state !== savedState) throw new Error('Invalid state parameter');

  // 2. Call the backend API to exchange the token
  // Note: this must go through the backend because ClientSecret cannot be exposed to the frontend
  const response = await fetch('/api/GitHubOAuth/callback', {
    method: 'POST',
    headers: { 'Content-Type': 'application/json' },
    body: JSON.stringify({ code, state, redirectUri: window.location.origin + '/settings?tab=github&oauth=callback' }),
  });

  if (!response.ok) throw new Error('Failed to exchange token');
  
  const token = await response.json();
  
  // 3. Save into LocalStorage
  saveToken(token);
  return token;
}
```

### 3. GitHub API İstemci Paketleyicisi

Tokenı aldıktan sonra GitHub API'sini çağırmak için sağlam bir araca ihtiyacımız var.

```typescript
// src/HagiCode.Client/src/services/githubApiClient.ts

const GITHUB_API_BASE = 'https://api.github.com';

// Core request wrapper
async function githubApi<T>(endpoint: string, options: RequestInit = {}): Promise<T> {
  const token = localStorage.getItem('gh_token');
  if (!token) throw new Error('Not connected to GitHub');
  
  const response = await fetch(`${GITHUB_API_BASE}${endpoint}`, {
    ...options,
    headers: {
      ...options.headers,
      Authorization: `Bearer ${token}`,
      Accept: 'application/vnd.github.v3+json', // Specify the API version
    },
  });
  
  // Error handling logic
  if (!response.ok) {
    if (response.status === 401) throw new Error('GitHub token expired, please reconnect');
    if (response.status === 403) throw new Error('No permission to access this repository or rate limit exceeded');
    if (response.status === 422) throw new Error('Issue validation failed, the title may be duplicated');
    throw new Error(`GitHub API Error: ${response.statusText}`);
  }
  
  return response.json();
}

// Create issue
export async function createIssue(owner: string, repo: string, data: { title: string, body: string, labels: string[] }) {
  return githubApi(`/repos/${owner}/${repo}/issues`, {
    method: 'POST',
    body: JSON.stringify(data),
  });
}
```

### 4. İçerik Formatlama ve Senkronizasyon

Son adım, HagiCode oturum verilerini GitHub sayısının formatına dönüştürmektir. Biraz çeviri işine benziyor.

```typescript
// Convert a Session object into a Markdown string
function formatIssueForSession(session: Session): string {
  let content = `# ${session.title}\n\n`;
  content += `**> HagiCode Session:** #${session.code}\n`;
  content += `**> Status:** ${session.status}\n\n`;
  content += `## Description\n\n${session.description || 'No description provided.'}\n\n`;
  
  // If this is a Proposal session, add extra fields
  if (session.type === 'proposal') {
    content += `## Chief Complaint\n\n${session.chiefComplaint || ''}\n\n`;
    // Add a deep link so users can jump back from GitHub to HagiCode
    content += `---\n\n**[View in HagiCode](hagicode://sessions/${session.id})**\n`;
  }
  
  return content;
}

// Main logic when clicking the sync button
const handleSync = async (session: Session) => {
  try {
    const repoInfo = parseRepositoryFromUrl(session.repoUrl); // Parse the repository URL
    if (!repoInfo) throw new Error('Invalid repository URL');

    toast.loading('Syncing to GitHub...');
    
    // 1. Format content
    const issueBody = formatIssueForSession(session);
    
    // 2. Call API
    const issue = await githubApiClient.createIssue(repoInfo.owner, repoInfo.repo, {
      title: `[HagiCode] ${session.title}`,
      body: issueBody,
      labels: ['hagicode', 'proposal', `status:${session.status}`],
    });
    
    // 3. Update Session Metadata (save the issue link)
    await SessionsService.patchApiSessionsSessionId(session.id, {
      metadata: {
        githubIssue: {
          owner: repoInfo.owner,
          repo: repoInfo.repo,
          issueNumber: issue.number,
          issueUrl: issue.html_url,
          syncedAt: new Date().toISOString(),
        }
      }
    });

    toast.success('Synced successfully!');
  } catch (err) {
    console.error(err);
    toast.error('Sync failed, please check your token or network');
  }
};
```

## Özet ve Görünüm

Bu "ön uç doğrudan bağlantı" yaklaşımıyla, mümkün olan en az arka uç koduyla kusursuz GitHub Issues entegrasyonunu sağladık.

### Temel kazanımlar
1.  **Yüksek geliştirme verimliliği**: Arka uç değişiklikleri minimum düzeydedir; esas olarak bir ekstra veritabanı alanı ve basit bir OAuth geri çağırma uç noktasıdır. Çoğu mantık ön uçta tamamlanır.
2.  **Güçlü güvenlik**: Belirteç sunucu veritabanından geçmez, bu da sızıntı riskini azaltır.
3.  **Mükemmel kullanıcı deneyimi**: İstekler doğrudan ön uçtan başlatıldığından yanıt hızı yüksektir ve arka uç yönlendirmeye gerek yoktur.

### Dikkat edilmesi gerekenler
Gerçek dağıtım sırasında akılda tutulması gereken birkaç tuzak vardır:
- **OAuth Uygulama ayarları**: Doğru bilgileri girmeyi unutmayın `Authorization callback URL` GitHub OAuth Uygulaması ayarlarınızda (genellikle `http://localhost:3000/settings?tab=github&oauth=callback`).
- **Hız sınırları**: GitHub API, kimliği doğrulanmamış istekleri oldukça sıkı bir şekilde sınırlar, ancak bir jetonla kota genellikle yeterlidir (5000 istek/saat).
- **URL ayrıştırma**: Kullanıcılar her türlü veri havuzu URL'sini girer; bu nedenle normal ifadenizin eşleşebildiğinden emin olun `.git` sonekler, SSH formatları ve benzer durumlar.

### Gelecekteki geliştirmeler
Mevcut özellik hala tek yönlü senkronizasyondur (HagiCode -> GitHub). Gelecekte GitHub Webhook'lar aracılığıyla iki yönlü senkronizasyon uygulamayı planlıyoruz. Örneğin GitHub'da bir konu kapatılırsa HagiCode tarafındaki oturum durumu da otomatik olarak güncellenebilir. Bu, arka uçta bir web kancası uç noktası ortaya çıkarmamızı gerektirecek ve bu da ilginç bir sonraki adım olacak.

Bu makalenin size kendi üçüncü taraf entegrasyon geliştirmeniz için biraz ilham vereceğini umuyoruz. Sorularınız varsa, tartışmak üzere bir konu açmaktan çekinmeyin. [HagiCode GitHub](https://github.com/HagiCode-org/site).
