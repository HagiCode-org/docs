Geleneksel yaklaşım, dosyaları toplu olarak manuel olarak hazırlamak, bunları birer birer işlemek ve işleme mesajları yazmaktır. Bu süreç hem zaman alıcıdır hem de hataya açıktır. ### Geleneksel Git Taahhütlerinin Sorunlu Noktaları

Bir sürüm kontrol sistemi olarak Git, geliştiricilere güçlü kod yönetimi yetenekleri sağlar. Ancak gerçek dünya kullanımında taahhüt, geliştirme iş akışında sıklıkla bir darboğaz haline gelir:

1. **Manuel gruplama zaman alıcıdır**: Çok sayıda dosya değişikliği olduğunda geliştiricilerin her dosyayı tek tek incelemesi ve hangi değişikliklerin aynı özelliğe ait olduğuna karar vermesi gerekir. Bu çok fazla zihinsel çaba gerektirir.
2. **Tutarsız taahhüt mesajı kalitesi**: Geleneksel Taahhüt spesifikasyonlarına uygun taahhüt mesajları yazmak deneyim ve beceri gerektirir ve yeni başlayanlar genellikle standart olmayan taahhütler üretir.
3. **Karmaşık çoklu depo yönetimi**: Bir monorepo ortamında, farklı depolar arasında geçiş yapmak operasyonel karmaşıklığı artırır.
4. **Kesintili iş akışı**: Kod işlemek, düşünce akışınızı kesintiye uğratır ve kodlama verimliliğine zarar verir.

Bu sorunlar özellikle büyük projelerde ve işbirliğine dayalı ekip ortamlarında belirgindir. İyi bir geliştirme aracı, geliştiricilerin hantal bir taahhüt iş akışında takılıp kalmak yerine temel kodlama çalışmalarına odaklanmasına olanak sağlamalıdır.

### Teknik Zorluklar

AI Compose Commit'i uygularken çeşitli teknik zorluklarla karşılaştık:

1. **Dosya anlamsal anlayışı**: Yapay zekanın, dosya değişiklikleri arasındaki anlamsal ilişkileri anlaması ve hangi dosyaların aynı işlevsel modüle ait olduğuna karar vermesi gerekir. Bu, dosya içeriğinin, dizin yapısının ve değişiklik bağlamının derinlemesine analizini gerektirir.

5. **Hata yönetimi ve geri alma**: Bir işlem başarısız olursa, halihazırda yürütülen işlemler nasıl ele alınmalıdır? Zaten hazırlanmış dosyaların geri alınması gerekiyor mu?

6. **Taahhüt mesajı tutarlılığı**: Oluşturulan taahhüt mesajlarının projenin mevcut stiliyle eşleşmesi ve geçmiş taahhütlerle tutarlı kalması gerekir.

## Çözüm

### Genel Mimari Tasarım

AI Compose Commit'i uygulamak için iyi ölçeklenebilirlik ve sürdürülebilirlik sağlayan katmanlı bir mimariyi benimsedik:

#### 1. API Katmanı (Web Katmanı)

**GitController** şunları sağlar: `POST /api/git/auto-compose-commit` giriş noktası olarak bitiş noktası. Kullanıcı deneyimini optimize etmek için, eşzamansız bir ateşle ve unut modelini benimsedik:

- İstemci bir istek gönderdikten sonra sunucu hemen HTTP 202 Kabul Edildi sonucunu döndürür
- Gerçek AI işleme arka planda eşzamansız olarak çalışır
- İşleme tamamlandığında istemci SignalR aracılığıyla bilgilendirilir.

Bu tasarım, yapay zekanın işlemesi birkaç dakika sürse bile kullanıcıların anında yanıt almasını ve sistemin donduğunu hissetmemesini sağlar.

#### 2. Uygulama Hizmeti Katmanı (Uygulama Katmanı)

**GitAppService** temel iş mantığından sorumludur:

- Depo tespiti: bir monorepoda çoklu depo yönetimini destekler
- Kilit yönetimi: eş zamanlı işlemlerden kaynaklanan çakışmaları önler
- Dosya hazırlama koordinasyonu: AI işleme akışıyla etkileşime girer
- Hatayı geri alma: arızalar meydana geldiğinde durumu geri yükler

#### 3. Dağıtılmış Bilgi İşlem Katmanı (Orleans Tahılları)

**AIGrain**, yapay zeka operasyonları için temel yürütme birimi olarak hizmet eder. Şunu uygular: `AutoComposeCommitAsync` gelen yöntem `IAIGrain` arayüz:

### Temel Uygulama Mantığı

#### Dosya İçeriği Oluşturma

Yapay zekanın akıllı kararlar alabilmesi için önce her dosyanın durumunu anlaması gerekiyor. Dosya içeriğini şu şekilde oluşturuyoruz: `BuildFileChangesXml` yöntem:

```csharp
/// <summary>
/// Build an XML representation of file changes to provide the AI with complete file context information
/// </summary>
/// <param name="stagedFiles">List of staged files, including file path, status, and old path (for rename operations)</param>
/// <returns>A formatted XML string containing metadata for all files</returns>
private static string BuildFileChangesXml(GitFileStatusDto[] stagedFiles)
{
    var sb = new StringBuilder();
    sb.AppendLine("<files>");

    foreach (var file in stagedFiles)
    {
        sb.AppendLine("  <file>");
        // Use XML escaping to ensure special characters do not break the XML structure
        sb.AppendLine($"    <path>{System.Security.SecurityElement.Escape(file.Path)}</path>");
        sb.AppendLine($"    <status>{System.Security.SecurityElement.Escape(file.Status)}</status>");

        // Handle file rename scenarios and record the old path so the AI can understand change relationships
        if (!string.IsNullOrEmpty(file.OldPath))
        {
            sb.AppendLine($"    <oldPath>{System.Security.SecurityElement.Escape(file.OldPath)}</oldPath>");
        }

        sb.AppendLine("  </file>");
    }

    sb.AppendLine("</files>");
    return sb.ToString();
}
```

Bu XML tabanlı içerik, yeniden adlandırma işlemleri için dosya yollarını, durumları ve eski yolları içerir ve yapay zekaya eksiksiz meta veriler sağlar. Yapılandırılmış bir XML formatı ile yapay zekanın her dosyanın durumunu ve türünü doğru bir şekilde anlayabilmesini sağlıyoruz.

#### Yapay Zeka İzin Yönetimi

Yapay zekanın Git işlemlerini doğrudan yürütmesine izin vermek için kapsamlı araç izinlerini yapılandırdık:

```csharp
// Define the set of tools the AI can use, including file operations and Git command execution permissions
// Read/Write/Edit: file reading, writing, and editing capabilities
// Bash(git:*): permission to execute all Git commands
// Other Bash commands: used to inspect file contents and directory structure so the AI can understand context
var allowedTools = new[]
{
    "Read", "Write", "Edit",
    "Bash(git:*)", "Bash(cat:*)", "Bash(ls:*)", "Bash(find:*)",
    "Bash(grep:*)", "Bash(head:*)", "Bash(tail:*)", "Bash(wc:*)"
};

// Build the complete AI request object

{

Bu, özellik tasarımının merkezinde yer alır, ancak aynı zamanda kötüye kullanımı önlemek için girişin sıkı bir şekilde doğrulanmasını da gerektirir. HagiCode'un üretim dağıtımında, arka uç parametre doğrulaması ve günlük izleme yoluyla bu mekanizmanın güvenliğini sağladık.

#### Sonuç Ayrıştırmanın Kaydedilmesi

Yapay zeka, yürütmeyi tamamladıktan sonra yapılandırılmış sonuçları döndürür. Uyumluluğu sağlamak için ikili bir ayrıştırma stratejisi uyguladık:

```csharp
/// <summary>
/// Parse commit execution results returned by the AI, supporting both delimiter format and regex format
/// </summary>
/// <param name="aiResponse">Raw response content returned by the AI</param>
/// <returns>A parsed list of commit results, where each result includes the commit hash and execution status</returns>
private List<CommitResultDto> ParseCommitExecutionResults(string aiResponse)
{
    var results = new List<CommitResultDto>();

    // Prefer delimiter-based parsing (new format), which is more explicit and reliable
    if (aiResponse.Contains("---"))
    {
        logger.LogDebug("Using delimiter-based parsing for AI response");
        results = ParseDelimitedFormat(aiResponse);

        if (results.Count > 0)
        {
            return results; // Successfully parsed, return the results directly
        }

        logger.LogWarning("Delimiter-based parsing produced no results, falling back to regex");
    }
    else
    {
        logger.LogDebug("No delimiter found, using legacy regex-based parsing");
    }

    // Fall back to regex parsing (old format) to ensure backward compatibility
    return ParseLegacyFormat(aiResponse);
}
```

Sınırlayıcı formatı şunları kullanır: `---` taahhütleri ayırmak, yapıyı netleştirmek ve ayrıştırılmasını kolaylaştırmak için:

```
---
Commit 1: abc123def456
feat(auth): add user login functionality

Implement JWT-based authentication with login form and API endpoints.

Co-Authored-By: Hagicode <noreply@hagicode.com>
---
Commit 2: 789ghi012jkl
docs(readme): update installation instructions

Add new setup steps for Docker environment.

Co-Authored-By: Hagicode <noreply@hagicode.com>
---
```

Bu format, ayrıştırmayı basit ve güvenilir hale getirirken aynı zamanda insanların okumasını da kolaylaştırır.

Eşzamanlı işlemlerin neden olduğu durum çakışmalarını önlemek için bir depo kilitleme mekanizması uyguladık:

```csharp
// Acquire the repository lock to prevent concurrent operations
// Parameter notes:
// - fullPath: full repository path, used to identify different repository instances
// - requestedBy: requester identifier, used for tracking and logging
await _autoComposeLockService.AcquireLockAsync(fullPath, requestedBy);

try

    // Execute the AI Compose Commit operation

### SignalR Gerçek Zamanlı Bildirimler

İşleme tamamlandıktan sonra sistem, SignalR aracılığıyla ön uca bir bildirim gönderir:

```csharp
/// <summary>
/// Send a notification when automatic commit composition is complete
/// </summary>
/// <param name="projectId">Project identifier, used to route the notification to the correct client</param>
/// <param name="totalCount">Total number of commits, including successes and failures</param>
/// <param name="successCount">Number of successful commits</param>
/// <param name="failureCount">Number of failed commits</param>
/// <param name="success">Whether the overall operation succeeded</param>
/// <param name="error">Error message (if the operation failed)</param>
private async Task SendAutoComposeCommitNotificationAsync(
    string projectId,
    int totalCount,
    int successCount,
    int failureCount,
    bool success,
    string? error)
{
    try
    {
        // Build the notification DTO containing detailed execution results
        var notification = new AutoComposeCommitCompletedDto
        {
            ProjectId = projectId,
            TotalCount = totalCount,
            SuccessCount = successCount,
            FailureCount = failureCount,
            Success = success,
            Error = error
        };

        // Broadcast the notification to all connected clients through the SignalR Hub
        await messageService.SendAutoComposeCommitCompletedAsync(notification);

        logger.LogInformation(
            "Auto compose commit notification sent for project {ProjectId}: {SuccessCount}/{TotalCount} succeeded",
            projectId, successCount, totalCount);
    }
    catch (Exception ex)
    {
        // Log notification errors without affecting the main operation flow
        // A notification failure should not cause the entire operation to fail
        logger.LogError(ex, "Failed to send auto compose commit notification for project {ProjectId}", projectId);
    }
}
```

Ön uç bildirimi aldıktan sonra kullanıcı deneyimini iyileştirerek, işlemin başarılı mı yoksa başarısız mı olduğunu gösterecek şekilde kullanıcı arayüzünü güncelleyebilir. Bu gerçek zamanlı geri bildirim mekanizması, işlemin ne zaman biteceğini ve sonucun ne olacağını açıkça görebilen HagiCode kullanıcılarından güçlü geri bildirimler aldı.

### Hızlı Mühendislik Tasarımı

Yapay zeka davranışı tamamen istem tarafından belirlenir, bu nedenle Otomatik Oluşturma Taahhüdü için bilgi istemi şablonunu dikkatli bir şekilde tasarladık. Örnek olarak Çin versiyonunu ele alalım (`auto-compose-commit.zh-CN.hbs`):

#### Etkileşimsiz Mod Desteği

İstemin başında, CI/CD ve otomasyon komut dosyaları için kritik bir gereksinim olan etkileşimli olmayan yürütme moduna yönelik desteği açıkça beyan ediyoruz:

```handlebars
**Important Note**: This prompt may run in a non-interactive environment (such as CI/CD or automation scripts).

**Non-Interactive Mode**:
- Do not use AskUserQuestion or any interactive tools
- When user input is required:
  - Use sensible defaults (for example, use feat as the commit type)
  - Skip optional confirmation steps
  - Record any assumptions made
```

Bu tasarım, AI Compose Commit'in yalnızca etkileşimli IDE ortamlarında kullanılmasını değil aynı zamanda tam otomatik bir taahhüt iş akışı sağlamak için CI/CD ardışık düzenlerine entegre edilmesini de sağlar.

#### Şube Koruma Mekanizması

Yapay zekanın tehlikeli işlemler yürütmesini önlemek için istemlere katı dal koruma kuralları ekledik:

```handlebars
**Branch Protection**:
- Do not perform any branch switching operations (git checkout, git switch)
- All `git commit` commands must run on the current branch
- Do not create, delete, or rename branches
- Do not modify untracked files or unstaged changes
- If branch switching is required to complete the operation, return an error instead of executing it
```

Bu kurallar, yapay zekanın araç kullanım kapsamını kısıtlayarak operasyonel güvenliği sağlar. HagiCode'un pratik testinde bu kısıtlamaların etkinliğini doğruladık: Yapay zeka, dal değişimi gerektirecek bir durumla karşılaştığında, tehlikeli eylemde bulunmak yerine güvenli bir şekilde hata döndürüyor.

#### Akıllı Gruplandırma Karar Ağacı

Bilgi istemi, dosya gruplamasına ilişkin karar mantığını ayrıntılı olarak tanımlar:

```handlebars
**File Grouping Decision Tree**:
├── Is it a configuration file (package.json, tsconfig.json, .env, etc.)?
│   ├── Yes -> separate commit (type: chore or build)
│   └── No -> continue
├── Is it a documentation file (README.md, *.md, docs/**)?
│   ├── Yes -> separate commit (type: docs)
│   └── No -> continue
├── Is it related to the same feature?
│   ├── Yes -> merge into the same commit
│   └── No -> commit separately
└── Is it a cross-module change?
    ├── Yes -> group by module
    └── No -> group by feature
```

Bu karar ağacı, yapay zekaya net bir gruplama mantığı vererek oluşturulan taahhütlerin anlamsal olarak makul kalmasını sağlar. Gerçek HagiCode kullanımında, bu karar ağacının yaygın senaryoların büyük çoğunluğunu karşılayabildiğini ve gruplandırma sonuçlarının geliştirici beklentileriyle eşleştiğini gördük.

#### Tarihsel Format Tutarlılık Analizi

Taahhüt mesajlarının proje geçmişiyle tutarlı olmasını sağlamak için istem, yapay zekanın oluşturmadan önce yakın zamandaki taahhüt geçmişini analiz etmesini gerektirir:

```handlebars
**Historical Format Consistency**: Before generating commit messages, you **must** analyze the current repository's commit history to match the existing style.

1. Use `git log -n 15 --pretty=format:"%H|%s|%b%n---%n"` to get the recent commit history
2. Analyze the commits to identify:
   - Structural patterns: does the project use multi-paragraph messages? Are there `Changes:` or `Capabilities:` sections?
   - Language patterns: are commit messages in English, Chinese, or mixed?
   - Common types: which commit types are most often used (`feat`, `fix`, `docs`, etc.)?
   - Special formatting: are there `Co-Authored-By` lines? Any other project-specific conventions?
3. Generate commit messages that follow the detected patterns
```

Bu analiz, yapay zeka tarafından oluşturulan taahhüt mesajlarının yersiz görünmemesini, bunun yerine stilistik olarak projenin geçmişiyle uyumlu kalmasını sağlar. HagiCode'un çok dilli projelerinde bu özellik özellikle önemlidir çünkü taahhüt geçmişine dayalı olarak uygun dili ve formatı otomatik olarak seçebilir.

#### Ortak Yazar-Gereksinime Göre

Her taahhütte Ortak Yazar bilgileri bulunmalıdır:

```handlebars
**Important**: Every commit must include Co-Authored-By information
- Use the following format: `git commit -m "type(scope): subject" -m "" -m "Co-Authored-By: Hagicode <noreply@hagicode.com>"`

```

Bu yalnızca katkı uyumluluğu için değil, aynı zamanda yapay zeka destekli taahhüt geçmişini izlemek için de geçerlidir. HagiCode, yapay zeka tarafından oluşturulan tüm taahhütlerin net bir kaynak işaretçisi taşımasını sağlamak için bunu zorunlu bir kural olarak ele alır.

### İş Akışı İzlenecek Yolu

AI Compose Commit iş akışının tamamı aşağıdaki gibidir:

1. **Kullanıcı tetikleyicisi**: Kullanıcı, Git Durumu panelinde veya Hızlı Eylemler Bölgesinde "AI Otomatik Oluşturma Taahhüdü" düğmesini tıklar.
2. **API isteği**: Ön uç, POST isteğini `/api/git/auto-compose-commit` uç nokta.
3. **Anında yanıt**: Sunucu, işlemin bitmesini beklemeden HTTP 202 Kabul Edildi sonucunu döndürür.
4. **Arka planda işleme**:
   - GitAppService depo kilidini alıyor
   - Aramalar `AIGrain.AutoComposeCommitAsync`
   - Dosya bağlamı XML'ini oluşturur
   - Yapay zekanın taahhütleri analiz edebilmesi ve gerçekleştirebilmesi için yapay zeka istemini yürütür
5. **Yapay zeka yürütme**:
   - Tüm düzenlenmemiş değişiklikleri elde etmek için Git komutlarını kullanır
   - Değişikliklerin doğasını anlamak için dosya içeriğini okur
   - Dosyaları anlamsal ilişkiye göre gruplandırır
   - Yürütür `git add` ve `git commit` her grup için
6. **Sonuç ayrıştırma**: Yapay zekanın döndürdüğü yürütme sonuçlarını ayrıştırır.
7. **Bildirim teslimi**: Ön ucu SignalR aracılığıyla bilgilendirir.
8. **Kilit açma**: İşlem başarılı olsa da başarısız olsa da depo kilidini serbest bırakır.

Bu iş akışı, kullanıcıların işlemi başlattıktan hemen sonra yapay zekanın bitmesini beklemeden diğer işlerine devam edebilmeleri için tasarlandı. HagiCode kullanıcılarından gelen geri bildirimler, bu eşzamansız işleme modelinin iş akışı deneyimini büyük ölçüde iyileştirdiğini gösteriyor.

## En İyi Uygulamalar

### AI Oluşturma Taahhüdünü Kullanmaya İlişkin En İyi Uygulamalar

#### 1. Doğru Anları Seçin

AI Compose Commit, aşağıdaki senaryolar için en uygunudur:

- Bir iş gününün sonunda, birçok dosyadaki değişiklikleri tek bir toplu işte işlemeniz gerektiğinde
- Yeniden düzenleme işleminden sonra, birkaç ilgili dosyanın ayrı ayrı işlenmesi gerektiğinde
- Bir özellik tamamlandıktan sonra ilgili değişikliklerin taahhütler halinde gruplanması gerektiğinde

Aşağıdaki senaryolar için uygun değildir:

- Tek bir dosya için hızlı taahhütler (normal bir taahhüt daha hızlıdır)
- Taahhüt içeriği üzerinde hassas kontrol gerektiren senaryolar
- İnsan incelemesi gerektiren hassas bilgiler içeren taahhütler

#### 2. Yapay Zeka Tarafından Oluşturulan Taahhütleri Gözden Geçirin

Yapay zeka destekli akıllı gruplama güçlü olsa da geliştiricilerin yine de oluşturulan taahhütleri incelemesi gerekir:

- Gruplandırmanın beklentilerle eşleşip eşleşmediğini kontrol edin
- Taahhüt mesajlarının doğruluğunu doğrulayın
- Hiçbir dosyanın atlanmadığını veya yanlış şekilde dahil edilmediğini doğrulayın

Mantıksız bir gruplama bulursanız şunu kullanabilirsiniz: `git reset --soft HEAD~N` geri almak ve yeniden gruplandırmak için. HagiCode'un deneyimi, yapay zeka gruplaması akıllı olsa bile, özellikle önemli özellik taahhütleri için manuel incelemenin hâlâ değerli olduğunu gösteriyor.

#### 3. Proje Standartlarına Uyum Sağlayın

Projenizin Git yapılandırmasının Geleneksel Taahhütleri desteklediğinden emin olun:

```bash
# Install commitlint
npm install -g @commitlint/cli @commitlint/config-conventional

# Configure commitlint
echo "module.exports = {extends: ['@commitlint/config-conventional']}" > commitlint.config.js
```

Bu, CI/CD iş akışlarında taahhüt mesajı formatını doğrulamanıza olanak tanır ve bunun AI Compose Commit tarafından oluşturulan formatla uyumlu kalmasını sağlar.
