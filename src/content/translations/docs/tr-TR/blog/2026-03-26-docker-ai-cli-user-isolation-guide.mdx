## AI CLI Araçlarını Docker Konteynerlerinde Çalıştırma: Kullanıcı Yalıtımı ve Kalıcı Birimler İçin Pratik Bir Kılavuz

> Claude Code, Codex ve OpenCode gibi yapay zeka kodlama araçlarını kapsayıcı ortamlara entegre etmek kulağa basit geliyor ancak her yerde gizli karmaşıklıklar var. Bu makale, HagiCode projesinin Docker dağıtımlarındaki kullanıcı izinleri, yapılandırma kalıcılığı ve sürüm yönetimi dahil temel zorlukları nasıl çözdüğüne derinlemesine bir bakış sunuyor; böylece sık karşılaşılan tuzaklardan kaçınabilirsiniz.

## Arka plan

Yapay zeka kodlama CLI araçlarını Docker kapsayıcılarının içinde çalıştırmaya karar verdiğimizde muhtemelen en sezgisel düşünce şuydu: "Kapsayıcılar yalnızca root değil mi? Neden her şeyi doğrudan kurup işlemin tamamlandığını söylemiyoruz?" Gerçekte, görünüşte basit olan bu fikir, çözülmesi gereken birçok temel sorunu gizliyor.

Öncelikle **güvenlik kısıtlamaları ilk engeldir**. Örnek olarak Claude CLI'yi ele alalım: kök kullanıcı olarak çalışmayı açıkça yasaklıyor. Bu zorunlu bir güvenlik kontrolüdür ve root tespit edilirse başlamayı reddeder. Kullanıcıları şu şekilde değiştiremez miyim diye düşünebilirsiniz. `USER` direktif? Bu o kadar basit değil. Kabın içindeki root olmayan kullanıcı ile ana makinedeki kullanıcı izinleri arasında hâlâ bir eşleme sorunu var.

İkincisi, **durumun devamlılığı ikinci tuzaktır**. Claude Code oturum açmayı gerektirir, Codex'in kendi yapılandırması vardır ve OpenCode'un ayrıca bir önbellek dizini vardır. Konteyner her yeniden başlatıldığında her şeyi yeniden yapılandırmanız gerekiyorsa, "otomasyon" fikrinin tamamı anlamını kaybeder. Konteynerin yaşam döngüsünün ötesinde de devam etmek için bu konfigürasyonlara ihtiyacımız var.

Üçüncü sorun **izin tutarlılığıdır**. Kapsayıcının içindeki işlemler, ana bilgisayar kullanıcısı tarafından oluşturulan yapılandırma dosyalarına erişebilir mi? UID/GID uyumsuzlukları genellikle dosya izin hatalarına neden olur ve bu, gerçek dağıtımlarda oldukça yaygındır.

Bu sorunlar birbirinden bağımsız görünebilir ancak pratikte birbiriyle sıkı sıkıya bağlantılıdır. HagiCode'un geliştirilmesi sırasında yavaş yavaş pratik bir çözüm üzerinde çalıştık. Daha sonra teknik ayrıntıları ve bu tuzaklardan öğrenilen dersleri paylaşacağım.

## Neden sadece root kullanamıyoruz?

Burada yaygın bir yanlış anlama var: Docker kapsayıcıları varsayılan olarak root olarak çalışıyor, öyleyse neden araçları sadece root olarak yüklemiyorsunuz? Eğer bu şekilde düşünüyorsanız, Claude CLI size hemen aksini öğretecektir.

```bash
# Run Claude CLI directly as root? No.
docker run --rm -it --user root myimage claude
# Output: Error: This command cannot be run as root user
```

Bu, Claude CLI'de katı bir güvenlik kısıtlamasıdır. Bunun nedeni basit: Bu CLI araçları, API belirteçleri, yerel önbellekler ve hatta kullanıcı tarafından yazılan komut dosyaları dahil olmak üzere hassas kullanıcı yapılandırmasını okur ve yazar. Bunları kök ayrıcalıklarıyla çalıştırmak çok fazla risk getirir.

Dolayısıyla şu soru ortaya çıkıyor: Konteyner yönetimini esnek tutarken CLI'nin güvenlik gereksinimlerini nasıl karşılayabiliriz? Bu konudaki düşünce şeklimizi değiştirmemiz gerekiyor: **çalışma zamanında kullanıcıları değiştirmek yerine, görüntü oluşturma aşamasında özel bir kullanıcı oluşturun**.

## Özel bir kullanıcı oluşturmak: yalnızca bir adı değiştirmekten daha fazlası

Tek bir eklemeyi düşünebilirsiniz `USER` Dockerfile'a bir satır eklemek yeterlidir. Bu aslında en basit yaklaşımdır ancak yeterince sağlam değildir.

### Statik oluşturma ve dinamik haritalama

HagiCode'un yaklaşımı bir `hagicode` Genellikle çoğu ana makinedeki varsayılan kullanıcıyla eşleşen UID 1000'e sahip kullanıcı:

```dockerfile
RUN groupadd -o -g 1000 hagicode && \
    useradd -o -u 1000 -g 1000 -s /bin/bash -m hagicode && \
    mkdir -p /home/hagicode/.claude && \
    chown -R hagicode:hagicode /home/hagicode
```

Ancak bu yalnızca görüntünün içindeki yerleşik kullanıcıyı çözer. Ana bilgisayar kullanıcısı UID 1001 ise ne olur? Kapsayıcı başlatıldığında dinamik eşlemeyi desteklemeye devam etmeniz gerekir.

docker-entrypoint.sh anahtar mantığını içerir:

```bash
if [ -n "$PUID" ] && [ -n "$PGID" ]; then
    if ! id hagicode >/dev/null 2>&1; then
        groupadd -g "$PGID" hagicode
        useradd -u "$PUID" -g "$PGID" -s /bin/bash -m hagicode
    fi
fi
```

Bu tasarımın avantajı açıktır: **görüntü oluşturma sırasında varsayılan UID 1000'i kullanın, ardından çalışma zamanında dinamik olarak ayarlayın. `PUID` ve `PGID` ortam değişkenleri**. Ana bilgisayar kullanıcısının sahip olduğu UID ne olursa olsun, yapılandırma dosyalarının sahipliği doğru kalır.

## Kalıcı hacimlerin tasarım felsefesi

Her AI CLI aracının kendi tercih edilen yapılandırma dizini vardır, dolayısıyla bunların tek tek eşlenmesi gerekir:

| CLI Aracı | Kapsayıcıdaki Yol | Adlandırılmış Birim |
|---------|-----------|--------|
| Claude | `/home/hagicode/.claude` | `claude-data` |
| Kodeks | `/home/hagicode/.codex` | `codex-data` |
| Açık Kod | `/home/hagicode/.config/opencode` | `opencode-config-data` |

Bağlama bağları yerine neden adlandırılmış birimler kullanılmalı? Üç neden:

1. **Daha basit yönetim**: Adlandırılmış birimler Docker tarafından otomatik olarak yönetilir, dolayısıyla ana bilgisayar dizinlerini manuel olarak oluşturmanıza gerek kalmaz.
2. **İzin yalıtımı**: Birimlerin ilk içerikleri, ana bilgisayarla izin çakışmalarını önleyecek şekilde kullanıcı tarafından kapsayıcı içinde oluşturulur.
3. **Bağımsız geçiş**: Birimler kapsayıcılardan bağımsız olarak mevcut olabilir, böylece görüntüler yükseltildiğinde veriler kaybolmaz.

docker-compose-builder-web otomatik olarak karşılık gelen birim yapılandırmasını oluşturur:

```yaml
volumes:
  claude-data:
  codex-data:
  opencode-config-data:

services:
  hagicode:
    volumes:
      - claude-data:/home/hagicode/.claude
      - codex-data:/home/hagicode/.codex
      - opencode-config-data:/home/hagicode/.config/opencode
    user: "${PUID:-1000}:${PGID:-1000}"
```

Şuna dikkat edin: `user` buradaki alan: `PUID` ve `PGID` kapsayıcı içindeki işlemlerin ana kullanıcıyla eşleşen bir kimlikle çalışmasını sağlamak için ortam değişkenleri aracılığıyla enjekte edilir. Bu ayrıntı önemlidir çünkü izin sorunları ortaya çıktıktan sonra hata ayıklamak zahmetlidir.

## Sürüm yönetimi: çalışma zamanı geçersiz kılmalarına sahip yerleşik sürümler

Tekrarlanabilirlik için Docker görüntü versiyonlarının sabitlenmesi önemlidir. Ancak gerçek geliştirmede sıklıkla daha yeni bir sürümü test etmemiz veya bir hatayı acilen düzeltmemiz gerekir. Görüntüyü her seferinde yeniden oluşturmak zorunda kalsaydık iş akışı çok verimsiz olurdu.

HagiCode'un stratejisi **varsayılan olarak sabit sürümler ve bir uzatma mekanizması olarak çalışma zamanı geçersiz kılmaları** şeklindedir**. Bu, stabilite ve esneklik arasında pragmatik bir mühendislik uzlaşmasıdır.

Dockerfile.template sürümleri buraya sabitler:

```dockerfile
USER hagicode
WORKDIR /home/hagicode

# Configure the global npm install path
RUN mkdir -p /home/hagicode/.npm-global && \
    npm config set prefix '/home/hagicode/.npm-global'

# Install CLI tools using pinned versions
RUN npm install -g @anthropic-ai/claude-code@2.1.71 && \
    npm install -g @openai/codex@0.112.0 && \
    npm install -g opencode-ai@1.2.25 && \
    npm cache clean --force
```

docker-entrypoint.sh çalışma zamanı geçersiz kılmalarını destekler:

```bash
install_cli_override_if_needed() {
    local package_name="$2"
    local override_version="$5"

    if [ -n "$override_version" ]; then
        gosu hagicode npm install -g "${package_name}@${override_version}"
    fi
}

# Example usage
install_cli_override_if_needed "" "@anthropic-ai/claude-code" "" "" "${CLAUDE_CODE_CLI_VERSION}"
```

Bu, görüntüyü yeniden oluşturmadan yeni bir sürümü bir ortam değişkeni aracılığıyla test etmenize olanak tanır:

```bash
docker run -e CLAUDE_CODE_CLI_VERSION=2.2.0 myimage
```

Bu tasarım pratiktir çünkü hiç kimse yeni bir özelliği her test ettiğinde görüntüyü yeniden oluşturmak istemez.

## Otomatik konfigürasyon ekleme

Bazı senaryolar, CLI araçlarını manuel olarak yapılandırmanın yanı sıra otomatik yapılandırma eklemeyi de gerektirir. En tipik örnek bir API belirtecidir.

```bash
if [ -n "$ANTHROPIC_AUTH_TOKEN" ]; then
    mkdir -p /home/hagicode/.claude
    cat > /home/hagicode/.claude/settings.json <<EOF
{
  "env": {
    "ANTHROPIC_AUTH_TOKEN": "${ANTHROPIC_AUTH_TOKEN}"
  }
}
EOF
    chown -R hagicode:hagicode /home/hagicode/.claude
fi
```

Burada iki şey önemlidir: **hassas bilgileri görüntüye sabit kodlamak yerine ortam değişkenleri aracılığıyla iletin** ve yapılandırma dosyalarının sahipliğinin doğru ayarlandığından emin olun, aksi takdirde CLI araçları bunları okuyamaz.

## En iyi uygulamalar ve tuzak kontrol listesi

### İzin uyumsuzluğu sorunları

Bu, düşülmesi en kolay tuzaktır. Ana bilgisayar kullanıcısı UID 1001'e sahipken konteyner 1000 kullanıyor, dolayısıyla bir tarafta oluşturulan dosyalara diğer taraftan erişilemiyor.

```bash
# Correct approach: make the container match the host user
docker run \
    -e PUID=$(id -u) \
    -e PGID=$(id -g) \
    myimage
```

Bu sorun çok yaygındır ve ilk kez karşılaştığınızda sinir bozucu olabilir.

### Kapsayıcı yeniden başlatıldıktan sonra yapılandırma kayboluyor

Her yeniden başlatmanın ardından kendinizi tekrar oturum açarken bulursanız, kalıcı bir birim eklemeyi unutup unutmadığınızı kontrol edin:

```yaml
volumes:
  - claude-data:/home/hagicode/.claude
```

Hiçbir şey, bir konfigürasyonu dikkatli bir şekilde kurup onun kaybolduğunu görmekten daha sinir bozucu olamaz.

### Sürümleri yükseltmenin doğru yolu

Koşma `npm install -g` doğrudan çalışan bir konteynerin içinde. Doğru yaklaşımlar şunlardır:

1. Geçersiz kılma yüklemesini tetiklemek için bir ortam değişkeni ayarlayın.
2. Veya görüntüyü yeniden oluşturun.

```bash
# Option 1: runtime override
docker run -e CLAUDE_CODE_CLI_VERSION=2.2.0 myimage

# Option 2: rebuild the image
docker build -t myimage:v2 .
```

Roma'ya giden birden fazla yol var ama bazı yollar diğerlerinden daha düzgün.

### Güvenliği güçlendirme kontrol listesi

- API belirteçlerini görüntüye yazmak yerine ortam değişkenleri aracılığıyla iletin.
- Yapılandırma dosyası izinlerini şu şekilde ayarlayın: `600`.
- Uygulamayı her zaman root olmayan bir kullanıcı olarak çalıştırın.
- Güvenlik açıklarını düzeltmek için CLI sürümlerini düzenli olarak güncelleyin.

Güvenlik her zaman önemlidir, ancak asıl zorluk onu pratikte tutarlı bir şekilde uygulamaktır.

## Yeni CLI araçları için desteğin genişletilmesi

Gelecekte yeni bir CLI aracını desteklemek istiyorsanız yalnızca üç adım vardır:

1. **Dockerfile.template**: kurulum adımını ekleyin.
2. **docker-entrypoint.sh**: sürüm geçersiz kılma mantığını ekleyin.
3. **docker-compose-builder-web**: kalıcı birim eşlemesini ekleyin.

Bu şablon tabanlı tasarım, temel mantığı değiştirmeden genişletmeyi basitleştirir.

## Sonuç

AI CLI araçlarını Docker kapsayıcılarında çalıştırmak üç temel zorluğu içerir: **kullanıcı izinleri, yapılandırma kalıcılığı ve sürüm yönetimi**. HagiCode projesi, özel kullanıcıları, adlandırılmış birim izolasyonunu ve ortam değişkeni tabanlı geçersiz kılmaları birleştirerek hem güvenli hem de esnek bir dağıtım mimarisi oluşturdu.

Anahtar tasarım noktaları:

- **Kullanıcı izolasyonu**: Dinamik için çalışma zamanı desteğiyle, görüntü oluşturma aşamasında özel bir kullanıcı oluşturun `PUID`/`PGID` haritalama.
- **Kalıcılık stratejisi**: Her CLI aracı kendi adlandırılmış birimine sahip olduğundan yeniden başlatmalar yapılandırmayı etkilemez.
- **Sürüm esnekliği**: Sabit varsayılanlar tekrarlanabilirlik sağlarken, çalışma zamanı geçersiz kılmaları test için alan sağlar.
- **Otomatik yapılandırma**: Hassas yapılandırma, ortam değişkenleri aracılığıyla otomatik olarak eklenebilir.
