## TypeScript'ten C#'a: Codex SDK için Diller Arası Taşıma Uygulaması

> Basitçe söylemek gerekirse, bu makale aynı zamanda bizim küçük bir makalemizdir: resmi TypeScript Codex SDK'yı C#'a taşıma işleminin tamamını kaydeder. Bunu bir "liman" olarak adlandırmak kulağa çok kolay geliyor; daha çok uzun bir maceraya benziyordu çünkü bu iki dilin çok farklı kişilikleri var ve onların işbirliği yapmasını sağlamanın bir yolunu bulmamız gerekiyordu.

## Arka plan

Codex, OpenAI tarafından piyasaya sürülen AI Agent CLI aracıdır ve gerçekten güçlüdür. Resmi ekip, bir TypeScript SDK'sı sağlar. `@openai/codex` paket. Codex CLI ile aşağıdakileri çağırarak etkileşime girer: `codex exec --experimental-json` JSONL olay akışını komut etme ve ayrıştırma.

Sorun şu ki, HagiCode projesinde onu saf bir .NET ortamında, özellikle de C# arka uç hizmetlerinde ve masaüstü uygulamalarında kullanmamız gerekiyor. Sadece bir CLI aracını çağırmak için bir .NET projesine Node.js çalışma zamanını makul bir şekilde dahil edemezdik. Bu çok zahmetli olurdu.

Dolayısıyla elimizde iki seçenek kaldı: karmaşık bir Node.js köprü katmanını sürdürmek veya yerel bir C# SDK'sını kendimiz oluşturmak.

Biz ikincisini seçtik.

## Temel İçerik

### Mimari Tasarım Karşılaştırması

Kodu bire bir çevirmeden önce her iki SDK'nın da mimari tasarımını anlamamız gerekiyordu. İyi bir şekilde aktarmadan önce her iki tarafı da anlamalısınız.

TypeScript SDK'nın çekirdek mimarisi şuna benzer:

```
Codex (entry class)
  └── CodexExec (executor, manages child processes)
      └── Thread (conversation thread)
          ├── run() / runStreamed() (synchronous/asynchronous execution)
          └── event stream parsing
```

C# SDK aynı mimari katmanlamayı korur ancak uygulama ayrıntılarını uyarlar. Genel fikir basittir: Uygulamada C# dili özelliklerinden tam olarak yararlanırken API tutarlılığını koruyun.

### Tip Sistem Dönüşümü

Bu işin en temel ve aynı zamanda en önemli kısmıdır. Temel zayıfsa bundan sonraki her şey zorlaşır.

TypeScript'in yazım sistemi C#'ınkinden daha esnektir ve bu bir gerçektir. Uygun bir haritalama stratejisi bulmamız gerekiyordu:

| TypeScript | C# | Notlar |
|------------|-----|------|
| `interface` / `type` | `record` | C# kullanımları `record` değişmez veri yapıları için |
| 'dize \| boş' | `string?` | Null yapılabilir referans türü |
| `boolean \| tanımsız` | `bool?` | Null yapılabilir Boolean |
| `AsyncGenerator` | `IAsyncEnumerable` | Eşzamansız yineleyici |

**Olay türü sistemi** tipik bir örnektir. TypeScript, olayları tanımlamak için birleşim türlerini kullanır:

```typescript
export type ThreadEvent =
  | ThreadStartedEvent
  | TurnStartedEvent
  | TurnCompletedEvent
  | ...
```

C#'ta benzer bir etki elde etmek için kalıtım hiyerarşisini ve kalıp eşleştirmeyi kullanırız:

```csharp
public abstract record ThreadEvent(string Type);

public sealed record ThreadStartedEvent(string ThreadId) : ThreadEvent("thread.started");
public sealed record TurnStartedEvent() : ThreadEvent("turn.started");
public sealed record TurnCompletedEvent(Usage Usage) : ThreadEvent("turn.completed");
// ...
```

Biz seçtik `record` yerine `class` çünkü olay nesnelerinin değişmez olması gerekir; bu da TypeScript'te düz nesnelerin kullanılmasının ardındaki amaç ile eşleşir. `sealed` anahtar kelime aynı zamanda ek kalıtımı önler ve derleyiciye optimizasyon için yer sağlar.

### Anahtar Taşıma Noktaları

#### 1. Olay ayrıştırıcı

Olay ayrıştırma tüm SDK'nın temelidir çünkü Codex CLI tarafından döndürülen her mesajı doğru şekilde anlayıp anlayamayacağımızı belirler. Eğer ayrıştırma yanlışsa bundan sonraki her şey boşa çabadır.

TypeScript sürümünün kullandığı `JSON.parse()` JSON'un her satırını ayrıştırmak için:

```typescript
export function parseEvent(line: string): ThreadEvent {
  const data = JSON.parse(line);
  // Handle different event types...
}
```

C# sürümü şunu kullanır: `System.Text.Json.JsonDocument` bunun yerine:

```csharp
public static ThreadEvent Parse(string line)
{
    using var document = JsonDocument.Parse(line);
    var root = document.RootElement;
    var type = GetRequiredString(root, "type", "event.type");

    return type switch
    {
        "thread.started" => new ThreadStartedEvent(GetRequiredString(root, "thread_id", ...)),
        "turn.started" => new TurnStartedEvent(),
        "turn.completed" => new TurnCompletedEvent(ParseUsage(...)),
        // ...
        _ => new UnknownThreadEvent(type, root.Clone()),
    };
}
```

Burada küçük ama önemli bir püf noktası var: `root.Clone()` gereklidir, çünkü öğeler `JsonDocument` belge imha edildikten sonra geçersiz hale gelir. Bilinmeyen olay türleri için bir kopyasını saklamamız gerekir. Bu, C# JSON işleme ile JavaScript arasındaki farklardan sadece biridir.

#### 2. Süreç yönetimi farklılıkları

İki SDK'nın en çok farklılık gösterdiği nokta burasıdır. Node.js ve .NET'in farklı çalışma zamanı kuralları vardır, bu nedenle uygulamanın buna uyum sağlaması gerekir.

TypeScript, Node.js'yi kullanır `spawn()` işlev:

```typescript
const child = spawn(this.executablePath, commandArgs, { env, signal });
```

C# .NET'i kullanıyor `System.Diagnostics.Process`:

```csharp
using var process = new Process { StartInfo = startInfo };
process.Start();

// stdin/stdout/stderr must be managed manually
```

Daha spesifik olarak, C# sürümünün süreci şu şekilde yapılandırması gerekir:

```csharp
var startInfo = new ProcessStartInfo
{
    FileName = _executablePath,
    RedirectStandardInput = true,
    RedirectStandardOutput = true,
    RedirectStandardError = true,
    UseShellExecute = false,
    CreateNoWindow = true,
};
```

En büyük fark iptal mekanizmasıdır. TypeScript'in kullanım alanları `AbortSignal`Web API'sinin bir parçası olan ve çalışması çok kolay olan:

```typescript
const child = spawn(cmd, args, { signal: cancellationSignal });
```

C# kullanımları `CancellationToken` bunun yerine:

```csharp
public async IAsyncEnumerable<string> RunAsync(
    CodexExecArgs args,
    [EnumeratorCancellation] CancellationToken cancellationToken = default)
{
    // Check cancellation status inside the loop
    while (!cancellationToken.IsCancellationRequested)
    {
        // Process output...
    }

    // Terminate the process when cancellation is requested
    if (cancellationToken.IsCancellationRequested)
    {
        try { process.Kill(entireProcessTree: true); } catch { }
    }
}
```

Yüksek düzeyde bu, Web API ekosistemi ile .NET ekosistemi arasındaki farkın yalnızca bir başka örneğidir.

#### 3. Yapılandırma serileştirmesinin korunması

Codex CLI, TOML biçimindeki yapılandırma geçersiz kılmalarını kabul ettiğinden, her iki SDK da JSON yapılandırmasını TOML yapılandırmasına dönüştüren mantığı uygular. Bu kısım tamamen tutarlı kalmalıdır, aksi takdirde aynı konfigürasyon iki SDK'da farklı davranacaktır.

Bu taviz veremeyeceğiniz türden bir ayrıntıdır. Başarı ya da başarısızlık çoğu zaman bunun gibi ayrıntılara bağlıdır.

### Uygulama Detayları

#### Proje yapısı

Aşağıdaki proje yapısını oluşturduk:

```
CodexSdk/
├── CodexSdk.csproj
├── Codex.cs           # Entry class
├── CodexThread.cs     # Conversation thread
├── CodexExec.cs       # Executor
├── Events.cs          # Event type definitions
├── Items.cs           # Item type definitions
├── EventParser.cs     # Event parser
├── OutputSchemaTempFile.cs  # Temporary file management
└── ...
```

Oldukça temiz bir yapı ve bu da liman sırasında çok yardımcı oldu.

#### Kullanım örneği

Temel kullanım TypeScript SDK ile tutarlı kalır:

```csharp
using CodexSdk;

// Create a Codex instance
var codex = new Codex();
var thread = codex.StartThread();

// Execute a query
var result = await thread.RunAsync("Summarize this repository.");
Console.WriteLine(result.FinalResponse);
```

Akış olayı işleme, C# desen eşleştirmesinden yararlanır:

```csharp
await foreach (var @event in thread.RunStreamedAsync("Analyze the code."))
{
    switch (@event)
    {
        case ItemCompletedEvent itemCompleted
            when itemCompleted.Item is AgentMessageItem msg:
            Console.WriteLine($"Assistant: {msg.Text}");
            break;
        case TurnCompletedEvent completed:
            Console.WriteLine($"Tokens: in={completed.Usage.InputTokens}");
            break;
        case CommandExecutionItem command:
            Console.WriteLine($"Command: {command.Command}");
            break;
    }
}
```

### Notlar

Uygulama sırasında birkaç pratik ders topladık:

1. **Süreç yönetimi**: C# sürümü, iptal sırasında sürecin sonlandırılması da dahil olmak üzere tüm süreç yaşam döngüsünü manuel olarak yönetmelidir. Kullanım `Kill(entireProcessTree: true)` alt süreçlerin de temizlendiğinden emin olmak için.

2. **Hata yönetimi**: Kullanıyoruz `InvalidOperationException` ayrıştırma hataları oluşturmak ve hata işleme stilini TypeScript SDK'ya benzer tutmak için.

3. **Kaynak temizleme**: `OutputSchemaTempFile` uygular `IAsyncDisposable` geçici dosyaların doğru şekilde temizlendiğinden emin olmak için.

4. **Ortam değişkenleri**: C# sürümü, süreç ortamı değişkenlerinin tamamen geçersiz kılınmasını destekler. `CodexOptions.Env`. Bu küçük bir özellik ama çok pratik bir özellik.

5. **Platform farklılıkları**: C# sürümü, TypeScript sürümünün npm paketleri içindeki ikili dosyaları otomatik olarak bulma mantığını içermez. .NET projeleri genellikle npm'ye bağlı olmadığından, `codex` yürütülebilir dosya aracılığıyla belirtilmelidir. `CODEX_EXECUTABLE` ortam değişkeni veya `CodexPathOverride`.

## Sonuç

Olgun bir TypeScript SDK'yı C#'a taşımak yalnızca sözdizimi dönüştürme meselesi değildir; aynı zamanda her iki dilin tasarım felsefelerinin anlaşılmasını da gerektirir. TypeScript'in esnekliği ve JavaScript ekosistemi gibi özellikleri `AbortSignal` C#'ta uygun karşılıklara ihtiyaç var.

Temel çıkarım şudur: **API tutarlılığını korumak, uygulama düzeyinde tutarlılığı sürdürmekten daha önemlidir**. Kullanıcılar, dahili uygulamanın aynı olup olmadığına değil, arayüzün kullanımının kolay olup olmadığına önem verirler. Kulağa basit geliyor ama bu ödünleşimleri yapmak muhakeme gerektirir.

Benzer bir diller arası bağlantı noktası üzerinde çalışıyorsanız deneyimimiz, önce orijinal SDK mimarisini tam olarak anlamak, ardından onu modül modül çevirmek ve son olarak davranışsal tutarlılığı sağlamak için eksiksiz bir test paketi kullanmaktır. Bu tür çalışmalar aceleye getirilemez.

Sonunda her şey yoluna girecek.
