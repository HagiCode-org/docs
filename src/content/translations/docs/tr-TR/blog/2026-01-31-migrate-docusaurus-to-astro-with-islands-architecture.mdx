## Docusaurus 3.x'ten Astro 5.x'e: HagiCode Sitesi Geçişine Retrospektif Bir Bakış

> Bu makale, HagiCode resmi web sitesinin Docusaurus 3.x'ten Astro 5.x'e tam geçişini ele alıyor. Astro's Islands Architecture'ın, mevcut React bileşen varlıklarımızı korurken hem derleme hızı hem de yükleme performansında iyileştirmeler sağlayarak performans darboğazlarını çözmemize nasıl yardımcı olduğunu derinlemesine inceleyeceğiz.

## Arka plan

Ocak 2026'da HagiCode resmi sitesinde temel çerçevesini Docusaurus 3.x'ten Astro 5.x'e tamamen geçirerek bir "kalp nakli" gerçekleştirdik. Bu, dürtüsel bir yeniden yazma değil, dikkatlice düşünülmüş bir teknik karardı.

Geçişten önce sitemiz işlevsel olarak tamamlanmıştı ancak bazı klasik "lüks sorunlar" göstermeye başlamıştı: şişirilmiş yapı yapıları, aşırı JavaScript yükleri ve karmaşık dokümantasyon sayfalarında idealden düşük sayfa yükleme hızı. Bir yapay zeka kodlama asistanı projesi olarak HagiCode'un sık sık dokümantasyona ve özellik güncellemelerine ihtiyacı vardır, bu nedenle derleme verimliliği sürüm hızını doğrudan etkiler. Aynı zamanda, daha fazla geliştiricinin projeyi keşfedebilmesi için sitenin daha arama motoru dostu (SEO) olmasını istedik.

Bu sıkıntılı noktaları çözmek için cesur bir karar aldık: tüm sistemi Astro'da yeniden inşa ettik. Bu kararın etkisi beklediğinizden daha büyük olabilir. Birazdan ayrıntılara gireceğim.

## HagiCode Hakkında

Bu makalede paylaşılan site taşıma yaklaşımı, uygulamalı deneyimimizden gelmektedir. [Hagi Kodu](https://github.com/HagiCode-org/site) proje.

HagiCode, geliştirme verimliliğini artırmaya odaklanan bir yapay zeka kodlama asistanıdır. Yalnızca temel özelliklerin yinelenmesini değil, aynı zamanda geliştirici deneyimini de önemsiyoruz. Bu site yeniden düzenlemesi aynı zamanda kullanıcılara belgelerimize ve resmi web sitemize göz atarken mümkün olan en hızlı deneyimi sunmayı amaçlıyordu.

## Neden Olgun Docusaurus Ekosisteminden Ayrılmalısınız?

React ekosisteminde Docusaurus uzun süredir dokümantasyon siteleri için "standart cevap" olmuştur. Kutudan çıktığı gibi çalışır, zengin bir eklenti ekosistemi sunar ve aktif bir topluluğa sahiptir. Ancak HagiCode daha fazla özellik kazandıkça sınırlamalarını da hissettik:

1.  **Performans darboğazları**: Docusaurus temelde bir React SPA'dır (tek sayfalı uygulama). Yalnızca statik sayfalar yazsanız bile istemcinin yine de React çalışma zamanını yüklemesi ve sayfayı nemlendirmesi gerekir; bu da basit doküman sayfaları için gereksiz derecede ağırdır.
2.  **Büyük varlık boyutu**: Bir sayfada çok az içerik bulunsa bile, paketlenen JS boyutu nispeten sabit kalır. Bu, mobil kullanıcılar veya kötü ağ koşulları için ideal değildir.
3.  **Sınırlı esneklik**: Genişletilebilir olmasına rağmen derleme hattı üzerinde daha düşük düzeyde kontrol istedik.

Astro bu sorunları çözmek için tam zamanında geldi. Yeni bir "Ada Mimarisi" tanıttı: Astro, varsayılan olarak sıfır JavaScript ile statik HTML oluşturur ve yalnızca etkileşim gerektiren bileşenler "etkinleştirilir" ve JS'yi yükler. Bu, sitemizin çoğunun saf HTML haline geldiği ve son derece hızlı yüklendiği anlamına gelir.

## Temel Geçiş Stratejisi: Sorunsuz Bir Mimari Geçiş

Göç sadece kopyala-yapıştırdan ibaret değildi. Zihniyet değişikliği gerektiriyordu. Docusaurus'un "all React" modelinden Astro'nun "Core + Islands" modeline geçtik.

### 1. Konfigürasyon Sisteminin Yeniden Oluşturulması

İlk önce taşınmak zorunda kaldık `docusaurus.config.ts` için `astro.config.mjs`. Bu yalnızca bir dosyanın yeniden adlandırılması değil, yönlendirme ve derleme mantığının yeniden yazılmasıydı.

Docusaurus'ta her şey bir eklentidir. Astro'da her şey bir bütünleşmedir. Sitenin temel yolunu, derleme çıktı modunu (statik ve SSR) ve varlık optimizasyon stratejisini yeniden tanımlamamız gerekiyordu.

**Taşımadan önce:**
```typescript
// docusaurus.config.ts
export default {
  title: 'HagiCode',
  url: 'https://hagicode.com',
  baseUrl: '/',
  // ... more configuration
};
```

**Taşıma sonrasında:**
```javascript
// astro.config.mjs
import { defineConfig } from 'astro/config';
import react from '@astrojs/react';

export default defineConfig({
  integrations: [react()],
  site: 'https://hagicode.com',
  base: '/',
  // Optimization settings for static assets
  build: {
    inlineStylesheets: 'auto',
  },
});
```

### 2. React Bileşenlerinde Neler Saklanmalı ve Neler Yeniden Düzenlenmeli

Bu, göçün en acı kısmıydı. Mevcut sitemizde Sekmeler, kod vurgulama, geri bildirim düğmeleri ve daha fazlası gibi birçok React bileşeni vardı. Bunları atmak israf olur, ancak her şeyi saklamak JavaScript yükünün çok ağır olmasına neden olur.

HagiCode **aşamalı sıvı alımı** stratejisini benimsedi:
*   **Saf statik bileşenler**: Üstbilgiler, altbilgiler ve düz metin belgeleri gibi sunum içerikleri için bunları Astro bileşenleri olarak yeniden yazdık (`.astro` dosyalar) ve bunları oluşturma sırasında doğrudan HTML'ye dönüştürdü.
*   **Etkileşimli adalar**: Tema değiştiriciler, sekme değiştirme ve kod bloğu kopyalama düğmeleri gibi etkileşimli kalması gereken bileşenler için React uygulamasını koruduk ve ekledik `client:load` veya `client:visible` direktifler.

Örneğin, belgelerde yaygın olarak kullanılan Sekmeler bileşenimiz:

```jsx
// src/components/Tabs.jsx
import { useState } from 'react';
import './Tabs.css'; // Import styles

export default function Tabs({ items }) {
  const [activeIndex, setActiveIndex] = useState(0);
  // ... state logic
  return (
    <div className="tabs-wrapper">
      {/* Rendering logic */}
    </div>
  );
}
```

Markdown'da kullanıldığında Astro'ya açıkça şunu söylüyoruz: "Bu bileşenin JS'ye ihtiyacı var."

```markdown
// src/content/docs/example.mdx
import Tabs from '../../components/Tabs.jsx';

<!-- Load JS only when the component enters the viewport -->
<Tabs client:visible items={...} />
```

Bu şekilde, görünüm alanının dışındaki etkileşimli bileşenler bant genişliği için rekabet etmez, bu da ilk ekranın yükleme hızını büyük ölçüde artırır.

### 3. Şekillendirme Sisteminin Uyarlanması: CSS Modüllerinden Kapsamlı CSS'ye

Docusaurus varsayılan olarak CSS Modüllerini desteklerken Astro, Kapsamlı CSS'yi teşvik eder. `<style>` etiketi. Her ikisinin de arkasındaki temel fikir stil yalıtımıdır, ancak sözdizimi farklıdır.

HagiCode geçişi sırasında, çoğu karmaşık CSS Modülünü Astro'nun kapsamlı stillerine dönüştürdük. Bunun aslında iyi bir şey olduğu ortaya çıktı, çünkü `.astro` Stiller ve şablonlar aynı dosyada bulunur, bu da bakımı daha sezgisel hale getirir.

**Yeniden düzenlemeden önce:**
```css
/* Tabs.module.css */
.wrapper { background: var(--ifm-background-color); }
```

**Yeniden düzenleme sonrasında (Astro Kapsamlı):**
```html
<!-- Tabs.astro -->
<div class="tabs-wrapper">
  <slot />
</div>

<style>
  .tabs-wrapper {
    /* Use CSS variables directly to adapt to the theme */
    background: var(--bg-color);
    padding: 1rem;
  }
</style>
```

Aynı zamanda, küresel CSS değişken sistemini birleştirdik ve karanlık modun sayfalar arasında sorunsuz bir şekilde geçiş yapmasını sağlamak için Astro'nun çevreye duyarlı yeteneklerini kullandık.

## Pratikte Karşılaştığımız Tuzaklar ve Bunları Nasıl Çözdük?

Gerçek HagiCode geçişi sırasında pek çok sorunla karşılaştık. İşte en tipik olanlardan birkaçı.

### 1. Yol ve Ortam Değişken Sorun Noktaları

HagiCode, GitHub Pages alt dizini altındaki dağıtım gibi alt yol dağıtımını destekler. Docusaurus'ta, `baseUrl` otomatik olarak işlenir. Ancak Astro'da resim bağlantılarını ve API isteklerini işlerken daha dikkatli olmamız gerekiyor.

Bunu tutarlı bir şekilde yönetmek için bir ortam değişkeni mekanizması geliştirdik:

```javascript
// Handle paths in the build script
const getBasePath = () => import.meta.env.VITE_SITE_BASE || '/';
```

İle başlayan yolları sabit kodlamadığınızdan emin olun. `/` kodunuzda. Geliştirmeye karşı üretimde veya bir yapılandırmanın ardından `base` yolu, bunu yapmak varlıklar için 404'lere neden olabilir.

### 2. CommonJS Komut Dosyası Uyumluluğu

Eski sitemizde Metrics verilerinin otomatik olarak getirilmesi ve site haritasının güncellenmesi gibi görevler için kullanılan bazı Node.js scriptleri vardı ve bunlar CommonJS'de yazılmıştı (`require`). Astro ve modern yapı araçları, ES Modüllerini tamamen benimsemiştir (`import`/`export`).

Benzer komut dosyalarınız varsa hepsini ES Modüllerine yeniden düzenlemeyi unutmayın. Ekosistemin hareket ettiği yön budur ve değişikliği ne kadar erken yaparsanız, daha sonra o kadar az sorunla karşılaşırsınız.

```javascript
// Old way
const fs = require('fs');

// New way
import fs from 'fs';
```

### 3. SEO ve Yönlendirmeleri Unutmayın

Arama motorları HagiCode'un eski Docusaurus sayfalarını zaten indeksledi. Doğrudan Astro'ya geçerseniz ve URL yapısı değişirse, çok sayıda 404'le karşılaşabilir ve arama sıralamasında büyük bir düşüş yaşayabilirsiniz.

Astro'da yönlendirme kurallarını yapılandırdık:

```javascript
// astro.config.mjs
export default defineConfig({
  redirects: {
    '/docs/old-path': '/docs/new-path',
    // Map old links to new links in bulk
  }
});
```

Veya bunu sunucu yapılandırma katmanında halledebilirsiniz. Eski bağlantıların 301 ile yeni adreslere yönlendirilebildiğinden emin olun, çünkü bu SEO açısından kritik öneme sahiptir.

## Özet

HagiCode için Docusaurus'tan Astro'ya geçiş yalnızca bir çerçeve yükseltmesi değildi. Bu aynı zamanda "önce performans" felsefesinin pratik bir uygulamasıydı.

**Kazandıklarımız:**
*   **Olağanüstü Lighthouse puanları**: Geçişten sonra HagiCode sitesinin performans puanı kolaylıkla mükemmel puana yaklaştı.
*   **Daha yüksek derleme hızı**: Astro'nun artımlı derleme yetenekleri, belge güncellemelerinin yayınlanma süresini yarı yarıya azaltır.
*   **Korunmuş esneklik**: Islands Architecture ile hiçbir etkileşimli özellikten ödün vermedik ve ihtiyaç duyulan yerde React'ı kullanmaya devam edebildik.

Ayrıca dokümantasyon odaklı bir siteniz varsa ve paket boyutu veya yükleme hızıyla mücadele ediyorsanız Astro denemeye değer. Geçiş süreci, PCode'u HagiCode olarak yeniden adlandırmak ve bileşenleri tek tek taşımak gibi bazı ameliyatlar gerektirse de, karşılığında alacağınız ipeksi pürüzsüz kullanıcı deneyimi, bunu kesinlikle değerli kılar.

Bu makalede paylaşılan derleme sistemi, HagiCode'u oluştururken gerçek deneme yanılma yoluyla geliştirdiğimiz yaklaşımın aynısıdır. Bu yaklaşımı değerli buluyorsanız, bu bizim mühendislik gücümüz hakkında bir şeyler söylüyor ve HagiCode'un kendisi de muhtemelen daha yakından incelenmeye değer.
