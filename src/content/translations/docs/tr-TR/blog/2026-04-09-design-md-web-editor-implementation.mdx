## DESIGN.md'yi Doğrudan Web Arayüzünde Düzenleyin: Fikirden Uygulamaya

> MonoSpecs proje yönetim sisteminde, `DESIGN.md` Bir projenin mimari tasarımını ve teknik kararlarını taşır. Ancak geleneksel düzenleme iş akışı, kullanıcıları harici bir düzenleyiciye geçmeye zorlar. Bu parçalanmış deneyim, bir şiir okurken yarıda kesilmeye benzer: İlham da, ruh hali de gider. Bu makale HagiCode projesinde uygulamaya koyduğumuz çözümü paylaşıyor: düzenleme `DESIGN.md` Çevrimiçi bir tasarım sitesinden şablonların içe aktarılması desteğiyle doğrudan web arayüzünde. Sonuçta her şeyi tek bir akışta tamamlama hissinden kim hoşlanmaz ki?

## Arka plan

Proje tasarım belgelerinin temel taşıyıcısı olarak, `DESIGN.md` mimari tasarım, teknik kararlar ve uygulama rehberliği gibi önemli bilgileri içerir. Ancak geleneksel düzenleme yaklaşımı, kullanıcıların VS Code gibi harici bir düzenleyiciye geçmesini, fiziksel yolu manuel olarak bulmasını ve ardından dosyayı düzenlemesini gerektirir. Özellikle karmaşık değildir, ancak işlemi birkaç kez tekrarladıktan sonra yorucu hale gelir.

Sorunlar esas olarak aşağıdaki şekillerde ortaya çıkıyor:

- **Parçalanmış iş akışı**: Kullanıcılar, tıpkı bir şarkının ortasında müziğin kesilmesi gibi, iş akışlarının sürekliliğini bozarak web yönetimi arayüzü ile yerel düzenleyici arasında sürekli geçiş yapmak zorundadır.
- **Yeniden kullanımı zor**: Tasarım sitesi zaten zengin bir tasarım şablonları kitaplığı yayınlıyor, ancak bunlar doğrudan proje düzenleme iş akışına entegre edilemiyor. İyi şeyler var ama yine de onları ihtiyaç duyduğunuz yerde kullanamıyorsunuz.
- **Deneyim döngüsü eksik**: "Önizleme-seç-içe aktarma" için kapalı bir döngü yoktur, dolayısıyla kullanıcıların manuel olarak kopyalayıp yapıştırması gerekir; bu da hata riskini artırır.
- **İşbirliği anlaşmazlığı**: Tasarım belgelerini ve kod uygulamasını senkronize tutmak, ekip verimliliğine zarar veren yüksek düzeyde sürtünmeli bir süreç haline gelir.

Bu sıkıntılı noktaları çözmek için doğrudan düzenleme desteği eklemeye karar verdik. `DESIGN.md` web arayüzünde ve çevrimiçi bir tasarım sitesinden tek tıklamayla şablonun içe aktarılmasına izin verin. Bu dünyayı sarsacak bir karar değildi. Biz sadece geliştirme deneyimini daha sorunsuz hale getirmek istedik.

## Teknik Çözüm

### Genel Mimari

Bu çözüm, esas olarak aşağıdaki katmanlardan oluşan, aynı kökenli proxy'ye sahip, ön uç-arka uçtan ayrılmış bir mimari kullanır. Uygulamada tasarım "her parçanın kendi işini yapması" şeklinde özetlenebilir:

**1. Ön uç düzenleyici katmanı**
```typescript
// Core component: DesignMdManagementDrawer
// Location: repos/web/src/components/project/DesignMdManagementDrawer.tsx
// Responsibility: handle editing, saving, version conflict detection, and import flow
```

**2. Arka uç hizmet katmanı**
```csharp
// Core service: ProjectAppService.DesignMd
// Location: repos/hagicode-core/src/PCode.Application/ProjectAppService.DesignMd.cs
// Responsibility: path resolution, file read/write, and version management
```

**3. Aynı kökenli proxy katmanı**
```csharp
// Proxy service: ProjectAppService.DesignMdSiteIndex
// Location: repos/hagicode-core/src/PCode.Application/ProjectAppService.DesignMdSiteIndex.cs
// Responsibility: proxy design site resources, preview image caching, and security validation
```

### Temel Teknik Kararlar

#### Karar 1: Global Çekmece Modeli

Durumun yönetildiği yerel açılır katmanlar yerine tek bir genel çekmece kullanıyoruz `layoutSlice`, kullanıcılara görünümler arasında tutarlı bir deneyim sunar (`classic` ve `kanban`). Kullanıcı editörü hangi görünümden açarsa açsın aynı etkileşim modelini elde eder. Tutarlı bir deneyim, insanların görüş değiştirdiğinde yönlerini kaybetmeleri yerine kendilerini daha rahat hissetmelerini sağlar.

#### Karar 2: Proje Kapsamlı API

Biz monte ettik `DESIGN.md`altında ilgili uç noktalar `ProjectController`, mevcut proje izin sınırının yeniden kullanılması ve ayrı bir denetleyici eklemenin karmaşıklığından kaçınılması. Bu, izin yönetimini daha net hale getirir ve aynı zamanda RESTful kaynak organizasyonuyla da uyumludur. Bazen yeniden kullanmak, sıfırdan yeni bir şey yaratmaktan daha anlamlıdır.

#### Karar 3: Sürüm Çakışmalarının Tespiti

Dosya sisteminin opak bir versiyonunu türetiyoruz. `LastWriteTimeUtc`, bu bize hafif, iyimser eşzamanlılık kontrolü sağlar. Birden fazla kullanıcı aynı dosyayı aynı anda düzenlediğinde, sistem çakışmaları algılayabilir ve kullanıcıdan dosyayı yenilemesini isteyebilir. Bu tasarım, veri tutarlılığını korurken düzenlemeyi engellemez.

#### Karar 4: Aynı Kökenli Proxy Modeli

Kullanıyoruz `IHttpClientFactory` hem çapraz köken sorunlarından hem de SSRF risklerinden kaçınarak harici tasarım sitesi kaynaklarını temsil etmek. Bu, ön uç çağrıları basitleştirirken aynı zamanda sistemi güvende tutar. Güvenlik konusunda çok dikkatli olamazsınız.

## Temel Uygulama

### 1. DESIGN.md'yi Doğrudan Düzenleyin

#### Arka Uç Uygulaması

Arka uç esas olarak yol çözümlemesinden, dosya okuma/yazma ve sürüm yönetiminden sorumludur. Bu görevler temeldir ancak bir evin temeli gibi vazgeçilmezdir:

```csharp
// Path resolution and security validation
private Task<string> ResolveDesignDocumentDirectoryAsync(string projectPath, string? repositoryPath)
{
    if (string.IsNullOrWhiteSpace(repositoryPath))
    {
        return Task.FromResult(Path.GetFullPath(projectPath));
    }
    return ValidateSubPathAsync(projectPath, repositoryPath);
}

// Version generation (based on file system timestamp and size)
private static string BuildDesignDocumentVersion(string path)
{
    var fileInfo = new FileInfo(path);
    fileInfo.Refresh();
    return string.Create(
        CultureInfo.InvariantCulture,
        $"{fileInfo.LastWriteTimeUtc.Ticks:x}-{fileInfo.Length:x}");
}
```

Sürüm tasarımı basitliği açısından ilgi çekicidir: benzersiz bir sürüm tanımlayıcısı oluşturmak için dosyanın son değiştirilme zamanını ve boyutunu kullanırız. Hafif ve güvenilirdir, bakımı gereken ekstra sürüm veritabanı yoktur. Basit çözümler çoğu zaman en etkili olanlardır.

#### Ön Uç Uygulaması

Ön uçta, kirli durum tespiti ve kaydetme mantığını uyguluyoruz. Bu tasarım, kullanıcıların yaptıkları değişikliklerin kaydedilip kaydedilmediğini anlamalarına yardımcı olur ve "ya kaybedersem?" kaygısını azaltır:

```typescript
// Dirty-state detection and save logic
const [draft, setDraft] = useState('');
const [savedDraft, setSavedDraft] = useState('');
const isDirty = draft !== savedDraft;

const handleSave = useCallback(async () => {
    const result = await saveProjectDesignMdDocument({
        ...activeTarget,
        content: draft,
        expectedVersion: document.version, // optimistic concurrency control
    });
    setSavedDraft(draft); // update saved state
}, [activeTarget, document, draft]);
```

Bu uygulamada iki durum parçasını koruyoruz: `draft` içerik şu anda düzenleniyor mu? `savedDraft` kaydedilen içeriktir. Bunları karşılaştırmak bize kaydedilmemiş değişikliklerin olup olmadığını söyler. Tasarım basit ama insanlara huzur veriyor. Hiç kimse üzerinde çok çalıştığı şeyin yok olmasını istemez.

### 2. Tasarım Dosyalarını Çevrimiçi Kaynaktan İçe Aktarın

#### Dizin Yapısı

```
repos/index/
└── src/data/public/design.json    # Design template index

repos/awesome-design-md-site/
├── vendor/awesome-design-md/       # Upstream design templates
│   └── design-md/
│       ├── clickhouse/
│       │   └── DESIGN.md
│       ├── linear/
│       │   └── DESIGN.md
│       └── ...
└── src/lib/content/
    └── awesomeDesignCatalog.ts     # Content pipeline
```

#### Dizin Veri Formatı

Tasarım sitesindeki indeks dosyası mevcut tüm şablonları tanımlar. Bu dizin sayesinde kullanıcılar istedikleri şablonu bir menüden sipariş verir gibi kolayca seçebilirler:

```json
{
  "entries": [
    {
      "slug": "linear.app",
      "title": "Linear Inspired Design System",
      "summary": "AI Product / Dark Feel",
      "detailUrl": "/designs/linear.app/",
      "designDownloadUrl": "/designs/linear.app/DESIGN.md",
      "previewLightImageUrl": "...",
      "previewDarkImageUrl": "..."
    }
  ]
}
```

Her giriş, şablonun temel bilgilerini ve indirme bağlantılarını içerir. Arka uç, bu dizindeki mevcut şablonların listesini okur ve bunları kullanıcının aralarından seçim yapması için sunar. Bu, insanları karanlıkta yollarını bulmaya zorlamak yerine seçimi sezgisel hale getiriyor.

#### Aynı Kökenli Proxy Uygulaması

İşleri güvende tutmak için arka uç, tasarım sitesine erişimde sıkı bir doğrulama gerçekleştirir. Güvenlik konusunda çok dikkatli olamazsınız:

```csharp
// Safe slug validation
private static readonly Regex SafeDesignSiteSlugRegex =
    new("^[A-Za-z0-9](?:[A-Za-z0-9._-]{0,127})$", RegexOptions.Compiled);

private static string NormalizeDesignSiteSlug(string slug)
{
    var normalizedSlug = slug?.Trim() ?? string.Empty;
    if (!IsSafeDesignSiteSlug(normalizedSlug))
    {
        throw new BusinessException(
            ProjectDesignSiteIndexErrorCodes.InvalidSlug,
            "Design site slug must be a single safe path segment.");
    }
    return normalizedSlug;
}

// Preview image caching (OS temp directory)
private static string ComputePreviewCacheKey(string slug, string theme, string previewUrl)
{
    var raw = $"{slug}|{theme}|{previewUrl}";
    var bytes = SHA256.HashData(Encoding.UTF8.GetBytes(raw));
    return Convert.ToHexString(bytes).ToLowerInvariant();
}
```

Burada iki şey yapıyoruz: birincisi, yol geçiş saldırılarını önlemek için sümüklü böcek formatını kesinlikle düzenli bir ifadeyle doğrularız; ikincisi, harici site üzerindeki baskıyı azaltmak için önizleme resimlerini önbelleğe alıyoruz. Birincisi koruma, ikincisi ise optimizasyondur ve her ikisi de önemlidir.

### 3. Tam İthalat Akışı

```typescript
// 1. Open the import drawer
const handleRequestImportDrawer = useCallback(() => {
    setIsImportDrawerOpen(true);
}, []);

// 2. Select and import
const handleImportRequest = useCallback((entry) => {
    if (isDirty) {
        setPendingImportEntry(entry);
        setConfirmMode('import'); // overwrite confirmation
        return;
    }
    void executeImport(entry);
}, [isDirty]);

// 3. Execute import
const executeImport = useCallback(async (entry) => {
    const result = await getProjectDesignMdSiteImportDocument(
        activeTarget.projectId,
        entry.slug
    );
    setDraft(result.content); // replace editor text only, do not save automatically
    setIsImportDrawerOpen(false);
}, [activeTarget?.projectId]);
```

İçe aktarma akışı bir "kullanıcı onayı" ilkesini izler: içe aktarmanın ardından yalnızca düzenleyici içeriği güncellenir ve hiçbir şey otomatik olarak kaydedilmez. Kullanıcılar içe aktarılan içeriği inceleyebilir ve yalnızca doğru göründüğünü onayladıktan sonra manuel olarak kaydedebilir. Nihai karar kullanıcının elinde kalmalıdır.

## Pratik Örnekler

### Senaryo 1: Proje Kökünde DESIGN.md Oluşturma

Ne zaman `DESIGN.md` mevcut değilse, arka uç bir sanal belge durumu döndürür. Bu, ön ucun "dosya mevcut değil" durumu için özel işlemlerden kaçınmasına olanak tanır ve birleştirilmiş bir API, kod mantığını basitleştirir:

```csharp
return new ProjectDesignDocumentDto
{
    Path = targetPath,
    Exists = false,  // virtual document state
    Content = string.Empty,
    Version = null
};

// Automatically create the file on first save
public async Task<SaveProjectDesignDocumentResultDto> SaveDesignDocumentAsync(...)
{
    Directory.CreateDirectory(targetDirectory);
    await File.WriteAllTextAsync(targetPath, input.Content);
    return new SaveProjectDesignDocumentResultDto { Created = !exists };
}
```

Bu tasarımın yararı, ön ucun eksik dosyalar için özel durum mantığına ihtiyaç duymamasıdır. Ön uç, bu karmaşıklığı arka uçta gizleyerek kullanıcı deneyimine daha kolay odaklanabilir.

### Senaryo 2: Tasarım Sitesinden Şablon İçe Aktarma

Kullanıcı içe aktarma çekmecesinde "Doğrusal" tasarım şablonunu seçtikten sonra sistem, `DESIGN.md` arka uç proxy aracılığıyla içerik. Tüm süreç kullanıcı için şeffaftır: yalnızca bir şablon seçerler ve sistem, ağ isteklerini ve veri dönüşümünü otomatik olarak gerçekleştirir.

```typescript
// 1. The system fetches DESIGN.md content through the backend proxy
GET /api/project/{id}/design-md/site-index/linear.app

// 2. The backend validates the slug and fetches content from upstream
var entry = FindDesignSiteEntry(catalog, "linear.app");
using var upstreamResponse = await httpClient.SendAsync(request);
var content = await upstreamResponse.Content.ReadAsStringAsync();

// 3. The frontend replaces the editor text
setDraft(result.content);
// The user reviews it and then saves it manually to disk
```

Akışın tamamı kullanıcıya şeffaf kalır. Sadece bir şablon seçerler ve sistem, ağ oluşturmayı ve perde arkasındaki dönüşümü yönetir. İstediğimiz deneyim budur: basit ama güçlü.

### Senaryo 3: Sürüm Çakışmalarını Ele Alma

Birden fazla kullanıcı aynı şeyi düzenlediğinde `DESIGN.md` aynı zamanda sistem sürüm çakışmalarını da algılar. Bu iyimser eşzamanlılık kontrol mekanizması, kullanıcının düzenlemelerini engellemeden veri tutarlılığını korur:

```csharp
if (!string.Equals(currentVersion, expectedVersion, StringComparison.Ordinal))
{
    throw new BusinessException(
        ProjectDesignDocumentErrorCodes.VersionConflict,
        $"DESIGN.md at '{targetPath}' changed on disk.");
}
```

Ön uç bu hatayı yakalar ve kullanıcıya şunu sorar:

```typescript
// Frontend prompts the user to refresh and retry
<Alert>
    <AlertTitle>Version conflict</AlertTitle>
    <AlertDescription>
        The file was modified by another process. Please refresh to get the latest version and try again.
    </AlertDescription>
</Alert>
```

Bu iyimser eşzamanlılık kontrol mekanizması, kullanıcıları çalışırken engellemeden verilerin tutarlı kalmasını sağlar. Çatışmalar kaçınılmazdır, ancak en azından kullanıcılar, değişikliklerini sessizce kaybetmek yerine ne olduğunu bilmelidir.

## Notlar ve En İyi Uygulamalar

### 1. Yol Güvenliği

Her zaman doğrula `repositoryPath` Yol geçiş saldırılarını önlemek için. Güvenlik söz konusu olduğunda asla çok fazla şey yapamazsınız:

```csharp
// Always validate repositoryPath to prevent path traversal attacks
return ValidateSubPathAsync(projectPath, repositoryPath);
// Reject dangerous inputs such as "../" and absolute paths
```

### 2. Önbellek Stratejisi

Önizleme görüntülerini maksimum 160 dosyayla 24 saat boyunca önbelleğe alın. Orta düzeyde önbelleğe alma performansı artırır ancak denge hâlâ önemlidir:

```csharp
// Cache preview images for 24 hours, with a maximum of 160 files
private static readonly TimeSpan PreviewCacheTtl = TimeSpan.FromHours(24);
private const int PreviewCacheMaxFiles = 160;
// Periodically clean up expired cache
```

### 3. Hata İşleme

Yukarı akış sitesi kullanılamadığında zarif bir şekilde bozulur. Bu tasarım, harici bir bağımlılık başarısız olsa bile temel düzenleme işlevinin normal şekilde çalışmaya devam etmesini sağlar:

```typescript
// Gracefully degrade when the upstream site is unavailable
try {
    const catalog = await getProjectDesignMdSiteImportCatalog(projectId);
} catch (error) {
    toast.error(t('project.designMd.siteImport.feedback.catalogLoadFailed'));
    // The main editing drawer remains available
}
```

Bu zarif bozulma, harici bağımlılıklar mevcut olmadığında bile temel düzenleme işlevinin çalışmaya devam etmesini sağlar. Bir şeyler ters gittiği anda çökmek yerine sistemin dayanıklı olması gerekir.

### 4. Kullanıcı Deneyimi Optimizasyonu

Üzerine yazma işlemlerini içe aktarmadan önce onaylayın ve içe aktarmanın ardından otomatik olarak kaydetmeyin. Kullanıcılar kendi eylemlerinin kontrolünü elinde tutmalıdır:

```typescript
// Confirm overwrite before import
if (isDirty) {
    setConfirmMode('import');
    return;
}

// Do not save automatically after import; let the user confirm
setDraft(result.content); // update draft only
// The content is written to disk only after the user reviews it and clicks Save
```

### 5. Performans Hususları

Çok fazla bağlantı oluşturmaktan kaçınmak için bir HTTP istemci fabrikası kullanın. Kaynak yönetimi küçük görünebilir ancak bunu iyi yapmak büyük bir fark yaratabilir:

```csharp
// Use an HTTP client factory to avoid creating too many connections
private const string DesignSiteProxyClientName = "ProjectDesignSiteProxy";
private static readonly TimeSpan DesignSiteProxyTimeout = TimeSpan.FromSeconds(8);
```

## Önerilen Uzantılar

1. **Markdown iyileştirmesi**: şu anda temel bir ürün kullanıyoruz `Textarea`, ancak sözdizimi vurgulama ve klavye kısayolları için CodeMirror'a yükseltebiliriz. Editör kendini daha iyi hissettiğinde belge yazmak da daha iyi hissettirir.
2. **Önizleme modu**: Düzenleme deneyimini geliştirmek için gerçek zamanlı Markdown önizlemesini ekleyin. Gördüğünüz şey, aldığınız şeydir, insanlara her zaman daha fazla güven verir.
3. **Fark birleştirme**: Basit tam metin değiştirme yerine akıllı bir birleştirme algoritması uygulayın. Çatışmalar kaçınılmazdır ancak çatışma çözümleme sürecinin sancılı olması gerekmez.
4. **Yerel önbelleğe alma**: önbellek `design.json` Harici siteye bağımlılığı azaltmak için veritabanında. Bir sistemin bağımlılığı ne kadar azsa, o kadar kararlı olma eğilimindedir.
