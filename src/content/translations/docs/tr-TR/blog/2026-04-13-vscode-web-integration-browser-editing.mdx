## VSCode ve kod sunucusu: Tarayıcı Tabanlı Kod Düzenleme Çözümü Seçme

> Tarayıcı tabanlı kod düzenleme yetenekleri geliştirirken geliştiriciler önemli bir seçimle karşı karşıya kalır: VSCode'un resmi sürümünü kullanın `code serve-web` özelliği kullanın veya topluluk odaklı özelliği benimseyin `code-server` çözüm? Bu karar yalnızca teknik mimariyi değil aynı zamanda lisans uyumluluğunu ve dağıtım esnekliğini de etkiler.

## Arka plan

Teknik seçim hayatta bir yol seçmeye çok benzer. Birini seçtikten sonra genellikle yürümeye devam etmek zorunda kalırsınız ve daha sonra geçiş yapmak çok pahalı olabilir.

Yapay zeka destekli programlama çağında, tarayıcı tabanlı kod düzenleme giderek daha önemli hale geliyor. Kullanıcılar, bir AI asistanının kodu analiz etmeyi bitirdikten sonra, aynı tarayıcı oturumunda hemen bir düzenleyici açabilmesini ve uygulamaları değiştirmeden değişiklik yapabilmesini bekler. Bu tür kusursuz bir deneyim, ihtiyaç duyduğunuz anda yanınızda olmalıdır.

Ancak geliştiriciler bu özelliği uygularken kritik bir teknik seçimle karşı karşıya kalırlar: VSCode'un resmi sürümünü kullanmalılar mı? `code serve-web` özellik veya topluluk odaklı `code-server` çözüm?

Her seçeneğin kendine göre güçlü yönleri ve değiş tokuşları vardır ve kötü seçim yapmak daha sonra çok fazla sorun yaratabilir. Lisanslama buna bir örnektir: Ürününüzün lisansla uyumlu olmadığını lansmandan sonra keşfederseniz, artık çok geç demektir. Dağıtım başka bir şeydir: Bir çözüm, geliştirme aşamasında mükemmel bir şekilde çalışabilir, ardından konteynerlere taşındığında her türlü sorunla karşılaşabilir. Bunlar tam olarak ekiplerin kaçınmak istediği türden tuzaklardır.

## Lisanslama Farklılıkları (En Önemlisi)

Bu, iki çözüm arasındaki en temel farktır ve değerlendirme sırasında dikkate aldığımız ilk faktördür. Teknik bir seçim yaparken yasal riskleri önceden anlamak önemlidir.

**kod sunucusu**
- MIT lisansı, tamamen açık kaynak
- Coder.com tarafından aktif bir toplulukla sürdürülmektedir
- Ticari olarak kullanmak, değiştirmek ve dağıtmak ücretsizdir
- Kullanım senaryolarında kısıtlama yok

**VSCode `code serve-web`**
- Microsoft VSCode ürününün bir parçası
- Microsoft'un lisansını kullanır (VS Code lisansı ticari kullanıma ilişkin kısıtlamaları içerir)
- Öncelikle bireysel geliştiricilere yöneliktir
- Kurumsal dağıtım ek ticari lisans incelemesi gerektirebilir

Lisanslama açısından bakıldığında kod sunucusu ticari projelere daha uygundur. Bu, ürün planlaması sırasında düşünmeniz gereken bir şeydir çünkü daha sonra geçiş yapmak çok maliyetli olabilir.

## Dağıtım Farklılıkları

Lisanslama işlemi tamamlandıktan sonra bir sonraki sorun dağıtımdır. Bu, operasyon maliyetinizi ve mimari tasarımınızı doğrudan etkiler.

**kod sunucusu**
- Bağımsız olarak dağıtılabilen bağımsız bir Node.js uygulaması
- Birden fazla çalışma zamanı kaynağını destekler:
  - Yürütülebilir yolu doğrudan belirtme
  - PATH sistemi aracılığıyla aranıyor
  - NVM Node.js 22.x ortamının otomatik algılanması
- Sunucuya VSCode masaüstü uygulamasını yüklemenize gerek yok
- Konteynerlerde dağıtımı daha kolay

**VSCode `code serve-web`**
- Yerel olarak kurulmuş bir VSCode CLI'ye bağlı olmalıdır
- Kullanılabilir bir ortam gerektirir `code` ana makinedeki komut
- Sistem, VS Code Remote CLI sarmalayıcılarını filtreler
- Öncelikle yerel kalkınma senaryoları için tasarlandı

code-server, sunucu ve konteyner dağıtım senaryoları için daha uygundur. Ürününüzün Docker'da çalışması gerekiyorsa veya kullanıcılarınızda VSCode yüklü değilse kod sunucusu genellikle doğru seçimdir.

## Özellik Parametre Farklılıkları

İki çözüm aynı zamanda birkaç özellik parametresinde de farklılık göstermektedir. Farklılıklar çok büyük değil ancak gerçek dünya kullanımında entegrasyon sürtüşmeleri yaratabilirler.

| Özellik | kod sunucusu | kod servisi-web |
|------|-------------|----------------|
| Genel taban yolu | `/` (yapılandırılabilir) | `/vscode-server` (sabit) |
| Kimlik doğrulama | `--auth` çoklu modlara sahip parametre | `--connection-token` / `--without-connection-token` |
| Veri dizini | `{DataDir}/code-server` | `{DataDir}/vscode-serve-web` |
| Telemetri | ile varsayılan olarak devre dışıdır `--disable-telemetry` | VSCode ayarlarına bağlıdır |
| Kontrolleri güncelle | İle devre dışı bırakılabilir `--disable-update-check` | VSCode ayarlarına bağlıdır |

Entegrasyon sırasında bu farklılıklara özel dikkat gösterilmesi gerekmektedir. Örneğin, farklı URL yolları, ön uç kodunuzun özel olarak işlenmesi gerektiği anlamına gelir.

## Kullanılabilirlik Algılama Farklılıkları

Düzenleyici değiştirmeyi uygularken kullanılabilirlik algılama mantığı da farklılık gösterir.

**kod sunucusu**
- Her zaman görünür bir uygulama olarak geri döndü
- Kullanılamadığında bile yine de gösterilir `install-required` durum
- NVM Node.js 22.x ortamının otomatik algılanmasını destekler

**kod servis-web**
- Yalnızca yerel olduğunda görünür `code` CLI algılandı
- Mevcut değilse ön uç bu seçeneği otomatik olarak gizler
- Yerel VSCode kurulum durumuna bağlıdır

Bu fark kullanıcı deneyimini doğrudan etkiliyor. kod sunucusu daha şeffaftır: kullanıcılar seçeneği görebilir ve kurulumun hala gerekli olduğunu anlayabilir. code serve-web daha gizlidir: kullanıcılar bu seçeneğin varlığının farkına bile varmayabilir. Hangi yaklaşımın daha iyi olduğu ürün konumlandırmasına bağlıdır.

## HagiCode'un İkili Uygulama Mimarisi

Derinlemesine analizin ardından HagiCode projesi, her iki çözümü de mimari düzeyde destekleyen ikili uygulamalı bir mimariyi benimsedi.

### Varsayılan olarak kod sunucusuna geçiş

```csharp
// The default active implementation is code-server
// If an explicit activeImplementation is saved, try that implementation first
// If the requested implementation is unavailable, the resolver tries the other one
// If a fallback occurs, return fallbackReason
```

Lisanslama ve dağıtım esnekliği nedeniyle varsayılan olarak kod sunucusunu tercih ediyoruz. Ancak halihazırda yerel bir VSCode ortamına sahip olan kullanıcılar için code serve-web de sağlam bir seçenektir.

### Uygulama Seçici

`CodeServerImplementationResolver` birleştirmekten sorumludur:
- Başlangıç ısınması sırasında uygulama seçimi
- Durum okunurken uygulama seçimi
- Projeleri açarken uygulama seçimi
- Kasaları açarken uygulama seçimi

Bu tasarım, sistemin farklı senaryolara esnek bir şekilde yanıt vermesini sağlar ve kullanıcılar, ortamlarına en uygun uygulamayı seçebilir.

### Ön Uç Uyarlama Kuralları

```typescript
// When localCodeAvailable=false, do not show code serve-web
// When localCodeAvailable=true, show the code serve-web configuration
```

Ön uç, ortama bağlı olarak mevcut seçenekleri otomatik olarak gösterir, böylece kullanıcıların kullanamayacakları özellikler nedeniyle kafaları karışmaz.

## Pratik Kılavuz

Bütün bu teoriden sonra, fiili dağıtım sırasında nelere dikkat etmelisiniz? Sonuçta önemli olan uygulamadır.

### Docker Dağıtım Önerisi

Konteynerli dağıtım için kod sunucusu daha iyi bir seçimdir:

```dockerfile
# Use the official code-server image directly
FROM codercom/code-server:latest

# Or install through npm
RUN npm install -g code-server
```

Bu, ek bir VSCode kurulumu gerektirmeden sorunu tek katmanda çözer.

### Yapılandırma Örnekleri

**kod sunucusu yapılandırması**
```json
{
  "vscodeServer": {
    "enabled": true,
    "activeImplementation": "code-server",
    "codeServer": {
      "host": "0.0.0.0",
      "port": 8080,
      "executablePath": "",
      "authMode": "none"
    }
  }
}
```

**kod hizmeti-web yapılandırması**
```json
{
  "vscodeServer": {
    "enabled": true,
    "activeImplementation": "serve-web",
    "serveWeb": {
      "host": "0.0.0.0",
      "port": 8080,
      "executablePath": "/usr/local/bin/code"
    }
  }
}
```

Yapılandırma ilk seferde biraz sıkıcı olabilir, ancak bir kez yerine oturduğunda bakımı çok daha kolay hale gelir.

### URL Yapısı Farklılıkları

**kod sunucusu**
```
http://localhost:8080/?folder=/path/to/project&vscode-lang=zh-CN
```

**kod servis-web**
```
http://localhost:8080/vscode-server/?folder=/path/to/project&tkn=xxx&vscode-lang=zh-CN
```

Yollar ve parametrelerdeki farklılıklara dikkat edin. Entegrasyon sırasında bunları ayrı ayrı ele almanız gerekir.

### Uygulamaları Değiştirme

Sistem, çalışma zamanı geçişini destekler ve geçiş sırasında önceki uygulamayı otomatik olarak durdurur:

```csharp
// VsCodeServerManager automatically handles mutual exclusion
// When switching activeImplementation, the old implementation will not keep running in the background
```

Bu tasarım, kullanıcıların istedikleri zaman farklı uygulamaları denemelerine ve kendileri için en uygun seçeneği bulmalarına olanak tanır.

### Durum İzleme

```typescript
const { settings, runtime } = await getVsCodeServerSettings();

// runtime.activeImplementation: "code-server" | "serve-web"
// runtime.fallbackReason: reason for switching
// runtime.status: "running" | "starting" | "stopped" | "unhealthy"
```

Durum görünür olduğunda kullanıcılar, sorunun sunucu tarafından mı yoksa kendi işlemlerinden mi kaynaklandığını hızlı bir şekilde belirleyebilir.
