# OpenSpec 소개

OpenSpec는 구조화된 제안과 사양을 통해 소프트웨어 개발 프로세스를 관리하는 사양 기반 개발 도구입니다. 이 안내서는 OpenSpec의 핵심 개념, 워크플로 및 사용 방법을 포괄적으로 소개합니다.

## OpenSpec란?

OpenSpec는 소프트웨어 개발 과정에서 제안(Proposals), 사양(Specifications), 변경(Changes)을 관리하기 위한 커맨드라인 도구입니다. 개발 워크플로를 표준화하여 아이디어에서 구현까지 모든 기능이 충분한 설계와 검토를 거치도록 보장합니다.

### 핵심 원칙

OpenSpec는 다음 핵심 원칙을 기반으로 합니다:

- **사양 우선**: 코드를 작성하기 전에 명확한 사양을 정의합니다
- **구조화된 협업**: 표준화된 문서 형식을 통해 팀 협업을 촉진합니다
- **추적 가능성**: 제안에서 구현까지의 전체 과정을 기록합니다
- **품질 보증**: 사양 검증을 통해 코드가 설계 의도에 부합하는지 확인합니다

### 주요 기능

OpenSpec는 다음과 같은 주요 기능을 제공합니다:

1. **제안 관리**: 기능 제안의 생성, 검토 및 추적
2. **사양 정의**: 기술 사양 문서 작성 및 유지
3. **변경 추적**: 검토된 모든 변경 이력을 기록
4. **사양 검증**: 코드 변경이 사전에 정의된 사양을 준수하는지 확인
5. **팀 협업**: 다수의 개발자 간 공동 개발 및 검토 프로세스를 지원

## OpenSpec 워크플로

일반적인 OpenSpec 워크플로는 다음 단계를 포함합니다:

### 1. 제안(Proposal) 생성

새 기능 또는 큰 변경 요구가 있을 때 먼저 제안을 생성합니다:

```bash
openspec proposal create
```

제안에는 다음이 포함됩니다:
- **문제 설명**: 해결해야 할 문제 또는 요구사항
- **해결책**: 제안된 해결책의 개요
- **영향 범위**: 기존 시스템에 미치는 영향
- **대안**: 고려된 다른 해결책

### 2. 사양(Specification) 작성

제안이 승인되면 자세한 기술 사양을 작성합니다:

```bash
openspec spec create
```

사양에는 다음이 포함됩니다:
- **기술 설계**: 상세한 기술 구현 계획
- **인터페이스 정의**: API, 데이터 구조 등 인터페이스 설계
- **구현 세부사항**: 핵심 구현 로직 및 알고리즘
- **테스트 계획**: 구현의 정확성을 검증하는 방법

### 3. 변경(Change) 생성

구현을 시작할 때 개발 과정을 추적하기 위해 변경을 생성합니다:

```bash
openspec change create
```

변경에는 다음이 포함됩니다:
- **연관된 사양**: 이 변경이 구현하는 사양
- **구현 작업**: 구체적인 개발 작업 목록
- **진행 추적**: 작업 완료 상태
- **코드 변경**: 실제 코드 수정사항

### 4. 검증 및 보관

개발을 완료한 후 변경이 사양을 준수하는지 검증하고 보관합니다:

```bash
openspec change validate
openspec change archive
```

## OpenSpec 디렉터리 구조

OpenSpec는 프로젝트 루트에 `openspec/` 디렉터리를 생성하며, 구조는 다음과 같습니다:

```
openspec/
├── proposals/           # 제안 디렉터리
│   ├── PROPOSAL-001.md
│   └── PROPOSAL-002.md
├── specs/              # 사양 디렉터리
│   ├── SPEC-001.md
│   └── SPEC-002.md
├── changes/            # 변경 디렉터리
│   ├── feature-name/
│   │   ├── proposal.md
│   │   ├── design.md
│   │   ├── specs.md
│   │   └── tasks.md
│   └── another-feature/
└── archived/           # 보관된 변경
    └── feature-name/
```

### 디렉터리 설명

- **proposals/**: 모든 제안 문서를 저장하며, 제안당 하나의 독립적인 Markdown 파일
- **specs/**: 제안에 대응하는 기술 사양 문서를 저장
- **changes/**: 진행 중인 변경을 저장하며, 변경당 하나의 하위 디렉터리
- **archived/**: 완료되어 보관된 변경을 저장

## 핵심 명령어

### 제안 관련 명령어

```bash
# 새 제안 생성
openspec proposal create

# 모든 제안 목록
openspec proposal list

# 제안 상세 보기
openspec proposal view <proposal-id>

# 제안 승인
openspec proposal approve <proposal-id>
```

### 사양 관련 명령어

```bash
# 새 사양 생성
openspec spec create

# 모든 사양 목록
openspec spec list

# 사양 상세 보기
openspec spec view <spec-id>

# 사양 업데이트
openspec spec update <spec-id>
```

### 변경 관련 명령어

```bash
# 새 변경 생성
openspec change create

# 모든 변경 목록
openspec change list

# 변경 상세 보기
openspec change view <change-name>

# 작업 추가
openspec change add-task <change-name>

# 변경 검증
openspec change validate <change-name>

# 변경 보관
openspec change archive <change-name>
```

### 일반 명령어

```bash
# 버전 보기
openspec --version

# 도움말 보기
openspec --help

# 특정 명령어의 도움말 보기
openspec <command> --help
```

## HagiCode 프로젝트에서 OpenSpec 사용

### 프로젝트 통합

HagiCode 프로젝트에서는 OpenSpec가 개발 워크플로에 통합되어 있습니다:

1. **초기화 검사**: 시작 마법사가 OpenSpec 설치 여부를 확인합니다
2. **제안 기반 개발**: 모든 새 기능은 OpenSpec 제안 프로세스를 거쳐야 합니다
3. **코드 리뷰**: 변경 사항은 OpenSpec 사양을 준수해야 병합됩니다
4. **문서 동기화**: OpenSpec 문서는 코드 구현과 동기화됩니다

### 버전 요구사항

HagiCode 프로젝트는 **OpenSpec >=1.0.0 <2.0.0** 버전 범위를 요구합니다.

설치 명령:

```bash
npm install -g @fission-ai/openspec@1
```

### 워크플로 예시

다음은 HagiCode 프로젝트에서 OpenSpec를 사용하는 전체 예시입니다:

#### 단계 1: 제안 생성

```bash
cd /path/to/hagicode-project
openspec proposal create
```

대화형 프롬프트에 입력:
- 제안 제목: 사용자 인증 기능 추가
- 문제 설명: 시스템에 로그인 및 권한 관리 필요
- 해결책: JWT 기반 인증 시스템 구현

#### 단계 2: 사양 작성

제안이 승인되면 상세 사양을 작성합니다:

```bash
openspec spec create
```

사양에 정의할 내용:
- 인증 흐름 설계
- API 인터페이스 정의
- 데이터베이스 테이블 구조
- 보안 고려사항

#### 단계 3: 변경 생성

구현을 시작합니다:

```bash
openspec change create
```

변경 이름 입력: `add-user-authentication`

#### 단계 4: 작업 추가

```bash
openspec change add-task add-user-authentication
```

구체적인 작업을 추가합니다:
- 사용자 등록 엔드포인트 구현
- 로그인 엔드포인트 구현
- JWT 미들웨어 구현
- 단위 테스트 작성

#### 단계 5: 구현 및 검증

코드를 완료한 후 변경을 검증합니다:

```bash
openspec change validate add-user-authentication
```

#### 단계 6: 보관

검증이 통과하면 변경을 보관합니다:

```bash
openspec change archive add-user-authentication
```

## 모범 사례

### 제안 작성

1. **명확한 문제 진술**: 해결하려는 문제를 명확히 정의하고 모호한 설명을 피합니다
2. **대안 고려**: 다양한 해결책을 나열하고 비교합니다
3. **영향 평가**: 기존 시스템에 대한 영향과 위험을 분석합니다
4. **간결성 유지**: 제안은 간결해야 하며, 세부 사항은 사양에 남깁니다

### 사양 작성

1. **상세하지만 중복은 피함**: 충분한 세부 정보를 제공하되 과도한 설계는 피합니다
2. **도표 사용**: 플로우차트나 아키텍처 다이어그램으로 설명을 보완합니다
3. **인터페이스 정의**: 모든 공용 인터페이스 및 데이터 구조를 명확히 정의합니다
4. **경계 사례 고려**: 예외 상황 및 오류 처리를 나열합니다

### 변경 관리

1. **작은 반복**: 큰 기능은 여러 작은 변경으로 나눕니다
2. **즉시 업데이트**: 작업 상태 및 문서를 동기화하여 최신 상태로 유지합니다
3. **코드 리뷰**: 변경 완료 후 코드 리뷰를 수행합니다
4. **빠른 보관**: 완료된 변경은 신속히 보관하여 작업 디렉터리를 정리합니다

### 팀 협업

1. **정기적인 검토**: 제안 및 사양을 정기적으로 검토하여 팀 합의를 유지합니다
2. **문서 우선**: 코딩을 시작하기 전에 문서를 완료합니다
3. **소통 유지**: 제안 및 사양에 논의와 결정을 기록합니다
4. **버전 관리**: OpenSpec 문서를 버전 관리에 포함시킵니다

## 일반적인 사용 사례

### 시나리오 1: 새 기능 추가

1. 새 기능 요구를 설명하는 제안을 생성
2. 팀에서 제안을 논의하고 승인
3. 상세 기술 사양 작성
4. 변경 생성 및 기능 구현
5. 변경 검증 및 보관

### 시나리오 2: 기존 코드 리팩터링

1. 리팩터링의 이유와 목표를 설명하는 제안 생성
2. 사양에서 리팩터링 계획 및 단계 정의
3. 변경 생성 후 단계적으로 리팩터링 수행
4. 테스트 커버리지를 확보하고 리팩터링 결과를 검증
5. 변경 보관

### 시나리오 3: 버그 수정

간단한 버그의 경우:
- 직접 수정 가능하며 전체 OpenSpec 프로세스가 필요하지 않을 수 있음

복잡한 버그의 경우:
1. 버그의 근본 원인을 분석하는 제안 생성
2. 사양에서 수정 방법 정의
3. 변경 생성하고 수정 구현
4. 회귀 방지를 위한 테스트 추가
5. 변경 보관

### 시나리오 4: 기술 부채 관리

1. 기술 부채 항목을 나열하는 제안 생성
2. 사양에서 개선 계획을 정의
3. 변경 생성하여 단계적으로 부채를 갚아 나감
4. 진행 상황과 효과 추적
5. 완료된 개선을 보관

## 다른 도구와의 통합

### Git 통합

OpenSpec는 Git와 원활하게 통합됩니다:

```bash
# OpenSpec 문서는 버전 관리에 포함되어야 합니다
git add openspec/
git commit -m "Add OpenSpec proposal for feature X"
```

### CI/CD 통합

CI/CD 파이프라인에서 OpenSpec 사양을 검증합니다:

```yaml
# .github/workflows/openspec.yml
name: OpenSpec Validation
on: [pull_request]
jobs:
  validate:
    runs-on: ubuntu-latest
    steps:
      - uses: actions/checkout@v2
      - name: Install OpenSpec
        run: npm install -g @fission-ai/openspec@1
      - name: Validate changes
        run: openspec change validate
```

### IDE 통합

OpenSpec 문서를 편집하기 위해 Markdown을 지원하는 에디터를 권장합니다:
- VS Code + Markdown 플러그인
- JetBrains IDE + Markdown 지원
- Vim/Neovim + Markdown 플러그인

## 문제 해결

### 일반적인 문제

#### 문제: OpenSpec 명령을 찾을 수 없음

**해결책**:
1. OpenSpec가 설치되어 있는지 확인: `npm list -g @fission-ai/openspec`
2. PATH 환경 변수에 npm 전역 bin 디렉터리가 포함되어 있는지 확인
3. 재설치: `npm install -g @fission-ai/openspec@1`

#### 문제: 제안 또는 변경을 생성할 수 없음

**해결책**:
1. 현재 디렉터리가 프로젝트 루트인지 확인
2. `openspec/` 디렉터리가 존재하는지 확인하고, 없으면 생성
3. 쓰기 권한을 확인

#### 문제: 검증 실패

**해결책**:
1. 변경 디렉터리 내의 문서가 완전한지 확인
2. 필요한 작업이 모두 완료되었는지 확인
3. 자세한 오류 정보 확인: `openspec change validate <change-name> --verbose`

#### 문제: 아카이브 실패

**해결책**:
1. 변경이 검증을 통과했는지 확인
2. `openspec/archived/` 디렉터리가 존재하는지 확인
3. 커밋되지 않은 코드 변경이 없는지 확인

## 고급 사용법

### 사용자 정의 템플릿

OpenSpec는 사용자 정의 제안 및 사양 템플릿을 지원합니다:

```bash
# 프로젝트 루트에 템플릿 디렉터리 생성
mkdir -p .openspec/templates

# 사용자 정의 제안 템플릿 생성
cat > .openspec/templates/proposal.md << 'EOF'
# Proposal: {{title}}

## Problem Statement
{{problem}}

## Proposed Solution
{{solution}}

## Custom Section
Your custom content here
EOF
```

### 구성 파일

프로젝트 루트에 `.openspec.json` 구성 파일을 생성합니다:

```json
{
  "version": "1.0",
  "templates": {
    "proposal": ".openspec/templates/proposal.md",
    "spec": ".openspec/templates/spec.md"
  },
  "validation": {
    "requireTests": true,
    "requireDocs": true
  },
  "hooks": {
    "pre-archive": "npm test"
  }
}
```

### 훅(Hooks)

OpenSpec는 특정 작업 전후에 커스텀 스크립트를 실행하는 것을 지원합니다:

```json
{
  "hooks": {
    "pre-create": "scripts/check-permissions.sh",
    "post-archive": "scripts/notify-team.sh"
  }
}
```
