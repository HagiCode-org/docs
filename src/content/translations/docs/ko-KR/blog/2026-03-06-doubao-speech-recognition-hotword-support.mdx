## 두바오 음성 인식 핫워드 기능 구현 가이드

> 이 글에서는 HagiCode 프로젝트에서 두바오 음성 인식의 핫워드 지원 기능을 구현하는 방법을 상세히 소개합니다. 커스텀 핫워드와 플랫폼 핫워드 테이블 두 가지 방식을 통해 특정 도메인 어휘의 인식 정확도를 크게 향상시킵니다.

## 배경

음성 인식 기술이 오랫동안 발전해 왔지만, 개발자들을 오랫동안 괴롭힌 문제가 하나 있습니다. 범용 음성 인식 모델은 일상 용어를 커버할 수 있지만, 전문 용어, 제품 이름, 인물 이름에 대해서는 인식 정확도가 항상 조금 부족합니다. 의료 분야의 음성 어시스턴트가 "고혈압", "당뇨병", "관상동맥경화"를 정확히 인식해야 하고, 법률 시스템이 "사유", "답변", "입증 책임"을 정밀하게 포착해야 하는 것을 생각해 보세요. 이러한 시나리오에서 범용 모델의 성능은 최선을 다했다고 해도 부족합니다.

HagiCode 프로젝트에서도 동일한 과제에 직면했습니다. 다기능 AI 코드 어시스턴트인 HagiCode는 다양한 기술 용어의 음성 인식 시나리오를 처리해야 합니다. 그러나 두바오 음성 인식 API는 기본적으로 전문 용어 정확도에 대한 우리의 요구를 완전히 충족하지 못했습니다. 사실 두바오가 부족한 것이 아니라, 각 분야마다 자체적인 용어 체계가 있기 때문입니다. 조사와 기술 탐색 끝에 두바오 음성 인식 API가 실제로 핫워드 지원 기능을 제공하며, 간단한 구성만으로 특정 어휘의 인식 정확도를 크게 향상시킬 수 있다는 것을 발견했습니다.

이 글에서 공유할 것은 HagiCode 프로젝트에서 두바오 음성 인식 핫워드 기능을 구현한 완전한 솔루션입니다. 두 가지 모드(커스텀 핫워드와 플랫폼 핫워드 테이블) 모두 사용 가능하며 조합하여 사용할 수도 있습니다. 이 솔루션을 통해 개발자는 비즈니스 시나리오에 따라 핫워드를 유연하게 구성하여 음성 인식 시스템이 전문적이고 희귀하지만 중요한 어휘를 "인식"할 수 있도록 할 수 있습니다.

## 핵심 구현

### 두 가지 핫워드 모드 분석

두바오 음성 인식 API는 두 가지 핫워드 구성 방식을 제공하며, 각 방식마다 고유한 응용 시나리오와 장점이 있습니다.

**커스텀 핫워드 모드**는 `corpus.context` 필드를 통해 핫워드 텍스트를 직접 전달할 수 있게 합니다. 이 방식은 소수의 핫워드를 빠르게 구성해야 하는 시나리오(예: 특정 제품 이름이나 인물 이름을 일시적으로 인식해야 할 때)에 매우 적합합니다. HagiCode 구현에서는 사용자가 입력한 다중 행 핫워드 텍스트를 문자열 목록으로 파싱한 다음 두바오 API의 요구 형식에 맞게 `context_data` 배열로 포맷팅합니다.

**플랫폼 핫워드 테이블 모드**는 `corpus.boosting_table_id` 필드를 통해 두바오 자학습 플랫폼에 미리 구성된 핫워드 테이블을 참조합니다. 이 방식은 대량의 핫워드를 관리해야 하는 시나리오에 적합합니다. 두바오 자학습 플랫폼에서 핫워드 테이블을 생성 및 유지보수하고 ID로 참조할 수 있습니다. HagiCode처럼 전문 용어를 지속적으로 업데이트하고 유지보수해야 하는 프로젝트에 더 나은 관리 가능성을 제공합니다.

흥미롭게도 이 두 모드는 조합하여 사용할 수 있습니다. 두바오 API는 동일한 요청에 커스텀 핫워드와 플랫폼 핫워드 테이블 ID를 모두 포함할 수 있으며, `combine_mode` 매개변수로 조합 전략을 제어합니다. 이 유연성 덕분에 HagiCode는 다양한 복잡한 전문 용어 인식 요구에 대응할 수 있습니다.

### 프론트엔드 타입 정의 및 검증

HagiCode의 프론트엔드 구현에서 완전한 핫워드 구성 타입과 검증 로직을 정의했습니다. 먼저 타입 정의:

```typescript
export interface HotwordConfig {
  contextText: string;           // 다중 행 핫워드 텍스트
  boostingTableId: string;      // 두바오 플랫폼 핫워드 테이블 ID
  combineMode: boolean;          // 조합 사용 여부
}
```

이 간단한 인터페이스는 핫워드 기능의 모든 구성 항목을 포함합니다. `contextText`는 사용자가 가장 직관적으로 느끼는 부분입니다 — 사용자가 한 줄에 하나의 핫워드 구를 입력하도록 허용하여 직관적인 UX를 제공합니다.

다음은 검증 함수의 구현입니다. 두바오 API의 제한을 고려하여 엄격한 검증 규칙을 제정했습니다: 핫워드 텍스트는 최대 100행, 행당 최대 50자, 총 최대 5000자; boosting_table_id는 최대 200자, 영문자, 숫자, 밑줄, 하이픈만 허용.

```typescript
export function validateContextText(contextText: string): HotwordValidationResult {
  if (!contextText || contextText.trim().length === 0) {
    return { isValid: true, errors: [] };
  }

  const lines = contextText.split('\n').filter(line => line.trim().length > 0);
  const errors: string[] = [];

  if (lines.length > 100) {
    errors.push(`热词行数不能超过 100 行，当前为 ${lines.length} 行`);
  }

  const totalChars = contextText.length;
  if (totalChars > 5000) {
    errors.push(`热词总字符数不能超过 5000，当前为 ${totalChars}`);
  }

  for (let i = 0; i < lines.length; i++) {
    if (lines[i].length > 50) {
      errors.push(`第 ${i + 1} 行热词超过 50 个字符限制`);
    }
  }

  return { isValid: errors.length === 0, errors };
}

export function validateBoostingTableId(boostingTableId: string): HotwordValidationResult {
  if (!boostingTableId || boostingTableId.trim().length === 0) {
    return { isValid: true, errors: [] };
  }

  const errors: string[] = [];

  if (boostingTableId.length > 200) {
    errors.push(`boosting_table_id 不能超过 200 个字符，当前为 ${boostingTableId.length}`);
  }

  if (!/^[a-zA-Z0-9_-]+$/.test(boostingTableId)) {
    errors.push('boosting_table_id 只能包含字母、数字、下划线和连字符');
  }

  return { isValid: errors.length === 0, errors };
}
```

이 검증 함수는 사용자가 핫워드를 구성할 때 즉시 실행되어 문제를 최초 단계에서 발견합니다.

### 프론트엔드 구성 지속화

HagiCode의 프론트엔드 구현에서 브라우저의 localStorage를 사용하여 핫워드 구성을 저장하기로 했습니다. 이 설계 결정에는 몇 가지 고려가 있습니다: 첫째, 핫워드 구성은 매우 개인화된 설정으로 다른 사용자가 다른 전문 분야 요구를 가질 수 있습니다. 둘째, 이 방식은 백엔드 구현을 단순화하여 추가 데이터베이스 테이블과 API 인터페이스가 필요 없습니다. 셋째, 사용자가 브라우저에서 한 번 구성하면 이후 사용 시 자동으로 로드되어 매우 편리합니다.

```typescript
const HOTWORD_STORAGE_KEYS = {
  contextText: 'hotword-context-text',
  boostingTableId: 'hotword-boosting-table-id',
  combineMode: 'hotword-combine-mode',
} as const;

export const DEFAULT_HOTWORD_CONFIG: HotwordConfig = {
  contextText: '',
  boostingTableId: '',
  combineMode: false,
};

export function loadHotwordConfig(): HotwordConfig {
  const contextText = localStorage.getItem(HOTWORD_STORAGE_KEYS.contextText) || '';
  const boostingTableId = localStorage.getItem(HOTWORD_STORAGE_KEYS.boostingTableId) || '';
  const combineMode = localStorage.getItem(HOTWORD_STORAGE_KEYS.combineMode) === 'true';

  return { contextText, boostingTableId, combineMode };
}

export function saveHotwordConfig(config: HotwordConfig): void {
  localStorage.setItem(HOTWORD_STORAGE_KEYS.contextText, config.contextText);
  localStorage.setItem(HOTWORD_STORAGE_KEYS.boostingTableId, config.boostingTableId);
  localStorage.setItem(HOTWORD_STORAGE_KEYS.combineMode, String(config.combineMode));
}
```

### 백엔드 SDK 구성 확장

HagiCode의 백엔드 구현에서 SDK 구성 클래스에 핫워드 관련 속성을 추가해야 합니다. C#의 언어 특성과 사용 관습을 고려하여 `List<string>`을 사용하여 커스텀 핫워드 컨텍스트를 저장합니다:

```csharp
public class DoubaoVoiceConfig
{
    public string AppId { get; set; } = string.Empty;
    public string AccessToken { get; set; } = string.Empty;
    public string ServiceUrl { get; set; } = string.Empty;
    public List<string>? HotwordContexts { get; set; }
    public string? BoostingTableId { get; set; }
}
```

### Payload 구성 로직

Payload 구성은 핫워드 기능의 핵심입니다. 핫워드 구성이 있으면 두바오 API의 요구 형식에 맞게 올바른 JSON 구조로 포맷팅해야 합니다:

```csharp
private void AddCorpusToRequest(Dictionary<string, object> request)
{
    var corpus = new Dictionary<string, object>();

    if (Config.HotwordContexts != null && Config.HotwordContexts.Count > 0)
    {
        corpus["context"] = new Dictionary<string, object>
        {
            ["context_type"] = "dialog_ctx",
            ["context_data"] = Config.HotwordContexts
                .Select(text => new Dictionary<string, object> { ["text"] = text })
                .ToList()
        };
    }

    if (!string.IsNullOrEmpty(Config.BoostingTableId))
    {
        corpus["boosting_table_id"] = Config.BoostingTableId;
    }

    if (corpus.Count > 0)
    {
        request["corpus"] = corpus;
    }
}
```

이 코드의 핵심은 실제로 핫워드 구성이 있을 때만 corpus 필드를 추가한다는 것입니다. 이 설계는 하위 호환성을 보장합니다 — 핫워드 구성이 없으면 요청 구조가 이전과 완전히 일치합니다.

## 실천 가이드

### 커스텀 핫워드 구성

실제 사용에서 커스텀 핫워드 구성은 매우 간단합니다. HagiCode의 음성 인식 설정 페이지에서 "핫워드 구성" 영역을 찾습니다. "커스텀 핫워드 텍스트" 입력란에 한 줄에 하나의 핫워드 구를 입력합니다.

예를 들어, 의료 관련 애플리케이션을 개발 중이라면:

```
고혈압
당뇨병
관상동맥경화
협심증
심근경색
심부전
```

구성을 저장한 후 매번 음성 인식을 시작할 때 이 핫워드가 자동으로 두바오 API에 전달됩니다. 실제 테스트에 따르면 핫워드 구성 후 관련 전문 용어의 인식 정확도가 눈에 띄게 향상되었습니다.

### 플랫폼 핫워드 테이블 구성

대량의 핫워드를 관리하거나 핫워드를 자주 업데이트해야 한다면 플랫폼 핫워드 테이블 모드가 더 적합합니다. 먼저 두바오 자학습 플랫폼에서 핫워드 테이블을 생성하고 생성된 boosting_table_id를 획득한 다음, HagiCode 설정 페이지에 이 ID를 입력합니다.

### 조합 모드 사용

복잡한 시나리오에서는 커스텀 핫워드와 플랫폼 핫워드 테이블을 동시에 사용해야 할 수 있습니다. 이때 HagiCode에서 두 가지 핫워드를 모두 구성하고 "조합 모드" 스위치를 활성화하기만 하면 됩니다.

조합 모드에서 두바오 API는 두 가지 핫워드 소스를 동시에 고려하며, 인식 정확도는 일반적으로 어느 것보다 높습니다. 단, 조합 모드는 요청의 복잡도를 증가시키므로 실제 테스트 후 활성화 여부를 결정하는 것을 권장합니다.

## 주의 사항

구현 및 사용 시 몇 가지 특히 주의해야 할 점이 있습니다.

첫째, 문자 제한. 두바오 API는 핫워드에 엄격한 제한(행 수, 행당 문자 수, 총 문자 수 등)이 있습니다. 제한을 초과하면 API가 오류를 반환합니다.

둘째, boosting_table_id의 형식. 이 필드는 영문자, 숫자, 밑줄, 하이픈만 허용하며 공백이나 기타 특수 문자는 포함할 수 없습니다.

셋째, 하위 호환성. 핫워드 매개변수는 완전히 선택 사항이며, 핫워드를 구성하지 않으면 시스템의 작동 방식이 이전과 완전히 일치합니다.

넷째, 오류 처리. 핫워드 구성이 유효하지 않으면 두바오 API가 해당 오류 정보를 반환합니다. HagiCode 구현은 상세한 로그를 기록하여 개발자가 문제를 조사할 수 있도록 합니다.

이 글을 통해 HagiCode 프로젝트에서 두바오 음성 인식 핫워드 기능을 구현한 완전한 솔루션을 상세히 소개했습니다. 이 솔루션은 요구 사항 분석, 기술 선택부터 코드 구현까지 전체 과정을 다루며 개발자에게 참고할 수 있는 실천 예시를 제공합니다.

핵심 포인트는 다음과 같이 요약할 수 있습니다: 첫째, 두바오 API는 커스텀 핫워드와 플랫폼 핫워드 테이블 두 가지 모드를 지원하며 독립적 또는 조합 사용이 가능합니다. 둘째, 프론트엔드는 localStorage로 구성을 저장하여 간단하고 효율적입니다. 셋째, 백엔드는 동적으로 corpus 필드를 구성하여 핫워드 매개변수를 전달하며 양호 호환성을 유지합니다. 넷째, 완전한 검증 로직이 구성의 정확성을 보장하고 무효한 요청을 방지합니다.

핫워드 기능의 구현으로 HagiCode의 음성 인식 분야 역량이 더욱 강화되었습니다. 비즈니스 관련 전문 용어를 유연하게 구성함으로써 개발자는 음성 인식 시스템이 특정 도메인의 콘텐츠를 더 잘 이해하도록 하여 더욱 정밀한 서비스를 제공할 수 있습니다.
