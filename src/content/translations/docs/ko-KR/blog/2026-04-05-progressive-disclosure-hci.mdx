## 점진적 공개: "적은 것이 더 낫다" 철학으로 AI 제품의 인간-컴퓨터 상호작용 개선

> AI 제품 설계에서 사용자 입력의 품질은 종종 출력의 품질을 결정합니다. 이 글에서는 HagiCode 프로젝트에서 실천한 "점진적 공개" 상호작용 접근 방식을 공유합니다. 단계별 안내, 지능형 완성 및 즉시 피드백을 통해 사용자의 짧고 모호한 입력을 구조화된 기술 제안으로 전환하여, 인간-컴퓨터 상호작용 효율을 크게 높입니다.

## 배경

AI 제품을 구축하는 사람이라면 아마 이 상황을 보셨을 것입니다. 사용자가 앱을 열고 의욕적으로 한 줄의 요청을 입력하지만, AI가 완전히 엉뚩맞은 것을 반환합니다. AI가 충분히 똑똘하지 않은 것이 아닙니다. 사용자가 충분한 정보를 제공하지 않은 것뿐입니다. 마음을 읽는 것은 누구에게나 어렵습니다.

이 문제는 HagiCode를 구축하면서 특히 두드러졌습니다. HagiCode는 AI 기반 코딩 어시스턴트로, 사용자가 자연어로 요구사항을 설명하여 기술 제안과 세션을 생성합니다. 실제 사용에서 사용자 입력에는 종종 다음과 같은 문제가 있었습니다:

- **일관되지 않은 입력 품질**: 일부 사용자는 "로그인 최적화"나 "버그 수정"과 같이 몇 단어만 입력하며, 필요한 컨텍스트가 없습니다
- **일관되지 않은 기술 용어**: 다른 사용자가 동일한 것에 대해 다른 용어를 사용합니다. 어떤 사람은 "프론트엔드"라고 하고, 어떤 사람은 "FE"라고 합니다
- **구조화된 정보 부족**: 프로젝트 배경, 저장소 범위, 영향 범위가 없는데, 이러한 것이 중요한 세부 사항입니다
- **반복되는 문제**: 동일한 유형의 요청이 다시 나타나고, 매번 처음부터 설명해야 합니다

직접적인 결과는 예측 가능합니다. AI가 요청을 이해하기 어려워지고, 제안 품질이 불안정해지며, 사용자 경험이 저하됩니다. 사용자는 "이 AI는 별로다"라고 생각하고, 우리는 억울하게 느낍니다. 한 문장만 주면 어떻게 정말 원하는 것을 추측해야 합니까?

솔직히 이것은 이해할 수 있습니다. 사람끼리 서로를 이해하는 데도 시간이 필요하고, 기계도 예외가 아닙니다.

이러한 문제를 해결하기 위해 우리는 대담한 결정을 내렸습니다. "점진적 공개" 설계 원칙을 도입하여 인간-컴퓨터 상호작용을 개선합니다. 이것이 가져온 변화는 예상보다 클 것이었습니다. 솔직히 그때는 이렇게 효과적일 줄 몰랐습니다.

## 점진적 공개란 무엇인가

"점진적 공개"는 HCI(인간-컴퓨터 상호작용) 분야의 설계 원칙입니다. 핵심 아이디어는 간단합니다: **사용자에게 모든 정보와 옵션을 한 번에 보여주지 마십시오. 대신 사용자의 동작과 필요에 따라 필요한 것만 단계별로 공개하십시오.**

이 원칙은 AI 제품에 특히 적합합니다. AI 상호작용은 본질적으로 점진적입니다. 사용자가 조금 말하면 AI가 조금 이해하고, 사용자가 더 추가하면 AI가 더 이해합니다. 사람 간의 소통과 매우 비슷합니다. 이해는 보통 점진적으로 발전합니다.

HagiCode의 시나리오에서 우리는 점진적 공개를 네 가지 영역에 적용했습니다:

### 1. 설명 최적화 메커니즘: AI가 더 명확하게 말하도록 돕기

사용자가 짧은 설명을 입력하면, AI 해석용으로 바로 보내지 않습니다. 대신 "설명 최적화" 흐름을 먼저 트리거합니다. 이 흐름의 핵심은 "구조화된 출력"입니다. 사용자의 자유 텍스트를 표준 형식으로 변환하는 것입니다. 흩어져 있는 진주를 목걸이에 꿰어 모든 것을 더 이해하기 쉽게 만드는 것과 비슷합니다.

최적화된 설명에는 다음 표준 섹션이 포함되어야 합니다:

- **배경**: 문제 배경과 컨텍스트
- **분석**: 기술 분석 및 추론
- **해결책**: 해결책 및 구현 단계
- **실무**: 구체적인 코드 예시 및 참고 사항

동시에 대상 저장소, 경로, 편집 권한과 같은 정보를 보여주는 Markdown 테이블을 자동 생성하여, 후속 AI 작업이 더 쉬워집니다. 명확한 목차가 있으면 무엇이든 찾기 쉬워집니다.

실제 구현은 다음과 같습니다:

```csharp
// ProposalDescriptionMemoryService.cs의 핵심 메서드
public async Task<string> OptimizeDescriptionAsync(
    string title,
    string description,
    string locale = "zh-CN",
    DescriptionOptimizationMemoryContext? memoryContext = null,
    CancellationToken cancellationToken = default)
{
    // 쿼리 매개변수 구축
    var queryContext = BuildQueryContext(title, description);

    // 기록 컨텍스트 검색
    var memoryContext = await RetrieveHistoricalContextAsync(queryContext, cancellationToken);

    // 구조화된 프롬프트 생성
    var prompt = await BuildOptimizationPromptAsync(
        title,
        description,
        memoryContext,
        cancellationToken);

    // AI 호출로 최적화
    return await _aiService.CompleteAsync(prompt, cancellationToken);
}
```

이 흐름의 핵심은 "메모리 주입"입니다. 프로젝트 규칙, 유사 사례, 부정적 패턴과 같은 기록 컨텍스트를 프롬프트에 주입하여, AI가 최적화 중 과거 경험을 참고할 수 있도록 합니다. 경험은 낭비되어서는 안 됩니다.

**참고 사항:**
- 현재 입력이 기록 메모리보다 우선하도록 하여, 사용자가 명시적으로 지정한 정보가 덮어쓰이지 않도록 하십시오
- HagIndex 참조는 사실적 출처로 취급되며, 기록 사례에 의해 변경되어서는 안 됩니다
- 신뢰도가 낮은 수정 제안은 강한 제약으로 주입하지 마십시오

### 2. 음성 입력 기능: 말하는 것이 타이핑보다 자연스럽습니다

텍스트 입력 외에도 음성 입력을 지원합니다. 이는 복잡한 요구사항을 설명할 때 특히 유용합니다. 기술 요청을 타이핑하려면 몇 분이 걸릴 수 있지만, 말로 하면 수십 초면 충분합니다.

음성 입력의 핵심 설계 초점은 "상태 관리"입니다. 사용자는 시스템이 현재 어떤 상태인지 명확히 알아야 합니다. 다음 상태를 정의했습니다:

- **유�(idle)**: 시스템이 준비되었으며 녹음을 시작할 수 있습니다
- **업스트림 대기(waiting-upstream)**: 시스템이 백엔드 서비스에 연결 중입니다
- **녹음 중(recording)**: 사용자의 음성이 녹음되고 있습니다
- **처리 중(processing)**: 음성이 텍스트로 변환되고 있습니다
- **오류(error)**: 오류가 발생했으며 사용자의 주의가 필요합니다

프론트엔드 상태 모델은 대략 다음과 같습니다:

```typescript
interface VoiceInputState {
  status: 'idle' | 'waiting-upstream' | 'recording' | 'processing' | 'error';
  duration: number;
  error?: string;
  deletedSet: Set<string>; // 삭제된 결과의 핑거프린트 집합
}

// 녹음 시작 시 상태 전환
const handleVoiceInputStart = async () => {
  // 먼저 대기 상태로 진입하고 로딩 애니메이션 표시
  setState({ status: 'waiting-upstream' });

  // 백엔드 준비 완료 대기
  const isReady = await waitForBackendReady();
  if (!isReady) {
    setState({ status: 'error', error: '백엔드 서비스가 준비되지 않았습니다' });
    return;
  }

  // 녹음 시작
  setState({ status: 'recording', startTime: Date.now() });
};

// 인식 결과 처리
const handleRecognitionResult = (result: RecognitionResult) => {
  const fingerprint = normalizeFingerprint(result.text);

  // 이미 삭제되었는지 확인
  if (state.deletedSet.has(fingerprint)) {
    return; // 삭제된 콘텐츠 건너뛰기
  }

  // 결과를 텍스트 상자에 병합
  appendResult(result);
};
```

여기에 중요한 세부 사항이 있습니다. "핑거프린트 집합"을 사용하여 삭제 동기화를 관리합니다. 음성 인식이 여러 결과를 반환할 때 사용자가 일부를 삭제할 수 있습니다. 삭제된 콘텐츠의 핑거프린트를 저장하여 나중에 동일한 콘텐츠가 나타나면 자동으로 건너뛰니다. 본질적으로 사용자가 이미 거부한 것을 기억하는 방식입니다.

### 3. 프롬프트 관리 시스템: AI의 "뇌" 외부화

HagiCode에는 유연한 프롬프트 관리 시스템이 있으며, 모든 프롬프트가 파일로 저장됩니다:

```
prompts/
├── metadata/
│   ├── optimize-description.zh-CN.json
│   └── optimize-description.en-US.json
└── templates/
    ├── optimize-description.zh-CN.hbs
    └── optimize-description.en-US.hbs
```

각 프롬프트는 두 부분으로 구성됩니다:
- **메타데이터 파일 (`.json`)**: 프롬프트 시나리오, 버전, 매개변수 등의 정보를 정의합니다
- **템플릿 파일 (`.hbs`)**: 실제 프롬프트 콘텐츠로 Handlebars 구문으로 작성됩니다

메타데이터 파일 형식은 다음과 같습니다:

```json
{
  "scenario": "optimize-description",
  "locale": "zh-CN",
  "version": "1.0.0",
  "syntax": "handlebars",
  "syntaxVersion": "1.0",
  "parameters": [
    {
      "name": "title",
      "type": "string",
      "required": true,
      "description": "Proposal 제목"
    },
    {
      "name": "description",
      "type": "string",
      "required": true,
      "description": "원본 설명"
    }
  ],
  "author": "HagiCode Team",
  "description": "사용자의 기술 제안 설명을 최적화합니다",
  "lastModified": "2026-04-05",
  "tags": ["optimization", "nlp"]
}
```

템플릿 파일은 Handlebars 구문을 사용하며 매개변수 주입을 지원합니다:

```handlebars
당신은 기술 제안 전문가입니다.

<task>
다음 정보를 기반으로 구조화된 기술 제안 설명을 생성합니다.
</task>

<input>
<title>{{title}}</title>
<description>{{description}}</description>
{{#if memoryContext}}
<memory_context>
{{memoryContext}}
</memory_context>
{{/if}}
</input>

<output_format>
## 배경
[문제 배경과 컨텍스트를 설명하십시오. 프로젝트 정보, 저장소 범위 등 포함]

## 분석
[기술 분석 및 추론 과정을 제공하고, 이 변경이 필요한 이유를 설명하십시오]

## 해결책
[해결책 및 구현 단계를 제공하고, 핵심 코드 위치를 나열하십시오]

## 실무
[구체적인 코드 예시 및 참고 사항을 제공하십시오]
</output_format>
```

이 설계의 장점은 명확합니다:
- 프롬프트를 코드처럼 버전 관리할 수 있습니다
- 다국어를 지원하며 사용자 환경설정에 따라 자동 전환됩니다
- 매개변수화된 설계로 컨텍스트를 동적으로 주입할 수 있습니다
- 시작 시 완전성을 검증하여 런타임 오류를 방지합니다

지식이 한 사람의 머릿속에만 있으면 잃어버리기 쉽습니다. 처음부터 구조화된 방식으로 기록하는 것이 훨씬 안전합니다.

### 4. 점진적 마법사: 복잡한 작업을 작은 단계로 분할

처음 설치 및 구성과 같은 복잡한 작업의 경우, 다단계 마법사 설계를 사용합니다. 각 단계에서 필요한 정보만 요청하고 명확한 진행 표시를 제공합니다. 큰 작업은 한 단계씩 처리하면 훨씬 관리하기 쉬워집니다.

마법사 상태 모델:

```typescript
interface WizardState {
  currentStep: number;           // 0-3, 4단계에 해당
  steps: WizardStep[];
  canGoNext: boolean;
  canGoBack: boolean;
  isLoading: boolean;
  error: string | null;
}

interface WizardStep {
  id: number;
  title: string;
  description: string;
  completed: boolean;
}

// 단계 탐색 로직
const goToNextStep = () => {
  if (wizardState.currentStep < wizardState.steps.length - 1) {
    // 현재 단계 입력 검증
    if (validateCurrentStep()) {
      wizardState.currentStep++;
      wizardState.steps[wizardState.currentStep - 1].completed = true;
    }
  }
};

const goToPreviousStep = () => {
  if (wizardState.currentStep > 0) {
    wizardState.currentStep--;
  }
};
```

각 단계에는 자체 검증 로직이 있으며, 완료된 단계에는 명확한 시각적 마커가 표시됩니다. 취소하면 확인 대화상자가 열려 실수로 진행 상황을 잃지 않도록 합니다.

## 결론

HagiCode의 점진적 공개 실무를 돌아보며, 몇 가지 핵심 원칙을 요약할 수 있습니다:

1. **단계별 안내**: 복잡한 작업을 더 작은 단계로 나누고 각 단계에서 필요한 정보만 요청합니다
2. **지능형 완성**: 기록 컨텍스트와 프로젝트 지식을 활용하여 정보를 자동으로 채웁니다
3. **즉시 피드백**: 모든 동작에 명확한 시각적 피드백과 상태 힌트를 제공합니다
4. **오류 허용 메커니즘**: 사용자가 실행 취소 및 재설정할 수 있도록 하여 실수가 되돌릴 수 없는 손실을 방지합니다
5. **입력 다양성**: 텍스트와 음성과 같은 여러 입력 방법을 지원합니다

HagiCode에서 이 접근 방식의 실제 결과는 명확했습니다. 사용자 입력의 평균 길이가 20자 미만에서 200-300자의 구조화된 설명으로 증가했고, AI가 생성한 제안의 품질이 눈에 띄게 향상되었으며, 사용자 만족도도 함께 증가했습니다.

이것은 놀라운 일이 아닙니다. 사용자가 더 많은 정보를 제공할수록 AI가 더 정확히 이해하고, 더 좋은 결과를 반환할 수 있습니다. 그 의미에서 사람 간의 소통과 크게 다르지 않습니다.

AI 관련 제품도 구축 중이시라면, 이 경험이 몇 가지 유용한 영감을 제공하기를 바랍니다. 기억하십시오. 사용자가 정보 제공을 거부하는 것이 아닙니다. 더 자주, 제품이 올바른 방식과 올바른 타이밍으로 질문하지 않았을 뿐입니다. 점진적 공개의 핵심은 그 질문들의 최적의 타이밍과 형태를 찾는 것입니다.
