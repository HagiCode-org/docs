## HagiCode Soul 플랫폼 기술 분석: 새로운 니즈에서 독립 플랫폼으로의 진화

> 기술 글을 쓰는 것은 그렇게 대단한 일이 아닙니다. 대부분은 겪은 함정과 우회한 길을 정리하는 것에 불과합니다. 우리 모두 초보였던 적이 있습니다. 이 글에서는 HagiCode 프로젝트에서 Soul의 설계 철학, 아키텍처 진화, 핵심 기술 구현을 심층적으로 살펴보고, 독립 플랫폼이 Agent 페르소나 생성과 공유를 위해 어떻게 더 집중된 경험을 제공할 수 있는지 탐구합니다.

## 배경

AI Agent 구축 실무에서 우리는 종종 단순해 보이지만 실제로는 매우 중요한 질문에 직면합니다: **다양한 Agent에 안정적이고 독특한 언어 스타일과 성격 특성을 어떻게 부여하는가?**

솔직히 약간 답답한 질문입니다. HagiCode의 초기 Hero 시스템에서는 다른 Hero(Agent 인스턴스)가 주로 직업 설정과 일반적인 프롬프트를 통해 구분되었습니다. 이 방식에는 몇 가지 명백한 고통점이 있었으며, 비슷한 것을 시도해 본 사람이라면 아마 같은 느낌을 받았을 것입니다.

첫째, 언어 스타일을 일관되게 유지하기 어려웠습니다. 같은 "개발 엔지니어" 역할이 하루는 전문적이고 엄격하게 들리고, 다음 날에는 캐주얼하고 느슨하게 들렸습니다. 이것은 모델의 문제라기보다는 출력 스타일을 제약하고 안내하는 독립적인 성격 구성 레이어가 부재한 문제였습니다.

둘째, 캐릭터의 감각이 일반적으로 약했습니다. Agent의 특성을 설명할 때 종종 "친절한", "전문적인", "유머러스한"과 같은 모호한 형용사에 의존해야 했고, 그 추상적인 설명을 뒷받침할 구체적인 언어 규칙이 없었습니다. 말하자면 이론적으로는 좋아 보였지만, 실제로는 붙잡을 것이 거의 없었습니다.

셋째, 페르소나 구성을 재사용하는 것은 거의 불가능했습니다. "고양이 소녀 웨이트리스"의 말투를 신중하게 설계하고 다른 비즈니스 시나리오에서 그 표현 스타일을 재사용하고 싶다면, 실제로는 거의 처음부터 다시 구성해야 했습니다. 때로는 아름다운 것을 소유하고 싶은 게 아니라 조금만 재사용하고 싶은데... 그것조차 어려웠습니다.

이러한 실제 문제를 해결하기 위해 우리는 **Soul** 메커니즘을 도입했습니다: 장비와 설명과 분리된 독립적인 **언어 스타일 구성 레이어**입니다. Soul은 Agent의 말하기 습관, 어조 선호도, 표현 경계를 정의할 수 있고, 여러 Hero 간에 공유 및 재사용될 수 있으며, 첫 번째 Session 호출 시 시스템 프롬프트에 자동으로 주입될 수 있습니다.

어떤 사람들은 이것이 그저 몇 가지 프롬프트를 구성하는 것이라고 말할 수도 있습니다. 하지만 때로 진짜 질문은 무언가를 할 수 있는지가 아니라 더 우아하게 하는 방법입니다. Soul이 성숙해지면서 우리는 이것이 독립적으로 발전할 만큼 충분한 깊이가 있다는 것을 깨달았습니다. 전용 Soul 플랫폼은 사용자가 Hero 시스템의 나머지 부분에 방해받지 않고 흥미로운 페르소나 구성을 생성, 공유, 탐색하는 데 집중할 수 있게 합니다. 그것이 **soul.hagicode.com**의 독립 플랫폼이 탄생한 방식입니다.

## Soul 플랫폼의 기술 아키텍처 진화

Soul 플랫폼은 한 번에 나타난 것이 아닙니다. 세 가지 명확한 단계를 거쳤습니다. 이야기는 갑자기 시작되어 자연스럽게 마무리되었습니다.

### 1단계: Hero에 내장된 Soul 구성

가장 초기 Soul 구현은 Hero 워크스페이스 내부의 기능 모듈로 존재했습니다. 우리는 Hero UI에 독립적인 SOUL 편집 영역을 추가하여 프리셋 적용과 텍스트 미세 조정을 모두 지원했습니다.

프리셋 적용을 통해 사용자는 "전문 개발 엔지니어" 및 "고양이 소녀 웨이트리스"와 같은 클래식 페르소나 템플릿에서 선택할 수 있었습니다. 텍스트 미세 조정을 통해 사용자는 해당 프리셋을 추가로 개인화할 수 있었습니다. 백엔드에서는 Hero 엔티티에 `Soul` 필드가 추가되었고, `SoulCatalogId`를 사용하여 출처를 식별했습니다.

이 단계는 기능이 존재하는지의 여부를 해결했고, 무엇이든 젊을 때처럼 다소 어색하게 성장했습니다. 하지만 Soul 콘텐츠가 풍부해지면서 Hero 시스템과 긴밀하게 결합된 아키텍처의 한계가 드러나기 시작했습니다.

### 2단계: 사내 마켓플레이스

더 나은 Soul 발견 및 재사용 경험을 제공하기 위해 SOUL 마켓플레이스 카탈로그 페이지를 구축하여 탐색, 검색, 상세 정보 확인, 즐겨찾기를 지원했습니다.

이 단계에서 **50개의 메인 카탈로그(기본 역할)**와 **10개의 직교 규칙(표현 스타일)**로 구성된 조합 설계를 도입했습니다. 메인 카탈로그는 Agent의 핵심 페르소나를 정의하고 "안개항 여행자" 및 "야간 사냥꾼"과 같은 추상적인 캐릭터 설정을 포함했습니다. 직교 규칙은 Agent가 *어떻게* 자신을 표현하는지를 정의하고 "간결하고 전문적인" 및 "상세하고 친절한"과 같은 언어 스타일 특성을 포함했습니다.

50 x 10 = 500가지 가능한 조합은 사용자에게 넓은 페르소나 구성 공간을 제공했습니다. 압도적인 숫자는 아니지만 작지도 않습니다. 로마로 가는 길은 많고, 어떤 길은 다른 길보다 걷기 쉬울 뿐입니다. 백엔드에서는 전체 SOUL 카탈로그가 `catalog-sources.json`을 통해 생성되었고, 프론트엔드에서는 해당 카탈로그 항목을 인터랙티브 카드 목록으로 표현했습니다.

사내 마켓플레이스는 좋은 과도기 해결책이었지만, 그것뿐이었습니다: 과도기적인. 여전히 메인 시스템에 부착되어 있었고, Soul 기능만 원하는 사용자에게는 접근 경로가 여전히 너무 깊었습니다. 간단한 일을 하기 위해 꼭 경치 좋은 길로 가고 싶은 사람은 아무도 없습니다.

### 3단계: 독립 플랫폼으로 분리

결국 우리는 Soul을 독립 저장소(`repos/soul`)로 이동하기로 결정했습니다. 기존 메인 시스템의 마켓플레이스는 외부 점프 가이드로 변경되었고, 새 플랫폼은 **Builder-first** 설계 철학을 채택했습니다: 홈페이지가 기본적으로 생성 워크스페이스이므로 사용자는 사이트를 여는 순간 자신만의 페르소나 구성을 만들기 시작할 수 있습니다.

이 단계에서 기술 스택도 전면적으로 업그레이드되었습니다: Vite 8 + React 19 + TypeScript 5.9, shadcn/ui 컴포넌트 시스템을 통한 통일된 디자인 언어, Tailwind CSS 4 테마 변수. 프론트엔드 엔지니어링의 향상은 향후 기능 반복을 위한 탄탄한 기반을 마련했습니다.

모든 것이 사라졌습니다... 아니, 실제로는 모든 것이 막 시작된 것입니다.

## 핵심 기술 설계 및 구현

### 소재 통합 전략

Soul 플랫폼의 핵심 설계 원칙 중 하나는 **local-first**입니다. 즉, 홈페이지는 백엔드 없이도 완전히 기능해야 하고, 원격 소재 로드 실패가 페이지 진입을 차단해서는 안 됩니다.

이것은 특별히 기적적인 것은 아닙니다. 시스템을 설계할 때 한 걸음 더 생각하는 것을 의미할 뿐입니다. 로컬 스냅샷을 베이스라인으로 사용하고 원격 데이터를 향상으로 사용하면 제품이 어떤 네트워크 조건에서도 기본적으로 사용 가능하게 됩니다. 구체적으로 우리는 2계층 소재 아키텍처를 구현했습니다:

```typescript
export async function loadBuilderMaterials(): Promise<BuilderMaterials> {
  const localMaterials = createLocalMaterials(snapshot)  // 로컬 베이스라인

  try {
    const inspirationFragments = await fetchMarketplaceItems()  // 원격 향상
    return { ...localMaterials, inspirationFragments, remoteState: "ready" }
  } catch (error) {
    return { ...localMaterials, remoteState: "fallback" }  // 우아한 저하
  }
}
```

로컬 소재는 빌드 시점의 메인 시스템 문서 스냅샷에서 오며, 50개의 기본 역할과 10개의 표현 규칙의 완전한 데이터를 포함합니다. 원격 소재는 사용자가 게시한 Soul에서 오며 Marketplace API를 통해 가져옵니다. 함께 사용자에게 공식 템플릿부터 커뮤니티 창작에 이르기까지 전체 스펙트럼의 소재를 제공합니다. 극적으로 들리지만, 실제로는 그저 로컬 더하기 원격일 뿐입니다.

### Soul Fragment 데이터 모델

Soul의 핵심 데이터 추상화는 **SoulFragment**입니다:

```typescript
export type SoulFragment = {
  fragmentId: string
  group: "main-catalog" | "expression-rule" | "published-soul"
  title: string
  summary: string
  content: string
  keywords: string[]
  localized?: Partial<Record<AppLocale, LocalizedFragmentContent>>
  sourceRef: SoulFragmentSourceRef
  meta: SoulFragmentMeta
}
```

`group` 필드는 프래그먼트 유형을 구분합니다: 메인 카탈로그는 캐릭터 코어를 정의하고, 직교 규칙은 표현 스타일을 정의하며, 사용자 게시 Soul은 `published-soul`로 표시됩니다. `localized` 필드는 다국어 표시를 지원하여 동일한 프래그먼트가 다른 언어 환경에서 다른 제목과 설명을 표시할 수 있습니다. 국제화는 정말 일찍 생각해야 하는 것이고, 이번에는 실제로 그렇게 했습니다.

Builder 초안 상태는 사용자의 현재 편집 상태를 캡슐화합니다:

```typescript
export type SoulBuilderDraft = {
  draftId: string
  name: string
  selectedMainFragmentId: string | null
  selectedRuleFragmentId: string | null
  inspirationSoulId: string | null
  mainSlotText: string
  ruleSlotText: string
  customPrompt: string
  previewText: string
  updatedAt: string
}
```

편집기에서 선택한 각 프래그먼트의 콘텐츠가 해당 슬롯에 연결되어 최종 미리보기 텍스트를 형성합니다. `mainSlotText`는 메인 역할 콘텐츠에 해당하고, `ruleSlotText`는 표현 규칙 콘텐츠에 해당하며, `customPrompt`는 사용자의 추가 지시 텍스트입니다.

### 미리보기 컴파일 메커니즘

미리보기 컴파일은 Soul Builder의 핵심 기능입니다. 사용자가 선택한 프래그먼트와 사용자 정의 텍스트를 직접 복사할 수 있는 시스템 프롬프트로 조립합니다:

```typescript
export function compilePreview(
  draft: Pick<SoulBuilderDraft, "mainSlotText" | "ruleSlotText" | "customPrompt">,
  fragments: {
    mainFragment: SoulFragment | null
    ruleFragment: SoulFragment | null
    inspirationFragment: SoulFragment | null
  }
): PreviewCompilation {
  // 조립 로직: 메인 역할 + 표현 규칙 + 영감 참조 + 사용자 정의 콘텐츠
}
```

컴파일 결과는 중앙 미리보기 패널에 표시되며, 사용자는 실시간으로 최종 효과를 볼 수 있고 원클릭으로 클립보드에 복사할 수 있습니다. 간단하게 들리고 실제로도 간단합니다. 하지만 간단한 것이 종종 가장 유용합니다.

### 프론트엔드 상태 관리

Soul Builder의 프론트엔드 상태 관리는 하나의 중요한 원칙을 따릅니다: **상태 경계의 명확한 분리**. 더 구체적으로, 서랍 상태는 지속되지 않으며 초안에 직접 쓰지 않습니다. 명시적인 Builder 동작만 의미 있는 상태 변경을 트리거합니다.

```typescript
// 도메인 상태 (useSoulBuilder)
export function useSoulBuilder() {
  // 소재 로딩 및 캐싱
  // 슬롯 집계 및 미리보기 컴파일
  // 복사 동작 및 피드백 메시지
  // 로케일 안전한 설명자
}

// 표현 상태 (useHomeEditorState)
export function useHomeEditorState() {
  // activeSlot, drawerSide, drawerOpen
  // 기본 포커스 동작
}
```

이 분리는 편집 상태 안전성과 반응형 UI 동작을 모두 보장합니다. 서랍을 열고 닫는 것은 순수한 UI 상호작용이며 복잡한 지속성 로직을 트리거해서는 안 됩니다. 당연하게 들릴 수 있지만 중요합니다: UI 상태와 비즈니스 상태를 명확하게 분리하여 인터페이스 상호작용이 핵심 데이터 모델을 오염시키지 않아야 합니다.

### 단일 서랍 수명 주기

Soul Builder는 단일 서랍 모드를 사용합니다: 한 번에 하나의 슬롯 서랍만 열릴 수 있습니다. 마스크 클릭, ESC 키 누르기, 슬롯 전환은 현재 서랍을 자동으로 닫습니다. 이는 상태 관리를 단순화하고 모바일의 일반적인 서랍 상호작용 패턴에도 부합합니다.

서랍을 닫아도 현재 편집 콘텐츠가 지워지지 않으므로, 사용자가 돌아올 때 컨텍스트가 보존됩니다. 이러한 "가벼운" 서랍 설계는 사용자의 흐름을 방해하지 않습니다. 누구도 신중하게 작성한 콘텐츠가 실수 클릭 하나로 사라지기를 원하지 않습니다.

### 이중 언어 지원 아키텍처

국제화는 Soul 플랫폼의 중요한 기능입니다. 시스템 카피는 이중 언어 전환을 완전히 지원하지만, 사용자 초안 텍스트는 언어 변경 시 재작성되지 않습니다. 초안 텍스트는 시스템 번역 콘텐츠가 아닌 사용자 작성 자유 입력이기 때문입니다.

공식 영감 카드(Marketplace Soul)는 업스트림 표시 이름을 유지하면서 최선을 다한 영어 요약도 제공합니다. 중국어 이름의 Soul에 대해서는 미리 정의된 매핑 규칙을 통해 영어 버전을 생성합니다:

```typescript
// 메인 역할 영어 이름 매핑
const mainNameEnglishMap = {
  "雾港旅人": "Mistport Traveler",
  "夜航猎手": "Night Hunter",
  // ...
}

// 직교 규칙 영어 이름 매핑
const ruleNameEnglishMap = {
  "简洁干练": "Concise & Professional",
  "啰嗦亲切": "Verbose & Friendly",
  // ...
}
```

매핑 테이블 자체는 충분히 간단해 보이지만, 이를 좋은 상태로 유지하려면 여전히 주의가 필요합니다. 50개의 메인 역할과 10개의 직교 규칙이 있으므로 총 500가지 조합입니다. 엄청나게 큰 숫자는 아니지만 존중받을 만큼은 됩니다.

## 백엔드 카탈로그 생성

Soul 카탈로그의 대량 생성은 백엔드에서 이루어지며, C#을 사용하여 50 x 10 = 500가지 조합의 생성을 자동화합니다:

```csharp
foreach (var main in source.MainCatalogs)
{
    foreach (var orthogonal in source.OrthogonalCatalogs)
    {
        var catalogId = $"soul-{main.Index:00}-{orthogonal.Index:00}";
        var displayName = BuildNickname(main, orthogonal);
        var soulSnapshot = BuildSoulSnapshot(main, orthogonal);
        // 데이터베이스에 쓰기...
    }
}
```

닉네임 생성 알고리즘은 메인 역할 이름과 표현 규칙 이름을 결합하여 상상력 넘치는 Agent 코드네임을 만듭니다:

```csharp
private static readonly string[] MainHandleRoots = [
    "雾港", "夜航", "零帧", "星渊", "霓虹", "断云", ...
];
private static readonly string[] OrthogonalHandleSuffixes = [
    "旅人", "猎手", "术师", "行者", "星使", ...
];
// 조합 예시: 雾港旅人, 夜航猎手, 零帧术师...
```

Soul 스냅샷 조립은 고정된 템플릿 형식을 따르며 메인 역할 코어, 서명 특성, 표현 규칙 코어, 출력 제약을 결합합니다:

```csharp
private static string BuildSoulSnapshot(main, orthogonal) => string.Join('\n', [
    $"你的人设内核来自「{main.Name}」：{main.Core}",
    $"保持以下标志性语言特征：{main.Signature}",
    $"你的表达规则来自「{orthogonal.Name}」：{orthogonal.Core}",
    $"必须遵循这些输出约束：{orthogonal.Signature}"
]);
```

템플릿 조립은 지루하게 들릴 수 있지만, 그런 지루한 작업 없이는 흥미로운 제품이 좀처럼 나오지 않습니다.

## 플랫폼 마이그레이션 전략

Soul을 메인 시스템에서 독립 플랫폼으로 분리한 후, 중요한 과제 중 하나는 기존 사용자 데이터를 처리하는 것이었습니다. 익숙한 문제입니다: 분리하는 것은 쉽지만 마이그레이션은 그렇지 않습니다. 우리는 세 가지 안전장치를 채택했습니다.

**하위 호환성 보호.** 이전에 저장된 Hero SOUL 스냅샷은 계속 표시되며, Marketplace 소스 ID가 없더라도 기록 스냅샷을 계속 미리볼 수 있습니다. 즉, 사용자의 이전 구성은 손실되지 않고 표시 위치만 변경된 것입니다.

**메인 시스템 API 폐기.** 사내 마켓플레이스 API는 HTTP 상태 `410 Gone`과 함께 사용자를 soul.hagicode.com으로 안내하는 마이그레이션 공지를 반환합니다.

**Hero SOUL 폼 리팩토링.** Hero Soul 편집 영역에 마이그레이션 공지 블록이 추가되어 Soul 플랫폼이 이제 독립적이라는 것을 사용자에게 명확히 알리고 원클릭 점프 버튼을 제공합니다:

```typescript
// HeroSoulForm.tsx
<div className="rounded-2xl border border-orange-200/70 bg-orange-50/80 p-4">
  <div>{t('hero.soul.migrationTitle')}</div>
  <p>{t('hero.soul.migrationDescription')}</p>
  <Button onClick={onOpenSoulPlatform}>
    {t('hero.soul.openSoulPlatformAction')}
  </Button>
</div>
```

## 실무 교훈

Soul 플랫폼의 전체 개발을 돌아보며 공유할 가치가 있는 몇 가지 실무 교훈이 있습니다. 이것은 위대한 원칙이 아니라 실제 실수에서 배운 것들입니다.

**Local-first 런타임 가정.** 원격 데이터에 의존하는 기능을 설계할 때 항상 네트워크를 사용할 수 없다고 가정하십시오. 로컬 스냅샷을 베이스라인으로 사용하고 원격 데이터를 향상으로 사용하면 제품이 어떤 네트워크 조건에서도 기본적으로 사용 가능하게 됩니다.

**상태 경계의 명확한 분리.** UI 상태와 비즈니스 상태를 명확하게 구분하여 인터페이스 상호작용이 핵심 데이터 모델을 오염시키지 않도록 해야 합니다. 서랍 토글은 순수한 UI 상태이며 초안 지속성과 혼합되어서는 안 됩니다.

**국제화를 일찍 설계.** 제품에 다국어 요구 사항이 있다면 데이터 모델 설계 단계에서 생각하는 것이 가장 좋습니다. `localized` 필드는 구조적 복잡성을 약간 추가하지만 다국어 콘텐츠의 장기 유지보수 비용을 크게 줄여줍니다.

**소재 동기화 워크플로우 자동화.** Soul 플랫폼의 로컬 소재는 메인 시스템 문서에서 옵니다. 업스트림 문서가 변경되면 프론트엔드 스냅샷으로 동기화할 메커니즘이 필요합니다. `npm run materials:sync` 스크립트를 설계하여 이 프로세스를 자동화하고 소재를 업스트림과 정렬시킵니다.

## 향후 전망

현재 아키텍처를 기반으로 Soul 플랫폼은 향후 여러 방향으로 발전할 수 있습니다. 이것들은 임시적인 아이디어이지만, 출발점으로 유용할 수 있습니다.

**커뮤니티 공유 생태계.** 사용자 정의 Soul의 업로드 및 공유를 지원하고, 평점, 댓글, 추천 메커니즘을 통해 우수한 Soul 구성이 더 많은 사람들에게 발견되고 재사용될 수 있도록 합니다.

**멀티모달 확장.** 텍스트 스타일을 넘어 음성 스타일 구성, 이모지 사용 선호도, 코드 스타일 및 포맷팅 규칙과 같은 차원도 지원할 수 있습니다. 이론적으로 매력적으로 들리지만, 구현은 더 복잡한 이야기를 할 것입니다.

**지능형 지원.** 사용 시나리오에 따라 Soul을 자동 추천하고, 스타일 전송 및 융합을 지원하며, 심지어 다른 Soul의 실제 효과를 A/B 테스트합니다. 시도해 보는 것보다 더 좋은 방법은 없습니다.

**크로스 플랫폼 동기화.** 다른 AI 플랫폼에서 페르소나 구성을 가져오고, 표준화된 Soul 내보내기 형식을 제공하며, 메인스트림 Agent 프레임워크와 통합합니다.

## 결론

이 글에서는 HagiCode Soul 플랫폼이 가장 초기의 새로운 니즈에서 독립 플랫폼으로의 전체 진화 과정을 공유했습니다. Agent 페르소나 일관성 문제를 해결하기 위해 Soul 메커니즘이 필요한 이유를 논의하고, 아키텍처 진화의 세 단계(내장 구성, 사내 마켓플레이스, 독립 플랫폼)를 분석하며, 핵심 데이터 모델, 상태 관리, 미리보기 컴파일, 국제화 설계를 심층적으로 검토하고, 실무 마이그레이션 교훈을 요약했습니다.

Soul의 본질은 **비즈니스 로직과 분리된 독립적인 페르소나 구성 레이어**입니다. 이것은 AI Agent의 언어 스타일을 정의 가능하고, 재사용 가능하고, 공유 가능하게 만듭니다. 기술적으로 이 설계 자체는 특별히 복잡하지 않지만, 해결하는 문제는 실제이고 광범위하게 관련이 있습니다.

AI Agent 제품을 구축 중이시라면, 페르소나 구성 솔루션이 충분히 유연한지 자문해 볼 가치가 있습니다. Soul 플랫폼의 실무 경험이 몇 가지 유용한 아이디어를 제공할 수 있습니다.

어느 날 비슷한 문제에 직면하실 수도 있습니다. 그때 이 글이 조금이라도 도움이 된다면, 그것으로 충분할 것입니다.
