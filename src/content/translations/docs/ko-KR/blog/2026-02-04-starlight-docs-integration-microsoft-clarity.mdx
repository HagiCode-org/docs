## 기술 선택과 탐색

초기에 Proposal 단계에서 여러 통합 방식을 논의했습니다. Starlight(Astro 기반 문서 프레임워크)를 사용하므로 가장 직관적인 생각은 Astro의 Hooks를 활용하는 것이었습니다.

먼저 `astro.config.mjs`를 수정하여 빌드 시 Clarity 스크립트를 주입하려고 시도했습니다. 이 방식은 전역 커버리지를 보장할 수 있지만 유연성이 부족하여 사용자 기본 설정에 따라 스크립트를 동적으로 로드하거나 언로드할 수 없습니다.

사용자 경험과 프라이버시 제어를 고려하여 최종적으로 **컴포넌트 오버라이드** 솔루션을 채택했습니다. Starlight는 개발자가 내부 컴포넌트를 오버라이드할 수 있으므로 `<footer>` 또는 `<head>`의 렌더링 로직을 인계받아 Clarity의 로딩 타이밍을 정밀하게 제어할 수 있습니다.

한 가지 에피소드가 있습니다: 원래 `StarlightWrapper.astro`라는 레이아웃 래퍼를 만들려고 했습니다. 하지만 실제 디버깅에서 Starlight의 라우팅 메커니즘이 이 커스텀 Wrapper를 자동으로 호출하지 않아 일부 페이지에서 스크립트가 작동하지 않는 것을 발견했습니다. 이것은 전형적인 "당연하다고 생각하는" 함정 경험으로, **프레임워크의 렌더링 흐름을 깊이 이해해야 하며 맹목적으로 일반적인 프레임워크 패턴을 적용하지 말아야 한다는 점**을 상기시킵니다.

## 핵심 솔루션: Footer 컴포넌트 오버라이드

Clarity 스크립트가 모든 페이지(문서 및 블로그 포함)에서 로드되고 기존 페이지 구조를 파괴하지 않도록 Starlight의 `Footer` 컴포넌트를 오버라이드하기로 했습니다.

### 왜 Footer일까요?

1.  **전역성**: Footer는 거의 모든 표준 페이지에 나타납니다.
2.  **비침습성**: 스크립트를 Footer 영역(실제로는 body 하단에 렌더링)에 두면 페이지의 핵심 렌더링 경로(LCP)를 차단하지 않고 성능에 미치는 영향이 최소화됩니다.
3.  **로직 집중**: 컴포넌트 내부에서 Cookie 동의 로직을 통합 처리할 수 있습니다.

### 구현 단계

#### 1. Clarity 프로젝트 준비

먼저 [Microsoft Clarity](https://clarity.microsoft.com/)에 등록하고 새 프로젝트를 생성해야 합니다. Project ID(예: `k8z2ab3xxx` 같은 문자열)를 획득합니다.

`src/components/` 디렉토리에 `StarlightFooter.astro` 파일을 생성합니다. Starlight는 이 파일을 자동으로 인식하고 기본 Footer를 오버라이드합니다.

핵심 코드 로직은 다음과 같습니다:

```astro
---
// src/components/StarlightFooter.astro
// 1. 引入原始组件以保留其默认功能
import DefaultFooter from '@astrojs/starlight/components/StarlightFooter.astro';

// 2. 获取环境变量
const clarityId = import.meta.env.PUBLIC_CLARITY_ID;

// 3. 定义简单的注入脚本（内联方式）
// 注意：生产环境建议将此逻辑抽离到单独的 .js 文件中以利用缓存
const initScript = `
(function(c,l,a,r,i,t,y){
    c[a]=c[a]||function(){(c[a].q=c[a].q||[]).push(arguments)};
    t=l.createElement(r);t.async=1;t.src="https://www.clarity.ms/tag/"+i;
    y=l.getElementsByTagName(r)[0];y.parentNode.insertBefore(t,y);
})(window, document, "clarity", "script", "${clarityId}");
`;
---

<DefaultFooter {...Astro.props} />

{/* 仅在生产环境且 ID 存在时注入脚本 */}
{import.meta.env.PROD && clarityId && (
  <script is:inline define:vars={{ clarityId }}>
    {initScript}
  </script>
)}
```

**핵심 포인트 분석**:
- `is:inline`: Astro에게 이 script 태그 내부의 콘텐츠를 처리하지 않고 HTML에 직접 출력하라고 지시합니다. 서드파티 통계 스크립트에 매우 중요하며, 그렇지 않으면 Astro의 번들링 최적화로 인해 스크립트가 작동하지 않을 수 있습니다.
- `define:vars`: Astro 3+의 기능으로 범위 내에서 변수를 안전하게 주입할 수 있습니다.
- `import.meta.env.PROD`: 로컬 개발 시(디버깅 목적이 아니면) 유효하지 않은 통계가 생성되지 않도록 하여 데이터의 순수성을 유지합니다.

### 고급: 프라이버시 규정 준수와 Cookie 제어

코드를 추가하는 것만으로는 충분하지 않습니다, 특히 GDPR 관할 지역에서는. 사용자의 선택을 존중해야 합니다.

## 경험 요약과 함정

이 솔루션을 HagiCode에 적용하는 과정에서 몇 가지 쉽게 간과되는 세부 사항을 요약했습니다:

1.  **`StarlightWrapper.astro`는 함정입니다**:
    앞서 말씀드린 대로 글로벌 Wrapper를 만들어 스크립트를 주입하려고 하지 마세요. Starlight에서는 작동하지 않습니다. 특정 컴포넌트(예: `StarlightFooter.astro` 또는 `StarlightHead.astro`)를 정직하게 오버라이드하는 것이 정답입니다.

2.  **스크립트 위치의 성능 고려**:
    Clarity는 데이터 정확성을 위해 `<head>`에 배치할 것을 권장하지만, 문서 사이트의 경우 첫 화면 로딩 속도(LCP)가 SEO와 사용자 유지에 직접적인 영향을 미칩니다. Footer(Body 하단)에 배치하면 극소수의 "즉시 이탈" 사용자 데이터를 약간 손실하지만 더 빠른 페이지 로딩 경험을 얻을 수 있으며, 이는 가치 있는 트레이드오프입니다.

3.  **개발 환경의 간섭**:
    반드시 `import.meta.env.PROD` 판단을 추가해야 합니다. 개발 모드에서는 페이지를 자주 새로고침하므로 Clarity 대시보드에 대량의 무의미한 테스트 데이터가 생성되어 오염됩니다.

## 효과 검증

배포 완료 후 Clarity 콘솔에서 실시간 데이터를 볼 수 있습니다. 일반적으로 몇 분 이내에 사용자의 히트맵과 녹화를 볼 수 있습니다.

HagiCode의 경우 이러한 데이터를 통해 다음을 발견했습니다:
- 많은 사용자가 "빠른 시작" 섹션을 반복적으로 확인하여 설치 안내가 아직 충분히 직관적이지 않음을 시사합니다.
- "API 참조" 페이지의 체류 시간이 가장 길어 핵심 사용자 그룹의 요구를 확인했습니다.

## 요약

Microsoft Clarity 연동은 복잡한 서버 측 개조나 무거운 SDK 도입이 필요하지 않습니다.

Starlight의 컴포넌트 오버라이드 메커니즘을 활용하여 가벼운 `StarlightFooter.astro` 컴포넌트 하나만으로 전역 데이터 통계를 구현했습니다. 이 "마이크로 통합" 방식은 코드의 깔끔함을 보장하면서 사용자 행동을 파악하는 능력을 부여합니다.

기술 프로젝트를 운영하고 있고, 특히 **HagiCode**처럼 문서를 지속적으로 반복해야 하는 프로젝트라면 Clarity 연동을 강력히 권장합니다. 데이터가 사용자의 진정한 문제점이 어디에 있는지 알려줄 것입니다.
