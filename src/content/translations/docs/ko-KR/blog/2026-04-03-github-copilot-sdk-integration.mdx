## CLI 호출에서 SDK 통합까지: .NET 프로젝트에서 GitHub Copilot 활용 모범 사례

> 명령줄 호출에서 공식 SDK 통합으로의 업그레이드 경로는 꽤 여정이었습니다. 오늘 HagiCode를 구축하면서 겪은 함정과 배운 점을 공유하고자 합니다.

## 배경

2025년 GitHub Copilot SDK가 공식 출시된 후, 우리는 AI 기능 계층에 통합하기 시작했습니다. 그 전에는 프로젝트가 주로 GitHub Copilot CLI 명령줄 도구를 직접 호출하는 방식을 사용했지만, 이 접근 방식에는 몇 가지 명확한 문제가 있었습니다:

- **복잡한 프로세스 관리**: CLI 프로세스 수명 주기, 시작 시간 초과, 프로세스 정리를 수동으로 관리해야 했습니다. 프로세스는 예고 없이 충돌할 수 있습니다.
- **불완전한 이벤트 처리**: 원시 CLI 호출은 모델 추론(`thinking`) 및 도구 실행의 세분화된 이벤트를 캡처하기 어렵습니다. 결과만 보이고 사고 과정은 보이지 않는 것과 같습니다.
- **어려운 세션 관리**: 세션 재사용 및 복구에 대한 효과적인 메커니즘이 없어 모든 상호작용을 처음부터 다시 시작해야 했습니다.
- **호환성 문제**: CLI 인수가 자주 변경되어, 해당 매개변수에 대한 호환성 로직을 계속 유지해야 했습니다.

이러한 문제는 일상 개발에서 점점 더 명확해졌으며, 특히 모델 추론(`thinking`)과 도구 실행 상태를 실시간으로 추적해야 할 때 더욱 그러했습니다. 그 시점에서 더 낮은 수준이고 더 완전한 통합 방식이 필요하다는 것이 명확해졌습니다.

## HagiCode 소개

이 글에서 공유하는 접근 방식은 [HagiCode](https://hagicode.com) 프로젝트의 실무 경험에서 나온 것입니다. HagiCode는 오픈소스 AI 코딩 어시스턴트 프로젝트이며, 개발 중에 GitHub Copilot 기능과 기본 코드 완성부터 복잡한 다중 턴 대화 및 도구 호출까지의 심층 통합이 필요했습니다. 그러한 실제 요구사항이 우리를 CLI 호출에서 공식 SDK로 이끌었습니다.

이 구현이 유용하게 들리신다면, 저희의 엔지니어링 경험이 도움이 될 가능성이 높습니다. 그렇다면 HagiCode 프로젝트 자체도 살펴볼 가치가 있을 것입니다. 이 글 끝부분에서 더 많은 프로젝트 정보와 링크를 찾으실 수 있습니다.

## 아키텍처 설계

프로젝트는 CLI 호출의 한계를 해결하기 위해 계층화 아키텍처를 사용합니다:

```
┌─────────────────────────────────────────────────────────┐
│  hagicode-core (Orleans Grains + AI Provider 계층)    │
│  - CopilotAIProvider: AIRequest를 CopilotOptions로 변환 │
│  - GitHubCopilotGrain: Orleans 분산 실행 인터페이스     │
└─────────────────────────────────────────────────────────┘
                          ↓
┌─────────────────────────────────────────────────────────┐
│  HagiCode.Libs (공통 Provider 계층)                  │
│  - CopilotProvider: CLI Provider 인터페이스 구현       │
│  - ICopilotSdkGateway: SDK 호출 추상화                │
│  - GitHubCopilotSdkGateway: SDK 세션 관리 및 이벤트 디스패치 │
└─────────────────────────────────────────────────────────┘
                          ↓
┌─────────────────────────────────────────────────────────┐
│  GitHub Copilot SDK (공식 .NET SDK)                  │
│  - CopilotClient: SDK 클라이언트                      │
│  - CopilotSession: 세션 관리                          │
│  - SessionEvent: 이벤트 스트림                         │
└─────────────────────────────────────────────────────────┘
```

이 계층화 설계는 몇 가지 실용적인 기술적 장점을 제공합니다:

1. **관심사의 분리**: 핵심 비즈니스 로직이 SDK 구현 세부 사항에서 분리됩니다.
2. **테스트 가능성**: `ICopilotSdkGateway` 인터페이스가 단위 테스트를 간단하게 만듭니다.
3. **재사용성**: `HagiCode.Libs`는 여러 프로젝트에서 참조할 수 있습니다.
4. **유지보수성**: SDK 업그레이드는 Gateway 계층의 변경만 필요하며, 상위 계층은 그대로 유지됩니다.

## 핵심 구현

### 인증 흐름

인증은 SDK 통합의 첫 번째이자 가장 중요한 단계입니다. 인증이 실패하면 다른 것은 의미가 없습니다. 우리는 여러 인증 소스를 지원하는 유연한 인증 구성을 설계했습니다:

```csharp
// CopilotProvider.cs - 인증 소스 구성
public class CopilotOptions
{
    public bool UseLoggedInUser { get; set; } = true;
    public string? GitHubToken { get; set; }
    public string? CliUrl { get; set; }
}

// SDK 요청으로 변환
return new CopilotSdkRequest(
    GitHubToken: options.AuthSource == CopilotAuthSource.GitHubToken
        ? options.GitHubToken
        : null,
    UseLoggedInUser: options.AuthSource != CopilotAuthSource.GitHubToken
);
```

이 설계의 장점은 꽤 명확합니다:

- 토큰 없이 로그인한 사용자 모드를 지원하며, 데스크톱 시나리오에 잘 맞습니다.
- GitHub Token 모드를 지원하며, 서버 측 배포 및 중앙 집중식 관리에 적합합니다.
- Copilot CLI URL 재정의를 지원하며, 기업 프록시 구성에 도움이 됩니다.

실제로 이 유연한 인증 모델은 다양한 배포 시나리오에서 구성을 크게 간소화했습니다. 데스크톱 클라이언트는 각 사용자의 Copilot 로그인 상태를 사용할 수 있고, 서버 측은 토큰을 통해 중앙에서 인증을 관리할 수 있습니다.

### 이벤트 스트림 처리

SDK의 가장 강력한 기능 중 하나는 완전한 이벤트 스트림 캡처입니다. 모든 종류의 SDK 이벤트를 실시간으로 처리할 수 있는 이벤트 디스패치 시스템을 구현했습니다:

```csharp
// GitHubCopilotSdkGateway.cs - 이벤트 디스패치 핵심 로직
internal static SessionEventDispatchResult DispatchSessionEvent(
    SessionEvent evt, bool sawDelta)
{
    switch (evt)
    {
        case AssistantReasoningEvent reasoningEvent:
            // 모델 추론 과정 캡처
            events.Add(new CopilotSdkStreamEvent(
                CopilotSdkStreamEventType.ReasoningDelta,
                Content: reasoningEvent.Data.Content));
            break;

        case ToolExecutionStartEvent toolStartEvent:
            // 도구 호출 시작 캡처
            events.Add(new CopilotSdkStreamEvent(
                CopilotSdkStreamEventType.ToolExecutionStart,
                ToolName: toolStartEvent.Data.ToolName,
                ToolCallId: toolStartEvent.Data.ToolCallId));
            break;

        case ToolExecutionCompleteEvent toolCompleteEvent:
            // 도구 완료 및 결과 캡처
            events.Add(new CopilotSdkStreamEvent(
                CopilotSdkStreamEventType.ToolExecutionEnd,
                Content: ExtractToolExecutionContent(toolCompleteEvent)));
            break;

        default:
            // 처리되지 않은 이벤트를 RawEvent로 보존
            events.Add(new CopilotSdkStreamEvent(
                CopilotSdkStreamEventType.RawEvent,
                RawEventType: evt.GetType().Name));
            break;
    }
}
```

이 구현의 가치는 상당합니다:

- **모델 추론 과정의 완전한 캡처** (`thinking`): 사용자는 AI가 추론하는 과정을 볼 수 있으며, 최종 답변만이 아닙니다.
- **도구 실행 상태의 실시간 추적**: 어떤 도구가 실행 중인지, 언제 완료되는지, 무엇을 반환하는지 알 수 있습니다.
- **이벤트 손실 제로**: `RawEvent`로 폴백하여 모든 이벤트가 기록되도록 보장합니다.

HagiCode에서 이러한 세분화된 이벤트는 사용자가 AI의 내부 작동 방식을 이해하는 데 도움이 되며, 특히 복잡한 작업을 디버깅할 때 유용합니다.

### CLI 호환성 처리

CLI 호출에서 SDK로 마이그레이션한 후, 일부 기존 CLI 매개변수가 SDK에서 더 이상 적용되지 않는다는 것을 발견했습니다. 하위 호환성을 유지하기 위해 매개변수 필터링 시스템을 구현했습니다:

```csharp
// CopilotCliCompatibility.cs - 인수 필터링
private static readonly Dictionary<string, string> RejectedFlags = new()
{
    ["--headless"] = "지원되지 않는 시작 인수",
    ["--model"] = "SDK 네이티브 필드를 통해 전달됨",
    ["--prompt"] = "SDK 네이티브 필드를 통해 전달됨",
    ["--interactive"] = "상호작용은 Provider가 관리",
};

public static CopilotCliArgumentBuildResult BuildCliArgs(CopilotOptions options)
{
    // 지원되지 않는 인수를 필터링하고 호환되는 것만 유지
    // 진단 정보 생성
}
```

이로써 여러 가지 이점이 있습니다:

- 호환되지 않는 CLI 인수를 자동으로 필터링하여 런타임 오류를 방지합니다.
- 명확한 진단 메시지를 생성하여 개발자가 문제를 빠르게 찾을 수 있도록 돕습니다.
- SDK를 안정적으로 유지하고 CLI 매개변수 변경으로부터 격리합니다.

업그레이드 중 이 호환성 메커니즘이 원활한 전환을 도왔습니다. 기존 구성 파일을 계속 사용할 수 있었고, 진단 출력에 따라 점진적으로 조정하기만 하면 되었습니다.

### 런타임 풀링

Copilot SDK 세션 생성은 상대적으로 비용이 많으며, 세션을 너무 자주 생성하고 파기하면 성능이 저하됩니다. 이를 해결하기 위해 세션 풀 관리 시스템을 구현했습니다:

```csharp
// CopilotProvider.cs - 세션 풀 관리
await using var lease = await _poolCoordinator.AcquireCopilotRuntimeAsync(
    request,
    async ct => await _gateway.CreateRuntimeAsync(sdkRequest, ct),
    cancellationToken);

if (lease.IsWarmLease)
{
    // 기존 세션 재사용
    yield return CreateSessionReusedMessage();
}

await foreach (var eventData in lease.Entry.Resource.SendPromptAsync(...))
{
    yield return MapEvent(eventData);
}
```

세션 풀링의 장점은 다음과 같습니다:

- **세션 재사용**: 동일한 `sessionId`를 가진 요청은 기존 세션을 재사용하여 시작 오버헤드를 줄입니다.
- **세션 복구 지원**: 네트워크 중단 후 이전 세션 상태를 복원할 수 있습니다.
- **자동 풀 관리**: 만료된 세션이 자동으로 정리되어 리소스 누수를 방지합니다.

HagiCode에서 세션 풀링은 반응성을 눈에 띄게 향상시켰으며, 특히 연속 대화에서 더욱 그러했습니다.

### Orleans 통합

HagiCode는 Orleans를 분산 프레임워크로 사용하며, Copilot SDK를 Orleans Grains에 통합했습니다:

```csharp
// GitHubCopilotGrain.cs - 분산 실행
public async IAsyncEnumerable<GitHubCopilotResponse> ExecuteCommandStreamAsync(
    string command,
    CancellationToken token = default)
{
    var provider = await aiProviderFactory.GetProviderAsync(AIProviderType.GitHubCopilot);

    await foreach (var chunk in provider.SendMessageAsync(request, null, token))
    {
        // 통합 응답 형식으로 매핑
        yield return BuildChunkResponse(chunk, startedAt);
    }
}
```

Orleans 통합의 장점은 상당합니다:

- **통합 AI Provider 추상화**: 다양한 AI Provider 간 전환을 쉽게 만듭니다.
- **다중 테넌트 격리**: 다른 사용자의 Copilot 세션이 서로 격리됩니다.
- **지속적 세션 상태**: 서버 재시작 후에도 세션 상태를 복원할 수 있습니다.

대량의 동시 요청을 처리해야 하는 시나리오에서 Orleans는 강력한 확장성을 제공합니다.

## 실무 가이드

### 구성 예시

완전한 구성 예시입니다:

```json
{
  "AI": {
    "Providers": {
      "Providers": {
        "GitHubCopilot": {
          "Enabled": true,
          "ExecutablePath": "copilot",
          "Model": "gpt-5",
          "WorkingDirectory": "/path/to/project",
          "Timeout": 7200,
          "StartupTimeout": 30,
          "UseLoggedInUser": true,
          "NoAskUser": true,
          "Permissions": {
            "AllowAllTools": false,
            "AllowedTools": ["Read", "Bash", "Grep"],
            "DeniedTools": ["Edit"]
          }
        }
      }
    }
  }
}
```

### 사용 시 참고 사항

실제 사용에서 몇 가지 주의사항을 정리했습니다:

**시작 시간 초과 구성**: Copilot CLI의 첫 시작은 비교적 오래 걸릴 수 있으므로, `StartupTimeout`을 최소 30초로 설정하는 것을 권장합니다. 처음 로그인하는 경우 더 오래 걸릴 수 있습니다.

**권한 관리**: 프로덕션 환경에서는 `AllowAllTools: true`를 사용하지 마십시오. `AllowedTools` 허용 목록을 사용하여 사용 가능한 도구를 제어하고, `DeniedTools` 거부 목록을 사용하여 위험한 작업을 차단하십시오. 이는 AI가 위험한 명령을 실행하는 것을 효과적으로 방지합니다.

**세션 관리**: 동일한 `sessionId`를 가진 요청은 자동으로 세션을 재사용합니다. 세션 상태는 `ProviderSessionId`를 통해 지속됩니다. 취소는 `CancellationTokenSource`를 통해 전파됩니다.

**진단 출력**: 호환되지 않는 CLI 인수는 `diagnostic` 유형의 메시지를 생성합니다. 원시 SDK 이벤트는 `event.raw`로 보존됩니다. 오류 메시지에는 시작 시간 초과 및 인수 호환성 불가와 같은 카테고리가 포함되어 문제 해결을 더 쉽게 만듭니다.

### 모범 사례

실무 경험을 바탕으로 몇 가지 모범 사례를 정리합니다:

**1. 도구 허용 목록 사용**

```csharp
var request = new AIRequest
{
    Prompt = "이 파일을 분석해 줘",
    AllowedTools = new[] { "Read", "Grep", "Bash(git:*)" }
};
```

허용 목록을 통해 허용된 도구를 명시적으로 지정하면 예상치 못한 AI 동작을 방지할 수 있습니다. 이는 `Edit`와 같은 쓰기 권한이 있는 도구에 특히 중요하며, 추가로 주의해야 합니다.

**2. 합리적인 시간 초과 설정**

```csharp
options.Timeout = 3600;  // 1시간
options.StartupTimeout = 60;  // 1분
```

작업 복잡도에 따라 적절한 시간 초과 값을 설정하십시오. 값이 너무 짧으면 작업이 중단될 수 있고, 너무 길면 응답 없는 요청을 기다리며 리소스를 낭비할 수 있습니다.

**3. 세션 재사용 활성화**

```csharp
options.SessionId = "my-session-123";
```

관련 작업에 동일한 `sessionId`를 사용하면 이전 세션 컨텍스트를 재사용하여 응답 속도를 높일 수 있습니다.

**4. 스트리밍 응답 처리**

```csharp
await foreach (var chunk in provider.StreamAsync(request))
{
    switch (chunk.Type)
    {
        case StreamingChunkType.ThinkingDelta:
            // 추론 과정 처리
            break;
        case StreamingChunkType.ToolCallDelta:
            // 도구 호출 처리
            break;
        case StreamingChunkType.ContentDelta:
            // 텍스트 출력 처리
            break;
    }
}
```

스트리밍 응답을 통해 AI 처리 진행 상황을 실시간으로 표시할 수 있어 사용자 경험이 향상됩니다. 이는 장시간 실행되는 작업에 특히 가치가 있습니다.

**5. 오류 처리 및 재시도**

```csharp
try
{
    await foreach (var chunk in provider.StreamAsync(request))
    {
        // 응답 처리
    }
}
catch (CopilotSessionException ex)
{
    // 세션 예외 처리
    logger.LogError(ex, "Copilot 세션 실패");
    // 예외 유형에 따라 재시도 여부 결정
}
```

적절한 오류 처리와 재시도 로직은 전체 시스템의 안정성을 향상시킵니다.

## 결론

CLI 호출에서 SDK 통합으로 업그레이드하는 것은 HagiCode 프로젝트에 상당한 가치를 제공했습니다:

- **향상된 안정성**: SDK는 CLI 버전 변경의 영향을 받지 않는 더 안정적인 인터페이스를 제공합니다.
- **더 완전한 기능**: 추론 및 도구 실행 상태를 포함한 전체 이벤트 스트림을 캡처합니다.
- **더 높은 개발 효율**: 타입 안전 SDK 인터페이스로 개발이 더 효율적이 되고 런타임 오류가 줄어듭니다.
- **더 나은 사용자 경험**: 실시간 이벤트 피드백으로 사용자가 AI가 무엇을 하는지 더 명확히 이해할 수 있습니다.

이 업그레이드는 기술적 구현의 단순한 교체가 아니었습니다. 전체 AI 기능 계층의 아키텍처 최적화이기도 했습니다. 계층화 설계와 추상화 인터페이스를 통해 더 나은 유지보수성과 확장성을 얻었습니다.

자체 .NET 프로젝트에 GitHub Copilot을 통합하는 것을 고려 중이시라면, 이러한 실무 교훈이 불필요한 우회를 피하는 데 도움이 되기를 바랍니다. 공식 SDK는 CLI 호출보다 확실히 더 안정적이고 완전하며, 이해하고 채택할 가치가 있습니다.
