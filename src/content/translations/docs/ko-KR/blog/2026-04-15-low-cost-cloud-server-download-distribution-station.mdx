## 저비용 클라우드 서버로 빠른 다운로드 배포站 구축하기

> 클라우드 스토리지 대역폭은 비싸기 짝이 없고, 크로스보더 접근은 끔찍하게 느리며, CDN 가격은 누구나 겁날 정도입니다... 파일 배포를 다루신다면 아마 이 문제들을 잘 아실 것입니다. 이 글에서는 [HagiCode](https://hagicode.com)를 구축하면서 알아낸 저비용 접근 방식을 공유합니다. 클라우드 서버에 Nginx 캐싱 레이어를 더하는 것입니다. 비용은 약 절반으로 줄었고, 다운로드 속도도 꽤 향상되어 적어도 약간의 위안이 되었습니다.

## 배경

인터넷에서 다운로드 속도와 안정성은 정말 사용자 경험의 일부입니다. 오픈소스 프로젝트든 상업 제품이든 사용자에게 신뢰할 수 있는 파일 다운로드 방법을 제공해야 합니다.

Azure Blob Storage나 AWS S3와 같은 클라우드 스토리지에서 파일을 직접 다운로드하는 것은 간단해 보이지만, 실제로는 꽤 몇 가지 문제가 있습니다.

**네트워크 지연**: 크로스보더 및 크로스 리전 접근은 키보드를 칠 만큼 느릴 수 있습니다. 사용자가 영원히 기다려야 한다면 경험은 분명 좋지 않을 것입니다.

**대역폭 비용**: 클라우드 스토리지 송신 트래픽은 끔찍하게 비쌉니다. 중국 본토에서 Azure Blob Storage에 접근하면 1GB당 약 0.5위안이 들며, 월 1TB이면 약 500위안에 달합니다. 소규모 팀에게 무시할 수 없는 금액입니다. 아무도 돈이 바람에서 떨어지지는 않으니까요.

**접근 제한**: 일부 지역에서는 해외 클라우드 서비스에 대한 접근이 불안정하며, 때로는 단순히 사용할 수 없습니다. 사용자는 파일을 다운로드하고 싶지만 할 수 없으며, 모두에게 좌절스럽습니다.

**CDN 비용**: 상업적 CDN은 이러한 문제를 해결할 수 있지만 가격도 현실적입니다. 대부분의 소규모 팀은 감당하기 어렵습니다.

그렇다면 저렴하면서도 실용적인 해결책이 있을까요? 있습니다. 클라우드 서버, 리버스 프록시, 캐싱 레이어를 사용하세요. 간단한 접근 방식이지만 작동합니다. 비용은 약 절반으로 줄고 속도도 향상되며, 꽤 괜찮은 트레이드오프입니다.

## 아키텍처 설계

### 전체 아키텍처

전체 아키텍처부터 시작해 보겠습니다.

```
사용자 요청
    ↓
DNS 확인
    ↓
┌─────────────────────────────────────┐
│   리버스 프록시 레이어 (Traefik/Bunker Web) │ ← SSL 종료, 라우팅, 보안 보호
├─────────────────────────────────────┤
│   포트: 80/443                     │
│   기능: Let's Encrypt 자동 인증서 │
│             호스트 라우팅            │
└─────────────────────────────────────┘
    ↓
┌─────────────────────────────────────┐
│   캐시 레이어 (Nginx)               │ ← 파일 캐싱, Gzip 압축
├─────────────────────────────────────┤
│   포트: 8080(서버) / 8081(데스크톱) │
│   캐시 전략:                   │
│   - index.json: 1시간              │
│   - 기타 파일: 7일              │
│   캐시 크기: 1GB                   │
└─────────────────────────────────────┘
    ↓
┌─────────────────────────────────────┐
│   오리진 (Azure Blob Storage)       │ ← 파일 저장
└─────────────────────────────────────┘
```

이 아키텍처의 핵심 아이디어는 간단합니다. **사용자와 클라우드 스토리지 사이에 캐시를 둡니다**.

사용자 요청은 먼저 클라우드 서버의 리버스 프록시 레이어에 도착한 다음 Nginx 캐시 레이어가 인계받습니다. 요청한 파일이 이미 캐시되어 있으면 즉시 반환됩니다. 그렇지 않으면 Nginx가 클라우드 스토리지에서 가져와 동시에 로컬 사본을 저장합니다. 다음에 누군가 같은 파일을 요청하면 클라우드 스토리지를 다시 거칠 필요가 없습니다.

### 이 아키텍처를 선택한 이유는?

**클라우드 서버의 장점**:
- 예측 가능한 비용: 알리바바 클라우드 등의 제공자는 저비용 클라우드 서버를 제공하며, 1-2 vCPU와 2GB RAM 인스턴스가 월 약 50-100위안 수준입니다
- 유연한 배포: 리버스 프록시 규칙과 캐싱 정책을 자유롭게 구성할 수 있습니다
- 유연한 지리: 사용자에게 더 가까운 서버 지역을 선택할 수 있습니다
- 좋은 확장성: 트래픽이 증가함에 따라 서버 사양을 업그레이드할 수 있습니다

**리버스 프록시 + 캐시 아키텍처**:
- 오리진 부하 감소: 핫 파일을 캐싱하여 클라우드 스토리지에 대한 직접 접근을 줄입니다
- 비용 절감: 클라우드 서버 트래픽은 클라우드 스토리지 송신보다 훨씬 저렴합니다
- 속도 향상: 근접 접근과 서버 대역폭은 일반적으로 직접 클라우드 스토리지 배달보다 낫습니다

**왜 Nginx를 캐시 레이어로 선택했는가?**

이것은 무작위 선택이 아닙니다. Nginx는 여기서 몇 가지 실제 장점이 있습니다.

1. 고성능: Nginx는 탁월한 리버스 프록시 성능으로 널리 인정받습니다
2. 성숙한 캐싱: 내장 `proxy_cache` 기능은 안정적이고 신뢰할 수 있습니다
3. 낮은 리소스 사용: 256MB 메모리만으로도 실행할 수 있습니다
4. 유연한 구성: 다른 파일 유형에 다른 캐시 정책을 적용할 수 있습니다

## 리버스 프록시 레이어: Traefik vs Bunker Web

HagiCode의 배포 솔루션은 두 가지 리버스 프록시 옵션을 지원하며, 각각 고유한 장점이 있습니다.

| 옵션 | 특징 | 적합한 시나리오 |
|------|------|----------|
| **Traefik** | 경량, 자동 SSL, 간단한 구성 | 기본 배포, 저트래픽 시나리오 |
| **Bunker Web** | 내장 WAF, DDoS 보호, 봇 보호 | 고보안, 고트래픽 시나리오 |

### Traefik: 경량 첫 번째 선택

Traefik은 현대적인 HTTP 리버스 프록시 및 로드 밸런서입니다. 가장 큰 장점은 구성이 간단하고 Let's Encrypt 인증서를 자동으로 얻을 수 있다는 것입니다.

초기 배포나 저트래픽 시나리오에서 Traefik은 종종 매우 좋은 선택입니다.
- 비교적 적은 리소스를 사용합니다. 1.5 CPU와 512MB 메모리면 충분합니다
- SSL 인증서가 자동으로 구성되므로 직접 관리할 필요가 없습니다
- 라우팅은 Docker 레이블을 통해 구성되어 편리합니다

### Bunker Web: 고보안 시나리오용

Bunker Web은 Nginx 기반 웹 애플리케이션 방화벽으로 더 포괄적인 보안 보호를 제공합니다.

언제 Bunker Web으로 전환을 고려해야 할까요? 일반적으로 다음과 같은 경우입니다.
- DDoS 공격을 받고 있을 때
- ModSecurity 보호가 필요할 때
- 봇 보호가 필요할 때
- 더 엄격한 보안 요구 사항이 있을 때

HagiCode는 `switch-deployment.sh` 스크립트를 제공하여 두 옵션 간에 빠르게 전환할 수 있습니다.

```bash
# Bunker Web으로 전환
./switch-deployment.sh bunkerweb

# Traefik으로 돌아가기
./switch-deployment.sh traefik

# 현재 상태 확인
./switch-deployment.sh status
```

스크립트는 사전 검사, 상태 확인 및 자동 롤백을 수행하므로 전환 과정이 꽤 안전하고 신뢰할 수 있습니다.

## Nginx 캐시 레이어 구성

캐시 레이어는 전체 아키텍처의 핵심이므로 Nginx 구성이 캐시 성능에 큰 차이를 만듭니다.

### 캐시 경로 구성

```nginx
# 캐시 경로 구성
proxy_cache_path /var/cache/nginx levels=1:2 keys_zone=azure_cache:10m
                   max_size=1g inactive=7d use_temp_path=off;
```

매개변수 상세:
- `levels=1:2`: 파일 접근 효율을 높이기 위한 두 수준의 캐시 디렉토리 계층
- `keys_zone=azure_cache:10m`: 캐시 키 저장 영역. 10MB는 많은 수의 키에 충분합니다
- `max_size=1g`: 최대 캐시 크기는 1GB입니다
- `inactive=7d`: 7일 동안 접근되지 않은 캐시 파일을 삭제합니다
- `use_temp_path=off`: 더 나은 성능을 위해 캐시 디렉토리에 직접 씁니다

### 계층화된 캐시 전략

다른 파일 유형에는 다른 캐시 전략이 필요합니다.

```nginx
# 서버 다운로드 서비스
server {
    listen 8080;

    # index.json 단기 캐시 (적시 업데이트 허용)
    location /index.json {
        proxy_cache azure_cache;
        proxy_cache_valid 200 1h;
        proxy_cache_key "$scheme$server_port$request_uri";
        add_header X-Cache-Status $upstream_cache_status;
        add_header Cache-Control "public, max-age=3600";

        # Azure OSS로 리버스 프록시
        proxy_pass https://${SERVER_DL_HOST}/${SERVER_DL_CONTAINER}$uri?${SERVER_DL_SAS_TOKEN};
        proxy_ssl_server_name on;
        proxy_ssl_protocols TLSv1.2 TLSv1.3;
    }

    # 설치 패키지 등 정적 파일 장기 캐시
    location / {
        proxy_cache azure_cache;
        proxy_cache_valid 200 7d;
        proxy_cache_key "$scheme$server_port$request_uri";
        add_header X-Cache-Status $upstream_cache_status;
        add_header Cache-Control "public, max-age=604800";

        proxy_pass https://${SERVER_DL_HOST}/${SERVER_DL_CONTAINER}$uri?${SERVER_DL_SAS_TOKEN};
        proxy_ssl_server_name on;
        proxy_ssl_protocols TLSv1.2 TLSv1.3;
    }
}
```

**왜 이렇게 설계했을까요?**

`index.json`은 버전 확인 파일이므로 즉시 업데이트해야 합니다. 1시간 캐시 윈도우를 사용하면 사용자는 게시 후 최대 1시간 이내에 새 릴리스를 감지할 수 있습니다.

설치 패키지와 같은 정적 파일은 자주 변경되지 않으므로 7일 동안 캐싱하면 오리진 접근이 크게 줄어듭니다. 업데이트가 필요할 때 캐시를 수동으로 삭제하면 됩니다.

**`X-Cache-Status` 응답 헤더**:

이 헤더는 캐시 적중 동작을 검사하는 데 도움이 됩니다.
- `HIT`: 캐시 적중
- `MISS`: 캐시 미스, 오리진에서 가져옴
- `EXPIRED`: 캐시 만료, 오리진에서 다시 가져옴
- `BYPASS`: 캐시 바이패스

확인 방법:
```bash
curl -I https://server.dl.hagicode.com/app.zip
```

## 비용 분석

월 1TB의 다운로드 트래픽을 가정해 보겠습니다. 계산해 보겠습니다.

| 옵션 | 트래픽 비용 | 서버 비용 | 합계 |
|------|----------|------------|------|
| Azure OSS 직접 | 약 500위안 | 0위안 | 500위안 |
| 클라우드 서버 + OSS (80% 캐시 적중률) | 100위안 + 80위안 | 60위안 | 240위안 |
| 상업적 CDN | 300-500위안 | 0위안 | 300-500위안 |

**결론**: 캐시 레이어를 추가하면 배포 비용이 약 50% 감소합니다.

이 추정은 80%의 캐시 적중률을 가정합니다. 실제로 파일이 자주 변경되지 않으면 적중률은 더 높을 수 있습니다.

## 운영 팁

### 캐시 관리

캐시도 때때로 유지 관리가 필요합니다.

**캐시 디스크 사용량 확인**:
```bash
docker volume inspect docker_nginx-cache
du -sh /var/lib/docker/volumes/docker_nginx-cache/_data
```

**캐시 수동 삭제**:
```bash
./clear-cache.sh
```

또는 필요한 경우 수동 명령어를 직접 실행할 수 있습니다.
```bash
docker exec nginx sh -c "rm -rf /var/cache/nginx/*"
docker restart nginx
```

### 리소스 제한

1코어, 2GB 서버에서 리소스 제한 구성은 다음과 같습니다.

```yaml
services:
  traefik:
    deploy:
      resources:
        limits:
          cpus: '1.50'
          memory: 512M

  nginx:
    deploy:
      resources:
        limits:
          cpus: '0.50'
          memory: 256M
```

리소스 사용량을 모니터링하려면 가끔 다음을 실행할 수 있습니다.
```bash
docker stats
```

### SAS Token 보안 관행

SAS Token은 Azure Blob Storage에 접근하는 데 사용되는 자격 증명이므로 유출되면 심각합니다.

- `.env` 파일을 버전 관리에 커밋하지 마세요. 이미 `.gitignore`에 있습니다
- 적절한 SAS Token 만료 시간을 설정하세요. 1년을 권장합니다
- SAS Token 권한을 읽기 전용으로 제한하세요
- SAS Token을 정기적으로 교체하세요

### 모니터링 및 알림

HagiCode는 Lark/Feishu Webhook 알림을 통합하여 다음 이벤트에 대한 알림을 보낼 수 있습니다.
- 배포 성공 또는 실패
- 캐시 삭제 상태
- 서비스 예외

알림에는 서버 정보, 타임스탬프, 오류 세부 사항이 포함되어 문제 해결이 훨씬 빨라집니다.

### 고가용성 확장

한 서버로 충분하지 않을 때 다음을 고려할 수 있습니다.

1. **수평 확장**: 여러 노드를 배포하고 DNS 라운드 로빈 또는 로드 밸런서로 트래픽을 분산합니다
2. **앞에 CDN**: 클라우드 서버 앞에 CDN을 두면 더 빠른 접근이 가능합니다
3. **캐시 웜업**: 스크립트를 사용하여 핫 파일을 캐시에 미리 로드합니다

## 주의 사항

몇 가지 기억해 둘 만한 것이 있습니다.

1. **SSL 인증서**: Let's Encrypt에는 속도 제한이 있으므로 배포를 너무 자주 전환하지 마세요. 인증서 발급이 실패할 수 있습니다
2. **캐시 삭제**: 중요한 파일을 업데이트한 후 캐시를 삭제하는 것을 잊지 마세요. 그렇지 않으면 사용자가 여전히 이전 버전을 다운로드할 수 있습니다
3. **로그 관리**: Docker 로그를 정기적으로 정리하세요. 그렇지 않으면 디스크가 가득 찰 수 있습니다
4. **백업 전략**: `Traefik acme.json` 및 Bunker Web 구성과 같은 파일을 백업하세요
5. **모니터링 및 알림**: Feishu 알림을 구성하여 배포 상태를 추적하고 문제에 빠르게 대응하세요

## 결론

클라우드 서버와 Nginx 캐싱 레이어면 충분합니다. HagiCode는 이 해결책을 사용하며 월 비용이 꽤 낮습니다. 서버 비용은 약 60-100위안이며, 결과는 매우 좋습니다. 주요 장점은 다음과 같습니다.

- **예측 가능한 비용**: 클라우드 스토리지 직접 사용이나 상업적 CDN보다 약 50% 저렴합니다
- **유연한 배포**: 필요에 따라 Traefik 또는 Bunker Web을 선택할 수 있습니다
- **강력한 확장성**: 나중에 수평 확장하거나 CDN을 추가할 수 있습니다
- **간단한 운영**: Shell 스크립트와 Ansible로 자동화된 배포가 간단해집니다

파일 배포가 필요한 소규모 팀과 인디 개발자에게 이것은 확실히 시도해 볼 가치가 있는 실용적인 옵션입니다.

HagiCode는 이 아키텍처를 프로덕션에서 안정적으로 실행해 왔으며, 전 세계 사용자의 다운로드는 신뢰할 수 있게 유지되었습니다. 비슷한 해결책을 찾고 계시다면 시도해 볼 가치가 있습니다.
