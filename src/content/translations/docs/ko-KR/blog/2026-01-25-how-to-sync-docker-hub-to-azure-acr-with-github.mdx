## Docker Hub에서 Azure ACR로 자동 이미지 동기화 구현하기

> 이 글에서는 GitHub Actions와 image-syncer 도구를 사용하여 Docker Hub 이미지에서 Azure Container Registry로 자동화된 동기화를 구현하는 방법을 소개합니다. 중국 내 및 일부 Azure 지역에서 Docker Hub 액세스 속도가 느린 문제를 해결하고, 이미지의 가용성과 Azure 환경의 배포 효율을 향상시킵니다.

## 배경/서론

HagiCode 프로젝트는 Docker 이미지를 핵심 런타임 컴포넌트로 사용하며, 주요 이미지는 Docker Hub에 호스팅되어 있습니다. 프로젝트가 발전하고 Azure 환경 배포 요구가 증가함에 따라 다음과 같은 문제점에 직면했습니다:
- 이미지 풀 속도가 느리고, 중국 내 및 일부 Azure 지역에서 Docker Hub 액세스가 제한됨
- 단일 이미지 소스에 의존하면 단일 장애점(SPOF) 위험이 존재함
- Azure 환경에서 Azure Container Registry를 사용하면 더 나은 네트워크 성능과 통합 경험을 얻을 수 있음

이러한 문제를 해결하기 위해 Docker Hub의 이미지를 정기적으로 Azure ACR에 동기화하는 자동화된 이미지 동기화 메커니즘을 구축해야 합니다. 사용자가 Azure 환경에서 더 빠른 이미지 풀 속도와 더 높은 가용성을 확보할 수 있도록 합니다.

## 핵심 해결 방안

GitHub Actions + image-syncer 자동화 방안을 채택하여 Docker Hub에서 Azure ACR로 이미지 동기화를 구현합니다.

## 구현 단계

### 1. 준비 단계
- Azure Portal에서 Azure Container Registry 생성 또는 확인
- ACR 액세스 키(사용자 이름과 비밀번호) 생성
- Docker Hub 이미지 저장소 접근 권한 확인

### 2. GitHub Secrets 구성
GitHub 저장소 설정에 다음 Secrets를 추가합니다:
- AZURE_ACR_USERNAME: Azure ACR 사용자 이름
- AZURE_ACR_PASSWORD: Azure ACR 비밀번호

### 3. GitHub Actions 워크플로우 생성
.github/workflows/sync-docker-acr.yml에 워크플로우를 구성합니다:
- 정기 트리거: 매일 UTC 00:00
- 수동 트리거: workflow_dispatch 지원
- 추가 트리거: publish 브랜치 푸시 시 트리거(빠른 동기화용)

### 4. 워크플로우 실행 흐름

| 순서 | 참여자 | 동작 | 설명 |
| --- | --- | --- | --- |
| 1 | GitHub Actions | 워크플로우 트리거 | 정기, 수동 또는 `publish` 브랜치 푸시로 트리거 |
| 2 | GitHub Actions → image-syncer | 동기화 도구 다운로드 및 실행 | 실제 동기화 단계 진입 |
| 3 | image-syncer → Docker Hub | 이미지 매니페스트 및 태그 목록 획득 | 소스 저장소 메타데이터 읽기 |
| 4 | image-syncer → Azure ACR | 대상 저장소의 기존 이미지 정보 획득 | 대상 측 현재 상태 판단 |
| 5 | image-syncer | 소스와 대상의 차이 비교 | 동기화가 필요한 이미지 레이어 식별 |
| 6 | image-syncer → Docker Hub | 변경된 이미지 레이어 풀 | 업데이트가 필요한 콘텐츠만 전송 |
| 7 | image-syncer → Azure ACR | 변경된 이미지 레이어 푸시 | 증분 동기화 완료 |
| 8 | image-syncer → GitHub Actions | 동기화 통계 정보 반환 | 결과, 차이 및 오류 정보 포함 |
| 9 | GitHub Actions | 로그 기록 및 artifact 업로드 | 향후 감사 및 문제 해결에 활용 |

## GitHub Actions 워크플로우 구현

다음은 실제로 사용 중인 워크플로우 구성입니다 (.github/workflows/sync-docker-acr.yml):

```yaml
name: Sync Docker Image to Azure ACR

on:
  schedule:
    - cron: "0 0 * * *" # 매일 UTC 00:00
  workflow_dispatch: # 수동 트리거
  push:
    branches: [publish]

permissions:
  contents: read

jobs:
  sync:
    runs-on: ubuntu-latest

    steps:
      - name: Checkout code
        uses: actions/checkout@v4

      - name: Download image-syncer
        run: |
          # image-syncer 바이너리 다운로드
          wget https://github.com/AliyunContainerService/image-syncer/releases/download/v1.5.5/image-syncer-v1.5.5-linux-amd64.tar.gz
          tar -zxvf image-syncer-v1.5.5-linux-amd64.tar.gz
          chmod +x image-syncer

      - name: Create auth config
        run: |
          # 인증 구성 파일 생성 (YAML 형식)
          cat > auth.yaml <<EOF
          hagicode.azurecr.io:
            username: "${{ secrets.AZURE_ACR_USERNAME }}"
            password: "${{ secrets.AZURE_ACR_PASSWORD }}"
          EOF

      - name: Create images config
        run: |
          # 이미지 동기화 구성 파일 생성 (YAML 형식)
          cat > images.yaml <<EOF
          docker.io/newbe36524/hagicode: hagicode.azurecr.io/hagicode
          EOF

      - name: Run image-syncer
        run: |
          # 동기화 실행 (새 버전 --auth 및 --images 매개변수 사용)
          ./image-syncer --auth=./auth.yaml --images=./images.yaml --proc=10 --retries=3

      - name: Upload logs
        if: always()
        uses: actions/upload-artifact@v4
        with:
          name: sync-logs
          path: image-syncer-*.log
          retention-days: 7
```

## 구성 설명

### 1. 트리거 조건
- 정기 트리거: cron: "0 0 * * *" - 매일 UTC 00:00 실행
- 수동 트리거: workflow_dispatch - GitHub UI에서 수동 실행 허용
- 푸시 트리거: push: branches: [publish] - publish 브랜치 푸시 시 트리거(빠른 동기화용)

### 2. 인증 구성 (auth.yaml)
```yaml
hagicode.azurecr.io:
  username: "${{ secrets.AZURE_ACR_USERNAME }}"
  password: "${{ secrets.AZURE_ACR_PASSWORD }}"
```

### 3. 이미지 동기화 구성
```yaml
docker.io/newbe36524/hagicode: hagicode.azurecr.io/hagicode
```
이 구성은 docker.io/newbe36524/hagicode의 모든 태그를 hagicode.azurecr.io/hagicode로 동기화하는 것을 의미합니다.

### 4. image-syncer 매개변수
- --auth=./auth.yaml: 인증 구성 파일 경로
- --images=./images.yaml: 이미지 동기화 구성 파일 경로
- --proc=10: 동시 스레드 수 10
- --retries=3: 실패 시 3회 재시도

## GitHub Secrets 구성 체크리스트

GitHub 저장소의 Settings → Secrets and variables → Actions에서 구성합니다:

| Secret 이름 | 설명 | 예시 값 | 획득 방법 |
|------------|------|--------|---------|
| AZURE_ACR_USERNAME | Azure ACR 사용자 이름 | hagicode | Azure Portal → ACR → Access keys |
| AZURE_ACR_PASSWORD | Azure ACR 비밀번호 | xxxxxxxx-xxxx-xxxx-xxxx-xxxxxxxxxxxx | Azure Portal → ACR → Access keys → Password |

## 주의사항 및 모범 사례

### 1. 보안 권장사항
- Azure ACR 비밀번호를 정기적으로 교체하세요(90일마다 권장)
- 전용 ACR 서비스 계정을 사용하고 권한을 푸시 전용으로 제한하세요
- ACR 액세스 로그를 모니터링하여 비정상적인 접근을 적시에 발견하세요
- 로그에 인증 정보를 출력하지 마세요
- 인증 정보를 코드 저장소에 커밋하지 마세요

### 2. 성능 최적화
- --proc 매개변수 조정: 네트워크 대역폭에 따라 동시성을 조정하세요(5-20 권장)
- 동기화 시간 모니터링: 동기화 시간이 너무 길면 동시성을 줄이는 것을 고려하세요
- 로그 정기 정리: 적절한 retention-days를 설정하세요(현재 7일)

### 3. 문제 해결

#### 문제 1: 인증 실패
```
Error: failed to authenticate to hagicode.azurecr.io
```
해결 방법:
1. GitHub Secrets가 올바르게 구성되었는지 확인합니다
2. Azure ACR 비밀번호가 만료되었는지 검증합니다
3. ACR 서비스 계정 권한이 올바른지 확인합니다

#### 문제 2: 네트워크 시간 초과
```
Error: timeout waiting for response
```
해결 방법:
1. 네트워크 연결을 확인합니다
2. 동시 스레드 수를 줄입니다(--proc 매개변수)
3. 네트워크 복구 후 워크플로우를 다시 트리거합니다

#### 문제 3: 이미지 동기화 불완전
```
Warning: some tags failed to sync
```
해결 방법:
1. 동기화 로그를 확인하여 실패한 태그를 식별합니다
2. 워크플로우를 수동으로 트리거하여 다시 동기화합니다
3. Docker Hub 소스 이미지가 정상인지 검증합니다

### 4. 모니터링 및 알림
- 정기적으로 Actions 페이지를 확인하여 워크플로우 실행 상태를 확인합니다
- GitHub 알림을 설정하여 워크플로우 실패 알림을 적시에 수신합니다
- Azure ACR의 스토리지 사용량을 모니터링합니다
- 정기적으로 이미지 태그 일관성을 검증합니다

## 자주 묻는 질문 및 해결 방안

### Q1: 모든 태그 대신 특정 태그를 동기화하려면 어떻게 하나요?

images.yaml 구성 파일을 수정합니다:
```yaml
# latest 및 v1.0 태그만 동기화
docker.io/newbe36524/hagicode:latest: hagicode.azurecr.io/hagicode:latest
docker.io/newbe36524/hagicode:v1.0: hagicode.azurecr.io/hagicode:v1.0
```

### Q2: 여러 이미지 저장소를 동기화하려면 어떻게 하나요?

images.yaml에 여러 줄의 구성을 추가합니다:
```yaml
docker.io/newbe36524/hagicode: hagicode.azurecr.io/hagicode
docker.io/newbe36524/another-image: hagicode.azurecr.io/another-image
```

### Q3: 동기화 실패 후 어떻게 재시도하나요?
- 자동 재시도: image-syncer에 내장된 재시도 메커니즘(기본 3회)
- 수동 재시도: GitHub Actions 페이지에서 Re-run all jobs 클릭

### Q4: 동기화의 상세 진행 상황은 어떻게 확인하나요?
- Actions 페이지에서 실시간 로그 확인
- sync-logs artifact를 다운로드하여 전체 로그 파일 확인
- 로그 파일에는 각 태그의 동기화 상태 및 전송 속도가 포함됩니다

### Q5: 동기화에 얼마나 걸리나요?
- 첫 전체 동기화: 이미지 크기에 따라 일반적으로 10-30분 소요
- 증분 동기화: 이미지 변경이 적으면 일반적으로 2-5분 소요
- 시간은 네트워크 대역폭, 이미지 크기 및 동시성 설정에 따라 다릅니다

## 확장 기능 제안

### 1. 동기화 알림 추가
워크플로우에 알림 단계를 추가합니다:
```yaml
- name: Notify on success
  if: success()
  run: |
    echo "Docker images synced successfully to Azure ACR"
```

### 2. 이미지 태그 필터링 구현
워크플로우에 태그 필터링 로직을 추가합니다:
```yaml
- name: Filter tags
  run: |
    # v로 시작하는 태그만 동기화
    echo "docker.io/newbe36524/hagicode:v* : hagicode.azurecr.io/hagicode:v*" > images.yaml
```

### 3. 동기화 통계 보고서 추가
```yaml
- name: Generate report
  if: always()
  run: |
    echo "## Sync Report" >> $GITHUB_STEP_SUMMARY
    echo "- Total tags: $(grep -c 'synced' image-syncer-*.log)" >> $GITHUB_STEP_SUMMARY
    echo "- Sync time: ${{ steps.sync.outputs.duration }}" >> $GITHUB_STEP_SUMMARY
```

## 요약

이 글에서 소개한 방법을 통해 Docker Hub에서 Azure ACR로의 자동화된 이미지 동기화를 성공적으로 구현했습니다. 이 방안은 GitHub Actions의 정기 트리거 및 수동 트리거 기능과 image-syncer의 증분 동기화 및 오류 처리 메커니즘을 결합하여 이미지의 적시 동기화와 일관성을 보장합니다.

또한 보안 모범 사례, 성능 최적화, 문제 해결 등의 주제를 논의하여 사용자가 이 동기화 메커니즘을 더 잘 관리하고 유지보수할 수 있도록 도왔습니다. Azure 환경에서 Docker 이미지를 배포해야 하는 개발자에게 가치 있는 참고 자료가 되기를 바랍니다.
