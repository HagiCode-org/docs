## TypeScript에서 C#으로: Codex SDK의 크로스 언어 포팅 실천

> 이 글은 공식 TypeScript Codex SDK를 완전히 C#으로 포팅한 전체 과정을 기록한 것입니다. "포팅"이라고 하지만 사실 더 긴 모험에 가깝습니다. 두 언어의 성격이 다르고, 잘 지내도록 해야 하니까요.

## 배경

Codex는 OpenAI가 출시한 AI Agent CLI 도구로 꽤 강력합니다. 공식 TypeScript SDK는 `@openai/codex` 패키지에 있습니다. `codex exec --experimental-json` 명령을 호출하여 Codex CLI와 상호작용하고 JSONL 형식의 이벤트 스트림을 파싱합니다.

하지만 우리 HagiCode 프로젝트에서는 순수 .NET 환경에서 사용해야 합니다. 구체적으로 C# 백엔드 서비스와 데스크톱 애플리케이션입니다. .NET 프로젝트에서 CLI 도구 하나를 호출하기 위해 Node.js 런타임을 도입하는 건 너무 번거롭습니다.

그래서 두 가지 선택지가 있었습니다: 복잡한 Node.js 브리징 레이어를 유지하거나, 직접 네이티브 C# SDK를 구현하는 것입니다.

우리는 후자를 선택했습니다.

## 핵심 내용

### 아키텍처 설계 비교

코드 레벨의 변환을 시작하기 전에 두 SDK의 아키텍처 설계를 먼저 이해해야 합니다. 지피를 알면 백 번 싸움입니다.

TypeScript SDK의 핵심 아키텍처는 다음과 같습니다:

```
Codex (입력 클래스)
  └── CodexExec (실행기, 하위 프로세스 관리)
      └── Thread (대화 스레드)
          ├── run() / runStreamed() (동기/비동기 실행)
          └── 이벤트 스트림 파싱
```

C# SDK는 동일한 아키텍처 계층을 유지하지만 구현 세부 사항에서 적응했습니다. 전체적인 접근 방식은 API의 일관성을 유지하면서 C# 언어의 특성을 최대한 활용하는 것입니다. 언어가 다르니 약간의 차이는 있어야 합니다.

### 타입 시스템 변환

이것은 가장 기초적이면서 가장 중요한 작업입니다. 기초가 튼튼튼히면 나중에 모두 문제입니다.

TypeScript의 타입 시스템은 C#보다 유연합니다. 적절한 매핑 방법을 찾아야 합니다:

| TypeScript | C# | 설명 |
|------------|-----|------|
| `interface` / `type` | `record` | C#은 record로 불변 데이터 구조를 구현합니다 |
| `string \| null` | `string?` | nullable 참조 타입 |
| `boolean \| undefined` | `bool?` | nullable bool 값 |
| `AsyncGenerator` | `IAsyncEnumerable` | 비동기 반복자 |

**이벤트 타입 시스템**이 전형적인 예입니다. TypeScript는 유니온 타입을 사용하여 이벤트를 정의합니다:

```typescript
export type ThreadEvent =
  | ThreadStartedEvent
  | TurnStartedEvent
  | TurnCompletedEvent
  | ...
```

C#에서는 상속 계층과 패턴 매칭으로 유사한 효과를 구현합니다:

```csharp
public abstract record ThreadEvent(string Type);

public sealed record ThreadStartedEvent(string ThreadId) : ThreadEvent("thread.started");
public sealed record TurnStartedEvent() : ThreadEvent("turn.started");
public sealed record TurnCompletedEvent(Usage Usage) : ThreadEvent("turn.completed");
// ...
```

`record`를 `class` 대신 사용하는 이유는 이벤트 객체가 불변이어야 하기 때문이며, TypeScript에서 일반 객체를 사용하는 것과 같습니다. `sealed` 키워드는 추가 상속이 없도록 하여 컴파일러가 최적화할 수 있게 합니다.

### 핵심 변환 포인트

#### 1. 이벤트 파서

이벤트 파싱은 전체 SDK의 핵심입니다. Codex CLI가 반환하는 모든 정보를 올바르게 이해하는 것이 중요합니다. 파싱이 틀리면 이후의 모든 작업이 헛수고입니다.

TypeScript 버전은 `JSON.parse()`를 사용하여 각 줄 JSON을 파싱합니다:

```typescript
export function parseEvent(line: string): ThreadEvent {
  const data = JSON.parse(line);
  // 처리各种事件类型...
}
```

C# 버전은 `System.Text.Json.JsonDocument`를 사용합니다:

```csharp
public static ThreadEvent Parse(string line)
{
    using var document = JsonDocument.Parse(line);
    var root = document.RootElement;
    var type = GetRequiredString(root, "type", "event.type");

    return type switch
    {
        "thread.started" => new ThreadStartedEvent(GetRequiredString(root, "thread_id", ...)),
        "turn.started" => new TurnStartedEvent(),
        "turn.completed" => new TurnCompletedEvent(ParseUsage(...)),
        // ...
        _ => new UnknownThreadEvent(type, root.Clone()),
    };
}
```

여기서 작은 팁: `root.Clone()`은 필수입니다. `JsonDocument`의 요소는 문서가 해제된 후 무효화되므로 알 수 없는 이벤트 타입을 위해 사본을 보존해야 합니다.

#### 2. 프로세스 관리 차이

이것이 두 SDK의 가장 큰 차이점입니다. Node.js와 .NET의 성격이 다르니 적응해야 합니다.

TypeScript는 Node.js의 `spawn()` 함수를 사용합니다:

```typescript
const child = spawn(this.executablePath, commandArgs, { env, signal });
```

C#는 .NET의 `System.Diagnostics.Process`를 사용합니다:

```csharp
using var process = new Process { StartInfo = startInfo };
process.Start();

// stdin/stdout/stderr를 수동으로 관리해야 합니다
```

구체적으로 C# 버전은 다음과 같이 프로세스를 구성합니다:

```csharp
var startInfo = new ProcessStartInfo
{
    FileName = _executablePath,
    RedirectStandardInput = true,
    RedirectStandardOutput = true,
    RedirectStandardError = true,
    UseShellExecute = false,
    CreateNoWindow = true,
};
```

가장 큰 차이는 취소 메커니즘입니다. TypeScript는 `AbortSignal`을 사용하며, 이는 Web API의 일부로 사용하기 매우 편리합니다:

```typescript
const child = spawn(cmd, args, { signal: cancellationSignal });
```

C#는 `CancellationToken`을 사용합니다:

```csharp
public async IAsyncEnumerable<string> RunAsync(
    CodexExecArgs args,
    [EnumeratorCancellation] CancellationToken cancellationToken = default)
{
    while (!cancellationToken.IsCancellationRequested)
    {
        // 출력 처리...
    }

    if (cancellationToken.IsCancellationRequested)
    {
        try { process.Kill(entireProcessTree: true); } catch { }
    }
}
```

이러한 차이는 Web API와 .NET 생태계의 차이라고 할 수 있습니다.

#### 3. 구성 직렬화 유지

두 SDK 모두 JSON 구성을 TOML 구성으로 변환하는 로직을 구현합니다. Codex CLI는 TOML 형식의 구성 덮어쓰기를 지원하므로 이 로직이 완전히 일치해야 합니다. 그렇지 않으면 동일한 구성이 두 SDK에서 다른 동작을 생성합니다.

### 구현 세부 사항

#### 프로젝트 구조

다음과 같은 프로젝트 구조를 생성했습니다:

```
CodexSdk/
├── CodexSdk.csproj
├── Codex.cs           # 입구 클래스
├── CodexThread.cs     # 대화 스레드
├── CodexExec.cs       # 실행기
├── Events.cs          # 이벤트 타입 정의
├── Items.cs           # 항목 타입 정의
├── EventParser.cs     # 이벤트 파서
├── OutputSchemaTempFile.cs  # 임시 파일 관리
└── ...
```

#### 사용 예시

기본 사용 방식은 TypeScript SDK와 일치하게 유지합니다:

```csharp
using CodexSdk;

var codex = new Codex();
var thread = codex.StartThread();

var result = await thread.RunAsync("Summarize this repository.");
Console.WriteLine(result.FinalResponse);
```

스트리밍 이벤트 처리는 C#의 패턴 매칭 능력을 활용합니다:

```csharp
await foreach (var @event in thread.RunStreamedAsync("Analyze the code."))
{
    switch (@event)
    {
        case ItemCompletedEvent itemCompleted
            when itemCompleted.Item is AgentMessageItem msg:
            Console.WriteLine($"Assistant: {msg.Text}");
            break;
        case TurnCompletedEvent completed:
            Console.WriteLine($"Tokens: in={completed.Usage.InputTokens}");
            break;
        case CommandExecutionItem command:
            Console.WriteLine($"Command: {command.Command}");
            break;
    }
}
```

### 주의 사항

구현 과정에서 다음과 같은 경험을 정리했습니다:

1. **프로세스 관리**: C# 버전은 프로세스의 수명 주기를 수동으로 관리해야 하며, 취소 시 프로세스 종료도 포함합니다. `Kill(entireProcessTree: true)`를 사용하여 하위 프로세스도 정리되도록 합니다.
2. **오류 처리**: `InvalidOperationException`을 사용하여 파싱 오류를 발생시켜 TypeScript SDK와 유사한 오류 처리 방식을 유지합니다.
3. **리소스 정리**: `OutputSchemaTempFile`이 `IAsyncDisposable`을 구현하여 임시 파일이 올바르게 정리되도록 합니다.
4. **환경 변수**: C# 버전은 `CodexOptions.Env`를 통해 프로세스 환경 변수를 완전히 덮어쓸는 것을 지원합니다.
5. **플랫폼 차이**: C# 버전은 TypeScript 버전에 있는 npm 패키지에서 바이너리를 자동 검색하는 로직을 포함하지 않습니다. 이는 .NET 프로젝트가 일반적으로 npm에 의존하지 않기 때문이며, `CODEX_EXECUTABLE` 환경 변수나 `CodexPathOverride`로 codex 실행 파일 경로를 지정해야 합니다.

## 요약

성숙한 TypeScript SDK를 C#으로 포팅하는 것은 단순한 구문 변환이 아니라 두 언어 설계 철학에 대한 이해가 필요합니다. TypeScript의 유연성과 JavaScript 생태계 특성(예: AbortSignal)을 C#에서 대응하는 방안을 찾아야 합니다.

핵심적인 깨달음: **API 일관성을 유지하는 것이 구현 세부 사항의 일관성을 유지하는 것보다 더 중요합니다**. 사용자가 관심하는 것은 인터페이스의 사용성이지, 내부 구현의 동일성이 아닙니다.

크로스 언어 포팅 작업을 하시는 경우, 저희 경험은 다음과 같습니다: 원 SDK의 아키텍처 설계를 완전히 이해한 후 모듈별로 변환하고, 마지막으로 완전한 테스트 케이스로 동작 일치성을 보장합니다. 서두르지 마세요, 한입에 모든 것을 이룰 수는 없습니다.
