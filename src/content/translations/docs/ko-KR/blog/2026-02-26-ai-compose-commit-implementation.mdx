## AI Compose Commit: AI 지능으로 Git 커밋 워크플로우를 스마트하게 재구성하기

## 소개

소프트웨어 개발 과정에서 코드를 커밋하는 것은 프로그래머가 매일 직면하는 일상적인 업무입니다. 하지만 이런 경험이 있으신가요: 하루 일과를 마친 후 Git을 열어보니 수십 개의 unstaged 변경 파일이 있는데, 이를 어떻게 합리적인 커밋으로 조직해야 할지 모르겠는 상황.

전통적인 방식은 파일을 수동으로 분배하여 staging하고, 하나씩 커밋하며, 커밋 메시지를 작성하는 것입니다. 이 과정은 시간이 많이 걸리고 오류가 발생하기 쉽습니다. 우리도 종종 이 작업에 많은 시간을 낭비했는데, 이미 피곤한 저녁에 이런 사소한 일로 골머리를 앓고 싶지는 않으니까요.

HagiCode 프로젝트에서 새 기능인 AI Compose Commit을 출시했습니다. 이 기능은 워크플로우를 완전히 바꾸는 것을 목표로 합니다. AI가 워크스페이스의 모든 uncommitted 변경을 지능적으로 분석하여 자동으로 여러 논리적 커밋으로 그룹화하고 규범에 맞는 커밋 작업을 수행합니다. 이 글에서는 이 기능의 구현 원리, 기술 아키텍처 및 실천 과정에서의 문제점과 해결 방안을 심층적으로 다룹니다.

## HagiCode에 대하여

## 배경

### 전통적 Git 커밋의 문제점

Git은 버전 관리 시스템으로 개발자에게 강력한 코드 관리 능력을 제공합니다. 하지만 실제 사용에서 커밋 작업은 종종 개발 프로세스의 병목이 됩니다:

1. **수동 그룹화에 시간 소요**: 대량의 파일 변경이 있을 때 개발자는 각 파일의 내용을 일일이 확인하여 동일 기능에 속하는지 판단해야 하므로 많은脑力이 소모됩니다
2. **커밋 메시지 품질 불균일**: Conventional Commits 규범에 맞는 커밋 메시지를 작성하려면 경험과 기술이 필요하며, 초보자는 종종 비표준 커밋을 작성합니다
3. **다중 저장소 관리 복잡**: monorepo 환경에서는 여러 저장소 간 전환해야 하므로 작업 복잡도가 증가합니다
4. **워크플로우 중단**: 코드를 커밋하면 개발 사고가 중단되고 코딩 효율에 영향을 미칩니다

이러한 문제는 대형 프로젝트와 팀 협업 환경에서 특히 두드러집니다. 우수한 개발 도구는 개발자가 핵심 코딩에 집중할 수 있게 해야 하며, 번거로운 커밋 프로세스에 방해받지 않아야 합니다.

### AI 보조 개발의 트렌드

최근 몇 년간 AI 기술은 소프트웨어 개발 분야에서 점점 더 널리 적용되고 있습니다. 코드 완성, 오류 감지에서 자동 문서 생성까지 AI는 점차 개발의 각 단계에 스며들고 있습니다. Git 워크플로우 측면에서도 일부 도구가 커밋 메시지 생성 기능을 제공하지만, 대부분 단일 커밋 시나리오에 국한되어 전체 워크스페이스 변경에 대한 지능적 분석 및 그룹화 능력이 부족합니다.

HagiCode도 개발 과정에서 이러한 문제점에 직면했으며, 여러 도구를 시도해 보았지만 어느 정도 한계가 있었습니다. 기능이 충분하지 않거나 사용자 경험이 좋지 않았습니다. 이것이 우리가 최종적으로 AI Compose Commit 기능을 직접 구현하기로 결정한 이유입니다.

HagiCode의 AI Compose Commit 기능은 이러한 공백을 메우기 위해 탄생했습니다. 단순히 커밋 메시지를 생성하는 것이 아니라 파일 분석부터 커밋 실행까지 전체 프로세스를 완전히 담당합니다.

## 문제

### 기술적 도전

AI Compose Commit 기능을 구현하는 과정에서 여러 기술적 도전에 직면했습니다:

1. **파일 의미 이해**: AI는 파일 변경의 의미 관계를 이해하고 어떤 파일이 동일 기능 모듈에 속하는지 판단해야 합니다. 파일 내용, 디렉토리 구조 및 변경의 컨텍스트를 심층 분석해야 합니다.

2. **커밋 그룹화 전략**: 합리적인 그룹화 기준을 어떻게 정의할까요? 기능별, 모듈별, 파일 유형별? 다른 프로젝트에는 다른 전략이 적용될 수 있습니다.

3. **실시간 피드백과 비동기 처리**: Git 작업은 특히 대량의 파일을 처리할 때 상당한 시간이 소요될 수 있습니다. 사용자 경험을 보장하면서 복잡한 작업을 완료하려면 어떻게 해야 할까요?

4. **다중 저장소 지원**: monorepo 아키텍처에서는 메인 저장소와 하위 저장소 간에 올바르게 작업을 라우팅해야 합니다.

5. **오류 처리와 롤백**: 특정 커밋이 실패하면 이미 실행된 커밋을 어떻게 처리할까요? 이미 staged된 파일을 롤백해야 할까요?

6. **커밋 메시지 일관성**: 생성된 커밋 메시지는 프로젝트의 기존 스타일에 맞아야 하며, 기존 커밋의 형식과 일치해야 합니다.

### 성능 고려

AI가 대량의 파일 변경을 처리하면 상당한 시간과 컴퓨팅 리소스가 소모됩니다. 다음 측면에서 최적화가 필요합니다:

- 불필요한 AI 호출 줄이기
- 파일 컨텍스트 구성 방식 최적화
- 효율적인 Git 작업 배치 처리 구현

이러한 문제는 HagiCode의 실제 사용에서 모두 실제로 발생했으며, 끊임없는 반복과 최적화를 통해 비교적 완벽한 해결 방안을 찾았습니다. 유사한 도구를 개발 중이시라면 우리의 경험이 영감을 주기를 바랍니다.

## 해결 방안

### 전체 아키텍처 설계

AI Compose Commit 기능을 구현하기 위해 계층화 아키텍처를 채택하여 시스템의 확장성과 유지보수성을 보장합니다:

#### 1. API 레이어 (Web 레이어)

**GitController**는 `POST /api/git/auto-compose-commit` 엔드포인트를 제공하여 기능의 진입점 역할을 합니다. 사용자 경험을 최적화하기 위해 Fire-and-Forget 비동기 모드를 채택했습니다:

- 클라이언트가 요청을 보내면 서버는 즉시 HTTP 202 Accepted를 반환합니다
- 실제 AI 처리는 백그라운드에서 비동기로 실행됩니다
- 처리 완료 후 SignalR을 통해 클라이언트에 알립니다

이 설계는 AI 처리에 몇 분이 걸리더라도 사용자가 즉시 응답을 받을 수 있어 시스템이 멈춘 것처럼 느끼지 않도록 보장합니다.

#### 2. 애플리케이션 서비스 레이어 (Application 레이어)

**GitAppService**는 핵심 비즈니스 로직을 담당합니다:

- 저장소 감지: monorepo에서의 다중 저장소 관리 지원
- 잠금 관리: 동시 작업으로 인한 충돌 방지
- 파일 staging 조정: AI 처리 프로세스와의 상호작용
- 오류 롤백: 실패 시나리오에서의 상태 복구
