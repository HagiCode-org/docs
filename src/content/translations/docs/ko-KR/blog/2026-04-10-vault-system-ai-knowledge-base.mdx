## AI 시대를 위한 크로스 프로젝트 지식 베이스 구축 - Vault 시스템

> 실제 프로젝트를 연구하고 재현하며 학습하는 것이 주류가 되고 있지만, 흩어진 학습 자료와 단절된 컨텍스트로 인해 AI 어시스턴트가 제 역할을 다하기 어렵습니다. 이 글에서는 HagiCode 프로젝트의 Vault 시스템 설계를 소개합니다. 통합된 저장소 추상화 레이어를 통해 AI 어시스턴트가 모든 학습 리소스를 이해하고 접근할 수 있게 하여, 진정한 크로스 프로젝트 지식 재사용을 실현합니다.

## 배경

사실 AI 시대에 새로운 기술을 배우는 방식은 조용히 변화하고 있습니다. 책을 읽고 영상을 보는 전통적인 방식도 여전히 중요하지만, "프로젝트를 연구하고 재현하는 것" - 우수한 오픈소스 프로젝트의 코드, 아키텍처, 디자인 패턴을 깊이 있게 연구하고 배우는 것 - 은 분명 더 효율적이 되고 있습니다. 고품질 오픈소스 프로젝트를 직접 실행하고 수정하는 것은 실제 엔지니어링 실무를 이해하는 가장 빠른 방법 중 하나입니다.

하지만 이 접근 방식도 새로운 과제를 가져옵니다.

**학습 자료가 너무 흩어져 있습니다.** 메모는 Obsidian에 있을 수 있고, 코드 저장소는 여러 폴더에 흩어져 있으며, AI 어시스턴트의 대화 기록은 별도의 데이터 섬이 됩니다. AI의 도움으로 프로젝트를 분석할 때마다 코드 스니펫을 수동으로 복사하고 컨텍스트를 정리해야 하며, 이는 꽤 번거로운 일입니다.

**컨텍스트가 계속 손실됩니다.** AI 어시스턴트는 로컬 학습 리소스에 직접 접근할 수 없으므로, 모든 대화마다 배경 정보를 다시 설명해야 합니다. 연구하는 코드 저장소는 빠르게 업데이트되고, 수동 동기화는 오류가 발생하기 쉽습니다. 더 나쁜 것은 여러 학습 프로젝트 간에 지식을 공유하기 어렵다는 것입니다. 프로젝트 A에서 배운 디자인 패턴은 AI가 프로젝트 B를 작업할 때 전혀 알지 못합니다.

핵심적으로 이러한 문제들은 모두 "데이터 섬"의 한 형태입니다. AI 어시스턴트가 모든 학습 리소스를 이해하고 접근할 수 있게 하는 통합된 저장소 추상화 레이어가 있다면 문제가 해결될 것입니다.

이러한 문제점들을 해결하기 위해 HagiCode를 개발하면서 핵심 설계 결정을 내렸습니다. Vault 시스템을 통합 지식 저장소 추상화 레이어로 구축합니다. 그 결정의 영향은 예상보다 클 수 있습니다. 곧 자세히 설명하겠습니다.

## HagiCode 소개

이 글에서 공유하는 접근 방식은 [HagiCode](https://hagicode.com) 프로젝트에서의 실무 경험에서 나온 것입니다. HagiCode는 OpenSpec 워크플로우 기반의 AI 코딩 어시스턴트입니다. 핵심 아이디어는 AI가 "말"할 수 있을 뿐만 아니라 "행동"할 수도 있어야 한다는 것입니다. 코드 저장소를 직접 조작하고, 명령어를 실행하고, 테스트를 실행하는 것입니다. GitHub: [github.com/HagiCode-org/site](https://github.com/HagiCode-org/site)

개발 과정에서 AI 어시스턴트가 사용자의 다양한 학습 리소스에 자주 접근해야 한다는 것을 발견했습니다. 코드 저장소, 메모, 설정 파일 등. 사용자가 매번 모든 것을 수동으로 제공해야 한다면 경험은 끔찍할 것입니다. 그래서 Vault 시스템을 설계하게 되었습니다.

## 핵심 설계

### 다중 유형 지원

HagiCode의 Vault 시스템은 네 가지 유형을 지원하며, 각각 다른 사용 시나리오에 해당합니다.

| 유형 | 용도 | 일반적인 시나리오 |
|------|------|----------|
| `folder` | 범용 폴더 유형 | 임시 학습 자료, 초안 |
| `coderef` | 코드 프로젝트 연구용으로 특별히 설계됨 | 오픈소스 프로젝트를 체계적으로 학습 |
| `obsidian` | Obsidian 메모장 소프트웨어와 통합 | 기존 메모 라이브러리 재사용 |
| `system-managed` | 시스템에 의해 자동 관리됨 | 프로젝트 설정, 프롬프트 템플릿 등 |

그 중 `coderef` 유형이 HagiCode에서 가장 자주 사용됩니다. 코드 학습 프로젝트를 위해 표준화된 디렉토리 구조와 AI가 읽을 수 있는 메타데이터 설명을 제공합니다. 왜 이 유형을 특별히 설계했을까요? 오픈소스 프로젝트를 연구하는 것이 "코드를 다운로드하는 것"만큼 간단하지 않기 때문입니다. 코드 자체, 학습 메모, 설정 파일 등의 콘텐츠를 동시에 관리해야 하며, `coderef`가 이 모든 것을 표준화합니다.

### 영구 저장 메커니즘

Vault 레지스트리는 파일 시스템에 JSON으로 저장됩니다.

```csharp
_registryFilePath = Path.Combine(absoluteDataDir, "personal-data", "vaults", "registry.json");
```

이 설계는 간단해 보이지만 신중하게 고려되었습니다.

**간단하고 신뢰할 수 있습니다.** JSON은 사람이 읽을 수 있어 디버깅과 수동 수정이 쉽습니다. 문제가 발생하면 파일을 직접 열어 상태를 검사하거나 수동으로 복구할 수 있습니다. 특히 개발 중에 유용합니다.

**종속성 감소.** 파일 시스템 저장은 데이터베이스의 복잡성을 피합니다. 추가적인 데이터베이스 서비스를 설치하고 구성할 필요가 없어 시스템 복잡성과 유지 관리 비용이 줄어듭니다.

**동시성 안전.** `SemaphoreSlim`을 사용하여 스레드 안전을 보장합니다. AI 코딩 어시스턴트 시나리오에서는 여러 작업이 동시에 Vault 레지스트리에 접근할 수 있으므로 동시성 제어가 필요합니다.

### AI 컨텍스트 통합

시스템의 핵심 기능은 Vault 정보를 AI 제안의 컨텍스트에 자동으로 주입할 수 있다는 것입니다.

```typescript
export function buildTargetVaultsText(
  vaults: VaultForText[],
  template: VaultPromptTemplate = DEFAULT_VAULT_PROMPT_TEMPLATE,
): string {
  const readOnlyVaults = vaults.filter((vault) => vault.accessType === 'read');
  const editableVaults = vaults.filter((vault) => vault.accessType === 'write');

  const sections = [
    buildVaultSection(readOnlyVaults, template.reference),
    buildVaultSection(editableVaults, template.editable),
  ].filter(Boolean);

  return `\n\n### ${template.heading}\n\n${sections.join('\n')}`;
}
```

이를 통해 AI 어시스턴트가 사용 가능한 학습 리소스를 자동으로 이해할 수 있으며, 사용자가 매번 컨텍스트를 수동으로 제공할 필요가 없습니다. HagiCode 경험을 특히 자연스럽게 만듭니다. AI에게 "React 동시성 렌더링을 분석해 줘"라고 말하면, 이전에 등록된 React 학습 Vault를 자동으로 찾아 코드를 반복해서 붙여넣으라고 요청하는 대신 처리합니다.

### 접근 제어 메커니즘

시스템은 Vault를 두 가지 접근 유형으로 나눕니다.

- **reference (읽기 전용)**: AI는 콘텐츠를 분석하고 이해하는 데만 사용할 수 있으며, 수정할 수 없습니다
- **editable (수정 가능)**: AI는 작업에 필요에 따라 콘텐츠를 수정할 수 있습니다

이 구분은 AI에게 어떤 콘텐츠가 "읽기 전용 참조"이고 어떤 콘텐츠를 수정할 수 있는지 알려주어, 실수로 변경할 위험을 줄입니다. 예를 들어, 오픈소스 프로젝트의 Vault를 학습 자료로 등록했다면 AI가 코드를 함부로 편집하기를 원하지 않을 것입니다. 따라서 `reference`로 표시합니다. 하지만 자체 프로젝트 Vault라면 `editable`로 표시하고 AI가 코드를 수정하도록 할 수 있습니다.

## 실용 가이드

### CodeRef Vault의 표준화된 구조

`coderef` Vault의 경우 시스템은 표준화된 디렉토리 구조를 제공합니다.

```
my-coderef-vault/
├── index.yaml          # Vault 메타데이터 설명
├── AGENTS.md           # AI 어시스턴트용 운영 가이드
├── docs/               # 학습 메모 및 문서 저장
└── repos/              # Git 서브모듈을 통해 참조된 코드 저장소 관리
```

이 구조 뒤의 설계 철학은 무엇일까요?

**docs/**는 학습 메모를 저장하며, Markdown을 사용하여 코드에 대한 이해, 아키텍처 분석, 디버깅 교훈을 기록합니다. 이 메모는 사용자만을 위한 것이 아닙니다. AI도 이해할 수 있으며, 관련 작업을 처리할 때 자동으로 참조합니다.

**repos/**는 코드를 직접 복사하는 대신 Git 서브모듈을 통해 연구하는 저장소를 관리합니다. 이에는 두 가지 이점이 있습니다. 첫째, 업스트림과 동기화 상태를 유지하며, `git submodule update` 한 번으로 최신 코드를 가져올 수 있습니다. 둘째, 공간을 절약합니다. 여러 Vault가 동일한 저장소의 다른 버전을 참조할 수 있기 때문입니다.

**index.yaml**에는 Vault 메타데이터가 포함되어 있어 AI 어시스턴트가 목적과 내용을 빠르게 이해할 수 있습니다. 본질적으로 Vault의 "자기소개"이며, AI가 처음 보는 순간 무엇을 위한 것인지 알 수 있습니다.

**AGENTS.md**는 AI 어시스턴트를 위해 특별히 작성된 가이드로, Vault 내부의 콘텐츠를 처리하는 방법을 설명합니다. "이 프로젝트를 분석할 때 성능 최적화와 관련된 코드에 집중해 주세요" 또는 "테스트 파일을 수정하지 마세요"와 같은 지시를 AI에게 줄 수 있습니다.

### Vault 생성 및 사용

CodeRef Vault를 생성하는 것은 간단합니다.

```typescript
const createCodeRefVault = async () => {
  const response = await VaultService.postApiVaults({
    requestBody: {
      name: "React Learning Vault",
      type: "coderef",
      physicalPath: "/Users/developer/vaults/react-learning",
      gitUrl: "https://github.com/facebook/react.git"
    }
  });

  // 시스템이 자동으로 다음을 수행합니다:
  // 1. React 저장소를 vault/repos/react에 클론
  // 2. 메모용 docs/ 디렉토리 생성
  // 3. index.yaml 메타데이터 생성
  // 4. AGENTS.md 가이드 파일 생성

  return response;
};
```

그런 다음 AI 제안에서 이 Vault를 참조합니다.

```typescript
const proposal = composeProposalChiefComplaint({
  chiefComplaint: "React의 동시성 렌더링 메커니즘을 분석해 줘",
  repositories: [
    { id: "react", gitUrl: "https://github.com/facebook/react.git" }
  ],
  vaults: [
    {
      id: "react-learning",
      name: "React Learning Vault",
      type: "coderef",
      physicalPath: "/vaults/react-learning",
      accessType: "read"  // AI는 읽기만 가능, 수정 불가
    }
  ],
  quickRequestText: "Fiber 아키텍처와 스케줄러 구현에 특별히 주의해 주세요"
});
```

### 일반적인 사용 시나리오

**시나리오 1: 오픈소스 프로젝트를 체계적으로 연구**

CodeRef Vault를 생성하고, Git 서브모듈을 통해 대상 저장소를 관리하며, `docs/` 디렉토리에 학습 메모를 기록합니다. AI는 코드와 메모를 동시에 접근하여 더 정확한 분석을 제공할 수 있습니다. 모듈을 학습하면서 작성한 메모는 나중에 AI가 관련 코드를 분석할 때 자동으로 참조됩니다. 이전의 생각을 기억하는 "조수"를 두는 것과 같습니다.

**시나리오 2: Obsidian 메모 라이브러리 재사용**

Obsidian을 사용하여 메모를 관리하고 있다면, 기존 Vault를 HagiCode에 직접 등록하기만 하면 됩니다. AI가 수동 복사-붙여넣기 없이 지식 베이스에 접근할 수 있습니다. 이 기능은 특히 실용적입니다. 많은 사람이 수년간 축적한 메모를 가지고 있으며, 연결되면 AI가 그 지식 체계를 "읽고" 이해할 수 있습니다.

**시나리오 3: 크로스 프로젝트 지식 재사용**

여러 AI 제안이 동일한 Vault를 참조할 수 있어 프로젝트 간 지식 재사용이 가능합니다. 예를 들어, 많은 디자인 패턴의 메모와 코드 예시를 포함하는 "디자인 패턴 학습 Vault"를 만들 수 있습니다. AI가 어느 프로젝트를 분석하든 해당 Vault의 콘텐츠를 참조할 수 있습니다. 지식을 반복해서 축적할 필요가 없습니다.

### 경로 안전 메커니즘

시스템은 경로를 엄격하게 검증하여 경로 순회 공격을 방지합니다.

```csharp
private static string ResolveFilePath(string vaultRoot, string relativePath)
{
    var rootPath = EnsureTrailingSeparator(Path.GetFullPath(vaultRoot));
    var combinedPath = Path.GetFullPath(Path.Combine(rootPath, relativePath));
    if (!combinedPath.StartsWith(rootPath, StringComparison.OrdinalIgnoreCase))
    {
        throw new BusinessException(VaultRelativePathTraversalCode,
            "Vault file paths must stay inside the registered vault root.");
    }
    return combinedPath;
}
```

이를 통해 모든 파일 작업이 Vault 루트 디렉토리 내에 유지되고 악의적인 경로 접근을 방지합니다. 보안은 가볍게 여겨서는 안 됩니다. AI 어시스턴트가 파일 시스템을 조작한다면, 경계를 명확하게 정의해야 합니다.

## 주의 사항

HagiCode Vault 시스템을 사용할 때 몇 가지에 특별히 주의해야 합니다.

1. **경로 안전**: 사용자 정의 경로가 허용된 범위 내에 있는지 확인하세요. 그렇지 않으면 시스템이 작업을 거부합니다. 이는 의도치 않은 오용과 잠재적인 보안 위험을 방지합니다.

2. **Git 서브모듈 관리**: CodeRef Vault는 코드를 직접 복사하는 대신 Git 서브모듈로 관리하는 것이 가장 좋습니다. 이점은 앞서 설명했습니다. 동기화와 공간 절약입니다. 하지만 서브모듈에는 자체 워크플로우가 있으므로 처음 사용하는 사용자는 익숙해지는 데 약간의 시간이 필요할 수 있습니다.

3. **파일 미리보기 제한**: 시스템은 파일 크기(256KB)와 수량(500개 파일)을 제한하므로, 초대형 파일은 배치로 처리해야 합니다. 이 제한은 성능을 위한 것입니다. 매우 큰 파일을 만나면 수동으로 분할하거나 다른 방식으로 처리할 수 있습니다.

4. **진단 정보**: Vault를 생성하면 실패 시 디버깅에 사용할 수 있는 진단 정보가 반환됩니다. 문제가 발생하면 먼저 진단 정보를 확인하세요. 대부분의 경우 거기서 단서를 찾을 수 있습니다.

## 결론

HagiCode Vault 시스템은 근본적으로 간단하지만 심오한 문제를 해결합니다. AI 어시스턴트가 로컬 지식 리소스를 이해하고 사용할 수 있도록 하는 것입니다.

통합된 저장소 추상화 레이어, 표준화된 디렉토리 구조, 자동화된 컨텍스트 주입을 통해 "한 번 등록하면 어디서나 재사용"하는 지식 관리 모델을 제공합니다. Vault가 생성되면 AI가 학습 메모, 코드 저장소, 문서 리소스에 자동으로 접근하고 이해할 수 있습니다.

이 설계의 경험 향상은 분명합니다. 더 이상 코드 스니펫을 수동으로 복사하거나 배경 정보를 반복해서 설명할 필요가 없습니다. AI 어시스턴트는 프로젝트를 진정으로 이해하는 팀원처럼 되며, 기존 지식을 기반으로 더 가치 있는 도움을 제공할 수 있습니다.

이 글에서 공유한 Vault 시스템은 HagiCode 개발 중 실제 시행착오와 최적화를 통해 만들어진 해결책입니다. 이 설계가 가치 있다고 생각하신다면, 그 뒤의 엔지니어링도 주목할 만한 것이며, HagiCode 자체도 확인해 볼 가치가 있습니다.
