## Skillsbase로 자체 Skills 컬렉션 저장소를 유지 관리해야 하는 이유

> 생각해보면 좀 재미있습니다. AI 프로그래밍 시대가 도래했고, 우리가 보유한 Agent Skills가 점점 많아지고 있습니다. 하지만 그에 따라 번거로운 일도 늘어나고 있습니다. 이 글에서는 우리가 skillsbase를 사용하여 이러한 문제들을 해결한 방법을 소개합니다.

## 배경

AI 프로그래밍 시대에 개발자들은 점점 더 많은 Agent Skills를 유지 관리해야 합니다. Agent Skills는 Claude Code, OpenCode, Cursor와 같은 코딩 어시스턴트의 기능을 확장하는 재사용 가능한 명령어 세트입니다. 하지만 Skills의 수가 증가함에 따라 실용적인 문제가 점진적으로 대두됩니다.

별로 큰 문제는 아니지만, 너무 많은 것들이 쌓이면 관리가 번거로워집니다.

### Skills가 여러 위치에 흩어져 있어 관리 비용이 높습니다

- 로컬 Skills가 여러 위치에 흩어져 있습니다: `~/.agents/skills/`, `~/.claude/skills/`, `~/.codex/skills/.system/` 등
- 다른 위치에 이름 충돌이 발생할 수 있습니다. 예를 들어 `skill-creator`가 사용자 디렉토리와 시스템 디렉토리에 모두 존재할 수 있습니다
- 통합된 관리 진입점이 없어 백업과 마이그레이션이 어렵습니다

이 부분은 정말 성가십니다. 때로는 특정 Skill이 실제로 어디에 있는지조차 모를 때가 있습니다. 무언가를 잃어버리고 그것을 찾기 위해 헤매는 것과 같습니다.

### 표준화된 유지 관리 워크플로우의 부재

- 수동으로 Skills를 복사하면 오류가 발생하기 쉽고 출처를 추적하기 어렵습니다
- 통합된 검증 메커니즘이 없어 Skill 저장소의 완전성을 보장할 수 없습니다
- 팀 협업 시 Skill 컬렉션의 동기화와 공유가 어렵습니다

수동 작업은 항상 실수가 발생하기 쉽습니다. 사람의 기억력에는 한계가 있으니까요. 모든 것이 어디서 왔는지 기억할 수 있는 사람이 누가 있겠습니까?

### 재현성 요구 사항을 충족하지 못합니다

- 개발 머신을 전환할 때 모든 Skills를 다시 구성해야 합니다
- CI/CD 환경에서 Skill 저장소를 자동으로 검증하고 동기화할 수 없습니다

다른 컴퓨터로 바꾸면 모든 것을 처음부터 다시 해야 합니다. 어찌 보면 이사하는 것과 같습니다. 매번 번거롭습니다. 새로운 환경에 적응하고 모든 것을 다시 구성해야 하니까요.

이러한 문제들을 해결하기 위해 우리는 여러 가지 접근 방식을 시도했습니다. 수동 복사에서 스크립트 자동화까지, 디렉토리 직접 관리에서 전역 설치 후 파일 복구까지. 각 접근 방식마다 고유한 단점이 있었습니다. 일부는 일관성을 보장하지 못했고, 일부는 환경을 오염시켰으며, 일부는 CI에서 사용하기 어려웠습니다.

꽤 많은 우회로를 거쳤습니다.

결국 우리는 더 우아한 해결책을 찾았습니다. 바로 skillsbase입니다. 이 접근 방식의 핵심 아이디어는 로컬에 먼저 설치하고 검증한 다음, 구조를 변환하여 저장소에 기록하고, 마지막으로 임시 파일을 제거하는 것입니다. 이를 통해 저장소 내용이 실제 설치 결과와 일치하도록 보장하면서 전역 환경의 오염을 방지합니다.

말로 하면 간단해 보이지만, 우리는 꽤 많은 시행착오를 겪은 후에야 이 방법을 알아냈습니다.

## HagiCode 소개

이 글에서 공유하는 해결책은 [HagiCode](https://hagicode.com) 프로젝트에서의 실무 경험에서 나온 것입니다.

HagiCode는 AI 코딩 어시스턴트 프로젝트입니다. 개발 과정에서 다양한 코딩 기능을 확장하기 위해 많은 수의 Agent Skills를 유지 관리해야 했습니다. 이러한 실제 요구 사항이 바로 우리가 Skill 저장소의 표준화된 관리를 위한 skillsbase 도구 모음을 구축하게 된 계기입니다.

이것은 무에서 유를 창조한 것이 아닙니다. 실제 요구 사항에 의해 만들어진 것입니다. Skills의 수가 증가하면 관리가 필요해지고, 관리 과정에서 문제가 발생하면 해결책도 필요해집니다. 단계별로 이렇게 여기까지 온 것입니다.

HagiCode에 관심이 있으시다면 [공식 웹사이트](https://hagicode.com)를 방문하여 자세한 정보를 확인하거나 [GitHub](https://github.com/HagiCode-org/site)에서 소스 코드를 살펴볼 수 있습니다.

## 분석

### 기술적 과제

유지 관리 가능한 Skills 컬렉션 저장소를 구축하려면 다음 핵심 문제들을 해결해야 합니다.

1. **통합된 네임스페이스 충돌**: 여러 소스에 동일한 이름의 Skills가 포함된 경우, 어떻게 덮어쓰기를 방지합니까?
2. **출처 추적 가능성**: 각 Skill의 출처를 어떻게 기록하여 향후 업데이트와 감사에 활용합니까?
3. **동기화 및 검증**: 저장소 내용이 실제 설치 결과와 일치하도록 어떻게 보장합니까?
4. **자동화 통합**: CI/CD 워크플로우와 통합하여 자동 동기화 및 검증을 어떻게 활성화합니까?

이 문제들은 간단해 보이지만, 각각 하나하나가 두통거리입니다. 하지만 가치 있는 일 중 쉬운 것이 어디 있겠습니까?

### 설계 트레이드오프

**옵션 1: 디렉토리를 직접 복사**

장점: 구현이 간단합니다
단점: `skills` CLI의 실제 설치 결과와의 일관성을 보장할 수 없습니다

이 방법을 생각해 본 적이 있습니다. 하지만 나중에 CLI가 설치 과정에서 일부 전처리 로직을 적용할 수 있다는 것을 깨달았습니다. 직접 복사하면 이 단계가 건너뛰게 됩니다. 결과적으로 복사한 것과 실제 설치된 것이 다르게 되고, 이것이 문제가 됩니다.

**옵션 2: 전역 설치 후 복구**

장점: 설치 과정을 검증할 수 있습니다
단점: 실행 환경을 오염시키고, CI와 로컬 결과의 일관성을 유지하기 어렵습니다

이 방법은 더 나쁩니다. 전역 설치는 환경을 오염시킵니다. 더 중요한 것은 CI 환경과 로컬 환경의 일관성을 유지하기 어려워서 "내 컴퓨터에서는 되는데 CI에서는 안 된다"는 전형적인 문제가 발생한다는 것입니다. 이 문제를 경험해 본 사람은 그 고통을 잘 알 것입니다.

**옵션 3: 로컬 설치 -> 변환 -> 제거 (최종 해결책)**

이것이 skillsbase가 채택한 접근 방식입니다.

- `npx skills`를 사용하여 Skills를 임시 위치에 먼저 설치합니다
- 디렉토리 구조를 변환하고 소스 메타데이터를 추가합니다
- 결과를 대상 저장소에 기록합니다
- 마지막으로 임시 파일을 제거합니다

이 접근 방식은 저장소 내용이 소비자가 보는 실제 설치 결과와 일치하도록 보장하고, 전역 환경의 오염을 방지하며, 변환 프로세스를 표준화하고, 멱등 연산을 지원합니다.

이 해결책도 처음부터 명확했던 것은 아닙니다. 우리는 충분한 시행착오를 통해 무엇이 효과가 있고 무엇이 효과가 없는지 배웠습니다.

### 아키텍처 결정

| 결정 항목 | 선택 | 이유 |
|--------|------|------|
| 런타임 | Node.js ESM | 빌드 단계가 불필요하며, `.mjs`만으로 파일 시스템 오케스트레이션이 가능합니다 |
| 설정 형식 | YAML (`sources.yaml`) | 가독성이 높고 수동 유지 관리에 적합합니다 |
| 명명 전략 | 네임스페이스 접두사 | 사용자 Skills는 원래 이름을 유지하고, 시스템 Skills는 `system-` 접두사를 받습니다 |
| 워크플로우 | `add`가 매니페스트를 업데이트 -> `sync`가 동기화를 실행 | 단일 동기화 엔진으로 동일한 규칙을 두 번 구현할 필요가 없습니다 |
| 파일 관리 | 관리 파일 마커 | 안전한 덮어쓰기를 지원하기 위해 주석 헤더를 추가합니다 |

이러한 결정은 모두 하나의 목표로 귀결됩니다. 단순하게 만드는 것입니다. 결국 단순함이 승리합니다.

## 해결책

### CLI 아키텍처

`skillsbase` CLI는 네 가지 핵심 명령어를 제공합니다.

```text
skillsbase
├── init          # 저장소 구조 초기화
├── sync          # Skill 내용 동기화
├── add           # 새 Skill 추가
└── github_action # GitHub Actions 설정 생성
```

명령어가 많지 않지만 충분합니다. 도구는 유용하기만 하면 됩니다.

### 핵심 워크플로우

```
┌─────────────┐    ┌─────────────┐    ┌─────────────┐    ┌─────────────┐
│    init     │───▶│    add      │───▶│    sync     │───▶│github_action│
│ initialize  │    │ add source  │    │ sync content│    │ generate CI │
└─────────────┘    └─────────────┘    └─────────────┘    └─────────────┘
```

한 단계씩 진행하면 됩니다. 서두를 필요가 없습니다.

### 동기화 흐름 설계

```
sources.yaml -> parse sources -> npx skills install -> convert structure -> write to skills/ -> uninstall temporary files
                              ↓
                        .skill-source.json (source metadata)
```

이 워크플로우는 꽤 명확합니다. 적어도 제가 보기에는 각 단계가 무엇을 하는지 이해할 수 있습니다.

### 저장소 구조

```
repos/skillsbase/
├── sources.yaml              # Source manifest (단일 진실 공급원)
├── skills/                   # Skills 디렉토리
│   ├── frontend-design/      # 사용자 Skill
│   ├── skill-creator/        # 사용자 Skill
│   └── system-skill-creator/ # 시스템 Skill (접두사 포함)
├── scripts/
│   ├── sync-skills.mjs       # 동기화 스크립트
│   └── validate-skills.mjs   # 검증 스크립트
├── docs/
│   └── maintainer-workflow.md # 관리자 문서
└── .github/
    ├── workflows/
    │   └── skills-sync.yml   # CI 워크플로우
    └── actions/
        └── skillsbase-sync/
            └── action.yml     # 재사용 가능한 Action
```

파일이 꽤 많지만 괜찮습니다. 구조가 명확하게 정리되면 유지 관리가 훨씬 쉬워집니다.

## 실습

### Skills 저장소 초기화

```bash
# 1. 빈 저장소 생성
mkdir repos/myskills && cd repos/myskills
git init

# 2. skillsbase로 초기화
npx skillsbase init

# 출력:
# [1/4] create manifest ................. done
# [2/4] create scripts .................. done
# [3/4] create docs ..................... done
# [4/4] create github workflow .......... done
#
# next: skillsbase add <skill-name>
```

이 단계에서 많은 파일이 생성되지만 걱정할 필요가 없습니다. 모두 자동으로 생성됩니다. 그 후에는 Skills를 추가하기 시작하면 됩니다.

### Skills 추가

```bash
# 단일 Skill 추가 (자동으로 동기화가 트리거됩니다)
npx skillsbase add frontend-design --source vercel-labs/agent-skills

# 로컬 소스에서 추가
npx skillsbase add documentation-writer --source /home/user/.agents/skills

# 출력:
# source: first-party ......... updated
# target: skills/frontend-design ... synced
# status: 1 skill added, 0 removed
```

Skill을 추가하는 것은 매우 간단합니다. 하나의 명령어로 충분합니다. 하지만 때로는 네트워크 상태 불량이나 권한 문제와 같은 예상치 못한 문제에 부딪힐 수 있습니다. 이는 관리 가능합니다. 하나씩 해결하면 됩니다.

### Skills 동기화

```bash
# 동기화 수행 (모든 소스 조정)
npx skillsbase sync

# 드리프트만 확인 (파일을 수정하지 않음)
npx skillsbase sync --check

# 누락된 소스 허용 (CI 시나리오)
npx skillsbase sync --allow-missing-sources
```

동기화 중에 시스템은 `sources.yaml`에 정의된 모든 소스를 확인하고 `skills/` 디렉토리의 내용과 조정합니다. 차이가 있으면 업데이트하고, 차이가 없으면 건너뜁니다. 이렇게 하면 "설정은 변경되었는데 파일은 변경되지 않은" 문제를 방지할 수 있습니다.

### CI 설정 생성

```bash
# 워크플로우 생성
npx skillsbase github_action --kind workflow

# Action 생성
npx skillsbase github_action --kind action

# 모든 것 생성
npx skillsbase github_action --kind all
```

CI 설정도 자동으로 생성됩니다. 트리거 조건이나 런타임 환경과 같은 일부 세부 사항은 직접 조정해야 하지만, 그것은 어렵지 않습니다.

### `sources.yaml` 설정 예시

```yaml
# Skills 루트 디렉토리 설정
skillsRoot: skills/
metadataFile: .skill-source.json

# 소스 정의
sources:
  # First-party: 로컬 사용자 Skills
  first-party:
    type: local
    path: /home/user/.agents/skills
    naming: original  # 원래 이름 유지
    includes:
      - documentation-writer
      - frontend-design
      - skill-creator

  # System: 시스템에서 제공하는 Skills
  system:
    type: local
    path: /home/user/.codex/skills/.system
    naming: prefix-system  # system- 접두사 추가
    includes:
      - imagegen
      - openai-docs
      - skill-creator  # system-skill-creator가 됩니다

  # Remote: 서드파티 저장소
  vercel:
    type: remote
    url: vercel-labs/agent-skills
    naming: original
    includes:
      - web-design-guidelines
```

이 설정 파일이 전체 시스템의 핵심입니다. 모든 소스가 여기에 정의됩니다. 이 파일을 변경하면 다음 동기화 시 새로운 상태가 적용됩니다. 그 의미에서 이것은 진정한 "단일 진실 공급원"입니다.

### `.skill-source.json` 메타데이터 예시

```json
{
  "source": "first-party",
  "originalPath": "/home/user/.agents/skills/documentation-writer",
  "originalName": "documentation-writer",
  "targetName": "documentation-writer",
  "syncedAt": "2026-04-07T00:00:00.000Z",
  "version": "unknown"
}
```

모든 Skill 디렉토리에는 이 파일이 포함되어 있으며, 소스 정보를 기록합니다. 그러면 나중에 문제가 발생했을 때 어디서 왔는지, 언제 동기화되었는지 빠르게 파악할 수 있습니다.

### 보안 및 검증

```bash
# 저장소 구조 검증
node scripts/validate-skills.mjs

# skills CLI로 검증
npx skills add . --list

# 업데이트 확인
npx skills check
```

검증은 중요하면서도 선택 사항처럼 느껴질 수 있습니다. 하지만 안전을 위해 가끔 실행하는 것은 나쁘지 않습니다. 언제 예상치 못한 일이 발생할지 모르니까요.

### GitHub Actions 통합

```yaml
# .github/workflows/skills-sync.yml
name: Skills Sync

on:
  push:
    paths:
      - 'sources.yaml'
      - 'skills/**'
  workflow_dispatch:

jobs:
  validate:
    runs-on: ubuntu-latest
    steps:
      - uses: actions/checkout@v4
      - uses: actions/setup-node@v4
        with:
          node-version: '20'
      - name: Validate repository
        run: |
          npx skills add . --list
          node scripts/validate-skills.mjs
      - name: Sync check
        run: npx skillsbase sync --check
```

CI 통합이 완료되면 `sources.yaml`이나 `skills/` 디렉토리에 대한 모든 변경이 자동으로 검증을 트리거합니다. 이렇게 하면 로컬에서 변경을 했지만 동기화를 잊어버리는 상황을 방지할 수 있습니다.

### 모범 사례

1. **이름 충돌 처리**: 시스템 Skills에 일관되게 `system-` 접두사를 추가합니다. 이렇게 하면 모든 Skill을 사용 가능하게 유지하면서 이름 충돌을 피할 수 있습니다.
2. **멱등 연산**: 모든 명령어는 반복 실행을 지원하며, `sync`를 여러 번 실행해도 부작용이 없습니다. 이것은 CI에서 특히 중요합니다.
3. **관리 파일**: 생성된 파일에는 `# Managed by skillsbase CLI` 주석이 포함되어 있어 식별和管理가 쉽습니다. 이러한 파일은 안전하게 덮어쓸 수 있으며, 수동 수정은 보존되지 않습니다.
4. **비대화형 모드**: CI 환경에서는 기본적으로 결정론적 동작을 사용하므로 대화형 프롬프트가 실행을 중단하지 않습니다. 모든 설정은 `sources.yaml`을 통해 선언됩니다.
5. **소스 추적 가능성**: 모든 Skill에는 소스 정보를 기록하는 `.skill-source.json` 파일이 있어 문제 해결이 훨씬 빨라집니다.

### 팀 협업

```bash
# 팀원이 공유 Skills 저장소를 설치합니다
npx skills add your-org/myskills -g --all

# 로컬에 클론하고 검증합니다
git clone https://github.com/your-org/myskills.git
cd myskills
npx skills add . --list
```

Git으로 Skills 저장소를 관리하면 팀원들이 Skill 컬렉션을 쉽게 동기화하고, 모두가 동일한 버전의 도구와 설정을 사용하도록 보장할 수 있습니다.

이것은 팀 협업에서 특히 유용합니다. 더 이상 "나에게는 되는데 너에게는 안 돼"라는 상황을 겪지 않게 됩니다. 환경이 통일되면 문제의 절반이 사라집니다.

## 결론

`skillsbase`를 사용하여 Skills 컬렉션 저장소를 유지 관리하는 핵심 가치는 다음과 같습니다.

- **보안**: 소스 검증, 충돌 감지, 관리 파일 보호
- **유지 관리성**: 통합 진입점, 멱등 연산, 설정 증 문서화
- **표준화**: 통합된 디렉토리 구조, 명명 규칙, 메타데이터 형식
- **자동화**: CI/CD 통합, 자동 동기화, 자동 검증

이 접근 방식을 통해 개발자들은 자신의 Agent Skills를 npm 패키지를 관리하는 것과 동일한 방식으로 관리할 수 있으며, 재현 가능하고 공유 가능하며 유지 관리 가능한 Skills 저장소 시스템을 구축할 수 있습니다.

이 글에서 공유한 도구와 워크플로우는 HagiCode를 구축하는 과정에서 실제 실수와 최적화를 통해 다듬어진 것입니다. 이 접근 방식이 가치 있다고 생각하신다면, 우리의 엔지니어링 방향이 올바르다는 좋은 신호이며, HagiCode 자체도 주목할 가치가 있습니다.

결국 좋은 도구는 더 많은 사람들이 사용할 가치가 있습니다.
