## GitHub Actions로 Steam 릴리스 자동화하기

> HagiCode Desktop 프로젝트에서 Steam 자동 릴리스를 위해 구현한 완전한 해결책을 공유합니다. GitHub Release부터 Steam 플랫폼까지의 엔드투엔드 자동화 흐름을 다루며, Steam Guard 인증 및 다중 플랫폼 Depot 업로드와 같은 핵심 기술 세부 사항을 포함합니다.

## 배경

Steam의 릴리스 워크플로우는 실제로 전통적인 애플리케이션 배포와 상당히 다릅니다. Steam에는 자체적인 완전한 업데이트 전송 시스템이 있습니다. 개발자는 다른 플랫폼처럼 다운로드 링크만 걸어두는 것이 아니라 SteamCMD를 사용하여 빌드 산출물을 Steam의 CDN 네트워크에 업로드해야 합니다.

HagiCode Desktop 프로젝트는 Steam 출시를 준비 중이며, 이것은 우리의 릴리스 워크플로우에 몇 가지 새로운 과제를 도입했습니다.

1. 기존 빌드 산출물을 Steam 호환 형식으로 변환해야 했습니다.
2. SteamCMD를 통해 Steam에 업로드해야 했습니다.
3. Steam Guard 인증도 처리해야 했습니다.
4. Linux, Windows, macOS를 위한 다중 플랫폼 Depot 업로드를 지원해야 했습니다.
5. GitHub Release부터 Steam까지의 완전 자동화 흐름을 원했습니다.

프로젝트는 이미 "portable version mode"를 구현했으며, 이를 통해 애플리케이션이 `extra` 디렉토리에 패키징된 고정 서비스 페이로드를 감지할 수 있습니다. 우리의 목표는 이 portable version mode를 Steam 배포와 매끄럽게 통합하는 것이었습니다.

## HagiCode 소개

이 글에서 공유하는 해결책은 [HagiCode](https://hagicode.com) 프로젝트에서의 실무 경험에서 나온 것입니다. HagiCode는 데스크톱 사용을 지원하는 AI 코딩 어시스턴트입니다. Steam 출시를 향해 적극적으로 노력하고 있으므로, 신뢰할 수 있는 자동 릴리스 워크플로우를 확립해야 했습니다.

## 아키텍처 설계

전체 Steam 릴리스 프로세스의 핵심은 GitHub Actions 워크플로우이며, 프로세스를 세 가지 주요 단계로 나눕니다.

```
┌─────────────────────────────────────────────────────────────┐
│ GitHub Actions 워크플로우 (Steam 릴리스)                    │
├─────────────────────────────────────────────────────────────┤
│ 1. 준비 단계:                                              │
│    - portable-version 코드 체크아웃                        │
│    - GitHub Release에서 빌드 산출물 다운로드               │
│    - Steam 콘텐츠 디렉토리 추출 및 준비                    │
│                                                             │
│ 2. SteamCMD 설정:                                          │
│    - SteamCMD 설치 또는 재사용                             │
│    - Steam Guard 인증                                      │
│                                                             │
│ 3. 릴리스 단계:                                             │
│    - Depot VDF 구성 파일 생성                              │
│    - App Build VDF 구성 파일 생성                          │
│    - SteamCMD 호출하여 Steam에 업로드                     │
└─────────────────────────────────────────────────────────────┘
```

이 설계는 몇 가지 장점이 있습니다.
- 기존 GitHub Release 산출물을 재사용하고 동일한 출력을 재빌드하는 것을 피합니다.
- self-hosted runner를 사용하여 안전한 격리를 보장합니다.
- 프리뷰 모드와 정식 릴리스 브랜치 간의 전환을 지원합니다.
- 완전한 오류 처리와 로깅이 포함되어 있어 실패를 더 쉽게 진단할 수 있습니다.

## 워크플로우 구현

### 트리거 매개변수 설계

우리의 워크플로우는 다음 핵심 매개변수를 지원합니다.

```yaml
inputs:
  release:           # Portable Version 릴리스 태그
    description: '릴리스할 버전 태그 (예: v1.0.0)'
    required: true
  steam_preview:     # 프리뷰 빌드를 생성할지 여부
    description: '프리뷰 모드를 사용할지 여부'
    required: false
    default: 'false'
  steam_branch:      # Steam 브랜치를 live로 설정
    description: '대상 Steam 브랜치'
    required: false
    default: 'preview'
  steam_description: # 빌드 설명 오버라이드
    description: '빌드 설명'
    required: false
```

### Self-hosted Runner 구성

보안을 위해 `steam` 레이블이 있는 self-hosted runner를 사용합니다.

```yaml
runs-on:
  - self-hosted
  - Linux
  - X64
  - steam
```

이렇게 하면 Steam 릴리스가 전용 runner에서 실행되며, 민감한 자격 증명이 안전하게 격리됩니다.

### 동시성 제어

동일한 버전의 릴리스가 서로 간섭하지 않도록 동시성 제어를 구성했습니다.

```yaml
concurrency:
  group: portable-version-steam-${{ github.event.inputs.release }}
  cancel-in-progress: false
```

여기서 `cancel-in-progress: false`로 설정된 것에 주의하세요. Steam 릴리스는 시간이 오래 걸릴 수 있으며, 새로 트리거된 실행이 이미 업로드 중인 실행을 취소하길 원하지 않기 때문입니다.

## 핵심 스크립트 구현

### 릴리스 입력 준비

`prepare-steam-release-input.mjs` 스크립트는 릴리스에 필요한 입력을 준비합니다.

```javascript
// GitHub Release 빌드 매니페스트 및 산출물 인벤토리 다운로드
const buildManifest = await downloadBuildManifest(releaseTag);
const artifactInventory = await downloadArtifactInventory(releaseTag);

// 각 플랫폼의 압축 아카이브 다운로드
for (const platform of ['linux-x64', 'win-x64', 'osx-universal']) {
  const artifactUrl = getArtifactUrl(artifactInventory, platform);
  await downloadArtifact(artifactUrl, platform);
}

// Steam 콘텐츠 디렉토리 구조로 추출
await extractToSteamContent(sources, contentRoot);
```

### Steam Guard 인증

Steam은 계정이 Steam Guard로 보호되어야 하므로, 공유 시크릿 기반 코드 생성 알고리즘을 구현했습니다.

```javascript
function generateSteamGuardCode(sharedSecret, timestamp = Date.now()) {
  const secret = decodeSharedSecret(sharedSecret);
  const time = Math.floor(timestamp / 1000 / 30);

  const timeBuffer = Buffer.alloc(8);
  timeBuffer.writeBigUInt64BE(BigInt(time));

  // HMAC-SHA1을 사용하여 시간 기반 일회용 코드 생성
  const hash = crypto.createHmac('sha1', secret)
    .update(timeBuffer)
    .digest();

  // 5자리 Steam Guard 코드로 변환
  const code = steamGuardCode(hash);
  return code;
}
```

이 구현은 Steam Guard의 TOTP (시간 기반 일회용 비밀번호) 메커니즘을 기반으로 하며, 30초마다 새로운 인증 코드를 생성합니다.

### VDF 구성 생성

VDF (Valve Data Format)는 Steam에서 사용하는 구성 형식입니다. 두 가지 유형의 VDF 파일을 생성해야 합니다.

**Depot VDF**는 각 플랫폼의 콘텐츠를 구성하는 데 사용됩니다.

```javascript
function buildDepotVdf(depotId, contentRoot) {
  return [
    '"DepotBuildConfig"',
    '{',
    `  "DepotID" "${escapeVdf(depotId)}"`,
    `  "ContentRoot" "${escapeVdf(contentRoot)}"`,
    '  "FileMapping"',
    '  {',
    '    "LocalPath" "*"',
    '    "DepotPath" "."',
    '    "recursive" "1"',
    '  }',
    '}'
  ].join('\n');
}
```

**App Build VDF**는 전체 애플리케이션 빌드를 구성하는 데 사용됩니다.

```javascript
function buildAppBuildVdf(appId, depotBuilds, description, setLive) {
  const vdf = [
    '"appbuild"',
    '{',
    `  "appid" "${appId}"`,
    `  "desc" "${escapeVdf(description)}"`,
    `  "contentroot" "${escapeVdf(contentRoot)}"`,
    '  "buildoutput" "build_output"',
    '  "depots"',
    '  {'
  ];

  for (const [depotId, depotVdfPath] of Object.entries(depotBuilds)) {
    vdf.push(`    "${depotId}" "${depotVdfPath}"`);
  }

  if (setLive) {
    vdf.push(`  }`);
    vdf.push(`  "setlive" "${setLive}"`);
  }

  vdf.push('}');
  return vdf.join('\n');
}
```

### SteamCMD 호출

마지막으로 SteamCMD를 호출하여 업로드를 수행합니다.

```javascript
await runCommand(steamcmdPath, [
  '+login', steamUsername, steamPassword, steamGuardCode,
  '+run_app_build', appBuildPath,
  '+quit'
]);
```

이것이 전체 워크플로우의 마지막 도약입니다. 성공하면 릴리스가 완료됩니다.

## 다중 플랫폼 Depot 처리

Steam은 Depot 시스템을 사용하여 다양한 플랫폼의 콘텐츠를 관리합니다. 세 가지 주요 Depot을 지원합니다.

| 플랫폼 | Depot 식별자 | 아키텍처 지원 |
|-----|-----------|---------|
| Linux | `linux-x64` | x64_64 |
| Windows | `win-x64` | x64_64 |
| macOS | `osx-universal` | universal, x64_64, arm64 |

각 Depot에는 자체 콘텐츠 디렉토리와 VDF 구성 파일이 있습니다. 이를 통해 다른 플랫폼의 사용자는 실제로 필요한 콘텐츠만 다운로드합니다.

## 릴리스 워크플로우

### 1단계: GitHub Release 준비

먼저 `portable-version` 저장소에 GitHub Release를 생성합니다. 다음이 포함됩니다.
- 각 플랫폼의 압축 아카이브
- 빌드 매니페스트 (`{tag}.build-manifest.json`)
- 산출물 인벤토리 (`{tag}.artifact-inventory.json`)

### 2단계: Steam 릴리스 워크플로우 트리거

GitHub Actions를 통해 워크플로우를 수동으로 트리거하고 필요한 매개변수를 입력합니다.
- `release`: 게시할 버전 태그 (예: `v1.0.0`)
- `steam_branch`: 대상 브랜치 (예: `preview` 또는 `public`)
- `steam_preview`: 프리뷰 모드를 사용할지 여부

### 3단계: 릴리스 흐름 자동 실행

워크플로우가 자동으로 다음 단계를 수행합니다.

1. GitHub Release 산출물을 다운로드하고 추출합니다.
2. SteamCMD를 설치하거나 업데이트합니다.
3. Steam VDF 구성 파일을 생성합니다.
4. Steam Guard로 인증합니다.
5. 콘텐츠를 Steam CDN에 업로드합니다.
6. 지정된 브랜치를 live로 설정합니다.

이 시퀀스가 완료되면 전체 릴리스 경로가 커버됩니다.

## 설정 가이드

### 필수 시크릿 구성

GitHub 저장소 설정에서 다음 시크릿을 구성합니다.

| 시크릿 이름 | 설명 |
|-----------|------|
| `STEAM_USERNAME` | Steam 계정 사용자 이름 |
| `STEAM_PASSWORD` | Steam 계정 비밀번호 |
| `STEAM_SHARED_SECRET` | Steam Guard 공유 시크릿 (선택 사항) |
| `STEAM_GUARD_CODE` | Steam Guard 코드 (선택 사항) |
| `STEAM_APP_ID` | Steam 애플리케이션 ID |
| `STEAM_DEPOT_ID_LINUX` | Linux Depot ID |
| `STEAM_DEPOT_ID_WINDOWS` | Windows Depot ID |
| `STEAM_DEPOT_ID_MACOS` | macOS Depot ID |

이러한 설정은 특별히 이상하지 않습니다. 필요한 모든 값이 제자리에 있으면 됩니다.

### 환경 변수 구성

| 변수 이름 | 설명 | 기본값 |
|---------|------|--------|
| `PORTABLE_VERSION_STEAMCMD_ROOT` | SteamCMD 설치 디렉토리 | `~/.local/share/portable-version/steamcmd` |

## 모범 사례

### Steam Guard 인증 관리

첫 번째 실행 시 Steam Guard 코드를 수동으로 입력해야 합니다. 그 후에는 공유 시크릿을 구성하여 코드를 자동으로 생성하는 것이 좋습니다. 이렇게 하면 모든 릴리스마다 수동 개입을 피할 수 있습니다.

SteamCMD는 로그인 토큰을 저장하고 나중에 재사용할 수 있습니다. 토큰 만료 후에는 재인증이 필요하므로 토큰 만료에도 주의해야 합니다.

### 콘텐츠 디렉토리 구조

Steam 콘텐츠 디렉토리 구조가 올바른지 확인하세요.

```
steam-content/
├── linux-x64/     # Linux 플랫폼 콘텐츠
├── win-x64/       # Windows 플랫폼 콘텐츠
└── osx-universal/ # macOS 범용 바이너리 콘텐츠
```

각 디렉토리에는 해당 플랫폼의 완전한 애플리케이션 파일이 포함되어야 합니다.

### 프리뷰 모드 사용

프리뷰 모드는 브랜치를 live로 설정하지 않으므로 테스트 및 검증에 적합합니다.

```yaml
if [ "$STEAM_PREVIEW_INPUT" = 'true' ]; then
  cmd+=(--preview)
fi
```

이렇게 하면 먼저 Steam에 업로드하여 검증한 다음, 모든 것이 확인되면 정식 브랜치로 전환할 수 있습니다.

### 오류 처리 및 로깅

스크립트에는 완전한 오류 처리와 로깅이 포함되어 있습니다.
- GitHub Release가 존재하는지 검증합니다.
- 필수 메타데이터 파일을 확인합니다.
- 플랫폼 콘텐츠가 있는지 확인합니다.
- GitHub Actions 요약 보고서를 생성합니다.

이 정보는 디버깅과 감사에 매우 가치가 있습니다.

### 산출물 관리

워크플로우는 두 가지 산출물을 생성합니다.
- `portable-steam-release-preparation-{tag}`: 릴리스 준비 메타데이터
- `portable-steam-build-metadata-{tag}`: Steam 빌드 메타데이터

이 산출물은 나중에 감사 및 디버깅에 사용할 수 있습니다. 30일 보유 기간이 실용적인 기본값입니다.

## 실제 사용

HagiCode 프로젝트에서 이 자동 릴리스 워크플로우는 이미 여러 버전에 걸쳐 성공적으로 실행되었습니다. GitHub Release부터 Steam 플랫폼까지 전체 경로가 완전히 자동화되며 수동 개입이 필요 없습니다.

이것은 릴리스 효율성과 신뢰성을 크게 향상시켰습니다. 과거에는 한 버전을 수동으로 게시하는 데 30분 이상이 걸렸습니다. 이제 전체 프로세스가 몇 분 안에 완료됩니다.

더 중요한 것은 자동화된 워크플로우가 인적 오류의 가능성을 줄인다는 것입니다. 모든 릴리스는 동일한 표준화된 프로세스를 따르므로 결과를 더 예측할 수 있습니다.

## 요약

이 글에서 설명한 접근 방식을 통해 다음을 달성했습니다.
1. GitHub Release부터 Steam 플랫폼까지의 완전 자동화.
2. 다중 플랫폼 Depot 업로드.
3. Steam Guard 기반의 안전한 인증.
4. 프리뷰 모드와 정식 릴리스 간의 유연한 전환.
5. 완전한 오류 처리 및 로깅.

이 해결책은 HagiCode 프로젝트뿐만 아니라 Steam 출시를 계획 중인 다른 프로젝트에도 참고 자료로 활용할 수 있습니다. Steam 릴리스 자동화를 고려 중이시라면 이 실무 경험이 도움이 되기를 바랍니다.

기술은 선택한 경로에 따라 복잡하게 느껴질 수도 있고 간단하게 느껴질 수도 있습니다. 핵심은 자신의 필요에 맞는 워크플로우를 찾는 것입니다.
