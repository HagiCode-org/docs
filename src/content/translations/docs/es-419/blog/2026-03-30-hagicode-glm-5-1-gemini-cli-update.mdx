## Compatibilidad total con GLM-5.1 e integración con Gemini CLI: el camino de evolución multimodelo de HagiCode

> Este artículo presenta dos actualizaciones recientes importantes de la plataforma HagiCode: soporte total para el modelo Zhipu AI GLM-5.1 y la integración exitosa de Gemini CLI como el décimo Agent CLI. Juntas, estas actualizaciones fortalecen aún más las capacidades multimodelo y el ecosistema multi-CLI de la plataforma.

:::nota[Aviso de contexto temporal]
Este artículo registra el progreso de la expansión de múltiples CLI tal como estaba en el momento del lanzamiento. ## Antecedentes

El tiempo realmente pasa rápido. El desarrollo de grandes modelos lingüísticos ha ido aumentando como el bambú en primavera. No hace mucho, todavía aplaudíamos por "una IA que pueda escribir código". Ahora ya estamos en una era de colaboración multimodelo e integración de múltiples herramientas. ¿Es eso emocionante? Tal vez. Después de todo, lo que los desarrolladores necesitan nunca ha sido solo la herramienta en sí, sino la facilidad de adaptarse a diferentes escenarios y cambiar de manera flexible cuando sea necesario.

Como plataforma de codificación asistida por IA, HagiCode ha dado la bienvenida recientemente a dos desarrollos importantes: primero, la integración completa del modelo GLM-5.1 de Zhipu AI; en segundo lugar, la incorporación oficial de Gemini CLI como el décimo Agent CLI compatible. Puede que estas dos actualizaciones no parezcan trascendentales, pero sin duda son buenas noticias para la continua maduración de la plataforma.

GLM-5.1 es el último modelo insignia de Zhipu AI. En comparación con GLM-5.0, ofrece un razonamiento más sólido, una comprensión más profunda del código y una llamada a herramientas más fluida. Más importante aún, es el primer modelo GLM que admite la entrada de imágenes. ¿Qué significa eso? Significa que los usuarios pueden dejar que la IA mire directamente una captura de pantalla en lugar de tener que esforzarse por describir el problema con palabras. Una vez que haya utilizado esa comodidad, inmediatamente comprenderá su valor.

Al mismo tiempo, a través de la `HagiCode.Libs.Providers` arquitectura, HagiCode integró con éxito Gemini CLI en la plataforma. Este es ahora el décimo Agente CLI. Para ser honesto, llegar a este punto genera una modesta sensación de logro.

También vale la pena mencionar que la función de carga de imágenes de HagiCode permite a los usuarios comunicarse con la IA directamente a través de capturas de pantalla. Incluso cuando se ejecuta GLM 4.7, la plataforma sigue funcionando bien y ya ha ayudado a completar muchas tareas de compilación importantes. En cuanto al GLM-5.1, naturalmente, va un paso más allá.

## Acerca de HagiCode

El enfoque compartido en este artículo proviene de nuestra experiencia práctica en el [HagiCode](https://hagicode.com) proyecto. HagiCode es una plataforma de codificación asistida por IA de código abierto diseñada para proporcionar a los desarrolladores un asistente de programación de IA potente y flexible a través de una arquitectura multimodelo y multi-CLI. Repositorio de proyectos: [github.com/HagiCode-org/site](https://github.com/HagiCode-org/site)

## Diseño de arquitectura multi-CLI

Una de las principales fortalezas de HagiCode es su soporte para múltiples herramientas CLI de programación de IA a través de una capa de abstracción unificada. La ventaja de este diseño es bastante simple: pueden aparecer nuevas herramientas, las antiguas pueden permanecer y el código base no se convierte en un caos. Para ser justos, así es como a todos les gustaría que fuera la vida.

### Enumeración AIProviderType

La plataforma define los tipos de proveedores CLI admitidos a través de `AIProviderType` enumeración:

```csharp
public enum AIProviderType
{
    ClaudeCodeCli = 0,    // Claude Code CLI
    CodexCli = 1,          // GitHub Copilot Codex
    GitHubCopilot = 2,     // GitHub Copilot
    CodebuddyCli = 3,     // Codebuddy CLI
    OpenCodeCli = 4,      // OpenCode CLI
    IFlowCli = 5,         // IFlow CLI
    HermesCli = 6,        // Hermes CLI
    QoderCli = 7,         // Qoder CLI
    KiroCli = 8,          // Kiro CLI
    KimiCli = 9,          // Kimi CLI
    GeminiCli = 10,       // Gemini CLI (new)
}
```

Como puedes ver, Gemini CLI se une a esta familia como el décimo miembro. Cada CLI tiene sus propias características y escenarios de uso, por lo que los usuarios pueden elegir de manera flexible según sus necesidades. Después de todo, muchos caminos conducen a Roma; algunos son simplemente más fáciles que otros.

### Arquitectura del proveedor

`HagiCode.Libs.Providers` proporciona una interfaz de proveedor unificada que hace que cada integración CLI sea estandarizada y concisa. Tomando Gemini CLI como ejemplo:

```csharp
public class GeminiProvider : ICliProvider<GeminiOptions>
{
    private static readonly string[] DefaultExecutableCandidates = ["gemini", "gemini-cli"];
    private const string ManagedBootstrapArgument = "--acp";

    public string Name => "gemini";
    public bool IsAvailable => _executableResolver.ResolveFirstAvailablePath(DefaultExecutableCandidates) is not null;
}
```

Los beneficios de este diseño son:
- La integración de una nueva CLI solo requiere implementar una clase de Proveedor
- Gestión unificada del ciclo de vida y agrupación de sesiones.
- Resolución de alias automatizada y descubrimiento de ejecutables

En pocas palabras, este diseño convierte las cosas complicadas en otras más simples y hace la vida un poco más fácil.

### Registro de proveedores

El Registro de proveedores maneja automáticamente la asignación y el registro de alias:

```csharp
if (provider is GeminiProvider)
{
    registry.Register(provider.Name, provider, ["gemini-cli"]);
    continue;
}
```

Esto significa que los usuarios pueden invocar Gemini CLI con `gemini` o `gemini-cli`, y el sistema lo reconocerá automáticamente. Es como un amigo con un nombre formal y un apodo; de cualquier manera, la gente sabe a quién te refieres.

## Soporte del modelo GLM-5.1

GLM-5.1 es el último modelo insignia de Zhipu AI y HagiCode ha completado el soporte completo para él.

### Catálogo de Profesiones Secundarias

HagiCode gestiona todos los modelos compatibles a través del Catálogo de profesiones secundarias. Aquí está la configuración para la serie GLM:

| ID del modelo | Nombre | SoportaImagen | Familias CLI compatibles |
|----------|------|----------------|-------------------------|
| `glm-4.7` | GLM 4.7 | - | claude, codebuddy, hermes, qoder, kiro |
| `glm-5` | GLM 5 | - | claude, codebuddy, hermes, qoder, kiro |
| `glm-5-turbo` | GLM 5 Turbo | - | claude, codebuddy, hermes, qoder, kiro |
| `glm-5.0` | GLM 5.0 (heredado) | - | claude, codebuddy, hermes, qoder, kiro |
| `glm-5.1` | GLM 5.1 | **verdadero** | claude, codebuddy, hermes, qoder, kiro |

Las características clave de GLM-5.1 se pueden resumir de la siguiente manera:
- Un identificador de versión independiente sin equipaje heredado
- El primer modelo GLM que admite la entrada de imágenes
- Mayor razonamiento y comprensión del código.
- Amplia compatibilidad con múltiples CLI

### GLM-5.1 frente a GLM-5.0

A nivel de código, la diferencia clave entre GLM-5.1 y GLM-5.0 se muestra aquí:

```csharp
// GLM-5.0 (Legacy) - contains special retention logic
private const string Glm50CodebuddySecondaryProfessionId = "secondary-glm-5-codebuddy";
private const string Glm50CodebuddyModelValue = "glm-5.0";

// GLM-5.1 - standalone new model identifier
private const string Glm51SecondaryProfessionId = "secondary-glm-5-1";
private const string Glm51ModelValue = "glm-5.1";
```

GLM-5.0 lleva la etiqueta "Legacy" porque es un identificador de versión antigua que se conserva por motivos de compatibilidad con versiones anteriores. GLM-5.1, por el contrario, es una versión independiente completamente nueva sin carga histórica. Algunas cosas quedan en el pasado; otros viajan más ligeros y se mueven más rápido.

### Configurar GLM-5.1

Aquí hay un ejemplo de configuración para usar GLM-5.1 en HagiCode:

```json
{
  "primaryProfessionId": "profession-claude-code",
  "secondaryProfessionId": "secondary-glm-5-1",
  "model": "glm-5.1",
  "reasoning": "high"
}
```

## Función de carga de imágenes

El soporte de imágenes de HagiCode se implementa a través de `SupportsImage` propiedad en `SecondaryProfession`:

```csharp
public class HeroSecondaryProfessionSettingDto
{
    public bool SupportsImage { get; set; }
}
```

En el Catálogo de Profesiones Secundarias, la configuración GLM-5.1 se ve así:

```json
{
  "id": "secondary-glm-5-1",
  "supportsImage": true
}
```

Esto significa que los usuarios pueden cargar capturas de pantalla directamente para el análisis de IA, como:
- Capturas de pantalla de mensajes de error
- Problemas en una pantalla de UI
- Gráficos de visualización de datos.
- Resultados de la ejecución del código

Ya no es necesario describir todo manualmente: simplemente cargue la captura de pantalla. La conveniencia de esta función es obvia una vez que la haya utilizado. A veces una mirada dice más que una larga explicación.

## Integración CLI de Géminis

Como décimo Agente CLI, Gemini CLI está integrado en HagiCode a través de la arquitectura de Proveedor estándar.

### Opciones de configuración

Gemini CLI admite un amplio conjunto de opciones de configuración:

```csharp
public class GeminiOptions
{
    public string? ExecutablePath { get; set; }
    public string? WorkingDirectory { get; set; }
    public string? SessionId { get; set; }
    public string? Model { get; set; }
    public string? AuthenticationMethod { get; set; }
    public string? AuthenticationToken { get; set; }
    public Dictionary<string, string?> AuthenticationInfo { get; set; }
    public Dictionary<string, string?> EnvironmentVariables { get; set; }
    public string[] ExtraArguments { get; set; }
    public TimeSpan? StartupTimeout { get; set; }
    public CliPoolSettings? PoolSettings { get; set; }
}
```

Estas opciones cubren todo, desde la configuración básica hasta funciones avanzadas, brindando a los usuarios la flexibilidad de configurar la CLI según sus propias necesidades. El flujo de trabajo de cada persona es diferente, por lo que siempre es bienvenido un poco de flexibilidad.

### Protocolo de comunicación ACP

Gemini CLI admite ACP (Protocolo de comunicación del agente), que es el estándar de comunicación CLI unificado de HagiCode. A través de ACP, diferentes CLI pueden interactuar con la plataforma de manera consistente, simplificando enormemente el trabajo de integración. En resumen, estandariza las partes complicadas para que todos puedan trabajar más fácilmente.

Una vez configurado, HagiCode puede llamar al modelo GLM-5.1 normalmente. No es ni especialmente difícil ni especialmente fácil: sólo tienes que seguir la configuración según lo previsto.

## Mejores prácticas para la integración multi-CLI

`HagiCode.Libs.Providers` proporciona un mecanismo unificado para el registro y uso:

```csharp
services.AddHagiCodeLibs();

var gemini = serviceProvider.GetRequiredService<ICliProvider<GeminiOptions>>();
var codebuddy = serviceProvider.GetRequiredService<ICliProvider<CodebuddyOptions>>();
var hermes = serviceProvider.GetRequiredService<ICliProvider<HermesOptions>>();
```

Este diseño de inyección de dependencia mantiene el uso en diferentes CLI muy conciso y también hace que las pruebas unitarias y las simulaciones sean más convenientes. El código limpio es una forma de ser responsable ante uno mismo.

### Notas

Hay algunas cosas a tener en cuenta en el uso real:

1. **Configuración de clave API**: asegúrese de `ANTHROPIC_AUTH_TOKEN` está configurado correctamente o no se puede llamar al modelo
2. **Disponibilidad del modelo**: GLM-5.1 debe ser habilitado por el proveedor del modelo correspondiente
3. **Característica de imagen**: Sólo modelos con `supportsImage: true` puede usar la carga de imágenes
4. **Instalación CLI**: antes de usar Gemini CLI, asegúrese `gemini` o `gemini-cli` está en la RUTA del sistema

Estos pueden ser pequeños detalles, pero los pequeños detalles mal manejados pueden convertirse en grandes problemas, por lo que vale la pena prestarles atención.
