## Integración profunda y uso práctico de StreamJsonRpc en HagiCode

> Este artículo explica en detalle cómo el proyecto HagiCode (anteriormente PCode) integró con éxito la biblioteca de comunicación StreamJsonRpc de Microsoft para reemplazar su implementación JSON-RPC personalizada original, al mismo tiempo que resolvió los problemas técnicos y los desafíos arquitectónicos encontrados durante el proceso de integración.

## Antecedentes

StreamJsonRpc es la biblioteca de comunicación JSON-RPC mantenida oficialmente por Microsoft para .NET y TypeScript, conocida por su sólida seguridad de tipos, generación automática de proxy y mecanismos maduros de manejo de excepciones. En el proyecto HagiCode, el equipo decidió integrar StreamJsonRpc para comunicarse con herramientas externas de IA, como iflow CLI y OpenCode CLI a través de ACP (Protocolo de comunicación de agente), al tiempo que eliminaba el costo de mantenimiento y los posibles errores introducidos por la implementación anterior personalizada de JSON-RPC. Sin embargo, el proceso de integración se topó con desafíos específicos de la transmisión JSON-RPC, especialmente al manejar el enlace de destino del proxy y el reconocimiento de parámetros genéricos.

Para abordar estos puntos débiles, tomamos una decisión audaz: reconstruir todo el sistema de compilación desde cero. El impacto de esa decisión puede ser incluso mayor de lo que espera y lo explicaré en detalle en breve.

## Análisis

El proyecto actual se encuentra en una etapa crítica de la integración del protocolo ACP y enfrenta los siguientes puntos técnicos y desafíos arquitectónicos:

### 1. Limitaciones de la implementación personalizada
La implementación JSON-RPC original se encuentra en `src/HagiCode.ClaudeHelper/AcpImp/` e incluye componentes como `JsonRpcEndpoint` y `ClientSideConnection`. Mantener esta base de código personalizada es costoso y carece de las capacidades avanzadas de una biblioteca madura, como informes de progreso y soporte de cancelación.

### 2. Obstáculos de la integración de StreamJsonRpc
Al intentar migrar el existente `CallbackProxyTarget` patrón a StreamJsonRpc, encontramos que el `_rpc.AddLocalRpcTarget(target)` El método no pudo reconocer los objetivos creados a través del patrón de proxy. Específicamente, StreamJsonRpc no pudo dividir automáticamente propiedades del tipo genérico `T` en los parámetros del método RPC, lo que provoca que el lado del servidor falle al procesar las llamadas al método iniciadas por el cliente.

### 3. Capas arquitectónicas confusas
El existente `ClientSideConnection` mezcla la capa de transporte (WebSocket/Stdio), la capa de protocolo (JSON-RPC) y la capa empresarial (interfaz del agente ACP), lo que genera responsabilidades poco claras. También sufre de enlaces de métodos faltantes en `AcpAgentCallbackRpcAdapter`.

### 4. Registros faltantes
La capa de transporte WebSocket carece de registro de contenido JSON sin formato, lo que dificulta determinar durante la depuración RPC si un problema se origina en la serialización o en la red.

## Solución

Para abordar los problemas anteriores, adoptamos la siguiente solución sistemática, optimizando desde tres dimensiones: refactorización arquitectónica, integración de bibliotecas y depuración mejorada.

### 1. Migrar completamente a StreamJsonRpc

#### Eliminar código heredado
Eliminar `JsonRpcEndpoint.cs`, `AgentSideConnection.cs`y convertidores de serialización personalizados relacionados, como `JsonRpcMessageJsonConverter`.

#### Integrar la Biblioteca Oficial
Introducir el `StreamJsonRpc` paquete NuGet y use su `JsonRpc` clase para manejar la lógica de comunicación central.

#### Resumen de la capa de transporte
Definir el `IAcpTransport` interfaz para manejar ambos `WebSocket` y `Stdio` modos de transporte de forma unificada, asegurando que la capa de protocolo esté desacoplada de la capa de transporte.

```csharp
// Definition of the IAcpTransport interface
public interface IAcpTransport
{
    Task SendAsync(string message, CancellationToken cancellationToken = default);
    Task<string> ReceiveAsync(CancellationToken cancellationToken = default);
    Task CloseAsync(CancellationToken cancellationToken = default);
}

// WebSocket transport implementation
public class WebSocketTransport : IAcpTransport
{
    private readonly WebSocket _webSocket;

    public WebSocketTransport(WebSocket webSocket)
    {
        _webSocket = webSocket;
    }

    // Implement send and receive methods
    // ...
}

// Stdio transport implementation
public class StdioTransport : IAcpTransport
{
    private readonly StreamReader _reader;
    private readonly StreamWriter _writer;

    public StdioTransport(StreamReader reader, StreamWriter writer)
    {
        _reader = reader;
        _writer = writer;
    }

    // Implement send and receive methods
    // ...
}
```

### 2. Solucionar problemas de reconocimiento de objetivos de proxy

#### analizar `CallbackProxyTarget`
Inspeccione la lógica de generación de proxy dinámico existente e identifique la causa raíz de por qué StreamJsonRpc no puede reconocerlo. En la mayoría de los casos, esto sucede porque el objeto proxy no expone públicamente las firmas del método real o utiliza tipos de parámetros no admitidos por StreamJsonRpc.

#### Paso de parámetros de refactorización
Divida las propiedades genéricas en parámetros explícitos del método RPC. En lugar de depender de propiedades dinámicas, defina DTO (objetos de transferencia de datos) de solicitud/respuesta concretos para que StreamJsonRpc pueda reconocer correctamente las firmas de métodos a través de la reflexión.

```csharp
// Original generic property approach
public class CallbackProxyTarget<T>
{
    public Func<T, Task> Callback { get; set; }
}

// Refactored concrete method approach
public class ReadTextFileRequest
{
    public string FilePath { get; set; }
}

public class ReadTextFileResponse
{
    public string Content { get; set; }
}

public interface IAcpAgentCallback
{
    Task<ReadTextFileResponse> ReadTextFileAsync(ReadTextFileRequest request);
    // Other methods...
}
```

#### uso `Attach` en lugar de `AddLocalRpcTarget`
En algunos escenarios complejos, proxy manualmente el `JsonRpc` objeto y manejo `RpcConnection` puede ser más flexible que agregar directamente un objetivo local.

### 3. Implementar vinculación de métodos y un registro más sólido

#### implementar `AcpAgentCallbackRpcAdapter`
Asegúrese de que este componente implemente explícitamente la interfaz de proxy StreamJsonRpc y asigne los métodos definidos por el protocolo ACP, como `ReadTextFileAsync`, a los controladores de devolución de llamada StreamJsonRpc.

#### Integrar el registro
Intercepte y registre el texto sin formato de solicitudes y respuestas JSON-RPC en la canalización de procesamiento de mensajes WebSocket o Stdio. uso `ILogger` para generar la carga útil sin procesar antes del análisis y después de la serialización para que los problemas de formato se puedan diagnosticar más fácilmente.

```csharp
// Transport wrapper with enhanced logging
public class LoggingAcpTransport : IAcpTransport
{
    private readonly IAcpTransport _innerTransport;
    private readonly ILogger<LoggingAcpTransport> _logger;

    public LoggingAcpTransport(IAcpTransport innerTransport, ILogger<LoggingAcpTransport> logger)
    {
        _innerTransport = innerTransport;
        _logger = logger;
    }

    public async Task SendAsync(string message, CancellationToken cancellationToken = default)
    {
        _logger.LogTrace("Sending message: {Message}", message);
        await _innerTransport.SendAsync(message, cancellationToken);
    }

    public async Task<string> ReceiveAsync(CancellationToken cancellationToken = default)
    {
        var message = await _innerTransport.ReceiveAsync(cancellationToken);
        _logger.LogTrace("Received message: {Message}", message);
        return message;
    }

    public async Task CloseAsync(CancellationToken cancellationToken = default)
    {
        _logger.LogDebug("Closing connection");
        await _innerTransport.CloseAsync(cancellationToken);
    }
}
```

### 4. Refactorizar capas arquitectónicas

#### Capa de transporte (`AcpRpcClient`)
Envuelva la conexión StreamJsonRpc y hágala responsable de `InvokeAsync` Gestión del ciclo de vida de llamadas y conexiones.

```csharp
public class AcpRpcClient : IDisposable
{
    private readonly JsonRpc _rpc;
    private readonly IAcpTransport _transport;

    public AcpRpcClient(IAcpTransport transport)
    {
        _transport = transport;
        _rpc = new JsonRpc(new StreamRpcTransport(transport));
        _rpc.StartListening();
    }

    public async Task<TResponse> InvokeAsync<TResponse>(string methodName, object parameters)
    {
        return await _rpc.InvokeAsync<TResponse>(methodName, parameters);
    }

    public void Dispose()
    {
        _rpc.Dispose();
        _transport.Dispose();
    }

    // StreamRpcTransport is the StreamJsonRpc adapter for IAcpTransport
    private class StreamRpcTransport : IDuplexPipe
    {
        // Implement the IDuplexPipe interface
        // ...
    }
}
```

#### Capa de protocolo (`IAcpAgentClient` / `IAcpAgentCallback`)
Defina interfaces claras de cliente a agente y de agente a cliente. Eliminar el patrón cíclico de fábrica `Func<IAcpAgent, IAcpClient>` y reemplácelo con inyección de dependencia o registro de devolución de llamada directa.

## Practica

Según las mejores prácticas de StreamJsonRpc y la experiencia en proyectos, las siguientes son las recomendaciones clave para la implementación:

### 1. Los DTO fuertemente tipados son mejores que los objetos dinámicos
La principal ventaja de StreamJsonRpc radica en la escritura segura. no usar `dynamic` o `JObject` para pasar parámetros. En su lugar, defina clases explícitas de C# POCO como parámetros para cada método RPC. Esto no solo resuelve el problema de reconocimiento de destino del proxy, sino que también detecta errores de tipo en tiempo de compilación.

Ejemplo: reemplazar las propiedades genéricas en `CallbackProxyTarget` con clases concretas como `ReadTextFileRequest` y `WriteTextFileRequest`.

### 2. Declarar explícitamente los nombres de los métodos
Utilice el `[JsonRpcMethod]` atributo para especificar explícitamente los nombres de los métodos RPC en lugar de depender de la asignación predeterminada de nombres de métodos. Esto evita errores de invocación causados por diferencias en el estilo de nombres, como PascalCase versus camelCase.

```csharp
public interface IAcpAgentCallback
{
    [JsonRpcMethod("readTextFile")]
    Task<ReadTextFileResponse> ReadTextFileAsync(ReadTextFileRequest request);
    
    [JsonRpcMethod("writeTextFile")]
    Task WriteTextFileAsync(WriteTextFileRequest request);
}
```

### 3. Aproveche las devoluciones de llamadas del estado de conexión
StreamJsonRpc proporciona la `JsonRpc.ConnectionLost` evento. Es absolutamente necesario escuchar este evento para manejar salidas inesperadas de procesos o desconexiones de red, lo cual es más oportuno que depender únicamente de la detección de fallas de Orleans Grain.

```csharp
_rpc.ConnectionLost += (sender, e) =>
{
    _logger.LogError("RPC connection lost: {Reason}", e.ToString());
    // Handle reconnection logic or notify the user
};
```

### 4. Grabar registros por capa
- **Nivel de seguimiento**: registre la carga útil completa de solicitud/respuesta JSON sin formato.
- **Nivel de depuración**: registra pilas de llamadas a métodos y resúmenes de parámetros.
- **Nota**: Asegúrese de que los registros no incluyan tokens de autorización confidenciales ni contenido de archivos grandes codificados en Base64.

### 5. Manejar la naturaleza especial del transporte por streaming
StreamJsonRpc admite de forma nativa `IAsyncEnumerable`. Al implementar respuestas rápidas de transmisión para ACP, utilice `IAsyncEnumerable` directamente en lugar de lógica de paginación personalizada. Esto puede simplificar enormemente la cantidad de código necesario para el procesamiento de transmisión.

```csharp
public interface IAcpAgentCallback
{
    [JsonRpcMethod("streamText")]
    IAsyncEnumerable<string> StreamTextAsync(StreamTextRequest request);
}
```

### 6. Patrón de adaptador
mantener `ACPSession` y `ClientSideConnection` separar. `ACPSession` debe centrarse en la gestión del estado de Orleans y la lógica empresarial, como la puesta en cola de mensajes, y debe utilizar el objeto de conexión StreamJsonRpc mediante composición en lugar de herencia.

## Conclusión

Al integrar completamente StreamJsonRpc, el proyecto HagiCode abordó con éxito el alto costo de mantenimiento, las limitaciones funcionales y la confusión de capas arquitectónicas de la implementación personalizada original. Las mejoras clave incluyen:

1. Reemplazo de propiedades dinámicas con DTO fuertemente tipados, mejorando la mantenibilidad y la confiabilidad
2. Implementar la abstracción de la capa de transporte y la separación de la capa de protocolo, mejorando la claridad arquitectónica
3. Fortalecer las capacidades de registro para facilitar el diagnóstico de los problemas de comunicación.
4. Presentamos soporte de transmisión para simplificar la implementación de la transmisión

Estas mejoras proporcionan a HagiCode una base de comunicación más estable y eficiente, lo que le permite interactuar mejor con herramientas de IA externas y sentar una base sólida para futuras expansiones de funciones.
