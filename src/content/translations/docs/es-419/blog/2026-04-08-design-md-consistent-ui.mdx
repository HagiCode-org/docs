## Design.md: una solución para un diseño de interfaz de usuario coherente impulsado por IA

> En la era del desarrollo frontend asistido por IA, ¿cómo se pueden mantener coherentes las IU generadas por IA? Este artículo comparte nuestra experiencia práctica en la creación de un sitio de galería de diseño basado en awesome-design-md, además de cómo crear un sitio estructurado. `design.md` para guiar la IA hacia un diseño de interfaz de usuario estandarizado.

## Antecedentes

Cualquiera que haya usado IA para escribir código frontend probablemente haya tenido una experiencia similar: pídale a AI que genere la misma página varias veces y cada resultado aparecerá en un estilo diferente. A veces las esquinas están redondeadas, a veces son afiladas. A veces el espaciado es de 8px y otras veces de 16px. Incluso el mismo botón puede verse diferente en diferentes conversaciones.

Ésta no es una cuestión aislada. A medida que el desarrollo asistido por IA se vuelve más común, la **falta de coherencia en la interfaz de usuario generada por IA** se ha convertido en un problema generalizado. Diferentes asistentes de IA, diferentes indicaciones e incluso el mismo asistente en diferentes conversaciones pueden producir diseños de interfaz dramáticamente diferentes. Eso crea un enorme costo de mantenimiento durante la iteración del producto.

La causa principal es realmente simple: no existe un **documento de referencia de diseño autorizado**. Las hojas de estilo CSS tradicionales pueden decirle a los desarrolladores "cómo implementar" algo, pero no pueden comunicar completamente "por qué está diseñado de esta manera" o "qué patrón de diseño debe usarse en cada escenario". Para la IA, una descripción clara y estructurada es aún más importante para comprender las convenciones de diseño.

Al mismo tiempo, la comunidad de código abierto ya ofrece excelentes recursos. el [VoltAgent/diseño-impresionante-md](https://github.com/VoltAgent/awesome-design-md) El proyecto recopila documentación del sistema de diseño de muchas empresas conocidas. Cada directorio contiene un `README.md`, un `DESIGN.md`y obtener una vista previa de HTML. Sin embargo, todo eso está disperso en el repositorio ascendente, lo que dificulta la navegación y la comparación rápida.

Entonces, ¿podemos consolidar esos recursos en una galería de diseño fácil de explorar y al mismo tiempo destilar una estructura estructurada? `design.md` para que la use la IA?

La respuesta es sí. A continuación, permítanme explicar nuestro enfoque.

## Análisis

Antes de escribir código, analicemos los desafíos técnicos detrás de este problema.

### Gestión de contenido fuente: ¿Cómo se unifican recursos de diseño dispersos?

El repositorio ascendente awesome-design-md contiene una gran cantidad de documentos de diseño, pero necesitábamos una forma de incorporarlos a nuestro propio proyecto.

**Solución: usar el submódulo git**

```
awesome-design-md-site
└── vendor/awesome-design-md          # Upstream resources (git submodule)
```

Esto nos aporta varios beneficios:
- Control de versiones: podemos fijar una versión ascendente específica
- Compilaciones sin conexión: no es necesario solicitar API externas durante la compilación
- Revisión de contenido: los cambios específicos son visibles en los RP

### Normalización de datos: ¿Cómo se estandarizan diferentes estructuras de documentos?

Cada empresa estructura sus documentos de diseño de forma diferente. A algunos les faltan archivos de vista previa y otros usan nombres inconsistentes. Necesitamos normalizarlos durante el proceso de construcción.

**Solución: escanear y generar entradas normalizadas en el momento de la compilación**

El módulo principal es `awesomeDesignCatalog.ts`, responsable de:
1. Escaneando el `vendor/awesome-design-md/design-md/*` directorio
2. Validar si cada entrada contiene los archivos requeridos (`README.md`, `DESIGN.md`y al menos un archivo de vista previa)
3. Extraer y renderizar contenido de Markdown en HTML
4. Generando datos de entrada normalizados

### Arquitectura de sitio estático: ¿Cómo se puede proporcionar una búsqueda dinámica sin dejar de ser completamente estático?

Dado que se trata de una galería de diseño, la búsqueda es imprescindible. Pero Astro es un generador de sitios estáticos, entonces, ¿cómo se implementa la búsqueda en tiempo real?

**Solución: React island + sincronización de parámetros de consulta de URL**

```typescript
// src/components/gallery/SearchToolbar.tsx

export function SearchToolbar() {
  const [query, setQuery] = useState('');

  // Sync with the URL
  useEffect(() => {
    const params = new URLSearchParams(window.location.search);
    setQuery(params.get('q') || '');
  }, []);

  // Filter in real time
  const filtered = entries.filter(entry =>
    entry.searchText.includes(query)
  );

  return <input value={query} onChange={e => {
    setQuery(e.target.value);
    updateURL(e.target.value);
  }} />;
}
```

La ventaja de este enfoque es que mantiene intacta la capacidad de implementación de un sitio estático, lo que significa que se puede implementar en cualquier servicio de alojamiento estático y, al mismo tiempo, ofrece una experiencia de filtrado instantánea.

### Documentación de diseño: ¿Cómo se puede ayudar a la IA a comprender y seguir los estándares de diseño?

Este es el núcleo de toda la solución. Necesitamos crear una estructura `design.md` que la IA pueda entender y aplicar.

**Solución: tomar prestada la estructura de ClickHouse `DESIGN.md`**

[Haga clic en Casa `DESIGN.md`](https://github.com/ClickHouse/ClickHouse/blob/master/DESIGN.md) es una excelente referencia. Incluye:
- Tema visual y atmósfera
- Paleta de colores y roles
- Reglas de tipografía
- Estilos de componentes
- Principios de diseño
- Profundidad y elevación
- Qué hacer y qué no hacer
- Comportamiento receptivo
- Guía de avisos para agentes

Nuestro enfoque es: **reutilizar la estructura, reescribir el contenido**. Mantenemos la estructura de secciones de ClickHouse `DESIGN.md`, pero reemplace el contenido con los tokens de diseño reales y las convenciones de componentes utilizadas en nuestro propio proyecto.

## Solución

Según el análisis anterior, nuestra solución consta de cuatro módulos principales.

### 1. Canal de ingesta de contenido

Esta es la base de todo el sistema, responsable de extraer y normalizar el contenido de los recursos ascendentes.

```typescript
// src/lib/content/awesomeDesignCatalog.ts

export async function scanSourceEntries(): Promise<DesignEntry[]> {
  const designDir = 'vendor/awesome-design-md/design-md';
  const entries: DesignEntry[] = [];

  for (const dir of await fs.readdir(designDir)) {
    const entryPath = path.join(designDir, dir);
    if (await isValidDesignEntry(entryPath)) {
      const entry = await normalizeDesignEntry(entryPath);
      entries.push(entry);
    }
  }

  return entries;
}

async function isValidDesignEntry(dir: string): Promise<boolean> {
  const requiredFiles = ['README.md', 'DESIGN.md'];
  for (const file of requiredFiles) {
    if (!(await fileExists(path.join(dir, file)))) {
      return false;
    }
  }
  return true;
}
```

### 2. Interfaz de navegación de la galería

La interfaz de la galería incluye tres partes principales:

**Página de inicio**: muestra una cuadrícula de tarjetas de todas las entradas de diseño, y cada tarjeta incluye:
- Título y resumen de la entrada del diseño.
- Imagen de vista previa, si está disponible
- Resaltado de búsqueda rápida

**Página de detalles**: agrega la información completa para una única entrada de diseño:
- `README` documento
- `DESIGN` documento
- Vista previa con cambio de tema claro/oscuro
- Navegación a entradas adyacentes

**Navegación**: admite regresar a la galería y explorar entradas adyacentes

La galería de la página de inicio utiliza un diseño de tarjeta de alta densidad, aplanando `design.md` entradas de diferentes fuentes en un marco visual unificado para que los equipos puedan comparar rápidamente estilos de marca, patrones de botones y ritmo tipográfico.

Después de abrir una entrada específica, la página de detalles coloca el resumen del diseño y la vista previa en vivo en la misma página, lo que reduce el costo de alternar entre documentación, vistas previas y código fuente.

### 3. Funcionalidad de búsqueda

La función de búsqueda se basa en el filtrado del lado del cliente, y el estado se conserva mediante parámetros de consulta de URL:

```typescript
// src/components/gallery/SearchToolbar.tsx

function SearchToolbar({ entries }: { entries: DesignEntry[] }) {
  const [query, setQuery] = useState('');
  const [results, setResults] = useState(entries);

  useEffect(() => {
    const params = new URLSearchParams(window.location.search);
    const initialQuery = params.get('q') || '';
    setQuery(initialQuery);
    filterEntries(initialQuery);
  }, []);

  const filterEntries = (searchQuery: string) => {
    const filtered = entries.filter(entry =>
      entry.searchText.toLowerCase().includes(searchQuery.toLowerCase())
    );
    setResults(filtered);
  };

  const handleChange = (e: React.ChangeEvent<HTMLInputElement>) => {
    const value = e.target.value;
    setQuery(value);
    filterEntries(value);

    // Update the URL without triggering a page refresh
    const newUrl = value
      ? `${window.location.pathname}?q=${encodeURIComponent(value)}`
      : window.location.pathname;
    window.history.replaceState({}, '', newUrl);
  };

  return (
    <div className="search-toolbar">
      <input
        type="text"
        value={query}
        onChange={handleChange}
        placeholder="Search design entries..."
      />
      <span className="result-count">{results.length} results</span>
    </div>
  );
}
```

## Practica

Ahora que hemos cubierto la solución, ¿cómo se implementa realmente?

### Pasos de implementación

**Paso 1: Inicializar el submódulo**

```bash
# Add the upstream repository as a submodule
git submodule add https://github.com/VoltAgent/awesome-design-md.git vendor/awesome-design-md

# Initialize and update the submodule
git submodule update --init --recursive
```

**Paso 2: crear el canal de contenido**

implementar `awesomeDesignCatalog.ts`, incluyendo:
- Lógica de validación y escaneo de archivos
- Representación de Markdown utilizando el renderizador integrado de Astro
- Extracción de datos de entrada

**Paso 3: crear la interfaz de usuario de la galería**

Utilice Astro + React Islands para crear:
- Diseño de la galería de la página de inicio (cuadrícula de tarjetas)
- Componentes de la tarjeta de diseño
- Barra de herramientas de búsqueda
- Diseño de página de detalles

**Paso 4: Escribe el documento de diseño**

Basado en la estructura de ClickHouse `DESIGN.md`, complete los tokens de diseño reales de su propio proyecto. Actualizar `README.md` y agregar un enlace a `design.md`.

### Notas

**Seguridad**: la representación de Markdown requiere filtrar HTML no seguro. Filtros de renderizado integrados de Astro `script` etiquetas de forma predeterminada, pero aún debe estar atento a los riesgos XSS.

**Rendimiento**: una gran cantidad de vistas previas de iframe puede afectar el rendimiento de la primera pintura. Se recomienda utilizar `loading="lazy"` para cargar contenido de vista previa de forma diferida.

**Mantenibilidad**: `design.md` necesita permanecer sincronizado con la implementación del código. Se recomienda agregar comprobaciones de CI para garantizar que las variables CSS permanezcan coherentes entre la documentación y el código.

**Accesibilidad**: asegúrese de que el contraste de color cumpla con el estándar WCAG AA (al menos 4,5:1).

### Guía de uso de IA

Después de crear `design.md`¿Cómo se consigue que la IA realmente la utilice? A continuación se ofrecen algunos consejos prácticos:

**Consejo 1: haga referencia explícita a él en el mensaje**

```
Please refer to the design.md file in the project root and use the design conventions defined there to implement the following components:
- Buttons: use the primary color with an 8px border radius
- Cards: use the elevation-2 shadow level
```

**Consejo 2: Exija que la IA haga referencia a variables CSS específicas**

```
Implement a navigation bar with the following requirements:
- Use --color-bg-primary for the background color
- Use --color-border-subtle for borders
- Use --text-color-primary for text
```

**Consejo 3: Incluir `design.md` contenido en el indicador del sistema**

Si su herramienta de IA admite indicaciones personalizadas del sistema, puede agregar el contenido principal de `design.md` directamente a él.

### Estrategia de prueba

**Pruebas de canalización de contenido**:
- Escenarios de archivos faltantes (faltantes `README.md` o `DESIGN.md`)
- Escenarios de error de formato (fallo en el análisis de Markdown)
- Escenarios de directorio vacío

**Prueba de la función de búsqueda**:
- Manejo de resultados vacíos
- Caracteres especiales como chino y emoji.
- Verificación de sincronización de URL

**Pruebas de componentes de UI**:
- Cambio de tema claro/oscuro
- Diseño responsivo
- Vista previa de los estados de carga

### Flujo de trabajo de implementación

```bash
# 1. Update the submodule to the latest version
git submodule update --remote

# 2. Rebuild the site
npm run build

# 3. Deploy static assets
npm run deploy
```

Se recomienda automatizar las actualizaciones de los submódulos, además de la compilación y la implementación, para que la CI pueda activarse automáticamente cada vez que se actualice el repositorio ascendente.

## Conclusión

La inconsistencia en las interfaces de usuario generadas por IA que encontró HagiCode durante el desarrollo se debió, en esencia, a la falta de un documento de referencia de diseño estructurado. Al construir un sitio de galería de diseño y crear un sitio estandarizado `design.md`, resolvimos con éxito este problema.

El valor central de esta solución radica en:
- **Recursos unificados**: consolidación de la documentación del sistema de diseño disperso
- **Estándares estructurados**: expresar convenciones de diseño en un formato que la IA pueda comprender
- **Mantenimiento continuo**: mantener el contenido actualizado `git submodule`

Si también utiliza IA para el desarrollo frontend, vale la pena probar este enfoque. Creando una estructura `design.md` no solo mejora la coherencia del código generado por IA, sino que también ayuda a su equipo a mantener estándares de diseño unificados internamente.
