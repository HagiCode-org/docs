## Resolviendo los desafíos de autenticación WebSocket del navegador: un patrón de proxy práctico para el reconocimiento de voz de Doubao

> La API WebSocket del navegador no admite encabezados HTTP personalizados, lo que crea un desafío para los servicios de reconocimiento de voz que requieren datos de autenticación en los encabezados. Este artículo comparte cómo resolvimos ese problema en el proyecto HagiCode con un patrón de proxy backend y cómo el enfoque evolucionó desde los experimentos en el patio de juegos hasta el uso en producción.


## Antecedentes

Cuando comenzamos a desarrollar el reconocimiento de voz para el proyecto HagiCode, elegimos con confianza el servicio de reconocimiento de voz Doubao de ByteDance. El diseño inicial fue sencillo: dejar que la interfaz se conectara directamente al servicio WebSocket de Doubao. ¿Qué tan difícil podría ser eso? Simplemente abre una conexión y envía algunos datos, ¿verdad?

Entonces vino la sorpresa: la API de Doubao requiere que la información de autenticación pase a través de encabezados HTTP, incluyendo cosas como `accessToken` y `secretKey`. Esto inmediatamente se volvió incómodo, porque la API WebSocket del navegador simplemente no admite la configuración de encabezados personalizados.

Entonces, ¿qué haces cuando el navegador no te permite enviarlos?

En ese momento, sopesamos dos opciones:

1. Coloque las credenciales en los parámetros de consulta de URL: simple y contundente
2. Agregue una capa de proxy en el backend: más trabajo a primera vista

La primera opción expone las credenciales directamente en el código de interfaz y en el almacenamiento local. ¿Es eso realmente seguro? No me sentía cómodo con eso. Además de eso, algunas API requieren verificación basada en encabezados, por lo que este enfoque ni siquiera es viable.

Al final, elegimos la segunda opción: implementar un proxy WebSocket en el backend. Casualmente, este patrón se validó por primera vez en nuestro entorno de juegos y solo después de que confirmamos su estabilidad lo pasamos a producción. Después de todo, nadie quiere que la producción sirva también como experimento de laboratorio.

## Acerca de HagiCode

El enfoque compartido en este artículo proviene de nuestra experiencia práctica en el [HagiCode](https://github.com/HagiCode-org/site) proyecto.

HagiCode es un proyecto de asistente de codificación de IA con soporte de interacción de voz. Como necesitábamos llamar a un servicio de reconocimiento de voz desde el frontend, nos topamos directamente con este problema de autenticación de WebSocket, que nos llevó a la solución que se describe aquí. A veces, estos obstáculos técnicos son frustrantes, pero también te obligan a aprender patrones que resultan útiles más adelante.

## Análisis de desafíos técnicos

### Limitaciones del WebSocket del navegador

La API WebSocket estándar parece maravillosamente simple:

```javascript
const ws = new WebSocket('wss://example.com/ws');
```

Pero esa simplicidad es exactamente donde radica el problema: solo pasa parámetros en la URL y no puede establecer encabezados como lo hace una solicitud HTTP:

```javascript
// This is not supported in the WebSocket API
const ws = new WebSocket('wss://example.com/ws', {
  headers: {
    'Authorization': 'Bearer token'
  }
});
```

Y esa es la cuestión central. Para servicios como el reconocimiento de voz de Doubao que dependen de la autenticación basada en encabezados, esta limitación es un obstáculo importante.

Una vez que acepta esa restricción, la arquitectura tiene que cambiar.

### Decisiones de diseño arquitectónico.

Al diseñar la solución, comparamos cuidadosamente las ventajas y desventajas.

**Decisión 1: elegir el patrón de proxy**

Comparamos dos enfoques:

| Opción | Ventajas | Contras | decisión |
|------|------|------|------|
| WebSocket nativo | Reenvío ligero, sencillo y directo | La gestión de la conexión debe realizarse manualmente | Elegido |
| SeñalR | Reconexión automática, escritura fuerte | Dependencias adicionales demasiado complejas | Rechazado |

Finalmente elegimos WebSocket nativo. Para ser honesto, era la opción más ligera y se adaptaba mejor al reenvío de flujo binario bidireccional simple. Incorporar SignalR habría parecido una ingeniería excesiva para este caso de uso y podría agregar latencia adicional.

**Decisión 2: estrategia de gestión de la conexión**

Adoptamos un modelo de "una conexión, una sesión": cada conexión WebSocket frontal se asigna a su propia conexión backend Doubao independiente.

Los beneficios son sencillos:
- Fácil de implementar y alineado con el patrón de uso común
- Más fácil de depurar y solucionar problemas
- Buen aislamiento de recursos, evitando interferencias entre sesiones.

En pocas palabras, la solución directa es a veces la mejor. La complejidad no mejora automáticamente un diseño.

**Decisión 3: Almacenamiento de datos de autenticación**

Las credenciales se almacenan en archivos de configuración de backend (`appsettings.yml` o variables de entorno) y cargado mediante inyección de dependencia:

- Modelo de configuración simple que coincide con las convenciones de backend existentes
- Los datos confidenciales nunca llegan al frontend
- Admite la configuración en múltiples entornos para desarrollo, pruebas y producción.

Ese nivel de separación importa. Nadie quiere que las credenciales floten en lugares donde no deberían estar.

### Diseño de flujo de datos

El flujo de datos general se ve así:

```
Frontend (browser)
  │
  │ ws://backend/api/voice/ws
  │ WebSocket (binary)
  ▼
Backend (proxy)
  │
  │ wss://openspeech.bytedance.com/
  │ (with authentication headers)
  ▼
Doubao API
```

El flujo en sí no es complicado:
1. El frontend se conecta al proxy backend a través de WebSocket
2. El proxy backend recibe datos de audio y se conecta a la API de Doubao con encabezados autenticados.
3. La API de Doubao devuelve resultados de reconocimiento y el proxy los reenvía al frontend
4. Todo el proceso permanece totalmente asíncrono con streaming bidireccional.

Una vez que las responsabilidades están claramente divididas, el diseño se vuelve bastante natural.

## Implementación del componente central

### 1. Configuración del punto final WebSocket

```csharp
app.Map("/ws", async context =>
{
    if (context.WebSockets.IsWebSocketRequest)
    {
        // Read configuration from query parameters
        var appId = context.Request.Query["appId"];
        var accessToken = context.Request.Query["accessToken"];

        // Validate required parameters
        if (string.IsNullOrEmpty(appId) || string.IsNullOrEmpty(accessToken))
        {
            context.Response.StatusCode = 400;
            return;
        }

        // Accept the WebSocket connection
        using var webSocket = await context.WebSockets.AcceptWebSocketAsync();

        // Message handling loop
        var buffer = new byte[4096];
        while (!webSocket.CloseStatus.HasValue)
        {
            var result = await webSocket.ReceiveAsync(buffer, CancellationToken.None);

            if (result.MessageType == WebSocketMessageType.Close)
            {
                await webSocket.CloseAsync(
                    result.CloseStatus.Value,
                    result.CloseStatusDescription,
                    CancellationToken.None);
                break;
            }

            // Process audio data
            await HandleAudioDataAsync(buffer, result.Count);
        }
    }
});
```

### 2. Gestión de sesiones

```csharp
public class DoubaoSessionManager : IDoubaoSessionManager
{
    private readonly ConcurrentDictionary<string, DoubaoSession> _sessions = new();

    public DoubaoSession CreateSession(string connectionId)
    {
        var session = new DoubaoSession(connectionId);
        _sessions[connectionId] = session;
        return session;
    }

    public async Task SendAudioAsync(string connectionId, byte[] audioData)
    {
        if (_sessions.TryGetValue(connectionId, out var session))
        {
            await session.SendAudioAsync(audioData);
        }
    }

    public void RemoveSession(string connectionId)
    {
        if (_sessions.TryRemove(connectionId, out var session))
        {
            session.Dispose();
        }
    }
}
```

Usando `ConcurrentDictionary` para la gestión de sesiones significa que la seguridad de los subprocesos la gestionamos en gran medida por nosotros. Cada conexión entrante obtiene su propia sesión y la limpieza se realiza automáticamente al desconectarse.

## Diseño de protocolo de mensajes.

El frontend y el backend utilizan mensajes de texto JSON para el control, mientras que los mensajes binarios transportan datos de audio.

Mensaje de control de ejemplo:

```json
{
    "type": "control",
    "messageId": "msg_123",
    "timestamp": "2026-03-03T10:00:00Z",
    "payload": {
        "command": "StartRecognition",
        "parameters": {
            "hotwordId": "hotword1",
            "boosting_table_id": "table123"
        }
    }
}
```

Resultado de reconocimiento de ejemplo:

```json
{
    "type": "result",
    "timestamp": "2026-03-03T10:00:03Z",
    "payload": {
        "text": "Hello world",
        "confidence": 0.95,
        "duration": 1500,
        "isFinal": true,
        "utterances": [
            {
                "text": "Hello",
                "startTime": 0,
                "endTime": 800,
                "definite": true
            }
        ]
    }
}
```

Este diseño separa las señales de control de las cargas útiles de audio, lo que hace que el sistema sea más fácil de razonar e implementar. Dividir las responsabilidades de manera limpia suele ser la decisión correcta.

## Integración frontend en la práctica

### Conexión WebSocket

```javascript
class DoubaoVoiceClient {
    constructor(config) {
        this.config = config;
        this.ws = null;
    }

    async connect() {
        const url = new URL(this.config.wsUrl);
        // Add query parameters
        Object.entries(this.config.params).forEach(([key, value]) => {
            url.searchParams.set(key, value);
        });

        this.ws = new WebSocket(url);

        return new Promise((resolve, reject) => {
            this.ws.onopen = () => {
                console.log('[DoubaoVoice] Connected');
                resolve();
            };

            this.ws.onmessage = (event) => {
                this._handleMessage(JSON.parse(event.data));
            };

            this.ws.onerror = reject;
        });
    }

    _handleMessage(message) {
        switch (message.type) {
            case 'status':
                this._handleStatus(message.payload);
                break;
            case 'result':
                this.onResult?.(message.payload);
                break;
            case 'error':
                console.error('[DoubaoVoice] Error:', message.payload);
                break;
        }
    }
}

// Usage example
const client = new DoubaoVoiceClient({
    wsUrl: 'ws://localhost:5000/ws',
    params: {
        appId: 'your-app-id',
        accessToken: 'your-access-token',
        sampleRate: 16000,
        bitsPerSample: 16,
        channels: 1
    }
});
```

### Captura y transmisión de audio.

Usando `AudioWorkletNode` para el procesamiento de audio ofrece un mejor rendimiento:

```javascript
// audio-worklet.js
class AudioProcessorWorklet extends AudioWorkletProcessor {
    process(inputs, outputs, parameters) {
        const input = inputs[0]?.[0];
        if (!input) return true;

        // Convert to 16-bit PCM
        const pcm = new Int16Array(input.length);
        for (let i = 0; i < input.length; i++) {
            pcm[i] = Math.max(-32768, Math.min(32767, input[i] * 32767));
        }

        this.port.postMessage({
            type: 'audioData',
            data: pcm.buffer
        }, [pcm.buffer]);

        return true;
    }
}

registerProcessor('audio-processor', AudioProcessorWorklet);

// Main thread code
async function startAudioRecording() {
    const stream = await navigator.mediaDevices.getUserMedia({
        audio: {
            echoCancellation: true,
            noiseSuppression: true,
            autoGainControl: true,
            sampleRate: 48000
        }
    });

    const audioContext = new AudioContext();
    const audioSource = audioContext.createMediaStreamSource(stream);

    await audioContext.audioWorklet.addModule('/audio-worklet.js');
    const audioWorkletNode = new AudioWorkletNode(audioContext, 'audio-processor');

    audioWorkletNode.port.onmessage = (event) => {
        if (event.data.type === 'audioData' && ws?.readyState === WebSocket.OPEN) {
            ws.send(event.data.data); // Send binary data directly
        }
    };

    audioSource.connect(audioWorkletNode);
}
```

`AudioWorklet` funciona mucho mejor que `ScriptProcessorNode` y evita los problemas de tartamudeo del audio que suelen introducir las rutas de procesamiento más antiguas.

## Notas y mejores prácticas

### 1. Monitoreo de conexión

- Registre periódicamente el estado de la sesión para rastrear el ciclo de vida completo de la conexión.
- Supervise el número y la duración de los segmentos de audio para detectar conexiones anormales
- Registre el estado de conexión y el comportamiento de reconexión para el servicio Doubao

Estas son prácticas operativas básicas, pero marcan una diferencia real en la producción.

### 2. Manejo de errores

- Capture y registre todas las excepciones de WebSocket
- uso `IAsyncDisposable` para garantizar que los recursos se limpien
- Implementar un apagado elegante de la conexión y un manejo del tiempo de espera

En resumen, favorecer la robustez.

### 3. Requisitos de formato de audio

- Frecuencia de muestreo: 16000 Hz (recomendado) u 8000 Hz
- Profundidad de bits: 16 bits
- Canales: mono
- Codificación: PCM (sin formato)

Si el formato es incorrecto, el reconocimiento puede fallar o la calidad puede degradarse significativamente. Estos requisitos son importantes.

### 4. Consideraciones de seguridad

- Mantenga las credenciales confidenciales solo en la configuración de backend
- Hacer cumplir los límites de conexión para evitar el agotamiento de los recursos
- Utilice HTTPS/WSS en producción

La seguridad nunca es una preocupación menor.

### 5. Optimización del rendimiento

- Utilice operaciones asincrónicas para evitar el bloqueo.
- Ajuste los tamaños de búfer según sea necesario (predeterminado: 4096 bytes)
- Considere estrategias de agrupación y reutilización de conexiones

Aplique estas optimizaciones cuando tengan sentido para su carga de trabajo.

## Recomendaciones de implementación

1. **Implementación de Docker**: empaquete el servicio de proxy como un contenedor para facilitar el escalado y la administración.
2. **Equilibrio de carga**: use Nginx o Envoy como proxy inverso para el tráfico de WebSocket
3. **Controles de estado**: implemente controles basados en latidos para monitorear la disponibilidad del servicio
4. **Agregación de registros**: envíe registros a un sistema de registro centralizado como ELK o Loki

La implementación puede ser simple o compleja según el equipo y el entorno, así que ajústela en consecuencia.

## Resumen

El patrón de proxy WebSocket resuelve el problema principal de que la API WebSocket del navegador no admite encabezados personalizados. En el proyecto HagiCode, este patrón demostró ser factible y estable a medida que pasaba de la validación en el campo de juego a la implementación en producción.

Conclusiones clave:
- Un proxy backend puede pasar información de autenticación de forma segura
- Native WebSocket es liviano y eficiente para escenarios simples
- El modelo "una conexión, una sesión" simplifica tanto la implementación como la depuración
- El protocolo frontend-backend debe separar las señales de control de los datos de audio.

Si está creando una función que también depende de la autenticación WebSocket, espero que este patrón le brinde un punto de partida útil.

Si tiene preguntas, no dude en discutirlas con nosotros. El progreso técnico ocurre más rápido cuando la gente compara notas.
