## Edición rápida de código en el navegador: integración web VSCode en la práctica

> Una vez que la IA termina de analizar el código, ¿cómo se abre inmediatamente un editor en el navegador y se empieza a realizar cambios? Este artículo comparte nuestra experiencia práctica en la integración del servidor de códigos en el proyecto HagiCode para crear un puente perfecto entre el asistente de IA y la experiencia de edición de código.

## Antecedentes

En la era de la programación asistida por IA, los desarrolladores suelen necesitar inspeccionar y editar el código rápidamente. El flujo de trabajo tradicional es simple: abra el proyecto en un IDE de escritorio, ubique el archivo, edítelo y guárdelo. Pero en algunas situaciones, ese flujo siempre se siente un poco fuera de lugar.

Escenario uno: desarrollo remoto. Cuando se utiliza un asistente de IA como HagiCode, el backend puede estar ejecutándose en un servidor remoto o dentro de un contenedor, y las máquinas locales no pueden acceder directamente a los archivos del proyecto. Cada vez que necesitas inspeccionar o modificar código, debes conectarte a través de SSH u otro método, y la experiencia se siente fragmentada. Es como querer conocer a alguien a través de un grueso cristal: puedes verlo, pero no puedes alcanzarlo.

Escenario dos: vistas previas rápidas. Después de que el asistente de IA analiza el código, es posible que el usuario solo desee explorar rápidamente un archivo o realizar un pequeño cambio. Iniciar un IDE de escritorio completo parece pesado, mientras que un editor liviano en el navegador se adapta mejor a la necesidad de una "vista rápida". Después de todo, ¿quién quiere movilizar toda una cadena de herramientas sólo para echar un vistazo?

Escenario tres: colaboración entre dispositivos. Cuando se trabaja en diferentes dispositivos, un editor basado en navegador proporciona un punto de acceso unificado sin necesidad de configurar cada máquina con un entorno de desarrollo. Sólo eso ahorra muchos problemas. La vida es corta; ¿Por qué repetir el mismo trabajo de configuración una y otra vez?

Para resolver estos puntos débiles, integramos VSCode Web en el proyecto HagiCode. Esto permite que el asistente de IA y la experiencia de edición de código se conecten sin problemas: después de que la IA analiza el código, los usuarios pueden abrir inmediatamente un editor y realizar cambios en la misma sesión del navegador, sin cambiar de aplicación. Es el tipo de experiencia que, cuando la necesitas, simplemente está ahí.

## Acerca de HagiCode

La solución compartida en este artículo proviene de nuestra experiencia práctica en el [HagiCode](https://hagicode.com) proyecto. HagiCode es un asistente de codificación impulsado por IA diseñado para mejorar la eficiencia del desarrollo a través de la interacción del lenguaje natural. Durante el desarrollo, descubrimos que los usuarios a menudo necesitan cambiar rápidamente entre el análisis de IA y la edición de código, lo que nos impulsó a explorar cómo integrar el editor directamente en el navegador.

Repositorio de proyectos: [github.com/HagiCode-org/site](https://github.com/HagiCode-org/site)

## Elección técnica: ¿Por qué servidor de códigos?

Entre las muchas soluciones web VSCode disponibles, elegimos el servidor de códigos. Había algunas razones concretas detrás de esa decisión.

Completitud de funciones. code-server es la versión web de VSCode y admite la mayoría de las funciones de escritorio, incluido el sistema de extensión, sugerencias inteligentes, depuración y más. Eso significa que los usuarios pueden obtener una experiencia de edición en el navegador muy parecida a la versión de escritorio. Después de todo, ¿quién quiere realmente renunciar a la funcionalidad?

Implementación flexible. code-server puede ejecutarse como un servicio independiente y también admite la implementación basada en Docker, que encaja bien con la arquitectura de HagiCode. Nuestro backend está escrito en C#, el frontend usa React y los dos se comunican con el servicio del servidor de códigos a través de API REST. Es como construir con bloques: cada pieza tiene su lugar.

Autenticación segura. El servidor de códigos incluye un `connection-token` Mecanismo para impedir el acceso no autorizado. Cada sesión tiene un token único para que sólo los usuarios autorizados puedan abrir el editor. La seguridad es una de esas cosas que sólo se aprecia plenamente una vez que se tiene.

## Diseño de Arquitectura

La integración web VSCode de HagiCode utiliza una arquitectura separada de front-end/back-end.

### Capa de servicio frontend

El frontend envuelve las interacciones con el backend a través de `vscodeServerService.ts`:

```typescript
// Open project
export async function openProjectInCodeServer(
  id: string,
  currentInterfaceLanguage?: string,
): Promise<VsCodeServerLaunchResponseDto>

// Open vault
export async function openVaultInCodeServer(
  id: string,
  path?: string,
  currentInterfaceLanguage?: string,
): Promise<VsCodeServerLaunchResponseDto>
```

La diferencia entre estos dos métodos es sencilla: `openProjectInCodeServer` abre todo el proyecto, mientras `openVaultInCodeServer` abre una ruta específica dentro de una bóveda. Para proyectos de repositorios múltiples de MonoSpecs, el sistema crea automáticamente un archivo de espacio de trabajo. Las responsabilidades claras suelen ser suficientes cuando cada parte hace bien su trabajo.

### Capa de servicio backend

El backend `VaultAppService.cs` implementa la lógica central:

```csharp
public async Task<VsCodeServerLaunchResponseDto> OpenInCodeServerAsync(
    string id,
    string? relativePath = null,
    string? currentInterfaceLanguage = null,
    CancellationToken cancellationToken = default)
{
    // 1. Get settings and check whether the feature is enabled
    var settings = await _vsCodeServerSettingsService.GetResolvedSettingsAsync(cancellationToken);
    if (!settings.Enabled) {
        throw new BusinessException(VsCodeServerErrorCodes.Disabled, "VSCode Server is disabled.");
    }

    // 2. Get vault and resolve the launch directory
    var vault = await RequireVaultAsync(id, cancellationToken);
    var launchDirectory = ResolveLaunchDirectory(vault, relativePath);

    // 3. Ensure code-server is running and get runtime info
    var runtime = await _vsCodeServerManager.EnsureStartedAsync(settings, cancellationToken);

    // 4. Resolve language settings
    var language = _vsCodeServerSettingsService.ResolveLaunchLanguage(
        settings.Language,
        currentInterfaceLanguage);

    // 5. Build launch URL
    return new VsCodeServerLaunchResponseDto {
        LaunchUrl = AppendQueryString(runtime.BaseUrl, new Dictionary<string, string?> {
            ["folder"] = launchDirectory,
            ["tkn"] = runtime.ConnectionToken,
            ["vscode-lang"] = language,
        }),
        ConnectionToken = runtime.ConnectionToken,
        OpenMode = "folder",
        Runtime = VsCodeServerSettingsService.MapRuntime(
            await _vsCodeServerManager.GetRuntimeSnapshotAsync(cancellationToken)),
    };
}
```

Este método tiene una responsabilidad muy clara: verificar la configuración, resolver rutas, iniciar el servicio y crear la URL. Entre ellos, el `ResolveLaunchDirectory` El método realiza comprobaciones de seguridad de rutas para evitar ataques de cruce de rutas. El código puede parecer un poco poesía cuando cada línea tiene un propósito.

### Gestión automática del tiempo de ejecución

El backend gestiona el proceso del servidor de códigos a través de `VsCodeServerManager`:

- Consultar estado del proceso
- Iniciar automáticamente servicios detenidos
- Devolver instantáneas del tiempo de ejecución, como puerto, ID de proceso y hora de inicio

Este diseño permite que el sistema maneje automáticamente el ciclo de vida del servidor de códigos, por lo que los usuarios no necesitan administrar los procesos de servicio manualmente. La vida ya es bastante complicada; todo lo que pueda automatizarse debería serlo.

### Sincronización de idiomas

HagiCode admite una interfaz multilingüe y el servidor de códigos debe seguir esa configuración. El sistema admite tres modos de idioma:

- `follow`: sigue el idioma actual de la interfaz
- `zh-CN`: arreglado en chino
- `en-US`: arreglado al inglés

La configuración se pasa al servidor de códigos a través del `vscode-lang` Parámetro de URL para que el idioma del editor se mantenga coherente con la interfaz de HagiCode. El lenguaje se siente mejor cuando está unificado.

### Espacio de trabajo de repositorios múltiples de MonoSpecs

Para proyectos MonoSpecs, que contienen múltiples subrepositorios dentro de un monorepo, el sistema crea automáticamente un `.code-workspace` archivo:

```csharp
private async Task<string> CreateWorkspaceFileAsync(Project project, Guid projectId)
{
    var folders = await ResolveWorkspaceFoldersAsync(project.Path);
    var workspaceDocument = new {
        folders = folders.Select(path => new { path }).ToArray(),
    };
    // Generate workspace file...
}
```

Esto hace posible editar múltiples subrepositorios en la misma instancia del servidor de códigos, lo cual es especialmente práctico para grandes proyectos monorepo. Múltiples repositorios en una ventana pueden parecer múltiples historias reunidas en el mismo libro.

## Integración frontal

La interfaz de HagiCode utiliza React + TypeScript y la integración del servidor de códigos no es especialmente complicada.

### Botón de acción rápida

Agregue un botón de Code Server a la tarjeta del proyecto:

```tsx
// QuickActionsZone.tsx
<Button
  size="sm"
  variant="default"
  onClick={() => onAction({ type: 'open-code-server' })}
>
  <Globe className="h-3 w-3 mr-1" />
  <span className="text-xs">{t('project.openCodeServer')}</span>
</Button>
```

Este botón activa la acción de apertura y llama a la API de backend para obtener la URL de inicio. Un botón, una acción, directa y sencilla.

### Manejo de la acción abierta

```tsx
const handleAction = async (action: ProjectAction) => {
  if (action.type === 'open-code-server') {
    const response = await openProjectInCodeServer(project.id, i18n.language);
    window.open(response.launchUrl, '_blank', 'noopener,noreferrer');
  }
};
```

uso `window.open` para abrir el servidor de códigos en una nueva pestaña. el `noopener,noreferrer` Los parámetros proporcionan seguridad adicional. Cuando se trata de seguridad, nunca se es demasiado cuidadoso.

### Entrada de edición de bóveda

Agregue un botón de edición similar en la lista de Vault:

```tsx
const handleEditVault = async (vault: VaultItemDto) => {
  const response = await openVaultInCodeServer(vault.id);
  window.open(response.launchUrl, '_blank', 'noopener,noreferrer');
};
```

Los proyectos y las bóvedas utilizan el mismo mecanismo abierto, lo que mantiene la interacción coherente. La coherencia importa casi tanto como la característica misma.

## Lógica de construcción de URL

El formato de URL para el servidor de códigos tiene algunos detalles que vale la pena mencionar.

**Modo carpeta**:
```
http://{host}:{port}/?folder={path}&tkn={token}&vscode-lang={lang}
```

**Modo de espacio de trabajo**:
```
http://{host}:{port}/?workspace={workspacePath}&tkn={token}&vscode-lang={lang}
```

Aquí, `tkn` es el token de conexión. Se genera automáticamente cada vez que se inicia el servidor de códigos, lo que garantiza un acceso seguro. el `vscode-lang` El parámetro controla el idioma de la interfaz de usuario del editor. Cada uno de estos parámetros tiene un papel que desempeñar.

## Escenarios de uso

### Escenario uno: revisión de código asistida por IA

El usuario habla con HagiCode, la IA analiza el código del proyecto y encuentra un problema potencial, y luego el usuario hace clic en el botón "Abrir en Code Server" para abrir el editor directamente en el navegador, inspeccionar el archivo afectado, solucionarlo y regresar a HagiCode para continuar la conversación. Todo el flujo ocurre en el navegador sin cambiar de aplicación. Se siente suave como se siente el agua corriente.

### Escenario dos: edición de materiales de estudio en una bóveda

Un usuario crea una Bóveda para estudiar un proyecto de código abierto y desea agregar notas de estudio bajo el `docs/` directorio. Con el servidor de códigos, pueden editar archivos Markdown directamente en el navegador, guardarlos y dejar que HagiCode lea inmediatamente las notas actualizadas. Esto es especialmente útil para construir una base de conocimientos personales. El conocimiento sólo se vuelve más valioso cuanto más lo acumulas.

### Escenario tres: desarrollo de repositorios múltiples de MonoSpecs

Un proyecto MonoSpecs contiene varios subrepositorios y el servidor de códigos crea automáticamente un espacio de trabajo de varias carpetas. En el navegador, los usuarios pueden editar código en varios repositorios a la vez y luego confirmar los cambios en sus respectivos repositorios Git. Este flujo de trabajo es particularmente adecuado para cambios que deben abarcar varios repositorios. Editar varios repositorios juntos requiere un poco de técnica, al igual que manejar múltiples tareas al mismo tiempo.

## Consideraciones de seguridad

Al implementar la integración del servidor de códigos, la seguridad merece una atención especial. Si la seguridad falla, siempre te das cuenta demasiado tarde.

### Token de conexión

el `connection-token` se genera aleatoriamente y no debe exponerse. Se utiliza mejor en HTTPS para evitar que un intermediario intercepte el token. Vale la pena proteger adecuadamente la información confidencial.

### Seguridad de la ruta del archivo

El backend implementa comprobaciones de recorrido de ruta:

```csharp
private static string ResolveLaunchDirectory(VaultRegistryEntry vault, string? relativePath)
{
    var vaultRoot = EnsureTrailingSeparator(Path.GetFullPath(vault.PhysicalPath));
    var combinedPath = Path.GetFullPath(Path.Combine(vaultRoot, relativePath ?? "."));
    if (!combinedPath.StartsWith(vaultRoot, StringComparison.OrdinalIgnoreCase))
    {
        throw new BusinessException(VaultRelativePathTraversalCode, "Relative path traversal detected.");
    }
    return combinedPath;
}
```

Este código garantiza que los usuarios no puedan utilizar `../` o patrones similares para acceder a archivos fuera del directorio de Vault. Siempre es mejor realizar comprobaciones de límites que omitirlas.

### Control de permisos

El proceso del servidor de códigos debe ejecutarse con los permisos de usuario adecuados para que no pueda acceder a archivos confidenciales del sistema. Es mejor ejecutar el servicio del servidor de códigos con un usuario dedicado. El control de permisos es uno de esos fundamentos que siempre debes mantener.

## Optimización del rendimiento

El servidor de códigos consume recursos del servidor, por lo que aquí hay algunas sugerencias de optimización:

- Supervise el uso de CPU y memoria, y ajuste los límites de recursos cuando sea necesario
- Los proyectos grandes pueden requerir tiempos de espera más prolongados
- Implementar la limpieza automática del tiempo de espera de la sesión para liberar recursos
- Considere el almacenamiento en caché para reducir el cálculo repetido

HagiCode proporciona una API de monitoreo del estado del tiempo de ejecución y la interfaz puede llamar `getVsCodeServerSettings()` para recuperar el estado actual:

```typescript
const { settings, runtime } = await getVsCodeServerSettings();
// runtime.status: 'disabled' | 'stopped' | 'starting' | 'running' | 'unhealthy'
// runtime.baseUrl: "http://localhost:8080"
// runtime.processId: 12345
```

Este diseño permite a los usuarios comprender claramente el estado del servidor de códigos y localizar rápidamente los problemas cuando algo sale mal. Cuando el estatus es visible, la gente se siente más en control.

## Detalles de la experiencia del usuario

Durante la implementación, descubrimos algunos detalles que afectan notablemente la experiencia del usuario y merecen atención especial.

Abrir el servidor de códigos por primera vez puede requerir esperar hasta el inicio, y ese retraso puede variar desde unos pocos segundos hasta medio minuto. Es una buena idea mostrar un estado de carga en la interfaz para que los usuarios sepan que el sistema todavía está funcionando. Esperar es más fácil cuando hay retroalimentación.

Los navegadores pueden bloquear la ventana emergente, por lo que se debe solicitar a los usuarios que la permitan manualmente. En el primer inicio, HagiCode muestra una guía que explica cómo otorgar los permisos necesarios del navegador. La experiencia del usuario a menudo reside exactamente en estos pequeños detalles.

También es una buena idea mostrar el estado del tiempo de ejecución, como inicio, ejecución o error, de modo que cuando ocurran problemas, los usuarios puedan saber rápidamente si el problema está en el lado del servidor o en su propia operación. Saber dónde está el problema al menos le dará un punto de partida.

## Ejemplo de configuración

La configuración del servidor de códigos no es complicada:

```json
{
  "vscodeServer": {
    "enabled": true,
    "host": "0.0.0.0",
    "port": 8080,
    "language": "follow"
  }
}
```

`enabled` controla si la función está activada, `host` y `port` definir la dirección de escucha, y `language` establece el modo de idioma. Estas configuraciones se pueden modificar a través de la interfaz de usuario y surten efecto inmediatamente. Las cosas simples suelen ser las más fáciles de usar.

## Conclusión

La integración web VSCode de HagiCode proporciona una solución elegante: permite que el asistente de IA y la experiencia de edición de código se conecten sin problemas. Al integrar el servidor de códigos en el navegador, los usuarios pueden actuar rápidamente sobre los resultados del análisis de IA y completar todo el flujo desde el análisis hasta la edición en la misma sesión del navegador.

Esta solución aporta varias ventajas clave: una experiencia unificada, porque los proyectos y Vaults utilizan el mismo mecanismo abierto; soporte para múltiples repositorios, porque los proyectos MonoSpecs crean automáticamente espacios de trabajo; y seguridad controlable, gracias a la supervisión del estado del tiempo de ejecución y a las comprobaciones de seguridad de las rutas.
