## De llamadas CLI a integración de SDK: mejores prácticas para GitHub Copilot en proyectos .NET

> El camino de actualización desde la invocación de la línea de comandos hasta la integración oficial del SDK ha sido todo un viaje. Hoy quiero compartir los obstáculos con los que nos topamos y lo que aprendimos mientras construíamos HagiCode.

## Antecedentes

Después de que el SDK de GitHub Copilot se lanzara oficialmente en 2025, comenzamos a integrarlo en nuestra capa de capacidad de IA. Antes de eso, el proyecto utilizaba principalmente GitHub Copilot invocando directamente la herramienta de línea de comandos Copilot CLI, pero ese enfoque tenía varios problemas obvios:

- **Gestión de procesos complejos**: Tuvimos que administrar manualmente el ciclo de vida del proceso CLI, los tiempos de espera de inicio y la limpieza del proceso. Los procesos pueden fallar sin previo aviso.
- **Manejo de eventos incompleto**: la invocación de CLI sin procesar dificulta la captura de eventos detallados del razonamiento del modelo y la ejecución de la herramienta. Es como ver sólo el resultado sin el proceso de pensamiento.
- **Gestión de sesiones difícil**: no existía un mecanismo eficaz para la reutilización y recuperación de sesiones, por lo que cada interacción tenía que empezar de nuevo.
- **Problemas de compatibilidad**: los argumentos de la CLI cambiaban con frecuencia, lo que significaba que teníamos que mantener constantemente la lógica de compatibilidad para esos parámetros.

Estos problemas se hicieron cada vez más evidentes en el desarrollo diario, especialmente cuando necesitábamos realizar un seguimiento del razonamiento del modelo (`thinking`) y estado de ejecución de la herramienta en tiempo real. En ese momento, quedó claro que necesitábamos un enfoque de integración más completo y de menor nivel.

## Acerca de HagiCode

El enfoque compartido en este artículo proviene de nuestra experiencia práctica en el [HagiCode](https://hagicode.com) proyecto. HagiCode es un proyecto de asistente de codificación de IA de código abierto y, durante el desarrollo, necesitábamos una integración profunda con las capacidades de GitHub Copilot, desde la finalización de código básico hasta conversaciones complejas de varios turnos y llamadas de herramientas. Esos requisitos del mundo real nos empujaron a pasar de la invocación CLI al SDK oficial.

Si esta implementación le parece útil, es muy probable que nuestra experiencia en ingeniería pueda ayudarle. En ese caso, también vale la pena echarle un vistazo al proyecto HagiCode. Incluso puede encontrar más información sobre el proyecto y enlaces al final de este artículo.

## Diseño de Arquitectura

El proyecto utiliza una arquitectura en capas para abordar las limitaciones de la invocación de CLI:

```
┌─────────────────────────────────────────────────────────┐
│  hagicode-core (Orleans Grains + AI Provider Layer)    │
│  - CopilotAIProvider: Converts AIRequest to CopilotOptions │
│  - GitHubCopilotGrain: Orleans distributed execution interface │
└─────────────────────────────────────────────────────────┘
                          ↓
┌─────────────────────────────────────────────────────────┐
│  HagiCode.Libs (Shared Provider Layer)                 │
│  - CopilotProvider: CLI Provider interface implementation │
│  - ICopilotSdkGateway: SDK invocation abstraction      │
│  - GitHubCopilotSdkGateway: SDK session management and event dispatch │
└─────────────────────────────────────────────────────────┘
                          ↓
┌─────────────────────────────────────────────────────────┐
│  GitHub Copilot SDK (Official .NET SDK)                │
│  - CopilotClient: SDK client                           │
│  - CopilotSession: Session management                  │
│  - SessionEvent: Event stream                          │
└─────────────────────────────────────────────────────────┘
```

Este diseño en capas aporta varias ventajas técnicas prácticas:

1. **Separación de preocupaciones**: la lógica empresarial central está desacoplada de los detalles de implementación del SDK.
2. **Comprobabilidad**: La `ICopilotSdkGateway` La interfaz hace que las pruebas unitarias sean sencillas.
3. **Reutilizabilidad**: `HagiCode.Libs` Puede ser referenciado por múltiples proyectos.
4. **Mantenibilidad**: las actualizaciones del SDK solo requieren cambios en la capa de puerta de enlace, mientras que las capas superiores permanecen intactas.

## Implementación central

### Flujo de autenticación

La autenticación es el primer y más importante paso de la integración del SDK. Si la autenticación falla, nada más importa. Diseñamos una configuración de autenticación flexible que admite múltiples fuentes de autenticación:

```csharp
// CopilotProvider.cs - Authentication source configuration
public class CopilotOptions
{
    public bool UseLoggedInUser { get; set; } = true;
    public string? GitHubToken { get; set; }
    public string? CliUrl { get; set; }
}

// Convert to SDK request
return new CopilotSdkRequest(
    GitHubToken: options.AuthSource == CopilotAuthSource.GitHubToken
        ? options.GitHubToken
        : null,
    UseLoggedInUser: options.AuthSource != CopilotAuthSource.GitHubToken
);
```

Los beneficios de este diseño son bastante claros:

- Admite el modo de usuario conectado sin necesidad de un token, lo que se adapta bien a escenarios de escritorio.
- Admite el modo GitHub Token, que es adecuado para implementaciones del lado del servidor y administración centralizada.
- Admite la anulación de la URL de la CLI de Copilot, lo que ayuda con la configuración del proxy empresarial.

En la práctica, este modelo de autenticación flexible simplificó enormemente la configuración en diferentes escenarios de implementación. El cliente de escritorio puede utilizar el estado de inicio de sesión de Copilot de cada usuario, mientras que el lado del servidor puede gestionar la autenticación de forma centralizada a través de tokens.

### Manejo de flujo de eventos

Una de las capacidades más potentes del SDK es la captura completa de flujos de eventos. Implementamos un sistema de despacho de eventos que puede manejar todo tipo de eventos del SDK en tiempo real:

```csharp
// GitHubCopilotSdkGateway.cs - Core logic for event dispatch
internal static SessionEventDispatchResult DispatchSessionEvent(
    SessionEvent evt, bool sawDelta)
{
    switch (evt)
    {
        case AssistantReasoningEvent reasoningEvent:
            // Capture the model reasoning process
            events.Add(new CopilotSdkStreamEvent(
                CopilotSdkStreamEventType.ReasoningDelta,
                Content: reasoningEvent.Data.Content));
            break;

        case ToolExecutionStartEvent toolStartEvent:
            // Capture the start of a tool invocation
            events.Add(new CopilotSdkStreamEvent(
                CopilotSdkStreamEventType.ToolExecutionStart,
                ToolName: toolStartEvent.Data.ToolName,
                ToolCallId: toolStartEvent.Data.ToolCallId));
            break;

        case ToolExecutionCompleteEvent toolCompleteEvent:
            // Capture tool completion and its result
            events.Add(new CopilotSdkStreamEvent(
                CopilotSdkStreamEventType.ToolExecutionEnd,
                Content: ExtractToolExecutionContent(toolCompleteEvent)));
            break;

        default:
            // Preserve unhandled events as RawEvent
            events.Add(new CopilotSdkStreamEvent(
                CopilotSdkStreamEventType.RawEvent,
                RawEventType: evt.GetType().Name));
            break;
    }
}
```

El valor de esta implementación es significativo:

- **Captura completa del proceso de razonamiento del modelo** (`thinking`): los usuarios pueden ver cómo razona la IA, no solo la respuesta final.
- **Seguimiento en tiempo real del estado de ejecución de las herramientas**: sabemos qué herramientas se están ejecutando, cuándo finalizan y qué devuelven.
- **Pérdida de evento cero**: al volver a `RawEvent`, nos aseguramos de que cada evento quede registrado.

En HagiCode, estos eventos detallados ayudan a los usuarios a comprender cómo funciona internamente la IA, especialmente al depurar tareas complejas.

### Manejo de compatibilidad CLI

Después de migrar de la invocación de CLI al SDK, descubrimos que algunos parámetros de CLI existentes ya no se aplicaban en el SDK. Para preservar la compatibilidad con versiones anteriores, implementamos un sistema de filtrado de parámetros:

```csharp
public static CopilotCliArgumentBuildResult BuildCliArgs(CopilotOptions options)
{
    // Filter out unsupported arguments and keep compatible ones
    // Generate diagnostic information
}
```

Esto nos aporta varios beneficios:

- Filtra automáticamente argumentos CLI incompatibles para evitar errores de tiempo de ejecución.
- Genera mensajes de diagnóstico claros para ayudar a los desarrolladores a localizar problemas rápidamente.
- Mantiene el SDK estable y aislado de cambios en los parámetros CLI.

Durante la actualización, este mecanismo de compatibilidad nos ayudó a realizar la transición sin problemas. Los archivos de configuración existentes aún se podían usar y solo necesitaban ajustes graduales según el resultado del diagnóstico.

### Agrupación de tiempo de ejecución

Crear sesiones de Copilot SDK es relativamente costoso y crear y destruir sesiones con demasiada frecuencia perjudica el rendimiento. Para solucionarlo, implementamos un sistema de gestión de grupos de sesiones:

```csharp
// CopilotProvider.cs - Session pool management
await using var lease = await _poolCoordinator.AcquireCopilotRuntimeAsync(
    request,
    async ct => await _gateway.CreateRuntimeAsync(sdkRequest, ct),
    cancellationToken);

if (lease.IsWarmLease)
{
    // Reuse an existing session
    yield return CreateSessionReusedMessage();
}

await foreach (var eventData in lease.Entry.Resource.SendPromptAsync(...))
{
    yield return MapEvent(eventData);
}
```

Los beneficios de la agrupación de sesiones incluyen:

- **Reutilización de sesión**: solicitudes con la misma `sessionId` Puede reutilizar sesiones existentes y reducir la sobrecarga de inicio.
- **Soporte de recuperación de sesión**: después de una interrupción de la red, se puede restaurar el estado de la sesión anterior.
- **Gestión automática de agrupaciones**: las sesiones caducadas se limpian automáticamente para evitar fugas de recursos.

En HagiCode, la agrupación de sesiones mejoró notablemente la capacidad de respuesta, especialmente para conversaciones continuas.

### Integración de Orleans

HagiCode utiliza Orleans como marco distribuido e integramos el SDK de Copilot en Orleans Grains:

```csharp
// GitHubCopilotGrain.cs - Distributed execution
public async IAsyncEnumerable<GitHubCopilotResponse> ExecuteCommandStreamAsync(
    string command,
    CancellationToken token = default)
{
    var provider = await aiProviderFactory.GetProviderAsync(AIProviderType.GitHubCopilot);

    await foreach (var chunk in provider.SendMessageAsync(request, null, token))
    {
        // Map to the unified response format
        yield return BuildChunkResponse(chunk, startedAt);
    }
}
```

Las ventajas de la integración de Orleans son sustanciales:

- **Abstracción del proveedor de IA unificado**: resulta fácil cambiar entre diferentes proveedores de IA.
- **Aislamiento multiinquilino**: las sesiones de Copilot para diferentes usuarios permanecen aisladas entre sí.
- **Estado de sesión persistente**: el estado de la sesión se puede restaurar incluso después de reiniciar el servidor.

Para escenarios que necesitan manejar una gran cantidad de solicitudes simultáneas, Orleans proporciona una gran escalabilidad.

### Notas de uso

En el uso en el mundo real, resumimos varios puntos a los que vale la pena prestar atención:

**Configuración del tiempo de espera de inicio**: el primer inicio de Copilot CLI puede tardar relativamente mucho tiempo, por lo que recomendamos configurar `StartupTimeout` a al menos 30 segundos. Si este es el primer inicio de sesión, puede llevar incluso más tiempo.

**Gestión de permisos**: En entornos de producción, evite utilizar `AllowAllTools: true`. Utilice el `AllowedTools` lista de permitidos para controlar qué herramientas están disponibles y utilizar la `DeniedTools` lista de denegación para bloquear operaciones peligrosas. Esto evita efectivamente que la IA ejecute comandos riesgosos.

**Gestión de sesiones**: Solicitudes con la misma `sessionId` reutilizar sesiones automáticamente. El estado de la sesión persiste durante `ProviderSessionId`. La cancelación se propaga a través de `CancellationTokenSource`.

**Resultado del diagnóstico**: Los argumentos CLI incompatibles generan mensajes de tipo `diagnostic`. Los eventos del SDK sin formato se conservan como `event.raw`. Los mensajes de error incluyen categorías como tiempo de espera de inicio e incompatibilidad de argumentos para facilitar la resolución de problemas.

### Mejores prácticas

Según nuestra experiencia práctica, a continuación presentamos algunas de las mejores prácticas:

**1. Utilice una lista de herramientas permitidas**

```csharp
var request = new AIRequest
{
    Prompt = "Analyze this file",
    AllowedTools = new[] { "Read", "Grep", "Bash(git:*)" }
};
```

Especificar explícitamente las herramientas permitidas a través de una lista de permitidos ayuda a prevenir acciones inesperadas de la IA. Esto es especialmente importante para herramientas con permisos de escritura, como `Edit`, que debe manejarse con especial cuidado.

**2. Establecer tiempos de espera razonables**

```csharp
options.Timeout = 3600;  // 1 hour
options.StartupTimeout = 60;  // 1 minute
```

Establezca valores de tiempo de espera adecuados según la complejidad de la tarea. Si el valor es demasiado corto, las tareas pueden interrumpirse. Si es demasiado largo, se pueden desperdiciar recursos esperando solicitudes que no responden.

**3. Habilitar la reutilización de sesiones**

```csharp
options.SessionId = "my-session-123";
```

usando lo mismo `sessionId` para tareas relacionadas le permite reutilizar el contexto de la sesión anterior y mejorar la velocidad de respuesta.

**4. Manejar las respuestas en streaming**

```csharp
await foreach (var chunk in provider.StreamAsync(request))
{
    switch (chunk.Type)
    {
        case StreamingChunkType.ThinkingDelta:
            // Handle the reasoning process
            break;
        case StreamingChunkType.ToolCallDelta:
            // Handle tool invocation
            break;
        case StreamingChunkType.ContentDelta:
            // Handle text output
            break;
    }
}
```

Las respuestas en streaming le permiten mostrar el progreso del procesamiento de la IA en tiempo real, lo que mejora la experiencia del usuario. Esto es especialmente valioso para tareas de larga duración.

**5. Manejo de errores y reintentos**

```csharp
try
{
    await foreach (var chunk in provider.StreamAsync(request))
    {
        // Handle the response
    }
}
catch (CopilotSessionException ex)
{
    // Handle session exceptions
    logger.LogError(ex, "Copilot session failed");
    // Decide whether to retry based on the exception type
}
```

El manejo adecuado de errores y la lógica de reintento mejoran la estabilidad general del sistema.

## Conclusión

La actualización de la invocación CLI a la integración SDK aportó un valor sustancial al proyecto HagiCode:

- **Estabilidad mejorada**: el SDK proporciona una interfaz más estable que no se ve afectada por los cambios de versión de CLI.
- **Funcionalidad más completa**: captura el flujo completo de eventos, incluido el razonamiento y el estado de ejecución de la herramienta.
- **Mayor eficiencia de desarrollo**: la interfaz SDK con seguridad de tipos hace que el desarrollo sea más eficiente y reduce los errores de tiempo de ejecución.
- **Mejor experiencia de usuario**: la retroalimentación de eventos en tiempo real brinda a los usuarios una comprensión más clara de lo que está haciendo la IA.

Esta actualización no fue sólo un reemplazo de la implementación técnica. También fue una optimización arquitectónica de toda la capa de capacidad de IA. A través del diseño en capas e interfaces abstractas, obtuvimos una mejor mantenibilidad y extensibilidad.

Si está considerando integrar GitHub Copilot en su propio proyecto .NET, espero que estas lecciones prácticas le ayuden a evitar desvíos innecesarios. De hecho, el SDK oficial es más estable y completo que la invocación CLI, y vale la pena dedicar tiempo a comprenderlo y adoptarlo.
