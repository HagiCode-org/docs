## Protección de código .NET en la práctica: de la ofuscación a la protección de máquinas virtuales

> Este artículo explica cómo implementar una estrategia de protección de código de múltiples capas en proyectos .NET, cubriendo el camino completo desde la ofuscación básica hasta la protección profesional de máquinas virtuales.

## Antecedentes

En el desarrollo de aplicaciones .NET, proteger el código central, como la validación de licencias, la lógica empresarial y la configuración confidencial, contra la descompilación y la ingeniería inversa es, francamente, un tema que no se puede evitar. A medida que el ecosistema .NET ha madurado, los desarrolladores han obtenido acceso a una variedad de opciones de protección, desde atributos de ofuscación integrados hasta herramientas de protección profesionales basadas en virtualización.

Como proyecto monorepo multilingüe complejo, HagiCode incluye aplicaciones de escritorio, sistemas de compilación y capacidades de administración de licencias. El código contiene inevitablemente lógica de validación de licencias, configuraciones confidenciales, como claves API e ID de productos, y lógica crítica para el negocio. Esas partes necesitan una protección seria, porque nadie quiere que su arduo trabajo quede expuesto tan fácilmente.

Este artículo comparte el enfoque de protección de código que adoptamos en el proyecto HagiCode y resume el recorrido completo desde los primeros problemas hasta la optimización posterior. Esperemos que te dé algunas ideas útiles.

## Acerca de HagiCode

[HagiCode](https://hagicode.com) es un proyecto de asistente de codificación de IA de código abierto dedicado a brindar a los desarrolladores una experiencia de programación inteligente. El proyecto utiliza una arquitectura monorepo y simultáneamente mantiene una extensión VSCode, servicios backend de IA, un cliente de escritorio multiplataforma y más. Esa complejidad multilingüe y multiplataforma hace que la protección del código sea un desafío de ingeniería que debemos afrontar de frente.

El enfoque compartido en este artículo es el resultado de prueba y error real durante el desarrollo de HagiCode. Si quieres ver cómo solucionamos estos problemas técnicos, sigue leyendo. Es posible que encuentre algunas conclusiones inesperadas.

## Contenido principal

### 1. Atributo de ofuscación integrado de Microsoft

.NET Framework proporciona una función integrada `[ObfuscationAttribute]`, que es el marcador de ofuscación de código más básico y más utilizado. Este atributo vive en el `System.Reflection` espacio de nombres y le permite aplicar protección básica al código sin introducir herramientas de terceros.

**Características principales**:
- `Feature` propiedad: especifica la característica de ofuscación, como `"ultra"` (alta ofuscación) o `"all"` (ofuscación total)
- `Exclude` propiedad: `true` significa excluir de la ofuscación, y `false` significa aplicar ofuscación
- Se puede aplicar a clases, métodos, propiedades y otros miembros de tipo.

En el proyecto HagiCode, puedes verlo usado así:

```csharp
[Obfuscation(Feature = "ultra", Exclude = false)]
public async Task<LicenseValidationResult?> ValidateLicenseAsync(...)
```

Las **ventajas** de este enfoque son bastante obvias:
- Sin dependencias adicionales; está integrado en .NET Framework listo para usar
- Puede ser reconocido y procesado por herramientas de ofuscación de terceros.
- No aumenta significativamente el tamaño del ensamblaje compilado.

Dicho esto, también tiene limitaciones. Es solo un marcador y el resultado real de la ofuscación depende de la implementación de la herramienta. No puede proporcionar seguridad a nivel de protección de máquinas virtuales.

### 2. VMP (Protección de máquinas virtuales)

VMP es una herramienta de protección de código profesional que brinda protección de alto nivel al compilar código en instrucciones de máquinas virtuales. A diferencia de la simple ofuscación de nombres, VMP en realidad transforma la lógica del código en una forma que los descompiladores convencionales no pueden reconstruir.

**Clasificación del nivel de protección**:

| Nivel | Virtualización | mutación | Anti-depuración | Cifrado de cadenas | Casos de uso |
|------|--------|------|--------|------------|----------|
| ALTA | lleno | alto | habilitado | habilitado | Validación de licencia, simultaneidad de sesiones, constantes sensibles |
| MEDIO | parcial | medio | habilitado | habilitado | Lógica empresarial, modelos de dominio. |
| BAJO | ninguno | bajo | discapacitado | discapacitado | Clases de utilidad, código no crítico |

El proyecto HagiCode define un sistema de atributos declarativos para marcar código que necesita protección:

```csharp
// High-priority protection
[VmProtect(VmProtectionPriority.High, Reason = "Contains license verification logic")]
public class KeygenClient { ... }

// Exclude from protection
[VmExclude(Reason = "Public API that must remain unchanged")]
public class PublicApi { ... }

// Inherited protection
[VmProtect(Priority.High, ProtectDerived = true)]
public class BaseLicenseValidator { ... }
```

### 3. Estrategia de protección durante el tiempo de construcción

La protección VMP no sólo importa en tiempo de ejecución. También debe automatizarse como parte del proceso de compilación, porque hacerlo manualmente sería demasiado tedioso. El sistema de compilación de HagiCode admite varios modos:

- **Modo nativo de Windows**: invoca la herramienta VMProtect directamente
- **Modo contenedor Docker de Linux**: ejecute VMP dentro de un contenedor para resolver problemas de compatibilidad entre plataformas
- **Escaneo de atributos**: descubre automáticamente marcadores de protección en el código
- **Mecanismo de validación**: Confirma que la protección se ha aplicado correctamente

En conjunto, estas capacidades hacen que el proceso sea mucho más fácil de gestionar.

## Solución

### 1. Uso del atributo de ofuscación integrado de Microsoft

Aplicar `ObfuscationAttribute` directamente en código:

```csharp
using System.Reflection;

[Obfuscation(Feature = "ultra", Exclude = false)]
public class LicenseService
{
    [Obfuscation(Feature = "ultra", Exclude = false)]
    public async Task<bool> ValidateLicenseAsync(string key)
    {
        // License validation logic
    }

    [Obfuscation(Feature = "flow", Exclude = false)]
    private string DecryptToken(string encrypted)
    {
        // Decryption logic
    }
}
```

A veces es necesario permitir que los ensamblados de prueba accedan a los miembros internos y al mismo tiempo mantener seguro el código de producción:

```csharp
// AssemblyInfo.cs
[assembly: InternalsVisibleTo("HagiCode.Application.Tests")]
[assembly: InternalsVisibleTo("DynamicProxyGenAssembly2")] // for Moq
```

Esto hace que las pruebas sean mucho más convenientes, porque el código aún debe probarse correctamente.

### 2. Definiciones de atributos personalizados para la protección VMP

Cree atributos de protección personalizados para controlar el comportamiento de VMP:

```csharp
using System;

[AttributeUsage(AttributeTargets.Class | AttributeTargets.Method | AttributeTargets.Property)]
public class VmProtectAttribute : Attribute
{
    public VmProtectionPriority Priority { get; set; }
    public string? Reason { get; set; }
    public bool ProtectDerived { get; set; }
}

[AttributeUsage(AttributeTargets.Class | AttributeTargets.Method | AttributeTargets.Property)]
public class VmExcludeAttribute : Attribute
{
    public string? Reason { get; set; }
}

public enum VmProtectionPriority
{
    None = 0,
    Low = 1,
    Medium = 2,
    High = 3
}
```

A menudo es más fácil trabajar con atributos personalizados porque reflejan directamente sus propios requisitos de protección.

### 3. Archivo de configuración VMP

```yaml
# vmp_config.yml
protection:
  priority_mode: "attribute"  # Attribute-based priority
  default_level: "medium"

tools:
  - name: "vmprotect"
    path: "C:\\Program Files\\VMProtect Ultimate\\VMProtect.exe"

protection_levels:
  high:
    virtualization: "full"
    mutation: "high"
    anti_debug: true
    anti_dump: true
    encrypt_strings: true
    encrypt_resources: true

  medium:
    virtualization: "partial"
    mutation: "medium"
    anti_debug: true
    encrypt_strings: true

  low:
    virtualization: "none"
    mutation: "low"
    anti_debug: false
```

Una configuración más clara hace que el sistema sea mucho más fácil de mantener en el futuro.

## Guía práctica

### 1. Prácticas de protección de componentes críticos

Según la especificación de protección del código de HagiCode, los siguientes componentes deben utilizar protección de ALTA prioridad:

```csharp
// Production constants - must be encrypted and protected by VMP
[VmProtect(VmProtectionPriority.High, Reason = "Production constants")]
public static class ProductionConstants
{
    // Encrypted string accessor, protected by VMP
    [VmProtect(VmProtectionPriority.High)]
    public static string GetLicenseServerUrl(IOptions<LicenseOptions> options) => ...;
}

// License validation logic
[VmProtect(VmProtectionPriority.High, Reason = "License verification logic")]
public class KeygenClient : IKeygenClient
{
    [Obfuscation(Feature = "ultra", Exclude = false)]
    public async Task<LicenseValidationResult?> ValidateLicenseAsync(...) { ... }
}

// Machine fingerprint service
[VmProtect(VmProtectionPriority.High)]
public class MachineFingerprintService : IMachineFingerprintService { ... }
```

El código crítico merece una protección más fuerte, porque exponer la lógica central causaría problemas reales.

### 2. Cifrado de cadenas y descifrado en tiempo de ejecución

Cifre cadenas en tiempo de compilación y descifrelas en tiempo de ejecución:

```csharp
public static class StringDecryption
{
    [VmProtect(VmProtectionPriority.High, Reason = "CRITICAL SECURITY")]
    public static string DecryptString(byte[] encryptedData, byte[] key, byte[] iv)
    {
        using var aes = Aes.Create();
        aes.Key = key;
        aes.IV = iv;

        using var decryptor = aes.CreateDecryptor();
        using var ms = new MemoryStream(encryptedData);
        using var cs = new CryptoStream(ms, decryptor, CryptoStreamMode.Read);
        using var reader = new StreamReader(cs);

        return reader.ReadToEnd();
    }
}

// Production constant accessor (lazy loading + caching)
public static class ProductionConstants
{
    private static string? _cachedLicenseServerUrl;

    public static string GetLicenseServerUrl(IOptions<LicenseOptions> options)
    {
        if (_cachedLicenseServerUrl == null)
        {
            var encrypted = GetEncryptedLicenseServerUrl();
#if DEBUG
            _cachedLicenseServerUrl = options.Value.PrimaryServer.Url;
#else
            _cachedLicenseServerUrl = StringDecryption.DecryptString(
                encrypted,
                GetEncryptionKey(),
                GetEncryptionIV());
#endif
        }
        return _cachedLicenseServerUrl;
    }
}
```

Este paso es importante porque la información confidencial nunca debe dejarse en texto sin formato.

### 3. Verificación de protección VMP

Después de la compilación, debe verificar si la protección se aplicó correctamente; de lo contrario, no podrá estar seguro de que realmente esté funcionando:

```csharp
// Example verification script
public bool VerifyProtection(string assemblyPath)
{
    // 1. Check the VMP signature
    var bytes = File.ReadAllBytes(assemblyPath);
    var vmpSignature = Encoding.ASCII.GetBytes("VMProtect");
    if (bytes.Any(b => vmpSignature.Contains(b)))
    {
        return true;
    }

    // 2. Check for file size changes (the protected file is usually larger)
    var originalInfo = new FileInfo(assemblyPath.Replace(".dll", ".bak"));
    if (originalInfo.Exists)
    {
        var sizeRatio = (double)new FileInfo(assemblyPath).Length / originalInfo.Length;
        return sizeRatio > 1.1;
    }

    return false;
}
```

Siempre vale la pena realizar una verificación, porque de lo contrario los problemas pueden pasar desapercibidos.

### 4. Notas y advertencias

Aquí hay varios obstáculos que merecen especial atención:

1. **No ofusque todo el código**: las API públicas, las definiciones de interfaz y las clases DTO generalmente no necesitan protección. La ofuscación excesiva puede perjudicar el rendimiento y la depuración. El proyecto HagiCode aprendió esto por las malas.

2. **Proteger a los usuarios de claves**: los métodos que recuperan claves de cifrado deben recibir el mismo nivel de protección o uno superior que los datos cifrados en sí; de lo contrario, toda la configuración pierde su valor.

3. **Equilibrar pruebas y producción**: las compilaciones DEBUG deben omitir el cifrado para facilitar el desarrollo y la depuración, mientras que las compilaciones RELEASE deben permitir una protección completa. Recuerde separarlos con compilación condicional como `#if DEBUG`.

4. **Considere el entorno Docker**: ejecutar VMP en Linux requiere un enfoque en contenedores para garantizar la compatibilidad de las herramientas. HagiCode utiliza una solución de contenedor Wine + VMP para resolver el problema multiplataforma.

5. **La verificación es obligatoria**: una vez finalizada la compilación, debes verificar que la protección se aplicó correctamente. De lo contrario, es posible que aún quede expuesto código confidencial, y el código de verificación mostrado anteriormente existe exactamente para este propósito.

## Conclusión

Con esta estrategia de protección multicapa, HagiCode creó un sistema integral de seguridad de código que abarca desde la ofuscación básica hasta la protección de máquinas virtuales:

- **Capa 1**: Usar `ObfuscationAttribute` para marcar líneas base y proporcionar sugerencias para herramientas de terceros
- **Capa 2**: Usar personalizado `VmProtectAttribute` declaraciones para expresar intención y prioridad de protección
- **Capa 3**: utilice la protección de máquinas virtuales VMP para transformar el código crítico en instrucciones irreversibles de máquinas virtuales
- **Capa 4**: escanea y aplica protección automáticamente durante la compilación, luego verifica el resultado

Este enfoque puede resistir las herramientas de descompilación ordinarias y, al mismo tiempo, resistir mejor los ataques avanzados de ingeniería inversa. Si está creando una aplicación .NET que necesita protección de código, espero que esto le brinde al menos un punto de referencia útil.
