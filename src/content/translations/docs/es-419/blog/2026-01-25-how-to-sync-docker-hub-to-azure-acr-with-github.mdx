## Automatización de la sincronización de imágenes desde Docker Hub a Azure ACR

> Este artículo explica cómo usar GitHub Actions y la herramienta de sincronización de imágenes para automatizar la sincronización de imágenes desde Docker Hub a Azure Container Registry, abordando el acceso lento a Docker Hub en China y algunas regiones de Azure y al mismo tiempo mejorando la disponibilidad de imágenes y la eficiencia de implementación en entornos de Azure.

## Antecedentes / Introducción

El proyecto HagiCode utiliza imágenes de Docker como componentes principales de tiempo de ejecución, y las imágenes principales se alojan en Docker Hub. A medida que el proyecto creció y las necesidades de implementación de Azure aumentaron, encontramos los siguientes puntos débiles:

- Velocidades de extracción de imágenes lentas, con acceso a Docker Hub restringido en China y algunas regiones de Azure
- Una única fuente de imagen crea un único punto de riesgo de falla
- Azure Container Registry proporciona mejor rendimiento e integración de la red cuando se ejecuta en Azure

Para resolver estos problemas, necesitábamos un mecanismo automatizado de sincronización de imágenes para sincronizar periódicamente imágenes desde Docker Hub con Azure ACR, garantizando que los usuarios puedan obtener velocidades de extracción de imágenes más rápidas y una mayor disponibilidad en entornos de Azure.

## Comparación de soluciones técnicas

Al definir la solución, comparamos múltiples opciones técnicas:

### 1. sincronizador de imágenes (elección final)
- Sincronización incremental: solo sincroniza las capas de imágenes modificadas, lo que reduce significativamente la transferencia de red
- Reanudar soporte: la sincronización puede continuar después de interrupciones de la red
- Control de concurrencia: los subprocesos de trabajo configurables mejoran la eficiencia de la sincronización de imágenes grandes
- Manejo sólido de errores: mecanismo de reintento incorporado para fallas (3 reintentos de forma predeterminada)
- Implementación ligera: un único binario sin dependencias
- Compatibilidad con registros múltiples: compatible con Docker Hub, Azure ACR, Harbor y más

### 2. CLI de Docker
- Sin sincronización incremental: el contenido completo de la imagen debe extraerse cada vez
- Menor eficiencia: gran volumen de transferencia de red y mayor tiempo de ejecución
- Fácil de usar: basado en familiar `docker pull` y `docker push` comandos

### 3. CLI de Azure
- Mayor complejidad: se debe configurar la autenticación de la CLI de Azure
- Funcionalidad limitada: `az acr import` tiene un alcance relativamente limitado
- Integración nativa: se integra bien con los servicios de Azure

## Decisiones de diseño arquitectónico

### Decisión 1: Establecer la frecuencia de sincronización diaria a las 00:00 UTC
- Equilibra la frescura de la imagen y el consumo de recursos.
- Evita las horas punta del negocio y reduce el impacto en otras operaciones
- Las imágenes de Docker Hub normalmente se actualizan después de las compilaciones diarias.

### Decisión 2: sincronizar todas las etiquetas de imágenes
- Mantiene total coherencia con Docker Hub
- Ofrece a los usuarios opciones de versión flexibles
- Simplifica la lógica de sincronización y evita reglas complejas de filtrado de etiquetas.

### Decisión 3: utilizar GitHub Secrets para almacenar credenciales
- Soporte nativo de GitHub Actions con seguridad sólida
- Configuración sencilla y fácil mantenimiento.
- Admite control de acceso a nivel de repositorio

## Evaluación y mitigación de riesgos

### Riesgo 1: fuga de credenciales de Azure ACR
- Almacenar credenciales cifradas en GitHub Secrets
- Rote las contraseñas ACR con regularidad
- Restringir la cuenta ACR a permisos de solo inserción
- Supervisar los registros de acceso de ACR

### Riesgo 2: fallos de sincronización que provocan incoherencia en la imagen
- image-syncer incluye un mecanismo de sincronización incremental incorporado
- Reintentos automáticos en caso de error (3 por defecto)
- Registros de errores detallados y notificaciones de fallas
- Capacidad de reanudar

### Riesgo 3: Consumo excesivo de recursos
- La sincronización incremental reduce la transferencia de red
- Simultaneidad configurable (`10` en la configuración actual)
- Supervise la cantidad y el tamaño de las imágenes sincronizadas.
- Ejecute la sincronización durante las horas de menor actividad

## Solución principal

Utilizamos una solución automatizada basada en GitHub Actions + image-syncer para sincronizar imágenes de Docker Hub a Azure ACR.

## Pasos de implementación

### 1. Preparación
- Cree o confirme un Azure Container Registry en Azure Portal
- Crear credenciales de acceso ACR (nombre de usuario y contraseña)
- Confirmar los permisos de acceso al repositorio de imágenes de Docker Hub

### 2. Configurar los secretos de GitHub
Agregue los siguientes secretos en la configuración del repositorio de GitHub:
- AZURE_ACR_USERNAME: nombre de usuario de Azure ACR
- AZURE_ACR_PASSWORD: contraseña de Azure ACR

### 3. Cree el flujo de trabajo de acciones de GitHub
Configurar el flujo de trabajo en `.github/workflows/sync-docker-acr.yml`:
- Activación programada: todos los días a las 00:00 UTC
- Gatillo manual: soportes `workflow_dispatch`
- Gatillo adicional: funciona al presionar hacia el `publish` rama para sincronización rápida

### 4. Flujo de ejecución del flujo de trabajo

| paso | Participante | acción | Descripción |
| --- | --- | --- | --- |
| 1 | Acciones de GitHub | Activar el flujo de trabajo | Activado por programación, ejecución manual o envíos al `publish` rama |
| 2 | Acciones de GitHub → sincronizador de imágenes | Descargue y ejecute la herramienta de sincronización | Entra en la fase de sincronización real |
| 3 | sincronizador de imágenes → Docker Hub | Obtener manifiestos de imágenes y lista de etiquetas | Lee metadatos del repositorio de origen |
| 4 | sincronizador de imágenes → Azure ACR | Obtener datos de imágenes existentes del repositorio de destino | Determina el estado actual del lado objetivo. |
| 5 | sincronizador de imágenes | Comparar las diferencias de origen y destino | Identifica las capas de imágenes que necesitan sincronización. |
| 6 | sincronizador de imágenes → Docker Hub | Extraer capas de imágenes modificadas | Transfiere solo el contenido que necesita actualización |
| 7 | sincronizador de imágenes → Azure ACR | Empujar capas de imágenes cambiadas | Completa la sincronización incremental |
| 8 | sincronizador de imágenes → Acciones de GitHub | Estadísticas de sincronización de devolución | Incluye resultados, diferencias e información de errores. |
| 9 | Acciones de GitHub | Grabar registros y cargar artefactos | Admite auditorías y resolución de problemas posteriores |

## Implementación del flujo de trabajo de acciones de GitHub

A continuación se muestra la configuración de flujo de trabajo real en uso (`.github/workflows/sync-docker-acr.yml`):

```yaml
name: Sync Docker Image to Azure ACR

on:
  schedule:
    - cron: "0 0 * * *" # Every day at 00:00 UTC
  workflow_dispatch: # Manual trigger
  push:
    branches: [publish]

permissions:
  contents: read

jobs:
  sync:
    runs-on: ubuntu-latest

    steps:
      - name: Checkout code
        uses: actions/checkout@v4

      - name: Download image-syncer
        run: |
          # Download the image-syncer binary
          wget https://github.com/AliyunContainerService/image-syncer/releases/download/v1.5.5/image-syncer-v1.5.5-linux-amd64.tar.gz
          tar -zxvf image-syncer-v1.5.5-linux-amd64.tar.gz
          chmod +x image-syncer

      - name: Create auth config
        run: |
          # Generate the authentication config file (YAML format)
          cat > auth.yaml <<EOF
          hagicode.azurecr.io:
            username: "${{ secrets.AZURE_ACR_USERNAME }}"
            password: "${{ secrets.AZURE_ACR_PASSWORD }}"
          EOF

      - name: Create images config
        run: |
          # Generate the image sync config file (YAML format)
          cat > images.yaml <<EOF
          docker.io/newbe36524/hagicode: hagicode.azurecr.io/hagicode
          EOF

      - name: Run image-syncer
        run: |
          # Run synchronization (using the newer --auth and --images parameters)
          ./image-syncer --auth=./auth.yaml --images=./images.yaml --proc=10 --retries=3

      - name: Upload logs
        if: always()
        uses: actions/upload-artifact@v4
        with:
          name: sync-logs
          path: image-syncer-*.log
          retention-days: 7
```

## Detalles de configuración

### 1. Condiciones de activación
- Activador programado: `cron: "0 0 * * *"` se ejecuta todos los días a las 00:00 UTC
- Gatillo manual: `workflow_dispatch` permite a los usuarios ejecutarlo manualmente desde la interfaz de usuario de GitHub
- Presione el gatillo: `push: branches: [publish]` Se activa al presionar la rama de lanzamiento para una sincronización rápida.

### 2. Configuración de autenticación (`auth.yaml`)
```yaml
hagicode.azurecr.io:
  username: "${{ secrets.AZURE_ACR_USERNAME }}"
  password: "${{ secrets.AZURE_ACR_PASSWORD }}"
```

### 3. Configuración de sincronización de imágenes
```yaml
docker.io/newbe36524/hagicode: hagicode.azurecr.io/hagicode
```
Esta configuración significa que todas las etiquetas de `docker.io/newbe36524/hagicode` se sincronizará con `hagicode.azurecr.io/hagicode`.

### 4. parámetros de sincronización de imágenes
- `--auth=./auth.yaml`: ruta al archivo de configuración de autenticación
- `--images=./images.yaml`: ruta al archivo de configuración de sincronización de imágenes
- `--proc=10`: uso `10` trabajadores concurrentes
- `--retries=3`: reintentar fallas `3` veces

## Notas y mejores prácticas

### 1. Recomendaciones de seguridad
- Rote las contraseñas de Azure ACR periódicamente (recomendado cada 90 días)
- Utilice una cuenta de servicio ACR dedicada con permisos de solo inserción
- Supervise los registros de acceso de ACR para detectar accesos anormales a tiempo
- No mostrar credenciales en registros
- No envíe credenciales al repositorio de código

### 2. Optimización del rendimiento
- Ajustar el `--proc` parámetro: ajuste la concurrencia según el ancho de banda de la red (recomendado `5-20`)
- Supervisar la duración de la sincronización: si la sincronización tarda demasiado, considere reducir la simultaneidad
- Limpie los registros con regularidad: establezca un tiempo razonable `retention-days` valor (`7` actualmente)

### 3. Solución de problemas

#### Problema 1: error de autenticación
```
Error: failed to authenticate to hagicode.azurecr.io
```
Solución:
1. Compruebe si los secretos de GitHub están configurados correctamente
2. Verifique si la contraseña de Azure ACR ha caducado
3. Confirme que los permisos de la cuenta de servicio ACR sean correctos

#### Problema 2: tiempo de espera de la red
```
Error: timeout waiting for response
```
Solución:
1. Verificar la conectividad de la red
2. Reducir el número de trabajadores concurrentes (los `--proc` parámetro)
3. Espere a que la red se recupere y vuelva a ejecutar el flujo de trabajo.

#### Problema 3: sincronización de imágenes incompleta
```
Warning: some tags failed to sync
```
Solución:
1. Verifique los registros de sincronización para identificar etiquetas fallidas
2. Vuelva a ejecutar manualmente el flujo de trabajo para sincronizarlo nuevamente
3. Verifique que la imagen de origen en Docker Hub esté en buen estado

### 4. Monitoreo y alerta
- Consulte periódicamente la página Acciones para confirmar el estado del flujo de trabajo.
- Habilite las notificaciones de GitHub para que las fallas en el flujo de trabajo se informen con prontitud
- Supervisar el uso del almacenamiento de Azure ACR
- Verifique periódicamente la coherencia de las etiquetas en todos los registros

## Preguntas frecuentes y soluciones

### P1: ¿Cómo sincronizo etiquetas específicas en lugar de todas las etiquetas?

Modificar el `images.yaml` archivo de configuración:
```yaml
# Sync only the latest and v1.0 tags
docker.io/newbe36524/hagicode:latest: hagicode.azurecr.io/hagicode:latest
docker.io/newbe36524/hagicode:v1.0: hagicode.azurecr.io/hagicode:v1.0
```

### P2: ¿Cómo sincronizo varios repositorios de imágenes?

Añade varias líneas a `images.yaml`:
```yaml
docker.io/newbe36524/hagicode: hagicode.azurecr.io/hagicode
docker.io/newbe36524/another-image: hagicode.azurecr.io/another-image
```

### P3: ¿Cómo puedo volver a intentarlo después de que falla la sincronización?
- Reintento automático: image-syncer incluye un mecanismo de reintento incorporado (`3` reintentos por defecto)
- Reintento manual: haga clic `Re-run all jobs` en la página de Acciones de GitHub

### P4: ¿Cómo puedo ver el progreso detallado de la sincronización?
- Ver registros en tiempo real en la página Acciones
- Descarga el `sync-logs` artefacto para inspeccionar los archivos de registro completos
- Los archivos de registro incluyen el estado de sincronización y la velocidad de transferencia de cada etiqueta.

### P5: ¿Cuánto tiempo lleva la sincronización?
- Sincronización completa inicial: normalmente `10-30` minutos dependiendo del tamaño de la imagen
- Sincronización incremental: normalmente `2-5` minutos si los cambios de imagen son pequeños
- El tiempo depende del ancho de banda de la red, el tamaño de la imagen y la configuración de simultaneidad.
