## Guía completa para el análisis de mensajes de la consola Codex SDK

> Este artículo explica el mecanismo de flujo de eventos del SDK de Codex, el análisis de tipos de mensajes y las mejores prácticas en proyectos reales, lo que ayuda a los desarrolladores a dominar rápidamente las habilidades básicas detrás de los servicios de ejecución de IA.

## Antecedentes

Al crear un servicio de ejecución de IA basado en el SDK de Codex, inevitablemente nos topamos con una pregunta práctica: ¿cómo debemos manejar los mensajes de eventos transmitidos devueltos por Codex? Estos mensajes contienen información importante, como el estado de ejecución, el contenido de la salida y detalles del error, por lo que merecen un manejo cuidadoso.

Como parte del proyecto HagiCode, necesitábamos un ejecutor confiable para escenarios de asistente de codificación de IA. Es exactamente por eso que decidimos estudiar en profundidad el mecanismo de flujo de eventos del Codex SDK. Después de todo, sólo comprendiendo cómo funcionan los mensajes subyacentes podremos construir una plataforma de ejecución de IA verdaderamente de nivel empresarial.

El SDK de Codex es un SDK de asistencia a la programación lanzado por OpenAI. Devuelve resultados de ejecución a través de un flujo de eventos. A diferencia del modelo tradicional de solicitud-respuesta, Codex utiliza eventos transmitidos para que podamos:

- Obtenga el progreso de la ejecución en tiempo real
- Manejar los errores rápidamente
- Obtenga estadísticas detalladas de uso de tokens
- Admite tareas complejas de larga duración

Comprender estos tipos de eventos y analizarlos correctamente es esencial para implementar un ejecutor de IA totalmente capaz. Al final nadie quiere trabajar con una caja negra.

## Mecanismo de flujo de eventos

### Conceptos básicos

El SDK del Codex utiliza el `thread.runStreamed()` método para devolver un iterador de eventos asincrónico:

```typescript
import { Codex } from '@openai/codex-sdk';

const client = new Codex({
  apiKey: process.env.CODEX_API_KEY,
  baseUrl: process.env.CODEX_BASE_URL,
});

const thread = client.startThread({
  workingDirectory: '/path/to/project',
  skipGitRepoCheck: false,
});

const { events } = await thread.runStreamed('your prompt here', {
  outputSchema: {
    type: 'object',
    properties: {
      output: { type: 'string' },
      status: { type: 'string', enum: ['ok', 'action_required'] },
    },
    required: ['output', 'status'],
  },
});

for await (const event of events) {
  // Handle each event
}
```

### Explicación detallada de los tipos de eventos

| Tipo de evento | Descripción | Datos clave |
|---------|------|----------|
| `thread.started` | El hilo comenzó exitosamente | `thread_id` |
| `item.updated` | Contenido del mensaje actualizado | `item.text` |
| `item.completed` | Mensaje completado | `item.text` |
| `turn.completed` | Ejecución completada | `usage` (uso de tokens) |
| `turn.failed` | La ejecución falló | `error.message` |
| `error` | evento de error | `message` |

En proyectos reales, el componente ejecutor de HagiCode se construye sobre estos tipos de eventos. Necesitamos manejar cada tipo de evento con cuidado para garantizar una experiencia de usuario fluida. Los buenos sistemas se construyen tomando en serio los detalles.

## Implementación del análisis de mensajes

### Extracción del contenido del mensaje

El contenido del mensaje se extrae mediante un controlador de eventos:

```typescript
private handleThreadEvent(event: ThreadEvent, onMessage: (content: string) => void): void {
  // Only handle message update and completion events
  if (event.type !== 'item.updated' && event.type !== 'item.completed') {
    return;
  }

  // Only handle agent message content
  if (event.item.type !== 'agent_message') {
    return;
  }

  // Extract text content
  onMessage(event.item.text);
}
```

Puntos clave:
- Solo manejar `item.updated` y `item.completed` eventos
- Solo maneja contenido de tipo `agent_message`
- El contenido del mensaje está en el `event.item.text` campo

### Análisis de salida estructurada

Codex admite salida estructurada JSON. Puede especificar el formato de devolución a través del `outputSchema` parámetro:

```typescript
const DEFAULT_OUTPUT_SCHEMA = {
  type: 'object',
  properties: {
    output: { type: 'string' },
    status: { type: 'string', enum: ['ok', 'action_required'] },
  },
  required: ['output', 'status'],
  additionalProperties: false,
} as const;
```

La función de análisis intenta analizar JSON y, si eso falla, vuelve al texto sin formato.

```typescript
function toStructuredOutput(raw: string): StructuredOutput {
  try {
    const parsed = JSON.parse(raw) as Partial<StructuredOutput>;
    if (typeof parsed.output === 'string') {
      return {
        output: parsed.output,
        status: parsed.status === 'action_required' ? 'action_required' : 'ok',
      };
    }
  } catch {
    // JSON parsing failed, fall back to the raw text
  }

  return {
    output: raw,
    status: 'ok',
  };
}
```

### Flujo completo de manejo de eventos

```typescript
private async runWithStreaming(
  thread: Thread,
  input: CodexStageExecutionInput
): Promise<{ output: string; usage: Usage | null }> {
  const abortController = new AbortController();
  const timeoutHandle = setTimeout(() => {
    abortController.abort();
  }, Math.max(1000, input.timeoutMs));

  let latestMessage = '';
  let usage: Usage | null = null;
  let emittedLength = 0;

  try {
    const { events } = await thread.runStreamed(input.prompt, {
      outputSchema: DEFAULT_OUTPUT_SCHEMA,
      signal: abortController.signal,
    });

    for await (const event of events) {
      // Handle message content
      this.handleThreadEvent(event, (nextContent) => {
        const delta = nextContent.slice(emittedLength);
        if (delta.length > 0) {
          emittedLength = nextContent.length;
          input.callbacks?.onChunk?.(delta);  // Streaming callback
        }
        latestMessage = nextContent;
      });

      // Process different data based on the event type
      if (event.type === 'thread.started') {
        this.threadId = event.thread_id;
      } else if (event.type === 'turn.completed') {
        usage = event.usage;
      } else if (event.type === 'turn.failed') {
        throw new CodexExecutorError('gateway_unavailable', event.error.message, true);
      } else if (event.type === 'error') {
        throw new CodexExecutorError('gateway_unavailable', event.message, true);
      }
    }
  } catch (error) {
    if (abortController.signal.aborted) {
      throw new CodexExecutorError(
        'upstream_timeout',
        `Codex stage timed out after ${input.timeoutMs}ms`,
        true
      );
    }
    throw error;
  } finally {
    clearTimeout(timeoutHandle);
  }

  const structured = toStructuredOutput(latestMessage);
  return { output: structured.output, usage };
}
```

## Estrategia de manejo de errores

### Asignación de códigos de error

Asigne patrones de error específicos a códigos de error concretos para que las capas superiores puedan manejarlos más fácilmente:

```typescript
function mapError(error: unknown): CodexExecutorError {
  if (error instanceof CodexExecutorError) {
    return error;
  }

  const message = error instanceof Error ? error.message : String(error);
  const normalized = message.toLowerCase();

  // Authentication errors - not retryable
  if (normalized.includes('401') ||
      normalized.includes('403') ||
      normalized.includes('api key') ||
      normalized.includes('auth')) {
    return new CodexExecutorError('auth_invalid', message, false);
  }

  // Rate limit errors - retryable
  if (normalized.includes('429') || normalized.includes('rate limit')) {
    return new CodexExecutorError('rate_limited', message, true);
  }

  // Timeout errors - retryable
  if (normalized.includes('timeout') || normalized.includes('aborted')) {
    return new CodexExecutorError('upstream_timeout', message, true);
  }

  // Default error
  return new CodexExecutorError('gateway_unavailable', message, true);
}
```

### Definiciones de tipos de errores

```typescript
export type CodexErrorCode =
  | 'auth_invalid'      // Authentication failure
  | 'upstream_timeout'  // Upstream timeout
  | 'rate_limited'      // Rate limited
  | 'gateway_unavailable'; // Gateway unavailable

export class CodexExecutorError extends Error {
  readonly code: CodexErrorCode;
  readonly retryable: boolean;

  constructor(code: CodexErrorCode, message: string, retryable: boolean) {
    super(message);
    this.name = 'CodexExecutorError';
    this.code = code;
    this.retryable = retryable;
  }
}
```

## Configuración del entorno y directorio de trabajo

### Validación del directorio de trabajo

El SDK de Codex requiere que el directorio de trabajo sea un repositorio Git válido.

```typescript
export function validateWorkingDirectory(
  workingDirectory: string,
  skipGitRepoCheck: boolean
): void {
  const resolvedWorkingDirectory = path.resolve(workingDirectory);

  if (!existsSync(resolvedWorkingDirectory)) {
    throw new CodexExecutorError(
      'gateway_unavailable',
      'Working directory does not exist.',
      false
    );
  }

  if (!statSync(resolvedWorkingDirectory).isDirectory()) {
    throw new CodexExecutorError(
      'gateway_unavailable',
      'Working directory is not a directory.',
      false
    );
  }

  if (skipGitRepoCheck) {
    return;
  }

  const gitDir = path.join(resolvedWorkingDirectory, '.git');
  if (!existsSync(gitDir)) {
    throw new CodexExecutorError(
      'gateway_unavailable',
      'Working directory is not a git repository.',
      false
    );
  }
}
```

### Cargando variables de entorno

El SDK de Codex necesita cargar variables de entorno desde el shell de inicio de sesión para que AI Agent pueda acceder a los comandos del sistema:

```typescript
function parseEnvironmentOutput(output: Buffer): Record<string, string> {
  const parsed: Record<string, string> = {};

  for (const entry of output.toString('utf8').split('\0')) {
    if (!entry) continue;

    const separatorIndex = entry.indexOf('=');
    if (separatorIndex <= 0) continue;

    const key = entry.slice(0, separatorIndex);
    const value = entry.slice(separatorIndex + 1);
    if (key.length > 0) {
      parsed[key] = value;
    }
  }

  return parsed;
}

function tryLoadEnvironmentFromShell(shellPath: string): Record<string, string> | null {
  const result = spawnSync(shellPath, ['-ilc', 'env -0'], {
    env: process.env,
    stdio: ['ignore', 'pipe', 'pipe'],
    timeout: 5000,
  });

  if (result.error || result.status !== 0) {
    return null;
  }

  return parseEnvironmentOutput(result.stdout);
}

export function createExecutorEnvironment(
  envOverrides: Record<string, string> = {}
): Record<string, string> {
  // Load environment variables from the login shell
  const consoleEnv = loadConsoleEnvironmentFromShell();

  return {
    ...process.env,
    ...consoleEnv,
    ...envOverrides,
  };
}
```

## Ejemplo de uso completo

### Uso básico

En el proyecto HagiCode, utilizamos el siguiente enfoque para inicializar el cliente Codex y ejecutar tareas:

```typescript
import { Codex } from '@openai/codex-sdk';

async function executeWithCodex(prompt: string, workingDir: string) {
  const client = new Codex({
    apiKey: process.env.CODEX_API_KEY,
    env: { PATH: process.env.PATH },
  });

  const thread = client.startThread({
    workingDirectory: workingDir,
  });

  const { events } = await thread.runStreamed(prompt);

  let result = '';
  for await (const event of events) {
    if (event.type === 'item.updated' && event.item.type === 'agent_message') {
      result = event.item.text;
    }
    if (event.type === 'turn.completed') {
      console.log('Token usage:', event.usage);
    }
  }

  // Try to parse JSON output
  try {
    const parsed = JSON.parse(result);
    return parsed.output;
  } catch {
    return result;
  }
}
```

### Implementación completa con reintentos

```typescript
export class CodexSdkExecutor {
  private readonly config: CodexRuntimeConfig;
  private readonly client: Codex;
  private threadId: string | null = null;

  async executeStage(input: CodexStageExecutionInput): Promise<CodexStageExecutionResult> {
    const maxAttempts = Math.max(1, this.config.retryCount + 1);
    let attempt = 0;
    let lastError: CodexExecutorError | null = null;

    while (attempt < maxAttempts) {
      attempt += 1;

      try {
        const thread = this.getThread(input.workingDirectory);
        const { output, usage } = await this.runWithStreaming(thread, input);

        return {
          output,
          usage,
          threadId: this.threadId!,
          attempts: attempt,
          latencyMs: Date.now() - startedAt,
        };
      } catch (error) {
        const mappedError = mapError(error);
        lastError = mappedError;

        // Non-retryable error or max retry attempts reached
        if (!mappedError.retryable || attempt >= maxAttempts) {
          throw mappedError;
        }

        // Wait before retrying
        await new Promise(resolve => setTimeout(resolve, 1000 * attempt));
      }
    }

    throw lastError!;
  }
}
```

## Mejores prácticas

### 1. Requisitos del directorio de trabajo

- Asegúrese de que el directorio de trabajo sea un repositorio Git válido
- Utilice el `PROJECT_ROOT` variable de entorno para especificarla explícitamente
- Durante la depuración, puede configurar `CODEX_SKIP_GIT_REPO_CHECK=true` saltarse el cheque

### 2. Configuración de variables de entorno

- Pase solo las variables de entorno requeridas a través de un mecanismo de lista blanca
- Utilice el shell de inicio de sesión para cargar el entorno completo
- Evite pasar información sensible

### 3. Estrategia de tiempo de espera y reintento

- Establezca tiempos de espera razonables según la complejidad de la tarea
- Implementar un retroceso exponencial para errores reintentables
- Registrar recuentos de reintentos y motivos

### 4. Manejo de errores

- Distinguir entre errores reintentables y no reintentables
- Proporcionar mensajes de error claros y sugerencias.
- Utilice códigos de error unificados para que las capas superiores puedan manejarlos de manera consistente

### 5. Salida de transmisión

- Implementar devoluciones de llamadas de salida incrementales para mejorar la experiencia del usuario
- Manejar correctamente las actualizaciones incrementales de mensajes
- Registre el uso de tokens para el análisis de costos
