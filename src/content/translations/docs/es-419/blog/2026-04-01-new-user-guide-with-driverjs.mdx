## Creación de una incorporación elegante de nuevos usuarios en proyectos de React: práctica driver.js de HagiCode

> Cuando los usuarios abren su producto por primera vez, ¿saben realmente por dónde empezar? En este artículo, quiero hablar un poco sobre cómo usamos driver.js para la incorporación de nuevos usuarios en el proyecto HagiCode. Considérelo un pequeño ejemplo práctico para iniciar la conversación.

## Antecedentes

¿Alguna vez te has encontrado con esta situación? Un nuevo usuario se registra para su producto, abre la página e inmediatamente parece perdido. Exploran a su alrededor, sin saber en qué hacer clic o qué hacer a continuación. Como desarrolladores, a menudo asumimos que los usuarios simplemente "explorarán por su cuenta" porque, después de todo, la curiosidad humana es ilimitada. Pero la realidad es diferente: la mayoría de los usuarios se van silenciosamente en cuestión de minutos si no pueden encontrar el punto de entrada correcto, como si la historia comenzara de repente y terminara con la misma naturalidad.

La incorporación de nuevos usuarios es una forma importante de resolver este problema, pero construirlo bien no es tan sencillo. Un buen sistema de incorporación necesita:

- Localice con precisión los elementos de la página y resáltelos
- Admite flujos de incorporación de varios pasos
- Recordar la elección del usuario (completar u omitir)
- Evite afectar el rendimiento de la página y la interacción normal.
- Mantenga la estructura del código clara y fácil de mantener.

Mientras construíamos HagiCode, nos encontramos con el mismo desafío. HagiCode es un asistente de codificación de IA y su flujo de trabajo principal es un flujo de trabajo OpenSpec que se ve así: "el usuario crea una propuesta -> la IA genera un plan -> el usuario lo revisa -> la IA lo ejecuta". Para los usuarios que se encuentran con este concepto por primera vez, el flujo de trabajo es completamente nuevo, por lo que necesitan una incorporación sólida para comenzar rápidamente. Siempre lleva un poco de tiempo acostumbrarse a las cosas nuevas.

## Acerca de HagiCode

El enfoque compartido en este artículo proviene de nuestra experiencia práctica en el [HagiCode](https://hagicode.com) proyecto. HagiCode es un asistente de codificación de IA basado en Claude que ayuda a los desarrolladores a completar tareas de codificación de manera más eficiente a través del flujo de trabajo OpenSpec. Puede ver nuestro código fuente abierto en [GitHub](https://github.com/HagiCode-org/site).

## Por qué elegimos driver.js

Durante la fase de evaluación técnica, analizamos varias bibliotecas de incorporación convencionales. Cada uno tenía sus propios puntos fuertes:

- **Intro.js**: Potente, pero relativamente grande, y la personalización de estilos es algo compleja
- **Shepherd.js**: API bien diseñada, pero demasiado "pesada" para nuestro caso de uso
- **driver.js**: API ligera, concisa e intuitiva y funciona bien en el ecosistema React

Al final, elegimos driver.js. No hubo ninguna razón especialmente dramática. La elección se debió principalmente a estas consideraciones:

1. **Ligero**: la biblioteca principal es pequeña y no aumenta significativamente el tamaño del paquete.
2. **API simple**: la configuración es clara e intuitiva, por lo que es fácil de aprender
3. **Flexible**: admite posicionamiento personalizado, estilo y comportamiento de interacción
4. **Importación dinámica**: se puede cargar según demanda sin afectar el rendimiento de la primera pantalla

En la selección de tecnología, rara vez existe una mejor respuesta universal. Por lo general, sólo existe la opción que mejor se adapta.

## Implementación técnica

### Configuración central

driver.js tiene un modelo de configuración muy intuitivo. Aquí está la configuración principal que utilizamos en el proyecto HagiCode:

```typescript
import { driver } from 'driver.js';
import 'driver.js/dist/driver.css';

const newConversationDriver = driver({
  allowClose: true,           // Allow users to close the guide
  animate: true,              // Enable animations
  overlayClickBehavior: 'close', // Close the guide when the overlay is clicked
  disableActiveInteraction: false, // Keep elements interactive
  showProgress: false,        // Do not show the progress bar (we manage progress ourselves)
  steps: guideSteps           // Array of guide steps
});
```

El razonamiento detrás de esta configuración es:

- `allowClose: true` - Respetar la elección del usuario y no obligarlo a terminar la guía.
- `disableActiveInteraction: false` - Algunos pasos requieren acciones reales del usuario, como escribir entradas, por lo que la interacción no se puede desactivar
- `overlayClickBehavior: 'close'` - Ofrecer a los usuarios una forma rápida de salir

### Gestión del Estado

El estado de incorporación persistente es fundamental. No queremos reiniciar la guía cada vez que se actualiza la página, porque eso se vuelve molesto rápidamente. Usos de HagiCode `localStorage` para gestionar el estado de la guía:

```typescript
export type GuideState = 'pending' | 'dismissed' | 'completed';

export interface UserGuideState {
  session: GuideState;
  detailGuides: Record<string, GuideState>;
}

// Read state
export const getUserGuideState = (): UserGuideState => {
  const state = localStorage.getItem('userGuideState');
  return state ? JSON.parse(state) : { session: 'pending', detailGuides: {} };
};

// Update state
export const setUserGuideState = (state: UserGuideState) => {
  localStorage.setItem('userGuideState', JSON.stringify(state));
};
```

Definimos tres estados:

- `pending`: La guía aún está en progreso y el usuario no la ha completado ni la ha omitido.
- `dismissed`: El usuario cerró la guía de forma proactiva.
- `completed`: El usuario completó todos los pasos.

Para la incorporación de la página de detalles de la propuesta, también admitimos un seguimiento de estado más detallado a través del `detailGuides` mapa, porque una propuesta puede pasar por varias etapas (borrador, revisión y ejecución completa) y cada etapa necesita una orientación diferente. El estado de las cosas siempre está cambiando y la incorporación debería reflejar eso.

### Selección del elemento objetivo

driver.js utiliza selectores CSS para localizar elementos de destino. HagiCode sigue una convención simple: use un código personalizado `data-guide` atributo para marcar objetivos de incorporación:

```typescript
const steps = [
  {
    element: '[data-guide="launch"]',
    popover: {
      title: 'Start a New Conversation',
      description: 'Click here to create a new conversation session...'
    }
  }
];
```

En componentes, se ve así:

```typescript
<button data-guide="launch" onClick={handleLaunch}>
  New Conversation
</button>
```

Los beneficios de este enfoque son:

- Evite conflictos con los nombres de clases de estilo empresarial
- Semántica clara, para que pueda saber inmediatamente que el elemento está relacionado con la incorporación
- Más fácil de gestionar y mantener de forma consistente

### Optimización de importación dinámica

Debido a que la función de incorporación solo es necesaria en escenarios específicos, como la primera visita de un usuario, utilizamos importaciones dinámicas para optimizar el rendimiento de carga inicial:

```typescript
const initNewUserGuide = async () => {
  // Dynamically import driver.js
  const { driver } = await import('driver.js');
  await import('driver.js/dist/driver.css');

  // Initialize the guide
  const newConversationDriver = driver({
    // ...configuration
  });

  newConversationDriver.drive();
};
```

De esta manera, driver.js y su hoja de estilo solo se cargan cuando es necesario y no afectan el rendimiento de la primera pantalla. No mucha gente disfruta esperando algo que ni siquiera necesita todavía.

## Diseño de flujo de incorporación

HagiCode implementa dos rutas de incorporación que cubren los escenarios principales del usuario.

### Incorporación de sesiones (10 pasos)

Esta ruta de incorporación ayuda a los usuarios a completar todo el flujo, desde la creación de una conversación hasta el envío de su primera propuesta completa:

1. **iniciar** - Inicia la guía e introduce el botón "Nueva conversación"
2. **componer**: guía al usuario para que escriba una solicitud en el cuadro de entrada
3. **enviar**: guía al usuario para que haga clic en el botón enviar
4. **proposal-launch-readme** - Guía al usuario para crear una propuesta README
5. **proposal-compose-readme**: guía al usuario para editar el contenido de la solicitud LÉAME
6. **proposal-submit-readme** - Guía al usuario para enviar la propuesta README
7. **proposal-launch-agents** - Guía al usuario para crear una `AGENTS.md` Propuesta
8. **proposal-compose-agents** - Guía al usuario para editar el `AGENTS.md` solicitud
9. **agentes-de-envío-de-propuestas** - Guía al usuario para que envíe la `AGENTS.md` Propuesta
10. **propuesta-espera**: explica que la IA está procesando y que el usuario debe esperar un momento.

La idea detrás de este camino es permitir a los usuarios experimentar el flujo de trabajo principal de HagiCode de primera mano a través de dos tareas reales de creación de propuestas, una para `README` y uno para `AGENTS.md`. Existe una gran diferencia entre escuchar acerca de un flujo de trabajo y realizarlo usted mismo.

### Incorporación de detalles de propuesta (3 pasos)

Cuando los usuarios ingresan a la página de detalles de la propuesta, HagiCode activa la guía correspondiente según el estado actual de la propuesta:

1. **redacción** (etapa de borrador): guía al usuario para que revise el plan generado por IA.
2. **revisión** (etapa de revisión): guía al usuario para ejecutar el plan
3. **executionCompleted** (etapa completada): guía al usuario para archivar el plan

La característica definitoria de esta guía es que está impulsada por el estado: decide dinámicamente qué paso de incorporación mostrar en función del estado real de la propuesta. Las cosas cambian y la incorporación debería cambiar con ellas.

En esta etapa, la guía centra la atención del usuario en la acción clave de generar un plan, para que no se pregunte qué hacer primero cuando ingresa a la página de detalles por primera vez.

### Mecanismo de reintento de renderizado de elementos

En las aplicaciones React, es posible que el elemento de incorporación de destino aún no haya terminado de renderizarse, por ejemplo, porque todavía se están cargando datos asincrónicos. Para manejar esto, HagiCode implementa un mecanismo de reintento:

```typescript
const waitForElement = (selector: string, maxRetries = 10, interval = 100) => {
  let retries = 0;
  return new Promise<HTMLElement>((resolve, reject) => {
    const checkElement = () => {
      const element = document.querySelector(selector) as HTMLElement;
      if (element) {
        resolve(element);
      } else if (retries < maxRetries) {
        retries++;
        setTimeout(checkElement, interval);
      } else {
        reject(new Error(`Element not found: ${selector}`));
      }
    };
    checkElement();
  });
};
```

Llame a esta función antes de inicializar la guía para asegurarse de que el elemento de destino ya exista. A veces vale la pena esperar un poco más.

## Resumen de mejores prácticas

Según la experiencia práctica de HagiCode, a continuación se presentan algunas de las mejores prácticas clave:

### 1. La incorporación debería ser fácil de finalizar

No obligue a los usuarios a completar la incorporación. Algunos usuarios son exploradores por naturaleza y prefieren resolver las cosas por sí mismos. Proporcione un botón claro para "Omitir", recuerde su elección y no los vuelva a interrumpir la próxima vez.

### 2. Mantenga el contenido de incorporación breve y conciso

Cada paso de incorporación debe centrarse en un único objetivo:

- **Título**: breve y claro, idealmente no más de unas pocas palabras.
- **Descripción**: Vaya directo al grano y dígale al usuario qué es esto y por qué es importante.

Evite las explicaciones largas. La atención del usuario es limitada durante la incorporación y cuanto más diga, es menos probable que lo lea.

### 3. Mantenga estables los selectores

Utilice marcadores de elementos estables que no cambien con frecuencia. la costumbre `data-guide` El atributo es una buena elección. Evite depender de los nombres de las clases o la estructura DOM, ya que son fáciles de cambiar durante las refactorizaciones. El código cambia todo el tiempo, pero algunas anclas deben permanecer estables siempre que sea posible.

### 4. Pruebe su incorporación

HagiCode incluye casos de prueba completos para la función de incorporación:

```typescript
describe('NewUserConversationGuide', () => {
  it('should initialize guide state correctly', () => {
    const state = getUserGuideState();
    expect(state.session).toBe('pending');
  });

  it('should update guide state correctly', () => {
    setUserGuideState({ session: 'completed', detailGuides: {} });
    const state = getUserGuideState();
    expect(state.session).toBe('completed');
  });
});
```

Las pruebas ayudan a garantizar que la refactorización no interrumpa accidentalmente el comportamiento de incorporación. Nadie quiere que un pequeño cambio de código dañe silenciosamente una funcionalidad que funcionaba anteriormente.

### 5. Optimización del rendimiento

- Utilice importaciones dinámicas para cargar de forma diferida la biblioteca de incorporación
- Evite inicializar la lógica de incorporación después de que el usuario ya haya completado la guía.
- Considere el impacto de las animaciones en el rendimiento y desactívelas en dispositivos de gama baja si es necesario.

El rendimiento, como muchas cosas en la vida, merece un presupuesto cuidadoso.

## Conclusión

La incorporación de nuevos usuarios es una parte importante para mejorar la experiencia del usuario del producto. En el proyecto HagiCode, utilizamos driver.js para crear un sistema de incorporación completo que cubra todo el flujo de trabajo desde la creación de la sesión hasta la ejecución de la propuesta.

Los puntos centrales que esperamos compartir a través de este artículo son:

1. **Las opciones técnicas deben coincidir con las necesidades reales**: driver.js no es la opción más poderosa, pero es la que mejor se adapta a nosotros
2. **La gestión estatal es crítica**: Utilice `localStorage` para mantener el estado de incorporación y evitar interrumpir repetidamente a los usuarios
3. **El diseño de incorporación debe mantenerse enfocado**: cada paso debe resolver un problema y no más
4. **La estructura del código debe permanecer clara**: configuración de incorporación, administración de estado y lógica de interfaz de usuario separadas para facilitar el mantenimiento

Si está agregando nuevos usuarios a su propio proyecto, espero que la experiencia práctica de este artículo le ayude. No hay nada especialmente místico en este tipo de tecnología. Sigue intentándolo, sigue resumiendo lo que aprendes y las cosas se irán facilitando poco a poco.
