## Cómo implementar el reintento automático para CLI de agentes como Claude Code y Codex

> La frase reintento automático parece un pequeño interruptor, pero una vez que se coloca en un entorno de ingeniería real, no es nada de eso. Hola a todos, soy el creador de HagiCode, Yu Kun. Hoy no quiero negociar con palabras vacías. Quiero hablar sobre cómo se debe realizar el reintento automático para las CLI de agentes como Claude Code y Codex, de modo que pueda recuperarse de excepciones y evitar arrastrar al sistema a ejecuciones repetidas e interminables.

## Antecedentes

Si también has estado trabajando en codificación de IA últimamente, probablemente ya te hayas encontrado con este tipo de problema: la tarea no falla inmediatamente, sino que se interrumpe a mitad de ejecución.

En una solicitud HTTP normal, eso a menudo significa simplemente enviarla de nuevo, tal vez con algún retraso exponencial. Pero las CLI del agente son diferentes. Herramientas como Claude Code y Codex generalmente se ejecutan en forma de transmisión, generando resultados fragmento a fragmento. Durante ese proceso, también pueden vincularse a un hilo, sesión o token de reanudación. En otras palabras, la pregunta no es simplemente: "¿Esta solicitud falló o no?" Se convierte en:

- ¿Sigue contando el contenido que ya fue emitido?
- ¿Puede seguir funcionando el contexto actual?
- ¿Debería recuperarse este fallo automáticamente?
- Si se debe recuperar, ¿cuánto tiempo debemos esperar antes de volver a intentarlo, qué debemos enviar durante el reintento y aún así debemos reutilizar el contexto original?

La primera vez que muchos equipos construyen esta parte, instintivamente escriben la versión más ingenua: si ocurre un error, inténtalo una vez más. Esa idea es perfectamente natural, pero una vez que se convierte en un proyecto real, empiezan a surgir un problema tras otro.

- Algunos errores son claramente fallas temporales, pero se tratan como fallas finales.
- No vale la pena volver a intentar algunos errores, pero el sistema los reproduce una y otra vez.
- Las solicitudes con un hilo y las solicitudes sin un hilo se tratan exactamente igual
- La estrategia de retroceso no tiene límites y las solicitudes en segundo plano se sobrecargan.

Al integrar múltiples CLI de agentes, HagiCode también cayó en estas trampas. Del lado del Codex en particular, el primer problema que expusimos fue que cierto tipo de mensaje de reconexión no se reconocía como un estado de terminal reintentable, por lo que el mecanismo de recuperación que ya teníamos nunca tuvo la oportunidad de surtir efecto. Para decirlo claramente, no es que el sistema careciera de reintento automático. El sistema simplemente no reconoció que valía la pena volver a intentar este fallo en particular.

Entonces, el punto central de este artículo es muy claro: **el reintento automático no es un botón, sino un diseño en capas.**

## Por qué el reintento automático para las CLI del agente es más difícil que el reintento normal

Esta es una pregunta muy práctica, así que vayamos directamente a la conclusión: la dificultad del reintento automático para las CLI del agente no es "intentar de nuevo después de unos segundos", sino "¿puede continuar en el contexto original?".

Puedes considerarlo como una larga conversación. El reintento normal de API se parece más a volver a marcar cuando la línea telefónica está ocupada. El reintento de la CLI del agente se parece más a la caída de la señal mientras la otra parte está a mitad de una frase, y luego usted tiene que decidir si vuelve a llamar, si comienza de nuevo cuando lo hace y si la otra parte aún recuerda dónde se detuvo la conversación. Estos no son en absoluto el mismo tipo de problema de ingeniería.

Más concretamente, hay cuatro dificultades especialmente típicas.

### 1. Está transmitiendo

Una vez que el resultado ya se ha enviado al usuario, ya no puede manejar el error como lo haría con una solicitud normal, donde se lo traga en silencio y vuelve a intentarlo en silencio. Esto se debe a que el contenido anterior ya se vio. Si la estrategia de reproducción es incorrecta, la interfaz puede mostrar fácilmente texto duplicado y un estado inconsistente, y el ciclo de vida de las llamadas a las herramientas también puede complicarse. Esto no es metafísica. Es ingeniería.

### 2. Generalmente se vincula al contexto de la sesión.

Proveedores como Codex se vinculan a un hilo y las implementaciones como Claude Code también tienen un objetivo de continuación o un contexto reanudable equivalente. El verdadero requisito previo para el reintento automático no es sólo que el error parezca una falla temporal, sino también que todavía exista un operador que permita que esta ejecución continúe.

### 3. No vale la pena volver a intentar todos los errores

Por lo general, vale la pena volver a intentarlo con la fluctuación de la red, el tiempo de espera de inactividad de SSE y las fallas temporales ascendentes. Pero si lo que enfrenta es una falla de autenticación, una pérdida de contexto o un proveedor que no tiene ninguna capacidad de reanudación, entonces volver a intentarlo generalmente no es una recuperación. Es generación de ruido.

### 4. Necesita límites

El reintento automático ilimitado casi siempre es incorrecto. Las tendencias tecnológicas pueden resultar ruidosas durante un tiempo, pero las leyes de la ingeniería suelen permanecer estables durante muchos años. Una de ellas es que la recuperación ante fallos debe tener límites. El sistema tiene que saber cuántas veces como máximo puede reintentar, cuánto tiempo debe esperar cada vez y cuándo debe detenerse y admitir que ésta realmente no se va a recuperar.

Debido a estas características, HagiCode finalmente no implementó el reintento automático, ya que unas pocas líneas de `try/catch` dentro de un proveedor específico. En cambio, lo extrajimos a una capa de capacidad compartida. Al final, los problemas de ingeniería aún deben resolverse con métodos de ingeniería.

## Enfoque de HagiCode: reintentar el proceso del proveedor

La implementación actual de HagiCode en el mundo real se puede comprimir en una oración:

**La capa compartida gestiona el flujo de reintento de manera uniforme, y cada proveedor concreto solo es responsable de responder dos preguntas: ¿vale la pena volver a intentar este estado terminal y puede continuar el contexto actual?**

Esto no es complicado, pero es fundamental. Una vez que las responsabilidades se separan de esta manera, Claude Code, Codex e incluso otras CLI de agentes pueden reutilizar el mismo esqueleto. Los modelos cambiarán, las herramientas evolucionarán, los flujos de trabajo se actualizarán, pero la base de la ingeniería seguirá ahí.

### Capa 1: utilice un coordinador unificado para gestionar el ciclo de reintento

El fragmento de implementación principal del proyecto se parece más o menos a esto:

```csharp
internal static class ProviderErrorAutoRetryCoordinator
{
    public static async IAsyncEnumerable<CliMessage> ExecuteAsync(
        string prompt,
        ProviderErrorAutoRetrySettings? settings,
        Func<string, IAsyncEnumerable<CliMessage>> executeAttemptAsync,
        Func<bool> canRetryInSameContext,
        Func<TimeSpan, CancellationToken, Task> delayAsync,
        Func<CliMessage, bool> isRetryableTerminalMessage,
        [EnumeratorCancellation] CancellationToken cancellationToken)
    {
        var normalizedSettings = ProviderErrorAutoRetrySettings.Normalize(settings);
        var retrySchedule = normalizedSettings.Enabled
            ? normalizedSettings.GetRetrySchedule()
            : [];

        for (var attempt = 0; ; attempt++)
        {
            var attemptPrompt = attempt == 0
                ? prompt
                : ProviderErrorAutoRetrySettings.ContinuationPrompt;

            CliMessage? terminalFailure = null;

            await foreach (var message in executeAttemptAsync(attemptPrompt)
                               .WithCancellation(cancellationToken))
            {
                if (isRetryableTerminalMessage(message))
                {
                    terminalFailure = message;
                    break;
                }

                yield return message;
            }

            if (terminalFailure is null)
            {
                yield break;
            }

            if (attempt >= retrySchedule.Count || !canRetryInSameContext())
            {
                yield return terminalFailure;
                yield break;
            }

            await delayAsync(retrySchedule[attempt], cancellationToken);
        }
    }
}
```

Lo que hace este código es realmente muy sencillo, pero también muy eficaz.

- No pase las fallas intermedias directamente al principio; el coordinador decide si la recuperación aún es posible
- Solo cuando se agota el presupuesto de reintento, la falla final realmente regresa a la capa superior.
- A partir del segundo intento, ya no se envía el mensaje original; en su lugar, se envía un mensaje de continuación de manera uniforme

Es por eso que seguí enfatizando antes que el reintento automático no es simplemente "hacer la solicitud nuevamente". No se trata solo de parchear una rama de excepción. Es gestionar el ciclo de vida de una ejecución. Puede parecer un lenguaje de gerente de producto, pero en términos de ingeniería, eso es exactamente lo que es.

### Capa 2: instantánea de la política de reintento

Otra cuestión que es muy fácil pasar por alto es la siguiente: ¿quién decide si se habilita el reintento automático para esta solicitud?

La respuesta de HagiCode es no depender de alguna "configuración global actual", sino convertir la política en una instantánea y dejarla viajar junto con esta solicitud. De esa manera, las colas de sesiones, la persistencia de mensajes, el reenvío de ejecución y la adaptación del proveedor no perderán la política en el camino. Una ejecución exitosa no es un sistema. El éxito sostenido es un sistema.

La estructura central se puede simplificar a esto:

```csharp
public sealed record ProviderErrorAutoRetrySnapshot
{
    public const string DefaultStrategy = "default";

    public bool Enabled { get; init; }

    public string Strategy { get; init; } = DefaultStrategy;

    public static ProviderErrorAutoRetrySnapshot Normalize(bool? enabled, string? strategy)
    {
        return new ProviderErrorAutoRetrySnapshot
        {
            Enabled = enabled ?? true,
            Strategy = string.IsNullOrWhiteSpace(strategy)
                ? DefaultStrategy
                : strategy.Trim()
        };
    }
}
```

Luego, en el lado de la ejecución, se asigna al objeto de configuración realmente consumido por el proveedor. El valor de este enfoque es muy directo:

- La capa empresarial decide si se debe permitir el reintento.
- El tiempo de ejecución decide cómo se debe realizar el reintento.

Cada lado gestiona su propia preocupación sin chocar con el otro. Muchos problemas no son imposibles de resolver. Su costo simplemente no se ha hecho explícito. Convertir la póliza en una instantánea es esencialmente una forma de contabilizar ese costo por adelantado.

### Capa 3: Los proveedores solo deciden el estado terminal y la viabilidad del contexto

Una vez que llegamos al proveedor concreto del Claude Code o Codex, la responsabilidad aquí se vuelve realmente muy escasa. Puedes considerarlo como una mejora, no como un reemplazo.

Tomando Codex como ejemplo, cuando se conecta al coordinador compartido, en realidad solo necesita proporcionar tres cosas:

```csharp
await foreach (var message in ProviderErrorAutoRetryCoordinator.ExecuteAsync(
                   prompt,
                   options.ProviderErrorAutoRetry,
                   retryPrompt => ExecuteCodexAttemptAsync(...),
                   () => !string.IsNullOrWhiteSpace(resolvedThreadId),
                   DelayAsync,
                   IsRetryableTerminalFailure,
                   cancellationToken))
{
    yield return message;
}
```

Notará que las decisiones específicas del proveedor son en realidad solo estas dos:

- `IsRetryableTerminalFailure`
- `canRetryInSameContext`

Codex verifica si el hilo aún puede continuar, mientras que Claude Code verifica si el objetivo de continuación aún existe. La política de retroceso, el recuento de reintentos y las indicaciones de seguimiento no deben ser reinventadas por cada proveedor por separado.

Una vez que se separa esta capa, el costo de integrar más CLI en HagiCode cae mucho. No es necesario duplicar una máquina de estado de reintento completa. Solo necesita ingresar las condiciones límite de ese proveedor. Escribir rápido no es lo mismo que escribir con firmeza. No es lo mismo poder conectar algo que conectarlo bien. Hacerlo funcionar tampoco es lo mismo que hacerlo mantenible en el tiempo.

## Un error fácil de cometer: no trate cada error como si se pudiera volver a intentar

En este análisis, el punto que más quiero destacar no es "cómo implementar el reintento", sino "cómo evitar los reintentos incorrectos".

El punto de entrada original al problema fue que Codex no pudo reconocer un mensaje de reconexión. Por intuición, muchas personas elegirían la solución más pequeña posible: agregar un prefijo de cadena más a la lista blanca. Esa idea no es exactamente incorrecta, pero se siente más como una solución en etapa de demostración que como una solución mantenible a largo plazo.

Desde la implementación actual de HagiCode, el sistema ya ha dado un paso en una dirección más sólida. Ya no mira sólo una cadena literal. En cambio, entrega los estados terminales recuperables al coordinador compartido de manera uniforme. Los beneficios son obvios:

- Es menos probable que falle por completo debido a un pequeño cambio de redacción en un mensaje.
- La cobertura de la prueba se puede construir alrededor del sobre del estado terminal en lugar de una sola línea de texto codificada
- La lógica de reintento se vuelve más consistente dentro del mismo proveedor

Por supuesto, aquí es necesario que haya un límite firme: ser más general no significa ser más permisivo. **Si el contexto actual no puede continuar, incluso si el error parece una falla temporal, no debe reproducirse a ciegas.**

Este punto es crítico. Lo que realmente hace que la gente confíe en un sistema no es que funcione ocasionalmente, sino que sea confiable la mayor parte del tiempo. Si un flujo sólo puede ser mantenido por expertos, todavía queda un largo camino para una adopción real.

## Las tres lecciones más valiosas que se deben poner en práctica

En este punto, tiene sentido comenzar a llevar la discusión nuevamente a la práctica de implementación. Si planea crear una capacidad similar en su propio proyecto, estas son las tres reglas que recomiendo proteger primero.

### 1. El presupuesto de reintento debe tener límites.

El ritmo de retroceso predeterminado actual de HagiCode es:

- 10 segundos
- 20 segundos
- 60 segundos

Puede que este ritmo no se ajuste a todos los sistemas, pero la existencia de límites debe permanecer. De lo contrario, el reintento automático deja rápidamente de ser un mecanismo de recuperación y se convierte en un amplificador de incidencias. No te apresures a darle un nombre impresionante. Primero, asegúrese de que el objeto pueda sobrevivir a dos iteraciones dentro de un equipo real.

### 2. El mensaje de continuación debe estar unificado.

El proyecto utiliza un mensaje de continuación fijo para que los intentos posteriores sigan claramente el camino de continuar con el contexto actual en lugar de iniciar una solicitud completa nueva. Esta capacidad no es llamativa, pero cuando construyes un proyecto real, no puedes prescindir de ella. Muchas cosas que parecen mágicas, una vez descompuestas, son sólo un pulido proceso de ingeniería.

### 3. Tanto la biblioteca compartida como la capa del adaptador necesitan pruebas reflejadas.

Especialmente quiero decir un poco más sobre este punto. Muchos equipos escribirán una capa de pruebas en el tiempo de ejecución compartido y pensarán que probablemente sea suficiente. No lo es.

La razón por la que me siento relativamente seguro acerca de la implementación de HagiCode es que ambas capas tienen cobertura de prueba:

- El proveedor compartido comprueba si realmente se produjo la continuación automática
- La capa del adaptador prueba si los errores finales y los mensajes de transmisión se conservaron correctamente

Esta vez también volví a ejecutar dos grupos de prueba relacionados y los 31 casos de prueba pasaron en ambos. Ese resultado por sí solo no prueba que el diseño sea perfecto, pero prueba al menos una cosa: el reintento automático actual no es un diseño en papel. Es una capacidad restringida tanto por el código como por las pruebas. Hablar es barato. Muéstrame el código. Encaja perfectamente aquí.

## Resumen

Si hubiera que resumir todo el artículo en una sola frase, sería esta:

**Para CLI de agentes como Claude Code y Codex, el reintento automático no debe implementarse como un truco local oculto dentro de un proveedor. Debería construirse como una combinación de un coordinador compartido, una instantánea de políticas, comprobaciones de viabilidad del contexto y pruebas reflejadas.**

Los beneficios de hacerlo de esta forma son muy prácticos:

- La lógica se escribe una vez y se reutiliza en varios proveedores.
- Si se permite que una solicitud se reintente puede viajar de manera estable con la cadena de ejecución
- Continuar corriendo cuando exista el contexto y detenerse en el tiempo cuando no exista.
- Lo que finalmente ve la interfaz es un estado estable completado o un estado fallido, no un montón de ruido intermedio abandonado.

Esta solución se fue puliendo poco a poco mientras HagiCode integraba múltiples CLI de agentes en escenarios reales. ¿Quién dice que la programación asistida por IA no es la nueva era de la programación en pareja? Los modelos lo ayudan a comenzar, completar el código y expandirse, pero lo que a menudo determina el límite superior de la experiencia sigue siendo el contexto, el proceso y las limitaciones.
