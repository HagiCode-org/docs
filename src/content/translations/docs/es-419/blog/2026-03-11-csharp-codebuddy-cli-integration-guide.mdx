## Guía práctica para integrar CodeBuddy CLI en un backend de C#

> Este artículo explica un enfoque completo para integrar CodeBuddy CLI en un proyecto backend de C# para que pueda ofrecer capacidades de asistente de codificación de IA de principio a fin.

El proyecto había implementado previamente la integración de CodeBuddy, pero el código relacionado se eliminó durante una refactorización. No hay mucho de qué quejarse allí; Durante el desarrollo iterativo, siempre queda algo atrás. Si también está considerando conectar varios asistentes de codificación de IA a su propio proyecto, el siguiente enfoque puede brindarle algunas ideas. Refleja las lecciones que resumimos después de haber tropezado con muchos obstáculos y tal vez pueda ayudarle a evitar algunos desvíos.

## Diseño de Arquitectura

### Descripción general de la arquitectura en capas

La integración entre C# y CodeBuddy utiliza una arquitectura en capas clara. Este diseño hace explícitas las responsabilidades y facilita mucho el mantenimiento a largo plazo:

```
┌─────────────────────────────────────────────┐
│           Provider Contract Layer           │
│   AIProviderType enum + extension methods   │
├─────────────────────────────────────────────┤
│           Provider Factory Layer            │
│   AIProviderFactory dependency injection factory │
├─────────────────────────────────────────────┤
│           Provider Implementation Layer     │
│   CodebuddyCliProvider concrete implementation │
├─────────────────────────────────────────────┤
│           ACP Infrastructure Layer          │
│  ACPSessionManager / StdioAcpTransport      │
│  AcpRpcClient / AcpAgentClient              │
└─────────────────────────────────────────────┘
```

¿Cuáles son los beneficios de esta estratificación? En pocas palabras, cada capa permanece fuera del camino de las demás. Si luego queremos cambiar el mecanismo de comunicación, por ejemplo de `stdio` para WebSocket, solo necesitamos modificar la capa inferior y la lógica empresarial superior permanece intacta. Nadie quiere que un cambio en la comunicación se extienda a todo el código base.

### Desglose de componentes principales

La **capa de contrato del proveedor** es la base de toda la arquitectura. Definimos el `AIProviderType` enumeración, donde `CodebuddyCli = 3` se utiliza como valor de enumeración e implementa el mapeo bidireccional entre cadenas y enumeraciones a través de métodos de extensión. Eso permite que las cadenas en los archivos de configuración se conviertan convenientemente en enumeraciones y que las enumeraciones se vuelvan a convertir en cadenas para la salida de depuración.

La **capa de fábrica de proveedores** es responsable de crear la instancia de proveedor correspondiente según la configuración. Utiliza inyección de dependencia .NET junto con `ActivatorUtilities.CreateInstance` para la creación dinámica. La ventaja del patrón de fábrica es que al agregar un nuevo proveedor, solo necesita agregar la lógica de creación en lugar de modificar el código existente.

La **capa de implementación del proveedor** es donde ocurre el trabajo real. `CodebuddyCliProvider` implementa el `IAIProvider` interfaz y proporciona dos modos de invocación: `ExecuteAsync` para llamadas sin transmisión y `StreamAsync` para llamadas en streaming.

La **capa de infraestructura ACP** proporciona la base de comunicación subyacente. Esta capa maneja todos los detalles del protocolo, incluida la gestión de procesos, la serialización de mensajes y el análisis de respuestas. Es la base que mantiene estable todo lo que está encima.

## Mecanismo de comunicación

### Modo de transporte estándar

CodeBuddy utiliza **Stdio (entrada/salida estándar)** para comunicarse con procesos externos. El comando de inicio es simple:

```bash
codebuddy --acp
```

Después de eso, los mensajes JSON-RPC se intercambian a través de entrada y salida estándar. Este enfoque tiene varias ventajas:

1. **Inicio rápido**: la comunicación del proceso local evita la latencia de la red
2. **Configuración sencilla**: sólo necesitas especificar la ruta del ejecutable
3. **Aislamiento del entorno**: cada sesión se ejecuta en un proceso independiente, por lo que no se afectan entre sí.

La inyección de variables de entorno se admite durante la comunicación. Los ejemplos comunes incluyen:

- `CODEBUDDY_API_KEY`: autenticación de clave API
- `CODEBUDDY_INTERNET_ENVIRONMENT`: configuración del entorno de red

Al igual que con la comunicación entre personas, es útil elegir primero un canal conveniente.

### Protocolo de mensajes

ACP se basa en JSON-RPC 2.0. El formato del mensaje se parece más o menos a esto:

```json
// Request message
{
  "jsonrpc": "2.0",
  "id": 1,
  "method": "agent/prompt",
  "params": {
    "prompt": "Help me write a sorting algorithm",
    "sessionId": "session-123"
  }
}

// Response message
{
  "jsonrpc": "2.0",
  "id": 1,
  "result": {
    "content": "Here is the AI response..."
  }
}
```

En la implementación real, encapsulamos todos estos detalles del protocolo para que la capa empresarial superior solo deba preocuparse por el aviso y la respuesta.

### 2. Integrar la fábrica de proveedores

Agregue una rama de creación de CodeBuddy en la clase de fábrica:

```csharp
// AIProviderFactory.cs
private IAIProvider? CreateProvider(AIProviderType providerType, ProviderConfiguration config)
{
    return providerType switch
    {
        AIProviderType.CodebuddyCli =>
            ActivatorUtilities.CreateInstance<CodebuddyCliProvider>(
                _serviceProvider,
                Options.Create(config)),
        // ... Other providers
        _ => throw new NotSupportedException($"Provider {providerType} not supported")
    };
}
```

Esto utiliza la inyección de dependencia a través de `ActivatorUtilities`, Maneja automáticamente la inyección de parámetros del constructor y es muy conveniente.

### 3. Implementación completa del proveedor

A continuación se muestra la implementación principal de `CodebuddyCliProvider`, que cubre los modos de invocación de transmisión y no transmisión:

```csharp
public class CodebuddyCliProvider : IAIProvider
{
    private readonly ILogger<CodebuddyCliProvider> _logger;
    private readonly IACPSessionManager _sessionManager;
    private readonly ProviderConfiguration _config;

    public string Name => "CodebuddyCli";
    public bool SupportsStreaming => true;
    public ProviderCapabilities Capabilities { get; }

    public CodebuddyCliProvider(
        ILogger<CodebuddyCliProvider> logger,
        IACPSessionManager sessionManager,
        IOptions<ProviderConfiguration> config)
    {
        _logger = logger;
        _sessionManager = sessionManager;
        _config = config.Value;

        // Define the capabilities of the current Provider
        Capabilities = new ProviderCapabilities
        {
            SupportsStreaming = true,
            SupportsTools = true,
            SupportsSystemMessages = true,
            SupportsArtifacts = false,
            MaxTokens = 8192
        };
    }

    // Non-streaming call: return all results together after completion
    public async Task<AIResponse> ExecuteAsync(
        AIRequest request,
        CancellationToken cancellationToken = default)
    {
        // Create an independent session for the request
        var session = await _sessionManager.CreateSessionAsync(
            "CodebuddyCli",
            request.WorkingDirectory,
            cancellationToken,
            request.SessionId);

        try
        {
            var fullPrompt = BuildPrompt(request);
            await session.SendPromptAsync(fullPrompt, cancellationToken);

            var responseBuilder = new StringBuilder();
            var toolCalls = new List<AIToolCall>();

            // Collect all response chunks
            await foreach (var chunk in StreamFromSession(session, cancellationToken))
            {
                if (!string.IsNullOrEmpty(chunk.Content))
                {
                    responseBuilder.Append(chunk.Content);
                }
                // Handle tool calls...
            }

            return new AIResponse
            {
                Content = AIResultContentSanitizer.SanitizeResultContent(
                    responseBuilder.ToString()),
                ToolCalls = toolCalls,
                Provider = Name,
                Model = string.Empty
            };
        }
        finally
        {
            // Release session resources
            await session.DisposeAsync();
        }
    }

    // Streaming call: return response chunks in real time
    public async IAsyncEnumerable<AIStreamingChunk> StreamAsync(
        AIRequest request,
        [EnumeratorCancellation] CancellationToken cancellationToken = default)
    {
        var session = await _sessionManager.CreateSessionAsync(
            "CodebuddyCli",
            request.WorkingDirectory,
            cancellationToken);

        try
        {
            var fullPrompt = BuildPrompt(request);
            await session.SendPromptAsync(fullPrompt, cancellationToken);

            await foreach (var chunk in StreamFromSession(session, cancellationToken))
            {
                yield return chunk;
            }
        }
        finally
        {
            await session.DisposeAsync();
        }
    }

    private async IAsyncEnumerable<AIStreamingChunk> StreamFromSession(
        IACPSession session,
        [EnumeratorCancellation] CancellationToken cancellationToken)
    {
        // Iterate through all updates in the session
        await foreach (var notification in session.ReceiveUpdatesAsync(cancellationToken))
        {
            switch (notification.Update)
            {
                case AgentMessageChunkSessionUpdate agentMessage:
                    // Handle text content chunks
                    if (agentMessage.Content is AcpImp.TextContentBlock textContent)
                    {
                        yield return new AIStreamingChunk
                        {
                            Content = textContent.Text,
                            Type = StreamingChunkType.ContentDelta,
                            IsComplete = false
                        };
                    }
                    break;

                case ToolCallSessionUpdate toolCall:
                    // Handle tool calls
                    yield return new AIStreamingChunk
                    {
                        Content = string.Empty,
                        Type = StreamingChunkType.ToolCallDelta,
                        ToolCallDelta = new AIToolCallDelta
                        {
                            Id = toolCall.ToolCallId,
                            Name = toolCall.Kind.ToString(),
                            Arguments = toolCall.RawInput?.ToString()
                        }
                    };
                    break;

                case AcpImp.PromptCompletedSessionUpdate:
                    // Response complete
                    yield break;
            }
        }
    }

    // Build the full prompt
    private string BuildPrompt(AIRequest request, string? embeddedCommandPrompt = null)
    {
        var sb = new StringBuilder();

        // Embedded command prompt, if present
        if (!string.IsNullOrEmpty(embeddedCommandPrompt))
        {
            sb.AppendLine(embeddedCommandPrompt);
            sb.AppendLine();
        }

        // System message
        if (!string.IsNullOrEmpty(request.SystemMessage))
        {
            sb.AppendLine(request.SystemMessage);
            sb.AppendLine();
        }

        // User prompt
        sb.Append(request.Prompt);
        return sb.ToString();
    }
}
```

Hay varios puntos clave en este código:

1. **Gestión de sesiones**: cada solicitud crea una sesión independiente y libera recursos una vez completada la solicitud. Esta es una lección aprendida mediante prueba y error. Si la reutilización de sesiones no se maneja bien, la contaminación estatal aparece fácilmente.

2. **Procesamiento de transmisión**: `IAsyncEnumerable` permite que la respuesta se devuelva mientras aún se está generando, en lugar de esperar a que finalice todo el contenido. Esto es especialmente importante para escenarios de texto largo y mejora significativamente la experiencia del usuario.

3. **Llamadas a herramientas**: CodeBuddy admite llamadas a herramientas (llamadas a funciones), manejadas a través de `ToolCallSessionUpdate`. Esta capacidad es fundamental para tareas complejas de edición de código.

4. **Filtrado de contenidos**: `AIResultContentSanitizer` se utiliza para filtrar el contenido del bloque Think y mantener limpia la salida.

public void ConfigureModule(IServiceCollection context)

## Ejemplo de configuración

### Archivo de configuración

Agregue la configuración relacionada con CodeBuddy a `appsettings.json`:

```yaml
AI:
  # Default Provider to use
  DefaultProvider: "CodebuddyCli"

  # Provider configuration
  Providers:
    CodebuddyCli:
      Type: "CodebuddyCli"
      WorkingDirectory: "C:/projects/my-app"
      ExecutablePath: "C:/tools/codebuddy.cmd"

  # Platform-specific configuration
  PlatformConfigurations:
    CodebuddyCli:
      ExecutablePath: "C:/tools/codebuddy.cmd"
      Arguments: "--acp"
      StartupTimeoutMs: 5000
      EnvironmentVariables:
        CODEBUDDY_API_KEY: "${CODEBUDDY_API_KEY}"
        CODEBUDDY_INTERNET_ENVIRONMENT: "production"
```

## Lecciones prácticas aprendidas

### Registro de errores

Nos topamos con varios errores típicos durante la implementación y compartirlos aquí puede ayudar a otros a evitar los mismos desvíos:

1. **Problema de pérdida de sesión**: al principio, las sesiones no se liberaban correctamente, lo que agotaba los recursos del proceso. La solución fue utilizar `try-finally` para garantizar que se liberen recursos para cada solicitud.

2. **Paso de variables de entorno**: Windows y Linux utilizan una sintaxis de variables de entorno diferente, por lo que luego estandarizamos `Dictionary<string, string?>` para manejar esto.

3. **Configuración del tiempo de espera**: el inicio de la CLI lleva tiempo, por lo que configuramos un tiempo de espera de inicio de 5 segundos para evitar fallas rápidas en las solicitudes.

4. **Problemas de codificación**: en Windows, la codificación predeterminada puede causar texto chino confuso, por lo que la codificación UTF-8 se especifica explícitamente al iniciar el proceso.

### Optimización del rendimiento

1. **Grupo de sesiones**: para solicitudes breves frecuentes, considere implementar un grupo de sesiones para reutilizar procesos
2. **Caché de conexión**: la clase de fábrica ya admite el almacenamiento en caché de instancias de proveedor
3. **Async first**: use programación asincrónica en todo momento para evitar bloquear subprocesos

Siempre vale la pena optimizar el rendimiento. Cuanto más esperen los usuarios, peor será la experiencia.

## Conclusión

Este artículo presentó una solución completa para integrar CodeBuddy CLI en un backend de C#, cubriendo todo el proceso desde el diseño de la arquitectura hasta la implementación concreta. A través de una arquitectura en capas, separamos los detalles del protocolo de la lógica empresarial, lo que hace que el código sea más claro y fácil de mantener.

Conclusiones clave:

- Utilice una arquitectura en capas con una capa de contrato de proveedor, una capa de fábrica, una capa de implementación y una capa de infraestructura.
- Utilice JSON-RPC sobre Stdio para la comunicación entre procesos
- Implemente una configuración flexible y extensibilidad mediante la inyección de dependencias.
- Proporcionar modos de invocación tanto de streaming como de no streaming

Este enfoque no sólo es adecuado para CodeBuddy; agregar nuevos proveedores de IA puede seguir el mismo patrón. Si también está creando una integración similar de múltiples proveedores de IA, espero que este artículo le brinde una referencia útil.
