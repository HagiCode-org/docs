## Diseño de herramientas CLI de ImgBin: enfoque de gestión de activos de imágenes de HagiCode

> Este artículo explica cómo crear una canalización de recursos de imágenes automatizable desde cero, y cubre el diseño de herramientas CLI, una arquitectura de adaptador de proveedor y estrategias de administración de metadatos.

## Antecedentes

Honestamente, no esperaba que la gestión de recursos de imágenes nos mantuviera enredados por tanto tiempo.

Durante el desarrollo de HagiCode, nos topamos con un problema que parecía simple en la superficie pero que era sorprendentemente complicado en la práctica: generar y administrar activos de imágenes. En cierto modo, era como los dramas de la adolescencia: calma por fuera, turbulencia por dentro.

A medida que el proyecto acumulaba más documentación y materiales de marketing, necesitábamos una gran cantidad de imágenes de apoyo. Algunos tenían que ser generados por IA, otros debían seleccionarse de una biblioteca de activos existente y otros necesitaban reconocimiento de IA y etiquetado automático. El problema era que todo esto se había manejado durante mucho tiempo mediante scripts dispersos y pasos manuales. Cada vez que generábamos una imagen, teníamos que ejecutar un script a mano, organizar los metadatos a mano y crear miniaturas a mano. Solo eso ya era bastante molesto, pero el problema más grande era que todo estaba esparcido por todas partes. Cuando quisimos encontrar algo, no pudimos. Cuando necesitábamos reutilizar algo, no podíamos.

Los puntos débiles eran concretos:

1. **Sin punto de entrada unificado**: la lógica para la generación de imágenes se distribuía entre diferentes scripts, por lo que la ejecución por lotes era básicamente imposible.
2. **Faltan metadatos**: las imágenes generadas no tenían unificado `metadata.json`, lo que significaba que no había posibilidades de búsqueda ni trazabilidad fiables.
3. **Alto costo de organización manual**: los títulos y etiquetas debían ordenarse uno por uno a mano, lo cual era ineficiente.
4. **Sin automatización**: ¿generar automáticamente activos visuales en un proceso de CI/CD? No es una posibilidad.

Pensamos en dejarlo así. Pero los proyectos todavía necesitan avanzar. Como no podíamos evitar el problema, pensamos que también podríamos resolverlo. Entonces decidimos actualizar ImgBin desde un conjunto de scripts dispersos a una canalización de activos de imágenes que se puede ejecutar automáticamente. Después de todo, algunos problemas no desaparecen sólo porque miras hacia otro lado.

## Acerca de HagiCode

El enfoque compartido en este artículo proviene de nuestra experiencia práctica en el [HagiCode](https://hagicode.com) proyecto. HagiCode es un proyecto de asistente de codificación de IA que mantiene simultáneamente múltiples componentes, incluida una extensión VSCode, servicios backend de IA y un cliente de escritorio multiplataforma. En un entorno complejo, multilingüe y multiplataforma como este, la gestión estandarizada de activos de imágenes se convierte en una parte clave para mejorar la eficiencia del desarrollo.

Se podría decir que este fue uno de esos pequeños problemas de crecimiento en el viaje de HagiCode. Cada proyecto tiene momentos como ese: un problema menor que parece insignificante, pero que de alguna manera logra ocupar la mitad del día.

El sistema de compilación de HagiCode se basa en el ecosistema TypeScript + Node.js, por lo que ImgBin naturalmente adoptó la misma pila tecnológica para mantener el proyecto técnicamente consistente. Una vez que estás acostumbrado a una pila, cambiar a otra cosa parece un problema innecesario.

## Diseño central

### Arquitectura general

ImgBin utiliza una arquitectura en capas que separa claramente los comandos CLI, los servicios de aplicaciones, los adaptadores API de terceros y la capa de infraestructura:

```
Component hierarchy
├── CLI Entry (cli.ts)              Global argument parsing, command routing
├── Commands (commands/*)           generate | batch | annotate | thumbnail
├── Application Services            job-runner | metadata | thumbnail | asset-writer
├── Provider Adapters               image-api-provider | vision-api-provider
└── Infrastructure Layer            config | logger | paths | schema
```

El beneficio de este diseño en capas son los límites de responsabilidad claros. También facilita las pruebas porque las dependencias externas se pueden burlar limpiamente. En la práctica, simplemente significa que cada capa hace su propio trabajo sin interponerse en el camino de las demás, de modo que cuando algo se rompe, es más fácil descubrir por qué.

### Modelo de directorio de un solo activo

ImgBin utiliza un modelo de "un activo, un directorio". Cada vez que se genera una imagen, crea una estructura como esta:

```
library/
└── 2026-03/
    └── orange-dashboard/
        ├── original.png      # Original image
        ├── thumbnail.webp    # 512x512 thumbnail
        └── metadata.json     # Structured metadata
```

Las ventajas de este modelo son:

1. **Autocontenido**: todos los archivos de un único activo se encuentran en el mismo directorio, lo que hace que la migración y la copia de seguridad sean convenientes.
2. **Rastreable**: `metadata.json` hace posible rastrear el tiempo de generación, el aviso, el modelo y otros detalles.
3. **Extensible**: si más adelante se necesitan más variantes, como miniaturas en varios tamaños, simplemente podemos agregar nuevos archivos en el mismo directorio.

No siempre es necesario poseer las cosas bellas. A veces basta con que sigan siendo bonitos y que puedas apreciarlos tranquilamente. Esto puede parecer un poco lejano, pero la lógica sigue siendo válida: una vez que las imágenes se mantienen juntas, son más agradables de ver y mucho más fáciles de encontrar.

### Almacenamiento de metadatos en capas

`metadata.json` es el núcleo de todo el sistema. Utiliza una estrategia de almacenamiento en capas que separa los campos en tres categorías:

```json
{
  "schemaVersion": 2,
  "assetId": "orange-dashboard",
  "slug": "orange-dashboard",
  "title": "Orange Dashboard",
  "tags": ["dashboard", "hero", "orange"],

  "source": { "type": "generated" },

  "paths": {
    "assetDir": "library/2026-03/orange-dashboard",
    "original": "original.png",
    "thumbnail": "thumbnail.webp"
  },

  "generated": {
    "prompt": "orange dashboard for docs hero",
    "provider": "azure-openai-image-api",
    "model": "gpt-image-1.5"
  },

  "recognized": {
    "title": "Orange Dashboard",
    "tags": ["dashboard", "ui", "orange"],
    "description": "A modern orange dashboard with charts and metrics"
  },

  "status": {
    "generation": "succeeded",
    "recognition": "succeeded",
    "thumbnail": "succeeded"
  },

  "timestamps": {
    "createdAt": "2026-03-11T04:01:19.570Z",
    "updatedAt": "2026-03-11T04:02:09.132Z"
  }
}
```

- **generado**: registra la información original de la generación de imágenes, como el mensaje, el proveedor y el modelo.
- **reconocido**: almacena resultados de reconocimiento de IA, como títulos, etiquetas y descripciones generados automáticamente.
- **manual**: almacena resultados seleccionados manualmente. Los datos de esta área tienen la máxima prioridad y el reconocimiento de IA no los sobrescribirá.

Esta estrategia en capas resuelve uno de nuestros conflictos centrales anteriores: cuando el reconocimiento de IA y la curación manual no están de acuerdo, ¿cuál debería ganar? La respuesta es la entrada manual. El reconocimiento de IA está ahí para ayudar, no para decidir. Esa pregunta también se volvió más clara con el tiempo: las máquinas siguen siendo máquinas y, al final, las personas todavía necesitan tomar la decisión.

## Patrón de adaptador de proveedor

Otra parte central de ImgBin es el patrón Adaptador de proveedor. Abstraemos las API externas detrás de una interfaz unificada, de modo que incluso si cambiamos de proveedor de servicios de IA, no necesitamos cambiar la lógica empresarial.

En cierto modo, es un poco como las relaciones: las apariencias externas pueden cambiar, pero lo que importa es que la estructura interna siga siendo la misma. Una vez fijada la interfaz, la implementación interna puede variar libremente.

### Proveedor de generación de imágenes

```typescript
interface ImageGenerationProvider {
  // Generate an image and return its Buffer
  generate(options: GenerateOptions): Promise<Buffer>;

  // Get the list of supported models
  getSupportedModels(): Promise<string[]>;
}

interface GenerateOptions {
  prompt: string;
  model?: string;
  size?: '1024x1024' | '1792x1024' | '1024x1792';
  quality?: 'standard' | 'hd';
  format?: 'png' | 'webp' | 'jpeg';
}
```

### Proveedor de reconocimiento de visión

```typescript
interface VisionRecognitionProvider {
  // Recognize image content and return structured metadata
  recognize(imageBuffer: Buffer): Promise<RecognitionResult>;

  // Get the list of supported models
  getSupportedModels(): Promise<string[]>;
}

interface RecognitionResult {
  title?: string;
  tags: string[];
  description?: string;
  confidence: number;
}
```

Las ventajas de este diseño de interfaz son:

1. **Comprobable**: en las pruebas unitarias, podemos pasar proveedores simulados en lugar de realizar llamadas API externas reales.
2. **Extensible**: agregar un nuevo proveedor solo requiere implementar la interfaz; No es necesario cambiar el código de la persona que llama.
3. **Reemplazable**: la producción puede usar Azure OpenAI, mientras que las pruebas pueden usar un modelo local, siendo la configuración lo único que cambia.

A veces, el trabajo en proyectos también se siente así. A primera vista, parece que acabamos de intercambiar una API, pero la lógica interna sigue siendo exactamente la misma, y ​​eso hace que todo sea mucho menos aterrador.

## Diseño de comandos CLI

ImgBin proporciona cuatro comandos principales para cubrir diferentes escenarios de uso:

### generar: generación de una sola imagen

```bash
# Simplest usage
imgbin generate --prompt "orange dashboard for docs hero"

# Generate a thumbnail and AI annotations at the same time
imgbin generate --prompt "orange dashboard" --annotate --thumbnail

# Specify an output directory
imgbin generate --prompt "orange dashboard" --output ./library
```

### lote: trabajos por lotes

Los trabajos por lotes se definen mediante archivos de manifiesto YAML o JSON, lo que los hace adecuados para flujos de trabajo CI/CD:

```yaml
# assets/jobs/launch.yaml
defaults:
  annotate: true
  thumbnail: true
  libraryRoot: ./library

jobs:
  - prompt: "orange dashboard hero"
    slug: orange-dashboard
    tags: [dashboard, hero, orange]

  - prompt: "pricing grid for docs"
    slug: pricing-grid
    tags: [pricing, grid, docs]
```

Ejecute el comando:

```bash
imgbin batch assets/jobs/launch.yaml
```

El diseño del trabajo por lotes admite el aislamiento de fallas: los elementos del manifiesto se procesan uno por uno y una falla en un elemento no afecta a los demás. También puede obtener una vista previa del trabajo con `--dry-run` sin llegar a ejecutarlo.

Y la mejor parte es que te dice exactamente qué tuvo éxito y qué fracasó. A diferencia de algunas cosas en la vida, donde ocurre el fracaso y ni siquiera sabes cómo sucedió.

### anotar: anotación AI

Ejecute el reconocimiento de IA en imágenes existentes para generar automáticamente títulos, etiquetas y descripciones:

```bash
# Annotate a single image
imgbin annotate ./library/2026-03/orange-dashboard

# Annotate an entire directory in batch
imgbin annotate ./library/2026-03/
```

### miniatura: generación de miniaturas

Genere miniaturas para imágenes existentes:

```bash
# Generate a thumbnail
imgbin thumbnail ./library/2026-03/orange-dashboard
```

## Diseño de manifiesto de trabajo por lotes

El formato de manifiesto para trabajos por lotes admite una configuración flexible. Los valores predeterminados se pueden establecer globalmente y los trabajos individuales pueden anularlos:

```yaml
# Global defaults
defaults:
  annotate: true        # Enable AI annotation by default
  thumbnail: true       # Generate thumbnails by default
  libraryRoot: ./library
  model: gpt-image-1.5

jobs:
  # Minimal configuration: only provide a prompt
  - prompt: "first image"

  # Full configuration
  - prompt: "second image"
    slug: custom-slug
    tags: [tag1, tag2]
    annotate: false     # Do not run AI annotation for this job
    model: dall-e-3    # Use a different model for this job
```

Cuando se ejecuta, ImgBin procesa los trabajos uno por uno. El resultado de cada trabajo se escribe en su correspondiente `metadata.json`. Incluso si un trabajo fracasa, los demás no se ven afectados. Una vez completados todos los trabajos, la CLI genera un informe resumido:

```
✓ orange-dashboard (succeeded)
✓ pricing-grid (succeeded)
✗ hero-banner (failed: API rate limit exceeded)

2/3 succeeded, 1 failed
```

Algunas cosas no se pueden apresurar. Tomarlos uno por uno suele ser el camino más firme. Quizás esa sea la filosofía detrás de los trabajos por lotes.

## Configuración de variables de entorno

ImgBin admite una configuración flexible a través de variables de entorno:

```bash
# ImgBin working directory
IMGBIN_WORKDIR=/path/to/imgbin

# Executable path (for invocation inside scripts)
IMGBIN_EXECUTABLE=/path/to/imgbin/dist/cli.js

# Asset library root
IMGBIN_LIBRARY_ROOT=./.imgbin-library

# Azure OpenAI configuration (if using the Azure provider)
AZURE_OPENAI_ENDPOINT=https://your-resource.openai.azure.com/
AZURE_OPENAI_API_KEY=***
AZURE_OPENAI_IMAGE_DEPLOYMENT=gpt-image-1
```

La configuración es una de esas cosas que pueden parecer importantes y no tan importantes al mismo tiempo. Al final, lo que le resulte más cómodo y se adapte mejor a su flujo de trabajo suele ser la elección correcta.

## Notas de implementación

Durante la implementación, resumimos algunos puntos clave:

### Diseño de interfaz de proveedor

Las definiciones de la interfaz deben ser claras y completas, incluidos los parámetros de entrada, los valores de retorno y el manejo de errores. También es una buena idea proporcionar estilos de invocación tanto sincrónicos como asincrónicos para diferentes escenarios.

Ésa es una pequeña parte de la experiencia ganada con tanto esfuerzo. Una vez configurada una interfaz, nadie quiere seguir cambiándola más tarde.

### Estrategia de manejo de fallas

Cuando un elemento falla en un trabajo por lotes, la CLI debería:

1. Escriba información detallada del error en un archivo de registro independiente.
2. Continúe ejecutando otros trabajos en lugar de interrumpir todo el proceso.
3. Devuelve un código de salida distinto de cero al final para indicar que algunos trabajos fallaron.
4. Muestre claramente el resultado de la ejecución de cada trabajo en el informe resumido.

Algunos fracasos son sólo fracasos. No tiene sentido pretender lo contrario. Es mejor reconocerlos abiertamente y luego descubrir cómo solucionarlos. La misma lógica se aplica a los proyectos y a la vida.

### Estrategia de fusión de metadatos

Los resultados del reconocimiento se escriben en el `recognized` sección de forma predeterminada, mientras que los campos editados manualmente están marcados en `manual`. Las actualizaciones de metadatos siguen una estrategia de solo agregar: a menos que `--force` se pasa explícitamente, los resultados existentes seleccionados manualmente no se sobrescriben.

Ese punto también quedó claro: algunas cosas, una vez sobrescritas, simplemente desaparecen. A menudo es mejor preservarlos, porque el registro en sí tiene valor.

### Atomicidad de la creación de directorios

uso `fs.mkdir({ recursive: true })` para garantizar que la creación de directorios siga siendo atómica y para evitar condiciones de carrera en escenarios concurrentes.

Quizás así es como se siente la seguridad: ser estable cuando la estabilidad importa, moverse rápido cuando la velocidad importa y nunca quedarse atascado dudando.

## Conclusión

Como herramienta principal para la gestión de activos de imágenes en el proyecto HagiCode, ImgBin resuelve nuestros problemas a través de las siguientes opciones de diseño:

1. **Punto de entrada unificado**: la CLI cubre la generación, anotación, miniaturas y todas las demás operaciones principales.
2. **Basado en metadatos**: cada activo tiene una información completa `metadata.json`, permitiendo la búsqueda y trazabilidad.
3. **Adaptador de proveedor**: abstracción flexible para API externas, lo que facilita las pruebas y la extensión.
4. **Compatibilidad con trabajos por lotes**: la generación de imágenes por lotes se puede automatizar dentro de los flujos de trabajo de CI/CD.

Es posible que todo lo demás se haya desvanecido, pero este enfoque realmente terminó resultando útil.

Esta solución no solo mejora la eficiencia del desarrollo propio de HagiCode, sino que también forma un marco reutilizable para la gestión de activos de imágenes. Si está creando un proyecto similar de múltiples componentes, creo que las ideas de diseño de ImgBin pueden brindarle algo de inspiración.

La juventud se trata de probar cosas y hacer un poco de desastre. Si nunca te sometes a eso, ¿cómo sabrás de lo que eres realmente capaz?
