## Guía para implementar la compatibilidad con palabras activas para el reconocimiento de voz de Doubao

> Este artículo explica en detalle cómo implementar el soporte de palabras activas para el reconocimiento de voz de Doubao en el proyecto HagiCode. Al utilizar tanto palabras activas personalizadas como tablas de palabras activas de la plataforma, puede mejorar significativamente la precisión del reconocimiento del vocabulario específico de un dominio.

## Antecedentes

La tecnología de reconocimiento de voz se ha desarrollado durante muchos años, pero un problema ha molestado constantemente a los desarrolladores. Los modelos de reconocimiento de voz de uso general pueden cubrir el lenguaje cotidiano, pero a menudo se quedan cortos cuando se trata de terminología profesional, nombres de productos y nombres personales. Piénselo: un asistente de voz en el campo médico necesita reconocer con precisión términos como "hipertensión", "diabetes" y "enfermedad coronaria"; un sistema legal necesita captar con precisión términos como "causa de acción", "defensa" y "carga de la prueba". En estos escenarios, un modelo de propósito general hace todo lo posible, pero a menudo eso no es suficiente.

Nos encontramos con el mismo desafío en el proyecto HagiCode. Como asistente de codificación de IA multifuncional, HagiCode necesita manejar el reconocimiento de voz para una amplia gama de terminología técnica. Sin embargo, la API de reconocimiento de voz de Doubao, en su configuración predeterminada, no pudo cumplir plenamente con nuestros requisitos de precisión para términos especializados. No es que Doubao no sea lo suficientemente bueno; más bien, cada dominio tiene su propio sistema terminológico. Después de un poco de investigación y exploración técnica, descubrimos que la API de reconocimiento de voz de Doubao en realidad proporciona soporte para palabras activas. Con una configuración sencilla, puede mejorar significativamente la precisión del reconocimiento de vocabulario específico. En cierto sentido, una vez que le dices a qué palabras prestar atención, las escucha con más atención.

Lo que comparte este artículo es la solución completa que utilizamos en el proyecto HagiCode para implementar palabras clave de reconocimiento de voz de Doubao. Ambos modos, palabras activas personalizadas y tablas de palabras activas de plataforma, están disponibles y también se pueden combinar. Con esta solución, los desarrolladores pueden configurar de manera flexible palabras activas basadas en escenarios comerciales para que el sistema de reconocimiento de voz pueda "reconocer" mejor vocabulario profesional, poco común pero crítico.

## Implementación central

### Comprender los dos modos de palabras activas

La API de reconocimiento de voz de Doubao proporciona dos formas de configurar palabras activas y cada una tiene sus propios casos de uso y ventajas ideales.

**Modo de palabra activa personalizada** nos permite pasar texto de palabra activa directamente a través del `corpus.context` campo. Este enfoque es especialmente adecuado para escenarios en los que necesita configurar rápidamente una pequeña cantidad de palabras activas, como reconocer temporalmente el nombre de un producto o el nombre de una persona. En la implementación de HagiCode, analizamos el texto de palabras activas de varias líneas ingresado por el usuario en una lista de cadenas y luego lo formateamos en el formato `context_data` matriz requerida por la API de Doubao. Este enfoque es muy directo: simplemente le dices al sistema a qué palabras prestar atención y él hace exactamente eso.

**Modo de tabla de palabras activas de plataforma** utiliza el `corpus.boosting_table_id` campo para hacer referencia a una tabla de palabras activas preconfigurada en la plataforma de autoaprendizaje Doubao. Este enfoque es adecuado para escenarios en los que es necesario gestionar una gran cantidad de palabras activas. Podemos crear y mantener tablas de palabras activas en la plataforma de autoaprendizaje Doubao y luego hacer referencia a ellas por ID. Para un proyecto como HagiCode, donde los términos especializados deben actualizarse y mantenerse continuamente, este modo ofrece una capacidad de gestión mucho mejor. Una vez que aumenta la cantidad de palabras clave, tener un lugar centralizado para administrarlas es mucho mejor que ingresarlas manualmente cada vez.

Curiosamente, estos dos modos también se pueden utilizar juntos. La API de Doubao admite la inclusión de palabras activas personalizadas y un ID de tabla de palabras activas de la plataforma en la misma solicitud, con la estrategia de combinación controlada por el `combine_mode` parámetro. Esta flexibilidad permite a HagiCode manejar una amplia gama de necesidades complejas de reconocimiento de terminología profesional. A veces, combinar múltiples enfoques produce mejores resultados.

### Definiciones y validación de tipos de frontend

En la implementación del frontend de HagiCode, definimos un conjunto completo de tipos de configuración de palabras clave y lógica de validación. La primera parte es la definición de tipo:

```typescript
export interface HotwordConfig {
  contextText: string;           // Multi-line hotword text
  boostingTableId: string;      // Doubao platform hotword table ID
  combineMode: boolean;          // Whether to use both together
}
```

Esta sencilla interfaz contiene todos los elementos de configuración para la función de palabras activas. Entre ellos, `contextText` es la parte con la que los usuarios interactúan más directamente: les permitimos ingresar una frase de palabras clave por línea, lo cual es muy intuitivo. Pedir a los usuarios que introduzcan un término por línea es mucho más fácil que hacerles comprender un formato de configuración complicado.

Luego viene la función de validación. Con base en las limitaciones de la API de Doubao, definimos reglas de validación estrictas: como máximo 100 líneas de texto de palabras activas, hasta 50 caracteres por línea y no más de 5000 caracteres en total; `boosting_table_id` puede tener como máximo 200 caracteres y solo puede contener letras, números, guiones bajos y guiones. Estos límites no son arbitrarios; provienen directamente de la documentación oficial de Doubao. Los límites de API son límites de API y tenemos que respetarlos.

```typescript
export function validateContextText(contextText: string): HotwordValidationResult {
  if (!contextText || contextText.trim().length === 0) {
    return { isValid: true, errors: [] };
  }

  const lines = contextText.split('\n').filter(line => line.trim().length > 0);
  const errors: string[] = [];

  if (lines.length > 100) {
    errors.push(`Hotword line count cannot exceed 100 lines; current count is ${lines.length}`);
  }

  const totalChars = contextText.length;
  if (totalChars > 5000) {
    errors.push(`Total hotword character count cannot exceed 5000; current count is ${totalChars}`);
  }

  for (let i = 0; i < lines.length; i++) {
    if (lines[i].length > 50) {
      errors.push(`Hotword on line ${i + 1} exceeds the 50-character limit`);
    }
  }

  return { isValid: errors.length === 0, errors };
}

export function validateBoostingTableId(boostingTableId: string): HotwordValidationResult {
  if (!boostingTableId || boostingTableId.trim().length === 0) {
    return { isValid: true, errors: [] };
  }

  const errors: string[] = [];

  if (boostingTableId.length > 200) {
    errors.push(`boosting_table_id cannot exceed 200 characters; current count is ${boostingTableId.length}`);
  }

  if (!/^[a-zA-Z0-9_-]+$/.test(boostingTableId)) {
    errors.push('boosting_table_id can contain only letters, numbers, underscores, and hyphens');
  }

  return { isValid: errors.length === 0, errors };
}
```

Estas funciones de validación se ejecutan inmediatamente cuando el usuario configura palabras activas, lo que garantiza que los problemas se detecten lo antes posible. Desde la perspectiva de la experiencia del usuario, este tipo de retroalimentación instantánea es muy importante. Siempre es mejor para los usuarios saber qué está mal mientras escriben que después de enviarlo.

### Persistencia de la configuración del front-end

En la implementación frontend de HagiCode, elegimos utilizar el navegador `localStorage` para almacenar la configuración de palabras clave. Hubo varias consideraciones detrás de esta decisión de diseño. En primer lugar, la configuración de las palabras activas es altamente personalizada y diferentes usuarios pueden tener diferentes necesidades específicas de dominio. En segundo lugar, este enfoque simplifica la implementación del backend porque no requiere tablas de base de datos adicionales ni puntos finales de API. Finalmente, después de que los usuarios lo configuran una vez en el navegador, la configuración se puede cargar automáticamente en usos posteriores, lo cual es muy conveniente. En pocas palabras, es el enfoque más sencillo.

```typescript
const HOTWORD_STORAGE_KEYS = {
  contextText: 'hotword-context-text',
  boostingTableId: 'hotword-boosting-table-id',
  combineMode: 'hotword-combine-mode',
} as const;

export const DEFAULT_HOTWORD_CONFIG: HotwordConfig = {
  contextText: '',
  boostingTableId: '',
  combineMode: false,
};

// Load hotword configuration
export function loadHotwordConfig(): HotwordConfig {
  const contextText = localStorage.getItem(HOTWORD_STORAGE_KEYS.contextText) || '';
  const boostingTableId = localStorage.getItem(HOTWORD_STORAGE_KEYS.boostingTableId) || '';
  const combineMode = localStorage.getItem(HOTWORD_STORAGE_KEYS.combineMode) === 'true';

  return { contextText, boostingTableId, combineMode };
}

// Save hotword configuration
export function saveHotwordConfig(config: HotwordConfig): void {
  localStorage.setItem(HOTWORD_STORAGE_KEYS.contextText, config.contextText);
  localStorage.setItem(HOTWORD_STORAGE_KEYS.boostingTableId, config.boostingTableId);
  localStorage.setItem(HOTWORD_STORAGE_KEYS.combineMode, String(config.combineMode));
}
```

La lógica de este código es sencilla y clara. leemos de `localStorage` al cargar la configuración y escribir en `localStorage` al guardarlo. También proporcionamos una configuración predeterminada para que el sistema pueda seguir funcionando correctamente cuando aún no existe ninguna configuración. Después de todo, tiene que haber un valor predeterminado sensato.

### Extensión de configuración del SDK de backend

En la implementación backend de HagiCode, necesitábamos agregar propiedades relacionadas con palabras activas a la clase de configuración del SDK. Teniendo en cuenta las características del lenguaje C# y los patrones de uso, utilizamos `List<string>` para almacenar contextos de palabras activas personalizados:

```csharp
public class DoubaoVoiceConfig
{
    /// <summary>
    /// App ID
    /// </summary>
    public string AppId { get; set; } = string.Empty;

    /// <summary>
    /// Access token
    /// </summary>
    public string AccessToken { get; set; } = string.Empty;

    /// <summary>
    /// Service URL
    /// </summary>
    public string ServiceUrl { get; set; } = string.Empty;

    /// <summary>
    /// Custom hotword context list
    /// </summary>
    public List<string>? HotwordContexts { get; set; }

    /// <summary>
    /// Doubao platform hotword table ID
    /// </summary>

```

El diseño de esta clase de configuración sigue el estilo conciso habitual de HagiCode. `HotwordContexts` es un tipo de lista que acepta valores NULL y `BoostingTableId` es una cadena que acepta valores NULL, por lo que cuando no hay una configuración de palabra activa, estas propiedades no tienen ningún efecto en la solicitud. Si no está utilizando la función, debe permanecer fuera del camino.

### Lógica de construcción de carga útil

La construcción de carga útil es el núcleo de toda la función de palabras activas. Una vez que tengamos la configuración de la palabra clave, debemos formatearla en la estructura JSON requerida por la API de Doubao. Este proceso ocurre antes de que el SDK envíe la solicitud:

```csharp
private void AddCorpusToRequest(Dictionary<string, object> request)
{
    var corpus = new Dictionary<string, object>();

    // Add custom hotwords
    if (Config.HotwordContexts != null && Config.HotwordContexts.Count > 0)
    {
        corpus["context"] = new Dictionary<string, object>
        {
            ["context_type"] = "dialog_ctx",
            ["context_data"] = Config.HotwordContexts
                .Select(text => new Dictionary<string, object> { ["text"] = text })
                .ToList()
        };
    }

    // Add platform hotword table ID
    if (!string.IsNullOrEmpty(Config.BoostingTableId))
    {
        corpus["boosting_table_id"] = Config.BoostingTableId;
    }

    // Add corpus to the request only when it is not empty
    if (corpus.Count > 0)
    {
        request["corpus"] = corpus;
    }
}
```

Este código muestra cómo construir dinámicamente el `corpus` campo basado en la configuración. El punto clave es que agregamos el `corpus` campo sólo cuando la configuración de palabras activas realmente existe. Este diseño garantiza la compatibilidad con versiones anteriores: cuando no se configuran palabras activas, la estructura de la solicitud sigue siendo exactamente la misma que antes. La compatibilidad con versiones anteriores es importante; Agregar una característica no debería alterar la lógica existente.

### Paso de parámetros de WebSocket

Entre el frontend y el backend, los parámetros de palabras activas se pasan a través de mensajes de control WebSocket. HagiCode está diseñado para que cuando el frontend comience a grabar, cargue la configuración de la palabra clave desde `localStorage` y lo envía al backend a través de un mensaje WebSocket.

```typescript
const controlMessage = {
  type: 'control',
  payload: {
    command: 'StartRecognition',
    contextText: '高血压\n糖尿病\n冠心病',
    boosting_table_id: 'medical_table',
    combineMode: false
  }
};
```

Hay un detalle a tener en cuenta aquí: la interfaz pasa texto de varias líneas separadas por caracteres de nueva línea y el backend necesita analizarlo. El controlador WebSocket de backend analiza estos parámetros y los pasa al SDK:

```csharp
private async Task HandleControlMessageAsync(
    string connectionId,
    DoubaoSession session,
    ControlMessage message)
{
    if (message.Payload is SessionControlRequest controlRequest)
    {
        // Parse hotword parameters
        string? contextText = controlRequest.ContextText;
        string? boostingTableId = controlRequest.BoostingTableId;
        bool? combineMode = controlRequest.CombineMode;

        // Parse multi-line text into a hotword list
        if (!string.IsNullOrEmpty(contextText))
        {
            var hotwords = contextText
                .Split('\n', StringSplitOptions.RemoveEmptyEntries)
                .Select(s => s.Trim())
                .Where(s => s.Length > 0)
                .ToList();

            session.HotwordContexts = hotwords;
        }

        session.BoostingTableId = boostingTableId;
    }
}
```

Con este diseño, pasar la configuración de palabras activas del frontend al backend se vuelve claro y eficiente. No hay nada especialmente misterioso en ello; los datos simplemente se pasan capa por capa.

## Guía práctica

### Configurar palabras activas personalizadas

En el uso real, configurar palabras activas personalizadas es muy sencillo. Abra la página de configuración de reconocimiento de voz en HagiCode y busque la sección "Configuración de palabras clave". En el cuadro de entrada "Texto de palabra clave personalizada", ingrese una frase de palabra clave por línea.

Por ejemplo, si está desarrollando una aplicación relacionada con la medicina, puede configurarla así:

```
高血压
糖尿病
冠心病
心绞痛
心肌梗死
心力衰竭
```

Después de guardar la configuración, estas palabras activas se pasan automáticamente a la API de Doubao cada vez que se inicia el reconocimiento de voz. En nuestras pruebas, una vez configuradas las palabras activas, la precisión del reconocimiento de términos profesionales relacionados mejoró notablemente. La mejora es real y claramente mejor que antes.

### Configurar una tabla de palabras clave de plataforma

Si necesita administrar una gran cantidad de palabras activas, o si las palabras activas necesitan actualizaciones frecuentes, el modo de tabla de palabras activas de la plataforma es una mejor opción. Primero, cree una tabla de palabras activas en la plataforma de autoaprendizaje Doubao y obtenga las generadas `boosting_table_id`, luego ingrese esta ID en la página de configuración de HagiCode.

La plataforma de autoaprendizaje Doubao proporciona capacidades como importación masiva y gestión categorizada de palabras activas, lo cual resulta muy práctico para equipos que necesitan gestionar grandes conjuntos de terminología especializada. Al administrar palabras clave en la plataforma, puede mantenerlas de manera centralizada e implementar actualizaciones de manera consistente. Una vez que la lista de palabras clave crece, tener un único lugar para administrarlas es mucho más práctico que ingresarlas manualmente cada vez.

### Usando el modo de combinación

En algunos escenarios complejos, es posible que necesite utilizar palabras activas personalizadas y una tabla de palabras activas de plataforma al mismo tiempo. En ese caso, simplemente configure ambos en HagiCode y habilite el interruptor "Modo de combinación".

En el modo combinado, la API de Doubao considera ambas fuentes de palabras activas al mismo tiempo, por lo que la precisión del reconocimiento suele ser mayor que si se utiliza cualquiera de las fuentes sola. Sin embargo, vale la pena señalar que el modo combinado aumenta la complejidad de la solicitud, por lo que es mejor decidir si habilitarlo después de pruebas prácticas. Una mayor complejidad sólo vale la pena si los resultados del mundo real lo justifican.

### Ejemplos de integración de código

Integrar la función hotword en el proyecto HagiCode es muy sencillo. A continuación se muestran algunos fragmentos de código de uso común:

```typescript
import {
  loadHotwordConfig,
  saveHotwordConfig,
  validateHotwordConfig,
  parseContextText,
  getEffectiveHotwordMode,
  type HotwordConfig
} from '@/types/hotword';

// Load and validate configuration
const config = loadHotwordConfig();
const validation = validateHotwordConfig(config);

if (!validation.isValid) {
  console.error('Hotword configuration validation failed:', validation.errors);
  return;
}

// Parse hotword text
const hotwords = parseContextText(config.contextText);
console.log('Parsed hotwords:', hotwords);

// Get effective hotword mode
const mode = getEffectiveHotwordMode(config);
console.log('Current hotword mode:', mode);
```

## Cosas a tener en cuenta

Hay varios puntos que merecen especial atención al implementar y utilizar la función de palabras activas.

Primero está el límite de caracteres. La API de Doubao tiene restricciones estrictas sobre las palabras activas, incluido el recuento de líneas, los caracteres por línea y el recuento total de caracteres. Si se excede algún límite, la API devuelve un error. En la implementación del frontend de HagiCode, verificamos estas restricciones durante la entrada del usuario a través de funciones de validación, lo que evita que se envíen configuraciones no válidas al backend. Detectar los problemas a tiempo siempre es mejor que esperar a que falle la API.

En segundo lugar está el formato de `boosting_table_id`. Este campo solo permite letras, números, guiones bajos y guiones, y no puede contener espacios ni otros caracteres especiales. Al crear una tabla de palabras activas en la plataforma de autoaprendizaje Doubao, asegúrese de seguir las reglas de nomenclatura. Ese tipo de validación de formato estricta es común para las API.

En tercer lugar está la compatibilidad con versiones anteriores. Los parámetros de palabras activas son completamente opcionales. Si no se configuran palabras activas, el sistema se comporta exactamente como antes. Este diseño garantiza que los usuarios existentes no se vean afectados de ninguna manera y también facilita la migración y las actualizaciones graduales. Agregar una función no debería alterar la lógica anterior.

Finalmente, está el manejo de errores. Cuando la configuración de palabras activas no es válida, la API de Doubao devuelve los mensajes de error correspondientes. La implementación de HagiCode registra registros detallados para ayudar a los desarrolladores a solucionar problemas. Al mismo tiempo, la interfaz muestra errores de validación en la interfaz de usuario para ayudar a los usuarios a corregir la configuración. Un buen manejo de errores conduce naturalmente a una mejor experiencia de usuario.
