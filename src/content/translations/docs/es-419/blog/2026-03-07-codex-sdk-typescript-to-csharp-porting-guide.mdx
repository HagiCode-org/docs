## De TypeScript a C#: práctica de portabilidad entre idiomas para el SDK del Codex

> En pocas palabras, este artículo también es un poco nuestro: registra el proceso completo de migración del SDK oficial de TypeScript Codex a C#. Llamarlo "puerto" casi hace que parezca demasiado fácil; fue más bien una larga aventura, porque estos dos idiomas tienen personalidades muy diferentes y teníamos que encontrar una manera de hacerlos cooperar.

## Antecedentes

Codex es la herramienta AI Agent CLI lanzada por OpenAI y es realmente poderosa. El equipo oficial proporciona un SDK de TypeScript en el `@openai/codex` paquete. Interactúa con la CLI del Codex llamando al `codex exec --experimental-json` comando y analizando un flujo de eventos JSONL.

El problema es que en el proyecto HagiCode, necesitamos usarlo en un entorno .NET puro, específicamente en servicios backend y aplicaciones de escritorio de C#. No podríamos introducir razonablemente un tiempo de ejecución de Node.js en un proyecto .NET solo para llamar a una herramienta CLI. Sería demasiado engorroso.

Así que nos quedaron dos opciones: mantener una capa puente Node.js compleja o crear nosotros mismos un SDK de C# nativo.

Elegimos este último.

## Acerca de HagiCode

Este artículo también proviene directamente de nuestra experiencia práctica en el [HagiCode](https://hagicode.com) proyecto. En términos sencillos, significa mantener varios componentes a la vez: una extensión VSCode en el frontend, servicios de inteligencia artificial en el backend y un cliente de escritorio multiplataforma. Esa complejidad multilingüe y multiplataforma es exactamente la razón por la que necesitábamos un SDK de C# nativo; realmente no queríamos ejecutar Node.js dentro de un proyecto .NET.

## Contenido principal

### Comparación de diseños arquitectónicos

Antes de traducir el código uno a uno, primero teníamos que comprender el diseño arquitectónico de ambos SDK. Tienes que entender ambas partes antes de poder portarlas bien.

La arquitectura central del SDK de TypeScript se ve así:

```
Codex (entry class)
  └── CodexExec (executor, manages child processes)
      └── Thread (conversation thread)
          ├── run() / runStreamed() (synchronous/asynchronous execution)
          └── event stream parsing
```

El SDK de C# mantiene las mismas capas arquitectónicas, pero adapta los detalles de implementación. La idea general es sencilla: preservar la coherencia de la API y al mismo tiempo aprovechar al máximo las características del lenguaje C# en la implementación.

### Tipo de conversión del sistema

Esta es la parte más fundamental y también la más importante del trabajo. Si la base es débil, todo lo que sigue se vuelve más difícil.

El sistema de tipos de TypeScript es más flexible que el de C#, y eso es simplemente un hecho. Necesitábamos encontrar una estrategia de mapeo adecuada:

| mecanografiado | c# | Notas |
|------------|-----|------|
| `interface` / `type` | `record` | usos de c# `record` para estructuras de datos inmutables |
| `cadena\| nulo` | `string?` | Tipo de referencia que acepta valores NULL |
| `booleano \| indefinido` | `bool?` | booleano anulable |
| `AsyncGenerator` | `IAsyncEnumerable` | iterador asíncrono |

El **sistema de tipo de evento** es un ejemplo típico. TypeScript usa tipos de unión para definir eventos:

```typescript
export type ThreadEvent =
  | ThreadStartedEvent
  | TurnStartedEvent
  | TurnCompletedEvent
  | ...
```

En C#, utilizamos una jerarquía de herencia y una coincidencia de patrones para lograr un efecto similar:

```csharp
public abstract record ThreadEvent(string Type);

public sealed record ThreadStartedEvent(string ThreadId) : ThreadEvent("thread.started");
public sealed record TurnStartedEvent() : ThreadEvent("turn.started");
public sealed record TurnCompletedEvent(Usage Usage) : ThreadEvent("turn.completed");
// ...
```

Nosotros elegimos `record` en lugar de `class` porque los objetos de eventos deben ser inmutables, lo que coincide con la intención detrás del uso de objetos simples en TypeScript. el `sealed` La palabra clave también evita la herencia adicional y le da al compilador espacio para optimizar.

### Puntos clave de portabilidad

#### 1. Analizador de eventos

El análisis de eventos es el núcleo de todo el SDK, porque determina si podemos comprender correctamente cada mensaje devuelto por la CLI del Codex. Si el análisis es incorrecto, todo lo posterior será un esfuerzo en vano.

La versión TypeScript utiliza `JSON.parse()` para analizar cada línea de JSON:

```typescript
export function parseEvent(line: string): ThreadEvent {
  const data = JSON.parse(line);
  // Handle different event types...
}
```

La versión C# utiliza `System.Text.Json.JsonDocument` en lugar de:

```csharp
public static ThreadEvent Parse(string line)
{
    using var document = JsonDocument.Parse(line);
    var root = document.RootElement;
    var type = GetRequiredString(root, "type", "event.type");

    return type switch
    {
        "thread.started" => new ThreadStartedEvent(GetRequiredString(root, "thread_id", ...)),
        "turn.started" => new TurnStartedEvent(),
        "turn.completed" => new TurnCompletedEvent(ParseUsage(...)),
        // ...
        _ => new UnknownThreadEvent(type, root.Clone()),
    };
}
```

Aquí hay un pequeño pero importante truco: `root.Clone()` es necesario, porque los elementos de `JsonDocument` dejarán de ser válidas una vez eliminado el documento. Necesitamos conservar una copia para tipos de eventos desconocidos. Esa es simplemente una de las diferencias entre el manejo de C# JSON y JavaScript.

#### 2. Diferencias en la gestión de procesos

Aquí es donde los dos SDK difieren más. Node.js y .NET tienen diferentes convenciones de ejecución, por lo que la implementación debe adaptarse.

TypeScript usa Node.js `spawn()` función:

```typescript
const child = spawn(this.executablePath, commandArgs, { env, signal });
```

C# utiliza .NET `System.Diagnostics.Process`:

```csharp
using var process = new Process { StartInfo = startInfo };
process.Start();

// stdin/stdout/stderr must be managed manually
```

Más específicamente, la versión C# necesita configurar el proceso de esta manera:

```csharp
var startInfo = new ProcessStartInfo
{
    FileName = _executablePath,
    RedirectStandardInput = true,
    RedirectStandardOutput = true,
    RedirectStandardError = true,
    UseShellExecute = false,
    CreateNoWindow = true,
};
```

La mayor diferencia es el mecanismo de cancelación. Usos de TypeScript `AbortSignal`, que es parte de la API web y es muy conveniente trabajar con él:

```typescript
const child = spawn(cmd, args, { signal: cancellationSignal });
```

usos de c# `CancellationToken` en lugar de:

```csharp
public async IAsyncEnumerable<string> RunAsync(
    CodexExecArgs args,
    [EnumeratorCancellation] CancellationToken cancellationToken = default)
{
    // Check cancellation status inside the loop
    while (!cancellationToken.IsCancellationRequested)
    {
        // Process output...
    }

    // Terminate the process when cancellation is requested
    if (cancellationToken.IsCancellationRequested)
    {
        try { process.Kill(entireProcessTree: true); } catch { }
    }
}
```

A alto nivel, este es sólo otro ejemplo de la diferencia entre el ecosistema Web API y el ecosistema .NET.

#### 3. Preservar la serialización de la configuración.

Ambos SDK implementan la lógica que convierte la configuración JSON en configuración TOML, porque la CLI del Codex acepta anulaciones de configuración en formato TOML. Esta parte debe permanecer completamente coherente; de ​​lo contrario, la misma configuración se comportará de manera diferente en los dos SDK.

Ése es el tipo de detalle en el que no se puede transigir. El éxito o el fracaso a menudo se reducen a detalles como este.

### Detalles de implementación

#### Estructura del proyecto

Creamos la siguiente estructura de proyecto:

```
CodexSdk/
├── CodexSdk.csproj
├── Codex.cs           # Entry class
├── CodexThread.cs     # Conversation thread
├── CodexExec.cs       # Executor
├── Events.cs          # Event type definitions
├── Items.cs           # Item type definitions
├── EventParser.cs     # Event parser
├── OutputSchemaTempFile.cs  # Temporary file management
└── ...
```

Es una estructura bastante limpia, y eso ayudó mucho durante el puerto.

#### Ejemplo de uso

El uso básico sigue siendo coherente con el SDK de TypeScript:

```csharp
using CodexSdk;

// Create a Codex instance
var codex = new Codex();
var thread = codex.StartThread();

// Execute a query
var result = await thread.RunAsync("Summarize this repository.");
Console.WriteLine(result.FinalResponse);
```

El manejo de eventos de transmisión aprovecha la coincidencia de patrones de C#:

```csharp
await foreach (var @event in thread.RunStreamedAsync("Analyze the code."))
{
    switch (@event)
    {
        case ItemCompletedEvent itemCompleted
            when itemCompleted.Item is AgentMessageItem msg:
            Console.WriteLine($"Assistant: {msg.Text}");
            break;
        case TurnCompletedEvent completed:
            Console.WriteLine($"Tokens: in={completed.Usage.InputTokens}");
            break;
        case CommandExecutionItem command:
            Console.WriteLine($"Command: {command.Command}");
            break;
    }
}
```

### Notas

Durante la implementación, recopilamos varias lecciones prácticas:

1. **Gestión de procesos**: la versión C# debe gestionar manualmente el ciclo de vida completo del proceso, incluida la terminación del proceso durante la cancelación. uso `Kill(entireProcessTree: true)` para asegurarse de que los procesos secundarios también se limpien.

2. **Manejo de errores**: Usamos `InvalidOperationException` para generar errores de análisis, manteniendo el estilo de manejo de errores similar al SDK de TypeScript.

3. **Limpieza de recursos**: `OutputSchemaTempFile` implementos `IAsyncDisposable` para garantizar que los archivos temporales se limpien correctamente.

4. **Variables de entorno**: la versión C# admite la anulación total de las variables de entorno del proceso a través de `CodexOptions.Env`. Es una característica pequeña, pero muy práctica.

5. **Diferencias de plataforma**: la versión C# no incluye la lógica de la versión TypeScript para ubicar automáticamente archivos binarios dentro de paquetes npm. Dado que los proyectos .NET normalmente no dependen de npm, la ruta al `codex` El ejecutable debe especificarse mediante el `CODEX_EXECUTABLE` variable de entorno o `CodexPathOverride`.

## Conclusión

Portar un SDK de TypeScript maduro a C# no es solo una cuestión de conversión de sintaxis, sino que también requiere comprender las filosofías de diseño de ambos lenguajes. La flexibilidad de TypeScript y las características del ecosistema JavaScript, como `AbortSignal` Necesita contrapartes apropiadas en C#.

La conclusión clave es la siguiente: **mantener la coherencia de la API es más importante que mantener la coherencia a nivel de implementación**. A los usuarios les importa si la interfaz es fácil de usar, no si la implementación interna es idéntica. Suena simple, pero hacer esas concesiones requiere criterio.

Si está trabajando en un puerto similar en varios idiomas, nuestra experiencia es comprender completamente la arquitectura del SDK original primero, luego traducirla módulo por módulo y finalmente utilizar un conjunto de pruebas completo para garantizar la coherencia del comportamiento. Este tipo de trabajo no se puede apresurar.

Todo saldrá bien al final.
