## VSCode y code-server: elección de una solución de edición de código basada en navegador

> Al crear capacidades de edición de código basadas en navegador, los desarrolladores se enfrentan a una elección clave: utilizar la versión oficial de VSCode. `code serve-web` característica, o adoptar la comunidad impulsada `code-server` solución? Esta decisión afecta no sólo a la arquitectura técnica, sino también al cumplimiento de la licencia y la flexibilidad de implementación.

## Antecedentes

La selección técnica se parece mucho a elegir un camino en la vida. Una vez que eliges uno, generalmente tienes que seguir recorriéndolo y cambiar más tarde puede resultar muy costoso.

En la era de la programación asistida por IA, la edición de código basada en navegador es cada vez más importante. Los usuarios esperan que después de que un asistente de IA termine de analizar el código, puedan abrir inmediatamente un editor en la misma sesión del navegador y realizar cambios sin cambiar de aplicación. Ese tipo de experiencia perfecta debería estar ahí cuando la necesites.

Sin embargo, al implementar esta característica, los desarrolladores enfrentan una elección técnica crítica: ¿deben usar la versión oficial de VSCode? `code serve-web` característica, o la comunidad impulsada `code-server` solución?

Cada opción tiene sus propias ventajas y desventajas, y elegir mal puede crear muchos problemas más adelante. La concesión de licencias es un ejemplo: si sólo después del lanzamiento descubre que su producto no cumple con las licencias, ya es demasiado tarde. La implementación es otra: una solución puede funcionar perfectamente en desarrollo y luego encontrarse con todo tipo de problemas una vez trasladada a contenedores. Estos son exactamente los tipos de trampas que los equipos quieren evitar.

## Diferencias de licencia (más importantes)

Esta es la diferencia más fundamental entre las dos soluciones y el primer factor que consideramos durante la evaluación. Al tomar una decisión técnica, es importante comprender los riesgos legales desde el principio.

**servidor de códigos**
- Licencia MIT, código totalmente abierto
- Mantenido por Coder.com con una comunidad activa.
- Gratis para usar comercialmente, modificar y distribuir
- Sin restricciones en los escenarios de uso

**Código VS `code serve-web`**
- Parte del producto Microsoft VSCode
- Utiliza la licencia de Microsoft (la licencia VS Code incluye restricciones de uso comercial)
- Destinado principalmente a desarrolladores individuales
- La implementación empresarial puede requerir una revisión adicional de las licencias comerciales

Desde una perspectiva de licencia, el servidor de códigos es más amigable para proyectos comerciales. Esto es algo que debe considerar durante la planificación del producto, porque migrar más tarde puede resultar muy costoso.

## Diferencias de implementación

Una vez que se resuelva la concesión de licencias, el siguiente tema es la implementación. Eso afecta directamente el costo de sus operaciones y el diseño arquitectónico.

**servidor de códigos**
- Una aplicación Node.js independiente que se puede implementar de forma independiente
- Admite múltiples fuentes de tiempo de ejecución:
  - Especificando directamente la ruta ejecutable
  - Buscándolo a través de la RUTA del sistema
  - Detección automática de un entorno NVM Node.js 22.x
- No es necesario instalar la aplicación de escritorio VSCode en el servidor
- Más fácil de implementar en contenedores

**Código VS `code serve-web`**
- Debe depender de una CLI VSCode instalada localmente
- Requiere un disponible `code` comando en la máquina host
- El sistema filtra los contenedores de CLI remota de VS Code
- Diseñado principalmente para escenarios de desarrollo local.

code-server es más adecuado para escenarios de implementación de servidores y contenedores. Si su producto necesita ejecutarse en Docker o sus usuarios no tienen VSCode instalado, el servidor de códigos suele ser la opción correcta.

## Diferencias de parámetros de características

Las dos soluciones también difieren en algunos parámetros de características. Las diferencias no son enormes, pero pueden crear fricciones de integración en el uso en el mundo real.

| Característica | servidor de códigos | código de servicio web |
|------|-------------|----------------|
| Camino de base pública | `/` (configurable) | `/vscode-server` (fijo) |
| Autenticación | `--auth` parámetro con múltiples modos | `--connection-token` / `--without-connection-token` |
| Directorio de datos | `{DataDir}/code-server` | `{DataDir}/vscode-serve-web` |
| Telemetria | Deshabilitado por defecto con `--disable-telemetry` | Depende de la configuración de VSCode |
| Comprobaciones de actualización | Se puede desactivar con `--disable-update-check` | Depende de la configuración de VSCode |

Estas diferencias necesitan atención especial durante la integración. Por ejemplo, diferentes rutas URL significan que su código de interfaz necesita un manejo dedicado.

## Diferencias de detección de disponibilidad

Al implementar el cambio de editor, la lógica de detección de disponibilidad también difiere.

**servidor de códigos**
- Siempre devuelto como una implementación visible
- Todavía se muestra incluso cuando no está disponible, con un `install-required` estado
- Admite la detección automática de un entorno NVM Node.js 22.x

**código de servicio-web**
- Sólo visible cuando un local `code` Se detecta CLI
- Si no está disponible, la interfaz oculta automáticamente esta opción.
- Depende del estado de instalación local de VSCode

Esta diferencia afecta directamente a la experiencia del usuario. Code-server es más transparente: los usuarios pueden ver la opción y comprender que aún es necesaria la instalación. El código de servicio web está más oculto: es posible que los usuarios ni siquiera se den cuenta de que existe la opción. Qué enfoque es mejor depende del posicionamiento del producto.

## Arquitectura de implementación dual de HagiCode

Después de un análisis en profundidad, el proyecto HagiCode adoptó una arquitectura de implementación dual que admite ambas soluciones a nivel arquitectónico.

### Por defecto al servidor de códigos

```csharp
// The default active implementation is code-server
// If an explicit activeImplementation is saved, try that implementation first
// If the requested implementation is unavailable, the resolver tries the other one
// If a fallback occurs, return fallbackReason
```

Por defecto utilizamos el servidor de códigos principalmente debido a la flexibilidad de licencias e implementación. Sin embargo, para los usuarios que ya tienen un entorno VSCode local, Codeserve-web también es una opción sólida.

### Selector de implementación

`CodeServerImplementationResolver` se encarga de unificar:
- Selección de implementación durante el calentamiento de inicio
- Selección de implementación al leer el estado.
- Selección de implementación al abrir proyectos.
- Selección de implementación al abrir Vaults

Este diseño permite que el sistema responda de manera flexible a diferentes escenarios y los usuarios puedan elegir la implementación que mejor se adapte a su entorno.

### Reglas de adaptación del frontend

```typescript
// When localCodeAvailable=false, do not show code serve-web
// When localCodeAvailable=true, show the code serve-web configuration
```

La interfaz muestra automáticamente las opciones disponibles según el entorno, para que los usuarios no se sientan confundidos por funciones que no pueden utilizar.

## Guía práctica

Después de toda esa teoría, ¿a qué debería prestar atención durante el despliegue real? Al final, lo que importa es la implementación.

### Recomendación de implementación de Docker

Para la implementación en contenedores, el servidor de códigos es la mejor opción:

```dockerfile
# Use the official code-server image directly
FROM codercom/code-server:latest

# Or install through npm
RUN npm install -g code-server
```

Esto resuelve el problema en una sola capa sin requerir una instalación adicional de VSCode.

### Ejemplos de configuración

**configuración del servidor de códigos**
```json
{
  "vscodeServer": {
    "enabled": true,
    "activeImplementation": "code-server",
    "codeServer": {
      "host": "0.0.0.0",
      "port": 8080,
      "executablePath": "",
      "authMode": "none"
    }
  }
}
```

**configuración web de servicio de código**
```json
{
  "vscodeServer": {
    "enabled": true,
    "activeImplementation": "serve-web",
    "serveWeb": {
      "host": "0.0.0.0",
      "port": 8080,
      "executablePath": "/usr/local/bin/code"
    }
  }
}
```

La configuración puede resultar un poco tediosa la primera vez, pero una vez realizada, todo resulta mucho más fácil de mantener.

### Diferencias en la construcción de URL

**servidor de códigos**
```
http://localhost:8080/?folder=/path/to/project&vscode-lang=zh-CN
```

**código de servicio-web**
```
http://localhost:8080/vscode-server/?folder=/path/to/project&tkn=xxx&vscode-lang=zh-CN
```

Preste atención a las diferencias en rutas y parámetros. Debe manejarlos por separado durante la integración.

### Implementaciones de conmutación

El sistema admite el cambio de tiempo de ejecución y detiene automáticamente la implementación anterior al cambiar:

```csharp
// VsCodeServerManager automatically handles mutual exclusion
// When switching activeImplementation, the old implementation will not keep running in the background
```

Este diseño permite a los usuarios probar diferentes implementaciones en cualquier momento y encontrar la opción que mejor les funcione.

### Monitoreo de estado

```typescript
const { settings, runtime } = await getVsCodeServerSettings();

// runtime.activeImplementation: "code-server" | "serve-web"
// runtime.fallbackReason: reason for switching
// runtime.status: "running" | "starting" | "stopped" | "unhealthy"
```

Cuando el estado es visible, los usuarios pueden determinar rápidamente si un problema proviene del lado del servidor o de su propia operación.

## Conclusión

| Dimensión de comparación | servidor de códigos | código de servicio web | Recomendación |
|---------|-------------|----------------|------|
| Licencia | MIT (apto para uso comercial) | Microsoft (restringido) | servidor de códigos |
| Flexibilidad de implementación | Despliegue independiente | Depende del VSCode local | servidor de códigos |
| Idoneidad del servidor | Diseñado para servidores | Principalmente para el desarrollo local. | servidor de códigos |
| Contenedorización | Soporte nativo | Requiere instalación de VSCode | servidor de códigos |
| Completitud de funciones | Cerca de la edición de escritorio | Versión oficial completa | código de servicio web |
| Actividad de mantenimiento | Comunidad activa | Mantenido oficialmente por Microsoft. | Ambos tienen fortalezas |

**Estrategia recomendada**: utilice primero el servidor de códigos y considere el servicio web de códigos cuando necesite una funcionalidad oficial completa y ya tenga un entorno VSCode local.
