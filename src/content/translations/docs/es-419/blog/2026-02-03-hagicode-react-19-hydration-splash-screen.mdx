## Diseño de 12 experiencias de inicio excepcionales para HagiCode: del minimalismo al cyberpunk

> El breve intervalo entre descargar una aplicación React 19 y completar Hydration es una oportunidad de oro para que los usuarios sientan la personalidad de su marca. En este artículo, compartimos un sistema de estilo de inicio completo que creamos para el proyecto HagiCode usando HTML/CSS/JS.

## Antecedentes

Como aplicación moderna creada con ASP.NET Core 10 y React 19 (Vite), HagiCode utiliza una arquitectura de implementación separada frontend-backend. El resultado de la compilación del frontend está empaquetado en el backend. `wwwroot/` directorio y alojado en ASP.NET Core.

Sin embargo, esta arquitectura introduce un problema clásico de UX: cuando los usuarios visitan la página, el navegador primero carga el HTML, luego descarga el paquete JS grande y finalmente permite que React realice la hidratación. Durante este "período de vacío" que dura desde unos cientos de milisegundos hasta varios segundos, los usuarios ven una pantalla en blanco o una página estática sin vida.

Para llenar ese vacío e inyectar la personalidad de marca de HagiCode, necesitábamos diseñar un sistema de estilo de inicio implementado completamente con código en línea en su interior. `index.html`.

## Acerca de HagiCode

El enfoque de diseño de la pantalla de presentación compartido en este artículo proviene de nuestra experiencia práctica en el [HagiCode](https://github.com/HagiCode-org/site) proyecto. Como asistente de codificación de IA, HagiCode no solo se preocupa por la eficiencia de la generación de código, sino también por la experiencia visual del desarrollador. Este sistema de inicio es uno de los resultados de nuestra búsqueda del máximo rendimiento frontend.

## Desafíos centrales y diseño de arquitectura

Antes de empezar a diseñar, primero tuvimos que aclarar las limitaciones técnicas. Dado que todo tuvo que implementarse en línea en `index.html`, no pudimos cargar ningún archivo CSS o JS externo que no sea el propio paquete de React.

### Análisis de restricciones técnicas

1.  **Principio de dependencia cero**: Todos los estilos deben vivir dentro de un `<style>` etiqueta, y toda la lógica debe vivir dentro de una `<script>` etiqueta.
2.  **CSS defensivo**: para evitar que los estilos globales contaminen la pantalla de inicio después de que se monte la aplicación React, decidimos envolver todos los estilos de inicio con un prefijo de ID de alta prioridad, como `#boot-screen`.
3.  **El rendimiento es lo primero**: las animaciones deben usar CSS `transform` y `opacity` siempre que sea posible para evitar el reflujo y garantizar que el hilo principal permanezca desbloqueado.
4.  **Consistencia visual**: los colores y las fuentes deben permanecer alineados con la configuración Tailwind de HagiCode.

### Patrón de arquitectura: carcasa e inyector

Adoptamos un **patrón variante**. La lógica central se encapsula dentro de una expresión de función invocada inmediatamente (IIFE), mientras que la lógica de representación específica se inyecta a través de la configuración. Esto nos permite cambiar entre diferentes estilos mediante una configuración simple en lugar de reescribir la lógica de manipulación DOM repetidamente.

Aquí está el código de arquitectura principal:

```html
<!-- Inline in index.html -->
<div id="boot-root"></div>

<script>
(function() {
  const BootSequence = {
    config: {
      theme: 'terminal', // Configurable as 'minimal', 'skeleton', 'code-rain', etc.
      color: '#3b82f6'   // Brand color
    },
    
    // Core lifecycle
    init() {
      this.render();
      this.listenForMount();
    },

    // Render the currently selected style
    render() {
      const root = document.getElementById('boot-root');
      if (this.variants[this.config.theme]) {
        root.innerHTML = this.variants[this.config.theme].render();
      }
    },

    // Listen for successful React mount and exit gracefully
    listenForMount() {
      window.addEventListener('hagicode:ready', () => {
        const screen = document.getElementById('boot-root');
        // Fade out first, then remove the DOM to avoid flicker
        screen.style.opacity = '0';
        screen.style.transition = 'opacity 0.3s ease';
        setTimeout(() => screen.remove(), 300);
      });
    },

    // The implementation logic for all 12 styles lives here
    variants: {
      // ...see details below
    }
  };

  BootSequence.init();
})();
</script>
```

## Una lista de verificación de 12 diseños de estilo de inicio

Agrupamos estos 12 estilos en seis categorías principales para satisfacer diferentes escenarios y preferencias estéticas.

### A. Minimalismo

> "Menos es más." Para escenarios que buscan la máxima velocidad de carga, ofrecemos las opciones más ligeras posibles.

#### 1. Punto minimalista
Un simple punto se encuentra en el centro de la pantalla, junto con una animación de respiración.
*   **Implementación**: CSS `@keyframes` controla la escala y la opacidad.
*   **Mejor para**: Cualquier caso en el que la página deba permanecer absolutamente limpia.

#### 2. Revelación de marca
Usando SVG `stroke-dasharray` animación, simula una revelación dibujada a mano de las líneas del logotipo de HagiCode, seguida de un texto que aparece gradualmente.
*   **Técnica**: Animación de ruta SVG con una sensación muy pulida.

### B. Mimetismo de pantalla de esqueleto

> "El arte de engañar a la vista". Al simular un diseño de interfaz de usuario real, los usuarios sienten que la página ya está medio cargada.

#### 3. Esqueleto de chat de la barra lateral
Esta puede ser la opción más práctica. Creamos manualmente un diseño en HTML que refleja React `Sidebar` y `ChatInput` componentes exactamente, luego lo superpuso con una animación de brillo gris.
*   **Valor**: Cuando se completa la hidratación de React, el esqueleto se convierte instantáneamente en el componente real y los usuarios apenas pueden percibir el cambio.

#### 4. Esqueleto de pila de cartas
Simula el movimiento apilado de las tarjetas de propuesta mientras se cargan, utilizando transformaciones 3D para hacer que las tarjetas floten sutilmente.

### C. Abstracción y Arte

> Muestra el ADN geek de HagiCode.

#### 5. Forma geométrica
Se representa una forma geométrica (un cuadrado) en el centro de la pantalla, luego se transforma suavemente con el tiempo en un círculo, un triángulo y, finalmente, el logotipo.
*   **Tecnología**: transiciones suaves con CSS `border-radius`.

#### 6. Lluvia de código
Un homenaje a *The Matrix*. Usando la fuente JetBrains Mono, débiles secuencias de caracteres caen en el fondo.
*   **Nota**: Para mejorar el rendimiento, las transmisiones de personajes deben permanecer dentro de un área más pequeña o utilizar una frecuencia de actualización más baja.

#### 7. Pulso de neón
Un anillo brillante de estilo cyberpunk que utiliza múltiples `box-shadow` capas para crear un poderoso brillo de neón.

### D. Marca y temas

> Haga que el sistema se sienta vivo.

#### 8. Tema estacional
Este es un cargador dinámico. Comprueba la fecha actual de días festivos como el Año Nuevo Lunar o Navidad y carga la animación SVG correspondiente.
*   **Ejemplo**: Durante el Año Nuevo Lunar, las linternas rojas se balancean suavemente en la parte inferior de la pantalla.

#### 9. Flujo de gradiente
El fondo utiliza un degradado fluido basado en los colores de la marca HagiCode. Combinado con animado `background-size` y `background-position`, crea una sensación de movimiento similar a una aurora.

### E. Sensación técnica

> Un saludo a los desarrolladores.

#### 10. Arranque de terminales
Simula la salida de la consola. Las líneas de código pasan rápidamente:
```text
> Initializing HagiCode Core...
> Loading models...
> Connecting to neural network...
```
Esto instantáneamente resulta familiar para todos los desarrolladores.

#### 11. Barra de progreso
Aparece una delgada barra de progreso en la parte superior de la pantalla, con un porcentaje mostrado a la derecha. Si bien no podemos acceder al progreso real de la descarga, podemos usar un temporizador para simular un proceso de carga "creíble": rápido durante el primer 80%, luego gradualmente más lento durante el último 20%.

### F. Creatividad

#### 12. Montaje de píxeles
Esta es una idea muy interesante. Pequeños cuadrados se encuentran dispersos por la pantalla, luego convergen hacia el centro y se ensamblan gradualmente en el ícono del logotipo de HagiCode. Simboliza el proceso de construcción del código.

## Mejores prácticas y errores

En el trabajo de desarrollo real de HagiCode, resumimos varios detalles de implementación de importancia crítica.

### 1. El CSS defensivo es obligatorio
Nunca seas perezoso y te saltes el prefijo. Una vez, nos olvidamos de definir el alcance de los estilos de la pantalla de presentación con un ID y global. `div` Los estilos después de que React se montó afectaron inesperadamente la pantalla de presentación, rompiendo el diseño.
**Lección aprendida**: coloque cada selector de CSS debajo `#boot-screen`y utilizar `!important` para aumentar la prioridad cuando sea necesario, pero solo dentro del CSS de la pantalla de presentación.

### 2. Transiciones elegantes
Después de que React se monte exitosamente, no lo hagas directamente. `remove()` la pantalla de presentación DOM.
**Enfoque correcto**:
1.  Reaccionar desencadenantes `window.dispatchEvent(new Event('hagicode:ready'))`.
2.  La pantalla de presentación escucha el evento y las primeras series. `opacity: 0`.
3.  Espere 300 ms, que coincide con la duración de la transición CSS, y llame `.remove()` sólo después de que la pantalla sea completamente invisible.

### 3. Sincronización de variables temáticas
Los valores de color de la pantalla de presentación están codificados en `index.html`. Si cambiamos el color primario de Tailwind, también debemos actualizar la pantalla de presentación.
**Optimización**: escriba un complemento simple en el script de compilación de Vite para leer `tailwind.config.js` e inyectar variables de color en el `index.html` variables de plantilla, creando una única fuente de verdad.

### 4. Precarga de fuentes
Las pantallas de presentación a menudo necesitan usar una fuente de marca, pero si la fuente se carga lentamente, puede aparecer FOUT (Flash of Unstyled Text).
**Solución**: Agregar `<link rel="preload" href="/fonts/JetBrainsMono.woff2" as="font" type="font/woff2" crossorigin>` dentro `<head>`. Esta es una forma de mejorar la experiencia de bajo costo y alto rendimiento.

### 5. Monitoreo del desempeño
nosotros inyectamos `performance.mark('boot-start')` en la parte inferior de `index.html`, y marcado `boot-end` cuando React se montó correctamente.
**Por qué es importante**: al recopilar estos datos a través de Application Insights, podemos medir directamente en qué medida la pantalla de presentación acorta el tiempo de espera percibido. Los datos muestran que una excelente pantalla esquelética puede mejorar la tolerancia de los usuarios a una "red lenta" en más de un 50%.

## Resumen

Una buena pantalla de presentación es más que "decoración mientras se espera". Es la señal de apretón de manos en la primera interacción entre el producto y el usuario. En el proyecto HagiCode, este sistema de inicio basado en el **patrón Variantes** nos permite cambiar estilos de manera flexible entre días festivos y lanzamientos, lo que mejora en gran medida la sensación de diversión y profesionalismo del producto.

La solución compartida en este artículo se basa completamente en estándares web nativos sin introducir grandes dependencias, lo que refleja la búsqueda de HagiCode de ser "ligero pero potente". Si este enfoque le resulta valioso, no dude en consultar el código fuente en el repositorio de HagiCode e incluso contribuir con sus propios diseños creativos.
