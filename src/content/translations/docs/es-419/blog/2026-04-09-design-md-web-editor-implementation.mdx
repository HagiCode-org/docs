## Edite DESIGN.md directamente en la interfaz web: de la idea a la implementación

> En el sistema de gestión de proyectos MonoSpecs, `DESIGN.md` Lleva el diseño arquitectónico y las decisiones técnicas de un proyecto. Pero el flujo de trabajo de edición tradicional obliga a los usuarios a recurrir a un editor externo. Esa experiencia fragmentada es como ser interrumpido en medio de la lectura de un poema: la inspiración se ha ido, y también el estado de ánimo. Este artículo comparte la solución que ponemos en práctica en el proyecto HagiCode: edición `DESIGN.md` directamente en la interfaz web, con soporte para importar plantillas desde un sitio de diseño en línea. Después de todo, ¿quién no disfruta de la sensación de completar todo de una sola vez?

## Antecedentes

Como portador principal de los documentos de diseño de proyectos, `DESIGN.md` contiene información clave como diseño de arquitectura, decisiones técnicas y orientación de implementación. Sin embargo, el enfoque de edición tradicional requiere que los usuarios cambien a un editor externo como VS Code, ubiquen manualmente la ruta física y luego editen el archivo. No es especialmente complicado, pero tras repetir el proceso unas cuantas veces resulta agotador.

Los problemas se manifiestan principalmente de las siguientes maneras:

- **Flujo de trabajo fragmentado**: los usuarios deben cambiar constantemente entre la interfaz de administración web y un editor local, rompiendo la continuidad de su flujo de trabajo, muy parecido a tener la música cortada en medio de una canción.
- **Difícil de reutilizar**: el sitio de diseño ya publica una rica biblioteca de plantillas de diseño, pero no se pueden integrar directamente en el flujo de trabajo de edición del proyecto. Lo bueno existe, pero aún así no puedes usarlo donde lo necesitas.
- **Falta un bucle de experiencia**: no hay un bucle cerrado para "vista previa-seleccionar-importar", por lo que los usuarios deben copiar y pegar manualmente, lo que aumenta el riesgo de errores.
- **Fricción de colaboración**: mantener sincronizados los documentos de diseño y la implementación del código se convierte en un proceso de alta fricción, lo que perjudica la eficiencia del equipo.

Para resolver estos puntos débiles, decidimos agregar soporte de edición directa para `DESIGN.md` en la interfaz web y permitir la importación de plantillas con un solo clic desde un sitio de diseño en línea. No fue una decisión trascendental. Simplemente queríamos que la experiencia de desarrollo fuera más fluida.

## Acerca de HagiCode

La solución compartida en este artículo proviene de nuestra experiencia práctica en el [HagiCode](https://hagicode.com) proyecto. HagiCode es un proyecto de asistente de codificación impulsado por IA y, durante el desarrollo, con frecuencia necesitamos mantener los documentos de diseño del proyecto. Para ayudar al equipo a colaborar de manera más eficiente, exploramos e implementamos esta solución de edición e importación en línea. No hay nada misterioso en ello. Nos encontramos con un problema y encontramos una manera de resolverlo.

## Solución técnica

### Arquitectura general

Esta solución utiliza una arquitectura separada frontend-backend con un proxy del mismo origen, compuesto principalmente por las siguientes capas. En la práctica, el diseño se puede resumir como "cada parte hace su propio trabajo":

**1. Capa del editor frontend**
```typescript
// Core component: DesignMdManagementDrawer
// Location: repos/web/src/components/project/DesignMdManagementDrawer.tsx
// Responsibility: handle editing, saving, version conflict detection, and import flow
```

**2. Capa de servicio backend**
```csharp
// Core service: ProjectAppService.DesignMd
// Location: repos/hagicode-core/src/PCode.Application/ProjectAppService.DesignMd.cs
// Responsibility: path resolution, file read/write, and version management
```

**3. Capa de proxy del mismo origen**
```csharp
// Proxy service: ProjectAppService.DesignMdSiteIndex
// Location: repos/hagicode-core/src/PCode.Application/ProjectAppService.DesignMdSiteIndex.cs
// Responsibility: proxy design site resources, preview image caching, and security validation
```

### Decisiones técnicas clave

#### Decisión 1: Patrón de cajón global

Usamos un único cajón global en lugar de capas emergentes locales, con el estado administrado a través de `layoutSlice`, que brinda a los usuarios una experiencia consistente en todas las vistas (`classic` y `kanban`). No importa desde qué vista el usuario abra el editor, obtiene el mismo modelo de interacción. Una experiencia consistente hace que las personas se sientan más cómodas en lugar de desorientarse cuando cambian de vista.

#### Decisión 2: API con alcance de proyecto

montamos `DESIGN.md`-Puntos finales relacionados en `ProjectController`, reutilizando el límite de permisos del proyecto existente y evitando la complejidad de agregar un controlador separado. Esto hace que el manejo de permisos sea más claro y también se alinea con la organización de recursos RESTful. A veces, reutilizar es más significativo que crear algo nuevo desde cero.

#### Decisión 3: Detección de conflictos de versiones

Derivamos una versión opaca del sistema de archivos. `LastWriteTimeUtc`, lo que nos brinda un control de concurrencia ligero y optimista. Cuando varios usuarios editan el mismo archivo a la vez, el sistema puede detectar conflictos y solicitar al usuario que actualice. Este diseño no bloquea la edición y al mismo tiempo protege la coherencia de los datos.

#### Decisión 4: Patrón de proxy del mismo origen

Usamos `IHttpClientFactory` para representar recursos externos del sitio de diseño, evitando tanto problemas de origen cruzado como riesgos de la SSRF. Esto mantiene el sistema seguro y al mismo tiempo simplifica las llamadas frontales. Difícilmente se puede ser demasiado cuidadoso con la seguridad.

## Implementación central

### 1. Edite DESIGN.md directamente

#### Implementación de back-end

El backend es el principal responsable de la resolución de rutas, la lectura/escritura de archivos y la gestión de versiones. Estas tareas son básicas, pero indispensables, como los cimientos de una casa:

```csharp
// Path resolution and security validation
private Task<string> ResolveDesignDocumentDirectoryAsync(string projectPath, string? repositoryPath)
{
    if (string.IsNullOrWhiteSpace(repositoryPath))
    {
        return Task.FromResult(Path.GetFullPath(projectPath));
    }
    return ValidateSubPathAsync(projectPath, repositoryPath);
}

// Version generation (based on file system timestamp and size)
private static string BuildDesignDocumentVersion(string path)
{
    var fileInfo = new FileInfo(path);
    fileInfo.Refresh();
    return string.Create(
        CultureInfo.InvariantCulture,
        $"{fileInfo.LastWriteTimeUtc.Ticks:x}-{fileInfo.Length:x}");
}
```

El diseño de la versión es interesante por su simplicidad: utilizamos la hora y el tamaño de la última modificación del archivo para generar un identificador de versión único. Es liviano y confiable, sin necesidad de mantener una base de datos de versión adicional. Las soluciones simples suelen ser las más efectivas.

#### Implementación de interfaz

En la interfaz, implementamos la detección de estado sucio y la lógica de guardado. Este diseño ayuda a los usuarios a comprender si sus cambios se han guardado y reduce la ansiedad de "¿y si lo pierdo?":

```typescript
// Dirty-state detection and save logic
const [draft, setDraft] = useState('');
const [savedDraft, setSavedDraft] = useState('');
const isDirty = draft !== savedDraft;

const handleSave = useCallback(async () => {
    const result = await saveProjectDesignMdDocument({
        ...activeTarget,
        content: draft,
        expectedVersion: document.version, // optimistic concurrency control
    });
    setSavedDraft(draft); // update saved state
}, [activeTarget, document, draft]);
```

En esta implementación, mantenemos dos partes de estado: `draft` es el contenido que se está editando actualmente, mientras `savedDraft` es el contenido guardado. Compararlos nos dice si hay cambios no guardados. El diseño es simple, pero brinda tranquilidad a la gente. Nadie quiere que desaparezca aquello en lo que trabajaron duro.

#### Formato de datos de índice

El archivo de índice en el sitio de diseño define todas las plantillas disponibles. Con este índice, los usuarios pueden elegir la plantilla que quieran tan fácilmente como pedir desde un menú:

```json
{

    {
      "slug": "linear.app",
      "title": "Linear Inspired Design System",
      "summary": "AI Product / Dark Feel",
      "detailUrl": "/designs/linear.app/",
      "designDownloadUrl": "/designs/linear.app/DESIGN.md",
      "previewLightImageUrl": "...",
      "previewDarkImageUrl": "..."
    }

```

Cada entrada incluye la información básica de la plantilla y enlaces de descarga. El backend lee la lista de plantillas disponibles en este índice y las presenta para que el usuario elija. Eso hace que la selección sea intuitiva en lugar de obligar a las personas a buscar el camino en la oscuridad.

#### Implementación de proxy del mismo origen

Para mantener la seguridad, el backend realiza una validación estricta al acceder al sitio de diseño. No se puede ser demasiado cauteloso con la seguridad:

```csharp
// Safe slug validation
private static readonly Regex SafeDesignSiteSlugRegex =
    new("^[A-Za-z0-9](?:[A-Za-z0-9._-]{0,127})$", RegexOptions.Compiled);

private static string NormalizeDesignSiteSlug(string slug)
{
    var normalizedSlug = slug?.Trim() ?? string.Empty;
    if (!IsSafeDesignSiteSlug(normalizedSlug))
    {
        throw new BusinessException(
            ProjectDesignSiteIndexErrorCodes.InvalidSlug,
            "Design site slug must be a single safe path segment.");
    }
    return normalizedSlug;
}

// Preview image caching (OS temp directory)
private static string ComputePreviewCacheKey(string slug, string theme, string previewUrl)
{
    var raw = $"{slug}|{theme}|{previewUrl}";
    var bytes = SHA256.HashData(Encoding.UTF8.GetBytes(raw));
    return Convert.ToHexString(bytes).ToLowerInvariant();
}
```

Aquí hacemos dos cosas: primero, validamos el formato slug estrictamente con una expresión regular para evitar ataques de recorrido de ruta; en segundo lugar, almacenamos en caché las imágenes de vista previa para reducir la presión en el sitio externo. Lo primero es protección, lo segundo es optimización y ambos importan.

### 3. Flujo de importación completo

```typescript
// 1. Open the import drawer
const handleRequestImportDrawer = useCallback(() => {
    setIsImportDrawerOpen(true);
}, []);

// 2. Select and import
const handleImportRequest = useCallback((entry) => {
    if (isDirty) {
        setPendingImportEntry(entry);
        setConfirmMode('import'); // overwrite confirmation
        return;
    }
    void executeImport(entry);
}, [isDirty]);

// 3. Execute import
const executeImport = useCallback(async (entry) => {
    const result = await getProjectDesignMdSiteImportDocument(
        activeTarget.projectId,
        entry.slug
    );
    setDraft(result.content); // replace editor text only, do not save automatically
    setIsImportDrawerOpen(false);
}, [activeTarget?.projectId]);
```

El flujo de importación sigue un principio de "confirmación del usuario": después de la importación, solo se actualiza el contenido del editor y no se guarda nada automáticamente. Los usuarios pueden inspeccionar el contenido importado y guardarlo manualmente solo después de confirmar que se ve bien. La decisión final debe quedar en manos del usuario.

## Ejemplos prácticos

### Escenario 1: creación de DESIGN.md en la raíz del proyecto

cuando `DESIGN.md` no existe, el backend devuelve un estado de documento virtual. Esto permite que la interfaz evite un manejo especial para el caso de "el archivo no existe", y una API unificada simplifica la lógica del código:

```csharp
return new ProjectDesignDocumentDto
{
    Path = targetPath,
    Exists = false,  // virtual document state
    Content = string.Empty,
    Version = null
};

// Automatically create the file on first save
public async Task<SaveProjectDesignDocumentResultDto> SaveDesignDocumentAsync(...)
{
    Directory.CreateDirectory(targetDirectory);
    await File.WriteAllTextAsync(targetPath, input.Content);
    return new SaveProjectDesignDocumentResultDto { Created = !exists };
}
```

El beneficio de este diseño es que la interfaz no necesita lógica de casos especiales para los archivos faltantes. Al ocultar esa complejidad en el backend, el frontend puede centrarse más fácilmente en la experiencia del usuario.

### Escenario 2: importar una plantilla desde el sitio de diseño

Después de que el usuario selecciona la plantilla de diseño "Lineal" en el cajón de importación, el sistema recupera la `DESIGN.md` contenido a través del proxy backend. Todo el proceso es transparente para el usuario: solo elige una plantilla y el sistema maneja las solicitudes de red y la transformación de datos automáticamente.

```typescript
// 1. The system fetches DESIGN.md content through the backend proxy
GET /api/project/{id}/design-md/site-index/linear.app

// 2. The backend validates the slug and fetches content from upstream
var entry = FindDesignSiteEntry(catalog, "linear.app");
using var upstreamResponse = await httpClient.SendAsync(request);
var content = await upstreamResponse.Content.ReadAsStringAsync();

// 3. The frontend replaces the editor text
setDraft(result.content);
// The user reviews it and then saves it manually to disk
```

Todo el flujo permanece transparente para el usuario. Simplemente eligen una plantilla y el sistema se encarga de la creación de redes y la transformación entre bastidores. Esa es la experiencia que queremos: simple, pero poderosa.

### Escenario 3: Manejo de conflictos de versiones

Cuando varios usuarios editan lo mismo `DESIGN.md` al mismo tiempo, el sistema detecta conflictos de versiones. Este mecanismo optimista de control de concurrencia preserva la coherencia de los datos sin bloquear las ediciones del usuario:

```csharp
if (!string.Equals(currentVersion, expectedVersion, StringComparison.Ordinal))
{
    throw new BusinessException(
        ProjectDesignDocumentErrorCodes.VersionConflict,
        $"DESIGN.md at '{targetPath}' changed on disk.");
}
```

La interfaz detecta este error y le pregunta al usuario:

```typescript
// Frontend prompts the user to refresh and retry
<Alert>
    <AlertTitle>Version conflict</AlertTitle>
    <AlertDescription>
        The file was modified by another process. Please refresh to get the latest version and try again.
    </AlertDescription>
</Alert>
```

Este mecanismo optimista de control de concurrencia mantiene la coherencia de los datos sin bloquear a los usuarios mientras trabajan. Los conflictos son inevitables, pero al menos los usuarios deberían saber qué sucedió en lugar de perder silenciosamente sus cambios.

## Notas y mejores prácticas

### 1. Seguridad del camino

Validar siempre `repositoryPath` para evitar ataques de cruce de ruta. Nunca se puede hacer demasiado cuando se trata de seguridad:

```csharp
// Always validate repositoryPath to prevent path traversal attacks
return ValidateSubPathAsync(projectPath, repositoryPath);
// Reject dangerous inputs such as "../" and absolute paths
```

### 2. Estrategia de caché

Vista previa en caché de imágenes durante 24 horas, con un máximo de 160 archivos. El almacenamiento en caché moderado mejora el rendimiento, pero el equilibrio sigue siendo importante:

```csharp
// Cache preview images for 24 hours, with a maximum of 160 files
private static readonly TimeSpan PreviewCacheTtl = TimeSpan.FromHours(24);
private const int PreviewCacheMaxFiles = 160;
// Periodically clean up expired cache
```

### 3. Manejo de errores

Degradar con gracia cuando el sitio ascendente no esté disponible. Este diseño garantiza que incluso si falla una dependencia externa, la funcionalidad de edición principal seguirá funcionando normalmente:

```typescript
// Gracefully degrade when the upstream site is unavailable
try {
    const catalog = await getProjectDesignMdSiteImportCatalog(projectId);
} catch (error) {
    toast.error(t('project.designMd.siteImport.feedback.catalogLoadFailed'));
    // The main editing drawer remains available
}
```

Esta degradación elegante garantiza que incluso cuando las dependencias externas no estén disponibles, la función de edición principal continúe funcionando. Un sistema debe ser resiliente en lugar de colapsar en el momento en que algo sale mal.

### 4. Optimización de la experiencia del usuario

Confirme las sobrescrituras antes de importar y no las guarde automáticamente después de la importación. Los usuarios deben mantener el control de sus propias acciones:

```typescript
// Confirm overwrite before import
if (isDirty) {
    setConfirmMode('import');
    return;
}

// Do not save automatically after import; let the user confirm
setDraft(result.content); // update draft only
// The content is written to disk only after the user reviews it and clicks Save
```

### 5. Consideraciones de desempeño

Utilice una fábrica de clientes HTTP para evitar crear demasiadas conexiones. La gestión de recursos puede parecer pequeña, pero hacerlo bien puede marcar una gran diferencia:

```csharp
// Use an HTTP client factory to avoid creating too many connections
private const string DesignSiteProxyClientName = "ProjectDesignSiteProxy";
private static readonly TimeSpan DesignSiteProxyTimeout = TimeSpan.FromSeconds(8);
```

## Extensiones sugeridas

1. **Mejora de Markdown**: actualmente utilizamos un básico `Textarea`, pero podríamos actualizar a CodeMirror para resaltar la sintaxis y utilizar métodos abreviados de teclado. Cuando el editor se siente mejor, escribir documentación también se siente mejor.
2. **Modo de vista previa**: agregue una vista previa de Markdown en tiempo real para mejorar la experiencia de edición. Lo que ves es lo que obtienes siempre da a las personas más confianza.
3. **Combinación de diferencias**: implemente un algoritmo de combinación inteligente en lugar de un simple reemplazo de texto completo. Los conflictos son inevitables, pero el proceso de resolución de conflictos no tiene por qué ser doloroso.
4. **Almacenamiento en caché local**: caché `design.json` en la base de datos para reducir la dependencia del sitio externo. Cuantas menos dependencias tenga un sistema, más estable tiende a ser.

## Resumen

En el proyecto HagiCode, implementamos una solución completa de edición e importación en línea para `DESIGN.md` a través de la colaboración frontend-backend. El valor central de esta solución radica en los siguientes puntos:

- **Mayor eficiencia**: no es necesario cambiar de herramienta; La edición e importación de documentos de diseño se puede realizar en una interfaz web unificada.
- **Barrera de entrada más baja**: la importación de plantillas de diseño con un solo clic ayuda a que los nuevos proyectos comiencen rápidamente.
- **Seguro y confiable**: la validación de rutas, la detección de conflictos de versiones y los mecanismos de degradación elegantes mantienen el sistema estable.
- **Mejor experiencia de usuario**: el cajón global, la detección de estado sucio y los cuadros de diálogo de confirmación refinan la experiencia de interacción general.

Esta solución ya se está ejecutando en el proyecto HagiCode y ha resuelto los puntos débiles del equipo en torno a la gestión de documentos de diseño. Si enfrenta problemas similares, espero que este artículo le brinde algunas ideas útiles. Aquí no hay una teoría particularmente profunda, sólo el trabajo práctico de toparse con un problema y encontrar una manera de resolverlo.
