## Cómo construir una estación de distribución de descarga rápida con servidores en la nube de bajo costo

> El ancho de banda del almacenamiento en la nube es absurdamente caro, el acceso transfronterizo es terriblemente lento y el precio de CDN es suficiente para asustar a cualquiera... Si maneja la distribución de archivos, probablemente conozca bien estos problemas. En esta publicación, quiero compartir un enfoque de bajo costo que elaboramos mientras construíamos [HagiCode](https://hagicode.com): un servidor en la nube más una capa de almacenamiento en caché Nginx. El costo se redujo aproximadamente a la mitad, mientras que la velocidad de descarga mejoró bastante, lo cual fue al menos un poco reconfortante.

## Antecedentes

Cuando se trata de Internet, la velocidad de descarga y la estabilidad son realmente parte de la experiencia del usuario. Ya sea que esté ejecutando un proyecto de código abierto o un producto comercial, aún necesita brindar a los usuarios una forma confiable de descargar archivos.

Descargar archivos directamente desde el almacenamiento en la nube, como Azure Blob Storage o AWS S3, parece sencillo, pero conlleva bastantes problemas prácticos:

**Latencia de red**: el acceso transfronterizo y entre regiones puede ser lo suficientemente lento como para hacerte querer romper el teclado. Si los usuarios tienen que esperar una eternidad, la experiencia obviamente no será excelente.

**Costo del ancho de banda**: el tráfico de salida del almacenamiento en la nube es tremendamente caro. Acceder a Azure Blob Storage desde China continental cuesta alrededor de CNY 0,5 por GB, lo que significa que 1 TB por mes suma aproximadamente CNY 500. Para un equipo pequeño, esa no es una cantidad insignificante. Después de todo, el dinero de nadie proviene del viento.

**Restricciones de acceso**: en algunas regiones, el acceso a servicios en la nube en el extranjero es inestable y, a veces, simplemente no está disponible. Los usuarios quieren descargar los archivos pero no pueden, lo que resulta frustrante para todos.

**Costo de CDN**: Las CDN comerciales pueden resolver estos problemas, pero el precio es igual de real. La mayoría de los equipos pequeños simplemente no pueden justificarlo.

Entonces, ¿existe una solución que sea a la vez asequible y práctica? Sí. Utilice un servidor en la nube, un proxy inverso y una capa de almacenamiento en caché. Es un enfoque sencillo, pero funciona. El costo se reduce aproximadamente a la mitad y la velocidad también mejora, lo cual es una compensación decente.

## Diseño de Arquitectura

### Arquitectura general

Comencemos con la arquitectura completa:

```
User request
    ↓
DNS resolution
    ↓
┌─────────────────────────────────────┐
│   Reverse proxy layer (Traefik/Bunker Web) │ ← SSL termination, routing, security protection
├─────────────────────────────────────┤
│   Ports: 80/443                     │
│   Features: Automatic Let's Encrypt certificates │
│             Host routing            │
└─────────────────────────────────────┘
    ↓
┌─────────────────────────────────────┐
│   Cache layer (Nginx)               │ ← File caching, Gzip compression
├─────────────────────────────────────┤
│   Ports: 8080(server) / 8081(desktop) │
│   Cache strategy:                   │
│   - index.json: 1 hour              │
│   - other files: 7 days             │
│   Cache size: 1GB                   │
└─────────────────────────────────────┘
    ↓
┌─────────────────────────────────────┐
│   Origin (Azure Blob Storage)       │ ← File storage
└─────────────────────────────────────┘
```

La idea central de esta arquitectura es simple: **colocar un caché entre los usuarios y el almacenamiento en la nube**.

Las solicitudes de los usuarios llegan primero a la capa de proxy inverso en el servidor en la nube y luego la capa de caché de Nginx toma el control. Si el archivo solicitado ya está almacenado en caché, se devuelve inmediatamente. De lo contrario, Nginx lo recupera del almacenamiento en la nube y almacena una copia local al mismo tiempo. La próxima vez que alguien solicite el mismo archivo, no será necesario volver a utilizar el almacenamiento en la nube.

### ¿Por qué elegir esta arquitectura?

**Ventajas de los servidores en la nube**:
- Costo predecible: proveedores como Alibaba Cloud ofrecen servidores en la nube de bajo costo, con instancias de 1-2 vCPU y 2 GB de RAM a un precio de entre 50 y 100 CNY por mes.
- Implementación flexible: puede configurar reglas de proxy inverso y políticas de almacenamiento en caché libremente
- Geografía flexible: puede elegir regiones de servidores más cercanas a sus usuarios
- Buena escalabilidad: puede actualizar las especificaciones del servidor a medida que crece el tráfico

**Proxy inverso + arquitectura de caché**:
- Reduzca la presión de origen: almacene en caché los archivos activos para reducir el acceso directo al almacenamiento en la nube
- Menor costo: el tráfico del servidor en la nube es mucho más barato que la salida del almacenamiento en la nube
- Mejorar la velocidad: el acceso cercano y el ancho de banda del servidor suelen ser mejores que la entrega directa de almacenamiento en la nube

**¿Por qué elegir Nginx como capa de caché?**

Esta no fue una elección aleatoria. Nginx tiene varias ventajas reales aquí:

1. Alto rendimiento: Nginx es ampliamente reconocido por su excelente rendimiento de proxy inverso
2. Almacenamiento en caché maduro: el incorporado `proxy_cache` La característica es estable y confiable.
3. Bajo uso de recursos: puede ejecutarse con tan solo 256 MB de memoria
4. Configuración flexible: puede aplicar diferentes políticas de caché a diferentes tipos de archivos

## Capa de proxy inverso: Traefik vs Bunker Web

La solución de implementación de HagiCode admite dos opciones de proxy inverso y cada una tiene sus propios puntos fuertes:

| Opción | Características | Escenarios adecuados |
|------|------|----------|
| **Traefik** | SSL ligero, automático, configuración sencilla | Implementación básica, escenarios de poco tráfico |
| **Red de búnker** | WAF incorporado, protección DDoS, protección anti-bot | Escenarios de alta seguridad y alto tráfico |

### Traefik: la primera opción ligera

Traefik es un moderno equilibrador de carga y proxy inverso HTTP. Su mayor ventaja es que la configuración es sencilla y puede obtener certificados Let's Encrypt automáticamente.

Para implementaciones iniciales o escenarios de poco tráfico, Traefik suele ser una muy buena opción:
- Utiliza relativamente pocos recursos; 1,5 CPU y 512 MB de memoria son suficientes
- Los certificados SSL se configuran automáticamente, por lo que no es necesario que los administre usted mismo
- El enrutamiento se configura a través de etiquetas Docker, lo cual es bastante conveniente

### Bunker Web: para escenarios de alta seguridad

Bunker Web es un firewall de aplicaciones web basado en Nginx con una protección de seguridad más completa.

¿Cuándo debería considerar cambiarse a Bunker Web? Generalmente en casos como estos:
- Estás bajo un ataque DDoS
- Necesitas protección ModSecurity
- Quieres protección anti-bot
- Tienes requisitos de seguridad más estrictos

HagiCode proporciona la `switch-deployment.sh` script para que puedas cambiar rápidamente entre las dos opciones:

```bash
# Switch to Bunker Web
./switch-deployment.sh bunkerweb

# Switch back to Traefik
./switch-deployment.sh traefik

# Check current status
./switch-deployment.sh status
```

El script realiza comprobaciones previas, comprobaciones de estado y reversión automática, por lo que el proceso de cambio es bastante seguro y confiable.

## Configuración de la capa de caché de Nginx

La capa de caché es el núcleo de toda la arquitectura, por lo que la configuración de Nginx marca una gran diferencia en el rendimiento de la caché.

### Configuración de ruta de caché

```nginx
# Cache path configuration
proxy_cache_path /var/cache/nginx levels=1:2 keys_zone=azure_cache:10m
                   max_size=1g inactive=7d use_temp_path=off;
```

Detalles del parámetro:
- `levels=1:2`: jerarquía de directorios de caché con dos niveles para mejorar la eficiencia del acceso a archivos
- `keys_zone=azure_cache:10m`: zona de almacenamiento de claves de caché; 10 MB son suficientes para una gran cantidad de claves
- `max_size=1g`: el tamaño máximo de caché es 1 GB
- `inactive=7d`: elimina los archivos almacenados en caché si no se ha accedido a ellos durante 7 días
- `use_temp_path=off`: escribe directamente en el directorio de caché para un mejor rendimiento

### Estrategia de caché escalonada

Los diferentes tipos de archivos necesitan diferentes estrategias de caché:

```nginx
# Server download service
server {
    listen 8080;

    # Short-term cache for index.json (to allow timely updates)
    location /index.json {
        proxy_cache azure_cache;
        proxy_cache_valid 200 1h;
        proxy_cache_key "$scheme$server_port$request_uri";
        add_header X-Cache-Status $upstream_cache_status;
        add_header Cache-Control "public, max-age=3600";

        # Reverse proxy to Azure OSS
        proxy_pass https://${SERVER_DL_HOST}/${SERVER_DL_CONTAINER}$uri?${SERVER_DL_SAS_TOKEN};
        proxy_ssl_server_name on;
        proxy_ssl_protocols TLSv1.2 TLSv1.3;
    }

    # Long-term cache for static files such as installation packages
    location / {
        proxy_cache azure_cache;
        proxy_cache_valid 200 7d;
        proxy_cache_key "$scheme$server_port$request_uri";
        add_header X-Cache-Status $upstream_cache_status;
        add_header Cache-Control "public, max-age=604800";

        proxy_pass https://${SERVER_DL_HOST}/${SERVER_DL_CONTAINER}$uri?${SERVER_DL_SAS_TOKEN};
        proxy_ssl_server_name on;
        proxy_ssl_protocols TLSv1.2 TLSv1.3;
    }
}
```

**¿Por qué está diseñado de esta manera?**

`index.json` es el archivo de verificación de versión, por lo que debe actualizarse rápidamente. Con una ventana de caché de 1 hora, los usuarios pueden detectar una nueva versión como máximo una hora después de su publicación.

Los archivos estáticos, como los paquetes de instalación, cambian con poca frecuencia, por lo que almacenarlos en caché durante 7 días reduce en gran medida el acceso al origen. Cuando se necesita una actualización, puede borrar el caché manualmente.

**`X-Cache-Status` encabezado de respuesta**:

Este encabezado le ayuda a inspeccionar el comportamiento de los hits de caché:
- `HIT`: acierto en caché
- `MISS`: error de caché, obtenido del origen
- `EXPIRED`: caché caducado, recuperado del origen nuevamente
- `BYPASS`: caché omitido

Cómo comprobarlo:
```bash
curl -I https://server.dl.hagicode.com/app.zip
```

## Análisis de costos

Supongamos 1 TB de tráfico de descarga por mes. Hagamos los cálculos:

| Opción | Costo de tráfico | Costo del servidor | totales |
|------|----------|------------|------|
| OSS directo de Azure | Alrededor de 500 yuanes | CNY 0 | 500 yuanes |
| Servidor en la nube + OSS (proporción de aciertos de caché del 80 %) | 100 yuanes + 80 yuanes | 60 yuanes | 240 yuanes |
| CDN comercial | 300-500 yuanes | CNY 0 | 300-500 yuanes |

**Conclusión**: agregar una capa de caché puede reducir el costo de distribución en aproximadamente un 50 %.

Esta estimación supone una tasa de aciertos de caché del 80 %. En la práctica, si los archivos no cambian con frecuencia, la tasa de aciertos puede ser incluso mayor.

## Práctica de implementación

### Preparación del entorno

Primero, configure las variables de entorno:

```bash
cd /path/to/hagicode_aliyun_deployment/docker
cp .env.example .env
vi .env  # Fill in the Azure OSS SAS URL and Lark Webhook URL
```

**Importante**: el `.env` El archivo contiene información confidencial, como el token SAS y la URL del webhook. Nunca lo comprometas con el control de versiones.

### Configuración de DNS

Agregue los siguientes registros DNS A:
- `server.dl.hagicode.com` → IP del servidor
- `desktop.dl.hagicode.com` → IP del servidor

### Inicializar el servidor

Utilice Ansible para inicializar el servidor automáticamente:

```bash
cd /path/to/hagicode_aliyun_deployment
ansible-playbook -i ./ansible/inventory/hosts.yml ./ansible/playbooks/init.yml
```

Este manual maneja las siguientes tareas automáticamente:
- Crear el usuario de implementación
- Instalar Docker y Docker Compose
- Configurar claves SSH
- Establecer reglas de firewall

Ese es el trabajo de configuración principal y la automatización ahorra mucho tiempo.

### Implementar los servicios

```bash
./deploy.sh
```

El script de implementación le ayuda a hacer lo siguiente:
- Verificar la configuración del entorno
- Extrae el código más reciente
- Iniciar contenedores Docker
- Ejecutar controles de salud
- Enviar notificaciones de implementación (Lark)

Un comando es suficiente, lo que hace que el proceso sea conveniente.

### Verificar la implementación

```bash
# Check container status
docker ps

# Test the download domains
curl -I https://server.dl.hagicode.com/index.json
curl -I https://desktop.dl.hagicode.com/index.json
```

## Consejos de operaciones

### Gestión de caché

Los cachés también necesitan mantenimiento de vez en cuando:

**Verifique el uso del disco de caché**:
```bash
docker volume inspect docker_nginx-cache
du -sh /var/lib/docker/volumes/docker_nginx-cache/_data
```

**Borrar el caché manualmente**:
```bash
./clear-cache.sh
```

O ejecute los comandos manuales directamente si es necesario:
```bash
docker exec nginx sh -c "rm -rf /var/cache/nginx/*"
docker restart nginx
```

### Límites de recursos

En un servidor de 1 núcleo y 2 GB, la configuración del límite de recursos se ve así:

```yaml
services:
  traefik:
    deploy:
      resources:
        limits:
          cpus: '1.50'
          memory: 512M

  nginx:
    deploy:
      resources:
        limits:
          cpus: '0.50'
          memory: 256M
```

Para monitorear el uso de recursos, ocasionalmente puede ejecutar:
```bash
docker stats
```

### Prácticas de seguridad de tokens SAS

El token SAS es la credencial utilizada para acceder a Azure Blob Storage, por lo que filtrarlo sería grave:

- No cometas el `.env` archivo a control de versiones; ya esta en `.gitignore`
- Establezca un tiempo de vencimiento apropiado para el token SAS; se recomienda 1 año
- Limite los permisos del token SAS a solo lectura
- Rote los tokens SAS con regularidad

### Monitoreo y Alertas

HagiCode integra notificaciones de Lark/Feishu Webhook, que pueden enviar alertas para los siguientes eventos:
- Éxito o fracaso de la implementación
- Estado de borrado de caché
- Excepciones de servicio

Las notificaciones incluyen información del servidor, marcas de tiempo y detalles de errores, lo que hace que la resolución de problemas sea mucho más rápida.

### Extensiones de alta disponibilidad

Cuando un servidor ya no es suficiente, puedes considerar lo siguiente:

1. **Escalado horizontal**: implemente múltiples nodos y distribuya el tráfico con DNS round-robin o un equilibrador de carga
2. **CDN delante**: coloque un CDN delante de los servidores en la nube para un acceso aún más rápido
3. **Calentamiento de caché**: use scripts para precargar archivos activos en el caché

## Notas

Hay algunas cosas que vale la pena tener en cuenta:

1. **Certificados SSL**: Let's Encrypt tiene límites de velocidad, por lo que no cambie las implementaciones con demasiada frecuencia o la emisión del certificado puede fallar
2. **Borrado de caché**: después de actualizar archivos importantes, recuerde borrar el caché o los usuarios aún podrán descargar la versión anterior.
3. **Administración de registros**: limpie los registros de Docker con regularidad o el disco podría llenarse
4. **Estrategia de copia de seguridad**: haga una copia de seguridad de archivos como `Traefik acme.json` y configuración Web Bunker
5. **Monitoreo y alertas**: configure las notificaciones de Feishu para que pueda realizar un seguimiento del estado de implementación y responder rápidamente a los problemas

## Conclusión

Todo lo que se necesita es un servidor en la nube más una capa de almacenamiento en caché Nginx. HagiCode utiliza esta solución con un costo mensual bastante bajo, alrededor de 60-100 CNY para el servidor, y los resultados han sido muy sólidos. Las principales ventajas son:

- **Costo predecible**: aproximadamente un 50% más barato que usar el almacenamiento en la nube directamente o pagar por una CDN comercial
- **Implementación flexible**: elija Traefik o Bunker Web según sus necesidades
- **Gran escalabilidad**: puedes escalar horizontalmente o agregar una CDN más adelante si es necesario
- **Operaciones simples**: los scripts de Shell más Ansible simplifican la implementación automatizada

Para equipos pequeños y desarrolladores independientes que necesitan distribución de archivos, esta es definitivamente una opción práctica que vale la pena probar.

HagiCode ha estado ejecutando esta arquitectura de manera estable en producción durante un tiempo y las descargas de usuarios globales se han mantenido confiables. Si está buscando una solución similar, vale la pena intentarlo.
