## Creación de un grupo de aventuras con IA: una guía práctica para la configuración de colaboración entre múltiples agentes en HagiCode

> En el desarrollo de software moderno, un único agente de IA ya no es suficiente para necesidades complejas. ¿Cómo pueden colaborar varios asistentes de IA de diferentes empresas en el mismo proyecto? Este artículo comparte el enfoque de configuración de colaboración de múltiples agentes que el proyecto HagiCode desarrolló a través de la práctica del mundo real.

:::nota[Nota de contexto de tiempo]
Este artículo registra la práctica de configuración de múltiples agentes **publicada el 17 de marzo de 2026**. El alcance actual admitido a largo plazo sigue la configuración actual del backend: Claude Code, Codex, GitHub Copilot, OpenCode, Hermes, QoderCLI, Kiro, Kimi, Gemini, DeepAgents y Codebuddy. Entre ellos, `IFlowCli` Ahora se conserva solo como un valor compatible con el legado y ya no forma parte del conjunto con soporte activo. Si desea conocer el alcance actual de la instalación y el soporte, consulte [Instalación de la CLI del Agente AI](/es-419/related-software-installation/ai-agent-cli/) y [Guía de instalación](/es-419/installation).

## Antecedentes

Es probable que muchos desarrolladores hayan tenido esta experiencia: incorporar un asistente de IA a un proyecto realmente mejora la eficiencia de la codificación. Pero a medida que los requisitos se vuelven más complejos, un agente de IA empieza a quedarse corto. Quiere que maneje la revisión de código, la generación de documentación, las pruebas unitarias y más al mismo tiempo, pero el resultado a menudo es que no puede equilibrar todo bien y la calidad de salida se vuelve inconsistente.

Lo que es aún más frustrante es que una vez que intentas introducir varios asistentes de IA, las cosas se vuelven más complicadas. Cada Agente tiene su propio método de configuración, interfaz API y lógica de ejecución, e incluso pueden entrar en conflicto entre sí. Es como un equipo deportivo en el que cada jugador es fuerte individualmente, pero nadie sabe cómo coordinarse, por lo que todo el partido se convierte en un caos.

El proyecto HagiCode se topó con el mismo problema durante el desarrollo. Como proyecto complejo que involucra una extensión VSCode frontend, servicios de IA backend y un cliente de escritorio multiplataforma, en la versión **2026-03 en ese momento** necesitábamos integrar múltiples asistentes de IA de diferentes compañías a la vez: Claude Code, Codex, CodeBuddy, iFlow y más. Descubrir cómo permitirles coexistir armoniosamente en el mismo proyecto mientras se aprovechaban al máximo sus fortalezas individuales se convirtió en un problema crítico que teníamos que resolver.

Sólo eso ya sería suficiente problema. Después de todo, ¿quién quiere lidiar con un grupo de herramientas de inteligencia artificial que luchan entre sí todos los días?

El enfoque compartido en este artículo es la práctica de configuración de colaboración de múltiples agentes que desarrollamos en el proyecto HagiCode mediante prueba y error real y optimización repetida. Si también tiene dificultades con varios asistentes de IA trabajando juntos, este artículo puede brindarle algunas ideas. Tal vez. Después de todo, cada proyecto es diferente.

## Acerca de HagiCode

[HagiCode](https://hagicode.com) es un proyecto de asistente de codificación de IA que adopta un modelo de "fiesta de aventuras" en el que varios motores de IA trabajan juntos. Repositorio de proyectos: [github.com/HagiCode-org/site](https://github.com/HagiCode-org/site).

El enfoque de configuración de múltiples agentes que se comparte aquí es una de las técnicas centrales que permite a HagiCode mantener un desarrollo eficiente en proyectos complejos. No tiene nada de místico: simplemente convierte a un grupo de IA en un grupo de aventuras que realmente pueden coordinarse.

## Diseño de arquitectura multiagente de HagiCode

### De "ir en solitario" a "colaboración en equipo"

En los primeros días del proyecto HagiCode, también intentamos utilizar un único agente de IA para manejar todo. Rápidamente descubrimos un claro cuello de botella en ese enfoque: diferentes tareas exigen diferentes fortalezas. Algunas tareas requieren una comprensión contextual más sólida, mientras que otras necesitan una edición de código más precisa. A un agente le resulta difícil sobresalir en todos ellos.

Eso nos hizo darnos cuenta de que varios agentes tenían que trabajar juntos. Pero el problema era este: ¿cómo se permite que productos de IA de diferentes empresas coexistan pacíficamente en el mismo proyecto? Necesitábamos resolver varios problemas centrales:

1. **Complejidad de la gestión de la configuración**: cada Agente tiene diferentes métodos de configuración, interfaces API y modos de ejecución
2. **Protocolo de comunicación unificado**: necesitamos una forma estandarizada para que diferentes Agentes intercambien datos
3. **Coordinación de tareas y división del trabajo**: ¿cómo asignamos el trabajo de manera razonable para que cada agente pueda aprovechar sus puntos fuertes?

Con esas preguntas en mente, comenzamos a diseñar la arquitectura multiagente de HagiCode. Al final no fue tan complicado; sólo teníamos que pensarlo con claridad.

### Arquitectura general de un vistazo

Después de múltiples iteraciones, esta es la arquitectura que elegimos:

```
┌─────────────────────────────────────────────────────────────────┐
│                    AIProviderFactory                             │
│  (Factory pattern for unified management of all AI Providers)    │
├─────────────────────────────────────────────────────────────────┤
│  ClaudeCodeCli  │  CodexCli  │  CodebuddyCli  │  IFlowCli    │
│  (Anthropic)   │  (OpenAI)  │  (Zhipu GLM)    │  (Zhipu)     │
└─────────────────────────────────────────────────────────────────┘
```

La idea central es permitir que diferentes agentes de IA sean administrados por el mismo código a través de una interfaz de proveedor unificada. Al mismo tiempo, el patrón de fábrica se utiliza para crear y configurar dinámicamente estos proveedores, asegurando escalabilidad y flexibilidad en todo el sistema.

Es como la división del trabajo en la vida diaria. Cada uno tiene un papel; aquí simplemente convertimos esa idea en arquitectura de código.

### Tipos de agentes y división de responsabilidades

Basándonos en la experiencia del mundo real de HagiCode, asignamos diferentes responsabilidades a cada Agente:

| Agente | Proveedor | modelo | Uso primario |
|-------|----------|-------|-------------|
| ClaudeCodeCli | antrópico | glm-5-turbo | Generar soluciones técnicas y propuestas. |
| CódiceCli | OpenAI/Zed | gpt-5.4 | Ejecute cambios de código precisos |
| CódigobuddyCli | Zhipu | glm-4.7 | Refinar las descripciones y la documentación de las propuestas. |
| IFlowCli | Zhipu | glm-4.7 | Propuestas de archivo y registros históricos (configuración en ese momento; ahora solo compatible con legado) |
| OpenCodeCli | - | - | Edición de código de propósito general |
| GitHubCopilot | microsoft | - | Programación asistida y finalización de código. |

La lógica detrás de esta división del trabajo es simple: cada Agente tiene su propia área de fortaleza. Claude Code se desempeña bien en la comprensión y el análisis de requisitos complejos, por lo que maneja el diseño temprano de soluciones. Codex es más preciso a la hora de modificar código, por lo que es más adecuado para trabajos de implementación concretos. CodeBuddy ofrece una sólida rentabilidad, lo que lo convierte en una excelente opción para perfeccionar la documentación.

Después de todo, la herramienta adecuada para el trabajo adecuado suele ser la mejor opción. Hay muchos caminos hacia Roma; algunos son simplemente más fáciles de caminar que otros.

## Mecanismos de configuración básicos

### Diseño de interfaz de proveedor unificada

Para gestionar diferentes agentes de IA de forma unificada, primero debemos definir una interfaz común. En HagiCode, esa interfaz se ve así:

```csharp
public interface IAIProvider
{
    // Unified Provider interface
    Task<IAIProvider?> GetProviderAsync(AIProviderType providerType);
    Task<IAIProvider?> GetProviderAsync(string providerName, CancellationToken cancellationToken);
}
```

La interfaz parece simple, pero es la base de todo el sistema multiagente. Con una interfaz unificada, podemos llamar a productos de IA de diferentes empresas exactamente de la misma manera, sin importar lo que haya debajo.

En realidad, se trata sólo de simplificar las cosas complejas. Después de todo, lo simple es hermoso.

### Implementación del patrón de fábrica de proveedores

Una vez unificada la interfaz, la siguiente pregunta es cómo crear estas instancias de Proveedor. HagiCode usa el patrón de fábrica:

```csharp
private IAIProvider? CreateProvider(AIProviderType providerType, ProviderConfiguration config)
{
    return providerType switch
    {
        AIProviderType.ClaudeCodeCli =>
            ActivatorUtilities.CreateInstance<ClaudeCodeCliProvider>(_serviceProvider, Options.Create(config)),
        AIProviderType.CodebuddyCli =>
            ActivatorUtilities.CreateInstance<CodebuddyCliProvider>(_serviceProvider, Options.Create(config)),
        AIProviderType.CodexCli =>
            ActivatorUtilities.CreateInstance<CodexCliProvider>(_serviceProvider, Options.Create(config)),
        AIProviderType.IFlowCli =>
            ActivatorUtilities.CreateInstance<IFlowCliProvider>(_serviceProvider, Options.Create(config)),
        _ => null
    };
}
```

Esto utiliza la inyección de dependencia a través de `ActivatorUtilities.CreateInstance`, que puede crear dinámicamente instancias de proveedor en tiempo de ejecución mientras inyecta dependencias automáticamente. El beneficio de este diseño es que cuando se agrega un nuevo tipo de Agente, solo necesita agregar la clase de Proveedor correspondiente y luego agregar una rama de caso más en el método de fábrica. No es necesario modificar el código existente en absoluto.

Ésa es razón suficiente. ¿Quién quiere reescribir un montón de código antiguo cada vez que se agrega una nueva característica?

### Resolución de configuración dinámica

Para hacer la configuración más flexible, también implementamos un mecanismo de mapeo de tipos:

```csharp
public static AIProviderTypeExtensions
{
    private static readonly Dictionary<string, AIProviderType> _typeMap = new(
        StringComparer.OrdinalIgnoreCase)
    {
        ["ClaudeCodeCli"] = AIProviderType.ClaudeCodeCli,
        ["CodebuddyCli"] = AIProviderType.CodebuddyCli,
        ["CodexCli"] = AIProviderType.CodexCli,
        ["IFlowCli"] = AIProviderType.IFlowCli,
        // ...more type mappings
    };
}
```

El propósito de esta tabla de mapeo es convertir nombres de proveedores en forma de cadena en tipos de enumeración. Esto permite que los archivos de configuración utilicen nombres de cadenas intuitivos, mientras que el código interno utiliza enumeraciones con seguridad de tipos para el procesamiento.

La configuración debe ser lo más intuitiva posible. Nadie quiere memorizar un montón de nombres en clave oscuros.

### Archivo de configuración de ejemplo

En la práctica, todo se puede configurar en `appsettings.json`:

```yaml
AI:
  Providers:
    Providers:
      ClaudeCodeCli:
        Enabled: true
        Model: glm-5-turbo
        WorkingDirectory: /path/to/project
      CodebuddyCli:
        Enabled: true
        Model: glm-4.7
      CodexCli:
        Enabled: true
        Model: gpt-5.4
      IFlowCli:
        Enabled: true
        Model: glm-4.7
```

Cada proveedor puede configurar de forma independiente parámetros como la habilitación, la versión del modelo y el directorio de trabajo. Este diseño preserva la flexibilidad sin dejar de ser fácil de administrar y mantener.

En cierto modo, los archivos de configuración son como las opciones de la vida: puedes elegir habilitar o deshabilitar ciertas cosas. La única diferencia es que es más fácil arrepentirse de las elecciones de código más adelante.

## Flujo de tareas del grupo de aventuras

### El arte de la división de tareas

Una vez implementada la arquitectura técnica unificada, el siguiente paso es hacer que varios agentes trabajen juntos. HagiCode diseñó un mecanismo de flujo de tareas para que diferentes Agentes puedan manejar diferentes etapas del trabajo:

```
Proposal creation (user)
    │
    ▼
[Claude Code] ──generate proposal──▶ Proposal document
    │                               │
    │                               ▼
    │                      [Codebuddy] ──refine description──▶ Refined proposal
    │                               │
    │                               ▼
    │                      [Codex] ──execute changes──▶ Code changes
    │                               │
    │                               ▼
    └──────────────────────▶ [iFlow] ──archive──▶ Historical records
```

El beneficio de esta división del trabajo es que cada Agente solo necesita concentrarse en las tareas que mejor hace, en lugar de intentar hacerlo todo. Claude Code genera propuestas desde cero. Codebuddy aclara las descripciones de las propuestas. Codex convierte las propuestas en cambios de código reales. iFlow archiva y conserva esos cambios.

En realidad, esto es solo trabajo en equipo, igual que en la vida diaria. Cada uno tiene un papel y sólo juntos se puede hacer algo grande. Aquí, los miembros del equipo resultan ser IA.

### Conclusiones prácticas clave

En la operación real, resumimos las siguientes lecciones:

**1. La estrategia de selección de agentes importa**

Las tareas no deben asignarse a la ligera; deben coincidir con las fortalezas de cada Agente:
- Generación de propuestas: use Claude Code, porque tiene una comprensión contextual más sólida
- Ejecución de código: utilice Codex, porque es más preciso para la modificación del código.
- Mejora de la propuesta: utilice Codebuddy, porque ofrece una sólida rentabilidad
- Almacenamiento de archivos: utilice iFlow, porque es estable y fiable

Después de todo, asignar a la persona adecuada la tarea adecuada es un principio atemporal.

**2. El aislamiento de la configuración garantiza la estabilidad**

La configuración de cada agente se administra de forma independiente, admite anulaciones de variables de entorno y utiliza directorios de trabajo separados. Como resultado, un error de configuración en un Agente no afecta a los demás.

Esto es como los límites personales en la vida. Cada uno necesita su propio espacio; la no interferencia hace posible la coexistencia.

**3. Mecanismo de manejo de errores**

Una falla en un solo Agente no debería afectar el flujo de trabajo general. Implementamos una estrategia alternativa: cuando un agente falla, el sistema puede cambiar automáticamente a un plan de respaldo u omitir ese paso y continuar con tareas posteriores. Al mismo tiempo, el registro completo facilita la resolución de problemas posteriormente.

Nadie puede garantizar que nunca se producirán errores. La clave es cómo los manejas. La vida funciona de la misma manera.

**4. Monitoreo y observabilidad**

A través del protocolo ACP (nuestro protocolo de comunicación personalizado basado en JSON-RPC 2.0), podemos rastrear el estado de ejecución de cada Agente. El aislamiento de sesiones garantiza la seguridad de la concurrencia, mientras que el almacenamiento en caché dinámico mejora el rendimiento.

Las cosas que no puedes ver suelen ser las que tienen más probabilidades de salir mal. Algo de visibilidad siempre es mejor que volar a ciegas.

## Resultados y beneficios del mundo real

Después de adoptar esta configuración de colaboración de múltiples agentes, la eficiencia de desarrollo del proyecto HagiCode mejoró significativamente. Específicamente:

1. **La capacidad de manejo de tareas se duplicó**: en el pasado, un Agente tenía que manejar muchos tipos de tareas a la vez; ahora las tareas se pueden procesar en paralelo y el rendimiento ha aumentado dramáticamente
2. **Calidad de salida más estable**: cada agente se centra solo en lo que hace mejor, por lo que tanto la coherencia como la calidad mejoran
3. **Menor coste de mantenimiento**: las interfaces unificadas y la gestión de la configuración hacen que todo el sistema sea más fácil de mantener y ampliar
4. **Agregar nuevos Agentes es simple**: para integrar un nuevo producto de IA, solo necesita implementar la interfaz y agregar la configuración, sin cambiar la lógica central

Este enfoque no solo resolvió los propios problemas de HagiCode, sino que también demostró que la colaboración entre múltiples agentes es una opción arquitectónica viable.

Las ganancias fueron bastante notables. El proceso fue un poco complicado.

## Conclusión

Este artículo comparte la experiencia práctica del proyecto HagiCode con la configuración de colaboración de múltiples agentes. Las principales conclusiones incluyen:

1. **Interfaces estandarizadas**: `IAIProvider` unifica el comportamiento de diferentes Agentes, permitiendo que el código ignore qué producto de la empresa está debajo
2. **Patrón de fábrica**: `ActivatorUtilities.CreateInstance` Crea dinámicamente instancias de proveedor, admitiendo la configuración del tiempo de ejecución y la inyección de dependencia.
3. **Unificación de protocolo**: el protocolo ACP proporciona comunicación estandarizada entre Agentes a través de un mecanismo bidireccional basado en JSON-RPC 2.0
4. **Enrutamiento de tareas**: asigne el trabajo de manera razonable entre diferentes agentes para que cada uno pueda aprovechar sus puntos fuertes, en lugar de esperar que un agente haga todo.

Este diseño no solo resuelve el problema de "múltiples agentes peleando entre sí", sino que también utiliza el mecanismo de flujo de tareas del grupo de aventuras para hacer que el proceso de desarrollo sea más automatizado y especializado.

Si también está considerando introducir varios asistentes de IA, espero que este artículo le brinde algunos puntos de referencia útiles. Por supuesto, cada proyecto es diferente y el enfoque específico aún debe adaptarse a la situación real. No existe una solución única para todos; la mejor solución es la que se adapta a ti.

No es necesario poseer cosas o personas hermosas. Mientras sigan siendo hermosos, simplemente apreciar esa belleza es suficiente. Las soluciones técnicas son las mismas: la que más te conviene es la mejor...
