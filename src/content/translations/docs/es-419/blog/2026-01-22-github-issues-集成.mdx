## Creación de una integración de problemas de GitHub desde cero: práctica de conexión directa frontend de HagiCode

> Este artículo documenta el proceso completo de integración de GitHub Issues en la plataforma HagiCode. Exploraremos cómo utilizar una arquitectura de "conexión directa de frontend + backend mínimo" para lograr una autenticación OAuth segura y una sincronización de problemas eficiente mientras se mantiene el backend liviano.

## Antecedentes: ¿Por qué integrar GitHub?

Como plataforma de desarrollo asistida por IA, el valor principal de HagiCode radica en conectar las ideas con la implementación. Pero en el uso real, descubrimos que después de que los usuarios completan una propuesta en HagiCode, a menudo necesitan copiar manualmente el contenido en GitHub Issues para realizar un seguimiento del proyecto.

Esto crea varios puntos débiles obvios:

1.  **Flujo de trabajo fragmentado**: los usuarios deben alternar entre dos sistemas. La experiencia no es fluida y la información clave se puede perder fácilmente al copiar y pegar.
2.  **Colaboración inconveniente**: Otros miembros del equipo están acostumbrados a verificar tareas en GitHub y no pueden ver directamente el progreso de la propuesta dentro de HagiCode.
3.  **Trabajo manual repetido**: cada vez que se actualiza una propuesta, alguien tiene que actualizar manualmente el problema correspondiente en GitHub, lo que agrega costos de mantenimiento innecesarios.

Para resolver este problema, decidimos introducir la función **Integración de problemas de GitHub**, conectando sesiones de HagiCode con repositorios de GitHub para habilitar la "sincronización con un solo clic".

## Elección técnica: conexión directa frontend frente a proxy backend

Al diseñar el enfoque de integración, teníamos dos opciones frente a nosotros: el tradicional "modelo de proxy backend" y el más agresivo "modelo de conexión directa frontend".

### Comparación de soluciones

En el **modelo de proxy de backend** tradicional, cada solicitud del frontend debe pasar primero por nuestro backend, que luego llama a la API de GitHub. Esto centraliza la lógica, pero también supone una carga significativa para el backend:

1.  **Backend inflado**: necesitaríamos escribir un contenedor de cliente API de GitHub dedicado y también manejar la compleja máquina de estado OAuth.
2.  **Riesgo de token**: el token de GitHub del usuario debería almacenarse en la base de datos backend. Incluso con el cifrado, esto sigue aumentando la superficie de seguridad.
3.  **Costo de desarrollo**: necesitaríamos migraciones de bases de datos para almacenar tokens y un servicio de sincronización adicional para mantener.

El **modelo de conexión directa frontend** es mucho más ligero. En este enfoque, utilizamos el backend solo para el paso de "intercambio secreto" más sensible (la devolución de llamada de OAuth). Después de obtener el token, lo almacenamos directamente en el navegador. `localStorage`. Las operaciones posteriores, como la creación de problemas y la actualización de comentarios, se envían directamente desde la interfaz a GitHub a través de HTTP.

| Dimensión de comparación | Modelo de proxy de back-end | Modelo de conexión directa frontend |
| :--- | :--- | :--- |
| **Complejidad del backend** | Requiere un servicio OAuth completo y un cliente API de GitHub | Solo necesita un punto final de devolución de llamada de OAuth |
| **Gestión de tokens** | Debe estar cifrado y almacenado en la base de datos, con riesgo de fuga. | Almacenado en el navegador y visible sólo para el usuario. |
| **Costo de implementación** | Requiere migraciones de bases de datos y desarrollo multiservicio. | Principalmente trabajo frontend |
| **Experiencia de usuario** | Lógica centralizada, pero la latencia del servidor puede ser ligeramente mayor | Respuesta extremadamente rápida con interacción directa de GitHub |

Como queríamos una integración rápida y cambios mínimos en el backend, **finalmente elegimos el "modelo de conexión directa del frontend"**. Es como darle al navegador un "pase temporal". Una vez que obtiene el pase, el navegador puede manejar las cosas en GitHub por sí solo sin pedirle aprobación al administrador backend cada vez.

## Diseño central: flujo de datos y seguridad

Después de decidirnos por la arquitectura, necesitábamos diseñar el flujo de datos específico. El núcleo del proceso de sincronización es cómo obtener el token de forma segura y utilizarlo de manera eficiente.

### Diagrama de arquitectura general

Todo el sistema se puede resumir en tres roles: el navegador (frontend), el backend de HagiCode y GitHub.

```text
+--------------+        +--------------+        +--------------+
| Frontend     |        |   Backend    |        |   GitHub     |
| React        |        |   ASP.NET    |        |   REST API   |
|              |        |              |        |              |
|  +--------+  |        |              |        |              |
|  | OAuth  |--+--------> /callback    |        |              |
|  | Flow   |  |        |              |        |              |
|  +--------+  |        |              |        |              |
|              |        |              |        |              |
|  +--------+  |        |  +--------+  |        |  +--------+  |
|  | GitHub |  +------------> Session |  +----------> Issues |  |
|  | API    |  |        |  |Metadata|  |        |  |        |  |
|  | Direct |  |        |  +--------+  |        |  +--------+  |
|  +--------+  |        |              |        |              |
+--------------+        +--------------+        +--------------+
```

**El punto clave es**: solo un pequeño paso en OAuth (intercambiar el código por un token) debe pasar por el backend. Después de eso, el trabajo pesado (crear problemas) se maneja directamente entre la interfaz y GitHub.

### Flujo de datos de sincronización en detalle

Cuando el usuario hace clic en el botón "Sincronizar con GitHub" en la interfaz de usuario de HagiCode, se llevan a cabo una serie de acciones complejas:

```text
User clicks "Sync to GitHub"
         │
         ▼
1. Frontend checks localStorage for the GitHub token
         │
         ▼
2. Format issue content (convert the Proposal into Markdown)
         │
         ▼
3. Frontend directly calls the GitHub API to create/update the issue
         │
         ▼
4. Call the HagiCode backend API to update Session.metadata (store issue URL and other info)
         │
         ▼
5. Backend broadcasts the SessionUpdated event via SignalR
         │
         ▼
6. Frontend receives the event and updates the UI to show the "Synced" state
```

### Diseño de seguridad

La seguridad es siempre la máxima prioridad al integrar servicios de terceros. Hicimos las siguientes consideraciones:

1.  **Defenderse contra ataques CSRF**: Generar un ataque aleatorio `state` parámetro durante la redirección de OAuth y guárdelo en `sessionStorage`. Valide estrictamente el estado en la devolución de llamada para evitar solicitudes falsificadas.
2.  **Almacenamiento de token aislado**: el token se almacena solo en el navegador `localStorage`. Al utilizar la política del mismo origen, solo los scripts de HagiCode pueden leerla, evitando el riesgo de que una fuga de base de datos del lado del servidor afecte a los usuarios.
3.  **Límites de error**: Diseñamos un manejo dedicado para errores comunes de la API de GitHub (como 401 token vencido, 422 falla de validación y 429 limitación de velocidad), para que los usuarios reciban comentarios amigables.

## En la práctica: detalles de implementación en código

La teoría sólo llega hasta cierto punto. Veamos cómo funciona realmente el código.

### 1. Cambios mínimos en el backend

El backend solo necesita hacer dos cosas: almacenar información de sincronización y manejar la devolución de llamada de OAuth.

### 2. Flujo de interfaz OAuth

Este es el punto de entrada de la conexión. Usamos el flujo de código de autorización estándar.

```typescript
// src/HagiCode.Client/src/services/githubOAuth.ts

// Generate the authorization URL and redirect
export async function generateAuthUrl(): Promise<string> {
  const state = generateRandomString(); // Generate a random string for CSRF protection
  sessionStorage.setItem('hagicode_github_state', state);
  
  const params = new URLSearchParams({
    client_id: clientId,
    redirect_uri: window.location.origin + '/settings?tab=github&oauth=callback',
    scope: ['repo', 'public_repo'].join(' '),
    state: state,
  });
  
  return `https://github.com/login/oauth/authorize?${params.toString()}`;
}

// Handle the code-to-token exchange on the callback page
export async function exchangeCodeForToken(code: string, state: string): Promise<GitHubToken> {
  // 1. Validate state to prevent CSRF
  const savedState = sessionStorage.getItem('hagicode_github_state');
  if (state !== savedState) throw new Error('Invalid state parameter');

  // 2. Call the backend API to exchange the token
  // Note: this must go through the backend because ClientSecret cannot be exposed to the frontend
  const response = await fetch('/api/GitHubOAuth/callback', {
    method: 'POST',
    headers: { 'Content-Type': 'application/json' },
    body: JSON.stringify({ code, state, redirectUri: window.location.origin + '/settings?tab=github&oauth=callback' }),
  });

  if (!response.ok) throw new Error('Failed to exchange token');
  
  const token = await response.json();
  
  // 3. Save into LocalStorage
  saveToken(token);
  return token;
}
```

### 3. Envoltorio del cliente API de GitHub

Una vez que tengamos el token, necesitamos una herramienta sólida para llamar a la API de GitHub.

```typescript
// src/HagiCode.Client/src/services/githubApiClient.ts

const GITHUB_API_BASE = 'https://api.github.com';

// Core request wrapper
async function githubApi<T>(endpoint: string, options: RequestInit = {}): Promise<T> {
  const token = localStorage.getItem('gh_token');
  if (!token) throw new Error('Not connected to GitHub');
  
  const response = await fetch(`${GITHUB_API_BASE}${endpoint}`, {
    ...options,
    headers: {
      ...options.headers,
      Authorization: `Bearer ${token}`,
      Accept: 'application/vnd.github.v3+json', // Specify the API version
    },
  });
  
  // Error handling logic
  if (!response.ok) {
    if (response.status === 401) throw new Error('GitHub token expired, please reconnect');
    if (response.status === 403) throw new Error('No permission to access this repository or rate limit exceeded');
    if (response.status === 422) throw new Error('Issue validation failed, the title may be duplicated');
    throw new Error(`GitHub API Error: ${response.statusText}`);
  }
  
  return response.json();
}

// Create issue
export async function createIssue(owner: string, repo: string, data: { title: string, body: string, labels: string[] }) {
  return githubApi(`/repos/${owner}/${repo}/issues`, {
    method: 'POST',
    body: JSON.stringify(data),
  });
}
```

### 4. Formato y sincronización de contenidos

El último paso es convertir los datos de la sesión de HagiCode al formato de una edición de GitHub. Es un poco como un trabajo de traducción.

```typescript
// Convert a Session object into a Markdown string
function formatIssueForSession(session: Session): string {
  let content = `# ${session.title}\n\n`;
  content += `**> HagiCode Session:** #${session.code}\n`;
  content += `**> Status:** ${session.status}\n\n`;
  content += `## Description\n\n${session.description || 'No description provided.'}\n\n`;
  
  // If this is a Proposal session, add extra fields
  if (session.type === 'proposal') {
    content += `## Chief Complaint\n\n${session.chiefComplaint || ''}\n\n`;
    // Add a deep link so users can jump back from GitHub to HagiCode
    content += `---\n\n**[View in HagiCode](hagicode://sessions/${session.id})**\n`;
  }
  
  return content;
}

// Main logic when clicking the sync button
const handleSync = async (session: Session) => {
  try {
    const repoInfo = parseRepositoryFromUrl(session.repoUrl); // Parse the repository URL
    if (!repoInfo) throw new Error('Invalid repository URL');

    toast.loading('Syncing to GitHub...');
    
    // 1. Format content
    const issueBody = formatIssueForSession(session);
    
    // 2. Call API
    const issue = await githubApiClient.createIssue(repoInfo.owner, repoInfo.repo, {
      title: `[HagiCode] ${session.title}`,
      body: issueBody,
      labels: ['hagicode', 'proposal', `status:${session.status}`],
    });
    
    // 3. Update Session Metadata (save the issue link)
    await SessionsService.patchApiSessionsSessionId(session.id, {
      metadata: {
        githubIssue: {
          owner: repoInfo.owner,
          repo: repoInfo.repo,
          issueNumber: issue.number,
          issueUrl: issue.html_url,
          syncedAt: new Date().toISOString(),
        }
      }
    });

    toast.success('Synced successfully!');
  } catch (err) {
    console.error(err);
    toast.error('Sync failed, please check your token or network');
  }
};
```

## Resumen y perspectivas

Con este enfoque de "conexión directa de frontend", logramos una integración perfecta de GitHub Issues con la menor cantidad de código de backend posible.

### Ganancias clave
1.  **Alta eficiencia de desarrollo**: los cambios en el backend son mínimos, principalmente un campo de base de datos adicional y un punto final de devolución de llamada OAuth simple. La mayor parte de la lógica se completa en la interfaz.
2.  **Seguridad sólida**: el token no pasa a través de la base de datos del servidor, lo que reduce el riesgo de fuga.
3.  **Excelente experiencia de usuario**: las solicitudes se inician directamente desde el frontend, por lo que la velocidad de respuesta es rápida y no hay necesidad de reenviar el backend.

### Cosas a tener en cuenta
Hay algunos errores a tener en cuenta durante la implementación real:
- **Configuración de la aplicación OAuth**: recuerde ingresar la información correcta `Authorization callback URL` en la configuración de tu aplicación GitHub OAuth (normalmente `http://localhost:3000/settings?tab=github&oauth=callback`).
- **Límites de velocidad**: La API de GitHub limita de manera bastante estricta las solicitudes no autenticadas, pero con un token la cuota suele ser suficiente (5000 solicitudes/hora).
- **Análisis de URL**: los usuarios ingresan todo tipo de URL de repositorio, así que asegúrese de que su expresión regular pueda coincidir `.git` sufijos, formatos SSH y casos similares.

### Mejoras futuras
La característica actual sigue siendo la sincronización unidireccional (HagiCode -> GitHub). En el futuro, planeamos implementar la sincronización bidireccional a través de GitHub Webhooks. Por ejemplo, si se cierra un problema en GitHub, el estado de la sesión en el lado de HagiCode también podría actualizarse automáticamente. Eso requerirá que expongamos un punto final de webhook en el backend, lo que será un siguiente paso interesante.
