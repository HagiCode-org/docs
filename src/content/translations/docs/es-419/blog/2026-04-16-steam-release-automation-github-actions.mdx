## Cómo automatizar los lanzamientos de Steam con acciones de GitHub

> Este artículo comparte la solución completa que implementamos para lanzamientos automatizados de Steam en el proyecto HagiCode Desktop, que cubre el flujo de automatización de un extremo a otro desde GitHub Release a la plataforma Steam, incluidos detalles técnicos clave como la autenticación Steam Guard y las cargas de Depot multiplataforma.

## Antecedentes

El flujo de trabajo de lanzamiento en Steam es bastante diferente de la distribución de aplicaciones tradicional. Steam tiene su propio sistema completo de entrega de actualizaciones. Los desarrolladores deben usar SteamCMD para cargar artefactos de compilación en la red CDN de Steam, en lugar de simplemente colocar un enlace de descarga como en otras plataformas.

El proyecto HagiCode Desktop se está preparando para un lanzamiento en Steam, lo que introdujo algunos desafíos nuevos en nuestro flujo de trabajo de lanzamiento:

1. Necesitábamos convertir los artefactos de compilación existentes a un formato compatible con Steam.
2. Tuvimos que subirlos a Steam a través de SteamCMD.
3. También tuvimos que manejar la autenticación Steam Guard.
4. Necesitábamos admitir cargas de Depot multiplataforma para Linux, Windows y macOS.
5. Queríamos un flujo totalmente automatizado desde GitHub Release a Steam.

El proyecto ya había implementado el "modo de versión portátil", que permite a la aplicación detectar cargas útiles de servicios fijos empaquetadas en el `extra` directorio. Nuestro objetivo era integrar ese modo de versión portátil a la perfección con la distribución de Steam.

## Acerca de HagiCode

La solución compartida en este artículo proviene de nuestra experiencia práctica en el [HagiCode](https://hagicode.com) proyecto. HagiCode es un asistente de codificación de IA que admite el uso de escritorio. Dado que estamos trabajando activamente para el lanzamiento en Steam, necesitábamos establecer un flujo de trabajo de lanzamiento automatizado confiable.

## Diseño de Arquitectura

El núcleo de todo el proceso de lanzamiento de Steam es un flujo de trabajo de GitHub Actions que divide el proceso en tres etapas principales:

```
┌─────────────────────────────────────────────────────────────┐
│ GitHub Actions Workflow (Steam Release)                    │
├─────────────────────────────────────────────────────────────┤
│ 1. Preparation Stage:                                      │
│    - Check out portable-version code                       │
│    - Download build artifacts from GitHub Release          │
│    - Extract and prepare the Steam content directory       │
│                                                             │
│ 2. SteamCMD Setup:                                         │
│    - Install or reuse SteamCMD                             │
│    - Authenticate with Steam Guard                         │
│                                                             │
│ 3. Release Stage:                                          │
│    - Generate Depot VDF configuration files                │
│    - Generate App Build VDF configuration files            │
│    - Invoke SteamCMD to upload to Steam                    │
└─────────────────────────────────────────────────────────────┘
```

Este diseño ofrece varias ventajas:
- Reutiliza los artefactos existentes de GitHub Release y evita reconstruir los mismos resultados.
- Utiliza corredores autohospedados para un aislamiento seguro.
- Admite cambiar entre el modo de vista previa y las ramas de lanzamiento formal.
- Incluye registro y manejo completo de errores, lo que facilita el diagnóstico de las fallas.

## Implementación del flujo de trabajo

### Diseño de parámetros de disparo

Nuestro flujo de trabajo admite los siguientes parámetros clave:

```yaml
inputs:
  release:           # Portable Version release tag
    description: 'Version tag to release (for example v1.0.0)'
    required: true
  steam_preview:     # Whether to generate a preview build
    description: 'Whether to use preview mode'
    required: false
    default: 'false'
  steam_branch:      # Steam branch to set live
    description: 'Target Steam branch'
    required: false
    default: 'preview'
  steam_description: # Build description override
    description: 'Build description'
    required: false
```

### Configuración del ejecutor autohospedado

Por razones de seguridad, utilizamos un corredor autohospedado con el `steam` etiqueta:

```yaml
runs-on:
  - self-hosted
  - Linux
  - X64
  - steam
```

Esto garantiza que las versiones de Steam se ejecuten en un ejecutor dedicado y mantenga las credenciales confidenciales aisladas de forma segura.

### Control de concurrencia

Para evitar que los lanzamientos de la misma versión interfieran entre sí, configuramos el control de concurrencia:

```yaml
concurrency:
  group: portable-version-steam-${{ github.event.inputs.release }}
  cancel-in-progress: false
```

Note que `cancel-in-progress: false` se establece aquí porque los lanzamientos de Steam pueden tardar un poco y no queremos que una ejecución recién activada cancele una que ya se está cargando.

## Implementación del script principal

### Preparación de entradas de lanzamiento

el `prepare-steam-release-input.mjs` El script es responsable de preparar los insumos necesarios para el lanzamiento:

```javascript
// Download the GitHub Release build manifest and artifact inventory
const buildManifest = await downloadBuildManifest(releaseTag);
const artifactInventory = await downloadArtifactInventory(releaseTag);

// Download compressed archives for each platform
for (const platform of ['linux-x64', 'win-x64', 'osx-universal']) {
  const artifactUrl = getArtifactUrl(artifactInventory, platform);
  await downloadArtifact(artifactUrl, platform);
}

// Extract into the Steam content directory structure
await extractToSteamContent(sources, contentRoot);
```

### Autenticación de Steam Guard

Steam requiere que las cuentas estén protegidas con Steam Guard, por lo que implementamos un algoritmo de generación de código basado en secreto compartido:

```javascript
function generateSteamGuardCode(sharedSecret, timestamp = Date.now()) {
  const secret = decodeSharedSecret(sharedSecret);
  const time = Math.floor(timestamp / 1000 / 30);

  const timeBuffer = Buffer.alloc(8);
  timeBuffer.writeBigUInt64BE(BigInt(time));

  // Use HMAC-SHA1 to generate a time-based one-time code
  const hash = crypto.createHmac('sha1', secret)
    .update(timeBuffer)
    .digest();

  // Convert it into a 5-character Steam Guard code
  const code = steamGuardCode(hash);
  return code;
}
```

Esta implementación se basa en el mecanismo TOTP (Contraseña de un solo uso basada en tiempo) de Steam Guard, generando un nuevo código de verificación cada 30 segundos.

### Generación de configuración VDF

VDF (Valve Data Format) es el formato de configuración utilizado por Steam. Necesitamos generar dos tipos de archivos VDF:

**Depot VDF** se utiliza para configurar el contenido para cada plataforma:

```javascript
function buildDepotVdf(depotId, contentRoot) {
  return [
    '"DepotBuildConfig"',
    '{',
    `  "DepotID" "${escapeVdf(depotId)}"`,
    `  "ContentRoot" "${escapeVdf(contentRoot)}"`,
    '  "FileMapping"',
    '  {',
    '    "LocalPath" "*"',
    '    "DepotPath" "."',
    '    "recursive" "1"',
    '  }',
    '}'
  ].join('\n');
}
```

**App Build VDF** se utiliza para configurar toda la compilación de la aplicación:

```javascript
function buildAppBuildVdf(appId, depotBuilds, description, setLive) {
  const vdf = [
    '"appbuild"',
    '{',
    `  "appid" "${appId}"`,
    `  "desc" "${escapeVdf(description)}"`,
    `  "contentroot" "${escapeVdf(contentRoot)}"`,
    '  "buildoutput" "build_output"',
    '  "depots"',
    '  {'
  ];

  for (const [depotId, depotVdfPath] of Object.entries(depotBuilds)) {
    vdf.push(`    "${depotId}" "${depotVdfPath}"`);
  }

  if (setLive) {
    vdf.push(`  }`);
    vdf.push(`  "setlive" "${setLive}"`);
  }

  vdf.push('}');
  return vdf.join('\n');
}
```

### Invocación de SteamCMD

Finalmente, la carga se realiza invocando SteamCMD:

```javascript
await runCommand(steamcmdPath, [
  '+login', steamUsername, steamPassword, steamGuardCode,
  '+run_app_build', appBuildPath,
  '+quit'
]);
```

Este es el salto final en todo el flujo de trabajo. Una vez que lo logra, se realiza la liberación.

## Manejo de depósitos multiplataforma

Steam utiliza el sistema Depot para gestionar contenido para diferentes plataformas. Admitimos tres depósitos principales:

| Plataforma | Identificador de depósito | Soporte de arquitectura |
|-----|-----------|---------|
| linux | `linux-x64` | x64_64 |
| ventanas | `win-x64` | x64_64 |
| macos | `osx-universal` | universal, x64_64, brazo64 |

Cada Depot tiene su propio directorio de contenido y archivo de configuración VDF. Esto garantiza que los usuarios de diferentes plataformas solo descarguen el contenido que realmente necesitan.

## Flujo de trabajo de lanzamiento

### Paso 1: preparar la versión de GitHub

Primero, cree una versión de GitHub en el `portable-version` repositorio que incluye:
- Archivos comprimidos para cada plataforma.
- Construir manifiesto (`{tag}.build-manifest.json`)
- Inventario de artefactos (`{tag}.artifact-inventory.json`)

### Paso 2: Activar el flujo de trabajo de lanzamiento de Steam

Active el flujo de trabajo manualmente a través de GitHub Actions y complete los parámetros requeridos:
- `release`: la etiqueta de versión para publicar, como `v1.0.0`
- `steam_branch`: la rama de destino, como `preview` o `public`
- `steam_preview`: si se utiliza el modo de vista previa

### Paso 3: ejecutar el flujo de liberación automáticamente

El flujo de trabajo realiza automáticamente los siguientes pasos:

1. Descargue y extraiga los artefactos de la versión GitHub.
2. Instale o actualice SteamCMD.
3. Genere archivos de configuración Steam VDF.
4. Autenticar con Steam Guard.
5. Sube contenido a Steam CDN.
6. Establece la rama especificada en vivo.

Una vez que se completa esta secuencia, se cubre todo el camino de liberación.

## Guía de configuración

### Configuración de secretos requerida

Configure los siguientes secretos en la configuración del repositorio de GitHub:

| Nombre secreto | Descripción |
|-----------|------|
| `STEAM_USERNAME` | nombre de usuario de la cuenta de Steam |
| `STEAM_PASSWORD` | contraseña de la cuenta de vapor |
| `STEAM_SHARED_SECRET` | Secreto compartido de Steam Guard (opcional) |
| `STEAM_GUARD_CODE` | Código Steam Guard (opcional) |
| `STEAM_APP_ID` | ID de la aplicación Steam |
| `STEAM_DEPOT_ID_LINUX` | ID de depósito de Linux |
| `STEAM_DEPOT_ID_WINDOWS` | ID de depósito de Windows |
| `STEAM_DEPOT_ID_MACOS` | ID de depósito de macOS |

No hay nada especialmente inusual en estos entornos. Simplemente necesita tener todos los valores esperados en su lugar.

### Configuración de variables de entorno

| Nombre de la variable | Descripción | Valor predeterminado |
|---------|------|--------|
| `PORTABLE_VERSION_STEAMCMD_ROOT` | Directorio de instalación de SteamCMD | `~/.local/share/portable-version/steamcmd` |

## Mejores prácticas

### Gestión de autenticación de Steam Guard

En la primera ejecución, debe ingresar el código Steam Guard manualmente. Después de eso, se recomienda configurar el secreto compartido para que el código se pueda generar automáticamente. Esto evita la intervención manual en cada versión.

SteamCMD guarda el token de inicio de sesión y puede reutilizarlo en ejecuciones posteriores. Aún debe estar atento a la caducidad del token, porque se requiere una nueva autenticación después de que caduque el token.

### Estructura del directorio de contenido

Asegúrese de que la estructura del directorio de contenido de Steam sea correcta:

```
steam-content/
├── linux-x64/     # Linux platform content
├── win-x64/       # Windows platform content
└── osx-universal/ # macOS universal binary content
```

Cada directorio debe contener los archivos completos de la aplicación para su plataforma correspondiente.

### Uso del modo de vista previa

El modo de vista previa no activa ninguna rama, por lo que es adecuado para pruebas y validación:

```yaml
if [ "$STEAM_PREVIEW_INPUT" = 'true' ]; then
  cmd+=(--preview)
fi
```

Esto le permite subir a Steam primero para su verificación y luego cambiar a la rama formal después de que todo esté verificado.

### Manejo y registro de errores

Los scripts incluyen registro y manejo de errores completos:
- Valide que exista la versión de GitHub.
- Verifique los archivos de metadatos requeridos.
- Asegúrese de que el contenido de la plataforma esté presente.
- Genere un informe resumido de GitHub Actions.

Esta información es muy valiosa tanto para la depuración como para la auditoría.

### Gestión de artefactos

El flujo de trabajo genera dos tipos de artefactos:
- `portable-steam-release-preparation-{tag}`: metadatos de preparación de lanzamiento
- `portable-steam-build-metadata-{tag}`: Metadatos de compilación de Steam

Estos artefactos se pueden utilizar para auditorías y depuraciones posteriores. Un período de retención de 30 días es un valor predeterminado práctico.

## Uso en el mundo real

En el proyecto HagiCode, este flujo de trabajo de lanzamiento automatizado ya se ha ejecutado con éxito en varias versiones. Todo el camino desde GitHub Release hasta la plataforma Steam está completamente automatizado y no requiere intervención manual.

Esto mejoró significativamente tanto la eficiencia como la confiabilidad de la liberación. En el pasado, publicar manualmente una versión llevaba más de 30 minutos. Ahora todo el proceso finaliza en tan solo unos minutos.

Más importante aún, el flujo de trabajo automatizado reduce la posibilidad de error humano. Cada lanzamiento sigue el mismo proceso estandarizado, lo que hace que los resultados sean más predecibles.
