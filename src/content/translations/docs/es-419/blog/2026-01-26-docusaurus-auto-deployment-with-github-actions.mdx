## Agregar soporte de implementación automática de páginas de GitHub a HagiCode

> Este proyecto originalmente tenía el nombre en código PCode y ahora pasó a llamarse oficialmente HagiCode. Este artículo documenta cómo introdujimos la implementación automatizada de sitios estáticos para que publicar contenido sea tan fácil como beber agua.

## Antecedentes / Introducción

Durante el desarrollo de HagiCode, nos topamos con un problema muy práctico: a medida que nuestra documentación y propuestas seguían creciendo, gestionar y presentar ese contenido de manera eficiente se volvió cada vez más urgente. Decidimos usar GitHub Pages para alojar nuestro sitio estático, pero crearlo e implementarlo manualmente era demasiado engorroso. Cada cambio requería una construcción local, un empaquetado y luego un envío manual al `gh-pages` rama. Esto no sólo era ineficaz, sino que también era fácil equivocarse.

Para resolver este problema, principalmente porque queríamos ahorrarnos algo de esfuerzo, necesitábamos un flujo de trabajo de implementación automatizado. Este artículo registra en detalle cómo agregamos soporte de implementación automática basado en GitHub Actions al proyecto HagiCode, para que podamos concentrarnos en escribir contenido y dejar que la automatización se encargue del resto.

## Análisis de objetivos

Antes de pasar a la implementación, primero debemos aclarar qué se supone que debe lograr esta tarea. Después de todo, afilar el hacha no retrasa el corte de leña.

### Requisitos básicos

1.  **Compilación automatizada**: activa automáticamente el flujo de trabajo de compilación cuando el código se envía al `main` rama.
2.  **Implementación automatizada**: después de una compilación exitosa, implemente automáticamente los archivos estáticos generados en GitHub Pages.
3.  **Coherencia del entorno**: asegúrese de que el entorno de CI coincida con el entorno de compilación local para evitar la clásica vergüenza de "funciona localmente, pero falla en línea".

### Elección técnica

Dado que HagiCode se basa en Docusaurus, un generador de sitios estáticos de React muy popular, podemos usar GitHub Actions para lograr este objetivo.

## Configurar el flujo de trabajo de acciones de GitHub

GitHub Actions es el servicio CI/CD proporcionado por GitHub. Al definir archivos de flujo de trabajo en formato YAML dentro del repositorio, podemos personalizar todo tipo de tareas de automatización.

### Crear el archivo de flujo de trabajo

Necesitamos crear un nuevo archivo de configuración en el `.github/workflows` directorio en la raíz del proyecto, por ejemplo `deploy.yml`. Si el directorio aún no existe, créelo primero.

La lógica central de este archivo de configuración es la siguiente:

1.  **Condición de activación**: Escuche `push` eventos en el `main` rama.
2.  **Entorno de ejecución**: la última versión de Ubuntu.
3.  **Pasos de construcción**:
    -   Mira el código
    -   Instalar Node.js
    -   Instalar dependencias (`npm install`)
    -   Construya los archivos estáticos (`npm run build`)
4.  **Paso de implementación**: utilice la versión oficial `action-gh-pages` acción para impulsar la salida de la compilación al `gh-pages` rama.

### Código de configuración clave

A continuación se muestra la plantilla de configuración que finalmente adoptamos:

```yaml
name: Deploy to GitHub Pages

# Trigger condition: run when code is pushed to the main branch
on:
  push:
    branches:
      - main
    # Add path filters if needed, for example only build when docs change
    # paths:
    #   - 'docs/**'
    #   - 'package.json'

# Set permissions, which is important for GitHub Pages deployment
permissions:
  contents: read
  pages: write
  id-token: write

# Concurrency control: cancel older builds for the same branch
concurrency:
  group: "pages"
  cancel-in-progress: false

jobs:
  build:
    runs-on: ubuntu-latest
    steps:
      - name: Checkout
        uses: actions/checkout@v4
        # Note: fetch-depth: 0 is required, otherwise the build version may be inaccurate
        with:
          fetch-depth: 0

      - name: Setup Node
        uses: actions/setup-node@v4
        with:
          node-version: 20 # Keep this consistent with the local development environment
          cache: 'npm'     # Enabling cache can speed up the build process

      - name: Install dependencies
        run: npm ci
        # Use npm ci instead of npm install because it is faster and stricter, which makes it better for CI

      - name: Build website
        run: npm run build
        env:
          # Configure environment variables here if your site build requires them
          # NODE_ENV: production
          # PUBLIC_URL: /your-repo-name
          
      - name: Upload artifact
        uses: actions/upload-pages-artifact@v3
        with:
          path: ./build # Default Docusaurus output directory

  deploy:
    environment:
      name: github-pages
      url: ${{ steps.deployment.outputs.page_url }}
    runs-on: ubuntu-latest
    needs: build
    steps:
      - name: Deploy to GitHub Pages
        id: deployment
        uses: actions/deploy-pages@v4
```

## Errores comunes durante la implementación

En la práctica, nos topamos con varios problemas. Los comparto aquí con la esperanza de que otros puedan evitarlos, o al menos preparar soluciones con anticipación.

### 1. Problemas de permisos de tokens de GitHub

Cuando configuramos las cosas por primera vez, la implementación seguía fallando con un 403 (Prohibido). Después de investigar mucho, encontramos que el valor predeterminado de GitHub `GITHUB_TOKEN` No tenía permiso para escribir en Pages.

**Solución**: En el repositorio `Settings` -> `Actions` -> `General` -> `Workflow permissions`, asegúrese de seleccionar **"Permisos de lectura y escritura"**.

### 2. Ruta de directorio de compilación incorrecta

Docusaurus coloca archivos estáticos creados en el `build` directorio por defecto. Sin embargo, algunos proyectos pueden configurarse de manera diferente. Por ejemplo, Crear aplicación React tiene como valor predeterminado `build`, mientras que Vite por defecto es `dist`. Si Actions informa que no puede encontrar los archivos de salida, verifique la ruta de salida en `docusaurus.config.js`.

### 3. Problemas con las rutas secundarias

Si su repositorio no es una página de inicio de usuario, lo que significa que no es `username.github.io`, sino una página de proyecto como `username.github.io/project-name`, entonces necesitas configurar `baseUrl`.

en `docusaurus.config.js`:

```javascript
module.exports = {
  // ...
  url: 'https://hagicode.com', // Your Hagicode URL
  baseUrl: '/', // Deploy from the root path
  // ...
};
```
Este detalle es fácil de pasar por alto. Si se configura incorrectamente, la página puede cargarse como una pantalla en blanco porque las rutas de los activos no se pueden resolver.

## Verificando el resultado

Después de configurar todo y enviar el código, podemos dirigirnos a la pestaña **Acciones** del repositorio de GitHub y ver el programa.

Verá un círculo amarillo mientras se ejecuta el flujo de trabajo y una vez que se vuelve verde, significa éxito. Si se vuelve rojo, haga clic e inspeccione los registros. En la mayoría de los casos, puedes rastrear el problema allí, generalmente un error tipográfico o una configuración de ruta incorrecta.

Una vez que la construcción sea exitosa, visite `https://<your-username>.github.io/<repository-name>/` y verá el sitio recién implementado.

## Resumen

Al introducir GitHub Actions, automatizamos con éxito la implementación del sitio de documentación de HagiCode. Esto no solo ahorra el tiempo que antes se dedicaba a operaciones manuales, sino que, lo que es más importante, estandariza el flujo de trabajo de lanzamiento. Ahora, no importa qué miembro del equipo actualice la documentación, una vez que el cambio se fusiona en `main`, el contenido más reciente estará disponible unos minutos más tarde.

**Beneficios principales**:
- **Mayor eficiencia**: Pasamos de "empaquetado y carga manual" a "código es igual a liberación".
- **Menos errores**: Eliminamos la posibilidad de error humano durante las operaciones manuales.
- **Mejor experiencia para desarrolladores**: los desarrolladores pueden centrarse más en la calidad del contenido en lugar de distraerse con tediosos pasos de implementación.

Aunque configurar CI/CD es un poco problemático al principio, especialmente con todos los permisos y detalles de la ruta, es una inversión única con importantes retornos a largo plazo. Recomiendo encarecidamente que cada proyecto de sitio estático adopte un flujo de trabajo automatizado similar.
