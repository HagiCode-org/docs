## De Docusaurus 3.x a Astro 5.x: una retrospectiva de la migración del sitio HagiCode

> Este artículo analiza nuestra migración completa del sitio web oficial de HagiCode de Docusaurus 3.x a Astro 5.x. Profundizaremos en cómo la arquitectura de las islas de Astro nos ayudó a resolver los cuellos de botella en el rendimiento y al mismo tiempo preservar nuestros activos de componentes React existentes, brindando mejoras tanto en la velocidad de construcción como en el rendimiento de carga.

## Antecedentes

En enero de 2026, realizamos un "trasplante de corazón" en el sitio oficial de HagiCode al migrar completamente su marco central de Docusaurus 3.x a Astro 5.x. Esta no fue una reescritura impulsiva, sino una decisión técnica cuidadosamente considerada.

Antes de la migración, nuestro sitio estaba funcionalmente completo, pero había comenzado a mostrar algunos "problemas de lujo" clásicos: artefactos de compilación inflados, cargas útiles de JavaScript excesivas y una velocidad de carga de página inferior a la ideal en páginas de documentación complejas. Como proyecto de asistente de codificación de IA, HagiCode necesita documentación frecuente y actualizaciones de funciones, por lo que la eficiencia de la compilación afecta directamente la velocidad de lanzamiento. Al mismo tiempo, queríamos que el sitio fuera más amigable para los motores de búsqueda (SEO) para que más desarrolladores pudieran descubrir el proyecto.

Para resolver estos problemas, tomamos una decisión audaz: reconstruir todo el sistema en Astro. El impacto de esa decisión puede ser incluso mayor de lo esperado. Entraré en los detalles en breve.

## Acerca de HagiCode

El enfoque de migración de sitios compartido en este artículo proviene de nuestra experiencia práctica en el [HagiCode](https://github.com/HagiCode-org/site) proyecto.

HagiCode es un asistente de codificación de IA centrado en mejorar la eficiencia del desarrollo. No solo nos preocupamos por iterar las funciones principales, sino también por la experiencia del desarrollador. Esta refactorización del sitio también tuvo como objetivo brindar a los usuarios la experiencia más rápida posible al navegar por nuestros documentos y el sitio web oficial.

## ¿Por qué abandonar el ecosistema maduro del Docusaurus?

Dentro del ecosistema React, Docusaurus ha sido durante mucho tiempo la "respuesta estándar" para los sitios de documentación. Funciona de inmediato, ofrece un rico ecosistema de complementos y tiene una comunidad activa. Pero a medida que HagiCode ganó más funciones, también sentimos sus limitaciones:

1.  **Cuellos de botella de rendimiento**: Docusaurus es fundamentalmente un React SPA (aplicación de una sola página). Incluso si solo escribe páginas estáticas, el cliente aún necesita cargar el tiempo de ejecución de React e hidratar la página, lo cual es innecesariamente pesado para páginas de documentos simples.
2.  **Tamaño de recurso grande**: incluso cuando una página contiene muy poco contenido, el tamaño del JS incluido permanece relativamente fijo. Esto no es ideal para usuarios de dispositivos móviles o condiciones de red deficientes.
3.  **Flexibilidad limitada**: aunque es extensible, queríamos tener más control de bajo nivel sobre el proceso de compilación.

Astro llegó exactamente en el momento adecuado para solucionar estos problemas. Introdujo una nueva "Arquitectura de islas": de forma predeterminada, Astro genera HTML estático sin JavaScript, y sólo los componentes que requieren interactividad se "activan" y cargan JS. Eso significa que la mayor parte de nuestro sitio se convierte en HTML puro y se carga extremadamente rápido.

## Estrategia de migración central: una transición arquitectónica fluida

La migración no fue sólo copiar y pegar. Requirió un cambio de mentalidad. Pasamos del modelo "todo React" de Docusaurus al modelo "Core + Islands" de Astro.

### 1. Reconstrucción del sistema de configuración

Primero, tuvimos que pasar de `docusaurus.config.ts` a `astro.config.mjs`. Esto no fue solo un cambio de nombre de archivo, sino una reescritura de la lógica de enrutamiento y construcción.

En Docusaurus, todo es un complemento. En Astro todo es una integración. Necesitábamos redefinir la ruta base del sitio, crear el modo de salida (estático frente a SSR) y la estrategia de optimización de activos.

**Antes de la migración:**
```typescript
// docusaurus.config.ts
export default {
  title: 'HagiCode',
  url: 'https://hagicode.com',
  baseUrl: '/',
  // ... more configuration
};
```

**Después de la migración:**
```javascript
// astro.config.mjs
import { defineConfig } from 'astro/config';
import react from '@astrojs/react';

export default defineConfig({
  integrations: [react()],
  site: 'https://hagicode.com',
  base: '/',
  // Optimization settings for static assets
  build: {
    inlineStylesheets: 'auto',
  },
});
```

### 2. Qué conservar y qué refactorizar en los componentes de React

Esta fue la parte más dolorosa de la migración. Nuestro sitio existente tenía muchos componentes de React, como pestañas, resaltado de código, botones de comentarios y más. Tirarlos sería un desperdicio, pero conservarlos todo haría que la carga útil de JavaScript fuera demasiado pesada.

HagiCode adoptó una estrategia de **hidratación progresiva**:
*   **Componentes estáticos puros**: para contenido de presentación, como encabezados, pies de página y documentación de texto sin formato, los reescribimos como componentes Astro (`.astro` archivos) y los renderizó directamente a HTML en el momento de la compilación.
*   **Islas interactivas**: Para los componentes que deben permanecer interactivos, como los conmutadores de temas, el cambio de pestañas y los botones de copia de bloques de código, mantuvimos la implementación de React y agregamos `client:load` o `client:visible` directivas.

Por ejemplo, nuestro componente de pestañas comúnmente utilizado en la documentación:

```jsx
// src/components/Tabs.jsx
import { useState } from 'react';
import './Tabs.css'; // Import styles

export default function Tabs({ items }) {
  const [activeIndex, setActiveIndex] = useState(0);
  // ... state logic
  return (
    <div className="tabs-wrapper">
      {/* Rendering logic */}
    </div>
  );
}
```

Cuando se usa en Markdown, le decimos explícitamente a Astro: "Este componente necesita JS".

```markdown
// src/content/docs/example.mdx
import Tabs from '../../components/Tabs.jsx';

<!-- Load JS only when the component enters the viewport -->
<Tabs client:visible items={...} />
```

De esta manera, los componentes interactivos fuera de la ventana gráfica no compiten por el ancho de banda, lo que mejora enormemente la velocidad de carga de la primera pantalla.

### 3. Adaptación del sistema de estilo: de módulos CSS a CSS con alcance

Docusaurus admite módulos CSS de forma predeterminada, mientras que Astro fomenta CSS con alcance a través del `<style>` etiqueta. La idea central detrás de ambos es el aislamiento de estilos, pero la sintaxis es diferente.

Durante la migración de HagiCode, convertimos los módulos CSS más complejos en estilos con alcance de Astro. En realidad esto resultó ser algo bueno, porque en `.astro` archivos los estilos y plantillas viven en el mismo archivo, lo que hace que el mantenimiento sea más intuitivo.

**Antes de refactorizar:**
```css
/* Tabs.module.css */
.wrapper { background: var(--ifm-background-color); }
```

**Después de la refactorización (Astro Scoped):**
```html
<!-- Tabs.astro -->
<div class="tabs-wrapper">
  <slot />
</div>

<style>
  .tabs-wrapper {
    /* Use CSS variables directly to adapt to the theme */
    background: var(--bg-color);
    padding: 1rem;
  }
</style>
```

Al mismo tiempo, unificamos el sistema global de variables CSS y utilizamos las capacidades ambientales de Astro para garantizar que el modo oscuro cambie sin problemas entre páginas.

## Errores que encontramos en la práctica y cómo los resolvimos

Durante la migración real de HagiCode, nos encontramos con bastantes problemas. A continuación te presentamos varios de los más típicos.

### 1. Puntos débiles variables de la ruta y el entorno

HagiCode admite la implementación de subrutas, como la implementación en un subdirectorio de páginas de GitHub. En Docusaurus, `baseUrl` se maneja automáticamente. En Astro, sin embargo, debemos tener más cuidado al manejar enlaces de imágenes y solicitudes de API.

Introdujimos un mecanismo de variables de entorno para gestionar esto de forma consistente:

```javascript
// Handle paths in the build script
const getBasePath = () => import.meta.env.VITE_SITE_BASE || '/';
```

Asegúrese de no codificar rutas que comiencen con `/` en tu código. En desarrollo versus producción, o después de configurar un `base` ruta, hacerlo puede causar 404 para los activos.

### 2. Compatibilidad de secuencias de comandos CommonJS

Nuestro sitio anterior tenía algunos scripts de Node.js utilizados para tareas como obtener automáticamente datos de métricas y actualizar el mapa del sitio, y estaban escritos en CommonJS (`require`). Astro y las herramientas de construcción modernas han adoptado completamente los módulos ES (`import`/`export`).

Si también tiene scripts similares, recuerde refactorizarlos todos en módulos ES. Esa es la dirección en la que se mueve el ecosistema, y cuanto antes hagas el cambio, menos problemas tendrás en el futuro.

```javascript
// Old way
const fs = require('fs');

// New way
import fs from 'fs';
```

### 3. No te olvides del SEO y los redireccionamientos

Los motores de búsqueda ya han indexado las antiguas páginas de Docusaurus de HagiCode. Si cambia directamente a Astro y la estructura de la URL cambia, puede terminar con una gran cantidad de 404 y una caída importante en la clasificación de búsqueda.

Configuramos reglas de redireccionamiento en Astro:

```javascript
// astro.config.mjs
export default defineConfig({
  redirects: {
    '/docs/old-path': '/docs/new-path',
    // Map old links to new links in bulk
  }
});
```

O puede manejar esto en la capa de configuración del servidor. Asegúrese de que los enlaces antiguos puedan redirigirse 301 a las nuevas direcciones, porque esto es fundamental para el SEO.

## Resumen

Para HagiCode, la migración de Docusaurus a Astro no fue solo una actualización del marco. También fue una implementación práctica de la filosofía de "primero el desempeño".

**Lo que ganamos:**
*   **Puntuaciones excepcionales de Lighthouse**: después de la migración, la puntuación de rendimiento del sitio HagiCode se acercó fácilmente a una puntuación perfecta.
*   **Velocidad de compilación más rápida**: las capacidades de compilación incremental de Astro reducen a la mitad el tiempo de publicación de las actualizaciones de la documentación.
*   **Flexibilidad preservada**: Con Islands Architecture, no sacrificamos ninguna característica interactiva y aún pudimos usar React cuando fuera necesario.

Si también mantiene un sitio orientado a la documentación y tiene problemas con el tamaño del paquete o la velocidad de carga, vale la pena probar Astro. Aunque el proceso de migración requiere algo de cirugía, como cambiar el nombre de PCode a HagiCode y mover los componentes uno por uno, la experiencia de usuario fluida que obtiene a cambio hace que valga la pena.

El sistema de compilación compartido en este artículo es el enfoque exacto que desarrollamos mediante prueba y error real mientras construíamos HagiCode. Si encuentra valioso este enfoque, eso dice algo sobre nuestra fortaleza de ingeniería, y probablemente también valga la pena analizar más de cerca el propio HagiCode.
