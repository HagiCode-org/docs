## De 120 segundos a 45 segundos: una guía práctica para optimizar el rendimiento de la compilación de Vite con subprocesos de trabajo

> Cuando se trabaja con grandes proyectos frontend, las compilaciones de producción pueden resultar tremendamente lentas. Este artículo comparte cómo utilizamos Node.js Worker Threads para reducir la etapa de ofuscación en una compilación de Vite de 120 segundos a 45 segundos, junto con los detalles de implementación y las lecciones aprendidas en el proyecto HagiCode.

## Antecedentes

En nuestra práctica de ingeniería frontend, los problemas de eficiencia de construcción se volvieron cada vez más prominentes a medida que crecía el proyecto. En particular, durante el proceso de construcción de producción, generalmente introducimos herramientas de ofuscación de JavaScript como `javascript-obfuscator` para proteger la lógica del código fuente. Este paso es necesario, pero también es computacionalmente costoso y depende mucho de la CPU.

Durante la etapa inicial de desarrollo de **HagiCode**, nos topamos con un cuello de botella de rendimiento muy complicado: los tiempos de compilación de producción se deterioraron rápidamente a medida que crecía el código base.

**Los puntos débiles específicos fueron**:
- Las tareas de ofuscación se ejecutaron en serie en un solo subproceso, maximizando un núcleo de CPU mientras los demás permanecían inactivos.
- El tiempo de construcción aumentó de los 30 segundos originales a 110-120 segundos.
- El ciclo de verificación de la compilación posterior al cambio se volvió extremadamente largo, lo que ralentizó seriamente la iteración del desarrollo.
- En el proceso de CI/CD, la etapa de construcción se convirtió en la parte que consumía más tiempo.

**¿Por qué HagiCode necesitaba esto?**
HagiCode es un asistente de código impulsado por IA cuya arquitectura frontal incluye lógica empresarial compleja y módulos de interacción con IA. Para garantizar la seguridad de nuestro código central, aplicamos una ofuscación de alta intensidad en las versiones de producción. Ante esperas de compilación cercanas a los dos minutos, decidimos llevar a cabo una optimización profunda del rendimiento del sistema de compilación.

## Análisis: encontrar el punto de ruptura en el cuello de botella del rendimiento

Antes de resolver el problema de rendimiento, primero necesitábamos aclarar nuestro pensamiento e identificar la mejor solución técnica.

### Decisión central: ¿Por qué elegir subprocesos trabajadores?

Hay tres formas principales de lograr cálculo paralelo en Node.js:
1. **child_process**: crea procesos secundarios independientes
2. **Web Workers**: utilizado principalmente en el lado del navegador
3. **worker_threads**: compatibilidad nativa con subprocesos múltiples en Node.js

Después de comparar las opciones, HagiCode finalmente eligió **Worker Threads** por las siguientes razones:
- **Gasto general de serialización cero**: los subprocesos de trabajo se ejecutan en el mismo proceso y pueden compartir memoria a través de `SharedArrayBuffer` o transferir la propiedad, evitando el alto costo de serialización de la comunicación entre procesos.
- **Soporte nativo**: integrado en Node.js 12+ sin necesidad de dependencias extra pesadas.
- **Contexto unificado**: la depuración y el registro son más convenientes que con los procesos secundarios.

### Granularidad de tareas: ¿Cómo deberían dividirse las tareas de ofuscación?

Es difícil paralelizar la ofuscación de un enorme archivo de paquete JS porque el código tiene dependencias, pero el resultado de la compilación de Vite se compone de múltiples **fragmentos**. Eso nos da un límite paralelo natural:

- **Independencia**: después del empaquetado de Vite, las dependencias entre diferentes fragmentos ya están desacopladas, por lo que se pueden procesar de forma segura en paralelo.
- **Granularidad adecuada**: los proyectos suelen tener entre 10 y 30 fragmentos, lo que constituye una escala excelente para la programación paralela.
- **Fácil integración**: el `generateBundle` El gancho en los complementos de Vite nos permite interceptar y procesar estos fragmentos antes de que se emitan los archivos.

### Diseño de Arquitectura

Diseñamos un sistema de procesamiento paralelo con cuatro componentes principales:

1. **Task Splitter**: itera sobre el objeto de paquete de Vite, filtra archivos que no necesitan ofuscación, como fragmentos de proveedores, y genera una cola de tareas.
2. **Administrador del grupo de trabajadores**: administra el ciclo de vida del trabajador y maneja la distribución, el reciclaje y el reintento de tareas en caso de falla.
3. **Progress Reporter**: los resultados generan progreso en tiempo real para reducir la ansiedad de espera.
4. **ObfuscationWorker**: el hilo de trabajo que realmente realiza la lógica de ofuscación.

## Solución: implementación práctica

Con base en el análisis anterior, comenzamos a implementar este sistema de ofuscación paralelo.

### 1. Configure el complemento Vite

Primero, integramos el complemento de ofuscación paralela en `vite.config.ts`. La configuración es sencilla. Solo necesita especificar la cantidad de trabajadores y las reglas de ofuscación.

```typescript
import { defineConfig } from 'vite'
import { parallelJavascriptObfuscator } from './buildTools/plugin'

export default defineConfig(({ mode }) => {
  const isProduction = mode === 'production'
  
  return {
    build: {
      rollupOptions: {
        ...(isProduction
          ? {
              plugins: [
                parallelJavascriptObfuscator({
                  enabled: true,
                  // Automatically adjust based on CPU core count; leave one core for the main thread
                  workerCount: 4, 
                  retryAttempts: 3,
                  fallbackToMainThread: true, // Automatically degrade to single-thread mode on failure
                  // Filter out vendor chunks; third-party libraries usually do not need obfuscation
                  isVendorChunk: (fileName: string) => fileName.includes('vendor-'),
                  obfuscationConfig: {
                    compact: true,
                    controlFlowFlattening: true,
                    deadCodeInjection: true,
                    disableConsoleOutput: true,
                    // ... more obfuscation options
                  },
                }),
              ],
            }
          : {}),
      },
    },
  }
})
```

### 2. Implementar la lógica del trabajador

Un Trabajador es la unidad que ejecuta tareas. Necesitamos definir claramente las estructuras de datos de entrada y salida.

**Nota**: aunque el código aquí es simple, existen varios errores a tener en cuenta, como verificar si `parentPort` es nulo y maneja los errores correctamente. En la implementación de HagiCode, encontramos que ciertos patrones de sintaxis especiales de ES6 podrían causar que el ofuscador fallara, por lo que agregamos `try-catch` protección.

```typescript
import { parentPort } from 'worker_threads'
import javascriptObfuscator from 'javascript-obfuscator'

export interface ObfuscationTask {
  chunkId: string
  code: string
  config: any
}

export interface ObfuscationResult {
  chunkId: string
  obfuscatedCode: string
  error?: string
}

// Listen for tasks sent from the main thread
if (parentPort) {
  parentPort.on('message', async (task: ObfuscationTask) => {
    try {
      // Perform obfuscation
      const obfuscated = javascriptObfuscator.obfuscate(task.code, task.config)
      const result: ObfuscationResult = {
        chunkId: task.chunkId,
        obfuscatedCode: obfuscated.getObfuscatedCode(),
      }
      // Send the result back to the main thread
      parentPort?.postMessage(result)
    } catch (error) {
      // Handle exceptions so one Worker crash does not block the whole build
      const result: ObfuscationResult = {
        chunkId: task.chunkId,
        obfuscatedCode: '',
        error: error instanceof Error ? error.message : 'Unknown error',
      }
      parentPort?.postMessage(result)
    }
  })
}
```

### 3. Administrador del grupo de trabajadores

Este es el núcleo de toda la solución. Necesitamos mantener un grupo de trabajadores de tamaño fijo y programar tareas utilizando una estrategia **FIFO (primero en entrar, primero en salir)**.

```typescript
import { Worker } from 'worker_threads'
import os from 'os'

export class WorkerPool {
  private workers: Worker[] = []
  private taskQueue: Array<{
    task: ObfuscationTask
    resolve: (result: ObfuscationResult) => void
    reject: (error: Error) => void
  }> = []
  
  constructor(options: WorkerPoolOptions = {}) {
    // Default to core count - 1 so the main thread still has some breathing room
    const workerCount = options.workerCount ?? Math.max(1, (os.cpus().length || 4) - 1)
    
    for (let i = 0; i < workerCount; i++) {
      this.createWorker()
    }
  }

  private createWorker() {
    const worker = new Worker('./worker.ts')
    
    worker.on('message', (result) => {
      // After one task finishes, take the next task from the queue
      const nextTask = this.taskQueue.shift()
      if (nextTask) {
        this.dispatchTask(worker, nextTask)
      } else {
        // If there are no pending tasks, mark the Worker as idle
        this.activeWorkers.delete(worker)
      }
    })
    
    this.workers.push(worker)
  }

  // Submit a task to the pool
  public runTask(task: ObfuscationTask): Promise<ObfuscationResult> {
    return new Promise((resolve, reject) => {
      const job = { task, resolve, reject }
      const idleWorker = this.workers.find(w => !this.activeWorkers.has(w))
      
      if (idleWorker) {
        this.dispatchTask(idleWorker, job)
      } else {
        this.taskQueue.push(job)
      }
    })
  }

  private dispatchTask(worker: Worker, job: any) {
    this.activeWorkers.set(worker, job.task)
    worker.postMessage(job.task)
  }
}
```

### 4. Informe de progreso

Esperar es doloroso, especialmente cuando no tienes idea de cuánto más llevará. Por eso agregamos un informe de progreso simple para brindar comentarios en tiempo real sobre el estado actual.

```typescript
export class ProgressReporter {
  private completed = 0
  private readonly total: number
  private readonly startTime: number

  constructor(total: number) {
    this.total = total
    this.startTime = Date.now()
  }

  increment(): void {
    this.completed++
    this.report()
  }

  private report(): void {
    const now = Date.now()
    const elapsed = now - this.startTime
    const percentage = (this.completed / this.total) * 100
    
    // Simple ETA estimate
    const avgTimePerChunk = elapsed / this.completed
    const remaining = (this.total - this.completed) * avgTimePerChunk

    console.log(
      `[Parallel Obfuscation] ${this.completed}/${this.total} chunks completed (${percentage.toFixed(1)}%) | ETA: ${(remaining / 1000).toFixed(1)}s`
    )
  }
}
```

## Práctica: resultados y lecciones aprendidas

Después de implementar esta solución, el rendimiento de compilación del proyecto HagiCode mejoró inmediatamente.

### Datos de referencia de rendimiento

Probamos en el siguiente entorno:
- CPU: Intel Core i7-12700K (12 núcleos / 20 subprocesos)
- RAM: 32 GB DDR4
- Nodo.js: v18.17.0
- Sistema operativo: Ubuntu 22.04

**Comparación de resultados**:
- **Subproceso único (antes de la optimización)**: 118 segundos
- **4 trabajadores**: 55 segundos (**53%** mejora)
- **8 trabajadores**: 48 segundos (**60 %** de mejora)
- **12 trabajadores**: 45 segundos (**62 %** de mejora)

Como puede ver, las ganancias no fueron lineales. Una vez que el recuento de trabajadores superó los 8, la mejora se hizo menor. Esto se vio limitado principalmente por la uniformidad de la distribución de tareas y los cuellos de botella en el ancho de banda de la memoria.

### Problemas comunes y soluciones

En el uso de HagiCode en el mundo real, también nos topamos con varios errores, así que aquí están como referencia:

**P1: ¿El tiempo de construcción no disminuyó mucho e incluso se volvió más lento?**
- **Razón**: la creación de trabajadores tiene su propia sobrecarga, o se configuraron demasiados trabajadores, lo que provoca cambios frecuentes de contexto.
- **Solución**: recomendamos configurar el recuento de trabajadores en `CPU core count - 1`. Compruebe también si algún fragmento es especialmente grande, por ejemplo `> 5MB`. Ese tipo de archivo "monstruoso" se convertirá en un cuello de botella, por lo que es posible que deba optimizar su estrategia de división de código.

**P2: ¿Los trabajadores ocasionalmente fallan y provocan fallas en la compilación?**
- **Motivo**: algunos patrones de sintaxis de código especiales pueden causar errores internos dentro del ofuscador.
- **Solución**: implementamos un **mecanismo de degradación automática**. Cuando un trabajador alcanza el umbral de falla, el complemento vuelve automáticamente al modo de subproceso único para garantizar que la compilación no se detenga. Al mismo tiempo, registra el nombre del archivo que causó el error para poder solucionarlo más tarde.

**P3: ¿El uso de memoria es demasiado alto (OOM)?**
- **Razón**: cada trabajador necesita su propio espacio de memoria para cargar el ofuscador y analizar el AST.
- **Solución**:
  - Reducir el número de Trabajadores.
  - Aumente el límite de memoria de Node.js: `NODE_OPTIONS="--max-old-space-size=4096" npm run build`.
  - Asegúrese de que los trabajadores no guarden referencias innecesarias a objetos grandes.

## Conclusión

Al introducir Node.js Worker Threads, redujimos con éxito el tiempo de construcción de producción del proyecto HagiCode de 120 segundos a alrededor de 45 segundos, mejorando enormemente la experiencia de desarrollo y la eficiencia de CI/CD.

El núcleo de esta solución es:
1. **Divida las tareas correctamente**: utilice fragmentos de Vite como unidad paralela.
2. **Controlar recursos**: utilice un grupo de trabajadores para evitar el agotamiento de los recursos.
3. **Diseño para tolerancia a fallos**: un mecanismo de degradación automática garantiza la estabilidad de la construcción.
