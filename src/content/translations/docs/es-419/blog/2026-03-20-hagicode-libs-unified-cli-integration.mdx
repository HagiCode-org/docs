## Hagicode.Libs: práctica de ingeniería para la integración unificada de múltiples CLI de asistente de codificación de IA

> Durante el desarrollo del proyecto HagiCode, necesitábamos integrar múltiples CLI de asistentes de codificación de IA al mismo tiempo, incluidos Claude Code, Codex y CodeBuddy. Cada CLI tiene diferentes interfaces, parámetros y formatos de salida, y el código de integración repetido hizo que el proyecto fuera cada vez más difícil de mantener. En este artículo, compartimos cómo construimos una capa de abstracción unificada con HagiCode.Libs para resolver este problema de ingeniería. También se podría decir que es simplemente una experiencia ganada con mucho esfuerzo y obtenida de los obstáculos que ya hemos topado.

## Antecedentes

El mercado de asistentes de codificación de IA está bastante activo ahora. Además de Claude Code, también existen Codex de OpenAI, CodeBuddy de Zhipu y más. Como proyecto de asistente de codificación de IA, HagiCode necesita integrar estas diferentes herramientas CLI en múltiples subproyectos, incluidos escritorio, backend y web.

Al principio, el problema era manejable. La integración de una CLI requirió solo unos cientos de líneas de código. Pero a medida que la cantidad de CLI que necesitábamos soportar siguió creciendo, las cosas empezaron a complicarse.

Cada CLI tiene su propio formato de argumentos de línea de comandos, diferentes requisitos de variables de entorno y una amplia variedad de formatos de salida. Algunos generan JSON, otros generan streaming de JSON y otros generan texto sin formato. Además de eso, existen problemas de compatibilidad multiplataforma. El descubrimiento de ejecutables y la gestión de procesos funcionan de manera muy diferente entre los sistemas Windows y Unix, por lo que la duplicación de código siguió aumentando. En verdad, fue solo un poco más Ctrl+C y Ctrl+V, pero el mantenimiento rápidamente se volvió doloroso.

La parte más frustrante fue que cada vez que queríamos agregar soporte para una nueva capacidad CLI, teníamos que cambiar el mismo código en varios proyectos. Es evidente que ese enfoque no es sostenible a largo plazo. El código también tiene mal carácter; duplíquelo demasiadas veces y comenzará a causar problemas.

## Acerca de HagiCode

El enfoque compartido en este artículo proviene de nuestra experiencia práctica en el [HagiCode](https://hagicode.com) proyecto. HagiCode es un proyecto de asistente de codificación de IA de código abierto que necesita mantener múltiples subproyectos al mismo tiempo, incluida una extensión VSCode frontend, servicios de IA backend y un cliente de escritorio multiplataforma. En cierto modo, fue exactamente este entorno complejo, multilingüe y multiplataforma lo que llevó al nacimiento de HagiCode.Libs. Se podría decir que nos vimos obligados a hacerlo, y que así sea.

## Análisis: encontrar puntos en común

Aunque cada una de estas CLI de asistente de codificación de IA tiene sus propias características, desde una perspectiva técnica comparten varios rasgos obvios:

**Patrones de interacción similares**: todos inician un proceso CLI, envían un mensaje, reciben respuestas en tiempo real, analizan mensajes y luego finalizan o continúan la sesión. Al fin y al cabo, todo el flujo sigue el mismo molde básico.

**Necesidades de configuración similares**: todos necesitan autenticación de clave API, configuración del directorio de trabajo, selección de modelo, control de permisos de herramientas y gestión de sesiones. Después de todo, todo el mundo se gana la vida con las API; las diferencias son principalmente una cuestión de sabor.

**Los mismos desafíos multiplataforma**: todos necesitan resolver la resolución de la ruta ejecutable (`claude` vs `claude.exe` vs `/usr/local/bin/claude`), inicio de procesos y manejo de variables de entorno, escape de comandos de shell y construcción de argumentos. El trabajo multiplataforma es doloroso sin importar cómo lo describas. Sólo las personas que han caído en las trampas entienden realmente la diferencia entre Windows y Unix.

Según este análisis, necesitábamos una capa de abstracción unificada que pudiera proporcionar una interfaz coherente, encapsular la lógica de descubrimiento de CLI multiplataforma, manejar el análisis de salida de streaming y admitir escenarios tanto de inyección de dependencia como sin DI. Es el tipo de problema que hace que te duela la cabeza con sólo pensarlo, pero aún así tienes que afrontarlo. Después de todo, es nuestro propio proyecto, así que tenemos que terminarlo incluso si tenemos que llorar para superarlo.

## Solución: HagiCode.Libs

Creamos **HagiCode.Libs**, un espacio de trabajo de biblioteca .NET 10 liviano lanzado bajo la licencia MIT y ahora publicado en [GitHub](https://github.com/HagiCode-org/Hagicode.Libs). Puede que no sea una obra maestra que conmueva al mundo, pero es realmente útil para resolver problemas reales.

### Estructura del proyecto

```
HagiCode.Libs/
├── src/
│   ├── HagiCode.Libs.Core/           # Core capabilities
│   │   ├── Discovery/                 # CLI executable discovery
│   │   ├── Process/                   # Cross-platform process management
│   │   ├── Transport/                 # Streaming message transport
│   │   └── Environment/               # Runtime environment resolution
│   ├── HagiCode.Libs.Providers/       # Provider implementations
│   │   ├── ClaudeCode/                # Claude Code provider
│   │   ├── Codex/                     # Codex provider
│   │   └── Codebuddy/                 # CodeBuddy provider
│   ├── HagiCode.Libs.ConsoleTesting/  # Testing framework
│   ├── HagiCode.Libs.ClaudeCode.Console/
│   ├── HagiCode.Libs.Codex.Console/
│   └── HagiCode.Libs.Codebuddy.Console/
└── tests/                             # xUnit tests
```

### Objetivos de diseño

Al diseñar HagiCode.Libs, seguimos algunos principios. Todos ellos provienen de lecciones aprendidas de la manera más difícil:

**Cero dependencias pesadas del marco**: no depende de ABP ni de ningún otro marco grande, lo que lo mantiene liviano. Hoy en día, cuantas menos dependencias tengas, menos dolores de cabeza tendrás. La mayoría de las personas ya han sido golpeadas por el infierno de la dependencia al menos una vez.

**Soporte multiplataforma**: soporte nativo para Windows, macOS y Linux, sin escribir código separado para diferentes plataformas. Una base de código que se ejecuta en todas partes es algo bastante bueno.

**Procesamiento de transmisión**: la salida CLI se maneja con transmisiones asincrónicas, lo que se adapta mucho mejor a los patrones de programación .NET modernos. Los tiempos cambian y async es el rey.

**Integración flexible**: admite escenarios de inyección de dependencia y al mismo tiempo permite la creación de instancias directas. Diferentes personas tienen diferentes preferencias, por lo que queríamos que fuera conveniente de cualquier manera.

## como usarlo

### Mediante inyección de dependencia

Si su proyecto ya utiliza inyección de dependencia, como ASP.NET Core o el host genérico, puede integrarlo directamente. Es una cosa pequeña, pero de buen comportamiento:

```csharp
using HagiCode.Libs.Providers;
using Microsoft.Extensions.DependencyInjection;

var services = new ServiceCollection();
services.AddHagiCodeLibs();

await using var provider = services.BuildServiceProvider();
var claude = provider.GetRequiredService<ICliProvider<ClaudeCodeOptions>>();

var options = new ClaudeCodeOptions
{
    ApiKey = "your-api-key",
    Model = "claude-sonnet-4-20250514"
};

await foreach (var message in claude.ExecuteAsync(options, "Hello, Claude!"))
{
    Console.WriteLine($"{message.Type}: {message.Content}");
}
```

### Creación de instancias directas

Si está escribiendo un script simple o trabajando en un escenario que no es DI, también funciona crear una instancia directamente. En pocas palabras, depende de tus preferencias personales:

```csharp
var claude = new ClaudeCodeProvider();
var options = new ClaudeCodeOptions
{
    ApiKey = "sk-ant-xxx",
    Model = "claude-sonnet-4-20250514"
};

await foreach (var message in claude.ExecuteAsync(options, "Help me write a quicksort"))
{
    // Handle messages
}
```

Ambos enfoques utilizan la misma implementación subyacente, por lo que puedes elegir el estilo de integración que mejor se adapte a tu proyecto. No existe una respuesta correcta universal en este mundo. Lo que más te convenga es la mejor opción. Puede parecer un cliché, pero es cierto.

## experiencia practica

### 1. Consolas de prueba dedicadas

Cada proveedor tiene su propio proyecto de consola de prueba dedicado, lo que facilita la validación de la integración de forma independiente. Las pruebas son una de esas cosas en las que, si vas a realizarlas, debes hacerlo correctamente:

```bash
# Claude Code tests
dotnet run --project src/HagiCode.Libs.ClaudeCode.Console -- --test-provider
dotnet run --project src/HagiCode.Libs.ClaudeCode.Console -- --test-all claude

# CodeBuddy tests
dotnet run --project src/HagiCode.Libs.Codebuddy.Console -- --test-provider codebuddy-cli

# Codex tests
dotnet run --project src/HagiCode.Libs.Codex.Console -- --test-provider codex-cli
```

Los escenarios de prueba cubren varios casos clave:
- **Ping**: verificación de estado para confirmar que la CLI está disponible
- **Mensaje simple**: prueba rápida básica
- **Mensaje complejo**: prueba de conversación de varios turnos
- **Restaurar/Reanudar sesión**: prueba de recuperación de sesión
- **Análisis de repositorio**: prueba de análisis de repositorio

Este diseño de consola de prueba independiente es especialmente útil durante la depuración porque nos permite identificar rápidamente si el problema está en la capa HagiCode.Libs o en la propia CLI. En realidad, la depuración consiste simplemente en encontrar dónde está el problema. Una vez que la dirección es correcta, ya estás a mitad de camino.

### 2. Validación de CI/CD multiplataforma

La compatibilidad multiplataforma es uno de los objetivos principales de HagiCode.Libs. Configuramos el flujo de trabajo de GitHub Actions `.github/workflows/cli-discovery-cross-platform.yml` para ejecutar una validación de descubrimiento de CLI real en `ubuntu-latest`, `macos-latest`, y `windows-latest`.

Esto garantiza que cada cambio de código no rompa la compatibilidad multiplataforma. Durante el desarrollo local, también puedes reproducirlo con los siguientes comandos. Después de todo, no se puede pedir a CI que asuma la culpa de todo. Su entorno local también debería poder ejecutarlo:

```bash
npm install --global @anthropic-ai/claude-code@2.1.79
HAGICODE_REAL_CLI_TESTS=1 dotnet test --filter "Category=RealCli"
```

### 3. Procesamiento de flujo de mensajes

HagiCode.Libs utiliza transmisiones asincrónicas para procesar la salida CLI. En comparación con los enfoques tradicionales de devolución de llamadas o basados ​​en eventos, esto se adapta mucho mejor al estilo de programación asincrónica del .NET moderno. Al final, así es simplemente como avanza la tecnología, le guste o no a alguien:

```csharp
public async IAsyncEnumerable<CliMessage> ExecuteAsync(
    TOptions options,
    string prompt,
    [EnumeratorCancellation] CancellationToken cancellationToken = default)
{
    // Start the CLI process
    // Parse streaming JSON output
    // Yield the CliMessage sequence
}
```

Los tipos de mensajes incluyen:
- `user`: mensaje de usuario
- `assistant`: respuesta del asistente
- `tool_use`: invocación de herramienta
- `result`: fin de sesión

Este diseño permite a las personas que llaman manejar la salida de streaming de manera flexible, ya sea para visualización en tiempo real, posprocesamiento almacenado en búfer o reenvío a otros servicios. ¿Por qué preocuparse si el cielo está soleado o nublado? Lo que importa es que una vez que la idea se abre, puedes usarla como quieras.

### 4. Exploración del repositorio Git

El módulo HagiCode.Libs.Exploration proporciona descubrimiento de repositorios Git y verificación de estado, lo cual es especialmente útil en escenarios de análisis de repositorios. Esta característica también nació por necesidad, porque HagiCode necesita analizar repositorios:

```csharp
// Discover Git repositories
var repositories = await GitRepositoryDiscovery.DiscoverAsync("/path/to/search");

// Get repository information
var info = await GitRepository.GetInfoAsync(repoPath);
Console.WriteLine($"Branch: {info.Branch}, Remote: {info.RemoteUrl}");
Console.WriteLine($"Has uncommitted changes: {info.HasUncommittedChanges}");
```

Las capacidades de análisis de código de HagiCode utilizan este módulo para identificar la estructura del proyecto y el estado de Git. Es un buen ejemplo de cómo aprovechar al máximo lo que construimos.

## Cosas a tener en cuenta

Según nuestra práctica en el proyecto HagiCode, hay varios puntos que merecen especial atención. Todos ellos son problemas reales que deben abordarse con cuidado:

**Seguridad de claves API**: no codifique claves API en su código. Utilice variables de entorno o gestión de configuración en su lugar. HagiCode.Libs admite pasar la configuración a través de objetos Opciones, lo que facilita la integración con diferentes fuentes de configuración. Cuando se trata de seguridad, nunca se es demasiado cuidadoso.

**Fijación de versión CLI**: en CI/CD, fijamos versiones específicas, como `@anthropic-ai/claude-code@2.1.79`, para reducir la incertidumbre causada por la deriva de la versión. También es una buena idea utilizar versiones fijas en el desarrollo local. El control de versiones puede ser doloroso. Si no fija las versiones, el problema le enseñará una lección muy rápidamente.

**Categorización de pruebas**: las pruebas predeterminadas utilizan proveedores falsos para mantenerlas deterministas y rápidas, mientras que las pruebas CLI reales deben habilitarse explícitamente. Esto proporciona a la CI una respuesta rápida y, al mismo tiempo, permite la validación en el entorno real cuando sea necesario. Lograr ese equilibrio nunca es fácil. La velocidad y la estabilidad siempre requieren concesiones.

**Gestión de sesiones**: diferentes CLI tienen diferentes mecanismos de recuperación de sesiones. Código Claude utiliza el `.claude/` directorio para almacenar sesiones, mientras que Codex y CodeBuddy tienen cada uno sus propios enfoques. Al usarlos, asegúrese de consultar su documentación respectiva y comprender los detalles de sus mecanismos de persistencia de sesión. No hay nada de malo en entenderlo claramente.
