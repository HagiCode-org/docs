## Adiós infierno de scripts: por qué elegimos C# para un sistema de compilación moderno

> Una inmersión profunda en cómo el proyecto HagiCode utiliza Nuke para crear un proceso de compilación automatizado, multiplataforma y altamente extensible, que aborda por completo los puntos débiles de mantenimiento de los scripts de compilación tradicionales.

## Antecedentes

A lo largo del largo camino del desarrollo de software, la palabra "construir" tiende a inspirar tanto amor como frustración. Amor, porque con un clic el código se convierte en producto, y ese es uno de los momentos más satisfactorios en la programación. Frustración, porque mantener un montón de scripts de compilación desordenados puede ser una pesadilla.

En muchos proyectos, estamos acostumbrados a escribir scripts en Python o confiar en archivos de configuración XML. Imagínese el miedo a ser gobernado por `<property>` bloques. Pero a medida que crece la complejidad del proyecto, especialmente en un proyecto como HagiCode que abarca trabajo frontend y backend, múltiples plataformas y múltiples lenguajes, los enfoques de construcción tradicionales comienzan a mostrar sus límites. Lógica de script dispersa, falta de verificación de tipos, soporte IDE débil... cada uno de estos problemas se convierte en un pequeño obstáculo que hace tropezar al equipo de vez en cuando.

Para resolver esos problemas, el proyecto HagiCode introdujo **Nuke**, un sistema de compilación moderno basado en C#. No es sólo una herramienta. Es una forma diferente de pensar en los flujos de trabajo de construcción. Hoy veremos por qué lo elegimos y cómo ha mejorado drásticamente nuestra experiencia de desarrollo.

## Análisis central: ¿Por qué Nuke?

Quizás se pregunte: "Ya existen tantos sistemas de compilación, como Make, Gradle o incluso scripts de Shell simples. ¿Por qué esforzarse por utilizar uno basado en C#?"

Ésa es una pregunta justa. El atractivo principal de Nuke es que trae las características del lenguaje de programación que ya conocemos bien al mundo de los scripts de compilación.

### 1. Modularizar los flujos de trabajo de construcción: el arte de los objetivos

Nuke tiene una filosofía de diseño muy clara: **todo es un objetivo**.

En los scripts tradicionales, es fácil terminar con cientos de líneas de lógica de ejecución lineal entrelazadas. En Nuke, dividimos el proceso de construcción en independientes. `Target`s. Cada objetivo hace exactamente una cosa, por ejemplo:

- `Clean`: limpia el directorio de salida
- `Restore`: restaurar paquetes de dependencia
- `Compile`: compila el código
- `Test`: ejecutar pruebas unitarias

Este diseño se alinea muy bien con el principio de responsabilidad única. Al igual que los bloques de construcción, estos objetivos se pueden combinar libremente. Más importante aún, Nuke nos permite definir dependencias entre objetivos. Por ejemplo, si desea ejecutar `Test`, el sistema comprobará automáticamente si `Compile` ya ha sido ejecutado. si quieres `Compile`, entonces `Restore` naturalmente es lo primero.

Este gráfico de dependencia hace que la lógica sea mucho más clara y mejora significativamente la eficiencia de la ejecución, porque Nuke puede analizar automáticamente la ruta de ejecución óptima.

### 2. Seguridad tipográfica: adiós a la pesadilla de los errores tipográficos

Cualquiera que haya escrito scripts de compilación en Python probablemente se haya encontrado con esta situación incómoda: el script se ejecuta durante cinco minutos y luego falla porque `Confi.guration` estaba mal escrito o porque se pasó una cadena donde se esperaba un parámetro numérico.

La mayor ventaja de escribir scripts de compilación en C# es la **seguridad de tipos**. Eso significa:

- **Comprobaciones en tiempo de compilación**: el IDE puede indicarle qué está mal mientras escribe en lugar de hacerle esperar hasta el tiempo de ejecución.
- **Refactorización segura**: si desea cambiar el nombre de una variable o método, el IDE puede manejarlo en un solo paso sin obligarlo a realizar una búsqueda y reemplazo global riesgosos.
- **Finalización inteligente**: el potente IntelliSense puede completar el código automáticamente, por lo que no necesita seguir hojeando documentación para recordar API oscuras.

### 3. Soporte multiplataforma: una experiencia de construcción consistente

En el pasado escribimos `.bat` en Windows y `.sh` en Linux, y luego tal vez agregue un script de Python para cerrar la brecha. Ahora, dondequiera que se pueda ejecutar .NET Core, o .NET 5+ hoy en día, Nuke también se puede ejecutar.

Eso significa que los miembros del equipo pueden usar Windows, Linux o macOS, y ya sea que prefieran Visual Studio, VS Code o Rider, todos ejecutan exactamente la misma lógica. Esto reduce en gran medida los problemas causados ​​por diferencias de entorno, especialmente el clásico problema de "funciona en mi máquina".

### 4. Gestión de parámetros y configuración.

Nuke proporciona un mecanismo de análisis de parámetros muy elegante. No es necesario analizar manualmente `string[] args`. Simplemente define una propiedad y anotala con `[Parameter]`y Nuke maneja automáticamente el mapeo desde los argumentos de la línea de comandos y los archivos de configuración.

Por ejemplo, podemos definir la configuración de compilación muy fácilmente:

```csharp
[Parameter("Configuration to build - Default is 'Debug'")]
readonly Configuration BuildConfiguration = IsLocalBuild ? Configuration.Debug : Configuration.Release;

Target Compile => _ => _
    .DependsOn(Restore)
    .Executes(() =>
    {
        // Use BuildConfiguration here; it is type-safe
        DotNetBuild(s => s
            .SetConfiguration(BuildConfiguration)
            .SetProjectFile(SolutionFile));
    });
```

Este estilo es a la vez intuitivo y resistente a los errores.

## Guía práctica: cómo ponerlo en práctica en un proyecto

La teoría por sí sola no nos lleva a ninguna parte. Veamos cómo aplicamos realmente este enfoque en el proyecto HagiCode.

### 1. Planificar la estructura del proyecto

No queríamos que los scripts de compilación saturaran la raíz del proyecto y tampoco queríamos una estructura de directorios tan profunda que se pareciera a ciertos proyectos Java. Así que colocamos todos los archivos de compilación relacionados con Nuke en un solo `nukeBuild/` directorio.

Los beneficios son sencillos:

- La raíz del proyecto se mantiene limpia.
- La lógica de construcción se mantiene coherente y es más fácil de gestionar.
- Los nuevos miembros del equipo pueden ver inmediatamente: "Aquí es donde reside la lógica relacionada con la construcción".

### 2. Diseñar una cadena de dependencia objetivo clara

Al diseñar Targets, seguimos un principio: **atomicidad más flujo de dependencia**.

Cada objetivo debe ser lo suficientemente pequeño como para hacer exactamente una cosa. Por ejemplo, `Clean` Sólo debe retirar las limas y no debe manipular silenciosamente el embalaje lateral.

Un flujo de dependencia recomendado se parece más o menos a esto:

`Clean` -> `Restore` -> `Compile` -> `Test` -> `Pack`

Por supuesto, esto no es absoluto. Por ejemplo, si solo desea ejecutar pruebas y no desea empaquetar nada, Nuke le permite ejecutar `nuke Test` directamente, y automáticamente manejará los requisitos requeridos. `Restore` y `Compile` pasos primero.

### 3. Manejo y registro de errores sólidos

¿Qué es lo más frustrante de los scripts de compilación? Mensajes de error poco claros. Una compilación fallida que registra sólo "Error: 1" es suficiente para volver loco a cualquiera.

En Nuke, debido a que podemos usar directamente el manejo de excepciones de C#, podemos capturar e informar errores con mucha mayor precisión.

```csharp
Target Publish => _ => _
    .DependsOn(Test)
    .Executes(() =>
    {
        try 
        {
            // Try publishing to NuGet
            DotNetNuGetPush(s => s
                .SetTargetPath(ArtifactPath)
                .SetSource("https://api.nuget.org/v3/index.json")
                .SetApiKey(ApiKey));
        }
        catch (Exception ex)
        {
            Log.Error($"Publishing failed. Team, please check whether the key is correct: {ex.Message}");
            throw; // Make sure the build process exits with a non-zero exit code
        }
    });
```

### 4. Utilice pruebas para proteger la calidad

Los scripts de compilación también son código, lo que significa que también necesitan pruebas. Nuke nos permite escribir pruebas para flujos de trabajo de compilación de modo que cuando cambiemos la lógica de compilación, no rompamos accidentalmente los procesos de lanzamiento existentes. Esto es especialmente importante en los canales de integración continua.

## Resumen

Al presentar Nuke, el proceso de construcción de HagiCode se volvió más fluido que nunca. Esto no fue sólo un reemplazo de herramienta. Fue una mejora en la mentalidad de ingeniería.

**¿Qué ganamos?**

- **Mantenibilidad**: configuración como código, lógica más clara y una ruta de incorporación más rápida para nuevos contribuyentes.
- **Estabilidad**: la escritura fuerte elimina más del 90% de los errores de bajo nivel.
- **Consistencia**: una experiencia multiplataforma unificada que elimina la deriva del entorno.

Si escribir guiones de compilación solía ser como "sentir el camino a través de la oscuridad", entonces usar Nuke se siente como "caminar de noche con las luces encendidas". Si está cansado de mantener scripts difíciles de depurar, considere trasladar su lógica de compilación al mundo de C#. Quizás descubras que las construcciones también pueden ser elegantes.
