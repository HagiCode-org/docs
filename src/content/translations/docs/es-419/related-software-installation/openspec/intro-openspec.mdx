# Introducción a OpenSpec

OpenSpec es una herramienta de desarrollo orientada a especificaciones que ayuda a los equipos a gestionar el proceso de desarrollo de software mediante propuestas y especificaciones estructuradas. Este artículo ofrece una introducción completa a los conceptos centrales, los flujos de trabajo y las formas de uso de OpenSpec.

## ¿Qué es OpenSpec?

OpenSpec es una herramienta de línea de comandos utilizada para gestionar propuestas (Proposals), especificaciones (Specifications) y cambios (Changes) en el proceso de desarrollo de software. Estandariza el flujo de trabajo de desarrollo y garantiza que cada función pase por un diseño y revisión adecuados desde la idea hasta la implementación.

### Principios fundamentales

OpenSpec se basa en los siguientes principios fundamentales:

- **Especificaciones primero**: Defina especificaciones claras antes de escribir código
- **Colaboración estructurada**: Fomente la colaboración del equipo mediante formatos de documentación estandarizados
- **Rastreabilidad**: Registre todo el proceso desde la propuesta hasta la implementación
- **Aseguramiento de la calidad**: Garantice que el código cumpla con la intención del diseño mediante la validación de especificaciones

### Funciones principales

OpenSpec proporciona las siguientes funciones principales:

1. **Gestión de propuestas**: Cree, revise y haga seguimiento de propuestas de funciones
2. **Definición de especificaciones**: Redacte y mantenga documentos de especificaciones técnicas
3. **Seguimiento de cambios**: Registre todo el historial de cambios revisados
4. **Validación de especificaciones**: Asegure que los cambios en el código cumplan las especificaciones predefinidas
5. **Colaboración del equipo**: Admite procesos de desarrollo y revisión con múltiples participantes

## Flujo de trabajo de OpenSpec

El flujo de trabajo típico de OpenSpec incluye las siguientes etapas:

### 1. Crear una propuesta (Proposal)

Cuando hay una nueva función o un requisito de cambio importante, primero cree una propuesta:

```bash
openspec proposal create
```

La propuesta incluye:
- **Descripción del problema**: El problema o requisito que debe resolverse
- **Solución**: Resumen de la solución propuesta
- **Alcance del impacto**: Impacto en el sistema existente
- **Alternativas**: Otras soluciones consideradas

### 2. Redactar la especificación (Specification)

Después de aprobar la propuesta, redacte una especificación técnica detallada:

```bash
openspec spec create
```

La especificación incluye:
- **Diseño técnico**: Plan detallado de implementación técnica
- **Definición de interfaces**: Diseño de interfaces como API, estructuras de datos, etc.
- **Detalles de implementación**: Lógica clave de implementación y algoritmos
- **Plan de pruebas**: Cómo verificar la corrección de la implementación

### 3. Crear un cambio (Change)

Al comenzar la implementación, cree un cambio para hacer seguimiento del proceso de desarrollo:

```bash
openspec change create
```

El cambio incluye:
- **Especificación asociada**: La especificación que implementa este cambio
- **Tareas de implementación**: Lista de tareas de desarrollo concretas
- **Seguimiento del progreso**: Estado de finalización de las tareas
- **Cambios de código**: Las modificaciones reales del código

### 4. Validación y archivo

Al terminar el desarrollo, valide si el cambio cumple con la especificación y luego archívelo:

```bash
openspec change validate
openspec change archive
```

## Estructura de directorios de OpenSpec

OpenSpec crea un directorio `openspec/` en la raíz del proyecto con la siguiente estructura:

```
openspec/
├── proposals/           # Directorio de propuestas
│   ├── PROPOSAL-001.md
│   └── PROPOSAL-002.md
├── specs/              # Directorio de especificaciones
│   ├── SPEC-001.md
│   └── SPEC-002.md
├── changes/            # Directorio de cambios
│   ├── feature-name/
│   │   ├── proposal.md
│   │   ├── design.md
│   │   ├── specs.md
│   │   └── tasks.md
│   └── another-feature/
└── archived/           # Cambios archivados
    └── feature-name/
```

### Descripción de directorios

- **proposals/**: Almacena todos los documentos de propuestas, cada propuesta en un archivo Markdown independiente
- **specs/**: Almacena documentos de especificaciones técnicas correspondientes a las propuestas
- **changes/**: Almacena los cambios en curso, cada cambio en un subdirectorio
- **archived/**: Almacena cambios completados y archivados

## Comandos principales

### Comandos relacionados con propuestas

```bash
# Crear una nueva propuesta
openspec proposal create

# Listar todas las propuestas
openspec proposal list

# Ver detalles de una propuesta
openspec proposal view <proposal-id>

# Aprobar una propuesta
openspec proposal approve <proposal-id>
```

### Comandos relacionados con especificaciones

```bash
# Crear una nueva especificación
openspec spec create

# Listar todas las especificaciones
openspec spec list

# Ver detalles de una especificación
openspec spec view <spec-id>

# Actualizar una especificación
openspec spec update <spec-id>
```

### Comandos relacionados con cambios

```bash
# Crear un nuevo cambio
openspec change create

# Listar todos los cambios
openspec change list

# Ver detalles de un cambio
openspec change view <change-name>

# Agregar una tarea
openspec change add-task <change-name>

# Validar un cambio
openspec change validate <change-name>

# Archivar un cambio
openspec change archive <change-name>
```

### Comandos generales

```bash
# Ver la versión
openspec --version

# Ver la ayuda
openspec --help

# Ver la ayuda de un comando específico
openspec <command> --help
```

## Uso de OpenSpec en el proyecto HagiCode

### Integración del proyecto

En el proyecto HagiCode, OpenSpec está integrado en el flujo de trabajo de desarrollo:

1. **Verificación de inicialización**: El asistente de inicio verificará si OpenSpec está instalado
2. **Desarrollo impulsado por propuestas**: Todas las nuevas funciones deben pasar por el proceso de propuestas de OpenSpec
3. **Revisión de código**: Los cambios deben cumplir las especificaciones de OpenSpec para ser fusionados
4. **Sincronización de documentación**: La documentación de OpenSpec se mantiene sincronizada con la implementación del código

### Requisitos de versión

El proyecto HagiCode requiere el uso de la versión **OpenSpec >=1.0.0 <2.0.0**.

Comando de instalación:

```bash
npm install -g @fission-ai/openspec@1
```

### Ejemplo de flujo de trabajo

A continuación se muestra un ejemplo completo de uso de OpenSpec en el proyecto HagiCode:

#### Paso 1: Crear propuesta

```bash
cd /path/to/hagicode-project
openspec proposal create
```

En el aviso interactivo, ingrese:
- Título de la propuesta: Agregar funcionalidad de autenticación de usuarios
- Descripción del problema: El sistema necesita inicio de sesión de usuarios y gestión de permisos
- Solución: Implementar un sistema de autenticación basado en JWT

#### Paso 2: Redactar la especificación

Después de aprobar la propuesta, cree una especificación detallada:

```bash
openspec spec create
```

Defina en la especificación:
- Diseño del flujo de autenticación
- Definición de la interfaz API
- Estructura de tablas de la base de datos
- Consideraciones de seguridad

#### Paso 3: Crear un cambio

Comience la implementación:

```bash
openspec change create
```

Ingrese el nombre del cambio: `add-user-authentication`

#### Paso 4: Agregar tareas

```bash
openspec change add-task add-user-authentication
```

Agregue tareas concretas:
- Implementar la interfaz de registro de usuarios
- Implementar la interfaz de inicio de sesión
- Implementar middleware JWT
- Escribir pruebas unitarias

#### Paso 5: Implementación y validación

Después de completar el código, valide el cambio:

```bash
openspec change validate add-user-authentication
```

#### Paso 6: Archivo

Después de la validación exitosa, archive el cambio:

```bash
openspec change archive add-user-authentication
```

## Mejores prácticas

### Redacción de propuestas

1. **Declaración clara del problema**: Defina claramente el problema que se debe resolver, evite descripciones vagas
2. **Considere alternativas**: Enumere y compare diferentes soluciones
3. **Evalúe el impacto**: Analice el impacto y riesgos para el sistema existente
4. **Manténgalo conciso**: Las propuestas deben ser breves y claras; deje los detalles para la especificación

### Redacción de especificaciones

1. **Detallado pero no redundante**: Proporcione suficientes detalles, pero evite el sobre-diseño
2. **Use diagramas**: Use diagramas de flujo, diagramas de arquitectura, etc., para ayudar en la explicación
3. **Defina interfaces**: Defina claramente todas las interfaces públicas y las estructuras de datos
4. **Considere casos límite**: Enumere condiciones excepcionales y manejo de errores

### Gestión de cambios

1. **Iteraciones pequeñas**: Divida grandes funcionalidades en varios cambios pequeños
2. **Actualice oportunamente**: Mantenga sincronizados el estado de tareas y la documentación
3. **Revisión de código**: Realice revisiones de código al completar un cambio
4. **Archive a tiempo**: Archive los cambios completados puntualmente y mantenga el directorio de trabajo limpio

### Colaboración del equipo

1. **Revisiones periódicas**: Revise periódicamente propuestas y especificaciones para asegurar consenso
2. **Documentación primero**: Complete la documentación antes de comenzar a codificar
3. **Mantenga la comunicación**: Registre discusiones y decisiones en las propuestas y especificaciones
4. **Control de versiones**: Incluya la documentación OpenSpec en el control de versiones

## Escenarios de uso comunes

### Escenario 1: Agregar una nueva función

1. Cree una propuesta que describa los requisitos de la nueva función
2. El equipo discute y aprueba la propuesta
3. Redacte una especificación técnica detallada
4. Cree un cambio e implemente la función
5. Valide y archive el cambio

### Escenario 2: Refactorización de código existente

1. Cree una propuesta que explique las razones y objetivos de la refactorización
2. Defina el plan y los pasos de la refactorización en la especificación
3. Cree un cambio y refactorice gradualmente
4. Asegure la cobertura de pruebas y valide los resultados de la refactorización
5. Archive el cambio

### Escenario 3: Corrección de errores

Para errores simples:
- Se pueden corregir directamente sin solicitar todo el proceso OpenSpec

Para errores complejos:
1. Cree una propuesta para analizar la causa raíz del error
2. Defina la solución de la corrección en la especificación
3. Cree un cambio e implemente la corrección
4. Agregue pruebas para evitar regresiones
5. Archive el cambio

### Escenario 4: Gestión de deuda técnica

1. Cree una propuesta que enumere los elementos de deuda técnica
2. Defina un plan de mejora en la especificación
3. Cree un cambio para pagar gradualmente la deuda técnica
4. Realice un seguimiento del progreso y el efecto
5. Archive las mejoras completadas

## Integración con otras herramientas

### Integración con Git

OpenSpec se integra sin problemas con Git:

```bash
# La documentación OpenSpec debe incluirse en el control de versiones
git add openspec/
git commit -m "Add OpenSpec proposal for feature X"
```

### Integración con CI/CD

Valide las especificaciones OpenSpec en la canalización CI/CD:

```yaml
# .github/workflows/openspec.yml
name: OpenSpec Validation
on: [pull_request]
jobs:
  validate:
    runs-on: ubuntu-latest
    steps:
      - uses: actions/checkout@v2
      - name: Install OpenSpec
        run: npm install -g @fission-ai/openspec@1
      - name: Validate changes
        run: openspec change validate
```

### Integración con IDE

Se recomienda usar un editor que admita Markdown para editar documentos OpenSpec:
- VS Code + complemento Markdown
- JetBrains IDEs + soporte Markdown
- Vim/Neovim + complemento Markdown

## Resolución de problemas

### Problemas comunes

#### Problema: Comando OpenSpec no encontrado

**Solución**:
1. Confirme que OpenSpec esté instalado: `npm list -g @fission-ai/openspec`
2. Verifique que la variable de entorno PATH incluya el directorio bin global de npm
3. Reinstale: `npm install -g @fission-ai/openspec@1`

#### Problema: No se puede crear una propuesta o un cambio

**Solución**:
1. Confirme que el directorio actual es la raíz del proyecto
2. Verifique si existe un directorio `openspec/`; si no, créelo
3. Confirme que tiene permisos de escritura

#### Problema: La validación falló

**Solución**:
1. Verifique que la documentación en el directorio de cambios esté completa
2. Confirme que todas las tareas necesarias estén completadas
3. Consulte información detallada del error: `openspec change validate <change-name> --verbose`

#### Problema: El archivado falló

**Solución**:
1. Asegúrese de que el cambio haya pasado la validación
2. Verifique si existe el directorio `openspec/archived/`
3. Confirme que no haya cambios de código sin confirmar

## Uso avanzado

### Plantillas personalizadas

OpenSpec admite plantillas personalizadas para propuestas y especificaciones:

```bash
# Cree un directorio de plantillas en la raíz del proyecto
mkdir -p .openspec/templates

# Cree una plantilla personalizada para propuestas
cat > .openspec/templates/proposal.md << 'EOF'
# Proposal: {{title}}

## Problem Statement
{{problem}}

## Proposed Solution
{{solution}}

## Custom Section
Your custom content here
EOF
```

### Archivo de configuración

Cree un archivo de configuración `.openspec.json` en la raíz del proyecto:

```json
{
  "version": "1.0",
  "templates": {
    "proposal": ".openspec/templates/proposal.md",
    "spec": ".openspec/templates/spec.md"
  },
  "validation": {
    "requireTests": true,
    "requireDocs": true
  },
  "hooks": {
    "pre-archive": "npm test"
  }
}
```

### Hooks

OpenSpec admite la ejecución de scripts personalizados antes y después de operaciones específicas:

```json
{
  "hooks": {
    "pre-create": "scripts/check-permissions.sh",
    "post-archive": "scripts/notify-team.sh"
  }
}
```
