> **HagiCode เป็นผลิตภัณฑ์ที่รวมเครื่องมือเข้ารหัส AI ระบบตอบรับแบบ gamified และพื้นที่ทำงานการพัฒนาเต็มรูปแบบไว้ในแพลตฟอร์มเดียว**

## เหตุใด HagiCode จึงดูไม่เหมือนเครื่องมือเข้ารหัส AI แบบดั้งเดิม

เครื่องมือเข้ารหัส AI แบบดั้งเดิมมักมุ่งเน้นไปที่การสร้าง HagiCode ให้ความสำคัญกับการดริฟท์น้อยลง จัดส่งได้อย่างน่าเชื่อถือ และสามารถตรวจสอบได้ในภายหลัง

นั่นหมายความว่าการออกแบบนั้นโน้มตัวไปสู่ขั้นตอนการพัฒนาซอฟต์แวร์จริง แทนที่จะเป็นการโต้ตอบคำถามและคำตอบแบบครั้งเดียว:

- ทำความเข้าใจกับ Repository ก่อนเปลี่ยน Code
- ชี้แจงเป้าหมายก่อนเริ่มดำเนินการ
- กำหนดขอบเขตก่อนที่ AI จะดำเนินการ
- ไม่เพียงแต่รักษาผลลัพธ์เท่านั้น แต่ยังรวมถึงกระบวนการและการใช้เหตุผลด้วย

นั่นเป็นรากฐานสำหรับอัตลักษณ์ทั้งสามของ HagiCode ที่ตามมา มันเป็นเครื่องมือเข้ารหัส AI พื้นที่ทำงานที่มีเกม และแพลตฟอร์มที่นำความสามารถในการพัฒนาที่หลากหลายมารวมกัน

## 1. HagiCode เป็นเครื่องมือเข้ารหัส AI

หากคุณดูเฉพาะเลเยอร์ "การเข้ารหัส AI" แสดงว่า HagiCode ไม่ได้พยายามทำให้ AI เขียนโค้ดที่กะพริบมากขึ้น กำลังพยายามทำให้ AI เขียนได้อย่างน่าเชื่อถือมากขึ้น

### 1. จัดระเบียบงานก่อนที่จะสร้างโค้ด

HagiCode มีเวิร์กโฟลว์ **OpenSpec** ในตัว สำหรับสิ่งที่ซับซ้อนกว่านี้เล็กน้อย AI จะไม่ข้ามไปที่การแก้ไขไฟล์โดยตรง ขั้นแรกจะเปลี่ยนคำขอเป็นข้อเสนอ งาน ขอบเขตผลกระทบ และขั้นตอนการตรวจสอบ

ประเด็นนั้นสำคัญ เครื่องมือเข้ารหัส AI จำนวนมากรู้สึกเสี่ยงไม่ใช่เพราะพวกเขาไม่สามารถสร้างโค้ดได้ แต่เพราะพวกเขาเริ่มเปลี่ยนแปลงสิ่งต่าง ๆ เร็วเกินไปเมื่อบริบทไม่สมบูรณ์ HagiCode พยายามย้อนกลับว่า:

- ชี้แจงปัญหาก่อน
- ยืนยันว่าโมดูลใดจะได้รับผลกระทบ
- แบ่งงานและเกณฑ์การยอมรับ
- แล้วจึงนำไปปฏิบัติ

ผลลัพธ์โดยตรงคือ AI มีโอกาสน้อยที่จะทำการแก้ไขแบบสุ่มตามสัญชาตญาณในโครงการที่ซับซ้อน พูดให้แตกต่างออกไป HagiCode ไม่ได้ไล่ตามเส้นทางที่สั้นที่สุด มันกำลังไล่ตามเส้นทางที่เชื่อถือได้มากขึ้น

### 2. เน้นความเข้าใจในระดับโครงการ ไม่ใช่แค่การทำงานปัจจุบันให้เสร็จสิ้น

IDE จำนวนมากสามารถแก้ไขไฟล์หลายไฟล์ได้แล้ว และบาง IDE ยังสามารถเปลี่ยนแปลงไดเร็กทอรีหลายรายการในเซสชันเดียวได้อีกด้วย ดังนั้นข้อดีของ HagiCode จึงไม่สามารถสรุปได้ง่ายๆ อีกต่อไปว่า "ไม่ใช่การเติมข้อความอัตโนมัติแบบไฟล์เดียว"

สิ่งที่ฉันต้องการเน้นแทนคือ HagiCode มุ่งเป้าไปที่ **มุมมองทั้งโครงการ**

ไม่สนใจเพียงว่า "ไฟล์ใดที่ต้องเปลี่ยนสำหรับงานนี้" นอกจากนี้ยังใส่ใจกับคำถามระดับสูงกว่าด้วย:

- ปัญหาที่โครงการโดยรวมกำลังแก้ไขคืออะไร
- ที่เก็บข้อมูลปัจจุบันเกี่ยวข้องกับที่เก็บข้อมูลอื่นอย่างไร
- ไม่ว่าการเปลี่ยนแปลงนี้จะกระทบกับส่วนหน้า แบ็กเอนด์ เอกสาร การนำไปใช้งาน หรือสคริปต์ด้วย
- ก่อนหน้านี้มีการตัดสินใจอะไรคล้าย ๆ กัน และเพราะเหตุใด
- ข้อเสนอ ความมุ่งมั่น และความรู้ของวันนี้ควรถูกนำมาใช้ซ้ำในภายหลังอย่างไร

กล่าวอีกนัยหนึ่ง HagiCode ไม่ใช่แค่พยายามทำงานหนึ่งงานให้คุณสำเร็จเท่านั้น กำลังพยายามดึง AI เข้าสู่มุมมองของการมีส่วนร่วมในโครงการในระยะยาว

จากมุมมองดังกล่าว งานเดียวเป็นเพียงพื้นผิวที่มองเห็นได้ สิ่งที่สำคัญกว่านั้นคือความสามารถต่อไปนี้สามารถเชื่อมต่อได้อย่างเป็นธรรมชาติ:

- การสลับและการประสานงานในหลายโครงการ
- สร้างความเข้าใจร่วมกันในที่เก็บข้อมูลหลายแห่ง
- การอนุรักษ์ข้อเสนอทางประวัติศาสตร์ ความมุ่งมั่น และความรู้เมื่อเวลาผ่านไป
- เปลี่ยนการสนทนาส่วนบุคคลให้เป็นบริบทของโครงการที่ยั่งยืน

นั่นคือเหตุผลที่ฉันออกแบบ HagiCode ให้เป็นพื้นที่ทำงานแทนที่จะเป็นหน้าต่างแชทธรรมดา ฉันต้องการให้ AI ไม่เห็นคำขอที่แยกจากกัน แต่ดูว่าโครงการทั้งหมดดำเนินไปในทิศทางใด

![มุมมองบอร์ดเซสชัน HagiCode แสดงคอลัมน์รอดำเนินการ อยู่ระหว่างดำเนินการ และเก็บถาวรแล้ว](../img/screenshots/shared/kanban-view/original.png)

จากมุมนี้ HagiCode ให้ความรู้สึกเหมือน "AI ที่คิดจากมุมมองของโปรเจ็กต์ทั้งหมด" มากกว่า "AI ที่ช่วยให้คุณแก้ไขชั่วคราวได้สำเร็จ"

### 3. รองรับ Agent CLI กระแสหลักหลายรายการ และแยก CLI ออกจากโมเดลอย่างชัดเจน

ช่วงการสนับสนุนที่ใช้งานอยู่ในปัจจุบันของ HagiCode ครอบคลุม Agent CLI หลักหลายรายการ รวมถึง:

- โคเด็กซ์
- คล็อด โค้ด
- นักบิน GitHub
- โอเพ่นโค้ด
- เฮอร์มีส
- QoderCLI
- คิโระ
- คิมิ
- ราศีเมถุน
- ดีพเอเจนท์
- รหัสบัดดี้

มีประเด็นสำคัญประการหนึ่งที่ฉันต้องการทำให้ชัดเจนที่นี่: **CLI และแบบจำลองไม่ได้เชื่อมโยงกันอย่างแน่นหนา**

ผลิตภัณฑ์จำนวนมากถือว่า "CLI ใดที่คุณใช้" และ "การสมัครสมาชิกรุ่นใดที่คุณใช้" เป็นการตัดสินใจเดียวกัน HagiCode ไม่ต้องการทำเช่นนั้น

### 4. OmniRoute แยกเลเยอร์โมเดลออกจาก CLI และทำให้การกำหนดเส้นทางมีความยืดหยุ่นมากขึ้น

HagiCode ผสานรวม **OmniRoute** เพื่อให้การเข้าถึงโมเดลกลายเป็นเลเยอร์โครงสร้างพื้นฐานของตัวเอง ด้วยวิธีนี้ CLI จะจัดการรูปแบบการโต้ตอบที่คุณต้องการ ในขณะที่โมเดลและการสมัครสมาชิกสามารถเลือกได้ผ่านเลเยอร์การกำหนดเส้นทางแบบรวม

คุณค่านั้นตรงไปตรงมา:

- คุณสามารถใช้ CLI ที่คุณชอบอยู่แล้วต่อไปได้
- คุณไม่ถูกบังคับให้สมัครรับข้อมูลแบบจำลองที่ CLI เกิดขึ้นเป็นค่าเริ่มต้น
- คุณสามารถจัดการการเลือกโมเดล แค็ตตาล็อกโมเดล และการเข้าถึงจุดสิ้นสุดได้ในเลเยอร์เดียว
- CLI หลายรายการสามารถใช้กลยุทธ์การกำหนดเส้นทางโมเดลเดียวกันซ้ำได้

กล่าวอีกนัยหนึ่ง แม้ว่าคุณต้องการใช้ **Claude Code** เป็น CLI คุณยังคงสามารถเชื่อมต่อกับแหล่งที่มาของโมเดลและการสมัครสมาชิกอื่น ๆ ผ่าน OmniRoute ได้ ตัวอย่างเช่น หากคุณต้องการใช้ความจุการสมัครรับข้อมูล **GitHub Copilot** แทนการเชื่อมโยง CLI เข้ากับเส้นทางการสมัครรับข้อมูลเริ่มต้น ซึ่งสามารถทำงานใน HagiCode ได้

สิ่งที่ฉันต้องการนั้นเรียบง่าย: คุณควรเลือก CLI เพราะคุณชอบความรู้สึกในการใช้งาน และเลือกรุ่นหรือการสมัครสมาชิกเพราะคุณเชื่อถือในเรื่องต้นทุน ความสามารถ และความพร้อมใช้งาน สิ่งเหล่านั้นไม่ควรถูกบังคับให้เลือกแบบรวมกลุ่มเดียว

## 2. HagiCode เป็นแพลตฟอร์มการพัฒนา AI ที่ครอบคลุม

หากส่วนแรกตอบว่า "สามารถจัดการโค้ดได้หรือไม่" ส่วนนี้จะตอบคำถามอื่น: **เหตุใดจึงรู้สึกเหมือนเป็น IDE และในบางแง่ก็เหมือนแพลตฟอร์มที่สมบูรณ์มากกว่า IDE แบบดั้งเดิม**

คำตอบก็คือ HagiCode ไม่ได้หยุดอยู่ที่การแชท และไม่ได้หยุดอยู่ที่ข้อเสนอเช่นกัน โดยดึงความสามารถที่ปกติจะกระจายไปตามเครื่องมือต่างๆ มารวมกัน และเปลี่ยนให้เป็นระบบต่อเนื่องระบบเดียว

### 1. MonoSpecs หยุดการพัฒนาข้ามพื้นที่เก็บข้อมูลจากการกลายเป็นงานปะติดปะต่อ

สำหรับทีมจริง ข้อกำหนดมักไม่อยู่ในที่เก็บข้อมูลเพียงแห่งเดียว ส่วนหน้า แบ็กเอนด์ เอกสารประกอบ สคริปต์ และการกำหนดค่าการใช้งานอาจจำเป็นต้องเปลี่ยนแปลงร่วมกัน

HagiCode แนะนำ **MonoSpecs** เพื่อนำการทำงานร่วมกันข้ามพื้นที่เก็บข้อมูลประเภทนี้กลับมาภายใต้มุมมองเดียว ในโปรเจ็กต์เดียว คุณสามารถรักษารายการพื้นที่เก็บข้อมูล ขอบเขตข้อเสนอ และกลยุทธ์การเก็บถาวรได้ คุณยังสามารถปล่อยให้ AI เข้าใจได้ชัดเจนยิ่งขึ้นว่าการเปลี่ยนแปลงข้ามขอบเขตใดจริงๆ

![ภาพรวมสถานะที่เก็บหลาย MonoSpecs แสดงสาขาและเปลี่ยนสถานะในที่เก็บหลายแห่ง](../img/screenshots/monospecs/hover-project-show-repository-status/original.png)

สำหรับผู้ใช้พื้นที่เก็บข้อมูลเดี่ยว นี่อาจไม่ใช่ความสามารถแรกที่พวกเขาสัมผัส แต่เมื่อคุณเริ่มจัดการกับการประสานงานส่วนหน้า-แบ็กเอนด์ การจัดเก็บเอกสารให้สอดคล้องกับผลิตภัณฑ์ หรือการบำรุงรักษาโปรเจ็กต์ย่อยหลายรายการ คุณค่าของมันก็จะชัดเจน

### 2. ระบบทักษะช่วยให้แพลตฟอร์มเติบโตต่อไป

ผลิตภัณฑ์ AI จำนวนมากขยายขอบเขตไปในทางคร่าวๆ: ไม่ว่าคุณจะรอฟีเจอร์อย่างเป็นทางการ หรือคุณทำให้ผู้ใช้ต้องปรับแต่งเทอร์มินัลด้วยตัวเอง HagiCode เปลี่ยน **ทักษะ** ให้เป็นโมดูลผลิตภัณฑ์อย่างเป็นทางการแทน

ภายใน HagiCode คุณสามารถ:

- ดูทักษะที่ติดตั้งไว้แล้วในเครื่อง
- ค้นหาแคตตาล็อกทักษะ
- รับคำแนะนำทักษะตามโครงการปัจจุบัน
- ตรวจสอบรายละเอียดทักษะ คำสั่งการติดตั้ง และสถานะความน่าเชื่อถือ
- อัปเดตทักษะท้องถิ่นเป็นชุด

![แกลเลอรีทักษะ HagiCode แสดงผลการค้นหา ตัวกรองแหล่งที่มา และจำนวนการติดตั้ง](../img/screenshots/shared/skills-gallary/original.png)

![หน้าทักษะท้องถิ่น HagiCode แสดงรายการทักษะที่ติดตั้งและการปฏิบัติงานเป็นชุด](../img/screenshots/shared/local-skills-view/original.png)

นั่นหมายความว่า HagiCode ไม่ใช่ผลิตภัณฑ์ที่ปิดสนิท เป็นเหมือนเกราะที่สามารถรับทักษะ ความสามารถ และขั้นตอนการทำงานใหม่ๆ ได้มากขึ้น

### 3. ระบบ Vault ช่วยให้ฐานความรู้ไม่กระจัดกระจายไปทุกที่

คุณสามารถนึกถึง **ห้องนิรภัย** ว่าเป็นชั้นพื้นที่จัดเก็บความรู้ของ HagiCode

รองรับการนำวัสดุประเภทต่าง ๆ มาสู่แพลตฟอร์ม ได้แก่ :

- ที่เก็บอ้างอิงรหัส
- โฟลเดอร์ธรรมดา
- ห้องนิรภัยออบซิเดียน
- ไดเร็กทอรีที่จัดการโดยระบบ

ด้วยวิธีนี้ บันทึกการวิเคราะห์ โค้ดอ้างอิง และบันทึกการออกแบบที่รวบรวมไว้ในโปรเจ็กต์เดียวจะไม่ติดอยู่ภายในเซสชันเดียว สามารถอ้างอิงได้อีกครั้ง จัดระเบียบเพิ่มเติม และนำมาใช้ใหม่เป็นบริบทในการทำงานในอนาคต

สำหรับหลายๆ ทีม สิ่งนี้สำคัญมาก AI มีคุณค่าอย่างแท้จริง ไม่ใช่เพราะ "ตอบครั้งเดียว" แต่เพราะสามารถทำงานต่อจากองค์ความรู้ที่จัดไว้แล้ว

![พื้นที่ทำงานของห้องนิรภัยแสดงห้องนิรภัยของผู้ใช้ ห้องนิรภัยของระบบ และการดำเนินการ เช่น การเปิดโค้ดเซิร์ฟเวอร์](../img/screenshots/shared/vault-view/original.png)

### 4. AI Compose Commit ขยาย "การเข้ารหัสที่เสร็จสิ้นแล้ว" ไปสู่ "การเขียน Commit ที่ชัดเจน"

สำหรับหลายทีม ปัญหาที่แท้จริงไม่ได้อยู่ที่การเขียนโค้ด แต่เป็นขั้นตอนสุดท้าย: โค้ดเสร็จสิ้นแล้ว แต่ไม่มีใครอยากเขียนข้อความ Commit อย่างระมัดระวัง

HagiCode มี **AI Compose Commit** ซึ่งนำการสร้างข้อความคอมมิตมาสู่เวิร์กโฟลว์เช่นกัน

- คุณไม่จำเป็นต้องจำการเปลี่ยนแปลงทุกรายการทีละบรรทัด
- คุณไม่จำเป็นต้องแก้ไขคำอธิบายการคอมมิตแบบเร่งรีบในนาทีสุดท้าย
- คุณสามารถปล่อยให้ AI จัดระเบียบข้อความยืนยันที่ชัดเจนยิ่งขึ้นตามความแตกต่างที่แท้จริงได้
- ในเวิร์กโฟลว์ Turbo Engine คุณยังสามารถปรับแต่ง AI ได้อีกด้วย `Co-Authored-By` ลายเซ็นต์และปล่อยให้การกำหนดค่าระดับพื้นที่เก็บข้อมูลแทนที่ค่าเริ่มต้นส่วนกลางของคุณ

![จุดเริ่มต้น AI Compose Commit ภายใน HagiCode](../img/screenshots/ai-compose-commit/trigger-ai-compose-commit-button/original.png)

คุณค่าของมันไม่ใช่แค่ประหยัดเวลาเพียงไม่กี่วินาทีเท่านั้น ในที่สุด "ความมุ่งมั่น" ก็หยุดแยกออกจากบริบทที่เหลือในที่สุด สำหรับทีม นั่นยังหมายถึงการคอมมิตที่ AI สร้างขึ้นสามารถใช้ชื่อบอท โดเมนอีเมลบริษัท และรูปแบบพื้นที่เก็บข้อมูลที่พวกเขาใช้อยู่แล้ว แทนที่จะถูกล็อคด้วยลายเซ็นคงที่เพียงลายเซ็นเดียว

### 5. การรวมเซิร์ฟเวอร์โค้ดทำให้การแก้ไขทั้งภายในและระยะไกลราบรื่นยิ่งขึ้น

HagiCode ยังรวมการแก้ไขบนเบราว์เซอร์ผ่าน **code-server** ไม่ว่าโปรเจ็กต์ของคุณจะอยู่ในเครื่อง บนเซิร์ฟเวอร์ ในคอนเทนเนอร์ หรือในรันไทม์ระยะไกล คุณสามารถเปิดโปรเจ็กต์หรือห้องนิรภัยได้ง่ายขึ้นและข้ามไปที่การแก้ไขโดยตรง

นั่นทำให้ HagiCode รู้สึกเหมือนเป็นแพลตฟอร์มการพัฒนาที่แท้จริง แทนที่จะเป็นเพียงส่วนหน้าที่วิเคราะห์โค้ด หลายครั้งที่ AI ได้ติดตามปัญหาไปยังไฟล์เฉพาะแล้ว หากคุณยังต้องกระโดดกลับไปใช้เครื่องมืออื่นและย้ายทุกอย่างด้วยตัวเอง ขั้นตอนการทำงานจะสูญเสียแรงผลักดัน การรวมเซิร์ฟเวอร์โค้ดช่วยแก้ปัญหาดังกล่าว

![สรุปรันไทม์ของเซิร์ฟเวอร์โค้ดที่แสดงการเปิดตัวล่าสุดและจุดเข้าใช้งานที่เปิดใหม่อีกครั้ง](../img/screenshots/shared/code-server-status/original.png)

### 6. ถือว่าคุณสมบัติด้านความสะดวกสบายเป็นเหมือนความสามารถที่แท้จริง ไม่ใช่สิ่งเหลือทิ้ง

นอกเหนือจากข้อเสนอ การดำเนินการ ทักษะ และการจัดการความรู้แล้ว HagiCode ยังมีคุณสมบัติบางประการที่ส่งผลต่อประสบการณ์ในแต่ละวันอย่างแท้จริง:

- บูรณาการ GitHub
- การรู้จำเสียงพูด
- การแจ้งเตือนความชุ่มชื้น
- การกำหนดธีมและอินเทอร์เฟซส่วนบุคคล
- จุดเข้าการรายงานและสถิติ

สิ่งเหล่านี้อาจดูเหมือนเป็น "คุณสมบัติเล็กๆ" แต่พวกเขาตัดสินใจว่าแพลตฟอร์มเป็นสิ่งที่ผู้คนต้องการให้เปิดอยู่ตลอดเวลาหรือไม่ HagiCode ไม่ได้ซ่อนไว้ที่ขอบ มันพยายามทำให้มองเห็นได้ สมบูรณ์ และกำหนดค่าชิ้นส่วนของผลิตภัณฑ์ได้

![หน้าการตั้งค่าทั่วไปที่แสดงการ์ดสำหรับภาษา การแสดงเซสชั่น และการแจ้งเตือนงานที่ต้องใช้เวลานาน](../img/screenshots/shared/settings-view/original.png)

## 3. HagiCode เป็นเกม

การเล่นเกมภายใน HagiCode ไม่ได้มีไว้เพื่อการตกแต่ง มีไว้เพื่อให้การใช้งานแพลตฟอร์มการพัฒนา AI ในระยะยาวรู้สึกตอบสนองมากขึ้น มีจังหวะมากขึ้น และยึดติดได้ง่ายขึ้น

### 1. คุณสามารถดูความคืบหน้าของคุณแทนที่จะเห็นเพียงบันทึกการสนทนา

ใน HagiCode การกระทำหลายอย่างจะกลายเป็นการตอบรับความคืบหน้าที่ชัดเจน การสร้างเซสชัน การส่งข้อความ การดำเนินการตามแผน การเปลี่ยนโปรเจ็กต์ และการส่งคำอธิบายประกอบจะไม่หายไปจากการดำเนินการเพียงครั้งเดียวอีกต่อไป โดยสะสมเป็น **ความสำเร็จรายวัน** ความคืบหน้าหลักชัย และบันทึกความสำเร็จ

จุดประสงค์ของการออกแบบนี้ไม่ใช่แค่ "ความสนุก" เท่านั้น มันเป็นการง่ายกว่าที่จะรู้สึกถึงสิ่งที่คุณก้าวไปข้างหน้าในหนึ่งวัน สำหรับนักพัฒนาระยะยาวหลายๆ คน ส่วนที่เหนื่อยล้าไม่ใช่ภาระงานในตัวมันเอง มันคือการขาดการตอบรับ HagiCode พยายามเติมเต็มช่องว่างนั้น

![ห้องโถงแห่งความสำเร็จแสดงความคืบหน้ารายวัน ความสำเร็จระดับโลก และการ์ดเมตริกเหตุการณ์สำคัญ](../img/screenshots/shared/achivement-view/original.png)

### 2. ไม่ได้หยุดอยู่ที่ความสำเร็จ นอกจากนี้ยังให้รายงานประจำวันแก่คุณอีกด้วย

นอกเหนือจากความสำเร็จแล้ว HagiCode ยังใช้รายงานรายวันเพื่อแสดงสิ่งที่คุณทำสำเร็จเมื่อวานนี้ คะแนนที่ได้มาจากไหน และความก้าวหน้าของคุณเป็นอย่างไร

นั่นหมายความว่าแพลตฟอร์มไม่เพียงแค่บันทึกสิ่งที่คุณทำเท่านั้น โดยจะจัดระเบียบการกระทำเหล่านั้นใหม่ให้เป็นพื้นผิวการทบทวนตามจังหวะจริง คุณสามารถบอกได้ง่ายขึ้นว่าคุณถูกบล็อกเกี่ยวกับความคืบหน้าของเซสชัน การใช้เครื่องมือ การเรียกใช้โค้ด หรือเพียงแค่เวลาที่ใช้งานและความต่อเนื่องของงาน

### 3. เปลี่ยนประสิทธิภาพการทำงานให้เป็นผลตอบรับที่มองเห็นได้ด้วยเช่นกัน

ผลิตภัณฑ์จำนวนมากบอกคุณว่า "AI ทำให้คุณมีประสิทธิผลมากขึ้น" แต่ไม่สามารถอธิบายได้ว่ามีประสิทธิผลมากขึ้นเพียงใด HagiCode อยากจะแสดงสิ่งนั้นด้วยข้อมูลที่มองเห็นได้

ในรายงานประสิทธิภาพการทำงานเหล่านี้ คุณสามารถดูระยะเวลารันไทม์ เวลาที่ใช้ AI การเพิ่มประสิทธิภาพ และการกระจายการทำงานพร้อมกันได้ มันไม่ใช่ตำนานของ AI กำลังพยายามเปลี่ยน "ประสิทธิภาพการทำงาน" จากสโลแกนเป็นการตอบรับที่เป็นรูปธรรม

### 4. มันยังเปลี่ยนการใช้งานโทเค็นให้เป็นสิ่งที่คุณสามารถสัมผัสได้แบบเรียลไทม์

หากคุณเป็นผู้ใช้งานหนัก คุณค่าของการออกแบบนี้จะเห็นได้ชัดเจน ในหลายกรณีปัญหาด้านต้นทุนและประสิทธิภาพของ AI จะไม่เปิดเผยในช่วงปลายเดือน จะปรากฏขึ้นในขณะที่เซสชั่นกำลังดำเนินการอยู่

HagiCode จะแสดงโทเค็นอินพุต โทเค็นเอาต์พุต จำนวนโทเค็นทั้งหมด และระดับปริมาณงานในผลิตภัณฑ์โดยตรง ซึ่งช่วยให้คุณทราบได้ทันทีว่าเซสชันมีน้ำหนักมากเพียงใด โมเดลปัจจุบันมีภาระงานสูงหรือไม่ และการสนทนามีมากเกินไปหรือไม่
