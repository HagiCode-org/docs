## ภาพรวม

Monospecs คือแนวทางการจัดการหลายพื้นที่เก็บข้อมูลของ HagiCode ติดตามและจัดการพื้นที่เก็บข้อมูลย่อยผ่าน `monospecs.yaml`และใช้งานได้กับโปรเจ็กต์ใดๆ ที่ต้องการที่เดียวเพื่อประสานงานที่เก็บข้อมูลอิสระหลายแห่ง

### แนวคิดหลัก

**ไฟล์กำหนดค่า**

`monospecs.yaml` เป็นแกนหลักของโซลูชันการจัดการทั้งหมด โดยกำหนด:
- รายชื่อและที่ตั้งของที่เก็บย่อย
- Git URL สำหรับแต่ละพื้นที่เก็บข้อมูล
- ชื่อที่แสดงและไอคอน
- ลักษณะการทำงานบูรณาการกับ OpenSpec

**ความแตกต่างจากคลังข้อมูลทั่วไป**

| คุณสมบัติ | พื้นที่เก็บข้อมูลปกติ | โครงการที่มีการจัดการ Monospecs |
|----------|-------------------|---------------------------|
| **ตำแหน่ง OpenSpec** | ไดเรกทอรีรากของ `openspec/` | ไดเร็กทอรีรากของที่เก็บหลัก `openspec/` |
| **เปลี่ยนขอบเขต** | จำกัด เฉพาะพื้นที่เก็บข้อมูลปัจจุบัน | สามารถเกี่ยวข้องกับที่เก็บข้อมูลย่อยหลายรายการ |
| **ความบริสุทธิ์ของพื้นที่เก็บข้อมูลย่อย** | สเปคผสมกับรหัส | ข้อมูลจำเพาะที่สร้างขึ้นในพื้นที่เก็บข้อมูลหลัก พื้นที่เก็บข้อมูลย่อยนั้นสะอาดกว่า |
| **เก็บถาวรอัตโนมัติคอมมิต** | ต้องการคอมมิตข้อมูลจำเพาะแบบแมนนวล | `commit_when_archive: true` ส่งมอบข้อกำหนดให้กับพื้นที่เก็บข้อมูลหลักโดยอัตโนมัติ |

### เหตุใดจึงต้องใช้ Monospecs

- **การจัดการแบบรวมศูนย์**: จัดการที่เก็บข้อมูลย่อยทั้งหมดในไฟล์การกำหนดค่าเดียว
- **การสนับสนุนอัตโนมัติ**: จัดเตรียมข้อมูลเมตาที่เก็บข้อมูลสำหรับสคริปต์และเครื่องมือ
- **ความเป็นมิตรต่อ AI**: การกำหนดค่าที่มีโครงสร้างช่วยให้ AI Agent เข้าใจความสัมพันธ์ของพื้นที่เก็บข้อมูล
- **การบูรณาการ OpenSpec**: การบูรณาการอย่างราบรื่นกับเวิร์กโฟลว์การจัดการการเปลี่ยนแปลง
- **พื้นที่เก็บข้อมูลย่อยที่สะอาดกว่า**: ข้อมูลจำเพาะจะถูกเก็บไว้ใน repo หลัก พื้นที่เก็บข้อมูลย่อยจะมีเฉพาะรหัสจริงเท่านั้น

### สถานการณ์ที่เกี่ยวข้อง

ใช้ Monospecs ก่อนในสถานการณ์เช่นนี้:

**1. การพัฒนาการทำงานร่วมกันหลายพื้นที่เก็บข้อมูล**
เมื่อโปรเจ็กต์ถูกแบ่งออกเป็นที่เก็บอิสระหลายแห่ง:
- แอปพลิเคชันส่วนหน้า บริการส่วนหลัง และแอปเดสก์ท็อปที่พัฒนาแยกกัน
- จำเป็นต้องประสานเวอร์ชันและการขึ้นต่อกันระหว่างที่เก็บข้อมูล
- โครงการเอกสารและตัวช่วยแยกจากรหัสหลัก

**2. การจัดการโครงการเสริม**
ใช้ Monospecs ในการจัดการ:
- โครงการเอกสาร
- เว็บไซต์อย่างเป็นทางการ
- สร้างเครื่องมือและสคริปต์
- แต่ละที่เก็บจะรักษาการควบคุมเวอร์ชันของตัวเอง

**3. การพัฒนาคุณสมบัติข้ามพื้นที่เก็บข้อมูล**
คุณลักษณะที่ต้องมีการเปลี่ยนแปลงในที่เก็บหลายแห่ง:
- จำเป็นต้องติดตามการเปลี่ยนแปลงในที่เก็บข้อมูล
- ยอมรับข้อกำหนดโดยอัตโนมัติเพื่อแก้ไขพื้นที่เก็บข้อมูลเป้าหมาย
- ประสานงานการอัปเดตเวอร์ชันและการขึ้นต่อกัน

## เริ่มต้นอย่างรวดเร็ว

### การเริ่มต้น

ทำตามขั้นตอนเหล่านี้เพื่อเริ่มต้น monospecs ให้เสร็จสมบูรณ์:

#### 1. สร้างพื้นที่เก็บข้อมูล git ที่ว่างเปล่า

สร้างที่ว่างเปล่า `git` โฟลเดอร์ในที่เก็บหลักของคุณ (หมายเหตุ: นี่คือ `git`ไม่ใช่ `repos/` - นี่คือแบบแผน monospecs):

```bash
mkdir git
cd git
git init
cd ..
```

#### 2. กำหนดค่าไฟล์ monospecs

สร้าง `monospecs.yaml` ไฟล์การกำหนดค่าในไดเร็กทอรีรากของที่เก็บหลักของคุณ:

```yaml
version: "1.0"
commit_when_archive: true

repositories:
  - path: "repos/frontend"
    url: "https://github.com/your-org/frontend.git"
    displayName: "Frontend App"
    icon: "🌐"

  - path: "repos/backend"
    url: "https://github.com/your-org/backend.git"
    displayName: "Backend Service"
    icon: "⚙️"
```

**รายการการกำหนดค่า**:
| รายการการกำหนดค่า | ประเภท | จำเป็น | คำอธิบาย |
|----------------|------|----------|-------------|
| `version` | เชือก | ใช่ | เวอร์ชันไฟล์การกำหนดค่า |
| `commit_when_archive` | บูลีน | ไม่ | ว่าจะคอมมิตข้อกำหนดอัตโนมัติเมื่อทำการเก็บถาวรหรือไม่ (อย่าคอมมิตโค้ด repo ย่อย) |
| `repositories` | อาร์เรย์ | ใช่ | รายการการกำหนดค่าที่เก็บ |
| `repositories[].path` | เชือก | ใช่ | เส้นทางที่เก็บ (สัมพันธ์กับรูทโปรเจ็กต์) |
| `repositories[].url` | เชือก | ใช่ | URL ที่เก็บ Git |
| `repositories[].displayName` | เชือก | ไม่ | ชื่อที่แสดงของพื้นที่เก็บข้อมูล |
| `repositories[].icon` | เชือก | ไม่ | ไอคอนที่เก็บ (อีโมจิ) |

### เพิ่มพื้นที่เก็บข้อมูลย่อยไปยังพื้นที่เก็บข้อมูลหลัก

เพิ่มที่เก็บย่อยทั้งหมดลงใน `repos/` ไดเรกทอรี ไดเรกทอรีนี้จะต้อง **แยกออกจากการควบคุมเวอร์ชัน Git ของที่เก็บหลัก** เพื่อหลีกเลี่ยงการติดตามเนื้อหาที่เก็บย่อย:

```plaintext
# Exclude all sub-repositories
repos/
```

#### 3. เพิ่มพื้นที่เก็บข้อมูลหลักใน HagiCode

หากคุณใช้แอปพลิเคชันเดสก์ท็อป HagiCode ให้เปิดแอปและเพิ่มพื้นที่เก็บข้อมูลหลักของคุณเป็นโปรเจ็กต์ คลิกปุ่ม "เพิ่มโครงการ" หรือ "+" บนหน้าจอหลัก เลือกโฟลเดอร์พื้นที่เก็บข้อมูลหลักของคุณ จากนั้นคุณจะเห็นพื้นที่เก็บข้อมูลย่อยทั้งหมดและข้อมูลสถานะ

#### 4. โคลนพื้นที่เก็บข้อมูลย่อย

ใช้ `monospecs.yaml` การกำหนดค่าเพื่อโคลนที่เก็บย่อยที่กำหนดค่าไว้ทั้งหมดไปยัง `repos/` ไดเรกทอรี:

สิ่งนี้จะโคลนที่เก็บย่อยทั้งหมดตาม URL ที่ระบุในไฟล์การกำหนดค่า

## โครงสร้างไฟล์การกำหนดค่า

### การกำหนดค่าระดับรูท

```yaml
version: "1.0"
commit_when_archive: true

repositories:
  # ... repository configurations
```

### การกำหนดค่าระดับพื้นที่เก็บข้อมูล

แต่ละที่เก็บย่อยสามารถมี:
- OpenSpec เปลี่ยนโฟลเดอร์ (`openspec/`)
- เป็นอิสระ `.git/` ไดเรกทอรี
- ไม่จำเป็น `AGENTS.md` สำหรับการกำหนดค่า AI Agent

## การดำเนินการจัดการพื้นที่เก็บข้อมูล

### เพิ่มพื้นที่เก็บข้อมูลใหม่

เมื่อเพิ่มพื้นที่เก็บข้อมูลย่อยใหม่ลงในไฟล์ `monospecs.yaml` `repositories` อาร์เรย์:

```yaml
repositories:
  - path: "repos/new-service"
    url: "https://github.com/HagiCode-org/new-service.git"
    displayName: "New Service"
    icon: "🆕"
```

### อัปเดตการกำหนดค่าที่เก็บ

เมื่อ URL ของที่เก็บหรือข้อมูลเมตาเปลี่ยนแปลง:
1. แก้ไข `monospecs.yaml` เพื่ออัปเดตรายการที่เกี่ยวข้อง
2. ตรวจสอบว่าไวยากรณ์ YAML ถูกต้อง
3. หากการซิงค์เปลี่ยนแปลง ให้อัปเดตการกำหนดค่าที่เก็บในเครื่องด้วยตนเอง

## รองรับการกำหนดค่าตัวแทน AI

### ตัวแทน.md

ที่ `AGENTS.md` ไฟล์ให้คำแนะนำเฉพาะพื้นที่เก็บข้อมูลสำหรับ AI Agent:

**ข้อมูลทางเทคนิค**
- กรอบและสร้างเครื่องมือ
- แบบแผนรหัสและแนวปฏิบัติในการตั้งชื่อ
- การจัดการเวอร์ชันและการกำหนดค่าการใช้งาน

**พฤติกรรมเฉพาะโครงการ**
- ส่วนขยายการกำหนดค่าและข้อกำหนดพิเศษ
- รูปแบบการรวม OpenSpec
- ความต้องการการประสานงานข้ามพื้นที่เก็บข้อมูล

### ประโยชน์ของเอไอ

การใช้ `AGENTS.md` ช่วยให้ AI Agent สามารถ:
- ทำความเข้าใจความสัมพันธ์และการขึ้นต่อกันของพื้นที่เก็บข้อมูล
- สร้างโค้ดและรีแฟกเตอร์ในที่เก็บข้อมูลหลายแห่ง
- รักษาความสม่ำเสมอในการควบคุมเวอร์ชันและการกำหนดค่า

## บูรณาการ OpenSpec

### Monospecs ทำงานอย่างไรกับ OpenSpec

ไม่ว่าจะจัดการโปรเจ็กต์ที่มีการจัดการ monospecs หรือพื้นที่เก็บข้อมูลปกติ คุณสามารถใช้เวิร์กโฟลว์ OpenSpec ของ HagiCode ได้:

**พื้นที่เก็บข้อมูลปกติ:**
- OpenSpec ในรูท `openspec/` ไดเรกทอรี
- การเปลี่ยนแปลงส่งผลต่อพื้นที่เก็บข้อมูลปัจจุบันเท่านั้น

**โครงการที่มีการจัดการ Monospecs:**
- OpenSpec อ่าน `monospecs.yaml` การกำหนดค่า
- ติดตามการเปลี่ยนแปลงโดยพื้นที่เก็บข้อมูลย่อย
- ข้อมูลจำเพาะมุ่งมั่นโดยอัตโนมัติเพื่อแก้ไขเป้าหมาย

### พฤติกรรมการเก็บถาวร

เมื่อ `commit_when_archive: true`:
- ข้อมูลจำเพาะจะถูกส่งไปยังพื้นที่เก็บข้อมูลหลักโดยอัตโนมัติ
- ไม่ได้คอมมิตโค้ดของที่เก็บย่อย (ยังคงแยกจากกัน)
- ลดความซับซ้อนในการจัดการข้อกำหนด - ไม่ต้องผูกมัดข้อกำหนดด้วยตนเองสำหรับพื้นที่เก็บข้อมูลย่อย

### คอมมิตพื้นที่เก็บข้อมูลย่อยแบบแมนนวล

ที่เก็บข้อมูลย่อยรักษาการควบคุมเวอร์ชันที่เป็นอิสระ:
- คอมมิตการเปลี่ยนแปลงโค้ดจริงกับที่เก็บของตนเอง
- ข้อกำหนดมีความมุ่งมั่นแยกต่างหาก (จากพื้นที่เก็บข้อมูลหลักเมื่อทำการเก็บถาวร)

## แนวทางปฏิบัติที่ดีที่สุด

### อนุสัญญาการตั้งชื่อพื้นที่เก็บข้อมูล

- ใช้ kebab-case (ตัวอักษรพิมพ์เล็กและขีดกลาง) สำหรับเส้นทางที่เก็บ
- ใช้ชื่อที่แสดงที่สื่อความหมายง่ายและเป็นภาษาจีน
- ใช้อิโมจิที่เกี่ยวข้องเป็นไอคอน

### หลักเกณฑ์เกี่ยวกับชื่อที่แสดง

- ทำให้มันกระชับและสื่อความหมาย
- ใช้การตั้งชื่อที่สอดคล้องกันในที่เก็บข้อมูลที่คล้ายกัน
- เลือกไอคอนที่แสดงถึงวัตถุประสงค์ของพื้นที่เก็บข้อมูล

### การบำรุงรักษาการกำหนดค่า

**การกำหนดค่าการซิงค์กับสถานะจริง**
- ตรวจสอบอย่างสม่ำเสมอ `monospecs.yaml` สะท้อนถึงโครงสร้างพื้นที่เก็บข้อมูลจริง
- อัปเดตหรือลบรายการเมื่อมีการเพิ่มหรือลบที่เก็บ
- ทดสอบการเปลี่ยนแปลงเพื่อให้แน่ใจว่าการกำหนดค่ามีความถูกต้อง

**การควบคุมเวอร์ชันสำหรับการกำหนดค่า**
- ติดตามการเปลี่ยนแปลงไฟล์การกำหนดค่าด้วย git
- เอกสารเหตุผลสำหรับการเปลี่ยนแปลงการกำหนดค่าที่สำคัญ
- เวอร์ชันไฟล์การกำหนดค่าแท็กเพื่อการย้อนกลับที่ง่ายดาย

## การแก้ไขปัญหา

### ไม่พบไฟล์การกำหนดค่า

ถ้า `monospecs.yaml` ไม่พบไฟล์ในรูทโปรเจ็กต์:
1. ตรวจสอบว่าไฟล์อยู่ในตำแหน่งที่ถูกต้องหรือไม่
2. ตรวจสอบว่าคุณอยู่ในไดเร็กทอรีการทำงานที่ถูกต้อง
3. ตรวจสอบการพิมพ์ผิดในชื่อไฟล์การกำหนดค่า

### ข้อผิดพลาดทางไวยากรณ์ YAML

ปัญหาไวยากรณ์ YAML ทั่วไป:
- การเยื้องไม่ถูกต้อง (ใช้ช่องว่าง ไม่ใช่แท็บ)
- เครื่องหมายคำพูดขาดหายไปหรือเพิ่มเติมรอบสตริง
- ประเภทข้อมูลไม่ถูกต้อง (สตริงต้องมีเครื่องหมายคำพูด ตัวเลขไม่ต้อง)
- ไม่มีช่องที่ต้องกรอก (like `path` หรือ `url`)

### ตรวจไม่พบพื้นที่เก็บข้อมูล

หากพื้นที่เก็บข้อมูลที่เพิ่มใหม่ไม่ปรากฏ:
1. ตรวจสอบ `path` ถูกต้องแล้วใน `monospecs.yaml`
2. ตรวจสอบว่าโคลนพื้นที่เก็บข้อมูลสำเร็จแล้ว
3. ตรวจสอบว่ามีที่เก็บอยู่ในตำแหน่งที่คาดหวังหรือไม่

### ความล้มเหลวของโคลน

หากการโคลนพื้นที่เก็บข้อมูลล้มเหลว:
1. ตรวจสอบการเชื่อมต่อเครือข่าย
2. ตรวจสอบว่า URL ที่เก็บ Git ถูกต้อง
3. ตรวจสอบปัญหาการตรวจสอบสิทธิ์
4. ตรวจสอบสถานะคอนเทนเนอร์ Docker
5. ตรวจสอบความพร้อมใช้งานของพื้นที่ดิสก์
