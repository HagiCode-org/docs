## วิธีสร้างสถานีกระจายการดาวน์โหลดที่รวดเร็วด้วยเซิร์ฟเวอร์คลาวด์ราคาประหยัด

> แบนด์วิดธ์การจัดเก็บข้อมูลบนคลาวด์มีราคาแพงอย่างไร้เหตุผล การเข้าถึงข้ามพรมแดนนั้นช้ามาก และราคา CDN ก็เพียงพอที่จะทำให้ทุกคนกลัว... หากคุณจัดการกับการแจกจ่ายไฟล์ คุณคงทราบปัญหาเหล่านี้เป็นอย่างดี ในโพสต์นี้ ฉันต้องการแบ่งปันแนวทางต้นทุนต่ำที่เราเคยใช้ขณะสร้าง [ฮากิโค้ด](https://hagicode.com): เซิร์ฟเวอร์คลาวด์พร้อมเลเยอร์แคช Nginx ค่าใช้จ่ายลดลงประมาณครึ่งหนึ่ง ในขณะที่ความเร็วในการดาวน์โหลดดีขึ้นเล็กน้อย ซึ่งอย่างน้อยก็สบายใจขึ้นเล็กน้อย

## พื้นหลัง

เมื่อพูดถึงอินเทอร์เน็ต ความเร็วในการดาวน์โหลดและความเสถียรเป็นส่วนหนึ่งของประสบการณ์ผู้ใช้อย่างแท้จริง ไม่ว่าคุณจะดำเนินโครงการโอเพ่นซอร์สหรือผลิตภัณฑ์เชิงพาณิชย์ คุณยังคงต้องจัดเตรียมวิธีการดาวน์โหลดไฟล์ที่เชื่อถือได้ให้กับผู้ใช้

การดาวน์โหลดไฟล์โดยตรงจากที่เก็บข้อมูลบนคลาวด์ เช่น Azure Blob Storage หรือ AWS S3 นั้นดูเรียบง่าย แต่มาพร้อมกับปัญหาในทางปฏิบัติบางประการ:

**เวลาแฝงของเครือข่าย**: การเข้าถึงข้ามพรมแดนและข้ามภูมิภาคอาจช้าพอที่จะทำให้คุณอยากทุบคีย์บอร์ดของคุณ หากผู้ใช้ต้องรอตลอดไป ประสบการณ์จะไม่ดีอย่างแน่นอน

**ต้นทุนแบนด์วิธ**: การรับส่งข้อมูลขาออกของพื้นที่เก็บข้อมูลบนคลาวด์มีราคาแพงมาก การเข้าถึง Azure Blob Storage จากจีนแผ่นดินใหญ่มีค่าใช้จ่ายประมาณ 0.5 หยวนต่อ GB ซึ่งหมายความว่า 1 TB ต่อเดือนจะรวมกันสูงถึงประมาณ 500 หยวนจีน สำหรับทีมขนาดเล็ก นั่นไม่ใช่จำนวนที่มีนัยสำคัญเลย ท้ายที่สุดแล้ว เงินของไม่มีใครมาจากลม

**ข้อจำกัดการเข้าถึง**: ในบางภูมิภาค การเข้าถึงบริการคลาวด์ในต่างประเทศไม่เสถียร และบางครั้งก็ไม่พร้อมใช้งานเลย ผู้ใช้ต้องการดาวน์โหลดไฟล์แต่ทำไม่ได้ ซึ่งทำให้ทุกคนหงุดหงิด

**ต้นทุน CDN**: CDN เชิงพาณิชย์สามารถแก้ปัญหาเหล่านี้ได้ แต่ราคาก็เป็นจริงเช่นกัน ทีมเล็กๆ ส่วนใหญ่ไม่สามารถหาเหตุผลมารองรับได้

มีวิธีแก้ไขปัญหาที่ทั้งประหยัดและใช้งานได้จริงหรือไม่? ใช่. ใช้เซิร์ฟเวอร์คลาวด์ พร็อกซีย้อนกลับ และเลเยอร์แคช มันเป็นแนวทางที่ตรงไปตรงมาแต่ได้ผล ค่าใช้จ่ายลดลงประมาณครึ่งหนึ่งและความเร็วก็ดีขึ้นเช่นกัน ซึ่งเป็นการแลกเปลี่ยนที่ดี

## การออกแบบสถาปัตยกรรม

### สถาปัตยกรรมโดยรวม

ให้เราเริ่มต้นด้วยสถาปัตยกรรมแบบเต็ม:

```
User request
    ↓
DNS resolution
    ↓
┌─────────────────────────────────────┐
│   Reverse proxy layer (Traefik/Bunker Web) │ ← SSL termination, routing, security protection
├─────────────────────────────────────┤
│   Ports: 80/443                     │
│   Features: Automatic Let's Encrypt certificates │
│             Host routing            │
└─────────────────────────────────────┘
    ↓
┌─────────────────────────────────────┐
│   Cache layer (Nginx)               │ ← File caching, Gzip compression
├─────────────────────────────────────┤
│   Ports: 8080(server) / 8081(desktop) │
│   Cache strategy:                   │
│   - index.json: 1 hour              │
│   - other files: 7 days             │
│   Cache size: 1GB                   │
└─────────────────────────────────────┘
    ↓
┌─────────────────────────────────────┐
│   Origin (Azure Blob Storage)       │ ← File storage
└─────────────────────────────────────┘
```

แนวคิดหลักของสถาปัตยกรรมนี้เรียบง่าย: **วางแคชระหว่างผู้ใช้และพื้นที่เก็บข้อมูลบนคลาวด์**

คำขอของผู้ใช้จะมาถึงเลเยอร์พร็อกซีย้อนกลับบนเซิร์ฟเวอร์คลาวด์ก่อน จากนั้นเลเยอร์แคช Nginx จะเข้ามาแทนที่ หากไฟล์ที่ร้องขอถูกแคชไว้แล้ว ไฟล์นั้นจะถูกส่งคืนทันที ถ้าไม่เช่นนั้น Nginx จะดึงข้อมูลจากที่เก็บข้อมูลบนคลาวด์และจัดเก็บสำเนาในเครื่องในเวลาเดียวกัน ครั้งถัดไปที่มีคนร้องขอไฟล์เดียวกัน พื้นที่เก็บข้อมูลบนคลาวด์ก็ไม่จำเป็นต้องเกี่ยวข้องอีก

### เหตุใดจึงเลือกสถาปัตยกรรมนี้

**ข้อดีของคลาวด์เซิร์ฟเวอร์**:
- ต้นทุนที่คาดการณ์ได้: ผู้ให้บริการอย่าง Alibaba Cloud นำเสนอเซิร์ฟเวอร์คลาวด์ราคาประหยัด โดยมี 1-2 vCPU และอินสแตนซ์ RAM 2 GB ราคาประมาณ 50-100 หยวนจีนต่อเดือน
- การใช้งานที่ยืดหยุ่น: คุณสามารถกำหนดค่ากฎพร็อกซีย้อนกลับและนโยบายการแคชได้อย่างอิสระ
- ภูมิศาสตร์ที่ยืดหยุ่น: คุณสามารถเลือกภูมิภาคเซิร์ฟเวอร์ที่ใกล้กับผู้ใช้ของคุณได้
- ความสามารถในการปรับขนาดที่ดี: คุณสามารถอัปเกรดข้อกำหนดเซิร์ฟเวอร์ได้เมื่อปริมาณการรับส่งข้อมูลเพิ่มขึ้น

**Reverse proxy + สถาปัตยกรรมแคช**:
- ลดแรงกดดันจากจุดเริ่มต้น: แคชไฟล์ร้อนเพื่อลดการเข้าถึงโดยตรงไปยังที่เก็บข้อมูลบนคลาวด์
- ต้นทุนที่ต่ำกว่า: ปริมาณการใช้เซิร์ฟเวอร์คลาวด์มีราคาถูกกว่าข้อมูลขาออกของที่เก็บข้อมูลบนคลาวด์มาก
- ปรับปรุงความเร็ว: การเข้าถึงในบริเวณใกล้เคียงและแบนด์วิธของเซิร์ฟเวอร์มักจะดีกว่าการจัดส่งที่เก็บข้อมูลบนคลาวด์โดยตรง

**เหตุใดจึงเลือก Nginx เป็นเลเยอร์แคช**

นี่ไม่ใช่การเลือกแบบสุ่ม Nginx มีข้อดีหลายประการที่นี่:

1. ประสิทธิภาพสูง: Nginx ได้รับการยอมรับอย่างกว้างขวางถึงประสิทธิภาพพร็อกซีย้อนกลับที่ยอดเยี่ยม
2. แคชสำหรับผู้ใหญ่: ในตัว `proxy_cache` คุณลักษณะมีเสถียรภาพและเชื่อถือได้
3. การใช้ทรัพยากรต่ำ: สามารถทำงานได้โดยใช้หน่วยความจำเพียง 256 MB
4. การกำหนดค่าที่ยืดหยุ่น: คุณสามารถใช้นโยบายแคชที่แตกต่างกันกับไฟล์ประเภทต่างๆ

## Reverse Proxy Layer: Traefik กับ Bunker Web

โซลูชันการปรับใช้ของ HagiCode รองรับตัวเลือกพร็อกซีย้อนกลับสองตัว และแต่ละตัวก็มีจุดแข็งของตัวเอง:

| ตัวเลือก | ลักษณะ | สถานการณ์ที่เหมาะสม |
|------|------|----------|
| **เทรฟิก** | SSL อัตโนมัติน้ำหนักเบา การกำหนดค่าที่เรียบง่าย | การใช้งานขั้นพื้นฐาน สถานการณ์ที่มีการรับส่งข้อมูลต่ำ |
| **บังเกอร์เว็บ** | WAF ในตัว, การป้องกัน DDoS, การป้องกันบอท | สถานการณ์ที่มีความปลอดภัยสูงและมีการจราจรหนาแน่น |

### Traefik: ตัวเลือกแรกที่มีน้ำหนักเบา

Traefik เป็นพร็อกซีย้อนกลับ HTTP และตัวโหลดบาลานซ์ที่ทันสมัย ข้อได้เปรียบที่ใหญ่ที่สุดคือการกำหนดค่านั้นง่ายและสามารถรับใบรับรอง Let's Encrypt ได้โดยอัตโนมัติ

สำหรับการปรับใช้ครั้งแรกหรือสถานการณ์ที่มีการรับส่งข้อมูลต่ำ Traefik มักเป็นตัวเลือกที่ดีมาก:
- ใช้ทรัพยากรค่อนข้างน้อย CPU 1.5 และหน่วยความจำ 512 MB ก็เพียงพอแล้ว
- ใบรับรอง SSL ได้รับการกำหนดค่าโดยอัตโนมัติ ดังนั้นคุณไม่จำเป็นต้องจัดการด้วยตนเอง
- การกำหนดเส้นทางได้รับการกำหนดค่าผ่านป้ายกำกับ Docker ซึ่งสะดวกเพียงพอ

### Bunker Web: สำหรับสถานการณ์ที่มีความปลอดภัยสูง

Bunker Web เป็นไฟร์วอลล์เว็บแอปพลิเคชันที่ใช้ Nginx พร้อมการป้องกันความปลอดภัยที่ครอบคลุมมากขึ้น

เมื่อใดที่คุณควรพิจารณาเปลี่ยนมาใช้ Bunker Web โดยปกติแล้วในกรณีเช่นนี้:
- คุณอยู่ภายใต้การโจมตี DDoS
- คุณต้องมีการป้องกัน ModSecurity
- คุณต้องการการป้องกันต่อต้านบอท
- คุณมีข้อกำหนดด้านความปลอดภัยที่เข้มงวดมากขึ้น

HagiCode ให้ `switch-deployment.sh` script เพื่อให้คุณสามารถสลับระหว่างสองตัวเลือกได้อย่างรวดเร็ว:

```bash
# Switch to Bunker Web
./switch-deployment.sh bunkerweb

# Switch back to Traefik
./switch-deployment.sh traefik

# Check current status
./switch-deployment.sh status
```

สคริปต์ดำเนินการตรวจสอบล่วงหน้า ตรวจสุขภาพ และการย้อนกลับอัตโนมัติ ดังนั้นกระบวนการเปลี่ยนจึงค่อนข้างปลอดภัยและเชื่อถือได้

## การกำหนดค่าเลเยอร์แคช Nginx

เลเยอร์แคชเป็นแกนหลักของสถาปัตยกรรมทั้งหมด ดังนั้นการกำหนดค่า Nginx จึงสร้างความแตกต่างอย่างมากในประสิทธิภาพของแคช

### การกำหนดค่าเส้นทางแคช

```nginx
# Cache path configuration
proxy_cache_path /var/cache/nginx levels=1:2 keys_zone=azure_cache:10m
                   max_size=1g inactive=7d use_temp_path=off;
```

รายละเอียดพารามิเตอร์:
- `levels=1:2`: ลำดับชั้นไดเรกทอรีแคชที่มีสองระดับเพื่อปรับปรุงประสิทธิภาพการเข้าถึงไฟล์
- `keys_zone=azure_cache:10m`: โซนจัดเก็บคีย์แคช 10 MB เพียงพอสำหรับคีย์จำนวนมาก
- `max_size=1g`: ขนาดแคชสูงสุดคือ 1 GB
- `inactive=7d`: ลบไฟล์แคชหากไม่มีการเข้าถึงเป็นเวลา 7 วัน
- `use_temp_path=off`: เขียนลงในไดเร็กทอรีแคชโดยตรงเพื่อประสิทธิภาพที่ดีขึ้น

### กลยุทธ์แคชแบบฉัตร

ไฟล์ประเภทต่างๆ ต้องใช้กลยุทธ์แคชที่แตกต่างกัน:

```nginx
# Server download service
server {
    listen 8080;

    # Short-term cache for index.json (to allow timely updates)
    location /index.json {
        proxy_cache azure_cache;
        proxy_cache_valid 200 1h;
        proxy_cache_key "$scheme$server_port$request_uri";
        add_header X-Cache-Status $upstream_cache_status;
        add_header Cache-Control "public, max-age=3600";

        # Reverse proxy to Azure OSS
        proxy_pass https://${SERVER_DL_HOST}/${SERVER_DL_CONTAINER}$uri?${SERVER_DL_SAS_TOKEN};
        proxy_ssl_server_name on;
        proxy_ssl_protocols TLSv1.2 TLSv1.3;
    }

    # Long-term cache for static files such as installation packages
    location / {
        proxy_cache azure_cache;
        proxy_cache_valid 200 7d;
        proxy_cache_key "$scheme$server_port$request_uri";
        add_header X-Cache-Status $upstream_cache_status;
        add_header Cache-Control "public, max-age=604800";

        proxy_pass https://${SERVER_DL_HOST}/${SERVER_DL_CONTAINER}$uri?${SERVER_DL_SAS_TOKEN};
        proxy_ssl_server_name on;
        proxy_ssl_protocols TLSv1.2 TLSv1.3;
    }
}
```

**เหตุใดจึงออกแบบเช่นนี้**

`index.json` เป็นไฟล์ตรวจสอบเวอร์ชันจึงต้องอัปเดตทันที ด้วยหน้าต่างแคช 1 ชั่วโมง ผู้ใช้สามารถตรวจพบรุ่นใหม่ได้ภายในเวลาสูงสุดหนึ่งชั่วโมงหลังจากการเผยแพร่

ไฟล์คงที่ เช่น แพ็คเกจการติดตั้งเปลี่ยนแปลงไม่บ่อยนัก ดังนั้นการแคชไฟล์เหล่านั้นเป็นเวลา 7 วันจะลดการเข้าถึงจากต้นทางได้อย่างมาก เมื่อจำเป็นต้องอัปเดต คุณสามารถล้างแคชด้วยตนเองได้

**`X-Cache-Status` ส่วนหัวการตอบกลับ**:

ส่วนหัวนี้ช่วยคุณตรวจสอบพฤติกรรมการเข้าถึงแคช:
- `HIT`: แคชโดน
- `MISS`: แคชหายไป ดึงมาจากจุดเริ่มต้น
- `EXPIRED`: แคชหมดอายุ ดึงข้อมูลจากต้นทางอีกครั้ง
- `BYPASS`: ข้ามแคชแล้ว

วิธีตรวจสอบ:
```bash
curl -I https://server.dl.hagicode.com/app.zip
```

## การวิเคราะห์ต้นทุน

สมมติว่ามีปริมาณการดาวน์โหลด 1 TB ต่อเดือน ให้เราทำคณิตศาสตร์:

| ตัวเลือก | ค่าเข้าชม | ต้นทุนเซิร์ฟเวอร์ | รวม |
|------|----------|------------|------|
| OSS Azure โดยตรง | ประมาณ 500 หยวน | 0 หยวน | 500 หยวนจีน |
| เซิร์ฟเวอร์คลาวด์ + OSS (อัตราส่วนการเข้าถึงแคช 80%) | 100 หยวนจีน + 80 หยวนจีน | 60 หยวนจีน | 240 หยวนจีน |
| CDN เชิงพาณิชย์ | 300-500 หยวนจีน | 0 หยวน | 300-500 หยวนจีน |

**บทสรุป**: การเพิ่มเลเยอร์แคชสามารถลดต้นทุนการจัดจำหน่ายได้ประมาณ 50%

ค่าประมาณนี้ใช้อัตราส่วนการเข้าถึงแคช 80% ในทางปฏิบัติ หากไฟล์ไม่เปลี่ยนแปลงบ่อย อัตราการเข้าชมอาจสูงกว่านี้อีก

## แนวทางปฏิบัติในการปรับใช้

### การเตรียมสิ่งแวดล้อม

```bash
cd /path/to/hagicode_aliyun_deployment/docker
cp .env.example .env
vi .env  # Fill in the Azure OSS SAS URL and Lark Webhook URL
```

**สำคัญ**: `.env` ไฟล์มีข้อมูลที่ละเอียดอ่อนเช่น SAS Token และ Webhook URL อย่าผูกมัดกับการควบคุมเวอร์ชัน

### การกำหนดค่า DNS

เพิ่มระเบียน DNS A ต่อไปนี้:
- `server.dl.hagicode.com` → ไอพีเซิร์ฟเวอร์
- `desktop.dl.hagicode.com` → ไอพีเซิร์ฟเวอร์

### เตรียมใช้งานเซิร์ฟเวอร์

ใช้ Ansible เพื่อเริ่มต้นเซิร์ฟเวอร์โดยอัตโนมัติ:

```bash
cd /path/to/hagicode_aliyun_deployment
ansible-playbook -i ./ansible/inventory/hosts.yml ./ansible/playbooks/init.yml
```

Playbook นี้จัดการงานต่อไปนี้โดยอัตโนมัติ:
- สร้างผู้ใช้การปรับใช้
- ติดตั้ง Docker และ Docker Compose
- กำหนดค่าคีย์ SSH
- ตั้งกฎไฟร์วอลล์

นั่นคืองานการตั้งค่าหลัก และระบบอัตโนมัติช่วยประหยัดเวลาได้มาก

### ปรับใช้บริการ

```bash
./deploy.sh
```

สคริปต์การปรับใช้ช่วยให้คุณทำสิ่งต่อไปนี้:
- ตรวจสอบการกำหนดค่าสภาพแวดล้อม
- ดึงรหัสล่าสุด
- เริ่มคอนเทนเนอร์นักเทียบท่า
- ดำเนินการตรวจสุขภาพ
- ส่งการแจ้งเตือนการปรับใช้ (Lark)

คำสั่งเดียวก็เพียงพอแล้วซึ่งทำให้กระบวนการสะดวก

## เคล็ดลับการดำเนินงาน

### การจัดการแคช

แคชยังต้องการการบำรุงรักษาเป็นครั้งคราว:

**ตรวจสอบการใช้งานดิสก์แคช**:
```bash
docker volume inspect docker_nginx-cache
du -sh /var/lib/docker/volumes/docker_nginx-cache/_data
```

**ล้างแคชด้วยตนเอง**:
```bash
./clear-cache.sh
```

หรือรันคำสั่งด้วยตนเองโดยตรงหากจำเป็น:
```bash
docker exec nginx sh -c "rm -rf /var/cache/nginx/*"
docker restart nginx
```

### ขีดจำกัดทรัพยากร

บนเซิร์ฟเวอร์ 1 คอร์ ขนาด 2 GB การกำหนดค่าขีดจำกัดทรัพยากรจะมีลักษณะดังนี้:

```yaml
services:
  traefik:
    deploy:
      resources:
        limits:
          cpus: '1.50'
          memory: 512M

  nginx:
    deploy:
      resources:
        limits:
          cpus: '0.50'
          memory: 256M
```

หากต้องการตรวจสอบการใช้ทรัพยากร คุณสามารถเรียกใช้ได้เป็นครั้งคราว:
```bash
docker stats
```

### แนวทางปฏิบัติด้านความปลอดภัยของโทเค็น SAS

SAS Token เป็นข้อมูลประจำตัวที่ใช้ในการเข้าถึง Azure Blob Storage ดังนั้นการรั่วไหลจะเป็นเรื่องร้ายแรง:

- อย่ากระทำการ `.env` การควบคุมไฟล์เป็นเวอร์ชัน มันอยู่ในแล้ว `.gitignore`
- ตั้งเวลาหมดอายุ SAS Token ที่เหมาะสม โดยแนะนำ 1 ปี
- จำกัดสิทธิ์ SAS Token ให้อ่านอย่างเดียว
- หมุนเวียนโทเค็น SAS เป็นประจำ

### การตรวจสอบและการแจ้งเตือน

HagiCode ผสานรวมการแจ้งเตือนของ Lark/Feishu Webhook ซึ่งสามารถส่งการแจ้งเตือนสำหรับเหตุการณ์ต่อไปนี้:
- การปรับใช้สำเร็จหรือล้มเหลว
- สถานะการล้างแคช
- ข้อยกเว้นการบริการ

การแจ้งเตือนประกอบด้วยข้อมูลเซิร์ฟเวอร์ การประทับเวลา และรายละเอียดข้อผิดพลาด ทำให้การแก้ไขปัญหารวดเร็วยิ่งขึ้น

### ส่วนขยายความพร้อมใช้งานสูง

เมื่อเซิร์ฟเวอร์เดียวไม่เพียงพออีกต่อไป คุณสามารถพิจารณาสิ่งต่อไปนี้:

1. **การปรับขนาดแนวนอน**: ปรับใช้หลายโหนดและกระจายการรับส่งข้อมูลด้วย DNS Round-Robin หรือโหลดบาลานเซอร์
2. **CDN ข้างหน้า**: วาง CDN ไว้หน้าเซิร์ฟเวอร์คลาวด์เพื่อการเข้าถึงที่รวดเร็วยิ่งขึ้น
3. **การอุ่นแคช**: ใช้สคริปต์เพื่อโหลดไฟล์ร้อนล่วงหน้าลงในแคช

## หมายเหตุ

มีบางสิ่งที่ควรคำนึงถึง:

1. **ใบรับรอง SSL**: Let's Encrypt มีขีดจำกัดอัตรา ดังนั้นอย่าสลับการใช้งานบ่อยเกินไป ไม่เช่นนั้นการออกใบรับรองอาจล้มเหลว
2. **การล้างแคช**: หลังจากอัปเดตไฟล์สำคัญแล้ว อย่าลืมล้างแคช ไม่เช่นนั้นผู้ใช้อาจยังดาวน์โหลดเวอร์ชันเก่าอยู่
3. **การจัดการบันทึก**: ทำความสะอาดบันทึก Docker เป็นประจำ ไม่เช่นนั้นดิสก์อาจเต็ม
4. **กลยุทธ์การสำรองข้อมูล**: สำรองไฟล์เช่น `Traefik acme.json` และการกำหนดค่า Bunker Web
5. **การตรวจสอบและการแจ้งเตือน**: กำหนดค่าการแจ้งเตือนของ Feishu เพื่อให้คุณสามารถติดตามสถานะการใช้งานและตอบสนองต่อปัญหาได้อย่างรวดเร็ว

## บทสรุป

เซิร์ฟเวอร์คลาวด์และเลเยอร์แคช Nginx ทำได้ทั้งหมด HagiCode ใช้โซลูชันนี้โดยมีค่าใช้จ่ายรายเดือนค่อนข้างต่ำ ประมาณ 60-100 หยวนจีนสำหรับเซิร์ฟเวอร์ และผลลัพธ์ที่ได้ก็ถือว่ามั่นคงมาก ข้อดีหลักคือ:

- **ต้นทุนที่คาดการณ์ได้**: ถูกกว่าการใช้พื้นที่เก็บข้อมูลบนคลาวด์โดยตรงประมาณ 50% หรือชำระค่า CDN เชิงพาณิชย์
- **การปรับใช้ที่ยืดหยุ่น**: เลือก Traefik หรือ Bunker Web ขึ้นอยู่กับความต้องการของคุณ
- **ความสามารถในการปรับขนาดที่แข็งแกร่ง**: คุณสามารถปรับขนาดในแนวนอนหรือเพิ่ม CDN ในภายหลังได้หากจำเป็น
- **การดำเนินการที่เรียบง่าย**: เชลล์สคริปต์และ Ansible ช่วยให้การปรับใช้อัตโนมัติตรงไปตรงมา

สำหรับทีมขนาดเล็กและนักพัฒนาอิสระที่ต้องการแจกจ่ายไฟล์ นี่เป็นตัวเลือกที่ใช้งานได้จริงซึ่งคุ้มค่าแก่การลองใช้อย่างแน่นอน

HagiCode ใช้งานสถาปัตยกรรมนี้อย่างเสถียรในการผลิตมาระยะหนึ่งแล้ว และการดาวน์โหลดของผู้ใช้ทั่วโลกยังคงเชื่อถือได้ หากคุณกำลังมองหาวิธีแก้ปัญหาที่คล้ายกัน มันก็คุ้มค่าที่จะลอง
