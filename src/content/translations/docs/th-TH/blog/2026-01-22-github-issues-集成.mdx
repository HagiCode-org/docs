## การสร้างปัญหาการรวม GitHub ตั้งแต่เริ่มต้น: แนวทางปฏิบัติในการเชื่อมต่อโดยตรงส่วนหน้าของ HagiCode

> บทความนี้รวบรวมกระบวนการทั้งหมดในการรวมปัญหา GitHub เข้ากับแพลตฟอร์ม HagiCode เราจะสำรวจวิธีใช้สถาปัตยกรรม "การเชื่อมต่อโดยตรงส่วนหน้า + แบ็กเอนด์ขั้นต่ำ" เพื่อให้ได้การรับรองความถูกต้อง OAuth ที่ปลอดภัย และการซิงโครไนซ์ปัญหาที่มีประสิทธิภาพ ในขณะเดียวกันก็รักษาแบ็คเอนด์ให้มีน้ำหนักเบา

## ความเป็นมา: เหตุใดจึงรวม GitHub

ในฐานะแพลตฟอร์มการพัฒนาที่ได้รับความช่วยเหลือจาก AI ค่านิยมหลักของ HagiCode อยู่ที่การเชื่อมโยงแนวคิดเข้ากับการนำไปปฏิบัติ แต่ในการใช้งานจริง เราพบว่าหลังจากที่ผู้ใช้กรอก Proposal ใน HagiCode แล้ว พวกเขามักจะต้องคัดลอกเนื้อหาไปยัง GitHub Issues ด้วยตนเองเพื่อติดตามโปรเจ็กต์

สิ่งนี้ทำให้เกิดปัญหาที่ชัดเจนหลายประการ:

1.  **ขั้นตอนการทำงานแบบแยกส่วน**: ผู้ใช้จำเป็นต้องสลับไปมาระหว่างสองระบบ ประสบการณ์ไม่ราบรื่น และข้อมูลสำคัญอาจสูญหายได้ง่ายระหว่างการคัดลอกและวาง
2.  **การทำงานร่วมกันไม่สะดวก**: สมาชิกในทีมคนอื่นๆ คุ้นเคยกับการตรวจสอบงานบน GitHub และไม่สามารถมองเห็นความคืบหน้าของข้อเสนอภายใน HagiCode ได้โดยตรง
3.  **การทำงานด้วยตนเองซ้ำๆ**: ทุกครั้งที่มีการอัปเดตข้อเสนอ จะต้องมีคนอัปเดตปัญหาที่เกี่ยวข้องบน GitHub ด้วยตนเอง ซึ่งจะทำให้ค่าบำรุงรักษาที่ไม่จำเป็นเพิ่มขึ้น

เพื่อแก้ไขปัญหานี้ เราจึงตัดสินใจเปิดตัวฟีเจอร์ **การรวมปัญหา GitHub** ซึ่งเชื่อมต่อเซสชัน HagiCode กับที่เก็บ GitHub เพื่อเปิดใช้งาน "การซิงค์ในคลิกเดียว"

## เกี่ยวกับ ฮากิโค้ด

> เฮ้ เรามาแนะนำสิ่งที่เรากำลังสร้างกันดีกว่า

เรากำลังสร้าง **HagiCode** — ผู้ช่วยเขียนโค้ดที่ขับเคลื่อนด้วย AI ซึ่งทำให้การพัฒนาฉลาดขึ้น ง่ายขึ้น และสนุกสนานยิ่งขึ้น

**ฉลาดกว่า** — AI ช่วยเหลือตลอดการเดินทาง ตั้งแต่แนวคิดไปจนถึงโค้ด เพิ่มประสิทธิภาพการพัฒนาแบบทวีคูณ **ง่ายขึ้น** — การดำเนินการพร้อมกันแบบมัลติเธรดทำให้ใช้ทรัพยากรได้อย่างเต็มที่ และทำให้ขั้นตอนการพัฒนาราบรื่น **สนุกยิ่งขึ้น** — Gamification และระบบความสำเร็จทำให้การเขียนโค้ดน่าเบื่อน้อยลงและคุ้มค่ามากขึ้น

โครงการกำลังทำซ้ำอย่างรวดเร็ว หากคุณสนใจในการเขียนเชิงเทคนิค การจัดการความรู้ หรือการพัฒนาโดยใช้ AI โปรดติดต่อเรา [GitHub](https://github.com/HagiCode-org/site)~

## ตัวเลือกทางเทคนิค: การเชื่อมต่อโดยตรงส่วนหน้าเทียบกับพร็อกซีแบ็กเอนด์

เมื่อออกแบบแนวทางบูรณาการ เรามีสองตัวเลือกตรงหน้า: "โมเดลพร็อกซีแบ็กเอนด์" แบบเดิมและ "โมเดลการเชื่อมต่อโดยตรงส่วนหน้า" ที่ก้าวร้าวมากขึ้น

### การเปรียบเทียบโซลูชัน

ใน **โมเดลพร็อกซีแบ็กเอนด์** แบบดั้งเดิม ทุกคำขอจากฟรอนต์เอนด์จะต้องผ่านแบ็กเอนด์ของเราก่อน ซึ่งจะเรียกใช้ GitHub API สิ่งนี้จะรวมศูนย์ตรรกะ แต่ยังสร้างภาระสำคัญให้กับแบ็กเอนด์ด้วย:

1.  **แบ็กเอนด์ที่มากเกินไป**: เราจะต้องเขียน Wrapper ไคลเอ็นต์ GitHub API โดยเฉพาะ และยังจัดการเครื่องสถานะ OAuth ที่ซับซ้อนด้วย
2.  **ความเสี่ยงของโทเค็น**: โทเค็น GitHub ของผู้ใช้จะต้องถูกจัดเก็บไว้ในฐานข้อมูลแบ็กเอนด์ แม้ว่าจะมีการเข้ารหัส แต่ก็ยังเพิ่มพื้นที่การรักษาความปลอดภัย
3.  **ต้นทุนการพัฒนา**: เราจำเป็นต้องมีการย้ายฐานข้อมูลเพื่อจัดเก็บโทเค็นและบริการซิงโครไนซ์เพิ่มเติมเพื่อบำรุงรักษา

**รูปแบบการเชื่อมต่อโดยตรงส่วนหน้า** มีน้ำหนักเบากว่ามาก ในแนวทางนี้ เราใช้แบ็กเอนด์เฉพาะสำหรับขั้นตอน "การแลกเปลี่ยนลับ" ที่ละเอียดอ่อนที่สุดเท่านั้น (การเรียกกลับ OAuth) หลังจากได้รับโทเค็นแล้ว เราจะจัดเก็บไว้ในเบราว์เซอร์โดยตรง `localStorage`. การดำเนินการในภายหลัง เช่น การสร้างปัญหาและการอัปเดตความคิดเห็นจะถูกส่งโดยตรงจากส่วนหน้าไปยัง GitHub ผ่าน HTTP

| มิติการเปรียบเทียบ | โมเดลพร็อกซีแบ็กเอนด์ | รูปแบบการเชื่อมต่อโดยตรงส่วนหน้า |
| :--- | :--- | :--- |
| **ความซับซ้อนของแบ็กเอนด์** | ต้องการบริการ OAuth เต็มรูปแบบและไคลเอ็นต์ GitHub API | ต้องการเพียงปลายทางการโทรกลับ OAuth เท่านั้น |
| **การจัดการโทเค็น** | จะต้องเข้ารหัสและจัดเก็บไว้ในฐานข้อมูลซึ่งมีความเสี่ยงต่อการรั่วไหล | เก็บไว้ในเบราว์เซอร์และมองเห็นได้เฉพาะผู้ใช้เท่านั้น |
| **ค่าใช้จ่ายในการดำเนินการ** | ต้องมีการย้ายฐานข้อมูลและการพัฒนาหลายบริการ | งานส่วนหน้าเป็นหลัก |
| **ประสบการณ์การใช้งาน** | ตรรกะแบบรวมศูนย์ แต่เวลาแฝงของเซิร์ฟเวอร์อาจสูงกว่าเล็กน้อย | ตอบสนองรวดเร็วมากด้วยการโต้ตอบ GitHub โดยตรง |

เนื่องจากเราต้องการการบูรณาการที่รวดเร็วและการเปลี่ยนแปลงแบ็กเอนด์น้อยที่สุด **ในที่สุดเราจึงเลือก "รูปแบบการเชื่อมต่อโดยตรงส่วนหน้า"** มันเหมือนกับการให้ "บัตรผ่านชั่วคราว" แก่เบราว์เซอร์ เมื่อผ่านแล้ว เบราว์เซอร์สามารถจัดการสิ่งต่าง ๆ บน GitHub ได้ด้วยตัวเอง โดยไม่ต้องขออนุมัติจากผู้ดูแลระบบแบ็กเอนด์ทุกครั้ง

## การออกแบบหลัก: การไหลของข้อมูลและความปลอดภัย

หลังจากตกลงเรื่องสถาปัตยกรรมแล้ว เราจำเป็นต้องออกแบบโฟลว์ข้อมูลเฉพาะ แกนหลักของกระบวนการซิงโครไนซ์คือวิธีการรับโทเค็นอย่างปลอดภัยและใช้งานอย่างมีประสิทธิภาพ

### แผนภาพสถาปัตยกรรมโดยรวม

ระบบทั้งหมดสามารถสรุปได้เป็นสามบทบาท: เบราว์เซอร์ (ส่วนหน้า), แบ็กเอนด์ HagiCode และ GitHub

```text
+--------------+        +--------------+        +--------------+
| Frontend     |        |   Backend    |        |   GitHub     |
| React        |        |   ASP.NET    |        |   REST API   |
|              |        |              |        |              |
|  +--------+  |        |              |        |              |
|  | OAuth  |--+--------> /callback    |        |              |
|  | Flow   |  |        |              |        |              |
|  +--------+  |        |              |        |              |
|              |        |              |        |              |
|  +--------+  |        |  +--------+  |        |  +--------+  |
|  | GitHub |  +------------> Session |  +----------> Issues |  |
|  | API    |  |        |  |Metadata|  |        |  |        |  |
|  | Direct |  |        |  +--------+  |        |  +--------+  |
|  +--------+  |        |              |        |              |
+--------------+        +--------------+        +--------------+
```

**ประเด็นสำคัญคือ**: มีเพียงขั้นตอนเล็กๆ เพียงหนึ่งขั้นตอนใน OAuth (การเปลี่ยนรหัสเป็นโทเค็น) ที่ต้องผ่านแบ็กเอนด์ หลังจากนั้น การทำงานหนัก (การสร้างปัญหา) จะได้รับการจัดการโดยตรงระหว่างส่วนหน้าและ GitHub

### การซิงโครไนซ์ข้อมูลอย่างละเอียด

เมื่อผู้ใช้คลิกปุ่ม "ซิงค์กับ GitHub" ใน HagiCode UI ชุดของการดำเนินการที่ซับซ้อนจะเกิดขึ้น:

```text
User clicks "Sync to GitHub"
         │
         ▼
1. Frontend checks localStorage for the GitHub token
         │
         ▼
2. Format issue content (convert the Proposal into Markdown)
         │
         ▼
3. Frontend directly calls the GitHub API to create/update the issue
         │
         ▼
4. Call the HagiCode backend API to update Session.metadata (store issue URL and other info)
         │
         ▼
5. Backend broadcasts the SessionUpdated event via SignalR
         │
         ▼
6. Frontend receives the event and updates the UI to show the "Synced" state
```

### การออกแบบความปลอดภัย

ความปลอดภัยเป็นสิ่งสำคัญที่สุดเสมอเมื่อรวมบริการของบุคคลที่สาม เราได้พิจารณาดังต่อไปนี้:

1.  **ป้องกันการโจมตี CSRF**: สร้างแบบสุ่ม `state` พารามิเตอร์ระหว่างการเปลี่ยนเส้นทาง OAuth และเก็บไว้ใน `sessionStorage`. ตรวจสอบสถานะในการเรียกกลับอย่างเคร่งครัดเพื่อป้องกันคำขอปลอมแปลง
2.  **การจัดเก็บโทเค็นแบบแยก**: โทเค็นจะถูกจัดเก็บไว้ในเบราว์เซอร์เท่านั้น `localStorage`. เมื่อใช้นโยบาย Same-Origin มีเพียงสคริปต์ HagiCode เท่านั้นที่สามารถอ่านได้ หลีกเลี่ยงความเสี่ยงที่ฐานข้อมูลฝั่งเซิร์ฟเวอร์จะรั่วไหลซึ่งส่งผลต่อผู้ใช้
3.  **ขอบเขตข้อผิดพลาด**: เราได้ออกแบบการจัดการโดยเฉพาะสำหรับข้อผิดพลาด GitHub API ทั่วไป (เช่น โทเค็นหมดอายุ 401 รายการ การตรวจสอบความถูกต้อง 422 ล้มเหลว และการจำกัดอัตรา 429) เพื่อให้ผู้ใช้ได้รับคำติชมที่เป็นมิตร

### 1. การเปลี่ยนแปลงแบ็กเอนด์น้อยที่สุด

### 2. โฟลว์ OAuth ส่วนหน้า

นี่คือจุดเริ่มต้นของการเชื่อมต่อ เราใช้ขั้นตอนรหัสอนุญาตมาตรฐาน

```typescript
// src/HagiCode.Client/src/services/githubOAuth.ts

// Generate the authorization URL and redirect
export async function generateAuthUrl(): Promise<string> {
  const state = generateRandomString(); // Generate a random string for CSRF protection
  sessionStorage.setItem('hagicode_github_state', state);
  
  const params = new URLSearchParams({
    client_id: clientId,
    redirect_uri: window.location.origin + '/settings?tab=github&oauth=callback',
    scope: ['repo', 'public_repo'].join(' '),
    state: state,
  });
  
  return `https://github.com/login/oauth/authorize?${params.toString()}`;
}

// Handle the code-to-token exchange on the callback page
export async function exchangeCodeForToken(code: string, state: string): Promise<GitHubToken> {
  // 1. Validate state to prevent CSRF
  const savedState = sessionStorage.getItem('hagicode_github_state');
  if (state !== savedState) throw new Error('Invalid state parameter');

  // 2. Call the backend API to exchange the token
  // Note: this must go through the backend because ClientSecret cannot be exposed to the frontend
  const response = await fetch('/api/GitHubOAuth/callback', {
    method: 'POST',
    headers: { 'Content-Type': 'application/json' },
    body: JSON.stringify({ code, state, redirectUri: window.location.origin + '/settings?tab=github&oauth=callback' }),
  });

  if (!response.ok) throw new Error('Failed to exchange token');
  
  const token = await response.json();
  
  // 3. Save into LocalStorage
  saveToken(token);
  return token;
}
```

### 3. Wrapper ไคลเอนต์ GitHub API

เมื่อเรามีโทเค็นแล้ว เราจำเป็นต้องมีเครื่องมือที่แข็งแกร่งสำหรับการเรียกใช้ GitHub API

```typescript
// src/HagiCode.Client/src/services/githubApiClient.ts

const GITHUB_API_BASE = 'https://api.github.com';

// Core request wrapper
async function githubApi<T>(endpoint: string, options: RequestInit = {}): Promise<T> {
  const token = localStorage.getItem('gh_token');
  if (!token) throw new Error('Not connected to GitHub');
  
  const response = await fetch(`${GITHUB_API_BASE}${endpoint}`, {
    ...options,
    headers: {
      ...options.headers,
      Authorization: `Bearer ${token}`,
      Accept: 'application/vnd.github.v3+json', // Specify the API version
    },
  });
  
  // Error handling logic
  if (!response.ok) {
    if (response.status === 401) throw new Error('GitHub token expired, please reconnect');
    if (response.status === 403) throw new Error('No permission to access this repository or rate limit exceeded');
    if (response.status === 422) throw new Error('Issue validation failed, the title may be duplicated');
    throw new Error(`GitHub API Error: ${response.statusText}`);
  }
  
  return response.json();
}

// Create issue
export async function createIssue(owner: string, repo: string, data: { title: string, body: string, labels: string[] }) {
  return githubApi(`/repos/${owner}/${repo}/issues`, {
    method: 'POST',
    body: JSON.stringify(data),
  });
}
```

### 4. การจัดรูปแบบเนื้อหาและการซิงโครไนซ์

ขั้นตอนสุดท้ายคือการแปลงข้อมูลเซสชัน HagiCode เป็นรูปแบบของปัญหา GitHub มันก็เหมือนกับงานแปลนิดหน่อย

// Convert a Session object into a Markdown string
function formatIssueForSession(session: Session): string {
  let content = `# ${session.title}\n\n`;
  content += `**> HagiCode Session:** #${session.code}\n`;
  content += `**> Status:** ${session.status}\n\n`;
  content += `## Description\n\n${session.description || 'No description provided.'}\n\n`;
  
  // If this is a Proposal session, add extra fields
  if (session.type === 'proposal') {
    content += `## Chief Complaint\n\n${session.chiefComplaint || ''}\n\n`;
    // Add a deep link so users can jump back from GitHub to HagiCode
    content += `---\n\n**[View in HagiCode](hagicode://sessions/${session.id})**\n`;
  }
  
  return content;
}

// Main logic when clicking the sync button
const handleSync = async (session: Session) => {
  try {
    const repoInfo = parseRepositoryFromUrl(session.repoUrl); // Parse the repository URL
    if (!repoInfo) throw new Error('Invalid repository URL');

    toast.loading('Syncing to GitHub...');
    
    // 1. Format content
    const issueBody = formatIssueForSession(session);
    
    // 2. Call API
    const issue = await githubApiClient.createIssue(repoInfo.owner, repoInfo.repo, {
      title: `[HagiCode] ${session.title}`,
      body: issueBody,
      labels: ['hagicode', 'proposal', `status:${session.status}`],
    });
    
    // 3. Update Session Metadata (save the issue link)
    await SessionsService.patchApiSessionsSessionId(session.id, {
      metadata: {
        githubIssue: {
          owner: repoInfo.owner,
          repo: repoInfo.repo,
          issueNumber: issue.number,
          issueUrl: issue.html_url,
          syncedAt: new Date().toISOString(),
        }
      }
    });

    toast.success('Synced successfully!');
  } catch (err) {
    console.error(err);
    toast.error('Sync failed, please check your token or network');
  }
};

## สรุปและแนวโน้ม

ด้วยแนวทาง "การเชื่อมต่อโดยตรงส่วนหน้า" นี้ เราจึงประสบความสำเร็จในการผสานรวมปัญหา GitHub เข้ากับโค้ดแบ็กเอนด์ที่น้อยที่สุดที่เป็นไปได้

### กำไรที่สำคัญ
1.  **ประสิทธิภาพการพัฒนาสูง**: การเปลี่ยนแปลงแบ็กเอนด์มีเพียงเล็กน้อย โดยส่วนใหญ่จะมีช่องฐานข้อมูลพิเศษเพียงช่องเดียวและมีจุดสิ้นสุดการโทรกลับ OAuth แบบธรรมดา ตรรกะส่วนใหญ่จะเสร็จสมบูรณ์ในส่วนหน้า
2.  **การรักษาความปลอดภัยที่แข็งแกร่ง**: โทเค็นไม่ผ่านฐานข้อมูลเซิร์ฟเวอร์ ซึ่งช่วยลดความเสี่ยงในการรั่วไหล
3.  **ประสบการณ์ผู้ใช้ที่ยอดเยี่ยม**: คำขอเริ่มต้นโดยตรงจากฟรอนต์เอนด์ ดังนั้นความเร็วในการตอบกลับจึงรวดเร็ว และไม่จำเป็นต้องส่งต่อแบ็กเอนด์

### สิ่งที่ต้องระวัง
มีข้อผิดพลาดบางประการที่ควรคำนึงถึงในระหว่างการปรับใช้จริง:
- **การตั้งค่าแอป OAuth**: อย่าลืมป้อนข้อมูลที่ถูกต้อง `Authorization callback URL` ในการตั้งค่าแอป GitHub OAuth ของคุณ (โดยปกติคือ `http://localhost:3000/settings?tab=github&oauth=callback`).
- **การจำกัดอัตรา**: GitHub API จะจำกัดคำขอที่ไม่ได้รับการรับรองความถูกต้องค่อนข้างเข้มงวด แต่โควต้ามักจะเพียงพอ (5,000 คำขอ/ชั่วโมง) ด้วยโทเค็น
- **การแยกวิเคราะห์ URL**: ผู้ใช้ป้อน URL ของที่เก็บทุกประเภท ดังนั้นโปรดตรวจสอบให้แน่ใจว่า regex ของคุณสามารถจับคู่ได้ `.git` ส่วนต่อท้าย รูปแบบ SSH และกรณีที่คล้ายกัน

### การปรับปรุงในอนาคต
คุณลักษณะปัจจุบันยังคงเป็นการซิงโครไนซ์ทางเดียว (HagiCode -> GitHub) ในอนาคต เราวางแผนที่จะใช้การซิงโครไนซ์แบบสองทางผ่าน GitHub Webhooks ตัวอย่างเช่น หากปัญหาถูกปิดบน GitHub สถานะเซสชันในด้าน HagiCode ก็อาจอัปเดตโดยอัตโนมัติได้เช่นกัน นั่นจะทำให้เราต้องเปิดเผยจุดสิ้นสุดของเว็บฮุคที่แบ็กเอนด์ ซึ่งจะเป็นขั้นตอนต่อไปที่น่าสนใจ

เราหวังว่าบทความนี้จะเป็นแรงบันดาลใจเล็กๆ น้อยๆ ให้กับคุณในการพัฒนาการบูรณาการโดยบุคคลที่สาม หากคุณมีคำถาม โปรดเปิดประเด็นเพื่อหารือเกี่ยวกับ [ฮากิโค้ด GitHub](https://github.com/HagiCode-org/site).
