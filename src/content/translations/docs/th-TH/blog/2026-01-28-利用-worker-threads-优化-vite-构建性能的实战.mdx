## จาก 120 วินาทีถึง 45 วินาที: คู่มือปฏิบัติเพื่อเพิ่มประสิทธิภาพ Vite Build ด้วย Worker Threads

> เมื่อทำงานกับโปรเจ็กต์ส่วนหน้าขนาดใหญ่ การสร้างเวอร์ชันที่ใช้งานจริงอาจรู้สึกช้ามาก บทความนี้แชร์วิธีที่เราใช้ Node.js Worker Threads เพื่อลดระยะการสร้างความสับสนใน Vite build จาก 120 วินาทีเหลือ 45 วินาที พร้อมด้วยรายละเอียดการใช้งานและบทเรียนที่เรียนรู้ในโปรเจ็กต์ HagiCode

## พื้นหลัง

ในการปฏิบัติงานด้านวิศวกรรมส่วนหน้าของเรา ปัญหาด้านประสิทธิภาพในการสร้างเริ่มมีความโดดเด่นมากขึ้นเมื่อโครงการเติบโตขึ้น โดยเฉพาะอย่างยิ่ง ในระหว่างกระบวนการสร้างที่ใช้งานจริง เรามักจะแนะนำเครื่องมือที่ทำให้สับสนของ JavaScript เช่น `javascript-obfuscator` เพื่อปกป้องตรรกะของซอร์สโค้ด ขั้นตอนนี้จำเป็น แต่ก็มีราคาแพงในการคำนวณและเชื่อมโยงกับ CPU อย่างมากเช่นกัน

ในช่วงเริ่มต้นของการพัฒนา **HagiCode** เราประสบปัญหาคอขวดด้านประสิทธิภาพที่ยุ่งยากมาก: เวลาในการสร้างการผลิตลดลงอย่างรวดเร็วเมื่อโค้ดเบสเติบโตขึ้น

**จุดปวดเฉพาะคือ**:
- งาน Obfuscation ทำงานตามลำดับบนเธรดเดียว โดยใช้งาน CPU คอร์หนึ่งคอร์จนเต็มประสิทธิภาพ ในขณะที่อีกคอร์ไม่ได้ใช้งาน
- เวลาในการสร้างเพิ่มขึ้นจากเดิม 30 วินาทีเป็น 110-120 วินาที
- วงจรการตรวจสอบหลังการเปลี่ยนแปลงใช้เวลานานมาก ซึ่งทำให้การพัฒนาซ้ำช้าลงอย่างมาก
- ในไปป์ไลน์ CI/CD ขั้นตอนการสร้างกลายเป็นส่วนที่ใช้เวลานานที่สุด

**เหตุใด HagiCode จึงต้องการสิ่งนี้**
HagiCode คือผู้ช่วยโค้ดที่ขับเคลื่อนด้วย AI ซึ่งมีสถาปัตยกรรมส่วนหน้าประกอบด้วยตรรกะทางธุรกิจที่ซับซ้อนและโมดูลการโต้ตอบของ AI เพื่อให้มั่นใจในความปลอดภัยของโค้ดหลักของเรา เราได้บังคับใช้การทำให้งงงันที่มีความเข้มข้นสูงในการเผยแพร่เวอร์ชันที่ใช้งานจริง เมื่อต้องเผชิญกับการรอบิลด์ที่ใกล้จะถึงสองนาที เราจึงตัดสินใจดำเนินการเพิ่มประสิทธิภาพเชิงลึกของระบบบิลด์

## การวิเคราะห์: การค้นหาจุดทะลุทะลวงในคอขวดของประสิทธิภาพ

ก่อนที่จะแก้ไขปัญหาด้านประสิทธิภาพ ก่อนอื่นเราต้องชี้แจงความคิดของเราและระบุวิธีแก้ปัญหาทางเทคนิคที่ดีที่สุด

### การตัดสินใจหลัก: เหตุใดจึงเลือกเธรดผู้ปฏิบัติงาน

มีสามวิธีหลักในการคำนวณแบบขนานใน Node.js:
1. **child_process**: สร้างกระบวนการย่อยที่เป็นอิสระ
2. **Web Workers**: ใช้งานบนฝั่งเบราว์เซอร์เป็นหลัก
3. **worker_threads**: รองรับมัลติเธรดดั้งเดิมใน Node.js

หลังจากเปรียบเทียบตัวเลือกต่างๆ ในที่สุด HagiCode ก็เลือก **Worker Threads** ด้วยเหตุผลต่อไปนี้:
- **ค่าใช้จ่ายในการซีเรียลไลซ์เป็นศูนย์**: Worker Threads ทำงานในกระบวนการเดียวกันและสามารถแชร์หน่วยความจำผ่านได้ `SharedArrayBuffer` หรือโอนกรรมสิทธิ์ หลีกเลี่ยงต้นทุนการซีเรียลไลซ์จำนวนมากของการสื่อสารระหว่างกระบวนการ
- **การสนับสนุนแบบเนทีฟ**: มีอยู่ใน Node.js 12+ โดยไม่จำเป็นต้องมีการพึ่งพาแบบเฮฟวี่เวทเพิ่มเติม
- **บริบทที่เป็นหนึ่งเดียว**: การดีบักและการบันทึกทำได้สะดวกกว่ากระบวนการย่อย

### รายละเอียดของงาน: งาน Obfuscation ควรแบ่งอย่างไร?

เป็นการยากที่จะขนานการสร้างความสับสนให้กับไฟล์บันเดิล JS ขนาดใหญ่ไฟล์เดียว เนื่องจากโค้ดมีการขึ้นต่อกัน แต่เอาต์พุตของ Vite build ประกอบด้วย **ชิ้น** หลายชิ้น นั่นทำให้เรามีขอบเขตขนานตามธรรมชาติ:

- **ความเป็นอิสระ**: หลังจากบรรจุภัณฑ์ของ Vite การขึ้นต่อกันระหว่างชิ้นต่างๆ จะถูกแยกออกแล้ว เพื่อให้สามารถประมวลผลพร้อมกันได้อย่างปลอดภัย
- **รายละเอียดที่เหมาะสม**: โดยปกติแล้วโปรเจ็กต์จะมีชิ้นส่วน 10-30 ชิ้น ซึ่งเป็นระดับที่ดีเยี่ยมสำหรับการจัดกำหนดการแบบคู่ขนาน
- **บูรณาการง่าย**: the `generateBundle` hook ในปลั๊กอิน Vite ช่วยให้เราสามารถสกัดกั้นและประมวลผลชิ้นส่วนเหล่านี้ก่อนที่ไฟล์จะถูกปล่อยออกมา

### การออกแบบสถาปัตยกรรม

เราได้ออกแบบระบบการประมวลผลแบบขนานโดยมีองค์ประกอบหลักสี่ประการ:

1. **ตัวแยกงาน**: วนซ้ำอ็อบเจ็กต์บันเดิลของ Vite กรองไฟล์ที่ไม่จำเป็นต้องสร้างความสับสน เช่น กลุ่มผู้จำหน่าย และสร้างคิวงาน
2. **ตัวจัดการพูลผู้ปฏิบัติงาน**: จัดการวงจรการใช้งานของผู้ปฏิบัติงาน และจัดการการกระจายงาน การรีไซเคิล และการลองใหม่อีกครั้งเมื่อเกิดความล้มเหลว
3. **รายงานความคืบหน้า**: ผลลัพธ์จะสร้างความคืบหน้าแบบเรียลไทม์เพื่อลดความวิตกกังวลในการรอคอย
4. **ObfuscationWorker**: เธรดผู้ปฏิบัติงานที่ดำเนินการตรรกะในการสร้างความสับสนจริงๆ

## วิธีแก้ปัญหา: การนำไปปฏิบัติจริง

จากการวิเคราะห์ข้างต้น เราได้เริ่มใช้ระบบสร้างความสับสนแบบขนานนี้

### 1. กำหนดค่าปลั๊กอิน Vite

ขั้นแรก เราได้รวมปลั๊กอินการทำให้งงงวยแบบขนานเข้าด้วยกัน `vite.config.ts`. การกำหนดค่าตรงไปตรงมา คุณจะต้องระบุจำนวนคนงานและกฎการสร้างความสับสนเท่านั้น

```typescript
import { defineConfig } from 'vite'
import { parallelJavascriptObfuscator } from './buildTools/plugin'

export default defineConfig(({ mode }) => {
  const isProduction = mode === 'production'
  
  return {
    build: {
      rollupOptions: {
        ...(isProduction
          ? {
              plugins: [
                parallelJavascriptObfuscator({
                  enabled: true,
                  // Automatically adjust based on CPU core count; leave one core for the main thread
                  workerCount: 4, 
                  retryAttempts: 3,
                  fallbackToMainThread: true, // Automatically degrade to single-thread mode on failure
                  // Filter out vendor chunks; third-party libraries usually do not need obfuscation
                  isVendorChunk: (fileName: string) => fileName.includes('vendor-'),
                  obfuscationConfig: {
                    compact: true,
                    controlFlowFlattening: true,
                    deadCodeInjection: true,
                    disableConsoleOutput: true,
                    // ... more obfuscation options
                  },
                }),
              ],
            }
          : {}),
      },
    },
  }
})
```

### 2. นำลอจิกของผู้ปฏิบัติงานไปใช้

ผู้ปฏิบัติงานคือหน่วยที่ดำเนินงาน เราจำเป็นต้องกำหนดโครงสร้างข้อมูลอินพุตและเอาต์พุตให้ชัดเจน

**หมายเหตุ**: แม้ว่าโค้ดที่นี่จะเรียบง่าย แต่ก็มีข้อผิดพลาดหลายประการที่ต้องระวัง เช่น การตรวจสอบว่า `parentPort` เป็นโมฆะและจัดการข้อผิดพลาดได้อย่างถูกต้อง ในการใช้งาน HagiCode เราพบว่ารูปแบบไวยากรณ์ ES6 พิเศษบางอย่างอาจทำให้ obfuscator หยุดทำงาน ดังนั้นเราจึงเพิ่ม `try-catch` การป้องกัน

```typescript
import { parentPort } from 'worker_threads'
import javascriptObfuscator from 'javascript-obfuscator'

export interface ObfuscationTask {
  chunkId: string
  code: string
  config: any
}

export interface ObfuscationResult {
  chunkId: string
  obfuscatedCode: string
  error?: string
}

// Listen for tasks sent from the main thread
if (parentPort) {
  parentPort.on('message', async (task: ObfuscationTask) => {
    try {
      // Perform obfuscation
      const obfuscated = javascriptObfuscator.obfuscate(task.code, task.config)
      const result: ObfuscationResult = {
        chunkId: task.chunkId,
        obfuscatedCode: obfuscated.getObfuscatedCode(),
      }
      // Send the result back to the main thread
      parentPort?.postMessage(result)
    } catch (error) {
      // Handle exceptions so one Worker crash does not block the whole build
      const result: ObfuscationResult = {
        chunkId: task.chunkId,
        obfuscatedCode: '',
        error: error instanceof Error ? error.message : 'Unknown error',
      }
      parentPort?.postMessage(result)
    }
  })
}
```

### 3. ผู้จัดการพูลคนงาน

นี่คือแกนหลักของโซลูชันทั้งหมด เราจำเป็นต้องรักษาพูลผู้ปฏิบัติงานที่มีขนาดคงที่ และจัดกำหนดการงานโดยใช้กลยุทธ์ **FIFO (เข้าก่อน ออกก่อน)**

```typescript
import { Worker } from 'worker_threads'
import os from 'os'

export class WorkerPool {
  private workers: Worker[] = []
  private taskQueue: Array<{
    task: ObfuscationTask
    resolve: (result: ObfuscationResult) => void
    reject: (error: Error) => void
  }> = []
  
  constructor(options: WorkerPoolOptions = {}) {
    // Default to core count - 1 so the main thread still has some breathing room
    const workerCount = options.workerCount ?? Math.max(1, (os.cpus().length || 4) - 1)
    
    for (let i = 0; i < workerCount; i++) {
      this.createWorker()
    }
  }

  private createWorker() {
    const worker = new Worker('./worker.ts')
    
    worker.on('message', (result) => {
      // After one task finishes, take the next task from the queue
      const nextTask = this.taskQueue.shift()
      if (nextTask) {
        this.dispatchTask(worker, nextTask)
      } else {
        // If there are no pending tasks, mark the Worker as idle
        this.activeWorkers.delete(worker)
      }
    })
    
    this.workers.push(worker)
  }

  // Submit a task to the pool
  public runTask(task: ObfuscationTask): Promise<ObfuscationResult> {
    return new Promise((resolve, reject) => {
      const job = { task, resolve, reject }
      const idleWorker = this.workers.find(w => !this.activeWorkers.has(w))
      
      if (idleWorker) {
        this.dispatchTask(idleWorker, job)
      } else {
        this.taskQueue.push(job)
      }
    })
  }

  private dispatchTask(worker: Worker, job: any) {
    this.activeWorkers.set(worker, job.task)
    worker.postMessage(job.task)
  }
}
```

### 4. การรายงานความคืบหน้า

การรอคอยเป็นสิ่งที่เจ็บปวด โดยเฉพาะอย่างยิ่งเมื่อคุณไม่รู้ว่าจะต้องใช้เวลานานเท่าใด ดังนั้นเราจึงเพิ่มรายงานความคืบหน้าง่ายๆ เพื่อให้ข้อเสนอแนะแบบเรียลไทม์เกี่ยวกับสถานะปัจจุบัน

```typescript
export class ProgressReporter {
  private completed = 0
  private readonly total: number
  private readonly startTime: number

  constructor(total: number) {
    this.total = total
    this.startTime = Date.now()
  }

  increment(): void {
    this.completed++
    this.report()
  }

  private report(): void {
    const now = Date.now()
    const elapsed = now - this.startTime
    const percentage = (this.completed / this.total) * 100
    
    // Simple ETA estimate
    const avgTimePerChunk = elapsed / this.completed
    const remaining = (this.total - this.completed) * avgTimePerChunk

    console.log(
      `[Parallel Obfuscation] ${this.completed}/${this.total} chunks completed (${percentage.toFixed(1)}%) | ETA: ${(remaining / 1000).toFixed(1)}s`
    )
  }
}
```

## แนวปฏิบัติ: ผลลัพธ์และบทเรียนที่ได้รับ

หลังจากปรับใช้โซลูชันนี้ ประสิทธิภาพการสร้างของโปรเจ็กต์ HagiCode ก็ดีขึ้นทันที

### ข้อมูลเกณฑ์มาตรฐานประสิทธิภาพ

เราทดสอบในสภาพแวดล้อมต่อไปนี้:
- ซีพียู: Intel Core i7-12700K (12 คอร์ / 20 เธรด)
- แรม: 32GB DDR4
- โหนด js: v18.17.0
- ระบบปฏิบัติการ: อูบุนตู 22.04

**การเปรียบเทียบผลลัพธ์**:
- **เธรดเดียว (ก่อนการปรับให้เหมาะสม)**: 118 วินาที
- **4 คน**: 55 วินาที (**ปรับปรุง 53%**)
- **8 คน**: 48 วินาที (**ปรับปรุง 60%**)
- **12 คน**: 45 วินาที (**ปรับปรุง 62%**)

อย่างที่คุณเห็น กำไรไม่ได้เชิงเส้น เมื่อจำนวนคนงานเกิน 8 การปรับปรุงก็เล็กลง ซึ่งส่วนใหญ่ถูกจำกัดโดยความสม่ำเสมอของการกระจายงานและปัญหาคอขวดของแบนด์วิธหน่วยความจำ

### ปัญหาทั่วไปและแนวทางแก้ไข

ในการใช้งานจริงของ HagiCode เรายังพบข้อผิดพลาดหลายประการ จึงมีข้อผิดพลาดดังนี้:

**คำถามที่ 1: เวลาในการสร้างไม่ได้ลดลงมากนักและช้าลงด้วยซ้ำใช่ไหม**
- **เหตุผล**: การสร้างผู้ปฏิบัติงานมีค่าใช้จ่ายของตัวเอง หรือมีการกำหนดค่าผู้ปฏิบัติงานมากเกินไป ทำให้เกิดการสลับบริบทบ่อยครั้ง
- **วิธีแก้ปัญหา**: เราขอแนะนำให้ตั้งค่าจำนวนผู้ปฏิบัติงานเป็น `CPU core count - 1`. ตรวจสอบด้วยว่าชิ้นเดียวมีขนาดใหญ่เป็นพิเศษหรือไม่ `> 5MB`. ไฟล์ "สัตว์ประหลาด" แบบนั้นจะกลายเป็นคอขวด ดังนั้นคุณอาจต้องปรับกลยุทธ์การแยกโค้ดให้เหมาะสม

**คำถามที่ 2: ผู้ปฏิบัติงานขัดข้องและทำให้เกิดความล้มเหลวในบางครั้ง**
- **เหตุผล**: รูปแบบไวยากรณ์ของโค้ดพิเศษบางรูปแบบอาจทำให้เกิดข้อผิดพลาดภายในภายใน obfuscator
- **แนวทางแก้ไข**: เราใช้ **กลไกการย่อยสลายอัตโนมัติ** เมื่อผู้ปฏิบัติงานถึงเกณฑ์ความล้มเหลว ปลั๊กอินจะถอยกลับไปเป็นโหมดเธรดเดียวโดยอัตโนมัติเพื่อให้แน่ใจว่าการสร้างจะไม่หยุดลง ในขณะเดียวกันก็บันทึกชื่อไฟล์ที่ทำให้เกิดข้อผิดพลาดเพื่อให้สามารถแก้ไขได้ในภายหลัง

**Q3: การใช้หน่วยความจำสูงเกินไป (OOM)?**
- **เหตุผล**: พนักงานแต่ละคนต้องการพื้นที่หน่วยความจำของตัวเองเพื่อโหลด obfuscator และแยกวิเคราะห์ AST
- **วิธีแก้ปัญหา**:
  - ลดจำนวนคนงาน
  - เพิ่มขีดจำกัดหน่วยความจำ Node.js: `NODE_OPTIONS="--max-old-space-size=4096" npm run build`.
  - ตรวจสอบให้แน่ใจว่าผู้ปฏิบัติงานไม่ได้เก็บการอ้างอิงที่ไม่จำเป็นไปยังวัตถุขนาดใหญ่

## บทสรุป

ด้วยการเปิดตัว Node.js Worker Threads เราประสบความสำเร็จในการลดเวลาการสร้างการผลิตของโปรเจ็กต์ HagiCode จาก 120 วินาทีเหลือประมาณ 45 วินาที ซึ่งช่วยปรับปรุงประสบการณ์การพัฒนาและประสิทธิภาพ CI/CD ได้อย่างมาก

หัวใจสำคัญของโซลูชันนี้คือ:
1. **แบ่งงานอย่างเหมาะสม**: ใช้ Vite chunks เป็นหน่วยคู่ขนาน
2. **ควบคุมทรัพยากร**: ใช้พูลผู้ปฏิบัติงานเพื่อหลีกเลี่ยงการใช้ทรัพยากรจนหมด
3. **การออกแบบเพื่อความทนทานต่อความเสียหาย**: กลไกการย่อยสลายอัตโนมัติช่วยให้มั่นใจได้ถึงความเสถียรของโครงสร้าง

หากคุณกำลังดิ้นรนกับประสิทธิภาพการสร้างส่วนหน้า หรือโปรเจ็กต์ของคุณก็มีการประมวลผลโค้ดจำนวนมากเช่นกัน โซลูชันนี้คุ้มค่าที่จะลองใช้ แน่นอน เราขอแนะนำให้ดู HagiCode โดยตรง ซึ่งรายละเอียดทางวิศวกรรมเหล่านี้ได้รวมเข้าด้วยกันแล้ว

หากบทความนี้ช่วยคุณได้ โปรดติดดาวให้กับเราบน GitHub หรือเข้าร่วมเวอร์ชันเบต้าสาธารณะแล้วลองใช้ดู
