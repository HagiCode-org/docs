## การสร้างความคุ้นเคยให้กับผู้ใช้ใหม่อย่างสง่างามในโครงการ React: แบบฝึกหัด driver.js ของ HagiCode

> เมื่อผู้ใช้เปิดผลิตภัณฑ์ของคุณเป็นครั้งแรก พวกเขารู้จริง ๆ ว่าจะเริ่มต้นอย่างไร ในบทความนี้ ฉันอยากจะพูดถึงวิธีที่เราใช้ driver.js สำหรับการเริ่มต้นใช้งานผู้ใช้ใหม่ในโปรเจ็กต์ HagiCode พิจารณาว่านี่เป็นตัวอย่างเล็กๆ น้อยๆ ที่นำไปใช้ได้จริงในการเริ่มต้นการสนทนา

## พื้นหลัง

คุณเคยเจอสถานการณ์นี้หรือไม่? ผู้ใช้ใหม่ลงทะเบียนผลิตภัณฑ์ของคุณ เปิดหน้าเพจ และดูหลงทางทันที พวกเขาสแกนไปรอบๆ ไม่แน่ใจว่าจะคลิกอะไรหรือจะทำอย่างไรต่อไป ในฐานะนักพัฒนา เรามักคิดว่าผู้ใช้จะ "สำรวจด้วยตัวเอง" เพราะท้ายที่สุดแล้ว ความอยากรู้อยากเห็นของมนุษย์นั้นไร้ขีดจำกัด แต่ความเป็นจริงนั้นแตกต่างออกไป ผู้ใช้ส่วนใหญ่จะออกไปอย่างเงียบ ๆ ภายในไม่กี่นาทีหากพวกเขาไม่สามารถหาจุดเริ่มต้นที่ถูกต้องได้ ราวกับว่าเรื่องราวเริ่มต้นอย่างกะทันหันและจบลงอย่างเป็นธรรมชาติ

การเริ่มต้นใช้งานผู้ใช้ใหม่เป็นวิธีสำคัญในการแก้ปัญหานี้ แต่การสร้างให้ดีนั้นไม่ใช่เรื่องง่ายขนาดนั้น ระบบการเริ่มต้นใช้งานที่ดีจะต้อง:

- ค้นหาองค์ประกอบของหน้าได้อย่างแม่นยำและเน้นองค์ประกอบเหล่านั้น
- รองรับโฟลว์การเริ่มต้นใช้งานแบบหลายขั้นตอน
- จำตัวเลือกของผู้ใช้ (สมบูรณ์หรือข้าม)
- หลีกเลี่ยงผลกระทบต่อประสิทธิภาพของเพจและการโต้ตอบตามปกติ
- รักษาโครงสร้างโค้ดให้ชัดเจนและดูแลรักษาง่าย

ในขณะที่สร้าง HagiCode เราก็พบกับความท้าทายเดียวกัน HagiCode เป็นตัวช่วยเขียนโค้ด AI และเวิร์กโฟลว์หลักของมันคือเวิร์กโฟลว์ OpenSpec ที่มีลักษณะดังนี้: "ผู้ใช้สร้างข้อเสนอ -> AI สร้างแผน -> ผู้ใช้ตรวจสอบ -> AI ดำเนินการ" สำหรับผู้ใช้ที่พบกับแนวคิดนี้เป็นครั้งแรก ขั้นตอนการทำงานถือเป็นเรื่องใหม่ ดังนั้นพวกเขาจึงจำเป็นต้องเริ่มต้นใช้งานอย่างมั่นคงเพื่อเริ่มต้นอย่างรวดเร็ว สิ่งใหม่ๆ ใช้เวลาในการทำความคุ้นเคยเสมอ

## เกี่ยวกับ ฮากิโค้ด

แนวทางที่แบ่งปันในบทความนี้มาจากประสบการณ์ตรงของเราใน [ฮากิโค้ด](https://hagicode.com) โครงการ. HagiCode คือผู้ช่วยเขียนโค้ด AI ที่ใช้ Claude ซึ่งช่วยให้นักพัฒนาทำงานเขียนโค้ดได้อย่างมีประสิทธิภาพมากขึ้นผ่านเวิร์กโฟลว์ OpenSpec คุณสามารถดูโค้ดโอเพ่นซอร์สของเราได้ที่ [GitHub](https://github.com/HagiCode-org/site).

## ทำไมเราถึงเลือก driver.js

ในระหว่างขั้นตอนการประเมินทางเทคนิค เราได้พิจารณาไลบรารีการเริ่มต้นใช้งานหลักๆ หลายแห่ง แต่ละคนมีจุดแข็งของตัวเอง:

- **Intro.js**: การปรับแต่งสไตล์ที่ทรงพลังแต่ค่อนข้างใหญ่นั้นค่อนข้างซับซ้อน
- **Shepherd.js**: API ที่ออกแบบมาอย่างดี แต่ "หนัก" เกินไปเล็กน้อยสำหรับกรณีการใช้งานของเรา
- **driver.js**: API น้ำหนักเบา กระชับ ใช้งานง่าย และทำงานได้ดีในระบบนิเวศ React

ในที่สุดเราก็เลือก driver.js ไม่มีเหตุผลที่น่าทึ่งเป็นพิเศษ ทางเลือกส่วนใหญ่มาจากการพิจารณาเหล่านี้:

1. **น้ำหนักเบา**: ไลบรารีหลักมีขนาดเล็กและไม่เพิ่มขนาดบันเดิลอย่างมีนัยสำคัญ
2. **Simple API**: การกำหนดค่ามีความชัดเจนและใช้งานง่าย ดังนั้นจึงง่ายต่อการรับ
3. **ยืดหยุ่น**: รองรับการวางตำแหน่ง สไตล์ และพฤติกรรมการโต้ตอบแบบกำหนดเอง
4. **การนำเข้าแบบไดนามิก**: สามารถโหลดได้ตามความต้องการโดยไม่กระทบต่อประสิทธิภาพของหน้าจอแรก

ด้วยการเลือกใช้เทคโนโลยี จึงไม่ค่อยมีคำตอบที่ดีที่สุดในระดับสากล โดยปกติแล้วจะมีเพียงตัวเลือกที่เหมาะสมที่สุดเท่านั้น

## การดำเนินการทางเทคนิค

### การกำหนดค่าหลัก

driver.js มีรูปแบบการกำหนดค่าที่ใช้งานง่ายมาก นี่คือการกำหนดค่าหลักที่เราใช้ในโครงการ HagiCode:

```typescript
import { driver } from 'driver.js';
import 'driver.js/dist/driver.css';

const newConversationDriver = driver({
  allowClose: true,           // Allow users to close the guide
  animate: true,              // Enable animations
  overlayClickBehavior: 'close', // Close the guide when the overlay is clicked
  disableActiveInteraction: false, // Keep elements interactive
  showProgress: false,        // Do not show the progress bar (we manage progress ourselves)
  steps: guideSteps           // Array of guide steps
});
```

เหตุผลเบื้องหลังการตั้งค่าเหล่านี้คือ:

- `allowClose: true` - เคารพตัวเลือกของผู้ใช้และอย่าบังคับให้พวกเขาจบคำแนะนำ
- `disableActiveInteraction: false` - บางขั้นตอนต้องมีการดำเนินการของผู้ใช้จริง เช่น การพิมพ์ข้อมูล ดังนั้นจึงไม่สามารถปิดใช้งานการโต้ตอบได้
- `overlayClickBehavior: 'close'` - ให้ผู้ใช้ออกจากระบบได้อย่างรวดเร็ว

### การจัดการของรัฐ

สถานะการเริ่มต้นใช้งานอย่างต่อเนื่องเป็นสิ่งสำคัญ เราไม่ต้องการเริ่มคำแนะนำใหม่ทุกครั้งที่รีเฟรชเพจ เนื่องจากจะทำให้น่ารำคาญอย่างรวดเร็ว การใช้ HagiCode `localStorage` เพื่อจัดการสถานะไกด์:

```typescript
export type GuideState = 'pending' | 'dismissed' | 'completed';

export interface UserGuideState {
  session: GuideState;
  detailGuides: Record<string, GuideState>;
}

// Read state
export const getUserGuideState = (): UserGuideState => {
  const state = localStorage.getItem('userGuideState');
  return state ? JSON.parse(state) : { session: 'pending', detailGuides: {} };
};

// Update state
export const setUserGuideState = (state: UserGuideState) => {
  localStorage.setItem('userGuideState', JSON.stringify(state));
};
```

เรากำหนดสามสถานะ:

- `pending`: คู่มือยังอยู่ในระหว่างดำเนินการ และผู้ใช้ยังดำเนินการไม่เสร็จสิ้นหรือข้ามไป
- `dismissed`: ผู้ใช้ปิดคำแนะนำในเชิงรุก
- `completed`: ผู้ใช้ทำครบทุกขั้นตอนแล้ว

สำหรับการเริ่มต้นใช้งานหน้ารายละเอียดข้อเสนอ เรายังสนับสนุนการติดตามสถานะที่ละเอียดยิ่งขึ้นผ่านทาง `detailGuides` เนื่องจากข้อเสนอหนึ่งสามารถผ่านได้หลายขั้นตอน - ร่าง ทบทวน และดำเนินการเสร็จสมบูรณ์ - และแต่ละขั้นตอนต้องการคำแนะนำที่แตกต่างกัน สถานะของสิ่งต่าง ๆ เปลี่ยนแปลงอยู่เสมอ และการเริ่มต้นใช้งานควรสะท้อนถึงสิ่งนั้น

### การเลือกองค์ประกอบเป้าหมาย

driver.js ใช้ตัวเลือก CSS เพื่อค้นหาองค์ประกอบเป้าหมาย HagiCode เป็นไปตามแบบแผนง่ายๆ: ใช้แบบกำหนดเอง `data-guide` คุณลักษณะเพื่อทำเครื่องหมายเป้าหมายการเริ่มต้นใช้งาน:

```typescript
const steps = [
  {
    element: '[data-guide="launch"]',
    popover: {
      title: 'Start a New Conversation',
      description: 'Click here to create a new conversation session...'
    }
  }
];
```

ในส่วนประกอบจะมีลักษณะดังนี้:

```typescript
<button data-guide="launch" onClick={handleLaunch}>
  New Conversation
</button>
```

ประโยชน์ของแนวทางนี้คือ:

- หลีกเลี่ยงความขัดแย้งกับชื่อคลาสสไตล์ธุรกิจ
- ความหมายที่ชัดเจน เพื่อให้คุณสามารถบอกได้ทันทีว่าองค์ประกอบนั้นเกี่ยวข้องกับการเริ่มต้นใช้งาน
- ง่ายต่อการจัดการและบำรุงรักษาอย่างสม่ำเสมอ

### การเพิ่มประสิทธิภาพการนำเข้าแบบไดนามิก

เนื่องจากคุณสมบัติการเริ่มต้นใช้งานจำเป็นในสถานการณ์เฉพาะเท่านั้น เช่น การเข้าชมครั้งแรกของผู้ใช้ เราจึงใช้การนำเข้าแบบไดนามิกเพื่อเพิ่มประสิทธิภาพการโหลดครั้งแรก:

```typescript
const initNewUserGuide = async () => {
  // Dynamically import driver.js
  const { driver } = await import('driver.js');
  await import('driver.js/dist/driver.css');

  // Initialize the guide
  const newConversationDriver = driver({
    // ...configuration
  });

  newConversationDriver.drive();
};
```

ด้วยวิธีนี้ driver.js และสไตล์ชีทจะถูกโหลดเมื่อจำเป็นเท่านั้น และไม่ส่งผลกระทบต่อประสิทธิภาพของหน้าจอแรก มีคนจำนวนไม่น้อยที่สนุกกับการรอคอยสิ่งที่พวกเขาไม่ต้องการด้วยซ้ำ

## การออกแบบโฟลว์การเริ่มต้นใช้งาน

HagiCode ใช้เส้นทางการเริ่มต้นใช้งานสองเส้นทางซึ่งครอบคลุมสถานการณ์หลักของผู้ใช้

### การเริ่มต้นเซสชัน (10 ขั้นตอน)

เส้นทางการเริ่มต้นใช้งานนี้ช่วยให้ผู้ใช้ดำเนินการขั้นตอนทั้งหมดตั้งแต่การสร้างการสนทนาไปจนถึงการส่งข้อเสนอที่สมบูรณ์ครั้งแรก:

1. **เปิดตัว** - เริ่มคำแนะนำและแนะนำปุ่ม "การสนทนาใหม่"
2. **เขียน** - แนะนำให้ผู้ใช้พิมพ์คำขอในช่องป้อนข้อมูล
3. **ส่ง** - แนะนำให้ผู้ใช้คลิกปุ่มส่ง
4. **proposal-launch-readme** - แนะนำผู้ใช้ในการสร้างข้อเสนอ README
5. **proposal-compose-readme** - แนะนำผู้ใช้ให้แก้ไขเนื้อหาคำขอ README
6. **proposal-submit-readme** - แนะนำผู้ใช้ในการส่งข้อเสนอ README
7. **proposal-launch-agents** - แนะนำผู้ใช้ในการสร้าง `AGENTS.md` ข้อเสนอ
8. **proposal-compose-agents** - แนะนำผู้ใช้ในการแก้ไข `AGENTS.md` คำขอ
9. **proposal-submit-agents** - แนะนำผู้ใช้ในการส่ง `AGENTS.md` ข้อเสนอ
10. **ข้อเสนอ-รอ** - อธิบายว่า AI กำลังประมวลผล และผู้ใช้ควรรอสักครู่

แนวคิดเบื้องหลังเส้นทางนี้คือการให้ผู้ใช้สัมผัสประสบการณ์เวิร์กโฟลว์หลักของ HagiCode ได้โดยตรงผ่านงานสร้างข้อเสนอจริงสองงาน งานหนึ่งงานสำหรับ `README` และอีกอันสำหรับ `AGENTS.md`. มีความแตกต่างอย่างมากระหว่างการได้ยินเกี่ยวกับเวิร์กโฟลว์และการดำเนินการด้วยตนเอง

### รายละเอียดข้อเสนอการเริ่มต้นใช้งาน (3 ขั้นตอน)

เมื่อผู้ใช้เข้าสู่หน้ารายละเอียดข้อเสนอ HagiCode จะทริกเกอร์คำแนะนำที่เกี่ยวข้องตามสถานะปัจจุบันของข้อเสนอ:

1. **การร่าง** (ขั้นตอนการร่าง) - แนะนำให้ผู้ใช้ตรวจสอบแผนที่สร้างโดย AI
2. **การทบทวน** (ขั้นตอนการทบทวน) - แนะนำให้ผู้ใช้ดำเนินการตามแผน
3. **การดำเนินการเสร็จสมบูรณ์** (ขั้นตอนที่เสร็จสมบูรณ์) - แนะนำให้ผู้ใช้เก็บถาวรแผน

คุณลักษณะที่กำหนดของคู่มือนี้คือ ขับเคลื่อนโดยรัฐ โดยจะตัดสินใจแบบไดนามิกว่าจะแสดงขั้นตอนการเริ่มต้นใช้งานใดตามสถานะจริงของข้อเสนอ สิ่งต่างๆ เปลี่ยนไป และการเริ่มใช้งานก็ควรเปลี่ยนแปลงตามไปด้วย

ภาพหน้าจอด้านล่างแสดงหน้ารายละเอียดข้อเสนอในสถานะการเริ่มต้นใช้งานในระหว่างขั้นตอนการร่าง:

![การเริ่มต้นใช้งานรายละเอียดข้อเสนอ: สร้างแผนก่อนระหว่างการร่าง](/article-images/new-user-guide-with-driverjs/1775022103729-2cb761ccff49.png)

ในขั้นตอนนี้ คู่มือจะเน้นความสนใจของผู้ใช้ไปที่การดำเนินการหลักในการสร้างแผน ดังนั้นพวกเขาจึงไม่สงสัยว่าจะต้องทำอะไรก่อนเมื่อเข้าสู่หน้ารายละเอียดเป็นครั้งแรก

### กลไกการลองแสดงผลองค์ประกอบอีกครั้ง

ในแอปพลิเคชัน React องค์ประกอบการเริ่มต้นใช้งานเป้าหมายอาจยังเรนเดอร์ไม่เสร็จ เช่น เนื่องจากข้อมูลอะซิงโครนัสยังคงโหลดอยู่ เพื่อจัดการกับสิ่งนี้ HagiCode จะใช้กลไกการลองใหม่:

```typescript
const waitForElement = (selector: string, maxRetries = 10, interval = 100) => {
  let retries = 0;
  return new Promise<HTMLElement>((resolve, reject) => {
    const checkElement = () => {
      const element = document.querySelector(selector) as HTMLElement;
      if (element) {
        resolve(element);
      } else if (retries < maxRetries) {
        retries++;
        setTimeout(checkElement, interval);
      } else {
        reject(new Error(`Element not found: ${selector}`));
      }
    };
    checkElement();
  });
};
```

เรียกใช้ฟังก์ชันนี้ก่อนที่จะเริ่มต้นคำแนะนำเพื่อให้แน่ใจว่าองค์ประกอบเป้าหมายมีอยู่แล้ว บางครั้งการรออีกสักหน่อยก็คุ้มค่า

## สรุปแนวทางปฏิบัติที่ดีที่สุด

จากประสบการณ์จริงของ HagiCode ต่อไปนี้เป็นแนวทางปฏิบัติที่ดีที่สุดบางประการ:

### 1. การเริ่มต้นใช้งานควรออกได้ง่าย

อย่าบังคับให้ผู้ใช้เริ่มต้นใช้งานให้เสร็จสิ้น ผู้ใช้บางคนเป็นนักสำรวจโดยธรรมชาติและชอบที่จะคิดหาสิ่งต่างๆ ด้วยตนเอง ระบุปุ่ม "ข้าม" ที่ชัดเจน จดจำตัวเลือกของพวกเขา และอย่าขัดจังหวะพวกเขาอีกในครั้งต่อไป

### 2. ทำให้เนื้อหาการเริ่มต้นใช้งานสั้นและคมชัด

แต่ละขั้นตอนการเริ่มต้นใช้งานควรมุ่งเน้นไปที่เป้าหมายเดียว:

- **หัวข้อ**: สั้นและชัดเจน ไม่ควรยาวเกิน 2-3 คำ
- **คำอธิบาย**: ตรงประเด็นและบอกผู้ใช้ว่านี่คืออะไร และเหตุใดจึงสำคัญ

หลีกเลี่ยงการอธิบายที่ยืดเยื้อ ความสนใจของผู้ใช้จะถูกจำกัดในระหว่างการเริ่มต้นใช้งาน และยิ่งคุณพูดมากเท่าใด พวกเขาก็จะมีโอกาสอ่านน้อยลงเท่านั้น

### 3. รักษาตัวเลือกให้คงที่

ใช้เครื่องหมายองค์ประกอบที่มั่นคงซึ่งไม่เปลี่ยนแปลงบ่อย ธรรมเนียม `data-guide` คุณลักษณะเป็นทางเลือกที่ดี หลีกเลี่ยงการขึ้นอยู่กับชื่อคลาสหรือโครงสร้าง DOM เนื่องจากสิ่งเหล่านี้เปลี่ยนแปลงได้ง่ายในระหว่างการรีแฟคเตอร์ รหัสเปลี่ยนแปลงตลอดเวลา แต่จุดยึดบางตัวควรคงที่เมื่อเป็นไปได้

### 4. ทดสอบการเริ่มต้นใช้งานของคุณ

HagiCode มีกรณีทดสอบที่สมบูรณ์สำหรับคุณสมบัติการเริ่มต้นใช้งาน:

```typescript
describe('NewUserConversationGuide', () => {
  it('should initialize guide state correctly', () => {
    const state = getUserGuideState();
    expect(state.session).toBe('pending');
  });

  it('should update guide state correctly', () => {
    setUserGuideState({ session: 'completed', detailGuides: {} });
    const state = getUserGuideState();
    expect(state.session).toBe('completed');
  });
});
```

การทดสอบช่วยให้มั่นใจได้ว่าการปรับโครงสร้างใหม่จะไม่ทำลายพฤติกรรมการเริ่มต้นใช้งานโดยไม่ได้ตั้งใจ ไม่มีใครอยากให้มีการเปลี่ยนแปลงโค้ดเล็กๆ น้อยๆ เพื่อสร้างความเสียหายให้กับฟังก์ชันการทำงานก่อนหน้านี้อย่างเงียบๆ

### 5. การเพิ่มประสิทธิภาพการทำงาน

- ใช้การนำเข้าแบบไดนามิกเพื่อโหลดไลบรารีการเริ่มต้นใช้งานแบบ Lazy Load
- หลีกเลี่ยงการเริ่มต้นตรรกะการเริ่มต้นใช้งานหลังจากที่ผู้ใช้ทำตามคำแนะนำเสร็จแล้ว
- พิจารณาผลกระทบด้านประสิทธิภาพของภาพเคลื่อนไหว และปิดใช้งานบนอุปกรณ์ระดับล่างหากจำเป็น

ประสิทธิภาพก็เหมือนกับหลายๆ สิ่งในชีวิต สมควรได้รับการจัดสรรงบประมาณอย่างรอบคอบ

## บทสรุป

การเริ่มต้นใช้งานผู้ใช้ใหม่เป็นส่วนสำคัญในการปรับปรุงประสบการณ์ผู้ใช้ผลิตภัณฑ์ ในโปรเจ็กต์ HagiCode เราใช้ driver.js เพื่อสร้างระบบการเริ่มต้นใช้งานที่สมบูรณ์ซึ่งครอบคลุมเวิร์กโฟลว์ทั้งหมดตั้งแต่การสร้างเซสชันไปจนถึงการดำเนินการตามข้อเสนอ

ประเด็นหลักที่เราหวังจะแบ่งปันผ่านบทความนี้คือ:

1. **ตัวเลือกทางเทคนิคควรตรงกับความต้องการที่แท้จริง**: driver.js ไม่ใช่ตัวเลือกที่ทรงพลังที่สุด แต่เหมาะสมที่สุดสำหรับเรา
2. **การจัดการของรัฐเป็นสิ่งสำคัญ**: การใช้งาน `localStorage` เพื่อคงสถานะการเริ่มต้นใช้งานและหลีกเลี่ยงการรบกวนผู้ใช้ซ้ำๆ
3. **การออกแบบการเริ่มต้นใช้งานควรมุ่งเน้น**: แต่ละขั้นตอนควรแก้ปัญหาเดียว และไม่มากไปกว่านี้
4. **โครงสร้างโค้ดควรมีความชัดเจน**: แยกการกำหนดค่าการเริ่มต้นใช้งาน การจัดการสถานะ และตรรกะ UI เพื่อให้การบำรุงรักษาง่ายขึ้น

หากคุณกำลังเพิ่มผู้ใช้ใหม่ในโปรเจ็กต์ของคุณเอง ฉันหวังว่าประสบการณ์เชิงปฏิบัติในบทความนี้จะช่วยคุณได้ ไม่มีอะไรลึกลับเป็นพิเศษเกี่ยวกับเทคโนโลยีประเภทนี้ พยายามต่อไป สรุปสิ่งที่คุณเรียนรู้ต่อไป และสิ่งต่างๆ จะค่อยๆ ง่ายขึ้น
