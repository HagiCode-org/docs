## อธิบายสถาปัตยกรรมการกระจายเดสก์ท็อปแบบไฮบริดของ HagiCode: P2P เร่งการดาวน์โหลดไฟล์ขนาดใหญ่ได้อย่างไร

> ฉันเก็บบทความนี้ไว้นานก่อนที่จะเขียนในที่สุดและฉันก็ยังไม่แน่ใจว่าจะอ่านได้ดีหรือไม่ การเขียนทางเทคนิคนั้นง่ายพอที่จะผลิต แต่ยากที่จะทำให้น่าสนใจอย่างแท้จริง ขอย้ำอีกครั้งว่า ฉันไม่ใช่ผู้เชี่ยวชาญด้านวรรณกรรมที่เก่งกาจ ดังนั้นฉันอาจจะเขียนคำอธิบายธรรมดาๆ นี้ขึ้นมาก็ได้

## พื้นหลัง

ทีมที่สร้างแอปพลิเคชันเดสก์ท็อปต่างก็ประสบปัญหาเดียวกันไม่ช้าก็เร็ว คุณจะแจกจ่ายไฟล์ขนาดใหญ่ได้อย่างไร

มันเป็นปัญหาที่น่าอึดอัดใจ การดาวน์โหลดโดยตรงผ่าน HTTP/HTTPS แบบเดิมยังคงสามารถรองรับได้เมื่อไฟล์มีขนาดเล็กและมีจำนวนผู้ใช้จำกัด แต่เวลาไม่ค่อยใจดี เมื่อโปรเจ็กต์เติบโตขึ้นเรื่อยๆ แพ็คเกจการติดตั้งก็เติบโตตามไปด้วย: แพ็คเกจ ZIP บนเดสก์ท็อป แพ็คเกจพกพา ไฟล์เก็บถาวรการปรับใช้เว็บ และอื่นๆ อีกมากมาย จากนั้นปัญหาก็เริ่มปรากฏ:

- **ความเร็วในการดาวน์โหลดถูกจำกัดโดยแบนด์วิดท์ต้นทาง**: ไม่ว่าเซิร์ฟเวอร์เดียวจะมีแบนด์วิดธ์เท่าใด ก็ยังคงประสบปัญหาเมื่อทุกคนดาวน์โหลดพร้อมกัน
- **เกือบจะไม่มีการสนับสนุนประวัติย่อ**: หากการดาวน์โหลด HTTP ถูกขัดจังหวะ คุณมักจะต้องเริ่มต้นใหม่ตั้งแต่ต้น ทำให้สิ้นเปลืองทั้งเวลาและแบนด์วิธ
- **เซิร์ฟเวอร์ต้นทางรับแรงกดดันทั้งหมด**: การรับส่งข้อมูลทั้งหมดไหลกลับไปยังเซิร์ฟเวอร์กลาง ต้นทุนแบนด์วิดท์เพิ่มขึ้นอย่างต่อเนื่อง และความสามารถในการขยายขนาดกลายเป็นปัญหาที่แท้จริง

โครงการ HagiCode Desktop ก็ไม่มีข้อยกเว้น เมื่อเราออกแบบระบบการกระจายสินค้า เราถามตัวเองอยู่เสมอว่า: เราจะแนะนำแนวทางการกระจายแบบไฮบริดโดยไม่ต้องเปลี่ยนแปลงวิธีที่มีอยู่ได้หรือไม่ `index.json` เครื่องบินควบคุมเหรอ? กล่าวอีกนัยหนึ่ง เราสามารถใช้ลักษณะแบบกระจายของเครือข่าย P2P เพื่อเร่งการดาวน์โหลดโดยที่ยังคงรักษา HTTP ต้นทางสำรองไว้ เพื่อให้ระบบยังคงใช้งานได้ในสภาพแวดล้อมที่จำกัด เช่น เครือข่ายองค์กร ได้หรือไม่

ผลกระทบของการตัดสินใจครั้งนั้นกลับกลายเป็นว่ามีขนาดใหญ่กว่าที่คุณคาดไว้ ให้เราเดินผ่านมันไปทีละขั้นตอน

## เกี่ยวกับ ฮากิโค้ด

แนวทางที่แบ่งปันในบทความนี้มาจากประสบการณ์จริงของเราใน [ฮากิโค้ด](https://hagicode.com) โครงการ HagiCode เป็นโครงการผู้ช่วยการเขียนโค้ด AI แบบโอเพ่นซอร์สที่มุ่งเน้นการช่วยให้ทีมพัฒนาปรับปรุงประสิทธิภาพทางวิศวกรรม โปรเจ็กต์ครอบคลุมระบบย่อยหลายระบบ รวมถึงฟรอนต์เอนด์ แบ็กเอนด์ ตัวเรียกใช้งานเดสก์ท็อป เอกสารประกอบ ไปป์ไลน์บิวด์ และการปรับใช้เซิร์ฟเวอร์

สถาปัตยกรรมการกระจายเดสก์ท็อปแบบไฮบริดนั้นเป็นโซลูชันประเภท HagiCode ที่ได้รับการปรับปรุงผ่านประสบการณ์การปฏิบัติงานจริงและการเพิ่มประสิทธิภาพซ้ำแล้วซ้ำเล่า หากการออกแบบนี้พิสูจน์ได้ว่ามีประโยชน์ บางทีก็อาจแสดงให้เห็นว่า HagiCode นั้นคุ้มค่าที่จะให้ความสนใจด้วย

พื้นที่เก็บข้อมูล GitHub ของโครงการคือ [HagiCode-org/site](https://github.com/HagiCode-org/site). หากคุณสนใจ อย่าลังเลที่จะให้ดาวแก่มันและบันทึกไว้ใช้ในภายหลัง

## ปรัชญาการออกแบบหลัก: P2P ก่อน, HTTP Fallback

โดยหัวใจสำคัญของโมเดลการกระจายแบบไฮบริดสามารถสรุปได้ในประโยคเดียว: **P2P ก่อน, HTTP ทางเลือก**

สิ่งสำคัญอยู่ที่คำว่า "ไฮบริด" นี่ไม่ใช่แค่การเพิ่ม BitTorrent และเรียกมันว่าวันเดียวเท่านั้น ประเด็นคือการทำให้วิธีการจัดส่งทั้งสองวิธีทำงานร่วมกันและเสริมซึ่งกันและกัน:

- **เครือข่าย P2P** ให้การเร่งความเร็วแบบกระจาย ยิ่งมีคนดาวน์โหลดมาก เพื่อนร่วมงานก็จะเข้าร่วมมากขึ้น และการถ่ายโอนก็จะเร็วขึ้นเท่านั้น
- **WebSeed/HTTP สำรอง** รับประกันความพร้อมใช้งาน ดังนั้นการดาวน์โหลดยังคงสามารถทำงานได้ในไฟร์วอลล์ขององค์กรและสภาพแวดล้อมเครือข่ายภายใน
- **ระนาบควบคุมยังคงเรียบง่าย** เราไม่เปลี่ยนตรรกะหลักของ `index.json`; เราเพิ่มช่องข้อมูลเมตาที่ไม่บังคับเพียงไม่กี่ช่องเท่านั้น

ประโยชน์ที่แท้จริงนั้นตรงไปตรงมา: ผู้ใช้รู้สึกว่า "การดาวน์โหลดเร็วขึ้น" ในขณะที่ทีมวิศวกรไม่จำเป็นต้องแบกรับความซับซ้อนเพิ่มเติมมากเกินไป ท้ายที่สุดแล้ว โปรโตคอล BT ก็ครบกำหนดแล้ว และไม่มีเหตุผลเพียงเล็กน้อยที่จะสร้างวงล้อขึ้นมาใหม่

## การออกแบบสถาปัตยกรรม

### ภาพรวมสถาปัตยกรรมแบบเลเยอร์

ให้เราเริ่มต้นด้วยแผนภาพสถาปัตยกรรมโดยรวมเพื่อสร้างแบบจำลองทางจิตระดับสูง:

```
┌─────────────────────────────────────┐
│     Renderer (UI layer)             │
├─────────────────────────────────────┤
│     IPC/Preload (bridge layer)      │
├─────────────────────────────────────┤
│   VersionManager (version manager)  │
├─────────────────────────────────────┤
│ HybridDownloadCoordinator (coord.)  │
│  ├── DistributionPolicyEvaluator    │
│  ├── DownloadEngineAdapter          │
│  ├── CacheRetentionManager          │
│  └── SHA256 Verifier                │
├─────────────────────────────────────┤
│   WebTorrent (download engine)      │
└─────────────────────────────────────┘
```

ตามแผนภาพที่แสดง ระบบใช้การออกแบบแบบเลเยอร์ เหตุผลในการแยกความรับผิดชอบอย่างชัดเจนนั้นง่ายมาก: **ความสามารถในการทดสอบและความสามารถในการเปลี่ยนทดแทน**

- **เลเยอร์ UI** มีหน้าที่แสดงความคืบหน้าในการดาวน์โหลดและการสลับการเร่งความเร็วการแชร์ มันคือพื้นผิว
- **ชั้นประสานงาน** คือแกนหลัก ประกอบด้วยการประเมินนโยบาย การปรับกลไก การจัดการแคช และการตรวจสอบความถูกต้อง
- **เลเยอร์ของกลไก** สรุปการดำเนินการดาวน์โหลดอย่างเป็นรูปธรรม ในขณะนี้มันใช้ WebTorrent

ชั้นเครื่องยนต์ถูกไล่ออกด้านหลัง `DownloadEngineAdapter` อินเตอร์เฟซ หากเราต้องการสลับในเอนจิ้น BT อื่นในภายหลัง หรือย้ายการใช้งานไปสู่กระบวนการช่วยเหลือเทียม นั่นก็จะง่ายขึ้นมาก

### การแยกระนาบควบคุมและระนาบข้อมูล

HagiCode Desktop เก็บไว้ `index.json` เป็นระนาบควบคุมเพียงอย่างเดียว และการออกแบบนั้นมีความสำคัญอย่างยิ่ง ส่วนควบคุมมีหน้าที่รับผิดชอบในการค้นหาเวอร์ชัน การเลือกช่องสัญญาณ และนโยบายแบบรวมศูนย์ ในขณะที่ส่วนข้อมูลคือจุดที่การถ่ายโอนไฟล์เกิดขึ้นจริง

ฟิลด์ใหม่ที่เพิ่มเข้ามา `index.json` เป็นทางเลือก:

```json
{
  "asset": {
    "torrentUrl": "https://cdn.example.com/app.torrent",
    "infoHash": "abc123...",
    "webSeeds": [
      "https://cdn.example.com/app.zip",
      "https://backup.example.com/app.zip"
    ],
    "sha256": "def456...",
    "directUrl": "https://cdn.example.com/app.zip"
  }
}
```

ฟิลด์ทั้งหมดนี้เป็นทางเลือก หากไม่มีหายไป ไคลเอ็นต์จะถอยกลับไปใช้โหมดดาวน์โหลด HTTP แบบเดิม ข้อดีของการออกแบบนี้คือ **ความเข้ากันได้แบบย้อนหลัง**: ไคลเอนต์รุ่นเก่าจะไม่ได้รับผลกระทบโดยสิ้นเชิง

### การตัดสินใจที่ขับเคลื่อนด้วยนโยบาย

ไม่ใช่ทุกไฟล์ที่จะคุ้มค่าแก่การแจกจ่ายผ่าน P2P

`DistributionPolicyEvaluator` มีหน้าที่ประเมินนโยบาย เฉพาะไฟล์ที่ตรงตามเงื่อนไขทั้งหมดต่อไปนี้เท่านั้นที่จะใช้การดาวน์โหลดแบบไฮบริด:

1. **ประเภทแหล่งที่มาต้องเป็นดัชนี HTTP**: การดาวน์โหลด GitHub โดยตรงหรือแหล่งที่มาของโฟลเดอร์ในเครื่องไม่ได้ใช้เส้นทางนี้
2. **ขนาดไฟล์ต้องมีอย่างน้อย 100 MB**: สำหรับไฟล์ขนาดเล็ก โอเวอร์เฮดของ P2P มีมากกว่าประโยชน์ที่ได้รับ
3. **ต้องมีข้อมูลเมตาไฮบริดที่สมบูรณ์**: `torrentUrl`, `webSeeds`และ `sha256` ล้วนจำเป็น
4. **เฉพาะแพ็คเกจเดสก์ท็อปล่าสุดและแพ็คเกจการปรับใช้เว็บเท่านั้นที่มีสิทธิ์**: เวอร์ชันในอดีตยังคงใช้เส้นทางการเผยแพร่แบบเดิม

```typescript
class DistributionPolicyEvaluator {
  evaluate(version: Version, settings: SharingAccelerationSettings): HybridDownloadPolicy {
    // Check source type
    if (version.sourceType !== 'http-index') {
      return { useHybrid: false, reason: 'not-http-index' };
    }

    // Check metadata completeness
    if (!version.hybrid) {
      return { useHybrid: false, reason: 'not-eligible' };
    }

    // Check whether the feature is enabled
    if (!settings.enabled) {
      return { useHybrid: false, reason: 'shared-disabled' };
    }

    // Check asset type (latest desktop/web packages only)
    if (!version.hybrid.isLatestDesktopAsset && !version.hybrid.isLatestWebAsset) {
      return { useHybrid: false, reason: 'latest-only' };
    }

    return { useHybrid: true, reason: 'shared-enabled' };
  }
}
```

สิ่งนี้ทำให้ระบบ **พฤติกรรมที่คาดการณ์ได้** ทั้งนักพัฒนาและผู้ใช้สามารถเข้าใจได้อย่างชัดเจนว่าไฟล์ใดจะใช้ P2P และไฟล์ใดจะไม่ใช้

## การดำเนินงานหลัก

### ประเภทระบบคำจำกัดความ

ให้เราเริ่มต้นด้วยคำจำกัดความของประเภท เนื่องจากเป็นรากฐานของทั้งระบบ

```typescript
// Hybrid distribution metadata
interface HybridDistributionMetadata {
  torrentUrl?: string;      // Torrent file URL
  infoHash?: string;        // InfoHash
  webSeeds: string[];       // WebSeed list
  sha256?: string;          // File hash
  directUrl?: string;       // HTTP direct link (for origin fallback)
  eligible: boolean;        // Whether hybrid distribution is applicable
  thresholdBytes: number;   // Threshold in bytes
  assetKind: VersionAssetKind;
  isLatestDesktopAsset: boolean;
  isLatestWebAsset: boolean;
}

// Sharing acceleration settings
interface SharingAccelerationSettings {
  enabled: boolean;           // Master switch
  uploadLimitMbps: number;    // Upload bandwidth limit
  cacheLimitGb: number;       // Cache limit
  retentionDays: number;      // Retention period
  hybridThresholdMb: number;  // Hybrid distribution threshold
  onboardingChoiceRecorded: boolean;
}

// Download progress
interface VersionDownloadProgress {
  current: number;
  total: number;
  percentage: number;
  stage: VersionInstallStage;  // queued, downloading, backfilling, verifying, extracting, completed, error
  mode: VersionDownloadMode;   // http-direct, shared-acceleration, source-fallback
  peers?: number;              // Number of connected peers
  p2pBytes?: number;           // Bytes received from P2P
  fallbackBytes?: number;      // Bytes received from fallback
  verified?: boolean;          // Whether verification has completed
}
```

เมื่อระบบประเภทชัดเจนแล้ว การนำไปใช้งานที่เหลือจะเป็นไปตามธรรมชาติ

### ผู้ประสานงานหลัก

`HybridDownloadCoordinator` จัดเตรียมขั้นตอนการดาวน์โหลดทั้งหมด โดยประสานการประเมินนโยบาย การทำงานของกลไก การตรวจสอบ SHA256 และการจัดการแคช

```typescript
class HybridDownloadCoordinator {
  async download(
    version: Version,
    cachePath: string,
    packageSource: PackageSource,
    onProgress?: DownloadProgressCallback,
  ): Promise<HybridDownloadResult> {
    // 1. Evaluate the policy: should hybrid download be used?
    const policy = this.policyEvaluator.evaluate(version, settings);

    // 2. Execute the download
    if (policy.useHybrid) {
      await this.engine.download(version, cachePath, settings, onProgress);
    } else {
      await packageSource.downloadPackage(version, cachePath, onProgress);
    }

    // 3. SHA256 verification (hard gate)
    const verified = await this.verify(version, cachePath, onProgress);
    if (!verified) {
      await this.cacheRetentionManager.discard(version.id, cachePath);
      throw new Error(`sha256 verification failed for ${version.id}`);
    }

    // 4. Mark as trusted cache and begin controlled seeding
    await this.cacheRetentionManager.markTrusted({
      versionId: version.id,
      cachePath,
      cacheSize,
    }, settings);

    return { cachePath, policy, verified };
  }
}
```

มีประเด็นสำคัญอย่างยิ่งประการหนึ่งที่นี่: **การตรวจสอบ SHA256 เป็นช่องทางที่ยาก** ไฟล์ที่ดาวน์โหลดจะต้องผ่านการยืนยันก่อนจึงจะเข้าสู่ขั้นตอนการติดตั้งได้ หากการตรวจสอบล้มเหลว แคชจะถูกยกเลิกเพื่อให้แน่ใจว่าไฟล์ที่ไม่ถูกต้องจะไม่ทำให้เกิดปัญหาในการติดตั้ง

### ดาวน์โหลด Engine Abstraction

`DownloadEngineAdapter` เป็นอินเทอร์เฟซเชิงนามธรรมที่กำหนดวิธีการที่ทุกเอ็นจิ้นต้องใช้:

```typescript
interface DownloadEngineAdapter {
  download(
    version: Version,
    destinationPath: string,
    settings: SharingAccelerationSettings,
    onProgress?: (progress: VersionDownloadProgress) => void,
  ): Promise<void>;

  stopAll(): Promise<void>;
}
```

การใช้งาน V1 นั้นอิงจาก WebTorrent และถูกรวมไว้ด้วย `InProcessTorrentEngineAdapter`:

```typescript
class InProcessTorrentEngineAdapter implements DownloadEngineAdapter {
  async download(...) {
    const client = this.getClient(settings);  // Apply upload rate limiting
    const torrent = client.add(torrentId, {
      path: path.dirname(destinationPath),
      destroyStoreOnDestroy: false,
      maxWebConns: 8,
    });

    // Add WebSeed sources
    torrent.on('ready', () => {
      for (const seed of hybrid.webSeeds) {
        torrent.addWebSeed(seed);
      }
      if (hybrid.directUrl) {
        torrent.addWebSeed(hybrid.directUrl);
      }
    });

    // Progress reporting - distinguish P2P from origin fallback
    torrent.on('download', () => {
      const hasP2PPeer = torrent.wires.some(w => w.type !== 'webSeed');
      const mode = hasP2PPeer ? 'shared-acceleration' : 'source-fallback';
      // ... report progress
    });
  }
}
```

การออกแบบเครื่องยนต์แบบเสียบปลั๊กได้ช่วยให้การปรับแต่งในอนาคตง่ายขึ้นมาก ตัวอย่างเช่น V2 สามารถรันเอ็นจิ้นในกระบวนการช่วยเหลือเพื่อหลีกเลี่ยงไม่ให้กระบวนการหลักหยุดทำงานหากเครื่องยนต์ขัดข้อง

### โหมดการรายงานความคืบหน้าที่แตกต่าง

ที่เลเยอร์ UI สิ่งที่ผู้ใช้ใส่ใจมากที่สุดนั้นเรียบง่าย: "ฉันกำลังดาวน์โหลดผ่าน P2P หรือผ่านทางเลือก HTTP หรือไม่" `InProcessTorrentEngineAdapter` กำหนดได้โดยการตรวจสอบประเภทภายใน `torrent.wires`:

```typescript
const hasP2PPeer = torrent.wires.some((wire) => wire.type !== 'webSeed');
const hasFallbackWire = torrent.wires.some((wire) => wire.type === 'webSeed');

const mode = hasP2PPeer ? 'shared-acceleration'
         : hasFallbackWire ? 'source-fallback'
         : 'shared-acceleration';

const stage = hasP2PPeer ? 'downloading'
           : hasFallbackWire ? 'backfilling'
           : 'downloading';
```

ตรรกะดูเรียบง่าย แต่เป็นส่วนสำคัญของประสบการณ์ผู้ใช้ ผู้ใช้สามารถเห็นได้อย่างชัดเจนว่าสถานะปัจจุบันคือ "การเร่งการแบ่งปัน" หรือ "การทดแทนต้นทาง" ซึ่งทำให้พฤติกรรมนี้เข้าใจได้ง่ายขึ้น

### การตรวจสอบสตรีมมิ่ง SHA256

การตรวจสอบความถูกต้องใช้ Node.js `crypto` โมดูลเพื่อคำนวณแฮชในลักษณะสตรีมมิ่ง ซึ่งหลีกเลี่ยงการโหลดไฟล์ทั้งหมดลงในหน่วยความจำ:

```typescript
private async computeSha256(filePath: string): Promise<string> {
  const hash = createHash('sha256');
  await new Promise<void>((resolve, reject) => {
    const stream = fs.createReadStream(filePath);
    stream.on('data', (chunk) => hash.update(chunk));
    stream.on('error', reject);
    stream.on('end', resolve);
  });
  return hash.digest('hex').toLowerCase();
}
```

การใช้งานนี้เป็นมิตรอย่างยิ่งสำหรับไฟล์ขนาดใหญ่ ลองนึกภาพการดาวน์โหลดแพ็คเกจการติดตั้งขนาด 2 GB แล้วลองโหลดสิ่งทั้งหมดลงในหน่วยความจำเพื่อตรวจสอบ การสตรีมมิ่งช่วยแก้ปัญหานั้นได้อย่างหมดจด

## การผลิต

แม้แต่การออกแบบทางเทคนิคที่ดีที่สุดก็ยังล้มเหลวหากประสบการณ์ผู้ใช้ไม่ดี HagiCode Desktop ทุ่มความพยายามพอสมควรในการสร้างความสามารถนี้

### ซ่อนคำศัพท์ BT

ผู้ใช้ส่วนใหญ่ไม่ทราบว่า BitTorrent หรือ InfoHash หมายถึงอะไร ดังนั้นในระดับผลิตภัณฑ์ เรานำเสนอคุณลักษณะโดยใช้วลี "การเร่งความเร็วในการแบ่งปัน":

- คุณลักษณะนี้เรียกว่า "การเร่งความเร็วในการแบ่งปัน" ไม่ใช่การดาวน์โหลด P2P
- การตั้งค่านี้เรียกว่า "ขีดจำกัดการอัปโหลด" ไม่ใช่การเริ่ม
- ป้ายกำกับความคืบหน้าระบุว่า "การทดแทนต้นทาง" ไม่ใช่การทดแทน WebSeed

สิ่งนี้จะช่วยลดภาระการรับรู้ของคำศัพท์และทำให้คุณสมบัตินี้ยอมรับได้ง่ายขึ้น

### เปิดใช้งานตามค่าเริ่มต้นในตัวช่วยสร้างการรันครั้งแรก

เมื่อผู้ใช้ใหม่เปิดแอปเดสก์ท็อปเป็นครั้งแรก พวกเขาจะเห็นหน้าวิซาร์ดที่แนะนำการเร่งความเร็วในการแชร์:

> เพื่อปรับปรุงความเร็วในการดาวน์โหลด เราจะแบ่งปันส่วนที่คุณดาวน์โหลดไปแล้วกับผู้ใช้รายอื่นในขณะที่การดาวน์โหลดของคุณเองอยู่ระหว่างดำเนินการ นี่เป็นทางเลือกโดยสมบูรณ์ และคุณสามารถปิดได้ตลอดเวลาในการตั้งค่า

โดยจะเปิดใช้งานตามค่าเริ่มต้น แต่ผู้ใช้จะได้รับวิธีที่ชัดเจนในการยกเลิก หากผู้ใช้ระดับองค์กรไม่ต้องการ ก็สามารถปิดการใช้งานได้ในระหว่างการเริ่มต้นใช้งาน

### พารามิเตอร์ที่ผู้ใช้ควบคุม

หน้าการตั้งค่าจะแสดงพารามิเตอร์ที่ปรับแต่งได้สามรายการ:

| พารามิเตอร์ | ค่าเริ่มต้น | คำอธิบาย |
|------|--------|------|
| ขีดจำกัดการอัปโหลด | 2 เมกะไบต์/วินาที | ป้องกันการใช้แบนด์วิธอัปสตรีมมากเกินไป |

## การตัดสินใจออกแบบที่สำคัญ

เมื่อมองย้อนกลับไปที่โซลูชันโดยรวม การตัดสินใจในการออกแบบหลายประการก็คุ้มค่าที่จะกล่าวถึง

### เครื่องยนต์ทำงานในกระบวนการหลัก (V1)

ทำไมไม่เริ่มด้วยรถเทียมข้างรถจักรยานยนต์หรือกระบวนการช่วยเหลือทันทีล่ะ เหตุผลง่ายๆ คือ **จัดส่งด่วน** การออกแบบระหว่างดำเนินการมีวงจรการพัฒนาที่สั้นกว่าและแก้ไขจุดบกพร่องได้ง่ายกว่า สิ่งสำคัญอันดับแรกคือการทำให้ฟีเจอร์ทำงาน จากนั้นจึงปรับปรุงความเสถียรในภายหลัง

แน่นอนว่าการตัดสินใจครั้งนี้มาพร้อมกับค่าใช้จ่าย หากเครื่องยนต์ขัดข้อง อาจส่งผลกระทบต่อกระบวนการหลักได้ เราลดความเสี่ยงดังกล่าวผ่านขอบเขตของอแด็ปเตอร์และการควบคุมการหมดเวลา และเรายังเปิดเส้นทางการโยกย้ายไว้เพื่อให้ V2 สามารถย้ายเข้าสู่กระบวนการที่แยกจากกันได้ง่ายขึ้น

### SHA256 เป็นการตรวจสอบความสมบูรณ์

เราใช้ SHA256 แทน MD5 หรือ CRC32 เนื่องจาก SHA256 มีความปลอดภัยมากกว่า ต้นทุนการชนกันของ MD5 และ CRC32 ต่ำเกินไป หากมีคนสร้างแพ็คเกจการติดตั้งปลอมโดยประสงค์ร้าย ผลที่ตามมาอาจรุนแรง SHA256 มีค่าใช้จ่ายในการคำนวณมากกว่า แต่ความปลอดภัยที่ได้รับก็คุ้มค่า

### เปิดใช้งานสำหรับแหล่งดัชนี HTTP เท่านั้น

สถานการณ์สมมติเช่นการดาวน์โหลด GitHub และแหล่งที่มาของโฟลเดอร์ในเครื่องไม่ได้ใช้การกระจายแบบไฮบริด นี่ไม่ใช่ข้อจำกัดทางเทคนิค เป็นเรื่องเกี่ยวกับ **การหลีกเลี่ยงความซับซ้อนที่ไม่จำเป็น** โปรโตคอล BT เพิ่มมูลค่าที่จำกัดภายในสถานการณ์เครือข่ายส่วนตัว และจะเพิ่มความซับซ้อนของโค้ดเท่านั้น

## หมายเหตุการปฏิบัติ

### การตั้งค่า การทำให้เป็นมาตรฐาน

ข้างใน `SharingAccelerationSettingsStore`ค่าตัวเลขทุกค่าต้องผ่านการตรวจสอบขอบเขตและการทำให้เป็นมาตรฐาน:

```typescript
private normalize(settings: SharingAccelerationSettings): SharingAccelerationSettings {
  return {
    enabled: Boolean(settings.enabled),
    uploadLimitMbps: this.clampNumber(settings.uploadLimitMbps, 1, 200, DEFAULT_SETTINGS.uploadLimitMbps),
    cacheLimitGb: this.clampNumber(settings.cacheLimitGb, 1, 500, DEFAULT_SETTINGS.cacheLimitGb),
    retentionDays: this.clampNumber(settings.retentionDays, 1, 90, DEFAULT_SETTINGS.retentionDays),
    hybridThresholdMb: DEFAULT_SETTINGS.hybridThresholdMb,  // Fixed value, not user-configurable
    onboardingChoiceRecorded: Boolean(settings.onboardingChoiceRecorded),
  };
}

private clampNumber(value: number, min: number, max: number, fallback: number): number {
  if (!Number.isFinite(value)) {
    return fallback;
  }
  return Math.min(max, Math.max(min, Math.round(value)));
}
```

วิธีนี้จะป้องกันไม่ให้ผู้ใช้แก้ไขไฟล์การกำหนดค่าเป็นค่าที่ไม่ถูกต้องด้วยตนเอง

### การล้างแคช LRU

`CacheRetentionManager.prune()` มีหน้าที่รับผิดชอบในการทำความสะอาดรายการแคชที่หมดอายุหรือมีขนาดใหญ่เกินไป กลยุทธ์การล้างข้อมูลใช้ LRU (ใช้ล่าสุดน้อยที่สุด):

```typescript
const records = [...this.listRecords()]
  .sort((left, right) =>
    new Date(left.lastUsedAt).getTime() - new Date(right.lastUsedAt).getTime()
  );

// When over the limit, evict the least recently used entries first
while (totalBytes > maxBytes && retainedEntries.length > 0) {
  const evicted = records.find((record) => retainedEntries.includes(record.versionId));
  retainedEntries.splice(retainedEntries.indexOf(evicted.versionId), 1);
  removedEntries.push(evicted.versionId);
  totalBytes -= evicted.cacheSize;
  await fs.rm(evicted.cachePath, { force: true });
}
```

ตรรกะนี้ช่วยให้แน่ใจว่าพื้นที่ดิสก์ถูกใช้อย่างมีประสิทธิภาพในขณะที่รักษาเวอร์ชันในอดีตที่ผู้ใช้อาจยังต้องการอยู่

### พฤติกรรมหยุดการเพาะทันที

เมื่อผู้ใช้ปิดการเร่งความเร็วการแบ่งปัน แอปจะต้องหยุดการเพาะและทำลายไคลเอนต์ทอร์เรนต์ทันที:

```typescript
async disableSharingAcceleration(): Promise<void> {
  this.settingsStore.updateSettings({ enabled: false });
  await this.cacheRetentionManager.stopAllSeeding();  // Stop seeding
  await this.engine.stopAll();  // Destroy the torrent client
}
```

หากผู้ใช้ปิดใช้งานคุณสมบัตินี้ ผลิตภัณฑ์ไม่ควรใช้ทรัพยากร P2P อีกต่อไป นั่นคือมารยาทพื้นฐานในการใช้ผลิตภัณฑ์

## ความเสี่ยงและการแลกเปลี่ยน

ไม่มีวิธีแก้ปัญหาที่สมบูรณ์แบบ และการกระจายแบบไฮบริดก็ไม่มีข้อยกเว้น นี่คือข้อเสียเปรียบหลัก:

**การแยกการชนนั้นอ่อนแอกว่ารถเทียมข้างรถจักรยานยนต์**: V1 ใช้เครื่องมือที่อยู่ในกระบวนการ ดังนั้นการชนของเครื่องยนต์อาจส่งผลต่อกระบวนการหลักได้ ขอบเขตของอะแดปเตอร์และการควบคุมการหมดเวลาช่วยลดความเสี่ยง แต่ไม่ใช่การแก้ไขขั้นพื้นฐาน V2 มีเส้นทางการย้ายที่วางแผนไว้ไปยังกระบวนการช่วยเหลือ

**เปิดใช้งานโดยค่าเริ่มต้นการใช้ทรัพยากร**: การตั้งค่าเริ่มต้นของการอัปโหลด 2 MB/s, แคช 10 GB และการเก็บรักษา 7 วันจะใช้ทรัพยากรเครื่องบางส่วน ความคาดหวังของผู้ใช้ได้รับการจัดการผ่านการคัดลอกการเริ่มต้นใช้งานและการตั้งค่าที่โปร่งใส

**ความเข้ากันได้ของเครือข่ายองค์กร**: ทางเลือก WebSeed/HTTPS อัตโนมัติช่วยรักษาความสามารถในการใช้งานในเครือข่ายองค์กร แต่สามารถลดการเร่งความเร็วที่ได้รับจาก P2P ได้ นี่เป็นการแลกเปลี่ยนโดยเจตนาโดยให้ความสำคัญกับความพร้อมใช้งาน

**ข้อมูลเมตาที่เข้ากันได้แบบย้อนหลัง**: ช่องใหม่ทั้งหมดจะระบุหรือไม่ก็ได้ หากหายไป ระบบจะย้อนกลับไปที่โหมด HTTP ลูกค้าเก่าไม่ได้รับผลกระทบใดๆ ทั้งสิ้น ทำให้การอัปเกรดเป็นไปอย่างราบรื่น

## บทสรุป

บทความนี้อธิบายเกี่ยวกับสถาปัตยกรรมการกระจายแบบไฮบริดที่ใช้ในโปรเจ็กต์ HagiCode Desktop ประเด็นสำคัญคือ:

1. **สถาปัตยกรรมแบบเลเยอร์**: ระนาบควบคุมและระนาบข้อมูลจะถูกแยกออกจากกัน และกลไกถูกแยกออกจากอินเทอร์เฟซแบบเสียบได้เพื่อการทดสอบและการขยายที่ง่ายขึ้น

2. **พฤติกรรมที่ขับเคลื่อนด้วยนโยบาย**: ไม่ใช่ทุกไฟล์ที่ใช้ P2P การกระจายแบบไฮบริดจะเปิดใช้งานสำหรับไฟล์ขนาดใหญ่ที่ตรงตามเงื่อนไขที่กำหนดเท่านั้น

3. **การตรวจสอบความสมบูรณ์**: SHA256 ทำหน้าที่เป็นฮาร์ดเกต และการตรวจสอบการสตรีมจะหลีกเลี่ยงแรงกดดันด้านหน่วยความจำ

4. **การนำเสนอที่มีประสิทธิภาพ**: คำศัพท์ BT ถูกซ่อนอยู่หลังวลี "การเร่งความเร็วในการแบ่งปัน" และฟีเจอร์นี้จะเปิดใช้งานตามค่าเริ่มต้นระหว่างการเริ่มต้นใช้งาน

5. **การควบคุมผู้ใช้**: ขีดจำกัดการอัปโหลด ขีดจำกัดแคช วันเก็บรักษา และพารามิเตอร์อื่นๆ ยังคงผู้ใช้สามารถปรับเปลี่ยนได้

สถาปัตยกรรมนี้ได้ถูกนำไปใช้ในโครงการ HagiCode Desktop แล้ว หากคุณลองใช้ เรายินดีรับฟังความคิดเห็นของคุณหลังการติดตั้งและการใช้งานจริง
