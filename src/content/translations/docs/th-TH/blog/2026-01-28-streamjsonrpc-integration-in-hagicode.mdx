## การบูรณาการเชิงลึกและการใช้งาน StreamJsonRpc ใน HagiCode ในทางปฏิบัติ

> บทความนี้อธิบายรายละเอียดว่าโปรเจ็กต์ HagiCode (เดิมเรียกว่า PCode) ประสบความสำเร็จในการผสานรวมไลบรารีการสื่อสาร StreamJsonRpc ของ Microsoft เพื่อแทนที่การใช้งาน JSON-RPC แบบกำหนดเองดั้งเดิมได้อย่างไร ขณะเดียวกันก็แก้ไขจุดบกพร่องทางเทคนิคและความท้าทายทางสถาปัตยกรรมที่พบในระหว่างกระบวนการผสานรวม

## พื้นหลัง

StreamJsonRpc คือไลบรารีการสื่อสาร JSON-RPC ที่ได้รับการดูแลอย่างเป็นทางการของ Microsoft สำหรับ .NET และ TypeScript ซึ่งเป็นที่รู้จักในเรื่องความปลอดภัยประเภทที่แข็งแกร่ง การสร้างพร็อกซีอัตโนมัติ และกลไกการจัดการข้อยกเว้นที่สมบูรณ์ ในโครงการ HagiCode ทีมงานตัดสินใจรวม StreamJsonRpc เพื่อสื่อสารกับเครื่องมือ AI ภายนอก เช่น ifflow CLI และ OpenCode CLI ผ่าน ACP (Agent Communication Protocol) ขณะเดียวกันก็ขจัดค่าบำรุงรักษาและจุดบกพร่องที่อาจเกิดขึ้นจากการใช้งาน JSON-RPC แบบกำหนดเองก่อนหน้านี้ อย่างไรก็ตาม กระบวนการบูรณาการพบกับความท้าทายเฉพาะสำหรับการสตรีม JSON-RPC โดยเฉพาะอย่างยิ่งเมื่อจัดการการเชื่อมโยงเป้าหมายพร็อกซีและการจดจำพารามิเตอร์ทั่วไป

เพื่อจัดการกับปัญหาเหล่านี้ เราได้ตัดสินใจอย่างกล้าหาญ: สร้างระบบการประกอบทั้งหมดใหม่ตั้งแต่ต้นจนจบ ผลกระทบของการตัดสินใจนั้นอาจยิ่งใหญ่กว่าที่คุณคาดไว้ และเราจะอธิบายโดยละเอียดในไม่ช้า

## การวิเคราะห์

โครงการปัจจุบันอยู่ในขั้นตอนสำคัญของการบูรณาการโปรโตคอล ACP และกำลังเผชิญกับปัญหาทางเทคนิคและความท้าทายทางสถาปัตยกรรมดังต่อไปนี้:

### 1. ข้อจำกัดของการใช้งานแบบกำหนดเอง
การใช้งาน JSON-RPC ดั้งเดิมนั้นอยู่ใน `src/HagiCode.ClaudeHelper/AcpImp/` และรวมถึงส่วนประกอบต่างๆ เช่น `JsonRpcEndpoint` และ `ClientSideConnection`. การดูแลรักษาโค้ดเบสที่กำหนดเองนี้มีค่าใช้จ่ายสูงและขาดความสามารถขั้นสูงของไลบรารีที่สมบูรณ์ เช่น การรายงานความคืบหน้าและการสนับสนุนการยกเลิก

### 2. อุปสรรคในการบูรณาการ StreamJsonRpc
เมื่อพยายามโยกย้ายสิ่งที่มีอยู่ `CallbackProxyTarget` รูปแบบเป็น StreamJsonRpc เราพบว่า `_rpc.AddLocalRpcTarget(target)` วิธีการไม่สามารถจดจำเป้าหมายที่สร้างผ่านรูปแบบพร็อกซีได้ โดยเฉพาะอย่างยิ่ง StreamJsonRpc ไม่สามารถแบ่งคุณสมบัติของประเภททั่วไปได้โดยอัตโนมัติ `T` ในพารามิเตอร์วิธี RPC ทำให้ฝั่งเซิร์ฟเวอร์ล้มเหลวเมื่อประมวลผลการเรียกวิธีการเริ่มต้นโดยไคลเอนต์

### 3. การแบ่งชั้นทางสถาปัตยกรรมที่สับสน
ที่มีอยู่ `ClientSideConnection` ผสมผสานเลเยอร์การขนส่ง (WebSocket/Stdio), เลเยอร์โปรโตคอล (JSON-RPC) และเลเยอร์ธุรกิจ (อินเทอร์เฟซตัวแทน ACP) นำไปสู่ความรับผิดชอบที่ไม่ชัดเจน นอกจากนี้ยังทนทุกข์ทรมานจากการเชื่อมโยงวิธีการที่ขาดหายไป `AcpAgentCallbackRpcAdapter`.

### 4. บันทึกที่หายไป
เลเยอร์การขนส่ง WebSocket ขาดการบันทึกเนื้อหา JSON แบบดิบ ทำให้ยากต่อการระบุในระหว่างการดีบัก RPC ว่าปัญหาเกิดจากการทำให้เป็นอนุกรมหรือจากเครือข่าย

## โซลูชั่น

เพื่อแก้ไขปัญหาข้างต้น เราได้นำโซลูชันที่เป็นระบบต่อไปนี้มาใช้ โดยปรับให้เหมาะสมจากสามมิติ: การปรับโครงสร้างสถาปัตยกรรมใหม่ การรวมไลบรารี และการแก้ไขข้อบกพร่องที่ได้รับการปรับปรุง

### 1. ย้ายไปยัง StreamJsonRpc โดยสมบูรณ์

#### ลบรหัสเดิม
ลบ `JsonRpcEndpoint.cs`, `AgentSideConnection.cs`และตัวแปลงซีเรียลไลซ์แบบกำหนดเองที่เกี่ยวข้อง เช่น `JsonRpcMessageJsonConverter`.

#### บูรณาการห้องสมุดอย่างเป็นทางการ
แนะนำตัว `StreamJsonRpc` แพ็คเกจ NuGet และใช้มัน `JsonRpc` คลาสเพื่อจัดการตรรกะการสื่อสารหลัก

#### บทคัดย่อเลเยอร์การขนส่ง
กำหนด `IAcpTransport` อินเทอร์เฟซเพื่อจัดการทั้งสองอย่าง `WebSocket` และ `Stdio` โหมดการขนส่งในลักษณะที่เป็นหนึ่งเดียว ทำให้มั่นใจได้ว่าชั้นโปรโตคอลจะแยกออกจากชั้นการขนส่ง

```csharp
// Definition of the IAcpTransport interface
public interface IAcpTransport
{
    Task SendAsync(string message, CancellationToken cancellationToken = default);
    Task<string> ReceiveAsync(CancellationToken cancellationToken = default);
    Task CloseAsync(CancellationToken cancellationToken = default);
}

// WebSocket transport implementation
public class WebSocketTransport : IAcpTransport
{
    private readonly WebSocket _webSocket;

    public WebSocketTransport(WebSocket webSocket)
    {
        _webSocket = webSocket;
    }

    // Implement send and receive methods
    // ...
}

// Stdio transport implementation
public class StdioTransport : IAcpTransport
{
    private readonly StreamReader _reader;
    private readonly StreamWriter _writer;

    public StdioTransport(StreamReader reader, StreamWriter writer)
    {
        _reader = reader;
        _writer = writer;
    }

    // Implement send and receive methods
    // ...
}
```

### 2. แก้ไขปัญหาการรับรู้เป้าหมายพร็อกซี

#### วิเคราะห์ `CallbackProxyTarget`
ตรวจสอบตรรกะการสร้างพร็อกซีแบบไดนามิกที่มีอยู่ และระบุสาเหตุหลักของสาเหตุที่ StreamJsonRpc ไม่สามารถจดจำได้ ในกรณีส่วนใหญ่ สิ่งนี้เกิดขึ้นเนื่องจากออบเจ็กต์พร็อกซีไม่เปิดเผยลายเซ็นวิธีการจริงต่อสาธารณะ หรือใช้ประเภทพารามิเตอร์ที่ StreamJsonRpc ไม่รองรับ

#### การส่งผ่านพารามิเตอร์รีแฟคเตอร์
แยกคุณสมบัติทั่วไปออกเป็นพารามิเตอร์วิธี RPC ที่ชัดเจน แทนที่จะอาศัยคุณสมบัติไดนามิก ให้กำหนด DTO คำขอ/การตอบสนองที่เป็นรูปธรรม (ออบเจ็กต์การถ่ายโอนข้อมูล) เพื่อให้ StreamJsonRpc สามารถจดจำลายเซ็นวิธีการได้อย่างถูกต้องผ่านการสะท้อนกลับ

```csharp
// Original generic property approach
public class CallbackProxyTarget<T>
{
    public Func<T, Task> Callback { get; set; }
}

// Refactored concrete method approach
public class ReadTextFileRequest
{
    public string FilePath { get; set; }
}

public class ReadTextFileResponse
{
    public string Content { get; set; }
}

public interface IAcpAgentCallback
{
    Task<ReadTextFileResponse> ReadTextFileAsync(ReadTextFileRequest request);
    // Other methods...
}
```

#### ใช้ `Attach` แทน `AddLocalRpcTarget`
ในบางสถานการณ์ที่ซับซ้อน การพร็อกซีด้วยตนเอง `JsonRpc` วัตถุและการจัดการ `RpcConnection` สามารถยืดหยุ่นได้มากกว่าการเพิ่มเป้าหมายท้องถิ่นโดยตรง

### 3. ใช้วิธีการผูกและการบันทึกที่แข็งแกร่งยิ่งขึ้น

#### ดำเนินการ `AcpAgentCallbackRpcAdapter`
ตรวจสอบให้แน่ใจว่าคอมโพเนนต์นี้ใช้อินเทอร์เฟซพร็อกซี StreamJsonRpc และวิธีการแมปที่กำหนดโดยโปรโตคอล ACP อย่างชัดเจน เช่น `ReadTextFileAsync`ไปยังตัวจัดการการเรียกกลับ StreamJsonRpc

#### บูรณาการการบันทึก
สกัดกั้นและบันทึกข้อความดิบของคำขอและการตอบกลับ JSON-RPC ในไปป์ไลน์การประมวลผลข้อความ WebSocket หรือ Stdio ใช้ `ILogger` เพื่อส่งออกเพย์โหลดดิบก่อนการแยกวิเคราะห์และหลังการทำให้เป็นอนุกรม เพื่อให้สามารถวินิจฉัยปัญหาการจัดรูปแบบได้ง่ายขึ้น

```csharp
// Transport wrapper with enhanced logging
public class LoggingAcpTransport : IAcpTransport
{
    private readonly IAcpTransport _innerTransport;
    private readonly ILogger<LoggingAcpTransport> _logger;

    public LoggingAcpTransport(IAcpTransport innerTransport, ILogger<LoggingAcpTransport> logger)
    {
        _innerTransport = innerTransport;
        _logger = logger;
    }

    public async Task SendAsync(string message, CancellationToken cancellationToken = default)
    {
        _logger.LogTrace("Sending message: {Message}", message);
        await _innerTransport.SendAsync(message, cancellationToken);
    }

    public async Task<string> ReceiveAsync(CancellationToken cancellationToken = default)
    {
        var message = await _innerTransport.ReceiveAsync(cancellationToken);
        _logger.LogTrace("Received message: {Message}", message);
        return message;
    }

    public async Task CloseAsync(CancellationToken cancellationToken = default)
    {
        _logger.LogDebug("Closing connection");
        await _innerTransport.CloseAsync(cancellationToken);
    }
}
```

### 4. การแบ่งชั้นทางสถาปัตยกรรมใหม่

#### ชั้นการขนส่ง (`AcpRpcClient`)
รวมการเชื่อมต่อ StreamJsonRpc และทำให้รับผิดชอบ `InvokeAsync` การจัดการวงจรการโทรและการเชื่อมต่อ

```csharp
public class AcpRpcClient : IDisposable
{
    private readonly JsonRpc _rpc;
    private readonly IAcpTransport _transport;

    public AcpRpcClient(IAcpTransport transport)
    {
        _transport = transport;
        _rpc = new JsonRpc(new StreamRpcTransport(transport));
        _rpc.StartListening();
    }

    public async Task<TResponse> InvokeAsync<TResponse>(string methodName, object parameters)
    {
        return await _rpc.InvokeAsync<TResponse>(methodName, parameters);
    }

    public void Dispose()
    {
        _rpc.Dispose();
        _transport.Dispose();
    }

    // StreamRpcTransport is the StreamJsonRpc adapter for IAcpTransport
    private class StreamRpcTransport : IDuplexPipe
    {
        // Implement the IDuplexPipe interface
        // ...
    }
}
```

#### เลเยอร์โปรโตคอล (`IAcpAgentClient` / `IAcpAgentCallback`)
กำหนดอินเทอร์เฟซระหว่างลูกค้ากับตัวแทนและตัวแทนกับลูกค้าที่ชัดเจน ลบรูปแบบโรงงานแบบวน `Func<IAcpAgent, IAcpClient>` และแทนที่ด้วยการฉีดการพึ่งพาหรือการลงทะเบียนการโทรกลับโดยตรง

## การปฏิบัติ

ตามแนวทางปฏิบัติที่ดีที่สุดของ StreamJsonRpc และประสบการณ์ของโปรเจ็กต์ คำแนะนำที่สำคัญสำหรับการนำไปปฏิบัติมีดังนี้

### 1. DTO ที่พิมพ์อย่างเข้มงวดนั้นดีกว่าวัตถุไดนามิก
ข้อได้เปรียบหลักของ StreamJsonRpc อยู่ที่การพิมพ์ที่แข็งแกร่ง อย่าใช้ `dynamic` หรือ `JObject` เพื่อส่งผ่านพารามิเตอร์ ให้กำหนดคลาส C# POCO ที่ชัดเจนเป็นพารามิเตอร์สำหรับวิธี RPC แต่ละวิธีแทน สิ่งนี้ไม่เพียงแต่แก้ปัญหาการรู้จำเป้าหมายของพร็อกซีเท่านั้น แต่ยังตรวจจับข้อผิดพลาดของประเภทในขณะคอมไพล์อีกด้วย

ตัวอย่าง: แทนที่คุณสมบัติทั่วไปใน `CallbackProxyTarget` ด้วยคลาสที่เป็นรูปธรรมเช่น `ReadTextFileRequest` และ `WriteTextFileRequest`.

### 2. ประกาศชื่อวิธีการอย่างชัดเจน
ใช้ `[JsonRpcMethod]` คุณลักษณะเพื่อระบุชื่อวิธีการ RPC อย่างชัดเจนแทนที่จะอาศัยการแมปชื่อวิธีการเริ่มต้น วิธีนี้จะช่วยป้องกันความล้มเหลวในการเรียกใช้ที่เกิดจากความแตกต่างในรูปแบบการตั้งชื่อ เช่น PascalCase กับ CamelCase

```csharp
public interface IAcpAgentCallback
{
    [JsonRpcMethod("readTextFile")]
    Task<ReadTextFileResponse> ReadTextFileAsync(ReadTextFileRequest request);
    
    [JsonRpcMethod("writeTextFile")]
    Task WriteTextFileAsync(WriteTextFileRequest request);
}
```

### 3. ใช้ประโยชน์จากการเรียกกลับสถานะการเชื่อมต่อ
StreamJsonRpc ให้ `JsonRpc.ConnectionLost` เหตุการณ์ คุณควรรับฟังเหตุการณ์นี้อย่างแน่นอนเพื่อจัดการกับการออกจากกระบวนการที่ไม่คาดคิดหรือการตัดการเชื่อมต่อของเครือข่าย ซึ่งจะทันเวลามากกว่าการพึ่งพาการตรวจจับความล้มเหลวของ Orleans Grain เท่านั้น

```csharp
_rpc.ConnectionLost += (sender, e) =>
{
    _logger.LogError("RPC connection lost: {Reason}", e.ToString());
    // Handle reconnection logic or notify the user
};
```

### 4. บันทึกบันทึกตามเลเยอร์
- **ระดับการติดตาม**: บันทึกเพย์โหลดคำขอ/การตอบกลับ JSON แบบดิบทั้งหมด
- **ระดับการแก้ไขข้อบกพร่อง**: บันทึกการเรียกใช้เมธอดสแต็กและข้อมูลสรุปพารามิเตอร์
- **หมายเหตุ**: ตรวจสอบให้แน่ใจว่าบันทึกไม่มีโทเค็นการให้สิทธิ์ที่ละเอียดอ่อนหรือเนื้อหาไฟล์ขนาดใหญ่ที่เข้ารหัส Base64

### 5. จัดการกับลักษณะพิเศษของการขนส่งแบบสตรีมมิ่ง
StreamJsonRpc รองรับโดยกำเนิด `IAsyncEnumerable`. เมื่อใช้การตอบกลับพร้อมต์สตรีมมิ่งสำหรับ ACP ให้ใช้ `IAsyncEnumerable` โดยตรงแทนตรรกะการแบ่งหน้าแบบกำหนดเอง ซึ่งช่วยลดความซับซ้อนของจำนวนโค้ดที่จำเป็นสำหรับการประมวลผลการสตรีมได้อย่างมาก

```csharp
public interface IAcpAgentCallback
{
    [JsonRpcMethod("streamText")]
    IAsyncEnumerable<string> StreamTextAsync(StreamTextRequest request);
}
```

### 6. รูปแบบอะแดปเตอร์
เก็บ `ACPSession` และ `ClientSideConnection` แยกจากกัน `ACPSession` ควรมุ่งเน้นไปที่การจัดการของรัฐและตรรกะทางธุรกิจของ Orleans เช่น การจัดคิวข้อความ และควรใช้ออบเจ็กต์การเชื่อมต่อ StreamJsonRpc ผ่านการเรียบเรียงมากกว่าการสืบทอด

## บทสรุป

ด้วยการบูรณาการ StreamJsonRpc อย่างครอบคลุม โปรเจ็กต์ HagiCode สามารถจัดการกับค่าบำรุงรักษาที่สูง ข้อจำกัดด้านการทำงาน และความสับสนในการแบ่งชั้นทางสถาปัตยกรรมของการใช้งานแบบกำหนดเองดั้งเดิมได้สำเร็จ การปรับปรุงที่สำคัญ ได้แก่ :

1. แทนที่คุณสมบัติไดนามิกด้วย DTO ที่พิมพ์อย่างแน่นหนา ปรับปรุงการบำรุงรักษาและความน่าเชื่อถือ
2. การใช้นามธรรมในชั้นการขนส่งและการแยกชั้นโปรโตคอล ปรับปรุงความชัดเจนทางสถาปัตยกรรม
3. เสริมสร้างความสามารถในการบันทึกข้อมูลเพื่อให้วินิจฉัยปัญหาการสื่อสารได้ง่ายขึ้น
4. ขอแนะนำการสนับสนุนการสตรีมเพื่อทำให้การใช้งานการสตรีมง่ายขึ้น

การปรับปรุงเหล่านี้ทำให้ HagiCode มีรากฐานการสื่อสารที่เสถียรและมีประสิทธิภาพมากขึ้น ทำให้สามารถโต้ตอบกับเครื่องมือ AI ภายนอกได้ดีขึ้น และวางรากฐานที่มั่นคงสำหรับการขยายคุณสมบัติในอนาคต
