## ลาก่อนสคริปต์นรก: ทำไมเราถึงเลือก C# สำหรับระบบ Build สมัยใหม่

> เจาะลึกถึงวิธีที่โปรเจ็กต์ HagiCode ใช้ Nuke เพื่อสร้างไปป์ไลน์การสร้างอัตโนมัติที่ปลอดภัยต่อประเภท ข้ามแพลตฟอร์ม และขยายได้สูง จัดการปัญหาในการบำรุงรักษาของสคริปต์บิลด์แบบดั้งเดิมได้อย่างสมบูรณ์

## พื้นหลัง

ตลอดการเดินทางอันยาวนานของการพัฒนาซอฟต์แวร์ คำว่า "สร้าง" มีแนวโน้มที่จะสร้างแรงบันดาลใจทั้งความรักและความคับข้องใจ ที่รัก เพราะว่าเพียงคลิกเดียว โค้ดก็จะกลายเป็นผลิตภัณฑ์ และนั่นเป็นหนึ่งในช่วงเวลาที่น่าพอใจที่สุดในการเขียนโปรแกรม ความหงุดหงิด เนื่องจากการรักษากองสคริปต์บิลด์ที่ยุ่งวุ่นวายอาจเป็นฝันร้ายได้

ในหลายโปรเจ็กต์ เราคุ้นเคยกับการเขียนสคริปต์ใน Python หรือใช้ไฟล์การกำหนดค่า XML ลองนึกภาพความกลัวที่จะถูกปกครองโดย `<property>` บล็อก แต่เมื่อความซับซ้อนของโปรเจ็กต์เพิ่มมากขึ้น โดยเฉพาะอย่างยิ่งในโปรเจ็กต์อย่าง HagiCode ที่ครอบคลุมงานฟรอนต์เอนด์และแบ็กเอนด์ หลายแพลตฟอร์ม และหลายภาษา วิธีสร้างแบบดั้งเดิมเริ่มแสดงขีดจำกัด ตรรกะของสคริปต์กระจัดกระจาย ไม่มีการตรวจสอบประเภท การสนับสนุน IDE ที่อ่อนแอ... แต่ละปัญหาเหล่านี้กลายเป็นข้อผิดพลาดเล็กๆ ที่ทำให้ทีมสะดุดเป็นครั้งคราว

เพื่อแก้ไขปัญหาเหล่านั้น โปรเจ็กต์ HagiCode ได้เปิดตัว **Nuke** ซึ่งเป็นระบบบิวด์สมัยใหม่ที่ใช้ C# มันไม่ใช่แค่เครื่องมือ มันเป็นวิธีคิดที่แตกต่างเกี่ยวกับเวิร์กโฟลว์การสร้าง วันนี้เราจะมาดูว่าทำไมเราถึงเลือกมัน และมันช่วยปรับปรุงประสบการณ์การพัฒนาของเราอย่างมากได้อย่างไร

## เกี่ยวกับ ฮากิโค้ด

> การแนะนำโดยย่อเกี่ยวกับสิ่งที่เรากำลังสร้าง

เรากำลังสร้าง **HagiCode** ซึ่งเป็นผู้ช่วยเขียนโค้ดที่ขับเคลื่อนด้วย AI ซึ่งจะทำให้ประสบการณ์การพัฒนาฉลาดขึ้น สะดวกยิ่งขึ้น และสนุกสนานยิ่งขึ้น

**อัจฉริยะ**: ความช่วยเหลือจาก AI มีให้ตลอดกระบวนการทั้งหมด ตั้งแต่แนวคิดไปจนถึงการเขียนโค้ด ซึ่งเพิ่มประสิทธิภาพในการพัฒนาเป็นทวีคูณ **สะดวก**: การทำงานพร้อมกันแบบมัลติเธรดทำให้ใช้ทรัพยากรได้อย่างเต็มที่ และทำให้ขั้นตอนการทำงานราบรื่น **ความสนุก**: Gamification และระบบความสำเร็จทำให้การเขียนโค้ดน่าเบื่อน้อยลงและคุ้มค่ามากขึ้น

โครงการกำลังพัฒนาอย่างรวดเร็ว หากคุณสนใจในการเขียนทางเทคนิค การจัดการความรู้ หรือการพัฒนาโดยใช้ AI เข้าไปเยี่ยมชมได้เลย [GitHub](https://github.com/HagiCode-org/site).

## การวิเคราะห์หลัก: ทำไมต้อง Nuke?

คุณอาจสงสัยว่า: "มีระบบบิลด์มากมายอยู่แล้ว เช่น Make, Gradle หรือแม้แต่เชลล์สคริปต์ธรรมดา ทำไมคุณถึงใช้ระบบที่ใช้ C# มากเกินไปล่ะ"

นั่นเป็นคำถามที่ยุติธรรม สิ่งดึงดูดใจหลักของ Nuke คือการนำฟีเจอร์ภาษาการเขียนโปรแกรมที่เรารู้จักดีมาสู่โลกแห่งสคริปต์บิลด์

### 1. เวิร์กโฟลว์การสร้างแบบแยกส่วน: ศิลปะแห่งเป้าหมาย

Nuke มีปรัชญาการออกแบบที่ชัดเจนมาก: **ทุกสิ่งเป็นเป้าหมาย**

ในสคริปต์แบบดั้งเดิม เป็นเรื่องง่ายที่จะจบลงด้วยตรรกะการดำเนินการเชิงเส้นหลายร้อยบรรทัดที่พันกัน ใน Nuke เราแบ่งกระบวนการสร้างให้เป็นอิสระ `Target`ส. แต่ละเป้าหมายจะทำสิ่งเดียวเท่านั้น ตัวอย่างเช่น:

- `Clean`: ทำความสะอาดไดเร็กทอรีเอาต์พุต
- `Restore`: กู้คืนแพ็คเกจการพึ่งพา
- `Compile`: รวบรวมโค้ด
- `Test`: รันการทดสอบหน่วย

การออกแบบนี้สอดคล้องกับหลักการความรับผิดชอบเดียวได้เป็นอย่างดี เช่นเดียวกับแบบเอกสารสำเร็จรูป เป้าหมายเหล่านี้สามารถรวมเข้าด้วยกันได้อย่างอิสระ ที่สำคัญกว่านั้น Nuke ช่วยให้เราสามารถกำหนดการพึ่งพาระหว่างเป้าหมายได้ เช่น หากคุณต้องการวิ่ง `Test`ระบบจะตรวจสอบโดยอัตโนมัติว่า `Compile` ได้ถูกดำเนินการแล้ว ถ้าคุณต้องการ `Compile`แล้ว `Restore` โดยธรรมชาติมาก่อน

กราฟการขึ้นต่อกันนี้ทำให้ตรรกะชัดเจนขึ้นมากและปรับปรุงประสิทธิภาพการดำเนินการอย่างมาก เนื่องจาก Nuke สามารถวิเคราะห์เส้นทางการดำเนินการที่เหมาะสมที่สุดได้โดยอัตโนมัติ

### 2. ประเภทความปลอดภัย: ลาก่อนฝันร้ายของการพิมพ์ผิด

ใครก็ตามที่เขียนบิลด์สคริปต์ใน Python อาจประสบปัญหานี้: สคริปต์ทำงานเป็นเวลาห้านาทีแล้วล้มเหลวเพราะ `Confi.guration` ถูกสะกดผิด หรือเนื่องจากมีการส่งผ่านสตริงโดยที่คาดว่าพารามิเตอร์ตัวเลข

ข้อได้เปรียบที่ใหญ่ที่สุดของการเขียนบิลด์สคริปต์ใน C# คือ **ความปลอดภัยประเภท** นั่นหมายถึง:

- **การตรวจสอบเวลาคอมไพล์**: IDE สามารถบอกคุณได้ว่ามีอะไรผิดปกติในขณะที่คุณกำลังพิมพ์ แทนที่จะต้องรอจนถึงรันไทม์
- **การปรับโครงสร้างใหม่อย่างปลอดภัย**: หากคุณต้องการเปลี่ยนชื่อตัวแปรหรือวิธีการ IDE สามารถจัดการตัวแปรหรือวิธีการนั้นได้ในขั้นตอนเดียวโดยไม่ต้องบังคับให้คุณทำการค้นหาและแทนที่ทั่วโลกที่มีความเสี่ยง
- **การทำงานอย่างชาญฉลาด**: IntelliSense อันทรงพลังสามารถกรอกโค้ดให้คุณโดยอัตโนมัติ ดังนั้นคุณจึงไม่จำเป็นต้องพลิกดูเอกสารประกอบเพื่อจำ API ที่คลุมเครือ

### 3. การสนับสนุนข้ามแพลตฟอร์ม: ประสบการณ์การสร้างที่สอดคล้องกัน

ที่ผ่านมาเราเขียนว่า `.bat` บน Windows และ `.sh` บน Linux จากนั้นอาจเพิ่มสคริปต์ Python เพื่อลดช่องว่าง ในตอนนี้ ไม่ว่า .NET Core หรือ .NET 5+ ในปัจจุบันจะรันที่ไหน Nuke ก็สามารถรันได้เช่นกัน

นั่นหมายความว่าสมาชิกในทีมสามารถใช้ Windows, Linux หรือ macOS ได้ และไม่ว่าพวกเขาจะชอบ Visual Studio, VS Code หรือ Rider พวกเขาก็ใช้ตรรกะเดียวกันทุกประการ ซึ่งช่วยลดปัญหาที่เกิดจากความแตกต่างของสภาพแวดล้อมได้อย่างมาก โดยเฉพาะปัญหา "ใช้งานได้กับเครื่องของฉัน" แบบคลาสสิก

### 4. การจัดการพารามิเตอร์และการกำหนดค่า

Nuke มีกลไกการแยกวิเคราะห์พารามิเตอร์ที่สวยงามมาก คุณไม่จำเป็นต้องแยกวิเคราะห์ด้วยตนเอง `string[] args`. คุณเพียงแค่กำหนดคุณสมบัติและใส่คำอธิบายประกอบด้วย `[Parameter]`และ Nuke จะจัดการการแมปจากอาร์กิวเมนต์บรรทัดคำสั่งและไฟล์การกำหนดค่าโดยอัตโนมัติ

ตัวอย่างเช่น เราสามารถกำหนดการกำหนดค่าบิลด์ได้อย่างง่ายดาย:

```csharp
[Parameter("Configuration to build - Default is 'Debug'")]
readonly Configuration BuildConfiguration = IsLocalBuild ? Configuration.Debug : Configuration.Release;

Target Compile => _ => _
    .DependsOn(Restore)
    .Executes(() =>
    {
        // Use BuildConfiguration here; it is type-safe
        DotNetBuild(s => s
            .SetConfiguration(BuildConfiguration)
            .SetProjectFile(SolutionFile));
    });
```

สไตล์นี้ทั้งใช้งานง่ายและทนทานต่อข้อผิดพลาด

## แนวทางปฏิบัติ: วิธีนำไปปฏิบัติในโครงการ

ทฤษฎีเพียงอย่างเดียวทำให้เราไปไหนไม่ได้ ให้เราดูว่าเราใช้แนวทางนี้ในโครงการ HagiCode อย่างไร

### 1. วางแผนโครงสร้างโครงการ

เราไม่ต้องการให้สคริปต์บิลด์ทำให้รูทโปรเจ็กต์เกะกะ และเรายังไม่ต้องการโครงสร้างไดเร็กทอรีที่ลึกมากจนคล้ายกับโปรเจ็กต์ Java บางโปรเจ็กต์ ดังนั้นเราจึงรวมไฟล์บิลด์ที่เกี่ยวข้องกับ Nuke ทั้งหมดไว้ในไฟล์เดียว `nukeBuild/` ไดเรกทอรี

ประโยชน์ที่ได้รับนั้นตรงไปตรงมา:

- รูทโปรเจ็กต์ยังคงสะอาดอยู่
- ตรรกะของบิลด์มีความสอดคล้องและจัดการได้ง่ายกว่า
- สมาชิกในทีมใหม่สามารถเห็นได้ทันทีว่า "นี่คือจุดที่ตรรกะที่เกี่ยวข้องกับบิวด์มีอยู่"

### 2. ออกแบบห่วงโซ่การพึ่งพาเป้าหมายที่ชัดเจน

เมื่อออกแบบเป้าหมาย เราปฏิบัติตามหลักการข้อหนึ่ง: **อะตอมมิกซิตีบวกกับโฟลว์การพึ่งพา**

แต่ละเป้าหมายควรมีขนาดเล็กพอที่จะทำสิ่งเดียวได้ ตัวอย่างเช่น `Clean` ควรลบเฉพาะไฟล์และไม่ควรจัดการบรรจุภัณฑ์ที่อยู่ด้านข้างอย่างเงียบๆ

โฟลว์การขึ้นต่อกันที่แนะนำจะมีลักษณะดังนี้:

`Clean` -> `Restore` -> `Compile` -> `Test` -> `Pack`

แน่นอนว่านี่ไม่ใช่เรื่องเด็ดขาด ตัวอย่างเช่น หากคุณต้องการรันการทดสอบเท่านั้นและไม่ต้องการทำแพ็คเกจใดๆ Nuke จะอนุญาตให้คุณดำเนินการได้ `nuke Test` โดยตรงและจะจัดการตามความต้องการโดยอัตโนมัติ `Restore` และ `Compile` ขั้นตอนแรก

### 3. การจัดการและการบันทึกข้อผิดพลาดที่มีประสิทธิภาพ

สิ่งที่น่าหงุดหงิดที่สุดเกี่ยวกับบิลด์สคริปต์คืออะไร? ข้อความแสดงข้อผิดพลาดที่ไม่ชัดเจน บิลด์ที่ล้มเหลวซึ่งบันทึกเฉพาะ "ข้อผิดพลาด: 1" ก็เพียงพอแล้วที่จะทำให้ใครก็ตามคลั่งไคล้

ใน Nuke เนื่องจากเราสามารถใช้การจัดการข้อยกเว้น C# ได้โดยตรง เราจึงสามารถจับภาพและรายงานข้อผิดพลาดได้อย่างแม่นยำยิ่งขึ้น

```csharp
Target Publish => _ => _
    .DependsOn(Test)
    .Executes(() =>
    {
        try 
        {
            // Try publishing to NuGet
            DotNetNuGetPush(s => s
                .SetTargetPath(ArtifactPath)
                .SetSource("https://api.nuget.org/v3/index.json")
                .SetApiKey(ApiKey));
        }
        catch (Exception ex)
        {
            Log.Error($"Publishing failed. Team, please check whether the key is correct: {ex.Message}");
            throw; // Make sure the build process exits with a non-zero exit code
        }
    });
```

### 4. ใช้การทดสอบเพื่อปกป้องคุณภาพ

บิลด์สคริปต์ก็เป็นโค้ดเช่นกัน ซึ่งหมายความว่าจำเป็นต้องมีการทดสอบด้วย Nuke ช่วยให้เราสามารถเขียนการทดสอบสำหรับเวิร์กโฟลว์การ build เพื่อที่ว่าเมื่อเราเปลี่ยนลอจิกการ build เราจะไม่ทำลายกระบวนการ release ที่มีอยู่โดยไม่ได้ตั้งใจ นี่เป็นสิ่งสำคัญอย่างยิ่งในไปป์ไลน์การรวมอย่างต่อเนื่อง

- **ความสามารถในการบำรุงรักษา**: การกำหนดค่าเป็นโค้ด ตรรกะที่ชัดเจนยิ่งขึ้น และเส้นทางการเริ่มต้นใช้งานที่เร็วขึ้นสำหรับผู้มีส่วนร่วมรายใหม่
- **ความเสถียร**: การพิมพ์ที่แข็งแกร่งช่วยลดข้อผิดพลาดในระดับต่ำได้มากกว่า 90%
- **ความสม่ำเสมอ**: ประสบการณ์ข้ามแพลตฟอร์มที่เป็นหนึ่งเดียวที่ช่วยขจัดการเบี่ยงเบนของสภาพแวดล้อม

หากการเขียนสคริปต์บิลด์เคยรู้สึกเหมือน "รู้สึกเหมือนกำลังฝ่าความมืด" การใช้ Nuke ก็ให้ความรู้สึกเหมือน "เดินในเวลากลางคืนโดยเปิดไฟ" หากคุณเบื่อหน่ายกับการรักษาสคริปต์ที่แก้ไขจุดบกพร่องได้ยาก ลองพิจารณาย้ายตรรกะการสร้างของคุณไปสู่โลกของ C# คุณอาจค้นพบว่างานสร้างก็สามารถหรูหราได้เช่นกัน
