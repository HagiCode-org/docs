## การแก้ปัญหาความท้าทายในการตรวจสอบสิทธิ์ WebSocket ของเบราว์เซอร์: รูปแบบพร็อกซีที่ใช้งานได้จริงสำหรับการรู้จำเสียง Doubao

> เบราว์เซอร์ WebSocket API ไม่รองรับส่วนหัว HTTP แบบกำหนดเอง ซึ่งสร้างความท้าทายสำหรับบริการรู้จำเสียงที่ต้องใช้ข้อมูลการตรวจสอบสิทธิ์ในส่วนหัว บทความนี้จะแชร์วิธีที่เราแก้ไขปัญหานั้นในโปรเจ็กต์ HagiCode ด้วยรูปแบบพร็อกซีแบ็กเอนด์ และวิธีการพัฒนาจากการทดลอง Playground ไปสู่การใช้งานจริง


## พื้นหลัง

เมื่อเราเริ่มสร้างการรู้จำคำพูดสำหรับโครงการ HagiCode เราก็เลือกบริการรู้จำเสียง Doubao ของ ByteDance อย่างมั่นใจ การออกแบบเบื้องต้นนั้นตรงไปตรงมา: ให้ส่วนหน้าเชื่อมต่อโดยตรงกับบริการ WebSocket ของ Doubao มันยากแค่ไหน? แค่เปิดการเชื่อมต่อแล้วส่งข้อมูลใช่ไหม?

จากนั้นก็เกิดความประหลาดใจ: API ของ Doubao ต้องการข้อมูลการตรวจสอบสิทธิ์ในการส่งผ่านส่วนหัว HTTP รวมถึงสิ่งต่างๆ เช่น `accessToken` และ `secretKey`. นั่นกลายเป็นเรื่องน่าอึดอัดใจทันที เนื่องจากเบราว์เซอร์ WebSocket API ไม่รองรับการตั้งค่าส่วนหัวที่กำหนดเอง

แล้วคุณจะทำอย่างไรเมื่อเบราว์เซอร์ไม่ยอมให้คุณส่งมัน?

ในเวลานั้น เรามีสองทางเลือก:

1. ใส่ข้อมูลรับรองลงในพารามิเตอร์การสืบค้น URL - เรียบง่ายและตรงไปตรงมา
2. เพิ่มเลเยอร์พร็อกซีที่แบ็กเอนด์ - ทำงานได้มากขึ้นตั้งแต่แรกเห็น

ตัวเลือกแรกเปิดเผยข้อมูลรับรองโดยตรงในโค้ดส่วนหน้าและที่จัดเก็บในเครื่อง นั่นปลอดภัยจริงๆเหรอ? ฉันไม่สบายใจกับมัน ยิ่งไปกว่านั้น API บางตัวจำเป็นต้องมีการตรวจสอบตามส่วนหัว ดังนั้นแนวทางนี้จึงใช้ไม่ได้ด้วยซ้ำ

ท้ายที่สุด เราเลือกตัวเลือกที่สอง: ใช้พร็อกซี WebSocket บนแบ็กเอนด์ โดยบังเอิญ รูปแบบนี้ได้รับการตรวจสอบครั้งแรกในสภาพแวดล้อมสนามเด็กเล่นของเรา และหลังจากที่เรายืนยันความเสถียรแล้วเท่านั้นจึงจะย้ายรูปแบบนี้ไปสู่การใช้งานจริง ท้ายที่สุดแล้ว ไม่มีใครอยากให้การผลิตเพิ่มขึ้นเป็นสองเท่าของการทดลองในห้องปฏิบัติการ

## เกี่ยวกับ ฮากิโค้ด

แนวทางที่แบ่งปันในบทความนี้มาจากประสบการณ์จริงของเราใน [ฮากิโค้ด](https://github.com/HagiCode-org/site) โครงการ

HagiCode เป็นโปรเจ็กต์ผู้ช่วยเขียนโค้ด AI พร้อมรองรับการโต้ตอบด้วยเสียง เนื่องจากเราจำเป็นต้องเรียกใช้บริการรู้จำคำพูดจากส่วนหน้า เราจึงพบปัญหาการตรวจสอบสิทธิ์ WebSocket โดยตรง ซึ่งนำเราไปสู่วิธีแก้ปัญหาที่อธิบายไว้ที่นี่ บางครั้งอุปสรรคทางเทคนิคเหล่านี้อาจทำให้คุณหงุดหงิด แต่ยังบังคับให้คุณเรียนรู้รูปแบบที่จะเป็นประโยชน์ในภายหลัง

## การวิเคราะห์ความท้าทายทางเทคนิค

### ข้อจำกัดของเบราว์เซอร์ WebSocket

WebSocket API มาตรฐานดูเรียบง่ายอย่างน่าอัศจรรย์:

```javascript
const ws = new WebSocket('wss://example.com/ws');
```

แต่ความเรียบง่ายนั้นตรงจุดที่ปัญหาอยู่ โดยจะส่งผ่านพารามิเตอร์ใน URL เท่านั้น และไม่สามารถตั้งค่าส่วนหัวในแบบที่คำขอ HTTP ทำได้:

```javascript
// This is not supported in the WebSocket API
const ws = new WebSocket('wss://example.com/ws', {
  headers: {
    'Authorization': 'Bearer token'
  }
});
```

และนั่นคือประเด็นหลัก สำหรับบริการต่างๆ เช่น การรู้จำเสียง Doubao ที่ต้องอาศัยการตรวจสอบสิทธิ์ตามส่วนหัว ข้อจำกัดนี้ถือเป็นฮาร์ดบล็อกเกอร์

เมื่อคุณยอมรับข้อจำกัดนั้น สถาปัตยกรรมก็ต้องเปลี่ยน

### การตัดสินใจออกแบบสถาปัตยกรรม

เมื่อออกแบบโซลูชัน เราเปรียบเทียบข้อดีข้อเสียอย่างรอบคอบ

**การตัดสินใจที่ 1: การเลือกรูปแบบพร็อกซี**

เราเปรียบเทียบสองวิธี:

| ตัวเลือก | ข้อดี | ข้อเสีย | การตัดสินใจ |
|------|------|------|------|
| WebSocket ดั้งเดิม | น้ำหนักเบา เรียบง่าย การส่งต่อโดยตรง | การจัดการการเชื่อมต่อจะต้องได้รับการจัดการด้วยตนเอง | เลือกแล้ว |
| สัญญาณร | การเชื่อมต่อใหม่อัตโนมัติ การพิมพ์ที่แข็งแกร่ง | ซับซ้อนเกินไปและมีการพึ่งพาเพิ่มเติม | ถูกปฏิเสธ |

ในที่สุดเราก็เลือก WebSocket ดั้งเดิม พูดตามตรง มันเป็นตัวเลือกที่เบาที่สุดและเหมาะสมกว่าสำหรับการส่งต่อไบนารี่สตรีมแบบสองทิศทางอย่างง่าย การดึง SignalR เข้ามาจะให้ความรู้สึกเหมือนเป็นวิศวกรรมมากเกินไปสำหรับกรณีการใช้งานนี้ และอาจเพิ่มเวลาแฝงเพิ่มเติมได้

**การตัดสินใจที่ 2: กลยุทธ์การจัดการการเชื่อมต่อ**

เราใช้โมเดล "หนึ่งการเชื่อมต่อ หนึ่งเซสชัน" - แต่ละการเชื่อมต่อฟรอนต์เอนด์ WebSocket จะแมปกับการเชื่อมต่อแบ็กเอนด์ Doubao ที่เป็นอิสระของตัวเอง

ประโยชน์ที่ได้รับนั้นตรงไปตรงมา:
- ง่ายต่อการนำไปใช้และสอดคล้องกับรูปแบบการใช้งานทั่วไป
- แก้จุดบกพร่องและแก้ไขปัญหาได้ง่ายขึ้น
- การแยกทรัพยากรที่ดี ป้องกันการรบกวนระหว่างเซสชัน

พูดง่ายๆ ก็คือ บางครั้งวิธีแก้ปัญหาโดยตรงก็อาจเป็นทางออกที่ดีที่สุด ความซับซ้อนไม่ได้ทำให้การออกแบบดีขึ้นโดยอัตโนมัติ

**การตัดสินใจที่ 3: การจัดเก็บข้อมูลการตรวจสอบความถูกต้อง**

ข้อมูลประจำตัวจะถูกจัดเก็บไว้ในไฟล์การกำหนดค่าแบ็กเอนด์ (`appsettings.yml` หรือตัวแปรสภาพแวดล้อม) และโหลดผ่านการฉีดการขึ้นต่อกัน:

- โมเดลการกำหนดค่าอย่างง่ายที่ตรงกับแบบแผนแบ็กเอนด์ที่มีอยู่
- ข้อมูลที่ละเอียดอ่อนไม่เคยไปถึงส่วนหน้า
- รองรับการตั้งค่าสภาพแวดล้อมที่หลากหลายสำหรับการพัฒนา การทดสอบ และการใช้งานจริง

ระดับของการแยกมีความสำคัญ ไม่มีใครอยากให้ข้อมูลประจำตัวลอยอยู่ในที่ที่ไม่ควรอยู่

### การออกแบบการไหลของข้อมูล

กระแสข้อมูลโดยรวมมีลักษณะดังนี้:

```
Frontend (browser)
  │
  │ ws://backend/api/voice/ws
  │ WebSocket (binary)
  ▼
Backend (proxy)
  │
  │ wss://openspeech.bytedance.com/
  │ (with authentication headers)
  ▼
Doubao API
```

การไหลนั้นไม่ซับซ้อน:
1. ส่วนหน้าเชื่อมต่อกับพร็อกซีส่วนหลังผ่าน WebSocket
2. พร็อกซีแบ็กเอนด์รับข้อมูลเสียงและเชื่อมต่อกับ Doubao API ด้วยส่วนหัวที่ได้รับการตรวจสอบสิทธิ์
3. Doubao API ส่งคืนผลลัพธ์การจดจำ และพร็อกซีส่งต่อไปยังส่วนหน้า
4. กระบวนการทั้งหมดยังคงไม่ตรงกันอย่างสมบูรณ์กับการสตรีมแบบสองทิศทาง

เมื่อแบ่งความรับผิดชอบอย่างชัดเจน การออกแบบจะค่อนข้างเป็นธรรมชาติ

## การใช้งานองค์ประกอบหลัก

### 1. การกำหนดค่าตำแหน่งข้อมูล WebSocket

```csharp
app.Map("/ws", async context =>
{
    if (context.WebSockets.IsWebSocketRequest)
    {
        // Read configuration from query parameters
        var appId = context.Request.Query["appId"];
        var accessToken = context.Request.Query["accessToken"];

        // Validate required parameters
        if (string.IsNullOrEmpty(appId) || string.IsNullOrEmpty(accessToken))
        {
            context.Response.StatusCode = 400;
            return;
        }

        // Accept the WebSocket connection
        using var webSocket = await context.WebSockets.AcceptWebSocketAsync();

        // Message handling loop
        var buffer = new byte[4096];
        while (!webSocket.CloseStatus.HasValue)
        {
            var result = await webSocket.ReceiveAsync(buffer, CancellationToken.None);

            if (result.MessageType == WebSocketMessageType.Close)
            {
                await webSocket.CloseAsync(
                    result.CloseStatus.Value,
                    result.CloseStatusDescription,
                    CancellationToken.None);
                break;
            }

            // Process audio data
            await HandleAudioDataAsync(buffer, result.Count);
        }
    }
});
```

### 2. การจัดการเซสชัน

```csharp
public class DoubaoSessionManager : IDoubaoSessionManager
{
    private readonly ConcurrentDictionary<string, DoubaoSession> _sessions = new();

    public DoubaoSession CreateSession(string connectionId)
    {
        var session = new DoubaoSession(connectionId);
        _sessions[connectionId] = session;
        return session;
    }

    public async Task SendAudioAsync(string connectionId, byte[] audioData)
    {
        if (_sessions.TryGetValue(connectionId, out var session))
        {
            await session.SendAudioAsync(audioData);
        }
    }

    public void RemoveSession(string connectionId)
    {
        if (_sessions.TryRemove(connectionId, out var session))
        {
            session.Dispose();
        }
    }
}
```

การใช้ `ConcurrentDictionary` สำหรับการจัดการเซสชันหมายถึงความปลอดภัยของเธรดส่วนใหญ่ได้รับการจัดการสำหรับเรา การเชื่อมต่อขาเข้าแต่ละรายการจะได้รับเซสชันของตัวเอง และการล้างข้อมูลจะเกิดขึ้นโดยอัตโนมัติเมื่อตัดการเชื่อมต่อ

## บูรณาการส่วนหน้าในทางปฏิบัติ

### การเชื่อมต่อเว็บซ็อกเก็ต

```javascript
class DoubaoVoiceClient {
    constructor(config) {
        this.config = config;
        this.ws = null;
    }

    async connect() {
        const url = new URL(this.config.wsUrl);
        // Add query parameters
        Object.entries(this.config.params).forEach(([key, value]) => {
            url.searchParams.set(key, value);
        });

        this.ws = new WebSocket(url);

        return new Promise((resolve, reject) => {
            this.ws.onopen = () => {
                console.log('[DoubaoVoice] Connected');
                resolve();
            };

            this.ws.onmessage = (event) => {
                this._handleMessage(JSON.parse(event.data));
            };

            this.ws.onerror = reject;
        });
    }

    _handleMessage(message) {
        switch (message.type) {
            case 'status':
                this._handleStatus(message.payload);
                break;
            case 'result':
                this.onResult?.(message.payload);
                break;
            case 'error':
                console.error('[DoubaoVoice] Error:', message.payload);
                break;
        }
    }
}

// Usage example
const client = new DoubaoVoiceClient({
    wsUrl: 'ws://localhost:5000/ws',
    params: {
        appId: 'your-app-id',
        accessToken: 'your-access-token',
        sampleRate: 16000,
        bitsPerSample: 16,
        channels: 1
    }
});
```

### การจับและส่งสัญญาณเสียง

การใช้ `AudioWorkletNode` สำหรับการประมวลผลเสียงให้ประสิทธิภาพที่ดีขึ้น:

```javascript
// audio-worklet.js
class AudioProcessorWorklet extends AudioWorkletProcessor {
    process(inputs, outputs, parameters) {
        const input = inputs[0]?.[0];
        if (!input) return true;

        // Convert to 16-bit PCM
        const pcm = new Int16Array(input.length);
        for (let i = 0; i < input.length; i++) {
            pcm[i] = Math.max(-32768, Math.min(32767, input[i] * 32767));
        }

        this.port.postMessage({
            type: 'audioData',
            data: pcm.buffer
        }, [pcm.buffer]);

        return true;
    }
}

registerProcessor('audio-processor', AudioProcessorWorklet);

// Main thread code
async function startAudioRecording() {
    const stream = await navigator.mediaDevices.getUserMedia({
        audio: {
            echoCancellation: true,
            noiseSuppression: true,
            autoGainControl: true,
            sampleRate: 48000
        }
    });

    const audioContext = new AudioContext();
    const audioSource = audioContext.createMediaStreamSource(stream);

    await audioContext.audioWorklet.addModule('/audio-worklet.js');
    const audioWorkletNode = new AudioWorkletNode(audioContext, 'audio-processor');

    audioWorkletNode.port.onmessage = (event) => {
        if (event.data.type === 'audioData' && ws?.readyState === WebSocket.OPEN) {
            ws.send(event.data.data); // Send binary data directly
        }
    };

    audioSource.connect(audioWorkletNode);
}
```

`AudioWorklet` ทำงานได้ดีกว่ามาก `ScriptProcessorNode` และหลีกเลี่ยงปัญหาเสียงติดขัดที่พาธการประมวลผลแบบเก่ามักแนะนำ

## หมายเหตุและแนวทางปฏิบัติที่ดีที่สุด

### 1. การตรวจสอบการเชื่อมต่อ

- บันทึกสถานะเซสชันเป็นประจำเพื่อติดตามวงจรการเชื่อมต่อทั้งหมด
- ตรวจสอบจำนวนและระยะเวลาของส่วนของเสียงเพื่อตรวจจับการเชื่อมต่อที่ผิดปกติ
- บันทึกสถานะการเชื่อมต่อและพฤติกรรมการเชื่อมต่อใหม่สำหรับบริการ Doubao

สิ่งเหล่านี้เป็นแนวทางปฏิบัติขั้นพื้นฐาน แต่สร้างความแตกต่างอย่างแท้จริงในการผลิต

### 2. การจัดการข้อผิดพลาด

- จับภาพและบันทึกข้อยกเว้น WebSocket ทั้งหมด
- ใช้ `IAsyncDisposable` เพื่อให้แน่ใจว่าทรัพยากรได้รับการทำความสะอาด
- ใช้การจัดการการปิดการเชื่อมต่อและการหมดเวลาอย่างค่อยเป็นค่อยไป

สรุปชอบความเข้มแข็งครับ

### 3. ข้อกำหนดรูปแบบเสียง

- อัตราตัวอย่าง: 16000 Hz (แนะนำ) หรือ 8000 Hz
- ความลึกของบิต: 16 บิต
- ช่อง: โมโน
- การเข้ารหัส: PCM (ดิบ)

หากรูปแบบไม่ถูกต้อง การจดจำอาจล้มเหลวหรือคุณภาพอาจลดลงอย่างมาก ข้อกำหนดเหล่านี้มีความสำคัญ

### 4. ข้อควรพิจารณาด้านความปลอดภัย

- เก็บข้อมูลรับรองที่ละเอียดอ่อนไว้ในการกำหนดค่าแบ็กเอนด์เท่านั้น
- บังคับใช้ขีดจำกัดการเชื่อมต่อเพื่อป้องกันการสิ้นเปลืองทรัพยากร
- ใช้ HTTPS/WSS ในการใช้งานจริง

ความปลอดภัยไม่ใช่เรื่องเล็กๆ น้อยๆ

### 5. การเพิ่มประสิทธิภาพการทำงาน

- ใช้การดำเนินการแบบอะซิงโครนัสเพื่อหลีกเลี่ยงการบล็อก
- ปรับขนาดบัฟเฟอร์ตามต้องการ (ค่าเริ่มต้น: 4096 ไบต์)
- พิจารณาการรวมการเชื่อมต่อและกลยุทธ์การนำกลับมาใช้ใหม่

ใช้การเพิ่มประสิทธิภาพเหล่านี้เมื่อเหมาะสมกับปริมาณงานของคุณ

## คำแนะนำในการปรับใช้

1. **การปรับใช้นักเทียบท่า**: รวมบริการพร็อกซีเป็นคอนเทนเนอร์เพื่อการปรับขนาดและการจัดการที่ง่ายขึ้น
2. **โหลดบาลานซ์**: ใช้ Nginx หรือ Envoy เป็นพร็อกซีย้อนกลับสำหรับการรับส่งข้อมูล WebSocket
3. **การตรวจสุขภาพ**: ใช้การตรวจสอบตามการเต้นของหัวใจเพื่อตรวจสอบความพร้อมในการให้บริการ
4. **การรวมบันทึก**: ส่งบันทึกไปยังระบบการบันทึกแบบรวมศูนย์ เช่น ELK หรือ Loki

การปรับใช้อาจทำได้ง่ายหรือซับซ้อน ขึ้นอยู่กับทีมและสภาพแวดล้อม ดังนั้นควรปรับเปลี่ยนให้เหมาะสม

## สรุป

รูปแบบพร็อกซี WebSocket แก้ปัญหาหลักที่เบราว์เซอร์ WebSocket API ไม่รองรับส่วนหัวที่กำหนดเอง ในโครงการ HagiCode รูปแบบนี้ได้รับการพิสูจน์แล้วว่าเป็นไปได้และมีเสถียรภาพ เมื่อเปลี่ยนจากการตรวจสอบสนามเด็กเล่นไปสู่การใช้งานจริง

ประเด็นสำคัญ:
- พร็อกซีแบ็กเอนด์สามารถส่งข้อมูลการตรวจสอบสิทธิ์ได้อย่างปลอดภัย
- Native WebSocket มีน้ำหนักเบาและมีประสิทธิภาพสำหรับสถานการณ์ง่ายๆ
- โมเดล "หนึ่งการเชื่อมต่อ หนึ่งเซสชัน" ช่วยให้ทั้งการใช้งานและการดีบักง่ายขึ้น
- โปรโตคอลส่วนหน้า-แบ็กเอนด์ควรแยกสัญญาณควบคุมออกจากข้อมูลเสียง

หากคุณกำลังสร้างฟีเจอร์ที่ต้องอาศัยการรับรองความถูกต้องของ WebSocket ด้วย ฉันหวังว่ารูปแบบนี้จะเป็นจุดเริ่มต้นที่เป็นประโยชน์แก่คุณ

หากคุณมีคำถาม โปรดพูดคุยกับเราได้ตลอดเวลา ความก้าวหน้าทางเทคนิคจะเกิดขึ้นเร็วขึ้นเมื่อผู้คนเปรียบเทียบบันทึกย่อ
