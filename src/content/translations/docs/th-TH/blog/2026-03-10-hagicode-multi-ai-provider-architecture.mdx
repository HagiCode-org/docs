## สถาปัตยกรรมผู้ให้บริการ Multi-AI ที่ใช้งานได้จริงในแพลตฟอร์ม HagiCode

> บทความนี้จะแชร์แนวทางทางเทคนิคที่เราใช้ภายใต้สถาปัตยกรรม Orleans Grain เพื่อผสานรวมเครื่องมือ AI สองตัว ได้แก่ iflow และ OpenCode ผ่านการรวมเป็นหนึ่งเดียว `IAIProvider` และเปรียบเทียบความแตกต่างในการใช้งานระหว่างการสื่อสาร WebSocket และ HTTP โดยละเอียด

:::note[บริบทเวลา]
บทความนี้บันทึกแนวทางปฏิบัติด้านสถาปัตยกรรมของผู้ให้บริการ ณ เวลาที่เผยแพร่ สำหรับขอบเขตที่ได้รับการสนับสนุนในปัจจุบัน โปรดดูที่ [การติดตั้ง AI Agent CLI](/th-TH/related-software-installation/ai-agent-cli/) และ [คู่มือการติดตั้ง](/th-TH/installation); `IFlowCli` ขณะนี้จะคงไว้เป็นค่าความเข้ากันได้ในอดีตเท่านั้น และไม่ได้เป็นส่วนหนึ่งของขอบเขตการสนับสนุนที่ใช้งานอยู่ในปัจจุบัน
:::

## พื้นหลัง

ไม่มีอะไรลึกลับเป็นพิเศษเกี่ยวกับเรื่องนี้ ในขณะที่สร้าง HagiCode เราประสบปัญหาในทางปฏิบัติ นั่นคือ ผู้ใช้ต้องการทำงานกับเครื่องมือ AI ที่แตกต่างกัน ไม่น่าแปลกใจเลยที่ทุกคนต่างก็มีนิสัยเป็นของตัวเอง บางคนชอบ Claude Code บางคนชอบ GitHub Copilot และบางทีมใช้เครื่องมือที่พวกเขาพัฒนาขึ้นเอง

โซลูชันเริ่มต้นของเรานั้นเรียบง่ายและตรงไปตรงมา: เขียนโค้ดการผสานรวมเฉพาะสำหรับเครื่องมือ AI แต่ละรายการ แต่ข้อเสียก็ปรากฏอย่างรวดเร็ว ฐานรหัสเต็มไปด้วย `if-else` สาขา การเปลี่ยนแปลงทุกครั้งต้องมีการทดสอบในหลายๆ แห่ง และเครื่องมือใหม่ทุกอันหมายถึงการเขียนตรรกะอีกกองหนึ่งตั้งแต่เริ่มต้น

ต่อมาฉันตระหนักว่าการสร้างเอกภาพจะดีกว่า `IAIProvider` อินเทอร์เฟซและสรุปความสามารถที่ใช้ร่วมกันโดยผู้ให้บริการ AI ทั้งหมด ด้วยวิธีนี้ไม่ว่าจะใช้เครื่องมือใดอยู่ข้างใต้ ชั้นบนก็สามารถเรียกมันได้ในลักษณะเดียวกัน

เมื่อเร็วๆ นี้ โปรเจ็กต์จำเป็นต้องผสานรวมเครื่องมือใหม่ 2 รายการ ได้แก่ iflow และ OpenCode ทั้งสองรองรับโปรโตคอล ACP แต่รูปแบบการสื่อสารแตกต่างกัน ifflow ใช้ WebSocket ในขณะที่ OpenCode ใช้ HTTP API นั่นกลายเป็นการทดสอบทางสถาปัตยกรรมที่มีประโยชน์: ปรับโหมดการขนส่งสองโหมดที่แตกต่างกันหลังอินเทอร์เฟซแบบรวมเดียว

## เกี่ยวกับ ฮากิโค้ด

แนวทางที่แบ่งปันในบทความนี้มาจากประสบการณ์จริงของเราใน [ฮากิโค้ด](https://hagicode.com) โครงการ HagiCode เป็นแพลตฟอร์มการพัฒนาที่ได้รับความช่วยเหลือจาก AI ที่สร้างขึ้นบนสถาปัตยกรรม Orleans Grain มันทำงานร่วมกับผู้ให้บริการ AI ที่แตกต่างกันผ่านระบบแบบครบวงจร `IAIProvider` อินเทอร์เฟซทำให้ผู้ใช้สามารถเลือกเครื่องมือ AI ที่ต้องการได้อย่างยืดหยุ่น

## การออกแบบสถาปัตยกรรม

### นามธรรมอินเทอร์เฟซแบบรวม

ขั้นแรกเรากำหนด `IAIProvider` อินเทอร์เฟซและสรุปความสามารถที่ผู้ให้บริการ AI ทุกรายจำเป็นต้องนำไปใช้:

```csharp
public interface IAIProvider
{
    string Name { get; }
    bool SupportsStreaming { get; }
    ProviderCapabilities Capabilities { get; }

    Task<AIResponse> ExecuteAsync(AIRequest request, CancellationToken cancellationToken = default);
    IAsyncEnumerable<AIStreamingChunk> StreamAsync(AIRequest request, CancellationToken cancellationToken = default);
    Task<ProviderTestResult> PingAsync(CancellationToken cancellationToken = default);
    IAsyncEnumerable<AIStreamingChunk> SendMessageAsync(AIRequest request, string? embeddedCommandPrompt = null, CancellationToken cancellationToken = default);
}
```

อินเทอร์เฟซนี้มีวิธีการหลักหลายวิธี:
- `ExecuteAsync`: ดำเนินการคำขอ AI ครั้งเดียว
- `StreamAsync`: รับการตอบสนองแบบสตรีมมิ่งสำหรับการแสดงผลแบบเรียลไทม์
- `PingAsync`: ดำเนินการตรวจสุขภาพเพื่อตรวจสอบว่าผู้ให้บริการพร้อมให้บริการหรือไม่
- `SendMessageAsync`: ส่งข้อความพร้อมรองรับคำสั่งแบบฝัง

### IFlowCliProvider: การใช้งานแบบ WebSocket

ifflow ใช้ WebSocket สำหรับการสื่อสาร ACP สถาปัตยกรรมโดยรวมมีลักษณะดังนี้:

```
IFlowCliProvider → ACPSessionManager → WebSocketAcpTransport → iflow CLI
                ↓
         Dynamic port allocation + process management
```

การไหลของแกนกลางยังค่อนข้างตรงไปตรงมา:
1. `ACPSessionManager` สร้างและจัดการเซสชัน ACP
2. `WebSocketAcpTransport` จัดการการสื่อสาร WebSocket
3. พอร์ตจะถูกจัดสรรแบบไดนามิก และกระบวนการ iflow จะเริ่มต้นด้วย `iflow --experimental-acp --port`.
4. `IAIRequestToAcpMapper` และ `IAcpToAIResponseMapper` แปลงคำขอและการตอบกลับ

นี่คือรหัสหลัก:

```csharp
private async IAsyncEnumerable<AIStreamingChunk> StreamCoreAsync(
    AIRequest request,
    string? embeddedCommandPrompt,
    [EnumeratorCancellation] CancellationToken cancellationToken)
{
    // Resolve working directory
    var resolvedWorkingDirectory = ResolveWorkingDirectory(request);
    var effectiveRequest = ApplyEmbeddedCommandPrompt(request, embeddedCommandPrompt);

    // Create ACP session
    await using var session = await _sessionManager.CreateSessionAsync(
        Name,
        resolvedWorkingDirectory,
        cancellationToken,
        request.SessionId);

    // Send prompt
    var prompt = _requestMapper.ToPromptString(effectiveRequest);
    var promptResponse = await session.SendPromptAsync(prompt, cancellationToken);

    // Receive streaming response
    await foreach (var notification in session.ReceiveUpdatesAsync(cancellationToken))
    {
        if (_responseMapper.TryConvertToStreamingChunk(notification, out var chunk))
        {
            if (chunk.Type == StreamingChunkType.Metadata && chunk.IsComplete)
            {
                yield return chunk;
                yield break;
            }
            yield return chunk;
        }
    }
}
```

มีจุดออกแบบสองสามจุดที่ควรค่าแก่การกล่าวถึงที่นี่:
- ใช้ `await using` เพื่อให้แน่ใจว่าเซสชันได้รับการเผยแพร่อย่างถูกต้องและหลีกเลี่ยงการรั่วไหลของทรัพยากร
- ส่งคืนการตอบกลับแบบสตรีมมิ่งผ่าน `IAsyncEnumerable`ซึ่งรองรับสตรีมอะซิงก์ตามธรรมชาติ
- ใช้ `Metadata` ชิ้นเพื่อพิจารณาความสมบูรณ์และให้แน่ใจว่าได้รับการตอบกลับอย่างครบถ้วน

### OpenCodeCliProvider: การใช้งานที่ใช้ HTTP API

OpenCode ให้บริการผ่าน HTTP API ดังนั้นสถาปัตยกรรมจึงแตกต่างออกไปเล็กน้อย:

```plaintext
OpenCodeCliProvider → OpenCodeRuntimeManager → OpenCodeClient → OpenCode HTTP API
                      ↓
                OpenCodeProcessManager → opencode process management
```

คุณลักษณะเด่นของ OpenCode คือใช้ฐานข้อมูล SQLite เพื่อคงการเชื่อมโยงเซสชันไว้ ซึ่งทำให้การกู้คืนเซสชันและการกู้คืนแบบตอบสนองทันทีเป็นไปได้:

```csharp
private async Task<OpenCodePromptExecutionResult> ExecutePromptAsync(
    AIRequest request,
    string? embeddedCommandPrompt,
    CancellationToken cancellationToken)
{
    var prompt = BuildPrompt(request, embeddedCommandPrompt);
    var resolvedWorkingDirectory = ResolveWorkingDirectory(request.WorkingDirectory);
    var client = await _runtimeManager.GetClientAsync(resolvedWorkingDirectory, cancellationToken);
    var bindingSessionId = request.SessionId;
    var boundSession = TryGetBinding(bindingSessionId, resolvedWorkingDirectory);

    // Try to use the already bound session
    if (boundSession is not null)
    {
        try
        {
            return await PromptSessionAsync(
                client,
                boundSession,
                BuildPromptRequest(request, prompt, CreatePromptMessageId()),
                request.Model ?? _settings.Model,
                cancellationToken);
        }
        catch (OpenCodeApiException ex) when (IsStaleBinding(ex))
        {
            // The session has expired, remove the binding
            RemoveBinding(bindingSessionId);
        }
    }

    // Create a new session
    var session = await client.Session.CreateAsync(new OpenCodeSessionCreateRequest
    {
        Title = BuildSessionTitle(request)
    }, cancellationToken);

    BindSession(bindingSessionId, session.Id, resolvedWorkingDirectory);
    return await PromptSessionAsync(client, session.Id, ...);
}
```

การใช้งานนี้มีจุดเด่นที่น่าสนใจหลายประการ:
- กลไกการผูกเซสชัน: เหมือนกัน `SessionId` ใช้เซสชัน OpenCode เดียวกันซ้ำ หลีกเลี่ยงการสร้างเซสชันซ้ำ
- การจัดการการหมดอายุ: เมื่อพบว่าเซสชันหมดอายุ การเชื่อมโยงจะถูกล้างข้อมูลโดยอัตโนมัติ
- การคงอยู่ของฐานข้อมูล: การผูกจะถูกจัดเก็บไว้ใน SQLite และยังคงมีผลหลังจากรีสตาร์ท

### เปรียบเทียบทั้งสองแนวทาง

| ด้าน | IFlowCliProvider | OpenCodeCliProvider |
|------|-----------------|---------------------|
| การสื่อสาร | เว็บซ็อกเก็ต (ACP) | HTTP API |
| การจัดการกระบวนการ | ACPSessionManager | OpenCodeProcessManager |
| การจัดสรรพอร์ต | พอร์ตแบบไดนามิก | ไม่มีพอร์ต (ใช้ HTTP) |
| การจัดการเซสชัน | ACPSession | OpenCodeSession |
| ความพากเพียร | แคชในหน่วยความจำ | ฐานข้อมูล SQLite |
| คำสั่งเริ่มต้น | `iflow --experimental-acp --port` | `opencode` |
| เวลาแฝง | ต่ำกว่า (การเชื่อมต่อที่มีอายุการใช้งานยาวนาน) | ค่อนข้างสูงกว่า (คำขอ HTTP) |

วิธีการที่คุณเลือกขึ้นอยู่กับความต้องการของคุณเป็นหลัก WebSocket ดีกว่าสำหรับสถานการณ์ที่มีความต้องการแบบเรียลไทม์สูง ในขณะที่ HTTP API นั้นง่ายกว่าและแก้ไขข้อบกพร่องได้ง่ายกว่า

## คู่มือการปฏิบัติ

### กำหนดค่าผู้ให้บริการ

ขั้นแรก ให้เปิดใช้งานผู้ให้บริการทั้งสองรายในไฟล์กำหนดค่า:

```yaml
AI:
  Providers:
    IFlowCli:
      Type: "IFlowCli"
      Enabled: true
      ExecutablePath: "iflow"
      Model: null
      WorkingDirectory: null
    OpenCodeCli:
      Type: "OpenCodeCli"
      Enabled: true
      ExecutablePath: "opencode"
      Model: "anthropic/claude-sonnet-4"
      WorkingDirectory: null

OpenCode:
  Enabled: true
  BaseUrl: "http://localhost:38376"
  ExecutablePath: "opencode"
  StartupTimeoutSeconds: 30
  RequestTimeoutSeconds: 120
```

### ใช้ IFlowCliProvider

```csharp
// Get provider through the factory
var provider = await _providerFactory.GetProviderAsync(AIProviderType.IFlowCli);

// Execute an AI request
var request = new AIRequest
{
    Prompt = "请帮我重构这个函数",
    WorkingDirectory = "/path/to/project",
    Model = "claude-sonnet-4"
};

// Get the complete response
var response = await provider.ExecuteAsync(request, cancellationToken);
Console.WriteLine(response.Content);

// Or use streaming responses
await foreach (var chunk in provider.StreamAsync(request, cancellationToken))
{
    if (chunk.Type == StreamingChunkType.ContentDelta)
    {
        Console.Write(chunk.Content);
    }
}
```

### ใช้ OpenCodeCliProvider

```csharp
// Get provider through the factory
var provider = await _providerFactory.GetProviderAsync(AIProviderType.OpenCodeCli);

var request = new AIRequest
{
    Prompt = "请帮我分析这个错误",
    WorkingDirectory = "/path/to/project",
    Model = "anthropic/claude-sonnet-4"
};

var response = await provider.ExecuteAsync(request, cancellationToken);
Console.WriteLine(response.Content);
```

### การตรวจสุขภาพ

ก่อนเริ่มต้นหรือก่อนใช้งาน คุณสามารถตรวจสอบว่าผู้ให้บริการพร้อมใช้งานหรือไม่:

```csharp
var iflowResult = await iflowProvider.PingAsync(cancellationToken);
if (!iflowResult.Success)
{
    Console.WriteLine($"IFlow is unavailable: {iflowResult.ErrorMessage}");
    return;
}

var openCodeResult = await openCodeProvider.PingAsync(cancellationToken);
if (!openCodeResult.Success)
{
    Console.WriteLine($"OpenCode is unavailable: {openCodeResult.ErrorMessage}");
    return;
}
```

### การสนับสนุนคำสั่งแบบฝัง

ผู้ให้บริการทั้งสองรายรองรับคำสั่งแบบฝัง เช่น `/file:xxx`:

```csharp
var request = new AIRequest
{
    Prompt = "分析这个文件的问题",
    SystemMessage = "你是一个代码分析专家"
};

await foreach (var chunk in provider.SendMessageAsync(
    request,
    embeddedCommandPrompt: "/file:src/main.cs",
    cancellationToken))
{
    Console.Write(chunk.Content);
}
```

## หมายเหตุและแนวปฏิบัติที่ดีที่สุด

### การจัดการทรัพยากร

IFlow ใช้การเชื่อมต่อ WebSocket ที่มีอายุการใช้งานยาวนาน ดังนั้นการจัดการทรัพยากรจึงสมควรได้รับความสนใจเป็นพิเศษ:
- ใช้ `await using` เพื่อให้แน่ใจว่าเซสชันต่างๆ ได้รับการเผยแพร่อย่างถูกต้อง
- การยกเลิกจะทำให้เกิดการล้างข้อมูลกระบวนการ
- `ACPSessionManager` รองรับการจำกัดจำนวนเซสชันสูงสุด

การจัดการกระบวนการ OpenCode นั้นค่อนข้างง่ายกว่าและ `OpenCodeRuntimeManager` จัดการมันโดยอัตโนมัติ

### การจัดการข้อผิดพลาด

ผู้ให้บริการทั้งสองมีการจัดการข้อผิดพลาดที่สมบูรณ์:
- ข้อผิดพลาด IFlow ได้รับการเผยแพร่ผ่านการอัปเดตเซสชัน ACP
- ข้อผิดพลาด OpenCode ถูกส่งผ่าน `OpenCodeApiException`.
- ขอแนะนำให้ผู้โทรจับและจัดการกับข้อยกเว้นเหล่านี้

### การพิจารณาประสิทธิภาพ

- การสื่อสาร IFlow WebSocket มีเวลาแฝงต่ำกว่า HTTP
- การใช้เซสชัน OpenCode ซ้ำสามารถลดโอเวอร์เฮดของคำขอ HTTP
- กลไกแคชจากโรงงานหลีกเลี่ยงการสร้างผู้ให้บริการซ้ำๆ
- ในสถานการณ์ที่เกิดพร้อมกันสูง ให้ใส่ใจกับขีดจำกัดในการนับกระบวนการและจำนวนการเชื่อมต่อ

### การตรวจสอบการกำหนดค่า

เส้นทางปฏิบัติการได้รับการตรวจสอบเมื่อเริ่มต้นระบบ แต่ปัญหารันไทม์ยังคงเกิดขึ้นได้ `PingAsync` เป็นเครื่องมือที่มีประโยชน์ในการตรวจสอบว่าการกำหนดค่าถูกต้องหรือไม่:

```csharp
// Check at startup
var provider = await _providerFactory.GetProviderAsync(providerType);
var result = await provider.PingAsync(cancellationToken);
if (!result.Success)
{
    _logger.LogError("Provider {ProviderType} is unavailable: {Error}", providerType, result.ErrorMessage);
}
```

## สรุป

บทความนี้จะแชร์แนวทางทางเทคนิคที่ใช้โดยแพลตฟอร์ม HagiCode เมื่อผสานรวมเครื่องมือ AI ทั้งสอง iflow และ OpenCode โดยผ่านการรวมเป็นหนึ่งเดียว `IAIProvider` เราได้ปรับรูปแบบการสื่อสารที่แตกต่างกัน WebSocket และ HTTP ในขณะที่ยังคงรักษารูปแบบการโทรชั้นบนให้สอดคล้องกัน

จริงๆ แล้ว แนวคิดหลักนั้นค่อนข้างเรียบง่าย:
1. กำหนดนามธรรมอินเทอร์เฟซแบบรวม
2. สร้างเลเยอร์อะแดปเตอร์สำหรับการใช้งานที่แตกต่างกัน
3. จัดการทุกอย่างอย่างเท่าเทียมกันผ่านรูปแบบโรงงาน

นั่นทำให้ระบบมีการขยายที่ดี เมื่อจำเป็นต้องผสานรวมเครื่องมือ AI ใหม่ในภายหลัง สิ่งที่เราต้องทำคือปรับใช้ `IAIProvider` อินเทอร์เฟซโดยไม่ต้องเปลี่ยนโค้ดที่มีอยู่มากเกินไป

หากคุณกำลังทำงานเกี่ยวกับการบูรณาการเครื่องมือ AI หลายตัวด้วย ฉันหวังว่าบทความนี้จะมีประโยชน์
