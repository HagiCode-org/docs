## AI Compose Commit: การใช้ AI เพื่อปรับโครงสร้างเวิร์กโฟลว์ Git Commit อย่างชาญฉลาด

## บทนำ

ในกระบวนการพัฒนาซอฟต์แวร์ การเขียนโค้ดเป็นงานประจำที่โปรแกรมเมอร์ทุกคนต้องเผชิญทุกวัน แต่คุณเคยเจอสถานการณ์เช่นนี้บ้างไหม เมื่อสิ้นสุดวันทำงาน คุณเปิด Git ดูไฟล์ที่แก้ไขแบบไม่มีการจัดฉากหลายสิบไฟล์ และไม่รู้ว่าจะจัดระเบียบไฟล์เหล่านั้นให้เป็น Commit ที่สมเหตุสมผลได้อย่างไร

วิธีการแบบดั้งเดิมคือการจัดเตรียมไฟล์เป็นชุดด้วยตนเอง คอมมิตไฟล์ทีละไฟล์ และเขียนข้อความคอมมิต กระบวนการนี้ใช้เวลานานและเกิดข้อผิดพลาดได้ง่าย เรามักจะเสียเวลาไปกับเรื่องนี้ และท้ายที่สุดก็ไม่มีใครอยากกังวลกับงานบ้านที่น่าเบื่อเหล่านี้ตอนดึกๆ เมื่อพวกเขาเหนื่อยแล้ว

ในโครงการ HagiCode เราได้เปิดตัวฟีเจอร์ใหม่ - AI Compose Commit - ออกแบบมาเพื่อเปลี่ยนแปลงเวิร์กโฟลว์นี้อย่างสมบูรณ์ ด้วยการใช้ AI เพื่อวิเคราะห์การเปลี่ยนแปลงที่ไม่มีข้อผูกมัดทั้งหมดในแผนผังการทำงานอย่างชาญฉลาด ระบบจะจัดกลุ่มการเปลี่ยนแปลงเหล่านั้นออกเป็นคอมมิตเชิงลอจิคัลหลายรายการโดยอัตโนมัติ และดำเนินการคอมมิตตามมาตรฐาน ในบทความนี้ เราจะเจาะลึกหลักการนำไปใช้งาน สถาปัตยกรรมทางเทคนิค ตลอดจนความท้าทายและแนวทางแก้ไขที่เราพบในทางปฏิบัติ

## พื้นหลัง

### Pain Points ของ Git Commits แบบดั้งเดิม

ในฐานะระบบควบคุมเวอร์ชัน Git ช่วยให้นักพัฒนามีความสามารถในการจัดการโค้ดอันทรงพลัง แต่ในการใช้งานจริง ความมุ่งมั่นมักจะกลายเป็นคอขวดในเวิร์กโฟลว์การพัฒนา:

1. **การจัดกลุ่มด้วยตนเองใช้เวลานาน**: เมื่อมีการเปลี่ยนแปลงไฟล์จำนวนมาก นักพัฒนาจำเป็นต้องตรวจสอบแต่ละไฟล์ทีละไฟล์ และตัดสินใจว่าการเปลี่ยนแปลงใดเป็นของฟีเจอร์เดียวกัน นั่นต้องใช้ความพยายามทางจิตอย่างมาก
2. **คุณภาพของข้อความคอมมิตไม่สอดคล้องกัน**: การเขียนข้อความคอมมิตที่เป็นไปตามข้อกำหนด Conventional Commits ต้องใช้ประสบการณ์และทักษะ และผู้เริ่มต้นมักจะสร้างคอมมิตที่ไม่ได้มาตรฐาน
3. **การจัดการพื้นที่เก็บข้อมูลหลายรายการที่ซับซ้อน**: ในสภาพแวดล้อม monorepo การสลับระหว่างพื้นที่เก็บข้อมูลต่างๆ จะเพิ่มความซับซ้อนในการปฏิบัติงาน
4. **เวิร์กโฟลว์ถูกขัดจังหวะ**: การคอมมิตโค้ดจะขัดจังหวะความคิดของคุณ และทำให้ประสิทธิภาพการเขียนโค้ดลดลง

ปัญหาเหล่านี้ชัดเจนเป็นพิเศษในโครงการขนาดใหญ่และสภาพแวดล้อมของทีมที่ทำงานร่วมกัน เครื่องมือการพัฒนาที่ดีควรช่วยให้นักพัฒนามุ่งเน้นไปที่งานการเขียนโค้ดหลัก แทนที่จะต้องจมอยู่กับเวิร์กโฟลว์การคอมมิตที่ยุ่งยาก

### แนวโน้มการพัฒนาโดยใช้ AI เป็นตัวช่วย

ในช่วงไม่กี่ปีที่ผ่านมา AI ถูกนำมาใช้อย่างกว้างขวางมากขึ้นในการพัฒนาซอฟต์แวร์ ตั้งแต่การเติมโค้ดให้สมบูรณ์และการตรวจจับจุดบกพร่องไปจนถึงการสร้างเอกสารอัตโนมัติ AI กำลังค่อยๆ เข้าถึงทุกขั้นตอนของกระบวนการพัฒนา ในเวิร์กโฟลว์ Git แม้ว่าเครื่องมือบางอย่างจะรองรับการสร้างข้อความคอมมิตอยู่แล้ว แต่เครื่องมือส่วนใหญ่จำกัดอยู่เพียงสถานการณ์คอมมิตเดียว และขาดความสามารถในการวิเคราะห์และจัดกลุ่มการเปลี่ยนแปลงทั่วทั้งแผนผังการทำงานอย่างชาญฉลาด

HagiCode พบกับปัญหาเหล่านี้ในระหว่างการพัฒนาเช่นกัน เราลองใช้เครื่องมือหลายอย่าง แต่แต่ละเครื่องมือก็มีข้อจำกัดอย่างใดอย่างหนึ่ง ฟังก์ชั่นการทำงานไม่สมบูรณ์หรือประสบการณ์การใช้งานไม่ดีพอ นั่นคือเหตุผลที่ท้ายที่สุดเราจึงตัดสินใจนำ AI Composit Commit ไปใช้ด้วยตนเอง

ฟีเจอร์ AI Compose Commit ของ HagiCode ถูกสร้างขึ้นเพื่อเติมเต็มช่องว่างนั้น มันไม่เพียงแค่สร้างข้อความยืนยันเท่านั้น แต่ยังครอบคลุมกระบวนการทั้งหมดตั้งแต่การวิเคราะห์ไฟล์ไปจนถึงการดำเนินการอีกด้วย

## ปัญหา

### ความท้าทายทางเทคนิค

ในขณะที่ใช้ AI Compose Commit เราเผชิญกับความท้าทายทางเทคนิคหลายประการ:

1. **การทำความเข้าใจความหมายของไฟล์**: AI จำเป็นต้องเข้าใจความสัมพันธ์เชิงความหมายระหว่างการเปลี่ยนแปลงไฟล์ และตัดสินใจว่าไฟล์ใดเป็นของโมดูลการทำงานเดียวกัน ซึ่งจำเป็นต้องมีการวิเคราะห์เชิงลึกเกี่ยวกับเนื้อหาไฟล์ โครงสร้างไดเร็กทอรี และบริบทการเปลี่ยนแปลง

2. **กำหนดกลยุทธ์การจัดกลุ่ม**: ควรกำหนดมาตรฐานการจัดกลุ่มที่สมเหตุสมผลอย่างไร ตามคุณสมบัติ ตามโมดูล หรือตามประเภทไฟล์? โครงการที่แตกต่างกันอาจต้องใช้กลยุทธ์ที่แตกต่างกัน

3. **การตอบสนองแบบเรียลไทม์และการประมวลผลแบบอะซิงโครนัส**: การทำงานของ Git อาจใช้เวลานาน โดยเฉพาะอย่างยิ่งเมื่อต้องจัดการกับไฟล์จำนวนมาก เราจะดำเนินการที่ซับซ้อนให้เสร็จสิ้นโดยยังคงรักษาประสบการณ์ผู้ใช้ที่ดีได้อย่างไร

4. **การสนับสนุนหลายพื้นที่เก็บข้อมูล**: ในสถาปัตยกรรม monorepo การดำเนินการจะต้องมีการกำหนดเส้นทางอย่างถูกต้องระหว่างพื้นที่เก็บข้อมูลหลักและพื้นที่เก็บข้อมูลย่อย

5. **การจัดการข้อผิดพลาดและการย้อนกลับ**: หากการคอมมิตหนึ่งล้มเหลว ควรจัดการการคอมมิตที่ดำเนินการแล้วอย่างไร? ไฟล์ที่จัดฉากแล้วจำเป็นต้องย้อนกลับหรือไม่

6. **ความสอดคล้องของข้อความคอมมิต**: ข้อความคอมมิตที่สร้างขึ้นจะต้องตรงกับสไตล์ที่มีอยู่ของโปรเจ็กต์ และยังคงสอดคล้องกับคอมมิตในอดีต

### การพิจารณาประสิทธิภาพ

การประมวลผล AI ในการเปลี่ยนแปลงไฟล์จำนวนมากกินเวลาและทรัพยากรการประมวลผลจำนวนมาก เราจำเป็นต้องเพิ่มประสิทธิภาพในด้านต่อไปนี้:

- ลดการเรียก AI ที่ไม่จำเป็น
- ปรับวิธีสร้างบริบทของไฟล์ให้เหมาะสม
- ใช้ชุดการดำเนินการ Git ที่มีประสิทธิภาพ

ปัญหาเหล่านี้ทั้งหมดปรากฏในการใช้งาน HagiCode จริง และเรามาถึงวิธีแก้ปัญหาที่ค่อนข้างสมบูรณ์ผ่านการทำซ้ำและการเพิ่มประสิทธิภาพซ้ำแล้วซ้ำเล่า หากคุณกำลังสร้างเครื่องมือที่คล้ายกัน เราหวังว่าประสบการณ์ของเราจะมอบแรงบันดาลใจให้กับคุณ

## โซลูชั่น

### การออกแบบสถาปัตยกรรมโดยรวม

เราใช้สถาปัตยกรรมแบบเลเยอร์เพื่อใช้ AI Compose Commit เพื่อให้มั่นใจถึงความสามารถในการปรับขนาดและการบำรุงรักษาที่ดี:

#### 1. เลเยอร์ API (เลเยอร์เว็บ)

**GitController** จัดเตรียมไฟล์ `POST /api/git/auto-compose-commit` จุดสิ้นสุดเป็นจุดเริ่มต้น เพื่อเพิ่มประสิทธิภาพประสบการณ์ผู้ใช้ เราได้นำรูปแบบอะซิงโครนัสแบบ fire-and-forget มาใช้:

- หลังจากที่ไคลเอ็นต์ส่งคำขอ เซิร์ฟเวอร์จะส่งคืน HTTP 202 Accepted ทันที
- การประมวลผล AI จริงจะทำงานแบบอะซิงโครนัสในเบื้องหลัง
- เมื่อการประมวลผลเสร็จสิ้น ไคลเอนต์จะได้รับแจ้งผ่าน SignalR

การออกแบบนี้ช่วยให้มั่นใจได้ว่าแม้ว่าการประมวลผล AI จะใช้เวลาหลายนาที แต่ผู้ใช้ยังคงได้รับการตอบสนองทันทีและไม่รู้สึกว่าระบบค้าง

#### 2. ชั้นบริการแอปพลิเคชัน (Application Layer)

**GitAppService** รับผิดชอบตรรกะทางธุรกิจหลัก:

- การตรวจจับพื้นที่เก็บข้อมูล: รองรับการจัดการพื้นที่เก็บข้อมูลหลายพื้นที่ใน monorepo
- การจัดการล็อค: ป้องกันข้อขัดแย้งที่เกิดจากการทำงานพร้อมกัน
- การประสานงานการแสดงไฟล์: โต้ตอบกับโฟลว์การประมวลผลของ AI
- การย้อนกลับข้อผิดพลาด: คืนสถานะเมื่อเกิดความล้มเหลว

#### 3. เลเยอร์คอมพิวเตอร์แบบกระจาย (Orleans Grains)

**AIGrain** ทำหน้าที่เป็นหน่วยดำเนินการหลักสำหรับการดำเนินงานของ AI มันดำเนินการ `AutoComposeCommitAsync` วิธีการจาก `IAIGrain` อินเตอร์เฟซ:

```csharp
// Define the interface method for AI-powered automatic commit composition
// Parameter notes:
// - projectId: unique project identifier
// - unstagedFiles: list of unstaged files, including file paths and status information
// - projectPath: project root directory path (optional), used to access project context
// Return value: a response object containing execution results, including success/failure status and detailed information
[Alias("AutoComposeCommitAsync")]
[ResponseTimeout("00:20:00")] // 20-minute timeout, suitable for handling large change sets
Task<AutoComposeCommitResponseDto> AutoComposeCommitAsync(
    string projectId,
    GitFileStatusDto[] unstagedFiles,
    string? projectPath = null);
```

วิธีนี้จะตั้งค่าการหมดเวลา 20 นาทีเพื่อจัดการกับชุดการเปลี่ยนแปลงขนาดใหญ่ ในการใช้งาน HagiCode ในโลกแห่งความเป็นจริง เราพบว่าบางโปรเจ็กต์อาจเกี่ยวข้องกับไฟล์ที่เปลี่ยนแปลงหลายร้อยไฟล์ในการผ่านครั้งเดียว ซึ่งต้องใช้เวลาในการประมวลผลมากขึ้น

#### 4. ชั้นบริการ AI

ผ่านทางนามธรรม `IAIService` อินเทอร์เฟซ เราได้ใช้สถาปัตยกรรมบริการ AI แบบเสียบได้ ขณะนี้เราใช้บริการ Claude Helper แต่สามารถเปลี่ยนไปใช้ผู้ให้บริการ AI รายอื่นได้อย่างง่ายดาย

### ตรรกะการใช้งานหลัก

#### การสร้างบริบทไฟล์

AI จำเป็นต้องเข้าใจสถานะของแต่ละไฟล์ก่อนจึงจะสามารถตัดสินใจได้อย่างชาญฉลาด เราสร้างบริบทของไฟล์ผ่าน `BuildFileChangesXml` วิธีการ:

```csharp
/// <summary>
/// Build an XML representation of file changes to provide the AI with complete file context information
/// </summary>
/// <param name="stagedFiles">List of staged files, including file path, status, and old path (for rename operations)</param>
/// <returns>A formatted XML string containing metadata for all files</returns>
private static string BuildFileChangesXml(GitFileStatusDto[] stagedFiles)
{
    var sb = new StringBuilder();
    sb.AppendLine("<files>");

    foreach (var file in stagedFiles)
    {
        sb.AppendLine("  <file>");
        // Use XML escaping to ensure special characters do not break the XML structure
        sb.AppendLine($"    <path>{System.Security.SecurityElement.Escape(file.Path)}</path>");
        sb.AppendLine($"    <status>{System.Security.SecurityElement.Escape(file.Status)}</status>");

        // Handle file rename scenarios and record the old path so the AI can understand change relationships
        if (!string.IsNullOrEmpty(file.OldPath))
        {
            sb.AppendLine($"    <oldPath>{System.Security.SecurityElement.Escape(file.OldPath)}</oldPath>");
        }

        sb.AppendLine("  </file>");
    }

    sb.AppendLine("</files>");
    return sb.ToString();
}
```

บริบทที่ใช้ XML นี้ประกอบด้วยเส้นทางไฟล์ สถานะ และเส้นทางเก่าสำหรับการดำเนินการเปลี่ยนชื่อ ทำให้ AI มีข้อมูลเมตาที่สมบูรณ์ ด้วยรูปแบบ XML ที่มีโครงสร้าง เรามั่นใจว่า AI สามารถเข้าใจสถานะและประเภทการเปลี่ยนแปลงของแต่ละไฟล์ได้อย่างแม่นยำ

#### การจัดการสิทธิ์ AI

เพื่อให้ AI ดำเนินการปฏิบัติการ Git ได้โดยตรง เราได้กำหนดค่าสิทธิ์อนุญาตเครื่องมือที่ครอบคลุม:

```csharp
// Define the set of tools the AI can use, including file operations and Git command execution permissions
// Read/Write/Edit: file reading, writing, and editing capabilities
// Bash(git:*): permission to execute all Git commands
// Other Bash commands: used to inspect file contents and directory structure so the AI can understand context
var allowedTools = new[]
{
    "Read", "Write", "Edit",
    "Bash(git:*)", "Bash(cat:*)", "Bash(ls:*)", "Bash(find:*)",
    "Bash(grep:*)", "Bash(head:*)", "Bash(tail:*)", "Bash(wc:*)"
};

// Build the complete AI request object
var request = new AIRequest
{
    Prompt = prompt,                          // Complete prompt template, including task instructions and constraints
    WorkingDirectory = projectPath ?? GetTempDirectory(), // Working directory, ensuring the AI runs in the correct project context
    AllowedTools = allowedTools,               // Allowed tool set
    PermissionMode = PermissionMode.bypassPermissions, // Bypass permission checks so Git operations can run directly
    LanguagePreference = languagePreference         // Language preference setting, ensuring commit messages match user expectations
};
```

ที่นี่เราใช้ `PermissionMode.bypassPermissions`ซึ่งช่วยให้ AI สามารถรันคำสั่ง Git ได้โดยตรงโดยไม่ต้องมีการยืนยันจากผู้ใช้ นี่เป็นหัวใจสำคัญของการออกแบบฟีเจอร์ แต่ยังต้องมีการตรวจสอบอินพุตอย่างเข้มงวดเพื่อป้องกันการละเมิด ในการปรับใช้การใช้งานจริงของ HagiCode เรารับประกันความปลอดภัยของกลไกนี้ผ่านการตรวจสอบพารามิเตอร์แบ็กเอนด์และการตรวจสอบบันทึก

#### ยอมรับการแยกวิเคราะห์ผลลัพธ์

หลังจากที่ AI ดำเนินการเสร็จสิ้น ก็จะส่งกลับผลลัพธ์ที่มีโครงสร้าง เราใช้กลยุทธ์การแยกวิเคราะห์แบบคู่เพื่อให้มั่นใจถึงความเข้ากันได้:

```csharp
/// <summary>
/// Parse commit execution results returned by the AI, supporting both delimiter format and regex format
/// </summary>
/// <param name="aiResponse">Raw response content returned by the AI</param>
/// <returns>A parsed list of commit results, where each result includes the commit hash and execution status</returns>
private List<CommitResultDto> ParseCommitExecutionResults(string aiResponse)
{
    var results = new List<CommitResultDto>();

    // Prefer delimiter-based parsing (new format), which is more explicit and reliable
    if (aiResponse.Contains("---"))
    {
        logger.LogDebug("Using delimiter-based parsing for AI response");
        results = ParseDelimitedFormat(aiResponse);

        if (results.Count > 0)
        {
            return results; // Successfully parsed, return the results directly
        }

        logger.LogWarning("Delimiter-based parsing produced no results, falling back to regex");
    }
    else
    {
        logger.LogDebug("No delimiter found, using legacy regex-based parsing");
    }

    // Fall back to regex parsing (old format) to ensure backward compatibility
    return ParseLegacyFormat(aiResponse);
}
```

ใช้รูปแบบตัวคั่น `---` เพื่อแยก Commit ออกจากกัน ทำให้โครงสร้างชัดเจนและง่ายต่อการแยกวิเคราะห์:

```
---
Commit 1: abc123def456
feat(auth): add user login functionality

Implement JWT-based authentication with login form and API endpoints.

Co-Authored-By: Hagicode <noreply@hagicode.com>
---
Commit 2: 789ghi012jkl
docs(readme): update installation instructions

Add new setup steps for Docker environment.

Co-Authored-By: Hagicode <noreply@hagicode.com>
---
```

รูปแบบนี้ทำให้การแยกวิเคราะห์ง่ายและเชื่อถือได้ ในขณะที่มนุษย์ยังอ่านได้ง่ายอีกด้วย

### กลไกการจัดการล็อค

เพื่อป้องกันความขัดแย้งของรัฐที่เกิดจากการดำเนินการพร้อมกัน เราได้ใช้กลไกการล็อคที่เก็บข้อมูล:

```csharp
// Acquire the repository lock to prevent concurrent operations
// Parameter notes:
// - fullPath: full repository path, used to identify different repository instances
// - requestedBy: requester identifier, used for tracking and logging
await _autoComposeLockService.AcquireLockAsync(fullPath, requestedBy);

try
{
    // Execute the AI Compose Commit operation
    // This section calls an Orleans Grain method to perform the actual AI processing and Git operations
    await aiGrain.AutoComposeCommitAsync(projectId, unstagedFiles, projectPath);
}
finally
{
    // Ensure the lock is released whether the operation succeeds or fails
    // Using a finally block guarantees lock release even when exceptions occur, preventing deadlocks
    await _autoComposeLockService.ReleaseLockAsync(fullPath);
}
```

การล็อคจะมีระยะหมดเวลา 20 นาที ซึ่งตรงกับระยะหมดเวลาที่ใช้สำหรับการดำเนินการของ AI หากการดำเนินการล้มเหลวหรือหมดเวลา ระบบจะปลดล็อกโดยอัตโนมัติเพื่อหลีกเลี่ยงการบล็อกถาวร ในการใช้งาน HagiCode จริง เราพบว่ากลไกการล็อกนี้มีความสำคัญอย่างยิ่ง โดยเฉพาะอย่างยิ่งในสภาพแวดล้อมการทำงานร่วมกันซึ่งนักพัฒนาหลายคนอาจเรียกใช้ AI Compose Commit ในเวลาเดียวกัน

### การแจ้งเตือนแบบเรียลไทม์ของ SignalR

หลังจากการประมวลผลเสร็จสิ้น ระบบจะส่งการแจ้งเตือนไปยังส่วนหน้าผ่าน SignalR:

```csharp
/// <summary>
/// Send a notification when automatic commit composition is complete
/// </summary>
/// <param name="projectId">Project identifier, used to route the notification to the correct client</param>
/// <param name="totalCount">Total number of commits, including successes and failures</param>
/// <param name="successCount">Number of successful commits</param>
/// <param name="failureCount">Number of failed commits</param>
/// <param name="success">Whether the overall operation succeeded</param>
/// <param name="error">Error message (if the operation failed)</param>
private async Task SendAutoComposeCommitNotificationAsync(
    string projectId,
    int totalCount,
    int successCount,
    int failureCount,
    bool success,
    string? error)
{
    try
    {
        // Build the notification DTO containing detailed execution results
        var notification = new AutoComposeCommitCompletedDto
        {
            ProjectId = projectId,
            TotalCount = totalCount,
            SuccessCount = successCount,
            FailureCount = failureCount,
            Success = success,
            Error = error
        };

        // Broadcast the notification to all connected clients through the SignalR Hub
        await messageService.SendAutoComposeCommitCompletedAsync(notification);

        logger.LogInformation(
            "Auto compose commit notification sent for project {ProjectId}: {SuccessCount}/{TotalCount} succeeded",
            projectId, successCount, totalCount);
    }
    catch (Exception ex)
    {
        // Log notification errors without affecting the main operation flow
        // A notification failure should not cause the entire operation to fail
        logger.LogError(ex, "Failed to send auto compose commit notification for project {ProjectId}", projectId);
    }
}
```

หลังจากที่ส่วนหน้าได้รับการแจ้งเตือน ก็สามารถอัปเดต UI เพื่อแสดงว่าการคอมมิตสำเร็จหรือล้มเหลว เพื่อปรับปรุงประสบการณ์ผู้ใช้ กลไกการตอบรับแบบเรียลไทม์นี้ได้รับการตอบรับอย่างดีจากผู้ใช้ HagiCode ซึ่งสามารถมองเห็นได้ชัดเจนว่าการดำเนินการเสร็จสิ้นเมื่อใดและผลลัพธ์เป็นอย่างไร

หากเกิดข้อผิดพลาดระหว่างการประมวลผล AI ระบบจะดำเนินการย้อนกลับและยกเลิกขั้นตอนไฟล์ที่จัดฉากไว้แล้ว เพื่อป้องกันไม่ให้สถานะที่ไม่สอดคล้องถูกทิ้งไว้ข้างหลัง ในการใช้งาน HagiCode จริง กลไกนี้ช่วยให้เรารอดจากการขัดจังหวะที่ไม่คาดคิดหลายครั้ง และรับประกันความสมบูรณ์ของสถานะพื้นที่เก็บข้อมูล

#### 3. การจัดการการหมดเวลา

การหมดเวลา 20 นาทีช่วยให้แน่ใจว่าการดำเนินการที่ใช้เวลานานจะไม่บล็อกทรัพยากรอย่างไม่มีกำหนด หลังจากหมดเวลา ระบบจะปลดล็อคและแจ้งให้ผู้ใช้ทราบว่าการดำเนินการล้มเหลว ในการใช้งาน HagiCode จริง เราพบว่าการดำเนินการส่วนใหญ่จะเสร็จสิ้นภายใน 2 ถึง 5 นาที และมีเพียงชุดการเปลี่ยนแปลงขนาดใหญ่มากเท่านั้นที่เข้าใกล้ขีดจำกัดการหมดเวลา

## แนวทางปฏิบัติที่ดีที่สุด

### แนวทางปฏิบัติที่ดีที่สุดสำหรับการใช้ AI Compose Commit

#### 1. เลือกช่วงเวลาที่เหมาะสม

AI Compose Commit เหมาะที่สุดสำหรับสถานการณ์ต่อไปนี้:

- เมื่อสิ้นสุดวันทำงาน เมื่อคุณต้องการประมวลผลการเปลี่ยนแปลงในไฟล์จำนวนมากในชุดเดียว
- หลังจากการดำเนินการปรับโครงสร้างใหม่ เมื่อไฟล์ที่เกี่ยวข้องหลายไฟล์จำเป็นต้องถูกคอมมิตแยกกัน
- หลังจากคุณลักษณะเสร็จสมบูรณ์ เมื่อการเปลี่ยนแปลงที่เกี่ยวข้องจำเป็นต้องจัดกลุ่มเป็นคอมมิต

ไม่เหมาะกับสถานการณ์ต่อไปนี้:

- คอมมิตด่วนสำหรับไฟล์เดียว (คอมมิตปกติจะเร็วกว่า)
- สถานการณ์ที่ต้องการการควบคุมเนื้อหาที่คอมมิตอย่างแม่นยำ
- คอมมิตที่มีข้อมูลที่ละเอียดอ่อนซึ่งต้องมีการตรวจสอบโดยเจ้าหน้าที่

#### 2. ตรวจสอบข้อผูกพันที่สร้างโดย AI

แม้ว่าการจัดกลุ่มอัจฉริยะที่ขับเคลื่อนด้วย AI จะมีประสิทธิภาพ แต่นักพัฒนาควรตรวจสอบความมุ่งมั่นที่สร้างขึ้น:

- ตรวจสอบว่าการจัดกลุ่มตรงกับความคาดหวังหรือไม่
- ตรวจสอบความถูกต้องของข้อความยืนยัน
- ยืนยันว่าไม่มีไฟล์ใดถูกละเว้นหรือรวมไว้อย่างไม่ถูกต้อง

หากคุณพบการจัดกลุ่มที่ไม่สมเหตุสมผลคุณสามารถใช้ `git reset --soft HEAD~N` เพื่อยกเลิกและจัดกลุ่มใหม่ ประสบการณ์ของ HagiCode แสดงให้เห็นว่าแม้การจัดกลุ่ม AI จะฉลาด แต่การตรวจสอบโดยเจ้าหน้าที่ยังคงมีคุณค่า โดยเฉพาะอย่างยิ่งสำหรับการยืนยันคุณสมบัติที่สำคัญ

#### 3. สอดคล้องกับมาตรฐานโครงการ

ตรวจสอบให้แน่ใจว่าการกำหนดค่า Git ของโปรเจ็กต์ของคุณรองรับ Conventional Commits:

```bash
# Install commitlint
npm install -g @commitlint/cli @commitlint/config-conventional

# Configure commitlint
echo "module.exports = {extends: ['@commitlint/config-conventional']}" > commitlint.config.js
```

ซึ่งช่วยให้คุณตรวจสอบรูปแบบข้อความคอมมิตในเวิร์กโฟลว์ CI/CD และรักษาให้สอดคล้องกับรูปแบบที่สร้างโดย AI Compose Commit

### คำแนะนำสำหรับการสร้างคุณลักษณะที่คล้ายกัน

หากคุณต้องการใช้คุณสมบัติการคอมมิตที่ได้รับความช่วยเหลือจาก AI ที่คล้ายกันในโปรเจ็กต์ของคุณเอง นี่คือคำแนะนำของเรา:

#### 1. เริ่มต้นจากสิ่งเล็กๆ

เริ่มต้นด้วยการสร้างข้อความคอมมิตเดี่ยว จากนั้นค่อยๆ ขยายไปสู่การจัดกลุ่มคอมมิตหลายรายการ ทำให้ง่ายต่อการตรวจสอบและทำซ้ำ HagiCode ดำเนินตามเส้นทางเดียวกัน: เวอร์ชันแรกๆ รองรับเฉพาะการคอมมิตเดี่ยว และต่อมาขยายไปสู่การจัดกลุ่มอัจฉริยะผ่านคอมมิตหลายรายการ

#### 2. ใช้ AI SDK ที่สมบูรณ์

อย่าใช้ตรรกะการร้องขอ AI ตั้งแต่เริ่มต้น การใช้ SDK ที่มีอยู่จะช่วยลดเวลาในการพัฒนาและข้อบกพร่องที่อาจเกิดขึ้น เราใช้บริการ Claude Helper ซึ่งมีอินเทอร์เฟซที่เสถียรและการจัดการข้อผิดพลาดที่มีประสิทธิภาพ

#### 3. ลงทุนในการออกแบบพร้อมท์

คุณภาพที่รวดเร็วจะกำหนดคุณภาพผลผลิตโดยตรง ใช้เวลาในการออกแบบพร้อมท์โดยละเอียด รวมถึง:

- ล้างคำอธิบายงาน
- ข้อกำหนดรูปแบบเอาต์พุตเฉพาะ
- กฎสำหรับการจัดการกรณีขอบ
- ตัวอย่างภาพประกอบ

HagiCode ลงทุนมหาศาลในการออกแบบที่รวดเร็ว และนี่เป็นหนึ่งในเหตุผลสำคัญที่ทำให้ฟีเจอร์นี้ประสบความสำเร็จ

#### 4. ใช้การจัดการข้อผิดพลาดที่ครอบคลุม

การดำเนินงานของ AI อาจล้มเหลวได้จากหลายสาเหตุ เช่น ปัญหาเครือข่าย ขีดจำกัดอัตรา API หรือการกลั่นกรองเนื้อหา ตรวจสอบให้แน่ใจว่าระบบของคุณสามารถจัดการกับข้อผิดพลาดเหล่านี้ได้อย่างสง่างามและให้ข้อมูลข้อผิดพลาดที่มีความหมาย

#### 5. จัดให้มีกลไกการแทรกแซงด้วยตนเอง

อย่าทำให้ทุกอย่างเป็นอัตโนมัติอย่างสมบูรณ์ ปล่อยให้ผู้ใช้ควบคุม มีตัวเลือกในการตรวจสอบผลลัพธ์การจัดกลุ่ม ปรับกลุ่ม และแก้ไขข้อความยืนยันด้วยตนเองเพื่อสร้างสมดุลระหว่างระบบอัตโนมัติและความยืดหยุ่น แม้ว่า HagiCode จะรองรับการดำเนินการอัตโนมัติ แต่ยังคงรักษาความสามารถในการแสดงตัวอย่างและการปรับแต่งไว้

### เคล็ดลับการเพิ่มประสิทธิภาพ

#### 1. การกรองไฟล์

เมื่อสร้างบริบทของไฟล์ ให้กรองไฟล์ที่ไม่ต้องการการวิเคราะห์ AI ออก:

```csharp
// Filter out generated files and excessively large files to reduce the AI processing burden
var relevantFiles = stagedFiles
    .Where(f => !IsGeneratedFile(f.Path))
    .Where(f => !IsLargeFile(f.Path))
    .ToArray();
```

#### 2. การประมวลผลแบบขนาน

หากรองรับที่เก็บข้อมูลอิสระหลายรายการ คอมมิตในที่เก็บข้อมูลที่แตกต่างกันสามารถประมวลผลพร้อมกันได้เพื่อปรับปรุงประสิทธิภาพโดยรวม

#### 3. การเพิ่มประสิทธิภาพแคช

โปรเจ็กต์แคชยอมรับผลการวิเคราะห์ประวัติเพื่อหลีกเลี่ยงการวิเคราะห์ซ้ำทุกครั้ง การตั้งค่ารูปแบบในอดีตสามารถจัดเก็บไว้ในไฟล์การกำหนดค่าเพื่อลดการเรียก AI

## บทสรุป

AI Compose Commit แสดงถึงการประยุกต์ใช้เทคโนโลยี AI ในเชิงลึกในเครื่องมือพัฒนาซอฟต์แวร์ ด้วยการวิเคราะห์การเปลี่ยนแปลงไฟล์อย่างชาญฉลาด จัดกลุ่มคอมมิตโดยอัตโนมัติ และสร้างข้อความคอมมิตที่เป็นไปตามมาตรฐาน จะช่วยปรับปรุงประสิทธิภาพของเวิร์กโฟลว์ Git ได้อย่างมาก และช่วยให้นักพัฒนามุ่งเน้นไปที่งานเขียนโค้ดหลักได้มากขึ้น

ในระหว่างการดำเนินการ เราได้เรียนรู้บทเรียนสำคัญหลายประการ:

1. **ความคิดเห็นของผู้ใช้เป็นสิ่งสำคัญ**: เวอร์ชันก่อนหน้าใช้การรอแบบซิงโครนัส และผู้ใช้รายงานว่าได้รับประสบการณ์ที่ไม่ดี หลังจากเปลี่ยนมาใช้โมเดล Fire and Forget ความพึงพอใจก็ดีขึ้นอย่างเห็นได้ชัด
2. **การออกแบบที่รวดเร็วจะกำหนดคุณภาพ**: ข้อความแจ้งที่ได้รับการออกแบบอย่างระมัดระวังจะช่วยรับประกันคุณภาพเอาต์พุตของ AI ได้ดีกว่าอัลกอริทึมที่ซับซ้อน
3. **ความปลอดภัยต้องมาก่อนเสมอ**: การให้สิทธิ์ AI เพื่อดำเนินการคำสั่ง Git จะช่วยเพิ่มประสิทธิภาพได้โดยตรง แต่จะต้องจับคู่กับข้อจำกัดและการตรวจสอบที่เข้มงวด
4. **การปรับปรุงแบบก้าวหน้าได้ผลดีที่สุด**: การเริ่มต้นด้วยสถานการณ์ง่ายๆ และค่อยๆ เพิ่มความซับซ้อนมีแนวโน้มที่จะประสบความสำเร็จมากกว่าการพยายามปรับใช้ทุกอย่างในคราวเดียว

ในอนาคต เราวางแผนที่จะเพิ่มประสิทธิภาพ AI Composed Commit เพิ่มเติม ซึ่งรวมถึง:

- สนับสนุนกลยุทธ์การจัดกลุ่มคอมมิตเพิ่มเติม (ตามเวลา โดยนักพัฒนา และอื่นๆ)
- การรวมเวิร์กโฟลว์การตรวจสอบโค้ดเพื่อกระตุ้นการตรวจสอบโดยอัตโนมัติก่อนที่จะคอมมิต
- รองรับเทมเพลตข้อความคอมมิตแบบกำหนดเองเพื่อตอบสนองความต้องการส่วนบุคคลของโปรเจ็กต์ต่างๆ

หากคุณพบว่าแนวทางที่แบ่งปันในบทความนี้มีประโยชน์ ให้ลองใช้ HagiCode และสัมผัสประสบการณ์ว่าฟีเจอร์นี้ทำงานอย่างไรในการพัฒนาจริง ท้ายที่สุดแล้ว การฝึกฝนเป็นเพียงเกณฑ์เดียวในการทดสอบความจริง
