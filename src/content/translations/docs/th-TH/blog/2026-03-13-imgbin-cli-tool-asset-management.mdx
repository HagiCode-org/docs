## การออกแบบเครื่องมือ ImgBin CLI: แนวทางการจัดการสินทรัพย์รูปภาพของ HagiCode

> บทความนี้จะอธิบายวิธีสร้างไปป์ไลน์สินทรัพย์รูปภาพอัตโนมัติตั้งแต่ต้น ครอบคลุมการออกแบบเครื่องมือ CLI สถาปัตยกรรมอะแดปเตอร์ของผู้ให้บริการ และกลยุทธ์การจัดการข้อมูลเมตา

## พื้นหลัง

จริงๆ แล้ว ฉันไม่ได้คาดหวังว่าการจัดการเนื้อหารูปภาพจะทำให้เรายุ่งวุ่นวายกันไปนานขนาดนี้

ในระหว่างการพัฒนา HagiCode เราประสบปัญหาที่ดูเรียบง่ายเมื่อดูเผินๆ แต่ในทางปฏิบัติกลับยุ่งยากอย่างน่าประหลาดใจ นั่นก็คือ การสร้างและการจัดการเนื้อหารูปภาพ ในทางหนึ่ง มันเหมือนกับละครของวัยรุ่น ภายนอกสงบ ความวุ่นวายอยู่ข้างใต้

เนื่องจากโปรเจ็กต์รวบรวมเอกสารและสื่อการตลาดมากขึ้น เราจึงจำเป็นต้องมีรูปภาพสนับสนุนจำนวนมาก บางส่วนต้องสร้างขึ้นโดย AI บางส่วนต้องเลือกจากคลังสินทรัพย์ที่มีอยู่ และบางส่วนจำเป็นต้องจดจำ AI พร้อมการติดป้ายกำกับอัตโนมัติ ปัญหาคือว่าทั้งหมดนี้ได้รับการจัดการมานานแล้วผ่านสคริปต์ที่กระจัดกระจายและขั้นตอนที่ต้องดำเนินการด้วยตนเอง ทุกครั้งที่เราสร้างภาพ เราต้องเรียกใช้สคริปต์ด้วยมือ จัดระเบียบข้อมูลเมตาด้วยมือ และสร้างภาพขนาดย่อด้วยมือ เพียงอย่างเดียวก็น่ารำคาญมากพอแล้ว แต่ปัญหาที่ใหญ่กว่าก็คือทุกอย่างกระจัดกระจายไปทั่ว เมื่อเราต้องการที่จะหาบางสิ่งบางอย่างเราไม่สามารถ เมื่อเราจำเป็นต้องนำบางสิ่งบางอย่างกลับมาใช้ใหม่ เราก็ทำไม่ได้

จุดปวดเป็นรูปธรรม:

1. **ไม่มีจุดเริ่มต้นแบบรวม**: ตรรกะสำหรับการสร้างอิมเมจกระจายไปตามสคริปต์ต่างๆ ดังนั้นการประมวลผลแบบกลุ่มจึงเป็นไปไม่ได้โดยทั่วไป
2. **ข้อมูลเมตาหายไป**: รูปภาพที่สร้างขึ้นไม่มีการรวมเป็นหนึ่งเดียว `metadata.json`ซึ่งหมายความว่าไม่มีความสามารถในการค้นหาหรือตรวจสอบย้อนกลับที่เชื่อถือได้
3. **ต้นทุนการจัดการด้วยตนเองสูง**: ต้องแยกชื่อและแท็กทีละรายการซึ่งไม่มีประสิทธิภาพ
4. **ไม่มีระบบอัตโนมัติ**: สร้างเนื้อหาภาพในไปป์ไลน์ CI/CD โดยอัตโนมัติหรือไม่ ไม่ใช่โอกาส

เราเคยคิดที่จะทิ้งมันไว้ตามลำพัง แต่โครงการยังคงต้องเดินหน้าต่อไป เนื่องจากเราไม่สามารถหลีกเลี่ยงปัญหาได้ เราจึงคิดว่าเราจะแก้ไขปัญหานั้นได้เช่นกัน ดังนั้นเราจึงตัดสินใจอัปเกรด ImgBin จากชุดสคริปต์ที่กระจัดกระจายไปเป็นไปป์ไลน์สินทรัพย์รูปภาพที่สามารถดำเนินการได้โดยอัตโนมัติ ปัญหาบางอย่างไม่ได้หายไปเพียงเพราะคุณมองข้ามไป

## เกี่ยวกับ ฮากิโค้ด

แนวทางที่แบ่งปันในบทความนี้มาจากประสบการณ์ตรงของเราใน [ฮากิโค้ด](https://hagicode.com) โครงการ HagiCode เป็นโปรเจ็กต์ผู้ช่วยเขียนโค้ด AI ที่ดูแลองค์ประกอบหลายอย่างพร้อมกัน รวมถึงส่วนขยาย VSCode บริการ AI แบ็กเอนด์ และไคลเอนต์เดสก์ท็อปข้ามแพลตฟอร์ม ในสภาพแวดล้อมข้ามแพลตฟอร์มที่ซับซ้อน หลายภาษา เช่นนี้ การจัดการสินทรัพย์รูปภาพที่ได้มาตรฐานกลายเป็นส่วนสำคัญในการปรับปรุงประสิทธิภาพการพัฒนา

คุณสามารถพูดได้ว่านี่เป็นหนึ่งในความเจ็บปวดเล็กๆ น้อยๆ ในการเดินทางของ HagiCode ทุกโปรเจ็กต์มีช่วงเวลาแบบนั้น: ปัญหาเล็กๆ น้อยๆ ที่ดูไม่สำคัญ แต่ก็กินเวลาไปครึ่งวัน

ระบบการสร้างของ HagiCode ขึ้นอยู่กับระบบนิเวศของ TypeScript + Node.js ดังนั้น ImgBin จึงนำ Tech Stack เดียวกันมาใช้เพื่อรักษาความสอดคล้องทางเทคนิคของโปรเจ็กต์ เมื่อคุณคุ้นเคยกับสแต็กเดียวแล้ว การเปลี่ยนไปใช้สแต็กอื่นก็รู้สึกเหมือนเป็นปัญหาที่ไม่จำเป็น

## การออกแบบหลัก

### สถาปัตยกรรมโดยรวม

ImgBin ใช้สถาปัตยกรรมแบบเลเยอร์ที่แยกคำสั่ง CLI บริการแอปพลิเคชัน อะแดปเตอร์ API ของบริษัทอื่น และเลเยอร์โครงสร้างพื้นฐานออกจากกันอย่างชัดเจน:

```
Component hierarchy
├── CLI Entry (cli.ts)              Global argument parsing, command routing
├── Commands (commands/*)           generate | batch | annotate | thumbnail
├── Application Services            job-runner | metadata | thumbnail | asset-writer
├── Provider Adapters               image-api-provider | vision-api-provider
└── Infrastructure Layer            config | logger | paths | schema
```

ประโยชน์ของการออกแบบแบบหลายชั้นนี้คือขอบเขตความรับผิดชอบที่ชัดเจน นอกจากนี้ยังทำให้การทดสอบง่ายขึ้นเนื่องจากการขึ้นต่อกันภายนอกสามารถจำลองได้อย่างหมดจด ในทางปฏิบัติ มันเพียงหมายความว่าแต่ละเลเยอร์ทำงานของตัวเองโดยไม่ไปขัดขวางเลเยอร์อื่น ดังนั้นเมื่อมีบางอย่างพัง จะง่ายกว่าที่จะเข้าใจว่าทำไม

### รูปแบบไดเรกทอรีสินทรัพย์เดียว

ImgBin ใช้โมเดล "หนึ่งเนื้อหา หนึ่งไดเรกทอรี" ทุกครั้งที่สร้างภาพ มันจะสร้างโครงสร้างดังนี้:

```
library/
└── 2026-03/
    └── orange-dashboard/
        ├── original.png      # Original image
        ├── thumbnail.webp    # 512x512 thumbnail
        └── metadata.json     # Structured metadata
```

ข้อดีของรุ่นนี้คือ:

1. **มีอยู่ในตัวเอง**: ไฟล์ทั้งหมดสำหรับเนื้อหาเดียวอยู่ในไดเร็กทอรีเดียวกัน ทำให้การโยกย้ายและการสำรองข้อมูลสะดวก
2. **ติดตามได้**: `metadata.json` ทำให้สามารถติดตามเวลาในการสร้าง พรอมต์ โมเดล และรายละเอียดอื่นๆ ได้
3. **ขยายได้**: หากต้องการรูปแบบเพิ่มเติมในภายหลัง เช่น ภาพขนาดย่อที่มีหลายขนาด เราก็สามารถเพิ่มไฟล์ใหม่ในไดเรกทอรีเดียวกันได้

สิ่งสวยงามไม่จำเป็นต้องครอบครองเสมอไป บางครั้งมันก็เพียงพอแล้วที่พวกมันจะยังคงสวยงาม และคุณสามารถชื่นชมพวกมันอย่างเงียบ ๆ ได้ อาจฟังดูห่างไกลสักหน่อย แต่ตรรกะยังคงอยู่: เมื่อเก็บภาพไว้ด้วยกัน ก็จะดูน่าดูยิ่งขึ้นและค้นหาได้ง่ายกว่ามาก

### การจัดเก็บข้อมูลเมตาแบบชั้น

`metadata.json` เป็นแกนหลักของระบบทั้งหมด ใช้กลยุทธ์การจัดเก็บข้อมูลแบบเลเยอร์ที่แบ่งฟิลด์ออกเป็นสามประเภท:

```json
{
  "schemaVersion": 2,
  "assetId": "orange-dashboard",
  "slug": "orange-dashboard",
  "title": "Orange Dashboard",
  "tags": ["dashboard", "hero", "orange"],

  "source": { "type": "generated" },

  "paths": {
    "assetDir": "library/2026-03/orange-dashboard",
    "original": "original.png",
    "thumbnail": "thumbnail.webp"
  },

  "generated": {
    "prompt": "orange dashboard for docs hero",
    "provider": "azure-openai-image-api",
    "model": "gpt-image-1.5"
  },

  "recognized": {
    "title": "Orange Dashboard",
    "tags": ["dashboard", "ui", "orange"],
    "description": "A modern orange dashboard with charts and metrics"
  },

  "status": {
    "generation": "succeeded",
    "recognition": "succeeded",
    "thumbnail": "succeeded"
  },

  "timestamps": {
    "createdAt": "2026-03-11T04:01:19.570Z",
    "updatedAt": "2026-03-11T04:02:09.132Z"
  }
}
```

- **สร้าง**: บันทึกข้อมูลต้นฉบับจากการสร้างภาพ เช่น พรอมต์ ผู้ให้บริการ และโมเดล
- **รู้จัก**: เก็บผลลัพธ์การจดจำ AI เช่น ชื่อ แท็ก และคำอธิบายที่สร้างขึ้นอัตโนมัติ
- **คู่มือ**: จัดเก็บผลลัพธ์ที่ดูแลจัดการด้วยตนเอง ข้อมูลในพื้นที่นี้มีลำดับความสำคัญสูงสุด และจะไม่ถูกเขียนทับโดยการรับรู้ของ AI

กลยุทธ์แบบแบ่งชั้นนี้แก้ไขข้อขัดแย้งหลักข้อหนึ่งของเราก่อนหน้านี้: เมื่อการจดจำ AI และการดูแลจัดการด้วยตนเองไม่ตรงกัน ข้อใดควรชนะ คำตอบคือการป้อนข้อมูลด้วยตนเอง การจดจำ AI มีไว้เพื่อช่วยเหลือ ไม่ใช่เพื่อการตัดสินใจ คำถามนั้นชัดเจนมากขึ้นเมื่อเวลาผ่านไป เครื่องจักรยังคงเป็นเครื่องจักร และในท้ายที่สุด ผู้คนก็ยังจำเป็นต้องโทรออก

## รูปแบบอะแดปเตอร์ของผู้ให้บริการ

ส่วนหลักอีกประการหนึ่งของ ImgBin คือรูปแบบอะแดปเตอร์ของผู้ให้บริการ เราสรุป API ภายนอกไว้เบื้องหลังอินเทอร์เฟซแบบรวม ดังนั้นแม้ว่าเราจะเปลี่ยนผู้ให้บริการ AI เราก็ไม่จำเป็นต้องเปลี่ยนตรรกะทางธุรกิจ

ในทางหนึ่ง มันก็เหมือนกับความสัมพันธ์เล็กน้อย รูปร่างภายนอกสามารถเปลี่ยนแปลงได้ แต่สิ่งสำคัญคือโครงสร้างภายในยังคงเหมือนเดิม เมื่ออินเทอร์เฟซได้รับการแก้ไขแล้ว การใช้งานภายในอาจแตกต่างกันได้อย่างอิสระ

### ผู้ให้บริการสร้างภาพ

```typescript
interface ImageGenerationProvider {
  // Generate an image and return its Buffer
  generate(options: GenerateOptions): Promise<Buffer>;

  // Get the list of supported models
  getSupportedModels(): Promise<string[]>;
}

interface GenerateOptions {
  prompt: string;
  model?: string;
  size?: '1024x1024' | '1792x1024' | '1024x1792';
  quality?: 'standard' | 'hd';
  format?: 'png' | 'webp' | 'jpeg';
}
```

### ผู้ให้บริการการรับรู้วิสัยทัศน์

```typescript
interface VisionRecognitionProvider {
  // Recognize image content and return structured metadata
  recognize(imageBuffer: Buffer): Promise<RecognitionResult>;

  // Get the list of supported models
  getSupportedModels(): Promise<string[]>;
}

interface RecognitionResult {
  title?: string;
  tags: string[];
  description?: string;
  confidence: number;
}
```

ข้อดีของการออกแบบอินเทอร์เฟซนี้คือ:

1. **ทดสอบได้**: ในการทดสอบหน่วย เราสามารถส่งผ่านผู้ให้บริการจำลอง แทนที่จะทำการเรียก API ภายนอกจริง
2. **ขยายได้**: การเพิ่มผู้ให้บริการใหม่ต้องใช้อินเทอร์เฟซเท่านั้น ไม่จำเป็นต้องเปลี่ยนรหัสผู้โทร
3. **ทดแทนได้**: การผลิตสามารถใช้ Azure OpenAI ในขณะที่การทดสอบสามารถใช้โมเดลในเครื่องได้ โดยการกำหนดค่าเป็นเพียงสิ่งเดียวที่เปลี่ยนแปลง

บางครั้งงานโครงการก็รู้สึกเช่นนั้นเช่นกัน ภายนอกดูเหมือนว่าเราเพิ่งสลับ API แต่ตรรกะภายในยังคงเหมือนเดิมทุกประการ และนั่นทำให้สิ่งทั้งหมดน่ากลัวน้อยลงมาก

## การออกแบบคำสั่ง CLI

ImgBin มีคำสั่งหลักสี่คำสั่งเพื่อให้ครอบคลุมสถานการณ์การใช้งานที่แตกต่างกัน:

### สร้าง: การสร้างภาพเดียว

```bash
# Simplest usage
imgbin generate --prompt "orange dashboard for docs hero"

# Generate a thumbnail and AI annotations at the same time
imgbin generate --prompt "orange dashboard" --annotate --thumbnail

# Specify an output directory
imgbin generate --prompt "orange dashboard" --output ./library
```

### แบทช์: งานแบทช์

งานแบทช์ถูกกำหนดผ่านไฟล์ Manifest ของ YAML หรือ JSON ซึ่งทำให้เหมาะสำหรับเวิร์กโฟลว์ CI/CD:

```yaml
# assets/jobs/launch.yaml
defaults:
  annotate: true
  thumbnail: true
  libraryRoot: ./library

jobs:
  - prompt: "orange dashboard hero"
    slug: orange-dashboard
    tags: [dashboard, hero, orange]

  - prompt: "pricing grid for docs"
    slug: pricing-grid
    tags: [pricing, grid, docs]
```

รันคำสั่ง:

```bash
imgbin batch assets/jobs/launch.yaml
```

การออกแบบงานแบทช์รองรับการแยกความล้มเหลว: รายการในรายการจะได้รับการประมวลผลทีละรายการ และความล้มเหลวในรายการหนึ่งจะไม่ส่งผลกระทบต่อรายการอื่นๆ คุณยังสามารถดูตัวอย่างงานได้ด้วย `--dry-run` โดยไม่ต้องดำเนินการจริง

และส่วนที่ดีที่สุดคือมันจะบอกคุณอย่างชัดเจนว่าอะไรสำเร็จและอะไรล้มเหลว ต่างจากบางสิ่งในชีวิตที่ความล้มเหลวเกิดขึ้นแต่คุณไม่รู้ด้วยซ้ำว่ามันเกิดขึ้นได้อย่างไร

### คำอธิบายประกอบ: คำอธิบายประกอบ AI

เรียกใช้การจดจำ AI บนรูปภาพที่มีอยู่เพื่อสร้างชื่อ แท็ก และคำอธิบายโดยอัตโนมัติ:

```bash
# Annotate a single image
imgbin annotate ./library/2026-03/orange-dashboard

# Annotate an entire directory in batch
imgbin annotate ./library/2026-03/
```

### ภาพขนาดย่อ: การสร้างภาพขนาดย่อ

สร้างภาพขนาดย่อสำหรับภาพที่มีอยู่:

```bash
# Generate a thumbnail
imgbin thumbnail ./library/2026-03/orange-dashboard
```

## การออกแบบรายการงานแบทช์

รูปแบบรายการสำหรับงานแบตช์รองรับการกำหนดค่าที่ยืดหยุ่น ค่าเริ่มต้นสามารถตั้งค่าได้ทั่วโลก และแต่ละงานสามารถแทนที่ได้:

```yaml
# Global defaults
defaults:
  annotate: true        # Enable AI annotation by default
  thumbnail: true       # Generate thumbnails by default
  libraryRoot: ./library
  model: gpt-image-1.5

jobs:
  # Minimal configuration: only provide a prompt
  - prompt: "first image"

  # Full configuration
  - prompt: "second image"
    slug: custom-slug
    tags: [tag1, tag2]
    annotate: false     # Do not run AI annotation for this job
    model: dall-e-3    # Use a different model for this job
```

เมื่อดำเนินการแล้ว ImgBin จะประมวลผลงานทีละรายการ ผลลัพธ์ของแต่ละงานจะถูกเขียนตามความเหมาะสม `metadata.json`. แม้ว่างานหนึ่งจะล้มเหลว แต่งานอื่นๆ ก็ไม่ได้รับผลกระทบ หลังจากงานทั้งหมดเสร็จสิ้น CLI จะแสดงรายงานสรุป:

```
✓ orange-dashboard (succeeded)
✓ pricing-grid (succeeded)
✗ hero-banner (failed: API rate limit exceeded)

2/3 succeeded, 1 failed
```

บางสิ่งบางอย่างไม่สามารถรีบเร่งได้ การพาพวกเขาไปทีละคนมักเป็นเส้นทางที่มั่นคงกว่า บางทีนั่นอาจเป็นปรัชญาเบื้องหลังงานแบบแบตช์

## การกำหนดค่าตัวแปรสภาพแวดล้อม

ImgBin รองรับการกำหนดค่าที่ยืดหยุ่นผ่านตัวแปรสภาพแวดล้อม:

```bash
# ImgBin working directory
IMGBIN_WORKDIR=/path/to/imgbin

# Executable path (for invocation inside scripts)
IMGBIN_EXECUTABLE=/path/to/imgbin/dist/cli.js

# Asset library root
IMGBIN_LIBRARY_ROOT=./.imgbin-library

# Azure OpenAI configuration (if using the Azure provider)
AZURE_OPENAI_ENDPOINT=https://your-resource.openai.azure.com/
AZURE_OPENAI_API_KEY=***
AZURE_OPENAI_IMAGE_DEPLOYMENT=gpt-image-1
```

การกำหนดค่าเป็นหนึ่งในสิ่งที่สามารถรู้สึกได้ทั้งสำคัญและไม่สำคัญในเวลาเดียวกัน ท้ายที่สุดแล้ว อะไรก็ตามที่ให้ความรู้สึกสบายใจและเหมาะกับขั้นตอนการทำงานของคุณมากที่สุดมักจะเป็นตัวเลือกที่ถูกต้อง

## หมายเหตุการใช้งาน

ในระหว่างการดำเนินการ เราได้สรุปประเด็นสำคัญบางประการ:

### การออกแบบส่วนต่อประสานของผู้ให้บริการ

คำจำกัดความของอินเทอร์เฟซควรมีความชัดเจนและครบถ้วน รวมถึงพารามิเตอร์อินพุต ค่าที่ส่งคืน และการจัดการข้อผิดพลาด เป็นความคิดที่ดีที่จะจัดเตรียมสไตล์การเรียกใช้ทั้งแบบซิงโครนัสและอะซิงโครนัสสำหรับสถานการณ์ที่แตกต่างกัน

นั่นเป็นประสบการณ์เล็กๆ น้อยๆ ที่ได้มาอย่างยากลำบาก เมื่อตั้งค่าอินเทอร์เฟซแล้ว จะไม่มีใครอยากเปลี่ยนแปลงในภายหลัง

### กลยุทธ์การจัดการความล้มเหลว

เมื่อรายการหนึ่งล้มเหลวในงานแบตช์ CLI ควร:

1. เขียนข้อมูลข้อผิดพลาดโดยละเอียดลงในไฟล์บันทึกแยกต่างหาก
2. ดำเนินงานอื่นๆ ต่อไป แทนที่จะขัดจังหวะกระบวนการทั้งหมด
3. ส่งกลับโค้ดทางออกที่ไม่ใช่ศูนย์ที่ส่วนท้ายเพื่อระบุว่างานบางงานล้มเหลว
4. แสดงผลการดำเนินการของทุกงานอย่างชัดเจนในรายงานสรุป

ความล้มเหลวบางอย่างเป็นเพียงความล้มเหลว ไม่มีประโยชน์ที่จะแสร้งทำเป็นอย่างอื่น เป็นการดีกว่าที่จะยอมรับพวกเขาอย่างเปิดเผยแล้วค่อยหาวิธีแก้ไข ตรรกะเดียวกันนี้ใช้ได้กับโครงการและชีวิต

### กลยุทธ์การรวมข้อมูลเมตา

มีการเขียนผลการรับรู้ถึง `recognized` ตามค่าเริ่มต้น ในขณะที่ช่องที่แก้ไขด้วยตนเองจะถูกทำเครื่องหมายไว้ `manual`. การอัปเดตข้อมูลเมตาเป็นไปตามกลยุทธ์แบบผนวกเท่านั้น: เว้นแต่ `--force` ถูกส่งผ่านอย่างชัดเจน ผลลัพธ์ที่ดูแลจัดการด้วยตนเองที่มีอยู่จะไม่ถูกเขียนทับ

ประเด็นนั้นก็ชัดเจนเช่นกัน - บางสิ่งเมื่อถูกเขียนทับก็หายไป มักจะดีกว่าที่จะเก็บรักษาไว้ เพราะบันทึกนั้นมีคุณค่า

### การสร้างไดเร็กทอรี Atomicity

ใช้ `fs.mkdir({ recursive: true })` เพื่อให้แน่ใจว่าการสร้างไดเร็กทอรียังคงเป็นแบบอะตอมมิกและเพื่อหลีกเลี่ยงสภาวะการแข่งขันในสถานการณ์ที่เกิดขึ้นพร้อมกัน

บางที นั่นอาจเป็นความรู้สึกของการรักษาความปลอดภัย เช่น มั่นคงเมื่อความเสถียรเป็นสิ่งสำคัญ เคลื่อนที่เร็วเมื่อความเร็วมีความสำคัญ และไม่เคยติดขัดเมื่อต้องคาดเดาซ้ำสอง

## บทสรุป

ในฐานะเครื่องมือหลักสำหรับการจัดการสินทรัพย์รูปภาพในโครงการ HagiCode ImgBin แก้ปัญหาของเราผ่านตัวเลือกการออกแบบต่อไปนี้:

1. **จุดเริ่มต้นแบบรวม**: CLI ครอบคลุมถึงการสร้าง คำอธิบายประกอบ ภาพขนาดย่อ และการดำเนินการหลักอื่นๆ ทั้งหมด
2. **ขับเคลื่อนด้วยข้อมูลเมตา**: ทุกเนื้อหามีความครบถ้วน `metadata.json`ทำให้สามารถค้นหาและติดตามได้
3. **อะแดปเตอร์ผู้ให้บริการ**: นามธรรมที่ยืดหยุ่นสำหรับ API ภายนอก ทำให้การทดสอบและการขยายง่ายขึ้น
4. **การสนับสนุนงานเป็นชุด**: การสร้างอิมเมจเป็นชุดสามารถทำได้อัตโนมัติภายในเวิร์กโฟลว์ CI/CD

ทุกสิ่งทุกอย่างอาจจางหายไป แต่แนวทางนี้กลับพิสูจน์แล้วว่ามีประโยชน์จริงๆ

โซลูชันนี้ไม่เพียงแต่ปรับปรุงประสิทธิภาพการพัฒนาของ HagiCode เท่านั้น แต่ยังสร้างกรอบการทำงานที่สามารถนำมาใช้ซ้ำได้สำหรับการจัดการสินทรัพย์รูปภาพอีกด้วย หากคุณกำลังสร้างโปรเจ็กต์ที่มีหลายองค์ประกอบคล้ายกัน ฉันเชื่อว่าแนวคิดการออกแบบของ ImgBin อาจให้แรงบันดาลใจแก่คุณได้

วัยหนุ่มสาวเป็นเรื่องของการลองทำสิ่งต่างๆ และสร้างความยุ่งวุ่นวายเล็กน้อย ถ้าคุณไม่เคยลองตัวเองผ่านเรื่องนั้น คุณจะรู้ได้อย่างไรว่าคุณมีความสามารถจริงๆ แค่ไหน?
