## การเพิ่มการสนับสนุนการปรับใช้อัตโนมัติของหน้า GitHub ให้กับ HagiCode

> โปรเจ็กต์นี้เดิมมีชื่อรหัสว่า PCode และปัจจุบันเปลี่ยนชื่อเป็น HagiCode อย่างเป็นทางการแล้ว บทความนี้บันทึกวิธีที่เราแนะนำการปรับใช้ไซต์คงที่แบบอัตโนมัติ เพื่อให้การเผยแพร่เนื้อหากลายเป็นเรื่องง่ายเหมือนกับน้ำดื่ม

## ความเป็นมา / บทนำ

ในระหว่างการพัฒนา HagiCode เราประสบปัญหาในทางปฏิบัติ เนื่องจากเอกสารและข้อเสนอของเราเติบโตขึ้นเรื่อยๆ การจัดการและการนำเสนอเนื้อหาจึงกลายเป็นเรื่องเร่งด่วนมากขึ้นอย่างมีประสิทธิภาพ เราตัดสินใจใช้ GitHub Pages เพื่อโฮสต์ไซต์แบบคงที่ของเรา แต่การสร้างและปรับใช้ด้วยตนเองนั้นยุ่งยากเกินไป การเปลี่ยนแปลงทุกครั้งจำเป็นต้องมีโครงสร้างภายในเครื่อง บรรจุภัณฑ์ จากนั้นจึงดำเนินการด้วยตนเอง `gh-pages` สาขา นั่นไม่เพียงไม่มีประสิทธิภาพเท่านั้น แต่ยังผิดพลาดได้ง่ายอีกด้วย

เพื่อแก้ไขปัญหานี้ สาเหตุหลักมาจากเราต้องการประหยัดความพยายาม เราจึงจำเป็นต้องมีเวิร์กโฟลว์การปรับใช้แบบอัตโนมัติ บทความนี้บันทึกรายละเอียดวิธีที่เราเพิ่มการสนับสนุนการปรับใช้อัตโนมัติตาม GitHub Actions ให้กับโปรเจ็กต์ HagiCode เพื่อให้เรามุ่งเน้นไปที่การเขียนเนื้อหาและให้ระบบอัตโนมัติจัดการส่วนที่เหลือได้

## เกี่ยวกับ ฮากิโค้ด

> เฮ้ เรามาแนะนำสิ่งที่เรากำลังสร้างกันดีกว่า

เรากำลังสร้าง **HagiCode** — ผู้ช่วยเขียนโค้ดที่ขับเคลื่อนด้วย AI ซึ่งทำให้การพัฒนาฉลาดขึ้น ง่ายขึ้น และสนุกสนานยิ่งขึ้น

**ฉลาดกว่า** — AI ช่วยเหลือตลอดการเดินทาง ตั้งแต่แนวคิดไปจนถึงโค้ด เพิ่มประสิทธิภาพการพัฒนาแบบทวีคูณ **ง่ายขึ้น** — การดำเนินการพร้อมกันแบบมัลติเธรดทำให้ใช้ทรัพยากรได้อย่างเต็มที่ และทำให้ขั้นตอนการพัฒนาราบรื่น **สนุกยิ่งขึ้น** — Gamification และระบบความสำเร็จทำให้การเขียนโค้ดน่าเบื่อน้อยลงและคุ้มค่ามากขึ้น

โครงการกำลังทำซ้ำอย่างรวดเร็ว หากคุณสนใจในการเขียนเชิงเทคนิค การจัดการความรู้ หรือการพัฒนาโดยใช้ AI โปรดติดต่อเรา [GitHub](https://github.com/HagiCode-org/site)~

## การวิเคราะห์เป้าหมาย

ก่อนที่จะเริ่มนำไปใช้งาน เราต้องชี้แจงก่อนว่าจริงๆ แล้วงานนี้ควรทำให้สำเร็จอย่างไร การลับขวานไม่ได้ช่วยชะลอการสับฟืน

### ข้อกำหนดหลัก

1.  **การสร้างอัตโนมัติ**: ทริกเกอร์เวิร์กโฟลว์บิวด์โดยอัตโนมัติเมื่อมีการพุชโค้ดไปที่ `main` สาขา
2.  **การปรับใช้อัตโนมัติ**: หลังจากสร้างสำเร็จแล้ว ให้ปรับใช้ไฟล์คงที่ที่สร้างขึ้นกับ GitHub Pages โดยอัตโนมัติ
3.  **ความสอดคล้องของสภาพแวดล้อม**: ตรวจสอบให้แน่ใจว่าสภาพแวดล้อม CI ตรงกับสภาพแวดล้อมบิลด์ในเครื่อง เพื่อหลีกเลี่ยงความลำบากใจแบบคลาสสิกที่ว่า "ใช้งานได้ในเครื่อง แต่ล้มเหลวทางออนไลน์"

### ทางเลือกทางเทคนิค

เนื่องจาก HagiCode สร้างขึ้นบน Docusaurus ซึ่งเป็นเครื่องมือสร้างไซต์คงที่ React ที่ได้รับความนิยมอย่างมาก เราจึงสามารถใช้ GitHub Actions เพื่อบรรลุเป้าหมายนี้ได้

## การกำหนดค่าเวิร์กโฟลว์การดำเนินการ GitHub

GitHub Actions คือบริการ CI/CD ที่ให้บริการโดย GitHub ด้วยการกำหนดไฟล์เวิร์กโฟลว์ในรูปแบบ YAML ภายในพื้นที่เก็บข้อมูล เราสามารถปรับแต่งงานอัตโนมัติทุกประเภทได้

### การสร้างไฟล์เวิร์กโฟลว์

เราจำเป็นต้องสร้างไฟล์กำหนดค่าใหม่ใน `.github/workflows` ไดเร็กทอรีที่รูทโปรเจ็กต์ เป็นต้น `deploy.yml`. หากยังไม่มีไดเร็กทอรี ให้สร้างไดเร็กทอรีก่อน

ตรรกะหลักของไฟล์การกำหนดค่านี้เป็นดังนี้:

1.  **เงื่อนไขทริกเกอร์**: ฟัง `push` เหตุการณ์บน `main` สาขา
2.  **สภาพแวดล้อมรันไทม์**: Ubuntu รุ่นล่าสุด
3.  **ขั้นตอนการสร้าง**:
    -   ตรวจสอบรหัส
    -   ติดตั้ง Node.js
    -   ติดตั้งการพึ่งพา (`npm install`)
    -   สร้างไฟล์สแตติก (`npm run build`)
4.  **ขั้นตอนการปรับใช้**: ใช้อย่างเป็นทางการ `action-gh-pages` การดำเนินการเพื่อผลักดันเอาท์พุตบิลด์ไปที่ `gh-pages` สาขา

### รหัสการกำหนดค่าคีย์

ด้านล่างนี้คือเทมเพลตการกำหนดค่าที่เรานำไปใช้ในท้ายที่สุด:

```yaml
name: Deploy to GitHub Pages

# Trigger condition: run when code is pushed to the main branch
on:
  push:
    branches:
      - main
    # Add path filters if needed, for example only build when docs change
    # paths:
    #   - 'docs/**'
    #   - 'package.json'

# Set permissions, which is important for GitHub Pages deployment
permissions:
  contents: read
  pages: write
  id-token: write

# Concurrency control: cancel older builds for the same branch
concurrency:
  group: "pages"
  cancel-in-progress: false

jobs:
  build:
    runs-on: ubuntu-latest
    steps:
      - name: Checkout
        uses: actions/checkout@v4
        # Note: fetch-depth: 0 is required, otherwise the build version may be inaccurate
        with:
          fetch-depth: 0

      - name: Setup Node
        uses: actions/setup-node@v4
        with:
          node-version: 20 # Keep this consistent with the local development environment
          cache: 'npm'     # Enabling cache can speed up the build process

      - name: Install dependencies
        run: npm ci
        # Use npm ci instead of npm install because it is faster and stricter, which makes it better for CI

      - name: Build website
        run: npm run build
        env:
          # Configure environment variables here if your site build requires them
          # NODE_ENV: production
          # PUBLIC_URL: /your-repo-name
          
      - name: Upload artifact
        uses: actions/upload-pages-artifact@v3
        with:
          path: ./build # Default Docusaurus output directory

  deploy:
    environment:
      name: github-pages
      url: ${{ steps.deployment.outputs.page_url }}
    runs-on: ubuntu-latest
    needs: build
    steps:
      - name: Deploy to GitHub Pages
        id: deployment
        uses: actions/deploy-pages@v4
```

## ข้อผิดพลาดทั่วไประหว่างการดำเนินการ

ในทางปฏิบัติ เราพบปัญหาหลายประการ ฉันกำลังแบ่งปันสิ่งเหล่านี้ที่นี่ด้วยความหวังว่าผู้อื่นจะสามารถหลีกเลี่ยงพวกเขาได้ หรืออย่างน้อยก็เตรียมวิธีแก้ปัญหาไว้ล่วงหน้า

### 1. ปัญหาการอนุญาตโทเค็น GitHub

เมื่อเราตั้งค่าต่างๆ เป็นครั้งแรก การปรับใช้ยังคงล้มเหลวด้วย 403 (ต้องห้าม) หลังจากขุดลึกมามาก เราพบว่าค่าเริ่มต้นของ GitHub `GITHUB_TOKEN` ไม่มีสิทธิ์เขียนลงเพจ

**วิธีแก้ปัญหา**: ในพื้นที่เก็บข้อมูล `Settings` -> `Actions` -> `General` -> `Workflow permissions`ตรวจสอบให้แน่ใจว่าได้เลือก **"สิทธิ์ในการอ่านและเขียน"**

### 2. เส้นทางไดเร็กทอรีบิลด์ผิด

Docusaurus สร้างไฟล์สแตติกไว้ในไฟล์ `build` ไดเร็กทอรีตามค่าเริ่มต้น อย่างไรก็ตาม บางโครงการอาจมีการกำหนดค่าแตกต่างออกไป ตัวอย่างเช่น สร้างค่าเริ่มต้นของแอป React เป็น `build`ในขณะที่ Vite มีค่าเริ่มต้นเป็น `dist`. หากการดำเนินการรายงานว่าไม่พบไฟล์เอาต์พุต ให้ตรวจสอบเส้นทางเอาต์พุต `docusaurus.config.js`.

### 3. ปัญหาเส้นทางย่อย

หากที่เก็บของคุณไม่ใช่โฮมเพจของผู้ใช้ แสดงว่าไม่ใช่ `username.github.io`แต่กลับเป็นหน้าโครงการเช่น `username.github.io/project-name`จากนั้นคุณจะต้องกำหนดค่า `baseUrl`.

ใน `docusaurus.config.js`:

```javascript
module.exports = {
  // ...
  url: 'https://hagicode.com', // Your Hagicode URL
  baseUrl: '/', // Deploy from the root path
  // ...
};
```
รายละเอียดนี้มองข้ามได้ง่าย หากมีการกำหนดค่าไม่ถูกต้อง เพจอาจโหลดเป็นหน้าจอว่างเนื่องจากไม่สามารถแก้ไขเส้นทางสินทรัพย์ได้

## การตรวจสอบผลลัพธ์

หลังจากกำหนดค่าทุกอย่างและกดโค้ดแล้ว เราก็ไปที่แท็บ **การดำเนินการ** ของที่เก็บ GitHub และดูการแสดงได้

คุณจะเห็นวงกลมสีเหลืองในขณะที่เวิร์กโฟลว์กำลังทำงาน และเมื่อเปลี่ยนเป็นสีเขียว แสดงว่าประสบความสำเร็จ หากเปลี่ยนเป็นสีแดง ให้คลิกเข้าไปและตรวจสอบบันทึก ในกรณีส่วนใหญ่ คุณสามารถติดตามปัญหาได้ที่นั่น ซึ่งโดยปกติจะเป็นการพิมพ์ผิดหรือการกำหนดค่าเส้นทางที่ไม่ถูกต้อง

เมื่อสร้างสำเร็จแล้วให้เยี่ยมชม `https://<your-username>.github.io/<repository-name>/` และคุณจะเห็นไซต์ที่เพิ่งปรับใช้ใหม่

## สรุป

ด้วยการแนะนำ GitHub Actions เราจึงประสบความสำเร็จในการปรับใช้อัตโนมัติสำหรับไซต์เอกสารประกอบ HagiCode ซึ่งไม่เพียงแต่ช่วยประหยัดเวลาที่ใช้ในการดำเนินการด้วยตนเองก่อนหน้านี้ แต่ยังช่วยสร้างมาตรฐานขั้นตอนการทำงานที่เผยแพร่อีกด้วย ในตอนนี้ ไม่ว่าสมาชิกในทีมคนใดจะอัปเดตเอกสารประกอบ เมื่อรวมการเปลี่ยนแปลงเข้าด้วยกันแล้ว `main`เนื้อหาล่าสุดจะเผยแพร่ในอีกไม่กี่นาทีต่อมา

**สิทธิประโยชน์หลัก**:
- **ประสิทธิภาพที่สูงขึ้น**: เราย้ายจาก "การบรรจุด้วยตนเองและการอัปโหลดด้วยตนเอง" เป็น "โค้ดเท่ากับการเปิดตัว"
- **ข้อผิดพลาดน้อยลง**: เราได้ขจัดความเป็นไปได้ที่จะเกิดข้อผิดพลาดจากมนุษย์ระหว่างการดำเนินการด้วยตนเอง
- **ประสบการณ์นักพัฒนาที่ดีขึ้น**: นักพัฒนาสามารถมุ่งเน้นไปที่คุณภาพของเนื้อหาได้มากขึ้น แทนที่จะถูกรบกวนด้วยขั้นตอนการปรับใช้ที่น่าเบื่อ

แม้ว่าการกำหนดค่า CI/CD จะค่อนข้างยุ่งยากในตอนแรก โดยเฉพาะอย่างยิ่งกับสิทธิ์และรายละเอียดพาธทั้งหมด แต่เป็นการลงทุนครั้งเดียวที่ให้ผลตอบแทนระยะยาวที่สำคัญ ฉันขอแนะนำอย่างยิ่งให้ทุกโครงการไซต์คงที่ใช้เวิร์กโฟลว์อัตโนมัติที่คล้ายกัน
