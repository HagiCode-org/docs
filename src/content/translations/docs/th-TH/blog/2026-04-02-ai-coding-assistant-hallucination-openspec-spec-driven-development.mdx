## ปัญหาอาการประสาทหลอนในผู้ช่วยเขียนโค้ด AI: วิธีบรรลุการพัฒนาที่ขับเคลื่อนด้วยข้อกำหนดด้วย OpenSpec

> ผู้ช่วยเขียนโค้ด AI นั้นทรงพลัง แต่มักจะสร้างโค้ดที่ไม่ตรงกับข้อกำหนดที่แท้จริงหรือฝ่าฝืนแบบแผนของโครงการ บทความนี้แบ่งปันวิธีที่โปรเจ็กต์ HagiCode ใช้เวิร์กโฟลว์ OpenSpec เพื่อใช้การพัฒนาที่ขับเคลื่อนด้วยข้อกำหนดและลดความเสี่ยงของอาการประสาทหลอน AI อย่างมากผ่านกลไกข้อเสนอที่มีโครงสร้าง

## พื้นหลัง

ใครก็ตามที่เคยใช้ GitHub Copilot หรือ ChatGPT ในการเขียนโค้ดอาจเคยมีประสบการณ์เช่นนี้: โค้ดที่สร้างโดย AI ดูขัดเกลา แต่เมื่อคุณใช้งานจริง ปัญหาก็จะปรากฏขึ้นทุกที่ บางทีอาจใช้ส่วนประกอบที่ไม่ถูกต้องจากโปรเจ็กต์ บางทีอาจเพิกเฉยต่อมาตรฐานการเขียนโค้ดของทีม หรือบางทีอาจเขียนตรรกะจำนวนมากตามสมมติฐานที่ไม่มีอยู่ด้วยซ้ำ

นี่คือปัญหาที่เรียกว่า "ภาพหลอน AI" ในการเขียนโปรแกรม จะปรากฏเป็นโค้ดที่ดูสมเหตุสมผลเมื่อดูเผินๆ แต่จริงๆ แล้วไม่เหมาะกับสถานะที่แท้จริงของโปรเจ็กต์

ยังมีเรื่องน่าหงุดหงิดนิดหน่อยเกี่ยวกับเรื่องนี้ เมื่อผู้ช่วยเขียนโค้ด AI แพร่หลายมากขึ้น ปัญหาก็รุนแรงมากขึ้น ท้ายที่สุดแล้ว AI ขาดความเข้าใจเกี่ยวกับประวัติโครงการ การตัดสินใจทางสถาปัตยกรรม และแบบแผนการเขียนโค้ด และเมื่อได้รับอิสระมากเกินไป AI ก็สามารถสร้างโค้ดที่ไม่ตรงกับความเป็นจริงได้อย่าง "สร้างสรรค์" มันก็เหมือนกับการเขียนบทความ หากไม่มีโครงสร้าง ก็สามารถท่องไปในจินตนาการได้ง่าย แม้ว่าสถานการณ์จริงจะมีเหตุผลมากกว่าก็ตาม

เพื่อแก้ไขปัญหาเหล่านี้ เราได้ตัดสินใจอย่างกล้าหาญ แทนที่จะพยายามทำให้ AI ฉลาดขึ้น เรากลับใส่มันไว้ในกรอบ "ข้อมูลจำเพาะ" การเปลี่ยนแปลงการตัดสินใจครั้งนี้อาจใหญ่กว่าที่คุณคาดไว้ และฉันจะอธิบายให้ทราบเร็วๆ นี้

## เกี่ยวกับ ฮากิโค้ด

แนวทางที่แบ่งปันในบทความนี้มาจากประสบการณ์จริงของเราใน [ฮากิโค้ด](https://hagicode.com) โครงการ HagiCode เป็นโครงการผู้ช่วยการเขียนโค้ด AI แบบโอเพ่นซอร์สที่อุทิศให้กับการแก้ปัญหาจริงในการเขียนโปรแกรม AI ผ่านการปฏิบัติงานด้านวิศวกรรมที่มีโครงสร้าง

## สาเหตุที่แท้จริงของภาพหลอน AI

ก่อนจะเจาะลึกถึงวิธีแก้ปัญหา ให้เราดูก่อนว่าจริงๆ แล้วปัญหามาจากไหน ท้ายที่สุดแล้ว หากคุณเข้าใจทั้งตัวคุณเองและคู่ต่อสู้ คุณสามารถต่อสู้นับร้อยครั้งโดยไม่พ่ายแพ้ นำไปใช้กับ AI คำพูดนั้นยังคงเหมาะสมอย่างน่าประหลาดใจ

### ไม่มีบริบท

โมเดล AI ได้รับการฝึกฝนบนที่เก็บรหัสสาธารณะ แต่โปรเจ็กต์ของคุณมีประวัติ แบบแผน และการตัดสินใจทางสถาปัตยกรรมเป็นของตัวเอง AI ไม่สามารถเข้าถึง "ความรู้โดยนัย" ประเภทนี้ได้โดยตรง ดังนั้นโค้ดที่สร้างขึ้นจึงมักถูกตัดการเชื่อมต่อจากโครงการจริง

นี่ไม่ใช่ความผิดของ AI ทั้งหมด มันไม่เคยอยู่ในโปรเจ็กต์ของคุณ ดังนั้นมันจะรู้กฎที่ไม่ได้เขียนไว้ทั้งหมดของคุณได้อย่างไร เช่นเดียวกับเด็กฝึกงานหน้าใหม่ การไม่เข้าใจประเพณีท้องถิ่นถือเป็นเรื่องปกติ ปัญหาเดียวคือต้นทุนอาจค่อนข้างสูง

### เสรีภาพมากเกินไป

เมื่อคุณถาม AI ว่า "ช่วยฉันใช้ฟีเจอร์การตรวจสอบสิทธิ์ผู้ใช้" AI อาจสร้างโค้ดในเกือบทุกรูปแบบ หากไม่มีข้อจำกัดที่ชัดเจน AI จะดำเนินการต่างๆ ในลักษณะที่ "คิดว่า" สมเหตุสมผล แทนที่จะปฏิบัติตามข้อกำหนดของโครงการของคุณ

นั่นก็เหมือนกับการขอให้คนที่ไม่เคยเรียนรู้มาตรฐานโครงการของคุณมาแสดงด้นสดอย่างอิสระ จะไม่สร้างปัญหาได้อย่างไร? ไม่ใช่ว่า AI จะไม่รับผิดชอบด้วยซ้ำ แค่ไม่รู้ว่าความรับผิดชอบหมายถึงอะไรในบริบทนี้

### ขาดการตรวจสอบ

หลังจากที่ AI สร้างโค้ด หากไม่มีกระบวนการตรวจสอบแบบมีโครงสร้าง โค้ดที่ตั้งอยู่บนสมมติฐานที่ผิดสามารถเข้าไปใน Repository ได้โดยตรง เมื่อถึงเวลาที่ปัญหาถูกค้นพบในการทดสอบหรือแม้แต่ในการใช้งานจริง ต้นทุนก็สูงเกินไปแล้ว

นั่นก็เหมือนกับการพยายามซ่อมปากกาหลังจากที่แกะจากไปแล้ว หลักการนี้ชัดเจน แต่ในทางปฏิบัติผู้คนมักจะพบว่างานพิเศษนั้นน่ารำคาญ ก่อนที่สิ่งต่างๆ จะผิดพลาด ใครอยากจะใช้เวลามากกว่านี้ล่ะ?

## OpenSpec: คำตอบของการพัฒนาที่ขับเคลื่อนด้วยข้อกำหนด

HagiCode เลือก OpenSpec เป็นโซลูชัน แนวคิดหลักนั้นเรียบง่าย: **การเปลี่ยนแปลงโค้ดทั้งหมดต้องผ่านขั้นตอนการทำงานของข้อเสนอที่มีโครงสร้าง** โดยเปลี่ยนแนวคิดที่เป็นนามธรรมให้เป็นแผนการใช้งานที่ปฏิบัติการได้

นั่นอาจฟังดูยิ่งใหญ่ แต่พูดง่ายๆ ก็คือการทำให้ AI เขียนเอกสารข้อกำหนดก่อนที่จะเขียนโค้ด ดังสุภาษิตโบราณที่ว่า การเตรียมตัวนำไปสู่ความสำเร็จ และการขาดการเตรียมตัวนำไปสู่ความล้มเหลว

### OpenSpec คืออะไร

OpenSpec เป็นเครื่องมือบรรทัดคำสั่งที่ใช้ npm (`@fission-ai/openspec`) ที่กำหนดโครงสร้างไฟล์ข้อเสนอมาตรฐานและกลไกการตรวจสอบ พูดง่ายๆ คือทำให้ AI "เขียนเอกสารข้อกำหนด" ก่อนที่จะเขียนโค้ด

### ขั้นตอนการทำงานสามขั้นตอนเพื่อป้องกันภาพหลอน

OpenSpec รับประกันคุณภาพของข้อเสนอผ่านขั้นตอนการทำงานสามขั้นตอน:

**ขั้นตอนที่ 1: เริ่มต้นข้อเสนอ** - ตั้งค่าสถานะเซสชันเป็น Openspecing
**ขั้นตอนที่ 2: การประมวลผลระดับกลาง** - คงสถานะ Openspecing ไว้ในขณะที่ค่อยๆ ปรับแต่งอาร์ติแฟกต์
**ขั้นตอนที่ 3: กรอกข้อเสนอให้เสร็จสิ้น** - เปลี่ยนเป็นสถานะกำลังตรวจสอบ

มีรายละเอียดที่ชาญฉลาดในการออกแบบนี้: ขั้นตอนแรกใช้ `ProposalGenerationStart` และดำเนินการให้เสร็จสิ้นจะไม่ทำให้เกิดการเปลี่ยนแปลงสถานะ เพื่อให้แน่ใจว่าจะไม่เข้าสู่ขั้นตอนการทบทวนเร็วเกินไปก่อนที่เวิร์กโฟลว์หลายขั้นตอนทั้งหมดจะเสร็จสิ้น

รายละเอียดนี้ค่อนข้างน่าสนใจจริงๆ มันก็เหมือนกับการทำอาหาร: หากคุณยกฝาขึ้นก่อนที่ไฟจะร้อน ทุกอย่างจะออกมาไม่ดี เพียงแค่ก้าวไปทีละขั้นด้วยความอดทนเพียงเล็กน้อยเท่านั้น คุณก็สามารถจะได้อาหารจานอร่อยได้

```csharp
// Implementation in the HagiCode project
public enum MessageAssociationType
{
    ProposalGeneration = 2,
    ProposalExecution = 3,

    /// <summary>
    /// Marks the start of the three-step proposal generation workflow
    /// Does not transition to the Reviewing state when completed
    /// </summary>
    ProposalGenerationStart = 5
}
```

### โครงสร้างไฟล์ที่ได้มาตรฐาน

ข้อเสนอ OpenSpec ทุกข้อเสนอมีโครงสร้างไดเร็กทอรีเดียวกัน:

```
openspec/
├── changes/                    # Active and archived changes
│   ├── {change-name}/
│   │   ├── proposal.md        # Proposal description
│   │   ├── design.md          # Design document
│   │   ├── specs/             # Technical specifications
│   │   └── tasks.md           # Executable task list
│   └── archive/               # Archived changes
└── specs/                     # Standalone specification library
```

ตามสถิติจากโครงการ HagiCode มีการเปลี่ยนแปลงที่เก็บถาวรมากกว่า 4,000 รายการและไฟล์ข้อกำหนดมากกว่า 150,000 บรรทัด การสะสมทางประวัติศาสตร์นี้ไม่เพียงแต่ให้แนวทางที่ชัดเจนแก่ AI ในการปฏิบัติตาม แต่ยังให้ฐานความรู้อันทรงคุณค่าแก่ทีมอีกด้วย

มันเหมือนกับความคลาสสิกที่คนรุ่นก่อนทิ้งไว้เบื้องหลัง อ่านมากพอแล้วรูปแบบก็เริ่มปรากฏ ข้อแตกต่างเพียงอย่างเดียวคือคลาสสิกเหล่านี้จะถูกจัดเก็บไว้ในไฟล์แทนที่จะเขียนลงบนแผ่นไม้ไผ่

### กลไกการตรวจสอบหลายชั้น

ระบบใช้การตรวจสอบความถูกต้องหลายชั้นเพื่อรับรองคุณภาพของข้อเสนอ:

```csharp
// Validate that required files exist
ValidateProposalFiles()

// Validate prerequisites before execution
ValidateExecuteAsync()

// Validate start conditions
ValidateStartAsync()

// Validate archive conditions
ValidateArchiveAsync()

// Validate proposal name format (kebab-case)
ValidateNameFormat()
```

การตรวจสอบเหล่านี้เปรียบเสมือนผู้เฝ้าประตูที่จุดตรวจหลายแห่ง เฉพาะข้อเสนอที่ผ่านการรับรองอย่างแท้จริงเท่านั้นที่สามารถผ่านได้ อาจดูน่าเบื่อ แต่ก็ยังดีกว่าปล่อยให้โค้ดที่แย่เข้าสู่ Repository มาก

### ข้อจำกัดของเทมเพลตพร้อมท์

เมื่อ AI ทำงานภายใน HagiCode ระบบจะใช้เทมเพลตแฮนด์บาร์ที่กำหนดไว้ล่วงหน้า เทมเพลตเหล่านี้มีคำแนะนำทีละขั้นตอนที่ชัดเจนและราวป้องกัน ตัวอย่างเช่น:

- อย่าดำเนินการต่อก่อนที่จะเข้าใจเจตนาของผู้ใช้
- อย่าสร้างรหัสที่ไม่ถูกต้อง
- กำหนดให้ผู้ใช้ระบุชื่ออีกครั้งหากไม่ถูกต้อง
- หากมีการเปลี่ยนแปลงอยู่แล้ว แนะนำให้ใช้คำสั่ง Continue แทนการสร้างใหม่

วิธีการ "เต้นพันธนาการ" นี้ช่วยให้ AI มุ่งเน้นไปที่การทำความเข้าใจข้อกำหนดและสร้างโค้ดที่เป็นไปตามมาตรฐานได้มากขึ้น ข้อจำกัดไม่ใช่สิ่งเลวร้ายเสมอไป บางครั้งอิสรภาพที่มากเกินไปก็ทำให้เกิดความสับสนวุ่นวาย

## แนวปฏิบัติ: วิธีใช้ OpenSpec ในโปรเจ็กต์

### การติดตั้งและการเริ่มต้น

```bash
npm install -g @fission-ai/openspec@1
openspec --version  # Verify the installation
```

ที่ `openspec/` โครงสร้างโฟลเดอร์จะถูกสร้างขึ้นโดยอัตโนมัติในรูทโปรเจ็กต์

ขั้นตอนนี้ไม่มีอะไรลึกลับมากนัก มันเป็นเพียงการติดตั้งเครื่องมือที่ทุกคนเข้าใจ เพียงจำไว้ว่าจะใช้ `@fission-ai/openspec@1`; เวอร์ชันที่ใหม่กว่าอาจมีข้อผิดพลาด และความเสถียรมีความสำคัญที่สุด

### สร้างข้อเสนอ

ในอินเทอร์เฟซการสนทนา HagiCode ให้ใช้คำสั่งทางลัด:

```
/opsx:new
```

หรือระบุการเปลี่ยนชื่อและที่เก็บเป้าหมาย:

```
/opsx:new "add-user-auth" --repos "repos/web"
```

การสร้างข้อเสนอก็เหมือนกับการสรุปบทความก่อนที่จะเขียน เมื่อคุณมีโครงร่างแล้ว ส่วนที่เหลือจะง่ายขึ้นมาก หลายๆ คนชอบที่จะกระโดดเข้าสู่การเขียนโดยตรง แต่กลับตระหนักได้เพียงครึ่งทางว่าแนวคิดนี้เข้ากันไม่ได้ นั่นคือตอนที่อาการปวดหัวที่แท้จริงเริ่มต้นขึ้น

### สร้างสิ่งประดิษฐ์

ใช้ `/opsx:continue` เพื่อสร้างสิ่งประดิษฐ์ที่จำเป็นทีละขั้นตอน:

**proposal.md** - อธิบายวัตถุประสงค์และขอบเขตของการเปลี่ยนแปลง
```markdown
# Proposal: Add User Authentication

## Why
The current system lacks user authentication and cannot protect sensitive APIs.

## What Changes
- Add JWT authentication middleware
- Implement login/registration APIs
- Update frontend integration
```

**design.md** - การออกแบบทางเทคนิคโดยละเอียด
```markdown
# Design: Add User Authentication

## Context
The system currently uses public APIs, so anyone can access them...

## Decisions
1. Choose JWT instead of Session...
2. Use the HS256 algorithm...

## Risks
- Risk of token leakage...
- Mitigation measures...
```

**ข้อมูลจำเพาะ/** - ข้อมูลจำเพาะทางเทคนิคและสถานการณ์การทดสอบ
```markdown
# user-auth Specification

## Requirements

### Requirement: JWT Token Generation
The system SHALL use the HS256 algorithm to generate JWT tokens.

#### Scenario: Valid login
- WHEN the user provides valid credentials
- THEN the system SHALL return a valid JWT token
```

**tasks.md** - รายการงานที่ปฏิบัติการได้
```markdown
# Tasks: Add User Authentication

## 1. Backend Changes
- [ ] 1.1 Create AuthController
- [ ] 1.2 Implement JWT middleware
- [ ] 1.3 Add unit tests
```

สิ่งประดิษฐ์เหล่านี้เหมือนกับร่างบทความมาก เมื่อร่างเสร็จสมบูรณ์ ข้อความหลักจะไหลลื่นอย่างเป็นธรรมชาติ หลายๆ คนไม่ชอบการเขียนฉบับร่างเพราะคิดว่ามันเสียเวลา แต่ในความเป็นจริงมักเป็นจุดที่ความคิดที่ชัดเจนที่สุดเกิดขึ้น

### ตรวจสอบและนำไปใช้

หลังจากสิ่งประดิษฐ์ทั้งหมดเสร็จสมบูรณ์แล้ว:

```
/opsx:apply
```

AI จะอ่านไฟล์บริบททั้งหมดและดำเนินงานทีละขั้นตอนตามรายการตรวจสอบใน `tasks.md`. ณ จุดนี้ เนื่องจากข้อกำหนดมีความชัดเจนอยู่แล้ว คุณภาพของโค้ดที่สร้างขึ้นจึงสูงกว่ามาก

เมื่อถึงขั้นตอนนี้ งานก็เสร็จสิ้นไปครึ่งหนึ่งแล้ว เมื่อมีรายการงานที่ชัดเจน ที่เหลือก็แค่ดำเนินการทีละขั้นตอน ปัญหาคือหลายๆ คนข้ามขั้นตอนก่อนหน้านี้และกระโดดตรงมาที่นี่ จากนั้นคุณภาพก็กลายเป็นเรื่องยากที่จะรับประกันโดยธรรมชาติ

### เก็บถาวร

หลังจากการเปลี่ยนแปลงเสร็จสิ้น:

```
/opsx:archive
```

ย้ายการเปลี่ยนแปลงที่เสร็จสมบูรณ์ไปที่ `archive/` ไดเร็กทอรีเพื่อให้สามารถตรวจทานและนำมาใช้ซ้ำได้ในภายหลัง

การเก็บถาวรเรื่องต่างๆ มันเหมือนกับการเก็บบทความที่เสร็จแล้วอย่างระมัดระวัง เมื่อปัญหาเดียวกันนี้เกิดขึ้นในอนาคต การมองย้อนกลับไปดูบันทึกเก่าๆ อาจให้คำตอบได้ หลายๆ คนพบว่าเป็นเรื่องยาก แต่วัสดุที่สะสมเหล่านี้มักเป็นทรัพย์สินที่มีค่าที่สุด

## หมายเหตุและแนวปฏิบัติที่ดีที่สุด

### กฎการตั้งชื่อข้อเสนอ

ใช้ kebab-case ขึ้นต้นด้วยตัวอักษร และใส่เฉพาะตัวอักษรพิมพ์เล็ก ตัวเลข และขีดกลางเท่านั้น:
- ✅ `add-user-auth`
- ❌ `AddUserAuth`
- ❌ `add--user-auth`

กฎการตั้งชื่ออาจดูเล็กน้อย แต่ความสม่ำเสมอนั้นคุ้มค่าเสมอ ในซอฟต์แวร์ ความสม่ำเสมอมีความสำคัญแม้ว่าผู้คนจะไม่ได้ใส่ใจกับมันเสมอไปก็ตาม

### หลีกเลี่ยงข้อผิดพลาดทั่วไป

1. **ใช้ผิดประเภทในขั้นตอนที่ 1 ของเวิร์กโฟลว์สามขั้นตอน** - ส่งผลให้สถานะเปลี่ยนเร็วเกินไป
2. **ลืมทริกเกอร์การเปลี่ยนแปลงสถานะในขั้นตอนสุดท้าย** - ซึ่งจะทำให้เวิร์กโฟลว์ค้างอยู่ในสถานะ Openspecing
3. **ข้ามการตรวจสอบและดำเนินการโดยตรง** - คุณควรตรวจสอบว่าอาร์ติแฟกต์ทั้งหมดเสร็จสมบูรณ์ก่อน

ข้อผิดพลาดเหล่านี้เป็นเรื่องปกติสำหรับผู้เริ่มต้น คนที่มีประสบการณ์ย่อมรู้วิธีหลีกเลี่ยงพวกเขาโดยธรรมชาติ ถึงกระนั้น ทุกคนก็มีประสบการณ์ในที่สุด และการออกนอกเส้นทางเล็กน้อยก็เป็นส่วนหนึ่งของกระบวนการ ความหวังเดียวคือการหลีกเลี่ยงการรับประทานมากเกินไป

### การจัดการการเปลี่ยนแปลงหลายรูปแบบ

OpenSpec รองรับการจัดการข้อเสนอหลายรายการพร้อมกัน ซึ่งมีประโยชน์อย่างยิ่งสำหรับคุณสมบัติขนาดใหญ่:

```bash
# View all active changes
openspec list

# Switch to a specific change
openspec apply "add-user-auth"

# View change status
openspec status --change "add-user-auth"
```

การจัดการการเปลี่ยนแปลงหลายรายการก็เหมือนกับการเขียนบทความหลายบทความในคราวเดียว ต้องใช้เทคนิคและความอดทน แต่เมื่อคุณคุ้นเคยแล้ว มันก็จะเป็นธรรมชาติมากขึ้น

### ทำความเข้าใจกับเครื่องสถานะเซสชัน

การทำความเข้าใจการเปลี่ยนสถานะช่วยในการแก้ไขปัญหา:

```
Init → Drafting → Openspecing → Reviewing → Executing → ExecutionCompleted → Completed → Archived
```

- **Openspecing**: กำลังสร้างแผน
- **การตรวจสอบ**: อยู่ระหว่างการตรวจสอบ (สิ่งประดิษฐ์สามารถแก้ไขได้ซ้ำหลายครั้ง)
- **กำลังดำเนินการ**: กำลังดำเนินการ (กำลังใช้ `tasks.md`)

ในที่สุดกลไกของรัฐก็เป็นเพียงกฎเกณฑ์ชุดหนึ่ง กฎเกณฑ์อาจรู้สึกน่ารำคาญในบางครั้ง แต่บ่อยครั้งที่กฎเหล่านี้มีประโยชน์ ดังคำกล่าวที่ว่า หากไม่มีกฎเกณฑ์ ไม่มีอะไรสามารถบรรลุผลสำเร็จได้อย่างถูกต้อง

## บทสรุป

ด้วยเวิร์กโฟลว์ OpenSpec โครงการ HagiCode ได้รับผลลัพธ์ที่สำคัญในการแก้ไขปัญหาภาพหลอนของ AI:

1. **ภาพหลอนน้อยลง** - AI ต้องปฏิบัติตามข้อกำหนดที่มีโครงสร้าง แทนที่จะสร้างโค้ดตามอำเภอใจ
2. **คุณภาพที่สูงขึ้น** - การตรวจสอบความถูกต้องหลายชั้นช่วยให้มั่นใจได้ว่าการเปลี่ยนแปลงเป็นไปตามมาตรฐานของโครงการ
3. **การทำงานร่วมกันเร็วขึ้น** - การเปลี่ยนแปลงที่เก็บไว้เป็นข้อมูลอ้างอิงสำหรับการพัฒนาในอนาคต
4. **การตรวจสอบย้อนกลับ** - ทุกการเปลี่ยนแปลงจะมีบันทึกข้อเสนอ การออกแบบ ข้อมูลจำเพาะ และงานที่สมบูรณ์

วิธีการนี้ไม่ได้ทำให้ AI ฉลาดขึ้น มันทำให้ AI อยู่ในกรง "ข้อกำหนด" การฝึกฝนได้แสดงให้เห็นว่าการเต้นรำในโซ่ตรวนสามารถนำไปสู่การแสดงที่ดีขึ้นได้

หลักการง่ายๆ ข้อจำกัดไม่จำเป็นต้องแย่เสมอไป เช่นเดียวกับการเขียน การมีรูปแบบในการติดตามมักจะช่วยให้ผลิตงานดีๆ ได้ง่ายขึ้น หลายๆ คนไม่ชอบข้อจำกัดเพราะพวกเขาคิดว่าข้อจำกัดจำกัดความคิดสร้างสรรค์ แต่ความคิดสร้างสรรค์ก็ต้องการพื้นที่ที่เหมาะสมในการเติบโตเช่นกัน

หากคุณใช้ผู้ช่วยเขียนโค้ด AI เหมือนกันและพบปัญหาที่คล้ายกัน ให้ลองใช้ OpenSpec การพัฒนาที่ขับเคลื่อนด้วยข้อกำหนดอาจดูเหมือนเพิ่มขั้นตอนพิเศษ แต่การลงทุนในช่วงแรกนั้นให้ผลตอบแทนมากกว่าคุณภาพโค้ดและประสิทธิภาพการบำรุงรักษาหลายเท่า

บางครั้งการชะลอตัวลงเล็กน้อยอาจเป็นหนทางที่เร็วที่สุด หลายๆ คนยังไม่ตระหนักเลย
