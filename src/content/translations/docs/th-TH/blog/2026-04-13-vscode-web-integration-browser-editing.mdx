## VSCode และโค้ดเซิร์ฟเวอร์: การเลือกโซลูชันการแก้ไขโค้ดบนเบราว์เซอร์

> เมื่อสร้างความสามารถในการแก้ไขโค้ดบนเบราว์เซอร์ นักพัฒนาต้องเผชิญกับทางเลือกหลัก: ใช้ VSCode อย่างเป็นทางการ `code serve-web` คุณลักษณะหรือนำมาใช้ขับเคลื่อนโดยชุมชน `code-server` วิธีแก้ปัญหา? การตัดสินใจนี้ไม่เพียงส่งผลต่อสถาปัตยกรรมทางเทคนิคเท่านั้น แต่ยังส่งผลต่อการปฏิบัติตามใบอนุญาตและความยืดหยุ่นในการปรับใช้อีกด้วย

## พื้นหลัง

การคัดเลือกทางเทคนิคก็เหมือนกับการเลือกเส้นทางชีวิต เมื่อคุณเลือกอันหนึ่งแล้ว คุณมักจะต้องเดินต่อไป และการเปลี่ยนทีหลังอาจมีราคาแพงมาก

ในยุคของการเขียนโปรแกรมแบบใช้ AI การแก้ไขโค้ดบนเบราว์เซอร์กำลังมีความสำคัญมากขึ้น ผู้ใช้คาดหวังว่าหลังจากที่ผู้ช่วย AI วิเคราะห์โค้ดเสร็จแล้ว พวกเขาสามารถเปิดตัวแก้ไขในเซสชันเบราว์เซอร์เดียวกันได้ทันที และทำการเปลี่ยนแปลงโดยไม่ต้องสลับแอปพลิเคชัน ประสบการณ์ที่ราบรื่นแบบนั้นควรจะอยู่ที่นั่นเมื่อคุณต้องการ

อย่างไรก็ตาม เมื่อใช้ฟีเจอร์นี้ นักพัฒนาต้องเผชิญกับทางเลือกทางเทคนิคที่สำคัญ: พวกเขาควรใช้ VSCode อย่างเป็นทางการหรือไม่ `code serve-web` คุณลักษณะหรือขับเคลื่อนโดยชุมชน `code-server` วิธีแก้ปัญหา?

แต่ละตัวเลือกมีจุดแข็งและข้อเสียของตัวเอง และการเลือกที่ไม่ดีอาจสร้างปัญหามากมายในภายหลัง การให้สิทธิ์ใช้งานเป็นตัวอย่างหนึ่ง: หากคุณค้นพบหลังจากเปิดตัวแล้วว่าผลิตภัณฑ์ของคุณไม่เป็นไปตามใบอนุญาต แสดงว่ามันสายเกินไปแล้ว การปรับใช้เป็นอีกวิธีหนึ่ง: โซลูชันอาจทำงานได้อย่างสมบูรณ์แบบในการพัฒนา จากนั้นจึงประสบปัญหาทุกประเภทเมื่อย้ายไปยังคอนเทนเนอร์ สิ่งเหล่านี้คือหลุมพรางที่ทีมต้องการหลีกเลี่ยง

## เกี่ยวกับ ฮากิโค้ด

แนวทางที่แบ่งปันในบทความนี้มาจากประสบการณ์จริงของเราใน [ฮากิโค้ด](https://hagicode.com) โครงการ HagiCode คือผู้ช่วยเขียนโค้ดที่ขับเคลื่อนด้วย AI ในขณะที่ใช้การแก้ไขโค้ดบนเบราว์เซอร์ เราได้ศึกษาทั้งสองโซลูชันในเชิงลึก และสุดท้ายได้ออกแบบสถาปัตยกรรมของเราเพื่อรองรับทั้งสองอย่าง ในขณะที่เลือกเซิร์ฟเวอร์โค้ดเป็นค่าเริ่มต้น

ที่เก็บโครงการ: [github.com/HagiCode-org/site](https://github.com/HagiCode-org/site)

## ความแตกต่างของใบอนุญาต (สำคัญที่สุด)

นี่คือความแตกต่างพื้นฐานที่สุดระหว่างทั้งสองโซลูชัน และเป็นปัจจัยแรกที่เราพิจารณาระหว่างการประเมิน เมื่อทำการตัดสินใจทางเทคนิค สิ่งสำคัญคือต้องเข้าใจความเสี่ยงทางกฎหมายล่วงหน้า

**โค้ดเซิร์ฟเวอร์**
- ใบอนุญาต MIT โอเพ่นซอร์สโดยสมบูรณ์
- ดูแลโดย Coder.com พร้อมชุมชนที่กระตือรือร้น
- นำไปใช้ในเชิงพาณิชย์ ดัดแปลง และแจกจ่ายได้ฟรี
- ไม่มีข้อจำกัดในสถานการณ์การใช้งาน

**VSCode `code serve-web`**
- ส่วนหนึ่งของผลิตภัณฑ์ Microsoft VSCode
- ใช้ใบอนุญาตของ Microsoft (ใบอนุญาต VS Code มีข้อจำกัดในการใช้งานเชิงพาณิชย์)
- มีไว้สำหรับนักพัฒนารายบุคคลเป็นหลัก
- การใช้งานระดับองค์กรอาจต้องมีการตรวจสอบใบอนุญาตเชิงพาณิชย์เพิ่มเติม

จากมุมมองของใบอนุญาต โค้ดเซิร์ฟเวอร์เป็นมิตรกับโครงการเชิงพาณิชย์มากกว่า นี่คือสิ่งที่คุณต้องพิจารณาในระหว่างการวางแผนผลิตภัณฑ์ เนื่องจากการย้ายในภายหลังอาจมีค่าใช้จ่ายสูงมาก

## ความแตกต่างในการปรับใช้

เมื่อจัดการสิทธิ์การใช้งานแล้ว ปัญหาถัดไปคือการปรับใช้ ซึ่งส่งผลโดยตรงต่อต้นทุนการดำเนินงานและการออกแบบสถาปัตยกรรมของคุณ

**โค้ดเซิร์ฟเวอร์**
- แอปพลิเคชัน Node.js แบบสแตนด์อโลนที่สามารถปรับใช้ได้อย่างอิสระ
- รองรับแหล่งรันไทม์หลายแหล่ง:
  - ระบุเส้นทางปฏิบัติการโดยตรง
  - ค้นหามันผ่านระบบ PATH
  - การตรวจหาสภาพแวดล้อม NVM Node.js 22.x โดยอัตโนมัติ
- ไม่จำเป็นต้องติดตั้งแอปพลิเคชันเดสก์ท็อป VSCode บนเซิร์ฟเวอร์
- ปรับใช้ในคอนเทนเนอร์ได้ง่ายขึ้น

**VSCode `code serve-web`**
- ต้องขึ้นอยู่กับ VSCode CLI ที่ติดตั้งในเครื่อง
- จำเป็นต้องมีที่มีอยู่ `code` คำสั่งบนเครื่องโฮสต์
- ระบบจะกรอง wrapper VS Code Remote CLI ออก
- ออกแบบมาเพื่อสถานการณ์การพัฒนาท้องถิ่นเป็นหลัก

โค้ดเซิร์ฟเวอร์เหมาะกว่าสำหรับสถานการณ์การปรับใช้เซิร์ฟเวอร์และคอนเทนเนอร์ หากผลิตภัณฑ์ของคุณจำเป็นต้องทำงานใน Docker หรือผู้ใช้ของคุณไม่ได้ติดตั้ง VSCode โค้ดเซิร์ฟเวอร์มักจะเป็นตัวเลือกที่เหมาะสม

## ความแตกต่างของพารามิเตอร์คุณลักษณะ

โซลูชันทั้งสองยังแตกต่างกันในพารามิเตอร์คุณลักษณะบางอย่างอีกด้วย ความแตกต่างไม่มากนัก แต่สามารถสร้างความขัดแย้งในการบูรณาการในการใช้งานจริงได้

| คุณสมบัติ | รหัสเซิร์ฟเวอร์ | โค้ดเสิร์ฟเว็บ |
|------|-------------|----------------|
| เส้นทางฐานสาธารณะ | `/` (กำหนดค่าได้) | `/vscode-server` (คงที่) |
| การรับรองความถูกต้อง | `--auth` พารามิเตอร์ที่มีหลายโหมด | `--connection-token` / `--without-connection-token` |
| ไดเร็กทอรีข้อมูล | `{DataDir}/code-server` | `{DataDir}/vscode-serve-web` |
| การวัดและส่งข้อมูลทางไกล | ปิดใช้งานโดยค่าเริ่มต้นด้วย `--disable-telemetry` | ขึ้นอยู่กับการตั้งค่า VSCode |
| อัพเดทเช็ค | สามารถปิดการใช้งานได้ด้วย `--disable-update-check` | ขึ้นอยู่กับการตั้งค่า VSCode |

ความแตกต่างเหล่านี้จำเป็นต้องได้รับการดูแลเป็นพิเศษในระหว่างการรวมระบบ ตัวอย่างเช่น เส้นทาง URL ที่แตกต่างกันหมายความว่าโค้ดส่วนหน้าของคุณต้องการการจัดการโดยเฉพาะ

## ความแตกต่างในการตรวจจับความพร้อมใช้งาน

เมื่อใช้การสลับตัวแก้ไข ตรรกะการตรวจจับความพร้อมใช้งานจะแตกต่างออกไปด้วย

**โค้ดเซิร์ฟเวอร์**
- กลับเป็นการดำเนินการที่มองเห็นได้เสมอ
- ยังคงแสดงอยู่แม้ว่าจะไม่พร้อมใช้งานก็ตาม โดยมี `install-required` สถานะ
- รองรับการตรวจจับสภาพแวดล้อม NVM Node.js 22.x โดยอัตโนมัติ

**โค้ดเสิร์ฟ-เว็บ**
- มองเห็นได้เฉพาะเมื่อเป็นคนท้องถิ่นเท่านั้น `code` ตรวจพบ CLI
- หากไม่พร้อมใช้งาน ส่วนหน้าจะซ่อนตัวเลือกนี้โดยอัตโนมัติ
- ขึ้นอยู่กับสถานะการติดตั้ง VSCode ในเครื่อง

ความแตกต่างนี้ส่งผลโดยตรงต่อประสบการณ์ผู้ใช้ โค้ดเซิร์ฟเวอร์มีความโปร่งใสมากขึ้น: ผู้ใช้สามารถเห็นตัวเลือกและเข้าใจว่ายังจำเป็นต้องมีการติดตั้ง code-serve-web ถูกซ่อนไว้มากขึ้น: ผู้ใช้อาจไม่รู้ด้วยซ้ำว่ามีตัวเลือกนี้อยู่ แนวทางใดจะดีกว่านั้นขึ้นอยู่กับการวางตำแหน่งผลิตภัณฑ์

## สถาปัตยกรรมการใช้งานแบบคู่ของ HagiCode

หลังจากการวิเคราะห์เชิงลึก โครงการ HagiCode ได้นำสถาปัตยกรรมการใช้งานแบบคู่มาใช้ ซึ่งสนับสนุนทั้งสองโซลูชันในระดับสถาปัตยกรรม

### กำลังเริ่มต้นเป็นโค้ดเซิร์ฟเวอร์

```csharp
// The default active implementation is code-server
// If an explicit activeImplementation is saved, try that implementation first
// If the requested implementation is unavailable, the resolver tries the other one
// If a fallback occurs, return fallbackReason
```

เราตั้งค่าเริ่มต้นเป็นโค้ดเซิร์ฟเวอร์เนื่องจากความยืดหยุ่นในการอนุญาตให้ใช้สิทธิ์และการปรับใช้เป็นหลัก อย่างไรก็ตาม สำหรับผู้ใช้ที่มีสภาพแวดล้อม VSCode ในเครื่องอยู่แล้ว โค้ดเสิร์ฟเว็บก็เป็นตัวเลือกที่ดีเช่นกัน

### ตัวเลือกการใช้งาน

`CodeServerImplementationResolver` มีหน้าที่รับผิดชอบในการรวม:
- การเลือกการใช้งานระหว่างการอุ่นเครื่องเมื่อเริ่มต้นระบบ
- การเลือกการใช้งานเมื่ออ่านสถานะ
- การเลือกการดำเนินการเมื่อเปิดโครงการ
- การเลือกการใช้งานเมื่อเปิด Vaults

การออกแบบนี้ช่วยให้ระบบสามารถตอบสนองสถานการณ์ต่างๆ ได้อย่างยืดหยุ่น และผู้ใช้สามารถเลือกการใช้งานที่ตรงกับสภาพแวดล้อมของตนมากที่สุด

### กฎการปรับตัวส่วนหน้า

```typescript
// When localCodeAvailable=false, do not show code serve-web
// When localCodeAvailable=true, show the code serve-web configuration
```

ส่วนหน้าจะแสดงตัวเลือกที่พร้อมใช้งานโดยอัตโนมัติตามสภาพแวดล้อม ดังนั้นผู้ใช้จึงไม่สับสนกับคุณลักษณะที่พวกเขาไม่สามารถใช้ได้

## คู่มือการปฏิบัติ

หลังจากทฤษฏีทั้งหมดแล้ว คุณควรใส่ใจอะไรในระหว่างการปรับใช้จริง ท้ายที่สุดแล้ว การนำไปปฏิบัติคือสิ่งสำคัญ

### คำแนะนำการปรับใช้นักเทียบท่า

สำหรับการปรับใช้แบบคอนเทนเนอร์ โค้ดเซิร์ฟเวอร์คือตัวเลือกที่ดีกว่า:

```dockerfile
# Use the official code-server image directly
FROM codercom/code-server:latest

# Or install through npm
RUN npm install -g code-server
```

วิธีนี้จะช่วยแก้ปัญหาในเลเยอร์เดียวโดยไม่ต้องติดตั้ง VSCode เพิ่มเติม

### ตัวอย่างการกำหนดค่า

**การกำหนดค่าเซิร์ฟเวอร์รหัส**
```json
{
  "vscodeServer": {
    "enabled": true,
    "activeImplementation": "code-server",
    "codeServer": {
      "host": "0.0.0.0",
      "port": 8080,
      "executablePath": "",
      "authMode": "none"
    }
  }
}
```

**โค้ดเสิร์ฟ-การกำหนดค่าเว็บ**
```json
{
  "vscodeServer": {
    "enabled": true,
    "activeImplementation": "serve-web",
    "serveWeb": {
      "host": "0.0.0.0",
      "port": 8080,
      "executablePath": "/usr/local/bin/code"
    }
  }
}
```

การกำหนดค่าอาจดูน่าเบื่อเล็กน้อยในครั้งแรก แต่เมื่อติดตั้งแล้ว สิ่งต่างๆ จะง่ายขึ้นมากในการบำรุงรักษา

### ความแตกต่างในการสร้าง URL

**โค้ดเซิร์ฟเวอร์**
```
http://localhost:8080/?folder=/path/to/project&vscode-lang=zh-CN
```

**โค้ดเสิร์ฟ-เว็บ**
```
http://localhost:8080/vscode-server/?folder=/path/to/project&tkn=xxx&vscode-lang=zh-CN
```

ให้ความสนใจกับความแตกต่างในเส้นทางและพารามิเตอร์ คุณต้องจัดการแยกกันในระหว่างการรวมระบบ

### การสลับการใช้งาน

ระบบรองรับการสลับรันไทม์และหยุดการใช้งานก่อนหน้าโดยอัตโนมัติเมื่อทำการสลับ:

```csharp
// VsCodeServerManager automatically handles mutual exclusion
// When switching activeImplementation, the old implementation will not keep running in the background
```

การออกแบบนี้ช่วยให้ผู้ใช้ลองใช้การใช้งานต่างๆ ได้ตลอดเวลาและค้นหาตัวเลือกที่เหมาะกับตนเองที่สุด

### การตรวจสอบสถานะ

```typescript
const { settings, runtime } = await getVsCodeServerSettings();

// runtime.activeImplementation: "code-server" | "serve-web"
// runtime.fallbackReason: reason for switching
// runtime.status: "running" | "starting" | "stopped" | "unhealthy"
```

เมื่อสถานะปรากฏ ผู้ใช้สามารถระบุได้อย่างรวดเร็วว่าปัญหามาจากฝั่งเซิร์ฟเวอร์หรือจากการดำเนินการของตนเอง
