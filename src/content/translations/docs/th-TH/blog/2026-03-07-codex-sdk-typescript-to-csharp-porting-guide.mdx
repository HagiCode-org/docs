## จาก TypeScript ถึง C#: แนวทางปฏิบัติในการย้ายข้อมูลข้ามภาษาสำหรับ Codex SDK

> พูดง่ายๆ ก็คือ บทความนี้ยังเหมือนบทความเล็กๆ น้อยๆ ของเราด้วย โดยจะบันทึกกระบวนการทั้งหมดในการย้าย TypeScript Codex SDK อย่างเป็นทางการไปยัง C# การเรียกมันว่า "ท่าเรือ" เกือบจะทำให้ฟังดูง่ายเกินไป - มันเหมือนกับการผจญภัยที่ยาวนานมากกว่า เพราะทั้งสองภาษามีบุคลิกที่แตกต่างกันมาก และเราต้องหาทางที่จะทำให้ทั้งสองภาษาร่วมมือกัน

## พื้นหลัง

Codex เป็นเครื่องมือ AI Agent CLI ที่เปิดตัวโดย OpenAI และเป็นเครื่องมือที่ทรงพลังอย่างแท้จริง ทีมงานอย่างเป็นทางการจัดเตรียม TypeScript SDK ใน `@openai/codex` แพคเกจ มันโต้ตอบกับ Codex CLI โดยการเรียก `codex exec --experimental-json` คำสั่งและแยกวิเคราะห์สตรีมเหตุการณ์ JSONL

ปัญหาคือในโครงการ HagiCode เราจำเป็นต้องใช้มันในสภาพแวดล้อม .NET โดยเฉพาะในบริการแบ็กเอนด์ C# และแอปพลิเคชันเดสก์ท็อป เราไม่สามารถแนะนำรันไทม์ Node.js ให้กับโปรเจ็กต์ .NET เพียงเพื่อเรียกใช้เครื่องมือ CLI ได้ นั่นคงจะยุ่งยากเกินไป

ดังนั้นเราจึงเหลือสองทางเลือก: รักษาเลเยอร์บริดจ์ Node.js ที่ซับซ้อน หรือสร้าง C# SDK ดั้งเดิมด้วยตนเอง

เราเลือกอย่างหลัง

หากคุณพบว่าบทความนี้มีประโยชน์ โปรดติดดาวให้กับเราบน GitHub: [github.com/HagiCode-org/site](https://github.com/HagiCode-org/site). คุณยังสามารถเยี่ยมชมเว็บไซต์อย่างเป็นทางการเพื่อเรียนรู้เพิ่มเติม: [hagicode.com](https://hagicode.com). เป็นเรื่องที่ให้กำลังใจเสมอเมื่อโครงการโอเพ่นซอร์สได้รับการสนับสนุน

## เนื้อหาหลัก

### การเปรียบเทียบการออกแบบสถาปัตยกรรม

ก่อนที่จะแปลโค้ดแบบหนึ่งต่อหนึ่ง เราต้องเข้าใจการออกแบบสถาปัตยกรรมของ SDK ทั้งสองก่อน คุณต้องเข้าใจทั้งสองฝ่ายก่อนจึงจะสามารถพอร์ตได้ดี

สถาปัตยกรรมหลักของ TypeScript SDK มีลักษณะดังนี้:

```
Codex (entry class)
  └── CodexExec (executor, manages child processes)
      └── Thread (conversation thread)
          ├── run() / runStreamed() (synchronous/asynchronous execution)
          └── event stream parsing
```

C# SDK ยังคงใช้สถาปัตยกรรมแบบเลเยอร์เดียวกัน แต่ปรับรายละเอียดการใช้งาน แนวคิดโดยรวมนั้นตรงไปตรงมา: รักษาความสอดคล้องของ API ในขณะที่ใช้ประโยชน์จากคุณลักษณะภาษา C# อย่างเต็มที่ในการใช้งาน

### พิมพ์การแปลงระบบ

นี่เป็นส่วนพื้นฐานที่สุดและเป็นส่วนที่สำคัญที่สุดของงานด้วย หากรากฐานอ่อนแอ ทุกอย่างที่ตามมาก็จะยากขึ้น

ระบบประเภทของ TypeScript มีความยืดหยุ่นมากกว่า C# และนั่นเป็นเพียงข้อเท็จจริง เราจำเป็นต้องค้นหากลยุทธ์การทำแผนที่ที่เหมาะสม:

| TypeScript | ค# | หมายเหตุ |
|------------|-----|------|
| `interface` / `type` | `record` | การใช้ C# `record` สำหรับโครงสร้างข้อมูลที่ไม่เปลี่ยนรูป |
| `สตริง \| โมฆะ` | `string?` | ประเภทการอ้างอิงที่เป็น Nullable |
| `บูลีน \| ไม่ได้กำหนด` | `bool?` | บูลีนที่เป็นโมฆะ |
| `AsyncGenerator` | `IAsyncEnumerable` | ตัววนซ้ำ Async |

**ระบบประเภทเหตุการณ์** เป็นตัวอย่างทั่วไป TypeScript ใช้ประเภทสหภาพเพื่อกำหนดเหตุการณ์:

```typescript
export type ThreadEvent =
  | ThreadStartedEvent
  | TurnStartedEvent
  | TurnCompletedEvent
  | ...
```

ใน C# เราใช้ลำดับชั้นการสืบทอดและการจับคู่รูปแบบเพื่อให้ได้ผลลัพธ์ที่คล้ายกัน:

```csharp
public abstract record ThreadEvent(string Type);

public sealed record ThreadStartedEvent(string ThreadId) : ThreadEvent("thread.started");
public sealed record TurnStartedEvent() : ThreadEvent("turn.started");
public sealed record TurnCompletedEvent(Usage Usage) : ThreadEvent("turn.completed");
// ...
```

เราเลือกแล้ว `record` แทน `class` เพราะออบเจ็กต์เหตุการณ์ควรไม่เปลี่ยนรูป ซึ่งตรงกับเจตนาเบื้องหลังการใช้ออบเจ็กต์ธรรมดาใน TypeScript ที่ `sealed` คีย์เวิร์ดยังป้องกันการสืบทอดเพิ่มเติมและให้ห้องคอมไพเลอร์เพื่อปรับให้เหมาะสม

### จุดย้ายที่สำคัญ

#### 1. ตัวแยกวิเคราะห์เหตุการณ์

การแยกวิเคราะห์เหตุการณ์เป็นแกนหลักของ SDK ทั้งหมด เนื่องจากเป็นตัวกำหนดว่าเราสามารถเข้าใจทุกข้อความที่ส่งคืนโดย Codex CLI ได้อย่างถูกต้องหรือไม่ หากการแยกวิเคราะห์ไม่ถูกต้อง ทุกอย่างหลังจากนั้นจะสูญเปล่า

เวอร์ชัน TypeScript ใช้ `JSON.parse()` เพื่อแยกวิเคราะห์ JSON แต่ละบรรทัด:

```typescript
export function parseEvent(line: string): ThreadEvent {
  const data = JSON.parse(line);
  // Handle different event types...
}
```

เวอร์ชัน C# ใช้ `System.Text.Json.JsonDocument` แทน:

```csharp
public static ThreadEvent Parse(string line)
{
    using var document = JsonDocument.Parse(line);
    var root = document.RootElement;
    var type = GetRequiredString(root, "type", "event.type");

    return type switch
    {
        "thread.started" => new ThreadStartedEvent(GetRequiredString(root, "thread_id", ...)),
        "turn.started" => new TurnStartedEvent(),
        "turn.completed" => new TurnCompletedEvent(ParseUsage(...)),
        // ...
        _ => new UnknownThreadEvent(type, root.Clone()),
    };
}
```

มีเคล็ดลับเล็กๆ น้อยๆ แต่สำคัญอย่างหนึ่งที่นี่: `root.Clone()` เป็นสิ่งจำเป็นเนื่องจากองค์ประกอบจาก `JsonDocument` จะใช้ไม่ได้หลังจากจำหน่ายเอกสารแล้ว เราจำเป็นต้องเก็บสำเนาไว้สำหรับประเภทเหตุการณ์ที่ไม่รู้จัก นั่นเป็นเพียงหนึ่งในความแตกต่างระหว่างการจัดการ C# JSON และ JavaScript

#### 2. ความแตกต่างในการจัดการกระบวนการ

นี่คือจุดที่ SDK ทั้งสองแตกต่างกันมากที่สุด Node.js และ .NET มีรูปแบบรันไทม์ที่แตกต่างกัน ดังนั้นการใช้งานจึงต้องปรับเปลี่ยน

TypeScript ใช้ Node.js `spawn()` ฟังก์ชั่น:

```typescript
const child = spawn(this.executablePath, commandArgs, { env, signal });
```

C# ใช้ .NET `System.Diagnostics.Process`:

```csharp
using var process = new Process { StartInfo = startInfo };
process.Start();

// stdin/stdout/stderr must be managed manually
```

โดยเฉพาะอย่างยิ่งเวอร์ชัน C# จำเป็นต้องกำหนดค่ากระบวนการดังนี้:

```csharp
var startInfo = new ProcessStartInfo
{
    FileName = _executablePath,
    RedirectStandardInput = true,
    RedirectStandardOutput = true,
    RedirectStandardError = true,
    UseShellExecute = false,
    CreateNoWindow = true,
};
```

ความแตกต่างที่ใหญ่ที่สุดคือกลไกการยกเลิก การใช้ TypeScript `AbortSignal`ซึ่งเป็นส่วนหนึ่งของ Web API และสะดวกมากในการทำงานกับ:

```typescript
const child = spawn(cmd, args, { signal: cancellationSignal });
```

การใช้ C# `CancellationToken` แทน:

```csharp
public async IAsyncEnumerable<string> RunAsync(
    CodexExecArgs args,
    [EnumeratorCancellation] CancellationToken cancellationToken = default)
{
    // Check cancellation status inside the loop
    while (!cancellationToken.IsCancellationRequested)
    {
        // Process output...
    }

    // Terminate the process when cancellation is requested
    if (cancellationToken.IsCancellationRequested)
    {
        try { process.Kill(entireProcessTree: true); } catch { }
    }
}
```

ในระดับสูง นี่เป็นเพียงอีกตัวอย่างหนึ่งของความแตกต่างระหว่างระบบนิเวศของ Web API และระบบนิเวศ .NET

#### 3. รักษาการกำหนดค่าให้เป็นอนุกรม

SDK ทั้งสองใช้ตรรกะที่แปลงการกำหนดค่า JSON เป็นการกำหนดค่า TOML เนื่องจาก Codex CLI ยอมรับการแทนที่การกำหนดค่าในรูปแบบ TOML ส่วนนี้จะต้องคงความสอดคล้องกันอย่างสมบูรณ์ มิฉะนั้นการกำหนดค่าเดียวกันจะทำงานแตกต่างออกไปใน SDK ทั้งสอง

นั่นคือรายละเอียดที่คุณไม่สามารถประนีประนอมได้ ความสำเร็จหรือความล้มเหลวมักจะขึ้นอยู่กับรายละเอียดเช่นนี้

### รายละเอียดการดำเนินการ

#### โครงสร้างโครงการ

เราสร้างโครงสร้างโครงการดังต่อไปนี้:

```
CodexSdk/
├── CodexSdk.csproj
├── Codex.cs           # Entry class
├── CodexThread.cs     # Conversation thread
├── CodexExec.cs       # Executor
├── Events.cs          # Event type definitions
├── Items.cs           # Item type definitions
├── EventParser.cs     # Event parser
├── OutputSchemaTempFile.cs  # Temporary file management
└── ...
```

มันเป็นโครงสร้างที่ค่อนข้างสะอาด และช่วยได้มากในระหว่างการท่าเรือ

#### ตัวอย่างการใช้งาน

การใช้งานพื้นฐานยังคงสอดคล้องกับ TypeScript SDK:

```csharp
using CodexSdk;

// Create a Codex instance
var codex = new Codex();
var thread = codex.StartThread();

// Execute a query
var result = await thread.RunAsync("Summarize this repository.");
Console.WriteLine(result.FinalResponse);
```

การจัดการเหตุการณ์สตรีมมิ่งใช้ประโยชน์จากการจับคู่รูปแบบ C#:

```csharp
await foreach (var @event in thread.RunStreamedAsync("Analyze the code."))
{
    switch (@event)
    {
        case ItemCompletedEvent itemCompleted
            when itemCompleted.Item is AgentMessageItem msg:
            Console.WriteLine($"Assistant: {msg.Text}");
            break;
        case TurnCompletedEvent completed:
            Console.WriteLine($"Tokens: in={completed.Usage.InputTokens}");
            break;
        case CommandExecutionItem command:
            Console.WriteLine($"Command: {command.Command}");
            break;
    }
}
```

### หมายเหตุ

ในระหว่างการดำเนินการ เราได้รวบรวมบทเรียนเชิงปฏิบัติหลายประการ:

1. **การจัดการกระบวนการ**: เวอร์ชัน C# ต้องจัดการวงจรการใช้งานกระบวนการทั้งหมดด้วยตนเอง รวมถึงการยุติกระบวนการระหว่างการยกเลิก ใช้ `Kill(entireProcessTree: true)` เพื่อให้แน่ใจว่ากระบวนการย่อยได้รับการล้างข้อมูลด้วย

2. **การจัดการข้อผิดพลาด**: เราใช้ `InvalidOperationException` เพื่อโยนข้อผิดพลาดในการแยกวิเคราะห์ ทำให้รูปแบบการจัดการข้อผิดพลาดคล้ายกับ TypeScript SDK

3. **การล้างทรัพยากร**: `OutputSchemaTempFile` ดำเนินการ `IAsyncDisposable` เพื่อให้แน่ใจว่าไฟล์ชั่วคราวได้รับการล้างอย่างถูกต้อง

4. **ตัวแปรสภาพแวดล้อม**: เวอร์ชัน C# รองรับการแทนที่ตัวแปรสภาพแวดล้อมกระบวนการอย่างสมบูรณ์ `CodexOptions.Env`. มันเป็นคุณสมบัติเล็กๆ แต่ใช้งานได้จริงมาก

5. **ความแตกต่างของแพลตฟอร์ม**: เวอร์ชัน C# ไม่มีตรรกะของเวอร์ชัน TypeScript สำหรับการค้นหาไบนารีภายในแพ็คเกจ npm โดยอัตโนมัติ เนื่องจากโดยทั่วไปแล้วโปรเจ็กต์ .NET จะไม่ขึ้นอยู่กับ npm ซึ่งเป็นพาธไปยัง `codex` จะต้องระบุปฏิบัติการผ่านทาง `CODEX_EXECUTABLE` ตัวแปรสภาพแวดล้อมหรือ `CodexPathOverride`.

## บทสรุป

การย้าย TypeScript SDK ที่สมบูรณ์ไปเป็น C# ไม่ใช่แค่เรื่องของการแปลงไวยากรณ์เท่านั้น แต่ยังต้องมีความเข้าใจปรัชญาการออกแบบของทั้งสองภาษาด้วย ความยืดหยุ่นของ TypeScript และคุณสมบัติระบบนิเวศ JavaScript เช่น `AbortSignal` ต้องการคู่หูที่เหมาะสมใน C #

ประเด็นสำคัญคือ: **การรักษาความสอดคล้องของ API มีความสำคัญมากกว่าการรักษาความสอดคล้องในระดับการใช้งาน** ผู้ใช้สนใจว่าอินเทอร์เฟซใช้งานง่ายหรือไม่ ไม่สำคัญว่าการใช้งานภายในจะเหมือนกันหรือไม่ ฟังดูง่าย แต่การแลกเปลี่ยนเหล่านั้นต้องใช้วิจารณญาณ

หากคุณกำลังทำงานกับพอร์ตข้ามภาษาที่คล้ายกัน ประสบการณ์ของเราคือการทำความเข้าใจสถาปัตยกรรม SDK ดั้งเดิมอย่างถ่องแท้ก่อน จากนั้นจึงแปลทีละโมดูล และสุดท้ายใช้ชุดทดสอบที่สมบูรณ์เพื่อให้แน่ใจว่าพฤติกรรมมีความสอดคล้องกัน งานประเภทนี้ไม่สามารถเร่งรีบได้
