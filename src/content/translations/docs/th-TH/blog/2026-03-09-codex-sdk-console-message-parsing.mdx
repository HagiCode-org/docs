## คำแนะนำฉบับสมบูรณ์เกี่ยวกับการแยกวิเคราะห์ข้อความคอนโซล Codex SDK

> บทความนี้จะอธิบายกลไกการสตรีมเหตุการณ์ Codex SDK การแยกวิเคราะห์ประเภทข้อความ และแนวทางปฏิบัติที่ดีที่สุดในโปรเจ็กต์จริง ช่วยให้นักพัฒนาเชี่ยวชาญทักษะหลักเบื้องหลังบริการดำเนินการ AI ได้อย่างรวดเร็ว

## พื้นหลัง

เมื่อสร้างบริการดำเนินการ AI โดยใช้ Codex SDK เราต้องพบกับคำถามเชิงปฏิบัติอย่างหลีกเลี่ยงไม่ได้: เราควรจัดการกับข้อความเหตุการณ์สตรีมที่ส่งคืนโดย Codex อย่างไร ข้อความเหล่านี้ประกอบด้วยข้อมูลที่สำคัญ เช่น สถานะการดำเนินการ เนื้อหาเอาต์พุต และรายละเอียดข้อผิดพลาด ดังนั้นจึงสมควรได้รับการจัดการอย่างระมัดระวัง

ในฐานะส่วนหนึ่งของโปรเจ็กต์ HagiCode เราต้องการตัวดำเนินการที่เชื่อถือได้สำหรับสถานการณ์ผู้ช่วยการเขียนโค้ด AI นั่นคือเหตุผลที่เราตัดสินใจศึกษากลไกการสตรีมเหตุการณ์ Codex SDK ในเชิงลึก ท้ายที่สุดแล้ว เพียงแค่เข้าใจว่าข้อความสำคัญทำงานอย่างไร เราจึงสามารถสร้างแพลตฟอร์มการดำเนินการ AI ระดับองค์กรได้อย่างแท้จริง

Codex SDK เป็น SDK ความช่วยเหลือด้านการเขียนโปรแกรมที่เผยแพร่โดย OpenAI ส่งคืนผลลัพธ์การดำเนินการผ่าน Event Stream ต่างจากโมเดลการตอบกลับคำขอแบบดั้งเดิม Codex ใช้กิจกรรมแบบสตรีมเพื่อให้เราสามารถ:

- รับความคืบหน้าในการดำเนินการแบบเรียลไทม์
- จัดการกับข้อผิดพลาดโดยทันที
- รับสถิติการใช้งานโทเค็นโดยละเอียด
- รองรับงานที่ซับซ้อนในระยะยาว

การทำความเข้าใจประเภทเหตุการณ์เหล่านี้และแยกวิเคราะห์อย่างถูกต้องถือเป็นสิ่งสำคัญสำหรับการนำ AI Executor ที่มีความสามารถเต็มรูปแบบไปใช้ ท้ายที่สุดแล้วไม่มีใครอยากทำงานกับกล่องดำ

## เกี่ยวกับ ฮากิโค้ด

วิธีแก้ปัญหาที่แบ่งปันในบทความนี้มาจากประสบการณ์จริงของเราใน [ฮากิโค้ด](https://hagicode.com) โครงการ HagiCode เป็นโปรเจ็กต์ผู้ช่วยการเขียนโค้ด AI แบบโอเพ่นซอร์สที่อุทิศตนเพื่อให้นักพัฒนาได้รับการสนับสนุนการเขียนโค้ดอัจฉริยะ ในระหว่างการพัฒนา เราจำเป็นต้องสร้างบริการดำเนินการ AI ที่เชื่อถือได้เพื่อจัดการกับคำขอเรียกใช้โค้ดของผู้ใช้ ซึ่งเป็นเหตุผลโดยตรงที่เราแนะนำ Codex SDK

ในฐานะผู้ช่วยเขียนโค้ด AI HagiCode จำเป็นต้องจัดการกับสถานการณ์การเรียกใช้โค้ดที่ซับซ้อนที่หลากหลาย: รับความคืบหน้าในการดำเนินการแบบเรียลไทม์ การจัดการข้อผิดพลาดทันที และรวบรวมสถิติการใช้งานโทเค็นโดยละเอียด ด้วยการทำความเข้าใจกลไกการสตรีมเหตุการณ์ Codex SDK อย่างลึกซึ้ง เราจึงสามารถสร้างตัวดำเนินการที่ตรงตามข้อกำหนดสภาพแวดล้อมการใช้งานจริงได้ ท้ายที่สุด ไม่ว่าจะเป็นซอฟต์แวร์หรือชีวิตจริง ทุกอย่างจะได้รับประโยชน์จากการสะสมและการปรับปรุงอย่างต่อเนื่อง

## กลไกการสตรีมเหตุการณ์

### แนวคิดพื้นฐาน

Codex SDK ใช้ `thread.runStreamed()` วิธีการส่งคืนตัววนซ้ำเหตุการณ์แบบอะซิงโครนัส:

```typescript
import { Codex } from '@openai/codex-sdk';

const client = new Codex({
  apiKey: process.env.CODEX_API_KEY,
  baseUrl: process.env.CODEX_BASE_URL,
});

const thread = client.startThread({
  workingDirectory: '/path/to/project',
  skipGitRepoCheck: false,
});

const { events } = await thread.runStreamed('your prompt here', {
  outputSchema: {
    type: 'object',
    properties: {
      output: { type: 'string' },
      status: { type: 'string', enum: ['ok', 'action_required'] },
    },
    required: ['output', 'status'],
  },
});

for await (const event of events) {
  // Handle each event
}
```

### คำอธิบายโดยละเอียดของประเภทเหตุการณ์

| ประเภทเหตุการณ์ | คำอธิบาย | ข้อมูลสำคัญ |
|---------|------|----------|
| `thread.started` | เริ่มกระทู้สำเร็จแล้ว | `thread_id` |
| `item.updated` | อัปเดตเนื้อหาข้อความแล้ว | `item.text` |
| `item.completed` | ข้อความเสร็จสมบูรณ์ | `item.text` |
| `turn.completed` | การดำเนินการเสร็จสิ้น | `usage` (การใช้โทเค็น) |
| `turn.failed` | การดำเนินการล้มเหลว | `error.message` |
| `error` | เหตุการณ์ข้อผิดพลาด | `message` |

ในโปรเจ็กต์จริง ส่วนประกอบตัวดำเนินการของ HagiCode ถูกสร้างขึ้นจากประเภทเหตุการณ์เหล่านี้ เราจำเป็นต้องจัดการกิจกรรมแต่ละประเภทอย่างระมัดระวังเพื่อให้แน่ใจว่าผู้ใช้จะได้รับประสบการณ์ที่ราบรื่น ระบบที่ดีถูกสร้างขึ้นจากการใส่ใจในรายละเอียดอย่างจริงจัง

## การดำเนินการแยกวิเคราะห์ข้อความ

### การแยกเนื้อหาข้อความ

เนื้อหาข้อความถูกแยกผ่านตัวจัดการเหตุการณ์:

```typescript
private handleThreadEvent(event: ThreadEvent, onMessage: (content: string) => void): void {
  // Only handle message update and completion events
  if (event.type !== 'item.updated' && event.type !== 'item.completed') {
    return;
  }

  // Only handle agent message content
  if (event.item.type !== 'agent_message') {
    return;
  }

  // Extract text content
  onMessage(event.item.text);
}
```

ประเด็นสำคัญ:
- ที่จับเท่านั้น `item.updated` และ `item.completed` เหตุการณ์ต่างๆ
- จัดการเฉพาะเนื้อหาประเภทเท่านั้น `agent_message`
- เนื้อหาข้อความอยู่ใน `event.item.text` สนาม

### การแยกวิเคราะห์เอาต์พุตที่มีโครงสร้าง

Codex รองรับเอาต์พุตที่มีโครงสร้าง JSON คุณสามารถระบุรูปแบบการคืนสินค้าผ่านทาง `outputSchema` พารามิเตอร์:

```typescript
const DEFAULT_OUTPUT_SCHEMA = {
  type: 'object',
  properties: {
    output: { type: 'string' },
    status: { type: 'string', enum: ['ok', 'action_required'] },
  },
  required: ['output', 'status'],
  additionalProperties: false,
} as const;
```

ฟังก์ชันการแยกวิเคราะห์จะพยายามแยกวิเคราะห์ JSON และหากล้มเหลว ก็จะกลับไปเป็นข้อความดิบ

```typescript
function toStructuredOutput(raw: string): StructuredOutput {
  try {
    const parsed = JSON.parse(raw) as Partial<StructuredOutput>;
    if (typeof parsed.output === 'string') {
      return {
        output: parsed.output,
        status: parsed.status === 'action_required' ? 'action_required' : 'ok',
      };
    }
  } catch {
    // JSON parsing failed, fall back to the raw text
  }

  return {
    output: raw,
    status: 'ok',
  };
}
```

### ทำตามขั้นตอนการจัดการเหตุการณ์ให้เสร็จสมบูรณ์

```typescript
private async runWithStreaming(
  thread: Thread,
  input: CodexStageExecutionInput
): Promise<{ output: string; usage: Usage | null }> {
  const abortController = new AbortController();
  const timeoutHandle = setTimeout(() => {
    abortController.abort();
  }, Math.max(1000, input.timeoutMs));

  let latestMessage = '';
  let usage: Usage | null = null;
  let emittedLength = 0;

  try {
    const { events } = await thread.runStreamed(input.prompt, {
      outputSchema: DEFAULT_OUTPUT_SCHEMA,
      signal: abortController.signal,
    });

    for await (const event of events) {
      // Handle message content
      this.handleThreadEvent(event, (nextContent) => {
        const delta = nextContent.slice(emittedLength);
        if (delta.length > 0) {
          emittedLength = nextContent.length;
          input.callbacks?.onChunk?.(delta);  // Streaming callback
        }
        latestMessage = nextContent;
      });

      // Process different data based on the event type
      if (event.type === 'thread.started') {
        this.threadId = event.thread_id;
      } else if (event.type === 'turn.completed') {
        usage = event.usage;
      } else if (event.type === 'turn.failed') {
        throw new CodexExecutorError('gateway_unavailable', event.error.message, true);
      } else if (event.type === 'error') {
        throw new CodexExecutorError('gateway_unavailable', event.message, true);
      }
    }
  } catch (error) {
    if (abortController.signal.aborted) {
      throw new CodexExecutorError(
        'upstream_timeout',
        `Codex stage timed out after ${input.timeoutMs}ms`,
        true
      );
    }
    throw error;
  } finally {
    clearTimeout(timeoutHandle);
  }

  const structured = toStructuredOutput(latestMessage);
  return { output: structured.output, usage };
}
```

## กลยุทธ์การจัดการข้อผิดพลาด

### การแมปรหัสข้อผิดพลาด

แมปรูปแบบข้อผิดพลาดเฉพาะกับรหัสข้อผิดพลาดที่เป็นรูปธรรม เพื่อให้ชั้นบนสามารถจัดการได้ง่ายขึ้น:

```typescript
function mapError(error: unknown): CodexExecutorError {
  if (error instanceof CodexExecutorError) {
    return error;
  }

  const message = error instanceof Error ? error.message : String(error);
  const normalized = message.toLowerCase();

  // Authentication errors - not retryable
  if (normalized.includes('401') ||
      normalized.includes('403') ||
      normalized.includes('api key') ||
      normalized.includes('auth')) {
    return new CodexExecutorError('auth_invalid', message, false);
  }

  // Rate limit errors - retryable
  if (normalized.includes('429') || normalized.includes('rate limit')) {
    return new CodexExecutorError('rate_limited', message, true);
  }

  // Timeout errors - retryable
  if (normalized.includes('timeout') || normalized.includes('aborted')) {
    return new CodexExecutorError('upstream_timeout', message, true);
  }

  // Default error
  return new CodexExecutorError('gateway_unavailable', message, true);
}
```

### คำจำกัดความประเภทข้อผิดพลาด

```typescript
export type CodexErrorCode =
  | 'auth_invalid'      // Authentication failure
  | 'upstream_timeout'  // Upstream timeout
  | 'rate_limited'      // Rate limited
  | 'gateway_unavailable'; // Gateway unavailable

export class CodexExecutorError extends Error {
  readonly code: CodexErrorCode;
  readonly retryable: boolean;

  constructor(code: CodexErrorCode, message: string, retryable: boolean) {
    super(message);
    this.name = 'CodexExecutorError';
    this.code = code;
    this.retryable = retryable;
  }
}
```

## ไดเร็กทอรีการทำงานและการกำหนดค่าสภาพแวดล้อม

### การตรวจสอบความถูกต้องของไดเร็กทอรีการทำงาน

Codex SDK กำหนดให้ไดเร็กทอรีการทำงานเป็นที่เก็บ Git ที่ถูกต้อง

```typescript
export function validateWorkingDirectory(
  workingDirectory: string,
  skipGitRepoCheck: boolean
): void {
  const resolvedWorkingDirectory = path.resolve(workingDirectory);

  if (!existsSync(resolvedWorkingDirectory)) {
    throw new CodexExecutorError(
      'gateway_unavailable',
      'Working directory does not exist.',
      false
    );
  }

  if (!statSync(resolvedWorkingDirectory).isDirectory()) {
    throw new CodexExecutorError(
      'gateway_unavailable',
      'Working directory is not a directory.',
      false
    );
  }

  if (skipGitRepoCheck) {
    return;
  }

  const gitDir = path.join(resolvedWorkingDirectory, '.git');
  if (!existsSync(gitDir)) {
    throw new CodexExecutorError(
      'gateway_unavailable',
      'Working directory is not a git repository.',
      false
    );
  }
}
```

### กำลังโหลดตัวแปรสภาพแวดล้อม

Codex SDK จำเป็นต้องโหลดตัวแปรสภาพแวดล้อมจากเชลล์การเข้าสู่ระบบเพื่อให้ AI Agent สามารถเข้าถึงคำสั่งระบบได้:

```typescript
function parseEnvironmentOutput(output: Buffer): Record<string, string> {
  const parsed: Record<string, string> = {};

  for (const entry of output.toString('utf8').split('\0')) {
    if (!entry) continue;

    const separatorIndex = entry.indexOf('=');
    if (separatorIndex <= 0) continue;

    const key = entry.slice(0, separatorIndex);
    const value = entry.slice(separatorIndex + 1);
    if (key.length > 0) {
      parsed[key] = value;
    }
  }

  return parsed;
}

function tryLoadEnvironmentFromShell(shellPath: string): Record<string, string> | null {
  const result = spawnSync(shellPath, ['-ilc', 'env -0'], {
    env: process.env,
    stdio: ['ignore', 'pipe', 'pipe'],
    timeout: 5000,
  });

  if (result.error || result.status !== 0) {
    return null;
  }

  return parseEnvironmentOutput(result.stdout);
}

export function createExecutorEnvironment(
  envOverrides: Record<string, string> = {}
): Record<string, string> {
  // Load environment variables from the login shell
  const consoleEnv = loadConsoleEnvironmentFromShell();

  return {
    ...process.env,
    ...consoleEnv,
    ...envOverrides,
  };
}
```

## ตัวอย่างการใช้งานที่สมบูรณ์

### การใช้งานขั้นพื้นฐาน

ในโครงการ HagiCode เราใช้วิธีการต่อไปนี้เพื่อเริ่มต้นไคลเอนต์ Codex และดำเนินงาน:

```typescript
import { Codex } from '@openai/codex-sdk';

async function executeWithCodex(prompt: string, workingDir: string) {
  const client = new Codex({
    apiKey: process.env.CODEX_API_KEY,
    env: { PATH: process.env.PATH },
  });

  const thread = client.startThread({
    workingDirectory: workingDir,
  });

  const { events } = await thread.runStreamed(prompt);

  let result = '';
  for await (const event of events) {
    if (event.type === 'item.updated' && event.item.type === 'agent_message') {
      result = event.item.text;
    }
    if (event.type === 'turn.completed') {
      console.log('Token usage:', event.usage);
    }
  }

  // Try to parse JSON output
  try {
    const parsed = JSON.parse(result);
    return parsed.output;
  } catch {
    return result;
  }
}
```

### ดำเนินการใช้งานให้เสร็จสิ้นด้วยการลองใหม่

```typescript
export class CodexSdkExecutor {
  private readonly config: CodexRuntimeConfig;
  private readonly client: Codex;
  private threadId: string | null = null;

  async executeStage(input: CodexStageExecutionInput): Promise<CodexStageExecutionResult> {
    const maxAttempts = Math.max(1, this.config.retryCount + 1);
    let attempt = 0;
    let lastError: CodexExecutorError | null = null;

    while (attempt < maxAttempts) {
      attempt += 1;

      try {
        const thread = this.getThread(input.workingDirectory);
        const { output, usage } = await this.runWithStreaming(thread, input);

        return {
          output,
          usage,
          threadId: this.threadId!,
          attempts: attempt,
          latencyMs: Date.now() - startedAt,
        };
      } catch (error) {
        const mappedError = mapError(error);
        lastError = mappedError;

        // Non-retryable error or max retry attempts reached
        if (!mappedError.retryable || attempt >= maxAttempts) {
          throw mappedError;
        }

        // Wait before retrying
        await new Promise(resolve => setTimeout(resolve, 1000 * attempt));
      }
    }

    throw lastError!;
  }
}
```

## แนวทางปฏิบัติที่ดีที่สุด

### 1. ข้อกำหนดของไดเร็กทอรีการทำงาน

- ตรวจสอบให้แน่ใจว่าไดเร็กทอรีการทำงานเป็นที่เก็บ Git ที่ถูกต้อง
- ใช้ `PROJECT_ROOT` ตัวแปรสภาพแวดล้อมเพื่อระบุอย่างชัดเจน
- ระหว่างการดีบัก คุณสามารถตั้งค่าได้ `CODEX_SKIP_GIT_REPO_CHECK=true` เพื่อข้ามการตรวจสอบ

### 2. การกำหนดค่าตัวแปรสภาพแวดล้อม

- ส่งผ่านเฉพาะตัวแปรสภาพแวดล้อมที่จำเป็นผ่านกลไกไวท์ลิสต์
- ใช้เชลล์การเข้าสู่ระบบเพื่อโหลดสภาพแวดล้อมทั้งหมด
- หลีกเลี่ยงการส่งข้อมูลที่ละเอียดอ่อน

### 3. หมดเวลาและลองกลยุทธ์อีกครั้ง

- ตั้งค่าการหมดเวลาที่เหมาะสมตามความซับซ้อนของงาน
- ใช้ Exponential Backoff สำหรับข้อผิดพลาดที่ลองใหม่ได้
- บันทึกจำนวนและเหตุผลในการลองซ้ำ

### 4. การจัดการข้อผิดพลาด

- แยกความแตกต่างระหว่างข้อผิดพลาดแบบลองซ้ำได้และลองซ้ำไม่ได้
- ระบุข้อความแสดงข้อผิดพลาดและข้อเสนอแนะที่ชัดเจน
- ใช้รหัสข้อผิดพลาดแบบรวมเพื่อให้ชั้นบนสามารถจัดการได้อย่างสม่ำเสมอ

### 5. เอาท์พุทสตรีมมิ่ง

- ใช้การเรียกกลับเอาต์พุตส่วนเพิ่มเพื่อปรับปรุงประสบการณ์ผู้ใช้
- จัดการการอัปเดตข้อความที่เพิ่มขึ้นอย่างถูกต้อง
- บันทึกการใช้โทเค็นเพื่อการวิเคราะห์ต้นทุน

ในสภาพแวดล้อมการผลิตจริงของโครงการ HagiCode เราได้ตรวจสอบประสิทธิภาพของแนวทางปฏิบัติที่ดีที่สุดข้างต้นแล้ว แนวทางนี้ได้ช่วยให้เราสร้างบริการดำเนินการ AI ที่มีเสถียรภาพและเชื่อถือได้ ในท้ายที่สุด การตรวจสอบเชิงปฏิบัติมีความสำคัญมากกว่าทฤษฎีเพียงอย่างเดียว

## บทสรุป

กลไกการสตรีมเหตุการณ์ Codex SDK มอบความสามารถที่แข็งแกร่งสำหรับการสร้างบริการการดำเนินการ AI ด้วยการแยกวิเคราะห์เหตุการณ์ประเภทต่างๆ อย่างถูกต้อง เราสามารถ:

- รับสถานะการดำเนินการและเอาต์พุตแบบเรียลไทม์
- ใช้กลไกการจัดการข้อผิดพลาดและลองใหม่ที่เชื่อถือได้
- รับสถิติการดำเนินการโดยละเอียด
- สร้างแพลตฟอร์มการดำเนินการ AI ที่มีคุณสมบัติครบถ้วน

แนวคิดหลักและตัวอย่างโค้ดที่แนะนำในบทความนี้สามารถนำไปใช้โดยตรงในโปรเจ็กต์จริง ช่วยให้นักพัฒนาเริ่มต้นได้อย่างรวดเร็วด้วยการผสานรวม Codex SDK หากคุณพบว่าแนวทางนี้มีคุณค่า ก็จะสะท้อนถึงจุดแข็งของการปฏิบัติงานด้านวิศวกรรมของ HagiCode และทำให้ HagiCode คุ้มค่าที่จะปฏิบัติตาม
