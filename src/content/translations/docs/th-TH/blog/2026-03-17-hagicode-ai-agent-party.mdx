## การสร้างปาร์ตี้ผจญภัยด้วย AI: คู่มือปฏิบัติสำหรับการกำหนดค่าการทำงานร่วมกันหลายตัวแทนใน HagiCode

> ในการพัฒนาซอฟต์แวร์ยุคใหม่ AI Agent ตัวเดียวไม่เพียงพอสำหรับความต้องการที่ซับซ้อนอีกต่อไป ผู้ช่วย AI หลายคนจากบริษัทต่างๆ สามารถทำงานร่วมกันภายในโปรเจ็กต์เดียวกันได้อย่างไร บทความนี้แบ่งปันแนวทางการกำหนดค่าการทำงานร่วมกันแบบหลายตัวแทนที่โครงการ HagiCode พัฒนาผ่านการปฏิบัติงานจริง

:::note[หมายเหตุบริบทเวลา]
บทความนี้บันทึกแนวทางปฏิบัติในการกำหนดค่าหลายตัวแทน **ตามที่เผยแพร่เมื่อวันที่ 17-03-2026** ขอบเขตที่รองรับระยะยาวในปัจจุบันเป็นไปตามการกำหนดค่าปัจจุบันของแบ็กเอนด์: Claude Code, Codex, GitHub Copilot, OpenCode, Hermes, QoderCLI, Kiro, Kimi, Gemini, DeepAgents และ Codebuddy ในหมู่พวกเขา `IFlowCli` ขณะนี้จะถูกเก็บไว้เป็นค่าที่เข้ากันได้กับระบบเดิมเท่านั้น และไม่ได้เป็นส่วนหนึ่งของชุดที่ได้รับการสนับสนุนอีกต่อไป หากคุณต้องการขอบเขตการติดตั้งและการสนับสนุนปัจจุบัน โปรดดู [การติดตั้ง AI Agent CLI](/th-TH/related-software-installation/ai-agent-cli/) และ [คู่มือการติดตั้ง](/th-TH/installation).
:::

## พื้นหลัง

นักพัฒนาหลายคนน่าจะเคยมีประสบการณ์นี้: การนำผู้ช่วย AI เข้ามาในโปรเจ็กต์ช่วยปรับปรุงประสิทธิภาพการเขียนโค้ดได้จริงๆ แต่เมื่อความต้องการเริ่มซับซ้อนมากขึ้น เจ้าหน้าที่ AI คนหนึ่งก็เริ่มขาดแคลน คุณต้องการให้ตรวจสอบโค้ด การสร้างเอกสาร การทดสอบหน่วย และอื่นๆ ในเวลาเดียวกัน แต่ผลลัพธ์มักจะทำให้ทุกอย่างสมดุลกันไม่ได้ และคุณภาพเอาต์พุตไม่สอดคล้องกัน

สิ่งที่น่าหงุดหงิดยิ่งกว่าคือเมื่อคุณพยายามแนะนำผู้ช่วย AI หลายคน สิ่งต่างๆ จะซับซ้อนมากขึ้น เอเจนต์แต่ละรายมีวิธีการกำหนดค่า อินเทอร์เฟซ API และตรรกะการดำเนินการของตนเอง และอาจขัดแย้งกันเองด้วยซ้ำ เปรียบเสมือนทีมกีฬาที่ผู้เล่นแต่ละคนแข็งแกร่งแต่ไม่มีใครรู้จักการประสานงานกันจนทำให้ทั้งแมตช์กลายเป็นความวุ่นวาย

โครงการ HagiCode ประสบปัญหาเดียวกันระหว่างการพัฒนา เนื่องจากเป็นโปรเจ็กต์ที่ซับซ้อนที่เกี่ยวข้องกับส่วนขยาย VSCode ส่วนหน้า บริการ AI แบ็กเอนด์ และไคลเอ็นต์เดสก์ท็อปข้ามแพลตฟอร์ม ในเวอร์ชัน **ปี 2026-03 ในขณะนั้น** เราจำเป็นต้องผสานรวมผู้ช่วย AI หลายคนจากบริษัทต่างๆ ในคราวเดียว: Claude Code, Codex, CodeBuddy, iFlow และอื่นๆ การหาวิธีให้พวกเขาอยู่ร่วมกันอย่างกลมกลืนในโครงการเดียวกันในขณะที่ใช้จุดแข็งของแต่ละคนให้เกิดประโยชน์สูงสุดกลายเป็นปัญหาสำคัญที่เราต้องแก้ไข

แค่นี้ก็คงเป็นปัญหามากพอแล้ว ท้ายที่สุดแล้วใครจะอยากจัดการกับกลุ่มเครื่องมือ AI ที่ทะเลาะกันทุกวัน?

แนวทางที่แบ่งปันในบทความนี้คือแนวปฏิบัติในการกำหนดค่าการทำงานร่วมกันแบบหลายตัวแทนที่เราพัฒนาในโครงการ HagiCode ผ่านการลองผิดลองถูกจริงและการเพิ่มประสิทธิภาพซ้ำหลายครั้ง หากคุณกำลังประสบปัญหากับผู้ช่วย AI หลายคนที่ทำงานร่วมกัน บทความนี้อาจให้แนวคิดบางอย่างแก่คุณ อาจจะ. ทุกโครงการมีความแตกต่างกัน

## การออกแบบสถาปัตยกรรมหลายตัวแทนของ HagiCode

### จาก "การลุยเดี่ยว" สู่ "การทำงานร่วมกันเป็นทีม"

ในช่วงแรกๆ ของโปรเจ็กต์ HagiCode เรายังลองใช้ AI Agent ตัวเดียวเพื่อจัดการทุกอย่าง เราค้นพบจุดคอขวดที่ชัดเจนอย่างรวดเร็วในแนวทางนั้น งานที่แตกต่างกันต้องการจุดแข็งที่แตกต่างกัน งานบางอย่างต้องการความเข้าใจบริบทที่แข็งแกร่ง ในขณะที่งานอื่นๆ ต้องการการแก้ไขโค้ดที่แม่นยำยิ่งขึ้น เอเจนท์หนึ่งคนมีช่วงเวลาที่ยากลำบากในการเป็นเลิศในทุกรายการ

นั่นทำให้เราตระหนักว่าตัวแทนหลายคนต้องทำงานร่วมกัน แต่ปัญหาคือ คุณจะปล่อยให้ผลิตภัณฑ์ AI จากบริษัทต่างๆ อยู่ร่วมกันอย่างสันติในโครงการเดียวกันได้อย่างไร เราจำเป็นต้องแก้ไขปัญหาหลักหลายประการ:

1. **ความซับซ้อนในการจัดการการกำหนดค่า**: เอเจนต์แต่ละตัวมีวิธีการกำหนดค่า อินเทอร์เฟซ API และโหมดการดำเนินการที่แตกต่างกัน
2. **โปรโตคอลการสื่อสารแบบครบวงจร**: เราต้องการวิธีที่เป็นมาตรฐานสำหรับตัวแทนต่างๆ ในการแลกเปลี่ยนข้อมูล
3. **การประสานงานงานและการแบ่งงาน**: เราจะมอบหมายงานอย่างสมเหตุสมผลได้อย่างไร เพื่อให้ตัวแทนแต่ละคนแสดงจุดแข็งของตนได้

เมื่อมีคำถามเหล่านี้แล้ว เราจึงเริ่มออกแบบสถาปัตยกรรมแบบ multi-Agent ของ HagiCode ท้ายที่สุดแล้วมันไม่ได้ซับซ้อนขนาดนั้น เราแค่ต้องคิดให้ชัดเจน

### สรุปสถาปัตยกรรมโดยรวม

หลังจากทำซ้ำหลายครั้ง นี่คือสถาปัตยกรรมที่เราตัดสินใจ:

```
┌─────────────────────────────────────────────────────────────────┐
│                    AIProviderFactory                             │
│  (Factory pattern for unified management of all AI Providers)    │
├─────────────────────────────────────────────────────────────────┤
│  ClaudeCodeCli  │  CodexCli  │  CodebuddyCli  │  IFlowCli    │
│  (Anthropic)   │  (OpenAI)  │  (Zhipu GLM)    │  (Zhipu)     │
└─────────────────────────────────────────────────────────────────┘
```

แนวคิดหลักคือการให้ AI Agent ที่แตกต่างกันได้รับการจัดการด้วยรหัสเดียวกันผ่านอินเทอร์เฟซผู้ให้บริการแบบรวม ในเวลาเดียวกัน รูปแบบโรงงานจะถูกใช้เพื่อสร้างและกำหนดค่าผู้ให้บริการเหล่านี้แบบไดนามิก เพื่อให้มั่นใจถึงความสามารถในการขยายขนาดและความยืดหยุ่นทั่วทั้งระบบ

เปรียบเสมือนการแบ่งงานในชีวิตประจำวัน ทุกคนมีบทบาท ที่นี่เราเพียงแค่เปลี่ยนแนวคิดนั้นให้เป็นสถาปัตยกรรมโค้ด

### ประเภทตัวแทนและฝ่ายความรับผิดชอบ

จากประสบการณ์จริงของ HagiCode เราได้มอบหมายความรับผิดชอบที่แตกต่างกันให้กับตัวแทนแต่ละคน:

| ตัวแทน | ผู้ให้บริการ | รุ่น | การใช้งานหลัก |
|-------|----------|-------|-------------|
| ClaudeCodeCli | มานุษยวิทยา | glm-5-เทอร์โบ | สร้างโซลูชันและข้อเสนอทางเทคนิค |
| CodexCli | OpenAI/Zed | gpt-5.4 | ดำเนินการเปลี่ยนแปลงโค้ดอย่างแม่นยำ |
| CodebuddyCli | ซิปู | glm-4.7 | ปรับแต่งคำอธิบายข้อเสนอและเอกสารประกอบ |
| IFlowCli | ซิปู | glm-4.7 | เก็บถาวรข้อเสนอและบันทึกทางประวัติศาสตร์ (การกำหนดค่าในขณะนั้น ขณะนี้เข้ากันได้กับระบบเดิมเท่านั้น) |
| OpenCodeCli | - | - | การแก้ไขโค้ดวัตถุประสงค์ทั่วไป |
| GitHubCopilot | ไมโครซอฟต์ | - | ช่วยเขียนโปรแกรมและเขียนโค้ดให้สมบูรณ์ |

ตรรกะเบื้องหลังการแบ่งงานนั้นเรียบง่าย: เอเจนท์ทุกคนมีพื้นที่ความแข็งแกร่งเป็นของตัวเอง Claude Code ทำงานได้ดีในการทำความเข้าใจและวิเคราะห์ข้อกำหนดที่ซับซ้อน ดังนั้นจึงสามารถรองรับการออกแบบโซลูชันตั้งแต่เนิ่นๆ ได้ Codex มีความแม่นยำมากขึ้นเมื่อแก้ไขโค้ด ดังนั้นจึงเหมาะกับการใช้งานที่เป็นรูปธรรมมากกว่า CodeBuddy นำเสนอประสิทธิภาพด้านต้นทุนที่แข็งแกร่ง ซึ่งทำให้เหมาะอย่างยิ่งสำหรับการปรับแต่งเอกสาร

ท้ายที่สุดแล้ว เครื่องมือที่เหมาะสมสำหรับงานที่เหมาะสมมักเป็นตัวเลือกที่ดีที่สุด มีถนนหลายสายสู่กรุงโรม บางอันก็เดินง่ายกว่าอันอื่น

## กลไกการกำหนดค่าหลัก

### การออกแบบส่วนต่อประสานผู้ให้บริการแบบรวม

ในการจัดการเอเจนต์ AI ต่างๆ ในลักษณะที่เป็นหนึ่งเดียว เราต้องกำหนดอินเทอร์เฟซทั่วไปก่อน ใน HagiCode อินเทอร์เฟซนั้นมีลักษณะดังนี้:

```csharp
public interface IAIProvider
{
    // Unified Provider interface
    Task<IAIProvider?> GetProviderAsync(AIProviderType providerType);
    Task<IAIProvider?> GetProviderAsync(string providerName, CancellationToken cancellationToken);
}
```

อินเทอร์เฟซดูเรียบง่าย แต่เป็นรากฐานของระบบหลายตัวแทนทั้งหมด ด้วยอินเทอร์เฟซแบบรวม เราสามารถเรียกผลิตภัณฑ์ AI จากบริษัทต่างๆ ได้ในลักษณะเดียวกันทุกประการ ไม่ว่าจะอยู่ใต้อะไรก็ตาม

นี่เป็นเพียงเรื่องของการทำให้สิ่งที่ซับซ้อนเป็นเรื่องง่าย เรียบง่ายก็สวยงาม

### การดำเนินการตามรูปแบบโรงงานของผู้ให้บริการ

เมื่ออินเทอร์เฟซรวมเป็นหนึ่งเดียว คำถามถัดไปคือจะสร้างอินสแตนซ์ของผู้ให้บริการเหล่านี้ได้อย่างไร HagiCode ใช้รูปแบบโรงงาน:

```csharp
private IAIProvider? CreateProvider(AIProviderType providerType, ProviderConfiguration config)
{
    return providerType switch
    {
        AIProviderType.ClaudeCodeCli =>
            ActivatorUtilities.CreateInstance<ClaudeCodeCliProvider>(_serviceProvider, Options.Create(config)),
        AIProviderType.CodebuddyCli =>
            ActivatorUtilities.CreateInstance<CodebuddyCliProvider>(_serviceProvider, Options.Create(config)),
        AIProviderType.CodexCli =>
            ActivatorUtilities.CreateInstance<CodexCliProvider>(_serviceProvider, Options.Create(config)),
        AIProviderType.IFlowCli =>
            ActivatorUtilities.CreateInstance<IFlowCliProvider>(_serviceProvider, Options.Create(config)),
        _ => null
    };
}
```

สิ่งนี้ใช้การฉีดการพึ่งพาผ่าน `ActivatorUtilities.CreateInstance`ซึ่งสามารถสร้างอินสแตนซ์ของผู้ให้บริการแบบไดนามิกในขณะรันไทม์ในขณะที่แทรกการขึ้นต่อกันโดยอัตโนมัติ ประโยชน์ของการออกแบบนี้คือ เมื่อมีการเพิ่มประเภท Agent ใหม่ คุณจะต้องเพิ่มคลาส Provider ที่เกี่ยวข้องเท่านั้น จากนั้นจึงเพิ่มสาขาเคสอีกหนึ่งรายการในวิธีการแบบโรงงาน ไม่จำเป็นต้องแก้ไขโค้ดที่มีอยู่เลย

นั่นเป็นเหตุผลเพียงพอแล้ว ใครอยากจะเขียนโค้ดเก่ากองใหม่ทุกครั้งที่มีการเพิ่มฟีเจอร์ใหม่?

### ความละเอียดการกำหนดค่าแบบไดนามิก

เพื่อให้การกำหนดค่ามีความยืดหยุ่นมากขึ้น เรายังใช้กลไกการแมปประเภท:

```csharp
public static AIProviderTypeExtensions
{
    private static readonly Dictionary<string, AIProviderType> _typeMap = new(
        StringComparer.OrdinalIgnoreCase)
    {
        ["ClaudeCodeCli"] = AIProviderType.ClaudeCodeCli,
        ["CodebuddyCli"] = AIProviderType.CodebuddyCli,
        ["CodexCli"] = AIProviderType.CodexCli,
        ["IFlowCli"] = AIProviderType.IFlowCli,
        // ...more type mappings
    };
}
```

วัตถุประสงค์ของตารางการแมปนี้คือการแปลงชื่อผู้ให้บริการในรูปแบบสตริงให้เป็นประเภทแจงนับ ซึ่งช่วยให้ไฟล์การกำหนดค่าใช้ชื่อสตริงที่เข้าใจง่าย ในขณะที่โค้ดภายในใช้ enum ที่ปลอดภัยสำหรับการประมวลผล

การกำหนดค่าควรใช้งานง่ายที่สุดเท่าที่จะเป็นไปได้ ไม่มีใครอยากจำกองชื่อโค้ดที่คลุมเครือ

### ตัวอย่างไฟล์การกำหนดค่า

ในทางปฏิบัติ ทุกอย่างสามารถกำหนดค่าได้ `appsettings.json`:

```yaml
AI:
  Providers:
    Providers:
      ClaudeCodeCli:
        Enabled: true
        Model: glm-5-turbo
        WorkingDirectory: /path/to/project
      CodebuddyCli:
        Enabled: true
        Model: glm-4.7
      CodexCli:
        Enabled: true
        Model: gpt-5.4
      IFlowCli:
        Enabled: true
        Model: glm-4.7
```

ผู้ให้บริการแต่ละรายสามารถกำหนดค่าพารามิเตอร์ต่างๆ เช่น การเปิดใช้งาน เวอร์ชันโมเดล และไดเร็กทอรีการทำงานได้อย่างอิสระ การออกแบบนี้รักษาความยืดหยุ่นในขณะที่ยังคงง่ายต่อการจัดการและบำรุงรักษา

ในบางแง่ ไฟล์การกำหนดค่าก็เหมือนกับตัวเลือกของชีวิต: คุณสามารถเลือกที่จะเปิดหรือปิดการใช้งานบางอย่างได้ ข้อแตกต่างเพียงอย่างเดียวคือตัวเลือกโค้ดจะง่ายกว่าที่จะเสียใจในภายหลัง

## ขั้นตอนงานปาร์ตี้ผจญภัย

### สาขาวิชาศิลปะงาน

ด้วยสถาปัตยกรรมทางเทคนิคที่เป็นหนึ่งเดียว ขั้นตอนต่อไปคือการทำให้ตัวแทนหลายรายทำงานร่วมกัน HagiCode ได้ออกแบบกลไกการไหลของงานเพื่อให้ตัวแทนที่แตกต่างกันสามารถจัดการขั้นตอนต่างๆ ของงานได้:

```
Proposal creation (user)
    │
    ▼
[Claude Code] ──generate proposal──▶ Proposal document
    │                               │
    │                               ▼
    │                      [Codebuddy] ──refine description──▶ Refined proposal
    │                               │
    │                               ▼
    │                      [Codex] ──execute changes──▶ Code changes
    │                               │
    │                               ▼
    └──────────────────────▶ [iFlow] ──archive──▶ Historical records
```

ประโยชน์ของการแบ่งงานนี้คือเอเจนต์แต่ละคนจะต้องมุ่งเน้นไปที่งานที่ทำได้ดีที่สุดเท่านั้น แทนที่จะพยายามทำทุกอย่าง Claude Code สร้างข้อเสนอตั้งแต่เริ่มต้น Codebuddy ทำให้คำอธิบายข้อเสนอชัดเจนยิ่งขึ้น Codex เปลี่ยนข้อเสนอให้เป็นการเปลี่ยนแปลงโค้ดจริง iFlow จัดเก็บและรักษาการเปลี่ยนแปลงเหล่านั้น

นี่เป็นเพียงการทำงานเป็นทีมเช่นเดียวกับในชีวิตประจำวัน ทุกคนมีบทบาท และมีเพียงความร่วมมือเท่านั้นที่จะทำสิ่งที่ยิ่งใหญ่ให้สำเร็จได้ สมาชิกในทีมนี้บังเอิญเป็น AI

### ประเด็นสำคัญในทางปฏิบัติ

ในการใช้งานจริง เราได้สรุปบทเรียนต่อไปนี้:

**1. กลยุทธ์การเลือกตัวแทนมีความสำคัญ**

ไม่ควรมอบหมายงานแบบไม่เป็นทางการ ควรจับคู่กับจุดแข็งของเอเจนท์แต่ละคน:
- การสร้างข้อเสนอ: ใช้ Claude Code เนื่องจากมีความเข้าใจบริบทที่ชัดเจนยิ่งขึ้น
- การดำเนินการโค้ด: ใช้ Codex เนื่องจากมีความแม่นยำในการแก้ไขโค้ดมากกว่า
- การปรับแต่งข้อเสนอ: ใช้ Codebuddy เพราะมันให้ประสิทธิภาพด้านต้นทุนที่สูง
- พื้นที่เก็บข้อมูลถาวร: ใช้ iFlow เนื่องจากมีความเสถียรและเชื่อถือได้

ท้ายที่สุดแล้ว การวางคนให้ถูกกับงานที่ถูกต้องเป็นหลักการที่อยู่เหนือกาลเวลา

**2. การแยกการกำหนดค่าทำให้มั่นใจถึงความเสถียร**

การกำหนดค่าของตัวแทนแต่ละรายได้รับการจัดการอย่างเป็นอิสระ รองรับการแทนที่ตัวแปรสภาพแวดล้อม และใช้ไดเร็กทอรีการทำงานที่แยกจากกัน ด้วยเหตุนี้ ข้อผิดพลาดในการกำหนดค่าในเอเจนต์หนึ่งจึงไม่ส่งผลกระทบต่อเอเจนต์อื่นๆ

นี่เป็นเหมือนขอบเขตส่วนบุคคลในชีวิต ทุกคนต้องการพื้นที่ของตัวเอง การไม่รบกวนทำให้สามารถอยู่ร่วมกันได้

**3. กลไกการจัดการข้อผิดพลาด**

ความล้มเหลวในเอเจนต์เดียวไม่ควรส่งผลกระทบต่อเวิร์กโฟลว์โดยรวม เราใช้กลยุทธ์ทางเลือก: เมื่อตัวแทนรายหนึ่งล้มเหลว ระบบสามารถสลับไปใช้แผนสำรองโดยอัตโนมัติหรือข้ามขั้นตอนนั้นและทำงานต่อในภายหลังได้ ในขณะเดียวกัน การบันทึกที่สมบูรณ์ทำให้การแก้ไขปัญหาง่ายขึ้นในภายหลัง

ไม่มีใครรับประกันได้ว่าข้อผิดพลาดจะไม่เกิดขึ้น สิ่งสำคัญคือคุณจะจัดการกับมันอย่างไร ชีวิตทำงานในลักษณะเดียวกันมาก

**4. การติดตามและการสังเกต**

ด้วยโปรโตคอล ACP (โปรโตคอลการสื่อสารแบบกำหนดเองของเราที่ใช้ JSON-RPC 2.0) เราสามารถติดตามสถานะการดำเนินการของตัวแทนแต่ละรายได้ การแยกเซสชันช่วยให้มั่นใจถึงความปลอดภัยในการทำงานพร้อมกัน ในขณะที่แคชแบบไดนามิกช่วยปรับปรุงประสิทธิภาพ

สิ่งที่คุณมองไม่เห็นมักเป็นสิ่งที่มีแนวโน้มที่จะผิดพลาดมากที่สุด ทัศนวิสัยบางอย่างย่อมดีกว่าการตาบอดเสมอ

## ผลลัพธ์และคุณประโยชน์ในโลกแห่งความเป็นจริง

หลังจากใช้การกำหนดค่าการทำงานร่วมกันแบบหลายตัวแทนนี้ ประสิทธิภาพการพัฒนาของโครงการ HagiCode ก็ดีขึ้นอย่างมาก โดยเฉพาะ:

1. **ความสามารถในการจัดการงานเพิ่มขึ้นสองเท่า**: ในอดีต เจ้าหน้าที่หนึ่งคนต้องจัดการงานหลายประเภทในคราวเดียว ขณะนี้งานสามารถประมวลผลแบบคู่ขนานได้ และปริมาณงานก็เพิ่มขึ้นอย่างมาก
2. **คุณภาพผลงานที่มีเสถียรภาพมากขึ้น**: ตัวแทนแต่ละรายมุ่งเน้นเฉพาะสิ่งที่ทำได้ดีที่สุดเท่านั้น ดังนั้นทั้งความสม่ำเสมอและคุณภาพจึงได้รับการปรับปรุง
3. **ค่าบำรุงรักษาต่ำกว่า**: อินเทอร์เฟซแบบรวมและการจัดการการกำหนดค่าทำให้ทั้งระบบง่ายต่อการบำรุงรักษาและขยาย
4. **การเพิ่มตัวแทนใหม่นั้นง่ายดาย**: ในการผสานรวมผลิตภัณฑ์ AI ใหม่ คุณเพียงแค่ต้องใช้อินเทอร์เฟซและเพิ่มการกำหนดค่า โดยไม่ต้องเปลี่ยนลอจิกหลัก

แนวทางนี้ไม่เพียงแต่แก้ปัญหาของ HagiCode เท่านั้น แต่ยังพิสูจน์ว่าการทำงานร่วมกันหลายตัวแทนเป็นทางเลือกทางสถาปัตยกรรมที่ใช้ได้

กำไรค่อนข้างชัดเจน กระบวนการนี้ยุ่งยากเล็กน้อย

## บทสรุป

บทความนี้ได้แบ่งปันประสบการณ์จริงของโปรเจ็กต์ HagiCode ด้วยการกำหนดค่าการทำงานร่วมกันแบบหลายตัวแทน ประเด็นหลัก ได้แก่ :

1. **อินเทอร์เฟซมาตรฐาน**: `IAIProvider` รวมพฤติกรรมของตัวแทนต่างๆ เข้าด้วยกัน โดยอนุญาตให้โค้ดเพิกเฉยต่อผลิตภัณฑ์ของบริษัทที่อยู่ข้างใต้
2. **แบบโรงงาน**: `ActivatorUtilities.CreateInstance` สร้างอินสแตนซ์ของผู้ให้บริการแบบไดนามิก รองรับการกำหนดค่ารันไทม์และการฉีดการพึ่งพา
3. **การรวมโปรโตคอล**: โปรโตคอล ACP ให้การสื่อสารที่เป็นมาตรฐานระหว่างเอเจนต์ผ่านกลไกสองทิศทางที่ใช้ JSON-RPC 2.0
4. **การกำหนดเส้นทางงาน**: มอบหมายงานอย่างสมเหตุสมผลให้กับตัวแทนที่แตกต่างกัน เพื่อให้แต่ละคนสามารถแสดงจุดแข็งของตนเองได้ แทนที่จะคาดหวังให้ตัวแทนเพียงคนเดียวทำทุกอย่าง

การออกแบบนี้ไม่เพียงแต่แก้ปัญหา "เอเจนต์หลายตัวต่อสู้กันเอง" เท่านั้น แต่ยังใช้กลไกโฟลว์งานปาร์ตี้ผจญภัยเพื่อทำให้กระบวนการพัฒนาเป็นแบบอัตโนมัติและเชี่ยวชาญมากขึ้น

หากคุณกำลังพิจารณาที่จะแนะนำผู้ช่วย AI หลายคนด้วย ฉันหวังว่าบทความนี้จะให้ข้อมูลอ้างอิงที่เป็นประโยชน์บางประการแก่คุณ แน่นอนว่าทุกโครงการมีความแตกต่างกัน และยังคงต้องมีการปรับแนวทางเฉพาะให้เข้ากับสถานการณ์จริง ไม่มีวิธีแก้ปัญหาแบบใดที่เหมาะกับทุกคน ทางออกที่ดีที่สุดคือทางออกที่เหมาะกับคุณ

สิ่งสวยงามหรือคนไม่จำเป็นต้องครอบครอง ตราบใดที่มันยังคงสวยงาม เพียงแค่ชื่นชมความงามนั้นก็เพียงพอแล้ว โซลูชันด้านเทคนิคเหมือนกัน: โซลูชันที่เหมาะกับคุณคือโซลูชันที่ดีที่สุด...
