## Design.md: โซลูชันสำหรับการออกแบบ UI ส่วนหน้าที่ขับเคลื่อนด้วย AI ที่สอดคล้องกัน

> ในยุคของการพัฒนาส่วนหน้าโดยใช้ AI คุณจะรักษา UI ที่สร้างโดย AI ให้สอดคล้องกันได้อย่างไร บทความนี้จะแชร์ประสบการณ์ตรงของเราในการสร้างไซต์แกลเลอรีการออกแบบโดยอิงจาก Awesome-design-md พร้อมด้วยวิธีสร้างไซต์แกลเลอรีที่มีโครงสร้าง `design.md` เพื่อนำทาง AI ไปสู่การออกแบบ UI ที่ได้มาตรฐาน

## พื้นหลัง

ใครก็ตามที่เคยใช้ AI ในการเขียนโค้ดส่วนหน้าอาจมีประสบการณ์คล้ายกัน: ขอให้ AI สร้างหน้าเดียวกันหลายครั้ง และผลลัพธ์แต่ละรายการจะออกมาในรูปแบบที่แตกต่างกัน บางทีก็โค้งมน บางทีก็คม บางครั้งระยะห่างคือ 8px และบางครั้งก็กลายเป็น 16px แม้แต่ปุ่มเดียวกันก็อาจดูแตกต่างไปตามการสนทนาต่างๆ

นี่ไม่ใช่ปัญหาที่แยกจากกัน เนื่องจากการพัฒนาที่ได้รับความช่วยเหลือจาก AI กลายเป็นเรื่องปกติมากขึ้น **การขาดความสอดคล้องใน UI ส่วนหน้าที่สร้างโดย AI** จึงกลายเป็นปัญหาที่แพร่หลาย ผู้ช่วย AI ที่แตกต่างกัน การแจ้งที่แตกต่างกัน และแม้แต่ผู้ช่วยคนเดียวกันในการสนทนาที่ต่างกัน ก็สามารถสร้างการออกแบบอินเทอร์เฟซที่แตกต่างกันอย่างมาก ซึ่งทำให้เกิดค่าบำรุงรักษาจำนวนมากในระหว่างการทำซ้ำผลิตภัณฑ์

สาเหตุที่แท้จริงนั้นง่ายมาก: ไม่มี **เอกสารอ้างอิงการออกแบบที่เชื่อถือได้** สไตล์ชีต CSS แบบดั้งเดิมสามารถบอกนักพัฒนาว่า "จะใช้งาน" บางอย่างได้อย่างไร แต่พวกเขาไม่สามารถสื่อสารได้ทั้งหมดว่า "เหตุใดจึงออกแบบด้วยวิธีนี้" หรือ "ควรใช้รูปแบบการออกแบบใดในสถานการณ์ใด" สำหรับ AI คำอธิบายที่ชัดเจนและมีโครงสร้างมีความสำคัญมากยิ่งขึ้นสำหรับการทำความเข้าใจแบบแผนการออกแบบ

ในขณะเดียวกัน ชุมชนโอเพ่นซอร์สก็มีทรัพยากรที่ยอดเยี่ยมอยู่แล้ว ที่ [VoltAgent/awesome-design-md](https://github.com/VoltAgent/awesome-design-md) โครงการรวบรวมเอกสารระบบการออกแบบจากบริษัทที่มีชื่อเสียงมากมาย แต่ละไดเร็กทอรีประกอบด้วย `README.md`, ก `DESIGN.md`และดูตัวอย่าง HTML อย่างไรก็ตาม ข้อมูลทั้งหมดนี้กระจัดกระจายไปทั่วพื้นที่เก็บข้อมูลต้นน้ำ ทำให้ยากต่อการเรียกดูและเปรียบเทียบอย่างรวดเร็ว

ดังนั้น เราสามารถรวมทรัพยากรเหล่านั้นไว้ในแกลเลอรีการออกแบบที่เรียกดูได้ง่าย ขณะเดียวกันก็กลั่นกรองโครงสร้างที่มีโครงสร้าง `design.md` ให้ AI ใช้?

คำตอบคือใช่ ต่อไปให้ฉันอธิบายแนวทางของเรา

## การวิเคราะห์

ก่อนที่จะเขียนโค้ด ให้เราแจกแจงความท้าทายทางเทคนิคที่อยู่เบื้องหลังปัญหานี้ก่อน

### การจัดการเนื้อหาที่มา: คุณจะรวมทรัพยากรการออกแบบที่กระจัดกระจายได้อย่างไร

พื้นที่เก็บข้อมูล Awesome-design-md อัพสตรีมประกอบด้วยเอกสารการออกแบบจำนวนมาก แต่เราต้องการวิธีที่จะนำเอกสารเหล่านั้นมาสู่โปรเจ็กต์ของเราเอง

**วิธีแก้ปัญหา: ใช้ git submodule**

```
awesome-design-md-site
└── vendor/awesome-design-md          # Upstream resources (git submodule)
```

สิ่งนี้ทำให้เราได้รับประโยชน์หลายประการ:
- การควบคุมเวอร์ชัน: เราสามารถปักหมุดเวอร์ชันอัปสตรีมเฉพาะได้
- บิลด์ออฟไลน์: ไม่จำเป็นต้องขอ API ภายนอกระหว่างบิลด์
- การตรวจสอบเนื้อหา: การเปลี่ยนแปลงเฉพาะสามารถมองเห็นได้ใน PR

### การทำให้ข้อมูลเป็นมาตรฐาน: คุณสร้างมาตรฐานให้กับโครงสร้างเอกสารที่แตกต่างกันอย่างไร

บริษัทต่างๆ มีโครงสร้างเอกสารการออกแบบที่แตกต่างกัน ไฟล์ตัวอย่างบางไฟล์หายไป และบางไฟล์ใช้การตั้งชื่อที่ไม่สอดคล้องกัน เราจำเป็นต้องทำให้พวกมันเป็นมาตรฐานในระหว่างกระบวนการสร้าง

**วิธีแก้ปัญหา: สแกนและสร้างรายการที่ทำให้เป็นมาตรฐาน ณ เวลาสร้าง**

### สถาปัตยกรรมไซต์แบบคงที่: คุณจะให้การค้นหาแบบไดนามิกในขณะที่คงที่ได้อย่างไร?

### เอกสารการออกแบบ: คุณช่วยให้ AI เข้าใจและปฏิบัติตามมาตรฐานการออกแบบได้อย่างไร

นี่คือแกนหลักของโซลูชันทั้งหมด เราจำเป็นต้องสร้างโครงสร้าง `design.md` ที่ AI สามารถเข้าใจและนำไปใช้ได้

**วิธีแก้ปัญหา: ยืมโครงสร้างของ ClickHouse `DESIGN.md`**

[ของคลิกเฮาส์ `DESIGN.md`](https://github.com/ClickHouse/ClickHouse/blob/master/DESIGN.md) เป็นข้อมูลอ้างอิงที่ดีเยี่ยม ประกอบด้วย:
- ธีมภาพและบรรยากาศ
- จานสีและบทบาท
- กฎการพิมพ์
- การจัดรูปแบบส่วนประกอบ
- หลักการจัดวาง
- ความลึกและระดับความสูง
- สิ่งที่ควรทำและไม่ควรทำ
- พฤติกรรมตอบสนอง
- คู่มือพร้อมท์ตัวแทน

แนวทางของเราคือ: **นำโครงสร้างกลับมาใช้ใหม่ เขียนเนื้อหาใหม่** เราคงโครงสร้างส่วนของ ClickHouse ไว้ `DESIGN.md`แต่แทนที่เนื้อหาด้วยโทเค็นการออกแบบจริงและแบบแผนส่วนประกอบที่ใช้ในโครงการของเราเอง

## โซลูชั่น

จากการวิเคราะห์ข้างต้น โซลูชันของเราประกอบด้วยโมดูลหลักสี่โมดูล

### 1. ไปป์ไลน์การนำเข้าเนื้อหา

นี่คือรากฐานของทั้งระบบ ซึ่งรับผิดชอบในการแยกและทำให้เนื้อหาเป็นมาตรฐานจากทรัพยากรต้นน้ำ

```typescript
// src/lib/content/awesomeDesignCatalog.ts

export async function scanSourceEntries(): Promise<DesignEntry[]> {
  const designDir = 'vendor/awesome-design-md/design-md';
  const entries: DesignEntry[] = [];

  for (const dir of await fs.readdir(designDir)) {
    const entryPath = path.join(designDir, dir);
    if (await isValidDesignEntry(entryPath)) {
      const entry = await normalizeDesignEntry(entryPath);
      entries.push(entry);
    }
  }

  return entries;
}

async function isValidDesignEntry(dir: string): Promise<boolean> {
  const requiredFiles = ['README.md', 'DESIGN.md'];
  for (const file of requiredFiles) {
    if (!(await fileExists(path.join(dir, file)))) {
      return false;
    }
  }
  return true;
}
```

### 2. อินเทอร์เฟซการเรียกดูแกลเลอรี

อินเทอร์เฟซแกลเลอรีประกอบด้วยสามส่วนหลัก:

**หน้าแรก**: แสดงตารางการ์ดของรายการการออกแบบทั้งหมด และการ์ดแต่ละใบประกอบด้วย:
- ชื่อรายการออกแบบและบทสรุป
- ดูตัวอย่างรูปภาพ หากมี
- การเน้นการค้นหาอย่างรวดเร็ว

**หน้ารายละเอียด**: รวบรวมข้อมูลทั้งหมดสำหรับรายการการออกแบบรายการเดียว:
- `README` เอกสาร
- `DESIGN` เอกสาร
- ดูตัวอย่างด้วยการสลับธีมสว่าง/มืด
- การนำทางไปยังรายการที่อยู่ติดกัน

**การนำทาง**: รองรับการกลับไปยังแกลเลอรีและเรียกดูรายการที่อยู่ติดกัน

แกลเลอรีหน้าแรกใช้เลย์เอาต์การ์ดที่มีความหนาแน่นสูงและแบนราบ `design.md` รายการจากแหล่งต่างๆ เข้าสู่กรอบภาพที่เป็นหนึ่งเดียว เพื่อให้ทีมสามารถเปรียบเทียบสไตล์ของแบรนด์ รูปแบบปุ่ม และจังหวะการพิมพ์ได้อย่างรวดเร็ว

หลังจากเปิดรายการเฉพาะ หน้ารายละเอียดจะวางสรุปการออกแบบและการแสดงตัวอย่างสดในหน้าเดียวกัน ซึ่งช่วยลดค่าใช้จ่ายในการสลับไปมาระหว่างเอกสารประกอบ การแสดงตัวอย่าง และซอร์สโค้ด

### 3. ฟังก์ชั่นการค้นหา

คุณลักษณะการค้นหาจะขึ้นอยู่กับการกรองฝั่งไคลเอ็นต์ โดยมีสถานะที่คงไว้ผ่านพารามิเตอร์การสืบค้น URL:

```typescript
// src/components/gallery/SearchToolbar.tsx

function SearchToolbar({ entries }: { entries: DesignEntry[] }) {
  const [query, setQuery] = useState('');
  const [results, setResults] = useState(entries);

  useEffect(() => {
    const params = new URLSearchParams(window.location.search);
    const initialQuery = params.get('q') || '';
    setQuery(initialQuery);
    filterEntries(initialQuery);
  }, []);

  const filterEntries = (searchQuery: string) => {
    const filtered = entries.filter(entry =>
      entry.searchText.toLowerCase().includes(searchQuery.toLowerCase())
    );
    setResults(filtered);
  };

  const handleChange = (e: React.ChangeEvent<HTMLInputElement>) => {
    const value = e.target.value;
    setQuery(value);
    filterEntries(value);

    // Update the URL without triggering a page refresh
    const newUrl = value
      ? `${window.location.pathname}?q=${encodeURIComponent(value)}`
      : window.location.pathname;
    window.history.replaceState({}, '', newUrl);
  };

  return (
    <div className="search-toolbar">
      <input
        type="text"
        value={query}
        onChange={handleChange}
        placeholder="Search design entries..."
      />
      <span className="result-count">{results.length} results</span>
    </div>
  );
}
```

### 4. เอกสารอ้างอิงการออกแบบ (`design.md`)

นี่คือผลลัพธ์หลักของโซลูชันทั้งหมด เราสร้างก `design.md` ในรูทโปรเจ็กต์โดยมีโครงสร้างดังต่อไปนี้:

นอกจากจะดิบแล้ว `design.md` เนื้อหาที่ AI บริโภค เรายังวางทั้ง `README` และ `DESIGN` เอกสารลงในอินเทอร์เฟซการอ่านเดียวกัน ทำให้ผู้คนสามารถตรวจทาน คัดลอกตัวอย่าง และเปรียบเทียบกับผลลัพธ์การแสดงตัวอย่างได้ง่ายขึ้น

![Awesome Design MD Gallery README และหน้าเอกสาร DESIGN](/article-images/design-md-consistent-ui/1775613231000-2d386de3f80f.png)

```markdown
# Design Reference for [Project Name]

## 1. Visual Theme & Atmosphere
- Overall style description
- Design philosophy and principles

## 2. Color Palette & Roles
- Primary and supporting colors
- Semantic colors (`success`, `warning`, `error`)
- CSS variable definitions

## 3. Typography Rules
- Font families
- Type scale (`h1-h6`, `body`, `small`)
- Line height and font weight

## 4. Component Stylings
- Button style conventions
- Form component styles
- Card and container styles

## 5. Layout Principles
- Spacing system
- Grid and breakpoints
- Alignment principles

## 6. Depth & Elevation
- Shadow levels
- `z-index` conventions

## 7. Do's and Don'ts
- Common mistakes and correct approaches

## 8. Responsive Behavior
- Breakpoint definitions
- Responsive adaptation rules

## 9. Agent Prompt Guide
- How to use this document in AI prompts
- Example prompt templates
```

## การปฏิบัติ

หลังจากที่เราได้กล่าวถึงโซลูชันแล้ว คุณจะนำไปใช้จริงได้อย่างไร

### ขั้นตอนการดำเนินการ

**ขั้นตอนที่ 1: เริ่มต้นโมดูลย่อย**

```bash
# Add the upstream repository as a submodule
git submodule add https://github.com/VoltAgent/awesome-design-md.git vendor/awesome-design-md

# Initialize and update the submodule
git submodule update --init --recursive
```

**ขั้นตอนที่ 2: สร้างไปป์ไลน์เนื้อหา**

ดำเนินการ `awesomeDesignCatalog.ts`รวมถึง:
- การสแกนไฟล์และตรรกะการตรวจสอบ
- การเรนเดอร์ Markdown โดยใช้ตัวเรนเดอร์ในตัวของ Astro
- การดึงข้อมูลเข้า

**ขั้นตอนที่ 3: สร้าง UI แกลเลอรี**

ใช้ Astro + React Islands เพื่อสร้าง:
- เค้าโครงแกลเลอรีหน้าแรก (ตารางการ์ด)
- การออกแบบส่วนประกอบของการ์ด
- แถบเครื่องมือค้นหา
- เค้าโครงหน้ารายละเอียด

**ขั้นตอนที่ 4: เขียนเอกสารการออกแบบ**

ขึ้นอยู่กับโครงสร้างของ ClickHouse `DESIGN.md`กรอกโทเค็นการออกแบบจริงจากโปรเจ็กต์ของคุณเอง อัปเดต `README.md` และเพิ่มลิงค์ไปที่ `design.md`.

### หมายเหตุ

**ความปลอดภัย**: การแสดงผล Markdown จำเป็นต้องกรอง HTML ที่ไม่ปลอดภัย ฟิลเตอร์เรนเดอร์ในตัวของ Astro `script` ตามค่าเริ่มต้น แต่คุณยังต้องระวังความเสี่ยง XSS

**ประสิทธิภาพ**: การแสดงตัวอย่าง iframe จำนวนมากอาจส่งผลต่อประสิทธิภาพการทำงานของสีแรก ขอแนะนำให้ใช้ `loading="lazy"` เพื่อโหลดเนื้อหาตัวอย่างแบบ Lazy Load

**การบำรุงรักษา**: `design.md` จำเป็นต้องซิงค์กับการติดตั้งโค้ด ขอแนะนำให้เพิ่มการตรวจสอบ CI เพื่อให้แน่ใจว่าตัวแปร CSS ยังคงสอดคล้องกันระหว่างเอกสารประกอบและโค้ด

**การเข้าถึงพิเศษ**: ตรวจสอบให้แน่ใจว่าคอนทราสต์ของสีเป็นไปตามมาตรฐาน WCAG AA (อย่างน้อย 4.5:1)

### คู่มือการใช้งาน AI

หลังจากสร้างแล้ว `design.md`ทำอย่างไรให้ AI ใช้งานได้จริง? คำแนะนำที่เป็นประโยชน์บางประการมีดังนี้:

**เคล็ดลับที่ 1: อ้างอิงอย่างชัดเจนในข้อความแจ้ง**

```
Please refer to the design.md file in the project root and use the design conventions defined there to implement the following components:
- Buttons: use the primary color with an 8px border radius
- Cards: use the elevation-2 shadow level
```

**เคล็ดลับ 2: ต้องใช้ AI เพื่ออ้างอิงตัวแปร CSS ที่เฉพาะเจาะจง**

```
Implement a navigation bar with the following requirements:
- Use --color-bg-primary for the background color
- Use --color-border-subtle for borders
- Use --text-color-primary for text
```

**เคล็ดลับ 3: รวม `design.md` เนื้อหาในระบบแจ้ง**

หากเครื่องมือ AI ของคุณรองรับข้อความแจ้งของระบบแบบกำหนดเอง คุณสามารถเพิ่มเนื้อหาหลักของ `design.md` โดยตรงกับมัน

### กลยุทธ์การทดสอบ

**การทดสอบไปป์ไลน์เนื้อหา**:
- สถานการณ์ไฟล์หายไป (missing `README.md` หรือ `DESIGN.md`)
- สถานการณ์ข้อผิดพลาดของรูปแบบ (การแยกวิเคราะห์ Markdown ล้มเหลว)
- สถานการณ์สมมติไดเรกทอรีว่างเปล่า

**การทดสอบคุณสมบัติการค้นหา**:
- การจัดการผลลัพธ์ที่ว่างเปล่า
- อักขระพิเศษ เช่น ภาษาจีนและอีโมจิ
- การตรวจสอบการซิงค์ URL

**การทดสอบองค์ประกอบ UI**:
- การสลับธีมสว่าง/มืด
- รูปแบบที่ตอบสนอง
- ดูตัวอย่างสถานะการโหลด

## บทสรุป

ความไม่สอดคล้องกันใน UI ที่สร้างโดย AI ที่ HagiCode พบระหว่างการพัฒนาเป็นแก่นแท้ เกิดจากการขาดเอกสารอ้างอิงการออกแบบที่มีโครงสร้าง โดยการสร้างไซต์แกลเลอรีการออกแบบและสร้างมาตรฐาน `design.md`เราแก้ไขปัญหานี้สำเร็จแล้ว

คุณค่าหลักของโซลูชันนี้อยู่ที่:
- **ทรัพยากรแบบครบวงจร**: รวบรวมเอกสารระบบการออกแบบที่กระจัดกระจาย
- **มาตรฐานที่มีโครงสร้าง**: แสดงแบบแผนการออกแบบในรูปแบบที่ AI เข้าใจได้
- **การบำรุงรักษาอย่างต่อเนื่อง**: ทำให้เนื้อหาอัปเดตอยู่เสมอ `git submodule`

หากคุณใช้ AI เพื่อการพัฒนาส่วนหน้าด้วย แนวทางนี้ก็คุ้มค่าที่จะลองใช้ การสร้างโครงสร้าง `design.md` ไม่เพียงปรับปรุงความสอดคล้องของโค้ดที่สร้างโดย AI เท่านั้น แต่ยังช่วยให้ทีมของคุณรักษามาตรฐานการออกแบบที่เป็นหนึ่งเดียวภายในอีกด้วย
