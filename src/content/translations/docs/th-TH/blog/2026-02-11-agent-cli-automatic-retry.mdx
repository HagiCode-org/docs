## วิธีใช้งานการลองใหม่อัตโนมัติสำหรับ Agent CLI เช่น Claude Code และ Codex

> วลี Automatic Retry ดูเหมือนสวิตช์สลับเล็กๆ แต่เมื่อคุณใส่เข้าไปในสภาพแวดล้อมทางวิศวกรรมจริงแล้ว มันก็ไม่มีอะไรเป็นเช่นนั้น สวัสดีทุกคน ฉันชื่อ Yu Kun ผู้สร้าง HagiCode วันนี้ผมไม่อยากเทรดแบบพูดเปล่าๆ ฉันต้องการพูดคุยเกี่ยวกับวิธีการลองใหม่อัตโนมัติสำหรับ Agent CLI เช่น Claude Code และ Codex จริง ๆ แล้วควรดำเนินการอย่างไร เพื่อให้สามารถกู้คืนจากข้อยกเว้นและหลีกเลี่ยงการลากระบบไปสู่การดำเนินการซ้ำอย่างไม่มีที่สิ้นสุด

## พื้นหลัง

หากคุณได้ทำงานเกี่ยวกับการเข้ารหัส AI เมื่อเร็ว ๆ นี้ คุณอาจประสบปัญหาประเภทนี้แล้ว: งานไม่ได้ล้มเหลวในทันที แต่หยุดดำเนินการไปครึ่งทาง

ในคำขอ HTTP ทั่วไป นั่นมักจะหมายถึงการส่งอีกครั้ง อาจมีแบ็คออฟแบบเอ็กซ์โปเนนเชียลบ้าง แต่ Agent CLI นั้นแตกต่างออกไป เครื่องมืออย่าง Claude Code และ Codex มักจะทำงานในลักษณะสตรีมมิ่ง โดยจะส่งออกเอาต์พุตทีละก้อน ในระหว่างกระบวนการนั้น พวกเขายังอาจผูกเข้ากับเธรด เซสชัน หรือโทเค็นดำเนินการต่อ กล่าวอีกนัยหนึ่ง คำถามไม่ใช่แค่ว่า "คำขอนี้ล้มเหลวหรือไม่" มันจะกลายเป็น:

- เนื้อหาที่ปล่อยออกมาแล้วยังนับรวมอยู่หรือไม่?
- บริบทปัจจุบันสามารถทำงานต่อไปได้หรือไม่
- ความล้มเหลวนี้ควรได้รับการกู้คืนโดยอัตโนมัติหรือไม่
- หากควรกู้คืน เราควรรอนานเท่าใดก่อนลองอีกครั้ง เราควรส่งสิ่งใดระหว่างการลองใหม่ และเราควรใช้บริบทดั้งเดิมซ้ำหรือไม่

ในครั้งแรกที่หลายทีมสร้างส่วนนี้ พวกเขาเขียนเวอร์ชันที่ไร้เดียงสาที่สุดโดยสัญชาตญาณ: หากมีข้อผิดพลาดเกิดขึ้น ให้ลองอีกครั้ง แนวคิดดังกล่าวเป็นไปตามธรรมชาติ แต่เมื่อบรรลุถึงโครงการจริง ปัญหาหนึ่งแล้วอีกอย่างหนึ่งก็เริ่มปรากฏให้เห็น

- ข้อผิดพลาดบางอย่างถือเป็นความล้มเหลวชั่วคราวอย่างชัดเจน แต่จะถือเป็นความล้มเหลวขั้นสุดท้าย
- ข้อผิดพลาดบางอย่างไม่คุ้มที่จะลองใหม่เลย แต่ระบบจะเล่นซ้ำซ้ำแล้วซ้ำเล่า
- คำขอที่มีเธรดและคำขอที่ไม่มีเธรดจะได้รับการปฏิบัติเหมือนกันทุกประการ
- กลยุทธ์การแบ็คออฟไม่มีขอบเขต และคำขอในเบื้องหลังก็มีภาระมากเกินไป

ในขณะที่บูรณาการ Agent CLI หลายรายการ HagiCode ก็ก้าวเข้าสู่กับดักเหล่านี้เช่นกัน โดยเฉพาะอย่างยิ่งในด้าน Codex ปัญหาแรกที่เราพบคือข้อความเชื่อมต่อใหม่บางประเภทไม่ได้รับการยอมรับว่าเป็นสถานะเทอร์มินัลที่ลองใหม่ได้ ดังนั้นกลไกการกู้คืนที่เราไม่เคยได้รับโอกาสที่จะมีผลเลย พูดให้ชัดเจนไม่ใช่ว่าระบบขาดการลองใหม่โดยอัตโนมัติ ระบบล้มเหลวในการรับรู้ว่าความล้มเหลวนี้คุ้มค่าที่จะลองใหม่

ดังนั้นประเด็นหลักของบทความนี้จึงชัดเจนมาก: **การลองใหม่อัตโนมัติไม่ใช่ปุ่ม แต่เป็นการออกแบบแบบเลเยอร์**

## เหตุใดการลองใหม่อัตโนมัติสำหรับ Agent CLI จึงยากกว่าการลองใหม่ตามปกติ

นี่เป็นคำถามที่ใช้งานได้จริง ดังนั้นให้เราตรงไปยังข้อสรุป: ความยากในการลองซ้ำอัตโนมัติสำหรับ Agent CLI ไม่ใช่ "ลองอีกครั้งหลังจากผ่านไปไม่กี่วินาที" แต่ "ยังสามารถดำเนินการต่อในบริบทดั้งเดิมได้หรือไม่"

คุณสามารถคิดว่ามันเป็นการสนทนาที่ยาวนาน การลอง API แบบธรรมดานั้นเหมือนกับการโทรซ้ำเมื่อสายโทรศัพท์ไม่ว่าง การลองอีกครั้งของ Agent CLI นั้นเหมือนกับสัญญาณที่ลดลงในขณะที่อีกฝ่ายยังอ่านไปได้ครึ่งประโยค จากนั้นคุณต้องตัดสินใจว่าจะโทรกลับหรือไม่ จะเริ่มใหม่เมื่อทำหรือไม่ และอีกฝ่ายจะยังจำจุดที่การสนทนาหยุดอยู่หรือไม่ นี่ไม่ใช่ปัญหาทางวิศวกรรมแบบเดียวกันเลย

กล่าวอย่างเป็นรูปธรรมมากขึ้น มีความยากลำบากโดยทั่วไปอยู่สี่ประการโดยเฉพาะ

### 1. เป็นการสตรีมมิ่ง

เมื่อเอาต์พุตถูกส่งไปยังผู้ใช้แล้ว คุณจะไม่สามารถจัดการกับความล้มเหลวแบบเดียวกับที่คุณทำกับคำขอทั่วไปได้อีกต่อไป โดยที่คุณกลืนมันลงไปอย่างเงียบๆ และลองอีกครั้งอย่างเงียบๆ นั่นเป็นเพราะว่าเนื้อหาก่อนหน้านี้ได้ถูกดูไปแล้ว หากกลยุทธ์การเล่นซ้ำไม่ถูกต้อง ส่วนหน้าสามารถแสดงข้อความที่ซ้ำกันและสถานะที่ไม่สอดคล้องกันได้อย่างง่ายดาย และวงจรการเรียกใช้เครื่องมือก็อาจพันกันได้เช่นกัน นี่ไม่ใช่อภิปรัชญา มันเป็นวิศวกรรม

### 2. โดยปกติแล้วจะเชื่อมโยงกับบริบทของเซสชัน

ผู้ให้บริการเช่น Codex เชื่อมโยงกับเธรด และการใช้งานเช่น Claude Code ก็มีเป้าหมายต่อเนื่องหรือบริบทที่เทียบเท่าต่อการทำงานต่อได้ ข้อกำหนดเบื้องต้นที่แท้จริงสำหรับการลองใหม่โดยอัตโนมัติไม่ใช่แค่ว่าข้อผิดพลาดดูเหมือนเป็นความล้มเหลวชั่วคราว แต่ยังรวมถึงผู้ให้บริการที่อนุญาตให้ดำเนินการนี้ต่อไปได้

### 3. ไม่ใช่ทุกข้อผิดพลาดที่คุ้มค่าที่จะลองใหม่

การกระวนกระวายใจของเครือข่าย การหมดเวลาที่ไม่ได้ใช้งาน SSE และความล้มเหลวของอัปสตรีมชั่วคราวมักจะคุ้มค่าที่จะลองอีกครั้ง แต่หากสิ่งที่คุณพบคือความล้มเหลวในการรับรองความถูกต้อง บริบทที่หายไป หรือผู้ให้บริการที่ไม่มีความสามารถในการดำเนินการต่อเลย การลองใหม่มักจะไม่ใช่การกู้คืน เป็นการสร้างเสียงรบกวน

### 4. ต้องมีขอบเขต

การลองซ้ำอัตโนมัติแบบไม่จำกัดมักผิดเสมอไป แนวโน้มเทคโนโลยีอาจมีเสียงดังอยู่ระยะหนึ่ง แต่กฎหมายทางวิศวกรรมมักจะคงที่เป็นเวลาหลายปี หนึ่งในนั้นคือการกู้คืนความล้มเหลวจะต้องมีขอบเขต ระบบจะต้องรู้ว่าสามารถลองใหม่ได้มากที่สุดกี่ครั้ง ควรรอนานเท่าใดในแต่ละครั้ง และเมื่อใดควรหยุดและยอมรับว่าสิ่งนี้จะไม่ฟื้นตัวจริงๆ

เนื่องจากคุณลักษณะเหล่านี้ ในที่สุด HagiCode จึงไม่ลองใช้การลองใหม่โดยอัตโนมัติเพียงไม่กี่บรรทัด `try/catch` ภายในผู้ให้บริการเฉพาะราย แต่เราแยกมันออกมาเป็นเลเยอร์ความสามารถที่ใช้ร่วมกันแทน สุดท้ายแล้วปัญหาทางวิศวกรรมก็ยังต้องได้รับการแก้ไขด้วยวิธีทางวิศวกรรม

## แนวทางของ HagiCode: ดึงการลองใหม่ออกจากผู้ให้บริการ

การใช้งานจริงในปัจจุบันของ HagiCode สามารถบีบอัดเป็นประโยคเดียว:

**เลเยอร์ที่ใช้ร่วมกันจัดการโฟลว์การลองใหม่อย่างสม่ำเสมอ และผู้ให้บริการที่เป็นรูปธรรมแต่ละรายมีหน้าที่รับผิดชอบในการตอบคำถามสองข้อเท่านั้น: สถานะเทอร์มินัลนี้คุ้มค่าที่จะลองอีกครั้ง และบริบทปัจจุบันจะยังคงดำเนินต่อไปได้หรือไม่**

สิ่งนี้ไม่ซับซ้อน แต่มีความสำคัญ เมื่อแยกความรับผิดชอบด้วยวิธีนี้ Claude Code, Codex และแม้แต่ Agent CLI อื่นๆ ก็สามารถนำโครงกระดูกเดียวกันกลับมาใช้ใหม่ได้ โมเดลจะเปลี่ยนไป เครื่องมือจะพัฒนาขึ้น ขั้นตอนการทำงานจะได้รับการอัพเกรด แต่รากฐานทางวิศวกรรมยังคงอยู่

### เลเยอร์ 1: ใช้ผู้ประสานงานแบบรวมเพื่อจัดการลูปการลองใหม่

ส่วนการใช้งานหลักในโครงการมีลักษณะดังนี้:

```csharp
internal static class ProviderErrorAutoRetryCoordinator
{
    public static async IAsyncEnumerable<CliMessage> ExecuteAsync(
        string prompt,
        ProviderErrorAutoRetrySettings? settings,
        Func<string, IAsyncEnumerable<CliMessage>> executeAttemptAsync,
        Func<bool> canRetryInSameContext,
        Func<TimeSpan, CancellationToken, Task> delayAsync,
        Func<CliMessage, bool> isRetryableTerminalMessage,
        [EnumeratorCancellation] CancellationToken cancellationToken)
    {
        var normalizedSettings = ProviderErrorAutoRetrySettings.Normalize(settings);
        var retrySchedule = normalizedSettings.Enabled
            ? normalizedSettings.GetRetrySchedule()
            : [];

        for (var attempt = 0; ; attempt++)
        {
            var attemptPrompt = attempt == 0
                ? prompt
                : ProviderErrorAutoRetrySettings.ContinuationPrompt;

            CliMessage? terminalFailure = null;

            await foreach (var message in executeAttemptAsync(attemptPrompt)
                               .WithCancellation(cancellationToken))
            {
                if (isRetryableTerminalMessage(message))
                {
                    terminalFailure = message;
                    break;
                }

                yield return message;
            }

            if (terminalFailure is null)
            {
                yield break;
            }

            if (attempt >= retrySchedule.Count || !canRetryInSameContext())
            {
                yield return terminalFailure;
                yield break;
            }

            await delayAsync(retrySchedule[attempt], cancellationToken);
        }
    }
}
```

สิ่งที่โค้ดนี้ทำนั้นตรงไปตรงมามาก แต่ก็มีประสิทธิภาพมากเช่นกัน

- อย่าผ่านความล้มเหลวระดับกลางโดยตรงในตอนแรก ผู้ประสานงานจะตัดสินใจว่ายังสามารถฟื้นตัวได้หรือไม่
- เฉพาะเมื่องบประมาณการลองใหม่หมดลงเท่านั้นที่ความล้มเหลวสุดท้ายจะกลับคืนสู่ชั้นบนสุด
- เริ่มต้นจากความพยายามครั้งที่สอง พรอมต์เดิมจะไม่ถูกส่งอีกต่อไป ข้อความแจ้งความต่อเนื่องจะถูกส่งอย่างสม่ำเสมอแทน

นั่นคือเหตุผลที่ฉันเน้นย้ำก่อนหน้านี้ว่าการลองใหม่โดยอัตโนมัติไม่ใช่แค่ "ส่งคำขออีกครั้ง" มันไม่ใช่แค่การแพตช์สาขาข้อยกเว้นเท่านั้น เป็นการจัดการวงจรชีวิตของการดำเนินการ นั่นอาจฟังดูเหมือนภาษาของผู้จัดการผลิตภัณฑ์ แต่ในแง่วิศวกรรม นั่นคือสิ่งที่เป็นอยู่

### เลเยอร์ 2: สแนปช็อตนโยบายการลองใหม่

อีกปัญหาหนึ่งที่มองข้ามได้ง่ายมากคือ ใครเป็นผู้ตัดสินใจว่าจะเปิดใช้งานการลองซ้ำอัตโนมัติสำหรับคำขอนี้หรือไม่

คำตอบของ HagiCode ไม่ได้ขึ้นอยู่กับ "การกำหนดค่าส่วนกลางในปัจจุบัน" บางอย่าง แต่ต้องเปลี่ยนนโยบายให้เป็นภาพรวมและปล่อยให้นโยบายเดินทางไปพร้อมกับคำขอนี้ ด้วยวิธีนี้ การเข้าคิวเซสชัน การคงอยู่ของข้อความ การส่งต่อการดำเนินการ และการปรับตัวของผู้ให้บริการ จะไม่สูญเสียนโยบายไปตลอดทาง การวิ่งที่ประสบความสำเร็จครั้งหนึ่งไม่ใช่ระบบ ความสำเร็จที่ยั่งยืนเป็นระบบ

โครงสร้างหลักสามารถทำให้ง่ายขึ้นดังนี้:

```csharp
public sealed record ProviderErrorAutoRetrySnapshot
{
    public const string DefaultStrategy = "default";

    public bool Enabled { get; init; }

    public string Strategy { get; init; } = DefaultStrategy;

    public static ProviderErrorAutoRetrySnapshot Normalize(bool? enabled, string? strategy)
    {
        return new ProviderErrorAutoRetrySnapshot
        {
            Enabled = enabled ?? true,
            Strategy = string.IsNullOrWhiteSpace(strategy)
                ? DefaultStrategy
                : strategy.Trim()
        };
    }
}
```

จากนั้นในด้านการดำเนินการ ข้อมูลจะถูกแมปเข้ากับออบเจ็กต์การตั้งค่าที่ผู้ให้บริการใช้จริง คุณค่าของแนวทางนี้ตรงมาก:

- ชั้นธุรกิจจะตัดสินใจว่าควรอนุญาตให้ลองใหม่หรือไม่
- รันไทม์จะตัดสินใจว่าควรดำเนินการลองใหม่อย่างไร

แต่ละฝ่ายจัดการข้อกังวลของตนเองโดยไม่ขัดแย้งกับอีกฝ่าย ปัญหามากมายไม่สามารถแก้ไขได้ ค่าใช้จ่ายของพวกเขาไม่ได้ระบุไว้อย่างชัดเจน การเปลี่ยนนโยบายให้เป็นภาพรวมถือเป็นวิธีการบัญชีต้นทุนดังกล่าวล่วงหน้า

### เลเยอร์ 3: ผู้ให้บริการตัดสินใจเฉพาะสถานะของเทอร์มินัลและความอยู่รอดของบริบทเท่านั้น

เมื่อเราเข้าถึงผู้ให้บริการ Claude Code หรือ Codex ที่เป็นรูปธรรมแล้ว ความรับผิดชอบที่นี่ก็น้อยลงจริงๆ คุณสามารถคิดว่ามันเป็นการเพิ่มประสิทธิภาพ ไม่ใช่การทดแทน

ยกตัวอย่าง Codex เมื่อเชื่อมโยงกับผู้ประสานงานที่ใช้ร่วมกัน จริงๆ แล้วจำเป็นต้องระบุสามสิ่งเท่านั้น:

```csharp
await foreach (var message in ProviderErrorAutoRetryCoordinator.ExecuteAsync(
                   prompt,
                   options.ProviderErrorAutoRetry,
                   retryPrompt => ExecuteCodexAttemptAsync(...),
                   () => !string.IsNullOrWhiteSpace(resolvedThreadId),
                   DelayAsync,
                   IsRetryableTerminalFailure,
                   cancellationToken))
{
    yield return message;
}
```

คุณจะสังเกตเห็นว่าการตัดสินใจเฉพาะของผู้ให้บริการเป็นเพียงสองสิ่งนี้เท่านั้น:

- `IsRetryableTerminalFailure`
- `canRetryInSameContext`

Codex ตรวจสอบว่าเธรดยังสามารถดำเนินการต่อได้หรือไม่ ในขณะที่ Claude Code ตรวจสอบว่าเป้าหมายการต่อเนื่องยังคงอยู่หรือไม่ นโยบาย Backoff จำนวนการลองใหม่ และข้อความแจ้งการติดตามผลไม่ควรถูกสร้างใหม่โดยผู้ให้บริการทุกรายแยกกัน

เมื่อเลเยอร์นี้ถูกแยกออก ค่าใช้จ่ายในการรวม CLI เข้ากับ HagiCode จะลดลงอย่างมาก คุณไม่จำเป็นต้องทำซ้ำเครื่องสถานะการลองใหม่ทั้งหมด คุณจะต้องเสียบเงื่อนไขขอบเขตของผู้ให้บริการรายนั้นเท่านั้น การเขียนอย่างรวดเร็วนั้นไม่เหมือนกับการเขียนอย่างแข็งแกร่ง ความสามารถในการเชื่อมต่อบางสิ่งบางอย่างไม่เหมือนกับการเชื่อมต่อได้ดี การทำให้มันรันนั้นไม่เหมือนกับการทำให้สามารถบำรุงรักษาได้ตลอดเวลา

## ข้อผิดพลาดง่ายๆ ที่ทำ: อย่าถือว่าทุกข้อผิดพลาดสามารถลองใหม่ได้

ในการวิเคราะห์นี้ ประเด็นที่ฉันต้องการเน้นมากที่สุดไม่ใช่ "วิธีการลองใหม่" แต่เป็น "วิธีหลีกเลี่ยงการลองผิดครั้ง"

จุดเริ่มต้นดั้งเดิมของปัญหาคือ Codex ล้มเหลวในการจดจำข้อความเชื่อมต่อใหม่หนึ่งข้อความ ตามสัญชาตญาณ ผู้คนจำนวนมากจะเลือกวิธีแก้ไขที่เล็กที่สุดที่เป็นไปได้: เพิ่มคำนำหน้าสตริงอีกหนึ่งรายการในรายการที่อนุญาต แนวคิดดังกล่าวไม่ได้ผิดแต่อย่างใด แต่ให้ความรู้สึกเหมือนเป็นโซลูชันในระยะสาธิตมากกว่าโซลูชันที่สามารถบำรุงรักษาได้ในระยะยาว

จากการนำ HagiCode ไปใช้ในปัจจุบัน ระบบได้ก้าวไปในทิศทางที่แข็งแกร่งยิ่งขึ้นแล้ว มันไม่ได้จ้องมองเพียงสตริงตัวอักษรเดียวอีกต่อไป แต่จะส่งมอบสถานะเทอร์มินัลที่สามารถกู้คืนได้ให้กับผู้ประสานงานที่ใช้ร่วมกันอย่างสม่ำเสมอ ประโยชน์ที่ได้ชัดเจน:

- มีโอกาสน้อยที่จะล้มเหลวโดยสิ้นเชิงเนื่องจากมีการเปลี่ยนแปลงถ้อยคำเล็กน้อยในข้อความเดียว
- ความครอบคลุมการทดสอบสามารถสร้างขึ้นรอบๆ ซองจดหมายสถานะเทอร์มินัล แทนที่จะเป็นบรรทัดข้อความแบบฮาร์ดโค้ดบรรทัดเดียว
- ตรรกะการลองใหม่มีความสอดคล้องกันมากขึ้นภายในผู้ให้บริการรายเดียวกัน

แน่นอนว่าจำเป็นต้องมีขอบเขตที่ชัดเจน การเป็นคนทั่วไปมากขึ้นไม่ได้หมายความว่าจะยอมให้มากขึ้น **หากบริบทปัจจุบันไม่สามารถดำเนินต่อไปได้ แม้ว่าข้อผิดพลาดจะดูเหมือนเป็นความล้มเหลวชั่วคราว ก็ไม่ควรเล่นซ้ำแบบสุ่มสี่สุ่มห้า**

จุดนี้เป็นสิ่งสำคัญ สิ่งที่ทำให้ผู้คนไว้วางใจระบบจริงๆ ไม่ใช่ว่าระบบใช้งานได้เป็นครั้งคราว แต่เชื่อถือได้เกือบตลอดเวลา หากผู้เชี่ยวชาญเท่านั้นที่สามารถรักษาโฟลว์ได้ ก็แสดงว่ายังอีกยาวไกลจากการนำไปใช้จริง

## บทเรียนอันมีค่าที่สุดสามบทที่ควรปฏิบัติ

ณ จุดนี้ เป็นเรื่องสมเหตุสมผลที่จะเริ่มนำการอภิปรายกลับไปสู่การปฏิบัติจริง หากคุณกำลังวางแผนที่จะสร้างความสามารถที่คล้ายกันในโปรเจ็กต์ของคุณเอง นี่คือกฎสามข้อที่ฉันแนะนำอย่างยิ่งให้ปกป้องเป็นอันดับแรก

### 1. งบประมาณการลองใหม่ต้องมีขอบเขต

จังหวะการถอยกลับเริ่มต้นปัจจุบันของ HagiCode คือ:

- 10 วินาที
- 20 วินาที
- 60 วินาที

จังหวะนี้อาจจะไม่เข้ากับทุกระบบแต่ขอบเขตก็ยังต้องคงอยู่ มิฉะนั้น การลองใหม่โดยอัตโนมัติจะหยุดการเป็นกลไกการกู้คืนอย่างรวดเร็วและเปลี่ยนเป็นตัวขยายเหตุการณ์ อย่ารีบเร่งที่จะตั้งชื่อให้น่าประทับใจ ขั้นแรกตรวจสอบให้แน่ใจว่าสิ่งนี้สามารถอยู่รอดได้สองครั้งภายในทีมจริง

### 2. ข้อความแจ้งความต่อเนื่องควรรวมเป็นหนึ่งเดียว

โปรเจ็กต์ใช้พรอมต์การต่อเนื่องแบบตายตัว เพื่อให้ความพยายามในภายหลังเป็นไปตามเส้นทางของการดำเนินต่อไปในบริบทปัจจุบันอย่างชัดเจน แทนที่จะเริ่มคำขอที่สมบูรณ์แบบใหม่ ความสามารถนี้ไม่ฉูดฉาด แต่เมื่อคุณสร้างโครงการจริง คุณไม่สามารถทำได้หากไม่มีมัน หลายสิ่งที่ดูเหมือนเวทมนตร์นั้น เมื่อแยกออกจากกัน เป็นเพียงกระบวนการทางวิศวกรรมที่ได้รับการขัดเกลาเท่านั้น

### 3. ทั้งไลบรารีที่ใช้ร่วมกันและเลเยอร์อะแดปเตอร์จำเป็นต้องมีการทดสอบแบบมิเรอร์

ฉันอยากจะพูดเพิ่มเติมเล็กน้อยเกี่ยวกับประเด็นนี้เป็นพิเศษ หลายทีมจะเขียนการทดสอบหนึ่งชั้นในรันไทม์ที่ใช้ร่วมกัน และคิดว่านั่นน่าจะเพียงพอแล้ว มันไม่ใช่.

เหตุผลที่ฉันรู้สึกค่อนข้างมั่นใจเกี่ยวกับการใช้งาน HagiCode ก็คือทั้งสองเลเยอร์มีความครอบคลุมการทดสอบ:

- ผู้ให้บริการที่ใช้ร่วมกันจะทดสอบว่ามีความต่อเนื่องโดยอัตโนมัติเกิดขึ้นจริงหรือไม่
- ชั้นอะแดปเตอร์จะทดสอบว่าข้อผิดพลาดสุดท้ายและข้อความสตรีมมิ่งได้รับการเก็บรักษาไว้อย่างถูกต้องหรือไม่

ครั้งนี้ ฉันยังจัดกลุ่มการทดสอบที่เกี่ยวข้องกันสองกลุ่มอีกครั้ง และกรณีทดสอบทั้งหมด 31 กรณีผ่านทั้งสองกลุ่ม ผลลัพธ์นั้นเพียงอย่างเดียวไม่ได้พิสูจน์ว่าการออกแบบนั้นสมบูรณ์แบบ แต่พิสูจน์ได้อย่างน้อยหนึ่งสิ่ง: การลองใหม่อัตโนมัติในปัจจุบันไม่ใช่การออกแบบกระดาษ เป็นความสามารถที่ถูกจำกัดโดยทั้งโค้ดและการทดสอบ คุยกันถูกครับ แสดงรหัสให้ฉันดู มันลงตัวพอดีที่นี่

## สรุป

หากต้องบีบอัดบทความทั้งหมดเป็นประโยคเดียว จะเป็นดังนี้:

**สำหรับ Agent CLI เช่น Claude Code และ Codex ไม่ควรนำการลองใหม่อัตโนมัติมาใช้เป็นกลอุบายเฉพาะที่ซ่อนอยู่ภายในผู้ให้บริการรายเดียว ควรสร้างขึ้นโดยผสมผสานระหว่างผู้ประสานงานที่ใช้ร่วมกัน สแน็ปช็อตนโยบาย การตรวจสอบความอยู่รอดของบริบท และการทดสอบแบบมิเรอร์**

ประโยชน์ของการทำเช่นนี้มีประโยชน์มาก:

- ตรรกะถูกเขียนเพียงครั้งเดียวและนำมาใช้ซ้ำกับผู้ให้บริการหลายราย
- ไม่ว่าคำขอจะได้รับอนุญาตให้ลองอีกครั้งก็สามารถเดินทางได้อย่างเสถียรด้วยห่วงโซ่การดำเนินการหรือไม่
- ทำงานต่อไปเมื่อมีบริบท และหยุดทันเวลาเมื่อไม่มีบริบท
- สิ่งที่ส่วนหน้าเห็นในท้ายที่สุดคือสถานะเสร็จสมบูรณ์หรือสถานะล้มเหลวอย่างเสถียร ไม่ใช่กองสัญญาณรบกวนกลางที่ถูกละทิ้ง

โซลูชันนี้ได้รับการขัดเกลาทีละน้อยในขณะที่ HagiCode กำลังผสานรวม Agent CLI หลายตัวในสถานการณ์จริง ใครบอกว่าการเขียนโปรแกรมโดยใช้ AI ไม่ใช่ยุคใหม่ของการเขียนโปรแกรมคู่ โมเดลช่วยให้คุณเริ่มต้น กรอกโค้ด และแตกแขนงออกไป แต่สิ่งที่มักกำหนดขอบเขตสูงสุดของประสบการณ์ยังคงเป็นบริบท กระบวนการ และข้อจำกัด
