## คู่มือปฏิบัติเพื่อรวม CodeBuddy CLI เข้ากับแบ็กเอนด์ C#

> บทความนี้อธิบายแนวทางที่สมบูรณ์ในการรวม CodeBuddy CLI เข้ากับโปรเจ็กต์แบ็กเอนด์ C# เพื่อให้คุณสามารถมอบความสามารถผู้ช่วยเขียนโค้ด AI ได้ตั้งแต่ต้นจนจบ

:::note[บริบทของเวลา]
บทความนี้จะกล่าวถึงการออกแบบการรวม CodeBuddy และโครงสร้างผู้ให้บริการตามที่มีอยู่ ณ เวลาที่เผยแพร่ สำหรับขอบเขตที่ได้รับการสนับสนุนอย่างถาวรในปัจจุบัน โปรดดูที่ [การติดตั้ง AI Agent CLI](/th-TH/related-software-installation/ai-agent-cli/) และ [คู่มือการติดตั้ง](/th-TH/installation); `IFlowCli` ขณะนี้จะคงไว้เป็นค่าความเข้ากันได้ในอดีตเท่านั้น และไม่ได้เป็นส่วนหนึ่งของขอบเขตการสนับสนุนที่ใช้งานอยู่ในปัจจุบัน
:::

## พื้นหลัง

ในการพัฒนาผู้ช่วยเขียนโค้ด AI สมัยใหม่ ผู้ให้บริการ AI รายเดียวมักไม่สามารถตอบสนองสถานการณ์การพัฒนาที่ซับซ้อนและเปลี่ยนแปลงได้ HagiCode ในฐานะผู้ช่วยเขียนโค้ด AI แบบมัลติฟังก์ชั่น จำเป็นต้องสนับสนุนผู้ให้บริการ AI หลายรายเพื่อมอบประสบการณ์ผู้ใช้ที่ดียิ่งขึ้น ผู้ใช้ควรมีอิสระในการเลือกมากพอ ในช่วงต้นปี 2026 โปรเจ็กต์เผชิญกับการตัดสินใจครั้งสำคัญ: วิธีคืนค่าความสามารถในการรวม CodeBuddy ACP (Agent Communication Protocol) ในแบ็กเอนด์ C#

โปรเจ็กต์เคยใช้การผสานรวม CodeBuddy มาก่อน แต่โค้ดที่เกี่ยวข้องถูกลบออกในระหว่างการรีแฟคเตอร์ ไม่มีอะไรจะบ่นมากนัก ในระหว่างการพัฒนาซ้ำ บางสิ่งมักจะถูกทิ้งไว้ข้างหลังเสมอ เป้าหมายของโซลูชันทางเทคนิคนี้คือการฟื้นฟูความสามารถดังกล่าวอย่างเต็มที่และปรับปรุงสถาปัตยกรรมเพื่อให้มีความแข็งแกร่งและบำรุงรักษาได้มากขึ้น

หากคุณกำลังพิจารณาที่จะเชื่อมต่อผู้ช่วยเขียนโค้ด AI หลายคนเข้ากับโปรเจ็กต์ของคุณเอง วิธีการด้านล่างนี้อาจให้แนวคิดบางอย่างแก่คุณได้ มันสะท้อนให้เห็นถึงบทเรียนที่เราสรุปหลังจากก้าวเข้าสู่หลุมพรางมากมาย และอาจช่วยให้คุณหลีกเลี่ยงการออกนอกเส้นทางได้

## เกี่ยวกับ ฮากิโค้ด

แนวทางที่แบ่งปันในบทความนี้มาจากประสบการณ์จริงของเราใน [ฮากิโค้ด](https://hagicode.com) โครงการ HagiCode เป็นโปรเจ็กต์ผู้ช่วยเขียนโค้ด AI แบบโอเพ่นซอร์สที่รองรับผู้ให้บริการ AI หลายรายและการทำงานข้ามแพลตฟอร์ม เพื่อตอบสนองความต้องการของผู้ใช้ที่แตกต่างกัน เราจำเป็นต้องสลับไปมาระหว่างผู้ช่วยเขียนโค้ด AI ต่างๆ อย่างยืดหยุ่น ซึ่งเป็นเหตุผลว่าทำไมเราจึงสร้างการบูรณาการ CodeBuddy ที่อธิบายไว้ที่นี่

HagiCode ใช้การออกแบบโมดูลาร์ โดยมี AI Providers นำมาใช้เป็นส่วนประกอบที่เสียบได้ สถาปัตยกรรมนี้ช่วยให้เราเพิ่มการรองรับ AI ใหม่ได้อย่างง่ายดายโดยไม่ส่งผลกระทบต่อคุณสมบัติที่มีอยู่ เมื่อการออกแบบทำได้ดีในส่วนหน้า จะช่วยประหยัดปัญหาได้มากในภายหลัง หากคุณสนใจสถาปัตยกรรมทางเทคนิคของเรา คุณสามารถดูซอร์สโค้ดแบบเต็มได้ที่ GitHub

## การออกแบบสถาปัตยกรรม

### ภาพรวมสถาปัตยกรรมแบบเลเยอร์

การบูรณาการระหว่าง C# และ CodeBuddy ใช้สถาปัตยกรรมแบบเลเยอร์ที่ชัดเจน การออกแบบนี้ทำให้ความรับผิดชอบชัดเจนและทำให้การบำรุงรักษาระยะยาวง่ายขึ้นมาก:

```
┌─────────────────────────────────────────────┐
│           Provider Contract Layer           │
│   AIProviderType enum + extension methods   │
├─────────────────────────────────────────────┤
│           Provider Factory Layer            │
│   AIProviderFactory dependency injection factory │
├─────────────────────────────────────────────┤
│           Provider Implementation Layer     │
│   CodebuddyCliProvider concrete implementation │
├─────────────────────────────────────────────┤
│           ACP Infrastructure Layer          │
│  ACPSessionManager / StdioAcpTransport      │
│  AcpRpcClient / AcpAgentClient              │
└─────────────────────────────────────────────┘
```

การเคลือบหลายชั้นนี้มีประโยชน์อย่างไร? พูดง่ายๆ ก็คือแต่ละชั้นจะไม่เกะกะกัน หากภายหลังเราต้องการเปลี่ยนกลไกการสื่อสาร เช่น จาก `stdio` ไปยัง WebSocket เราเพียงแต่ต้องแก้ไขชั้นล่างสุด และตรรกะทางธุรกิจที่อยู่ด้านบนนั้นยังคงไม่มีใครแตะต้อง ไม่มีใครอยากให้การเปลี่ยนแปลงการสื่อสารกระเพื่อมไปทั่วทั้งโค้ดเบส

### การแยกส่วนประกอบหลัก

**ชั้นสัญญาของผู้ให้บริการ** เป็นรากฐานของสถาปัตยกรรมทั้งหมด เรากำหนด `AIProviderType` แจงนับ, ที่ไหน `CodebuddyCli = 3` ใช้เป็นค่าแจงนับ และใช้การแมปแบบสองทิศทางระหว่างสตริงและแจงนับผ่านวิธีการขยาย ซึ่งช่วยให้สามารถแปลงสตริงในไฟล์คอนฟิกูเรชันเป็น enums ได้อย่างสะดวก และ enum สามารถแปลงกลับเป็นสตริงสำหรับการดีบักเอาต์พุต

**เลเยอร์โรงงานของผู้ให้บริการ** มีหน้าที่สร้างอินสแตนซ์ของผู้ให้บริการที่เกี่ยวข้องตามการกำหนดค่า มันใช้การฉีดการพึ่งพา. NET ร่วมกับ `ActivatorUtilities.CreateInstance` เพื่อการสร้างสรรค์แบบไดนามิก ข้อดีของรูปแบบโรงงานคือเมื่อเพิ่มผู้ให้บริการรายใหม่ คุณจะต้องเพิ่มตรรกะการสร้างแทนการแก้ไขโค้ดที่มีอยู่เท่านั้น

**เลเยอร์การใช้งานผู้ให้บริการ** คือจุดที่งานเกิดขึ้นจริง `CodebuddyCliProvider` ดำเนินการ `IAIProvider` อินเทอร์เฟซและมีโหมดการเรียกใช้สองโหมด: `ExecuteAsync` สำหรับการโทรที่ไม่สตรีมมิ่งและ `StreamAsync` สำหรับการสตรีมการโทร

**เลเยอร์โครงสร้างพื้นฐาน ACP** มอบรากฐานการสื่อสารภายใต้ เลเยอร์นี้จัดการรายละเอียดโปรโตคอลทั้งหมด รวมถึงการจัดการกระบวนการ การทำให้เป็นอนุกรมของข้อความ และการแยกวิเคราะห์การตอบสนอง เป็นรากฐานที่ทำให้ทุกสิ่งที่อยู่เบื้องบนมั่นคง

## กลไกการสื่อสาร

### โหมดการขนส่ง Stdio

CodeBuddy ใช้ **Stdio (อินพุต/เอาต์พุตมาตรฐาน)** เพื่อสื่อสารกับกระบวนการภายนอก คำสั่งเริ่มต้นนั้นง่าย:

```bash
codebuddy --acp
```

หลังจากนั้น ข้อความ JSON-RPC จะถูกแลกเปลี่ยนผ่านอินพุตและเอาต์พุตมาตรฐาน วิธีนี้มีข้อดีหลายประการ:

1. **เริ่มต้นอย่างรวดเร็ว**: การสื่อสารกระบวนการภายในเครื่องช่วยหลีกเลี่ยงเวลาแฝงของเครือข่าย
2. **การกำหนดค่าอย่างง่าย**: คุณเพียงแค่ต้องระบุเส้นทางปฏิบัติการเท่านั้น
3. **การแยกสภาพแวดล้อม**: แต่ละเซสชันทำงานในกระบวนการที่เป็นอิสระ ดังนั้นจึงไม่มีผลกระทบต่อกัน

รองรับการฉีดตัวแปรสภาพแวดล้อมระหว่างการสื่อสาร ตัวอย่างทั่วไป ได้แก่:

- `CODEBUDDY_API_KEY`: การรับรองความถูกต้องของคีย์ API
- `CODEBUDDY_INTERNET_ENVIRONMENT`: การกำหนดค่าสภาพแวดล้อมเครือข่าย

เช่นเดียวกับการสื่อสารระหว่างบุคคลก็ควรเลือกช่องทางที่สะดวกก่อน

### โปรโตคอลข้อความ

ACP ขึ้นอยู่กับ JSON-RPC 2.0 รูปแบบข้อความจะมีลักษณะประมาณนี้:

```json
// Request message
{
  "jsonrpc": "2.0",
  "id": 1,
  "method": "agent/prompt",
  "params": {
    "prompt": "Help me write a sorting algorithm",
    "sessionId": "session-123"
  }
}

// Response message
{
  "jsonrpc": "2.0",
  "id": 1,
  "result": {
    "content": "Here is the AI response..."
  }
}
```

ในการใช้งานจริง เราจะสรุปรายละเอียดโปรโตคอลทั้งหมดเหล่านี้ไว้ ดังนั้นชั้นธุรกิจระดับบนจะต้องใส่ใจกับความรวดเร็วและการตอบสนองเท่านั้น

## การดำเนินงานหลัก

### 1. เรียกคืนสัญญาของผู้ให้บริการ

ขั้นแรก ให้คืนค่าประเภท CodeBuddy ในไฟล์ enum:

```csharp
// PCode.Models/AIProviderType.cs
public enum AIProviderType
{
    ClaudeCodeCli = 0,
    CodexCli = 1,
    GitHubCopilot = 2,
    CodebuddyCli = 3,  // Restore this enum value
    OpenCodeCli = 4,
    IFlowCli = 5,
}
```

จากนั้นเพิ่มการแมปสตริงในวิธีการขยายเพื่อให้ไฟล์การกำหนดค่าสามารถระบุผู้ให้บริการด้วยสตริง:

```csharp
// AIProviderTypeExtensions.cs
private static readonly Dictionary<string, AIProviderType> _typeMap = new(
    StringComparer.OrdinalIgnoreCase)
{
    ["CodebuddyCli"] = AIProviderType.CodebuddyCli,
    ["Codebuddy"] = AIProviderType.CodebuddyCli,
    ["codebuddy"] = AIProviderType.CodebuddyCli,
    // ... Mappings for other providers
};
```

### 2. บูรณาการโรงงานผู้ให้บริการ

เพิ่มสาขาการสร้าง CodeBuddy ในคลาสโรงงาน:

```csharp
// AIProviderFactory.cs
private IAIProvider? CreateProvider(AIProviderType providerType, ProviderConfiguration config)
{
    return providerType switch
    {
        AIProviderType.CodebuddyCli =>
            ActivatorUtilities.CreateInstance<CodebuddyCliProvider>(
                _serviceProvider,
                Options.Create(config)),
        // ... Other providers
        _ => throw new NotSupportedException($"Provider {providerType} not supported")
    };
}
```

สิ่งนี้ใช้การฉีดการพึ่งพาผ่าน `ActivatorUtilities`ซึ่งจัดการการแทรกพารามิเตอร์คอนสตรัคเตอร์โดยอัตโนมัติและสะดวกมาก

### 3. การดำเนินการของผู้ให้บริการให้เสร็จสมบูรณ์

ด้านล่างนี้คือการดำเนินการหลักของ `CodebuddyCliProvider`ครอบคลุมทั้งโหมดการเรียกใช้แบบสตรีมมิ่งและแบบไม่สตรีม:

```csharp
public class CodebuddyCliProvider : IAIProvider
{
    private readonly ILogger<CodebuddyCliProvider> _logger;
    private readonly IACPSessionManager _sessionManager;
    private readonly ProviderConfiguration _config;

    public string Name => "CodebuddyCli";
    public bool SupportsStreaming => true;
    public ProviderCapabilities Capabilities { get; }

    public CodebuddyCliProvider(
        ILogger<CodebuddyCliProvider> logger,
        IACPSessionManager sessionManager,
        IOptions<ProviderConfiguration> config)
    {
        _logger = logger;
        _sessionManager = sessionManager;
        _config = config.Value;

        // Define the capabilities of the current Provider
        Capabilities = new ProviderCapabilities
        {
            SupportsStreaming = true,
            SupportsTools = true,
            SupportsSystemMessages = true,
            SupportsArtifacts = false,
            MaxTokens = 8192
        };
    }

    // Non-streaming call: return all results together after completion
    public async Task<AIResponse> ExecuteAsync(
        AIRequest request,
        CancellationToken cancellationToken = default)
    {
        // Create an independent session for the request
        var session = await _sessionManager.CreateSessionAsync(
            "CodebuddyCli",
            request.WorkingDirectory,
            cancellationToken,
            request.SessionId);

        try
        {
            var fullPrompt = BuildPrompt(request);
            await session.SendPromptAsync(fullPrompt, cancellationToken);

            var responseBuilder = new StringBuilder();
            var toolCalls = new List<AIToolCall>();

            // Collect all response chunks
            await foreach (var chunk in StreamFromSession(session, cancellationToken))
            {
                if (!string.IsNullOrEmpty(chunk.Content))
                {
                    responseBuilder.Append(chunk.Content);
                }
                // Handle tool calls...
            }

            return new AIResponse
            {
                Content = AIResultContentSanitizer.SanitizeResultContent(
                    responseBuilder.ToString()),
                ToolCalls = toolCalls,
                Provider = Name,
                Model = string.Empty
            };
        }
        finally
        {
            // Release session resources
            await session.DisposeAsync();
        }
    }

    // Streaming call: return response chunks in real time
    public async IAsyncEnumerable<AIStreamingChunk> StreamAsync(
        AIRequest request,
        [EnumeratorCancellation] CancellationToken cancellationToken = default)
    {
        var session = await _sessionManager.CreateSessionAsync(
            "CodebuddyCli",
            request.WorkingDirectory,
            cancellationToken);

        try
        {
            var fullPrompt = BuildPrompt(request);
            await session.SendPromptAsync(fullPrompt, cancellationToken);

            await foreach (var chunk in StreamFromSession(session, cancellationToken))
            {
                yield return chunk;
            }
        }
        finally
        {
            await session.DisposeAsync();
        }
    }

    private async IAsyncEnumerable<AIStreamingChunk> StreamFromSession(
        IACPSession session,
        [EnumeratorCancellation] CancellationToken cancellationToken)
    {
        // Iterate through all updates in the session
        await foreach (var notification in session.ReceiveUpdatesAsync(cancellationToken))
        {
            switch (notification.Update)
            {
                case AgentMessageChunkSessionUpdate agentMessage:
                    // Handle text content chunks
                    if (agentMessage.Content is AcpImp.TextContentBlock textContent)
                    {
                        yield return new AIStreamingChunk
                        {
                            Content = textContent.Text,
                            Type = StreamingChunkType.ContentDelta,
                            IsComplete = false
                        };
                    }
                    break;

                case ToolCallSessionUpdate toolCall:
                    // Handle tool calls
                    yield return new AIStreamingChunk
                    {
                        Content = string.Empty,
                        Type = StreamingChunkType.ToolCallDelta,
                        ToolCallDelta = new AIToolCallDelta
                        {
                            Id = toolCall.ToolCallId,
                            Name = toolCall.Kind.ToString(),
                            Arguments = toolCall.RawInput?.ToString()
                        }
                    };
                    break;

                case AcpImp.PromptCompletedSessionUpdate:
                    // Response complete
                    yield break;
            }
        }
    }

    // Build the full prompt
    private string BuildPrompt(AIRequest request, string? embeddedCommandPrompt = null)
    {
        var sb = new StringBuilder();

        // Embedded command prompt, if present
        if (!string.IsNullOrEmpty(embeddedCommandPrompt))
        {
            sb.AppendLine(embeddedCommandPrompt);
            sb.AppendLine();
        }

        // System message
        if (!string.IsNullOrEmpty(request.SystemMessage))
        {
            sb.AppendLine(request.SystemMessage);
            sb.AppendLine();
        }

        // User prompt
        sb.Append(request.Prompt);
        return sb.ToString();
    }
}
```

มีประเด็นสำคัญหลายประการในรหัสนี้:

1. **การจัดการเซสชัน**: แต่ละคำขอจะสร้างเซสชันอิสระและเผยแพร่ทรัพยากรหลังจากคำขอเสร็จสมบูรณ์ นี่คือบทเรียนที่ได้รับจากการลองผิดลองถูก หากการจัดการการนำเซสชันกลับมาใช้ใหม่ไม่ดีพอ มลภาวะของรัฐก็จะปรากฏขึ้นได้อย่างง่ายดาย

2. **การประมวลผลแบบสตรีมมิ่ง**: `IAsyncEnumerable` อนุญาตให้มีการตอบกลับในขณะที่ยังคงสร้างอยู่ แทนที่จะรอให้เนื้อหาทั้งหมดเสร็จสิ้น นี่เป็นสิ่งสำคัญอย่างยิ่งสำหรับสถานการณ์ที่เป็นข้อความยาวและปรับปรุงประสบการณ์ผู้ใช้อย่างมาก

3. **การเรียกเครื่องมือ**: CodeBuddy รองรับการเรียกเครื่องมือ (การเรียกฟังก์ชัน) ซึ่งจัดการผ่าน `ToolCallSessionUpdate`. ความสามารถนี้มีความสำคัญอย่างยิ่งสำหรับงานแก้ไขโค้ดที่ซับซ้อน

4. **การกรองเนื้อหา**: `AIResultContentSanitizer` ใช้เพื่อกรองเนื้อหา Think block และรักษาเอาต์พุตให้สะอาด

### 4. การกำหนดค่าการฉีดพึ่งพา

เพิ่มบริการที่เกี่ยวข้องระหว่างการลงทะเบียนโมดูล:

```csharp
// PCodeClaudeHelperModule.cs
public void ConfigureModule(IServiceCollection context)
{
    // Register Provider
    context.Services.AddTransient<CodebuddyCliProvider>();

    // Register ACP infrastructure
    context.Services.AddSingleton<IACPSessionManager, ACPSessionManager>();
    context.Services.AddSingleton<IAcpPlatformConfigurationResolver, AcpPlatformConfigurationResolver>();
    context.Services.AddSingleton<IAIRequestToAcpMapper, AIRequestToAcpMapper>();
    context.Services.AddSingleton<IAcpToAIResponseMapper, AcpToAIResponseMapper>();
}
```

## ตัวอย่างการกำหนดค่า

### ไฟล์การกำหนดค่า

เพิ่มการกำหนดค่าที่เกี่ยวข้องกับ CodeBuddy ไปที่ `appsettings.json`:

```yaml
AI:
  # Default Provider to use
  DefaultProvider: "CodebuddyCli"

  # Provider configuration
  Providers:
    CodebuddyCli:
      Type: "CodebuddyCli"
      WorkingDirectory: "C:/projects/my-app"
      ExecutablePath: "C:/tools/codebuddy.cmd"

  # Platform-specific configuration
  PlatformConfigurations:
    CodebuddyCli:
      ExecutablePath: "C:/tools/codebuddy.cmd"
      Arguments: "--acp"
      StartupTimeoutMs: 5000
      EnvironmentVariables:
        CODEBUDDY_API_KEY: "${CODEBUDDY_API_KEY}"
        CODEBUDDY_INTERNET_ENVIRONMENT: "production"
```

### รูปแบบการกำหนดค่า

คำจำกัดความของรูปแบบการกำหนดค่าที่สอดคล้องกัน:

```csharp
public class CodebuddyPlatformConfiguration : IAcpPlatformConfiguration
{
    public string ProviderName => "CodebuddyCli";
    public AcpTransportType TransportType => AcpTransportType.Stdio;

    public string ExecutablePath { get; set; } = "codebuddy";
    public string Arguments { get; set; } = "--acp";
    public int StartupTimeoutMs { get; set; } = 5000;

    public Dictionary<string, string?>? EnvironmentVariables { get; set; }
}
```

## บทเรียนภาคปฏิบัติที่ได้รับ

### บันทึกหลุมพราง

เราพบข้อผิดพลาดทั่วไปหลายประการระหว่างการใช้งาน และการแบ่งปันที่นี่อาจช่วยให้ผู้อื่นหลีกเลี่ยงการเลี่ยงแบบเดียวกัน:

1. **ปัญหาการรั่วไหลของเซสชัน**: ในตอนแรก เซสชันไม่ได้รับการเผยแพร่อย่างถูกต้อง ซึ่งทำให้ทรัพยากรกระบวนการหมดลง วิธีแก้ไขคือการใช้ `try-finally` เพื่อให้แน่ใจว่าทรัพยากรได้รับการเผยแพร่สำหรับทุกคำขอ

2. **การส่งผ่านตัวแปรสภาพแวดล้อม**: Windows และ Linux ใช้ไวยากรณ์ของตัวแปรสภาพแวดล้อมที่แตกต่างกัน ดังนั้นเราจึงสร้างมาตรฐานในภายหลัง `Dictionary<string, string?>` เพื่อจัดการกับสิ่งนี้

3. **การกำหนดค่าการหมดเวลา**: การเริ่มต้น CLI ต้องใช้เวลา ดังนั้นเราจึงตั้งค่าการหมดเวลาการเริ่มต้น 5 วินาทีเพื่อหลีกเลี่ยงความล้มเหลวของคำขอที่รวดเร็ว

4. **ปัญหาการเข้ารหัส**: บน Windows การเข้ารหัสเริ่มต้นอาจทำให้ข้อความภาษาจีนอ่านไม่ออก ดังนั้นการเข้ารหัส UTF-8 จึงมีการระบุอย่างชัดเจนเมื่อเริ่มกระบวนการ

### การเพิ่มประสิทธิภาพการทำงาน

1. **กลุ่มเซสชัน**: สำหรับคำขอสั้นๆ บ่อยครั้ง ให้พิจารณาใช้กลุ่มเซสชันเพื่อนำกระบวนการกลับมาใช้ใหม่
2. **แคชการเชื่อมต่อ**: คลาสโรงงานรองรับอินสแตนซ์ของผู้ให้บริการแคชแล้ว
3. **Async ก่อน**: ใช้การเขียนโปรแกรมแบบอะซิงโครนัสตลอดเพื่อหลีกเลี่ยงการบล็อกเธรด

ประสิทธิภาพคุ้มค่ากับการเพิ่มประสิทธิภาพเสมอ ยิ่งผู้ใช้รอนานเท่าไร ประสบการณ์ก็จะยิ่งแย่ลงเท่านั้น

## บทสรุป

บทความนี้แนะนำโซลูชันที่สมบูรณ์สำหรับการผสานรวม CodeBuddy CLI เข้ากับแบ็กเอนด์ C# ซึ่งครอบคลุมกระบวนการทั้งหมดตั้งแต่การออกแบบสถาปัตยกรรมไปจนถึงการใช้งานอย่างเป็นรูปธรรม เราแยกรายละเอียดโปรโตคอลออกจากตรรกะทางธุรกิจผ่านสถาปัตยกรรมแบบเลเยอร์ ทำให้โค้ดมีความชัดเจนและง่ายต่อการบำรุงรักษา

ประเด็นสำคัญ:

- ใช้สถาปัตยกรรมแบบเลเยอร์กับเลเยอร์สัญญาของผู้ให้บริการ เลเยอร์โรงงาน เลเยอร์การใช้งาน และเลเยอร์โครงสร้างพื้นฐาน
- ใช้ JSON-RPC บน Stdio สำหรับการสื่อสารระหว่างกระบวนการ
- ใช้การกำหนดค่าที่ยืดหยุ่นและความสามารถในการขยายผ่านการฉีดขึ้นต่อกัน
- จัดให้มีโหมดการร้องขอทั้งแบบสตรีมมิ่งและแบบไม่สตรีมมิ่ง

วิธีการนี้ไม่เพียงแต่เหมาะสำหรับ CodeBuddy เท่านั้น แต่ยังเหมาะสำหรับ CodeBuddy ด้วย การเพิ่มผู้ให้บริการ AI รายใหม่ก็สามารถทำตามรูปแบบเดียวกันได้ หากคุณกำลังสร้างการบูรณาการหลาย AI-Provider ที่คล้ายกัน ฉันหวังว่าบทความนี้จะเป็นข้อมูลอ้างอิงที่เป็นประโยชน์แก่คุณ
