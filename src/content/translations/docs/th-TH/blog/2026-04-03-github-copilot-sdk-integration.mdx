## จากการเรียก CLI ไปจนถึงการรวม SDK: วิธีปฏิบัติที่ดีที่สุดสำหรับ GitHub Copilot ในโปรเจ็กต์ .NET

> เส้นทางการอัพเกรดจากการเรียกใช้บรรทัดคำสั่งไปจนถึงการรวม SDK อย่างเป็นทางการนั้นค่อนข้างเป็นการเดินทาง วันนี้ ฉันต้องการแบ่งปันข้อผิดพลาดที่เราพบและสิ่งที่เราเรียนรู้ขณะสร้าง HagiCode

## พื้นหลัง

หลังจากที่ GitHub Copilot SDK เปิดตัวอย่างเป็นทางการในปี 2025 เราก็เริ่มรวมเข้ากับเลเยอร์ความสามารถ AI ของเรา ก่อนหน้านั้น โปรเจ็กต์ใช้ GitHub Copilot เป็นหลักโดยการเรียกใช้เครื่องมือบรรทัดคำสั่ง Copilot CLI โดยตรง แต่แนวทางนั้นมีปัญหาที่ชัดเจนหลายประการ:

- **การจัดการกระบวนการที่ซับซ้อน**: เราต้องจัดการวงจรกระบวนการ CLI การหมดเวลาเริ่มต้น และการล้างกระบวนการด้วยตนเอง กระบวนการอาจหยุดทำงานโดยไม่มีการเตือนล่วงหน้า
- **การจัดการเหตุการณ์ที่ไม่สมบูรณ์**: การเรียกใช้ Raw CLI ทำให้ยากต่อการจับภาพเหตุการณ์แบบละเอียดจากการให้เหตุผลของโมเดลและการใช้เครื่องมือ เหมือนเห็นแต่ผลโดยไม่ใช้กระบวนการคิด
- **การจัดการเซสชันที่ยากลำบาก**: ไม่มีกลไกที่มีประสิทธิภาพสำหรับการใช้ซ้ำและการกู้คืนเซสชัน ดังนั้นทุกการโต้ตอบจึงต้องเริ่มต้นใหม่
- **ปัญหาความเข้ากันได้**: อาร์กิวเมนต์ CLI มีการเปลี่ยนแปลงบ่อยครั้ง ซึ่งหมายความว่าเราต้องรักษาตรรกะความเข้ากันได้สำหรับพารามิเตอร์เหล่านั้นอยู่ตลอดเวลา

ปัญหาเหล่านี้ปรากฏชัดเจนมากขึ้นในการพัฒนาในแต่ละวัน โดยเฉพาะอย่างยิ่งเมื่อเราจำเป็นต้องติดตามการใช้เหตุผลของโมเดล (`thinking`) และสถานะการทำงานของเครื่องมือแบบเรียลไทม์ เมื่อถึงจุดนั้น เห็นได้ชัดว่าเราต้องการแนวทางบูรณาการในระดับที่ต่ำกว่าและสมบูรณ์ยิ่งขึ้น

## เกี่ยวกับ ฮากิโค้ด

แนวทางที่แบ่งปันในบทความนี้มาจากประสบการณ์จริงของเราใน [ฮากิโค้ด](https://hagicode.com) โครงการ HagiCode เป็นโปรเจ็กต์ผู้ช่วยเขียนโค้ด AI แบบโอเพ่นซอร์ส และในระหว่างการพัฒนา เราต้องการการบูรณาการเชิงลึกกับความสามารถของ GitHub Copilot ตั้งแต่การเติมโค้ดพื้นฐานให้สมบูรณ์ไปจนถึงการสนทนาหลายรอบที่ซับซ้อนและการเรียกใช้เครื่องมือ ข้อกำหนดในโลกแห่งความเป็นจริงเหล่านั้นผลักดันให้เราเปลี่ยนจากการเรียกใช้ CLI มาเป็น SDK อย่างเป็นทางการ

หากการใช้งานนี้ฟังดูมีประโยชน์สำหรับคุณ ก็มีโอกาสที่ดีที่ประสบการณ์ด้านวิศวกรรมของเราสามารถช่วยได้ ในกรณีดังกล่าว โครงการ HagiCode เองก็อาจคุ้มค่าที่จะลองดูเช่นกัน คุณอาจพบข้อมูลโครงการและลิงก์เพิ่มเติมในตอนท้ายของบทความนี้

## การออกแบบสถาปัตยกรรม

โปรเจ็กต์ใช้สถาปัตยกรรมแบบเลเยอร์เพื่อแก้ไขข้อจำกัดของการเรียกใช้ CLI:

```
┌─────────────────────────────────────────────────────────┐
│  hagicode-core (Orleans Grains + AI Provider Layer)    │
│  - CopilotAIProvider: Converts AIRequest to CopilotOptions │
│  - GitHubCopilotGrain: Orleans distributed execution interface │
└─────────────────────────────────────────────────────────┘
                          ↓
┌─────────────────────────────────────────────────────────┐
│  HagiCode.Libs (Shared Provider Layer)                 │
│  - CopilotProvider: CLI Provider interface implementation │
│  - ICopilotSdkGateway: SDK invocation abstraction      │
│  - GitHubCopilotSdkGateway: SDK session management and event dispatch │
└─────────────────────────────────────────────────────────┘
                          ↓
┌─────────────────────────────────────────────────────────┐
│  GitHub Copilot SDK (Official .NET SDK)                │
│  - CopilotClient: SDK client                           │
│  - CopilotSession: Session management                  │
│  - SessionEvent: Event stream                          │
└─────────────────────────────────────────────────────────┘
```

การออกแบบหลายชั้นนี้นำมาซึ่งข้อดีทางเทคนิคที่ใช้งานได้จริงหลายประการ:

1. **การแยกข้อกังวล**: ตรรกะทางธุรกิจหลักแยกออกจากรายละเอียดการใช้งาน SDK
2. **ความสามารถในการทดสอบ**: The `ICopilotSdkGateway` อินเทอร์เฟซทำให้การทดสอบหน่วยตรงไปตรงมา
3. **การนำกลับมาใช้ใหม่**: `HagiCode.Libs` สามารถอ้างอิงได้หลายโครงการ
4. **การบำรุงรักษา**: การอัปเกรด SDK ต้องการการเปลี่ยนแปลงในเลเยอร์เกตเวย์เท่านั้น ในขณะที่ชั้นบนยังคงไม่ถูกแตะต้อง

## การดำเนินงานหลัก

### ขั้นตอนการรับรองความถูกต้อง

การรับรองความถูกต้องเป็นขั้นตอนแรกและสำคัญที่สุดของการรวม SDK หากการรับรองความถูกต้องล้มเหลว ก็ไม่มีอะไรสำคัญอีกต่อไป เราได้ออกแบบการกำหนดค่าการรับรองความถูกต้องที่ยืดหยุ่นซึ่งสนับสนุนแหล่งที่มาของการรับรองความถูกต้องหลายแหล่ง:

```csharp
// CopilotProvider.cs - Authentication source configuration
public class CopilotOptions
{
    public bool UseLoggedInUser { get; set; } = true;
    public string? GitHubToken { get; set; }
    public string? CliUrl { get; set; }
}

// Convert to SDK request
return new CopilotSdkRequest(
    GitHubToken: options.AuthSource == CopilotAuthSource.GitHubToken
        ? options.GitHubToken
        : null,
    UseLoggedInUser: options.AuthSource != CopilotAuthSource.GitHubToken
);
```

ประโยชน์ของการออกแบบนี้ค่อนข้างชัดเจน:

- รองรับโหมดผู้ใช้ที่เข้าสู่ระบบโดยไม่ต้องใช้โทเค็น ซึ่งเหมาะกับสถานการณ์เดสก์ท็อปเป็นอย่างดี
- รองรับโหมด GitHub Token ซึ่งเหมาะสำหรับการปรับใช้ฝั่งเซิร์ฟเวอร์และการจัดการแบบรวมศูนย์
- รองรับการแทนที่ Copilot CLI URL ซึ่งช่วยในการกำหนดค่าพร็อกซีขององค์กร

ในทางปฏิบัติ โมเดลการตรวจสอบสิทธิ์แบบยืดหยุ่นนี้ทำให้การกำหนดค่าในสถานการณ์การใช้งานต่างๆ ง่ายขึ้นอย่างมาก ไคลเอนต์เดสก์ท็อปสามารถใช้สถานะการเข้าสู่ระบบ Copilot ของผู้ใช้แต่ละคนได้ ในขณะที่ฝั่งเซิร์ฟเวอร์สามารถจัดการการรับรองความถูกต้องจากส่วนกลางผ่านโทเค็น

### การจัดการสตรีมเหตุการณ์

ความสามารถที่ทรงพลังที่สุดประการหนึ่งของ SDK คือการจับภาพสตรีมเหตุการณ์ที่สมบูรณ์ เราใช้ระบบจัดส่งเหตุการณ์ที่สามารถจัดการเหตุการณ์ SDK ทุกประเภทแบบเรียลไทม์:

```csharp
// GitHubCopilotSdkGateway.cs - Core logic for event dispatch
internal static SessionEventDispatchResult DispatchSessionEvent(
    SessionEvent evt, bool sawDelta)
{
    switch (evt)
    {
        case AssistantReasoningEvent reasoningEvent:
            // Capture the model reasoning process
            events.Add(new CopilotSdkStreamEvent(
                CopilotSdkStreamEventType.ReasoningDelta,
                Content: reasoningEvent.Data.Content));
            break;

        case ToolExecutionStartEvent toolStartEvent:
            // Capture the start of a tool invocation
            events.Add(new CopilotSdkStreamEvent(
                CopilotSdkStreamEventType.ToolExecutionStart,
                ToolName: toolStartEvent.Data.ToolName,
                ToolCallId: toolStartEvent.Data.ToolCallId));
            break;

        case ToolExecutionCompleteEvent toolCompleteEvent:
            // Capture tool completion and its result
            events.Add(new CopilotSdkStreamEvent(
                CopilotSdkStreamEventType.ToolExecutionEnd,
                Content: ExtractToolExecutionContent(toolCompleteEvent)));
            break;

        default:
            // Preserve unhandled events as RawEvent
            events.Add(new CopilotSdkStreamEvent(
                CopilotSdkStreamEventType.RawEvent,
                RawEventType: evt.GetType().Name));
            break;
    }
}
```

คุณค่าของการดำเนินการนี้มีความสำคัญ:

- **บันทึกกระบวนการให้เหตุผลแบบจำลองโดยสมบูรณ์** (`thinking`): ผู้ใช้สามารถดูว่า AI ให้เหตุผลอย่างไร ไม่ใช่แค่คำตอบสุดท้าย
- **การติดตามสถานะการทำงานของเครื่องมือแบบเรียลไทม์**: เรารู้ว่าเครื่องมือใดกำลังทำงานอยู่ ทำงานเสร็จเมื่อใด และส่งคืนอะไรบ้าง
- **การสูญเสียเหตุการณ์เป็นศูนย์**: โดยการถอยกลับไป `RawEvent`เรารับรองว่าทุกเหตุการณ์จะถูกบันทึกไว้

ใน HagiCode เหตุการณ์ที่ละเอียดเหล่านี้ช่วยให้ผู้ใช้เข้าใจวิธีการทำงานของ AI ภายใน โดยเฉพาะอย่างยิ่งเมื่อทำการดีบักงานที่ซับซ้อน

### การจัดการความเข้ากันได้ของ CLI

หลังจากย้ายจากการเรียกใช้ CLI ไปยัง SDK เราพบว่าพารามิเตอร์ CLI ที่มีอยู่บางรายการไม่ได้นำไปใช้กับ SDK อีกต่อไป เพื่อรักษาความเข้ากันได้แบบย้อนหลัง เราได้ใช้ระบบการกรองพารามิเตอร์:

```csharp
// CopilotCliCompatibility.cs - Argument filtering
private static readonly Dictionary<string, string> RejectedFlags = new()
{
    ["--headless"] = "Unsupported startup argument",
    ["--model"] = "Passed through an SDK-native field",
    ["--prompt"] = "Passed through an SDK-native field",
    ["--interactive"] = "Interaction is managed by the provider",
};

public static CopilotCliArgumentBuildResult BuildCliArgs(CopilotOptions options)
{
    // Filter out unsupported arguments and keep compatible ones
    // Generate diagnostic information
}
```

สิ่งนี้ทำให้เราได้รับประโยชน์หลายประการ:

- โดยจะกรองอาร์กิวเมนต์ CLI ที่เข้ากันไม่ได้โดยอัตโนมัติเพื่อป้องกันข้อผิดพลาดรันไทม์
- สร้างข้อความวินิจฉัยที่ชัดเจนเพื่อช่วยให้นักพัฒนาระบุปัญหาได้อย่างรวดเร็ว
- ช่วยให้ SDK มีเสถียรภาพและเป็นฉนวนจากการเปลี่ยนแปลงในพารามิเตอร์ CLI

ในระหว่างการอัปเกรด กลไกความเข้ากันได้นี้ช่วยให้เราเปลี่ยนผ่านได้อย่างราบรื่น ไฟล์การกำหนดค่าที่มีอยู่ยังคงสามารถใช้ได้ และจำเป็นต้องปรับเปลี่ยนทีละน้อยตามเอาต์พุตการวินิจฉัยเท่านั้น

### การรวมรันไทม์

การสร้างเซสชัน Copilot SDK มีราคาค่อนข้างแพง และการสร้างและทำลายเซสชันบ่อยเกินไปจะทำให้ประสิทธิภาพลดลง เพื่อแก้ปัญหาดังกล่าว เราได้ใช้ระบบการจัดการเซสชันพูล:

```csharp
// CopilotProvider.cs - Session pool management
await using var lease = await _poolCoordinator.AcquireCopilotRuntimeAsync(
    request,
    async ct => await _gateway.CreateRuntimeAsync(sdkRequest, ct),
    cancellationToken);

if (lease.IsWarmLease)
{
    // Reuse an existing session
    yield return CreateSessionReusedMessage();
}

await foreach (var eventData in lease.Entry.Resource.SendPromptAsync(...))
{
    yield return MapEvent(eventData);
}
```

ประโยชน์ของการรวมเซสชันได้แก่:

- **การใช้เซสชันซ้ำ**: คำขอที่เหมือนกัน `sessionId` สามารถใช้เซสชันที่มีอยู่ซ้ำและลดค่าใช้จ่ายในการเริ่มต้นระบบได้
- **การสนับสนุนการกู้คืนเซสชัน**: หลังจากที่เครือข่ายหยุดชะงัก สถานะเซสชันก่อนหน้าสามารถกู้คืนได้
- **การจัดการการรวมกลุ่มอัตโนมัติ**: เซสชันที่หมดอายุจะถูกล้างข้อมูลโดยอัตโนมัติเพื่อหลีกเลี่ยงการรั่วไหลของทรัพยากร

ใน HagiCode การรวมเซสชั่นการตอบสนองดีขึ้นอย่างเห็นได้ชัด โดยเฉพาะอย่างยิ่งสำหรับการสนทนาต่อเนื่อง

### บูรณาการออร์ลีนส์

HagiCode ใช้ Orleans เป็นเฟรมเวิร์กแบบกระจาย และเราได้รวม Copilot SDK เข้ากับ Orleans Grains:

```csharp
// GitHubCopilotGrain.cs - Distributed execution
public async IAsyncEnumerable<GitHubCopilotResponse> ExecuteCommandStreamAsync(
    string command,
    CancellationToken token = default)
{
    var provider = await aiProviderFactory.GetProviderAsync(AIProviderType.GitHubCopilot);

    await foreach (var chunk in provider.SendMessageAsync(request, null, token))
    {
        // Map to the unified response format
        yield return BuildChunkResponse(chunk, startedAt);
    }
}
```

ข้อดีของการรวมกลุ่มของออร์ลีนส์นั้นมีมากมาย:

- **นามธรรมของผู้ให้บริการ AI แบบครบวงจร**: การสลับระหว่างผู้ให้บริการ AI ต่างๆ กลายเป็นเรื่องง่าย
- **การแยกผู้เช่าหลายราย**: เซสชัน Copilot สำหรับผู้ใช้ที่แตกต่างกันยังคงแยกออกจากกัน
- **สถานะเซสชันถาวร**: สถานะเซสชันสามารถกู้คืนได้แม้หลังจากที่เซิร์ฟเวอร์รีสตาร์ทแล้ว

สำหรับสถานการณ์ที่ต้องจัดการกับคำขอจำนวนมากที่เกิดขึ้นพร้อมกัน Orleans มอบความสามารถในการปรับขนาดที่แข็งแกร่ง

```json
          "UseLoggedInUser": true,
          "NoAskUser": true,
          "Permissions": {
            "AllowAllTools": false,
            "AllowedTools": ["Read", "Bash", "Grep"],
            "DeniedTools": ["Edit"]
          }
        }
      }
    }
  }
}
```

### หมายเหตุการใช้งาน

ในการใช้งานจริง เราได้สรุปประเด็นต่างๆ ที่ควรคำนึงถึงไว้ดังนี้:

**การกำหนดค่าการหมดเวลาการเริ่มต้น**: การเริ่มต้น Copilot CLI ครั้งแรกอาจใช้เวลานานพอสมควร ดังนั้นเราขอแนะนำให้ตั้งค่า `StartupTimeout` อย่างน้อย 30 วินาที หากเป็นการเข้าสู่ระบบครั้งแรก อาจใช้เวลานานกว่านี้อีก

**การจัดการสิทธิ์**: ในสภาพแวดล้อมการใช้งานจริง ให้หลีกเลี่ยงการใช้ `AllowAllTools: true`. ใช้ `AllowedTools` รายการที่อนุญาตเพื่อควบคุมเครื่องมือที่พร้อมใช้งาน และใช้ `DeniedTools` รายชื่อที่ปฏิเสธเพื่อป้องกันการดำเนินการที่เป็นอันตราย สิ่งนี้จะป้องกัน AI จากการรันคำสั่งที่มีความเสี่ยงได้อย่างมีประสิทธิภาพ

**การจัดการเซสชัน**: คำขอที่เหมือนกัน `sessionId` ใช้เซสชันซ้ำโดยอัตโนมัติ สถานะเซสชันยังคงอยู่ผ่าน `ProviderSessionId`. การยกเลิกจะเผยแพร่ผ่านทาง `CancellationTokenSource`.

**เอาต์พุตการวินิจฉัย**: อาร์กิวเมนต์ CLI ที่เข้ากันไม่ได้จะสร้างข้อความประเภท `diagnostic`. เหตุการณ์ Raw SDK จะถูกเก็บรักษาไว้เป็น `event.raw`. ข้อความแสดงข้อผิดพลาดประกอบด้วยหมวดหมู่ต่างๆ เช่น การหมดเวลาเริ่มต้นและความไม่เข้ากันของอาร์กิวเมนต์ เพื่อให้การแก้ไขปัญหาง่ายขึ้น

### แนวทางปฏิบัติที่ดีที่สุด

จากประสบการณ์จริงของเรา แนวทางปฏิบัติที่ดีที่สุดบางประการมีดังนี้:

**1. ใช้รายการที่อนุญาตของเครื่องมือ**

```csharp
var request = new AIRequest
{
    Prompt = "Analyze this file",
    AllowedTools = new[] { "Read", "Grep", "Bash(git:*)" }
};
```

การระบุเครื่องมือที่อนุญาตอย่างชัดเจนผ่านรายการที่อนุญาตจะช่วยป้องกันการกระทำของ AI ที่ไม่คาดคิด นี่เป็นสิ่งสำคัญอย่างยิ่งสำหรับเครื่องมือที่มีสิทธิ์ในการเขียน เช่น `Edit`ซึ่งควรได้รับการดูแลเป็นพิเศษ

**2. ตั้งค่าการหมดเวลาที่เหมาะสม**

```csharp
options.Timeout = 3600;  // 1 hour
options.StartupTimeout = 60;  // 1 minute
```

ตั้งค่าการหมดเวลาที่เหมาะสมตามความซับซ้อนของงาน หากค่าสั้นเกินไป งานอาจถูกขัดจังหวะ หากยาวเกินไป ทรัพยากรอาจสูญเปล่าเพื่อรอคำขอที่ไม่ตอบสนอง

**3. เปิดใช้งานการใช้เซสชันซ้ำ**

```csharp
options.SessionId = "my-session-123";
```

ใช้เหมือนกัน `sessionId` สำหรับงานที่เกี่ยวข้องช่วยให้คุณสามารถนำบริบทของเซสชันก่อนหน้ากลับมาใช้ใหม่ได้ และปรับปรุงความเร็วในการตอบสนอง

**4. จัดการการตอบกลับแบบสตรีมมิ่ง**

```csharp
await foreach (var chunk in provider.StreamAsync(request))
{
    switch (chunk.Type)
    {
        case StreamingChunkType.ThinkingDelta:
            // Handle the reasoning process
            break;
        case StreamingChunkType.ToolCallDelta:
            // Handle tool invocation
            break;
        case StreamingChunkType.ContentDelta:
            // Handle text output
            break;
    }
}
```

การตอบกลับแบบสตรีมช่วยให้คุณแสดงความคืบหน้าในการประมวลผลของ AI ได้แบบเรียลไทม์ ซึ่งจะช่วยปรับปรุงประสบการณ์ผู้ใช้ สิ่งนี้มีประโยชน์อย่างยิ่งสำหรับงานที่ต้องใช้เวลานาน

**5. การจัดการข้อผิดพลาดและลองใหม่อีกครั้ง**

```csharp
try
{
    await foreach (var chunk in provider.StreamAsync(request))
    {
        // Handle the response
    }
}
catch (CopilotSessionException ex)
{
    // Handle session exceptions
    logger.LogError(ex, "Copilot session failed");
    // Decide whether to retry based on the exception type
}
```

การจัดการข้อผิดพลาดที่เหมาะสมและตรรกะการลองใหม่ช่วยปรับปรุงความเสถียรของระบบโดยรวม

## บทสรุป

การอัปเกรดจากการเรียกใช้ CLI เป็นการบูรณาการ SDK มอบคุณค่าที่สำคัญให้กับโครงการ HagiCode:

- **ปรับปรุงความเสถียร**: SDK มอบอินเทอร์เฟซที่เสถียรยิ่งขึ้นซึ่งไม่ได้รับผลกระทบจากการเปลี่ยนแปลงเวอร์ชันของ CLI
- **ฟังก์ชันการทำงานที่สมบูรณ์ยิ่งขึ้น**: บันทึกสตรีมเหตุการณ์ทั้งหมด รวมถึงการให้เหตุผลและสถานะการทำงานของเครื่องมือ
- **ประสิทธิภาพการพัฒนาที่สูงขึ้น**: อินเทอร์เฟซ SDK ที่ปลอดภัยต่อประเภททำให้การพัฒนามีประสิทธิภาพมากขึ้น และลดข้อผิดพลาดรันไทม์
- **ประสบการณ์ผู้ใช้ที่ดีขึ้น**: ความคิดเห็นเกี่ยวกับเหตุการณ์แบบเรียลไทม์ช่วยให้ผู้ใช้มีความเข้าใจที่ชัดเจนยิ่งขึ้นว่า AI กำลังทำอะไรอยู่

การอัปเกรดนี้ไม่ใช่แค่การแทนที่การใช้งานด้านเทคนิคเท่านั้น นอกจากนี้ยังเป็นการเพิ่มประสิทธิภาพทางสถาปัตยกรรมของชั้นความสามารถ AI ทั้งหมดอีกด้วย ด้วยการออกแบบแบบเลเยอร์และอินเทอร์เฟซแบบนามธรรม ทำให้เราได้รับการบำรุงรักษาและความสามารถในการขยายได้ดียิ่งขึ้น

หากคุณกำลังพิจารณาที่จะรวม GitHub Copilot เข้ากับโปรเจ็กต์ .NET ของคุณเอง ฉันหวังว่าบทเรียนเชิงปฏิบัติเหล่านี้จะช่วยให้คุณหลีกเลี่ยงการเลี่ยงโดยไม่จำเป็น SDK อย่างเป็นทางการมีความเสถียรและสมบูรณ์กว่าการเรียกใช้ CLI อย่างแน่นอน และคุ้มค่ากับเวลาที่จะทำความเข้าใจและปรับใช้
