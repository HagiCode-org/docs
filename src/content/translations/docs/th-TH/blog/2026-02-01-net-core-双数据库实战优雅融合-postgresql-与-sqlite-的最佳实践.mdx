## .NET Core Dual-Database ในทางปฏิบัติ: ให้ PostgreSQL และ SQLite อยู่ร่วมกันอย่างสันติ

> เมื่อสร้างแอปพลิเคชันที่ทันสมัย เรามักจะเผชิญกับข้อเสียเปรียบนี้: สภาพแวดล้อมการพัฒนาต้องการสิ่งที่มีน้ำหนักเบาและสะดวกสบาย ในขณะที่สภาพแวดล้อมการใช้งานจริงต้องการการทำงานพร้อมกันสูงและมีความพร้อมใช้งานสูง บทความนี้แบ่งปันวิธีการสนับสนุนทั้ง PostgreSQL และ SQLite อย่างดีในโครงการ .NET Core และใช้แนวปฏิบัติที่ดีที่สุดของ "SQLite สำหรับการพัฒนา PostgreSQL สำหรับการผลิต"


## พื้นหลัง

ในการพัฒนาซอฟต์แวร์ ความแตกต่างระหว่างสภาพแวดล้อมถือเป็นปัญหาที่ยากที่สุดประการหนึ่งสำหรับทีมวิศวกรมาโดยตลอด ยกตัวอย่างแพลตฟอร์ม **HagiCode** ที่เรากำลังสร้าง โดยเป็นระบบการพัฒนาที่ได้รับความช่วยเหลือจาก AI ซึ่งใช้ ASP.NET Core 10 และ React โดยมี Orleans บูรณาการภายในเพื่อการจัดการสถานะแบบกระจาย สแต็กมีความทันสมัยและค่อนข้างซับซ้อน

ในช่วงต้นของโปรเจ็กต์ เราพบปัญหาทางวิศวกรรมแบบคลาสสิก: นักพัฒนาต้องการให้สภาพแวดล้อมภายในเครื่องทำงานนอกกรอบ โดยไม่ต้องติดตั้งและกำหนดค่าฐานข้อมูล PostgreSQL จำนวนมาก แต่ในการผลิต เราจำเป็นต้องจัดการกับการเขียนที่เกิดขึ้นพร้อมกันสูงและแบบสอบถาม JSON ที่ซับซ้อน และนั่นคือสิ่งที่ SQLite แบบเบาเริ่มแสดงขีดจำกัดของมัน

เราจะเก็บโค้ดเบสเดียวในขณะที่ปล่อยให้แอปพลิเคชันได้รับประโยชน์จากการพกพาของ SQLite เช่น แอปเดสก์ท็อป และยังใช้ประโยชน์จากประสิทธิภาพอันทรงพลังของ PostgreSQL เช่น บริการระดับองค์กรได้อย่างไร นั่นคือคำถามหลักที่บทความนี้สำรวจ

## เกี่ยวกับ ฮากิโค้ด

วิธีการปรับฐานข้อมูลคู่ที่แชร์ในบทความนี้มาจากประสบการณ์ตรงของเราในโปรเจ็กต์ **HagiCode** HagiCode เป็นแพลตฟอร์มการพัฒนายุคใหม่ที่ผสานรวมการจัดการพร้อมท์ AI และเวิร์กโฟลว์ OpenSpec มันเป็นการสร้างสมดุลระหว่างประสบการณ์ของนักพัฒนากับความเสถียรในการผลิตที่เราได้มาถึงรูปแบบสถาปัตยกรรมที่ได้รับการพิสูจน์แล้วนี้

โปรดเยี่ยมชมพื้นที่เก็บข้อมูล GitHub ของเราเพื่อดูโปรเจ็กต์ทั้งหมด: [HagiCode-org/site](https://github.com/HagiCode-org/site).

## หัวข้อหลักที่ 1: การออกแบบสถาปัตยกรรมและนามธรรมแบบครบวงจร

เพื่อรองรับฐานข้อมูลสองฐานข้อมูลใน .NET Core แนวคิดหลักคือการขึ้นอยู่กับนามธรรมมากกว่าการใช้งานที่เป็นรูปธรรม เราจำเป็นต้องแยกการเลือกฐานข้อมูลออกจากรหัสธุรกิจ และปล่อยให้เลเยอร์การกำหนดค่าตัดสินใจ

### แนวทางการออกแบบ

1.  **อินเทอร์เฟซแบบรวม**: ตรรกะทางธุรกิจทั้งหมดควรขึ้นอยู่กับ `DbContext` คลาสพื้นฐานหรืออินเทอร์เฟซแบบกำหนดเอง แทนที่จะเป็นคลาสเฉพาะ `PostgreSqlDbContext`.
2.  **ขับเคลื่อนด้วยการกำหนดค่า**: ใช้รายการการกำหนดค่าใน `appsettings.json` เพื่อตัดสินใจแบบไดนามิกว่าจะโหลดผู้ให้บริการฐานข้อมูลใดเมื่อเริ่มต้นแอปพลิเคชัน
3.  **การแยกคุณสมบัติ**: เพิ่มตรรกะการปรับตัวสำหรับความสามารถเฉพาะของ PostgreSQL เช่น JSONB เพื่อให้แอปพลิเคชันยังสามารถลดระดับลงได้อย่างสวยงามบน SQLite

### การติดตั้งโค้ด: การกำหนดค่าบริบทแบบไดนามิก

ใน ASP.NET Core `Program.cs`เราไม่ควรฮาร์ดโค้ด `UseNpgsql` หรือ `UseSqlite`. แต่เราควรอ่านการกำหนดค่าและตัดสินใจแบบไดนามิกแทน

ขั้นแรก ให้กำหนดคลาสการกำหนดค่า:

```csharp
public class DatabaseSettings
{
    public const string SectionName = "Database";
    
    // Database type: PostgreSQL or SQLite
    public string DbType { get; set; } = "PostgreSQL"; 
    
    // Connection string
    public string ConnectionString { get; set; } = string.Empty;
}
```

จากนั้นลงทะเบียนใช้บริการใน `Program.cs` ขึ้นอยู่กับการกำหนดค่า:

```csharp
// Read configuration
var databaseSettings = builder.Configuration.GetSection(DatabaseSettings.SectionName).Get<DatabaseSettings>();

// Register DbContext
builder.Services.AddDbContext<ApplicationDbContext>(options =>
{
    if (databaseSettings?.DbType?.ToLower() == "sqlite")
    {
        // SQLite configuration
        options.UseSqlite(databaseSettings.ConnectionString);
        
        // Handling SQLite's concurrent write limitations
        // Note: in production, enabling WAL mode is recommended to improve concurrency performance
    }
    else
    {
        // PostgreSQL configuration (default)
        options.UseNpgsql(databaseSettings.ConnectionString, npgsqlOptions =>
        {
            // Enable JSONB support, which is very useful when handling AI conversation records
            npgsqlOptions.UseJsonNet(); 
        });
        
        // Configure connection pool retry policy
        options.EnableRetryOnFailure(3);
    }
});
```

## หัวข้อหลัก 2: การจัดการความแตกต่างและกลยุทธ์การย้ายถิ่น

แม้ว่า PostgreSQL และ SQLite จะรองรับมาตรฐาน SQL ทั้งคู่ แต่ก็มีความแตกต่างกันอย่างมากในด้านความสามารถและลักษณะการทำงานเฉพาะ หากความแตกต่างเหล่านี้ไม่ได้รับการจัดการอย่างระมัดระวัง คุณอาจจบลงด้วยสถานการณ์ที่น่าอึดอัดใจที่ทุกอย่างทำงานได้ภายในเครื่องแต่ล้มเหลวหลังจากการปรับใช้

### 1. การจัดการประเภท JSON

ใน HagiCode เราจำเป็นต้องจัดเก็บพร้อมต์และข้อมูลเมตาของ AI จำนวนมาก ซึ่งโดยปกติจะเกี่ยวข้องกับคอลัมน์ JSON
- **PostgreSQL**: มีเจ้าของภาษา `JSONB` ประเภทที่มีประสิทธิภาพการสืบค้นที่ยอดเยี่ยม
- **SQLite**: ไม่มีประเภท JSON ดั้งเดิม (เวอร์ชันใหม่กว่ามีส่วนขยาย JSON1 แต่การแมปวัตถุยังคงแตกต่างออกไป) ดังนั้นข้อมูลจึงมักจะจัดเก็บเป็น TEXT

**วิธีแก้ปัญหา**:
ในการแมปเอนทิตี EF Core เรากำหนดค่าให้เป็นประเภทที่เปลี่ยนแปลงได้

```csharp
protected override void OnModelCreating(ModelBuilder modelBuilder)
{
    base.OnModelCreating(modelBuilder);

    // Configure entity
    modelBuilder.Entity<PromptTemplate>(entity => 
    {
        entity.Property(e => e.Metadata)
              .HasColumnType("jsonb") // PG uses jsonb
              .HasConversion(
                  v => JsonSerializer.Serialize(v, (JsonSerializerOptions)null),
                  v => JsonSerializer.Deserialize<Dictionary<string, object>>(v, (JsonSerializerOptions)null)
              ); 
    });
}
```

เมื่อใช้ SQLite `HasColumnType("jsonb")` อาจถูกละเลยหรือแจ้งเตือน แต่เนื่องจาก `HasConversion` ได้รับการกำหนดค่า ข้อมูลยังคงเป็นอนุกรมและดีซีเรียลไลซ์อย่างถูกต้องเป็นสตริงที่จัดเก็บไว้ในฟิลด์ TEXT เพื่อให้มั่นใจถึงความเข้ากันได้

### 2. การแยกกลยุทธ์การย้ายถิ่นฐาน

อย่าพยายามทำให้สคริปต์การย้ายข้อมูลชุดเดียวทำงานได้ทั้ง PostgreSQL และ SQLite ในเวลาเดียวกัน ความแตกต่างในกลยุทธ์การสร้างคีย์หลัก ไวยากรณ์ดัชนี และรายละเอียดฐานข้อมูลอื่นๆ จะทำให้เกิดความล้มเหลวอย่างหลีกเลี่ยงไม่ได้

**ข้อปฏิบัติที่แนะนำ**:
รักษาสาขาหรือโครงการการโยกย้ายสองแห่ง ในเวิร์กโฟลว์การพัฒนา HagiCode นี่คือวิธีที่เราจัดการ:

1.  **ขั้นตอนการพัฒนา**: ทำงานกับ SQLite เป็นหลัก ใช้ `Add-Migration Init_Sqlite -OutputDir Migrations/Sqlite`.
2.  **ขั้นตอนการปรับตัว**: หลังจากพัฒนาคุณสมบัติแล้ว ให้สลับสตริงการเชื่อมต่อเป็น PostgreSQL แล้วเรียกใช้ `Add-Migration Init_Postgres -OutputDir Migrations/Postgres`.
3.  **สคริปต์อัตโนมัติ**: เขียนสคริปต์ PowerShell หรือ Bash แบบง่ายเพื่อใช้การโยกย้ายที่ถูกต้องโดยอัตโนมัติตามตัวแปรสภาพแวดล้อมปัจจุบัน

```bash
# Pseudocode for simple deployment logic
if [ "$DATABASE_PROVIDER" = "PostgreSQL" ]; then
    dotnet ef database update --project Migrations.Postgres
else
    dotnet ef database update --project Migrations.Sqlite
fi
```

## หัวข้อหลัก 3: บทเรียนที่ได้รับจาก HagiCode ในการผลิต

ในขณะที่ปรับโครงสร้าง **HagiCode** จากโมเดลฐานข้อมูลเดียวไปเป็นการสนับสนุนฐานข้อมูลคู่ เราก็พบข้อผิดพลาดบางประการและรวบรวมบทเรียนสำคัญที่อาจช่วยให้คุณหลีกเลี่ยงข้อผิดพลาดเดียวกันได้

### 1. ความแตกต่างในการดำเนินการพร้อมกันและธุรกรรม

PostgreSQL ใช้สถาปัตยกรรมเซิร์ฟเวอร์-ไคลเอ็นต์ และรองรับการเขียนที่มีการทำงานพร้อมกันสูงพร้อมระดับการแยกธุรกรรมที่มีประสิทธิภาพ SQLite ใช้การล็อกไฟล์ ดังนั้นการดำเนินการเขียนจะล็อกไฟล์ฐานข้อมูลทั้งหมด เว้นแต่จะเปิดใช้งานโหมด WAL

**คำแนะนำ**:
เมื่อเขียนตรรกะทางธุรกิจที่เกี่ยวข้องกับการเขียนบ่อยครั้ง เช่น การบันทึกสถานะการแก้ไขของผู้ใช้แบบเรียลไทม์ คุณต้องคำนึงถึงโมเดลการล็อกของ SQLite ด้วย เมื่อออกแบบโมดูลการทำงานร่วมกันของ OpenSpec ใน **HagiCode** เราได้แนะนำกลไก "ผสานก่อนเขียน" เพื่อลดความถี่ในการเขียนฐานข้อมูลโดยตรง ทำให้เราสามารถรักษาประสิทธิภาพที่ดีบนฐานข้อมูลทั้งสองได้

### 2. การจัดการวงจรชีวิตของสตริงการเชื่อมต่อ

การสร้างการเชื่อมต่อ PostgreSQL มีราคาค่อนข้างแพงและขึ้นอยู่กับการรวมการเชื่อมต่อ การเชื่อมต่อ SQLite มีน้ำหนักเบามาก แต่ถ้าไม่ได้เผยแพร่ทันที การล็อกไฟล์อาจทำให้การดำเนินการในภายหลังหมดเวลา

ใน `Program.cs`เราสามารถปรับแต่งพฤติกรรมของแต่ละฐานข้อมูลได้:

```csharp
if (databaseSettings?.DbType?.ToLower() == "sqlite")
{
    // SQLite: keeping connections open can improve performance, but watch out for file locks
    options.UseSqlite(connectionString, sqliteOptions =>
    {
        // Set command timeout
        sqliteOptions.CommandTimeout(30);
    });
}
else
{
    // PG: make full use of connection pooling
    options.UseNpgsql(connectionString, npgsqlOptions =>
    {
        npgsqlOptions.MaxBatchSize(100);
        npgsqlOptions.CommandTimeout(30);
    });
}
```

### 3. ความสำคัญของความครอบคลุมการทดสอบ

นักพัฒนาหลายคน รวมถึงสมาชิกในทีมของเราในยุคแรก ๆ มักจะทำข้อผิดพลาดทั่วไปอย่างหนึ่ง นั่นคือ การรันการทดสอบหน่วยเฉพาะในสภาพแวดล้อมการพัฒนา ซึ่งโดยปกติคือ SQLite

ในไปป์ไลน์ CI/CD ของ HagiCode เราได้บังคับใช้ขั้นตอน GitHub Actions เพื่อให้แน่ใจว่าทุกคำขอดึงจะทำการทดสอบการรวม PostgreSQL

```yaml
# Example snippet from .github/workflows/test.yml
- name: Run Integration Tests (PostgreSQL)
  run: |
    docker-compose up -d db_postgres
    dotnet test --filter "Category=Integration"
```

สิ่งนี้ช่วยให้เราตรวจพบข้อบกพร่องนับไม่ถ้วนที่เกี่ยวข้องกับความแตกต่างทางไวยากรณ์ SQL และความไวของตัวพิมพ์

## บทสรุป

ด้วยการแนะนำเลเยอร์นามธรรมและการฉีดการขึ้นต่อกันที่ขับเคลื่อนด้วยการกำหนดค่า เราจึงนำการตั้งค่า PostgreSQL และ SQLite แบบติดตามคู่มาใช้ในโปรเจ็กต์ **HagiCode** ได้สำเร็จ สิ่งนี้ไม่เพียงลดอุปสรรคในการเริ่มใช้งานสำหรับนักพัฒนารายใหม่ลงอย่างมากโดยขจัดความจำเป็นในการติดตั้ง PostgreSQL แต่ยังให้การรับประกันประสิทธิภาพที่แข็งแกร่งสำหรับการผลิตอีกด้วย

หากต้องการสรุปประเด็นสำคัญ:
1.  **บทคัดย่อต้องมาก่อน**: รหัสธุรกิจไม่ควรขึ้นอยู่กับการใช้งานฐานข้อมูลที่เป็นรูปธรรม
2.  **การกำหนดค่าแยกต่างหาก**: ใช้การกำหนดค่าอื่น `appsettings.json` ไฟล์สำหรับการพัฒนาและการผลิต
3.  **แยกการโยกย้าย**: อย่าพยายามทำให้ชุดการโยกย้ายชุดเดียวทำงานได้ในทุกที่
4.  **การลดคุณสมบัติลง**: จัดลำดับความสำคัญความเข้ากันได้ใน SQLite และประสิทธิภาพใน PostgreSQL
