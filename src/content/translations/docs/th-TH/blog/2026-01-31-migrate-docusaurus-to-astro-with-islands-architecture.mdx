## จาก Docusaurus 3.x ถึง Astro 5.x: การย้อนหลังเกี่ยวกับการย้ายไซต์ HagiCode

> บทความนี้จะย้อนกลับไปดูการย้ายเว็บไซต์อย่างเป็นทางการของ HagiCode จาก Docusaurus 3.x ไปเป็น Astro 5.x โดยสมบูรณ์ เราจะเจาะลึกว่าสถาปัตยกรรม Astro's Islands ช่วยเราแก้ปัญหาคอขวดด้านประสิทธิภาพได้อย่างไร ขณะเดียวกันก็รักษาแอสเซทส่วนประกอบ React ที่มีอยู่ของเรา โดยมอบการปรับปรุงทั้งความเร็วในการสร้างและประสิทธิภาพการโหลด

## พื้นหลัง

ในเดือนมกราคม ปี 2026 เราได้ดำเนินการ "ปลูกถ่ายหัวใจ" บนเว็บไซต์อย่างเป็นทางการของ HagiCode โดยการย้ายเฟรมเวิร์กหลักจาก Docusaurus 3.x ไปยัง Astro 5.x โดยสมบูรณ์ นี่ไม่ใช่การเขียนซ้ำแบบหุนหันพลันแล่น แต่เป็นการตัดสินใจทางเทคนิคที่พิจารณาอย่างรอบคอบ

ก่อนการย้ายข้อมูล ไซต์ของเราทำงานได้เสร็จสมบูรณ์แล้ว แต่เริ่มแสดง "ปัญหาหรูหรา" แบบคลาสสิกแล้ว: อาร์ติแฟกต์บิลด์ที่บวม เพย์โหลด JavaScript มากเกินไป และความเร็วในการโหลดหน้าเว็บที่น้อยกว่าอุดมคติบนหน้าเอกสารประกอบที่ซับซ้อน ในฐานะโปรเจ็กต์ผู้ช่วยเขียนโค้ด AI HagiCode ต้องการเอกสารประกอบและการอัปเดตฟีเจอร์บ่อยครั้ง ดังนั้นประสิทธิภาพในการสร้างจึงส่งผลโดยตรงต่อความเร็วในการเผยแพร่ ในเวลาเดียวกัน เราต้องการให้ไซต์เป็นมิตรกับเครื่องมือค้นหา (SEO) มากขึ้น เพื่อให้นักพัฒนาค้นพบโครงการได้มากขึ้น

เพื่อแก้ไขปัญหาเหล่านี้ เราได้ตัดสินใจอย่างกล้าหาญ: สร้างระบบทั้งหมดบน Astro ใหม่ ผลกระทบของการตัดสินใจนั้นอาจยิ่งใหญ่กว่าที่คุณคาดไว้ด้วยซ้ำ ฉันจะเข้าไปดูรายละเอียดเร็วๆ นี้

## เกี่ยวกับ ฮากิโค้ด

วิธีการโยกย้ายไซต์ที่แบ่งปันในบทความนี้มาจากประสบการณ์จริงของเราใน [ฮากิโค้ด](https://github.com/HagiCode-org/site) โครงการ

HagiCode คือผู้ช่วยเขียนโค้ด AI ที่มุ่งเน้นการปรับปรุงประสิทธิภาพการพัฒนา เราไม่เพียงแต่ใส่ใจในการวนซ้ำคุณสมบัติหลักเท่านั้น แต่ยังรวมถึงประสบการณ์ของนักพัฒนาด้วย การปรับปรุงไซต์นี้มีจุดมุ่งหมายเพื่อให้ผู้ใช้ได้รับประสบการณ์ที่เร็วที่สุดเมื่อเรียกดูเอกสารและเว็บไซต์อย่างเป็นทางการของเรา

## เหตุใดจึงละทิ้งระบบนิเวศ Docusaurus ที่เป็นผู้ใหญ่?

ภายในระบบนิเวศ React Docusaurus เป็น "คำตอบมาตรฐาน" สำหรับไซต์เอกสารมานานแล้ว มันทำงานนอกกรอบ นำเสนอระบบนิเวศปลั๊กอินที่หลากหลาย และมีชุมชนที่กระตือรือร้น แต่เมื่อ HagiCode ได้รับคุณสมบัติมากขึ้น เราก็รู้สึกถึงข้อจำกัดของมันด้วย:

1.  **ปัญหาคอขวดด้านประสิทธิภาพ**: โดยพื้นฐานแล้ว Docusaurus นั้นเป็น React SPA (แอปพลิเคชันหน้าเดียว) แม้ว่าคุณจะเขียนเฉพาะหน้าคงที่ แต่ไคลเอ็นต์ยังคงต้องโหลด React runtime และไฮเดรตของหน้า ซึ่งมีน้ำหนักมากโดยไม่จำเป็นสำหรับหน้าเอกสารทั่วไป
2.  **ขนาดเนื้อหาขนาดใหญ่**: แม้ว่าหน้าเว็บจะมีเนื้อหาน้อยมาก แต่ขนาด JS ที่รวมกลุ่มไว้ก็ยังคงค่อนข้างคงที่ ซึ่งไม่เหมาะสำหรับผู้ใช้มือถือหรือสภาพเครือข่ายที่ไม่ดี
3.  **ความยืดหยุ่นจำกัด**: แม้ว่าจะสามารถขยายได้ แต่เราต้องการการควบคุมไปป์ไลน์การสร้างในระดับต่ำมากขึ้น

แอสโตรมาถูกเวลาเพื่อแก้ไขปัญหาเหล่านี้ เปิดตัว "สถาปัตยกรรมเกาะ" ใหม่: ตามค่าเริ่มต้น Astro จะสร้าง HTML แบบคงที่โดยไม่มี JavaScript เป็นศูนย์ และมีเพียงส่วนประกอบที่ต้องการการโต้ตอบเท่านั้นที่จะ "เปิดใช้งาน" และโหลด JS นั่นหมายความว่าไซต์ส่วนใหญ่ของเรากลายเป็น HTML ล้วนๆ และโหลดได้เร็วมาก

## กลยุทธ์การย้ายถิ่นหลัก: การเปลี่ยนแปลงทางสถาปัตยกรรมที่ราบรื่น

การย้ายข้อมูลไม่ใช่แค่การคัดลอกและวาง มันจำเป็นต้องเปลี่ยนความคิด เราย้ายจากโมเดล "all React" ของ Docusaurus ไปเป็นโมเดล "Core + Islands" ของ Astro

### 1. การสร้างระบบการกำหนดค่าใหม่

ก่อนอื่นเราต้องย้ายจาก `docusaurus.config.ts` ถึง `astro.config.mjs`. นี่ไม่ใช่แค่การเปลี่ยนชื่อไฟล์ แต่เป็นการเขียนตรรกะการกำหนดเส้นทางและการสร้างใหม่

ใน Docusaurus ทุกอย่างเป็นปลั๊กอิน ใน Astro ทุกอย่างเป็นการบูรณาการ เราจำเป็นต้องกำหนดเส้นทางพื้นฐานของไซต์ใหม่ สร้างโหมดเอาต์พุต (คงที่เทียบกับ SSR) และกลยุทธ์การเพิ่มประสิทธิภาพสินทรัพย์

**ก่อนการย้าย:**
```typescript
// docusaurus.config.ts
export default {
  title: 'HagiCode',
  url: 'https://hagicode.com',
  baseUrl: '/',
  // ... more configuration
};
```

**หลังการย้าย:**
```javascript
// astro.config.mjs
import { defineConfig } from 'astro/config';
import react from '@astrojs/react';

export default defineConfig({
  integrations: [react()],
  site: 'https://hagicode.com',
  base: '/',
  // Optimization settings for static assets
  build: {
    inlineStylesheets: 'auto',
  },
});
```

### 2. สิ่งที่ควรเก็บไว้และสิ่งที่ต้องปรับโครงสร้างใหม่ในส่วนประกอบของปฏิกิริยา

นี่เป็นส่วนที่เจ็บปวดที่สุดของการย้ายถิ่นฐาน ไซต์ที่มีอยู่ของเรามีส่วนประกอบ React มากมาย เช่น แท็บ การเน้นโค้ด ปุ่มตอบรับ และอื่นๆ การทิ้งพวกมันไปจะสิ้นเปลือง แต่การเก็บทุกอย่างไว้จะทำให้เพย์โหลด JavaScript หนักเกินไป

HagiCode ใช้กลยุทธ์ **การให้ความชุ่มชื้นแบบก้าวหน้า**:
*   **ส่วนประกอบคงที่ล้วนๆ**: สำหรับเนื้อหาการนำเสนอ เช่น ส่วนหัว ส่วนท้าย และเอกสารประกอบข้อความธรรมดา เราได้เขียนเนื้อหาเหล่านั้นใหม่เป็นส่วนประกอบ Astro (`.astro` ไฟล์) และเรนเดอร์เป็น HTML โดยตรงในขณะสร้าง
*   **หมู่เกาะแบบโต้ตอบ**: สำหรับส่วนประกอบที่ต้องคงการโต้ตอบ เช่น ตัวสลับธีม การสลับแท็บ และปุ่มคัดลอกบล็อคโค้ด เราจะเก็บการใช้งาน React ไว้และเพิ่ม `client:load` หรือ `client:visible` คำสั่ง

ตัวอย่างเช่น ส่วนประกอบ Tabs ที่ใช้กันทั่วไปในเอกสารประกอบ:

```jsx
// src/components/Tabs.jsx
import { useState } from 'react';
import './Tabs.css'; // Import styles

export default function Tabs({ items }) {
  const [activeIndex, setActiveIndex] = useState(0);
  // ... state logic
  return (
    <div className="tabs-wrapper">
      {/* Rendering logic */}
    </div>
  );
}
```

เมื่อใช้ใน Markdown เราจะบอก Astro อย่างชัดเจนว่า: "ส่วนประกอบนี้ต้องการ JS"

```markdown
// src/content/docs/example.mdx
import Tabs from '../../components/Tabs.jsx';

<!-- Load JS only when the component enters the viewport -->
<Tabs client:visible items={...} />
```

ด้วยวิธีนี้ ส่วนประกอบเชิงโต้ตอบที่อยู่นอกวิวพอร์ตจะไม่แย่งแบนด์วิธ ซึ่งช่วยเพิ่มความเร็วในการโหลดหน้าจอแรกได้อย่างมาก

### 3. การปรับระบบจัดแต่งทรงผม: จากโมดูล CSS ไปจนถึง CSS ที่มีขอบเขต

Docusaurus รองรับโมดูล CSS เป็นค่าเริ่มต้น ในขณะที่ Astro สนับสนุน CSS แบบกำหนดขอบเขตผ่านทาง `<style>` แท็ก แนวคิดหลักที่อยู่เบื้องหลังทั้งสองคือการแยกสไตล์ แต่ไวยากรณ์แตกต่างกัน

ในระหว่างการโยกย้าย HagiCode เราได้แปลงโมดูล CSS ที่ซับซ้อนส่วนใหญ่ให้เป็นรูปแบบที่กำหนดขอบเขตของ Astro นี่กลับกลายเป็นเรื่องดีจริงๆ เพราะว่าใน `.astro` ไฟล์สไตล์และเทมเพลตอยู่ในไฟล์เดียวกัน ซึ่งทำให้การบำรุงรักษาง่ายขึ้น

**ก่อนการปรับโครงสร้างใหม่:**
```css
/* Tabs.module.css */
.wrapper { background: var(--ifm-background-color); }
```

**หลังการปรับโครงสร้างใหม่ (ขอบเขต Astro):**
```html
<!-- Tabs.astro -->
<div class="tabs-wrapper">
  <slot />
</div>

<style>
  .tabs-wrapper {
    /* Use CSS variables directly to adapt to the theme */
    background: var(--bg-color);
    padding: 1rem;
  }
</style>
```

ในเวลาเดียวกัน เราได้รวมระบบตัวแปร CSS ส่วนกลางเข้าด้วยกัน และใช้ความสามารถที่คำนึงถึงสภาพแวดล้อมของ Astro เพื่อให้แน่ใจว่าโหมดมืดจะสลับระหว่างหน้าต่างๆ ได้อย่างราบรื่น

## ข้อผิดพลาดที่เราพบในทางปฏิบัติและวิธีแก้ไข

ในระหว่างการย้ายข้อมูล HagiCode จริง เราพบปัญหาบางประการ ต่อไปนี้เป็นตัวอย่างทั่วไปหลายประการ

### 1. จุดปวดที่แปรผันของเส้นทางและสภาพแวดล้อม

HagiCode รองรับการปรับใช้เส้นทางย่อย เช่น การใช้งานภายใต้ไดเร็กทอรีย่อย GitHub Pages ในดอคูซอรัส `baseUrl` ได้รับการจัดการโดยอัตโนมัติ อย่างไรก็ตาม ใน Astro เราจำเป็นต้องระมัดระวังมากขึ้นในการจัดการลิงก์รูปภาพและคำขอ API

เราได้แนะนำกลไกตัวแปรสภาพแวดล้อมเพื่อจัดการสิ่งนี้อย่างต่อเนื่อง:

```javascript
// Handle paths in the build script
const getBasePath = () => import.meta.env.VITE_SITE_BASE || '/';
```

ตรวจสอบให้แน่ใจว่าไม่ได้ฮาร์ดโค้ดพา ธ ที่ขึ้นต้นด้วย `/` ในรหัสของคุณ อยู่ระหว่างการพัฒนากับการใช้งานจริงหรือหลังการกำหนดค่า `base` เส้นทาง การทำเช่นนี้อาจทำให้เกิด 404 สำหรับเนื้อหาได้

### 2. ความเข้ากันได้ของสคริปต์ CommonJS

ไซต์เก่าของเรามีสคริปต์ Node.js บางตัวที่ใช้สำหรับงานต่างๆ เช่น การดึงข้อมูล Metrics โดยอัตโนมัติและการอัปเดตแผนผังไซต์ และสคริปต์เหล่านี้เขียนด้วย CommonJS (`require`). เครื่องมือสร้าง Astro และสมัยใหม่ได้นำ ES Modules มาใช้อย่างเต็มที่ (`import`/`export`).

หากคุณมีสคริปต์ที่คล้ายกัน อย่าลืมปรับโครงสร้างสคริปต์ทั้งหมดเป็น ES Modules นั่นคือทิศทางที่ระบบนิเวศกำลังเคลื่อนไป และยิ่งคุณทำการเปลี่ยนแปลงเร็วเท่าไร ปัญหาที่ตามมาก็จะน้อยลงเท่านั้น

```javascript
// Old way
const fs = require('fs');

// New way
import fs from 'fs';
```

### 3. อย่าลืม SEO และการเปลี่ยนเส้นทาง

เครื่องมือค้นหาได้จัดทำดัชนีหน้า Docusaurus เก่าของ HagiCode แล้ว หากคุณเปลี่ยนมาใช้ Astro โดยตรงและโครงสร้าง URL เปลี่ยนไป คุณอาจได้รับ 404 จำนวนมากและอันดับการค้นหาลดลงอย่างมาก

เรากำหนดค่ากฎการเปลี่ยนเส้นทางใน Astro:

```javascript
// astro.config.mjs
export default defineConfig({
  redirects: {
    '/docs/old-path': '/docs/new-path',
    // Map old links to new links in bulk
  }
});
```

หรือคุณสามารถจัดการสิ่งนี้ได้ที่เลเยอร์การกำหนดค่าเซิร์ฟเวอร์ ตรวจสอบให้แน่ใจว่าลิงก์เก่าสามารถเปลี่ยนเส้นทาง 301 ไปยังที่อยู่ใหม่ได้ เนื่องจากนี่เป็นสิ่งสำคัญสำหรับ SEO

สำหรับ HagiCode การย้ายจาก Docusaurus ไปยัง Astro ไม่ใช่แค่การอัปเกรดเฟรมเวิร์กเท่านั้น นอกจากนี้ยังเป็นการนำไปปฏิบัติตามปรัชญา "ประสิทธิภาพต้องมาก่อน" ในทางปฏิบัติด้วย

**สิ่งที่เราได้รับ:**
*   **คะแนน Lighthouse ที่โดดเด่น**: หลังจากการโยกย้าย คะแนนประสิทธิภาพของไซต์ HagiCode ก็เข้าใกล้คะแนนสมบูรณ์แบบได้อย่างง่ายดาย
*   **ความเร็วในการสร้างที่เร็วขึ้น**: ความสามารถในการสร้างที่เพิ่มขึ้นของ Astro ช่วยลดเวลาในการเผยแพร่สำหรับการอัพเดตเอกสารลงครึ่งหนึ่ง
*   **รักษาความยืดหยุ่น**: ด้วยสถาปัตยกรรม Islands เราไม่ได้เสียสละคุณสมบัติเชิงโต้ตอบใดๆ และยังคงสามารถใช้ React ได้เมื่อจำเป็น

หากคุณดูแลไซต์ที่เน้นเอกสารเป็นหลักและกำลังประสบปัญหากับขนาดบันเดิลหรือความเร็วในการโหลด Astro ก็คุ้มค่าที่จะลองใช้ แม้ว่ากระบวนการโยกย้ายจะต้องได้รับการผ่าตัดบางอย่าง เช่น การเปลี่ยนชื่อ PCode เป็น HagiCode และการย้ายส่วนประกอบไปทีละชิ้น ประสบการณ์การใช้งานที่ราบรื่นที่คุณได้รับเป็นการตอบแทนจะทำให้คุ้มค่าอย่างยิ่ง

ระบบบิลด์ที่แชร์ในบทความนี้เป็นแนวทางที่เราพัฒนาผ่านการลองผิดลองถูกจริงในขณะที่สร้าง HagiCode หากคุณพบว่าแนวทางนี้มีคุณค่า นั่นบ่งบอกถึงความแข็งแกร่งทางวิศวกรรมของเรา และตัว HagiCode เองก็น่าจะคุ้มค่าที่จะพิจารณาอย่างใกล้ชิดเช่นกัน
