## การเรียกใช้เครื่องมือ AI CLI ในคอนเทนเนอร์ Docker: คู่มือปฏิบัติสำหรับการแยกผู้ใช้และวอลุ่มถาวร

> การรวมเครื่องมือการเขียนโค้ด AI เช่น Claude Code, Codex และ OpenCode เข้ากับสภาพแวดล้อมแบบคอนเทนเนอร์ฟังดูเป็นเรื่องง่าย แต่มีความซับซ้อนที่ซ่อนอยู่ทุกที่ บทความนี้เจาะลึกว่าโปรเจ็กต์ HagiCode แก้ปัญหาความท้าทายหลักในการปรับใช้ Docker ได้อย่างไร รวมถึงการอนุญาตผู้ใช้ ความคงอยู่ของการกำหนดค่า และการจัดการเวอร์ชัน เพื่อให้คุณสามารถหลีกเลี่ยงข้อผิดพลาดทั่วไปได้

## พื้นหลัง

เมื่อเราตัดสินใจที่จะเรียกใช้เครื่องมือ CLI การเข้ารหัส AI ภายในคอนเทนเนอร์ Docker ความคิดตามสัญชาตญาณที่สุดน่าจะเป็น: "คอนเทนเนอร์ไม่ใช่แค่รูทใช่ไหม ทำไมไม่ติดตั้งทุกอย่างโดยตรงแล้วเรียกมันว่าเสร็จแล้วล่ะ" ในความเป็นจริง แนวคิดที่ดูเรียบง่ายนั้นซ่อนปัญหาสำคัญหลายประการที่ต้องแก้ไขไว้

ประการแรก **ข้อจำกัดด้านความปลอดภัยเป็นอุปสรรคประการแรก** ยกตัวอย่าง Claude CLI: ห้ามมิให้เรียกใช้ในฐานะผู้ใช้รูทอย่างชัดเจน นี่เป็นการตรวจสอบความปลอดภัยภาคบังคับ และหากตรวจพบรูท ระบบจะไม่ยอมเริ่มทำงาน คุณอาจคิดว่า ฉันไม่สามารถเปลี่ยนผู้ใช้ด้วย `USER` คำสั่ง? มันไม่ง่ายอย่างนั้น ยังคงมีปัญหาการแมประหว่างผู้ใช้ที่ไม่ใช่รูทภายในคอนเทนเนอร์และการอนุญาตผู้ใช้บนเครื่องโฮสต์

ประการที่สอง **ความคงอยู่ของรัฐเป็นกับดักที่สอง** Claude Code จำเป็นต้องเข้าสู่ระบบ Codex มีการกำหนดค่าของตัวเอง และ OpenCode ก็มีไดเร็กทอรีแคชด้วย หากคุณต้องกำหนดค่าทุกอย่างใหม่ทุกครั้งที่รีสตาร์ทคอนเทนเนอร์ แนวคิดเรื่อง "ระบบอัตโนมัติ" ทั้งหมดจะสูญเสียความหมายไป เราต้องการให้การกำหนดค่าเหล่านี้คงอยู่เกินวงจรการใช้งานของคอนเทนเนอร์

ปัญหาที่สามคือ **ความสอดคล้องของสิทธิ์** สามารถดำเนินการภายในไฟล์การกำหนดค่าการเข้าถึงคอนเทนเนอร์ที่สร้างโดยผู้ใช้โฮสต์ได้หรือไม่ UID/GID ที่ไม่ตรงกันมักทำให้เกิดข้อผิดพลาดในการอนุญาตไฟล์ และนี่เป็นเรื่องปกติอย่างยิ่งในการปรับใช้จริง

ปัญหาเหล่านี้อาจดูเป็นอิสระ แต่ในทางปฏิบัติแล้วปัญหาเหล่านี้เชื่อมโยงกันอย่างแน่นหนา ในระหว่างการพัฒนา HagiCode เราได้ค่อยๆ คิดค้นวิธีแก้ปัญหาที่ใช้งานได้จริง ต่อไป ฉันจะแบ่งปันรายละเอียดด้านเทคนิคและบทเรียนที่ได้รับจากข้อผิดพลาดเหล่านั้น

## เกี่ยวกับ ฮากิโค้ด

วิธีแก้ปัญหาที่แบ่งปันในบทความนี้มาจากประสบการณ์จริงของเราใน [ฮากิโค้ด](https://hagicode.com) โครงการ HagiCode เป็นแพลตฟอร์มการเขียนโปรแกรมแบบโอเพ่นซอร์สที่ได้รับความช่วยเหลือจาก AI ซึ่งผสานรวมผู้ช่วยด้านการเขียนโค้ด AI กระแสหลักหลายตัว รวมถึง Claude Code, Codex และ OpenCode เนื่องจากเป็นโครงการที่ต้องการการใช้งานข้ามแพลตฟอร์มและมีความพร้อมใช้งานสูง HagiCode จึงต้องแก้ไขความท้าทายทั้งหมดที่เกี่ยวข้องกับการใช้งานแบบคอนเทนเนอร์

หากคุณพบว่าโซลูชันด้านเทคนิคในบทความนี้มีคุณค่า นั่นเป็นสัญญาณว่า HagiCode มีบางสิ่งบางอย่างที่เป็นประโยชน์ในทางปฏิบัติทางวิศวกรรม ในกรณีนั้น [เว็บไซต์อย่างเป็นทางการของ HagiCode](https://hagicode.com) และ [พื้นที่เก็บข้อมูล GitHub](https://github.com/HagiCode-org/site) น่าติดตามทั้งคู่

## ทำไมเราถึงใช้รูทไม่ได้?

มีความเข้าใจผิดที่พบบ่อยที่นี่: คอนเทนเนอร์ Docker ทำงานเป็นรูทตามค่าเริ่มต้น แล้วทำไมไม่ติดตั้งเครื่องมือเป็นรูทล่ะ? หากคุณคิดเช่นนั้น Claude CLI จะสอนคุณเป็นอย่างอื่นอย่างรวดเร็ว

```bash
# Run Claude CLI directly as root? No.
docker run --rm -it --user root myimage claude
# Output: Error: This command cannot be run as root user
```

นี่เป็นข้อจำกัดด้านความปลอดภัยที่เข้มงวดใน Claude CLI เหตุผลง่ายๆ คือ เครื่องมือ CLI เหล่านี้อ่านและเขียนการกำหนดค่าผู้ใช้ที่ละเอียดอ่อน รวมถึงโทเค็น API แคชในเครื่อง และแม้แต่สคริปต์ที่เขียนโดยผู้ใช้ การเรียกใช้งานด้วยสิทธิ์รูททำให้เกิดความเสี่ยงมากเกินไป

คำถามจึงเกิดขึ้น: เราจะปฏิบัติตามข้อกำหนดด้านความปลอดภัยของ CLI ขณะเดียวกันก็ทำให้การจัดการคอนเทนเนอร์มีความยืดหยุ่นได้อย่างไร เราจำเป็นต้องเปลี่ยนวิธีคิด: **แทนที่จะเปลี่ยนผู้ใช้ขณะรันไทม์ ให้สร้างผู้ใช้เฉพาะในระหว่างขั้นตอนการสร้างอิมเมจ**

## การสร้างผู้ใช้เฉพาะ: มากกว่าการเปลี่ยนชื่อ

คุณอาจคิดว่าการเพิ่มเพียงอย่างเดียว `USER` บรรทัดไปที่ Dockerfile ก็เพียงพอแล้ว นั่นเป็นวิธีที่ง่ายที่สุด แต่ก็ยังไม่แข็งแกร่งพอ

### การสร้างแบบคงที่เทียบกับการทำแผนที่แบบไดนามิก

แนวทางของ HagiCode คือการสร้าง `hagicode` ผู้ใช้ที่มี UID 1000 ซึ่งโดยปกติจะตรงกับผู้ใช้เริ่มต้นในเครื่องโฮสต์ส่วนใหญ่:

```dockerfile
RUN groupadd -o -g 1000 hagicode && \
    useradd -o -u 1000 -g 1000 -s /bin/bash -m hagicode && \
    mkdir -p /home/hagicode/.claude && \
    chown -R hagicode:hagicode /home/hagicode
```

แต่นี่จะแก้ปัญหาเฉพาะผู้ใช้ภายในรูปภาพเท่านั้น จะเกิดอะไรขึ้นถ้าผู้ใช้โฮสต์คือ UID 1001? คุณยังคงต้องรองรับการแมปแบบไดนามิกเมื่อคอนเทนเนอร์เริ่มทำงาน

docker-entrypoint.sh มีตรรกะที่สำคัญ:

```bash
if [ -n "$PUID" ] && [ -n "$PGID" ]; then
    if ! id hagicode >/dev/null 2>&1; then
        groupadd -g "$PGID" hagicode
        useradd -u "$PUID" -g "$PGID" -s /bin/bash -m hagicode
    fi
fi
```

ข้อดีของการออกแบบนี้ชัดเจน: **ใช้ UID 1000 เริ่มต้น ณ เวลาสร้างอิมเมจ จากนั้นปรับแบบไดนามิก ณ รันไทม์ผ่าน `PUID` และ `PGID` ตัวแปรสภาพแวดล้อม** ไม่ว่าผู้ใช้โฮสต์จะมี UID ใดก็ตาม ความเป็นเจ้าของไฟล์การกำหนดค่ายังคงถูกต้อง

## ปรัชญาการออกแบบวอลุ่มถาวร

เครื่องมือ AI CLI แต่ละรายการมีไดเร็กทอรีการกำหนดค่าที่ต้องการของตัวเอง ดังนั้นจึงจำเป็นต้องแมปทีละรายการ:

| เครื่องมือ CLI | เส้นทางในคอนเทนเนอร์ | ชื่อปริมาณ |
|---------|-----------|--------|
| คลอดด์ | `/home/hagicode/.claude` | `claude-data` |
| โคเด็กซ์ | `/home/hagicode/.codex` | `codex-data` |
| โอเพ่นโค้ด | `/home/hagicode/.config/opencode` | `opencode-config-data` |

เหตุใดจึงใช้วอลุ่มที่มีชื่อแทนการผูกเมานต์ เหตุผลสามประการ:

1. **การจัดการที่ง่ายขึ้น**: Docker จัดการวอลุ่มที่มีชื่อโดยอัตโนมัติ ดังนั้นคุณไม่จำเป็นต้องสร้างไดเร็กทอรีโฮสต์ด้วยตนเอง
2. **การแยกสิทธิ์**: เนื้อหาเริ่มต้นของวอลุ่มถูกสร้างขึ้นโดยผู้ใช้ภายในคอนเทนเนอร์ เพื่อหลีกเลี่ยงความขัดแย้งในการอนุญาตกับโฮสต์
3. **การโยกย้ายแบบอิสระ**: ไดรฟ์ข้อมูลสามารถมีอยู่ได้โดยอิสระจากคอนเทนเนอร์ ดังนั้นข้อมูลจึงไม่สูญหายเมื่อมีการอัปเกรดรูปภาพ

docker-compose-builder-web สร้างการกำหนดค่าวอลุ่มที่เกี่ยวข้องโดยอัตโนมัติ:

```yaml
volumes:
  claude-data:
  codex-data:
  opencode-config-data:

services:
  hagicode:
    volumes:
      - claude-data:/home/hagicode/.claude
      - codex-data:/home/hagicode/.codex
      - opencode-config-data:/home/hagicode/.config/opencode
    user: "${PUID:-1000}:${PGID:-1000}"
```

ให้ความสนใจกับ `user` ฟิลด์ที่นี่: `PUID` และ `PGID` ถูกฉีดผ่านตัวแปรสภาพแวดล้อมเพื่อให้แน่ใจว่ากระบวนการภายในคอนเทนเนอร์ทำงานด้วยข้อมูลประจำตัวที่ตรงกับผู้ใช้โฮสต์ รายละเอียดนี้มีความสำคัญเนื่องจากปัญหาการอนุญาตเป็นเรื่องยากที่จะแก้ไขเมื่อปรากฏขึ้น

## การจัดการเวอร์ชัน: เวอร์ชันที่ฝังมาพร้อมกับการแทนที่รันไทม์

การปักหมุดเวอร์ชันอิมเมจ Docker ถือเป็นสิ่งสำคัญสำหรับความสามารถในการทำซ้ำ แต่ในการพัฒนาจริง เรามักจะต้องทดสอบเวอร์ชันที่ใหม่กว่าหรือแก้ไขข้อบกพร่องอย่างเร่งด่วน หากเราต้องสร้างอิมเมจใหม่ทุกครั้ง ขั้นตอนการทำงานก็จะไร้ประสิทธิภาพมากเกินไป

กลยุทธ์ของ HagiCode คือ **เวอร์ชันคงที่เป็นค่าเริ่มต้น โดยมีการแทนที่รันไทม์เป็นกลไกส่วนขยาย** เป็นการประนีประนอมทางวิศวกรรมเชิงปฏิบัติระหว่างความเสถียรและความยืดหยุ่น

Dockerfile.template ปักหมุดเวอร์ชันที่นี่:

```dockerfile
USER hagicode
WORKDIR /home/hagicode

# Configure the global npm install path
RUN mkdir -p /home/hagicode/.npm-global && \
    npm config set prefix '/home/hagicode/.npm-global'

# Install CLI tools using pinned versions
RUN npm install -g @anthropic-ai/claude-code@2.1.71 && \
    npm install -g @openai/codex@0.112.0 && \
    npm install -g opencode-ai@1.2.25 && \
    npm cache clean --force
```

docker-entrypoint.sh รองรับการแทนที่รันไทม์:

```bash
install_cli_override_if_needed() {
    local package_name="$2"
    local override_version="$5"

    if [ -n "$override_version" ]; then
        gosu hagicode npm install -g "${package_name}@${override_version}"
    fi
}

# Example usage
install_cli_override_if_needed "" "@anthropic-ai/claude-code" "" "" "${CLAUDE_CODE_CLI_VERSION}"
```

ซึ่งช่วยให้คุณทดสอบเวอร์ชันใหม่ผ่านตัวแปรสภาพแวดล้อมโดยไม่ต้องสร้างอิมเมจใหม่:

```bash
docker run -e CLAUDE_CODE_CLI_VERSION=2.2.0 myimage
```

การออกแบบนี้ใช้งานได้จริงเพราะไม่มีใครอยากสร้างอิมเมจใหม่ทุกครั้งที่ทดสอบฟีเจอร์ใหม่

## การฉีดการกำหนดค่าอัตโนมัติ

นอกเหนือจากการกำหนดค่าเครื่องมือ CLI ด้วยตนเองแล้ว บางสถานการณ์ยังจำเป็นต้องมีการแทรกการกำหนดค่าอัตโนมัติ ตัวอย่างทั่วไปที่สุดคือโทเค็น API

```bash
if [ -n "$ANTHROPIC_AUTH_TOKEN" ]; then
    mkdir -p /home/hagicode/.claude
    cat > /home/hagicode/.claude/settings.json <<EOF
{
  "env": {
    "ANTHROPIC_AUTH_TOKEN": "${ANTHROPIC_AUTH_TOKEN}"
  }
}
EOF
    chown -R hagicode:hagicode /home/hagicode/.claude
fi
```

มีสองสิ่งที่สำคัญที่นี่: **ส่งข้อมูลที่ละเอียดอ่อนผ่านตัวแปรสภาพแวดล้อม แทนที่จะฮาร์ดโค้ดลงในรูปภาพ** และตรวจสอบให้แน่ใจว่าได้ตั้งค่าความเป็นเจ้าของไฟล์การกำหนดค่าอย่างถูกต้อง ไม่เช่นนั้นเครื่องมือ CLI จะไม่สามารถอ่านได้

## แนวทางปฏิบัติที่ดีที่สุดและรายการตรวจสอบข้อผิดพลาด

### ปัญหาการอนุญาตไม่ตรงกัน

นี่เป็นกับดักที่ง่ายที่สุดที่จะตกเข้าไป ผู้ใช้โฮสต์มี UID 1001 ในขณะที่คอนเทนเนอร์ใช้ 1,000 ดังนั้นไฟล์ที่สร้างในด้านหนึ่งจึงไม่สามารถเข้าถึงได้จากอีกด้านหนึ่ง

```bash
# Correct approach: make the container match the host user
docker run \
    -e PUID=$(id -u) \
    -e PGID=$(id -g) \
    myimage
```

ปัญหานี้เกิดขึ้นบ่อยมาก และอาจทำให้หงุดหงิดในครั้งแรกที่คุณพบปัญหานี้

### การกำหนดค่าหายไปหลังจากรีสตาร์ทคอนเทนเนอร์

หากคุณพบว่าตัวเองเข้าสู่ระบบอีกครั้งหลังจากรีสตาร์ททุกครั้ง ให้ตรวจสอบว่าคุณลืมติดตั้งวอลุ่มถาวรหรือไม่:

```yaml
volumes:
  - claude-data:/home/hagicode/.claude
```

ไม่มีอะไรน่าหงุดหงิดไปกว่าการตั้งค่าอย่างระมัดระวังเพียงเพื่อให้การกำหนดค่าหายไป

### วิธีที่ถูกต้องในการอัพเกรดเวอร์ชัน

อย่าวิ่ง `npm install -g` โดยตรงภายในคอนเทนเนอร์ที่ทำงานอยู่ แนวทางที่ถูกต้องคือ:

1. ตั้งค่าตัวแปรสภาพแวดล้อมเพื่อทริกเกอร์การติดตั้งแทนที่
2. หรือสร้างภาพขึ้นมาใหม่

```bash
# Option 1: runtime override
docker run -e CLAUDE_CODE_CLI_VERSION=2.2.0 myimage

# Option 2: rebuild the image
docker build -t myimage:v2 .
```

มีถนนไปโรมมากกว่าหนึ่งสาย แต่ถนนบางสายก็เรียบกว่าถนนสายอื่น

### รายการตรวจสอบการรักษาความปลอดภัย

- ส่งโทเค็น API ผ่านตัวแปรสภาพแวดล้อมแทนที่จะเขียนลงในรูปภาพ
- ตั้งค่าการอนุญาตไฟล์การกำหนดค่าเป็น `600`.
- เรียกใช้แอปพลิเคชันในฐานะผู้ใช้ที่ไม่ใช่รูทเสมอ
- อัปเดตเวอร์ชัน CLI เป็นประจำเพื่อแก้ไขช่องโหว่ด้านความปลอดภัย

การรักษาความปลอดภัยเป็นสิ่งสำคัญเสมอ แต่ความท้าทายที่แท้จริงคือการบังคับใช้ในทางปฏิบัติอย่างสม่ำเสมอ

## ขยายการรองรับเครื่องมือ CLI ใหม่

หากคุณต้องการสนับสนุนเครื่องมือ CLI ใหม่ในอนาคต มีเพียงสามขั้นตอนเท่านั้น:

1. **Dockerfile.template**: เพิ่มขั้นตอนการติดตั้ง
2. **docker-entrypoint.sh**: เพิ่มตรรกะการแทนที่เวอร์ชัน
3. **docker-compose-builder-web**: เพิ่มการแมปวอลุ่มถาวร

การออกแบบตามเทมเพลตนี้ทำให้ส่วนขยายเป็นเรื่องง่ายโดยไม่ต้องเปลี่ยนตรรกะหลัก

## บทสรุป

การเรียกใช้เครื่องมือ AI CLI ในคอนเทนเนอร์ Docker เกี่ยวข้องกับความท้าทายหลักสามประการ: **สิทธิ์ของผู้ใช้ ความคงอยู่ของการกำหนดค่า และการจัดการเวอร์ชัน** ด้วยการรวมผู้ใช้เฉพาะ การแยกชื่อ-วอลุ่ม และการแทนที่ตามตัวแปรสภาพแวดล้อม โปรเจ็กต์ HagiCode ได้สร้างสถาปัตยกรรมการปรับใช้ที่มีทั้งความปลอดภัยและยืดหยุ่น

จุดออกแบบที่สำคัญ:

- **การแยกผู้ใช้**: สร้างผู้ใช้เฉพาะในระหว่างขั้นตอนการสร้างอิมเมจ พร้อมการรองรับรันไทม์สำหรับไดนามิก `PUID`/`PGID` การทำแผนที่
- **กลยุทธ์การคงอยู่**: เครื่องมือ CLI แต่ละรายการได้รับวอลุ่มที่ระบุชื่อเป็นของตัวเอง ดังนั้นการรีสตาร์ทจึงไม่ส่งผลต่อการกำหนดค่า
- **ความยืดหยุ่นของเวอร์ชัน**: ค่าเริ่มต้นคงที่ช่วยให้มั่นใจในความสามารถในการทำซ้ำ ในขณะที่การแทนที่รันไทม์จะทำให้มีพื้นที่สำหรับการทดสอบ
- **การกำหนดค่าอัตโนมัติ**: การกำหนดค่าที่ละเอียดอ่อนสามารถฉีดได้โดยอัตโนมัติผ่านตัวแปรสภาพแวดล้อม

โซลูชันนี้ทำงานได้อย่างเสถียรในโปรเจ็กต์ HagiCode มาระยะหนึ่งแล้ว และฉันหวังว่าโซลูชันนี้จะเป็นจุดอ้างอิงที่มีประโยชน์สำหรับนักพัฒนาที่มีความต้องการคล้ายกัน
