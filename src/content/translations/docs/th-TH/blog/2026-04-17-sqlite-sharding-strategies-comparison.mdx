## SQLite Sharding ในทางปฏิบัติ: การเปรียบเทียบเชิงลึกของสามกลยุทธ์ Sharding

> เมื่อฐานข้อมูล SQLite ไฟล์เดียวประสบปัญหาคอขวดในการทำงานพร้อมกัน เราจะฝ่าฟันผ่านได้อย่างไร บทความนี้จะแชร์แนวทางการแบ่งกลุ่มย่อย SQLite สามวิธีจากโปรเจ็กต์ HagiCode ในสถานการณ์ต่างๆ ซึ่งช่วยให้คุณเข้าใจวิธีเลือกกลยุทธ์การแบ่งกลุ่มย่อยที่เหมาะสม

สวัสดีทุกคน ฉันชื่อยูคุน ผู้สร้าง HagiCode

## พื้นหลัง

เมื่อสร้างแอปพลิเคชันประสิทธิภาพสูง ฐานข้อมูล SQLite ไฟล์เดียวจะประสบปัญหาในทางปฏิบัติ เมื่อจำนวนผู้ใช้และปริมาณข้อมูลเพิ่มขึ้น ปัญหาเหล่านี้ก็เริ่มเรียงกัน:

- การดำเนินการเขียนเริ่มเข้าคิว และเวลาตอบสนองเพิ่มขึ้นอย่างเห็นได้ชัด
- ประสิทธิภาพการสืบค้นลดลงเมื่อปริมาณข้อมูลเพิ่มขึ้น
- เป็นประจำ `database is locked` ข้อผิดพลาดปรากฏขึ้นภายใต้การเข้าถึงแบบมัลติเธรด

หลายๆ คนถามโดยสัญชาตญาณว่า เราควรย้ายโดยตรงไปยัง PostgreSQL หรือ MySQL หรือไม่ ซึ่งสามารถแก้ปัญหาได้ แต่ความซับซ้อนในการปรับใช้ก็เพิ่มขึ้นอย่างรวดเร็ว มีตัวเลือกที่น้ำหนักเบากว่าหรือไม่?

คำตอบคือการแบ่งส่วน สุดท้ายแล้วปัญหาทางวิศวกรรมก็ยังควรได้รับการแก้ไขด้วยวิธีทางวิศวกรรม ด้วยการกระจายข้อมูลไปยังไฟล์ SQLite หลายไฟล์ เราสามารถปรับปรุงประสิทธิภาพการทำงานพร้อมกันและการสืบค้นได้อย่างมาก ขณะเดียวกันก็รักษาคุณลักษณะน้ำหนักเบาของ SQLite ไว้

## เกี่ยวกับ ฮากิโค้ด

แนวทางที่แบ่งปันในบทความนี้มาจากประสบการณ์เชิงปฏิบัติของเราใน [ฮากิโค้ด](https://hagicode.com) โครงการ ในฐานะโปรเจ็กต์ผู้ช่วยเขียนโค้ด AI HagiCode จำเป็นต้องจัดการกับข้อความการสนทนา สถานะคงอยู่ และบันทึกประวัติเหตุการณ์จำนวนมาก การแก้ปัญหาที่แท้จริงเหล่านี้ทำให้เราสรุปแนวทางการแบ่งส่วนข้อมูล 3 วิธีสำหรับสถานการณ์ที่แตกต่างกันได้

เครื่องมือที่ดีมีความสำคัญ แต่วิธีใช้งานจะขึ้นอยู่กับงานที่คุณต้องทำจริงๆ

ที่เก็บโค้ดของเราอยู่ที่ [github.com/HagiCode-org/site](https://github.com/HagiCode-org/site). อย่าลังเลที่จะมองลึกลงไปหากคุณสนใจ

## ภาพรวมของแนวทาง Three Sharding

หลังจากวิเคราะห์ฐานโค้ด HagiCode เราได้ระบุแนวทางการแบ่งส่วน SQLite สามวิธีสำหรับสถานการณ์ทางธุรกิจที่แตกต่างกัน:

1. **พื้นที่เก็บข้อมูลเซสชันข้อความที่แบ่งส่วน**: พื้นที่จัดเก็บสำหรับข้อความการสนทนา AI โดดเด่นด้วยการเขียนความถี่สูงและการสืบค้นแบบแยกตามเซสชัน
2. **Orleans Grain sharded storage**: การคงอยู่ของสถานะสำหรับเฟรมเวิร์กแบบกระจาย โดยมีคุณลักษณะเฉพาะคือการเข้าถึงแบบข้ามโหนดและความจำเป็นในการกำหนดเส้นทางที่กำหนด
3. **พื้นที่เก็บข้อมูลแบ่งส่วนประวัติฮีโร่**: บันทึกเหตุการณ์ในอดีตสำหรับระบบเกม โดดเด่นด้วยการจัดหาเหตุการณ์และความจำเป็นในความเข้ากันได้ของการย้ายข้อมูล

แม้ว่าสถานการณ์ทางธุรกิจจะแตกต่างกัน แต่ทั้งสามก็ใช้หลักการออกแบบหลักที่เหมือนกัน:

- **การกำหนดเส้นทางแบบกำหนด**: คำนวณชาร์ดโดยตรงจากรหัสธุรกิจ โดยไม่ต้องใช้ตารางข้อมูลเมตา
- **การเข้าถึงที่โปร่งใส**: ชั้นบนใช้อินเทอร์เฟซแบบรวมและยังคงไม่ทราบถึงชาร์ดที่ซ่อนอยู่
- **พื้นที่เก็บข้อมูลอิสระ**: แต่ละชาร์ดเป็นไฟล์ SQLite ที่เป็นอิสระอย่างสมบูรณ์
- **การเพิ่มประสิทธิภาพการทำงานพร้อมกัน**: โหมด WAL บวก `busy_timeout` ลดการโต้แย้งล็อค

หลายๆ คนถามว่า: ทำไมไม่สร้างโซลูชันการแบ่งส่วนทั่วไปขึ้นมาสักโซลูชันล่ะ นั่นเป็นคำถามที่ใช้งานได้จริง และข้อสรุปก็ตรงไปตรงมา ในทางวิศวกรรมไม่มีวิธีแก้ปัญหาแบบสากล มีเพียงวิธีเดียวที่เหมาะกับสถานการณ์ทางธุรกิจในปัจจุบันมากที่สุด ต่อไป เราจะเปรียบเทียบการใช้งานที่เป็นรูปธรรมของทั้งสามแนวทางนี้ในเชิงลึก

## การเปรียบเทียบกลยุทธ์การแบ่งกลุ่ม

### กฎการนับเศษและการตั้งชื่อ

| ด้าน | ข้อความเซสชัน | ข้าวออร์ลีนส์ | ประวัติศาสตร์ฮีโร่ |
|------|----------------|---------------|--------------|
| **จำนวนชาร์ด** | 256 (16²) | 100 | 10 |
| **กฎการตั้งชื่อ** | เลขฐานสิบหก (00-ff) | ทศนิยม (00-99) | ทศนิยม (0-9) |
| **ไดเร็กทอรีที่เก็บข้อมูล** | `DataDir/messages/` | `DataDir/orleans/grains/` | `DataDir/hero-history/` |
| **รูปแบบชื่อไฟล์** | `{shard}.db` | `grains-{shard}.db` | `{shard}.db` |

เหตุใดจำนวนชาร์ดจึงแตกต่างกันมาก? ขึ้นอยู่กับลักษณะธุรกิจ กล่าวอีกนัยหนึ่ง โมเดลจะเปลี่ยนไป เครื่องมือจะพัฒนาขึ้น และเวิร์กโฟลว์จะได้รับการอัพเกรด แต่พื้นฐานทางวิศวกรรมยังคงเหมือนเดิม ก่อนอื่นให้ทำความเข้าใจปัญหาที่คุณพยายามแก้ไขจริงๆ ก่อน

- **ข้อความเซสชัน** ใช้ส่วนแบ่งข้อมูล 256 ส่วนเนื่องจากข้อความการสนทนามีความถี่ในการเขียนสูงสุดและต้องการส่วนแบ่งข้อมูลมากขึ้นเพื่อกระจายโหลด
- **Orleans Grain** ใช้ชิ้นส่วน 100 ชิ้น ปรับสมดุลประสิทธิภาพการทำงานพร้อมกันและความซับซ้อนในการปฏิบัติงาน
- **ประวัติฮีโร่** ใช้เพียง 10 ชิ้นส่วน เนื่องจากการเขียนเหตุการณ์ในอดีตมีความถี่น้อยกว่า และต้องพิจารณาค่าใช้จ่ายในการย้ายข้อมูลด้วย

### ความแตกต่างในอัลกอริทึมการกำหนดเส้นทาง

อัลกอริธึมการกำหนดเส้นทางเป็นแกนหลักของโครงร่างการแบ่งส่วน จะกำหนดวิธีการกระจายข้อมูลข้ามส่วนต่างๆ ทั้งสามแนวทางใช้กลยุทธ์การกำหนดเส้นทางที่แตกต่างกัน:

```csharp
// Session Message: last two hexadecimal characters of the GUID
var normalized = Guid.Parse(sessionId.Value).ToString("N").ToLowerInvariant();
return normalized[^2..];  // Take the last two hexadecimal characters

// Orleans Grain: extract digits, then use the last two digits modulo shard count
var digits = ExtractDigits(grainId);  // Extract all digits
var lastTwoDigits = (digits[^2] * 10) + digits[^1];
return lastTwoDigits % shardCount;

// Hero History: modulo 10 using the ASCII value of the last character
return heroId[^1] % 10;
```

**การวิเคราะห์การออกแบบ**:

- **ข้อความเซสชัน** รหัสคือ GUID หลังจากแปลงเป็นเลขฐานสิบหกแล้ว การใช้อักขระสองตัวสุดท้ายจะทำให้มีการกระจายเท่าๆ กันทั่วทั้ง 256 ส่วน
- รหัส **Orleans Grain** มีรูปแบบไม่สอดคล้องกันและอาจมีทั้งตัวอักษรและตัวเลข ดังนั้นตัวเลขทั้งหมดจะถูกแยกออกมาก่อนที่จะใช้โมดูโล
- **ประวัติฮีโร่** ID เป็นสตริง ดังนั้นค่า ASCII ของอักขระตัวสุดท้ายจึงถูกใช้โดยตรงกับโมดูโล เป็นเรื่องง่าย แต่การกระจายตัวอาจจะสม่ำเสมอน้อยกว่า

**ประเด็นสำคัญ**: ไม่ว่าคุณจะใช้อัลกอริทึมใดก็ตาม รหัสเดียวกันจะต้องจับคู่กับชาร์ดเดียวกันเสมอ นี่เป็นหนึ่งในข้อกำหนดพื้นฐานที่สุดในระบบแบบกระจาย มิฉะนั้นข้อมูลที่ไม่สอดคล้องกันจะหลีกเลี่ยงไม่ได้ หากการกำหนดเส้นทางไม่เสถียร ความพยายามอื่นๆ ทั้งหมดจะยุบลงเหลือศูนย์

### ความแตกต่างในกลยุทธ์การเริ่มต้น

| ด้าน | ข้อความเซสชัน | ข้าวออร์ลีนส์ | ประวัติศาสตร์ฮีโร่ |
|------|----------------|---------------|--------------|
| **ระยะเวลาในการเริ่มต้น** | ขี้เกียจโหลดตามความต้องการ | การเริ่มต้นแบบขนานเต็มรูปแบบเมื่อเริ่มต้น | ขี้เกียจโหลดตามความต้องการ |
| **การควบคุมการทำงานพร้อมกัน** | `Lazy<Task>` ป้องกันการเริ่มต้นซ้ำซ้อน | `Parallel.ForEachAsync` | `Lazy<Task>` ป้องกันการเริ่มต้นซ้ำซ้อน |

**เหตุใด Orleans Grain จึงเลือกการกำหนดค่าเริ่มต้นแบบเต็มเมื่อเริ่มต้นระบบ**

เนื่องจาก Orleans เป็นเฟรมเวิร์กแบบกระจาย Grain จึงอาจถูกกำหนดเวลาให้กับโหนดใดก็ได้ หากพบว่าไฟล์ชาร์ดหายไปเฉพาะตอนรันไทม์ คำขออาจล้มเหลวได้ การเริ่มต้นแบบเต็มเมื่อเริ่มต้นระบบจะขยายเวลาเริ่มต้น แต่จะรับประกันความเสถียรของรันไทม์ การทำให้มันทำงานเป็นเพียงจุดเริ่มต้นเท่านั้น การบำรุงรักษาให้คงอยู่เป็นทักษะที่แท้จริง

**ข้อดีของการขี้เกียจโหลด**:

สำหรับข้อความเซสชันและประวัติฮีโร่ การโหลดแบบ Lazy Loading จะช่วยลดเวลาการเริ่มต้นระบบ ไฟล์และสคีมาจะถูกสร้างขึ้นเมื่อจำเป็นต้องใช้ชาร์ดจริงๆ เท่านั้น การใช้ `Lazy<Task>` ยังป้องกันสภาพการแข่งขันในระหว่างการเริ่มต้นพร้อมกัน การออกแบบดูเรียบง่าย แต่ในโครงการจริงจะช่วยประหยัดปัญหาที่ไม่จำเป็นได้มากมาย

### ลักษณะการออกแบบสคีมา

การออกแบบสคีมาของทั้งสามแนวทางสะท้อนถึงคุณลักษณะทางธุรกิจตามลำดับ:

**ข้อความเซสชัน**:
- รองรับรูปแบบการจัดหากิจกรรม (ตารางกิจกรรมพร้อมตารางสแน็ปช็อต)
- รวมตารางลูกสำหรับบล็อกเนื้อหาข้อความ (`MessageContentBlocks`)
- มีฟิลด์การบีบอัดและแฟล็กการบีบอัดเพื่อรองรับการปรับให้เหมาะสมในอนาคต

**ออร์ลีนส์เกรน**:
- การออกแบบที่เรียบง่าย: เดี่ยว `GrainState` ตาราง
- จัดเก็บสถานะเป็น JSON แบบอนุกรม
- ใช้การควบคุมการทำงานพร้อมกันในแง่ดีตาม ETag

**ประวัติฮีโร่**:
- ดัชนีการเพิ่มประสิทธิภาพการค้นหาไทม์ไลน์
- มีเอกลักษณ์เฉพาะตัว `DedupeKey` ข้อ จำกัด ป้องกันการทำซ้ำ
- รองรับประเภทและสถานะของเหตุการณ์หลายประเภท

การออกแบบเหล่านี้แสดงให้เห็นว่าการออกแบบสคีมาควรสอดคล้องกับข้อกำหนดทางธุรกิจอย่างใกล้ชิด แทนที่จะไล่ตามความทั่วไป Orleans Grain นั้นเรียบง่ายเพราะต้องการเพียงจัดเก็บสถานะซีเรียลไลซ์และไม่ต้องใช้ความสามารถในการสืบค้นที่ซับซ้อน นี่ไม่ใช่เวทย์มนต์ มันเป็นวิศวกรรม อย่ารีบด่วนตั้งชื่ออะไรให้ยิ่งใหญ่ก่อนที่จะตรวจสอบว่าจะสามารถอยู่รอดได้ภายในทีมจริงหรือไม่

### การเปรียบเทียบการกำหนดค่าการทำงานพร้อมกัน

ทั้งสามวิธีใช้การตั้งค่าการเพิ่มประสิทธิภาพการทำงานพร้อมกันของ SQLite เดียวกัน:

```sql
PRAGMA journal_mode=WAL;      -- Write-ahead logging mode
PRAGMA synchronous=NORMAL;     -- Reduce persistence overhead
PRAGMA busy_timeout=5000;      -- 5-second busy wait
PRAGMA foreign_keys=ON;        -- Foreign key constraints
```

**ข้อดีของโหมด WAL**:

โหมดบันทึกการย้อนกลับแบบดั้งเดิมทำให้เกิดการโต้แย้งการล็อกในระหว่างการเขียน ในขณะที่โหมด WAL ช่วยให้สามารถอ่านและเขียนดำเนินการไปพร้อมๆ กัน ในสถานการณ์ที่มีข้อมูลขนาดใหญ่ สิ่งนี้สามารถปรับปรุงประสิทธิภาพได้อย่างมาก นักพัฒนาหลายคนมองข้ามการตั้งค่านี้ แต่มันสำคัญมากกว่าที่พวกเขาคิด

**ข้อแลกเปลี่ยนของ. `synchronous=NORMAL`**:

การตั้งค่าให้เป็น `FULL` ให้ความปลอดภัยสูงสุดแต่ลดประสิทธิภาพลงอย่างมาก `NORMAL` สร้างสมดุลระหว่างความปลอดภัยและประสิทธิภาพ ทำให้เป็นตัวเลือกที่เหมาะสมสำหรับการใช้งานส่วนใหญ่ ไม่จำเป็นต้องคิดมากเรื่องนี้ `NORMAL` ก็เพียงพอแล้ว

## วิธีการเลือกกลยุทธ์การแบ่งส่วน

จากการวิเคราะห์แนวทางทั้งสามของ HagiCode เราสามารถสรุปเมทริกซ์การตัดสินใจต่อไปนี้ได้:

```
High-throughput scenarios -> more shards (for example, Message uses 256)
Simple maintainability   -> fewer shards (for example, Hero History uses 10)
Mostly numeric IDs       -> modulo algorithm (Orleans Grain)
Mostly GUIDs             -> hexadecimal suffix (Session Message)
String IDs               -> ASCII modulo (Hero History)
```

**กฎง่ายๆ ในการเลือกจำนวนชาร์ด**:

- น้อยเกินไป (`< 10`): การปรับปรุงการทำงานพร้อมกันอย่างจำกัด ทำให้การแบ่งส่วนมีความหมายน้อยลง
- มากเกินไป (`> 1000`): การจัดการไฟล์มีความซับซ้อนและค่าใช้จ่ายพูลการเชื่อมต่อเพิ่มขึ้น
- หลักทั่วไป: 10 ถึง 100 ชิ้นส่วนที่เหมาะกับสถานการณ์ส่วนใหญ่
- สถานการณ์การทำงานพร้อมกันที่สูงมาก: สามารถพิจารณาชิ้นส่วนได้ 256 ชิ้น

หากคุณดูแค่การสาธิต มันก็ง่ายที่จะลืมไป แต่เมื่อคุณเข้าสู่การผลิตแล้ว ต้นทุนทั้งหมดจะต้องได้รับการคำนวณอย่างรอบคอบ หลายๆ อย่างไม่ใช่เรื่องที่เป็นไปไม่ได้ เพียงแค่ไม่ได้ตั้งราคาอย่างตรงไปตรงมา

## คู่มือการปฏิบัติ

### ใช้ตัวแก้ไขเศษมาตรฐาน

```csharp
public interface IShardResolver<TId>
{
    string ResolveShardKey(TId id);
}

// Hexadecimal sharding (for GUIDs)
public class HexSuffixShardResolver : IShardResolver<string>
{
    private readonly int _suffixLength;

    public HexSuffixShardResolver(int suffixLength = 2)
    {
        _suffixLength = suffixLength;
    }

    public string ResolveShardKey(string id)
    {
        var normalized = id.Replace("-", "").ToLowerInvariant();
        return normalized[^_suffixLength..];
    }
}

// Numeric modulo sharding (for purely numeric IDs)
public class NumericModuloShardResolver : IShardResolver<long>
{
    private readonly int _shardCount;

    public NumericModuloShardResolver(int shardCount)
    {
        _shardCount = shardCount;
    }

    public string ResolveShardKey(long id)
    {
        return (id % _shardCount).ToString("D2");
    }
}
```

### รูปแบบโรงงานการเชื่อมต่อแบบครบวงจร

```csharp
public class ShardedConnectionFactory<TOptions>
{
    private readonly ConcurrentDictionary<string, Lazy<Task>> _initializationTasks = new();
    private readonly TOptions _options;
    private readonly IShardSchemaInitializer _initializer;

    public ShardedConnectionFactory(
        TOptions options,
        IShardSchemaInitializer initializer)
    {
        _options = options;
        _initializer = initializer;
    }

    public async Task<TDbContext> CreateAsync(string shardKey, CancellationToken ct)
    {
        var connectionString = BuildConnectionString(shardKey);

        // Use Lazy<Task> to prevent concurrent initialization
        var initTask = _initializationTasks.GetOrAdd(
            connectionString,
            _ => new Lazy<Task>(() => InitializeShardAsync(connectionString, ct))
        );

        await initTask.Value;
        return CreateDbContext(connectionString);
    }

    private async Task InitializeShardAsync(string connectionString, CancellationToken ct)
    {
        await _initializer.InitializeAsync(connectionString, ct);
    }

    private string BuildConnectionString(string shardKey)
    {
        var shardPath = Path.Combine(_options.BaseDirectory, $"{shardKey}.db");
        return $"Data Source={shardPath}";
    }

    private TDbContext CreateDbContext(string connectionString)
    {
        // Create the DbContext according to the specific ORM
        return Activator.CreateInstance(typeof(TDbContext), connectionString) as TDbContext;
    }
}
```

### แนวทางปฏิบัติที่ดีที่สุดสำหรับการเริ่มต้น Schema

```csharp
public class SqliteShardInitializer : IShardSchemaInitializer
{
    public async Task InitializeAsync(string connectionString, CancellationToken ct)
    {
        await using var connection = new SqliteConnection(connectionString);
        await connection.OpenAsync(ct);

        // Concurrency optimization settings
        await connection.ExecuteAsync("""
            PRAGMA journal_mode=WAL;
            PRAGMA synchronous=NORMAL;
            PRAGMA busy_timeout=5000;
            PRAGMA foreign_keys=ON;
        """);

        // Create table schema
        await connection.ExecuteAsync("""
            CREATE TABLE IF NOT EXISTS Entities (
                Id TEXT PRIMARY KEY,
                CreatedAt TEXT NOT NULL,
                UpdatedAt TEXT NOT NULL,
                Data TEXT NOT NULL,
                ETag TEXT
            );
        """);

        // Create indexes
        await connection.ExecuteAsync("""
            CREATE INDEX IF NOT EXISTS IX_Entities_CreatedAt
            ON Entities(CreatedAt DESC);

            CREATE INDEX IF NOT EXISTS IX_Entities_UpdatedAt
            ON Entities(UpdatedAt DESC);
        """);
    }
}
```

### ข้อควรพิจารณาที่สำคัญ

**1. ความเสถียรของเส้นทาง**

อัลกอริธึมการกำหนดเส้นทางต้องรับประกันว่า ID เดียวกันจะจับคู่กับชาร์ดเดียวกันเสมอ หลีกเลี่ยงการคำนวณแบบสุ่มหรือขึ้นอยู่กับเวลา และอย่าแนะนำพารามิเตอร์ที่ไม่แน่นอนในอัลกอริทึม

**2. การเลือกจำนวนชาร์ด**

ควรตัดสินใจจำนวนชิ้นส่วนในระหว่างขั้นตอนการออกแบบ การเปลี่ยนในภายหลังเป็นเรื่องยากมาก พิจารณา:
- ปริมาณการทำงานพร้อมกันในปัจจุบันและอนาคต
- ต้นทุนการจัดการของแต่ละส่วน
- ความซับซ้อนของการย้ายข้อมูล

**3. การวางแผนการย้ายถิ่น**

แนวทางประวัติฮีโร่แสดงให้เห็นถึงเส้นทางการโยกย้ายที่สมบูรณ์:

1. สร้างโครงสร้างพื้นฐานการจัดเก็บข้อมูลแบบแบ่งส่วนใหม่
2. ใช้บริการย้ายข้อมูลเพื่อคัดลอกข้อมูลจากฐานข้อมูลหลักไปยังชาร์ด
3. ตรวจสอบความเข้ากันได้ของแบบสอบถามหลังจากการย้ายข้อมูล
4. สลับเส้นทางการอ่านและเขียนไปยังชาร์ด
5. ทำความสะอาดตารางดั้งเดิมในฐานข้อมูลหลัก

ข้อกำหนดการย้ายข้อมูลในอนาคตจำเป็นต้องได้รับการพิจารณาในขณะที่ออกแบบโครงร่างการแบ่งส่วน คุยกันถูกครับ แสดงรหัสให้ฉันดู แต่รหัสเพียงอย่างเดียวไม่เพียงพอ คุณต้องมีเส้นทางการโยกย้ายที่สมบูรณ์ด้วย ความสำเร็จเพียงครั้งเดียวไม่ใช่ระบบ ความสำเร็จที่ยั่งยืนก็คือ

**4. การติดตามและการดำเนินงาน**

- ตรวจสอบการกระจายขนาดในชาร์ดเพื่อตรวจจับข้อมูลที่บิดเบือนตั้งแต่เนิ่นๆ
- ตั้งค่าการแจ้งเตือนสำหรับชาร์ดฮอตสปอตเพื่อป้องกันไม่ให้ชาร์ดเดียวกลายเป็นคอขวด
- ตรวจสอบขนาดไฟล์ WAL เป็นประจำเพื่อหลีกเลี่ยงการใช้ดิสก์มากเกินไป
- สร้างกลไกการตรวจสุขภาพชาร์ด

**5. ความครอบคลุมการทดสอบ**

- ทดสอบเงื่อนไขขอบเขต เช่น รหัสว่าง อักขระพิเศษ และรหัสที่ยาวเกินไป
- ตรวจสอบการกำหนดเส้นทางเพื่อให้แน่ใจว่า ID เดียวกันจะแมปกับชาร์ดเดียวกันเสมอ
- รันการทดสอบความเครียดในการเขียนพร้อมกันเพื่อยืนยันว่าความขัดแย้งในการล็อคลดลงอย่างมีประสิทธิภาพ
- เรียกใช้การทดสอบการย้ายข้อมูลเพื่อให้มั่นใจถึงความสมบูรณ์และความสอดคล้องของข้อมูล

## สรุป

โดยการเปรียบเทียบวิธีการแบ่งส่วนข้อมูล SQLite ทั้งสามวิธีในโครงการ HagiCode เราจะเห็นได้ว่า:

1. **ไม่มีวิธีแก้ปัญหาที่เป็นสากล**: สถานการณ์ทางธุรกิจที่แตกต่างกันต้องใช้กลยุทธ์การแบ่งส่วนที่แตกต่างกัน
2. **มีการแบ่งปันหลักการสำคัญ**: การกำหนดเส้นทางที่กำหนด การเข้าถึงที่โปร่งใส พื้นที่เก็บข้อมูลอิสระ และการเพิ่มประสิทธิภาพการทำงานพร้อมกัน
3. **การออกแบบควรเผชิญกับอนาคต**: พิจารณาเส้นทางการโยกย้ายและต้นทุนการดำเนินงาน

หากโครงการของคุณใช้ SQLite และเริ่มประสบปัญหาคอขวดในการทำงานพร้อมกัน ฉันหวังว่าบทความนี้จะให้แนวคิดที่เป็นประโยชน์แก่คุณ ไม่จำเป็นต้องรีบเร่งในการโยกย้ายไปยังฐานข้อมูลรุ่นเฮฟวี่เวท บางครั้งกลยุทธ์การแบ่งกลุ่มข้อมูลที่ถูกต้องก็เพียงพอที่จะแก้ไขปัญหาได้

แน่นอนว่าการแบ่งส่วนไม่ใช่กระสุนเงิน ก่อนที่จะเลือกกลยุทธ์การแบ่งส่วน อันดับแรกต้องแน่ใจว่า:
- คุณได้ปรับประสิทธิภาพการสืบค้นตารางเดี่ยวให้เหมาะสมแล้ว
- คุณได้เพิ่มดัชนีที่เหมาะสมแล้ว
- คุณได้เปิดใช้งานโหมด WAL แล้ว

หลังจากที่การเพิ่มประสิทธิภาพเหล่านี้เสร็จสิ้นแล้ว และปัญหาคอขวดด้านประสิทธิภาพยังคงอยู่ คุณควรพิจารณานำการแบ่งกลุ่มข้อมูลมาใช้ การทำสิ่งง่ายๆ ได้ดีนั้นเป็นความสามารถในตัวเอง

บางครั้งการทำงานเพียงครั้งเดียวก็มีความหมายมากกว่าการอธิบายสิบครั้ง จากตรงนี้ ให้ผลลัพธ์ทางวิศวกรรมพูดเพื่อตัวมันเอง
