## วิธีเผยแพร่ Steam อัตโนมัติด้วย GitHub Actions

> บทความนี้จะแชร์โซลูชันฉบับสมบูรณ์ที่เรานำมาใช้สำหรับการเผยแพร่ Steam แบบอัตโนมัติในโปรเจ็กต์ HagiCode Desktop ซึ่งครอบคลุมขั้นตอนการทำงานอัตโนมัติแบบ end-to-end จาก GitHub Release ไปยังแพลตฟอร์ม Steam รวมถึงรายละเอียดทางเทคนิคที่สำคัญ เช่น การรับรองความถูกต้อง Steam Guard และการอัปโหลด Depot หลายแพลตฟอร์ม

## พื้นหลัง

ขั้นตอนการเปิดตัวบน Steam ค่อนข้างแตกต่างจากการเผยแพร่แอปพลิเคชันทั่วไป Steam มีระบบส่งมอบการอัปเดตที่สมบูรณ์ของตัวเอง นักพัฒนาจำเป็นต้องใช้ SteamCMD เพื่ออัปโหลดบิลด์อาร์ติแฟกต์ไปยังเครือข่าย CDN ของ Steam แทนที่จะปล่อยลิงก์ดาวน์โหลดเหมือนบนแพลตฟอร์มอื่น ๆ

โปรเจ็กต์เดสก์ท็อป HagiCode กำลังเตรียมการวางจำหน่ายบน Steam ซึ่งนำเสนอความท้าทายใหม่บางประการในขั้นตอนการเปิดตัวของเรา:

1. เราจำเป็นต้องแปลงอาร์ติแฟกต์บิลด์ที่มีอยู่ให้เป็นรูปแบบที่เข้ากันได้กับ Steam
2. เราต้องอัปโหลดไปยัง Steam ผ่าน SteamCMD
3. เรายังต้องจัดการการรับรองความถูกต้องของ Steam Guard ด้วย
4. เราจำเป็นต้องรองรับการอัพโหลด Depot หลายแพลตฟอร์มสำหรับ Linux, Windows และ macOS
5. เราต้องการโฟลว์อัตโนมัติเต็มรูปแบบจาก GitHub Release ไปยัง Steam

โครงการได้ใช้ "โหมดเวอร์ชันพกพา" ซึ่งช่วยให้แอปพลิเคชันสามารถตรวจจับเพย์โหลดบริการคงที่ที่บรรจุอยู่ใน `extra` ไดเรกทอรี เป้าหมายของเราคือการผสานรวมโหมดเวอร์ชันพกพานั้นเข้ากับการจัดจำหน่าย Steam ได้อย่างราบรื่น

## เกี่ยวกับ ฮากิโค้ด

วิธีแก้ปัญหาที่แบ่งปันในบทความนี้มาจากประสบการณ์จริงของเราใน [ฮากิโค้ด](https://hagicode.com) โครงการ HagiCode เป็นตัวช่วยเขียนโค้ด AI ที่รองรับการใช้งานเดสก์ท็อป เนื่องจากเรากำลังดำเนินการอย่างกระตือรือร้นเพื่อเปิดตัวบน Steam เราจึงจำเป็นต้องสร้างขั้นตอนการทำงานการเผยแพร่อัตโนมัติที่เชื่อถือได้

## การออกแบบสถาปัตยกรรม

แกนหลักของกระบวนการวางจำหน่าย Steam ทั้งหมดคือเวิร์กโฟลว์ GitHub Actions ที่แบ่งกระบวนการออกเป็นสามขั้นตอนหลัก:

```
┌─────────────────────────────────────────────────────────────┐
│ GitHub Actions Workflow (Steam Release)                    │
├─────────────────────────────────────────────────────────────┤
│ 1. Preparation Stage:                                      │
│    - Check out portable-version code                       │
│    - Download build artifacts from GitHub Release          │
│    - Extract and prepare the Steam content directory       │
│                                                             │
│ 2. SteamCMD Setup:                                         │
│    - Install or reuse SteamCMD                             │
│    - Authenticate with Steam Guard                         │
│                                                             │
│ 3. Release Stage:                                          │
│    - Generate Depot VDF configuration files                │
│    - Generate App Build VDF configuration files            │
│    - Invoke SteamCMD to upload to Steam                    │
└─────────────────────────────────────────────────────────────┘
```

การออกแบบนี้มีข้อดีหลายประการ:
- โดยจะนำอาร์ติแฟกต์ GitHub Release ที่มีอยู่กลับมาใช้ใหม่และหลีกเลี่ยงการสร้างเอาต์พุตเดียวกันขึ้นมาใหม่
- ใช้นักวิ่งที่โฮสต์เองเพื่อการแยกตัวอย่างปลอดภัย
- รองรับการสลับระหว่างโหมดแสดงตัวอย่างและสาขาการเผยแพร่อย่างเป็นทางการ
- รวมถึงการจัดการและการบันทึกข้อผิดพลาดที่สมบูรณ์ ซึ่งทำให้การวินิจฉัยความล้มเหลวง่ายขึ้น

## การดำเนินการเวิร์กโฟลว์

### การออกแบบพารามิเตอร์ทริกเกอร์

ขั้นตอนการทำงานของเรารองรับพารามิเตอร์หลักต่อไปนี้:

```yaml
inputs:
  release:           # Portable Version release tag
    description: 'Version tag to release (for example v1.0.0)'
    required: true
  steam_preview:     # Whether to generate a preview build
    description: 'Whether to use preview mode'
    required: false
    default: 'false'
  steam_branch:      # Steam branch to set live
    description: 'Target Steam branch'
    required: false
    default: 'preview'
  steam_description: # Build description override
    description: 'Build description'
    required: false
```

### การกำหนดค่านักวิ่งที่โฮสต์ด้วยตนเอง

ด้วยเหตุผลด้านความปลอดภัย เราใช้นักวิ่งที่โฮสต์เองกับ `steam` ป้ายกำกับ:

```yaml
runs-on:
  - self-hosted
  - Linux
  - X64
  - steam
```

สิ่งนี้ทำให้แน่ใจได้ว่าการวางจำหน่าย Steam จะทำงานบนตัวรันเฉพาะและแยกข้อมูลรับรองที่ละเอียดอ่อนไว้อย่างปลอดภัย

### การควบคุมภาวะพร้อมกัน

เพื่อป้องกันไม่ให้การเผยแพร่เวอร์ชันเดียวกันรบกวนซึ่งกันและกัน เราได้กำหนดค่าการควบคุมการทำงานพร้อมกัน:

```yaml
concurrency:
  group: portable-version-steam-${{ github.event.inputs.release }}
  cancel-in-progress: false
```

สังเกตว่า `cancel-in-progress: false` ถูกตั้งค่าไว้ที่นี่เนื่องจากการวางจำหน่าย Steam อาจใช้เวลาสักครู่ และเราไม่ต้องการให้เรียกใช้งานใหม่เพื่อยกเลิกรายการที่กำลังอัปโหลดอยู่แล้ว

## การใช้งานสคริปต์หลัก

### การเตรียมอินพุตที่วางจำหน่าย

ที่ `prepare-steam-release-input.mjs` script มีหน้าที่เตรียมอินพุตที่จำเป็นสำหรับการเปิดตัว:

```javascript
// Download the GitHub Release build manifest and artifact inventory
const buildManifest = await downloadBuildManifest(releaseTag);
const artifactInventory = await downloadArtifactInventory(releaseTag);

// Download compressed archives for each platform
for (const platform of ['linux-x64', 'win-x64', 'osx-universal']) {
  const artifactUrl = getArtifactUrl(artifactInventory, platform);
  await downloadArtifact(artifactUrl, platform);
}

// Extract into the Steam content directory structure
await extractToSteamContent(sources, contentRoot);
```

### การรับรองความถูกต้อง Steam Guard

Steam ต้องการให้บัญชีได้รับการปกป้องด้วย Steam Guard ดังนั้นเราจึงปรับใช้อัลกอริธึมการสร้างโค้ดที่อิงความลับร่วมกัน:

```javascript
function generateSteamGuardCode(sharedSecret, timestamp = Date.now()) {
  const secret = decodeSharedSecret(sharedSecret);
  const time = Math.floor(timestamp / 1000 / 30);

  const timeBuffer = Buffer.alloc(8);
  timeBuffer.writeBigUInt64BE(BigInt(time));

  // Use HMAC-SHA1 to generate a time-based one-time code
  const hash = crypto.createHmac('sha1', secret)
    .update(timeBuffer)
    .digest();

  // Convert it into a 5-character Steam Guard code
  const code = steamGuardCode(hash);
  return code;
}
```

การใช้งานนี้เป็นไปตามกลไก TOTP (รหัสผ่านครั้งเดียวตามเวลา) ของ Steam Guard โดยจะสร้างรหัสยืนยันใหม่ทุกๆ 30 วินาที

### การสร้างการกำหนดค่า VDF

VDF (Valve Data Format) คือรูปแบบการกำหนดค่าที่ Steam ใช้ เราจำเป็นต้องสร้างไฟล์ VDF สองประเภท:

**Depot VDF** ใช้เพื่อกำหนดค่าเนื้อหาสำหรับแต่ละแพลตฟอร์ม:

```javascript
function buildDepotVdf(depotId, contentRoot) {
  return [
    '"DepotBuildConfig"',
    '{',
    `  "DepotID" "${escapeVdf(depotId)}"`,
    `  "ContentRoot" "${escapeVdf(contentRoot)}"`,
    '  "FileMapping"',
    '  {',
    '    "LocalPath" "*"',
    '    "DepotPath" "."',
    '    "recursive" "1"',
    '  }',
    '}'
  ].join('\n');
}
```

**App Build VDF** ใช้เพื่อกำหนดค่าบิลด์แอปพลิเคชันทั้งหมด:

```javascript
function buildAppBuildVdf(appId, depotBuilds, description, setLive) {
  const vdf = [
    '"appbuild"',
    '{',
    `  "appid" "${appId}"`,
    `  "desc" "${escapeVdf(description)}"`,
    `  "contentroot" "${escapeVdf(contentRoot)}"`,
    '  "buildoutput" "build_output"',
    '  "depots"',
    '  {'
  ];

  for (const [depotId, depotVdfPath] of Object.entries(depotBuilds)) {
    vdf.push(`    "${depotId}" "${depotVdfPath}"`);
  }

  if (setLive) {
    vdf.push(`  }`);
    vdf.push(`  "setlive" "${setLive}"`);
  }

  vdf.push('}');
  return vdf.join('\n');
}
```

### การร้องขอ SteamCMD

สุดท้ายนี้ การอัปโหลดจะดำเนินการโดยการเรียกใช้ SteamCMD:

```javascript
await runCommand(steamcmdPath, [
  '+login', steamUsername, steamPassword, steamGuardCode,
  '+run_app_build', appBuildPath,
  '+quit'
]);
```

นี่เป็นการกระโดดครั้งสุดท้ายในเวิร์กโฟลว์ทั้งหมด เมื่อทำสำเร็จแล้วการปลดปล่อยก็เสร็จสิ้น

## การจัดการคลังสินค้าหลายแพลตฟอร์ม

Steam ใช้ระบบดีโปเพื่อจัดการเนื้อหาสำหรับแพลตฟอร์มต่างๆ เราสนับสนุนดีโปหลักสามแห่ง:

| แพลตฟอร์ม | ตัวระบุคลังสินค้า | การสนับสนุนสถาปัตยกรรม |
|-----|-----------|---------|
| ลินุกซ์ | `linux-x64` | x64_64 |
| หน้าต่าง | `win-x64` | x64_64 |
| ระบบปฏิบัติการ macOS | `osx-universal` | สากล, x64_64, arm64 |

แต่ละ Depot มีไดเร็กทอรีเนื้อหาและไฟล์การกำหนดค่า VDF ของตัวเอง สิ่งนี้ทำให้แน่ใจได้ว่าผู้ใช้บนแพลตฟอร์มที่แตกต่างกันจะดาวน์โหลดเฉพาะเนื้อหาที่พวกเขาต้องการจริงๆ เท่านั้น

## ปล่อยเวิร์กโฟลว์

### ขั้นตอนที่ 1: เตรียมการเปิดตัว GitHub

ขั้นแรก สร้าง GitHub Release ในไฟล์ `portable-version` พื้นที่เก็บข้อมูลที่ประกอบด้วย:
- ไฟล์บีบอัดสำหรับแต่ละแพลตฟอร์ม
- สร้างรายการ (`{tag}.build-manifest.json`)
- สินค้าคงคลังสิ่งประดิษฐ์ (`{tag}.artifact-inventory.json`)

### ขั้นตอนที่ 2: ทริกเกอร์เวิร์กโฟลว์การวางจำหน่าย Steam

ทริกเกอร์เวิร์กโฟลว์ด้วยตนเองผ่าน GitHub Actions และกรอกพารามิเตอร์ที่จำเป็น:
- `release`: แท็กเวอร์ชันที่จะเผยแพร่ เช่น `v1.0.0`
- `steam_branch`: สาขาเป้าหมาย เช่น `preview` หรือ `public`
- `steam_preview`: จะใช้โหมดดูตัวอย่างหรือไม่

### ขั้นตอนที่ 3: ดำเนินการขั้นตอนการเผยแพร่โดยอัตโนมัติ

เวิร์กโฟลว์จะดำเนินการตามขั้นตอนต่อไปนี้โดยอัตโนมัติ:

1. ดาวน์โหลดและแยกสิ่งประดิษฐ์ GitHub Release
2. ติดตั้งหรืออัปเดต SteamCMD
3. สร้างไฟล์การกำหนดค่า Steam VDF
4. ตรวจสอบสิทธิ์กับ Steam Guard
5. อัปโหลดเนื้อหาไปยัง Steam CDN
6. ตั้งค่าสาขาที่ระบุสด

เมื่อลำดับนี้เสร็จสิ้น เส้นทางการเผยแพร่ทั้งหมดจะถูกครอบคลุม

## คู่มือการกำหนดค่า

### การกำหนดค่าความลับที่จำเป็น

กำหนดค่าข้อมูลลับต่อไปนี้ในการตั้งค่าที่เก็บ GitHub:

| ชื่อลับ | คำอธิบาย |
|-----------|------|
| `STEAM_USERNAME` | ชื่อผู้ใช้บัญชี Steam |
| `STEAM_PASSWORD` | รหัสผ่านบัญชี Steam |
| `STEAM_SHARED_SECRET` | Steam Guard แบ่งปันความลับ (ไม่บังคับ) |
| `STEAM_GUARD_CODE` | รหัส Steam Guard (ไม่จำเป็น) |
| `STEAM_APP_ID` | รหัสแอปพลิเคชัน Steam |
| `STEAM_DEPOT_ID_LINUX` | รหัสดีโปลินุกซ์ |
| `STEAM_DEPOT_ID_WINDOWS` | รหัสวินโดวส์ดีโป |
| `STEAM_DEPOT_ID_MACOS` | รหัส macOS Depot |

ไม่มีอะไรผิดปกติเป็นพิเศษเกี่ยวกับการตั้งค่าเหล่านี้ คุณเพียงแค่ต้องมีค่าที่คาดหวังทั้งหมดไว้

### การกำหนดค่าตัวแปรสภาพแวดล้อม

| ชื่อตัวแปร | คำอธิบาย | ค่าเริ่มต้น |
|---------|------|--------|
| `PORTABLE_VERSION_STEAMCMD_ROOT` | ไดเรกทอรีการติดตั้ง SteamCMD | `~/.local/share/portable-version/steamcmd` |

## แนวทางปฏิบัติที่ดีที่สุด

### การจัดการการรับรองความถูกต้อง Steam Guard

ในการรันครั้งแรก คุณจะต้องป้อนรหัส Steam Guard ด้วยตนเอง หลังจากนั้น ขอแนะนำให้กำหนดค่าความลับที่ใช้ร่วมกันเพื่อให้สามารถสร้างโค้ดได้โดยอัตโนมัติ วิธีนี้จะหลีกเลี่ยงการแทรกแซงด้วยตนเองในทุก ๆ การเปิดตัว

SteamCMD จะบันทึกโทเค็นการเข้าสู่ระบบและสามารถนำมาใช้ซ้ำได้ในการรันในภายหลัง คุณยังคงต้องจับตาดูการหมดอายุของโทเค็น เนื่องจากต้องมีการตรวจสอบสิทธิ์อีกครั้งหลังจากที่โทเค็นหมดอายุ

### โครงสร้างไดเร็กทอรีเนื้อหา

ตรวจสอบให้แน่ใจว่าโครงสร้างไดเรกทอรีเนื้อหา Steam ถูกต้อง:

```
steam-content/
├── linux-x64/     # Linux platform content
├── win-x64/       # Windows platform content
└── osx-universal/ # macOS universal binary content
```

แต่ละไดเร็กทอรีควรมีไฟล์แอปพลิเคชันที่สมบูรณ์สำหรับแพลตฟอร์มที่เกี่ยวข้อง

### การใช้โหมดดูตัวอย่าง

โหมดดูตัวอย่างไม่ได้ตั้งค่าสาขาใดที่ใช้งานอยู่ ดังนั้นจึงเหมาะสำหรับการทดสอบและการตรวจสอบ:

```yaml
if [ "$STEAM_PREVIEW_INPUT" = 'true' ]; then
  cmd+=(--preview)
fi
```

ซึ่งจะช่วยให้คุณอัปโหลดไปยัง Steam ก่อนเพื่อตรวจสอบ จากนั้นจึงสลับไปที่สาขาอย่างเป็นทางการหลังจากที่ทุกอย่างเสร็จสิ้นแล้ว

### การจัดการและการบันทึกข้อผิดพลาด

สคริปต์ประกอบด้วยการจัดการข้อผิดพลาดและการบันทึกที่สมบูรณ์:
- ตรวจสอบว่ามี GitHub Release อยู่
- ตรวจสอบไฟล์ข้อมูลเมตาที่จำเป็น
- ตรวจสอบให้แน่ใจว่ามีเนื้อหาแพลตฟอร์มอยู่
- สร้างรายงานสรุปการดำเนินการ GitHub

ข้อมูลนี้มีคุณค่าอย่างมากสำหรับทั้งการดีบักและการตรวจสอบ

### การจัดการสิ่งประดิษฐ์

เวิร์กโฟลว์สร้างอาร์ติแฟกต์สองประเภท:
- `portable-steam-release-preparation-{tag}`: ข้อมูลเมตาการเตรียมการเผยแพร่
- `portable-steam-build-metadata-{tag}`: ข้อมูลเมตาบิลด์ Steam

อาร์ติแฟกต์เหล่านี้สามารถใช้เพื่อการตรวจสอบและการดีบักในภายหลัง ระยะเวลาการเก็บรักษา 30 วันถือเป็นค่าเริ่มต้นในทางปฏิบัติ

## การใช้งานจริง

ในโครงการ HagiCode เวิร์กโฟลว์การเผยแพร่อัตโนมัตินี้ทำงานได้สำเร็จในหลายเวอร์ชันแล้ว เส้นทางทั้งหมดจาก GitHub Release ไปยังแพลตฟอร์ม Steam นั้นเป็นไปโดยอัตโนมัติโดยสมบูรณ์ และไม่จำเป็นต้องมีการแทรกแซงด้วยตนเอง

สิ่งนี้ช่วยปรับปรุงทั้งประสิทธิภาพการเปิดตัวและความน่าเชื่อถืออย่างมีนัยสำคัญ ในอดีต การเผยแพร่เวอร์ชันหนึ่งด้วยตนเองใช้เวลานานกว่า 30 นาที ตอนนี้กระบวนการทั้งหมดเสร็จสิ้นในเวลาเพียงไม่กี่นาที

ที่สำคัญกว่านั้น กระบวนการทำงานแบบอัตโนมัติช่วยลดโอกาสที่จะเกิดข้อผิดพลาดจากมนุษย์ ทุกการเปิดตัวเป็นไปตามกระบวนการที่เป็นมาตรฐานเดียวกัน ซึ่งทำให้สามารถคาดเดาผลลัพธ์ได้มากขึ้น

## สรุป

ด้วยแนวทางที่อธิบายไว้ในบทความนี้ เราจึงได้:
1. ระบบอัตโนมัติเต็มรูปแบบตั้งแต่ GitHub Release ไปจนถึงแพลตฟอร์ม Steam
2. การอัปโหลด Depot หลายแพลตฟอร์ม
3. การรับรองความถูกต้องที่ปลอดภัยโดยใช้ Steam Guard
4. การสลับระหว่างโหมดดูตัวอย่างและการเปิดตัวอย่างเป็นทางการได้อย่างยืดหยุ่น
5. การจัดการและการบันทึกข้อผิดพลาดเสร็จสมบูรณ์

โซลูชันนี้ไม่เพียงแต่เหมาะสำหรับโปรเจ็กต์ HagiCode เท่านั้น แต่ยังสามารถใช้เป็นข้อมูลอ้างอิงสำหรับโปรเจ็กต์อื่น ๆ ที่วางแผนจะเปิดตัวบน Steam ได้อีกด้วย หากคุณกำลังพิจารณาที่จะปล่อย Steam โดยอัตโนมัติ ฉันหวังว่าประสบการณ์เชิงปฏิบัตินี้จะเป็นประโยชน์สำหรับคุณ

เทคโนโลยีอาจรู้สึกซับซ้อนหรือเรียบง่ายขึ้นอยู่กับเส้นทางที่คุณเลือก สิ่งสำคัญคือการหาขั้นตอนการทำงานที่เหมาะกับความต้องการของคุณ

หากบทความนี้ช่วยคุณได้ คุณสามารถติดดาวพื้นที่เก็บข้อมูล HagiCode GitHub หรือเยี่ยมชมเว็บไซต์อย่างเป็นทางการเพื่อเรียนรู้เพิ่มเติม
