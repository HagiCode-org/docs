## การเปิดเผยข้อมูลแบบก้าวหน้า: การปรับปรุงปฏิสัมพันธ์ระหว่างมนุษย์กับคอมพิวเตอร์ในผลิตภัณฑ์ AI ด้วยปรัชญา "Less Is More"

> ในการออกแบบผลิตภัณฑ์ AI คุณภาพของอินพุตของผู้ใช้มักจะเป็นตัวกำหนดคุณภาพของเอาต์พุต บทความนี้แบ่งปันแนวทางการโต้ตอบ "การเปิดเผยข้อมูลแบบก้าวหน้า" ที่เราปฏิบัติในโครงการ HagiCode ด้วยคำแนะนำทีละขั้นตอน การกรอกข้อมูลอย่างชาญฉลาด และการตอบรับทันที จะเปลี่ยนข้อมูลที่สั้นและคลุมเครือของผู้ใช้ให้เป็นข้อเสนอทางเทคนิคที่มีโครงสร้าง ซึ่งช่วยปรับปรุงประสิทธิภาพการโต้ตอบระหว่างมนุษย์กับคอมพิวเตอร์ได้อย่างมาก

## พื้นหลัง

ใครก็ตามที่สร้างผลิตภัณฑ์ AI อาจเคยเห็นสถานการณ์นี้: ผู้ใช้เปิดแอปของคุณและพิมพ์คำขอบรรทัดเดียวอย่างกระตือรือร้น เพื่อให้ AI ส่งคืนสิ่งที่ผิดเป้าหมายโดยสิ้นเชิง ไม่ใช่ว่า AI ไม่ฉลาดพอ ผู้ใช้ไม่ได้ให้ข้อมูลที่เพียงพอ การอ่านใจเป็นเรื่องยากสำหรับทุกคน

ปัญหานี้เห็นได้ชัดเจนเป็นพิเศษในขณะที่เรากำลังสร้าง HagiCode HagiCode เป็นตัวช่วยเขียนโค้ดที่ขับเคลื่อนด้วย AI ซึ่งผู้ใช้อธิบายข้อกำหนดในภาษาธรรมชาติเพื่อสร้างข้อเสนอทางเทคนิคและเซสชัน ในการใช้งานจริง เราพบว่าข้อมูลที่ผู้ใช้ป้อนมักมีปัญหาเหล่านี้:

- **คุณภาพการป้อนข้อมูลไม่สอดคล้องกัน**: ผู้ใช้บางคนพิมพ์เพียงไม่กี่คำ เช่น "เพิ่มประสิทธิภาพการเข้าสู่ระบบ" หรือ "แก้ไขข้อบกพร่อง" โดยไม่มีบริบทที่จำเป็น
- **คำศัพท์ทางเทคนิคที่ไม่สอดคล้องกัน**: ผู้ใช้ที่แตกต่างกันใช้คำที่แตกต่างกันสำหรับสิ่งเดียวกัน บางคนพูดว่า "ส่วนหน้า" ในขณะที่บางคนพูดว่า "FE"
- **ไม่มีข้อมูลที่มีโครงสร้าง**: ไม่มีความเป็นมาของโครงการ ขอบเขตพื้นที่เก็บข้อมูล หรือขอบเขตผลกระทบ แม้ว่าสิ่งเหล่านี้จะเป็นรายละเอียดที่สำคัญก็ตาม
- **ปัญหาที่เกิดซ้ำ**: คำขอประเภทเดียวกันปรากฏขึ้นซ้ำแล้วซ้ำอีก และทุกครั้งที่ต้องอธิบายใหม่ตั้งแต่ต้น

ผลลัพธ์โดยตรงเป็นสิ่งที่คาดเดาได้: AI เข้าใจคำขอได้ยากขึ้น คุณภาพของข้อเสนอไม่เสถียร และประสบการณ์ผู้ใช้ก็แย่ลง ผู้ใช้คิดว่า "AI นี้ไม่ดีมาก" ในขณะที่เรารู้สึกว่าถูกตำหนิอย่างไม่ยุติธรรม ถ้าให้ฉันแค่ประโยคเดียว ฉันจะเดาได้อย่างไรว่าคุณต้องการอะไรจริงๆ?

ความจริงเรื่องนี้เป็นที่เข้าใจได้ แม้แต่ผู้คนก็ยังต้องการเวลาเพื่อทำความเข้าใจซึ่งกันและกัน และเครื่องจักรก็ไม่มีข้อยกเว้น

เพื่อแก้ไขปัญหาเหล่านี้ เราได้ตัดสินใจอย่างกล้าหาญ: แนะนำหลักการออกแบบ "การเปิดเผยข้อมูลแบบก้าวหน้า" เพื่อปรับปรุงปฏิสัมพันธ์ระหว่างมนุษย์กับคอมพิวเตอร์ การเปลี่ยนแปลงที่เกิดขึ้นนี้อาจยิ่งใหญ่กว่าที่คุณจะจินตนาการได้ พูดตามตรง เราไม่ได้คาดหวังว่ามันจะได้ผลขนาดนี้ในขณะนั้น

## การเปิดเผยข้อมูลแบบก้าวหน้าคืออะไร

"การเปิดเผยข้อมูลแบบก้าวหน้า" เป็นหลักการออกแบบจากสาขา HCI (ปฏิสัมพันธ์ระหว่างมนุษย์กับคอมพิวเตอร์) แนวคิดหลักนั้นเรียบง่าย: **อย่าแสดงข้อมูลและตัวเลือกทั้งหมดแก่ผู้ใช้ในคราวเดียว แต่ให้เปิดเผยเฉพาะสิ่งที่จำเป็นทีละขั้นตอน โดยพิจารณาจากการกระทำและความต้องการของผู้ใช้**

หลักการนี้เหมาะอย่างยิ่งสำหรับผลิตภัณฑ์ AI เนื่องจากการโต้ตอบของ AI นั้นมีความก้าวหน้าตามธรรมชาติ ผู้ใช้พูดเพียงเล็กน้อย AI เข้าใจเพียงเล็กน้อย จากนั้นผู้ใช้ก็เพิ่มมากขึ้น และ AI ก็เข้าใจมากขึ้น มันคล้ายกันมากกับวิธีที่ผู้คนสื่อสารกัน: ความเข้าใจมักจะพัฒนาทีละน้อย

ในสถานการณ์ของ HagiCode เราใช้การเปิดเผยข้อมูลแบบก้าวหน้าในสี่ด้าน:

### 1. กลไกการเพิ่มประสิทธิภาพคำอธิบาย: ให้ AI ช่วยคุณพูดสิ่งต่าง ๆ ได้ชัดเจนยิ่งขึ้น

เมื่อผู้ใช้ป้อนคำอธิบายสั้น ๆ เราจะไม่ส่งคำอธิบายนั้นโดยตรงไปยัง AI เพื่อตีความ แต่ก่อนอื่น เราจะทริกเกอร์โฟลว์ "การเพิ่มประสิทธิภาพคำอธิบาย" แทน แกนหลักของโฟลว์นี้คือ "เอาต์พุตที่มีโครงสร้าง": การแปลงข้อความอิสระของผู้ใช้ให้เป็นรูปแบบมาตรฐาน มันเหมือนกับการร้อยไข่มุกหลวมๆ ลงในสร้อยคอ ทุกอย่างจึงเข้าใจได้ง่ายขึ้น

คำอธิบายที่ปรับให้เหมาะสมจะต้องมีส่วนมาตรฐานต่อไปนี้:

- **ความเป็นมา**: ความเป็นมาของปัญหาและบริบท
- **การวิเคราะห์**: การวิเคราะห์ทางเทคนิคและการใช้เหตุผล
- **แนวทางแก้ไข**: แนวทางแก้ไขและขั้นตอนการใช้งาน
- **แนวทางปฏิบัติ**: ตัวอย่างและหมายเหตุโค้ดที่เป็นรูปธรรม

ในเวลาเดียวกัน เราสร้างตาราง Markdown โดยอัตโนมัติเพื่อแสดงข้อมูล เช่น พื้นที่เก็บข้อมูลเป้าหมาย เส้นทาง และสิทธิ์ในการแก้ไข ทำให้การทำงานของ AI ในภายหลังง่ายขึ้น ไดเร็กทอรีที่ชัดเจนจะทำให้การค้นหาสิ่งต่างๆ ง่ายขึ้นเสมอ

นี่คือการใช้งานจริง:

```csharp
// Core method in ProposalDescriptionMemoryService.cs
public async Task<string> OptimizeDescriptionAsync(
    string title,
    string description,
    string locale = "zh-CN",
    DescriptionOptimizationMemoryContext? memoryContext = null,
    CancellationToken cancellationToken = default)
{
    // Build query parameters
    var queryContext = BuildQueryContext(title, description);

    // Retrieve historical context
    var memoryContext = await RetrieveHistoricalContextAsync(queryContext, cancellationToken);

    // Generate a structured prompt
    var prompt = await BuildOptimizationPromptAsync(
        title,
        description,
        memoryContext,
        cancellationToken);

    // Call AI for optimization
    return await _aiService.CompleteAsync(prompt, cancellationToken);
}
```

หัวใจสำคัญของโฟลว์นี้คือ "การฉีดหน่วยความจำ" เราแทรกบริบททางประวัติศาสตร์ เช่น แบบแผนโครงการ กรณีที่คล้ายกัน และรูปแบบเชิงลบลงในข้อความแจ้ง ช่วยให้ AI สามารถอ้างอิงประสบการณ์ในอดีตในระหว่างการปรับให้เหมาะสม ประสบการณ์ไม่ควรสูญเปล่า

**หมายเหตุ:**
- ตรวจสอบให้แน่ใจว่าอินพุตปัจจุบันมีลำดับความสำคัญเหนือหน่วยความจำในอดีต ดังนั้นข้อมูลผู้ใช้ที่ระบุอย่างชัดเจนจะไม่ถูกแทนที่
- การอ้างอิง HagIndex จะต้องถือเป็นแหล่งข้อมูลที่เป็นข้อเท็จจริง และจะต้องไม่มีการเปลี่ยนแปลงตามกรณีทางประวัติศาสตร์
- คำแนะนำการแก้ไขที่มีความมั่นใจต่ำไม่ควรถูกแทรกเข้าไปในข้อจำกัดที่เข้มงวด

### 2. ความสามารถในการป้อนข้อมูลด้วยเสียง: การพูดเป็นธรรมชาติมากกว่าการพิมพ์

นอกจากการป้อนข้อความแล้ว เรายังรองรับการป้อนด้วยเสียงอีกด้วย สิ่งนี้มีประโยชน์อย่างยิ่งในการอธิบายข้อกำหนดที่ซับซ้อน การพิมพ์คำขอด้านเทคนิคอาจใช้เวลาไม่กี่นาที ในขณะที่การพูดออกเสียงอาจใช้เวลาเพียงไม่กี่วินาที

การออกแบบหลักสำหรับการป้อนข้อมูลด้วยเสียงคือ "การจัดการของรัฐ" ผู้ใช้จะต้องทราบอย่างชัดเจนว่าระบบอยู่ในสถานะใด เรากำหนดสถานะต่อไปนี้:

- **ไม่ได้ใช้งาน**: ระบบพร้อมและสามารถเริ่มการบันทึกได้
- **กำลังรออัปสตรีม**: ระบบกำลังเชื่อมต่อกับบริการแบ็กเอนด์
- **การบันทึก**: เสียงของผู้ใช้กำลังถูกบันทึกเสียง
- **กำลังประมวลผล**: คำพูดกำลังแปลงเป็นข้อความ
- **ข้อผิดพลาด**: มีข้อผิดพลาดเกิดขึ้นและต้องการความสนใจจากผู้ใช้

โมเดลสถานะส่วนหน้ามีลักษณะดังนี้:

```typescript
interface VoiceInputState {
  status: 'idle' | 'waiting-upstream' | 'recording' | 'processing' | 'error';
  duration: number;
  error?: string;
  deletedSet: Set<string>; // Fingerprint set of deleted results
}

// State transition when recording starts
const handleVoiceInputStart = async () => {
  // Enter waiting state first and show a loading animation
  setState({ status: 'waiting-upstream' });

  // Wait for backend readiness confirmation
  const isReady = await waitForBackendReady();
  if (!isReady) {
    setState({ status: 'error', error: 'Backend service is not ready' });
    return;
  }

  // Start recording
  setState({ status: 'recording', startTime: Date.now() });
};

// Handle recognition results
const handleRecognitionResult = (result: RecognitionResult) => {
  const fingerprint = normalizeFingerprint(result.text);

  // Check whether it has already been deleted
  if (state.deletedSet.has(fingerprint)) {
    return; // Skip deleted content
  }

  // Merge the result into the text box
  appendResult(result);
};
```

มีรายละเอียดที่สำคัญอยู่ที่นี่: เราใช้ "ชุดลายนิ้วมือ" เพื่อจัดการการซิงโครไนซ์การลบ เมื่อการรู้จำเสียงส่งกลับผลลัพธ์หลายรายการ ผู้ใช้อาจลบบางส่วนออกได้ เราจัดเก็บลายนิ้วมือของเนื้อหาที่ถูกลบ ดังนั้นหากเนื้อหาเดิมปรากฏขึ้นอีกครั้งในภายหลัง เนื้อหานั้นจะถูกข้ามโดยอัตโนมัติ โดยพื้นฐานแล้วมันเป็นวิธีจดจำสิ่งที่ผู้ใช้ปฏิเสธไปแล้ว

### 3. ระบบการจัดการที่รวดเร็ว: ส่งออก "สมอง" ของ AI ไปภายนอก

HagiCode มีระบบการจัดการพรอมต์ที่ยืดหยุ่น โดยพรอมต์ทั้งหมดจะถูกจัดเก็บเป็นไฟล์:

```
prompts/
├── metadata/
│   ├── optimize-description.zh-CN.json
│   └── optimize-description.en-US.json
└── templates/
    ├── optimize-description.zh-CN.hbs
    └── optimize-description.en-US.hbs
```

แต่ละพรอมต์ประกอบด้วยสองส่วน:
- **ไฟล์ข้อมูลเมตา (`.json`)**: กำหนดข้อมูล เช่น สถานการณ์พร้อมต์ เวอร์ชัน และพารามิเตอร์
- **ไฟล์เทมเพลต (`.hbs`)**: เนื้อหาพร้อมท์จริงที่เขียนด้วยไวยากรณ์ของแฮนด์บาร์

```json
{
  "scenario": "optimize-description",
  "locale": "zh-CN",
  "version": "1.0.0",
  "syntax": "handlebars",
  "syntaxVersion": "1.0",
  "parameters": [
    {
      "name": "title",
```

ประโยชน์ของการออกแบบนี้ชัดเจน:
- พรอมต์สามารถควบคุมเวอร์ชันได้เช่นเดียวกับโค้ด
- รองรับหลายภาษาและสามารถเปลี่ยนได้โดยอัตโนมัติตามความต้องการของผู้ใช้
- การออกแบบที่กำหนดพารามิเตอร์ช่วยให้สามารถแทรกบริบทแบบไดนามิกได้
- สามารถตรวจสอบความสมบูรณ์ได้เมื่อเริ่มต้นระบบ หลีกเลี่ยงข้อผิดพลาดรันไทม์

หากความรู้ยังอยู่ในหัวของใครซักคน ก็สูญเปล่าได้ง่าย การบันทึกอย่างมีโครงสร้างตั้งแต่ต้นจะปลอดภัยกว่ามาก

### 4. ตัวช่วยสร้างแบบก้าวหน้า: แบ่งงานที่ซับซ้อนออกเป็นขั้นตอนเล็กๆ

สำหรับงานที่ซับซ้อน เช่น การติดตั้งและการกำหนดค่าครั้งแรก เราใช้การออกแบบวิซาร์ดหลายขั้นตอน แต่ละขั้นตอนขอเฉพาะข้อมูลที่จำเป็นและให้ตัวบ่งชี้ความคืบหน้าที่ชัดเจน งานขนาดใหญ่จะจัดการได้ง่ายขึ้นมากเมื่อจัดการทีละขั้นตอน

โมเดลสถานะตัวช่วยสร้าง:

```typescript
interface WizardState {
  currentStep: number;           // 0-3, corresponding to 4 steps
  steps: WizardStep[];
  canGoNext: boolean;
  canGoBack: boolean;
  isLoading: boolean;
  error: string | null;
}

interface WizardStep {
  id: number;
  title: string;
  description: string;
  completed: boolean;
}

// Step navigation logic
const goToNextStep = () => {
  if (wizardState.currentStep < wizardState.steps.length - 1) {
    // Validate input for the current step
    if (validateCurrentStep()) {
      wizardState.currentStep++;
      wizardState.steps[wizardState.currentStep - 1].completed = true;
    }
  }
};

const goToPreviousStep = () => {
  if (wizardState.currentStep > 0) {
    wizardState.currentStep--;
  }
};
```

แต่ละขั้นตอนมีตรรกะในการตรวจสอบของตัวเอง และขั้นตอนที่เสร็จสมบูรณ์จะได้รับเครื่องหมายที่มองเห็นได้ชัดเจน การยกเลิกจะเปิดกล่องโต้ตอบการยืนยันเพื่อป้องกันไม่ให้ผู้ใช้สูญเสียความคืบหน้าจากการดำเนินการโดยไม่ตั้งใจ

## บทสรุป

เมื่อมองย้อนกลับไปที่แนวทางปฏิบัติในการเปิดเผยข้อมูลที่ก้าวหน้าของ HagiCode เราสามารถสรุปหลักการสำคัญหลายประการได้:

1. **คำแนะนำทีละขั้นตอน**: แบ่งงานที่ซับซ้อนออกเป็นขั้นตอนเล็กๆ และขอเฉพาะข้อมูลที่จำเป็นในแต่ละขั้นตอน
2. **การสำเร็จอย่างชาญฉลาด**: ใช้บริบททางประวัติศาสตร์และความรู้ของโครงการเพื่อกรอกข้อมูลโดยอัตโนมัติ
3. **ข้อเสนอแนะทันที**: ให้ทุกการกระทำที่มีภาพตอบรับและคำแนะนำสถานะที่ชัดเจน
4. **กลไกการทนทานต่อข้อผิดพลาด**: อนุญาตให้ผู้ใช้เลิกทำและรีเซ็ต ดังนั้นข้อผิดพลาดจะไม่นำไปสู่การสูญเสียที่ไม่สามารถย้อนกลับได้
5. **ความหลากหลายของอินพุต**: รองรับวิธีการป้อนข้อมูลที่หลากหลาย เช่น ข้อความและเสียง

ใน HagiCode ผลลัพธ์ในทางปฏิบัติของแนวทางนี้ชัดเจน: ความยาวเฉลี่ยของการป้อนข้อมูลของผู้ใช้เพิ่มขึ้นจากน้อยกว่า 20 อักขระเป็นคำอธิบายที่มีโครงสร้าง 200-300 อักขระ คุณภาพของข้อเสนอที่สร้างโดย AI ได้รับการปรับปรุงอย่างมีนัยสำคัญ และความพึงพอใจของผู้ใช้ก็เพิ่มขึ้นตามไปด้วย

นี่ไม่ใช่เรื่องน่าแปลกใจเลย ยิ่งผู้ใช้ให้ข้อมูลมากเท่าไร AI ก็สามารถเข้าใจพวกเขาได้แม่นยำมากขึ้นเท่านั้น และผลลัพธ์ที่ได้ก็จะยิ่งดียิ่งขึ้นเท่านั้น ในแง่นั้นก็ไม่ต่างจากการสื่อสารระหว่างผู้คนมากนัก

หากคุณกำลังสร้างผลิตภัณฑ์ที่เกี่ยวข้องกับ AI ฉันหวังว่าประสบการณ์เหล่านี้จะเป็นแรงบันดาลใจที่เป็นประโยชน์ ข้อควรจำ: ผู้ใช้ไม่จำเป็นต้องปฏิเสธที่จะให้ข้อมูล บ่อยครั้งที่ผลิตภัณฑ์ยังไม่ได้ถามคำถามที่ถูกต้องด้วยวิธีที่ถูกต้อง หัวใจสำคัญของการเปิดเผยข้อมูลอย่างต่อเนื่องคือการหาช่วงเวลาและรูปแบบที่ดีที่สุดสำหรับคำถามเหล่านั้น
