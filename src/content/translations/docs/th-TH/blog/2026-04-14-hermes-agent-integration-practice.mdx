## แนวทางปฏิบัติในการบูรณาการตัวแทนของ Hermes: จากพิธีสารสู่การผลิต

> การแบ่งปันประสบการณ์ HagiCode ที่สมบูรณ์ของการบูรณาการ Hermes Agent รวมถึงบทเรียนหลักเกี่ยวกับการปรับโปรโตคอล ACP การจัดการเซสชันพูล และการซิงโครไนซ์สัญญาส่วนหน้า/ส่วนหลัง

## พื้นหลัง

ในขณะที่สร้าง HagiCode ซึ่งเป็นแพลตฟอร์มการเขียนโค้ดที่ได้รับความช่วยเหลือจาก AI ทีมของเราจำเป็นต้องผสานรวมเฟรมเวิร์กตัวแทนที่สามารถทำงานในพื้นที่และปรับขนาดไปยังระบบคลาวด์ได้ด้วย หลังการวิจัย ตัวแทน Hermes จาก Nous Research ได้รับเลือกให้เป็นเครื่องมือพื้นฐานสำหรับความสามารถตัวแทนทั่วไปของเรา

ความจริงแล้ว การเลือกเทคโนโลยีไม่ได้ยากหรือง่ายเป็นพิเศษ มีเฟรมเวิร์ก Agent ที่แข็งแกร่งมากมายในตลาด แต่ Hermes มีความโดดเด่นเนื่องจากโปรโตคอล ACP และระบบเครื่องมือนั้นสอดคล้องกับข้อกำหนดความต้องการของ HagiCode เป็นอย่างดี: การพัฒนาในท้องถิ่น การทำงานร่วมกันเป็นทีม และการขยายระบบคลาวด์ อย่างไรก็ตาม การนำ Hermes เข้าสู่ระบบการผลิตจริงหมายถึงการแก้ปัญหาทางวิศวกรรมอันยาวนาน ส่วนนี้ไม่มีอะไรนอกจากเรื่องเล็กน้อย

สแต็กของ HagiCode ใช้ Orleans เพื่อสร้างระบบแบบกระจาย ในขณะที่ส่วนหน้าสร้างด้วย React + TypeScript การบูรณาการ Hermes หมายถึงการรักษาความสอดคล้องทางสถาปัตยกรรมในขณะเดียวกันก็ทำให้ Hermes เป็นผู้ดำเนินการระดับเฟิร์สคลาสควบคู่ไปกับ ClaudeCode และ OpenCode ฟังดูง่ายพอ แต่การนำไปปฏิบัติจะบอกเล่าเรื่องราวที่แท้จริงเสมอ

บทความนี้แบ่งปันประสบการณ์เชิงปฏิบัติของเราในการบูรณาการ Hermes Agent เข้ากับ HagiCode และเราหวังว่าบทความนี้จะนำเสนอข้อมูลอ้างอิงที่เป็นประโยชน์สำหรับทีมที่เผชิญกับความต้องการที่คล้ายกัน ท้ายที่สุด เมื่อคุณตกลงไปในหลุมแล้ว ก็ไม่มีเหตุผลที่จะปล่อยให้คนอื่นตกหลุมเดียวกันได้

## เกี่ยวกับ ฮากิโค้ด

วิธีแก้ปัญหาที่อธิบายไว้ในบทความนี้มาจากการทำงานจริงของเราใน [ฮากิโค้ด](https://hagicode.com) โครงการ HagiCode เป็นแพลตฟอร์มความช่วยเหลือในการเขียนโค้ดที่ขับเคลื่อนด้วย AI ซึ่งสนับสนุนการเข้าถึงและการจัดการแบบรวมศูนย์ของผู้ให้บริการ AI หลายราย ในระหว่างการบูรณาการ Hermes Agent เราได้ออกแบบเลเยอร์นามธรรมของผู้ให้บริการทั่วไป เพื่อให้ตัวแทนประเภทใหม่สามารถเสียบเข้ากับระบบที่มีอยู่ได้อย่างราบรื่น

หากคุณสนใจ HagiCode เข้าไปเยี่ยมชมได้ฟรี [GitHub](https://github.com/HagiCode-org/site) เพื่อเรียนรู้เพิ่มเติม ยิ่งมีคนให้ความสนใจมากเท่าไร โมเมนตัมก็จะยิ่งแข็งแกร่งขึ้นเท่านั้น

## การออกแบบสถาปัตยกรรม

### วิธีการออกแบบแบบเป็นชั้น

การบูรณาการ Hermes ของ HagiCode ใช้สถาปัตยกรรมแบบเลเยอร์ที่ชัดเจน โดยแต่ละเลเยอร์มุ่งเน้นไปที่ความรับผิดชอบของตัวเอง:

**ชั้นแกนส่วนหลัง**
- `HermesCliProvider`: ดำเนินการ `IAIProvider` อินเทอร์เฟซเป็นจุดเริ่มต้นผู้ให้บริการ AI แบบครบวงจร
- `HermesPlatformConfiguration`: จัดการพาธปฏิบัติการของ Hermes อาร์กิวเมนต์ การรับรองความถูกต้อง และการตั้งค่าที่เกี่ยวข้อง
- `ICliProvider<HermesOptions>`: นามธรรม CLI ระดับต่ำจัดทำโดย HagiCode.Libs สำหรับการจัดการวงจรชีวิตของกระบวนการย่อย

**ชั้นขนส่ง**
- `StdioAcpTransport`: สื่อสารกับกระบวนการย่อย Hermes ACP ผ่านอินพุตและเอาต์พุตมาตรฐาน
- วิธีการโปรโตคอล ACP: `initialize`, `authenticate`, `session/new`, `session/prompt`

**เลเยอร์รันไทม์**
- `HermesGrain`: การใช้งาน Orleans Grain ที่จัดการการดำเนินการเซสชันแบบกระจาย
- `CliAcpSessionPool`: เซสชันพูลที่ใช้กระบวนการย่อย ACP ซ้ำเพื่อหลีกเลี่ยงค่าใช้จ่ายในการเริ่มต้นระบบบ่อยครั้ง

**ส่วนหน้า**
- `ExecutorAvatar`: ภาพลักษณ์และไอคอนของ Hermes
- `executorTypeAdapter`: ตรรกะการแมปประเภทผู้ให้บริการ
- การส่งข้อความแบบเรียลไทม์ของ SignalR: รักษาความสอดคล้องของข้อมูลประจำตัวของ Hermes ตลอดทั้งสตรีมข้อความ

การออกแบบเป็นชั้นนี้ช่วยให้แต่ละชั้นสามารถพัฒนาได้อย่างอิสระ ตัวอย่างเช่น หากเราต้องการเพิ่มกลไกการขนส่งใหม่ในอนาคต เช่น WebSocket เราเพียงแต่ต้องแก้ไขเลเยอร์การขนส่งเท่านั้น ไม่จำเป็นต้องพลิกทั้งระบบเพียงเพราะการขนส่งครั้งหนึ่งมีการเปลี่ยนแปลง

### นามธรรมอินเทอร์เฟซแบบรวม

ผู้ให้บริการ AI ทั้งหมดใช้งาน `IAIProvider` อินเทอร์เฟซ ซึ่งเป็นหนึ่งในตัวเลือกการออกแบบหลักในสถาปัตยกรรมของ HagiCode:

```csharp
public interface IAIProvider
{
    string Name { get; }
    ProviderCapabilities Capabilities { get; }

    IAsyncEnumerable<AIStreamingChunk> StreamAsync(
        AIRequest request,
        CancellationToken cancellationToken = default);

    Task<AIResponse> ExecuteAsync(
        AIRequest request,
        CancellationToken cancellationToken = default);
}
```

`HermesCliProvider` ใช้อินเทอร์เฟซนี้และยืนหยัดอย่างเท่าเทียมกันด้วย `ClaudeCodeProvider`, `OpenCodeProvider`และอื่น ๆ ประโยชน์ของการออกแบบนี้ ได้แก่ :

1. **ความสามารถในการทดแทน**: ผู้ให้บริการสวิตช์ไม่ส่งผลต่อตรรกะทางธุรกิจชั้นบน
2. **ความสามารถในการทดสอบ**: สามารถจำลองผู้ให้บริการสำหรับการทดสอบหน่วยได้อย่างง่ายดาย
3. **ความสามารถในการขยาย**: การเพิ่มผู้ให้บริการใหม่จำเป็นต้องมีการใช้งานอินเทอร์เฟซเท่านั้น

ท้ายที่สุดแล้ว อินเทอร์เฟซก็เหมือนกับกฎเกณฑ์ เมื่อมีกฎเกณฑ์แล้ว ทุกคนสามารถอยู่ร่วมกันได้อย่างกลมกลืน เล่นตามจุดแข็งของตนเอง และหลีกเลี่ยงการเหยียบย่ำกัน มีความสง่างามบางอย่างอยู่ในนั้น

## การดำเนินงานหลัก

### การใช้งานเลเยอร์ของผู้ให้บริการ

`HermesCliProvider` เป็นหัวใจสำคัญของการบูรณาการทั้งหมด โดยประสานองค์ประกอบต่างๆ ที่จำเป็นในการเรียกใช้ AI เดียวให้เสร็จสมบูรณ์:

```csharp
public sealed class HermesCliProvider : IAIProvider, IVersionedAIProvider
{
    private readonly ICliProvider<LibsHermesOptions> _provider;
    private readonly ConcurrentDictionary<string, string> _sessionBindings;

    public ProviderCapabilities Capabilities { get; } = new()
    {
        SupportsStreaming = true,
        SupportsTools = true,
        SupportsSystemMessages = true,
        SupportsArtifacts = false
    };

    public async IAsyncEnumerable<AIStreamingChunk> StreamAsync(
        AIRequest request,
        [EnumeratorCancellation] CancellationToken cancellationToken = default)
    {
        // 1. Resolve the session binding key
        var bindingKey = ResolveBindingKey(request.CessionId);

        // 2. Get or create a Hermes session through the session pool
        var options = new HermesOptions
        {
            ExecutablePath = _platformConfiguration.ExecutablePath,
            Arguments = _platformConfiguration.Arguments,
            SessionId = _sessionBindings.TryGetValue(bindingKey, out var sessionId) ? sessionId : null,
            WorkingDirectory = request.WorkingDirectory,
            Model = request.Model
        };

        // 3. Execute and collect the streaming response
        await foreach (var message in _provider.ExecuteAsync(options, request.Prompt, cancellationToken))
        {
            // 4. Map ACP messages to AIStreamingChunk
            if (_responseMapper.TryConvertToStreamingChunk(message, out var chunk))
            {
                yield return chunk;
            }
        }
    }
}
```

จุดการออกแบบหลายจุดมีความสำคัญอย่างยิ่งที่นี่:

1. **การเชื่อมโยงเซสชัน**: การใช้งาน `CessionId` เพื่อผูกคำขอหลายรายการเข้ากับกระบวนการย่อยของ Hermes เดียวกัน โดยรักษาความต่อเนื่องของบริบทในการสนทนาแบบหลายรอบ
2. **การแมปการตอบสนอง**: แปลงรูปแบบข้อความ Hermes ACP ให้เป็นแบบรวม `AIStreamingChunk` รูปแบบ
3. **รองรับการสตรีมมิ่ง**: การใช้งาน `IAsyncEnumerable` เพื่อรองรับการตอบสนองการสตรีมมิ่งที่แท้จริง

การเชื่อมโยงเซสชันก็เหมือนกับความสัมพันธ์ของมนุษย์ เมื่อสร้างการเชื่อมต่อแล้ว การสื่อสารในอนาคตจะมีบริบท ดังนั้นคุณไม่จำเป็นต้องเริ่มจากศูนย์ในแต่ละครั้ง แน่นอนว่าความสัมพันธ์นั้นยังคงต้องรักษาไว้

### การปรับโปรโตคอล ACP

Hermes ใช้ ACP (Agent Communication Protocol) ซึ่งแตกต่างจาก HTTP API แบบดั้งเดิม ACP เป็นโปรโตคอลที่ใช้อินพุตและเอาต์พุตมาตรฐาน และมีลักษณะเฉพาะหลายประการ:

1. **เครื่องหมายเริ่มต้น**: หลังจากที่กระบวนการ Hermes เริ่มต้นขึ้น มันจะส่งออกไฟล์ `//ready` เครื่องหมาย
2. **การตรวจสอบสิทธิ์แบบไดนามิก**: วิธีการตรวจสอบความถูกต้องไม่ได้รับการแก้ไขและต้องมีการเจรจาผ่านโปรโตคอล
3. **การใช้เซสชันซ้ำ**: เซสชันที่สร้างไว้จะถูกนำมาใช้ซ้ำผ่าน `SessionId`
4. **คำตอบแบบแยกส่วน**: คำตอบที่สมบูรณ์อาจแบ่งออกเป็นหลายคำตอบได้ `session/update` การแจ้งเตือน

HagiCode จัดการคุณลักษณะเหล่านี้ผ่าน `StdioAcpTransport`:

```csharp
public class StdioAcpTransport
{
    public async Task InitializeAsync(CancellationToken cancellationToken)
    {
        // Wait for the //ready marker
        var readyLine = await _outputReader.ReadLineAsync(cancellationToken);
        if (readyLine != "//ready")
        {
            throw new InvalidOperationException("Hermes did not send ready signal");
        }

        // Send the initialize request
        await SendRequestAsync(new
        {
            jsonrpc = "2.0",
            id = 1,
            method = "initialize",
            @params = new
            {
                protocolVersion = "2024-11-05",
                capabilities = new { },
                clientInfo = new { name = "HagiCode", version = "1.0.0" }
            }
        }, cancellationToken);
    }
}
```

โปรโตคอลก็เหมือนกับความเข้าใจร่วมกันระหว่างผู้คน เมื่อเข้าใจแล้ว การสื่อสารก็จะราบรื่นมากขึ้น การสร้างมันต้องใช้เวลา

### การจัดการพูลเซสชัน

การเริ่มกระบวนการย่อยของ Hermes บ่อยครั้งมีราคาแพง ดังนั้นเราจึงใช้กลไกเซสชันพูล:

```csharp
services.AddSingleton(static _ =>
{
    var registry = new CliProviderPoolConfigurationRegistry();
    registry.Register("hermes", new CliPoolSettings
    {
        MaxActiveSessions = 50,
        IdleTimeout = TimeSpan.FromMinutes(10)
    });
    return registry;
});
```

พารามิเตอร์พูลเซสชันหลัก:
- `MaxActiveSessions`: ควบคุมขีดจำกัดการทำงานพร้อมกันเพื่อหลีกเลี่ยงไม่ให้ทรัพยากรหมด
- `IdleTimeout`: การหมดเวลาที่ไม่ได้ใช้งานซึ่งจะสมดุลต้นทุนการเริ่มต้นกับการใช้หน่วยความจำ

ในทางปฏิบัติ เราพบว่า:
1. หากการหมดเวลาของการไม่ได้ใช้งานสั้นเกินไป เซสชันจะรีสตาร์ทบ่อยครั้ง ถ้ามันยาวเกินไป หน่วยความจำจะยังคงอยู่
2. ต้องปรับขีดจำกัดการทำงานพร้อมกันตามโหลดจริง เนื่องจากการตั้งค่าสูงเกินไปอาจทำให้ระบบเชื่องช้า
3. การใช้งานเซสชันพูลจำเป็นต้องมีการตรวจสอบเพื่อให้สามารถปรับเปลี่ยนพารามิเตอร์ได้ทันเวลา

นี่เป็นเหมือนทางเลือกอื่นๆ ในชีวิต การก้าวร้าวเกินไปสร้างปัญหา ในขณะที่การเป็นคนอนุรักษ์นิยมเกินไปทำให้พลาดโอกาส เป้าหมายคือการค้นหาสมดุลที่เหมาะสม

### พิมพ์การแมป

ส่วนหน้าจำเป็นต้องระบุผู้ให้บริการ Hermes อย่างถูกต้อง และแสดงองค์ประกอบภาพที่เกี่ยวข้อง:

```typescript
// executorTypeAdapter.ts
export const resolveExecutorVisualTypeFromProviderType = (
  providerType: PCode_Models_AIProviderType | null | undefined
): ExecutorVisualType => {
  switch (providerType) {
    case PCode_Models_AIProviderType.HERMES_CLI:
      return 'Hermes';
    default:
      return 'Unknown';
  }
};
```

```typescript
};
```

ท้ายที่สุดแล้ว สิ่งสวยงามก็สมควรได้รับการนำเสนอที่สวยงาม การทำให้แน่ใจว่าความสวยงามนั้นสามารถมองเห็นได้จริงนั้นยังคงขึ้นอยู่กับงานฝีมือส่วนหน้า

### การซิงโครไนซ์สัญญา

ส่วนหน้าและส่วนหลังทำให้สัญญาสอดคล้องกันผ่านการสร้าง OpenAPI ด้านหลังกำหนด `AIProviderType` แจงนับ:

```csharp
public enum AIProviderType
{
    Unknown,
    ClaudeCode,
    OpenCode,
    HermesCli  // Newly added
}
```

ส่วนหน้าจะสร้างประเภท TypeScript ที่สอดคล้องกันผ่าน OpenAPI เพื่อให้แน่ใจว่าค่าแจงนับมีความสอดคล้องกัน นี่คือกุญแจสำคัญในการหลีกเลี่ยงการแสดงส่วนหน้า `Unknown`.

สัญญาก็เหมือนกับคำมั่นสัญญา เมื่อตกลงกันแล้วก็ต้องให้เกียรติ ไม่เช่นนั้น คุณจะตกอยู่ในสถานการณ์ที่น่าอึดอัดใจเช่น `Unknown`.

## การจัดการการกำหนดค่า

การกำหนดค่า Hermes ได้รับการจัดการผ่าน `appsettings.json`:

```json
{
  "Providers": {
    "HermesCli": {
      "ExecutablePath": "hermes",
      "Arguments": "acp",
      "StartupTimeoutMs": 10000,
      "ClientName": "HagiCode",
      "Authentication": {
        "PreferredMethodId": "api-key",
        "MethodInfo": {
          "api-key": "your-api-key-here"
        }
      },
      "SessionDefaults": {
        "Model": "claude-sonnet-4-20250514",
        "ModeId": "default"
      }
    }
  }
}
```

การออกแบบที่ขับเคลื่อนด้วยการกำหนดค่านี้นำมาซึ่งความยืดหยุ่น:
- สามารถแทนที่เส้นทางปฏิบัติการได้ ซึ่งสะดวกสำหรับการพัฒนาและการทดสอบ
- อาร์กิวเมนต์เริ่มต้นสามารถปรับแต่งให้ตรงกับรุ่น Hermes ที่แตกต่างกัน
- ข้อมูลการตรวจสอบความถูกต้องสามารถกำหนดค่าเพื่อรองรับวิธีการตรวจสอบความถูกต้องได้หลายวิธี

การกำหนดค่าก็เหมือนกับคำถามแบบปรนัยในชีวิต หากมีตัวเลือกเพียงพอ ก็มักจะมีตัวเลือกที่เหมาะสม กล่าวคือ ตัวเลือกมากเกินไปอาจสร้างความเหนื่อยล้าในการตัดสินใจได้ด้วยตนเอง

## ประสบการณ์เชิงปฏิบัติ

### การตรวจสุขภาพ

การสร้างผู้ให้บริการที่เชื่อถือได้จำเป็นต้องมีการตรวจสุขภาพที่ครอบคลุม:

```csharp
public async Task<ProviderTestResult> PingAsync(CancellationToken cancellationToken = default)
{
    var response = await ExecuteAsync(new AIRequest
    {
        Prompt = "Reply with exactly PONG.",
        CessionId = null,
        AllowedTools = Array.Empty<string>(),
        WorkingDirectory = ResolveWorkingDirectory(null)
    }, cancellationToken);

    var success = string.Equals(response.Content.Trim(), "PONG", StringComparison.OrdinalIgnoreCase);
    return new ProviderTestResult
    {
        ProviderName = Name,
        Success = success,
        ResponseTimeMs = stopwatch.ElapsedMilliseconds,
        ErrorMessage = success ? null : $"Unexpected Hermes ping response: '{response.Content}'."
    };
}
```

ข้อควรสังเกตในการตรวจสุขภาพ:
1. ใช้กรณีทดสอบง่ายๆ และหลีกเลี่ยงสถานการณ์ที่ซับซ้อนเกินไป
2. ตั้งค่าการหมดเวลาที่เหมาะสม
3. บันทึกเวลาตอบสนองเพื่อรองรับการวิเคราะห์ประสิทธิภาพ

เช่นเดียวกับที่ผู้คนต้องการการตรวจร่างกาย ระบบก็ต้องการการตรวจสุขภาพด้วย ยิ่งพบปัญหาได้เร็วเท่าไรก็ยิ่งแก้ไขได้ง่ายขึ้นเท่านั้น

### เครื่องมือตรวจสอบความถูกต้อง

HagiCode จัดเตรียมคอนโซลเฉพาะสำหรับการตรวจสอบความถูกต้องของการบูรณาการ Hermes:

```bash
# Basic validation
HagiCode.Libs.Hermes.Console --test-provider

# Full suite (including repository analysis)
HagiCode.Libs.Hermes.Console --test-provider-full --repo .

# Custom executable
HagiCode.Libs.Hermes.Console --test-provider-full --executable /path/to/hermes
```

เครื่องมือนี้มีประโยชน์อย่างยิ่งในระหว่างการพัฒนาเพราะช่วยให้เราตรวจสอบได้อย่างรวดเร็วว่าการบูรณาการนั้นถูกต้องหรือไม่ ท้ายที่สุดแล้ว ไม่มีใครอยากรอจนกว่าปัญหาจะเกิดขึ้นก่อนที่จะจำการทดสอบ

### การจัดการปัญหาทั่วไป

**การตรวจสอบสิทธิ์ล้มเหลว**
- ตรวจสอบว่า `Authentication.PreferredMethodId` ตรงกับวิธีการรับรองความถูกต้องที่ Hermes รองรับจริง
- ยืนยันว่ารูปแบบข้อมูลการรับรองความถูกต้องถูกต้อง เช่น คีย์ API หรือ Bearer Token

**หมดเวลาเซสชัน**
- เพิ่ม `StartupTimeoutMs` ค่า
- ตรวจสอบความสามารถในการเข้าถึงเซิร์ฟเวอร์ MCP
- ตรวจสอบการใช้ทรัพยากรระบบ

**ตอบไม่ครบ**
- มั่นใจ `session/update` การแจ้งเตือนและผลลัพธ์สุดท้ายจะถูกรวบรวมอย่างถูกต้อง
- ตรวจสอบตรรกะการยกเลิกในการจัดการการสตรีม
- ตรวจสอบว่าการจัดการข้อผิดพลาดเสร็จสมบูรณ์

**ไม่ทราบการแสดงผลส่วนหน้า**
- ยืนยันว่าการสร้าง OpenAPI มีไฟล์ `HermesCli` ค่าแจงนับ
- ตรวจสอบว่าการแมปประเภทถูกต้องหรือไม่
- ล้างแคชของเบราว์เซอร์และสร้างประเภทใหม่

ปัญหาก็จะตามมาเสมอ เมื่อปรากฏขึ้นสิ่งสำคัญคือไม่ต้องตื่นตระหนก ติดตามสาเหตุทีละขั้นตอน และในที่สุด สาเหตุส่วนใหญ่ก็สามารถแก้ไขได้

### คำแนะนำการเพิ่มประสิทธิภาพการทำงาน

1. **ใช้กลุ่มเซสชัน**: นำกระบวนการย่อย ACP มาใช้ซ้ำเพื่อลดค่าใช้จ่ายในการเริ่มต้นระบบ
2. **ตั้งค่าการหมดเวลาอย่างเหมาะสม**: ปรับสมดุลการใช้หน่วยความจำกับต้นทุนการเริ่มต้นระบบ
3. **ใช้รหัสเซสชันซ้ำ**: ใช้รหัสเดิม `CessionId` สำหรับงานแบทช์
4. **กำหนดค่า MCP ตามความต้องการ**: หลีกเลี่ยงการเรียกใช้เครื่องมือที่ไม่จำเป็น

ประสิทธิภาพก็เหมือนกับประสิทธิภาพในชีวิตประจำวันมาก เมื่อคุณทำถูกต้อง คุณจะประสบความสำเร็จมากขึ้นโดยใช้เวลาน้อยลง เมื่อคุณทำผิด ความพยายามจะทวีคูณในขณะที่ผลลัพธ์ลดลง การค้นหาจุดที่ "ถูกต้อง" นั้นต้องใช้ทั้งประสบการณ์และโชคลาภ

## บทสรุป

การรวม Hermes Agent เข้ากับระบบการผลิตจำเป็นต้องพิจารณาปัญหาในหลายมิติ:

1. **สถาปัตยกรรม**: ออกแบบอินเทอร์เฟซผู้ให้บริการแบบรวม และใช้สถาปัตยกรรมส่วนประกอบที่ทดแทนได้
2. **โปรโตคอล**: จัดการพฤติกรรมเฉพาะของ ACP ได้อย่างถูกต้อง เช่น เครื่องหมายเริ่มต้นระบบและการตรวจสอบสิทธิ์แบบไดนามิก
3. **ประสิทธิภาพ**: นำทรัพยากรกลับมาใช้ใหม่ผ่านพูลเซสชัน และปรับต้นทุนการเริ่มต้นให้สมดุลกับการใช้หน่วยความจำ
4. **ส่วนหน้า**: รับประกันการประสานสัญญาและมอบประสบการณ์การมองเห็นที่สอดคล้องกัน

ประสบการณ์ของ HagiCode แสดงให้เห็นว่าด้วยการออกแบบแบบหลายชั้นที่ดีและการใช้งานที่ขับเคลื่อนด้วยการกำหนดค่า ระบบ Agent ที่ซับซ้อนสามารถบูรณาการเข้ากับสถาปัตยกรรมที่มีอยู่ได้อย่างราบรื่น

หลักการเหล่านี้ฟังดูง่ายเมื่ออธิบายเป็นคำพูด แต่การนำไปปฏิบัติจริงมักนำมาซึ่งปัญหาหลายประเภทเสมอ นั่นเป็นเรื่องปกติ หากปัญหาได้รับการแก้ไข มันก็จะกลายเป็นประสบการณ์ ถ้าไม่ทำก็จะกลายเป็นบทเรียน ไม่ว่าอย่างไรก็ยังมีคุณค่า

สิ่งสวยงามหรือคนไม่จำเป็นต้องครอบครอง ตราบใดที่ยังคงสวยงามอยู่ ก็เพียงพอที่จะชื่นชมความงามนั้นอย่างเงียบๆ เทคโนโลยีก็เหมือนกันมาก ถ้ามันช่วยทำให้ระบบดีขึ้น กรอบงานหรือโปรโตคอลเฉพาะก็มีความสำคัญน้อยกว่าที่คนคิดกันมาก
