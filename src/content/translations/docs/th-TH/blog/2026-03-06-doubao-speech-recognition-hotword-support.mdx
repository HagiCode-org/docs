## คำแนะนำในการใช้การสนับสนุน Hotword สำหรับการรู้จำเสียง Doubao

> บทความนี้จะอธิบายรายละเอียดวิธีการใช้การรองรับคำที่นิยมสำหรับการรู้จำเสียง Doubao ในโปรเจ็กต์ HagiCode ด้วยการใช้ทั้งคำที่นิยมที่กำหนดเองและตารางคำที่นิยมของแพลตฟอร์ม คุณสามารถปรับปรุงความแม่นยำในการจดจำสำหรับคำศัพท์เฉพาะโดเมนได้อย่างมาก

## พื้นหลัง

เทคโนโลยีการรู้จำเสียงได้รับการพัฒนามาหลายปีแล้ว แต่ปัญหาหนึ่งที่รบกวนนักพัฒนาอย่างต่อเนื่อง โมเดลการรู้จำเสียงสำหรับวัตถุประสงค์ทั่วไปสามารถครอบคลุมภาษาในชีวิตประจำวันได้ แต่มักขาดคำศัพท์เฉพาะทางวิชาชีพ ชื่อผลิตภัณฑ์ และชื่อส่วนบุคคล ลองคิดดู: ผู้ช่วยด้านเสียงในสาขาการแพทย์จำเป็นต้องจดจำคำศัพท์ต่างๆ เช่น "ความดันโลหิตสูง" "เบาหวาน" และ "โรคหลอดเลือดหัวใจ" ได้อย่างถูกต้อง ระบบกฎหมายจำเป็นต้องจับคำศัพท์ต่างๆ เช่น "สาเหตุของการกระทำ" "การป้องกัน" และ "ภาระในการพิสูจน์" อย่างแม่นยำ ในสถานการณ์เหล่านี้ โมเดลสำหรับวัตถุประสงค์ทั่วไปกำลังพยายามอย่างดีที่สุด แต่นั่นมักจะไม่เพียงพอ

เราพบกับความท้าทายเดียวกันในโครงการ HagiCode ในฐานะผู้ช่วยเขียนโค้ด AI แบบมัลติฟังก์ชั่น HagiCode จำเป็นต้องจัดการการรู้จำเสียงสำหรับคำศัพท์ทางเทคนิคที่หลากหลาย อย่างไรก็ตาม API การรู้จำเสียง Doubao ในการกำหนดค่าเริ่มต้น ไม่สามารถตอบสนองข้อกำหนดด้านความแม่นยำของเราสำหรับข้อกำหนดเฉพาะได้อย่างสมบูรณ์ ไม่ใช่ว่าโต่วเป่าไม่ดีพอ แต่ทุกโดเมนมีระบบคำศัพท์เฉพาะของตัวเอง หลังจากการวิจัยและสำรวจทางเทคนิคแล้ว เราพบว่า API การรู้จำเสียง Doubao ให้การสนับสนุนคำที่นิยมจริงๆ ด้วยการกำหนดค่าที่ตรงไปตรงมา จึงสามารถปรับปรุงความแม่นยำในการจดจำคำศัพท์เฉพาะได้อย่างมาก ในแง่หนึ่ง เมื่อคุณบอกว่าควรใส่ใจคำไหน คำเหล่านั้นก็จะฟังอย่างระมัดระวังมากขึ้น

สิ่งที่เราแบ่งปันในบทความนี้คือโซลูชันที่สมบูรณ์ที่เราใช้ในโครงการ HagiCode เพื่อใช้คำที่นิยมในการรู้จำเสียง Doubao ทั้งสองโหมด คำสั่งที่นิยมที่กำหนดเอง และตารางคำสั่งที่นิยมของแพลตฟอร์ม มีให้บริการ และยังสามารถรวมเข้าด้วยกันได้อีกด้วย ด้วยโซลูชันนี้ นักพัฒนาสามารถกำหนดค่าคำที่นิยมได้อย่างยืดหยุ่นตามสถานการณ์ทางธุรกิจ ดังนั้นระบบการรู้จำเสียงพูดจึงสามารถ "จดจำ" คำศัพท์ที่เป็นมืออาชีพ ไม่ธรรมดา แต่มีความสำคัญได้ดียิ่งขึ้น

## เกี่ยวกับ ฮากิโค้ด

วิธีแก้ปัญหาที่แบ่งปันในบทความนี้มาจากประสบการณ์จริงของเราใน [ฮากิโค้ด](https://hagicode.com) โครงการ HagiCode เป็นโปรเจ็กต์ผู้ช่วยเขียนโค้ด AI แบบโอเพ่นซอร์สพร้อมกลุ่มเทคโนโลยีสมัยใหม่ ออกแบบมาเพื่อมอบประสบการณ์ความช่วยเหลือด้านการเขียนโปรแกรมอัจฉริยะแก่นักพัฒนา เนื่องจากเป็นโครงการที่ซับซ้อนหลายภาษาและหลายแพลตฟอร์ม HagiCode จำเป็นต้องจัดการกับสถานการณ์การรู้จำเสียงที่เกี่ยวข้องกับคำศัพท์ทางเทคนิคมากมาย ซึ่งจะผลักดันการวิจัยของเราและการนำคุณลักษณะคำที่นิยมไปใช้

หากคุณสนใจในการใช้งานด้านเทคนิคของ HagiCode คุณสามารถเยี่ยมชมได้ที่ [พื้นที่เก็บข้อมูล GitHub](https://github.com/HagiCode-org/site) สำหรับรายละเอียดเพิ่มเติมหรือตรวจสอบของเรา [เอกสารอย่างเป็นทางการ](https://docs.hagicode.com) สำหรับคู่มือการติดตั้งและการใช้งานฉบับสมบูรณ์

## การดำเนินงานหลัก

### ทำความเข้าใจกับโหมดคำที่นิยมสองโหมด

API การรู้จำเสียง Doubao มีสองวิธีในการกำหนดค่าคำที่นิยม และแต่ละวิธีก็มีกรณีการใช้งานและข้อดีในอุดมคติของตัวเอง

**โหมดคำที่นิยมที่กำหนดเอง** ช่วยให้เราสามารถส่งข้อความคำที่นิยมได้โดยตรงผ่าน `corpus.context` สนาม แนวทางนี้เหมาะอย่างยิ่งสำหรับสถานการณ์ที่คุณต้องกำหนดค่าคำที่นิยมจำนวนเล็กน้อยอย่างรวดเร็ว เช่น การจดจำชื่อผลิตภัณฑ์หรือชื่อของบุคคลชั่วคราว ในการใช้งาน HagiCode เราจะแยกวิเคราะห์ข้อความคำที่นิยมหลายบรรทัดที่ผู้ใช้ป้อนลงในรายการสตริง จากนั้นจัดรูปแบบเป็น `context_data` อาร์เรย์ที่ต้องการโดย Doubao API แนวทางนี้ตรงไปตรงมามาก คุณเพียงแค่บอกระบบว่าคำไหนที่ควรใส่ใจ และมันก็ทำอย่างนั้นจริงๆ

**โหมดตารางคำที่นิยมของแพลตฟอร์ม** ใช้ `corpus.boosting_table_id` เพื่ออ้างอิงตารางคำที่นิยมที่กำหนดค่าไว้ล่วงหน้าในแพลตฟอร์มการเรียนรู้ด้วยตนเองของ Doubao วิธีนี้เหมาะสำหรับสถานการณ์ที่คุณต้องการจัดการคำที่นิยมจำนวนมาก เราสามารถสร้างและดูแลรักษาตารางคำยอดนิยมบนแพลตฟอร์มการเรียนรู้ด้วยตนเองของ Doubao จากนั้นอ้างอิงด้วย ID สำหรับโปรเจ็กต์อย่าง HagiCode ซึ่งคำศัพท์เฉพาะทางจำเป็นต้องได้รับการอัปเดตและบำรุงรักษาอย่างต่อเนื่อง โหมดนี้มอบความสามารถในการจัดการที่ดีกว่ามาก เมื่อจำนวนคำที่นิยมเพิ่มมากขึ้น การมีศูนย์กลางในการจัดการคำเหล่านี้ย่อมดีกว่าการป้อนคำด้วยตนเองทุกครั้ง

ที่น่าสนใจทั้งสองโหมดนี้สามารถใช้ร่วมกันได้เช่นกัน Doubao API รองรับทั้งคำที่นิยมที่กำหนดเองและรหัสตารางคำที่นิยมของแพลตฟอร์มในคำขอเดียวกัน โดยมีกลยุทธ์การผสมผสานที่ควบคุมโดย `combine_mode` พารามิเตอร์ ความยืดหยุ่นนี้ช่วยให้ HagiCode สามารถรองรับความต้องการในการจดจำคำศัพท์ทางวิชาชีพที่ซับซ้อนได้หลากหลาย บางครั้งการรวมหลายๆ วิธีเข้าด้วยกันก็ให้ผลลัพธ์ที่ดีกว่า

### คำจำกัดความและการตรวจสอบประเภทส่วนหน้า

ในการใช้งานส่วนหน้าของ HagiCode เราได้กำหนดประเภทการกำหนดค่าคำที่นิยมและตรรกะการตรวจสอบที่สมบูรณ์ ส่วนแรกคือคำจำกัดความของประเภท:

```typescript
export interface HotwordConfig {
  contextText: string;           // Multi-line hotword text
  boostingTableId: string;      // Doubao platform hotword table ID
  combineMode: boolean;          // Whether to use both together
}
```

อินเทอร์เฟซที่เรียบง่ายนี้ประกอบด้วยรายการการกำหนดค่าทั้งหมดสำหรับคุณลักษณะคำที่นิยม ในหมู่พวกเขา `contextText` คือส่วนที่ผู้ใช้โต้ตอบด้วยโดยตรงมากที่สุด: เราอนุญาตให้ผู้ใช้ป้อนวลีที่นิยมหนึ่งวลีต่อบรรทัด ซึ่งใช้งานง่ายมาก การขอให้ผู้ใช้ป้อนหนึ่งคำต่อบรรทัดนั้นง่ายกว่าการทำให้พวกเขาเข้าใจรูปแบบการกำหนดค่าที่ซับซ้อนมาก

ถัดมาคือฟังก์ชันการตรวจสอบความถูกต้อง ตามข้อจำกัด Doubao API เราได้กำหนดกฎการตรวจสอบที่เข้มงวด: ข้อความที่นิยมสูงสุด 100 บรรทัด สูงสุด 50 อักขระต่อบรรทัด และทั้งหมดไม่เกิน 5,000 อักขระ `boosting_table_id` สามารถมีอักขระได้สูงสุด 200 ตัวและประกอบด้วยตัวอักษร ตัวเลข ขีดล่าง และขีดกลางเท่านั้น ข้อจำกัดเหล่านี้ไม่ได้เกิดขึ้นโดยพลการ พวกเขามาจากเอกสาร Doubao อย่างเป็นทางการโดยตรง ขีดจำกัดของ API คือขีดจำกัดของ API และเราต้องปฏิบัติตาม

```typescript
export function validateContextText(contextText: string): HotwordValidationResult {
  if (!contextText || contextText.trim().length === 0) {
    return { isValid: true, errors: [] };
  }

  const lines = contextText.split('\n').filter(line => line.trim().length > 0);
  const errors: string[] = [];

  if (lines.length > 100) {
    errors.push(`Hotword line count cannot exceed 100 lines; current count is ${lines.length}`);
  }

  const totalChars = contextText.length;
  if (totalChars > 5000) {
    errors.push(`Total hotword character count cannot exceed 5000; current count is ${totalChars}`);
  }

  for (let i = 0; i < lines.length; i++) {
    if (lines[i].length > 50) {
      errors.push(`Hotword on line ${i + 1} exceeds the 50-character limit`);
    }
  }

  return { isValid: errors.length === 0, errors };
}

export function validateBoostingTableId(boostingTableId: string): HotwordValidationResult {
  if (!boostingTableId || boostingTableId.trim().length === 0) {
    return { isValid: true, errors: [] };
  }

  const errors: string[] = [];

  if (boostingTableId.length > 200) {
    errors.push(`boosting_table_id cannot exceed 200 characters; current count is ${boostingTableId.length}`);
  }

  if (!/^[a-zA-Z0-9_-]+$/.test(boostingTableId)) {
    errors.push('boosting_table_id can contain only letters, numbers, underscores, and hyphens');
  }

  return { isValid: errors.length === 0, errors };
}
```

ฟังก์ชันการตรวจสอบความถูกต้องเหล่านี้จะทำงานทันทีเมื่อผู้ใช้กำหนดค่าคำที่นิยม เพื่อให้มั่นใจว่าปัญหาจะได้รับการแก้ไขโดยเร็วที่สุด จากมุมมองของประสบการณ์ผู้ใช้ การตอบรับทันทีประเภทนี้มีความสำคัญมาก จะดีกว่าเสมอสำหรับผู้ใช้ที่จะรู้ว่ามีอะไรผิดปกติในขณะที่พิมพ์มากกว่าหลังจากที่ส่งไปแล้ว

### ความคงอยู่ของการกำหนดค่าส่วนหน้า

ในการใช้งานส่วนหน้าของ HagiCode เราเลือกใช้เบราว์เซอร์ `localStorage` เพื่อจัดเก็บการกำหนดค่าคำที่นิยม มีข้อควรพิจารณาหลายประการที่อยู่เบื้องหลังการตัดสินใจออกแบบนี้ ประการแรก การกำหนดค่าคำที่นิยมนั้นมีความเป็นส่วนตัวสูง และผู้ใช้ที่แตกต่างกันอาจมีความต้องการเฉพาะโดเมนที่แตกต่างกัน ประการที่สอง แนวทางนี้ทำให้การใช้งานแบ็กเอนด์ง่ายขึ้น เนื่องจากไม่ต้องการตารางฐานข้อมูลเพิ่มเติมหรือตำแหน่งข้อมูล API ในที่สุด หลังจากที่ผู้ใช้กำหนดค่าเพียงครั้งเดียวในเบราว์เซอร์ การตั้งค่าต่างๆ ก็สามารถโหลดได้โดยอัตโนมัติในการใช้งานครั้งต่อไป ซึ่งสะดวกมาก พูดง่ายๆ ว่าเป็นแนวทางที่ง่ายที่สุด

```typescript
const HOTWORD_STORAGE_KEYS = {
  contextText: 'hotword-context-text',
  boostingTableId: 'hotword-boosting-table-id',
  combineMode: 'hotword-combine-mode',
} as const;

export const DEFAULT_HOTWORD_CONFIG: HotwordConfig = {
  contextText: '',
  boostingTableId: '',
  combineMode: false,
};

// Load hotword configuration
export function loadHotwordConfig(): HotwordConfig {
  const contextText = localStorage.getItem(HOTWORD_STORAGE_KEYS.contextText) || '';
  const boostingTableId = localStorage.getItem(HOTWORD_STORAGE_KEYS.boostingTableId) || '';
  const combineMode = localStorage.getItem(HOTWORD_STORAGE_KEYS.combineMode) === 'true';

  return { contextText, boostingTableId, combineMode };
}

// Save hotword configuration
export function saveHotwordConfig(config: HotwordConfig): void {
  localStorage.setItem(HOTWORD_STORAGE_KEYS.contextText, config.contextText);
  localStorage.setItem(HOTWORD_STORAGE_KEYS.boostingTableId, config.boostingTableId);
  localStorage.setItem(HOTWORD_STORAGE_KEYS.combineMode, String(config.combineMode));
}
```

ตรรกะในโค้ดนี้ตรงไปตรงมาและชัดเจน เราอ่านจาก `localStorage` เมื่อโหลดการกำหนดค่าและเขียนถึง `localStorage` เมื่อทำการบันทึก นอกจากนี้เรายังมีการกำหนดค่าเริ่มต้นเพื่อให้ระบบยังคงสามารถทำงานได้อย่างถูกต้องเมื่อยังไม่มีการกำหนดค่าใดๆ จะต้องมีค่าเริ่มต้นที่สมเหตุสมผล

### ส่วนขยายการกำหนดค่า SDK แบ็กเอนด์

ในการใช้งานแบ็กเอนด์ของ HagiCode เราจำเป็นต้องเพิ่มคุณสมบัติที่เกี่ยวข้องกับคำที่นิยมให้กับคลาสการกำหนดค่า SDK เราใช้โดยคำนึงถึงคุณลักษณะของภาษา C# และรูปแบบการใช้งาน `List<string>` เพื่อจัดเก็บบริบทคำที่นิยมที่กำหนดเอง:

```csharp
public class DoubaoVoiceConfig
{
    /// <summary>
    /// App ID
    /// </summary>
    public string AppId { get; set; } = string.Empty;

    /// <summary>
    /// Access token
    /// </summary>
    public string AccessToken { get; set; } = string.Empty;

    /// <summary>
    /// Service URL
    /// </summary>
    public string ServiceUrl { get; set; } = string.Empty;

    /// <summary>
    /// Custom hotword context list
    /// </summary>
    public List<string>? HotwordContexts { get; set; }

    /// <summary>
    /// Doubao platform hotword table ID
    /// </summary>
    public string? BoostingTableId { get; set; }
}
```

การออกแบบคลาสการกำหนดค่านี้เป็นไปตามสไตล์ที่กระชับตามปกติของ HagiCode `HotwordContexts` เป็นประเภทรายการที่เป็นโมฆะและ `BoostingTableId` เป็นสตริงที่เป็นโมฆะ ดังนั้นเมื่อไม่มีการกำหนดค่าคำที่นิยม คุณสมบัติเหล่านี้จะไม่มีผลกระทบต่อคำขอเลย หากคุณไม่ได้ใช้คุณสมบัตินี้ ก็ควรหลีกเลี่ยงไม่ให้เกะกะ

### ลอจิกการก่อสร้างเพย์โหลด

โครงสร้างเพย์โหลดเป็นแกนหลักของฟีเจอร์คำที่นิยมทั้งหมด เมื่อเรามีการกำหนดค่าคำที่นิยมแล้ว เราจำเป็นต้องจัดรูปแบบให้เป็นโครงสร้าง JSON ที่กำหนดโดย Doubao API กระบวนการนี้เกิดขึ้นก่อนที่ SDK จะส่งคำขอ:

```csharp
private void AddCorpusToRequest(Dictionary<string, object> request)
{
    var corpus = new Dictionary<string, object>();

    // Add custom hotwords
    if (Config.HotwordContexts != null && Config.HotwordContexts.Count > 0)
    {
        corpus["context"] = new Dictionary<string, object>
        {
            ["context_type"] = "dialog_ctx",
            ["context_data"] = Config.HotwordContexts
                .Select(text => new Dictionary<string, object> { ["text"] = text })
                .ToList()
        };
    }

    // Add platform hotword table ID
    if (!string.IsNullOrEmpty(Config.BoostingTableId))
    {
        corpus["boosting_table_id"] = Config.BoostingTableId;
    }

    // Add corpus to the request only when it is not empty
    if (corpus.Count > 0)
    {
        request["corpus"] = corpus;
    }
}
```

รหัสนี้แสดงวิธีการสร้างแบบไดนามิก `corpus` ฟิลด์ตามการกำหนดค่า ประเด็นสำคัญคือเราเพิ่ม `corpus` ฟิลด์เฉพาะเมื่อมีการกำหนดค่าคำที่นิยมจริงเท่านั้น การออกแบบนี้รับประกันความเข้ากันได้แบบย้อนหลัง: เมื่อไม่ได้กำหนดค่าคำที่นิยม โครงสร้างคำขอจะยังคงเหมือนเดิมทุกประการ ความเข้ากันได้แบบย้อนหลังมีความสำคัญ การเพิ่มคุณลักษณะไม่ควรรบกวนตรรกะที่มีอยู่

### การส่งผ่านพารามิเตอร์ WebSocket

ระหว่างส่วนหน้าและส่วนหลัง พารามิเตอร์คำที่นิยมจะถูกส่งผ่านข้อความควบคุม WebSocket HagiCode ได้รับการออกแบบเพื่อให้เมื่อส่วนหน้าเริ่มบันทึก จะโหลดการกำหนดค่าคำที่นิยม `localStorage` และส่งไปยังแบ็กเอนด์ผ่านข้อความ WebSocket

```typescript
const controlMessage = {
  type: 'control',
  payload: {
    command: 'StartRecognition',
    contextText: '高血压\n糖尿病\n冠心病',
    boosting_table_id: 'medical_table',
    combineMode: false
  }
};
```

มีรายละเอียดหนึ่งที่ควรทราบที่นี่: ส่วนหน้าส่งข้อความหลายบรรทัดโดยคั่นด้วยอักขระขึ้นบรรทัดใหม่ และส่วนหลังจำเป็นต้องแยกวิเคราะห์ ตัวจัดการ WebSocket แบ็กเอนด์จะแยกวิเคราะห์พารามิเตอร์เหล่านี้และส่งผ่านไปยัง SDK:

```csharp
private async Task HandleControlMessageAsync(
    string connectionId,
    DoubaoSession session,
    ControlMessage message)
{
    if (message.Payload is SessionControlRequest controlRequest)
    {
        // Parse hotword parameters
        string? contextText = controlRequest.ContextText;
        string? boostingTableId = controlRequest.BoostingTableId;
        bool? combineMode = controlRequest.CombineMode;

        // Parse multi-line text into a hotword list
        if (!string.IsNullOrEmpty(contextText))
        {
            var hotwords = contextText
                .Split('\n', StringSplitOptions.RemoveEmptyEntries)
                .Select(s => s.Trim())
                .Where(s => s.Length > 0)
                .ToList();

            session.HotwordContexts = hotwords;
        }

        session.BoostingTableId = boostingTableId;
    }
}
```

ด้วยการออกแบบนี้ การส่งการกำหนดค่าคำที่นิยมจากส่วนหน้าไปยังส่วนหลังจึงมีความชัดเจนและมีประสิทธิภาพ ไม่มีอะไรลึกลับเป็นพิเศษเกี่ยวกับเรื่องนี้ ข้อมูลจะถูกส่งผ่านทีละชั้น

## คู่มือการปฏิบัติ

### กำหนดค่าคำที่นิยมที่กำหนดเอง

ในการใช้งานจริง การกำหนดค่าคำที่นิยมที่กำหนดเองนั้นง่ายมาก เปิดหน้าการตั้งค่าการรู้จำเสียงใน HagiCode และค้นหาส่วน "การกำหนดค่า Hotword" ในช่องป้อนข้อมูล "ข้อความที่นิยมที่กำหนดเอง" ให้ป้อนหนึ่งวลีที่นิยมต่อบรรทัด

ตัวอย่างเช่น หากคุณกำลังพัฒนาแอปพลิเคชันที่เกี่ยวข้องกับการแพทย์ คุณสามารถกำหนดค่าได้ดังนี้:

```
高血压
糖尿病
冠心病
心绞痛
心肌梗死
心力衰竭
```

หลังจากที่คุณบันทึกการกำหนดค่าแล้ว คำที่นิยมเหล่านี้จะถูกส่งไปที่ Doubao API โดยอัตโนมัติทุกครั้งที่เริ่มการรู้จำเสียง ในการทดสอบของเรา เมื่อกำหนดค่าคำที่นิยมแล้ว ความแม่นยำในการจดจำสำหรับคำศัพท์ทางวิชาชีพที่เกี่ยวข้องก็ดีขึ้นอย่างเห็นได้ชัด การปรับปรุงเกิดขึ้นจริงและดีขึ้นกว่าเดิมอย่างเห็นได้ชัด

### กำหนดค่าตารางคำที่นิยมแพลตฟอร์ม

หากคุณต้องการจัดการคำที่นิยมจำนวนมาก หรือหากคำที่นิยมต้องการการอัปเดตบ่อยๆ โหมดตารางคำที่นิยมของแพลตฟอร์มจะเหมาะสมกว่า ขั้นแรก สร้างตารางคำที่นิยมบนแพลตฟอร์มการเรียนรู้ด้วยตนเองของ Doubao และรับค่าที่สร้างขึ้น `boosting_table_id`จากนั้นป้อน ID นี้ในหน้าการตั้งค่า HagiCode

แพลตฟอร์มการเรียนรู้ด้วยตนเองของ Doubao มอบความสามารถต่างๆ เช่น การนำเข้าจำนวนมากและการจัดการหมวดหมู่สำหรับคำที่นิยม ซึ่งเป็นประโยชน์อย่างมากสำหรับทีมที่ต้องการจัดการคำศัพท์เฉพาะทางชุดใหญ่ ด้วยการจัดการคำที่นิยมบนแพลตฟอร์ม คุณสามารถรักษาคำเหล่านี้จากส่วนกลางและเผยแพร่การอัปเดตได้อย่างสม่ำเสมอ เมื่อรายการคำที่นิยมมีขนาดใหญ่ขึ้น การมีที่เดียวในการจัดการจะมีประโยชน์มากกว่าการป้อนด้วยตนเองทุกครั้ง

### การใช้โหมดผสมผสาน

ในบางสถานการณ์ที่ซับซ้อน คุณอาจจำเป็นต้องใช้ทั้งคำที่นิยมที่กำหนดเองและตารางคำที่นิยมของแพลตฟอร์มในเวลาเดียวกัน ในกรณีนั้น เพียงกำหนดค่าทั้งใน HagiCode และเปิดใช้งานสวิตช์ "Combination Mode"

ในโหมดผสม Doubao API จะพิจารณาแหล่งที่มาของคำที่นิยมทั้งสองพร้อมกัน ดังนั้นความแม่นยำในการจดจำจึงมักจะสูงกว่าการใช้แหล่งที่มาใดแหล่งหนึ่งเพียงอย่างเดียว อย่างไรก็ตาม เป็นที่น่าสังเกตว่าโหมดผสมผสานจะเพิ่มความซับซ้อนของคำขอ ดังนั้นจึงควรตัดสินใจว่าจะเปิดใช้งานหรือไม่หลังจากการทดสอบภาคปฏิบัติ ความซับซ้อนที่มากขึ้นจะคุ้มค่าก็ต่อเมื่อผลลัพธ์ในโลกแห่งความเป็นจริงพิสูจน์ได้

### ตัวอย่างการรวมโค้ด

การรวมฟีเจอร์คำที่นิยมเข้ากับโปรเจ็กต์ HagiCode นั้นตรงไปตรงมามาก ต่อไปนี้คือตัวอย่างโค้ดที่ใช้กันทั่วไปบางส่วน:

```typescript
import {
  loadHotwordConfig,
  saveHotwordConfig,
  validateHotwordConfig,
  parseContextText,
  getEffectiveHotwordMode,
  type HotwordConfig
} from '@/types/hotword';

// Load and validate configuration
const config = loadHotwordConfig();
const validation = validateHotwordConfig(config);

if (!validation.isValid) {
  console.error('Hotword configuration validation failed:', validation.errors);
  return;
}

// Parse hotword text
const hotwords = parseContextText(config.contextText);
console.log('Parsed hotwords:', hotwords);

// Get effective hotword mode
const mode = getEffectiveHotwordMode(config);
console.log('Current hotword mode:', mode);
```

การใช้งานแบ็กเอนด์มีความกระชับในทำนองเดียวกัน:

```csharp
var config = new DoubaoVoiceConfig
{

        "高血压",
```

## สิ่งที่ควรคำนึงถึง

มีหลายจุดที่สมควรได้รับความสนใจเป็นพิเศษเมื่อใช้งานและใช้งานฟีเจอร์คำที่นิยม

ประการแรกคือจำนวนอักขระสูงสุด Doubao API มีข้อจำกัดที่เข้มงวดเกี่ยวกับคำที่นิยม รวมถึงจำนวนบรรทัด อักขระต่อบรรทัด และจำนวนอักขระทั้งหมด หากเกินขีดจำกัด API จะส่งคืนข้อผิดพลาด ในการใช้งานส่วนหน้าของ HagiCode เราจะตรวจสอบข้อจำกัดเหล่านี้ระหว่างที่ผู้ใช้ป้อนข้อมูลผ่านฟังก์ชันการตรวจสอบ ซึ่งจะป้องกันไม่ให้การกำหนดค่าที่ไม่ถูกต้องถูกส่งไปยังแบ็กเอนด์ การตรวจจับปัญหาตั้งแต่เนิ่นๆ ดีกว่าการรอให้ API ล้มเหลวเสมอ

ประการที่สองคือรูปแบบของ `boosting_table_id`. ฟิลด์นี้อนุญาตเฉพาะตัวอักษร ตัวเลข ขีดล่าง และขีดกลางเท่านั้น และไม่สามารถมีการเว้นวรรคหรืออักขระพิเศษอื่นๆ ได้ เมื่อสร้างตารางคำที่นิยมบนแพลตฟอร์มการเรียนรู้ด้วยตนเอง Doubao ต้องแน่ใจว่าได้ปฏิบัติตามกฎการตั้งชื่อ การตรวจสอบความถูกต้องของรูปแบบที่เข้มงวดเช่นนั้นถือเป็นเรื่องปกติสำหรับ API

ประการที่สามคือความเข้ากันได้แบบย้อนหลัง พารามิเตอร์คำที่นิยมเป็นทางเลือกทั้งหมด หากไม่มีการกำหนดค่าคำที่นิยม ระบบจะทำงานเหมือนเดิมทุกประการ การออกแบบนี้ช่วยให้แน่ใจว่าผู้ใช้ที่มีอยู่จะไม่ได้รับผลกระทบใดๆ และยังทำให้การโยกย้ายและอัปเกรดทีละน้อยง่ายขึ้น การเพิ่มคุณลักษณะไม่ควรรบกวนตรรกะก่อนหน้านี้

ในที่สุดก็มีการจัดการข้อผิดพลาด เมื่อการกำหนดค่าคำที่นิยมไม่ถูกต้อง Doubao API จะส่งคืนข้อความแสดงข้อผิดพลาดที่เกี่ยวข้อง การใช้งานของ HagiCode จะบันทึกบันทึกโดยละเอียดเพื่อช่วยนักพัฒนาในการแก้ไขปัญหา ในขณะเดียวกัน ส่วนหน้าจะแสดงข้อผิดพลาดในการตรวจสอบความถูกต้องใน UI เพื่อช่วยให้ผู้ใช้แก้ไขการกำหนดค่าได้ การจัดการข้อผิดพลาดที่ดีย่อมนำไปสู่ประสบการณ์ผู้ใช้ที่ดีขึ้น

## บทสรุป

ในบทความนี้ เราได้ให้ข้อมูลเบื้องต้นโดยละเอียดเกี่ยวกับโซลูชันที่สมบูรณ์สำหรับการนำคำที่นิยมการรู้จำเสียง Doubao ไปใช้งานในโครงการ HagiCode โซลูชันนี้ครอบคลุมกระบวนการทั้งหมดตั้งแต่การวิเคราะห์ความต้องการและการเลือกทางเทคนิคไปจนถึงการนำโค้ดไปใช้ ทำให้นักพัฒนามีตัวอย่างเชิงปฏิบัติที่พวกเขาสามารถใช้เพื่ออ้างอิงได้

ประเด็นสำคัญสามารถสรุปได้ดังนี้ ประการแรก Doubao API รองรับทั้งคำที่นิยมที่กำหนดเองและตารางคำที่นิยมของแพลตฟอร์ม และสามารถใช้แยกกันหรือรวมกันก็ได้ ประการที่สอง ส่วนหน้าใช้ `localStorage` เพื่อจัดเก็บการกำหนดค่าด้วยวิธีที่ง่ายและมีประสิทธิภาพ ประการที่สาม แบ็กเอนด์ส่งผ่านพารามิเตอร์คำที่นิยมโดยการสร้างแบบไดนามิก `corpus` สนาม โดยรักษาความเข้ากันได้แบบย้อนหลังที่แข็งแกร่ง ประการที่สี่ ตรรกะการตรวจสอบที่ครอบคลุมช่วยให้มั่นใจว่าการกำหนดค่าถูกต้องและหลีกเลี่ยงคำขอที่ไม่ถูกต้อง โดยรวมแล้วการแก้ปัญหาไม่ซับซ้อน เพียงปฏิบัติตามข้อกำหนด API อย่างระมัดระวัง

การใช้ฟีเจอร์คำที่นิยมจะช่วยเพิ่มความแข็งแกร่งให้กับความสามารถของ HagiCode ในโดเมนการรู้จำเสียงพูด ด้วยการกำหนดค่าข้อกำหนดทางวิชาชีพที่เกี่ยวข้องกับธุรกิจอย่างยืดหยุ่น นักพัฒนาสามารถช่วยให้ระบบรู้จำเสียงเข้าใจเนื้อหาจากโดเมนเฉพาะได้ดีขึ้น และให้บริการที่แม่นยำยิ่งขึ้น ท้ายที่สุดแล้ว เทคโนโลยีควรตอบสนองความต้องการทางธุรกิจที่แท้จริง และการแก้ปัญหาเชิงปฏิบัติคือสิ่งที่สำคัญที่สุด

หากคุณพบว่าบทความนี้มีประโยชน์ คุณสามารถติดดาว HagiCode บน GitHub ได้เลย การสนับสนุนของคุณเป็นแรงบันดาลใจให้เราแบ่งปันแนวทางปฏิบัติและประสบการณ์ด้านเทคนิคต่อไป ท้ายที่สุดแล้ว การเขียนและแบ่งปันเนื้อหาทางเทคนิคที่ช่วยเหลือผู้อื่นก็เป็นเรื่องที่น่าพึงพอใจในตัวมันเอง
