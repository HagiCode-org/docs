## การซิงโครไนซ์รูปภาพอัตโนมัติจาก Docker Hub ไปยัง Azure ACR

> บทความนี้จะอธิบายวิธีใช้ GitHub Actions และเครื่องมือซิงค์รูปภาพเพื่อทำให้การซิงโครไนซ์รูปภาพจาก Docker Hub ไปยัง Azure Container Registry เป็นแบบอัตโนมัติ จัดการกับการเข้าถึง Docker Hub ในประเทศจีนและภูมิภาค Azure บางส่วนได้ช้า ในขณะเดียวกันก็ปรับปรุงความพร้อมใช้งานของรูปภาพและประสิทธิภาพการปรับใช้ในสภาพแวดล้อม Azure

## ความเป็นมา / บทนำ

โปรเจ็กต์ HagiCode ใช้อิมเมจ Docker เป็นส่วนประกอบรันไทม์หลัก โดยอิมเมจหลักโฮสต์บน Docker Hub เมื่อโครงการเติบโตขึ้นและความต้องการใช้งาน Azure เพิ่มขึ้น เราพบปัญหาต่อไปนี้:

- ความเร็วในการดึงรูปภาพช้า โดยจำกัดการเข้าถึง Docker Hub ในจีนและภูมิภาค Azure บางแห่ง
- แหล่งที่มาของรูปภาพเพียงแหล่งเดียวทำให้เกิดความเสี่ยงต่อความล้มเหลวเพียงจุดเดียว
- Azure Container Registry มอบประสิทธิภาพเครือข่ายและการบูรณาการที่ดีขึ้นเมื่อทำงานใน Azure

เพื่อแก้ไขปัญหาเหล่านี้ เราจำเป็นต้องมีกลไกการซิงโครไนซ์รูปภาพอัตโนมัติเพื่อซิงค์รูปภาพจาก Docker Hub ไปยัง Azure ACR เป็นประจำ เพื่อให้มั่นใจว่าผู้ใช้จะได้รับความเร็วในการดึงรูปภาพที่เร็วขึ้นและความพร้อมใช้งานที่สูงขึ้นในสภาพแวดล้อม Azure

## เกี่ยวกับ ฮากิโค้ด

เรากำลังสร้าง HagiCode ซึ่งเป็นผู้ช่วยเขียนโค้ดที่ขับเคลื่อนด้วย AI ซึ่งทำให้การพัฒนาชาญฉลาดขึ้น สะดวกยิ่งขึ้น และสนุกสนานยิ่งขึ้น

ความชาญฉลาดหมายถึงความช่วยเหลือจาก AI ทั่วทั้งขั้นตอนการทำงาน ตั้งแต่แนวคิดไปจนถึงการเขียนโค้ด เพิ่มประสิทธิภาพการเขียนโค้ดแบบทวีคูณ สะดวกยิ่งขึ้นหมายถึงการดำเนินการพร้อมกันแบบมัลติเธรดที่ใช้ทรัพยากรอย่างเต็มที่และทำให้กระบวนการพัฒนาราบรื่น ความสนุกสนานที่มากขึ้นหมายถึงการเล่นเกมและระบบความสำเร็จที่ทำให้การเขียนโค้ดน่าเบื่อน้อยลงและคุ้มค่ามากขึ้น

โครงการกำลังทำซ้ำอย่างรวดเร็ว หากคุณสนใจในการเขียนเชิงเทคนิค การจัดการความรู้ หรือการพัฒนาโดยใช้ AI ยินดีต้อนรับเข้าสู่ GitHub

## การเปรียบเทียบโซลูชันทางเทคนิค

เมื่อกำหนดโซลูชัน เราได้เปรียบเทียบตัวเลือกทางเทคนิคหลายตัวเลือก:

### 1. image-syncer (ตัวเลือกสุดท้าย)
- การซิงโครไนซ์แบบเพิ่มหน่วย: ซิงค์เฉพาะเลเยอร์ภาพที่เปลี่ยนแปลง ซึ่งช่วยลดการถ่ายโอนข้อมูลผ่านเครือข่ายได้อย่างมาก
- ดำเนินการสนับสนุนต่อ: การซิงโครไนซ์สามารถดำเนินต่อไปได้หลังจากการหยุดชะงักของเครือข่าย
- การควบคุมการทำงานพร้อมกัน: เธรดผู้ปฏิบัติงานที่กำหนดค่าได้ช่วยปรับปรุงประสิทธิภาพการซิงโครไนซ์รูปภาพขนาดใหญ่
- การจัดการข้อผิดพลาดที่มีประสิทธิภาพ: กลไกการลองใหม่ในตัวสำหรับความล้มเหลว (ลองใหม่ 3 ครั้งตามค่าเริ่มต้น)
- การปรับใช้แบบน้ำหนักเบา: ไบนารีเดี่ยวที่ไม่มีการขึ้นต่อกัน
- รองรับหลายรีจิสทรี: เข้ากันได้กับ Docker Hub, Azure ACR, Harbor และอีกมากมาย

### 2. นักเทียบท่า CLI
- ไม่มีการซิงโครไนซ์แบบเพิ่มหน่วย: ต้องดึงเนื้อหาภาพเต็มทุกครั้ง
- ประสิทธิภาพที่ต่ำกว่า: ปริมาณการถ่ายโอนเครือข่ายขนาดใหญ่และเวลาดำเนินการนานขึ้น
- ใช้งานง่าย: ขึ้นอยู่กับความคุ้นเคย `docker pull` และ `docker push` คำสั่ง

### 3. Azure CLI
- ความซับซ้อนที่สูงขึ้น: ต้องกำหนดค่าการรับรองความถูกต้องของ Azure CLI
- ฟังก์ชั่นที่จำกัด: `az acr import` มีขอบเขตค่อนข้างแคบ
- บูรณาการแบบเนทีฟ: ผสานรวมกับบริการ Azure ได้ดี

## การตัดสินใจออกแบบสถาปัตยกรรม

### การตัดสินใจที่ 1: ตั้งค่าความถี่ในการซิงโครไนซ์เป็นรายวัน เวลา 00:00 UTC
- ปรับสมดุลความสดของภาพและการใช้ทรัพยากร
- หลีกเลี่ยงชั่วโมงเร่งด่วนทางธุรกิจและลดผลกระทบต่อการดำเนินงานอื่นๆ
- โดยทั่วไปแล้ว อิมเมจ Docker Hub จะได้รับการอัปเดตหลังจากบิวด์รายวัน

### การตัดสินใจที่ 2: ซิงโครไนซ์แท็กรูปภาพทั้งหมด
- รักษาความสอดคล้องอย่างสมบูรณ์กับ Docker Hub
- ให้ผู้ใช้มีตัวเลือกเวอร์ชันที่ยืดหยุ่น
- ลดความซับซ้อนของตรรกะการซิงโครไนซ์และหลีกเลี่ยงกฎการกรองแท็กที่ซับซ้อน

### การตัดสินใจที่ 3: ใช้ความลับ GitHub เพื่อจัดเก็บข้อมูลประจำตัว
- รองรับ Native GitHub Actions พร้อมความปลอดภัยที่แข็งแกร่ง
- การกำหนดค่าที่เรียบง่ายและการบำรุงรักษาง่าย
- รองรับการควบคุมการเข้าถึงระดับพื้นที่เก็บข้อมูล

## การประเมินความเสี่ยงและการบรรเทาผลกระทบ

### ความเสี่ยงที่ 1: การรั่วไหลของข้อมูลรับรอง Azure ACR
- จัดเก็บข้อมูลรับรองที่เข้ารหัสใน GitHub Secrets
- หมุนเวียนรหัสผ่าน ACR เป็นประจำ
- จำกัดบัญชี ACR ให้อนุญาตแบบพุชเท่านั้น
- ตรวจสอบบันทึกการเข้าถึง ACR

### ความเสี่ยงที่ 2: การซิงโครไนซ์ล้มเหลวทำให้เกิดภาพไม่สอดคล้องกัน
- image-syncer มีกลไกการซิงโครไนซ์แบบเพิ่มหน่วยในตัว
- ลองใหม่อัตโนมัติเมื่อเกิดความล้มเหลว (3 ตามค่าเริ่มต้น)
- บันทึกข้อผิดพลาดโดยละเอียดและการแจ้งเตือนความล้มเหลว
- ความสามารถในการดำเนินการต่อ

### ความเสี่ยง 3: การใช้ทรัพยากรมากเกินไป
- การซิงโครไนซ์แบบเพิ่มหน่วยช่วยลดการถ่ายโอนเครือข่าย
- การทำงานพร้อมกันที่กำหนดค่าได้ (`10` ในการตั้งค่าปัจจุบัน)
- ตรวจสอบจำนวนและขนาดของภาพที่ซิงโครไนซ์
- เรียกใช้การซิงโครไนซ์ในช่วงเวลาที่มีการใช้งานน้อย

## โซลูชั่นหลัก

เราใช้โซลูชันอัตโนมัติที่ใช้ GitHub Actions + ตัวซิงค์รูปภาพเพื่อซิงโครไนซ์รูปภาพจาก Docker Hub ไปยัง Azure ACR

## ขั้นตอนการดำเนินการ

### 1. การเตรียมการ
- สร้างหรือยืนยัน Azure Container Registry ใน Azure Portal
- สร้างข้อมูลรับรองการเข้าถึง ACR (ชื่อผู้ใช้และรหัสผ่าน)
- ยืนยันสิทธิ์การเข้าถึงพื้นที่เก็บข้อมูลอิมเมจ Docker Hub

### 2. กำหนดค่าความลับ GitHub
เพิ่มข้อมูลลับต่อไปนี้ในการตั้งค่าพื้นที่เก็บข้อมูล GitHub:
- AZURE_ACR_USERNAME: ชื่อผู้ใช้ Azure ACR
- AZURE_ACR_PASSWORD: รหัสผ่าน Azure ACR

### 3. สร้างเวิร์กโฟลว์การดำเนินการ GitHub
กำหนดค่าเวิร์กโฟลว์ใน `.github/workflows/sync-docker-acr.yml`:
- ทริกเกอร์ตามกำหนดการ: ทุกวัน เวลา 00:00 UTC
- ทริกเกอร์ด้วยตนเอง: รองรับ `workflow_dispatch`
- ทริกเกอร์เพิ่มเติม: วิ่งเมื่อกดไปที่ `publish` สาขาเพื่อการซิงโครไนซ์ที่รวดเร็ว

### 4. ขั้นตอนการดำเนินการเวิร์กโฟลว์

| ขั้นตอน | ผู้เข้าร่วม | การดำเนินการ | คำอธิบาย |
| --- | --- | --- | --- |
| 1 | การดำเนินการ GitHub | ทริกเกอร์เวิร์กโฟลว์ | ทริกเกอร์ตามกำหนดการ การดำเนินการด้วยตนเอง หรือการพุชไปที่ `publish` สาขา |

## การดำเนินการเวิร์กโฟลว์ GitHub Actions

ด้านล่างนี้คือการกำหนดค่าเวิร์กโฟลว์จริงที่ใช้งานอยู่ (`.github/workflows/sync-docker-acr.yml`):

```yaml
name: Sync Docker Image to Azure ACR

on:
  schedule:
    - cron: "0 0 * * *" # Every day at 00:00 UTC
  workflow_dispatch: # Manual trigger
  push:
    branches: [publish]

permissions:
  contents: read

jobs:
  sync:
    runs-on: ubuntu-latest

    steps:
      - name: Checkout code
        uses: actions/checkout@v4

      - name: Download image-syncer
        run: |
          # Download the image-syncer binary
          wget https://github.com/AliyunContainerService/image-syncer/releases/download/v1.5.5/image-syncer-v1.5.5-linux-amd64.tar.gz
          tar -zxvf image-syncer-v1.5.5-linux-amd64.tar.gz
          chmod +x image-syncer

      - name: Create auth config
        run: |
          # Generate the authentication config file (YAML format)
          cat > auth.yaml <<EOF
          hagicode.azurecr.io:
            username: "${{ secrets.AZURE_ACR_USERNAME }}"
            password: "${{ secrets.AZURE_ACR_PASSWORD }}"
          EOF

      - name: Create images config
        run: |
          # Generate the image sync config file (YAML format)
          cat > images.yaml <<EOF
          docker.io/newbe36524/hagicode: hagicode.azurecr.io/hagicode
          EOF

      - name: Run image-syncer
        run: |
          # Run synchronization (using the newer --auth and --images parameters)
          ./image-syncer --auth=./auth.yaml --images=./images.yaml --proc=10 --retries=3

      - name: Upload logs
        if: always()
        uses: actions/upload-artifact@v4
        with:
          name: sync-logs
          path: image-syncer-*.log
          retention-days: 7
```

## รายละเอียดการกำหนดค่า

### 1. เงื่อนไขทริกเกอร์
- ทริกเกอร์ตามกำหนดการ: `cron: "0 0 * * *"` วิ่งทุกวันเวลา 00:00 UTC
- ทริกเกอร์แบบแมนนวล: `workflow_dispatch` อนุญาตให้ผู้ใช้เรียกใช้ด้วยตนเองจาก GitHub UI
- ทริกเกอร์: `push: branches: [publish]` ทริกเกอร์เมื่อพุชไปยังสาขาการเผยแพร่เพื่อการซิงโครไนซ์ที่รวดเร็ว

### 2. การกำหนดค่าการรับรองความถูกต้อง (`auth.yaml`)
```yaml
hagicode.azurecr.io:
  username: "${{ secrets.AZURE_ACR_USERNAME }}"
  password: "${{ secrets.AZURE_ACR_PASSWORD }}"
```

### 3. การกำหนดค่าการซิงโครไนซ์รูปภาพ
```yaml
docker.io/newbe36524/hagicode: hagicode.azurecr.io/hagicode
```
การกำหนดค่านี้หมายความว่าแท็กทั้งหมดมาจาก `docker.io/newbe36524/hagicode` จะถูกซิงโครไนซ์กับ `hagicode.azurecr.io/hagicode`.

### 4. พารามิเตอร์การซิงค์รูปภาพ
- `--auth=./auth.yaml`: เส้นทางไปยังไฟล์กำหนดค่าการตรวจสอบสิทธิ์
- `--images=./images.yaml`: พาธไปยังไฟล์กำหนดค่าการซิงโครไนซ์รูปภาพ
- `--proc=10`: การใช้งาน `10` คนงานพร้อมกัน
- `--retries=3`: ลองใหม่ล้มเหลว `3` ครั้ง

## หมายเหตุและแนวปฏิบัติที่ดีที่สุด

### 1. คำแนะนำด้านความปลอดภัย
- หมุนเวียนรหัสผ่าน Azure ACR เป็นประจำ (แนะนำทุก 90 วัน)
- ใช้บัญชีบริการ ACR เฉพาะที่มีสิทธิ์แบบพุชเท่านั้น
- ตรวจสอบบันทึกการเข้าถึง ACR เพื่อตรวจจับการเข้าถึงที่ผิดปกติทันเวลา
- อย่าส่งออกข้อมูลประจำตัวในบันทึก
- อย่าส่งข้อมูลรับรองไปยังที่เก็บโค้ด

### 2. การเพิ่มประสิทธิภาพการทำงาน
- ปรับ `--proc` พารามิเตอร์: ปรับการทำงานพร้อมกันตามแบนด์วิธเครือข่าย (แนะนำ `5-20`)
- ตรวจสอบระยะเวลาการซิงโครไนซ์: หากการซิงโครไนซ์ใช้เวลานานเกินไป ให้พิจารณาลดการทำงานพร้อมกันลง
- ทำความสะอาดบันทึกเป็นประจำ: ตั้งค่าให้สมเหตุสมผล `retention-days` ค่า (`7` ปัจจุบัน)

### 3. การแก้ไขปัญหา

#### ปัญหาที่ 1: การรับรองความถูกต้องล้มเหลว
```
Error: failed to authenticate to hagicode.azurecr.io
```
วิธีแก้ปัญหา:
1. ตรวจสอบว่า GitHub Secrets ได้รับการกำหนดค่าอย่างถูกต้องหรือไม่
2. ตรวจสอบว่ารหัสผ่าน Azure ACR หมดอายุแล้วหรือไม่
3. ยืนยันว่าสิทธิ์บัญชีบริการ ACR ถูกต้อง

#### ปัญหาที่ 2: การหมดเวลาของเครือข่าย
```
Error: timeout waiting for response
```
วิธีแก้ปัญหา:
1. ตรวจสอบการเชื่อมต่อเครือข่าย
2. ลดจำนวนคนงานพร้อมกัน (ก `--proc` พารามิเตอร์)
3. รอให้เครือข่ายกู้คืนและรันเวิร์กโฟลว์อีกครั้ง

#### ปัญหาที่ 3: การซิงโครไนซ์รูปภาพไม่สมบูรณ์
```
Warning: some tags failed to sync
```
วิธีแก้ปัญหา:
1. ตรวจสอบบันทึกการซิงโครไนซ์เพื่อระบุแท็กที่ล้มเหลว
2. รันเวิร์กโฟลว์อีกครั้งด้วยตนเองเพื่อซิงโครไนซ์อีกครั้ง
3. ตรวจสอบว่าอิมเมจต้นฉบับบน Docker Hub นั้นแข็งแรง

### 4. การติดตามและแจ้งเตือน
- ตรวจสอบหน้าการดำเนินการเป็นประจำเพื่อยืนยันสถานะเวิร์กโฟลว์
- เปิดใช้งานการแจ้งเตือน GitHub เพื่อให้รายงานความล้มเหลวของเวิร์กโฟลว์ทันที
- ตรวจสอบการใช้พื้นที่เก็บข้อมูล Azure ACR
- ตรวจสอบความสอดคล้องของแท็กในรีจิสทรีเป็นประจำ

## คำถามและแนวทางแก้ไขที่พบบ่อย

### คำถามที่ 1: ฉันจะซิงโครไนซ์แท็กเฉพาะแทนที่จะเป็นแท็กทั้งหมดได้อย่างไร

ปรับเปลี่ยน `images.yaml` ไฟล์การกำหนดค่า:
```yaml
# Sync only the latest and v1.0 tags
docker.io/newbe36524/hagicode:latest: hagicode.azurecr.io/hagicode:latest
docker.io/newbe36524/hagicode:v1.0: hagicode.azurecr.io/hagicode:v1.0
```

### คำถามที่ 2: ฉันจะซิงโครไนซ์ที่เก็บรูปภาพหลายอันได้อย่างไร

เพิ่มหลายบรรทัดเข้าไป `images.yaml`:
```yaml
docker.io/newbe36524/hagicode: hagicode.azurecr.io/hagicode
docker.io/newbe36524/another-image: hagicode.azurecr.io/another-image
```

### คำถามที่ 3: ฉันจะลองอีกครั้งหลังจากการซิงโครไนซ์ล้มเหลวได้อย่างไร
- ลองใหม่อัตโนมัติ: image-syncer มีกลไกการลองใหม่ในตัว (`3` ลองอีกครั้งตามค่าเริ่มต้น)
- ลองอีกครั้งด้วยตนเอง: คลิก `Re-run all jobs` บนหน้าการดำเนินการ GitHub

### คำถามที่ 4: ฉันจะดูความคืบหน้าของการซิงโครไนซ์โดยละเอียดได้อย่างไร
- ดูบันทึกแบบเรียลไทม์บนหน้าการดำเนินการ
- ดาวน์โหลด `sync-logs` สิ่งประดิษฐ์เพื่อตรวจสอบไฟล์บันทึกทั้งหมด
- ไฟล์บันทึกประกอบด้วยสถานะการซิงโครไนซ์และความเร็วในการถ่ายโอนสำหรับแต่ละแท็ก

### Q5: การซิงโครไนซ์ใช้เวลานานเท่าใด?
- การซิงโครไนซ์แบบเต็มครั้งแรก: ปกติ `10-30` นาที ขึ้นอยู่กับขนาดภาพ
- การซิงโครไนซ์แบบเพิ่มหน่วย: ปกติ `2-5` นาทีหากภาพมีการเปลี่ยนแปลงเล็กน้อย
- ระยะเวลาขึ้นอยู่กับแบนด์วิดท์เครือข่าย ขนาดภาพ และการตั้งค่าการทำงานพร้อมกัน

## ส่วนขยายที่แนะนำ

### 1. เพิ่มการแจ้งเตือนการซิงโครไนซ์
เพิ่มขั้นตอนการแจ้งเตือนลงในเวิร์กโฟลว์:
```yaml
- name: Notify on success
  if: success()
  run: |
    echo "Docker images synced successfully to Azure ACR"
```

### 2. ใช้การกรองแท็กรูปภาพ
เพิ่มตรรกะการกรองแท็กให้กับเวิร์กโฟลว์:
```yaml
- name: Filter tags
  run: |
    # Sync only tags prefixed with v
    echo "docker.io/newbe36524/hagicode:v* : hagicode.azurecr.io/hagicode:v*" > images.yaml
```

### 3. เพิ่มรายงานสถิติการซิงโครไนซ์
```yaml
- name: Generate report
  if: always()
  run: |
    echo "## Sync Report" >> $GITHUB_STEP_SUMMARY
    echo "- Total tags: $(grep -c 'synced' image-syncer-*.log)" >> $GITHUB_STEP_SUMMARY
    echo "- Sync time: ${{ steps.sync.outputs.duration }}" >> $GITHUB_STEP_SUMMARY
```
