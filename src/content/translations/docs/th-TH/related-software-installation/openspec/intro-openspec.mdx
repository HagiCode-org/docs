# บทนำสู่ OpenSpec

OpenSpec เป็นเครื่องมือพัฒนาแบบขับเคลื่อนด้วยสเปคที่ช่วยให้ทีมงานจัดการกระบวนการพัฒนาซอฟต์แวร์ผ่านข้อเสนอและสเปคที่มีโครงสร้าง บทความนี้ให้การแนะนำอย่างครอบคลุมเกี่ยวกับแนวคิดหลัก กระบวนงาน และวิธีการใช้งาน OpenSpec

## OpenSpec คืออะไร?

OpenSpec เป็นเครื่องมือบรรทัดคำสั่งที่ใช้จัดการข้อเสนอ (Proposals) สเปค (Specifications) และการเปลี่ยนแปลง (Changes) ในกระบวนการพัฒนาซอฟต์แวร์ มันช่วยมาตรฐานกระบวนการพัฒนาและรับประกันว่าแต่ละฟีเจอร์จะผ่านการออกแบบและตรวจทานอย่างละเอียดตั้งแต่แนวคิดจนถึงการนำไปใช้

### หลักการสำคัญ

OpenSpec มีหลักการสำคัญดังต่อไปนี้:

- **สเปคมาก่อน**: กำหนดสเปคที่ชัดเจนก่อนเขียนโค้ด
- **การทำงานร่วมกันอย่างมีโครงสร้าง**: ส่งเสริมการทำงานร่วมกันของทีมผ่านรูปแบบเอกสารที่เป็นมาตรฐาน
- **การติดตามย้อนกลับได้**: บันทึกกระบวนการทั้งหมดตั้งแต่ข้อเสนอจนถึงการนำไปใช้
- **การประกันคุณภาพ**: ยืนยันว่าโค้ดสอดคล้องกับเจตนาการออกแบบผ่านการตรวจสอบสเปค

### ฟีเจอร์หลัก

OpenSpec ให้ฟีเจอร์หลักดังต่อไปนี้:

1. **การจัดการข้อเสนอ**: สร้าง ตรวจทาน และติดตามข้อเสนอฟีเจอร์
2. **การกำหนดสเปค**: เขียนและดูแลเอกสารสเปคทางเทคนิค
3. **การติดตามการเปลี่ยนแปลง**: บันทึกประวัติของการเปลี่ยนแปลงที่ผ่านการตรวจทานทั้งหมด
4. **การตรวจสอบสเปค**: ยืนยันว่าการเปลี่ยนแปลงของโค้ดตรงตามสเปคที่กำหนดไว้
5. **การทำงานร่วมกันของทีม**: รองรับกระบวนการพัฒนาและการตรวจทานที่มีหลายผู้เข้าร่วม

## กระบวนงานของ OpenSpec

กระบวนงานทั่วไปของ OpenSpec ประกอบด้วยขั้นตอนดังต่อไปนี้:

### 1. สร้างข้อเสนอ (Proposal)

เมื่อมีฟีเจอร์ใหม่หรือข้อกำหนดการเปลี่ยนแปลงครั้งใหญ่ ให้สร้างข้อเสนอก่อน:

```bash
openspec proposal create
```

ข้อเสนอประกอบด้วย:
- **คำอธิบายปัญหา**: ปัญหาหรือข้อกำหนดที่ต้องแก้ไข
- **แนวทางแก้ไข**: ภาพรวมของแนวทางที่เสนอ
- **ขอบเขตผลกระทบ**: ผลกระทบต่อระบบที่มีอยู่
- **ตัวเลือกอื่น ๆ**: ทางเลือกอื่นที่พิจารณาแล้ว

### 2. เขียนสเปค (Specification)

หลังจากข้อเสนอได้รับการอนุมัติ ให้เขียนสเปคทางเทคนิคโดยละเอียด:

```bash
openspec spec create
```

สเปคประกอบด้วย:
- **การออกแบบทางเทคนิค**: แผนการดำเนินการทางเทคนิคอย่างละเอียด
- **การกำหนดอินเทอร์เฟซ**: การออกแบบอินเทอร์เฟซเช่น API โครงสร้างข้อมูล ฯลฯ
- **รายละเอียดการใช้งาน**: ลอจิกการใช้งานหลักและอัลกอริทึม
- **แผนการทดสอบ**: วิธีการยืนยันความถูกต้องของการใช้งาน

### 3. สร้างการเปลี่ยนแปลง (Change)

เมื่อเริ่มการพัฒนา ให้สร้างการเปลี่ยนแปลงเพื่อติดตามกระบวนการ:

```bash
openspec change create
```

การเปลี่ยนแปลงประกอบด้วย:
- **สเปคที่เกี่ยวข้อง**: สเปคที่การเปลี่ยนแปลงนี้นำไปใช้งาน
- **งานการพัฒนา**: รายการงานการพัฒนาเฉพาะ
- **การติดตามความคืบหน้า**: สถานะการทำงานของงานต่าง ๆ
- **การเปลี่ยนแปลงโค้ด**: การปรับเปลี่ยนโค้ดจริง

### 4. การตรวจสอบและการเก็บถาวร

เมื่อพัฒนาสำเร็จ ให้ตรวจสอบว่าการเปลี่ยนแปลงสอดคล้องกับสเปคหรือไม่ จากนั้นทำการเก็บถาวร:

```bash
openspec change validate
openspec change archive
```

## โครงสร้างไดเรกทอรี OpenSpec

OpenSpec สร้างไดเรกทอรี `openspec/` ในไดเรกทอรีรูทของโปรเจกต์โดยมีโครงสร้างดังนี้:

```
openspec/
├── proposals/           # โฟลเดอร์ข้อเสนอ
│   ├── PROPOSAL-001.md
│   └── PROPOSAL-002.md
├── specs/              # โฟลเดอร์สเปค
│   ├── SPEC-001.md
│   └── SPEC-002.md
├── changes/            # โฟลเดอร์การเปลี่ยนแปลง
│   ├── feature-name/
│   │   ├── proposal.md
│   │   ├── design.md
│   │   ├── specs.md
│   │   └── tasks.md
│   └── another-feature/
└── archived/           # การเปลี่ยนแปลงที่เก็บถาวร
    └── feature-name/
```

### คำอธิบายไดเรกทอรี

- **proposals/**: เก็บเอกสารข้อเสนอทั้งหมด แต่ละข้อเสนอเป็นไฟล์ Markdown แยก
- **specs/**: เก็บเอกสารสเปคทางเทคนิคที่สอดคล้องกับข้อเสนอ
- **changes/**: เก็บการเปลี่ยนแปลงที่กำลังดำเนินการ แต่ละการเปลี่ยนแปลงอยู่ในโฟลเดอร์ย่อย
- **archived/**: เก็บการเปลี่ยนแปลงที่เสร็จสิ้นและถูกเก็บถาวร

## คำสั่งหลัก

### คำสั่งที่เกี่ยวข้องกับข้อเสนอ

```bash
# สร้างข้อเสนอใหม่
openspec proposal create

# แสดงรายการข้อเสนอทั้งหมด
openspec proposal list

# ดูรายละเอียดข้อเสนอ
openspec proposal view <proposal-id>

# อนุมัติข้อเสนอ
openspec proposal approve <proposal-id>
```

### คำสั่งที่เกี่ยวข้องกับสเปค

```bash
# สร้างสเปคใหม่
openspec spec create

# แสดงรายการสเปคทั้งหมด
openspec spec list

# ดูรายละเอียดสเปค
openspec spec view <spec-id>

# อัปเดตสเปค
openspec spec update <spec-id>
```

### คำสั่งที่เกี่ยวข้องกับการเปลี่ยนแปลง

```bash
# สร้างการเปลี่ยนแปลงใหม่
openspec change create

# แสดงรายการการเปลี่ยนแปลงทั้งหมด
openspec change list

# ดูรายละเอียดการเปลี่ยนแปลง
openspec change view <change-name>

# เพิ่มงาน
openspec change add-task <change-name>

# ตรวจสอบการเปลี่ยนแปลง
openspec change validate <change-name>

# เก็บถาวรการเปลี่ยนแปลง
openspec change archive <change-name>
```

### คำสั่งทั่วไป

```bash
# แสดงเวอร์ชัน
openspec --version

# แสดงความช่วยเหลือ
openspec --help

# แสดงความช่วยเหลือสำหรับคำสั่งเฉพาะ
openspec <command> --help
```

## การใช้ OpenSpec ในโปรเจกต์ HagiCode

### การผสานเข้ากับโปรเจกต์

ในโปรเจกต์ HagiCode OpenSpec ถูกผสานเข้ากับเวิร์กโฟลว์การพัฒนา:

1. **การตรวจสอบเมื่อติดตั้งเริ่มต้น**: ตัวช่วยเริ่มต้นจะตรวจสอบว่า OpenSpec ติดตั้งอยู่หรือไม่
2. **การพัฒนาโดยการเสนอ**: ฟีเจอร์ใหม่ทั้งหมดต้องผ่านกระบวนการข้อเสนอของ OpenSpec
3. **การตรวจทานโค้ด**: การเปลี่ยนแปลงต้องเป็นไปตามสเปค OpenSpec เพื่อที่จะถูกผสาน
4. **การซิงค์เอกสาร**: เอกสาร OpenSpec ถูกเก็บให้สอดคล้องกับการนำไปใช้ของโค้ด

### ข้อกำหนดเวอร์ชัน

โปรเจกต์ HagiCode ต้องการใช้เวอร์ชัน **OpenSpec >=1.0.0 {'<'}2.0.0**.

คำสั่งติดตั้ง:

```bash
npm install -g @fission-ai/openspec@1
```

### ตัวอย่างเวิร์กโฟลว์

ด้านล่างเป็นตัวอย่างการใช้ OpenSpec ในโปรเจกต์ HagiCode:

#### ขั้นตอนที่ 1: สร้างข้อเสนอ

```bash
cd /path/to/hagicode-project
openspec proposal create
```

ในพรอมต์แบบโต้ตอบ ให้ป้อน:
- ชื่อข้อเสนอ: เพิ่มฟีเจอร์ยืนยันตัวตนผู้ใช้
- คำอธิบายปัญหา: ระบบต้องการการเข้าสู่ระบบของผู้ใช้และการจัดการสิทธิ์
- การแก้ไข: นำระบบยืนยันตัวตนแบบ JWT มาใช้

#### ขั้นตอนที่ 2: เขียนสเปค

หลังการอนุมัติข้อเสนอ ให้สร้างสเปคโดยละเอียด:

```bash
openspec spec create
```

กำหนดในสเปค:
- การออกแบบกระบวนการยืนยันตัวตน
- การกำหนด API
- โครงสร้างตารางฐานข้อมูล
- ข้อควรพิจารณาด้านความปลอดภัย

#### ขั้นตอนที่ 3: สร้างการเปลี่ยนแปลง

เริ่มการพัฒนา:

```bash
openspec change create
```

ป้อนชื่อการเปลี่ยนแปลง: `add-user-authentication`

#### ขั้นตอนที่ 4: เพิ่มงาน

```bash
openspec change add-task add-user-authentication
```

เพิ่มงานเฉพาะ:
- ดำเนินการสร้าง endpoint ลงทะเบียนผู้ใช้
- ดำเนินการสร้าง endpoint เข้าสู่ระบบ
- ติดตั้ง JWT middleware
- เขียน unit tests

#### ขั้นตอนที่ 5: การพัฒนาและการตรวจสอบ

เมื่อโค้ดเสร็จ ให้ตรวจสอบการเปลี่ยนแปลง:

```bash
openspec change validate add-user-authentication
```

#### ขั้นตอนที่ 6: เก็บถาวร

หลังการตรวจสอบเรียบร้อย ให้เก็บถาวรการเปลี่ยนแปลง:

```bash
openspec change archive add-user-authentication
```

## แนวทางปฏิบัติที่ดีที่สุด

### การเขียนข้อเสนอ

1. **ระบุปัญหาอย่างชัดเจน**: กำหนดปัญหาที่ต้องแก้ไขอย่างชัดเจน หลีกเลี่ยงคำอธิบายที่กำกวม
2. **พิจารณาทางเลือก**: ระบุและเปรียบเทียบวิธีแก้ไขต่าง ๆ
3. **ประเมินผลกระทบ**: วิเคราะห์ผลกระทบและความเสี่ยงต่อระบบที่มีอยู่
4. **สั้นและกระชับ**: ข้อเสนอควรกระชับและชัดเจน — ปล่อยรายละเอียดไว้ในสเปค

### การเขียนสเป็ค

1. **ละเอียดแต่ไม่เกินความจำเป็น**: ให้รายละเอียดเพียงพอ แต่หลีกเลี่ยงการออกแบบเกินจำเป็น
2. **ใช้แผนภาพ**: ใช้ flowchart, แผนภาพสถาปัตยกรรม ฯลฯ เพื่อช่วยในการอธิบาย
3. **กำหนดอินเทอร์เฟซ**: กำหนดอินเทอร์เฟซสาธารณะและโครงสร้างข้อมูลอย่างชัดเจน
4. **พิจารณากรณีขอบเขต**: ระบุสถานการณ์ข้อยกเว้นและการจัดการข้อผิดพลาด

### การจัดการการเปลี่ยนแปลง

1. **ทำเป็นขั้นตอนเล็ก ๆ**: แบ่งฟีเจอร์ขนาดใหญ่เป็นการเปลี่ยนแปลงเล็ก ๆ หลายรายการ
2. **อัปเดตทันเวลา**: รักษาสถานะงานและเอกสารให้สอดคล้องเสมอ
3. **ตรวจสอบโค้ด**: ดำเนินการตรวจสอบโค้ดหลังจากการเปลี่ยนแปลงเสร็จ
4. **เก็บถาวรทันเวลา**: เก็บถาวรการเปลี่ยนแปลงที่เสร็จแล้วอย่างทันท่วงทีเพื่อรักษาความเป็นระเบียบ

### การทำงานร่วมกันในทีม

1. **การตรวจสอบเป็นประจำ**: ตรวจสอบข้อเสนอและสเปคอย่างสม่ำเสมอเพื่อให้ทีมมีความเห็นร่วมกัน
2. **เอกสารก่อนการพัฒนา**: ทำเอกสารให้เสร็จก่อนเริ่มเขียนโค้ด
3. **รักษาการสื่อสาร**: บันทึกการสนทนาและการตัดสินใจในข้อเสนอและสเปค
4. **ควบคุมเวอร์ชัน**: รวมเอกสาร OpenSpec ในระบบควบคุมเวอร์ชัน

## กรณีการใช้งานทั่วไป

### กรณีที่ 1: เพิ่มฟีเจอร์ใหม่

1. สร้างข้อเสนอที่อธิบายข้อกำหนดของฟีเจอร์ใหม่
2. ทีมอภิปรายและอนุมัติข้อเสนอ
3. เขียนสเปคทางเทคนิคโดยละเอียด
4. สร้างการเปลี่ยนแปลงและดำเนินการฟีเจอร์
5. ตรวจสอบและเก็บถาวรการเปลี่ยนแปลง

### กรณีที่ 2: รีแฟกเตอริงโค้ดที่มีอยู่

1. สร้างข้อเสนออธิบายสาเหตุและเป้าหมายของการรีแฟกเตอริง
2. กำหนดแผนการและขั้นตอนในสเปค
3. สร้างการเปลี่ยนแปลงและรีแฟกเตอริงเป็นขั้นตอน
4. ตรวจสอบความครอบคลุมด้วยเทสต์และยืนยันผลลัพธ์
5. เก็บถาวรการเปลี่ยนแปลง

### กรณีที่ 3: แก้ไขบั๊ก

สำหรับบั๊กทั่วไป:
- สามารถแก้ไขโดยตรงได้โดยไม่ต้องกระบวนการ OpenSpec ทั้งหมด

สำหรับบั๊กซับซ้อน:
1. สร้างข้อเสนอเพื่อวิเคราะห์สาเหตุรากของบั๊ก
2. กำหนดแนวทางแก้ไขในสเปค
3. สร้างการเปลี่ยนแปลงและนำแนวทางแก้ไขไปใช้
4. เพิ่มเทสต์เพื่อป้องกันการเกิด regression
5. เก็บถาวรการเปลี่ยนแปลง

### กรณีที่ 4: จัดการหนี้ทางเทคนิค

1. สร้างข้อเสนอที่ระบุรายการหนี้ทางเทคนิค
2. กำหนดแผนการปรับปรุงในสเปค
3. สร้างการเปลี่ยนแป่งเพื่อชำระหนี้ทางเทคนิคทีละน้อย
4. ติดตามความคืบหน้าและผลลัพธ์
5. เก็บถาวรการปรับปรุงที่เสร็จแล้ว

## การผสานกับเครื่องมืออื่น ๆ

### การผสานกับ Git

OpenSpec ผสานการทำงานกับ Git ได้อย่างราบรื่น:

```bash
# ควรรวมเอกสาร OpenSpec ไว้ในการควบคุมเวอร์ชัน
git add openspec/
git commit -m "Add OpenSpec proposal for feature X"
```

### การผสานกับ CI/CD

ตรวจสอบสเปค OpenSpec ใน pipeline ของ CI/CD:

```yaml
# .github/workflows/openspec.yml
name: OpenSpec Validation
on: [pull_request]
jobs:
  validate:
    runs-on: ubuntu-latest
    steps:
      - uses: actions/checkout@v2
      - name: Install OpenSpec
        run: npm install -g @fission-ai/openspec@1
      - name: Validate changes
        run: openspec change validate
```

### การผสานกับ IDE

แนะนำให้ใช้ editor ที่รองรับ Markdown ในการแก้ไขเอกสาร OpenSpec:
- VS Code + Markdown plugin
- JetBrains IDEs + การสนับสนุน Markdown
- Vim/Neovim + Markdown plugin

## การแก้ไขปัญหา

### ปัญหาทั่วไป

#### ปัญหา: ไม่พบคำสั่ง OpenSpec

**วิธีแก้ไข**:
1. ยืนยันว่า OpenSpec ติดตั้งแล้ว: `npm list -g @fission-ai/openspec`
2. ตรวจสอบว่า PATH มีไดเรกทอรี bin ของ npm ที่ติดตั้งแบบ global
3. ติดตั้งใหม่: `npm install -g @fission-ai/openspec@1`

#### ปัญหา: ไม่สามารถสร้างข้อเสนอหรือการเปลี่ยนแปลงได้

**วิธีแก้ไข**:
1. ยืนยันว่าไดเรกทอรีปัจจุบันเป็น root ของโปรเจกต์
2. ตรวจสอบว่ามีไดเรกทอรี `openspec/` หากไม่มีให้สร้าง
3. ยืนยันว่าคุณมีสิทธิ์เขียน

#### ปัญหา: การตรวจสอบล้มเหลว

**วิธีแก้ไข**:
1. ตรวจสอบว่าเอกสารในไดเรกทอรีการเปลี่ยนแปลงครบถ้วน
2. ยืนยันว่าได้ทำงานที่จำเป็นทั้งหมดเสร็จแล้ว
3. ดูข้อมูลข้อผิดพลาดโดยละเอียด: `openspec change validate <change-name> --verbose`

#### ปัญหา: เก็บถาวรไม่สำเร็จ

**วิธีแก้ไข**:
1. ยืนยันว่าเปลี่ยนแปลงผ่านการตรวจสอบแล้ว
2. ตรวจสอบว่าไดเรกทอรี `openspec/archived/` มีอยู่
3. ยืนยันว่าไม่มีการเปลี่ยนแปลงโค้ดที่ยังไม่ได้ commit

## การใช้งานขั้นสูง

### เทมเพลตที่ปรับแต่งได้

OpenSpec รองรับเทมเพลตที่กำหนดเองสำหรับข้อเสนอและสเปค:

```bash
# สร้างไดเรกทอรีเทมเพลตใน root ของโปรเจกต์
mkdir -p .openspec/templates

# สร้างเทมเพลตข้อเสนอที่กำหนดเอง
cat > .openspec/templates/proposal.md << 'EOF'
# Proposal: {{title}}

## Problem Statement
{{problem}}

## Proposed Solution
{{solution}}

## Custom Section
Your custom content here
EOF
```

### ไฟล์การตั้งค่า

สร้างไฟล์การตั้งค่า `.openspec.json` ใน root ของโปรเจกต์:

```json
{
  "version": "1.0",
  "templates": {
    "proposal": ".openspec/templates/proposal.md",
    "spec": ".openspec/templates/spec.md"
  },
  "validation": {
    "requireTests": true,
    "requireDocs": true
  },
  "hooks": {
    "pre-archive": "npm test"
  }
}
```

### Hooks

OpenSpec รองรับการรันสคริปต์ที่กำหนดเองก่อนและหลังการดำเนินการบางอย่าง:

```json
{
  "hooks": {
    "pre-create": "scripts/check-permissions.sh",
    "post-archive": "scripts/notify-team.sh"
  }
}
```
