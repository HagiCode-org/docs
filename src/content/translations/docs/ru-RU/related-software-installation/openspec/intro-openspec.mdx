# Введение в OpenSpec

OpenSpec — это инструмент разработки на основе спецификаций, который помогает командам управлять процессом разработки программного обеспечения через структурированные предложения и спецификации. Это руководство предоставляет всестороннее введение в основные концепции, рабочие процессы и методы использования OpenSpec.

## Что такое OpenSpec?

OpenSpec — это инструмент командной строки для управления предложениями (Proposals), спецификациями (Specifications) и изменениями (Changes) на протяжении всего процесса разработки программного обеспечения. Он стандартизирует рабочий процесс разработки, гарантируя, что каждая функция проходит тщательное проектирование и проверку от идеи до реализации.

### Основная философия

OpenSpec основан на следующих ключевых принципах:

- **Спецификация в первую очередь**: Определение четких спецификаций перед написанием кода
- **Структурированное сотрудничество**: Содействие командной работе через стандартизированные форматы документов
- **Отслеживаемость**: Запись полного процесса от предложения до реализации
- **Обеспечение качества**: Гарантия соответствия кода проектному замыслу через валидацию спецификаций

### Ключевые функции

OpenSpec предоставляет следующие ключевые функции:

1. **Управление предложениями**: Создание, проверка и отслеживание предложений функций
2. **Определение спецификаций**: Написание и поддержка документов технических спецификаций
3. **Отслеживание изменений**: Запись всей истории проверенных изменений
4. **Валидация спецификаций**: Обеспечение соответствия изменений кода предопределенным спецификациям
5. **Командное сотрудничество**: Поддержка процессов совместной разработки и проверки

## Рабочий процесс OpenSpec

Типичный рабочий процесс OpenSpec включает следующие этапы:

### 1. Создание предложения

Когда появляется новая функция или важное изменение, сначала создайте предложение:

```bash
openspec proposal create
```

Предложение содержит:
- **Описание проблемы**: Проблема или требование, которое нужно решить
- **Решение**: Обзор предлагаемого решения
- **Область влияния**: Влияние на существующие системы
- **Альтернативы**: Другие рассмотренные решения

### 2. Написание спецификации

После утверждения предложения напишите подробную техническую спецификацию:

```bash
openspec spec create
```

Спецификация содержит:
- **Техническое проектирование**: Подробный план технической реализации
- **Определение интерфейсов**: API, структуры данных и другие определения интерфейсов
- **Детали реализации**: Ключевая логика реализации и алгоритмы
- **План тестирования**: Как проверить корректность реализации

### 3. Создание изменения

При начале реализации создайте change для отслеживания процесса разработки:

```bash
openspec change create
```

Change содержит:
- **Связанная спецификация**: Спецификация, которую реализует это изменение
- **Задачи реализации**: Конкретный список задач разработки
- **Отслеживание прогресса**: Статус выполнения задач
- **Изменения кода**: Фактические модификации кода

### 4. Валидация и архивирование

После завершения разработки проверьте соответствие изменения спецификации, затем заархивируйте:

```bash
openspec change validate
openspec change archive
```

## Структура каталогов OpenSpec

OpenSpec создает каталог `openspec/` в корне проекта со следующей структурой:

```
openspec/
├── proposals/           # Каталог предложений
│   ├── PROPOSAL-001.md
│   └── PROPOSAL-002.md
├── specs/              # Каталог спецификаций
│   ├── SPEC-001.md
│   └── SPEC-002.md
├── changes/            # Каталог изменений
│   ├── feature-name/
│   │   ├── proposal.md
│   │   ├── design.md
│   │   ├── specs.md
│   │   └── tasks.md
│   └── another-feature/
└── archived/           # Архивированные изменения
    └── feature-name/
```

### Описание каталогов

- **proposals/**: Хранит все документы предложений, один независимый файл Markdown на предложение
- **specs/**: Хранит документы технических спецификаций, соответствующие предложениям
- **changes/**: Хранит текущие изменения, один подкаталог на change
- **archived/**: Хранит завершенные и архивированные изменения

## Основные команды

### Команды для предложений

```bash
# Создать новое предложение
openspec proposal create

# Список всех предложений
openspec proposal list

# Просмотр деталей предложения
openspec proposal view <proposal-id>

# Утвердить предложение
openspec proposal approve <proposal-id>
```

### Команды для спецификаций

```bash
# Создать новую спецификацию
openspec spec create

# Список всех спецификаций
openspec spec list

# Просмотр деталей спецификации
openspec spec view <spec-id>

# Обновить спецификацию
openspec spec update <spec-id>
```

### Команды для изменений

```bash
# Создать новое изменение
openspec change create

# Список всех изменений
openspec change list

# Просмотр деталей изменения
openspec change view <change-name>

# Добавить задачу
openspec change add-task <change-name>

# Валидировать изменение
openspec change validate <change-name>

# Архивировать изменение
openspec change archive <change-name>
```

### Общие команды

```bash
# Просмотр версии
openspec --version

# Просмотр справки
openspec --help

# Просмотр справки для конкретной команды
openspec <command> --help
```

## Использование OpenSpec в проектах HagiCode

### Интеграция проекта

В проектах HagiCode OpenSpec интегрирован в рабочий процесс разработки:

1. **Проверка инициализации**: Мастер запуска проверяет, установлен ли OpenSpec
2. **Разработка на основе предложений**: Все новые функции должны пройти через процесс предложений OpenSpec
3. **Проверка кода**: Изменения должны соответствовать спецификациям OpenSpec для слияния
4. **Синхронизация документации**: Документация OpenSpec остается синхронизированной с реализацией кода

### Требования к версии

Проекты HagiCode требуют версию **OpenSpec >=1.0.0 <2.0.0**.

Команда установки:

```bash
npm install -g @fission-ai/openspec@1
```

### Пример рабочего процесса

Вот полный пример использования OpenSpec в проекте HagiCode:

#### Шаг 1: Создание предложения

```bash
cd /path/to/hagicode-project
openspec proposal create
```

Введите в интерактивном приглашении:
- Название предложения: Добавить функцию аутентификации пользователей
- Описание проблемы: Системе нужна авторизация и управление правами
- Решение: Реализовать систему аутентификации на основе JWT

#### Шаг 2: Написание спецификации

После утверждения предложения создайте подробную спецификацию:

```bash
openspec spec create
```

Определите в спецификации:
- Проектирование потока аутентификации
- Определение интерфейса API
- Структура таблиц базы данных
- Соображения безопасности

#### Шаг 3: Создание изменения

Начните реализацию:

```bash
openspec change create
```

Введите имя изменения: `add-user-authentication`

#### Шаг 4: Добавление задач

```bash
openspec change add-task add-user-authentication
```

Добавьте конкретные задачи:
- Реализовать endpoint регистрации пользователя
- Реализовать endpoint входа
- Реализовать JWT middleware
- Написать модульные тесты

#### Шаг 5: Реализация и валидация

После завершения кода проверьте изменение:

```bash
openspec change validate add-user-authentication
```

#### Шаг 6: Архивирование

После успешной валидации заархивируйте изменение:

```bash
openspec change archive add-user-authentication
```

## Лучшие практики

### Написание предложений

1. **Четкая формулировка проблемы**: Четко определите проблему, которую нужно решить, избегайте расплывчатых описаний
2. **Рассмотрение альтернатив**: Перечислите и сравните различные решения
3. **Оценка влияния**: Проанализируйте влияние и риски для существующих систем
4. **Краткость**: Предложения должны быть краткими; детали оставьте для спецификации

### Написание спецификаций

1. **Детально, но не избыточно**: Предоставьте достаточно деталей, избегайте избыточного проектирования
2. **Использование диаграмм**: Используйте блок-схемы, архитектурные диаграммы и т.д. для пояснения
3. **Определение интерфейсов**: Четко определите все публичные интерфейсы и структуры данных
4. **Учет граничных случаев**: Перечислите исключительные ситуации и обработку ошибок

### Управление изменениями

1. **Малые итерации**: Разбивайте большие функции на несколько небольших изменений
2. **Своевременные обновления**: Поддерживайте синхронизацию статуса задач и документации
3. **Проверка кода**: Проводите проверку кода после завершения изменений
4. **Своевременное архивирование**: Архивируйте завершенные изменения для поддержания чистоты рабочего каталога

### Командное сотрудничество

1. **Регулярные проверки**: Регулярно проверяйте предложения и спецификации для обеспечения консенсуса команды
2. **Документация в первую очередь**: Завершите документацию перед началом кодирования
3. **Поддержание коммуникации**: Записывайте обсуждения и решения в предложениях и спецификациях
4. **Контроль версий**: Включите документы OpenSpec в систему контроля версий

## Распространенные сценарии использования

### Сценарий 1: Добавление новых функций

1. Создайте предложение с описанием требований к новой функции
2. Команда обсуждает и утверждает предложение
3. Напишите подробную техническую спецификацию
4. Создайте изменение и реализуйте функцию
5. Валидируйте и заархивируйте изменение

### Сценарий 2: Рефакторинг существующего кода

1. Создайте предложение с объяснением причин и целей рефакторинга
2. Определите план рефакторинга и шаги в спецификации
3. Создайте изменение и проводите рефакторинг поэтапно
4. Обеспечьте покрытие тестами и валидируйте результаты рефакторинга
5. Заархивируйте изменение

### Сценарий 3: Исправление ошибок

Для простых ошибок:
- Могут быть исправлены напрямую без полного процесса OpenSpec

Для сложных ошибок:
1. Создайте предложение с анализом первопричины ошибки
2. Определите исправление в спецификации
3. Создайте изменение и реализуйте исправление
4. Добавьте тесты для предотвращения регрессий
5. Заархивируйте изменение

### Сценарий 4: Управление техническим долгом

1. Создайте предложение со списком элементов технического долга
2. Определите план улучшения в спецификации
3. Создайте изменения для постепенного погашения долга
4. Отслеживайте прогресс и эффективность
5. Архивируйте завершенные улучшения

## Интеграция с другими инструментами

### Интеграция с Git

OpenSpec легко интегрируется с Git:

```bash
# Документы OpenSpec должны быть включены в систему контроля версий
git add openspec/
git commit -m "Add OpenSpec proposal for feature X"
```

### Интеграция CI/CD

Валидируйте спецификации OpenSpec в конвейерах CI/CD:

```yaml
# .github/workflows/openspec.yml
name: OpenSpec Validation
on: [pull_request]
jobs:
  validate:
    runs-on: ubuntu-latest
    steps:
      - uses: actions/checkout@v2
      - name: Install OpenSpec
        run: npm install -g @fission-ai/openspec@1
      - name: Validate changes
        run: openspec change validate
```

### Интеграция с IDE

Рекомендуемые редакторы с поддержкой Markdown для редактирования документов OpenSpec:
- VS Code + плагины Markdown
- IDE JetBrains + поддержка Markdown
- Vim/Neovim + плагины Markdown

## Устранение неполадок

### Распространенные проблемы

#### Проблема: команда OpenSpec не найдена

**Решение**:
1. Убедитесь, что OpenSpec установлен: `npm list -g @fission-ai/openspec`
2. Проверьте, что переменная PATH включает глобальный каталог bin npm
3. Переустановите: `npm install -g @fission-ai/openspec@1`

#### Проблема: невозможно создать предложения или изменения

**Решение**:
1. Убедитесь, что текущий каталог является корнем проекта
2. Проверьте наличие каталога `openspec/`; если его нет, создайте
3. Проверьте права на запись

#### Проблема: валидация не проходит

**Решение**:
1. Проверьте, что документы в каталоге изменения полные
2. Убедитесь, что все необходимые задачи выполнены
3. Просмотрите подробные ошибки: `openspec change validate <change-name> --verbose`

#### Проблема: архивирование не удается

**Решение**:
1. Убедитесь, что изменение прошло валидацию
2. Проверьте наличие каталога `openspec/archived/`
3. Убедитесь, что нет незакоммиченных изменений кода

## Расширенное использование

### Пользовательские шаблоны

OpenSpec поддерживает пользовательские шаблоны для предложений и спецификаций:

```bash
# Создайте каталог шаблонов в корне проекта
mkdir -p .openspec/templates

# Создайте пользовательский шаблон предложения
cat > .openspec/templates/proposal.md << 'EOF'
# Proposal: {{title}}

## Problem Statement
{{problem}}

## Proposed Solution
{{solution}}

## Custom Section
Your custom content here
EOF
```

### Файл конфигурации

Создайте файл конфигурации `.openspec.json` в корне проекта:

```json
{
  "version": "1.0",
  "templates": {
    "proposal": ".openspec/templates/proposal.md",
    "spec": ".openspec/templates/spec.md"
  },
  "validation": {
    "requireTests": true,
    "requireDocs": true
  },
  "hooks": {
    "pre-archive": "npm test"
  }
}
```

### Хуки

OpenSpec поддерживает выполнение пользовательских скриптов до и после определенных операций:

```json
{
  "hooks": {
    "pre-create": "scripts/check-permissions.sh",
    "post-archive": "scripts/notify-team.sh"
  }
}
```
