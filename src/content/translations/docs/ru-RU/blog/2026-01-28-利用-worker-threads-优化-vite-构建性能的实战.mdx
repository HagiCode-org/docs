## От 120 секунд до 45 секунд: практическое руководство по оптимизации производительности сборки Vite с помощью Worker Threads

> При работе с крупными фронтенд-проектами сборка для продакшена может казаться мучительно медленной. В этой статье описано, как с помощью Node.js Worker Threads сократить этап обфускации в сборке Vite с 120 секунд до 45 секунд, а также приводятся детали реализации и опыт из проекта HagiCode.

## Предпосылки

В нашей практике фронтенд-инженерии по мере роста проекта проблемы эффективности сборки становились всё более заметными. В частности, в процессе продакшен-сборки мы обычно используем инструменты обфускации JavaScript, такие как `javascript-obfuscator`, для защиты логики исходного кода. Этот шаг необходим, но он требует значительных вычислительных ресурсов и интенсивно нагружает CPU.

На раннем этапе разработки **HagiCode** мы столкнулись с очень неприятным узким местом производительности: время продакшен-сборки быстро ухудшалось по мере роста кодовой базы.

**Конкретные проблемы были следующими:**
- Обфускация выполнялась последовательно на одном потоке, загружая одно ядро CPU, в то время как другие простаивали
- Время сборки выросло с первоначальных 30 секунд до 110–120 секунд
- Цикл проверки сборки после изменений стал крайне долгим, серьёзно замедляя итерации разработки
- В CI/CD-конвейере этап сборки стал самой времязатратной частью

**Почему HagiCode нуждался в этом?**
HagiCode — это AI-ассистент программирования, чья фронтенд-архитектура включает сложную бизнес-логику и модули AI-взаимодействия. Для обеспечения безопасности основного кода мы принудительно включили интенсивную обфускацию при релизах. Столкнувшись с ожиданием сборки до двух минут, мы решили провести глубокую оптимизацию производительности системы сборки.

## Анализ: поиск точки прорыва в узком месте производительности

Прежде чем решать проблему производительности, нам нужно было чётко определить мысли и найти оптимальное техническое решение.

### Ключевое решение: почему Worker Threads?

Существует три основных способа реализации параллельных вычислений в Node.js:
1. **child_process**: создание независимых дочерних процессов
2. **Web Workers**: в основном для браузерной стороны
3. **worker_threads**: нативная поддержка многопоточности в Node.js

После сравнения вариантов HagiCode в конечном итоге выбрал **Worker Threads** по следующим причинам:
- **Нулевые затраты на сериализацию**: Worker Threads работают в том же процессе и могут использовать `SharedArrayBuffer` или передачу владения для совместного использования памяти, избегая значительных затрат на сериализацию при межпроцессном взаимодействии.
- **Нативная поддержка**: встроено в Node.js 12+ без необходимости дополнительных тяжёлых зависимостей.
- **Единый контекст**: отладка и журналирование удобнее, чем с дочерними процессами.

### Гранулярность задач: как разделить задачи обфускации?

Сложно распараллелить обфускацию одного огромного JS-бандла, поскольку код имеет зависимости, но выходные данные сборки Vite состоят из нескольких **chunks**. Это даёт нам естественную границу параллелизма:

- **Независимость**: после упаковки Vite зависимости между разными чанками уже развязаны, поэтому они могут безопасно обрабатываться параллельно.
- **Подходящая гранулярность**: обычно в проекте 10–30 чанков, что является отличным масштабом для параллельного планирования.
- **Простая интеграция**: хук `generateBundle` в плагинах Vite позволяет перехватывать и обрабатывать эти чанки до записи файлов.

### Архитектурный дизайн

Мы разработали параллельную систему обработки с четырьмя основными компонентами:

1. **Task Splitter**: обход объекта bundle Vite, фильтрация файлов, не требующих обфускации (таких как vendor), и создание очереди задач.
2. **Worker Pool Manager**: управление жизненным циклом Worker-ов, распределение задач, перезапуск и повторная попытка при ошибках.
3. **Progress Reporter**: вывод прогресса сборки в реальном времени для снижения тревожности ожидания.
4. **ObfuscationWorker**: рабочий поток, фактически выполняющий логику обфускации.

## Решение: практическая реализация

На основе описанного анализа мы приступили к реализации этой системы параллельной обфускации.

### 1. Настройка плагина Vite

Сначала мы интегрировали плагин параллельной обфускации в `vite.config.ts`. Конфигурация интуитивно понятна — достаточно указать количество Worker-ов и правила обфускации.

```typescript
import { defineConfig } from 'vite'
import { parallelJavascriptObfuscator } from './buildTools/plugin'

export default defineConfig(({ mode }) => {
  const isProduction = mode === 'production'

  return {
    build: {
      rollupOptions: {
        ...(isProduction
          ? {
              plugins: [
                parallelJavascriptObfuscator({
                  enabled: true,
                  // Автоматическая настройка по количеству ядер CPU; оставляем одно ядро для основного потока
                  workerCount: 4,
                  retryAttempts: 3,
                  fallbackToMainThread: true, // Автоматический откат к однопоточному режиму при ошибке
                  // Фильтрация vendor-чанков; сторонние библиотеки обычно не нуждаются в обфускации
                  isVendorChunk: (fileName: string) => fileName.includes('vendor-'),
                  obfuscationConfig: {
                    compact: true,
                    controlFlowFlattening: true,
                    deadCodeInjection: true,
                    disableConsoleOutput: true,
                    // ... дополнительные параметры обфускации
                  },
                }),
              ],
            }
          : {}),
      },
    },
  }
})
```

### 2. Реализация логики Worker-а

Worker — это единица выполнения задач. Нам нужно чётко определить структуры входных и выходных данных.

**Примечание**: хотя код здесь простой, есть несколько подводных камней. Например, проверка `parentPort` на null и правильная обработка ошибок. В практике HagiCode мы обнаружили, что некоторые специфические паттерны синтаксиса ES6 могут привести к сбою обфускатора, поэтому добавили защиту `try-catch`.

```typescript
import { parentPort } from 'worker_threads'
import javascriptObfuscator from 'javascript-obfuscator'

export interface ObfuscationTask {
  chunkId: string
  code: string
  config: any
}

export interface ObfuscationResult {
  chunkId: string
  obfuscatedCode: string
  error?: string
}

// Прослушивание задач от основного потока
if (parentPort) {
  parentPort.on('message', async (task: ObfuscationTask) => {
    try {
      // Выполнение обфускации
      const obfuscated = javascriptObfuscator.obfuscate(task.code, task.config)
      const result: ObfuscationResult = {
        chunkId: task.chunkId,
        obfuscatedCode: obfuscated.getObfuscatedCode(),
      }
      // Отправка результата обратно в основной поток
      parentPort?.postMessage(result)
    } catch (error) {
      // Обработка исключений, чтобы сбой одного Worker-а не блокировал всю сборку
      const result: ObfuscationResult = {
        chunkId: task.chunkId,
        obfuscatedCode: '',
        error: error instanceof Error ? error.message : 'Unknown error',
      }
      parentPort?.postMessage(result)
    }
  })
}
```

### 3. Менеджер пула Worker-ов

Это ядро всего решения. Нам нужно поддерживать пул Worker-ов фиксированного размера и планировать задачи по стратегии **FIFO (первым пришёл — первым обслужен)**.

```typescript
import { Worker } from 'worker_threads'
import os from 'os'

export class WorkerPool {
  private workers: Worker[] = []
  private taskQueue: Array<{
    task: ObfuscationTask
    resolve: (result: ObfuscationResult) => void
    reject: (error: Error) => void
  }> = []

  constructor(options: WorkerPoolOptions = {}) {
    // По умолчанию количество ядер - 1, чтобы основной поток имел место для работы
    const workerCount = options.workerCount ?? Math.max(1, (os.cpus().length || 4) - 1)

    for (let i = 0; i < workerCount; i++) {
      this.createWorker()
    }
  }

  private createWorker() {
    const worker = new Worker('./worker.ts')

    worker.on('message', (result) => {
      // После завершения задачи берём следующую из очереди
      const nextTask = this.taskQueue.shift()
      if (nextTask) {
        this.dispatchTask(worker, nextTask)
      } else {
        // Если нет ожидающих задач, помечаем Worker как свободный
        this.activeWorkers.delete(worker)
      }
    })

    this.workers.push(worker)
  }

  // Отправка задачи в пул
  public runTask(task: ObfuscationTask): Promise<ObfuscationResult> {
    return new Promise((resolve, reject) => {
      const job = { task, resolve, reject }
      const idleWorker = this.workers.find(w => !this.activeWorkers.has(w))

      if (idleWorker) {
        this.dispatchTask(idleWorker, job)
      } else {
        this.taskQueue.push(job)
      }
    })
  }

  private dispatchTask(worker: Worker, job: any) {
    this.activeWorkers.set(worker, job.task)
    worker.postMessage(job.task)
  }
}
```

### 4. Отчёт о прогрессе

Ожидание — это мучительно, особенно когда не знаешь, сколько ещё осталось. Мы добавили простой репортёр прогресса для обратной связи в реальном времени.

```typescript
export class ProgressReporter {
  private completed = 0
  private readonly total: number
  private readonly startTime: number

  constructor(total: number) {
    this.total = total
    this.startTime = Date.now()
  }

  increment(): void {
    this.completed++
    this.report()
  }

  private report(): void {
    const now = Date.now()
    const elapsed = now - this.startTime
    const percentage = (this.completed / this.total) * 100

    // Простая оценка оставшегося времени
    const avgTimePerChunk = elapsed / this.completed
    const remaining = (this.total - this.completed) * avgTimePerChunk

    console.log(
      `[Parallel Obfuscation] ${this.completed}/${this.total} chunks completed (${percentage.toFixed(1)}%) | ETA: ${(remaining / 1000).toFixed(1)}s`
    )
  }
}
```

### Данные эталонного тестирования

Мы тестировали в следующей среде:
- CPU: Intel Core i7-12700K (12 ядер / 20 потоков)
- RAM: 32GB DDR4
- Node.js: v18.17.0
- OS: Ubuntu 22.04

**Сравнение результатов**:
- **Один поток (до оптимизации)**: 118 секунд
- **4 Worker-а**: 55 секунд (улучшение на **53%**)
- **8 Worker-ов**: 48 секунд (улучшение на **60%**)
- **12 Worker-ов**: 45 секунд (улучшение на **62%**)

Как видно, прирост не является линейным. При количестве Worker-ов более 8 улучшение уменьшается. Это в основном ограничено равномерностью распределения задач и пропускной способностью памяти.

### Частые проблемы и решения

В реальном использовании HagiCode мы также столкнулись с несколькими проблемами, которыми делимся:

**В1: Время сборки не уменьшилось значительно, а даже увеличилось?**
- **Причина**: создание Worker-ов имеет свои накладные расходы, или настроено слишком много Worker-ов, что вызывает частое переключение контекста.
- **Решение**: рекомендуем устанавливать количество Worker-ов равным `количеству ядер CPU - 1`. Также проверьте, нет ли особенно крупных чанков (например, > 5 МБ). Такой «монстр»-файл станет узким местом, и может потребоваться оптимизация стратегии разделения кода.

**В2: Worker-ы иногда падают и вызывают сбои сборки?**
- **Причина**: некоторые специфические паттерны синтаксиса кода могут вызывать внутренние ошибки обфускатора.
- **Решение**: мы реализовали **механизм автоматического понижения**. Когда Worker достигает порога ошибок, плагин автоматически откатывается к однопоточному режиму, обеспечивая непрерывность сборки. При этом записывается имя файла, вызвавшего ошибку, для последующего исправления.

**В3: Слишком высокое потребление памяти (OOM)?**
- **Причина**: каждый Worker нуждается в собственном адресном пространстве для загрузки обфускатора и парсинга AST.
- **Решение**:
  - Уменьшите количество Worker-ов.
  - Увеличьте лимит памяти Node.js: `NODE_OPTIONS="--max-old-space-size=4096" npm run build`.
  - Убедитесь, что Worker-ы не удерживают ненужные ссылки на крупные объекты.

## Заключение

С помощью внедрения Node.js Worker Threads мы успешно сократили время продакшен-сборки проекта HagiCode с 120 секунд до примерно 45 секунд, значительно улучшив опыт разработки и эффективность CI/CD.

Суть этого решения:
1. **Правильное разделение задач**: использование чанков Vite как единиц параллелизма.
2. **Контроль ресурсов**: использование пула Worker-ов для предотвращения исчерпания ресурсов.
3. **Проектирование отказоустойчивости**: механизм автоматического понижения обеспечивает стабильность сборки.
