## From Docusaurus 3.x to Astro 5.x: HagiCode Site Migration Practical Review

> This article reviews our complete process of migrating the HagiCode official website from Docusaurus 3.x to Astro 5.x. We will dive deep into how to solve performance bottlenecks through Astro's Islands Architecture while preserving existing React component assets, achieving dual improvements in build speed and loading performance.


## Background

In January 2026, we performed a "heart transplant surgery" on HagiCode's official site—a complete migration of the core framework from Docusaurus 3.x to Astro 5.x. This was not an impulsive major refactoring, but a carefully considered technical decision.

Before the migration, our site was fully functional but gradually showed some "rich people's problems": bloated build output size, excessive JavaScript load, and less-than-ideal page loading speeds for complex documentation pages. As an AI code assistant project, HagiCode needs frequent documentation and feature updates, making build efficiency directly impact release speed. At the same time, we wanted the site to be more search engine friendly (SEO) to help more developers discover this project.

To address these pain points, we made a bold decision: completely rebuild the build system and migrate to Astro. The changes brought by this decision may be bigger than you imagine—I'll elaborate shortly.

## Why Give Up a Mature Docusaurus?

In the React ecosystem, Docusaurus has always been the "standard answer" for documentation sites. It's plug-and-play, plugin-rich, and has an active community. However, as HagiCode's functionality increased, we also felt its limitations:

1.  **Performance Bottleneck**: Docusaurus is essentially a React SPA (Single Page Application). Even if you're writing purely static pages, the client needs to load the React runtime and perform hydration, which is too heavy for simple documentation pages.
2.  **Resource Size**: Even with minimal page content, the bundled JS size is relatively fixed, which isn't friendly for mobile users and poor network environments.
3.  **Insufficient Flexibility**: While extensible, we craved more low-level control over the build process customization.

The emergence of Astro perfectly solves these problems. It provides a brand new "Islands Architecture": by default, Astro generates zero-JavaScript static HTML, and only components requiring interactivity "activate" and load JS. This means most of our site content is pure HTML with extremely fast speeds.

## Core Migration Strategy: Smooth Architecture Transition

Migration is not simple copy-paste, but a paradigm shift. We switched from Docusaurus's "All React Mode" to Astro's "Core + Islands" mode.

### 1. Configuration System Restructuring

First, we needed to transition from `docusaurus.config.ts` to `astro.config.mjs`. This is not just a filename change, but a rewrite of routing and build logic.

In Docusaurus, everything is a plugin; in Astro, everything is an integration. We needed to redefine the site's base path, build output mode (static vs SSR), and asset optimization strategy.

**Before Migration:**
```typescript
// docusaurus.config.ts
export default {
  title: 'HagiCode',
  url: 'https://hagicode.com',
  baseUrl: '/',
  // ... more configuration
};
```

**After Migration:**
```javascript
// astro.config.mjs
import { defineConfig } from 'astro/config';
import react from '@astrojs/react';

export default defineConfig({
  integrations: [react()],
  site: 'https://hagicode.com',
  base: '/',
  // Optimization configuration for static assets
  build: {
    inlineStylesheets: 'auto',
  },
});
```

### 2. React Component Retention and Transformation

This was the most headache-inducing part of the migration. Our existing site had many React components (like Tabs component, code highlighting, feedback buttons, etc.). Discarding them outright was wasteful, but keeping all of them would result in excessive JS load.

HagiCode adopted a **progressive hydration** strategy:
*   **Pure Static Components**: For display content (like headers, footers, plain text documentation), rewrite as Astro components (`.astro` files), rendered directly as HTML at build time.
*   **Interactive Islands**: For components that must retain interactivity (like theme switcher, Tabs switching, code block copy button), keep the React implementation and add `client:load` or `client:visible` directives.

For example, our commonly used Tabs component in documentation:

```jsx
// src/components/Tabs.jsx
import { useState } from 'react';
import './Tabs.css'; // import styles

export default function Tabs({ items }) {
  const [activeIndex, setActiveIndex] = useState(0);
  // ... state logic
  return (
    <div className="tabs-wrapper">
      {/* rendering logic */}
    </div>
  );
}
```

When using in Markdown, we explicitly tell Astro: "this component needs JS":

```markdown
// src/content/docs/example.mdx
import Tabs from '../../components/Tabs.jsx';

<!-- Only load JS when component enters viewport -->
<Tabs client:visible items={...} />
```

This way, interactive components outside the viewport don't compete for bandwidth, greatly optimizing first-screen load speed.

### 3. Style System Adaptation: CSS Modules to Scoped

Docusaurus supports CSS Modules by default, while Astro favors using Scoped CSS (via `<style>` tags). Both have the core concept of style isolation, but different syntax.

In HagiCode's migration, we broke down most complex CSS Modules into Astro's Scoped styles. This is actually a good thing, because in `.astro` files, styles and templates are written in the same file, making maintenance more intuitive.

**Before Transformation:**
```css
/* Tabs.module.css */
.wrapper { background: var(--ifm-background-color); }
```

**After Transformation (Astro Scoped):**
```html
<!-- Tabs.astro -->
<div class="tabs-wrapper">
  <slot />
</div>

<style>
  .tabs-wrapper {
    /* Use CSS variables directly, adapt to theme */
    background: var(--bg-color);
    padding: 1rem;
  }
</style>
```

At the same time, we unified the global CSS variable system, leveraging Astro's environment awareness to ensure seamless dark mode switching across different pages.

## Pitfalls and Solutions in Practice

During HagiCode's actual migration process, we encountered many pitfalls. Here are a few of the most typical ones to share.

### 1. Path and Environment Variable Pain Points

HagiCode supports subpath deployment (like deploying to GitHub Pages subdirectories). In Docusaurus, it automatically handles `baseUrl`. But in Astro, we need to be more careful when handling image links and API requests.

We introduced an environment variable mechanism for unified management:

```javascript
// Handle paths in build scripts
const getBasePath = () => import.meta.env.VITE_SITE_BASE || '/';
```

Remember, don't hardcode paths starting with `/` in code. In development and production environments, or after configuring `base` path, this will cause 404s for resources.

### 2. CommonJS Script Compatibility

Our old site had some Node.js scripts (for automatically fetching Metrics data, updating sitemaps, etc.) that were written in CommonJS (`require`). Astro and modern build tools fully embrace ES Modules (`import`/`export`).

If you have similar scripts, remember to refactor them all to ES Modules. This is the trend of the future—better to change early and save trouble later.

```javascript
// Old way
const fs = require('fs');

// New way
import fs from 'fs';
```

### 3. Don't Forget SEO and Redirects

Search engines have already indexed HagiCode's old Docusaurus pages. If we directly switched to Astro with changed URL structure, it would cause大量 404s and significant authority loss.

We configured redirect rules in Astro:

```javascript
// astro.config.mjs
export default defineConfig({
  redirects: {
    '/docs/old-path': '/docs/new-path',
    // Batch map old links to new links
  }
});
```
Or handle at the server configuration level. Ensure old links can 301 redirect to new addresses—this is crucial for SEO.
