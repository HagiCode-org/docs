## Architecture Design

### Layered Architecture Overview

The integration between C# and CodeBuddy adopts a clear layered architecture, which makes code responsibilities distinct and later maintenance more convenient:

```
┌─────────────────────────────────────────────┐
│           Provider Contract Layer           │
│   AIProviderType enum + extension methods   │
├─────────────────────────────────────────────┤
│            Provider Factory Layer           │
│   AIProviderFactory dependency inject factory│
├─────────────────────────────────────────────┤
│           Provider Implementation Layer     │
│   CodebuddyCliProvider specific implementation│
├─────────────────────────────────────────────┤
│          ACP Infrastructure Layer           │
│  ACPSessionManager / StdioAcpTransport      │
│  AcpRpcClient / AcpAgentClient              │
└─────────────────────────────────────────────┘
```

What's the benefit of this layering? Simply put, each layer doesn't interfere with the others. Suppose you want to change a communication method later (like from stdio to WebSocket), you only need to modify the bottom layer, and the business code above remains completely untouched. After all, no one wants to pull one hair and move the whole body—changing a communication method but having to modify business code for a long time would be too miserable.

### Core Component Analysis

The **Provider Contract Layer** is the cornerstone of the entire architecture. We defined the `AIProviderType` enum, where `CodebuddyCli = 3` serves as an enum value, implementing bidirectional mapping between strings and enums through extension methods. This way, strings in configuration files can be easily converted to enums, and enums can also be converted to strings for output during debugging. That's all, it's just a mapping relationship, but done well saves worry.

The **Provider Factory Layer** is responsible for creating corresponding Provider instances based on configuration. Here we use .NET's dependency injection mechanism, combined with `ActivatorUtilities.CreateInstance` to achieve dynamic creation. The benefit of the factory pattern is that when adding a new Provider, you only need to add creation logic without modifying existing code. This is similar to writing articles—if you want to add a new chapter, just add a new chapter, no need to rewrite all the previous ones.

The **Provider Implementation Layer** is where the real work happens. `CodebuddyCliProvider` implements the `IAIProvider` interface, providing both `ExecuteAsync` (non-streaming) and `StreamAsync` (streaming) calling methods.

The **ACP Infrastructure Layer** is the underlying support for communication. This layer handles all protocol details, including process management, message serialization, response parsing, etc. Like the foundation of a house, no matter how beautiful the top is, the bottom must be stable.

## Communication Mechanism

### Stdio Transport Mode

CodeBuddy uses **Stdio (Standard Input/Output)** to communicate with external processes. The startup command is simple:

```bash
codebuddy --acp
```

Then JSON-RPC message exchange is performed through standard input/output. The advantages of this approach include:

1. **Fast startup**: Local process communication has no network latency
2. **Simple configuration**: Only need to specify the executable file path
3. **Environment isolation**: Each session is an independent process, not affecting each other

During communication, environment variable injection is supported, commonly including:

- `CODEBUDDY_API_KEY`: API key authentication
- `CODEBUDDY_INTERNET_ENVIRONMENT`: Network environment configuration

It's like, communication between people, find a convenient way, then you can talk.

### Message Protocol

ACP is based on the JSON-RPC 2.0 protocol, and the message format is roughly like this:

```json
// Request message
{
  "jsonrpc": "2.0",
  "id": 1,
  "method": "agent/prompt",
  "params": {
    "prompt": "Help me write a sorting algorithm",
    "sessionId": "session-123"
  }
}

// Response message
{
  "jsonrpc": "2.0",
  "id": 1,
  "result": {
    "content": "This is the AI's reply..."
  }
}
```

In actual implementation, we've encapsulated all these protocol details, so upper-level business code only needs to focus on prompt and response. That's all, encapsulated well, people using it later will be more comfortable.

### 2. Provider Factory Integration

Add the CodeBuddy creation branch in the factory class:

```csharp
// AIProviderFactory.cs
private IAIProvider? CreateProvider(AIProviderType providerType, ProviderConfiguration config)
{
    return providerType switch
    {
        AIProviderType.CodebuddyCli =>
            ActivatorUtilities.CreateInstance<CodebuddyCliProvider>(
                _serviceProvider,
                Options.Create(config)),
        // ... other providers
        _ => throw new NotSupportedException($"Provider {providerType} not supported")
    };
}
```

Here we use dependency injection's `ActivatorUtilities`, which automatically handles constructor parameter injection, very convenient. That's all, .NET stuff, used correctly saves worry.

### 3. Complete Provider Implementation

Below is the core implementation of `CodebuddyCliProvider`, including both streaming and non-streaming calling methods:

```csharp
public class CodebuddyCliProvider : IAIProvider
{
    private readonly ILogger<CodebuddyCliProvider> _logger;
    private readonly IACPSessionManager _sessionManager;
    private readonly ProviderConfiguration _config;

    public string Name => "CodebuddyCli";
    public bool SupportsStreaming => true;
    public ProviderCapabilities Capabilities { get; }

    public CodebuddyCliProvider(
        ILogger<CodebuddyCliProvider> logger,
        IACPSessionManager sessionManager,
        IOptions<ProviderConfiguration> config)
    {
        _logger = logger;
        _sessionManager = sessionManager;
        _config = config.Value;

        // Define current Provider's capabilities
        Capabilities = new ProviderCapabilities
        {
            SupportsStreaming = true,
            SupportsTools = true,
            SupportsSystemMessages = true,
            SupportsArtifacts = false,
            MaxTokens = 8192
        };
    }

    // Non-streaming call: wait for all results to return together
    public async Task<AIResponse> ExecuteAsync(
        AIRequest request,
        CancellationToken cancellationToken = default)
    {
        // Create independent session for request
        var session = await _sessionManager.CreateSessionAsync(
            "CodebuddyCli",
            request.WorkingDirectory,
            cancellationToken,
            request.SessionId);

        try
        {
            var fullPrompt = BuildPrompt(request);
            await session.SendPromptAsync(fullPrompt, cancellationToken);

            var responseBuilder = new StringBuilder();
            var toolCalls = new List<AIToolCall>();

            // Collect all response chunks
            await foreach (var chunk in StreamFromSession(session, cancellationToken))
            {
                if (!string.IsNullOrEmpty(chunk.Content))
                {
                    responseBuilder.Append(chunk.Content);
                }
                // Handle tool calls...
            }

            return new AIResponse
            {
                Content = AIResultContentSanitizer.SanitizeResultContent(
                    responseBuilder.ToString()),
                ToolCalls = toolCalls,
                Provider = Name,
                Model = string.Empty
            };
        }
        finally
        {
            // Release session resources
            await session.DisposeAsync();
        }
    }

    // Streaming call: return response chunks in real-time
    public async IAsyncEnumerable<AIStreamingChunk> StreamAsync(
        AIRequest request,
        [EnumeratorCancellation] CancellationToken cancellationToken = default)
    {
        var session = await _sessionManager.CreateSessionAsync(
            "CodebuddyCli",
            request.WorkingDirectory,
            cancellationToken);

        try
        {
            var fullPrompt = BuildPrompt(request);
            await session.SendPromptAsync(fullPrompt, cancellationToken);

            await foreach (var chunk in StreamFromSession(session, cancellationToken))
            {
                yield return chunk;
            }
        }
        finally
        {
            await session.DisposeAsync();
        }
    }

    private async IAsyncEnumerable<AIStreamingChunk> StreamFromSession(
        IACPSession session,
        [EnumeratorCancellation] CancellationToken cancellationToken)
    {
        // Iterate through all updates in the session
        await foreach (var notification in session.ReceiveUpdatesAsync(cancellationToken))
        {
            switch (notification.Update)
            {
                case AgentMessageChunkSessionUpdate agentMessage:
                    // Handle text content chunks
                    if (agentMessage.Content is AcpImp.TextContentBlock textContent)
                    {
                        yield return new AIStreamingChunk
                        {
                            Content = textContent.Text,
                            Type = StreamingChunkType.ContentDelta,
                            IsComplete = false
                        };
                    }
                    break;

                case ToolCallSessionUpdate toolCall:
                    // Handle tool calls
                    yield return new AIStreamingChunk
                    {
                        Content = string.Empty,
                        Type = StreamingChunkType.ToolCallDelta,
                        ToolCallDelta = new AIToolCallDelta
                        {
                            Id = toolCall.ToolCallId,
                            Name = toolCall.Kind.ToString(),
                            Arguments = toolCall.RawInput?.ToString()
                        }
                    };
                    break;

                case AcpImp.PromptCompletedSessionUpdate:
                    // Response complete
                    yield break;
            }
        }
    }

    // Build complete prompt
    private string BuildPrompt(AIRequest request, string? embeddedCommandPrompt = null)
    {
        var sb = new StringBuilder();

        // Embed command prompt (if any)
        if (!string.IsNullOrEmpty(embeddedCommandPrompt))
        {
            sb.AppendLine(embeddedCommandPrompt);
            sb.AppendLine();
        }

        // System message
        if (!string.IsNullOrEmpty(request.SystemMessage))
        {
            sb.AppendLine(request.SystemMessage);
            sb.AppendLine();
        }

        // User prompt
        sb.Append(request.Prompt);
        return sb.ToString();
    }
}
```

This code has several key points:

1. **Session Management**: Each request creates an independent session, and resources are released after the request completes. This is experience from pitfalls—if session reuse isn't done well, state pollution problems easily occur. After all, after using something, clean it up, otherwise the next person to use it will have trouble.

2. **Streaming Processing**: `IAsyncEnumerable` allows responses to be returned as they're generated, without waiting for all content to be generated. This is particularly important for long text scenarios, greatly improving user experience. Like, people waiting for results don't want to just keep waiting.

3. **Tool Calls**: CodeBuddy supports tool calls (Function Calling), handled through `ToolCallSessionUpdate`. This capability is critical for complex code editing tasks.

4. **Content Filtering**: Use `AIResultContentSanitizer` to filter Think block content, keeping output clean.

### 4. Dependency Injection Configuration

Add related services in module registration:

```csharp
// PCodeClaudeHelperModule.cs
public void ConfigureModule(IServiceCollection context)
{
    // Register Provider
    context.Services.AddTransient<CodebuddyCliProvider>();

    // Register ACP infrastructure
    context.Services.AddSingleton<IACPSessionManager, ACPSessionManager>();
    context.Services.AddSingleton<IAcpPlatformConfigurationResolver, AcpPlatformConfigurationResolver>();
    context.Services.AddSingleton<IAIRequestToAcpMapper, AIRequestToAcpMapper>();
    context.Services.AddSingleton<IAcpToAIResponseMapper, AcpToAIResponseMapper>();
}
```

## Configuration Examples

### Configuration File

Add CodeBuddy related configuration in `appsettings.json`:

```yaml
AI:
  # Default Provider to use
  DefaultProvider: "CodebuddyCli"

  # Provider configuration
  Providers:
    CodebuddyCli:
      Type: "CodebuddyCli"
      WorkingDirectory: "C:/projects/my-app"
      ExecutablePath: "C:/tools/codebuddy.cmd"

  # Platform-related configuration
  PlatformConfigurations:
    CodebuddyCli:
      ExecutablePath: "C:/tools/codebuddy.cmd"
      Arguments: "--acp"
      StartupTimeoutMs: 5000
      EnvironmentVariables:
        CODEBUDDY_API_KEY: "${CODEBUDDY_API_KEY}"
        CODEBUDDY_INTERNET_ENVIRONMENT: "production"
```

## Practical Experience Summary

### Pitfall Record

We encountered several typical pitfalls during implementation, sharing them to help you avoid detours. After all, avoiding other people's pitfalls is a good thing:

1. **Session Leak Issue**: Initially didn't properly release sessions, leading to process resource exhaustion. The solution is to use `try-finally` to ensure each request releases resources. That's all, used things should be put back, otherwise what will the next person use.

2. **Environment Variable Passing**: Windows and Linux have different environment variable syntax, later unified using `Dictionary<string, string?>` to handle. Cross-platform things, unified standards from the start save worry later.

3. **Timeout Configuration**: CLI startup takes time, set a 5-second startup timeout to avoid rapid request failures. Everything needs a degree, being too hasty prevents getting things done.

4. **Encoding Issues**: Default encoding on Windows may cause Chinese character garbling, explicitly specify UTF-8 encoding when starting processes. Chinese not displaying properly would be very uncomfortable.

### Performance Optimization

1. **Session Pool**: For frequent short requests, consider implementing a session pool to reuse processes
2. **Connection Caching**: Factory class already supports Provider instance caching
3. **Async First**: Use asynchronous programming throughout to avoid blocking threads

Performance, optimize when you can, after all the longer users wait, the worse the experience.
