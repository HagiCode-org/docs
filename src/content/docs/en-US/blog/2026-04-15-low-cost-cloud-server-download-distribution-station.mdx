## How to Build an Accelerated Download Distribution Station Using Low-Cost Cloud Servers

> Cloud storage traffic is outrageously expensive, cross-border access is frustratingly slow, and CDN prices are daunting... These are pains that anyone doing distribution understands. This article shares a low-cost solution we developed through experimentation in the [HagiCode](https://hagicode.com) project. Cloud server + Nginx caching layer reduced costs by half while improving speed—a small comfort indeed.

## Background

In the internet world, download speed and stability ultimately come down to user experience. Whether open source or commercial, you always need to provide users with a reliable download method.

Downloading files directly from cloud storage (such as Azure Blob Storage, AWS S3) looks simple, but there are actually quite a few problems:

**Network Latency**: Cross-border and cross-region access is slow enough to make you want to smash your keyboard. Users wait forever—how can the experience be good?

**Bandwidth Costs**: Cloud storage egress traffic is heartbreakingly expensive. Accessing Azure Blob Storage from mainland China costs about 0.5 yuan per GB, which is 500 yuan for 1TB per month. For small teams, this amount is neither too much nor too little—after all, money doesn't grow on trees.

**Access Restrictions**: Access to foreign cloud services from certain regions is intermittent, and sometimes simply impossible. Users can't download when they want to—this situation itself is quite helpless.

**CDN Costs**: Commercial CDNs can indeed solve the problem, but the prices are indeed beautiful. How can small teams afford them?

So is there a way that's both cost-effective and easy to use? Actually, yes. Cloud server + reverse proxy + caching layer—simple and crude. Costs reduced by about half, speed improved—a small comfort indeed.

## Architecture Design

### Overall Architecture Approach

Let's first look at the overall architecture design:

```
User Request
    ↓
DNS Resolution
    ↓
┌─────────────────────────────────────┐
│   Reverse Proxy Layer (Traefik/Bunker Web)   │ ← SSL termination, routing distribution, security protection
├─────────────────────────────────────┤
│   Port: 80/443                       │
│   Functions: Automatic Let's Encrypt certificate      │
│         Host routing                    │
└─────────────────────────────────────┘
    ↓
┌─────────────────────────────────────┐
│   Caching Layer (Nginx)                     │ ← File caching, Gzip compression
├─────────────────────────────────────┤
│   Port: 8080(server) / 8081(desktop) │
│   Cache strategy:                          │
│   - index.json: 1 hour               │
│   - Other files: 7 days                   │
│   Cache size: 1GB                      │
└─────────────────────────────────────┘
    ↓
┌─────────────────────────────────────┐
│   Origin (Azure Blob Storage)         │ ← File storage
└─────────────────────────────────────┘
```

The core idea of this architecture, simply put, is: **Add a caching layer between users and cloud storage**.

User requests first reach the reverse proxy layer on the cloud server, then the Nginx caching layer takes over. File in cache? Give it directly to the user. Not there? Fetch a copy from cloud storage and conveniently store a copy locally. Next time accessing the same file, no need to bother cloud storage. It's actually like memory—once you remember something, you don't have to work hard to think about it again next time...

### Why Choose This Architecture?

**Cloud Server Advantages**:
- Controllable costs: Cloud providers like Alibaba Cloud offer cheap cloud servers (1-2 core 2GB configuration costs about 50-100 yuan per month)
- Flexible deployment: Freely configure reverse proxy and caching strategies
- Flexible geographic location: Choose server regions close to users
- Strong scalability: Upgrade configuration based on traffic demand

**Reverse Proxy + Caching Architecture**:
- Reduce origin pressure: Cache hot files, reduce access to cloud storage
- Lower costs: Cloud server traffic fees are far lower than cloud storage egress traffic
- Improve speed: Nearby access, server bandwidth is usually better than cloud storage

**Why Choose Nginx as the Caching Layer?**

This choice wasn't made casually—after all, Nginx does have its reasons:

1. High performance: Nginx reverse proxy performance is well-recognized in the industry
2. Mature caching: Built-in proxy_cache functionality, stable and reliable, won't frequently cause issues
3. Low resource usage: Can run with 256MB memory, friendly to the server
4. Flexible configuration: Different file types can have different caching strategies—this is quite thoughtful

## Reverse Proxy Layer: Traefik vs Bunker Web

HagiCode's deployment solution actually supports two reverse proxies—each has its own characteristics:

| Solution | Features | Use Cases |
|------|------|----------|
| **Traefik** | Lightweight, automatic SSL, simple configuration | Basic deployment, low traffic scenarios |
| **Bunker Web** | Built-in WAF, anti-DDoS, anti-crawler | High security requirements, high traffic scenarios |

### Traefik: Lightweight First Choice

Traefik is a modern HTTP reverse proxy and load balancer whose biggest feature is simple configuration and automatic Let's Encrypt certificates.

For initial deployment or low traffic scenarios, Traefik is actually quite a good choice:
- Low resource usage (1.5 CPU/512MB memory is enough)
- Automatic SSL certificate configuration, no need to worry about it yourself
- Routing configuration based on Docker labels, also quite convenient

### Bunker Web: High Security Scenarios

Bunker Web is an Nginx-based web application firewall with more comprehensive security protection.

When should you consider switching to Bunker Web? Probably in these situations:
- Suffering DDoS attacks (although no one hopes to encounter this)
- Need ModSecurity protection
- Want anti-crawler functionality
- Have higher security requirements

HagiCode provides the `switch-deployment.sh` script to quickly switch between the two solutions:

```bash
# Switch to Bunker Web
./switch-deployment.sh bunkerweb

# Switch back to Traefik
./switch-deployment.sh traefik

# View current status
./switch-deployment.sh status
```

The script automatically performs pre-checks, health checks, and can automatically rollback—the switching process is safe and reliable, won't crash while switching.

## Nginx Caching Layer Configuration

The caching layer is the core of the entire architecture. Good or bad Nginx configuration makes a huge difference in caching effectiveness. This matter is quite critical after all.

### Cache Path Configuration

```nginx
# Cache path configuration
proxy_cache_path /var/cache/nginx levels=1:2 keys_zone=azure_cache:10m
                   max_size=1g inactive=7d use_temp_path=off;
```

Parameter explanation:
- `levels=1:2`: Cache directory hierarchy, 2-level directory structure to improve file access efficiency
- `keys_zone=azure_cache:10m`: Cache key storage area, 10MB is enough to store a large number of keys
- `max_size=1g`: Maximum cache size 1GB
- `inactive=7d`: Cache files not accessed for 7 days are deleted
- `use_temp_path=off`: Write directly to cache directory to improve performance

### Tiered Caching Strategy

Different file types need different caching strategies:

```nginx
# Server download service
server {
    listen 8080;

    # index.json short-term caching (for timely updates)
    location /index.json {
        proxy_cache azure_cache;
        proxy_cache_valid 200 1h;
        proxy_cache_key "$scheme$server_port$request_uri";
        add_header X-Cache-Status $upstream_cache_status;
        add_header Cache-Control "public, max-age=3600";

        # Reverse proxy to Azure OSS
        proxy_pass https://${SERVER_DL_HOST}/${SERVER_DL_CONTAINER}$uri?${SERVER_DL_SAS_TOKEN};
        proxy_ssl_server_name on;
        proxy_ssl_protocols TLSv1.2 TLSv1.3;
    }

    # Installation packages and other static files long-term caching
    location / {
        proxy_cache azure_cache;
        proxy_cache_valid 200 7d;
        proxy_cache_key "$scheme$server_port$request_uri";
        add_header X-Cache-Status $upstream_cache_status;
        add_header Cache-Control "public, max-age=604800";

        proxy_pass https://${SERVER_DL_HOST}/${SERVER_DL_CONTAINER}$uri?${SERVER_DL_SAS_TOKEN};
        proxy_ssl_server_name on;
        proxy_ssl_protocols TLSv1.2 TLSv1.3;
    }
}
```

**Why Design It This Way?**

`index.json` is the version check file and needs timely updates, so cache time is set to 1 hour. This way after releasing a new version, users can detect the update within at most 1 hour—not too long.

Static files like installation packages don't change often, so caching for 7 days can dramatically reduce origin access. When updates are needed, manually clearing the cache is enough—not too troublesome.

**X-Cache-Status Response Header**:

This response header lets you see how well caching is hitting—also quite useful:
- `HIT`: Cache hit
- `MISS`: Cache miss, fetched from origin
- `EXPIRED`: Cache expired, re-fetched from origin
- `BYPASS`: Cache bypassed

View method:
```bash
curl -I https://server.dl.hagicode.com/app.zip
```

## Cost Analysis

Assuming 1TB download traffic per month, let's do the math:

| Solution | Traffic Cost | Server Cost | Total |
|------|----------|------------|------|
| Direct Azure OSS | About ¥500 | ¥0 | ¥500 |
| Cloud server + OSS (80% cache hit rate) | ¥100 + ¥80 | ¥60 | ¥240 |
| Commercial CDN | ¥300-500 | ¥0 | ¥300-500 |

**Conclusion**: The caching layer can save approximately 50% of distribution costs.

This calculation assumes an 80% cache hit rate. In actual situations, if file update frequency is not high, the hit rate may be even higher—this is also a natural outcome.

## Deployment Practice

### Environment Preparation

First configure environment variables:

```bash
cd /path/to/hagicode_aliyun_deployment/docker
cp .env.example .env
vi .env  # Fill in Azure OSS SAS URL, Lark Webhook URL
```

**Important**: The `.env` file contains sensitive information (SAS Token, Webhook URL)—never commit it to version control, this matter is quite critical.

### DNS Configuration

Add the following DNS A records—don't forget this step:
- `server.dl.hagicode.com` → Server IP
- `desktop.dl.hagicode.com` → Server IP

### Initialize Server

Use Ansible to automatically initialize the server:

```bash
cd /path/to/hagicode_aliyun_deployment
ansible-playbook -i ./ansible/inventory/hosts.yml ./ansible/playbooks/init.yml
```

This playbook will automatically handle these:
- Create deployment user
- Install Docker and Docker Compose
- Configure SSH keys
- Set up firewall rules

Not too complex—after all, automation saves time and effort.

### Deploy Services

```bash
./deploy.sh
```

The deployment script will help you with these:

One command to handle it all—quite convenient.

## Operations Tips

### Cache Management

Cache occasionally needs some attention:

**View cache disk usage**:
```bash
docker volume inspect docker_nginx-cache
du -sh /var/lib/docker/volumes/docker_nginx-cache/_data
```

**Manually clear cache**:
```bash
./clear-cache.sh
```

Or execute manually—although a bit troublesome, it works:
```bash
docker exec nginx sh -c "rm -rf /var/cache/nginx/*"
docker restart nginx
```

### Resource Limits

On a 1 core 2GB server, resource limit configuration is as follows:

```yaml
services:
  traefik:
    deploy:
      resources:
        limits:
          cpus: '1.50'
          memory: 512M

  nginx:
    deploy:
      resources:
        limits:
          cpus: '0.50'
          memory: 256M
```

Monitor resource usage—occasionally checking is also good:
```bash
docker stats
```

### SAS Token Security Practices

SAS Token is the credential for accessing Azure Blob Storage—leaking it is no joke:

- `.env` file not committed to version control (already in .gitignore)
- SAS Token set with appropriate expiration time (recommend 1 year)
- Restrict SAS Token permissions (read only)
- Regularly rotate SAS Token

### Monitoring Alerts

HagiCode integrates Lark/Feishu Webhook notifications, which can send notifications in the following situations:
- Deployment success/failure
- Cache clearing status
- Service anomalies

Notifications include server information, timestamps, error details—making it easier to quickly locate problems.

### High Availability Expansion

When a single server can't handle the load, you can also consider:

1. **Horizontal scaling**: Deploy multiple nodes, distribute through DNS round-robin or load balancing
2. **CDN support**: Add CDN in front of cloud server to further improve access speed
3. **Cache warming**: Use scripts to preload popular files into cache

## Precautions

A few points still need to be reminded—after all, no one wants to encounter issues:

1. **SSL certificates**: Let's Encrypt has rate limits, don't frequently switch deployments, or you might not be able to obtain certificates
2. **Cache clearing**: Remember to clear cache after updating important files, or users might not download the new version
3. **Log management**: Regularly clean Docker logs, or disk filling up becomes troublesome
4. **Backup strategy**: Traefik acme.json, Bunker Web configurations—better to back these up
5. **Monitoring alerts**: Configure Lark notifications to stay informed about deployment status and react quickly when problems occur
