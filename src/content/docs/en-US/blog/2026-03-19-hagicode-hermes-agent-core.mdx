## Why HagiCode Chooses Hermes as the Comprehensive Agent Core

> When building an AI-assisted coding platform, choosing the right Agent core directly determines the ceiling of system capabilities. After all, some things can't be forced—choose the wrong framework, and nothing will feel right no matter how much you tweak it. This article shares HagiCode's thinking in technology selection and the integration practice of Hermes Agent.

## Background

The most headache-inducing part of building AI-assisted coding is choosing the underlying Agent framework. Actually, there are quite a few options available on the market—it's just that some are too simple in functionality, some are too complex to deploy, and some have insufficient extensibility. We need a solution that can run on a $5 VPS and also connect to a GPU cluster. This requirement isn't too high, but it's enough to discourage quite a few people.

But the reality is that many so-called "all-purpose Agents" either can only run in the cloud or have ridiculously high local deployment requirements. After spending two weeks researching various solutions, we made a bold decision: to completely rebuild the Agent core and adopt Hermes as the underlying engine for the comprehensive Agent.

Everything that this decision brought about was perhaps meant to be.

## Why HagiCode Needs Hermes

Before introducing Hermes in detail, let me first explain why HagiCode has such requirements. Things in this world often don't work out the way you want them to—you always need to find a suitable excuse.

As an AI code assistant, HagiCode needs to support multiple usage scenarios simultaneously:
- Local development environment: Developers want to run on their own computers, keeping data local—these days, data security is neither too big nor too small of a deal
- Team collaboration environment: Small teams can share an Agent deployed on a server—to save money, everyone's having a hard time
- Cloud elastic scaling: When handling complex tasks, can automatically scale to a GPU cluster—better safe than sorry

This "having your cake and eating it too" requirement led us to Hermes. I don't know if this choice is right, but there weren't any better options at the time.

## What is Hermes Agent

Hermes Agent is an autonomous AI Agent created by **Nous Research**. Some people might not be familiar with Nous Research—they're the lab that developed open-source large models like Hermes, Nomos, and Psyché. Speaking of which, they've had it pretty tough too—having made so many good things, yet not many people know about them.

Unlike traditional IDE programming assistants or simple API chat wrappers, Hermes has one characteristic: the longer it runs, the stronger its capabilities. It doesn't just complete a one-time task and call it done, but can continuously learn and accumulate experience over long-term operation. This is quite like humans, isn't it?

## How HagiCode Integrates Hermes

Now that we've covered "why," let's look at "how." Some things, once you figure them out, you need to take action—just thinking without doing isn't the way to go.

### Provider Layer Abstraction

In HagiCode's architecture, all AI Providers implement a unified `IAIProvider` interface:

```csharp
public sealed class HermesCliProvider : IAIProvider, IVersionedAIProvider
{
    public ProviderCapabilities Capabilities { get; } = new ProviderCapabilities
    {
        SupportsStreaming = true,   // Support streaming output
        SupportsTools = true,       // Support tool calling
        SupportsSystemMessages = true, // Support system prompts
        SupportsArtifacts = false
    };
}
```

This abstraction layer allows HagiCode to seamlessly switch between different AI Providers—whether OpenAI, Claude, or Hermes, the upper-layer calling method is completely consistent. To put it plainly, it saves trouble.

### ACP Communication Protocol

Hermes uses **ACP (Agent Communication Protocol)** for communication. This is a protocol specifically designed for Agent communication, with main methods including:

| Method | Description |
|------|------|
| `initialize` | Initialize connection, get protocol version and client capabilities |
| `authenticate` | Handle authentication, supports multiple authentication methods |
| `session/new` | Create new session, set working directory and MCP servers |
| `session/prompt` | Send prompt and get response |

HagiCode implements the ACP transport layer through `StdioAcpTransport`, launching Hermes subprocess and communicating through standard input/output. This thing sounds complex, but it's actually okay to implement—mainly you need patience.

### Configuration Management

Configuration is managed through the `HermesPlatformConfiguration` class:

```csharp
public sealed class HermesPlatformConfiguration : IAcpPlatformConfiguration
{
    public string ExecutablePath { get; set; } = "hermes";
    public string Arguments { get; set; } = "acp";
    public int StartupTimeoutMs { get; set; } = 5000;
    public string ClientName { get; set; } = "HagiCode";
    public HermesAuthenticationConfiguration Authentication { get; set; }
    public HermesSessionDefaultsConfiguration SessionDefaults { get; set; }
}
```

### Complete Execution Flow

The following is the core execution logic of the Hermes Provider:

```csharp
private async IAsyncEnumerable<AIStreamingChunk> StreamCoreAsync(
    AIRequest request,
    string? embeddedCommandPrompt,
    [EnumeratorCancellation] CancellationToken cancellationToken)
{
    // 1. Create transport layer, launch Hermes subprocess
    await using var transport = new StdioAcpTransport(
        platformConfiguration.GetExecutablePath(),
        platformConfiguration.GetArguments(),
        platformConfiguration.GetEnvironmentVariables(),
        platformConfiguration.GetStartupTimeout(),
        _loggerFactory.CreateLogger<StdioAcpTransport>());
    await transport.ConnectAsync(cancellationToken);

    // 2. Initialize, get protocol version and authentication methods
    var initializeResult = await SendHermesRequestAsync(
        transport, nextRequestId++, "initialize",
        BuildInitializeParameters(platformConfiguration), cancellationToken);

    // 3. Handle authentication
    var authMethods = ParseAuthMethods(initializeResult);
    if (!isAuthenticated)
    {
        var methodId = platformConfiguration.Authentication.ResolveMethodId(authMethods);
        await SendHermesRequestAsync(transport, nextRequestId++, "authenticate", ...);
    }

    // 4. Create session
    var newSessionResult = await SendHermesRequestAsync(
        transport, nextRequestId++, "session/new",
        BuildNewSessionParameters(platformConfiguration, workingDirectory, model), cancellationToken);
    var sessionId = ParseSessionId(newSessionResult);

    // 5. Execute prompt and collect streaming response
    await foreach (var payload in transport.ReceiveMessagesAsync(cancellationToken))
    {
        // Handle session/update notifications, convert to streaming chunks
        if (TryParseSessionNotification(root, out var notification))
        {
            if (_responseMapper.TryConvertToStreamingChunk(notification, out var chunk))
            {
                yield return chunk;
            }
        }
    }
}
```

Code, seen enough, is just like that. What's important is the approach, right?

### Health Check

To ensure the availability of Hermes services, HagiCode implements a health check mechanism:

```csharp
public async Task<ProviderTestResult> PingAsync(CancellationToken cancellationToken = default)
{
    var response = await ExecuteAsync(
        new AIRequest
        {
            Prompt = "Reply with exactly PONG.",
            CessionId = null,
            AllowedTools = Array.Empty<string>(),
            WorkingDirectory = ResolveWorkingDirectory(null)
        },
        cancellationToken);

    var success = string.Equals(response.Content.Trim(), "PONG", StringComparison.OrdinalIgnoreCase);
    return new ProviderTestResult
    {
        ProviderName = Name,
        Success = success,
        ResponseTimeMs = stopwatch.ElapsedMilliseconds,
        ErrorMessage = success ? null : $"Unexpected Hermes ping response: '{response.Content}'."
    };
}
```

## Considerations in Practice

During the process of integrating Hermes, there are some pitfalls worth knowing about in advance. These days, who hasn't stepped in a few pits?

### Authentication Method Configuration

Hermes supports multiple authentication methods (API Key, Token, etc.), and you need to choose based on the actual deployment situation. Configuration errors will lead to connection failures, but error messages may not be intuitive enough. Sometimes error messages are far from the actual cause, and you need to troubleshoot slowly.

### MCP Server Configuration

When creating a session, you can configure a list of MCP servers for Hermes to call external tools. But note:
- MCP server addresses must be accessible
- Timeout settings should be reasonable
- Degradation handling when servers are unavailable

In this world, it's hard to guard against everything.

### Working Directory Management

Each session needs to specify a working directory to ensure Hermes can correctly access project files. For multi-project scenarios, you need to dynamically switch working directories. It sounds simple, but there are quite a few cases to consider.

### Response Aggregation Processing

Hermes responses may be scattered across `session/update` notifications and final results, requiring proper merging and handling, otherwise content loss will occur. I've suffered from this too, it gets better slowly.

### Error Handling Strategy

Runtime errors should be explicitly returned rather than silently falling back to other Providers. This way users know it's Hermes that has a problem, not inexplicably switching to another model. After all, muddling through isn't the way to muddle through.

HagiCode choosing Hermes as the comprehensive Agent core wasn't a snap decision, but a careful choice based on actual requirements and technical characteristics. Whether this choice is right, it's too early to say, but so far it's working quite well.
