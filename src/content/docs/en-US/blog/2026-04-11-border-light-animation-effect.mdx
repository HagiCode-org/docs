## Border Light Animation Effect Implementation Guide

> How to create that eye-catching element with pure CSS? It's not that difficult, just takes a bit of a roundabout approach. This article will guide you through implementing border light rotation animation from scratch, and also share the pitfalls we encountered in the HagiCode project.

## Background

As frontend developers, we've all had this experience: a product manager comes over with that "this requirement is simple" look on their face — "Can you add an effect to this running task so users can see it at a glance?"

You say sure, let's add a border color change. But they shake their head, with that "you don't get it" look in their eyes: "Not obvious enough, I want that effect where light runs around the border, like in sci-fi movies."

At this point you might be wondering: how to implement this? Use Canvas? SVG? Or can CSS handle it? After all, nobody wants to admit they don't know.

Actually, border light rotation animation is quite common in modern web applications, mainly used in these scenarios:

- **Status indication**: Mark running tasks or active items
- **Visual focus**: Highlight important content areas
- **Brand enhancement**: Create a technological and modern visual experience
- **Holiday themes**: Create celebratory atmosphere for special occasions

We encountered this requirement when building [HagiCode](https://hagicode.com) — users need to see at a glance which sessions are running and which proposals are being processed. We tried several approaches, some easier, some more roundabout, and finally settled on a fairly mature implementation.

## About HagiCode

The solution shared in this article comes from our practical experience in the [HagiCode](https://hagicode.com) project. HagiCode is an AI-driven code assistant project that extensively uses border light animation in the interface to indicate various running states, such as session list running status, proposal flow diagram state transitions, throughput indicator intensity display, and more.

Actually, these effects aren't complicated in principle, but we encountered many pitfalls during implementation. If you want to see the actual effect, you can visit our [GitHub repository](https://github.com/HagiCode-org/site) or check out the [official website](https://hagicode.com). After all, what works is what matters.

## Core Implementation Ideas

Through analysis of the HagiCode code, we've summarized several core implementation patterns below. Each has its applicable scenarios, or rather, each has its reason to exist.

### 1. Conic Gradient Rotating Glow (Most Common)

This is the most classic border light rotation implementation. The core idea is to use CSS's `conic-gradient` to create a conical gradient, then make it spin. Like a streetlamp at night, just spinning round and round.

**Key Elements:**
- Create a glow layer with `::before` pseudo-element
- Define gradient color distribution with `conic-gradient`
- Mask the center area with `::after` pseudo-element (optional)
- Implement rotation animation with `@keyframes`

### 2. Side Glow Line

This is suitable for list item status indication. Just create a glowing thin line on one side of the element, no need to animate the entire border. Sometimes, a little light is enough, no need to illuminate the whole world.

**Key Elements:**
- Absolute positioned thin line element
- Create glow effect with `box-shadow`
- Implement breathing animation with `scale` and `opacity`

### 3. Box-Shadow Glow Background

If you don't need the wrapping effect, just want a soft background glow, then multiple `box-shadow` layers are enough. Sometimes, simpler is better.

### 4. Accessibility Support

This is easily overlooked but particularly important. All animations should consider the `prefers-reduced-motion` media query to provide a static alternative for users who don't like animations. After all, not everyone likes things moving around, and respecting everyone's choice is the right thing to do.

## Implementation Solutions

### Solution 1: Conic Gradient Rotating Border (Recommended)

This is the most complete surrounding light effect implementation, and also the most used solution in HagiCode. After all, if something works well, why change it?

```css
/* Parent container */
.glow-border-container {
  position: relative;
  overflow: hidden;
}

/* Rotating glow layer */
.glow-border-container::before {
  content: '';
  position: absolute;
  top: -50%;
  left: -50%;
  width: 200%;
  height: 200%;
  background: conic-gradient(
    transparent 0deg,
    rgba(59, 130, 246, 0.6) 60deg,
    rgba(59, 130, 246, 0.3) 120deg,
    rgba(59, 130, 246, 0.6) 180deg,
    transparent 240deg
  );
  animation: border-rotate 3s linear infinite;
  z-index: -1;
}

/* Mask layer (optional, for creating hollow border effect) */
.glow-border-container::after {
  content: '';
  position: absolute;
  inset: 2px;
  background: inherit;
  border-radius: inherit;
  z-index: -1;
}

@keyframes border-rotate {
  from {
    transform: rotate(0deg);
  }
  to {
    transform: rotate(360deg);
  }
}
```

The principle of this solution is quite simple: create a pseudo-element larger than the parent container, draw a conical gradient on it, then make it spin continuously. The parent container sets `overflow: hidden`, so you can only see the light at the border portion spinning. It's like watching a streetlamp outside from a window, you can only see that small segment as it passes.

### Solution 2: Simplified Rotating Light Border

If you don't need such a complex effect, HagiCode has a lighter utility class implementation. Sometimes simpler is better.

```css
/* Rotating light border utility class */
.running-light-border {
  position: absolute;
  inset: -2px;
  background: conic-gradient(
    from 0deg,
    transparent 0deg 270deg,
    var(--theme-running-color) 270deg 360deg
  );
  border-radius: inherit;
  animation: lightRayRotate 3s linear infinite;
  will-change: transform;
  z-index: 0;
}

@keyframes lightRayRotate {
  from {
    transform: rotate(0deg);
  }
  to {
    transform: rotate(360deg);
  }
}

/* Accessibility support */
@media (prefers-reduced-motion: reduce) {
  .running-light-border {
    animation: none;
  }
}
```

Note the `will-change: transform` here. This tells the browser "this element will keep changing", so the browser will do some optimizations in advance, making the animation smoother. After all, being prepared is better than scrambling at the last minute.

### Solution 3: Side Glow Line

This is particularly suitable for list item status indication. HagiCode's session list uses this solution. A thin line that stands out among many items — isn't that also a life philosophy?

```css
.side-glow {
  position: relative;
  isolation: isolate;
}

.side-glow::before {
  content: '';
  position: absolute;
  left: 0;
  top: 14px;
  bottom: 14px;
  width: 1px;
  border-radius: 999px;
  background: var(--theme-running-color);
  box-shadow:
    0 0 16px var(--theme-running-color),
    0 0 28px var(--theme-running-color);
  z-index: 1;
  pointer-events: none;
  animation: sidePulse 2.6s ease-in-out infinite;
}

.side-glow > * {
  position: relative;
  z-index: 2;
}

@keyframes sidePulse {
  0%, 100% {
    opacity: 0.55;
    transform: scaleY(0.96);
  }
  50% {
    opacity: 0.95;
    transform: scaleY(1);
  }
}
```

Here we use `isolation: isolate` to create a new stacking context, then control the display order of layers with `z-index`. `pointer-events: none` is also crucial, otherwise the pseudo-element will block user click operations. Like some things — nice to look at, but they shouldn't get in the way.

### Solution 4: React Component Encapsulation

If your project uses React, you can wrap a component to handle these logic parts, especially the accessibility portion. After all, writing code once and using it many times is what we want.

```tsx
import React from 'react';
import { useReducedMotion } from 'framer-motion';
import styles from './GlowBorder.module.css';

interface GlowBorderProps {
  isActive: boolean;
  children: React.ReactNode;
  className?: string;
}

export const GlowBorder = React.memo<GlowBorderProps>(
  ({ isActive, children, className = '' }) => {
    const prefersReducedMotion = useReducedMotion();

    if (!isActive) {
      return <div className={className}>{children}</div>;
    }

    if (prefersReducedMotion) {
      return (
        <div className={`${styles.glowStatic} ${className}`}>
          {children}
        </div>
      );
    }

    return (
      <div className={`${styles.glowAnimated} ${className}`}>
        {children}
      </div>
    );
  }
);
```

Corresponding CSS module:

```css
/* GlowBorder.module.css */

/* Animated version */
.glowAnimated {
  position: relative;
  overflow: hidden;
}

.glowAnimated::before {
  content: '';
  position: absolute;
  top: -50%;
  left: -50%;
  width: 200%;
  height: 200%;
  background: conic-gradient(
    from 0deg,
    transparent,
    rgba(59, 130, 246, 0.6),
    transparent,
    rgba(59, 130, 246, 0.6),
    transparent
  );
  animation: rotateGlow 3s linear infinite;
  z-index: -1;
}

.glowAnimated::after {
  content: '';
  position: absolute;
  inset: 2px;
  background: inherit;
  border-radius: inherit;
  z-index: -1;
}

/* Static version (accessibility) */
.glowStatic {
  position: relative;
  border: 1px solid rgba(59, 130, 246, 0.5);
  box-shadow: 0 0 15px rgba(59, 130, 246, 0.3);
}

@keyframes rotateGlow {
  from {
    transform: rotate(0deg);
  }
  to {
    transform: rotate(360deg);
  }
}
```

The `useReducedMotion` hook from `framer-motion` automatically detects the user's system preferences. If the user has enabled "reduce motion", it returns true, and then the static version is displayed. After all, respecting user choice is more important than forcing display.

## Practical Experience Sharing

Below are the experiences we summarized from pitfalls encountered while building HagiCode. Just some ramblings, hope they can help you who come after.

### 1. Theme Variable System

Using CSS variables to implement multi-theme support is particularly convenient. After all, who wants to modify a bunch of code every time you switch themes?

```css
:root {
  --glow-color-light: rgb(16, 185, 129);
  --glow-color-dark: rgb(16, 185, 129);
  --theme-glow-color: var(--glow-color-light);
}

html.dark {
  --theme-glow-color: var(--glow-color-dark);
}

/* Usage */
.glow-effect {
  background: var(--theme-glow-color);
  box-shadow: 0 0 20px var(--theme-glow-color);
}
```

This way, when switching themes, you only need to change the `html` tag's class, and all animation colors will automatically update. One codebase, two styles — isn't that what we pursue?

### 2. Performance Optimization

**Use `will-change` to hint browser optimization:**

```css
.animated-glow {
  will-change: transform, opacity;
}
```

Tell the browser in advance, and it will help you do some optimizations. Like many things in life, being prepared is always good.

**Avoid using complex box-shadow on large area elements:**

```css
/* Bad - using blurry shadow on large area elements */
.large-card {
  box-shadow: 0 0 50px rgba(0, 0, 0, 0.5);
}

/* Better - use pseudo-element to limit glow area */
.large-card::before {
  content: '';
  position: absolute;
  inset: 0;
  border-radius: inherit;
  box-shadow: 0 0 20px var(--glow-color);
  pointer-events: none;
}
```

We tested this in HagiCode — adding blurry shadows directly on large cards would drop scroll frame rate below 30fps, but after switching to pseudo-elements, it stayed at 60fps. Users can feel this difference in experience.

### 3. Accessibility

This really can't be skipped. Some users feel dizzy or find animations noisy, respecting their choice is basic etiquette for product design. Beautiful things shouldn't be forced on people.

**CSS Media Query:**

```css
@media (prefers-reduced-motion: reduce) {
  .glow-animation {
    animation: none;
  }

  .glow-animation::before {
    /* Provide static alternative */
    opacity: 1;
  }
}
```

**Detect user preferences in React:**

```tsx
import { useReducedMotion } from 'framer-motion';

const Component = () => {
  const prefersReducedMotion = useReducedMotion();

  return (
    <div className={prefersReducedMotion ? 'static-glow' : 'animated-glow'}>
      Content
    </div>
  );
};
```

### 4. Intensity Level Control

The Token throughput indicator in HagiCode displays different colored lights based on real-time throughput, this is dynamically implemented. Different states should have different expression methods.

```tsx
const colors = [
  null,       // Level 0 - no color
  '#3b82f6',  // Level 1 - Blue
  '#34d399',  // Level 2 - Emerald
  '#facc15',  // Level 3 - Yellow
  '#fbbf24',  // Level 4 - Amber
  '#f97316',  // Level 5 - Orange
  '#22d3ee',  // Level 6 - Cyan
  '#d946ef',  // Level 7 - Fuchsia
  '#f43f5e',  // Level 8 - Rose
];

const IntensityGlow = ({ intensity }) => {
  const glowColor = colors[Math.min(intensity, colors.length - 1)];

  return (
    <div
      className="glow-effect"
      style={{
        '--glow-color': glowColor,
        opacity: 0.6 + (intensity * 0.08),
      }}
    />
  );
};
```

### 5. Notes

Some details still need attention, otherwise you'll only know after falling into a pit.

| Note | Description |
|---------|------|
| **z-index management** | Glow layer should have appropriate z-index to avoid affecting content interaction |
| **pointer-events** | Glow pseudo-elements should have `pointer-events: none` |
| **Boundary overflow** | Parent container needs `overflow: hidden` or adjust pseudo-element size |
| **Performance impact** | Complex animations may affect performance on mobile devices, testing needed |
| **Dark mode** | Ensure glow colors are clearly visible on dark backgrounds |
| **Theme switching** | Use CSS variables to ensure animation colors update correctly when switching themes |

### 6. Debugging Tips

Pseudo-elements are sometimes hard to find in developer tools. You can temporarily add a border to see the position.

```css
/* Temporarily show pseudo-element boundary for debugging */
.glow-effect::before {
  /* debug: border: 1px solid red; */
}
```

After adjusting the position, remember to comment out or delete this line, otherwise production environment will be awkward. Some things are better left in development environment.

## Summary

Border light rotation animation is neither difficult nor simple. The core is `conic-gradient` plus rotation, but to achieve good performance, maintainability, and accessibility friendliness, there are many details to pay attention to.

HagiCode encountered many pitfalls here and also summarized some best practices. This is how projects are — trial and error, improvement after improvement. If you're working on similar requirements, hope this article helps you take fewer detours.

After all, some things you have to practice yourself to know the depth.
