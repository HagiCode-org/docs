## Design.md: A Solution for Consistent AI-Driven Frontend UI Design

> In the era of AI-assisted frontend development, how can we maintain consistency in AI-generated UI? This article shares our practical experience building a design gallery site based on awesome-design-md, and how to create structured design.md to guide AI in standardized UI design.

## Background

Anyone who has used AI to write frontend code has likely had a similar experience: the same page, generated by AI multiple times, yields different styles each time. Sometimes it's rounded corners, sometimes square corners; sometimes spacing is 8px, sometimes 16px; even the same button looks different across different conversations.

This is not an isolated phenomenon. With the proliferation of AI-assisted development, **inconsistency in AI-generated frontend UI** has become a widespread problem. Different AI assistants, different prompts, and even the same assistant in different conversations all produce vastly different interface designs. This brings enormous maintenance costs to product iteration.

The root of the problem is actually quite simple: lack of an **authoritative design reference document**. Traditional CSS stylesheet files can only tell developers "how to implement," but cannot fully convey "why it's designed this way" and "what design patterns to use in what scenarios." For AI, it needs a clear structured description to understand design specifications.

At the same time, the open source community already has some excellent resources. The [VoltAgent/awesome-design-md](https://github.com/VoltAgent/awesome-design-md) project collects design system documentation from many well-known companies, with each directory containing README.md, DESIGN.md, and preview HTML. But these are scattered across upstream repositories, making them difficult to quickly review and compare.

So can we integrate these resources into a convenient-to-browse design gallery, while consolidating a structured design.md for AI to use?

The answer is yes. Let me share our solution.

## Analysis

Before writing code, let's break down several technical challenges of this problem.

### Content Source Management: How to unify scattered design resources?

The upstream awesome-design-md repository contains a large number of design documents, but we need a way to incorporate it into our project.

**Solution: Use git submodule**

```
awesome-design-md-site
└── vendor/awesome-design-md          # Upstream resources (git submodule)
```

This has several benefits:
- Version control: can lock specific upstream versions
- Offline build: no need to request external APIs during build
- Content review: can see specific changes in PRs

## Solution

Based on the above analysis, our solution contains four core modules.

### 1. Content Ingestion Pipeline

This is the foundation of the entire system, responsible for extracting and standardizing content from upstream resources.

```typescript
// src/lib/content/awesomeDesignCatalog.ts

export async function scanSourceEntries(): Promise<DesignEntry[]> {
  const designDir = 'vendor/awesome-design-md/design-md';
  const entries: DesignEntry[] = [];

  for (const dir of await fs.readdir(designDir)) {
    const entryPath = path.join(designDir, dir);
    if (await isValidDesignEntry(entryPath)) {
      const entry = await normalizeDesignEntry(entryPath);
      entries.push(entry);
    }
  }

  return entries;
}

async function isValidDesignEntry(dir: string): Promise<boolean> {
  const requiredFiles = ['README.md', 'DESIGN.md'];
  for (const file of requiredFiles) {
    if (!(await fileExists(path.join(dir, file)))) {
      return false;
    }
  }
  return true;
}
```

### 3. Search Functionality

Search functionality is based on client-side filtering, using URL query parameters to maintain state:

```typescript
// src/components/gallery/SearchToolbar.tsx

function SearchToolbar({ entries }: { entries: DesignEntry[] }) {
  const [query, setQuery] = useState('');
  const [results, setResults] = useState(entries);

  useEffect(() => {
    const params = new URLSearchParams(window.location.search);
    const initialQuery = params.get('q') || '';
    setQuery(initialQuery);
    filterEntries(initialQuery);
  }, []);

  const filterEntries = (searchQuery: string) => {
    const filtered = entries.filter(entry =>
      entry.searchText.toLowerCase().includes(searchQuery.toLowerCase())
    );
    setResults(filtered);
  };

  const handleChange = (e: React.ChangeEvent<HTMLInputElement>) => {
    const value = e.target.value;
    setQuery(value);
    filterEntries(value);

    // Update URL (doesn't trigger page refresh)
    const newUrl = value
      ? `${window.location.pathname}?q=${encodeURIComponent(value)}`
      : window.location.pathname;
    window.history.replaceState({}, '', newUrl);
  };

  return (
    <div className="search-toolbar">
      <input
        type="text"
        value={query}
        onChange={handleChange}
        placeholder="Search design entries..."
      />
      <span className="result-count">{results.length} results</span>
    </div>
  );
}
```

## Practice

**Step 1: Initialize Submodule**

```bash
# Add upstream repository as submodule
git submodule add https://github.com/VoltAgent/awesome-design-md.git vendor/awesome-design-md

# Initialize and update submodule
git submodule update --init --recursive
```

**Step 2: Create Content Pipeline**

Implement `awesomeDesignCatalog.ts`, including:
- File scanning and validation logic
- Markdown rendering (using Astro's built-in renderer)
- Entry data extraction

**Step 3: Build Gallery UI**

Use Astro + React Islands to create:
- Homepage gallery layout (card grid)
- Design card component
- Search toolbar
- Detail page layout

**Step 4: Write Design Document**

Based on ClickHouse DESIGN.md structure, fill in actual design tokens from your project. Update README.md, add links pointing to design.md.

### Considerations

**Security**: Markdown rendering needs to filter unsafe HTML. Astro's built-in renderer filters script tags by default, but still need to be aware of XSS risks.

**Performance**: Large numbers of iframe previews may affect first-screen load. Recommend using `loading="lazy"` to defer loading preview content.

**Maintainability**: design.md needs to stay synchronized with code implementation. Recommend adding checks in CI to ensure CSS variables are consistent between documentation and code.

**Accessibility**: Ensure color contrast meets WCAG AA standards (at least 4.5:1).

### AI Usage Guide

After creating design.md, how to make AI actually use it? Here are some practical tips:

**Tip 1: Explicitly reference in prompts**

```
Please refer to the design.md file in the project root directory, using the design specifications defined therein to implement the following components:
- Button: use primary color, 8px border radius
- Card: use elevation-2 shadow level
```

**Tip 2: Require AI to reference specific CSS variables**

```
Implement a navigation bar, requiring:
- Background color use --color-bg-primary
- Border use --color-border-subtle
- Text use --text-color-primary
```

**Tip 3: Include design.md content in system prompt**

If your AI tool supports custom system prompts, you can directly add the core content of design.md.

### Testing Strategy

**Content Pipeline Testing**:
- File missing scenarios (missing README.md or DESIGN.md)

**Search Functionality Testing**:

**UI Component Testing**:
- Light/dark theme switching
- Responsive layout
