## Automated Image Synchronization from Docker Hub to Azure ACR

> This article introduces how to use GitHub Actions and the image-syncer tool to achieve automated synchronization of Docker Hub images to Azure Container Registry, addressing slow access to Docker Hub in China and some Azure regions, and improving image availability and deployment efficiency in Azure environments.

## Background/Introduction

The HagiCode project uses Docker images as core runtime components, with primary images hosted on Docker Hub. As the project evolves and Azure deployment requirements increase, we encountered the following pain points:
- Slow image pull speeds, with Docker Hub access restricted in China and some Azure regions
- Single point of failure risk from relying on a single image source
- Better network performance and integration experience when using Azure Container Registry in Azure environments

To address these issues, we need to establish an automated image synchronization mechanism to periodically sync Docker Hub images to Azure ACR, ensuring users can achieve faster image pull speeds and higher availability in Azure environments.

## Technical Solution Comparison

When developing our solution, we compared multiple technical approaches:

### 1. image-syncer (Final Choice)
- Incremental sync: Only syncs changed image layers, significantly reducing network transfer
- Resumable transfer: Can resume sync after network interruption
- Concurrency control: Supports configurable concurrent thread count, improving efficiency for large images
- Comprehensive error handling: Built-in failure retry mechanism (default 3 times)
- Lightweight deployment: Single binary file, no dependencies
- Multi-registry support: Compatible with Docker Hub, Azure ACR, Harbor, etc.

### 2. Docker CLI
- No incremental sync support: Requires pulling complete image content each time
- Lower efficiency: Large network transfer, long duration
- Simple and easy to use: Uses familiar docker pull/push commands

### 3. Azure CLI
- High complexity: Requires Azure CLI authentication configuration
- Functional limitations: az acr import functionality is relatively basic
- Native integration: Integrates well with Azure services

## Architecture Design Decisions

### Decision 1: Set sync frequency to daily at UTC 00:00
- Balances image freshness and resource consumption
- Avoids business peak hours, reducing impact on other operations
- Docker Hub images are typically updated after daily builds

### Decision 2: Sync all image tags
- Maintains complete consistency with Docker Hub
- Provides flexible version selection for users
- Simplifies sync logic, avoiding complex tag filtering rules

### Decision 3: Use GitHub Secrets to store authentication information
- Native GitHub Actions support with high security
- Simple configuration, easy to manage and maintain
- Supports repository-level access control

## Risk Assessment and Mitigation

### Risk 1: Azure ACR authentication information leakage
- Use GitHub Secrets for encrypted storage
- Regularly rotate ACR passwords
- Limit ACR user permissions to push-only
- Monitor ACR access logs

### Risk 2: Sync failure causing image inconsistency
- image-syncer built-in incremental sync mechanism
- Automatic failure retry (default 3 times)
- Detailed error logs and failure notifications
- Resumable transfer functionality

### Risk 3: Excessive resource consumption
- Incremental sync reduces network transfer
- Configurable concurrent thread count (currently set to 10)
- Monitor number and size of synced images
- Run sync during off-peak hours

## Core Solution

We adopted the GitHub Actions + image-syncer automation approach to achieve image synchronization from Docker Hub to Azure ACR.

## Implementation Steps

### 1. Preparation Phase
- Create or confirm Azure Container Registry in Azure Portal
- Create ACR access keys (username and password)
- Confirm Docker Hub image repository access permissions

### 2. Configure GitHub Secrets
Add the following Secrets in GitHub repository settings:
- AZURE_ACR_USERNAME: Azure ACR username
- AZURE_ACR_PASSWORD: Azure ACR password

### 3. Create GitHub Actions Workflow
Configure the workflow in .github/workflows/sync-docker-acr.yml:
- Scheduled trigger: Daily at UTC 00:00
- Manual trigger: Support workflow_dispatch
- Additional trigger: Triggered when pushing to publish branch (for quick sync)

### 4. Workflow Execution Flow

| Step | Participant | Action | Description |
| --- | --- | --- | --- |
| 1 | GitHub Actions | Trigger workflow | Triggered by schedule, manual, or `publish` branch push |
| 2 | GitHub Actions → image-syncer | Download and execute sync tool | Enter actual sync phase |
| 3 | image-syncer → Docker Hub | Fetch image manifest and tag list | Read source repository metadata |
| 4 | image-syncer → Azure ACR | Fetch existing image info from target repository | Determine current target state |
| 5 | image-syncer | Compare source and target differences | Identify image layers needing sync |
| 6 | image-syncer → Docker Hub | Pull changed image layers | Only transfer content needing updates |
| 7 | image-syncer → Azure ACR | Push changed image layers | Complete incremental sync |
| 8 | image-syncer → GitHub Actions | Return sync statistics | Include results, differences, and error information |
| 9 | GitHub Actions | Log and upload artifact | Facilitate subsequent audit and troubleshooting |

## GitHub Actions Workflow Implementation

Below is the actual workflow configuration (.github/workflows/sync-docker-acr.yml):

```yaml
name: Sync Docker Image to Azure ACR

on:
  schedule:
    - cron: "0 0 * * *" # Daily at UTC 00:00
  workflow_dispatch: # Manual trigger
  push:
    branches: [publish]

permissions:
  contents: read

jobs:
  sync:
    runs-on: ubuntu-latest

    steps:
      - name: Checkout code
        uses: actions/checkout@v4

      - name: Download image-syncer
        run: |
          # Download image-syncer binary
          wget https://github.com/AliyunContainerService/image-syncer/releases/download/v1.5.5/image-syncer-v1.5.5-linux-amd64.tar.gz
          tar -zxvf image-syncer-v1.5.5-linux-amd64.tar.gz
          chmod +x image-syncer

      - name: Create auth config
        run: |
          # Generate authentication config file (YAML format)
          cat > auth.yaml <<EOF
          hagicode.azurecr.io:
            username: "${{ secrets.AZURE_ACR_USERNAME }}"
            password: "${{ secrets.AZURE_ACR_PASSWORD }}"
          EOF

      - name: Create images config
        run: |
          # Generate image sync config file (YAML format)
          cat > images.yaml <<EOF
          docker.io/newbe36524/hagicode: hagicode.azurecr.io/hagicode
          EOF

      - name: Run image-syncer
        run: |
          # Execute sync (using new --auth and --images parameters)
          ./image-syncer --auth=./auth.yaml --images=./images.yaml --proc=10 --retries=3

      - name: Upload logs
        if: always()
        uses: actions/upload-artifact@v4
        with:
          name: sync-logs
          path: image-syncer-*.log
          retention-days: 7
```

## Configuration Description

### 1. Trigger Conditions
- Scheduled trigger: cron: "0 0 * * *" - Execute daily at UTC 00:00
- Manual trigger: workflow_dispatch - Allows users to manually run via GitHub UI
- Push trigger: push: branches: [publish] - Triggered when pushing to publish branch (for quick sync)

### 2. Authentication Configuration (auth.yaml)
```yaml
hagicode.azurecr.io:
  username: "${{ secrets.AZURE_ACR_USERNAME }}"
  password: "${{ secrets.AZURE_ACR_PASSWORD }}"
```

### 3. Image Sync Configuration (images.yaml)
```yaml
docker.io/newbe36524/hagicode: hagicode.azurecr.io/hagicode
```
This configuration means syncing all tags of docker.io/newbe36524/hagicode to hagicode.azurecr.io/hagicode

### 4. image-syncer Parameters
- --auth=./auth.yaml: Path to authentication config file
- --images=./images.yaml: Path to image sync config file
- --proc=10: Concurrent thread count set to 10
- --retries=3: Retry 3 times on failure

## Notes and Best Practices

### 1. Security Recommendations
- Regularly rotate Azure ACR passwords (recommended every 90 days)
- Use a dedicated ACR service account with push-only permissions
- Monitor ACR access logs to detect anomalous access promptly
- Do not output authentication information in logs
- Do not commit authentication information to the code repository

### 2. Performance Optimization
- Adjust --proc parameter: Adjust concurrency based on network bandwidth (recommended 5-20)
- Monitor sync time: If sync takes too long, consider reducing concurrency
- Regularly clean up logs: Set reasonable retention-days (currently 7 days)

### 3. Troubleshooting

#### Issue 1: Authentication Failure
```
Error: failed to authenticate to hagicode.azurecr.io
```
Solution:
1. Check if GitHub Secrets are correctly configured
2. Verify if Azure ACR password has expired
3. Confirm if ACR service account permissions are correct

#### Issue 2: Network Timeout
```
Error: timeout waiting for response
```
Solution:
1. Check network connection
2. Reduce concurrent thread count (--proc parameter)
3. Re-trigger workflow after network recovery

#### Issue 3: Incomplete Image Sync
```
Warning: some tags failed to sync
```
Solution:
1. Check sync logs to identify failed tags
2. Manually trigger workflow to re-sync
3. Verify if Docker Hub source images are normal

### 4. Monitoring and Alerts
- Regularly check Actions page to confirm workflow run status
- Set up GitHub notifications to get timely workflow failure alerts
- Monitor Azure ACR storage usage
- Regularly verify image tag consistency

## FAQs and Solutions

### Q1: How to sync specific tags instead of all tags?

Modify the images.yaml configuration file:
```yaml
# Only sync latest and v1.0 tags
docker.io/newbe36524/hagicode:latest: hagicode.azurecr.io/hagicode:latest
docker.io/newbe36524/hagicode:v1.0: hagicode.azurecr.io/hagicode:v1.0
```

### Q2: How to sync multiple image repositories?

Add multiple configuration lines in images.yaml:
```yaml
docker.io/newbe36524/hagicode: hagicode.azurecr.io/hagicode
docker.io/newbe36524/another-image: hagicode.azurecr.io/another-image
```

### Q3: How to retry after sync failure?
- Auto retry: image-syncer built-in retry mechanism (default 3 times)
- Manual retry: Click Re-run all jobs on GitHub Actions page

### Q4: How to view detailed sync progress?
- View real-time logs on Actions page
- Download sync-logs artifact to view complete log file
- Log files contain sync status and transfer speed for each tag

### Q5: How long does sync take?
- First full sync: Typically 10-30 minutes depending on image size
- Incremental sync: Usually 2-5 minutes if image changes are small
- Time depends on network bandwidth, image size, and concurrency settings
