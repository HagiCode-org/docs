## Automating Image Synchronization from Docker Hub to Azure ACR

> This article explains how to use GitHub Actions and the image-syncer tool to automate image synchronization from Docker Hub to Azure Container Registry, addressing the slow access to Docker Hub in China and some Azure regions while improving image availability and deployment efficiency in Azure environments.

## Background / Introduction

The HagiCode project uses Docker images as its core runtime components, with the primary images hosted on Docker Hub. As the project has grown and Azure deployment needs have increased, we encountered the following pain points:

- Slow image pull speeds, with Docker Hub access constrained in China and some Azure regions
- A single image source creates a single point of failure risk
- Azure Container Registry provides better network performance and integration when running in Azure

To solve these problems, we needed an automated image synchronization mechanism to regularly sync images from Docker Hub to Azure ACR, ensuring that users can get faster image pull speeds and higher availability in Azure environments.

## Technical Solution Comparison

When defining the solution, we compared multiple technical options:

### 1. image-syncer (final choice)
- Incremental synchronization: only syncs changed image layers, significantly reducing network transfer
- Resume support: synchronization can continue after network interruptions
- Concurrency control: configurable worker threads improve large image synchronization efficiency
- Robust error handling: built-in retry mechanism for failures (3 retries by default)
- Lightweight deployment: a single binary with no dependencies
- Multi-registry support: compatible with Docker Hub, Azure ACR, Harbor, and more

### 2. Docker CLI
- No incremental synchronization: the full image content must be pulled every time
- Lower efficiency: large network transfer volume and longer execution time
- Simple to use: based on familiar `docker pull` and `docker push` commands

### 3. Azure CLI
- Higher complexity: Azure CLI authentication must be configured
- Limited functionality: `az acr import` is relatively narrow in scope
- Native integration: integrates well with Azure services

## Architecture Design Decisions

### Decision 1: Set synchronization frequency to daily at 00:00 UTC
- Balances image freshness and resource consumption
- Avoids business peak hours and reduces impact on other operations
- Docker Hub images are typically updated after daily builds

### Decision 2: Synchronize all image tags
- Maintains full consistency with Docker Hub
- Gives users flexible version choices
- Simplifies synchronization logic and avoids complex tag filtering rules

### Decision 3: Use GitHub Secrets to store credentials
- Native GitHub Actions support with strong security
- Simple configuration and easy maintenance
- Supports repository-level access control

## Risk Assessment and Mitigation

### Risk 1: Azure ACR credential leakage
- Store credentials encrypted in GitHub Secrets
- Rotate ACR passwords regularly
- Restrict the ACR account to push-only permissions
- Monitor ACR access logs

### Risk 2: Synchronization failures causing image inconsistency
- image-syncer includes a built-in incremental synchronization mechanism
- Automatic retries on failure (3 by default)
- Detailed error logs and failure notifications
- Resume capability

### Risk 3: Excessive resource consumption
- Incremental synchronization reduces network transfer
- Configurable concurrency (`10` in the current setup)
- Monitor the number and size of synchronized images
- Run synchronization during off-peak hours

## Core Solution

We use an automated solution based on GitHub Actions + image-syncer to synchronize images from Docker Hub to Azure ACR.

## Implementation Steps

### 1. Preparation
- Create or confirm an Azure Container Registry in Azure Portal
- Create ACR access credentials (username and password)
- Confirm access permissions for the Docker Hub image repository

### 2. Configure GitHub Secrets
Add the following Secrets in the GitHub repository settings:
- AZURE_ACR_USERNAME: Azure ACR username
- AZURE_ACR_PASSWORD: Azure ACR password

### 3. Create the GitHub Actions workflow
Configure the workflow in `.github/workflows/sync-docker-acr.yml`:
- Scheduled trigger: every day at 00:00 UTC
- Manual trigger: supports `workflow_dispatch`
- Additional trigger: runs on pushes to the `publish` branch for fast synchronization

### 4. Workflow execution flow

| Step | Participant | Action | Description |
| --- | --- | --- | --- |
| 1 | GitHub Actions | Trigger the workflow | Triggered by schedule, manual execution, or pushes to the `publish` branch |
| 2 | GitHub Actions → image-syncer | Download and execute the sync tool | Enters the actual synchronization phase |
| 3 | image-syncer → Docker Hub | Fetch image manifests and tag list | Reads metadata from the source repository |
| 4 | image-syncer → Azure ACR | Fetch existing image data from the target repository | Determines the current state of the target side |
| 5 | image-syncer | Compare source and target differences | Identifies the image layers that need synchronization |
| 6 | image-syncer → Docker Hub | Pull changed image layers | Transfers only the content that needs updating |
| 7 | image-syncer → Azure ACR | Push changed image layers | Completes incremental synchronization |
| 8 | image-syncer → GitHub Actions | Return synchronization statistics | Includes results, differences, and error information |
| 9 | GitHub Actions | Record logs and upload artifacts | Supports later auditing and troubleshooting |

## GitHub Actions Workflow Implementation

Below is the actual workflow configuration in use (`.github/workflows/sync-docker-acr.yml`):

```yaml
name: Sync Docker Image to Azure ACR

on:
  schedule:
    - cron: "0 0 * * *" # Every day at 00:00 UTC
  workflow_dispatch: # Manual trigger
  push:
    branches: [publish]

permissions:
  contents: read

jobs:
  sync:
    runs-on: ubuntu-latest

    steps:
      - name: Checkout code
        uses: actions/checkout@v4

      - name: Download image-syncer
        run: |
          # Download the image-syncer binary
          wget https://github.com/AliyunContainerService/image-syncer/releases/download/v1.5.5/image-syncer-v1.5.5-linux-amd64.tar.gz
          tar -zxvf image-syncer-v1.5.5-linux-amd64.tar.gz
          chmod +x image-syncer

      - name: Create auth config
        run: |
          # Generate the authentication config file (YAML format)
          cat > auth.yaml <<EOF
          hagicode.azurecr.io:
            username: "${{ secrets.AZURE_ACR_USERNAME }}"
            password: "${{ secrets.AZURE_ACR_PASSWORD }}"
          EOF

      - name: Create images config
        run: |
          # Generate the image sync config file (YAML format)
          cat > images.yaml <<EOF
          docker.io/newbe36524/hagicode: hagicode.azurecr.io/hagicode
          EOF

      - name: Run image-syncer
        run: |
          # Run synchronization (using the newer --auth and --images parameters)
          ./image-syncer --auth=./auth.yaml --images=./images.yaml --proc=10 --retries=3

      - name: Upload logs
        if: always()
        uses: actions/upload-artifact@v4
        with:
          name: sync-logs
          path: image-syncer-*.log
          retention-days: 7
```

## Configuration Details

### 1. Trigger conditions
- Scheduled trigger: `cron: "0 0 * * *"` runs every day at 00:00 UTC
- Manual trigger: `workflow_dispatch` allows users to run it manually from the GitHub UI
- Push trigger: `push: branches: [publish]` triggers on pushes to the release branch for fast synchronization

### 2. Authentication configuration (`auth.yaml`)
```yaml
hagicode.azurecr.io:
  username: "${{ secrets.AZURE_ACR_USERNAME }}"
  password: "${{ secrets.AZURE_ACR_PASSWORD }}"
```

### 3. Image synchronization configuration
```yaml
docker.io/newbe36524/hagicode: hagicode.azurecr.io/hagicode
```
This configuration means that all tags from `docker.io/newbe36524/hagicode` will be synchronized to `hagicode.azurecr.io/hagicode`.

### 4. image-syncer parameters
- `--auth=./auth.yaml`: path to the authentication config file
- `--images=./images.yaml`: path to the image synchronization config file
- `--proc=10`: use `10` concurrent workers
- `--retries=3`: retry failures `3` times

## Notes and Best Practices

### 1. Security recommendations
- Rotate Azure ACR passwords regularly (recommended every 90 days)
- Use a dedicated ACR service account with push-only permissions
- Monitor ACR access logs to detect abnormal access in time
- Do not output credentials in logs
- Do not commit credentials to the code repository

### 2. Performance optimization
- Adjust the `--proc` parameter: tune concurrency based on network bandwidth (recommended `5-20`)
- Monitor synchronization duration: if synchronization takes too long, consider lowering concurrency
- Clean up logs regularly: set a reasonable `retention-days` value (`7` currently)

### 3. Troubleshooting

#### Issue 1: Authentication failure
```
Error: failed to authenticate to hagicode.azurecr.io
```
Solution:
1. Check whether GitHub Secrets are configured correctly
2. Verify whether the Azure ACR password has expired
3. Confirm that the ACR service account permissions are correct

#### Issue 2: Network timeout
```
Error: timeout waiting for response
```
Solution:
1. Check network connectivity
2. Reduce the number of concurrent workers (the `--proc` parameter)
3. Wait for the network to recover and rerun the workflow

#### Issue 3: Incomplete image synchronization
```
Warning: some tags failed to sync
```
Solution:
1. Check the synchronization logs to identify failed tags
2. Manually rerun the workflow to synchronize again
3. Verify that the source image on Docker Hub is healthy

### 4. Monitoring and alerting
- Regularly check the Actions page to confirm workflow status
- Enable GitHub notifications so workflow failures are reported promptly
- Monitor Azure ACR storage usage
- Regularly verify tag consistency across registries

## Frequently Asked Questions and Solutions

### Q1: How do I synchronize specific tags instead of all tags?

Modify the `images.yaml` configuration file:
```yaml
# Sync only the latest and v1.0 tags
docker.io/newbe36524/hagicode:latest: hagicode.azurecr.io/hagicode:latest
docker.io/newbe36524/hagicode:v1.0: hagicode.azurecr.io/hagicode:v1.0
```

### Q2: How do I synchronize multiple image repositories?

Add multiple lines to `images.yaml`:
```yaml
docker.io/newbe36524/hagicode: hagicode.azurecr.io/hagicode
docker.io/newbe36524/another-image: hagicode.azurecr.io/another-image
```

### Q3: How do I retry after synchronization fails?
- Automatic retry: image-syncer includes a built-in retry mechanism (`3` retries by default)
- Manual retry: click `Re-run all jobs` on the GitHub Actions page

### Q4: How can I view detailed synchronization progress?
- View real-time logs on the Actions page
- Download the `sync-logs` artifact to inspect the full log files
- The log files include synchronization status and transfer speed for each tag

### Q5: How long does synchronization take?
- Initial full synchronization: usually `10-30` minutes depending on image size
- Incremental synchronization: usually `2-5` minutes if image changes are small
- Timing depends on network bandwidth, image size, and concurrency settings
