## HagiCode's Multi-Agent Architecture Design

### From "Solo Fighting" to "Team Collaboration"

In the early stages of the HagiCode project, we also tried using only one AI Agent to handle all tasks. We soon discovered that this approach had obvious bottlenecks: different tasks require different capability emphases—some tasks need stronger context understanding, while others require more precise code modification abilities. It's difficult for one Agent to excel in all aspects.

This made us realize that we must enable multiple Agents to work collaboratively. But the question is: how to make AI products from different companies coexist peacefully in the same project? We needed to solve several core problems:

1. **Configuration Management Complexity**: Each Agent has different configuration methods, API interfaces, and execution modes
2. **Unified Communication Protocol**: Need a standardized way for data exchange between different Agents
3. **Task Division and Coordination**: How to reasonably assign tasks so each Agent can leverage its strengths

With these problems in mind, we began designing HagiCode's multi-agent architecture. Actually, it's not that complicated—we just figured it out.

### Overall Architecture Overview

After multiple iterations, our final architecture is this:

```
┌─────────────────────────────────────────────────────────────────┐
│                    AIProviderFactory                             │
│  (Factory pattern uniformly manages all AI Providers)            │
├─────────────────────────────────────────────────────────────────┤
│  ClaudeCodeCli  │  CodexCli  │  CodebuddyCli  │  IFlowCli    │
│  (Anthropic)   │  (OpenAI)  │  (Zhipu GLM)     │  (Zhipu)      │
└─────────────────────────────────────────────────────────────────┘
```

The core idea is: through a unified Provider interface, different AI Agents can be managed by the same set of code. Meanwhile, using the factory pattern to dynamically create and configure these Providers ensures system extensibility and flexibility.

It's like the division of labor in life—everyone has their own role, only here we've turned this division of labor into code architecture.

### Agent Types and Responsibility Division

Based on HagiCode project's actual usage experience, we assigned different responsibilities to each Agent:

| Agent | Provider | Model | Primary Use |
|-------|--------|------|---------|
| ClaudeCodeCli | Anthropic | glm-5-turbo | Generate technical solutions and proposals |
| CodexCli | OpenAI/Zed | gpt-5.4 | Execute precise code modifications |
| CodebuddyCli | Zhipu | glm-4.7 | Optimize proposal descriptions and documentation |
| IFlowCli | Zhipu | glm-4.7 | Archive proposals and historical records |
| OpenCodeCli | - | - | General code editing |
| GitHubCopilot | Microsoft | - | Assisted programming and code completion |

The logic behind this division is: each Agent has its area of expertise. Claude Code excels at understanding and analyzing complex requirements, so it's responsible for early-stage solution design; Codex is more precise in code modification, suitable for handling specific implementation tasks; CodeBuddy is cost-effective, making it perfect for optimizing documentation.

After all, what suits you best is best—all roads lead to Rome, it's just that some roads are easier to travel, while others are a bit more winding.

## Core Configuration Mechanism

### Unified Provider Interface Design

To enable different AI Agents to be managed uniformly, we first need to define a unified interface. In HagiCode, we defined this interface:

```csharp
public interface IAIProvider
{
    // Unified Provider interface
    Task<IAIProvider?> GetProviderAsync(AIProviderType providerType);
    Task<IAIProvider?> GetProviderAsync(string providerName, CancellationToken cancellationToken);
}
```

This interface looks simple, but it's the cornerstone of the entire multi-agent system. Through the unified interface, we can ignore which company's AI product is at the bottom and invoke them all in the same way.

Actually, this just simplifies complex matters—after all, simplicity is beauty.

### Provider Factory Pattern Implementation

With a unified interface, the next step is how to create these Provider instances. HagiCode uses the factory pattern:

```csharp
private IAIProvider? CreateProvider(AIProviderType providerType, ProviderConfiguration config)
{
    return providerType switch
    {
        AIProviderType.ClaudeCodeCli =>
            ActivatorUtilities.CreateInstance<ClaudeCodeCliProvider>(_serviceProvider, Options.Create(config)),
        AIProviderType.CodebuddyCli =>
            ActivatorUtilities.CreateInstance<CodebuddyCliProvider>(_serviceProvider, Options.Create(config)),
        AIProviderType.CodexCli =>
            ActivatorUtilities.CreateInstance<CodexCliProvider>(_serviceProvider, Options.Create(config)),
        AIProviderType.IFlowCli =>
            ActivatorUtilities.CreateInstance<IFlowCliProvider>(_serviceProvider, Options.Create(config)),
        _ => null
    };
}
```

Here we use dependency injection's `ActivatorUtilities.CreateInstance`, which can dynamically create Provider instances at runtime and automatically inject dependencies. The benefit of this design is: when adding a new Agent type, you only need to add the corresponding Provider class and add a case branch in the factory method, without any need to modify existing code.

That's fine—after all, who wants to modify a bunch of old code every time they add a new feature?

### Dynamic Configuration Resolution

To make configuration more flexible, we also implemented a type mapping mechanism:

```csharp
public static AIProviderTypeExtensions
{
    private static readonly Dictionary<string, AIProviderType> _typeMap = new(
        StringComparer.OrdinalIgnoreCase)
    {
        ["ClaudeCodeCli"] = AIProviderType.ClaudeCodeCli,
        ["CodebuddyCli"] = AIProviderType.CodebuddyCli,
        ["CodexCli"] = AIProviderType.CodexCli,
        ["IFlowCli"] = AIProviderType.IFlowCli,
        // ...more type mappings
    };
}
```

The purpose of this mapping table is to convert string-form Provider names to enum types. This way, configuration files can use intuitive string names, while code internally uses type-safe enums for processing.

After all, with configuration, the more intuitive the better—who wants to remember a bunch of complex code?

In actual use, you only need to configure in `appsettings.json`:

Each Provider can be independently configured with parameters such as enable/disable switch, model version, working directory, etc. This design ensures both flexibility and ease of management and maintenance.

## Adventure Party Task Flow

### The Art of Task Division

With a unified technical architecture, the next step is how to enable multiple Agents to work collaboratively. HagiCode designed a task flow mechanism that lets different Agents handle tasks at different stages:

```
Proposal Creation (User)
    │
    ▼
[Claude Code] ──Generate Proposal──▶ Proposal Document
    │                        │
    │                        ▼
    │               [Codebuddy] ──Optimize Description──▶ Optimized Proposal
    │                        │
    │                        ▼
    │               [Codex] ──Execute Modifications──▶ Code Changes
    │                        │
    │                        ▼
    └───────────────▶ [iFlow] ──Archive──▶ Historical Records
```

The benefit of this division is: each Agent only needs to focus on tasks they're good at, without needing to "know everything." Claude Code is responsible for generating proposals from scratch, Codebuddy is responsible for making proposal descriptions clearer, Codex is responsible for turning proposals into actual code changes, and iFlow is responsible for archiving and saving these changes.

Actually, it's like teamwork in life—everyone has their own role, and only together can they accomplish something big. It's just that the team members here are AIs.

### Key Practice Points

In actual operation, we summarized the following experience:

**1. Agent Selection Strategy is Important**

Don't just assign tasks randomly—assign them based on each Agent's strengths:
- Proposal Generation: Use Claude Code because it has stronger context understanding capabilities
- Code Execution: Use Codex because it's more precise in code modification
- Proposal Optimization: Use CodeBuddy because of its high cost-performance ratio
- Archive Storage: Use iFlow because it's stable and reliable

After all, having the right person do the right thing is an eternal truth.

**2. Configuration Isolation Ensures Stability**

Each Agent's configuration is managed independently, supports environment variable overrides, and working directories are also independent. This way, one Agent's configuration error won't affect other Agents.

It's like boundaries in life—everyone has their own space, and not interfering with each other is the only way to coexist harmoniously.

**3. Error Handling Mechanism**

Single Agent failure shouldn't affect the overall process. We implemented degradation strategies: when a certain Agent fails to execute, the system can automatically switch to a backup solution, or directly skip that step and continue executing subsequent tasks. At the same time, complete logging facilitates post-mortem troubleshooting.

After all, no one can guarantee they'll never make mistakes—the key is how to handle errors. It's like life—you'll always encounter setbacks, what matters is how to get out of them.

**4. Monitoring and Observability**

Through the ACP protocol (our custom communication protocol based on JSON-RPC 2.0), we can track each Agent's execution status. Session isolation ensures concurrency safety, and dynamic caching optimizes performance.

After all, things you can't see are most prone to problems—having some monitoring is better than being completely in the dark.
