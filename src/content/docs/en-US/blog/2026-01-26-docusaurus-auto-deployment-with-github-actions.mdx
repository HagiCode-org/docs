## Adding GitHub Pages Auto-Deployment Support for HagiCode

> This project was originally codenamed PCode and has now been officially renamed HagiCode. This article documents how to introduce automated static site deployment capabilities to the project, making content publishing as easy as drinking water.

## Background/Introduction

During the development of HagiCode, we encountered a very practical problem: as documentation and proposals increased, how to efficiently manage and display this content became urgent. We decided to use GitHub Pages to host our static site, but manual building and deployment were too cumbersome—every change required local building, packaging, and then manual pushing to the `gh-pages` branch. This was not only inefficient but also error-prone.

To solve this problem (mainly to be lazy), we needed an automated deployment workflow. This article will detail how to add GitHub Actions auto-deployment support for the HagiCode project, allowing us to focus solely on content creation while leaving the rest to the automation process.

## Goal Analysis

Before diving in, we need to clarify what this task actually entails. After all, sharpening the axe before cutting trees doesn't delay the work.

### Core Requirements

1.  **Automated Build**: Automatically trigger the build process when code is pushed to the `main` branch.
2.  **Automated Deployment**: Automatically deploy the generated static files to GitHub Pages after successful build.
3.  **Environment Consistency**: Ensure the CI environment matches the local build environment to avoid the awkward "works locally but fails online" situation.

### Technology Selection

Considering HagiCode is built with Docusaurus (a very popular React static site generator), we can leverage GitHub Actions to achieve this goal.

## Configuring GitHub Actions Workflow

GitHub Actions is the CI/CD service provided by GitHub. By defining YAML format workflow files in the code repository, we can customize various automated tasks.

### Creating the Workflow File

We need to create a new configuration file in the `.github/workflows` folder in the project root directory, such as `deploy.yml`. If the folder doesn't exist, remember to create it manually first.

The core logic of this configuration file is as follows:

1.  **Trigger Conditions**: Listen for `push` events on the `main` branch.
2.  **Runtime Environment**: Latest version of Ubuntu.
3.  **Build Steps**:
    -   Checkout code
    -   Install Node.js
    -   Install dependencies (`npm install`)
    -   Build static files (`npm run build`)
4.  **Deployment Steps**: Use the official `action-gh-pages` to push build artifacts to the `gh-pages` branch.

### Key Configuration Code

Below is the configuration template we ultimately adopted:

```yaml
name: Deploy to GitHub Pages

# Trigger condition: when pushed to main branch
on:
  push:
    branches:
      - main
    # You can add path filtering as needed, e.g., only build when docs change
    # paths:
    #   - 'docs/**'
    #   - 'package.json'

# Set permissions, which is important for deploying to GitHub Pages
permissions:
  contents: read
  pages: write
  id-token: write

# Concurrency control: cancel old builds on the same branch
concurrency:
  group: "pages"
  cancel-in-progress: false

jobs:
  build:
    runs-on: ubuntu-latest
    steps:
      - name: Checkout
        uses: actions/checkout@v4
        # Note: Must set fetch-depth: 0, otherwise build version numbers may be inaccurate
        with:
          fetch-depth: 0

      - name: Setup Node
        uses: actions/setup-node@v4
        with:
          node-version: 20 # Recommended to match local development environment
          cache: 'npm'     # Enable caching to speed up build process

      - name: Install dependencies
        run: npm ci
        # Use npm ci instead of npm install, as it's faster and stricter, suitable for CI environment

      - name: Build website
        run: npm run build
        env:
          # If your site build requires environment variables, configure them here
          # NODE_ENV: production
          # PUBLIC_URL: /your-repo-name
          
      - name: Upload artifact
        uses: actions/upload-pages-artifact@v3
        with:
          path: ./build # Docusaurus default output directory

  deploy:
    environment:
      name: github-pages
      url: ${{ steps.deployment.outputs.page_url }}
    runs-on: ubuntu-latest
    needs: build
    steps:
      - name: Deploy to GitHub Pages
        id: deployment
        uses: actions/deploy-pages@v4
```

## Pitfalls During Implementation

In actual operation, we encountered some issues. Here we share them in hopes that you can avoid them (or prepare solutions in advance).

### 1. GitHub Token Permission Issues

When first configuring, deployment always failed with a 403 (Forbidden) error. After investigating for a long time, we discovered that GitHub's default `GITHUB_TOKEN` doesn't have write permissions for Pages by default.

**Solution**: In the repository's `Settings` -> `Actions` -> `General` -> `Workflow permissions`, make sure to select **"Read and write permissions"**.

### 2. Build Directory Path Errors

Docusaurus places built static files in the `build` directory by default. However, some projects (e.g., Create React App defaults to `build`, Vite defaults to `dist`) may have different configurations. If you get file not found errors in Actions, remember to check the output path configuration in `docusaurus.config.js`.

### 3. Subpath Issues

If your repository is not a user homepage (i.e., not `username.github.io`) but a project homepage (e.g., `username.github.io/project-name`), you need to configure `baseUrl`.

In `docusaurus.config.js`:

```javascript
module.exports = {
  // ...
  url: 'https://hagicode.com', // Your Hagicode URL
  baseUrl: '/', // Root path deployment
  // ...
};
```
This is easily overlooked. Incorrect configuration will result in all pages showing a white screen because resource paths cannot be loaded.

## Verifying Results

After configuring everything and pushing the code, we can go to the **Actions** tab of the GitHub repository to watch the show.

You'll see a yellow circle (workflow running), and when it turns green, it means success! If it turns red, click in to check the logs. Usually, you can troubleshoot the problem (most of the time it's a spelling error or incorrect path configuration).

After a successful build, visit `https://<your-username>.github.io/<repository-name>/` to see your brand new site.
