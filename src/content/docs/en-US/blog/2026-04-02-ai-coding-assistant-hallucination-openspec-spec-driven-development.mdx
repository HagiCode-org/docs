## AI Coding Assistant Hallucination: How to Use OpenSpec for Spec-Driven Development

> AI coding assistants are powerful, but they often generate code that doesn't meet actual requirements or violates project standards. This article shares how the HagiCode project implements "spec-driven development" through the OpenSpec process, significantly reducing AI hallucination risks with a structured proposal mechanism.

## Background

Anyone who has used GitHub Copilot or ChatGPT to write code has likely experienced this: the AI-generated code looks beautiful, but it's full of problems when you actually use it. It might use a component from your project incorrectly, ignore your team's coding standards, or write long segments of logic based on non-existent assumptions.

This is the so-called "AI hallucination" problem — in the programming domain, it manifests as generating code that seems reasonable but doesn't actually match the project's reality.

Actually, this is quite a helpless situation. AI coding assistants are becoming more widespread, and this problem is getting worse. After all, AI lacks understanding of project history, architectural decisions, and coding standards, while excessive freedom easily leads it to "creatively" generate code that doesn't match reality. This is quite similar to writing articles — without structure, it's easy to write wild and fanciful content, but that's not actually how things are.

To address these pain points, we made a bold decision: instead of trying to make AI smarter, we put it in a "specification" cage. The changes brought by this decision might be bigger than you imagine — I'll get into the specifics shortly.

## About HagiCode

The solution shared in this article comes from our practical experience in the [HagiCode](https://hagicode.com) project. HagiCode is an open-source AI code assistant project dedicated to solving practical problems in AI programming through structured engineering practices.

## Root Causes of AI Hallucination

Before diving into the solution, let's first look at where the problem actually lies. After all, know yourself and know your enemy, and you'll never be defeated in a hundred battles — though applying this to AI seems interesting too.

### Lack of Context

AI models are trained on public code repositories, but your project has its own history, conventions, and architectural decisions. This "tacit knowledge" cannot be directly accessed by AI, so the generated code often disconnects from the project's actual situation.

You can't entirely blame AI for this — after all, it hasn't worked on your project, so how would it know your unwritten rules? It's like a new intern who doesn't understand the rules — that's normal, but the cost might be a bit high.

### Excessive Freedom

When you ask AI "help me implement a user authentication feature," it might generate code in any form. Without clear constraints, AI will implement it in the way it "thinks" is reasonable, not according to your project's requirements.

This is like letting someone who has never learned your project's standards freely express themselves — can problems be avoided? Actually, it's not that it's irresponsible, it just doesn't know what responsibility is.

### Lack of Validation

After AI generates code, if there's no structured review process, code based on wrong assumptions will enter the codebase directly. By the time problems are discovered in testing or even production, the cost is too high.

This is no different from mending the pen after the sheep are lost — but when all the sheep have run away, what's the use of mending the pen? Everyone understands this principle, but when it comes to actually doing it, it always feels troublesome. After all, before things go bad, who wants to spend extra time?

## OpenSpec: The Answer to Spec-Driven Development

HagiCode chose OpenSpec as the solution, with the core idea being: **all code changes must go through a structured proposal process**, transforming abstract ideas into executable implementation plans.

This sounds quite grand, but it's essentially just having AI write the requirements document before writing the code. After all, preparation leads to success, lack of preparation leads to failure — the ancients didn't deceive me.

### What is OpenSpec

OpenSpec is an npm-based command-line tool (`@fission-ai/openspec`) that defines a standard proposal file structure and validation mechanism. Simply put, it's about having AI "write the requirements document" before writing code.

### Three-Step Process to Prevent Hallucination

OpenSpec ensures proposal quality through a three-step process:

**Step 1: Initialize proposal** - Set session state to Openspecing
**Step 2: Intermediate processing** - Maintain Openspecing state, gradually refine artifacts
**Step 3: Complete proposal** - Transition to Reviewing state

This design has a clever detail: the first step uses the `ProposalGenerationStart` type, and completion doesn't trigger a state transition. This ensures the entire multi-step process is complete before prematurely entering the review phase.

Actually, this detail is quite interesting — it's like cooking. If you uncover the pot before the cooking time is reached, you definitely won't produce a good dish. Only by being patient and taking it step by step can you finally create a good dish.

```csharp
// Implementation in HagiCode project
public enum MessageAssociationType
{
    ProposalGeneration = 2,
    ProposalExecution = 3,

    /// <summary>
    /// Marks the start of the three-step proposal generation process
    /// Does not transition to Reviewing state after completion
    /// </summary>
    ProposalGenerationStart = 5
}
```

### Standardized File Structure

Each OpenSpec proposal follows the same directory structure:

```
openspec/
├── changes/                    # Active and archived changes
│   ├── {change-name}/
│   │   ├── proposal.md        # Proposal description
│   │   ├── design.md          # Design document
│   │   ├── specs/             # Technical specifications
│   │   └── tasks.md           # Executable task list
│   └── archive/               # Archived changes
└── specs/                     # Independent specification library
```

According to HagiCode project statistics, there are currently 4000+ archived changes and 150,000+ lines of specification files. This historical accumulation not only gives AI rules to follow but also provides a valuable knowledge base for the team.

This is like the classics left by the ancients — read enough and you'll naturally gain some insight. It's just that these classics aren't written on bamboo slips, but stored in files.

### Multi-Layer Validation Mechanism

The system implements multi-layer validation to ensure proposal quality:

```csharp
// Validate required files exist
ValidateProposalFiles()

// Validate execution prerequisites
ValidateExecuteAsync()

// Validate startup conditions
ValidateStartAsync()

// Validate archive conditions
ValidateArchiveAsync()

// Proposal name format validation (kebab-case)
ValidateNameFormat()
```

These validations are like gatekeepers with layers of checks — only truly qualified proposals can pass. Although it looks cumbersome, it's better than letting bad code enter the codebase, right?

### Prompt Template Constraints

AI execution in HagiCode uses predefined Handlebars templates that contain clear step-by-step guidance and safeguards. For example:

- Prohibit continuing without understanding user intent
- Prohibit generating unverified code
- Require re-providing when names are invalid
- If change already exists, suggest using continue command instead of recreating

This approach of "dancing with shackles" actually makes AI more focused on understanding requirements and generating code that conforms to specifications. Actually, constraints aren't necessarily bad — after all, too much freedom easily leads to chaos.

## Practice: How to Use OpenSpec in Your Project

### Installation and Initialization

```bash
npm install -g @fission-ai/openspec@1
openspec --version  # Verify installation
```

The `openspec/` folder structure will be automatically created in the project root directory.

There's not much to say about this step — installing tools, everyone understands. Just remember to use the `@fission-ai/openspec@1` version, as newer versions might have pitfalls — after all, stability comes first.

### Creating a Proposal

In HagiCode's conversation interface, use the shortcut command:

```
/opsx:new
```

Or specify the change name and target repository:

```
/opsx:new "add-user-auth" --repos "repos/web"
```

Creating a proposal is like outlining an article — once you have the outline, the rest is easy to write. It's just that many people like to start writing directly, only to find halfway through that their thinking doesn't flow — that's truly a headache.

### Generating Artifacts

Use `/opsx:continue` to gradually generate the required artifacts:

**proposal.md** - Describes the purpose and scope of the change
```markdown
# Proposal: Add User Authentication

## Why
The current system lacks user authentication functionality and cannot protect sensitive APIs.

## What Changes
- Add JWT authentication middleware
- Implement login/registration API
- Update frontend integration
```

**design.md** - Detailed technical design
```markdown
# Design: Add User Authentication

## Context
Currently using public APIs, accessible to anyone...

## Decisions
1. Choose JWT over Session...
2. Use HS256 algorithm...

## Risks
- Token leakage risk...
- Mitigation measures...
```

**specs/** - Technical specifications and test scenarios
```markdown
# user-auth Specification

## Requirements

### Requirement: JWT Token Generation
The system SHALL generate JWT tokens using the HS256 algorithm.

#### Scenario: Valid login
- WHEN user provides valid credentials
- THEN the system SHALL return a valid JWT token
```

**tasks.md** - Executable task list
```markdown
# Tasks: Add User Authentication

## 1. Backend Changes
- [ ] 1.1 Create AuthController
- [ ] 1.2 Implement JWT middleware
- [ ] 1.3 Add unit tests
```

These artifacts are actually like drafts for writing an article — when the draft is well-written, the main text naturally flows smoothly. It's just that many people don't like writing drafts, thinking it's a waste of time, but actually drafts are the best place to clarify your thinking.

### Review and Application

After completing all artifacts:

```
/opsx:apply
```

AI will read all context files and gradually execute tasks according to the list in tasks.md. At this point, because there are clear specifications, the quality of generated code will be much higher.

Actually, at this point, things are already half done. With a clear task list, the rest is just methodical execution. It's just that many people skip the previous steps and go directly here, so quality is naturally hard to guarantee.

### Archiving

After completing the change:

```
/opsx:archive
```

Move completed changes to the `archive/` directory for future reference and reuse.

Archiving is quite important — it's like properly storing a finished article. When you encounter similar problems in the future, looking through past records might give you answers. It's just that many people are lazy about it, thinking it's troublesome, but actually these accumulations are the most precious wealth.

## Notes and Best Practices

### Proposal Naming Convention

Use kebab-case format, starting with a letter, containing only lowercase letters, numbers, and hyphens:
- ✅ `add-user-auth`
- ❌ `AddUserAuth`
- ❌ `add--user-auth`

Naming conventions — there's nothing major to say about them, but being consistent is generally good. After all, in coding, consistency is important, though many people don't pay attention to it.

### Avoiding Common Errors

1. **Using the wrong type in step 1 of the three-step process** - Will transition state prematurely
2. **Forgetting to trigger state transition in the final step** - Will get stuck in Openspecing state
3. **Skipping review and executing directly** - Should first verify all artifacts are complete

These errors are easily made by beginners — experienced hands naturally know how to avoid them. It's just that beginners will eventually become experienced, and taking some detours is fine, I just hope they don't take too many.

### Managing Multiple Changes

OpenSpec supports managing multiple proposals simultaneously, which is particularly useful when handling large features:

```bash
# View all active changes
openspec list

# Switch to a specific change
openspec apply "add-user-auth"

# View change status
openspec status --change "add-user-auth"
```

Managing multiple changes is like writing several articles simultaneously — it requires some skill and patience. But you'll get used to it — after all, humans can always adapt.

### Understanding the Session State Machine

Understanding state transitions helps with troubleshooting:

```
Init → Drafting → Openspecing → Reviewing → Executing → ExecutionCompleted → Completed → Archived
```

- **Openspecing**: Planning in progress
- **Reviewing**: Under review (artifacts can be repeatedly modified)
- **Executing**: Executing (applying tasks.md)

State machines — simply put, it's a set of rules. Rules can be annoying sometimes, but more often they're useful. After all, nothing can be accomplished without norms — the ancients said this long ago.

## Summary

Through the OpenSpec process, the HagiCode project has achieved significant results in addressing AI hallucination:

1. **Reduce hallucination** - AI must follow structured specifications and cannot arbitrarily generate code
2. **Improve quality** - Multi-layer validation ensures changes meet project standards
3. **Accelerate collaboration** - Archived changes provide reference for subsequent development
4. **Traceability** - Each change has complete proposal, design, specification, and task records

This solution doesn't make AI smarter, but puts it in a "specification" cage. Practice has shown that dancing with shackles actually leads to better dancing.

Actually, this principle is simple — constraints aren't necessarily bad. It's like writing articles — with format constraints, it's easier to write good content. It's just that many people don't like constraints, thinking they limit their creativity, but actually creativity also needs soil to bloom and bear fruit.

If you're also using AI coding assistants and have encountered similar problems, why not try OpenSpec? Spec-driven development might seem like adding some steps, but this upfront investment will pay back many times over in code quality and maintenance efficiency.
