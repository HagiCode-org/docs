## Building a Cross-Project Knowledge Base for the AI Era with Vault System

> Project-based learning is becoming mainstream, but scattered learning materials and fragmented context prevent AI assistants from delivering maximum value. This article introduces HagiCode's Vault system design—through a unified storage abstraction layer, AI assistants can understand and access all learning resources, achieving true cross-project knowledge reuse.

## Background

Actually, in the AI era, our approach to learning new technologies is quietly changing. Traditional reading and watching videos are certainly important, but "project-based learning"—deeply studying and learning from excellent open-source projects' code, architecture, and design patterns—is indeed becoming more efficient. Directly running and modifying high-quality open-source projects lets you understand real-world engineering practices fastest.

However, this approach also brings new challenges.

**Learning materials are too scattered.** Notes might be in Obsidian, code repositories scattered across various folders, and AI assistant conversation history is yet another isolated data silo. Every time you need AI to help analyze a project, you have to manually copy code snippets and organize context, which is quite tedious.

**Context frequently breaks.** AI assistants cannot directly access local learning resources, so background information needs to be provided anew for each conversation. Code repositories being studied update quickly, and manual synchronization is error-prone. Even worse, knowledge is difficult to share between multiple learning projects—design patterns learned in project A are completely unknown to AI when processing project B.

The essence of these problems is "data silos." If there were a unified storage abstraction layer allowing AI assistants to understand and access all learning resources, the problem would be readily solved.

To address these pain points, we made a key design decision during HagiCode development: build a Vault system as a unified knowledge storage abstraction layer. The impact of this decision might be greater than imagined—more on this shortly.

## About HagiCode

The solution shared in this article comes from practical experience in the [HagiCode](https://hagicode.com) project. HagiCode is an AI code assistant based on the OpenSpec workflow, with the core philosophy of making AI not just "talk" but "do"—directly manipulating code repositories, executing commands, and running tests. GitHub: [github.com/HagiCode-org/site](https://github.com/HagiCode-org/site)

During development, we discovered that AI assistants need frequent access to users' various learning resources: code repositories, notes, configuration files, etc. If users had to manually provide these each time, the experience would be terrible. This prompted the Vault system design.

## Core Design

### Multi-Type Support

HagiCode's Vault system supports four types, each corresponding to different usage scenarios:

| Type | Purpose | Typical Scenarios |
|------|---------|-------------------|
| `folder` | General folder type | Temporary learning materials, drafts |
| `coderef` | Specialized for studying code projects | Systematic learning of an open-source project |
| `obsidian` | Integration with Obsidian note-taking software | Reuse of existing note libraries |
| `system-managed` | Automatic system management | Project configuration, prompt templates, etc. |

Among these, the `coderef` type is most commonly used in HagiCode, providing a standardized directory structure and AI-readable metadata descriptions for studying code projects. Why specifically design this type? Because studying an open-source project isn't simply "downloading code"—it requires managing the code itself, learning notes, configuration files, and other content simultaneously. `coderef` standardizes all of this.

### Persistent Storage Mechanism

Vault's registry is persistently stored to the file system in JSON format:

```csharp
_registryFilePath = Path.Combine(absoluteDataDir, "personal-data", "vaults", "registry.json");
```

This design seems simple but is actually well-considered:

**Simple and reliable.** JSON format is human-readable, facilitating debugging and manual modification. When system issues arise, you can directly open the file to check status and even manually fix it—particularly useful during development.

**Reduced dependencies.** File system storage avoids database complexity. No additional installation and configuration of database services are needed, reducing system complexity and maintenance costs.

**Concurrency safety.** Uses `SemaphoreSlim` to ensure multi-thread safety. In AI code assistant scenarios, multiple operations might access the vault registry simultaneously, requiring proper concurrency control.

### AI Context Integration

The system's core capability lies in automatically injecting vault information into AI proposal context:

```typescript
export function buildTargetVaultsText(
  vaults: VaultForText[],
  template: VaultPromptTemplate = DEFAULT_VAULT_PROMPT_TEMPLATE,
): string {
  const readOnlyVaults = vaults.filter((vault) => vault.accessType === 'read');
  const editableVaults = vaults.filter((vault) => vault.accessType === 'write');

  const sections = [
    buildVaultSection(readOnlyVaults, template.reference),
    buildVaultSection(editableVaults, template.editable),
  ].filter(Boolean);

  return `\n\n### ${template.heading}\n\n${sections.join('\n')}`;
}
```

This way, AI assistants can automatically understand available learning resources without users manually providing context each time. This design makes HagiCode's experience particularly natural—tell AI "help me analyze React's concurrent rendering," and AI can automatically find the previously registered React learning vault instead of repeatedly pasting code.

### Access Control Mechanism

The system divides vaults into two access types:

- **reference (read-only)**: AI only uses for analysis and understanding, cannot modify content
- **editable (writable)**: AI can modify content based on task requirements

This distinction lets AI know which content is "read-only reference" and which is "can be modified," avoiding accidental operation risks. For example, if you register an open-source project vault as learning material, you certainly don't want AI arbitrarily modifying code inside—mark it as `reference`. But if it's your own project vault, you can mark it as `editable` to let AI help modify code.

## Practice Guide

### CodeRef Vault Standardized Structure

For coderef-type vaults, the system provides a standardized directory structure:

```
my-coderef-vault/
├── index.yaml          # vault metadata description
├── AGENTS.md           # AI assistant operation guide
├── docs/               # store learning notes and documentation
└── repos/              # manage studied code repositories via Git submodules
```

What's the design philosophy behind this structure?

**docs/** stores learning notes, recording code understanding, architecture analysis, and pitfalls encountered in Markdown format. These notes aren't just for yourself—AI can also understand them and automatically reference them when handling related tasks.

**repos/** manages studied repositories via Git submodules rather than directly copying code. This has two benefits: first, maintaining synchronization with upstream—one `git submodule update` gets the latest code; second, saving space, as multiple vaults can reference different versions of the same repository.

**index.yaml** contains vault metadata, letting AI assistants quickly understand purpose and content. It's like writing a "self-introduction" for the vault—AI knows what it's for when first encountering it.

**AGENTS.md** is a guide specifically written for AI assistants, explaining how to handle vault content. You can tell AI here: "focus on performance optimization related code when analyzing this project" or "don't modify test files."

### Creating and Using Vaults

Creating a CodeRef vault is simple:

```typescript
const createCodeRefVault = async () => {
  const response = await VaultService.postApiVaults({
    requestBody: {
      name: "React Learning Vault",
      type: "coderef",
      physicalPath: "/Users/developer/vaults/react-learning",
      gitUrl: "https://github.com/facebook/react.git"
    }
  });

  // System will automatically:
  // 1. Clone React repository to vault/repos/react
  // 2. Create docs/ directory for notes
  // 3. Generate index.yaml metadata
  // 4. Create AGENTS.md guide file

  return response;
};
```

Then reference this vault in an AI proposal:

```typescript
const proposal = composeProposalChiefComplaint({
  chiefComplaint: "Help me analyze React's concurrent rendering mechanism",
  repositories: [
    { id: "react", gitUrl: "https://github.com/facebook/react.git" }
  ],
  vaults: [
    {
      id: "react-learning",
      name: "React Learning Vault",
      type: "coderef",
      physicalPath: "/vaults/react-learning",
      accessType: "read"  // AI can only read, not modify
    }
  ],
  quickRequestText: "Focus on fiber architecture and scheduler implementation"
});
```

### Typical Usage Scenarios

**Scenario 1: Systematic Open-Source Project Learning**

Create a CodeRef vault, manage target repositories via Git submodules, and record learning notes in the `docs/` directory. AI can access both code and notes simultaneously, providing more precise analysis. Notes written while learning a module are automatically referenced by AI in subsequent related code analysis—like having an "assistant" remember previous thinking.

**Scenario 2: Reusing Obsidian Note Libraries**

If you're already using Obsidian for note management, simply register your existing vault to HagiCode. AI can directly access your knowledge base without manual copy-pasting. This feature is particularly practical—many people have note libraries accumulated over years, and after integration, AI can "read" and understand the knowledge system.

**Scenario 3: Cross-Project Knowledge Reuse**

Multiple AI proposals can reference the same vault, achieving cross-project knowledge reuse. For example, create a "design pattern learning vault" containing notes and code examples for various design patterns. Whatever project you're analyzing, AI can reference content from this vault—no need to accumulate knowledge repeatedly.

### Path Safety Mechanism

The system strictly validates paths to prevent path traversal attacks:

```csharp
private static string ResolveFilePath(string vaultRoot, string relativePath)
{
    var rootPath = EnsureTrailingSeparator(Path.GetFullPath(vaultRoot));
    var combinedPath = Path.GetFullPath(Path.Combine(rootPath, relativePath));
    if (!combinedPath.StartsWith(rootPath, StringComparison.OrdinalIgnoreCase))
    {
        throw new BusinessException(VaultRelativePathTraversalCode,
            "Vault file paths must stay inside the registered vault root.");
    }
    return combinedPath;
}
```

This ensures all file operations stay within the vault's root directory scope, preventing malicious path access. Security can't be careless—when AI assistants operate on the file system, boundaries must be clearly defined.

## Considerations

When using the HagiCode Vault system, several points need special attention:

1. **Path safety**: Ensure custom paths are within allowed ranges, otherwise the system will refuse operations. This prevents accidental operations and potential security risks.

2. **Git submodule management**: CodeRef vaults recommend using Git submodules rather than directly copying code. The benefits were mentioned earlier—maintaining synchronization, saving space. However, submodules have their own usage patterns and may require familiarization when first used.

3. **File preview limitations**: The system limits file size (256KB) and count (500 files), oversized files need batch processing. This limitation is for performance considerations—if you encounter oversized files, you can manually split them or use other processing methods.

4. **Diagnostic information**: Creating a vault returns diagnostic information usable for debugging when failures occur. When encountering problems, check diagnostic information first—in most cases, you can find clues.

## Summary

HagiCode's Vault system essentially addresses a simple but profound problem: how to enable AI assistants to understand and use local knowledge resources.

Through a unified storage abstraction layer, standardized directory structure, and automated context injection, it achieves a "register once, reuse everywhere" knowledge management approach. After creating a vault, whether it's learning notes, code repositories, or documentation materials, AI can automatically access and understand them.

The experience improvement from this design is evident. No more manually copying code snippets or repeatedly explaining background information—AI assistants are like colleagues who truly understand project situations, able to provide more valuable help based on existing knowledge.

The Vault system shared in this article is a solution that emerged from actual pitfalls and optimizations during HagiCode development. If you find this design valuable, it indicates solid engineering capability—then HagiCode itself is also worth attention.
