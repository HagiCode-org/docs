## Hagicode.Libs: Engineering Practice for Unified Integration of Multiple AI Programming Assistant CLIs

> During the development of the HagiCode project, we needed to integrate multiple AI programming assistant CLIs such as Claude Code, Codex, and CodeBuddy simultaneously. Each CLI has different interfaces, parameters, and output formats, and the repetitive integration code made the project increasingly difficult to maintain. This article shares how we built a unified abstraction layer through the HagiCode.Libs library to solve this engineering pain point—it's also a record of the pitfalls we've encountered and the experience we've accumulated along the way.

## Background

The current AI programming assistant market is quite lively. Besides Claude Code, there's OpenAI's Codex, Zhipu's CodeBuddy, and more. As an AI code assistant project, HagiCode needs to integrate these different CLI tools across multiple subprojects (desktop, backend, web, etc.).

At first, the problem wasn't too big—integrating one CLI was just a few hundred lines of code. But as we needed to support more and more CLIs, things started getting troublesome—

Each CLI has different command-line parameter formats, different requirements for environment variables, and even more diverse output formats—some output JSON, some stream JSON, and some just output plain text. Add to that cross-platform compatibility issues: Windows and Unix systems have completely different executable discovery and process management mechanisms, and code repetition kept increasing. Actually, it's just a matter of more Ctrl+C and Ctrl+V, but maintenance becomes a real headache.

The biggest headache is that every time we need to add support for a new CLI feature, we have to modify the same code in several projects. This approach is obviously not sustainable in the long run—code has a personality too, and too much repetition makes it misbehave.

## About HagiCode

The solution shared in this article comes from our practical experience in the [HagiCode](https://hagicode.com) project. HagiCode is an open-source AI code assistant project that needs to maintain multiple subprojects including a VSCode extension, backend AI service, and cross-platform desktop client. Let's just say it was this complex multi-language, multi-platform scenario that drove the birth of HagiCode.Libs—forced into existence, so be it.

## Analysis: Finding Common Ground

Although these AI programming assistant CLIs each have their characteristics, from a technical perspective, they share obvious common features:

**Similar interaction patterns**: They all start a CLI process, send prompts, receive streaming responses, parse messages, and finally end or continue the session—this whole process is basically cut from the same mold.

**Similar configuration requirements**: They all need API key authentication, working directory setup, model selection, tool permission control, and session management. After all, everyone's livelihood depends on APIs—the differences are just a matter of taste.

**Consistent cross-platform challenges**: They all need to solve executable path resolution (`claude` vs `claude.exe` vs `/usr/local/bin/claude`), process startup and environment variable handling, shell command escaping and parameter construction, and other issues. Cross-platform work... the more you talk about it, the more it hurts—only those who've stepped in these potholes know the differences between Windows and Unix.

Based on this analysis, we needed a unified abstraction layer to provide consistent interfaces, encapsulate cross-platform CLI discovery logic, handle streaming output parsing, and support both dependency injection and non-DI scenarios. It's overwhelming just thinking about it, but we still have to face it—after all, it's our own project, and we have to finish it even if we're crying.

## Solution: HagiCode.Libs

We created **HagiCode.Libs**—a lightweight .NET 10 library workspace, released under the MIT open-source license, now published on [GitHub](https://github.com/HagiCode-org/Hagicode.Libs). Although it's not some earth-shattering masterpiece, it's pretty satisfying to solve real problems.

### Project Structure

```
HagiCode.Libs/
├── src/
│   ├── HagiCode.Libs.Core/           # Core functionality
│   │   ├── Discovery/                 # CLI executable discovery
│   │   ├── Process/                   # Cross-platform process management
│   │   ├── Transport/                 # Streaming message transport
│   │   └── Environment/               # Runtime environment resolution
│   ├── HagiCode.Libs.Providers/       # Provider implementations
│   │   ├── ClaudeCode/                # Claude Code provider
│   │   ├── Codex/                     # Codex provider
│   │   └── Codebuddy/                 # CodeBuddy provider
│   ├── HagiCode.Libs.ConsoleTesting/  # Testing framework
│   ├── HagiCode.Libs.ClaudeCode.Console/
│   ├── HagiCode.Libs.Codex.Console/
│   └── HagiCode.Libs.Codebuddy.Console/
└── tests/                             # xUnit tests
```

### Design Goals

When designing HagiCode.Libs, we followed a few principles—after all, they're all lessons learned from stepping in potholes:

**Zero heavy framework dependencies**: No dependency on ABP or other large frameworks, keep it lightweight. These days, fewer dependencies mean fewer troubles—who hasn't been tortured by dependency hell?

**Cross-platform support**: Native support for Windows, macOS, and Linux, no need to write different code for different platforms. One codebase to rule them all—pretty nice, actually.

**Streaming processing**: Use async streams to handle CLI output, more aligned with modern .NET programming patterns. Times are changing, async is the way forward.

**Flexible integration**: Support both dependency injection scenarios and allow direct instantiation. Different strokes for different folks—whatever is most convenient.

## Usage

### Through Dependency Injection

If your project already uses dependency injection (like ASP.NET Core or Generic Host), you can integrate directly—it's like a child, though small, quite well-behaved:

```csharp
using HagiCode.Libs.Providers;
using Microsoft.Extensions.DependencyInjection;

var services = new ServiceCollection();
services.AddHagiCodeLibs();

await using var provider = services.BuildServiceProvider();
var claude = provider.GetRequiredService<ICliProvider<ClaudeCodeOptions>>();

var options = new ClaudeCodeOptions
{
    ApiKey = "your-api-key",
    Model = "claude-sonnet-4-20250514"
};

await foreach (var message in claude.ExecuteAsync(options, "Hello, Claude!"))
{
    Console.WriteLine($"{message.Type}: {message.Content}");
}
```

### Direct Instantiation

For simple scripts or non-DI scenarios, creating an instance directly works too—basically, it's a matter of personal preference:

```csharp
var claude = new ClaudeCodeProvider();
var options = new ClaudeCodeOptions
{
    ApiKey = "sk-ant-xxx",
    Model = "claude-sonnet-4-20250514"
};

await foreach (var message in claude.ExecuteAsync(options, "帮我写一个快速排序"))
{
    // Handle messages
}
```

Both approaches use the same underlying implementation, and you can choose the appropriate integration method based on your project's actual situation. There's no standard answer in this world—whatever suits you is best—though this is a cliché, it's really the truth.

## Practical Experience Sharing

### 1. Dedicated Test Consoles

Each provider has a dedicated test console project for convenient independent verification of integration效果—let's just say, when it comes to testing, either don't do it or do it thoroughly:

```bash
# Claude Code testing
dotnet run --project src/HagiCode.Libs.ClaudeCode.Console -- --test-provider
dotnet run --project src/HagiCode.Libs.ClaudeCode.Console -- --test-all claude

# CodeBuddy testing
dotnet run --project src/HagiCode.Libs.Codebuddy.Console -- --test-provider codebuddy-cli

# Codex testing
dotnet run --project src/HagiCode.Libs.Codex.Console -- --test-provider codex-cli
```

Test scenarios cover several key scenarios:
- **Ping**: Health check, confirm CLI availability
- **Simple Prompt**: Basic prompt testing
- **Complex Prompt**: Multi-turn conversation testing
- **Session Restore/Resume**: Session recovery testing
- **Repository Analysis**: Codebase analysis testing

This independent test console design is particularly useful during debugging, allowing you to quickly locate whether the problem is in the HagiCode.Libs layer or the CLI itself. Debugging, basically, is figuring out where the problem is—once you have the right direction, you're halfway there.

### 2. Cross-platform CI/CD Validation

Cross-platform compatibility is one of HagiCode.Libs' core goals. We configured a GitHub Actions workflow `.github/workflows/cli-discovery-cross-platform.yml` that runs real CLI discovery validation on three platforms: `ubuntu-latest`, `macos-latest`, and `windows-latest`.

This ensures that every code change doesn't break cross-platform compatibility. You can also reproduce this locally during development—you can't let CI take all the blame, you need to be able to run it locally too:

```bash
npm install --global @anthropic-ai/claude-code@2.1.79
HAGICODE_REAL_CLI_TESTS=1 dotnet test --filter "Category=RealCli"
```

### 3. Message Stream Processing

HagiCode.Libs uses async streams to process CLI output, which is more aligned with modern .NET async programming style than traditional callback or event patterns—ultimately, this is technological progress, and no one can stop it:

```csharp
public async IAsyncEnumerable<CliMessage> ExecuteAsync(
    TOptions options,
    string prompt,
    [EnumeratorCancellation] CancellationToken cancellationToken = default)
{
    // Start CLI process
    // Parse streaming JSON output
    // Generate CliMessage sequence
}
```

Message types include:
- `user`: User message
- `assistant`: Assistant response
- `tool_use`: Tool call
- `result`: Session end

This design allows callers to handle streaming output flexibly, such as real-time display, buffered processing, or forwarding to other services. Why care about sunny or cloudy? The important thing is that your thinking is open—any way you use it works.

### 4. Git Repository Exploration

The HagiCode.Libs.Exploration module provides Git repository discovery and status checking functionality, which is particularly useful in codebase analysis scenarios—this is also a feature forced into existence, since HagiCode needs to analyze codebases:

```csharp
// Discover Git repositories
var repositories = await GitRepositoryDiscovery.DiscoverAsync("/path/to/search");

// Get repository information
var info = await GitRepository.GetInfoAsync(repoPath);
Console.WriteLine($"Branch: {info.Branch}, Remote: {info.RemoteUrl}");
Console.WriteLine($"Has uncommitted changes: {info.HasUncommittedChanges}");
```

HagiCode's code analysis functionality uses this module to identify project structure and Git status. Making the best of things, and raising a child well—it wasn't raised in vain.

## Important Notes

Based on our practice in the HagiCode project, there are a few areas that need special attention—they're all just things, things, things:

**API Key Security**: Don't hardcode API keys into code; use environment variables or configuration management. HagiCode.Libs supports passing configuration through Options objects, making it easy to integrate various configuration sources. After all, you can't be too careful with security.

**CLI Version Locking**: In CI/CD, we lock specific versions (like `@anthropic-ai/claude-code@2.1.79`) to reduce uncertainty from version drift. We recommend using fixed versions for local development as well. Version issues... the more you talk, the more you cry—if you don't fix versions, you'll get taught a lesson in no time.

**Test Categorization**: By default, tests use fake providers to maintain determinism and speed, while real CLI tests need explicit enabling. This ensures both fast CI feedback and real environment validation when needed. Walking this balance beam isn't easy—fast and stable always require trade-offs.

**Session Management**: Different CLIs have different session recovery mechanisms. Claude Code uses the `.claude/` directory to store sessions, while Codex and CodeBuddy each have their own methods. When using them, pay attention to their respective documentation to understand the specific mechanisms of session persistence. That's fine—understanding clearly never hurts.
