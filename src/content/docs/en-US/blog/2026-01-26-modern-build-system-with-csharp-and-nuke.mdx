## Goodbye Script Hell: Why We Chose to Build a Modern Build System with C#

> Revealing how the HagiCode project leverages Nuke to implement a type-safe, cross-platform, and highly scalable automated build process, completely solving the maintenance pain points of traditional build scripts.

## Background

We love it because with one click, code transforms into product—that's one of the most captivating moments for programmers. But we hate it because maintaining that messy pile of build scripts is simply a nightmare.

In many projects, we've grown accustomed to writing scripts in Python, or using XML configuration files (imagine the dread of being dominated by `<property>` tags). But as project complexity increases—especially for projects like HagiCode that involve frontend and backend, multi-platform, and multi-language mixed development—traditional build approaches start to show their limitations. To solve these pain points, in the HagiCode project, we decided to introduce **Nuke**—a modern build system based on C#. It's not just a tool, but more of a rethinking of the build process. Today, let's talk about why we chose it and how it made our development experience "take off."

## Core Analysis: Why Nuke?

You might be thinking: "There are so many build systems out there, like Make, Gradle, or even just using Shell scripts. Why do we have to use a C# one?"

That's actually a great question. Nuke's core appeal lies in bringing the programming language features we're most familiar with into the world of build scripts.

### 1. Modularize the Build Process: The Art of Targets

Nuke's design philosophy is very clear: **everything is a target**.

In traditional scripts, we might write hundreds of lines of linearly executing code with complex logic. In Nuke, we break down the build process into independent `Target`s. Each target is responsible for only one thing, such as:

- `Clean`: Clean output directories
- `Restore`: Restore dependency packages
- `Compile`: Compile code
- `Test`: Run unit tests

This design fits perfectly with the single responsibility principle. Like building blocks, we can freely combine these Targets. More importantly, Nuke allows us to define dependencies between Targets. For example, if you want `Test`, the system automatically checks if you've executed `Compile` first; if you want `Compile`, naturally you need `Restore` first.

This dependency graph not only makes logic clearer but also greatly improves execution efficiency, as Nuke automatically analyzes the optimal execution path.

### 2. Type Safety: Goodbye Spelling Error Nightmares

Friends who have used Python to write build scripts have definitely encountered this awkward situation: the script runs for five minutes, only to report an error that `Confi.guration` is misspelled, or a string was passed to a parameter that should have been a number.

The biggest advantage of using C# to write build scripts is **type safety**. This means:

- **Compile-time checking**: While you're coding, the IDE tells you where you went wrong, without waiting until runtime to discover it.
- **Refactoring without worry**: If you want to change a variable or method name, the IDE's refactoring feature handles it in one click, without the anxiety of global search and replace.
- **Intelligent prompts**: Powerful IntelliSense automatically completes code, so you don't need to flip through documentation to memorize obscure APIs.

### 3. Cross-Platform: A Unified Build Experience

Before, we wrote `.bat` on Windows, `.sh` on Linux, and to make both compatible, we even had to write a Python script. Now, wherever .NET Core (now .NET 5+) can run, Nuke can run.

This means whether team members use Windows, Linux, or macOS, whether they use Visual Studio, VS Code, or Rider, everyone is executing the same set of logic. This greatly eliminates problems caused by environmental differences like "it works on my machine."

### 4. Parameter and Configuration Management

Nuke provides a very elegant parameter parsing mechanism. You don't need to manually parse `string[] args`; just define a property, add the `[Parameter]` attribute, and Nuke automatically handles the mapping between command-line arguments and configuration files.

For example, we can easily define build configurations:

```csharp
[Parameter("Configuration to build - Default is 'Debug'")]
readonly Configuration BuildConfiguration = IsLocalBuild ? Configuration.Debug : Configuration.Release;

Target Compile => _ => _
    .DependsOn(Restore)
    .Executes(() =>
    {
        // Use BuildConfiguration here, it's type-safe
        DotNetBuild(s => s
            .SetConfiguration(BuildConfiguration)
            .SetProjectFile(SolutionFile));
    });
```

This approach is both intuitive and error-resistant.

## Practical Guide: How to Implement in Projects

Empty talk harms the nation, practical action prospers it. Let's see how we specifically implemented this solution in the HagiCode project.

### 1. Plan Project Structure

We didn't want build scripts to pollute the project root directory, nor did we want a directory structure as deep as some Java projects. So, we placed all Nuke-related build files uniformly in the `nukeBuild/` folder.

The benefits of this approach are:
- The project root directory stays clean.
- Build logic is cohesive and easy to manage.
- When new members join, they can see at a glance "oh, this is build-related logic."

### 2. Design Clear Target Dependency Chains

When designing Targets, we followed a principle: **atomic + dependency flow**.

Each Target should be small enough and do only one thing. For example, `Clean` should only handle deleting files, don't conveniently do packaging in it.

The recommended dependency flow looks something like this:

`Clean` -> `Restore` -> `Compile` -> `Test` -> `Pack`

Of course, this isn't absolute. For example, if you just want to run tests and don't want to package, Nuke allows you to directly execute `nuke Test`, and it will automatically handle the prerequisite Restore and Compile steps.

### 3. Comprehensive Error Handling and Logging

What's the most feared thing about build scripts? Unclear error messages. For example, a build fails, and the log only shows "Error: 1"—that's frustrating.

In Nuke, since we can directly use C#'s exception handling mechanism, we can capture and report errors very precisely.

```csharp
Target Publish => _ => _
    .DependsOn(Test)
    .Executes(() =>
    {
        try 
        {
            // Try publishing to NuGet
            DotNetNuGetPush(s => s
                .SetTargetPath(ArtifactPath)
                .SetSource("https://api.nuget.org/v3/index.json")
                .SetApiKey(ApiKey));
        }
        catch (Exception ex)
        {
            Log.Error($"Publishing failed, folks check if the Key is correct: {ex.Message}");
            throw; // Ensure the build process ends with a non-zero exit code
        }
    });
```

### 4. Integrated Testing to Ensure Quality

Build scripts are also code and need testing. Nuke allows us to write tests for the build process, ensuring that when we modify build logic, we don't break existing publishing workflows. This is particularly important in continuous integration (CI) pipelines.

## Summary

By introducing Nuke, HagiCode's build process became unprecedentedly smooth. It's not just a tool replacement, but an elevation of engineering thinking.

**What have we gained?**
- **Maintainability**: Code as configuration, clear logic, new team members can quickly get started.
- **Stability**: Strong type checking reduces over 90% of silly mistakes.
- **Consistency**: Unified cross-platform experience eliminates environmental differences.

If writing build scripts before was "groping in the dark," then using Nuke is like "walking at night with the lights on." If you're tired of maintaining those hard-to-debug scripting languages, try moving build logic into the C# world—you might find that building can be elegant too.
