## HagiCode Soul Platform Technical Analysis: From Requirements to Independent Platform Evolution

> Writing technical articles isn't really a big deal—it's just documenting the pitfalls encountered and detours taken. After all, who hasn't been young before, right? This article will delve into the design philosophy, architecture evolution, and core technical implementation of Soul (AI Agent Personality Configuration System) in the HagiCode project, exploring how to provide a more focused experience for creating and sharing Agent personalities through an independent platform.

## Background

In AI Agent development practice, we often encounter a seemingly simple yet extremely important question: **How can different Agents have stable and unique language styles and personality traits?**

This problem is actually quite frustrating. In the early HagiCode Hero system, different heroes (Agent instances) primarily relied on profession configurations and generic prompts to distinguish their modes of expression. This approach brought some obvious pain points—perhaps those who have done this can relate.

First, language styles are difficult to maintain consistently. The same "Development Engineer" role might respond professionally and rigorously today, but tomorrow's output becomes casual and scattered. This isn't a problem with the model itself, but rather the lack of an independent personality configuration layer to constrain and guide the output style.

Second, the sense of character is generally weak. When describing an Agent's characteristics, we can only use vague adjectives like "friendly," "professional," or "humorous," without specific language rules to support these abstract descriptions. To put it plainly, it sounds nice but doesn't work in practice.

Third, the reusability of personality configurations is nearly zero. Suppose we carefully design a "cat maid waitress" speaking style and want to reuse this mode of expression in another business scenario—almost needs to be configured from scratch. Beautiful things or people don't necessarily need to be possessed, just reused... but it's truly difficult.

It was precisely to solve these practical problems that we introduced the **Soul** mechanism—a **language style configuration layer** independent of equipment and descriptions. Soul can define an Agent's speaking habits, tone preferences, and word boundaries. It can be shared and reused across multiple heroes, and can automatically inject system prompts when a Session is first called.

Some might think this is nothing special—just configuring a few prompts, right? But sometimes, the key to a problem isn't whether it can be done, but how to do it more elegantly. As Soul's capabilities gradually matured, we realized it had potential for independent development. A dedicated Soul platform could allow users to more focusedly create, share, and browse various interesting personality configurations without being distracted by other features of the Hero system. Thus, the **soul.hagicode.com** independent platform came into being.

## Soul Platform's Technical Architecture Evolution

The development of the Soul platform wasn't achieved in one step, but went through three distinct stages. This story began suddenly and ended naturally.

### Stage 1: Hero-Embedded Soul Configuration

The earliest Soul implementation existed as a functional module within the Hero workspace. We added an independent SOUL editing area to the Hero interface, supporting both preset application and text fine-tuning methods.

Preset application allows users to select from classic personality templates, such as "Professional Development Engineer" or "Cat Maid Waitress." Text fine-tuning allows users to make personalized modifications based on presets. The backend Hero entity correspondingly added a `Soul` field and identified sources through `SoulCatalogId`.

This stage solved the problem of "whether it exists," and was still like a child, growing stumblingly. However, as Soul content became increasingly rich, the architecture coupled with the Hero system began to show limitations.

### Stage 2: In-Site Marketplace

To provide better Soul discovery and reuse experience, we built a SOUL Marketplace catalog page, supporting browsing, searching, detail viewing, and bookmarking functions.

At this stage, we introduced a combination design of **50 main Catalogs (base roles)** and **10 orthogonal rules (expression modes)**. Main Catalogs define an Agent's core persona, such as abstract character settings like "Mistport Traveler" or "Night Hunter"; orthogonal rules define modes of expression, such as language style features like "Concise & Professional" or "Verbose & Friendly."

50 × 10 = 500 combination possibilities provide users with a rich personality configuration space. This quantity is neither much nor little—how should I put it, all roads lead to Rome, just some roads are easier to walk than others. The backend generates a complete SOUL catalog through `catalog-sources.json`, while the frontend presents these catalog items as an interactive card list.

The in-site Marketplace was a good transitional solution, but still only a transition. It remained attached to the main system, and for users who only wanted to use Soul functionality, the access path was still too deep. After all, who wants to go around in circles just to do one simple thing?

### Stage 3: Independent Platform Split

Ultimately, we decided to migrate Soul capabilities to an independent repository (`repos/soul`), changing the main system's Marketplace to external jump guidance. The new platform adopts a **Builder-first** design philosophy—the default homepage is the creation workbench, and users can start creating their own personality configurations the moment they open the website.

The technology stack at this stage also underwent a comprehensive upgrade: adopting the Vite 8 + React 19 + TypeScript 5.9 combination, using the shadcn/ui component system to unify design language, and introducing Tailwind CSS 4's theme variable system. The improvement in frontend engineering level laid a solid foundation for subsequent feature iteration.

Everything has faded... no, everything has just begun.

## Core Technical Design and Implementation

### Material Integration Strategy

A core design philosophy of the Soul platform is **local-first**. This means the homepage must be fully operational without a backend, and remote material failures must not block page entry.

This isn't really remarkable—just considering one more step when designing the system. Local snapshot as baseline, remote as enhancement, this approach ensures the product can provide basic usability under any network conditions. Specifically, we adopted a two-layer material architecture:

```typescript
export async function loadBuilderMaterials(): Promise<BuilderMaterials> {
  const localMaterials = createLocalMaterials(snapshot)  // Local baseline

  try {
    const inspirationFragments = await fetchMarketplaceItems()  // Remote enhancement
    return { ...localMaterials, inspirationFragments, remoteState: "ready" }
  } catch (error) {
    return { ...localMaterials, remoteState: "fallback" }  // Graceful degradation
  }
}
```

Local materials come from build-time snapshots of the main system documentation, containing complete data for 50 base roles and 10 expression rules. Remote materials come from user-published Souls, obtained through the Marketplace API. The combination provides users with a complete material spectrum from official templates to community creativity. Wanting to laugh to mask fallen tears... no, actually nothing, just local plus remote.

### Soul Fragment Data Model

Soul's core data abstraction is **SoulFragment** (Soul Fragment):

```typescript
export type SoulFragment = {
  fragmentId: string
  group: "main-catalog" | "expression-rule" | "published-soul"
  title: string
  summary: string
  content: string
  keywords: string[]
  localized?: Partial<Record<AppLocale, LocalizedFragmentContent>>
  sourceRef: SoulFragmentSourceRef
  meta: SoulFragmentMeta
}
```

The `group` field distinguishes fragment types: main catalog defines character core, orthogonal rules define expression modes, and user-published Souls are marked as `published-soul`. The `localized` field supports multilingual, allowing the same fragment to present different titles and descriptions in different language environments. Internationalization design should be done early—we've also put this into practice.

Builder draft state encapsulates the user's current editing state:

```typescript
export type SoulBuilderDraft = {
  draftId: string
  name: string
  selectedMainFragmentId: string | null

  inspirationSoulId: string | null

  ruleSlotText: string
  customPrompt: string

  updatedAt: string

```

Each fragment selected by the user in the editor has its content spliced into the corresponding slot, forming the final preview text. `mainSlotText` corresponds to main role content, `ruleSlotText` corresponds to expression rule content, and `customPrompt` is the user's additional supplementary instructions.

### Frontend State Management

Soul Builder's frontend state management follows an important principle: **clear state boundary division**. Specifically, drawer state is not persisted and not directly written to drafts; only explicit Builder operations trigger state changes.

```typescript
// Domain state (useSoulBuilder)
export function useSoulBuilder() {
  // Material loading and caching
  // Slot aggregation and preview compilation
  // Copy behavior and feedback messages
  // Locale-safe descriptors
}

// Presentation state (useHomeEditorState)
export function useHomeEditorState() {
  // activeSlot, drawerSide, drawerOpen
  // Default focus behavior
}
```

This separation ensures edit state security and interface response speed. Drawer opening and closing is pure UI interaction and doesn't need to trigger complex persistence logic. This is nothing but stating the obvious! No, actually it's very important—interface state and business state must be clearly distinguished to avoid UI interaction polluting the core data model.

### Single Drawer Lifecycle

Soul Builder uses a single drawer mode: only one slot drawer is allowed to be open at a time. Clicking the mask layer, pressing ESC, or switching slots all automatically close the current drawer. This design simplifies state management and conforms to common mobile drawer interaction patterns.

Drawer closing doesn't clear current edit content; when users switch back, context is preserved. This "lightweight" drawer design avoids user operation interruption. After all, who wants their hard-written content to disappear because of an accidental click?

### Bilingual Support Architecture

Internationalization is an important feature of the Soul platform. System text fully supports bilingual switching, while user draft text is never rewritten due to language switching—because draft text is user-input content and doesn't involve system translation.

Official inspiration cards (Marketplace Soul) maintain upstream display names but provide best-effort English summaries. For Chinese-named Souls, we generate English versions through predefined mapping rules:


## Platform Migration Strategy

After splitting Soul from the main system to an independent platform, we faced an important challenge: how to handle existing user data. This is a common problem—splitting is easy, migration is hard. We adopted three safeguard measures:

**Backward compatibility guarantee**. Saved Hero SOUL snapshots remain visible, and historical snapshots can still be previewed even without Marketplace source IDs. This means all previous user configurations are not lost, just displayed in a different location. After all, no one wants their hard-earned configurations to suddenly disappear.

**Main system API deprecation**. The in-site Marketplace API returns a 410 Gone status code with migration guidance, directing users to visit soul.hagicode.com.

**Hero SOUL form modification**. Added a migration notification block to the Hero Soul editing area, clearly informing users that the Soul platform has become independent and providing a one-click jump button:

```typescript
// HeroSoulForm.tsx
<div className="rounded-2xl border border-orange-200/70 bg-orange-50/80 p-4">
```

## Practice Point Summary

Looking back at the entire Soul platform development process, several practical experiences are worth sharing. This is also some insight from an experienced person—not great principles, just pitfalls encountered.

**Local-first runtime assumptions**. When designing features that depend on remote data, always assume the network may be unavailable. Local snapshot as baseline, remote as enhancement—this approach ensures the product can provide basic usability under any network conditions. After all, these days, networks can disconnect at any time, you never know.

**Clear state boundary division**. Interface state and business state must be clearly distinguished to avoid UI interaction polluting the core data model. Drawer switch is pure UI state and doesn't need to be mixed with draft persistence.

**Internationalization design should be done early**. If your product has internationalization needs, it's best to consider them during the data model design phase. Although the `localized` field increases data structure complexity, the cost of maintaining multilingual content is greatly reduced later.

**Material sync workflow should be automated**. Soul platform's local materials come from main system documentation. When upstream documentation updates, there needs to be a mechanism to sync to frontend snapshots. We designed the `npm run materials:sync` script to automate this process, ensuring materials always stay consistent with upstream.
