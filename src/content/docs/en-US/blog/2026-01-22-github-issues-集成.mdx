## Building GitHub Issues Integration from Scratch: HagiCode's Frontend Direct-Connect Practice

> This article documents the complete process of integrating GitHub Issues into the HagiCode platform. We explore how to implement secure OAuth authentication and efficient Issues synchronization while maintaining a lightweight backend through a "frontend-direct-connect + backend-minimization" architecture.

## Background: Why Integrate GitHub?

HagiCode, as an AI-assisted development platform, has its core value in connecting ideas with implementation. However, in actual use, we discovered that after users complete Proposals in HagiCode, they often need to manually copy the content to GitHub Issues for project tracking.

This brings several obvious pain points:

1.  **Workflow Fragmentation**: Users need to switch back and forth between two systems. The experience is not smooth, and it's easy for key information to be lost during copy-paste operations.
2.  **Inconvenient Collaboration**: Other team members are accustomed to viewing tasks on GitHub and cannot directly see the progress of proposals in HagiCode.
3.  **Duplicate Effort**: Every time a proposal is updated, someone has to manually update the corresponding Issue on GitHub, increasing unnecessary maintenance costs.

To solve this problem, we decided to introduce **GitHub Issues Integration** functionality, connecting HagiCode conversations with GitHub repositories to achieve "one-click sync."

## Technology Selection: Frontend Direct-Connect vs Backend Proxy

When designing the integration solution, we faced two paths: the traditional "backend proxy mode" and the more aggressive "frontend direct-connect mode."

### Solution Comparison

In the traditional **backend proxy mode**, all frontend requests must first go through our backend, which then calls the GitHub API. Although this centralizes logic, it places a significant burden on the backend:

1.  **Backend Bloat**: Requires writing specialized GitHub API client wrappers and handling the complex state machine of OAuth.
2.  **Token Risk**: Users' GitHub Tokens must be stored in the backend database. Although encryption is possible, it increases the security risk surface.
3.  **Development Cost**: Requires database migrations to store Tokens and maintaining an additional synchronization service.

The **frontend direct-connect mode** is much lighter. In this solution, we only use the backend to handle the most sensitive "key exchange" step (OAuth callback). After obtaining the Token, it's stored directly in the browser's localStorage. Subsequent operations like creating Issues and updating comments are done directly by the frontend making HTTP requests to GitHub.

| Comparison Dimension | Backend Proxy Mode | Frontend Direct-Connect Mode |
| :--- | :--- | :--- |
| **Backend Complexity** | Requires complete OAuth service and GitHub API client | Only needs one OAuth callback endpoint |
| **Token Management** | Needs encrypted storage in database with leakage risk | Stored in browser, visible only to user |
| **Implementation Cost** | Requires database migration, multi-service development | Primarily frontend effort |
| **User Experience** | Unified logic, but server latency may be slightly higher | Extremely fast response, direct interaction with GitHub |

Considering we wanted rapid integration and minimal backend changes, **we ultimately adopted the "frontend direct-connect mode."** This is like issuing a "temporary pass" to the browser. After getting the pass, the browser can handle GitHub business on its own without needing approval from the backend administrator every time.

## Core Design: Data Flow and Security

After determining the architecture, we needed to design the specific data flow. The core of the entire synchronization process lies in how to securely obtain the Token and efficiently utilize it.

### Overall Architecture Diagram

The entire system can be abstracted into three roles: browser (frontend), HagiCode backend, and GitHub.

```text
+--------------+        +--------------+        +--------------+
|  Frontend    |        |    Backend   |        |    GitHub    |
|   React      |        |   ASP.NET    |        |    REST API  |
|              |        |              |        |              |
|  +--------+  |        |              |        |              |
|  |  OAuth |--+--------> /callback    |        |              |
|  |  Flow  |  |        |              |        |              |
|  +--------+  |        |              |        |              |
|              |        |              |        |              |
|  +--------+  |        |  +--------+  |        |  +--------+  |
|  |GitHub  |  +------------>Session |  +----------> Issues |  |
|  |API     |  |        |  |Metadata|  |        |  |        |  |
|  |Direct  |  |        |  +--------+  |        |  +--------+  |
|  +--------+  |        |              |        |              |
+--------------+        +--------------+        +--------------+
```

**The key point is**: Only a small step of OAuth (getting code to exchange for token) needs to go through the backend. After that, the heavy work (creating Issues) is handled directly by the frontend with GitHub.

### Detailed Synchronization Data Flow

When the user clicks the "Sync to GitHub" button on the HagiCode interface, a series of complex actions occur:

```text
User clicks "Sync to GitHub"
         │
         ▼
1. Frontend checks localStorage to get GitHub Token
         │
         ▼
2. Format Issue content (convert Proposal to Markdown)
         │
         ▼
3. Frontend directly calls GitHub API to create/update Issue
         │
         ▼
4. Call HagiCode backend API to update Session.metadata (store Issue URL and other info)
         │
         ▼
5. Backend broadcasts SessionUpdated event through SignalR
         │
         ▼
6. Frontend receives event and updates UI to show "synced" status
```

### Security Design

Security issues are always paramount when integrating third-party services. We made the following considerations:

1.  **CSRF Protection**: During OAuth redirects, generate a random `state` parameter and store it in sessionStorage. Strictly verify the state during callback to prevent request forgery.
2.  **Token Storage Isolation**: Tokens are only stored in the browser's `localStorage`. Leveraging the Same-Origin Policy, only HagiCode scripts can read them, avoiding the risk of server-side database breaches affecting users.
3.  **Error Boundaries**: For common GitHub API errors (such as 401 Token expiration, 422 validation failure, 429 rate limiting), we designed specialized error handling logic to provide user-friendly prompts.

### 2. Frontend OAuth Flow

This is the entry point for the connection. We use the standard Authorization Code Flow.

```typescript
// src/HagiCode.Client/src/services/githubOAuth.ts

// Generate authorization URL and redirect
export async function generateAuthUrl(): Promise<string> {
  const state = generateRandomString(); // Generate anti-CSRF random string
  sessionStorage.setItem('hagicode_github_state', state);
  
  const params = new URLSearchParams({
    client_id: clientId,
    redirect_uri: window.location.origin + '/settings?tab=github&oauth=callback',
    scope: ['repo', 'public_repo'].join(' '),
    state: state,
  });
  
  return `https://github.com/login/oauth/authorize?${params.toString()}`;
}

// Handle Code-to-Token exchange on callback page
export async function exchangeCodeForToken(code: string, state: string): Promise<GitHubToken> {
  // 1. Verify State to prevent CSRF
  const savedState = sessionStorage.getItem('hagicode_github_state');
  if (state !== savedState) throw new Error('Invalid state parameter');

  // 2. Call backend API for Token exchange
  // Note: This must go through the backend because ClientSecret is required and cannot be exposed in the frontend
  const response = await fetch('/api/GitHubOAuth/callback', {
    method: 'POST',
    headers: { 'Content-Type': 'application/json' },
    body: JSON.stringify({ code, state, redirectUri: window.location.origin + '/settings?tab=github&oauth=callback' }),
  });

  if (!response.ok) throw new Error('Failed to exchange token');
  
  const token = await response.json();
  
  // 3. Store in LocalStorage
  saveToken(token);
  return token;
}
```

### 3. GitHub API Client Wrapper

Once we have the Token, we need a powerful tool to call the GitHub API.

```typescript
// src/HagiCode.Client/src/services/githubApiClient.ts

const GITHUB_API_BASE = 'https://api.github.com';

// Core request wrapper
async function githubApi<T>(endpoint: string, options: RequestInit = {}): Promise<T> {
  const token = localStorage.getItem('gh_token');
  if (!token) throw new Error('Not connected to GitHub');
  
  const response = await fetch(`${GITHUB_API_BASE}${endpoint}`, {
    ...options,
    headers: {
      ...options.headers,
      Authorization: `Bearer ${token}`,
      Accept: 'application/vnd.github.v3+json', // Specify API version
    },
  });
  
  // Error handling logic
  if (!response.ok) {
    if (response.status === 401) throw new Error('GitHub Token 失效，请重新连接');
    if (response.status === 403) throw new Error('无权访问该仓库或超出速率限制');
    if (response.status === 422) throw new Error('Issue 验证失败，可能标题重复');
    throw new Error(`GitHub API Error: ${response.statusText}`);
  }
  
  return response.json();
}

// Create Issue
export async function createIssue(owner: string, repo: string, data: { title: string, body: string, labels: string[] }) {
  return githubApi(`/repos/${owner}/${repo}/issues`, {
    method: 'POST',
    body: JSON.stringify(data),
  });
}
```

### 4. Content Formatting and Sync

The final step is to convert HagiCode Session data into GitHub Issue format. This is somewhat like "translation" work.

```typescript
// Convert Session object to Markdown string
function formatIssueForSession(session: Session): string {
  let content = `# ${session.title}\n\n`;
  content += `**> HagiCode Session:** #${session.code}\n`;
  content += `**> Status:** ${session.status}\n\n`;
  content += `## Description\n\n${session.description || 'No description provided.'}\n\n`;
  
  // If it's a Proposal type, add extra fields
  if (session.type === 'proposal') {
    content += `## Chief Complaint\n\n${session.chiefComplaint || ''}\n\n`;
    // Add a deep link for easy navigation from GitHub back to HagiCode
    content += `---\n\n**[View in HagiCode](hagicode://sessions/${session.id})**\n`;
  }
  
  return content;
}

// Main logic for sync button click
const handleSync = async (session: Session) => {
  try {
    const repoInfo = parseRepositoryFromUrl(session.repoUrl); // Parse repository URL
    if (!repoInfo) throw new Error('Invalid repository URL');

    toast.loading('正在同步到 GitHub...');
    
    // 1. Format content
    const issueBody = formatIssueForSession(session);
    
    // 2. Call API
    const issue = await githubApiClient.createIssue(repoInfo.owner, repoInfo.repo, {
      title: `[HagiCode] ${session.title}`,
      body: issueBody,
      labels: ['hagicode', 'proposal', `status:${session.status}`],
    });
    
    // 3. Update Session Metadata (save Issue link)
    await SessionsService.patchApiSessionsSessionId(session.id, {
      metadata: {
        githubIssue: {
          owner: repoInfo.owner,
          repo: repoInfo.repo,
          issueNumber: issue.number,
          issueUrl: issue.html_url,
          syncedAt: new Date().toISOString(),
        }
      }
    });

    toast.success('同步成功！');
  } catch (err) {
    console.error(err);
    toast.error('同步失败，请检查 Token 或网络');
  }
};
```

## Summary and Outlook

Through this "frontend direct-connect" solution, we achieved seamless integration of GitHub Issues with minimal backend code.

### Benefits
1.  **High Development Efficiency**: Backend changes are minimal - primarily adding one database field and a simple OAuth callback interface, with most logic completed on the frontend.
2.  **Good Security**: Tokens don't go through the server database, reducing leakage risk.
3.  **Great User Experience**: Requests are initiated directly from the frontend with fast response times, no need for backend relay.

### Considerations
In actual deployment, there are several pitfalls to watch out for:
- **OAuth App Settings**: Remember to fill in the correct `Authorization callback URL` in the GitHub OAuth App settings (usually `http://localhost:3000/settings?tab=github&oauth=callback`).
- **Rate Limiting**: GitHub API has stricter limits on unauthenticated requests, but with Token it's usually sufficient (5000 times/hour).
- **URL Parsing**: User-entered Repo URLs come in various forms. Remember to match `.git` suffixes, SSH formats, and other cases with regex.

### Future Enhancements
Current functionality is one-way sync (HagiCode -> GitHub). In the future, we plan to implement bidirectional sync through GitHub Webhooks, such as automatically updating HagiCode session status when an Issue is closed on GitHub. This requires exposing a Webhook receiver endpoint on the backend, which is the next interesting work to be done.
