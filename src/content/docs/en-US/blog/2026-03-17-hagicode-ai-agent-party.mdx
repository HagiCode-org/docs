## Background

I believe many developers have had this experience: introducing an AI assistant into a project to assist with programming does indeed improve efficiency. But as requirements become increasingly complex, one AI Agent starts to be insufficient—you want it to simultaneously handle code review, documentation generation, unit testing, and other tasks, but the result is often sacrificing one thing for another, with uneven output quality.

Even more troublesome is that when you try to introduce multiple AI assistants, the problem becomes more complex. Each Agent has its own configuration approach, API interface, and execution logic, and may even conflict with each other. It's like a soccer team where every player is amazing, but no one knows how to cooperate, resulting in a messy game.

The HagiCode project encountered the same dilemma during development. As a complex project involving a frontend VSCode extension, backend AI service, and cross-platform desktop client, in the **March 2026 version** we needed to simultaneously interface with multiple AI assistants from different companies: Claude Code, Codex, CodeBuddy, iFlow, and others. How to make them coexist harmoniously in the same project and leverage their respective strengths became a key problem that had to be solved.

Actually, that's fine—after all, who wants to deal with a bunch of fighting AIs every day?

The solution shared in this article is exactly the multi-Agent collaboration configuration practice that we actually stumbled through and optimized in the HagiCode project. If you're also struggling with multi-AI assistant collaboration, I believe this article will give you some inspiration. Maybe, after all, everyone's situation is different.

## HagiCode's Multi-Agent Architecture Design

### From "Solo Fighting" to "Team Collaboration"

In the early stages of the HagiCode project, we also tried using only one AI Agent to handle all tasks. We quickly discovered that this approach had obvious bottlenecks: different tasks require different capability emphases—some tasks need stronger context understanding ability, while others require more precise code modification ability. It's difficult for one Agent to excel in all aspects.

This made us realize that we must enable multiple Agents to work collaboratively. But the problem is, how to make AI products from different companies coexist peacefully in the same project? We needed to solve several core problems:

1. **Configuration Management Complexity**: Each Agent has different configuration approaches, API interfaces, and execution patterns
2. **Unified Communication Protocol**: Need a standardized way to enable data exchange between different Agents
3. **Task Division Coordination**: How to reasonably allocate tasks so that each Agent can leverage its strengths

With these problems in mind, we began designing HagiCode's multi-Agent architecture. Actually, it's not that complex, we just figured it out.

### Overall Architecture Overview

After multiple iterations, our final determined architecture is this:

```
┌─────────────────────────────────────────────────────────────────┐
│                    AIProviderFactory                             │
│  (Factory pattern uniformly manages all AI Providers)            │
├─────────────────────────────────────────────────────────────────┤
│  ClaudeCodeCli  │  CodexCli  │  CodebuddyCli  │  IFlowCli    │
│  (Anthropic)   │  (OpenAI)  │  (Zhipu GLM)    │  (Zhipu)      │
└─────────────────────────────────────────────────────────────────┘
```

The core idea is: through a unified Provider interface, different AI Agents can be managed by the same set of code. At the same time, using the factory pattern to dynamically create and configure these Providers ensures system extensibility and flexibility.

It's like the division of labor in life—everyone has their own role, it's just that here this division of labor is transformed into code architecture.

### Agent Types and Responsibility Division

Based on HagiCode project's actual usage experience, we assigned different responsibilities to each Agent:

| Agent | Provider | Model | Primary Use |
|-------|----------|-------|-------------|
| ClaudeCodeCli | Anthropic | glm-5-turbo | Generate technical proposals and Proposals |
| CodexCli | OpenAI/Zed | gpt-5.4 | Execute precise code modifications |
| CodebuddyCli | Zhipu | glm-4.7 | Optimize proposal descriptions and documentation |
| IFlowCli | Zhipu | glm-4.7 | Archive proposals and historical records (configuration at the time; currently historical compatibility only) |
| OpenCodeCli | - | - | General code editing |
| GitHubCopilot | Microsoft | - | Assisted programming and code completion |

The logic behind this division is: each Agent has its own area of expertise. Claude Code excels at understanding and analyzing complex requirements, so it's tasked with early-stage solution design; Codex is more precise in code modification, making it suitable for handling specific implementation tasks; CodeBuddy has high cost-effectiveness, making it perfect for optimizing documentation.

After all, what suits you is best. All roads lead to Rome, it's just that some roads are easier to walk, while others are a bit more winding.

## Core Configuration Mechanism

### Unified Provider Interface Design

To enable different AI Agents to be managed uniformly, first we need to define a unified interface. HagiCode defines this interface:

```csharp
public interface IAIProvider
{
    // Unified Provider interface
    Task<IAIProvider?> GetProviderAsync(AIProviderType providerType);
    Task<IAIProvider?> GetProviderAsync(string providerName, CancellationToken cancellationToken);
}
```

This interface looks simple, but it's the cornerstone of the entire multi-Agent system. Through the unified interface, we can ignore which company's AI product is at the bottom, and invoke them all in the same way.

Actually, this just simplifies complex things—after all, simplicity is beauty.

### Provider Factory Pattern Implementation

With the unified interface, the next step is how to create these Provider instances. HagiCode uses the factory pattern:

```csharp
private IAIProvider? CreateProvider(AIProviderType providerType, ProviderConfiguration config)
{
    return providerType switch
    {
        AIProviderType.ClaudeCodeCli =>
            ActivatorUtilities.CreateInstance<ClaudeCodeCliProvider>(_serviceProvider, Options.Create(config)),
        AIProviderType.CodebuddyCli =>
            ActivatorUtilities.CreateInstance<CodebuddyCliProvider>(_serviceProvider, Options.Create(config)),
        AIProviderType.CodexCli =>
            ActivatorUtilities.CreateInstance<CodexCliProvider>(_serviceProvider, Options.Create(config)),
        AIProviderType.IFlowCli =>
            ActivatorUtilities.CreateInstance<IFlowCliProvider>(_serviceProvider, Options.Create(config)),
        _ => null
    };
}
```

Here we use dependency injection's `ActivatorUtilities.CreateInstance`, which can dynamically create Provider instances at runtime and automatically inject dependencies. The benefit of this design is: when adding a new Agent type, you only need to add the corresponding Provider class, then add a case branch in the factory method—no need to modify existing code at all.

That's fine, after all, who wants to modify a bunch of old code every time they add new features.

### Dynamic Configuration Resolution

To make configuration more flexible, we also implemented a type mapping mechanism:

```csharp
public static AIProviderTypeExtensions
{
    private static readonly Dictionary<string, AIProviderType> _typeMap = new(
        StringComparer.OrdinalIgnoreCase)
    {
        ["ClaudeCodeCli"] = AIProviderType.ClaudeCodeCli,
        ["CodebuddyCli"] = AIProviderType.CodebuddyCli,
        ["CodexCli"] = AIProviderType.CodexCli,
        ["IFlowCli"] = AIProviderType.IFlowCli,
        // ...more type mappings
    };
}
```

The purpose of this mapping table is to convert string-form Provider names into enum types. This way, configuration files can use intuitive string names, while code internally uses type-safe enums for processing.

After all, with configuration, the more intuitive the better—who wants to remember a bunch of complex code.

In actual use, you only need to configure it in `appsettings.json`:

Each Provider can be independently configured with parameters such as enable/disable switch, model version, working directory, etc. This design ensures both flexibility and easy management and maintenance.

## Adventure Party Task Flow

### The Art of Task Division

With a unified technical architecture, the next step is how to enable multiple Agents to work collaboratively. HagiCode designed a task flow mechanism that lets different Agents handle tasks at different stages:

```
Proposal Creation (User)
    │
    ▼
[Claude Code] ──Generate Proposal──▶ Proposal Document
    │                        │
    │                        ▼
    │               [Codebuddy] ──Optimize Description──▶ Optimized Proposal
    │                        │
    │                        ▼
    │               [Codex] ──Execute Modification──▶ Code Changes
    │                        │
    │                        ▼
    └───────────────▶ [iFlow] ──Archive──▶ Historical Records
```

The benefit of this division is: each Agent only needs to focus on the tasks it's good at, no need to "know everything." Claude Code is responsible for generating proposals from scratch, Codebuddy is responsible for making proposal descriptions clearer, Codex is responsible for turning proposals into actual code changes, and iFlow is responsible for archiving and saving these changes.

Actually, this is like teamwork in life—everyone has their own role, and only together can they accomplish something big. It's just that the team members here are AIs.

### Key Practice Points

In actual operation, we summarized the following experiences:

**1. Agent Selection Strategy is Important**

It's not about randomly assigning tasks, but allocating them based on each Agent's strengths:
- Proposal Generation: Use Claude Code, because it has stronger context understanding ability
- Code Execution: Use Codex, because it's more precise in code modification
- Proposal Optimization: Use CodeBuddy, because it has high cost-effectiveness
- Archival Storage: Use iFlow, because it's stable and reliable

After all, letting the right person do the right thing is an eternal truth.

**2. Configuration Isolation Ensures Stability**

Each Agent's configuration is independently managed, supporting environment variable overrides, and working directories are also independent from each other. This way, one Agent's configuration error won't affect other Agents.

It's like boundaries in life—everyone has their own space, and not interfering with each other is the only way to coexist harmoniously.

**3. Error Handling Mechanism**

Single Agent failure shouldn't affect the overall process. We implemented a degradation strategy: when a certain Agent fails to execute, the system can automatically switch to a backup solution, or directly skip that step and continue executing subsequent tasks. At the same time, complete logging facilitates事后 troubleshooting.

After all, no one can guarantee they'll never make mistakes—the key is how to handle errors. It's like life, you'll always encounter setbacks, what matters is how to get out of them.

**4. Monitoring and Observability**

Through the ACP protocol (our custom communication protocol based on JSON-RPC 2.0), we can track each Agent's execution status. Session isolation ensures concurrent safety, and dynamic caching optimizes performance.

After all, things you can't see are most prone to problems—having some monitoring is better than flying blind.

## Actual Results and Benefits

After adopting this multi-Agent collaboration configuration, HagiCode project's development efficiency significantly improved. Specific manifestations include:

1. **Doubled Task Processing Capability**: Previously one Agent needed to handle multiple tasks simultaneously, now they can be processed in parallel, throughput more than doubled
2. **More Stable Output Quality**: Each Agent only focuses on tasks it's good at, resulting in higher consistency and quality of output
3. **Reduced Maintenance Costs**: Unified interface and configuration management make the entire system easier to maintain and extend
4. **Simple Agent Addition**: To integrate a new AI product, you only need to implement the interface and add configuration—no need to modify core logic

This solution not only solved HagiCode's own problems, but also proved that multi-Agent collaboration is indeed a viable architectural choice.

## Summary

This article shared HagiCode project's practical experience in multi-Agent collaboration configuration. Core points include:
