## Guide to Implementing Doubao Speech Recognition Hotword Support

> This article will detail how to implement Doubao speech recognition hotword support in the HagiCode project, significantly improving the recognition accuracy of domain-specific vocabulary through both custom hotwords and platform hotword tables.

## Background

Speech recognition technology has developed for many years, but one problem has always troubled developers. While general speech recognition models can cover daily expressions, they fall short when it comes to professional terminology, product names, and personal names—the recognition accuracy just isn't quite there. Think about it: voice assistants in the medical field need to accurately recognize "hypertension," "diabetes," and "coronary heart disease"; legal systems need to precisely capture "case cause," "defense," and "burden of proof"—in these scenarios, well, general models do their best.

In the HagiCode project, we encountered the same challenge. As a versatile AI coding assistant, HagiCode needs to handle speech recognition scenarios for various technical terms. However, the Doubao speech recognition API doesn't fully meet our requirements for professional term accuracy by default. It's not that Doubao isn't good enough, it's just that each field has its own terminology system. After some research and technical exploration, we discovered that the Doubao speech recognition API actually provides hotword support functionality—with simple configuration, it can significantly improve the recognition accuracy of specific vocabulary. It's kind of like telling it what words to pay attention to, and then it listens more carefully for those words.

This article shares the complete solution for implementing Doubao speech recognition hotword functionality in the HagiCode project. Both modes—custom hotwords and platform hotword tables—can be used independently or in combination. Through this solution, developers can flexibly configure hotwords based on business scenarios, enabling the speech recognition system to "recognize" those professional, rare, yet crucial vocabulary terms.

## Core Implementation

### Two Hotword Modes Explained

The Doubao speech recognition API provides us with two hotword configuration methods, each with its unique application scenarios and advantages.

**Custom Hotword Mode** allows us to pass hotword text directly through the `corpus.context` field. This method is perfect for scenarios that require quick configuration of a small number of hotwords, such as temporarily needing to recognize a certain product name or person's name. In HagiCode's implementation, we parse the multi-line hotword text entered by the user into a list of strings, then format it as a `context_data` array according to Doubao API requirements. To put it simply, this approach is very direct—like telling the other party "pay attention to these words," and then it does.

**Platform Hotword Table Mode** references pre-configured hotword tables from the Doubao self-learning platform through the `corpus.boosting_table_id` field. This method is suitable for scenarios that need to manage large numbers of hotwords—we can create and maintain hotword tables on the Doubao self-learning platform, then reference them by ID. For projects like HagiCode that need to continuously update and maintain professional terminology, this mode provides better manageability. After all, when you have many hotwords, having a place to manage them centrally is better than manually entering them every time.

Interestingly, these two modes can also be used in combination. The Doubao API supports including both custom hotwords and platform hotword table IDs in the same request, with the combination strategy controlled through the `combine_mode` parameter. This flexibility enables HagiCode to handle various complex professional term recognition needs. This is also quite good—sometimes combining multiple methods can yield better results.

### Frontend Type Definitions and Validation

In HagiCode's frontend implementation, we defined a complete set of hotword configuration types and validation logic. First, the type definition part:

```typescript
export interface HotwordConfig {
  contextText: string;           // Multi-line hotword text
  boostingTableId: string;      // Doubao platform hotword table ID
  combineMode: boolean;          // Whether to use combination mode
}
```

This simple interface contains all configuration items for the hotword functionality. Among them, `contextText` is the part users experience most intuitively—we allow users to enter one hotword phrase per line, which is very intuitive. After all, letting users enter one word per line is better than making them understand complex configuration rules.

Next is the implementation of validation functions. Considering the Doubao API's limitations, we established strict validation rules: hotword text can have at most 100 lines, each line at most 50 characters, total at most 5000 characters; boosting_table_id at most 200 characters, only allowing letters, numbers, underscores, and hyphens. These limitations aren't something we imagined—they're based on the actual requirements of Doubao's official documentation. After all, API limitations are API limitations, we can't do anything about it except comply.

```typescript
export function validateContextText(contextText: string): HotwordValidationResult {
  if (!contextText || contextText.trim().length === 0) {
    return { isValid: true, errors: [] };
  }

  const lines = contextText.split('\n').filter(line => line.trim().length > 0);
  const errors: string[] = [];

  if (lines.length > 100) {
    errors.push(`Hotword lines cannot exceed 100, currently ${lines.length} lines`);
  }

  const totalChars = contextText.length;
  if (totalChars > 5000) {
    errors.push(`Hotword total characters cannot exceed 5000, currently ${totalChars}`);
  }

  for (let i = 0; i < lines.length; i++) {
    if (lines[i].length > 50) {
      errors.push(`Line ${i + 1} hotword exceeds 50 character limit`);
    }
  }

  return { isValid: errors.length === 0, errors };
}

export function validateBoostingTableId(boostingTableId: string): HotwordValidationResult {
  if (!boostingTableId || boostingTableId.trim().length === 0) {
    return { isValid: true, errors: [] };
  }

  const errors: string[] = [];

  if (boostingTableId.length > 200) {
    errors.push(`boosting_table_id cannot exceed 200 characters, currently ${boostingTableId.length}`);
  }

  if (!/^[a-zA-Z0-9_-]+$/.test(boostingTableId)) {
    errors.push('boosting_table_id can only contain letters, numbers, underscores, and hyphens');
  }

  return { isValid: errors.length === 0, errors };
}
```

These validation functions execute immediately when users configure hotwords, ensuring problems are discovered at the earliest stage. For user experience, this immediate feedback is very important. After all, knowing where the error is while inputting is better than discovering it after submission.

### Frontend Configuration Persistence

In HagiCode's frontend implementation, we chose to use the browser's localStorage to store hotword configuration. Several considerations went behind this design decision: first, hotword configuration is a highly personalized setting, different users may have different professional field requirements; second, this approach simplifies backend implementation, no additional database tables or API interfaces needed; finally, users configure once in the browser, and subsequent uses automatically load the configuration, which is very convenient. To put it plainly, it just saves effort.

```typescript
const HOTWORD_STORAGE_KEYS = {
  contextText: 'hotword-context-text',
  boostingTableId: 'hotword-boosting-table-id',
  combineMode: 'hotword-combine-mode',
} as const;

export const DEFAULT_HOTWORD_CONFIG: HotwordConfig = {
  contextText: '',
  boostingTableId: '',
  combineMode: false,
};

// Load hotword configuration
export function loadHotwordConfig(): HotwordConfig {
  const contextText = localStorage.getItem(HOTWORD_STORAGE_KEYS.contextText) || '';
  const boostingTableId = localStorage.getItem(HOTWORD_STORAGE_KEYS.boostingTableId) || '';
  const combineMode = localStorage.getItem(HOTWORD_STORAGE_KEYS.combineMode) === 'true';

  return { contextText, boostingTableId, combineMode };
}

// Save hotword configuration
export function saveHotwordConfig(config: HotwordConfig): void {
  localStorage.setItem(HOTWORD_STORAGE_KEYS.contextText, config.contextText);
  localStorage.setItem(HOTWORD_STORAGE_KEYS.boostingTableId, config.boostingTableId);
  localStorage.setItem(HOTWORD_STORAGE_KEYS.combineMode, String(config.combineMode));
}
```

The logic of this code is very simple and clear. When loading configuration, read from localStorage; when saving configuration, write to localStorage. We also provide default configuration to ensure the system works normally when there's no configuration at all. After all, there has to be a default value.

The design of this configuration class follows HagiCode's consistent minimalist style. `HotwordContexts` is a nullable list type, `BoostingTableId` is a nullable string, so when there's no hotword configuration, these properties won't affect the request at all. After all, when not in use, they shouldn't exist—that's what being clean means.

### Payload Construction Logic

Payload construction is the core of the entire hotword functionality. Once we have hotword configuration, we need to format it into the correct JSON structure according to Doubao API requirements. This process happens before the SDK sends the request:

```csharp
private void AddCorpusToRequest(Dictionary<string, object> request)
{
    var corpus = new Dictionary<string, object>();

    // Add custom hotwords
    if (Config.HotwordContexts != null && Config.HotwordContexts.Count > 0)
    {
        corpus["context"] = new Dictionary<string, object>
        {
            ["context_type"] = "dialog_ctx",
            ["context_data"] = Config.HotwordContexts
                .Select(text => new Dictionary<string, object> { ["text"] = text })
                .ToList()
        };
    }

    // Add platform hotword table ID
    if (!string.IsNullOrEmpty(Config.BoostingTableId))
    {
        corpus["boosting_table_id"] = Config.BoostingTableId;
    }

    // Only add corpus to request when it's not empty
    if (corpus.Count > 0)
    {
        request["corpus"] = corpus;
    }
}
```

This code shows how to dynamically build the corpus field based on configuration. The key point is: we only add the corpus field when hotword configuration actually exists. This design ensures backward compatibility—when no hotwords are configured, the request structure remains exactly the same as before. After all, compatibility is important, we can't mess up the previous logic just by adding a feature.

### WebSocket Parameter Passing

Between frontend and backend, hotword parameters are passed through WebSocket control messages. HagiCode's design is: the frontend loads hotword configuration from localStorage when starting recording, then sends it to the backend via WebSocket messages.

```typescript
const controlMessage = {
  type: 'control',
  payload: {
    command: 'StartRecognition',
    contextText: '高血压\n糖尿病\n冠心病',
    boosting_table_id: 'medical_table',
    combineMode: false
  }
};
```

One detail to note here: the frontend passes multi-line text (separated by newline characters), and the backend needs to parse it. The backend's WebSocket Handler will parse these parameters and pass them to the SDK:

```csharp
private async Task HandleControlMessageAsync(
    string connectionId,
    DoubaoSession session,
    ControlMessage message)
{
    if (message.Payload is SessionControlRequest controlRequest)
    {
        // Parse hotword parameters
        string? contextText = controlRequest.ContextText;
        string? boostingTableId = controlRequest.BoostingTableId;
        bool? combineMode = controlRequest.CombineMode;

        // Parse multi-line text into hotword list
        if (!string.IsNullOrEmpty(contextText))
        {
            var hotwords = contextText
                .Split('\n', StringSplitOptions.RemoveEmptyEntries)
                .Select(s => s.Trim())
                .Where(s => s.Length > 0)
                .ToList();

            session.HotwordContexts = hotwords;
        }

        session.ostingTableId = boostingTableId;
    }
}
```

Through this design, the passing of hotword configuration from frontend to backend becomes clear and efficient. Actually, there's nothing special about it, just passing it down layer by layer.

## Practice Guide

### Configuring Custom Hotwords

In actual use, configuring custom hotwords is very simple. Open HagiCode's speech recognition settings page and find the "Hotword Configuration" area. In the "Custom Hotword Text" input box, enter one hotword phrase per line.

For example, if you're developing a medical-related application, you can configure it like this:

```
高血压
糖尿病
冠心病
心绞痛
心肌梗死
心力衰竭
```

After saving the configuration, these hotwords will be automatically passed to the Doubao API every time speech recognition starts. Actual testing shows that after configuring hotwords, the recognition accuracy of relevant professional terms has significantly improved. To put it simply, there's definitely an effect, at least much better than before.

### Configuring Platform Hotword Tables

If you need to manage a large number of hotwords, or hotwords need frequent updates, then the platform hotword table mode is more suitable for you. First, you need to create a hotword table on the Doubao self-learning platform, get the generated boosting_table_id, then enter this ID in HagiCode's settings page.

The Doubao self-learning platform provides features like batch import and categorized management of hotwords, which is very practical for teams that need to manage large numbers of professional terms. Managing hotwords through the platform enables centralized maintenance and unified updates. After all, when you have many hotwords, having a place to manage them centrally is better than manually entering them every time.

### Using Combination Mode

In some complex scenarios, you may need to use both custom hotwords and platform hotword tables simultaneously. In this case, just configure both types of hotwords in HagiCode and enable the "Combination Mode" switch.

In combination mode, the Doubao API considers both hotword sources, and recognition accuracy is usually higher than using either method alone. However, note that combination mode increases request complexity, so it's recommended to test before deciding whether to enable it. After all, complexity has increased, whether it's really worth it depends on actual results.

## Important Considerations

When implementing and using hotword functionality, there are several points that need special attention.

First are character limitations. The Doubao API has strict limits on hotwords, including line count, characters per line, total characters, etc. If limits are exceeded, the API returns an error. In HagiCode's frontend implementation, we check during user input through validation functions, avoiding sending invalid configuration to the backend. After all, discovering problems early is better than waiting for API errors.

Second is the format of boosting_table_id. This field only allows letters, numbers, underscores, and hyphens, and cannot contain spaces or other special characters. When creating hotword tables on the Doubao self-learning platform, attention needs to be paid to naming conventions. This is actually understandable—APIs always have relatively strict format requirements.

Third is backward compatibility. Hotword parameters are completely optional; when no hotwords are configured, the system works exactly as before. This design ensures existing users aren't affected at all, and facilitates gradual migration and upgrade. After all, we can't mess up previous logic just by adding a feature.

Finally is error handling. When hotword configuration is invalid, the Doubao API returns corresponding error information. HagiCode's implementation logs detailed information to help developers troubleshoot problems. At the same time, the frontend also displays validation errors in the interface, helping users correct configuration. Good error handling naturally leads to good user experience.
