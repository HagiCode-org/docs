## Technology Selection and Exploration

Initially, we discussed multiple integration approaches during the Proposal phase. Since we use Starlight (a documentation framework based on Astro), the most intuitive idea was to leverage Astro's Hooks.

We first tried modifying `astro.config.mjs`, planning to inject the Clarity script at build time. Although this approach guarantees global coverage, it lacks flexibility—we cannot dynamically load or unload scripts based on user preferences.

Considering user experience and privacy control, we finally decided to adopt the **component override** approach. Starlight allows developers to override its internal components, which means we can take over the rendering logic of `<footer>` or `<head>`, thereby finely controlling the timing of Clarity loading.

Here's a small episode: originally we wanted to create a layout wrapper called `StarlightWrapper.astro`. But during actual debugging, we discovered that Starlight's routing mechanism doesn't automatically call this custom Wrapper, which caused the script to fail on some pages. This counts as a typical "taking it for granted" pitfall, reminding us that **we must deeply understand the framework's rendering process, rather than blindly applying general framework patterns**.

## Core Solution: Footer Component Override

To ensure that the Clarity script loads on all pages (including documentation and blogs) without breaking the original page structure, we chose to override Starlight's `Footer` component.

### Why Footer?

1.  **Global**: Footer appears on almost all standard pages.
2.  **Non-intrusive**: Placing the script in the Footer area (actually rendered at the bottom of the body) won't block the page's critical rendering path (LCP), minimizing performance impact.
3.  **Centralized logic**: Cookie consent logic can be handled uniformly within the component.

## Experience Summary and Pitfalls

In the process of implementing this solution in HagiCode, we summarized several easily overlooked details:

1.  **`StarlightWrapper.astro` is a trap**:
    As mentioned above, don't try to create a global Wrapper to inject scripts. This doesn't work in Starlight. Properly overriding specific components (such as `StarlightFooter.astro` or `StarlightHead.astro`) is the right approach.

2.  **Performance considerations for script placement**:
    Although Clarity recommends placing it in `<head>` to ensure data accuracy, for documentation sites, first-screen loading speed (LCP) directly affects SEO and user retention. We chose to place it in the Footer (bottom of Body), which will slightly lose a very small amount of data from "bounce" users, but in exchange for faster page loading experience. This is a worthwhile trade-off.

3.  **Interference from development environment**:
    Be sure to add `import.meta.env.PROD` judgment. In development mode, you will refresh pages frequently, which will generate a lot of meaningless test data and pollute your Clarity dashboard.
