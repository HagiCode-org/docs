## The Practice of Integrating Pi Agent into HagiCode

> This article shares our practical experience in integrating Pi agent as a first-class workflow entry point in the HagiCode project, including architecture design, core component implementation, and frontend-backend adaptation details.

## Background

Actually, everything started from that question: in the HagiCode Mono project, although `repos/Hagicode.Libs` has already implemented a reusable `PiProvider`, `repos/hagicode-core` and `repos/web` haven't yet elevated Pi to a project-level first-class Agent CLI. This is like having a good pair of shoes but not having tied the shoelaces - you can walk, but it feels like something is missing.

The existing `PiProvider` is located at `HagiCode.Libs/src/HagiCode.Libs.Providers/Pi/PiProvider.cs`. It implements a CLI communication protocol based on line-delimited JSON (`--mode json --print`), supporting session management, tool calling, and streaming output. This code is well-written, it's just lying there, not yet awakened.

This situation created a gap: Pi's underlying capabilities are complete, but the project-level integration chain (from user configuration to execution monitoring) is missing. This is like painting a good picture but not hanging it on the wall for people to appreciate. So we need to integrate Pi as a new active provider `PiCli` into the entire system, making it a first-class workflow entry point like other Agent CLIs.

After all, we want a complete experience, not fragmented pieces.

## About HagiCode

The solution shared in this article comes from our practical experience in the [HagiCode](https://hagicode.com) project. HagiCode is an intelligent code assistant project. During development, we need to integrate multiple AI capability providers. The Pi agent integration solution described in this article is a reusable integration pattern we summarized from practice, which has reference value for similar multi-provider integration scenarios. The project's GitHub address is [github.com/HagiCode-org/site](https://github.com/HagiCode-org/site).

## Architecture Design

Pi agent integration adopts the **thin adapter pattern**, instead of re-implementing the Pi process protocol in the core layer. This design decision isn't actually complicated - just thought, why reinvent the wheel?

After all:

1. **Avoid duplicate implementation**: The parameter building, process startup, JSON parsing, and error handling logic in `PiProvider` are already complete. Re-implementing would create two sources of behavior and two test matrices. That's one thing, but maintenance would be troublesome.
2. **Maintain consistency**: Keep consistent with the integration methods of existing CLIs like Kimi, Gemini, Reasonix, and DeepAgents, all delegating to the libs layer implementation through thin adapter. Everyone does this, just follow along.
3. **Separation of concerns**: `hagicode-core` focuses on runtime contracts and business logic, while process details are left to `HagiCode.Libs` to handle. Each does its job, isn't that nice?

This design allows HagiCode to quickly integrate new AI capability providers while keeping the core layer concise. After all, simplicity is beauty, efficiency is money.

## Core Component Implementation

### Provider Enum and Factory

Added `PiCli = 13` enum value in `hagicode-core/src/PCode.Models/AIProviderType.cs`:

```csharp
public enum AIProviderType
{
    ClaudeCodeCli = 0,
    // ... other providers
    PiCli = 13,
}
```

This enum value is the root of provider identity, affecting the OpenAPI-generated `PCode_Models_AIProviderType.ts` frontend enum, the creation branch of `AIProviderFactory`, and Provider parsing and active provider determination logic. Register the new creation branch in `AIProviderFactory`:

```csharp
case AIProviderType.PiCli:
    provider = new PiCliProvider(
        logger,
        configuration,
        providerFactory.GetRequiredService<ICliProvider<PiOptions>>(),
        providerFactory.GetService<IAgentCliRuntimeEnvironmentResolver>());
    break;
```

Actually, these code snippets aren't anything special, just an enum and a switch case. But they're important, like notes on sheet music, added one by one to play a complete piece.

### Thin Adapter Implementation

`PiCliProvider.cs` is the core thin adapter, implementing the `IAIProvider`, `IVersionedAIProvider`, and `IAsyncDisposable` interfaces. It receives `ICliProvider<PiOptions>` (from `HagiCode.Libs`) through the constructor, maps `AIRequest` / `ProviderConfiguration` to `PiOptions`, and then delegates execution, ping, version query, and other operations to the underlying provider.

The key is to handle Pi-specific JSON event streams, including `assistant.thought`, `assistant`, `terminal.completed`, and other events. These events need to be correctly parsed and converted to the system's standard format during streaming output. This is a bit like translation - translating one language to another, the meaning must be conveyed accurately.

## Frontend Adaptation

### Executor Type Adaptation

The frontend needs to update `executorTypeAdapter.ts`, adding Pi's type recognition logic:

```typescript
const PI_IDS = new Set<string>([
  PCode_Models_AIProviderType.PI_CLI,
  'Pi',
  'PiCli',
  'pi',
  'picli',
  'pi-cli',
]);

export function isPi(value: string): boolean {
  const normalized = normalize(value);
  const normalizedLower = normalized.toLowerCase();
  return PI_IDS.has(normalized)
    || normalizedLower.includes('pi-cli')
    || normalizedLower.includes('picli')
    || normalizedLower === 'pi';
}
```

This is like giving Pi several names. No matter how you call it, it knows you're calling it. After all, what name you use isn't important, what's important is knowing who you are.

### Localization Copy and Forms

Add Pi-related translations in `locales/*/common/{hero,settings}.yml`, and add a configuration field section for Pi in `HeroCliEquipmentForm.tsx`, supporting binary, provider, thinking, sessionDirectory, and tool/session switch fields.

The first version of Pi only exposes minimally necessary fields. Complex features like tool allowlist/denylist and environment variable editors are explicitly postponed to subsequent changes. After all, you can't get fat in one bite, it's faster to take it slow.

## Common Issue Troubleshooting

### Pi Executable Not Found

If the health check returns "Pi executable was not found.", you need to check if pi is in PATH, or confirm the configured path is correct. The solution is to ensure `pi` is installed and in PATH, or configure the correct `ExecutablePath` in `appsettings.yml`.

This is like not being able to find your house key - you have to wonder if you put it in the wrong place. Actually, the solution is quite simple - either put the key back in its original place, or change the lock.

### Configuration Fields Not Recognized

If an error "PiCli runtime settings [...] are not supported" is thrown at startup, check if you're only using configuration fields supported in the first version. Fields supported in the first version include: `provider`, `thinking`, `sessionDirectory`, `noSession`, `disableAllTools`, `disableBuiltinTools`.

Sometimes it's just greed - wanting too many features, but the system doesn't support them. Actually, the features in the first version are already sufficient. Greedy people bite off more than they can chew.

### Unable to Select Pi in Frontend

If there's no Pi option in the Hero editor, check if you've run `npm run generate:api` to regenerate frontend enums, if there's a `profession-pi` entry in `hero.ts`, and if localization copy has been correctly added.

Troubleshooting is like looking for lost items - you have to go step by step. After all, blind searching won't find it, you need to search logically.

## Best Practices

1. **Use thin adapter pattern**: Don't re-implement process protocols in the core layer, delegate to the libs layer provider. This avoids duplicate implementation and maintains code consistency. After all, reinventing the wheel is not only tiring but also prone to problems.

2. **Maintain naming consistency**: Use unified naming conventions across frontend and backend to avoid confusion. Provider enum uses `PiCli`, CLI ID uses `"pi"`, display name uses `"Pi"`. Good naming means lower communication costs.

3. **Prioritize using presets**: The first version should be based on the `profession-pi` preset, rather than requiring manual user configuration. This allows users to get started quickly and reduces configuration complexity. Users like simple things; let them come to me for complex ones.

4. **Focus on error messages**: Ensure error messages are clear and actionable, helping users quickly locate problems. Clear error messages mean users won't panic over an error.

5. **Version compatibility**: Considering the serialization stability of `AIProviderType` enum values, changes need to be handled cautiously. The enum value of `AIProviderType.PiCli = 13` cannot be easily modified. After all, changing this value might break backward compatibility, which would be troublesome.

## Summary

Through the thin adapter pattern, we successfully integrated Pi agent into the HagiCode system, making it a first-class workflow entry point like other Agent CLIs. The core advantages of this solution are:

- Avoided duplicate implementation, reused the existing `PiProvider` from the libs layer
- Maintained consistent integration methods with existing providers, reducing maintenance costs
- Implemented the complete chain from user configuration to execution monitoring

HagiCode's practice proves that the thin adapter pattern is an effective solution for integrating AI capability providers. It allows us to quickly support new agents while maintaining system stability and maintainability.
