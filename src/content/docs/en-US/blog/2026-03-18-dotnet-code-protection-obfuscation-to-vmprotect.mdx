## .NET Code Protection in Practice: From Obfuscation to Virtual Machine Protection

> This article introduces how to implement multi-layer code protection strategies in .NET projects, covering all solutions from basic obfuscation to professional virtual machine protection.

## Background

In .NET application development, protecting core code (such as license verification, business logic, sensitive configuration, etc.) from decompilation and reverse analysis is, well, an unavoidable topic. With the maturity of the .NET ecosystem, developers have various code protection methods, from built-in obfuscation attributes to professional virtualization protection tools—there are quite a few choices.

As a complex multi-language monorepo project, HagiCode includes desktop applications, build systems, and license management functionality. The code inevitably involves license verification logic, sensitive configurations (such as API keys, product IDs), and core business logic. These things still need to be properly protected—after all, no one wants their hard work to be easily seen by others.

This article will share the code protection solutions we actually adopted in the HagiCode project, summarizing the complete process from trial and error to optimization. Perhaps it can give you some inspiration.

## About HagiCode

[HagiCode](https://hagicode.com) is an open-source AI code assistant project dedicated to providing developers with intelligent programming assistance. The project adopts a monorepo architecture, maintaining multiple components including VSCode extensions, backend AI services, and cross-platform desktop clients. This multi-language, multi-platform complexity makes code protection an engineering challenge that must be faced—there's no way around it, given how complex the project is.

The solution shared in this article is exactly what we actually developed through trial and error during HagiCode development. If you want to know how we solved these technical problems, please continue reading—perhaps there will be some unexpected insights.

## Core Content

### 1. Microsoft Built-in Obfuscation Attribute

.NET Framework provides a built-in `[ObfuscationAttribute]`, which is the most basic and commonly used code obfuscation marker. This attribute is located in the `System.Reflection` namespace and can provide basic code protection without introducing third-party tools—it's quite convenient.

**Core Features**:
- `Feature` property: Specifies obfuscation features, such as `"ultra"` (high obfuscation), `"all"` (full obfuscation)
- `Exclude` property: `true` means exclude from obfuscation, `false` means apply obfuscation
- Can be applied to classes, methods, properties, and other type members

In the HagiCode project, you can see actual usage examples:

```csharp
[Obfuscation(Feature = "ultra", Exclude = false)]
public async Task<LicenseValidationResult?> ValidateLicenseAsync(...)
```

The **advantages** of this approach are quite obvious:
- No additional dependencies needed, comes with .NET Framework, saves a lot of trouble
- Can be recognized and processed by third-party obfuscation tools
- Does not significantly increase the size of compiled assemblies

However, it also has limitations: it only serves as a marker, actual obfuscation effectiveness depends on tool implementation, and cannot provide virtual machine protection-level security—that's fine, after all, it wasn't designed for that.

### 2. VMP (Virtual Machine Protection)

VMP is a professional code protection tool that provides high-level protection by compiling code into virtual machine instructions. Unlike simple name obfuscation, VMP truly converts code logic into a form that cannot be restored by conventional decompilers—this is quite impressive.

**Protection Level Classification**:

| Level | Virtualization | Mutation | Anti-Debug | String Encryption | Applicable Scenarios |
|------|----------------|----------|------------|-------------------|----------------------|
| HIGH | full | high | Enabled | Enabled | License verification, session concurrency, sensitive constants |
| MEDIUM | partial | medium | Enabled | Enabled | Business logic, domain models |
| LOW | none | low | Disabled | Disabled | Utility classes, non-critical code |

The HagiCode project defines a declarative attribute system to mark code that needs protection:

```csharp
// High priority protection
[VmProtect(VmProtectionPriority.High, Reason = "Contains license verification logic")]
public class KeygenClient { ... }

// Exclude protection
[VmExclude(Reason = "Public API that must remain unchanged")]
public class PublicApi { ... }

// Inherited protection
[VmProtect(Priority.High, ProtectDerived = true)]
public class BaseLicenseValidator { ... }
```

### 3. Build-time Protection Strategy

VMP protection not only takes effect at runtime but also needs to be automated in the build process—after all, doing it manually is too troublesome. HagiCode's build system supports multiple modes:

- **Windows native mode**: Directly calls VMProtect tools
- **Linux Docker container mode**: Runs VMP in containers (solves cross-platform compatibility issues)
- **Attribute scanning**: Automatically discovers protection markers in code
- **Verification mechanism**: Confirms protection has been successfully applied

When combined, these features are quite reassuring.

## Solution

### 1. Using Microsoft Built-in Obfuscation Attribute

Apply `ObfuscationAttribute` directly in code:

```csharp
using System.Reflection;

[Obfuscation(Feature = "ultra", Exclude = false)]
public class LicenseService
{
    [Obfuscation(Feature = "ultra", Exclude = false)]
    public async Task<bool> ValidateLicenseAsync(string key)
    {
        // License verification logic
    }

    [Obfuscation(Feature = "flow", Exclude = false)]
    private string DecryptToken(string encrypted)
    {
        // Decryption logic
    }
}
```

Sometimes you need to allow test assemblies to access internal members while maintaining production code security:

```csharp
// AssemblyInfo.cs
[assembly: InternalsVisibleTo("HagiCode.Application.Tests")]
[assembly: InternalsVisibleTo("DynamicProxyGenAssembly2")] // for Moq
```

This makes testing much more convenient—after all, code still needs to be tested.

### 2. Custom Attribute Definition for VMP Protection

Create custom protection attributes to control VMP behavior:

```csharp
using System;

[AttributeUsage(AttributeTargets.Class | AttributeTargets.Method | AttributeTargets.Property)]
public class VmProtectAttribute : Attribute
{
    public VmProtectionPriority Priority { get; set; }
    public string? Reason { get; set; }
    public bool ProtectDerived { get; set; }
}

[AttributeUsage(AttributeTargets.Class | AttributeTargets.Method | AttributeTargets.Property)]
public class VmExcludeAttribute : Attribute
{
    public string? Reason { get; set; }
}

public enum VmProtectionPriority
{
    None = 0,
    Low = 1,
    Medium = 2,
    High = 3
}
```

Self-defined attributes are also more convenient to use—after all, you understand your own needs.

Writing the configuration clearly makes later maintenance more convenient.

## Practice Guide

### 1. Key Component Protection Practices

According to HagiCode's code-protection specification, the following components must use HIGH priority protection:

```csharp
// Production constants - must be encrypted and protected by VMP
[VmProtect(VmProtectionPriority.High, Reason = "Production constants")]
public static class ProductionConstants
{
    // Encrypted string accessor, protected by VMP
    [VmProtect(VmProtectionPriority.High)]
    public static string GetLicenseServerUrl(IOptions<LicenseOptions> options) => ...;
}

// License verification logic
[VmProtect(VmProtectionPriority.High, Reason = "License verification logic")]
public class KeygenClient : IKeygenClient
{
    [Obfuscation(Feature = "ultra", Exclude = false)]
    public async Task<LicenseValidationResult?> ValidateLicenseAsync(...) { ... }
}

// Machine fingerprint service
[VmProtect(VmProtectionPriority.High)]
public class MachineFingerprintService : IMachineFingerprintService { ... }
```

Critical code still needs focused protection—after all, core logic leakage would be troublesome.

### 2. Runtime Decryption of String Encryption

Encrypt strings at build time, decrypt at runtime:

```csharp
public static class StringDecryption
{
    [VmProtect(VmProtectionPriority.High, Reason = "CRITICAL SECURITY")]
    public static string DecryptString(byte[] encryptedData, byte[] key, byte[] iv)
    {
        using var aes = Aes.Create();
        aes.Key = key;
        aes.IV = iv;

        using var decryptor = aes.CreateDecryptor();
        using var ms = new MemoryStream(encryptedData);
        using var cs = new CryptoStream(ms, decryptor, CryptoStreamMode.Read);
        using var reader = new StreamReader(cs);

        return reader.ReadToEnd();
    }
}

// Production constants accessor (lazy loading + caching)
public static class ProductionConstants
{
    private static string? _cachedLicenseServerUrl;

    public static string GetLicenseServerUrl(IOptions<LicenseOptions> options)
    {
        if (_cachedLicenseServerUrl == null)
        {
            var encrypted = GetEncryptedLicenseServerUrl();
#if DEBUG
            _cachedLicenseServerUrl = options.Value.PrimaryServer.Url;
#else
            _cachedLicenseServerUrl = StringDecryption.DecryptString(
                encrypted,
                GetEncryptionKey(),
                GetEncryptionIV());
#endif
        }
        return _cachedLicenseServerUrl;
    }
}
```

String encryption is also quite important—after all, sensitive information cannot be stored in plain text.

### 3. VMP Protection Verification

After building, must verify that protection was successfully applied—otherwise how do you know the protection is effective:

```csharp
// Verification script example
public bool VerifyProtection(string assemblyPath)
{
    // 1. Check VMP signature
    var bytes = File.ReadAllBytes(assemblyPath);
    var vmpSignature = Encoding.ASCII.GetBytes("VMProtect");
    if (bytes.Any(b => vmpSignature.Contains(b)))
    {
        return true;
    }

    // 2. Check file size change (usually increases after protection)
    var originalInfo = new FileInfo(assemblyPath.Replace(".dll", ".bak"));
    if (originalInfo.Exists)
    {
        var sizeRatio = (double)new FileInfo(assemblyPath).Length / originalInfo.Length;
        return sizeRatio > 1.1;
    }

    return false;
}
```

Verifying is always good—otherwise you might not know when something goes wrong.

### 4. Precautions

Here are a few pitfalls to pay special attention to—after all, we learned them the hard way:

1. **Don't obfuscate all code**: Public APIs, interface definitions, DTO classes usually don't need protection. Excessive obfuscation affects performance and debugging. The HagiCode project suffered from this—pay attention to this.

2. **Protect key accessors**: Methods for obtaining encryption keys must enjoy the same or higher level of protection as encrypted data. Otherwise, it's just window dressing and meaningless.

3. **Balance between testing and production**: DEBUG builds should skip encryption for development and debugging convenience, while RELEASE builds enable full protection. Remember to use conditional compilation `#if DEBUG` to distinguish—this makes development more convenient.

4. **Docker environment considerations**: Running VMP in Linux environments requires containerized solutions to ensure tool compatibility. HagiCode uses a Wine + VMP container solution—cross-platform issues are solved this way.

5. **Verification is essential**: After building, must verify that protection was successful. Otherwise, sensitive code could be exposed. The verification code above serves this purpose—better to check.

## Summary

Through this multi-layer protection strategy, HagiCode achieves a comprehensive code security system from basic obfuscation to virtual machine protection:

- **First layer**: Use `ObfuscationAttribute` for basic marking, providing hints for third-party tools
- **Second layer**: Declare protection intent and priority through custom `VmProtectAttribute`
- **Third layer**: VMP virtual machine protection converts critical code into irreversible virtual machine instructions
- **Fourth layer**: Automatically scan and apply protection at build time, verify protection results

This solution can withstand both ordinary decompilation tools and advanced reverse analysis attacks. If you're also developing .NET applications that need code protection, I hope this gives you some reference—even a little inspiration is enough.
