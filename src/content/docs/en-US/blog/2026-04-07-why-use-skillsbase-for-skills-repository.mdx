## Why Use Skillsbase to Maintain Your Skills Collection Repository

> It's kind of funny—the era of AI programming is here, we have more and more Agent Skills in hand, but with that comes more and more trouble. This article is about how we use skillsbase to solve these problems.

## Background

In the AI programming era, developers need to maintain more and more Agent Skills—these are reusable instruction sets for extending the capabilities of coding assistants like Claude Code, OpenCode, Cursor, and more. However, as the number of skills grows, a practical issue gradually emerges:

Actually, it's not really a big problem, it's just that when you have more stuff, managing it becomes troublesome.

### Skills Are Scattered, High Management Overhead

- Local skills are scattered across multiple locations: `~/.agents/skills/`, `~/.claude/skills/`, `~/.codex/skills/.system/`, etc.
- Different locations may have naming conflicts (e.g., `skill-creator` exists in both user and system directories)
- Lack of unified management entry point, making backup and migration difficult

This is quite annoying. Sometimes you don't even know where a particular skill is, like losing something, it's hard to find.

### Lack of Standardized Maintenance Process

- Manual copying of skills is error-prone and difficult to trace sources
- No unified validation mechanism, unable to guarantee the integrity of the skills repository
- During team collaboration, difficult to synchronize and share skill collections

Manual operations are always error-prone. After all, human memory is limited—who can remember where so many things came from?

### Unable to Meet Reproducibility Requirements

- When switching development machines, need to reconfigure all skills
- Unable to automatically verify and synchronize skills repository in CI/CD environments

Switching computers means starting all over again—how should I put it, it's as troublesome as moving house. Every time you have to adapt to a new environment and reconfigure everything.

To solve these pain points, we tried various solutions: from manual copying to script automation, from direct directory management to global installation and then cleanup. Each solution had its own defects—either unable to guarantee consistency, or polluting the environment, or difficult to use in CI.

Actually, we took quite a few detours.

Eventually, we found a more elegant solution—skillsbase. The core idea of this solution is: first install locally for validation, then transform the structure and write to the repository, finally uninstall temporary files. This ensures repository content matches actual installation results while not polluting the global environment.

Sounds simple, but it took quite a few pitfalls to figure out.

## About HagiCode

The solution shared in this article comes from our practical experience in the [HagiCode](https://hagicode.com) project.

HagiCode is an AI coding assistant project. During development, we need to maintain a large number of Agent Skills to extend various coding capabilities. It was these actual needs that drove us to develop the skillsbase toolset to standardize skills repository management.

Actually, this wasn't thought up out of thin air—we were forced into it. When you have more skills, you naturally need to manage them. When you encounter problems during management, you naturally need to solve them. Step by step, we've reached where we are today.

If you're interested in HagiCode, you can visit the [official website to learn more](https://hagicode.com) or check the source code on [GitHub](https://github.com/HagiCode-org/site).

## Analysis

### Technical Challenges

To build a maintainable skills collection repository, the following core issues need to be addressed:

1. **Unified namespace conflicts**: When multiple sources have skills with the same name, how do we avoid overwriting?
2. **Source traceability**: How to record the source of each skill for future updates and auditing?
3. **Synchronization and validation**: How to ensure repository content matches actual installation results?
4. **Automation integration**: How to integrate with CI/CD workflows for automatic synchronization and validation?

These problems seem simple, but each one is enough to give you a headache. But then again, what isn't difficult?

### Design Trade-offs

**Option 1: Direct Directory Copy**

Pros: Simple implementation
Cons: Cannot guarantee consistency with actual `skills` CLI installation results

We actually considered this approach. But later we discovered that the CLI might have some preprocessing logic during installation, which direct copying would skip. The result is that what you copy differs from what actually gets installed—problematic.

**Option 2: Global Installation Then Cleanup**

Pros: Can validate installation process
Cons: Pollutes execution environment, difficult to maintain consistency between CI and local results

This approach is even worse. Global installation pollutes the environment. Even more troublesome is that CI environments and local environments are difficult to keep consistent, leading to "works locally, fails in CI" problems. Who understands this feeling?

**Option 3: Local Install → Transform → Uninstall (Final Solution)**

This is the solution adopted by skillsbase:

- First use `npx skills` to install skills to a temporary location
- Transform directory structure and add source metadata
- Write to target repository
- Finally uninstall temporary files

This approach ensures repository content matches actual consumer installation results while not polluting the global environment. The transformation process can be standardized and supports idempotent operations.

Actually, we didn't think of this solution from the start either—it was only after many trial and errors that we naturally learned what works and what doesn't.

### Architecture Decisions

| Decision | Choice | Rationale |
|----------|--------|-----------|
| Runtime | Node.js ESM | No build step needed, `.mjs` is sufficient for file system orchestration |
| Configuration format | YAML (`sources.yaml`) | Highly readable, supports manual maintenance |
| Naming strategy | Namespace prefix | User skills keep original names, system skills get `system-` prefix |
| Workflow | `add` modifies manifest → `sync` executes synchronization | Single sync engine, avoiding duplicate rule implementation |
| File management | Managed file标识 | Add comment headers, support safe overwriting |

These decisions, when it comes down to it, are all for one goal: making things simple. After all, simplicity is king.

## Solution

### CLI Architecture

The `skillsbase` CLI provides four core commands:

```text
skillsbase
├── init          # Initialize repository structure
├── sync          # Sync skills content
├── add           # Add new skills
└── github_action # Generate GitHub Actions configuration
```

Not many commands, but enough. After all, when it comes to tools, enough is good.

### Core Workflow

```
┌─────────────┐    ┌─────────────┐    ┌─────────────┐    ┌─────────────┐
│    init     │───▶│    add      │───▶│    sync     │───▶│github_action│
│ Initialize  │    │ Add source  │    │ Sync content│    │ Generate CI │
└─────────────┘    └─────────────┘    └─────────────┘    └─────────────┘
```

Step by step, no rush.

### Sync Flow Design

```
sources.yaml → Parse sources → npx skills install → Transform structure → Write to skills/ → Uninstall temp files
                              ↓
                        .skill-source.json (Source metadata)
```

This flow design is fairly clear. At least when I look at it myself, I understand what each step is doing.

### Repository Structure

```
repos/skillsbase/
├── sources.yaml              # Source manifest (single source of truth)
├── skills/                   # Skills directory
│   ├── frontend-design/      # User skill
│   ├── skill-creator/        # User skill
│   └── system-skill-creator/ # System skill (with prefix)
├── scripts/
│   ├── sync-skills.mjs       # Sync script
│   └── validate-skills.mjs   # Validation script
├── docs/
│   └── maintainer-workflow.md # Maintainer documentation
└── .github/
    ├── workflows/
    │   └── skills-sync.yml   # CI workflow
    └── actions/
        └── skillsbase-sync/
            └── action.yml     # Reusable Action
```

A lot of files, but still okay. After all, with clear organizational structure, maintenance becomes easier.

## Practice

### Initialize Skills Repository

```bash
# 1. Create empty repository
mkdir repos/myskills && cd repos/myskills
git init

# 2. Initialize with skillsbase
npx skillsbase init

# Output:
# [1/4] create manifest ................. done
# [2/4] create scripts .................. done
# [3/4] create docs ..................... done
# [4/4] create github workflow .......... done
#
# next: skillsbase add <skill-name>
```

This step generates a bunch of files, but don't worry—they're all auto-generated. Next, you can start adding skills.

### Add Skills

```bash
# Add a single skill (automatically executes sync)
npx skillsbase add frontend-design --source vercel-labs/agent-skills

# Add to local source
npx skillsbase add documentation-writer --source /home/user/.agents/skills

# Output:
# source: first-party ......... updated
# target: skills/frontend-design ... synced
# status: 1 skill added, 0 removed
```

Adding skills is quite simple, just one command. Sometimes you encounter unexpected situations like poor network or permission issues, but these are minor issues—take it slow.

### Sync Skills

```bash
# Execute sync (reconcile all sources)
npx skillsbase sync

# Only check for drift (don't modify files)
npx skillsbase sync --check

# Allow missing sources (CI scenario)
npx skillsbase sync --allow-missing-sources
```

During sync, the system checks all sources defined in `sources.yaml` and reconciles them with the content in the `skills/` directory. Update if there are differences, skip if there aren't. This prevents "configuration changed but files didn't" problems.

### Generate CI Configuration

```bash
# Generate workflow
npx skillsbase github_action --kind workflow

# Generate action
npx skillsbase github_action --kind action

# Generate all
npx skillsbase github_action --kind all
```

CI configuration is also auto-generated. You just need to adjust some details yourself, like trigger conditions, runtime environment, etc. But these aren't difficult.

### sources.yaml Configuration Example

```yaml
# Skills root directory configuration
skillsRoot: skills/
metadataFile: .skill-source.json

# Source definitions
sources:
  # First-party: local user skills
  first-party:
    type: local
    path: /home/user/.agents/skills
    naming: original  # Keep original name
    includes:
      - documentation-writer
      - frontend-design
      - skill-creator

  # System: system-provided skills
  system:
    type: local
    path: /home/user/.codex/skills/.system
    naming: prefix-system  # Add system- prefix
    includes:
      - imagegen
      - openai-docs
      - skill-creator  # Will become system-skill-creator

  # Remote: third-party repository
  vercel:
    type: remote
    url: vercel-labs/agent-skills
    naming: original
    includes:
      - web-design-guidelines
```

This configuration file is the core of the entire system. All sources are defined here—change this and it takes effect on the next sync. So this counts as a "single source of truth."

### .skill-source.json Metadata Example

```json
{
  "source": "first-party",
  "originalPath": "/home/user/.agents/skills/documentation-writer",
  "originalName": "documentation-writer",
  "targetName": "documentation-writer",
  "syncedAt": "2026-04-07T00:00:00.000Z",
  "version": "unknown"
}
```

Each skill directory has this file, recording its source information. This way, when problems occur later, you can quickly locate where it came from and when it was synced.

### Security and Validation

```bash
# Validate repository structure
node scripts/validate-skills.mjs

# Validate using skills CLI
npx skills add . --list

# Check for updates
npx skills check
```

Validation—important if you think it is, unnecessary if you don't. But just to be safe, running it occasionally doesn't hurt. After all, who knows what unexpected things might happen?

### GitHub Actions Integration

```yaml
# .github/workflows/skills-sync.yml
name: Skills Sync

on:
  push:
    paths:
      - 'sources.yaml'
      - 'skills/**'
  workflow_dispatch:

jobs:
  validate:
    runs-on: ubuntu-latest
    steps:
      - uses: actions/checkout@v4
      - uses: actions/setup-node@v4
        with:
          node-version: '20'
      - name: Validate repository
        run: |
          npx skills add . --list
          node scripts/validate-skills.mjs
      - name: Sync check
        run: npx skillsbase sync --check
```

After CI integration, every time you modify `sources.yaml` or the `skills/` directory, validation is automatically triggered. This prevents "changed locally but forgot to sync" problems.

### Best Practices

1. **Naming conflict handling**: System skills uniformly get the `system-` prefix. This preserves all skills while avoiding naming conflicts.
2. **Idempotent operations**: All commands support repeated execution—running `sync` multiple times has no side effects. This is especially important in CI.
3. **Managed files**: Generated files contain `# Managed by skillsbase CLI` comments for easy identification and management. These files can be safely overwritten—manual modifications won't be preserved.
4. **Non-interactive mode**: CI environments use deterministic behavior by default and won't be interrupted by interactive prompts. All configuration is declared through `sources.yaml`.
5. **Source traceability**: Each skill has `.skill-source.json` recording source information, enabling quick localization when problems occur.

### Team Collaboration

```bash
# Team members install shared skills repository
npx skills add your-org/myskills -g --all

# Local clone validation
git clone https://github.com/your-org/myskills.git
cd myskills
npx skills add . --list
```

By managing the skills repository through Git, team members can easily synchronize skill collections, ensuring everyone uses the same version of tools and configuration.

This is particularly useful in team collaboration—avoiding "works on my machine but not yours" situations. After all, once the environment is unified, problems are halved.

## Summary

The core value of using `skillsbase` to maintain a skills collection repository lies in:

- **Security**: Source validation, conflict detection, managed file protection
- **Maintainability**: Unified entry point, idempotent operations, configuration as documentation
- **Standardization**: Unified directory structure, naming conventions, metadata format
- **Automation**: CI/CD integration, automatic sync, automatic validation

Through this solution, developers can manage their Agent Skills like managing npm packages, achieving a reproducible, shareable, maintainable skills repository system.

The tools and workflows shared in this article are exactly what we developed through actual pitfalls and optimizations during HagiCode development. If you find this solution valuable, it shows our engineering practice direction is correct—then HagiCode itself is also worth paying attention to.

After all, good tools deserve to be used by more people.
