## VSCode and code-server: Browser-based Code Editing Solution Selection

> When building browser-based code editing capabilities, developers face a critical choice: use VSCode's official `code serve-web` functionality, or adopt the community-driven `code-server` solution? This choice not only affects technical architecture but also relates to license compliance and deployment flexibility.

## Background

Actually, making technology selections is a bit like choosing a life path. Once you choose a path, you have to keep walking on it, and when you want to switch paths, the cost becomes enormous.

In the era of AI-assisted programming, browser-based code editing capabilities are becoming increasingly important. Users expect to open an editor in the same browser session for immediate modifications after an AI assistant analyzes the code, without switching applications. This seamless experience—well, it's like when you need it, it's just there—but sometimes it's偏偏 not there.

Both solutions have their pros and cons, and choosing the wrong one may bring considerable trouble later. ## License Differences (Most Critical)

This is the most fundamental difference between the two solutions and the first factor we considered in selection. After all, in technology selection, the first step is to think clearly about legal risks, otherwise when problems arise later, whom can you blame?

**code-server**
- MIT license, fully open source
- Maintained by Coder.com company, active community
- Can be freely used for commercial purposes, modified, and distributed
- No usage scenario restrictions

**VSCode `code serve-web`**
- Part of Microsoft VSCode product
- Uses Microsoft's license (VS Code license has commercial use restrictions)
- Mainly designed for individual developers
- Enterprise deployment may require additional commercial authorization considerations

From a license perspective, code-server is more friendly to commercial projects. This needs to be clear in the product planning phase, otherwise waiting until scale grows to migrate, the cost becomes enormous. After all, migration—easy to say, hard to do, those who've experienced it know.

## Deployment Differences

After resolving license issues, next comes deployment approach. This directly affects your operational costs and architecture design, and indirectly affects your daily mood—the simpler the deployment, the better the mood, everyone understands this principle.

**code-server**
- Standalone Node.js application, can be deployed independently
- Supports multiple runtime sources:
  - Directly specify executable file path
  - System PATH lookup
  - NVM Node.js 22.x environment auto-detection
- No need to install VSCode desktop version on server
- Containerized deployment is simpler

**VSCode `code serve-web`**
- Must depend on locally installed VSCode CLI
- Requires available `code` command on the local machine
- System will filter out VS Code Remote CLI wrappers
- Mainly designed for local development scenarios

code-server is more suitable for server/container deployment scenarios. If your product needs to run in Docker, or the user environment doesn't have VSCode, then choosing code-server is right. After all, simplicity is beauty; complexity leads to problems, problems need fixing, fixing may introduce new problems, this endless cycle—who wants to experience it?

## Feature Parameter Differences

The two solutions also have some differences in feature parameters. Although not major, they may bring some trouble in actual use. These details are like minor friction in life—not many, but when accumulated, they become annoying.

| Feature | code-server | code serve-web |
|---------|-------------|----------------|
| Public base path | `/` (configurable) | `/vscode-server` (fixed) |
| Authentication | `--auth` parameter, supports multiple modes | `--connection-token` / `--without-connection-token` |
| Data directory | `{DataDir}/code-server` | `{DataDir}/vscode-serve-web` |
| Telemetry | Disabled by default `--disable-telemetry` | Depends on VSCode settings |
| Update check | Can be disabled `--disable-update-check` | Depends on VSCode settings |

These differences need special attention during integration. For example, different URL paths mean frontend code needs targeted handling. All developers know that handling such details takes the most time, but there's no way—without it, it won't run.

## Availability Detection Differences

When implementing editor switching functionality, availability detection logic also differs. This difference is like interpersonal relationship styles—some people prefer directness, others prefer subtlety.

**code-server**
- Always returned as visible implementation
- Even when unavailable, it displays, prompting `install-required` status
- Supports automatic detection of NVM Node.js 22.x environment

**code serve-web**
- Only visible when local `code` CLI is detected
- If unavailable, frontend automatically hides this option
- Depends on local VSCode installation status

This difference directly affects user experience. code-server's approach is more transparent—users know about this option, just not installed yet; code serve-web's approach is more hidden—users may not even know about this choice. Which approach is better? It depends on product positioning—after all, user experience has no standard answer, only suitability or unsuitability.

## HagiCode's Dual Implementation Architecture

After in-depth analysis, the HagiCode project adopted a dual implementation architecture, supporting both solutions at the architectural level. This isn't because we have technology selection difficulty syndrome, but because there are genuine practical needs. After all, in the technology world, there's no absolutely correct choice, only the choice that best suits you.

### Default Choice of code-server

```csharp
// The default active implementation is code-server
// If an explicit activeImplementation is saved, prioritize attempting that implementation
// If the requested implementation is unavailable, the resolver will attempt the other implementation
// If fallback occurs, return fallbackReason
```

We default to code-server, mainly considering license and deployment flexibility. But for users with local VSCode environments, code serve-web is also a good choice. After all, giving users one more choice is always good—I can't force others to accept a single solution.

### Implementation Selector

`CodeServerImplementationResolver` is uniformly responsible for:
- Implementation selection during startup warmup
- Implementation selection during state reading
- Implementation selection during project opening
- Implementation selection during Vault opening

This design allows the system to flexibly handle different scenarios, and users can choose the most suitable implementation according to their environment. Flexible design—takes more time upfront, but saves worry later; after all, no one wants to modify code everywhere.

### Frontend Adaptation Rules

```typescript
// When localCodeAvailable=false, don't display code serve-web
// When localCodeAvailable=true, display code serve-web configuration
```

The frontend automatically displays available options based on environment, avoiding user confusion about unusable features. When users are confused, they come to ask you; too many questions annoy you, annoyance makes you want to modify code, modifying code may introduce bugs, this vicious cycle—who wants to experience it?

## Practice Guide

After so much theory, what should you pay attention to during actual deployment? Actually, no matter how good the theory, if implementation doesn't work, it's useless—after all, practice is the only criterion for testing truth.

### Docker Deployment Recommendations

For containerized deployment, code-server is the better choice:

```dockerfile
# Directly use code-server official image
FROM codercom/code-server:latest

# Or install via npm
RUN npm install -g code-server
```

This one layer handles it, no need to additionally install VSCode. Simple is good, complexity leads to errors—this principle applies everywhere.

### Configuration Examples

**code-server configuration**
```json
{
  "vscodeServer": {
    "enabled": true,
    "activeImplementation": "code-server",
    "codeServer": {
      "host": "0.0.0.0",
      "port": 8080,
      "executablePath": "",
      "authMode": "none"
    }
  }
}
```

**code serve-web configuration**
```json
{
  "vscodeServer": {
    "enabled": true,
    "activeImplementation": "serve-web",
    "serveWeb": {
      "host": "0.0.0.0",
      "port": 8080,
      "executablePath": "/usr/local/bin/code"
    }
  }
}
```

Configuration—troublesome the first time, but after setup, it saves worry later. Like life, invest more upfront, and days become easier later.

### URL Construction Differences

**code-server**
```
http://localhost:8080/?folder=/path/to/project&vscode-lang=en-US
```

**code serve-web**
```
http://localhost:8080/vscode-server/?folder=/path/to/project&tkn=xxx&vscode-lang=en-US
```

Pay attention to path and parameter differences, integration needs separate handling. Details determine success or failure—this couldn't be more true; missing one parameter might prevent the page from opening.

### Switching Implementations

The system supports runtime switching, automatically stopping the old implementation when switching:

```csharp
// VsCodeServerManager automatically handles mutual exclusion
// When switching activeImplementation, the old implementation won't continue running in background
```

This design allows users to try different implementations at any time and find the solution that best suits them. After all, what suits you is best—others' suggestions are only reference, ultimately you have to try yourself.

### Status Monitoring

```typescript
const { settings, runtime } = await getVsCodeServerSettings();

// runtime.activeImplementation: "code-server" | "serve-web"
// runtime.fallbackReason: switching reason
// runtime.status: "running" | "starting" | "stopped" | "unhealthy"
```

With visible status, you know what's going on. Users can quickly locate whether it's a server issue or their own operational issue when encountering problems. When status is unknown, people tend to panic, panic leads to wrong judgments, once this chain starts, it can't be stopped.

## Summary

| Comparison Dimension | code-server | code serve-web | Recommendation |
|---------------------|-------------|----------------|----------------|
| License | MIT (commercial friendly) | Microsoft (restricted) | code-server |
| Deployment Flexibility | Independent deployment | Depends on local VSCode | code-server |
| Server Applicability | Designed for servers | Mainly for local development | code-server |
| Containerization | Native support | Needs VSCode installation | code-server |
| Feature Completeness | Close to desktop version | Official complete version | code serve-web |
| Maintenance Activity | Active community | Microsoft official | Each has advantages |

**Recommended Strategy**: Prioritize code-server, consider code serve-web when complete official functionality is needed and local VSCode environment is available.
