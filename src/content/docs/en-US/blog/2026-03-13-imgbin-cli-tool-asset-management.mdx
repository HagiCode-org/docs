## ImgBin CLI Tool Design: HagiCode Image Asset Management Solution

> This article introduces how to build an automated image asset pipeline from scratch, including CLI tool design, Provider Adapter architecture, and metadata management strategies.

## Background

Actually, I never expected that image asset management would be something that would trouble us for so long.

During the development of the HagiCode project, we encountered a problem that seemed simple but was quite tricky: the generation and management of image assets. How should I put it—it's like those things during adolescence—on the surface it's calm, but underneath there are undercurrents.

As the project documentation and marketing materials increased, a large number of images were needed. Some of these images needed to be AI-generated, some needed to be selected from existing asset libraries, and some needed AI recognition and automatic annotation of existing images. The problem is that these tasks have long been completed using scattered scripts plus manual operations—every time an image is generated, you need to manually execute scripts, manually organize metadata, and manually generate thumbnails. That's one thing, but the key issue is that these scattered things are all over the place—when you want to find them, you can't find them, and when you want to use them, you can't use them.

Specific pain points include:

1. **Lack of unified entry**: Image generation logic is scattered across different scripts, making batch execution impossible
2. **Missing metadata**: Generated images don't have unified metadata.json, making them impossible to retrieve and track
3. **High manual organization cost**: Image titles and tags all need manual organization one by one, resulting in low efficiency
4. **Cannot be automated**: Want to automatically generate images in CI/CD processes? No way.

We also thought about just ignoring it, but after all, we still need to work on the project. Since we can't avoid it, we might as well find a solution. So we decided to upgrade ImgBin from "scattered scripts" to an automated image asset pipeline. After all, for some things, avoidance is not the solution.

## About HagiCode

The solution shared in this article comes from our practical experience in the [HagiCode](https://hagicode.com) project. HagiCode is an AI code assistant project that maintains multiple components including VSCode extensions, backend AI services, and cross-platform desktop clients. In this complex scenario of multiple languages and platforms, standardized management of image assets became a key link in improving development efficiency.

How should I put it, this is also a small annoyance during HagiCode's growth. Every project will have times like this—small problems that seem insignificant can actually make you struggle for a long time.

HagiCode's build system uses the TypeScript + Node.js ecosystem, so ImgBin naturally chose the same tech stack to ensure technical consistency across the entire project. After all, we're used to it, and switching to something else would be troublesome.

## Core Design

### Overall Architecture

ImgBin adopts a layered architecture, clearly separating CLI commands, application services, third-party API adapters, and infrastructure layers:

```
Component Hierarchy
├── CLI Entry (cli.ts)              Global parameter parsing, command routing
├── Commands (commands/*)           generate | batch | annotate | thumbnail
├── Application Services            job-runner | metadata | thumbnail | asset-writer
├── Provider Adapters               image-api-provider | vision-api-provider
└── Infrastructure Layer            config | logger | paths | schema
```

The advantage of this layered design is that each layer has clear responsibilities, and when testing, it's easy to mock out external dependencies. Actually, it's just about letting each do its own thing without interfering with each other, making it easier to find the cause when problems occur, isn't it?

### Single Asset Directory Model

ImgBin adopts a "one asset, one directory" model. Each time an image is generated, the following structure is created:

```
library/
└── 2026-03/
    └── orange-dashboard/
        ├── original.png      # Original image
        ├── thumbnail.webp    # 512x512 thumbnail
        └── metadata.json     # Structured metadata
```

The advantages of this model are:

1. **Self-contained**: All files for each asset are in the same directory, making migration and backup convenient
2. **Traceable**: Through metadata.json, you can trace the image's generation time, used prompt, model, and other information
3. **Extensible**: If more variants are needed in the future (such as thumbnails of different sizes), you only need to add new files in the same directory

Beautiful things or people don't have to be possessed. As long as they remain beautiful, it's enough to just appreciate their beauty. Although this saying goes a bit far, the principle is the same—when images are placed together, they look comfortable and are easy to find.

### Metadata Layered Storage

metadata.json is the core of the entire system. It adopts a layered storage strategy that distinguishes three types of fields:

```json
{
  "schemaVersion": 2,
  "assetId": "orange-dashboard",
  "slug": "orange-dashboard",
  "title": "Orange Dashboard",
  "tags": ["dashboard", "hero", "orange"],

  "source": { "type": "generated" },

  "paths": {
    "assetDir": "library/2026-03/orange-dashboard",
    "original": "original.png",
    "thumbnail": "thumbnail.webp"
  },

  "generated": {
    "prompt": "orange dashboard for docs hero",
    "provider": "azure-openai-image-api",
    "model": "gpt-image-1.5"
  },

  "recognized": {
    "title": "Orange Dashboard",
    "tags": ["dashboard", "ui", "orange"],
    "description": "A modern orange dashboard with charts and metrics"
  },

  "status": {
    "generation": "succeeded",
    "recognition": "succeeded",
    "thumbnail": "succeeded"
  },

  "timestamps": {
    "createdAt": "2026-03-11T04:01:19.570Z",
    "updatedAt": "2026-03-11T04:02:09.132Z"
  }
}
```

- **generated**: Records original information when the image is generated, such as the used prompt, provider, model, etc.
- **recognized**: AI recognition results, such as automatically generated titles, tags, descriptions
- **manual**: Manual organization results, data in this area has the highest priority and will not be overwritten by AI recognition

This layered strategy solved a core contradiction we had before: which takes priority, AI recognition results or manual organization results? The answer is manual priority, AI recognition is just auxiliary. We also figured this out—for some things, machines are ultimately machines, and ultimately people need to be the gatekeepers.

## Provider Adapter Pattern

Another core design of ImgBin is the Provider Adapter pattern. We abstract external APIs into unified interfaces, so even if we change AI service providers, we don't need to modify business logic.

How should I put it, it's just like relationships—how the appearance changes doesn't matter, what matters is that the inner core remains unchanged. Once the interface is set, the internal implementation can be changed however you want.

### Image Generation Provider

```typescript
interface ImageGenerationProvider {
  // Generate image, return image Buffer
  generate(options: GenerateOptions): Promise<Buffer>;

  // Get supported model list
  getSupportedModels(): Promise<string[]>;
}

interface GenerateOptions {
  prompt: string;
  model?: string;
  size?: '1024x1024' | '1792x1024' | '1024x1792';
  quality?: 'standard' | 'hd';
  format?: 'png' | 'webp' | 'jpeg';
}
```

### Vision Recognition Provider

```typescript
interface VisionRecognitionProvider {
  // Recognize image content, return structured metadata
  recognize(imageBuffer: Buffer): Promise<RecognitionResult>;

  // Get supported model list
  getSupportedModels(): Promise<string[]>;
}

interface RecognitionResult {
  title?: string;
  tags: string[];
  description?: string;
  confidence: number;
}
```

The advantages of this interface design are:

1. **Testable**: During unit testing, you can pass in a mock provider without needing to actually call external APIs
2. **Extensible**: Adding a new provider only requires implementing the interface without modifying caller code
3. **Replaceable**: Use Azure OpenAI in production, local models in testing, just need to switch configurations

Wanting to laugh to disguise your shedding tears, wanting to cry to test if you're numb—sometimes doing projects is like this,表面上看起来是换了个 API，实际上内在的那套逻辑一点没变，也就没什么好怕的了。

## CLI Command Design

ImgBin provides four core commands to meet different usage scenarios:

### generate: Single Image Generation

```bash
# Simplest usage
imgbin generate --prompt "orange dashboard for docs hero"

# Generate thumbnail and AI annotation at the same time
imgbin generate --prompt "orange dashboard" --annotate --thumbnail

# Specify output directory
imgbin generate --prompt "orange dashboard" --output ./library
```

### batch: Batch Tasks

Batch tasks are defined through YAML or JSON manifest files, suitable for use in CI/CD processes:

```yaml
# assets/jobs/launch.yaml
defaults:
  annotate: true
  thumbnail: true
  libraryRoot: ./library

jobs:
  - prompt: "orange dashboard hero"
    slug: orange-dashboard
    tags: [dashboard, hero, orange]

  - prompt: "pricing grid for docs"
    slug: pricing-grid
    tags: [pricing, grid, docs]
```

Execute command:

```bash
imgbin batch assets/jobs/launch.yaml
```

The batch task design supports failure isolation: items in the manifest are processed one by one, and single item failure doesn't affect other tasks. You can preview through `--dry-run` without actually executing.

That's one thing, but the key is it can also tell you what succeeded and what failed, unlike some things where when they fail, you don't even know how they failed.

### annotate: AI Annotation

Perform AI recognition on existing images to automatically generate titles, tags, and descriptions:

```bash
# Annotate single image
imgbin annotate ./library/2026-03/orange-dashboard

# Batch annotate entire directory
imgbin annotate ./library/2026-03/
```

### thumbnail: Thumbnail Generation

Generate thumbnails for existing images:

```bash
# Generate thumbnail
imgbin thumbnail ./library/2026-03/orange-dashboard
```

## Batch Task Manifest Design

The manifest for batch tasks supports flexible configuration. Default values can be set uniformly, and individual tasks can also override:

```yaml
# Global defaults
defaults:
  annotate: true        # Enable AI annotation by default
  thumbnail: true       # Generate thumbnails by default
  libraryRoot: ./library
  model: gpt-image-1.5

jobs:
  # Minimum configuration, only provide prompt
  - prompt: "first image"

  # Complete configuration
  - prompt: "second image"
    slug: custom-slug
    tags: [tag1, tag2]
    annotate: false     # This task doesn't execute AI annotation
    model: dall-e-3    # This task uses a different model
```

When executing, ImgBin processes tasks one by one. Each task's result is written to the corresponding metadata.json. Even if a task fails, it won't affect other tasks. After tasks are completed, a summary report is output:

```
✓ orange-dashboard (succeeded)
✓ pricing-grid (succeeded)
✗ hero-banner (failed: API rate limit exceeded)

2/3 succeeded, 1 failed
```

Some things can't be rushed, take them one by one, and you'll feel more at ease. This is perhaps the philosophy of batch tasks.

## Environment Variable Configuration

ImgBin supports flexible configuration through environment variables:

```bash
# ImgBin working directory
IMGBIN_WORKDIR=/path/to/imgbin

# Executable path (for calling in scripts)
IMGBIN_EXECUTABLE=/path/to/imgbin/dist/cli.js

# Asset library root directory
IMGBIN_LIBRARY_ROOT=./.imgbin-library

# Azure OpenAI configuration (if using Azure provider)
AZURE_OPENAI_ENDPOINT=https://your-resource.openai.azure.com/
AZURE_OPENAI_API_KEY=your-api-key
AZURE_OPENAI_IMAGE_DEPLOYMENT=gpt-image-1
```

Configuration is both important and not important. After all, whatever makes you comfortable is best, and what suits you is the best.

## Implementation Points

During implementation, we summarized the following key points:

### Provider Interface Design

Interface definitions should be clear and complete, including input parameters, return values, and error handling. It's recommended to provide both synchronous and asynchronous calling methods for different scenarios.

This is also a bit of experience from a veteran—after all, once an interface is set, you don't want to change it again, it's troublesome.

### Failure Handling Strategy

When a task fails in batch processing, it should:

1. Log detailed error information to a separate log file
2. Continue executing other tasks without interrupting the entire process
3. Finally return a non-zero exit code indicating task failure
4. Clearly display each task's execution result in the summary report

Some things just fail, and avoiding it is useless. It's better to admit it generously and then find a solution. This principle applies to both doing projects and being a person.

### Metadata Merge Strategy

Recognition results are written to the `recognized` area by default, and manually modified fields have a `manual` mark. When updating metadata, a "only add, don't delete" strategy is adopted: unless explicitly passing the `--force` parameter, existing manual organization results won't be overwritten.

We also figured this out—some things, once missed, are missed. If overwritten, they're gone. It's better to keep them, after all, the record itself is also a kind of beauty.

### Directory Creation Atomicity

Use `fs.mkdir({ recursive: true })` to ensure directory creation atomicity, avoiding race conditions in concurrent scenarios.

This is probably the so-called sense of security—be steady when you need to be steady, be fast when you need to be fast, without dragging your feet or looking forward and backward.

## Summary

As the core tool for image asset management in the HagiCode project, ImgBin solved the problems we faced through the following designs:

1. **Unified entry**: CLI commands cover all operations including generation, annotation, and thumbnails
2. **Metadata-driven**: Each asset has complete metadata.json supporting retrieval and tracking
3. **Provider Adapter**: Flexible external API abstraction for easy testing and extension
4. **Batch task support**: CI/CD processes can automatically execute batch image generation
