## Editing DESIGN.md Directly in Web Interface: From Concept to Implementation

> In the MonoSpecs project management system, DESIGN.md carries the project's architectural design and technical decisions. But traditional editing requires users to switch to external editors. This fragmented workflow, well, it's like being interrupted while reading a poem—the inspiration is gone, and so is the mood. This article shares the solution we practiced in the HagiCode project: editing DESIGN.md directly in the web interface, with support for importing templates from online design sites. After all, who doesn't love the feeling of getting things done in one go?

## Background

`DESIGN.md` serves as the core carrier for project design documents, holding critical information such as architectural design, technical decisions, and implementation guidance. However, traditional editing requires users to switch to external editors (like VS Code), manually locate the physical path, and then edit. The process isn't particularly complex, but after repeating it several times, one gets tired.

The specific problems manifest in several aspects:

- **Fragmented workflow**: Users need to frequently switch between the web management interface and local editors, breaking workflow continuity—like suddenly losing internet connection while listening to music, the rhythm is completely disrupted.
- **Difficult reuse**: The design site has already published a rich library of design templates, but they cannot be directly integrated into the project editing process.明明 have good stuff, just can't use it, which does feel a bit regrettable.
- **Missing experience**: Lack of "preview-select-import" closed loop, users must manually copy and paste, increasing error risk. The more manual operations, the more opportunities for errors.
- **Collaboration barriers**: Synchronized maintenance of design documents and code implementation becomes high-friction, hindering team collaboration efficiency. Team collaboration is already not easy, why add these obstacles?

To address these pain points, we decided to implement direct editing capability for DESIGN.md in the web interface, with support for one-click template import from online design sites. This isn't exactly an earth-shattering decision, just wanting to make the development experience a bit smoother.

## About HagiCode

The solution shared in this article comes from our practical experience in the [HagiCode](https://hagicode.com) project. HagiCode is an AI-driven code assistant project. During development, we need to frequently maintain project design documents. To enable the team to collaborate more efficiently, we explored and implemented this online editing and import solution. Actually nothing special, just encountered a problem and found a way to solve it.

## Technical Solution

### Overall Architecture

The solution adopts a same-origin proxy architecture with frontend-backend separation, mainly consisting of the following layers. This architecture design, well, it's just about "each doing their own job":

### Key Technical Decisions

#### Decision 1: Global Drawer Mode

Using a single global drawer instead of local popups, managing state through `layoutSlice`, achieving consistent experience across views (classic/kanban). This approach ensures that users get a unified interactive experience regardless of which view they open the editor from. After all, a consistent experience makes users feel more at ease, not getting lost just because they switched views.

#### Decision 2: Project-Scoped API

Mounting DESIGN.md related interfaces under `ProjectController`, reusing existing project permission boundaries, avoiding the complexity of adding a separate controller. The benefit of this design is clearer permission management and compliance with RESTful resource organization principles. Sometimes, reuse makes more sense than creating from scratch, doesn't it?

#### Decision 3: Version Conflict Detection

Deriving opaque version based on file system `LastWriteTimeUtc`, implementing lightweight optimistic concurrency control. When multiple users edit the same file simultaneously, the system can detect conflicts and prompt users to refresh. This design neither blocks user editing operations nor ensures data consistency—like boundaries in interpersonal relationships, neither too distant nor overstepping.

#### Decision 4: Same-Origin Proxy Mode

Proxying external design site resources through `IHttpClientFactory`, avoiding CORS issues and SSRF risks. This design ensures security while simplifying frontend calls. Security, you can never do too much, after all data security is like health, you only regret it when it's lost.

## Core Implementation

### 1. Direct Editing of DESIGN.md

#### Backend Implementation

The backend is mainly responsible for path resolution, file read/write, and version management. Although these tasks are basic, they are indispensable, like the foundation of a house:

```csharp
// Path resolution and security validation
private Task<string> ResolveDesignDocumentDirectoryAsync(string projectPath, string? repositoryPath)
{
    if (string.IsNullOrWhiteSpace(repositoryPath))
    {
        return Task.FromResult(Path.GetFullPath(projectPath));
    }
    return ValidateSubPathAsync(projectPath, repositoryPath);
}

// Version number generation (based on file system timestamp and size)
private static string BuildDesignDocumentVersion(string path)
{
    var fileInfo = new FileInfo(path);
    fileInfo.Refresh();
    return string.Create(
        CultureInfo.InvariantCulture,
        $"{fileInfo.LastWriteTimeUtc.Ticks:x}-{fileInfo.Length:x}");
}
```

The version number design is actually quite interesting—we use the file's last modification time and size to generate a unique version identifier. This is both lightweight and reliable, no need to maintain additional version databases. Simple things are often more effective, aren't they?

#### Frontend Implementation

The frontend implements dirty state detection and save logic. This design lets users know at any time whether their modifications are saved, reducing the anxiety of "what if it gets lost":

```typescript
// Dirty state detection and save logic
const [draft, setDraft] = useState('');
const [savedDraft, setSavedDraft] = useState('');
const isDirty = draft !== savedDraft;

const handleSave = useCallback(async () => {
    const result = await saveProjectDesignMdDocument({
        ...activeTarget,
        content: draft,
        expectedVersion: document.version, // Optimistic concurrency control
    });
    setSavedDraft(draft); // Update saved state
}, [activeTarget, document, draft]);
```

In this implementation, we maintain two states: `draft` is the current editing content, `savedDraft` is the saved content. By comparing the two, we determine if there are unsaved modifications. Although this design is simple, it provides peace of mind—after all, who wants their hard-written content to suddenly disappear?

#### Index Data Format

The design site's index file defines all available templates. With this index, users can easily select the template they want, just like ordering dishes in a restaurant:

```json
{
  "entries": [
    {
      "slug": "linear.app",
      "title": "Linear Inspired Design System",
      "summary": "AI Product / Dark Theme",
      "detailUrl": "/designs/linear.app/",
      "designDownloadUrl": "/designs/linear.app/DESIGN.md",
      "previewLightImageUrl": "...",
      "previewDarkImageUrl": "..."
    }
  ]
}
```

Each entry contains basic template information and download links. The backend reads the list of available templates from this index and displays them for user selection. This design makes selection intuitive rather than fumbling in the dark.

#### Same-Origin Proxy Implementation

To ensure security, the backend strictly validates access to design sites. Security, you can never be too careful:

```csharp
// Safe slug validation
private static readonly Regex SafeDesignSiteSlugRegex =
    new("^[A-Za-z0-9](?:[A-Za-z0-9._-]{0,127})$", RegexOptions.Compiled);

private static string NormalizeDesignSiteSlug(string slug)
{
    var normalizedSlug = slug?.Trim() ?? string.Empty;
    if (!IsSafeDesignSiteSlug(normalizedSlug))
    {
        throw new BusinessException(
            ProjectDesignSiteIndexErrorCodes.InvalidSlug,
            "Design site slug must be a single safe path segment.");
    }
    return normalizedSlug;
}

// Preview image caching (OS temp directory)
private static string ComputePreviewCacheKey(string slug, string theme, string previewUrl)
{
    var raw = $"{slug}|{theme}|{previewUrl}";
    var bytes = SHA256.HashData(Encoding.UTF8.GetBytes(raw));
    return Convert.ToHexString(bytes).ToLowerInvariant();
}
```

Here we do two things: first, use regex to strictly validate slug format to prevent path traversal attacks; second, cache preview images to reduce request pressure on external sites. The former is protection, the latter is optimization, both indispensable.

### 3. Complete Import Process

```typescript
// 1. Open import drawer
const handleRequestImportDrawer = useCallback(() => {
    setIsImportDrawerOpen(true);
}, []);

// 2. Select and import
const handleImportRequest = useCallback((entry) => {
    if (isDirty) {
        setPendingImportEntry(entry);
        setConfirmMode('import'); // Overwrite confirmation
        return;
    }
    void executeImport(entry);
}, [isDirty]);

// 3. Execute import
const executeImport = useCallback(async (entry) => {
    const result = await getProjectDesignMdSiteImportDocument(
        activeTarget.projectId,
        entry.slug
    );
    setDraft(result.content); // Only replace editor text, no auto-save
    setIsImportDrawerOpen(false);
}, [activeTarget?.projectId]);
```

The import process design follows the "user confirmation" principle: after importing, it only updates the editor content and does not auto-save. Users can review the imported content and manually save after confirmation. After all, users should have the final say on what they write, shouldn't they?

## Practical Examples

### Scenario 1: Project Root DESIGN.md Creation

When DESIGN.md doesn't exist, the backend returns a virtual document state. This design eliminates the need for the frontend to handle "file not found" specially, and the unified API interface simplifies code logic:

```csharp
return new ProjectDesignDocumentDto
{
    Path = targetPath,
    Exists = false,  // Virtual document state
    Content = string.Empty,
    Version = null
};

// Auto-create file on first save
public async Task<SaveProjectDesignDocumentResultDto> SaveDesignDocumentAsync(...)
{
    Directory.CreateDirectory(targetDirectory);
    await File.WriteAllTextAsync(targetPath, input.Content);
    return new SaveProjectDesignDocumentResultDto { Created = !exists };
}
```

The benefit of this design is that the frontend doesn't need special handling for "file not found" situations, and the unified API interface simplifies code logic. Sometimes, hiding complexity in the backend makes the frontend more relaxed in focusing on user experience.

### Scenario 2: Import Template from Design Site

After users select the "Linear" design template in the import drawer, the system obtains the DESIGN.md content through backend proxy. The entire process is transparent to users—they only need to select templates, and the system automatically handles all network requests and data conversion:


The entire process is transparent to users—they only need to select templates, and the system automatically handles all network requests and data conversion. Users don't need to care about the underlying complexity, this is the experience we pursue—simple, yet powerful.

### Scenario 3: Version Conflict Handling

When multiple users edit the same DESIGN.md simultaneously, the system detects version conflicts. This optimistic concurrency control mechanism ensures data consistency without blocking user editing operations:

```csharp
if (!string.Equals(currentVersion, expectedVersion, StringComparison.Ordinal))
{
    throw new BusinessException(
        ProjectDesignDocumentErrorCodes.VersionConflict,
        $"DESIGN.md at '{targetPath}' changed on disk.");
}
```

The frontend catches this error and prompts the user:

```typescript
// Frontend prompts user to refresh and retry
<Alert>
    <AlertTitle>Version Conflict</AlertTitle>
    <AlertDescription>
        File has been modified by another process. Please refresh to the latest version and try again.
    </AlertDescription>
</Alert>
```

This optimistic concurrency control mechanism ensures data consistency without blocking user editing operations. Conflicts are inevitable, but at least users can be informed of what happened rather than silently losing modifications.

## Considerations and Best Practices

### 1. Path Security

Always validate `repositoryPath` to prevent path traversal attacks. Security, you can never do too much:

```csharp
// Always validate repositoryPath to prevent path traversal attacks
return ValidateSubPathAsync(projectPath, repositoryPath);
// Reject dangerous inputs like "../", absolute paths, etc.
```

### 2. Caching Strategy

Cache preview images for 24 hours, maximum 160 files. Moderate caching improves performance, but don't overdo it—balance is key:

```csharp
// Preview image cache 24 hours, max 160 files
private static readonly TimeSpan PreviewCacheTtl = TimeSpan.FromHours(24);
private const int PreviewCacheMaxFiles = 160;
// Periodically clean expired cache
```

### 3. Error Handling

Graceful degradation when upstream site is unavailable. This graceful degradation design ensures that even if external dependencies are unavailable, core editing functionality still works normally. After all, the entire system shouldn't be paralyzed just because one external service is down:

```typescript
// Graceful degradation when upstream site is unavailable
try {
    const catalog = await getProjectDesignMdSiteImportCatalog(projectId);
} catch (error) {
    toast.error(t('project.designMd.siteImport.feedback.catalogLoadFailed'));
    // Main edit drawer still available
}
```

This graceful degradation design ensures that even if external dependencies are unavailable, core editing functionality still works normally. Systems should be resilient, not collapse at the first problem.

### 4. User Experience Optimization

Confirm overwrite before import, no auto-save after import. Users should have control over their operations, not the system making decisions:

```typescript
// Confirm overwrite before import
if (isDirty) {
    setConfirmMode('import');
    return;
}

// No auto-save after import, user confirms
setDraft(result.content); // Only update draft
// User clicks Save after review to actually write to disk
```

### 5. Performance Considerations

Use HTTP client factory to avoid creating too many connections. Resource management, though seemingly insignificant, can bring unexpected results when done well:

```csharp
// Use HTTP client factory to avoid creating too many connections
private const string DesignSiteProxyClientName = "ProjectDesignSiteProxy";
private static readonly TimeSpan DesignSiteProxyTimeout = TimeSpan.FromSeconds(8);
```

## Extension Suggestions

1. **Markdown Enhancement**: Currently using basic `Textarea`, consider upgrading to CodeMirror to support syntax highlighting and keyboard shortcuts. Better editor experience makes writing documents more enjoyable.
2. **Preview Mode**: Add Markdown real-time preview to improve editing experience. WYSIWYG always gives more confidence.
3. **Diff Merging**: Implement intelligent merge algorithms instead of simple full-text replacement. Conflicts are inevitable, but at least make conflict handling less painful.
4. **Local Caching**: Cache design.json to database to reduce dependency on external sites. Fewer dependencies, more stable system—simple logic.
