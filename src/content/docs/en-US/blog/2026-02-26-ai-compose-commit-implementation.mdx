## AI Compose Commit: Revolutionizing Git Commit Workflow with AI

## Introduction

In the software development process, committing code is a daily task that programmers face every day. But have you ever experienced this scenario: at the end of a day's work, opening Git to see dozens of unstaged modified files, not knowing how to organize them into reasonable commits?

The traditional approach is to manually stage files in batches, commit them one by one, and write commit messages. This process is both time-consuming and error-prone. We often waste a lot of time on this, as no one wants to be bothered by these trivialities when already exhausted at the end of the day.

We've introduced a new feature in the HagiCode project—AI Compose Commit, designed to revolutionize this workflow. It intelligently analyzes all uncommitted changes in the workspace through AI, automatically groups them into multiple logical commits, and executes compliant commit operations. This article will delve into the implementation principles, technical architecture, and the challenges and solutions we encountered in practice.

## About HagiCode

The solution shared in this article comes from our practical experience in the [HagiCode](https://hagicode.com) project.

## Background

### Pain Points of Traditional Git Commits

Git, as a version control system, provides developers with powerful code management capabilities. However, in actual use, commit operations often become a bottleneck in the development process:

1. **Time-consuming manual grouping**: When there are numerous file changes, developers need to check file contents one by one to determine which ones belong to the same functionality, which requires significant mental effort
2. **Inconsistent commit message quality**: Writing commit messages that conform to Conventional Commits specifications requires experience and skill, and beginners often write non-standard commits
3. **Complex multi-repository management**: In monorepo environments, switching between different repositories adds operational complexity
4. **Interrupted workflow**: Committing code interrupts development flow, affecting coding efficiency

These issues are particularly prominent in large projects and team collaboration environments. An excellent development tool should allow developers to focus on core coding work rather than being troubled by tedious commit processes.

### The Trend of AI-Assisted Development

In recent years, AI technology applications in software development have become increasingly widespread. From code completion, error detection to automatic documentation generation, AI is gradually penetrating every aspect of development. Regarding Git workflows, although some tools provide commit message generation functionality, most are limited to single commit scenarios and lack intelligent analysis and grouping capabilities for entire workspace changes.

Actually, HagiCode encountered these pain points during development. We tried various tools, but they all had some limitations. Either the functionality wasn't complete enough, or the user experience wasn't good enough. This is why we ultimately decided to implement the AI Compose Commit feature ourselves.

HagiCode's AI Compose Commit feature was created to fill this gap. It doesn't just generate commit messages but completely takes over the entire process from file analysis to executing commits.

## Problem

### Technical Challenges

In implementing the AI Compose Commit feature, we faced multiple technical challenges:

1. **File semantic understanding**: AI needs to understand the semantic relationships between file changes and determine which files belong to the same functional module. This requires deeply analyzing file content, directory structure, and change context.

2. **Commit grouping strategy**: How to define reasonable grouping standards? By functionality, module, or file type? Different projects may require different strategies.

3. **Real-time feedback and asynchronous processing**: Git operations may require significant time, especially when handling numerous files. How to complete complex operations while ensuring user experience?

4. **Multi-repository support**: Under monorepo architecture, operations need to be correctly routed between the main repository and sub-repositories.

5. **Error handling and rollback**: If a commit fails, how to handle executed commits? Is it necessary to roll back staged files?

6. **Commit message consistency**: Generated commit messages need to conform to the project's existing style, maintaining format consistency with historical commits.

### Performance Considerations

AI processing of numerous file changes consumes significant time and computational resources. We need to optimize in the following aspects:

- Reduce unnecessary AI calls
- Optimize file context construction methods
- Implement efficient Git operation batching

These issues actually occurred in HagiCode's actual use. Through continuous iteration and optimization, we found a relatively perfect solution. If you're also developing similar tools, we hope our experience can provide you with some inspiration.

## Solution

### Overall Architecture Design

We adopted a layered architecture to implement the AI Compose Commit feature, ensuring the system has good scalability and maintainability:

#### 1. API Layer (Web Layer)

**GitController** provides the `POST /api/git/auto-compose-commit` endpoint as the feature entry point. To optimize user experience, we adopted the Fire-and-Forget asynchronous pattern:

- After the client initiates a request, the server immediately returns HTTP 202 Accepted
- Actual AI processing executes asynchronously in the background
- Upon completion, the client is notified through SignalR

This design ensures that even if AI processing takes several minutes, users get immediate response without feeling the system is stuck.

#### 2. Application Service Layer (Application Layer)

**GitAppService** is responsible for core business logic:

- Repository detection: supports multi-repository management in monorepos
- Lock management: prevents conflicts caused by concurrent operations
- File staging coordination: interaction with AI processing workflow
- Error rollback: handles state recovery in failure scenarios

#### 3. Distributed Computing Layer (Orleans Grains)

**AIGrain** serves as the core execution unit for AI operations, implementing the `AutoComposeCommitAsync` method from the `IAIGrain` interface:

```csharp
// Define the interface method for AI automatic composition of commits
// Parameter explanations:
// - projectId: Project unique identifier
// - unstagedFiles: List of unstaged files, including file paths and status information
// - projectPath: Project root directory path (optional), used to access project context
// Return value: Response object containing execution results, including success/failure status and detailed information
[Alias("AutoComposeCommitAsync")]
[ResponseTimeout("00:20:00")] // 20 minute timeout, suitable for processing large changesets
Task<AutoComposeCommitResponseDto> AutoComposeCommitAsync(
    string projectId,
    GitFileStatusDto[] unstagedFiles,
    string? projectPath = null);
```

This method sets a 20-minute timeout to handle large changesets. In actual use, HagiCode found that some projects' single changes might involve hundreds of files, requiring longer processing time.

#### 4. AI Service Layer

Through the abstract `IAIService` interface, we implemented a pluggable architecture for AI services. Currently using Claude Helper service, but can easily switch to other AI providers.

### Core Implementation Logic

#### File Context Construction

AI needs to understand each file's status to make intelligent decisions. We construct file context through the `BuildFileChangesXml` method:

```csharp
/// <summary>
/// Builds an XML representation of file changes, providing complete file context information for AI
/// </summary>
/// <param name="stagedFiles">List of staged files, including file paths, status, and old paths (for rename operations)</param>
/// <returns>Formatted XML string containing metadata information for all files</returns>
private static string BuildFileChangesXml(GitFileStatusDto[] stagedFiles)
{
    var sb = new StringBuilder();
    sb.AppendLine("<files>");

    foreach (var file in stagedFiles)
    {
        sb.AppendLine("  <file>");
        // Use XML escaping to ensure special characters don't break XML structure
        sb.AppendLine($"    <path>{System.Security.SecurityElement.Escape(file.Path)}</path>");
        sb.AppendLine($"    <status>{System.Security.SecurityElement.Escape(file.Status)}</status>");

        // Handle file rename scenarios, recording old paths for AI to understand change relationships
        if (!string.IsNullOrEmpty(file.OldPath))
        {
            sb.AppendLine($"    <oldPath>{System.Security.SecurityElement.Escape(file.OldPath)}</oldPath>");
        }

        sb.AppendLine("  </file>");
    }

    sb.AppendLine("</files>");
    return sb.ToString();
}
```

This XML-formatted context includes file paths, status, and old paths (for rename operations), providing complete metadata for AI. Through structured XML format, we ensure AI can accurately understand each file's status and change type.

#### Commit Result Parsing

After AI execution completes, it returns structured results. We implemented a dual parsing strategy to ensure compatibility:

```csharp
/// <summary>
/// Parses commit execution results returned by AI, supporting both delimiter format and regular expression format
/// </summary>
/// <param name="aiResponse">Original response content returned by AI</param>
/// <returns>List of parsed commit results, each result containing commit hash and execution status</returns>
private List<CommitResultDto> ParseCommitExecutionResults(string aiResponse)
{
    var results = new List<CommitResultDto>();

    // Prioritize delimiter parsing (new format), which is clearer and more reliable
    if (aiResponse.Contains("---"))
    {
        logger.LogDebug("Using delimiter-based parsing for AI response");
        results = ParseDelimitedFormat(aiResponse);

        if (results.Count > 0)
        {
            return results; // Successfully parsed, return results directly
        }

        logger.LogWarning("Delimiter-based parsing produced no results, falling back to regex");
    }
    else
    {
        logger.LogDebug("No delimiter found, using legacy regex-based parsing");
    }

    // Fall back to regular expression parsing (old format), ensuring backward compatibility
    return ParseLegacyFormat(aiResponse);
}
```

The delimiter format uses `---` as a separator between commits, with a clear and easy-to-parse format:

```
---
Commit 1: abc123def456
feat(auth): add user login functionality

Implement JWT-based authentication with login form and API endpoints.

Co-Authored-By: Hagicode <noreply@hagicode.com>
---
Commit 2: 789ghi012jkl
docs(readme): update installation instructions

Add new setup steps for Docker environment.

Co-Authored-By: Hagicode <noreply@hagicode.com>
---
```

This format design makes parsing simple and reliable while remaining clear for human reading.

### Lock Management Mechanism

To prevent state conflicts caused by concurrent operations, we implemented a repository lock mechanism:

```csharp
// Acquire repository lock, preventing concurrent operations
// Parameter explanations:
// - fullPath: Complete path of the repository, used to identify different repository instances
// - requestedBy: Requester identifier, used for tracking and logging
await _autoComposeLockService.AcquireLockAsync(fullPath, requestedBy);

try
{
    // Execute AI Compose Commit operation
    // This code calls the Orleans Grain method, executing actual AI processing and Git operations
    await aiGrain.AutoComposeCommitAsync(projectId, unstagedFiles, projectPath);
}
finally
{
    // Ensure the lock is released, whether operation succeeds or fails
    // Using finally block guarantees lock release even in exception cases, avoiding deadlocks
    await _autoComposeLockService.ReleaseLockAsync(fullPath);
}
```

The lock has a 20-minute timeout, consistent with the AI operation's timeout setting. If the operation fails or times out, the system automatically releases the lock to avoid permanent blocking. In actual use, HagiCode found this lock mechanism very important, especially in team collaboration environments where multiple developers might trigger AI Compose Commit operations simultaneously.

### SignalR Real-time Notification

After processing completes, the system sends notifications to the frontend through SignalR:

```csharp
/// <summary>
/// Sends notification of auto composition commit completion
/// </summary>
/// <param name="projectId">Project identifier, used to route notifications to the correct client</param>
/// <param name="totalCount">Total number of commits, including successful and failed ones</param>
/// <param name="successCount">Number of successful commits</param>
/// <param name="failureCount">Number of failed commits</param>
/// <param name="success">Flag indicating whether the overall operation succeeded</param>
/// <param name="error">Error message (if operation failed)</param>
private async Task SendAutoComposeCommitNotificationAsync(
    string projectId,
    int totalCount,
    int successCount,
    int failureCount,
    bool success,
    string? error)
{
    try
    {
        // Build notification data transfer object, containing detailed execution results
        var notification = new AutoComposeCommitCompletedDto
        {
            ProjectId = projectId,
            TotalCount = totalCount,
            SuccessCount = successCount,
            FailureCount = failureCount,
            Success = success,
            Error = error
        };

        // Broadcast notification to all connected clients through SignalR Hub
        await messageService.SendAutoComposeCommitCompletedAsync(notification);

        logger.LogInformation(
            "Auto compose commit notification sent for project {ProjectId}: {SuccessCount}/{TotalCount} succeeded",
            projectId, successCount, totalCount);
    }
    catch (Exception ex)
    {
        // Log notification errors without affecting the main operation flow
        // Notification failure shouldn't cause the entire operation to fail
        logger.LogError(ex, "Failed to send auto compose commit notification for project {ProjectId}", projectId);
    }
}
```

After receiving the notification, the frontend can update the UI, displaying commit success or failure status, improving user experience. This real-time feedback mechanism received good user feedback in HagiCode's use, allowing users to clearly know when the operation completes and its results.

## Implementation

### Prompt Engineering Design

AI behavior is entirely determined by the Prompt. We carefully designed the Auto Compose Commit Prompt template. Taking the Chinese version as an example (`auto-compose-commit.zh-CN.hbs`):

#### Non-interactive Mode Support

The Prompt beginning explicitly declares support for non-interactive running mode, a key requirement for CI/CD and automation scripts:

```handlebars
**Important Note**: This prompt may run in non-interactive environments (such as CI/CD, automation scripts).

**Non-interactive Mode**:
- Prohibit using AskUserQuestion or any interactive tools
- When user input is needed:
  - Use reasonable default values (such as using feat for commit type)
  - Skip optional confirmation steps
  - Record assumptions made
```

This design ensures that AI Compose Commit functionality can not only be used in interactive IDE environments but can also be integrated into CI/CD workflows, achieving fully automated commit processes.

#### Branch Protection Mechanism

To prevent AI from executing dangerous operations, we added strict branch protection rules in the Prompt:

```handlebars
**Branch Protection**:
- Prohibit executing any branch switching operations (git checkout, git switch)
- All git commit commands must be executed on the current branch
- Must not create, delete, or rename branches
- Must not modify untracked files or unstaged changes
- If branch switching is needed to complete operations, should return an error rather than execute
```

These rules ensure operation safety by constraining AI's tool usage scope. In actual testing, HagiCode verified the effectiveness of these constraints. When AI encounters scenarios requiring branch switching, it safely returns errors rather than executing dangerous operations.

#### Intelligent Grouping Decision Tree

The Prompt defines detailed file grouping decision logic:

```handlebars
**File Grouping Decision Tree**:
├── Is it a configuration file (package.json, tsconfig.json, .env, etc.)?
│   ├── Yes → Independent commit (type: chore or build)
│   └── No → Continue
├── Is it a documentation file (README.md, *.md, docs/**)?
│   ├── Yes → Independent commit (type: docs)
│   └── No → Continue
├── Is it related to the same functionality?
│   ├── Yes → Merge into the same commit
│   └── No → Commit separately
└── Is it a cross-module change?
    ├── Yes → Group by module
    └── No → Group by functionality
```

This decision tree provides AI with clear grouping logic, ensuring generated commits conform to semantic rationality. In actual use, HagiCode found this decision tree can handle most common scenarios, and generated grouping results meet developer expectations.

#### Historical Format Consistency Analysis

To keep commit messages consistent with project history, the Prompt requires AI to analyze recent commit history before generating:

```handlebars
**Historical Format Consistency**: Before generating commit messages, you **must** analyze the current repository's commit history to match existing style

1. Use git log -n 15 --pretty=format:"%H|%s|%b%n---%n" to get recent commit history
2. Analyze commits to identify:
   - Structure patterns: Does the project use multiple paragraphs? Are there "Changes:" or "Capabilities:" sections?
   - Language patterns: Are commit messages in English, Chinese, or mixed?
   - Common types: Which commit types are most frequently used (feat, fix, docs, etc.)?
   - Special formats: Are there Co-Authored-By lines? Other project-specific conventions?
3. Generate commit messages that follow detected patterns
```

This analysis ensures AI-generated commit messages don't appear jarring but maintain style consistency with the project's commit history. In HagiCode's multilingual projects, this feature is particularly important, automatically selecting appropriate language and format based on the project's commit history.

#### Co-Authored-By Requirement

Each commit must include Co-Authored-By information:

```handlebars
**Important**: Each commit must add Co-Authored-By information
- Use the following format: git commit -m "type(scope): subject" -m "" -m "Co-Authored-By: Hagicode <noreply@hagicode.com>"
- Or include the Co-Authored-By line directly in the commit message
```

This is not only for contribution standards but also for tracking AI-assisted commit history. HagiCode made this requirement a mandatory rule, ensuring all AI-generated commits carry clear source attribution.

### Detailed Workflow

The complete AI Compose Commit workflow is as follows:

1. **User trigger**: User clicks the "AI Auto Compose Commit" button in Git Status panel or Quick Actions Zone
2. **API request**: Frontend sends POST request to `/api/git/auto-compose-commit` endpoint
3. **Immediate response**: Server returns HTTP 202 Accepted, not waiting for processing completion
4. **Background processing**:
   - GitAppService acquires repository lock
   - Calls AIGrain's AutoComposeCommitAsync method
   - Builds file context XML
   - Executes AI Prompt, letting AI analyze and execute commits
5. **AI execution**:
   - Use Git commands to get all unstaged changes
   - Read file contents to understand change nature
   - Group files by semantic relationships
   - Execute `git add` and `git commit` operations for each group
6. **Result parsing**: Parse execution results returned by AI
7. **Notification sending**: Notify frontend through SignalR
8. **Lock release**: Whether successful or failed, release repository lock

This workflow design ensures users can immediately continue other work after initiating operations without waiting for AI processing to complete. HagiCode user feedback indicates this asynchronous processing approach greatly improves workflow experience.

## Best Practices

### Best Practices for Using AI Compose Commit

#### 1. Appropriate Usage Timing

AI Compose Commit is most suitable for the following scenarios:

- At the end of a day's work, batch processing changes to multiple files
- After refactoring operations, multiple related files need separate commits
- After feature development completion, related changes need to be grouped and committed

Not suitable for the following scenarios:

- Quick commits for single files (direct regular commit is faster)
- Scenarios requiring precise control over commit content
- Commits containing sensitive information (requiring manual review)

#### 2. Review AI-Generated Commits

Although AI intelligent grouping is powerful, developers should still review generated commits:

- Check if commit grouping meets expectations
- Verify accuracy of commit messages
- Confirm no files are missed or incorrectly included

If unreasonable grouping is found, use `git reset --soft HEAD~N` to undo and regroup. HagiCode's experience shows that even with intelligent AI grouping, manual review remains valuable, especially for important feature commits.

#### 3. Coordinate with Project Standards

Ensure the project's Git configuration supports Conventional Commits:

```bash
# Install commitlint
npm install -g @commitlint/cli @commitlint/config-conventional

# Configure commitlint
echo "module.exports = {extends: ['@commitlint/config-conventional']}" > commitlint.config.js
```

This allows validating commit message format in CI/CD workflows, maintaining consistency with AI Compose Commit generated formats.

### Recommendations for Implementing Similar Functionality

If you want to implement similar AI-assisted commit functionality in your project, here are our recommendations:

#### 1. Start Small

First implement single commit message generation, then gradually expand to multi-commit grouping functionality. This makes verification and iteration easier. HagiCode also gradually improved functionality following this path. Early versions only supported single commits, later expanding to intelligent multi-commit grouping.

#### 2. Use Mature AI SDKs

Don't implement AI calling logic yourself. Using existing SDKs can reduce development time and potential bugs. We used Claude Helper service, which provides stable interfaces and comprehensive error handling.

#### 3. Emphasize Prompt Design

Prompt quality directly determines AI output quality. Invest time designing detailed Prompts, including:

- Clear task descriptions
- Specific output format requirements
- Rules for handling edge cases
- Example explanations

HagiCode invested significant time in Prompt design, which is one of the key factors in the feature's success.

#### 4. Implement Comprehensive Error Handling

AI operations can fail for various reasons (network issues, API rate limiting, content moderation, etc.). Ensure your system can gracefully handle these errors and provide meaningful error messages.

#### 5. Provide Manual Intervention Mechanisms

Don't fully automate; retain user control. Provide options to view grouping results, adjust groupings, manually edit commit messages, etc., balancing automation with flexibility. Although HagiCode implemented automatic execution, it still retained preview and adjustment capabilities.

### Performance Optimization Techniques

#### 1. File Filtering

When building file context, filter out files that don't need AI analysis:

```csharp
// Filter out auto-generated files and overly large files, reducing AI processing burden
var relevantFiles = stagedFiles
    .Where(f => !IsGeneratedFile(f.Path))
    .Where(f => !IsLargeFile(f.Path))
    .ToArray();
```

#### 2. Parallel Processing

If supporting multiple independent repositories, process different repository commits in parallel to improve overall efficiency.

#### 3. Cache Optimization

Cache project commit history analysis results to avoid reanalyzing each time. Store historical format preferences in configuration files to reduce AI calls.
