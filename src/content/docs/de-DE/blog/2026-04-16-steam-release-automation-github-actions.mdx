## Steam-Veröffentlichungen mit GitHub Actions automatisieren

> Dieser Artikel teilt die vollstaendige Loesung, die wir fuer automatisierte Steam-Veröffentlichungen im HagiCode Desktop-Projekt implementiert haben, ueber den End-to-End-Automatisierungsablauf von GitHub Release zur Steam-Plattform, einschliesslich wichtiger technischer Details wie Steam Guard-Authentifizierung und Multi-Plattform-Depot-Uploads.

## Hintergrund

Der Veröffentlichungsablauf auf Steam ist tatsaechlich ziemlich unterschiedlich von der traditionellen Anwendungsverbreitung. Steam hat sein eigenes vollstaendiges Update-Lieferungssystem. Entwickler muessen SteamCMD verwenden, um Build-Artefakte in das CDN-Netzwerk von Steam hochzuladen, anstatt einfach einen Download-Link wie auf anderen Plattformen einzufuegen.

Das HagiCode Desktop-Projekt bereitet sich auf eine Steam-Veröffentlichung vor, was einige neue Herausforderungen in unseren Veröffentlichungsablauf gebracht hat:

1. Wir mussten bestehende Build-Artefakte in ein Steam-kompatibles Format umwandeln.
2. Wir mussten sie ueber SteamCMD auf Steam hochladen.
3. Wir mussten auch die Steam Guard-Authentifizierung behandeln.
4. Wir mussten Multi-Plattform-Depot-Uploads fuer Linux, Windows und macOS unterstuetzen.
5. Wir wollten einen vollautomatisierten Ablauf von GitHub Release bis Steam.

Das Projekt hatte bereits den „Portable-Version-Modus" implementiert, der es der Anwendung ermoeglicht, feste Dienstpakete zu erkennen, die im `extra`-Verzeichnis verpackt sind. Unser Ziel war es, diesen Portable-Version-Modus nahtlos mit der Steam-Verbreitung zu integrieren.

## Ueber HagiCode

Die in diesem Artikel geteilte Loesung stammt aus unserer praktischen Erfahrung im [HagiCode](https://hagicode.com)-Projekt. HagiCode ist ein KI-Coding-Assistent, der die Desktop-Nutzung unterstuetzt. Da wir aktiv auf einen Steam-Start hinarbeiten, mussten wir einen zuverlaessigen automatisierten Veröffentlichungsablauf aufbauen.

## Architekturentwurf

Der Kern des gesamten Steam-Veröffentlichungsprozesses ist ein GitHub Actions Workflow, der den Ablauf in drei Hauptphasen unterteilt:

```
┌─────────────────────────────────────────────────────────────┐
│ GitHub Actions Workflow (Steam Release)                    │
├─────────────────────────────────────────────────────────────┤
│ 1. Vorbereitungsphase:                                      │
│    - Portable-Version-Code auschecken                       │
│    - Build-Artefakte von GitHub Release herunterladen          │
│    - Steam-Inhaltsverzeichnis extrahieren und vorbereiten     │
│                                                             │
│ 2. SteamCMD-Einrichtung:                                    │
│    - SteamCMD installieren oder wiederverwenden               │
│    - Mit Steam Guard authentifizieren                        │
│                                                             │
│ 3. Veröffentlichungsphase:                                   │
│    - Depot VDF Konfigurationsdateien generieren              │
│    - App Build VDF Konfigurationsdateien generieren          │
│    - SteamCMD zum Hochladen auf Steam aufrufen               │
└─────────────────────────────────────────────────────────────┘
```

Dieses Design bietet mehrere Vorteile:
- Es wiederverwendet bestehende GitHub Release Artefakte und vermeidet, dieselben Ausgaben neu zu bauen.
- Es verwendet Self-hosted Runner fuer sichere Isolierung.
- Es unterstuetzt das Umschalten zwischen Preview-Modus und formellen Veröffentlichungszweigen.
- Es enthaelt vollstaendige Fehlerbehandlung und Protokollierung, was Fehlerdiagnosen erleichtert.

## Workflow-Implementierung

### Trigger-Parameter-Design

Unser Workflow unterstuetzt die folgenden Schluesselparameter:

```yaml
inputs:
  release:           # Portable-Version Release-Tag
    description: 'Version-Tag zur Veröffentlichung (z.B. v1.0.0)'
    required: true
  steam_preview:     # Ob ein Preview-Build generiert werden soll
    description: 'Ob der Preview-Modus verwendet werden soll'
    required: false
    default: 'false'
  steam_branch:      # Steam-Branch, der live gesetzt werden soll
    description: 'Ziel-Steam-Branch'
    required: false
    default: 'preview'
  steam_description: # Build-Beschreibung Ueberschreibung
    description: 'Build-Beschreibung'
    required: false
```

### Self-hosted Runner Konfiguration

Aus Sicherheitsgruenden verwenden wir einen Self-hosted Runner mit dem Label `steam`:

```yaml
runs-on:
  - self-hosted
  - Linux
  - X64
  - steam
```

Dies stellt sicher, dass Steam-Veröffentlichungen auf einem dedizierten Runner laufen und sensible Berechtigungen sicher isoliert bleiben.

### Nebenlaeufigkeitskontrolle

Um zu verhindern, dass Veröffentlichungen derselben Version sich gegenseitig stoeren, haben wir eine Nebenlaeufigkeitskontrolle konfiguriert:

```yaml
concurrency:
  group: portable-version-steam-${{ github.event.inputs.release }}
  cancel-in-progress: false
```

Beachten Sie, dass `cancel-in-progress: false` gesetzt ist, da Steam-Veröffentlichungen eine Weile dauern koennen und wir nicht wollen, dass ein neu getriggerter Lauf einen bereits hochladenden Lauf abbricht.

## Kern-Skript-Implementierung

### Veröffentlichungseingaben vorbereiten

Das Skript `prepare-steam-release-input.mjs` ist dafuer zustaendig, die fuer die Veröffentlichung erforderlichen Eingaben vorzubereiten:

```javascript
// GitHub Release Build-Manifest und Artefakt-Inventar herunterladen
const buildManifest = await downloadBuildManifest(releaseTag);
const artifactInventory = await downloadArtifactInventory(releaseTag);

// Komprimierte Archive fuer jede Plattform herunterladen
for (const platform of ['linux-x64', 'win-x64', 'osx-universal']) {
  const artifactUrl = getArtifactUrl(artifactInventory, platform);
  await downloadArtifact(artifactUrl, platform);
}

// In die Steam-Inhaltsverzeichnisstruktur extrahieren
await extractToSteamContent(sources, contentRoot);
```

### Steam Guard Authentifizierung

Steam erfordert, dass Konten mit Steam Guard geschuetzt sind, daher haben wir einen Code-Generierungsalgorithmus basierend auf Shared Secret implementiert:

```javascript
function generateSteamGuardCode(sharedSecret, timestamp = Date.now()) {
  const secret = decodeSharedSecret(sharedSecret);
  const time = Math.floor(timestamp / 1000 / 30);

  const timeBuffer = Buffer.alloc(8);
  timeBuffer.writeBigUInt64BE(BigInt(time));

  // HMAC-SHA1 zur Generierung eines zeitbasierten Einmalpassworts verwenden
  const hash = crypto.createHmac('sha1', secret)
    .update(timeBuffer)
    .digest();

  // In einen 5-stelligen Steam Guard Code umwandeln
  const code = steamGuardCode(hash);
  return code;
}
```

Diese Implementierung basiert auf Steam Guards TOTP-Mechanismus (Time-based One-Time Password) und generiert alle 30 Sekunden einen neuen Verifizierungscode.

### VDF-Konfigurationsgenerierung

VDF (Valve Data Format) ist das von Steam verwendete Konfigurationsformat. Wir muessen zwei Arten von VDF-Dateien generieren:

**Depot VDF** wird verwendet, um den Inhalt fuer jede Plattform zu konfigurieren:

```javascript
function buildDepotVdf(depotId, contentRoot) {
  return [
    '"DepotBuildConfig"',
    '{',
    `  "DepotID" "${escapeVdf(depotId)}"`,
    `  "ContentRoot" "${escapeVdf(contentRoot)}"`,
    '  "FileMapping"',
    '  {',
    '    "LocalPath" "*"',
    '    "DepotPath" "."',
    '    "recursive" "1"',
    '  }',
    '}'
  ].join('\n');
}
```

**App Build VDF** wird verwendet, um den gesamten Anwendungsbau zu konfigurieren:

```javascript
function buildAppBuildVdf(appId, depotBuilds, description, setLive) {
  const vdf = [
    '"appbuild"',
    '{',
    `  "appid" "${appId}"`,
    `  "desc" "${escapeVdf(description)}"`,
    `  "contentroot" "${escapeVdf(contentRoot)}"`,
    '  "buildoutput" "build_output"',
    '  "depots"',
    '  {'
  ];

  for (const [depotId, depotVdfPath] of Object.entries(depotBuilds)) {
    vdf.push(`    "${depotId}" "${depotVdfPath}"`);
  }

  if (setLive) {
    vdf.push(`  }`);
    vdf.push(`  "setlive" "${setLive}"`);
  }

  vdf.push('}');
  return vdf.join('\n');
}
```

### SteamCMD-Aufruf

Schliesslich wird der Upload durch Aufruf von SteamCMD durchgefuehrt:

```javascript
await runCommand(steamcmdPath, [
  '+login', steamUsername, steamPassword, steamGuardCode,
  '+run_app_build', appBuildPath,
  '+quit'
]);
```

Das ist der letzte Sprung im gesamten Workflow. Wenn es erfolgreich ist, ist die Veröffentlichung abgeschlossen.

## Multi-Plattform-Depot-Behandlung

Steam verwendet das Depot-System, um Inhalte fuer verschiedene Plattformen zu verwalten. Wir unterstuetzen drei Hauptdepots:

| Plattform | Depot-Kennung | Architektur-Unterstuetzung |
|-----|-----------|---------|
| Linux | `linux-x64` | x64_64 |
| Windows | `win-x64` | x64_64 |
| macOS | `osx-universal` | universal, x64_64, arm64 |

Jedes Depot hat sein eigenes Inhaltsverzeichnis und seine eigene VDF-Konfigurationsdatei. Dadurch laden Benutzer auf verschiedenen Plattformen nur den Inhalt herunter, den sie tatsaechlich benoetigen.

## Veröffentlichungsablauf

### Schritt 1: GitHub Release vorbereiten

Erstellen Sie zuerst ein GitHub Release im `portable-version`-Repository, das folgende enthaelt:
- Komprimierte Archive fuer jede Plattform
- Build-Manifest (`{tag}.build-manifest.json`)
- Artefakt-Inventar (`{tag}.artifact-inventory.json`)

### Schritt 2: Steam-Veröffentlichungs-Workflow triggern

Triggeren Sie den Workflow manuell ueber GitHub Actions und fuellen Sie die erforderlichen Parameter aus:
- `release`: der Versions-Tag, der veroeffentlicht werden soll, z.B. `v1.0.0`
- `steam_branch`: der Ziel-Branch, z.B. `preview` oder `public`
- `steam_preview`: ob der Preview-Modus verwendet werden soll

### Schritt 3: Den Veröffentlichungsablauf automatisch ausfuehren

Der Workflow fuehrt automatisch folgende Schritte durch:

1. GitHub Release Artefakte herunterladen und extrahieren.
2. SteamCMD installieren oder aktualisieren.
3. Steam VDF Konfigurationsdateien generieren.
4. Mit Steam Guard authentifizieren.
5. Inhalte auf das Steam CDN hochladen.
6. Den angegebenen Branch live setzen.

Sobald diese Sequenz abgeschlossen ist, ist der gesamte Veröffentlichungspfad abgedeckt.

## Konfigurationsanleitung

### Erforderliche Secrets Konfiguration

Konfigurieren Sie die folgenden Secrets in den GitHub-Repository-Einstellungen:

| Secret-Name | Beschreibung |
|-----------|------|
| `STEAM_USERNAME` | Steam-Konto Benutzername |
| `STEAM_PASSWORD` | Steam-Konto Passwort |
| `STEAM_SHARED_SECRET` | Steam Guard Shared Secret (optional) |
| `STEAM_GUARD_CODE` | Steam Guard Code (optional) |
| `STEAM_APP_ID` | Steam-Anwendungs-ID |
| `STEAM_DEPOT_ID_LINUX` | Linux Depot ID |
| `STEAM_DEPOT_ID_WINDOWS` | Windows Depot ID |
| `STEAM_DEPOT_ID_MACOS` | macOS Depot ID |

An diesen Einstellungen gibt es nichts Besonderes. Sie muessen einfach alle erwarteten Werte haben.

### Umgebungsvariablen-Konfiguration

| Variablenname | Beschreibung | Standardwert |
|---------|------|--------|
| `PORTABLE_VERSION_STEAMCMD_ROOT` | SteamCMD Installationsverzeichnis | `~/.local/share/portable-version/steamcmd` |

## Bewaehrte Praktiken

### Steam Guard Authentifizierungsverwaltung

Beim ersten Lauf muessen Sie den Steam Guard Code manuell eingeben. Danach wird empfohlen, das Shared Secret zu konfigurieren, damit der Code automatisch generiert werden kann. Das vermeidet manuelle Eingriffe bei jeder Veröffentlichung.

SteamCMD speichert den Login-Token und kann ihn bei spaeteren Läufen wiederverwenden. Sie muessen dennoch auf die Token-Ablauf achten, da nach Ablauf eine erneute Authentifizierung erforderlich ist.

### Inhaltsverzeichnisstruktur

Stellen Sie sicher, dass die Steam-Inhaltsverzeichnisstruktur korrekt ist:

```
steam-content/
├── linux-x64/     # Linux Plattforminhalt
├── win-x64/       # Windows Plattforminhalt
└── osx-universal/ # macOS Universal Binary Inhalt
```

Jedes Verzeichnis sollte die vollstaendigen Anwendungsdateien fuer die entsprechende Plattform enthalten.

### Preview-Modus Verwendung

Der Preview-Modus setzt keinen Branch live und eignet sich daher zum Testen und Validieren:

```yaml
if [ "$STEAM_PREVIEW_INPUT" = 'true' ]; then
  cmd+=(--preview)
fi
```

Dadurch koennen Sie zuerst auf Steam hochladen zur Verifizierung und dann, wenn alles stimmt, auf den formellen Branch wechseln.

### Fehlerbehandlung und Protokollierung

Die Skripte enthaelten vollstaendige Fehlerbehandlung und Protokollierung:
- Validierung, dass das GitHub Release existiert.
- Erforderliche Metadatendateien pruefen.
- Sicherstellen, dass Plattforminhalte vorhanden sind.
- Einen GitHub Actions Zusammenfassungsbericht generieren.

Diese Informationen sind sowohl fuer Debugging als auch fuer Auditing hochgradig wertvoll.

### Artefaktverwaltung

Der Workflow generiert zwei Arten von Artefakten:
- `portable-steam-release-preparation-{tag}`: Veröffentlichungsvorbereitungsmetadaten
- `portable-steam-build-metadata-{tag}`: Steam Build Metadaten

Diese Artefakte koennen fuer spaetere Auditing und Debugging verwendet werden. Ein Aufbewahrungszeitraum von 30 Tagen ist ein praktischer Standardwert.

## Reale Nutzung

Im HagiCode-Projekt hat dieser automatisierte Veröffentlichungsworkflow bereits erfolgreich ueber mehrere Versionen gelaufen. Der gesamte Weg von GitHub Release zur Steam-Plattform ist vollautomatisiert und erfordert keine manuelle Eingriffe.

Dies hat sowohl die Veröffentlichungseffizienz als auch die Zuverlaessigkeit deutlich verbessert. Frueher dauerte die manuelle Veröffentlichung einer Version mehr als 30 Minuten. Jetzt ist der gesamte Prozess in nur wenigen Minuten abgeschlossen.

Noch wichtiger ist, dass der automatisierte Workflow die Chance menschlicher Fehler verringert. Jede Veröffentlichung folgt demselben standardisierten Prozess, was die Ergebnisse vorhersehbarer macht.
