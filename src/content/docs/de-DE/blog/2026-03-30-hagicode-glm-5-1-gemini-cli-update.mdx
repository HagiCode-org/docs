## Volle GLM-5.1-Unterstuetzung und Gemini CLI-Integration: HagiCodes Weg der Multi-Modell-Evolution

> Dieser Artikel stellt zwei bedeutende aktuelle Aktualisierungen der HagiCode-Plattform vor: die volle Unterstuetzung fuer das Zhipu AI GLM-5.1-Modell und die erfolgreiche Integration von Gemini CLI als zehntes Agent CLI. Zusammen stärken diese Aktualisierungen die Multi-Modell-Fähigkeiten und das Multi-CLI-Ökosystem der Plattform weiter.

:::note[Zeitkontext-Hinweis]
Dieser Artikel dokumentiert den Multi-CLI-Erweiterungsfortschritt zum Zeitpunkt der Veröffentlichung. Für den aktuell dauerhaft unterstützten Umfang konsultieren Sie bitte [AI Agent CLI Installation](/en/related-software-installation/ai-agent-cli/) und den [Installationsleitfaden](/en/installation); das hier erwähnte `IFlowCli` stellt lediglich eine historische Konfiguration zu jenem Zeitpunkt dar und gehört nicht zum aktuellen aktiven Unterstützungsumfang.
:::

## Hintergrund

Die Zeit vergeht wirklich schnell. Die Entwicklung großer Sprachmodelle steigt wie Bambus im Frühling. Noch vor kurzem haben wir für „eine KI, die Code schreiben kann" gejubelt. Jetzt befinden wir uns bereits in einer Ära der Multi-Modell-Zusammenarbeit und Multi-Tool-Integration. Ist das aufregend? Vielleicht. Schließlich brauchen Entwickler nie nur das Werkzeug selbst, sondern die Leichtigkeit, sich an verschiedene Szenarien anzupassen und bei Bedarf flexibel zu wechseln.

Als KI-unterstützte Programmierplattform hat HagiCode kürzlich zwei wichtige Entwicklungen begrüßt: erstens die vollständige Integration des GLM-5.1-Modells von Zhipu AI; zweitens die offizielle Aufnahme von Gemini CLI als zehntes unterstütztes Agent CLI. Diese beiden Aktualisierungen klingen vielleicht nicht bahnbrechend, sind aber zweifellos gute Nachrichten für die kontinuierliche Reifung der Plattform.

GLM-5.1 ist das neueste Flaggschiffmodell von Zhipu AI. Verglichen mit GLM-5.0 bietet es stärkeres Reasoning, tieferes Codeverständnis und reibungsloseres Tool-Aufrufen. Noch wichtiger ist, dass es das erste GLM-Modell ist, das Bildeingabe unterstützt. Was bedeutet das? Nutzer können die KI direkt auf einen Screenshot schauen lassen, anstatt das Problem mühsam in Worten beschreiben zu müssen. Sobald Sie diese Bequemlichkeit erlebt haben, verstehen Sie sofort ihren Wert.

Gleichzeitig hat HagiCode durch die `HagiCode.Libs.Providers`-Architektur erfolgreich Gemini CLI in die Plattform integriert. Dies ist nun das zehnte Agent CLI. Um ehrlich zu sein, diesen Punkt erreicht zu haben, vermittelt ein gewisses Gefühl der Errungenschaft.

Es ist auch erwähnenswert, dass die Bildupload-Funktion von HagiCode es Nutzern ermöglicht, direkt über Screenshots mit der KI zu kommunizieren. Selbst mit GLM 4.7 funktioniert die Plattform gut und hat bereits viele wichtige Build-Aufgaben abgeschlossen. GLM-5.1 geht natürlich noch einen Schritt weiter.

## Ueber HagiCode

Der in diesem Artikel vorgestellte Ansatz stammt aus unserer praktischen Erfahrung im [HagiCode](https://hagicode.com)-Projekt. HagiCode ist eine Open-Source-KI-unterstützte Programmierplattform, die Entwicklern durch eine Multi-Modell-, Multi-CLI-Architektur einen flexiblen und leistungsstarken KI-Programmierassistenten bieten soll. Projekt-Repository: [github.com/HagiCode-org/site](https://github.com/HagiCode-org/site)

## Multi-CLI-Architekturdesign

Eine der Kernstärken von HagiCode ist die Unterstützung mehrerer KI-Programmier-CLI-Tools durch eine einheitliche Abstraktionsschicht. Der Vorteil dieses Designs ist eigentlich recht einfach: neue Tools können hinzukommen, alte Tools können bleiben, und die Codebasis verwandelt sich nicht in Chaos. Um fair zu sein, so würde jeder gerne sein Leben gestalten.

### AIProviderType-Enum

Die Plattform definiert unterstützte CLI-Anbietertypen durch den `AIProviderType`-Enum:

```csharp
public enum AIProviderType
{
    ClaudeCodeCli = 0,    // Claude Code CLI
    CodexCli = 1,          // GitHub Copilot Codex
    GitHubCopilot = 2,     // GitHub Copilot
    CodebuddyCli = 3,     // Codebuddy CLI
    OpenCodeCli = 4,      // OpenCode CLI
    IFlowCli = 5,         // IFlow CLI
    HermesCli = 6,        // Hermes CLI
    QoderCli = 7,         // Qoder CLI
    KiroCli = 8,          // Kiro CLI
    KimiCli = 9,          // Kimi CLI
    GeminiCli = 10,       // Gemini CLI (neu)
}
```

Wie Sie sehen, tritt Gemini CLI als zehntes Mitglied dieser Familie bei. Jedes CLI hat seine eigenen Merkmale und Nutzungsszenarien, sodass Nutzer flexibel nach ihren Bedürfnissen wählen können. Viele Wege führen nach Rom; einige sind einfach leichter als andere.

### Provider-Architektur

`HagiCode.Libs.Providers` bietet eine einheitliche Provider-Schnittstelle, die jede CLI-Integration standardisiert und übersichtlich macht. Am Beispiel von Gemini CLI:

```csharp
public class GeminiProvider : ICliProvider<GeminiOptions>
{
    private static readonly string[] DefaultExecutableCandidates = ["gemini", "gemini-cli"];
    private const string ManagedBootstrapArgument = "--acp";

    public string Name => "gemini";
    public bool IsAvailable => _executableResolver.ResolveFirstAvailablePath(DefaultExecutableCandidates) is not null;
}
```

Die Vorteile dieses Designs sind:
- Die Integration eines neuen CLI erfordert nur die Implementierung einer Provider-Klasse
- Einheitliche Lebenszyklusverwaltung und Session-Pooling
- Automatische Alias-Auflösung und ausführbare Datei-Erkennung

Kurz gesagt, dieses Design verwandelt Kompliziertes in Einfacheres und macht das Leben ein wenig leichter.

### Provider-Registry

Die Provider-Registry übernimmt automatisch die Alias-Zuordnung und Registrierung:

```csharp
if (provider is GeminiProvider)
{
    registry.Register(provider.Name, provider, ["gemini-cli"]);
    continue;
}
```

Das bedeutet, dass Nutzer Gemini CLI mit entweder `gemini` oder `gemini-cli` aufrufen können, und das System erkennt es automatisch. Es ist wie ein Freund mit sowohl einem formellen Namen als auch einem Spitznamen - auf beide Arten weiß jeder, wen Sie meinen.

## GLM-5.1-Modellunterstützung

GLM-5.1 ist das neueste Flaggschiffmodell von Zhipu AI, und HagiCode hat die volle Unterstützung dafür abgeschlossen.

### Sekundärberufskatalog

HagiCode verwaltet alle unterstützten Modelle über den Sekundärberufskatalog. Hier ist die Konfiguration für die GLM-Reihe:

| Modell-ID | Name | SupportsImage | Kompatible CLI-Familien |
|----------|------|----------------|-------------------------|
| `glm-4.7` | GLM 4.7 | - | claude, codebuddy, hermes, qoder, kiro |
| `glm-5` | GLM 5 | - | claude, codebuddy, hermes, qoder, kiro |
| `glm-5-turbo` | GLM 5 Turbo | - | claude, codebuddy, hermes, qoder, kiro |
| `glm-5.0` | GLM 5.0 (Legacy) | - | claude, codebuddy, hermes, qoder, kiro |
| `glm-5.1` | GLM 5.1 | **true** | claude, codebuddy, hermes, qoder, kiro |

Die wesentlichen Merkmale von GLM-5.1 lassen sich wie folgt zusammenfassen:
- Ein eigenständiger Versionsidentifikator ohne Legacy-Ballast
- Das erste GLM-Modell mit Unterstützung für Bildeingabe
- Stärkeres Reasoning und Codeverständnis
- Breite Multi-CLI-Kompatibilität

### GLM-5.1 vs GLM-5.0

Auf Code-Ebene zeigt sich der wesentliche Unterschied zwischen GLM-5.1 und GLM-5.0 hier:

```csharp
// GLM-5.0 (Legacy) - enthält spezielle Beibehaltungslogik
private const string Glm50CodebuddySecondaryProfessionId = "secondary-glm-5-codebuddy";
private const string Glm50CodebuddyModelValue = "glm-5.0";

// GLM-5.1 - eigenständiger neuer Modellidentifikator
private const string Glm51SecondaryProfessionId = "secondary-glm-5-1";
private const string Glm51ModelValue = "glm-5.1";
```

GLM-5.0 trägt das Label „Legacy", da es ein alter Versionsidentifikator ist, der aus Gründen der Abwärtskompatibilität beibehalten wurde. GLM-5.1 hingegen ist eine brandneue eigenständige Version ohne historische Last. Manches gehört in die Vergangenheit; anderes reist leichter und bewegt sich schneller.

### GLM-5.1 konfigurieren

Hier ist ein Konfigurationsbeispiel für die Verwendung von GLM-5.1 in HagiCode:

```json
{
  "primaryProfessionId": "profession-claude-code",
  "secondaryProfessionId": "secondary-glm-5-1",
  "model": "glm-5.1",
  "reasoning": "high"
}
```

## Bildupload-Funktion

HagiCodes Bildunterstützung wird über die Eigenschaft `SupportsImage` auf `SecondaryProfession` implementiert:

```csharp
public class HeroSecondaryProfessionSettingDto
{
    public bool SupportsImage { get; set; }
}
```

Im Sekundärberufskatalog sieht die GLM-5.1-Konfiguration wie folgt aus:

```json
{
  "id": "secondary-glm-5-1",
  "supportsImage": true
}
```

Das bedeutet, dass Nutzer Screenshots direkt zur KI-Analyse hochladen können, beispielsweise:
- Screenshots von Fehlermeldungen
- Probleme in einer UI-Ansicht
- Datenvisualisierungsdiagramme
- Code-Ausführungsergebnisse

Es ist nicht mehr nötig, alles manuell zu beschreiben - laden Sie einfach den Screenshot hoch. Die Bequemlichkeit dieser Funktion wird offensichtlich, sobald Sie sie verwendet haben. Manchmal sagt ein Blick mehr als eine lange Erklärung.

## Gemini CLI-Integration

Als zehntes Agent CLI wird Gemini CLI über die Standard-Provider-Architektur in HagiCode integriert.

### Konfigurationsoptionen

Gemini CLI unterstützt eine reiche Auswahl an Konfigurationsoptionen:

```csharp
public class GeminiOptions
{
    public string? ExecutablePath { get; set; }
    public string? WorkingDirectory { get; set; }
    public string? SessionId { get; set; }
    public string? Model { get; set; }
    public string? AuthenticationMethod { get; set; }
    public string? AuthenticationToken { get; set; }
    public Dictionary<string, string?> AuthenticationInfo { get; set; }
    public Dictionary<string, string?> EnvironmentVariables { get; set; }
    public string[] ExtraArguments { get; set; }
    public TimeSpan? StartupTimeout { get; set; }
    public CliPoolSettings? PoolSettings { get; set; }
}
```

Diese Optionen decken alles von der Grundeinrichtung bis zu erweiterten Funktionen ab und geben Nutzern die Flexibilität, das CLI nach ihren eigenen Bedürfnissen zu konfigurieren. Der Workflow jedes Nutzers ist unterschiedlich, daher ist ein wenig Flexibilität immer willkommen.

### ACP-Kommunikationsprotokoll

Gemini CLI unterstützt das ACP (Agent Communication Protocol), das HagiCodes einheitlicher CLI-Kommunikationsstandard ist. Durch ACP können verschiedene CLIs auf konsistente Weise mit der Plattform interagieren, was die Integrationsarbeit erheblich vereinfacht. Kurz gesagt, es standardisiert die komplizierten Teile, damit alle leichter arbeiten können.

## HagiCodes eigene Build-Praxis

Was die praktische Anwendung betrifft, ist das beste Beispiel der Build-Workflow der HagiCode-Plattform selbst. Der Entwicklungsprozess von HagiCode macht bereits umfassend von den KI-Fähigkeiten Gebrauch.

### Funktioniert gut auch mit GLM 4.7

Das Plattformdesign von HagiCode ist gut optimiert, sodass es selbst mit GLM 4.7 eine gute Entwicklungserfahrung bieten kann. Die Plattform hat bereits mehrere wichtige Build-Projekte abgeschlossen, darunter:
- Integration mehrerer CLI-Provider
- Implementierung der Bildupload-Funktion
- Dokumentationsgenerierung und Inhaltspublishing

Das ist eigentlich eine gute Sache. Nicht jeder braucht immer das Neueste. Was einem am besten passt, ist oft das Wichtigste.

### GLM-5.1 bringt mehr mit weniger Aufwand

Nach dem Upgrade auf GLM-5.1 werden diese Fähigkeiten noch stärker:
- Stärkeres Codeverständnis, reduziert die Hin- und Her-Kommunikation
- Präzisere Abhängigkeitsanalyse, zeigt sofort die richtige Richtung
- Effizientere Fehlerdiagnose, lokalisiert Probleme schneller
- Unterstützung für Bildeingabe, beschleunigt Problembeschreibungen

Es ist wie der Wechsel vom Fahrrad zum Auto. Sie erreichen dasselbe Ziel, aber Geschwindigkeit und Komfort sind nicht gleich.

## Bewährte Praktiken für Multi-CLI-Integration

`HagiCode.Libs.Providers` bietet einen einheitlichen Mechanismus für Registrierung und Nutzung:

```csharp
services.AddHagiCodeLibs();

var gemini = serviceProvider.GetRequiredService<ICliProvider<GeminiOptions>>();
var codebuddy = serviceProvider.GetRequiredService<ICliProvider<CodebuddyOptions>>();
var hermes = serviceProvider.GetRequiredService<ICliProvider<HermesOptions>>();
```

Dieses Dependency-Injection-Design hält die Verwendung über verschiedene CLIs sehr übersichtlich und macht Unit-Testing und Mocking zudem bequemer. Sauberer Code ist eine Art, verantwortungsvoll mit sich selbst umzugehen.

### Hinweise

Einige Dinge sollten bei der tatsächlichen Verwendung beachtet werden:

1. **API-Schlüsselkonfiguration**: Stellen Sie sicher, dass `ANTHROPIC_AUTH_TOKEN` korrekt gesetzt ist, sonst kann das Modell nicht aufgerufen werden
2. **Modellverfügbarkeit**: GLM-5.1 muss vom entsprechenden Modellanbieter aktiviert werden
3. **Bildfunktion**: Nur Modelle mit `supportsImage: true` können den Bildupload verwenden
4. **CLI-Installation**: Stellen Sie vor der Verwendung von Gemini CLI sicher, dass `gemini` oder `gemini-cli` im System-PATH vorhanden ist

Das sind vielleicht kleine Details, aber schlecht gehandhabte kleine Details können zu großen Problemen werden, also sind sie es wert, beachtet zu werden.

## Zusammenfassung

Mit vollständiger Unterstützung für GLM-5.1 und der erfolgreichen Integration von Gemini CLI stärkt HagiCode seine Fähigkeiten als Multi-Modell-, Multi-CLI-KI-Programmierplattform weiter. Diese Aktualisierungen geben Nutzern nicht nur mehr Auswahl, sondern zeigen auch HagiCodes zukunftsorientierte Architektur und Skalierbarkeit.

Die Bildunterstützung von GLM-5.1 in Kombination mit HagiCodes Screenshot-Upload-Funktion ermöglicht es der KI, „aus dem Bild zu verstehen", und reduziert die Kosten der Problembeschreibung erheblich. Mit Unterstützung für zehn CLIs können Nutzer flexibel den KI-Programmierassistenten wählen, der am besten zu ihren Präferenzen und Szenarien passt. Mehr Auswahl ist fast immer eine gute Sache.

Am wichtigsten ist, dass HagiCodes eigene Build-Praxis beweist, dass die Plattform auch mit GLM 4.7 bereits gut läuft und komplexe Aufgaben abschließen kann, während ein Upgrade auf GLM-5.1 die Entwicklungseffizienz weiter verbessern kann. Das Leben ist oft so: Man braucht nicht immer das absolut Beste, nur das, was passt. Natürlich, wenn das, was passt, noch besser werden kann, umso besser.

Wenn Sie an einer Multi-Modell-, Multi-CLI-KI-Programmierplattform interessiert sind, probieren Sie HagiCode aus - Open Source, kostenlos und noch in Entwicklung. Ausprobieren kostet nichts, und es stellt sich vielleicht als genau das heraus, was Sie brauchen.
