## Hintergrund

Die Zeit vergeht wirklich schnell, die Entwicklung großer Sprachmodelle schießt wie Bambus im Frühling in die Höhe. Früher jubelten wir noch über "eine KI, die Code schreiben kann", heute sind wir in einer Ära der Multi-Modell-Kooperation und Multi-Tool-Integration angelangt. Ist das interessant? Vielleicht, schließlich brauchen Entwickler nie nur das Werkzeug selbst, sondern eine Gelassenheit, die sich an verschiedene Szenarien anpassen und flexibel wechseln kann.

HagiCode als KI-unterstützte Kodierungsplattform hat kürzlich zwei wichtige Ereignisse erlebt:一是 die vollständige Integration des GLM-5.1-Modells von Zhipu AI und二是 dass Gemini CLI offiziell der zehnte unterstützte Agent CLI wurde. Diese beiden Dinge sind nicht groß, aber auch nicht klein – für die Vervollkommnung der Plattform ist es auf jeden Fall eine gute Sache.

GLM-5.1 ist das neueste Flaggschiff-Modell von Zhipu AI. Im Vergleich zu GLM-5.0 sind die Schlussfolgerungsfähigkeit stärker, das Codeverständnis tiefer und der Toolaufruf reibungsloser geworden. Wichtiger noch: es ist das erste GLM-Modell, das Bildeingabe unterstützt – was bedeutet das? Dass Benutzer direkt Screenshots machen können, damit die KI das Problem sieht, ohne sich mühsam zu beschreiben. Diese Bequemlichkeit versteht man, wenn man sie benutzt hat.

Gleichzeitig hat HagiCode durch die HagiCode.Libs.Providers-Architektur Gemini CLI erfolgreich integriert. Das ist der zehnte Agent CLI – ehrlich gesagt, bis zu diesem Punkt zu kommen, ist schon ein kleines Erfolgsgefühl.

Erwähnenswert ist, dass HagiCodes Bild-Upload-Funktion es Benutzern ermöglicht, direkt Screenshots mit der KI zu kommunizieren. Selbst wenn die GLM 4.7-Version ausgeführt wird, kann die Plattform immer noch gut laufen und hat bereits geholfen, viele wichtige Build-Arbeiten abzuschließen. Was GLM-5.1 betrifft? Das wird natürlich noch einen Schritt weiter gehen.

## Über HagiCode

Das in diesem Artikel geteilte Konzept stammt aus unserer praktischen Erfahrung im [HagiCode](https://hagicode.com) Projekt. Projektadresse: [github.com/HagiCode-org/site](https://github.com/HagiCode-org/site)

## Multi-CLI-Architekturdesign

Einer der Kernvorteile von HagiCode ist die Unterstützung verschiedener KI-Programmier-CLI-Tools durch eine einheitliche Abstraktionsschicht. Die Vorteile dieses Designs sind, wenn man es direkt sagt, eigentlich nur das: Neue Dinge können hereinkommen, alte Dinge können bleiben, und der Code wird nicht chaotisch. Schließlich hofft jeder, dass das Leben so sein könnte, oder?

### AIProviderType-Enumeration

Die Plattform definiert die unterstützten CLI-Anbietertypen durch die `AIProviderType`-Enumeration:

```csharp
public enum AIProviderType
{
    ClaudeCodeCli = 0,    // Claude Code CLI
    CodexCli = 1,          // GitHub Copilot Codex
    GitHubCopilot = 2,     // GitHub Copilot
    CodebuddyCli = 3,     // Codebuddy CLI
    OpenCodeCli = 4,      // OpenCode CLI
    IFlowCli = 5,         // IFlow CLI
    HermesCli = 6,        // Hermes CLI
    QoderCli = 7,         // Qoder CLI
    KiroCli = 8,          // Kiro CLI
    KimiCli = 9,          // Kimi CLI
    GeminiCli = 10,       // Gemini CLI (neu hinzugefügt)
}
```

Man可以看到，Gemini CLI ist als zehntes Mitglied dieser großen Familie beigetreten. Jede CLI hat ihre einzigartigen Merkmale und Anwendungsszenarien, Benutzer können entsprechend ihren Bedürfnissen flexibel wählen. Schließlich führen alle Wege nach Rom, manche Straßen sind nur etwas einfacher zu begehen, andere etwas kurviger.

### Provider-Architektur

HagiCode.Libs.Providers bietet eine einheitliche Provider-Schnittstelle, sodass die Integration jeder CLI normiert und prägnant wird. Am Beispiel von Gemini CLI:

```csharp
public class GeminiProvider : ICliProvider<GeminiOptions>
{
    private static readonly string[] DefaultExecutableCandidates = ["gemini", "gemini-cli"];
    private const string ManagedBootstrapArgument = "--acp";

    public string Name => "gemini";
    public bool IsAvailable => _executableResolver.ResolveFirstAvailablePath(DefaultExecutableCandidates) is not null;
}
```

Die Vorteile dieses Designs sind:
- Die Integration neuer CLI erfordert nur die Implementierung einer Provider-Klasse
- Einheitliche Lebenszyklusverwaltung und Session-Pooling
- Automatisierte Aliasauflösung und ausführbare Dateisuche

Wenn man es direkt sagt: Dieses Design macht komplexe Dinge einfach und macht das Leben ein bisschen angenehmer.

### Provider Registry

Die Provider Registry verarbeitet Aliaszuordnungen und Registrierungen automatisch:

```csharp
if (provider is GeminiProvider)
{
    registry.Register(provider.Name, provider, ["gemini-cli"]);
    continue;
}
```

Das bedeutet, dass Benutzer `gemini` oder `gemini-cli` verwenden können, um Gemini CLI aufzurufen, das System erkennt es automatisch. Das ist wie wenn Sie viele Freunde haben, einige werden beim Vornamen gerufen, andere beim Spitznamen – egal wie, es ist immer er.

## GLM-5.1-Modellunterstützung

GLM-5.1 ist das neueste Flaggschiff-Modell von Zhipu AI, HagiCode hat die volle Unterstützung abgeschlossen.

### Secondary Professions Catalog

HagiCode verwaltet alle unterstützten Modelle durch den Secondary Professions Catalog. Hier ist die Konfiguration der GLM-Serie:

| Model ID | Name | SupportsImage | Compatible CLI Families |
|----------|------|----------------|-------------------------|
| `glm-4.7` | GLM 4.7 | - | claude, codebuddy, hermes, qoder, kiro |
| `glm-5` | GLM 5 | - | claude, codebuddy, hermes, qoder, kiro |
| `glm-5-turbo` | GLM 5 Turbo | - | claude, codebuddy, hermes, qoder, kiro |
| `glm-5.0` | GLM 5.0 (Legacy) | - | claude, codebuddy, hermes, qoder, kiro |
| `glm-5.1` | GLM 5.1 | **true** | claude, codebuddy, hermes, qoder, kiro |

Die Schlüsseleigenschaften von GLM-5.1 können zusammengefasst werden als:
- Unabhängige Versionkennung, kein legacy-Ballast
- Erstes GLM-Modell mit Bildeingabeunterstützung
- Stärkere Schlussfolgerungsfähigkeit und Codeverständnis
- Breite Multi-CLI-Kompatibilität

### GLM-5.1 vs GLM-5.0

Auf Codeebene gesehen, der Hauptunterschied zwischen GLM-5.1 und GLM-5.0:

```csharp
// GLM-5.0 (Legacy) - hat spezielle Behaltenslogik
private const string Glm50CodebuddySecondaryProfessionId = "secondary-glm-5-codebuddy";
private const string Glm50CodebuddyModelValue = "glm-5.0";

// GLM-5.1 - unabhängige neue Modellkennung
private const string Glm51SecondaryProfessionId = "secondary-glm-5-1";
private const string Glm51ModelValue = "glm-5.1";
```

GLM-5.0 trägt die "Legacy"-Markierung und ist eine aus Gründen der Rückwärtskompatibilität beibehaltene alte Versionkennung. GLM-5.1 ist eine völlig neue unabhängige Version ohne historischen Ballast. Das ist wie bei manchen Leuten, die immer in der Vergangenheit leben; und andere, die leicht gestartet sind und schneller gehen.

### GLM-5.1 konfigurieren

## Gemini CLI Integration

Gemini CLI als zehnter Agent CLI wurde durch die Standard-Provider-Architektur in HagiCode integriert.

### Konfigurationsoptionen

Gemini CLI unterstützt umfangreiche Konfigurationsoptionen:

```csharp
public class GeminiOptions
{
    public string? ExecutablePath { get; set; }
    public string? WorkingDirectory { get; set; }
    public string? SessionId { get; set; }
    public string? Model { get; set; }
    public string? AuthenticationMethod { get; set; }
    public string? AuthenticationToken { get; set; }
    public Dictionary<string, string?> AuthenticationInfo { get; set; }
    public Dictionary<string, string?> EnvironmentVariables { get; set; }
    public string[] ExtraArguments { get; set; }
    public TimeSpan? StartupTimeout { get; set; }
    public CliPoolSettings? PoolSettings { get; set; }
}
```

Diese Optionen decken alles von der Grundkonfiguration bis zu erweiterten Funktionen ab, Benutzer können sie entsprechend ihren Bedürfnissen flexibel konfigurieren. Schließlich sind die Bedürfnisse jedes anders, etwas Flexibilität ist immer gut.

### ACP-Kommunikationsprotokoll

Gemini CLI unterstützt das ACP (Agent Communication Protocol) Kommunikationsprotokoll, dies ist der einheitliche CLI-Kommunikationsstandard von HagiCode. Durch ACP können verschiedene CLI auf konsistente Weise mit der Plattform interagieren, was die Integrationsarbeit erheblich vereinfacht. Wenn man es direkt sagt: Es macht komplexe Dinge einheitlich, damit alle etwas entspannter sein können.

## HagiCode eigene Build-Praxis

Wenn es um Praxis geht, ist das beste Beispiel der Build-Prozess der HagiCode-Plattform selbst. Der Entwicklungsprozess von HagiCode hat bereits die KI-Fähigkeiten voll genutzt:

### Gut laufen mit GLM 4.7

Das Design der HagiCode-Plattform ist gut optimiert, selbst mit GLM 4.7 kann man eine gute Entwicklungserfahrung erzielen. Die Plattform hat bereits geholfen, mehrere wichtige Build-Projekte abzuschließen, einschließlich:

- Stärkere Codeverständnisfähigkeit, reduziert Hin-und-Her-Kommunikation
- Genauere Abhängigkeitsanalyse, weist einmalig in die richtige Richtung
- Effizientere Fehlerdiagnose, schnelle Problemlokalisierung
- Unterstützung von Bildeingabe, beschleunigt Problembeschreibung

Das ist wie vom Fahrrad zum Auto wechseln, man kann an dieselben Orte kommen, nur Geschwindigkeit und Komfort sind anders.

## Best Practices für Multi-CLI-Integration

HagiCode.Libs.Providers bietet einen einheitlichen Registrierungs- und Verwendungsmechanismus:

```csharp
services.AddHagiCodeLibs();

var gemini = serviceProvider.GetRequiredService<ICliProvider<GeminiOptions>>();
var codebuddy = serviceProvider.GetRequiredService<ICliProvider<CodebuddyOptions>>();
var hermes = serviceProvider.GetRequiredService<ICliProvider<HermesOptions>>();
```

 Dieses Dependency-Injection-Design macht die Verwendung jeder CLI sehr prägnant und erleichtert auch Unit-Tests und Mocking. Schließlich, sauberer Code zu schreiben, ist auch eine Verantwortung gegen sich selbst.

### Zu beachtende Punkte

In der praktischen Verwendung gibt es einige Dinge zu beachten:

1. **API Key Konfiguration**: Stellen Sie sicher, dass `ANTHROPIC_AUTH_TOKEN` korrekt eingestellt ist, sonst kann das Modell nicht aufgerufen werden
2. **Modellverfügbarkeit**: GLM-5.1 muss beim entsprechenden Modellanbieter berechtigt werden
3. **Bildfunktion**: Nur Modelle mit `supportsImage: true` können die Bild-Upload-Funktion verwenden
4. **CLI Installation**: Stellen Sie vor der Verwendung von Gemini CLI sicher, dass `gemini` oder `gemini-cli` im system PATH ist

Das sind alles Kleinigkeiten, aber wenn Kleinigkeiten nicht gut behandelt werden, können sie zu großen Dingen werden. Also besser etwas aufpassen.
