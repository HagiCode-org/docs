## Tippen ist langsamer als Sprechen, und Sprechen ist langsamer als ein Screenshot - Multimodale Eingabepraktiken fuer KI-Programmierassistenten

> Code schreiben hat ein Geschwindigkeitslimit, egal wie schnell Sie tippen. Manchmal dauert etwas, das man in einem Satz sagen könnte, ewig zum Tippen; manchmal erklärt ein einziger Screenshot alles, und Sie müssen es trotzdem mit einem Berg Text beschreiben. Dieser Artikel handelt davon, worauf wir beim Bau von [HagiCode](https://hagicode.com) gestoßen sind, von der Spracherkennung bis zum Bildupload. Am Ende wollten wir einfach einen KI-Programmierassistenten ein wenig benutzerfreundlicher machen.

## Hintergrund

Beim Bau von HagiCode bemerkten wir ein Problem - oder vielmehr ein Problem, das natürlich zutage trat, sobald Menschen begannen, es intensiv zu nutzen: sich allein auf Tippen zu verlassen, kann ermüdend sein.

Denken Sie darüber nach: Die Interaktion zwischen Nutzern und dem Agent ist ein Kernszenario. Aber wenn jeder Austausch ununterbrochenes Tippen an der Tastatur erfordert, ist die Effizienz nicht besonders groß:

1. **Tippen ist zu langsam**: Für komplizierte Probleme wie Fehlermeldungen oder UI-Probleme kann das Abtippen eine halbe Minute dauern, während das Aussprechen vielleicht nur zehn Sekunden in Anspruch nimmt. Diese Lücke ist real.

2. **Bilder sind direkter**: Manchmal wirft die UI einen Fehler, manchmal möchte man einen Entwurf vergleichen, und manchmal muss man eine Codestruktur zeigen. „Ein Bild sagt mehr als tausend Worte" mag ein altes Sprichwort sein, aber es gilt immer noch. Der KI das Problem direkt „sehen" zu lassen, ist viel klarer, als es ewig zu beschreiben.

3. **Interaktion sollte natürlich fühlen**: Moderne KI-Assistenten sollten Text, Sprache und Bilder unterstützen. Nutzer sollten die Eingabemethode wählen können, die sich am natürlichsten anfühlt.

Also beschlossen wir, HagiCode um Spracherkennung und Bildupload-Unterstützung zu erweitern, um die Agent-Interaktion bequemer zu gestalten. Wenn Nutzer ein wenig weniger tippen können, ist das bereits ein Gewinn.

## Ueber HagiCode

Die in diesem Artikel vorgestellten Lösungen stammen aus unserer praktischen Arbeit im [HagiCode](https://hagicode.com)-Projekt - oder genauer gesagt, aus Lehren, die wir beim Stolpern über einige Fallstricke gezogen haben.

HagiCode ist ein Open-Source-KI-Programmierassistent-Projekt mit einem einfachen Ziel: KI nutzen, um die Entwicklungseffizienz zu verbessern. Im Laufe des Baus wurde klar, dass Nutzer multimodale Eingaben stark wünschen. Manchmal ist ein einziger Satz schneller als ein langer Absatz, und manchmal ist ein Screenshot weit klarer als eine lange Erklärung.

Diese Bedürfnisse trieben uns voran, und so nahmen Funktionen wie Spracherkennung und Bilduploads schließlich Gestalt an. Nutzer können nun auf die natürlichste Weise mit der KI interagieren, und das fühlt sich gut an.

## Analyse

### Technische Herausforderungen bei der Spracherkennung

Beim Bau der Spracherkennung stießen wir auf ein kniffliges Problem: **Die Browser-WebSocket-API unterstützt keine benutzerdefinierten HTTP-Header**.

Der von uns gewählte Spracherkennungsdienst war die Doubao Spracherkennungs-API von ByteDance. Leider erfordert diese API, dass Authentifizierungsinformationen wie `accessToken` und `secretKey` über HTTP-Header übergeben werden. Das schuf einen sofortigen technischen Konflikt:

```javascript
// Die Browser-WebSocket-API unterstützt diesen Ansatz nicht
const ws = new WebSocket('wss://api.com/ws', {
  headers: {
    'Authorization': 'Bearer token'  // Nicht unterstützt
  }
});
```

Wir hatten im Grunde zwei Möglichkeiten:

1. **URL-Abfrageparameter-Ansatz**: Die Authentifizierungsinformationen in der URL unterbringen
   - Der Vorteil ist, dass die Implementierung einfach ist
   - Der Nachteil ist, dass die Zugangsdaten für das Frontend offengelegt werden, was unsicher ist; einige APIs erfordern zudem header-basierte Authentifizierung

2. **Backend-Proxy-Ansatz**: Einen WebSocket-Proxy auf dem Backend implementieren
   - Der Vorteil ist, dass die Zugangsdaten sicher auf dem Backend gespeichert bleiben und die Lösung vollständig kompatibel mit den API-Anforderungen ist
   - Der Nachteil ist, dass die Implementierung etwas komplexer ist

Letztendlich wählten wir den **Backend-Proxy-Ansatz**. Bei Sicherheit gibt es keine Kompromisse.

### Funktionale Anforderungen an Bilduploads

Unsere Anforderungen an Bilduploads waren eigentlich recht unkompliziert:

1. **Mehrere Upload-Methoden**: Klick zum Dateiauswählen, Drag & Drop und Einfügen aus der Zwischenablage
2. **Dateivalidierung**: Typbeschränkungen (PNG, JPG, WebP, GIF) und Größenlimits (5-10 MB) sind Grundanforderungen
3. **Benutzererfahrung**: Upload-Fortschritt, Vorschauen und Fehlermeldungen, damit Nutzer immer wissen, was passiert
4. **Sicherheit**: Serverseitige Validierung und Schutz gegen bösartige Datei-Uploads sind unerlässlich

## Loesung

### Spracherkennung: WebSocket-Proxy-Architektur

Wir entwarfen eine dreischichtige Architektur für die Spracherkennung und fanden einen funktionierenden Weg:

```
Browser WebSocket
       |
       | ws://backend/api/voice/ws
       | (binäre Audiodaten)
       v
Backend-Proxy
       |
       | wss://openspeech.bytedance.com/ (mit Auth-Header)
       v
Doubao API
```

**Kernkomponentenimplementierungen**:

1. **Frontend AudioWorklet-Prozessor**:

```javascript
class AudioProcessorWorklet extends AudioWorkletProcessor {
  process(inputs, outputs, parameters) {
    const input = inputs[0]?.[0];
    if (!input) return true;

    // Resampling auf 16 kHz (erforderlich von der Doubao API)
    const samples = this.resampleAudio(input, 48000, 16000);

    // Samples in 500-ms-Chunks akkumulieren
    this.accumulatedSamples.push(...samples);

    if (this.accumulatedSamples.length >= 8000) {
      // In 16-bit PCM konvertieren und senden
      const pcm = this.floatToPcm16(this.accumulatedSamples);
      this.port.postMessage({ type: 'audioData', data: pcm.buffer }, [pcm.buffer]);
      this.accumulatedSamples = [];
    }
    return true;
  }
}
```

2. **Backend-WebSocket-Handler** (C#):

```csharp
[HttpGet("ws")]
public async Task GetWebSocket()
{
    if (HttpContext.WebSockets.IsWebSocketRequest)
    {
        await _webSocketHandler.HandleAsync(HttpContext);
    }
}
```

3. **Frontend `VoiceTextArea`-Komponente**:

```tsx
export const VoiceTextArea = forwardRef<HTMLTextAreaElement, VoiceTextAreaProps>(
  ({ value, onChange, onTextRecognized, maxDuration }, ref) => {
    const { isRecording, interimText, volume, duration, startRecording, stopRecording } =
      useVoiceRecording({ onTextRecognized, maxDuration });

    return (

        </button>
        {/* Texteingabebereich */}

    );
  }

```

### Bilduploads: Multi-Methoden-Upload-Komponente

Wir bauten eine voll ausgestattete Bildupload-Komponente mit Unterstützung für alle drei Upload-Methoden, die die gängigsten Szenarien abdeckt, auf die Nutzer stoßen.

**Kernfunktionen**:

1. **Drei Upload-Methoden**:

```tsx
// Klick zum Hochladen
const handleClick = () => fileInputRef.current?.click();

// Drag-and-Drop-Upload
const handleDrop = (e: React.DragEvent) => {
  const file = e.dataTransfer.files?.[0];
  if (file) uploadFile(file);
};

// Zwischenablage-Einfügen
const handlePaste = (e: ClipboardEvent) => {
  for (const item of Array.from(e.clipboardData?.items || [])) {
    if (item.type.startsWith('image/')) {
      const file = item.getAsFile();
      if (file) uploadFile(file);
    }
  }
};
```

2. **Frontend-Validierung**:

```tsx
const validateFile = (file: File): { valid: boolean; error?: string } => {
  if (!acceptedTypes.includes(file.type)) {
    return { valid: false, error: 'Nur PNG, JPG, JPEG, WebP und GIF Bilder sind erlaubt' };
  }
  if (file.size > maxSize) {
    return { valid: false, error: `Maximale Dateigröße ist ${(maxSize / 1024 / 1024).toFixed(1)}MB` };
  }
  return { valid: true };
};
```

3. **Backend-Upload-Handler** (TypeScript):

```typescript
export const Route = createFileRoute('/api/upload')({
  server: {
    handlers: {
      POST: async ({ request }) => {
        const formData = await request.formData();
        const file = formData.get('file') as File;

        // Validierung
        const validation = validateFile(file);
        if (!validation.isValid) {
          return Response.json({ error: validation.error }, { status: 400 });
        }

        // Datei speichern
        const uuid = uuidv4();
        const filePath = join(uploadDir, `${uuid}${extension}`);
        await writeFile(filePath, buffer);

        return Response.json({ url: `/uploaded/${today}/${uuid}${extension}` });
      }
    }
  }
});
```

## Praktischer Leitfaden

### Spracherkennung verwenden

1. **Spracherkennungsdienst konfigurieren**:
   - Seite der Spracherkennungseinstellungen öffnen
   - Doubao Speech `AppId` und `AccessToken` konfigurieren
   - Optional Hotwords konfigurieren, um die Erkennungsgenauigkeit für domänenspezifische Begriffe zu verbessern

2. **Im Eingabefeld verwenden**:
   - Auf das Mikrofonsymbol links neben dem Eingabefeld klicken
   - Nach Erscheinen der Wellenform-Animation zu sprechen beginnen
   - Erneut auf das Symbol klicken, um die Aufnahme zu stoppen
   - Der erkannte Text wird automatisch an der Cursorposition eingefügt

3. **Hotword-Konfigurationsbeispiel**:

```text
TypeScript
React
useState
useEffect
```

### Bilduploads verwenden

1. **Upload-Methoden**:
   - Auf den Upload-Button klicken, um eine Datei auszuwählen
   - Ein Bild direkt in den Upload-Bereich ziehen
   - `Strg+V` verwenden, um einen Screenshot aus der Zwischenablage einzufügen

2. **Unterstützte Formate**: PNG, JPG, JPEG, WebP, GIF
3. **Größenlimit**: 5 MB standardmäßig (konfigurierbar)

### Hinweise

1. **Spracherkennung**:
   - Mikrofonberechtigung ist erforderlich
   - Möglichst in ruhiger Umgebung verwenden
   - Die maximal unterstützte Aufnahmedauer beträgt standardmäßig 300 Sekunden (konfigurierbar)

2. **Bilduploads**:
   - Nur gängige Bildformate werden unterstützt
   - Auf Dateigrößenlimits achten
   - Hochgeladene Bilder erhalten automatisch eine Vorschau-URL

3. **Sicherheitsüberlegungen**:
   - Spracherkennungs-Zugangsdaten werden auf dem Backend gespeichert
   - Bilduploads durchlaufen eine strenge serverseitige Validierung
   - HTTPS/WSS wird in Produktionsumgebungen empfohlen

## Zusammenfassung

Nach Hinzufügen von Spracherkennung und Bilduploads verbesserte sich die HagiCode-Benutzererfahrung spürbar. Nutzer können nun auf natürlichere Weise mit der KI interagieren - sprechen statt tippen und Screenshots teilen statt alles manuell zu beschreiben. Es fühlt sich an, als hätte man endlich einen bequemeren Weg der Kommunikation gefunden.

Beim Bau dieser Funktion stießen wir auf das Problem, dass Browser-WebSocket-APIs keine benutzerdefinierten Header unterstützen. Letztendlich lösten wir es mit einem Backend-Proxy-Ansatz. Diese Lösung bewahrte nicht nur die Sicherheit, sondern legte auch das Fundament für die spätere Integration anderer authentifizierter WebSocket-Dienste.

Die Bildupload-Komponente profitiert ebenfalls von der Unterstützung mehrerer Upload-Methoden, sodass Nutzer die jeweils bequemste Methode wählen können. Klicken, Ziehen oder Einfügen - alles funktioniert, und jeder Pfad erledigt die Aufgabe schnell.

„Tippen ist langsamer als Sprechen, und Sprechen ist langsamer als ein Screenshot" passt hier recht gut zum Thema. Wenn Sie ein ähnliches KI-Assistenten-Produkt bauen, hoffe ich, dass diese Erfahrungen helfen, auch wenn nur ein wenig.
