Zuvor hatte das Projekt bereits eine CodeBuddy-Anbindung implementiert, aber der relevante Code wurde bei einem Refactoring entfernt. Eigentlich gibt es auch nichts zu beanstanden – bei der Code-Iteration gehen immer wieder Dinge vergessen. Das Ziel dieser technischen Lösung ist es, diese Fähigkeit vollständig wiederherzustellen und die Architektur zu optimieren, damit sie robuster und wartbarer wird.

Wenn Sie auch darüber nachdenken, mehrere AI-Programmierassistenten für Ihr Projekt anzubinden, könnte die folgende Lösung einige Inspiration bieten – das sind Erfahrungen, die wir nach unzähligen Fehlschlägen zusammengefasst haben. ## Architekturdesign

### Überblick über die geschichtete Architektur

Die Anbindung von C# an CodeBuddy verwendet eine klare geschichtete Architektur, die für eine klare Trennung der Verantwortlichkeiten sorgt und die spätere Wartung erleichtert:

```
┌─────────────────────────────────────────────┐
│           Provider-Vertragsebene             │
│   AIProviderType-Enumeration + Erweiterungsmethoden             │
├─────────────────────────────────────────────┤
│           Provider-Factory-Ebene             │
│   AIProviderFactory-Dependency-Injection-Factory             │
├─────────────────────────────────────────────┤
│           Provider-Implementierungsebene     │
│   CodebuddyCliProvider konkrete Implementierung              │
├─────────────────────────────────────────────┤
│           ACP-Infrastrukturebene             │
│  ACPSessionManager / StdioAcpTransport      │
│  AcpRpcClient / AcpAgentClient              │
└─────────────────────────────────────────────┘
```

Was ist der Vorteil dieser Schichtung? Ganz einfach: Die Ebenen stören sich nicht gegenseitig. Angenommen, Sie müssen später die Kommunikationsmethode ändern (z. B. von stdio auf WebSocket), dann müssen Sie nur die unterste Ebene ändern, der Geschäftscode oben bleibt völlig unberührt. Schließlich will niemand, dass eine kleine Änderung Auswirkungen auf den gesamten Code hat – nur die Kommunikationsmethode zu ändern und dann noch stundenlang Geschäftscode anpassen zu müssen, wäre zu tragisch.

### Analyse der Kernkomponenten

Die **Provider-Vertragsebene** ist das Fundament der gesamten Architektur. Wir definieren die Enumeration `AIProviderType`, wobei `CodebuddyCli = 3` als Enumerationswert dient. Durch Erweiterungsmethoden wird eine bidirektionale Zuordnung zwischen Zeichenfolgen und Enumerationen realisiert. So können Zeichenfolgen in Konfigurationsdateien bequem in Enumerationen umgewandelt werden, und bei der Fehlersuche können Enumerationen in Zeichenfolgen ausgegeben werden. Das ist auch okay – es ist im Grunde nur eine Zuordnungsbeziehung, aber gut gemacht spart viel Ärger.

Die **Provider-Factory-Ebene** ist dafür verantwortlich, entsprechend der Konfiguration entsprechende Provider-Instanzen zu erstellen. Hier wird der Dependency-Injection-Mechanismus von .NET verwendet, zusammen mit `ActivatorUtilities.CreateInstance` für die dynamische Erstellung. Der Vorteil des Factory-Musters besteht darin, dass beim Hinzufügen eines neuen Providers nur die Erstellungslogik hinzugefügt werden muss, ohne vorhandenen Code ändern zu müssen. Das ist ähnlich wie beim Schreiben eines Artikels: Wenn Sie ein neues Kapitel hinzufügen wollen, fügen Sie einfach ein neues Kapitel hinzu, ohne alles vorherige neu schreiben zu müssen.

Die **Provider-Implementierungsebene** ist der Ort, an dem die eigentliche Arbeit erledigt wird. `CodebuddyCliProvider` implementiert das `IAIProvider`-Interface und bietet zwei Aufrufarten: `ExecuteAsync` (nicht-streaming) und `StreamAsync` (streaming).

Die **ACP-Infrastrukturebene** ist die darunterliegende Unterstützung für die Kommunikation. Diese Ebene处理 alle Protokolldetails, einschließlich Prozessverwaltung, Nachrichtenserialisierung, Antwortparsing usw. Wie das Fundament eines Hauses – egal wie schön es oben ist, das Fundament muss stabil sein.

## Kommunikationsmechanismus

### Stdio-Transportmodus

CodeBuddy verwendet den **Stdio-(Standard Input/Output)**-Modus zur Kommunikation mit externen Prozessen. Der Startbefehl ist einfach:

```bash
codebuddy --acp
```

Dann werden JSON-RPC-Nachrichten über Standard Input/Output ausgetauscht. Die Vorteile dieses Ansatzes sind:

1. **Schneller Start**: Keine Netzwerkverzögerung bei lokaler Prozesskommunikation
2. **Einfache Konfiguration**: Nur den Pfad zur ausführbaren Datei angeben
3. **Umgebungsisolation**: Jede Sitzung als unabhängiger Prozess, ohne gegenseitige Beeinflussung

Während der Kommunikationsprozess wird die Injektion von Umgebungsvariablen unterstützt, häufig verwendet werden:

- `CODEBUDDY_API_KEY`: API-Schlüssel-Authentifizierung
- `CODEBUDDY_INTERNET_ENVIRONMENT`: Netzwerkumgebungskonfiguration

Das ist wie die Kommunikation zwischen Menschen – man braucht eine bequeme Methode, um überhaupt ins Gespräch zu kommen.

### Nachrichtenprotokoll

ACP basiert auf dem JSON-RPC-2.0-Protokoll, das Nachrichtenformat sieht in etwa so aus:

```json
// Anfragenachricht
{
  "jsonrpc": "2.0",
  "id": 1,
  "method": "agent/prompt",
  "params": {
    "prompt": "Hilf mir, einen Sortieralgorithmus zu schreiben",
    "sessionId": "session-123"
  }
}

// Antwortnachricht
{
  "jsonrpc": "2.0",
  "id": 1,
  "result": {
    "content": "Hier ist die KI-Antwort..."
  }
}
```

In der tatsächlichen Implementierung haben wir alle diese Protokolldetails gekapselt, sodass der obere Geschäftscode nur auf prompt und response achten muss. Das ist auch gut so – gute Kapselung macht die spätere Nutzung angenehmer.

### 2. Provider-Factory-Integration

Fügen Sie in der Factory-Klasse den CodeBuddy-Erstellungszweig hinzu:

```csharp
// AIProviderFactory.cs
private IAIProvider? CreateProvider(AIProviderType providerType, ProviderConfiguration config)
{
    return providerType switch
    {
        AIProviderType.CodebuddyCli =>
            ActivatorUtilities.CreateInstance<CodebuddyCliProvider>(
                _serviceProvider,
                Options.Create(config)),
        // ... andere Provider
        _ => throw new NotSupportedException($"Provider {providerType} nicht unterstützt")
    };
}
```

Hier wird der `ActivatorUtilities` der Dependency-Injection verwendet, der automatisch die Konstruktorparameter-Injektion处理, sehr praktisch. Das ist auch okay – .NET-Dinge, richtig verwendet, sparen viel Ärger.

### 3. Vollständige Provider-Implementierung

Hier ist die Kernimplementierung von `CodebuddyCliProvider`, die sowohl Streaming- als auch Nicht-Streaming-Aufrufe unterstützt:

```csharp
public class CodebuddyCliProvider : IAIProvider
{
    private readonly ILogger<CodebuddyCliProvider> _logger;
    private readonly IACPSessionManager _sessionManager;
    private readonly ProviderConfiguration _config;

    public string Name => "CodebuddyCli";
    public bool SupportsStreaming => true;
    public ProviderCapabilities Capabilities { get; }

    public CodebuddyCliProvider(
        ILogger<CodebuddyCliProvider> logger,
        IACPSessionManager sessionManager,
        IOptions<ProviderConfiguration> config)
    {
        _logger = logger;
        _sessionManager = sessionManager;
        _config = config.Value;

        // Fähigkeiten des aktuellen Providers definieren
        Capabilities = new ProviderCapabilities
        {
            SupportsStreaming = true,
            SupportsTools = true,
            SupportsSystemMessages = true,
            SupportsArtifacts = false,
            MaxTokens = 8192
        };
    }

    // Nicht-Streaming-Aufruf: Warten, bis alle Ergebnisse zusammen zurückgegeben werden
    public async Task<AIResponse> ExecuteAsync(
        AIRequest request,
        CancellationToken cancellationToken = default)
    {
        // Unabhängige Sitzung für die Anfrage erstellen
        var session = await _sessionManager.CreateSessionAsync(
            "CodebuddyCli",
            request.WorkingDirectory,
            cancellationToken,
            request.SessionId);

        try
        {
            var fullPrompt = BuildPrompt(request);
            await session.SendPromptAsync(fullPrompt, cancellationToken);

            var responseBuilder = new StringBuilder();
            var toolCalls = new List<AIToolCall>();

            // Alle Antwortblöcke sammeln
            await foreach (var chunk in StreamFromSession(session, cancellationToken))
            {
                if (!string.IsNullOrEmpty(chunk.Content))
                {
                    responseBuilder.Append(chunk.Content);
                }
                // Tool-Aufrufe verarbeiten...
            }

            return new AIResponse
            {
                Content = AIResultContentSanitizer.SanitizeResultContent(
                    responseBuilder.ToString()),
                ToolCalls = toolCalls,
                Provider = Name,
                Model = string.Empty
            };
        }
        finally
        {
            // Sitzungsressourcen freigeben
            await session.DisposeAsync();
        }
    }

    // Streaming-Aufruf: Echtzeit-Rückgabe von Antwortblöcken
    public async IAsyncEnumerable<AIStreamingChunk> StreamAsync(
        AIRequest request,
        [EnumeratorCancellation] CancellationToken cancellationToken = default)
    {
        var session = await _sessionManager.CreateSessionAsync(
            "CodebuddyCli",
            request.WorkingDirectory,
            cancellationToken);

        try
        {
            var fullPrompt = BuildPrompt(request);
            await session.SendPromptAsync(fullPrompt, cancellationToken);

            await foreach (var chunk in StreamFromSession(session, cancellationToken))
            {
                yield return chunk;
            }
        }
        finally
        {
            await session.DisposeAsync();
        }
    }

    private async IAsyncEnumerable<AIStreamingChunk> StreamFromSession(
        IACPSession session,
        [EnumeratorCancellation] CancellationToken cancellationToken)
    {
        // Alle Updates in der Sitzung durchlaufen
        await foreach (var notification in session.ReceiveUpdatesAsync(cancellationToken))
        {
            switch (notification.Update)
            {
                case AgentMessageChunkSessionUpdate agentMessage:
                    // Textinhaltsblöcke verarbeiten
                    if (agentMessage.Content is AcpImp.TextContentBlock textContent)
                    {
                        yield return new AIStreamingChunk
                        {
                            Content = textContent.Text,
                            Type = StreamingChunkType.ContentDelta,
                            IsComplete = false
                        };
                    }
                    break;

                case ToolCallSessionUpdate toolCall:
                    // Tool-Aufrufe verarbeiten
                    yield return new AIStreamingChunk
                    {
                        Content = string.Empty,
                        Type = StreamingChunkType.ToolCallDelta,
                        ToolCallDelta = new AIToolCallDelta
                        {
                            Id = toolCall.ToolCallId,
                            Name = toolCall.Kind.ToString(),
                            Arguments = toolCall.RawInput?.ToString()
                        }
                    };
                    break;

                case AcpImp.PromptCompletedSessionUpdate:
                    // Antwort abgeschlossen
                    yield break;
            }
        }
    }

    // Vollständigen Prompt aufbauen
    private string BuildPrompt(AIRequest request, string? embeddedCommandPrompt = null)
    {
        var sb = new StringBuilder();

        // Eingebetteten Befehlsprompt (falls vorhanden)
        if (!string.IsNullOrEmpty(embeddedCommandPrompt))
        {
            sb.AppendLine(embeddedCommandPrompt);
            sb.AppendLine();
        }

        // Systemnachricht
        if (!string.IsNullOrEmpty(request.SystemMessage))
        {
            sb.AppendLine(request.SystemMessage);
            sb.AppendLine();
        }

        // Benutzer-Prompt
        sb.Append(request.Prompt);
        return sb.ToString();
    }
}
```

Dieser Code hat mehrere wichtige Punkte:

1. **Sitzungsverwaltung**: Jede Anfrage erstellt eine unabhängige Sitzung, Ressourcen werden nach Abschluss der Anfrage freigegeben. Das ist eine Erfahrung aus Fehlern – wenn die Sitzungswiederverwendung nicht gut gemacht ist, treten leicht Probleme mit Zustandsverschmutzung auf. Schließlich muss man das, was man benutzt hat, aufräumen, sonst haben die nächsten Personen Probleme.

2. **Streaming-Verarbeitung**: `IAsyncEnumerable` ermöglicht es, Antworten zurückzugeben, während sie generiert werden, ohne auf die vollständige Generierung zu warten. Das ist besonders wichtig für Langtext-Szenarien, die Benutzererfahrung wird viel besser. Wie beim Warten auf Ergebnisse – niemand will ewig untätig warten.

3. **Tool-Aufrufe**: CodeBuddy unterstützt Tool-Aufrufe (Function Calling), die durch `ToolCallSessionUpdate` verarbeitet werden. Diese Fähigkeit ist entscheidend für komplexe Codebearbeitungsaufgaben.

4. **Inhaltsfilterung`: Verwendung von `AIResultContentSanitizer` zum Filtern von Think-Block-Inhalten, um die Ausgabe sauber zu halten.

### 4. Dependency-Injection-Konfiguration

Fügen Sie relevante Dienste in der Modulregistrierung hinzu:

```csharp
// PCodeClaudeHelperModule.cs
public void ConfigureModule(IServiceCollection context)
{
    // Provider registrieren
    context.Services.AddTransient<CodebuddyCliProvider>();

    // ACP-Infrastruktur registrieren
    context.Services.AddSingleton<IACPSessionManager, ACPSessionManager>();
    context.Services.AddSingleton<IAcpPlatformConfigurationResolver, AcpPlatformConfigurationResolver>();
    context.Services.AddSingleton<IAIRequestToAcpMapper, AIRequestToAcpMapper>();
    context.Services.AddSingleton<IAcpToAIResponseMapper, AcpToAIResponseMapper>();
}
```

## Zusammenfassung der praktischen Erfahrungen

### Fehlerprotokoll

Wir sind bei der Implementierung auf mehrere typische Fallen gestoßen, die wir teilen, damit andere Umwege vermeiden können. Schließlich ist es gut, wenn man die Fehler anderer vermeiden kann:

1. **Sitzungsleckproblem**: Anfangs wurden Sitzungen nicht korrekt freigegeben, was zur Erschöpfung von Prozessressourcen führte. Die Lösung ist die Verwendung von `try-finally`, um sicherzustellen, dass bei jeder Anfrage Ressourcen freigegeben werden. Das ist auch okay – was man benutzt hat, muss zurückgestellt werden, sonst haben die nächsten Personen nichts zu verwenden.

2. **Übergabe von Umgebungsvariablen**: Die Syntax für Umgebungsvariablen unter Windows und Linux ist unterschiedlich, später haben wir uns auf die Verwendung von `Dictionary<string, string?>` geeinigt. Bei plattformübergreifenden Dingen sollte man von Anfang an einheitliche Standards festlegen, das spart später Ärger.

3. **Timeout-Konfiguration**: Der CLI-Start benötigt Zeit, wir haben einen Start-Timeout von 5 Sekunden eingestellt, um schnelle Anforderungsfehler zu vermeiden. Bei allem muss ein Maß geben, zu viel Eile führt zum Scheitern.

4. **Kodierungsproblem**: Die Standardkodierung unter Windows kann zu chinesischen Zeichenfehlern führen, beim Starten des Prozesses wird explizit UTF-8-Kodierung angegeben. Wenn chinesische Zeichen nicht angezeigt werden können, ist das sehr unangenehm.

### Leistungsoptimierung

1. **Sitzungspool**: Für häufige kurze Anfragen kann ein Sitzungspool implementiert werden, um Prozesse wiederzuverwenden
2. **Verbindungscache**: Die Factory-Klasse unterstützt bereits das Caching von Provider-Instanzen
3. **Asynchronie zuerst**: Durchgehend asynchrone Programmierung verwenden,线程blockierung vermeiden

Bei Leistungsfragen: Optimieren, wo es möglich ist – je länger Benutzer warten, desto schlechter die Erfahrung.
