## Praktischer Leitfaden zur Integration von CodeBuddy CLI in ein C#-Backend

> Dieser Artikel führt durch einen vollständigen Ansatz zur Integration von CodeBuddy CLI in ein C#-Backend-Projekt, um KI-Code-Assistent-Funktionen durchgängig bereitzustellen.

:::note[Zeitlicher Kontext]
Dieser Artikel behandelt das CodeBuddy-Integrationsdesign und die Provider-Struktur wie sie zum Zeitpunkt der Veröffentlichung bestanden. Informationen zum aktuell dauerhaft unterstützten Bereich finden Sie unter [KI-Agent-CLI-Installation](/en/related-software-installation/ai-agent-cli/) und im [Installationsleitfaden](/en/installation); `IFlowCli` wird jetzt nur noch als historischer Kompatibilitätswert beibehalten und ist nicht Teil des aktuellen aktiven Unterstützungsbereichs.
:::

## Hintergrund

In der modernen Entwicklung von KI-Code-Assistenten kann ein einzelner KI-Anbieter oft nicht den komplexen und sich ändernden Entwicklungsszenarien gerecht werden. HagiCode als multifunktionaler KI-Code-Assistent muss mehrere KI-Anbieter unterstützen, um eine bessere Benutzererfahrung zu bieten. Benutzer sollten ausreichend Freiheit bei der Wahl haben. Anfang 2026 stand das Projekt vor einer Schlüsselentscheidung: wie die CodeBuddy-ACP-Integrationsfähigkeit (Agent Communication Protocol) im C#-Backend wiederhergestellt werden kann.

Das Projekt hatte zuvor die CodeBuddy-Integration implementiert, aber der zugehörige Code wurde bei einem Refactoring entfernt. Darüber gibt es nicht viel zu beschweren; bei iterativer Entwicklung bleibt immer etwas auf der Strecke. Das Ziel dieser technischen Lösung war, diese Fähigkeit vollständig wiederherzustellen und die Architektur zu verbessern, damit sie robuster und wartbarer wird.

Wenn Sie ebenfalls erwägen, mehrere KI-Code-Assistenten in Ihr eigenes Projekt einzubinden, kann der folgende Ansatz Ihnen einige Ideen geben. Er spiegelt Erkenntnisse wider, die wir nach vielen Fehlern zusammengefasst haben, und kann Ihnen vielleicht einige Umwege ersparen.

## Ueber HagiCode

Der in diesem Artikel geteilte Ansatz stammt aus unserer praktischen Erfahrung im [HagiCode](https://hagicode.com)-Projekt. HagiCode ist ein Open-Source-KI-Code-Assistent-Projekt, das mehrere KI-Anbieter und plattformübergreifenden Betrieb unterstützt. Um unterschiedliche Benutzerpräferenzen zu erfüllen, müssen wir flexibel zwischen verschiedenen KI-Code-Assistenten wechseln können, weshalb wir die hier beschriebene CodeBuddy-Integration aufgebaut haben.

HagiCode verwendet einen modularen Entwurf, bei dem KI-Anbieter als einsteckbare Komponenten implementiert sind. Diese Architektur ermöglicht es uns, neue KI-Unterstützung einfach hinzuzufügen, ohne bestehende Funktionen zu beeinträchtigen. Wenn ein Entwurf von Anfang an gut gemacht ist, spart das später viel Ärger. Wenn Sie sich für unsere technische Architektur interessieren, können Sie den vollständigen Quellcode auf GitHub einsehen.

## Architekturentwurf

### Übersicht über die Schichtenarchitektur

Die Integration zwischen C# und CodeBuddy verwendet eine klare Schichtenarchitektur. Dieser Entwurf macht Verantwortlichkeiten explizit und erleichtert die langfristige Wartung erheblich:

```
┌─────────────────────────────────────────────┐
│           Provider-Vertragsschicht          │
│   AIProviderType-Aufzählung + Erweiterungsmethoden   │
├─────────────────────────────────────────────┤
│           Provider-Factory-Schicht          │
│   AIProviderFactory-Abhängigkeitsinjektions-Factory │
├─────────────────────────────────────────────┤
│           Provider-Implementierungsschicht   │
│   CodebuddyCliProvider konkrete Implementierung │
├─────────────────────────────────────────────┤
│           ACP-Infrastrukturschicht          │
│  ACPSessionManager / StdioAcpTransport      │
│  AcpRpcClient / AcpAgentClient              │
└─────────────────────────────────────────────┘
```

Was sind die Vorteile dieser Schichtung? Ganz einfach: jede Schicht bleibt den anderen aus dem Weg. Wenn wir später beispielsweise den Kommunikationsmechanismus ändern wollen, etwa von `stdio` auf WebSocket, müssen wir nur die unterste Schicht anpassen, und die darüberliegende Geschäftslogik bleibt unberührt. Niemand möchte, dass sich eine Kommunikationsänderung durch die gesamte Codebasis zieht.

### Aufschlüsselung der Kernkomponenten

Die **Provider-Vertragsschicht** ist das Fundament der gesamten Architektur. Wir definieren die `AIProviderType`-Aufzählung, wobei `CodebuddyCli = 3` als Aufzählungswert verwendet wird, und implementieren eine bidirektionale Zuordnung zwischen Zeichenketten und Aufzählungen über Erweiterungsmethoden. Dadurch können Zeichenketten in Konfigurationsdateien bequem in Aufzählungen umgewandelt werden und Aufzählungen für Debugging-Ausgaben zurück in Zeichenketten.

Die **Provider-Factory-Schicht** ist für die Erstellung der entsprechenden Anbieterinstanz basierend auf der Konfiguration verantwortlich. Sie verwendet die .NET-Abhängigkeitsinjektion zusammen mit `ActivatorUtilities.CreateInstance` zur dynamischen Erstellung. Der Vorteil des Factory-Musters besteht darin, dass beim Hinzufügen eines neuen Anbieters nur die Erstellungslogik hinzugefügt werden muss, ohne vorhandenen Code zu ändern.

Die **Provider-Implementierungsschicht** ist, wo die eigentliche Arbeit stattfindet. `CodebuddyCliProvider` implementiert die `IAIProvider`-Schnittstelle und bietet zwei Aufrufmodi: `ExecuteAsync` für nicht-streaming Aufrufe und `StreamAsync` für Streaming-Aufrufe.

Die **ACP-Infrastrukturschicht** bietet das Kommunikationsfundament darunter. Diese Schicht behandelt alle Protokolldetails, einschließlich Prozessverwaltung, Nachrichtenserialisierung und Antwortanalyse. Sie ist das Fundament, das alles darüber stabil hält.

## Kommunikationsmechanismus

### Stdio-Transportmodus

CodeBuddy verwendet **Stdio (Standardeingabe/-ausgabe)** zur Kommunikation mit externen Prozessen. Der Startbefehl ist einfach:

```bash
codebuddy --acp
```

Danach werden JSON-RPC-Nachrichten über Standardeingabe und -ausgabe ausgetauscht. Dieser Ansatz bietet mehrere Vorteile:

1. **Schneller Start**: Lokale Prozesskommunikation vermeidet Netzwerklatenz
2. **Einfache Konfiguration**: Es muss nur der ausführbare Pfad angegeben werden
3. **Umgebungsisolierung**: Jede Sitzung läuft in einem unabhängigen Prozess und beeinflusst die anderen nicht

Die Injektion von Umgebungsvariablen wird während der Kommunikation unterstützt. Häufige Beispiele sind:

- `CODEBUDDY_API_KEY`: API-Schlüssel-Authentifizierung
- `CODEBUDDY_INTERNET_ENVIRONMENT`: Netzwerkumgebungskonfiguration

Wie bei der Kommunikation zwischen Menschen hilft es, zunächst einen bequemen Kanal zu wählen.

### Nachrichtenprotokoll

ACP basiert auf JSON-RPC 2.0. Das Nachrichtenformat sieht grob so aus:

```json
// Anfragenachricht
{
  "jsonrpc": "2.0",
  "id": 1,
  "method": "agent/prompt",
  "params": {
    "prompt": "Helfen Sie mir, einen Sortieralgorithmus zu schreiben",
    "sessionId": "session-123"
  }
}

// Antwortnachricht
{
  "jsonrpc": "2.0",
  "id": 1,
  "result": {
    "content": "Hier ist die KI-Antwort..."
  }
}
```

In der tatsächlichen Implementierung kapseln wir all diese Protokolldetails, sodass die obere Geschäftsschicht sich nur um den Prompt und die Antwort kümmern muss.

### 1. Provider-Vertrag wiederherstellen

Stellen Sie zunächst den CodeBuddy-Typ in der Aufzählungsdatei wieder her:

### 2. Provider-Factory integrieren

Fügen Sie eine CodeBuddy-Erstellungsverzweigung in der Factory-Klasse hinzu:

```csharp
// AIProviderFactory.cs
private IAIProvider? CreateProvider(AIProviderType providerType, ProviderConfiguration config)
{
    return providerType switch
    {
        AIProviderType.CodebuddyCli =>
            ActivatorUtilities.CreateInstance<CodebuddyCliProvider>(
                _serviceProvider,
                Options.Create(config)),
        // ... Weitere Anbieter
        _ => throw new NotSupportedException($"Anbieter {providerType} wird nicht unterstützt")
    };
}
```

Dies verwendet die Abhängigkeitsinjektion über `ActivatorUtilities`, was automatisch die Konstruktorparameterinjektion übernimmt und sehr bequem ist.

### 3. Vollständige Provider-Implementierung

Nachfolgend die Kernimplementierung von `CodebuddyCliProvider`, die sowohl Streaming- als auch Nicht-Streaming-Aufrufmodi abdeckt:

```csharp
public class CodebuddyCliProvider : IAIProvider
{
    private readonly ILogger<CodebuddyCliProvider> _logger;
    private readonly IACPSessionManager _sessionManager;
    private readonly ProviderConfiguration _config;

    public string Name => "CodebuddyCli";
    public bool SupportsStreaming => true;
    public ProviderCapabilities Capabilities { get; }

    public CodebuddyCliProvider(
        ILogger<CodebuddyCliProvider> logger,
        IACPSessionManager sessionManager,
        IOptions<ProviderConfiguration> config)
    {
        _logger = logger;
        _sessionManager = sessionManager;
        _config = config.Value;

        // Fähigkeiten des aktuellen Providers definieren
        Capabilities = new ProviderCapabilities
        {
            SupportsStreaming = true,
            SupportsTools = true,
            SupportsSystemMessages = true,
            SupportsArtifacts = false,
            MaxTokens = 8192
        };
    }

    // Nicht-Streaming-Aufruf: alle Ergebnisse nach Abschluss zurückgeben
    public async Task<AIResponse> ExecuteAsync(
        AIRequest request,
        CancellationToken cancellationToken = default)
    {
        // Unabhängige Sitzung für die Anfrage erstellen
        var session = await _sessionManager.CreateSessionAsync(
            "CodebuddyCli",
            request.WorkingDirectory,
            cancellationToken,
            request.SessionId);

        try
        {
            var fullPrompt = BuildPrompt(request);
            await session.SendPromptAsync(fullPrompt, cancellationToken);

            var responseBuilder = new StringBuilder();
            var toolCalls = new List<AIToolCall>();

            // Alle Antwort-Chunks sammeln
            await foreach (var chunk in StreamFromSession(session, cancellationToken))
            {
                if (!string.IsNullOrEmpty(chunk.Content))
                {
                    responseBuilder.Append(chunk.Content);
                }
                // Werkzeugaufrufe behandeln...
            }

            return new AIResponse
            {
                Content = AIResultContentSanitizer.SanitizeResultContent(
                    responseBuilder.ToString()),
                ToolCalls = toolCalls,
                Provider = Name,
                Model = string.Empty
            };
        }
        finally
        {
            // Sitzungsressourcen freigeben
            await session.DisposeAsync();
        }
    }

    // Streaming-Aufruf: Antwort-Chunks in Echtzeit zurückgeben
    public async IAsyncEnumerable<AIStreamingChunk> StreamAsync(
        AIRequest request,
        [EnumeratorCancellation] CancellationToken cancellationToken = default)
    {
        var session = await _sessionManager.CreateSessionAsync(
            "CodebuddyCli",
            request.WorkingDirectory,
            cancellationToken);

        try
        {
            var fullPrompt = BuildPrompt(request);
            await session.SendPromptAsync(fullPrompt, cancellationToken);

            await foreach (var chunk in StreamFromSession(session, cancellationToken))
            {
                yield return chunk;
            }
        }
        finally
        {
            await session.DisposeAsync();
        }
    }

    private async IAsyncEnumerable<AIStreamingChunk> StreamFromSession(
        IACPSession session,
        [EnumeratorCancellation] CancellationToken cancellationToken)
    {
        // Alle Aktualisierungen in der Sitzung durchlaufen
        await foreach (var notification in session.ReceiveUpdatesAsync(cancellationToken))
        {
            switch (notification.Update)
            {
                case AgentMessageChunkSessionUpdate agentMessage:
                    // Textinhalt-Chunks behandeln
                    if (agentMessage.Content is AcpImp.TextContentBlock textContent)
                    {
                        yield return new AIStreamingChunk
                        {
                            Content = textContent.Text,
                            Type = StreamingChunkType.ContentDelta,
                            IsComplete = false
                        };
                    }
                    break;

                case ToolCallSessionUpdate toolCall:
                    // Werkzeugaufrufe behandeln
                    yield return new AIStreamingChunk
                    {
                        Content = string.Empty,
                        Type = StreamingChunkType.ToolCallDelta,
                        ToolCallDelta = new AIToolCallDelta
                        {
                            Id = toolCall.ToolCallId,
                            Name = toolCall.Kind.ToString(),
                            Arguments = toolCall.RawInput?.ToString()
                        }
                    };
                    break;

                case AcpImp.PromptCompletedSessionUpdate:
                    // Antwort abgeschlossen
                    yield break;
            }
        }
    }

    // Den vollständigen Prompt erstellen
    private string BuildPrompt(AIRequest request, string? embeddedCommandPrompt = null)
    {
        var sb = new StringBuilder();

        // Eingebetteter Befehls-Prompt, falls vorhanden
        if (!string.IsNullOrEmpty(embeddedCommandPrompt))
        {
            sb.AppendLine(embeddedCommandPrompt);
            sb.AppendLine();
        }

        // Systemnachricht
        if (!string.IsNullOrEmpty(request.SystemMessage))
        {
            sb.AppendLine(request.SystemMessage);
            sb.AppendLine();
        }

        // Benutzer-Prompt
        sb.Append(request.Prompt);
        return sb.ToString();
    }
}
```

In diesem Code gibt es mehrere Schlüsselpunkte:

1. **Sitzungsverwaltung**: Jede Anfrage erstellt eine unabhängige Sitzung und gibt Ressourcen nach Abschluss der Anfrage frei. Dies ist eine durch Versuch und Irrtum gewonnene Erkenntnis. Wenn die Sitzungswiederverwendung nicht gut gehandhabt wird, treten leicht Zustandsverschmutzungen auf.

2. **Streaming-Verarbeitung**: `IAsyncEnumerable` ermöglicht die Rückgabe der Antwort, während sie noch generiert wird, anstatt auf den Abschluss aller Inhalte zu warten. Dies ist besonders wichtig für Szenarien mit langen Texten und verbessert die Benutzererfahrung erheblich.

3. **Werkzeugaufrufe**: CodeBuddy unterstützt Werkzeugaufrufe (Function Calling), die über `ToolCallSessionUpdate` behandelt werden. Diese Fähigkeit ist entscheidend für komplexe Codebearbeitungsaufgaben.

4. **Inhaltsfilterung**: `AIResultContentSanitizer` wird verwendet, um Think-Block-Inhalte herauszufiltern und die Ausgabe sauber zu halten.

```csharp
// PCodeClaudeHelperModule.cs
public void ConfigureModule(IServiceCollection context)

    context.Services.AddSingleton<IAcpToAIResponseMapper, AcpToAIResponseMapper>();
}
```

## Konfigurationsbeispiel

### Konfigurationsdatei

Fügen Sie die CodeBuddy-bezogene Konfiguration zu `appsettings.json` hinzu:

```yaml
AI:
  # Standard-Provider
  DefaultProvider: "CodebuddyCli"

  # Provider-Konfiguration
  Providers:
    CodebuddyCli:
      Type: "CodebuddyCli"
      WorkingDirectory: "C:/projects/my-app"
      ExecutablePath: "C:/tools/codebuddy.cmd"

  # Plattformspezifische Konfiguration
  PlatformConfigurations:
    CodebuddyCli:
      ExecutablePath: "C:/tools/codebuddy.cmd"
      Arguments: "--acp"
      StartupTimeoutMs: 5000
      EnvironmentVariables:
        CODEBUDDY_API_KEY: "${CODEBUDDY_API_KEY}"
        CODEBUDDY_INTERNET_ENVIRONMENT: "production"
```

## Praktische Erfahrungswerte

### Fallenprotokoll

Wir sind während der Implementierung auf mehrere typische Fallen gestoßen, und sie hier zu teilen kann anderen helfen, dieselben Umwege zu vermeiden:

1. **Sitzungsleck-Problem**: Anfänglich wurden Sitzungen nicht korrekt freigegeben, was die Prozessressourcen erschöpfte. Die Lösung war die Verwendung von `try-finally`, um sicherzustellen, dass Ressourcen für jede Anfrage freigegeben werden.

2. **Umgebungsvariablenübergabe**: Windows und Linux verwenden unterschiedliche Syntax für Umgebungsvariablen, daher haben wir später auf `Dictionary<string, string?>` standardisiert, um dies zu behandeln.

3. **Timeout-Konfiguration**: Der CLI-Start benötigt Zeit, daher haben wir ein 5-sekündiges Start-Timeout festgelegt, um schnelle Anfrageausfälle zu vermeiden.

4. **Kodierungsprobleme**: Unter Windows kann die Standardkodierung zu verstümmeltem chinesischem Text führen, daher wird beim Start des Prozesses explizit die UTF-8-Kodierung angegeben.

### Leistungsoptimierung

1. **Sitzungspool**: Für häufige kurze Anfragen sollte ein Sitzungspool implementiert werden, um Prozesse wiederzuverwenden
2. **Verbindungscache**: Die Factory-Klasse unterstützt bereits das Caching von Anbieterinstanzen
3. **Async first**: Durchgehend asynchrone Programmierung verwenden, um Threads nicht zu blockieren

Leistung ist immer eine Optimierung wert. Je länger Benutzer warten, desto schlechter wird die Erfahrung.

Dieser Artikel stellte eine vollständige Lösung zur Integration von CodeBuddy CLI in ein C#-Backend vor, die den gesamten Prozess vom Architekturentwurf bis zur konkreten Implementierung abdeckt. Durch eine Schichtenarchitektur trennen wir Protokolldetails von der Geschäftslogik, wodurch der Code klarer und wartbarer wird.

Wesentliche Erkenntnisse:

- Verwendung einer Schichtenarchitektur mit Provider-Vertragsschicht, Factory-Schicht, Implementierungsschicht und Infrastrukturschicht

- Flexible Konfiguration und Erweiterbarkeit durch Abhängigkeitsinjektion

Dieser Ansatz eignet sich nicht nur für CodeBuddy; das Hinzufügen neuer KI-Anbieter kann demselben Muster folgen. Wenn Sie ebenfalls eine ähnliche Multi-AI-Provider-Integration aufbauen, hoffe ich, dass dieser Artikel einen nützlichen Bezugspunkt bietet.
