## GitHub Pages-Automatische Bereitstellung für HagiCode hinzufügen

> Das Projekt trug früher den Codenamen PCode und wurde inzwischen offiziell in HagiCode umbenannt. Dieser Artikel dokumentiert, wie wir die automatische Bereitstellung einer statischen Website für das Projekt eingeführt haben, damit das Veröffentlichen von Inhalten so einfach ist wie Wasser trinken.

## Hintergrund / Einleitung

Während der Entwicklung von HagiCode stießen wir auf ein sehr praktisches Problem: Mit der wachsenden Anzahl an Dokumentation und Proposals wurde die effiziente Verwaltung und Präsentation dieser Inhalte immer dringlicher. Wir entschieden uns, GitHub Pages zum Hosten unserer statischen Website zu nutzen, aber das manuelle Bauen und Bereitstellen war einfach zu mühsam — jede Änderung erforderte einen lokalen Build, ein Packen und dann einen manuellen Push in den `gh-pages`-Branch. Das war nicht nur ineffizient, sondern auch fehleranfällig.

Um dieses Problem zu lösen (hauptsächlich, weil wir faul sein wollten), brauchten wir einen automatisierten Bereitstellungsworkflow. Dieser Artikel dokumentiert im Detail, wie wir die GitHub Actions-basierte automatische Bereitstellung für das HagiCode-Projekt eingerichtet haben, damit wir uns auf die Content-Erstellung konzentrieren können und den Rest der Automatisierung überlassen können.

## Zielanalyse

Bevor wir loslegen, müssen wir zunächst klären, was genau diese Aufgabe bewirken soll. Schließlich schadet es nicht, die Axt vorher zu schärfen.

### Kernanforderungen

1.  **Automatischer Build**: Der Build-Prozess wird automatisch ausgelöst, wenn Code in den `main`-Branch gepusht wird.
2.  **Automatische Bereitstellung**: Nach einem erfolgreichen Build werden die generierten statischen Dateien automatisch auf GitHub Pages bereitgestellt.
3.  **Umgebungskonsistenz**: Sicherstellen, dass die CI-Umgebung mit der lokalen Build-Umgebung übereinstimmt, um die peinliche Situation „lokal funktioniert es, in der Produktion schlägt es fehl" zu vermeiden.

### Technologieentscheidung

Da HagiCode auf Docusaurus basiert (einem sehr beliebten React Static Site Generator), können wir GitHub Actions nutzen, um dieses Ziel zu erreichen.

## GitHub Actions-Workflow konfigurieren

GitHub Actions ist der CI/CD-Dienst von GitHub. Durch die Definition von Workflow-Dateien im YAML-Format innerhalb des Repositorys können wir verschiedene Automatisierungsaufgaben anpassen.

### Workflow-Datei erstellen

Wir müssen eine neue Konfigurationsdatei im Ordner `.github/workflows` unter dem Projektstamm erstellen, zum Beispiel `deploy.yml`. Wenn der Ordner nicht existiert, denken Sie daran, ihn manuell zu erstellen.

Die Kernlogik dieser Konfigurationsdatei sieht wie folgt aus:

1.  **Auslösebedingung**: Lauscht auf `push`-Ereignisse im `main`-Branch.
2.  **Laufzeitumgebung**: Das neueste Ubuntu.
3.  **Build-Schritte**:
    -   Code auschecken
    -   Node.js installieren
    -   Abhängigkeiten installieren (`npm install`)
    -   Statische Dateien bauen (`npm run build`)
4.  **Bereitstellungsschritt**: Das offizielle `action-gh-pages` verwenden, um die Build-Artefakte in den `gh-pages`-Branch zu pushen.

### Schlüsselkonfigurationscode

Nachfolgend die von uns übernommene Konfigurationsvorlage:

```yaml
name: Deploy to GitHub Pages

# Auslösebedingung: beim Push in den main-Branch
on:
  push:
    branches:
      - main
    # Bei Bedarf können Pfadfilter hinzugefügt werden, z. B. nur bei Dokumentationsänderungen bauen
    # paths:
    #   - 'docs/**'
    #   - 'package.json'

# Berechtigungen festlegen, wichtig für die Bereitstellung auf GitHub Pages
permissions:
  contents: read
  pages: write
  id-token: write

# Nebenläufigkeitssteuerung: ältere Builds desselben Branchs abbrechen
concurrency:
  group: "pages"
  cancel-in-progress: false

jobs:
  build:
    runs-on: ubuntu-latest
    steps:
      - name: Checkout
        uses: actions/checkout@v4
        # Hinweis: fetch-depth: 0 muss gesetzt sein, sonst kann die Build-Versionsnummer ungenau sein
        with:
          fetch-depth: 0

      - name: Setup Node
        uses: actions/setup-node@v4
        with:
          node-version: 20 # Empfohlen: mit der lokalen Entwicklungsumgebung übereinstimmen
          cache: 'npm'     # Cache aktiviert beschleunigt den Build-Prozess

      - name: Install dependencies
        run: npm ci
        # npm ci statt npm install verwenden, da es schneller, strenger und besser für CI geeignet ist

      - name: Build website
        run: npm run build
        env:
          # Falls Ihr Site-Build Umgebungsvariablen benötigt, hier konfigurieren
          # NODE_ENV: production
          # PUBLIC_URL: /your-repo-name

      - name: Upload artifact
        uses: actions/upload-pages-artifact@v3
        with:
          path: ./build # Standard-Ausgabeverzeichnis von Docusaurus

  deploy:
    environment:
      name: github-pages
      url: ${{ steps.deployment.outputs.page_url }}
    runs-on: ubuntu-latest
    needs: build
    steps:
      - name: Deploy to GitHub Pages
        id: deployment
        uses: actions/deploy-pages@v4
```

## Fallstricke bei der Implementierung

In der Praxis sind wir auf einige Probleme gestoßen. Wir teilen sie hier in der Hoffnung, dass andere sie vermeiden oder zumindest im Vorfeld Lösungen vorbereiten können.

### 1. GitHub Token-Berechtigungsprobleme

Bei der anfänglichen Konfiguration schlug die Bereitstellung stets mit einem 403 (Forbidden)-Fehler fehl. Nach langer Untersuchung stellten wir fest, dass der standardmäßige `GITHUB_TOKEN` von GitHub keine Schreibberechtigung für Pages hatte.

**Lösung**: In den Repository-Einstellungen `Settings` → `Actions` → `General` → `Workflow permissions` unbedingt **„Read and write permissions"** auswählen.

### 2. Falscher Build-Verzeichnispfad

Docusaurus speichert die gebauten statischen Dateien standardmäßig im Verzeichnis `build`. Einige Projekte können jedoch andere Konfigurationen verwenden — Create React App nutzt standardmäßig `build`, Vite standardmäßig `dist`. Wenn Actions meldet, dass Dateien nicht gefunden werden können, prüfen Sie die Ausgabepfad-Konfiguration in `docusaurus.config.js`.

### 3. Unterpfad-Probleme

Wenn Ihr Repository keine Benutzer-Homepage (also nicht `username.github.io`), sondern eine Projektseite (z. B. `username.github.io/project-name`) ist, müssen Sie `baseUrl` konfigurieren.

In `docusaurus.config.js`:

```javascript
module.exports = {
  // ...
  url: 'https://hagicode.com', // Ihre HagiCode-URL
  baseUrl: '/', // Bereitstellung im Stammverzeichnis
  // ...
};
```

Dieses Detail wird leicht übersehen. Bei falscher Konfiguration wird die Seite als weißer Bildschirm geladen, weil die Ressourcenpfade nicht aufgelöst werden können.

## Ergebnis überprüfen

Nachdem alles konfiguriert und der Code gepusht wurde, können wir in den **Actions**-Tab des GitHub-Repositorys gehen und zuschauen.

Sie sehen einen gelben Kreis, während der Workflow läuft. Wenn er grün wird, bedeutet das Erfolg. Wenn er rot wird, klicken Sie in die Protokolle, um das Problem zu untersuchen. Meistens lässt es sich dort nachverfolgen — in der Regel handelt es sich um einen Tippfehler oder eine falsche Pfadkonfiguration.

Nach erfolgreichem Build besuchen Sie `https://<ihre-benutzername>.github.io/<repo-name>/` und Sie sollten Ihre brandneue Website sehen.

## Zusammenfassung

Durch die Einführung von GitHub Actions haben wir die automatische Bereitstellung der HagiCode-Dokumentationswebsite erfolgreich implementiert. Dies spart nicht nur die Zeit für manuelle Operationen, sondern vor allem standardisiert es den Veröffentlichungsprozess. Jetzt kann jedes Teammitglied die Dokumentation aktualisieren — sobald die Änderungen in den `main`-Branch gemerged werden, erscheint der aktuelle Inhalt wenige Minuten später online.

**Kernvorteile**:
- **Höhere Effizienz**: Vom „manuellen Packen und manuellen Hochladen" zu „Code ist die Veröffentlichung".
- **Weniger Fehler**: Beseitigt die Möglichkeit menschlicher Bedienungsfehler.
- **Bessere Erfahrung**: Ermöglicht Entwicklern, sich mehr auf die Content-Qualität zu konzentrieren, anstatt von lästigen Bereitstellungsschritten abgelenkt zu werden.

Obwohl die Einrichtung von CI/CD anfangs etwas mühsam sein kann — insbesondere mit den verschiedenen Berechtigungs- und Pfadproblemen — ist es eine einmalige Investition mit enormer langfristiger Rendite. Wir empfehlen jedem statischen Website-Projekt dringend, einen ähnlichen Automatisierungsworkflow zu nutzen.
