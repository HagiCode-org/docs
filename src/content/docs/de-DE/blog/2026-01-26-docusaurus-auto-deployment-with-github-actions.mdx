## HagiCode GitHub Pages automatische Bereitstellung hinzufügen

> Die frühere Codenummer dieses Projekts war PCode, jetzt wurde es offiziell in HagiCode umbenannt. Dieser Artikel zeichnet auf, wie automatische Static-Site-Bereitstellungsfunktionen für das Projekt eingeführt wurden, um die Inhaltveröffentlichung so einfach wie das Trinken von Wasser zu machen.

## Hintergrund/Einleitung

Während der Entwicklung von HagiCode sind wir auf ein sehr实际问题 gestoßen: Mit immer mehr Dokumentation und Vorschlägen wurde die effiziente Verwaltung und Präsentation dieser Inhalte zu einer dringenden Aufgabe. Wir entschieden uns, GitHub Pages zu verwenden, um unsere statische Site zu hosten, aber manuelles Erstellen und Bereitstellen war wirklich zu umständlich – bei jeder Änderung mussten wir lokal erstellen, packen und dann manuell in den `gh-pages`-Zweig pushen. Das ist nicht nur ineffizient, sondern auch fehleranfällig.

Um dieses Problem zu lösen (hauptsächlich um faul zu sein), benötigten wir einen automatisierten Bereitstellungsprozess. Dieser Artikel wird detailliert aufzeichnen, wie GitHub Actions automatische Bereitstellungsunterstützung für das HagiCode-Projekt hinzugefügt wurde, damit wir uns nur auf die Inhaltserstellung konzentrieren können und den Rest dem automatisierten Prozess überlassen können.

## Zielanalyse

Bevor wir loslegen, müssen wir zunächst klären, was diese Aufgabe eigentlich tun soll. Immerhin (hier wurde ein Fehler gemacht, es sollte Immerhin sein) schadet das Wetzen der Axt dem Holzhauen nicht.

### Kernanforderungen

1. **Automatisches Erstellen**: Wenn Code in den `main`-Zweig gepusht wird, wird der Erstellungsprozess automatisch ausgelöst.
2. **Automatische Bereitstellung**: Nach erfolgreichem Erstellen werden die generierten statischen Dateien automatisch auf GitHub Pages bereitgestellt.
3. **Umgebungskonsistenz**: Sicherstellen, dass die CI-Umgebung und die lokale Erstellungsumgebung konsistent sind, um die Peinlichkeit von "lokal läuft es, online Fehler" zu vermeiden.

### Technologieauswahl

Da HagiCode auf Docusaurus basiert (ein sehr beliebter React-Static-Site-Generator), können wir GitHub Actions verwenden, um dieses Ziel zu erreichen.

## GitHub Actions-Workflow konfigurieren

GitHub Actions ist der von GitHub bereitgestellte CI/CD-Service. Durch die Definition von Workflow-Dateien im YAML-Format im Code-Repository können wir verschiedene automatisierte Aufgaben anpassen.

### Workflow-Datei erstellen

Wir müssen eine neue Konfigurationsdatei im Ordner `.github/workflows` im Stammverzeichnis des Projekts erstellen, zum Beispiel `deploy.yml`. Wenn der Ordner nicht existiert, denken Sie daran, ihn zuerst manuell zu erstellen.

Die Kernlogik dieser Konfigurationsdatei lautet wie folgt:

1. **Auslösebedingungen**: Überwacht `push`-Ereignisse des `main`-Zweigs.
2. **Laufumgebung**: Neueste Version von Ubuntu.
3. **Erstellungsschritte**:
   - Code auschecken
   - Node.js installieren
   - Abhängigkeiten installieren (`npm install`)
   - Statische Dateien erstellen (`npm run build`)
4. **Bereitstellungsschritte**: Verwenden Sie die offiziell bereitgestellte `action-gh-pages`, um die Erstellungsergebnisse in den `gh-pages`-Zweig zu pushen.

### Wichtiger Konfigurationscode

Das Folgende ist die von uns übernommene Konfigurationsvorlage:

```yaml
name: Deploy to GitHub Pages

# Auslösebedingungen: Wenn in den main-Zweig gepusht wird
on:
  push:
    branches:
      - main
    # Bei Bedarf können Pfadfilter hinzugefügt werden, zum Beispiel nur bei Dokumentänderungen erstellen
    # paths:
    #   - 'docs/**'
    #   - 'package.json'

# Berechtigungen festlegen, dies ist wichtig für die Bereitstellung auf GitHub Pages
permissions:
  contents: read
  pages: write
  id-token: write

# Parallelitätssteuerung: Alte Erstellungen desselben Zweigs abbrechen
concurrency:
  group: "pages"
  cancel-in-progress: false

jobs:
  build:
    runs-on: ubuntu-latest
    steps:
      - name: Checkout
        uses: actions/checkout@v4
        # Hinweis: fetch-depth: 0 muss festgelegt werden, sonst kann die Erstellungsversion ungenau sein
        with:
          fetch-depth: 0

      - name: Setup Node
        uses: actions/setup-node@v4
        with:
          node-version: 20 # Es wird empfohlen, mit der lokalen Entwicklungsumgebung konsistent zu bleiben
          cache: 'npm'     # Das Aktivieren des Caches kann den Erstellungsprozess beschleunigen

      - name: Install dependencies
        run: npm ci
        # Verwenden Sie npm ci statt npm install, da es schneller und strenger ist und für CI-Umgebungen geeignet ist

      - name: Build website
        run: npm run build
        env:
          # Wenn Ihre Site-Erstellung Umgebungsvariablen benötigt, konfigurieren Sie sie hier
          # NODE_ENV: production
          # PUBLIC_URL: /your-repo-name
          
      - name: Upload artifact
        uses: actions/upload-pages-artifact@v3
        with:
          path: ./build # Docusaurus-Standardausgabeverzeichnis

  deploy:
    environment:
      name: github-pages
      url: ${{ steps.deployment.outputs.page_url }}
    runs-on: ubuntu-latest
    needs: build
    steps:
      - name: Deploy to GitHub Pages
        id: deployment
        uses: actions/deploy-pages@v4
```

## Fallstricke während der Implementierung

In der Praxis sind wir auf einige Probleme gestoßen. Hier teilen wir sie in der Hoffnung, dass Sie sie vermeiden können (oder sich im Voraus auf Lösungen vorbereiten können).

### 1. GitHub-Token-Berechtigungsproblem

Am Anfang der Konfiguration schlug die Bereitstellung immer mit Fehler 403 (Forbidden) fehl. Nach langer Suche stellte sich heraus, dass das Standard-`GITHUB_TOKEN` von GitHub keine Schreibberechtigung für Pages hatte.

**Lösung**: In den Repository-`Settings` -> `Actions` -> `General` -> `Workflow permissions` müssen Sie unbedingt **"Read and write permissions"** auswählen.

### 2. Erstellungsverzeichnispfad falsch

Docusaurus speichert die erstellten statischen Dateien standardmäßig im `build`-Verzeichnis. Aber einige Projekte (z. B. Create React App standardmäßig `build`, Vite standardmäßig `dist`) können anders konfiguriert sein. Wenn in Actions Fehler auftreten, dass Dateien nicht gefunden werden, denken Sie daran, die Pfadkonfiguration in `docusaurus.config.js` zu überprüfen.

### 3. Unterpfadproblem

Wenn Ihr Repository keine Benutzerhauptseite ist (d. h. nicht `username.github.io`), sondern eine Projekthauptseite (z. B. `username.github.io/project-name`), müssen Sie `baseUrl` konfigurieren.

In `docusaurus.config.js`:

```javascript
module.exports = {
  // ...
  url: 'https://hagicode.com', // Ihre Hagicode-URL
  baseUrl: '/', // Bereitstellung im Stammverzeichnis
  // ...
};
```
Dies wird leicht übersehen. Eine falsche Konfiguration führt dazu, dass die Seite beim Öffnen komplett weiß ist, da die Ressourcenpfade nicht geladen werden können.

## Ergebnisse validieren

Nach der Konfiguration aller Inhalte und dem Pushen des Codes können wir im GitHub-Repository auf der **Actions**-Registerkarte zusehen.

Sie sehen einen gelben Kreis (Workflow läuft), wenn er grün wird, bedeutet das Erfolg! Wenn er rot wird, klicken Sie hinein, um die Protokolle anzuzeigen. Normalerweise können Probleme diagnostiziert werden (meistens Tippfehler oder falsche Pfadkonfiguration).

Nach erfolgreichem Erstellen können Sie unter `https://<Ihr-Benutzername>.github.io/<Repository-Name>/` die brandneue Site sehen.

## Zusammenfassung

Durch die Einführung von GitHub Actions haben wir die automatische Bereitstellung der HagiCode-Dokumentationsseite erfolgreich realisiert. Dies spart nicht nur Zeit für manuelle Operationen, sondern更重要的是 garantiert die Standardisierung des Veröffentlichungsprozesses. Egal welcher Kollege die Dokumentation aktualisiert, sobald sie in den `main`-Zweig zusammengeführt wird, ist der neueste Inhalt nach wenigen Minuten online zu sehen.

**Kernvorteile**:
- **Effizienzsteigerung**: Von "manuell packen, manuell hochladen" zu "Code ist Veröffentlichung".
- **Fehlerreduzierung**: Beseitigt die Möglichkeit von menschlichen Bedienungsfehlern.
- **Erlebnisoptimierung**: Ermöglicht Entwicklern, sich mehr auf die Inhaltsqualität zu konzentrieren, anstatt von mühsamen Bereitstellungsprozessen gestört zu werden.

Obwohl die Einrichtung von CI/CD anfangs etwas umständlich ist (insbesondere bei verschiedenen Berechtigungs- und Pfadproblemen), ist es eine einmalige Investition mit langfristig hoher Rendite. Es wird dringend empfohlen, dass alle Static-Site-Projekte ähnliche automatisierte Prozesse einbinden.
