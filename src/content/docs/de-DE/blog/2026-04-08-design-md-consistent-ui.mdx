## Design.md: Eine Loesung fuer konsistentes KI-gesteuertes Frontend-UI-Design

> Im Zeitalter der KI-unterstuetzten Frontend-Entwicklung: Wie koennen Sie KI-generierte UIs konsistent halten? Dieser Artikel teilt unsere praktische Erfahrung beim Bau einer Design-Galerie-Website basierend auf awesome-design-md, sowie darueber, wie man ein strukturiertes `design.md` erstellt, um die KI zu standardisiertem UI-Design zu fuehren.

## Hintergrund

Jeder, der KI zum Schreiben von Frontend-Code verwendet hat, hat wahrscheinlich eine aehnliche Erfahrung gemacht: Man bittet die KI, dieselbe Seite mehrmals zu generieren, und jedes Ergebnis kommt in einem anderen Stil. Manchmal sind die Ecken abgerundet, manchmal scharf. Manchmal betraegt der Abstand 8px, und andere Male werden es 16px. Sogar derselbe Button kann in verschiedenen Gespraechen anders aussehen.

Dies ist kein Einzelfall. Je weiter die KI-unterstuetzte Entwicklung verbreitet wird, desto groesser wird das Problem der **mangelnden Konsistenz in KI-generierten Frontend-UIs**. Verschiedene KI-Assistenten, verschiedene Prompts und sogar derselbe Assistent in verschiedenen Gespraechen koennen drastisch unterschiedliche Interface-Designs produzieren.

Die Ursache ist eigentlich einfach: Es gibt kein **massgebliches Design-Referenzdokument**. Traditionelle CSS-Stylesheets koennen Entwicklern sagen, „wie man etwas implementiert", aber sie koennen nicht vollstaendig vermitteln, „warum es so gestaltet ist" oder „welches Designmuster in welchem Szenario verwendet werden sollte". Fuer die KI ist eine klare und strukturierte Beschreibung noch wichtiger, um Designkonventionen zu verstehen.

Gleichzeitig bietet die Open-Source-Community bereits einige ausgezeichnete Ressourcen. Das [VoltAgent/awesome-design-md](https://github.com/VoltAgent/awesome-design-md)-Projekt sammelt Design-System-Dokumentation vieler bekannter Unternehmen. Jedes Verzeichnis enthaelt eine `README.md`, eine `DESIGN.md` und eine Vorschau-HTML.

Koennen wir diese Ressourcen also in einer leicht durchsuchbaren Design-Galerie konsolidieren und gleichzeitig ein strukturiertes `design.md` fuer die KI destillieren?

Die Antwort ist ja. Lassen Sie uns unseren Ansatz durchgehen.

## Ueber HagiCode

Die in diesem Artikel geteilte Loesung stammt aus unserer praktischen Erfahrung im [HagiCode](https://hagicode.com)-Projekt. HagiCode ist eine KI-unterstuetzte Entwicklungsplattform, und waehrend der Entwicklung stießen wir auf dasselbe Problem inkonsistenter KI-generierter UIs. Um es zu loesen, bauten wir eine Design-Galerie-Website und erstellten ein standardisiertes `design.md`. Dieser Artikel ist eine Zusammenfassung dieser Loesung.

## Analyse

Bevor wir Code schreiben, lassen Sie uns die technischen Herausforderungen hinter diesem Problem zerlegen.

### Quellinhaltsverwaltung: Wie vereinheitlicht man verstreute Designressourcen?

Das Upstream awesome-design-md-Repository enthaelt eine grosse Anzahl von Designdokumenten, aber wir brauchten einen Weg, sie in unser eigenes Projekt zu bringen.

**Loesung: git submodule verwenden**

```
awesome-design-md-site
└── vendor/awesome-design-md          # Upstream-Ressourcen (git submodule)
```

### Datennormalisierung: Wie standardisiert man verschiedene Dokumentstrukturen?

Unterschiedliche Unternehmen strukturieren ihre Designdokumente unterschiedlich. Einige fehlen Vorschau-Dateien, und einige verwenden inkonsistente Benennung. Wir muessen sie waehrend des Build-Prozesses normalisieren.

**Loesung: Zur Build-Zeit scannen und normalisierte Eintraege generieren**

Das Kernmodul ist `awesomeDesignCatalog.ts`, verantwortlich fuer:
1. Scannen des Verzeichnisses `vendor/awesome-design-md/design-md/*`
2. Validieren, ob jeder Eintrag die erforderlichen Dateien enthaelt
3. Extrahieren und Rendern von Markdown-Inhalt in HTML
4. Generieren von normalisierten Eintragsdaten

### Statische Site-Architektur: Wie bietet man dynamische Suche bei vollstaendig statischer Site?

Da dies eine Design-Galerie ist, ist Suche ein Muss. Aber Astro ist ein statischer Site-Generator, also wie implementiert man Echtzeit-Suche?

**Loesung: React-Island + URL-Abfrageparameter-Synchronisation**

```typescript
// src/components/gallery/SearchToolbar.tsx

export function SearchToolbar() {
  const [query, setQuery] = useState('');

  // Mit URL synchronisieren
  useEffect(() => {
    const params = new URLSearchParams(window.location.search);
    setQuery(params.get('q') || '');
  }, []);

  // In Echtzeit filtern
  const filtered = entries.filter(entry =>
    entry.searchText.includes(query)
  );

  return <input value={query} onChange={e => {
    setQuery(e.target.value);
    updateURL(e.target.value);
  }} />;
}
```

### Designdokumentation: Wie hilft man der KI, Designstandards zu verstehen und zu befolgen?

Dies ist der Kern der gesamten Loesung. Wir muessen ein strukturiertes `design.md` erstellen, das die KI verstehen und anwenden kann.

**Loesung: Die Struktur des ClickHouse `DESIGN.md` verwenden**

[Das `DESIGN.md` von ClickHouse](https://github.com/ClickHouse/ClickHouse/blob/master/DESIGN.md) ist eine ausgezeichnete Referenz. Unser Ansatz ist: **Die Struktur wiederverwenden, den Inhalt neu schreiben**.

## Loesung

Basierend auf der obigen Analyse besteht unsere Loesung aus vier Kernmodulen.

### 1. Content-Ingestion-Pipeline

Dies ist das Fundament des gesamten Systems, verantwortlich fuer die Extraktion und Normalisierung von Inhalten aus Upstream-Ressourcen.

### 2. Galerien-Interface

Die Galerie-Oberflaeche umfasst drei Hauptteile:

**Startseite**: Zeigt ein Kartenraster aller Design-Eintraege

**Detailseite**: Bündelt die vollstaendigen Informationen fuer einen einzelnen Design-Eintrag

**Navigation**: Unterstuetzt die Rueckkehr zur Galerie und das Durchsuchen angrenzender Eintraege

### 3. Suchfunktion

Die Suchfunktion basiert auf clientseitiger Filterung, wobei der Zustand durch URL-Abfrageparameter erhalten bleibt.

### 4. Design-Referenzdokument (`design.md`)

Dies ist das Kernlieferobjekt der gesamten Loesung. Wir erstellen ein `design.md` im Projektstamm mit folgender Struktur:

```markdown
# Design-Referenz fuer [Projektname]

## 1. Visuelles Thema & Atmosphaere
- Gesamtstil-Beschreibung

## 2. Farbpalette & Rollen
- Primaer- und Unterstuetzungsfarben
- Semantische Farben (`success`, `warning`, `error`)

## 3. Typografie-Regeln
- Schriftfamilien
- Typskala (`h1-h6`, `body`, `small`)

## 4. Komponentenstile
- Button-Stil-Konventionen
- Formularkomponenten-Stile

## 5. Layout-Prinzipien
- Abstandssystem
- Grid und Breakpoints

## 6. Tiefe & Erhabenheit
- Schattenstufen
- `z-index`-Konventionen

## 7. Do's and Don'ts
- Haeufige Fehler und korrekte Ansaetze

## 8. Responsives Verhalten
- Breakpoint-Definitionen

## 9. Agent Prompt-Leitfaden
- Wie man dieses Dokument in KI-Prompts verwendet
```

## Praxis

### Implementierungsschritte

**Schritt 1: Das Submodul initialisieren**

```bash
# Das Upstream-Repository als Submodul hinzufuegen
git submodule add https://github.com/VoltAgent/awesome-design-md.git vendor/awesome-design-md

# Das Submodul initialisieren und aktualisieren
git submodule update --init --recursive
```

**Schritt 2: Die Content-Pipeline erstellen**

`awesomeDesignCatalog.ts` implementieren, einschliesslich:
- Dateiscan- und Validierungslogik
- Markdown-Rendering mit Astros integriertem Renderer
- Eintragsdatenextraktion

**Schritt 3: Die Galerien-UI bauen**

Astro + React Islands verwenden, um zu erstellen:
- Galerie-Layout der Startseite (Kartenraster)
- Designkarten-Komponenten
- Suchwerkzeugleiste
- Detailseiten-Layout

**Schritt 4: Das Designdokument schreiben**

Basierend auf der Struktur von ClickHouse `DESIGN.md`, die tatsaechlichen Design-Token des eigenen Projekts eintragen.

### Hinweise

**Sicherheit**: Markdown-Rendering erfordert das Filtern unsicheren HTML.

**Leistung**: Eine grosse Anzahl von iframe-Vorschauen kann die Erstbildschirm-Leistung beeintraechtigen. Es wird empfohlen, `loading="lazy"` zu verwenden.

**Wartbarkeit**: `design.md` muss mit der Code-Implementierung synchron bleiben. Es wird empfohlen, CI-Pruefungen hinzuzufuegen.

**Barrierefreiheit**: Stellen Sie sicher, dass der Farbkontrast den WCAG AA-Standard erfuellt (mindestens 4,5:1).

### KI-Nutzungsleitfaden

Nach der Erstellung von `design.md`, wie bringt man die KI dazu, es tatsaechlich zu verwenden? Hier sind einige praktische Tipps:

**Tipp 1: Im Prompt explizit referenzieren**

```
Bitte beziehen Sie sich auf die design.md-Datei im Projektstamm und verwenden Sie die dort definierten Designkonventionen, um die folgenden Komponenten zu implementieren:
- Buttons: Primaerfarbe mit 8px Rahmenradius
- Karten: Schattenstufe elevation-2
```

**Tipp 2: KI auffordern, spezifische CSS-Variablen zu referenzieren**

```
Implementieren Sie eine Navigationsleiste mit folgenden Anforderungen:
- --color-bg-primary fuer die Hintergrundfarbe
- --color-border-subtle fuer Rahmen
- --text-color-primary fuer Text
```

**Tipp 3: `design.md`-Inhalt in den System-Prompt aufnehmen**

Wenn Ihr KI-Werkzeug benutzerdefinierte System-Prompts unterstuetzt, koennen Sie den Kerninhalt von `design.md` direkt hinzufuegen.

## Zusammenfassung

Die Inkonsistenz in KI-generierten UIs, auf die HagiCode waehrend der Entwicklung stiess, lag im Kern an einem fehlenden strukturierten Design-Referenzdokument. Durch den Bau einer Design-Galerie-Website und die Erstellung eines standardisierten `design.md` loesten wir dieses Problem erfolgreich.

Der Kernwert dieser Loesung liegt in:
- **Vereinheitlichte Ressourcen**: Konsolidierung verstreuter Design-System-Dokumentation
- **Strukturierte Standards**: Ausdruck von Designkonventionen in einem Format, das die KI verstehen kann
- **Kontinuierliche Pflege**: Inhalte aktuell halten durch `git submodule`

Wenn Sie ebenfalls KI fuer die Frontend-Entwicklung verwenden, ist dieser Ansatz einen Versuch wert. Die Erstellung eines strukturierten `design.md` verbessert nicht nur die Konsistenz des KI-generierten Codes, sondern hilft auch Ihrem Team, interne einheitliche Designstandards beizubehalten.
