## Design.md: Eine Lösung für konsistente Frontend-UI-Designs mit AI

> Wie lässt sich die Konsistenz von KI-generierten UIs in der Ära der KI-gestützten Frontend-Entwicklung gewährleisten? Dieser Artikel teilt unsere praktischen Erfahrungen beim Aufbau eines Design-Galerie-Sites auf der Grundlage von awesome-design-md sowie wie man strukturierte design.md erstellt, um die KI bei der Erstellung standardisierter UI-Designs zu leiten.

## Hintergrund

Wer KI zum Schreiben von Frontend-Code verwendet hat, wahrscheinlich ähnliche Erfahrungen gemacht: Für dieselbe Seite die KI mehrfach Generieren zu lassen, führt jedes Mal zu einem anderen Stil. Manchmal sind es abgerundete Ecken, manchmal eckige; manchmal ist der Abstand 8px, dann wieder 16px; sogar derselbe Button sieht in verschiedenen Konversationen anders aus.

Dies ist kein Einzelfall. Mit der Verbreitung der KI-gestützten Entwicklung ist **die Inkonsistenz von KI-generierten Frontend-UIs** zu einem allgemeinen Problem geworden. Verschiedene KI-Assistenten, verschiedene Prompts und sogar derselbe Assistent in verschiedenen Konversationen produzieren stark unterschiedliche Interface-Designs. Dies bringt enorme Wartungskosten mit sich.

Die Wurzel des Problems ist eigentlich ganz einfach: Es fehlt eine **autoritative Design-Referenzdokumentation**. Herkömmliche CSS-Stildateien können Entwicklern nur sagen, "wie man es implementiert", aber nicht vollständig vermitteln, "warum es so entworfen wurde" und "in welchen Szenarien welches Design-Muster verwendet wird". Für eine KI ist es noch wichtiger, eine klare strukturierte Beschreibung zu haben, um Design-Richtlinien zu verstehen.

Gleichzeitig gibt es in der Open-Source-Community bereits einige gute Ressourcen. Das Projekt [VoltAgent/awesome-design-md](https://github.com/VoltAgent/awesome-design-md) sammelt Design-System-Dokumentationen vieler bekannter Unternehmen, wobei jedes Verzeichnis README.md, DESIGN.md und eine Vorschau-HTML enthält. Aber diese sind über die Upstream-Repositories verteilt und schwer schnell nachzuschlagen und zu vergleichen.

Könnte man diese Ressourcen zusammenfassen, eine leicht zugängliche Design-Galerie erstellen und gleichzeitig eine strukturierte design.md für die KI erstellen?

Die Antwort ist ja. Im Folgenden teilen wir unsere Lösung.

## Analyse

Bevor wir mit dem Codieren beginnen, lassen Sie uns die几个技术挑战 dieses Problems zerlegen.

### Inhaltsquellen-Verwaltung: Wie man verteilte Design-Ressourcen vereinheitlicht?

Das Upstream awesome-design-md Repository enthält eine große Anzahl an Design-Dokumentationen, aber wir brauchen eine Möglichkeit, es in unser Projekt zu integrieren.

**Lösung: git submodule verwenden**

```
awesome-design-md-site
└── vendor/awesome-design-md          # Upstream-Ressourcen (git submodule)
```

Dies hat mehrere Vorteile:
- Versionskontrolle: Kann bestimmte Upstream-Versionen festlegen
- Offline-Build: Keine Notwendigkeit, während des Builds externe APIs abzufragen
- Inhaltsprüfung: Konnte konkrete Änderungen in PRs sehen

### Daten-Standardisierung: Wie vereinheitlicht man verschiedene Dokumentstrukturen?

Die Design-Dokumentstrukturen verschiedener Unternehmen können unterschiedlich sein; einige fehlen Vorschau-Dateien, einige Benennungen sind nicht einheitlich. Wir müssen während des Builds eine Standardisierung durchführen.

**Lösung: Buildzeit-Scan und Generierung standardisierter Einträge**

3. Extrahieren und Rendern von Markdown-Inhalten als HTML
4. Generieren standardisierter Eintragsdaten

### Statische Site-Architektur: Wie man dynamische Suche bietet, während statische Bereitstellung beibehalten wird?

Da es sich um eine Design-Galerie handelt, ist die Suchfunktion erforderlich. Aber Astro ist ein statischer Site-Generator, wie implementiert man Echtzeitsuche?

**Lösung: React Island + URL-Abfrageparameter-Synchronisierung**

```typescript
// src/components/gallery/SearchToolbar.tsx

export function SearchToolbar() {
  const [query, setQuery] = useState('');

  // URL-Synchronisierung
  useEffect(() => {
    const params = new URLSearchParams(window.location.search);
    setQuery(params.get('q') || '');
  }, []);

  // Echtzeit-Filterung
  const filtered = entries.filter(entry =>
    entry.searchText.includes(query)
  );

  return <input value={query} onChange={e => {
    setQuery(e.target.value);
    updateURL(e.target.value);
  }} />;
}
```

Der Vorteil davon ist, dass die Bereitbarkeit der statischen Site erhalten bleibt (kann auf jedem statischen Hosting-Service bereitgestellt werden), während gleichzeitig eine sofortige Filter-Benutzererfahrung geboten wird.

### Design-Dokumentation: Wie lässt die KI Design-Richtlinien verstehen und einhalten?

Dies ist der Kern der gesamten Lösung. Wir müssen eine strukturierte design.md erstellen, damit die KI unsere Design-Richtlinien verstehen und anwenden kann.

**Lösung: Struktur von ClickHouse DESIGN.md übernehmen**

[ClickHouse's DESIGN.md](https://github.com/ClickHouse/ClickHouse/blob/master/DESIGN.md) ist eine gute Referenz. Sie enthält:
- Visual Theme & Atmosphere
- Color Palette & Roles
- Typography Rules
- Component Stylings
- Layout Principles
- Depth & Elevation
- Do's and Don'ts
- Responsive Behavior
- Agent Prompt Guide

Unser Ansatz ist: **Struktur übernehmen, Inhalt neu schreiben**. Die Kapitelstruktur der ClickHouse DESIGN.md beibehalten, aber den Inhalt durch die tatsächlichen Design-Tokens und Komponenten-Richtlinien unseres Projekts ersetzen.

## Lösung

Basierend auf der obigen Analyse umfasst unsere Lösung vier Kernmodule.

### 1. Inhaltsaufnahme-Pipeline

Dies ist die Grundlage des gesamten Systems, zuständig für das Extrahieren und Standardisieren von Inhalten aus Upstream-Ressourcen.

```typescript
// src/lib/content/awesomeDesignCatalog.ts

export async function scanSourceEntries(): Promise<DesignEntry[]> {
  const designDir = 'vendor/awesome-design-md/design-md';
  const entries: DesignEntry[] = [];

  for (const dir of await fs.readdir(designDir)) {
    const entryPath = path.join(designDir, dir);
    if (await isValidDesignEntry(entryPath)) {
      const entry = await normalizeDesignEntry(entryPath);
      entries.push(entry);
    }
  }

  return entries;
}

async function isValidDesignEntry(dir: string): Promise<boolean> {
  const requiredFiles = ['README.md', 'DESIGN.md'];
  for (const file of requiredFiles) {
    if (!(await fileExists(path.join(dir, file)))) {
      return false;
    }
  }
  return true;
}
```

### 2. Galerie-Browsing-Oberfläche

Die Galerie-Oberfläche umfasst drei Hauptteile:

**Homepage**: Zeigt ein Raster von Karten für alle Design-Einträge, jede Karte enthält:
- Titel und Zusammenfassung des Design-Eintrags
- Vorschau-Bild (falls vorhanden)
- Schnellsuch-Highlight

**Detailseite**: Aggregiert die vollständigen Informationen eines einzelnen Design-Eintrags:
- README-Dokumentation
- DESIGN-Dokumentation
- Vorschau (mit hellem/dunklem Theme-Wechsel)
- Navigation zu benachbarten Einträgen

**Navigation**: Unterstützt Rückkehr zur Galerie, Durchsuchen benachbarter Einträge

Die Homepage-Galerie verwendet ein hochdichtes Karten-Layout, das design.md-Einträge aus verschiedenen Quellen in einem einheitlichen visuellen Rahmen darstellt, was den schnellen Vergleich von Markenstilen, Button-Mustern und Typografie-Rhythmen erleichtert.

![Awesome Design MD Gallery Design-Karten-Raster](/article-images/design-md-consistent-ui/1775613199000-e652f7155dd5.png)

Nach dem Betreten eines spezifischen Eintrags platziert die Detailseite die Design-Zusammenfassung und die Echtzeitvorschau auf derselben Seite, was die Kosten des Hin- und Herwechsels zwischen Dokumentation, Vorschau und Quellcode reduziert.

### 3. Suchfunktion

Die Suchfunktion basiert auf clientseitiger Filterung und verwendet URL-Abfrageparameter zur Zustandsbeibehaltung:

```typescript
// src/components/gallery/SearchToolbar.tsx

function SearchToolbar({ entries }: { entries: DesignEntry[] }) {
  const [query, setQuery] = useState('');
  const [results, setResults] = useState(entries);

  useEffect(() => {
    const params = new URLSearchParams(window.location.search);
    const initialQuery = params.get('q') || '';
    setQuery(initialQuery);
    filterEntries(initialQuery);
  }, []);

  const filterEntries = (searchQuery: string) => {
    const filtered = entries.filter(entry =>
      entry.searchText.toLowerCase().includes(searchQuery.toLowerCase())
    );
    setResults(filtered);
  };

  const handleChange = (e: React.ChangeEvent<HTMLInputElement>) => {
    const value = e.target.value;
    setQuery(value);
    filterEntries(value);

    // URL aktualisieren (ohne Seiten-Refresh auszulösen)
    const newUrl = value
      ? `${window.location.pathname}?q=${encodeURIComponent(value)}`
      : window.location.pathname;
    window.history.replaceState({}, '', newUrl);
  };

  return (
    <div className="search-toolbar">
      <input
        type="text"
        value={query}
        onChange={handleChange}
        placeholder="Design-Einträge durchsuchen..."
      />
      <span className="result-count">{results.length} Ergebnisse</span>
    </div>
  );
}
```

## Praxis

Nachdem wir die Lösung verstanden haben, wie wird sie konkret umgesetzt?

### Umsetzungsschritte

**Schritt 1: Submodul initialisieren**

```bash
# Upstream-Repository als Submodul hinzufügen
git submodule add https://github.com/VoltAgent/awesome-design-md.git vendor/awesome-design-md

# Submodul initialisieren und aktualisieren
git submodule update --init --recursive
```

**Schritt 2: Inhaltspipeline erstellen**

Implementieren von `awesomeDesignCatalog.ts`, einschließlich:
- Dateiscan- und Validierungslogik
- Markdown-Rendering (unter Verwendung von Astros eingebautem Renderer)
- Eintragsdaten-Extraktion

**Schritt 3: Galerie-UI aufbauen**

Mit Astro + React Islands erstellen:
- Homepage-Galerie-Layout (Kartenraster)
- Design-Karten-Komponente
- Suchwerkzeugleiste
- Detailseiten-Layout

**Schritt 4: Design-Dokumentation schreiben**

Basierend auf der ClickHouse DESIGN.md Struktur die tatsächlichen Design-Tokens des Projekts ausfüllen. README.md aktualisieren und Links zu design.md hinzufügen.

### Zu beachtende Punkte

**Sicherheit**: Markdown-Rendering muss unsicheres HTML filtern. Astros eingebauter Renderer filtert standardmäßig script-Tags, aber XSS-Risiken sollten weiterhin beachtet werden.

**Leistung**: Viele iframe-Vorschauen können das erste Laden der Seite beeinträchtigen. Es wird empfohlen, `loading="lazy"` zu verwenden, um Vorschauinhalte verzögert zu laden.

**Wartbarkeit**: design.md muss mit der Code-Implementierung synchron gehalten werden. Es wird empfohlen, in CI Prüfungen hinzuzufügen, um sicherzustellen, dass CSS-Variablen in Dokumentation und Code konsistent sind.

**Barrierefreiheit**: Sicherstellen, dass Farbkontraste WCAG AA Standards erfüllen (mindestens 4,5:1).

### KI-Nutzungsleitfaden

Nach dem Erstellen von design.md, wie lässt sich die KI tatsächlich dazu bringen, es zu verwenden? Hier sind einige praktische Tipps:

**Tipp 1: Explizites Referenzieren im Prompt**

```
Bitte beziehen Sie sich auf die design.md-Datei im Projektstammverzeichnis und verwenden die darin definierten Design-Richtlinien, um die folgenden Komponenten zu implementieren:
- Button: Verwendung des primary-Farbtons, abgerundete Ecken 8px
- Karte: Verwendung der elevation-2 Schatten-Hierarchie
```

**Tipp 2: KI auffordern, spezifische CSS-Variablen zu referenzieren**

```
Implementieren Sie eine Navigationsleiste mit folgenden Anforderungen:
- Hintergrundfarbe Verwendung von --color-bg-primary
- Rahmen Verwendung von --color-border-subtle
- Text Verwendung von --text-color-primary
```

**Tipp 3: design.md-Inhalte in System-Prompt aufnehmen**

Wenn Ihr KI-Tool benutzerdefinierte System-Prompts unterstützt, können Sie den Kerninhalt von design.md direkt hinzufügen.

### Teststrategie

**Inhaltspipelinetests**:
- Dateifehl-Szenarien (fehlende README.md oder DESIGN.md)
- Formatfehler-Szenarien (Markdown-Parsing fehlgeschlagen)
- Leere Verzeichnis-Szenarien

**Suchfunktionstests**:
- Leere Ergebnisbehandlung
- Sonderzeichen (wie Chinesisch, Emoji)
- URL-Synchronisierungsverifizierung

**UI-Komponententests**:
- Hell/Dunkel Theme-Wechsel
- Responsives Layout
- Vorschauladestatus

### Bereitstellungsprozess

```bash
# 1. Submodul auf neueste Version aktualisieren
git submodule update --remote

# 2. Site neu erstellen
npm run build

# 3. Statische Ressourcen bereitstellen
npm run deploy
```

Es wird empfohlen, die Submodul-Aktualisierung und Build-Bereitstellung zu automatisieren. CI-Prozesse können automatisch ausgelöst werden, wenn das Upstream-Repository aktualisiert wird.

## Zusammenfassung

Das Problem der inkonsistenten KI-generierten UIs, auf das HagiCode während der Entwicklung gestoßen ist, liegt im Wesentlichen an fehlenden strukturierten Design-Referenzdokumentationen. Durch den Aufbau einer Design-Galerie-Site und das Erstellen standardisierter design.md haben wir dieses Problem erfolgreich gelöst.

Der Kernwert dieser Lösung liegt in:
- **Vereinheitlichte Ressourcen**: Integration verteilter Design-System-Dokumentationen
- **Strukturierte Richtlinien**: Darstellung von Design-Richtlinien in einer für KI verständlichen Form
- **Kontinuierliche Wartung**: Inhaltliche Aktualisierung durch git submodule

Wenn Sie auch KI-gestützte Frontend-Entwicklung verwenden, empfehlen wir, diese Lösung zu versuchen. Erstellen Sie eine strukturierte design.md, die nicht nur die Konsistenz der KI-generierten Codes verbessert, sondern auch dem Team hilft, Design-Richtlinien intern einheitlich zu halten.
