## Hermes Agent Integrationspraxis: Vom Protokoll zum Produktionseinsatz

> Teilen der vollstaendigen HagiCode-Erfahrung bei der Integration von Hermes Agent, einschliesslich zentraler Erkenntnisse zur ACP-Protokollanpassung, Session-Pool-Verwaltung und Frontend-Backend-Vertragssynchronisation.

## Hintergrund

Waehrend des Baus von HagiCode, einer KI-gestuetzten Coding-Plattform, musste unser Team ein Agent-Framework integrieren, das lokal laufen und auch in die Cloud skalieren konnte. Nach Recherche wurde Hermes Agent von Nous Research als zugrundeliegende Engine fuer unsere allgemeinen Agent-Faehigkeiten gewaehlt.

Wahrlich ist Technologieauswahl weder besonders schwer noch besonders leicht. Es gibt viele starke Agent-Frameworks auf dem Markt, aber Hermes stach hervor, weil sein ACP-Protokoll und sein Toolsystem besonders gut zu HagiCodes anspruchsvollen Anforderungen passten: lokale Entwicklung, Teamzusammenarbeit und Cloud-Erweiterung. Dennoch bedeutete die Integration von Hermes in ein echtes Produktionssystem, eine lange Liste von Engineering-Problemen zu loesen. Dieser Teil war alles andere als trivial.

HagiCodes Stack verwendet Orleans zum Aufbau eines verteilten Systems, waehrend das Frontend mit React + TypeScript gebaut ist. Die Integration von Hermes bedeutete, die architektonische Konsistenz zu wahren und Hermes gleichzeitig als erstklassigen Ausfuehrer neben ClaudeCode und OpenCode zu machen. Es klingt einfach genug, aber die Implementierung erzaehlt immer die echte Geschichte.

Dieser Artikel teilt unsere praktische Erfahrung bei der Integration von Hermes Agent in HagiCode, und wir hoffen, dass er nuetzliches Referenzmaterial fuer Teams mit aehnlichen Beduerfnissen bietet. Denn wenn man einmal in eine Grube gefallen ist, gibt es keinen Grund, jemanden anderen in dieselbe fallen zu lassen.

## Architekturentwurf

### Schichtenweiser Designansatz

HagiCodes Hermes-Integration verwendet eine klare Schichtenarchitektur, wobei jede Schicht ihre eigenen Verantwortlichkeiten hat:

**Backend-Kernschicht**
- `HermesCliProvider`: implementiert die `IAIProvider`-Schnittstelle als einheitlichen KI-Provider-Einstiegspunkt
- `HermesPlatformConfiguration`: verwaltet Hermes-Ausfuehrungspfad, Argumente, Authentifizierung und verwandte Einstellungen
- `ICliProvider<HermesOptions>`: die Low-Level-CLI-Abstraktion von HagiCode.Libs fuer die Verwaltung von Subprozess-Lebenszyklen

**Transportschicht**
- `StdioAcpTransport`: kommuniziert mit dem Hermes ACP-Subprozess ueber Standard-Ein- und Ausgabe
- ACP-Protokollmethoden: `initialize`, `authenticate`, `session/new`, `session/prompt`

**Laufzeitschicht**
- `HermesGrain`: Orleans Grain-Implementierung, die verteilte Sitzungsausfuehrung behandelt
- `CliAcpSessionPool`: Session-Pool, der ACP-Subprozesse wiederverwendet, um haeufige Start-Overhead zu vermeiden

**Frontend-Schicht**
- `ExecutorAvatar`: Hermes visuelle Identitaet und Icon
- `executorTypeAdapter`: Provider-Typ-Zuordnungslogik
- SignalR Echtzeit-Messaging: bewahrt die Hermes-Identitaetskonsistenz im gesamten Nachrichtenstrom

Dieses Schichtendesign ermoeglicht es jeder Schicht, sich unabhaengig weiterzuentwickeln. Wenn wir beispielsweise in Zukunft einen neuen Transportmechanismus wie WebSocket hinzufuegen moechten, muessen wir nur die Transportschicht aendern. Es ist nicht noetig, das gesamte System umzukrempeln, nur weil sich ein Transport aendert.

### Einheitliche Schnittstellenabstraktion

Alle KI-Provider implementieren die `IAIProvider`-Schnittstelle, was eine der zentralen Designentscheidungen in HagiCodes Architektur ist:

```csharp
public interface IAIProvider
{
    string Name { get; }
    ProviderCapabilities Capabilities { get; }

    IAsyncEnumerable<AIStreamingChunk> StreamAsync(
        AIRequest request,
        CancellationToken cancellationToken = default);

    Task<AIResponse> ExecuteAsync(
        AIRequest request,
        CancellationToken cancellationToken = default);
}
```

`HermesCliProvider` implementiert diese Schnittstelle und steht auf Augenhoehe mit `ClaudeCodeProvider`, `OpenCodeProvider` und anderen. Die Vorteile dieses Designs umfassen:

1. **Austauschbarkeit**: Das Wechseln von Providern beeinflusst die obere Geschaeftslogik nicht
2. **Testbarkeit**: Provider koennen einfach fuer Unit-Tests gemockt werden
3. **Erweiterbarkeit**: Das Hinzufuegen eines neuen Providers erfordert nur die Implementierung der Schnittstelle

Letztendlich sind Schnittstellen viel wie Regeln. Wenn die Regeln einmal feststehen, koennen alle harmonisch zusammenexistieren, ihre Staerken ausspielen und sich nicht in die Quere kommen. Darin liegt eine gewisse Eleganz.

## Kernimplementierung

### Provider-Schicht-Implementierung

`HermesCliProvider` ist der Kern der gesamten Integration. Er koordiniert die verschiedenen Komponenten, die fuer einen einzelnen KI-Aufruf benoetigt werden:

```csharp
public sealed class HermesCliProvider : IAIProvider, IVersionedAIProvider
{
    private readonly ICliProvider<LibsHermesOptions> _provider;
    private readonly ConcurrentDictionary<string, string> _sessionBindings;

    public ProviderCapabilities Capabilities { get; } = new()
    {
        SupportsStreaming = true,
        SupportsTools = true,
        SupportsSystemMessages = true,
        SupportsArtifacts = false
    };

    public async IAsyncEnumerable<AIStreamingChunk> StreamAsync(
        AIRequest request,
        [EnumeratorCancellation] CancellationToken cancellationToken = default)
    {
        // 1. Den Session-Binding-Schluessel aufloesen
        var bindingKey = ResolveBindingKey(request.CessionId);

        // 2. Ueber den Session-Pool eine Hermes-Sitzung abrufen oder erstellen
        var options = new HermesOptions
        {
            ExecutablePath = _platformConfiguration.ExecutablePath,
            Arguments = _platformConfiguration.Arguments,
            SessionId = _sessionBindings.TryGetValue(bindingKey, out var sessionId) ? sessionId : null,
            WorkingDirectory = request.WorkingDirectory,
            Model = request.Model
        };

        // 3. Ausfuehren und Streaming-Antwort sammeln
        await foreach (var message in _provider.ExecuteAsync(options, request.Prompt, cancellationToken))
        {
            // 4. ACP-Nachrichten in AIStreamingChunk zuordnen
            if (_responseMapper.TryConvertToStreamingChunk(message, out var chunk))
            {
                yield return chunk;
            }
        }
    }
}
```

Mehrere Designpunkte sind hier besonders wichtig:

1. **Session-Binding**: verwendet `CessionId`, um mehrere Anfragen an denselben Hermes-Subprozess zu binden, um die Kontinuitaet des Kontexts ueber mehrrundige Konversationen hinweg zu bewahren
2. **Response-Zuordnung**: konvertiert das Hermes ACP-Nachrichtenformat in das einheitliche `AIStreamingChunk`-Format
3. **Streaming-Unterstuetzung**: verwendet `IAsyncEnumerable` fuer echte Streaming-Antworten

Session-Binding ist ein bisschen wie menschliche Beziehungen. Wenn eine Verbindung hergestellt ist, hat die zukuenftige Kommunikation Kontext, sodass man nicht jedes Mal bei null anfangen muss. Diese Beziehung muss aber trotzdem gepflegt werden.

### ACP-Protokollanpassung

Hermes verwendet ACP (Agent Communication Protocol), das sich von einer traditionellen HTTP-API unterscheidet. ACP ist ein Protokoll basierend auf Standard-Ein- und Ausgabe und hat mehrere Merkmale:

1. **Startmarker**: Nach dem Start des Hermes-Prozesses gibt er den `//ready`-Marker aus
2. **Dynamische Authentifizierung**: Authentifizierungsmethoden sind nicht fest und muessen ueber das Protokoll verhandelt werden
3. **Session-Wiederverwendung**: bestehende Sitzungen werden ueber `SessionId` wiederverwendet
4. **Fragmentierte Antworten**: Eine vollstaendige Antwort kann ueber mehrere `session/update`-Benachrichtigungen aufgeteilt werden

HagiCode behandelt diese Merkmale durch `StdioAcpTransport`:

```csharp
public class StdioAcpTransport
{
    public async Task InitializeAsync(CancellationToken cancellationToken)
    {
        // Warten auf den //ready-Marker
        var readyLine = await _outputReader.ReadLineAsync(cancellationToken);
        if (readyLine != "//ready")
        {
            throw new InvalidOperationException("Hermes did not send ready signal");
        }

        // Initialize-Anfrage senden
        await SendRequestAsync(new
        {
            jsonrpc = "2.0",
            id = 1,
            method = "initialize",
            @params = new
            {
                protocolVersion = "2024-11-05",
                capabilities = new { },
                clientInfo = new { name = "HagiCode", version = "1.0.0" }
            }
        }, cancellationToken);
    }
}
```

Protokolle sind ein bisschen wie gegenseitiges Verstaendnis zwischen Menschen. Wenn dieses Verstaendnis da ist, fliesst die Kommunikation viel reibungsloser. Es braucht nur Zeit, es aufzubauen.

### Session-Pool-Verwaltung

Das haeufige Starten von Hermes-Subprozessen ist teuer, daher haben wir einen Session-Pool-Mechanismus implementiert:

```csharp
services.AddSingleton(static _ =>
{
    var registry = new CliProviderPoolConfigurationRegistry();
    registry.Register("hermes", new CliPoolSettings
    {
        MaxActiveSessions = 50,
        IdleTimeout = TimeSpan.FromMinutes(10)
    });
    return registry;
});
```

Schluessel-Session-Pool-Parameter:
- `MaxActiveSessions`: steuert das Nebenlaeufigkeitslimit, um Ressourcenerschöpfung zu vermeiden
- `IdleTimeout`: Idle-Timeout, das Startkosten gegen Speichernutzung ausbalanciert

In der Praxis haben wir festgestellt, dass:
1. Wenn das Idle-Timeout zu kurz ist, starten Sessions haeufig neu; ist es zu lang, bleibt der Speicher belegt
2. Das Nebenlaeufigkeitslimit muss an die tatsaechliche Last angepasst werden, da eine zu hohe Einstellung das System traege machen kann
3. Die Session-Pool-Auslastung muss ueberwacht werden, damit Parameter rechtzeitig angepasst werden koennen

Das ist vieles wie viele Entscheidungen im Leben: zu aggressiv schafft Probleme, zu konservativ verpasst man Chancen. Das Ziel ist einfach, die richtige Balance zu finden.

## Frontend-Integration

Das Frontend muss den Hermes-Provider korrekt identifizieren und die entsprechenden visuellen Elemente anzeigen:

```typescript
// executorTypeAdapter.ts

      return 'Hermes';

      return 'Unknown';
```

Schoene Dinge verdienen schliesslich eine schoene Praesentation. Dass diese Schoenheit tatsaechlich sichtbar wird, haengt aber von der Frontend-Handwerkskunst ab.

### Vertragssynchronisation

Frontend und Backend halten ihren Vertrag ueber OpenAPI-Generierung synchron. Das Backend definiert den `AIProviderType`-Enum:

```csharp
public enum AIProviderType
{
    Unknown,
    ClaudeCode,
    OpenCode,
    HermesCli  // Neu hinzugefuegt
}
```

Das Frontend generiert den entsprechenden TypeScript-Typ ueber OpenAPI, um sicherzustellen, dass Enum-Werte konsistent bleiben. Das ist der Schluessel, um zu vermeiden, dass das Frontend `Unknown` anzeigt.

Ein Vertrag ist ein bisschen wie ein Versprechen. Wenn er einmal vereinbart ist, muss er eingehalten werden, sonst landet man in peinlichen Situationen wie `Unknown`.

## Konfigurationsverwaltung

Hermes-Konfiguration wird ueber `appsettings.json` verwaltet:

```json
{
  "Providers": {
    "HermesCli": {
      "ExecutablePath": "hermes",
      "Arguments": "acp",
      "StartupTimeoutMs": 10000,
      "ClientName": "HagiCode",
      "Authentication": {
        "PreferredMethodId": "api-key",
        "MethodInfo": {
          "api-key": "your-api-key-here"
        }
      },
      "SessionDefaults": {
        "Model": "claude-sonnet-4-20250514",
        "ModeId": "default"
      }
    }
  }
}
```

Dieses konfigurationsgesteuerte Design bringt Flexibilitaet:
- Ausfuehrungspfade koennen ueberschrieben werden, was bequem fuer Entwicklung und Tests ist
- Startargumente koennen an verschiedene Hermes-Versionen angepasst werden
- Authentifizierungsinformationen koennen konfiguriert werden, um mehrere Authentifizierungsmethoden zu unterstuetzen

Konfiguration ist ein bisschen wie Multiple-Choice-Fragen im Leben. Wenn genug Optionen verfuegbar sind, gibt es normalerweise eine, die passt. Allerdings koennen zu viele Optionen auch zu Entscheidungsmuedigkeit fuehren.

## Praktische Erfahrung

### Gesundheitspruefungen

Ein zuverlaessiger Provider erfordert umfassende Gesundheitspruefungen:

```csharp
public async Task<ProviderTestResult> PingAsync(CancellationToken cancellationToken = default)
{
    var response = await ExecuteAsync(new AIRequest
    {
        Prompt = "Reply with exactly PONG.",
        CessionId = null,
        AllowedTools = Array.Empty<string>(),
        WorkingDirectory = ResolveWorkingDirectory(null)
    }, cancellationToken);

    var success = string.Equals(response.Content.Trim(), "PONG", StringComparison.OrdinalIgnoreCase);
    return new ProviderTestResult
    {
        ProviderName = Name,
        Success = success,
        ResponseTimeMs = stopwatch.ElapsedMilliseconds,
        ErrorMessage = success ? null : $"Unexpected Hermes ping response: '{response.Content}'."
    };
}
```

Punkte, die bei Gesundheitspruefungen zu beachten sind:
1. Verwenden Sie einfache Testfaelle und vermeiden Sie ueberkomplexe Szenarien
2. Setzen Sie angemessene Timeout-Werte
3. Zeichnen Sie die Antwortzeit auf, um Leistungsanalysen zu unterstuetzen

So wie Menschen körperliche Vorsorgeuntersuchungen brauchen, braucht auch ein System Gesundheitspruefungen. Je frueher Probleme gefunden werden, desto leichter sind sie zu beheben.

### Umgang mit haeufigen Problemen

**Authentifizierungsfehler**
- Pruefen Sie, ob `Authentication.PreferredMethodId` mit der Authentifizierungsmethode uebereinstimmt, die Hermes tatsaechlich unterstuetzt
- Bestaetigen Sie, dass das Format der Authentifizierungsinformationen korrekt ist, wie API Key oder Bearer Token

**Session-Timeout**
- Erhoehen Sie den Wert von `StartupTimeoutMs`
- Pruefen Sie die Erreichbarkeit des MCP-Servers
- Pruefen Sie die Systemressourcenauslastung

**Unvollstaendige Antwort**
- Stellen Sie sicher, dass `session/update`-Benachrichtigungen und das Endergebnis korrekt aggregiert werden
- Pruefen Sie die Abbruchlogik in der Streaming-Verarbeitung
- Verifizieren Sie, dass die Fehlerbehandlung vollstaendig ist

**Frontend zeigt Unknown**
- Bestaetigen Sie, dass die OpenAPI-Generierung bereits den `HermesCli`-Enum-Wert enthaelt
- Pruefen Sie, ob die Typzuordnung korrekt ist
- Leeren Sie den Browser-Cache und generieren Sie Typen neu

Probleme wird es immer geben. Wenn sie auftreten, ist es wichtig, nicht in Panik zu geraten. Verfolgen Sie die Ursache Schritt fuer Schritt, und am Ende koennen die meisten geloest werden.

### Leistungsoptimierungsvorschlaege

1. **Session-Pool verwenden**: ACP-Subprozesse wiederverwenden, um Start-Overhead zu reduzieren
2. **Timeouts angemessen setzen**: Speichernutzung gegen Startkosten ausbalancieren
3. **Session-IDs wiederverwenden**: dieselbe `CessionId` fuer Stapelaufgaben verwenden
4. **MCP bei Bedarf konfigurieren**: unnötige Tool-Aufrufe vermeiden

Leistung ist ein bisschen wie Effizienz im Alltag. Wenn man es richtig macht, erreicht man mehr mit weniger; wenn man es falsch macht, vervielfachen sich die Anstrengungen, waehrend die Ergebnisse schrumpfen. Den „richtigen" Punkt zu finden, erfordert sowohl Erfahrung als auch Glueck.

## Zusammenfassung

Die Integration von Hermes Agent in ein Produktionssystem erfordert die Beruecksichtigung von Problemen ueber mehrere Dimensionen:

1. **Architektur**: eine einheitliche Provider-Schnittstelle entwerfen und eine austauschbare Komponentenarchitektur implementieren
2. **Protokoll**: ACP-spezifisches Verhalten wie Startmarker und dynamische Authentifizierung korrekt behandeln
3. **Leistung**: Ressourcen ueber den Session-Pool wiederverwenden und Startkosten gegen Speichernutzung ausbalancieren
4. **Frontend**: Vertragssynchronisation sicherstellen und ein einheitliches visuelles Erlebnis bieten

HagiCodes Erfahrung zeigt, dass mit gutem Schichtendesign und konfigurationsgesteuerter Implementierung ein komplexes Agent-System nahtlos in eine bestehende Architektur integriert werden kann.

Diese Prinzipien klingen einfach, wenn sie in Worten beschrieben werden, aber die tatsaechliche Implementation fuehrt immer viele verschiedene Arten von Problemen mit sich. Das ist in Ordnung. Wenn ein Problem geloest wird, wird es zu Erfahrung. Wenn nicht, wird es zu einer Lektion. Beides hat Wert.

Schoene Dinge oder Menschen muessen nicht besessen werden; solange sie schoen bleiben, reicht es, ihre Schoenheit still zu wuerdigen. Technik ist nicht anders. Wenn es hilft, das System besser zu machen, dann ist das spezifische Framework oder Protokoll viel weniger wichtig, als Menschen manchmal denken.
