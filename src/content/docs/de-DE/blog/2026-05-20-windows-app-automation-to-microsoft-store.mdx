## Automatisierungspraxis für das automatische Veröffentlichen von Windows-Apps im Microsoft Store

> Wie man die vollständige Prozessautomatisierung von Versionsauflösung bis Store-Veröffentlichung实现，abschied vom umständlichen manuellen Prozess.

## Hintergrund

Wenn Sie eine Electron-Anwendung im Microsoft Store veröffentlichen möchten，werden Sie wahrscheinlich auf dieses Problem stoßen: Der Store unterstützt keine getrennten Installationsschritte – Sie müssen die Desktop-Anwendung und die Server-Payload in ein einzelnene完整的 AppX/MSIX-Paket paketieren.

Das wäre noch nicht das Ende der Probleme. Bei jeder neuen Version müssen Sie:

1. Überprüfen，ob Desktop- und Server-Versionen aktualisiert wurden
2. Code aus dem entsprechenden Tag auschecken
3. Server-Payload herunterladen und injizieren
4. MSIX-Paket erstellen
5. Manuell zum Microsoft Store hochladen
6. Shop-Informationen und Preise konfigurieren

Wenn jeder Schritt manuell ausgeführt werden muss，ist das zu mühsam. Außerdem ist es fehleranfällig – man weiß vielleicht nicht mehr genau，welcher Schritt bereits ausgeführt wurde und welcher noch aussteht.

Eigentlich kann man niemandem die Schuld geben，denn manuelle Operationen sind ohnehin anfällig für Übersehungsfehler. Aber wir wollten nicht jedes Mal so viel Aufwand betreiben，also beschlossen wir，das Problem gründlich zu lösen – den gesamten Prozess automatisch ablaufen zu lassen.

## Analyse der technischen Architektur

Dieses Problem实际上 involve die Koordination mehrerer technischer Ebenen. Wir können es in fünf Ebenen unterteilen:

### Version-Koordinations-Ebene

Zuerst muss man wissen，wann eine neue Version veröffentlicht werden muss. Wir müssen die neuesten Versionen der Desktop- und Server-Komponenten aus dem Azure-Index (unserer blob storage für Build-Artefakte)解析，然后判断，ob ein neues Store-Paket generiert werden muss.

Das ist wie die Frage an sich selbst: Jetzt ist es an der Zeit，das zu tun?

### Arbeitsbereich-Management-Ebene

Der AppX-Bau hängt von Konfigurationen auf Quellcode-Ebene und Layout zur Laufzeit ab，daher können einfache fertige Build-Artefakte nicht einfach neu paketiert werden. Wir müssen Code aus einem bestimmten Tag des Desktop-Repositorys auschecken，um sicherzustellen，dass der Quellcode-Zustand，der für den Build verwendet wird，korrekt ist.

Schließlich，wenn der Quellcode nicht stimmt，sind alle nachfolgenden Bemühungen umsonst.

### Laufzeit-Paketierungs-Ebene

Das ist der Kern-Teil. Wir müssen die Server-Payload in das Paketierungs-Layout der Desktop-Anwendung injizieren. So erkennt die Anwendung beim Start，dass die paketierte Laufzeit vorhanden ist und geht in den Steam-Modus (Offline-Modus).

Wenn dieser Schritt nicht ordentlich durchgeführt wird，sind alle vorherigen Bemühungen vergeblich.

### Build-Ausgabe-Ebene

Verwenden von electron-builder，um ein MSIX-Paket zu generieren，das den Store-Anforderungen entspricht. Dieser Schritt muss in einer Windows-Umgebung ausgeführt werden，da der AppX-Bau das Windows SDK benötigt.

Manche Dinge müssen einfach am richtigen Ort getan werden – am falschen Ort geht es nicht.

### Veröffentlichungs- und Distributions-Ebene

Zum Schluss wird es auf GitHub Releases und im Microsoft Store veröffentlicht. GitHub Release dient als Sicherung und Versionsnachverfolgung，während der Microsoft Store für Endbenutzer gedacht ist.

Alles ist bereit，es fehlt nur noch dieser letzte Schritt.

Der gesamte Prozess wird von GitHub Actions gesteuert und kann zeitgesteuert (alle 4 Stunden) oder manuell ausgelöst werden. Bei manueller Auslösung können spezifische Desktop- und Server-Versionen angegeben oder ein erzwungener Neubau durchgeführt werden.

Alle vier Stunden ist eigentlich nicht zu häufig – schließlich gehen Code-Updates mal schneller，mal langsamer，aber so ist es auf der sicheren Seite.

## Detaillierte Implementierung

### 1. Bau-Plan-Auflösung

Der erste Schritt besteht darin，festzulegen，ob eine neue Version erstellt werden muss. Wir müssen die aktuellen Versionen der Desktop- und Server-Komponenten解析 und dann überprüfen，ob bereits entsprechende Veröffentlichungen existieren.

```javascript
// scripts/resolve-dispatch-build-plan.mjs
export async function resolveDispatchBuildPlan({
  eventName,
  eventPayload,
  desktopAzureSasUrl,
  serverAzureSasUrl,
}) {
  // Trigger-Inputs解析
  const trigger = normalizeTriggerInputs({ eventName, eventPayload });
  
  // Desktop- und Server-Versionen aus Azure-Index解析
  const [desktopRelease, serverRelease] = await Promise.all([
    resolveIndexRelease({ 
      azureSasUrl: desktopAzureSasUrl,
      platformId: 'win-x64' 
    }),
    resolveIndexRelease({ 
      azureSasUrl: serverAzureSasUrl,
      platformId: 'win-x64' 
    })
  ]);
  
  // Release-Tag generieren
  const desktopTag = normalizeGitTag(desktopRelease.version);
  const releaseTag = deriveStoreReleaseTag(desktopRelease.version, serverRelease.version);
  // Zum Beispiel: desktop-v1.2.3-server-v4.5.6
  
  // Überprüfen，ob bereits vorhanden (doppelten Bau vermeiden)
  const existingRelease = await findReleaseByTag(packerRepository, releaseTag);
  const shouldBuild = !existingRelease || trigger.forceRebuild;
  
  return {
    release: { tag: releaseTag, exists: Boolean(existingRelease) },
    build: { shouldBuild, skipReason },
    upstream: { desktop: { tag: desktopTag }, server }
  };
}
```

Der Schlüssel in diesem Schritt ist die Überspring-Logik – wenn die Kombination aus gleichen Versionen bereits gebaut wurde，gibt es keinen Grund，den kompletten Bau-Prozess erneut ablaufen zu lassen. Das spart CI-Kosten und Zeit.

Dasselbe noch einmal zu tun，hat wohl kaum Sinn. Schließlich sind Zeit und Ressourcen begrenzt.

### 2. Arbeitsbereich-Vorbereitung

Nachdem festgelegt wurde，dass gebaut werden soll，muss eine saubere Bau-Umgebung vorbereitet werden. Wir verwenden git worktree，um Code aus einem bestimmten Tag auszuchecken.

```javascript
// scripts/prepare-packaging-workspace.mjs
export async function preparePackagingWorkspace({
  planPath,
  platformId,
  workspacePath,
  desktopSourcePath
}) {
  // Mit git worktree Desktop-Code aus einem bestimmten Tag auschecken
  await runCommand('git', [
    '-C', resolvedDesktopSourcePath,
    'worktree', 'add', '--detach',
    desktopWorkspace,
    `refs/tags/${plan.upstream.desktop.tag}`
  ]);
  
  // Arbeitsbereich validieren
  const validation = await validateDesktopWorkspace({
    desktopWorkspace,
    storePackageConfig
  });
  
  // Arbeitsbereich-Manifest erstellen
  const workspaceManifest = {
    desktopWorkspace,
    runtimeInjectionRoot: validation.runtimeRoot,
    desktopTag: plan.upstream.desktop.tag,
    desktopRef,
  };
  
  return workspaceManifest;
}
```

Der Vorteil von worktree besteht darin，dass das Hauptarbeitsverzeichnis nicht beeinflusst wird，Baut parallel durchgeführt werden können und nach Abschluss des Baus einfach aufgeräumt werden kann.

Schließlich möchte niemand，dass der Bau das Hauptarbeitsverzeichnis durcheinander bringt.

### 3. Server-Payload-Injektion

Dieser Schritt lädt die Server-Payload herunter und injiziert sie an die richtige Stelle.

```javascript
// scripts/stage-server-payload.mjs
export async function stageServerPayload({
  planPath,
  workspacePath,
  platformId,
  azureSasUrl
}) {
  // Server-Payload aus Azure-Index herunterladen
  const assetSource = resolveAssetDownloadUrl({ asset, sasUrl: azureSasUrl });
  await downloadFromSource({ 
    sourceUrl: assetSource, 
    destinationPath: downloadPath 
  });
  
  // Entpacken und validieren
  await extractArchive(downloadPath, extractionPath);
  const runtimeRoot = await resolveRuntimeRoot(extractionPath);
  const validation = await validateServerPayloadRoot(runtimeRoot, platformId);
  
  // In Paketierungs-Laufzeit-Layout injizieren
  // Zielpfad ist resources/portable-fixed/current
  // Dieser Pfad wird in AppX auf extra/portable-fixed/current abgebildet
  await copyDir(runtimeRoot, targetPath);
}
```

Der Schlüssel ist die Pfad-Abbildung – `resources/portable-fixed/current` wird im AppX als `extra/portable-fixed/current` paketiert，so dass die Anwendung beim Start die lokale Laufzeit erkennt und in den Offline-Modus geht.

Bei Pfaden darf nichts falsch sein – eine kleine Abweichung und es wird nicht gefunden.

### 4. MSIX-Bau

Mit dem vorbereiteten Arbeitsbereich und der Server-Payload kann das MSIX-Paket erstellt werden.

```javascript
// scripts/build-appx.mjs
export async function buildAppx({
  planPath,
  workspacePath,
  platformId
}) {
  // Store-spezifische electron-builder-Konfigurations-Überlagerung generieren
  const overlayConfig = await writeStoreElectronBuilderConfig({
    desktopWorkspace: workspaceManifest.desktopWorkspace,
    sourceConfigPath: storePackageConfig.desktop.electronBuilderConfigPath,
    outputConfigPath: 'electron-builder.store.yml'
  });
  
  // Desktop-Bau-Befehl ausführen
  await runShellCommand(
    buildDesktopStoreCommand(overlayConfig.outputPath, desktopScripts),
    workspaceManifest.desktopWorkspace
  );
  
  // MSIX-Ausgabe sammeln
  const storeOutputs = await findStoreOutputs(pkgDirectory);
  const artifactPath = path.join(
    workspaceManifest.outputDirectory, 
    artifactFileName
  );
  await copySingleFile(primaryOutput, artifactPath);
}
```

Die electron-builder-Konfiguration muss Store-spezifische Metadaten enthalten，wie zum Beispiel Paket-Identität，Anzeigename des Herausgebers usw. Diese Informationen werden bei der Store-Einreichung verwendet.

Wenn die Konfiguration nicht stimmt，kann auch das Paket nicht erstellt werden. Das ist wohl klar.

### 5. Veröffentlichung auf GitHub

Nach Abschluss des Baus müssen die Artefakte auf GitHub Releases veröffentlicht werden.

```javascript
// scripts/publish-release.mjs
export async function publishRelease({
  planPath,
  artifactsDir,
  outputDir,
  forceDryRun
}) {
  // Veröffentlichungs-Artefakt-Manifest erstellen
  const publicationArtifacts = await buildPublicationArtifacts({
    plan,
    artifactsDir,
    outputDir
  });
  
  if (!dryRun) {
    // GitHub Release erstellen oder aktualisieren
    const releaseResult = await upsertReleaseNotes(
      plan.release.repository,
      plan.release.tag,
      token,
      { name, body }
    );
    
    // Assets hochladen
    for (const upload of publicationArtifacts.uploads) {
      await uploadReleaseAsset({
        release: releaseResult.release,
        filePath: upload.filePath,
        fileName: upload.fileName
      });
    }
  }
}
```

GitHub Release dient als Versionsnachverfolgung und Sicherung – selbst wenn die Store-Veröffentlichung fehlschlägt，werden die Bau-Artefakte erhalten.

Einen Rückweg zu haben，ist immer gut. Vielleicht muss man eines Tages zurückgehen.

### 6. Veröffentlichung im Microsoft Store

Der letzte Schritt ist die Veröffentlichung im Store mit dem Microsoft Store CLI.

```yaml
# .github/workflows/package-release.yml
publish_store:
  runs-on: windows-latest
  steps:
    - name: Configure Microsoft Store CLI
      uses: microsoft/microsoft-store-apppublisher@v1.2
    
    - name: Publish MSIX packages to Store
      shell: pwsh
      run: |
        msstore reconfigure --tenantId $env:AZURE_AD_TENANT_ID ...
        msstore publish "$($package.FullName)" -id $env:MICROSOFT_STORE_PRODUCT_ID
```

Dieser Schritt muss auf einem Windows-Runner ausgeführt werden，da das Microsoft Store CLI eine Windows-Umgebung benötigt.

Manche Dinge müssen einfach am richtigen Ort getan werden – in einer anderen Umgebung geht es nicht.

### Workflow-Trigger

Kann zeitgesteuert oder manuell ausgelöst werden:

```yaml
on:
  schedule:
    - cron: '0 */4 * * *'  # Alle 4 Stunden ausführen
  workflow_dispatch:
    inputs:
      desktop_version:
        description: 'Desktop-Version-Auswahl'
        required: false
      server_version:
        description: 'Server-Version-Auswahl'
        required: false
      force_rebuild:
        description: 'Erzwungener Neubau，auch wenn bereits vorhanden'
        type: boolean
      dry_run:
        description: 'Nur bauen，nicht veröffentlichen'
        type: boolean
```

Automatisch oder manuell – am Ende muss es laufen.

### Wichtige Hinweise

1. **Git-Tag-Anforderungen**: Das Desktop-Repository muss einen Tag enthalten，der der Veröffentlichungs-Version entspricht (z.B. v1.2.3)，sonst schlägt der Bau fehl

2. **Azure SAS URL**: Die Umgebungsvariablen `DESKTOP_AZURE_SAS_URL` und `SERVER_AZURE_SAS_URL` müssen konfiguriert werden，um auf den Index zuzugreifen

3. **Microsoft Store-Anmeldedaten**: Folgende Schlüssel müssen konfiguriert werden:
   - `AZURE_AD_APPLICATION_CLIENT_ID`
   - `AZURE_AD_APPLICATION_SECRET`
   - `AZURE_AD_TENANT_ID`
   - `SELLER_ID`
   - `MICROSOFT_STORE_PRODUCT_ID`

4. **Laufzeit-Validierung**: Die Server-Payload muss die erforderlichen Laufzeit-Dateien enthalten，sonst schlägt die Paketierung fehl

5. **Windows Runner**: AppX-Bau und Store-Veröffentlichung müssen auf einem Windows-Runner ausgeführt werden

6. **Duplikat-Erkennung**: Der geplante Lauf überprüft，ob der gleiche Release-Tag bereits existiert，um unnötige Bauten zu vermeiden

Diese Fallstricke haben wir wahrscheinlich schon einmal erlebt，also schreiben wir sie hier als Reminder auf. Schließlich möchte niemand denselben Fehler zweimal machen.

## Zusammenfassung

Die automatische Veröffentlichung im Microsoft Store sieht kompliziert aus，aber wenn man sie in unabhängige Schritte aufteilt，ist jeder Schritt nicht schwierig. Der Schlüssel liegt darin:
- git worktree für die Bau-Umgebung-Verwaltung verwenden
- Die Server-Payload in den richtigen Pfad injizieren
- electron-builder verwenden，um MSIX-Pakete zu generieren
- Das Store CLI für die endgültige Veröffentlichung verwenden

Diese Lösung läuft im HagiCode-Projekt seit einigen Monaten stabil und hat das Ziel der "vollständigen Automatisierung"基本上实现 – abgesehen von der ersten Konfiguration der Anmeldedaten und Einstellungen sind bei den nachfolgenden Version-Veröffentlichungen keine manuellen Eingriffe mehr erforderlich.

Eigentlich sind viele Dinge so: sie sehen schwierig aus，aber in der Praxis sind sie es dann doch nicht. Es braucht nur etwas Geduld und Ausprobieren.
