## GitHub Issues Integration von Grund auf: HagiCodes Frontend-Direktverbindung in der Praxis

> Dieser Artikel dokumentiert den vollständigen Prozess der Integration von GitHub Issues in die HagiCode-Plattform. Wir untersuchen, wie man mit einer „Frontend-Direktverbindung + minimales Backend"-Architektur eine sichere OAuth-Authentifizierung und effiziente Issues-Synchronisation erreicht und dabei das Backend schlank hält.

## Hintergrund: Warum GitHub integrieren?

Als KI-gestützte Entwicklungsplattform liegt der Kernwert von HagiCode darin, Ideen mit der Umsetzung zu verbinden. In der Praxis haben wir jedoch festgestellt, dass Benutzer nach Abschluss einer Proposal in HagiCode den Inhalt oft manuell in GitHub Issues kopieren müssen, um den Projektfortschritt zu verfolgen.

Daraus ergeben sich mehrere offensichtliche Probleme:

1.  **Zersplitterter Workflow**: Benutzer müssen zwischen zwei Systemen hin- und herwechseln. Die Erfahrung ist nicht flüssig, und bei Copy & Paste gehen leicht wichtige Informationen verloren.
2.  **Erschwerte Zusammenarbeit**: Andere Teammitglieder sind es gewohnt, Aufgaben auf GitHub zu prüfen, und können den Fortschritt der Proposals in HagiCode nicht direkt einsehen.
3.  **Doppelte Arbeit**: Jedes Mal, wenn eine Proposal aktualisiert wird, muss manuell das entsprechende Issue auf GitHub aktualisiert werden, was unnötige Wartungskosten verursacht.

Um dieses Problem zu lösen, haben wir beschlossen, die Funktion **GitHub Issues Integration** einzuführen, die HagiCode-Sessions mit GitHub-Repositories verbindet und eine „Ein-Klick-Synchronisation" ermöglicht.

## Technische Entscheidung: Frontend-Direktverbindung vs. Backend-Proxy

Bei der Gestaltung der Integrationslösung standen zwei Wege zur Auswahl: das traditionelle „Backend-Proxy-Modell" und das aggressivere „Frontend-Direktverbindungs-Modell".

### Lösungsvergleich

Im traditionellen **Backend-Proxy-Modell** müssen alle Anfragen vom Frontend zunächst über unser Backend geleitet werden, das dann die GitHub API aufruft. Dies zentralisiert die Logik, bringt dem Backend jedoch eine erhebliche Belastung:

1.  **Aufgeblähtes Backend**: Es müsste ein dedizierter GitHub API-Client-Wrapper geschrieben werden und außerdem die komplexe OAuth-Zustandsmaschine verwaltet werden.
2.  **Token-Risiko**: Der GitHub-Token des Benutzers müsste in der Backend-Datenbank gespeichert werden. Selbst mit Verschlüsselung vergrößert dies die Angriffsfläche.
3.  **Entwicklungskosten**: Es wären Datenbankmigrationen zum Speichern der Token und ein zusätzlicher Synchronisationsdienst zu pflegen.

Das **Frontend-Direktverbindungs-Modell** ist deutlich leichtgewichtiger. In diesem Ansatz nutzen wir das Backend nur für den sensibelsten Schritt — den „Schlüsselaustausch" (den OAuth-Callback). Nach Erhalt des Tokens wird er direkt im `localStorage` des Browsers gespeichert. Spätere Operationen wie das Erstellen von Issues und das Aktualisieren von Kommentaren werden direkt vom Frontend per HTTP an GitHub gesendet.

| Vergleichsdimension | Backend-Proxy-Modell | Frontend-Direktverbindungs-Modell |
| :--- | :--- | :--- |
| **Backend-Komplexität** | Erfordert einen vollständigen OAuth-Dienst und GitHub API-Client | Benötigt nur einen OAuth-Callback-Endpunkt |
| **Token-Verwaltung** | Muss verschlüsselt in der Datenbank gespeichert werden, mit Leckage-Risiko | Im Browser gespeichert, nur für den Benutzer sichtbar |
| **Implementierungskosten** | Erfordert Datenbankmigrationen und Entwicklung mehrerer Dienste | Hauptsächlich Frontend-Arbeit |
| **Benutzererfahrung** | Zentralisierte Logik, aber Serverlatenz kann etwas höher sein | Extrem schnelle Antwort durch direkte GitHub-Interaktion |

Da wir eine schnelle Integration mit minimalen Backend-Änderungen anstrebten, **haben wir uns letztendlich für das „Frontend-Direktverbindungs-Modell" entschieden**. Das ist vergleichbar mit der Ausstellung eines „temporären Passes" an den Browser. Hat er den Pass einmal erhalten, kann der Browser eigenständig auf GitHub erledigen, was zu tun ist, ohne jedes Mal den Backend-Administrator um Erlaubnis bitten zu müssen.

## Kerndesign: Datenfluss und Sicherheit

Nach der Festlegung der Architektur mussten wir den konkreten Datenfluss entwerfen. Der Kern des Synchronisationsprozesses liegt darin, wie das Token sicher beschafft und effizient genutzt wird.

### Gesamtarchitekturdiagramm

Das gesamte System lässt sich auf drei Rollen abstrahieren: den Browser (Frontend), das HagiCode-Backend und GitHub.

```text
+--------------+        +--------------+        +--------------+
| Frontend     |        |   Backend    |        |   GitHub     |
| React        |        |   ASP.NET    |        |   REST API   |
|              |        |              |        |              |
|  +--------+  |        |              |        |              |
|  | OAuth  |--+--------> /callback    |        |              |
|  | Flow   |  |        |              |        |              |
|  +--------+  |        |              |        |              |
|              |        |              |        |              |
|  +--------+  |        |  +--------+  |        |  +--------+  |
|  | GitHub |  +------------> Session |  +----------> Issues |  |
|  | API    |  |        |  |Metadata|  |        |  |        |  |
|  | Direct |  |        |  +--------+  |        |  +--------+  |
|  +--------+  |        |              |        |              |
+--------------+        +--------------+        +--------------+
```

**Der entscheidende Punkt**: Nur ein kleiner Schritt bei OAuth — der Austausch des Codes gegen ein Token — muss über das Backend geleitet werden. Danach erledigt die Schwerstarbeit (das Erstellen von Issues) das Frontend direkt mit GitHub.

### Synchronisationsdatenfluss im Detail

Wenn der Benutzer in der HagiCode-Benutzeroberfläche auf die Schaltfläche „Sync to GitHub" klickt, findet eine Reihe komplexer Aktionen statt:

```text
Benutzer klickt „Sync to GitHub"
         │
         ▼
1. Frontend prüft localStorage auf den GitHub-Token
         │
         ▼
2. Issue-Inhalt formatieren (Proposal in Markdown umwandeln)
         │
         ▼
3. Frontend ruft direkt die GitHub API auf, um das Issue zu erstellen/aktualisieren
         │
         ▼
4. HagiCode-Backend-API aufrufen, um Session.metadata zu aktualisieren (Issue-URL usw. speichern)
         │
         ▼
5. Backend broadcastet das SessionUpdated-Ereignis über SignalR
         │
         ▼
6. Frontend empfängt das Ereignis und aktualisiert die UI auf „Synchronisiert"-Status
```

### Sicherheitsdesign

Sicherheit ist bei der Integration von Drittanbieterdiensten stets oberste Priorität. Wir haben folgende Überlegungen angestellt:

1.  **CSRF-Schutz**: Bei der OAuth-Weiterleitung wird ein zufälliger `state`-Parameter generiert und im `sessionStorage` gespeichert. Im Callback wird der state streng validiert, um gefälschte Anfragen zu verhindern.
2.  **Isolierte Token-Speicherung**: Der Token wird ausschließlich im `localStorage` des Browsers gespeichert. Dank der Same-Origin Policy können nur HagiCode-Skripte ihn lesen, wodurch das Risiko ausgeschlossen wird, dass eine serverseitige Datenbank-Leckage die Benutzer betrifft.
3.  **Fehlerbehandlung**: Wir haben spezielle Behandlungslogiken für häufige GitHub API-Fehler entworfen (z. B. 401 Token abgelaufen, 422 Validierung fehlgeschlagen, 429 Rate-Limit), damit Benutzer freundliche Rückmeldungen erhalten.

## In der Praxis: Implementierungsdetails im Code

Theorie reicht nicht weit. Schauen wir uns an, wie der Code tatsächlich funktioniert.

### 1. Minimale Backend-Änderungen

Das Backend muss nur zwei Dinge erledigen: Synchronisationsinformationen speichern und den OAuth-Callback verarbeiten.

**Datenbankänderungen**

### 2. Frontend OAuth-Flow

Dies ist der Einstiegspunkt der Verbindung. Wir verwenden den standardmäßigen Authorization Code Flow.

```typescript
// src/HagiCode.Client/src/services/githubOAuth.ts

// Autorisierungs-URL generieren und weiterleiten
export async function generateAuthUrl(): Promise<string> {
  const state = generateRandomString(); // Zufällige Zeichenfolge für CSRF-Schutz generieren
  sessionStorage.setItem('hagicode_github_state', state);

  const params = new URLSearchParams({
    client_id: clientId,
    redirect_uri: window.location.origin + '/settings?tab=github&oauth=callback',
    scope: ['repo', 'public_repo'].join(' '),
    state: state,
  });

  return `https://github.com/login/oauth/authorize?${params.toString()}`;
}

// Code-zu-Token-Austausch auf der Callback-Seite verarbeiten
export async function exchangeCodeForToken(code: string, state: string): Promise<GitHubToken> {
  // 1. State validieren, um CSRF zu verhindern
  const savedState = sessionStorage.getItem('hagicode_github_state');
  if (state !== savedState) throw new Error('Invalid state parameter');

  // 2. Backend-API aufrufen, um den Token auszutauschen
  // Hinweis: Dies muss über das Backend erfolgen, da ClientSecret nicht im Frontend offengelegt werden darf
  const response = await fetch('/api/GitHubOAuth/callback', {
    method: 'POST',
    headers: { 'Content-Type': 'application/json' },
    body: JSON.stringify({ code, state, redirectUri: window.location.origin + '/settings?tab=github&oauth=callback' }),
  });

  if (!response.ok) throw new Error('Failed to exchange token');

  const token = await response.json();

  // 3. Im LocalStorage speichern
  saveToken(token);
  return token;
}
```

### 3. GitHub API Client-Wrapper

Sobald wir den Token haben, benötigen wir ein robustes Werkzeug zum Aufrufen der GitHub API.

```typescript
// src/HagiCode.Client/src/services/githubApiClient.ts

const GITHUB_API_BASE = 'https://api.github.com';

// Kernanfrage-Wrapper
async function githubApi<T>(endpoint: string, options: RequestInit = {}): Promise<T> {
  const token = localStorage.getItem('gh_token');
  if (!token) throw new Error('Nicht mit GitHub verbunden');

  const response = await fetch(`${GITHUB_API_BASE}${endpoint}`, {
    ...options,
    headers: {
      ...options.headers,
      Authorization: `Bearer ${token}`,
      Accept: 'application/vnd.github.v3+json', // API-Version festlegen
    },
  });

  // Fehlerbehandlungslogik
  if (!response.ok) {
    if (response.status === 401) throw new Error('GitHub-Token abgelaufen, bitte erneut verbinden');
    if (response.status === 403) throw new Error('Kein Zugriff auf dieses Repository oder Rate-Limit überschritten');
    if (response.status === 422) throw new Error('Issue-Validierung fehlgeschlagen, Titel möglicherweise doppelt');
    throw new Error(`GitHub API Error: ${response.statusText}`);
  }

  return response.json();
}

// Issue erstellen
export async function createIssue(owner: string, repo: string, data: { title: string, body: string, labels: string[] }) {
  return githubApi(`/repos/${owner}/${repo}/issues`, {
    method: 'POST',
    body: JSON.stringify(data),
  });
}
```

### 4. Inhaltsformatierung und Synchronisation

Der letzte Schritt besteht darin, die HagiCode-Sessionsdaten in das Format eines GitHub-Issues umzuwandeln. Das ist gewissermaßen Übersetzungsarbeit.

```typescript
// Session-Objekt in einen Markdown-String umwandeln
function formatIssueForSession(session: Session): string {
  let content = `# ${session.title}\n\n`;
  content += `**> HagiCode Session:** #${session.code}\n`;
  content += `**> Status:** ${session.status}\n\n`;
  content += `## Description\n\n${session.description || 'No description provided.'}\n\n`;

  // Bei Proposal-Sessions zusätzliche Felder hinzufügen
  if (session.type === 'proposal') {
    content += `## Chief Complaint\n\n${session.chiefComplaint || ''}\n\n`;
    // Deep Link hinzufügen, um von GitHub zurück zu HagiCode zu gelangen
    content += `---\n\n**[In HagiCode ansehen](hagicode://sessions/${session.id})**\n`;
  }

  return content;
}

// Hauptlogik beim Klicken der Synchronisationsschaltfläche
const handleSync = async (session: Session) => {
  try {
    const repoInfo = parseRepositoryFromUrl(session.repoUrl); // Repository-URL parsen
    if (!repoInfo) throw new Error('Invalid repository URL');

    toast.loading('Synchronisiere mit GitHub...');

    // 1. Inhalt formatieren
    const issueBody = formatIssueForSession(session);

    // 2. API aufrufen
    const issue = await githubApiClient.createIssue(repoInfo.owner, repoInfo.repo, {
      title: `[HagiCode] ${session.title}`,
      body: issueBody,
      labels: ['hagicode', 'proposal', `status:${session.status}`],
    });

    // 3. Session-Metadaten aktualisieren (Issue-Link speichern)
    await SessionsService.patchApiSessionsSessionId(session.id, {
      metadata: {
        githubIssue: {
          owner: repoInfo.owner,
          repo: repoInfo.repo,
          issueNumber: issue.number,
          issueUrl: issue.html_url,
          syncedAt: new Date().toISOString(),
        }
      }
    });

    toast.success('Erfolgreich synchronisiert!');
  } catch (err) {
    console.error(err);
    toast.error('Synchronisation fehlgeschlagen, bitte Token oder Netzwerk prüfen');
  }
};
```

## Zusammenfassung und Ausblick

Mit diesem „Frontend-Direktverbindungs"-Ansatz haben wir eine nahtlose GitHub Issues Integration mit dem geringstmöglichen Backend-Aufwand erreicht.

### Gewonnene Erkenntnisse
1.  **Hohe Entwicklungseffizienz**: Backend-Änderungen sind minimal — hauptsächlich ein zusätzliches Datenbankfeld und ein einfacher OAuth-Callback-Endpunkt. Der Großteil der Logik wird im Frontend abgearbeitet.
2.  **Starke Sicherheit**: Der Token durchläuft nicht die Serverdatenbank, was das Leckage-Risiko verringert.
3.  **Hervorragende Benutzererfahrung**: Anfragen werden direkt vom Frontend gestellt, die Antwortgeschwindigkeit ist hoch und es bedarf keines Backend-Umwegs.

### Zu beachtende Punkte
Bei der tatsächlichen Bereitstellung gibt es einige Fallstricke, die man beachten sollte:
- **OAuth App-Einstellungen**: Denken Sie daran, die korrekte `Authorization callback URL` in den GitHub OAuth App-Einstellungen einzutragen (in der Regel `http://localhost:3000/settings?tab=github&oauth=callback`).
- **Rate-Limits**: Die GitHub API schränkt nicht authentifizierte Anfragen stark ein, aber mit einem Token ist das Kontingent in der Regel ausreichend (5000 Anfragen/Stunde).
- **URL-Parsing**: Benutzer geben alle möglichen Repository-URLs ein — stellen Sie sicher, dass Ihr Regex `.git`-Suffixe, SSH-Formate und ähnliche Fälle abdeckt.

### Zukünftige Erweiterungen
Die aktuelle Funktion ist noch eine Einweg-Synchronisation (HagiCode -> GitHub). In Zukunft planen wir, eine bidirektionale Synchronisation über GitHub Webhooks zu implementieren. Wenn beispielsweise ein Issue auf GitHub geschlossen wird, könnte der Session-Status auf HagiCode-Seite automatisch aktualisiert werden. Dafür müssten wir einen Webhook-Endpunkt auf dem Backend bereitstellen — das wird ein interessanter nächster Schritt.

Wir hoffen, dieser Artikel gibt Ihnen etwas Inspiration für Ihre eigene Integration von Drittanbieterdiensten. Bei Fragen können Sie gerne ein Issue auf [HagiCode GitHub](https://github.com/HagiCode-org/site) eröffnen, um die Diskussion zu beginnen.
