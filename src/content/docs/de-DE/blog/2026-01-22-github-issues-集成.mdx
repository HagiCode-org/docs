## GitHub Issues Integration von Grund auf neu aufbauen: HagiCodes Frontend-Direktverbindung Praxis

> Dieser Artikel dokumentiert den vollständigen Prozess der Integration von GitHub Issues in die HagiCode-Plattform. Wir untersuchen, wie wir durch eine „Frontend-Direktverbindung + Backend-Minimierung“-Architektur sichere OAuth-Authentifizierung und effiziente Issues-Synchronisation bei gleichzeitigem schlankem Backend realisieren.

## Hintergrund: Warum GitHub integrieren?

HagiCode ist eine KI-gestützte Entwicklungsplattform, deren Kernwert darin besteht, Ideen mit ihrer Umsetzung zu verbinden. In der praktischen Nutzung haben wir jedoch festgestellt, dass Benutzer nach Abschluss eines Proposals (Vorschlags) in HagiCode den Inhalt oft manuell in GitHub Issues kopieren müssen, um Projekte zu verfolgen.

Dies bringt einige offensichtliche Schmerzpunkte mit sich:

1.  **Unterbrochener Workflow**: Benutzer müssen zwischen zwei Systemen hin- und herwechseln, was nicht nur zu einer reibungslosen Erfahrung führt, sondern auch dazu führt, dass wichtige Informationen beim Kopieren und Einfügen verloren gehen.
2.  **Erschwerte Zusammenarbeit**: Andere Teammitglieder gewöhnt sind, Aufgaben auf GitHub zu überprüfen, und können den Fortschritt der Proposals in HagiCode nicht direkt sehen.
3.  **Doppelte Arbeit**: Jedes Mal, wenn ein Proposal aktualisiert wird, muss das entsprechende Issue manuell auf GitHub aktualisiert werden, was unnötige Wartungskosten erhöht.

Um dieses Problem zu lösen, haben wir beschlossen, die **GitHub Issues Integration** Funktion einzuführen, um die Verbindung zwischen HagiCode-Sitzungen und GitHub-Repositories zu verknüpfen und eine „Ein-Klick-Synchronisation“ zu ermöglichen.

## Technologiewahl: Frontend-Direktverbindung vs. Backend-Proxy

Bei der Gestaltung der Integrationslösung standen zwei Wege vor uns: die traditionelle „Backend-Proxy-Methode“ und die radikalere „Frontend-Direktverbindungsmethode“.

### Vergleich der Lösungen

Im traditionellen **Backend-Proxy-Modell** müssen alle Frontend-Anfragen zuerst über unser Backend geleitet werden, das dann die GitHub API aufruft. Dies ist zwar logisch zentralisiert, bringt aber eine erhebliche Belastung für das Backend mit sich:

1.  **Aufgeblähtes Backend**: Es müssen spezielle GitHub API Client-Kapselungen geschrieben werden und die komplexe Zustandsmaschine von OAuth verarbeitet werden.
2.  **Token-Risiko**: Das GitHub Token des Benutzers muss in der Backend-Datenbank gespeichert werden. Obwohl es verschlüsselt werden kann, erhöht dies dennoch das Sicherheitsrisiko.
3.  **Entwicklungskosten**: Datenbankmigrationen sind erforderlich, um Tokens zu speichern, und ein zusätzlicher Synchronisationsdienst muss gewartet werden.

Das **Frontend-Direktverbindungsmodell** ist viel leichtgewichtiger. In dieser Lösung nutzen wir das Backend nur für den empfindlichsten „Schlüsselaustausch“-Schritt (OAuth Callback). Nach Erhalt des Tokens wird es direkt im localStorage des Browsers gespeichert. Nachfolgende Operationen wie das Erstellen von Issues oder das Aktualisieren von Kommentaren werden vom Frontend direkt als HTTP-Anfrage an GitHub gesendet.

| Vergleichsdimension | Backend-Proxy-Modell | Frontend-Direktverbindungsmodell |
| :--- | :--- | :--- |
| **Backend-Komplexität** | Erfordert vollständigen OAuth-Service und GitHub API Client | Nur ein OAuth Callback-Endpunkt erforderlich |
| **Token-Verwaltung** | Muss verschlüsselt in der Datenbank gespeichert werden, Risiko von Lecks | Im Browser gespeichert, nur für den Benutzer selbst sichtbar |
| **Implementierungskosten** | Erfordert Datenbankmigration, Multi-Service-Entwicklung | Hauptsächlich Frontend-Arbeitsaufwand |
| **Benutzererfahrung** | Logisch einheitlich, aber Server-Latenz kann etwas höher sein | Extrem schnelle Reaktion, direkte Interaktion mit GitHub |

Da wir eine schnelle Integration und minimale Backend-Änderungen anstreben, **haben wir uns letztendlich für das „Frontend-Direktverbindungsmodell“ entschieden**. Dies ist vergleichbar mit der Ausstellung eines „temporären Ausweises“ an den Browser – nach Erhalt des Ausweises kann der Browser selbstständig Geschäfte auf GitHub erledigen, ohne jedes Mal die Genehmigung des Backend-Administrators einzuholen.

## Kern-Design: Datenfluss und Sicherheit

Nach Festlegung der Architektur müssen wir den konkreten Datenfluss gestalten. Der Kern des gesamten Synchronisationsprozesses liegt darin, wie man das Token sicher erhält und effizient nutzt.

### Architekturdiagramm

Das gesamte System kann auf drei Rollen abstrahiert werden: Browser (Frontend), HagiCode Backend, GitHub.

```text
+--------------+        +--------------+        +--------------+
|  Frontend    |        |    Backend   |        |    GitHub    |
|   React      |        |   ASP.NET    |        |    REST API  |
|  +--------+  |        |              |        |              |
|  |  OAuth |--+--------> /callback    |        |              |
|  |  Flow  |  |        |              |        |              |
|  +--------+  |        |              |        |              |
|              |        |              |        |              |
|  +--------+  |        |  +--------+  |        |  +--------+  |
|  |GitHub  |  +------------>Session |  +----------> Issues |  |
|  |API     |  |        |  |Metadata|  |        |  |        |  |
|  |Direct  |  |        |  +--------+  |        |  +--------+  |
|  +--------+  |        |              |        |              |
+--------------+        +--------------+        +--------------+
```

**Der Schlüssel liegt darin**: Nur ein kleiner Schritt von OAuth (Code erhalten gegen Token tauschen) muss über das Backend laufen, die grobe Arbeit danach (Issue erstellen) wird direkt vom Frontend mit GitHub abgewickelt.

### Detaillierte Erklärung des Synchronisationsdatenflusses

Wenn der Benutzer auf die Schaltfläche „Sync to GitHub“ in der HagiCode-Benutzeroberfläche klickt, findet eine Reihe komplexer Aktionen statt:

```text
Benutzer klickt auf „Sync to GitHub“
         │
         ▼
1. Frontend prüft localStorage für GitHub Token
         │
         ▼
2. Issue-Inhalt formatieren (Proposal zu Markdown konvertieren)
         │
         ▼
3. Frontend ruft direkt GitHub API auf, um Issue zu erstellen/aktualisieren
         │
         ▼
4. Aufruf der HagiCode Backend API zum Aktualisieren von Session.metadata (Speichern von Issue URL etc.)
         │
         ▼
5. Backend sendet SessionUpdated Event über SignalR
         │
         ▼
6. Frontend empfängt Event, aktualisiert UI mit „synchronisiert“ Status
```

### Sicherheits-Design

Sicherheitsfragen sind bei der Integration von Drittanbieterdiensten von größter Bedeutung. Wir haben folgende Überlegungen angestellt:

1.  **CSRF-Angriffe verhindern**: Bei OAuth-Weiterleitungen wird ein zufälliger `state`-Parameter generiert und im sessionStorage gespeichert. Beim Callback wird der state strikt validiert, um zu verhindern, dass Anfragen gefälscht werden.
2.  **Token-Speicher-Isolierung**: Tokens werden nur im `localStorage` des Browsers gespeichert. Durch die Same-Origin Policy können nur HagiCode-Skripte sie lesen, wodurch vermieden wird, dass Server-seitige Datenbank-Lecks die Benutzer betreffen.
3.  **Fehlergrenzen**: Für häufige GitHub API Fehler (wie 401 Token abgelaufen, 422 Validierung fehlgeschlagen, 429 Rate Limiting) wurde eine spezielle Fehlerbehandlungslogik entwickelt, um dem Benutzer benutzerfreundliche Hinweise zu geben.

## Praxis: Code-Implementierungsdetails

Theorie ist gut, aber Praxis ist besser. Lassen Sie uns sehen, wie der konkrete Code implementiert ist.

### 1. Minimale Backend-Änderungen

Das Backend muss nur zwei Dinge tun: Synchronisationsinformationen speichern und OAuth Callbacks verarbeiten.

### 2. Frontend OAuth Prozess

Dies ist der Einstiegspunkt für die Verbindung. Wir verwenden den Standard Authorization Code Flow.

```typescript
// src/HagiCode.Client/src/services/githubOAuth.ts

// Autorisierungs-URL generieren und weiterleiten
export async function generateAuthUrl(): Promise<string> {
  const state = generateRandomString(); // Zufällige Zeichenfolge für CSRF-Schutz
  sessionStorage.setItem('hagicode_github_state', state);
  
  const params = new URLSearchParams({
    client_id: clientId,
    redirect_uri: window.location.origin + '/settings?tab=github&oauth=callback',
    scope: ['repo', 'public_repo'].join(' '),
    state: state,
  });
  
  return `https://github.com/login/oauth/authorize?${params.toString()}`;
}

// Code gegen Token auf der Callback-Seite tauschen
export async function exchangeCodeForToken(code: string, state: string): Promise<GitHubToken> {
  // 1. State validieren, um CSRF zu verhindern
  const savedState = sessionStorage.getItem('hagicode_github_state');
  if (state !== savedState) throw new Error('Invalid state parameter');

  // 2. Backend API für Token-Austausch aufrufen
  // Hinweis: Dies muss über das Backend laufen, da ClientSecret erforderlich ist und nicht im Frontend exponiert werden darf
  const response = await fetch('/api/GitHubOAuth/callback', {
    method: 'POST',
    headers: { 'Content-Type': 'application/json' },
    body: JSON.stringify({ code, state, redirectUri: window.location.origin + '/settings?tab=github&oauth=callback' }),
  });

  if (!response.ok) throw new Error('Failed to exchange token');
  
  const token = await response.json();
  
  // 3. In LocalStorage speichern
  saveToken(token);
  return token;
}
```

### 3. GitHub API Client Kapselung

Nach Erhalt des Tokens benötigen wir ein leistungsstarkes Werkzeug, um die GitHub API aufzurufen.

```typescript
// src/HagiCode.Client/src/services/githubApiClient.ts

const GITHUB_API_BASE = 'https://api.github.com';

// Kern-Anfragekapselung
async function githubApi<T>(endpoint: string, options: RequestInit = {}): Promise<T> {
  const token = localStorage.getItem('gh_token');
  if (!token) throw new Error('Not connected to GitHub');
  
  const response = await fetch(`${GITHUB_API_BASE}${endpoint}`, {
    ...options,
    headers: {
      ...options.headers,
      Authorization: `Bearer ${token}`,
      Accept: 'application/vnd.github.v3+json', // API-Version angeben
    },
  });
  
  // Fehlerbehandlungslogik
  if (!response.ok) {
    if (response.status === 401) throw new Error('GitHub Token abgelaufen, bitte erneut verbinden');
    if (response.status === 403) throw new Error('Kein Zugriff auf das Repository oder Rate-Limit überschritten');
    if (response.status === 422) throw new Error('Issue-Validierung fehlgeschlagen, möglicherweise doppelter Titel');
    throw new Error(`GitHub API Error: ${response.statusText}`);
  }
  
  return response.json();
}

// Issue erstellen
export async function createIssue(owner: string, repo: string, data: { title: string, body: string, labels: string[] }) {
  return githubApi(`/repos/${owner}/${repo}/issues`, {
    method: 'POST',
    body: JSON.stringify(data),
  });
}
```

### 4. Inhaltsformatierung und Synchronisation

Der letzte Schritt besteht darin, die HagiCode-Sitzungsdaten in das GitHub Issue-Format zu konvertieren. Dies ähnelt einer „Übersetzungsarbeit“.

```typescript
// Session-Objekt in Markdown-Zeichenfolge konvertieren
function formatIssueForSession(session: Session): string {
  let content = `# ${session.title}\n\n`;
  content += `**> HagiCode Session:** #${session.code}\n`;
  content += `**> Status:** ${session.status}\n\n`;
  content += `## Description\n\n${session.description || 'No description provided.'}\n\n`;
  
  // Wenn es vom Typ Proposal ist, zusätzliche Felder hinzufügen
  if (session.type === 'proposal') {
    content += `## Chief Complaint\n\n${session.chiefComplaint || ''}\n\n`;
    // Deep-Link hinzufügen, um von GitHub zurück zu HagiCode zu gelangen
    content += `---\n\n**[View in HagiCode](hagicode://sessions/${session.id})**\n`;
  }
  
  return content;
}

// Hauptlogik beim Klicken auf die Synchronisationsschaltfläche
const handleSync = async (session: Session) => {
  try {
    const repoInfo = parseRepositoryFromUrl(session.repoUrl); // Repository URL parsen
    if (!repoInfo) throw new Error('Invalid repository URL');

    toast.loading('Synchronisiere mit GitHub...');
    
    // 1. Inhalt formatieren
    const issueBody = formatIssueForSession(session);
    
    // 2. API aufrufen
    const issue = await githubApiClient.createIssue(repoInfo.owner, repoInfo.repo, {
      title: `[HagiCode] ${session.title}`,
      body: issueBody,
      labels: ['hagicode', 'proposal', `status:${session.status}`],
    });
    
    // 3. Session Metadata aktualisieren (Issue Link speichern)
    await SessionsService.patchApiSessionsSessionId(session.id, {
      metadata: {
        githubIssue: {
          owner: repoInfo.owner,
          repo: repoInfo.repo,
          issueNumber: issue.number,
          issueUrl: issue.html_url,
          syncedAt: new Date().toISOString(),
        }
      }
    });

    toast.success('Synchronisation erfolgreich!');
  } catch (err) {
    console.error(err);
    toast.error('Synchronisation fehlgeschlagen, bitte Token oder Netzwerk überprüfen');
  }
};
```

## Zusammenfassung und Ausblick

Mit diesem „Frontend-Direktverbindung“-Ansatz haben wir eine nahtlose Integration von GitHub Issues mit minimalem Backend-Code realisiert.

### Erkenntnisse
1.  **Hohe Entwicklungseffizienz**: Backend-Änderungen sind minimal, hauptsächlich eine Spalte in der Datenbank und ein einfacher OAuth Callback-Endpunkt, die meisten Logik wird im Frontend erledigt.
2.  **Gute Sicherheit**: Tokens gehen nicht durch die Server-Datenbank, was das Risiko von Lecks verringert.
3.  **Gute Benutzererfahrung**: Direkte Anfragen vom Frontend, schnelle Reaktionszeit, keine Umleitung über das Backend erforderlich.

### Zu beachten
Bei der tatsächlichen Bereitstellung gibt es einige Stolpersteine, die Sie beachten sollten:
- **OAuth App Einstellungen**: Denken Sie daran, die korrekte `Authorization callback URL` in den GitHub OAuth App Einstellungen einzugeben (normalerweise `http://localhost:3000/settings?tab=github&oauth=callback`).
- **Rate Limiting**: Die GitHub API hat strenge Limits für nicht authentifizierte Anfragen, aber mit Token ist es normalerweise ausreichend (5000 Anfragen/Stunde).
- **URL Parsing**: Die vom Benutzer eingegebene Repo URL ist vielfältig, denken Sie daran, dass der reguläre Ausdruck `.git` Endungen, SSH-Formate etc. abdecken sollte.

### Zukünftige Erweiterungen
Die aktuelle Funktion ist noch eine Einweg-Synchronisation (HagiCode -> GitHub). In Zukunft planen wir, durch GitHub Webhooks eine bidirektionale Synchronisation zu实现, zum Beispiel das Schließen eines Issues in GitHub würde den Sitzungsstatus in HagiCode automatisch aktualisieren. Dies erfordert einen Webhook-Empfangs-Endpunkt im Backend, was die nächste interessante Aufgabe ist.
