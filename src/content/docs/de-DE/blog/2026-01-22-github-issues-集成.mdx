## GitHub Issues-Integration von Grund auf: HagiCodes Frontend-Direktverbindung in der Praxis

> Dieser Artikel dokumentiert den gesamten Prozess der Integration von GitHub Issues in die HagiCode-Plattform. Wir erläutern, wie durch eine „Frontend-Direktverbindung + minimales Backend"-Architektur bei gleichzeitigem Backend-Leichtbau eine sichere OAuth-Authentifizierung und eine effiziente Issues-Synchronisierung erreicht werden.

## Hintergrund: Warum GitHub integrieren?

HagiCode ist als KI-gestützte Entwicklungsplattform darauf ausgelegt, Ideen mit deren Umsetzung zu verbinden. In der Praxis stellten wir jedoch fest, dass Benutzer nach dem Erstellen eines Proposal (Vorschlags) in HagiCode den Inhalt häufig manuell in GitHub Issues kopieren mussten, um die Projektaufgaben zu verfolgen.

Dies führte zu mehreren offensichtlichen Problemen:

1.  **Gefragmentierter Workflow**: Benutzer mussten zwischen zwei Systemen hin- und herwechseln, was nicht nur die Benutzererfahrung beeinträchtigte, sondern auch zum Verlust wichtiger Informationen beim Kopieren und Einfügen führte.
2.  **Erschwerte Zusammenarbeit**: Andere Teammitglieder, die Aufgaben auf GitHub verfolgen, konnten den Fortschritt der Proposal in HagiCode nicht direkt einsehen.
3.  **Doppelte Arbeit**: Bei jeder Aktualisierung eines Proposal musste das entsprechende Issue manuell auf GitHub aktualisiert werden, was unnötigen Wartungsaufwand bedeutete.

Um dieses Problem zu lösen, entschieden wir uns für die Einführung der **GitHub Issues Integration**, um HagiCode-Sitzungen mit GitHub-Repositories zu verknüpfen und eine „Ein-Klick-Synchronisierung" zu realisieren.

## Technologische Entscheidung: Frontend-Direktverbindung vs. Backend-Proxy

Bei der Konzeption der Integrationslösung standen zwei Ansätze zur Wahl: das traditionelle „Backend-Proxy-Modell" und das radikalere „Frontend-Direktverbindungsmodell".

### Vergleich der Ansätze

Im traditionellen **Backend-Proxy-Modell** werden alle Frontend-Anfragen zunächst über unser Backend geleitet, welches dann die GitHub API aufruft. Obwohl die Logik zentralisiert ist, bedeutet dies eine erhebliche Belastung für das Backend:

1.  **Aufgeblähtes Backend**: Es müssen spezifische GitHub-API-Client-Wrapper entwickelt und komplexe OAuth-Zustandsautomaten verarbeitet werden.
2.  **Token-Risiko**: Der GitHub Token des Benutzers muss in der Backend-Datenbank gespeichert werden. Auch wenn er verschlüsselt werden kann, vergrößert dies die Angriffsfläche.
3.  **Entwicklungskosten**: Datenbankmigrationen zur Token-Speicherung und die Wartung eines zusätzlichen Synchronisierungsdienstes sind erforderlich.

Das **Frontend-Direktverbindungsmodell** ist wesentlich leichtgewichtiger. In diesem Ansatz wird das Backend nur für den sensibelsten Teil – den „Schlüsselaustausch" (OAuth Callback) – genutzt. Nach Erhalt des Tokens wird dieser direkt im localStorage des Browsers gespeichert. Alle nachfolgenden Vorgänge wie das Erstellen von Issues oder das Aktualisieren von Kommentaren werden vom Frontend über HTTP-Anfragen direkt an GitHub gesendet.

| Vergleichskriterium | Backend-Proxy-Modell | Frontend-Direktverbindungsmodell |
| :--- | :--- | :--- |
| **Backend-Komplexität** | Vollständiger OAuth-Dienst und GitHub-API-Client erforderlich | Nur ein OAuth-Callback-Endpunkt erforderlich |
| **Token-Verwaltung** | Verschlüsselte Speicherung in der Datenbank, Risiko von Datenlecks | Speicherung im Browser, nur für den Benutzer sichtbar |
| **Implementierungskosten** | Datenbankmigrationen, Entwicklung mehrerer Dienste | Hauptsächlich Frontend-Aufwand |
| **Benutzererfahrung** | Einheitliche Logik, aber mögliche Serververzögerungen | Sehr schnelle Reaktion, direkte Interaktion mit GitHub |

Da wir eine schnelle Integration bei minimalen Backend-Änderungen anstrebten, **haben wir uns schließlich für das „Frontend-Direktverbindungsmodell" entschieden**. Das ist vergleichbar mit der Ausstellung einer „temporären Zugangskarte" an den Browser – nach Erhalt der Karte kann der Browser selbstständig mit GitHub kommunizieren, ohne bei jedem Vorgang die Genehmigung des Backend-Administrators einholen zu müssen.

## Kernentwurf: Datenfluss und Sicherheit

Nach Festlegung der Architektur musste der konkrete Datenfluss entworfen werden. Der Kern des gesamten Synchronisierungsprozesses liegt in der sicheren Beschaffung und effizienten Nutzung des Tokens.

### Gesamtarchitekturdiagramm

Das gesamte System lässt sich auf drei Akteure abstrahieren: Browser (Frontend), HagiCode Backend und GitHub.

```text
+--------------+        +--------------+        +--------------+
|  前端 React  |        |    后端      |        |    GitHub    |
|              |        |   ASP.NET    |        |    REST API  |
|  +--------+  |        |              |        |              |
|  |  OAuth |--+--------> /callback    |        |              |
|  |  流程  |  |        |              |        |              |
|  +--------+  |        |              |        |              |
|              |        |              |        |              |
|  +--------+  |        |  +--------+  |        |  +--------+  |
|  |GitHub  |  +------------>Session |  +----------> Issues |  |
|  |API     |  |        |  |Metadata|  |        |  |        |  |
|  |直连    |  |        |  +--------+  |        |  +--------+  |
|  +--------+  |        |              |        |              |
+--------------+        +--------------+        +--------------+
```

**Der entscheidende Punkt**: Nur ein kleiner Teil des OAuth-Prozesses (Code gegen Token eintauschen) erfordert die Beteiligung des Backends. Die eigentliche Arbeit (Erstellung von Issues) wird danach direkt vom Frontend mit GitHub abgewickelt.

### Detaillierte Erklärung des Synchronisierungsdatenflusses

Wenn ein Benutzer in der HagiCode-Benutzeroberfläche auf die Schaltfläche „Sync to GitHub" klickt, erfolgt eine Reihe komplexer Aktionen:

```text
用户点击 "Sync to GitHub"
         │
         ▼
1. 前端检查 localStorage 获取 GitHub Token
         │
         ▼
2. 格式化 Issue 内容（将 Proposal 转换为 Markdown）
         │
         ▼
3. 前端直接调用 GitHub API 创建/更新 Issue
         │
         ▼
4. 调用 HagiCode 后端 API 更新 Session.metadata (存储 Issue URL 等信息)
         │
         ▼
5. 后端通过 SignalR 广播 SessionUpdated 事件
         │
         ▼
6. 前端接收事件，更新 UI 显示"已同步"状态
```

### Sicherheitsdesign

Sicherheitsfragen sind bei der Integration von Drittanbieterdiensten von größter Bedeutung. Wir haben folgende Aspekte berücksichtigt:

1.  **CSRF-Schutz**: Bei der OAuth-Weiterleitung wird ein zufälliger `state`-Parameter generiert und im sessionStorage gespeichert. Beim Callback wird der state streng validiert, um Anfragefälschungen zu verhindern.
2.  **Isolierte Token-Speicherung**: Der Token wird ausschließlich im `localStorage` des Browsers gespeichert. Durch die Same-Origin Policy können nur HagiCode-Skripte darauf zugreifen, wodurch ein serverseitiges Datenbankleck die Benutzer nicht beeinträchtigt.
3.  **Fehlergrenzen**: Für häufige GitHub-API-Fehler (z. B. 401 Token abgelaufen, 422 Validierung fehlgeschlagen, 429 Rate-Limit) wurde eine spezialisierte Fehlerbehandlungslogik entwickelt, um dem Benutzer benutzerfreundliche Hinweise zu geben.

## Praxis: Details der Code-Implementierung

Theorie ist gut, Praxis ist besser. Sehen wir uns an, wie die konkrete Implementierung aussieht.

### 1. Minimale Backend-Änderungen

Das Backend muss nur zwei Dinge erledigen: Synchronisierungsinformationen speichern und den OAuth-Callback verarbeiten.

**Datenbankänderung**

### 2. Frontend OAuth-Ablauf

Dies ist der Einstiegspunkt der Verbindung. Wir verwenden den Standard Authorization Code Flow.

```typescript
// src/HagiCode.Client/src/services/githubOAuth.ts

// 生成授权 URL 并跳转
export async function generateAuthUrl(): Promise<string> {
  const state = generateRandomString(); // 生成防 CSRF 的随机串
  sessionStorage.setItem('hagicode_github_state', state);

  const params = new URLSearchParams({
    client_id: clientId,
    redirect_uri: window.location.origin + '/settings?tab=github&oauth=callback',
    scope: ['repo', 'public_repo'].join(' '),
    state: state,
  });

  return `https://github.com/login/oauth/authorize?${params.toString()}`;
}

// 在回调页面处理 Code 换取 Token
export async function exchangeCodeForToken(code: string, state: string): Promise<GitHubToken> {
  // 1. 验证 State 防止 CSRF
  const savedState = sessionStorage.getItem('hagicode_github_state');
  if (state !== savedState) throw new Error('Invalid state parameter');

  // 2. 调用后端 API 进行 Token 交换
  // 注意：这里必须经过后端，因为需要 ClientSecret，不能暴露在前端
  const response = await fetch('/api/GitHubOAuth/callback', {
    method: 'POST',
    headers: { 'Content-Type': 'application/json' },
    body: JSON.stringify({ code, state, redirectUri: window.location.origin + '/settings?tab=github&oauth=callback' }),
  });

  if (!response.ok) throw new Error('Failed to exchange token');

  const token = await response.json();

  // 3. 存入 LocalStorage
  saveToken(token);
  return token;
}
```

### 3. GitHub API Client-Wrapper

Mit dem Token benötigen wir ein leistungsstarkes Werkzeug zum Aufrufen der GitHub API.

```typescript
// src/HagiCode.Client/src/services/githubApiClient.ts

const GITHUB_API_BASE = 'https://api.github.com';

// 核心请求封装
async function githubApi<T>(endpoint: string, options: RequestInit = {}): Promise<T> {
  const token = localStorage.getItem('gh_token');
  if (!token) throw new Error('Not connected to GitHub');

  const response = await fetch(`${GITHUB_API_BASE}${endpoint}`, {
    ...options,
    headers: {
      ...options.headers,
      Authorization: `Bearer ${token}`,
      Accept: 'application/vnd.github.v3+json', // 指定 API 版本
    },
  });

  // 错误处理逻辑
  if (!response.ok) {
    if (response.status === 401) throw new Error('GitHub Token 失效，请重新连接');
    if (response.status === 403) throw new Error('无权访问该仓库或超出速率限制');
    if (response.status === 422) throw new Error('Issue 验证失败，可能标题重复');
    throw new Error(`GitHub API Error: ${response.statusText}`);
  }

  return response.json();
}

// 创建 Issue
export async function createIssue(owner: string, repo: string, data: { title: string, body: string, labels: string[] }) {
  return githubApi(`/repos/${owner}/${repo}/issues`, {
    method: 'POST',
    body: JSON.stringify(data),
  });
}
```

### 4. Inhaltsformatierung und Synchronisierung

Der letzte Schritt besteht darin, die Session-Daten von HagiCode in das GitHub Issue-Format umzuwandeln. Dies ist gewissermaßen eine „Übersetzungsarbeit".

```typescript
// 将 Session 对象转换为 Markdown 字符串
function formatIssueForSession(session: Session): string {
  let content = `# ${session.title}\n\n`;
  content += `**> HagiCode Session:** #${session.code}\n`;
  content += `**> Status:** ${session.status}\n\n`;
  content += `## Description\n\n${session.description || 'No description provided.'}\n\n`;

  // 如果是 Proposal 类型，添加额外字段
  if (session.type === 'proposal') {
    content += `## Chief Complaint\n\n${session.chiefComplaint || ''}\n\n`;
    // 添加一个深链接，方便从 GitHub 跳回 HagiCode
    content += `---\n\n**[View in HagiCode](hagicode://sessions/${session.id})**\n`;
  }

  return content;
}

// 点击同步按钮的主逻辑
const handleSync = async (session: Session) => {
  try {
    const repoInfo = parseRepositoryFromUrl(session.repoUrl); // 解析仓库 URL
    if (!repoInfo) throw new Error('Invalid repository URL');

    toast.loading('正在同步到 GitHub...');

    // 1. 格式化内容
    const issueBody = formatIssueForSession(session);

    // 2. 调用 API
    const issue = await githubApiClient.createIssue(repoInfo.owner, repoInfo.repo, {
      title: `[HagiCode] ${session.title}`,
      body: issueBody,
      labels: ['hagicode', 'proposal', `status:${session.status}`],
    });

    // 3. 更新 Session Metadata (保存 Issue 链接)
    await SessionsService.patchApiSessionsSessionId(session.id, {
      metadata: {
        githubIssue: {
          owner: repoInfo.owner,
          repo: repoInfo.repo,
          issueNumber: issue.number,
          issueUrl: issue.html_url,
          syncedAt: new Date().toISOString(),
        }
      }
    });

    toast.success('同步成功！');
  } catch (err) {
    console.error(err);
    toast.error('同步失败，请检查 Token 或网络');
  }
};
```

## Zusammenfassung und Ausblick

Mit diesem „Frontend-Direktverbindungs"-Ansatz haben wir eine nahtlose GitHub Issues-Integration mit minimalem Backend-Code realisiert.

### Gewonnene Erkenntnisse
1.  **Hohe Entwicklungseffizienz**: Das Backend wurde nur minimal geändert – hauptsächlich ein neues Datenbankfeld und ein einfacher OAuth-Callback-Endpunkt. Der Großteil der Logik resides im Frontend.
2.  **Gute Sicherheit**: Der Token durchläuft nicht die Serverdatenbank, was das Risiko von Datenlecks verringert.
3.  **Hervorragende Benutzererfahrung**: Direkte Anfragen vom Frontend sorgen für schnelle Reaktionszeiten ohne Backend-Umwege.

### Hinweise
Bei der tatsächlichen Bereitstellung sollten einige Stolpersteine beachtet werden:
- **OAuth App-Einstellungen**: Vergessen Sie nicht, die korrekte `Authorization callback URL` in den GitHub OAuth App-Einstellungen anzugeben (in der Regel `http://localhost:3000/settings?tab=github&oauth=callback`).
- **Rate-Limiting**: Die GitHub API hat strenge Limits für nicht authentifizierte Anfragen, aber mit einem Token sind normalerweise 5000 Anfragen pro Stunde möglich.
- **URL-Parsing**: Die von Benutzern eingegebenen Repo-URLs können sehr unterschiedlich sein. Stellen Sie sicher, dass Ihre Regex auch Suffixe wie `.git` und SSH-Formate berücksichtigt.

### Zukünftige Erweiterungen
Derzeit ist die Funktion auf eine einseitige Synchronisierung (HagiCode -> GitHub) beschränkt. In Zukunft planen wir durch GitHub Webhooks eine bidirektionale Synchronisierung – beispielsweise soll das Schließen eines Issue auf GitHub automatisch den Session-Status in HagiCode aktualisieren. Dafür müssen wir im Backend einen Webhook-Empfangsendpunkt bereitstellen, was die nächste interessante Aufgabe darstellt.
