## Projekte im KI-Zeitalter nachahmen: Vault-System für projektübergreifende persistente Speicherung

> Im Zeitalter der KI-unterstützten Entwicklung: Wie können wir KI-Assistenten helfen, unsere Lernressourcen besser zu verstehen? Das HagiCode-Projekt hat durch das Vault-System eine einheitliche, von KI verständliche Wissensspeicher-Abstraktionsschicht implementiert, die die Lerneffizienz beim Nachahmen von Projekten erheblich verbessert.

## Hintergrund

Im KI-Zeitalter verändert sich die Art und Weise, wie Entwickler neue Technologien und Architekturen lernen, tiefgreifend. Das „Nachahmen von Projekten" – also das eingehende Studium und Lernen von Code, Architektur und Designmustern hervorragender Open-Source-Projekte – ist zu einer effektiven Lernmethode geworden. Im Vergleich zum traditionellen Lesen von Büchern und Ansehen von Videos können Sie durch das direkte Lesen und Ausführen hochwertiger Open-Source-Projekte die realen Ingenieurpraktiken schneller verstehen.

Aber diese Lernmethode steht auch vor zahlreichen Herausforderungen.

Lernmaterialien sind zu zerstreut. Ihre Notizen könnten in Obsidian sein, Code-Repositories in verschiedenen Ordnern verstreut, und der Gesprächsverlauf des KI-Assistenten ist wieder eine eigenständige Datensilbe. Wenn Sie möchten, dass die KI Ihnen bei der Analyse eines Projekts hilft, müssen Sie Codefragmente manuell kopieren und den Kontext zusammenstellen – dieser Prozess ist ziemlich umständlich.

Eigentlich noch problematischer ist der Kontextabriss. KI-Assistenten können nicht direkt auf Ihre lokalen Lernressourcen zugreifen, bei jedem Gespräch müssen Hintergrundinformationen erneut bereitgestellt werden. Außerdem werden die nachgeahmten Code-Repositories schnell aktualisiert, manuelle Synchronisierung ist fehleranfällig, und zwischen mehreren Lernprojekten ist es schwierig, Wissen zu teilen.

Diese Probleme sind im Wesentlichen auf „Datensilben" zurückzuführen. Wenn es eine einheitliche Speicher-Abstraktionsschicht gäbe, die KI-Assistenten verstehen und auf alle Ihre Lernressourcen zugreifen könnten, wäre das Problem gelöst.

## Designkonzepte des Vault-Systems

Der Kerngedanke des Vault-Systems ist sehr einfach: eine einheitliche, von KI verständliche Wissensspeicher-Abstraktionsschicht erstellen. Aus der Implementierungsperspektive hat das System mehrere Schlüsselmerkmale.

### Unterstützung mehrerer Typen

Das System unterstützt vier Vault-Typen, die verschiedenen Anwendungsszenarien entsprechen:

```typescript
// folder: allgemeiner Ordnertyp
export const DEFAULT_VAULT_TYPE = 'folder';

// coderef: speziell für das Nachahmen von Code-Projekten
export const CODEREF_VAULT_TYPE = 'coderef';

// obsidian: Integration mit der Obsidian-Notizsoftware
export const OBSIDIAN_VAULT_TYPE = 'obsidian';

// system-managed: automatisch vom System verwaltete vaults
export const SYSTEM_MANAGED_VAULT_TYPE = 'system-managed';
```

Der Typ **coderef** ist der am häufigsten verwendete in HagiCode. Er wurde speziell für das Nachahmen von Code-Projekten entworfen und bietet eine standardisierte Verzeichnisstruktur und von KI lesbare Metadatenbeschreibungen.

### Persistenter Speichermechanismus

Die Registry von Vault wird im JSON-Format persistent gespeichert, um sicherzustellen, dass die Konfiguration nach einem Application-Neustart verfügbar bleibt:

```csharp
public class VaultRegistryStore : IVaultRegistryStore
{
    private readonly string _registryFilePath;

    public VaultRegistryStore(IConfiguration configuration, ILogger<VaultRegistryStore> logger)
    {
        var dataDir = configuration["DataDir"] ?? "./data";
        var absoluteDataDir = Path.IsPathRooted(dataDir)
            ? dataDir
            : Path.GetFullPath(Path.Combine(Directory.GetCurrentDirectory(), dataDir));

        _registryFilePath = Path.Combine(absoluteDataDir, "personal-data", "vaults", "registry.json");
    }
}
```

Der Vorteil dieses Designs ist Einfachheit und Zuverlässigkeit. Das JSON-Format ist menschenlesbar, erleichtert das Debugging und manuelle Änderungen; Dateisystemspeicher vermeidet die Komplexität von Datenbanken und reduziert die Abhängigkeiten des Systems. Schließlich ist manchmal das Einfache das Beste.

### KI-Kontext-Integration

Am wichtigsten ist, dass das System automatisch Vault-Informationen in den Kontext des KI-Proposals injizieren kann:

```typescript
export function buildTargetVaultsText(
  vaults: VaultForText[],
  template: VaultPromptTemplate = DEFAULT_VAULT_PROMPT_TEMPLATE,
): string {
  const readOnlyVaults = vaults.filter((vault) => vault.accessType === 'read');
  const editableVaults = vaults.filter((vault) => vault.accessType === 'write');

  if (readOnlyVaults.length === 0 && editableVaults.length === 0) {
    return '';
  }

  const sections = [
    buildVaultSection(readOnlyVaults, template.reference),
    buildVaultSection(editableVaults, template.editable),
  ].filter(Boolean);

  return `\n\n### ${template.heading}\n\n${sections.join('\n')}`;
}
```

Auf diese Weise wird eine wichtige Funktion realisiert: Der KI-Assistent kann automatisch verfügbare Lernressourcen verstehen, ohne dass der Benutzer manuell Kontext bereitstellen muss. Das ist auch eine Art stillschweigendes Verständnis.

## Standardisierte Struktur von CodeRef Vault

Für Vault vom Typ coderef bietet HagiCode eine standardisierte Verzeichnisstruktur:

```
my-coderef-vault/
├── index.yaml          # vault-Metadatenbeschreibung
├── AGENTS.md           # Bedienungsanleitung für KI-Assistenten
├── docs/               # Speicherung von Lernnotizen und Dokumentationen
└── repos/              # Verwaltung der nachgeahmten Code-Repositories durch Git-Submodule
```

Beim Erstellen eines Vault initialisiert das System automatisch diese Struktur:

```csharp
private async Task EnsureCodeRefStructureAsync(
    string vaultName,
    string physicalPath,
    ICollection<VaultBootstrapDiagnosticDto> diagnostics,
    CancellationToken cancellationToken)
{
    Directory.CreateDirectory(physicalPath);

    var indexPath = Path.Combine(physicalPath, CodeRefIndexFileName);
    var docsPath = Path.Combine(physicalPath, CodeRefDocsDirectoryName);
    var reposPath = Path.Combine(physicalPath, CodeRefReposDirectoryName);

    // Standard-Verzeichnisstruktur erstellen
    if (!Directory.Exists(docsPath))
    {
        Directory.CreateDirectory(docsPath);
    }

    if (!Directory.Exists(reposPath))
    {
        Directory.CreateDirectory(reposPath);
    }

    // AGENTS.md-Anleitung erstellen
    await EnsureCodeRefAgentsDocumentAsync(physicalPath, cancellationToken);

    // index.yaml-Metadaten erstellen
    await WriteCodeRefIndexDocumentAsync(indexPath, mergedDocument, cancellationToken);
}
```

Das Design dieser Struktur ist auch wohldurchdacht:

- **docs/** Verzeichnis speichert Ihre Lernnotizen, Sie können das Verständnis von Code, Architekturanalyse, Erfahrungen mit Problemen usw. im Markdown-Format aufzeichnen
- **repos/** Verzeichnis verwaltet die nachgeahmten Repositories durch Git-Submodule anstatt Code direkt zu kopieren. So bleibt der Code synchron und spart Platz
- **index.yaml** enthält die Metadaten des Vault, damit der KI-Assistent schnell den Zweck und Inhalt dieses Vaults versteht
- **AGENTS.md** ist eine speziell für KI-Assistenten geschriebene Anleitung, die erklärt, wie mit dem Inhalt in diesem Vault umgegangen wird

Vielleicht kann die KI Ihre Ideen so leichter verstehen.

## Automatische Initialisierung durch Systemverwaltung

Neben dem manuellen Erstellen von Vaults unterstützt HagiCode auch automatisch vom System verwaltete Vaults:

```csharp
public async Task<IReadOnlyList<VaultRegistryEntry>> EnsureAllSystemManagedVaultsAsync(
    CancellationToken cancellationToken = default)
{
    var definitions = GetAllResolvedDefinitions();
    var entries = new List<VaultRegistryEntry>(definitions.Count);

    foreach (var definition in definitions)
    {
        entries.Add(await EnsureResolvedSystemManagedVaultAsync(definition, cancellationToken));
    }

    return entries;
}
```

Das System erstellt und verwaltet automatisch die folgenden Vaults:

- **hagiprojectdata**: Projektdatenspeicher, zum Speichern von Projektkonfigurationen und -status
- **personaldata**: Persönlicher Datenspeicher, zum Speichern von Benutzereinstellungen
- **hbsprompt**: Prompt-Vorlagenbibliothek, zum Verwalten häufig verwendeter KI-Prompts

Diese Vaults werden beim Systemstart automatisch initialisiert, ohne dass der Benutzer sie manuell konfigurieren muss. Schließlich, einige Dinge sollte man dem System überlassen, warum sollten sich Menschen darum kümmern?

## Zugriffssteuerungsmechanismus

Ein wichtiges Design ist die Zugriffssteuerung. Das System unterteilt Vaults in zwei Zugriffstypen:

```typescript
export interface VaultForText {
  id: string;
  name: string;
  type: string;
  physicalPath: string;
  accessType: 'read' | 'write';  // Schlüssel: Unterscheidung zwischen schreibgeschützt und bearbeitbar
}
```

- **reference（schreibgeschützt）**: KI wird nur für Analyse und Verständnis verwendet, kann Inhalte nicht ändern. Geeignet für referenzierte Open-Source-Projekte, Dokumentationen usw.
- **editable（bearbeitbar）**: KI kann Inhalte je nach Aufgabenbedarf ändern. Geeignet für Ihre Notizen, Entwürfe usw.

Diese Unterscheidung ist wichtig. Sie lässt die KI wissen, welche Inhalte „nur zu lesen Referenz" sind und welche „geändert werden können", und vermeidet das Risiko von Fehlbedienungen. Schließlich will niemand, dass sein hartes Arbeit versehentlich gelöscht wird.

## Praxis: Vault erstellen und verwenden

Nachdem wir die Prinzipien verstanden haben, lassen Sie uns sehen, wie es in der Praxis verwendet wird.

### CodeRef Vault erstellen

Das Folgende ist ein vollständiges Beispiel für einen Frontend-Aufruf:

```typescript
const createCodeRefVault = async () => {
  const response = await VaultService.postApiVaults({
    requestBody: {
      name: "React Learning Vault",
      type: "coderef",
      physicalPath: "/Users/developer/vaults/react-learning",
      gitUrl: "https://github.com/facebook/react.git"
    }
  });

  // Das System wird automatisch:
  // 1. React-Repository in vault/repos/react klonen
  // 2. docs/-Verzeichnis für Notizen erstellen
  // 3. index.yaml-Metadaten generieren
  // 4. AGENTS.md-Leitfaddatei erstellen

  return response;
};
```

Dieser API-Aufruf führt eine Reihe von Operationen durch: Verzeichnisstruktur erstellen, Git-Submodule initialisieren, Metadatendateien generieren usw. Sie müssen nur grundlegende Informationen bereitstellen, den Rest erledigt das System. Das ist auch ziemlich beruhigend.

## Best Practices und Hinweise

Bei der Verwendung des Vault-Systems haben wir einige Erfahrungen und Lektionen zusammengefasst.

### Pfadsicherheit

Das System überprüft Pfade strikt, um Pfad-Traversal-Angriffe zu verhindern:

```csharp
private static string ResolveFilePath(string vaultRoot, string relativePath)
{
    var rootPath = EnsureTrailingSeparator(Path.GetFullPath(vaultRoot));
    var combinedPath = Path.GetFullPath(Path.Combine(rootPath, relativePath));
    if (!combinedPath.StartsWith(rootPath, StringComparison.OrdinalIgnoreCase))
    {
        throw new BusinessException(VaultRelativePathTraversalCode,
            "Vault file paths must stay inside the registered vault root.");
    }
    return combinedPath;
}
```

Das ist sehr wichtig. Wenn Sie einen benutzerdefinierten Vault-Pfad festlegen, stellen Sie sicher, dass der Pfad innerhalb des zulässigen Bereichs liegt, sonst wird das System die Operation ablehnen. Bei Sicherheit kann man nicht genug betonen.

### Git-Submodul-Verwaltung

Für CodeRef-Vaults wird die Verwendung von Git-Submodulen anstatt direktem Kopieren von Code empfohlen:

```csharp
private static string BuildCodeRefAgentsContent()
{
    return """
    # CodeRef Vault Guide

    Repositories under `repos/` should be maintained through Git submodules
    rather than copied directly into the vault root.

    Keep this structure stable so assistants and tools can understand the vault quickly.
    """ + Environment.NewLine;
}
```

Dies hat mehrere Vorteile: Code mit Upstream synchron halten, Festplattenplatz sparen, einfache Verwaltung mehrerer Code-Versionen. Schließlich will niemand immer wieder dieselben Dinge herunterladen.

### Dateivorschau-Begrenzungen

Um Leistungsprobleme zu vermeiden, begrenzt das System Dateigröße und -typ:

```csharp
private const int FileEnumerationLimit = 500;
private const int PreviewByteLimit = 256 * 1024;  // 256KB
```

Wenn Ihr Vault eine große Anzahl von Dateien oder sehr große Dateien enthält, kann dies die Leistung der Vorschaufunktion beeinträchtigen. In diesem Fall können Sie die Verarbeitung in Stapeln oder spezielle Suchwerkzeuge in Betracht ziehen. Schließlich, einige Dinge sind zu groß und schwer zu verarbeiten.

### Diagnoseinformationen

Beim Erstellen eines Vaults werden Diagnoseinformationen zurückgegeben, um beim Debuggen zu helfen:

```csharp
List<VaultBootstrapDiagnosticDto> bootstrapDiagnostics = [];

if (IsCodeRefVaultType(normalizedType))
{
    bootstrapDiagnostics = await EnsureCodeRefBootstrapAsync(
        normalizedName,
        normalizedPhysicalPath,
        normalizedGitUrl,
        cancellationToken);
}
```

Wenn das Erstellen fehlschlägt, können Sie die Diagnoseinformationen überprüfen, um die genaue Ursache zu erfahren. Bei Fehlern die Diagnoseinformationen zu überprüfen, ist auch eine Methode zur Problemlösung.

Das Vault-System löst durch eine einheitliche Speicher-Abstraktionsschicht die Kernschmerzpunkte beim Nachahmen von Projekten im KI-Zeitalter:

- **Zentralisiertes Wissensmanagement**: Alle Lernressourcen an einem Ort, nicht mehr verstreut
