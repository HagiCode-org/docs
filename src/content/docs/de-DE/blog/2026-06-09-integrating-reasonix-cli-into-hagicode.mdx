## Von Grund auf: Integration von Reasonix CLI in das HagiCode-System

> Dieser Artikel teilt die vollständige technische Praxis der Integration von Reasonix CLI als erstklassigen Agent Provider in das HagiCode-System, einschließlich dreischichtigem Architekturdesign, Schlüsseltechnologieentscheidungen und Frontend- und Backend-Implementierungsdetails.

## Hintergrund

Reasonix CLI ist eigentlich eine ziemlich interessante Sache. Es ist ein AI-Codierungsassistent-Tool basierend auf ACP (Agent Communication Protocol), das leistungsstarke Streaming- und Sitzungsverwaltungsfunktionen bietet.其实在 HagiCode.Libs 层，我们已经把它的底层实现都搞定了，只是这些组件还处在孤立的状态，就像一个个漂亮的珍珠，还没串成项链。用户无法通过 Hero 职业选择、会话执行链路或监控面板来使用它，这多少有点可惜。

Das Problem, dem wir gegenüberstehen, ist: Wie man Reasonix auf denselben Rang wie Codex, Hermes usw. als erstklassigen Agent Provider hebt und vollständige Backend-Routing- und Frontend-Darstellung实现？这可不是简单地注册一个枚举值就能了事的，而是需要构建从底层抽象到用户界面的完整链路。就像盖房子，不能只打个地基就完事了，总得把墙砌起来，屋顶盖上去。

Die Herausforderung bei dieser Integration liegt darin, dass Reasonix als lokales CLI-Tool seinen eigenen Charakter und seine eigenen Eigenheiten hat. Zum Beispiel benötigt es keine Verbindungszeichenfolge, alle Parameter werden vom Benutzer zur Laufzeit konfiguriert; es ist möglicherweise gar nicht installiert und erfordert ein graceful Degradation-Handling; es ist mit der anthropic-Modellreihe kompatibel, hat aber eigene ACP-spezifische Parameter wie effort, budget usw. Das ist wie eine Person mit ihrer eigenen einzigartigen Arbeitsweise, man kann sie nicht zwingen.

Nach sorgfältigem Architekturdesign und mehreren Diskussionen haben wir uns schließlich für ein klares dreischichtiges Architekturkonzept entschieden und Reasonix erfolgreich in das System integriert. Dieses Konzept löst nicht nur das aktuelle Problem, sondern bietet auch ein wiederverwendbares Muster für die Integration ähnlicher CLI Provider in Zukunft. Viele Dinge sind so: Sobald man die richtige Methode gefunden hat, wird der Weg viel einfacher.

## Über HagiCode

Die in diesem Artikel geteilten Lösungen stammen aus unserer praktischen Erfahrung im [HagiCode](https://hagicode.com)-Projekt. HagiCode ist ein Open-Source-AI-Codierungsassistent-Projekt, das sich der Bereitstellung leistungsstarker Codegenerierungs-, Refactoring- und Optimierungsfähigkeiten für Entwickler verschrieben hat. Im Entwicklungsprozess sind wir auf verschiedene technische Herausforderungen gestoßen, von denen die Integration von Reasonix als erstklassiger Agent Provider eine ist. Wenn Sie die in diesem Artikel geteilten Lösungen für wertvoll halten, bedeutet das, dass unsere technische Praxis nicht schlecht ist, und dann ist HagiCode selbst auch einen Blick wert.

## Kerninhalt

### Technisches Architekturdesign

Das System verwendet eine klare dreischichtige Architektur zur Trennung von Zuständigkeiten, wobei jede Schicht klare Zuständigkeitsgrenzen hat:

**HagiCode.Libs-Schicht**: Diese Schicht ist bereits abgeschlossen und bietet Abstraktionen und konkrete Implementierungen für CLI provider. Sie definiert die Schnittstelle `ICliProvider<ReasonixOptions>`, implementiert `ReasonixProvider` zur Verarbeitung von ACP-Streaming und Sitzungsverwaltung und unterstützt gleichzeitig Parameter wie effort, budget, yolo, transcript usw. Die Zuständigkeit dieser Schicht besteht darin, stabile, wiederverwendbare Basisfunktionen bereitzustellen, ohne Geschäftslogik zu berühren. Wie das Fundament eines Hauses, obwohl man es nicht sieht, ist es sehr wichtig.

**hagicode-core-Schicht**: Dies ist der Schwerpunkt unserer Integration. Sie ist dafür verantwortlich, die Basisabstraktion mit der einheitlichen Systemschnittstelle zu verbinden. Die konkreten Aufgaben umfassen die Registrierung des Enumerationswerts `AIProviderType.ReasonixCli = 12`, die Erstellung von `ReasonixCliProvider` als thin adapter zur Verbindung mit der Libs-Schicht, die Implementierung von `ReasonixGrain` zur Verarbeitung von Sitzungszuständen und Ausführungsströmen sowie die Integration des Hero-Systems für Parameterzuordnung und Konfigurationsverwaltung. Der Kern dieser Schicht ist die Koordination der Komponenten und der Aufbau vollständiger Geschäftsstränge. Wie die tragenden Wände eines Hauses, die alle Teile verbinden.

**web-Schicht**: Verantwortlich für die Darstellung und Konfigurationserfassung für den Benutzer. Wir müssen OpenAPI-Typen neu generieren, um neue Enumerationswerte zu unterstützen, visuelle Typzuordnungen zu implementieren, damit Reasonix sein eigenes Symbol und seinen eigenen Anzeigenamen hat, CLI-Parameterkonfigurationsformulare zu erstellen, damit Benutzer die einzelnen Parameter konfigurieren können, sowie mehrsprachige Unterstützung hinzuzufügen. Der Schwerpunkt dieser Schicht liegt auf Benutzererlebnis und Interaktionsdesign. Wie die Inneneinrichtung eines Hauses - ob sie gut gemacht ist, beeinflusst direkt, wie angenehm es ist zu wohnen.

Diese dreischichtige Gestaltung ermöglicht es jeder Schicht, sich auf ihre eigenen Zuständigkeiten zu konzentrieren, verringert die Komplexität des Systems und erleichtert die spätere Wartung und Erweiterung. Oft ist es einfacher, Dinge klar zu trennen.

### Schlüsseltechnologieentscheidungen

Während der Implementierung haben wir mehrere wichtige technische Entscheidungen getroffen, die einen wichtigen Einfluss auf die endgültige Architektur und das Benutzererlebnis haben.

**Entscheidung 1: Verwendung eines dedizierten Grain**

Wir haben ein eigenständiges `ReasonixGrain : IReasonixGrain, IExecutorStreamGrain` erstellt, anstatt zu versuchen, ein gemeinsam genutztes Grain wiederzuverwenden. Diese Entscheidung folgt dem etablierten Muster der 11 vorhandenen provider im System. Obwohl es möglicherweise zu etwas Codeduplizierung zu kommen scheint, ermöglicht ein dediziertes Grain eine präzise Kontrolle der spezifischen Merkmale von Reasonix, wie etwa seinen einzigartigen Sitzungsbindungsmechanismus und die ACP-Nachrichtenzuordnung. Wir haben auch ein leeres Antwort-DTO `ReasonixResponse` als Typunterscheider definiert. Obwohl es keine tatsächlichen Daten enthält, spielt es eine wichtige Rolle im Typsystem. Wie jeder sein eigenes Zimmer hat, auch wenn es leer ist, ist es sein eigener Raum.

**Entscheidung 2: Keine Erstellung einer dedizierten Settings-Klasse**

Im Gegensatz zu einigen Provider, die Verbindungszeichenfolgen benötigen, werden alle Konfigurationen von Reasonix vom Benutzer zur Laufzeit festgelegt und benötigen keine Validierung beim Start. Daher haben wir keine dedizierte Settings-Klasse erstellt, sondern alle Konfigurationen im Wörterbuch `AIProviderOptions.Providers[ReasonixCli].Settings` gespeichert. Dieses Muster ist mit anderen lokalen CLI Provider wie Qoder, Kiro, Kimi usw. konsistent, vereinfacht die Codestruktur und vermeidet unnötige Abstraktionsschichten. Die unterstützten Einstellungsschlüssel umfassen: `effort`, `budgetUsd`, `transcriptPath`, `enableYolo`, `arguments`, `startupTimeoutMs`, `reasoning`. Manchmal ist einfacher besser.

**Entscheidung 3: Provider-Strategie-Gesundheitsüberwachung**

Reasonix ist ein lokal vom Benutzer installiertes CLI, das möglicherweise gar nicht installiert ist oder sich nicht im System-PATH befindet. In diesem Fall sollten wir nicht direkt einen Fehler ausgeben, sondern graceful degradation durchführen. Wir verwenden die `Provider`-Strategie und überprüfen über `CommandUtil.TryResolveExecutablePath`, ob das CLI verfügbar ist. Wenn die Überprüfung fehlschlägt, wird die UI als "nicht verfügbar" angezeigt, dies wirkt sich jedoch nicht auf andere Teile des Systems aus. Dieses Design macht das System robuster und gibt dem Benutzer eine klare Rückmeldung. Schließlich möchte niemand, dass das gesamte System wegen eines kleinen Problems ausfällt.

**Entscheidung 4: Wirtschaftssystem-Klassifizierung**

Im HagiCode-System haben unterschiedliche Provider unterschiedliche Wirtschaftssystem-Klassifizierungen. Wir haben entschieden, dass Reasonix standardmäßig die `'claude'`-Wirtschaftssystem-Klassifizierung verwendet, da Reasonix selbst mit der anthropic-Modellreihe kompatibel ist. Derzeit haben nur Codex und Copilot dedizierte Wirtschaftssystem-Klassifizierungen, andere Provider verwenden vorhandene Klassifizierungen wieder. Dies hält das System einfach und kann dennoch Abrechnung und Kostenstatistiken korrekt verarbeiten. Wiederverwendung ist auch eine Weisheit, man muss nicht alles von vorne beginnen.

**Entscheidung 5: Modellkompatibilität**

Reasonix unterstützt mehrere Modelle über das `--model`-Flag, insbesondere die anthropic-Reihe. Wir haben Kompatibilitätszuordnungen in `secondary-professions.index.json` hinzugefügt, damit Benutzer diese Modelle in Reasonix auswählen können. Dieses Design respektiert die Fähigkeiten von Reasonix und behält die Konsistenz des Systems bei, sodass Benutzer ohne Verständnis der zugrundeliegenden Unterschiede reibungslos verschiedene Modelle verwenden können. Benutzern ist es auch nicht leicht - lassen Sie es sie einfach halten.

**Thin Adapter-Implementierung**

`ReasonixCliProvider` ist die Schlüsselkomponente, die die Libs-Schicht mit der einheitlichen Systemschnittstelle verbindet:

```csharp
public sealed class ReasonixCliProvider : IAIProvider, IVersionedAIProvider, IAsyncDisposable
{
    private static readonly IReadOnlyList<string> SupportedSettingKeys =
    [
        "effort",
        "budgetUsd",
        "transcriptPath",
        "enableYolo",
        "arguments",
        "startupTimeoutMs",
        "reasoning"
    ];

    private readonly ICliProvider<ReasonixOptions> _provider;
    private readonly ConcurrentDictionary<string, string> _sessionBindings = new(StringComparer.Ordinal);

    public async IAsyncEnumerable<AIStreamingChunk> StreamCoreAsync(
        AIRequest request,
        string? sessionId = null,
        [EnumeratorCancellation] CancellationToken cancellationToken = default)
    {
        var options = BuildOptions(request, sessionId);
        await foreach (var message in _provider.StreamAsync(options, cancellationToken))
        {
            yield return MapToStreamingChunk(message);
        }
    }

    private ReasonixOptions BuildOptions(AIRequest request, string? sessionId)
    {
        return new ReasonixOptions
        {
            ExecutablePath = GetExecutablePath(),
            WorkingDirectory = GetWorkingDirectory(),
            Model = _config.Model,
            Effort = _config.Settings.GetValueOrDefault("effort", "medium"),
            Budget = _config.Settings.GetValueOrDefault("budgetUsd", 10.0),
            Yolo = _config.Settings.GetValueOrDefault("enableYolo", false),
            TranscriptPath = _config.Settings.GetValueOrDefault("transcriptPath"),
            Arguments = _config.Settings.GetValueOrDefault("arguments", ""),
            StartupTimeout = GetStartupTimeout(),
            EnvironmentVariables = _environmentVariables,
            CessionId = sessionId ?? GetCessionId()
        };
    }
}
```

Die wichtigsten Zuständigkeiten dieses Adapters sind:
1. Validierung von Konfigurationsparametern, Ablehnung nicht unterstützter Einstellungsschlüssel
2. Aufrechterhaltung von Sitzungsbindungen, Unterstützung der Sitzungswiederherstellung
3. Zuordnung von ACP-Nachrichten zum einheitlichen Format des Systems
4. Verwendung von `ProviderErrorAutoRetryCoordinator` zur Implementierung automatischer Wiederholungen

Wie ein Dolmetscher, der eine Sprache in eine andere übersetzt und dabei sicherstellt, dass die Bedeutung korrekt ist.

**Orleans Grain-Implementierung**

`ReasonixGrain` ist für die Verarbeitung von Sitzungszuständen und Ausführungsströmen verantwortlich:

```csharp
public class ReasonixGrain : Grain, IReasonixGrain, IExecutorStreamGrain
{
    private readonly Dictionary<string, ExecutorToolLifecycleStatus> _toolLifecycleState =
        new(StringComparer.Ordinal);

    public IAsyncEnumerable<ReasonixResponse> ExecuteCommandStreamAsync(
        string command,
        string? heroId = null,
        CancellationToken token = default,
        string? executionMessageId = null,
        string? systemMessage = null,
        Dictionary<string, string>? requestSettings = null)
    {
        var request = BuildRequest(command, isEdit: false, heroId, executionMessageId, systemMessage, requestSettings);
        return SendAsync(request, heroId, token);
    }

    private async IAsyncEnumerable<ReasonixResponse> SendAsync(
        AIRequest request,
        string? heroId,
        [EnumeratorCancellation] CancellationToken token)
    {
        _cancellationTokenSource = new CancellationTokenSource();
        var linkedToken = CancellationTokenSource.CreateLinkedTokenSource(token, _cancellationTokenSource.Token);

        var provider = await ResolveReasonixProviderAsync(heroId);

        await foreach (var chunk in provider.StreamAsync(request, linkedToken.Token))
        {
            var response = BuildChunkResponse(chunk);
            yield return response;
        }
    }

    private async Task<IAIProvider> ResolveReasonixProviderAsync(string? heroId)
    {
        // Hero-bewusste Konfigurationsfallback-Logik
        var config = await HeroProviderResolver.ResolveAsync(AIProviderType.ReasonixCli, heroId);
        return _aiProviderFactory.CreateProvider(AIProviderType.ReasonixCli, config);
    }
}
```

Die Kernfunktionen des Grain umfassen:
1. Verwendung von `[PersistentState("reasonix-interop")]` zur Aufrechterhaltung von Sitzungszuständen
2. Implementierung der Hero-bewussten Konfigurationsfallback-Logik
3. Verfolgung von Tool-Lebenszyklusstatus
4. Unterstützung von Abbruchtokenketten, um sicherzustellen, dass die Ausführung rechtzeitig unterbrochen werden kann

Wie ein Hausmeister, der die Dinge ordentlich organisiert.

### Best Practices und Hinweise

Während der Implementierung haben wir einige Best Practices und zu beachtende Punkte zusammengefasst.

**Parametervalidierung**

ReasonixCliProvider muss Konfigurationsparameter strikt validieren und nicht unterstützte Einstellungsschlüssel ablehnen:

```csharp
public void ValidateConfigurationOverrides(Dictionary<string, string?> overrides)
{
    foreach (var key in overrides.Keys)
    {
        if (!SupportedSettingKeys.Contains(key))
        {
            throw new HeroProviderConfigurationException(
                $"Unsupported setting key '{key}' for Reasonix provider");
        }
    }
}
```

Dies verhindert, dass Benutzer falsche Parameter konfigurieren und vermeidet Laufzeitfehler. Wie ein Torwart - man kann nicht zulassen, dass Dinge, die nicht hineingehören, hereinkommen.

**Sitzungsbindungsverwaltung**

Verwendung von `ConcurrentDictionary` zur Verwaltung von Sitzungsbindungen, Unterstützung der Sitzungswiederherstellung:

```csharp
_sessionBindings[cessionId] = sessionId;

// Bindung einer vorhandenen Sitzung in nachfolgenden Anforderungen
if (_sessionBindings.TryGetValue(cessionId, out var boundSessionId))
{
    options.SessionId = boundSessionId;
}
```

Dieses Design ermöglicht es Benutzern, eine vorherige Sitzung nach einer Unterbrechung wiederherzustellen und bietet ein besseres Erlebnis. Wie das Merken einer Geschichte, damit man beim nächsten Mal weitererzählen kann.

**Graceful Degradation-Handling**

Das Frontend sollte die CLI-Verfügbarkeit überprüfen und freundliche Hinweise geben:

```typescript
const reasonixAvailable = await healthApi.checkCliAvailable('reasonix');
if (!reasonixAvailable) {
  showMessage('Reasonix CLI ist nicht installiert, bitte im System-PATH konfigurieren');
}
```

Lassen Sie Benutzer nicht auf unverständliche Fehler stoßen - überprüfen Sie vorher und geben Sie klare Anweisungen. Schließlich will niemand unverständlich steckenbleiben.

**Testabdeckung**

Umfassende Tests sind der Schlüssel zur Qualitätssicherung:

