## Onboarding neuer Benutzer in React-Projekten elegant umsetzen: HagiCodes driver.js-Praxis

> Wenn Benutzer Ihr Produkt zum ersten Mal öffnen, wissen sie wirklich, wo sie anfangen sollen? Dieser Artikel diskutiert unsere Erfahrungen mit driver.js für das Onboarding neuer Benutzer im HagiCode-Projekt – als Anregung für andere.

## Hintergrund

Sind Sie schon einmal in eine solche Situation geraten: Ein neuer Benutzer registriert sich für Ihr Produkt, öffnet die Seite und schaut völlig verloren drein, weiß nicht, wo er klicken oder was er tun soll. Als Entwickler gehen wir oft davon aus, dass Benutzer "selbst erkunden" werden – schließlich ist die Neugier des Menschen unbegrenzt. Die Realität ist jedoch: Die meisten Benutzer werden innerhalb weniger Minuten leise verlassen, weil sie den Einstieg nicht finden – die Geschichte beginnt plötzlich und endet ebenso natürlich.

Das Onboarding neuer Benutzer ist ein wichtiges Mittel, um dieses Problem zu lösen, aber die Umsetzung ist auch nicht so einfach. Ein gutes Onboarding-System muss:

- Seite-Elemente präzise定位 und hervorheben können
- Mehrstufige Onboarding-Prozesse unterstützen
- Benutzerentscheidungen merken können (abgeschlossen/übersprungen)
- Seitenleistung und normale Interaktion nicht beeinträchtigen
- Klare Codestruktur haben, leicht zu warten

Bei der Entwicklung von HagiCode sind wir auf die gleichen Herausforderungen gestoßen. HagiCode ist ein KI-Coding-Assistent-Projekt, dessen Kernworkflow "Benutzer erstellt Proposal → KI generiert Plan → Benutzer prüft → KI führt aus" ist – ein solcher OpenSpec-Prozess. Für Benutzer, die zum ersten Mal mit diesem Konzept in Berührung kommen, ist dieser Prozess völlig neu, und sie benötigen ein gutes Onboarding, um schnell einzusteigen. Schließlich brauchen neue Dinge immer etwas Zeit.

## Über HagiCode

Die in diesem Artikel vorgestellte Lösung stammt aus unserer Praxis im [HagiCode](https://hagicode.com)-Projekt. HagiCode ist ein auf Claude basierender KI-Coding-Assistent, der Entwicklern durch den OpenSpec-Workflow hilft, Code-Aufgaben effizienter zu erledigen. Sie können unseren Open-Source-Code auf [GitHub](https://github.com/HagiCode-org/site) einsehen.

## Warum driver.js

In der Phase der Technikauswahl haben wir mehrere gängige Onboarding-Bibliotheken evaluiert, jede hat ihre eigenen Besonderheiten:

- **Intro.js**: Leistungsstark aber großvolumig, Stil-Anpassung relativ komplex
- **Shepherd.js**: API-Design gut, aber für unser Szenario etwas "schwer"
- **driver.js**: Leichtgewichtig, einfach, API intuitiv, unterstützt React-Ökosystem

Schließlich haben wir driver.js gewählt, ohne besonderen Grund, hauptsächlich basierend auf folgenden Erwägungen:

1. **Leichtgewichtig**: Kleine Kernbibliothek, erhöht die Bundle-Größe nicht signifikant
2. **Einfache API**: Klare, intuitive Konfiguration, schnelles Erlernen
3. **Flexibilität**: Unterstützt benutzerdefinierte Positionierung, Stile und Interaktionsverhalten
4. **Dynamischer Import**: Kann bei Bedarf geladen werden, beeinträchtigt die First-Screen-Leistung nicht

Bei der Technikauswahl gibt es eigentlich nicht das Beste, nur das Geeignete.

## Technische Umsetzung

### Kernkonfiguration

Die Konfiguration von driver.js ist sehr intuitiv, folgend die Kernkonfiguration im HagiCode-Projekt:

```typescript
import { driver } from 'driver.js';
import 'driver.js/dist/driver.css';

const newConversationDriver = driver({
  allowClose: true,           // Benutzer dürfen Onboarding schließen
  animate: true,              // Animationseffekte aktivieren
  overlayClickBehavior: 'close', // Klick auf Overlay schließt Onboarding
  disableActiveInteraction: false, // Elemente interaktiv halten
  showProgress: false,        // Keine Fortschrittsanzeige (wir haben benutzerdefiniertes Fortschrittsmanagement)
  steps: guideSteps           // Array mit Onboarding-Schritten
});
```

Die Überlegungen hinter diesen Konfigurationen sind:

- `allowClose: true` - Benutzerentscheidung respektieren, nicht zum Abschluss zwingen – schließlich schmeckt eine erzwungene Melone nicht süß
- `disableActiveInteraction: false` - Einige Schritte erfordern tatsächliche Benutzeroperationen (z. B. Texteingabe), daher Interaktion nicht deaktivieren
- `overlayClickBehavior: 'close'` - Ein schnelles Exit für Benutzer

### Zustandsverwaltung

Die Persistenz des Onboarding-Zustands ist entscheidend – wir wollen nicht bei jedem Seiten-Refresh erneut onboarden, das wäre ziemlich ärgerlich. HagiCode verwendet localStorage zur Verwaltung des Onboarding-Zustands:

```typescript
export type GuideState = 'pending' | 'dismissed' | 'completed';

export interface UserGuideState {
  session: GuideState;
  detailGuides: Record<string, GuideState>;
}

// Zustand lesen
export const getUserGuideState = (): UserGuideState => {
  const state = localStorage.getItem('userGuideState');
  return state ? JSON.parse(state) : { session: 'pending', detailGuides: {} };
};

// Zustand aktualisieren
export const setUserGuideState = (state: UserGuideState) => {
  localStorage.setItem('userGuideState', JSON.stringify(state));
};
```

Wir definieren drei Zustände:

- `pending`: Onboarding läuft, Benutzer noch nicht abgeschlossen oder übersprungen
- `dismissed`: Benutzer hat Onboarding aktiv geschlossen
- `completed`: Benutzer hat alle Schritte abgeschlossen

Für das Onboarding der Proposal-Detailseite unterstützen wir auch eine feingranularere Zustandsverfolgung (über `detailGuides`-Dictionary), da ein Proposal mehrere Phasen durchlaufen kann (Entwurf, Prüfung, Ausführung abgeschlossen), jede Phase benötigt unterschiedliche Anleitung. Schließlich ändern sich die Zustände von Dingen ständig.

### Ziel-Element-Positionierung

driver.js verwendet CSS-Selektoren zur Positionierung von Ziel-Elementen. HagiCode verwendet eine Konvention: `data-guide`-Custom-Attribut zur Markierung von Onboarding-Zielen:

```typescript
const steps = [
  {
    element: '[data-guide="launch"]',
    popover: {
      title: 'Neue Konversation starten',
      description: 'Klicken Sie hier, um eine neue Konversationssitzung zu erstellen...'
    }
  }
];
```

Verwendung in Komponenten:

```typescript
<button data-guide="launch" onClick={handleLaunch}>
  Neue Konversation
</button>
```

Die Vorteile dieser Vorgehensweise sind:

- Konfliktvermeidung mit Geschäftsstil-Klassennamen
- Klare Semantik, auf den ersten Blick erkennbar, dass dieses Element mit Onboarding zusammenhängt
- Einfach zu verwalten und warten

### Optimierung durch dynamischen Import

Da die Onboarding-Funktion nur in bestimmten Szenarien benötigt wird (z. B. beim ersten Besuch neuer Benutzer), verwenden wir dynamischen Import zur Optimierung der initialen Ladeleistung:

```typescript
const initNewUserGuide = async () => {
  // driver.js dynamisch importieren
  const { driver } = await import('driver.js');
  await import('driver.js/dist/driver.css');

  // Onboarding initialisieren
  const newConversationDriver = driver({
    // ...Konfiguration
  });

  newConversationDriver.drive();
};
```

So werden driver.js und seine Stildateien nur bei Bedarf geladen und beeinträchtigen nicht die First-Screen-Leistung. Schließlich will niemand für etwas warten, das er vorerst nicht benötigt.

## Onboarding-Flow-Design

HagiCode implementiert zwei Onboarding-Pfade, die die Kernnutzungsszenarien der Benutzer abdecken.

### Konversations-Onboarding (10 Schritte)

Dieses Onboarding hilft dem Benutzer, den gesamten Prozess vom Erstellen einer Konversation bis zum Einreichen des ersten vollständigen Proposals abzuschließen:

1. **launch** - Onboarding starten, Button "Neue Konversation" vorstellen
2. **compose** - Benutzer zur Eingabe einer Anfrage im Eingabefeld führen
3. **send** - Zum Klicken des Senden-Buttons führen
4. **proposal-launch-readme** - Zum Erstellen eines README-Proposals führen
5. **proposal-compose-readme** - Zum Bearbeiten der README-Anfrage führen
6. **proposal-submit-readme** - Zum Einreichen des README-Proposals führen
7. **proposal-launch-agents** - Zum Erstellen eines AGENTS.md-Proposals führen
8. **proposal-compose-agents** - Zum Bearbeiten der AGENTS.md-Anfrage führen
9. **proposal-submit-agents** - Zum Einreichen des AGENTS.md-Proposals führen
10. **proposal-wait** - Erklären, dass die KI verarbeitet, bitte warten

Das Designkonzept dieses Onboardings ist: Durch zwei tatsächliche Proposal-Erstellungsaufgaben (README und AGENTS.md) den Benutzer den Kern-Workflow von HagiCode persönlich erleben lassen. Schließlich: Auf Papier ist es immer flach, echtes Verständnis kommt durch praktische Anwendung.

### Proposal-Detail-Onboarding (3 Schritte)

Wenn der Benutzer die Proposal-Detailseite betritt, wird je nach aktuellem Proposal-Status das entsprechende Onboarding ausgelöst:

1. **drafting** (Entwurfsphase) - Benutzer zum Ansehen des KI-generierten Plans führen
2. **reviewing** (Prüfungsphase) - Benutzer zur Ausführung des Plans führen
3. **executionCompleted** (Abschlussphase) - Benutzer zum Archivieren des Plans führen

Dieses Onboarding ist statusgesteuert – entscheidet dynamisch, welcher Onboarding-Schritt angezeigt wird, basierend auf dem tatsächlichen Status des Proposals. Dinge ändern sich ständig, das Onboarding sollte sich ebenfalls ändern.

In dieser Phase fokussiert das Onboarding die Aufmerksamkeit des Benutzers auf die Schlüsselaktion "Plan erstellen", um Verwirrung beim ersten Betreten der Detailseite zu vermeiden.

### Element-Rendering-Wiederholungsmechanismus

In React-Anwendungen könnten Onboarding-Zielelemente noch nicht fertig gerendert sein (z. B. warten auf asynchrone Datenladung). Um diese Situation zu behandeln, implementiert HagiCode einen Wiederholungsmechanismus:

```typescript
const waitForElement = (selector: string, maxRetries = 10, interval = 100) => {
  let retries = 0;
  return new Promise<HTMLElement>((resolve, reject) => {
    const checkElement = () => {
      const element = document.querySelector(selector) as HTMLElement;
      if (element) {
        resolve(element);
      } else if (retries < maxRetries) {
        retries++;
        setTimeout(checkElement, interval);
      } else {
        reject(new Error(`Element not found: ${selector}`));
      }
    };
    checkElement();
  });
};
```

Vor der Initialisierung des Onboardings diese Funktion aufrufen, um sicherzustellen, dass die Zielelemente bereits existieren. Manchmal ist es wert, etwas zu warten.

## Zusammenfassung der Best Practices

Basierend auf den Praxiserfahrungen von HagiCode werden hier einige wichtige Best Practices geteilt:

### 1. Onboarding sollte "verlassbar" sein

Zwingen Sie Benutzer nicht zum Abschluss des Onboardings. Einige Benutzer sind Entdecker-Typ, sie bevorzugen Selbsterkundung. Bieten Sie einen klaren "Überspringen"-Button und merken Sie sich die Wahl des Benutzers, um ihn下次 nicht zu stören. Schließlich: Schöne Dinge oder Menschen muss man nicht besitzen, solange sie schön sind, kann man ihre Schönheit gut genießen.

### 2. Onboarding-Inhalte sollten prägnant und kraftvoll sein

Jeder Onboarding-Schritt sollte sich auf ein einzelnes Ziel konzentrieren:

- **Title**: Kurz und klar, nicht mehr als 10 Zeichen
- **Description**: Auf den Punkt kommen, dem Benutzer sagen "Was ist das" und "Warum nutzen"

Vermeiden Sie lange Erklärungen – die Aufmerksamkeit der Benutzer in der Onboarding-Phase ist sehr begrenzt. Zu viele Worte und niemand liest sie gerne.

### 3. Selektoren sollten stabil sein

Verwenden Sie stabile, nicht häufig ändernde Element-Markierungsmethoden. Das `data-guide`-Custom-Attribut ist eine gute Wahl, vermeiden Sie Abhängigkeit von Klassennamen oder DOM-Struktur, da sich diese leicht bei Refactorings ändern können. Code ändert sich ständig, aber einige Dinge sollten so stabil wie möglich bleiben.

### 4. Testen Sie Ihr Onboarding

HagiCode hat vollständige Testfälle für die Onboarding-Funktion geschrieben:

```typescript
describe('NewUserConversationGuide', () => {
  it('sollte Onboarding-Zustand korrekt initialisieren', () => {
    const state = getUserGuideState();
    expect(state.session).toBe('pending');
  });

  it('sollte Onboarding-Zustand korrekt aktualisieren', () => {
    setUserGuideState({ session: 'completed', detailGuides: {} });
    const state = getUserGuideState();
    expect(state.session).toBe('completed');
  });
});
```

Tests können sicherstellen, dass beim Refactoring von Code die Onboarding-Funktion nicht versehentlich beschädigt wird. Schließlich will niemand, dass ein paar Codeänderungen die vorherige Funktion zerstören.

### 5. Leistungsoptimierung

- Dynamischen Import verwenden, um Onboarding-Bibliothek verzögert zu laden
- Vermeiden, dass Onboarding-Logik noch initialisiert wird, nachdem Benutzer das Onboarding bereits abgeschlossen haben
- Leistungseinfluss von Onboarding-Animationen berücksichtigen, auf Low-End-Geräten können Animationen deaktiviert werden

Leistung ist wie das Leben – wo gespart werden sollte, sollte man sparen.

## Zusammenfassung

Das Onboarding neuer Benutzer ist ein wichtiger Aspekt zur Verbesserung der Produkt-Benutzererfahrung. Im HagiCode-Projekt haben wir mit driver.js ein vollständiges Onboarding-System aufgebaut, das den gesamten Workflow vom Erstellen von Konversationen bis zur Proposal-Ausführung abdeckt.

Durch diesen Artikel möchten wir unsere Kernansichten vermitteln:

1. **Technikauswahl muss zu Anforderungen passen**: driver.js ist nicht das stärkste, aber für uns das geeignete
2. **Zustandsverwaltung ist entscheidend**: Onboarding-Zustand mit localStorage persistieren, wiederholte Störung von Benutzern vermeiden
3. **Onboarding-Design sollte fokussiert sein**: Jeder Schritt löst ein Problem, nicht gierig werden
4. **Codestruktur sollte klar sein**: Onboarding-Konfiguration, Zustandsverwaltung und UI-Logik trennen, einfach zu warten

Wenn Sie gerade eine Onboarding-Funktion für neues Benutzer für Ihr Projekt hinzufügen, hoffen wir, dass die Praxiserfahrungen in diesem Artikel hilfreich für Sie sind. Technologie ist eigentlich nicht mysteriös – mehr ausprobieren, mehr zusammenfassen, langsam wird es besser...
