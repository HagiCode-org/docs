## Von Docusaurus 3.x zu Astro 5.x: Praxisbericht zur HagiCode-Sitenmigration

> Dieser Artikel fasst unseren vollständigen Migrationsprozess von Docusaurus 3.x zu Astro 5.x für die offizielle HagiCode-Website zusammen. Wir werden eingehend diskutieren, wie wir durch Astros Islands-Architektur Leistungsengpässe lösen, während wir unsere bestehenden React-KomponentenAssets behalten und eine doppelte Verbesserung von Buildgeschwindigkeit und Ladeleistung erzielen.


## Hintergrund

Im Januar 2026 haben wir an der offiziellen HagiCode-Website eine "Herztransplantation" durchgeführt – die vollständige Migration des Kernframeworks von Docusaurus 3.x zu Astro 5.x. Dies war keine impulsive große Refactoring-Maßnahme, sondern eine wohlüberlegte technische Entscheidung.

Vor der Migration war unsere Site zwar funktional vollständig, zeigte aber allmählich einige "Wohlstandsprobleme": aufgeblähte Build-Artefakte, übermäßige JavaScript-Last und unzureichende Seitenladegeschwindigkeit bei komplexen Dokumentationsseiten. Als ein AI-Kode-Assistent-Projekt muss HagiCode Dokumentation und Funktionsbeschreibungen häufig aktualisieren; die Build-Effizienz wirkt sich direkt auf die Veröffentlichungsgeschwindigkeit aus. Gleichzeitig wollten wir, dass die Site für Suchmaschinen (SEO) freundlicher ist, damit mehr Entwickler dieses Projekt entdecken.

Um diese Schmerzpunkte zu lösen, trafen wir eine kühne Entscheidung: das gesamte Build-System komplett neu aufzubauen und zu Astro zu migrieren. Die Veränderungen, die diese Entscheidung mit sich brachte, waren größer, als Sie sich vielleicht vorstellen – dazu später mehr.

## Über HagiCode

Das in diesem Artikel geteilte Site-Migrationsschema stammt aus unserer praktischen Erfahrung im [HagiCode](https://github.com/HagiCode-org/site)-Projekt.

HagiCode ist ein KI-Kode-Assistent, der sich der Verbesserung der Entwicklungseffizienz widmet. Wir konzentrieren uns nicht nur auf die Iteration der Kernfunktionen, sondern legen gleichen Wert auf die Developer Experience. Dieses Refactoring der Site dient dazu, Benutzern beim Durchsuchen der Dokumentation und der offiziellen Website extreme Ladegeschwindigkeit zu bieten.

## Warum das etablierte Docusaurus aufgeben?

Im React-Ökosystem war Docusaurus immer die "Standardantwort" für Dokumentations-Sites. Es ist sofort einsatzbereit, hat reichhaltige Plugins und eine aktive Community. Aber mit dem Wachstum der HagiCode-Funktionen spürten wir auch seine Grenzen:

1.  **Leistungsengpass**: Docusaurus ist im Wesentlichen eine React-SPA (Single-Page-Anwendung). Selbst wenn Sie rein statische Seiten schreiben, muss der Client die React-Runtime laden und hydrieren, was für einfache Dokumentationsseiten zu schwer ist.
2.  **Ressourcenvolumen**: Selbst wenn der Seiteninhalt sehr gering ist, ist das gebündelte JS-Volumen relativ fest, was für Mobilnutzer und Umgebungen mit schlechter Netzwerkverbindung nicht freundlich ist.
3.  **Unzureichende Flexibilität**: Obwohl Erweiterungen möglich sind, wollten wir bei der Anpassung des Build-Prozesses mehr grundlegende Kontrolle.

Die appearance von Astro löst diese Probleme genau. Es bietet eine brandneue "Islands-Architektur": Standardmäßig generiert Astro statisches HTML mit null JavaScript; nur Komponenten, die Interaktion benötigen, werden "aktiviert" und laden JS. Das bedeutet, dass der Großteil unserer Site aus reinem HTML besteht und extrem schnell ist.

## Kernstrategie der Migration: Architektur-Übergang glatt gestalten

Migration ist kein einfaches Kopieren und Einfügen, sondern ein Wandel des Denkens. Wir wechselten von Docusaurus "vollständiger React-Modus" zu Astros "Core + Islands"-Modus.

### 1. Refactoring des Konfigurationssystems

Zuerst mussten wir von `docusaurus.config.ts` zu `astro.config.mjs` übergehen. Dies ist nicht nur eine Änderung des Dateinamens, sondern ein Umschreiben von Routing- und Build-Logik.

In Docusaurus ist alles ein Plugin; in Astro ist alles eine Integration. Wir müssen den Basispfad der Site, den Build-Ausgabemodus (statisch vs. SSR) und die Ressourcenkomprimierungsstrategie neu definieren.

**Vor der Migration:**
```typescript
// docusaurus.config.ts
export default {
  title: 'HagiCode',
  url: 'https://hagicode.com',
  baseUrl: '/',
  // ... mehr Konfigurationen
};
```

**Nach der Migration:**
```javascript
// astro.config.mjs
import { defineConfig } from 'astro/config';
import react from '@astrojs/react';

export default defineConfig({
  integrations: [react()],
  site: 'https://hagicode.com',
  base: '/',
  // Optimierungskonfiguration für statische Ressourcen
  build: {
    inlineStylesheets: 'auto',
  },
});
```

### 2. Ob und wie React-Komponenten umgebaut werden

Dies war der kopfschmerzendste Teil der Migration. Unsere bestehende Site hatte viele React-Komponenten (wie Tabs-Komponente, Code-Highlighting, Feedback-Buttons usw.). Einfach wegzuwerfen wäre zu schade; alle zu behalten würde zu hoher JS-Last führen.

HagiCode übernahm eine **strategie der progressiven Hydration**:
*   **Rein statische Komponenten**: Für Präsentationsinhalte (wie Header, Footer, rein Text-Dokumentation) als Astro-Komponenten (`.astro`-Dateien) neu schreiben, die beim Build direkt in HTML gerendert werden.
*   **Interaktive Inseln**: Für Komponenten, die Interaktion erfordern müssen (wie Theme-Switcher, Tabs-Umschaltung, Codeblock-Kopier-Button), React-Implementierung behalten und `client:load`- oder `client:visible`-Direktiven hinzufügen.

Zum Beispiel unsere oft verwendete Tabs-Komponente in der Dokumentation:

```jsx
// src/components/Tabs.jsx
import { useState } from 'react';
import './Tabs.css'; // Styles importieren

export default function Tabs({ items }) {
  const [activeIndex, setActiveIndex] = useState(0);
  // ... Zustandslogik
  return (
    <div className="tabs-wrapper">
      {/* Renderlogik */}
    </div>
  );
}
```

Bei der Verwendung in Markdown sagen wir Astro explizit: "Diese Komponente braucht JS":

```markdown
// src/content/docs/example.mdx
import Tabs from '../../components/Tabs.jsx';

<!-- JS nur laden, wenn die Komponente in den sichtbaren Bereich eintritt -->
<Tabs client:visible items={...} />
```

So beanspruchen interaktive Komponenten außerhalb des sichtbaren Bereichs keine Bandbreite, was die First-Screen-Ladegeschwindigkeit extrem optimiert.

### 3. Anpassung des Stylesystems: Von CSS Modules zu Scoped

Docusaurus unterstützt CSS Modules standardmäßig, während Astro die Verwendung von Scoped CSS (durch `<style>`-Tags) bevorzugt. Beide haben den gleichen Kerngedanken der Stilisolierung, aber unterschiedliche Syntax.

In der HagiCode-Migration zerlegten wir die meisten komplexen CSS Modules in Astros Scoped Styles. Das ist eigentlich eine gute Sache, da in `.astro`-Dateien Styles und Templates in derselben Datei geschrieben werden, was die Wartung intuitiver macht.

**Vor dem Umbau:**
```css
/* Tabs.module.css */
.wrapper { background: var(--ifm-background-color); }
```

**Nach dem Umbau (Astro Scoped):**
```html
<!-- Tabs.astro -->
<div class="tabs-wrapper">
  <slot />
</div>

<style>
  .tabs-wrapper {
    /* CSS-Variablen direkt verwenden, Theme anpassen */
    background: var(--bg-color);
    padding: 1rem;
  }
</style>
```

Gleichzeitig vereinheitlichten wir das globale CSS-Variablensystem und nutzten Astros Umgebungsawareness, um einen nahtlosen Übergang des Dark Mode zwischen verschiedenen Seiten sicherzustellen.

## Praktische Fallstricke und Lösungen

Im tatsächlichen Migrationsprozess von HagiCode stießen wir auf viele Fallstricke. Hier teile ich einige der typischsten.

### 1. Schmerzpunkte bei Pfaden und Umgebungsvariablen

HagiCode unterstützt Subpfad-Bereitstellung (z. B. Bereitstellung in einem Unterverzeichnis von GitHub Pages). In Docusaurus wird dies automatisch durch `baseUrl` verarbeitet. In Astro müssen wir bei der Handhabung von Bildlinks und API-Anfragen vorsichtiger sein.

Wir führten einen Umgebungsvariablen-Mechanismus ein, um dies einheitlich zu verwalten:

```javascript
// Pfade im Build-Skript verarbeiten
const getBasePath = () => import.meta.env.VITE_SITE_BASE || '/';
```

Denken Sie daran, keine Pfade, die mit `/` beginnen, im Code hartzucodieren. In Entwicklungs- und Produktionsumgebung oder nach Konfiguration des `base`-Pfads führt dies zu Ressourcen-404.

### 2. Kompatibilität von CommonJS-Skripten

Unsere alte Site hatte einige Node.js-Skripte (für automatisches Abrufen von Metrics-Daten, Aktualisieren von Sitemaps usw.), die in CommonJS (`require`) geschrieben waren. Astro und moderne Build-Tools umarmen vollständig ES Modules (`import`/`export`).

Wenn Sie ähnliche Skripte haben, denken Sie daran, sie alle zu ES Modules zu refaktorieren. Das ist der allgemeine Trend; früher umgebaut, früher sorgenfrei.

```javascript
// Alte Weise
const fs = require('fs');

// Neue Weise
import fs from 'fs';
```

### 3. Vergessen Sie SEO und Redirects nicht

Suchmaschinen haben bereits die alten Docusaurus-Seiten von HagiCode indexiert. Wenn direkt zu Astro gewechselt wird und sich die URL-Struktur ändert, führt dies zu vielen 404s und einem massiven Gewichtsverlust.

Wir konfigurierten Redirect-Regeln in Astro:

```javascript
// astro.config.mjs
export default defineConfig({
  redirects: {
    '/docs/old-path': '/docs/new-path',
    // Batch-Mapping alter Links zu neuen Links
  }
});
```
Oder auf Server-Konfigurationsebene verarbeiten. Stellen Sie sicher, dass alte Links 301 zu neuen Adressen redirecten; dies ist für SEO entscheidend.

## Zusammenfassung

Die Migration von Docusaurus zu Astro war für HagiCode nicht nur ein Framework-Upgrade, sondern eine Praxis der Philosophie "Leistung zuerst".

**Unsere Gewinne:**
*   **Extreme Lighthouse-Punktzahlen**: Nach der Migration nähert sich die Leistungsbewertung der HagiCode-Site problemlos der vollen Punktzahl.
*   **Schnellere Build-Geschwindigkeit**: Astros inkrementelles Build halbierte die Veröffentlichungszeit unserer Dokumentations-Updates.
*   **Flexibilität behalten**: Durch die Islands-Architektur opferten wir keine interaktiven Funktionen und können immer noch React dort verwenden, wo wir es brauchen.

Wenn Sie auch eine dokumentationsorientierte Site warten und von Paketgröße oder Ladegeschwindigkeit geplagt werden, probieren Sie Astro. Obwohl der Migrationsprozess einige chirurgische Eingriffe erfordert (z. B. den Namen von PCode in HagiCode ändern, Komponenten einzeln ausgraben), erhalten Sie eine seidige Benutzererfahrung, die absolut lohnenswert ist.

Das in diesem Artikel geteilte Build-System ist genau das Schema, das wir bei der Entwicklung von HagiCode durch tatsächliche Fallstricke und tatsächliche Optimierung entwickelt haben. Wenn Sie dieses Schema wertvoll finden, zeigt das, dass unsere Ingenieursfähigkeiten ganz ordentlich sind – dann ist HagiCode selbst auch einen Blick wert.
