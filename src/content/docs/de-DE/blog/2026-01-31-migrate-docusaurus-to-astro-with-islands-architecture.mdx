## Von Docusaurus 3.x zu Astro 5.x: Rückblick auf die HagiCode-Seitenmigration

> Dieser Artikel blickt auf unsere vollständige Migration der HagiCode-Website von Docusaurus 3.x zu Astro 5.x zurück. Wir beleuchten im Detail, wie Astros Islands-Architektur uns geholfen hat, Leistungsengpässe zu beseitigen und gleichzeitig unsere bestehenden React-Komponenten zu erhalten, wodurch wir sowohl die Build-Geschwindigkeit als auch die Ladeleistung verbessern konnten.

## Hintergrund

Im Januar 2026 führten wir an der HagiCode-Website eine „Herztransplantation" durch — die vollständige Migration des Kernframeworks von Docusaurus 3.x zu Astro 5.x. Das war keine impulsiv durchgeführte Neuentwicklung, sondern eine sorgfältig abgewogene technische Entscheidung.

Vor der Migration war unsere Website funktional vollständig, zeigte aber zunehmend einige klassische „Luxusprobleme": Aufgeblähte Build-Artefakte, übermäßige JavaScript-Payloads und weniger als ideale Seitenladezeiten bei komplexen Dokumentationsseiten. Als KI-Coding-Assistant-Projekt benötigt HagiCode häufige Dokumentations- und Feature-Updates, weshalb die Build-Effizienz direkt die Veröffentlichungsgeschwindigkeit beeinflusst. Gleichzeitig wollten wir die Website suchmaschinenfreundlicher gestalten (SEO), damit mehr Entwickler das Projekt entdecken.

Um diese Schmerzpunkte zu beheben, trafen wir eine gewagte Entscheidung: Den gesamten Systemaufbau auf Astro neu erstellen. Die Auswirkungen dieser Entscheidung waren größer, als man erwarten würde — dazu gleich mehr.

## Über HagiCode

Der in diesem Artikel geteilte Migrationsansatz stammt aus unserer praktischen Erfahrung im [HagiCode](https://github.com/HagiCode-org/site)-Projekt.

HagiCode ist ein KI-Coding-Assistent mit Fokus auf die Verbesserung der Entwicklungseffizienz. Wir kümmern uns nicht nur um die Weiterentwicklung der Kernfunktionen, sondern legen ebenso großen Wert auf die Entwicklererfahrung. Dieses Website-Refactoring diente auch dazu, Nutzern beim Durchsuchen der Dokumentation und der offiziellen Website ein möglichst schnelles Erlebnis zu bieten.

## Warum das reife Docusaurus-Ökosystem verlassen?

Im React-Ökosystem war Docusaurus lange die „Standardantwort" für Dokumentationswebsites. Es funktioniert sofort, bietet ein reichhaltiges Plugin-Ökosystem und hat eine aktive Community. Doch mit dem Wachstum von HagiCode spürten wir auch seine Grenzen:

1.  **Leistungsengpässe**: Docusaurus ist im Kern eine React SPA (Single-Page-Application). Selbst wenn Sie nur statische Seiten schreiben, muss der Client die React-Runtime laden und die Seite hydratisieren — das ist für einfache Dokumentationsseiten unnötig schwer.
2.  **Große Asset-Größe**: Selbst bei sehr wenig Seiteninhalt bleibt die gebündelte JS-Größe relativ konstant. Das ist für mobile Nutzer oder bei schlechten Netzwerkbedingungen nicht ideal.
3.  **Eingeschränkte Flexibilität**: Obwohl es erweiterbar ist, wollten wir mehr Low-Level-Kontrolle über die Build-Pipeline.

Astro kam genau richtig, um diese Probleme zu lösen. Es führte eine neue „Islands-Architektur" ein: Standardmäßig generiert Astro statisches HTML ohne JavaScript, und nur Komponenten, die Interaktivität erfordern, werden „aktiviert" und laden JS. Das bedeutet, dass der Großteil unserer Website reines HTML ist und extrem schnell lädt.

## Kernmigrationsstrategie: Ein sanfter architektonischer Übergang

Migration bedeutete nicht einfach Kopieren und Einfügen, sondern einen Wechsel des Denkmodells. Wir bewegten uns von Docusaurus' „Alles-React"-Modell zu Astros „Core + Islands"-Modell.

### 1. Neuaufbau des Konfigurationssystems

Zuerst mussten wir von `docusaurus.config.ts` zu `astro.config.mjs` wechseln. Das war nicht nur eine Dateiumbenennung, sondern ein Neuschreiben der Routing- und Build-Logik.

In Docusaurus ist alles ein Plugin. In Astro ist alles eine Integration. Wir mussten den Basispfad der Website, den Build-Ausgabemodus (statisch vs. SSR) und die Asset-Optimierungsstrategie neu definieren.

**Vor der Migration:**
```typescript
// docusaurus.config.ts
export default {
  title: 'HagiCode',
  url: 'https://hagicode.com',
  baseUrl: '/',
  // ... weitere Konfiguration
};
```

**Nach der Migration:**
```javascript
// astro.config.mjs
import { defineConfig } from 'astro/config';
import react from '@astrojs/react';

export default defineConfig({
  integrations: [react()],
  site: 'https://hagicode.com',
  base: '/',
  // Optimierungseinstellungen für statische Assets
  build: {
    inlineStylesheets: 'auto',
  },
});
```

### 2. Was an React-Komponenten behalten und was refaktoriert werden sollte

Das war der schmerzhafteste Teil der Migration. Unsere bestehende Website hatte viele React-Komponenten wie Tabs, Code-Highlighting, Feedback-Buttons und mehr. Sie einfach wegzuwerfen wäre verschwenderisch gewesen, alles zu behalten hätte den JavaScript-Payload zu schwer gemacht.

HagiCode verfolgte eine **progressive Hydration**-Strategie:
*   **Reine statische Komponenten**: Für Präsentationsinhalte wie Header, Footer und reine Textdokumentation schrieben wir sie als Astro-Komponenten (`.astro`-Dateien) neu und rendern sie zur Build-Zeit direkt als HTML.
*   **Interaktive Inseln**: Für Komponenten, die interaktiv bleiben müssen, wie Theme-Switcher, Tab-Umschaltung und Code-Block-Kopierbuttons, behielten wir die React-Implementierung und fügt `client:load`- oder `client:visible`-Direktiven hinzu.

Beispielsweise unsere häufig verwendete Tabs-Komponente in der Dokumentation:

```jsx
// src/components/Tabs.jsx
import { useState } from 'react';
import './Tabs.css'; // Styles importieren

export default function Tabs({ items }) {
  const [activeIndex, setActiveIndex] = useState(0);
  // ... Zustandslogik
  return (
    <div className="tabs-wrapper">
      {/* Rendering-Logik */}
    </div>
  );
}
```

Bei der Verwendung in Markdown sagen wir Astro explizit: „Diese Komponente benötigt JS."

```markdown
// src/content/docs/example.mdx
import Tabs from '../../components/Tabs.jsx';

<!-- JS nur laden, wenn die Komponente in den Viewport eintritt -->
<Tabs client:visible items={...} />
```

So konkurrieren interaktive Komponenten außerhalb des Viewports nicht um Bandbreite, was die First-Screen-Ladezeit erheblich verbessert.

### 3. Anpassung des Stylesystems: Von CSS Modules zu Scoped CSS

Docusaurus unterstützt standardmäßig CSS Modules, während Astro Scoped CSS über das `<style>`-Tag bevorzugt. Der Grundgedanke beider Ansätze ist die Style-Isolation, jedoch unterscheidet sich die Syntax.

Während der HagiCode-Migration wandelten wir die meisten komplexen CSS Modules in Astros Scoped Styles um. Das stellte sich als vorteilhaft heraus, da in `.astro`-Dateien Styles und Templates in derselben Datei liegen, was die Wartung intuitiver macht.

**Vor dem Refactoring:**
```css
/* Tabs.module.css */
.wrapper { background: var(--ifm-background-color); }
```

**Nach dem Refactoring (Astro Scoped):**
```html
<!-- Tabs.astro -->
<div class="tabs-wrapper">
  <slot />
</div>

<style>
  .tabs-wrapper {
    /* CSS-Variablen direkt verwenden, themenangepasst */
    background: var(--bg-color);
    padding: 1rem;
  }
</style>
```

Gleichzeitig vereinheitlichten wir das globale CSS-Variablensystem und nutzten Astros Umgebungserkennung, um sicherzustellen, dass der Dark Mode reibungslos zwischen Seiten wechselt.

## Fallstricke in der Praxis und deren Lösung

Während der tatsächlichen HagiCode-Migration stießen wir auf einige Probleme. Hier sind die typischsten davon.

### 1. Pfad- und Umgebungsvariable-Probleme

HagiCode unterstützt die Subpfad-Bereitstellung, z. B. unter einem GitHub Pages-Unterverzeichnis. In Docusaurus wird `baseUrl` automatisch behandelt. In Astro müssen wir bei der Verarbeitung von Bildlinks und API-Anfragen jedoch vorsichtiger sein.

Wir führten einen Umgebungsvariablenmechanismus zur einheitlichen Verwaltung ein:

```javascript
// Pfade im Build-Skript behandeln
const getBasePath = () => import.meta.env.VITE_SITE_BASE || '/';
```

Bitte vermeiden Sie es, Pfade mit führendem `/` im Code hart zu codieren. In der Entwicklungs- versus Produktionsumgebung oder nach der Konfiguration eines `base`-Pfads kann dies zu 404-Fehlern für Assets führen.

### 2. CommonJS-Skriptkompatibilität

Unsere alte Website verfügte über einige Node.js-Skripte für Aufgaben wie das automatische Abrufen von Metriken und das Aktualisieren der Sitemap, die in CommonJS (`require`) geschrieben waren. Astro und moderne Build-Tools haben sich vollständig ES Modules (`import`/`export`) zugewandt.

Wenn Sie ähnliche Skripte haben, denken Sie daran, sie alle auf ES Modules umzustellen. Das ist der Trend im Ökosystem — je früher Sie wechseln, desto weniger Probleme haben Sie später.

```javascript
// Alter Weg
const fs = require('fs');

// Neuer Weg
import fs from 'fs';
```

### 3. SEO und Weiterleitungen nicht vergessen

Suchmaschinen haben die alten Docusaurus-Seiten von HagiCode bereits indiziert. Wenn Sie direkt zu Astro wechseln und sich die URL-Struktur ändert, erhalten Sie viele 404-Fehler und einen erheblichen Ranking-Verlust.

Wir konfigurierten Weiterleitungsregeln in Astro:

```javascript
// astro.config.mjs
export default defineConfig({
  redirects: {
    '/docs/old-path': '/docs/new-path',
    // Alte Links massenhaft auf neue Links abbilden
  }
});
```

Alternativ können Sie dies auf Server-Konfigurationsebene behandeln. Stellen Sie sicher, dass alte Links per 301-Weiterleitung auf die neuen Adressen umgeleitet werden — das ist für SEO entscheidend.

## Zusammenfassung

Für HagiCode war die Migration von Docusaurus zu Astro nicht nur ein Framework-Upgrade, sondern auch die praktische Umsetzung einer „Leistung-zuerst"-Philosophie.

**Was wir gewonnen haben:**
*   **Hervorragende Lighthouse-Ergebnisse**: Nach der Migration erreichte die Leistungsbewertung der HagiCode-Website fast die volle Punktzahl.
*   **Schnellere Build-Geschwindigkeit**: Astros inkrementelle Build-Fähigkeiten halbierten die Veröffentlichungszeit für Dokumentations-Updates.
*   **Bewahrte Flexibilität**: Mit der Islands-Architektur opferten wir keine interaktiven Funktionen und konnten weiterhin React dort verwenden, wo es benötigt wurde.

Wenn Sie ebenfalls eine dokumentationsorientierte Website pflegen und mit Bundle-Größe oder Ladezeiten kämpfen, ist Astro einen Versuch wert. Zwar erfordert der Migrationsprozess einigen Aufwand, aber die geschmeidige Benutzererfahrung, die Sie dafür erhalten, ist es absolut wert.

Das in diesem Artikel geteilte Build-System ist der genaue Ansatz, den wir durch reale Versuche und Irrtümer während der Entwicklung von HagiCode entwickelt haben. Wenn Sie diesen Ansatz wertvoll finden, spricht das für unsere Engineering-Stärke — und HagiCode selbst ist es wahrscheinlich ebenfalls wert, genauer betrachtet zu werden.
