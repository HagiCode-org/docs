## Von TypeScript zu C#: Praxis der sprachuebergreifenden Portierung des Codex SDK

> Einfach gesagt, dieser Artikel ist auch ein bisschen unser Kind: Er dokumentiert den vollstaendigen Prozess der Portierung des offiziellen TypeScript Codex SDK nach C#. Es als ,,Portierung`` zu bezeichnen klingt fast zu einfach -- es war eher ein langes Abenteuer, denn diese beiden Sprachen haben sehr unterschiedliche Eigenheiten, und wir mussten einen Weg finden, sie zusammenarbeiten zu lassen.

## Hintergrund

Codex ist das von OpenAI veroeffentlichte AI Agent CLI-Tool, und es ist wirklich maechtig. Das offizielle Team stellt ein TypeScript SDK im Paket `@openai/codex` bereit. Es kommuniziert mit dem Codex CLI, indem es den Befehl `codex exec --experimental-json` aufruft und einen JSONL-Ereignisstrom parst.

Das Problem ist, dass wir im HagiCode-Projekt eine reine .NET-Umgebung benoetigen -- konkret in C#-Backend-Diensten und Desktop-Anwendungen. Es waere unzumutbar, in ein .NET-Projekt eine Node.js-Laufzeitumgebung einzufuehren, nur um ein CLI-Tool aufzurufen.

So blieben uns zwei Moeglichkeiten: Eine komplexe Node.js-Bridge-Schicht pflegen oder einen nativen C#-SDK selbst entwickeln.

Wir haben uns fuer Letzteres entschieden.

## Ueber HagiCode

Dieser Artikel stammt ebenfalls aus unserer praktischen Erfahrung im [HagiCode](https://hagicode.com)-Projekt. HagiCode ist ein Open-Source-AI-Code-Assistent-Projekt. Es bedeutet, gleichzeitig mehrere Komponenten zu pflegen: eine VSCode-Erweiterung im Frontend, AI-Dienste im Backend und einen plattformuebergreifenden Desktop-Client. Diese komplexe Mehrsprachen- und Mehrplattformarchitektur ist genau der Grund, warum wir einen nativen C#-SDK benoetigten.

## Kerninhalt

### Architekturvergleich

Bevor wir Code auf Code uebertragen, muessen wir zunaechst die Architektur beider SDKs verstehen.

Die Kernarchitektur des TypeScript SDK sieht so aus:

```
Codex (Einstiegsklasse)
  └── CodexExec (Executor, verwaltet Kindprozesse)
      └── Thread (Konversationsthread)
          ├── run() / runStreamed() (synchrone/asynchrone Ausfuehrung)
          └── Ereignisstrom-Parsing
```

Das C#-SDK behaelt dieselbe Architekturebene bei, passt jedoch die Implementierungsdetails an. Die Grundidee: API-Konsistenz bewahren und gleichzeitig die C#-Sprachmerkmale voll ausnutzen.

### Typsystemkonvertierung

Dies ist der grundlegendste und wichtigste Teil der Arbeit.

TypeScripts Typsystem ist flexibler als das von C#. Wir mussten eine geeignete Abbildungsstrategie finden:

| TypeScript | C# | Hinweise |
|------------|-----|------|
| `interface` / `type` | `record` | C# verwendet record fuer unveranderliche Datenstrukturen |
| `string \| null` | `string?` | Nullable Referenztyp |
| `boolean \| undefined` | `bool?` | Nullable Boolean |
| `AsyncGenerator` | `IAsyncEnumerable` | Async-Iterator |

Das **Ereignistyp-System** ist ein typisches Beispiel. TypeScript verwendet Vereinigungstypen:

```typescript
export type ThreadEvent =
  | ThreadStartedEvent
  | TurnStartedEvent
  | TurnCompletedEvent
  | ...
```

In C# verwenden wir eine Vererbungshierarchie und Musterabgleich:

```csharp
public abstract record ThreadEvent(string Type);

public sealed record ThreadStartedEvent(string ThreadId) : ThreadEvent("thread.started");
public sealed record TurnStartedEvent() : ThreadEvent("turn.started");
public sealed record TurnCompletedEvent(Usage Usage) : ThreadEvent("turn.completed");
// ...
```

Wir haben `record` gewaehlt, da Ereignisobjekte unveranderlich sein sollten. `sealed` verhindert zusaetzliche Vererbung.

### Wesentliche Portierungspunkte

#### 1. Ereignisparser

Das Parsen von Ereignissen ist der Kern des gesamten SDK. Die TypeScript-Version verwendet `JSON.parse()`:

```typescript
export function parseEvent(line: string): ThreadEvent {
  const data = JSON.parse(line);
  // Verschiedene Ereignistypen verarbeiten...
}
```

Die C#-Version verwendet `System.Text.Json.JsonDocument`:

```csharp
public static ThreadEvent Parse(string line)
{
    using var document = JsonDocument.Parse(line);
    var root = document.RootElement;
    var type = GetRequiredString(root, "type", "event.type");

    return type switch
    {
        "thread.started" => new ThreadStartedEvent(GetRequiredString(root, "thread_id", ...)),
        "turn.started" => new TurnStartedEvent(),
        "turn.completed" => new TurnCompletedEvent(ParseUsage(...)),
        _ => new UnknownThreadEvent(type, root.Clone()),
    };
}
```

Ein wichtiger Unterschied: `root.Clone()` ist erforderlich, da Elemente eines `JsonDocument` nach der Freigabe ungueltig werden.

#### 2. Prozessverwaltungsunterschiede

TypeScript verwendet die `spawn()`-Funktion von Node.js:

```typescript
const child = spawn(this.executablePath, commandArgs, { env, signal });
```

C# verwendet `System.Diagnostics.Process`:

```csharp
using var process = new Process { StartInfo = startInfo };
process.Start();
```

Der groesste Unterschied liegt im Abbruchmechanismus. TypeScript verwendet `AbortSignal`, C# verwendet `CancellationToken`:

```csharp
public async IAsyncEnumerable<string> RunAsync(
    CodexExecArgs args,
    [EnumeratorCancellation] CancellationToken cancellationToken = default)
{
    while (!cancellationToken.IsCancellationRequested)
    {
        // Ausgabe verarbeiten...
    }

    if (cancellationToken.IsCancellationRequested)
    {
        try { process.Kill(entireProcessTree: true); } catch { }
    }
}
```

#### 3. Beibehaltung der Konfigurationsserialisierung

Beide SDKs implementieren die Logik zur Konvertierung von JSON-Konfiguration in TOML-Konfiguration, da der Codex CLI Konfigurationsoptionen im TOML-Format akzeptiert.

### Projektstruktur

```
CodexSdk/
├── CodexSdk.csproj
├── Codex.cs           # Einstiegsklasse
├── CodexThread.cs     # Konversationsthread
├── CodexExec.cs       # Executor
├── Events.cs          # Ereignistypdefinitionen
├── Items.cs           # Elementtypdefinitionen
├── EventParser.cs     # Ereignisparser
└── ...
```

### Verwendungsbeispiel

```csharp
using CodexSdk;

var codex = new Codex();
var thread = codex.StartThread();

var result = await thread.RunAsync("Summarize this repository.");
Console.WriteLine(result.FinalResponse);
```

Streaming-Ereignisse:

```csharp
await foreach (var @event in thread.RunStreamedAsync("Analyze the code."))
{
    switch (@event)
    {
        case ItemCompletedEvent itemCompleted
            when itemCompleted.Item is AgentMessageItem msg:
            Console.WriteLine($"Assistant: {msg.Text}");
            break;
        case TurnCompletedEvent completed:
            Console.WriteLine($"Tokens: in={completed.Usage.InputTokens}");
            break;
    }
}
```

### Hinweise

1. **Prozessverwaltung**: Die C#-Version muss den vollstaendigen Prozesslebenszyklus manuell verwalten. Verwenden Sie `Kill(entireProcessTree: true)`.
2. **Fehlerbehandlung**: Wir verwenden `InvalidOperationException` fuer Parsing-Fehler.
3. **Ressourcenbereinigung**: `OutputSchemaTempFile` implementiert `IAsyncDisposable`.
4. **Umgebungsvariablen**: Die C#-Version unterstuetzt das vollstaendige Ueberschreiben von Prozessumgebungsvariablen.
5. **Plattformunterschiede**: Der Pfad zur Codex-Ausfuehrbaren Datei muss ueber `CODEX_EXECUTABLE` oder `CodexPathOverride` angegeben werden.

## Zusammenfassung

Die Portierung eines ausgereiften TypeScript SDK nach C# ist nicht nur eine Syntaxkonvertierung -- sie erfordert auch das Verstaendnis der Designphilosophie beider Sprachen. Die wichtigste Erkenntnis: **Die Beibehaltung der API-Konsistenz ist wichtiger als die Beibehaltung der Implementierungskonsistenz**.

Wenn Sie an einer aehnlichen Portierung arbeiten: Zunaechst die urspruengliche SDK-Architektur vollstaendig verstehen, dann Modul fuer Modul uebertragen und schliesslich durch Testfaelle die Verhaltenskonsistenz sicherstellen.
