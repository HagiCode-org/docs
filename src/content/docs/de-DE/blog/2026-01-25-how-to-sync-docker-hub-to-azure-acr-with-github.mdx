## Automatische Images-Synchronisation von Docker Hub zu Azure ACR

> Dieser Artikel erklärt, wie man GitHub Actions und das image-syncer-Tool nutzt, um die Image-Synchronisation von Docker Hub zu Azure Container Registry zu automatisieren. Damit wird das Problem langsamer Docker Hub-Zugriffe in einigen Regionen und Azure-Bereichen gelöst, während die Verfügbarkeit der Images und die Bereitstellungseffizienz in Azure-Umgebungen verbessert werden.

## Hintergrund / Einleitung

Das HagiCode-Projekt verwendet Docker-Images als Kern-Runtime-Komponenten, wobei die Hauptimages auf Docker Hub gehostet werden. Mit dem Wachstum des Projekts und steigenden Azure-Bereitstellungsanforderungen stellten wir folgende Probleme fest:
- Langsame Image-Pulls, da der Zugriff auf Docker Hub in einigen Regionen und Azure-Bereichen eingeschränkt ist
- Die Abhängigkeit von einer einzigen Image-Quelle birgt ein Single-Point-of-Failure-Risiko
- Die Verwendung von Azure Container Registry in Azure-Umgebungen bietet eine bessere Netzwerkleistung und Integration

Um diese Probleme zu lösen, mussten wir einen automatisierten Image-Synchronisationsmechanismus einrichten, der regelmäßig Images von Docker Hub zu Azure ACR synchronisiert, damit Benutzer in Azure-Umgebungen schnellere Pull-Geschwindigkeiten und höhere Verfügbarkeit erhalten.

## Vergleich der technischen Lösungen

Bei der Definition der Lösung verglichen wir mehrere technische Ansätze:

### 1. image-syncer (endgültige Wahl)
- Inkrementelle Synchronisation: Nur geänderte Image-Layer werden synchronisiert, was die Netzwerkübertragung deutlich reduziert
- Fortsetzungsunterstützung: Die Synchronisation kann nach Netzwerkunterbrechungen fortgesetzt werden
- Parallelitätssteuerung: Unterstützt konfigurierbare Parallelität zur Verbesserung der Effizienz bei großen Images
- Robuste Fehlerbehandlung: Integrierter Wiederholungsmechanismus (standardmäßig 3 Versuche)
- Leichtgewichtige Bereitstellung: Einzelne Binärdatei ohne Abhängigkeiten
- Multi-Registry-Unterstützung: Kompatibel mit Docker Hub, Azure ACR, Harbor und mehr

### 2. Docker CLI
- Keine inkrementelle Synchronisation: Jeder Durchlauf erfordert das vollständige Pullen des Image-Inhalts
- Geringere Effizienz: Hohe Netzwerkübertragungsmenge und längere Ausführungszeit
- Einfach und benutzerfreundlich: Basiert auf den vertrauten `docker pull`/`docker push`-Befehlen

### 3. Azure CLI
- Höhere Komplexität: Erfordert die Einrichtung der Azure CLI-Authentifizierung
- Funktionale Einschränkungen: `az acr import` ist relativ begrenzt
- Native Integration: Gut integriert in Azure-Dienste

## Architekturentscheidungen

### Entscheidung 1: Synchronisationsfrequenz auf täglich 00:00 UTC festgelegt
- Ausgewogenes Verhältnis zwischen Image-Aktualität und Ressourcenverbrauch
- Vermeidung von Geschäftszeiten, um Auswirkungen auf andere Vorgänge zu minimieren
- Docker Hub-Images werden in der Regel nach täglichen Builds aktualisiert

### Entscheidung 2: Alle Image-Tags synchronisieren
- Aufrechterhaltung der vollen Konsistenz mit Docker Hub
- Flexible Versionsauswahl für Benutzer
- Vereinfachung der Synchronisationslogik ohne komplexe Tag-Filterregeln

### Entscheidung 3: Anmeldeinformationen in GitHub Secrets speichern
- Native Unterstützung durch GitHub Actions mit hoher Sicherheit
- Einfach zu konfigurieren, leicht zu verwalten und zu warten
- Unterstützung der Repository-Level-Zugriffskontrolle

## Risikobewertung und Minderung

### Risiko 1: Preisgabe von Azure ACR-Anmeldeinformationen
- Verschlüsselte Speicherung in GitHub Secrets
- Regelmäßiger Wechsel der ACR-Passwörter
- Beschränkung der ACR-Kontoberechtigungen auf Nur-Push
- Überwachung der ACR-Zugriffsprotokolle

### Risiko 2: Synchronisationsfehler führen zu Image-Inkonsistenz
- image-syncer enthält einen integrierten inkrementellen Synchronisationsmechanismus
- Automatischer Wiederholungsversuch bei Fehlern (standardmäßig 3-mal)
- Detaillierte Fehlerprotokolle und Fehlerbenachrichtigungen
- Fortsetzungsunterstützung

### Risiko 3: Übermäßiger Ressourcenverbrauch
- Inkrementelle Synchronisation reduziert die Netzwerkübertragung
- Konfigurierbare Parallelität (aktuell auf 10 eingestellt)
- Überwachung der Anzahl und Größe der synchronisierten Images
- Synchronisation während Schwachlastzeiten

## Kernlösung

Wir verwenden eine automatisierte GitHub Actions + image-syncer-Lösung zur Synchronisation von Images von Docker Hub zu Azure ACR.

## Implementierungsschritte

### 1. Vorbereitung
- Azure Container Registry im Azure Portal erstellen oder bestätigen
- ACR-Zugangsdaten erstellen (Benutzername und Passwort)
- Zugriffsrechte für das Docker Hub-Image-Repository bestätigen

### 2. GitHub Secrets konfigurieren
Folgende Secrets in den GitHub-Repository-Einstellungen hinzufügen:
- AZURE_ACR_USERNAME: Azure ACR-Benutzername
- AZURE_ACR_PASSWORD: Azure ACR-Passwort

### 3. GitHub Actions-Workflow erstellen
Workflow in `.github/workflows/sync-docker-acr.yml` konfigurieren:
- Geplanter Trigger: Jeden Tag um 00:00 UTC
- Manueller Trigger: Unterstützt `workflow_dispatch`
- Zusätzlicher Trigger: Läuft bei Push auf den `publish`-Branch (für schnelle Synchronisation)

### 4. Workflow-Ausführungsablauf

| Reihenfolge | Teilnehmer | Aktion | Beschreibung |
| --- | --- | --- | --- |
| 1 | GitHub Actions | Workflow auslösen | Ausgelöst durch Zeitplan, manuellen Start oder Push auf den `publish`-Branch |
| 2 | GitHub Actions → image-syncer | Sync-Tool herunterladen und ausführen | Eintritt in die tatsächliche Synchronisationsphase |
| 3 | image-syncer → Docker Hub | Image-Manifeste und Tag-Liste abrufen | Metadaten des Quell-Repositorys lesen |
| 4 | image-syncer → Azure ACR | Vorhandene Image-Informationen des Ziel-Repositorys abrufen | Aktuellen Zustand der Zielseite bestimmen |
| 5 | image-syncer | Quell- und Zieldifferenzen vergleichen | Zu synchronisierende Image-Layer identifizieren |
| 6 | image-syncer → Docker Hub | Geänderte Image-Layer pullen | Nur zu aktualisierende Inhalte übertragen |
| 7 | image-syncer → Azure ACR | Geänderte Image-Layer pushen | Inkrementelle Synchronisation abschließen |
| 8 | image-syncer → GitHub Actions | Synchronisationsstatistiken zurückgeben | Enthält Ergebnisse, Differenzen und Fehlerinformationen |
| 9 | GitHub Actions | Protokolle aufzeichnen und Artifacts hochladen | Nützlich für nachfolgende Audits und Fehlerbehebung |

## GitHub Actions-Workflow-Implementierung

Hier ist die tatsächliche Workflow-Konfiguration (`.github/workflows/sync-docker-acr.yml`):

```yaml
name: Sync Docker Image to Azure ACR

on:
  schedule:
    - cron: "0 0 * * *" # Jeden Tag um 00:00 UTC
  workflow_dispatch: # Manueller Trigger
  push:
    branches: [publish]

permissions:
  contents: read

jobs:
  sync:
    runs-on: ubuntu-latest

    steps:
      - name: Checkout code
        uses: actions/checkout@v4

      - name: Download image-syncer
        run: |
          # image-syncer-Binärdatei herunterladen
          wget https://github.com/AliyunContainerService/image-syncer/releases/download/v1.5.5/image-syncer-v1.5.5-linux-amd64.tar.gz
          tar -zxvf image-syncer-v1.5.5-linux-amd64.tar.gz
          chmod +x image-syncer

      - name: Create auth config
        run: |
          # Authentifizierungskonfigurationsdatei generieren (YAML-Format)
          cat > auth.yaml <<EOF
          hagicode.azurecr.io:
            username: "${{ secrets.AZURE_ACR_USERNAME }}"
            password: "${{ secrets.AZURE_ACR_PASSWORD }}"
          EOF

      - name: Create images config
        run: |
          # Image-Synchronisationskonfigurationsdatei generieren (YAML-Format)
          cat > images.yaml <<EOF
          docker.io/newbe36524/hagicode: hagicode.azurecr.io/hagicode
          EOF

      - name: Run image-syncer
        run: |
          # Synchronisation ausführen (mit den neueren --auth und --images Parametern)
          ./image-syncer --auth=./auth.yaml --images=./images.yaml --proc=10 --retries=3

      - name: Upload logs
        if: always()
        uses: actions/upload-artifact@v4
        with:
          name: sync-logs
          path: image-syncer-*.log
          retention-days: 7
```

## Konfigurationsdetails

### 1. Auslösebedingungen
- Geplanter Trigger: `cron: "0 0 * * *"` — läuft jeden Tag um 00:00 UTC
- Manueller Trigger: `workflow_dispatch` — ermöglicht manuelle Ausführung in der GitHub-Benutzeroberfläche
- Push-Trigger: `push: branches: [publish]` — wird bei Push auf den publish-Branch ausgelöst (für schnelle Synchronisation)

### 2. Authentifizierungskonfiguration (`auth.yaml`)
```yaml
hagicode.azurecr.io:
  username: "${{ secrets.AZURE_ACR_USERNAME }}"
  password: "${{ secrets.AZURE_ACR_PASSWORD }}"
```

### 3. Image-Synchronisationskonfiguration
```yaml
docker.io/newbe36524/hagicode: hagicode.azurecr.io/hagicode
```
Diese Konfiguration bedeutet, dass alle Tags von `docker.io/newbe36524/hagicode` zu `hagicode.azurecr.io/hagicode` synchronisiert werden.

### 4. image-syncer-Parameter
- `--auth=./auth.yaml`: Pfad zur Authentifizierungskonfigurationsdatei
- `--images=./images.yaml`: Pfad zur Image-Synchronisationskonfigurationsdatei
- `--proc=10`: Parallelität auf `10` setzen
- `--retries=3`: Bei Fehlern `3`-mal wiederholen

## Hinweise und Best Practices

### 1. Sicherheitsempfehlungen
- Azure ACR-Passwörter regelmäßig wechseln (empfohlen alle 90 Tage)
- Ein dediziertes ACR-Dienstkonto mit Nur-Push-Berechtigungen verwenden
- ACR-Zugriffsprotokolle überwachen, um anomale Zugriffe rechtzeitig zu erkennen
- Anmeldeinformationen nicht in Protokollen ausgeben
- Anmeldeinformationen nicht im Code-Repository committen

### 2. Leistungsoptimierung
- Den Parameter `--proc` anpassen: Parallelität basierend auf der Netzwerkbandbreite einstellen (empfohlen 5-20)
- Synchronisationszeit überwachen: Bei zu langer Dauer die Parallelität reduzieren
- Protokolle regelmäßig bereinigen: Einen sinnvollen `retention-days`-Wert festlegen (aktuell 7 Tage)

### 3. Fehlerbehebung

#### Problem 1: Authentifizierung fehlgeschlagen
```
Error: failed to authenticate to hagicode.azurecr.io
```
Lösung:
1. Prüfen, ob die GitHub Secrets korrekt konfiguriert sind
2. Überprüfen, ob das Azure ACR-Passwort abgelaufen ist
3. Bestätigen, ob die ACR-Dienstkonto-Berechtigungen korrekt sind

#### Problem 2: Netzwerk-Timeout
```
Error: timeout waiting for response
```
Lösung:
1. Netzwerkverbindung prüfen
2. Parallelität reduzieren (Parameter `--proc`)
3. Auf Netzwerkwiederherstellung warten und den Workflow erneut auslösen

#### Problem 3: Unvollständige Image-Synchronisation
```
Warning: some tags failed to sync
```
Lösung:
1. Synchronisationsprotokolle prüfen, um fehlgeschlagene Tags zu identifizieren
2. Workflow manuell erneut auslösen
3. Überprüfen, ob das Quell-Image auf Docker Hub ordnungsgemäß funktioniert

### 4. Überwachung und Benachrichtigungen
- Regelmäßig die `Actions`-Seite überprüfen, um den Workflow-Status zu bestätigen
- GitHub-Benachrichtigungen konfigurieren, um Workflow-Fehler rechtzeitig zu erhalten
- Azure ACR-Speichernutzung überwachen
- Regelmäßig die Tag-Konsistenz überprüfen

## Häufig gestellte Fragen und Lösungen

### F1: Wie synchronisiere ich bestimmte Tags statt aller Tags?

Ändern Sie die `images.yaml`-Konfigurationsdatei:
```yaml
# Nur latest und v1.0 Tags synchronisieren
docker.io/newbe36524/hagicode:latest: hagicode.azurecr.io/hagicode:latest
docker.io/newbe36524/hagicode:v1.0: hagicode.azurecr.io/hagicode:v1.0
```

### F2: Wie synchronisiere ich mehrere Image-Repositories?

Fügen Sie mehrere Zeilen in `images.yaml` hinzu:
```yaml
docker.io/newbe36524/hagicode: hagicode.azurecr.io/hagicode
docker.io/newbe36524/another-image: hagicode.azurecr.io/another-image
```

### F3: Wie wiederhole ich nach einem Synchronisationsfehler?
- Automatischer Wiederholungsversuch: image-syncer enthält einen integrierten Wiederholungsmechanismus (standardmäßig 3-mal)
- Manueller Wiederholungsversuch: `Re-run all jobs` auf der GitHub Actions-Seite anklicken

### F4: Wie zeige ich den detaillierten Synchronisationsfortschritt an?
- Echtzeitprotokolle auf der `Actions`-Seite anzeigen
- Das `sync-logs`-Artifact herunterladen, um die vollständige Protokolldatei zu sehen
- Die Protokolldatei enthält den Synchronisationsstatus und die Übertragungsgeschwindigkeit für jeden Tag

### F5: Wie lange dauert die Synchronisation?
- Erste vollständige Synchronisation: In der Regel 10-30 Minuten je nach Image-Größe
- Inkrementelle Synchronisation: Meist 2-5 Minuten bei geringen Image-Änderungen
- Die Dauer hängt von der Netzwerkbandbreite, der Image-Größe und den Parallelitätseinstellungen ab
