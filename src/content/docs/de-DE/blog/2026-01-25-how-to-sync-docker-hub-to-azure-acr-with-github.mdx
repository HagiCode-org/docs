## Automatische Imagesynchronisierung von Docker Hub zu Azure ACR implementieren

> Dieser Artikel stellt vor, wie Sie mit GitHub Actions und dem image-syncer-Tool eine automatische Synchronisierung von Docker Hub-Images zu Azure Container Registry implementieren. Dies löst das Problem langsamer Zugriffszeiten auf Docker Hub in China und某些 Azure-Regionen und verbessert die Verfügbarkeit von Images sowie die Bereitstellungseffizienz in Azure-Umgebungen.

## Hintergrund/Einführung

Das HagiCode-Projekt verwendet Docker-Images als Kernlaufzeitkomponenten, wobei die Haupt-Images auf Docker Hub gehostet werden. Mit der Entwicklung des Projekts und der zunehmenden Nachfrage nach Azure-Bereitstellungen stießen wir auf folgende Schmerzpunkte:
- Langsames Image-Pulling, Docker Hub ist in China und einigen Azure-Regionen eingeschränkt zugänglich
- Abhängigkeit von einer einzigen Image-Quelle birgt Risiko eines Single Points of Failure
- In Azure-Umgebungen bietet Azure Container Registry bessere Netzwerkleistung und Integration

Um diese Probleme zu lösen, mussten wir einen automatischen Image-Synchronisierungsmechanismus einrichten, der Docker Hub-Images regelmäßig zu Azure ACR synchronisiert und sicherstellt, dass Benutzer in Azure-Umgebungen schnellere Image-Pull-Geschwindigkeiten und höhere Verfügbarkeit erhalten.

## Vergleich der technischen Lösungen

Bei der Entwicklung der Lösung haben wir verschiedene technische Ansätze verglichen:

### 1. image-syncer (endgültige Wahl)
- Inkrementelle Synchronisierung: Synchronisiert nur geänderte Image-Layer, reduziert die Netzwerkübertragung erheblich
- Resume-Funktion: Synchronisierung kann nach Netzwerkunterbrechung fortgesetzt werden
- Parallelitätssteuerung: Unterstützt konfigurierbare Anzahl von Parallel-Threads, verbessert die Effizienz bei großen Images
- Umfassende Fehlerbehandlung: Integrierter Wiederholungsmechanismus bei Fehlern (standardmäßig 3 Mal)
- Leichte Bereitstellung: Einzelne Binärdatei, keine Abhängigkeiten
- Multi-Registry-Unterstützung: Kompatibel mit Docker Hub, Azure ACR, Harbor usw.

### 2. Docker CLI
- Keine inkrementelle Synchronisierung: Jedes Mal müssen die vollständigen Image-Inhalte gezogen werden
- Geringere Effizienz: Große Netzwerkübertragungsmenge, lange Dauer
- Einfach zu bedienen: Verwendung der vertrauten Befehle docker pull/push

### 3. Azure CLI
- Hohe Komplexität: Erfordert Konfiguration der Azure CLI-Authentifizierung
- Funktionseinschränkungen: Die Funktion az acr import ist relativ einfach
- Native Integration: Gute Integration mit Azure-Diensten

## Architekturdesign-Entscheidungen

### Entscheidung 1: Synchronisierungsfrequenz auf täglich UTC 00:00 eingestellt
- Ausgleich zwischen Image-Aktualität und Ressourcenverbrauch
- Vermeidung von Geschäftszeiten, reduziert Auswirkungen auf andere Operationen
- Docker Hub-Images werden normalerweise nach täglichen Builds aktualisiert

### Entscheidung 2: Alle Image-Tags synchronisieren
- Vollständige Konsistenz mit Docker Hub beibehalten
- Flexible Versionsauswahl für Benutzer bieten
- Synchronisierungslogik vereinfachen, komplexe Tag-Filterregeln vermeiden

### Entscheidung 3: GitHub Secrets zum Speichern von Authentifizierungsinformationen verwenden
- Native Unterstützung durch GitHub Actions, hohe Sicherheit
- Einfache Konfiguration, leicht zu verwalten und zu warten
- Repository-level Zugriffssteuerung unterstützt

## Risikobewertung und Minderung

### Risiko 1: Offenlegung von Azure ACR-Authentifizierungsinformationen
- Verschlüsselte Speicherung mit GitHub Secrets
- Regelmäßiger Wechsel von ACR-Passwörtern
- Einschränkung von ACR-Benutzerrechten auf nur Push
- Überwachung von ACR-Zugriffsprotokollen

### Risiko 2: Synchronisierungsfehler führen zu Image-Inkonsistenz
- image-syncer integrierter inkrementeller Synchronisierungsmechanismus
- Automatische Wiederholung bei Fehlern (standardmäßig 3 Mal)
- Detaillierte Fehlerprotokolle und Fehlerbenachrichtigungen
- Resume-Funktion

### Risiko 3: Zu hoher Ressourcenverbrauch
- Inkrementelle Synchronisierung reduziert Netzwerkübertragung
- Konfigurierbare Anzahl von Parallel-Threads (derzeit auf 10 eingestellt)
- Überwachung der Anzahl und Größe der synchronisierten Images
- Ausführung der Synchronisierung außerhalb der Hauptzeiten

## Kernlösung

Wir verwenden die automatische Lösung GitHub Actions + image-syncer, um die Imagesynchronisierung von Docker Hub zu Azure ACR zu implementieren.

## Implementierungsschritte

### 1. Vorbereitungsphase
- Erstellen oder Bestätigen von Azure Container Registry im Azure Portal
- Erstellen von ACR-Zugriffsschlüsseln (Benutzername und Passwort)
- Bestätigen der Docker Hub-Image-Repository-Zugriffsrechte

### 2. GitHub Secrets konfigurieren
Fügen Sie in den GitHub-Repository-Einstellungen folgende Secrets hinzu:
- AZURE_ACR_USERNAME: Azure ACR-Benutzername
- AZURE_ACR_PASSWORD: Azure ACR-Passwort

### 3. GitHub Actions-Workflow erstellen
Konfigurieren Sie den Workflow in .github/workflows/sync-docker-acr.yml:
- Zeitgesteuerte Auslösung: Täglich UTC 00:00
- Manuelle Auslösung: Unterstützt workflow_dispatch
- Zusätzliche Auslösung: Auslösung bei Push des Publish-Zweigs (für schnelle Synchronisierung)

### 4. Workflow-Ausführungsablauf

| Reihenfolge | Teilnehmer | Aktion | Beschreibung |
| --- | --- | --- | --- |
| 1 | GitHub Actions | Workflow auslösen | Durch Zeitplan, manuell oder `publish`-Zweig-Push ausgelöst |
| 2 | GitHub Actions → image-syncer | Synchronisierungstool herunterladen und ausführen | Eintritt in eigentliche Synchronisierungsphase |
| 3 | image-syncer → Docker Hub | Image-Manifest und Tag-Liste abrufen | Quell-Repository-Metadaten lesen |
| 4 | image-syncer → Azure ACR | Vorhandene Image-Informationen des Ziel-Repositorys abrufen | Aktuellen Status auf Zielseite bestimmen |
| 5 | image-syncer | Unterschiede zwischen Quelle und Ziel vergleichen | Zu synchronisierende Image-Layer identifizieren |
| 6 | image-syncer → Docker Hub | Geänderte Image-Layer pullen | Nur zu aktualisierende Inhalte übertragen |
| 7 | image-syncer → Azure ACR | Geänderte Image-Layer pushen | Inkrementelle Synchronisierung abschließen |
| 8 | image-syncer → GitHub Actions | Synchronisierungsstatistiken zurückgeben | Enthält Ergebnisse, Unterschiede und Fehlerinformationen |
| 9 | GitHub Actions | Protokolle aufzeichnen und Artifacts hochladen | Ermöglicht spätere Auditierung und Fehlerbehebung |

## GitHub Actions-Workflow-Implementierung

Hier ist die tatsächliche Workflow-Konfiguration (.github/workflows/sync-docker-acr.yml):

```yaml
name: Sync Docker Image to Azure ACR

on:
  schedule:
    - cron: "0 0 * * *" # Täglich UTC 00:00
  workflow_dispatch: # Manuelle Auslösung
  push:
    branches: [publish]

permissions:
  contents: read

jobs:
  sync:
    runs-on: ubuntu-latest

    steps:
      - name: Checkout code
        uses: actions/checkout@v4

      - name: Download image-syncer
        run: |
          # image-syncer-Binärdatei herunterladen
          wget https://github.com/AliyunContainerService/image-syncer/releases/download/v1.5.5/image-syncer-v1.5.5-linux-amd64.tar.gz
          tar -zxvf image-syncer-v1.5.5-linux-amd64.tar.gz
          chmod +x image-syncer

      - name: Create auth config
        run: |
          # Authentifizierungskonfigurationsdatei generieren (YAML-Format)
          cat > auth.yaml <<EOF
          hagicode.azurecr.io:
            username: "${{ secrets.AZURE_ACR_USERNAME }}"
            password: "${{ secrets.AZURE_ACR_PASSWORD }}"
          EOF

      - name: Create images config
        run: |
          # Image-Synchronisierungskonfigurationsdatei generieren (YAML-Format)
          cat > images.yaml <<EOF
          docker.io/newbe36524/hagicode: hagicode.azurecr.io/hagicode
          EOF

      - name: Run image-syncer
        run: |
          # Synchronisierung ausführen (neue Parameter --auth und --images verwenden)
          ./image-syncer --auth=./auth.yaml --images=./images.yaml --proc=10 --retries=3

      - name: Upload logs
        if: always()
        uses: actions/upload-artifact@v4
        with:
          name: sync-logs
          path: image-syncer-*.log
          retention-days: 7
```

## Konfigurationserklärungen

### 1. Auslösebedingungen
- Zeitgesteuerte Auslösung: cron: "0 0 * * *" - Täglich um UTC 00:00 ausgeführt
- Manuelle Auslösung: workflow_dispatch - Ermöglicht Benutzern die manuelle Ausführung in der GitHub-Benutzeroberfläche
- Push-Auslösung: push: branches: [publish] - Auslösung bei Push des Publish-Zweigs (für schnelle Synchronisierung)

### 2. Authentifizierungskonfiguration (auth.yaml)
```yaml
hagicode.azurecr.io:
  username: "${{ secrets.AZURE_ACR_USERNAME }}"
  password: "${{ secrets.AZURE_ACR_PASSWORD }}"
```

### 3. Image-Synchronisierungskonfiguration
```yaml
docker.io/newbe36524/hagicode: hagicode.azurecr.io/hagicode
```
Diese Konfiguration bedeutet, dass alle Tags von docker.io/newbe36524/hagicode zu hagicode.azurecr.io/hagicode synchronisiert werden

### 4. image-syncer-Parameter
- --auth=./auth.yaml: Pfad zur Authentifizierungskonfigurationsdatei
- --images=./images.yaml: Pfad zur Image-Synchronisierungskonfigurationsdatei
- --proc=10: Anzahl der Parallel-Threads ist 10
- --retries=3: 3 Wiederholungsversuche bei Fehlern

## Hinweise und Best Practices

### 1. Sicherheitsempfehlungen
- Regelmäßiger Wechsel von Azure ACR-Passwörtern (empfohlen alle 90 Tage)
- Verwendung dedizierter ACR-Dienstkonten mit auf nur Push beschränkten Rechten
- Überwachung der ACR-Zugriffsprotokolle, rechtzeitige Erkennung anomalen Zugriffs
- Keine Ausgabe von Authentifizierungsinformationen in Protokollen
- Keine Übermittlung von Authentifizierungsinformationen an das Code-Repository

### 2. Leistungsoptimierung
- Anpassung des --proc-Parameters: Parallelität basierend auf Netzwerkbandbreite anpassen (empfohlen 5-20)
- Überwachung der Synchronisierungszeit: Wenn die Synchronisierungszeit zu lang ist, Reduzierung der Parallelzahl in Betracht ziehen
- Regelmäßige Bereinigung von Protokollen: Angemessene retention-days einstellen (derzeit 7 Tage)

### 3. Fehlerbehebung

#### Problem 1: Authentifizierungsfehler
```
Error: failed to authenticate to hagicode.azurecr.io
```
Lösung:
1. Überprüfen, ob GitHub Secrets korrekt konfiguriert sind
2. Verifizieren, ob das Azure ACR-Passwort abgelaufen ist
3. Bestätigen, ob die ACR-Dienstkonto-Rechte korrekt sind

#### Problem 2: Netzwerk-Timeout
```
Error: timeout waiting for response
```
Lösung:
1. Netzwerkverbindung überprüfen
2. Anzahl der Parallel-Threads reduzieren (--proc-Parameter)
3. Nach Wiederherstellung des Netzwerks Workflow erneut auslösen

#### Problem 3: Image-Synchronisierung unvollständig
```
Warning: some tags failed to sync
```
Lösung:
1. Synchronisierungsprotokolle überprüfen, fehlgeschlagene Tags identifizieren
2. Workflow manuell erneut auslösen
3. Verifizieren, ob Docker Hub-Quelldateien normal sind

### 4. Überwachung und Alarmierung
- Regelmäßige Überprüfung der Actions-Seite, Bestätigung des Workflow-Status
- GitHub-Benachrichtigungen einrichten, rechtzeitige Erhalt von Workflow-Fehlerbenachrichtigungen
- Überwachung der Azure ACR-Speichernutzung
- Regelmäßige Verifizierung der Image-Tag-Konsistenz

## Häufige Fragen und Lösungen

### F1: Wie synchronisiere ich bestimmte Tags anstatt aller Tags?

Ändern Sie die Konfigurationsdatei images.yaml:
```yaml
# Nur latest und v1.0 Tags synchronisieren
docker.io/newbe36524/hagicode:latest: hagicode.azurecr.io/hagicode:latest
docker.io/newbe36524/hagicode:v1.0: hagicode.azurecr.io/hagicode:v1.0
```

### F2: Wie synchronisiere ich mehrere Image-Repositories?

Fügen Sie mehrere Konfigurationszeilen zu images.yaml hinzu:
```yaml
docker.io/newbe36524/hagicode: hagicode.azurecr.io/hagicode
docker.io/newbe36524/another-image: hagicode.azurecr.io/another-image
```

### F3: Wie wiederhole ich nach Synchronisierungsfehler?
- Automatische Wiederholung: image-syncer integrierter Wiederholungsmechanismus (standardmäßig 3 Mal)
- Manuelle Wiederholung: Klicken Sie auf der GitHub Actions-Seite auf Re-run all jobs

### F4: Wie zeige ich den detaillierten Synchronisierungsfortschritt an?
- Anzeige von Echtzeitprotokollen auf der Actions-Seite
- Download von sync-logs artifact für vollständige Protokolldatei
- Protokolldatei enthält Synchronisierungsstatus und Übertragungsgeschwindigkeit für jedes Tag

### F5: Wie lange dauert die Synchronisierung?
- Erste vollständige Synchronisierung: Abhängig von der Image-Größe, normalerweise 10-30 Minuten
- Inkrementelle Synchronisierung: Wenn Image-Änderungen klein sind, normalerweise 2-5 Minuten
- Zeit hängt von Netzwerkbandbreite, Image-Größe und Parallelitätseinstellungen ab
