## Automatische Dockerspiegel-Synchronisierung von Docker Hub zu Azure ACR

> Dieser Artikel stellt vor, wie Sie mit GitHub Actions und dem Tool image-syncer eine automatische Synchronisierung von Docker Hub-Images zum Azure Container Registry realisieren können. Dadurch werden die langsamen Zugriffszeiten auf Docker Hub in bestimmten Regionen behoben und die Verfügbarkeit der Images sowie die Bereitstellungseffizienz in Azure-Umgebungen verbessert.

## Hintergrund/Einleitung

Das HagiCode-Projekt verwendet Docker-Images als zentrale Laufzeitkomponenten, die hauptsächlich auf Docker Hub gehostet werden. Mit dem Projektwachstum und steigenden Anforderungen an Azure-Bereitstellungen traten folgende Probleme auf:
- Langsame Image-Pull-Geschwindigkeit, da Docker Hub in bestimmten Regionen eingeschränkt ist
- Risiko eines Single Point of Failure bei Abhängigkeit von einer einzigen Image-Quelle
- Bessere Netzwerkleistung und Integration in Azure-Umgebungen durch Nutzung von Azure Container Registry

Um diese Probleme zu lösen, mussten wir einen automatisierten Synchronisierungsmechanismus einrichten, der Docker Hub-Images regelmäßig in die Azure ACR synchronisiert und sicherstellt, dass Benutzer in der Azure-Umgebung schnellere Pull-Geschwindigkeiten und höhere Verfügbarkeit erhalten.

## Vergleich der technischen Lösungsansätze

Bei der Ausarbeitung der Lösung verglichen wir mehrere technische Ansätze:

### 1. image-syncer (endgültige Wahl)
- Inkrementelle Synchronisierung: Nur geänderte Image-Layer werden synchronisiert, was den Netzwerkverkehr deutlich reduziert
- Wiederaufnahme bei Unterbrechung: Synchronisierung kann nach Netzwerkausfall fortgesetzt werden
- Parallelitätssteuerung: Konfigurierbare Anzahl paralleler Threads zur Steigerung der Effizienz
- Umfassende Fehlerbehandlung: Integrierter Retry-Mechanismus (standardmäßig 3 Versuche)
- Leichtgewichtige Bereitstellung: Einzelne Binärdatei, keine Abhängigkeiten
- Multi-Registry-Support: Kompatibel mit Docker Hub, Azure ACR, Harbor usw.

### 2. Docker CLI
- Keine inkrementelle Synchronisierung: Jedes Mal müssen die vollständigen Image-Inhalte heruntergeladen werden
- Geringe Effizienz: Hoher Netzwerkaufwand, lange Dauer
- Einfache Bedienung: Verwendung der vertrauten docker pull/push-Befehle

### 3. Azure CLI
- Hohe Komplexität: Azure CLI-Authentifizierung muss konfiguriert werden
- Funktionsbeschränkungen: az acr import ist relativ begrenzt
- Native Integration: Gute Integration mit Azure-Diensten

## Architekturentscheidungen

### Entscheidung 1: Synchronisierungsfrequenz auf täglich UTC 00:00
- Ausgewogene Balance zwischen Image-Aktualität und Ressourcenverbrauch
- Vermeidung von Geschäftsverkehrsspitzen, um Auswirkungen auf andere Vorgänge zu minimieren
- Docker Hub-Images werden in der Regel nach dem täglichen Build aktualisiert

### Entscheidung 2: Synchronisierung aller Image-Tags
- Aufrechterhaltung vollständiger Konsistenz mit Docker Hub
- Flexible Versionsauswahl für Benutzer
- Vereinfachung der Synchronisierungslogik ohne komplexe Tag-Filterregeln

### Entscheidung 3: Verwendung von GitHub Secrets für die Speicherung von Anmeldedaten
- Native Unterstützung durch GitHub Actions, hohe Sicherheit
- Einfache Konfiguration, leicht zu verwalten und zu warten
- Repository-weite Zugriffskontrolle

## Risikobewertung und Minderung

### Risiko 1: Leakage von Azure ACR-Anmeldedaten
- Verschlüsselte Speicherung in GitHub Secrets
- Regelmäßige Rotation der ACR-Passwörter
- Einschränkung der ACR-Benutzerberechtigungen auf Push-only
- Überwachung der ACR-Zugriffsprotokolle

### Risiko 2: Synchronisierungsfehler führen zu Image-Inkonsistenz
- Integrierter inkrementeller Synchronisierungsmechanismus von image-syncer
- Automatische Wiederholungsversuche bei Fehlern (standardmäßig 3 Versuche)
- Detaillierte Fehlerprotokolle und Fehlerbenachrichtigungen
- Wiederaufnahmefunktion bei Unterbrechungen

### Risiko 3: Übermäßiger Ressourcenverbrauch
- Inkrementelle Synchronisierung reduziert den Netzwerkverkehr
- Konfigurierbare Thread-Anzahl (aktuell auf 10 eingestellt)
- Überwachung der Anzahl und Größe synchronisierter Images
- Ausführung der Synchronisierung außerhalb der Spitzenzeiten

## Kernlösung

Wir verwenden die automatisierte Lösung GitHub Actions + image-syncer zur Image-Synchronisierung von Docker Hub zu Azure ACR.

## Implementierungsschritte

### 1. Vorbereitung
- Erstellen oder Bestätigen der Azure Container Registry im Azure Portal
- Erstellung von ACR-Zugriffsschlüsseln (Benutzername und Passwort)
- Bestätigung der Zugriffsberechtigungen für Docker Hub-Image-Repositories

### 2. Konfiguration der GitHub Secrets
Fügen Sie in den GitHub-Repository-Einstellungen folgende Secrets hinzu:
- AZURE_ACR_USERNAME: Azure ACR-Benutzername
- AZURE_ACR_PASSWORD: Azure ACR-Passwort

### 3. Erstellung des GitHub Actions-Workflows
Konfigurieren Sie den Workflow in .github/workflows/sync-docker-acr.yml:
- Zeitgesteuerte Auslösung: Täglich UTC 00:00
- Manuelle Auslösung: Unterstützung für workflow_dispatch
- Zusätzliche Auslösung: Bei Push auf den publish-Zweig (für schnelle Synchronisierung)

### 4. Workflow-Ausführungsablauf

| Reihenfolge | Akteur | Aktion | Beschreibung |
| --- | --- | --- | --- |
| 1 | GitHub Actions | Workflow auslösen | Durch Zeitplan, manuelle Auslösung oder Push auf den `publish`-Zweig |
| 2 | GitHub Actions -> image-syncer | Synchronisierungstool herunterladen und ausführen | Eintritt in die eigentliche Synchronisierungsphase |
| 3 | image-syncer -> Docker Hub | Image-Manifest und Tag-Liste abrufen | Lesen der Quell-Repository-Metadaten |
| 4 | image-syncer -> Azure ACR | Vorhandene Images im Ziel-Repository abrufen | Bestimmung des aktuellen Zustands auf der Zielseite |
| 5 | image-syncer | Quelle und Ziel vergleichen | Identifikation der zu synchronisierenden Image-Layer |
| 6 | image-syncer -> Docker Hub | Geänderte Image-Layer pullen | Nur zu aktualisierende Inhalte übertragen |
| 7 | image-syncer -> Azure ACR | Geänderte Image-Layer pushen | Abschließung der inkrementellen Synchronisierung |
| 8 | image-syncer -> GitHub Actions | Synchronisierungsstatistiken zurückgeben | Enthält Ergebnisse, Differenzen und Fehlerinformationen |
| 9 | GitHub Actions | Protokolle aufzeichnen und Artifact hochladen | Zur späteren Auditierung und Fehlerbehebung |

## Implementierung des GitHub Actions-Workflows

Im Folgenden die tatsächlich verwendete Workflow-Konfiguration (.github/workflows/sync-docker-acr.yml):

```yaml
name: Sync Docker Image to Azure ACR

on:
  schedule:
    - cron: "0 0 * * *" # 每天 UTC 00:00
  workflow_dispatch: # 手动触发
  push:
    branches: [publish]

permissions:
  contents: read

jobs:
  sync:
    runs-on: ubuntu-latest

    steps:
      - name: Checkout code
        uses: actions/checkout@v4

      - name: Download image-syncer
        run: |
          # 下载 image-syncer 二进制文件
          wget https://github.com/AliyunContainerService/image-syncer/releases/download/v1.5.5/image-syncer-v1.5.5-linux-amd64.tar.gz
          tar -zxvf image-syncer-v1.5.5-linux-amd64.tar.gz
          chmod +x image-syncer

      - name: Create auth config
        run: |
          # 生成认证配置文件 (YAML 格式)
          cat > auth.yaml <<EOF
          hagicode.azurecr.io:
            username: "${{ secrets.AZURE_ACR_USERNAME }}"
            password: "${{ secrets.AZURE_ACR_PASSWORD }}"
          EOF

      - name: Create images config
        run: |
          # 生成镜像同步配置文件 (YAML 格式)
          cat > images.yaml <<EOF
          docker.io/newbe36524/hagicode: hagicode.azurecr.io/hagicode
          EOF

      - name: Run image-syncer
        run: |
          # 执行同步 (使用新版 --auth 和 --images 参数)
          ./image-syncer --auth=./auth.yaml --images=./images.yaml --proc=10 --retries=3

      - name: Upload logs
        if: always()
        uses: actions/upload-artifact@v4
        with:
          name: sync-logs
          path: image-syncer-*.log
          retention-days: 7
```

## Konfigurationserklärung

### 1. Auslösebedingungen
- Zeitgesteuerte Auslösung: cron: "0 0 * * *" - Täglich um UTC 00:00
- Manuelle Auslösung: workflow_dispatch - Ermöglicht die manuelle Ausführung in der GitHub-Benutzeroberfläche
- Push-Auslösung: push: branches: [publish] - Auslösung bei Push auf den Veröffentlichungszweig

### 2. Authentifizierungskonfiguration (auth.yaml)
```yaml
hagicode.azurecr.io:
  username: "${{ secrets.AZURE_ACR_USERNAME }}"
  password: "${{ secrets.AZURE_ACR_PASSWORD }}"
```

### 3. Image-Synchronisierungskonfiguration
```yaml
docker.io/newbe36524/hagicode: hagicode.azurecr.io/hagicode
```
Diese Konfiguration bedeutet, dass alle Tags von docker.io/newbe36524/hagicode zu hagicode.azurecr.io/hagicode synchronisiert werden.

### 4. image-syncer-Parameter
- --auth=./auth.yaml: Pfad zur Authentifizierungskonfigurationsdatei
- --images=./images.yaml: Pfad zur Image-Synchronisierungskonfigurationsdatei
- --proc=10: Anzahl paralleler Threads: 10
- --retries=3: 3 Wiederholungsversuche bei Fehlschlagen

## Hinweise und Best Practices

### 1. Sicherheitsempfehlungen
- Regelmäßige Rotation der Azure ACR-Passwörter (empfohlen alle 90 Tage)
- Verwendung eines dedizierten ACR-Dienstkontos mit auf Push beschränkten Berechtigungen
- Überwachung der ACR-Zugriffsprotokolle zur rechtzeitigen Erkennung anomaler Zugriffe
- Keine Ausgabe von Anmeldedaten in Protokollen
- Keine Einbindung von Anmeldedaten in das Code-Repository

### 2. Leistungsoptimierung
- Anpassung des --proc-Parameters: Parallelitätsanzahl entsprechend der Netzwerkbandbreite (empfohlen 5-20)
- Überwachung der Synchronisierungsdauer: Bei zu langer Dauer sollte die Parallelitätsanzahl reduziert werden
- Regelmäßige Bereinigung von Protokollen: Festlegung eines angemessenen retention-days-Werts (aktuell 7 Tage)

### 3. Fehlerbehebung

#### Problem 1: Authentifizierungsfehler
```
Error: failed to authenticate to hagicode.azurecr.io
```
Lösung:
1. Überprüfen Sie die Konfiguration der GitHub Secrets
2. Überprüfen Sie, ob das Azure ACR-Passwort abgelaufen ist
3. Bestätigen Sie, ob die Berechtigungen des ACR-Dienstkontos korrekt sind

#### Problem 2: Netzwerk-Timeout
```
Error: timeout waiting for response
```
Lösung:
1. Überprüfen Sie die Netzwerkverbindung
2. Reduzieren Sie die Anzahl paralleler Threads (--proc-Parameter)
3. Warten Sie auf die Wiederherstellung des Netzwerks und lösen Sie den Workflow erneut aus

#### Problem 3: Unvollständige Image-Synchronisierung
```
Warning: some tags failed to sync
```
Lösung:
1. Überprüfen Sie die Synchronisierungsprotokolle und identifizieren Sie die fehlgeschlagenen Tags
2. Lösen Sie den Workflow manuell erneut aus
3. Überprüfen Sie, ob die Docker Hub-Quellimages ordnungsgemäß verfügbar sind

### 4. Überwachung und Alarmierung
- Regelmäßige Überprüfung der Actions-Seite zur Bestätigung des Workflow-Status
- Konfiguration von GitHub-Benachrichtigungen für zeitnahe Fehlermeldungen
- Überwachung der Speichernutzung von Azure ACR
- Regelmäßige Überprüfung der Konsistenz der Image-Tags

## Häufig gestellte Fragen und Lösungen

### F1: Wie synchronisiere ich bestimmte Tags anstelle aller Tags?

Ändern Sie die images.yaml-Konfigurationsdatei:
```yaml
# 仅同步 latest 和 v1.0 标签
docker.io/newbe36524/hagicode:latest: hagicode.azurecr.io/hagicode:latest
docker.io/newbe36524/hagicode:v1.0: hagicode.azurecr.io/hagicode:v1.0
```

### F2: Wie synchronisiere ich mehrere Image-Repositories?

Fügen Sie mehrere Zeilen in der images.yaml hinzu:
```yaml
docker.io/newbe36524/hagicode: hagicode.azurecr.io/hagicode
docker.io/newbe36524/another-image: hagicode.azurecr.io/another-image
```

### F3: Wie wiederhole ich nach einem Synchronisierungsfehler?
- Automatischer Wiederholungsversuch: image-syncer verfügt über einen integrierten Retry-Mechanismus (standardmäßig 3 Versuche)
- Manueller Wiederholungsversuch: Klicken Sie auf der GitHub Actions-Seite auf „Re-run all jobs"

### F4: Wie zeige ich den detaillierten Synchronisierungsfortschritt an?
- Echtzeitprotokolle auf der Actions-Seite anzeigen
- sync-logs-Artifact herunterladen für vollständige Protokolldateien
- Protokolldateien enthalten den Synchronisierungsstatus und die Übertragungsgeschwindigkeit für jeden Tag

### F5: Wie lange dauert die Synchronisierung?
- Erste Vollsynchronisierung: Abhängig von der Image-Größe in der Regel 10-30 Minuten
- Inkrementelle Synchronisierung: Bei geringen Image-Änderungen in der Regel 2-5 Minuten
- Die Dauer hängt von der Netzwerkbandbreite, der Image-Größe und den Parallelitätseinstellungen ab
