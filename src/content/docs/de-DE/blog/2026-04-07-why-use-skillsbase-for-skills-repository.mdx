## Warum Skillsbase verwenden, um Ihr eigenes Skills-Sammlungs-Repository zu pflegen

> Es ist irgendwie lustig, wenn man darueber nachdenkt: Das Zeitalter der KI-Programmierung ist angekommen, und die Agent Skills, die wir zur Verfuegung haben, werden immer zahlreicher. Aber damit kommt auch immer mehr Aerger. Dieser Artikel handelt davon, wie wir skillsbase verwendet haben, um diese Probleme zu loesen.

## Hintergrund

Im Zeitalter der KI-Programmierung muessen Entwickler eine zunehmende Anzahl von Agent Skills pflegen - wiederverwendbare Anweisungssätze, die die Faehigkeiten von Programmierassistenten wie Claude Code, OpenCode und Cursor erweitern. Allerdings taucht mit der wachsenden Anzahl von Skills allmaehlich ein praktisches Problem auf:

Es ist nicht gerade ein grosses Problem, aber wenn man zu viele Dinge hat, wird die Verwaltung mühseelig.

### Skills sind an verschiedenen Orten verstreut, was die Verwaltung kostspielig macht

- Lokale Skills sind an mehreren Stellen verstreut: `~/.agents/skills/`, `~/.claude/skills/`, `~/.codex/skills/.system/` usw.
- Verschiedene Orte koennen Namenskonflikte aufweisen, z.B. `skill-creator` existiert sowohl im Nutzerverzeichnis als auch im Systemverzeichnis
- Es gibt keinen einheitlichen Verwaltungseinstiegspunkt, was Backup und Migration erschwert

### Fehlt ein standardisierter Pflege-Workflow

- Manuelles Kopieren von Skills ist fehleranfaellig und erschwert die Rueckverfolgung des Ursprungs
- Ohne einheitlichen Validierungsmechanismus gibt es keine Garantie, dass das Skills-Repository vollstaendig bleibt
- Waehrend der Teamzusammenarbeit ist die Synchronisierung und Freigabe einer Skills-Sammlung schwierig

### Wiedergabeanforderungen werden nicht erfuellt

- Beim Wechsel des Entwicklungsrechners muessen alle Skills neu konfiguriert werden
- In CI/CD-Umgebungen koennen das Skills-Repository nicht automatisch validiert und synchronisiert werden

Um diese Schmerzpunkte anzugehen, versuchten wir viele verschiedene Ansaetze: vom manuellen Kopieren ueber skriptgesteuerte Automatisierung bis hin zur direkten Verzeichnisverwaltung und globalen Installation mit anschliessender Dateiwiederherstellung. Jeder Ansatz hatte seine eigenen Maengel.

Letztendlich fanden wir eine elegantere Loesung: skillsbase. Die Kernidee hinter diesem Ansatz ist, zuerst lokal zu installieren und zu validieren, dann die Struktur umzuwandeln und in das Repository zu schreiben und schliesslich die temporaeren Dateien zu deinstallieren.

## Ueber HagiCode

Die in diesem Artikel geteilte Loesung stammt aus unserer praktischen Erfahrung im [HagiCode](https://hagicode.com)-Projekt.

HagiCode ist ein KI-Programmierassistent-Projekt. Waehrend der Entwicklung muessen wir eine grosse Anzahl von Agent Skills pflegen, um verschiedene Programmierfaehigkeiten zu erweitern. Diese realen Anforderungen trieben uns dazu, das skillsbase-Werkzeugset fuer die standardisierte Verwaltung von Skills-Repositories zu bauen.

Wenn Sie an HagiCode interessiert sind, koennen Sie die [offizielle Website](https://hagicode.com) besuchen, um mehr zu erfahren, oder den Quellcode auf [GitHub](https://github.com/HagiCode-org/site) einsehen.

## Analyse

### Technische Herausforderungen

Um ein wartbares Skills-Sammlungs-Repository zu bauen, muessen folgende Kernprobleme geloest werden:

1. **Einheitliche Namensraumkonflikte**: Wenn mehrere Quellen Skills mit demselben Namen enthalten, wie vermeiden wir ein Ueberschreiben?
2. **Quellen-Rueckverfolgbarkeit**: Wie erfassen wir die Quelle jedes Skills fuer kuenftige Updates und Audits?
3. **Synchronisierung und Validierung**: Wie stellen wir sicher, dass der Repository-Inhalt mit den tatsaechlichen Installationsergebnissen uebereinstimmt?
4. **Automatisierungsintegration**: Wie integrieren wir uns in CI/CD-Workflows, um automatische Synchronisierung und Validierung zu ermoeglichen?

### Design-Entscheidungen

**Option 1: Verzeichnisse direkt kopieren**

Vorteile: Einfach zu implementieren
Nachteile: Keine Garantie fuer Konsistenz mit dem tatsaechlichen Installationsergebnis der `skills` CLI

**Option 2: Global installieren und dann wiederherstellen**

Vorteile: Der Installationsprozess kann validiert werden
Nachteile: Verschmutzt die Ausfuehrungsumgebung, und es ist schwierig, CI und lokale Ergebnisse konsistent zu halten

**Option 3: Lokale Installation -> Umwandlung -> Deinstallation (endgueltige Loesung)**

Dies ist der von skillsbase gewaehlte Ansatz:

- Zunaechst Skills mit `npx skills` an einem temporaeren Ort installieren
- Die Verzeichnisstruktur umwandeln und Quellmetadaten hinzufuegen
- Das Ergebnis in das Ziel-Repository schreiben
- Schliesslich die temporaeren Dateien deinstallieren

### Architekturentscheidungen

| Entscheidungspunkt | Wahl | Grund |
|--------|------|------|
| Runtime | Node.js ESM | Kein Build-Schritt erforderlich; `.mjs` reicht zur Orchestrierung des Dateisystems |
| Konfigurationsformat | YAML (`sources.yaml`) | Hoechst lesbar und geeignet fuer manuelle Pflege |
| Namensstrategie | Namespace-Praefix | Nutzer-Skills behalten ihre urspruenglichen Namen, System-Skills erhalten den `system-`-Praefix |
| Workflow | `add` aktualisiert Manifest -> `sync` fuehrt Synchronisierung aus | Ein einziger Synchronisierungsmechanismus vermeidet die doppelte Implementierung derselben Regeln |
| Dateiverwaltung | Verwaltete Dateimarker | Kommentarkopf hinzufuegen, um sicheres Ueberschreiben zu unterstuetzen |

## Loesung

### CLI-Architektur

Die `skillsbase`-CLI bietet vier Kernbefehle:

```text
skillsbase
├── init          # Repository-Struktur initialisieren
├── sync          # Skill-Inhalte synchronisieren
├── add           # Neue Skills hinzufuegen
└── github_action # GitHub Actions-Konfiguration generieren
```

### Kern-Workflow

```
┌─────────────┐    ┌─────────────┐    ┌─────────────┐    ┌─────────────┐
│    init     │───▶│    add      │───▶│    sync     │───▶│github_action│
│ initialisieren│    │ Quelle adden│    │ Inhalt sync │    │ CI generier │
└─────────────┘    └─────────────┘    └─────────────┘    └─────────────┘
```

### Repository-Struktur

```
repos/skillsbase/
├── sources.yaml              # Quellmanifest (einzige Quelle der Wahrheit)
├── skills/                   # Skills-Verzeichnis
│   ├── frontend-design/      # Nutzer-Skill
│   ├── skill-creator/        # Nutzer-Skill
│   └── system-skill-creator/ # System-Skill (mit Praefix)
├── scripts/
│   ├── sync-skills.mjs       # Synchronisierungsskript
│   └── validate-skills.mjs   # Validierungsskript
├── docs/
│   └── maintainer-workflow.md # Maintainer-Dokumentation
└── .github/
    ├── workflows/
    │   └── skills-sync.yml   # CI-Workflow
    └── actions/
        └── skillsbase-sync/
            └── action.yml     # Wiederverwendbare Action
```

## Praxis

### Ein Skills-Repository initialisieren

```bash
# 1. Ein leeres Repository erstellen
mkdir repos/myskills && cd repos/myskills
git init

# 2. Mit skillsbase initialisieren
npx skillsbase init
```

### Skills hinzufuegen

```bash
# Einen einzelnen Skill hinzufuegen (loest automatisch Synchronisierung aus)
npx skillsbase add frontend-design --source vercel-labs/agent-skills

# Aus einer lokalen Quelle hinzufuegen
npx skillsbase add documentation-writer --source /home/user/.agents/skills
```

### Skills synchronisieren

```bash
# Synchronisierung durchfuehren (alle Quellen abgleichen)
npx skillsbase sync

# Nur auf Abweichung pruefen (Dateien nicht aendern)
npx skillsbase sync --check

# Fehlende Quellen erlauben (CI-Szenario)
npx skillsbase sync --allow-missing-sources
```

### CI-Konfiguration generieren

```bash
# Workflow generieren
npx skillsbase github_action --kind workflow

# Action generieren
npx skillsbase github_action --kind action

# Alles generieren
npx skillsbase github_action --kind all
```

### `sources.yaml`-Konfigurationsbeispiel

```yaml
# Skills-Stammverzeichniskonfiguration
skillsRoot: skills/
metadataFile: .skill-source.json

# Quellendefinitionen
sources:
  # First-party: lokale Nutzer-Skills
  first-party:
    type: local
    path: /home/user/.agents/skills
    naming: original
    includes:
      - documentation-writer
      - frontend-design
      - skill-creator

  # System: vom System bereitgestellte Skills
  system:
    type: local
    path: /home/user/.codex/skills/.system
    naming: prefix-system
    includes:
      - imagegen
      - openai-docs
      - skill-creator  # Wird zu system-skill-creator

  # Remote: Drittanbieter-Repository
  vercel:
    type: remote
    url: vercel-labs/agent-skills
    naming: original
    includes:
      - web-design-guidelines
```

### Sicherheit und Validierung

```bash
# Repository-Struktur validieren
node scripts/validate-skills.mjs

# Mit der skills CLI validieren
npx skills add . --list

# Auf Updates pruefen
npx skills check
```

### GitHub Actions-Integration

```yaml
# .github/workflows/skills-sync.yml
name: Skills Sync

on:
  push:
    paths:
      - 'sources.yaml'
      - 'skills/**'
  workflow_dispatch:

jobs:
  validate:
    runs-on: ubuntu-latest
    steps:
      - uses: actions/checkout@v4
      - uses: actions/setup-node@v4
        with:
          node-version: '20'
      - name: Repository validieren
        run: |
          npx skills add . --list
          node scripts/validate-skills.mjs
      - name: Sync-Pruefung
        run: npx skillsbase sync --check
```

Sobald die CI-Integration vorhanden ist, loest jede Aenderung an `sources.yaml` oder dem `skills/`-Verzeichnis automatisch eine Validierung aus.

### Teamzusammenarbeit

```bash
# Teammitglieder installieren das freigegebene Skills-Repository
npx skills add your-org/myskills -g --all

# Lokal klonen und validieren
git clone https://github.com/your-org/myskills.git
cd myskills
npx skills add . --list
```

Durch die Verwaltung des Skills-Repository mit Git koennen Teammitglieder ihre Skills-Sammlung leicht synchronisieren und sicherstellen, dass jeder dieselben Versionen von Werkzeugen und Konfigurationen verwendet.

## Zusammenfassung

Der Kernwert der Verwendung von `skillsbase` zur Pflege eines Skills-Sammlungs-Repositorys liegt in Folgendem:

- **Sicherheit**: Quellvalidierung, Konflikterkennung und verwaltete Datei-Schutz
- **Wartbarkeit**: Einheitlicher Einstiegspunkt, idempotente Operationen und Konfiguration-als-Dokumentation
- **Standardisierung**: Einheitliche Verzeichnisstruktur, Namenskonventionen und Metadatenformat
- **Automatisierung**: CI/CD-Integration, automatische Synchronisierung und automatische Validierung

Mit diesem Ansatz koennen Entwickler ihre eigenen Agent Skills auf dieselbe Weise verwalten wie npm-Pakete, wodurch ein reproduzierbares, teilbares und wartbares Skills-Repository-System entsteht.

Nach all ist gesagt, gute Werkzeuge verdienen es, von mehr Menschen verwendet zu werden.
