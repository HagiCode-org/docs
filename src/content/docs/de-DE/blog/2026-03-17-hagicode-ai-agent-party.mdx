## Hintergrund

Viele Entwickler haben sicherlich schon ähnliche Erfahrungen gemacht: In ein Projekt einen KI-Assistenten zur Programmierunterstützung einzuführen, erhöht tatsächlich die Effizienz. Mit zunehmender Komplexität der Anforderungen reicht ein AI-Agent nicht mehr aus – Sie möchten, dass er gleichzeitig Code-Reviews, Dokumentationserstellung, Unit-Tests und mehrere andere Aufgaben bewältigt, aber das Ergebnis ist oft, dass das eine vernachlässigt wird, während das andere getan wird, und die Ausgabequalität schwankt.

Noch problematischer wird es, wenn Sie versuchen, mehrere KI-Assistenten einzuführen – die Probleme werden komplexer. Jeder Agent hat seine eigene Konfigurationsmethode, API-Schnittstelle und Ausführungslogik, und es können sogar Konflikte zwischen ihnen entstehen. Das ist wie eine Fußballmannschaft, in der jeder Spieler großartig ist, aber niemand weiß, wie man zusammenarbeitet, und das Ergebnis ist ein chaotisches Spiel.

Das HagiCode-Projekt stieß während der Entwicklung auf das gleiche Problem. Als ein komplexes Projekt, das ein VSCode-Frontend-Plugin, einen Backend-AI-Dienst und einen plattformübergreifenden Desktop-Client umfasst, mussten wir in der **Version von 2026-03** gleichzeitig mehrere KI-Assistenten verschiedener Unternehmen anbinden: Claude Code, Codex, CodeBuddy, iFlow usw. Wie sie im selben Projekt harmonisch zusammenleben und ihre jeweiligen Stärken ausspielen können, wurde zu einem entscheidenden Problem, das gelöst werden musste.

Eigentlich ist das auch in Ordnung – schließlich möchte niemand täglich mit einer Gruppe kämpfender KI zu tun haben.

Die in diesem Artikel vorgestellte Lösung ist genau die Multi-Agent-Kollaborationskonfigurationspraxis, die wir im HagiCode-Projekt durch tatsächliche Erfahrungen und Optimiationen entwickelt haben. Wenn Sie auch mit der Multi-KI-Assistenten-Kollaboration zu kämpfen haben, wird Ihnen dieser Artikel hoffentlich einige Anregungen geben. Vielleicht schließlich ist die Situation bei jedem anders.

[HagiCode](https://hagicode.com) ist ein AI-Coding-Assistent-Projekt, das den „Abenteurerteam“-Modus mit mehreren KI-Engines nutzt, die zusammenarbeiten. Projektadresse: [github.com/HagiCode-org/site](https://github.com/HagiCode-org/site).

Die in diesem Artikel vorgestellte Multi-Agent-Konfigurationslösung ist eine der Schlüsseltechnologien, mit der HagiCode eine effiziente Entwicklung in komplexen Projekten aufrechterhalten kann. ## Multi-Agent-Architekturdesign von HagiCode

### Vom „Alleingang“ zur „Teamarbeit“

In den frühen Tagen des HagiCode-Projekts haben wir auch versucht, nur einen AI-Agenten für alle Aufgaben zu verwenden. Schnell stellten wir fest, dass dieser Ansatz offensichtliche Flaschenhälse hatte: Verschiedene Aufgaben erfordern unterschiedliche Fähigkeitsschwerpunkte. Einige Aufgaben benötigen stärkere Kontextverständnisfähigkeiten, andere präzisere Codeänderungsfähigkeiten. Ein Agent kann kaum in allen Bereichen herausragend sein.

Das hat uns zu der Erkenntnis geführt, dass mehrere Agenten zusammenarbeiten müssen. Aber das Problem war: Wie können KI-Produkte verschiedener Unternehmen im selben Projekt friedlich koexistieren? Wir mussten einige Kernprobleme lösen:

1. **Komplexität der Konfigurationsverwaltung**: Jeder Agent hat unterschiedliche Konfigurationsmethoden, API-Schnittstellen und Ausführungsmodi
2. **Einheitliche Kommunikationsprotokolle**: Es ist eine standardisierte Methode erforderlich, damit verschiedene Agenten Daten austauschen können
3. **Aufgabenteilung und -koordination**: Wie Aufgaben sinnvoll zugewiesen werden können, damit jeder Agent seine Stärken ausspielt

Mit diesen Fragen begannen wir, die Multi-Agent-Architektur von HagiCode zu entwerfen. Eigentlich gar nicht so komplex, man muss es nur verstanden haben.

### Überblick über die Gesamtarchitektur

Nach mehreren Iterationen haben wir schließlich folgende Architektur festgelegt:

```
┌─────────────────────────────────────────────────────────────────┐
│                    AIProviderFactory                             │
│  (Factory-Mode zur einheitlichen Verwaltung aller AI Provider)   │
├─────────────────────────────────────────────────────────────────┤
│  ClaudeCodeCli  │  CodexCli  │  CodebuddyCli  │  IFlowCli    │
│  (Anthropic)   │  (OpenAI)  │  (Zhipu GLM)    │  (Zhipu)     │
└─────────────────────────────────────────────────────────────────┘
```

Der Kerngedanke ist: Durch eine einheitliche Provider-Schnittstelle können verschiedene AI-Agenten mit demselben Code verwaltet werden. Gleichzeitig wird das Factory-Muster verwendet, um diese Provider dynamisch zu erstellen und zu konfigurieren, wodurch die Erweiterbarkeit und Flexibilität des Systems gewährleistet wird.

Das ist wie die Arbeitsteilung im Leben – jeder hat seine eigene Rolle, nur dass hier diese Arbeitsteilung in eine Codearchitektur verwandelt wird.

### Agent-Typen und Verantwortungsteilung

Basierend auf den tatsächlichen Nutzungserfahrungen des HagiCode-Projekts haben wir jedem Agent unterschiedliche Verantwortungen zugewiesen:

| Agent | Anbieter | Modell | Hauptzweck |
|-------|---------|--------|------------|
| ClaudeCodeCli | Anthropic | glm-5-turbo | Erstellung technischer Lösungen und Proposals |
| CodexCli | OpenAI/Zed | gpt-5.4 | Ausführung präziser Codeänderungen |
| CodebuddyCli | Zhipu | glm-4.7 | Optimierung von Proposal-Beschreibungen und Dokumentationen |
| IFlowCli | Zhipu | glm-4.7 | Archivierung von Proposals und Verlauf (damalige Konfiguration; aktuell nur historische Kompatibilität) |
| OpenCodeCli | - | - | Allgemeine Codebearbeitung |
| GitHubCopilot | Microsoft | - | Unterstützende Programmierung und Codevervollständigung |

Die Logik hinter dieser Aufgabenteilung ist: Jeder Agent hat sein eigenes Spezialgebiet. Claude Code zeichnet sich durch Verstehen und Analysieren komplexer Anforderungen aus, daher lässt es ihn für das frühe Lösungsentwer verantwortlich sein; Codex ist präziser bei Codeänderungen und eignet sich für die Bearbeitung konkreter Implementierungsaufgaben; CodeBuddy hat ein gutes Preis-Leistungs-Verhältnis und eignet sich perfekt zur Optimierung von Dokumentationen.

Schließlich ist das, was passt, das Beste – alle Wege führen nach Rom, nur sind manche Wege leichter zu begehen, andere etwas kurvenreicher.

## Kernkonfigurationsmechanismen

### Einheitliche Provider-Schnittstellendesign

Damit verschiedene AI-Agenten einheitlich verwaltet werden können, muss zunächst eine einheitliche Schnittstelle definiert werden. HagiCode definierte diese Schnittstelle:

```csharp
public interface IAIProvider
{
    // Einheitliche Provider-Schnittstelle
    Task<IAIProvider?> GetProviderAsync(AIProviderType providerType);
    Task<IAIProvider?> GetProviderAsync(string providerName, CancellationToken cancellationToken);
}
```

Diese Schnittstelle mag einfach aussehen, aber sie ist das Fundament des gesamten Multi-Agent-Systems. Durch die einheitliche Schnittstelle können wir ignorieren, von welchem Unternehmen das zugrundeliegende KI-Produkt stammt, und alle auf die gleiche Weise aufrufen.

Eigentlich macht das Kompliziertes einfach – schließlich ist Schlichtheit schön.

### Provider-Factory-Pattern-Implementierung

Mit der einheitlichen Schnittstelle stellt sich die Frage, wie diese Provider-Instanzen erstellt werden. HagiCode verwendet das Factory-Pattern:

```csharp
private IAIProvider? CreateProvider(AIProviderType providerType, ProviderConfiguration config)
{
    return providerType switch
    {
        AIProviderType.ClaudeCodeCli =>
            ActivatorUtilities.CreateInstance<ClaudeCodeCliProvider>(_serviceProvider, Options.Create(config)),
        AIProviderType.CodebuddyCli =>
            ActivatorUtilities.CreateInstance<CodebuddyCliProvider>(_serviceProvider, Options.Create(config)),
        AIProviderType.CodexCli =>
            ActivatorUtilities.CreateInstance<CodexCliProvider>(_serviceProvider, Options.Create(config)),
        AIProviderType.IFlowCli =>
            ActivatorUtilities.CreateInstance<IFlowCliProvider>(_serviceProvider, Options.Create(config)),
        _ => null
    };
}
```

Hier wird die `ActivatorUtilities.CreateInstance` der Dependency Injection verwendet, die zur Laufzeit Provider-Instanzen dynamisch erstellen und Abhängigkeiten automatisch injizieren kann. Der Vorteil dieses Designs ist: Wenn ein neuer Agent-Typ hinzugefügt wird, muss nur die entsprechende Provider-Klasse hinzugefügt und dann ein Case-Zweig zur Factory-Methode hinzugefügt werden – vorhandener Code muss überhaupt nicht geändert werden.

Das ist auch in Ordnung – schließlich möchte niemand bei jedem Hinzufügen einer neuen Funktion einen Haufen alten Code ändern.

### Dynamische Konfigurationsanalyse

Um die Konfiguration flexibler zu gestalten, haben wir auch einen Typzuordnungsmechanismus implementiert:

```csharp
public static AIProviderTypeExtensions
{
    private static readonly Dictionary<string, AIProviderType> _typeMap = new(
        StringComparer.OrdinalIgnoreCase)
    {
        ["ClaudeCodeCli"] = AIProviderType.ClaudeCodeCli,
        ["CodebuddyCli"] = AIProviderType.CodebuddyCli,
        ["CodexCli"] = AIProviderType.CodexCli,
        ["IFlowCli"] = AIProviderType.IFlowCli,
        // ... weitere Typzuordnungen
    };
}
```

Der Zweck dieser Zuordnungstabelle ist die Umwandlung von Provider-Namen in Stringform in Enumerationstypen. Auf diese Weise können Konfigurationsdateien intuitive Stringnamen verwenden, während der Code intern typsichere Enumerationen zur Verarbeitung verwendet.

Bei Konfiguration gilt: Je intuitiver, desto besser – schließlich möchte niemand eine Reihe komplexer Codes auswendig lernen.

### Konfigurationsdateibeispiel

In der tatsächlichen Verwendung muss man nur in `appsettings.json` konfigurieren:

```yaml
AI:
  Providers:
    Providers:
      ClaudeCodeCli:
        Enabled: true
        Model: glm-5-turbo
        WorkingDirectory: /path/to/project
      CodebuddyCli:
        Enabled: true
        Model: glm-4.7
      CodexCli:
        Enabled: true
        Model: gpt-5.4
      IFlowCli:
        Enabled: true
        Model: glm-4.7
```

Jeder Provider kann unabhängig mit Parametern wie Ein/Aus-Schalter, Modellversion, Arbeitsverzeichnis usw. konfiguriert werden. Dieses Design gewährleistet sowohl Flexibilität als auch einfache Verwaltung und Wartung.

Eigentlich ist eine Konfigurationsdatei wie die Optionen im Leben – Sie können wählen, ob Sie bestimmte Funktionen aktivieren oder deaktivieren, nur sind Entscheidungen im Code leichter zu bereuen.

## Aufgabenfluss des Abenteurerteams

### Die Kunst der Aufgabenteilung

Mit der einheitlichen technischen Architektur stellt sich als Nächstes die Frage, wie mehrere Agenten zusammenarbeiten können. HagiCode hat einen Aufgabenflussmechanismus entworfen, bei dem verschiedene Agenten Aufgaben in verschiedenen Phasen bearbeiten:

```
Proposal-Erstellung (Benutzer)
    │
    ▼
[Claude Code] ──Proposal generieren──▶ Proposal-Dokument
    │                        │
    │                        ▼
    │               [Codebuddy] ──Beschreibung optimieren──▶ Optimiertes Proposal
    │                        │
    │                        ▼
    │               [Codex] ──Änderung ausführen──▶ Code-Änderung
    │                        │
    │                        ▼
    └───────────────▶ [iFlow] ──Archivieren──▶ Verlauf
```

Der Vorteil dieser Aufgabenteilung ist: Jeder Agent muss sich nur auf die Aufgaben konzentrieren, die er beherrscht, und muss nicht „alles können“. Claude Code ist für die Generierung von Proposals aus dem Nichts verantwortlich, Codebuddy für die klarere Beschreibung von Proposals, Codex für die Umwandlung von Proposals in tatsächliche Codeänderungen und iFlow für die Archivierung dieser Änderungen.

Eigentlich ist das wie die Teamarbeit im Leben – jeder hat seine eigene Rolle, und nur zusammen kann man eine große Sache vollbringen. Nur sind die Teammitglieder hier KI.

### Wichtige Praxispunkte

Im tatsächlichen Betrieb haben wir folgende Erfahrungen zusammengefasst:

**1. Agent-Auswahlstrategie ist wichtig**

Aufgaben werden nicht zufällig zugewiesen, sondern basierend auf den Stärken jedes Agenten:
- Proposal-Generierung: Verwendung von Claude Code, da es über stärkere Kontextverständnisfähigkeiten verfügt
- Code-Ausführung: Verwendung von Codex, da es bei Codeänderungen präziser ist
- Proposal-Optimierung: Verwendung von Codebuddy, da es ein gutes Preis-Leistungs-Verhältnis hat
- Archivierungsspeicherung: Verwendung von iFlow, da es stabil und zuverlässig ist

Schließlich ist es eine zeitlose Wahrheit, dass der richtige Mensch die richtige Aufgabe tun sollte.

**2. Konfigurationsisolierung gewährleistet Stabilität**

Die Konfiguration jedes Agenten wird unabhängig verwaltet, unterstützt das Überschreiben durch Umgebungsvariablen, und die Arbeitsverzeichnisse sind ebenfalls voneinander unabhängig. Dadurch wirkt sich ein Konfigurationsfehler bei einem Agenten nicht auf andere Agenten aus.

Das ist wie die Grenzen im Leben – jeder hat seinen eigenen Raum, und nur so kann man harmonisch zusammenleben, ohne sich gegenseitig zu stören.

**3. Fehlerbehandlungsmechanismus**

Das Scheitern eines einzelnen Agenten sollte den Gesamtprozess nicht beeinträchtigen. Wir haben eine Fallback-Strategie implementiert: Wenn ein Agent bei der Ausführung scheitert, kann das System automatisch auf eine Ersatzlösung umschalten oder diesen Schritt direkt überspringen und mit nachfolgenden Aufgaben fortfahren. Gleichzeitig erleichtern vollständige Protokolle die späretere Problemanalyse.

Schließlich kann niemand garantieren, dass niemals Fehler auftreten – entscheidend ist, wie man mit Fehlern umgeht. Das ist wie im Leben: Man wird immer auf Hindernisse stoßen, wichtig ist, wie man daraus hervorgeht.

**4. Überwachung und Beobachtbarkeit**

Durch das ACP-Protokoll (unser benutzerdefiniertes Kommunikationsprotokoll, basierend auf JSON-RPC 2.0) kann der Ausführungsstatus jedes Agenten verfolgt werden. Die Sitzungsisolierung gewährleistet dieConcurrency-Sicherheit, während dynamisches Caching die Leistung optimiert.

Schließlich gehen am leichtesten Dinge schief, die man nicht sieht – ein bisschen Überwachung ist besser als blindlings durchs Leben zu tapeln.

## Tatsächliche Ergebnisse und Vorteile

Nach der Einführung dieser Multi-Agent-Kollaborationskonfiguration hat die Entwicklungseffizienz des HagiCode-Projekts deutlich zugenommen. Konkret zeigt sich das in:

1. **Verdoppelte Aufgabenverarbeitungskapazität**: Früher musste ein Agent mehrere Aufgaben gleichzeitig bearbeiten, jetzt kann dies parallel geschehen, und der Durchsatz hat sich mehr als verdoppelt
2. **Stabilere Ausgabequalität**: Jeder Agent konzentriert sich nur auf die Aufgaben, die er beherrscht, was zu höherer Konsistenz und Qualität der Ergebnisse führt
3. **Geringere Wartungskosten**: Einheitliche Schnittstellen und Konfigurationsverwaltung machen das gesamte System einfacher zu warten und zu erweitern
4. **Einfache Hinzufügung neuer Agenten**: Wenn ein neues KI-Produkt angebunden werden soll, muss nur die Schnittstelle implementiert und die Konfiguration hinzugefügt werden, ohne die Kernlogik zu ändern

Diese Lösung hat nicht nur die Probleme von HagiCode selbst gelöst, sondern auch bewiesen, dass die Multi-Agent-Kollaboration tatsächlich eine gangbare Architekturwahl ist.

Eigentlich ist der Effekt ziemlich offensichtlich, nur der Prozess war etwas mühsam.

## Zusammenfassung

Dieser Artikel teilte die praktischen Erfahrungen des HagiCode-Projekts im Bereich der Multi-Agent-Kollaborationskonfiguration. Die Kernpunkte umfassen:

1. **Standardisierte Schnittstelle**: Durch `IAIProvider` werden die Verhaltensweisen verschiedener Agenten vereinheitlicht, sodass Code ignorieren kann, von welchem Unternehmen das zugrundeliegende Produkt stammt
2. **Factory-Pattern**: Verwendung von `ActivatorUtilities.CreateInstance` zur dynamischen Erstellung von Provider-Instanzen, unterstützt Laufzeitkonfiguration und Dependency Injection
3. **Einheitliches Protokoll**: Das ACP-Protokoll implementiert standardisierte Kommunikation zwischen Agenten, ein bidirektionales Kommunikationsmechanismus basierend auf JSON-RPC 2.0
4. **Aufgabenaufteilung**: Sinnvolle Zuweisung von Aufgaben an verschiedene Agenten, damit sie ihre Stärken ausspielen können, anstatt zu versuchen, einen Agenten alles tun zu lassen

Dieses Design löst nicht nur das Problem „mehrere Agenten bekämpfen sich“, sondern realisiert durch den Aufgabenflussmechanismus des Abenteurerteams auch die Automatisierung und Spezialisierung des Entwicklungsprozesses.
