## Tiefe Integration und praktische Nutzung von StreamJsonRpc in HagiCode

> Dieser Artikel beschreibt im Detail, wie das HagiCode-Projekt (früher PCode) die StreamJsonRpc-Kommunikationsbibliothek von Microsoft erfolgreich integriert hat, um die bestehende benutzerdefinierte JSON-RPC-Implementierung zu ersetzen, sowie die technischen Herausforderungen und architektonischen Probleme bei der Integration gelöst wurden.

## Hintergrund

StreamJsonRpc ist eine von Microsoft offiziell gepflegte JSON-RPC-Kommunikationsbibliothek für .NET und TypeScript, bekannt für ihre leistungsstarke Typsicherheit, automatische Proxy-Generierung und ausgereifte Ausnahmebehandlungsmechanismen. Im HagiCode-Projekt wurde die Integration von StreamJsonRpc beschlossen, um über ACP (Agent Communication Protocol) mit externen KI-Tools (wie iflow CLI, OpenCode CLI) kommunizieren zu können und die Wartungskosten sowie potenziellen Bugs der frühen benutzerdefinierten JSON-RPC-Implementierung zu beseitigen. Bei der Integration traten jedoch spezifische Herausforderungen des Streaming-JSON-RPC auf, insbesondere bei der Behandlung von Proxy-Zielbindungen und der Erkennung generischer Parameter.

Um diese Probleme zu lösen, trafen wir eine mutige Entscheidung: Das gesamte Build-System wurde neu aufgebaut. Die daraus resultierenden Veränderungen waren größer als man erwarten könnte – dazu später mehr.

## Analyse

Das Projekt befindet sich in einer kritischen Phase der ACP-Protokollintegration und steht vor folgenden technischen Herausforderungen und architektonischen Problemen:

### 1. Grenzen der benutzerdefinierten Implementierung
Die bestehende JSON-RPC-Implementierung befindet sich in `src/HagiCode.ClaudeHelper/AcpImp/` und enthält Komponenten wie `JsonRpcEndpoint` und `ClientSideConnection`. Die Wartung dieses benutzerdefinierten Codes ist kostenintensiv und es fehlen fortgeschrittene Funktionen ausgereifter Bibliotheken (wie Fortschrittsberichte, Abbruchunterstützung).

### 2. StreamJsonRpc-Integrationshindernisse
Bei dem Versuch, das bestehende `CallbackProxyTarget`-Muster auf StreamJsonRpc zu migrieren, stellte sich heraus, dass die Methode `_rpc.AddLocalRpcTarget(target)` nicht in der Lage war, über das Proxy-Muster erstellte Ziele zu erkennen. Konkret konnte StreamJsonRpc die Eigenschaften des generischen Typs `T` nicht automatisch in RPC-Methodenparameter aufteilen, was dazu führte, dass der Server die vom Client initiierten Methodenaufrufe nicht korrekt verarbeiten konnte.

### 3. Unübersichtliche Architekturschichten
Die bestehende `ClientSideConnection` vermischte Transportschicht (WebSocket/Stdio), Protokollschicht (JSON-RPC) und Geschäftsschicht (ACP Agent-Schnittstelle), was zu unklaren Zuständigkeiten und dem Problem fehlender Methodenbindungen in `AcpAgentCallbackRpcAdapter` führte.

### 4. Fehlende Protokollierung
Der WebSocket-Transportschicht fehlte die Protokollausgabe der ursprünglichen JSON-Inhalte, was die Fehlerdiagnose bei RPC-Kommunikationsproblemen erschwerte – es war schwer zu unterscheiden, ob es sich um ein Serialisierungs- oder ein Netzwerkproblem handelte.

## Lösung

Für die oben genannten Probleme wandten wir die folgenden systematischen Lösungsansätze aus drei Dimensionen an: Architektur-Refactoring, Bibliotheksintegration und Debugging-Verbesserung:

### 1. Vollständige Migration auf StreamJsonRpc

#### Alten Code entfernen
Löschung von `JsonRpcEndpoint.cs`, `AgentSideConnection.cs` und zugehörigen benutzerdefinierten Serialisierungskonvertern (`JsonRpcMessageJsonConverter` usw.).

#### Offizielle Bibliothek integrieren
Einführung des `StreamJsonRpc` NuGet-Pakets und Nutzung der `JsonRpc`-Klasse für die Kernkommunikationslogik.

#### Transportschicht abstrahieren
Definition der `IAcpTransport`-Schnittstelle zur einheitlichen Behandlung von `WebSocket`- und `Stdio`-Transportmodi, um die Entkopplung von Protokoll- und Transportschicht sicherzustellen.

```csharp
// IAcpTransport 接口定义
public interface IAcpTransport
{
    Task SendAsync(string message, CancellationToken cancellationToken = default);
    Task<string> ReceiveAsync(CancellationToken cancellationToken = default);
    Task CloseAsync(CancellationToken cancellationToken = default);
}

// WebSocket 传输实现
public class WebSocketTransport : IAcpTransport
{
    private readonly WebSocket _webSocket;

    public WebSocketTransport(WebSocket webSocket)
    {
        _webSocket = webSocket;
    }

    // 实现发送和接收方法
    // ...
}

// Stdio 传输实现
public class StdioTransport : IAcpTransport
{
    private readonly StreamReader _reader;
    private readonly StreamWriter _writer;

    public StdioTransport(StreamReader reader, StreamWriter writer)
    {
        _reader = reader;
        _writer = writer;
    }

    // 实现发送和接收方法
    // ...
}
```

### 2. Proxy-Zielerkennungsproblem beheben

#### `CallbackProxyTarget` analysieren
Überprüfung der bestehenden dynamischen Proxy-Generierungslogik zur Bestimmung der Grundursache, warum StreamJsonRpc das Ziel nicht erkennen kann (meistens weil das Proxy-Objekt keine tatsächlichen Methodensignaturen offenlegt oder StreamJsonRpc nicht unterstützte Parametertypen verwendet).

#### Parameterübergabe refaktorieren
Aufteilung generischer Eigenschaften in eindeutige RPC-Methodenparameter. Anstatt dynamischer Eigenschaften werden konkrete Request/Response DTOs (Data Transfer Objects) definiert, um sicherzustellen, dass StreamJsonRpc die Methodensignaturen durch Reflexion korrekt erkennen kann.

```csharp
// 原有的泛型属性方式
public class CallbackProxyTarget<T>
{
    public Func<T, Task> Callback { get; set; }
}

// 重构后的具体方法方式
public class ReadTextFileRequest
{
    public string FilePath { get; set; }
}

public class ReadTextFileResponse
{
    public string Content { get; set; }
}

public interface IAcpAgentCallback
{
    Task<ReadTextFileResponse> ReadTextFileAsync(ReadTextFileRequest request);
    // 其他方法...
}
```

#### `Attach` statt `AddLocalRpcTarget` verwenden
In komplexen Szenarien kann das manuelle Proxying des `JsonRpc`-Objekts und die Behandlung der `RpcConnection` flexibler sein als das direkte Hinzufügen von Zielen.

### 3. Methodenbindung und Protokollverbesserung implementieren

#### `AcpAgentCallbackRpcAdapter` implementieren
Sicherstellen, dass diese Komponente die StreamJsonRpc-Proxy-Schnittstelle explizit implementiert und die durch das ACP-Protokoll definierten Methoden (wie `ReadTextFileAsync`) den StreamJsonRpc-Callback-Handlern zuordnet.

#### Protokollierung integrieren
In der Nachrichtenverarbeitungspipeline von WebSocket oder Stdio werden JSON-RPC-Anfragen und -Antworten abgefangen und protokolliert. Mit `ILogger` wird der ursprüngliche Payload vor dem Parsing und nach der Serialisierung ausgegeben, um Formatierungsfehler zu identifizieren.

```csharp
// 日志增强的传输包装器
public class LoggingAcpTransport : IAcpTransport
{
    private readonly IAcpTransport _innerTransport;
    private readonly ILogger<LoggingAcpTransport> _logger;

    public LoggingAcpTransport(IAcpTransport innerTransport, ILogger<LoggingAcpTransport> logger)
    {
        _innerTransport = innerTransport;
        _logger = logger;
    }

    public async Task SendAsync(string message, CancellationToken cancellationToken = default)
    {
        _logger.LogTrace("Sending message: {Message}", message);
        await _innerTransport.SendAsync(message, cancellationToken);
    }

    public async Task<string> ReceiveAsync(CancellationToken cancellationToken = default)
    {
        var message = await _innerTransport.ReceiveAsync(cancellationToken);
        _logger.LogTrace("Received message: {Message}", message);
        return message;
    }

    public async Task CloseAsync(CancellationToken cancellationToken = default)
    {
        _logger.LogDebug("Closing connection");
        await _innerTransport.CloseAsync(cancellationToken);
    }
}
```

### 4. Architekturschichten-Refactoring

#### Transportschicht (`AcpRpcClient`)
Kapselung der StreamJsonRpc-Verbindung, verantwortlich für `InvokeAsync` und das Verbindungslebenszyklusmanagement.

```csharp
public class AcpRpcClient : IDisposable
{
    private readonly JsonRpc _rpc;
    private readonly IAcpTransport _transport;

    public AcpRpcClient(IAcpTransport transport)
    {
        _transport = transport;
        _rpc = new JsonRpc(new StreamRpcTransport(transport));
        _rpc.StartListening();
    }

    public async Task<TResponse> InvokeAsync<TResponse>(string methodName, object parameters)
    {
        return await _rpc.InvokeAsync<TResponse>(methodName, parameters);
    }

    public void Dispose()
    {
        _rpc.Dispose();
        _transport.Dispose();
    }

    // StreamRpcTransport 是对 IAcpTransport 的 StreamJsonRpc 适配器
    private class StreamRpcTransport : IDuplexPipe
    {
        // 实现 IDuplexPipe 接口
        // ...
    }
}
```

#### Protokollschicht (`IAcpAgentClient` / `IAcpAgentCallback`)
Definition klarer Client-to-Agent- und Agent-to-Client-Schnittstellen, Entfernung des zirkulären Abhängigkeitsfabrikmusters `Func<IAcpAgent, IAcpClient>` zugunsten von Dependency Injection oder direkter Callback-Registrierung.

## Praxis

Basierend auf den Best Practices von StreamJsonRpc und unserer Projekterfahrung sind dies die wichtigsten Empfehlungen:

### 1. Stark typisierte DTOs statt dynamischer Objekte
Der Kernvorteil von StreamJsonRpc liegt in der starken Typisierung. Verwenden Sie nicht `dynamic` oder `JObject` für Parameter. Für jede RPC-Methode sollten Sie konkrete C# POCO-Klassen als Parameter definieren. Dies löst nicht nur das Proxy-Zielerkennungsproblem, sondern ermöglicht auch die Erkennung von Typfehlern zur Kompilierzeit.

Beispiel: Ersetzen der generischen Eigenschaften in `CallbackProxyTarget` durch konkrete Klassen wie `ReadTextFileRequest` und `WriteTextFileRequest`.

### 2. Methodennamen explizit deklarieren
Verwenden Sie das `[JsonRpcMethod]`-Attribut, um RPC-Methodennamen explizit anzugeben, anstatt auf die Standardmethodennamenszuordnung zu vertrauen. Dies verhindert Aufruffehler aufgrund unterschiedlicher Namenskonventionen (z. B. PascalCase vs. camelCase).

```csharp
public interface IAcpAgentCallback
{
    [JsonRpcMethod("readTextFile")]
    Task<ReadTextFileResponse> ReadTextFileAsync(ReadTextFileRequest request);

    [JsonRpcMethod("writeTextFile")]
    Task WriteTextFileAsync(WriteTextFileRequest request);
}
```

### 3. Verbindungsstatus-Callbacks nutzen
StreamJsonRpc bietet das Ereignis `JsonRpc.ConnectionLost`. Sie sollten dieses Ereignis unbedingt überwachen, um unerwartete Prozessabbrüche oder Netzwerkunterbrechungen zu behandeln – dies ist zeitnaher als die bloße Nutzung des Orleans Grain-Ausfallerkennungsmechanismus.

```csharp
_rpc.ConnectionLost += (sender, e) =>
{
    _logger.LogError("RPC connection lost: {Reason}", e.ToString());
    // 处理重连逻辑或通知用户
};
```

### 4. Gestaffelte Protokollierung
- **Trace-Ebene**: Vollständiger JSON-Request/Response-Originaltext.
- **Debug-Ebene**: Methodenaufrufstack und Parameterzusammenfassung.
- **Hinweis**: Stellen Sie sicher, dass die Protokolle keine sensiblen Authorization Tokens oder Base64-codierte große Dateiinhalte enthalten.

### 5. Besonderheiten des Streaming-Transports behandeln
StreamJsonRpc unterstützt nativ `IAsyncEnumerable`. Bei der Implementierung von ACP-Streaming-Prompt-Antworten sollte `IAsyncEnumerable` direkt verwendet werden anstelle einer benutzerdefinierten Paginierungslogik. Dies vereinfacht den Streaming-Code erheblich.

```csharp
public interface IAcpAgentCallback
{
    [JsonRpcMethod("streamText")]
    IAsyncEnumerable<string> StreamTextAsync(StreamTextRequest request);
}
```

### 6. Adapter-Muster
Trennung von `ACPSession` und `ClientSideConnection` beibehalten. `ACPSession` sollte sich auf das Orleans-Zustandsmanagement und die Geschäftslogik (wie Nachrichten-Warteschlangen) konzentrieren und das StreamJsonRpc-Verbindungsobjekt durch Komposition statt Vererbung verwenden.

## Zusammenfassung

Durch die vollständige Integration von StreamJsonRpc hat das HagiCode-Projekt die Probleme der hohen Wartungskosten, der funktionalen Einschränkungen und der unübersichtlichen Architekturschichten der ursprünglichen benutzerdefinierten Implementierung erfolgreich gelöst. Die wichtigsten Verbesserungen umfassen:

1. Verwendung stark typisierter DTOs statt dynamischer Eigenschaften, was die Wartbarkeit und Zuverlässigkeit des Codes erhöht hat
2. Implementierung der Transportschichtabstraktion und Protokollschichttrennung, was die Klarheit der Architektur verbessert hat
3. Verbesserung der Protokollierungsfunktionen zur erleichterten Fehlerdiagnose bei Kommunikationsproblemen
4. Einführung von Streaming-Unterstützung zur Vereinfachung der Streaming-Implementierung

Diese Verbesserungen bieten HagiCode ein stabileres und effizienteres Kommunikationsfundament,使其能够更好地与外部 AI 工具进行交互，并为未来的功能扩展奠定了坚实的基础。
