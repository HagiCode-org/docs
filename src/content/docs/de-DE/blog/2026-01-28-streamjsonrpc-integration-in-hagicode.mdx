## Tiefe Integration und praktische Nutzung von StreamJsonRpc in HagiCode

> Dieser Artikel erklärt im Detail, wie das HagiCode-Projekt (früher PCode) erfolgreich die StreamJsonRpc-Kommunikationsbibliothek von Microsoft integriert hat, um die ursprüngliche benutzerdefinierte JSON-RPC-Implementierung zu ersetzen und gleichzeitig die technischen Schmerzpunkte und architektonischen Herausforderungen während des Integrationsprozesses zu bewältigen.

## Hintergrund

StreamJsonRpc ist die offiziell von Microsoft gepflegte JSON-RPC-Kommunikationsbibliothek für .NET und TypeScript, bekannt für ihre starke Typsicherheit, automatische Proxy-Generierung und ausgereifte Ausnahmebehandlungsmechanismen. Im HagiCode-Projekt wurde entschieden, StreamJsonRpc zu integrieren, um über ACP (Agent Communication Protocol) mit externen KI-Tools wie iflow CLI und OpenCode CLI zu kommunizieren und gleichzeitig die Wartungskosten und potenziellen Bugs der früheren benutzerdefinierten JSON-RPC-Implementierung zu beseitigen. Bei der Integration stießen wir jedoch auf spezifische Herausforderungen des Streaming-JSON-RPC, insbesondere bei der Verarbeitung von Proxy-Ziel-Bindung und der Erkennung generischer Parameter.

Um diese Schmerzpunkte zu beheben, haben wir eine gewagte Entscheidung getroffen: das gesamte Build-System von Grund auf neu aufzubauen. Die Auswirkungen dieser Entscheidung sind größer, als man erwarten könnte — dazu später mehr.

## Analyse

Das Projekt befindet sich derzeit in einer kritischen Phase der ACP-Protokollintegration und steht vor folgenden technischen Schmerzpunkten und architektonischen Herausforderungen:

### 1. Einschränkungen der benutzerdefinierten Implementierung
Die ursprüngliche JSON-RPC-Implementierung befindet sich in `src/HagiCode.ClaudeHelper/AcpImp/` und umfasst Komponenten wie `JsonRpcEndpoint` und `ClientSideConnection`. Die Wartung dieser benutzerdefinierten Codebasis ist kostspielig, und es fehlen die fortgeschrittenen Funktionen einer ausgereiften Bibliothek wie Fortschrittsberichterstattung und Abbruchunterstützung.

### 2. StreamJsonRpc-Integrationshindernisse
Bei dem Versuch, das bestehende `CallbackProxyTarget`-Muster auf StreamJsonRpc zu migrieren, stellten wir fest, dass die Methode `_rpc.AddLocalRpcTarget(target)` keine über das Proxy-Muster erstellten Ziele erkennen konnte. Konkret konnte StreamJsonRpc die Eigenschaften des generischen Typs `T` nicht automatisch in RPC-Methodenparameter aufteilen, wodurch die Serverseite beim Verarbeiten von Methodenaufrufen des Clients scheiterte.

### 3. Verworrene Architekturschichtung
Das bestehende `ClientSideConnection` vermengt die Transportschicht (WebSocket/Stdio), die Protokollschicht (JSON-RPC) und die Geschäftsschicht (ACP Agent-Schnittstelle), was zu unklaren Verantwortlichkeiten führt. Zudem fehlen Methodenbindungen in `AcpAgentCallbackRpcAdapter`.

### 4. Fehlende Protokolle
Die WebSocket-Transportschicht verfügt über keine Protokollierung des rohen JSON-Inhalts, was die Fehlersuche bei RPC-Kommunikationsproblemen erschwert — es lässt sich schwer unterscheiden, ob ein Serialisierungs- oder ein Netzwerkproblem vorliegt.

## Lösung

Um die oben genannten Probleme anzugehen, haben wir folgende systematische Lösung aus drei Dimensionen gewählt: Architektur-Refactoring, Bibliotheksintegration und verbessertes Debugging.

### 1. Vollständige Migration zu StreamJsonRpc

#### Alten Code entfernen
`JsonRpcEndpoint.cs`, `AgentSideConnection.cs` und zugehörige benutzerdefinierte Serialisierungskonverter wie `JsonRpcMessageJsonConverter` löschen.

#### Offizielle Bibliothek integrieren
Das `StreamJsonRpc` NuGet-Paket einführen und dessen `JsonRpc`-Klasse für die zentrale Kommunikationslogik nutzen.

#### Transportschicht abstrahieren
Die `IAcpTransport`-Schnittstelle definieren, um `WebSocket`- und `Stdio`-Transportmodi einheitlich zu verarbeiten und sicherzustellen, dass Protokoll- und Transportschicht entkoppelt sind.

```csharp
// Definition der IAcpTransport-Schnittstelle
public interface IAcpTransport
{
    Task SendAsync(string message, CancellationToken cancellationToken = default);
    Task<string> ReceiveAsync(CancellationToken cancellationToken = default);
    Task CloseAsync(CancellationToken cancellationToken = default);
}

// WebSocket-Transportimplementierung
public class WebSocketTransport : IAcpTransport
{
    private readonly WebSocket _webSocket;

    public WebSocketTransport(WebSocket webSocket)
    {
        _webSocket = webSocket;
    }

    // Sende- und Empfangsmethoden implementieren
    // ...
}

// Stdio-Transportimplementierung
public class StdioTransport : IAcpTransport
{
    private readonly StreamReader _reader;
    private readonly StreamWriter _writer;

    public StdioTransport(StreamReader reader, StreamWriter writer)
    {
        _reader = reader;
        _writer = writer;
    }

    // Sende- und Empfangsmethoden implementieren
    // ...
}
```

### 2. Proxy-Ziel-Erkennungsprobleme beheben

#### `CallbackProxyTarget` analysieren
Die bestehende dynamische Proxy-Generierungslogik untersuchen und die Ursache identifizieren, warum StreamJsonRpc sie nicht erkennt. In den meisten Fällen geschieht dies, weil das Proxy-Objekt die tatsächlichen Methodensignaturen nicht öffentlich verfügbar macht oder Parametertypen verwendet, die von StreamJsonRpc nicht unterstützt werden.

#### Parameterübergabe refaktorieren
Generische Eigenschaften in explizite RPC-Methodenparameter aufteilen. Anstatt sich auf dynamische Eigenschaften zu verlassen, konkrete Request/Response-DTOs definieren, damit StreamJsonRpc die Methodensignaturen per Reflexion korrekt erkennen kann.

```csharp
// Ursprünglicher generischer Eigenschaftsansatz
public class CallbackProxyTarget<T>
{
    public Func<T, Task> Callback { get; set; }
}

// Refaktorierter konkreter Methodenansatz
public class ReadTextFileRequest
{
    public string FilePath { get; set; }
}

public class ReadTextFileResponse
{
    public string Content { get; set; }
}

public interface IAcpAgentCallback
{
    Task<ReadTextFileResponse> ReadTextFileAsync(ReadTextFileRequest request);
    // Weitere Methoden...
}
```

#### `Attach` statt `AddLocalRpcTarget` verwenden
In bestimmten komplexen Szenarien kann das manuelle Proxying des `JsonRpc`-Objekts und die Behandlung von `RpcConnection` flexibler sein als das direkte Hinzufügen eines lokalen Ziels.

### 3. Methodenbindung und verbesserte Protokollierung implementieren

#### `AcpAgentCallbackRpcAdapter` implementieren
Sicherstellen, dass diese Komponente die StreamJsonRpc-Proxy-Schnittstelle explizit implementiert und die vom ACP-Protokoll definierten Methoden wie `ReadTextFileAsync` auf StreamJsonRpc-Callback-Handler abbildet.

#### Protokollierung integrieren
In der WebSocket- oder Stdio-Nachrichtenverarbeitungspipeline den rohen Text von JSON-RPC-Anfragen und -Antworten abfangen und protokollieren. Mit `ILogger` den rohen Payload vor dem Parsen und nach der Serialisierung ausgeben, um Formatierungsfehler leichter diagnostizieren zu können.

```csharp
// Transport-Wrapper mit erweiterter Protokollierung
public class LoggingAcpTransport : IAcpTransport
{
    private readonly IAcpTransport _innerTransport;
    private readonly ILogger<LoggingAcpTransport> _logger;

    public LoggingAcpTransport(IAcpTransport innerTransport, ILogger<LoggingAcpTransport> logger)
    {
        _innerTransport = innerTransport;
        _logger = logger;
    }

    public async Task SendAsync(string message, CancellationToken cancellationToken = default)
    {
        _logger.LogTrace("Sending message: {Message}", message);
        await _innerTransport.SendAsync(message, cancellationToken);
    }

    public async Task<string> ReceiveAsync(CancellationToken cancellationToken = default)
    {
        var message = await _innerTransport.ReceiveAsync(cancellationToken);
        _logger.LogTrace("Received message: {Message}", message);
        return message;
    }

    public async Task CloseAsync(CancellationToken cancellationToken = default)
    {
        _logger.LogDebug("Closing connection");
        await _innerTransport.CloseAsync(cancellationToken);
    }
}
```

### 4. Architekturschichtung refaktorieren

#### Transportschicht (`AcpRpcClient`)
Die StreamJsonRpc-Verbindung kapseln und für `InvokeAsync`-Aufrufe sowie das Verbindungslebenszyklusmanagement verantwortlich machen.

```csharp
public class AcpRpcClient : IDisposable
{
    private readonly JsonRpc _rpc;
    private readonly IAcpTransport _transport;

    public AcpRpcClient(IAcpTransport transport)
    {
        _transport = transport;
        _rpc = new JsonRpc(new StreamRpcTransport(transport));
        _rpc.StartListening();
    }

    public async Task<TResponse> InvokeAsync<TResponse>(string methodName, object parameters)
    {
        return await _rpc.InvokeAsync<TResponse>(methodName, parameters);
    }

    public void Dispose()
    {
        _rpc.Dispose();
        _transport.Dispose();
    }

    // StreamRpcTransport ist der StreamJsonRpc-Adapter für IAcpTransport
    private class StreamRpcTransport : IDuplexPipe
    {
        // IDuplexPipe-Schnittstelle implementieren
        // ...
    }
}
```

#### Protokollschicht (`IAcpAgentClient` / `IAcpAgentCallback`)
Klare Client-zu-Agent- und Agent-zu-Client-Schnittstellen definieren. Das zyklische Factory-Muster `Func<IAcpAgent, IAcpClient>` entfernen und durch Dependency Injection oder direkte Callback-Registrierung ersetzen.

## Praxis

Basierend auf den Best Practices für StreamJsonRpc und unserer Projekterfahrung sind dies die wichtigsten Empfehlungen für die Implementierung:

### 1. Stark typisierte DTOs sind dynamischen Objekten vorzuziehen
Der Kernvorteil von StreamJsonRpc liegt in der starken Typisierung. Verwenden Sie nicht `dynamic` oder `JObject` zur Parameterübergabe. Definieren Sie stattdessen für jede RPC-Methode explizite C# POCO-Klassen als Parameter. Dies löst nicht nur das Problem der Proxy-Ziel-Erkennung, sondern fängt auch Typfehler bereits zur Compile-Zeit ab.

Beispiel: Die generischen Eigenschaften in `CallbackProxyTarget` durch konkrete Klassen wie `ReadTextFileRequest` und `WriteTextFileRequest` ersetzen.

### 2. Methodennamen explizit deklarieren
Verwenden Sie das `[JsonRpcMethod]`-Attribut, um RPC-Methodennamen explizit anzugeben, anstatt sich auf die Standard-Methodennamenszuordnung zu verlassen. Dies verhindert Aufruffehler durch Unterschiede in der Namenskonvention wie PascalCase versus camelCase.

```csharp
public interface IAcpAgentCallback
{
    [JsonRpcMethod("readTextFile")]
    Task<ReadTextFileResponse> ReadTextFileAsync(ReadTextFileRequest request);

    [JsonRpcMethod("writeTextFile")]
    Task WriteTextFileAsync(WriteTextFileRequest request);
}
```

### 3. Verbindungsstatus-Callbacks nutzen
StreamJsonRpc bietet das Ereignis `JsonRpc.ConnectionLost`. Sie sollten dieses Ereignis unbedingt überwachen, um unerwartete Prozessbeendigungen oder Netzwerkunterbrechungen zu behandeln — das ist zeitnaher als sich nur auf die Orleans Grain-Ausfallerkennung zu verlassen.

```csharp
_rpc.ConnectionLost += (sender, e) =>
{
    _logger.LogError("RPC connection lost: {Reason}", e.ToString());
    // Reconnect-Logik behandeln oder Benutzer benachrichtigen
};
```

### 4. Protokolle nach Schichten aufzeichnen
- **Trace-Ebene**: Den vollständigen rohen JSON-Request/Response-Payload aufzeichnen.
- **Debug-Ebene**: Methodenaufruf-Stacks und Parameterzusammenfassungen aufzeichnen.
- **Hinweis**: Sicherstellen, dass die Protokolle keine sensiblen Authorization-Tokens oder Base64-codierte große Dateiinhalte enthalten.

### 5. Besonderheiten des Streaming-Transports behandeln
StreamJsonRpc unterstützt nativ `IAsyncEnumerable`. Bei der Implementierung von Streaming-Prompt-Antworten für ACP sollten Sie `IAsyncEnumerable` direkt verwenden statt einer benutzerdefinierten Paginierungslogik. Das vereinfacht den Code für die Streaming-Verarbeitung erheblich.

```csharp
public interface IAcpAgentCallback
{
    [JsonRpcMethod("streamText")]
    IAsyncEnumerable<string> StreamTextAsync(StreamTextRequest request);
}
```

### 6. Adapter-Muster
`ACPSession` und `ClientSideConnection` getrennt halten. `ACPSession` sollte sich auf Orleans-Zustandsverwaltung und Geschäftslogik wie das Einreihen von Nachrichten konzentrieren und das StreamJsonRpc-Verbindungsobjekt über Komposition statt Vererbung verwenden.

## Zusammenfassung

Durch die umfassende Integration von StreamJsonRpc hat das HagiCode-Projekt die hohen Wartungskosten, die funktionalen Einschränkungen und die unübersichtliche Architekturschichtung der ursprünglichen benutzerdefinierten Implementierung erfolgreich behoben. Die wichtigsten Verbesserungen umfassen:

1. Ersetzen dynamischer Eigenschaften durch stark typisierte DTOs zur Verbesserung von Wartbarkeit und Zuverlässigkeit
2. Implementierung einer Transportschicht-Abstraktion und Protokollschicht-Trennung zur Verbesserung der architektonischen Klarheit
3. Verstärkung der Protokollierungsfunktionen zur leichteren Diagnose von Kommunikationsproblemen
4. Einführung von Streaming-Unterstützung zur Vereinfachung der Streaming-Implementierung

Diese Verbesserungen bieten HagiCode ein stabileres und effizienteres Kommunikationsfundament, ermöglichen eine bessere Interaktion mit externen KI-Tools und legen einen soliden Grundstein für zukünftige Funktionserweiterungen.
