## Code schnell im Browser bearbeiten: VSCode-Web-Integration

> Nach der AI-Analyse: Wie öffnet man sofort einen Editor im Browser zur Bearbeitung? Dieser Artikel teilt die Erfahrungen der Integration von code-server im HagiCode-Projekt und ermöglicht eine nahtlose Verbindung zwischen AI-Assistent und Code-Editor-Erlebnis.

## Hintergrund

Im Zeitalter der AI-unterstützten Programmierung müssen Entwickler häufig schnell Code anzeigen und bearbeiten. Der traditionelle Entwicklungsablauf: Projekt in der Desktop-IDE öffnen, Datei finden, bearbeiten, speichern. Aber in manchen Szenarien fühlt sich dieser Ablauf irgendwie nicht ganz richtig an.

Szenario 1: Remote-Entwicklung. Bei Verwendung eines AI-Assistenten wie HagiCode läuft das Backend möglicherweise auf einem Remote-Server oder in einem Container, und lokal kann nicht direkt auf Projektdateien zugegriffen werden. Jedes Mal, wenn Sie Code anzeigen oder ändern möchten, müssen Sie sich über SSH oder andere Methoden verbinden – eine zerrissene Erfahrung. Das fühlt sich an, als möchten Sie jemanden treffen, aber es gibt eine dicke Glasscheibe zwischen Ihnen: Sie können es sehen, aber nicht berühren.

Szenario 2: schnelle Vorschau. Nachdem der AI-Assistent den Code analysiert hat, möchten Benutzer vielleicht nur schnell eine Datei durchsuchen oder kleine Änderungen vornehmen. Das Starten einer vollständigen Desktop-IDE wirkt schwerfällig, ein leichtgewichtiger Editor im Browser entspricht eher dem Bedürfnis nach "schnellem Ansehen". Schließlich, wer will für einen kurzen Blick einen großen Aufwand betreiben?

Szenario 3: geräteübergreifende Zusammenarbeit. Bei der Arbeit auf verschiedenen Geräten bietet ein Editor im Browser einen einheitlichen Zugriffspunkt, ohne dass auf jedem Gerät eine Entwicklungsumgebung eingerichtet werden muss. Das spart Zeit – schließlich ist das Leben kurz, warum也应重复劳动?

Um diese Schmerzpunkte zu lösen, haben wir VSCode Web in das HagiCode-Projekt integriert. Damit werden AI-Assistent und Code-Editor-Erlebnis nahtlos verbunden – nach der AI-Analyse können Benutzer sofort in derselben Browsersitzung den Editor zur Bearbeitung öffnen, ohne Anwendungen zu wechseln. Diese Erfahrung, nun ja, ist wie wenn Sie es wollen, ist es da.

## Über HagiCode

Die in diesem Artikel vorgestellte Lösung stammt aus unseren praktischen Erfahrungen im [HagiCode](https://hagicode.com)-Projekt. HagiCode ist ein KI-gesteuerter Code-Assistent, der die Entwicklungseffizienz durch natürliche Sprachinteraktion verbessern soll. Bei der Entwicklung stellten wir fest, dass Benutzer häufig zwischen AI-Analyse und Code-Bearbeitung schnell wechseln mussten, was uns dazu veranlasste zu erkunden, wie man den Editor direkt in den Browser integrieren kann.

Projektadresse: [github.com/HagiCode-org/site](https://github.com/HagiCode-org/site)

## Technologieauswahl: Warum code-server?

Unter den vielen VSCode-Web-Lösungen haben wir uns für code-server entschieden. Diese Entscheidung hatte tatsächlich mehrere Überlegungen:

Umfangreiche Funktionen. code-server ist die Web-Version von VSCode und unterstützt die meisten Funktionen der Desktop-Version, einschließlich Erweiterungssystem, intelligenter Vervollständigung, Debugging usw. Das bedeutet, dass Benutzer auch im Browser ein Editor-Erlebnis nahe der Desktop-Version erhalten können. Schließlich, wer will bei Funktionen Kompromisse eingehen?

Flexible Bereitstellung. code-server kann als eigenständiger Service ausgeführt werden und unterstützt Docker-Container-Bereitstellung, was zur Architektur von HagiCode passt. Unser Backend ist in C# geschrieben, das Frontend ist React, und wir kommunizieren über REST-API mit dem code-server-Service. Das ist wie Bauklötze: jeder Block hat seinen Platz.

Sichere Authentifizierung. code-server hat einen eingebauten Connection-Token-Mechanismus, um unbefugten Zugriff zu verhindern. Jede Sitzung hat ein eindeutiges Token, um sicherzustellen, dass nur autorisierte Benutzer auf den Editor zugreifen können. Sicherheit – wenn man sie hat, weiß man ihre Bedeutung.

## Architekturdesign

Die VSCode-Web-Integration von HagiCode verwendet eine Architektur mit Trennung von Frontend und Backend.

### Frontend-Service-Schicht

Das Frontend kapselt die Interaktion mit dem Backend durch `vscodeServerService.ts`:

```typescript
// Projekt öffnen
export async function openProjectInCodeServer(
  id: string,
  currentInterfaceLanguage?: string,
): Promise<VsCodeServerLaunchResponseDto>

// Vault öffnen
export async function openVaultInCodeServer(
  id: string,
  path?: string,
  currentInterfaceLanguage?: string,
): Promise<VsCodeServerLaunchResponseDto>
```

Der Unterschied zwischen diesen beiden Methoden besteht darin, dass `openProjectInCodeServer` zum Öffnen des gesamten Projekts verwendet wird, während `openVaultInCodeServer` zum Öffnen eines bestimmten Pfads im Vault verwendet wird. Für MonoSpecs-Multi-Repository-Projekte erstellt das System automatisch Arbeitsbereichsdateien. Klare Aufgabenteilung – jeder macht seinen Job, das reicht.

### Backend-Service-Schicht

Das Backends `VaultAppService.cs` implementiert die Kernlogik:

```csharp
public async Task<VsCodeServerLaunchResponseDto> OpenInCodeServerAsync(
    string id,
    string? relativePath = null,
    string? currentInterfaceLanguage = null,
    CancellationToken cancellationToken = default)
{
    // 1. Einstellungen abrufen und prüfen, ob aktiviert
    var settings = await _vsCodeServerSettingsService.GetResolvedSettingsAsync(cancellationToken);
    if (!settings.Enabled) {
        throw new BusinessException(VsCodeServerErrorCodes.Disabled, "VSCode Server is disabled.");
    }

    // 2. Vault abrufen und Startverzeichnis auflösen
    var vault = await RequireVaultAsync(id, cancellationToken);
    var launchDirectory = ResolveLaunchDirectory(vault, relativePath);

    // 3. Sicherstellen, dass code-server läuft und Laufzeitinformationen abrufen
    var runtime = await _vsCodeServerManager.EnsureStartedAsync(settings, cancellationToken);

    // 4. Spracheinstellungen auflösen
    var language = _vsCodeServerSettingsService.ResolveLaunchLanguage(
        settings.Language,
        currentInterfaceLanguage);

    // 5. Start-URL erstellen
    return new VsCodeServerLaunchResponseDto {
        LaunchUrl = AppendQueryString(runtime.BaseUrl, new Dictionary<string, string?> {
            ["folder"] = launchDirectory,
            ["tkn"] = runtime.ConnectionToken,
            ["vscode-lang"] = language,
        }),
        ConnectionToken = runtime.ConnectionToken,
        OpenMode = "folder",
        Runtime = VsCodeServerSettingsService.MapRuntime(
            await _vsCodeServerManager.GetRuntimeSnapshotAsync(cancellationToken)),
    };
}
```

Die Verantwortung dieser Methode ist klar: Einstellungen prüfen, Pfade auflösen, Service starten, URL erstellen. Die Methode `ResolveLaunchDirectory` führt Pfadsicherheitsprüfungen durch, um Pfad-Traversierungsangriffe zu verhindern. Code ist wie Poesie – jede Zeile hat ihren eigenen Rhythmus.

### Automatische Laufzeitverwaltung

Das Backend verwaltet den code-server-Prozess durch `VsCodeServerManager`:

- Prozessstatus prüfen
- Automatisches Starten gestoppter Services
- Zurückgeben von Laufzeit-Snapshots (Port, Prozess-ID, Startzeit usw.)

Dieses Design ermöglicht es dem System, den Lebenszyklus von code-server automatisch zu verarbeiten, ohne dass Benutzer Service-Prozesse manuell verwalten müssen. Schließlich ist das Leben bereits komplex genug genug – automatisieren Sie, was sich automatisieren lässt.

### Sprachfolgemechanismus

HagiCode unterstützt mehrsprachige Benutzeroberflächen, und code-server muss dieser Einstellung folgen. Das System unterstützt drei Sprachmodi:

- `follow`: Der aktuellen Oberflächensprache folgen
- `zh-CN`: Fest auf Chinesisch
- `en-US`: Fest auf Englisch

Der URL-Parameter `vscode-lang` wird an code-server übergeben, um sicherzustellen, dass die Editor-Sprache mit der HagiCode-Oberfläche übereinstimmt. Einheitlichkeit bei der Sprache – erst dann fühlt es sich angenehm an.

### MonoSpecs-Multi-Repository-Arbeitsbereiche

Für MonoSpecs-Projekte (mono-repo mit mehreren Unter-Repositories) erstellt das System automatisch `.code-workspace`-Dateien:

```csharp
private async Task<string> CreateWorkspaceFileAsync(Project project, Guid projectId)
{
    var folders = await ResolveWorkspaceFoldersAsync(project.Path);
    var workspaceDocument = new {
        folders = folders.Select(path => new { path }).ToArray(),
    };
    // Workspace-Datei generieren...
}
```

Auf diese Weise können mehrere Unter-Repositories in einer code-server-Instanz gleichzeitig bearbeitet werden, was für große mono-repo-Projekte sehr praktisch ist. Mehrere Repositories in einem Fenster – wie mehrere Geschichten in einem Buch.

## Frontend-Integration

Das HagiCode-Frontend verwendet React + TypeScript, und die Integration von code-server ist nicht besonders kompliziert.

### Schnellaktionsbuttons

Code Server-Button zur Projektkarte hinzufügen:

```tsx
// QuickActionsZone.tsx
<Button
  size="sm"
  variant="default"
  onClick={() => onAction({ type: 'open-code-server' })}
>
  <Globe className="h-3 w-3 mr-1" />
  <span className="text-xs">{t('project.openCodeServer')}</span>
</Button>
```

Dieser Button löst die Öffnen-Aktion aus und ruft das Backend-API auf, um die Start-URL zu erhalten. Ein Button, eine Aktion – einfach und direkt.

### Öffnen-Aktion verarbeiten

```tsx
const handleAction = async (action: ProjectAction) => {
  if (action.type === 'open-code-server') {
    const response = await openProjectInCodeServer(project.id, i18n.language);
    window.open(response.launchUrl, '_blank', 'noopener,noreferrer');
  }
};
```

Verwendung von `window.open`, um code-server in einem neuen Tab zu öffnen, die Parameter `noopener,noreferrer` gewährleisten die Sicherheit. Bei Sicherheit kann man nicht vorsichtig genug sein.

### Vault-Bearbeitungseinstieg

Ähnlichen Bearbeiten-Button zur Vault-Liste hinzufügen:

```tsx
const handleEditVault = async (vault: VaultItemDto) => {
  const response = await openVaultInCodeServer(vault.id);
  window.open(response.launchUrl, '_blank', 'noopener,noreferrer');
};
```

Projekt und Vault verwenden die gleiche Öffnen-Methode, was die Konsistenz der Interaktion aufrechterhält. Die Bedeutung der Konsistenz ist nicht geringer als die der Funktion selbst.

## URL-Erstellungslogik

Das URL-Format von code-server hat tatsächlich seine Besonderheiten:

**Ordnermodus**:
```
http://{host}:{port}/?folder={path}&tkn={token}&vscode-lang={lang}
```

**Arbeitsbereichsmodus**:
```
http://{host}:{port}/?workspace={workspacePath}&tkn={token}&vscode-lang={lang}
```

Dabei ist `tkn` das Verbindungstoken, das bei jedem Start von code-server automatisch generiert wird und den Zugriffssicherheit gewährleistet. Der Parameter `vscode-lang` steuert die Editor-Oberflächensprache. Jeder dieser Parameter hat seine eigene Funktion – keiner ist entbehrlich.

## Verwendungsszenarien

### Szenario 1: AI-unterstützte Code-Review

Benutzer unterhalten sich mit HagiCode, die AI analysiert den Projektcode und erkennt potenzielle Probleme. Benutzer klicken auf "Open in Code Server"-Button, um den Editor direkt im Browser zu öffnen, die Problemdateien anzusehen und zu beheben, dann kehren sie zu HagiCode zurück, um das Gespräch fortzusetzen. Der gesamte Prozess wird im Browser abgeschlossen, ohne Anwendungen zu wechseln. Dieses Gefühl, nun ja, ist wie fließendes Wasser – reibungslos.

### Szenario 2: Vault-Lernmaterialbearbeitung

Benutzer erstellen einen Vault zum Lernen eines Open-Source-Projekts und möchten Lernnotizen im Verzeichnis `docs/` hinzufügen. Durch code-server können Markdown-Dateien direkt im Browser bearbeitet werden, und nach dem Speichern kann HagiCode die aktualisierten Notizeninhalte synchron lesen. Dies ist sehr nützlich für den Aufbau einer persönlichen Wissensdatenbank. Wissen – je mehr man es anhäuft, desto wertvoller wird es.

### Szenario 3: MonoSpecs-Multi-Repository-Entwicklung

MonoSpecs-Projekte enthalten mehrere Unter-Repositories, und code-server erstellt automatisch Multi-Ordner-Arbeitsbereiche. Benutzer können im Browser gleichzeitig Code aus mehreren Repositories bearbeiten und nach Änderungen an die jeweiligen Git-Repositories committen. Diese Arbeitsweise ist besonders geeignet für Szenarien, die repository-übergreifende Änderungen erfordern. Mehrere Repositories gleichzeitig ändern – wie mehrere Aufgaben gleichzeitig bearbeiten, erfordert etwas Geschicklichkeit.

## Sicherheitsüberlegungen

Bei der Implementierung der code-server-Integration ist Sicherheit ein Problem, das besondere Aufmerksamkeit erfordert. Schließlich, bei Sicherheitsproblemen ist es dann zu spät.

### Connection Token

Der connection-token wird zufällig generiert und sollte nicht preisgegeben werden. Die Verwendung in einer HTTPS-Umgebung wird empfohlen, um zu verhindern, dass das Token von Man-in-the-Middle-Angriffen abgefangen wird. Sensible Informationen – besser gut schützen.

### Dateipfadsicherheit

Das Backend implementiert Pfad-Traversierungsprüfungen:

```csharp
private static string ResolveLaunchDirectory(VaultRegistryEntry vault, string? relativePath)
{
    var vaultRoot = EnsureTrailingSeparator(Path.GetFullPath(vault.PhysicalPath));
    var combinedPath = Path.GetFullPath(Path.Combine(vaultRoot, relativePath ?? "."));
    if (!combinedPath.StartsWith(vaultRoot, StringComparison.OrdinalIgnoreCase))
    {
        throw new BusinessException(VaultRelativePathTraversalCode, "Relative path traversal detected.");
    }
    return combinedPath;
}
```

Dieser Code stellt sicher, dass Benutzer nicht durch `../` etc. auf Dateien außerhalb des Vault-Verzeichnisses zugreifen können. Grenzprüfungen – besser gemacht als nicht gemacht.

### Berechtigungssteuerung

Der code-server-Prozess läuft mit entsprechenden Benutzerberechtigungen, um den Zugriff auf sensible Systemdateien zu vermeiden. Es wird empfohlen, einen dedizierten Benutzer für den code-server-Service zu verwenden. Berechtigungssteuerung – das, was sein sollte, sollte auch da sein.

## Leistungsoptimierung

code-server verbraucht Serverressourcen. Hier sind einige Optimierungsvorschläge:

- CPU-/Speichernutzung überwachen und bei Bedarf Ressourcenlimits anpassen
- Bei großen Projekten möglicherweise die Zeitüberschreitung erhöhen
- Automatische Bereinigung bei Sitzungszeitüberschreitung implementieren, Ressourcen freigeben
- Verwendung von Caching in Betracht ziehen, um wiederholte Berechnungen zu reduzieren

HagiCode bietet eine Laufzeit-Statusüberwachungs-API, und das Frontend kann den aktuellen Status über `getVsCodeServerSettings()` abrufen:

```typescript
const { settings, runtime } = await getVsCodeServerSettings();
// runtime.status: 'disabled' | 'stopped' | 'starting' | 'running' | 'unhealthy'
// runtime.baseUrl: "http://localhost:8080"
// runtime.processId: 12345
```

Dieses Design ermöglicht es Benutzern, den Gesundheitsstatus von code-server klar zu verstehen und Probleme schnell zu lokalisieren, wenn sie auftreten. Status sichtbar – dann weiß man, wo man steht.

## Benutzererfahrungs-Details

Bei der Implementation haben wir einige Details entdeckt, die die Benutzererfahrung beeinflussen und besondere Aufmerksamkeit verdienen:

Das erste Öffnen von code-server möglicherweise warten müssen, bis es startet – diese Zeit kann von wenigen Sekunden bis zu einer halben Minute variieren. Es wird empfohlen, im Frontend einen Ladestatus anzuzeigen, damit Benutzer wissen, dass das System verarbeitet. Warten – mit Feedback ist es in Ordnung.

Browser können Popups blockieren, daher müssen Benutzer möglicherweise manuell dazu aufgefordert werden, sie zuzulassen. HagiCode zeigt beim ersten Öffnen eine Anleitung, die Benutzern zeigt, wie sie Browserberechtigungen einstellen. Benutzererfahrung zeigt sich genau in diesen Details.

Es wird empfohlen, den Laufzeitstatus anzuzeigen (startend/läuft/Fehler), damit Benutzer bei Problemen wissen, ob es ein Serverproblem oder ein eigenes Operationsproblem ist. Zu wissen, wo das Problem liegt, gibt zumindest Sicherheit.

## Konfigurationsbeispiel

Die Konfiguration von code-server ist nicht besonders komplex:

```json
{
  "vscodeServer": {
    "enabled": true,
    "host": "0.0.0.0",
    "port": 8080,
    "language": "follow"
  }
}
```

`enabled` steuert den Funktionsschalter, `host` und `port` geben die Listener-Adresse an, `language` setzt den Sprachmodus. Diese Konfigurationen können über die UI-Oberfläche geändert werden und wirksam werden, ohne Neustart. Einfache Dinge sind oft am besten zu verwenden.

## Zusammenfassung

Die VSCode-Web-Integration von HagiCode bietet eine elegante Lösung: eine nahtlose Verbindung zwischen AI-Assistent und Code-Editor-Erlebnis. Durch die Integration von code-server im Browser können Benutzer schnell auf die Analyseergebnisse der AI reagieren und den vollständigen Prozess von Analyse bis Bearbeitung in derselben Browsersitzung abschließen.

Die wichtigsten Vorteile dieser Lösung sind: erstens einheitliche Erfahrung – Projekt und Vault verwenden die gleiche Öffnen-Methode; zweitens Multi-Repository-Unterstützung – MonoSpecs-Projekte erstellen automatisch Arbeitsbereiche; drittens Sicherheit und Kontrolle – Laufzeit-Statusüberwachung und Pfadsicherheitsprüfungen.

Die in diesem Artikel vorgestellte Lösung wurde von HagiCode aus der tatsächlichen Entwicklung zusammengefasst. Wenn Sie diese Lösung wertvoll finden, bedeutet das, dass unsere technische Praxis nicht schlecht ist – dann lohnt es sich, HagiCode selbst zu beachten.
