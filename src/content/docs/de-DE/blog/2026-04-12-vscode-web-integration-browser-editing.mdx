## Schneller Code-Editor im Browser: VSCode Web Integration in der Praxis

> Nachdem die KI die Codeanalyse abgeschlossen hat, wie oeffnet man sofort einen Editor im Browser und beginnt mit Aenderungen? Dieser Artikel teilt unsere praktische Erfahrung bei der Integration von code-server im HagiCode-Projekt, um eine nahtlose Bruecke zwischen dem KI-Assistenten und dem Code-Bearbeitungserlebnis zu schaffen.

## Hintergrund

Im Zeitalter der KI-gestuetzten Programmierung muessen Entwickler haeufig Code schnell pruefen und bearbeiten. Der traditionelle Workflow ist einfach: Projekt in einer Desktop-IDE oeffnen, Datei suchen, bearbeiten und speichern. Aber in manchen Situationen fuehlt sich dieser Ablauf immer etwas holprig an.

Szenario eins: Remote-Entwicklung. Bei der Verwendung eines KI-Assistenten wie HagiCode laeuft das Backend moeglicherweise auf einem Remote-Server oder in einem Container, und lokale Maschinen koennen nicht direkt auf die Projektdateien zugreifen. Jedes Mal, wenn Sie Code pruefen oder aendern muessen, muessen Sie sich ueber SSH oder eine andere Methode verbinden, und die Erfahrung fuehlt sich fragmentiert an. Es ist, als wuerde man durch eine dicke Glasscheibe jemanden treffen wollen: man sieht ihn, kann ihn aber nicht erreichen.

Szenario zwei: schnelle Vorschauen. Nachdem der KI-Assistent den Code analysiert hat, moechte der Benutzer moeglicherweise nur schnell eine Datei durchsehen oder eine kleine Aenderung vornehmen. Das Starten einer vollstaendigen Desktop-IDE fuehlt sich schwer an, waehrend ein leichtgewichtiger In-Browser-Editor besser zum Beduerfnis eines „schnellen Blicks" passt. Schliesslich moechte niemand eine gesamte Toolchain mobilisieren, nur um einen kurzen Blick zu werfen.

Szenario drei: geraetuebergreifende Zusammenarbeit. Bei der Arbeit ueber verschiedene Geraete hinweg bietet ein browserbasierter Editor einen einheitlichen Zugriffspunkt, ohne dass jede Maschine mit einer Entwicklungsumgebung konfiguriert werden muss. Das allein spart viel Muehe. Das Leben ist kurz; warum dieselbe Einrichtungsarbeit immer wiederholen?

Um diese Schmerzpunkte zu loesen, haben wir VSCode Web in das HagiCode-Projekt integriert. Dadurch koennen der KI-Assistent und das Code-Bearbeitungserlebnis nahtlos verbunden werden: Nach der KI-Analyse koennen Benutzer sofort einen Editor oeffnen und Aenderungen in derselben Browsersitzung vornehmen, ohne die Anwendung zu wechseln. Es ist die Art von Erfahrung, bei der sie einfach da ist, wenn man sie braucht.

## Ueber HagiCode

Die in diesem artikel geteilte Loesung stammt aus unserer praktischen Erfahrung im [HagiCode](https://hagicode.com)-Projekt. HagiCode ist ein KI-gesteuerter Coding-Assistent, der die Entwicklungseffizienz durch natuerliche Sprachinteraktion verbessern soll. Waehrend der Entwicklung stellten wir fest, dass Benutzer haeufig schnell zwischen KI-Analyse und Code-Bearbeitung wechseln muessen, was uns dazu brachte, zu erkunden, wie der Editor direkt in den Browser integriert werden kann.

Projekt-Repository: [github.com/HagiCode-org/site](https://github.com/HagiCode-org/site)

## Technische Entscheidung: Warum code-server?

Unter den vielen verfuegbaren VSCode Web-Loesungen haben wir uns fuer code-server entschieden. Dafuer gab es einige konkrete Gruende.

Funktionsvollstaendigkeit. code-server ist die Web-Version von VSCode und unterstuetzt die meisten Desktop-Funktionen, einschliesslich des Erweiterungssystems, intelligenter Vorschlaege, Debugging und mehr. Das bedeutet, dass Benutzer im Browser ein Bearbeitungserlebnis bekommen, das dem Desktop sehr nahe kommt. Schliesslich moechte wirklich niemand bei der Funktionalitaet Kompromisse eingehen.

Flexible Bereitstellung. code-server kann als unabhaengiger Dienst laufen und unterstuetzt auch Docker-basierte Bereitstellung, was gut zur HagiCode-Architektur passt. Unser Backend ist in C# geschrieben, das Frontend verwendet React, und beide kommunizieren mit dem code-server-Dienst ueber REST-APIs. Es ist, als wuerde man mit Bausteinen bauen: jedes Stueck hat seinen Platz.

Sichere Authentifizierung. code-server enthaelt einen eingebauten `connection-token`-Mechanismus, um unbefugten Zugriff zu verhindern. Jede Sitzung hat einen eindeutigen Token, sodass nur autorisierte Benutzer den Editor oeffnen koennen. Sicherheit ist so etwas, das man erst vollstaendig zu schaetzen lernt, wenn man sie hat.

## Architekturentwurf

HagiCodes VSCode Web Integration verwendet eine Frontend-Backend-getrennte Architektur.

### Frontend-Dienstschicht

Das Frontend kapselt die Interaktion mit dem Backend durch `vscodeServerService.ts`:

```typescript
// Projekt oeffnen
export async function openProjectInCodeServer(
  id: string,
  currentInterfaceLanguage?: string,
): Promise<VsCodeServerLaunchResponseDto>

// Vault oeffnen
export async function openVaultInCodeServer(
  id: string,
  path?: string,
  currentInterfaceLanguage?: string,
): Promise<VsCodeServerLaunchResponseDto>
```

Der Unterschied zwischen diesen beiden Methoden ist einfach: `openProjectInCodeServer` oeffnet das gesamte Projekt, waehrend `openVaultInCodeServer` einen bestimmten Pfad innerhalb eines Vaults oeffnet. Bei MonoSpecs-Multi-Repository-Projekten erstellt das System automatisch eine Workspace-Datei. Klare Verantwortlichkeiten genuegen oft, wenn jeder Teil seine eigene Arbeit gut macht.

### Backend-Dienstschicht

Das Backend `VaultAppService.cs` implementiert die Kernlogik:

```csharp
public async Task<VsCodeServerLaunchResponseDto> OpenInCodeServerAsync(
    string id,
    string? relativePath = null,
    string? currentInterfaceLanguage = null,
    CancellationToken cancellationToken = default)
{
    // 1. Einstellungen abrufen und pruefen, ob die Funktion aktiviert ist
    var settings = await _vsCodeServerSettingsService.GetResolvedSettingsAsync(cancellationToken);
    if (!settings.Enabled) {
        throw new BusinessException(VsCodeServerErrorCodes.Disabled, "VSCode Server is disabled.");
    }

    // 2. Vault abrufen und das Startverzeichnis aufloesen
    var vault = await RequireVaultAsync(id, cancellationToken);
    var launchDirectory = ResolveLaunchDirectory(vault, relativePath);

    // 3. Sicherstellen, dass code-server laeuft und Runtime-Info abrufen
    var runtime = await _vsCodeServerManager.EnsureStartedAsync(settings, cancellationToken);

    // 4. Spracheinstellungen aufloesen
    var language = _vsCodeServerSettingsService.ResolveLaunchLanguage(
        settings.Language,
        currentInterfaceLanguage);

    // 5. Start-URL aufbauen
    return new VsCodeServerLaunchResponseDto {
        LaunchUrl = AppendQueryString(runtime.BaseUrl, new Dictionary<string, string?> {
            ["folder"] = launchDirectory,
            ["tkn"] = runtime.ConnectionToken,
            ["vscode-lang"] = language,
        }),
        ConnectionToken = runtime.ConnectionToken,
        OpenMode = "folder",
        Runtime = VsCodeServerSettingsService.MapRuntime(
            await _vsCodeServerManager.GetRuntimeSnapshotAsync(cancellationToken)),
    };
}
```

Diese Methode hat eine sehr klare Verantwortung: Einstellungen pruefen, Pfade aufloesen, Dienst starten und URL aufbauen. Darunter fuehrt die Methode `ResolveLaunchDirectory` Pfadsicherheitspruefungen durch, um Path-Traversal-Angriffe zu verhindern. Code kann sich manchmal ein wenig wie Poesie anfuehlen, wenn jede Zeile einen Zweck hat.

### Automatische Runtime-Verwaltung

Das Backend verwaltet den code-server-Prozess durch `VsCodeServerManager`:

- Prozessstatus pruefen
- Gestoppte Dienste automatisch starten
- Runtime-Snapshots wie Port, Prozess-ID und Startzeit zurueckgeben

Dieses Design laesst das System den Lebenszyklus von code-server automatisch verwalten, sodass Benutzer Dienstprozesse nicht manuell verwalten muessen. Das Leben ist bereits kompliziert genug; alles, was automatisiert werden kann, sollte es auch sein.

### Sprachsynchronisation

HagiCode unterstuetzt eine mehrsprachige Benutzeroberflaeche, und code-server muss dieser Einstellung folgen. Das System unterstuetzt drei Sprachmodi:

- `follow`: der aktuellen Oberflaechensprache folgen
- `zh-CN`: fest auf Chinesisch
- `en-US`: fest auf Englisch

Die Einstellung wird ueber den `vscode-lang`-URL-Parameter an code-server uebergeben, sodass die Editorsprache mit der HagiCode-Oberflaeche konsistent bleibt. Sprache fuehlt sich am besten an, wenn sie einheitlich ist.

### MonoSpecs Multi-Repository Workspace

Fuer MonoSpecs-Projekte, die mehrere Sub-Repositories innerhalb eines Monorepos enthalten, erstellt das System automatisch eine `.code-workspace`-Datei:

```csharp
private async Task<string> CreateWorkspaceFileAsync(Project project, Guid projectId)
{
    var folders = await ResolveWorkspaceFoldersAsync(project.Path);
    var workspaceDocument = new {
        folders = folders.Select(path => new { path }).ToArray(),
    };
    // Workspace-Datei generieren...
}
```

Dadurch ist es moeglich, mehrere Sub-Repositories in derselben code-server-Instanz zu bearbeiten, was besonders praktisch fuer grosse Monorepo-Projekte ist. Mehrere Repositories in einem Fenster – das fuehlt sich an wie mehrere Geschichten, die in demselben Buch gesammelt sind.

## Frontend-Integration

Das HagiCode-Frontend verwendet React + TypeScript, und die Integration von code-server ist nicht besonders kompliziert.

### Schnellaktionsbutton

Fuegen Sie einen Code Server-Button zur Projektkarte hinzu:

```tsx
// QuickActionsZone.tsx
<Button
  size="sm"
  variant="default"
  onClick={() => onAction({ type: 'open-code-server' })}
>
  <Globe className="h-3 w-3 mr-1" />
  <span className="text-xs">{t('project.openCodeServer')}</span>
</Button>
```

Dieser Button loest die Oeffnen-Aktion aus und ruft die Backend-API auf, um die Start-URL zu erhalten. Ein Button, eine Aktion, direkt und einfach.

### Behandlung der Oeffnen-Aktion

```tsx
const handleAction = async (action: ProjectAction) => {
  if (action.type === 'open-code-server') {
    const response = await openProjectInCodeServer(project.id, i18n.language);
    window.open(response.launchUrl, '_blank', 'noopener,noreferrer');
  }
};
```

Verwenden Sie `window.open`, um code-server in einem neuen Tab zu oeffnen. Die Parameter `noopener,noreferrer` bieten zusaetzliche Sicherheit. Bei Sicherheit gibt es kein „zu vorsichtig."

### Vault-Bearbeitungseingang

Fuegen Sie einen aehnlichen Bearbeitungsbutton in der Vault-Liste hinzu:

```tsx
const handleEditVault = async (vault: VaultItemDto) => {
  const response = await openVaultInCodeServer(vault.id);
  window.open(response.launchUrl, '_blank', 'noopener,noreferrer');
};
```

Projekte und Vaults verwenden denselben Oeffnen-Mechanismus, was die Interaktion konsistent haelt. Konsistenz ist fast genauso wichtig wie die Funktion selbst.

## URL-Aufbaulogik

Das URL-Format fuer code-server hat einige Details, die Beachtung verdienen.

**Ordnermodus**:
```
http://{host}:{port}/?folder={path}&tkn={token}&vscode-lang={lang}
```

**Workspace-Modus**:
```
http://{host}:{port}/?workspace={workspacePath}&tkn={token}&vscode-lang={lang}
```

Hierbei ist `tkn` das Verbindungstoken. Es wird bei jedem Start von code-server automatisch generiert und stellt einen sicheren Zugriff sicher. Der Parameter `vscode-lang` steuert die Sprache der Editor-Benutzeroberflaeche. Jeder dieser Parameter hat seine Rolle zu spielen.

## Nutzungsszenarien

### Szenario eins: KI-gestuetztes Code-Review

Der Benutzer spricht mit HagiCode, die KI analysiert den Projektcode und findet ein potenzielles Problem, dann klickt der Benutzer auf den „In Code Server oeffnen"-Button, um den Editor direkt im Browser zu oeffnen, die betroffene Datei zu pruefen, sie zu beheben und zu HagiCode zurueckzukehren, um das Gespraech fortzusetzen. Der gesamte Ablauf findet im Browser statt, ohne die Anwendung zu wechseln. Das fuehlt sich an wie fliessendes Wasser – geschmeidig.

### Szenario zwei: Bearbeitung von Studienmaterialien in einem Vault

Ein Benutzer erstellt einen Vault zum Studium eines Open-Source-Projekts und moechte Lernnotizen im `docs/`-Verzeichnis hinzufuegen. Mit code-server kann er Markdown-Dateien direkt im Browser bearbeiten, speichern und HagiCode die aktualisierten Notizen sofort lesen lassen. Dies ist besonders nuetzlich fuer den Aufbau einer persoenlichen Wissensbasis. Wissen wird umso wertvoller, je mehr man es ansammelt.

### Szenario drei: MonoSpecs Multi-Repository-Entwicklung

Ein MonoSpecs-Projekt enthaelt mehrere Sub-Repositories, und code-server erstellt automatisch einen Multi-Ordner-Workspace. Im Browser koennen Benutzer Code ueber mehrere Repositories gleichzeitig bearbeiten und Aenderungen dann in ihre jeweiligen Git-Repositories committen. Dieser Workflow ist besonders gut geeignet fuer Aenderungen, die mehrere Repositories betreffen. Gleichzeitiges Bearbeiten mehrerer Repositories erfordert etwas Technik, aehnlich wie das gleichzeitige Bearbeiten mehrerer Aufgaben.

## Sicherheitsueberlegungen

Bei der Implementierung der code-server-Integration verdient Sicherheit besondere Aufmerksamkeit. Wenn bei der Sicherheit etwas schiefgeht, bemerkt man es immer zu spaet.

### Verbindungstoken

Der `connection-token` wird zufaellig generiert und sollte nicht offengelegt werden. Er sollte am besten unter HTTPS verwendet werden, um zu verhindern, dass der Token von einem Man-in-the-Middle abgefangen wird. Sensible Informationen sind es wert, richtig gesichert zu werden.

### Dateipfadsicherheit

Das Backend implementiert Pfad-Traversal-Pruefungen:

```csharp
private static string ResolveLaunchDirectory(VaultRegistryEntry vault, string? relativePath)
{
    var vaultRoot = EnsureTrailingSeparator(Path.GetFullPath(vault.PhysicalPath));
    var combinedPath = Path.GetFullPath(Path.Combine(vaultRoot, relativePath ?? "."));
    if (!combinedPath.StartsWith(vaultRoot, StringComparison.OrdinalIgnoreCase))
    {
        throw new BusinessException(VaultRelativePathTraversalCode, "Relative path traversal detected.");
    }
    return combinedPath;
}
```

Dieser Code stellt sicher, dass Benutzer nicht ueber `../` oder aehnliche Muster auf Dateien ausserhalb des Vault-Verzeichnisses zugreifen koennen. Grenzpruefungen sind immer besser durchgefuehrt als uebersprungen.

### Berechtigungssteuerung

Der code-server-Prozess sollte mit angemessenen Benutzerberechtigungen laufen, damit er nicht auf sensible Systemdateien zugreifen kann. Es ist am besten, den code-server-Dienst unter einem dedizierten Benutzer laufen zu lassen. Berechtigungssteuerung ist so etwas Grundlegendes, das man immer beachten sollte.

## Leistungsoptimierung

code-server verbraucht Serverressourcen, also hier einige Optimierungsvorschlaege:

- CPU- und Speichernutzung ueberwachen und Ressourcenlimits bei Bedarf anpassen
- Grosse Projekte koennen laengere Timeouts erfordern
- Automatische Sitzungs-Timeout-Bereinigung implementieren, um Ressourcen freizugeben
- Caching in Betracht ziehen, um wiederholte Berechnungen zu reduzieren

HagiCode bietet eine Runtime-Statusueberwachungs-API, und das Frontend kann `getVsCodeServerSettings()` aufrufen, um den aktuellen Zustand abzurufen:

```typescript
const { settings, runtime } = await getVsCodeServerSettings();
// runtime.status: 'disabled' | 'stopped' | 'starting' | 'running' | 'unhealthy'
// runtime.baseUrl: "http://localhost:8080"
// runtime.processId: 12345
```

Dieses Design ermoeglicht es Benutzern, den Gesundheitszustand von code-server klar zu verstehen und Probleme schnell zu lokalisieren, wenn etwas schiefgeht. Wenn der Status sichtbar ist, fuehlen sich die Menschen mehr in der Kontrolle.

## Benutzererfahrungs-Details

Waehrend der Implementation haben wir einige Details entdeckt, die die Benutzererfahrung merklich beeinflussen und besondere Beachtung verdienen.

Das erste Oeffnen von code-server erfordert moeglicherweise Warten auf den Start, und diese Verzoegerung kann von wenigen Sekunden bis zu einer halben Minute reichen. Es ist eine gute Idee, im Frontend einen Ladezustand anzuzeigen, damit Benutzer wissen, dass das System noch arbeitet. Warten ist einfacher, wenn es Rueckmeldung gibt.

Browser koennen das Pop-up blockieren, daher sollten Benutzer aufgefordert werden, es manuell zuzulassen. Beim ersten Start zeigt HagiCode eine Anleitung, die erklaert, wie die notwendigen Browserberechtigungen erteilt werden. Benutzererfahrung lebt oft genau in diesen kleinen Details.

Es ist auch eine gute Idee, den Laufzeitstatus wie Starten, Laufen oder Fehler anzuzeigen, sodass Benutzer bei Problemen schnell erkennen koennen, ob das Problem auf der Serverseite oder in ihrem eigenen Vorgang liegt. Zu wissen, wo das Problem liegt, gibt zumindest einen Anhaltspunkt.

## Konfigurationsbeispiel

Die Konfiguration fuer code-server ist nicht kompliziert:

```json
{
  "vscodeServer": {
    "enabled": true,
    "host": "0.0.0.0",
    "port": 8080,
    "language": "follow"
  }
}
```

`enabled` steuert, ob die Funktion aktiviert ist, `host` und `port` definieren die Abhoeradresse, und `language` setzt den Sprachmodus. Diese Einstellungen koennen ueber die Benutzeroberflaeche geaendert werden und sofort wirksam werden. Einfache Dinge sind oft am einfachsten zu verwenden.
