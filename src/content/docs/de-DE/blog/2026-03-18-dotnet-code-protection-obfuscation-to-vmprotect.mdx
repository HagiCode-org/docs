## .NET Code-Schutz in der Praxis: Von Obfuscation bis VM-Schutz

> Dieser Artikel stellt vor, wie man eine mehrschichtige Code-Schutzstrategie in .NET-Projekten implementiert, die von grundlegender Obfuscation bis hin zu professionellem VM-Schutz alle Lösungen abdeckt.

## Hintergrund

Bei der Entwicklung von .NET-Anwendungen ist der Schutz von Kerncode (wie Lizenzvalidierung, Geschäftslogik, sensible Konfigurationen etc.) vor Dekompilierung und Reverse-Engineering, nun ja, ein unvermeidliches Thema. Mit der Reifung des .NET-Ökosystems haben Entwickler verschiedene Möglichkeiten zum Code-Schutz, von integrierten Obfuscation-Attributen bis hin zu professionellen Virtualisierungsschutztools – die Auswahl ist durchaus groß.

Als ein komplexes mehrsprachiges Monorepo-Projekt enthält HagiCode Desktop-Anwendungen, Build-Systeme und Lizenzverwaltungsfunktionen. Der Code beinhaltet unvermeidlich Lizenzvalidierungslogik, sensible Konfigurationen (wie API-Schlüssel, Produkt-IDs) sowie geschäftskritische Kernlogik – diese Dinge müssen gut geschützt werden, schließlich möchte niemand, dass sein Herzblut einfach von anderen eingesehen wird.

Dieser Artikel teilt die im HagiCode-Projekt tatsächlich angewendeten Code-Schutzlösungen und fasst den vollständigen Prozess von der Fehlersuche bis zur Optimierung zusammen. Vielleicht kann er Ihnen einige Anregungen geben.

## Über HagiCode

[HagiCode](https://hagicode.com) ist ein Open-Source-KI-Coding-Assistant-Projekt, das sich der Bereitstellung intelligenter Programmierunterstützung für Entwickler widmet. Das Projekt verwendet eine Monorepo-Architektur und pflegt gleichzeitig mehrere Komponenten wie VSCode-Erweiterungen, Backend-KI-Dienste und plattformübergreifende Desktop-Clients. Diese Komplexität aus mehreren Sprachen und Plattformen macht den Code-Schutz zu einer unvermeidlichen technischen Herausforderung – nun ja, so ist das nun mal bei so komplexen Projekten.

Die in diesem Artikel vorgestellten Lösungen sind genau die, die wir während der Entwicklung von HagiCode durch praktische Erfahrung und Optimierung erarbeitet haben. Wenn Sie erfahren möchten, wie wir diese technischen Herausforderungen gelöst haben, lesen Sie weiter – vielleicht gibt es einige unerwartete Erkenntnisse.

## Kerninhalte

### 1. Microsofts integriertes Obfuscation-Attribut

.NET Framework bietet ein integriertes `[ObfuscationAttribute]`, das grundlegendste und am häufigsten verwendete Code-Obfuscation-Markierung. Dieses Attribut befindet sich im Namespace `System.Reflection` und ermöglicht grundlegenden Schutz ohne die Notwendigkeit von Drittanbieter-Tools – durchaus praktisch.

**Kernmerkmale**:
- `Feature`-Eigenschaft: Spezifiziert Obfuscation-Features wie `"ultra"` (hohe Obfuscation), `"all"` (vollständige Obfuscation)
- `Exclude`-Eigenschaft: `true` bedeutet von Obfuscation ausschließen, `false` bedeutet Obfuscation anwenden
- Kann auf Klassen, Methoden, Eigenschaften und andere Typmitglieder angewendet werden

Im HagiCode-Projekt sehen Sie ein praktisches Verwendungsbeispiel:

```csharp
[Obfuscation(Feature = "ultra", Exclude = false)]
public async Task<LicenseValidationResult?> ValidateLicenseAsync(...)
```

Die **Vorteile** dieser Methode sind durchaus offensichtlich:
- Keine zusätzlichen Abhängigkeiten, in .NET Framework integriert – spart viel Arbeit
- Kann von Drittanbieter-Obfuscation-Tools erkannt und verarbeitet werden
- Erhöht die Größe der kompilierten Assembly nicht signifikant

Allerdings gibt es auch Einschränkungen: Es ist nur eine Markierung, der tatsächliche Obfuscation-Effekt hängt von der Tool-Implementierung ab und kann keinen VM-Schutz-level Sicherheit bieten – aber das ist auch in Ordnung, schließlich wurde es nicht dafür entwickelt.

### 2. VMP (Virtual Machine Protection)

VMP ist ein professionelles Code-Schutztool, das fortgeschrittenen Schutz bietet, indem Code in VM-Anweisungen kompiliert wird. Im Gegensatz zu einfacher Namensobfuscation konvertiert VMP Code-Logik tatsächlich in eine Form, die von herkömmlichen Decompilern nicht wiederhergestellt werden kann – das ist schon ziemlich beeindruckend.

**Schutzlevel-Klassifizierung**:

| Level | Virtualisierung | Mutation | Anti-Debugging | String-Verschlüsselung | Anwendungsszenarien |
|-------|------------------|----------|----------------|------------------------|---------------------|
| HIGH | full | high | Aktiviert | Aktiviert | Lizenzvalidierung, Sitzungskonkurrenz, sensible Konstanten |
| MEDIUM | partial | medium | Aktiviert | Aktiviert | Geschäftslogik, Domänenmodelle |
| LOW | none | low | Deaktiviert | Deaktiviert | Hilfsklassen, nicht-kritischer Code |

Das HagiCode-Projekt definiert ein deklaratives Attributsystem zum Markieren von zu schützendem Code:

```csharp
// Hochprioritätsschutz
[VmProtect(VmProtectionPriority.High, Reason = "Contains license verification logic")]
public class KeygenClient { ... }

// Schutz ausschließen
[VmExclude(Reason = "Public API that must remain unchanged")]
public class PublicApi { ... }

// Vererbungsschutz
[VmProtect(Priority.High, ProtectDerived = true)]
public class BaseLicenseValidator { ... }
```

### 3. Buildzeit-Schutzstrategie

VMP-Schutz wirkt nicht nur zur Laufzeit, sondern muss auch im Build-Prozess automatisiert verarbeitet werden – manuell zu machen ist schließlich zu mühsam. Das Build-System von HagiCode unterstützt mehrere Modi:

- **Windows-Nativmodus**: Direkter Aufruf des VMProtect-Tools
- **Linux-Docker-Container-Modus**: VMP im Container ausführen (Lösung von Kompatibilitätsproblemen über Plattformen hinweg)
- **Attribut-Scanning**: Automatisches Auffinden von Schutzmarkierungen im Code
- **Verifizierungsmechanismus**: Bestätigung, dass Schutz erfolgreich angewendet wurde

In Kombination sind diese Funktionen ziemlich praktisch.

## Lösungen

### 1. Verwendung von Microsofts integriertem Obfuscation-Attribut

Wenden Sie `ObfuscationAttribute` direkt im Code an:

```csharp
using System.Reflection;

[Obfuscation(Feature = "ultra", Exclude = false)]
public class LicenseService
{
    [Obfuscation(Feature = "ultra", Exclude = false)]
    public async Task<bool> ValidateLicenseAsync(string key)
    {
        // Lizenzvalidierungslogik
    }

    [Obfuscation(Feature = "flow", Exclude = false)]
    private string DecryptToken(string encrypted)
    {
        // Entschlüsselungslogik
    }
}
```

Manchmal müssen Test-Assemblys auf interne Member zugreifen, während die Sicherheit des Produktionscodes erhalten bleibt:

```csharp
// AssemblyInfo.cs
[assembly: InternalsVisibleTo("HagiCode.Application.Tests")]
[assembly: InternalsVisibleTo("DynamicProxyGenAssembly2")] // for Moq
```

So ist das Testen viel bequemer – schließlich muss der Code getestet werden.

### 2. Definition von benutzerdefinierten VMP-Schutzattributen

Erstellen Sie benutzerdefinierte Schutzattribute zur Steuerung des VMP-Verhaltens:

```csharp
using System;

[AttributeUsage(AttributeTargets.Class | AttributeTargets.Method | AttributeTargets.Property)]
public class VmProtectAttribute : Attribute
{
    public VmProtectionPriority Priority { get; set; }
    public string? Reason { get; set; }
    public bool ProtectDerived { get; set; }
}

[AttributeUsage(AttributeTargets.Class | AttributeTargets.Method | AttributeTargets.Property)]
public class VmExcludeAttribute : Attribute
{
    public string? Reason { get; set; }
}

public enum VmProtectionPriority
{
    None = 0,
    Low = 1,
    Medium = 2,
    High = 3
}
```

Selbstdefinierte Attribute sind auch bequemer zu verwenden – man kennt schließlich seine eigenen Anforderungen.

### 3. VMP-Konfigurationsdatei

```yaml
# vmp_config.yml
protection:

  default_level: "medium"

  high:
    virtualization: "full"
    mutation: "high"
    anti_debug: true
    anti_dump: true
    encrypt_strings: true
    encrypt_resources: true

  medium:
    virtualization: "partial"
    mutation: "medium"
    anti_debug: true
    encrypt_strings: true

  low:
    virtualization: "none"
    mutation: "low"
    anti_debug: false
```

Eine klare Konfiguration erleichtert die spätere Wartung.

## Praktischer Leitfaden

### 1. Schutzpraxis für kritische Komponenten

Gemäß der code-protection-Spezifikation von HagiCode müssen die folgenden Komponenten mit HIGH-Priorität geschützt werden:

```csharp
// Produktionsumgebungskonstanten - müssen verschlüsselt und durch VMP geschützt sein
[VmProtect(VmProtectionPriority.High, Reason = "Production constants")]
public static class ProductionConstants
{
    // Verschlüsselter String-Accessor, durch VMP geschützt
    [VmProtect(VmProtectionPriority.High)]
    public static string GetLicenseServerUrl(IOptions<LicenseOptions> options) => ...;
}

// Lizenzvalidierungslogik
[VmProtect(VmProtectionPriority.High, Reason = "License verification logic")]
public class KeygenClient : IKeygenClient
{
    [Obfuscation(Feature = "ultra", Exclude = false)]
    public async Task<LicenseValidationResult?> ValidateLicenseAsync(...) { ... }
}

// Maschinenerkennungs-Service
[VmProtect(VmProtectionPriority.High)]
public class MachineFingerprintService : IMachineFingerprintService { ... }
```

Kritischer Code muss schwerpunktmäßig geschützt werden – wenn die Kernlogik durchsickert, wird es problematic.

### 2. String-Verschlüsselung mit Laufzeitentschlüsselung

Verschlüsselung von Strings zur Buildzeit, Entschlüsselung zur Laufzeit:

```csharp
public static class StringDecryption
{
    [VmProtect(VmProtectionPriority.High, Reason = "CRITICAL SECURITY")]
    public static string DecryptString(byte[] encryptedData, byte[] key, byte[] iv)
    {
        using var aes = Aes.Create();
        aes.Key = key;
        aes.IV = iv;

        using var decryptor = aes.CreateDecryptor();
        using var ms = new MemoryStream(encryptedData);
        using var cs = new CryptoStream(ms, decryptor, CryptoStreamMode.Read);
        using var reader = new StreamReader(cs);

        return reader.ReadToEnd();
    }
}

// Produktionskonstanten-Accessor (Lazy Loading + Caching)
public static class ProductionConstants
{
    private static string? _cachedLicenseServerUrl;

    public static string GetLicenseServerUrl(IOptions<LicenseOptions> options)
    {
        if (_cachedLicenseServerUrl == null)
        {
            var encrypted = GetEncryptedLicenseServerUrl();
#if DEBUG
            _cachedLicenseServerUrl = options.Value.PrimaryServer.Url;
#else
            _cachedLicenseServerUrl = StringDecryption.DecryptString(
                encrypted,
                GetEncryptionKey(),
                GetEncryptionIV());
#endif
        }
        return _cachedLicenseServerUrl;
    }
}
```

Die String-Verschlüsselung ist auch ein ziemlich wichtiger Schritt – sensible Informationen sollten nicht im Klartext herumliegen.

### 3. VMP-Schutzverifizierung

Nach dem Build muss überprüft werden, ob der Schutz erfolgreich angewendet wurde – sonst weiß man nicht, ob der Schutz wirkt:

```csharp
// Verifizierungsskript-Beispiel
public bool VerifyProtection(string assemblyPath)
{
    // 1. VMP-Signatur prüfen
    var bytes = File.ReadAllBytes(assemblyPath);
    var vmpSignature = Encoding.ASCII.GetBytes("VMProtect");
    if (bytes.Any(b => vmpSignature.Contains(b)))
    {
        return true;
    }

    // 2. Dateigrößenänderung prüfen (nach Schutz normalerweise größer)
    var originalInfo = new FileInfo(assemblyPath.Replace(".dll", ".bak"));
    if (originalInfo.Exists)
    {
        var sizeRatio = (double)new FileInfo(assemblyPath).Length / originalInfo.Length;
        return sizeRatio > 1.1;
    }

    return false;
}
```

Zu verifizieren ist immer gut – vermeidet Probleme, wenn man später gar nicht weiß, was passiert ist.

### 4. Zu beachtende Punkte

Hier gibt es einige Fallstricke, die besonders beachtet werden müssen – schließlich sind wir alle schon hineingetappt:

1. **Nicht allen Code obfuscaten**: Öffentliche APIs, Schnittstellendefinitionen und DTO-Klassen benötigen normalerweise keinen Schutz, übermäßige Obfuscation beeinträchtigt Leistung und Debugging, HagiCode-Projekt hat hier bereits negative Erfahrungen gemacht – das muss beachtet werden

2. **Schlüssel-Accessor schützen**: Die Abrufmethode für Verschlüsselungsschlüssel muss die gleiche oder höhere Schutzebene wie die verschlüsselten Daten haben, sonst ist es sinnlos

3. **Balance zwischen Test und Produktion**: DEBUG-Builds sollten Verschlüsselung überspringen, um Entwicklung und Debugging zu erleichtern, RELEASE-Builds vollständigen Schutz aktivieren, bedenken Sie bedingte Kompilierung mit `#if DEBUG` zur Unterscheidung – so ist die Entwicklung auch bequemer

4. **Docker-Umgebung beachten**: Die Ausführung von VMP unter Linux erfordert containerisierte Lösungen, um die Kompatibilität der Schutztools sicherzustellen, HagiCode verwendet eine Container-Lösung mit Wine + VMP – so werden plattformübergreifende Probleme gelöst

5. **Verifizierung unerlässlich**: Nach Abschluss des Builds muss überprüft werden, ob der Schutz erfolgreich war, sonst kann sensibler Code offengelegt werden – der vorherige Verifizierungscode hat genau diesen Zweck, besser einmal zu prüfen

## Zusammenfassung

Durch diese mehrschichtige Schutzstrategie hat HagiCode ein umfassendes Code-Sicherheitssystem von grundlegender Obfuscation bis hin zu VM-Schutz realisiert:

- **Erste Schicht**: Verwendung von `ObfuscationAttribute` für grundlegende Markierungen, Hinweise für Drittanbieter-Tools bereitstellen
- **Zweite Schicht**: Deklaration von Schutzabsicht und Priorität durch benutzerdefinierte `VmProtectAttribute`
- **Dritte Schicht**: VMP-VM-Schutz konvertiert kritischen Code in nicht umkehrbare VM-Anweisungen
- **Vierte Schicht**: Automatisches Scannen und Anwenden von Schutz zur Buildzeit, Verifizierung der Schutzergebnisse

Diese Lösung kann sowohl gewöhnlichen Decompilertools als auch fortgeschrittenen Reverse-Engineering-Angriffen standhalten. Wenn Sie auch eine .NET-Anwendung mit Code-Schutz entwickeln, hoffe ich, dass dies Ihnen einige Referenzen geben kann – selbst ein kleiner Einblick ist genug.
