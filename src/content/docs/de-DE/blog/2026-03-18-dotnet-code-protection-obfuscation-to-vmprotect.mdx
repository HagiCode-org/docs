## .NET-Codeschutz in der Praxis: Von Verschleierung bis VM-Schutz

> Dieser Artikel erklärt, wie man eine mehrschichtige Codeschutzstrategie in .NET-Projekten implementiert, die den vollständigen Weg von der grundlegenden Verschleierung bis zum professionellen VM-Schutz abdeckt.

## Hintergrund

In der .NET-Anwendungsentwicklung ist der Schutz von Kerncode wie Lizenzvalidierung, Geschäftslogik und sensiblen Konfiguration vor Dekompilierung und Reverse Engineering ein Thema, das man nicht vermeiden kann.Da das .NET-OEkosystem gereift ist, stehen Entwicklern eine Reihe von Schutzoptionen zur Verfügung, von integrierten Verschleierungsattributen bis hin zu professionellen virtualisierungsbasierten Schutzwerkzeugen.

Als komplexes mehrsprachiges Monorepo-Projekt umfasst HagiCode Desktop-Anwendungen, Build-Systeme und Lizenzverwaltungsfunktionen. Der Code enthält unvermeidlich Lizenzvalidierungslogik, sensible Konfigurationen wie API-Schlüssel und Produkt-IDs sowie geschäftskritische Logik. Diese Teile benötigen ernsthaften Schutz, denn niemand möchte, dass seine harte Arbeit so leicht offengelegt wird.

Dieser Artikel teilt den Codeschutzansatz, den wir im HagiCode-Projekt tatsächlich angewendet haben, und fasst die gesamte Reise von frühen Fehlern bis zur späteren Optimierung zusammen. Hoffentlich gibt es Ihnen einige nützliche Ideen.

## Ueber HagiCode

[HagiCode](https://hagicode.com) ist ein Open-Source-KI-Code-Assistent-Projekt, das Entwicklern eine intelligente Programmiererfahrung bieten möchte. Das Projekt verwendet eine Monorepo-Architektur und pflegt gleichzeitig eine VSCode-Erweiterung, Backend-KI-Dienste, einen plattformübergreifenden Desktop-Client und mehr. Diese mehrsprachige, plattformübergreifende Komplexität macht den Codeschutz zu einer technischen Herausforderung, der wir uns stellen müssen.

Der in diesem Artikel geteilte Ansatz ist das Ergebnis echter Versuche und Irrtümer während der HagiCode-Entwicklung. Wenn Sie sehen möchten, wie wir diese technischen Probleme gelöst haben, lesen Sie weiter. Vielleicht finden Sie einige unerwartete Erkenntnisse.

## Kerninhalt

### 1. Microsofts integriertes Verschleierungsattribut

.NET Framework bietet ein integriertes `[ObfuscationAttribute]`, das die grundlegendste und am häufigsten verwendete Code-Verschleierungsmarkierung ist. Dieses Attribut befindet sich im Namespace `System.Reflection` und ermöglicht es, eine Basis-Code-Verschleierung anzuwenden, ohne Drittanbieter-Werkzeuge einzuführen.

**Kernfunktionen**:
- `Feature`-Eigenschaft: Gibt die Verschleierungsfunktion an, z. B. `"ultra"` (hohe Verschleierung) oder `"all"` (vollständige Verschleierung)
- `Exclude`-Eigenschaft: `true` bedeutet Ausschluss von der Verschleierung, `false` bedeutet Verschleierung anwenden
- Kann auf Klassen, Methoden, Eigenschaften und andere Typmitglieder angewendet werden

Im HagiCode-Projekt können Sie es wie folgt verwendet sehen:

```csharp
[Obfuscation(Feature = "ultra", Exclude = false)]
public async Task<LicenseValidationResult?> ValidateLicenseAsync(...)
```

Die **Vorteile** dieses Ansatzes sind recht offensichtlich:
- Keine zusätzlichen Abhängigkeiten; es ist in .NET Framework out-of-the-box integriert
- Kann von Drittanbieter-Verschleierungswerkzeugen erkannt und verarbeitet werden
- Erhöht die Größe der kompilierten Assembly nicht signifikant

Jedoch hat es auch Einschränkungen. Es ist nur eine Markierung, und das tatsächliche Verschleierungsergebnis hängt von der Werkzeugimplementierung ab. Es kann keinen VM-Schutz-Level an Sicherheit bieten.

### 2. VMP (Virtual Machine Protection)

VMP ist ein professionelles Codeschutzwerkzeug, das hohen Schutz bietet, indem Code in virtuelle Maschinenanweisungen kompiliert wird. Im Gegensatz zur einfachen Namensverschleierung transformiert VMP die Codelogik tatsächlich in eine Form, die herkömmliche Dekompiler nicht rekonstruieren können.

**Schutzklassifizierung**:

| Stufe | Virtualisierung | Mutation | Anti-Debugging | Zeichenkettenverschlüsselung | Anwendungsfälle |
|------|--------|------|--------|------------|----------|
| HOCH | voll | hoch | aktiviert | aktiviert | Lizenzvalidierung, Sitzungsnebenläufigkeit, sensible Konstanten |
| MITTEL | teilweise | mittel | aktiviert | aktiviert | Geschäftslogik, Domänenmodelle |
| NIEDRIG | keine | niedrig | deaktiviert | deaktiviert | Hilfsklassen, nicht-kritischer Code |

Das HagiCode-Projekt definiert ein deklaratives Attributsystem zur Markierung von Code, der geschützt werden muss:

```csharp
// Hochprioritätsschutz
[VmProtect(VmProtectionPriority.High, Reason = "Enthält Lizenzüberprüfungslogik")]
public class KeygenClient { ... }

// Vom Schutz ausschließen
[VmExclude(Reason = "Öffentliche API, die unverändert bleiben muss")]
public class PublicApi { ... }

// Vererbter Schutz
[VmProtect(Priority.High, ProtectDerived = true)]
public class BaseLicenseValidator { ... }
```

### 3. Build-Zeit-Schutzstrategie

VMP-Schutz ist nicht nur zur Laufzeit wichtig. Er muss auch als Teil der Build-Pipeline automatisiert werden, da manuelle Ausführung viel zu mühsam wäre. HagiCodes Build-System unterstützt mehrere Modi:

- **Nativer Windows-Modus**: Direkten Aufruf des VMProtect-Werkzeugs
- **Linux Docker-Container-Modus**: Ausführung von VMP in einem Container zur Lösung plattformübergreifender Kompatibilitätsprobleme
- **Attribut-Scanning**: Automatische Entdeckung von Schutzmarkierungen im Code
- **Validierungsmechanismus**: Bestätigung, dass der Schutz erfolgreich angewendet wurde

Zusammen machen diese Fähigkeiten den Prozess viel einfacher zu verwalten.

## Lösung

### 1. Microsofts integriertes Verschleierungsattribut verwenden

Wenden Sie `ObfuscationAttribute` direkt im Code an:

```csharp
using System.Reflection;

[Obfuscation(Feature = "ultra", Exclude = false)]
public class LicenseService
{
    [Obfuscation(Feature = "ultra", Exclude = false)]
    public async Task<bool> ValidateLicenseAsync(string key)
    {
        // Lizenzvalidierungslogik
    }

    [Obfuscation(Feature = "flow", Exclude = false)]
    private string DecryptToken(string encrypted)
    {
        // Entschlüsselungslogik
    }
}
```

Manchmal muss man Test-Assemblys den Zugriff auf interne Mitglieder erlauben, während der Produktionscode geschützt bleibt:

```csharp
// AssemblyInfo.cs
[assembly: InternalsVisibleTo("HagiCode.Application.Tests")]
[assembly: InternalsVisibleTo("DynamicProxyGenAssembly2")] // für Moq
```

Das macht das Testen viel bequemer, denn der Code muss immer noch ordnungsgemäß getestet werden.

### 2. Benutzerdefinierte Attributdefinitionen für VMP-Schutz

Erstellen Sie benutzerdefinierte Schutzattribute zur Steuerung des VMP-Verhaltens:

```csharp
using System;

[AttributeUsage(AttributeTargets.Class | AttributeTargets.Method | AttributeTargets.Property)]
public class VmProtectAttribute : Attribute
{
    public VmProtectionPriority Priority { get; set; }
    public string? Reason { get; set; }
    public bool ProtectDerived { get; set; }
}

[AttributeUsage(AttributeTargets.Class | AttributeTargets.Method | AttributeTargets.Property)]
public class VmExcludeAttribute : Attribute
{
    public string? Reason { get; set; }
}

public enum VmProtectionPriority
{
    None = 0,
    Low = 1,
    Medium = 2,
    High = 3
}
```

Benutzerdefinierte Attribute sind oft einfacher zu handhaben, da sie die eigenen Schutzanforderungen direkt widerspiegeln.

### 3. VMP-Konfigurationsdatei

```yaml
# vmp_config.yml
protection:
  priority_mode: "attribute"  # Attributbasierte Priorität
  default_level: "medium"

tools:
  - name: "vmprotect"
    path: "C:\\Program Files\\VMProtect Ultimate\\VMProtect.exe"

protection_levels:
  high:
    virtualization: "full"
    mutation: "high"
    anti_debug: true
    anti_dump: true
    encrypt_strings: true
    encrypt_resources: true

  medium:
    virtualization: "partial"
    mutation: "medium"
    anti_debug: true
    encrypt_strings: true

  low:
    virtualization: "none"
    mutation: "low"
    anti_debug: false
```

Eine klarere Konfiguration macht das System später viel leichter wartbar.

## Praktischer Leitfaden

### 1. Schutzpraktiken für kritische Komponenten

Gemäß HagiCodes Code-Schutzspezifikation müssen die folgenden Komponenten HOCH-Prioritätsschutz verwenden:

```csharp
// Produktionskonstanten - müssen verschlüsselt und durch VMP geschützt werden
[VmProtect(VmProtectionPriority.High, Reason = "Produktionskonstanten")]
public static class ProductionConstants
{
    // Verschlüsselter Zeichenketten-Zugriff, geschützt durch VMP
    [VmProtect(VmProtectionPriority.High)]
    public static string GetLicenseServerUrl(IOptions<LicenseOptions> options) => ...;
}

// Lizenzvalidierungslogik
[VmProtect(VmProtectionPriority.High, Reason = "Lizenzüberprüfungslogik")]
public class KeygenClient : IKeygenClient
{
    [Obfuscation(Feature = "ultra", Exclude = false)]
    public async Task<LicenseValidationResult?> ValidateLicenseAsync(...) { ... }
}

// Maschinen-Fingerabdruck-Dienst
[VmProtect(VmProtectionPriority.High)]
public class MachineFingerprintService : IMachineFingerprintService { ... }
```

Kritischer Code verdient stärkeren Schutz, denn die Offenlegung der Kernlogik würde echte Probleme verursachen.

### 2. Zeichenkettenverschlüsselung und Laufzeitentschlüsselung

Verschlüsseln Sie Zeichenketten zur Build-Zeit und entschlüsseln Sie sie zur Laufzeit:

```csharp
public static class StringDecryption
{
    [VmProtect(VmProtectionPriority.High, Reason = "KRITISCHE SICHERHEIT")]
    public static string DecryptString(byte[] encryptedData, byte[] key, byte[] iv)
    {
        using var aes = Aes.Create();
        aes.Key = key;
        aes.IV = iv;

        using var decryptor = aes.CreateDecryptor();
        using var ms = new MemoryStream(encryptedData);
        using var cs = new CryptoStream(ms, decryptor, CryptoStreamMode.Read);
        using var reader = new StreamReader(cs);

        return reader.ReadToEnd();
    }
}

// Produktionskonstanten-Zugriff (Lazy Loading + Caching)
public static class ProductionConstants
{
    private static string? _cachedLicenseServerUrl;

    public static string GetLicenseServerUrl(IOptions<LicenseOptions> options)
    {
        if (_cachedLicenseServerUrl == null)
        {
            var encrypted = GetEncryptedLicenseServerUrl();
#if DEBUG
            _cachedLicenseServerUrl = options.Value.PrimaryServer.Url;
#else
            _cachedLicenseServerUrl = StringDecryption.DecryptString(
                encrypted,
                GetEncryptionKey(),
                GetEncryptionIV());
#endif
        }
        return _cachedLicenseServerUrl;
    }
}
```

Dieser Schritt ist wichtig, weil sensible Informationen niemals im Klartext belassen werden sollten.

### 3. VMP-Schutzüberprüfung

Nach dem Build müssen Sie überprüfen, ob der Schutz erfolgreich angewendet wurde; andernfalls können Sie nicht sicher sein, dass er tatsächlich funktioniert:

```csharp
// Beispiel-Überprüfungsskript
public bool VerifyProtection(string assemblyPath)
{
    // 1. VMP-Signatur prüfen
    var bytes = File.ReadAllBytes(assemblyPath);
    var vmpSignature = Encoding.ASCII.GetBytes("VMProtect");
    if (bytes.Any(b => vmpSignature.Contains(b)))
    {
        return true;
    }

    // 2. Dateigrößenänderung prüfen (die geschützte Datei ist normalerweise größer)
    var originalInfo = new FileInfo(assemblyPath.Replace(".dll", ".bak"));
    if (originalInfo.Exists)
    {
        var sizeRatio = (double)new FileInfo(assemblyPath).Length / originalInfo.Length;
        return sizeRatio > 1.1;
    }

    return false;
}
```

Überprüfung ist immer wert, denn sonst können Probleme unbemerkt durchrutschen.

### 4. Hinweise und Warnungen

Es gibt hier mehrere Fallen, die besondere Beachtung verdienen:

1. **Nicht den gesamten Code verschleiern**: Öffentliche APIs, Schnittstellendefinitionen und DTO-Klassen benötigen normalerweise keinen Schutz. Übermäßige Verschleierung kann die Leistung und das Debugging beeinträchtigen. Das HagiCode-Projekt hat dies auf die harte Tour gelernt.

2. **Schlüssige Zugriffsmethoden schützen**: Methoden, die Verschlüsselungsschlüssel abrufen, müssen die gleiche oder eine höhere Schutzstufe erhalten als die verschlüsselten Daten selbst; andernfalls verliert die gesamte Einrichtung ihren Wert.

3. **Testen und Produktion ausbalancieren**: DEBUG-Builds sollten die Verschlüsselung überspringen, um die Entwicklung und das Debugging zu erleichtern, während RELEASE-Builds den vollen Schutz aktivieren sollten. Denken Sie daran, sie mit bedingter Kompilierung wie `#if DEBUG` zu trennen.

4. **Docker-Umgebung berücksichtigen**: Die Ausführung von VMP unter Linux erfordert einen containerisierten Ansatz, um die Werkzeugkompatibilität sicherzustellen. HagiCode verwendet eine Wine + VMP Container-Lösung, um das plattformübergreifende Problem zu lösen.

5. **Überprüfung ist obligatorisch**: Nach Abschluss des Builds müssen Sie überprüfen, dass der Schutz erfolgreich angewendet wurde. Andernfalls kann sensibler Code noch offengelegt sein, und der oben gezeigte Überprüfungscode existiert genau für diesen Zweck.

## Zusammenfassung

Mit dieser mehrschichtigen Schutzstrategie hat HagiCode ein umfassendes Codesicherheitssystem aufgebaut, das von der Basisverschleierung bis zum VM-Schutz reicht:

- **Schicht 1**: Verwendung von `ObfuscationAttribute` für Basis-Markierung und Bereitstellung von Hinweisen für Drittanbieter-Werkzeuge
- **Schicht 2**: Verwendung von benutzerdefinierten `VmProtectAttribute`-Deklarationen zum Ausdrücken von Schutzabsicht und Priorität
- **Schicht 3**: Verwendung von VMP-VM-Schutz, um kritischen Code in irreversible VM-Anweisungen zu transformieren
- **Schicht 4**: Automatisches Scannen und Anwenden des Schutzes während des Builds, gefolgt von einer Ergebnisüberprüfung

Dieser Ansatz kann herkömmliche Dekompilierungswerkzeuge abwehren und hält auch fortgeschrittenen Reverse-Engineering-Angriffen besser stand. Wenn Sie eine .NET-Anwendung erstellen, die Codeschutz benötigt, hoffe ich, dass dies zumindest einen nützlichen Bezugspunkt bietet.
