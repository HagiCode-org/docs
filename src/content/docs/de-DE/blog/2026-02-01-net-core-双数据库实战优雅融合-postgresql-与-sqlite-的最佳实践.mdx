## .NET Core Dual-Datenbank in der Praxis: PostgreSQL und SQLite friedlich koexistieren lassen

> Bei der Entwicklung moderner Anwendungen stehen wir oft vor der Entscheidung: Die Entwicklungsumgebung soll leichtgewichtig sein, während die Produktionsumgebung hohe Parallelität und Verfügbarkeit benötigt. Dieser Artikel zeigt, wie man in .NET Core elegant PostgreSQL und SQLite gleichzeitig unterstützt, um die Best Practice „SQLite für die Entwicklung, PostgreSQL für die Produktion" zu verwirklichen.


## Hintergrund

In der Softwareentwicklung ist die Umweltunterscheidung schon immer eine der Hürden für Entwicklungsteams gewesen. Bei dem von uns entwickelten **HagiCode**-Projekt handelt es sich um ein auf ASP.NET Core 10 und React basierendes KI-gestütztes Entwicklungssystem mit integriertem Orleans für verteiltes Zustandsmanagement – ein recht moderner und komplexer Technologie-Stack.

In der frühen Projektphase stießen wir auf einen typischen Engineering-Schmerzpunkt: Entwickler wünschten sich eine lokale Umgebung, die „out of the box" funktioniert, ohne eine schwere PostgreSQL-Datenbank installieren und konfigurieren zu müssen. In der Produktionsumgebung mussten wir jedoch hohe Schreibparallelität und komplexe JSON-Abfragen verarbeiten, bei denen das leichtgewichtige SQLite nicht mehr mithielt.

Wie kann man bei einheitlicher Codebasis die Anwendung sowohl von der Portabilität von SQLite als Client-Software profitieren lassen als auch die Leistungsfähigkeit von PostgreSQL als Enterprise-Service nutzen? Das ist die Kernfrage dieses Artikels.

## Kerninhalt 1: Architekturentwurf und einheitliche Abstraktion

Um in .NET Core eine Dual-Datenbank-Unterstützung zu implementieren, lautet der Kerngedanke: „Abhängig von Abstraktionen, nicht von konkreten Implementierungen". Wir müssen die Datenbankauswahl aus der Geschäftslogik herauslösen und der Konfigurationsebene überlassen.

### Entwurfsansatz

1.  **Einheitliche Schnittstelle**: Die gesamte Geschäftslogik sollte von der `DbContext`-Basisklasse oder einer benutzerdefinierten Schnittstelle abhängen, nicht von einem konkreten `PostgreSqlDbContext`.
2.  **Konfigurationsgesteuert**: Über Konfigurationseinträge in `appsettings.json` wird beim Anwendungsstart dynamisch entschieden, welcher Datenbankanbieter geladen wird.
3.  **Feature-Isolierung**: PostgreSQL-spezifische Funktionen (wie JSONB) werden adaptiv behandelt, um auch in SQLite eine Abwärtskompatibilität zu gewährleisten.

### Code-Implementierung: Dynamische Kontextkonfiguration

Im `Program.cs` von ASP.NET Core sollten wir `UseNpgsql` oder `UseSqlite` nicht hartcodieren. Stattdessen sollten wir die Konfiguration lesen und dynamisch entscheiden.

Zuerst die Konfigurationsklasse definieren:

```csharp
public class DatabaseSettings
{
    public const string SectionName = "Database";

    // 数据库类型：PostgreSQL 或 SQLite
    public string DbType { get; set; } = "PostgreSQL";

    // 连接字符串
    public string ConnectionString { get; set; } = string.Empty;
}
```

Dann im `Program.cs` den Dienst basierend auf der Konfiguration registrieren:

```csharp
// 读取配置
var databaseSettings = builder.Configuration.GetSection(DatabaseSettings.SectionName).Get<DatabaseSettings>();

// 注册 DbContext
builder.Services.AddDbContext<ApplicationDbContext>(options =>
{
    if (databaseSettings?.DbType?.ToLower() == "sqlite")
    {
        // SQLite 配置
        options.UseSqlite(databaseSettings.ConnectionString);

        // SQLite 的并发写入限制处理
        // 注意：在生产环境中建议开启 WAL 模式以提高并发性能
    }
    else
    {
        // PostgreSQL 配置（默认）
        options.UseNpgsql(databaseSettings.ConnectionString, npgsqlOptions =>
        {
            // 开启 JSONB 支持，这在处理 AI 对话记录时非常有用
            npgsqlOptions.UseJsonNet();
        });

        // 配置连接池重连策略
        options.EnableRetryOnFailure(3);
    }
});
```

## Kerninhalt 2: Umgang mit Unterschieden und Migrationsstrategie

Obwohl PostgreSQL und SQLite beide den SQL-Standard unterstützen, gibt es signifikante Unterschiede in spezifischen Features und Verhalten. Wenn diese Unterschiede nicht richtig behandelt werden, droht die unangenehme Situation „funktioniert lokal, schlägt in Produktion fehl".

### 1. Behandlung von JSON-Typen

In HagiCode müssen wir大量 von Prompts und KI-Metadaten speichern, was typischerweise JSON-Spalten betrifft.
- **PostgreSQL**: Verfügt über den nativen `JSONB`-Typ mit hervorragender Abfrageleistung.
- **SQLite**: Hat keinen nativen JSON-Typ (neuere Versionen haben die JSON1-Erweiterung, aber beim Objekt-Mapping gibt es noch Unterschiede), wird normalerweise als TEXT gespeichert.

**Lösung**:
In der EF Core-Entitätszuweisung konfigurieren wir es als konvertierbaren Typ.

```csharp
protected override void OnModelCreating(ModelBuilder modelBuilder)
{
    base.OnModelCreating(modelBuilder);

    // 配置实体
    modelBuilder.Entity<PromptTemplate>(entity =>
    {
        entity.Property(e => e.Metadata)
              .HasColumnType("jsonb") // PG 使用 jsonb
              .HasConversion(
                  v => JsonSerializer.Serialize(v, (JsonSerializerOptions)null),
                  v => JsonSerializer.Deserialize<Dictionary<string, object>>(v, (JsonSerializerOptions)null)
              );
    });
}
```

Bei Verwendung von SQLite wird `HasColumnType("jsonb")` zwar ignoriert oder erzeugt eine Warnung, aber da `HasConversion` konfiguriert ist, werden Daten korrekt serialisiert und deserialisiert als String in ein TEXT-Feld gespeichert, was die Kompatibilität gewährleistet.

### 2. Trennung der Migrationsstrategie

Versuchen Sie niemals, denselben Migrationsskript-Satz sowohl für PG als auch für SQLite zu verwenden. Aufgrund unterschiedlicher Primärschlüssel-Generierungsstrategien, Indexsyntax usw. wird dies unweigerlich scheitern.

**Empfohlene Vorgehensweise**:
Pflegen Sie zwei Migrationszweige oder -projekte. In der HagiCode-Entwicklung gehen wir so vor:

1.  **Entwicklungsphase**: Hauptsächliche Arbeit unter SQLite. Verwenden Sie `Add-Migration Init_Sqlite -OutputDir Migrations/Sqlite`.
2.  **Anpassungsphase**: Nach Abschluss einer Funktion wechseln Sie die Verbindungszeichenfolge zu PostgreSQL und führen Sie `Add-Migration Init_Postgres -OutputDir Migrations/Postgres` aus.
3.  **Automatisierungsskript**: Schreiben Sie ein einfaches PowerShell- oder Bash-Skript, das je nach aktueller Umgebungsvariable automatisch die entsprechende Migration anwendet.

```bash
# 简单的部署逻辑伪代码
if [ "$DATABASE_PROVIDER" = "PostgreSQL" ]; then
    dotnet ef database update --project Migrations.Postgres
else
    dotnet ef database update --project Migrations.Sqlite
fi
```

## Kerninhalt 3: Praktische Erfahrungen aus HagiCode

Bei der Umstrukturierung von **HagiCode** von einer einzelnen Datenbank zu Dual-Datenbank-Unterstützung haben wir einige Stolpersteine erlebt und wichtige Erfahrungen gesammelt, die wir teilen möchten.

### 1. Unterschiede bei Parallelität und Transaktionen

PostgreSQL ist eine Client-Server-Architektur, die hohe Schreibparallelität unterstützt, mit sehr starken Transaktionsisolationsstufen. SQLite verwendet eine Dateisperre, bei der Schreibvorgänge die gesamte Datenbankdatei sperren (außer im WAL-Modus).

**Empfehlung**:
Beim Schreiben von Geschäftslogik mit häufigen Schreibvorgängen (z. B. Echtzeitspeicherung des Bearbeitungszustands des Benutzers) muss unbedingt die Sperrmechanik von SQLite berücksichtigt werden. Bei der Entwicklung des OpenSpec-Kollaborationsmoduls von **HagiCode** haben wir einen „Merge-before-Write"-Mechanismus eingeführt, um die direkte Schreibfrequenz in die Datenbank zu reduzieren und so hohe Leistung bei beiden Datenbanken zu gewährleisten.

### 2. Lebenszyklusmanagement von Verbindungszeichenfolgen

Der Verbindungsaufbau bei PostgreSQL ist teuer und nutzt Verbindungspools. SQLite-Verbindungen sind sehr leichtgewichtig, aber wenn sie nicht rechtzeitig freigegeben werden, können Dateisperren zu Timeouts bei nachfolgenden Vorgängen führen.

Im `Program.cs` können wir für verschiedene Datenbanken feingranulare Anpassungen vornehmen:

```csharp
if (databaseSettings?.DbType?.ToLower() == "sqlite")
{
    // SQLite：保持连接开启能提升性能，但要注意文件锁
    options.UseSqlite(connectionString, sqliteOptions =>
    {
        // 设置命令超时时间
        sqliteOptions.CommandTimeout(30);
    });
}
else
{
    // PG：利用连接池
    options.UseNpgsql(connectionString, npgsqlOptions =>
    {
        npgsqlOptions.MaxBatchSize(100);
        npgsqlOptions.CommandTimeout(30);
    });
}
```

### 3. Bedeutung von Testabdeckung

Viele Entwickler (einschließlich unserer Teammitglieder in der Frühphase) begehen einen Fehler: Sie führen Unit-Tests nur in der Entwicklungsumgebung (meistens SQLite) aus.

In der CI/CD-Pipeline von HagiCode haben wir einen GitHub Action-Schritt erzwungen, der sicherstellt, dass bei jedem Pull Request auch PostgreSQL-Integrationstests ausgeführt werden.

```yaml
# .github/workflows/test.yml 示例片段
- name: Run Integration Tests (PostgreSQL)
  run: |
    docker-compose up -d db_postgres
    dotnet test --filter "Category=Integration"
```

Das hat uns unzählige Male Bugs bezüglich SQL-Syntaxunterschieden und Groß-/Kleinschreibung abgefangen.

## Zusammenfassung

Durch die Einführung einer Abstraktionsschicht und konfigurationsgesteuerte Dependency Injection haben wir im **HagiCode**-Projekt erfolgreich einen „Zweigleisigen" Betrieb von PostgreSQL und SQLite realisiert. Dies senkt die Einstiegshürde für neue Entwickler erheblich (keine PG-Installation nötig) und bietet gleichzeitig eine solide Leistungsgarantie für die Produktionsumgebung.

Rückblick auf die wichtigsten Punkte:
1.  **Abstraktion zuerst**: Geschäftscode hängt nicht von konkreter Datenbankimplementierung ab.
2.  **Konfigurationstrennung**: Entwicklung und Produktion verwenden unterschiedliche `appsettings.json`.
3.  **Migrationstrennung**: Versuchen Sie nicht, eine Migration für alles zu verwenden.
4.  **Feature-Downgrade**: In SQLite Kompatibilität priorisieren, in PostgreSQL Leistung priorisieren.

Dieses Architekturmuster ist nicht nur für HagiCode geeignet, sondern für jedes .NET-Projekt, das eine Balance zwischen leichtgewichtiger Entwicklung und schwerer Produktionsumgebung finden muss.
