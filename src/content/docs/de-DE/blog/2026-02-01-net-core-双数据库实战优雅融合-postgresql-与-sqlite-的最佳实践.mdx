## .NET Core Dual-Datenbank in der Praxis: PostgreSQL und SQLite friedlich koexistieren lassen

> Bei der Entwicklung moderner Anwendungen stehen wir oft vor folgendem Dilemma: Entwicklungsumgebungen bevorzugen Leichtgewichtigkeit und Bequemlichkeit, während Produktionsumgebungen hohe Nebenläufigkeit und hohe Verfügbarkeit erfordern. Dieser Artikel zeigt, wie man PostgreSQL und SQLite in einem .NET Core-Projekt elegant unterstützt und die Best Practice „SQLite für die Entwicklung, PostgreSQL für die Produktion" umsetzt.


## Hintergrund

In der Softwareentwicklung sind Umweltunterschiede schon immer eines der schwersten Probleme für Engineering-Teams. Nehmen wir die **HagiCode**-Plattform, die wir gerade entwickeln: Es ist ein KI-gestütztes Entwicklungssystem auf Basis von ASP.NET Core 10 und React, mit intern integriertem Orleans für verteilte Zustandsverwaltung. Der Tech-Stack ist modern und ziemlich anspruchsvoll.

Früh im Projekt stießen wir auf einen klassischen Engineering-Schmerzpunkt: Entwickler wollten, dass die lokale Umgebung „out of the box" funktioniert, ohne eine schwere PostgreSQL-Datenbank installieren und konfigurieren zu müssen. In der Produktion mussten wir jedoch nebenläufige Schreibvorgänge und komplexe JSON-Abfragen bewältigen — genau hier zeigt sich die Grenze von leichtgewichtigem SQLite.

Wie kann man eine einheitliche Codebasis beibehalten und gleichzeitig ermöglichen, dass die Anwendung sowohl von der Portabilität von SQLite wie eine Desktop-Anwendung profitiert als auch von der leistungsstarken Performance von PostgreSQL wie ein Enterprise-Service? Das ist die Kernfrage dieses Artikels.

## Kernthema 1: Architekturentwurf und einheitliche Abstraktion

Um zwei Datenbanken in .NET Core zu unterstützen, ist der Kerngedanke: „Von Abstraktionen abhängen, nicht von konkreten Implementierungen." Wir müssen die Datenbankauswahl aus dem Geschäftscode herauslösen und die Konfigurationsebene entscheiden lassen.

### Entwurfsansatz

1.  **Einheitliche Schnittstelle**: Die gesamte Geschäftslogik sollte von der `DbContext`-Basisklasse oder benutzerdefinierten Schnittstellen abhängen, nicht von einem spezifischen `PostgreSqlDbContext`.
2.  **Konfigurationsgesteuert**: Konfigurationseinträge in `appsettings.json` nutzen, um zur Laufzeit dynamisch zu entscheiden, welcher Datenbankanbieter geladen wird.
3.  **Funktionsisolation**: Anpassungslogik für PostgreSQL-spezifische Funktionen wie JSONB hinzufügen, damit die Anwendung auf SQLite graceful degradieren kann.

### Code-Implementierung: Dynamische Kontextkonfiguration

In ASP.NET Core `Program.cs` sollten wir `UseNpgsql` oder `UseSqlite` nicht hart codieren. Stattdessen lesen wir die Konfiguration und entscheiden dynamisch.

Zuerst die Konfigurationsklasse definieren:

```csharp
public class DatabaseSettings
{
    public const string SectionName = "Database";

    // Datenbanktyp: PostgreSQL oder SQLite
    public string DbType { get; set; } = "PostgreSQL";

    // Verbindungszeichenfolge
    public string ConnectionString { get; set; } = string.Empty;
}
```

Dann den Dienst in `Program.cs` basierend auf der Konfiguration registrieren:

```csharp
// Konfiguration lesen
var databaseSettings = builder.Configuration.GetSection(DatabaseSettings.SectionName).Get<DatabaseSettings>();

// DbContext registrieren
builder.Services.AddDbContext<ApplicationDbContext>(options =>
{
    if (databaseSettings?.DbType?.ToLower() == "sqlite")
    {
        // SQLite-Konfiguration
        options.UseSqlite(databaseSettings.ConnectionString);

        // Handhabung der gleichzeitigen Schreib-Limitierungen von SQLite
        // Hinweis: In der Produktion wird der WAL-Modus empfohlen, um die Nebenläufigkeitsleistung zu verbessern
    }
    else
    {
        // PostgreSQL-Konfiguration (Standard)
        options.UseNpgsql(databaseSettings.ConnectionString, npgsqlOptions =>
        {
            // JSONB-Unterstützung aktivieren, sehr nützlich bei der Verarbeitung von KI-Konversationsdaten
            npgsqlOptions.UseJsonNet();
        });

        // Verbindungspool-Wiederholungsrichtlinie konfigurieren
        options.EnableRetryOnFailure(3);
    }
});
```

## Kernthema 2: Umgang mit Unterschieden und Migrationsstrategie

Obwohl PostgreSQL und SQLite beide den SQL-Standard unterstützen, unterscheiden sie sich in spezifischen Funktionen und Verhalten erheblich. Wenn diese Unterschiede nicht sorgfältig behandelt werden, landet man leicht in der peinlichen Situation, dass alles lokal funktioniert, aber nach der Bereitstellung fehlschlägt.

### 1. Umgang mit JSON-Typen

In HagiCode müssen wir eine große Menge an Prompts und KI-Metadaten speichern, was normalerweise JSON-Spalten betrifft.
- **PostgreSQL**: Hat einen nativen `JSONB`-Typ mit hervorragender Abfrageleistung.
- **SQLite**: Hat keinen nativen JSON-Typ ( neuere Versionen enthalten die JSON1-Erweiterung, aber das Object-Mapping unterscheidet sich dennoch), sodass Daten normalerweise als TEXT gespeichert werden.

**Lösung**:
In der EF Core-Entity-Zuordnung konfigurieren wir es als konvertierbaren Typ.

```csharp
protected override void OnModelCreating(ModelBuilder modelBuilder)
{
    base.OnModelCreating(modelBuilder);

    // Entity konfigurieren
    modelBuilder.Entity<PromptTemplate>(entity =>
    {
        entity.Property(e => e.Metadata)
              .HasColumnType("jsonb") // PG verwendet jsonb
              .HasConversion(
                  v => JsonSerializer.Serialize(v, (JsonSerializerOptions)null),
                  v => JsonSerializer.Deserialize<Dictionary<string, object>>(v, (JsonSerializerOptions)null)
              );
    });
}
```

Wenn SQLite verwendet wird, wird `HasColumnType("jsonb")` möglicherweise ignoriert oder löst eine Warnung aus. Da jedoch `HasConversion` konfiguriert ist, werden die Daten weiterhin korrekt serialisiert und deserialisiert als in einem TEXT-Feld gespeicherte Strings, was die Kompatibilität sicherstellt.

### 2. Migrationen trennen

Versuchen Sie niemals, einen Satz Migrationsskripte gleichzeitig für PostgreSQL und SQLite zu nutzen. Unterschiede in der Primärschlüssel-Generierung, Index-Syntax und anderen Datenbankdetails werden unweigerlich zu Fehlern führen.

**Empfohlene Vorgehensweise**:
Zwei Migrationszweige oder -projekte pflegen. Im HagiCode-Entwicklungsworkflow gehen wir so vor:

1.  **Entwicklungsphase**: Hauptsächlich mit SQLite arbeiten. `Add-Migration Init_Sqlite -OutputDir Migrations/Sqlite` verwenden.
2.  **Anpassungsphase**: Nach der Entwicklung eines Features die Verbindungszeichenfolge auf PostgreSQL umstellen und `Add-Migration Init_Postgres -OutputDir Migrations/Postgres` ausführen.
3.  **Automatisierungsskripte**: Ein einfaches PowerShell- oder Bash-Skript schreiben, das automatisch die richtige Migration basierend auf der aktuellen Umgebungsvariable anwendet.

```bash
# Pseudocode für einfache Bereitstellungslogik
if [ "$DATABASE_PROVIDER" = "PostgreSQL" ]; then
    dotnet ef database update --project Migrations.Postgres
else
    dotnet ef database update --project Migrations.Sqlite
fi
```

## Kernthema 3: Erkenntnisse aus dem HagiCode-Produktivbetrieb

Bei der Umstrukturierung von **HagiCode** von einem Einzel- zu einem Dual-Datenbankmodell stießen wir auf einige Fallstricke und sammelten wichtige Erkenntnisse, die Ihnen helfen können, dieselben Fehler zu vermeiden.

### 1. Unterschiede bei Nebenläufigkeit und Transaktionen

PostgreSQL verwendet eine Server-Client-Architektur und unterstützt nebenläufige Schreibvorgänge mit leistungsstarken Transaktionsisolationsebenen. SQLite verwendet Dateisperren, sodass Schreibvorgänge die gesamte Datenbankdatei sperren, es sei denn, der WAL-Modus ist aktiviert.

**Empfehlung**:
Beim Schreiben von Geschäftslogik, die häufige Schreibvorgänge umfasst, wie das Echtzeit-Speichern des Bearbeitungszustands eines Benutzers, müssen Sie das Sperrmodell von SQLite berücksichtigen. Bei der Entwicklung des OpenSpec-Kollaborationsmoduls in **HagiCode** führten wir einen „Merge vor dem Schreiben"-Mechanismus ein, um die Häufigkeit direkter Datenbankschreibvorgänge zu reduzieren und gute Leistung auf beiden Datenbanken zu erhalten.

### 2. Lebenszyklusverwaltung von Verbindungszeichenfolgen

Das Herstellen einer PostgreSQL-Verbindung ist relativ teuer und hängt vom Verbindungspooling ab. SQLite-Verbindungen sind sehr leichtgewichtig, aber wenn sie nicht rechtzeitig freigegeben werden, können Dateisperren nachfolgende Operationen mit Timeouts belegen.

In `Program.cs` können wir das Verhalten für jede Datenbank feinabstimmen:

```csharp
if (databaseSettings?.DbType?.ToLower() == "sqlite")
{
    // SQLite: Offene Verbindungen können die Leistung verbessern, aber auf Dateisperren achten
    options.UseSqlite(connectionString, sqliteOptions =>
    {
        // Kommando-Timeout festlegen
        sqliteOptions.CommandTimeout(30);
    });
}
else
{
    // PG: Verbindungspooling voll ausnutzen
    options.UseNpgsql(connectionString, npgsqlOptions =>
    {
        npgsqlOptions.MaxBatchSize(100);
        npgsqlOptions.CommandTimeout(30);
    });
}
```

### 3. Die Bedeutung von Testabdeckung

Viele Entwickler — einschließlich einiger früherer Teammitglieder — neigen zu einem häufigen Fehler: Unit-Tests nur in der Entwicklungsumgebung auszuführen, die normalerweise SQLite verwendet.

In der CI/CD-Pipeline von HagiCode haben wir einen GitHub Actions-Schritt erzwungen, der sicherstellt, dass jeder Pull Request PostgreSQL-Integrationstests durchläuft.

```yaml
# Beispielausschnitt aus .github/workflows/test.yml
- name: Run Integration Tests (PostgreSQL)
  run: |
    docker-compose up -d db_postgres
    dotnet test --filter "Category=Integration"
```

Das hat uns geholfen, unzählige Bugs im Zusammenhang mit SQL-Syntaxunterschieden und Groß-/Kleinschreibung zu finden.

## Zusammenfassung

Durch die Einführung einer Abstraktionsschicht und konfigurationsgesteuerte Dependency Injection haben wir in **HagiCode** erfolgreich ein Dual-Track-Setup mit PostgreSQL und SQLite implementiert. Das hat nicht nur die Einstiegshürde für neue Entwickler erheblich gesenkt, da keine PostgreSQL-Installation mehr erforderlich ist, sondern auch starke Leistungsgarantien für die Produktion geboten.

Die wichtigsten Punkte im Überblick:
1.  **Zuerst abstrahieren**: Geschäftscode sollte nicht von konkreten Datenbankimplementierungen abhängen.
2.  **Konfiguration trennen**: Unterschiedliche `appsettings.json`-Dateien für Entwicklung und Produktion verwenden.
3.  **Migrationen trennen**: Nicht versuchen, einen Migrationssatz überall zum Laufen zu bringen.
4.  **Funktionsdegradation**: Kompatibilität in SQLite priorisieren und Leistung in PostgreSQL.

Dieses Architekturmuster eignet sich nicht nur für HagiCode, sondern für jedes .NET-Projekt, das einen Ausgleich zwischen leichtgewichtiger Entwicklung und leistungsstarker Produktion finden muss.
