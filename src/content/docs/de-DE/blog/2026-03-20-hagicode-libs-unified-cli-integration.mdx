## Hagicode.Libs: Einheitliche Integration mehrerer KI-Programmier-Assistenten-CLIs – Ein Praxisbericht

> Bei der Entwicklung des HagiCode-Projekts mussten wir gleichzeitig die CLIs mehrerer KI-Programmier-Assistenten wie Claude Code, Codex, CodeBuddy usw. integrieren. Jede CLI hat unterschiedliche Schnittstellen, Parameter und Ausgabeformate, und der doppelte Integrationscode macht das Projekt immer schwieriger zu warten. Dieser Artikel teilt, wie wir durch die HagiCode.Libs-Bibliothek eine einheitliche Abstraktionsschicht erstellt haben, um dieses technische Problem zu lösen – man könnte sagen, es ist ein bisschen Erfahrung, die wir aus unseren Fehlern gesammelt haben.

## Hintergrund

Der Markt für KI-Programmier-Assistenten ist mittlerweile recht lebhaft, neben Claude Code gibt es OpenAIs Codex, Zhipus CodeBuddy und viele mehr. Als ein KI-Code-Assistent-Projekt muss HagiCode diese verschiedenen CLI-Tools in mehrere Unterprojekte (Desktop, Backend, Web usw.) integrieren.

Anfangs war das Problem noch nicht groß, die Integration einer CLI war eine Sache von ein paar hundert Zeilen Code. Aber mit zunehmender Anzahl der zu unterstützenden CLIs begannen die Probleme:

Jede CLI hat ein anderes Befehlszeilenparameterformat, unterschiedliche Anforderungen an Umgebungsvariablen und die Ausgabeformate sind noch vielfältiger – manche geben JSON aus, manche Streaming-JSON, manche einfach nur Klartext. Hinzu kommen plattformübergreifende Kompatibilitätsprobleme: Die Erkennungs- und Prozessverwaltungsmechanismen für ausführbare Dateien unter Windows und Unix-Systemen sind völlig unterschiedlich, und der Code-Duplikationsgrad wird immer höher. Eigentlich ist das nur ein bisschen mehr Strg+C, Strg+V, aber die Wartung macht Kopfschmerzen.

Am problematischsten ist, dass jedes Mal, wenn eine neue CLI-Funktion unterstützt werden soll, derselbe Code in mehreren Projekten geändert werden muss. Diese Methode ist offensichtlich keine langfristige Lösung – Code hat auch seinen Charakter, zu viel Wiederholung und er wird stur.

## Über HagiCode

Die in diesem Artikel vorgestellte Lösung stammt aus unserer praktischen Erfahrung im [HagiCode](https://hagicode.com)-Projekt. HagiCode ist ein Open-Source-KI-Code-Assistent-Projekt, das mehrere Unterprojekte gleichzeitig warten muss: das VSCode-Erweiterungs-Frontend, den KI-Dienst-Backend, den plattformübergreifenden Desktop-Client und mehr. Man könnte sagen, genau dieses komplexe Szenario mit mehreren Sprachen und Plattformen hat zur Geburt von HagiCode.Libs geführt – sozusagen aus der Not geboren, nun ja.

## Analyse: Gemeinsamkeiten finden

Obwohl diese KI-Programmier-Assistenten-CLIs jeweils ihre Besonderheiten haben, gibt es auf technischer Ebene offensichtliche gemeinsame Merkmale:

**Ähnliche Interaktionsmuster**: Alle starten einen CLI-Prozess, senden Prompts, empfangen Streaming-Antworten, parsen Nachrichten und beenden die Sitzung oder setzen sie fort – dieser Prozess ist im Grunde aus derselben Form geschnitten.

**Ähnliche Konfigurationsanforderungen**: Alle benötigen API-Schlüssel-Authentifizierung, Arbeitsverzeichniseinstellung, Modellauswahl, Tool-Berechtigungssteuerung, Sitzungsverwaltung. Schließlich leben alle alle von der API, der Unterschied liegt nur im Geschmack.

**Konsistente plattformübergreifende Herausforderungen**: Alle müssen Probleme wie Pfad-Auflösung von ausführbaren Dateien (`claude` vs. `claude.exe` vs `/usr/local/bin/claude`), Prozessstart und Umgebungsvariablenbehandlung, Shell-Befehls-Escaping und Parameter-Building lösen. Diese plattformübergreifenden Dinge sind eine Geschichte voller Tränen – nur die, die in die Falle getappt sind, wissen den Unterschied zwischen Windows und Unix zu schätzen.

Auf der Grundlage dieser Analysen benötigen wir eine einheitliche Abstraktionsschicht, die eine konsistente Schnittstelle bietet, die plattformübergreifende CLI-Erkennungslogik kapselt, das Parsing von Streaming-Ausgaben verarbeitet und gleichzeitig die Verwendung mit und ohne Dependency Injection unterstützt. Diese Sache macht einen kopfschmerzend, aber man muss ihr stellen – es ist schließlich mein eigenes Projekt, ich muss es fertigstellen, auch wenn ich dabei weinen muss.

## Lösung: HagiCode.Libs

Wir haben **HagiCode.Libs** erstellt – einen leichten .NET 10-Bibliotheksarbeitsbereich unter der MIT-Open-Source-Lizenz, jetzt veröffentlicht auf [GitHub](https://github.com/HagiCode-org/Hagicode.Libs). Obwohl es kein Welt bewegendes Meisterwerk ist, löst es echte Probleme ziemlich gut.

### Projektstruktur

```
HagiCode.Libs/
├── src/
│   ├── HagiCode.Libs.Core/           # Kernfunktionen
│   │   ├── Discovery/                 # CLI-ausführbare Datei-Erkennung
│   │   ├── Process/                   # Plattformübergreifende Prozessverwaltung
│   │   ├── Transport/                 # Streaming-Nachrichtentransport
│   │   └── Environment/               # Laufzeitumgebungsauflösung
│   ├── HagiCode.Libs.Providers/       # Provider-Implementierungen
│   │   ├── ClaudeCode/                # Claude Code Provider
│   │   ├── Codex/                     # Codex Provider
│   │   └── Codebuddy/                 # CodeBuddy Provider
│   ├── HagiCode.Libs.ConsoleTesting/  # Testframework
│   ├── HagiCode.Libs.ClaudeCode.Console/
│   ├── HagiCode.Libs.Codex.Console/
│   └── HagiCode.Libs.Codebuddy.Console/
└── tests/                             # xUnit-Tests
```

### Designziele

Bei der Entwicklung von HagiCode.Libs haben wir einige Prinzipien befolgt – schließlich sind es alles Erfahrungen aus Fehlern:

**Keine Abhängigkeit von schweren Frameworks**: Keine Abhängigkeit von ABP oder anderen großen Frameworks, leichtgewichtig bleiben. In diesen Tagen: weniger Abhängigkeiten, weniger Probleme – wer wurde nicht schon von der Abhängigkeitshölle gequält?

**Plattformübergreifende Unterstützung**: Native Unterstützung für Windows, macOS, Linux, ohne unterschiedlichen Code für verschiedene Plattformen schreiben zu müssen. Ein Code für alle, auch ganz nett.

**Streaming-Verarbeitung**: Verwendung von asynchronen Streams zur Verarbeitung der CLI-Ausgabe, mehr im Einklang mit modernem .NET-Programmierstil. Schließlich ändern sich die Zeiten, asynchron ist der Königsweg.

**Flexible Integration**: Sowohl Szenarien mit Dependency Injection als auch direkte Instanziierung werden unterstützt. Jedem das Seine, wie es am bequemsten ist.

## Verwendungsweise

### Über Dependency Injection

Wenn Ihr Projekt bereits Dependency Injection verwendet (z. B. ASP.NET Core oder generischer Host), können Sie es direkt integrieren – das ist auch ein Kind, zwar nicht groß, aber ziemlich brav:

```csharp
using HagiCode.Libs.Providers;
using Microsoft.Extensions.DependencyInjection;

var services = new ServiceCollection();
services.AddHagiCodeLibs();

await using var provider = services.BuildServiceProvider();
var claude = provider.GetRequiredService<ICliProvider<ClaudeCodeOptions>>();

var options = new ClaudeCodeOptions
{
    ApiKey = "your-api-key",
    Model = "claude-sonnet-4-20250514"
};

await foreach (var message in claude.ExecuteAsync(options, "Hello, Claude!"))
{
    Console.WriteLine($"{message.Type}: {message.Content}");
}
```

### Direkte Instanziierung

Bei einfachen Skripten oder Szenarien ohne DI funktioniert auch die direkte Erstellung einer Instanz – ganz einfach gesagt: es ist Geschmackssache:

```csharp
var claude = new ClaudeCodeProvider();
var options = new ClaudeCodeOptions
{
    ApiKey = "sk-ant-xxx",
    Model = "claude-sonnet-4-20250514"
};

await foreach (var message in claude.ExecuteAsync(options, "帮我写一个快速排序"))
{
    // Nachricht verarbeiten
}
```

Beide Methoden verwenden dieselbe zugrundeliegende Implementierung, Sie können die passende Integrationsmethode je nach Projekt wählen. In dieser Welt gibt es keine Standardantwort, was für einen passt, ist das Beste – auch wenn das etwas abgedroschen klingt, es ist wirklich so.

## Erfahrungen aus der Praxis

### 1. Spezielle Testkonsole

Jeder Provider hat ein spezielles Testkonsolenprojekt, um die Integration unabhängig zu validieren – ganz ehrlich: Testing, entweder man macht es nicht oder man macht es gründlich:

```bash
# Claude Code Test
dotnet run --project src/HagiCode.Libs.ClaudeCode.Console -- --test-provider
dotnet run --project src/HagiCode.Libs.ClaudeCode.Console -- --test-all claude

# CodeBuddy Test
dotnet run --project src/HagiCode.Libs.Codebuddy.Console -- --test-provider codebuddy-cli

# Codex Test
dotnet run --project src/HagiCode.Libs.Codex.Console -- --test-provider codex-cli
```

Die Testszenarien decken mehrere Schlüsselszenarien ab:
- **Ping**: Gesundheitscheck, Bestätigung dass CLI verfügbar ist
- **Simple Prompt**: Basis-Prompt-Test
- **Complex Prompt**: Multi-Runden-Konversationstest
- **Session Restore/Resume**: Sitzungswiederherstellungstest
- **Repository Analysis**: Codebasis-Analyse-Test

Dieses unabhängige Testkonsolen-Design ist beim Debuggen besonders nützlich, um schnell zu lokalisieren, ob das Problem in der HagiCode.Libs-Schicht oder in der CLI selbst liegt. Beim Debuggen geht es eben darum, festzustellen, wo das Problem liegt – mit der richtigen Richtung ist man zur Hälfte erfolgreich.

### 2. Plattformübergreifende CI/CD-Validierung

Plattformübergreifende Kompatibilität ist eines der Kernziele von HagiCode.Libs. Wir haben einen GitHub Actions-Workflow `.github/workflows/cli-discovery-cross-platform.yml` konfiguriert, der auf den drei Plattformen `ubuntu-latest`, `macos-latest` und `windows-latest` echte CLI-Erkennungsvalidierungen durchführt.

Dies stellt sicher, dass jede Codeänderung die plattformübergreifende Kompatibilität nicht zerstört. Bei der lokalen Entwicklung können Sie dies mit dem folgenden Befehl reproduzieren – man kann nicht alle Schuld auf die CI schieben, es muss auch lokal laufen:

```bash
npm install --global @anthropic-ai/claude-code@2.1.79
HAGICODE_REAL_CLI_TESTS=1 dotnet test --filter "Category=RealCli"
```

### 3. Nachrichtenstromverarbeitung

HagiCode.Libs verwendet asynchrone Streams zur Verarbeitung der CLI-Ausgabe, eine Methode, die mehr im Einklang mit modernem .NET-asynchronen Programmierstil steht als herkömmliche Callback- oder Ereignismuster – am Ende ist das einfach technologischer Fortschritt, niemand kann ihn aufhalten:

```csharp
public async IAsyncEnumerable<CliMessage> ExecuteAsync(
    TOptions options,
    string prompt,
    [EnumeratorCancellation] CancellationToken cancellationToken = default)
{
    // CLI-Prozess starten
    // Streaming-JSON-Ausgabe parsen
    // CliMessage-Sequenz generieren
}
```

Die Nachrichtentypen umfassen:
- `user`: Benutzernachricht
- `assistant`: Assistentenantwort
- `tool_use`: Tool-Aufruf
- `result`: Sitzungsende

 Dieses Design ermöglicht es dem Aufrufer, Streaming-Ausgaben flexibel zu verarbeiten, z. B. Echtzeitanzeige, Pufferung zur Nachbearbeitung oder Weiterleitung an andere Dienste. Was das Schöne angeht, warum sollte man sich um Sonnenschein oder Wolken kümmern? Wichtig ist, dass der Denkprozess offen ist, jede Verwendung ist möglich.

### 4. Git-Repository-Erkundung

Das HagiCode.Libs.Exploration-Modul bietet Git-Repository-Erkennungs- und Statusprüfungsfunktionen, besonders nützlich in Codeanalyse-Szenarien – auch eine Funktion aus der Not geboren, HagiCode muss schließlich Codebasen analysieren:

```csharp
// Git-Repositories entdecken
var repositories = await GitRepositoryDiscovery.DiscoverAsync("/path/to/search");

// Repository-Informationen abrufen
var info = await GitRepository.GetInfoAsync(repoPath);
Console.WriteLine($"Branch: {info.Branch}, Remote: {info.RemoteUrl}");
Console.WriteLine($"Has uncommitted changes: {info.HasUncommittedChanges}");
```

HagiCodes Codeanalysefunktion verwendet dieses Modul, um Projektstruktur und Git-Status zu erkennen. So wird alles bestens genutzt, auch ein Kind, nicht umsonst aufgezogen.

## Zu beachten

Basierend auf unserer Erfahrung im HagiCode-Projekt gibt es einige Dinge, die besonders beachtet werden müssen – alles Probleme, Probleme, Probleme:

**API-Schlüsselsicherheit**: Hardcodieren Sie keine API-Schlüssel im Code, verwenden Sie Umgebungsvariablen oder Konfigurationsmanagement. HagiCode.Libs unterstützt die Übergabe von Konfigurationen über Options-Objekte, einfach zur Integration verschiedener Konfigurationsquellen. Sicherheit ist schließlich etwas, bei dem man nicht vorsichtig genug sein kann.

**CLI-Versionssperre**: In CI/CD haben wir bestimmte Versionen gesperrt (wie `@anthropic-ai/claude-code@2.1.79`), um Unsicherheiten durch Versionsdrift zu reduzieren. Bei der lokalen Entwicklung wird empfohlen, ebenfalls feste Versionen zu verwenden. Diese Versionsgeschichte ist eine Geschichte voller Tränen – ohne feste Versionen, bringt es dich im Handumdrehen bei.

**Testklassifizierung**: Standardtests verwenden gefälschte Provider, um Determinismus und Geschwindigkeit zu gewährleisten, echte CLI-Tests müssen explizit aktiviert werden. Dies garantiert sowohl schnelles CI-Feedback als auch die Möglichkeit bei Bedarf echte Umgebungsvalidierungen durchzuführen. Auf diesem Balanceakt zu laufen, ist auch nicht leicht – schnell und stabil, immer ein Kompromiss.

**Sitzungsverwaltung**: Die Sitzungswiederherstellungsmechanismen verschiedener CLIs sind unterschiedlich, Claude Code verwendet das `.claude/`-Verzeichnis zur Speicherung von Sitzungen, Codex und CodeBuddy haben jeweils ihre eigenen Methoden. Achten Sie bei der Verwendung auf die jeweilige Dokumentation, um den spezifischen Mechanismus der Sitzungspersistenz zu verstehen. Das ist auch okay, verstanden zu haben, schadet nie.

## Zusammenfassung

HagiCode.Libs ist eine einheitliche Abstraktionsschicht, die wir während der Entwicklung von HagiCode erstellt haben, um das Problem der doppelten Ingenieursarbeit bei der Integration mehrerer CLIs zu lösen. Durch die Bereitstellung einer konsistenten Schnittstelle, die Kapselung plattformübergreifender Details und die Unterstützung flexibler Integrationsmethoden wird die technische Komplexität der Integration mehrerer KI-Programmier-Assistenten erheblich reduziert. Alles ist verblasst – aber die Erfahrung bleibt.

Wenn Sie auch mehrere AI-CLI-Tools in Ihrem Projekt integrieren müssen oder sich für plattformübergreifende Prozessverwaltung oder Streaming-Nachrichtenverarbeitung interessieren, besuchen Sie uns gerne auf [GitHub](https://github.com/HagiCode-org/Hagicode.Libs). Das Projekt verwendet die MIT-Open-Source-Lizenz, Beiträge und Feedback sind willkommen. Am Ende ist es eine Schicksalsgemeinschaft, wir sind schon da, stellen wir uns Freunde vor.
