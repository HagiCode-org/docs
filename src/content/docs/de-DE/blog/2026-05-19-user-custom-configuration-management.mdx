## Best Practices für benutzerdefinierte Konfigurationsverwaltung

> Nichts Besonderes, nur eine Konfigurationsverwaltungslösung zum Teilen. Von Architektur bis Praxis – nun ja, ich hoffe, es hilft Ihnen, einige Fallstricke zu vermeiden.

## Hintergrund

Konfigurationsverwaltung ist wichtig, aber auch Details sind wichtig. Es ist wie die kleinen Gewohnheiten im Leben – manche trinken morgens gerne Kaffee, andere codieren lieber spät in die Nacht. Diese scheinbar unbedeutenden Vorlieben definieren eigentlich stillschweigend, wer Sie sind. Benutzerdefinierte Konfigurationen sind ähnlich: Themenwechsel, Sprachauswahl, Tastenkürzel-Anpassung – wenn diese Funktionen gut gemacht sind, fühlen sich die Benutzer vom Produkt „verstanden“, und die Bindung steigt natürlich.

Allerdings ist es nicht so einfach, dieses System vollständig umzusetzen. Versionsverwaltung, Abwärtskompatibilität, Datenvalidierung, Nebenläufigkeitssteuerung, DLC-Funktionsgates – diese Probleme tauchen wie Pubertätsprobleme eins nach dem anderen auf und lassen sich nicht vermeiden. Auch wir bei HagiCode wurden von diesen Problemen gequält, aber am Ende haben wir einige Lösungen gefunden.

## Über HagiCode

Diese Lösung ist eigentlich das, was wir in HagiCode herausgearbeitet haben. HagiCode ist ein KI-Codierungsassistent mit vielen Funktionen – Oberflächensprache, KI-Sprachepräferenzen, Themen, Spracherkennung, Benachrichtigungen, Schnellaktionen – es gibt viel zu konfigurieren. Gerade weil Benutzer so viel wollen, mussten wir dieses System entwickeln.

## Architekturdesign

### Schichtenarchitektur

Das Konfigurationsverwaltungssystem von HagiCode ist nun ja, in mehrere Schichten unterteilt, jede mit ihren eigenen Aufgaben:

```
┌─────────────────────────────────────────┐
│         Frontend (React + Redux)        │
│  - Konfigurationszustandsverwaltung     │
│  - UI-Formular-Rendering                │
│  - Konfiguration nach Gruppen speichern │
└─────────────────────────────────────────┘
                    ↓ HTTP/REST API
┌─────────────────────────────────────────┐
│   Application Service Layer             │
│  - FrontendConfigAppService             │
│  - Geschäftslogikverarbeitung           │
│  - Berechtigungssteuerung               │
└─────────────────────────────────────────┘
                    ↓
┌─────────────────────────────────────────┐
│   Domain Layer (Config Store)           │
│  - FrontendConfigStore                  │
│  - Konfiguration lesen/schreiben        │
│  - Datenvalidierung und Normalisierung  │
└─────────────────────────────────────────┘
                    ↓
┌─────────────────────────────────────────┐
│   Infrastructure Layer                  │
│  - YAML-Dateispeicherung                │
│  - ISystemManagedVaultService           │
│  - Nebenläufigkeitssteuerung (SemaphoreSlim) │
└─────────────────────────────────────────┘
```

Die Vorteile dieser Schichtung sind eigentlich ziemlich offensichtlich:
- **Klare Verantwortlichkeiten**: Jede Schicht verwaltet ihre eigenen Aufgaben, ohne sich gegenseitig zu stören
- **Einfach zu testen**: Jede Schicht kann separat getestet werden, was Änderungen sicherer macht
- **Flexibel erweiterbar**: Speichermethode oder API ändern – die anderen Schichten laufen weiter

### Kernschnittstellendesign

Die Schnittstelle für die Backend-Konfigurationsspeicherung ist ungefähr so definiert:

```csharp
public interface IFrontendConfigStore
{
    // Vollständige Benutzerkonfiguration abrufen
    Task<FrontendConfigStoreResult> GetAsync(CancellationToken cancellationToken = default);
    
    // Konfiguration aktualisieren (unterstützt Teilaktualisierungen)
    Task<FrontendConfigStoreResult> UpdateAsync(
        UpdateFrontendConfigRequestDto input,
        CancellationToken cancellationToken = default);
    
    // KI-Sprachzustandsverwaltung
    Task<FrontendConfigAiLanguageState> GetAiLanguageStateAsync(
        string userId,
        CancellationToken cancellationToken = default);
        
    Task<FrontendConfigAiLanguageState> SetAiLanguageAsync(
        string userId,
        string language,
        CancellationToken cancellationToken = default);
}
```

Die wichtigsten Punkte dieser Schnittstelle sind eigentlich ziemlich verständlich:
- **Asynchrone Operationen**: Alles ist asynchron, schließlich will niemand warten
- **Abbruchtoken**: Operationen, die zu lange dauern, timeouten, nicht ewig blockieren
- **Teilaktualisierungen**: Nur aktualisieren, was geändert werden muss, nicht die gesamte Konfiguration durchgehen

Konfigurationsgruppen sind wie das Klassifizieren von Lebensdingen – Arbeit ist Arbeit, Freizeit ist Freizeit, Beziehungen sind Beziehungen. Wenn man alles mischt, wird es chaotisch; klar getrennt wird es einfacher:
- **Aktualisierung bei Bedarf**: Nur aktualisieren, was geändert wird, nicht alles auf einmal beeinflussen
- **Berechtigungssteuerung**: Verschiedene Konfigurationen können verschiedene Berechtigungen haben, schließlich kann nicht alles einfach so geändert werden
- **DLC-Gates**: Erweiterte Funktionen können mit DLCs verknüpft werden – wer gute Funktionen nutzen will, muss dafür bezahlen

## Datenvalidierung und Normalisierung

### Sprachnormalisierungsbeispiel

Konfigurationsnormalisierung bedeutet im Grunde, Daten konsistent zu halten. Am Beispiel der Sprachkonfiguration:

```csharp
public static class FrontendConfigLanguageRules
{
    // Liste der unterstützten Oberflächensprachen
    public static readonly HashSet<string> ValidUiLanguages = new(StringComparer.OrdinalIgnoreCase)
    {
        "zh-CN", "zh-Hant", "en-US", "ja-JP", "ko-KR", 
        "de-DE", "fr-FR", "es-ES", "pt-BR", "ru-RU"
    };
    
    public static string NormalizeUiLanguage(string? value)
    {
        // Leere Wertbehandlung: Standardsprache zurückgeben
        if (string.IsNullOrWhiteSpace(value)) return "en-US";
        
        var normalized = value.Trim();
        
        // Aliasbehandlung: Häufige Aliase in Standardcodes umwandeln
        normalized = normalized.ToLower() switch
        {
            "zh" or "chinese" or "cn" => "zh-CN",
            "en" or "english" => "en-US",
            "ja" or "japanese" => "ja-JP",
            "ko" or "korean" => "ko-KR",
            _ => normalized
        };
        
        // Dialektvariantenbehandlung
        if (normalized.StartsWith("zh-Hans", StringComparison.OrdinalIgnoreCase))
            return "zh-CN";
        if (normalized.StartsWith("zh-TW", StringComparison.OrdinalIgnoreCase))
            return "zh-Hant";
            
        // Validierung und Rückgabe
        return ValidUiLanguages.Contains(normalized) ? normalized : "en-US";
    }
}
```

Normalisierung ist wie das Aufräumen eines Zimmers – wenn Dinge chaotisch sind, müssen sie sortiert werden, sonst findet man selbst nichts mehr:
1. **Leere-Wert-Grundlage**: Leeren Werten einen angemessenen Standardwert geben, kann nicht einfach leer bleiben
2. **Aliaszuordnung**: Häufige Aliase, Abkürzungen, einheitlich in Standardformat umwandeln
3. **Dialektvereinheitlichung**: Dialektvarianten zu Standardcodes zusammenfassen, schließlich ist Codieren keine Dialektforschung
4. **Endgültige Validierung**: Sicherstellen, dass der zurückgegebene Wert unbedingt in der gültigen Liste ist, sonst war alles umsonst

## DLC-Funktionsgates

HagiCode unterstützt DLC-Funktionsschalter. Einige erweiterte Konfigurationsoptionen erfordern den Kauf eines DLCs. Dies ist in kommerzieller Software quite häufig – Basisfunktionen kostenlos, wer gute Funktionen nutzen will, muss bezahlen, schließlich müssen Entwickler auch essen.

### DLC-Zugriffsprüfung

```csharp
private async Task<PreparedFrontendConfigUpdate> PrepareUpdateAsync(
    UpdateFrontendConfigRequestDto input,
    FrontendConfigStoreResult current,
    CancellationToken cancellationToken)
{
    // DLC-Zugriffsrechte prüfen
    var accessState = await _grainFactory
        .GetDlcAccessStateGrain(TurboEngineDlcId)
        .GetAccessStateAsync();
        
    var blockedFields = new List<string>();
    
    if (!accessState.IsActive)
    {
        // DLC nicht aktiviert, aktuelle Konfiguration beibehalten
        if (input.GeneralSettings?.BrandingLogo != null)
        {
            blockedFields.Add("brandingLogo");
            // Alten Wert beibehalten
            input.GeneralSettings.BrandingLogo = current.Snapshot.GeneralSettings.BrandingLogo;
        }
        
        if (input.GeneralSettings?.BrandingTitle != null)
        {
            blockedFields.Add("brandingTitle");
            input.GeneralSettings.BrandingTitle = current.Snapshot.GeneralSettings.BrandingTitle;
        }
    }
    
    return new PreparedFrontendConfigUpdate(
        input,
        new FrontendConfigUpdateDiagnosticsDto
        {
            Status = blockedFields.Count > 0 ? "partially-applied" : "success",
            BlockedFields = blockedFields,
        });
}
```

## Frontend-Zustandsverwaltung

### Redux Slice

Das Frontend verwaltet den Konfigurationszustand mit Redux, eigentlich auch quite Standard:

```typescript
export const frontendConfigSlice = createSlice({
  name: 'frontendConfig',
  initialState,
  reducers: {
    setConfigStatus(state, action: PayloadAction<FrontendConfigStatus>) {
      state.status = action.payload;
    },
    updateConfigGroups(state, action: PayloadAction<Partial<FrontendConfigSnapshot>>) {
      // Konfigurationsaktualisierungen zusammenführen
      Object.assign(state.snapshot, action.payload);
    },
  },
  extraReducers: (builder) => {
    builder
      .addCase(fetchConfig.pending, (state) => {
        state.status = 'loading';
      })
      .addCase(fetchConfig.fulfilled, (state, action) => {
        state.status = 'succeeded';
        state.snapshot = action.payload;
      })
      .addCase(fetchConfig.rejected, (state, action) => {
        state.status = 'failed';
        state.error = action.error.message;
      });
  },
});
```

## Nebenläufigkeitssteuerung

Konfigurationsaktualisierungsoperationen müssen threadsicher sein, sonst wenn zwei Personen gleichzeitig Konfigurationen ändern, wessen Änderungen werden am Ende gespeichert? HagiCode verwendet `SemaphoreSlim` für Nebenläufigkeitssteuerung, nun ja, eine quite häufige Methode:

```csharp
private readonly SemaphoreSlim _semaphore = new(1, 1);

public async Task<FrontendConfigStoreResult> UpdateAsync(
    UpdateFrontendConfigRequestDto input,
    CancellationToken cancellationToken = default)
{
    await _semaphore.WaitAsync(cancellationToken);
    try
    {
        // Aktuelle Konfiguration lesen
        var current = await GetAsync(cancellationToken);
        
        // Aktualisierung vorbereiten
        var prepared = await PrepareUpdateAsync(input, current, cancellationToken);
        
        // Konfiguration schreiben
        await WriteConfigAsync(prepared.UpdatedConfig, cancellationToken);
        
        return new FrontendConfigStoreResult(prepared.UpdatedConfig, prepared.Diagnostics);
    }
    finally
    {
        _semaphore.Release();
    }
}
```

### Umgang mit häufigen Problemen

1. **Konfigurationsverlust**: Backup- und Wiederherstellungsmechanismen, immer auf Nummer sicher gehen
2. **Nebenläufigkeitskonflikte**: Optimistische oder pessimistische Sperren, wie SemaphoreSlim
3. **Leistungsprobleme**: Teilaktualisierungen unterstützen, nicht jedes Mal das gesamte Konfigurationsobjekt hin- und herschieben
4. **Sicherheit**: Sensible Konfigurationen (wie API-Schlüssel) müssen verschlüsselt gespeichert werden, sonst wird es problematisch, wenn sie gestohlen werden
