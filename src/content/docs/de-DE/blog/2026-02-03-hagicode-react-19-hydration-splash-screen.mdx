## 12 außergewöhnliche Startup-Erlebnisse für HagiCode: Von Minimalismus bis Cyberpunk

> Die kurze Lücke zwischen dem Herunterladen einer React 19-Anwendung und dem Abschluss der Hydration ist eine goldene Gelegenheit, um Nutzer mit der Markenpersönlichkeit in Berührung zu bringen. In diesem Artikel teilen wir ein vollständiges Startup-Stilsystem, das wir für das HagiCode-Projekt mit HTML/CSS/JS entwickelt haben.

## Hintergrund

Als moderne Anwendung, die mit ASP.NET Core 10 und React 19 (Vite) gebaut ist, verwendet HagiCode eine Frontend-Backend-getrennte Bereitstellungsarchitektur. Das Frontend-Build-Ergebnis wird im Backend-Verzeichnis `wwwroot/` verpackt und von ASP.NET Core gehostet.

Diese Architektur führt jedoch zu einem klassischen UX-Schmerzpunkt: Wenn Nutzer die Seite besuchen, lädt der Browser zuerst das HTML, dann das große JS-Bundle und lässt schließlich React die Hydration durchführen. Während dieser „Vakuumzeit" von einigen hundert Millisekunden bis mehreren Sekunden sehen die Nutzer entweder einen weißen Bildschirm oder eine leblose statische Seite.

Um diese Lücke zu schließen und die Markenpersönlichkeit von HagiCode einzubringen, mussten wir ein Startup-Stilsystem entwerfen, das vollständig mit Inline-Code in `index.html` implementiert ist.

## Über HagiCode

Der in diesem Artikel geteilte Splash-Screen-Designansatz stammt aus unserer praktischen Erfahrung im [HagiCode](https://github.com/HagiCode-org/site)-Projekt. ## Kernherausforderungen und Architekturentwurf

Bevor wir mit dem Design begannen, mussten wir zunächst die technischen Rahmenbedingungen klären. Da alles inline in `index.html` implementiert werden musste, konnten wir keine externen CSS- oder JS-Dateien laden, abgesehen von Reacts eigenem Bundle.

### Analyse der technischen Rahmenbedingungen

1.  **Null-Abhängigkeits-Prinzip**: Alle Styles müssen in einem `<style>`-Tag liegen, die gesamte Logik in einem `<script>`-Tag.
2.  **Defensives CSS**: Um zu verhindern, dass globale Styles den Splash Screen nach dem Mounten der React-App verunreinigen, entschieden wir uns, alle Startup-Styles mit einem hochpriorisierten ID-Präfix wie `#boot-screen` zu umschließen.
3.  **Leistung zuerst**: Animationen sollten nach Möglichkeit CSS `transform` und `opacity` verwenden, um Reflows zu vermeiden und sicherzustellen, dass der Hauptthread nicht blockiert wird.
4.  **Visuelle Konsistenz**: Farben und Schriftarten müssen mit HagiCodes Tailwind-Konfiguration übereinstimmen.

### Architekturmuster: Shell & Injector

Wir adoptierten ein **Varianten-Muster**. Die Kernlogik ist in einem Immediately Invoked Function Expression (IIFE) gekapselt, während die spezifische Renderlogik durch Konfiguration injiziert wird. Das ermöglicht uns, zwischen verschiedenen Stilen durch einfache Konfiguration zu wechseln, anstatt DOM-Manipulationslogik wiederholt neu zu schreiben.

Hier ist der Kernarchitekturcode:

```html
<!-- Inline in index.html -->
<div id="boot-root"></div>

<script>
(function() {
  const BootSequence = {
    config: {
      theme: 'terminal', // Konfigurierbar als 'minimal', 'skeleton', 'code-rain' usw.
      color: '#3b82f6'   // Markenfarbe
    },

    // Kern-Lebenszyklus
    init() {
      this.render();
      this.listenForMount();
    },

    // Den aktuell gewählten Stil rendern
    render() {
      const root = document.getElementById('boot-root');
      if (this.variants[this.config.theme]) {
        root.innerHTML = this.variants[this.config.theme].render();
      }
    },

    // Auf erfolgreiches React-Mount lauschen und sauber beenden
    listenForMount() {
      window.addEventListener('hagicode:ready', () => {
        const screen = document.getElementById('boot-root');
        // Zuerst ausblenden, dann DOM entfernen, um Flackern zu vermeiden
        screen.style.opacity = '0';
        screen.style.transition = 'opacity 0.3s ease';
        setTimeout(() => screen.remove(), 300);
      });
    },

    // Die Implementierungslogik für alle 12 Stile befindet sich hier
    variants: {
      // ...Details unten
    }
  };

  BootSequence.init();
})();
</script>
```

## Checkliste: 12 Startup-Stil-Designs

Wir haben diese 12 Stile in sechs Hauptkategorien unterteilt, um verschiedene Szenarien und ästhetische Präferenzen abzudecken.

### A. Minimalismus

> „Weniger ist mehr." Für Szenarien, die maximale Ladegeschwindigkeit anstreben, bieten wir die leichtgewichtigsten Optionen.

#### 1. Minimalistischer Punkt
Ein einfacher Punkt sitzt in der Bildschirmmitte, gepaart mit einer Atem-Animation.
*   **Implementierung**: CSS `@keyframes` steuert Scale und Opacity.
*   **Am besten für**: Jeden Fall, in dem die Seite absolut sauber bleiben muss.

#### 2. Marken-Enthüllung
Mithilfe der SVG `stroke-dasharray`-Animation wird eine handgezeichnete Enthüllung der HagiCode-Logo-Linien simuliert, gefolgt von einem Text-Fade-In.
*   **Technik**: SVG-Pfad-Animation mit hochgradig veredeltem Gefühl.

### B. Skeleton-Screen-Nachahmung

> „Die Kunst der optischen Täuschung." Durch die Simulation eines echten UI-Layouts haben Nutzer das Gefühl, die Seite sei bereits halb geladen.

#### 3. Sidebar-Chat-Skeleton
Das ist möglicherweise die praktischste Option. Wir bauten manuell ein Layout in HTML, das die React `Sidebar`- und `ChatInput`-Komponenten exakt widerspiegelt, und legten darüber eine graue Shimmer-Animation.
*   **Wert**: Wenn die React-Hydration abgeschlossen ist, wird das Skeleton sofort zur echten Komponente, und Nutzer bemerken den Wechsel kaum.

#### 4. Kartenstapel-Skeleton
Simuliert die gestapelte Bewegung von Proposal-Karten beim Laden, wobei 3D-Transforms verwendet werden, um die Karten subtil schweben zu lassen.

### C. Abstraktion und Kunst

> HagiCodes Geek-DNA zeigen.

#### 5. Geometrische Metamorphose
Eine geometrische Form (ein Quadrat) wird in der Bildschirmmitte gerendert und verwandelt sich dann reibungslos über die Zeit in einen Kreis, ein Dreieck und schließlich das Logo.
*   **Technologie**: Sanfte Übergänge mit CSS `border-radius`.

#### 6. Code-Regen
Eine Hommage an *The Matrix*. Mit der Schriftart JetBrains Mono fallen schwache Zeichenströme im Hintergrund.
*   **Hinweis**: Aus Performance-Gründen sollten die Zeichenströme auf einen kleineren Bereich beschränkt oder eine niedrigere Refresh-Rate verwendet werden.

#### 7. Neon-Puls
Ein Cyberpunk-artiger Leuchtring, der mehrere `box-shadow`-Schichten nutzt, um ein starkes Neon-Glow zu erzeugen.

### D. Branding und Themes

> Das System lebendig machen.

#### 8. Saisonales Theme
Das ist ein dynamischer Loader. Er prüft das aktuelle Datum auf Feiertage wie das Mondneujahr oder Weihnachten und lädt die entsprechende SVG-Animation.
*   **Beispiel**: Während des Mondneujahrs schwenken rote Laternen sanft am unteren Bildschirmrand.

#### 9. Gradientenfluss
Der Hintergrund verwendet einen fließenden Gradienten basierend auf HagiCode-Markenfarben. Kombiniert mit animiertem `background-size` und `background-position` entsteht ein aurora-artiges Bewegungsgefühl.

### E. Technisches Gefühl

> Ein Gruß an Entwickler.

#### 10. Terminal-Boot
Simuliert die Konsolenausgabe. Zeilen von Code scrollen schnell vorbei:
```text
> HagiCode Core initialisieren...
> Modelle laden...
> Mit neuronalem Netzwerk verbinden...
```
Das fühlt sich für jeden Entwickler sofort vertraut an.

#### 11. Fortschrittsbalken
Ein dünner Fortschrittsbalken erscheint oben auf dem Bildschirm, mit einem Prozentwert auf der rechten Seite. Während wir nicht auf den tatsächlichen Download-Fortschritt zugreifen können, können wir einen Timer nutzen, um einen „glaubwürdigen" Ladevorgang zu simulieren: schnell für die ersten 80 %, dann allmählich langsamer für die letzten 20 %.

### F. Kreativität

#### 12. Pixel-Assembly
Das ist eine sehr interessante Idee. Kleine Quadrate sind über den Bildschirm verstreut, konvergieren dann zur Mitte und setzen sich schrittweise zum HagiCode-Logo-Icon zusammen. Es symbolisiert den Prozess des Code-Aufbaus.

## Best Practices und Fallstricke

In der praktischen Entwicklungsarbeit von HagiCode haben wir mehrere kritisch wichtige Implementierungsdetails zusammengefasst.

### 1. Defensives CSS ist obligatorisch
Werden Sie nicht faul und überspringen Sie das Präfix nicht. Einmal vergaßen wir, die Splash-Screen-Styles mit einer ID einzuschränken, und globale `div`-Styles nach dem React-Mount beeinflussten unerwartet den Splash Screen und zerstörten das Layout.
**Gelernte Lektion**: Setzen Sie jeden CSS-Selektor unter `#boot-screen` und verwenden Sie `!important`, um die Priorität zu erhöhen, aber nur innerhalb des Splash-Screen-CSS.

### 2. Sanfte Übergänge
Nach erfolgreichem React-Mount den Splash-Screen-DOM nicht direkt `remove()`.
**Korrekter Ansatz**:
1.  React löst `window.dispatchEvent(new Event('hagicode:ready'))` aus.
2.  Der Splash Screen lauscht auf das Ereignis und setzt zuerst `opacity: 0`.
3.  300 ms warten (entspricht der CSS-Übergangsdauer) und erst dann `.remove()` aufrufen, wenn der Bildschirm vollständig unsichtbar ist.

### 3. Theme-Variablen-Synchronisation
Die Farbwerte des Splash Screens sind in `index.html` hart codiert. Wenn wir die Primärfarbe von Tailwind ändern, müssen wir auch den Splash Screen aktualisieren.
**Optimierung**: Schreiben Sie ein einfaches Plugin im Vite-Build-Skript, das `tailwind.config.js` liest und Farbvariablen in die `index.html`-Template-Variablen injiziert, um eine Single Source of Truth zu schaffen.

### 4. Schriftart-Vorladen
Splash Screens benötigen oft eine Markenschriftart, aber wenn die Schriftart langsam lädt, kann FOUT (Flash of Unstyled Text) auftreten.
**Lösung**: Fügen Sie `<link rel="preload" href="/fonts/JetBrainsMono.woff2" as="font" type="font/woff2" crossorigin>` in den `<head>` ein. Das ist ein kostengünstiger, ertragreicher Weg zur Verbesserung des Erlebnisses.

### 5. Leistungsüberwachung
Wir injizierten `performance.mark('boot-start')` am Ende von `index.html` und markierten `boot-end`, wenn React erfolgreich gemountet wurde.
**Warum das wichtig ist**: Durch die Erfassung dieser Daten über Application Insights können wir direkt messen, wie viel der Splash Screen die wahrgenommene Wartezeit verkürzt. Die Daten zeigen, dass ein hervorragender Skeleton Screen die Toleranz der Nutzer für ein „langsames Netzwerk" um mehr als 50 % verbessern kann.

Ein guter Splash Screen ist mehr als „Dekoration beim Warten". Er ist das Handshake-Signal bei der allerersten Interaktion zwischen Produkt und Nutzer. Im HagiCode-Projekt lässt uns dieses Startup-System auf Basis des **Varianten-Musters** flexibel zwischen Feiertagen und Releases Stile wechseln, was den Spaßfaktor und die Professionalität des Produkts erheblich steigert.
