## Vollständige Leitfaden zur Codex SDK-Konsolennachrichtenanalyse

> Dieser Artikel stellt detailliert den Event-Stream-Mechanismus des Codex SDK, die Analyse verschiedener Nachrichtentypen sowie bewährte Praktiken in realen Projekten vor, um Entwicklern zu helfen, die Kernfähigkeiten des AI-Ausführungsdiensts schnell zu beherrschen.

## Hintergrund

Bei der Entwicklung von AI-Ausführungsdiensten auf der Grundlage von Codex SDK müssen wir uns mit einem Problem auseinandersetzen: dem Umgang mit den Streaming-Ereignisnachrichten, die von Codex zurückgegeben werden. Diese Nachrichten enthalten wichtige Informationen wie Ausführungsstatus, Ausgabeinhalte und Fehlerinformationen – ähnlich wie die unklaren Gefühle in der Jugend, die man sorgfältig analysieren muss.

Als Teil des HagiCode-Projekts müssen wir einen zuverlässigen Executor in AI-Codeszenarien implementieren. Das ist ungefähr der Grund, warum wir uns entschlossen haben, den Event-Stream-Mechanismus des Codex SDK gründlich zu untersuchen – schließlich kann man nur durch das Verständnis, wie die zugrundeliegenden Nachrichten funktionieren, eine wirklich unternehmensweit einsetzbare AI-Ausführungsplattform aufbauen. Das ist wie bei einer Beziehung: Wenn man die Gedanken des anderen nicht versteht, wie kann man dann weitermachen?

Codex SDK ist das von OpenAI entwickelte SDK für Programmiertools, das Ausführungsergebnisse über Event Streams zurückgibt. Im Gegensatz zum herkömmlichen Anfrage-Antwort-Modell verwendet Codex Streaming-Ereignisse, die es uns ermöglichen:

- Den Ausführungsfortschritt in Echtzeit zu verfolgen
- Fehler rechtzeitig zu behandeln
- Detaillierte Token-Verwendungsstatistiken zu erhalten
- Lange laufende komplexe Aufgaben zu unterstützen

Das Verständnis dieser Ereignistypen und ihre korrekte Analyse sind für die Implementierung eines funktionsfähigen AI-Executors ziemlich wichtig. Schließlich möchte niemand mit einer Black Box konfrontiert werden.

## Über HagiCode

Die in diesem Artikel vorgestellte Lösung stammt aus unserer praktischen Erfahrung im [HagiCode](https://hagicode.com)-Projekt. HagiCode ist ein Open-Source-AI-Codesassistent-Projekt, das Entwicklern intelligente Codeunterstützung bieten möchte. Bei der Entwicklung müssen wir einen zuverlässigen AI-Ausführungsdienst erstellen, um die Codeausführungsanforderungen der Benutzer zu verarbeiten – genau das ist der direkte Grund, warum wir Codex SDK eingeführt haben.

Als AI-Codesassistent muss HagiCode verschiedene komplexe Codeausführungsszenarien verarbeiten: Echtzeit-Verfolgung des Ausführungsfortschritts, rechtzeitige Fehlerbehandlung, detaillierte Token-Verwendungsstatistiken usw. Durch das tiefgreifende Verständnis des Event-Stream-Mechanismus des Codex SDK können wir Executors erstellen, die den Produktionsanforderungen gerecht werden. Am Ende brauchen sowohl Code als auch Leben ein wenig Ansammlung und Reifung.

## Event-Stream-Mechanismus

### Grundlegende Konzepte

Codex SDK verwendet die Methode `thread.runStreamed()`, um einen asynchronen Ereignis-Iterator zurückzugeben:

```typescript
import { Codex } from '@openai/codex-sdk';

const client = new Codex({
  apiKey: process.env.CODEX_API_KEY,
  baseUrl: process.env.CODEX_BASE_URL,
});

const thread = client.startThread({
  workingDirectory: '/path/to/project',
  skipGitRepoCheck: false,
});

const { events } = await thread.runStreamed('your prompt here', {
  outputSchema: {
    type: 'object',
    properties: {
      output: { type: 'string' },
      status: { type: 'string', enum: ['ok', 'action_required'] },
    },
    required: ['output', 'status'],
  },
});

for await (const event of events) {
  // Verarbeite jedes Ereignis
}
```

### Detaillierte Erläuterung der Ereignistypen

| Ereignistyp | Beschreibung | Schlüsseldaten |
|---------|------|----------|
| `thread.started` | Thread erfolgreich gestartet | `thread_id` |
| `item.updated` | Nachrichteninhalt aktualisiert | `item.text` |
| `item.completed` | Nachricht abgeschlossen | `item.text` |
| `turn.completed` | Ausführung abgeschlossen | `usage` (Token-Verwendung) |
| `turn.failed` | Ausführung fehlgeschlagen | `error.message` |
| `error` | Fehlerereignis | `message` |

In realen Projekts basiert die Executor-Komponente von HagiCode genau auf diesen Ereignistypen. Wir müssen jedes Ereignis sorgfältig verarbeiten, um ein reibungsloses Benutzererlebnis zu gewährleisten. Das ist wie bei einer Beziehung: Jedes Detail muss mit Sorgfalt behandelt werden, sonst kann es kein gutes Ergebnis geben.

## Implementierung der Nachrichtenanalyse

### Extraktion des Nachrichteninhalts

Der Nachrichteninhalt wird durch die Ereignisverarbeitungsfunktion extrahiert:

```typescript
private handleThreadEvent(event: ThreadEvent, onMessage: (content: string) => void): void {
  // Verarbeite nur Nachrichtenaktualisierungs- und Abschlussereignisse
  if (event.type !== 'item.updated' && event.type !== 'item.completed') {
    return;
  }

  // Verarbeite nur Agentennachrichtentypen
  if (event.item.type !== 'agent_message') {
    return;
  }

  // Extrahiere Textinhalt
  onMessage(event.item.text);
}
```

Wichtige Punkte:
- Verarbeite nur `item.updated`- und `item.completed`-Ereignisse
- Verarbeite nur Inhalte vom Typ `agent_message`
- Der Nachrichteninhalt befindet sich im Feld `event.item.text`

### Analyse strukturierter Ausgaben

Codex unterstützt strukturierte JSON-Ausgaben. Das Rückgabeformat wird über den Parameter `outputSchema` angegeben:

```typescript
const DEFAULT_OUTPUT_SCHEMA = {
  type: 'object',
  properties: {
    output: { type: 'string' },
    status: { type: 'string', enum: ['ok', 'action_required'] },
  },
  required: ['output', 'status'],
  additionalProperties: false,
} as const;
```

Die Analysefunktion versucht, JSON zu analysieren. Wenn dies fehlschlägt, wird der ursprüngliche Text zurückgegeben – ähnlich wie im Leben: Manchmal möchten Sie eine perfekte Antwort, aber die Realität gibt Ihnen oft eine vage Reaktion, und Sie müssen sie selbst verdauen.

```typescript
function toStructuredOutput(raw: string): StructuredOutput {
  try {
    const parsed = JSON.parse(raw) as Partial<StructuredOutput>;
    if (typeof parsed.output === 'string') {
      return {
        output: parsed.output,
        status: parsed.status === 'action_required' ? 'action_required' : 'ok',
      };
    }
  } catch {
    // JSON-Analyse fehlgeschlagen, Fallback auf ursprünglichen Text
  }

  return {
    output: raw,
    status: 'ok',
  };
}
```

### Vollständiger Ereignisverarbeitungsablauf

```typescript
private async runWithStreaming(
  thread: Thread,
  input: CodexStageExecutionInput
): Promise<{ output: string; usage: Usage | null }> {
  const abortController = new AbortController();
  const timeoutHandle = setTimeout(() => {
    abortController.abort();
  }, Math.max(1000, input.timeoutMs));

  let latestMessage = '';
  let usage: Usage | null = null;
  let emittedLength = 0;

  try {
    const { events } = await thread.runStreamed(input.prompt, {
      outputSchema: DEFAULT_OUTPUT_SCHEMA,
      signal: abortController.signal,
    });

    for await (const event of events) {
      // Verarbeite Nachrichteninhalt
      this.handleThreadEvent(event, (nextContent) => {
        const delta = nextContent.slice(emittedLength);
        if (delta.length > 0) {
          emittedLength = nextContent.length;
          input.callbacks?.onChunk?.(delta);  // Streaming-Callback
        }
        latestMessage = nextContent;
      });

      // Verarbeite verschiedene Daten nach Ereignistyp
      if (event.type === 'thread.started') {
        this.threadId = event.thread_id;
      } else if (event.type === 'turn.completed') {
        usage = event.usage;
      } else if (event.type === 'turn.failed') {
        throw new CodexExecutorError('gateway_unavailable', event.error.message, true);
      } else if (event.type === 'error') {
        throw new CodexExecutorError('gateway_unavailable', event.message, true);
      }
    }
  } catch (error) {
    if (abortController.signal.aborted) {
      throw new CodexExecutorError(
        'upstream_timeout',
        `Codex stage timed out after ${input.timeoutMs}ms`,
        true
      );
    }
    throw error;
  } finally {
    clearTimeout(timeoutHandle);
  }

  const structured = toStructuredOutput(latestMessage);
  return { output: structured.output, usage };
}
```

## Fehlerbehandlungsstrategien

### Fehlercode-Zuordnung

Zuordnung zu spezifischen Fehlercodes basierend auf Fehlermerkmalen, um eine einfachere Verarbeitung auf höherer Ebene zu ermöglichen:

```typescript
function mapError(error: unknown): CodexExecutorError {
  if (error instanceof CodexExecutorError) {
    return error;
  }

  const message = error instanceof Error ? error.message : String(error);
  const normalized = message.toLowerCase();

  // Authentifizierungsfehler - nicht wiederholbar
  if (normalized.includes('401') ||
      normalized.includes('403') ||
      normalized.includes('api key') ||
      normalized.includes('auth')) {
    return new CodexExecutorError('auth_invalid', message, false);
  }

  // Ratenbegrenzung - wiederholbar
  if (normalized.includes('429') || normalized.includes('rate limit')) {
    return new CodexExecutorError('rate_limited', message, true);
  }

  // Timeout-Fehler - wiederholbar
  if (normalized.includes('timeout') || normalized.includes('aborted')) {
    return new CodexExecutorError('upstream_timeout', message, true);
  }

  // Standardfehler
  return new CodexExecutorError('gateway_unavailable', message, true);
}
```

### Definition der Fehlertypen

```typescript
export type CodexErrorCode =
  | 'auth_invalid'      // Authentifizierung fehlgeschlagen
  | 'upstream_timeout'  // Upstream-Timeout
  | 'rate_limited'      // Ratenbegrenzung
  | 'gateway_unavailable'; // Gateway nicht verfügbar

export class CodexExecutorError extends Error {
  readonly code: CodexErrorCode;
  readonly retryable: boolean;

  constructor(code: CodexErrorCode, message: string, retryable: boolean) {
    super(message);
    this.name = 'CodexExecutorError';
    this.code = code;
    this.retryable = retryable;
  }
}
```

## Arbeitsverzeichnis und Umgebungskonfiguration

### Validierung des Arbeitsverzeichnisses

Codex SDK erfordert, dass das Arbeitsverzeichnis ein gültiges Git-Repository ist – ähnlich wie bei Menschen: Man braucht eine Wurzel, einen Ursprung, sonst fühlt man sich nicht sicher.

```typescript
export function validateWorkingDirectory(
  workingDirectory: string,
  skipGitRepoCheck: boolean
): void {
  const resolvedWorkingDirectory = path.resolve(workingDirectory);

  if (!existsSync(resolvedWorkingDirectory)) {
    throw new CodexExecutorError(
      'gateway_unavailable',
      'Working directory does not exist.',
      false
    );
  }

  if (!statSync(resolvedWorkingDirectory).isDirectory()) {
    throw new CodexExecutorError(
      'gateway_unavailable',
      'Working directory is not a directory.',
      false
    );
  }

  if (skipGitRepoCheck) {
    return;
  }

  const gitDir = path.join(resolvedWorkingDirectory, '.git');
  if (!existsSync(gitDir)) {
    throw new CodexExecutorError(
      'gateway_unavailable',
      'Working directory is not a git repository.',
      false
    );
  }
}
```

### Laden von Umgebungsvariablen

Codex SDK muss Umgebungsvariablen von der Anmeldeshell laden, um sicherzustellen, dass der AI-Agent auf Systembefehle zugreifen kann:

```typescript
function parseEnvironmentOutput(output: Buffer): Record<string, string> {
  const parsed: Record<string, string> = {};

  for (const entry of output.toString('utf8').split('\0')) {
    if (!entry) continue;

    const separatorIndex = entry.indexOf('=');
    if (separatorIndex <= 0) continue;

    const key = entry.slice(0, separatorIndex);
    const value = entry.slice(separatorIndex + 1);
    if (key.length > 0) {
      parsed[key] = value;
    }
  }

  return parsed;
}

function tryLoadEnvironmentFromShell(shellPath: string): Record<string, string> | null {
  const result = spawnSync(shellPath, ['-ilc', 'env -0'], {
    env: process.env,
    stdio: ['ignore', 'pipe', 'pipe'],
    timeout: 5000,
  });

  if (result.error || result.status !== 0) {
    return null;
  }

  return parseEnvironmentOutput(result.stdout);
}

export function createExecutorEnvironment(
  envOverrides: Record<string, string> = {}
): Record<string, string> {
  // Laden der Umgebungsvariablen der Anmeldeshell
  const consoleEnv = loadConsoleEnvironmentFromShell();

  return {
    ...process.env,
    ...consoleEnv,
    ...envOverrides,
  };
}
```

## Vollständige Verwendungsbeispiele

### Grundlegende Verwendung

Im HagiCode-Projekt initialisieren wir den Codex-Client und führen Aufgaben auf folgende Weise aus:

```typescript
import { Codex } from '@openai/codex-sdk';

async function executeWithCodex(prompt: string, workingDir: string) {
  const client = new Codex({
    apiKey: process.env.CODEX_API_KEY,
    env: { PATH: process.env.PATH },
  });

  const thread = client.startThread({
    workingDirectory: workingDir,
  });

  const { events } = await thread.runStreamed(prompt);

  let result = '';
  for await (const event of events) {
    if (event.type === 'item.updated' && event.item.type === 'agent_message') {
      result = event.item.text;
    }
    if (event.type === 'turn.completed') {
      console.log('Token usage:', event.usage);
    }
  }

  // Versuche, JSON-Ausgabe zu analysieren
  try {
    const parsed = JSON.parse(result);
    return parsed.output;
  } catch {
    return result;
  }
}
```

### Vollständige Implementierung mit Wiederholungsmechanismus

```typescript
export class CodexSdkExecutor {
  private readonly config: CodexRuntimeConfig;
  private readonly client: Codex;
  private threadId: string | null = null;

  async executeStage(input: CodexStageExecutionInput): Promise<CodexStageExecutionResult> {
    const maxAttempts = Math.max(1, this.config.retryCount + 1);
    let attempt = 0;
    let lastError: CodexExecutorError | null = null;

    while (attempt < maxAttempts) {
      attempt += 1;

      try {
        const thread = this.getThread(input.workingDirectory);
        const { output, usage } = await this.runWithStreaming(thread, input);

        return {
          output,
          usage,
          threadId: this.threadId!,
          attempts: attempt,
          latencyMs: Date.now() - startedAt,
        };
      } catch (error) {
        const mappedError = mapError(error);
        lastError = mappedError;

        // Nicht wiederholbarer Fehler oder maximale Wiederholungsversuche erreicht
        if (!mappedError.retryable || attempt >= maxAttempts) {
          throw mappedError;
        }

        // Warte vor dem Wiederholen
        await new Promise(resolve => setTimeout(resolve, 1000 * attempt));
      }
    }

    throw lastError!;
  }
}
```

## Bewährte Praktiken

### 1. Anforderungen an das Arbeitsverzeichnis

- Stellen Sie sicher, dass das Arbeitsverzeichnis ein gültiges Git-Repository ist
- Verwenden Sie die Umgebungsvariable `PROJECT_ROOT`, um es explizit anzugeben
- Bei der Entwicklung und Fehlerbehebung kann `CODEX_SKIP_GIT_REPO_CHECK=true` gesetzt werden, um die Prüfung zu überspringen

### 2. Konfiguration der Umgebungsvariablen

- Übergeben Sie notwendige Umgebungsvariablen über einen Whitelist-Mechanismus
- Verwenden Sie die Anmeldeshell, um die vollständige Umgebung zu laden
- Vermeiden Sie die Übertragung vertraulicher Informationen

### 3. Timeout und Wiederholung

- Legen Sie basierend auf der Aufgabenkomplexität eine angemessene Timeout-Zeit fest
- Implementieren Sie exponentielles Backoff für wiederholbare Fehler
- Protokollieren Sie die Anzahl der Wiederholungsversuche und die Gründe

### 4. Fehlerbehandlung

- Unterscheiden Sie zwischen wiederholbaren und nicht wiederholbaren Fehlern
- Geben Sie klare Fehlerinformationen und Empfehlungen
- Einheitliche Fehlercodes erleichtern die Verarbeitung auf höherer Ebene

### 5. Streaming-Ausgabe

- Implementieren Sie inkrementelle Ausgabe-Callbacks, um das Benutzererlebnis zu verbessern
- Verarbeiten Sie inkrementelle Aktualisierungen von Nachrichten korrekt
- Zeichnen Sie die Token-Verwendung für die Kostenanalyse auf

In der tatsächlichen Produktionsumgebung des HagiCode-Projekts haben wir die Wirksamkeit der oben genannten bewährten Praktiken überprüft.
