## Von 120 auf 45 Sekunden: Praktische Optimierung der Vite-Build-Leistung mit Worker Threads

> Bei großen Frontend-Projekten können Produktions-Builds schmerzhaft langsam sein. Dieser Artikel zeigt, wie wir mit Node.js Worker Threads die Obfuscation-Phase in einem Vite-Build von 120 auf 45 Sekunden reduziert haben, zusammen mit den Implementierungsdetails und den gesammelten Erfahrungen im HagiCode-Projekt.

## Hintergrund

In unserer Frontend-Engineering-Praxis traten Probleme der Build-Effizienz mit zunehmender Projektgröße immer deutlicher zutage. Insbesondere beim Produktions-Build-Prozess führen wir in der Regel JavaScript-Obfuscation-Tools wie `javascript-obfuscator` ein, um die Quellcode-Logik zu schützen. Dieser Schritt ist zwar notwendig, aber rechenintensiv und stark CPU-gebunden.

Während der frühen Entwicklungsphase von **HagiCode** stießen wir auf einen sehr kniffligen Leistungsengpass: Die Produktions-Build-Zeiten verschlechterten sich rapide mit dem Anwachsen der Codebasis.

**Die konkreten Schmerzpunkte waren**:
- Obfuscation-Aufgaben liefen seriell auf einem einzigen Thread, wobei ein CPU-Kern volllastete, während die anderen ungenutzt blieben
- Die Build-Zeit stieg von ursprünglich 30 Sekunden auf 110-120 Sekunden
- Die Build-Verifizierungsschleife nach Codeänderungen wurde extrem lang, was die Entwicklungsiteration ernsthaft verlangsamte
- In der CI/CD-Pipeline wurde der Build-Schritt zum zeitaufwändigsten Teil

**Warum brauchte HagiCode das?**
HagiCode ist ein KI-gestützter Code-Assistent, dessen Frontend-Architektur komplexe Geschäftslogik und KI-Interaktionsmodule umfasst. Um die Sicherheit unseres Kerncodes zu gewährleisten, haben wir bei Produktions-Releases eine hochintensive Obfuscation erzwungen. Angesichts von Build-Wartezeiten von nahezu zwei Minuten entschieden wir uns für eine umfassende Leistungsoptimierung des Build-Systems.

## Analyse: Den Durchbruch beim Leistungsengpass finden

Bevor wir das Leistungsproblem lösten, mussten wir zunächst unsere Gedanken ordnen und die beste technische Lösung identifizieren.

### Kernentscheidung: Warum Worker Threads?

In Node.js gibt es drei Hauptwege zur Realisierung paralleler Berechnungen:
1. **child_process**: Unabhängige Kindprozesse erstellen
2. **Web Workers**: Hauptsächlich für den Browser
3. **worker_threads**: Native Multithreading-Unterstützung in Node.js

Nach dem Vergleich der Optionen entschied sich HagiCode letztendlich für **Worker Threads** aus folgenden Gründen:
- **Null-Serialisierungs-Overhead**: Worker Threads laufen im selben Prozess und können Speicher über `SharedArrayBuffer` oder Transfer-Ownership teilen, wodurch die hohen Serialisierungskosten der Inter-Prozess-Kommunikation vermieden werden.
- **Native Unterstützung**: In Node.js 12+ integriert, ohne zusätzliche schwerabhängige Pakete.
- **Einheitlicher Kontext**: Debugging und Protokollierung sind bequemer als bei Kindprozessen.

### Aufgabengranularität: Wie sollen Obfuscation-Aufgaben aufgeteilt werden?

Es ist schwierig, die Obfuscation einer riesigen JS-Bundle-Datei zu parallelisieren, da der Code Abhängigkeiten aufweist. Das Vite-Build-Ergebnis besteht jedoch aus mehreren **Chunks**. Das gibt uns eine natürliche Parallelitätsgrenze:

- **Unabhängigkeit**: Nach dem Vite-Packaging sind die Abhängigkeiten zwischen verschiedenen Chunks bereits entkoppelt und können sicher parallel verarbeitet werden.
- **Angemessene Granularität**: Projekte haben normalerweise 10-30 Chunks — eine hervorragende Skalierung für paralleles Scheduling.
- **Einfache Integration**: Der `generateBundle`-Hook in Vite-Plugins ermöglicht es uns, diese Chunks vor der Dateiausgabe abzufangen und zu verarbeiten.

### Architekturentwurf

Wir entwarfen ein paralleles Verarbeitungssystem mit vier Kernkomponenten:

1. **Task Splitter**: Iteriert über das Vite-Bundle-Objekt, filtert Dateien, die keine Obfuscation benötigen (wie Vendor-Chunks), und generiert eine Aufgabenwarteschlange.
2. **Worker Pool Manager**: Verwaltet den Worker-Lebenszyklus und übernimmt Aufgabenverteilung, -rückgewinnung und Fehlerwiederholung.
3. **Progress Reporter**: Gibt den Build-Fortschritt in Echtzeit aus, um die Warteangst zu reduzieren.
4. **ObfuscationWorker**: Der Worker-Thread, der die eigentliche Obfuscation-Logik ausführt.

## Lösung: Praktische Implementierung

Basierend auf der obigen Analyse begannen wir mit der Implementierung dieses parallelen Obfuscation-Systems.

### 1. Das Vite-Plugin konfigurieren

Zuerst integrierten wir das parallele Obfuscation-Plugin in `vite.config.ts`. Die Konfiguration ist unkompliziert — Sie müssen nur die Anzahl der Worker und die Obfuscation-Regeln angeben.

```typescript
import { defineConfig } from 'vite'
import { parallelJavascriptObfuscator } from './buildTools/plugin'

export default defineConfig(({ mode }) => {
  const isProduction = mode === 'production'

  return {
    build: {
      rollupOptions: {
        ...(isProduction
          ? {
              plugins: [
                parallelJavascriptObfuscator({
                  enabled: true,
                  // Automatisch an CPU-Kernanzahl anpassen; einen Kern für den Hauptthread freihalten
                  workerCount: 4,
                  retryAttempts: 3,
                  fallbackToMainThread: true, // Bei Fehler automatisch auf Einzelthread-Modus zurückgreifen
                  // Vendor-Chunks herausfiltern; Drittanbieterbibliotheken benötigen meist keine Obfuscation
                  isVendorChunk: (fileName: string) => fileName.includes('vendor-'),
                  obfuscationConfig: {
                    compact: true,
                    controlFlowFlattening: true,
                    deadCodeInjection: true,
                    disableConsoleOutput: true,
                    // ... weitere Obfuscation-Optionen
                  },
                }),
              ],
            }
          : {}),
      },
    },
  }
})
```

### 2. Die Worker-Logik implementieren

Ein Worker ist die Einheit, die Aufgaben ausführt. Wir müssen die Eingabe- und Ausgabedatenstrukturen klar definieren.

**Hinweis**: Obwohl der Code hier einfach aussieht, gibt es einige Fallstricke zu beachten, wie die Prüfung, ob `parentPort` null ist, und die korrekte Fehlerbehandlung. In HagiCodes Implementierung stellten wir fest, dass bestimmte spezielle ES6-Syntaxmuster den Obfuscator zum Absturz bringen können, weshalb wir einen `try-catch`-Schutz hinzufügten.

```typescript
import { parentPort } from 'worker_threads'
import javascriptObfuscator from 'javascript-obfuscator'

export interface ObfuscationTask {
  chunkId: string
  code: string
  config: any
}

export interface ObfuscationResult {
  chunkId: string
  obfuscatedCode: string
  error?: string
}

// Auf Aufgaben vom Hauptthread lauschen
if (parentPort) {
  parentPort.on('message', async (task: ObfuscationTask) => {
    try {
      // Obfuscation durchführen
      const obfuscated = javascriptObfuscator.obfuscate(task.code, task.config)
      const result: ObfuscationResult = {
        chunkId: task.chunkId,
        obfuscatedCode: obfuscated.getObfuscatedCode(),
      }
      // Ergebnis an den Hauptthread zurücksenden
      parentPort?.postMessage(result)
    } catch (error) {
      // Ausnahmen behandeln, damit ein Worker-Absturz nicht den gesamten Build blockiert
      const result: ObfuscationResult = {
        chunkId: task.chunkId,
        obfuscatedCode: '',
        error: error instanceof Error ? error.message : 'Unknown error',
      }
      parentPort?.postMessage(result)
    }
  })
}
```

### 3. Worker Pool Manager

Das ist der Kern der gesamten Lösung. Wir müssen einen Worker Pool fester Größe verwalten und Aufgaben mit einer **FIFO-Strategie (First In, First Out)** planen.

```typescript
import { Worker } from 'worker_threads'
import os from 'os'

export class WorkerPool {
  private workers: Worker[] = []
  private taskQueue: Array<{
    task: ObfuscationTask
    resolve: (result: ObfuscationResult) => void
    reject: (error: Error) => void
  }> = []

  constructor(options: WorkerPoolOptions = {}) {
    // Standardmäßig Kernanzahl - 1, damit der Hauptthread etwas Luft hat
    const workerCount = options.workerCount ?? Math.max(1, (os.cpus().length || 4) - 1)

    for (let i = 0; i < workerCount; i++) {
      this.createWorker()
    }
  }

  private createWorker() {
    const worker = new Worker('./worker.ts')

    worker.on('message', (result) => {
      // Nach Abschluss einer Aufgabe die nächste aus der Warteschlange nehmen
      const nextTask = this.taskQueue.shift()
      if (nextTask) {
        this.dispatchTask(worker, nextTask)
      } else {
        // Wenn keine Aufgaben anstehen, Worker als leerlaufend markieren
        this.activeWorkers.delete(worker)
      }
    })

    this.workers.push(worker)
  }

  // Aufgabe an den Pool übergeben
  public runTask(task: ObfuscationTask): Promise<ObfuscationResult> {
    return new Promise((resolve, reject) => {
      const job = { task, resolve, reject }
      const idleWorker = this.workers.find(w => !this.activeWorkers.has(w))

      if (idleWorker) {
        this.dispatchTask(idleWorker, job)
      } else {
        this.taskQueue.push(job)
      }
    })
  }

  private dispatchTask(worker: Worker, job: any) {
    this.activeWorkers.set(worker, job.task)
    worker.postMessage(job.task)
  }
}
```

### 4. Fortschrittsbericht

Warten ist schmerzhaft, besonders wenn man nicht weiß, wie lange es noch dauert. Daher haben wir einen einfachen Fortschrittsreporter hinzugefügt, der in Echtzeit über den aktuellen Stand informiert.

```typescript
export class ProgressReporter {
  private completed = 0
  private readonly total: number
  private readonly startTime: number

  constructor(total: number) {
    this.total = total
    this.startTime = Date.now()
  }

  increment(): void {
    this.completed++
    this.report()
  }

  private report(): void {
    const now = Date.now()
    const elapsed = now - this.startTime
    const percentage = (this.completed / this.total) * 100

    // Einfache ETA-Schätzung
    const avgTimePerChunk = elapsed / this.completed
    const remaining = (this.total - this.completed) * avgTimePerChunk

    console.log(
      `[Parallel Obfuscation] ${this.completed}/${this.total} Chunks abgeschlossen (${percentage.toFixed(1)}%) | ETA: ${(remaining / 1000).toFixed(1)}s`
    )
  }
}
```

## Praxis: Ergebnisse und gesammelte Erfahrungen

Nach der Bereitstellung dieser Lösung verbesserte sich die Build-Leistung des HagiCode-Projekts sofort spürbar.

### Leistungs-Benchmark-Daten

Wir testeten in folgender Umgebung:
- CPU: Intel Core i7-12700K (12 Kerne / 20 Threads)
- RAM: 32GB DDR4
- Node.js: v18.17.0
- OS: Ubuntu 22.04

**Ergebnisvergleich**:
- **Einzelthread (vor der Optimierung)**: 118 Sekunden
- **4 Worker**: 55 Sekunden (**53 %** Verbesserung)
- **8 Worker**: 48 Sekunden (**60 %** Verbesserung)
- **12 Worker**: 45 Sekunden (**62 %** Verbesserung)

Wie man sieht, sind die Gewinne nicht linear. Sobald die Worker-Anzahl 8 überstieg, wurde die Verbesserung geringer. Dies war hauptsächlich durch die Gleichmäßigkeit der Aufgabenverteilung und Speicherbandbreitenengpässe begrenzt.

### Häufige Probleme und Lösungen

In der Praxis mit HagiCode stießen wir ebenfalls auf einige Fallstricke, die wir hier teilen möchten:

**F1: Die Build-Zeit hat sich nicht nennenswert verringert und wurde sogar langsamer?**
- **Ursache**: Die Erstellung von Workers verursacht eigenen Overhead, oder es wurden zu viele Worker konfiguriert, was zu häufigem Kontextwechsel führt.
- **Lösung**: Wir empfehlen, die Worker-Anzahl auf `CPU-Kernanzahl - 1` zu setzen. Prüfen Sie außerdem, ob ein einzelner Chunk besonders groß ist (z. B. > 5 MB). Eine solche „Monster"-Datei wird zum Engpass — hier sollte die Code-Splitting-Strategie optimiert werden.

**F2: Worker stürzen gelegentlich ab und verursachen Build-Fehler?**
- **Ursache**: Bestimmte spezielle Code-Syntaxmuster können interne Fehler im Obfuscator auslösen.
- **Lösung**: Wir implementierten einen **automatischen Degradationsmechanismus**. Wenn ein Worker den Fehler-Schwellenwert erreicht, fällt das Plugin automatisch auf den Einzelthread-Modus zurück, um sicherzustellen, dass der Build nicht stoppt. Gleichzeitig wird der Dateiname des Fehlers protokolliert, um später gezielt beheben zu können.

**F3: Speicherverbrauch zu hoch (OOM)?**
- **Ursache**: Jeder Worker benötigt eigenen Speicherplatz zum Laden des Obfuscators und Parsen des AST.
- **Lösung**:
  - Worker-Anzahl reduzieren.
  - Node.js-Speicherlimit erhöhen: `NODE_OPTIONS="--max-old-space-size=4096" npm run build`.
  - Sicherstellen, dass Worker keine unnötigen Referenzen auf große Objekte halten.

## Zusammenfassung

Durch die Einführung von Node.js Worker Threads haben wir die Produktions-Build-Zeit des HagiCode-Projekts erfolgreich von 120 Sekunden auf etwa 45 Sekunden reduziert, was die Entwicklungserfahrung und die CI/CD-Effizienz erheblich verbesserte.

Der Kern dieser Lösung liegt in:
1. **Aufgaben sinnvoll aufteilen**: Vite-Chunks als parallele Einheiten nutzen.
2. **Ressourcen kontrollieren**: Einen Worker Pool verwenden, um Ressourcenerschöpfung zu vermeiden.
3. **Fehlertoleranz entwerfen**: Ein automatischer Degradationsmechanismus stellt die Build-Stabilität sicher.
