Steam 商品、DLC 和 Bundle 的图片展示不应该在前端硬编码资源路径。图片由 Index 发布到 Steam payload，消费端只读取描述符，再按自己的布局选择合适的图。

当前这个契约覆盖三类 key：

- `hagicode`：Steam 主应用。
- `turbo-engine`：Turbo Engine DLC。
- `hagicode-plus`：Steam 官方组合包。

## Payload 位置

默认入口是：

```text
https://index.hagicode.com/steam/index.json
```

`repos/web` 支持通过 `VITE_STEAM_INDEX_URL` 覆盖这个入口，便于测试、预发或临时切换 Index 源。

## 顶层结构

Steam payload 至少需要包含三个顶层字段：

```json
{
  "applications": [],
  "bundles": [],
  "images": {}
}
```

消费端会在加载边界验证这三个字段：

- `applications` 必须是数组，用于主应用和 DLC 记录。
- `bundles` 必须是数组，用于组合包记录。
- `images` 必须是对象，用于按 product key 补充或共享图片描述符。

## 图片描述符

每个图片描述符使用同一个结构：

```json
{
  "src": "https://cdn.example.invalid/steam/hagicode-wide.jpg",
  "variant": "wide-capsule",
  "alt": "Hagicode Steam wide capsule artwork",
  "width": 1920,
  "height": 1080
}
```

字段含义：

- `src`：必填。图片 URL 或已发布的静态资源路径。
- `variant`：必填。告诉消费端这张图适合什么布局。
- `alt`：可选，但推荐填写。消费端会优先使用它作为图片替代文本。
- `width` / `height`：可选。用于未来更精确的尺寸选择或预留空间。

支持的常用 `variant`：

| Variant | 用途 | 消费端布局 |
| --- | --- | --- |
| `store-capsule` | 商店紧凑卡片图 | 小卡片或缩略图 |
| `wide-capsule` | 横向宣传图 | 默认主图 |
| `library-capsule` | 纵向库封面 | 缩略图或备选主图 |
| `hero` | 大幅头图 | Hero 区或大横幅 |

未知 `variant` 不应该让页面崩溃。当前消费端可以忽略未知变体，也可以用通用比例展示。

## Application 与 DLC 记录

应用和 DLC 记录位于 `applications`：

```json
{
  "key": "turbo-engine",
  "displayName": "Hagicode: Turbo Engine DLC",
  "kind": "dlc",
  "parentKey": "hagicode",
  "storeAppId": 4635480,
  "storeUrl": "https://store.steampowered.com/app/4635480/Hagicode__Turbo_Engine/",
  "images": []
}
```

要求：

- `key` 应稳定、小写，并作为消费端查找主键。
- DLC 应保留 `parentKey`，让 UI 可以展示父应用关系。
- `images` 是向后兼容字段，旧客户端可以直接忽略。
- 如果记录内没有 `images`，消费端可以从顶层 `images[key]` 读取补充描述符。

## Bundle 记录

Bundle 记录位于 `bundles`：

```json
{
  "key": "hagicode-plus",
  "displayName": "Hagicode Plus",
  "storeBundleId": 73989,
  "storeUrl": "https://store.steampowered.com/bundle/73989/Hagicode_Plus/",
  "includedApplicationKeys": ["hagicode", "turbo-engine"],
  "images": []
}
```

要求：

- Bundle 图片只描述 Bundle 自己，不应该替代 `includedApplicationKeys`。
- 消费端优先展示 Bundle 自己的图片。
- 如果 Bundle 没有可用图片，仍然必须保留名称、Steam 链接和组合内容信息。

## 无障碍要求

每张实际渲染的图片都必须有可读替代文本：

- 如果描述符提供 `alt`，直接使用 `alt`。
- 如果没有 `alt`，消费端生成确定性文本，例如 `Hagicode wide capsule image`。
- 图片加载失败后，不要把破图留在页面上；应显示品牌占位、商品名称和 Steam CTA。

## 维护流程

添加或修改 Steam 图片时，按这个顺序处理：

1. 在 Index 发布源里上传或登记图片资源。
2. 更新 `/steam/index.json` 对应记录的 `images`，或更新顶层 `images[key]`。
3. 确认 `src`、`variant` 和可选 `alt` 均符合本页契约。
4. 前端 `repos/web` 会在浏览器页面生命周期内缓存已解析 payload；用户刷新页面或 HTTP 缓存重新验证后会拿到新数据。
5. 官网 `repos/site` 是静态 Astro 站点，本次实现使用静态描述符快照和占位回退；当 Index 图片正式发布后，同步该快照即可让静态构建稳定输出图片。
6. 文档 `repos/docs` 只记录契约和维护方法，不直接参与图片发布。

## 实现边界

本次变更的实现边界如下：

- `repos/index` 是参考来源，用来定义和发布 Steam payload；本次变更不编辑它。
- `repos/web` 是主要运行时消费端，负责动态加载、缓存、归一化和展示图片。
- `repos/site` 是公开营销站点，采用静态构建安全的描述符快照和可见占位回退。
- `repos/docs` 负责记录字段契约、无障碍要求、兼容性和维护流程。

这个边界可以避免旧客户端被 `images` 字段破坏，也让新消费端逐步接入 Steam 视觉资源。
