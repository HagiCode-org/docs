## Construindo Integração com GitHub Issues do Zero: A Prática de Conexão Direta do Frontend da HagiCode

> Este artigo documenta o processo completo de integração do GitHub Issues na plataforma HagiCode. Exploraremos como, por meio da arquitetura "conexão direta do frontend + backend mínimo", podemos manter o backend leve enquanto implementamos autenticação OAuth segura e sincronização eficiente de Issues.

## Contexto: Por que integrar com o GitHub?

Como uma plataforma de desenvolvimento assistida por IA, o valor central do HagiCode está em conectar ideias e implementações. Mas, durante o uso prático, descobrimos que, após os usuários completarem uma Proposal (proposta) no HagiCode, frequentemente precisam copiar manualmente o conteúdo para o GitHub Issues para acompanhamento do projeto.

Isso traz várias dores óbvias:

1.  **Fluxo de trabalho fragmentado**: Os usuários precisam alternar entre dois sistemas, o que não é apenas uma experiência não fluida, mas também propenso a perda de informações críticas durante o processo de copiar e colar.
2.  **Colaboração inconveniente**: Outros membros da equipe estão acostumados a visualizar tarefas no GitHub e não podem ver diretamente o progresso das propostas no HagiCode.
3.  **Trabalho repetitivo**: Sempre que uma proposta é atualizada, é necessário atualizar manualmente a Issue correspondente no GitHub, aumentando custos de manutenção desnecessários.

Para resolver esse problema, decidimos introduzir o recurso **GitHub Issues Integration**, conectando as sessões do HagiCode com os repositórios do GitHub,实现 "sincronização com um clique".

## Seleção Tecnológica: Conexão Direta do Frontend vs Proxy do Backend

Ao projetar a solução de integração, tínhamos dois caminhos à nossa frente: o tradicional "modo de proxy do backend" e o mais agressivo "modo de conexão direta do frontend".

### Comparação de Soluções

No tradicional **modo de proxy do backend**, todas as requisições do frontend precisam passar primeiro pelo nosso backend, e então o backend chama a API do GitHub. Embora a lógica seja centralizada, isso traz uma carga significativa para o backend:

1.  **Backend inchado**: É necessário escrever um encapsulamento de cliente de API do GitHub dedicado e também lidar com a máquina de estados complexa do OAuth.
2.  **Risco de Token**: O Token do GitHub do usuário deve ser armazenado no banco de dados do backend. Embora possa ser criptografado, isso aumenta a superfície de risco de segurança.
3.  **Custo de desenvolvimento**: São necessárias migrações de banco de dados para armazenar o Token e também manter um serviço de sincronização adicional.

O **modo de conexão direta do frontend** é muito mais leve. Nesta solução, usamos o backend apenas para lidar com a etapa mais sensível de "troca de chaves" (OAuth callback). Após obter o Token, ele é armazenado diretamente no localStorage do navegador. Operações subsequentes como criar Issues e atualizar comentários são feitas diretamente pelo frontend enviando requisições HTTP para o GitHub.

| Dimensão de Comparação | Modo de Proxy do Backend | Modo de Conexão Direta do Frontend |
| :--- | :--- | :--- |
| **Complexidade do Backend** | Precisa de serviço OAuth completo e cliente da API do GitHub | Apenas um endpoint de callback OAuth |
| **Gerenciamento de Token** | Precisa ser criptografado e armazenado no banco de dados, com risco de vazamento | Armazenado no navegador, visível apenas pelo usuário |
| **Custo de Implementação** | Precisa de migração de banco de dados, desenvolvimento de múltiplos serviços | Principalmente trabalho do frontend |
| **Experiência do Usuário** | Lógica unificada, mas latência do servidor pode ser ligeiramente maior | Resposta extremamente rápida, interação direta com o GitHub |

Considerando que queríamos integração rápida e mudanças mínimas no backend, **finalmente adotamos o "modo de conexão direta do frontend"**. É como emitir um "cartão de acesso temporário" para o navegador — após obter o cartão, o navegador pode ir ao GitHub por conta própria, sem precisar pedir aprovação do administrador do backend a cada vez.

## Design Central: Fluxo de Dados e Segurança

Após determinar a arquitetura, precisamos projetar o fluxo de dados específico. O núcleo de todo o processo de sincronização reside em como obter o Token com segurança e usá-lo de forma eficiente.

### Diagrama de Arquitetura Geral

Todo o sistema pode ser abstraído em três papéis: navegador (frontend), backend HagiCode e GitHub.

```text
+--------------+        +--------------+        +--------------+
|  Frontend    |        |    Backend   |        |    GitHub    |
|   React      |        |   ASP.NET    |        |    REST API  |
|              |        |              |        |              |
|  +--------+  |        |              |        |              |
|  |  OAuth |--+--------> /callback    |        |              |
|  |  Fluxo |  |        |              |        |              |
|  +--------+  |        |              |        |              |
|              |        |              |        |              |
|  +--------+  |        |  +--------+  |        |  +--------+  |
|  |GitHub  |  +------------>Session |  +----------> Issues |  |
|  |API     |  |        |  |Metadata|  |        |  |        |  |
|  |Direto  |  |        |  +--------+  |        |  +--------+  |
|  +--------+  |        |              |        |              |
+--------------+        +--------------+        +--------------+
```

**O ponto chave é**: apenas um pequeno passo do OAuth (obter code para trocar por token) precisa passar pelo backend; o trabalho pesado subsequente (criar Issue) é feito diretamente pelo frontend com o GitHub.

### Detalhamento do Fluxo de Dados de Sincronização

Quando o usuário clica no botão "Sync to GitHub" na interface do HagiCode, uma série de ações complexas ocorre:

```text
Usuário clica em "Sync to GitHub"
         │
         ▼
1. Frontend verifica o localStorage para obter o GitHub Token
         │
         ▼
2. Formata o conteúdo da Issue (converte Proposal em Markdown)
         │
         ▼
3. Frontend chama diretamente a API do GitHub para criar/atualizar Issue
         │
         ▼
4. Chama a API do backend HagiCode para atualizar Session.metadata (armazena URL da Issue etc.)
         │
         ▼
5. Backend transmite evento SessionUpdated via SignalR
         │
         ▼
6. Frontend recebe o evento, atualiza a UI para exibir status "sincronizado"
```

### Design de Segurança

A segurança é sempre a parte mais importante da integração com serviços de terceiros. Fazemos as seguintes considerações:

1.  **Prevenção de ataques CSRF**: Durante o redirecionamento OAuth, geramos um parâmetro `state` aleatório e o armazenamos no sessionStorage. No callback, verificamos rigorosamente o state para evitar que requisições sejam forjadas.
2.  **Isolamento de armazenamento de Token**: O Token é armazenado apenas no `localStorage` do navegador, utilizando a política de mesma origem (Same-Origin Policy), apenas os scripts do HagiCode podem lê-lo, evitando que vazamentos do banco de dados do servidor afetem os usuários.
3.  **Limites de erro**: Para erros comuns da API do GitHub (como 401 Token expirado, 422 falha de validação, 429 limite de taxa), projetamos lógica dedicada de tratamento de erros, fornecendo avisos amigáveis aos usuários.

### 1. Mudanças Mínimas no Backend

O backend só precisa fazer duas coisas: armazenar informações de sincronização e processar o callback OAuth.

### 2. Fluxo OAuth do Frontend

Este é o ponto de entrada da conexão. Usamos o fluxo padrão de Authorization Code.

```typescript
// src/HagiCode.Client/src/services/githubOAuth.ts

// Gerar URL de autorização e redirecionar
export async function generateAuthUrl(): Promise<string> {
  const state = generateRandomString(); // Gera string aleatória para prevenir CSRF
  sessionStorage.setItem('hagicode_github_state', state);
  
  const params = new URLSearchParams({
    client_id: clientId,
    redirect_uri: window.location.origin + '/settings?tab=github&oauth=callback',
    scope: ['repo', 'public_repo'].join(' '),
    state: state,
  });
  
  return `https://github.com/login/oauth/authorize?${params.toString()}`;
}

// Na página de callback, processa troca de Code por Token
export async function exchangeCodeForToken(code: string, state: string): Promise<GitHubToken> {
  // 1. Verificar State para prevenir CSRF
  const savedState = sessionStorage.getItem('hagicode_github_state');
  if (state !== savedState) throw new Error('Invalid state parameter');

  // 2. Chamar API do backend para troca de Token
  // Nota: aqui deve passar pelo backend, pois precisa do ClientSecret, não pode ser exposto no frontend
  const response = await fetch('/api/GitHubOAuth/callback', {
    method: 'POST',
    headers: { 'Content-Type': 'application/json' },
    body: JSON.stringify({ code, state, redirectUri: window.location.origin + '/settings?tab=github&oauth=callback' }),
  });

  if (!response.ok) throw new Error('Failed to exchange token');
  
  const token = await response.json();
  
  // 3. Armazenar no LocalStorage
  saveToken(token);
  return token;
}
```

### 3. Encapsulamento do Cliente da API do GitHub

Com o Token em mãos, precisamos de uma ferramenta poderosa para chamar a API do GitHub.

```typescript
// src/HagiCode.Client/src/services/githubApiClient.ts

const GITHUB_API_BASE = 'https://api.github.com';

// Encapsulamento de requisição central
async function githubApi<T>(endpoint: string, options: RequestInit = {}): Promise<T> {
  const token = localStorage.getItem('gh_token');
  if (!token) throw new Error('Not connected to GitHub');
  
  const response = await fetch(`${GITHUB_API_BASE}${endpoint}`, {
    ...options,
    headers: {
      ...options.headers,
      Authorization: `Bearer ${token}`,
      Accept: 'application/vnd.github.v3+json', // Especifica versão da API
    },
  });
  
  // Lógica de tratamento de erros
  if (!response.ok) {
    if (response.status === 401) throw new Error('GitHub Token expirado, por favor reconecte');
    if (response.status === 403) throw new Error('Sem permissão para acessar o repositório ou limite de taxa excedido');
    if (response.status === 422) throw new Error('Validação de Issue falhou, título possivelmente duplicado');
    throw new Error(`GitHub API Error: ${response.statusText}`);
  }
  
  return response.json();
}

// Criar Issue
export async function createIssue(owner: string, repo: string, data: { title: string, body: string, labels: string[] }) {
  return githubApi(`/repos/${owner}/${repo}/issues`, {
    method: 'POST',
    body: JSON.stringify(data),
  });
}
```

### 4. Formatação de Conteúdo e Sincronização

O último passo é converter os dados da Session do HagiCode para o formato de Issue do GitHub. É como um trabalho de "tradução".

```typescript
// Converter objeto Session em string Markdown
function formatIssueForSession(session: Session): string {
  let content = `# ${session.title}\n\n`;
  content += `**> HagiCode Session:** #${session.code}\n`;
  content += `**> Status:** ${session.status}\n\n`;
  content += `## Description\n\n${session.description || 'No description provided.'}\n\n`;
  
  // Se for tipo Proposal, adiciona campos extras
  if (session.type === 'proposal') {
    content += `## Chief Complaint\n\n${session.chiefComplaint || ''}\n\n`;
    // Adiciona um deep link, conveniente para voltar ao HagiCode pelo GitHub
    content += `---\n\n**[View in HagiCode](hagicode://sessions/${session.id})**\n`;
  }
  
  return content;
}

// Lógica principal ao clicar no botão de sincronização
const handleSync = async (session: Session) => {
  try {
    const repoInfo = parseRepositoryFromUrl(session.repoUrl); // Analisa URL do repositório
    if (!repoInfo) throw new Error('Invalid repository URL');

    toast.loading('Sincronizando com o GitHub...');
    
    // 1. Formatar conteúdo
    const issueBody = formatIssueForSession(session);
    
    // 2. Chamar API
    const issue = await githubApiClient.createIssue(repoInfo.owner, repoInfo.repo, {
      title: `[HagiCode] ${session.title}`,
      body: issueBody,
      labels: ['hagicode', 'proposal', `status:${session.status}`],
    });
    
    // 3. Atualizar Session Metadata (salvar link da Issue)
    await SessionsService.patchApiSessionsSessionId(session.id, {
      metadata: {
        githubIssue: {
          owner: repoInfo.owner,
          repo: repoInfo.repo,
          issueNumber: issue.number,
          issueUrl: issue.html_url,
          syncedAt: new Date().toISOString(),
        }
      }
    });

    toast.success('Sincronização bem-sucedida!');
  } catch (err) {
    console.error(err);
    toast.error('Falha na sincronização, verifique o Token ou rede');
  }
};
```

## Conclusão e Perspectivas

Por meio desta solução de "conexão direta do frontend",实现 integração perfeita com o GitHub Issues com a quantidade mínima de código backend.

### Conquistas
1.  **Alta eficiência de desenvolvimento**: Mudanças mínimas no backend, principalmente adicionando um campo ao banco de dados e uma interface simples de callback OAuth, a maior parte da lógica é concluída no frontend.
2.  **Boa segurança**: O Token não passa pelo banco de dados do servidor, reduzindo o risco de vazamento.
3.  **Excelente experiência do usuário**: Requisições iniciadas diretamente do frontend, resposta rápida, sem necessidade de passagem pelo backend.

### Pontos de Atenção
Na implantação real, há alguns armadilhas a serem observadas:
- **Configuração do OAuth App**: Lembre-se de preencher o `Authorization callback URL` correto nas configurações do GitHub OAuth App (geralmente `http://localhost:3000/settings?tab=github&oauth=callback`).
- **Limite de taxa**: A API do GitHub tem restrições mais rigorosas para requisições não autenticadas, mas com o Token geralmente é suficiente (5000 vezes/hora).
- **Análise de URL**: As URLs de Repo inseridas pelos usuário são variadas, lembre-se de corresponder regex para sufixo `.git`, formato SSH, etc.

### Melhorias Futuras
Atualmente, a funcionalidade ainda é sincronização unidirecional (HagiCode -> GitHub). No futuro, planejamos实现 sincronização bidirecional por meio dos Webhooks do GitHub, por exemplo, ao fechar uma Issue no GitHub, o status da sessão no HagiCode também pode ser atualizado automaticamente. Isso requer expormos um endpoint de recebimento de Webhook no backend, que é o próximo trabalho interessante a ser feito.
