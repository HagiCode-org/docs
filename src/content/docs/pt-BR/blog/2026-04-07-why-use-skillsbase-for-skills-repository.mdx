## Por que usar o Skillsbase para manter seu próprio repositório de Skills

> É até engraçado, a era da programação com IA chegou, temos cada vez mais Agent Skills em mãos, mas com isso vem cada vez mais problemas. Este artigo é sobre como usamos o skillsbase para resolver esses problemas.

## Contexto

Na era da programação com IA, os desenvolvedores precisam manter cada vez mais Agent Skills — conjuntos de instruções reutilizáveis para estender as capacidades de assistentes de codificação como Claude Code, OpenCode, Cursor, etc. No entanto, com o crescimento do número de skills, um problema real gradualmente emerge:

Na verdade, não é um grande problema, é só que quando há muitas coisas, fica difícil gerenciar.

### Skills armazenadas de forma dispersa, alto custo de gerenciamento

- Skills locais espalhadas em vários locais: `~/.agents/skills/`, `~/.claude/skills/`, `~/.codex/skills/.system/`, etc.
- Diferentes locais podem ter conflitos de nomenclatura (por exemplo, `skill-creator` existe simultaneamente no diretório do usuário e no diretório do sistema)
- Falta de um ponto de gerenciamento unificado, backup e migração difíceis

Isso é bem irritante. Às vezes você nem sabe onde uma determinada skill está, como se tivesse perdido algo, é difícil de encontrar.

### Falta de processo de manutenção padronizado

- Copiar skills manualmente é propenso a erros, difícil rastrear a origem
- Não há mecanismo de validação unificado, não é possível garantir a integridade do repositório de skills
- Ao colaborar em equipe, é difícil sincronizar e compartilhar o conjunto de skills

Operações manuais, bem, sempre são propensas a erros. Afinal, a memória humana é limitada, quem se lembra de onde veio tanta coisa?

### Incapacidade de atender aos requisitos de reprodutibilidade

- Ao trocar de máquina de desenvolvimento, é necessário reconfigurar todas as skills
- Em ambientes de CI/CD, não é possível validar e sincronizar automaticamente o repositório de skills

Trocar de computador significa começar tudo de novo, essa sensação, como dizer, é tão problemática quanto se mudar de casa. Toda vez é preciso se readaptar ao novo ambiente, reconfigurar tudo.

Para resolver essas dores, tentamos várias soluções: desde cópia manual até automação com scripts, desde gerenciar diretórios diretamente até instalação global e depois reciclagem. Cada solução tem seus defeitos, não consegue garantir consistência, ou polui o ambiente, ou é difícil de usar em CI.

Na verdade, tomamos muitos atalhos errados.

Finalmente, encontramos uma solução mais elegante — o skillsbase. A ideia central desta solução é: primeiro instalar localmente para validar, depois converter a estrutura e escrever no repositório, finalmente desinstalar os arquivos temporários. Isso garante que o conteúdo do repositório seja consistente com o resultado real da instalação, sem poluir o ambiente global.

Parece simples, mas pisamos em muitas armadilhas para descobrir isso.

O HagiCode é um projeto de assistente de código IA. Durante o desenvolvimento, precisamos manter um grande número de Agent Skills para estender várias capacidades de codificação. Na verdade, essa coisa não surgiu do nada, fomos forçados a isso. Com muitas skills, naturalmente é preciso gerenciar. Ao encontrar problemas no gerenciamento, naturalmente é preciso resolvê-los. Passo a passo, chegamos até aqui.

## Análise

### Desafios técnicos

Para estabelecer um repositório de skills mantível, é preciso resolver os seguintes problemas principais:

1. **Conflitos de namespace unificados**: quando várias fontes têm skills com o mesmo nome, como evitar sobrescrita?
2. **Rastreamento de origem**: como registrar a origem de cada skill para atualizações e auditorias futuras?
3. **Sincronização e validação**: como garantir que o conteúdo do repositório seja consistente com o resultado real da instalação?
4. **Integração automatizada**: como integrar com processos de CI/CD para实现 sincronização e validação automáticas?

Esses problemas parecem simples, mas cada um é suficiente para dar dor de cabeça. Mas falando nisso, o que não é difícil na vida?

### Trade-offs de design

**Solução 1: copiar diretório diretamente**

Vantagens: implementação simples
Desvantagens: não é possível garantir consistência com o resultado real da instalação da CLI `skills`

Essa solução, para ser honesto, também pensamos nela. Só descobrimos depois que, ao instalar com a CLI, pode haver alguma lógica de pré-processamento, e copiar diretamente pula isso. O resultado é que a coisa copiada é diferente da coisa instalada, e isso é um problema.

**Solução 2: instalar globalmente e depois recuperar**

Vantagens: pode validar o processo de instalação
Desvantagens: polui o ambiente de execução, difícil manter resultados consistentes entre CI e local

Essa solução é ainda pior. Instalação global polui o ambiente. O problema é que é difícil manter consistência entre o ambiente de CI e o ambiente local, levando ao problema "funciona localmente, falha no CI". Essa sensação, quem entende?

**Solução 3: instalação local → conversão → desinstalação (solução final)**

Esta é a solução adotada pelo skillsbase:

- Primeiro instalar skills em um local temporário via `npx skills`
- Converter estrutura de diretório e adicionar metadados de origem
- Escrever no repositório de destino
- Finalmente desinstalar arquivos temporários

Essa solução garante que o conteúdo do repositório seja consistente com o resultado real da instalação do consumidor, sem poluir o ambiente global, o processo de conversão pode ser padronizado, suporta operações idempotentes.

Na verdade, essa solução não foi pensada desde o início, só depois de tentar muito é que sabemos naturalmente o que é viável e o que não é.

### Decisões de arquitetura

| Decisão | Escolha | Motivo |
|---------|---------|--------|
| Runtime | Node.js ESM | Não precisa de etapa de build, `.mjs` é suficiente para orquestração de sistema de arquivos |
| Formato de configuração | YAML (`sources.yaml`) | Alta legibilidade, suporta manutenção manual |
| Estratégia de nomenclatura | Prefixo de namespace | Skills de usuário mantêm o nome original, skills do sistema adicionam prefixo `system-` |
| Fluxo de trabalho | `add` modifica lista → `sync` executa sincronização | Motor de sincronização único, evita implementação dupla de regras |
| Gerenciamento de arquivos | Identificação de arquivos gerenciados | Adiciona cabeçalho de comentário, suporta sobrescrita segura |

Essas decisões, no fim das contas, são para um objetivo: tornar as coisas simples. Afinal, simplicidade é rei.

## Solução

### Arquitetura da CLI

A CLI `skillsbase` fornece quatro comandos principais:

```text
skillsbase
├── init          # Inicializa estrutura do repositório
├── sync          # Sincroniza conteúdo das skills
├── add           # Adiciona nova skill
└── github_action # Gera configuração do GitHub Actions
```

Não são muitos comandos, mas são suficientes. Afinal, com ferramentas, suficiente é bom o suficiente.

### Fluxo de trabalho principal

```
┌─────────────┐    ┌─────────────┐    ┌─────────────┐    ┌─────────────┐
│    init     │───▶│    add      │───▶│    sync     │───▶│github_action│
│Inicializa repo│   │Adiciona fonte│   │Sincroniza   │   │Gera CI      │
└─────────────┘    └─────────────┘    └─────────────┘    └─────────────┘
```

Passo a passo, sem pressa.

### Design do fluxo de sincronização

```
sources.yaml → Analisa fontes → npx skills instala → Converte estrutura → Escreve skills/ → Desinstala temporários
                              ↓
                        .skill-source.json (metadados de origem)
```

Este fluxo é projetado de forma bastante clara. Pelo menos quando eu mesmo olho, consigo entender o que cada passo está fazendo.

### Estrutura do repositório

```
repos/skillsbase/
├── sources.yaml              # Lista de fontes (única fonte de verdade)
├── skills/                   # Diretório de skills
│   ├── frontend-design/      # Skill do usuário
│   ├── skill-creator/        # Skill do usuário
│   └── system-skill-creator/ # Skill do sistema (com prefixo)
├── scripts/
│   ├── sync-skills.mjs       # Script de sincronização
│   └── validate-skills.mjs   # Script de validação
├── docs/
│   └── maintainer-workflow.md # Documentação do mantenedor
└── .github/
    ├── workflows/
    │   └── skills-sync.yml   # Workflow de CI
    └── actions/
        └── skillsbase-sync/
            └── action.yml     # Action reutilizável
```

São muitos arquivos, mas ainda está ok. Afinal, com a estrutura organizacional clara, a manutenção também é conveniente.

## Prática

### Inicializar repositório de skills

```bash
# 1. Criar repositório vazio
mkdir repos/myskills && cd repos/myskills
git init

# 2. Usar skillsbase para inicializar
npx skillsbase init

# Saída:
# [1/4] create manifest ................. done
# [2/4] create scripts .................. done
# [3/4] create docs ..................... done
# [4/4] create github workflow .......... done
#
# next: skillsbase add <skill-name>
```

Este passo gera um monte de arquivos, mas não se preocupe, todos são gerados automaticamente. Em seguida, você pode começar a adicionar skills.

### Adicionar skills

```bash
# Adicionar uma única skill (executa sincronização automaticamente)
npx skillsbase add frontend-design --source vercel-labs/agent-skills

# Adicionar a fonte local
npx skillsbase add documentation-writer --source /home/user/.agents/skills

# Saída:
# source: first-party ......... updated
# target: skills/frontend-design ... synced
# status: 1 skill added, 0 removed
```

Adicionar skills é bastante simples, um comando é suficiente. Às vezes encontramos algumas situações inesperadas, como rede ruim, problemas de permissão, etc. Mas essas são coisas pequenas, vá com calma.

### Sincronizar skills

```bash
# Executar sincronização (reconciliar todas as fontes)
npx skillsbase sync

# Apenas verificar se há deriva (não modifica arquivos)
npx skillsbase sync --check

# Permitir fontes ausentes (cenário de CI)
npx skillsbase sync --allow-missing-sources
```

Ao sincronizar, o sistema verifica todas as fontes definidas em `sources.yaml` e depois reconcilia com o conteúdo no diretório `skills/`. Se houver diferença, atualiza, se não houver diferença, pula. Dessa forma, não ocorre o problema de "configuração alterada mas arquivos não alterados".

### Gerar configuração de CI

```bash
# Gerar workflow
npx skillsbase github_action --kind workflow

# Gerar action
npx skillsbase github_action --kind action

# Gerar tudo
npx skillsbase github_action --kind all
```

A configuração de CI também é gerada automaticamente. Só que você precisa ajustar alguns detalhes você mesmo, como condições de disparo, ambiente de execução, etc. Mas essas coisas não são difíceis.

### Exemplo de configuração sources.yaml

```yaml
# Configuração do diretório raiz de skills
skillsRoot: skills/
metadataFile: .skill-source.json

# Definições de fontes
sources:
  # Primeira parte: skills de usuário locais
  first-party:
    type: local
    path: /home/user/.agents/skills
    naming: original  # Mantém nome original
    includes:
      - documentation-writer
      - frontend-design
      - skill-creator

  # Sistema: skills fornecidas pelo sistema
  system:
    type: local
    path: /home/user/.codex/skills/.system
    naming: prefix-system  # Adiciona prefixo system-
    includes:
      - imagegen
      - openai-docs
      - skill-creator  # Vai se tornar system-skill-creator

  # Remoto: repositórios de terceiros
  vercel:
    type: remote
    url: vercel-labs/agent-skills
    naming: original
    includes:
      - web-design-guidelines
```

Este arquivo de configuração é o núcleo de todo o sistema. Todas as fontes são definidas aqui, altere aqui, na próxima sincronização entrará em vigor. Portanto, isso conta como uma "única fonte de verdade".

### Exemplo de metadados .skill-source.json

```json
{
  "source": "first-party",
  "originalPath": "/home/user/.agents/skills/documentation-writer",
  "originalName": "documentation-writer",
  "targetName": "documentation-writer",
  "syncedAt": "2026-04-07T00:00:00.000Z",
  "version": "unknown"
}
```

Cada diretório de skill terá este arquivo, registrando suas informações de origem. Dessa forma, quando ocorrer problemas no futuro, é possível localizar rapidamente de onde veio e quando foi sincronizado.

### Segurança e validação

```bash
# Validar estrutura do repositório
node scripts/validate-skills.mjs

# Usar skills CLI para validar
npx skills add . --list

# Verificar atualizações
npx skills check
```

Validação, digamos, é importante e ao mesmo tempo não é necessária. Mas por segurança, ocasionalmente executar também não faz mal. Afinal, quem sabe se pode haver algum imprevisto?

### Integração com GitHub Actions

```yaml
# .github/workflows/skills-sync.yml
name: Skills Sync

on:
  push:
    paths:
      - 'sources.yaml'
      - 'skills/**'
  workflow_dispatch:

jobs:
  validate:
    runs-on: ubuntu-latest
    steps:
      - uses: actions/checkout@v4
      - uses: actions/setup-node@v4
        with:
          node-version: '20'
      - name: Validate repository
        run: |
          npx skills add . --list
          node scripts/validate-skills.mjs
      - name: Sync check
        run: npx skillsbase sync --check
```

Após a integração com CI, cada vez que você alterar `sources.yaml` ou o diretório `skills/`, a validação será acionada automaticamente. Dessa forma, não ocorre o problema de "alterou localmente e esqueceu de sincronizar".

### Melhores práticas

1. **Tratamento de conflitos de nomenclatura**: skills do sistema adicionam uniformemente o prefixo `system-`. Dessa forma, todas as skills podem ser mantidas e conflitos de nomenclatura podem ser evitados.
2. **Operações idempotentes**: todos os comandos suportam execução repetida, executar `sync` várias vezes não produz efeitos colaterais. Isso é especialmente importante no CI.
3. **Arquivos gerenciados**: arquivos gerados contêm o comentário `# Managed by skillsbase CLI`, conveniente para identificação e gerenciamento. Esses arquivos podem ser sobrescritos com segurança, modificações manuais não serão mantidas.
4. **Modo não interativo**: ambientes de CI usam comportamento determinístico por padrão, não serão interrompidos por prompts interativos. Toda configuração é declarada via `sources.yaml`.
5. **Rastreamento de origem**: cada skill tem `.skill-source.json` registrando informações de origem, quando ocorrer problemas é possível localizar rapidamente.

### Colaboração em equipe

```bash
# Membros da equipe instalam biblioteca de skills compartilhada
npx skills add your-org/myskills -g --all

# Clonagem local para validação
git clone https://github.com/your-org/myskills.git
cd myskills
npx skills add . --list
```

Ao gerenciar o repositório de skills via Git, os membros da equipe podem facilmente sincronizar o conjunto de skills, garantindo que todos usem a mesma versão de ferramentas e configurações.

Isso é especialmente útil na colaboração em equipe, não ocorre a situação de "funciona do meu lado mas não do seu". Afinal, com o ambiente unificado, os problemas são reduzidos pela metade.

## Conclusão

O valor central de usar `skillsbase` para manter um repositório de skills está em:

- **Segurança**: validação de origem, detecção de conflitos, proteção de arquivos gerenciados
- **Manutenibilidade**: ponto de entrada unificado, operações idempotentes, configuração como documentação
- **Padronização**: estrutura de diretório unificada, especificações de nomenclatura, formato de metadados
- **Automação**: integração CI/CD, sincronização automática, validação automática

Através desta solução, os desenvolvedores podem gerenciar seus Agent Skills como gerenciam pacotes npm,实现 um sistema de repositório de skills reproduzível, compartilhável e mantível.

As ferramentas e processos compartilhados neste artigo são exatamente as soluções que desenvolvemos pisando em armadilhas e otimizando durante o desenvolvimento do HagiCode. Se você acha que esta solução tem valor, significa que nossa direção de prática de engenharia está correta — então o HagiCode em si também vale a pena ser notado.

Afinal, boas ferramentas merecem ser usadas por mais pessoas.
