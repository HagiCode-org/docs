## Hagicode.Libs: Práticas de Engenharia para Integração Unificada de Múltiplos CLIs de Assistentes de Programação AI

> No desenvolvimento do projeto HagiCode, precisamos integrar simultaneamente múltiplos CLIs de assistentes de programação AI como Claude Code, Codex, CodeBuddy, etc. Cada CLI tem interfaces, parâmetros e formatos de saída diferentes, e o código de integração repetido tornava o projeto cada vez mais difícil de manter. Este artigo compartilha como construímos uma camada de abstração unificada através da biblioteca HagiCode.Libs para resolver essa dor de engenharia — também pode ser considerado como os buracos que pisamos e um pouco de experiência acumulada.

## Contexto

O mercado atual de assistentes de programação AI está bastante animado, além do Claude Code, há o Codex da OpenAI, o CodeBuddy da Zhipu, entre outros. Como um projeto de assistente de código AI, o HagiCode precisa integrar essas diferentes ferramentas CLI em múltiplos subprojetos (desktop, backend, Web, etc.).

No início o problema não era tão grande, integrar um CLI eram apenas algumas centenas de linhas de código. Mas conforme o número de CLIs a serem suportados aumentava, as coisas começaram a ficar complicadas —

Cada CLI tem diferentes formatos de parâmetros de linha de comando, os requisitos de variáveis de ambiente também são diferentes, e os formatos de saída são ainda mais variados — alguns saem JSON, outros JSON em stream, outros apenas texto puro. Além disso, problemas de compatibilidade entre plataformas, os mecanismos de descoberta de executáveis e gerenciamento de processos são completamente diferentes entre Windows e sistemas Unix, o grau de repetição de código ficou cada vez maior. Na verdade, isso é apenas Ctrl+C, Ctrl+V um pouco mais, mas a manutenção se torna uma dor de cabeça.

O mais doloroso é que, cada vez que precisamos adicionar suporte a uma nova funcionalidade CLI, temos que modificar o mesmo código em vários projetos. Essa abordagem obviamente não é uma solução sustentável — código também tem temperamento, repetição demais ele também faz birra.

## Sobre o HagiCode

A solução compartilhada neste artigo vem da nossa experiência prática no projeto [HagiCode](https://hagicode.com). O HagiCode é um projeto de assistente de código AI open source que precisa manter múltiplos subprojetos incluindo extensão VSCode, serviço AI de backend, cliente desktop multiplataforma, etc. Como dizer, é exatamente esse cenário complexo de multi-linguagem e multi-plataforma que impulsionou o nascimento do HagiCode.Libs — foi forçado pelas circunstâncias, mas tudo bem.

## Análise: Buscando Pontos em Comum

Embora esses CLIs de assistentes de programação AI tenham suas características próprias, do ponto de vista técnico, eles existem características comuns óbvias:

**Padrões de Interação Similares**: Todos iniciam o processo CLI, enviam prompts, recebem respostas em stream, analisam as mensagens, e finalmente a sessão termina ou continua — esse fluxo, em última análise, é igualzinho.

**Necessidades de Configuração Similares**: Todos precisam de autenticação com chave API, configuração de diretório de trabalho, seleção de modelo, controle de permissões de ferramentas, gerenciamento de sessões. Afinal, todos vivem de API, a diferença é apenas o sabor.

**Desafios Multiplataforma Consistentes**: Todos precisam resolver problemas de resolução de caminho de executável (`claude` vs `claude.exe` vs `/usr/local/bin/claude`), inicialização de processos e tratamento de variáveis de ambiente, escape de comandos Shell e construção de parâmetros, etc. Essas coisas multiplataforma, falar demais é choro — as diferenças entre Windows e Unix, só quem já pisou no buraco sabe.

Com base nessas análises, precisamos de uma camada de abstração unificada para fornecer uma interface consistente, encapsular a lógica de descoberta CLI multiplataforma, processar a análise de saída em stream, e ao mesmo tempo suportar injeção de dependência e uso sem DI. Isso parece uma dor de cabeça só de pensar, mas temos que enfrentar — afinal é nosso próprio projeto, chorando temos que terminar.

## Solução: HagiCode.Libs

Criamos o **HagiCode.Libs** — um workspace de biblioteca .NET 10 leve, adotando a licença open source MIT, agora publicado no [GitHub](https://github.com/HagiCode-org/Hagicode.Libs). Embora não seja nada monumental, resolver problemas reais é bastante satisfatório.

### Estrutura do Projeto

```
HagiCode.Libs/
├── src/
│   ├── HagiCode.Libs.Core/           # Funcionalidades principais
│   │   ├── Discovery/                 # Descoberta de executáveis CLI
│   │   ├── Process/                   # Gerenciamento de processos multiplataforma
│   │   ├── Transport/                 # Transporte de mensagens em stream
│   │   └── Environment/               # Resolução de ambiente de runtime
│   ├── HagiCode.Libs.Providers/       # Implementações de provedores
│   │   ├── ClaudeCode/                # Provedor Claude Code
│   │   ├── Codex/                     # Provedor Codex
│   │   └── Codebuddy/                 # Provedor CodeBuddy
│   ├── HagiCode.Libs.ConsoleTesting/  # Framework de teste
│   ├── HagiCode.Libs.ClaudeCode.Console/
│   ├── HagiCode.Libs.Codex.Console/
│   └── HagiCode.Libs.Codebuddy.Console/
└── tests/                             # Testes xUnit
```

### Objetivos de Design

Ao projetar o HagiCode.Libs, seguimos alguns princípios — afinal, também são buracos que pisamos e experiência resumida:

**Zero Dependências de Framework Pesado**: Não depende de ABP ou outros frameworks grandes, mantendo-se leve. Nesses dias, menos dependências, menos problemas — quem nunca foi torturado pelo inferno de dependências?

**Suporte Multiplataforma**: Suporte nativo para Windows, macOS, Linux, sem precisar escrever código diferente para cada plataforma. Um código para todos os lugares, também é bom.

**Processamento em Stream**: Usa streams assíncronos para processar saídas CLI, mais adequado ao padrão de programação .NET moderno. Afinal, os tempos mudam, assincronia é o rei.

**Integração Flexível**: Tanto suporta cenários de injeção de dependência quanto permite instanciação direta. Para cada gosto, o que for mais conveniente.

## Formas de Uso

### Através de Injeção de Dependência

Se seu projeto já está usando injeção de dependência (como ASP.NET Core ou Host Genérico), pode integrar diretamente — é como criar um filho, embora pequeno, mas bem comportado:

```csharp
using HagiCode.Libs.Providers;
using Microsoft.Extensions.DependencyInjection;

var services = new ServiceCollection();
services.AddHagiCodeLibs();

await using var provider = services.BuildServiceProvider();
var claude = provider.GetRequiredService<ICliProvider<ClaudeCodeOptions>>();

var options = new ClaudeCodeOptions
{
    ApiKey = "your-api-key",
    Model = "claude-sonnet-4-20250514"
};

await foreach (var message in claude.ExecuteAsync(options, "Hello, Claude!"))
{
    Console.WriteLine($"{message.Type}: {message.Content}");
}
```

### Instanciação Direta

Para scripts simples ou cenários sem DI, criar a instância diretamente também funciona — basicamente é questão de preferência pessoal:

```csharp
var claude = new ClaudeCodeProvider();
var options = new ClaudeCodeOptions
{
    ApiKey = "sk-ant-xxx",
    Model = "claude-sonnet-4-20250514"
};

await foreach (var message in claude.ExecuteAsync(options, "帮我写一个快速排序"))
{
    // Processar mensagem
}
```

Ambas as formas usam a mesma implementação subjacente, você pode escolher a forma de integração adequada conforme a situação real do projeto. Neste mundo não há resposta padrão, o que serve para você é o melhor — embora essa frase seja um pouco clichê, mas realmente é assim.

## Compartilhamento de Experiência Prática

### 1. Console de Teste Dedicado

Cada provedor tem um projeto de console de teste dedicado, facilitando a verificação independente do efeito de integração — como dizer, sobre testes, ou não faz, ou faz direito:

```bash
# Teste Claude Code
dotnet run --project src/HagiCode.Libs.ClaudeCode.Console -- --test-provider
dotnet run --project src/HagiCode.Libs.ClaudeCode.Console -- --test-all claude

# Teste CodeBuddy
dotnet run --project src/HagiCode.Libs.Codebuddy.Console -- --test-provider codebuddy-cli

# Teste Codex
dotnet run --project src/HagiCode.Libs.Codex.Console -- --test-provider codex-cli
```

Os cenários de teste cobrem alguns cenários chave:
- **Ping**: Verificação de saúde, confirma que o CLI está disponível
- **Simple Prompt**: Teste básico de prompt
- **Complex Prompt**: Teste de múltiplas rodadas de conversa
- **Session Restore/Resume**: Teste de recuperação de sessão
- **Repository Analysis**: Teste de análise de repositório de código

Este design de console de teste independente é especialmente útil na depuração, podendo rapidamente localizar se o problema está na camada HagiCode.Libs ou no próprio CLI. Sobre depuração, basicamente é ver onde está o problema — direção certa, metade do sucesso.

### 2. Verificação CI/CD Multiplataforma

Compatibilidade multiplataforma é um dos objetivos centrais do HagiCode.Libs. Configuramos o workflow GitHub Actions `.github/workflows/cli-discovery-cross-platform.yml`, executando verificação de descoberta CLI real em três plataformas: `ubuntu-latest`, `macos-latest`, `windows-latest`.

Isso garante que cada mudança de código não quebre a compatibilidade multiplataforma. No desenvolvimento local também pode reproduzir através dos seguintes comandos — não pode deixar o CI levar toda culpa, você também precisa conseguir rodar localmente:

```bash
npm install --global @anthropic-ai/claude-code@2.1.79
HAGICODE_REAL_CLI_TESTS=1 dotnet test --filter "Category=RealCli"
```

### 3. Processamento de Stream de Mensagens

O HagiCode.Libs usa streams assíncronos para processar saídas CLI, essa abordagem é mais adequada ao estilo de programação assíncrona .NET moderno do que os padrões tradicionais de callback ou evento — em última análise, isso é progresso tecnológico, ninguém pode segurar:

```csharp
public async IAsyncEnumerable<CliMessage> ExecuteAsync(
    TOptions options,
    string prompt,
    [EnumeratorCancellation] CancellationToken cancellationToken = default)
{
    // Iniciar processo CLI
    // Analisar saída JSON em stream
    // Gerar sequência CliMessage
}
```

Tipos de mensagem incluem:
- `user`: Mensagem do usuário
- `assistant`: Resposta do assistente
- `tool_use`: Chamada de ferramenta
- `result`: Fim da sessão

Este design permite que o chamador processe flexivelmente a saída em stream, como exibição em tempo real, buffer para processamento posterior, ou encaminhamento para outros serviços. Para que se importar com o tempo? O importante é abrir a mente, como usar tudo dá certo.

### 4. Exploração de Repositório Git

O módulo HagiCode.Libs.Exploration fornece funcionalidades de descoberta e verificação de estado de repositórios Git, isso é especialmente útil em cenários de análise de repositórios de código — esta também é uma funcionalidade forçada pelas circunstâncias, quem mandou o HagiCode precisar analisar repositórios de código:

```csharp
// Descobrir repositórios Git
var repositories = await GitRepositoryDiscovery.DiscoverAsync("/path/to/search");

// Obter informações do repositório
var info = await GitRepository.GetInfoAsync(repoPath);
Console.WriteLine($"Branch: {info.Branch}, Remote: {info.RemoteUrl}");
Console.WriteLine($"Has uncommitted changes: {info.HasUncommittedChanges}");
```

A funcionalidade de análise de código do HagiCode usa este módulo para identificar estrutura de projeto e estado Git. Pode ser considerado aproveitamento máximo, também é como criar um filho, não criado em vão.

## Precauções

Com base na nossa prática no projeto HagiCode, há alguns lugares que precisam de atenção especial — tudo é problema, problema, problema:

**Segurança de Chave API**: Não codifique chaves API diretamente no código, use variáveis de ambiente ou gerenciamento de configuração. O HagiCode.Libs suporta passar configuração através de objetos Options, facilitando a integração com várias fontes de configuração. Afinal, segurança, quanto mais cuidado, melhor.

**Lock de Versão CLI**: No CI/CD travamos versões específicas (como `@anthropic-ai/claude-code@2.1.79`) para reduzir incerteza causada por drift de versão. No desenvolvimento local também recomendamos usar versão fixa. Sobre versões, falar demais é choro — sem travar versão, em segundos te ensina a ser gente.

**Classificação de Testes**: Por padrão os testes usam provedores falsos para manter determinismo e velocidade, testes CLI reais precisam ser ativados explicitamente. Isso garante feedback rápido do CI, e ao mesmo tempo permite verificação de ambiente real quando necessário. Esse equilíbrio também não é fácil — rápido e estável, sempre precisa de trade-off.

**Gerenciamento de Sessão**: Diferentes CLIs têm mecanismos diferentes de recuperação de sessão, Claude Code usa o diretório `.claude/` para armazenar sessões, Codex e CodeBuddy têm suas próprias formas. Ao usar, preste atenção para verificar a documentação respectiva, entender o mecanismo específico de persistência de sessão. Isso tudo, entender claramente nunca faz mal.
