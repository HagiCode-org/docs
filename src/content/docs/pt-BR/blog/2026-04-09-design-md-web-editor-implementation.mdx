## Editando DESIGN.md diretamente na interface Web: do conceito à implementação

> No sistema de gerenciamento de projetos MonoSpecs, DESIGN.md contém a arquitetura do projeto e decisões técnicas. Mas o método tradicional de edição exige que os usuários alternem para editores externos, esse fluxo fragmentado, como diremos, é como quando você está lendo um poema e é subitamente interrompido—a inspiração se foi, o humor também. Este artigo compartilha a solução que praticamos no projeto HagiCode: editar DESIGN.md diretamente na interface Web, com suporte para importar modelos de sites de design online. Afinal, quem não gosta da sensação de fluidez?

## Contexto

`DESIGN.md` como portador principal da documentação de design do projeto, contém informações críticas como design de arquitetura, decisões técnicas e orientações de implementação. No entanto, o método tradicional de edição exige que os usuários alternem para editores externos (como VS Code), localizando manualmente o caminho físico antes de editar. Esse processo, para ser honesto, não é complexo, mas depois de várias repetições, as pessoas se cansam.

Os problemas específicos se manifestam nos seguintes aspectos:

- **Fluxo fragmentado**: Os usuários precisam alternar frequentemente entre a interface de gerenciamento Web e o editor local, quebrando a continuidade do fluxo de trabalho—como quando a rede cai subitamente enquanto você ouve uma música, o ritmo todo se desorganiza.
- **Dificuldade de reutilização**: O site de design já publicou uma rica biblioteca de modelos de design, mas não pode ser integrado diretamente ao fluxo de edição do projeto. Claramente há coisas boas, mas não podem ser usadas, o que é realmente uma pena.
- **Experiência insuficiente**: Falta o ciclo fechado de "pré-visualização-seleção-importação", os usuários devem copiar e colar manualmente, aumentando o risco de erros. Quanto mais operações manuais, maiores as chances de erros.
- **Barreiras de colaboração**: A manutenção síncrona da documentação de design e da implementação do código se torna de alta fricção, dificultando a eficiência da colaboração em equipe. A colaboração em equipe já não é fácil, por que adicionar mais essas resistências?

Para resolver essas dores, decidimos implementar a capacidade de edição direta do DESIGN.md na interface Web, com suporte para importação de modelos de sites de design online com um clique. Isso não é uma decisão surpreendente, apenas queremos tornar a experiência de desenvolvimento mais suave.

## Sobre HagiCode

A solução compartilhada neste artigo vem de nossa experiência prática no projeto [HagiCode](https://hagicode.com). HagiCode é um projeto de assistente de código impulsionado por IA, durante o desenvolvimento, precisamos manter frequentemente a documentação de design do projeto. Para permitir que a equipe colabore com mais eficiência, exploramos e implementamos esta solução de edição online e importação. Na verdade, não há nada de especial, apenas encontramos um problema e procuramos resolvê-lo.

## Solução técnica

### Arquitetura geral

A solução adota uma arquitetura de proxy de mesma origem com separação frontend-backend, composta principalmente pelas seguintes camadas. O design desta arquitetura, para ser honesto, é apenas "cada um com sua função":

**1. Camada de editor frontend**
```typescript
// Componente principal: DesignMdManagementDrawer
// Localização: repos/web/src/components/project/DesignMdManagementDrawer.tsx
// Funcionalidade: suporta edição, salvamento, detecção de conflitos de versão, fluxo de importação
```

**2. Camada de serviço backend**
```csharp
// Serviço principal: ProjectAppService.DesignMd
// Localização: repos/hagicode-core/src/PCode.Application/ProjectAppService.DesignMd.cs
// Funcionalidade: resolução de caminho, leitura/escrita de arquivos, gerenciamento de versão
```

**3. Camada de proxy de mesma origem**
```csharp
// Serviço de proxy: ProjectAppService.DesignMdSiteIndex
// Localização: repos/hagicode-core/src/PCode.Application/ProjectAppService.DesignMdSiteIndex.cs
// Funcionalidade: proxy de recursos do site de design, cache de imagens de pré-visualização, validação de segurança
```

### Decisões técnicas-chave

#### Decisão 1: Modo de gaveta global

Adotar uma única gaveta global em vez de camadas pop-up locais, gerenciando estado através de `layoutSlice`,实现了experiência consistente entre visualizações (classic/kanban). Esta abordagem garante que os usuários obtenham uma experiência de interação unificada, independentemente de qual visualização abrem o editor. Afinal, uma experiência consistente faz os usuários se sentirem mais confortáveis, não se perdendo ao trocar de visualização.

#### Decisão 2: API com escopo de projeto

Colocar as interfaces relacionadas ao DESIGN.md sob `ProjectController`, reaproveitando os limites de permissão de projeto existentes e evitando a complexidade de adicionar um novo controlador independente. Os benefícios deste design são permissões mais claras e estão em conformidade com os princípios de organização de recursos RESTful. Às vezes, reaproveitar faz mais sentido do que criar do zero, não concorda?

#### Decisão 3: Detecção de conflitos de versão

Derivar uma versão opaca baseada em `LastWriteTimeUtc` do sistema de arquivos,实现了controle de concorrência otimista leve. Quando vários usuários editam o mesmo arquivo simultaneamente, o sistema pode detectar conflitos e solicitar que os usuários atualizem. Este design não bloqueia as operações de edição do usuário, garantindo a consistência dos dados—como o senso de limites nas relações interpessoais, nem muito distante nem ultrapassando limites.

#### Decisão 4: Modo de proxy de mesma origem

Através de `IHttpClientFactory` proxy de recursos externos do site de design, evitando problemas de CORS e riscos de SSRF. Este design garante segurança e simplifica as chamadas frontend. Segurança, não é demais falar, afinal a segurança dos dados é como saúde, só nos arrependemos quando perdida.

## Implementação principal

### 1. Editando DESIGN.md diretamente

#### Implementação backend

O backend é responsável principalmente pela resolução de caminho, leitura/escrita de arquivos e gerenciamento de versão. Este trabalho, embora básico, é indispensável, como a fundação de uma casa:

```csharp
// Resolução de caminho e validação de segurança
private Task<string> ResolveDesignDocumentDirectoryAsync(string projectPath, string? repositoryPath)
{
    if (string.IsNullOrWhiteSpace(repositoryPath))
    {
        return Task.FromResult(Path.GetFullPath(projectPath));
    }
    return ValidateSubPathAsync(projectPath, repositoryPath);
}

// Geração de número de versão (baseado em timestamp e tamanho do sistema de arquivos)
private static string BuildDesignDocumentVersion(string path)
{
    var fileInfo = new FileInfo(path);
    fileInfo.Refresh();
    return string.Create(
        CultureInfo.InvariantCulture,
        $"{fileInfo.LastWriteTimeUtc.Ticks:x}-{fileInfo.Length:x}");
}
```

O design do número de versão é bastante interessante, usamos o horário da última modificação e o tamanho do arquivo para gerar um identificador de versão único. Isso é leve e confiável, não precisando manter um banco de dados de versão adicional. Coisas simples são muitas vezes mais eficazes, não concorda?

#### Implementação frontend

O frontend实现了detecção de estado sujo e lógica de salvamento. Este design permite que os usuários saibam a qualquer momento se suas modificações foram salvas, reduzindo a ansiedade de "e se isso for perdido":

```typescript
// Detecção de estado sujo e lógica de salvamento
const [draft, setDraft] = useState('');
const [savedDraft, setSavedDraft] = useState('');
const isDirty = draft !== savedDraft;

const handleSave = useCallback(async () => {
    const result = await saveProjectDesignMdDocument({
        ...activeTarget,
        content: draft,
        expectedVersion: document.version, // controle de concorrência otimista
    });
    setSavedDraft(draft); // atualiza estado salvo
}, [activeTarget, document, draft]);
```

Nesta implementação, mantemos dois estados: `draft` é o conteúdo sendo editado atualmente, `savedDraft` é o conteúdo salvo. Comparamos ambos para determinar se há modificações não salvas. Este design, embora simples, pode trazer tranquilidade, afinal quem quer que algo trabalhoso duramente desapareça subitamente?

### 2. Importando arquivos de design online

#### Estrutura de diretórios

```
repos/index/
└── src/data/public/design.json    # índice de modelos de design

repos/awesome-design-md-site/
├── vendor/awesome-design-md/       # modelos de design upstream
│   └── design-md/
│       ├── clickhouse/
│       │   └── DESIGN.md
│       ├── linear/
│       │   └── DESIGN.md
│       └── ...
└── src/lib/content/
    └── awesomeDesignCatalog.ts     # pipeline de conteúdo
```

#### Formato de dados de índice

O arquivo de índice do site de design define todos os modelos disponíveis. Com este índice, os usuários podem selecionar facilmente o modelo desejado como em um restaurante:

```json
{
  "entries": [
    {
      "slug": "linear.app",
      "title": "Linear Inspired Design System",
      "summary": "AI product / Dark theme",
      "detailUrl": "/designs/linear.app/",
      "designDownloadUrl": "/designs/linear.app/DESIGN.md",
      "previewLightImageUrl": "...",
      "previewDarkImageUrl": "..."
    }
  ]
}
```

Cada entrada contém informações básicas e links de download do modelo. O backend lê a lista de modelos disponível deste índice e a exibe para os usuários selecionarem. Este design torna a seleção intuitiva, em vez de procurar no escuro.

#### Implementação de proxy de mesma origem

Para garantir segurança, o backend realiza validação estrita do acesso ao site de design. Segurança, nunca é demais ser cuidadoso:

```csharp
// Validação segura de slug
private static readonly Regex SafeDesignSiteSlugRegex =
    new("^[A-Za-z0-9](?:[A-Za-z0-9._-]{0,127})$", RegexOptions.Compiled);

private static string NormalizeDesignSiteSlug(string slug)
{
    var normalizedSlug = slug?.Trim() ?? string.Empty;
    if (!IsSafeDesignSiteSlug(normalizedSlug))
    {
        throw new BusinessException(
            ProjectDesignSiteIndexErrorCodes.InvalidSlug,
            "Design site slug must be a single safe path segment.");
    }
    return normalizedSlug;
}

// Cache de imagens de pré-visualização (diretório temporário do SO)
private static string ComputePreviewCacheKey(string slug, string theme, string previewUrl)
{
    var raw = $"{slug}|{theme}|{previewUrl}";
    var bytes = SHA256.HashData(Encoding.UTF8.GetBytes(raw));
    return Convert.ToHexString(bytes).ToLowerInvariant();
}
```

Aqui fizemos duas coisas: primeiro, validamos estritamente o formato do slug com expressão regular para prevenir ataques de traversia de caminho; segundo, cacheamos as imagens de pré-visualização para reduzir a pressão no site externo. O primeiro é proteção, o segundo é otimização, ambos indispensáveis.

### 3. Fluxo completo de importação

```typescript
// 1. Abrir gaveta de importação
const handleRequestImportDrawer = useCallback(() => {
    setIsImportDrawerOpen(true);
}, []);

// 2. Selecionar e importar
const handleImportRequest = useCallback((entry) => {
    if (isDirty) {
        setPendingImportEntry(entry);
        setConfirmMode('import'); // confirmação de sobrescrita
        return;
    }
    void executeImport(entry);
}, [isDirty]);

// 3. Executar importação
const executeImport = useCallback(async (entry) => {
    const result = await getProjectDesignMdSiteImportDocument(
        activeTarget.projectId,
        entry.slug
    );
    setDraft(result.content); // substitui apenas o texto do editor, não salva automaticamente
    setIsImportDrawerOpen(false);
}, [activeTarget?.projectId]);
```

O design do fluxo de importação segue o princípio de "confirmação do usuário": após a importação, apenas o conteúdo do editor é atualizado, sem salvamento automático. Os usuários podem verificar o conteúdo importado e salvar manualmente após confirmação. Afinal, os usuários devem ter a decisão final sobre o que escrevem, não concorda?

## Exemplos práticos

### Cenário 1: Criação de DESIGN.md no diretório raiz do projeto

Quando DESIGN.md não existe, o backend retorna um estado de documento virtual. Este design permite que o frontend não precise tratamento especial para "arquivo não existe", a interface de API unificada simplifica a lógica do código:

```csharp
return new ProjectDesignDocumentDto
{
    Path = targetPath,
    Exists = false,  // estado de documento virtual
    Content = string.Empty,
    Version = null
};

// Arquivo criado automaticamente no primeiro salvamento
public async Task<SaveProjectDesignDocumentResultDto> SaveDesignDocumentAsync(...)
{
    Directory.CreateDirectory(targetDirectory);
    await File.WriteAllTextAsync(targetPath, input.Content);
    return new SaveProjectDesignDocumentResultDto { Created = !exists };
}
```

Os benefícios deste design são que o frontend não precisa tratamento especial para "arquivo não existe", a interface de API unificada simplifica a lógica do código. Às vezes, esconder a complexidade no backend permite que o frontend se concentre mais facilmente na experiência do usuário.

### Cenário 2: Importando modelo do site de design

Quando o usuário seleciona o modelo de design "Linear" na gaveta de importação, o sistema obtém o conteúdo DESIGN.md através do proxy backend. Todo o fluxo é transparente para o usuário, eles apenas selecionam o modelo e o sistema processa automaticamente todas as requisições de rede e conversão de dados:

```typescript
// 1. Sistema obtém conteúdo DESIGN.md através do proxy backend
GET /api/project/{id}/design-md/site-index/linear.app

// 2. Backend valida slug e obtém conteúdo do upstream
var entry = FindDesignSiteEntry(catalog, "linear.app");
using var upstreamResponse = await httpClient.SendAsync(request);
var content = await upstreamResponse.Content.ReadAsStringAsync();

// 3. Frontend substitui o texto do editor
setDraft(result.content);
// Usuário verifica e salva manualmente no disco
```

Todo o fluxo é transparente para o usuário, eles apenas selecionam o modelo e o sistema processa automaticamente todas as requisições de rede e conversão de dados. Os usuários não precisam se preocupar com a complexidade por trás, essa é a experiência que buscamos—simples, mas poderosa.

### Cenário 3: Tratamento de conflitos de versão

Quando vários usuários editam o mesmo DESIGN.md simultaneamente, o sistema detecta o conflito de versão. Este mecanismo de controle de concorrência otimista garante a consistência dos dados sem bloquear as operações de edição do usuário:

```csharp
if (!string.Equals(currentVersion, expectedVersion, StringComparison.Ordinal))
{
    throw new BusinessException(
        ProjectDesignDocumentErrorCodes.VersionConflict,
        $"DESIGN.md at '{targetPath}' changed on disk.");
}
```

O frontend captura este erro e alerta o usuário:

```typescript
// Frontend alerta o usuário para atualizar e tentar novamente
<Alert>
    <AlertTitle>Conflito de versão</AlertTitle>
    <AlertDescription>
        O arquivo foi modificado por outro processo. Atualize para a versão mais recente e tente novamente.
    </AlertDescription>
</Alert>
```

Este mecanismo de controle de concorrência otimista garante a consistência dos dados sem bloquear as operações de edição do usuário. Conflitos são inevitáveis, mas pelo menos podemos deixar os usuários saberem o que aconteceu, em vez de perder modificações silenciosamente.

## Considerações e melhores práticas

### 1. Segurança de caminho

Sempre validar `repositoryPath` para prevenir ataques de traversia de caminho. Segurança, nunca é demais falar:

```csharp
// Sempre validar repositoryPath, prevenir ataques de traversia de caminho
return ValidateSubPathAsync(projectPath, repositoryPath);
// Rejeita "../", caminhos absolutos e outras entradas perigosas
```

### 2. Estratégia de cache

Cache de imagens de pré-visualização por 24 horas, máximo 160 arquivos. Cache moderado melhora o desempenho, mas não excessivo, afinal o equilíbrio é a chave:

```csharp
// Cache de imagens de pré-visualização por 24 horas, máximo 160 arquivos
private static readonly TimeSpan PreviewCacheTtl = TimeSpan.FromHours(24);
private const int PreviewCacheMaxFiles = 160;
// Limpeza periódica de cache expirado
```

### 3. Tratamento de erros

Degradação graciosa quando o site upstream não está disponível. Este design de degradação graciosa garante que mesmo quando dependências externas não estão disponíveis, a funcionalidade principal de edição ainda funciona. Afinal, não é porque um serviço externo caiu que todo o sistema deve parar:

```typescript
// Degradação graciosa quando o site upstream não está disponível
try {
    const catalog = await getProjectDesignMdSiteImportCatalog(projectId);
} catch (error) {
    toast.error(t('project.designMd.siteImport.feedback.catalogLoadFailed'));
    // Gaveta de edição principal ainda está disponível
}
```

Este design de degradação graciosa garante que mesmo quando dependências externas não estão disponíveis, a funcionalidade principal de edição ainda funciona. O sistema deve ser resiliente, não colapsar ao encontrar problemas.

### 4. Otimização da experiência do usuário

Confirmar sobrescrita antes de importar, não salvar automaticamente após importar. Os usuários devem ter controle sobre suas operações, não o sistema tomando decisões por si só:

```typescript
// Confirmar sobrescrita antes de importar
if (isDirty) {
    setConfirmMode('import');
    return;
}

// Não salvar automaticamente após importar, confirmado pelo usuário
setDraft(result.content); // apenas atualiza o rascunho
// Usuário verifica e clica em Save para escrever no disco
```

### 5. Considerações de desempenho

Usar fábrica de cliente HTTP, evitar criar muitas conexões. Gerenciamento de recursos, parece insignificante, mas bem feito pode trazer efeitos surpreendentes:

```csharp
// Usar fábrica de cliente HTTP, evitar criar muitas conexões
private const string DesignSiteProxyClientName = "ProjectDesignSiteProxy";
private static readonly TimeSpan DesignSiteProxyTimeout = TimeSpan.FromSeconds(8);
```

## Sugestões de extensão

1. **Melhorias de Markdown**: Atualmente usando `Textarea` básico, considere atualizar para CodeMirror para suportar destaque de sintaxe e atalhos. Com a experiência do editor melhorada, o humor de escrever documentação também melhora.
2. **Modo de pré-visualização**: Adicionar pré-visualização Markdown em tempo real para melhorar a experiência de edição. Ver o que você obtém sempre traz mais confiança.
3. **Mesclagem de diferenças**: Implementar algoritmo de mesclagem inteligente, em vez de simples substituição de texto completo. Conflitos são inevitáveis, mas pelo menos podemos tornar o processo de tratamento menos doloroso.
4. **Cache local**: Cache design.json no banco de dados, reduzir dependência do site externo. Menos dependências, sistema mais estável, é lógica simples.

## Conclusão

No projeto HagiCode, através da colaboração frontend-backend implementamos uma solução completa de edição online e importação de DESIGN.md. O valor central desta solução reside em:

- **Melhoria de eficiência**: Sem necessidade de alternar ferramentas, edite e importe documentação de design em uma interface Web unificada. O tempo economizado pode ser usado para coisas mais significativas.
- **Redução de barreiras**: Importação de modelos de design com um clique, novos projetos podem começar rapidamente. Quanto mais fácil começar, maior a probabilidade de continuar.
- **Segurança e confiabilidade**: Mecanismos como validação de caminho, detecção de conflitos de versão e degradação graciosa garantem operação estável do sistema. Estabilidade é a base, sem estabilidade, tudo é conversa vazia.
- **Experiência do usuário**: Detalhes como gaveta global, detecção de estado sujo e diálogos de confirmação poliram a experiência de interação. Detalhes determinam o sucesso ou fracasso, especialmente aplicável à experiência do usuário.

Esta solução está funcionando no projeto HagiCode, resolvendo dores da equipe no gerenciamento de documentação de design.
