## Guia de Implementação de Suporte a Palavras-chave para Reconhecimento de Fala Doubao

> Este artigo apresentará em detalhes como implementar o suporte a palavras-chave para reconhecimento de fala Doubao no projeto HagiCode, melhorando significativamente a precisão de reconhecimento de vocabulário em áreas específicas através de dois métodos: palavras-chave personalizadas e tabelas de palavras-chave da plataforma.

## Contexto

A tecnologia de reconhecimento de fala evoluiu tanto ao longo dos anos, mas há um problema que continua desafinando os desenvolvedores. Modelos de reconhecimento de fala gerals podem cobrir a linguagem do dia a dia, mas para termos técnicos, nomes de produtos, nomes de pessoas, a precisão do reconhecimento sempre deixa um pouco a desejar. Pense nisso, assistentes de voz na área médica precisam reconhecer com precisão "hipertensão", "diabetes", "doença coronariana"; sistemas jurídicos precisam capturar com precisão "causa da ação", "defesa", "ônus da prova" - nestes cenários, o desempenho dos modelos gerais, bem, eles fazem o melhor que podem.

No projeto HagiCode, também enfrentamos o mesmo desafio. Como um assistente de código de IA multifuncional, o HagiCode precisa lidar com vários cenários de reconhecimento de fala de termos técnicos. No entanto, a API de reconhecimento de fala Doubao por padrão não consegue atender completamente aos nossos requisitos de precisão para termos técnicos. Na verdade, não é que o Doubao não é bom o suficiente, é que cada área tem seu próprio sistema de terminologia. Após alguma pesquisa e exploração técnica, descobrimos que a API de reconhecimento de fala Doubao na verdade fornece funcionalidade de suporte a palavras-chave, e com uma configuração simples, pode melhorar significativamente a precisão de reconhecimento de vocabulário específico. É mais ou menos assim, você diz a ela para prestar atenção a certas palavras, e ela vai ouvir com mais atenção essas palavras.

O que este artigo vai compartilhar é a solução completa para implementar a funcionalidade de palavras-chave para reconhecimento de fala Doubao no projeto HagiCode. Dois modos, palavras-chave personalizadas e tabela de palavras-chave da plataforma, ambos podem ser usados e também podem ser combinados. Através desta solução, os desenvolvedores podem configurar palavras-chave flexivelmente de acordo com cenários de negócios, permitindo que o sistema de reconhecimento de fala "conheça" aquele vocabulário profissional, raro mas crítico.

## Implementação Principal

### Análise dos Dois Modos de Palavras-chave

A API de reconhecimento de fala Doubao nos fornece dois métodos de configuração de palavras-chave, cada um com seus cenários de aplicação únicos e vantagens.

O **Modo de Palavras-chave Personalizadas** nos permite passar texto de palavras-chave diretamente através do campo `corpus.context`. Este método é muito adequado para cenários que precisam configurar rapidamente algumas palavras-chave, como precisar temporariamente reconhecer um determinado nome de produto ou nome de pessoa. Na implementação do HagiCode, analisamos o texto de palavras-chave de múltiplas linhas inserido pelo usuário em uma lista de strings, e então o formatamos como um array `context_data` de acordo com os requisitos da API Doubao. Como dizer, este método é muito direto, como dizer ao outro lado "você precisa prestar atenção a estas palavras", e então ele vai prestar atenção.

O **Modo de Tabela de Palavras-chave da Plataforma** referencia a tabela de palavras-chave pré-configurada na plataforma de autoaprendizado Doubao através do campo `corpus.boosting_table_id`. Este método é adequado para cenários que precisam gerenciar um grande número de palavras-chave. Podemos criar e manter tabelas de palavras-chave na plataforma de autoaprendizado Doubao, e então referenciá-las através de ID. Para projetos como o HagiCode que precisam atualizar e manter continuamente terminologia profissional, este modo oferece melhor capacidade de gerenciamento. Afinal, quando há muitas palavras-chave, encontrar um lugar para gerenciá-las de forma centralizada é melhor do que ter que inseri-las manualmente cada vez.

O interessante é que estes dois modos também podem ser combinados. A API Doubao suporta incluir palavras-chave personalizadas e ID de tabela de palavras-chave da plataforma no mesmo pedido, controlando a estratégia de combinação através do parâmetro `combine_mode`. Esta flexibilidade permite que o HagiCode possa lidar com várias necessidades complexas de reconhecimento de terminologia profissional. Isso também é muito bom, às vezes combinando múltiplos métodos, o efeito pode ser melhor.

### Definições de Tipos e Validação Frontend

Na implementação frontend do HagiCode, definimos um conjunto completo de tipos de configuração de palavras-chave e lógica de validação. Primeiro, a parte de definição de tipos:

```typescript
export interface HotwordConfig {
  contextText: string;           // texto de palavras-chave de múltiplas linhas
  boostingTableId: string;      // ID da tabela de palavras-chave da plataforma Doubao
  combineMode: boolean;          // se deve usar em combinação
}
```

Esta interface simples contém todos os itens de configuração da funcionalidade de palavras-chave. Onde `contextText` é a parte que o usuário percebe mais diretamente - permitimos que o usuário insira uma frase de palavra-chave por linha, o que é muito intuitivo. Afinal, deixar o usuário inserir uma palavra por linha é melhor do que deixar o usuário entender regras de configuração complexas.

A seguir está a implementação da função de validação. Considerando as limitações da API Doubao, estabelecemos regras de validação estritas: texto de palavras-chave no máximo 100 linhas, cada linha no máximo 50 caracteres, total no máximo 5000 caracteres; boosting_table_id no máximo 200 caracteres, permitindo apenas letras, números, sublinhados e hífens. Estas limitações não foram imaginadas por nós, mas são baseadas nos requisitos reais da documentação oficial Doubao. Afinal, as limitações da API são limitações da API, não temos escolha, só podemos obedecer.

```typescript
export function validateContextText(contextText: string): HotwordValidationResult {
  if (!contextText || contextText.trim().length === 0) {
    return { isValid: true, errors: [] };
  }

  const lines = contextText.split('\n').filter(line => line.trim().length > 0);
  const errors: string[] = [];

  if (lines.length > 100) {
    errors.push(`O número de linhas de palavras-chave não pode exceder 100, atualmente ${lines.length} linhas`);
  }

  const totalChars = contextText.length;
  if (totalChars > 5000) {
    errors.push(`O número total de caracteres de palavras-chave não pode exceder 5000, atualmente ${totalChars}`);
  }

  for (let i = 0; i < lines.length; i++) {
    if (lines[i].length > 50) {
      errors.push(`A palavra-chave na linha ${i + 1} excede o limite de 50 caracteres`);
    }
  }

  return { isValid: errors.length === 0, errors };
}

export function validateBoostingTableId(boostingTableId: string): HotwordValidationResult {
  if (!boostingTableId || boostingTableId.trim().length === 0) {
    return { isValid: true, errors: [] };
  }

  const errors: string[] = [];

  if (boostingTableId.length > 200) {
    errors.push(`boosting_table_id não pode exceder 200 caracteres, atualmente ${boostingTableId.length}`);
  }

  if (!/^[a-zA-Z0-9_-]+$/.test(boostingTableId)) {
    errors.push('boosting_table_id só pode conter letras, números, sublinhados e hífens');
  }

  return { isValid: errors.length === 0, errors };
}
```

Essas funções de validação são executadas imediatamente quando o usuário configura as palavras-chave, garantindo que os problemas sejam descobertos na fase mais precoce. Para a experiência do usuário, este feedback imediato é muito importante. Afinal, saber onde está o erro durante a entrada é melhor do que descobrir após o envio.

### Persistência de Configuração Frontend

Na implementação frontend do HagiCode, escolhemos usar o localStorage do navegador para armazenar a configuração de palavras-chave. Há algumas considerações por trás desta decisão de design: primeiro, a configuração de palavras-chave é uma configuração muito personalizada, diferentes usuários podem ter diferentes necessidades de áreas profissionais; segundo, este método simplifica a implementação backend, não precisa de tabelas de banco de dados adicionais e interfaces API; finalmente, depois que o usuário configura uma vez no navegador, o uso subsequente pode carregar automaticamente, muito conveniente. Na verdade, para ser honesto, é apenas para economizar trabalho.

```typescript
const HOTWORD_STORAGE_KEYS = {
  contextText: 'hotword-context-text',
  boostingTableId: 'hotword-boosting-table-id',
  combineMode: 'hotword-combine-mode',
} as const;

export const DEFAULT_HOTWORD_CONFIG: HotwordConfig = {
  contextText: '',
  boostingTableId: '',
  combineMode: false,
};

// carregar configuração de palavras-chave
export function loadHotwordConfig(): HotwordConfig {
  const contextText = localStorage.getItem(HOTWORD_STORAGE_KEYS.contextText) || '';
  const boostingTableId = localStorage.getItem(HOTWORD_STORAGE_KEYS.boostingTableId) || '';
  const combineMode = localStorage.getItem(HOTWORD_STORAGE_KEYS.combineMode) === 'true';

  return { contextText, boostingTableId, combineMode };
}

// salvar configuração de palavras-chave
export function saveHotwordConfig(config: HotwordConfig): void {
  localStorage.setItem(HOTWORD_STORAGE_KEYS.contextText, config.contextText);
  localStorage.setItem(HOTWORD_STORAGE_KEYS.boostingTableId, config.boostingTableId);
  localStorage.setItem(HOTWORD_STORAGE_KEYS.combineMode, String(config.combineMode));
}
```

A lógica deste código é muito simples e clara. Ao carregar a configuração, lê-se do localStorage; ao salvar a configuração, escreve-se no localStorage. Também fornecemos uma configuração padrão, garantindo que o sistema possa funcionar normalmente mesmo sem nenhuma configuração. Afinal, sempre deve haver um valor padrão.

Afinal, quando não está em uso, não deveria existir, isso é que é limpo.

### Lógica de Construção de Payload

A construção do Payload é o núcleo de toda a funcionalidade de palavras-chave. Quando temos a configuração de palavras-chave, precisamos formatá-la na estrutura JSON correta de acordo com os requisitos da API Doubao. Este processo acontece antes do SDK enviar o pedido:

```csharp
private void AddCorpusToRequest(Dictionary<string, object> request)
{
    var corpus = new Dictionary<string, object>();

    // Adicionar palavras-chave personalizadas
    if (Config.HotwordContexts != null && Config.HotwordContexts.Count > 0)
    {
        corpus["context"] = new Dictionary<string, object>
        {
            ["context_type"] = "dialog_ctx",
            ["context_data"] = Config.HotwordContexts
                .Select(text => new Dictionary<string, object> { ["text"] = text })
                .ToList()
        };
    }

    // Adicionar ID da tabela de palavras-chave da plataforma
    if (!string.IsNullOrEmpty(Config.BoostingTableId))
    {
        corpus["boosting_table_id"] = Config.BoostingTableId;
    }

    // Só adiciona ao pedido quando corpus não está vazio
    if (corpus.Count > 0)
    {
        request["corpus"] = corpus;
    }
}
```

Este código mostra como construir dinamicamente o campo corpus com base na configuração. O ponto chave é: só quando realmente existe configuração de palavras-chave, adicionaremos o campo corpus. Este design garante compatibilidade reversa - quando nenhuma palavra-chave está configurada, a estrutura do pedido é completamente consistente com a anterior. Afinal, compatibilidade é importante, não podemos estragar a lógica anterior apenas por adicionar uma funcionalidade.

### Transmissão de Parâmetros WebSocket

Entre frontend e backend, os parâmetros de palavras-chave são transmitidos através de mensagens de controle WebSocket. O design do HagiCode é: o frontend carrega a configuração de palavras-chave do localStorage quando começa a gravar, e então a envia para o backend através de mensagens WebSocket.

```typescript
const controlMessage = {
  type: 'control',
  payload: {
    command: 'StartRecognition',
    contextText: '高血压\n糖尿病\n冠心病',
    boosting_table_id: 'medical_table',
    combineMode: false
  }
};
```

Aqui há um detalhe que precisa de atenção: o frontend passa texto de múltiplas linhas (separado por caracteres de nova linha), o backend precisa analisar. O Handler WebSocket do backend analisará esses parâmetros e os passará para o SDK:

```csharp
private async Task HandleControlMessageAsync(
    string connectionId,
    DoubaoSession session,
    ControlMessage message)
{
    if (message.Payload is SessionControlRequest controlRequest)
    {
        // Analisar parâmetros de palavras-chave
        string? contextText = controlRequest.ContextText;
        string? boostingTableId = controlRequest.BoostingTableId;
        bool? combineMode = controlRequest.CombineMode;

        // Analisar texto de múltiplas linhas em lista de palavras-chave
        if (!string.IsNullOrEmpty(contextText))
        {
            var hotwords = contextText
                .Split('\n', StringSplitOptions.RemoveEmptyEntries)
                .Select(s => s.Trim())
                .Where(s => s.Length > 0)
                .ToList();

            session.HotwordContexts = hotwords;
        }

        session.BoostingTableId = boostingTableId;
    }
}
```

Através deste design, a transmissão da configuração de palavras-chave do frontend para o backend torna-se clara e eficiente. Na verdade, não há nada de especial, é apenas passar camada por camada.

Testes reais mostram que, após configurar palavras-chave, a precisão de reconhecimento de terminologia profissional relacionada melhorou significativamente. Como dizer, o efeito existe, pelo menos muito melhor do que antes.

### Configurar Tabela de Palavras-chave da Plataforma

Se você precisa gerenciar um grande número de palavras-chave, ou as palavras-chave precisam ser atualizadas com frequência, então o modo de tabela de palavras-chave da plataforma é mais adequado para você. Primeiro você precisa criar uma tabela de palavras-chave na plataforma de autoaprendizado Doubao, obter o boosting_table_id gerado, e então inserir este ID na página de configurações do HagiCode.

A plataforma de autoaprendizado Doubao fornece funções como importação em lote de palavras-chave, gerenciamento categorizado, etc., o que é muito prático para equipes que precisam gerenciar um grande número de termos profissionais. Através da plataforma gerenciar palavras-chave, pode-se realizar manutenção centralizada e atualização unificada de palavras-chave. Afinal, quando há muitas palavras-chave, ter um lugar para gerenciamento centralizado é melhor do que ter que inserir manualmente cada vez.

### Uso do Modo de Combinação

Em alguns cenários complexos, você pode precisar usar palavras-chave personalizadas e tabela de palavras-chave da plataforma ao mesmo tempo. Neste momento, você só precisa configurar os dois tipos de palavras-chave no HagiCode e ativar o interruptor "Modo de Combinação".

No modo de combinação, a API Doubao considerará ambas as fontes de palavras-chave, e a precisão de reconhecimento geralmente é maior do que usar qualquer uma delas sozinha. No entanto, é preciso notar que o modo de combinação aumentará a complexidade do pedido, recomenda-se decidir se deve habilitar após testes reais. Afinal, a complexidade aumentou, se realmente vale a pena, ainda depende do efeito real.

## Precauções

Ao implementar e usar a funcionalidade de palavras-chave, há alguns pontos que precisam de atenção especial.

Primeiro são as limitações de caracteres. A API Doubao tem limitações estritas em palavras-chave, incluindo número de linhas, caracteres por linha, número total de caracteres, etc. Se exceder as limitações, a API retornará um erro. Na implementação frontend do HagiCode, verificamos durante a fase de entrada do usuário através de funções de validação, evitando enviar configurações inválidas para o backend. Afinal, descobrir problemas antecipadamente é melhor do que esperar a API retornar um erro.

Em segundo lugar está o formato do boosting_table_id. Este campo só permite letras, números, sublinhados e hífens, não permite conter espaços ou outros caracteres especiais. Ao criar tabelas de palavras-chave na plataforma de autoaprendizado Doubao, é preciso prestar atenção às convenções de nomenclatura. Na verdade, nem é para menos, os requisitos de formato da API são sempre relativamente estritos.

Terceiro é a compatibilidade reversa. Os parâmetros de palavras-chave são completamente opcionais, quando nenhuma palavra-chave está configurada, a forma como o sistema funciona é completamente consistente com a anterior. Este design garante que usuários existentes não sejam afetados de forma alguma, e também facilita migração e atualização graduais. Afinal, não podemos estragar a lógica anterior apenas por adicionar uma funcionalidade.

Finalmente está o tratamento de erros. Quando a configuração de palavras-chave é inválida, a API Doubao retornará as informações de erro correspondentes. A implementação do HagiCode registrará logs detalhados, facilitando para os desenvolvedores investigarem problemas. Ao mesmo tempo, o frontend também exibirá erros de validação na interface, ajudando os usuários a corrigir a configuração. Quando o tratamento de erros é bem feito, a experiência do usuário naturalmente também fica boa.
