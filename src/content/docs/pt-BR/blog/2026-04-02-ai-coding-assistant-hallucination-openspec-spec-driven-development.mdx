## Alucinações de Assistentes de Programação de IA: Como Implementar Desenvolvimento Orientado a Especificações com OpenSpec

> Assistentes de programação de IA são poderosos, mas frequentemente geram código que não atende aos requisitos reais e viola as convenções do projeto. Este artigo compartilha como o projeto HagiCode implementa "desenvolvimento orientado a especificações" através do processo OpenSpec, usando um mecanismo de propostas estruturado para reduzir significativamente os riscos de alucinações de IA.

## Contexto

Quem já usou GitHub Copilot ou ChatGPT para escrever código provavelmente já teve essa experiência: o código gerado pela IA parece muito bonito, mas quando você realmente usa, está cheio de problemas. Pode ser que usou incorretamente algum componente do projeto, pode ter ignorado as convenções de codificação da equipe, ou pode ter escrito grandes blocos de lógica baseados em suposições que não existem.

Este é o chamado problema de "alucinação de IA" - no campo de programação, ele se manifesta como código que parece razoável, mas na realidade não corresponde à situação real do projeto.

Na verdade, isso é bastante inevitável. Assistentes de programação de IA estão se tornando cada vez mais populares, e esse problema está se tornando cada vez mais grave. Afinal, a IA carece de compreensão do histórico do projeto, decisões de arquitetura e convenções de codificação, e um grau de liberdade muito alto facilita que ela gere código de forma "criativa" que não corresponde à realidade. Isso é bastante parecido com escrever artigos - sem método, é fácil escrever de forma desorganizada, quando na realidade não é bem assim.

Para resolver essas dores, tomamos uma decisão ousada: em vez de tentar tornar a IA mais inteligente, colocamos uma "especificação" ao redor dela. As mudanças que essa decisão trouxe podem ser maiores do que você imagina - falarei mais sobre isso em breve.

## Sobre HagiCode

A solução compartilhada neste artigo vem da nossa experiência prática no projeto [HagiCode](https://hagicode.com). HagiCode é um projeto de assistente de código de IA de código aberto, dedicado a resolver problemas práticos na programação de IA através de práticas de engenharia estruturadas.

## A Raiz das Alucinações de IA

Antes de mergulharmos na solução, vejamos primeiro onde está o problema. Afinal, conheça a si mesmo e ao seu inimigo, e você nunca perderá uma batalha - embora aplicar isso à IA também seja um pouco interessante.

### Falta de Contexto

Os modelos de IA são treinados com repositórios de código públicos, mas seu projeto tem seu próprio histórico, convenções e decisões de arquitetura. A IA não pode acessar diretamente esse "conhecimento tácito", então o código gerado frequentemente se desvincula da situação real do projeto.

Isso não é inteiramente culpa da IA - afinal, ela nunca esteve no seu projeto, como ela saberia quais são as regras não escritas? É como um estagiário recém-chegado, não entender as regras é normal, apenas o preço pode ser um pouco alto.

### Grau de Liberdade Excessivo

Quando você pergunta à IA "me ajude a implementar uma funcionalidade de autenticação de usuário", ela pode gerar código de qualquer forma. Sem restrições claras, a IA irá implementar da maneira que ela "acha" razoável, em vez de seguir os requisitos do seu projeto.

É como deixar alguém que nunca aprendeu as convenções do seu projeto agir livremente - como não dar problema? Na verdade, não é que ela é irresponsável, é que ela simplesmente não sabe o que é responsabilidade.

### Falta de Validação

Depois que a IA gera código, se não passar por um processo de revisão estruturado, aquele código baseado em suposições erradas entrará diretamente no repositório. Esperar até os testes ou até o ambiente de produção para descobrir problemas, o preço é muito alto.

Isso é como fechar a porta do estábulo depois que o cavalo já fugiu - só que os cavalos já fugiram todos, de que adianta fechar a porta? Todos entendem essa lógica, mas quando realmente precisamos fazer, sempre parece chato. Afinal, antes que as coisas deem errado, quem quer gastar mais tempo?

## OpenSpec: A Resposta para o Desenvolvimento Orientado a Especificações

HagiCode escolheu OpenSpec como solução, com a ideia central: **todas as mudanças de código devem passar por um processo de propostas estruturado**, transformando ideias abstratas em planos de implementação executáveis.

Isso parece muito sofisticado, mas na verdade é apenas fazer a IA escrever o documento de requisitos antes de escrever código. Afinal, para cada tarefa, o planejamento antecipado leva ao sucesso, e a falta de planejamento leva ao fracasso - os antigos não me enganaram.

### O que é OpenSpec

OpenSpec é uma ferramenta de linha de comando baseada em npm (`@fission-ai/openspec`), que define uma estrutura padrão de arquivos de proposta e mecanismo de validação. Simplificando, é fazer com que a IA "escreva o documento de requisitos" antes de escrever código.

### Processo de Três Passos Contra Alucinações

OpenSpec usa um processo de três passos para garantir a qualidade das propostas:

**Passo 1: Inicializar proposta** - Define o estado da sessão como Openspecing
**Passo 2: Processamento intermediário** - Mantém o estado Openspecing, aprimorando gradualmente os artefatos
**Passo 3: Completar proposta** - Transição para o estado Reviewing

Este design tem um detalhe muito engenhoso: o primeiro passo usa o tipo `ProposalGenerationStart`, e quando completado não aciona a transição de estado. Isso garante que antes de completar todo o fluxo de múltiplos passos, não entre prematuramente na fase de revisão.

Na verdade, esse detalhe é bastante interessante - é como cozinhar, se você tirar a tampa antes da hora certa, definitivamente não vai dar certo. Só com paciência, passo a passo, você pode finalmente fazer um bom prato.

```csharp
// Implementação no projeto HagiCode
public enum MessageAssociationType
{
    ProposalGeneration = 2,
    ProposalExecution = 3,

    /// <summary>
    /// Marca o início do fluxo de geração de propostas de três passos
    /// Quando completado não transiciona para o estado Reviewing
    /// </summary>
    ProposalGenerationStart = 5
}
```

### Estrutura de Arquivo Padronizada

Cada proposta OpenSpec segue a mesma estrutura de diretórios:

```
openspec/
├── changes/                    # Mudanças ativas e arquivadas
│   ├── {change-name}/
│   │   ├── proposal.md        # Descrição da proposta
│   │   ├── design.md          # Documento de design
│   │   ├── specs/             # Especificações técnicas
│   │   └── tasks.md           # Lista de tarefas executáveis
│   └── archive/               # Mudanças arquivadas
└── specs/                     # Biblioteca de especificações independentes
```

De acordo com as estatísticas do projeto HagiCode, atualmente já existem mais de 4000 mudanças arquivadas e mais de 150 mil linhas de arquivos de especificação. Esse acúmulo histórico não apenas dá à IA algo para seguir, mas também fornece uma base de conhecimento valiosa para a equipe.

É como os clássicos deixados pelos antigos - lendo muito, você naturalmente pode perceber alguns princípios. Só que agora esses clássicos não estão escritos em bambu, mas armazenados em arquivos.

### Mecanismo de Validação em Múltiplas Camadas

O sistema implementa validação em múltiplas camadas para garantir a qualidade das propostas:

```csharp
// Valida a existência de arquivos obrigatórios
ValidateProposalFiles()

// Valida pré-condições de execução
ValidateExecuteAsync()

// Valida condições de inicialização
ValidateStartAsync()

// Valida condições de arquivamento
ValidateArchiveAsync()

// Validação do formato do nome da proposta (kebab-case)
ValidateNameFormat()
```

Essas validações são como guardiões em camadas - apenas propostas verdadeiramente qualificadas podem passar. Embora pareça tedioso, é melhor do que deixar código ruim entrar no repositório, certo?

### Restrições de Template de Prompt

A IA no HagiCode usa templates Handlebars predefinidos durante a execução, que incluem orientações claras de passos e medidas de proteção. Por exemplo:

- Proibido de continuar sem entender a intenção do usuário
- Proibido de gerar código não validado
- Exige fornecer novamente quando o nome é inválido
- Se a mudança já existe, sugere usar o comando continue em vez de recriar

Essa maneira de "dançar com algemas" faz com que a IA se concentre mais em entender os requisitos e gerar código que esteja em conformidade com as especificações. Na verdade, restrições não são necessariamente ruins - afinal, liberdade excessiva facilmente leva à bagunça.

## Prática: Como Usar OpenSpec em Projetos

### Instalação e Inicialização

```bash
npm install -g @fission-ai/openspec@1
openspec --version  # Verificar instalação
```

A estrutura de pasta `openspec/` será criada automaticamente no diretório raiz do projeto.

Este passo na verdade não tem muito o que dizer - instalar ferramentas, todo mundo entende. Apenas lembre-se de usar a versão `@fission-ai/openspec@1`, novas versões podem ter problemas - afinal, estabilidade acima de tudo.

### Criar Proposta

Na interface de conversação do HagiCode, use o comando de atalho:

```
/opsx:new
```

Ou especifique o nome da mudança e o repositório de destino:

```
/opsx:new "add-user-auth" --repos "repos/web"
```

Criar uma proposta é como fazer um esboço antes de escrever um artigo - com o esboço, o resto fica mais fácil de escrever. Só que muitas pessoas gostam de começar a escrever diretamente, e no meio do caminho descobrem que o raciocínio não funciona - isso é que dá dor de cabeça.

### Gerar Artefatos

Use `/opsx:continue` para gerar gradualmente os artefatos necessários:

**proposal.md** - Descreve o propósito e escopo da mudança
```markdown
# Proposal: Add User Authentication

## Why
O sistema atual carece de funcionalidade de autenticação de usuário, incapaz de proteger APIs sensíveis.

## What Changes
- Adicionar middleware de autenticação JWT
- Implementar APIs de login/registro
- Atualizar integração frontend
```

**design.md** - Design técnico detalhado
```markdown
# Design: Add User Authentication

## Context
Atualmente usando API pública, acessível a qualquer pessoa...

## Decisions
1. Escolher JWT em vez de Session...
2. Usar algoritmo HS256...

## Risks
- Risco de vazamento de token...
- Medidas de mitigação...
```

**specs/** - Especificações técnicas e cenários de teste
```markdown
# user-auth Specification

## Requirements

### Requirement: JWT Token Generation
O sistema SHALL usar o algoritmo HS256 para gerar tokens JWT.

#### Scenario: Login válido
- WHEN O usuário fornece credenciais válidas
- THEN O sistema SHALL retornar um token JWT válido
```

**tasks.md** - Lista de tarefas executáveis
```markdown
# Tasks: Add User Authentication

## 1. Backend Changes
- [ ] 1.1 Criar AuthController
- [ ] 1.2 Implementar middleware JWT
- [ ] 1.3 Adicionar testes unitários
```

Esses artefatos são como rascunhos ao escrever artigos - quando o rascunho está pronto, o texto principal flui naturalmente. Só que muitas pessoas não gostam de escrever rascunhos, achando que é perda de tempo, mas na verdade o rascunho é onde você melhor esclarece seu raciocínio.

### Revisar e Aplicar

Depois de completar todos os artefatos:

```
/opsx:apply
```

A IA lerá todos os arquivos de contexto e executará as tarefas gradualmente de acordo com a lista em tasks.md. Neste momento, com especificações claras, a qualidade do código gerado será muito maior.

Na verdade, neste ponto, as coisas já estão halfway completas. Com uma lista de tarefas clara, o resto é apenas executar passo a passo. Só que muitas pessoas pulam as etapas anteriores e vêm direto para cá - então a qualidade naturalmente é difícil de garantir.

### Arquivar

Após completar a mudança:

```
/opsx:archive
```

Move as mudanças concluídas para o diretório `archive/`, facilitando consulta e reutilização futuras.

Arquivar é bastante importante - é como guardar adequadamente um artigo que você escreveu. Quando encontrar problemas similares no futuro, consulte os registros anteriores - pode haver uma resposta. Só que muitas pessoas são preguiçosas para fazer isso, achando chato, mas na verdade esses acúmulos são a riqueza mais valiosa.

## Notas e Melhores Práticas

### Convenções de Nomenclatura de Propostas

Use formato kebab-case, começando com uma letra, contendo apenas letras minúsculas, números e hífens:
- ✅ `add-user-auth`
- ❌ `AddUserAuth`
- ❌ `add--user-auth`

Convenções de nomenclatura, em suma, não é grande coisa - só que um pouco de uniformidade é bom. Afinal, em código, consistência é importante - só que muitas pessoas não se importam com isso.

### Evitar Erros Comuns

1. **Usar o tipo errado no passo 1 do fluxo de três passos** - Irá transicionar o estado prematuramente
2. **Esquecer de acionar a transição de estado no último passo** - Ficará preso no estado Openspecing
3. **Pular a revisão e executar diretamente** - Deve validar primeiro que todos os artefatos estão completos

Esses erros são fáceis para iniciantes cometerem - veteranos naturalmente sabem como evitá-los. Só que iniciantes sempre se tornarão veteranos um dia - pegar atalhos é aceitável, só espero que não peguem muitos atalhos.

### Gerenciamento de Múltiplas Mudanças

OpenSpec suporta gerenciar múltiplas propostas simultaneamente, o que é particularmente útil ao lidar com funcionalidades grandes:

```bash
# Ver todas as mudanças ativas
openspec list

# Alternar para uma mudança específica
openspec apply "add-user-auth"

# Ver status da mudança
openspec status --change "add-user-auth"
```

Gerenciar múltiplas mudanças é como escrever vários artigos ao mesmo tempo - requer um pouco de habilidade e paciência. Só que você se acostuma - afinal, as pessoas sempre se adaptam.

### Compreensão da Máquina de Estados de Sessão

Entender as transições de estado ajuda a solucionar problemas:

```
Init → Drafting → Openspecing → Reviewing → Executing → ExecutionCompleted → Completed → Archived
```

- **Openspecing**: Gerando planejamento
- **Reviewing**: Em revisão (pode modificar artefatos repetidamente)
- **Executing**: Em execução (aplicando tasks.md)

Máquina de estados, em suma, é apenas um conjunto de regras. Regras às vezes são irritantes, mas mais frequentemente são úteis. Afinal, sem regras não há ordem - os antigos já disseram isso.

## Conclusão

Através do processo OpenSpec, o projeto HagiCode alcançou resultados significativos na resolução do problema de alucinações de IA:

1. **Reduzir alucinações** - A IA deve seguir especificações estruturadas, não pode gerar código livremente
2. **Melhorar qualidade** - Validação em múltiplas camadas garante que mudanças estejam em conformidade com os padrões do projeto
3. **Acelerar colaboração** - Mudanças arquivadas fornecem referência para desenvolvimento subsequente
4. **Rastreabilidade** - Cada mudança tem registros completos de proposta, design, especificações e tarefas

Esta solução não torna a IA mais inteligente, mas coloca uma "especificação" ao redor dela. A prática prova que dançar com algemas, na verdade, dança-se melhor.

Na verdade, essa lógica é simples - restrições não são necessariamente ruins. É como escrever artigos, com restrições de formato, é mais fácil escrever coisas boas. Só que muitas pessoas não gostam de restrições, sentindo que limitam sua criatividade, mas na verdade a criatividade também precisa de solo para florescer e dar frutos.

Se você também está usando assistentes de programação de IA e encontrou problemas similares, experimente OpenSpec. Desenvolvimento orientado a especificações pode parecer adicionar algumas etapas, mas esse investimento inicial será recompensado muitas vezes na qualidade do código e eficiência de manutenção.

Afinal, ir devagar às vezes é ir mais rápido - só que muitas pessoas não entendem essa lógica...
