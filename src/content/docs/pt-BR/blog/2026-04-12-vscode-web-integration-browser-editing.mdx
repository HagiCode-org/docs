## Edição rápida de código no navegador: Prática de integração VSCode Web

> Após a IA analisar o código, como abrir o editor no navegador imediatamente para fazer modificações? Este artigo compartilha a experiência prática de integração do code-server no projeto HagiCode, alcançando uma conexão perfeita entre o assistente de IA e a experiência de edição de código.

## Contexto

Na era da programação assistida por IA, os desenvolvedores frequentemente precisam visualizar e editar código rapidamente. O fluxo de desenvolvimento tradicional é: abrir o projeto em uma IDE desktop, localizar o arquivo, editar e salvar. Mas em alguns cenários, este fluxo parece sempre um pouco inadequado.

Cenário 1: Desenvolvimento remoto. Ao usar um assistente de IA como o HagiCode, o backend pode estar rodando em um servidor remoto ou container, e o local não pode acessar diretamente os arquivos do projeto. Cada vez que você precisa visualizar ou modificar código, é necessário conectar via SSH ou outros meios, criando uma experiência fragmentada. É como querer encontrar alguém, mas com uma camada grossa de vidro no meio - você pode ver, mas não pode tocar.

Cenário 2: Visualização rápida. Depois que o assistente de IA analisa o código, o usuário pode apenas querer navegar rapidamente por um arquivo ou fazer pequenas modificações. Iniciar uma IDE desktop completa parece pesado, e um editor leve no navegador é mais adequado para a necessidade de "visualização rápida". Afinal, quem quer fazer uma tempestade em copo d'água só para dar uma olhada?

Cenário 3: Colaboração entre dispositivos. Ao trabalhar em diferentes dispositivos, o editor no navegador fornece um ponto de acesso unificado, sem a necessidade de configurar o ambiente de desenvolvimento em cada dispositivo. Isso é econômico - afinal, a vida é curta, por que repetir o trabalho?

Para resolver essas dores, integramos o VSCode Web no projeto HagiCode. Permitindo uma conexão perfeita entre o assistente de IA e a experiência de edição de código - após a IA analisar o código, os usuários podem abrir imediatamente o editor na mesma sessão do navegador para fazer modificações, sem alternar aplicativos. Esta experiência, como dizer, quando você precisa, está lá.

O HagiCode é um assistente de código impulsionado por IA, projetado para melhorar a eficiência do desenvolvimento através de interação em linguagem natural. Durante o desenvolvimento, descobrimos que os usuários frequentemente precisam alternar rapidamente entre a análise da IA e a edição de código, o que nos levou a explorar como integrar o editor diretamente no navegador.

## Seleção tecnológica: Por que code-server?

Entre as diversas soluções VSCode Web, escolhemos o code-server. Esta escolha, na verdade, tem algumas considerações:

Funcionalidade completa. O code-server é a versão Web do VSCode, suportando a maioria das funcionalidades da versão desktop, incluindo sistema de extensões, sugestões inteligentes, depuração, etc. Isso significa que os usuários também podem obter uma experiência de edição próxima à versão desktop no navegador. Afinal, quem quer fazer concessões em funcionalidades?

Implantação flexível. O code-server pode ser executado como um serviço independente, também suporta implantação em container Docker, encaixando-se na arquitetura do HagiCode. Nosso backend é escrito em C#, o frontend é React, comunicando-se com o serviço code-server via API REST. É como montar blocos - cada um tem seu próprio lugar.

Segurança de autenticação. O code-server possui um mecanismo interno de connection-token, prevenindo acesso não autorizado. Cada sessão possui um token único, garantindo que apenas usuários autorizados possam acessar o editor. Segurança - você só percebe a importância quando a tem.

## Design de arquitetura

A integração VSCode Web do HagiCode adota um design de arquitetura de separação entre frontend e backend.

### Camada de serviço frontend

O frontend encapsula a interação com o backend através de `vscodeServerService.ts`:

```typescript
// Abrir projeto
export async function openProjectInCodeServer(
  id: string,
  currentInterfaceLanguage?: string,
): Promise<VsCodeServerLaunchResponseDto>

// Abrir vault
export async function openVaultInCodeServer(
  id: string,
  path?: string,
  currentInterfaceLanguage?: string,
): Promise<VsCodeServerLaunchResponseDto>
```

A diferença entre esses dois métodos é: `openProjectInCodeServer` é usado para abrir o projeto inteiro, enquanto `openVaultInCodeServer` é usado para abrir um caminho específico do Vault. Para projetos mono-repo MonoSpecs, o sistema cria automaticamente arquivos de workspace. Divisão clara de tarefas, cada um fazendo seu próprio trabalho - isso é suficiente.

### Camada de serviço backend

O backend `VaultAppService.cs` implementa a lógica principal:

```csharp
public async Task<VsCodeServerLaunchResponseDto> OpenInCodeServerAsync(
    string id,
    string? relativePath = null,
    string? currentInterfaceLanguage = null,
    CancellationToken cancellationToken = default)
{
    // 1. Obter configurações e verificar se está habilitado
    var settings = await _vsCodeServerSettingsService.GetResolvedSettingsAsync(cancellationToken);
    if (!settings.Enabled) {
        throw new BusinessException(VsCodeServerErrorCodes.Disabled, "VSCode Server is disabled.");
    }

    // 2. Obter vault e resolver diretório de inicialização
    var vault = await RequireVaultAsync(id, cancellationToken);
    var launchDirectory = ResolveLaunchDirectory(vault, relativePath);

    // 3. Garantir que o code-server esteja rodando e obter informações de runtime
    var runtime = await _vsCodeServerManager.EnsureStartedAsync(settings, cancellationToken);

    // 4. Resolver configurações de idioma
    var language = _vsCodeServerSettingsService.ResolveLaunchLanguage(
        settings.Language,
        currentInterfaceLanguage);

    // 5. Construir URL de inicialização
    return new VsCodeServerLaunchResponseDto {
        LaunchUrl = AppendQueryString(runtime.BaseUrl, new Dictionary<string, string?> {
            ["folder"] = launchDirectory,
            ["tkn"] = runtime.ConnectionToken,
            ["vscode-lang"] = language,
        }),
        ConnectionToken = runtime.ConnectionToken,
        OpenMode = "folder",
        Runtime = VsCodeServerSettingsService.MapRuntime(
            await _vsCodeServerManager.GetRuntimeSnapshotAsync(cancellationToken)),
    };
}
```

As responsabilidades deste método são claras: verificar configurações, resolver caminhos, iniciar serviço, construir URL. O método `ResolveLaunchDirectory` realiza verificações de segurança de caminho, prevenindo ataques de path traversal. Código é como poesia - cada linha tem seu próprio ritmo.

### Gerenciamento automático de runtime

O backend gerencia o processo code-server através do `VsCodeServerManager`:

- Verificar status do processo
- Iniciar automaticamente serviços parados
- Retornar snapshot de runtime (porta, ID do processo, tempo de inicialização, etc.)

Este design permite que o sistema processe automaticamente o ciclo de vida do code-server, sem que os usuários precisem gerenciar manualmente os processos do serviço. Afinal, a vida já é complexa o suficiente - automatize o que for possível.

### Mecanismo de acompanhamento de idioma

O HagiCode suporta interface multilíngue, e o code-server também precisa seguir esta configuração. O sistema suporta três modos de idioma:

- `follow`: Seguir o idioma atual da interface
- `zh-CN`: Chinês fixo
- `en-US`: Inglês fixo

Através do parâmetro URL `vscode-lang` passado para o code-server, garantindo que o idioma do editor permaneça consistente com a interface do HagiCode. Idioma - quando unificado, fica mais confortável.

Para projetos MonoSpecs (mono-repo contendo múltiplos sub-repositórios), o sistema cria automaticamente arquivos `.code-workspace`:

Isso permite editar múltiplos sub-repositórios simultaneamente em uma instância code-server, muito prático para grandes projetos mono-repo. Múltiplos repositórios em uma janela - como múltiplas histórias no mesmo livro.

## Integração frontend

O frontend do HagiCode usa React + TypeScript, e os passos para integrar o code-server não são complexos.

### Botões de ação rápida

Adicionar botão Code Server no card do projeto:

```tsx
// QuickActionsZone.tsx
<Button
  size="sm"
  variant="default"
  onClick={() => onAction({ type: 'open-code-server' })}
>
  <Globe className="h-3 w-3 mr-1" />
  <span className="text-xs">{t('project.openCodeServer')}</span>
</Button>
```

Este botão aciona a ação de abertura, chamando a API backend para obter a URL de inicialização. Um botão, carregando uma ação - simples e direto.

### Tratamento da ação de abertura

```tsx
const handleAction = async (action: ProjectAction) => {
  if (action.type === 'open-code-server') {
    const response = await openProjectInCodeServer(project.id, i18n.language);
    window.open(response.launchUrl, '_blank', 'noopener,noreferrer');
  }
};
```

Usar `window.open` para abrir o code-server em uma nova aba, os parâmetros `noopener,noreferrer` garantem segurança. Segurança - nunca é demais cuidar.

### Ponto de entrada de edição Vault

Adicionar botão de edição similar na lista Vault:

```tsx
const handleEditVault = async (vault: VaultItemDto) => {
  const response = await openVaultInCodeServer(vault.id);
  window.open(response.launchUrl, '_blank', 'noopener,noreferrer');
};
```

Projeto e Vault usam a mesma forma de abertura, mantendo a consistência da interação. A importância da consistência não é inferior à da funcionalidade em si.

## Lógica de construção de URL

O formato de URL do code-server, na verdade, tem seus detalhes:

**Modo pasta**:
```
http://{host}:{port}/?folder={path}&tkn={token}&vscode-lang={lang}
```

**Modo workspace**:
```
http://{host}:{port}/?workspace={workspacePath}&tkn={token}&vscode-lang={lang}
```

Onde `tkn` é o token de conexão, gerado automaticamente a cada inicialização do code-server, garantindo acesso seguro. O parâmetro `vscode-lang` controla o idioma da interface do editor. Cada parâmetro tem seu propósito, nenhum é dispensável.

## Considerações de segurança

Ao implementar a integração code-server, segurança é uma questão que precisa de atenção especial. Afinal, em segurança, quando problemas aparecem, é tarde demais.

### Connection Token

O connection-token é gerado aleatoriamente e não deve ser vazado. Recomenda-se usar em ambiente HTTPS para evitar que o token seja interceptado por ataques man-in-the-middle. Informações sensíveis - é melhor protegê-las.

### Segurança de caminho de arquivo

O backend implementa verificação de path traversal:

```csharp
private static string ResolveLaunchDirectory(VaultRegistryEntry vault, string? relativePath)
{
    var vaultRoot = EnsureTrailingSeparator(Path.GetFullPath(vault.PhysicalPath));
    var combinedPath = Path.GetFullPath(Path.Combine(vaultRoot, relativePath ?? "."));
    if (!combinedPath.StartsWith(vaultRoot, StringComparison.OrdinalIgnoreCase))
    {
        throw new BusinessException(VaultRelativePathTraversalCode, "Relative path traversal detected.");
    }
    return combinedPath;
}
```

Este código garante que os usuários não possam acessar arquivos fora do diretório vault através de `../` etc. Verificação de limites - melhor fazer do que não fazer.

### Controle de permissões

O processo code-server é executado com permissões de usuário apropriadas, evitando acesso a arquivos sensíveis do sistema. Recomenda-se usar um usuário dedicado para executar o serviço code-server. Controle de permissões - o necessário deve ter.

## Otimização de performance

O code-server consome recursos do servidor, aqui estão algumas sugestões de otimização:

- Monitorar uso de CPU/memória, ajustar limites de recursos quando necessário
- Projetos grandes podem precisar aumentar o tempo de timeout
- Implementar limpeza automática de timeout de sessão, liberando recursos
- Considerar usar cache para reduzir cálculos repetidos

O HagiCode fornece API de monitoramento de status de runtime, e o frontend pode obter o status atual através de `getVsCodeServerSettings()`:

```typescript
const { settings, runtime } = await getVsCodeServerSettings();
// runtime.status: 'disabled' | 'stopped' | 'starting' | 'running' | 'unhealthy'
// runtime.baseUrl: "http://localhost:8080"
// runtime.processId: 12345
```

Este design permite que os usuários entendam claramente o status de saúde do code-server e localizem rapidamente problemas quando eles ocorrem. Status visível - assim sabemos onde estamos.

## Detalhes de experiência do usuário

Durante a implementação, descobrimos alguns detalhes que afetam a experiência do usuário e que merecem atenção especial:

A primeira abertura do code-server pode precisar esperar a inicialização, este tempo pode variar de alguns segundos a meio minuto. Recomenda-se mostrar status de carregamento no frontend, para que os usuários saibam que o sistema está processando. Esperar - ter feedback já ajuda.

Navegadores podem bloquear pop-ups, precisando alertar os usuários para permitir manualmente. O HagiCode mostra informações de guia na primeira abertura, dizendo aos usuários como configurar permissões do navegador. Experiência do usuário - é nesses detalhes que se manifesta.

Recomenda-se mostrar o status de runtime (iniciando/rodando/erro), para que os usuários possam saber se é um problema do servidor ou seu próprio problema de operação quando encontram problemas. Saber onde está o problema - pelo menos temos uma base.

## Exemplo de configuração

A configuração do code-server não é complexa:

```json
{
  "vscodeServer": {
    "enabled": true,
    "host": "0.0.0.0",
    "port": 8080,
    "language": "follow"
  }
}
```

`enabled` controla o interruptor de funcionalidade, `host` e `port` especificam o endereço de escuta, `language` define o modo de idioma. Estas configurações podem ser modificadas através da interface UI, entrando em vigor em tempo real. Coisas simples - frequentemente são as mais úteis.

A integração VSCode Web do HagiCode fornece uma solução elegante: conectar perfeitamente o assistente de IA com a experiência de edição de código.
