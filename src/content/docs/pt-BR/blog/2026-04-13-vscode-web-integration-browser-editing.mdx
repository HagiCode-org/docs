## VSCode e code-server: escolha de solução de edição de código no navegador

> Ao construir capacidades de edição de código no navegador, os desenvolvedores enfrentam uma escolha crítica: usar a funcionalidade `code serve-web` oficial do VSCode ou adotar a solução `code-server` impulsionada pela comunidade? Esta escolha não afeta apenas a arquitetura técnica, mas também está relacionada à conformidade de licença e flexibilidade de implantação.

## Contexto

Na verdade, fazer escolhas tecnológicas é um pouco como escolher caminhos na vida. Você escolhe um caminho e tem que continuar seguindo-o; quando quer mudar, o custo é alto.

Na era da programação assistida por IA, a capacidade de editar código no navegador torna-se cada vez mais importante. Os usuários esperam que, depois que um assistente IA analisar o código, possam abrir imediatamente o editor na mesma sessão do navegador para fazer modificações, sem precisar alternar aplicativos. Esta experiência perfeita, bem, é como quando você quer algo, ele está lá – só que às vezes ele simplesmente não está.

No entanto, ao implementar esta funcionalidade, os desenvolvedores enfrentam uma escolha tecnológica crítica: usar a funcionalidade `code serve-web` oficial do VSCode ou adotar a solução `code-server` impulsionada pela comunidade?

Ambas as soluções têm seus prós e contras, e escolher a errada pode trazer muitos problemas mais tarde. Por exemplo, problemas de licença – descobrir que a licença não está em conformidade apenas quando o produto já está no ar, isso é tarde demais. É um pouco como namorar; se você não pensar claramente no início, no final descobre que os valores de duas pessoas são fundamentalmente incompatíveis, e o preço pago é alto. Outro exemplo é a forma de implantação – uma solução funciona bem no ambiente de desenvolvimento, mas quando entra no container surgem vários problemas; ninguém quer cair nessas armadilhas, afinal, depois de cair em muitas armadilhas, a pessoa fica insensível.

A solução compartilhada neste artigo vem da nossa experiência prática no projeto [HagiCode](https://hagicode.com). HagiCode é um assistente de código impulsionado por IA; ao implementar capacidades de edição de código no navegador, pesquisamos profundamente ambas as soluções e, finalmente, no design da arquitetura, suportamos ambas simultaneamente, mas por padrão priorizamos o code-server.

Endereço do projeto: [github.com/HagiCode-org/site](https://github.com/HagiCode-org/site)

## Diferenças de licença (o mais crítico)

Esta é a diferença mais fundamental entre as duas soluções e o primeiro fator que consideramos na seleção. Afinal, em seleção tecnológica, o primeiro passo é pensar claramente sobre os riscos legais; caso contrário, se algo der errado depois, de quem será a culpa?

**code-server**
- Licença MIT, completamente open source
- Mantido pela empresa Coder.com, comunidade ativa
- Pode ser usado comercialmente, modificado e distribuído livremente
- Sem restrições de cenários de uso

**VSCode `code serve-web`**
- Faz parte do produto Microsoft VSCode
- Usa a licença Microsoft (a licença do VS Code tem restrições de uso comercial)
- Voltado principalmente para desenvolvedores individuais
- Implantação empresarial pode precisar considerar autorizações comerciais adicionais

Do ponto de vista da licença, o code-server é mais amigável para projetos comerciais. Isso precisa ser pensado claramente na fase de planejamento do produto; caso contrário, esperar até que a escala cresça para então migrar, o custo será muito alto. Afinal, migração é fácil de falar, difícil de fazer; quem já passou por isso, sabe.

## Diferenças na forma de implantação

Depois de resolver os problemas de licença, o seguinte é a forma de implantação. Isso afeta diretamente seus custos operacionais e design de arquitetura, e também afeta indiretamente seu humor diário – quanto mais simples a implantação, melhor o humor, todo mundo entende isso.

**code-server**
- Aplicativo Node.js independente, pode ser implantado separadamente
- Suporta várias origens de runtime:
  - Especificar diretamente o caminho do executável
  - Pesquisa no PATH do sistema
  - Detecção automática do ambiente NVM Node.js 22.x
- Não precisa instalar o VSCode desktop no servidor
- Implantação em container mais simples

**VSCode `code serve-web`**
- Deve depender do VSCode CLI instalado localmente
- Precisa ter o comando `code` disponível na máquina local
- O sistema filtra o wrapper do VS Code Remote CLI
- Projetado principalmente para cenários de desenvolvimento local

O code-server é mais adequado para cenários de implantação em servidor/container. Se seu produto precisa rodar no Docker, ou o ambiente do usuário não tem VSCode, então escolher code-server é a decisão certa. Afinal, simples é belo; o complexo facilmente traz problemas; problemas precisam ser corrigidos; depois de corrigir podem ser introduzidos novos problemas; este ciclo infinito, quem quer experimentar?

## Diferenças de parâmetros funcionais

As duas soluções também têm algumas diferenças em parâmetros funcionais, embora não grandes, mas no uso real podem trazer alguns problemas. Esses detalhes são como pequenos atritos na vida; não são muitos, mas quando se acumulam, incomodam.

| Recurso | code-server | code serve-web |
|---------|-------------|----------------|
| Caminho base público | `/` (configurável) | `/vscode-server` (fixo) |
| Método de autenticação | `--auth` parâmetro, suporta múltiplos modos | `--connection-token` / `--without-connection-token` |
| Diretório de dados | `{DataDir}/code-server` | `{DataDir}/vscode-serve-web` |
| Telemetria | Desabilitado por padrão `--disable-telemetry` | Depende das configurações do VSCode |
| Verificação de atualizações | Pode ser desabilitada `--disable-update-check` | Depende das configurações do VSCode |

Essas diferenças precisam ser特别注意 na integração. Por exemplo, diferentes caminhos de URL significam que o código frontend precisa processar especificamente. Desenvolvedores sabem que lidar com esses detalhes consome mais tempo, mas não há outra opção; sem fazer, não funciona.

## Diferenças na detecção de disponibilidade

Ao implementar a funcionalidade de alternância de editores, a lógica de detecção de disponibilidade também é diferente. Esta diferença é como as formas de interação entre pessoas; algumas gostam de ir direto ao ponto, outras preferem ser sutis e indiretas.

**code-server**
- Sempre retornado como implementação visível
- Mesmo quando indisponível, é exibido, indicando o estado `install-required`
- Suporta detecção automática do ambiente NVM Node.js 22.x

**code serve-web**
- Visível apenas quando o CLI `code` local é detectado
- Se indisponível, o frontend oculta automaticamente esta opção
- Depende do estado da instalação local do VSCode

Esta diferença afeta diretamente a experiência do usuário. A abordagem do code-server é mais transparente; o usuário sabe que existe esta opção, apenas não está instalada; a abordagem do code serve-web é mais discreta, o usuário pode nem saber que existe esta escolha. Qual abordagem é melhor? Depende do posicionamento do produto; afinal, em experiência do usuário, não há resposta padrão, apenas adequado ou inadequado.

## Arquitetura de dupla implementação do HagiCode

Após análise profunda, o projeto HagiCode adotou uma arquitetura de dupla implementação, suportando ambas as soluções no nível da arquitetura. Não é que temos dificuldade de escolha tecnológica, mas sim que existem necessidades reais. Afinal, no mundo tecnológico, não há escolha absolutamente correta, apenas a mais adequada para si.

### Escolha padrão do code-server

```csharp
// A implementação ativa padrão é code-server
// Se uma activeImplementation explícita foi salva, tenta essa implementação primeiro
// Se a implementação solicitada não estiver disponível, o resolvedor tenta a outra implementação
// Se ocorrer fallback, retorna fallbackReason
```

Escolhemos o code-server por padrão, considerando principalmente a licença e a flexibilidade de implantação. Mas para usuários com ambiente VSCode local, o code serve-web também é uma boa escolha. Afinal, dar aos usuários mais uma escolha é sempre bom; forçar outros a aceitar uma única solução, isso eu não faço.

### Seletor de implementação

O `CodeServerImplementationResolver` é responsável uniformemente por:
- Seleção de implementação durante aquecimento de inicialização
- Seleção de implementação durante leitura de estado
- Seleção de implementação durante abertura de projeto
- Seleção de implementação durante abertura de Vault

Este design permite que o sistema lide flexivelmente com diferentes cenários, e os usuários podem escolher a implementação mais adequada de acordo com seu ambiente. Design flexível gasta mais tempo no início, mas economiza preocupações mais tarde; afinal, ninguém quer modificar código em todo lugar.

### Regras de adaptação do frontend

```typescript
// Quando localCodeAvailable=false, não exibir code serve-web
// Quando localCodeAvailable=true, exibir configuração code serve-web
```

O frontend exibe automaticamente as opções disponíveis com base no ambiente, evitando que usuários vejam funcionalidades inutilizáveis e fiquem confusos. Quando usuários ficam confusos, eles vêm perguntar você; quando fazem muitas perguntas, você fica irritado; quando fica irritado, quer modificar o código; modificar o código pode introduzir bugs; este ciclo vicioso, quem quiser experimentar, que experimente.

## Guia prático

Depois de tanta teoria, o que precisa ser notado na implantação real? Na verdade, não importa quão boa a teoria seja, se não funciona na prática, é inútil; afinal, a prática é o único critério para testar a verdade.

### Recomendações de implantação Docker

Para implantação em container, o code-server é a escolha ideal:

```dockerfile
# Usar diretamente a imagem oficial do code-server
FROM codercom/code-server:latest

# Ou instalar via npm
RUN npm install -g code-server
```

Com uma única camada está pronto, sem precisar instalar adicionalmente o VSCode. Simples é bom; o complexo facilmente comete erros; este princípio se aplica em todo lugar.

### Exemplos de configuração

**Configuração do code-server**
```json
{
  "vscodeServer": {
    "enabled": true,
    "activeImplementation": "code-server",
    "codeServer": {
      "host": "0.0.0.0",
      "port": 8080,
      "executablePath": "",
      "authMode": "none"
    }
  }
}
```

**Configuração do code serve-web**
```json
{
  "vscodeServer": {
    "enabled": true,
    "activeImplementation": "serve-web",
    "serveWeb": {
      "host": "0.0.0.0",
      "port": 8080,
      "executablePath": "/usr/local/bin/code"
    }
  }
}
```

Configuração é chata na primeira vez; depois de configurado, economiza preocupações. É como a vida; investir mais no início torna os dias subsequentes mais fáceis.

### Diferenças na construção de URL

**code-server**
```
http://localhost:8080/?folder=/path/to/project&vscode-lang=pt-BR
```

**code serve-web**
```
http://localhost:8080/vscode-server/?folder=/path/to/project&tkn=xxx&vscode-lang=pt-BR
```

Note as diferenças de caminho e parâmetros; a integração precisa processar separadamente. Os detalhes determinam o sucesso ou fracasso; isso não é falso, falta um parâmetro e a página pode não abrir.

### Alternar implementação

O sistema suporta alternância em tempo de execução; ao alternar, a implementação antiga é automaticamente parada:

```csharp
// VsCodeServerManager processa exclusão mútua automaticamente
// Ao alternar activeImplementation, a implementação antiga não continua em segundo plano
```

Este design permite que usuários tentem diferentes implementações a qualquer momento, encontrando a solução mais adequada. Afinal, a mais adequada é a melhor; as sugestões de outros são apenas para referência; no final, você mesmo precisa experimentar.

### Monitoramento de estado

```typescript
const { settings, runtime } = await getVsCodeServerSettings();

// runtime.activeImplementation: "code-server" | "serve-web"
// runtime.fallbackReason: motivo da alternância
// runtime.status: "running" | "starting" | "stopped" | "unhealthy"
```

Com estado visível, você sabe o que está acontecendo. Quando usuários encontram problemas, podem rapidamente identificar se é um problema do servidor ou seu próprio erro de operação. Quando não conhece o estado, a pessoa facilmente entra em pânico; em pânico, facilmente faz julgamentos errados; uma vez iniciada esta cadeia, não para mais.

## Conclusão

| Dimensão de comparação | code-server | code serve-web | Recomendação |
|------------------------|-------------|----------------|--------------|
| Licença | MIT (amigável para comércio) | Microsoft (com restrições) | code-server |
| Flexibilidade de implantação | Implantação independente | Depende do VSCode local | code-server |
| Adequação para servidor | Projetado para servidores | Principalmente para desenvolvimento local | code-server |
| Containerização | Suporte nativo | Precisa instalar VSCode | code-server |
| Completude funcional | Próximo da versão desktop | Versão oficial completa | code serve-web |
| Atividade de manutenção | Comunidade ativa | Oficial Microsoft | Cada um tem vantagens |

**Estratégia recomendada**: Priorizar o uso do code-server; considerar o code serve-web quando precisar de funcionalidades oficiais completas e tiver ambiente VSCode local.
