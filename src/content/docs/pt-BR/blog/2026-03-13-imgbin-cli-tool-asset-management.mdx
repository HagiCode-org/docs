## Design da Ferramenta CLI ImgBin: Solução de Gerenciamento de Ativos de Imagens do HagiCode

> Este artigo apresenta como construir do zero um pipeline de ativos de imagens automatizável, incluindo design da ferramenta CLI, arquitetura Provider Adapter e estratégias de gerenciamento de metadados.

## Contexto

Na verdade, nem imaginávamos que o gerenciamento de ativos de imagens poderia nos preocupar por tanto tempo.

Durante o desenvolvimento do projeto HagiCode, encontramos um problema aparentemente simples, mas muito difícil: a geração e o gerenciamento de ativos de imagens. Como dizer, é como aquelas coisas da adolescência — na superfície tudo está calmo, mas por baixo há ondas agitadas.

Com o aumento da documentação do projeto e materiais de marketing, foi necessário um grande número de imagens. Algumas dessas imagens precisam ser geradas por IA, outras precisam ser selecionadas de bibliotecas de materiais existentes, e outras ainda precisam ser reconhecidas por IA e anotadas automaticamente. O problema é que esse trabalho há muito tempo é feito com scripts dispersos mais operações manuais — cada vez que uma imagem é gerada, é necessário executar scripts manualmente, organizar metadados manualmente, gerar miniaturas manualmente. Isso já seria ruim, o problema é que essas coisas dispersas estão espalhadas por todos os lados, quando você quer encontrar não encontra, quando você quer usar não consegue usar.

Os pontos de dor específicos incluem:

1. **Falta de entrada unificada**: a lógica de geração de imagens está dispersa em diferentes scripts, não há como executar em lote
2. **Metadados ausentes**: as imagens geradas não têm um metadata.json unificado, impossibilitando recuperação e rastreamento
3. **Alto custo de organização manual**: títulos e tags de imagens precisam ser organizados manualmente um por um, com baixa eficiência
4. **Impossibilidade de automação**: quer gerar imagens automaticamente no fluxo CI/CD? Sem chance.

Também pensamos em simplesmente ignorar, mas afinal ainda precisamos fazer o projeto. Como não dá para escapar, então encontramos uma maneira de resolver. Assim decidimos atualizar o ImgBin de "scripts dispersos" para um pipeline de ativos de imagens executável automaticamente. Afinal, algumas coisas, fugir não é a solução.

## Sobre o HagiCode

A solução compartilhada neste artigo vem da nossa experiência prática no projeto [HagiCode](https://hagicode.com). HagiCode é um projeto de assistente de código IA, que também mantém vários componentes como extensão VSCode, serviços de IA de backend, cliente de desktop multiplataforma, etc. Neste cenário complexo de múltiplas linguagens e múltiplas plataformas, o gerenciamento padronizado de ativos de imagens tornou-se um elo chave para melhorar a eficiência do desenvolvimento.

Como dizer, isso também pode ser considerado uma pequena frustração no crescimento do HagiCode. Todo projeto tem esses momentos, problemas pequenos que parecem insignificantes, mas que podem fazer as pessoas se preocuparem por muito tempo.

O sistema de build do HagiCode adota o ecossistema TypeScript + Node.js, portanto o ImgBin naturalmente escolheu a mesma stack tecnológica, garantindo a consistência tecnológica de todo o projeto. Afinal, já estamos acostumados, mudar para outra coisa também é chato.

## Design Principal

### Arquitetura Geral

O ImgBin adota uma arquitetura em camadas, separando claramente os comandos CLI, serviços de aplicação, adaptadores de API de terceiros e a camada de infraestrutura:

```
Hierarquia de Componentes
├── CLI Entry (cli.ts)              Análise de parâmetros globais, roteamento de comandos
├── Commands (commands/*)           generate | batch | annotate | thumbnail
├── Application Services            job-runner | metadata | thumbnail | asset-writer
├── Provider Adapters               image-api-provider | vision-api-provider
└── Infrastructure Layer            config | logger | paths | schema
```

A vantagem deste design em camadas é: as responsabilidades de cada camada são claras, durante os testes é fácil fazer mock das dependências externas. Na verdade, é apenas deixar cada um fazer seu trabalho, sem se incomodar, assim quando há problemas também é fácil encontrar a causa, não é?

### Modelo de Diretório de Ativo Único

O ImgBin adota o modelo "um ativo um diretório", cada vez que uma imagem é gerada, a seguinte estrutura é criada:

```
library/
└── 2026-03/
    └── orange-dashboard/
        ├── original.png      # Imagem original
        ├── thumbnail.webp    # Miniatura 512x512
        └── metadata.json     # Metadados estruturados
```

As vantagens deste modelo são:

1. **Autocontido**: todos os arquivos de cada ativo estão no mesmo diretório, migração e backup são muito convenientes
2. **Rastreável**: através do metadata.json é possível rastrear o tempo de geração da imagem, o prompt usado, informações do modelo, etc.
3. **Extensível**: no futuro, se precisar adicionar mais variantes (como miniaturas de diferentes tamanhos), basta adicionar novos arquivos no mesmo diretório

Coisas ou pessoas bonitas, não precisam ser possuídas, contanto que ela ainda seja bonita, basta apreciar sua beleza. Embora estas palavras pareçam um pouco distantes, a lógica é esta — as imagens juntas, parecem bonitas, e também são fáceis de encontrar.

### Armazenamento em Camadas de Metadados

O metadata.json é o núcleo de todo o sistema, ele adota uma estratégia de armazenamento em camadas, distinguindo três tipos de campos:

```json
{
  "schemaVersion": 2,
  "assetId": "orange-dashboard",
  "slug": "orange-dashboard",
  "title": "Orange Dashboard",
  "tags": ["dashboard", "hero", "orange"],

  "source": { "type": "generated" },

  "paths": {
    "assetDir": "library/2026-03/orange-dashboard",
    "original": "original.png",
    "thumbnail": "thumbnail.webp"
  },

  "generated": {
    "prompt": "orange dashboard for docs hero",
    "provider": "azure-openai-image-api",
    "model": "gpt-image-1.5"
  },

  "recognized": {
    "title": "Orange Dashboard",
    "tags": ["dashboard", "ui", "orange"],
    "description": "A modern orange dashboard with charts and metrics"
  },

  "status": {
    "generation": "succeeded",
    "recognition": "succeeded",
    "thumbnail": "succeeded"
  },

  "timestamps": {
    "createdAt": "2026-03-11T04:01:19.570Z",
    "updatedAt": "2026-03-11T04:02:09.132Z"
  }
}
```

- **generated**: registra as informações originais quando a imagem foi gerada, como o prompt usado, provedor, modelo, etc.
- **recognized**: resultados de reconhecimento de IA, como título gerado automaticamente, tags, descrição
- **manual**: resultados de organização manual, os dados desta área têm a prioridade mais alta e não serão sobrescritos pelo reconhecimento de IA

Esta estratégia em camadas resolveu uma contradição central que tínhamos antes: resultados de reconhecimento de IA e resultados de organização manual, quem tem prioridade? A resposta é manual primeiro, reconhecimento de IA é apenas auxiliar. Esta coisa também ficou clara — algumas coisas, a máquina no final das contas é máquina, no final das contas ainda precisa de pessoas para verificar.

## Padrão Provider Adapter

Outro design central do ImgBin é o padrão Provider Adapter. Nós abstraímos APIs externas como uma interface unificada, de modo que mesmo mudando de provedor de serviços de IA, não há necessidade de modificar a lógica de negócio.

Como dizer, é como em relacionamentos — como a aparência muda não é importante, o importante é que o interior permaneça o mesmo. A interface definida, a implementação interna pode ser trocada à vontade.

### Image Generation Provider

```typescript
interface ImageGenerationProvider {
  // Gerar imagem, retornar Buffer da imagem
  generate(options: GenerateOptions): Promise<Buffer>;

  // Obter lista de modelos suportados
  getSupportedModels(): Promise<string[]>;
}

interface GenerateOptions {
  prompt: string;
  model?: string;
  size?: '1024x1024' | '1792x1024' | '1024x1792';
  quality?: 'standard' | 'hd';
  format?: 'png' | 'webp' | 'jpeg';
}
```

### Vision Recognition Provider

```typescript
interface VisionRecognitionProvider {
  // Reconhecer conteúdo da imagem, retornar metadados estruturados
  recognize(imageBuffer: Buffer): Promise<RecognitionResult>;

  // Obter lista de modelos suportados
  getSupportedModels(): Promise<string[]>;
}

interface RecognitionResult {
  title?: string;
  tags: string[];
  description?: string;
  confidence: number;
}
```

As vantagens deste design de interface são:

1. **Testável**: em testes unitários pode-se passar um mock provider, sem necessidade de chamar APIs externas
2. **Extensível**: adicionar um novo provider só precisa implementar a interface, sem necessidade de modificar o código do chamador
3. **Substituível**: ambiente de produção usa Azure OpenAI, ambiente de teste usa modelo local, só precisa trocar a configuração

Quero rir para disfarçar as lágrimas que caíram, quero chorar para testar se fiquei anestesiado — às vezes fazer projetos é assim, superficialmente parece que trocou de API, mas na verdade a lógica interna não mudou nada, então não há do que ter medo.

## Design de Comandos CLI

O ImgBin fornece quatro comandos principais, atendendo a diferentes cenários de uso:

### generate: Geração de Imagem Única

```bash
# Uso mais simples
imgbin generate --prompt "orange dashboard for docs hero"

# Gerar miniatura e anotação IA simultaneamente
imgbin generate --prompt "orange dashboard" --annotate --thumbnail

# Especificar diretório de saída
imgbin generate --prompt "orange dashboard" --output ./library
```

### batch: Tarefas em Lote

Tarefas em lote são definidas através de arquivos manifest YAML ou JSON, adequadas para uso em fluxos CI/CD:

```yaml
# assets/jobs/launch.yaml
defaults:
  annotate: true
  thumbnail: true
  libraryRoot: ./library

jobs:
  - prompt: "orange dashboard hero"
    slug: orange-dashboard
    tags: [dashboard, hero, orange]

  - prompt: "pricing grid for docs"
    slug: pricing-grid
    tags: [pricing, grid, docs]
```

Comando de execução:

```bash
imgbin batch assets/jobs/launch.yaml
```

O design de tarefas em lote suporta isolamento de falhas: processamento item por item no manifest, falha de um item não afeta outras tarefas. Pode-se pré-visualizar através de `--dry-run` sem executar realmente.

Isso já seria bom, o关键是 que ele ainda pode te dizer onde teve sucesso e onde falhou, unlike algumas coisas, falhou e nem sabia como falhou.

### annotate: Anotação IA

Executa reconhecimento de IA em imagens existentes, gerando automaticamente título, tags, descrição:

```bash
# Anotar uma única imagem
imgbin annotate ./library/2026-03/orange-dashboard

# Anotar em lote um diretório inteiro
imgbin annotate ./library/2026-03/
```

### thumbnail: Geração de Miniaturas

Gera miniaturas para imagens existentes:

```bash
# Gerar miniatura
imgbin thumbnail ./library/2026-03/orange-dashboard
```

## Design do Manifest de Tarefas em Lote

O manifest de tarefas em lote suporta configuração flexível, valores padrão podem ser definidos uniformemente, tarefas individuais também podem sobrescrever:

```yaml
# Valores padrão globais
defaults:
  annotate: true        # Ativar anotação IA por padrão
  thumbnail: true       # Gerar miniaturas por padrão
  libraryRoot: ./library
  model: gpt-image-1.5

jobs:
  # Configuração mínima, fornece apenas prompt
  - prompt: "first image"

  # Configuração completa
  - prompt: "second image"
    slug: custom-slug
    tags: [tag1, tag2]
    annotate: false     # Esta tarefa não executa anotação IA
    model: dall-e-3    # Esta tarefa usa modelo diferente
```

Durante a execução, o ImgBin processa as tarefas uma por uma, os resultados de cada tarefa são gravados no metadata.json correspondente, mesmo que uma tarefa falhe, não afetará outras tarefas. Após a conclusão das tarefas, um relatório汇总 é exibido:

```
✓ orange-dashboard (succeeded)
✓ pricing-grid (succeeded)
✗ hero-banner (failed: API rate limit exceeded)

2/3 succeeded, 1 failed
```

Algumas coisas, não adianta ter pressa, uma de cada vez, ao contrário, é mais tranquilo. Esta é, talvez, a filosofia das tarefas em lote.

## Configuração de Variáveis de Ambiente

O ImgBin suporta configuração flexível através de variáveis de ambiente:

```bash
# Diretório de trabalho do ImgBin
IMGBIN_WORKDIR=/path/to/imgbin

# Caminho do executável (para chamada em scripts)
IMGBIN_EXECUTABLE=/path/to/imgbin/dist/cli.js

# Diretório raiz da biblioteca de ativos
IMGBIN_LIBRARY_ROOT=./.imgbin-library

# Configuração do Azure OpenAI (se usando o provider Azure)
AZURE_OPENAI_ENDPOINT=https://your-resource.openai.azure.com/
AZURE_OPENAI_API_KEY=your-api-key
AZURE_OPENAI_IMAGE_DEPLOYMENT=gpt-image-1
```

Configuração, é importante e não é importante. Afinal, como for mais confortável, o que se adequa a você é o melhor.

## Pontos de Implementação

Durante o processo de implementação, resumimos os seguintes pontos-chave:

### Design da Interface Provider

A definição da interface deve ser clara e completa, incluindo parâmetros de entrada, valores de retorno, tratamento de erros. Recomenda-se fornecer tanto chamadas síncronas quanto assíncronas, convenientes para diferentes cenários de uso.

Isso também pode ser considerado um pouco de experiência de quem já passou por isso, afinal, interface é algo, uma vez definido não se quer mais mudar, é chato.

### Estratégia de Tratamento de Falhas

Quando uma tarefa em lote falhar, deve-se:

1. Registrar informações detalhadas de erro em um arquivo de log separado
2. Continuar executando outras tarefas, sem interromper todo o fluxo
3. Finalmente retornar código de saída não zero, indicando que houve falha de tarefa
4. Exibir claramente os resultados de execução de cada tarefa no relatório汇总

Algumas coisas falharam é falharam, fugir não adianta, melhor admitir abertamente e então encontrar uma maneira de resolver. Este princípio, fazer projetos e ser pessoa é o mesmo.

### Estratégia de Fusão de Metadados

Resultados de reconhecimento são gravados por padrão na área `recognized`, campos modificados manualmente têm marcação `manual`. Atualizações de metadados adotam a estratégia "apenas adicionar, não remover": a menos que seja explicitamente passado o parâmetro `--force`, caso contrário não sobrescreve resultados de organização manual existentes.

Esta coisa também ficou clara — algumas coisas, perdidas é perdidas, sobrescritas também se foram. Melhor preservar, afinal o registro em si também é uma espécie de beleza.

### Atomicidade de Criação de Diretório

Usar `fs.mkdir({ recursive: true })` garante a atomicidade da criação de diretórios, evitando condições de corrida em cenários concorrentes.

Isso é mais ou menos o chamado senso de segurança — o que deve ser firme é firme, o que deve ser rápido é rápido, sem se arrastar, sem olhar para frente e para trás.

## Conclusão

Como ferramenta principal de gerenciamento de ativos de imagens do projeto HagiCode, o ImgBin resolve os problemas que enfrentamos através dos seguintes designs:

1. **Entrada unificada**: comandos CLI cobrem todas as operações de geração, anotação, miniaturas
2. **Dirigido por metadados**: cada ativo tem um metadata.json completo, suporta recuperação e rastreamento
3. **Provider Adapter**: abstração flexível de APIs externas, fácil de testar e estender
4. **Suporte a tarefas em lote**: processos CI/CD podem executar automaticamente geração de imagens em lote

Tudo desbotou......mas esta solução, realmente foi usada.

Esta solução não só melhorou a eficiência de desenvolvimento do próprio HagiCode, como também formou um框架 de gerenciamento de ativos de imagens reutilizável. Se você também está desenvolvendo projetos multi-componentes similares, acredito que as ideias de design do ImgBin vão te dar algumas inspirações.

Juventude, sempre precisa se aventurar. Sem se aventurar, como saber o próprio valor?
