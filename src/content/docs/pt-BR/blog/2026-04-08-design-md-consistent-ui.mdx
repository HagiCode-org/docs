## Design.md: Uma Solução para Design de UI Frontend Consistente com IA

> Na era do desenvolvimento frontend assistido por IA, como manter a consistência das UIs geradas por IA? Este artigo compartilha nossa experiência prática na construção de um site de galeria de designs baseado em awesome-design-md, e como criar design.md estruturados para orientar a IA a realizar designs de UI padronizados.

## Contexto

Quem usa IA para escrever código frontend provavelmente já teve uma experiência similar: a mesma página, gerada pela IA várias vezes, com estilos diferentes a cada vez. Às vezes são bordas arredondadas, às vezes quadradas; às vezes o espaçamento é 8px, às vezes 16px; até mesmo o mesmo botão pode ter aparências diferentes em conversas distintas.

Isso não é um fenômeno isolado. Com a popularização do desenvolvimento assistido por IA, **a falta de consistência nas UIs frontend geradas por IA** tornou-se um problema generalizado. Diferentes assistentes de IA, diferentes prompts, e até mesmo o mesmo assistente em conversas diferentes podem produzir designs de interface com estilos completamente distintos. Isso traz custos de manutenção enormes para iterações de produtos.

A raiz do problema é bastante simples: falta um **documento de referência de design autoritativo**. Arquivos CSS tradicionais só podem dizer aos desenvolvedores "como implementar", mas não conseguem transmitir completamente "por que projetar desta forma" e "em que cenários usar quais padrões de design". Para a IA, ela precisa de uma descrição estruturada clara para entender as especificações de design.

Ao mesmo tempo, a comunidade open source já tem alguns recursos excelentes. O projeto [VoltAgent/awesome-design-md](https://github.com/VoltAgent/awesome-design-md) coleta documentos de sistemas de design de muitas empresas conhecidas, cada diretório contém README.md, DESIGN.md e HTML de pré-visualização. Mas tudo isso está disperso nos repositórios upstream, dificultando consulta rápida e comparação.

Então, seria possível integrar esses recursos, criar uma galeria de designs conveniente para consulta, e ao mesmo tempo consolidar um design.md estruturado para uso da IA?

A resposta é sim. A seguir, compartilharemos nossa solução.

## Análise

Antes de começar a escrever código, vamos decomprimir os vários desafios técnicos deste problema.

### Gerenciamento de Fontes de Conteúdo: Como unificar recursos de design dispersos?

O repositório upstream awesome-design-md contém muitos documentos de design, mas precisamos de uma maneira de incorporá-lo ao nosso projeto.

**Solução: usar git submodule**

```
awesome-design-md-site
└── vendor/awesome-design-md          # Recursos upstream (git submodule)
```

Isso tem vários benefícios:
- Controle de versão: pode bloquear uma versão upstream específica
- Build offline: não precisa solicitar APIs externas durante o build
- Revisão de conteúdo: pode ver alterações específicas no PR

### Padronização de Dados: Como unificar estruturas de documentos diferentes?

Os documentos de design de diferentes empresas podem ter estruturas diferentes, alguns faltam arquivos de pré-visualização, alguns têm nomenclatura não uniforme. Precisamos realizar processamento de padronização durante o build.

**Solução: escanear e gerar itens padronizados durante o build**

O módulo principal é `awesomeDesignCatalog.ts`, responsável por:
1. Escanear o diretório `vendor/awesome-design-md/design-md/*`
2. Validar se cada item contém os arquivos necessários (README.md, DESIGN.md, pelo menos um arquivo de pré-visualização)
3. Extrair e renderizar conteúdo Markdown como HTML
4. Gerar dados de itens padronizados

### Arquitetura de Site Estático: Como fornecer busca dinâmica mantendo deploy estático?

Já que é uma galeria de designs, a funcionalidade de busca é essencial. Mas Astro é um gerador de sites estáticos, como implementar busca em tempo real?

**Solução: React island + sincronização de parâmetros de consulta de URL**

```typescript
// src/components/gallery/SearchToolbar.tsx

export function SearchToolbar() {
  const [query, setQuery] = useState('');

  // Sincronização de URL

    const params = new URLSearchParams(window.location.search);

```

O benefício disso é manter a capacidade de deploy de sites estáticos (pode fazer deploy para qualquer serviço de hospedagem estática), enquanto fornece experiência de usuário de filtragem instantânea.

### Documentação de Design: Como fazer a IA entender e seguir especificações de design?

Este é o núcleo de toda a solução. Precisamos criar um design.md estruturado que permita a IA entender e aplicar nossas especificações de design.

**Solução:借鉴 a estrutura do DESIGN.md do ClickHouse**

O [DESIGN.md do ClickHouse](https://github.com/ClickHouse/ClickHouse/blob/master/DESIGN.md) é uma excelente referência, ele inclui:
- Visual Theme & Atmosphere
- Color Palette & Roles
- Typography Rules
- Component Stylings
- Layout Principles
- Depth & Elevation
- Do's and Don'ts
- Responsive Behavior
- Agent Prompt Guide

Nossa abordagem é: **referenciar a estrutura, reescrever o conteúdo**. Manter a estrutura de capítulos do DESIGN.md do ClickHouse, mas substituir o conteúdo com os tokens de design e especificações de componentes realmente usados em nosso projeto.

## Solução

Com base na análise acima, nossa solução contém quatro módulos principais.

### 1. Pipeline de Ingestão de Conteúdo

Esta é a base de todo o sistema, responsável por extrair e padronizar conteúdo dos recursos upstream.

```typescript
// src/lib/content/awesomeDesignCatalog.ts

export async function scanSourceEntries(): Promise<DesignEntry[]> {
  const designDir = 'vendor/awesome-design-md/design-md';
  const entries: DesignEntry[] = [];

  for (const dir of await fs.readdir(designDir)) {
    const entryPath = path.join(designDir, dir);
    if (await isValidDesignEntry(entryPath)) {
      const entry = await normalizeDesignEntry(entryPath);
      entries.push(entry);
    }
  }

  return entries;
}

async function isValidDesignEntry(dir: string): Promise<boolean> {
  const requiredFiles = ['README.md', 'DESIGN.md'];
  for (const file of requiredFiles) {
    if (!(await fileExists(path.join(dir, file)))) {
      return false;
    }
  }
  return true;
}
```

### 2. Interface de Galeria

A interface da galeria inclui três partes principais:

**Página inicial**: Exibe uma grade de cards com todos os itens de design, cada card contém:
- Título e resumo do item de design
- Imagem de pré-visualização (se disponível)
- Destaque de busca rápida

**Página de detalhes**: Exibe de forma agregada todas as informações de um único item de design:
- Documento README
- Documento DESIGN
- Pré-visualização (com suporte a alternância de tema claro/escuro)
- Navegação para itens adjacentes

**Navegação**: Suporta voltar à galeria, navegar itens adjacentes

A galeria da página inicial usa layout de cards de alta densidade, espalhando itens de design.md de diferentes fontes em uma estrutura visual unificada, facilitando comparação rápida de estilos de marca, padrões de botões e ritmo de tipografia.

Ao entrar em um item específico, a página de detalhes coloca o resumo do design e a pré-visualização em tempo real na mesma página, reduzindo o custo de alternar entre documentos, pré-visualizações e código fonte.

### 3. Funcionalidade de Busca

A funcionalidade de busca é baseada em filtragem no lado do cliente, usando parâmetros de consulta de URL para manter estado:

```typescript
// src/components/gallery/SearchToolbar.tsx

function SearchToolbar({ entries }: { entries: DesignEntry[] }) {
  const [query, setQuery] = useState('');
  const [results, setResults] = useState(entries);

  useEffect(() => {
    const params = new URLSearchParams(window.location.search);
    const initialQuery = params.get('q') || '';
    setQuery(initialQuery);
    filterEntries(initialQuery);
  }, []);

  const filterEntries = (searchQuery: string) => {
    const filtered = entries.filter(entry =>
      entry.searchText.toLowerCase().includes(searchQuery.toLowerCase())
    );
    setResults(filtered);
  };

  const handleChange = (e: React.ChangeEvent<HTMLInputElement>) => {
    const value = e.target.value;
    setQuery(value);
    filterEntries(value);

    // Atualizar URL (sem acionar recarregamento da página)
    const newUrl = value
      ? `${window.location.pathname}?q=${encodeURIComponent(value)}`
      : window.location.pathname;
    window.history.replaceState({}, '', newUrl);
  };

  return (
    <div className="search-toolbar">
      <input
        type="text"
        value={query}
        onChange={handleChange}
        placeholder="Buscar itens de design..."
      />
      <span className="result-count">{results.length} resultados</span>
    </div>
  );
}
```

## Prática

Depois de entender a solução, como implementá-la especificamente?

### Passos de Implementação

**Primeiro passo: inicializar submódulo**

```bash
# Adicionar repositório upstream como submódulo
git submodule add https://github.com/VoltAgent/awesome-design-md.git vendor/awesome-design-md

# Inicializar e atualizar submódulo
git submodule update --init --recursive
```

**Segundo passo: criar pipeline de conteúdo**

Implementar `awesomeDesignCatalog.ts`, incluindo:
- Lógica de escaneamento e validação de arquivos
- Renderização de Markdown (usando o renderizador integrado do Astro)
- Extração de dados de itens

**Terceiro passo: construir UI da galeria**

Usar Astro + React Islands para criar:
- Layout da galeria da página inicial (grade de cards)
- Componente de card de design
- Barra de ferramentas de busca
- Layout da página de detalhes

**Quarto passo: escrever documento de design**

Com base na estrutura do DESIGN.md do ClickHouse, preencher os tokens de design reais do seu projeto. Atualizar README.md, adicionando links para design.md.

### Considerações Importantes

**Segurança**: A renderização de Markdown precisa filtrar HTML inseguro. O renderizador integrado do Astro filtra tags script por padrão, mas ainda é preciso atenção aos riscos de XSS.

**Performance**: Muitos iframes de pré-visualização podem afetar o carregamento inicial. Recomenda-se usar `loading="lazy"` para carregar pré-visualizações de forma adiada.

**Manutenibilidade**: design.md precisa manter sincronização com a implementação do código. Recomenda-se adicionar verificações no CI para garantir que variáveis CSS sejam consistentes entre documento e código.

**Acessibilidade**: Garantir que o contraste de cores esteja em conformidade com os padrões WCAG AA (pelo menos 4.5:1).

### Guia de Uso com IA

Depois de criar design.md, como fazer a IA realmente usá-lo? Aqui estão algumas dicas práticas:

**Dica 1: referência explícita no prompt**

```
Por favor, consulte o arquivo design.md no diretório raiz do projeto e use as especificações de design definidas nele para implementar os seguintes componentes:
- Botão: usar cor primária, borda arredondada 8px
- Card: usar nível de sombra elevation-2
```

**Dica 2: exigir que a IA referencie variáveis CSS específicas**

```
Implementar uma barra de navegação, requer:
- Cor de fundo usar --color-bg-primary
- Bordas usar --color-border-subtle
- Texto usar --text-color-primary
```

**Dica 3: incluir conteúdo design.md no prompt do sistema**

Se sua ferramenta de IA suporta prompts de sistema personalizados, você pode adicionar diretamente o conteúdo principal do design.md.

### Estratégia de Testes

**Testes do pipeline de conteúdo**:
- Cenários de arquivos ausentes (faltando README.md ou DESIGN.md)
- Cenários de erro de formato (falha na análise de Markdown)
- Cenários de diretório vazio

**Testes de funcionalidade de busca**:
- Tratamento de resultados vazios
- Caracteres especiais (como chinês, emoji)
- Validação de sincronização de URL

**Testes de componentes de UI**:
- Alternância de tema claro/escuro
- Layout responsivo
- Estado de carregamento de pré-visualização

## Resumo

O problema de inconsistência nas UIs geradas por IA encontrado durante o desenvolvimento do HagiCode é essencialmente falta de documentos de referência de design estruturados. Através da construção de um site de galeria de designs e criação de design.md padronizados, resolvemos com sucesso esse problema.

O valor central desta solução está em:
- **Unificar recursos**: integrar documentos de sistemas de design dispersos
- **Especificações estruturadas**: apresentar especificações de design de forma que a IA possa entender
- **Manutenção contínua**: manter conteúdo atualizado através de git submodule

Se você também está usando IA auxiliando desenvolvimento frontend, recomendo tentar esta solução. Criar um design.md estruturado não só melhora a consistência do código gerado por IA, mas também ajuda a equipe a manter a unidade das especificações de design.
