## Contexto

O tempo voa, o desenvolvimento de modelos de linguagem grande é como bambô na primavera, crescendo rapidamente. Antes, ainda aplaudíamos por "uma IA capaz de escrever código", hoje estamos na era de colaboração entre múltiplos modelos e fusão de múltiplas ferramentas. Isso é interessante? Talvez, afinal, os desenvolvedores nunca precisaram apenas das ferramentas em si, mas sim de uma capacidade de se adaptar a diferentes cenários e alternar com flexibilidade e tranquilidade.

Como uma plataforma de codificação assistida por IA, o HagiCode recentemente teve dois grandes eventos: primeiro, a integração completa do modelo GLM-5.1 da Zhipu AI, e segundo, o Gemini CLI tornando-se oficialmente o décimo Agent CLI suportado. Essas duas coisas não são grandes demais nem pequenas demais, mas para o aperfeiçoamento da plataforma, é sempre uma coisa boa.

O GLM-5.1 é o modelo mais recente da Zhipu AI. Comparado com o GLM-5.0, tem capacidades de raciocínio mais fortes, compreensão de código mais profunda e chamadas de ferramentas mais suaves. Mais importante, é o primeiro modelo GLM a suportar entrada de imagens — o que isso significa? Significa que os usuários podem capturar telas diretamente para a IA ver o problema, sem precisar se esforçar para descrevê-lo. Essa conveniência, quem usa sabe.

Ao mesmo tempo, através da arquitetura HagiCode.Libs.Providers, o HagiCode integrou com sucesso o Gemini CLI. Este é o décimo Agent CLI, honestamente, chegar a este ponto também traz um pouco de senso de realização.

Vale a pena mencionar que a funcionalidade de upload de imagens do HagiCode permite que os usuários capturem telas diretamente para conversar com a IA. Mesmo executando a versão GLM 4.7, a plataforma ainda funciona bem e já ajudou o projeto a completar muitos trabalhos de construção importantes. E o GLM-5.1? Naturalmente será um passo além.

## Sobre o HagiCode

A solução compartilhada neste artigo vem de nossa experiência prática no projeto [HagiCode](https://hagicode.com). O HagiCode é uma plataforma de codificação assistida por IA de código aberto, projetada para fornecer aos desenvolvedores um assistente de programação com IA flexível e poderoso através de uma arquitetura de múltiplos modelos e múltiplos CLIs. Endereço do projeto: [github.com/HagiCode-org/site](https://github.com/HagiCode-org/site)

## Design de arquitetura multi-CLI

Uma das principais vantagens do HagiCode é suportar várias ferramentas CLI de programação com IA diferentes através de uma camada de abstração unificada. Os benefícios desse design, em última análise, são apenas isso: coisas novas podem entrar, coisas velhas podem ficar, e o código não fica bagunçado. Afinal, todo mundo espera que a vida possa ser assim, certo?

### Enumeração AIProviderType

A plataforma define os tipos de provedores CLI suportados através da enumeração `AIProviderType`:

```csharp
public enum AIProviderType
{
    ClaudeCodeCli = 0,    // Claude Code CLI
    CodexCli = 1,          // GitHub Copilot Codex
    GitHubCopilot = 2,     // GitHub Copilot
    CodebuddyCli = 3,     // Codebuddy CLI
    OpenCodeCli = 4,      // OpenCode CLI
    IFlowCli = 5,         // IFlow CLI
    HermesCli = 6,        // Hermes CLI
    QoderCli = 7,         // Qoder CLI
    KiroCli = 8,          // Kiro CLI
    KimiCli = 9,          // Kimi CLI
    GeminiCli = 10,       // Gemini CLI (novo)
}
```

Como pode ser visto, o Gemini CLI junta-se a esta grande família como o décimo membro. Cada CLI tem características únicas e cenários aplicáveis, e os usuários podem escolher flexivelmente de acordo com suas necessidades. Afinal, todos os caminhos levam a Roma, apenas alguns caminhos são mais fáceis de percorrer, outros um pouco mais tortuosos.

### Arquitetura Provider

O HagiCode.Libs.Providers fornece uma interface Provider unificada, tornando a integração de cada CLI padronizada e concisa. Tomando o Gemini CLI como exemplo:

```csharp
public class GeminiProvider : ICliProvider<GeminiOptions>
{
    private static readonly string[] DefaultExecutableCandidates = ["gemini", "gemini-cli"];
    private const string ManagedBootstrapArgument = "--acp";

    public string Name => "gemini";
    public bool IsAvailable => _executableResolver.ResolveFirstAvailablePath(DefaultExecutableCandidates) is not null;
}
```

Os benefícios deste design são:
- A integração de novos CLIs requer apenas a implementação de uma classe Provider
- Gerenciamento de ciclo de vida unificado e pool de sessões
- Resolução automática de aliases e localização de executáveis

Em última análise, esse design é apenas simplificar coisas complexas e tornar a vida um pouco mais fácil.

### Provider Registry

O Provider Registry processa automaticamente mapeamento de alias e registro:

```csharp
if (provider is GeminiProvider)
{
    registry.Register(provider.Name, provider, ["gemini-cli"]);
    continue;
}
```

Isso significa que os usuários podem usar `gemini` ou `gemini-cli` para chamar o Gemini CLI, e o sistema reconhecerá automaticamente. É como ter muitos amigos, alguns chamados pelo nome completo, outros pelo apelido, afinal é ele mesmo, pode chamar como quiser.

## Suporte ao modelo GLM-5.1

O GLM-5.1 é o modelo mais recente da Zhipu AI, e o HagiCode completou o suporte completo a ele.

### Catálogo de Secondary Professions

O HagiCode gerencia todos os modelos suportados através do Secondary Professions Catalog. Abaixo está a configuração da série GLM:

| Model ID | Name | SupportsImage | Compatible CLI Families |
|----------|------|----------------|-------------------------|
| `glm-4.7` | GLM 4.7 | - | claude, codebuddy, hermes, qoder, kiro |
| `glm-5` | GLM 5 | - | claude, codebuddy, hermes, qoder, kiro |
| `glm-5-turbo` | GLM 5 Turbo | - | claude, codebuddy, hermes, qoder, kiro |
| `glm-5.0` | GLM 5.0 (Legacy) | - | claude, codebuddy, hermes, qoder, kiro |
| `glm-5.1` | GLM 5.1 | **true** | claude, codebuddy, hermes, qoder, kiro |

As principais características do GLM-5.1 podem ser resumidas como:
- Identificação de versão independente, sem bagagem legacy
- Primeiro modelo GLM a suportar entrada de imagens
- Capacidades de raciocínio mais fortes e compreensão de código
- Ampla compatibilidade com múltiplos CLIs

### GLM-5.1 vs GLM-5.0

Do ponto de vista do código, as principais diferenças entre GLM-5.1 e GLM-5.0:

```csharp
// GLM-5.0 (Legacy) - tem lógica especial de retenção
private const string Glm50CodebuddySecondaryProfessionId = "secondary-glm-5-codebuddy";
private const string Glm50CodebuddyModelValue = "glm-5.0";

// GLM-5.1 - identificador de novo modelo independente
private const string Glm51SecondaryProfessionId = "secondary-glm-5-1";
private const string Glm51ModelValue = "glm-5.1";
```

O GLM-5.0 possui a marca "Legacy", que é um identificador de versão antigo mantido para compatibilidade com versões anteriores. O GLM-5.1 é uma nova versão completamente independente, sem qualquer bagagem histórica. É como algumas pessoas, sempre vivendo no passado; enquanto outras, viajam leve e caminham mais rápido.

### Configurando o GLM-5.1

Exemplo de configuração para usar o GLM-5.1 no HagiCode:

```json
{
  "primaryProfessionId": "profession-claude-code",
  "secondaryProfessionId": "secondary-glm-5-1",
  "model": "glm-5.1",
  "reasoning": "high"
}
```

## Funcionalidade de upload de imagens

O suporte a imagens do HagiCode é implementado através da propriedade `SupportsImage` do SecondaryProfession:

```csharp
public class HeroSecondaryProfessionSettingDto
{
    public bool SupportsImage { get; set; }
}
```

No Secondary Professions Catalog, a configuração do GLM-5.1 é a seguinte:

```json
{
  "id": "secondary-glm-5-1",
  "supportsImage": true
}
```

Isso significa que os usuários podem fazer upload diretamente de capturas de tela para a IA analisar, por exemplo:
- Capturas de tela de mensagens de erro
- Problemas de interface do usuário
- Gráficos de visualização de dados
- Resultados de execução de código

Não é mais necessário descrever manualmente o problema, basta capturar a tela — a conveniência dessa funcionalidade, quem usa sabe. Afinal, algumas coisas, não importa o quanto você fale, não é melhor do que dar uma olhada.

## Integração do Gemini CLI

O Gemini CLI, como o décimo Agent CLI, é integrado ao HagiCode através da arquitetura Provider padrão.

### Opções de configuração

O Gemini CLI suporta várias opções de configuração:

```csharp
public class GeminiOptions
{
    public string? ExecutablePath { get; set; }
    public string? WorkingDirectory { get; set; }
    public string? SessionId { get; set; }
    public string? Model { get; set; }
    public string? AuthenticationMethod { get; set; }
    public string? AuthenticationToken { get; set; }
    public Dictionary<string, string?> AuthenticationInfo { get; set; }
    public Dictionary<string, string?> EnvironmentVariables { get; set; }
    public string[] ExtraArguments { get; set; }
    public TimeSpan? StartupTimeout { get; set; }
    public CliPoolSettings? PoolSettings { get; set; }
}
```

Essas opções cobrem tudo, desde configurações básicas até recursos avançados, e os usuários podem configurar flexivelmente de acordo com suas necessidades. Afinal, as necessidades de cada pessoa são diferentes, e ter flexibilidade é sempre bom.

### Protocolo de comunicação ACP

O Gemini CLI suporta o protocolo de comunicação ACP (Agent Communication Protocol), que é o padrão de comunicação CLI unificado do HagiCode. Através do ACP, diferentes CLIs podem interagir com a plataforma de maneira consistente, simplificando bastante o trabalho de integração. Em última análise, é apenas unificar coisas complexas para que todos possam ficar um pouco mais relaxados.

## Melhores práticas para integração de múltiplos CLIs

O HagiCode.Libs.Providers fornece um mecanismo unificado de registro e uso:

```csharp
services.AddHagiCodeLibs();

var gemini = serviceProvider.GetRequiredService<ICliProvider<GeminiOptions>>();
var codebuddy = serviceProvider.GetRequiredService<ICliProvider<CodebuddyOptions>>();
var hermes = serviceProvider.GetRequiredService<ICliProvider<HermesOptions>>();
```

Esse design de injeção de dependência torna o uso de cada CLI muito conciso, e também facilita testes unitários e simulações. Afinal, escrever código mais limpo também é uma responsabilidade consigo mesmo.

### Precauções

No uso real, existem alguns pontos que precisam de atenção:

1. **Configuração da API Key**: Certifique-se de definir corretamente `ANTHROPIC_AUTH_TOKEN`, caso contrário não será possível chamar o modelo
2. **Disponibilidade do modelo**: O GLM-5.1 precisa ter permissões ativadas no provedor de modelo correspondente
3. **Funcionalidade de imagem**: Apenas modelos que suportam `supportsImage: true` podem usar a funcionalidade de upload de imagens
4. **Instalação do CLI**: Antes de usar o Gemini CLI, certifique-se de que `gemini` ou `gemini-cli` está no PATH do sistema

Essas são coisas pequenas, mas se coisas pequenas não forem bem tratadas, podem se tornar grandes coisas. Portanto, é melhor prestar atenção.
