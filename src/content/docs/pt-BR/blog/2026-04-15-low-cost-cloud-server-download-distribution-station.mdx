## Como usar servidores em nuvem baratos para criar uma estação de distribuição de downloads acelerada

> O tráfego de armazenamento em nuvem é absurdamente caro, o acesso transfronteiriço é lento o suficiente para deixar alguém louco, e o preço de CDNs é proibitivo...... Quem trabalha com distribuição entende essas dores. Este artigo compartilha uma solução de baixo custo que exploramos no projeto [HagiCode](https://hagicode.com). Servidor em nuvem + camada de cache Nginx, os custos caíram pela metade e a velocidade aumentou, o que é um pequeno consolo.

## Contexto

Na internet, a velocidade de download e a estabilidade, em última análise, são tudo sobre experiência do usuário. Seja código aberto ou comercial, sempre é preciso fornecer aos usuários uma maneira confiável de download.

Baixar arquivos diretamente do armazenamento em nuvem (como Azure Blob Storage, AWS S3) parece simples, mas na verdade há vários problemas:

**Latência de rede**: Acesso transfronteiriço e transregional, lento o suficiente para fazer você querer quebrar o teclado. Os usuários esperam até as flores murcharem, como a experiência pode ser boa?

**Custo de largura de banda**: O tráfego de saída do armazenamento em nuvem é caro o suficiente para doer. Azure Blob Storage acessado da China continental custa cerca de 0,5 yuans por GB, 1TB por mês são 500 yuans. Para equipes pequenas, esse dinheiro não é muito, mas também não é pouco - afinal, o dinheiro de ninguém veio do vento.

**Restrições de acesso**: Em certas regiões, o acesso a serviços de nuvem estrangeiros é irregular, às vezes simplesmente não acessível. Os usuários não conseguem baixar, o que já é frustrante por si só.

**Custo de CDN**: CDNs comerciais realmente resolvem o problema, mas o preço também é realmente bonito. Equipes pequenas não podem pagar?

Então existe uma maneira que seja econômica e fácil de usar? Na verdade, existe. Servidor em nuvem + proxy reverso + camada de cache, simples e direto. Os custos caíram cerca de metade, e a velocidade aumentou, o que também é um pequeno consolo.

## Design de Arquitetura

### Ideia Geral de Arquitetura

Primeiro, vamos dar uma olhada no design geral da arquitetura:

```
Solicitação do usuário
    ↓
Resolução DNS
    ↓
┌─────────────────────────────────────┐
│   Camada de proxy reverso (Traefik/Bunker Web)   │ ← Término SSL, distribuição de rota, proteção de segurança
├─────────────────────────────────────┤
│   Portas: 80/443                       │
│   Funcionalidades: Certificados automáticos Let's Encrypt      │
│         Roteamento Host                    │
└─────────────────────────────────────┘
    ↓
┌─────────────────────────────────────┐
│   Camada de cache (Nginx)                     │ ← Cache de arquivo, compressão Gzip
├─────────────────────────────────────┤
│   Portas: 8080(server) / 8081(desktop) │
│   Estratégia de cache:                          │
│   - index.json: 1 hora               │
│   - Outros arquivos: 7 dias                   │
│   Tamanho do cache: 1GB                      │
└─────────────────────────────────────┘
    ↓
┌─────────────────────────────────────┐
│   Origem (Azure Blob Storage)         │ ← Armazenamento de arquivo
└─────────────────────────────────────┘
```

A ideia central desta arquitetura, em poucas palavras, é: **adicionar uma camada de cache entre usuários e armazenamento em nuvem**.

As solicitações dos usuários primeiro chegam à camada de proxy reverso no servidor em nuvem, depois a camada de cache Nginx assume. Tem o arquivo em cache? Entrega diretamente ao usuário. Não? Busca uma cópia do armazenamento em nuvem e convenientemente armazena uma localmente. Na próxima vez que acessar o mesmo arquivo, não precisa mais incomodar o armazenamento em nuvem. É como memória, uma vez que você se lembra de algumas coisas, na próxima vez não precisa mais se esforçar para pensar......

### Por que escolher esta arquitetura?

**Vantagens do servidor em nuvem**:
- Custo controlável: Alibaba Cloud e outros fornecem servidores em nuvem baratos (configuração de 1-2 núcleos 2GB custa cerca de 50-100 yuans por mês)
- Implantação flexível: pode configurar livremente proxy reverso e estratégias de cache
- Localização geográfica flexível: pode escolher região de servidor próxima aos usuários
- Forte escalabilidade: pode atualizar configuração de acordo com demanda de tráfego

**Arquitetura de proxy reverso + cache**:
- Reduz pressão na origem: cache de arquivos quentes, reduz acesso ao armazenamento em nuvem
- Reduz custos: custos de tráfego de servidor em nuvem são muito menores que tráfego de saída de armazenamento em nuvem
- Melhora velocidade: acesso próximo, largura de banda de servidor geralmente é melhor que armazenamento em nuvem

**Por que escolher Nginx como camada de cache?**

Esta escolha não foi aleatória, afinal Nginx tem suas razões:

1. Alta performance: a performance de proxy reverso Nginx é reconhecida na indústria
2. Cache maduro: função proxy_cache embutida, estável e confiável, não dá problemas
3. Baixo uso de recursos: 256MB de memória é suficiente, também é amigável para o servidor
4. Configuração flexível: diferentes tipos de arquivo podem ter diferentes estratégias de cache, isso é muito atencioso

## Camada de Proxy Reverso: Traefik vs Bunker Web

A solução de implantação HagiCode na verdade suporta dois proxies reversos, cada um com suas características:

| Solução | Características | Cenário de Uso |
|------|------|----------|
| **Traefik** | Leve, SSL automático, configuração simples | Implantação básica, cenários de baixo tráfego |
| **Bunker Web** | WAF embutido, proteção DDoS, proteção contra crawlers | Requisitos de alta segurança, cenários de alto tráfego |

### Traefik: Escolha Leve

Traefik é um proxy reverso HTTP moderno e balanceador de carga, a maior característica é configuração simples e pode automaticamente obter certificados Let's Encrypt.

Para implantação inicial ou cenários com pouco tráfego, Traefik é uma escolha bastante boa:
- Uso de recursos não é muito (1.5 CPU/512MB de memória é suficiente)
- Certificados SSL configurados automaticamente, sem precisar se preocupar
- Configuração de roteamento baseada em tags Docker, também é conveniente

### Bunker Web: Cenários de Alta Segurança

Bunker Web é um firewall de aplicação web baseado em Nginx, com proteção de segurança mais abrangente.

Quando considerar mudar para Bunker Web? Aproximadamente nestas situações:
- Sofrendo ataques DDoS (embora ninguém queira encontrar)
- Precisa de proteção ModSecurity
- Quer funcionalidade anti-crawler
- Requisitos de segurança mais altos

HagiCode fornece o script `switch-deployment.sh`, que pode alternar rapidamente entre as duas soluções:

```bash
# Mudar para Bunker Web
./switch-deployment.sh bunkerweb

# Mudar de volta para Traefik
./switch-deployment.sh traefik

# Ver status atual
./switch-deployment.sh status
```

O script automaticamente faz pré-verificações, verificações de saúde e pode rollback automático, o processo de troca também é seguro e confiável, não vai quebrar no meio da troca.

## Configuração da Camada de Cache Nginx

A camada de cache é o núcleo de toda a arquitetura, se a configuração Nginx é boa ou não, o efeito do cache varia drasticamente. Afinal, essa coisa é bastante crítica.

### Configuração do Caminho de Cache

```nginx
# Configuração do caminho de cache
proxy_cache_path /var/cache/nginx levels=1:2 keys_zone=azure_cache:10m
                   max_size=1g inactive=7d use_temp_path=off;
```

Descrição dos parâmetros:
- `levels=1:2`: níveis de diretório de cache, estrutura de diretório de 2 níveis, melhora eficiência de acesso a arquivos
- `keys_zone=azure_cache:10m`: zona de armazenamento de chaves de cache, 10MB é suficiente para armazenar muitas chaves
- `max_size=1g`: tamanho máximo de cache 1GB
- `inactive=7d`: arquivos de cache não acessados por 7 dias são deletados
- `use_temp_path=off`: escreve diretamente no diretório de cache, melhora performance

### Estratégia de Cache em Níveis

Diferentes tipos de arquivos precisam de diferentes estratégias de cache:

```nginx
# Serviço de download Server
server {
    listen 8080;

    # Cache de curto prazo para index.json (para atualizações oportunas)
    location /index.json {
        proxy_cache azure_cache;
        proxy_cache_valid 200 1h;
        proxy_cache_key "$scheme$server_port$request_uri";
        add_header X-Cache-Status $upstream_cache_status;
        add_header Cache-Control "public, max-age=3600";

        # Proxy reverso para Azure OSS
        proxy_pass https://${SERVER_DL_HOST}/${SERVER_DL_CONTAINER}$uri?${SERVER_DL_SAS_TOKEN};
        proxy_ssl_server_name on;
        proxy_ssl_protocols TLSv1.2 TLSv1.3;
    }

    # Cache de longo prazo para pacotes de instalação e outros arquivos estáticos
    location / {
        proxy_cache azure_cache;
        proxy_cache_valid 200 7d;
        proxy_cache_key "$scheme$server_port$request_uri";
        add_header X-Cache-Status $upstream_cache_status;
        add_header Cache-Control "public, max-age=604800";

        proxy_pass https://${SERVER_DL_HOST}/${SERVER_DL_CONTAINER}$uri?${SERVER_DL_SAS_TOKEN};
        proxy_ssl_server_name on;
        proxy_ssl_protocols TLSv1.2 TLSv1.3;
    }
}
```

**Por que projetado assim?**

`index.json` é um arquivo de verificação de versão, precisa ser atualizado oportunamente, então o tempo de cache foi definido como 1 hora. Assim, após lançar uma nova versão, no máximo 1 hora os usuários podem detectar a atualização, o que não é muito tempo.

Pacotes de instalação e outros arquivos estáticos mudam raramente, cache de 7 dias pode reduzir drasticamente o acesso à origem. Quando precisa atualizar, limpe manualmente o cache, o que não é muito problemático.

**Cabeçalho de Resposta X-Cache-Status**:

Este cabeçalho de resposta permite ver como está o命中率 de cache, também é bastante útil:
- `HIT`: cache atingido
- `MISS`: cache perdido, buscado da origem
- `EXPIRED`: cache expirado, buscado novamente da origem
- `BYPASS`: cache contornado

Método de visualização:
```bash
curl -I https://server.dl.hagicode.com/app.zip
```

## Análise de Custos

Assumindo tráfego de download mensal de 1TB, vamos fazer as contas:

| Solução | Custo de Tráfego | Custo de Servidor | Total |
|------|----------|------------|------|
| Azure OSS direto | cerca de ¥500 | ¥0 | ¥500 |
| Servidor em nuvem + OSS (taxa de acerto de cache 80%) | ¥100 + ¥80 | ¥60 | ¥240 |
| CDN comercial | ¥300-500 | ¥0 | ¥300-500 |

**Conclusão**: A camada de cache pode economizar cerca de 50% dos custos de distribuição.

Este cálculo assume uma taxa de acerto de cache de 80%. Na situação real, se a frequência de atualização de arquivo não é alta, a taxa de acerto pode ser ainda maior, afinal, isso também é natural.

## Prática de Implantação

### Preparação do Ambiente

Primeiro configure as variáveis de ambiente:

```bash
cd /path/to/hagicode_aliyun_deployment/docker
cp .env.example .env
vi .env  # Preencha Azure OSS SAS URL, Lark Webhook URL
```

**Importante**: O arquivo `.env` contém informações sensíveis (SAS Token, Webhook URL), nunca submeta ao controle de versão, esta coisa é bastante crítica.

### Configuração DNS

Adicione os seguintes registros DNS A, não esqueça este passo:
- `server.dl.hagicode.com` → IP do servidor
- `desktop.dl.hagicode.com` → IP do servidor

### Inicializar Servidor

Use Ansible para automatizar a inicialização do servidor:

```bash
cd /path/to/hagicode_aliyun_deployment
ansible-playbook -i ./ansible/inventory/hosts.yml ./ansible/playbooks/init.yml
```

Este Playbook automaticamente cuidará destas coisas:
- Criar usuário de implantação
- Instalar Docker e Docker Compose
- Configurar chaves SSH
- Definir regras de firewall

Não é muito complexo, afinal automação economiza tempo e esforço.

### Implantar Serviço

```bash
./deploy.sh
```

O script de implantação fará estas coisas para você:
- Verificar configuração do ambiente
- Puxar o código mais recente
- Iniciar contêineres Docker
- Executar verificações de saúde
- Enviar notificações de implantação (Feishu)

Um comando resolve, também é conveniente.

## Dicas de Operação

### Gerenciamento de Cache

O cache ocasionalmente precisa de algum cuidado:

**Verificar uso de disco do cache**:
```bash
docker volume inspect docker_nginx-cache
du -sh /var/lib/docker/volumes/docker_nginx-cache/_data
```

**Limpar cache manualmente**:
```bash
./clear-cache.sh
```

Ou execute manualmente, embora seja mais problemático, mas funciona:
```bash
docker exec nginx sh -c "rm -rf /var/cache/nginx/*"
docker restart nginx
```

### Limites de Recursos

Em servidor de 1 núcleo 2GB, a configuração de limites de recursos é a seguinte:

```yaml
services:
  traefik:
    deploy:
      resources:
        limits:
          cpus: '1.50'
          memory: 512M

  nginx:
    deploy:
      resources:
        limits:
          cpus: '0.50'
          memory: 256M
```

Monitore o uso de recursos, ocasionalmente dar uma olhada também é bom:
```bash
docker stats
```

### Práticas de Segurança de SAS Token

SAS Token é a credencial para acessar Azure Blob Storage, vazamento não é brincadeira:

- Arquivo `.env` não submetido ao controle de versão (já está em .gitignore)
- SAS Token com tempo de expiração apropriado (recomendado 1 ano)
- Limitar permissões do SAS Token (apenas leitura)
- Rotacionar SAS Token regularmente

### Monitoramento e Alertas

HagiCode integra notificações de webhook Lark/Feishu, pode enviar notificações nas seguintes situações:
- Implantação bem-sucedida/falhada
- Status de limpeza de cache
- Anomalia de serviço

As notificações incluem informações do servidor, timestamp, detalhes do erro, localização rápida de problemas também é mais conveniente.

### Extensão de Alta Disponibilidade

Quando um único servidor não suporta mais, também pode considerar:

1. **Escalonamento horizontal**: implantar múltiplos nós, distribuir através de DNS round-robin ou balanceamento de carga
2. **Suporte CDN**: acessar CDN antes do servidor em nuvem, melhorar ainda mais a velocidade de acesso
3. **Aquecimento de cache**: usar scripts para carregar arquivos populares no cache antecipadamente

## Precauções

Algumas coisas ainda precisam ser lembradas, afinal ninguém quer encontrar problemas:

1. **Certificados SSL**: Let's Encrypt tem limites de taxa, não altere implantações com frequência, caso contrário pode não conseguir obter certificados
2. **Limpeza de cache**: após atualizar arquivos importantes, lembre-se de limpar o cache, caso contrário os usuários podem não baixar a nova versão
3. **Gerenciamento de logs**: limpe logs Docker regularmente, caso contrário quando o disco encher é problema
4. **Estratégia de backup**: Traefik acme.json, configurações do Bunker Web, é melhor fazer backup
5. **Monitoramento e alertas**: configure notificações Feishu, entenda o status de implantação oportunamente, reaja rapidamente quando houver problemas

## Resumo

Servidor em nuvem + camada de cache Nginx, é simples assim. HagiCode usa esta solução, o custo não é muito alto (custo de servidor cerca de 60-100 yuans/mês), o efeito é bastante bom. As vantagens principais são aproximadamente estas:

- **Custo controlável**: comparado com usar armazenamento em nuvem diretamente ou CDN comercial, o custo caiu cerca de 50%
- **Implantação flexível**: Traefik ou Bunker Web, você escolhe
- **Forte escalabilidade**: se necessário pode escalar horizontalmente, ou adicionar um CDN
- **Operação simples**: scripts Shell + Ansible, implantação automatizada também é conveniente

Para equipes pequenas e desenvolvedores individuais que precisam de distribuição de arquivos, esta solução vale a pena tentar.

HagiCode usa esta arquitetura em produção estável por um tempo, downloads de usuários globais não tiveram grandes problemas. Se você também está procurando soluções semelhantes, por que não tentar, pode ser que ajude você também.
