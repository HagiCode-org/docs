## Adicionando Suporte à Implantação Automatizada do GitHub Pages ao HagiCode

> O codinome inicial do projeto era PCode, agora oficialmente renomeado para HagiCode. Este artigo registra como introduzimos a capacidade de implantação automatizada de site estático para o projeto, tornando a publicação de conteúdo tão fácil quanto beber água.

## Contexto / Introdução

Durante o desenvolvimento do HagiCode, encontramos um problema muito prático: à medida que a documentação e as propostas cresciam, gerenciar e apresentar esse conteúdo de forma eficiente tornou-se cada vez mais urgente. Decidimos usar o GitHub Pages para hospedar nosso site estático, mas construir e implantar manualmente era simplesmente muito trabalhoso. Cada alteração exigia uma build local, empacotamento e então um push manual para o branch `gh-pages`. Isso não era apenas ineficiente, mas também propenso a erros.

Para resolver esse problema (principalmente por preguiça), precisávamos de um fluxo de implantação automatizado. Este artigo documenta em detalhes como adicionamos suporte à implantação automatizada com GitHub Actions ao projeto HagiCode, para que possamos nos concentrar apenas na criação de conteúdo e deixar o resto para a automação.

## Análise dos Objetivos

Antes de começar, precisamos esclarecer exatamente o que esta tarefa deve realizar. Afinal, afiar o machado não atrasa o trabalho.

### Requisitos Centrais

1.  **Build automatizada**: acionar automaticamente o processo de build quando o código é enviado para o branch `main`.
2.  **Implantação automatizada**: após uma build bem-sucedida, implantar automaticamente os arquivos estáticos gerados no GitHub Pages.
3.  **Consistência de ambiente**: garantir que o ambiente de CI corresponda ao ambiente de build local, evitando a situação embaraçosa de "funciona localmente mas falha em produção".

### Escolha Tecnológica

Como o HagiCode é construído com Docusaurus (um gerador de sites estáticos em React muito popular), podemos usar GitHub Actions para alcançar esse objetivo.

## Configurando o Workflow do GitHub Actions

GitHub Actions é o serviço de CI/CD fornecido pelo GitHub. Ao definir arquivos de workflow em formato YAML dentro do repositório, podemos personalizar várias tarefas automatizadas.

### Criando o Arquivo de Workflow

Precisamos criar um novo arquivo de configuração na pasta `.github/workflows` sob o diretório raiz do projeto, por exemplo `deploy.yml`. Se a pasta não existir, lembre-se de criá-la manualmente primeiro.

A lógica central deste arquivo de configuração é a seguinte:

1.  **Condição de gatilho**: monitorar eventos de `push` no branch `main`.
2.  **Ambiente de execução**: a versão mais recente do Ubuntu.
3.  **Etapas de build**:
    -   Fazer checkout do código
    -   Instalar o Node.js
    -   Instalar dependências (`npm install`)
    -   Construir os arquivos estáticos (`npm run build`)
4.  **Etapa de implantação**: usar o `action-gh-pages` oficial para enviar os artefatos de build para o branch `gh-pages`.

### Código de Configuração Principal

Abaixo está o modelo de configuração que adotamos:

```yaml
name: Deploy to GitHub Pages

# Condição de gatilho: ao enviar para o branch main
on:
  push:
    branches:
      - main
    # Você pode adicionar filtros de caminho conforme necessário, por exemplo construir apenas quando os docs mudarem
    # paths:
    #   - 'docs/**'
    #   - 'package.json'

# Definir permissões, o que é importante para implantar no GitHub Pages
permissions:
  contents: read
  pages: write
  id-token: write

# Controle de concorrência: cancelar builds antigos no mesmo branch
concurrency:
  group: "pages"
  cancel-in-progress: false

jobs:
  build:
    runs-on: ubuntu-latest
    steps:
      - name: Checkout
        uses: actions/checkout@v4
        # Nota: você deve definir fetch-depth: 0, caso contrário a versão do build pode ser imprecisa
        with:
          fetch-depth: 0

      - name: Setup Node
        uses: actions/setup-node@v4
        with:
          node-version: 20 # Recomendado corresponder ao seu ambiente de desenvolvimento local
          cache: 'npm'     # Habilitar cache pode acelerar o processo de build

      - name: Install dependencies
        run: npm ci
        # Usar npm ci em vez de npm install porque é mais rápido, mais estrito e melhor adequado para CI

      - name: Build website
        run: npm run build
        env:
          # Se a build do seu site requer variáveis de ambiente, configure-as aqui
          # NODE_ENV: production
          # PUBLIC_URL: /your-repo-name

      - name: Upload artifact
        uses: actions/upload-pages-artifact@v3
        with:
          path: ./build # Diretório de saída padrão do Docusaurus

  deploy:
    environment:
      name: github-pages
      url: ${{ steps.deployment.outputs.page_url }}
    runs-on: ubuntu-latest
    needs: build
    steps:
      - name: Deploy to GitHub Pages
        id: deployment
        uses: actions/deploy-pages@v4
```

## Armadilhas Encontradas Durante a Implementação

Na prática, encontramos alguns problemas. Estamos compartilhando aqui na esperança de que todos possam evitá-los (ou pelo menos preparar soluções com antecedência).

### 1. Problemas de Permissão do GitHub Token

Quando configuramos inicialmente, a implantação falhava continuamente com erro 403 (Forbidden). Após uma longa investigação, descobrimos que o `GITHUB_TOKEN` padrão do GitHub não tinha permissão para gravar no Pages.

**Solução**: Nas configurações do repositório `Settings` -> `Actions` -> `General` -> `Workflow permissions`, certifique-se de escolher **"Read and write permissions"**.

### 2. Caminho Incorreto do Diretório de Build

Por padrão, o Docusaurus coloca os arquivos estáticos construídos no diretório `build`. No entanto, alguns projetos podem usar configurações diferentes. Por exemplo, o Create React App usa `build` por padrão, enquanto o Vite usa `dist`. Se o Actions relatar que não consegue encontrar arquivos, lembre-se de verificar a configuração do caminho de saída em `docusaurus.config.js`.

### 3. Problemas de Subcaminho

Se o seu repositório não é uma página de usuário (ou seja, não é `username.github.io`) mas sim uma página de projeto (como `username.github.io/nome-do-projeto`), você precisa configurar o `baseUrl`.

Em `docusaurus.config.js`:

```javascript
module.exports = {
  // ...
  url: 'https://hagicode.com', // Sua URL do HagiCode
  baseUrl: '/', // Implantação no caminho raiz
  // ...
};
```
Este detalhe é fácil de ser esquecido. Se configurado incorretamente, a página pode carregar como uma tela em branco porque os caminhos dos recursos não podem ser resolvidos.

## Verificando o Resultado

Após configurar tudo e enviar o código, podemos ir para a aba **Actions** no repositório GitHub e assistir ao espetáculo.

Você verá um círculo amarelo enquanto o workflow está em execução. Quando ficar verde, significa sucesso. Se ficar vermelho, clique nos logs para inspecionar o problema. Geralmente, é possível rastrear o problema ali, e na maioria das vezes é um erro de digitação ou uma configuração de caminho incorreta.

Após a build ter sucesso, visite `https://<seu-usuario>.github.io/<nome-do-repo>/` e você deverá ver seu site totalmente novo.

## Conclusão

Ao introduzir GitHub Actions, implementamos com sucesso a implantação automatizada do site de documentação do HagiCode. Isso não apenas economiza o tempo gasto anteriormente em operações manuais, mas mais importante, padroniza o processo de lançamento. Agora, não importa qual membro da equipe atualize a documentação, desde que as alterações sejam mescladas no branch `main`, o conteúdo mais recente aparecerá online alguns minutos depois.

**Benefícios principais**:
- **Maior eficiência**: de "empacotamento manual e upload manual" para "código é o lançamento".
- **Menos erros**: elimina a possibilidade de erros operacionais humanos.
- **Melhor experiência**: permite que os desenvolvedores se concentrem mais na qualidade do conteúdo em vez de serem distraídos por etapas tediosas de implantação.

Embora configurar CI/CD possa ser um pouco trabalhoso no início, especialmente com todas as questões de permissões e caminhos, é um investimento único com grandes retornos a longo prazo. Recomendo fortemente que todo projeto de site estático adote um workflow automatizado semelhante.
