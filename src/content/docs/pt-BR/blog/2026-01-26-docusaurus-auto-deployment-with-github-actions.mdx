## Adicionando suporte de implantação automática do GitHub Pages ao HagiCode

> Este projeto teve anteriormente o codinome PCode e agora foi oficialmente renomeado para HagiCode. Este artigo documenta como introduzir a capacidade de implantação automatizada de sites estáticos no projeto, tornando a publicação de conteúdo tão simples quanto beber água.

## Contexto/Introdução

Durante o desenvolvimento do HagiCode, enfrentamos um problema muito prático: à medida que os documentos e propostas aumentavam, como gerenciar e exibir esses conteúdo de forma eficiente tornou-se uma tarefa urgente. Decidimos introduzir o GitHub Pages para hospedar nosso site estático, mas a construção e implantação manual era muito problemática — cada alteração exigia construção local, empacotamento e envio manual para o branch `gh-pages`. Isso não só era ineficiente, mas também propenso a erros.

Para resolver este problema (principalmente para ser preguiçoso), precisávamos de um fluxo de implantação automatizado. Este artigo documentará em detalhes como adicionar suporte de implantação automática do GitHub Actions ao projeto HagiCode, permitindo que nos concentremos apenas na criação de conteúdo, deixando o resto para o processo automatizado.

## Análise de objetivos

Antes de começar, precisamos esclarecer o que esta tarefa realmente precisa fazer. Afinal (aqui foi digitado errado, deveria ser afinal) amolar o machado não atrapalha o trabalho de cortar lenha.

### Requisitos principais

1.  **Construção automatizada**: quando o código é enviado para o branch `main`, acionar automaticamente o processo de construção.
2.  **Implantação automática**: após a construção bem-sucedida, implantar automaticamente os arquivos estáticos gerados no GitHub Pages.
3.  **Consistência de ambiente**: garantir que o ambiente CI e o ambiente de construção local sejam consistentes, evitando o constrangimento de "funciona localmente, mas falha online".

### Seleção de tecnologia

Considerando que o HagiCode é construído com base no Docusaurus (um gerador de sites estáticos React muito popular), podemos usar o GitHub Actions para alcançar este objetivo.

## Configurando o fluxo de trabalho do GitHub Actions

O GitHub Actions é o serviço CI/CD fornecido pelo GitHub. Ao definir arquivos de fluxo de trabalho em formato YAML no repositório de código, podemos personalizar várias tarefas automatizadas.

### Criando o arquivo de fluxo de trabalho

Precisamos criar um novo arquivo de configuração na pasta `.github/workflows` no diretório raiz do projeto, chamado `deploy.yml`, por exemplo. Se a pasta não existir, lembre-se de criá-la manualmente primeiro.

A lógica principal deste arquivo de configuração é a seguinte:

1.  **Condição de acionamento**: monitorar o evento `push` do branch `main`.
2.  **Ambiente de execução**: versão mais recente do Ubuntu.
3.  **Etapas de construção**:
    -   Fazer checkout do código
    -   Instalar o Node.js
    -   Instalar dependências (`npm install`)
    -   Construir arquivos estáticos (`npm run build`)
4.  **Etapas de implantação**: usar o `action-gh-pages` oficial para enviar os artefatos de construção para o branch `gh-pages`.

### Código de configuração principal

Abaixo está o modelo de configuração que finalmente adotamos:

```yaml
name: Deploy to GitHub Pages

# Condição de acionamento: ao enviar para o branch main
on:
  push:
    branches:
      - main
    # Você pode adicionar filtragem de caminho conforme necessário, por exemplo, construir apenas quando a documentação for alterada
    # paths:
    #   - 'docs/**'
    #   - 'package.json'

# Definir permissões, isso é importante para implantação no GitHub Pages
permissions:
  contents: read
  pages: write
  id-token: write

# Controle de concorrência: cancelar construções antigas do mesmo branch
concurrency:
  group: "pages"
  cancel-in-progress: false

jobs:
  build:
    runs-on: ubuntu-latest
    steps:
      - name: Checkout
        uses: actions/checkout@v4
        # Nota: deve definir fetch-depth: 0, caso contrário pode causar número de versão de construção impreciso
        with:
          fetch-depth: 0

      - name: Setup Node
        uses: actions/setup-node@v4
        with:
          node-version: 20 # Recomendado manter consistente com o ambiente de desenvolvimento local
          cache: 'npm'     # Habilitar cache pode acelerar o processo de construção

      - name: Install dependencies
        run: npm ci
        # Use npm ci em vez de npm install, porque é mais rápido e mais estrito, adequado para ambiente CI

      - name: Build website
        run: npm run build
        env:
          # Se a construção do seu site precisa de variáveis de ambiente, configure aqui
          # NODE_ENV: production
          # PUBLIC_URL: /your-repo-name
          
      - name: Upload artifact
        uses: actions/upload-pages-artifact@v3
        with:
          path: ./build # Diretório de saída padrão do Docusaurus

  deploy:
    environment:
      name: github-pages
      url: ${{ steps.deployment.outputs.page_url }}
    runs-on: ubuntu-latest
    needs: build
    steps:
      - name: Deploy to GitHub Pages
        id: deployment
        uses: actions/deploy-pages@v4
```

## Problemas encontrados durante a implementação

Na operação real, encontramos alguns problemas, compartilharemos aqui esperando que você possa evitá-los (ou preparar soluções com antecedência).

### 1. Problema de permissão do GitHub Token

No início da configuração, a implantação sempre relatava erro 403 (Forbidden). Depois de procurar muito tempo, descobrimos que o `GITHUB_TOKEN` padrão do GitHub não tinha permissão para escrever no Pages.

**Solução**: nas `Settings` -> `Actions` -> `General` -> `Workflow permissions` do repositório, certifique-se de escolher **"Read and write permissions"**.

### 2. Erro no caminho do diretório de construção

O Docusaurus coloca os arquivos estáticos construídos no diretório `build` por padrão. Mas alguns projetos (como Create React App padrão é `build`, Vite padrão é `dist`) podem ter configurações diferentes. Se o Actions relatar erro ao encontrar arquivos, lembre-se de verificar a configuração do caminho de saída em `docusaurus.config.js`.

### 3. Problema de subcaminho

Se o seu repositório não é uma página de usuário (ou seja, não é `username.github.io`), mas uma página de projeto (como `username.github.io/project-name`), você precisa configurar o `baseUrl`.

Em `docusaurus.config.js`:

```javascript
module.exports = {
  // ...
  url: 'https://hagicode.com', // Seu URL do Hagicode
  baseUrl: '/', // Implantação no caminho raiz
  // ...
};
```
Este ponto é facilmente ignorado, a configuração incorreta levará a uma tela branca ao abrir a página, porque os caminhos dos recursos não podem ser carregados.

## Verificando os resultados

Depois de configurar tudo e enviar o código, podemos ir à aba **Actions** do repositório GitHub para assistir.

Você verá círculos amarelos (fluxo de trabalho em execução), quando ficar verde significa sucesso! Se ficar vermelho, clique para ver os logs, geralmente é possível investigar o problema (na maioria das vezes são erros de ortografia ou configuração de caminho incorreta).

Após a construção bem-sucedida, acesse `https://<seu-nome-de-usuário>.github.io/<nome-do-repositório>/` para ver o novo site.

## Resumo

Ao introduzir o GitHub Actions, alcançamos com sucesso a implantação automatizada do site de documentação do HagiCode. Isso não só economiza tempo de operação manual, mas o mais importante garante a padronização do processo de publicação. Agora, não importa qual amigo atualize a documentação, desde que seja mesclado no branch `main`, em poucos minutos o conteúdo mais recente pode ser visto online.

**Benefícios principais**:
- **Melhoria de eficiência**: de "empacotamento manual, upload manual" para "código como publicação".
- **Redução de erros**: elimina a possibilidade de erros operacionais humanos.
- **Otimização de experiência**: permite que os desenvolvedores se concentrem mais na qualidade do conteúdo, em vez de serem incomodados por processos de implantação complicados.

Embora a configuração de CI/CD seja um pouco problemática no início (especialmente vários problemas de permissão e caminho), é um investimento único com retorno enorme a longo prazo. É altamente recomendado que todos os projetos de sites estáticos integrem processos automatizados semelhantes.
