## Formando um Grupo de Aventuras de IA: Prática de Configuração de Colaboração Multi-Agent do HagiCode

> No desenvolvimento de software moderno, um único Agente de IA já não consegue atender necessidades complexas. Como fazer com que vários assistentes de IA de diferentes empresas trabalhem juntos no mesmo projeto? Este artigo compartilhará a solução de configuração de colaboração multi-Agent que o projeto HagiCode explorou no desenvolvimento real.

:::note[Nota sobre Contexto Temporal]
Este artigo registra a prática de configuração multi-Agent da **data de lançamento em 2026-03-17** Para o escopo de suporte residente atual, consulte [Instalação da CLI do AI Agent](/related-software-installation/ai-agent-cli/) e [Guia de Instalação](/installation); `IFlowCli` agora é mantido apenas como valor de compatibilidade histórica e não pertence ao escopo de suporte ativo atual.
:::

## Contexto

Muitos desenvolvedores provavelmente já passaram por essa experiência: o projeto introduziu um assistente de IA para auxiliar na programação e a eficiência realmente melhorou. Mas à medida que as necessidades se tornam cada vez mais complexas, um Agente de IA começa a não ser suficiente - você quer que ele processe simultaneamente revisão de código, geração de documentação, testes unitários e outras tarefas, e o resultado geralmente é cuidar de uma coisa e negligenciar outra, com qualidade de output irregular.

O que é ainda mais irritante é quando você tenta introduzir múltiplos assistentes de IA, os problemas ficam mais complexos. Cada Agente tem sua própria forma de configuração, interface API e lógica de execução, e podem até entrar em conflito uns com os outros. É como um time de futebol onde cada jogador é muito habilidoso, mas ninguém sabe como cooperar, resultando em uma bagunça total.

O projeto HagiCode também enfrentou o mesmo dilema durante o desenvolvimento. Como um projeto complexo envolvendo extensão VSCode frontend, serviços de AI backend e cliente desktop multiplataforma, precisamos integrar simultaneamente múltiplos assistentes de IA de diferentes empresas: Claude Code, Codex, CodeBuddy, iFlow, entre outros. Como fazê-los coexistir harmoniosamente no mesmo projeto e exercer seus pontos fortes tornou-se um problema-chave que precisávamos resolver.

Na verdade, até que tá bom, afinal quem quer lidar com um grupo de IAs brigando todos os dias, né.

A solução compartilhada neste artigo é exatamente a prática de configuração de colaboração multi-Agent que refinamos e otimizamos no projeto HagiCode. Se você também está sofrendo com colaboração de múltiplos assistentes de IA, acredito que este artigo lhe dará algumas inspirações. Talvez, afinal as situações de cada pessoa são diferentes.

## Sobre o HagiCode

[HagiCode](https://hagicode.com) é um projeto de assistente de código de IA que adota o modo de "grupo de aventuras" com múltiplos motores de AI trabalhando em colaboração. Endereço do projeto: [github.com/HagiCode-org/site](https://github.com/HagiCode-org/site).

A solução de configuração multi-Agent compartilhada neste artigo é exatamente uma das tecnologias centrais que permitem ao HagiCode manter desenvolvimento eficiente em projetos complexos. Não há nada de especial, é só transformar um grupo de IAs em um grupo de aventuras que sabe trabalhar em equipe.

## Design de Arquitetura Multi-Agent do HagiCode

### De "Luta Individual" para "Trabalho em Equipe"

No estágio inicial do projeto HagiCode, também tentamos usar apenas um Agente de IA para lidar com todas as tarefas. Logo descobrimos que essa abordagem tinha gargalos óbvios: diferentes tarefas precisam de diferentes pontos focais de capacidade, algumas tarefas precisam de capacidade de compreensão de contexto mais forte, outras precisam de capacidade de modificação de código mais precisa. Um Agente dificilmente pode se sobressair em todos os aspectos.

Isso nos fez perceber que precisamos fazer múltiplos Agents trabalharem em colaboração. Mas o problema é: como fazer produtos de IA de diferentes empresas coexistirem pacificamente no mesmo projeto? Precisamos resolver alguns problemas centrais:

1. **Complexidade de Gerenciamento de Configuração**: Cada Agente tem diferentes formas de configuração, interfaces API e padrões de execução
2. **Unificação de Protocolo de Comunicação**: É necessário uma maneira padronizada permitir troca de dados entre diferentes Agents
3. **Coordenação de Divisão de Tarefas**: Como alocar razoavelmente tarefas para que cada Agente exercite seus pontos fortes

Com esses problemas, começamos a desenhar a arquitetura multi-Agent do HagiCode. Na verdade não é tão complexo, é só entender direito.

### Visão Geral da Arquitetura

Após múltiplas iterações, a arquitetura que finalmente determinamos é esta:

```
┌─────────────────────────────────────────────────────────────────┐
│                    AIProviderFactory                             │
│  (Padrão Factory gerencia uniformemente todos os AI Providers)   │
├─────────────────────────────────────────────────────────────────┤
│  ClaudeCodeCli  │  CodexCli  │  CodebuddyCli  │  IFlowCli    │
│  (Anthropic)   │  (OpenAI)  │  (Zhipu GLM)    │  (Zhipu)      │
└─────────────────────────────────────────────────────────────────┘
```

A ideia central é: através da interface Provider unificada, diferentes Agents de IA podem ser gerenciados pelo mesmo conjunto de código. Ao mesmo tempo, usa-se o padrão Factory para criar e configurar dinamicamente esses Providers, garantindo escalabilidade e flexibilidade do sistema.

É como a divisão de trabalho na vida, cada pessoa tem seu próprio papel, só que aqui transformamos essa divisão de trabalho em arquitetura de código.

### Tipos de Agent e Divisão de Responsabilidades

Com base na experiência de uso real do projeto HagiCode, alocamos responsabilidades diferentes para cada Agente:

| Agente | Provedor | Modelo | Uso Principal |
|--------|----------|--------|---------------|
| ClaudeCodeCli | Anthropic | glm-5-turbo | Gera soluções técnicas e Propostas |
| CodexCli | OpenAI/Zed | gpt-5.4 | Executa modificações de código precisas |
| CodebuddyCli | Zhipu | glm-4.7 | Otimiza descrições de propostas e documentação |
| IFlowCli | Zhipu | glm-4.7 | Arquiva propostas e registros históricos |
| OpenCodeCli | - | - | Edição de código geral |
| GitHubCopilot | Microsoft | - | Programação assistida e autocomplete de código |

A lógica por trás dessa divisão é: cada Agente tem sua própria área de especialização. Claude Code se destaca na compreensão e análise de requisitos complexos, então o deixamos responsável pelo design de solução inicial; Codex é mais preciso em modificações de código, adequado para lidar com tarefas de implementação específicas; CodeBuddy tem bom custo-benefício, perfeito para otimizar documentação.

Afinal, o que se adequa a você é o melhor, todos os caminhos levam a Roma, só que alguns caminhos são mais fáceis, outros um pouco mais tortuosos.

## Mecanismo de Configuração Central

### Design de Interface Provider Unificada

Para fazer diferentes Agents de IA serem gerenciados uniformemente, primeiro é necessário definir um conjunto de interfaces unificadas. No HagiCode definimos esta interface:

```csharp
public interface IAIProvider
{
    // Interface Provider unificada
    Task<IAIProvider?> GetProviderAsync(AIProviderType providerType);
    Task<IAIProvider?> GetProviderAsync(string providerName, CancellationToken cancellationToken);
}
```

Esta interface parece simples, mas é a pedra angular de todo o sistema multi-Agent. Através da interface unificada, podemos ignorar qual produto de IA está por baixo e chamá-los todos da mesma forma.

Na verdade, é só simplificar coisas complexas, afinal simples é belo.

### Implementação do Padrão Factory de Provider

Com a interface unificada, o próximo passo é como criar essas instâncias de Provider. O HagiCode usa o padrão Factory:

```csharp
private IAIProvider? CreateProvider(AIProviderType providerType, ProviderConfiguration config)
{
    return providerType switch
    {
        AIProviderType.ClaudeCodeCli =>
            ActivatorUtilities.CreateInstance<ClaudeCodeCliProvider>(_serviceProvider, Options.Create(config)),
        AIProviderType.CodebuddyCli =>
            ActivatorUtilities.CreateInstance<CodebuddyCliProvider>(_serviceProvider, Options.Create(config)),
        AIProviderType.CodexCli =>
            ActivatorUtilities.CreateInstance<CodexCliProvider>(_serviceProvider, Options.Create(config)),
        AIProviderType.IFlowCli =>
            ActivatorUtilities.CreateInstance<IFlowCliProvider>(_serviceProvider, Options.Create(config)),
        _ => null
    };
}
```

Aqui usa-se `ActivatorUtilities.CreateInstance` de injeção de dependência, que pode criar dinamicamente instâncias Provider em tempo de execução e automaticamente injetar dependências. O benefício deste design é: ao adicionar um novo tipo de Agente, só precisa adicionar a classe Provider correspondente e adicionar um case no método Factory, sem precisar modificar código existente.

Isso é aceitável, afinal quem quer modificar um monte de código antigo toda vez que adiciona nova funcionalidade.

### Resolução de Configuração Dinâmica

Para tornar a configuração mais flexível, também implementamos um mecanismo de mapeamento de tipos:

```csharp
public static AIProviderTypeExtensions
{
    private static readonly Dictionary<string, AIProviderType> _typeMap = new(
        StringComparer.OrdinalIgnoreCase)
    {
        ["ClaudeCodeCli"] = AIProviderType.ClaudeCodeCli,
        ["CodebuddyCli"] = AIProviderType.CodebuddyCli,
        ["CodexCli"] = AIProviderType.CodexCli,
        ["IFlowCli"] = AIProviderType.IFlowCli,
        // ...mais mapeamentos de tipos
    };
}
```

O papel desta tabela de mapeamento é converter nomes de Provider em formato de string para tipos enumerados. Dessa forma, os arquivos de configuração podem usar nomes de string intuitivos, enquanto o código interno usa enums type-safe para processamento.

Afinal, configuração, quanto mais intuitiva melhor, quem quer memorizar um monte de código complexo.

### Exemplo de Arquivo de Configuração

No uso real, só precisa configurar em `appsettings.json`:

```yaml
AI:
  Providers:
    Providers:
      ClaudeCodeCli:
        Enabled: true
        Model: glm-5-turbo
        WorkingDirectory: /path/to/project
      CodebuddyCli:
        Enabled: true
        Model: glm-4.7
      CodexCli:
        Enabled: true
        Model: gpt-5.4
      IFlowCli:
        Enabled: true
        Model: glm-4.7
```

Cada Provider pode ser configurado independentemente com parâmetros como switch, versão do modelo, diretório de trabalho, etc. Este design garante flexibilidade e facilita gerenciamento e manutenção.

Na verdade, arquivos de configuração são como opções na vida, você pode escolher ligar ou desligar certas funcionalidades, só que as escolhas no código são mais fáceis de se arrepender.

## Fluxo de Tarefas do Grupo de Aventuras

### A Arte da Divisão de Tarefas

Com a arquitetura técnica unificada, o próximo passo é fazer múltiplos Agents trabalharem em colaboração. O HagiCode desenhou um mecanismo de fluxo de tarefas para que diferentes Agents lidem com tarefas de diferentes estágios:

```
Criação de Proposta (usuário)
    │
    ▼
[Claude Code] ──gera proposta──▶ Documento de proposta
    │                        │
    │                        ▼
    │               [Codebuddy] ──otimiza descrição──▶ Proposta otimizada
    │                        │
    │                        ▼
    │               [Codex] ──executa modificação──▶ Mudança de código
    │                        │
    │                        ▼
    └───────────────▶ [iFlow] ──arquiva──▶ Registro histórico
```

O benefício desta divisão é: cada Agente só precisa focar na tarefa que é bom, não precisa "saber tudo". Claude Code é responsável por gerar propostas do zero, Codebuddy é responsável por tornar as descrições das propostas mais claras, Codex é responsável por transformar propostas em mudanças de código reais, iFlow é responsável por arquiving essas mudanças.

Na verdade, é como trabalho em equipe na vida, cada pessoa tem seu próprio papel, só juntos podem completar algo grande. Só que aqui os membros da equipe são IAs.

### Pontos Chave da Prática

Na operação real, resumimos as seguintes experiências:

**1. Estratégia de Seleção de Agent é Importante**

Não é alocar tarefas aleatoriamente, mas alocar com base nos pontos fortes de cada Agente:
- Geração de propostas: Use Claude Code, porque tem capacidade de compreensão de contexto mais forte
- Execução de código: Use Codex, porque é mais preciso em modificações de código
- Otimização de propostas: Use Codebuddy, porque tem bom custo-benefício
- Armazenamento de arquivo: Use iFlow, porque é estável e confiável

Afinal, deixar a pessoa certa fazer a coisa certa é uma verdade eterna.

**2. Isolamento de Configuração Garante Estabilidade**

A configuração de cada Agente é gerenciada independentemente, suporta override por variáveis de ambiente, e os diretórios de trabalho também são independentes. Dessa forma, erro de configuração de um Agente não afeta outros Agents.

É como limites na vida, cada pessoa tem seu próprio espaço, sem interferência mútua para coexistir harmoniosamente.

**3. Mecanismo de Tratamento de Erros**

Falha de um único Agente não deve afetar o fluxo geral. Implementamos estratégias de degradação: quando um Agente falha na execução, o sistema pode mudar automaticamente para solução alternativa, ou pular essa etapa e continuar executando tarefas subsequentes. Ao mesmo tempo, registros completos de log facilitam investigação posterior de problemas.

Afinal ninguém pode garantir que nunca errará, o importante é como tratar erros. É como a vida, sempre haverá contratempos, o importante é como superar.

**4. Monitoramento e Observabilidade**

Através do protocolo ACP (nosso protocolo de comunicação personalizado, baseado em JSON-RPC 2.0), é possível rastrear o status de execução de cada Agente. Isolamento de sessão garante segurança de concorrência, cache dinâmico otimiza performance.

Afinal, coisas invisíveis são as mais fáceis de dar problema, ter um pouco de monitoramento é melhor do que estar no escuro total.

## Resultados e Benefícios Reais

Após adotar esta configuração de colaboração multi-Agent, a eficiência de desenvolvimento do projeto HagiCode melhorou significativamente. Especificamente:

1. **Capacidade de Processamento de Tarefas Dobrada**: Antes um Agente precisava lidar com múltiplas tarefas simultaneamente, agora pode processar em paralelo, throughput dobrou mais do que duplo
2. **Qualidade de Output Mais Estável**: Cada Agente só foca na tarefa que é bom, consistência e qualidade dos resultados são maiores
3. **Custo de Manutenção Reduzido**: Interface e gerenciamento de configuração unificados tornam todo o sistema mais fácil de manter e expandir
4. **Adicionar Novo Agents é Simples**: Para integrar novo produto de IA, só precisa implementar interface, adicionar configuração, sem precisar modificar lógica central

Esta solução não apenas resolveu os problemas do próprio HagiCode, mas também provou que colaboração multi-Agent é realmente uma escolha de arquitetura viável.

Na verdade, o efeito é bastante óbvio, só que o processo foi um pouco trabalhoso.

## Conclusão

Este artigo compartilhou experiência prática em configuração de colaboração multi-Agent do projeto HagiCode. Pontos centrais incluem:

1. **Interface Padronizada**: Através de `IAIProvider` unificar comportamentos de diferentes Agents, permitindo que código ignore qual produto está por baixo
2. **Padrão Factory**: Usa `ActivatorUtilities.CreateInstance` para criar dinamicamente instâncias Provider, suportando configuração em tempo de execução e injeção de dependência
3. **Protocolo Unificado**: Protocolo ACP implementa comunicação padronizada entre Agents, mecanismo de comunicação bidirecional baseado em JSON-RPC 2.0
4. **Divisão de Fluxo de Tarefas**: Aloca razoavelmente tarefas para diferentes Agents, deixando-os exercitar pontos fortes, em vez de tentar fazer um Agente fazer tudo

Este design não apenas resolveu o problema de "múltiplos Agents brigando", mas também através do mecanismo de fluxo de tarefas do grupo de aventuras, realizou automação e especialização do fluxo de desenvolvimento.
