## Implementando sincronização automática de imagens do Docker Hub para Azure ACR

> Este artigo apresenta como usar GitHub Actions e a ferramenta image-syncer para implementar a sincronização automática de imagens do Docker Hub para o Azure Container Registry, resolvendo o problema de lentidão no acesso ao Docker Hub na China e em algumas regiões do Azure, melhorando a disponibilidade de imagens e a eficiência de implantação em ambientes Azure.

## Contexto/Introdução

O projeto HagiCode usa imagens Docker como componentes principais de runtime, com imagens principais hospedadas no Docker Hub. Com o desenvolvimento do projeto e o aumento das demandas de implantação em ambientes Azure, enfrentamos os seguintes desafios:
- Lentidão na pulling de imagens, acesso limitado ao Docker Hub na China e em algumas regiões do Azure
- Risco de ponto único de falha ao depender de uma única fonte de imagens
- O uso do Azure Container Registry em ambientes Azure proporciona melhor desempenho de rede e experiência de integração

Para resolver esses problemas, precisamos estabelecer um mecanismo de sincronização automática de imagens, sincronizando periodicamente as imagens do Docker Hub para o Azure ACR, garantindo que os usuários possam obter velocidades de pulling mais rápidas e maior disponibilidade em ambientes Azure.

## Comparação de soluções técnicas

Ao desenvolver a solução, comparamos várias abordagens técnicas:

### 1. image-syncer (escolha final)
- Sincronização incremental: sincroniza apenas as camadas de imagens alteradas, reduzindo significativamente a transferência de rede
- Retoma de quebra: pode retomar a sincronização após interrupção da rede
- Controle de concorrência: suporta configuração do número de threads simultâneas, melhorando a eficiência de sincronização de imagens grandes
- Tratamento de erros robusto: mecanismo integrado de retry em caso de falha (padrão 3 vezes)
- Implantação leve: único arquivo binário, sem dependências
- Suporte a múltiplos repositórios: compatível com Docker Hub, Azure ACR, Harbor, etc.

### 2. Docker CLI
- Não suporta sincronização incremental: precisa puxar o conteúdo completo da imagem toda vez
- Eficiência baixa: grande volume de transferência de rede, tempo longo
- Fácil de usar: usa comandos familiares docker pull/push

### 3. Azure CLI
- Alta complexidade: precisa configurar autenticação do Azure CLI
- Limitações funcionais: função az acr import relativamente limitada
- Integração nativa: integra-se bem com serviços Azure

## Decisões de design de arquitetura

### Decisão 1: Frequência de sincronização definida para diariamente às 00:00 UTC
- Equilibra atualização de imagens e consumo de recursos
- Evita horários de pico de negócios, reduzindo o impacto em outras operações
- Imagens do Docker Hub geralmente são atualizadas após builds diários

### Decisão 2: Sincronizar todas as tags de imagens
- Mantém consistência completa com o Docker Hub
- Fornece escolhas flexíveis de versão para os usuários
- Simplifica a lógica de sincronização, evitando regras complexas de filtragem de tags

### Decisão 3: Usar GitHub Secrets para armazenar informações de autenticação
- Suporte nativo do GitHub Actions, alta segurança
- Configuração simples, fácil de gerenciar e manter
- Suporta controle de acesso em nível de repositório

## Avaliação e mitigação de riscos

### Risco 1: Vazamento de informações de autenticação do Azure ACR
- Usar GitHub Secrets para armazenamento criptografado
- Rotacionar regularmente a senha do ACR
- Limitar permissões de usuário do ACR apenas para push
- Monitorar logs de acesso do ACR

### Risco 2: Falha de sincronização causando inconsistência de imagens
- image-syncer possui mecanismo integrado de sincronização incremental
- Retry automático em caso de falha (padrão 3 vezes)
- Logs de erro detalhados e notificações de falha
- Funcionalidade de retoma de quebra

### Risco 3: Consumo excessivo de recursos
- Sincronização incremental reduz transferência de rede
- Número de threads configurável (atualmente definido para 10)
- Monitorar quantidade e tamanho das imagens sincronizadas
- Executar sincronização em horários fora do pico

## Solução principal

Adotamos a solução automatizada GitHub Actions + image-syncer para implementar a sincronização de imagens do Docker Hub para o Azure ACR.

## Passos de implementação

### 1. Fase de preparação
- Criar ou confirmar o Azure Container Registry no Azure Portal
- Criar chaves de acesso do ACR (nome de usuário e senha)
- Confirmar permissões de acesso ao repositório de imagens do Docker Hub

### 2. Configurar GitHub Secrets
Adicione os seguintes Secrets nas configurações do repositório GitHub:
- AZURE_ACR_USERNAME: Nome de usuário do Azure ACR
- AZURE_ACR_PASSWORD: Senha do Azure ACR

### 3. Criar workflow do GitHub Actions
Configure o workflow em .github/workflows/sync-docker-acr.yml:
- Trigger agendado: diariamente às 00:00 UTC
- Trigger manual: suporta workflow_dispatch
- Trigger adicional: acionado ao push para branch publish (para sincronização rápida)

### 4. Fluxo de execução do workflow

| Ordem | Participante | Ação | Descrição |
| --- | --- | --- | --- |
| 1 | GitHub Actions | Acionar workflow | Acionado por agendamento, manual ou push para branch `publish` |
| 2 | GitHub Actions → image-syncer | Baixar e executar ferramenta de sincronização | Entrar na fase de sincronização real |
| 3 | image-syncer → Docker Hub | Obter manifesto e lista de tags da imagem | Ler metadados do repositório de origem |
| 4 | image-syncer → Azure ACR | Obter informações de imagens existentes no repositório de destino | Determinar estado atual do lado de destino |
| 5 | image-syncer | Comparar diferenças entre origem e destino | Identificar camadas de imagens que precisam ser sincronizadas |
| 6 | image-syncer → Docker Hub | Puxar camadas de imagens alteradas | Transferir apenas o conteúdo que precisa ser atualizado |
| 7 | image-syncer → Azure ACR | Push de camadas de imagens alteradas | Completar sincronização incremental |
| 8 | image-syncer → GitHub Actions | Retornar estatísticas de sincronização | Incluindo resultados, diferenças e informações de erro |
| 9 | GitHub Actions | Registrar logs e fazer upload de artifact | Facilitar auditoria e troubleshooting posterior |

## Implementação do workflow do GitHub Actions

Abaixo está a configuração do workflow realmente em execução (.github/workflows/sync-docker-acr.yml):

```yaml
name: Sync Docker Image to Azure ACR

on:
  schedule:
    - cron: "0 0 * * *" # Diariamente às 00:00 UTC
  workflow_dispatch: # Trigger manual
  push:
    branches: [publish]

permissions:
  contents: read

jobs:
  sync:
    runs-on: ubuntu-latest

    steps:
      - name: Checkout code
        uses: actions/checkout@v4

      - name: Download image-syncer
        run: |
          # Baixar binário image-syncer
          wget https://github.com/AliyunContainerService/image-syncer/releases/download/v1.5.5/image-syncer-v1.5.5-linux-amd64.tar.gz
          tar -zxvf image-syncer-v1.5.5-linux-amd64.tar.gz
          chmod +x image-syncer

      - name: Create auth config
        run: |
          # Gerar arquivo de configuração de autenticação (formato YAML)
          cat > auth.yaml <<EOF
          hagicode.azurecr.io:
            username: "${{ secrets.AZURE_ACR_USERNAME }}"
            password: "${{ secrets.AZURE_ACR_PASSWORD }}"
          EOF

      - name: Create images config
        run: |
          # Gerar arquivo de configuração de sincronização de imagens (formato YAML)
          cat > images.yaml <<EOF
          docker.io/newbe36524/hagicode: hagicode.azurecr.io/hagicode
          EOF

      - name: Run image-syncer
        run: |
          # Executar sincronização (usando parâmetros --auth e --images da nova versão)
          ./image-syncer --auth=./auth.yaml --images=./images.yaml --proc=10 --retries=3

      - name: Upload logs
        if: always()
        uses: actions/upload-artifact@v4
        with:
          name: sync-logs
          path: image-syncer-*.log
          retention-days: 7
```

## Explicação da configuração

### 1. Condições de trigger
- Trigger agendado: cron: "0 0 * * *" - Executa diariamente às 00:00 UTC
- Trigger manual: workflow_dispatch - Permite que usuários executem manualmente na interface do GitHub
- Trigger de push: push: branches: [publish] - Acionado ao push para branch publish (para sincronização rápida)

### 2. Configuração de autenticação (auth.yaml)
```yaml
hagicode.azurecr.io:
  username: "${{ secrets.AZURE_ACR_USERNAME }}"
  password: "${{ secrets.AZURE_ACR_PASSWORD }}"
```

### 3. Configuração de sincronização de imagens (images.yaml)
```yaml
docker.io/newbe36524/hagicode: hagicode.azurecr.io/hagicode
```
Esta configuração indica sincronizar todas as tags de docker.io/newbe36524/hagicode para hagicode.azurecr.io/hagicode

### 4. Parâmetros do image-syncer
- --auth=./auth.yaml: Caminho do arquivo de configuração de autenticação
- --images=./images.yaml: Caminho do arquivo de configuração de sincronização de imagens
- --proc=10: Número de threads simultâneas é 10
- --retries=3: Retry 3 vezes em caso de falha

## Considerações e melhores práticas

### 1. Recomendações de segurança
- Rotacionar regularmente a senha do Azure ACR (recomendado a cada 90 dias)
- Usar conta de serviço dedicada do ACR, limitar permissões apenas para push
- Monitorar logs de acesso do ACR, detectar acessos anômalos oportunamente
- Não imprimir informações de autenticação nos logs
- Não commitar informações de autenticação no repositório de código

### 2. Otimização de desempenho
- Ajustar parâmetro --proc: Ajustar concorrência com base na largura de banda da rede (recomendado 5-20)
- Monitorar tempo de sincronização: Se o tempo de sincronização for muito longo, considere reduzir a concorrência
- Limpar logs regularmente: Definir retention-days razoável (atualmente 7 dias)

### 3. Troubleshooting

#### Problema 1: Falha de autenticação
```
Error: failed to authenticate to hagicode.azurecr.io
```
Solução:
1. Verificar se os GitHub Secrets estão configurados corretamente
2. Verificar se a senha do Azure ACR expirou
3. Confirmar se as permissões da conta de serviço do ACR estão corretas

#### Problema 2: Timeout de rede
```
Error: timeout waiting for response
```
Solução:
1. Verificar conexão de rede
2. Reduzir número de threads simultâneas (parâmetro --proc)
3. Aguardar recuperação da rede e acionar o workflow novamente

#### Problema 3: Sincronização de imagens incompleta
```
Warning: some tags failed to sync
```
Solução:
1. Verificar logs de sincronização, identificar tags com falha
2. Acionar workflow manualmente para ressincronizar
3. Verificar se a imagem de origem do Docker Hub está normal

### 4. Monitoramento e alertas
- Verificar regularmente a página Actions, confirmar status de execução do workflow
- Configurar notificações do GitHub, obter notificações de falha do workflow oportunamente
- Monitorar uso de armazenamento do Azure ACR
- Verificar regularmente consistência de tags de imagens

## Perguntas frequentes e soluções

### Q1: Como sincronizar tags específicas em vez de todas as tags?

Modifique o arquivo de configuração images.yaml:
```yaml
# Sincronizar apenas tags latest e v1.0
docker.io/newbe36524/hagicode:latest: hagicode.azurecr.io/hagicode:latest
docker.io/newbe36524/hagicode:v1.0: hagicode.azurecr.io/hagicode:v1.0
```

### Q2: Como sincronizar múltiplos repositórios de imagens?

Adicione múltiplas linhas de configuração em images.yaml:
```yaml
docker.io/newbe36524/hagicode: hagicode.azurecr.io/hagicode
docker.io/newbe36524/another-image: hagicode.azurecr.io/another-image
```

### Q3: Como tentar novamente após falha de sincronização?
- Retry automático: image-syncer possui mecanismo integrado de retry (padrão 3 vezes)
- Retry manual: Clique em Re-run all jobs na página GitHub Actions

### Q4: Como ver o progresso detalhado da sincronização?
- Ver logs em tempo real na página Actions
- Baixar artifact sync-logs para ver o arquivo de logs completo
- O arquivo de logs contém status de sincronização e velocidade de transferência de cada tag

### Q5: Quanto tempo leva a sincronização?
- Primeira sincronização completa: Dependendo do tamanho da imagem, geralmente 10-30 minutos
- Sincronização incremental: Se as alterações da imagem forem pequenas, geralmente 2-5 minutos
- O tempo depende da largura de banda da rede, tamanho da imagem e configurações de concorrência
