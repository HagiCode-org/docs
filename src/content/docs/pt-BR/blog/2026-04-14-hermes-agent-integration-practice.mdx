## Prática de Integração do Hermes Agent: do Protocolo à Produção

> Compartilha a prática completa de integração do Hermes Agent no HagiCode, incluindo adaptação do protocolo ACP, gerenciamento de pool de sessões, sincronização de contratos front-end/back-end e outras experiências essenciais.

## Contexto

Durante a construção da plataforma de codificação assistida por IA HagiCode, a equipe precisava integrar um framework Agent que pudesse ser executado localmente e também escalado para a nuvem. Após pesquisa, o Hermes Agent da Nous Research foi selecionado como o mecanismo subjacente para o Agent integrado.

Na verdade, a escolha de tecnologia não é nem tão difícil nem tão simples. Afinal, existem bastantes frameworks Agent competentes no mercado, mas o protocolo ACP e o sistema de ferramentas do Hermes realmente têm algo especial, que se encaixa perfeitamente no cenário de requisitos "ter tudo ao mesmo tempo" do HagiCode — desenvolvimento local, colaboração em equipe e expansão na nuvem. Mas para integrar o Hermes verdadeiramente em um sistema de produção, é necessário resolver uma série de problemas de engenharia, o que não é brincadeira.

A stack tecnológica do HagiCode é baseada no Orleans para construir sistemas distribuídos, com front-end usando React + TypeScript. A integração do Hermes requer manter a unidade da arquitetura existente, tornando o Hermes um executor de "primeira classe" em paralelo com ClaudeCode, OpenCode, etc. Parece fácil, mas na prática é assim que é.

Este artigo compartilha nossa experiência prática na integração do Hermes Agent no projeto HagiCode, esperando fornecer referência para equipes com necessidades semelhantes. Afinal, os obstáculos que superamos não precisam ser superados novamente por outros.

## Sobre o HagiCode

A solução compartilhada neste artigo vem de nossa experiência prática no projeto [HagiCode](https://hagicode.com). O HagiCode é uma plataforma de assistência de codificação movida por IA, suportando acesso unificado e gerenciamento de múltiplos Providers de IA. Durante o processo de integração do Hermes Agent, desenhamos uma camada de abstração de Provider genérica, permitindo que novos tipos de Agent se integrem perfeitamente ao sistema existente.

## Design de Arquitetura

### Abordagem de Design em Camadas

A integração do Hermes no HagiCode adota uma arquitetura em camadas clara, onde cada camada tem sua responsabilidade:

**Camada Core Back-end**
- `HermesCliProvider`: Implementa a interface `IAIProvider`, servindo como entrada unificada de AI Provider
- `HermesPlatformConfiguration`: Gerencia caminhos de executáveis, parâmetros, autenticação e outras configurações do Hermes
- `ICliProvider<HermesOptions>`: Abstração CLI de baixo nível fornecida pelo HagiCode.Libs, tratando o ciclo de vida de subprocessos

**Camada de Transporte**
- `StdioAcpTransport`: Comunica com subprocessos Hermes ACP através de entrada/saída padrão
- Métodos do protocolo ACP: `initialize`, `authenticate`, `session/new`, `session/prompt`

**Camada de Runtime**
- `HermesGrain`: Implementação de Orleans Grain, processando execução de sessões distribuídas
- `CliAcpSessionPool`: Pool de sessões, reutiliza subprocessos ACP, evitando sobrecarga de inicialização frequente

**Camada Front-end**
- `ExecutorAvatar`: Identidade visual e ícone do Hermes
- `executorTypeAdapter`: Lógica de mapeamento de tipos de Provider
- Transmissão de mensagens em tempo real SignalR: mantém a consistência da identidade Hermes no fluxo de mensagens

Este design em camadas permite que cada camada evolua independentemente. Por exemplo, para adicionar novos métodos de transporte no futuro (como WebSocket), basta modificar a camada de transporte. Afinal, quem quer refazer todo o sistema só por mudar um método de transporte? Que cansaço.

### Abstração de Interface Unificada

Todos os AI Providers implementam a interface `IAIProvider`, que é o design central da arquitetura HagiCode:

```csharp
public interface IAIProvider
{
    string Name { get; }
    ProviderCapabilities Capabilities { get; }

    IAsyncEnumerable<AIStreamingChunk> StreamAsync(
        AIRequest request,
        CancellationToken cancellationToken = default);

    Task<AIResponse> ExecuteAsync(
        AIRequest request,
        CancellationToken cancellationToken = default);
}
```

`HermesCliProvider` implementa esta interface, estando em pé de igualdade com `ClaudeCodeProvider`, `OpenCodeProvider`, etc. Os benefícios deste design:

1. **Substituibilidade**: Trocar de Provider não afeta a lógica de negócios da camada superior
2. **Testabilidade**: Pode facilmente fazer Mock de Provider para testes unitários
3. **Extensibilidade**: Adicionar novo Provider só precisa implementar a interface

No fim das contas, interfaces são como regras. Com regras, todos podem coexistir harmoniosamente, cada um exercitando seus pontos fortes, sem interferência mútua. Isso não é uma forma de beleza?

## Implementação Core

### Implementação da Camada Provider

`HermesCliProvider` é o núcleo de toda a integração, responsável por coordenar os componentes para completar uma chamada de IA:

```csharp
public sealed class HermesCliProvider : IAIProvider, IVersionedAIProvider
{
    private readonly ICliProvider<LibsHermesOptions> _provider;
    private readonly ConcurrentDictionary<string, string> _sessionBindings;

    public ProviderCapabilities Capabilities { get; } = new()
    {
        SupportsStreaming = true,
        SupportsTools = true,
        SupportsSystemMessages = true,
        SupportsArtifacts = false
    };

    public async IAsyncEnumerable<AIStreamingChunk> StreamAsync(
        AIRequest request,
        [EnumeratorCancellation] CancellationToken cancellationToken = default)
    {
        // 1. Resolve a chave de binding de sessão
        var bindingKey = ResolveBindingKey(request.CessionId);

        // 2. Obtém ou cria sessão Hermes através do pool de sessões
        var options = new HermesOptions
        {
            ExecutablePath = _platformConfiguration.ExecutablePath,
            Arguments = _platformConfiguration.Arguments,
            SessionId = _sessionBindings.TryGetValue(bindingKey, out var sessionId) ? sessionId : null,
            WorkingDirectory = request.WorkingDirectory,
            Model = request.Model
        };

        // 3. Executa e coleta resposta streaming
        await foreach (var message in _provider.ExecuteAsync(options, request.Prompt, cancellationToken))
        {
            // 4. Mapeia mensagens ACP para AIStreamingChunk
            if (_responseMapper.TryConvertToStreamingChunk(message, out var chunk))
            {
                yield return chunk;
            }
        }
    }
}
```

Aqui estão alguns pontos chave de design:

1. **Binding de Sessão**: Através de `CessionId`, vincula múltiplas requisições ao mesmo subprocesso Hermes, implementando continuidade de contexto em conversas multi-turno
2. **Mapeamento de Resposta**: Converte o formato de mensagem Hermes ACP para o formato unificado `AIStreamingChunk`
3. **Processamento Streaming**: Usa `IAsyncEnumerable` para suportar verdadeiras respostas streaming

Na verdade, binding de sessão é como o relacionamento entre pessoas. Uma vez estabelecida a conexão, a comunicação subsequente tem contexto, sem precisar começar do zero toda vez. Só que este relacionamento precisa ser bem mantido, caso contrário, se quebra, se quebra.

### Adaptação do Protocolo ACP

O Hermes usa o protocolo ACP (Agent Communication Protocol), diferente das APIs HTTP tradicionais. ACP é um protocolo baseado em entrada/saída padrão, com várias características:

1. **Marcador de Inicialização**: O processo Hermes envia o marcador `//ready` após iniciar
2. **Autenticação Dinâmica**: O método de autenticação não é fixo, precisa ser negociado através do protocolo
3. **Reutilização de Sessão**: Reutiliza sessões estabelecidas através de `SessionId`
4. **Resposta Dispersa**: A resposta completa pode estar dispersa em múltiplas notificações `session/update`

O HagiCode lida com essas características através de `StdioAcpTransport`:

```csharp
public class StdioAcpTransport
{
    public async Task InitializeAsync(CancellationToken cancellationToken)
    {
        // Aguarda marcador //ready
        var readyLine = await _outputReader.ReadLineAsync(cancellationToken);
        if (readyLine != "//ready")
        {
            throw new InvalidOperationException("Hermes did not send ready signal");
        }

        // Envia requisição initialize
        await SendRequestAsync(new
        {
            jsonrpc = "2.0",
            id = 1,
            method = "initialize",
            @params = new
            {
                protocolVersion = "2024-11-05",
                capabilities = new { },
                clientInfo = new { name = "HagiCode", version = "1.0.0" }
            }
        }, cancellationToken);
    }
}
```

Protocolos são como a química entre pessoas. Com química, a comunicação flui muito melhor. Só que estabelecer química leva tempo, ajuste mútuo, todo mundo passa por isso.

### Gerenciamento de Pool de Sessões

Iniciar subprocessos Hermes frequentemente tem sobrecarga significativa, por isso implementamos um mecanismo de pool de sessões:

```csharp
services.AddSingleton(static _ =>
{
    var registry = new CliProviderPoolConfigurationRegistry();
    registry.Register("hermes", new CliPoolSettings
    {
        MaxActiveSessions = 50,
        IdleTimeout = TimeSpan.FromMinutes(10)
    });
    return registry;
});
```

Parâmetros chave do pool de sessões:
- `MaxActiveSessions`: Controla o limite de concorrência, evita esgotamento de recursos
- `IdleTimeout`: Timeout de ociosidade, balanceia custo de inicialização e uso de memória

Na prática descobrimos:
1. Timeout de ociosidade muito curto causa reinícios frequentes, muito longo ocupa memória
2. O limite de concorrência precisa ser ajustado conforme a carga real, muito grande pode causar travamentos
3. Precisa monitorar o uso do pool de sessões para ajustar parâmetros oportunamente

É como muitas escolhas na vida, muito agressivo facilmente causa problemas, muito conservador perde oportunidades. Encontrar o ponto de equilíbrio.

### Sincronização de Contratos

Front-end e back-end mantêm contratos consistentes através de geração OpenAPI. O back-end define o enum `AIProviderType`:

```csharp
public enum AIProviderType
{
    Unknown,
    ClaudeCode,
    OpenCode,
    HermesCli  // Novo
}
```

O front-end gera os tipos TypeScript correspondentes através de OpenAPI, garantindo que os valores do enum sejam consistentes. Esta é a chave para evitar que o front-end exiba "Unknown".

Contratos são como promessas. O que foi combinado deve ser cumprido, caso contrário aparece essa situação embaraçosa de "Unknown".

Este design driven por configuração traz flexibilidade:
- Pode sobrescrever o caminho do executável, facilitando desenvolvimento e testes
- Pode customizar parâmetros de inicialização, adaptando a diferentes versões do Hermes
- Pode configurar informações de autenticação, suportando múltiplos métodos de autenticação

Configuração é como questões de múltipla escolha na vida. Com opções suficientes, sempre se encontra a adequada. Só que às vezes tantas opções causam dificuldade de escolha.

## Experiência Prática

### Health Check

Implementar um Provider confiável precisa de health check completo:

```csharp
public async Task<ProviderTestResult> PingAsync(CancellationToken cancellationToken = default)
{
    var response = await ExecuteAsync(new AIRequest
    {
        Prompt = "Reply with exactly PONG.",
        CessionId = null,
        AllowedTools = Array.Empty<string>(),
        WorkingDirectory = ResolveWorkingDirectory(null)
    }, cancellationToken);

    var success = string.Equals(response.Content.Trim(), "PONG", StringComparison.OrdinalIgnoreCase);
    return new ProviderTestResult
    {
        ProviderName = Name,
        Success = success,
        ResponseTimeMs = stopwatch.ElapsedMilliseconds,
        ErrorMessage = success ? null : $"Unexpected Hermes ping response: '{response.Content}'."
    };
}
```

Health check precisa atenção:
1. Use casos de teste simples, evite cenários complexos
2. Defina timeout razoável
3. Registre tempo de resposta, facilitando análise de performance

Como pessoas precisam de exames de saúde, sistemas também precisam de health check. Descobrir cedo, tratar cedo, evita problemas maiores depois.

### Tratamento de Problemas Comuns

**Falha de Autenticação**
- Verifique se `Authentication.PreferredMethodId` corresponde aos métodos de autenticação realmente suportados pelo Hermes
- Confirme que o formato da informação de autenticação está correto (API Key, Bearer Token, etc.)

**Timeout de Sessão**
- Aumente o valor de `StartupTimeoutMs`
- Verifique a acessibilidade do servidor MCP
- Verifique o uso de recursos do sistema

**Resposta Incompleta**
- Garanta agregação correta de notificações `session/update` e resultado final
- Verifique a lógica de cancelamento do processamento streaming
- Valide se o tratamento de erros está completo

**Front-end Exibe Unknown**
- Confirme que a geração OpenAPI inclui o valor do enum `HermesCli`
- Verifique se o mapeamento de tipos está correto
- Limpe o cache do navegador e regenere os tipos

Problemas sempre existem. Só que quando encontrar problemas, não entre em pânico, procure a causa calmamente, sempre dá para resolver. Afinal, soluções são mais numerosas que dificuldades.

### Sugestões de Otimização de Performance

1. **Use Pool de Sessões**: Reutiliza subprocessos ACP, reduz sobrecarga de inicialização
2. **Defina Timeout Razoável**: Balanceia memória e custo de inicialização
3. **Reuse ID de Sessão**: Use o mesmo `CessionId` para tarefas em lote
4. **Configure MCP sob Demanda**: Evite chamadas de ferramentas desnecessárias

Performance é como eficiência na vida. Feito certo,事半功倍; feito errado,事倍功半. Só encontrar aquele ponto "certo" precisa experiência e sorte.

## Conclusão

Integrar o Hermes Agent em um sistema de produção precisa considerar problemas em múltiplas camadas:

1. **Camada de Arquitetura**: Desenhe interfaces unificadas de Provider, implemente arquitetura de componentes substituíveis
2. **Camada de Protocolo**: Trate corretamente as particularidades do protocolo ACP, como marcadores de inicialização, autenticação dinâmica, etc.
3. **Camada de Performance**: Reutilize recursos através de pool de sessões, balanceie custo de inicialização e uso de memória
4. **Camada Front-end**: Garanta sincronização de contratos, forneça experiência visual consistente

A prática do HagiCode mostra que através de bom design em camadas e configuração driver, sistemas Agent complexos podem ser integrados perfeitamente na arquitetura existente.

Na verdade, esses princípios parecem simples, mas na prática real, sempre encontramos vários problemas. Sem problemas, é experiência; não resolvidos, é lição, tudo tem valor.

Coisas ou pessoas belas não precisam ser possuídas, desde que ela permaneça bela, basta contemplar sua beleza. O mesmo vale para tecnologia, desde que torne o sistema melhor, que framework, que protocolo, na verdade não é tão importante...
