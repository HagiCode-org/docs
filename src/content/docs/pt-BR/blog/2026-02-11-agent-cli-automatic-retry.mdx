## Como implementar repetição automática para Agent CLI como Claude Code e Codex

> A palavra repetição automática pode parecer um pequeno interruptor, mas quando se trata de engenharia real, é completamente diferente. Olá a todos os produtores, sou Yu Kun, produtor do HagiCode. Hoje não vamos falar de conversa vazia, vamos falar de como fazer a repetição automática para Agent CLI como Claude Code e Codex da maneira correta, para que possamos tanto lidar com exceções quanto não levar o sistema a execuções repetidas infinitas.

## Contexto

Se você também tem trabalhado com programação AI recentemente, provavelmente já encontrou este tipo de problema: a tarefa não falha no início, mas para no meio do caminho.

Em solicitações HTTP comuns, muitas vezes é só reenviar, no máximo com um backoff exponencial. Mas Agent CLI é diferente. Ferramentas como Claude Code e Codex geralmente são executadas em streaming, a saída é empurrada em partes, e durante o processo ainda bindam thread, session ou resume token. Em outras palavras, não é "esta solicitação falhou ou não", mas sim:

- O conteúdo já enviado ainda conta
- O contexto atual ainda pode continuar
- Esta falha deve ser recuperada automaticamente
- Se for recuperar, quando tentar novamente, o que enviar, o contexto original deve ser reutilizado

Muitas equipes fazendo isso pela primeira vez vão instintivamente escrever a versão mais simples: se der erro, tente novamente. Você está muito certo, essa ideia é natural, mas quando entra no projeto, os problemas surgem um após o outro.

- Alguns erros que são claramente falhas temporárias são tratados como falhas finais
- Alguns erros que não valem a pena repetir são repetidos pelo sistema repetidamente
- Solicitações com thread e sem thread são tratadas igualmente
- Estratégia de backoff sem limite, solicitações em segundo plano se autodestruem

HagiCode também passou por essas armadilhas ao integrar vários Agent CLI. Especialmente no lado do Codex, o problema inicialmente exposto foi que certas mensagens de reconnect não foram reconhecidas como um estado terminal repetível, resultando no mecanismo de recuperação já existente nunca ter chance de funcionar. Basicamente, não é que o sistema não tem repetição automática, é que o sistema não reconheceu "esta vez vale a pena repetir".

Então o ponto principal que este artigo quer explicar é muito claro: **repetição automática não é um botão, é um design em camadas.**

## Por que a repetição automática de Agent CLI é mais difícil que a repetição comum

Esta pergunta é muito prática, vamos direto à conclusão: a dificuldade da repetição automática de Agent CLI não está em "tentar novamente após alguns segundos", mas em "ainda pode continuar no contexto original".

Você pode entender como uma longa conversa. Repetir API comum é mais como telefone ocupado e ligar novamente; repetir Agent CLI é mais como a outra pessoa falando ao meio e o sinal cair, você precisa primeiro julgar se deve retornar a chamada, após retornar deve começar do início, a outra pessoa ainda lembra onde a conversa parou. Quem diz que são a mesma coisa? Não são o mesmo problema de engenharia.

Especificamente, há quatro dificuldades mais típicas.

### 1. É streaming

Uma vez que a saída já foi enviada ao usuário, você não pode como tratar solicitações comuns, engolir silenciosamente a falha e depois tentar novamente secretamente. Porque aquela parte do conteúdo já foi vista, ao reproduzir novamente se a estratégia não estiver correta, o frontend facilmente verá texto duplicado, estado confuso, o ciclo de vida da chamada de ferramenta também ficará confuso. Isso não é misticismo, é engenharia.

### 2. Geralmente binda contexto de sessão

Providers como Codex bindam thread, implementações como Claude Code também terão continuation target ou contexto de continuação equivalente. A premissa para realmente repetir automaticamente não é apenas "este erro parece uma falha temporária", mas também "esta execução ainda tem um portador para continuar".

### 3. Nem todos os erros valem a pena repetir

Flutuação de rede, SSE idle timeout, falha temporária upstream, estes geralmente podem ser tentados. Mas se você encontrar falha de autenticação, contexto já perdido, ou provider simplesmente não tem capacidade de resume, continuar repetindo na maioria das vezes não é recuperação, é criar ruído.

### 4. Precisa de limite

Repetição automática infinita quase sempre está errada. Tendências tecnológicas podem animar por um tempo, leis de engenharia geralmente estabilizam por muitos anos, uma delas é: recuperação de falha deve ter limite. O sistema deve saber quantas vezes tentar no máximo, quanto tempo intervalar entre cada vez, quando parar e admitir que desta vez realmente não funcionou.

Também por causa dessas características, HagiCode finalmente não escreveu repetição automática como algumas linhas de `try/catch` em algum provider, mas sim extraiu como uma camada de capacidade compartilhada. No final das contas, problemas de engenharia precisam voltar aos métodos de engenharia para resolver.

## A abordagem do HagiCode: tirar a repetição do Provider

A implementação real atual do HagiCode pode ser comprimida em uma frase:

**Camada compartilhada gerencia uniformemente o fluxo de repetição, Provider específico só precisa responder duas perguntas: este estado terminal vale a pena repetir? O contexto atual ainda pode continuar?**

Isso não é complexo, mas é muito crítico. Porque uma vez que as responsabilidades são separadas, Claude Code, Codex, e até outros Agent CLI podem reutilizar o mesmo esqueleto. Modelos mudam, ferramentas mudam, workflows atualizam, mas a base de engenharia sempre está lá.

### Primeira camada: usar coordenador unificado para gerenciar ciclo de repetição

O fragmento de implementação central no projeto é aproximadamente assim:

```csharp
internal static class ProviderErrorAutoRetryCoordinator
{
    public static async IAsyncEnumerable<CliMessage> ExecuteAsync(
        string prompt,
        ProviderErrorAutoRetrySettings? settings,
        Func<string, IAsyncEnumerable<CliMessage>> executeAttemptAsync,
        Func<bool> canRetryInSameContext,
        Func<TimeSpan, CancellationToken, Task> delayAsync,
        Func<CliMessage, bool> isRetryableTerminalMessage,
        [EnumeratorCancellation] CancellationToken cancellationToken)
    {
        var normalizedSettings = ProviderErrorAutoRetrySettings.Normalize(settings);
        var retrySchedule = normalizedSettings.Enabled
            ? normalizedSettings.GetRetrySchedule()
            : [];

        for (var attempt = 0; ; attempt++)
        {
            var attemptPrompt = attempt == 0
                ? prompt
                : ProviderErrorAutoRetrySettings.ContinuationPrompt;

            CliMessage? terminalFailure = null;

            await foreach (var message in executeAttemptAsync(attemptPrompt)
                               .WithCancellation(cancellationToken))
            {
                if (isRetryableTerminalMessage(message))
                {
                    terminalFailure = message;
                    break;
                }

                yield return message;
            }

            if (terminalFailure is null)
            {
                yield break;
            }

            if (attempt >= retrySchedule.Count || !canRetryInSameContext())
            {
                yield return terminalFailure;
                yield break;
            }

            await delayAsync(retrySchedule[attempt], cancellationToken);
        }
    }
}
```

O que este código faz é muito simples, mas muito poderoso.

- Falha intermediária primeiro não é passada diretamente, coordenador primeiro julga se pode recuperar
- Apenas quando o orçamento de repetição se esgota, falha final realmente retorna para camada superior
- A partir da segunda rodada não envia prompt original, mas sim envia uniformemente continuation prompt

É por isso que eu一直在 enfatizando, repetição automática não é simplesmente "solicitar novamente". Não é complementar uma ramificação de exceção, é gerenciar um ciclo de vida de execução. Parece um pouco como gerente de produto, mas em engenharia é realmente assim.

### Segunda camada: snapshot de estratégia de repetição

Outro problema facilmente ignorado é: quem decide se esta solicitação deve habilitar repetição automática?

A resposta do HagiCode é: não depender de "configuração global neste momento", mas sim fazer a estratégia como snapshot, seguir junto com esta solicitação. Desta forma, enfileiramento de sessão, persistência de mensagem, encaminhamento de execução, adaptação de provider, não perderão a estratégia. Um sucesso não é sistema, sucesso contínuo é sistema.

A estrutura central pode ser simplificada assim:

```csharp
public sealed record ProviderErrorAutoRetrySnapshot
{
    public const string DefaultStrategy = "default";

    public bool Enabled { get; init; }

    public string Strategy { get; init; } = DefaultStrategy;

    public static ProviderErrorAutoRetrySnapshot Normalize(bool? enabled, string? strategy)
    {
        return new ProviderErrorAutoRetrySnapshot
        {
            Enabled = enabled ?? true,
            Strategy = string.IsNullOrWhiteSpace(strategy)
                ? DefaultStrategy
                : strategy.Trim()
        };
    }
}
```

Então no lado de execução mapeia para objeto de configuração que provider realmente consome. O valor desta abordagem é muito direto:

- Camada de negócio decide "deve repetir"
- Runtime decide "como repetir"

Cada lado gerencia sua parte, não brigam. Muitos problemas não são que não podem ser feitos, apenas não calcularam o custo claramente. Snapshot da estratégia, essencialmente é calcular o custo antecipadamente.

### Terceira camada: Provider só faz julgamento de estado terminal e contexto

Chegando ao provider específico de Claude Code ou Codex, as responsabilidades aqui são muito finas. Você pode entender como melhoria, não como substituição.

Tomando Codex como exemplo, quando finalmente integra ao coordenador compartilhado, essencialmente só precisa fornecer três coisas:

```csharp
await foreach (var message in ProviderErrorAutoRetryCoordinator.ExecuteAsync(
                   prompt,
                   options.ProviderErrorAutoRetry,
                   retryPrompt => ExecuteCodexAttemptAsync(...),
                   () => !string.IsNullOrWhiteSpace(resolvedThreadId),
                   DelayAsync,
                   IsRetryableTerminalFailure,
                   cancellationToken))
{
    yield return message;
}
```

Você vai descobrir, o julgamento que realmente pertence ao Provider só tem dois:

- `IsRetryableTerminalFailure`
- `canRetryInSameContext`

Codex olha se thread ainda pode continuar, Claude Code olha se continuation target ainda existe. Estratégia de backoff, número de repetições, prompt subsequente, estes absolutamente não devem deixar Provider reinventar.

Depois que esta camada é separada, o custo de HagiCode integrar mais CLI é muito menor. Você não precisa copiar um conjunto completo de máquina de estado de repetição, só precisa conectar "condições de limite deste provider". Escrever rápido, não significa escrever estável; conseguir integrar, não significa integrar bem; conseguir rodar, não significa conseguir manter a longo prazo.

## Um ponto fácil de errar: não trate todos os erros como repetíveis

Nesta análise, acho o que vale mais a pena destacar não é "como implementar repetição", mas "como evitar repetição errada".

O ponto de entrada inicial do problema foi Codex não reconhecer uma mensagem de reconnect. Por intuição, muitas pessoas escolheriam a correção mínima: adicionar outro prefixo de string à whitelist. Esta ideia não está errada, apenas é mais como solução do período Demo, não muito como solução de manutenção a longo prazo.

Da implementação atual do HagiCode, o sistema já deu um passo em direção mais estável. Não fica mais preso a alguma string literal, mas entrega estado terminal recuperável uniformemente ao coordenador compartilhado processar. Os benefícios disso são óbvios:

- Não facilmente falhar completamente por pequenas mudanças em algum texto
- Cobertura de teste pode se expandir em torno de "envelope de estado terminal", não texto único hardcoded
- Lógica de repetição do mesmo provider será mais consistente

Claro, aqui precisa definir um limite: mais genérico, não igual a mais relaxado. **Enquanto contexto atual não pode continuar, mesmo que erro pareça falha temporária, não deve replay cegamente.**

Isso é muito crítico. O que realmente deixa as pessoas tranquilas não é ocasionalmente funcionar uma vez, é maioria das vezes ser confiável. Se um processo só pode ser mantido por especialistas, ainda está longe de popularização.

## Três experiências mais vale a pena reter na prática

O artigo chegou aqui, quase pode terminar na camada de prática. Se você prepara para implementar capacidade similar em seu projeto, eu mais sugiro primeiro manter as três seguintes.

### 1. Orçamento de repetição deve ter limite

O ritmo de backoff padrão atual do HagiCode é:

- 10 segundos
- 20 segundos
- 60 segundos

Este ritmo não necessariamente serve todos os sistemas, mas "ter limite" deve ser mantido. Caso contrário, repetição automática rapidamente mudará de mecanismo de recuperação para amplificador de acidentes. Não tenha pressa em dar um nome grande, primeiro veja se esta coisa pode sobreviver duas iterações na equipe.

### 2. continuation prompt deve ser unificado

O projeto usa continuation prompt fixo, permitindo tentativa subsequente claramente seguir caminho "continuar contexto atual", em vez de iniciar novamente uma solicitação completa. Esta capacidade não é chamativa, mas quando você realmente faz projeto não pode viver sem. Muitas capacidades parecem mágica, após abrir não passam de um processo de engenharia polido.

### 3. Biblioteca compartilhada e camada de adaptação devem ter teste espelho

Este ponto eu quero falar mais. Muitas equipes escrevem uma camada de teste no runtime compartilhado, e então acham que é suficiente. Na verdade não.

O motivo pelo qual HagiCode me deixa mais tranquilo é porque duas camadas complementaram testes:

- Provider compartilhado testa "se realmente ocorreu continuação automática"
- Camada de adapção testa "se erro final e mensagens streaming foram quebradas"

Desta vez também executei adicionalmente dois grupos de testes relacionados, resultado todos 31 casos passaram. Este resultado em si não prova que design é perfeito, mas pelo menos pode prover uma coisa: este conjunto de repetição automática não é solução no papel, é capacidade já restrita por código e teste. ## Resumo

Se comprimir todo artigo em uma frase, é:

**Repetição automática de Agent CLI como Claude Code, Codex, etc., é melhor não fazer como truque local dentro de algum Provider, mas sim fazer como combinação de coordenador compartilhado + snapshot de estratégia + julgamento de contexto + teste espelho.**

Os benefícios que isto traz são realmente muito práticos:

- Lógica escrita uma vez, múltiplos Providers podem reutilizar
- Se solicitação permite repetir pode seguir estável com cadeia de execução
- Com contexto continua executar, sem contexto para a tempo
- Frontend finalmente vê estado completo ou falha estável, não monte de ruído intermediário abandonado no meio
