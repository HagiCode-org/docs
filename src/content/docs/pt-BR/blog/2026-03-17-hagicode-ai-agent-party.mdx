## Criando uma Party de Aventura de IA: Prática de Configuração de Multi-Agent Collaboration do HagiCode

> No desenvolvimento de software moderno, um único Agente de IA já não consegue mais atender às demandas complexas. Como fazer com que múltiplos assistentes de IA de diferentes empresas trabalhem em conjunto no mesmo projeto? Este artigo compartilhará a solução de configuração de colaboração multi-Agent explorada pelo projeto HagiCode no desenvolvimento prático.

:::note[Nota sobre Contexto Temporal]
Este artigo registra a prática de configuração multi-Agent **no lançamento de 2026-03-17**. O escopo de suporte atual permanente segue a configuração atual do backend: Claude Code, Codex, GitHub Copilot, OpenCode, Hermes, QoderCLI, Kiro, Kimi, Gemini, DeepAgents, Codebuddy; onde `IFlowCli` agora é mantido apenas como um valor de compatibilidade histórica, não pertencendo ao escopo de suporte ativo atual. ## Contexto

Acredito que muitos desenvolvedores já passaram por essa experiência: o projeto introduziu assistentes de IA para auxiliar na programação e a eficiência realmente melhorou. Mas conforme as demandas ficam cada vez mais complexas, um AI Agent começa a não ser suficiente - você quer que ele processe simultaneamente múltiplas tarefas como revisão de código, geração de documentação, testes unitários, etc., e o resultado é frequentemente perder de vista o geral, com qualidade de output inconsistente.

O que é ainda mais frustrante é quando você tenta introduzir múltiplos assistentes de IA, os problemas ficam ainda mais complexos. Cada Agent tem sua própria forma de configuração, interface de API e lógica de execução, e podem até entrar em conflito uns com os outros. É como um time de futebol, cada jogador é talentoso, mas ninguém sabe como cooperar, resultando em uma desordem total.

O projeto HagiCode também encontrou o mesmo dilema durante o desenvolvimento. Como um projeto complexo envolvendo extensão VSCode frontend, serviços de IA backend e cliente desktop multiplataforma, precisamos **na versão de 2026-03 da época** integrar simultaneamente múltiplos assistentes de IA de diferentes empresas: Claude Code, Codex, CodeBuddy, iFlow, entre outros. Como fazê-los coexistir harmoniosamente no mesmo projeto e tirar proveito de seus pontos fortes individuais tornou-se um problema chave que precisava ser resolvido.

Na verdade, isso é aceitável, afinal quem quer lidar diariamente com um grupo de IAs brigando?

A solução compartilhada neste artigo é precisamente a prática de configuração de colaboração multi-Agent que o projeto HagiCode calçou os espinhos e otimizaou na prática. Se você também está sofrendo com colaboração de múltiplos assistentes de IA, acredito que este artigo lhe dará alguma inspiração. Talvez, afinal as situações de cada pessoa são diferentes.

## Sobre o HagiCode

[HagiCode](https://hagicode.com) é um projeto de assistente de código de IA, adotando o modo de "party de aventura" onde múltiplos motores de IA trabalham em colaboração. Endereço do projeto: [github.com/HagiCode-org/site](https://github.com/HagiCode-org/site).

A solução de configuração multi-Agent compartilhada neste artigo é precisamente uma das tecnologias centrais que permitem ao HagiCode manter desenvolvimento eficiente em projetos complexos. Nada de especial, é só transformar um grupo de IAs em uma party de aventura capaz de trabalhar em equipe.

## Design de Arquitetura Multi-Agent do HagiCode

### De "Ação Individual" para "Trabalho em Equipe"

Nos estágios iniciais do projeto HagiCode, também tentamos usar apenas um AI Agent para processar todas as tarefas. Rapidamente descobrimos que essa abordagem tem gargalos óbvios: diferentes tarefas requerem diferentes ênfases de capacidades, algumas tarefas precisam de maior capacidade de compreensão de contexto, outras precisam de capacidade de modificação de código mais precisa. Um Agent dificilmente pode se destacar em todos os aspectos.

Isso nos fez perceber que必须让多个 Agent 协同工作。但问题是，如何让不同公司的 AI 产品在同一个项目中和平共处？Precisamos resolver几个核心问题：

1. **Complexidade de Gerenciamento de Configuração**: Cada Agent tem diferentes formas de configuração, interfaces de API e modos de execução
2. **Unificação de Protocolo de Comunicação**: É necessário um método padronizado para troca de dados entre diferentes Agents
3. **Coordenação de Divisão de Tarefas**: Como distribuir razoavelmente tarefas para que cada Agent aproveite seus pontos fortes

Com essas questões em mente, começamos a projetar a arquitetura multi-Agent do HagiCode. Na verdade não é tão complexo, é só entender direito.

### Visão Geral da Arquitetura

Após múltiplas iterações, a arquitetura que finalmente determinamos é esta:

```
┌─────────────────────────────────────────────────────────────────┐
│                    AIProviderFactory                             │
│  (Padrão Factory gerencia uniformemente todos os AI Providers)   │
├─────────────────────────────────────────────────────────────────┤
│  ClaudeCodeCli  │  CodexCli  │  CodebuddyCli  │  IFlowCli    │
│  (Anthropic)   │  (OpenAI)  │  (Zhipu GLM)     │  (Zhipu)      │
└─────────────────────────────────────────────────────────────────┘
```

A ideia central é: através da interface Provider unificada, diferentes AI Agents podem ser gerenciados pelo mesmo conjunto de código. Ao mesmo tempo, usando o padrão Factory para criar e configurar dinamicamente esses Providers, garantindo escalabilidade e flexibilidade do sistema.

É como a divisão de trabalho na vida, cada um tem seu próprio papel, só que aqui transformamos essa divisão de trabalho em arquitetura de código.

### Tipos de Agent e Divisão de Responsabilidades

Com base na experiência prática de uso do projeto HagiCode, atribuímos responsabilidades diferentes a cada Agent:

| Agent | Provedor | Modelo | Uso Principal |
|-------|----------|--------|---------------|
| ClaudeCodeCli | Anthropic | glm-5-turbo | Gerar soluções técnicas e Proposals |
| CodexCli | OpenAI/Zed | gpt-5.4 | Executar modificações de código precisas |
| CodebuddyCli | Zhipu | glm-4.7 | Otimizar descrições de proposal e documentação |
| IFlowCli | Zhipu | glm-4.7 | Arquivar proposals e registros históricos (configuração da época; atualmente apenas compatibilidade histórica) |
| OpenCodeCli | - | - | Edição de código geral |
| GitHubCopilot | Microsoft | - | Programação assistida e autocomplete de código |

A lógica por trás dessa divisão de trabalho é: cada Agent tem sua própria área de especialização. Claude Code se destaca em compreender e analisar demandas complexas, por isso é responsável pelo design de soluções na fase inicial; Codex é mais preciso em modificações de código, adequado para processar tarefas de implementação específicas; CodeBuddy tem alto custo-benefício, perfeito para otimizar documentação.

Afinal, o que se adapta a você é o melhor, todos os caminhos levam a Roma, só que alguns caminhos são mais fáceis, outros um pouco mais sinuosos.

## Mecanismo de Configuração Central

### Design de Interface Provider Unificada

Para permitir que diferentes AI Agents sejam gerenciados uniformemente, primeiro é necessário definir um conjunto de interfaces unificadas. O HagiCode define esta interface:

```csharp
public interface IAIProvider
{
    // Interface Provider unificada
    Task<IAIProvider?> GetProviderAsync(AIProviderType providerType);
    Task<IAIProvider?> GetProviderAsync(string providerName, CancellationToken cancellationToken);
}
```

Esta interface parece simples, mas é a pedra angular de todo o sistema multi-Agent. Através da interface unificada, podemos ignorar de qual empresa é o produto de IA subjacente, todos são chamados da mesma forma.

Na verdade, isso é simplificar coisas complexas, afinal a simplicidade é beleza.

### Implementação do Padrão Factory de Provider

Com a interface unificada, o próximo passo é como criar instâncias desses Providers. O HagiCode usa o padrão Factory:

```csharp
private IAIProvider? CreateProvider(AIProviderType providerType, ProviderConfiguration config)
{
    return providerType switch
    {
        AIProviderType.ClaudeCodeCli =>
            ActivatorUtilities.CreateInstance<ClaudeCodeCliProvider>(_serviceProvider, Options.Create(config)),
        AIProviderType.CodebuddyCli =>
            ActivatorUtilities.CreateInstance<CodebuddyCliProvider>(_serviceProvider, Options.Create(config)),
        AIProviderType.CodexCli =>
            ActivatorUtilities.CreateInstance<CodexCliProvider>(_serviceProvider, Options.Create(config)),
        AIProviderType.IFlowCli =>
            ActivatorUtilities.CreateInstance<IFlowCliProvider>(_serviceProvider, Options.Create(config)),
        _ => null
    };
}
```

Aqui usamos `ActivatorUtilities.CreateInstance` de injeção de dependência, que pode criar dinamicamente instâncias Provider em tempo de execução e injetar automaticamente dependências. O benefício deste design é: ao adicionar um novo tipo de Agent, basta adicionar a classe Provider correspondente e então adicionar um case branch no método Factory, sem precisar modificar o código existente.

Isso é aceitável, afinal quem quer modificar um monte de código antigo toda vez que adiciona nova funcionalidade.

### Análise de Configuração Dinâmica

Para tornar a configuração mais flexível, também implementamos um mecanismo de mapeamento de tipos:

```csharp
public static AIProviderTypeExtensions
{
    private static readonly Dictionary<string, AIProviderType> _typeMap = new(
        StringComparer.OrdinalIgnoreCase)
    {
        ["ClaudeCodeCli"] = AIProviderType.ClaudeCodeCli,
        ["CodebuddyCli"] = AIProviderType.CodebuddyCli,
        ["CodexCli"] = AIProviderType.CodexCli,
        ["IFlowCli"] = AIProviderType.IFlowCli,
        // ...mais mapeamentos de tipos
    };
}
```

O papel desta tabela de mapeamento é converter nomes de Provider em formato de string em tipos enum. Desta forma, os arquivos de configuração podem usar nomes de string intuitivos, enquanto o código interno usa enums type-safe para processamento.

Afinal, com configuração, quanto mais intuitivo melhor, quem quer memorizar um monte de código complexo.

### Exemplo de Arquivo de Configuração

No uso prático, basta configurar em `appsettings.json`:

```yaml
AI:
  Providers:
    Providers:
      ClaudeCodeCli:
        Enabled: true
        Model: glm-5-turbo
        WorkingDirectory: /path/to/project
      CodebuddyCli:
        Enabled: true
        Model: glm-4.7
      CodexCli:
        Enabled: true
        Model: gpt-5.4
      IFlowCli:
        Enabled: true
        Model: glm-4.7
```

Cada Provider pode ser configurado independentemente com parâmetros como switch, versão do modelo, diretório de trabalho, etc. Este design garante flexibilidade e facilita gerenciamento e manutenção.

Na verdade, arquivos de configuração são como opções na vida, você pode escolher ativar ou desativar certas funcionalidades, só que as escolhas no código são mais fáceis de se arrepender.

## Fluxo de Tarefas da Party de Aventura

### A Arte da Divisão de Tarefas

Com a arquitetura técnica unificada, o próximo passo é como fazer múltiplos Agents trabalharem em colaboração. O HagiCode projetou um mecanismo de fluxo de tarefas, permitindo que diferentes Agents processem tarefas de diferentes estágios:

```
Criação de Proposal (Usuário)
    │
    ▼
[Claude Code] ──Gerar proposal──▶ Documento de proposal
    │                        │
    │                        ▼
    │               [Codebuddy] ──Otimizar descrição──▶ Proposal otimizada
    │                        │
    │                        ▼
    │               [Codex] ──Executar modificação──▶ Mudança de código
    │                        │
    │                        ▼
    └───────────────▶ [iFlow] ──Arquivar──▶ Registro histórico
```

O benefício dessa divisão de trabalho é: cada Agent só precisa se concentrar na tarefa em que é bom, não precisa "saber tudo". Claude Code é responsável por gerar proposals do zero, Codebuddy é responsável por tornar as descrições de proposals mais claras, Codex é responsável por transformar proposals em mudanças de código reais, iFlow é responsável por arquivar e salvar essas mudanças.

Na verdade, é como trabalho em equipe na vida, cada um tem seu próprio papel, juntos podem completar algo grande. Só que aqui os membros da equipe são IA.

### Pontos Chave da Prática

Na operação real, resumimos as seguintes experiências:

**1. Estratégia de Seleção de Agent é Importante**

Não é distribuir tarefas aleatoriamente, mas sim alocar com base nos pontos fortes de cada Agent:
- Geração de proposal: Use Claude Code, porque tem maior capacidade de compreensão de contexto
- Execução de código: Use Codex, porque é mais preciso em modificações de código
- Otimização de proposal: Use Codebuddy, porque tem alto custo-benefício
- Arquivamento e armazenamento: Use iFlow, porque é estável e confiável

Afinal, deixar a pessoa certa fazer a coisa certa é uma verdade imutável através dos tempos.

**2. Isolamento de Configuração Garante Estabilidade**

A configuração de cada Agent é gerenciada independentemente, suporta override por variável de ambiente, e os diretórios de trabalho também são independentes. Desta forma, erros de configuração de um Agent não afetam outros Agents.

É como limites na vida, cada um tem seu próprio espaço, sem interferência mútua para coexistir harmoniosamente.

**3. Mecanismo de Tratamento de Erros**

A falha de um único Agent não deve afetar o fluxo geral. Implementamos estratégias de degradação: quando um Agent falha na execução, o sistema pode alternar automaticamente para uma solução de backup, ou pular essa etapa e continuar executando tarefas subsequentes. Ao mesmo tempo, logs completos facilitam a investigação posterior de problemas.

Afinal, ninguém pode garantir que nunca haverá erros, o chave é como tratar erros. É como a vida, sempre haverá contratempos, o importante é como sair deles.

**4. Monitoramento e Observabilidade**

Através do protocolo ACP (nosso protocolo de comunicação personalizado, baseado em JSON-RPC 2.0), é possível rastrear o status de execução de cada Agent. Isolamento de sessão garante segurança concorrente, cache dinâmico otimiza o desempenho.

Afinal, coisas invisíveis são as mais propensas a problemas, ter algum monitoramento é melhor do que voar às cegas.

## Efeitos Práticos e Benefícios

Após adotar esta configuração de colaboração multi-Agent, a eficiência de desenvolvimento do projeto HagiCode melhorou significativamente. Especificamente:

1. **Capacidade de Processamento de Tarefas Dobrada**: Antes um Agent precisava processar simultaneamente múltiplas tarefas, agora pode processar em paralelo, throughput mais que dobrou
2. **Qualidade de Output Mais Estável**: Cada Agent só se concentra nas tarefas em que é bom, a consistência e qualidade dos resultados são maiores
3. **Custo de Manutenção Reduzido**: Interface e gerenciamento de configuração unificados tornam todo o sistema mais fácil de manter e expandir
4. **Adicionar Novo Agents é Simples**: Para integrar novos produtos de IA, basta implementar a interface, adicionar configuração, sem precisar modificar a lógica central

Esta solução não apenas resolveu os problemas do próprio HagiCode, mas também provou que colaboração multi-Agent é de fato uma escolha de arquitetura viável.

Na verdade, o efeito é bastante óbvio, só que o processo é um pouco conturbado.

## Conclusão

Este artigo compartilhou a experiência prática do projeto HagiCode em configuração de colaboração multi-Agent. Os pontos centrais incluem:

1. **Interface Padronizada**: Através de `IAIProvider` unificar o comportamento de diferentes Agents, permitindo que o código ignore de qual empresa é o produto subjacente
2. **Padrão Factory**: Usar `ActivatorUtilities.CreateInstance` para criar dinamicamente instâncias Provider, suportando configuração em tempo de execução e injeção de dependência
3. **Unificação de Protocolo**: Protocolo ACP implementa comunicação padronizada entre Agents, mecanismo de comunicação bidirecional baseado em JSON-RPC 2.0
4. **Divisão de Fluxo de Tarefas**: Distribuir razoavelmente tarefas para diferentes Agents, deixá-los mostrar seus pontos fortes, em vez de tentar fazer um Agent fazer tudo

Este design não apenas resolveu o problema de "múltiplos Agents brigando", mas também através do mecanismo de fluxo de tarefas da party de aventura, realizou automação e especialização do fluxo de desenvolvimento.

Se você também está considerando introduzir múltiplos assistentes de IA, espero que este artigo possa lhe dar algumas referências. Claro, cada projeto tem situações diferentes, a solução específica precisa ser ajustada de acordo com a situação real. Afinal, não há solução que sirva para todos, a que se adapta a você é a melhor.

Coisas ou pessoas belas não precisam ser possuídas, contanto que ela ainda seja bela, basta admirar sua beleza em silêncio. O mesmo vale para soluções técnicas, a que se adapta a você é a melhor...
