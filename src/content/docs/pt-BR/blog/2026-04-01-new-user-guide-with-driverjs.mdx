## Implementando elegantemente o guia de novos usuários em projetos React: A prática do driver.js do HagiCode

> Quando os usuários abrem seu produto pela primeira vez, eles realmente sabem por onde começar? Este artigo discute nossa experiência com driver.js para guia de novos usuários no projeto HagiCode.

## Contexto

Você já encontrou este cenário: novos usuários se registram em seu produto, abrem a página e ficam perplexos, sem saber onde clicar ou o que fazer. Como desenvolvedores, sempre achamos que os usuários "explorarão por conta própria", afinal a curiosidade humana é infinita. Mas a realidade é—a maioria dos usuários silenciosamente sairá em minutos porque não conseguem encontrar a entrada, como uma história que começa abruptamente e termina naturalmente.

O guia de novos usuários é uma importante maneira de resolver este problema, mas implementá-lo não é tão simples. Um bom sistema de guia precisa:

- Ser capaz de localizar com precisão os elementos da página e destacá-los
- Suportar fluxos de guia com múltiplas etapas
- Poder lembrar as escolhas do usuário (concluído/ignorado)
- Não afetar o desempenho da página e interações normais
- Estrutura de código clara, fácil de manter

Durante o desenvolvimento do HagiCode, encontramos o mesmo desafio. O HagiCode é um projeto de assistente de código AI, cujo fluxo de trabalho principal é "usuário cria proposta → AI gera plano → usuário revisa → AI executa", um processo OpenSpec. Para usuários que estão entrando em contato com este conceito pela primeira vez, este fluxo é completamente novo, e deve haver um bom guia para ajudá-los a começar rapidamente. Afinal, coisas novas sempre precisam de um pouco de tempo.

## Sobre o HagiCode

A solução compartilhada neste artigo vem de nossa experiência prática no projeto [HagiCode](https://hagicode.com). O HagiCode é um assistente de código AI baseado em Claude que ajuda desenvolvedores a completar tarefas de código com mais eficiência através do fluxo de trabalho OpenSpec. Você pode ver nosso código open source no [GitHub](https://github.com/HagiCode-org/site).

## Por que escolher driver.js

Na fase de seleção tecnológica, avaliamos várias bibliotecas de guia mainstream, cada uma com suas próprias características:

- **Intro.js**: Poderoso mas com volume grande, customização de estilos relativamente complexa
- **Shepherd.js**: API bem projetada, mas um pouco "pesado" para nosso cenário
- **driver.js**: Leve, conciso, API intuitiva, e suporta o ecossistema React

Finalmente escolhemos driver.js, baseado principalmente nas seguintes considerações:

1. **Leve**: Biblioteca principal com volume pequeno, não aumenta significativamente o tamanho do bundle
2. **API concisa**: Opções de configuração claras e intuitivas, curva de aprendizado rápida
3. **Flexibilidade**: Suporta posicionamento customizado, estilos e comportamentos de interação
4. **Importação dinâmica**: Pode ser carregado sob demanda, não afeta o desempenho da primeira tela

Na seleção tecnológica, na verdade não há a melhor, apenas a mais adequada.

## Implementação técnica

### Configuração principal

A configuração do driver.js é muito intuitiva, abaixo está a configuração principal no projeto HagiCode:

```typescript
import { driver } from 'driver.js';
import 'driver.js/dist/driver.css';

const newConversationDriver = driver({
  allowClose: true,           // Permite que o usuário feche o guia
  animate: true,              // Ativa efeitos de animação
  overlayClickBehavior: 'close', // Clique na máscara fecha o guia
  disableActiveInteraction: false, // Mantém elementos interativos
  showProgress: false,        // Não mostra barra de progresso (temos gerenciamento de progresso customizado)
  steps: guideSteps           // Array de etapas do guia
});
```

As considerações por trás dessas configurações são:

- `allowClose: true` - Respeita a escolha do usuário, não força conclusão do guia, afinal forçar não é bom
- `disableActiveInteraction: false` - Algumas etapas precisam que o usuário opere realmente (como digitar texto), então não pode desabilitar interação
- `overlayClickBehavior: 'close'` - Dá ao usuário uma maneira rápida de sair

### Gerenciamento de estado

A persistência do estado do guia é fundamental—não queremos guiar novamente a cada refresh de página, isso seria bastante irritante. O HagiCode usa localStorage para gerenciar o estado do guia:

```typescript
export type GuideState = 'pending' | 'dismissed' | 'completed';

export interface UserGuideState {
  session: GuideState;
  detailGuides: Record<string, GuideState>;
}

// Ler estado
export const getUserGuideState = (): UserGuideState => {
  const state = localStorage.getItem('userGuideState');
  return state ? JSON.parse(state) : { session: 'pending', detailGuides: {} };
};

// Atualizar estado
export const setUserGuideState = (state: UserGuideState) => {
  localStorage.setItem('userGuideState', JSON.stringify(state));
};
```

Definimos três estados:

- `pending`: Guia em andamento, usuário ainda não completou ou ignorou
- `dismissed`: Usuario fechou ativamente o guia
- `completed`: Usuario completou todas as etapas

Para o guia da página de detalhes da proposta, também suportamos rastreamento de estado mais granular (através do dicionário `detailGuides`), pois uma proposta pode passar por múltiplas fases (rascunho, revisão, execução concluída), cada fase precisando de um guia diferente. Afinal, o estado das coisas está sempre mudando.

### Localização de elementos alvo

O driver.js usa seletores CSS para localizar elementos alvo. O HagiCode adota uma convenção: usar o atributo customizado `data-guide` para marcar os alvos do guia:

```typescript
const steps = [
  {
    element: '[data-guide="launch"]',
    popover: {
      title: 'Iniciar nova conversa',
      description: 'Clique aqui para criar uma nova sessão de conversa...'
    }
  }
];
```

Usando no componente assim:

```typescript
<button data-guide="launch" onClick={handleLaunch}>
  Nova conversa
</button>
```

As vantagens desta abordagem são:

- Evita conflito com nomes de classes de estilo de negócio
- Semântica clara, pode-se ver imediatamente que este elemento está relacionado ao guia
- Fácil de gerenciar e manter de forma unificada

### Otimização de importação dinâmica

Como a funcionalidade de guia só é necessária em cenários específicos (como primeiro acesso de novos usuários), usamos importação dinâmica para otimizar o desempenho de carregamento inicial:

```typescript
const initNewUserGuide = async () => {
  // Importação dinâmica de driver.js
  const { driver } = await import('driver.js');
  await import('driver.js/dist/driver.css');

  // Inicializar guia
  const newConversationDriver = driver({
    // ...configuração
  });

  newConversationDriver.drive();
};
```

Assim o driver.js e seus arquivos de estilo só serão carregados quando necessários, não afetando o desempenho da primeira tela. Afinal, quem quer pagar o preço de espera por algo que não será usado por enquanto?

## Design do fluxo do guia

O HagiCode implementa dois caminhos de guia, cobrindo os cenários principais de uso do usuário.

### Guia de sessão (10 etapas)

Este guia ajuda o usuário a completar todo o fluxo desde criar conversa até submeter a primeira proposta completa:

1. **launch** - Iniciar o guia, apresentar o botão "nova conversa"
2. **compose** - Guiar o usuário a inserir solicitação na caixa de entrada
3. **send** - Guiar para clicar no botão enviar
4. **proposal-launch-readme** - Guiar para criar proposta README
5. **proposal-compose-readme** - Guiar para editar conteúdo da solicitação README
6. **proposal-submit-readme** - Guiar para submeter proposta README
7. **proposal-launch-agents** - Guiar para criar proposta AGENTS.md
8. **proposal-compose-agents** - Guiar para editar solicitação AGENTS.md
9. **proposal-submit-agents** - Guiar para submeter proposta AGENTS.md
10. **proposal-wait** - Explicar que AI está processando, aguarde

A ideia de design deste guia é: através de duas tarefas reais de criação de propostas (README e AGENTS.md), permitir que o usuário experimente pessoalmente o fluxo de trabalho principal do HagiCode. Afinal, o conhecimento no papel é superficial, só a prática traz entendimento verdadeiro.

### Guia de detalhes da proposta (3 etapas)

Quando o usuário entra na página de detalhes da proposta, o guia correspondente é acionado baseado no estado atual da proposta:

1. **drafting** (fase de rascunho) - Guia o usuário a visualizar o plano gerado por AI
2. **reviewing** (fase de revisão) - Guia o usuário a executar o plano
3. **executionCompleted** (fase concluída) - Guia o usuário a arquivar o plano

A característica deste guia é ser orientado a estado—dinamicamente decide qual etapa do guia mostrar baseado no estado real da proposta. As coisas sempre mudam, e o guia também deveria mudar junto.

A imagem abaixo mostra o estado do guia na página de detalhes da proposta na "fase de rascunho":

Nesta fase, o guia foca a atenção do usuário na ação chave "gerar plano", evitando que ao entrar na página de detalhes pela primeira vez não saiba por onde começar.

### Mecanismo de retry de renderização de elementos

Em aplicações React, os elementos alvo do guia podem não ter terminado de renderizar (como aguardando carregamento de dados assíncronos). Para lidar com este caso, o HagiCode implementa um mecanismo de retry:

```typescript
const waitForElement = (selector: string, maxRetries = 10, interval = 100) => {
  let retries = 0;
  return new Promise<HTMLElement>((resolve, reject) => {
    const checkElement = () => {
      const element = document.querySelector(selector) as HTMLElement;
      if (element) {
        resolve(element);
      } else if (retries < maxRetries) {
        retries++;
        setTimeout(checkElement, interval);
      } else {
        reject(new Error(`Element not found: ${selector}`));
      }
    };
    checkElement();
  });
};
```

Chame esta função antes de inicializar o guia para garantir que o elemento alvo já existe. Às vezes, esperar um pouco mais vale a pena.

## Resumo de melhores práticas

Baseado na experiência prática do HagiCode, aqui estão algumas melhores práticas chave:

### 1. O guia deve ser "escapável"

Não force o usuário a completar o guia. Alguns usuários são exploradores, eles preferem descobrir por conta própria. Forneça um botão "ignorar" claro e lembre a escolha do usuário, não incomode na próxima vez. Afinal, coisas belas ou pessoas, não é necessário possuir, apenas aprecie sua beleza.

### 2. O conteúdo do guia deve ser conciso e poderoso

Cada etapa do guia deve focar em um único objetivo:

- **Title**: Curto e claro, não mais que 10 caracteres
- **Description**: Direto ao ponto, diga ao usuário "o que é" e "por que usar"

Evite explicações longas—a atenção do usuário na fase do guia é muito limitada. Falar demais, ao contrário, ninguém quer ver.

### 3. Os seletores devem ser estáveis

Use métodos de marcação de elementos estáveis que não mudem frequentemente. O atributo customizado `data-guide` é uma boa escolha, evite depender de nomes de classe ou estrutura DOM, pois estes mudam facilmente durante refatoração. O código sempre muda, mas algumas coisas devem permanecer estáveis tanto quanto possível.

### 4. Teste seu guia

O HagiCode escreveu casos de teste completos para a funcionalidade de guia:

```typescript
describe('NewUserConversationGuide', () => {
  it('deve inicializar corretamente o estado do guia', () => {
    const state = getUserGuideState();
    expect(state.session).toBe('pending');
  });

  it('deve atualizar corretamente o estado do guia', () => {
    setUserGuideState({ session: 'completed', detailGuides: {} });
    const state = getUserGuideState();
    expect(state.session).toBe('completed');
  });
});
```

Testes podem garantir que durante a refatoração do código não quebraremos acidentalmente a funcionalidade de guia. Afinal, ninguém quer que mudar um pouco de código quebre a funcionalidade anterior.

### 5. Otimização de desempenho

- Use importação dinâmica para carregar a biblioteca de guia com atraso
- Evite inicializar a lógica do guia após o usuário ter completado o guia
- Considere o impacto de desempenho das animações do guia, em dispositivos de baixo specs pode desabilitar animações

Desempenho, como a vida, deve ser economizado onde deve ser economizado.

## Conclusão

O guia de novos usuários é uma parte importante da experiência do usuário do produto. No projeto HagiCode, usamos driver.js para construir um sistema de guia completo, cobrindo todo o fluxo de trabalho desde criação de sessão até execução de propostas.

Através do compartilhamento deste artigo, esperamos transmitir os pontos principais:

1. **A seleção tecnológica deve corresponder às necessidades**: driver.js não é o mais forte, mas para nós é o mais adequado
2. **Gerenciamento de estado é chave**: Use localStorage para persistir o estado do guia, evitando incomodar o usuário repetidamente
3. **Design do guia deve ser focado**: Cada etapa resolve um problema, não seja ganancioso
4. **Estrutura de código deve ser clara**: Separe configuração do guia, gerenciamento de estado e lógica de UI, fácil de manter

Se você está adicionando funcionalidade de guia de novos usuários ao seu projeto, esperamos que a experiência prática compartilhada neste artigo possa ajudá-lo. Na verdade, tecnologia não tem nada de misterioso, tente mais, resuma mais, e aos poucos ficará melhor...
