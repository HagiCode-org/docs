## Revelação Progressiva: Como melhorar a interação humano-computador de produtos de IA com o conceito de "menos é mais"

> No design de produtos de IA, a qualidade da entrada do usuário frequentemente determina a qualidade da saída. Este artigo compartilha uma solução de interação de "revelação progressiva" que praticamos no projeto HagiCode, por meio de orientação passo a passo, preenchimento inteligente e feedback imediato, transformando entradas breves e vagas dos usuários em propostas técnicas estruturadas, melhorando significativamente a eficiência da interação humano-computador.

## Contexto

Quem trabalha com produtos de IA provavelmente já encontrou este cenário: o usuário abre seu aplicativo, entusiasmado digita uma linha de requisito, e o conteúdo retornado pela IA não tem nada a ver. Não é que a IA não seja inteligente, é apenas que as informações fornecidas pelo usuário são muito poucas - afinal, adivinhar o que os outros pensam não é algo que ninguém faça bem.

Esse fenômeno foi especialmente evidente durante o desenvolvimento do HagiCode. O HagiCode é um assistente de código impulsionado por IA, onde os usuários criam propostas técnicas e conversas descrevendo requisitos em linguagem natural. Mas no uso real, descobrimos que o conteúdo inserido pelos usuários frequentemente apresenta estes problemas:

- **Qualidade de entrada irregular**: alguns usuários inserem apenas algumas palavras, como "otimizar login", "corrigir bug", sem o contexto necessário
- **Terminologia técnica inconsistente**: diferentes usuários usam palavras diferentes para dizer a mesma coisa, alguns dizem "frontend", outros dizem "FE"
- **Falta de informações estruturadas**: não há contexto do projeto, escopo do repositório, escopo de impacto e outras informações-chave
- **Problemas repetitivos**: o mesmo tipo de requisito aparece repetidamente, sendo necessário explicar do zero cada vez

As consequências diretas desses problemas são: dificuldade de compreensão da IA, qualidade instável das propostas geradas e má experiência do usuário. Os usuários acham que "esta IA não presta", e nós também nos sentimos injustiçados - você só dá uma frase, como vou adivinhar o que você quer?

Na verdade, não há como evitar, afinal a compreensão entre pessoas leva tempo, quanto mais com máquinas?

Para resolver essas dores, tomamos uma decisão ousada: introduzir o conceito de design de "revelação progressiva" para melhorar a interação humano-computador. As mudanças que essa decisão trouxe podem ser maiores do que você imagina, só que na época não imaginávamos que seria tão eficaz.

## O que é Revelação Progressiva

"Revelação Progressiva" (Progressive Disclosure) é um princípio de design originário do campo de HCI (Interação Humano-Computador), cuja ideia central é muito simples: **não apresente todas as informações e opções aos usuários de uma só vez, mas sim mostre gradualmente o conteúdo necessário com base nas operações e necessidades dos usuários**.

Este princípio é especialmente adequado para produtos de IA, pois a interação com IA é naturalmente progressiva - o usuário diz um pouco, a IA entende um pouco, depois complementa um pouco, e entende mais. É como a comunicação entre pessoas, tem que ser devagar, afinal ninguém pode se abrir assim que se encontra, certo?

Especificamente no cenário do HagiCode, implementamos a revelação progressiva em quatro aspectos:

### 1. Mecanismo de Otimização de Descrição: Deixe a IA ajudá-lo a se expressar claramente

Quando o usuário insere uma descrição breve, não deixamos a IA entendê-la diretamente, mas primeiro acionamos um processo de "otimização de descrição". O núcleo deste processo é "saída estruturada" - transformar o texto livre do usuário em um formato padrão. É como pegar pérolas espalhadas pelo chão e transformá-las em um colar, ficando menos bagunçado.

A descrição otimizada deve conter os seguintes capítulos padrão:

- **Contexto**: contexto do problema e informações de fundo
- **Análise**: análise técnica e processo de pensamento
- **Solução**: solução e etapas de implementação
- **Prática**: exemplos de código reais e precauções

Ao mesmo tempo, geramos automaticamente uma tabela Markdown mostrando informações como repositório alvo, caminho, permissões de edição, facilitando as operações subsequentes da IA. Afinal, com um índice claro, é mais fácil encontrar as coisas.

Abaixo está a implementação real do código:

```csharp
// Método principal em ProposalDescriptionMemoryService.cs
public async Task<string> OptimizeDescriptionAsync(
    string title,
    string description,
    string locale = "pt-BR",
    DescriptionOptimizationMemoryContext? memoryContext = null,
    CancellationToken cancellationToken = default)
{
    // Construir parâmetros de consulta
    var queryContext = BuildQueryContext(title, description);

    // Recuperar contexto histórico
    var memoryContext = await RetrieveHistoricalContextAsync(queryContext, cancellationToken);

    // Gerar prompt estruturado
    var prompt = await BuildOptimizationPromptAsync(
        title,
        description,
        memoryContext,
        cancellationToken);

    // Chamar IA para otimização
    return await _aiService.CompleteAsync(prompt, cancellationToken);
}
```

A chave deste processo é "injeção de memória" - injetamos contexto histórico como convenções do projeto, casos similares, padrões negativos no prompt, para que a IA possa referenciar experiências passadas ao otimizar. Afinal, aprendemos com os erros, as experiências passadas não podem ser desperdiçadas, certo?

**Precauções**:
- Garantir que a entrada atual tenha prioridade sobre a memória histórica, evitando sobrescrever informações explicitamente especificadas pelo usuário
- Referências do HagIndex devem servir como fonte de fatos e não podem ser modificadas por casos históricos
- Sugestões de correção de baixa confiança não devem ser injetadas como restrições fortes

### 2. Capacidade de Entrada por Voz: Falar é mais natural do que digitar

Além da entrada de texto, também suportamos entrada por voz. Isso é particularmente útil ao descrever requisitos complexos - pense, digitar um requisito técnico pode levar alguns minutos, mas falar pode levar apenas dezenas de segundos, afinal a boca é sempre mais rápida que a mão.

O foco do design da entrada por voz é "gerenciamento de estado", o usuário deve estar claramente ciente do estado atual do sistema. Definimos os seguintes estados:

- **Ocioso**: sistema pronto, pode começar a gravar
- **Aguardando upstream**: conectando ao serviço de backend
- **Gravando**: gravando a voz do usuário
- **Processando**: convertendo voz em texto
- **Erro**: ocorreu um erro, requer ação do usuário

O modelo de estado do frontend é mais ou menos assim:

```typescript
interface VoiceInputState {
  status: "idle" | "waiting-upstream" | "recording" | "processing" | "error";
  duration: number;
  error?: string;
  deletedSet: Set<string>; // Conjunto de fingerprints de resultados já excluídos
}

// Transição de estado ao iniciar a gravação
const handleVoiceInputStart = async () => {
  // Primeiro entra no estado de espera, mostra animação de carregamento
  setState({ status: "waiting-upstream" });

  // Aguarda confirmação de prontidão do backend
  const isReady = await waitForBackendReady();
  if (!isReady) {
    setState({ status: "error", error: "Serviço de backend não está pronto" });
    return;
  }

  // Iniciar gravação
  setState({ status: "recording", startTime: Date.now() });
};

// Processar resultado de reconhecimento
const handleRecognitionResult = (result: RecognitionResult) => {
  const fingerprint = normalizeFingerprint(result.text);

  // Verificar se já foi excluído
  if (state.deletedSet.has(fingerprint)) {
    return; // Pular conteúdo já excluído
  }

  // Mesclar resultado na caixa de texto
  appendResult(result);
};
```

Aqui está um detalhe: usamos "conjunto de fingerprints" para gerenciar a sincronização de exclusão. Quando o reconhecimento de voz retorna múltiplos resultados, o usuário pode excluir alguns deles. Armazenamos os fingerprints do conteúdo excluído, e se o mesmo conteúdo aparecer novamente, ele será automaticamente ignorado. É como lembrar quais pratos não gostamos, da próxima vez não pediremos mais, afinal ninguém quer ser incomodado pelo mesmo problema duas vezes.

### 3. Sistema de Gerenciamento de Prompts: Externalizar o "cérebro" da IA

O HagiCode tem um sistema flexível de gerenciamento de prompts, onde todos os prompts são armazenados como arquivos:

```
prompts/
├── metadata/
│   ├── optimize-description.zh-CN.json
│   └── optimize-description.en-US.json
└── templates/
    ├── optimize-description.zh-CN.hbs
    └── optimize-description.en-US.hbs
```

Cada prompt consiste em duas partes:
- **Arquivo de metadados (.json)**: define cenário, versão, parâmetros e outras informações do prompt
- **Arquivo de modelo (.hbs)**: conteúdo real do prompt usando sintaxe Handlebars

O formato do arquivo de metadados é assim:

```json
{
  "scenario": "optimize-description",
  "locale": "pt-BR",
  "version": "1.0.0",
  "syntax": "handlebars",
  "syntaxVersion": "1.0",
  "parameters": [
    {
      "name": "title",
      "type": "string",
      "required": true,
      "description": "Título da proposta"
    },
    {
      "name": "description",
      "type": "string",
      "required": true,
      "description": "Descrição original"
    }
  ],
  "author": "HagiCode Team",
  "description": "Otimiza a descrição da proposta técnica inserida pelo usuário",
  "lastModified": "2026-04-05",
  "tags": ["optimization", "nlp"]
}
```

Os arquivos de modelo usam sintaxe Handlebars e suportam injeção de parâmetros:

```handlebars
Você é um especialista em propostas técnicas.

<task>
Gere uma descrição estruturada de proposta técnica com base nas seguintes informações.
</task>

<input>
<title>{{title}}</title>
<description>{{description}}</description>
{{#if memoryContext}}
<memory_context>
{{memoryContext}}
</memory_context>
{{/if}}
</input>

<output_format>
## Contexto
[Descreva o contexto do problema e informações de fundo, incluindo informações do projeto, escopo do repositório, etc.]

## Análise
[Análise técnica e processo de pensamento, explicando por que essa mudança é necessária]

## Solução
[Solução e etapas de implementação, listando locais-chave do código]

## Prática
[Exemplos de código reais e precauções]
</output_format>
```

As vantagens deste design são:
- Prompts podem ser versionados como código
- Suporte multilíngue, alternando automaticamente conforme preferência do usuário
- Design parametrizado, pode injetar contexto dinamicamente
- Validação de integridade na inicialização, evitando erros em tempo de execução

Afinal, se não escrevermos o que está na mente, nunca sabemos quando vamos esquecer, em vez de nos arrependermos depois, é melhor fazer um bom registro desde o início.

### 4. Assistente Progressivo: Dividir tarefas complexas em pequenos passos

Para tarefas complexas (como primeira instalação e configuração), usamos o design de assistente de múltiplas etapas. Cada etapa solicita apenas as informações necessárias e fornece indicações claras de progresso. A vida também é assim, não se pode engordar de uma só vez, passo a passo é mais seguro.

O modelo de estado do assistente:

```typescript
interface WizardState {
  currentStep: number;           // 0-3, correspondendo a 4 etapas
  steps: WizardStep[];
  canGoNext: boolean;
  canGoBack: boolean;
  isLoading: boolean;
  error: string | null;
}

interface WizardStep {
  id: number;
  title: string;
  description: string;
  completed: boolean;
}

// Lógica de navegação de etapas
const goToNextStep = () => {
  if (wizardState.currentStep < wizardState.steps.length - 1) {
    // Validar entrada da etapa atual
    if (validateCurrentStep()) {
      wizardState.currentStep++;
      wizardState.steps[wizardState.currentStep - 1].completed = true;
    }
  }
};

const goToPreviousStep = () => {
  if (wizardState.currentStep > 0) {
    wizardState.currentStep--;
  }
};
```

Cada etapa tem lógica de validação independente, etapas concluídas têm marcadores visuais claros. Operações de cancelamento mostram uma caixa de diálogo de confirmação, impedindo que os usuários percam o progresso por engano. Afinal, se você pegar o caminho errado pode voltar, mas se você destruir o caminho, então não há solução.

## Conclusão

Revendo a prática de revelação progressiva do HagiCode, podemos resumir alguns princípios fundamentais:

1. **Orientação passo a passo**: divida tarefas complexas em pequenos passos, cada passo solicitando apenas informações necessárias
2. **Preenchimento inteligente**: use contexto histórico e conhecimento do projeto para preencher informações automaticamente
3. **Feedback imediato**: cada operação tem feedback visual claro e indicações de estado
4. **Mecanismo de tolerância a falhas**: permite que os usuários desfaçam ou redefinam, evitando perdas irreversíveis devido a erros
5. **Entrada diversificada**: suporta texto, voz e outras formas de entrada

O efeito real desta solução no HagiCode é: o comprimento médio da entrada do usuário aumentou de menos de 20 caracteres para 200-300 caracteres estruturados, a qualidade das propostas geradas pela IA melhorou significativamente, e a satisfação do usuário também aumentou.

Na verdade, não é surpreendente, afinal quanto mais informações você der, mais precisamente a IA entenderá, e melhores serão os resultados retornados, não é diferente da comunicação entre pessoas.

Se você também está trabalhando em produtos relacionados à IA, espero que essas experiências possam trazer alguma inspiração. Lembre-se: os usuários não não querem fornecer informações, você simplesmente ainda não fez as perguntas certas. O núcleo da revelação progressiva é encontrar o momento e a maneira certos de fazer perguntas, só que esse momento e maneira requerem um pouco de paciência para explorar.
