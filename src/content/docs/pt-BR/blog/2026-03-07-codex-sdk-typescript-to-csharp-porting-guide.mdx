## Do TypeScript ao C#: Guia de Portabilidade do Codex SDK

> Como dizer, este artigo também é como um filho, registrando o processo completo de portarmos o TypeScript Codex SDK oficial para C#. Dizer "portar" é mais como uma longa aventura, afinal as duas linguagens têm personalidades diferentes, e precisamos encontrar uma maneira para que elas convivam bem.

## Contexto

O Codex é uma ferramenta CLI de AI Agent lançada pela OpenAI, realmente muito poderosa. O oficial fornece o TypeScript SDK, colocado no pacote `@openai/codex`. Ele interage com o Codex CLI chamando o comando `codex exec --experimental-json`, analisando o fluxo de eventos no formato JSONL.

Mas aqui está o problema, no projeto HagiCode, precisamos usá-lo em um ambiente .NET puro. Especificamente, é o serviço backend em C# e o aplicativo desktop. Diga-me, como podemos introduzir o runtime do Node.js em um projeto .NET apenas para chamar uma ferramenta CLI? Isso seria muito trabalhoso.

Assim, temos dois caminhos à nossa frente: primeiro, manter uma camada de ponte Node.js complexa, ou segundo, fazer com nossas próprias mãos e implementar um SDK C# nativo.

Escolhemos a última opção.

## Conteúdo Principal

### Comparação de Design de Arquitetura

Antes de começar a conversão no nível de código, precisamos primeiro entender o design de arquitetura dos dois SDKs. Afinal, conheça a si mesmo e ao inimigo, e você nunca será derrotado em cem batalhas.

A arquitetura central do TypeScript SDK é assim:

```
Codex (classe de entrada)
  └── CodexExec (executor, gerencia subprocessos)
      └── Thread (thread de conversa)
          ├── run() / runStreamed() (execução síncrona/assíncrona)
          └── análise de fluxo de eventos
```

O C# SDK, por outro lado, mantém a mesma hierarquia de arquitetura, mas fez adaptações nos detalhes de implementação. A ideia geral é: manter a consistência da API, mas aproveitar ao máximo os recursos da linguagem C# na implementação específica. Afinal, as linguagens são diferentes, deve haver algumas diferenças.

### Conversão de Sistema de Tipos

Este é o trabalho mais básico e importante. Afinal, arranha-se o céu arranha-se o chão, se a base não for boa, tudo depois será problema.

O sistema de tipos do TypeScript é mais flexível que o do C#, isso é um fato. Precisamos encontrar uma maneira de mapeamento adequada:

| TypeScript | C# | Descrição |
|------------|-----|------|
| `interface` / `type` | `record` | C# usa record para implementar estruturas de dados imutáveis |
| `string \| null` | `string?` | Tipos de referência anuláveis |
| `boolean \| undefined` | `bool?` | Valor booleano anulável |
| `AsyncGenerator` | `IAsyncEnumerable` | Iterador assíncrono |

**O sistema de tipos de eventos** é um exemplo típico. O TypeScript usa tipos de união para definir eventos:

```typescript
export type ThreadEvent =
  | ThreadStartedEvent
  | TurnStartedEvent
  | TurnCompletedEvent
  | ...
```

Em C#, usamos hierarquia de herança e correspondência de padrões para alcançar efeitos semelhantes:

```csharp
public abstract record ThreadEvent(string Type);

public sealed record ThreadStartedEvent(string ThreadId) : ThreadEvent("thread.started");
public sealed record TurnStartedEvent() : ThreadEvent("turn.started");
public sealed record TurnCompletedEvent(Usage Usage) : ThreadEvent("turn.completed");
// ...
```

Usar `record` em vez de `class`, porque os objetos de evento devem ser imutáveis, o que é o mesmo que usar objetos comuns no TypeScript. A palavra-chave `sealed` garante que não haja subclasses adicionais herdando, o compilador pode otimizar. Na verdade, é só isso, acostume-se.

### Pontos de Conversão Principais

#### 1. Analisador de Eventos

A análise de eventos é o núcleo de todo o SDK, afinal isso determina se podemos entender corretamente cada mensagem retornada pelo Codex CLI. Se analisar errado, tudo depois é trabalho em vão.

A versão TypeScript usa `JSON.parse()` para analisar cada linha de JSON:

```typescript
export function parseEvent(line: string): ThreadEvent {
  const data = JSON.parse(line);
  // Processa vários tipos de eventos...
}
```

A versão C# usa `System.Text.Json.JsonDocument`:

```csharp
public static ThreadEvent Parse(string line)
{
    using var document = JsonDocument.Parse(line);
    var root = document.RootElement;
    var type = GetRequiredString(root, "type", "event.type");

    return type switch
    {
        "thread.started" => new ThreadStartedEvent(GetRequiredString(root, "thread_id", ...)),
        "turn.started" => new TurnStartedEvent(),
        "turn.completed" => new TurnCompletedEvent(ParseUsage(...)),
        // ...
        _ => new UnknownThreadEvent(type, root.Clone()),
    };
}
```

Aqui há um pequeno truque: `root.Clone()` é necessário, porque os elementos do `JsonDocument` se tornarão inválidos após a liberação do documento, precisamos manter uma cópia para tipos de eventos desconhecidos. Isso também é inevitável, afinal o processamento JSON de C# e JavaScript não é o mesmo.

#### 2. Diferenças no Gerenciamento de Processos

Este é o lugar onde os dois SDKs diferem mais. Afinal, Node.js e .NET têm personalidades diferentes, precisamos nos adaptar.

TypeScript usa a função `spawn()` do Node.js:

```typescript
const child = spawn(this.executablePath, commandArgs, { env, signal });
```

C# usa `System.Diagnostics.Process` do .NET:

```csharp
using var process = new Process { StartInfo = startInfo };
process.Start();

// Precisa gerenciar stdin/stdout/stderr manualmente
```

Especificamente, a versão C# precisa configurar o processo assim:

```csharp
var startInfo = new ProcessStartInfo
{
    FileName = _executablePath,
    RedirectStandardInput = true,
    RedirectStandardOutput = true,
    RedirectStandardError = true,
    UseShellExecute = false,
    CreateNoWindow = true,
};
```

A maior diferença está no mecanismo de cancelamento. O TypeScript usa `AbortSignal`, que é parte da Web API, muito conveniente de usar:

```typescript
const child = spawn(cmd, args, { signal: cancellationSignal });
```

C# usa `CancellationToken`:

```csharp
public async IAsyncEnumerable<string> RunAsync(
    CodexExecArgs args,
    [EnumeratorCancellation] CancellationToken cancellationToken = default)
{
    // Verifica o status de cancelamento no loop
    while (!cancellationToken.IsCancellationRequested)
    {
        // Processa a saída...
    }

    // Termina o processo quando cancelado
    if (cancellationToken.IsCancellationRequested)
    {
        try { process.Kill(entireProcessTree: true); } catch { }
    }
}
```

A diferença aqui é mais ou menos a diferença entre a Web API e o ecossistema .NET, no final das contas é só isso.

#### 3. Manutenção da Serialização de Configuração

Ambos os SDKs implementam a lógica de converter configuração JSON para configuração TOML, porque o Codex CLI aceita substituição de configuração no formato TOML. Esta parte da lógica deve ser mantida completamente consistente, caso contrário a mesma configuração produzirá comportamentos diferentes nos dois SDKs.

O que isso se chama? Isso se chama espírito artesanal. Afinal, os detalhes determinam o sucesso ou o fracasso, algumas coisas não podem ser aceitas.

### Detalhes de Implementação

#### Estrutura do Projeto

Criamos esta estrutura de projeto:

```
CodexSdk/
├── CodexSdk.csproj
├── Codex.cs           # classe de entrada
├── CodexThread.cs     # thread de conversa
├── CodexExec.cs       # executor
├── Events.cs          # definições de tipos de eventos
├── Items.cs           # definições de tipos de itens
├── EventParser.cs     # analisador de eventos
├── OutputSchemaTempFile.cs  # gerenciamento de arquivos temporários
└── ...
```

Parece bastante arrumado, não é?

#### Exemplos de Uso

A maneira básica de usar é consistente com o TypeScript SDK:

```csharp
using CodexSdk;

// Cria uma instância do Codex
var codex = new Codex();
var thread = codex.StartThread();

// Executa a consulta
var result = await thread.RunAsync("Summarize this repository.");
Console.WriteLine(result.FinalResponse);
```

O processamento de eventos em fluxo aproveita a capacidade de correspondência de padrões do C#:

```csharp
await foreach (var @event in thread.RunStreamedAsync("Analyze the code."))
{
    switch (@event)
    {
        case ItemCompletedEvent itemCompleted
            when itemCompleted.Item is AgentMessageItem msg:
            Console.WriteLine($"Assistant: {msg.Text}");
            break;
        case TurnCompletedEvent completed:
            Console.WriteLine($"Tokens: in={completed.Usage.InputTokens}");
            break;
        case CommandExecutionItem command:
            Console.WriteLine($"Command: {command.Command}");
            break;
    }
}
```

### Considerações

Durante o processo de implementação, não foi trabalho em vão, resumimos algumas experiências da seguinte forma:

1. **Gerenciamento de Processos**: A versão C# precisa gerenciar manualmente o ciclo de vida do processo, incluindo a terminação do processo ao cancelar. Usar `Kill(entireProcessTree: true)` garante que os subprocessos também sejam limpos. O que isso se chama? Isso se chama começo e fim.

2. **Tratamento de Erros**: Usamos `InvalidOperationException` para lançar erros de análise, mantendo uma maneira de tratamento de erros semelhante ao TypeScript SDK. Afinal, o tratamento de erros não pode ser muito casual.

3. **Limpeza de Recursos**: `OutputSchemaTempFile` implementa `IAsyncDisposable`, garantindo que os arquivos temporários sejam limpos corretamente. Isso também é inevitável, se os recursos não forem limpos, sempre haverá problemas.

4. **Variáveis de Ambiente**: A versão C# suporta substituição completa das variáveis de ambiente do processo através de `CodexOptions.Env`. Esta função, embora pequena, é bastante prática.

5. **Diferenças de Plataforma**: A versão C# não contém a lógica de encontrar automaticamente arquivos binários em pacotes npm como na versão TypeScript. Isso ocorre porque projetos .NET geralmente não dependem de npm, então é necessário especificar o caminho do executável codex através da variável de ambiente `CODEX_EXECUTABLE` ou `CodexPathOverride`. O que isso se chama? Isso se chama adaptar-se às condições locais.

## Resumo

Portar um SDK TypeScript maduro para C# não é apenas uma conversão no nível de sintaxe, mas também uma compreensão das filosofias de design de duas linguagens. A flexibilidade do TypeScript e as características do ecossistema JavaScript (como AbortSignal) precisam encontrar soluções alternativas correspondentes em C#. Os amargos e doces disso, provavelmente apenas quem realmente fez pode entender.

A principal lição é: **manter a consistência da API é mais importante do que manter a consistência dos detalhes de implementação**. Os usuários se preocupam se a interface é fácil de usar, não se a implementação interna é a mesma. Isso soa simples, mas precisa de trade-offs para fazer.

Se você também está fazendo trabalho de portabilidade multilíngue semelhante, nossa experiência é: primeiro entender completamente o design de arquitetura do SDK original, depois converter módulo por módulo, e finalmente garantir o comportamento consistente através de casos de teste completos. Afinal, não tenha pressa, não se pode engolir um elefante de uma vez.

Tudo ficará bem, tudo terá......
