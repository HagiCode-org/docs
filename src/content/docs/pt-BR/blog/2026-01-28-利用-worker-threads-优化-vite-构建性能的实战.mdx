## De 120s para 45s: Otimizando o Desempenho de Build do Vite com Worker Threads

> Ao lidar com grandes projetos frontend, o build de código de produção muitas vezes parece uma eternidade. Este artigo compartilha como usar Node.js Worker Threads para reduzir o tempo de ofuscação de código no build do Vite de 120 segundos para 45 segundos, com detalhes de implementação e lições aprendidas no projeto HagiCode.

## Contexto

Em nossa prática de engenharia frontend, à medida que o projeto cresce, os problemas de eficiência de build se tornam mais evidentes. Especialmente no fluxo de build de produção, para proteger a lógica do código-fonte, geralmente introduzimos ferramentas de ofuscação JavaScript (como `javascript-obfuscator`). Este passo, embora necessário, requer muito poder computacional e consome recursos de CPU intensamente.

Nos estágios iniciais de desenvolvimento do projeto **HagiCode**, enfrentamos um gargalo de performance muito difícil: o tempo de build de produção piorava rapidamente com o aumento da quantidade de código.

**Dores específicas**:
- Execução serial de tarefas de ofuscação em single-thread, com um núcleo da CPU a 100% e outros núcleos ociosos
- Tempo de build aumentou de 30 segundos para 110-120 segundos
- O fluxo de validação de build após cada modificação de código era extremamente longo, prejudicando seriamente a eficiência de iteração de desenvolvimento
- No pipeline CI/CD, a etapa de build se tornava a mais demorada

**Por que o HagiCode tem essa necessidade?**
HagiCode é um assistente de código inteligente alimentado por IA, cuja arquitetura frontend contém lógica de negócios complexa e módulos de interação com IA. Para garantir a segurança do código central, forçamos a ofuscação de alta intensidade no lançamento de produção. Diante de uma espera de build de dois minutos, decidimos fazer uma otimização profunda de performance no sistema de build.

## Análise: Encontrando o Ponto de Entrada para o Gargalo de Performance

Antes de começar a resolver o problema de performance, precisamos esclarecer as ideias e determinar a solução técnica ideal.

### Decisão Central: Por que escolher Worker Threads?

No ambiente Node.js, existem principalmente três soluções para implementar computação paralela:
1. **child_process**: Cria processos filhos independentes
2. **Web Workers**: Principalmente para uso em browsers
3. **worker_threads**: Suporte nativo multi-thread do Node.js

Após análise comparativa, o HagiCode finalmente escolheu **Worker Threads**, pelos seguintes motivos:
- **Zero sobrecarga de serialização**: Worker Threads estão no mesmo processo, podem compartilhar memória através de `SharedArrayBuffer` ou transferência de controle, evitando os grandes custos de serialização na comunicação entre processos.
- **Suporte nativo**: Node.js versão 12+ tem suporte integrado, sem necessidade de introduzir dependências pesadas adicionais.
- **Contexto unificado**: Depuração e logging são mais convenientes do que subprocessos.

### Granularidade de Tarefas: Como dividir tarefas de ofuscação?

Ofuscar um único e enorme arquivo JS Bundle é difícil de paralelizar (porque o código tem dependências), mas os artefatos de build do Vite são compostos por múltiplos **Chunks**. Isso nos dá uma fronteira natural de paralelização:

- **Independência**: Após o empacotamento do Vite, as dependências entre diferentes Chunks foram desacopladas, podendo ser processadas em paralelo com segurança.
- **Granularidade adequada**: Geralmente os projetos terão 10-30 Chunks, essa quantidade é muito adequada para escalonamento paralelo.
- **Fácil integração**: O hook `generateBundle` do plugin Vite nos permite interceptar e processar esses Chunks antes da geração dos arquivos.

### Design da Arquitetura

Desenhamos um sistema de processamento paralelo contendo quatro componentes principais:

1. **Task Splitter**: Percorre o objeto bundle do Vite, filtra arquivos que não precisam de ofuscação (como vendor), gera fila de tarefas.
2. **Worker Pool Manager**: Gerencia o ciclo de vida dos Workers, responsável pelo dispatch, reciclagem e retry de erros das tarefas.
3. **Progress Reporter**: Output em tempo real do progresso do build, eliminando a ansiedade de espera do usuário.
4. **ObfuscationWorker**: Thread de trabalho que executa a lógica de ofuscação.

## Solução: Codificação Prática e Implementação

Com base na análise acima, começamos a implementar este sistema de ofuscação paralela.

### 1. Configurar Plugin Vite

Primeiro, integramos o plugin de ofuscação paralela no `vite.config.ts`. A configuração é muito intuitiva, basta especificar o número de Workers e regras de ofuscação.

```typescript
import { defineConfig } from 'vite'
import { parallelJavascriptObfuscator } from './buildTools/plugin'

export default defineConfig(({ mode }) => {
  const isProduction = mode === 'production'
  
  return {
    build: {
      rollupOptions: {
        ...(isProduction
          ? {
              plugins: [
                parallelJavascriptObfuscator({
                  enabled: true,
                  // Ajusta automaticamente conforme o número de núcleos da CPU, recomenda-se deixar um núcleo para a thread principal
                  workerCount: 4, 
                  retryAttempts: 3,
                  fallbackToMainThread: true, // Automaticamente degrada para single-thread em caso de erro
                  // Filtra vendor chunk, geralmente não há necessidade de ofuscar bibliotecas de terceiros
                  isVendorChunk: (fileName: string) => fileName.includes('vendor-'),
                  obfuscationConfig: {
                    compact: true,
                    controlFlowFlattening: true,
                    deadCodeInjection: true,
                    disableConsoleOutput: true,
                    // ... mais opções de ofuscação
                  },
                }),
              ],
            }
          : {}),
      },
    },
  }
})
```

### 2. Implementar Lógica do Worker

O Worker é a unidade de execução de tarefas. Precisamos definir bem as estruturas de dados de entrada e saída.

**Nota**: Embora o código aqui seja simples, há alguns pontos de atenção a serem observados. Por exemplo, a verificação de nulidade do `parentPort`, e o tratamento de erros. Na prática do HagiCode, descobrimos que algumas sintaxes ES6 especiais podem causar crashes do ofuscador, então adicionamos proteção `try-catch`.

```typescript
import { parentPort } from 'worker_threads'
import javascriptObfuscator from 'javascript-obfuscator'

export interface ObfuscationTask {
  chunkId: string
  code: string
  config: any
}

export interface ObfuscationResult {
  chunkId: string
  obfuscatedCode: string
  error?: string
}

// Escuta tarefas enviadas pela thread principal
if (parentPort) {
  parentPort.on('message', async (task: ObfuscationTask) => {
    try {
      // Executa ofuscação
      const obfuscated = javascriptObfuscator.obfuscate(task.code, task.config)
      const result: ObfuscationResult = {
        chunkId: task.chunkId,
        obfuscatedCode: obfuscated.getObfuscatedCode(),
      }
      // Envia o resultado de volta para a thread principal
      parentPort?.postMessage(result)
    } catch (error) {
      // Trata exceções, garantindo que uma falha individual do Worker não bloqueie todo o build
      const result: ObfuscationResult = {
        chunkId: task.chunkId,
        obfuscatedCode: '',
        error: error instanceof Error ? error.message : 'Unknown error',
      }
      parentPort?.postMessage(result)
    }
  })
}
```

### 3. Gerenciador de Worker Pool

Este é o núcleo de toda a solução. Precisamos manter um Worker Pool de tamanho fixo, adotando a estratégia **FIFO (First-In-First-Out)** para escalonar tarefas.

```typescript
import { Worker } from 'worker_threads'
import os from 'os'

export class WorkerPool {
  private workers: Worker[] = []
  private taskQueue: Array<{
    task: ObfuscationTask
    resolve: (result: ObfuscationResult) => void
    reject: (error: Error) => void
  }> = []
  
  constructor(options: WorkerPoolOptions = {}) {
    // Padrão é número de núcleos - 1, deixando um pouco de espaço para a thread principal respirar
    const workerCount = options.workerCount ?? Math.max(1, (os.cpus().length || 4) - 1)
    
    for (let i = 0; i < workerCount; i++) {
      this.createWorker()
    }
  }

  private createWorker() {
    const worker = new Worker('./worker.ts')
    
    worker.on('message', (result) => {
      // Após completar a tarefa, tira a próxima tarefa da fila
      const nextTask = this.taskQueue.shift()
      if (nextTask) {
        this.dispatchTask(worker, nextTask)
      } else {
        // Se não há tarefas pendentes, marca o Worker como ocioso
        this.activeWorkers.delete(worker)
      }
    })
    
    this.workers.push(worker)
  }

  // Submete tarefa ao pool
  public runTask(task: ObfuscationTask): Promise<ObfuscationResult> {
    return new Promise((resolve, reject) => {
      const job = { task, resolve, reject }
      const idleWorker = this.workers.find(w => !this.activeWorkers.has(w))
      
      if (idleWorker) {
        this.dispatchTask(idleWorker, job)
      } else {
        this.taskQueue.push(job)
      }
    })
  }

  private dispatchTask(worker: Worker, job: any) {
    this.activeWorkers.set(worker, job.task)
    worker.postMessage(job.task)
  }
}
```

### 4. Relatório de Progresso

Esperar é doloroso, especialmente quando não se sabe quanto tempo ainda resta. Adicionamos um simples reportador de progresso, fornecendo feedback em tempo real do status atual.

```typescript
export class ProgressReporter {
  private completed = 0
  private readonly total: number
  private readonly startTime: number

  constructor(total: number) {
    this.total = total
    this.startTime = Date.now()
  }

  increment(): void {
    this.completed++
    this.report()
  }

  private report(): void {
    const now = Date.now()
    const elapsed = now - this.startTime
    const percentage = (this.completed / this.total) * 100
    
    // Estimativa simples de ETA
    const avgTimePerChunk = elapsed / this.completed
    const remaining = (this.total - this.completed) * avgTimePerChunk

    console.log(
      `[Parallel Obfuscation] ${this.completed}/${this.total} chunks completed (${percentage.toFixed(1)}%) | ETA: ${(remaining / 1000).toFixed(1)}s`
    )
  }
}
```

## Prática: Resultados e Armadilhas

Após implantar esta solução, a performance de build do projeto HagiCode teve uma melhoria imediata.

### Dados de Benchmark de Performance

Testamos no seguinte ambiente:
- CPU: Intel Core i7-12700K (12 cores / 20 threads)
- RAM: 32GB DDR4
- Node.js: v18.17.0
- OS: Ubuntu 22.04

**Comparação de resultados**:
- **Single-thread (antes da otimização)**: 118 segundos
- **4 Workers**: 55 segundos (melhoria de **53%**)
- **8 Workers**: 48 segundos (melhoria de **60%**)
- **12 Workers**: 45 segundos (melhoria de **62%**)

Pode-se ver que o ganho não é linear. Quando o número de Workers ultrapassa 8, a magnitude de melhoria diminui. Isto é principalmente limitado pela uniformidade da distribuição de tarefas e gargalo de largura de banda de memória.

### Problemas Comuns e Soluções

No uso real do HagiCode, também encontramos algumas armadilhas, aqui compartilhamos com todos:

**Q1: O tempo de build não diminuiu significativamente, ao contrário, ficou mais lento?**
- **Causa**: A criação do Worker em si tem sobrecarga, ou o número de Workers está configurado muito alto, causando troca de contexto frequente.
- **Solução**: Recomenda-se configurar o número de Workers como `número de núcleos da CPU - 1`. Ao mesmo tempo, verifique se há algum Chunk individual particularmente grande (por exemplo > 5MB), esse arquivo "gigante" se tornará o ponto fraco, considere otimizar a estratégia de code splitting.

**Q2: Ocasionalmente o Worker trava, build falha?**
- **Causa**: Certas sintaxes de código especiais podem causar erros internos do ofuscador.
- **Solução**: Implementamos um **mecanismo de degradação automática**. Quando o número de falhas consecutivas do Worker atinge o limite, o plugin automaticamente volta ao modo single-thread, garantindo que o build não seja interrompido. Ao mesmo tempo, registra o nome do arquivo com erro, facilitando correções específicas posteriores.

**Q3: Uso de memória muito alto (OOM)?**
- **Causa**: Cada Worker precisa de espaço de memória independente para carregar o ofuscador e analisar AST.
- **Solução**:
  - Reduza o número de Workers.
  - Aumente o limite de memória do Node.js: `NODE_OPTIONS="--max-old-space-size=4096" npm run build`.
  - Certifique-se de não manter referências a grandes objetos desnecessários dentro do Worker.

## Conclusão

Ao introduzir Node.js Worker Threads, conseguimos reduzir com sucesso o tempo de build de produção do projeto HagiCode de 120 segundos para cerca de 45 segundos, melhorando drasticamente a experiência de desenvolvimento e a eficiência do CI/CD.

O núcleo desta solução está em:
1. **Divisão razoável de tarefas**: Usar os Chunks do Vite como unidade de paralelização.
2. **Controle de recursos**: Usar Worker Pool para evitar esgotamento de recursos.
3. **Design de tolerância a falhas**: Mecanismo de degradação automática garante estabilidade do build.
