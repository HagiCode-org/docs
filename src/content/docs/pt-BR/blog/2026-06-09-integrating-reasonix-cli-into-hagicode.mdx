## Do zero: como integrar Reasonix CLI no sistema HagiCode

> Este artigo compartilha a prática técnica completa de integração do Reasonix CLI como um Agent Provider de primeira classe no sistema HagiCode, cobrindo design de arquitetura de três camadas, decisões técnicas-chave e detalhes de implementação de frontend e backend.

## Background

Reasonix CLI, falando sobre isso, é uma coisa bem interessante. É uma ferramenta de assistente de código AI baseada em ACP (Agent Communication Protocol), fornecendo poderosas capacidades de streaming e gerenciamento de sessões. Na verdade, na camada HagiCode.Libs, já resolvemos toda a implementação de baixo nível, mas esses componentes ainda estão em estado isolado, como pérolas bonitas que ainda não foram enfiadas em um colar. Os usuários não podem usá-lo através da seleção de carreira Hero, fluxo de execução de sessões ou painel de monitoramento, o que é um pouco lamentável.

O problema que enfrentamos é: como elevar Reasonix ao nível de um Agent Provider de primeira classe como Codex, Hermes, etc., implementando roteamento de backend completo e exibição de frontend? Isso não é algo que possa ser resolvido simplesmente registrando um valor de enumeração, mas sim construir o fluxo completo desde a abstração de baixo nível até a interface do usuário. É como construir uma casa, não basta apenas fazer a fundação, é preciso levantar as paredes e colocar o telhado.

O desafio desta integração está em Reasonix, como uma ferramenta CLI local, ter sua própria personalidade e temperamento. Por exemplo, ele não precisa de connection string, todos os parâmetros são configurados pelo usuário em tempo de execução; ele pode nem estar instalado, precisando de tratamento de degradação graciosa; ele é compatível com a série de modelos anthropic, mas tem seus próprios parâmetros específicos ACP como effort, budget, etc. É como uma pessoa, com sua própria maneira de fazer as coisas, não pode ser forçado.

Após cuidadoso design de arquitetura e múltiplas rodadas de discussão, finalmente adotamos um esquema de arquitetura de três camadas claro, integrando com sucesso Reasonix no sistema. Este esquema não apenas resolve os problemas atuais, mas também fornece um padrão reutilizável para integrações futuras de CLI Provider similares. Na verdade, muitas coisas são assim, uma vez que encontramos o método correto, o caminho seguinte se torna muito mais fácil.

## Sobre HagiCode

O esquema compartilhado neste artigo vem de nossa experiência prática no projeto [HagiCode](https://hagicode.com). HagiCode é um projeto de assistente de código AI de código aberto, dedicado a fornecer poderosas capacidades de geração, refatoração e otimização de código para desenvolvedores. Durante o desenvolvimento, encontramos vários desafios técnicos, e integrar Reasonix como um Agent Provider de primeira classe foi um deles. Se você achar que o esquema compartilhado neste artigo tem valor, isso mostra que nossa prática de engenharia é boa, então o próprio HagiCode também merece atenção.

## Conteúdo Principal

### Design de Arquitetura Técnica

O sistema adota uma arquitetura de três camadas clara para separar preocupações, cada camada com limites de responsabilidade claros:

**Camada HagiCode.Libs**: Esta camada já está concluída, fornecendo abstração e implementação concreta de CLI provider. Define a interface `ICliProvider<ReasonixOptions>`, implementa `ReasonixProvider` para processar streaming ACP e gerenciamento de sessões, enquanto suporta parâmetros como effort, budget, yolo, transcript, etc. A responsabilidade desta camada é fornecer capacidades de baixo nível estáveis e reutilizáveis, sem envolver qualquer lógica de negócio. É como a fundação de uma casa, embora invisível, é muito importante.

**Camada hagicode-core**: Este é o foco desta integração. É responsável por conectar as abstrações de baixo nível à interface unificada do sistema. O trabalho específico inclui registrar o valor de enumeração `AIProviderType.ReasonixCli = 12`, criar `ReasonixCliProvider` como thin adapter conectando a camada Libs, implementar `ReasonixGrain` para processar estado de sessão e fluxo de execução, e integrar o sistema Hero para mapeamento de parâmetros e gerenciamento de configuração. O núcleo desta camada é coordenar os componentes, construindo o fluxo completo de negócio. É como as paredes de carga de uma casa, conectando todas as partes.

**Camada web**: Responsável por exibir e coletar configurações para os usuários. Precisamos regenerar os tipos OpenAPI para suportar o novo valor de enumeração, implementar mapeamento de tipos visuais para que Reasonix tenha seu próprio ícone e nome de exibição, criar formulários de configuração de parâmetros CLI para que os usuários possam configurar cada parâmetro, e adicionar suporte multilíngue. O foco desta camada é experiência do usuário e design de interação. É como a decoração de uma casa, o bem feito ou não afeta diretamente o conforto da moradia.

Este design de camadas permite que cada camada foque em suas responsabilidades, reduzindo a complexidade do sistema e facilitando manutenção e extensão futuras. Na verdade, muitas vezes, separar as coisas claramente as torna mais simples.

### Decisões Técnicas-Chave

Durante o processo de implementação, fizemos várias decisões técnicas-chave que tiveram impacto importante na arquitetura final e na experiência do usuário.

**Decisão 1: Usar Grain Dedicado**

Criamos um `ReasonixGrain : IReasonixGrain, IExecutorStreamGrain` independente, em vez de tentar reutilizar algum Grain compartilhado. Esta decisão segue o padrão estabelecido dos 11 providers existentes no sistema. Embora possa parecer que haverá alguma duplicação de código, um Grain dedicado nos permite fazer controle refinado para as características específicas de Reasonix, como seu mecanismo específico de binding de sessão e mapeamento de mensagens ACP. Também definimos um DTO de resposta vazio `ReasonixResponse` como discriminador de tipo, embora não contenha dados reais, desempenha um papel importante no sistema de tipos. É como cada pessoa ter seu próprio quarto, mesmo vazio, é seu próprio espaço.

**Decisão 2: Não Criar Classe Settings Dedicada**

Diferente de alguns providers que precisam de connection string, todas as configurações de Reasonix são definidas pelo usuário em tempo de execução, não precisando de validação na inicialização. Portanto, não criamos uma classe Settings dedicada, mas sim armazenamos todas as configurações no dicionário `AIProviderOptions.Providers[ReasonixCli].Settings`. Este padrão mantém consistência com outros providers CLI locais como Qoder, Kiro, Kimi, simplificando a estrutura do código e evitando camadas de abstração desnecessárias. As chaves de configuração suportadas incluem: `effort`, `budgetUsd`, `transcriptPath`, `enableYolo`, `arguments`, `startupTimeoutMs`, `reasoning`. Às vezes, o mais simples é melhor.

**Decisão 3: Monitoramento de Saúde de Estratégia Provider**

Reasonix é um CLI instalado localmente pelo usuário, pode não estar instalado, ou não estar no PATH do sistema. Neste caso, não devemos lançar um erro diretamente, mas sim tratar com degradação graciosa. Usamos a estratégia `Provider`, verificando se o CLI está disponível através de `CommandUtil.TryResolveExecutablePath`. Se a verificação falhar, a UI exibirá "indisponível", mas não afetará outras partes do sistema. Este design torna o sistema mais robusto e dá feedback claro ao usuário. Afinal, ninguém quer que o sistema inteiro caia por causa de um pequeno problema.

**Decisão 4: Classificação do Sistema Econômico**

No sistema HagiCode, diferentes providers têm diferentes classificações do sistema econômico. Decidimos fazer Reasonix usar por padrão a classificação do sistema econômico `'claude'`, pois Reasonix em si é compatível com a série de modelos anthropic. Atualmente apenas Codex e Copilot têm classificação de sistema econômico dedicada, outros providers reutilizam as classificações existentes. Isso mantém a simplicidade do sistema e permite processar corretamente cobrança e estatísticas de custos. Reutilização também é sabedoria, não precisa começar tudo do zero.

**Decisão 5: Compatibilidade de Modelos**

Reasonix suporta múltiplos modelos através do sinalizador `--model`, especialmente a série anthropic. Adicionamos mapeamento de compatibilidade em `secondary-professions.index.json`, permitindo que os usuários selecionem esses modelos em Reasonix. Este design respeita as capacidades de Reasonix e mantém a consistência do sistema, permitindo que os usuários usem vários modelos sem precisar entender as diferenças subjacentes. Os usuários também não têm vida fácil, é melhor deixá-los mais simples.

**Implementação de Thin Adapter**

`ReasonixCliProvider` é o componente-chave conectando a camada Libs à interface unificada do sistema:

```csharp
public sealed class ReasonixCliProvider : IAIProvider, IVersionedAIProvider, IAsyncDisposable
{
    private static readonly IReadOnlyList<string> SupportedSettingKeys =
    [
        "effort",
        "budgetUsd",
        "transcriptPath",
        "enableYolo",
        "arguments",
        "startupTimeoutMs",
        "reasoning"
    ];

    private readonly ICliProvider<ReasonixOptions> _provider;
    private readonly ConcurrentDictionary<string, string> _sessionBindings = new(StringComparer.Ordinal);

    public async IAsyncEnumerable<AIStreamingChunk> StreamCoreAsync(
        AIRequest request,
        string? sessionId = null,
        [EnumeratorCancellation] CancellationToken cancellationToken = default)
    {
        var options = BuildOptions(request, sessionId);
        await foreach (var message in _provider.StreamAsync(options, cancellationToken))
        {
            yield return MapToStreamingChunk(message);
        }
    }

    private ReasonixOptions BuildOptions(AIRequest request, string? sessionId)
    {
        return new ReasonixOptions
        {
            ExecutablePath = GetExecutablePath(),
            WorkingDirectory = GetWorkingDirectory(),
            Model = _config.Model,
            Effort = _config.Settings.GetValueOrDefault("effort", "medium"),
            Budget = _config.Settings.GetValueOrDefault("budgetUsd", 10.0),
            Yolo = _config.Settings.GetValueOrDefault("enableYolo", false),
            TranscriptPath = _config.Settings.GetValueOrDefault("transcriptPath"),
            Arguments = _config.Settings.GetValueOrDefault("arguments", ""),
            StartupTimeout = GetStartupTimeout(),
            EnvironmentVariables = _environmentVariables,
            CessionId = sessionId ?? GetCessionId()
        };
    }
}
```

As responsabilidades-chave deste adapter são:
1. Validar parâmetros de configuração, rejeitar chaves de configuração não suportadas
2. Manter relações de binding de sessão, suportando recuperação de sessão
3. Mapear mensagens ACP para o formato unificado do sistema
4. Usar `ProviderErrorAutoRetryCoordinator` para implementar retry automático

É como um tradutor, traduzindo de uma linguagem para outra, garantindo que o significado seja preciso e correto.

**Implementação de Orleans Grain**

`ReasonixGrain` é responsável por processar estado de sessão e fluxo de execução:

```csharp
public class ReasonixGrain : Grain, IReasonixGrain, IExecutorStreamGrain
{
    private readonly Dictionary<string, ExecutorToolLifecycleStatus> _toolLifecycleState =
        new(StringComparer.Ordinal);

    public IAsyncEnumerable<ReasonixResponse> ExecuteCommandStreamAsync(
        string command,
        string? heroId = null,
        CancellationToken token = default,
        string? executionMessageId = null,
        string? systemMessage = null,
        Dictionary<string, string>? requestSettings = null)
    {
        var request = BuildRequest(command, isEdit: false, heroId, executionMessageId, systemMessage, requestSettings);
        return SendAsync(request, heroId, token);
    }

    private async IAsyncEnumerable<ReasonixResponse> SendAsync(
        AIRequest request,
        string? heroId,
        [EnumeratorCancellation] CancellationToken token)
    {
        _cancellationTokenSource = new CancellationTokenSource();
        var linkedToken = CancellationTokenSource.CreateLinkedTokenSource(token, _cancellationTokenSource.Token);

        var provider = await ResolveReasonixProviderAsync(heroId);

        await foreach (var chunk in provider.StreamAsync(request, linkedToken.Token))
        {
            var response = BuildChunkResponse(chunk);
            yield return response;
        }
    }

    private async Task<IAIProvider> ResolveReasonixProviderAsync(string? heroId)
    {
        // Lógica de fallback de configuração com感知 de Hero
        var config = await HeroProviderResolver.ResolveAsync(AIProviderType.ReasonixCli, heroId);
        return _aiProviderFactory.CreateProvider(AIProviderType.ReasonixCli, config);
    }
}
```

As funções-chave do Grain incluem:
1. Usar `[PersistentState("reasonix-interop")]` para manter estado de sessão
2. Implementar lógica de fallback de configuração com感知 de Hero
3. Rastrear estado de ciclo de vida de ferramentas
4. Suportar cadeia de tokens de cancelamento, garantindo que a execução possa ser interrompida oportunamente

É como um mordomo, organizando as coisas de forma ordenada.

### Melhores Práticas e Considerações

Durante o processo de implementação, resumimos algumas melhores práticas e pontos a serem observados.

**Validação de Parâmetros**

ReasonixCliProvider deve validar estritamente parâmetros de configuração, rejeitando chaves de configuração não suportadas:

```csharp
public void ValidateConfigurationOverrides(Dictionary<string, string?> overrides)
{
    foreach (var key in overrides.Keys)
    {
        if (!SupportedSettingKeys.Contains(key))
        {
            throw new HeroProviderConfigurationException(
                $"Unsupported setting key '{key}' for Reasonix provider");
        }
    }
}
```

Isso evita que os usuários configurem parâmetros incorretos, evitando erros de tempo de execução. É como um goleiro, não pode deixar coisas que não devem entrar se misturarem.

**Gerenciamento de Binding de Sessão**

Usar `ConcurrentDictionary` para gerenciar bindings de sessão, suportando recuperação de sessão:

```csharp
_sessionBindings[cessionId] = sessionId;

// Vincular sessão existente em solicitações subsequentes
if (_sessionBindings.TryGetValue(cessionId, out var boundSessionId))
{
    options.SessionId = boundSessionId;
}
```

Este design permite que os usuários recuperem sessões anteriores após interrupção, proporcionando melhor experiência. É como lembrar uma história, da próxima vez ainda pode continuar contando.

**Tratamento de Degradação Graciosa**

O frontend deve verificar a disponibilidade do CLI e fornecer aviso amigável:

```typescript
const reasonixAvailable = await healthApi.checkCliAvailable('reasonix');
if (!reasonixAvailable) {
  showMessage('Reasonix CLI não está instalado, por favor configure no PATH do sistema');
}
```

Não deixe os usuários encontrarem erros inexplicáveis, verifique antecipadamente e forneça orientação clara. Afinal, ninguém quer ficar travado sem motivo claro.

**Cobertura de Testes**

Testes completos são chave para garantia de qualidade:

```csharp
[Fact]
public async Task ExecuteCommandStreamAsync_WithValidCommand_StreamsReasonixResponse()
{
    // Arrange
    var grain = _grainFactory.GetGrain<IReasonixGrain>("test-cession");
    var responses = new List<ReasonixResponse>();

    // Act
    await foreach (var response in grain.ExecuteCommandStreamAsync("help"))
    {
        responses.Add(response);
    }

    // Assert
    responses.Should().NotBeEmpty();
    responses.All(r => r.Kind == ExecutorResponseKind.Content).Should().BeTrue();
}
```

Este tipo de teste unitário pode verificar se as funções principais estão funcionando corretamente, prevenindo erros de regressão. É como fazer exercícios antes do exame, precisa garantir que realmente dominou.
