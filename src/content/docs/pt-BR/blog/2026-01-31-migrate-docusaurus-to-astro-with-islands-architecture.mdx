## Do Docusaurus 3.x ao Astro 5.x: Recapitulação Prática da Migração do Site HagiCode

> Este artigo recapitula o processo completo de migração do site oficial da HagiCode do Docusaurus 3.x para o Astro 5.x. Exploraremos profundamente como resolver gargalos de performance através da arquitetura Islands da Astro, enquanto preservamos os ativos de componentes React existentes, alcançando melhorias duplas na velocidade de build e performance de carregamento.


## Contexto

Em janeiro de 2026, realizamos uma "cirurgia de transplante de coração" no site oficial da HagiCode — migrando totalmente o framework principal do Docusaurus 3.x para o Astro 5.x. Não foi uma grande refactoragem impulsiva, mas uma decisão técnica cuidadosamente considerada.

Antes da migração, nosso site, embora funcional, começou a mostrar alguns "sinais de riqueza": volume excessivo de artefatos de build, carga JavaScript muito alta, e velocidade de carregamento de página não ideal em páginas de documentação complexas. Como um projeto de assistente de código AI, a HagiCode precisa atualizar frequentemente a documentação e introduções de recursos; a eficiência de build afeta diretamente a velocidade de publicação. Além disso, queríamos que o site fosse mais amigável para motores de busca (SEO), para que mais desenvolvedores possam descobrir este projeto.

Para resolver essas dores, tomamos uma decisão ousada: reconstruir completamente todo o sistema de build e migrar para Astro. As mudanças trazidas por esta decisão podem ser maiores do que você imagina — falarei mais sobre isso logo a seguir.

## Sobre a HagiCode

O plano de migração de site compartilhado neste artigo vem da nossa experiência prática no projeto [HagiCode](https://github.com/HagiCode-org/site).

A HagiCode é um assistente de código AI dedicado a melhorar a eficiência de desenvolvimento. Não apenas focamos na iteração de recursos principais, mas também valorizamos a experiência do desenvolvedor. Esta refactoragem do site também visa permitir que os usuários obtenham velocidade de carregamento extrema ao navegar na documentação e site oficial.

## Por que abandonar o Docusaurus maduro?

No ecossistema React, o Docusaurus sempre foi a "resposta padrão" para sites de documentação. É pronto para uso, com plugins ricos e comunidade ativa. No entanto, com o aumento de funcionalidades da HagiCode, também sentimos suas limitações:

1.  **Gargalos de performance**: O Docusaurus é essencialmente um React SPA (Single Page Application). Mesmo que você escreva páginas puramente estáticas, o cliente precisa carregar o runtime React e realizar hidratação, o que é muito pesado para páginas de documentação simples.
2.  **Volume de recursos**: Mesmo com pouco conteúdo de página, o volume JS empacotado permanece relativamente fixo, o que não é amigável para usuários móveis e ambientes com rede ruim.
3.  **Flexibilidade insuficiente**: Embora possa ser estendido, na customização do fluxo de build, desejávamos ter mais controle de baixo nível.

O surgimento do Astro resolve exatamente esses problemas. Ele fornece uma nova "arquitetura de ilhas" (Islands Architecture): por padrão, o Astro gera HTML estático com zero JavaScript, apenas componentes que necessitam de interação são "ativados" e carregam JS. Isso significa que a maior parte do conteúdo do nosso site é HTML puro, extremamente rápido.

## Estratégia Central de Migração: Transição Suave de Arquitetura

Migração não é simplesmente copiar e colar, mas uma mudança de mentalidade. Mudamos do "modo totalmente React" do Docusaurus para o modo "Core + Islands" do Astro.

### 1. Refatoração do Sistema de Configuração

Primeiro, precisamos mudar de `docusaurus.config.ts` para `astro.config.mjs`. Isso não é apenas uma mudança de nome de arquivo, mas uma reescrita da lógica de roteamento e build.

No Docusaurus, tudo é plugin; no Astro, tudo é integração. Precisamos redefinir o caminho base do site, o modo de saída de build (estático vs SSR) e a estratégia de compressão de recursos.

**Antes da migração:**
```typescript
// docusaurus.config.ts
export default {
  title: 'HagiCode',
  url: 'https://hagicode.com',
  baseUrl: '/',
  // ... mais configurações
};
```

**Depois da migração:**
```javascript
// astro.config.mjs
import { defineConfig } from 'astro/config';
import react from '@astrojs/react';

export default defineConfig({
  integrations: [react()],
  site: 'https://hagicode.com',
  base: '/',
  // Configurações de otimização para recursos estáticos
  build: {
    inlineStylesheets: 'auto',
  },
});
```

### 2. Manutenção e Transformação de Componentes React

Esta foi a parte mais dolorosa da migração. Nosso site existente tinha muitos componentes React (como componente Tabs, highlight de código, botões de feedback, etc.). Descartar todos seria um desperdício, mas manter todos resultaria em carga JS excessiva.

A HagiCode adotou a estratégia de **hidratação progressiva**:
*   **Componentes puramente estáticos**: Para conteúdo de exibição (como cabeçalho, rodapé, documentação de texto puro), reescrever como componentes Astro (arquivos `.astro`), renderizados diretamente como HTML no build.
*   **Ilhas interativas**: Para componentes que devem manter interação (como alternador de tema, Tabs, botões de cópia de bloco de código), manter implementação React e adicionar diretivas `client:load` ou `client:visible`.

Por exemplo, o componente Tabs frequentemente usado em nossa documentação:

```jsx
// src/components/Tabs.jsx
import { useState } from 'react';
import './Tabs.css'; // importar estilos

export default function Tabs({ items }) {
  const [activeIndex, setActiveIndex] = useState(0);
  // ... lógica de estado
  return (
    <div className="tabs-wrapper">
      {/* lógica de renderização */}
    </div>
  );
}
```

Ao usar em Markdown, dizemos explicitamente ao Astro: "este componente precisa de JS":

```markdown
// src/content/docs/example.mdx
import Tabs from '../../components/Tabs.jsx';

<!-- Carrega JS apenas quando o componente entra na viewport -->
<Tabs client:visible items={...} />
```

Desta forma, componentes interativos fora da viewport não monopolizam a largura de banda, otimizando enormemente a velocidade de carregamento da primeira tela.

### 3. Adaptação do Sistema de Estilos: CSS Modules para Scoped

O Docusaurus suporta CSS Modules por padrão, enquanto o Astro prefere usar Scoped CSS (através de tags `<style>`). Ambos compartilham a ideia central de isolar estilos, mas com sintaxe diferente.

Na migração da HagiCode, decompusemos a maioria dos CSS Modules complexos em estilos Scoped do Astro. Na verdade, isso é uma coisa boa, porque em arquivos `.astro`, estilos e templates são escritos no mesmo arquivo, tornando a manutenção mais intuitiva.

**Antes da transformação:**
```css
/* Tabs.module.css */
.wrapper { background: var(--ifm-background-color); }
```

**Depois da transformação (Astro Scoped):**
```html
<!-- Tabs.astro -->
<div class="tabs-wrapper">
  <slot />
</div>

<style>
  .tabs-wrapper {
    /* Usa variáveis CSS diretamente, adaptando ao tema */
    background: var(--bg-color);
    padding: 1rem;
  }
</style>
```

Ao mesmo tempo, unificamos o sistema de variáveis CSS globais, utilizando a capacidade de感知 de ambiente do Astro para garantir que a troca de modo escuro entre diferentes páginas seja perfeita.

## Armadilhas Práticas e Soluções

Durante o processo real de migração da HagiCode, encontramos muitas armadilhas. Aqui estão algumas das mais típicas.

### 1. Dor de Caminhos e Variáveis de Ambiente

A HagiCode suporta implantação em subcaminhos (como implantação em subdiretórios do GitHub Pages). No Docusaurus, ele trata automaticamente o `baseUrl`. Mas no Astro, ao lidar com links de imagem e requisições API, precisamos ser mais cuidadosos.

Introduzimos um mecanismo de variáveis de ambiente para gerenciar uniformemente:

```javascript
// Trata caminhos no script de build
const getBasePath = () => import.meta.env.VITE_SITE_BASE || '/';
```

Lembre-se, não codifique caminhos que começam com `/` no código. No ambiente de desenvolvimento e produção, ou após configurar o caminho `base`, isso levará a recursos 404.

### 2. Compatibilidade de Scripts CommonJS

Nosso site antigo tinha alguns scripts Node.js (para coleta automática de dados Metrics, atualização de sitemap, etc.), escritos em CommonJS (`require`). O Astro e ferramentas de build modernas abraçam completamente ES Modules (`import`/`export`).

Se você também tem scripts similares, lembre-se de refatorá-los todos para ES Modules. Esta é a tendência geral, quanto antes mudar, mais cedo ficará tranquilo.

```javascript
// Maneira antiga
const fs = require('fs');

// Nova maneira
import fs from 'fs';
```

### 3. Não Esqueça SEO e Redirecionamentos

Os motores de busca já indexaram as páginas antigas do Docusaurus da HagiCode. Se mudar diretamente para Astro e a estrutura de URL mudar, levará a muitos 404 e queda drástica de autoridade.

Configuramos regras de redirecionamento no Astro:

```javascript
// astro.config.mjs
export default defineConfig({
  redirects: {
    '/docs/old-path': '/docs/new-path',
    // Mapeia links antigos para novos em lote
  }
});
```
Ou trate no nível de configuração do servidor. Certifique-se de que links antigos possam redirecionar 301 para novos endereços, o que é crucial para SEO.

## Conclusão

Migrar do Docusaurus para Astro, para a HagiCode, não é apenas uma atualização de framework, mas também uma prática da filosofia "performance primeiro".

**Nossos ganhos:**
*   **Pontuações extremas do Lighthouse**: Após a migração, as pontuações de performance do site HagiCode facilmente se aproximam da pontuação máxima.
*   **Velocidade de build mais rápida**: O build incremental do Astro reduziu pela metade o tempo de publicação de atualizações de documentação.
*   **Flexibilidade mantida**: Através da arquitetura Islands, não sacrificamos nenhuma funcionalidade interativa, ainda podemos usar React onde necessário.

Se você também mantém um site de documentação e sofre com volume de empacotamento ou velocidade de carregamento, experimente o Astro. Embora o processo de migração exija alguma cirurgia (como mudar o nome de PCode para HagiCode, escavar componentes um por um), o que você ganha é uma experiência de usuário suave como seda, definitivamente vale a pena.

O sistema de build compartilhado neste artigo é exatamente a solução que desenvolvemos ao encontrar e otimizar armadilhas práticas durante o desenvolvimento da HagiCode. Se você acha que esta solução tem valor, mostra que nossa capacidade de engenharia não é ruim — então a própria HagiCode também merece atenção.
