## 從 Docusaurus 3.x 到 Astro 5.x：HagiCode 站點遷移實戰複盤

> 本文複盤了我們將 HagiCode 官方網站從 Docusaurus 3.x 遷移至 Astro 5.x 的全過程。我們將深入探討如何透過 Astro 的 Islands 架構解決效能瓶頸，同時保留現有的 React 組件資產，實現建構速度與載入效能的雙重提升。


## 背景

2026 年 1 月，我們對 HagiCode 的官方站點進行了一次「心臟移植手術」——將核心框架從 Docusaurus 3.x 全面遷移至 Astro 5.x。這不是一次衝動的大重構，而是經過深思熟慮的技術抉擇。

在遷移前，我們的站點雖然功能完善，但逐漸顯露出一些「富貴病」：建構產物體積臃腫、JavaScript 負載過高，且頁面載入速度在複雜文件頁面下不夠理想。作為一個 AI 程式碼助手專案，HagiCode 需要頻繁更新文件和功能介紹，建構效率直接影響發布速度。同時，我們希望站點對搜尋引擎（SEO）更加友善，以便讓更多開發者發現這個專案。

為了解決這些痛點，我們做了一個大膽的決定：整個建構系統推倒重來，遷移到 Astro。這個決定帶來的變化，可能比你想像的還要大——稍後我會具體說。

## 關於 HagiCode

本文分享的站點遷移方案，來自我們在 [HagiCode](https://github.com/HagiCode-org/site) 專案中的實踐經驗。

HagiCode 是一款致力於提升開發效率的 AI 程式碼助手，我們不僅關注核心功能的疊代，同樣重視開發者體驗。這次站點的重構，也是為了讓使用者在瀏覽文件和官網時能獲得極致的載入速度。

## 為什麼要放棄成熟的 Docusaurus？

在 React 生態中，Docusaurus 一直是文件站點的「標準答案」。它開箱即用，插件豐富，社群活躍。但是，隨著 HagiCode 功能的增加，我們也感受到了它的侷限性：

1.  **效能瓶頸**：Docusaurus 本質上是一個 React SPA（單頁應用）。哪怕你是寫純靜態頁面，客戶端也需要載入 React 執行時並進行水合，這對於簡單的文件頁面來說太重了。
2.  **資源體積**：即便頁面內容很少，打包後的 JS 體積也相對固定，這對行動裝置使用者和網路較差的環境不夠友善。
3.  **靈活性不足**：雖然也能擴展，但在建構流程的客製上，我們渴望擁有更底層的控制權。

Astro 的出現正好解決了這些問題。它提供了一個全新的「島嶼架構」（Islands Architecture）：預設情況下，Astro 產生零 JavaScript 的靜態 HTML，只有需要互動的組件才會「啟動」並載入 JS。這意味著我們的站點大部分內容都是純 HTML，速度極快。

## 遷移核心策略：架構平滑過渡

遷移不是簡單的複製貼上，而是思維模式的轉變。我們從 Docusaurus 的「全 React 模式」切換到了 Astro 的「Core + Islands」模式。

### 1. 配置系統的重構

首先，我們需要從 `docusaurus.config.ts` 轉向 `astro.config.mjs`。這不僅是檔案名的變化，更是路由和建構邏輯的重寫。

在 Docusaurus 中，一切皆插件；而在 Astro 中，一切皆整合。我們需要重新定義站點的基礎路徑、建構輸出模式（靜態 vs SSR）以及資源壓縮策略。

**遷移前：**
```typescript
// docusaurus.config.ts
export default {
  title: 'HagiCode',
  url: 'https://hagicode.com',
  baseUrl: '/',
  // ... 更多配置
};
```

**遷移後：**
```javascript
// astro.config.mjs
import { defineConfig } from 'astro/config';
import react from '@astrojs/react';

export default defineConfig({
  integrations: [react()],
  site: 'https://hagicode.com',
  base: '/',
  // 針對靜態資源的優化配置
  build: {
    inlineStylesheets: 'auto',
  },
});
```

### 2. React 組件的去留與改造

這是遷移中最頭痛的部分。我們現有的站點有很多 React 組件（比如 Tabs 組件、程式碼高亮、回饋按鈕等）。直接扔掉太可惜，全都保留又會導致 JS 負載過重。

HagiCode 採用了**漸進式水合**策略：
*   **純靜態組件**：對於展示型內容（如頁首、頁尾、純文字文件），重寫為 Astro 組件（`.astro` 檔案），在建構時直接渲染為 HTML。
*   **互動式島嶼**：對於必須保留互動的組件（如主題切換器、Tabs 切換、程式碼區塊複製按鈕），保留 React 實作，並新增 `client:load` 或 `client:visible` 指令。

例如，我們的文件中常用的 Tabs 組件：

```jsx
// src/components/Tabs.jsx
import { useState } from 'react';
import './Tabs.css'; // 引入樣式

export default function Tabs({ items }) {
  const [activeIndex, setActiveIndex] = useState(0);
  // ... 狀態邏輯
  return (
    <div className="tabs-wrapper">
      {/* 渲染邏輯 */}
    </div>
  );
}
```

在 Markdown 中使用時，我們明確告訴 Astro：「這個組件需要 JS」：

```markdown
// src/content/docs/example.mdx
import Tabs from '../../components/Tabs.jsx';

<!-- 只在組件進入視口時才載入 JS -->
<Tabs client:visible items={...} />
```

這樣，非視口內的互動組件不會搶佔頻寬，極大地優化了首屏載入速度。

### 3. 樣式系統的適配：CSS Modules 到 Scoped

Docusaurus 預設支援 CSS Modules，而 Astro 推崇使用 Scoped CSS（透過 `<style>` 標籤）。兩者的核心思想都是隔離樣式，但語法不同。

在 HagiCode 的遷移中，我們將大部分複雜的 CSS Modules 拆解為 Astro 的 Scoped 樣式。這其實是件好事，因為在 `.astro` 檔案中，樣式和模板寫在同一個檔案裡，維護起來更加直觀。

**改造前：**
```css
/* Tabs.module.css */
.wrapper { background: var(--ifm-background-color); }
```

**改造後 (Astro Scoped):**
```html
<!-- Tabs.astro -->
<div class="tabs-wrapper">
  <slot />
</div>

<style>
  .tabs-wrapper {
    /* 直接使用 CSS 變數，適配主題 */
    background: var(--bg-color);
    padding: 1rem;
  }
</style>
```

同時，我們統一了全域 CSS 變數系統，利用 Astro 的環境感知能力，確保暗色模式在不同頁面間的切換無縫銜接。

## 實踐中的坑與解決方案

在 HagiCode 的實際遷移過程中，我們遇到了不少坑，這裡挑幾個最典型的分享一下。

### 1. 路徑與環境變數的痛點

HagiCode 支援子路徑部署（比如部署到 GitHub Pages 的子目錄）。在 Docusaurus 中，它自動處理 `baseUrl`。但在 Astro 中，處理圖片連結和 API 請求時，我們需要更小心。

我們引入了環境變數機制來統一管理：

```javascript
// 在建構腳本中處理路徑
const getBasePath = () => import.meta.env.VITE_SITE_BASE || '/';
```

切記，不要在程式碼中硬編碼 `/` 開頭的路徑。在開發環境和生產環境，或者配置了 `base` 路徑後，這會導致資源 404。

### 2. CommonJS 腳本的相容

我們的舊站點有一些 Node.js 腳本（用於自動抓取 Metrics 資料、更新 sitemap 等），它們是用 CommonJS (`require`) 寫的。Astro 和現代建構工具全面擁抱 ES Modules (`import`/`export`)。

如果你也有類似的腳本，記得把它們全部重構為 ES Modules。這是大勢所趨，早點改了早點省心。

```javascript
// 舊方式
const fs = require('fs');

// 新方式
import fs from 'fs';
```

### 3. 別忘了 SEO 與重新導向

搜尋引擎已經收錄了 HagiCode 舊的 Docusaurus 頁面。如果直接切到 Astro，URL 結構發生變化，會導致大量 404，權重大跌。

我們在 Astro 中配置了重新導向規則：

```javascript
// astro.config.mjs
export default defineConfig({
  redirects: {
    '/docs/old-path': '/docs/new-path',
    // 批量映射舊連結到新連結
  }
});
```
或者在伺服器配置層面處理。確保舊連結能 301 重新導向到新地址，這對 SEO 至關重要。

## 總結

從 Docusaurus 遷移到 Astro，對 HagiCode 來說，不僅僅是一次框架升級，更是一次對「效能優先」理念的實踐。

**我們的收穫：**
*   **極致的 Lighthouse 分數**：遷移後，HagiCode 站點的效能評分輕鬆接近滿分。
*   **更快的建構速度**：Astro 的增量建構讓我們文件更新的發布時間縮短了一半。
*   **保留了靈活性**：透過 Islands 架構，我們沒有犧牲任何互動功能，依然可以在需要的地方使用 React。

如果你也在維護文件型站點，並且深受打包體積或載入速度的困擾，不妨試試 Astro。雖然遷移過程需要動動手術（比如把 PCode 的名字改成 HagiCode，把組件一個個挖過來），但換來的是如絲般順滑的使用者體驗，絕對值得。

本文分享的建構系統，正是我們在開發 HagiCode 過程中實際踩坑、實際優化出來的方案。如果你覺得這套方案有價值，說明我們的工程實力還不錯——那麼 HagiCode 本身也值得關注一下。
