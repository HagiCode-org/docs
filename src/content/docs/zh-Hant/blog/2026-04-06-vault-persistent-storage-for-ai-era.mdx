## AI時代如何臨摹項目：Vault跨項目持久化存儲系統

> 在AI輔助開發時代，如何讓AI助手更好地理解我們的學習資源？HagiCode項目通過Vault系統實現了一個統一的、可被AI理解的知識存儲抽象層，讓臨摹項目的學習效率大幅提升。

## 背景

在AI時代，開發者學習新技術和架構的方式正在發生深刻變化。"臨摹項目"——即深入研究和學習優秀開源項目的代碼、架構和設計模式——已經成為一種高效的學習方法。相比傳統的讀書、看視頻，直接閱讀和運行高質量的開源項目能讓你更快地理解真實世界的工程實踐。

只是這種學習方式也面臨著不少挑戰。

學習資料太分散了。你的筆記可能在Obsidian裡，代碼倉庫散落在各個文件夾，AI助手的對話歷史又是一個獨立的數據孤島。當你想讓AI幫你分析某個項目時，得手動複製代碼片段、整理上下文，這個過程相當繁瑣。

其實更麻煩的是上下文斷裂。AI助手無法直接訪問你的本地學習資源，每次對話都得重新提供背景信息。而且臨摹的代碼倉庫更新很快，手動同步容易出錯，多個學習項目之間也難以共享知識。

這些問題本質上都是"數據孤島"導致的。如果能有一個統一的存儲抽象層，讓AI助手能夠理解和訪問你的所有學習資源，問題就迎刃而解了。

## 關於 HagiCode

本文分享的Vault系統，正是我們在開發 [HagiCode](https://hagicode.com) 過程中實踐出來的解決方案。HagiCode 是一個AI代碼助手項目，在我們的日常開發中，經常需要學習和參考各種開源項目。為了讓AI助手更好地理解這些學習資源，我們設計了Vault跨項目持久化存儲系統。

這套方案已經在HagiCode中經過了實際驗證，如果你也面臨類似的知識管理難題，希望這些經驗能給你一些啟發。畢竟，有些坑踩過了，總得留下點什麼給後來的人。

## Vault系統設計理念

Vault系統的核心思想很簡單：創建一個統一的、可被AI理解的知識存儲抽象層。從實現角度來看，系統具有幾個關鍵特徵。

### 多類型支持

系統支持四種vault類型，分別對應不同的使用場景：

```typescript
// folder：通用文件夾類型
export const DEFAULT_VAULT_TYPE = 'folder';

// coderef：專門用於臨摹代碼項目的類型
export const CODEREF_VAULT_TYPE = 'coderef';

// obsidian：與Obsidian筆記軟件集成
export const OBSIDIAN_VAULT_TYPE = 'obsidian';

// system-managed：系統自動管理的vault
export const SYSTEM_MANAGED_VAULT_TYPE = 'system-managed';
```

其中 **coderef** 類型是HagiCode中最常用的。它專門為臨摹代碼項目設計，提供了標準化的目錄結構和AI可讀的元數據描述。

### 持久化存儲機制

Vault的註冊表以JSON格式持久化存儲，確保配置在應用重啟後仍然可用：

```csharp
public class VaultRegistryStore : IVaultRegistryStore
{
    private readonly string _registryFilePath;

    public VaultRegistryStore(IConfiguration configuration, ILogger<VaultRegistryStore> logger)
    {
        var dataDir = configuration["DataDir"] ?? "./data";
        var absoluteDataDir = Path.IsPathRooted(dataDir)
            ? dataDir
            : Path.GetFullPath(Path.Combine(Directory.GetCurrentDirectory(), dataDir));

        _registryFilePath = Path.Combine(absoluteDataDir, "personal-data", "vaults", "registry.json");
    }
}
```

這種設計的好處是簡單可靠。JSON格式人類可讀，便於調試和手動修改；文件系統存儲避免了數據庫的複雜性，降低了系統的依賴。畢竟，有時候簡單的反而是最好的。

### AI上下文集成

最關鍵的是，系統能夠自動將vault信息注入到AI提案的上下文中：

```typescript
export function buildTargetVaultsText(
  vaults: VaultForText[],
  template: VaultPromptTemplate = DEFAULT_VAULT_PROMPT_TEMPLATE,
): string {
  const readOnlyVaults = vaults.filter((vault) => vault.accessType === 'read');
  const editableVaults = vaults.filter((vault) => vault.accessType === 'write');

  if (readOnlyVaults.length === 0 && editableVaults.length === 0) {
    return '';
  }

  const sections = [
    buildVaultSection(readOnlyVaults, template.reference),
    buildVaultSection(editableVaults, template.editable),
  ].filter(Boolean);

  return `\n\n### ${template.heading}\n\n${sections.join('\n')}`;
}
```

這樣就實現了一個重要的功能：AI助手能夠自動理解可用的學習資源，無需用戶手動提供上下文。這倒也算是一種默契了。

## CodeRef Vault的標準化結構

對於coderef類型的vault，HagiCode提供了一套標準化的目錄結構：

```
my-coderef-vault/
├── index.yaml          # vault元數據描述
├── AGENTS.md           # AI助手的操作指南
├── docs/               # 存放學習筆記和文檔
└── repos/              # 通過Git子模塊管理臨摹的代碼倉庫
```

創建vault時，系統會自動初始化這個結構：

```csharp
private async Task EnsureCodeRefStructureAsync(
    string vaultName,
    string physicalPath,
    ICollection<VaultBootstrapDiagnosticDto> diagnostics,
    CancellationToken cancellationToken)
{
    Directory.CreateDirectory(physicalPath);

    var indexPath = Path.Combine(physicalPath, CodeRefIndexFileName);
    var docsPath = Path.Combine(physicalPath, CodeRefDocsDirectoryName);
    var reposPath = Path.Combine(physicalPath, CodeRefReposDirectoryName);

    // 創建標準目錄結構
    if (!Directory.Exists(docsPath))
    {
        Directory.CreateDirectory(docsPath);
    }

    if (!Directory.Exists(reposPath))
    {
        Directory.CreateDirectory(reposPath);
    }

    // 創建AGENTS.md指南
    await EnsureCodeRefAgentsDocumentAsync(physicalPath, cancellationToken);

    // 創建index.yaml元數據
    await WriteCodeRefIndexDocumentAsync(indexPath, mergedDocument, cancellationToken);
}
```

這套結構的設計也是有講究的：

- **docs/** 目錄存放你的學習筆記，可以用Markdown格式記錄對代碼的理解、架構分析、踩坑經驗等
- **repos/** 目錄通過Git子模塊管理臨摹的倉庫，而不是直接複製代碼。這樣既能保持代碼同步，又能節省空間
- **index.yaml** 包含vault的元數據，讓AI助手快速理解這個vault的用途和內容
- **AGENTS.md** 是專門寫給AI助手看的指南，說明如何處理這個vault中的內容

或許這樣組織起來，AI也能更容易理解你的想法吧。

## 系統管理的自動初始化

除了手動創建vault，HagiCode還支持系統自動管理的vault：

```csharp
public async Task<IReadOnlyList<VaultRegistryEntry>> EnsureAllSystemManagedVaultsAsync(
    CancellationToken cancellationToken = default)
{
    var definitions = GetAllResolvedDefinitions();
    var entries = new List<VaultRegistryEntry>(definitions.Count);

    foreach (var definition in definitions)
    {
        entries.Add(await EnsureResolvedSystemManagedVaultAsync(definition, cancellationToken));
    }

    return entries;
}
```

系統會自動創建和管理以下vault：

- **hagiprojectdata**：項目數據存儲，用於保存項目的配置和狀態
- **personaldata**：個人數據存儲，用於保存用戶的偏好設置
- **hbsprompt**：提示詞模板庫，用於管理常用的AI提示詞

這些vault在系統啟動時自動初始化，無需用戶手動配置。畢竟，有些事情交給系統去做就好了，人類何必操心呢。

## 訪問控制機制

一個重要的設計是訪問控制。系統將vault分為兩種訪問類型：

```typescript
export interface VaultForText {
  id: string;
  name: string;
  type: string;
  physicalPath: string;
  accessType: 'read' | 'write';  // 關鍵：區分只讀和可編輯
}
```

- **reference（只讀）**：AI僅用於分析和理解，不能修改內容。適用於參考的開源項目、文檔等
- **editable（可編輯）**：AI可以根據任務需要修改內容。適用於你的筆記、草稿等

這種區分很重要。它讓AI知道哪些內容是"只讀參考"，哪些是"可以動手改的"，避免了誤操作風險。畢竟，誰也不想自己的心血被無意中改沒了。

## 實戰：創建和使用Vault

看完了原理，咱們來看看實際怎麼用。

### 創建CodeRef Vault

以下是一個完整的前端調用示例：

```typescript
const createCodeRefVault = async () => {
  const response = await VaultService.postApiVaults({
    requestBody: {
      name: "React Learning Vault",
      type: "coderef",
      physicalPath: "/Users/developer/vaults/react-learning",
      gitUrl: "https://github.com/facebook/react.git"
    }
  });

  // 系統會自動：
  // 1. 克隆React倉庫到vault/repos/react
  // 2. 創建docs/目錄用於筆記
  // 3. 生成index.yaml元數據
  // 4. 創建AGENTS.md指南文件

  return response;
};
```

這個API調用會完成一系列操作：創建目錄結構、初始化Git子模塊、生成元數據文件等。你只需要提供基本信息，剩下的交給系統處理。其實這樣也挺省心的。

### 在AI提案中使用Vault

創建好vault後，就可以在AI提案中引用它了：

```typescript
const proposal = composeProposalChiefComplaint({
  chiefComplaint: "幫我分析React的並發渲染機制",
  repositories: [
    { id: "react", gitUrl: "https://github.com/facebook/react.git" }
  ],
  vaults: [
    {
      id: "react-learning",
      name: "React Learning Vault",
      type: "coderef",
      physicalPath: "/vaults/react-learning",
      accessType: "read"  // AI只能讀取，不能修改
    }
  ],
  quickRequestText: "重點關注fiber架構和scheduler實現"
});
```

系統會自動將vault信息注入到AI的上下文中，讓AI知道你有哪些學習資源可用。AI能理解你的想法，這倒也算是一種難得的默契了。

## 最佳實踐和注意事項

在使用Vault系統的過程中，我們總結了一些經驗教訓。

### 路徑安全

系統會嚴格校驗路徑，防止路徑穿越攻擊：

```csharp
private static string ResolveFilePath(string vaultRoot, string relativePath)
{
    var rootPath = EnsureTrailingSeparator(Path.GetFullPath(vaultRoot));
    var combinedPath = Path.GetFullPath(Path.Combine(rootPath, relativePath));
    if (!combinedPath.StartsWith(rootPath, StringComparison.OrdinalIgnoreCase))
    {
        throw new BusinessException(VaultRelativePathTraversalCode,
            "Vault file paths must stay inside the registered vault root.");
    }
    return combinedPath;
}
```

這點很重要。如果你在自定義vault路徑，一定要確保路徑在允許的範圍內，否則系統會拒絕操作。安全這東西，怎麼強調都不過分。

### Git子模塊管理

CodeRef vault推薦使用Git子模塊而非直接複製代碼：

```csharp
private static string BuildCodeRefAgentsContent()
{
    return """
    # CodeRef Vault Guide

    Repositories under `repos/` should be maintained through Git submodules
    rather than copied directly into the vault root.

    Keep this structure stable so assistants and tools can understand the vault quickly.
    """ + Environment.NewLine;
}
```

這樣做有幾個好處：保持代碼與上游同步、節省磁盤空間、便於管理多個版本的代碼。畢竟，誰願意一遍遍地重複下載同樣的東西呢。

### 文件預覽限制

為了防止性能問題，系統限制了文件大小和類型：

```csharp
private const int FileEnumerationLimit = 500;
private const int PreviewByteLimit = 256 * 1024;  // 256KB
```

如果你的vault包含大量文件或超大文件，可能會影響預覽功能的性能。這種情況下可以考慮分批處理或使用專門的搜索工具。畢竟，有些東西太大了，反而不好處理。

### 診斷信息

創建vault時會返回診斷信息，幫助調試：

```csharp
List<VaultBootstrapDiagnosticDto> bootstrapDiagnostics = [];

if (IsCodeRefVaultType(normalizedType))
{
    bootstrapDiagnostics = await EnsureCodeRefBootstrapAsync(
        normalizedName,
        normalizedPhysicalPath,
        normalizedGitUrl,
        cancellationToken);
}
```

如果創建失敗，可以查看診斷信息了解具體原因。出錯了就看看診斷信息，這倒也是一種解決問題的方法。

## 總結

Vault系統通過統一的存儲抽象層，解決了AI時代臨摹項目的核心痛點：

- **知識集中管理**：所有學習資源集中在一個地方，不再散落各處
- **AI上下文自動注入**：AI助手能夠自動理解可用的學習資源，無需手動提供上下文
- **跨項目知識複用**：多個學習項目之間可以共享和複用知識
- **標準化目錄結構**：提供一致的目錄結構，降低學習成本

這套方案在HagiCode項目中已經經過了實際驗證。如果你也在做AI輔助開發相關的工具，或者面臨類似的知識管理問題，希望這些經驗能給你一些參考。

其實技術方案的價值不在於有多複雜，而在於能不能解決實際問題。Vault系統的核心思想很簡單——就是建立一個統一的、AI可理解的知識存儲層。但正是這個簡單的抽象，讓我們的開發效率提升了不少。

有時候，簡單的反而是最好的。畢竟，複雜的東西往往藏著更多的坑......
