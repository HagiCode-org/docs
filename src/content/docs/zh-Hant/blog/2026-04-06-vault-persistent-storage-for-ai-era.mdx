## AI時代如何臨摹專案：Vault跨專案永續性存儲系統

> 在AI輔助開發時代，如何讓AI助手更好地理解我們的學習資源？HagiCode專案透過Vault系統實作了一個統一的、可被AI理解的知識存儲抽象層，讓臨摹專案的學習效率大幅提升。

## 背景

在AI時代，開發人員學習新技術和架構的方式正在發生深刻變化。"臨摹專案"——即深入研究和學習優秀開放原始碼專案的程式碼、架構和設計模式——已經成爲一種高效的學習方法。相比傳統的讀書、看影片，直接閱讀和執行高質量的開放原始碼專案能讓你更快地理解真實世界的工程實踐。

只是這種學習方式也面臨着不少挑戰。

學習資料太分散了。你的筆記可能在Obsidian裏，程式碼存放庫散落在各個檔案夾，AI助手的對話歷史又是一個獨立的資料孤島。當你想讓AI幫你分析某個專案時，得手動複製程式碼片段、整理上下文，這個程序相當繁瑣。

其實更麻煩的是上下文斷裂。AI助手無法直接存取你的本地學習資源，每次對話都得重新提供背景資訊。而且臨摹的程式碼存放庫更新很快，手動同步容易出錯，多個學習專案之間也難以共享知識。

這些問題本質上都是"資料孤島"導致的。如果能有一個統一的存儲抽象層，讓AI助手能夠理解和存取你的所有學習資源，問題就迎刃而解了。

## 關於 HagiCode

本文分享的Vault系統，正是我們在開發 [HagiCode](https://hagicode.com) 程序中實踐出來的解決方案。HagiCode 是一個AI程式碼助手專案，在我們的日常開發中，經常需要學習和參考各種開放原始碼專案。爲了讓AI助手更好地理解這些學習資源，我們設計了Vault跨專案永續性存儲系統。

這套方案已經在HagiCode中經過了實際驗證，如果你也面臨類似的知識管理難題，希望這些經驗能給你一些啓發。畢竟，有些坑踩過了，總得留下點什麼給後來的人。

## Vault系統設計理念

Vault系統的核心思想很簡單：建立一個統一的、可被AI理解的知識存儲抽象層。從實作角度來看，系統具有幾個關鍵特徵。

### 多型別支援

系統支援四種vault型別，分別對應不同的使用場景：

```typescript
// folder：通用文件夹类型
export const DEFAULT_VAULT_TYPE = 'folder';

// coderef：专门用于临摹代码项目的类型
export const CODEREF_VAULT_TYPE = 'coderef';

// obsidian：与Obsidian笔记软件集成
export const OBSIDIAN_VAULT_TYPE = 'obsidian';

// system-managed：系统自动管理的vault
export const SYSTEM_MANAGED_VAULT_TYPE = 'system-managed';
```

其中 **coderef** 型別是HagiCode中最常用的。它專門爲臨摹程式碼專案設計，提供了標準化的目錄結構和AI可讀的元資料描述。

### 永續性存儲機制

Vault的註冊表以JSON格式永續性存儲，確保設定在應用重啓後仍然可用：

```csharp
public class VaultRegistryStore : IVaultRegistryStore
{
    private readonly string _registryFilePath;

    public VaultRegistryStore(IConfiguration configuration, ILogger<VaultRegistryStore> logger)
    {
        var dataDir = configuration["DataDir"] ?? "./data";
        var absoluteDataDir = Path.IsPathRooted(dataDir)
            ? dataDir
            : Path.GetFullPath(Path.Combine(Directory.GetCurrentDirectory(), dataDir));

        _registryFilePath = Path.Combine(absoluteDataDir, "personal-data", "vaults", "registry.json");
    }
}
```

這種設計的好處是簡單可靠。JSON格式人類可讀，便於調試和手動修改；檔案系統存儲避免了資料庫的複雜性，降低了系統的相依。畢竟，有時候簡單的反而是最好的。

### AI上下文集成

最關鍵的是，系統能夠自動將vault資訊注入到AI提案的上下文中：

```typescript
export function buildTargetVaultsText(
  vaults: VaultForText[],
  template: VaultPromptTemplate = DEFAULT_VAULT_PROMPT_TEMPLATE,
): string {
  const readOnlyVaults = vaults.filter((vault) => vault.accessType === 'read');
  const editableVaults = vaults.filter((vault) => vault.accessType === 'write');

  if (readOnlyVaults.length === 0 && editableVaults.length === 0) {
    return '';
  }

  const sections = [
    buildVaultSection(readOnlyVaults, template.reference),
    buildVaultSection(editableVaults, template.editable),
  ].filter(Boolean);

  return `\n\n### ${template.heading}\n\n${sections.join('\n')}`;
}
```

這樣就實作了一個重要的功能：AI助手能夠自動理解可用的學習資源，無需使用者手動提供上下文。這倒也算是一種默契了。

## CodeRef Vault的標準化結構

對於coderef型別的vault，HagiCode提供了一套標準化的目錄結構：

```
my-coderef-vault/
├── index.yaml          # vault元数据描述
├── AGENTS.md           # AI助手的操作指南
├── docs/               # 存放学习笔记和文档
└── repos/              # 通过Git子模块管理临摹的代码仓库
```

建立vault時，系統會自動初始化這個結構：

```csharp
private async Task EnsureCodeRefStructureAsync(
    string vaultName,
    string physicalPath,
    ICollection<VaultBootstrapDiagnosticDto> diagnostics,
    CancellationToken cancellationToken)
{
    Directory.CreateDirectory(physicalPath);

    var indexPath = Path.Combine(physicalPath, CodeRefIndexFileName);
    var docsPath = Path.Combine(physicalPath, CodeRefDocsDirectoryName);
    var reposPath = Path.Combine(physicalPath, CodeRefReposDirectoryName);

    // 创建标准目录结构
    if (!Directory.Exists(docsPath))
    {
        Directory.CreateDirectory(docsPath);
    }

    if (!Directory.Exists(reposPath))
    {
        Directory.CreateDirectory(reposPath);
    }

    // 创建AGENTS.md指南
    await EnsureCodeRefAgentsDocumentAsync(physicalPath, cancellationToken);

    // 创建index.yaml元数据
    await WriteCodeRefIndexDocumentAsync(indexPath, mergedDocument, cancellationToken);
}
```

這套結構的設計也是有講究的：

- **docs/** 目錄存放你的學習筆記，可以用Markdown格式記錄對程式碼的理解、架構分析、踩坑經驗等
- **repos/** 目錄透過Git子模組管理臨摹的存放庫，而不是直接複製程式碼。這樣既能保持程式碼同步，又能節省空間
- **index.yaml** 包含vault的元資料，讓AI助手快速理解這個vault的用途和內容
- **AGENTS.md** 是專門寫給AI助手看的指南，說明如何處理這個vault中的內容

或許這樣組織起來，AI也能更容易理解你的想法吧。

## 系統管理的自動初始化

除了手動建立vault，HagiCode還支援系統自動管理的vault：

```csharp
public async Task<IReadOnlyList<VaultRegistryEntry>> EnsureAllSystemManagedVaultsAsync(
    CancellationToken cancellationToken = default)
{
    var definitions = GetAllResolvedDefinitions();
    var entries = new List<VaultRegistryEntry>(definitions.Count);

    foreach (var definition in definitions)
    {
        entries.Add(await EnsureResolvedSystemManagedVaultAsync(definition, cancellationToken));
    }

    return entries;
}
```

系統會自動建立和管理以下vault：

- **hagiprojectdata**：專案資料存儲，用於保存專案的設定和狀態
- **personaldata**：個人資料存儲，用於保存使用者的偏好設定設定
- **hbsprompt**：提示詞範本程式庫，用於管理常用的AI提示詞

這些vault在系統啓動時自動初始化，無需使用者手動設定。畢竟，有些事情交給系統去做就好了，人類何必操心呢。

## 存取控制機制

一個重要的設計是存取控制。系統將vault分爲兩種存取型別：

```typescript
export interface VaultForText {
  id: string;
  name: string;
  type: string;
  physicalPath: string;
  accessType: 'read' | 'write';  // 关键：区分只读和可编辑
}
```

- **reference（只讀）**：AI僅用於分析和理解，不能修改內容。適用於參考的開放原始碼專案、文件等
- **editable（可編輯）**：AI可以根據任務需要修改內容。適用於你的筆記、草稿等

這種區分很重要。它讓AI知道哪些內容是"只讀參考"，哪些是"可以動手改的"，避免了誤操作風險。畢竟，誰也不想自己的心血被無意中改沒了。

## 實戰：建立和使用Vault

看完了原理，咱們來看看實際怎麼用。

### 建立CodeRef Vault

以下是一個完整的前端呼叫示例：

```typescript
const createCodeRefVault = async () => {
  const response = await VaultService.postApiVaults({
    requestBody: {
      name: "React Learning Vault",
      type: "coderef",
      physicalPath: "/Users/developer/vaults/react-learning",
      gitUrl: "https://github.com/facebook/react.git"
    }
  });

  // 系统会自动：
  // 1. 克隆React仓库到vault/repos/react
  // 2. 创建docs/目录用于笔记
  // 3. 生成index.yaml元数据
  // 4. 创建AGENTS.md指南文件

  return response;
};
```

這個API呼叫會完成一系列操作：建立目錄結構、初始化Git子模組、產生元資料檔案等。你只需要提供基本資訊，剩下的交給系統處理。其實這樣也挺省心的。

### 在AI提案中使用Vault

建立好vault後，就可以在AI提案中參考它了：

系統會自動將vault資訊注入到AI的上下文中，讓AI知道你有哪些學習資源可用。AI能理解你的想法，這倒也算是一種難得的默契了。

## 最佳實踐和注意事項

在使用Vault系統的程序中，我們總結了一些經驗教訓。

### 路徑安全

系統會嚴格校驗路徑，防止路徑穿越攻擊：

```csharp
private static string ResolveFilePath(string vaultRoot, string relativePath)
{
    var rootPath = EnsureTrailingSeparator(Path.GetFullPath(vaultRoot));
    var combinedPath = Path.GetFullPath(Path.Combine(rootPath, relativePath));
    if (!combinedPath.StartsWith(rootPath, StringComparison.OrdinalIgnoreCase))
    {
        throw new BusinessException(VaultRelativePathTraversalCode,
            "Vault file paths must stay inside the registered vault root.");
    }
    return combinedPath;
}
```

這點很重要。如果你在自定義vault路徑，一定要確保路徑在允許的範圍內，否則系統會拒絕操作。安全這東西，怎麼強調都不過分。

### Git子模組管理

CodeRef vault推薦使用Git子模組而非直接複製程式碼：

```csharp
private static string BuildCodeRefAgentsContent()
{
    return """
    # CodeRef Vault Guide

    Repositories under `repos/` should be maintained through Git submodules
    rather than copied directly into the vault root.

    Keep this structure stable so assistants and tools can understand the vault quickly.
    """ + Environment.NewLine;
}
```

這樣做有幾個好處：保持程式碼與上游同步、節省磁盤空間、便於管理多個版本的程式碼。畢竟，誰願意一遍遍地重複下載同樣的東西呢。

### 檔案預覽限制

爲了防止效能問題，系統限制了檔案大小和型別：

```csharp
private const int FileEnumerationLimit = 500;
private const int PreviewByteLimit = 256 * 1024;  // 256KB
```

如果你的vault包含大量檔案或超大檔案，可能會影響預覽功能的效能。這種情況下可以考慮分批處理或使用專門的搜索工具。畢竟，有些東西太大了，反而不好處理。

### 診斷資訊

建立vault時會傳回診斷資訊，說明調試：

```csharp
List<VaultBootstrapDiagnosticDto> bootstrapDiagnostics = [];

if (IsCodeRefVaultType(normalizedType))
{
    bootstrapDiagnostics = await EnsureCodeRefBootstrapAsync(
        normalizedName,
        normalizedPhysicalPath,
        normalizedGitUrl,
        cancellationToken);
}
```

如果建立失敗，可以查看診斷資訊瞭解具體原因。出錯了就看看診斷資訊，這倒也是一種解決問題的方法。

## 總結

Vault系統透過統一的存儲抽象層，解決了AI時代臨摹專案的核心痛點：

- **知識集中管理**：所有學習資源集中在一個地方，不再散落各處
- **AI上下文自動注入**：AI助手能夠自動理解可用的學習資源，無需手動提供上下文
- **跨專案知識複用**：多個學習專案之間可以共享和複用知識
- **標準化目錄結構**：提供一致的目錄結構，降低學習成本

這套方案在HagiCode專案中已經經過了實際驗證。如果你也在做AI輔助開發相關的工具，或者面臨類似的知識管理問題，希望這些經驗能給你一些參考。

其實技術方案的價值不在於有多複雜，而在於能不能解決實際問題。Vault系統的核心思想很簡單——就是建立一個統一的、AI可理解的知識存儲層。但正是這個簡單的抽象，讓我們的開發效率提升了不少。

有時候，簡單的反而是最好的。畢竟，複雜的東西往往藏着更多的坑......
