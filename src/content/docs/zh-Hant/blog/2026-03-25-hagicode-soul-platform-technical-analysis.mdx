## HagiCode Soul 平台技術剖析：從需求萌發到獨立平台的演進之路

> 其實寫技術文章這事兒，也沒什麼了不起的，不過是把一些趟過的坑、繞過的彎路整理出來罷了。畢竟誰還沒年輕過呢，對吧？本文將深入剖析 HagiCode 專案中 Soul（AI Agent 人格設定系統）的設計理念、架構演進和核心技術實作，探討如何透過獨立平台提供更聚焦的 Agent 人格建立與分享體驗。

## 背景

在 AI Agent 的開發實踐中，我們經常會遇到一個看似簡單卻極其重要的問題：**如何讓不同的 Agent 擁有穩定且獨特的語言風格和人格特徵？**

這問題說起來也挺無奈的。早期 HagiCode 的 Hero 體系中，不同英雄（Agent 執行個體）主要相依職業設定和通用提示詞來區分表達方式。這種方式帶來了一些明顯的痛點，或許做過的朋友都有同感。

首先，語言風格難以保持一致。同樣是"開發工程師"角色，今天的回覆可能專業嚴謹，明天的輸出又變得隨意散漫。這不是模型本身的問題，而是缺乏一個獨立的人格設定層來約束和引導輸出風格罷了。

其次，角色感普遍較弱。當我們描述一個 Agent 的特徵時，往往只能用"友好"、"專業"、"幽默"這樣模糊的形容詞，卻沒有具體的語言規則來支撐這些抽象的描述。說白了，就是說起來挺美好，做起來卻沒轍。

第三，人格設定的複用性幾乎爲零。假設我們精心設計了一個"貓娘服務員"的說話風格，想要在另一個業務場景中複用這套表達方式，幾乎需要從頭開始設定。美的事物或人，不一定要佔用，只是想複用一下罷了......可是真的難。

正是爲了解決這些實際問題，我們引入了 **Soul** 機制——一個獨立於裝備和描述的**語言風格設定層**。Soul 可以定義 Agent 的說話習慣、語氣偏好設定和用詞邊界，可以在多個英雄間共享複用，還能在 Session 首次呼叫時自動注入系統提示詞。

或許有人會覺得這也罷了，不就是設定幾個提示詞嗎？可是有時候啊，問題的關鍵不在於能不能做，而在於怎麼做更優雅。隨着 Soul 能力的逐步成熟，我們意識到它已經具備了獨立發展的潛力。一個專門的 Soul 平台可以讓使用者更聚焦地建立、分享和瀏覽各種有趣的人格設定，而不必被 Hero 系統的其他功能所幹擾。於是，**soul.hagicode.com** 獨立平台應運而生。

## Soul 平台的技術架構演進

Soul 平台的發展並非一蹴而就，而是經歷了三個清晰的階段。這故事開始得突然，結束得自然。

### 第一階段：Hero 內嵌 Soul 設定

最早的 Soul 實作是作爲 Hero 工作區的一個功能模組存在的。我們在 Hero 界面中增加了獨立的 SOUL 編輯區域，支援預設套用和文字微調兩種方式。

預設套用允許使用者從一些經典人格範本中選擇，比如"專業開發工程師"、"貓娘服務員"等。文字微調則讓使用者可以在預設基礎上進行個性化修改。後端 Hero 實體相應地增加了 `Soul` 字段，並透過 `SoulCatalogId` 標識來源。

這個階段解決了"有沒有"的問題，也還算是個孩子，磕磕絆絆地成長着。但隨着 Soul 內容越來越豐富，與 Hero 系統耦合在一起的架構開始顯現出侷限性。

### 第二階段：站內 Marketplace

爲了提供更好的 Soul 發現和複用體驗，我們建置了 SOUL Marketplace 目錄頁，支援瀏覽、搜索、詳情查看和收藏功能。

在這個階段，我們引入了 **50 組主 Catalog（基礎角色）** 和 **10 組正交規則（表達方式）** 的組合設計。主 Catalog 定義了 Agent 的核心人設，比如"霧港旅人"、"夜航獵手"這類抽象的角色設定；正交規則則定義了表達的方式，比如"簡潔幹練"、"囉嗦親切"等語言風格特徵。

50 × 10 = 500 個組合可能性，爲使用者提供了豐富的人格設定空間。這數量說多不多，說少不少，怎麼說呢，條條大路通羅馬，只是有的路好走一點罷了。後端透過 `catalog-sources.json` 產生完整的 SOUL 目錄，前端則負責將這些目錄項呈現爲可互動的卡片清單。

站內 Marketplace 是一個很好的過渡方案，但也只是過渡而已。它仍然依附於主系統，對於只想使用 Soul 功能的使用者來說，存取路徑還是太深了。畢竟誰願意繞一大圈才能做一件簡單的事呢？

### 第三階段：獨立平台拆分

最終，我們決定將 Soul 能力遷移到獨立存放庫（`repos/soul`），原主系統的 Marketplace 改爲外部跳轉引導，新平台採用 **Builder-first** 設計理念——預設首頁即爲建立工作臺，使用者開啟網站的第一時間就可以開始建立自己的人格設定。

這個階段的技術棧也進行了全面升級：採用 Vite 8 + React 19 + TypeScript 5.9 組合，使用 shadcn/ui 元件系統統一設計語言，引入 Tailwind CSS 4 的主題變數系統。前端工程化水平的提升，爲後續的功能疊代打下了堅實基礎。

一切都淡了......不，一切纔剛剛開始。

## 核心技術設計與實作

### 素材整合策略

Soul 平台的一個核心設計理念是**本地優先**。這意味着首頁必須在無後端情況下可完全執行，遠端素材失敗時不得阻斷頁面進入。

其實這也沒什麼了不起的，只是在設計系統時多考慮了一步罷了。本地快照作爲基線，遠端作爲增強，這種思路讓產品在任何網路條件下都能提供基本的可用性。具體實作上，我們採用了兩層素材架構：

```typescript
export async function loadBuilderMaterials(): Promise<BuilderMaterials> {
  const localMaterials = createLocalMaterials(snapshot)  // 本地基线

  try {
    const inspirationFragments = await fetchMarketplaceItems()  // 远程增强
    return { ...localMaterials, inspirationFragments, remoteState: "ready" }
  } catch (error) {
    return { ...localMaterials, remoteState: "fallback" }  // 优雅降级
  }
}
```

本地素材來自主系統文件的建置期快照，包含 50 組基礎角色和 10 組表達規則的完整資料。遠端素材則來自使用者發佈的 Soul，透過 Marketplace API 取得。兩者的結合，爲使用者提供了從官方範本到社區創意的完整素材光譜。想笑來僞裝自己掉下的淚......不，其實沒什麼，就是本地加遠程罷了。

### Soul 碎片資料模型

Soul 的核心資料抽象是 **SoulFragment**（靈魂碎片）：

```typescript
export type SoulFragment = {
  fragmentId: string
  group: "main-catalog" | "expression-rule" | "published-soul"
  title: string
  summary: string
  content: string
  keywords: string[]
  localized?: Partial<Record<AppLocale, LocalizedFragmentContent>>
  sourceRef: SoulFragmentSourceRef
  meta: SoulFragmentMeta
}
```

`group` 字段區分了碎片的型別：主目錄定義角色核心，正交規則定義表達方式，使用者發佈的 Soul 則標記爲 `published-soul`。`localized` 字段支援多語言，讓同一個碎片可以在不同語言環境下呈現不同的標題和描述。國際化設計要趁早，這話我們也算是用上了。

Builder 草稿狀態則封裝了使用者當前的編輯狀態：

```typescript
export type SoulBuilderDraft = {
  draftId: string
  name: string
  selectedMainFragmentId: string | null
  selectedRuleFragmentId: string | null
  inspirationSoulId: string | null
  mainSlotText: string
  ruleSlotText: string
  customPrompt: string
  previewText: string
  updatedAt: string
}
```

使用者在編輯器中選擇的每個碎片，其內容都會被拼接到對應的 slot（槽位）中，形成最終的預覽文字。`mainSlotText` 對應主角色內容，`ruleSlotText` 對應表達規則內容，`customPrompt` 則是使用者的額外補充指令。

```typescript
export function compilePreview(

  fragments: {

    ruleFragment: SoulFragment | null

  }

  // 组装逻辑：主角色 + 表达规则 + 灵感参考 + 自定义内容
}
```

編譯結果會展示在中央預覽面板中，使用者可以實時看到最終效果，並一鍵複製到剪貼簿。這功能說起來也挺簡單的，不是嗎？可是簡單的東西往往最實用。

### 前端狀態管理

Soul Builder 的前端狀態管理遵循一個重要原則：**狀態邊界清晰劃分**。具體來說，抽屜狀態不永續性，不直接寫入草稿；只有明確的 Builder 操作纔會觸發狀態變更。

```typescript
// 领域状态（useSoulBuilder）
export function useSoulBuilder() {
  // 素材加载与缓存
  // 槽位聚合与预览编译
  // 复制行为与反馈消息
  // Locale 安全的描述符
}

// 呈现状态（useHomeEditorState）
export function useHomeEditorState() {
  // activeSlot, drawerSide, drawerOpen
  // 默认焦点行为
}
```

這種分離確保了編輯狀態的安全性和界面的回應速度。抽屜的開啟關閉是純粹的 UI 互動，不需要觸發複雜的永續性邏輯。這無異於廢話了！不，其實很重要——界面狀態和業務狀態要明確區分，避免 UI 互動污染核心資料模型。

### 單抽屜生命週期

Soul Builder 採用單抽屜模式：同時只允許一個槽位抽屜開啟。點擊遮罩層、按 ESC 鍵或切換槽位都會自動關閉當前抽屜。這個設計簡化了狀態管理，也符合移動端抽屜互動的常見模式。

抽屜關閉不會清空當前編輯內容，使用者切換回來時，上下文得以保留。這種"輕量級"的抽屜設計，避免了使用者操作的中斷感。畢竟誰願意辛辛苦苦寫的東西，因爲不小心點錯就全沒了嗎？

### 雙語支援架構

國際化是 Soul 平台的重要特性。系統文案完全支援雙語切換，而使用者草稿文字則永遠不會因語言切換而被覆寫——因爲草稿文字本身就是使用者自由輸入的內容，不涉及系統翻譯。

官方靈感卡（Marketplace Soul）保持上游顯示名稱，但提供最佳努力的英文摘要。對於中文名稱的 Soul，我們透過預定義的映射規則產生英文版本：

```typescript
// 主角色英文名映射
const mainNameEnglishMap = {
  "雾港旅人": "Mistport Traveler",
  "夜航猎手": "Night Hunter",
  // ...
}

// 正交规则英文名映射
const ruleNameEnglishMap = {
  "简洁干练": "Concise & Professional",
  "啰嗦亲切": "Verbose & Friendly",
  // ...
}
```

這映射表看起來也挺簡單的，可是要維護好它，也得花不少心思。畢竟有 50 組主角色和 10 組正交規則，乘起來就是 500 個組合，這數量說大不大，說小也不小。

## 後端目錄產生

Soul Catalog 的批量產生在後端完成，使用 C# 實作了 50 × 10 = 500 個組合的自動化建立：

```csharp

        var soulSnapshot = BuildSoulSnapshot(main, orthogonal);

    }

```

暱稱產生演算法將主角色名和表達規則名組合在一起，創造出富有想象力的 Agent 代號：

```csharp
private static readonly string[] MainHandleRoots = [
    "雾港", "夜航", "零帧", "星渊", "霓虹", "断云", ...
];
private static readonly string[] OrthogonalHandleSuffixes = [
    "旅人", "猎手", "术师", "行者", "星使", ...
];
// 组合示例：雾港旅人、夜航猎手、零帧术师...
```

Soul 快照的拼裝則按照固定的範本格式，將主角色核心、標誌特徵、表達規則核心和輸出約束組合在一起：

```csharp
private static string BuildSoulSnapshot(main, orthogonal) => string.Join('\n', [
    $"你的人设内核来自「{main.Name}」：{main.Core}",
    $"保持以下标志性语言特征：{main.Signature}",
    $"你的表达规则来自「{orthogonal.Name}」：{orthogonal.Core}",
    $"必须遵循这些输出约束：{orthogonal.Signature}"
]);
```

這範本拼裝說起來也是無聊透頂的活兒，可是沒有這些無聊的工作，哪來有趣的產品呢？

## 平台遷移策略

Soul 從主系統拆分到獨立平台後，我們面臨的一個重要挑戰是如何處理已有使用者資料。這問題說起來也挺常見的——拆分容易，遷移難。我們採取了三項保障措施：

**向後相容保障**。已保存的 Hero SOUL 快照保持可見，歷史快照即使失去 Marketplace 來源 ID 仍可預覽。這意味着使用者之前的所有設定都不會丟失，只是展示位置發生了變化。畢竟誰也不想辛辛苦苦的設定，說沒就沒了。

**主系統介面棄用**。站內 Marketplace API 傳回 410 Gone 狀態碼，並附帶遷移提示，引導使用者存取 soul.hagicode.com。

**Hero SOUL 表單改造**。在 Hero Soul 編輯區域新增遷移提示區塊，明確告知使用者 Soul 平台已經獨立，並提供一鍵跳轉按鈕：

```typescript
// HeroSoulForm.tsx
<div className="rounded-2xl border border-orange-200/70 bg-orange-50/80 p-4">
  <div>{t('hero.soul.migrationTitle')}</div>
  <p>{t('hero.soul.migrationDescription')}</p>
  <Button onClick={onOpenSoulPlatform}>
    {t('hero.soul.openSoulPlatformAction')}
  </Button>
</div>
```

## 實踐要點總結

回顧 Soul 平台的整個開發程序，有幾點實踐經驗值得分享。這也算是過來人的一點心得，不是什麼大道理，只是踩過的坑罷了。

**本地優先的執行階段假設**。在設計相依遠端資料的特性時，始終假設網路可能不可用。本地快照作爲基線，遠端作爲增強，這種思路讓產品在任何網路條件下都能提供基本的可用性。畢竟這年頭，網路這東西，說斷就斷，誰也說不準。

**狀態邊界清晰劃分**。界面狀態和業務狀態要明確區分，避免 UI 互動污染核心資料模型。抽屜開關是純粹的 UI 狀態，不需要和草稿永續性混在一起。

**國際化設計要趁早**。如果你的產品有國際化需求，最好在資料模型設計階段就考慮進去。`localized` 字段雖然增加了資料結構的複雜度，但後續維護多語言內容的成本會大大降低。

**素材同步工作流要自動化**。Soul 平台的本地素材來自主系統文件，當上遊文件更新時，需要有機制同步到前端快照。我們設計了 `npm run materials:sync` 腳本自動化這個程序，確保素材始終和上游保持一致。

## 未來展望

基於當前的架構設計，Soul 平台未來可以考慮以下發展方向。這也只是一些粗淺的想法，不一定對，權當拋磚引玉罷了。

**社區共享生態**。支援使用者上傳和分享自定義 Soul，增加評分、評論和推薦機制，讓優秀的 Soul 設定能夠被更多人發現和使用。畢竟獨樂樂不如衆樂樂。

**多模態擴充**。除了文字風格，還可以考慮支援語音風格設定、表情符號使用偏好設定、程式碼風格與格式化規則等維度。這事兒說起來挺美好，做起來可能就......

**智能輔助**。基於使用場景自動推薦 Soul，風格遷移與融合，甚至 A/B 測試不同 Soul 的實際效果。美又何必在乎天晴陰呢？試試就知道了。

**跨平台同步**。支援從其他 AI 平台匯入人格設定，提供標準化的 Soul 匯出格式，與主流 Agent 框架集成。
