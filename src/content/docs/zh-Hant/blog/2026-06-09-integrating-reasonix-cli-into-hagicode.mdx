## 從零開始：如何將 Reasonix CLI 集成到 HagiCode 系統中

> 本文分享了將 Reasonix CLI 作為一等 Agent Provider 集成到 HagiCode 系統的完整技術實踐，涵蓋三層架構設計、關鍵技術決策和前後端實現細節。

## 背景

Reasonix CLI，說起來也是個挺有意思的東西。它是一個基於 ACP（Agent Communication Protocol）的 AI 代碼助手工具，提供了強大的流式傳輸和會話管理能力。其實在 HagiCode.Libs 層，我們已經把它的底層實現都搞定了，只是這些組件還處在孤立的狀態，就像一個個漂亮的珍珠，還沒串成項鍊。用戶無法通過 Hero 職業選擇、會話執行鏈路或監控面板來使用它，這多少有點可惜。

我們面臨的問題是：如何將 Reasonix 提升為與 Codex、Hermes 等同級別的一等 Agent Provider，實現完整的後端路由和前端展示？這可不是簡單地註冊一個枚舉值就能了事的，而是需要構建從底層抽象到用戶界面的完整鏈路。就像蓋房子，不能只打個地基就完事了，總得把牆砌起來，屋頂蓋上去。

這個集成的挑戰在於，Reasonix 作為一個本地 CLI 工具，有自己的性格和脾氣。比如它不需要連接字符串，所有參數都是用戶運行時配置；它可能根本沒有安裝，需要優雅降級處理；它兼容 anthropic 系列模型，但又有自己的 effort、budget 等 ACP 特有參數。這就像一個人，有自己獨特的處事方式，不能硬來。

經過仔細的架構設計和多輪討論，我們最終採用了一套清晰的三層架構方案，將 Reasonix 成功集成到系統中。這套方案不僅解決了眼前的問題，也為後續類似 CLI Provider 的集成提供了可複用的模式。其實很多事情就是這樣，一旦找到了正確的方法，後面的路就好走多了。

## 關於 HagiCode

本文分享的方案來自我們在 [HagiCode](https://hagicode.com) 項目中的實踐經驗。HagiCode 是一個開源的 AI 代碼助手項目，致力於為開發者提供強大的代碼生成、重構和優化能力。在開發過程中，我們遇到了各種各樣的技術挑戰，將 Reasonix 集成為一等 Agent Provider 就是其中之一。如果你覺得本文分享的方案有價值，說明我們的工程實踐還不錯，那麼 HagiCode 本身也值得關注一下。

## 核心內容

### 技術架構設計

系統採用了清晰的三層架構來分離關注點，每層都有明確的職責邊界：

**HagiCode.Libs 層**：這一層已經完成，提供了 CLI provider 的抽象和具體實現。它定義了 `ICliProvider<ReasonixOptions>` 接口，實現了 `ReasonixProvider` 來處理 ACP 流式傳輸和會話管理，同時支持 effort、budget、yolo、transcript 等參數。這一層的職責是提供穩定、可複用的底層能力，不涉及任何業務邏輯。就像房子的地基，雖然看不見，但很重要。

**hagicode-core 層**：這是我們本次集成的重點。它負責將底層抽象橋接到系統的統一接口上。具體工作包括註冊 `AIProviderType.ReasonixCli = 12` 枚舉值，創建 `ReasonixCliProvider` 作為 thin adapter 橋接 Libs 層，實現 `ReasonixGrain` 處理會話狀態和執行流，以及集成 Hero 系統進行參數映射和配置管理。這一層的核心是協調各組件，構建完整的業務鏈路。就像房子的承重牆，把各個部分連接起來。

**web 層**：負責向用戶展示和收集配置。我們需要重新生成 OpenAPI 類型以支持新枚舉值，實現視覺類型映射讓 Reasonix 有自己的圖標和顯示名稱，創建 CLI 參數配置表單讓用戶可以配置各個參數，以及添加多語言支持。這一層的重點是用戶體驗和交互設計。就像房子的裝修，做得好不好直接影響住得舒不舒服。

這樣的分層設計讓每一層都專注於自己的職責，降低了系統的複雜度，也便於後續維護和擴展。其實很多時候，把事情分清楚，反而會更簡單。

### 關鍵技術決策

在實現過程中，我們做了幾個關鍵的技術決策，這些決策對最終的架構和用戶體驗都有重要影響。

**決策一：使用專用 Grain**

我們創建了獨立的 `ReasonixGrain : IReasonixGrain, IExecutorStreamGrain`，而不是嘗試複用某個共享的 Grain。這個決策遵循了系統現有的 11 個 provider 的既定模式。雖然看起來可能會有些代碼重複，但專用 Grain 讓我們可以針對 Reasonix 的特性做精細化的控制，比如它特有的會話綁定機制和 ACP 消息映射。我們還定義了一個空的響應 DTO `ReasonixResponse` 作為類型區分符，雖然它不包含實際數據，但在類型系統中起到了重要的作用。就像每個都有自己的房間，哪怕空著，也是自己的空間。

**決策二：不創建專用 Settings 類**

與某些需要連接字符串的 Provider 不同，Reasonix 的所有配置都是用戶運行時設置的，不需要啟動時驗證。因此我們沒有創建專用的 Settings 類，而是將所有配置存儲在 `AIProviderOptions.Providers[ReasonixCli].Settings` 字典中。這種模式與 Qoder、Kiro、Kimi 等其他本地 CLI Provider 保持一致，簡化了代碼結構，也避免了不必要的抽象層。支持的設置鍵包括：`effort`、`budgetUsd`、`transcriptPath`、`enableYolo`、`arguments`、`startupTimeoutMs`、`reasoning`。有時候簡單點，反而更好。

**決策三：Provider 策略健康監控**

Reasonix 是用戶本地安裝的 CLI，可能根本沒有安裝，或者不在系統 PATH 中。這種情況下，我們不應該直接報錯，而是應該優雅地降級處理。我們使用了 `Provider` 策略，通過 `CommandUtil.TryResolveExecutablePath` 來檢查 CLI 是否可用。如果檢查失敗，UI 會顯示為"不可用"，但不會影響系統的其他部分。這種設計讓系統更加健壯，也給用戶清晰的反饋。畢竟誰也不會希望因為一個小問題，整個系統都掛了。

**決策四：經濟系統分類**

在 HagiCode 系統中，不同的 Provider 有不同的經濟系統分類。我們決定讓 Reasonix 默認使用 `'claude'` 經濟系統分類，因為 Reasonix 本身兼容 anthropic 系列模型。目前只有 Codex 和 Copilot 有專用的經濟系統分類，其他 Provider 都是複用現有的分類。這樣既保持了系統的簡潔性，又能正確處理計費和成本統計。複用也是個智慧，不需要什麼都從頭來。

**決策五：模型兼容性**

Reasonix 通過 `--model` 標志支持多種模型，特別是 anthropic 系列。我們在 `secondary-professions.index.json` 中添加了兼容性映射，讓用戶可以在 Reasonix 中選擇這些模型。這種設計既尊重了 Reasonix 的能力，又保持了系統的一致性，用戶無需理解底層的區別，就能順暢地使用各種模型。用戶也不容易，還是讓他們簡單點好。

**Thin Adapter 實現**

`ReasonixCliProvider` 是連接 Libs 層和系統統一接口的關鍵組件：

```csharp
public sealed class ReasonixCliProvider : IAIProvider, IVersionedAIProvider, IAsyncDisposable
{
    private static readonly IReadOnlyList<string> SupportedSettingKeys =
    [
        "effort",
        "budgetUsd",
        "transcriptPath",
        "enableYolo",
        "arguments",
        "startupTimeoutMs",
        "reasoning"
    ];

    private readonly ICliProvider<ReasonixOptions> _provider;
    private readonly ConcurrentDictionary<string, string> _sessionBindings = new(StringComparer.Ordinal);

    public async IAsyncEnumerable<AIStreamingChunk> StreamCoreAsync(
        AIRequest request,
        string? sessionId = null,
        [EnumeratorCancellation] CancellationToken cancellationToken = default)
    {
        var options = BuildOptions(request, sessionId);
        await foreach (var message in _provider.StreamAsync(options, cancellationToken))
        {
            yield return MapToStreamingChunk(message);
        }
    }

    private ReasonixOptions BuildOptions(AIRequest request, string? sessionId)
    {
        return new ReasonixOptions
        {
            ExecutablePath = GetExecutablePath(),
            WorkingDirectory = GetWorkingDirectory(),
            Model = _config.Model,
            Effort = _config.Settings.GetValueOrDefault("effort", "medium"),
            Budget = _config.Settings.GetValueOrDefault("budgetUsd", 10.0),
            Yolo = _config.Settings.GetValueOrDefault("enableYolo", false),
            TranscriptPath = _config.Settings.GetValueOrDefault("transcriptPath"),
            Arguments = _config.Settings.GetValueOrDefault("arguments", ""),
            StartupTimeout = GetStartupTimeout(),
            EnvironmentVariables = _environmentVariables,
            CessionId = sessionId ?? GetCessionId()
        };
    }
}
```

這個 adapter 的關鍵職責是：
1. 驗證配置參數，拒絕不支持的設置鍵
2. 維護會話綁定關係，支持會話恢復
3. 將 ACP 消息映射到系統的統一格式
4. 使用 `ProviderErrorAutoRetryCoordinator` 實現自動重試

就像一個翻譯官，把一種語言翻譯成另一種語言，還得保證意思準確無誤。

**Orleans Grain 實現**

`ReasonixGrain` 負責處理會話狀態和執行流：

```csharp
public class ReasonixGrain : Grain, IReasonixGrain, IExecutorStreamGrain
{
    private readonly Dictionary<string, ExecutorToolLifecycleStatus> _toolLifecycleState =
        new(StringComparer.Ordinal);

    public IAsyncEnumerable<ReasonixResponse> ExecuteCommandStreamAsync(
        string command,
        string? heroId = null,
        CancellationToken token = default,
        string? executionMessageId = null,
        string? systemMessage = null,
        Dictionary<string, string>? requestSettings = null)
    {
        var request = BuildRequest(command, isEdit: false, heroId, executionMessageId, systemMessage, requestSettings);
        return SendAsync(request, heroId, token);
    }

    private async IAsyncEnumerable<ReasonixResponse> SendAsync(
        AIRequest request,
        string? heroId,
        [EnumeratorCancellation] CancellationToken token)
    {
        _cancellationTokenSource = new CancellationTokenSource();
        var linkedToken = CancellationTokenSource.CreateLinkedTokenSource(token, _cancellationTokenSource.Token);

        var provider = await ResolveReasonixProviderAsync(heroId);

        await foreach (var chunk in provider.StreamAsync(request, linkedToken.Token))
        {
            var response = BuildChunkResponse(chunk);
            yield return response;
        }
    }

    private async Task<IAIProvider> ResolveReasonixProviderAsync(string? heroId)
    {
        // Hero 感知的配置回退邏輯
        var config = await HeroProviderResolver.ResolveAsync(AIProviderType.ReasonixCli, heroId);
        return _aiProviderFactory.CreateProvider(AIProviderType.ReasonixCli, config);
    }
}
```

Grain 的核心功能包括：
1. 使用 `[PersistentState("reasonix-interop")]` 維護會話狀態
2. 實現 Hero 感知的配置回退邏輯
3. 追蹤工具生命週期狀態
4. 支持取消令牌鏈，確保執行可以及時中斷

這就像一個管家，把事情安排得井井有條。

### 最佳實踐和注意事項

在實現過程中，我們總結了一些最佳實踐和需要注意的地方。

**參數驗證**

ReasonixCliProvider 必須嚴格驗證配置參數，拒絕不支持的設置鍵：

```csharp
public void ValidateConfigurationOverrides(Dictionary<string, string?> overrides)
{
    foreach (var key in overrides.Keys)
    {
        if (!SupportedSettingKeys.Contains(key))
        {
            throw new HeroProviderConfigurationException(
                $"Unsupported setting key '{key}' for Reasonix provider");
        }
    }
}
```

這樣可以防止用戶配置錯誤的參數，避免運行時錯誤。就像守門員，不能讓不該進的東西混進來。

**會話綁定管理**

使用 `ConcurrentDictionary` 管理會話綁定，支持會話恢復：

```csharp
_sessionBindings[cessionId] = sessionId;

// 在後續請求中綁定已存在的會話
if (_sessionBindings.TryGetValue(cessionId, out var boundSessionId))
{
    options.SessionId = boundSessionId;
}
```

這種設計讓用戶可以在中斷後恢復之前的會話，提供更好的體驗。就像把一個故事記住，下次還能繼續講。

**優雅降級處理**

前端應該檢查 CLI 可用性，並給出友好的提示：

```typescript
const reasonixAvailable = await healthApi.checkCliAvailable('reasonix');
if (!reasonixAvailable) {
  showMessage('Reasonix CLI 未安裝，請在系統 PATH 中配置');
}
```

不要讓用戶遇到莫名其妙的錯誤，提前檢查並給出明確的指導。畢竟誰也不願意莫名其妙地被卡住。

**測試覆蓋**

完善的測試是質量保證的關鍵：

```csharp
[Fact]
public async Task ExecuteCommandStreamAsync_WithValidCommand_StreamsReasonixResponse()
{
    // Arrange
    var grain = _grainFactory.GetGrain<IReasonixGrain>("test-cession");
    var responses = new List<ReasonixResponse>();

    // Act

        responses.Add(response);

    responses.Should().NotBeEmpty();
    responses.All(r => r.Kind == ExecutorResponseKind.Content).Should().BeTrue();

```

這樣的單元測試可以驗證核心功能是否正常工作，防止回歸錯誤。就像考試前做練習題，總得確保自己真的掌握了。
