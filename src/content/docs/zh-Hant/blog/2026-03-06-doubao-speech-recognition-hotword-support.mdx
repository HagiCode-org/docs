## 豆包語音識別熱詞功能實作指南

> 本文將詳細介紹如何在 HagiCode 專案中實作豆包語音識別的熱詞支援功能，透過自定義熱詞和平台熱詞表兩種方式，顯著提升特定領域詞彙的識別準確率。

## 背景

語音識別技術發展這麼多年了，其實有個問題一直困擾着開發人員們。通用語音識別模型雖然能覆寫日常用語，可對於專業術語、產品名稱、人名這些詞，識別準確率總差那麼點意思。想想看，醫療領域的語音助手要準確識別"高血壓"、"糖尿病"、"冠心病"；法律系統要精準捕捉"案由"、"答辯"、"舉證責任"——這些場景下，通用模型的表現怎麼說呢，也算盡力了。

在 HagiCode 專案中，我們也遇到了同樣的挑戰。作爲一個多功能的 AI 程式碼助手，HagiCode 需要處理各種技術術語的語音識別場景。然而，豆包語音識別 API 在預設情況下，並不能完全滿足我們對專業術語準確率的那些要求。其實也不是豆包不夠好，只是每個領域都有自己的一套術語體系。經過一番調研和技術探索，我們發現豆包語音識別 API 實際上提供了熱詞支援功能，只要簡單設定一下，就能顯著提升特定詞彙的識別準確率。這倒是有點像，你告訴它你要注意什麼詞，它就會更用心去聽那些詞。

本文要分享的，就是在 HagiCode 專案中實作豆包語音識別熱詞功能的完整方案。兩種模式，自定義熱詞和平台熱詞表，都可以用，也都能組合用。透過這套方案，開發人員可以根據業務場景靈活設定熱詞，讓語音識別系統"認識"那些專業、罕見但又至關重要的詞彙。

## 核心實作

### 兩種熱詞模式剖析

豆包語音識別 API 爲我們提供了兩種熱詞設定方式，每種方式都有其獨特的應用場景和優勢。

**自定義熱詞模式**允許我們透過 `corpus.context` 字段直接傳遞熱詞文字。這種方式非常適合需要快速設定少量熱詞的場景，比如臨時需要識別某個產品名稱或者人名。在 HagiCode 的實作中，我們將使用者輸入的多行熱詞文字剖析爲字串清單，然後按照豆包 API 的要求格式化爲 `context_data` 陣列。怎麼說呢，這種方式很直接，就像告訴對方"你要注意這些詞"，然後它就去注意了。

**平台熱詞表模式**則透過 `corpus.boosting_table_id` 字段參考豆包自學習平台預設定的熱詞表。這種方式適合需要管理大量熱詞的場景，我們可以在豆包自學習平台上建立和維護熱詞表，然後透過 ID 進行參考。對於 HagiCode 這類需要持續更新和維護專業術語的專案來說，這種模式提供了更好的可管理性。畢竟，熱詞多了之後，找個地方統一管理，總比每次都要手動輸入要好。

有意思的是，這兩種模式還可以組合使用。豆包 API 支援在同一個請求中同時包含自定義熱詞和平台熱詞表 ID，透過 `combine_mode` 參數控制組合策略。這種靈活性使得 HagiCode 能夠應對各種複雜的專業術語識別需求。這也倒是挺好，有時候多種方式組合一下，效果可能更好。

### 前端型別定義與驗證

在 HagiCode 的前端實作中，我們定義了一套完整的熱詞設定型別和驗證邏輯。首先是型別定義部分：

```typescript
export interface HotwordConfig {
  contextText: string;           // 多行热词文本
  boostingTableId: string;      // 豆包平台热词表 ID
  combineMode: boolean;          // 是否组合使用
}
```

這個簡單的介面包含了熱詞功能的所有設定項。其中 `contextText` 是使用者最直觀感受到的部分——我們允許使用者每行輸入一個熱詞短語，這種方式非常符合直覺。畢竟，讓使用者一行一個詞，總比讓使用者理解複雜的設定規則要好。

接下來是驗證函式的實作。考慮到豆包 API 的限制，我們制定了嚴格的驗證規則：熱詞文字最多 100 行，每行最多 50 個字符，總共最多 5000 個字符；boosting_table_id 最多 200 個字符，只允許字母、數字、下劃線和連字符。這些限制不是我們憑空想象的，而是基於豆包官方文件的實際要求。畢竟，API 的限制就是 API 的限制，我們也沒辦法，只能遵守。

```typescript
export function validateContextText(contextText: string): HotwordValidationResult {
  if (!contextText || contextText.trim().length === 0) {
    return { isValid: true, errors: [] };
  }

  const lines = contextText.split('\n').filter(line => line.trim().length > 0);
  const errors: string[] = [];

  if (lines.length > 100) {
    errors.push(`热词行数不能超过 100 行，当前为 ${lines.length} 行`);
  }

  const totalChars = contextText.length;
  if (totalChars > 5000) {
    errors.push(`热词总字符数不能超过 5000，当前为 ${totalChars}`);
  }

  for (let i = 0; i < lines.length; i++) {
    if (lines[i].length > 50) {
      errors.push(`第 ${i + 1} 行热词超过 50 个字符限制`);
    }
  }

  return { isValid: errors.length === 0, errors };
}

export function validateBoostingTableId(boostingTableId: string): HotwordValidationResult {
  if (!boostingTableId || boostingTableId.trim().length === 0) {
    return { isValid: true, errors: [] };
  }

  const errors: string[] = [];

  if (boostingTableId.length > 200) {
    errors.push(`boosting_table_id 不能超过 200 个字符，当前为 ${boostingTableId.length}`);
  }

  if (!/^[a-zA-Z0-9_-]+$/.test(boostingTableId)) {
    errors.push('boosting_table_id 只能包含字母、数字、下划线和连字符');
  }

  return { isValid: errors.length === 0, errors };
}
```

這些驗證函式在使用者設定熱詞時就會立即執行，確保問題在最早階段被發現。對於使用者體驗來說，這種即時反饋是非常重要的。畢竟，使用者輸入的時候就知道哪裏錯了，總比提交後才發現要好。

### 前端設定永續性

在 HagiCode 的前端實作中，我們選擇使用瀏覽器的 localStorage 來存儲熱詞設定。這個設計決策背後有幾點考量：首先，熱詞設定是非常個性化的設定，不同使用者可能有不同的專業領域需求；其次，這種方式簡化了後端實作，不需要額外的資料庫表和 API 介面；最後，使用者在瀏覽器中設定一次後，後續使用都能自動載入，非常方便。其實說白了，就是省事。

```typescript
const HOTWORD_STORAGE_KEYS = {
  contextText: 'hotword-context-text',
  boostingTableId: 'hotword-boosting-table-id',
  combineMode: 'hotword-combine-mode',
} as const;

export const DEFAULT_HOTWORD_CONFIG: HotwordConfig = {
  contextText: '',
  boostingTableId: '',
  combineMode: false,
};

// 加载热词配置
export function loadHotwordConfig(): HotwordConfig {
  const contextText = localStorage.getItem(HOTWORD_STORAGE_KEYS.contextText) || '';
  const boostingTableId = localStorage.getItem(HOTWORD_STORAGE_KEYS.boostingTableId) || '';
  const combineMode = localStorage.getItem(HOTWORD_STORAGE_KEYS.combineMode) === 'true';

  return { contextText, boostingTableId, combineMode };
}

// 保存热词配置
export function saveHotwordConfig(config: HotwordConfig): void {
  localStorage.setItem(HOTWORD_STORAGE_KEYS.contextText, config.contextText);
  localStorage.setItem(HOTWORD_STORAGE_KEYS.boostingTableId, config.boostingTableId);
  localStorage.setItem(HOTWORD_STORAGE_KEYS.combineMode, String(config.combineMode));
}
```

這段程式碼的邏輯非常簡單清晰。載入設定時從 localStorage 讀取，保存設定時寫入 localStorage。我們還提供了預設設定，確保在沒有任何設定時系統也能正常工作。畢竟，總得有個預設值吧。

### 後端 SDK 設定擴充

在 HagiCode 的後端實作中，我們需要在 SDK 設定類中新增熱詞相關的屬性。考慮到 C# 的語言特性和使用習慣，我們採用了 `List<string>` 來存儲自定義熱詞上下文：

```csharp
public class DoubaoVoiceConfig
{
    /// <summary>
    /// 应用 ID
    /// </summary>
    public string AppId { get; set; } = string.Empty;

    /// <summary>
    /// 访问令牌
    /// </summary>

    /// <summary>
    /// 服务 URL
    /// </summary>
    public string ServiceUrl { get; set; } = string.Empty;

    /// <summary>
    /// 自定义热词上下文列表
    /// </summary>
    public List<string>? HotwordContexts { get; set; }

    /// <summary>
    /// 豆包平台热词表 ID
    /// </summary>
    public string? BoostingTableId { get; set; }

```

這個設定類的設計遵循了 HagiCode 一貫的簡潔風格。`HotwordContexts` 是可空的清單型別，`BoostingTableId` 是可空的字串，這樣在沒有任何熱詞設定時，這些屬性不會對請求造成任何影響。畢竟，不用的時候就不應該存在，這才叫乾淨。

### Payload 建置邏輯

Payload 的建置是整個熱詞功能的核心。當我們有了熱詞設定後，需要按照豆包 API 的要求格式化爲正確的 JSON 結構。這個程序發生在 SDK 發送請求之前：

```csharp
private void AddCorpusToRequest(Dictionary<string, object> request)
{
    var corpus = new Dictionary<string, object>();

    // 添加自定义热词
    if (Config.HotwordContexts != null && Config.HotwordContexts.Count > 0)
    {
        corpus["context"] = new Dictionary<string, object>
        {
            ["context_type"] = "dialog_ctx",
            ["context_data"] = Config.HotwordContexts
                .Select(text => new Dictionary<string, object> { ["text"] = text })
                .ToList()
        };
    }

    // 添加平台热词表 ID
    if (!string.IsNullOrEmpty(Config.BoostingTableId))
    {
        corpus["boosting_table_id"] = Config.BoostingTableId;
    }

    // 只有当 corpus 不为空时才添加到请求中
    if (corpus.Count > 0)
    {
        request["corpus"] = corpus;
    }
}
```

這段程式碼展示瞭如何根據設定動態建置 corpus 字段。關鍵點在於：只有當確實存在熱詞設定時，我們纔會新增 corpus 字段。這種設計確保了向後相容性——沒有設定熱詞時，請求的結構與之前完全一致。畢竟，相容性很重要，不能因爲加個功能就把之前的邏輯搞亂了。

### WebSocket 參數傳遞

在前端和後端之間，熱詞參數透過 WebSocket 控制訊息進行傳遞。HagiCode 的設計是：前端在開始錄音時從 localStorage 載入熱詞設定，然後透過 WebSocket 訊息發送給後端。

```typescript
const controlMessage = {
  type: 'control',
  payload: {
    command: 'StartRecognition',
    contextText: '高血压\n糖尿病\n冠心病',
    boosting_table_id: 'medical_table',
    combineMode: false
  }
};
```

這裏有一個細節需要注意：前端傳遞的是多行文字（用換行符分隔），後端需要進行剖析。後端的 WebSocket Handler 會剖析這些參數並傳遞給 SDK：

```csharp
private async Task HandleControlMessageAsync(
    string connectionId,
    DoubaoSession session,
    ControlMessage message)
{
    if (message.Payload is SessionControlRequest controlRequest)
    {
        // 解析热词参数
        string? contextText = controlRequest.ContextText;
        string? boostingTableId = controlRequest.BoostingTableId;
        bool? combineMode = controlRequest.CombineMode;

        // 解析多行文本为热词列表
        if (!string.IsNullOrEmpty(contextText))
        {
            var hotwords = contextText
                .Split('\n', StringSplitOptions.RemoveEmptyEntries)
                .Select(s => s.Trim())
                .Where(s => s.Length > 0)
                .ToList();

            session.HotwordContexts = hotwords;
        }

        session.BoostingTableId = boostingTableId;
    }
}
```

透過這樣的設計，熱詞設定從前端到後端的傳遞變得清晰而高效。其實也沒什麼特別的，就是一層一層傳下去而已。

## 實踐指南

### 設定自定義熱詞

在實際使用中，設定自定義熱詞非常簡單。開啟 HagiCode 的語音識別設定頁面，找到"熱詞設定"區域。在"自定義熱詞文字"輸入框中，每行輸入一個熱詞短語。

比如，如果你正在開發一個醫療相關的應用，可以這樣設定：

```
高血压
糖尿病
冠心病
心绞痛
心肌梗死
心力衰竭
```

保存設定後，每次開始語音識別時，這些熱詞都會自動傳遞給豆包 API。實際測試表明，設定熱詞後，相關專業術語的識別準確率有了明顯提升。怎麼說呢，效果還是有的，至少比之前好多了。

### 設定平台熱詞表

如果你需要管理大量的熱詞，或者熱詞需要頻繁更新，那麼平台熱詞表模式更適合你。首先需要在豆包自學習平台上建立熱詞表，取得產生的 boosting_table_id，然後在 HagiCode 的設定頁面中輸入這個 ID。

豆包自學習平台提供了熱詞的批量匯入、分類管理等功能，對於需要管理大量專業術語的團隊來說非常實用。透過平台管理熱詞，可以實作熱詞的集中維護和統一更新。畢竟，熱詞多了之後，有個地方統一管理，總比每次都要手動輸入要好。

### 組合模式的使用

在某些複雜場景下，你可能需要同時使用自定義熱詞和平台熱詞表。這時只需要在 HagiCode 中同時設定兩種熱詞，並開啓"組合模式"開關。

組合模式下，豆包 API 會同時考慮兩種熱詞來源，識別準確率通常比單獨使用任意一種更高。不過需要注意的是，組合模式會增加請求的複雜度，建議在實際測試後再決定是否啓用。畢竟，複雜度增加了，是不是真的值得，還是得看實際效果。

## 注意事項

在實作和使用熱詞功能時，有幾點需要特別注意。

首先是字符限制。豆包 API 對熱詞有嚴格的限制，包括行數、每行字符數、總字符數等。如果超出限制，API 會傳回錯誤。在 HagiCode 的前端實作中，我們透過驗證函式在使用者輸入階段就進行檢查，避免將無效設定發送到後端。畢竟，提前發現問題，總比等 API 傳回錯誤要好。

其次是 boosting_table_id 的格式。這個字段只允許字母、數字、下劃線和連字符，不允許包含空格或其他特殊字元。在豆包自學習平台上建立熱詞表時，需要注意命名規範。其實這也難怪，API 對格式的要求總是比較嚴格的。

第三是向後相容性。熱詞參數是完全可選的，不設定熱詞時，系統的工作方式與之前完全一致。這種設計確保了現有使用者不會受到任何影響，也便於逐步遷移和升級。畢竟，不能因爲加個功能就把之前的邏輯搞亂了。

最後是錯誤處理。當熱詞設定無效時，豆包 API 會傳回相應的錯誤資訊。HagiCode 的實作會記錄詳細的日誌，便於開發人員排查問題。同時，前端也會在界面上展示驗證錯誤，說明使用者修正設定。錯誤處理做得好，使用者體驗自然也就好了。
