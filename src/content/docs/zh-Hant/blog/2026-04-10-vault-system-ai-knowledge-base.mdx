## 用 Vault 系統建構 AI 時代的跨專案知識庫

> 臨摹專案學習法正在成為主流，只是學習資料分散、上下文斷裂的痛點讓 AI 助手難以發揮最大價值。本文介紹 HagiCode 專案的 Vault 系統設計——透過統一的儲存抽象層，讓 AI 助手能夠理解和存取所有學習資源，實現真正的跨專案知識複用。

## 背景

其實，在 AI 時代，我們學習新技術的方式正在悄然改變。傳統的讀書、看視頻固然重要，但「臨摹專案」——深入研究和學習優秀開源專案的程式碼、架構和設計模式——確實越來越高效。直接執行和修改高品質的開源專案，能讓你最快理解真實世界的工程實踐。

只是這種方式也帶來了新的挑戰。

**學習資料太分散**。筆記可能在 Obsidian 裡，程式碼倉庫散落在各個資料夾，AI 助手的對話歷史又是一個獨立的資料孤島。每次需要 AI 幫忙分析某個專案時，都得手動複製程式碼片段、整理上下文，過程相當繁瑣。

**上下文經常斷掉**。AI 助手無法直接存取本機學習資源，每次對話都得重新提供背景資訊。臨摹的程式碼倉庫更新快，手動同步容易出錯。更糟的是，多個學習專案之間難以共享知識——在 A 專案中學到的設計模式，AI 處理 B 專案時完全不知道。

這些問題的本質是「資料孤島」。如果能有個統一的儲存抽象層，讓 AI 助手能夠理解和存取所有學習資源，問題就迎刃而解了。

為了解決這些痛點，我們在開發 HagiCode 時做了一個關鍵的設計決策：建構一個 Vault 系統作為統一的知識儲存抽象層。這個決策帶來的變化，可能比想像的還要大——稍後具體說。

## 關於 HagiCode

本文分享的方案來自在 [HagiCode](https://hagicode.com) 專案中的實踐經驗。HagiCode 是一個基於 OpenSpec 工作流的 AI 程式碼助手，它的核心理念是讓 AI 不僅會「說」，更會「做」——能夠直接操作程式碼倉庫、執行指令、執行測試。GitHub：[github.com/HagiCode-org/site](https://github.com/HagiCode-org/site)

在開發過程中，我們發現 AI 助手需要頻繁存取使用者的各類學習資源：程式碼倉庫、筆記文件、設定檔等。如果每次都要使用者手動提供，體驗就太糟糕了。這促使設計了 Vault 系統。

## 核心設計

### 多類型支援

HagiCode 的 Vault 系統支援四種類型，分別對應不同的使用情境：

| 類型 | 用途 | 典型情境 |
|------|------|----------|
| `folder` | 通用資料夾類型 | 臨時學習資料、草稿 |
| `coderef` | 專門用於臨摹程式碼專案 | 系統化學習某個開源專案 |
| `obsidian` | 與 Obsidian 筆記軟體整合 | 現有筆記庫的複用 |
| `system-managed` | 系統自動管理 | 專案設定、提示詞模板等 |

其中 `coderef` 類型是 HagiCode 中最常用的，它為臨摹程式碼專案提供了標準化的目錄結構和 AI 可讀的元數據描述。為什麼要專門設計這個類型？因為臨摹一個開源專案不是簡單的「下載程式碼」，需要同時管理程式碼本身、學習筆記、設定檔等多種內容，`coderef` 把這些都規範好了。

### 持久化儲存機制

Vault 的登錄檔以 JSON 格式持久化儲存到檔案系統：

```csharp
_registryFilePath = Path.Combine(absoluteDataDir, "personal-data", "vaults", "registry.json");
```

這個設計看起來簡單，實則經過深思熟慮：

**簡單可靠**。JSON 格式人類可讀，便於除錯和手動修改。當系統出現問題時，可以直接開啟檔案檢查狀態，甚至手動修復——這在開發階段特別有用。

**降低依賴**。檔案系統儲存避免了資料庫的複雜性。不需要額外安裝和設定資料庫服務，降低了系統複雜度和維護成本。

**並發安全**。使用 `SemaphoreSlim` 確保多執行緒安全。在 AI 程式碼助手的情境下，可能會有多個操作同時存取 vault 登錄檔，需要做好並發控制。

### AI 上下文整合

系統的核心能力在於能夠自動將 vault 資訊注入到 AI 提案的上下文中：

```typescript
export function buildTargetVaultsText(
  vaults: VaultForText[],
  template: VaultPromptTemplate = DEFAULT_VAULT_PROMPT_TEMPLATE,
): string {
  const readOnlyVaults = vaults.filter((vault) => vault.accessType === 'read');
  const editableVaults = vaults.filter((vault) => vault.accessType === 'write');

  const sections = [
    buildVaultSection(readOnlyVaults, template.reference),
    buildVaultSection(editableVaults, template.editable),
  ].filter(Boolean);

  return `\n\n### ${template.heading}\n\n${sections.join('\n')}`;
}
```

這樣 AI 助手就能自動理解可用的學習資源，無需使用者每次手動提供上下文。這個設計讓 HagiCode 的體驗變得特別自然——告訴 AI「幫我分析 React 的並發渲染」，AI 就能自動找到之前註冊的 React 學習 vault，而不是一遍遍貼程式碼。

### 存取控制機制

系統將 vault 分為兩種存取類型：

- **reference（唯讀）**：AI 僅用於分析和理解，不能修改內容
- **editable（可編輯）**：AI 可以根據任務需要修改內容

這種區分讓 AI 知道哪些內容是「唯讀參考」，哪些是「可以動手改的」，避免了誤操作風險。比如註冊了一個開源專案的 vault 作為學習材料，肯定不希望 AI 隨手修改裡面的程式碼——那就標記為 `reference`。但如果是自己的專案 vault，就可以標記為 `editable`，讓 AI 幫著改程式碼。

## 實踐指南

### CodeRef Vault 的標準化結構

對於 coderef 類型的 vault，系統提供了一套標準化的目錄結構：

```
my-coderef-vault/
├── index.yaml          # vault 元數據描述
├── AGENTS.md           # AI 助手的操作指南
├── docs/               # 存放學習筆記和文件
└── repos/              # 透過 Git 子模組管理臨摹的程式碼倉庫
```

這個結構的設計理念是什麼？

**docs/** 存放學習筆記，用 Markdown 格式記錄對程式碼的理解、架構分析、踩坑經驗。這些筆記不僅自己看，AI 也能讀懂——在處理相關任務時會自動參考。

**repos/** 透過 Git 子模組管理臨摹的倉庫，而不是直接複製程式碼。這樣做有兩個好處：一是保持與上游同步，一個 `git submodule update` 就能拿到最新程式碼；二是節省空間，多個 vault 可以引用同一個倉庫的不同版本。

**index.yaml** 包含 vault 的元數據，讓 AI 助手快速理解用途和內容。相當於給 vault 寫了個「自我介紹」，AI 第一次見到就知道這是幹嘛的。

**AGENTS.md** 是專門寫給 AI 助手看的指南，說明如何處理 vault 中的內容。可以在這裡告訴 AI：「分析這個專案時重點關注效能優化相關的程式碼」或者「不要修改測試檔案」。

### 建立和使用 Vault

建立一個 CodeRef vault 很簡單：

```typescript
const createCodeRefVault = async () => {
  const response = await VaultService.postApiVaults({
    requestBody: {
      name: "React Learning Vault",
      type: "coderef",
      physicalPath: "/Users/developer/vaults/react-learning",
      gitUrl: "https://github.com/facebook/react.git"
    }
  });

  // 系統會自動：
  // 1. 克隆 React 倉庫到 vault/repos/react
  // 2. 建立 docs/ 目錄用於筆記
  // 3. 產生 index.yaml 元數據
  // 4. 建立 AGENTS.md 指南檔案

  return response;
};
```

然後在 AI 提案中引用這個 vault：

```typescript
const proposal = composeProposalChiefComplaint({
  chiefComplaint: "幫我分析 React 的並發渲染機制",
  repositories: [
    { id: "react", gitUrl: "https://github.com/facebook/react.git" }
  ],
  vaults: [
    {
      id: "react-learning",
      name: "React Learning Vault",
      type: "coderef",
      physicalPath: "/vaults/react-learning",
      accessType: "read"  // AI 只能讀取，不能修改
    }
  ],
  quickRequestText: "重點關注 fiber 架構和 scheduler 實作"
});
```

### 典型使用情境

**情境一：系統化學習開源專案**

建立一個 CodeRef vault，透過 Git 子模組管理目標倉庫，在 `docs/` 目錄記錄學習筆記。AI 可以同時存取程式碼和筆記，提供更精準的分析。在學習某個模組時寫的筆記，AI 後續分析相關程式碼時會自動參考——就像有個「助手」記住了之前的思考。

**情境二：複用 Obsidian 筆記庫**

如果已經在用 Obsidian 管理筆記，直接把現有的 vault 註冊到 HagiCode 中就行。AI 可以直接存取知識庫，無需手動複製貼上。這個功能特別實用，很多人都有累積多年的筆記庫，接入之後 AI 就能「讀」懂知識體系。

**情境三：跨專案知識複用**

多個 AI 提案可以引用同一個 vault，實現知識的跨專案複用。比如建立了一個「設計模式學習 vault」，裡面記錄了各種設計模式的筆記和程式碼範例。無論在分析哪個專案，AI 都能參考這個 vault 中的內容——知識不用重複累積。

### 路徑安全機制

系統嚴格校驗路徑，防止路徑穿越攻擊：

```csharp
private static string ResolveFilePath(string vaultRoot, string relativePath)
{
    var rootPath = EnsureTrailingSeparator(Path.GetFullPath(vaultRoot));
    var combinedPath = Path.GetFullPath(Path.Combine(rootPath, relativePath));
    if (!combinedPath.StartsWith(rootPath, StringComparison.OrdinalIgnoreCase))
    {
        throw new BusinessException(VaultRelativePathTraversalCode,
            "Vault file paths must stay inside the registered vault root.");
    }
    return combinedPath;
}
```

這確保了所有檔案操作都在 vault 的根目錄範圍內，防止惡意路徑存取。安全這塊不能馬虎，AI 助手要操作檔案系統，必須把邊界劃清楚。

## 注意事項

使用 HagiCode Vault 系統時，有幾點需要特別注意：

1. **路徑安全**：確保自訂路徑在允許的範圍內，否則系統會拒絕操作。這是為了防止誤操作和潛在的安全風險。

2. **Git 子模組管理**：CodeRef vault 推薦使用 Git 子模組而非直接複製程式碼。好處前面說過——保持同步、節省空間。只是子模組有自己的使用方式，第一次使用可能需要熟悉一下。

3. **檔案預覽限制**：系統限制檔案大小（256KB）和數量（500個），超大檔案需分批處理。這個限制是為了效能考慮，如果遇到超大檔案，可以手動拆分或者用其他方式處理。

4. **診斷資訊**：建立 vault 會返回診斷資訊，失敗時可用於除錯。遇到問題時先看診斷資訊，大部分情況下都能找到線索。

## 總結

HagiCode 的 Vault 系統本質上是在解決一個簡單但深刻的問題：如何讓 AI 助手理解和使用本機知識資源。

透過統一的儲存抽象層、標準化的目錄結構、自動化的上下文注入，實現了「一次註冊，處處複用」的知識管理方式。建立一個 vault 後，無論是學習筆記、程式碼倉庫還是文件資料，AI 都能自動存取和理解。

這種設計帶來的體驗提升是明顯的。不再需要手動複製程式碼片段、重複解釋背景資訊——AI 助手就像一個真正了解專案情況的同事，能夠基於已有知識提供更有價值的幫助。

本文分享的 Vault 系統，正是開發 HagiCode 過程中實際踩坑、實際優化出來的方案。如果覺得這套設計有價值，說明工程實力還不錯——那麼 HagiCode 本身也值得關注一下。
