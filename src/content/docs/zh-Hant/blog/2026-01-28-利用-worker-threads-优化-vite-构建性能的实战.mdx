## 120秒到45秒：利用 Worker Threads 優化 Vite 建置效能的實戰

> 在處理大型前端專案時，生產環境的程式碼建置往往讓人望眼欲穿。本文分享如何透過 Node.js Worker Threads 將 Vite 建置中的程式碼混淆環節耗時從 120 秒降低至 45 秒，並詳細介紹 HagiCode 專案中的實施細節與踩坑經驗。

## 分析：尋找效能瓶頸的突破口

在着手解決效能問題之前，我們需要先理清思路，確定最優的技術方案。

### 核心決策：爲什麼選擇 Worker Threads？

Node.js 環境下實作平行計算主要有三種方案：
1. **child_process**：建立獨立的子處理序
2. **Web Workers**：主要用於瀏覽器端
3. **worker_threads**：Node.js 原生多執行緒支援

經過對比分析，HagiCode 最終選擇了 **Worker Threads**，原因如下：
- **零序列化開銷**：Worker Threads 位於同一處理序，可以透過 `SharedArrayBuffer` 或轉移控制權的方式共享記憶體，避免了處理序間通信的大額序列化成本。
- **原生支援**：Node.js 12+ 版本內建支援，無需引入額外的重相依。
- **上下文統一**：調試和日誌記錄比子處理序更方便。

### 任務粒度：如何拆分混淆任務？

混淆一個巨大的 JS Bundle 檔案很難平行（因爲程式碼有相依關係），但 Vite 的建置產物是由多個 **Chunk** 組成的。這給了我們一個天然的平行邊界：

- **獨立性**：Vite 打包後的不同 Chunk 之間相依關係已解耦，可以安全地平行處理。
- **粒度適中**：通常專案會有 10-30 個 Chunk，這個數量級非常適合平行調度。
- **易於集成**：Vite 外掛的 `generateBundle` 鉤子允許我們在檔案產生前攔截並處理這些 Chunk。

### 架構設計

我們設計了一個包含四個核心元件的平行處理系統：

1. **Task Splitter**：走訪 Vite 的 bundle 物件，過濾不需要混淆的檔案（如 vendor），產生任務佇列。
2. **Worker Pool Manager**：管理 Worker 的生命週期，負責任務的分發、回收和錯誤重試。
3. **Progress Reporter**：實時輸出建置進度，消除使用者的等待焦慮。
4. **ObfuscationWorker**：實際執行混淆邏輯的工作執行緒。

## 解決：實戰編碼與實施

基於上述分析，我們開始動手實作這套平行混淆系統。

### 1. 設定 Vite 外掛

首先，我們在 `vite.config.ts` 中集成平行混淆外掛。設定非常直觀，只需指定 Worker 數量和混淆規則。

```typescript
import { defineConfig } from 'vite'
import { parallelJavascriptObfuscator } from './buildTools/plugin'

export default defineConfig(({ mode }) => {
  const isProduction = mode === 'production'
  
  return {
    build: {
      rollupOptions: {
        ...(isProduction
          ? {
              plugins: [
                parallelJavascriptObfuscator({
                  enabled: true,
                  // 根据 CPU 核心数自动调整，建议留出一个核心给主线程
                  workerCount: 4, 
                  retryAttempts: 3,
                  fallbackToMainThread: true, // 出错时自动降级为单线程
                  // 过滤掉 vendor chunk，通常不需要混淆第三方库
                  isVendorChunk: (fileName: string) => fileName.includes('vendor-'),
                  obfuscationConfig: {
                    compact: true,
                    controlFlowFlattening: true,
                    deadCodeInjection: true,
                    disableConsoleOutput: true,
                    // ... 更多混淆选项
                  },
                }),
              ],
            }
          : {}),
      },
    },
  }
})
```

### 2. 實作 Worker 邏輯

Worker 是執行任務的單元。我們需要定義好輸入和輸出的資料結構。

**注意**：這裏的程式碼雖然簡單，但有幾個坑點需要注意。比如 `parentPort` 的空值檢查，以及錯誤處理。在 HagiCode 的實踐中，我們發現有些特殊的 ES6 語法可能會導致混淆器崩潰，所以加上了 `try-catch` 保護。

```typescript
import { parentPort } from 'worker_threads'
import javascriptObfuscator from 'javascript-obfuscator'

export interface ObfuscationTask {
  chunkId: string
  code: string
  config: any
}

export interface ObfuscationResult {
  chunkId: string
  obfuscatedCode: string
  error?: string
}

// 监听主线程发来的任务
if (parentPort) {
  parentPort.on('message', async (task: ObfuscationTask) => {
    try {
      // 执行混淆
      const obfuscated = javascriptObfuscator.obfuscate(task.code, task.config)
      const result: ObfuscationResult = {
        chunkId: task.chunkId,
        obfuscatedCode: obfuscated.getObfuscatedCode(),
      }
      // 将结果发回主线程
      parentPort?.postMessage(result)
    } catch (error) {
      // 处理异常，确保单个 Worker 崩溃不会阻塞整个构建
      const result: ObfuscationResult = {
        chunkId: task.chunkId,
        obfuscatedCode: '',
        error: error instanceof Error ? error.message : 'Unknown error',
      }
      parentPort?.postMessage(result)
    }
  })
}
```

### 3. Worker 池管理器

這是整個方案的核心。我們需要維護一個固定大小的 Worker 池，採用 **FIFO（先進先出）** 策略調度任務。

```typescript
import { Worker } from 'worker_threads'
import os from 'os'

export class WorkerPool {
  private workers: Worker[] = []
  private taskQueue: Array<{
    task: ObfuscationTask
    resolve: (result: ObfuscationResult) => void
    reject: (error: Error) => void
  }> = []
  
  constructor(options: WorkerPoolOptions = {}) {
    // 默认为核心数 - 1，给主线程留一点喘息的空间
    const workerCount = options.workerCount ?? Math.max(1, (os.cpus().length || 4) - 1)
    
    for (let i = 0; i < workerCount; i++) {
      this.createWorker()
    }
  }

  private createWorker() {
    const worker = new Worker('./worker.ts')
    
    worker.on('message', (result) => {
      // 任务完成后，从队列中取出下一个任务
      const nextTask = this.taskQueue.shift()
      if (nextTask) {
        this.dispatchTask(worker, nextTask)
      } else {
        // 如果没有待处理任务，标记 Worker 为空闲
        this.activeWorkers.delete(worker)
      }
    })
    
    this.workers.push(worker)
  }

  // 提交任务到池中
  public runTask(task: ObfuscationTask): Promise<ObfuscationResult> {
    return new Promise((resolve, reject) => {
      const job = { task, resolve, reject }
      const idleWorker = this.workers.find(w => !this.activeWorkers.has(w))
      
      if (idleWorker) {
        this.dispatchTask(idleWorker, job)
      } else {
        this.taskQueue.push(job)
      }
    })
  }

  private dispatchTask(worker: Worker, job: any) {
    this.activeWorkers.set(worker, job.task)
    worker.postMessage(job.task)
  }
}
```

### 4. 進度報告

等待是痛苦的，尤其是不知道還要等多久。我們增加了一個簡單的進度報告器，實時反饋當前狀態。

```typescript
export class ProgressReporter {
  private completed = 0
  private readonly total: number
  private readonly startTime: number

  constructor(total: number) {
    this.total = total
    this.startTime = Date.now()
  }

  increment(): void {
    this.completed++
    this.report()
  }

  private report(): void {
    const now = Date.now()
    const elapsed = now - this.startTime
    const percentage = (this.completed / this.total) * 100
    
    // 简单的 ETA 估算
    const avgTimePerChunk = elapsed / this.completed
    const remaining = (this.total - this.completed) * avgTimePerChunk

    console.log(
      `[Parallel Obfuscation] ${this.completed}/${this.total} chunks completed (${percentage.toFixed(1)}%) | ETA: ${(remaining / 1000).toFixed(1)}s`
    )
  }
}
```

## 實踐：效果與踩坑

部署這套方案後，HagiCode 專案的建置效能有了立竿見影的提升。

### 效能基準資料

我們在以下環境進行了測試：
- CPU：Intel Core i7-12700K (12 cores / 20 threads)
- RAM：32GB DDR4
- Node.js：v18.17.0
- OS：Ubuntu 22.04

**結果對比**：
- **單執行緒（優化前）**：118 秒
- **4 Workers**：55 秒（提升 **53%**）
- **8 Workers**：48 秒（提升 **60%**）
- **12 Workers**：45 秒（提升 **62%**）

可以看出，收益並不是線性的。當 Worker 數量超過 8 個後，提升幅度變小。這主要受限於任務分配的均勻度和記憶體頻寬瓶頸。

### 常見問題與解決方案

在 HagiCode 的實際使用中，我們也遇到了一些坑，這裏分享給大家：

**Q1: 建置時間沒有明顯減少，反而變慢了？**
- **原因**：Worker 建立本身有開銷，或者 Worker 數量設定過多導致上下文切換頻繁。
- **解決**：建議 Worker 數量設定爲 `CPU 核心數 - 1`。同時檢查是否有單個 Chunk 特別大（例如 > 5MB），這種"巨無霸"檔案會成爲短板，可以考慮優化程式碼分割策略。

**Q2: 偶爾出現 Worker 崩潰，建置失敗？**
- **原因**：某些特殊的程式碼語法可能導致混淆器內部報錯。
- **解決**：我們實作了 **自動降級機制**。當 Worker 連續失敗次數達到閾值時，外掛會自動回退到單執行緒模式，確保建置不中斷。同時記錄下錯誤的檔案名，方便後續針對性修復。

**Q3: 記憶體佔用過高（OOM）？**
- **原因**：每個 Worker 都需要獨立記憶體空間來載入混淆器和剖析 AST。
- **解決**：
  - 減少 Worker 數量。
  - 增加 Node.js 的記憶體限制：`NODE_OPTIONS="--max-old-space-size=4096" npm run build`。
  - 確保不在 Worker 內部持有不必要的大物件參考。

## 總結

透過引入 Node.js Worker Threads，我們成功將 HagiCode 專案的生產建置時間從 120 秒降低到了 45 秒左右，極大提升了開發體驗和 CI/CD 效率。

這套方案的核心在於：
1. **合理拆分任務**：利用 Vite 的 Chunk 作爲平行單元。
2. **資源控制**：使用 Worker 池避免資源耗盡。
3. **容錯設計**：自動降級機制確保建置穩定性。

如果你也在爲前端建置效率發愁，或者你的專案也在做重度程式碼處理，不妨試試這套方案。當然，更推薦你直接關注我們的 HagiCode 專案，這些工程化的細節都已經集成在裏面了。
