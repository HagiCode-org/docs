## 背景

時間過得真快，大型語言模型的發展就像春天的竹子一樣，蹭蹭地往上竄。曾經我們還在為「一個能寫代碼的 AI」而歡呼雀躍，如今已是多模型協同、多工具融合的時代了。這有意思嗎？或許吧，畢竟開發者需要的從來都不只是工具本身，而是一種能夠適應不同場景、靈活切換的從容。

HagiCode 作為一個 AI 輔助編碼平台，最近也算是迎來了兩件大事：一是智譜 AI 的 GLM-5.1 模型全面接入，二是 Gemini CLI 正式成為第十個支援的 Agent CLI。這兩件事說大不大，說小也不小，只是對於平台的完善而言，總歸是好事一樁。

GLM-5.1 是智譜 AI 的最新旗艦模型，相比 GLM-5.0，推理能力更強了，代碼理解更深了，工具調用也更順滑了。更重要的是，它是首個支援圖片輸入的 GLM 模型——這意味著什麼？意味著用戶可以直接截圖讓 AI 看問題，不用再費勁巴力地描述了。這便利性，用過就懂了。

與此同時，HagiCode 通過 HagiCode.Libs.Providers 架構，把 Gemini CLI 成功整合了進來。這是第十個 Agent CLI 了，說實話，能走到這一步，也算是有些許成就感罷了。

值得一提的是，HagiCode 的圖片上傳功能讓用戶可以直接截圖與 AI 交流。即使運行的是 GLM 4.7 版本，平台依然能夠良好運行，並且已經幫助項目完成了許多重要的構建工作。至於 GLM-5.1？那自然會更進一步。

## 多 CLI 架構設計

HagiCode 的核心優勢之一，就是通過統一的抽象層支援多種不同的 AI 編程 CLI 工具。這種設計的好處，說穿了也就那麼回事：新東西能進來，舊東西能留下，代碼還不亂。畢竟，誰都希望生活能這樣吧？

### AIProviderType 枚舉

平台通過 `AIProviderType` 枚舉定義了支援的 CLI 提供商類型：

```csharp
public enum AIProviderType
{
    ClaudeCodeCli = 0,    // Claude Code CLI
    CodexCli = 1,          // GitHub Copilot Codex
    GitHubCopilot = 2,     // GitHub Copilot
    CodebuddyCli = 3,     // Codebuddy CLI
    OpenCodeCli = 4,      // OpenCode CLI
    IFlowCli = 5,         // IFlow CLI
    HermesCli = 6,        // Hermes CLI
    QoderCli = 7,         // Qoder CLI
    KiroCli = 8,          // Kiro CLI
    KimiCli = 9,          // Kimi CLI
    GeminiCli = 10,       // Gemini CLI (新增)
}
```

可以看到，Gemini CLI 作為第十個成員加入了這個大家庭。每個 CLI 都有獨特的特點和適用場景，用戶可以根據自己的需求靈活選擇。畢竟，條條大路通羅馬，只是有的路好走一點，有的路稍微曲折一點罷了。

### Provider 架構

HagiCode.Libs.Providers 提供了統一的 Provider 接口，讓每個 CLI 的整合變得規範而簡潔。以 Gemini CLI 為例：

```csharp
public class GeminiProvider : ICliProvider<GeminiOptions>
{
    private static readonly string[] DefaultExecutableCandidates = ["gemini", "gemini-cli"];
    private const string ManagedBootstrapArgument = "--acp";

    public string Name => "gemini";
    public bool IsAvailable => _executableResolver.ResolveFirstAvailablePath(DefaultExecutableCandidates) is not null;
}
```

這種設計的好處是：
- 新 CLI 的整合只需要實現一個 Provider 類
- 統一的生命週期管理和會話池化
- 自動化的別名解析和可執行文件查找

說穿了，這種設計其實就是把複雜的事情簡單化，讓生活更輕鬆一點罷了。

### Provider Registry

Provider Registry 自動處理別名映射和註冊：

```csharp
if (provider is GeminiProvider)
{
    registry.Register(provider.Name, provider, ["gemini-cli"]);
    continue;
}
```

這意味著用戶可以使用 `gemini` 或 `gemini-cli` 兩種方式來調用 Gemini CLI，系統會自動識別。這就像你朋友多，有的叫大名，有的叫小名，反正都是他，怎麼叫都行。

## GLM-5.1 模型支援

GLM-5.1 是智譜 AI 的最新旗艦模型，HagiCode 已完成對其的全面支援。

### Secondary Professions Catalog

HagiCode 通過 Secondary Professions Catalog 管理所有支援的模型。以下是 GLM 系列的配置：

| Model ID | Name | SupportsImage | Compatible CLI Families |
|----------|------|----------------|-------------------------|
| `glm-4.7` | GLM 4.7 | - | claude, codebuddy, hermes, qoder, kiro |
| `glm-5` | GLM 5 | - | claude, codebuddy, hermes, qoder, kiro |
| `glm-5-turbo` | GLM 5 Turbo | - | claude, codebuddy, hermes, qoder, kiro |
| `glm-5.0` | GLM 5.0 (Legacy) | - | claude, codebuddy, hermes, qoder, kiro |
| `glm-5.1` | GLM 5.1 | **true** | claude, codebuddy, hermes, qoder, kiro |

GLM-5.1 的關鍵特性可以總結為：
- 獨立的版本標識，沒有 legacy 包袱
- 首個支援圖片輸入的 GLM 模型
- 更強的推理能力和代碼理解
- 廣泛的多 CLI 兼容性

### GLM-5.1 vs GLM-5.0

從代碼層面來看，GLM-5.1 與 GLM-5.0 的關鍵區別：

```csharp
// GLM-5.0 (Legacy) - 有特殊保留邏輯
private const string Glm50CodebuddySecondaryProfessionId = "secondary-glm-5-codebuddy";
private const string Glm50CodebuddyModelValue = "glm-5.0";

// GLM-5.1 - 獨立的新模型標識
private const string Glm51SecondaryProfessionId = "secondary-glm-5-1";
private const string Glm51ModelValue = "glm-5.1";
```

GLM-5.0 帶有 "Legacy" 標記，是為了向後兼容而保留的舊版本標識。而 GLM-5.1 是一個全新的獨立版本，沒有任何歷史包袱。這就像有些人，總是活在过去；而有些人，輕裝上陣，走得更快罷了。

### 配置 GLM-5.1

```json
  "secondaryProfessionId": "secondary-glm-5-1",
  "model": "glm-5.1",

```

## 圖片上傳功能

HagiCode 的圖片支援是通過 SecondaryProfession 的 `SupportsImage` 屬性實現的：

```csharp
public class HeroSecondaryProfessionSettingDto
{
    public bool SupportsImage { get; set; }
}
```

在 Secondary Professions Catalog 中，GLM-5.1 的配置如下：

```json
{
  "id": "secondary-glm-5-1",
  "supportsImage": true
}
```

這意味著用戶可以直接上傳截圖讓 AI 分析，比如：
- 錯誤信息的截圖
- UI 界面的問題
- 數據可視化圖表
- 代碼運行結果

不用再手動描述問題了，直接截圖就行——這個功能的便利性用過就知道了。畢竟，有些事情，說再多不如看一眼。

## Gemini CLI 整合

Gemini CLI 作為第十個 Agent CLI，通過標準的 Provider 架構整合到 HagiCode 中。

### 配置選項

Gemini CLI 支援豐富的配置選項：

```csharp
public class GeminiOptions
{
    public string? ExecutablePath { get; set; }
    public string? WorkingDirectory { get; set; }
    public string? SessionId { get; set; }
    public string? Model { get; set; }
    public string? AuthenticationMethod { get; set; }
    public string? AuthenticationToken { get; set; }
    public Dictionary<string, string?> AuthenticationInfo { get; set; }
    public Dictionary<string, string?> EnvironmentVariables { get; set; }
    public string[] ExtraArguments { get; set; }
    public TimeSpan? StartupTimeout { get; set; }
    public CliPoolSettings? PoolSettings { get; set; }
}
```

這些選項覆蓋了從基本配置到高級特性的方方面面，用戶可以根據自己的需求進行靈活配置。畢竟，每個人的需求都不一樣，能靈活一點總是好的。

### ACP 通信協議

Gemini CLI 支援 ACP (Agent Communication Protocol) 通信協議，這是 HagiCode 統一的 CLI 通信標準。通過 ACP，不同的 CLI 可以以一致的方式與平台交互，大大簡化了整合工作。說穿了，就是把複雜的事情統一化，讓大家都能輕鬆一點罷了。

## 多 CLI 整合最佳實踐

HagiCode.Libs.Providers 提供了統一的註冊和使用機制：

```csharp
services.AddHagiCodeLibs();

var gemini = serviceProvider.GetRequiredService<ICliProvider<GeminiOptions>>();
var codebuddy = serviceProvider.GetRequiredService<ICliProvider<CodebuddyOptions>>();
var hermes = serviceProvider.GetRequiredService<ICliProvider<HermesOptions>>();
```

這種依賴注入的設計讓各個 CLI 的使用變得非常簡潔，也方便進行單元測試和模擬。畢竟，代碼寫得乾淨一點，對自己也是一種負責。

### 注意事項

在實際使用中，有幾個地方需要注意：

1. **API Key 配置**：確保正確設置 `ANTHROPIC_AUTH_TOKEN`，否則無法調用模型
2. **模型可用性**：GLM-5.1 需要在對應的模型提供商處開通權限
3. **圖片功能**：只有支援 `supportsImage: true` 的模型才能使用圖片上傳功能
4. **CLI 安裝**：使用 Gemini CLI 前，確保 `gemini` 或 `gemini-cli` 在系統 PATH 中

這些都是小事，但小事處理不好，也可能變成大事。所以還是要注意一下的。
