## Windows 應用程式自動上架 Microsoft Store 的自動化實踐

> 如何從版本解析到 Store 發布實現全流程自動化，告別手動操作的繁瑣流程。

## 背景

如果你有 Electron 應用程式想要上架 Microsoft Store，大概會碰到這樣的麻煩：Store 不支援分離的安裝流程——你得把桌面應用程式和伺服器負載打包成一個完整的 AppX/MSIX 套件。

這倒也罷了。問題還在後面呢。每次發新版本，你得：

1. 檢查桌面和伺服器版本是否更新
2. 從對應的 tag 檢出程式碼
3. 下載並注入伺服器負載
4. 建置 MSIX 套件
5. 手動上傳到 Microsoft Store
6. 組態商店資訊和定價

如果每一步都要手動操作，那也太折騰人了。而且容易出錯，哪一步做了哪一步沒做，自己都未必記得清楚。

其實這也不能怪誰，畢竟手動操作本就容易遺漏。只是我們實在不想每次都這樣折騰，於是決定徹底解決這個問題——讓整個流程自動跑起來。

## 關於 HagiCode

本文分享的方案來自我們在 [HagiCode](https://hagicode.com) 專案中的實踐經驗。作為一個 AI 程式碼助手，HagiCode 提供桌面端和 Web 端，需要支援多種分發渠道。在實作 Microsoft Store 自動上架的過程中，我們總結出了一套完整的自動化方案。

說起來，這大概也算個意外收穫。本來只是為了省點時間，沒想到最後做出來這麼一套東西。

## 技術架構分析

這個問題其實涉及多個技術層次的協調。我們可以把它分成五層：

### 版本協調層

首先需要知道什麼時候需要發布新版本。我們需要從 Azure 索引（我們用來儲存建置產物的 blob 儲存體）中解析出桌面和伺服器元件的最新版本，然後判斷是否需要產生新的 Store 套件。

這就像是在問自己：現在該做這件事了嗎？

### 工作空間管理層

AppX 的建置依賴於原始碼層級的組態和運行時佈局，所以不能簡單地用現成的建置產物重新打包。我們需要從桌面倉庫的特定 tag 檢出程式碼，確保建置使用的原始碼狀態是正確的。

畢竟，原始碼不對，後面再怎麼努力也是白搭。

### 運行時打包層

這是核心部分。我們需要把伺服器負載注入到桌面應用程式的打包佈局中。這樣當應用程式啟動時，會偵測到打包的運行時並進入 Steam 模式（離線模式）。

這一步做不好，前面所有的努力也都白費了。

### 建置輸出層

使用 electron-builder 產生符合 Store 要求的 MSIX 套件。這一步需要在 Windows 環境中執行，因為 AppX 建置需要 Windows SDK。

有些事情就是要在對的地方做，換地方就不行。

### 發布分發層

最後是發布到 GitHub Releases 和 Microsoft Store。GitHub Release 作為備份和版本追蹤，Microsoft Store 則面向終端使用者。

一切準備就緒，就差最後這一哆嗦了。

其實四小時一次也不算頻繁，畢竟程式碼更新總是有快有慢，只是這樣比較保險罷了。

## 實作詳解

### 1. 建置計畫解析

第一步是確定是否需要建置新版本。我們需要解析桌面和伺服器元件的目前版本，然後檢查是否已經存在對應的發布。

```javascript
// scripts/resolve-dispatch-build-plan.mjs
export async function resolveDispatchBuildPlan({
  eventName,
  eventPayload,
  desktopAzureSasUrl,
  serverAzureSasUrl,
}) {
  // 解析觸發器輸入
  const trigger = normalizeTriggerInputs({ eventName, eventPayload });
  
  // 從 Azure 索引解析桌面和伺服器版本
  const [desktopRelease, serverRelease] = await Promise.all([
    resolveIndexRelease({ 
      azureSasUrl: desktopAzureSasUrl,
      platformId: 'win-x64' 
    }),
    resolveIndexRelease({ 
      azureSasUrl: serverAzureSasUrl,
      platformId: 'win-x64' 
    })
  ]);
  
  // 產生發布 tag
  const desktopTag = normalizeGitTag(desktopRelease.version);
  const releaseTag = deriveStoreReleaseTag(desktopRelease.version, serverRelease.version);
  // 比如：desktop-v1.2.3-server-v4.5.6
  
  // 檢查是否已存在（避免重複建置）
  const existingRelease = await findReleaseByTag(packerRepository, releaseTag);
  const shouldBuild = !existingRelease || trigger.forceRebuild;
  
  return {
    release: { tag: releaseTag, exists: Boolean(existingRelease) },
    build: { shouldBuild, skipReason },
    upstream: { desktop: { tag: desktopTag }, server }
  };
}
```

這個步驟的關鍵在於跳過邏輯——如果相同版本的組合已經建置過，就沒有必要再跑一遍完整的建置流程。這樣可以節省 CI 成本和時間。

重複做同樣的事情，大概也沒什麼意義。畢竟，時間和資源都是有限的。

### 2. 工作空間準備

確定要建置後，需要準備一個乾淨的建置環境。我們使用 git worktree 來從特定的 tag 檢出程式碼。

```javascript
// scripts/prepare-packaging-workspace.mjs
export async function preparePackagingWorkspace({
  planPath,
  platformId,
  workspacePath,
  desktopSourcePath
}) {
  // 使用 git worktree 從特定 tag 檢出桌面程式碼
  await runCommand('git', [
    '-C', resolvedDesktopSourcePath,
    'worktree', 'add', '--detach',
    desktopWorkspace,
    `refs/tags/${plan.upstream.desktop.tag}`
  ]);
  
  // 驗證工作空間
  const validation = await validateDesktopWorkspace({
    desktopWorkspace,
    storePackageConfig
  });
  
  // 建立工作空間清單
  const workspaceManifest = {
    desktopWorkspace,
    runtimeInjectionRoot: validation.runtimeRoot,
    desktopTag: plan.upstream.desktop.tag,
    desktopRef,
  };
  
  return workspaceManifest;
}
```

使用 worktree 的好處是不影響主工作目錄，建置可以並行進行，而且建置完成後可以輕鬆清理。

畢竟，誰也不希望因為建置把主工作目錄搞得亂七八糟。

### 3. 伺服器負載注入

這一步是把伺服器負載下載並注入到正確的位置。

```javascript
// scripts/stage-server-payload.mjs
export async function stageServerPayload({
  planPath,
  workspacePath,
  platformId,
  azureSasUrl
}) {
  // 從 Azure 索引下載伺服器負載
  const assetSource = resolveAssetDownloadUrl({ asset, sasUrl: azureSasUrl });
  await downloadFromSource({ 
    sourceUrl: assetSource, 
    destinationPath: downloadPath 
  });
  
  // 解壓縮並驗證
  await extractArchive(downloadPath, extractionPath);
  const runtimeRoot = await resolveRuntimeRoot(extractionPath);
  const validation = await validateServerPayloadRoot(runtimeRoot, platformId);
  
  // 注入到打包運行時佈局
  // 目標路徑是 resources/portable-fixed/current
  // 這個路徑會被映射到 AppX 內的 extra/portable-fixed/current
  await copyDir(runtimeRoot, targetPath);
}
```

關鍵是路徑映射——`resources/portable-fixed/current` 會被打包到 AppX 的 `extra/portable-fixed/current`，這樣應用程式啟動時就會偵測到本地運行時並進入離線模式。

路徑這東西，錯一点都不行。一點偏差，可能就找不到了。

### 4. MSIX 建置

有了準備好的工作空間和伺服器負載，就可以建置 MSIX 套件了。

```javascript
// scripts/build-appx.mjs
export async function buildAppx({
  planPath,
  workspacePath,
  platformId
}) {
  // 產生 Store 特定的 electron-builder 組態覆蓋
  const overlayConfig = await writeStoreElectronBuilderConfig({
    desktopWorkspace: workspaceManifest.desktopWorkspace,
    sourceConfigPath: storePackageConfig.desktop.electronBuilderConfigPath,
    outputConfigPath: 'electron-builder.store.yml'
  });
  
  // 執行桌面建置命令
  await runShellCommand(
    buildDesktopStoreCommand(overlayConfig.outputPath, desktopScripts),
    workspaceManifest.desktopWorkspace
  );
  
  // 收集 MSIX 輸出
  const storeOutputs = await findStoreOutputs(pkgDirectory);
  const artifactPath = path.join(
    workspaceManifest.outputDirectory, 
    artifactFileName
  );
  await copySingleFile(primaryOutput, artifactPath);
}
```

electron-builder 組態需要包含 Store 特定的元資料，比如套件身分、發布者顯示名稱等。這些資訊會在 Store 提交時使用。

組態不對，套件也打出來。這也沒什麼好說的。

### 5. 發布到 GitHub

建置完成後，需要把產物發布到 GitHub Releases。

```javascript
// scripts/publish-release.mjs
export async function publishRelease({
  planPath,
  artifactsDir,
  outputDir,
  forceDryRun
}) {
  // 建立發布產物清單
  const publicationArtifacts = await buildPublicationArtifacts({
    plan,
    artifactsDir,
    outputDir
  });
  
  if (!dryRun) {
    // 建立或更新 GitHub Release
    const releaseResult = await upsertReleaseNotes(
      plan.release.repository,
      plan.release.tag,
      token,
      { name, body }
    );
    
    // 上傳資產
    for (const upload of publicationArtifacts.uploads) {
      await uploadReleaseAsset({
        release: releaseResult.release,
        filePath: upload.filePath,
        fileName: upload.fileName
      });
    }
  }
}
```

GitHub Release 作為版本追蹤和備份，即使 Store 發布失敗，也能保留建置產物。

留一條後路總是好的。萬一哪天需要回溯呢？

### 6. 發布到 Microsoft Store

最後一步是使用 Microsoft Store CLI 發布到商店。

```yaml
# .github/workflows/package-release.yml
publish_store:
  runs-on: windows-latest
  steps:
    - name: Configure Microsoft Store CLI
      uses: microsoft/microsoft-store-apppublisher@v1.2
    
    - name: Publish MSIX packages to Store
      shell: pwsh
      run: |
        msstore reconfigure --tenantId $env:AZURE_AD_TENANT_ID ...
        msstore publish "$($package.FullName)" -id $env:MICROSOFT_STORE_PRODUCT_ID
```

這一步需要在 Windows runner 上執行，因為 Microsoft Store CLI 需要 Windows 環境。

有些事情就是要在對的地方做，換了環境就不行。

## 實踐指南

### 組態檔案

Store 套件組態檔案定義了套件身分和建置設定：

```json
{
  "packageIdentity": {
    "displayName": "Hagicode",
    "publisherDisplayName": "newbe36524",
    "publisher": "CN=8B6C8A94-AAE5-4C8B-9202-A29EA42B042F",
    "identityName": "newbe36524.Hagicode",
    "backgroundColor": "transparent",
    "languages": ["en-US", "zh-CN", "zh-Hant"]
  },
  "supportedWindowsTargets": ["win-x64"],
  "desktop": {
    "submodulePath": "inputs/hagicode-desktop",
    "electronBuilderConfigPath": "electron-builder.yml",
    "buildScript": "build:appx",
    "runtimeInjectionPath": "resources/portable-fixed/current"
  }
}
```

組態這東西，改一次就夠頭疼的了，所以最好一次搞對。

### 工作流程觸發

可以定時觸發或手動觸發：

```yaml
on:
  schedule:
    - cron: '0 */4 * * *'  # 每 4 小時執行一次
  workflow_dispatch:
    inputs:
      desktop_version:
        description: 'Desktop 版本選擇器'
        required: false
      server_version:
        description: 'Server 版本選擇器'
        required: false
      force_rebuild:
        description: '即使已存在也強制重建'
        type: boolean
      dry_run:
        description: '僅建置，不發布'
        type: boolean
```

自動也好，手動也罷，反正最後能跑起來就行。

### 關鍵注意事項

1. **Git Tag 要求**：桌面倉庫必須包含與發布版本對應的 tag（如 v1.2.3），否則建置會失敗

2. **Azure SAS URL**：需要組態 `DESKTOP_AZURE_SAS_URL` 和 `SERVER_AZURE_SAS_URL` 環境變數以存取索引

3. **Microsoft Store 憑證**：需要組態以下金鑰：
   - `AZURE_AD_APPLICATION_CLIENT_ID`
   - `AZURE_AD_APPLICATION_SECRET`
   - `AZURE_AD_TENANT_ID`
   - `SELLER_ID`
   - `MICROSOFT_STORE_PRODUCT_ID`

4. **運行時驗證**：伺服器負載必須包含必需的運行時檔案，否則打包會失敗

5. **Windows Runner**：AppX 建置和 Store 發布需要在 Windows runner 上執行

6. **重複偵測**：計畫執行會檢查是否存在相同的 release tag，避免不必要的建置

這些坑我們大概都踩過一遍了，所以寫出來提醒一下。畢竟，誰願意重複犯錯呢？

## 總結

Microsoft Store 自動上架看起來複雜，但拆解成獨立步驟後，每一步都不難。關鍵是：
- 用 git worktree 管理建置環境
- 把伺服器負載注入到正確的路徑
- 使用 electron-builder 產生 MSIX 套件
- 用 Store CLI 完成最終發布

這套方案在 HagiCode 專案中已經穩定執行了几個月，基本實現了「完全自動化」的目標——除了第一次組態憑證和設定，後續的版本發布都不需要人工介入。

其實很多事情都是這樣，看起來難，做起來也就那麼回事。只是需要一點耐心和嘗試罷了。

如果你也在做類似的自動化工作，希望這篇文章能給你一些參考。當然，每個專案的情況不一樣，可能需要根據具體需求調整方案。

畢竟，條條大路通羅馬，只是有的路好走一點，有的路稍微曲折一點而已......
