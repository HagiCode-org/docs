## 實現 Docker Hub 到 Azure ACR 的自動化映像同步

> 本文介紹如何使用 GitHub Actions 和 image-syncer 工具，實現 Docker Hub 映像到 Azure Container Registry 的自動化同步，解決了國內及部分 Azure 區域訪問 Docker Hub 速度慢的問題，提升了映像的可用性和 Azure 環境的部署效率。

## 背景/引言

HagiCode 專案使用 Docker 映像作為核心運行時元件，主要映像託管在 Docker Hub。隨著專案發展和 Azure 環境部署需求的增加，我們遇到了以下痛點：
- 映像拉取速度慢，Docker Hub 在國內及部分 Azure 區域訪問受限
- 依賴單一映像源存在單點故障風險
- Azure 環境下使用 Azure Container Registry 能獲得更好的網路效能和整合體驗

為解決這些問題，我們需要建立一個自動化的映像同步機制，將 Docker Hub 的映像定期同步到 Azure ACR，確保使用者能夠在 Azure 環境中獲得更快的映像拉取速度和更高的可用性。

## 技術方案對比

在制定解決方案時，我們對比了多種技術方案：

### 1. image-syncer（最終選擇）
- 增量同步：僅同步變更的映像層，顯著減少網路傳輸
- 斷點續傳：網路中斷後可恢復同步
- 並發控制：支援配置並發執行緒數，提升大映像同步效率
- 完善的錯誤處理：內建失敗重試機制（預設 3 次）
- 輕量級部署：單一二進位檔案，無依賴
- 多倉庫支援：相容 Docker Hub、Azure ACR、Harbor 等

### 2. Docker CLI
- 不支援增量同步：每次都需要拉取完整的映像內容
- 效率較低：網路傳輸量大，時間長
- 簡單易用：使用熟悉的 docker pull/push 指令

### 3. Azure CLI
- 複雜度高：需要配置 Azure CLI 認證
- 功能限制：az acr import 功能相對單一
- 原生整合：與 Azure 服務整合良好

## 架構設計決策

### 決策 1：同步頻率設定為每日 UTC 00:00
- 平衡映像新鮮度和資源消耗
- 避開業務高峰期，減少對其他操作的影響
- Docker Hub 映像通常在每日建置後更新

### 決策 2：同步所有映像標籤
- 保持與 Docker Hub 的完全一致性
- 為使用者提供靈活的版本選擇
- 簡化同步邏輯，避免複雜的標籤過濾規則

### 決策 3：使用 GitHub Secrets 儲存認證資訊
- GitHub Actions 原生支援，安全性高
- 配置簡單，易於管理和維護
- 支援倉庫層級的存取控制

## 風險評估與緩解

### 風險 1：Azure ACR 認證資訊洩露
- 使用 GitHub Secrets 加密儲存
- 定期輪換 ACR 密碼
- 限制 ACR 使用者權限為僅推送
- 監控 ACR 存取日誌

### 風險 2：同步失敗導致映像不一致
- image-syncer 內建增量同步機制
- 自動失敗重試（預設 3 次）
- 詳細的錯誤日誌和失敗通知
- 斷點續傳功能

### 風險 3：資源消耗過大
- 增量同步減少網路傳輸
- 可配置並發執行緒數（目前設定為 10）
- 監控同步的映像數量和大小
- 在非高峰時段執行同步

## 核心解決方案

我們採用 GitHub Actions + image-syncer 的自動化方案，實現從 Docker Hub 到 Azure ACR 的映像同步。

## 實施步驟

### 1. 準備階段
- 在 Azure Portal 中建立或確認 Azure Container Registry
- 建立 ACR 存取金鑰（使用者名稱和密碼）
- 確認 Docker Hub 映像倉庫存取權限

### 2. 配置 GitHub Secrets
在 GitHub 倉庫設定中新增以下 Secrets：
- AZURE_ACR_USERNAME: Azure ACR 使用者名稱
- AZURE_ACR_PASSWORD: Azure ACR 密碼

### 3. 建立 GitHub Actions 工作流程
在 .github/workflows/sync-docker-acr.yml 中配置工作流程：
- 定時觸發：每天 UTC 00:00
- 手動觸發：支援 workflow_dispatch
- 額外觸發：publish 分支推送時觸發（用於快速同步）

### 4. 工作流程執行流程

| 順序 | 參與方 | 動作 | 說明 |
| --- | --- | --- | --- |
| 1 | GitHub Actions | 觸發工作流程 | 由定時、手動或 `publish` 分支推送觸發 |
| 2 | GitHub Actions → image-syncer | 下載並執行同步工具 | 進入實際同步階段 |
| 3 | image-syncer → Docker Hub | 取得映像 manifest 與標籤列表 | 讀取來源倉庫元資料 |
| 4 | image-syncer → Azure ACR | 取得目標倉庫已存在映像資訊 | 判斷目標側目前狀態 |
| 5 | image-syncer | 對比來源與目標差異 | 識別需要同步的映像層 |
| 6 | image-syncer → Docker Hub | 拉取變更的映像層 | 僅傳輸需要更新的內容 |
| 7 | image-syncer → Azure ACR | 推送變更的映像層 | 完成增量同步 |
| 8 | image-syncer → GitHub Actions | 傳回同步統計資訊 | 包含結果、差異和錯誤資訊 |
| 9 | GitHub Actions | 記錄日誌並上傳 artifact | 便於後續稽核與排查 |

## GitHub Actions 工作流程實作

以下是實際執行的工作流程配置（.github/workflows/sync-docker-acr.yml）：

```yaml
name: Sync Docker Image to Azure ACR

on:
  schedule:
    - cron: "0 0 * * *" # 每天 UTC 00:00
  workflow_dispatch: # 手動觸發
  push:
    branches: [publish]

permissions:
  contents: read

jobs:
  sync:
    runs-on: ubuntu-latest

    steps:
      - name: Checkout code
        uses: actions/checkout@v4

      - name: Download image-syncer
        run: |
          # 下載 image-syncer 二進位檔案
          wget https://github.com/AliyunContainerService/image-syncer/releases/download/v1.5.5/image-syncer-v1.5.5-linux-amd64.tar.gz
          tar -zxvf image-syncer-v1.5.5-linux-amd64.tar.gz
          chmod +x image-syncer

      - name: Create auth config
        run: |
          # 產生認證配置檔案 (YAML 格式)
          cat > auth.yaml <<EOF
          hagicode.azurecr.io:
            username: "${{ secrets.AZURE_ACR_USERNAME }}"
            password: "${{ secrets.AZURE_ACR_PASSWORD }}"
          EOF

      - name: Create images config
        run: |
          # 產生映像同步配置檔案 (YAML 格式)
          cat > images.yaml <<EOF
          docker.io/newbe36524/hagicode: hagicode.azurecr.io/hagicode
          EOF

      - name: Run image-syncer
        run: |
          # 執行同步 (使用新版 --auth 和 --images 參數)
          ./image-syncer --auth=./auth.yaml --images=./images.yaml --proc=10 --retries=3

      - name: Upload logs
        if: always()
        uses: actions/upload-artifact@v4
        with:
          name: sync-logs
          path: image-syncer-*.log
          retention-days: 7
```

## 配置說明

### 1. 觸發條件
- 定時觸發：cron: "0 0 * * *" - 每天 UTC 00:00 執行
- 手動觸發：workflow_dispatch - 允許使用者在 GitHub UI 手動執行
- 推送觸發：push: branches: [publish] - 發布分支推送時觸發（用於快速同步）

### 2. 認證配置 (auth.yaml)
```yaml
hagicode.azurecr.io:
  username: "${{ secrets.AZURE_ACR_USERNAME }}"
  password: "${{ secrets.AZURE_ACR_PASSWORD }}"
```

### 3. 映像同步配置
```yaml
docker.io/newbe36524/hagicode: hagicode.azurecr.io/hagicode
```
此配置表示將 docker.io/newbe36524/hagicode 的所有標籤同步到 hagicode.azurecr.io/hagicode

### 4. image-syncer 參數
- --auth=./auth.yaml: 認證配置檔案路徑
- --images=./images.yaml: 映像同步配置檔案路徑
- --proc=10: 並發執行緒數為 10
- --retries=3: 失敗重試 3 次

## 注意事項和最佳實踐

### 1. 安全建議
- 定期輪換 Azure ACR 密碼（建議每 90 天）
- 使用專用的 ACR 服務帳戶，限制權限為僅推送
- 監控 ACR 的存取日誌，及時發現異常存取
- 不要在日誌中輸出認證資訊
- 不要將認證資訊提交到程式碼倉庫

### 2. 效能最佳化
- 調整 --proc 參數：根據網路頻寬調整並發數（建議 5-20）
- 監控同步時間：如果同步時間過長，考慮減少並發數
- 定期清理日誌：設定合理的 retention-days（目前為 7 天）

### 3. 故障排查

#### 問題 1：認證失敗
```
Error: failed to authenticate to hagicode.azurecr.io
```
解決方案：
1. 檢查 GitHub Secrets 是否正確配置
2. 驗證 Azure ACR 密碼是否過期
3. 確認 ACR 服務帳戶權限是否正確

#### 問題 2：網路逾時
```
Error: timeout waiting for response
```
解決方案：
1. 檢查網路連線
2. 減少並發執行緒數（--proc 參數）
3. 等待網路恢復後重新觸發工作流程

#### 問題 3：映像同步不完整
```
Warning: some tags failed to sync
```
解決方案：
1. 檢查同步日誌，識別失敗的標籤
2. 手動觸發工作流程重新同步
3. 驗證 Docker Hub 來源映像是否正常

### 4. 監控和告警
- 定期檢查 Actions 頁面，確認工作流程執行狀態
- 設定 GitHub 通知，及時取得工作流程失敗通知
- 監控 Azure ACR 的儲存使用情況
- 定期驗證映像標籤一致性

## 常見問題和解決方案

### Q1: 如何同步特定標籤而不是所有標籤？

修改 images.yaml 配置檔案：
```yaml
# 僅同步 latest 和 v1.0 標籤
docker.io/newbe36524/hagicode:latest: hagicode.azurecr.io/hagicode:latest
docker.io/newbe36524/hagicode:v1.0: hagicode.azurecr.io/hagicode:v1.0
```

### Q2: 如何同步多個映像倉庫？

在 images.yaml 中新增多行配置：
```yaml
docker.io/newbe36524/hagicode: hagicode.azurecr.io/hagicode
docker.io/newbe36524/another-image: hagicode.azurecr.io/another-image
```

### Q3: 同步失敗後如何重試？
- 自動重試：image-syncer 內建重試機制（預設 3 次）
- 手動重試：在 GitHub Actions 頁面點擊 Re-run all jobs

### Q4: 如何查看同步的詳細進度？
- 在 Actions 頁面查看即時日誌
- 下載 sync-logs artifact 查看完整日誌檔案
- 日誌檔案包含每個標籤的同步狀態和傳輸速度

### Q5: 同步需要多長時間？
- 首次全量同步：根據映像大小，通常需要 10-30 分鐘
- 增量同步：如果映像變更小，通常 2-5 分鐘
- 時間取決於網路頻寬、映像大小和並發設定
