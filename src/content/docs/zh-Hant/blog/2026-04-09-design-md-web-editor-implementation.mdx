## 在 Web 介面直接編輯 DESIGN.md：從思路到實現

> 在 MonoSpecs 專案管理系統中，DESIGN.md 承載著專案的架構設計和技術決策。但傳統的編輯方式要求使用者必須切換到外部編輯器，這種割裂的流程，怎麼說呢，就像在讀一首詩的時候突然被打斷了——靈感沒了，心情也沒了。本文分享了我們在 HagiCode 專案中實踐的解決方案：在 Web 介面直接編輯 DESIGN.md，並支援從線上設計站點匯入範本。畢竟，誰不喜歡一氣呵成的感覺呢？

## 背景

`DESIGN.md` 作為專案設計文件的核心載體，承載著架構設計、技術決策和實作指導等關鍵資訊。然而，傳統的編輯方式要求使用者必須切換到外部編輯器（如 VS Code），手動定位實體路徑後再進行編輯。這過程說起來也不算複雜，只是反覆幾次之後，人也就乏了。

具體問題體現在以下幾個方面：

- **流程割裂**：使用者需要在 Web 管理介面和本地編輯器之間頻繁切換，破壞了工作流連貫性——就像聽歌的時候突然斷網了，節奏全亂了。
- **復用困難**：設計站點已經發布了豐富的設計範本庫，但無法直接整合到專案編輯流程中。明明有好東西，就是用不上，這感覺確實有點遺憾。
- **體驗缺失**：缺少"預覽-選擇-匯入"的閉環，使用者必須手動複製貼上，增加了出錯風險。手動操作的次數多了，出錯的機會自然也多了。
- **協作障礙**：設計文件與程式碼實作的同步維護變得高摩擦，阻礙團隊協作效率。團隊協作本就不易，何必再添這些阻力呢？

為了解決這些痛點，我們決定在 Web 介面中實作 DESIGN.md 的直接編輯能力，並支援從線上設計站點一鍵匯入範本。這也不算是什麼驚天動地的決策，只是想讓開發體驗更順暢一點罷了。

## 關於 HagiCode

本文分享的方案來自我們在 [HagiCode](https://hagicode.com) 專案中的實踐經驗。HagiCode 是一個 AI 驅動的程式碼助手專案，在開發過程中，我們需要頻繁維護專案的設計文件。為了讓團隊能夠更高效地協作，我們探索並實作了這套線上編輯和匯入方案。其實也沒什麼特別的，只是遇到了問題，想辦法解決而已。

### 關鍵技術決策

#### 決策 1：全域抽屜模式

採用單一全域抽屜而非局部彈層，透過 `layoutSlice` 管理狀態，實現了跨檢視（classic/kanban）的一致體驗。這種方式確保使用者無論在哪個檢視中開啟編輯器，都能獲得統一的互動體驗。畢竟，一致的體驗能讓使用者感覺更自在，不會因為換個檢視就迷失方向。

#### 決策 2：專案作用域 API

將 DESIGN.md 相關介面掛在 `ProjectController` 下，復用現有專案權限邊界，避免了新增獨立控制器的複雜度。這樣設計的好處是權限管理更清晰，也符合 RESTful 的資源組織原則。有時候，復用比重新建立更有意義，不是嗎？

#### 決策 3：版本衝突檢測

基於檔案系統 `LastWriteTimeUtc` 派生 opaque version，實現了輕量級的樂觀並發控制。當多個使用者同時編輯同一檔案時，系統能夠檢測到衝突並提示使用者重新整理。這種設計既不阻塞使用者的編輯操作，又能保證資料的一致性——就像人際交往中的邊界感，既不過分疏離，也不越界。

#### 決策 4：同源代理模式

透過 `IHttpClientFactory` 代理外部設計站點資源，避免了跨域問題和 SSRF 風險。這種設計既保證了安全性，又簡化了前端呼叫。安全這件事，做再多也不為過，畢竟資料安全就像健康，失去了才後悔就晚了。

## 核心實作

### 1. 直接編輯 DESIGN.md

#### 後端實作

後端主要負責路徑解析、檔案讀寫和版本管理。這些工作雖然基礎，但必不可少，就像房子的地基一樣：

```csharp
// 路徑解析與安全校驗
private Task<string> ResolveDesignDocumentDirectoryAsync(string projectPath, string? repositoryPath)
{
    if (string.IsNullOrWhiteSpace(repositoryPath))
    {
        return Task.FromResult(Path.GetFullPath(projectPath));
    }
    return ValidateSubPathAsync(projectPath, repositoryPath);
}

// 版本號生成（基於檔案系統時間戳記和大小）
private static string BuildDesignDocumentVersion(string path)
{
    var fileInfo = new FileInfo(path);
    fileInfo.Refresh();
    return string.Create(
        CultureInfo.InvariantCulture,
        $"{fileInfo.LastWriteTimeUtc.Ticks:x}-{fileInfo.Length:x}");
}
```

版本號的設計其實也挺有意思的，我們用檔案的最後修改時間和大小來生成一個唯一的版本識別。這樣既輕量又可靠，不需要維護額外的版本資料庫。簡單的東西，往往更有效，不是嗎？

#### 前端實作

前端實現了髒狀態檢測和儲存邏輯。這種設計讓使用者隨時知道自己的修改是否已儲存，減少"萬一丟失了怎麼辦"的焦慮：

```typescript
// 髒狀態檢測與儲存邏輯
const [draft, setDraft] = useState('');
const [savedDraft, setSavedDraft] = useState('');
const isDirty = draft !== savedDraft;

const handleSave = useCallback(async () => {
    const result = await saveProjectDesignMdDocument({
        ...activeTarget,
        content: draft,
        expectedVersion: document.version, // 樂觀並發控制
    });
    setSavedDraft(draft); // 更新已儲存狀態
}, [activeTarget, document, draft]);
```

這個實作中，我們維護了兩個狀態：`draft` 是當前編輯的內容，`savedDraft` 是已儲存的內容。透過比較兩者來判斷是否有未儲存的修改。這種設計雖然簡單，但能給人安心感，畢竟誰願意辛辛苦苦寫的東西突然消失呢？

#### 索引資料格式

設計站點的索引檔案定義了所有可用的範本。有了這個索引，使用者就能像在餐廳點菜一樣，輕鬆選擇自己想要的範本：

```json

      "slug": "linear.app",

```

每個條目包含了範本的基本資訊和下載連結。後端會從這個索引中讀取可用的範本列表，然後展示給使用者選擇。這種設計讓選擇變得直觀，而不是在黑暗中摸索。

#### 同源代理實作

為了保證安全性，後端對設計站點的存取做了嚴格的校驗。安全這件事，再怎麼小心也不為過：

```csharp
// 安全的 slug 校驗
private static readonly Regex SafeDesignSiteSlugRegex =
    new("^[A-Za-z0-9](?:[A-Za-z0-9._-]{0,127})$", RegexOptions.Compiled);

private static string NormalizeDesignSiteSlug(string slug)
{
    var normalizedSlug = slug?.Trim() ?? string.Empty;
    if (!IsSafeDesignSiteSlug(normalizedSlug))
    {
        throw new BusinessException(
            ProjectDesignSiteIndexErrorCodes.InvalidSlug,
            "Design site slug must be a single safe path segment.");
    }
    return normalizedSlug;
}

// 預覽圖快取（OS 暫存目錄）
private static string ComputePreviewCacheKey(string slug, string theme, string previewUrl)
{
    var raw = $"{slug}|{theme}|{previewUrl}";
    var bytes = SHA256.HashData(Encoding.UTF8.GetBytes(raw));
    return Convert.ToHexString(bytes).ToLowerInvariant();
}
```

這裡我們做了兩件事：一是用正則表達式嚴格校驗 slug 的格式，防止路徑遍歷攻擊；二是對預覽圖進行快取，減少對外部站點的請求壓力。前者是防護，後者是優化，缺一不可罷了。

### 3. 完整的匯入流程

```typescript
// 1. 開啟匯入抽屜
const handleRequestImportDrawer = useCallback(() => {
    setIsImportDrawerOpen(true);
}, []);

// 2. 選擇並匯入
const handleImportRequest = useCallback((entry) => {
    if (isDirty) {
        setPendingImportEntry(entry);
        setConfirmMode('import'); // 覆蓋確認
        return;
    }
    void executeImport(entry);
}, [isDirty]);

// 3. 執行匯入
const executeImport = useCallback(async (entry) => {
    const result = await getProjectDesignMdSiteImportDocument(
        activeTarget.projectId,
        entry.slug
    );
    setDraft(result.content); // 只替換編輯器文字，不自動儲存
    setIsImportDrawerOpen(false);
}, [activeTarget?.projectId]);
```

匯入流程的設計遵循了"使用者確認"的原則：匯入後只更新編輯器內容，不會自動儲存。使用者可以檢查匯入的內容，確認無誤後再手動儲存。畢竟，使用者對自己寫的東西應該有最終決定權，不是嗎？

## 實踐範例

### 場景 1：專案根目錄 DESIGN.md 建立

當 DESIGN.md 不存在時，後端傳回虛擬文件狀態。這種設計讓前端不需要特殊處理"檔案不存在"的情況，統一的 API 介面簡化了程式碼邏輯：

```csharp
return new ProjectDesignDocumentDto
{
    Path = targetPath,
    Exists = false,  // 虛擬文件狀態
    Content = string.Empty,
    Version = null
};

// 首次儲存時自動建立檔案
public async Task<SaveProjectDesignDocumentResultDto> SaveDesignDocumentAsync(...)
{
    Directory.CreateDirectory(targetDirectory);
    await File.WriteAllTextAsync(targetPath, input.Content);
    return new SaveProjectDesignDocumentResultDto { Created = !exists };
}
```

這種設計的好處是前端不需要特殊處理"檔案不存在"的情況，統一的 API 介面簡化了程式碼邏輯。有時候，把複雜性隱藏在後端，前端就能更輕鬆地專注於使用者體驗。

### 場景 2：從設計站點匯入範本

使用者在匯入抽屜中選擇 "Linear" 設計範本後，系統會透過後端代理取得 DESIGN.md 內容。整個流程對使用者來說是透明的，他們只需要選擇範本，系統會自動處理所有的網路請求和資料轉換：

```typescript
// 1. 系統透過後端代理取得 DESIGN.md 內容
GET /api/project/{id}/design-md/site-index/linear.app

// 2. 後端驗證 slug 並從上游取得內容
var entry = FindDesignSiteEntry(catalog, "linear.app");
using var upstreamResponse = await httpClient.SendAsync(request);
var content = await upstreamResponse.Content.ReadAsStringAsync();

// 3. 前端替換編輯器文字
setDraft(result.content);
// 使用者檢查後手動儲存到磁碟
```

整個流程對使用者來說是透明的，他們只需要選擇範本，系統會自動處理所有的網路請求和資料轉換。使用者不需要關心背後的複雜性，這就是我們追求的體驗——簡單，但強大。

### 場景 3：版本衝突處理

當多個使用者同時編輯同一 DESIGN.md 時，系統會檢測到版本衝突。這種樂觀並發控制機制確保了資料的一致性，同時又不會阻塞使用者的編輯操作：

```csharp
if (!string.Equals(currentVersion, expectedVersion, StringComparison.Ordinal))
{
    throw new BusinessException(
        ProjectDesignDocumentErrorCodes.VersionConflict,
        $"DESIGN.md at '{targetPath}' changed on disk.");
}
```

前端會捕獲這個錯誤並提示使用者：

```typescript
// 前端提示使用者重新整理並重試
<Alert>
    <AlertTitle>版本衝突</AlertTitle>
    <AlertDescription>
        檔案已被其他程序修改。請重新整理最新版本後重試。
    </AlertDescription>
</Alert>
```

這種樂觀並發控制機制確保了資料的一致性，同時又不會阻塞使用者的編輯操作。衝突不可避免，但至少可以讓使用者知道發生了什麼，而不是默默丟失修改。

## 注意事項與最佳實踐

### 1. 路徑安全

始終校驗 `repositoryPath`，防止路徑遍歷攻擊。安全這種事，做再多也不為過：

```csharp
// 始終校驗 repositoryPath，防止路徑遍歷攻擊
return ValidateSubPathAsync(projectPath, repositoryPath);
// 拒絕 "../", 絕對路徑等危險輸入
```

### 2. 快取策略

預覽圖快取 24 小時，最大 160 個檔案。適度的快取能提升效能，但也不能過度，畢竟平衡才是關鍵：

```csharp
// 預覽圖快取 24 小時，最大 160 個檔案
private static readonly TimeSpan PreviewCacheTtl = TimeSpan.FromHours(24);
private const int PreviewCacheMaxFiles = 160;
// 定期清理過期快取
```

### 3. 錯誤處理

上游站點不可用時降級處理。這種優雅降級的設計確保了即使外部依賴不可用，核心編輯功能仍然正常工作。畢竟，不能因為一個外部服務掛了，整個系統就癱瘓了：

```typescript
// 上游站點不可用時降級處理
try {
    const catalog = await getProjectDesignMdSiteImportCatalog(projectId);
} catch (error) {
    toast.error(t('project.designMd.siteImport.feedback.catalogLoadFailed'));
    // 主編輯抽屜仍然可用
}
```

這種優雅降級的設計確保了即使外部依賴不可用，核心編輯功能仍然正常工作。系統應該有韌性，而不是一遇到問題就倒下。

### 4. 使用者體驗優化

匯入前確認覆蓋，匯入後不自動儲存。使用者應該對自己的操作有控制權，而不是系統自作主張：

```typescript
// 匯入前確認覆蓋
if (isDirty) {
    setConfirmMode('import');
    return;
}

// 匯入後不自動儲存，由使用者確認
setDraft(result.content); // 只更新草稿
// 使用者檢查後點擊 Save 才真正寫入磁碟
```

### 5. 效能考量

使用 HTTP 用戶端工廠，避免建立過多連線。資源管理這種事，看似不起眼，但做好了能帶來意想不到的效果：

```csharp
// 使用 HTTP 用戶端工廠，避免建立過多連線
private const string DesignSiteProxyClientName = "ProjectDesignSiteProxy";
private static readonly TimeSpan DesignSiteProxyTimeout = TimeSpan.FromSeconds(8);
```

## 擴展建議

1. **Markdown 增強**：當前使用基礎 `Textarea`，可考慮升級為 CodeMirror 以支援語法高亮和快捷鍵。編輯器的體驗好了，寫文件的心情也會好一些。
2. **預覽模式**：新增 Markdown 即時預覽，提升編輯體驗。所見即所得，總能給人更多信心。
3. **差異合併**：實現智慧合併演算法，而非簡單的全文替換。衝突是難免的，但至少可以讓處理衝突的過程不那么痛苦。
4. **本地快取**：將 design.json 快取到資料庫，減少對外部站點的依賴。依賴越少，系統越穩定，這是簡單的道理。

## 總結

在 HagiCode 專案中，我們透過前後端協作實作了一套完整的 DESIGN.md 線上編輯和匯入方案。這套方案的核心價值在於：

- **提升效率**：無需切換工具，在統一的 Web 介面完成設計文件的編輯和匯入。省下的時間，可以做更有意義的事情。
- **降低門檻**：設計範本一鍵匯入，新專案可以快速起步。開始得越容易，堅持下去的可能性就越大。
- **安全可靠**：路徑校驗、版本衝突檢測、優雅降級等機制確保系統穩定運行。穩定是基礎，沒有穩定，一切都是空談。
- **使用者體驗**：全域抽屜、髒狀態檢測、確認對話方塊等細節打磨了互動體驗。細節決定成敗，這句話在使用者體驗上尤其適用。

這套方案已經在 HagiCode 專案中實際運行，解決了團隊在設計文件管理方面的痛點。如果你也在面臨類似的問題，希望這篇文章能給你一些啟發。其實也沒什麼高深的理論，只是遇到了問題，想辦法解決而已。
