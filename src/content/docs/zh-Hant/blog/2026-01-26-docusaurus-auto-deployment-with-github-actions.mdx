## 為 HagiCode 添加 GitHub Pages 自動部署支持

> 本項目早期代號為 PCode，現已正式更名為 HagiCode。本文記錄了如何為項目引入自動化靜態站點部署能力，讓內容發布像喝水一樣簡單。

## 背景/引言

在 HagiCode 的開發過程中，我們遇到了一個很現實的問題：隨著文檔和提案越來越多，如何高效地管理和展示這些內容成了當務之急。我們決定引入 GitHub Pages 來託管我們的靜態站點，但是手動構建和部署實在是太麻煩了——每次改動都要本地構建、打包，然後手動推送到 `gh-pages` 分支。這不僅效率低下，還容易出錯。

為了解決這個問題（主要是為了偷懶），我們需要一套自動化的部署流程。本文將詳細記錄如何為 HagiCode 項目添加 GitHub Actions 自動部署支持，讓我們只需專注於內容創作，剩下的交給自動化流程。

## 目標分析

在動手之前，我們得先明確這次任務到底要幹啥。畢竟（這裡打錯了，應該是畢竟）磨刀不誤砍柴工嘛。

### 核心需求

1.  **自動化構建**：當代碼推送到 `main` 分支時，自動觸發構建流程。
2.  **自動部署**：構建成功後，自動將生成的靜態文件部署到 GitHub Pages。
3.  **環境一致性**：確保 CI 環境和本地構建環境一致，避免"本地能跑，線上報錯"的尷尬。

### 技術選型

考慮到 HagiCode 是基於 Docusaurus 構建的（一種非常流行的 React 靜態站點生成器），我們可以利用 GitHub Actions 來實現這一目標。

## 配置 GitHub Actions 工作流

GitHub Actions 是 GitHub 提供的 CI/CD 服務。通過在代碼倉庫中定義 YAML 格式的工作流文件，我們可以定製各種自動化任務。

### 創建工作流文件

我們需要在項目根目錄下的 `.github/workflows` 文件夾中創建一個新的配置文件，比如叫 `deploy.yml`。如果文件夾不存在，記得先手動創建一下。

這個配置文件的核心邏輯如下：

1.  **觸發條件**：監聽 `main` 分支的 `push` 事件。
2.  **運行環境**：最新版的 Ubuntu。
3.  **構建步驟**：
    -   檢出代碼
    -   安裝 Node.js
    -   安裝依賴 (`npm install`)
    -   構建靜態文件 (`npm run build`)
4.  **部署步驟**：使用官方提供的 `action-gh-pages` 將構建產物推送到 `gh-pages` 分支。

### 關鍵配置代碼

以下是我們最終採用的配置模板：

```yaml
name: Deploy to GitHub Pages

# 觸發條件：當推送到 main 分支時
on:
  push:
    branches:
      - main
    # 可以根據需要添加路徑過濾，比如只有文檔變動才構建
    # paths:
    #   - 'docs/**'
    #   - 'package.json'

# 設置權限，這對於部署到 GitHub Pages 很重要
permissions:
  contents: read
  pages: write
  id-token: write

# 並發控制：取消同一分支的舊構建
concurrency:
  group: "pages"
  cancel-in-progress: false

jobs:
  build:
    runs-on: ubuntu-latest
    steps:
      - name: Checkout
        uses: actions/checkout@v4
        # 注意：必須設置 fetch-depth: 0，否則可能導致構建版本號不準確
        with:
          fetch-depth: 0

      - name: Setup Node
        uses: actions/setup-node@v4
        with:
          node-version: 20 # 建議與本地開發環境保持一致
          cache: 'npm'     # 啟用緩存可以加速構建過程

      - name: Install dependencies
        run: npm ci
        # 使用 npm ci 而不是 npm install，因為它更快、更嚴格，適合 CI 環境

      - name: Build website
        run: npm run build
        env:
          # 如果你的站點構建需要環境變量，在這裡配置
          # NODE_ENV: production
          # PUBLIC_URL: /your-repo-name
          
      - name: Upload artifact
        uses: actions/upload-pages-artifact@v3
        with:
          path: ./build # Docusaurus 默認輸出目錄

  deploy:
    environment:
      name: github-pages
      url: ${{ steps.deployment.outputs.page_url }}
    runs-on: ubuntu-latest
    needs: build
    steps:
      - name: Deploy to GitHub Pages
        id: deployment
        uses: actions/deploy-pages@v4
```

## 實施過程中的坑點

在實際操作中，我們遇到了一些問題，這裡分享出來希望大家能避开（或者提前準備好解決方案）。

### 1. GitHub Token 權限問題

最開始配置的時候，部署總是報錯 403 (Forbidden)。查了好久才發現，是因為 GitHub 默認的 `GITHUB_TOKEN` 並沒有寫入 Pages 的權限。

**解決方案**：在倉庫的 `Settings` -> `Actions` -> `General` -> `Workflow permissions` 中，務必選擇 **"Read and write permissions"**。

### 2. 構建目錄路徑錯誤

Docusaurus 默認把構建好的靜態文件放在 `build` 目錄。但是有些項目（比如 Create React App 默認是 `build`，Vite 默認是 `dist`）可能配置不一樣。如果在 Actions 中報錯找不到文件，記得去 `docusaurus.config.js` 裡檢查一下輸出路徑配置。

### 3. 子路徑問題

如果你的倉庫不是用戶主頁（即不是 `username.github.io`），而是項目主頁（比如 `username.github.io/project-name`），你需要配置 `baseUrl`。

在 `docusaurus.config.js` 中：

```javascript
module.exports = {
  // ...
  url: 'https://hagicode.com', // 你的 Hagicode URL
  baseUrl: '/', // 根路徑部署
  // ...
};
```
這一點很容易被忽略，配置不對會導致頁面打開全是白屏，因為資源路徑加載不到。

## 驗證成果

配置完所有東西並推送代碼後，我們就可以去 GitHub 倉庫的 **Actions** 標籤頁看戲了。

你會看到黃色的圓圈（工作流正在運行），變綠就代表成功啦！如果變紅了，點擊進去查看日誌，通常都能排查出問題（大部分時候是拼寫錯誤或者路徑配置不對）。

構建成功後，訪問 `https://<你的用戶名>.github.io/<倉庫名>/` 就能看到嶄新的站點了。

## 總結

通過引入 GitHub Actions，我們成功實現了 HagiCode 文檔站的自動化部署。這不僅節省了手動操作的時間，更重要的是保證了發布流程的標準化。現在不管是哪位小夥伴更新了文檔，只要合併到 `main` 分支，幾分鐘後就能在線上看到最新的內容。

**核心收益**：
- **效率提升**：從"手動打包、手動上傳"變成"代碼即發布"。
- **降低錯誤**：消除了人為操作失誤的可能性。
- **體驗優化**：讓開發者更專注於內容質量，而不是被繁瑣的部署流程困擾。

雖然配置 CI/CD 剛開始有點麻煩（尤其是各種權限和路徑問題），但這是一次性投入，長期回報巨大的工作。強烈建議所有靜態站點項目都接入類似的自動化流程。
