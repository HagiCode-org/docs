## 爲 HagiCode 新增 GitHub Pages 自動部署支援

> 本專案早期代號爲 PCode，現已正式更名爲 HagiCode。本文記錄瞭如何爲專案引入自動化靜態站點部署能力，讓內容發佈像喝水一樣簡單。

## 背景/引言

在 HagiCode 的開發程序中，我們遇到了一個很現實的問題：隨着文件和提案越來越多，如何高效地管理和展示這些內容成了當務之急。我們決定引入 GitHub Pages 來代管我們的靜態站點，但是手動建置和部署實在是太麻煩了——每次改動都要本地建置、打包，然後手動推送到 `gh-pages` 分支。這不僅效率低下，還容易出錯。

爲了解決這個問題（主要是爲了偷懶），我們需要一套自動化的部署流程。本文將詳細記錄如何爲 HagiCode 專案新增 GitHub Actions 自動部署支援，讓我們只需專注於內容創作，剩下的交給自動化流程。

## 目標分析

在動手之前，我們得先明確這次任務到底要幹啥。畢竟（這裏打錯了，應該是畢竟）磨刀不誤砍柴工嘛。

### 核心需求

1.  **自動化建置**：當程式碼推送到 `main` 分支時，自動觸發建置流程。
2.  **自動部署**：建置成功後，自動將產生的靜態檔案部署到 GitHub Pages。
3.  **環境一致性**：確保 CI 環境和本地建置環境一致，避免"本地能跑，線上報錯"的尷尬。

### 技術選型

考慮到 HagiCode 是基於 Docusaurus 建置的（一種非常流行的 React 靜態站點產生器），我們可以利用 GitHub Actions 來實作這一目標。

## 設定 GitHub Actions 工作流

GitHub Actions 是 GitHub 提供的 CI/CD 服務。透過在程式碼存放庫中定義 YAML 格式的工作流檔案，我們可以定製各種自動化任務。

### 建立工作流檔案

我們需要在專案根目錄下的 `.github/workflows` 檔案夾中建立一個新的設定檔案，比如叫 `deploy.yml`。如果檔案夾不存在，記得先手動建立一下。

這個設定檔案的核心邏輯如下：

1.  **觸發條件**：監聽 `main` 分支的 `push` 事件。
2.  **執行環境**：最新版的 Ubuntu。
3.  **建置步驟**：
    -   檢出程式碼
    -   安裝 Node.js
    -   安裝相依 (`npm install`)
    -   建置靜態檔案 (`npm run build`)
4.  **部署步驟**：使用官方提供的 `action-gh-pages` 將建置產物推送到 `gh-pages` 分支。

### 關鍵設定程式碼

以下是我們最終採用的設定範本：

```yaml
name: Deploy to GitHub Pages

# 触发条件：当推送到 main 分支时
on:
  push:
    branches:
      - main
    # 可以根据需要添加路径过滤，比如只有文档变动才构建
    # paths:
    #   - 'docs/**'
    #   - 'package.json'

# 设置权限，这对于部署到 GitHub Pages 很重要
permissions:
  contents: read
  pages: write
  id-token: write

# 并发控制：取消同一分支的旧构建
concurrency:
  group: "pages"
  cancel-in-progress: false

jobs:
  build:
    runs-on: ubuntu-latest
    steps:
      - name: Checkout
        uses: actions/checkout@v4
        # 注意：必须设置 fetch-depth: 0，否则可能导致构建版本号不准确
        with:
          fetch-depth: 0

      - name: Setup Node
        uses: actions/setup-node@v4
        with:
          node-version: 20 # 建议与本地开发环境保持一致
          cache: 'npm'     # 启用缓存可以加速构建过程

      - name: Install dependencies
        run: npm ci
        # 使用 npm ci 而不是 npm install，因为它更快、更严格，适合 CI 环境

      - name: Build website
        run: npm run build
        env:
          # 如果你的站点构建需要环境变量，在这里配置
          # NODE_ENV: production
          # PUBLIC_URL: /your-repo-name
          
      - name: Upload artifact
        uses: actions/upload-pages-artifact@v3
        with:
          path: ./build # Docusaurus 默认输出目录

  deploy:
    environment:
      name: github-pages
      url: ${{ steps.deployment.outputs.page_url }}
    runs-on: ubuntu-latest
    needs: build
    steps:
      - name: Deploy to GitHub Pages
        id: deployment
        uses: actions/deploy-pages@v4
```

## 實施程序中的坑點

在實際操作中，我們遇到了一些問題，這裏分享出來希望大家能避開（或者提前準備好解決方案）。

### 1. GitHub Token 權限問題

最開始設定的時候，部署總是報錯 403 (Forbidden)。查了好久才發現，是因爲 GitHub 預設的 `GITHUB_TOKEN` 並沒有寫入 Pages 的權限。

**解決方案**：在存放庫的 `Settings` -> `Actions` -> `General` -> `Workflow permissions` 中，務必選擇 **"Read and write permissions"**。

### 2. 建置目錄路徑錯誤

Docusaurus 預設把建置好的靜態檔案放在 `build` 目錄。但是有些專案（比如 Create React App 預設是 `build`，Vite 預設是 `dist`）可能設定不一樣。如果在 Actions 中報錯找不到檔案，記得去 `docusaurus.config.js` 裏檢查一下輸出路徑設定。

### 3. 子路徑問題

如果你的存放庫不是使用者主頁（即不是 `username.github.io`），而是專案主頁（比如 `username.github.io/project-name`），你需要設定 `baseUrl`。

在 `docusaurus.config.js` 中：

```javascript
module.exports = {
  // ...
  url: 'https://hagicode.com', // 你的 Hagicode URL
  baseUrl: '/', // 根路径部署
  // ...
};
```
這一點很容易被忽略，設定不對會導致頁面開啟全是白屏，因爲資源路徑載入不到。

## 驗證成果

設定完所有東西並推送程式碼後，我們就可以去 GitHub 存放庫的 **Actions** 標籤頁看戲了。

你會看到黃色的圓圈（工作流正在執行），變綠就代表成功啦！如果變紅了，點擊進去查看日誌，通常都能排查出問題（大部分時候是拼寫錯誤或者路徑設定不對）。

建置成功後，存取 `https://<你的使用者名>.github.io/<存放庫名>/` 就能看到嶄新的站點了。

## 總結

透過引入 GitHub Actions，我們成功實作了 HagiCode 文件站的自動化部署。這不僅節省了手動操作的時間，更重要的是保證了發佈流程的標準化。現在不管是哪位小夥伴更新了文件，只要合併到 `main` 分支，幾分鐘後就能在線上看到最新的內容。

**核心收益**：
- **效率提升**：從"手動打包、手動上傳"變成"程式碼即發佈"。
- **降低錯誤**：消除了人爲操作失誤的可能性。
- **體驗優化**：讓開發人員更專注於內容質量，而不是被繁瑣的部署流程困擾。

雖然設定 CI/CD 剛開始有點麻煩（尤其是各種權限和路徑問題），但這是一次性投入，長期回報巨大的工作。強烈建議所有靜態站點專案都接入類似的自動化流程。
