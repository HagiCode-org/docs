## VSCode 與 code-server：瀏覽器端程式碼編輯方案選型

> 在建構瀏覽器端的程式碼編輯能力時，開發者面臨一個關鍵選擇：使用 VSCode 官方的 `code serve-web` 功能，還是採用社群驅動的 `code-server` 方案？這個選擇不僅影響技術架構，還關係到許可證合規性和部署彈性。

## 背景

其實做技術選型這件事，跟選擇人生道路有點像。你選了一條路，就得一直走下去，想換路的時候，成本可就大了。

在 AI 輔助程式設計的時代，瀏覽器端的程式碼編輯能力變得越來越重要。用戶期望在 AI 助手分析完程式碼後，能夠立即在同一個瀏覽器會話中打開編輯器進行修改，無需切換應用。這種無縫的體驗，怎麼說呢，就像你想的時候，它就在那裡——只是有時候它偏偏不在。

然而，在實現這個功能時，開發者面臨一個關鍵的技術選型：是使用 VSCode 官方的 `code serve-web` 功能，還是採用社群驅動的 `code-server` 方案？

這兩個方案各有優劣，選擇錯了可能會在後期帶來不少麻煩。比如許可證問題——等到產品上線了才發現許可證不合規，那可就晚了。這跟談戀愛有點像，一開始沒想清楚，到頭來才發現兩個人的價值觀根本不合，那付出的代價就大了。再比如部署方式——某個方案在開發環境跑得好好的，一上容器就各種問題，這種坑誰也不想踩，畢竟踩坑踩多了，人也就麻了。

## 關於 HagiCode

本文分享的方案來自我們在 [HagiCode](https://hagicode.com) 專案中的實踐經驗。HagiCode 是一個 AI 驅動的程式碼助手，在實現瀏覽器端程式碼編輯能力時，我們深入研究了這兩個方案，最終在架構設計中同時支援兩者，但預設優先選擇 code-server。

專案地址：[github.com/HagiCode-org/site](https://github.com/HagiCode-org/site)

## 許可證差異（最關鍵）

這是兩個方案最根本的區別，也是我們在選型時最先考慮的因素。畢竟選型這件事，第一步就要想清楚法律風險，不然以後出事了，怪誰呢？

**code-server**
- MIT 許可證，完全開源
- 由 Coder.com 公司維護，社群活躍
- 可以自由商用、修改和分發
- 無使用場景限制

**VSCode `code serve-web`**
- 屬於 Microsoft VSCode 產品的一部分
- 使用 Microsoft 的許可證（VS Code 的許可證有商業使用限制）
- 主要面向個人開發者使用
- 企業級部署可能需要額外的商業授權考慮

從許可證角度看，code-server 對商業專案更友善。這一點在產品規劃階段就要想清楚，不然等到規模上來了再去遷移，那成本可就大了。畢竟遷移這件事，說起來容易，做起來難，誰經歷過誰知道。

## 部署方式差異

許可證問題解決後，接下來就是部署方式。這直接影響到你的運維成本和架構設計，也間接影響著你每天的心情——部署越簡單，心情越好，這道理大家都懂。

**code-server**
- 獨立的 Node.js 應用，可單獨部署
- 支援多種執行時來源：
  - 直接指定可執行檔路徑
  - 系統 PATH 查找
  - NVM Node.js 22.x 環境自動檢測
- 伺服器上無需安裝 VSCode 桌面版
- 容器化部署更簡單

**VSCode `code serve-web`**
- 必須依賴本地安裝的 VSCode CLI
- 需要本機有可用的 `code` 命令
- 系統會過濾掉 VS Code Remote CLI 包裝器
- 主要設計用於本地開發場景

code-server 更適合伺服器/容器部署場景。如果你的產品需要跑在 Docker 裡，或者用戶環境沒有 VSCode，那選 code-server 就對了。畢竟簡單就是美，複雜了容易出問題，出了問題還得修，修完了還可能引入新問題，這無窮無盡的循環，誰願意經歷呢？

## 功能參數差異

兩個方案在功能參數上也有一些差異，雖然不大，但在實際使用中可能會帶來一些麻煩。這些細節就像生活中的小摩擦，不多，但多了就讓人煩。

| 特性 | code-server | code serve-web |
|------|-------------|----------------|
| 公開基路徑 | `/` (可配置) | `/vscode-server` (固定) |
| 認證方式 | `--auth` 參數，支援多種模式 | `--connection-token` / `--without-connection-token` |
| 資料目錄 | `{DataDir}/code-server` | `{DataDir}/vscode-serve-web` |
| 遙測 | 預設停用 `--disable-telemetry` | 依賴 VSCode 設定 |
| 更新檢查 | 可停用 `--disable-update-check` | 依賴 VSCode 設定 |

這些差異在整合時需要特別注意。比如 URL 路徑的不同，意味著前端程式碼需要做針對性處理。做開發的都知道，這種細節處理起來最費時間，但也沒辦法，不做就跑不通。

## 可用性檢測差異

在實現編輯器切換功能時，可用性檢測的邏輯也有所不同。這種差異就像人與人之間的相處方式，有的人喜歡直來直去，有的人喜歡含蓄委婉。

**code-server**
- 始終作為可見實作返回
- 即使不可用也會顯示，提示 `install-required` 狀態
- 支援自動檢測 NVM Node.js 22.x 環境

**code serve-web**
- 只有檢測到本機 `code` CLI 時才可見
- 如果不可用，前端會自動隱藏此選項
- 依賴本地 VSCode 安裝狀態

這種差異直接影響用戶體驗。code-server 的方式更透明，用戶知道有這個選項，只是還沒安裝；code serve-web 的方式更隱蔽，用戶可能都不知道還有這個選擇。哪種方式更好？這得看產品定位了，畢竟用戶體驗這件事，沒有標準答案，只有合適不合適。

## HagiCode 的雙實作架構

經過深入分析，HagiCode 專案採用了雙實作架構，在架構層面同時支援兩種方案。這倒不是我們技術選型困難症，而是真的有實際需求。畢竟在技術世界裡，沒有什麼絕對正確的選擇，只有最適合自己的選擇。

### 預設選擇 code-server

```csharp
// 預設活動實作是 code-server
// 如果保存了顯式的 activeImplementation，優先嘗試該實作
// 如果所請求實作不可用，解析器會嘗試另一個實作
// 如果發生回退，返回 fallbackReason
```

我們預設選擇 code-server，主要考慮是許可證和部署彈性。但對於有本地 VSCode 環境的用戶，code serve-web 也是一個不錯的選擇。畢竟給用戶多一種選擇，總歸是好的，強迫別人接受單一方案，這件事我幹不出來。

### 實作選擇器

`CodeServerImplementationResolver` 統一負責：
- 啟動預熱時的實作選擇
- 狀態讀取時的實作選擇
- 專案打開時的實作選擇
- Vault 打開時的實作選擇

這種設計讓系統可以靈活應對不同場景，用戶可以根據自己的環境選擇最合適的實作。靈活設計這件事，前期花的時間多一點，但後期省心，畢竟誰也不想到處改程式碼。

### 前端適配規則

```typescript
// localCodeAvailable=false 時，不顯示 code serve-web
// localCodeAvailable=true 時，顯示 code serve-web 設定
```

前端根據環境自動顯示可用選項，避免用戶看到無法使用的功能而困惑。用戶困惑了，就來問你，問多了你也煩，煩多了就想改程式碼，改程式碼又可能引入 bug，這惡性循環，誰愛經歷誰經歷。

## 實踐指南

說了這麼多理論，實際部署時要注意什麼呢？其實理論說得再好，落地不行也沒用，畢竟實踐是檢驗真理的唯一標準。

### Docker 部署建議

對於容器化部署，code-server 是更優選擇：

```dockerfile
# 直接使用 code-server 官方映像檔
FROM codercom/code-server:latest

# 或者透過 npm 安裝
RUN npm install -g code-server
```

這樣一層就搞定，不需要額外安裝 VSCode。簡單就是好，複雜了容易出錯，這道理放哪兒都適用。

### 設定範例

**code-server 設定**
```json
{
  "vscodeServer": {
    "enabled": true,
    "activeImplementation": "code-server",
    "codeServer": {
      "host": "0.0.0.0",
      "port": 8080,
      "executablePath": "",
      "authMode": "none"
    }
  }
}
```

**code serve-web 設定**
```json
{
  "vscodeServer": {
    "enabled": true,
    "activeImplementation": "serve-web",
    "serveWeb": {
      "host": "0.0.0.0",
      "port": 8080,
      "executablePath": "/usr/local/bin/code"
    }
  }
}
```

設定這件事，第一次麻煩點，配好了後面就省心了。就像生活一樣，前期投入多一點，後面日子就好過一點。

### URL 構建差異

**code-server**
```
http://localhost:8080/?folder=/path/to/project&vscode-lang=zh-CN
```

**code serve-web**
```
http://localhost:8080/vscode-server/?folder=/path/to/project&tkn=xxx&vscode-lang=zh-CN
```

注意路徑和參數的差異，整合時需要分別處理。細節決定成敗，這話一點都不假，少一個參數，可能就打不開頁面。

### 切換實作

系統支援執行時切換，切換時會自動停止舊實作：

```csharp
// VsCodeServerManager 自動處理互斥
// 切換 activeImplementation 時，舊實作不會繼續後台保活
```

這種設計讓用戶可以隨時嘗試不同實作，找到最適合自己的方案。畢竟適合自己的才是最好的，別人的建議僅供參考，最終還得自己試。

### 狀態監控

```typescript
const { settings, runtime } = await getVsCodeServerSettings();

// runtime.activeImplementation: "code-server" | "serve-web"
// runtime.fallbackReason: 切換原因
// runtime.status: "running" | "starting" | "stopped" | "unhealthy"
```

狀態可見，心裡才有數。用戶在遇到問題時可以快速定位是伺服器端問題還是自己操作問題。不知道狀態的時候，人就容易慌，慌了就容易做出錯誤判斷，這鏈條一旦啟動，就停不下來了。

## 總結

| 對比維度 | code-server | code serve-web | 推薦 |
|---------|-------------|----------------|------|
| 許可證 | MIT（商用友善） | Microsoft（有限制） | code-server |
| 部署彈性 | 獨立部署 | 依賴本地 VSCode | code-server |
| 伺服器適用性 | 專為伺服器設計 | 主要面向本地開發 | code-server |
| 容器化 | 原生支援 | 需要安裝 VSCode | code-server |
| 功能完整性 | 接近桌面版 | 官方完整版 | code serve-web |
| 維護活躍度 | 社群活躍 | Microsoft 官方 | 各有優勢 |

**推薦策略**：優先使用 code-server，在需要完整官方功能且具備本地 VSCode 環境時考慮 code serve-web。
