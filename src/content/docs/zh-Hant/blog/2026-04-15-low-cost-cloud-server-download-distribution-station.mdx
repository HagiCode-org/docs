## 如何利用廉價的雲伺服器搭建加速下載的分發站

> 雲存儲的流量貴得離譜，跨國存取慢得讓人抓狂，CDN 的價格又讓人望而卻步......這些痛，大概做分發的都懂吧。本文分享一下我們在 [HagiCode](https://hagicode.com) 專案裏摸索出來的一套低成本方案。雲伺服器 + Nginx 快取層，成本降了一半，速度倒是提上去了，也算是一點小小的安慰。

## 背景

互聯網嘛，下載速度和穩定性，說到底都是使用者體驗。開放原始碼也好，商業也罷，總得給使用者一個靠譜的下載方式。

直接從雲存儲（比如 Azure Blob Storage、AWS S3）下檔案，看起來簡單，實際上問題還真不少：

**網路延遲**：跨國跨地域的存取，慢得讓人想砸鍵盤。使用者等得花兒都謝了，體驗能好到哪去？

**頻寬成本**：雲存儲的出口流量，貴得讓人心疼。Azure Blob Storage 在中國大陸存取，差不多每 GB 0.5 元，一個月 1TB 就是 500 塊。對於小團隊來說，這筆錢說多不多，說少也不少，畢竟誰的錢都不是大風颳來的。

**存取限制**：某些地區存取國外雲服務，時好時壞，有時候乾脆就存取不了。使用者想下都下不了，這事兒本身就挺無奈的。

**CDN 成本**：商業 CDN 確實能解決問題，但價格也確實美麗。小團隊哪裏用得起？

那有沒有既省錢又好用的辦法呢？其實也是有的。雲伺服器 + 反向代理 + 快取層，就這麼簡單粗暴。成本降了一半左右，速度還提上去了，也算是一點小小的慰藉。

## 架構設計

### 整體架構思路

先來看一下整體的架構設計：

```
用户请求
    ↓
DNS 解析
    ↓
┌─────────────────────────────────────┐
│   反向代理层 (Traefik/Bunker Web)   │ ← SSL 终止、路由分发、安全防护
├─────────────────────────────────────┤
│   端口: 80/443                       │
│   功能: 自动 Let's Encrypt 证书      │
│         Host 路由                    │
└─────────────────────────────────────┘
    ↓
┌─────────────────────────────────────┐
│   缓存层 (Nginx)                     │ ← 文件缓存、Gzip 压缩
├─────────────────────────────────────┤
│   端口: 8080(server) / 8081(desktop) │
│   缓存策略:                          │
│   - index.json: 1 小时               │
│   - 其他文件: 7 天                   │
│   缓存大小: 1GB                      │
└─────────────────────────────────────┘
    ↓
┌─────────────────────────────────────┐
│   源站 (Azure Blob Storage)         │ ← 文件存储
└─────────────────────────────────────┘
```

這個架構的核心思想，說白了就是：**在使用者和雲存儲之間，加一層快取罷了**。

使用者請求先到雲伺服器上的反向代理層，然後 Nginx 快取層接手。快取裏有檔案？直接給使用者。沒有？去雲存儲拉一份，順便在本地存一份。下次再存取同一個檔案，就不用麻煩雲存儲了。其實就像記憶一樣，有些東西記住了，下次就不用再費勁去想......

### 爲什麼選擇這個架構？

**雲伺服器優勢**：
- 成本可控：阿里雲等提供廉價雲伺服器（1-2 核 2GB 設定月費約 50-100 元）
- 部署靈活：可自由設定反向代理、快取策略
- 地理位置靈活：可選擇靠近使用者的伺服器區域
- 擴充性強：可根據流量需求升級設定

**反向代理 + 快取架構**：
- 減少源站壓力：快取熱點檔案，減少對雲存儲的存取
- 降低成本：雲伺服器流量費用遠低於雲存儲出口流量
- 提升速度：就近存取，伺服器頻寬通常優於雲存儲

**爲什麼選擇 Nginx 作爲快取層？**

這事兒也不是隨便選的，畢竟 Nginx 確實有它的道理：

1. 高效能：Nginx 反向代理的效能，業界是有目共睹的
2. 快取成熟：內建的 proxy_cache 功能，穩定可靠，不會動不動就出幺蛾子
3. 資源佔用低：256MB 記憶體就能跑，對伺服器也算友好
4. 設定靈活：不同的檔案型別可以設定不同的快取策略，這點還是很貼心的

## 反向代理層：Traefik vs Bunker Web

HagiCode 的部署方案，其實支援兩種反向代理，怎麼說呢，各有各的特點：

| 方案 | 特點 | 適用場景 |
|------|------|----------|
| **Traefik** | 輕量級、自動 SSL、設定簡單 | 基礎部署、低流量場景 |
| **Bunker Web** | 內建 WAF、防 DDoS、防爬蟲 | 高安全要求、高流量場景 |

### Traefik：輕量級首選

Traefik 是一個現代 HTTP 反向代理和負載均衡器，最大的特點就是設定簡單，還能自動搞 Let's Encrypt 證書。

對於初始部署或者流量不大的場景，Traefik 其實是挺不錯的選擇：
- 資源佔用不多（1.5 CPU/512MB 記憶體就夠了）
- SSL 證書自動設定，不用自己操心
- 路由設定基於 Docker 標籤，也算方便

### Bunker Web：高安全場景

Bunker Web 是一個基於 Nginx 的 Web 應用防火牆，安全防護更全面一些。

什麼時候可以考慮切換到 Bunker Web 呢？大概就是這些情況吧：
- 遭受 DDoS 攻擊（雖然誰都不希望遇到）
- 需要 ModSecurity 防護
- 想要防爬蟲功能
- 對安全有更高的要求

HagiCode 提供了 `switch-deployment.sh` 腳本，可以在兩種方案之間快速切換：

```bash
# 切换到 Bunker Web
./switch-deployment.sh bunkerweb

# 切换回 Traefik
./switch-deployment.sh traefik

# 查看当前状态
./switch-deployment.sh status
```

腳本會自動做預檢查、健康檢查，還能自動回復，切換程序也算安全可靠，不會說換着換着就掛了。

## Nginx 快取層設定

快取層是整個架構的核心，Nginx 設定得好不好，快取效果天差地別。畢竟這事兒還挺關鍵的。

### 快取路徑設定

```nginx
# 缓存路径配置
proxy_cache_path /var/cache/nginx levels=1:2 keys_zone=azure_cache:10m
                   max_size=1g inactive=7d use_temp_path=off;
```

參數說明：
- `levels=1:2`：快取目錄層級，2 級目錄結構，提高檔案存取效率
- `keys_zone=azure_cache:10m`：快取鍵存儲區域，10MB 足夠存儲大量鍵
- `max_size=1g`：最大快取大小 1GB
- `inactive=7d`：快取檔案 7 天未被存取則刪除
- `use_temp_path=off`：直接寫入快取目錄，提高效能

### 分級快取策略

不同型別的檔案需要不同的快取策略：

```nginx
# Server 下载服务
server {
    listen 8080;

    # index.json 短期缓存（便于及时更新）
    location /index.json {
        proxy_cache azure_cache;
        proxy_cache_valid 200 1h;
        proxy_cache_key "$scheme$server_port$request_uri";
        add_header X-Cache-Status $upstream_cache_status;
        add_header Cache-Control "public, max-age=3600";

        # 反向代理到 Azure OSS
        proxy_pass https://${SERVER_DL_HOST}/${SERVER_DL_CONTAINER}$uri?${SERVER_DL_SAS_TOKEN};
        proxy_ssl_server_name on;
        proxy_ssl_protocols TLSv1.2 TLSv1.3;
    }

    # 安装包等静态文件长期缓存
    location / {
        proxy_cache azure_cache;
        proxy_cache_valid 200 7d;
        proxy_cache_key "$scheme$server_port$request_uri";
        add_header X-Cache-Status $upstream_cache_status;
        add_header Cache-Control "public, max-age=604800";

        proxy_pass https://${SERVER_DL_HOST}/${SERVER_DL_CONTAINER}$uri?${SERVER_DL_SAS_TOKEN};
        proxy_ssl_server_name on;
        proxy_ssl_protocols TLSv1.2 TLSv1.3;
    }
}
```

**爲什麼這樣設計？**

`index.json` 是版本檢查檔案，得及時更新，所以快取時間設成了 1 小時。這樣發佈新版本後，最多 1 小時使用者就能檢測到更新，也不算太久。

安裝包這些靜態檔案，變化本來就少，快取 7 天能大幅減少源站存取。需要更新的時候手動清理一下快取就好了，也不算麻煩。

**X-Cache-Status 回應頭**：

這個回應頭能讓你看看快取命中情況怎麼樣，也挺有用的：
- `HIT`：快取命中
- `MISS`：快取未命中，從源站拉取
- `EXPIRED`：快取過期，重新從源站拉取
- `BYPASS`：快取被繞過

查看方法：
```bash
curl -I https://server.dl.hagicode.com/app.zip
```

## 成本分析

假設每月下載流量 1TB，我們來算算賬：

| 方案 | 流量費用 | 伺服器費用 | 合計 |
|------|----------|------------|------|
| 直接 Azure OSS | 約 ¥500 | ¥0 | ¥500 |
| 雲伺服器 + OSS (快取命中率 80%) | ¥100 + ¥80 | ¥60 | ¥240 |
| 商業 CDN | ¥300-500 | ¥0 | ¥300-500 |

**結論**：快取層能省下大概 50% 的分發成本。

這個計算假設快取命中率 80%，實際情況中，如果檔案更新頻率不高，命中率可能還會更高一些，畢竟這也是自然而然的事。

## 運維技巧

### 快取管理

快取這東西，偶爾也得打理一下：

**查看快取磁盤使用**：
```bash
docker volume inspect docker_nginx-cache
du -sh /var/lib/docker/volumes/docker_nginx-cache/_data
```

**手動清除快取**：
```bash
./clear-cache.sh
```

或者手動執行，雖然麻煩一點，但也管用：
```bash
docker exec nginx sh -c "rm -rf /var/cache/nginx/*"
docker restart nginx
```

### 資源限制

在 1 核 2GB 伺服器上，資源限制設定如下：

```yaml
services:
  traefik:
    deploy:
      resources:
        limits:
          cpus: '1.50'
          memory: 512M

  nginx:
    deploy:
      resources:
        limits:
          cpus: '0.50'
          memory: 256M
```

監控資源使用情況，偶爾看看也挺好：
```bash
docker stats
```

### SAS Token 安全實踐

SAS Token 是存取 Azure Blob Storage 的憑證，泄露了可不是鬧着玩的：

- `.env` 檔案不提交到版本控制（已在 .gitignore）
- SAS Token 設定適當過期時間（推薦 1 年）
- 限制 SAS Token 權限（僅讀取）
- 定期輪換 SAS Token

### 監控告警

HagiCode 集成了 Lark/飛書 Webhook 通知，可以在以下情況發送通知：
- 部署成功/失敗
- 快取清除狀態
- 服務例外

通知包含伺服器資訊、時間戳、錯誤詳情，快速定位問題也方便一些。

### 高可用擴充

當單臺伺服器撐不住的時候，也可以考慮：

1. **水平擴充**：部署多個節點，透過 DNS 輪詢或負載均衡分發
2. **CDN 加持**：在雲伺服器前接入 CDN，進一步提升存取速度
3. **快取預熱**：使用腳本提前將熱門檔案載入到快取

## 注意事項

有幾點還是得提醒一下，畢竟誰都不想遇到幺蛾子：

1. **SSL 證書**：Let's Encrypt 有速率限制，別頻繁切換部署，不然可能申請不到證書
2. **快取清理**：更新重要檔案後記得清理快取，不然使用者可能下載不到新版本
3. **日誌管理**：定期清理 Docker 日誌，不然磁盤滿了就麻煩了
4. **備份策略**：Traefik acme.json、Bunker Web 設定這些，還是備份一下比較好
5. **監控告警**：設定飛書通知，及時瞭解部署狀態，出問題也能快速反應

## 總結

雲伺服器 + Nginx 快取層，就這麼簡單。HagiCode 用這套方案，成本不算高（伺服器費用大概 60-100 元/月），效果還挺不錯的。核心優勢大概也就這些：

- **成本可控**：比直接用雲存儲或商業 CDN，成本降了大概 50%
- **部署靈活**：Traefik 還是 Bunker Web，看你自己選
- **擴充性強**：需要的話可以水平擴充，或者再加個 CDN
- **運維簡單**：Shell 腳本 + Ansible，自動化部署也方便

對於需要檔案分發的小團隊和個人開發人員來說，這方案倒是可以試試。

HagiCode 用這套架構在生產環境穩定執行了一段時間，全球使用者下載也沒出什麼大問題。如果你也在找類似的解決方案，不妨試試看，說不定對你也有說明。
