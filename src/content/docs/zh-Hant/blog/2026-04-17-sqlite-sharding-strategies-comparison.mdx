## SQLite 分片方案實戰：三種分片策略的深度對比

> 當單檔案 SQLite 遇到並行瓶頸，我們該如何破局？本文分享 HagiCode 專案中三種不同場景下的 SQLite 分片方案，幫你理解如何選擇合適的分片策略。

全民製作人們大家好，我是 HagiCode 製作人俞坤。

## 背景

在建置高效能應用時，單檔案 SQLite 資料庫會碰到很現實的問題。使用者量和資料量一上來，這些狀況就會排隊找上門：

- 寫入操作開始排隊，回應時間肉眼可見地變長
- 查詢效能隨資料增長往下掉
- 多執行緒存取時頻繁出現 "database is locked" 錯誤

很多人第一反應是：要不要直接遷移到 PostgreSQL 或者 MySQL？這波操作雖然能解決問題，但部署複雜度會直線上升。有沒有更輕量的方案？

答案是：分片。說到底，工程問題還是要回到工程方法裏解決，透過將資料分散到多個 SQLite 檔案，可以顯著提升並行能力和查詢效能，同時保持 SQLite 的輕量級特性。

## 關於 HagiCode

本文分享的方案來自我們在 [HagiCode](https://hagicode.com) 專案中的實踐經驗。作爲一個 AI 程式碼助手專案，HagiCode 需要處理大量的對話訊息、狀態永續性和事件瀏覽記錄。正是在解決這些實際問題的程序中，我們總結出了三種不同場景下的分片方案。

工欲善其事，必先利其器，但這些"器"怎麼用，還得看具體的"事"是什麼。

我們的程式碼存放庫在 [github.com/HagiCode-org/site](https://github.com/HagiCode-org/site)，歡迎感興趣的朋友深入瞭解。

## 三種分片方案概覽

經過對 HagiCode 程式碼程式庫的分析，我們發現了三種針對不同業務場景的 SQLite 分片方案：

1. **Session Message 分片存儲**：AI 對話訊息存儲，特點是高頻寫入、基於 Session 的隔離查詢
2. **Orleans Grain 分片存儲**：分佈式框架狀態永續性，特點是跨節點存取、需要確定性路由
3. **Hero History 分片存儲**：遊戲化系統歷史事件記錄，特點是事件溯源、需要遷移相容

雖然業務場景不同，但三者都遵循相同的核心設計原則：

- **確定性路由**：直接從業務 ID 計算分片，無需元資料表
- **透明存取**：上層透過統一介面操作，不感知分片存在
- **獨立存儲**：每個分片是完全獨立的 SQLite 檔案
- **並行優化**：WAL 模式 + busy_timeout 降低鎖定競爭

很多人會問：爲什麼不搞一套通用的分片方案？這個問題問得很實在，我們直接上結論：工程上沒有萬能方案，只有最貼合當前業務場景的方案。接下來我們深入對比這三種方案的具體實作。

## 分片策略對比

### 分片數量與命名規則

| 方面 | Session Message | Orleans Grain | Hero History |
|------|----------------|---------------|--------------|
| **分片數量** | 256 (16²) | 100 | 10 |
| **命名規則** | 16 進制 (00-ff) | 10 進制 (00-99) | 10 進制 (0-9) |
| **存儲目錄** | `DataDir/messages/` | `DataDir/orleans/grains/` | `DataDir/hero-history/` |
| **檔案名模式** | `{shard}.db` | `grains-{shard}.db` | `{shard}.db` |

爲什麼分片數量差異這麼大？這取決於業務特點。換句話說，模型會說，工具會變，工作流會升級，但工程上的基本盤一直都在那裏：你得先搞清楚自己要解決什麼問題。

- **Session Message** 使用 256 個分片，因爲對話訊息的寫入頻率最高，需要更多的分片來分散負載
- **Orleans Grain** 使用 100 個分片，平衡了並行效能和管理複雜度
- **Hero History** 只用 10 個分片，因爲歷史事件寫入頻率較低，且需要考慮遷移成本

### 路由演算法差異

路由演算法是分片方案的核心，決定了資料如何分佈到各個分片。三種方案使用了不同的路由策略：

```csharp
// Session Message: GUID 后两位 16 进制
var normalized = Guid.Parse(sessionId.Value).ToString("N").ToLowerInvariant();
return normalized[^2..];  // 取末两位 16 进制字符

// Orleans Grain: 提取数字后两位取模
var digits = ExtractDigits(grainId);  // 提取所有数字
var lastTwoDigits = (digits[^2] * 10) + digits[^1];
return lastTwoDigits % shardCount;

// Hero History: 末位字符 ASCII 值取模
return heroId[^1] % 10;
```

**設計思路剖析**：

- **Session Message** 的 ID 是 GUID，轉換爲 16 進制後取末兩位，可以得到均勻分佈的 256 個分片
- **Orleans Grain** 的 ID 格式不統一，可能包含字母和數字，所以提取所有數字後取模
- **Hero History** 的 ID 是字串，直接用末位字符的 ASCII 值取模，簡單但分佈可能不夠均勻

**關鍵點**：無論使用哪種演算法，都必須保證同一 ID 永遠映射到同一分片。這是分佈式系統中最基本的要求，否則會導致資料不一致。說到底，路由不穩定，一切努力都是零。

### 初始化策略差異

| 方面 | Session Message | Orleans Grain | Hero History |
|------|----------------|---------------|--------------|
| **初始化時機** | 按需懶載入 | 啓動時全量平行初始化 | 按需懶載入 |
| **並行控制** | `Lazy<Task>` 防重複初始化 | `Parallel.ForEachAsync` | `Lazy<Task>` 防重複初始化 |

**爲什麼 Orleans Grain 選擇啓動時全量初始化？**

因爲 Orleans 是分佈式框架，Grain 可能被調度到任意節點。如果在執行階段才發現分片檔案不存在，會導致請求失敗。啓動時全量初始化雖然會延長啓動時間，但能確保執行階段的穩定性。能跑起來只是開始，能維護下去纔算本事。

**懶載入的優勢**：

對於 Session Message 和 Hero History，使用懶載入可以減少啓動時間，只有在真正需要存取某個分片時才建立檔案和初始化 Schema。使用 `Lazy<Task>` 可以防止並行初始化時的競爭條件。這個設計看着簡單，但在真實專案裏能省掉很多不必要的麻煩。

### Schema 設計特點

三種方案的 Schema 設計反映了各自的業務特點：

**Session Message**：
- 支援 Event Sourcing 模式（事件表 + 快照表）
- 包含訊息內容塊子表（MessageContentBlocks）
- 具有壓縮和壓縮標記字段，支援後續優化

**Orleans Grain**：
- 最簡設計：單表 GrainState
- JSON 序列化存儲狀態
- ETag 樂觀並行控制

**Hero History**：
- 時間線查詢優化索引
- DedupeKey 唯一約束防重複
- 支援多種事件型別和狀態

從這些設計中可以看出，Schema 設計應該緊密貼合業務需求，而不是追求通用性。Orleans Grain 的簡單設計正是因爲它只需要存儲序列化後的狀態，不需要複雜的查詢能力。這波不是玄學，是工程。別急着把名字起得太大，先看看這東西能不能在團隊裏活過兩個疊代。

### 並行設定對比

三種方案都使用了相同的 SQLite 並行優化設定：

```sql
PRAGMA journal_mode=WAL;      -- 写前日志模式
PRAGMA synchronous=NORMAL;     -- 降低持久化开销
PRAGMA busy_timeout=5000;      -- 5秒忙等待
PRAGMA foreign_keys=ON;        -- 外键约束
```

**WAL 模式的優勢**：

傳統的回復日誌模式在寫入時會產生鎖定競爭，而 WAL 模式允許讀寫並行進行。這在大資料量場景下可以顯著提升效能。很多人不知道這個設定，其實它比你想的要重要得多。

**synchronous=NORMAL 的權衡**：

設定爲 FULL 可以保證最高安全性，但會顯著降低效能。NORMAL 模式在安全性和效能之間取得了平衡，對於大多數應用來說是合適的選擇。這個設定不需要糾結太久，NORMAL 就夠了。

## 如何選擇分片策略

基於對 HagiCode 三種方案的分析，我們可以總結出以下決策矩陣：

```
高吞吐量场景 → 更多分片（如 Message 用 256）
简单维护性   → 较少分片（如 Hero History 用 10）
数字 ID 为主 → 取模算法（Orleans Grain）
GUID 为主    → 16 进制后缀（Session Message）
字符串 ID    → ASCII 取模（Hero History）
```

**分片數量選擇的經驗值**：

- 太少（< 10）：並行提升有限，分片意義不大
- 太多（> 1000）：檔案管理複雜，連接池開銷大
- 經驗值：10-100 個分片適用於大多數場景
- 極高並行場景：可以考慮 256 個分片

這事你要是隻看演示，確實容易上頭；可一旦進了生產環境，賬就得一筆一筆算清楚。很多問題不是不能做，只是沒把代價算明白。

## 實踐指南

### 實作標準化分片路由器

```csharp
public interface IShardResolver<TId>
{
    string ResolveShardKey(TId id);
}

// 16 进制分片（适用于 GUID）
public class HexSuffixShardResolver : IShardResolver<string>
{
    private readonly int _suffixLength;

    public HexSuffixShardResolver(int suffixLength = 2)
    {
        _suffixLength = suffixLength;
    }

    public string ResolveShardKey(string id)
    {
        var normalized = id.Replace("-", "").ToLowerInvariant();
        return normalized[^_suffixLength..];
    }
}

// 数字取模分片（适用于纯数字 ID）
public class NumericModuloShardResolver : IShardResolver<long>
{
    private readonly int _shardCount;

    public NumericModuloShardResolver(int shardCount)
    {
        _shardCount = shardCount;
    }

    public string ResolveShardKey(long id)
    {
        return (id % _shardCount).ToString("D2");
    }
}
```

### 統一連接工廠模式

```csharp
public class ShardedConnectionFactory<TOptions>
{
    private readonly ConcurrentDictionary<string, Lazy<Task>> _initializationTasks = new();
    private readonly TOptions _options;
    private readonly IShardSchemaInitializer _initializer;

    public ShardedConnectionFactory(
        TOptions options,
        IShardSchemaInitializer initializer)
    {
        _options = options;
        _initializer = initializer;
    }

    public async Task<TDbContext> CreateAsync(string shardKey, CancellationToken ct)
    {
        var connectionString = BuildConnectionString(shardKey);

        // 使用 Lazy<Task> 防止并发初始化
        var initTask = _initializationTasks.GetOrAdd(
            connectionString,
            _ => new Lazy<Task>(() => InitializeShardAsync(connectionString, ct))
        );

        await initTask.Value;
        return CreateDbContext(connectionString);
    }

    private async Task InitializeShardAsync(string connectionString, CancellationToken ct)
    {
        await _initializer.InitializeAsync(connectionString, ct);
    }

    private string BuildConnectionString(string shardKey)
    {
        var shardPath = Path.Combine(_options.BaseDirectory, $"{shardKey}.db");
        return $"Data Source={shardPath}";
    }

    private TDbContext CreateDbContext(string connectionString)
    {
        // 根据具体的 ORM 创建 DbContext
        return Activator.CreateInstance(typeof(TDbContext), connectionString) as TDbContext;
    }
}
```

### Schema 初始化最佳實踐

```csharp
public class SqliteShardInitializer : IShardSchemaInitializer
{
    public async Task InitializeAsync(string connectionString, CancellationToken ct)
    {
        await using var connection = new SqliteConnection(connectionString);
        await connection.OpenAsync(ct);

        // 并发优化配置
        await connection.ExecuteAsync("""
            PRAGMA journal_mode=WAL;
            PRAGMA synchronous=NORMAL;
            PRAGMA busy_timeout=5000;
            PRAGMA foreign_keys=ON;
        """);

        // 创建表结构
        await connection.ExecuteAsync("""
            CREATE TABLE IF NOT EXISTS Entities (
                Id TEXT PRIMARY KEY,
                CreatedAt TEXT NOT NULL,
                UpdatedAt TEXT NOT NULL,
                Data TEXT NOT NULL,
                ETag TEXT
            );
        """);

        // 创建索引
        await connection.ExecuteAsync("""
            CREATE INDEX IF NOT EXISTS IX_Entities_CreatedAt
            ON Entities(CreatedAt DESC);

            CREATE INDEX IF NOT EXISTS IX_Entities_UpdatedAt
            ON Entities(UpdatedAt DESC);
        """);
    }
}
```

### 關鍵注意事項

**1. 路由穩定性**

路由演算法必須保證同一 ID 永遠映射到同一分片。避免使用隨機或時間相關的計算，也不要在演算法中引入可變參數。

**2. 分片數量選擇**

分片數量應該在設計階段確定，後期修改非常困難。需要考慮：
- 當前和未來的並行量
- 單個分片的管理成本
- 資料遷移的複雜度

**3. 遷移考慮**

Hero History 方案展示了完整的遷移路徑：

1. 新建分片存儲基礎設施
2. 實作遷移服務將主程式庫資料複製到分片
3. 驗證遷移後查詢相容性
4. 切換讀寫路徑到分片
5. 清理主程式庫舊錶

設計分片方案時就需要考慮未來的遷移需求。Talk is cheap. Show me the code，但光有程式碼還不夠，你還得有完整的遷移路徑。一次成功不叫體系，持續成功才叫體系。

**4. 監控與運維**

- 監控各分片的大小分佈，及時發現資料傾斜
- 設定告警檢測分片熱點，避免單個分片成爲瓶頸
- 定期檢查 WAL 檔案大小，防止磁盤空間佔用過多
- 建立分片健康檢查機制

**5. 測試覆寫**

- 測試邊界條件（空 ID、特殊字元、超長 ID）
- 驗證路由確定性，確保同一 ID 總是映射到同一分片
- 並行寫入壓力測試，驗證鎖定競爭得到有效緩解
- 遷移測試，確保資料完整性和一致性

## 總結

透過對比 HagiCode 專案中的三種 SQLite 分片方案，我們可以看到：

1. **沒有萬能的解決方案**：不同業務場景需要不同的分片策略
2. **核心原則是通用的**：確定性路由、透明存取、獨立存儲、並行優化
3. **設計要面向未來**：考慮遷移路徑和運維成本

如果你的專案正在使用 SQLite，並且開始遇到並行瓶頸，希望這篇文章能爲你提供一些思路。不需要急着遷移到重量級資料庫，有時候合適的分片方案就能解決問題。

當然，分片不是銀彈。在選擇分片方案之前，先確保：
- 你已經優化了單表查詢效能
- 你已經使用了合適的索引
- 你已經啓用了 WAL 模式

只有在這些優化都做完之後，仍然存在效能瓶頸時，才考慮引入分片。你能把簡單的事情做好，這本身就是一種能力。
