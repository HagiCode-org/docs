## 如何用 GitHub Actions 實作 Steam 自動化發佈

> 本文分享了 HagiCode Desktop 專案中實作 Steam 自動化發佈的完整方案，從 GitHub Release 到 Steam 平台的全鏈路自動化流程，包括 Steam Guard 認證、多平台 Depot 上傳等關鍵技術細節。

## 背景

Steam 平台的發佈流程，其實和傳統的應用分發方式挺不一樣的。Steam 有自己的一套完整更新分發系統，開發人員得透過 SteamCMD 工具把建置產物上傳到 Steam 的 CDN 網路，而不是像其他平台那樣直接丟個下載連結就完事了。

HagiCode Desktop 專案計劃上架 Steam 平台，這也算是給我們的發佈流程帶來了點新挑戰：

1. 需要把現有的建置產物轉換成 Steam 相容的格式
2. 得透過 SteamCMD 工具上傳到 Steam 平台
3. 還必須處理 Steam Guard 認證這玩意兒
4. 需要支援多平台（Linux、Windows、macOS）的 Depot 上傳
5. 還要實作從 GitHub Release 到 Steam 的自動化流轉

專案此前已經實作了"便攜版模式"（portable version mode），允許應用檢測打包在 extra 目錄中的固定服務載荷。我們的目標，其實就是讓這套便攜版模式和 Steam 分發能無縫集成罷了。

## 關於 HagiCode

本文分享的方案，來自我們在 [HagiCode](https://hagicode.com) 專案中的實踐經驗。HagiCode 是一個 AI 程式碼助手專案，支援桌面端執行，我們正在推進 Steam 平台的上架工作，因此才需要建立一套可靠的自動化發佈流程。

## 架構設計

整個 Steam 發佈流程的核心是一個 GitHub Actions 工作流，它把整個程序分爲三個主要階段：

```
┌─────────────────────────────────────────────────────────────┐
│ GitHub Actions Workflow (Steam Release)                      │
├─────────────────────────────────────────────────────────────┤
│ 1. 准备阶段:                                                 │
│    - 检出 portable-version 代码                             │
│    - 从 GitHub Release 下载构建产物                         │
│    - 解压并准备 Steam 内容目录                              │
│                                                             │
│ 2. SteamCMD 设置:                                           │
│    - 安装/复用 SteamCMD                                     │
│    - 使用 Steam Guard 进行认证                              │
│                                                             │
│ 3. 发布阶段:                                                 │
│    - 生成 Depot VDF 配置文件                                │
│    - 生成 App Build VDF 配置文件                            │
│    - 调用 SteamCMD 上传到 Steam                             │
└─────────────────────────────────────────────────────────────┘
```

這種設計的優勢，怎麼說呢：
- 複用現有的 GitHub Release 產物，避免重複建置，畢竟誰願意重複勞動呢
- 透過自代管執行器實作安全隔離，多一層保障總是好的
- 支援預覽模式和正式發佈分支切換，靈活一點
- 完整的錯誤處理和日誌記錄，出問題的時候不至於太迷茫

## 工作流實作

### 觸發參數設計

我們的工作流支援以下關鍵參數：

```yaml
inputs:
  release:           # Portable Version 发布标签
    description: '要发布的版本标签（如 v1.0.0）'
    required: true
  steam_preview:     # 是否生成预览构建
    description: '是否为预览模式'
    required: false
    default: 'false'
  steam_branch:      # 设置为 live 的 Steam 分支
    description: '目标 Steam 分支'
    required: false
    default: 'preview'
  steam_description: # 构建描述覆盖
    description: '构建描述'
    required: false
```

### 自代管執行器設定

出於安全考慮，我們使用帶 steam 標籤的自代管執行器：

```yaml
runs-on:
  - self-hosted
  - Linux
  - X64
  - steam
```

這樣可以確保 Steam 發佈在專用執行器上執行，保持敏感憑據的安全隔離。畢竟安全這事兒，多注意一點總是好的。

### 並行控制

爲了避免同一版本的發佈相互干擾，我們設定了並行控制：

```yaml
concurrency:
  group: portable-version-steam-${{ github.event.inputs.release }}
  cancel-in-progress: false
```

注意這裏設定 `cancel-in-progress: false`，因爲 Steam 發佈程序可能較長，我們也不想因爲新的觸發就取消正在進行的發佈。畢竟發佈個版本也不容易，總得讓人家跑完不是？

## 核心腳本實作

### 準備發佈輸入

`prepare-steam-release-input.mjs` 腳本負責準備發佈所需的輸入：

```javascript
// 下载 GitHub Release 的构建清单和产物清单
const buildManifest = await downloadBuildManifest(releaseTag);
const artifactInventory = await downloadArtifactInventory(releaseTag);

// 下载各平台的压缩包
for (const platform of ['linux-x64', 'win-x64', 'osx-universal']) {
  const artifactUrl = getArtifactUrl(artifactInventory, platform);
  await downloadArtifact(artifactUrl, platform);
}

// 解压到 Steam 内容目录结构
await extractToSteamContent(sources, contentRoot);
```

### Steam Guard 認證

Steam 要求使用 Steam Guard 保護賬戶，我們實作了基於共享密鑰的程式碼產生演算法：

```javascript
function generateSteamGuardCode(sharedSecret, timestamp = Date.now()) {
  const secret = decodeSharedSecret(sharedSecret);
  const time = Math.floor(timestamp / 1000 / 30);

  const timeBuffer = Buffer.alloc(8);
  timeBuffer.writeBigUInt64BE(BigInt(time));

  // 使用 HMAC-SHA1 生成时间基础的一次性代码
  const hash = crypto.createHmac('sha1', secret)
    .update(timeBuffer)
    .digest();

  // 转换为 5 字符的 Steam Guard 代码
  const code = steamGuardCode(hash);
  return code;
}
```

這個實作基於 Steam Guard 的 TOTP（Time-based One-Time Password）機制，每 30 秒產生一個新的驗證碼。畢竟安全這東西，還是得用靠譜的方式纔行。

### VDF 設定產生

VDF（Valve Data Format）是 Steam 使用的設定格式，我們需要產生兩種型別的 VDF 檔案：

**Depot VDF** 用於設定各個平台的內容：

```javascript
function buildDepotVdf(depotId, contentRoot) {
  return [
    '"DepotBuildConfig"',
    '{',
    `  "DepotID" "${escapeVdf(depotId)}"`,
    `  "ContentRoot" "${escapeVdf(contentRoot)}"`,
    '  "FileMapping"',
    '  {',
    '    "LocalPath" "*"',
    '    "DepotPath" "."',
    '    "recursive" "1"',
    '  }',
    '}'
  ].join('\n');
}
```

**App Build VDF** 用於設定整個應用建置：

```javascript
function buildAppBuildVdf(appId, depotBuilds, description, setLive) {
  const vdf = [
    '"appbuild"',
    '{',
    `  "appid" "${appId}"`,
    `  "desc" "${escapeVdf(description)}"`,
    `  "contentroot" "${escapeVdf(contentRoot)}"`,
    '  "buildoutput" "build_output"',
    '  "depots"',
    '  {'
  ];

  for (const [depotId, depotVdfPath] of Object.entries(depotBuilds)) {
    vdf.push(`    "${depotId}" "${depotVdfPath}"`);
  }

  if (setLive) {
    vdf.push(`  }`);
    vdf.push(`  "setlive" "${setLive}"`);
  }

  vdf.push('}');
  return vdf.join('\n');
}
```

### SteamCMD 呼叫

最後，透過呼叫 SteamCMD 執行上傳：

```javascript
await runCommand(steamcmdPath, [
  '+login', steamUsername, steamPassword, steamGuardCode,
  '+run_app_build', appBuildPath,
  '+quit'
]);
```

這一步算是整個流程的最後一躍，跨過去就完成了......

## 多平台 Depot 處理

Steam 使用 Depot 系統管理不同平台的內容，我們支援三種主要的 Depot：

| 平台 | Depot 標識 | 架構支援 |
|-----|-----------|---------|
| Linux | `linux-x64` | x64_64 |
| Windows | `win-x64` | x64_64 |
| macOS | `osx-universal` | universal, x64_64, arm64 |

每個 Depot 都有獨立的內容目錄和 VDF 設定檔案，這樣可以確保不同平台的使用者只下載自己需要的內容。畢竟流量也是錢，能省一點是一點。

## 發佈流程

### 步驟 1：準備 GitHub Release

首先需要在 portable-version 存放庫建立一個 GitHub Release，包含：
- 各平台的壓縮包
- 建置清單（`{tag}.build-manifest.json`）
- 產物清單（`{tag}.artifact-inventory.json`）

### 步驟 2：觸發 Steam 發佈工作流

透過 GitHub Actions 手動觸發工作流，填寫必要參數：
- `release`: 要發佈的版本標籤（如 v1.0.0）
- `steam_branch`: 目標分支（如 `preview` 或 `public`）
- `steam_preview`: 是否預覽模式

### 步驟 3：自動執行發佈流程

工作流會自動執行以下步驟：

1. 下載並解壓 GitHub Release 產物
2. 安裝/更新 SteamCMD
3. 產生 Steam VDF 設定檔案
4. 使用 Steam Guard 認證
5. 上傳內容到 Steam CDN
6. 設定指定分支爲 live

這一套流程走下來，也算是把該做的都做了。

## 設定指南

### 必需的 Secrets 設定

在 GitHub 存放庫設定中設定以下密鑰：

| Secret 名稱 | 說明 |
|-----------|------|
| `STEAM_USERNAME` | Steam 賬戶使用者名 |
| `STEAM_PASSWORD` | Steam 賬戶密碼 |
| `STEAM_SHARED_SECRET` | Steam Guard 共享密鑰（可選） |
| `STEAM_GUARD_CODE` | Steam Guard 程式碼（可選） |
| `STEAM_APP_ID` | Steam 應用 ID |
| `STEAM_DEPOT_ID_LINUX` | Linux Depot ID |
| `STEAM_DEPOT_ID_WINDOWS` | Windows Depot ID |
| `STEAM_DEPOT_ID_MACOS` | macOS Depot ID |

這些設定項，其實也沒什麼特別的，就是該有的都得有罷了。

### 環境變數設定

| 變數名稱 | 說明 | 預設值 |
|---------|------|--------|
| `PORTABLE_VERSION_STEAMCMD_ROOT` | SteamCMD 安裝目錄 | `~/.local/share/portable-version/steamcmd` |

## 最佳實踐

### Steam Guard 認證管理

首次執行需要手動輸入 Steam Guard 程式碼，之後建議設定共享密鑰自動產生程式碼。這樣可以避免每次發佈都需要手動干預，畢竟誰也不想每次都重複同樣的操作。

SteamCMD 會保存登入令牌，後續可以複用。但要注意令牌的有效期，過期後還是得重新認證的，這也沒辦法。

### 內容目錄結構

確保 Steam 內容目錄結構正確：

```
steam-content/
├── linux-x64/     # Linux 平台内容
├── win-x64/       # Windows 平台内容
└── osx-universal/ # macOS 通用二进制内容
```

每個目錄下應該包含對應平台的完整應用檔案。這點倒也沒什麼好說的，該怎麼做就怎麼做。

### 預覽模式使用

預覽模式不會設定任何分支爲 live，適合測試驗證：

```yaml
if [ "$STEAM_PREVIEW_INPUT" = 'true' ]; then
  cmd+=(--preview)
fi
```

這樣可以先上傳到 Steam 平台進行驗證，確認無誤後再切換到正式分支。多一層驗證，總是好的。

### 錯誤處理和日誌

腳本包含了完善的錯誤處理和日誌記錄：
- 驗證 GitHub Release 存在性
- 檢查必需的元資料檔案
- 確保平台內容存在
- 產生 GitHub Actions 摘要報告

這些資訊對於調試和審計都非常有價值，畢竟出問題的時候能有個線索，總比一頭霧水要好。

### 產物管理

工作流產生兩種產物：
- `portable-steam-release-preparation-{tag}`: 發佈準備元資料
- `portable-steam-build-metadata-{tag}`: Steam 建置元資料

這些產物可以用於後續的審計和調試，保存時間建議設定爲 30 天。反正也不佔多少地方，留着也無妨。

這大大提高了我們的發佈效率和可靠性。之前手動發佈一個版本需要 30 分鐘以上的時間，現在只需要幾分鐘就能完成整個流程。時間這東西，省下來總歸是好的。
