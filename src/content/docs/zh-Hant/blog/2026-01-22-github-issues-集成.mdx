## 從零構建 GitHub Issues 整合：HagiCode 的前端直連實踐

> 本文記錄了在 HagiCode 平台中整合 GitHub Issues 的全過程。我們將探討如何透過「前端直連 + 後端最小化」的架構，在保持後端輕量的同時，實現安全的 OAuth 認證與高效的 Issues 同步。

## 背景：為什麼要整合 GitHub？

HagiCode 作為一個 AI 輔助開發平台，核心價值在於連接想法與實現。但在實際使用中，我們發現用戶在 HagiCode 中完成了 Proposal（提案）後，往往需要手動將內容複製到 GitHub Issues 中進行專案追蹤。

這帶來了幾個明顯的痛點：

1.  **工作流割裂**：用戶需要在兩個系統之間來回切換，體驗不僅不流暢，還容易導致關鍵資訊在複製貼上的過程中丟失。
2.  **協作不便**：團隊其他成員習慣在 GitHub 上查看任務，無法直接看到 HagiCode 中的提案進展。
3.  **重複勞動**：每當提案更新，就要人工去 GitHub 更新對應的 Issue，增加不必要的維護成本。

為了解決這個問題，我們決定引入 **GitHub Issues Integration** 功能，打通 HagiCode 會話與 GitHub 儲存庫的連接，實現「一鍵同步」。

## 技術選型：前端直連 vs 後端代理

在設計整合方案時，擺在我們面前的有兩條路：傳統的「後端代理模式」和更激進的「前端直連模式」。

### 方案對比

在傳統的**後端代理模式**中，前端所有的請求都要先經過我們的後端，再由後端去呼叫 GitHub API。這雖然邏輯集中，但給後端帶來了不小的負擔：

1.  **後端臃腫**：需要編寫專門的 GitHub API 客戶端封裝，還要處理 OAuth 的複雜狀態機。
2.  **Token 風險**：用戶的 GitHub Token 必須儲存在後端資料庫中，雖然可以加密，但畢竟增加了安全風險面。
3.  **開發成本**：需要資料庫遷移來儲存 Token，還需要維護一套額外的同步服務。

而**前端直連模式**則要輕量得多。在這個方案中，我們只利用後端來處理最敏感的「金鑰交換」環節（OAuth callback），取得 Token 後，直接存在瀏覽器的 localStorage 裡。後續建立 Issue、更新評論等操作，直接由前端發 HTTP 請求到 GitHub。

| 對比維度 | 後端代理模式 | 前端直連模式 |
| :--- | :--- | :--- |
| **後端複雜度** | 需要完整的 OAuth 服務和 GitHub API 客戶端 | 僅需一個 OAuth 回呼端點 |
| **Token 管理** | 需加密儲存在資料庫，有洩露風險 | 儲存在瀏覽器，僅用戶自己可見 |
| **實施成本** | 需資料庫遷移、多服務開發 | 主要是前端工作量 |
| **用戶體驗** | 邏輯統一，但伺服器延遲可能稍高 | 回應極快，直接與 GitHub 互動 |

考慮到我們要的是快速整合和最小化後端改動，**最終我們採用了「前端直連模式」**。這就像給瀏覽器發了一張「臨時通行證」，拿到證之後，瀏覽器就可以自己去 GitHub 辦事了，不需要每次都找後端管理員批准。

## 核心設計：資料流與安全

在確定架構後，我們需要設計具體的資料流。整個同步流程的核心在於如何安全地取得 Token 並高效地利用它。

### 整體架構圖

整個系統可以抽象為三個角色：瀏覽器（前端）、HagiCode 後端、GitHub。

```text
+--------------+        +--------------+        +--------------+
|  前端 React  |        |    後端      |        |    GitHub    |
|              |        |   ASP.NET    |        |    REST API  |
|  +--------+  |        |              |        |              |
|  |  OAuth |--+--------> /callback    |        |              |
|  |  流程  |  |        |              |        |              |
|  +--------+  |        |              |        |              |
|              |        |              |        |              |
|  +--------+  |        |  +--------+  |        |  +--------+  |
|  |GitHub  |  +------------>Session |  +----------> Issues |  |
|  |API     |  |        |  |Metadata|  |        |  |        |  |
|  |直連    |  |        |  +--------+  |        |  +--------+  |
|  +--------+  |        |              |        |              |
+--------------+        +--------------+        +--------------+
```

**關鍵點在於**：只有 OAuth 的一小步（取得 code 換 token）需要經過後端，之後的粗活累活（建立 Issue）都是前端直接跟 GitHub 打交道。

### 同步資料流詳解

當用戶點擊 HagiCode 介面上的「Sync to GitHub」按鈕時，會發生一系列複雜的動作：

```text
用戶點擊 "Sync to GitHub"
         │
         ▼
1. 前端檢查 localStorage 取得 GitHub Token
         │
         ▼
2. 格式化 Issue 內容（將 Proposal 轉換為 Markdown）
         │
         ▼
3. 前端直接呼叫 GitHub API 建立/更新 Issue
         │
         ▼
4. 呼叫 HagiCode 後端 API 更新 Session.metadata (儲存 Issue URL 等資訊)
         │
         ▼
5. 後端透過 SignalR 廣播 SessionUpdated 事件
         │
         ▼
6. 前端接收事件，更新 UI 顯示「已同步」狀態
```

### 安全設計

安全問題始終是整合第三方服務的重中之重。我們做了以下考量：

1.  **防 CSRF 攻擊**：在 OAuth 跳轉時，產生隨機的 `state` 參數並存入 sessionStorage。回呼時嚴格驗證 state，防止請求被偽造。
2.  **Token 儲存隔離**：Token 僅儲存在瀏覽器的 `localStorage` 中，利用同源策略（Same-Origin Policy），只有 HagiCode 的腳本才能讀取，避免了伺服器端資料庫洩露波及用戶。
3.  **錯誤邊界**：針對 GitHub API 常見的錯誤（如 401 Token 過期、422 驗證失敗、429 速率限制），設計了專門的錯誤處理邏輯，給用戶以友善的提示。

### 2. 前端 OAuth 流程

這是連接的入口。我們使用標準的 Authorization Code Flow。

```typescript
// src/HagiCode.Client/src/services/githubOAuth.ts

// 產生授權 URL 並跳轉
export async function generateAuthUrl(): Promise<string> {
  const state = generateRandomString(); // 產生防 CSRF 的隨機串
  sessionStorage.setItem('hagicode_github_state', state);
  
  const params = new URLSearchParams({
    client_id: clientId,
    redirect_uri: window.location.origin + '/settings?tab=github&oauth=callback',
    scope: ['repo', 'public_repo'].join(' '),
    state: state,
  });
  
  return `https://github.com/login/oauth/authorize?${params.toString()}`;
}

// 在回呼頁面處理 Code 換取 Token
export async function exchangeCodeForToken(code: string, state: string): Promise<GitHubToken> {
  // 1. 驗證 State 防止 CSRF
  const savedState = sessionStorage.getItem('hagicode_github_state');
  if (state !== savedState) throw new Error('Invalid state parameter');

  // 2. 呼叫後端 API 進行 Token 交換
  // 注意：這裡必須經過後端，因為需要 ClientSecret，不能暴露在前端
  const response = await fetch('/api/GitHubOAuth/callback', {
    method: 'POST',
    headers: { 'Content-Type': 'application/json' },
    body: JSON.stringify({ code, state, redirectUri: window.location.origin + '/settings?tab=github&oauth=callback' }),
  });

  if (!response.ok) throw new Error('Failed to exchange token');
  
  const token = await response.json();
  
  // 3. 存入 LocalStorage
  saveToken(token);
  return token;
}
```

### 3. GitHub API 客戶端封裝

有了 Token 之後，我們就需要一個強有力的工具來呼叫 GitHub API。

```typescript
// src/HagiCode.Client/src/services/githubApiClient.ts

const GITHUB_API_BASE = 'https://api.github.com';

// 核心請求封裝
async function githubApi<T>(endpoint: string, options: RequestInit = {}): Promise<T> {
  const token = localStorage.getItem('gh_token');
  if (!token) throw new Error('Not connected to GitHub');
  
  const response = await fetch(`${GITHUB_API_BASE}${endpoint}`, {
    ...options,
    headers: {
      ...options.headers,
      Authorization: `Bearer ${token}`,
      Accept: 'application/vnd.github.v3+json', // 指定 API 版本
    },
  });
  
  // 錯誤處理邏輯
  if (!response.ok) {
    if (response.status === 401) throw new Error('GitHub Token 失效，請重新連接');
    if (response.status === 403) throw new Error('無權存取該儲存庫或超出速率限制');
    if (response.status === 422) throw new Error('Issue 驗證失敗，可能標題重複');
    throw new Error(`GitHub API Error: ${response.statusText}`);
  }
  
  return response.json();
}

// 建立 Issue
export async function createIssue(owner: string, repo: string, data: { title: string, body: string, labels: string[] }) {
  return githubApi(`/repos/${owner}/${repo}/issues`, {
    method: 'POST',
    body: JSON.stringify(data),
  });
}
```

### 4. 內容格式化與同步

最後一步，就是把 HagiCode 的 Session 資料轉換成 GitHub Issue 的格式。這有點像「翻譯」工作。

```typescript
// 將 Session 物件轉換為 Markdown 字串
function formatIssueForSession(session: Session): string {
  let content = `# ${session.title}\n\n`;
  content += `**> HagiCode Session:** #${session.code}\n`;
  content += `**> Status:** ${session.status}\n\n`;
  content += `## Description\n\n${session.description || 'No description provided.'}\n\n`;
  
  // 如果是 Proposal 類型，新增額外欄位
  if (session.type === 'proposal') {
    content += `## Chief Complaint\n\n${session.chiefComplaint || ''}\n\n`;
    // 新增一個深層連結，方便從 GitHub 跳回 HagiCode
    content += `---\n\n**[View in HagiCode](hagicode://sessions/${session.id})**\n`;
  }
  
  return content;
}

// 點擊同步按鈕的主邏輯
const handleSync = async (session: Session) => {
  try {
    const repoInfo = parseRepositoryFromUrl(session.repoUrl); // 解析儲存庫 URL
    if (!repoInfo) throw new Error('Invalid repository URL');

    toast.loading('正在同步到 GitHub...');
    
    // 1. 格式化內容
    const issueBody = formatIssueForSession(session);
    
    // 2. 呼叫 API
    const issue = await githubApiClient.createIssue(repoInfo.owner, repoInfo.repo, {
      title: `[HagiCode] ${session.title}`,
      body: issueBody,
      labels: ['hagicode', 'proposal', `status:${session.status}`],
    });
    
    // 3. 更新 Session Metadata (儲存 Issue 連結)
    await SessionsService.patchApiSessionsSessionId(session.id, {
      metadata: {
        githubIssue: {
          owner: repoInfo.owner,
          repo: repoInfo.repo,
          issueNumber: issue.number,
          issueUrl: issue.html_url,
          syncedAt: new Date().toISOString(),
        }
      }
    });

    toast.success('同步成功！');
  } catch (err) {
    console.error(err);
    toast.error('同步失敗，請檢查 Token 或網路');
  }
};
```

## 總結與展望

透過這套「前端直連」方案，我們用最少的後端程式碼實現了 GitHub Issues 的無縫整合。

### 收穫
1.  **開發效率高**：後端改動極小，主要是資料庫加一個欄位和一個簡單的 OAuth 回呼介面，大部分邏輯都在前端完成。
2.  **安全性好**：Token 不經過伺服器資料庫，降低了洩露風險。
3.  **用戶體驗佳**：直接從前端發起請求，回應速度快，不需要經過後端中轉。

### 注意事項
在實際部署時，有幾個坑大家要注意：
- **OAuth App 設定**：記得在 GitHub OAuth App 設定裡填正確的 `Authorization callback URL`（通常是 `http://localhost:3000/settings?tab=github&oauth=callback`）。
- **速率限制**：GitHub API 對未認證請求限制較嚴，但用 Token 後通常足夠（5000次/小時）。
- **URL 解析**：用戶輸入的 Repo URL 千奇百怪，記得正則要匹配 `.git` 後綴、SSH 格式等情況。

### 後續增強
目前的功能還是單向同步（HagiCode -> GitHub）。未來我們計劃透過 GitHub Webhooks 實現雙向同步，比如在 GitHub 裡關閉 Issue，HagiCode 這邊的會話狀態也能自動更新。這需要我們在後端暴露一個 Webhook 接收端點，這也是下一步要做的有趣工作。
