## StreamJsonRpc 在 HagiCode 中的深度集成與實踐

> 本文詳細介紹了 HagiCode（原 PCode）專案如何成功集成 Microsoft 的 StreamJsonRpc 通信程式庫，以取代原有的自定義 JSON-RPC 實作，並解決了集成程序中的技術痛點與架構挑戰。

## 背景

StreamJsonRpc 是微軟官方維護的用於 .NET 和 TypeScript 的 JSON-RPC 通信程式庫，以其強大的型別安全、自動代理產生和成熟的例外處理機制著稱。在 HagiCode 專案中，爲了透過 ACP (Agent Communication Protocol) 與外部 AI 工具（如 iflow CLI、OpenCode CLI）進行通信，並消除早期自定義 JSON-RPC 實作帶來的維護成本和潛在 Bug，專案決定集成 StreamJsonRpc。然而，在集成程序中遇到了流式 JSON-RPC 特有的挑戰，特別是在處理代理目標綁定和泛型參數識別時。

爲了解決這些痛點，我們做了一個大膽的決定：整個建置系統推倒重來。這個決定帶來的變化，可能比你想象的還要大——稍後我會具體說。

## 分析

當前專案處於 ACP 通訊協定集成的關鍵階段，面臨着以下幾個技術痛點和架構挑戰：

### 1. 自定義實作的侷限
原有的 JSON-RPC 實作位於 `src/HagiCode.ClaudeHelper/AcpImp/`，包含 `JsonRpcEndpoint` 和 `ClientSideConnection` 等元件。維護這套自定義程式碼成本高，且缺乏成熟程式庫的高級功能（如進度報告、取消支援）。

### 2. StreamJsonRpc 集成障礙
在嘗試將現有的 `CallbackProxyTarget` 模式遷移到 StreamJsonRpc 時，發現 `_rpc.AddLocalRpcTarget(target)` 方法無法識別透過代理模式建立的目標。具體表現爲，StreamJsonRpc 無法自動將泛型型別 `T` 的屬性拆分爲 RPC 方法參數，導致伺服器端無法正確處理用戶端發起的方法呼叫。

### 3. 架構分層混亂
現有的 `ClientSideConnection` 混合了傳輸層（WebSocket/Stdio）、通訊協定層（JSON-RPC）和業務層（ACP Agent 介面），導致職責不清，且存在 `AcpAgentCallbackRpcAdapter` 方法綁定缺失的問題。

### 4. 日誌缺失
WebSocket 傳輸層缺少對原始 JSON 內容的日誌輸出，導致在調試 RPC 通信問題時難以定位是序列化問題還是網路問題。

## 解決

針對上述問題，我們採用了以下系統化的解決方案，從架構重構、程式庫集成和調試增強三個維度進行優化：

### 1. 全面遷移至 StreamJsonRpc

#### 移除舊程式碼
刪除 `JsonRpcEndpoint.cs`、`AgentSideConnection.cs` 及相關的自定義序列化轉換器（`JsonRpcMessageJsonConverter` 等）。

#### 集成官方程式庫
引入 `StreamJsonRpc` NuGet 包，利用其 `JsonRpc` 類處理核心通信邏輯。

#### 抽象傳輸層
定義 `IAcpTransport` 介面，統一處理 `WebSocket` 和 `Stdio` 兩種傳輸模式，確保通訊協定層與傳輸層解耦。

```csharp
// IAcpTransport 接口定义
public interface IAcpTransport
{
    Task SendAsync(string message, CancellationToken cancellationToken = default);
    Task<string> ReceiveAsync(CancellationToken cancellationToken = default);
    Task CloseAsync(CancellationToken cancellationToken = default);
}

// WebSocket 传输实现
public class WebSocketTransport : IAcpTransport
{
    private readonly WebSocket _webSocket;

    public WebSocketTransport(WebSocket webSocket)
    {
        _webSocket = webSocket;
    }

    // 实现发送和接收方法
    // ...
}

// Stdio 传输实现
public class StdioTransport : IAcpTransport
{
    private readonly StreamReader _reader;
    private readonly StreamWriter _writer;

    public StdioTransport(StreamReader reader, StreamWriter writer)
    {
        _reader = reader;
        _writer = writer;
    }

    // 实现发送和接收方法
    // ...
}
```

### 2. 修復代理目標識別問題

#### 分析 `CallbackProxyTarget`
檢查現有的動態代理產生邏輯，確定 StreamJsonRpc 無法識別的根本原因（通常是因爲代理物件沒有公開實際的方法簽名，或者使用了 StreamJsonRpc 不支援的參數型別）。

#### 重構參數傳遞
將泛型屬性拆分爲明確的 RPC 方法參數。不再相依動態屬性，而是定義具體的 Request/Response DTO（資料傳輸物件），確保 StreamJsonRpc 能透過反射正確識別方法簽名。

```csharp
// 原有的泛型属性方式
public class CallbackProxyTarget<T>
{
    public Func<T, Task> Callback { get; set; }
}

// 重构后的具体方法方式
public class ReadTextFileRequest
{
    public string FilePath { get; set; }
}

public class ReadTextFileResponse
{
    public string Content { get; set; }
}

public interface IAcpAgentCallback
{
    Task<ReadTextFileResponse> ReadTextFileAsync(ReadTextFileRequest request);
    // 其他方法...
}
```

#### 使用 `Attach` 替代 `AddLocalRpcTarget`
在某些複雜場景下，手動代理 `JsonRpc` 物件並處理 `RpcConnection` 可能比直接新增目標更靈活。

### 3. 實作方法綁定與日誌增強

#### 實作 `AcpAgentCallbackRpcAdapter`
確保該元件顯式實作 StreamJsonRpc 的代理介面，將 ACP 通訊協定定義的方法（如 `ReadTextFileAsync`）映射到 StreamJsonRpc 的回呼處理器上。

#### 集成日誌記錄
在 WebSocket 或 Stdio 的訊息處理管道中，攔截並記錄 JSON-RPC 請求和回應的原始文字。利用 `ILogger` 在剖析前和序列化後輸出原始 payload，以便排查格式錯誤。

```csharp
// 日志增强的传输包装器
public class LoggingAcpTransport : IAcpTransport
{
    private readonly IAcpTransport _innerTransport;
    private readonly ILogger<LoggingAcpTransport> _logger;

    public LoggingAcpTransport(IAcpTransport innerTransport, ILogger<LoggingAcpTransport> logger)
    {
        _innerTransport = innerTransport;
        _logger = logger;
    }

    public async Task SendAsync(string message, CancellationToken cancellationToken = default)
    {
        _logger.LogTrace("Sending message: {Message}", message);
        await _innerTransport.SendAsync(message, cancellationToken);
    }

    public async Task<string> ReceiveAsync(CancellationToken cancellationToken = default)
    {
        var message = await _innerTransport.ReceiveAsync(cancellationToken);
        _logger.LogTrace("Received message: {Message}", message);
        return message;
    }

    public async Task CloseAsync(CancellationToken cancellationToken = default)
    {
        _logger.LogDebug("Closing connection");
        await _innerTransport.CloseAsync(cancellationToken);
    }
}
```

### 4. 架構分層重構

#### 傳輸層 (`AcpRpcClient`)
封裝 StreamJsonRpc 連接，負責 `InvokeAsync` 和連接生命週期管理。

```csharp
public class AcpRpcClient : IDisposable
{
    private readonly JsonRpc _rpc;
    private readonly IAcpTransport _transport;

    public AcpRpcClient(IAcpTransport transport)
    {
        _transport = transport;
        _rpc = new JsonRpc(new StreamRpcTransport(transport));
        _rpc.StartListening();
    }

    public async Task<TResponse> InvokeAsync<TResponse>(string methodName, object parameters)
    {
        return await _rpc.InvokeAsync<TResponse>(methodName, parameters);
    }

    public void Dispose()
    {
        _rpc.Dispose();
        _transport.Dispose();
    }

    // StreamRpcTransport 是对 IAcpTransport 的 StreamJsonRpc 适配器
    private class StreamRpcTransport : IDuplexPipe
    {
        // 实现 IDuplexPipe 接口
        // ...
    }
}
```

#### 通訊協定層 (`IAcpAgentClient` / `IAcpAgentCallback`)
定義清晰的 client-to-agent 和 agent-to-client 介面，移除 `Func<IAcpAgent, IAcpClient>` 這種循環相依的工廠模式，改用相依注入或直接註冊回呼。

## 實踐

基於 StreamJsonRpc 的最佳實踐和專案經驗，以下是實施程序中的關鍵建議：

### 1. 強型別 DTO 優於動態物件
StreamJsonRpc 的核心優勢在於強型別。不要使用 `dynamic` 或 `JObject` 傳遞參數。應爲每個 RPC 方法定義明確的 C# POCO 類作爲參數。這不僅解決了代理目標識別問題，還能在編譯時期發現型別錯誤。

示例：將 `CallbackProxyTarget` 中的泛型屬性取代爲 `ReadTextFileRequest` 和 `WriteTextFileRequest` 等具體類。

### 2. 顯式宣告 Method Name
使用 `[JsonRpcMethod]` 特性顯式指定 RPC 方法名稱，不要相依預設的方法名映射。這可以防止因命名風格差異（如 PascalCase vs camelCase）導致的呼叫失敗。

```csharp
public interface IAcpAgentCallback
{
    [JsonRpcMethod("readTextFile")]
    Task<ReadTextFileResponse> ReadTextFileAsync(ReadTextFileRequest request);
    
    [JsonRpcMethod("writeTextFile")]
    Task WriteTextFileAsync(WriteTextFileRequest request);
}
```

### 3. 利用連接狀態回呼
StreamJsonRpc 提供了 `JsonRpc.ConnectionLost` 事件。務必監聽此事件以處理處理序意外退出或網路斷開的情況，這比單純相依 Orleans 的 Grain 失效檢測更及時。

```csharp
_rpc.ConnectionLost += (sender, e) =>
{
    _logger.LogError("RPC connection lost: {Reason}", e.ToString());
    // 处理重连逻辑或通知用户
};
```

### 4. 日誌分層記錄
- **Trace 級別**：記錄完整的 JSON Request/Response 原文。
- **Debug 級別**：記錄方法呼叫棧和參數摘要。
- **注意**：確保日誌中不包含敏感的 Authorization Token 或大檔案內容的 Base64 編碼。

### 5. 處理流式傳輸的特殊性
StreamJsonRpc 原生支援 `IAsyncEnumerable`。在實作 ACP 的流式 Prompt 回應時，應直接使用 `IAsyncEnumerable` 而不是自定義的分頁邏輯。這能極大簡化流式處理的程式碼量。

```csharp
public interface IAcpAgentCallback
{
    [JsonRpcMethod("streamText")]
    IAsyncEnumerable<string> StreamTextAsync(StreamTextRequest request);
}
```

### 6. 配接器模式 (Adapter Pattern)
保持 `ACPSession` 和 `ClientSideConnection` 的分離。`ACPSession` 應專注於 Orleans 的狀態管理和業務邏輯（如訊息入隊），透過組合而非繼承的方式使用 StreamJsonRpc 連接物件。

## 總結

透過全面集成 StreamJsonRpc，HagiCode 專案成功解決了原自定義實作的維護成本高、功能侷限性和架構分層混亂等問題。關鍵改進包括：

1. 採用強型別 DTO 替代動態屬性，提高了程式碼的可維護性和可靠性
2. 實作了傳輸層抽象和通訊協定層分離，提升了架構的清晰性
3. 增強了日誌記錄功能，便於排查通信問題
4. 引入了流式傳輸支援，簡化了流式處理的實作

這些改進爲 HagiCode 提供了更穩定、更高效的通信基礎，使其能夠更好地與外部 AI 工具進行互動，併爲未來的功能擴充奠定了堅實的基礎。
