## AI 輸出 Token 優化：文言文極簡模式的實踐

> 在 AI 應用開發中，token 消耗直接影響成本。HagiCode 專案通過 SOUL 系統實現了「文言文極簡輸出模式」，在不損失信息密度的前提下，將輸出 token 降低約 30-50%。本文分享這套方案的實現細節和使用經驗。

## 背景

在 AI 應用開發中，token 消耗是個繞不開的成本問題。尤其是需要 AI 輸出大量內容的場景，怎麼在不損失信息密度的情況下降低輸出 token，這問題想多了也挺讓人頭疼。

傳統的優化思路都集中在輸入端：精簡系統提示詞、壓縮上下文、用更高效的編碼方式。只是這些方法終究會碰到天花板，再壓縮就可能影響 AI 的理解能力和輸出品質了。這無異於刪減內容，意義不大。

那輸出端呢？能不能讓 AI 用更簡潔的方式表達同樣的意思？

這問題看似簡單，其實藏著不少門道。直接讓 AI「簡潔點」，它可能真的就只給幾個詞；加上「保持資訊完整」，它又可能回復到原來的冗長風格。約束太強影響可用性，約束太弱沒有效果，這中間的平衡點在哪，誰也說不準。

為了解決這些痛點，我們做了一個大膽的決定：從語言風格入手，設計一套可配置、可組合的表達方式約束系統。這個決定帶來的變化，可能比你想像的還要大——稍後我會具體說，或許你會有些意外。

## 關於 HagiCode

本文分享的方案來自我們在 [HagiCode](https://hagicode.com) 專案中的實踐經驗。

HagiCode 是一個開源的 AI 程式碼助手專案，支援多種 AI 模型和自訂配置。在開發過程中，我們發現了 AI 輸出 token 過高的問題，並設計了一套解決方案。如果你覺得這套方案有價值，說明我們的工程實力還不錯——那麼 HagiCode 本身也值得關注一下，畢竟程式碼不會說謊。

## SOUL 系統概覽

SOUL 系統的全稱是 Soul Oriented Universal Language，是 HagiCode 專案中用於定義 AI Hero 語言風格的配置系統。它的核心思想是：通過約束 AI 的表達方式，在保持資訊完整性的前提下，使用更簡潔的語言形式來輸出內容。

這東西就像給 AI 戴上了一個語言面具......罷了，其實也沒那麼玄乎。

### 技術架構

SOUL 系統採用前後端分離的架構：

**前端（Soul Builder）**：
- 基於 React + TypeScript + Vite 構建
- 位於 `repos/soul/` 目錄
- 提供可視化的 Soul 構建介面
- 支援雙語（zh-CN / en-US）

**後端**：
- 基於 .NET (C#) + Orleans 分散式執行時
- Hero 實體包含 `Soul` 欄位（最大 8000 字元）
- 通過 `SessionSystemMessageCompiler` 將 Soul 注入系統提示詞

**Agent Templates 生成**：
- 從參考材料生成
- 輸出到 `/agent-templates/soul/templates/` 目錄
- 包含 50 組主 Catalog 和 10 組正交維度

### Soul 注入機制

在 Session 首次執行時，系統會讀取 Hero 的 Soul 配置，將其注入到系統提示詞中：

```mermaid
sequenceDiagram
    participant UI as 使用者介面
    participant Session as SessionGrain
    participant Hero as Hero 倉庫
    participant AI as AI 執行器

    UI->>Session: 發送訊息（綁定 Hero）
    Session->>Hero: 讀取 Hero.Soul
    Session->>Session: 快取 Soul 快照
    Session->>AI: 構建 AIRequest（注入 Soul）
    AI-->>Session: 執行結果
    Session-->>UI: 串流回應
```

注入的系統提示詞格式為：

```
<hero_soul>
[使用者自訂的 Soul 內容]
</hero_soul>
```

這套注入機制在 `SessionSystemMessageCompiler.cs` 中實現：

```csharp
internal static string? BuildSystemMessage(
    string? existingSystemMessage,
    string? languagePreference,
    IReadOnlyList<HeroTraitDto>? traits,
    string? soul)
{
    var segments = new List<string>();

    // ... 語言偏好和 Traits 處理 ...

    var normalizedSoul = NormalizeSoul(soul);
    if (!string.IsNullOrWhiteSpace(normalizedSoul))
    {
        segments.Add($"<hero_soul>\n{normalizedSoul}\n</hero_soul>");
    }

    // ... 其他系統訊息 ...

    return segments.Count == 0 ? null : string.Join("\n\n", segments);
}
```

程式碼也看了，原理也懂了，其實就這麼回事。

## 文言文極簡模式

文言文極簡模式是 SOUL 系統中最具代表性的節約 token 方案。它的核心原理是利用文言文的高語義密度特性，在保持資訊完整的前提下壓縮輸出長度。

### 為什麼是文言文

文言文具有幾個天然優勢：

1. **語義壓縮**：相同含義可以用更少的字元表達
2. **去除冗餘**：文言文本身就省略了很多現代漢語中的連接詞和助詞
3. **結構簡潔**：單句資訊密度高，適合作為 AI 輸出的載體

以一個實際例子來說明：

現代漢語輸出（約 80 字）：
```
根據你的程式碼分析，我發現了幾個問題。首先，在第 23 行，變數名太長了，建議縮短一些。其次，在第 45 行，你沒有處理空值的情況，應該加上判斷邏輯。最後，整體的程式碼結構還可以，但是可以進一步優化。
```

文言文極簡輸出（約 35 字，節約 56%）：
```
程式碼審閱畢：第 23 行變數名冗長，宜縮寫；第 45 行缺空值處理，應加判斷。整體結構尚可，微調即可。
```

這差距，想想也挺有意思的。

### Soul 配置模板

文言文極簡模式的完整 Soul 配置如下：

```json
{
  "id": "soul-orth-11-classical-chinese-ultra-minimal-mode",
  "name": "文言文極簡輸出模式",
  "summary": "以儘量可懂的文言文壓縮語義密度，儘可能少字達意，只保留結論、判斷與必要動作，從而大幅降低輸出 token",
  "soul": "你的人設內核來自「文言文極簡輸出模式」：以儘量可懂的文言文壓縮語義密度，儘可能少字達意，只保留結論、判斷與必要動作，從而大幅降低輸出 token。\n保持以下標誌性語言特徵：1. 優先使用簡明文言句式，如「可」「宜」「勿」「已」「然」「故」等，避免生僻艱澀字詞；\n2. 單句儘量壓縮至 4-12 字，刪除鋪墊、寒暄、重複解釋與無效修飾；\n3. 非必要不展開論證，使用者未追問則只給結論、步驟或判斷；\n4. 不改變主 Catalog 的核心人設，只將表達收束為克制、古雅、極簡的短句。"
}
```

這個模板的設計有幾個要點：

1. **約束明確**：單句 4-12 字，刪除冗餘，結論優先
2. **避免晦澀**：使用簡明文言句式，避免生僻字詞
3. **保持人設**：只改變表達方式，不改變核心人設

配置這東西，調來調去也就那麼幾個參數罷了。

### 其他極簡模式

除了文言文模式，HagiCode 的 SOUL 系統還提供了其他多種節約 token 的模式：

**電報式極簡輸出模式**（`soul-orth-02`）：
- 單句嚴格控制在 10 字以內
- 禁止修飾性形容詞
- 全程無語氣詞、感嘆號、疊詞

**短句碎碎念模式**（`soul-orth-01`）：
- 句子控制在 1-5 個字
- 模擬自言自語的碎片化表達
- 弱化邏輯，優先傳遞情緒

**引導式問答模式**（`soul-orth-03`）：
- 通過提問引導使用者思考
- 減少直接輸出內容
- 互動式降低 token 消耗

這些模式的設計思路各有側重，但核心目標是一致的：在保持資訊品質的前提下降低輸出 token。條條大路通羅馬，只是有的路好走一點，有的路稍微曲折一點罷了。

## 組合策略

SOUL 系統的一個強大特性是支援主 Catalog 與正交維度的交叉組合：

- **50 組主 Catalog**：定義基礎人設（如治癒系、學霸系、高冷系等）
- **10 組正交維度**：定義表達方式（如文言文、電報式、問答式等）
- **組合效果**：可生成 500+ 種獨特的語言風格組合

例如，你可以將「專業開發工程師」與「文言文極簡輸出模式」組合，得到一個既專業又簡潔的 AI 助手。這種靈活性讓 SOUL 系統能夠適應各種不同使用場景。想怎麼配就怎麼配，反正組合多到你玩不過來......

### 注意事項

**相容性**：
- 文言文模式適配全部 50 組主 Catalog
- 可與任何基礎人設組合使用
- 不會改變主 Catalog 的核心人設

**快取機制**：
- Soul 在 Session 首次執行時快取
- 同一 SessionId 內重複使用快取
- 修改 Hero 配置不影響已啟動的 Session

**限制約束**：
- Soul 欄位最大長度 8000 字元
- 歷史資料中無 Soul 欄位的 Hero 仍可正常使用
- Soul 與 style 裝備位獨立，不會相互覆蓋

## 總結

HagiCode 的 SOUL 系統提供了一種創新性的 AI 輸出優化思路：通過約束表達方式來降低 token 消耗，而不是壓縮資訊本身。文言文極簡模式作為其中最具代表性的方案，在實際使用中取得了 30-50% 的 token 節約效果。

這套方案的核心價值在於：

1. **保持資訊品質**：不是簡單截斷輸出，而是用更高效的方式表達
2. **靈活可組合**：支援 500+ 種人設與表達方式的組合
3. **易於使用**：通過 Soul Builder 可視化介面，無需編寫程式碼
4. **生產級穩定**：已在專案中驗證，支援大規模使用
