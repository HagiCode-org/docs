## 在瀏覽器中快速編輯程式碼：VSCode Web 整合實踐

> AI 分析完程式碼後，如何立即在瀏覽器中開啟編輯器進行修改？本文分享 HagiCode 專案中整合 code-server 的實踐經驗，實現 AI 助手與程式碼編輯體驗的無縫連接。

## 背景介紹

在 AI 輔助程式設計的時代，開發者經常需要快速檢視和編輯程式碼。傳統的開發流程是：在桌面端 IDE 中開啟專案，定位檔案，編輯，儲存。只是這個流程在某些場景下，總覺得哪裡不太對勁。

場景一：遠端開發。當使用 HagiCode 這樣的 AI 助手時，後端可能執行在遠端伺服器或容器中，本地無法直接存取專案檔案。每次要檢視或修改程式碼，都需要透過 SSH 或其他方式連接，體驗割裂。這感覺就像你想見一個人，卻隔著一層厚厚的玻璃，能看見卻摸不著。

場景二：快速預覽。AI 助手分析程式碼後，使用者可能只是想快速瀏覽某個檔案或做小幅修改。啟動完整的桌面 IDE 顯得重量級，瀏覽器內的輕量級編輯器更符合「快速檢視」的需求。畢竟，誰願意為了看一眼就大動干戈呢？

場景三：跨裝置協作。在不同裝置上工作時，瀏覽器中的編輯器提供了統一的存取入口，無需每台裝置都配置開發環境。這倒也省事，畢竟人生苦短，何必重複勞動。

為了解決這些痛點，我們在 HagiCode 專案中整合了 VSCode Web。讓 AI 助手與程式碼編輯體驗無縫連接——AI 分析完程式碼後，使用者可以立即在同一個瀏覽器工作階段中開啟編輯器進行修改，無需切換應用程式。這種體驗，怎麼說呢，就像你想的時候，它就在那裡。

## 關於 HagiCode

本文分享的方案來自我們在 [HagiCode](https://hagicode.com) 專案中的實踐經驗。HagiCode 是一個 AI 驅動的程式碼助手，旨在透過自然語言互動提升開發效率。在開發過程中，我們發現使用者經常需要在 AI 分析和程式碼編輯之間快速切換，這促使我們探索如何將編輯器直接整合到瀏覽器中。

專案位址：[github.com/HagiCode-org/site](https://github.com/HagiCode-org/site)

## 技術選型：為什麼是 code-server？

在眾多 VSCode Web 解決方案中，我們選擇了 code-server。這個選擇，其實也有幾個考量：

功能完整。code-server 是 VSCode 的 Web 版本，支援桌面版的大部分功能，包括擴充功能系統、智慧提示、除錯等。這意味著使用者在瀏覽器中也能獲得接近桌面版的編輯體驗。畢竟，誰又願意在功能上妥協呢？

部署靈活。code-server 可以作為獨立服務執行，也支援 Docker 容器化部署，與 HagiCode 的架構契合。我們的後端用 C# 撰寫，前端是 React，透過 REST API 與 code-server 服務通訊。這就像搭積木，每塊都有自己的位置。

認證安全。code-server 內建 connection-token 機制，防止未授權存取。每個工作階段都有唯一的 token，確保只有授權使用者才能存取編輯器。安全感這東西，有了才知道重要。

## 架構設計

HagiCode 的 VSCode Web 整合採用前後端分離的架構設計。

### 前端服務層

前端透過 `vscodeServerService.ts` 封裝了與後端的互動：

```typescript
// 開啟專案
export async function openProjectInCodeServer(
  id: string,
  currentInterfaceLanguage?: string,
): Promise<VsCodeServerLaunchResponseDto>

// 開啟 vault
export async function openVaultInCodeServer(
  id: string,
  path?: string,
  currentInterfaceLanguage?: string,
): Promise<VsCodeServerLaunchResponseDto>
```

這兩個方法的區別在於：`openProjectInCodeServer` 用於開啟整個專案，而 `openVaultInCodeServer` 用於開啟 Vault 的特定路徑。對於 MonoSpecs 多倉庫專案，系統會自動建立工作區檔案。分工明確，各自做好自己的事，這就夠了。

### 後端服務層

後端的 `VaultAppService.cs` 實現了核心邏輯：

```csharp
public async Task<VsCodeServerLaunchResponseDto> OpenInCodeServerAsync(
    string id,
    string? relativePath = null,
    string? currentInterfaceLanguage = null,
    CancellationToken cancellationToken = default)
{
    // 1. 取得設定並檢查是否啟用
    var settings = await _vsCodeServerSettingsService.GetResolvedSettingsAsync(cancellationToken);
    if (!settings.Enabled) {
        throw new BusinessException(VsCodeServerErrorCodes.Disabled, "VSCode Server is disabled.");
    }

    // 2. 取得 vault 並解析啟動目錄
    var vault = await RequireVaultAsync(id, cancellationToken);
    var launchDirectory = ResolveLaunchDirectory(vault, relativePath);

    // 3. 確保 code-server 執行並取得執行時資訊
    var runtime = await _vsCodeServerManager.EnsureStartedAsync(settings, cancellationToken);

    // 4. 解析語言設定
    var language = _vsCodeServerSettingsService.ResolveLaunchLanguage(
        settings.Language,
        currentInterfaceLanguage);

    // 5. 建構啟動 URL
    return new VsCodeServerLaunchResponseDto {
        LaunchUrl = AppendQueryString(runtime.BaseUrl, new Dictionary<string, string?> {
            ["folder"] = launchDirectory,
            ["tkn"] = runtime.ConnectionToken,
            ["vscode-lang"] = language,
        }),
        ConnectionToken = runtime.ConnectionToken,
        OpenMode = "folder",
        Runtime = VsCodeServerSettingsService.MapRuntime(
            await _vsCodeServerManager.GetRuntimeSnapshotAsync(cancellationToken)),
    };
}
```

這個方法的職責很清晰：檢查設定、解析路徑、啟動服務、建構 URL。其中 `ResolveLaunchDirectory` 方法會進行路徑安全檢查，防止路徑穿越攻擊。程式碼就像詩，每一行都有自己的韻律。

### 自動執行時管理

後端透過 `VsCodeServerManager` 管理 code-server 行程：

- 檢查行程狀態
- 自動啟動已停止的服務
- 傳回執行時快照（連接埠、行程 ID、啟動時間等）

這種設計讓系統可以自動處理 code-server 的生命週期，使用者無需手動管理服務行程。畢竟，人生已經夠複雜了，能自動化的就自動化吧。

### 語言跟隨機制

HagiCode 支援多語言介面，code-server 也需要跟隨這個設定。系統支援三種語言模式：

- `follow`：跟隨當前介面語言
- `zh-CN`：固定中文
- `en-US`：固定英文

透過 URL 參數 `vscode-lang` 傳遞給 code-server，確保編輯器語言與 HagiCode 介面保持一致。語言這東西，統一了才舒服。

### MonoSpecs 多倉庫工作區

對於 MonoSpecs 專案（包含多個子倉庫的 mono-repo），系統會自動建立 `.code-workspace` 檔案：

```csharp
private async Task<string> CreateWorkspaceFileAsync(Project project, Guid projectId)
{
    var folders = await ResolveWorkspaceFoldersAsync(project.Path);
    var workspaceDocument = new {
        folders = folders.Select(path => new { path }).ToArray(),
    };
    // 產生 workspace 檔案...
}
```

這樣可以在一個 code-server 實例中同時編輯多個子倉庫，對於大型 mono-repo 專案非常實用。多個倉庫在一個視窗裡，就像多個故事在同一本書裡。

## 前端整合

HagiCode 前端使用 React + TypeScript，整合 code-server 的步驟倒也不複雜。

### 快速操作按鈕

在專案卡片中新增 Code Server 按鈕：

```tsx
// QuickActionsZone.tsx
<Button
  size="sm"
  variant="default"
  onClick={() => onAction({ type: 'open-code-server' })}
>
  <Globe className="h-3 w-3 mr-1" />
  <span className="text-xs">{t('project.openCodeServer')}</span>
</Button>
```

這個按鈕會觸發開啟動作，呼叫後端 API 取得啟動 URL。一個按鈕，承載一個動作，簡單直接。

### 處理開啟動作

```tsx
const handleAction = async (action: ProjectAction) => {
  if (action.type === 'open-code-server') {
    const response = await openProjectInCodeServer(project.id, i18n.language);
    window.open(response.launchUrl, '_blank', 'noopener,noreferrer');
  }
};
```

使用 `window.open` 在新分頁中開啟 code-server，`noopener,noreferrer` 參數確保安全性。安全這東西，再怎麼小心都不為過。

### Vault 編輯入口

在 Vault 列表中新增類似的編輯按鈕：

```tsx
const handleEditVault = async (vault: VaultItemDto) => {
  const response = await openVaultInCodeServer(vault.id);
  window.open(response.launchUrl, '_blank', 'noopener,noreferrer');
};
```

專案和 Vault 使用相同的開啟方式，保持了互動的一致性。一致性的重要，不亞於功能本身。

## URL 建構邏輯

code-server 的 URL 格式，其實也有點講究：

**資料夾模式**：
```
http://{host}:{port}/?folder={path}&tkn={token}&vscode-lang={lang}
```

**工作區模式**：
```
http://{host}:{port}/?workspace={workspacePath}&tkn={token}&vscode-lang={lang}
```

其中 `tkn` 是連接 token，每次啟動 code-server 時自動產生，確保存取安全。`vscode-lang` 參數控制編輯器介面語言。這些參數各有各的用途，缺一不可。

## 使用場景

### 場景一：AI 輔助程式碼審查

使用者與 HagiCode 對話，AI 分析專案程式碼並發現潛在問題。使用者點擊「Open in Code Server」按鈕直接在瀏覽器中開啟編輯器，檢視問題檔案並修復，然後返回 HagiCode 繼續對話。整個流程在瀏覽器中完成，無需切換應用程式。這種感覺，怎麼說呢，就像流水一樣順暢。

### 場景二：Vault 學習資料編輯

使用者建立了學習某個開源專案的 Vault，想在 `docs/` 目錄新增學習筆記。透過 code-server 直接在瀏覽器中編輯 Markdown 檔案，儲存後 HagiCode 可以同步讀取更新的筆記內容。這對於建構個人知識庫非常有用。知識這東西，越積累越有價值。

### 場景三：MonoSpecs 多倉庫開發

MonoSpecs 專案包含多個子倉庫，code-server 自動建立多資料夾工作區。使用者在瀏覽器中可以同時編輯多個倉庫的程式碼，修改後提交到各自的 Git 倉庫。這種工作方式特別適合需要跨倉庫修改的場景。多個倉庫一起改，就像同時處理多件事情，需要點技巧。

## 安全性考量

在實作 code-server 整合時，安全性是需要重點關注的問題。畢竟，安全這種事，出了問題就晚了。

### Connection Token

connection-token 是隨機產生的，不應洩露。建議在 HTTPS 環境下使用，防止 token 被中間人截獲。敏感資訊，還是保護好為妙。

### 檔案路徑安全

後端實作了路徑穿越檢查：

```csharp
private static string ResolveLaunchDirectory(VaultRegistryEntry vault, string? relativePath)
{
    var vaultRoot = EnsureTrailingSeparator(Path.GetFullPath(vault.PhysicalPath));
    var combinedPath = Path.GetFullPath(Path.Combine(vaultRoot, relativePath ?? "."));
    if (!combinedPath.StartsWith(vaultRoot, StringComparison.OrdinalIgnoreCase))
    {
        throw new BusinessException(VaultRelativePathTraversalCode, "Relative path traversal detected.");
    }
    return combinedPath;
}
```

這段程式碼確保使用者無法透過 `../` 等方式存取 vault 目錄之外的檔案。邊界檢查這種事，做總比不做強。

### 權限控制

code-server 行程以適當的使用者權限執行，避免存取系統敏感檔案。建議使用專用使用者執行 code-server 服務。權限控制，該有的還是要有。

## 效能最佳化

code-server 會消耗伺服器資源，以下是一些最佳化建議：

- 監控 CPU/記憶體使用，必要時調整資源限制
- 大型專案可能需要增加逾時時間
- 實作工作階段逾時自動清理，釋放資源
- 考慮使用快取減少重複計算

HagiCode 提供了執行時狀態監控 API，前端可以透過 `getVsCodeServerSettings()` 取得當前狀態：

```typescript
const { settings, runtime } = await getVsCodeServerSettings();
// runtime.status: 'disabled' | 'stopped' | 'starting' | 'running' | 'unhealthy'
// runtime.baseUrl: "http://localhost:8080"
// runtime.processId: 12345
```

這個設計讓使用者可以清楚了解 code-server 的健康狀態，在出現問題時快速定位。狀態可見，心裡才有數。

## 使用者體驗細節

在實作過程中，我們發現了一些影響使用者體驗的細節，值得特別關注：

首次開啟 code-server 可能需要等待啟動，這個時間可能從幾秒到半分鐘不等。建議在前端顯示載入狀態，讓使用者知道系統正在處理。等待這事兒，有回饋就好。

瀏覽器可能會阻止彈出視窗，需要提示使用者手動允許。HagiCode 在首次開啟時會顯示引導資訊，告訴使用者如何設定瀏覽器權限。使用者體驗，就是在這些細節裡體現的。

建議顯示執行時狀態（啟動中/執行中/錯誤），這樣使用者在遇到問題時可以知道是伺服器端問題還是自己操作問題。知道問題出在哪裡，至少心裡有底。

## 設定範例

code-server 的設定倒也不複雜：

```json
{
  "vscodeServer": {
    "enabled": true,
    "host": "0.0.0.0",
    "port": 8080,
    "language": "follow"
  }
}
```

`enabled` 控制功能開關，`host` 和 `port` 指定監聽位址，`language` 設定語言模式。這些設定可以透過 UI 介面修改，即時生效。簡單的東西，往往最好用。

## 總結

HagiCode 的 VSCode Web 整合提供了一個優雅的解決方案：讓 AI 助手與程式碼編輯體驗無縫連接。透過在瀏覽器中整合 code-server，使用者可以快速回應 AI 的分析結果，在同一個瀏覽器工作階段中完成從分析到編輯的完整流程。

這套方案的幾個關鍵優勢：一是統一體驗，專案和 Vault 使用相同的開啟方式；二是多倉庫支援，MonoSpecs 專案自動建立工作區；三是安全可控，執行時狀態監控和路徑安全檢查。

本文分享的方案是 HagiCode 在實際開發中總結出來的。如果你覺得這套方案有價值，說明我們的工程實踐還不錯——那麼 HagiCode 本身也值得關注一下。畢竟，好東西值得被更多人看見。
