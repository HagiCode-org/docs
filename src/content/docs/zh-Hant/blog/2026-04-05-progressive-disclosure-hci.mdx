## 漸進式披露：如何用"少即是多"的理念改進 AI 產品的人機互動

> 在 AI 產品設計中，使用者輸入的質量往往決定了輸出的質量。本文分享我們在 HagiCode 專案中實踐的一套"漸進式披露"互動方案，透過分步引導、智能補全和即時反饋，將使用者簡短模糊的輸入轉化爲結構化的技術提案，顯著提升了人機互動效率。

## 背景

做 AI 產品的同學應該都遇到過這樣的場景：使用者開啟你的應用，興致勃勃地輸入一行需求，結果 AI 傳回的內容完全不搭邊。不是 AI 不聰明，只是使用者給的資訊太少了，畢竟猜心這種事，誰也做不好。

這種現象在我們開發 HagiCode 的程序中尤爲明顯。HagiCode 是一個 AI 驅動程式的程式碼助手，使用者透過自然語言描述需求來建立技術提案和會話。可在實際使用中，我們發現使用者輸入的內容往往存在這些問題：

- **輸入質量參差不齊**：有的使用者只輸入幾個字，比如"優化登入"、"修復 bug"，缺乏必要的上下文
- **技術術語不統一**：不同使用者用不同的詞說同一件事，有人說"前端"有人說"FE"
- **缺少結構化資訊**：沒有專案背景、沒有存放庫範圍、沒有影響範圍這些關鍵資訊
- **重複性問題**：相同型別的需求反覆出現，每次都要從頭解釋

這些問題導致的直接後果就是：AI 理解困難、產生的提案質量不穩定、使用者體驗差。使用者覺得"這 AI 不行啊"，我們也很委屈——你只給一句話，讓我怎麼猜你想要啥？

其實這也沒辦法，畢竟人和人之間的理解都需要時間，更何況是機器呢？

爲了解決這些痛點，我們做了一個大膽的決定：引入"漸進式披露"的設計理念來改進人機互動。這個決定帶來的變化，可能比你想象的還要大，只是當時我們也沒想到會這麼有效罷了。

## 什麼是漸進式披露

"漸進式披露"（Progressive Disclosure）是一個源自 HCI（人機互動）領域的設計原則，核心思想很簡單：**不要一次性把所有資訊和選項都展示給使用者，而是根據使用者的操作和需求，逐步展示必要的內容**。

這個原則特別適合 AI 產品，因爲 AI 互動天然就是漸進式的——使用者說一點，AI 理解一點，然後補充一點，再理解更多。就像人與人之間的交流一樣，總得慢慢來，畢竟誰也不能一見面就把心掏出來不是？

具體到 HagiCode 的場景，我們從四個方面實施了漸進式披露：

### 1. 描述優化機制：讓 AI 幫你把話說清楚

當使用者輸入簡短描述時，我們不是直接讓 AI 去理解，而是先觸發一個"描述優化"流程。這個流程的核心是"結構化輸出"——把使用者的自由文字轉化爲標準格式。就像把散落一地的珍珠串成項鍊，看起來也就不那麼亂了。

優化後的描述必須包含以下幾個標準章節：

- **背景**：問題背景和上下文
- **分析**：技術分析和思考程序
- **解決**：解決方案和實施步驟
- **實踐**：實際程式碼示例和注意事項

同時，我們還會自動產生一個 Markdown 表格，展示目標存放庫、路徑、編輯權限等資訊，方便 AI 後續操作。畢竟有個清晰的目錄，找起東西來也方便。

下面是實際的程式碼實作：

```csharp
// ProposalDescriptionMemoryService.cs 中的核心方法
public async Task<string> OptimizeDescriptionAsync(
    string title,
    string description,
    string locale = "zh-CN",
    DescriptionOptimizationMemoryContext? memoryContext = null,
    CancellationToken cancellationToken = default)
{
    // 构建查询参数
    var queryContext = BuildQueryContext(title, description);

    // 检索历史上下文
    var memoryContext = await RetrieveHistoricalContextAsync(queryContext, cancellationToken);

    // 生成结构化提示词
    var prompt = await BuildOptimizationPromptAsync(
        title,
        description,
        memoryContext,
        cancellationToken);

    // 调用 AI 进行优化
    return await _aiService.CompleteAsync(prompt, cancellationToken);
}
```

這個流程的關鍵在於"記憶注入"——我們會把專案慣例、相似案例、負面模式等歷史上下文注入到提示詞中，讓 AI 在優化時能夠參考過去的經驗。畢竟喫一塹長一智，過去的經驗總不能白白浪費了不是？

**注意事項**：
- 確保當前輸入優先於歷史記憶，避免覆寫使用者顯式指定的資訊
- HagIndex 參考必須作爲事實來源，不得被歷史案例修改
- 低置信度的糾錯建議不應作爲強約束注入

### 2. 語音輸入能力：說話比打字更自然

除了文字輸入，我們還支援語音輸入。這在描述複雜需求時特別有用——你想想，打一段技術需求可能要幾分鐘，但說可能幾十秒就完事了，畢竟嘴總是比手快。

語音輸入的設計重點是"狀態管理"，使用者必須清楚當前系統處於什麼狀態。我們定義了以下幾種狀態：

- **空閒**：系統就緒，可以開始錄製
- **等待上游**：正在連接後端服務
- **錄製中**：正在錄製使用者語音
- **處理中**：正在將語音轉換爲文字
- **錯誤**：發生錯誤，需要使用者處理

前端的狀態模型大概是這樣的：

```typescript
interface VoiceInputState {
  status: 'idle' | 'waiting-upstream' | 'recording' | 'processing' | 'error';
  duration: number;
  error?: string;
  deletedSet: Set<string>; // 已删除结果的指纹集合
}

// 开始录制时的状态转换
const handleVoiceInputStart = async () => {
  // 先进入等待状态，显示加载动画
  setState({ status: 'waiting-upstream' });

  // 等待后端就绪确认
  const isReady = await waitForBackendReady();
  if (!isReady) {
    setState({ status: 'error', error: '后端服务未就绪' });
    return;
  }

  // 开始录制
  setState({ status: 'recording', startTime: Date.now() });
};

// 处理识别结果
const handleRecognitionResult = (result: RecognitionResult) => {
  const fingerprint = normalizeFingerprint(result.text);

  // 检查是否已被删除
  if (state.deletedSet.has(fingerprint)) {
    return; // 跳过已删除的内容
  }

  // 合并结果到文本框
  appendResult(result);
};
```

這裏有個細節：我們用"指紋集合"來管理刪除同步。當語音識別傳回多條結果時，使用者可能會刪除其中一些。我們把已刪除內容的指紋存起來，後續如果相同內容再出現就自動跳過。這就像記住了哪些菜不愛喫，下次就不會再點了，畢竟誰也不想被同樣的問題困擾兩次。

### 3. 提示詞管理系統：把 AI 的"腦子"外置

HagiCode 有一個靈活的提示詞管理系統，所有提示詞都以檔案形式存儲：

```
prompts/
├── metadata/
│   ├── optimize-description.zh-CN.json
│   └── optimize-description.en-US.json
└── templates/
    ├── optimize-description.zh-CN.hbs
    └── optimize-description.en-US.hbs
```

每個提示詞由兩部分組成：
- **元資料檔案（.json）**：定義提示詞的場景、版本、參數等資訊
- **範本檔案（.hbs）**：使用 Handlebars 語法的實際提示詞內容

元資料檔案的格式是這樣的：

```json
{
  "scenario": "optimize-description",
  "locale": "zh-CN",
  "version": "1.0.0",
  "syntax": "handlebars",
  "syntaxVersion": "1.0",
  "parameters": [
    {
      "name": "title",
      "type": "string",
      "required": true,
      "description": "提案标题"
    },
    {
      "name": "description",
      "type": "string",
      "required": true,
      "description": "原始描述"
    }
  ],
  "author": "HagiCode Team",
  "description": "优化用户输入的技术提案描述",
  "lastModified": "2026-04-05",
  "tags": ["optimization", "nlp"]
}
```

範本檔案使用 Handlebars 語法，支援參數注入：

```handlebars
你是一个技术提案专家。

<task>
根据以下信息生成结构化的技术提案描述。
</task>

<input>
<title>{{title}}</title>
<description>{{description}}</description>
{{#if memoryContext}}
<memory_context>
{{memoryContext}}
</memory_context>
{{/if}}
</input>

<output_format>
## 背景
[描述问题背景和上下文，包括项目信息、仓库范围等]

## 分析
[技术分析和思考过程，说明为什么需要这个改动]

## 解决
[解决方案和实施步骤，列出关键代码位置]

## 实践
[实际代码示例和注意事项]
</output_format>
```

這種設計的好處是：
- 提示詞可以像程式碼一樣版本管理
- 支援多語言，根據使用者偏好設定自動切換
- 參數化設計，可以動態注入上下文
- 啓動時驗證完備性，避免執行階段出錯

畢竟腦子裏的東西不寫下來，誰也不知道什麼時候就忘了，與其到時候懊悔，不如一開始就做好記錄罷了。

### 4. 漸進式嚮導：複雜任務拆成小步

對於複雜任務（比如首次安裝設定），我們使用了多步驟嚮導的設計。每個步驟只請求必要資訊，並提供清晰的進度指示。生活也是這樣嘛，一口吃不成胖子，一步一步來反而更穩妥。

嚮導的狀態模型：

```typescript
interface WizardState {
  currentStep: number;           // 0-3，对应 4 个步骤
  steps: WizardStep[];
  canGoNext: boolean;
  canGoBack: boolean;
  isLoading: boolean;
  error: string | null;
}

interface WizardStep {
  id: number;
  title: string;
  description: string;
  completed: boolean;
}

// 步骤导航逻辑
const goToNextStep = () => {
  if (wizardState.currentStep < wizardState.steps.length - 1) {
    // 验证当前步骤的输入
    if (validateCurrentStep()) {
      wizardState.currentStep++;
      wizardState.steps[wizardState.currentStep - 1].completed = true;
    }
  }
};

const goToPreviousStep = () => {
  if (wizardState.currentStep > 0) {
    wizardState.currentStep--;
  }
};
```

每個步驟都有獨立的驗證邏輯，已完成步驟會有清晰的視覺標記。取消操作會彈出確認對話方塊，防止使用者誤操作丟失進度。畢竟走錯路可以回頭，但如果把路都拆了，那就真的沒轍了。

## 總結

回顧 HagiCode 的漸進式披露實踐，我們可以總結出幾個核心原則：

1. **分步引導**：把複雜任務拆成小步，每步只請求必要資訊
2. **智能補全**：利用歷史上下文和專案知識自動補全資訊
3. **即時反饋**：每個操作都有清晰的視覺反饋和狀態提示
4. **容錯機制**：允許使用者撤銷、重置，避免錯誤造成不可逆損失
5. **輸入多樣化**：支援文字、語音等多種輸入方式

這套方案在 HagiCode 中的實際效果是：使用者輸入的平均長度從不到 20 字提升到了結構化的 200-300 字，AI 產生的提案質量顯著提高，使用者滿意度也跟着上來了。

其實這也不奇怪，畢竟你給的資訊越多，AI 理解得越準確，傳回的結果自然就越好，這和人與人之間的交流也沒什麼兩樣。

如果你也在做 AI 相關的產品，希望這些經驗能給你帶來一些啓發。記住：使用者不是不想給資訊，而是你還沒問對問題。漸進式披露的核心，就是找到問問題的最佳時機和方式，只是這個時機和方式，需要一點耐心去摸索罷了。
