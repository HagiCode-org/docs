## Design.md：讓 AI 一致性進行前端 UI 設計的解決方案

> 在 AI 輔助前端開發時代，如何讓 AI 生成的 UI 保持一致性？本文分享了我們基於 awesome-design-md 構建設計畫廊站點的實踐經驗，以及如何創建結構化的 design.md 來指導 AI 進行規範化的 UI 設計。

## 背景

用過 AI 寫前端代碼的朋友應該都有過類似的經歷：同一個頁面，讓 AI 多生成幾次，每次的風格都不一樣。有時候是圓角有時候是方角，有時候間距是 8px 有時候又變成 16px，甚至同一個按鈕在不同對話裡長得都不一樣。

這不僅僅是個別現象。隨著 AI 輔助開發的普及，**AI 生成的前端 UI 缺乏一致性**已經成為一個普遍問題。不同的 AI 助手、不同的提示詞，甚至同一助手在不同對話中，都會產生風格迥異的界面設計。這給產品疊代帶來了巨大的維護成本。

問題的根源其實很簡單：缺少一份**權威的設計參考文檔**。傳統的 CSS 樣式文件只能告訴開發者"怎麼實現"，卻無法完整傳達"為什麼這樣設計"以及"在什麼場景下使用什麼設計模式"。而對於 AI 來說，它更需要一個清晰的結構化描述來理解設計規範。

與此同時，開源社區已經有了一些很好的資源。[VoltAgent/awesome-design-md](https://github.com/VoltAgent/awesome-design-md) 項目收集了大量知名公司的設計系統文檔，每個目錄包含 README.md、DESIGN.md 和預覽 HTML。但這些都分散在上游倉庫中，難以快速查閱和比較。

那能不能把這些資源整合起來，做成一個方便查閱的設計畫廊，同時沉澱出一份結構化的 design.md 給 AI 用呢？

答案是肯定的。接下來分享我們的方案。

## 關於 HagiCode

本文分享的方案來自我們在 [HagiCode](https://hagicode.com) 項目中的實踐經驗。HagiCode 是一個 AI 輔助開發平台，在開發過程中，我們也遇到了 AI 生成 UI 不一致的問題。為了解決這個問題，我們構建設計畫廊站點並創建規範化的 design.md，本文就是這套方案的總結。

[GitHub - HagiCode-org/site](https://github.com/HagiCode-org/site)

## 分析

在動手寫代碼之前，我們先來拆解一下這個問題的幾個技術挑戰。

### 內容源管理：如何統一分散的設計資源？

上游的 awesome-design-md 倉庫包含了大量設計文檔，但我們需要一種方式把它納入到我們的項目中。

**方案：使用 git submodule**

```
awesome-design-md-site
└── vendor/awesome-design-md          # 上游資源（git submodule）
```

這樣做有幾個好處：
- 版本可控：可以鎖定特定的上游版本
- 離線構建：不需要在構建時請求外部 API
- 內容審閱：可以在 PR 中看到具體變更

### 數據標準化：不同文檔結構怎麼統一？

不同公司的設計文檔結構可能不同，有些缺少預覽文件，有些命名不統一。我們需要在構建期進行標準化處理。

### 靜態站點架構：怎麼在保持靜態部署的同時提供動態搜索？

既然是設計畫廊，搜索功能是必須的。但 Astro 是靜態站點生成器，怎麼實現實時搜索呢？

**方案：React island + URL 查詢參數同步**

```typescript
// src/components/gallery/SearchToolbar.tsx

export function SearchToolbar() {
  const [query, setQuery] = useState('');

  // URL 同步
  useEffect(() => {
    const params = new URLSearchParams(window.location.search);
    setQuery(params.get('q') || '');
  }, []);

  // 實時過濾
  const filtered = entries.filter(entry =>
    entry.searchText.includes(query)
  );

  return <input value={query} onChange={e => {
    setQuery(e.target.value);
    updateURL(e.target.value);
  }} />;
}
```

這樣做的好處是保留了靜態站點的可部署性（可以部署到任何靜態託管服務），同時提供了即時過濾的用戶體驗。

### 設計文檔化：怎麼讓 AI 理解並遵守設計規範？

這是整個方案的核心。我們需要創建一份結構化的 design.md，讓 AI 能夠理解並應用我們的設計規範。

**方案：借鑑 ClickHouse DESIGN.md 的結構**

[ClickHouse 的 DESIGN.md](https://github.com/ClickHouse/ClickHouse/blob/master/DESIGN.md) 是一個很好的參考，它包含了：
- Visual Theme & Atmosphere
- Color Palette & Roles
- Typography Rules
- Component Stylings
- Layout Principles
- Depth & Elevation
- Do's and Don'ts
- Responsive Behavior
- Agent Prompt Guide

我們的做法是：**結構參考，內容重寫**。保留 ClickHouse DESIGN.md 的章節結構，但把內容替換成我們自己項目實際使用的設計 token 和組件規範。

## 解決

基於上述分析，我們的解決方案包含四個核心模塊。

### 1. 內容攝取管線

這是整個系統的基礎，負責從上游資源中提取和標準化內容。

```typescript
// src/lib/content/awesomeDesignCatalog.ts

export async function scanSourceEntries(): Promise<DesignEntry[]> {
  const designDir = 'vendor/awesome-design-md/design-md';
  const entries: DesignEntry[] = [];

  for (const dir of await fs.readdir(designDir)) {
    const entryPath = path.join(designDir, dir);
    if (await isValidDesignEntry(entryPath)) {
      const entry = await normalizeDesignEntry(entryPath);
      entries.push(entry);
    }
  }

  return entries;
}

async function isValidDesignEntry(dir: string): Promise<boolean> {
  const requiredFiles = ['README.md', 'DESIGN.md'];
  for (const file of requiredFiles) {
    if (!(await fileExists(path.join(dir, file)))) {
      return false;
    }
  }
  return true;
}
```

### 2. 畫廊瀏覽界面

畫廊界面包括三個主要部分：

**首頁**：展示所有設計條目的卡片網格，每個卡片包含：
- 設計條目標題和簡介
- 預覽圖（如果有）
- 快速搜索高亮

**詳情頁**：聚合展示單個設計條目的完整信息：
- README 文檔
- DESIGN 文檔
- 預覽（支持明/暗主題切換）
- 相鄰條目導航

**導航**：支持返回畫廊、瀏覽相鄰條目

首頁畫廊使用高密度卡片布局，把不同來源的 design.md 條目平鋪在一個統一的視覺框架裡，方便快速對比品牌風格、按鈕模式和排版節奏。

![Awesome Design MD Gallery 設計卡片網格](/article-images/design-md-consistent-ui/1775613199000-e652f7155dd5.png)

進入具體條目後，詳情頁會把設計摘要和實時預覽放在同一個頁面中，減少在文檔、預覽和源碼之間來回切換的成本。

### 3. 搜索功能

搜索功能基於客戶端過濾，使用 URL 查詢參數保持狀態：

```typescript
// src/components/gallery/SearchToolbar.tsx

function SearchToolbar({ entries }: { entries: DesignEntry[] }) {
  const [query, setQuery] = useState('');
  const [results, setResults] = useState(entries);

  useEffect(() => {
    const params = new URLSearchParams(window.location.search);
    const initialQuery = params.get('q') || '';
    setQuery(initialQuery);
    filterEntries(initialQuery);
  }, []);

  const filterEntries = (searchQuery: string) => {
    const filtered = entries.filter(entry =>
      entry.searchText.toLowerCase().includes(searchQuery.toLowerCase())
    );
    setResults(filtered);
  };

  const handleChange = (e: React.ChangeEvent<HTMLInputElement>) => {
    const value = e.target.value;
    setQuery(value);
    filterEntries(value);

    // 更新 URL（不觸發頁面刷新）
    const newUrl = value
      ? `${window.location.pathname}?q=${encodeURIComponent(value)}`
      : window.location.pathname;
    window.history.replaceState({}, '', newUrl);
  };

  return (
    <div className="search-toolbar">
      <input
        type="text"
        value={query}
        onChange={handleChange}
        placeholder="搜索設計條目..."
      />
      <span className="result-count">{results.length} 個結果</span>
    </div>
  );
}
```

## 實踐

了解了方案之後，具體怎麼實施呢？

### 實施步驟

**第一步：初始化子模塊**

```bash
# 添加上游倉庫為子模塊
git submodule add https://github.com/VoltAgent/awesome-design-md.git vendor/awesome-design-md

# 初始化並更新子模塊
git submodule update --init --recursive
```

**第二步：創建內容管線**

實現 `awesomeDesignCatalog.ts`，包括：
- 文件掃描和校驗邏輯
- Markdown 渲染（使用 Astro 的內置渲染器）
- 條目數據提取

**第三步：構建畫廊 UI**

使用 Astro + React Islands 創建：
- 首頁畫廊布局（卡片網格）
- 設計卡片組件
- 搜索工具欄
- 詳情頁布局

**第四步：編寫設計文檔**

基於 ClickHouse DESIGN.md 結構，填充自己項目的實際設計 token。更新 README.md，添加指向 design.md 的鏈接。

### 注意事項

**安全性**：Markdown 渲染需要過濾不安全的 HTML。Astro 的內置渲染器默認會過濾 script 標籤，但仍需注意 XSS 風險。

**性能**：大量 iframe 預覽可能影響首屏加載。建議使用 `loading="lazy"` 延遲加載預覽內容。

**維護性**：design.md 需要與代碼實現保持同步。建議在 CI 中添加檢查，確保 CSS 變量在文檔和代碼中一致。

**可訪問性**：確保顏色對比度符合 WCAG AA 標準（至少 4.5:1）。

### AI 使用指南

創建 design.md 之後，怎麼讓 AI 真正用它呢？這裡有幾個實用技巧：

**技巧一：在提示詞中明確引用**

```
請參考項目根目錄的 design.md 文件，使用其中定義的設計規範來實現以下組件：
- 按鈕：使用 primary 色調，圓角 8px
- 卡片：使用 elevation-2 陰影層級
```

**技巧二：要求 AI 引用具體的 CSS 變量**

```
實現一個導航欄，要求：
- 背景色使用 --color-bg-primary
- 邊框使用 --color-border-subtle
- 文字使用 --text-color-primary
```

**技巧三：在系統提示詞中包含 design.md 內容**

如果你的 AI 工具支持自定義系統提示詞，可以將 design.md 的核心內容直接添加進去。

### 測試策略

**內容管線測試**：
- 文件缺失場景（缺少 README.md 或 DESIGN.md）
- 格式錯誤場景（Markdown 解析失敗）
- 空目錄場景

**搜索功能測試**：
- 空結果處理
- 特殊字符（如中文、emoji）
- URL 同步驗證

**UI 組件測試**：
- 明/暗主題切換
- 響應式布局
- 預覽加載狀態

### 部署流程

```bash
# 1. 更新子模塊到最新版本
git submodule update --remote

# 2. 重新構建站點
npm run build

# 3. 部署靜態資源
npm run deploy
```

建議將子模塊更新和構建部署自動化，可以在上游倉庫更新時自動觸發 CI 流程。

## 總結

HagiCode 在開發過程中遇到的 AI 生成 UI 不一致問題，本質上是缺少結構化的設計參考文檔。通過構建設計畫廊站點和創建規範化的 design.md，我們成功解決了這個問題。

這套方案的核心價值在於：
- **統一資源**：整合分散的設計系統文檔
- **結構化規範**：將設計規範以 AI 可理解的形式呈現
- **持續維護**：通過 git submodule 保持內容更新

如果你也在使用 AI 輔助前端開發，建議嘗試一下這個方案。創建一份結構化的 design.md，不僅能提升 AI 生成代碼的一致性，也能幫助團隊內部保持設計規範的統一。
