## 告別腳本地獄：為什麼我們選擇用 C# 打造現代化構建系統

> 揭秘 HagiCode 專案如何利用 Nuke 實現類型安全、跨平台且高度可擴展的自動化構建流程，徹底解決傳統構建腳本的維護痛點。

## 背景

在軟體開發的漫長旅途中，"構建"這個詞往往讓人又愛又恨。愛的是，一鍵點擊，代碼變成產品，那是程序員最迷人的時刻；恨的是，維護那一堆亂糟糟的構建腳本，簡直是噩夢。

在很多專案中，我們習慣了用 Python 寫腳本，或者用 XML 配置文件（想像一下那段被 `<property>` 支配的恐懼）。但隨著專案複雜度的提升，尤其是像 HagiCode 這樣涉及前後端、多平台、多語言混合開發的專案，傳統的構建方式開始顯得力不從心。腳本邏輯分散、缺乏類型檢查、IDE 支援弱……這些問題像一個個小坑，時不時就讓開發團隊絆個跟頭。

為了解決這些痛點，在 HagiCode 專案中，我們決定引入 **Nuke** —— 一個基於 C# 的現代化構建系統。它不僅僅是一個工具，更像是一種對構建流程的重新思考。今天，我們就來聊聊為什麼選擇它，以及它是如何讓我們的開發體驗"起飛"的。

我們正在開發 **HagiCode** —— 一款 AI 驅動的代碼智能助手，讓開發體驗變得更智能、更便捷、更有趣。

**智能** —— AI 全程輔助，從想法到代碼，讓編碼效率提升數倍。**便捷** —— 多執行緒並發操作，充分利用資源，開發流程順暢無阻。**有趣** —— 遊戲化機制和成就系統，讓編碼不再枯燥，充滿成就感。

專案正在快速迭代中，如果你對技術寫作、知識管理或者 AI 輔助開發感興趣，歡迎來 [GitHub](https://github.com/HagiCode-org/site) 看看～

## 核心剖析：為什麼是 Nuke？

你可能心裡會犯嘀咕："哎呀，構建系統那麼多，比如 Make、Gradle，甚至直接用 Shell 腳本不行嗎？為啥非得整一個 C# 的？"

這其實是個好問題。Nuke 的核心魅力在於它把我們最熟悉的編程語言特性帶進了構建腳本的世界。

### 1. 將構建流程模組化：Target 的藝術

Nuke 的設計理念非常清晰：**一切皆為目標**。

在傳統的腳本裡，我們可能會寫出幾百行線性執行的代碼，邏輯錯綜複雜。而在 Nuke 中，我們將構建流程分解為獨立的 `Target`（目標）。每個目標只負責一件事，比如：

- `Clean`: 清理輸出目錄
- `Restore`: 還原依賴包
- `Compile`: 編譯代碼
- `Test`: 運行單元測試

這種設計非常符合單一職責原則。就像搭積木一樣，我們可以隨意組合這些 Target。更重要的是，Nuke 允許我們定義 Target 之間的依賴關係。比如，你想要 `Test`，那系統會自動檢查你是否先執行了 `Compile`；想要 `Compile`，自然得先 `Restore`。

這種依賴關係圖不僅讓邏輯更清晰，還極大地提高了執行效率，Nuke 會自動分析最優執行路徑。

### 2. 類型安全：告別拼寫錯誤的噩夢

用過 Python 寫構建腳本的朋友肯定遇到過這種尷尬：腳本跑了五分鐘，最後報錯說 `Confi.guration` 拼寫錯了，或者傳了一個字符串給了一個本該是數字的參數。

使用 C# 編寫構建腳本最大的優勢就是 **類型安全**。這意味著：

- **編譯時檢查**：你在敲代碼的時候，IDE 就會告訴你哪裡錯了，不用等到運行時才發現。
- **重構無憂**：如果你想改個變量名或者方法名，IDE 的重構功能一鍵搞定，不用全局搜索替換提心吊膽。
- **智能提示**：強大的 IntelliSense 會自動補全代碼，你不需要去翻文檔記那些生僻的 API。

### 3. 跨平台：統一的構建體驗

以前在 Windows 上寫 `.bat`，在 Linux 上寫 `.sh`，為了兼容兩者，還得寫個 Python 腳本。現在，只要是 .NET Core（現 .NET 5+）能跑的地方，Nuke 就能跑。

這意味著無論團隊成員是使用 Windows、Linux 還是 macOS，無論是用 Visual Studio、VS Code 還是 Rider，大家執行的都是同一套邏輯。這就極大地消除了"在我機器上能跑"這類環境差異導致的問題。

### 4. 參數與配置管理

Nuke 提供了一套非常優雅的參數解析機制。你不需要手動去解析 `string[] args`，只需要定義一個屬性，加上 `[Parameter]` 特性，Nuke 就會自動處理命令行參數和配置文件的映射。

比如，我們可以輕鬆定義構建配置：

```csharp
[Parameter("Configuration to build - Default is 'Debug'")]
readonly Configuration BuildConfiguration = IsLocalBuild ? Configuration.Debug : Configuration.Release;

Target Compile => _ => _
    .DependsOn(Restore)
    .Executes(() =>
    {
        // 在這裡使用 BuildConfiguration，它是類型安全的
        DotNetBuild(s => s
            .SetConfiguration(BuildConfiguration)
            .SetProjectFile(SolutionFile));
    });
```

這種寫法既直觀又不容易出錯。

## 實踐指南：如何在專案中落地

空談誤國，實干興邦。讓我們看看在 HagiCode 專案中，具體是怎麼落地這套方案的。

### 1. 規劃專案結構

我們不想讓構建腳本污染專案根目錄，也不想搞得像某些 Java 專案那樣目錄結構深不見底。所以，我們將所有與 Nuke 相關的構建文件統一放置在 `nukeBuild/` 文件夾中。

這樣做的好處是：
- 專案根目錄保持清爽。
- 構建邏輯內聚，方便管理。
- 新成員加入時，一眼就能看到"哦，這是構建相關的邏輯"。

### 2. 設計清晰的 Target 依賴鏈

在設計 Target 時，我們遵循了一個原則：**原子化 + 依賴流**。

每個 Target 應該足夠小，只做一件事。比如 `Clean` 就只管刪文件，不要在裡面順便做打包。

推薦的依賴流大概是这个樣子的：

`Clean` -> `Restore` -> `Compile` -> `Test` -> `Pack`

當然，這不是絕對的。比如如果你只想跑個測試，不想打包，Nuke 允許你直接執行 `nuke Test`，它會自動處理好前置的 Restore 和 Compile 步驟。

### 3. 完善的錯誤處理與日誌

構建腳本最怕的是什麼？是報錯信息不明確。比如構建失敗了，日誌只顯示 "Error: 1"，這就讓人很抓狂。

在 Nuke 中，由於我們可以直接使用 C# 的異常處理機制，因此可以非常精確地捕獲和報告錯誤。

```csharp
Target Publish => _ => _
    .DependsOn(Test)
    .Executes(() =>
    {
        try 
        {
            // 嘗試發布到 NuGet
            DotNetNuGetPush(s => s
                .SetTargetPath(ArtifactPath)
                .SetSource("https://api.nuget.org/v3/index.json")
                .SetApiKey(ApiKey));
        }
        catch (Exception ex)
        {
            Log.Error($"發布失敗了，兄弟們檢查一下 Key 對不對: {ex.Message}");
            throw; // 確保構建進程以非零退出碼結束
        }
    });
```

### 4. 集成測試保障質量

構建腳本本身也是代碼，也需要測試。Nuke 允許我們為構建流程編寫測試，確保當我們修改了構建邏輯後，不會破壞現有的發布流程。這在持續集成（CI）流水線中尤為重要。

## 總結

通過引入 Nuke，HagiCode 的構建流程變得前所未有的順暢。它不僅僅是一個工具的替換，更是工程化思維的提升。

**我們收穫了什麼？**
- **可維護性**：代碼即配置，邏輯清晰，新人也能快速上手。
- **穩定性**：強類型檢查減少了 90% 以上的低級錯誤。
- **一致性**：跨平台的統一體驗，消除了環境差異。

如果說以前寫構建腳本是"在黑暗中摸索"，那麼使用 Nuke 就像是"開著燈走夜路"。如果你受夠了維護那些難以調試的腳本語言，不妨試試把構建邏輯也搬到 C# 的世界裡來，也許你會發現，原來構建也可以這麼優雅。
