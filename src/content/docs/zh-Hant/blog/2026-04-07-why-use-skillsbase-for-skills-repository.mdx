## 爲什麼使用 Skillsbase 維護自己的 Skills 收藏存放庫

> 說起來也挺好笑的，AI 程式設計時代來了，我們手裏的 Agent Skills 越來越多，可隨之而來的麻煩也越來越多。這篇文章就是想聊聊我們是怎麼用 skillsbase 解決這些問題的。

## 背景

在 AI 程式設計時代，開發人員需要維護越來越多的 Agent Skills——這些是可複用的指令集，用於擴充 Claude Code、OpenCode、Cursor 等編碼助手的能力。然而，隨着技能數量的增長，一個現實問題逐漸浮現：

其實也不能說是什麼大問題，只是東西多了，管理起來就麻煩了。

### 技能分散存儲，管理成本高

- 本地技能散落在多個位置：`~/.agents/skills/`、`~/.claude/skills/`、`~/.codex/skills/.system/` 等
- 不同位置可能存在命名衝突（例如 `skill-creator` 同時存在於使用者目錄和系統目錄）
- 缺乏統一的管理入口，備份和遷移困難

這點挺煩人的。有時候你自己都不知道某個技能到底在哪，像丟了東西一樣，找起來費勁。

### 缺乏標準化維護流程

- 手工複製技能容易出錯，難以追蹤來源
- 沒有統一的驗證機制，無法保證技能存放庫的完整性
- 團隊協作時，難以同步和共享技能集合

手工操作嘛，總是容易出錯的。畢竟人的記憶力有限，誰記得住那麼多東西是從哪來的呢？

### 無法滿足可復現性要求

- 更換開發機器時，需要重新設定所有技能
- CI/CD 環境中無法自動驗證和同步技能存放庫

換個電腦就得重新來一遍，這種感覺，怎麼說呢，就像搬家一樣麻煩。每次都得重新適應新的環境，重新設定所有東西。

爲了解決這些痛點，我們嘗試過多種方案：從手工複製到腳本自動化，從直接管理目錄到全局安裝再回收。每種方案都有各自的缺陷，要麼無法保證一致性，要麼污染環境，要麼難以在 CI 中使用。

其實也是走了不少彎路。

最終，我們找到了一套更優雅的解決方案——skillsbase。這個方案的核心思想是：先本地安裝驗證，再轉換結構寫入存放庫，最後卸載臨時檔案。這樣既能確保存放庫內容與實際安裝結果一致，又不會污染全局環境。

說起來簡單，只是踩了不少坑才琢磨出來罷了。

## 關於 HagiCode

本文分享的方案來自我們在 [HagiCode](https://hagicode.com) 專案中的實踐經驗。

HagiCode 是一個 AI 程式碼助手專案，在開發程序中我們需要維護大量的 Agent Skills 來擴充各種編碼能力。正是這些實際需求，促使我們開發了 skillsbase 這套工具來規範化管理技能存放庫。

其實這東西也不是憑空想出來的，都是被逼的。技能多了，自然就需要管理。管理的程序中遇到問題，自然就需要解決。一步一步，就走到今天了。

如果你對 HagiCode 感興趣，可以存取 [官網瞭解更多](https://hagicode.com) 或在 [GitHub](https://github.com/HagiCode-org/site) 上查看源碼。

## 分析

### 技術挑戰

要建立一個可維護的技能收藏存放庫，需要解決以下核心問題：

1. **統一命名空間衝突**：當多個來源存在同名技能時，如何避免覆寫？
2. **來源可追溯性**：如何記錄每個技能的來源，以便後續更新和審計？
3. **同步與驗證**：如何確保存放庫內容與實際安裝結果一致？
4. **自動化集成**：如何與 CI/CD 流程集成，實作自動同步和驗證？

這些問題看似簡單，但每一個都夠頭疼的。不過話說回來，做什麼事不難呢？

### 設計權衡

**方案一：直接複製目錄**

優點：實作簡單
缺點：無法保證與 `skills` CLI 實際安裝結果一致

這個方案嘛，說真的，我們也想過。只是後來發現，CLI 安裝的時候可能有一些預處理邏輯，直接複製就跳過了。結果就是，複製的東西和實際裝的東西不一樣，這就有問題了。

**方案二：全局安裝後回收**

優點：可以驗證安裝程序
缺點：污染執行環境，CI 與本地結果難以保持一致

這個方案更糟糕。全局安裝，會污染環境。更麻煩的是，CI 環境和本地環境很難保持一致，導致"本地能跑，CI 失敗"的問題。這種感覺，誰懂啊？

**方案三：本地安裝 → 轉換 → 卸載（最終方案）**

這是 skillsbase 採用的方案：

- 先透過 `npx skills` 把技能安裝到臨時位置
- 轉換目錄結構並新增來源元資料
- 寫入目標存放庫
- 最後卸載臨時檔案

這種方案確保了存放庫內容與實際消費者安裝結果一致，同時不污染全局環境，轉換程序可標準化，支援冪等操作。

其實這個方案也不是一開始就想到的，只是試錯試多了，自然就知道什麼可行、什麼不可行了。

### 架構決策

| 決策項 | 選擇 | 理由 |
|--------|------|------|
| 執行階段 | Node.js ESM | 無需建置步驟，`.mjs` 足以完成檔案系統編排 |
| 設定格式 | YAML (`sources.yaml`) | 可讀性強，支援人工維護 |
| 命名策略 | 命名空間前綴 | 使用者技能保持原名，系統技能新增 `system-` 前綴 |
| 工作流 | `add` 修改清單 → `sync` 執行同步 | 單一同步引擎，避免規則雙份實作 |
| 檔案管理 | 受管檔案標識 | 新增註釋頭，支援安全覆寫 |

這些決策，說到底都是爲了一個目標：讓事情變得簡單。畢竟，簡單纔是王道。

## 解決

### CLI 架構

`skillsbase` CLI 提供四個核心命令：

```text
skillsbase
├── init          # 初始化仓库结构
├── sync          # 同步技能内容
├── add           # 添加新技能
└── github_action # 生成 GitHub Actions 配置
```

命令不多，但也夠了。畢竟，工具這東西，夠用就好。

### 核心工作流

```
┌─────────────┐    ┌─────────────┐    ┌─────────────┐    ┌─────────────┐
│    init     │───▶│    add      │───▶│    sync     │───▶│github_action│
│  初始化仓库  │    │  添加来源   │    │  同步内容   │    │  生成 CI    │
└─────────────┘    └─────────────┘    └─────────────┘    └─────────────┘
```

一步一步來，急不得。

### 同步流程設計

```
sources.yaml → 解析来源 → npx skills 安装 → 转换结构 → 写入 skills/ → 卸载临时文件
                              ↓
                        .skill-source.json (来源元数据)
```

這個流程設計得還算清晰。至少我自己看的時候，能明白每一步在做什麼。

### 存放庫結構

```
repos/skillsbase/
├── sources.yaml              # 来源清单（单一真相源）
├── skills/                   # 技能目录
│   ├── frontend-design/      # 用户技能
│   ├── skill-creator/        # 用户技能
│   └── system-skill-creator/ # 系统技能（带前缀）
├── scripts/
│   ├── sync-skills.mjs       # 同步脚本
│   └── validate-skills.mjs   # 验证脚本
├── docs/
│   └── maintainer-workflow.md # 维护者文档
└── .github/
    ├── workflows/
    │   └── skills-sync.yml   # CI 工作流
    └── actions/
        └── skillsbase-sync/
            └── action.yml     # 复用型 Action
```

檔案多了點，不過也還好。畢竟，組織結構清晰了，維護起來也方便。

## 實踐

### 初始化技能存放庫

```bash
# 1. 创建空仓库
mkdir repos/myskills && cd repos/myskills
git init

# 2. 使用 skillsbase 初始化
npx skillsbase init

# 输出：
# [1/4] create manifest ................. done
# [2/4] create scripts .................. done
# [3/4] create docs ..................... done
# [4/4] create github workflow .......... done
#
# next: skillsbase add <skill-name>
```

這一步會產生一堆檔案，不過不用擔心，都是自動產生的。接下來就可以開始新增技能了。

### 新增技能

```bash
# 添加单个技能（会自动执行同步）
npx skillsbase add frontend-design --source vercel-labs/agent-skills

# 添加到本地来源
npx skillsbase add documentation-writer --source /home/user/.agents/skills

# 输出：
# source: first-party ......... updated
# target: skills/frontend-design ... synced
# status: 1 skill added, 0 removed
```

新增技能挺簡單的，一條命令就夠了。只是有時候會遇到一些意外情況，比如網路不好、權限問題之類的。不過這些都是小事，慢慢來。

### 同步技能

```bash
# 执行同步（对账所有来源）
npx skillsbase sync

# 仅检查是否漂移（不修改文件）
npx skillsbase sync --check

# 允许缺失来源（CI 场景）
npx skillsbase sync --allow-missing-sources
```

同步的時候，系統會把 `sources.yaml` 裏定義的來源都檢查一遍，然後和 `skills/` 目錄裏的內容對賬。有差異就更新，沒差異就跳過。這樣就不會出現"設定改了但檔案沒變"的問題。

### 產生 CI 設定

```bash
# 生成 workflow
npx skillsbase github_action --kind workflow

# 生成 action
npx skillsbase github_action --kind action

# 生成全部
npx skillsbase github_action --kind all
```

CI 設定也是自動產生的。只是你需要自己調整一些細節，比如觸發條件、執行環境之類的。不過這些都不難。

### sources.yaml 設定示例

```yaml
# 技能根目录配置
skillsRoot: skills/
metadataFile: .skill-source.json

# 来源定义
sources:
  # 第一方：本地用户技能
  first-party:
    type: local
    path: /home/user/.agents/skills
    naming: original  # 保持原名
    includes:
      - documentation-writer
      - frontend-design
      - skill-creator

  # 系统：系统提供技能
  system:
    type: local
    path: /home/user/.codex/skills/.system
    naming: prefix-system  # 添加 system- 前缀
    includes:
      - imagegen
      - openai-docs
      - skill-creator  # 会变成 system-skill-creator

  # 远程：第三方仓库
  vercel:
    type: remote
    url: vercel-labs/agent-skills
    naming: original
    includes:
      - web-design-guidelines
```

這個設定檔案是整個系統的核心。所有的來源都在這裏定義，改了這裏，下次同步就會生效。所以說，這算是一個"單一真相源"。

### .skill-source.json 元資料示例

```json
{
  "source": "first-party",
  "originalPath": "/home/user/.agents/skills/documentation-writer",
  "originalName": "documentation-writer",
  "targetName": "documentation-writer",
  "syncedAt": "2026-04-07T00:00:00.000Z",
  "version": "unknown"
}
```

每個技能目錄下都會有這個檔案，記錄了它的來源資訊。這樣以後出問題的時候，能快速定位是從哪來的、什麼時候同步的。

### 安全與驗證

```bash
# 验证仓库结构
node scripts/validate-skills.mjs

# 使用 skills CLI 验证
npx skills add . --list

# 检查更新
npx skills check
```

驗證這東西，說重要也重要，說沒必要也沒必要。不過爲了保險起見，偶爾跑一下也無妨。畢竟，誰知道會不會有什麼意外呢？

### GitHub Actions 集成

```yaml
# .github/workflows/skills-sync.yml
name: Skills Sync

on:
  push:
    paths:
      - 'sources.yaml'
      - 'skills/**'
  workflow_dispatch:

jobs:
  validate:
    runs-on: ubuntu-latest
    steps:
      - uses: actions/checkout@v4
      - uses: actions/setup-node@v4
        with:
          node-version: '20'
      - name: Validate repository
        run: |
          npx skills add . --list
          node scripts/validate-skills.mjs
      - name: Sync check
        run: npx skillsbase sync --check
```

CI 集成之後，每次改 `sources.yaml` 或者 `skills/` 目錄，都會自動觸發驗證。這樣就不會出現"本地改了忘了同步"的問題了。

### 最佳實踐

1. **命名衝突處理**：系統技能統一新增 `system-` 前綴。這樣既能保留所有技能，又能避免命名衝突。
2. **冪等操作**：所有命令支援重複執行，多次執行 `sync` 不會產生副作用。這點在 CI 裏特別重要。
3. **受管檔案**：產生的檔案包含 `# Managed by skillsbase CLI` 註釋，方便識別和管理。這些檔案可以安全覆寫，手工修改不會被保留。
4. **非互動模式**：CI 環境預設使用確定性行爲，不會因爲互動式提示而中斷。所有設定都透過 `sources.yaml` 宣告。
5. **來源可追溯**：每個技能都有 `.skill-source.json` 記錄來源資訊，出問題的時候能快速定位。

### 團隊協作

```bash
# 团队成员安装共享技能库
npx skills add your-org/myskills -g --all

# 本地克隆验证
git clone https://github.com/your-org/myskills.git
cd myskills
npx skills add . --list
```

透過 Git 管理技能存放庫，團隊成員可以輕鬆同步技能集合，確保每個人都使用相同版本的工具和設定。

這點在團隊協作裏特別有用，不會出現"我這邊能跑你那邊不行"的情況。畢竟，環境統一了，問題就少了一半。

## 總結

使用 `skillsbase` 維護技能收藏存放庫的核心價值在於：

- **安全性**：來源驗證、衝突檢測、受管檔案保護
- **可維護性**：統一入口、冪等操作、設定即文件
- **標準化**：統一的目錄結構、命名規範、元資料格式
- **自動化**：CI/CD 集成、自動同步、自動驗證

透過這套方案，開發人員可以像管理 npm 包一樣管理自己的 Agent Skills，實作可復現、可共享、可維護的技能存放庫體系。

本文分享的這套工具和流程，正是我們在開發 HagiCode 程序中實際踩坑、實際優化出來的方案。如果你覺得這套方案有價值，說明我們的工程實踐方向是正確的——那麼 HagiCode 本身也值得關注一下。

畢竟，好的工具是值得被更多人使用的。
