## AI Compose Commit：用 AI 智能重構 Git 提交工作流

## Introduction

在軟體開發過程中,提交代碼是程序員每天都要面對的日常工作。可是你有沒有經歷過這樣的場景:一天工作結束後,打開 Git 看到幾十個未暫存的修改文件,卻不知道該如何將它們組織成合理的提交?

傳統的方式是手動將文件分批暫存、逐個提交、撰寫提交信息,這個過程既耗時又容易出錯。咱們就常常在這上面浪費了不少時間,畢竟誰也不想在建經疲憊的晚上還要為這些瑣事煩心。

我們在 HagiCode 項目中推出了一項新功能——AI Compose Commit,旨在徹底改變這個工作流程。它通過 AI 智能分析工作區中的所有未提交變更,自動將它們分組為多個邏輯提交,並執行符合規範的提交操作。本文將深入探討這個功能的實現原理、技術架構以及我們在實踐中遇到的挑戰與解決方案。

## About HagiCode

本文分享的方案來自我們在 [HagiCode](https://hagicode.com) 項目中的實踐經驗。

## Background

### 傳統 Git 提交的痛點

Git 作為版本控制系統,為開發者提供了強大的代碼管理能力。但在實際使用中,提交操作往往成為開發流程中的瓶頸:

1. **手動分組耗時**: 當有大量文件變更時,開發者需要逐個檢查文件內容,判斷哪些屬於同一個功能,這需要耗費大量腦力
2. **提交信息質量參差**: 撰寫符合 Conventional Commits 規範的提交信息需要經驗和技巧,新手常常寫出不規範的提交
3. **多倉庫管理複雜**: 在 monorepo 環境中,需要在不同倉庫間切換,增加了操作複雜度
4. **工作流被打斷**: 提交代碼會打斷開發思路,影響編碼效率

這些問題在大型項目和團隊協作環境中尤為明顯。一個優秀的開發工具應該讓開發者專注於核心的編碼工作,而不是被繁瑣的提交流程所困擾。

### AI 輔助開發的趨勢

近年來,AI 技術在軟體開發領域的應用日益廣泛。從代碼補全、錯誤檢測到自動生成文檔,AI 正在逐步滲透到開發的各個環節。在 Git 工作流方面,雖然已有一些工具提供提交信息生成的功能,但大多侷限於單次提交的場景,缺乏對整個工作區變更的智能分析和分組能力。

其實 HagiCode 在開發過程中也遇到了這些痛點,我們曾嘗試過多種工具,但都或多或少存在一些局限性。要麼是功能不夠完善,要麼是用戶體驗不夠好。這也是為什麼我們最終決定自己實現 AI Compose Commit 功能的原因。

HagiCode 的 AI Compose Commit 功能正是為了填補這一空白而生,它不僅是生成提交信息,而是完整接管從文件分析到執行提交的整個流程。

## Problem

### 技術挑戰

在實現 AI Compose Commit 功能的過程中,我們面臨了多個技術挑戰:

1. **文件語義理解**: AI 需要理解文件變更的語義關係,判斷哪些文件屬於同一個功能模塊。這需要深入分析文件內容、目錄結構以及變更的上下文。

2. **提交分組策略**: 如何定義合理的分組標準?是按功能、按模塊,還是按文件類型?不同的項目可能適用不同的策略。

3. **實時反饋與異步處理**: Git 操作可能需要較長時間,特別是處理大量文件時。如何在保證用戶體驗的同時完成複雜操作?

4. **多倉庫支持**: 在 monorepo 架構下,需要在主倉庫和子倉庫之間正確路由操作。

5. **錯誤處理與回滾**: 如果某個提交失敗,如何處理已執行的提交?是否需要回滾已暫存的文件?

6. **提交信息一致性**: 生成的提交信息需要符合項目現有的風格,保持歷史提交的格式一致。

### 性能考量

AI 處理大量文件變更會消耗顯著的時間和計算資源。我們需要在以下方面進行優化:

- 減少不必要的 AI 調用
- 優化文件上下文的構建方式
- 實現高效的 Git 操作批處理

這些問題在 HagiCode 的實際使用中都真實出現過,我們通過不斷的迭代和優化才找到了相對完美的解決方案。如果你也在開發類似的工具,希望我們的經驗能給你一些啟發。

## Solution

### 整體架構設計

我們採用了分層架構來實現 AI Compose Commit 功能,確保系統具有良好的可擴展性和可維護性:

#### 1. API 層(Web 層)

**GitController** 提供了 `POST /api/git/auto-compose-commit` 端點,作為功能入口。為了優化用戶體驗,我們採用了 Fire-and-Forget 異步模式:

- 客戶端發起請求後,服務器立即返回 HTTP 202 Accepted
- 實際的 AI 處理在後台異步執行
- 處理完成後通過 SignalR 通知客戶端

這種設計確保了即使 AI 處理需要幾分鐘,用戶也能立即得到響應,不會感覺系統卡頓。

#### 2. 應用服務層(Application 層)

**GitAppService** 負責核心業務邏輯:

- 倉庫檢測:支持 monorepo 中的多倉庫管理
- 鎖管理:防止並發操作導致的衝突
- 文件暫存協調:與 AI 處理流程的交互
- 錯誤回滾:處理失敗場景下的狀態恢復

#### 3. 分佈式計算層(Orleans Grains)

**AIGrain** 作為 AI 操作的核心執行單元,實現了 `IAIGrain` 接口中的 `AutoComposeCommitAsync` 方法:

```csharp
// 定義 AI 自動組合提交的接口方法
// 參數說明:
// - projectId: 項目唯一標識符
// - unstagedFiles: 未暫存文件列表,包含文件路徑和狀態信息
// - projectPath: 項目根目錄路徑(可選),用於訪問項目上下文
// 返回值: 包含執行結果的響應對象,包括成功/失敗狀態和詳細信息
[Alias("AutoComposeCommitAsync")]
[ResponseTimeout("00:20:00")] // 20 分鐘超時,適用於處理大型變更集
Task<AutoComposeCommitResponseDto> AutoComposeCommitAsync(
    string projectId,
    GitFileStatusDto[] unstagedFiles,
    string? projectPath = null);
```

這個方法設置了 20 分鐘的超時時間,以處理大型變更集。HagiCode 在實際使用中發現,有些項目的單次變更可能涉及上百個文件,需要更長的處理時間。

#### 4. AI 服務層

通過抽象的 `IAIService` 接口,我們實現了 AI 服務的可插拔架構。目前使用 Claude Helper 服務,但可以輕鬆切換到其他 AI 提供商。

### 核心實現邏輯

#### 文件上下文構建

AI 需要了解每個文件的狀態才能做出智能決策。我們通過 `BuildFileChangesXml` 方法構建文件上下文:

```csharp
/// <summary>
/// 構建文件變更的 XML 表示形式,用於為 AI 提供完整的文件上下文信息
/// </summary>
/// <param name="stagedFiles">已暫存的文件列表,包含文件路徑、狀態和舊路徑(針對重命名操作)</param>
/// <returns>格式化的 XML 字符串,包含所有文件的元數據信息</returns>
private static string BuildFileChangesXml(GitFileStatusDto[] stagedFiles)
{
    var sb = new StringBuilder();
    sb.AppendLine("<files>");

    foreach (var file in stagedFiles)
    {
        sb.AppendLine("  <file>");
        // 使用 XML 轉義確保特殊字符不會破壞 XML 結構
        sb.AppendLine($"    <path>{System.Security.SecurityElement.Escape(file.Path)}</path>");
        sb.AppendLine($"    <status>{System.Security.SecurityElement.Escape(file.Status)}</status>");

        // 處理文件重命名場景,記錄舊路徑以便 AI 理解變更關係
        if (!string.IsNullOrEmpty(file.OldPath))
        {
            sb.AppendLine($"    <oldPath>{System.Security.SecurityElement.Escape(file.OldPath)}</oldPath>");
        }

        sb.AppendLine("  </file>");
    }

    sb.AppendLine("</files>");
    return sb.ToString();
}
```

這個 XML 格式的上下文包含文件路徑、狀態和舊路徑(針對重命名操作),為 AI 提供了完整的元數據。通過結構化的 XML 格式,我們確保了 AI 能夠準確理解每個文件的狀態和變更類型。

#### AI 權限管理

為了讓 AI 能夠直接執行 Git 操作,我們配置了全面的工具權限:

```csharp
// 定義 AI 可以使用的工具集合,包括文件操作和 Git 命令執行權限
// Read/Write/Edit: 文件讀寫和編輯能力
// Bash(git:*): 執行所有 Git 命令的權限
// 其他 Bash 命令: 用於查看文件內容和目錄結構,輔助 AI 理解上下文
var allowedTools = new[]
{
    "Read", "Write", "Edit",
    "Bash(git:*)", "Bash(cat:*)", "Bash(ls:*)", "Bash(find:*)",
    "Bash(grep:*)", "Bash(head:*)", "Bash(tail:*)", "Bash(wc:*)"
};

// 構建完整的 AI 請求對象
var request = new AIRequest
{
    Prompt = prompt,                          // 完整的 Prompt 模板,包含任務指令和約束條件
    WorkingDirectory = projectPath ?? GetTempDirectory(), // 工作目錄,確保 AI 在正確的項目上下文中執行
    AllowedTools = allowedTools,               // 允許使用的工具集合
    PermissionMode = PermissionMode.bypassPermissions, // 繞過權限檢查,允許直接執行 Git 操作
    LanguagePreference = languagePreference         // 語言偏好設置,確保生成符合用戶期望的提交信息
};
```

這裡使用了 `PermissionMode.bypassPermissions` 模式,允許 AI 直接執行 Git 命令而無需用戶確認。這是功能設計的核心,但同時也需要嚴格的輸入驗證來防止濫用。HagiCode 在實際部署中,通過後端的參數驗證和日誌監控,確保了這個機制的安全性。

#### 提交結果解析

AI 執行完成後,會返回結構化的結果。我們實現了雙重解析策略以確保兼容性:

```csharp
/// <summary>
/// 解析 AI 返回的提交執行結果,支持分隔符格式和正則表達式格式
/// </summary>
/// <param name="aiResponse">AI 返回的原始響應內容</param>
/// <returns>解析後的提交結果列表,每個結果包含提交哈希和執行狀態</returns>
private List<CommitResultDto> ParseCommitExecutionResults(string aiResponse)
{
    var results = new List<CommitResultDto>();

    // 優先使用分隔符解析(新格式),這種格式更加明確和可靠
    if (aiResponse.Contains("---"))
    {
        logger.LogDebug("Using delimiter-based parsing for AI response");
        results = ParseDelimitedFormat(aiResponse);

        if (results.Count > 0)
        {
            return results; // 成功解析,直接返回結果
        }

        logger.LogWarning("Delimiter-based parsing produced no results, falling back to regex");
    }
    else
    {
        logger.LogDebug("No delimiter found, using legacy regex-based parsing");
    }

    // 回退到正則表達式解析(舊格式),確保向後兼容性
    return ParseLegacyFormat(aiResponse);
}
```

分隔符格式使用 `---` 作為提交之間的分隔,格式清晰且易於解析:

```
---
Commit 1: abc123def456
feat(auth): add user login functionality

Implement JWT-based authentication with login form and API endpoints.

Co-Authored-By: Hagicode <noreply@hagicode.com>
---
Commit 2: 789ghi012jkl
docs(readme): update installation instructions

Add new setup steps for Docker environment.

Co-Authored-By: Hagicode <noreply@hagicode.com>
---
```

這種格式設計讓解析變得簡單可靠,同時人類閱讀也很清晰。

### 鎖管理機制

為了防止並發操作導致的狀態衝突,我們實現了倉庫鎖機制:

```csharp
// 獲取倉庫鎖,防止並發操作
// 參數說明:
// - fullPath: 倉庫的完整路徑,用於標識不同的倉庫實例
// - requestedBy: 請求者標識,用於追蹤和日誌記錄
await _autoComposeLockService.AcquireLockAsync(fullPath, requestedBy);

try
{
    // 執行 AI Compose Commit 操作
    // 這部分代碼會調用 Orleans Grain 的方法,執行實際的 AI 處理和 Git 操作
    await aiGrain.AutoComposeCommitAsync(projectId, unstagedFiles, projectPath);
}
finally
{
    // 確保鎖被釋放,無論操作成功或失敗
    // 使用 finally 塊可以保證異常情況下也能釋放鎖,避免死鎖
    await _autoComposeLockService.ReleaseLockAsync(fullPath);
}
```

鎖具有 20 分鐘的超時時間,與 AI 操作的超時設置保持一致。如果操作失敗或超時,系統會自動釋放鎖,避免永久阻塞。HagiCode 在實際使用中發現,這個鎖機制非常重要,特別是在團隊協作環境中,多個開發者可能同時觸發 AI Compose Commit 操作。

### SignalR 實時通知

處理完成後,系統通過 SignalR 向前端發送通知:

```csharp
/// <summary>
/// 發送自動組合提交完成的通知
/// </summary>
/// <param name="projectId">項目標識符,用於路由通知到正確的客戶端</param>
/// <param name="totalCount">總提交數量,包括成功和失敗</param>
/// <param name="successCount">成功提交的數量</param>
/// <param name="failureCount">失敗提交的數量</param>
/// <param name="success">整體操作是否成功標志</param>
/// <param name="error">錯誤信息(如果操作失敗)</param>
private async Task SendAutoComposeCommitNotificationAsync(
    string projectId,
    int totalCount,
    int successCount,
    int failureCount,
    bool success,
    string? error)
{
    try
    {
        // 構建通知數據傳輸對象,包含詳細的執行結果
        var notification = new AutoComposeCommitCompletedDto
        {
            ProjectId = projectId,
            TotalCount = totalCount,
            SuccessCount = successCount,
            FailureCount = failureCount,
            Success = success,
            Error = error
        };

        // 通過 SignalR Hub 廣播通知到所有連接的客戶端
        await messageService.SendAutoComposeCommitCompletedAsync(notification);

        logger.LogInformation(
            "Auto compose commit notification sent for project {ProjectId}: {SuccessCount}/{TotalCount} succeeded",
            projectId, successCount, totalCount);
    }
    catch (Exception ex)
    {
        // 記錄通知錯誤但不影響主操作流程
        // 通知失敗不應該導致整個操作失敗
        logger.LogError(ex, "Failed to send auto compose commit notification for project {ProjectId}", projectId);
    }
}
```

前端收到通知後可以更新 UI,顯示提交成功或失敗的狀態,提升用戶體驗。這種實時反饋機制在 HagiCode 的使用中獲得了很好的用戶反饋,用戶可以清楚地知道操作何時完成以及結果如何。

## Implementation

### Prompt 工程設計

AI 的行為完全由 Prompt 決定,我們精心設計了 Auto Compose Commit 的 Prompt 模板。以中文版本為例(`auto-compose-commit.zh-CN.hbs`):

#### 非交互式模式支持

Prompt 開頭明確聲明支持非交互式運行模式,這是 CI/CD 和自動化腳本的關鍵需求:

```handlebars
**重要提示**:此提示詞可能在非交互式環境中運行(如 CI/CD、自動化腳本)。

**非交互式模式**:
- 禁止使用 AskUserQuestion 或任何交互式工具
- 當需要用戶輸入時:
  - 使用合理的默認值(如提交類型使用 feat)
  - 跳過可選的確認步驟
  - 記錄所做的假設
```

這個設計確保了 AI Compose Commit 功能不僅能在交互式 IDE 環境中使用,也能集成到 CI/CD 流程中,實現完全自動化的提交流程。

#### 分支保護機制

為了防止 AI 執行危險操作,我們在 Prompt 中添加了嚴格的分支保護規則:

```handlebars
**分支保護**:
- 禁止執行任何分支切換操作(git checkout、git switch)
- 所有 git commit 命令必須在當前分支上執行
- 不得創建、刪除或重命名分支
- 不得修改未跟踪文件或未暫存變更
- 如果需要分支切換才能完成操作,應返回錯誤而非執行
```

這些規則通過約束 AI 的工具使用範圍,確保操作的安全性。HagiCode 在實際測試中驗證了這些約束的有效性,AI 在遇到需要分支切換的場景時會安全地返回錯誤,而不是執行危險操作。

#### 智能分組決策樹

Prompt 中詳細定義了文件分組的決策邏輯:

```handlebars
**文件分組決策樹**:
├── 是否為配置文件(package.json、tsconfig.json、.env 等)?
│   ├── 是 → 獨立提交(類型:chore 或 build)
│   └── 否 → 繼續
├── 是否為文檔文件(README.md、*.md、docs/**)?
│   ├── 是 → 獨立提交(類型:docs)
│   └── 否 → 繼續
├── 是否與同一功能相關?
│   ├── 是 → 合併到同一提交
│   └── 否 → 分別提交
└── 是否為跨模塊變更?
    ├── 是 → 按模塊分組
    └── 否 → 按功能分組
```

這個決策樹為 AI 提供了清晰的分組邏輯,確保生成的提交符合語義合理性。HagiCode 在實際使用中發現,這個決策樹能夠處理絕大多數常見場景,生成的分組結果符合開發者預期。

#### 歷史格式一致性分析

為了讓提交信息與項目歷史保持一致,Prompt 要求 AI 在生成前分析最近的提交歷史:

```handlebars
**歷史格式一致性**:在生成提交信息之前,你**必須**分析當前倉庫的提交歷史以匹配現有風格

1. 使用 git log -n 15 --pretty=format:"%H|%s|%b%n---%n" 獲取最近的提交歷史
2. 分析提交以識別:
   - 結構模式:項目是否使用多段落?是否有 "Changes:" 或 "Capabilities:" 部分?
   - 語言模式:提交信息是英文、中文還是混合?
   - 常用類型:最常使用哪些提交類型(feat、fix、docs 等)?
   - 特殊格式:是否有 Co-Authored-By 行?其他項目特定的約定?
3. 生成遵循檢測到的模式的提交信息
```

這個分析確保了 AI 生成的提交信息不會顯得突兀,而是與項目的提交歷史保持風格一致。在 HagiCode 的多語言項目中,這個功能特別重要,它能夠根據項目的提交歷史自動選擇合適的語言和格式。

#### Co-Authored-By 要求

每個提交必須包含 Co-Authored-By 信息:

```handlebars
**重要**:每個提交必須添加 Co-Authored-By 信息
- 使用以下格式:git commit -m "type(scope): subject" -m "" -m "Co-Authored-By: Hagicode <noreply@hagicode.com>"
- 或者直接在提交信息中包含 Co-Authored-By 行
```

這不僅是為了貢獻規範,也是為了追蹤 AI 輔助的提交歷史。HagiCode 將這個要求作為強製規則,確保所有 AI 生成的提交都帶有明確的來源標識。

### 工作流程詳解

完整的 AI Compose Commit 工作流程如下:

1. **用戶觸發**: 用戶在 Git Status 面板或 Quick Actions Zone 點擊"AI Auto Compose Commit"按鈕
2. **API 請求**: 前端發送 POST 請求到 `/api/git/auto-compose-commit` 端點
3. **立即響應**: 服務器返回 HTTP 202 Accepted,不等待處理完成
4. **後台處理**:
   - GitAppService 獲取倉庫鎖
   - 調用 AIGrain 的 AutoComposeCommitAsync 方法
   - 構建文件上下文 XML
   - 執行 AI Prompt,讓 AI 分析並執行提交
5. **AI 執行**:
   - 使用 Git 命令獲取所有未暫存變更
   - 讀取文件內容理解變更性質
   - 按語義關係對文件分組
   - 對每組執行 `git add` 和 `git commit` 操作
6. **結果解析**: 解析 AI 返回的執行結果
7. **通知發送**: 通過 SignalR 通知前端
8. **鎖釋放**: 無論成功或失敗,都釋放倉庫鎖

這個流程的設計確保了用戶可以在發起操作後立即繼續其他工作,而不需要等待 AI 處理完成。HagiCode 的用戶反饋表明,這種異步處理方式大大提升了工作流體驗。

#### 3. 超時處理

20 分鐘的超時設置確保了長時間運行的操作不會無限期阻塞資源。超時後,系統會釋放鎖並通知用戶操作失敗。HagiCode 在實際使用中發現,大部分操作能夠在 2-5 分鐘內完成,只有處理超大型變更集時才會接近超時限制。

## Best Practices

### 使用 AI Compose Commit 的最佳實踐

#### 1. 合理使用時機

AI Compose Commit 最適合以下場景:

- 一天工作結束後,批量處理多個文件的變更
- 重構操作後,多個相關文件需要分別提交
- 功能開發完成,需要將相關變更分組提交

不適合以下場景:

- 單個文件的快速提交(直接使用普通提交更快)
- 需要精確控制提交內容的場景
- 包含敏感信息的提交(需要人工審核)

#### 2. 審查 AI 生成的提交

雖然 AI 智能分組很強大,但開發者仍應審查生成的提交:

- 檢查提交的分組是否符合預期
- 驗證提交信息的準確性
- 確認沒有遺漏或錯誤包含文件

如果發現不合理的分組,可以使用 `git reset --soft HEAD~N` 撤銷後重新分組。HagiCode 的經驗表明,即使 AI 分組很智能,人工審查仍然是有價值的,特別是在重要的功能提交時。

#### 3. 配合項目規範

確保項目的 Git 配置支持 Conventional Commits:

```bash
# 安裝 commitlint
npm install -g @commitlint/cli @commitlint/config-conventional

# 配置 commitlint
echo "module.exports = {extends: ['@commitlint/config-conventional']}" > commitlint.config.js
```

這樣可以在 CI/CD 流程中驗證提交信息格式,與 AI Compose Commit 生成的格式保持一致。

### 實現類似功能的建議

如果你想在項目中實現類似的 AI 輔助提交功能,以下是我們的建議:

#### 1. 從小規模開始

先實現單次提交信息生成,再逐步擴展到多提交分組功能。這樣更容易驗證和迭代。HagiCode 也是按照這個路徑逐步完善功能的,早期版本只支持單次提交,後來才擴展到多提交智能分組。

#### 2. 使用成熟的 AI SDK

不要自己實現 AI 調用邏輯,使用現有的 SDK 可以減少開發時間和潛在 bug。我們使用了 Claude Helper 服務,它提供了穩定的接口和完善的錯誤處理。

#### 3. 重視 Prompt 設計

Prompt 的質量直接決定了 AI 輸出的質量。投入時間設計詳細的 Prompt,包括:

- 明確的任務描述
- 具體的輸出格式要求
- 邊界情況的處理規則
- 示例說明

HagiCode 在 Prompt 設計上投入了大量時間,這是功能成功的關鍵因素之一。

#### 4. 實現全面的錯誤處理

AI 操作可能因為各種原因失敗(網絡問題、API 限流、內容審查等)。確保你的系統能夠優雅地處理這些錯誤,並提供有意義的錯誤信息。

#### 5. 提供手動干预機制

不要完全自動化,給用戶保留控制權。提供查看分組結果、調整分組、手動編輯提交信息等選項,平衡自動化與靈活性。HagiCode 雖然實現了自動執行,但仍然保留了預覽和調整的能力。

### 性能優化技巧

#### 1. 文件過濾

在構建文件上下文時,過濾掉不需要 AI 分析的文件:

```csharp
// 過濾掉自動生成的文件和過大的文件,減少 AI 處理負擔
var relevantFiles = stagedFiles
    .Where(f => !IsGeneratedFile(f.Path))
    .Where(f => !IsLargeFile(f.Path))
    .ToArray();
```

#### 2. 並行處理

如果支持多個獨立倉庫,可以並行處理不同倉庫的提交,提高整體效率。

#### 3. 緩存優化

緩存項目提交歷史分析結果,避免每次都重新分析。可以在配置文件中存儲歷史格式偏好,減少 AI 調用次數。

## Conclusion

AI Compose Commit 功能代表了 AI 技術在軟體開發工具中的深度應用。通過智能分析文件變更、自動分組提交、生成規範的提交信息,它顯著提升了 Git 工作流的效率,讓開發者能夠更專注於核心的編碼工作。

在實現過程中,我們學到了幾個重要的經驗:

1. **用戶反饋是關鍵**: 早期版本採用同步等待方式,用戶反饋體驗不佳,改為 Fire-and-Forget 模式後滿意度大幅提升
2. **Prompt 設計決定質量**: 一個精心設計的 Prompt 比複雜的算法更能保證 AI 輸出的質量
3. **安全永遠是第一位的**: 雖然賦予 AI 直接執行 Git 命令的權限帶來了效率提升,但必須配合嚴格的約束和驗證
4. **漸進式改進**: 從簡單場景開始,逐步增加複雜度,比一次性實現所有功能更容易成功

未來,我們計劃進一步優化 AI Compose Commit 功能,包括:

- 支持更多提交分組策略(按時間、按開發者等)

- 支持自定義提交信息模板,滿足不同項目的個性化需求
