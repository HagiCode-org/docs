## 從 CLI 呼叫到 SDK 集成：GitHub Copilot 在 .NET 專案中的最佳實踐

> 從命令列呼叫到官方 SDK 集成的升級之路，說起來也算是一段經歷，今天就分享我們在 HagiCode 專案中踩過的坑和學到的東西。

## 背景

GitHub Copilot SDK 在 2025 年正式發佈後，我們開始將其集成到 AI 能力層中。在此之前，專案主要透過直接呼叫 Copilot CLI 命令列工具來使用 GitHub Copilot 能力，這種方式其實也存在幾個明顯問題：

- **處理序管理複雜**：需要手動管理 CLI 處理序的生命週期、啓動超時和處理序清理——畢竟處理序這東西，說崩潰就崩潰了，也沒什麼預兆
- **事件處理不完整**：原始 CLI 呼叫難以捕獲模型推理程序和工具執行的細粒度事件，就像只能看到結果，卻看不到思考的程序
- **會話管理困難**：缺乏有效的會話複用和復原機制，每次都得重新開始，想想也是挺累的
- **相容性問題**：CLI 參數更新頻繁，需要持續維護參數相容性邏輯，這無異於和風車作戰了

這些問題在日常開發中逐漸顯現，特別是在需要實時追蹤模型推理程序（thinking）和工具執行狀態時，CLI 呼叫的侷限性尤爲明顯。我們也算是想明白了，需要一個更底層、更完整的集成方式——畢竟，條條大路通羅馬，只是有的路好走一點，有的路稍微曲折一點罷了。

## 關於 HagiCode

本文分享的方案來自我們在 [HagiCode](https://hagicode.com) 專案中的實踐經驗。HagiCode 是一個開放原始碼的 AI 程式碼助手專案，在開發程序中我們需要深度集成 GitHub Copilot 的各種能力——從基礎的程式碼補全到複雜的多輪對話和工具呼叫。這些實際需求推動我們從 CLI 呼叫升級到了官方 SDK 集成。

如果你對本文的實踐方案感興趣，說明我們的工程實踐可能對你有說明——那麼 HagiCode 專案本身也值得關注一下。或許在文末你會發現更多關於專案的資訊和連結，誰知道呢......

## 架構設計

專案採用了分層架構來解決 CLI 呼叫的問題：

```
┌─────────────────────────────────────────────────────────┐
│  hagicode-core (Orleans Grains + AI Provider Layer)    │
│  - CopilotAIProvider: 将 AIRequest 转换为 CopilotOptions │
│  - GitHubCopilotGrain: Orleans 分布式执行接口            │
└─────────────────────────────────────────────────────────┘
                          ↓
┌─────────────────────────────────────────────────────────┐
│  HagiCode.Libs (Shared Provider Layer)                 │
│  - CopilotProvider: CLI Provider 接口实现               │
│  - ICopilotSdkGateway: SDK 调用抽象                     │
│  - GitHubCopilotSdkGateway: SDK 会话管理与事件分发     │
└─────────────────────────────────────────────────────────┘
                          ↓
┌─────────────────────────────────────────────────────────┐
│  GitHub Copilot SDK (Official .NET SDK)                │
│  - CopilotClient: SDK 客户端                            │
│  - CopilotSession: 会话管理                             │
│  - SessionEvent: 事件流                                 │
└─────────────────────────────────────────────────────────┘
```

這種分層設計帶來的技術優勢，其實也還挺實用的：

1. **關注點分離**：核心業務邏輯與 SDK 實作細節解耦——畢竟，什麼層做什麼事，井水不犯河水
2. **可測試性**：透過 `ICopilotSdkGateway` 介面可以輕鬆進行單元測試，測試起來也不那麼費勁
3. **複用性**：HagiCode.Libs 可被多個專案參考，寫一次，多處用
4. **可維護性**：SDK 升級只需修改 Gateway 層，上面的程式碼不用動，美得很

## 核心實作

### 認證流程

認證是 SDK 集成的第一步，也是最重要的一步——畢竟，門都進不去，後面的事情就免談了。我們設計了一個靈活的認證設定，支援多種認證來源：

```csharp
// CopilotProvider.cs - 认证来源配置
public class CopilotOptions
{
    public bool UseLoggedInUser { get; set; } = true;
    public string? GitHubToken { get; set; }
    public string? CliUrl { get; set; }
}

// 转换为 SDK 请求
return new CopilotSdkRequest(
    GitHubToken: options.AuthSource == CopilotAuthSource.GitHubToken
        ? options.GitHubToken
        : null,
    UseLoggedInUser: options.AuthSource != CopilotAuthSource.GitHubToken
);
```

這個設計的好處，其實也挺明顯的：

- 支援已登入使用者模式（無需 token），適合桌面端場景——使用者用自己的賬號登入就行
- 支援 GitHub Token 模式，適用於服務端部署——統一管理也方便
- 支援 Copilot CLI URL 覆寫，方便企業代理設定——企業環境嘛，總有些特殊的規矩

在實際使用中，這種靈活的認證方式大大簡化了不同部署場景的設定工作。桌面端可以使用使用者自己的 Copilot 登入狀態，服務端則可以透過 Token 進行統一管理。怎麼說呢，各取所需罷了。

### 事件流處理

SDK 最強大的能力之一，應該就是對事件流的完整捕獲了。我們實作了一個事件分發系統，能夠實時處理各種 SDK 事件——畢竟，知道程序和只知道結果，感覺還是不一樣的：

```csharp
// GitHubCopilotSdkGateway.cs - 事件分发核心逻辑
internal static SessionEventDispatchResult DispatchSessionEvent(
    SessionEvent evt, bool sawDelta)
{
    switch (evt)
    {
        case AssistantReasoningEvent reasoningEvent:
            // 捕获模型推理过程
            events.Add(new CopilotSdkStreamEvent(
                CopilotSdkStreamEventType.ReasoningDelta,
                Content: reasoningEvent.Data.Content));
            break;

        case ToolExecutionStartEvent toolStartEvent:
            // 捕获工具调用开始
            events.Add(new CopilotSdkStreamEvent(
                CopilotSdkStreamEventType.ToolExecutionStart,
                ToolName: toolStartEvent.Data.ToolName,
                ToolCallId: toolStartEvent.Data.ToolCallId));
            break;

        case ToolExecutionCompleteEvent toolCompleteEvent:
            // 捕获工具调用完成及结果
            events.Add(new CopilotSdkStreamEvent(
                CopilotSdkStreamEventType.ToolExecutionEnd,
                Content: ExtractToolExecutionContent(toolCompleteEvent)));
            break;

        default:
            // 未处理事件作为 RawEvent 保留
            events.Add(new CopilotSdkStreamEvent(
                CopilotSdkStreamEventType.RawEvent,
                RawEventType: evt.GetType().Name));
            break;
    }
}
```

這個實作帶來的價值，怎麼說呢：

- **完整捕獲模型推理程序**（thinking）：使用者可以看到 AI 的思考程序，而不僅僅是最終結果——就像知道答案不如知道怎麼思考出來的
- **實時追蹤工具執行狀態**：知道哪些工具正在執行、何時完成、傳回了什麼結果
- **零事件丟失**：透過 fallback 到 RawEvent 機制，確保所有事件都被記錄，什麼都不落下

在 HagiCode 的實際使用中，這些細粒度的事件讓使用者能夠更深入地理解 AI 的工作程序，特別是在調試複雜任務時——這還是有點用處的。

### CLI 相容性處理

從 CLI 呼叫遷移到 SDK 後，我們發現一些原有的 CLI 參數在 SDK 中不再適用。爲了保持向後相容，我們實作了一個參數過濾系統——畢竟，舊設定不能用，也挺讓人頭疼的：

```csharp
// CopilotCliCompatibility.cs - 参数过滤
private static readonly Dictionary<string, string> RejectedFlags = new()
{
    ["--headless"] = "不支持的启动参数",
    ["--model"] = "通过 SDK 原生字段传递",
    ["--prompt"] = "通过 SDK 原生字段传递",
    ["--interactive"] = "由 provider 管理交互",
};

public static CopilotCliArgumentBuildResult BuildCliArgs(CopilotOptions options)
{
    // 过滤不支持的参数，保留兼容参数
    // 生成诊断信息
}
```

這樣做的好處：

- 自動過濾不相容的 CLI 參數，避免執行階段錯誤——程式崩潰可不是鬧着玩的
- 產生清晰的錯誤診斷資訊，說明開發人員快速定位問題
- 保證 SDK 穩定性，不受 CLI 參數變化的影響

在升級程序中，這個相容性處理機制說明我們平滑過渡，舊的設定檔案仍然可以使用，只需要根據診斷資訊逐步調整即可——也算是個漸進的程序了。

### 執行階段池化

Copilot SDK 的會話建立成本較高，頻繁建立和銷燬會話會影響效能。我們實作了一個會話池管理系統——就像池子裏的水，用完了再裝，不如留着下次接着用：

```csharp
// CopilotProvider.cs - 会话池管理
await using var lease = await _poolCoordinator.AcquireCopilotRuntimeAsync(
    request,
    async ct => await _gateway.CreateRuntimeAsync(sdkRequest, ct),
    cancellationToken);

if (lease.IsWarmLease)
{
    // 复用已有会话
    yield return CreateSessionReusedMessage();
}

await foreach (var eventData in lease.Entry.Resource.SendPromptAsync(...))
{
    yield return MapEvent(eventData);
}
```

會話池化的好處：

- **會話複用**：相同 sessionId 的請求可以複用已有會話，減少啓動開銷
- **支援會話復原**：網路中斷後可以復原之前的會話狀態——畢竟網路這東西，誰敢保證一直穩定呢
- **自動池化管理**：自動清理過期會話，避免資源泄漏

在 HagiCode 的實際使用中，會話池化顯著提升了回應速度，特別是在處理連續對話時效果明顯——這種提升還是能感覺到的。

### Orleans 集成

HagiCode 使用 Orleans 作爲分佈式框架，我們將 Copilot SDK 集成到了 Orleans Grain 中——分佈式這東西，說起來複雜，用起來倒也挺順手：

```csharp
// GitHubCopilotGrain.cs - 分布式执行
public async IAsyncEnumerable<GitHubCopilotResponse> ExecuteCommandStreamAsync(
    string command,
    CancellationToken token = default)
{
    var provider = await aiProviderFactory.GetProviderAsync(AIProviderType.GitHubCopilot);

    await foreach (var chunk in provider.SendMessageAsync(request, null, token))
    {
        // 映射为统一的响应格式
        yield return BuildChunkResponse(chunk, startedAt);
    }
}
```

Orleans 集成帶來的優勢：

- **統一的 AI Provider 抽象**：可以輕鬆切換不同的 AI 提供商——今天用這個，明天用那個，也挺靈活
- **多租戶隔離**：不同使用者的 Copilot 會話相互隔離，井水不犯河水
- **永續性會話狀態**：會話狀態可以跨伺服器重啓復原，重啓也不怕丟資料

對於需要處理大量並行請求的場景，Orleans 的分佈式能力提供了很好的擴充性——畢竟，單機扛不住的時候，只能靠分佈式頂上了。

## 實踐指南

### 設定示例

以下是一個完整的設定示例——直接複製貼上改改就能用：

```json
{
  "AI": {
    "Providers": {
      "Providers": {
        "GitHubCopilot": {
          "Enabled": true,
          "ExecutablePath": "copilot",
          "Model": "gpt-5",
          "WorkingDirectory": "/path/to/project",
          "Timeout": 7200,
          "StartupTimeout": 30,
          "UseLoggedInUser": true,
          "NoAskUser": true,
          "Permissions": {
            "AllowAllTools": false,
            "AllowedTools": ["Read", "Bash", "Grep"],
            "DeniedTools": ["Edit"]
          }
        }
      }
    }
  }
}
```

### 使用注意事項

在實際使用中，我們總結了一些需要注意的地方——有些是踩坑得來的經驗：

**啓動超時設定**：首次啓動 Copilot CLI 需要較長時間，建議設定 `StartupTimeout` 至少 30 秒。如果是首次登入，可能需要更長的時間——畢竟首次登入總得驗證一下，這也沒辦法。

**權限管理**：生產環境避免使用 `AllowAllTools: true`。使用 `AllowedTools` 白名單控制可用工具，使用 `DeniedTools` 黑名單禁止危險操作。這樣可以有效防止 AI 執行危險命令——安全這東西，小心點總是對的。

**會話管理**：相同 `sessionId` 的請求會自動複用會話。會話狀態透過 `ProviderSessionId` 永續性。取消操作透過 `CancellationTokenSource` 傳遞——會話管理做得好，體驗自然就好。

**診斷輸出**：不相容的 CLI 參數會產生 `diagnostic` 型別訊息。原始 SDK 事件以 `event.raw` 型別保留。錯誤資訊包含分類（啓動超時、參數不相容等），方便排查問題——出了問題能快速定位，也算是一種安慰了。

### 最佳實踐

基於我們的實際經驗，這裏分享一些最佳實踐——算是一些總結吧：

**1. 使用工具白名單**

```csharp
var request = new AIRequest
{
    Prompt = "分析这个文件",
    AllowedTools = new[] { "Read", "Grep", "Bash(git:*)" }
};
```

透過白名單明確指定允許的工具，避免 AI 執行意外操作。特別是對於有寫入權限的工具（如 Edit），需要格外謹慎——畢竟刪程式庫這種事，誰也不想經歷。

**2. 設定合理的超時**

```csharp
options.Timeout = 3600;  // 1小时
options.StartupTimeout = 60;  // 1分钟
```

根據任務的複雜度設定合適的超時時間。太短可能導致任務中斷，太長則可能浪費資源等待無回應的請求——凡事適度，過猶不及。

**3. 啓用會話複用**

```csharp
options.SessionId = "my-session-123";
```

爲相關任務設定相同的 sessionId，可以複用之前的會話上下文，提升回應速度——上下文這東西，有時候還挺重要的。

**4. 處理流式回應**

```csharp
await foreach (var chunk in provider.StreamAsync(request))
{
    switch (chunk.Type)
    {
        case StreamingChunkType.ThinkingDelta:
            // 处理推理过程
            break;
        case StreamingChunkType.ToolCallDelta:
            // 处理工具调用
            break;
        case StreamingChunkType.ContentDelta:
            // 处理文本输出
            break;
    }
}
```

流式回應可以實時顯示 AI 的處理進度，提升使用者體驗。特別是對於耗時任務，實時反饋非常重要——看着進度條總比干等着強。

**5. 錯誤處理和重試**

```csharp
try
{
    await foreach (var chunk in provider.StreamAsync(request))
    {
        // 处理响应
    }
}
catch (CopilotSessionException ex)
{
    // 处理会话异常
    logger.LogError(ex, "Copilot session failed");
    // 根据异常类型决定是否重试
}
```

適當的錯誤處理和重試機制可以提升系統的穩定性——誰也不能保證程式永遠不出錯，出了錯能處理好就行。

## 總結

從 CLI 呼叫到 SDK 集成的升級，爲 HagiCode 專案帶來了顯著的價值——怎麼說呢，這次升級還是挺值的：

- **穩定性提升**：SDK 提供了更穩定的介面，不受 CLI 版本變化影響——不用天天擔心版本更新了
- **功能完整性**：能夠捕獲完整的事件流，包括推理程序和工具執行狀態——程序和結果都能看到
- **開發效率**：型別安全的 SDK 介面讓開發更高效，減少執行階段錯誤——有型別檢查，心裏踏實
- **使用者體驗**：實時的事件反饋讓使用者更清晰地瞭解 AI 的工作程序——知道它在想什麼，總比一無所知強

這次升級不僅僅是技術方案的取代，更是對整個 AI 能力層架構的優化。透過分層設計和抽象介面，我們獲得了更好的可維護性和可擴充性——架構做好了，後面的事情就好辦了。

如果你正在考慮將 GitHub Copilot 集成到你的 .NET 專案中，希望本文的實踐經驗能夠說明你少走一些彎路。官方 SDK 確實比 CLI 呼叫更加穩定和完整，值得投入時間去理解和掌握——畢竟，正確的工具能讓事情事半功倍，這話也不是沒有道理的。
