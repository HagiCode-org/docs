## Implementación elegante de guía para nuevos usuarios en proyectos React: La práctica de driver.js de HagiCode

> Cuando los usuarios abren tu producto por primera vez, ¿realmente saben por dónde empezar? Este artículo habla sobre nuestra experiencia usando driver.js para guiar a nuevos usuarios en el proyecto HagiCode, es simplemente una introducción al tema.

## Antecedentes

¿Alguna vez te has encontrado con esta situación: un nuevo usuario se registra en tu producto, abre la página y se queda confundido, mirando a su alrededor, sin saber dónde hacer clic o qué hacer. Como desarrolladores, siempre asumimos que los usuarios "explorarán por sí mismos", después de todo, la curiosidad humana es infinita. Pero la realidad es que la mayoría de los usuarios abandonarán silenciosamente en unos minutos por no encontrar el punto de entrada, como una historia que comienza de repente y termina naturalmente.

La guía para nuevos usuarios es un medio importante para resolver este problema, aunque su implementación no es tan sencilla. Un buen sistema de guía necesita:

- Poder localizar con precisión los elementos de la página y resaltarlos
- Soportar flujos de guía de múltiples pasos
- Poder recordar las elecciones del usuario (completado/omitido)
- No afectar el rendimiento de la página y las interacciones normales
- Estructura de código clara, fácil de mantener

Durante el desarrollo de HagiCode, nos enfrentamos al mismo desafío. HagiCode es un proyecto de asistente de código con IA, cuyo flujo de trabajo central es "usuario crea propuesta → IA genera plan → usuario revisa → IA ejecuta", un proceso OpenSpec. Para los usuarios que se encuentran por primera vez con este concepto, este flujo es completamente nuevo, por lo que debe haber una buena guía para ayudarlos a comenzar rápidamente. Después de todo, las cosas nuevas siempre necesitan un poco de tiempo.

## Sobre HagiCode

La solución compartida en este artículo proviene de nuestra experiencia práctica en el proyecto [HagiCode](https://hagicode.com). HagiCode es un asistente de código con IA basado en Claude que ayuda a los desarrolladores a completar tareas de código de manera más eficiente a través del flujo de trabajo OpenSpec. Puedes ver nuestro código de código abierto en [GitHub](https://github.com/HagiCode-org/site).

## Por qué elegir driver.js

En la etapa de selección técnica, evaluamos varias bibliotecas de guía principales, cada una con sus propias características:

- **Intro.js**: Potente pero con un volumen grande, la personalización de estilos es relativamente compleja
- **Shepherd.js**: Buen diseño de API, pero un poco "pesado" para nuestro escenario
- **driver.js**: Ligero, conciso, API intuitiva, y soporta el ecosistema React

Finalmente elegimos driver.js, sin ninguna razón especial, principalmente basado en las siguientes consideraciones:

1. **Ligero**: El volumen de la biblioteca central es pequeño, no aumenta significativamente el volumen del empaquetado
2. **API concisa**: Las opciones de configuración son claras e intuitivas, rápido de aprender
3. **Flexibilidad**: Soporta posicionamiento personalizado, estilos y comportamientos de interacción
4. **Importación dinámica**: Se puede cargar bajo demanda, no afecta el rendimiento de la primera pantalla

En cuanto a la selección de tecnología, en realidad no hay la mejor, solo la más adecuada.

## Implementación técnica

### Configuración central

La configuración de driver.js es muy intuitiva, a continuación se muestra la configuración central en el proyecto HagiCode:

```typescript
import { driver } from 'driver.js';
import 'driver.js/dist/driver.css';

const newConversationDriver = driver({
  allowClose: true,           // Permite al usuario cerrar la guía
  animate: true,              // Habilita efectos de animación
  overlayClickBehavior: 'close', // Hacer clic en la capa de superposición cierra la guía
  disableActiveInteraction: false, // Mantiene los elementos interactivos
  showProgress: false,        // No muestra la barra de progreso (tenemos gestión de progreso personalizada)
  steps: guideSteps           // Array de pasos de guía
});
```

Las consideraciones detrás de esta configuración son:

- `allowClose: true` - Respeta la elección del usuario, no fuerza completar la guía, después de todo, forzar las cosas no es agradable
- `disableActiveInteraction: false` - Algunos pasos requieren que el usuario realice operaciones reales (como ingresar texto), por lo que no se puede deshabilitar la interacción
- `overlayClickBehavior: 'close'` - Proporciona al usuario una forma rápida de salir

### Gestión de estado

La persistencia del estado de la guía es clave - no queremos guiar nuevamente cada vez que se actualiza la página, eso es bastante molesto. HagiCode usa localStorage para gestionar el estado de la guía:

```typescript
export type GuideState = 'pending' | 'dismissed' | 'completed';

export interface UserGuideState {
  session: GuideState;
  detailGuides: Record<string, GuideState>;
}

// Leer estado
export const getUserGuideState = (): UserGuideState => {
  const state = localStorage.getItem('userGuideState');
  return state ? JSON.parse(state) : { session: 'pending', detailGuides: {} };
};

// Actualizar estado
export const setUserGuideState = (state: UserGuideState) => {
  localStorage.setItem('userGuideState', JSON.stringify(state));
};
```

Definimos tres estados:

- `pending`: La guía está en progreso, el usuario aún no ha completado u omitido
- `dismissed`: El usuario cerró activamente la guía
- `completed`: El usuario completó todos los pasos

Para la guía de la página de detalles de la propuesta, también admitimos un seguimiento de estado más granular (a través del diccionario `detailGuides`), porque una propuesta puede pasar por múltiples etapas (borrador, revisión, ejecución completada), cada etapa necesita diferentes guías. Después de todo, el estado de las cosas siempre está cambiando.

### Localización de elementos objetivo

driver.js usa selectores CSS para localizar elementos objetivo. HagiCode adopta una convención: usar el atributo personalizado `data-guide` para marcar los objetivos de guía:

```typescript
const steps = [
  {
    element: '[data-guide="launch"]',
    popover: {
      title: 'Comenzar nueva conversación',
      description: 'Haz clic aquí para crear una nueva sesión de conversación...'
    }
  }
];
```

Usado así en el componente:

```typescript
<button data-guide="launch" onClick={handleLaunch}>
  Nueva conversación
</button>
```

Las ventajas de este enfoque son:

- Evita conflictos con nombres de clases de estilo comerciales
- Semántica clara, se puede ver a simple vista que este elemento está relacionado con la guía
- Fácil de gestionar y mantener de manera unificada

### Optimización de importación dinámica

Debido a que la función de guía solo se necesita en escenarios específicos (como cuando un nuevo usuario accede por primera vez), usamos importación dinámica para optimizar el rendimiento de carga inicial:

```typescript
const initNewUserGuide = async () => {
  // Importación dinámica de driver.js
  const { driver } = await import('driver.js');
  await import('driver.js/dist/driver.css');

  // Inicializar guía
  const newConversationDriver = driver({
    // ...configuración
  });

  newConversationDriver.drive();
};
```

De esta manera, driver.js y sus archivos de estilo solo se cargarán cuando sea necesario, sin afectar el rendimiento de la primera pantalla. Después de todo, ¿quién quiere pagar el precio de esperar por cosas que aún no se usan?

## Diseño del flujo de guía

HagiCode implementa dos rutas de guía, cubriendo los escenarios de uso principal del usuario.

### Guía de conversación (10 pasos)

Esta guía ayuda al usuario a completar todo el proceso desde crear una conversación hasta enviar la primera propuesta completa:

1. **launch** - Iniciar guía, presentar el botón "Nueva conversación"
2. **compose** - Guiar al usuario para ingresar la solicitud en el cuadro de entrada
3. **send** - Guiar para hacer clic en el botón enviar
4. **proposal-launch-readme** - Guiar para crear propuesta README
5. **proposal-compose-readme** - Guiar para editar contenido de solicitud README
6. **proposal-submit-readme** - Guiar para enviar propuesta README
7. **proposal-launch-agents** - Guiar para crear propuesta AGENTS.md
8. **proposal-compose-agents** - Guiar para editar solicitud AGENTS.md
9. **proposal-submit-agents** - Guiar para enviar propuesta AGENTS.md
10. **proposal-wait** - Explicar que la IA está procesando, espere un momento

La idea de diseño de esta guía es: a través de dos tareas reales de creación de propuestas (README y AGENTS.md), permitir que los usuarios experimenten personalmente el flujo de trabajo central de HagiCode. Después de todo, el conocimiento en papel es superficial, debes practicarlo personalmente para entenderlo realmente.

### Guía de detalles de propuesta (3 pasos)

Cuando el usuario entra en la página de detalles de la propuesta, se activa la guía correspondiente según el estado actual de la propuesta:

1. **drafting** (fase de borrador) - Guía al usuario para ver el plan generado por la IA
2. **reviewing** (fase de revisión) - Guía al usuario para ejecutar el plan
3. **executionCompleted** (fase completada) - Guía al usuario para archivar el plan

Esta guía se caracteriza por ser impulsada por el estado - decide dinámicamente qué paso de guía mostrar según el estado real de la propuesta. Las cosas siempre están cambiando, la guía también debería cambiar con ellas.

### Mecanismo de reintento de renderizado de elementos

En las aplicaciones React, los elementos objetivo de la guía pueden no haber terminado de renderizarse (por ejemplo, esperando la carga de datos asíncronos). Para manejar esta situación, HagiCode implementa un mecanismo de reintento:

```typescript
const waitForElement = (selector: string, maxRetries = 10, interval = 100) => {
  let retries = 0;
  return new Promise<HTMLElement>((resolve, reject) => {
    const checkElement = () => {
      const element = document.querySelector(selector) as HTMLElement;
      if (element) {
        resolve(element);
      } else if (retries < maxRetries) {
        retries++;
        setTimeout(checkElement, interval);
      } else {
        reject(new Error(`Element not found: ${selector}`));
      }
    };
    checkElement();
  });
};
```

Llama a esta función antes de inicializar la guía para asegurar que el elemento objetivo ya existe. A veces, esperar un poco más vale la pena.

## Resumen de mejores prácticas

Basado en la experiencia práctica de HagiCode, aquí compartimos algunas mejores prácticas clave:

### 1. La guía debe ser "escapable"

No fuerces al usuario a completar la guía. Algunos usuarios son exploradores, prefieren descubrir por sí mismos. Proporciona un botón claro de "omitir" y recuerda la elección del usuario, no molestes la próxima vez. Después de todo, las cosas o personas hermosas no tienen que ser poseídas, siempre que sigan siendo hermosas, basta con admirar su belleza.

### 2. El contenido de la guía debe ser conciso y potente

Cada paso de la guía debe enfocarse en un solo objetivo:

- **Title**: Corto y claro, no más de 10 caracteres
- **Description**: Directo al grano, dile al usuario "qué es esto" y "por qué usarlo"

Evita explicaciones largas - la atención de los usuarios en la fase de guía es muy limitada. Si se dicen demasiadas cosas, nadie querrá leerlas.

### 3. Los selectores deben ser estables

Usa métodos de marcado de elementos estables que no cambien con frecuencia. El atributo personalizado `data-guide` es una buena opción, evita depender de nombres de clase o estructuras DOM, ya que estos pueden cambiar fácilmente durante las refactorizaciones. El código siempre está cambiando, pero algunas cosas deben mantenerse estables tanto como sea posible.

### 4. Prueba tu guía

HagiCode escribió casos de prueba completos para la función de guía:

```typescript
describe('NewUserConversationGuide', () => {
  it('debería inicializar correctamente el estado de la guía', () => {
    const state = getUserGuideState();
    expect(state.session).toBe('pending');
  });

  it('debería actualizar correctamente el estado de la guía', () => {
    setUserGuideState({ session: 'completed', detailGuides: {} });
    const state = getUserGuideState();
    expect(state.session).toBe('completed');
  });
});
```

Las pruebas pueden asegurar que al refactorizar el código no se rompa accidentalmente la función de guía. Después de todo, nadie quiere que al cambiar un poco de código se arruine la funcionalidad anterior.

### 5. Optimización del rendimiento

- Usa importación dinámica para cargar la biblioteca de guía de manera diferida
- Evita inicializar la lógica de guía después de que el usuario haya completado la guía
- Considera el impacto en el rendimiento de las animaciones de guía, en dispositivos de gama baja se pueden desactivar las animaciones

El rendimiento, como la vida, debe ahorrarse donde sea necesario.

## Conclusión

La guía para nuevos usuarios es un enlace importante para mejorar la experiencia del usuario del producto. En el proyecto HagiCode, usamos driver.js para construir un sistema de guía completo, cubriendo todo el flujo de trabajo desde la creación de conversaciones hasta la ejecución de propuestas.

A través de este artículo, esperamos transmitir los puntos centrales:

1. **La selección técnica debe coincidir con los requisitos**: driver.js no es el más potente, pero para nosotros es el más adecuado
2. **La gestión del estado es clave**: Usa localStorage para persistir el estado de la guía, evitando molestar repetidamente al usuario
3. **El diseño de la guía debe enfocarse**: Cada paso resuelve un problema, no seas codicioso
4. **La estructura del código debe ser clara**: Separa la configuración de la guía, la gestión del estado y la lógica de UI, facilitando el mantenimiento

Si estás agregando funcionalidad de guía para nuevos usuarios a tu propio proyecto, espero que la experiencia práctica de este artículo te sea útil. En realidad, la tecnología no es nada misterioso, intenta más, resume más, poco a poco mejorará...
