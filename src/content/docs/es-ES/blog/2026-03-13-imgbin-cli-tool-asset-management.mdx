## Diseño de la herramienta CLI ImgBin: Solución de gestión de activos de imágenes de HagiCode

> Este artículo presenta cómo construir desde cero una canalización de activos de imágenes ejecutable automáticamente, incluido el diseño de herramientas CLI, la arquitectura del adaptador de proveedores y las estrategias de gestión de metadatos.

## Antecedentes

Realmente no esperábamos que la gestión de activos de imágenes nos hiciera dudar tanto tiempo.

Durante el desarrollo del proyecto HagiCode, encontramos un problema que parecía simple pero era muy difícil: la generación y gestión de activos de imágenes. ¿Cómo decirlo? Es como esas cosas de la adolescencia: en la superficie todo está tranquilo, pero por debajo hay olas turbulentas.

Con el aumento de la documentación del proyecto y los materiales de marketing, se necesitan muchas imágenes. Algunas de estas imágenes deben ser generadas por IA, otras deben seleccionarse de bibliotecas de materiales existentes y otras deben ser reconocidas automáticamente por IA. El problema es que este trabajo se ha completado durante mucho tiempo con scripts dispersos más operaciones manuales: cada vez que se genera una imagen, se debe ejecutar manualmente el script, organizar manualmente los metadatos y generar manualmente miniaturas. Esto no sería gran cosa, pero lo clave es que estas cosas dispersas están esparcidas por todas partes; cuando quieres encontrarlas, no puedes, y cuando quieres usarlas, no puedes.

Los puntos de dolor específicos incluyen:

1. **Falta de un punto de entrada unificado**: la lógica de generación de imágenes está dispersa en diferentes scripts, no hay forma de ejecutarla por lotes
2. **Falta de metadatos**: las imágenes generadas no tienen un metadata.json unificado, lo que impide la recuperación y el seguimiento
3. **Alto costo de organización manual**: los títulos y etiquetas de las imágenes deben organizarse manualmente uno por uno, lo cual es ineficiente
4. **Imposibilidad de automatización**: ¿quieres generar automáticamente imágenes en el proceso de CI/CD? No hay manera.

También pensamos en ignorarlo, pero después de todo todavía tenemos que hacer el proyecto. Dado que no podemos evitarlo, busquemos una solución. Así que decidimos actualizar ImgBin de "scripts dispersos" a una canalización de activos de imágenes ejecutable automáticamente. Después de todo, para algunas cosas, escapar no es la solución.

## Sobre HagiCode

La solución compartida en este artículo proviene de nuestra experiencia práctica en el proyecto [HagiCode](https://hagicode.com). HagiCode es un proyecto de asistente de código IA que mantiene múltiples componentes como extensiones de VSCode, servicios de IA de backend y clientes de escritorio multiplataforma. En este escenario complejo de múltiples idiomas y plataformas, la gestión estandarizada de activos de imágenes se ha convertido en un enlace clave para mejorar la eficiencia del desarrollo.

¿Cómo decirlo? Esto también se considera una pequeña molestia en el crecimiento de HagiCode. Cada proyecto tendrá tales momentos, problemas pequeños que parecen insignificantes pero que pueden hacer que la gente se preocupen durante mucho tiempo.

El sistema de construcción de HagiCode utiliza el ecosistema TypeScript + Node.js, por lo que ImgBin también eligió naturalmente la misma pila tecnológica para garantizar la coherencia tecnológica de todo el proyecto. Después de todo, estamos acostumbrados a usarlo y cambiar a otro sería problemático.

## Diseño central

### Arquitectura general

ImgBin adopta una arquitectura en capas, separando claramente los comandos CLI, los servicios de aplicación, los adaptadores de API de terceros y la capa de infraestructura:

```
Jerarquía de componentes
├── CLI Entry (cli.ts)              Análisis de parámetros globales, enrutamiento de comandos
├── Commands (commands/*)           generate | batch | annotate | thumbnail
├── Application Services            job-runner | metadata | thumbnail | asset-writer
├── Provider Adapters               image-api-provider | vision-api-provider
└── Infrastructure Layer            config | logger | paths | schema
```

El beneficio de este diseño en capas es: cada capa tiene responsabilidades claras y las dependencias externas se pueden simular fácilmente durante las pruebas. En realidad, es dejar que cada uno haga lo suyo, sin interferir entre sí, para que cuando surja un problema sea fácil encontrar la causa, ¿verdad?

### Modelo de directorio de activo único

ImgBin adopta el modelo de "un activo, un directorio". Cada vez que se genera una imagen, se crea la siguiente estructura:

```
library/
└── 2026-03/
    └── orange-dashboard/
        ├── original.png      # Imagen original
        ├── thumbnail.webp    # Miniatura 512x512
        └── metadata.json     # Metadatos estructurados
```

Las ventajas de este modelo son:

1. **Autónomo**: todos los archivos de cada activo están en el mismo directorio, lo que facilita la migración y la copia de seguridad
2. **Rastreable**: a través de metadata.json se puede rastrear el tiempo de generación de la imagen, el prompt utilizado, el modelo y otra información
3. **Extensible**: en el futuro, si necesita agregar más variantes (como miniaturas de diferentes tamaños), solo necesita agregar nuevos archivos en el mismo directorio

Las cosas o personas hermosas no necesitan ser poseídas, siempre que sigan siendo hermosas, uno puede admirar su belleza. Aunque esto se diga un poco lejos, la lógica es esta: las imágenes juntas se ven bien y son fáciles de encontrar.

### Almacenamiento estratificado de metadatos

metadata.json es el núcleo de todo el sistema. Adopta una estrategia de almacenamiento estratificado que distingue tres tipos de campos:

```json
{
  "schemaVersion": 2,
  "assetId": "orange-dashboard",
  "slug": "orange-dashboard",
  "title": "Orange Dashboard",
  "tags": ["dashboard", "hero", "orange"],

  "source": { "type": "generated" },

  "paths": {
    "assetDir": "library/2026-03/orange-dashboard",
    "original": "original.png",
    "thumbnail": "thumbnail.webp"
  },

  "generated": {
    "prompt": "orange dashboard for docs hero",
    "provider": "azure-openai-image-api",
    "model": "gpt-image-1.5"
  },

  "recognized": {
    "title": "Orange Dashboard",
    "tags": ["dashboard", "ui", "orange"],
    "description": "A modern orange dashboard with charts and metrics"
  },

  "status": {
    "generation": "succeeded",
    "recognition": "succeeded",
    "thumbnail": "succeeded"
  },

  "timestamps": {
    "createdAt": "2026-03-11T04:01:19.570Z",
    "updatedAt": "2026-03-11T04:02:09.132Z"
  }
}
```

- **generated**: registra la información original cuando se genera la imagen, como el prompt utilizado, el proveedor, el modelo, etc.
- **recognized**: resultado del reconocimiento de IA, como título, etiquetas y descripción generados automáticamente
- **manual**: resultado de la organización manual, los datos en esta área tienen la prioridad más alta y no serán sobrescritos por el reconocimiento de IA

Esta estrategia estratificada resuelve una contradicción central que teníamos antes: ¿qué tiene prioridad, el resultado del reconocimiento de IA o el resultado de la organización manual? La respuesta es que la organización manual tiene prioridad y el reconocimiento de IA es solo auxiliar. Esto también quedó claro: algunas cosas, la máquina es después de todo una máquina, y al final debe ser revisada por humanos.

## Patrón de adaptador de proveedor

Otro diseño central de ImgBin es el patrón de adaptador de proveedor. Abstraemos las API externas como una interfaz unificada, de modo que incluso si cambiamos de proveedor de servicios de IA, no necesitamos modificar la lógica empresarial.

¿Cómo decirlo? Es como las relaciones: no importa cómo cambie la apariencia, lo importante es que el interior no cambie. Una vez definida la interfaz, se puede cambiar la implementación interna.

### Proveedor de generación de imágenes

```typescript
interface ImageGenerationProvider {
  // Genera imágenes, devuelve el Buffer de la imagen
  generate(options: GenerateOptions): Promise<Buffer>;

  // Obtiene la lista de modelos admitidos
  getSupportedModels(): Promise<string[]>;
}

interface GenerateOptions {
  prompt: string;
  model?: string;
  size?: '1024x1024' | '1792x1024' | '1024x1792';
  quality?: 'standard' | 'hd';
  format?: 'png' | 'webp' | 'jpeg';
}
```

### Proveedor de reconocimiento visual

```typescript
interface VisionRecognitionProvider {
  // Reconoce el contenido de la imagen, devuelve metadatos estructurados
  recognize(imageBuffer: Buffer): Promise<RecognitionResult>;

  // Obtiene la lista de modelos admitidos
  getSupportedModels(): Promise<string[]>;
}

interface RecognitionResult {
  title?: string;
  tags: string[];
  description?: string;
  confidence: number;
}
```

Las ventajas de este diseño de interfaz son:

1. **Probable**: en las pruebas unitarias se puede pasar un proveedor simulado sin llamar realmente a la API externa
2. **Extensible**: agregar un nuevo proveedor solo necesita implementar la interfaz, sin modificar el código de la persona que llama
3. **Reemplazable**: en producción usar Azure OpenAI, en pruebas usar modelos locales, solo necesita cambiar la configuración

Querer reír para disimular las lágrimas que caen, querer llorar para probar si uno está entumecido: a veces hacer proyectos es así, superficialmente parece que se cambió una API, pero en realidad la lógica interna no ha cambiado en absoluto, así que no hay nada de qué temer.

## Diseño de comandos CLI

ImgBin proporciona cuatro comandos centrales para satisfacer diferentes escenarios de uso:

### generate: Generación de imagen única

```bash
# El uso más simple
imgbin generate --prompt "orange dashboard for docs hero"

# Genera miniaturas y anotaciones de IA al mismo tiempo
imgbin generate --prompt "orange dashboard" --annotate --thumbnail

# Especifica el directorio de salida
imgbin generate --prompt "orange dashboard" --output ./library
```

### batch: Tareas por lotes

Las tareas por lotes se definen mediante archivos manifiesto YAML o JSON, adecuados para usar en procesos de CI/CD:

```yaml
# assets/jobs/launch.yaml
defaults:
  annotate: true
  thumbnail: true
  libraryRoot: ./library

jobs:
  - prompt: "orange dashboard hero"
    slug: orange-dashboard
    tags: [dashboard, hero, orange]

  - prompt: "pricing grid for docs"
    slug: pricing-grid
    tags: [pricing, grid, docs]
```

Comando de ejecución:

```bash
imgbin batch assets/jobs/launch.yaml
```

El diseño de tareas por lotes admite el aislamiento de fallos: procesamiento elemento por elemento en el manifiesto, el fallo de un elemento no afecta a otras tareas. Se puede previsualizar con `--dry-run` sin ejecutar realmente.

Esto no sería gran cosa, pero lo clave es que también puede decirte dónde tuvo éxito y dónde falló, a diferencia de algunas cosas, cuando fallas ni siquiera sabes cómo fallaste.

### annotate: Anotación de IA

Ejecuta reconocimiento de IA en imágenes existentes para generar automáticamente títulos, etiquetas y descripciones:

```bash
# Anotar una sola imagen
imgbin annotate ./library/2026-03/orange-dashboard

# Anotar todo el directorio por lotes
imgbin annotate ./library/2026-03/
```

### thumbnail: Generación de miniaturas

Genera miniaturas para imágenes existentes:

```bash
# Generar miniatura
imgbin thumbnail ./library/2026-03/orange-dashboard
```

## Diseño de manifiesto de tareas por lotes

El manifiesto de tareas por lotes admite una configuración flexible. Los valores predeterminados se pueden establecer de manera uniforme y las tareas individuales también se pueden anular:

```yaml
# Valores predeterminados globales
defaults:
  annotate: true        # Habilita anotación de IA por defecto
  thumbnail: true       # Genera miniaturas por defecto
  libraryRoot: ./library
  model: gpt-image-1.5

jobs:
  # Configuración mínima, solo proporciona prompt
  - prompt: "first image"

  # Configuración completa
  - prompt: "second image"
    slug: custom-slug
    tags: [tag1, tag2]
    annotate: false     # Esta tarea no ejecuta anotación de IA
    model: dall-e-3    # Esta tarea usa un modelo diferente
```

Durante la ejecución, ImgBin procesa las tareas una por una. El resultado de cada tarea se escribe en el metadata.json correspondiente. Incluso si una tarea falla, no afectará a otras tareas. Después de completar las tareas, se generará un informe resumido:

```
✓ orange-dashboard (succeeded)
✓ pricing-grid (succeeded)
✗ hero-banner (failed: API rate limit exceeded)

2/3 succeeded, 1 failed
```

Algunas cosas no se pueden apresurar, una por una es más seguro. Esto, quizás, es la filosofía de las tareas por lotes.

## Configuración de variables de entorno

ImgBin admite una configuración flexible a través de variables de entorno:

```bash
# Directorio de trabajo de ImgBin
IMGBIN_WORKDIR=/path/to/imgbin

# Ruta del archivo ejecutable (para llamadas en scripts)
IMGBIN_EXECUTABLE=/path/to/imgbin/dist/cli.js

# Directorio raíz de la biblioteca de activos
IMGBIN_LIBRARY_ROOT=./.imgbin-library

# Configuración de Azure OpenAI (si usa el proveedor de Azure)
AZURE_OPENAI_ENDPOINT=https://your-resource.openai.azure.com/
AZURE_OPENAI_API_KEY=your-api-key
AZURE_OPENAI_IMAGE_DEPLOYMENT=gpt-image-1
```

La configuración, se podría decir que es importante y que no lo es. Después de todo, haz lo que te sea cómodo, lo que te convenga es lo mejor.

## Puntos clave de implementación

Durante el proceso de implementación, resumimos los siguientes puntos clave:

### Diseño de interfaz de proveedor

La definición de la interfaz debe ser clara y completa, incluidos los parámetros de entrada, los valores de retorno y el manejo de errores. Se recomienda proporcionar métodos de llamada síncronos y asíncronos para facilitar diferentes escenarios de uso.

Esto también es algo de experiencia de los que ya pasaron por esto, después de todo, una vez que se define la interfaz, no se quiere cambiar, es problemático.

### Estrategia de manejo de fallos

Cuando una tarea falla en una tarea por lotes, se debe:

1. Registrar información detallada del error en un archivo de registro separado
2. Continuar ejecutando otras tareas sin interrumpir todo el proceso
3. Finalmente devolver un código de salida distinto de cero, indicando que hay tareas fallidas
4. Mostrar claramente el resultado de ejecución de cada tarea en el informe resumido

Algunas cosas fallan y fallan, escapar no sirve, mejor admitirlo con franqueza y luego encontrar una solución. Esta lógica es la misma para hacer proyectos y para ser persona.

### Estrategia de fusión de metadatos

Los resultados de reconocimiento se escriben por defecto en el área `recognized`, y los campos modificados manualmente tienen la marca `manual`. Al actualizar metadatos se adopta la estrategia de "solo agregar, no eliminar": a menos que se pase explícitamente el parámetro `--force`, no se sobrescribirán los resultados de organización manual existentes.

Esto también quedó claro: algunas cosas, si se pierden, se pierden, si se sobrescriben, desaparecen. Es mejor mantenerlas, después de todo, el registro en sí también es una belleza.

### Atomicidad en la creación de directorios

Use `fs.mkdir({ recursive: true })` para garantizar la atomicidad en la creación de directorios, evitando condiciones de carrera en escenarios concurrentes.

Esto es probablemente lo que se llama sentido de seguridad: lo que debe ser estable, estable; lo que debe ser rápido, rápido; sin arrastrar los pies ni mirar hacia adelante y hacia atrás.

## Conclusión

Como herramienta central de gestión de activos de imágenes del proyecto HagiCode, ImgBin resolvió los problemas que enfrentamos a través de los siguientes diseños:

1. **Punto de entrada unificado**: los comandos CLI cubren todas las operaciones de generación, anotación y miniaturas
2. **Impulsado por metadatos**: cada activo tiene un metadata.json completo que admite recuperación y seguimiento
3. **Adaptador de proveedor**: abstracción flexible de API externa, fácil de probar y extender
4. **Soporte de tareas por lotes**: puede ejecutar automáticamente la generación de imágenes por lotes en el proceso de CI/CD
