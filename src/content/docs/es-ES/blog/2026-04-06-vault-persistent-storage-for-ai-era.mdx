## Cómo estudiar proyectos en la era de la IA: Sistema de almacenamiento persistente entre proyectos Vault

> En la era del desarrollo asistido por IA, ¿cómo hacer que los asistentes de IA comprendan mejor nuestros recursos de aprendizaje? El proyecto HagiCode implementa, a través del sistema Vault, una capa de abstracción de almacenamiento de conocimiento unificada y comprensible para la IA, mejorando significativamente la eficiencia del aprendizaje mediante el estudio de proyectos.

## Antecedentes

En la era de la IA, la forma en que los desarrolladores aprenden nuevas tecnologías y arquitecturas está experimentando cambios profundos. "Estudiar proyectos" — es decir, investigar y aprender a fondo el código, la arquitectura y los patrones de diseño de proyectos de código abierto excelentes — se ha convertido en un método de aprendizaje eficiente. En comparación con la lectura tradicional de libros o ver videos, leer y ejecutar directamente proyectos de código abierto de alta calidad te permite comprender más rápido las prácticas de ingeniería del mundo real.

Sin embargo, este método de aprendizaje también enfrenta numerosos desafíos.

Los materiales de aprendizaje están demasiado dispersos. Tus notas pueden estar en Obsidian, los repositorios de código esparcidos en diferentes carpetas, y el historial de conversaciones del asistente de IA es otra isla de datos independiente. Cuando quieres que la IA te ayude a analizar un proyecto, tienes que copiar manualmente fragmentos de código y organizar el contexto, un proceso bastante tedioso.

En realidad, lo más problemático es la discontinuidad del contexto. El asistente de IA no puede acceder directamente a tus recursos de aprendizaje locales; cada conversación requiere volver a proporcionar información de antecedentes. Además, los repositorios de código que se estudian se actualizan muy rápido, la sincronización manual es propensa a errores, y es difícil compartir conocimiento entre múltiples proyectos de aprendizaje.

Esencialmente, todos estos problemas son causados por "islas de datos". Si pudiera haber una capa de abstracción de almacenamiento unificada que permita al asistente de IA comprender y acceder a todos tus recursos de aprendizaje, el problema estaría resuelto.

## Sobre HagiCode

El sistema Vault que se comparte en este artículo es precisamente la solución que hemos practicado durante el desarrollo de [HagiCode](https://hagicode.com). HagiCode es un proyecto de asistente de código con IA. En nuestro desarrollo diario, a menudo necesitamos aprender y referirnos a varios proyectos de código abierto. Para que el asistente de IA comprenda mejor estos recursos de aprendizaje, diseñamos el sistema de almacenamiento persistente entre proyectos Vault.

Esta solución ya ha sido verificada en la práctica en HagiCode. Si también enfrentas problemas similares de gestión del conocimiento, espero que estas experiencias te sirvan de inspiración. Después de todo, algunos obstáculos ya se han superado y siempre es bueno dejar algo para quienes vengan después.

## Filosofía de diseño del sistema Vault

La idea central del sistema Vault es muy simple: crear una capa de abstracción de almacenamiento de conocimiento unificada y comprensible para la IA. Desde una perspectiva de implementación, el sistema tiene varias características clave.

### Soporte de múltiples tipos

El sistema admite cuatro tipos de vault, correspondientes respectivamente a diferentes escenarios de uso:

```typescript
// folder: tipo de carpeta de uso general
export const DEFAULT_VAULT_TYPE = 'folder';

// coderef: tipo especializado para estudiar proyectos de código
export const CODEREF_VAULT_TYPE = 'coderef';

// obsidian: integración con el software de notas Obsidian
export const OBSIDIAN_VAULT_TYPE = 'obsidian';

// system-managed: vault administrado automáticamente por el sistema
export const SYSTEM_MANAGED_VAULT_TYPE = 'system-managed';
```

Entre ellos, el tipo **coderef** es el más utilizado en HagiCode. Está diseñado específicamente para estudiar proyectos de código y proporciona una estructura de directorios estandarizada y descripciones de metadatos legibles por IA.

### Mecanismo de almacenamiento persistente

El registro de Vault se almacena de forma persistente en formato JSON, asegurando que la configuración siga disponible después de reiniciar la aplicación:

```csharp
public class VaultRegistryStore : IVaultRegistryStore
{
    private readonly string _registryFilePath;

    public VaultRegistryStore(IConfiguration configuration, ILogger<VaultRegistryStore> logger)
    {
        var dataDir = configuration["DataDir"] ?? "./data";
        var absoluteDataDir = Path.IsPathRooted(dataDir)
            ? dataDir
            : Path.GetFullPath(Path.Combine(Directory.GetCurrentDirectory(), dataDir));

        _registryFilePath = Path.Combine(absoluteDataDir, "personal-data", "vaults", "registry.json");
    }
}
```

El beneficio de este diseño es la simplicidad y confiabilidad. El formato JSON es legible por humanos, lo que facilita la depuración y la modificación manual; el almacenamiento en el sistema de archivos evita la complejidad de una base de datos y reduce las dependencias del sistema. Después de todo, a veces lo simple es lo mejor.

### Integración del contexto de IA

Lo más crucial es que el sistema puede inyectar automáticamente la información del vault en el contexto de las propuestas de IA:

```typescript
export function buildTargetVaultsText(
  vaults: VaultForText[],
  template: VaultPromptTemplate = DEFAULT_VAULT_PROMPT_TEMPLATE,
): string {
  const readOnlyVaults = vaults.filter((vault) => vault.accessType === 'read');
  const editableVaults = vaults.filter((vault) => vault.accessType === 'write');

  if (readOnlyVaults.length === 0 && editableVaults.length === 0) {
    return '';
  }

  const sections = [
    buildVaultSection(readOnlyVaults, template.reference),
    buildVaultSection(editableVaults, template.editable),
  ].filter(Boolean);

  return `\n\n### ${template.heading}\n\n${sections.join('\n')}`;
}
```

Así se implementa una función importante: el asistente de IA puede compender automáticamente los recursos de aprendizaje disponibles, sin que el usuario necesite proporcionar contexto manualmente. Podríamos decir que es una especie de entendimiento tácito.

## Estructura estandarizada de CodeRef Vault

Para los vault de tipo coderef, HagiCode proporciona una estructura de directorios estandarizada:

```
my-coderef-vault/
├── index.yaml          # descripción de metadatos del vault
├── AGENTS.md           # guía de operación del asistente de IA
├── docs/               # almacena notas de aprendizaje y documentos
└── repos/              # gestiona repositorios de código estudiados a través de submódulos Git
```

Al crear un vault, el sistema inicializa automáticamente esta estructura:

```csharp
private async Task EnsureCodeRefStructureAsync(
    string vaultName,
    string physicalPath,
    ICollection<VaultBootstrapDiagnosticDto> diagnostics,
    CancellationToken cancellationToken)
{
    Directory.CreateDirectory(physicalPath);

    var indexPath = Path.Combine(physicalPath, CodeRefIndexFileName);
    var docsPath = Path.Combine(physicalPath, CodeRefDocsDirectoryName);
    var reposPath = Path.Combine(physicalPath, CodeRefReposDirectoryName);

    // crear estructura de directorios estándar
    if (!Directory.Exists(docsPath))
    {
        Directory.CreateDirectory(docsPath);
    }

    if (!Directory.Exists(reposPath))
    {
        Directory.CreateDirectory(reposPath);
    }

    // crear guía AGENTS.md
    await EnsureCodeRefAgentsDocumentAsync(physicalPath, cancellationToken);

    // crear metadatos index.yaml
    await WriteCodeRefIndexDocumentAsync(indexPath, mergedDocument, cancellationToken);
}
```

El diseño de esta estructura también tiene su razón de ser:

- El directorio **docs/** almacena tus notas de aprendizaje, en las que puedes registrar tu comprensión del código, análisis de arquitectura, experiencias con obstáculos, etc., en formato Markdown
- El directorio **repos/** gestiona los repositorios estudiados a través de submódulos Git, en lugar de copiar el código directamente. Esto mantiene el código sincronizado y ahorra espacio
- **index.yaml** contiene los metadatos del vault, permitiendo que el asistente de IA comprenda rápidamente el propósito y contenido de este vault
- **AGENTS.md** es una guía escrita específicamente para el asistente de IA, explicando cómo procesar el contenido de este vault

Quizás así organizado, la IA también pueda comprender mejor tus ideas.

## Inicialización automática gestionada por el sistema

Además de crear vault manualmente, HagiCode también admite vault gestionados automáticamente por el sistema:

```csharp
public async Task<IReadOnlyList<VaultRegistryEntry>> EnsureAllSystemManagedVaultsAsync(
    CancellationToken cancellationToken = default)
{
    var definitions = GetAllResolvedDefinitions();
    var entries = new List<VaultRegistryEntry>(definitions.Count);

    foreach (var definition in definitions)
    {
        entries.Add(await EnsureResolvedSystemManagedVaultAsync(definition, cancellationToken));
    }

    return entries;
}
```

El sistema crea y gestiona automáticamente los siguientes vault:

- **hagiprojectdata**: almacenamiento de datos del proyecto, utilizado para guardar la configuración y el estado del proyecto
- **personaldata**: almacenamiento de datos personales, utilizado para guardar las preferencias del usuario
- **hbsprompt**: biblioteca de plantillas de prompts, utilizada para gestionar los prompts de IA de uso común

Estos vault se inicializan automáticamente al iniciar el sistema, sin que el usuario necesite configurarlos manualmente. Después de todo, algunas cosas son mejor dejarlas al sistema, ¿para qué preocuparse los humanos?

## Mecanismo de control de acceso

Un diseño importante es el control de acceso. El sistema divide los vault en dos tipos de acceso:

```typescript
export interface VaultForText {
  id: string;
  name: string;
  type: string;
  physicalPath: string;
  accessType: 'read' | 'write';  // clave: distingue entre solo lectura y editable
}
```

- **reference (solo lectura)**: la IA solo lo usa para analizar y comprender, no puede modificar el contenido. Es adecuado para proyectos de código abierto de referencia, documentación, etc.
- **editable (editable)**: la IA puede modificar el contenido según las necesidades de la tarea. Es adecuado para tus notas, borradores, etc.

Esta distinción es importante. Permite que la IA sepa qué contenido es "referencia de solo lectura" y qué contenido es "puede modificarse", evitando riesgos de operación incorrecta. Después de todo, nadie quiere que su trabajo se pierda inadvertidamente.

## Práctica: crear y usar Vault

Después de ver la teoría, veamos cómo usarlo en la práctica.

### Crear CodeRef Vault

A continuación, un ejemplo completo de llamada desde el frontend:

```typescript
const createCodeRefVault = async () => {
  const response = await VaultService.postApiVaults({
    requestBody: {
      name: "React Learning Vault",
      type: "coderef",
      physicalPath: "/Users/developer/vaults/react-learning",
      gitUrl: "https://github.com/facebook/react.git"
    }
  });

  // el sistema automáticamente:
  // 1. clona el repositorio React en vault/repos/react
  // 2. crea el directorio docs/ para notas
  // 3. genera metadatos index.yaml
  // 4. crea el archivo de guía AGENTS.md

  return response;
};
```

Esta llamada a la API completa una serie de operaciones: crear estructura de directorios, inicializar submódulos Git, generar archivos de metadatos, etc. Solo necesitas proporcionar información básica, el sistema se encarga del resto. En realidad, esto también es bastante conveniente.

## Mejores prácticas y consideraciones

Durante el uso del sistema Vault, hemos resumido algunas experiencias y lecciones.

### Seguridad de rutas

El sistema valida estrictamente las rutas para prevenir ataques de salto de ruta:

```csharp
private static string ResolveFilePath(string vaultRoot, string relativePath)
{
    var rootPath = EnsureTrailingSeparator(Path.GetFullPath(vaultRoot));
    var combinedPath = Path.GetFullPath(Path.Combine(rootPath, relativePath));
    if (!combinedPath.StartsWith(rootPath, StringComparison.OrdinalIgnoreCase))
    {
        throw new BusinessException(VaultRelativePathTraversalCode,
            "Vault file paths must stay inside the registered vault root.");
    }
    return combinedPath;
}
```

Esto es muy importante. Si personalizas la ruta del vault, asegúrate de que la ruta esté dentro del rango permitido; de lo contrario, el sistema rechazará la operación. La seguridad es algo que nunca se puede enfatizar lo suficiente.

### Gestión de submódulos Git

CodeRef vault recomienda usar submódulos Git en lugar de copiar el código directamente:

```csharp
private static string BuildCodeRefAgentsContent()
{
    return """
    # CodeRef Vault Guide

    Repositories under `repos/` should be maintained through Git submodules
    rather than copied directly into the vault root.

    Keep this structure stable so assistants and tools can understand the vault quickly.
    """ + Environment.NewLine;
}
```

Hacer esto tiene varios beneficios: mantiene el código sincronizado con upstream, ahorra espacio en disco y facilita la gestión de múltiples versiones del código. Después de todo, ¿quién querría descargar lo mismo una y otra vez?

### Límites de vista previa de archivos

Para prevenir problemas de rendimiento, el sistema limita el tamaño y tipo de archivos:

```csharp
private const int FileEnumerationLimit = 500;
private const int PreviewByteLimit = 256 * 1024;  // 256KB
```

Si tu vault contiene una gran cantidad de archivos o archivos muy grandes, puede afectar el rendimiento de la función de vista previa. En este caso, puedes considerar procesar por lotes o usar herramientas de búsqueda especializadas. Después de todo, algunas cosas demasiado grandes son difíciles de procesar.

### Información de diagnóstico

Al crear un vault se devuelve información de diagnóstico para ayudar en la depuración:

```csharp
List<VaultBootstrapDiagnosticDto> bootstrapDiagnostics = [];

if (IsCodeRefVaultType(normalizedType))
{
    bootstrapDiagnostics = await EnsureCodeRefBootstrapAsync(
        normalizedName,
        normalizedPhysicalPath,
        normalizedGitUrl,
        cancellationToken);
}
```

Si la creación falla, puedes consultar la información de diagnóstico para conocer la causa específica. Cuando algo sale mal, mira la información de diagnóstico; también es una forma de resolver el problema.
