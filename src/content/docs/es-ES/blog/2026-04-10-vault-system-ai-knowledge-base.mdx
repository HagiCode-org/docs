## Construyendo una base de conocimientos multi-proyecto para la era de la IA con el sistema Vault

> El método de aprendizaje por imitación de proyectos se está convirtiendo en mainstream, pero los puntos de dolor de materiales de aprendizaje dispersos y contextos rotos dificultan que los asistentes de IA cumplan su máximo valor. Este artículo presenta el diseño del sistema Vault del proyecto HagiCode: a través de una capa de abstracción de almacenamiento unificada, permite que los asistentes de IA entiendan y accedan a todos los recursos de aprendizaje, logrando una verdadera reutilización de conocimientos entre proyectos.

## Antecedentes

De hecho, en la era de la IA, nuestra forma de aprender nuevas tecnologías está cambiando silenciosamente. La lectura tradicional y ver videos siguen siendo importantes, pero "imitar proyectos": estudiar y aprender profundamente el código, arquitectura y patrones de diseño de proyectos open source excelentes, es cada vez más eficiente. Ejecutar y modificar directamente proyectos open source de alta calidad te permite entender las prácticas de ingeniería del mundo real más rápido.

Sin embargo, este enfoque también trae nuevos desafíos.

**Los materiales de aprendizaje están demasiado dispersos**. Las notas pueden estar en Obsidian, los repositorios de código esparcidos en varias carpetas, y el historial de conversaciones del asistente de IA es otro islote de datos independiente. Cada vez que necesitas que la IA ayude a analizar un proyecto, tienes que copiar manualmente fragmentos de código y organizar el contexto, un proceso bastante tedioso.

**El contexto se rompe frecuentemente**. Los asistentes de IA no pueden acceder directamente a los recursos de aprendizaje locales, cada conversación debe proporcionar información de fondo repetidamente. Los repositorios de código imitados se actualizan rápido, la sincronización manual es propensa a errores. Peor aún, es difícil compartir conocimiento entre múltiples proyectos de aprendizaje: los patrones de diseño aprendidos en el proyecto A, la IA procesando el proyecto B no lo sabe en absoluto.

La esencia de estos problemas son los "islotes de datos". Si puede haber una capa de abstracción de almacenamiento unificada que permita a los asistentes de IA entender y acceder a todos los recursos de aprendizaje, el problema se resuelve fácilmente.

Para resolver estos puntos de dolor, al desarrollar HagiCode tomamos una decisión de diseño clave: construir un sistema Vault como una capa de abstracción de almacenamiento de conocimiento unificada. El cambio traído por esta decisión puede ser mayor de lo imaginado: hablaremos de eso más adelante.

## Sobre HagiCode

La solución compartida en este artículo proviene de la experiencia práctica en el proyecto [HagiCode](https://hagicode.com). HagiCode es un asistente de código basado en el flujo de trabajo OpenSpec, su concepto central es hacer que la IA no solo sepa "decir", sino también "hacer": puede operar directamente repositorios de código, ejecutar comandos, ejecutar pruebas. GitHub: [github.com/HagiCode-org/site](https://github.com/HagiCode-org/site)

Durante el desarrollo, descubrimos que el asistente de IA necesita acceder frecuentemente a varios recursos de aprendizaje del usuario: repositorios de código, documentos de notas, archivos de configuración, etc. Si cada vez requiere que el usuario lo proporcione manualmente, la experiencia es demasiado terrible. Esto impulsó el diseño del sistema Vault.

## Diseño central

### Soporte multi-tipo

El sistema Vault de HagiCode soporta cuatro tipos, correspondientes respectivamente a diferentes escenarios de uso:

| Tipo | Uso | Escenario típico |
|------|------|----------|
| `folder` | Tipo de carpeta general | Materiales de aprendizaje temporales, borradores |
| `coderef` | Especialmente para imitar proyectos de código | Aprendizaje sistemático de algún proyecto open source |
| `obsidian` | Integración con el software de notas Obsidian | Reutilización de bibliotecas de notas existentes |
| `system-managed` | Gestión automática del sistema | Configuración de proyectos, plantillas de prompts, etc. |

Entre ellos, el tipo `coderef` es el más usado en HagiCode, proporciona una estructura de directorios estandarizada y descripciones de metadatos legibles por IA para proyectos de código imitados. ¿Por qué diseñar específicamente este tipo? Porque imitar un proyecto open source no es simplemente "descargar código", necesita gestionar simultáneamente el código en sí, notas de aprendizaje, archivos de configuración y otros contenidos, `coderef` estandariza todo esto.

### Mecanismo de almacenamiento persistente

El registro de Vault se almacena persistentemente en formato JSON en el sistema de archivos:

```csharp
_registryFilePath = Path.Combine(absoluteDataDir, "personal-data", "vaults", "registry.json");
```

Este diseño parece simple, pero en realidad está cuidadosamente considerado:

**Simple y confiable**. El formato JSON es legible por humanos, facilita la depuración y modificación manual. Cuando el sistema tiene problemas, puede abrir directamente el archivo para ver el estado, incluso repararlo manualmente: esto es particularmente útil en la etapa de desarrollo.

**Reducir dependencias**. El almacenamiento del sistema de archivos evita la complejidad de la base de datos. No necesita instalar y configurar adicionalmente servicios de base de datos, reduciendo la complejidad del sistema y los costos de mantenimiento.

**Seguridad de concurrencia**. Usa `SemaphoreSlim` para garantizar la seguridad de multi-hilos. En el escenario del asistente de código IA, pueden haber múltiples operaciones accediendo simultáneamente al registro de vaults, necesita hacer un buen control de concurrencia.

### Integración de contexto de IA

La capacidad central del sistema reside en poder inyectar automáticamente información de vaults en el contexto de la propuesta de IA:

```typescript
export function buildTargetVaultsText(
  vaults: VaultForText[],
  template: VaultPromptTemplate = DEFAULT_VAULT_PROMPT_TEMPLATE,
): string {
  const readOnlyVaults = vaults.filter((vault) => vault.accessType === 'read');
  const editableVaults = vaults.filter((vault) => vault.accessType === 'write');

  const sections = [
    buildVaultSection(readOnlyVaults, template.reference),
    buildVaultSection(editableVaults, template.editable),
  ].filter(Boolean);

  return `\n\n### ${template.heading}\n\n${sections.join('\n')}`;
}
```

Así el asistente de IA puede entender automáticamente los recursos de aprendizaje disponibles, sin que el usuario tenga que proporcionar contexto manualmente cada vez. Este diseño hace que la experiencia de HagiCode sea particularmente natural: dile a la IA "ayúdame a analizar el renderizado concurrente de React", la IA puede encontrar automáticamente el vault de aprendizaje de React registrado previamente, en lugar de pegar código una y otra vez.

### Mecanismo de control de acceso

El sistema divide vaults en dos tipos de acceso:

- **reference (solo lectura)**: IA solo se usa para analizar y entender, no puede modificar el contenido
- **editable (editable)**: IA puede modificar el contenido según las necesidades de la tarea

Esta distinción permite que la IA sepa qué contenido es "referencia de solo lectura" y qué es "que se puede modificar", evitando el riesgo de operaciones incorrectas. Por ejemplo, si registras un vault de un proyecto open source como material de aprendizaje, definitivamente no quieres que la IA modifique el código dentro a voluntad: entonces márcalo como `reference`. Pero si es tu propio proyecto vault, puedes marcarlo como `editable`, dejando que la IA ayude a modificar el código.

## Guía práctica

### Estructura estandarizada de CodeRef Vault

Para vaults de tipo coderef, el sistema proporciona un conjunto de estructura de directorios estandarizada:

```
my-coderef-vault/
├── index.yaml          # descripción de metadatos del vault
├── AGENTS.md           # guía de operación del asistente de IA
├── docs/               # almacena notas de aprendizaje y documentos
└── repos/              # gestiona repositorios de código imitados a través de submódulos Git
```

¿Cuál es la filosofía de diseño de esta estructura?

**docs/** Almacena notas de aprendizaje, usando formato Markdown para registrar la comprensión del código, análisis de arquitectura, experiencia de pitfalls. Estas notas no solo las lees tú, la IA también puede entenderlas: al procesar tareas relacionadas se referirá automáticamente.

**repos/** Gestiona repositorios imitados a través de submódulos Git, en lugar de copiar código directamente. Hacer esto tiene dos beneficios: uno es mantener la sincronización con upstream, un `git submodule update` puede obtener el último código; dos es ahorrar espacio, múltiples vaults pueden referenciar diferentes versiones del mismo repositorio.

**index.yaml** Contiene metadatos del vault, permitiendo que el asistente de IA entienda rápidamente el propósito y contenido. Equivale a escribir una "auto-presentación" para el vault, la IA la primera vez que lo ve sabe para qué sirve.

**AGENTS.md** Es una guía escrita específicamente para el asistente de IA, explicando cómo procesar el contenido en el vault. Puedes decirle aquí a la IA: "al analizar este proyecto, enfócate en código relacionado con optimización de rendimiento" o "no modifiques archivos de prueba".

### Crear y usar Vault

Crear un CodeRef vault es simple:

```typescript
const createCodeRefVault = async () => {
  const response = await VaultService.postApiVaults({
    requestBody: {
      name: "React Learning Vault",
      type: "coderef",
      physicalPath: "/Users/developer/vaults/react-learning",
      gitUrl: "https://github.com/facebook/react.git"
    }
  });

  // El sistema automáticamente:
  // 1. Clona el repositorio React a vault/repos/react
  // 2. Crea el directorio docs/ para notas
  // 3. Genera metadatos index.yaml
  // 4. Crea el archivo de guía AGENTS.md

  return response;
};
```

Luego referencia este vault en la propuesta de IA:

```typescript
const proposal = composeProposalChiefComplaint({
  chiefComplaint: "ayúdame a analizar el mecanismo de renderizado concurrente de React",
  repositories: [
    { id: "react", gitUrl: "https://github.com/facebook/react.git" }
  ],
  vaults: [
    {
      id: "react-learning",
      name: "React Learning Vault",
      type: "coderef",
      physicalPath: "/vaults/react-learning",
      accessType: "read"  // IA solo puede leer, no modificar
    }
  ],
  quickRequestText: "enfócate en la arquitectura fiber y la implementación del scheduler"
});
```

### Escenarios de uso típicos

**Escenario uno: aprendizaje sistemático de proyectos open source**

Crea un CodeRef vault, gestiona el repositorio objetivo a través de submódulos Git, registra notas de aprendizaje en el directorio `docs/`. La IA puede acceder simultáneamente al código y las notas, proporcionando un análisis más preciso. Las notas escritas al aprender un módulo, la IA se referirá automáticamente al analizar código relacionado posteriormente: como tener un "asistente" que recuerda el pensamiento previo.

**Escenario dos: reutilizar biblioteca de notas de Obsidian**

Si ya estás usando Obsidian para gestionar notas, solo registra el vault existente en HagiCode. La IA puede acceder directamente a la base de conocimientos, sin necesidad de copiar y pegar manualmente. Esta función es particularmente práctica, muchas personas tienen bibliotecas de notas acumuladas durante años, después de conectarse la IA puede "leer" y entender el sistema de conocimiento.

**Escenario tres: reutilización de conocimiento entre proyectos**

Múltiples propuestas de IA pueden referenciar el mismo vault, logrando la reutilización de conocimiento entre proyectos. Por ejemplo, creaste un "vault de aprendizaje de patrones de diseño", que registra notas y ejemplos de código de varios patrones de diseño. Sin importar qué proyecto estés analizando, la IA puede referirse al contenido en este vault: el conocimiento no necesita acumularse repetidamente.

### Mecanismo de seguridad de rutas

El sistema estrictamente valida las rutas, previniendo ataques de path traversal:

```csharp
private static string ResolveFilePath(string vaultRoot, string relativePath)
{
    var rootPath = EnsureTrailingSeparator(Path.GetFullPath(vaultRoot));
    var combinedPath = Path.GetFullPath(Path.Combine(rootPath, relativePath));
    if (!combinedPath.StartsWith(rootPath, StringComparison.OrdinalIgnoreCase))
    {
        throw new BusinessException(VaultRelativePathTraversalCode,
            "Vault file paths must stay inside the registered vault root.");
    }
    return combinedPath;
}
```

Esto asegura que todas las operaciones de archivos estén dentro del rango del directorio raíz del vault, previniendo acceso a rutas maliciosas. La seguridad no puede ser descuidada, el asistente de IA va a operar el sistema de archivos, debe delimitar claramente los límites.

## Precauciones

Al usar el sistema Vault de HagiCode, hay varios puntos que necesitan atención especial:

1. **Seguridad de rutas**: Asegura que las rutas personalizadas estén dentro del rango permitido, de lo contrario el sistema rechazará la operación. Esto es para prevenir operaciones incorrectas y riesgos de seguridad potenciales.

2. **Gestión de submódulos Git**: CodeRef vault recomienda usar submódulos Git en lugar de copiar código directamente. Los beneficios ya se mencionaron: mantener sincronización, ahorrar espacio. Solo que los submódulos tienen su propia forma de uso, la primera vez puede necesitar familiarizarse.

3. **Límites de vista previa de archivos**: El sistema limita el tamaño de archivos (256KB) y cantidad (500), archivos超大 necesitan procesamiento por lotes. Este límite es por consideraciones de rendimiento, si encuentras archivos超大, puedes dividirlos manualmente o usar otras formas de procesar.

4. **Información de diagnóstico**: Crear un vault devuelve información de diagnóstico, cuando falla puede usarse para depuración. Cuando encuentres problemas mira primero la información de diagnóstico, en la mayoría de los casos puedes encontrar pistas.

## Resumen

El sistema Vault de HagiCode esencialmente está resolviendo un problema simple pero profundo: cómo hacer que el asistente de IA entienda y use recursos de conocimiento local.

A través de una capa de abstracción de almacenamiento unificada, estructura de directorios estandarizada, inyección de contexto automatizada, logra un método de gestión de conocimiento de "registrar una vez, reutilizar en todas partes". Después de crear un vault, ya sean notas de aprendizaje, repositorios de código o materiales de documentos, la IA puede acceder y entender automáticamente.

La mejora de experiencia traída por este diseño es obvia. Ya no necesita copiar manualmente fragmentos de código, explicar repetidamente información de fondo: el asistente de IA es como un colega que realmente entiende la situación del proyecto, capaz de proporcionar ayuda más valiosa basada en conocimiento existente.

El sistema Vault compartido en este artículo es exactamente la solución obtenida al pisar pozos y optimizar realmente durante el desarrollo de HagiCode. Si sientes que este diseño tiene valor, significa que la capacidad de ingeniería no está mal: entonces HagiCode mismo también vale la pena prestar atención.
