## Edición rápida de código en el navegador: Prácticas de integración con VSCode Web

> Después de que la IA analiza el código, ¿cómo abrir inmediatamente el editor en el navegador para modificarlo? Este artículo comparte la experiencia práctica de integrar code-server en el proyecto HagiCode, logrando una conexión perfecta entre el asistente de IA y la experiencia de edición de código.

## Antecedentes

En la era de la programación asistida por IA, los desarrolladores necesitan frecuentemente ver y editar código rápidamente. El flujo de desarrollo tradicional es: abrir el proyecto en un IDE de escritorio, localizar el archivo, editarlo y guardarlo. Sin embargo, en algunos escenarios, este proceso siempre parece que algo no está del todo bien.

Escenario 1: Desarrollo remoto. Al usar un asistente de IA como HagiCode, el backend puede ejecutarse en un servidor remoto o contenedor, sin acceso local directo a los archivos del proyecto. Cada vez que necesitas ver o modificar código, debes conectarte a través de SSH u otros medios, lo que fragmenta la experiencia. Se siente como si quisieras ver a alguien, pero hay un cristal grueso de por medio: puedes ver, pero no tocar.

Escenario 2: Vista previa rápida. Después de que el asistente de IA analiza el código, el usuario puede solo querer navegar rápidamente algún archivo o hacer modificaciones menores. Iniciar el IDE de escritorio completo parece demasiado pesado, un editor ligero dentro del navegador se ajusta mejor a la necesidad de "visualización rápida". Después de todo, ¿quién querría armarse un gran lío solo para echar un vistazo?

Escenario 3: Colaboración entre dispositivos. Al trabajar en diferentes dispositivos, el editor en el navegador proporciona un punto de acceso unificado, sin necesidad de configurar el entorno de desarrollo en cada dispositivo. Esto ahorra trabajo,毕竟 la vida es corta, ¿para qué repetir el trabajo?

Para resolver estos puntos problemáticos, integramos VSCode Web en el proyecto HagiCode. Permitimos una conexión perfecta entre el asistente de IA y la experiencia de edición de código: después de que la IA analiza el código, los usuarios pueden abrir inmediatamente el editor en la misma sesión del navegador para modificarlo, sin cambiar de aplicación. Esta experiencia, digamos, es como cuando lo piensas, ya está ahí.

## Sobre HagiCode

La solución compartida en este artículo proviene de nuestra experiencia práctica en el proyecto [HagiCode](https://hagicode.com). HagiCode es un asistente de código impulsado por IA, diseñado para mejorar la eficiencia del desarrollo a través de la interacción en lenguaje natural. Durante el desarrollo, descubrimos que los usuarios necesitan cambiar rápidamente entre el análisis de IA y la edición de código, lo que nos impulsó a explorar cómo integrar el editor directamente en el navegador.

Dirección del proyecto: [github.com/HagiCode-org/site](https://github.com/HagiCode-org/site)

## Selección tecnológica: ¿Por qué code-server?

Entre las muchas soluciones de VSCode Web, elegimos code-server. Esta elección, en realidad, tiene varias consideraciones:

Funcionalidad completa. code-server es la versión web de VSCode, que admite la mayoría de las funciones de la versión de escritorio, incluyendo el sistema de extensiones, sugerencias inteligentes, depuración, etc. Esto significa que los usuarios pueden obtener una experiencia de edición cercana a la de escritorio también en el navegador. Después de todo, ¿quién quiere comprometerse en funcionalidad?

Despliegue flexible. code-server puede ejecutarse como un servicio independiente, también admite despliegue en contenedores Docker, encajando con la arquitectura de HagiCode. Nuestro backend está escrito en C#, el frontend es React, y se comunica con el servicio code-server a través de REST API. Es como construir con bloques, cada uno tiene su lugar.

Seguridad de autenticación. code-server tiene un mecanismo integrado de connection-token para evitar el acceso no autorizado. Cada sesión tiene un token único, asegurando que solo los usuarios autorizados puedan acceder al editor. La sensación de seguridad, una vez que la tienes, sabes que es importante.

## Diseño de arquitectura

La integración de VSCode Web de HagiCode utiliza un diseño de arquitectura de separación frontend-backend.

### Capa de servicios frontend

El frontend encapsula la interacción con el backend a través de `vscodeServerService.ts`:

```typescript
// Abrir proyecto
export async function openProjectInCodeServer(
  id: string,
  currentInterfaceLanguage?: string,
): Promise<VsCodeServerLaunchResponseDto>

// Abrir vault
export async function openVaultInCodeServer(
  id: string,
  path?: string,
  currentInterfaceLanguage?: string,
): Promise<VsCodeServerLaunchResponseDto>
```

La diferencia entre estos dos métodos es: `openProjectInCodeServer` se usa para abrir todo el proyecto, mientras que `openVaultInCodeServer` se usa para abrir una ruta específica de Vault. Para proyectos MonoSpecs de múltiples repositorios, el sistema crea automáticamente archivos de espacio de trabajo. División clara del trabajo, cada uno hace lo suyo, eso es suficiente.

### Capa de servicios backend

El backend `VaultAppService.cs` implementa la lógica central:

```csharp
public async Task<VsCodeServerLaunchResponseDto> OpenInCodeServerAsync(
    string id,
    string? relativePath = null,
    string? currentInterfaceLanguage = null,
    CancellationToken cancellationToken = default)
{
    // 1. Obtener configuración y verificar si está habilitado
    var settings = await _vsCodeServerSettingsService.GetResolvedSettingsAsync(cancellationToken);
    if (!settings.Enabled) {
        throw new BusinessException(VsCodeServerErrorCodes.Disabled, "VSCode Server is disabled.");
    }

    // 2. Obtener vault y resolver directorio de inicio
    var vault = await RequireVaultAsync(id, cancellationToken);
    var launchDirectory = ResolveLaunchDirectory(vault, relativePath);

    // 3. Asegurar que code-server se esté ejecutando y obtener información de tiempo de ejecución
    var runtime = await _vsCodeServerManager.EnsureStartedAsync(settings, cancellationToken);

    // 4. Resolver configuración de idioma
    var language = _vsCodeServerSettingsService.ResolveLaunchLanguage(
        settings.Language,
        currentInterfaceLanguage);

    // 5. Construir URL de inicio
    return new VsCodeServerLaunchResponseDto {
        LaunchUrl = AppendQueryString(runtime.BaseUrl, new Dictionary<string, string?> {
            ["folder"] = launchDirectory,
            ["tkn"] = runtime.ConnectionToken,
            ["vscode-lang"] = language,
        }),
        ConnectionToken = runtime.ConnectionToken,
        OpenMode = "folder",
        Runtime = VsCodeServerSettingsService.MapRuntime(
            await _vsCodeServerManager.GetRuntimeSnapshotAsync(cancellationToken)),
    };
}
```

Las responsabilidades de este método son muy claras: verificar configuración, resolver rutas, iniciar servicio, construir URL. El método `ResolveLaunchDirectory` realiza verificaciones de seguridad de rutas para prevenir ataques de path traversal. El código es como poesía, cada línea tiene su propio ritmo.

### Gestión automática de tiempo de ejecución

El backend gestiona el proceso de code-server a través de `VsCodeServerManager`:

- Verificar estado del proceso
- Iniciar automáticamente servicios detenidos
- Devolver instantánea de tiempo de ejecución (puerto, ID de proceso, tiempo de inicio, etc.)

Este diseño permite que el sistema maneje automáticamente el ciclo de vida de code-server, sin que los usuarios necesiten gestionar manualmente los procesos del servicio. Después de todo, la vida ya es lo suficientemente compleja, automatiza lo que se pueda.

### Mecanismo de seguimiento de idioma

HagiCode admite interfaces multiidioma, y code-server también debe seguir esta configuración. El sistema admite tres modos de idioma:

- `follow`: seguir el idioma actual de la interfaz
- `zh-CN`: fijar chino
- `en-US`: fijar inglés

Se pasa a code-server a través del parámetro URL `vscode-lang`, asegurando que el idioma del editor se mantenga consistente con la interfaz de HagiCode. El idioma, cuando está unificado, se siente mejor.

El frontend de HagiCode usa React + TypeScript, los pasos para integrar code-server no son complicados.


Este botón activa la acción de abrir, llamando a la API del backend para obtener la URL de inicio. Un botón, lleva una acción, simple y directo.

### Manejar la acción de abrir

```tsx
const handleAction = async (action: ProjectAction) => {
  if (action.type === 'open-code-server') {
    const response = await openProjectInCodeServer(project.id, i18n.language);
    window.open(response.launchUrl, '_blank', 'noopener,noreferrer');
  }
};
```

Usar `window.open` para abrir code-server en una nueva pestaña, los parámetros `noopener,noreferrer` aseguran la seguridad. En seguridad, nunca se es demasiado cuidadoso.

### Punto de entrada de edición de Vault

Agregar un botón de edición similar en la lista de Vaults:

```tsx

```

Proyecto y Vault usan la misma forma de apertura, manteniendo la consistencia de la interacción. La importancia de la consistencia no es menor que la funcionalidad misma.

## Lógica de construcción de URL

El formato de URL de code-server, en realidad, tiene su detalle:

**Modo carpeta**:
```
http://{host}:{port}/?folder={path}&tkn={token}&vscode-lang={lang}
```

**Modo espacio de trabajo**:
```
http://{host}:{port}/?workspace={workspacePath}&tkn={token}&vscode-lang={lang}
```

Donde `tkn` es el token de conexión, generado automáticamente cada vez que se inicia code-server, asegurando el acceso seguro. El parámetro `vscode-lang` controla el idioma de la interfaz del editor. Cada parámetro tiene su uso, todos son necesarios.

## Consideraciones de seguridad

Al implementar la integración de code-server, la seguridad es un problema que necesita atención especial. Después de todo, en seguridad, cuando surge un problema, ya es tarde.

### Connection Token

El connection-token se genera aleatoriamente y no debe filtrarse. Se recomienda usarlo en entorno HTTPS para evitar que el token sea interceptado por un atacante intermediario. La información sensible, mejor protegerla.

### Seguridad de rutas de archivo

El backend implementa verificación de path traversal:

```csharp
private static string ResolveLaunchDirectory(VaultRegistryEntry vault, string? relativePath)
{
    var vaultRoot = EnsureTrailingSeparator(Path.GetFullPath(vault.PhysicalPath));
    var combinedPath = Path.GetFullPath(Path.Combine(vaultRoot, relativePath ?? "."));
    if (!combinedPath.StartsWith(vaultRoot, StringComparison.OrdinalIgnoreCase))
    {
        throw new BusinessException(VaultRelativePathTraversalCode, "Relative path traversal detected.");
    }
    return combinedPath;
}
```

Este código asegura que los usuarios no puedan acceder a archivos fuera del directorio vault a través de `../` y otros métodos. En verificación de límites, mejor hacer que no hacer.

### Control de permisos

El proceso de code-server se ejecuta con permisos de usuario apropiados, evitando acceder a archivos sensibles del sistema. Se recomienda usar un usuario dedicado para ejecutar el servicio code-server. El control de permisos, lo necesario hay que tenerlo.

## Optimización de rendimiento

code-server consumirá recursos del servidor, aquí hay algunas sugerencias de optimización:

- Monitorear el uso de CPU/memoria, ajustar límites de recursos cuando sea necesario
- Los proyectos grandes pueden necesitar aumentar el tiempo de espera
- Implementar limpieza automática de tiempo de espera de sesión para liberar recursos
- Considerar usar caché para reducir cálculos repetidos

HagiCode proporciona una API de monitoreo de estado de tiempo de ejecución, el frontend puede obtener el estado actual a través de `getVsCodeServerSettings()`:

```typescript
const { settings, runtime } = await getVsCodeServerSettings();
// runtime.status: 'disabled' | 'stopped' | 'starting' | 'running' | 'unhealthy'
// runtime.baseUrl: "http://localhost:8080"
// runtime.processId: 12345
```

Este diseño permite a los usuarios entender claramente el estado de salud de code-server y localizar rápidamente problemas cuando surgen. Estado visible, para tener tranquilidad.

## Detalles de experiencia de usuario

Durante la implementación, descubrimos algunos detalles que afectan la experiencia del usuario que merecen atención especial:

Abrir code-server por primera vez puede requerir esperar el inicio, este tiempo puede variar desde unos segundos hasta medio minuto. Se recomienda mostrar el estado de carga en el frontend para que el usuario sepa que el sistema está procesando. En la espera, tener retroalimentación es suficiente.

Los navegadores pueden bloquear ventanas emergentes, es necesario pedir a los usuarios que permitan manualmente. HagiCode muestra información de guía al abrir por primera vez, diciendo a los usuarios cómo configurar los permisos del navegador. La experiencia del usuario se refleja en estos detalles.

Se recomienda mostrar el estado de tiempo de ejecución (iniciando/ejecutándose/error), para que cuando los usuarios encuentren problemas puedan saber si es un problema del servidor o su propia operación. Saber dónde está el problema, al menos tener seguridad.
