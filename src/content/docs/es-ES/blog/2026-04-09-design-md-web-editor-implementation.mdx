## Editar DESIGN.md directamente en la interfaz web: de la idea a la implementación

> En el sistema de gestión de proyectos MonoSpecs, DESIGN.md contiene el diseño arquitectónico y las decisiones técnicas del proyecto. Pero el método de edición tradicional requiere que los usuarios cambien a un editor externo, este flujo fragmentado, ¿cómo decirlo?, es como cuando se interrumpe la lectura de un poema de repente: se pierde la inspiración y también el estado de ánimo. Este artículo comparte la solución que practicamos en el proyecto HagiCode: editar DESIGN.md directamente en la interfaz web y admitir la importación de plantillas desde sitios de diseño en línea. Después de todo, ¿a quién no le gusta esa sensación de fluidez?

## Antecedentes

`DESIGN.md`, como portador principal de los documentos de diseño del proyecto, contiene información clave como el diseño arquitectónico, las decisiones técnicas y la guía de implementación. Sin embargo, el método de edición tradicional requiere que los usuarios cambien a un editor externo (como VS Code) y localicen manualmente la ruta física antes de editar. Este proceso no es complejo, pero después de repetirlo varias veces, uno se cansa.

Los problemas específicos se reflejan en los siguientes aspectos:

- **Flujo fragmentado**: Los usuarios necesitan cambiar con frecuencia entre la interfaz de gestión web y el editor local, lo que rompe la continuidad del flujo de trabajo; es como cuando la red se desconecta repentinamente mientras escuchas música, el ritmo se desordena por completo.
- **Dificultad de reutilización**: El sitio de diseño ya ha publicado una rica biblioteca de plantillas de diseño, pero no se puede integrar directamente en el flujo de edición del proyecto. Claramente hay cosas buenas, pero no se pueden usar, lo cual es realmente una lástima.
- **Falta de experiencia**: Falta el ciclo cerrado de "vista previa-selección-importación", los usuarios deben copiar y pegar manualmente, lo que aumenta el riesgo de errores. Cuantas más operaciones manuales, más oportunidades de error.
- **Barreras de colaboración**: El mantenimiento sincrónico de los documentos de diseño y la implementación del código se vuelve de alta fricción, lo que obstaculiza la eficiencia de la colaboración en equipo. La colaboración en equipo ya no es fácil, ¿por qué añadir más resistencias?

Para resolver estos problemas, decidimos implementar la capacidad de editar directamente DESIGN.md en la interfaz web y admitir la importación con un clic de plantillas desde sitios de diseño en línea. No es una decisión trascendental, solo queremos que la experiencia de desarrollo sea más fluida.

## Sobre HagiCode

La solución compartida en este artículo proviene de nuestra experiencia práctica en el proyecto [HagiCode](https://hagicode.com). HagiCode es un proyecto de asistente de código impulsado por IA. Durante el desarrollo, necesitamos mantener frecuentemente los documentos de diseño del proyecto. Para permitir que el equipo colabore de manera más eficiente, exploramos e implementamos esta solución de edición e importación en línea. En realidad no hay nada especial, solo encontramos un problema y buscamos una solución.

## Solución técnica

### Arquitectura general

Esta solución adopta una arquitectura de proxy del mismo origen con separación entre frontend y backend, compuesta principalmente por las siguientes capas. El diseño de esta arquitectura, para decirlo simplemente, se trata de "cada uno cumple su función":

**1. Capa de editor frontend**
```typescript
// Componente principal: DesignMdManagementDrawer
// Ubicación: repos/web/src/components/project/DesignMdManagementDrawer.tsx
// Función: alojar edición, guardado, detección de conflictos de versión, flujo de importación
```

**2. Capa de servicios backend**
```csharp
// Servicio principal: ProjectAppService.DesignMd
// Ubicación: repos/hagicode-core/src/PCode.Application/ProjectAppService.DesignMd.cs
// Función: resolución de rutas, lectura/escritura de archivos, gestión de versiones
```

**3. Capa de proxy del mismo origen**
```csharp
// Servicio de proxy: ProjectAppService.DesignMdSiteIndex
// Ubicación: repos/hagicode-core/src/PCode.Application/ProjectAppService.DesignMdSiteIndex.cs
// Función: proxy de recursos del sitio de diseño, caché de vistas previas, validación de seguridad
```

### Decisiones técnicas clave

#### Decisión 1: Modo de cajón global

Se adopta un único cajón global en lugar de capas emergentes locales, gestionando el estado a través de `layoutSlice`, logrando una experiencia consistente entre vistas (classic/kanban). Este enfoque asegura que los usuarios obtengan una experiencia de interacción unificada independientemente de la vista desde la que abran el editor. Después de todo, una experiencia consistente hace que los usuarios se sientan más cómodos y no se pierdan al cambiar de vista.

#### Decisión 2: API con ámbito de proyecto

Se coloca las interfaces relacionadas con DESIGN.md bajo `ProjectController`, reutilizando los límites de permisos del proyecto existentes y evitando la complejidad de agregar un controlador independiente. Las ventajas de este diseño son una gestión de permisos más clara y el cumplimiento de los principios de organización de recursos RESTful. A veces, reutilizar tiene más sentido que crear de nuevo, ¿no?

#### Decisión 3: Detección de conflictos de versión

Se deriva una versión opaca basada en `LastWriteTimeUtc` del sistema de archivos, implementando un control de concurrencia optimista ligero. Cuando varios usuarios editan el mismo archivo simultáneamente, el sistema puede detectar conflictos y pedir a los usuarios que actualicen. Este diseño no bloquea las operaciones de edición del usuario y garantiza la coherencia de los datos; es como el sentido de límites en las relaciones interpersonales, ni demasiado distante ni invasivo.

#### Decisión 4: Modo de proxy del mismo origen

A través de `IHttpClientFactory`, se hace proxy de los recursos del sitio de diseño externo, evitando problemas de CORS y riesgos de SSRF. Este diseño garantiza la seguridad y simplifica las llamadas del frontend. En cuanto a seguridad, nunca es suficiente, después de todo, la seguridad de los datos es como la salud, uno se arrepiente cuando la pierde.

## Implementación principal

### 1. Editar DESIGN.md directamente

#### Implementación del backend

El backend es responsable principalmente de la resolución de rutas, lectura/escritura de archivos y gestión de versiones. Este trabajo, aunque básico, es indispensable, como los cimientos de una casa:

```csharp
// Resolución de ruta y validación de seguridad
private Task<string> ResolveDesignDocumentDirectoryAsync(string projectPath, string? repositoryPath)
{
    if (string.IsNullOrWhiteSpace(repositoryPath))
    {
        return Task.FromResult(Path.GetFullPath(projectPath));
    }
    return ValidateSubPathAsync(projectPath, repositoryPath);
}

// Generación de número de versión (basado en marca de tiempo y tamaño del sistema de archivos)
private static string BuildDesignDocumentVersion(string path)
{
    var fileInfo = new FileInfo(path);
    fileInfo.Refresh();
    return string.Create(
        CultureInfo.InvariantCulture,
        $"{fileInfo.LastWriteTimeUtc.Ticks:x}-{fileInfo.Length:x}");
}
```

El diseño del número de versión también es bastante interesante, usamos la última hora de modificación y el tamaño del archivo para generar un identificador de versión único. Esto es ligero y confiable, no requiere mantener una base de datos de versiones adicional. Las cosas simples suelen ser más efectivas, ¿no?

#### Implementación del frontend

El frontend implementa la detección de estado sucio y la lógica de guardado. Este diseño permite que los usuarios sepan en todo momento si sus modificaciones se han guardado, reduciendo la ansiedad de "¿y si se pierde?":

```typescript
// Detección de estado sucio y lógica de guardado
const [draft, setDraft] = useState('');
const [savedDraft, setSavedDraft] = useState('');
const isDirty = draft !== savedDraft;

const handleSave = useCallback(async () => {
    const result = await saveProjectDesignMdDocument({
        ...activeTarget,
        content: draft,
        expectedVersion: document.version, // control de concurrencia optimista
    });
    setSavedDraft(draft); // actualizar estado guardado
}, [activeTarget, document, draft]);
```

En esta implementación, mantenemos dos estados: `draft` es el contenido que se está editando actualmente, `savedDraft` es el contenido guardado. Comparamos ambos para determinar si hay modificaciones sin guardar. Este diseño, aunque simple, brinda tranquilidad, después de todo, ¿quién quiere que lo que escribió con tanto esfuerzo desaparezca repentinamente?

### 2. Importar archivos de diseño desde línea

#### Estructura de directorios

```
repos/index/
└── src/data/public/design.json    # índice de plantillas de diseño

repos/awesome-design-md-site/

    └── awesomeDesignCatalog.ts     # tubería de contenido
```

#### Formato de datos del índice

El archivo de índice del sitio de diseño define todas las plantillas disponibles. Con este índice, los usuarios pueden seleccionar fácilmente la plantilla que desean, como pedir en un restaurante:

```json
{
  "entries": [
    {
      "slug": "linear.app",
      "title": "Linear Inspired Design System",
      "summary": "Producto AI / estilo oscuro",
      "detailUrl": "/designs/linear.app/",
      "designDownloadUrl": "/designs/linear.app/DESIGN.md",
      "previewLightImageUrl": "...",
      "previewDarkImageUrl": "..."
    }
  ]
}
```

Cada entrada contiene la información básica y el enlace de descarga de la plantilla. El backend lee la lista de plantillas disponibles desde este índice y luego las presenta a los usuarios para su selección. Este diseño hace que la selección sea intuitiva en lugar de estar a ciegas.

#### Implementación del proxy del mismo origen

Para garantizar la seguridad, el backend realiza una validación estricta del acceso al sitio de diseño. En cuanto a seguridad, nunca es demasiado cuidadoso:

```csharp
// Validación segura de slug
private static readonly Regex SafeDesignSiteSlugRegex =
    new("^[A-Za-z0-9](?:[A-Za-z0-9._-]{0,127})$", RegexOptions.Compiled);

private static string NormalizeDesignSiteSlug(string slug)
{
    var normalizedSlug = slug?.Trim() ?? string.Empty;
    if (!IsSafeDesignSiteSlug(normalizedSlug))
    {
        throw new BusinessException(
            ProjectDesignSiteIndexErrorCodes.InvalidSlug,
            "Design site slug must be a single safe path segment.");
    }
    return normalizedSlug;
}

// Caché de vista previa (directorio temporal del SO)
private static string ComputePreviewCacheKey(string slug, string theme, string previewUrl)
{
    var raw = $"{slug}|{theme}|{previewUrl}";
    var bytes = SHA256.HashData(Encoding.UTF8.GetBytes(raw));
    return Convert.ToHexString(bytes).ToLowerInvariant();
}
```

Aquí hicimos dos cosas: primero, usar expresiones regulares para validar estrictamente el formato del slug y prevenir ataques de path traversal; segundo, hacer caché de las vistas previas para reducir la presión de solicitudes en el sitio externo. Lo primero es protección, lo segundo es optimización, ambos son indispensables.

### 3. Flujo completo de importación

```typescript
// 1. Abrir cajón de importación
const handleRequestImportDrawer = useCallback(() => {
    setIsImportDrawerOpen(true);
}, []);

// 2. Seleccionar e importar
const handleImportRequest = useCallback((entry) => {
    if (isDirty) {
        setPendingImportEntry(entry);
        setConfirmMode('import'); // confirmación de sobrescritura
        return;
    }
    void executeImport(entry);
}, [isDirty]);

// 3. Ejecutar importación
const executeImport = useCallback(async (entry) => {
    const result = await getProjectDesignMdSiteImportDocument(
        activeTarget.projectId,
        entry.slug
    );
    setDraft(result.content); // solo reemplazar texto del editor, no guardar automáticamente
    setIsImportDrawerOpen(false);
}, [activeTarget?.projectId]);
```

El diseño del flujo de importación sigue el principio de "confirmación del usuario": después de importar, solo se actualiza el contenido del editor, no se guarda automáticamente. Los usuarios pueden revisar el contenido importado y guardarlo manualmente después de confirmar que es correcto. Después de todo, los usuarios deberían tener la decisión final sobre lo que escriben, ¿no?

### Escenario 3: Manejo de conflictos de versión

Cuando varios usuarios editan el mismo DESIGN.md simultáneamente, el sistema detectará conflictos de versión. Este mecanismo de control de concurrencia optimista asegura la coherencia de los datos sin bloquear las operaciones de edición del usuario:

```csharp
if (!string.Equals(currentVersion, expectedVersion, StringComparison.Ordinal))
{
    throw new BusinessException(
        ProjectDesignDocumentErrorCodes.VersionConflict,

```

El frontend capturará este error y avisará al usuario:

Este mecanismo de control de concurrencia optimista asegura la coherencia de los datos sin bloquear las operaciones de edición del usuario. Los conflictos son inevitables, pero al menos podemos informar al usuario qué sucedió en lugar de perder silenciosamente las modificaciones.

## Consideraciones y mejores prácticas

### 1. Seguridad de rutas

Siempre valide `repositoryPath` para prevenir ataques de path traversal. En cuanto a seguridad, nunca es suficiente:

```csharp
// Siempre validar repositoryPath para prevenir ataques de path traversal
return ValidateSubPathAsync(projectPath, repositoryPath);
// rechazar entradas peligrosas como "../", rutas absolutas, etc.
```

### 2. Estrategia de caché

Caché de vistas previas durante 24 horas, máximo 160 archivos. Un caché moderado mejora el rendimiento, pero tampoco se debe exagerar, después de todo, el equilibrio es la clave:

```csharp
// Caché de vistas previas durante 24 horas, máximo 160 archivos
private static readonly TimeSpan PreviewCacheTtl = TimeSpan.FromHours(24);
private const int PreviewCacheMaxFiles = 160;
// limpiar periódicamente el caché expirado
```

### 3. Manejo de errores

Degradación elegante cuando el sitio upstream no está disponible. Este diseño de degradación elegante asegura que incluso si las dependencias externas no están disponibles, la funcionalidad de edición principal siga funcionando normalmente. Después de todo, el sistema no debería colapsar solo porque un servicio externo falla:

```typescript
// Degradación elegante cuando el sitio upstream no está disponible
try {
    const catalog = await getProjectDesignMdSiteImportCatalog(projectId);
} catch (error) {
    toast.error(t('project.designMd.siteImport.feedback.catalogLoadFailed'));
    // El cajón de edición principal todavía está disponible
}
```

Este diseño de degradación elegante asegura que incluso si las dependencias externas no están disponibles, la funcionalidad de edición principal siga funcionando normalmente. El sistema debe ser resistente en lugar de caerse ante cualquier problema.

### 4. Optimización de la experiencia del usuario

Confirmar sobrescritura antes de importar, no guardar automáticamente después de importar. Los usuarios deberían tener control sobre sus operaciones, no que el sistema decida por sí solo:

```typescript
// Confirmar sobrescritura antes de importar
if (isDirty) {
    setConfirmMode('import');
    return;
}

// No guardar automáticamente después de importar, confirmar por el usuario
setDraft(result.content); // solo actualizar borrador
// El usuario revisa y hace clic en Save para escribir realmente en disco
```

### 5. Consideraciones de rendimiento

Usar fábrica de clientes HTTP para evitar crear demasiadas conexiones. La gestión de recursos, aunque parece insignificante, si se hace bien puede traer efectos inesperados:

```csharp
// Usar fábrica de clientes HTTP para evitar crear demasiadas conexiones
private const string DesignSiteProxyClientName = "ProjectDesignSiteProxy";
```

## Sugerencias de extensión

1. **Mejoras de Markdown**: Actualmente se usa un `Textarea` básico, se podría considerar actualizar a CodeMirror para admitir resaltado de sintaxis y atajos de teclado. Si la experiencia del editor es buena, el ánimo de escribir documentos también mejora.
2. **Modo de vista previa**: Agregar vista previa en tiempo real de Markdown para mejorar la experiencia de edición. Ver lo que se obtiene siempre da más confianza.
3. **Fusión de diferencias**: Implementar un algoritmo de fusión inteligente en lugar de un simple reemplazo de texto completo. Los conflictos son inevitables, pero al menos el proceso de manejarlos puede ser menos doloroso.
4. **Caché local**: Almacenar en caché design.json en la base de datos para reducir la dependencia del sitio externo. Menos dependencias, más estable el sistema, es una simple verdad.

## Conclusión

En el proyecto HagiCode, implementamos una solución completa de edición e importación en línea de DESIGN.md a través de la colaboración entre frontend y backend. El valor central de esta solución radica en:

- **Mejorar la eficiencia**: No es necesario cambiar de herramientas, completar la edición e importación de documentos de diseño en una interfaz web unificada. El tiempo ahorrado se puede usar para cosas más significativas.
- **Reducir la barrera**: Importación de plantillas de diseño con un clic, los nuevos proyectos pueden comenzar rápidamente. Cuanto más fácil es empezar, más probable es continuar.
- **Seguro y confiable**: Mecanismos como validación de rutas, detección de conflictos de versión y degradación elegante aseguran el funcionamiento estable del sistema. La estabilidad es la base, sin estabilidad todo es palabrería.
- **Experiencia de usuario**: Detalles como el cajón global, detección de estado sucio y diálogos de confirmación pulieron la experiencia de interacción. Los detalles determinan el éxito o el fracaso, esta frase es especialmente aplicable a la experiencia del usuario.

Esta solución ya está funcionando en el proyecto HagiCode y ha resuelto los puntos dolorosos del equipo en la gestión de documentos de diseño.
