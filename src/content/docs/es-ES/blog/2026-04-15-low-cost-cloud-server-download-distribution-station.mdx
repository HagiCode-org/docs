## Cómo utilizar servidores en la nube de bajo costo para construir una estación de distribución de descargas aceleradas

> El tráfico de almacenamiento en la nube es absurdamente costoso, el acceso transfronterizo es frustrantemente lento, los precios de CDN son desalentadores...... estos dolores, probablemente todos los que hacen distribución los entiendan. Este artículo comparte una solución de bajo costo que exploramos en el proyecto [HagiCode](https://hagicode.com). Servidor en la nube + capa de caché Nginx, los costos se redujeron a la mitad y la velocidad mejoró, también es un pequeño consuelo.

## Antecedentes

En internet, la velocidad de descarga y la estabilidad, al final todo es experiencia del usuario. Sea de código abierto o comercial, siempre hay que dar a los usuarios una forma de descarga confiable.

Descargar archivos directamente desde almacenamiento en la nube (como Azure Blob Storage, AWS S3) parece simple, pero en realidad hay bastantes problemas:

**Latencia de red**: el acceso transfronterizo y transregional es tan lento que te hace querer romper el teclado. Los usuarios esperan hasta que las flores se marchiten, ¿puede ser buena la experiencia?

**Costo de ancho de banda**: el tráfico de salida del almacenamiento en la nube es dolorosamente caro. Azure Blob Storage en China continental cuesta alrededor de 0.5 yuanes por GB, 1TB al mes son 500 yuanes. Para equipos pequeños, este dinero no es mucho ni poco, después de todo el dinero de nadie viene del viento.

**Restricciones de acceso**: en ciertas regiones el acceso a servicios en la nube extranjeros es irregular, a veces simplemente no se puede acceder. Los usuarios no pueden descargar ni siquiera si quieren, esta situación en sí es bastante impotente.

**Costo de CDN**: las CDN comerciales ciertamente pueden resolver el problema, pero el precio también es realmente hermoso. ¿Qué equipo pequeño puede permitírselo?

¿Entonces hay una forma que sea tanto económica como fácil de usar? De hecho la hay. Servidor en la nube + proxy inverso + capa de caché, simple y brutalmente. Los costos se redujeron aproximadamente a la mitad, y la velocidad mejoró, también es un pequeño consuelo.

## Diseño de arquitectura

### Ideas de arquitectura general

Primero veamos el diseño de arquitectura general:

```
Solicitud de usuario
    ↓
Resolución DNS
    ↓
┌─────────────────────────────────────┐
│   Capa de proxy inverso (Traefik/Bunker Web)   │ ← Terminación SSL, distribución de rutas, protección de seguridad
├─────────────────────────────────────┤
│   Puerto: 80/443                       │
│   Función: certificados automáticos Let's Encrypt      │
│         enrutamiento Host                    │
└─────────────────────────────────────┘
    ↓
┌─────────────────────────────────────┐
│   Capa de caché (Nginx)                     │ ← caché de archivos, compresión Gzip
├─────────────────────────────────────┤
│   Puerto: 8080(server) / 8081(desktop) │
│   Estrategia de caché:                          │
│   - index.json: 1 hora               │
│   - otros archivos: 7 días                   │
│   Tamaño de caché: 1GB                      │
└─────────────────────────────────────┘
    ↓
┌─────────────────────────────────────┐
│   Sitio de origen (Azure Blob Storage)         │ ← almacenamiento de archivos
└─────────────────────────────────────┘
```

La idea central de esta arquitectura, dicho simplemente, es: **agregar una capa de caché entre el usuario y el almacenamiento en la nube**.

La solicitud del usuario llega primero a la capa de proxy inverso en el servidor en la nube, luego la capa de caché Nginx toma el control. ¿Hay archivo en caché? Directamente al usuario. ¿No? Traer una copia del almacenamiento en la nube, convenientemente guardar una copia localmente. La próxima vez que accedas al mismo archivo, no tienes que molestar al almacenamiento en la nube. En realidad es como la memoria, algunas cosas se recuerdan, la próxima vez no tienes que esforzarte de nuevo......

### ¿Por qué elegir esta arquitectura?

**Ventajas del servidor en la nube**:
- Costo controlable: Alibaba Cloud y otros proporcionan servidores en la nube económicos (configuración de 1-2 núcleos 2GB costo mensual alrededor de 50-100 yuanes)
- Despliegue flexible: puedes configurar libremente proxy inverso y estrategias de caché
- Ubicación geográfica flexible: puedes elegir áreas de servidor cercanas a los usuarios
- Fuerte escalabilidad: puedes actualizar la configuración según las necesidades de tráfico

**Arquitectura de proxy inverso + caché**:
- Reducir la presión del sitio de origen: cachear archivos calientes, reducir el acceso al almacenamiento en la nube
- Reducir costos: las tarifas de tráfico del servidor en la nube son mucho más bajas que el tráfico de salida del almacenamiento en la nube
- Mejorar velocidad: acceso cercano, el ancho de banda del servidor suele ser mejor que el almacenamiento en la nube

**¿Por qué elegir Nginx como capa de caché?**

Esto no se eligió al azar, después de todo Nginx ciertamente tiene su razón:

1. Alto rendimiento: el rendimiento de proxy inverso de Nginx, la industria lo tiene a la vista
2. Caché maduro: la función proxy_cache integrada, estable y confiable, no da problemas fácilmente
3. Bajo uso de recursos: 256MB de memoria pueden correr, también es amigable para el servidor
4. Configuración flexible: diferentes tipos de archivos pueden establecer diferentes estrategias de caché, esto es bastante considerado

## Capa de proxy inverso: Traefik vs Bunker Web

El plan de despliegue de HagiCode, en realidad admite dos proxies inversos, digamos, cada uno tiene sus características:

| Plan | Características | Escenario aplicable |
|------|------|----------|
| **Traefik** | Ligero, SSL automático, configuración simple | Despliegue básico, escenarios de bajo tráfico |
| **Bunker Web** | WAF incorporado, anti-DDoS, anti-crawler | Altos requisitos de seguridad, escenarios de alto tráfico |

### Traefik: primera opción ligera

Traefik es un proxy inverso HTTP moderno y balanceador de carga, la característica más grande es configuración simple, también puede hacer automáticamente certificados Let's Encrypt.

Para despliegue inicial o escenarios con poco tráfico, Traefik es bastante buena elección:
- No ocupa muchos recursos (1.5 CPU/512MB de memoria es suficiente)
- Certificados SSL configurados automáticamente, no tienes que preocuparte
- Configuración de enrutamiento basada en etiquetas Docker, también es conveniente

### Bunker Web: escenarios de alta seguridad

Bunker Web es un firewall de aplicaciones web basado en Nginx, la protección de seguridad es más completa.

¿Cuándo considerar cambiar a Bunker Web? Probablemente en estas situaciones:
- Sufrir ataques DDoS (aunque nadie espera encontrarlos)
- Necesitar protección ModSecurity
- Querer funciones anti-crawler
- Tener requisitos más altos de seguridad

HagiCode proporciona el script `switch-deployment.sh`, puede cambiar rápidamente entre los dos planes:

```bash
# Cambiar a Bunker Web
./switch-deployment.sh bunkerweb

# Cambiar de vuelta a Traefik
./switch-deployment.sh traefik

# Ver estado actual
./switch-deployment.sh status
```

El script automáticamente hace verificaciones previas, verificaciones de salud, también puede revertir automáticamente, el proceso de cambio también es seguro y confiable, no se va a caer mientras cambias.

## Configuración de capa de caché Nginx

La capa de caché es el núcleo de toda la arquitectura, la configuración de Nginx es buena o mala, el efecto de caché es muy diferente. Después de todo esto es bastante crítico.

### Configuración de ruta de caché

```nginx
# Configuración de ruta de caché
proxy_cache_path /var/cache/nginx levels=1:2 keys_zone=azure_cache:10m
                   max_size=1g inactive=7d use_temp_path=off;
```

Descripción de parámetros:
- `levels=1:2`: nivel de directorio de caché, estructura de directorio de 2 niveles, mejora la eficiencia de acceso a archivos
- `keys_zone=azure_cache:10m`: zona de almacenamiento de claves de caché, 10MB es suficiente para almacenar muchas claves
- `max_size=1g`: tamaño máximo de caché 1GB
- `inactive=7d`: archivos de caché no accedidos durante 7 días se eliminan
- `use_temp_path=off`: escribir directamente en el directorio de caché, mejorar rendimiento

### Estrategia de caché jerárquico

Diferentes tipos de archivos necesitan diferentes estrategias de caché:

```nginx
# Servicio de descarga Server
server {
    listen 8080;

    # caché a corto plazo para index.json (para actualización oportuna)
    location /index.json {
        proxy_cache azure_cache;
        proxy_cache_valid 200 1h;
        proxy_cache_key "$scheme$server_port$request_uri";
        add_header X-Cache-Status $upstream_cache_status;
        add_header Cache-Control "public, max-age=3600";

        # Proxy inverso a Azure OSS
        proxy_pass https://${SERVER_DL_HOST}/${SERVER_DL_CONTAINER}$uri?${SERVER_DL_SAS_TOKEN};
        proxy_ssl_server_name on;
        proxy_ssl_protocols TLSv1.2 TLSv1.3;
    }

    # caché a largo plazo para archivos estáticos como paquetes de instalación
    location / {
        proxy_cache azure_cache;
        proxy_cache_valid 200 7d;
        proxy_cache_key "$scheme$server_port$request_uri";
        add_header X-Cache-Status $upstream_cache_status;
        add_header Cache-Control "public, max-age=604800";

        proxy_pass https://${SERVER_DL_HOST}/${SERVER_DL_CONTAINER}$uri?${SERVER_DL_SAS_TOKEN};
        proxy_ssl_server_name on;
        proxy_ssl_protocols TLSv1.2 TLSv1.3;
    }
}
```

**¿Por qué diseñarlo así?**

`index.json` es un archivo de verificación de versión, debe actualizarse a tiempo, así que el tiempo de caché se estableció en 1 hora. De esta manera, después de lanzar una nueva versión, como máximo 1 hora los usuarios pueden detectar la actualización, tampoco es demasiado largo.

Los archivos estáticos como paquetes de instalación, cambian poco, cachear 7 días puede reducir drásticamente el acceso al sitio de origen. Cuando necesitas actualizar, limpia manualmente el caché y ya está, tampoco es molesto.

**Encabezado de respuesta X-Cache-Status**:

Este encabezado de respuesta puede dejarte ver cómo está el acierto de caché, también es bastante útil:
- `HIT`: acierto de caché
- `MISS`: fallo de caché, obtener del sitio de origen
- `EXPIRED`: caché expirado, volver a obtener del sitio de origen
- `BYPASS`: caché omitido

## Análisis de costos

Asumiendo tráfico de descarga mensual de 1TB, calculemos la cuenta:

| Plan | Tarifa de tráfico | Tarifa de servidor | Total |
|------|----------|------------|------|
| Azure OSS directo | aprox. ¥500 | ¥0 | ¥500 |
| Servidor en la nube + OSS (tasa de aciertos de caché 80%) | ¥100 + ¥80 | ¥60 | ¥240 |
| CDN comercial | ¥300-500 | ¥0 | ¥300-500 |

**Conclusión**: la capa de caché puede ahorrar aproximadamente el 50% del costo de distribución.

Este cálculo asume una tasa de aciertos de caché del 80%, en situaciones reales, si la frecuencia de actualización de archivos no es alta, la tasa de aciertos puede ser incluso más alta, después de todo esto también es algo natural.

## Práctica de despliegue

### Preparación del entorno

Primero configura las variables de entorno:

```bash
cd /path/to/hagicode_aliyun_deployment/docker
cp .env.example .env
vi .env  # ingresa URL SAS de Azure OSS, URL de Webhook de Lark
```

**Importante**: el archivo `.env` contiene información sensible (Token SAS, URL de Webhook), nunca lo envíes al control de versiones, esto es bastante crítico.

### Configuración DNS

Agrega los siguientes registros DNS A, no olvides este paso:
- `server.dl.hagicode.com` → IP del servidor
- `desktop.dl.hagicode.com` → IP del servidor

### Inicializar servidor

Use Ansible para inicializar automáticamente el servidor:

```bash
cd /path/to/hagicode_aliyun_deployment
ansible-playbook -i ./ansible/inventory/hosts.yml ./ansible/playbooks/init.yml
```

Este Playbook automáticamente resolverá estos:
- Crear usuario de despliegue
- Instalar Docker y Docker Compose
- Configurar claves SSH
- Establecer reglas de firewall

Tampoco es demasiado complejo, después de todo la automatización, ahorra tiempo y esfuerzo.

### Desplegar servicio

```bash
./deploy.sh
```

El script de despliegue te ayudará a hacer estos:
- Verificar configuración del entorno
- Obtener el último código
- Iniciar contenedores Docker
- Ejecutar verificaciones de salud
- Enviar notificación de despliegue (Lark)

Un comando para resolverlo, también es conveniente.

## Consejos de operación

### Gestión de caché

El caché, ocasionalmente también hay que cuidarlo:

**Ver uso de disco de caché**:
```bash
docker volume inspect docker_nginx-cache
du -sh /var/lib/docker/volumes/docker_nginx-cache/_data
```

**Limpiar caché manualmente**:
```bash
./clear-cache.sh
```

O ejecutar manualmente, aunque es un poco más molesto, pero también funciona:
```bash
docker exec nginx sh -c "rm -rf /var/cache/nginx/*"
docker restart nginx
```

### Límites de recursos

En servidor de 1 núcleo 2GB, la configuración de límites de recursos es la siguiente:

```yaml
services:
  traefik:
    deploy:
      resources:
        limits:
          cpus: '1.50'
          memory: 512M

  nginx:
    deploy:
      resources:
        limits:
          cpus: '0.50'
          memory: 256M
```

Monitorear el uso de recursos, ocasionalmente ver también está bien:
```bash
docker stats
```

### Prácticas de seguridad de SAS Token

El Token SAS es la credencial para acceder a Azure Blob Storage, si se filtra no es broma:

- El archivo `.env` no se envía al control de versiones (ya está en .gitignore)
- Establecer tiempo de expiración apropiado para el Token SAS (recomendado 1 año)
- Limitar permisos del Token SAS (solo lectura)
- Rotar periódicamente el Token SAS

### Monitoreo y alertas

HagiCode integra notificaciones de Webhook de Lark/Feishu, puede enviar notificaciones en las siguientes situaciones:
- Despliegue exitoso/fracasado
- Estado de limpieza de caché
- Anomalía de servicio

Las notificaciones incluyen información del servidor, marca de tiempo, detalles de error, localizar rápidamente problemas también es más conveniente.

### Extensión de alta disponibilidad

Cuando un solo servidor no puede soportar, también puedes considerar:

1. **Escalado horizontal**: desplegar múltiples nodos, distribuir a través de round-robin DNS o balanceo de carga
2. **Soporte CDN**: agregar CDN frente al servidor en la nube, mejorar aún más la velocidad de acceso
3. **Precalentamiento de caché**: usar scripts para cargar archivos populares en el caché por adelantado

## Precauciones

Hay algunos puntos que todavía hay que recordar, después de todo nadie quiere encontrar problemas:

1. **Certificado SSL**: Let's Encrypt tiene límites de velocidad, no cambies de despliegue frecuentemente, de lo contrario puede que no puedas obtener certificados
2. **Limpieza de caché**: después de actualizar archivos importantes recuerda limpiar el caché, de lo contrario los usuarios pueden no descargar la nueva versión
3. **Gestión de registros**: limpia periódicamente los registros de Docker, de lo contrario si el disco se llena es problemático
4. **Estrategia de respaldo**: Traefik acme.json, configuración de Bunker Web, estos es mejor hacer una copia de seguridad
5. **Monitoreo y alertas**: configura notificaciones de Feishu, conoce oportunamente el estado de despliegue, si hay problemas también puedes reaccionar rápidamente
