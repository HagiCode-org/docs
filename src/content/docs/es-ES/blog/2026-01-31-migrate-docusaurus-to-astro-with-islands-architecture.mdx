## De Docusaurus 3.x a Astro 5.x: Repaso práctico de la migración del sitio HagiCode

> Este artículo repasa el proceso completo de migración del sitio web oficial de HagiCode de Docusaurus 3.x a Astro 5.x. Profundizaremos en cómo resolver los cuellos de botella de rendimiento a través de la arquitectura Islands de Astro, al tiempo que conservamos los activos de componentes React existentes, logrando una doble mejora en la velocidad de construcción y el rendimiento de carga.


## Antecedentes

En enero de 2026, realizamos una "operación de trasplante de corazón" en el sitio oficial de HagiCode: migramos por completo el framework central de Docusaurus 3.x a Astro 5.x. Esta no fue una gran refactorización impulsiva, sino una decisión técnica cuidadosamente considerada.

Antes de la migración, nuestro sitio, aunque funcional, comenzó a mostrar algunos "síntomas de prosperidad": el volumen de los artefactos de construcción era hinchado, la carga de JavaScript era excesiva, y la velocidad de carga de las páginas no era ideal en páginas de documentación complejas. Como un proyecto de asistente de código AI, HagiCode necesita actualizar con frecuencia la documentación y las descripciones de funciones, por lo que la eficiencia de construcción afecta directamente la velocidad de publicación. Al mismo tiempo, deseábamos que el sitio fuera más amigable para los motores de búsqueda (SEO) para que más desarrolladores pudieran descubrir el proyecto.

Para resolver estos puntos de dolor, tomamos una decisión audaz: reconstruir todo el sistema de construcción y migrar a Astro. Los cambios que esta decisión trajo son mayores de lo que imaginas—hablaré de esto en detalle más adelante.

## Sobre HagiCode

El plan de migración del sitio compartido en este artículo proviene de nuestra experiencia práctica en el proyecto [HagiCode](https://github.com/HagiCode-org/site).

HagiCode es un asistente de código AI dedicado a mejorar la eficiencia de desarrollo. No solo nos enfocamos en la iteración de funciones centrales, sino que también valoramos la experiencia del desarrollador. Esta refactorización del sitio también tiene como objetivo que los usuarios obtengan una velocidad de carga extrema al navegar por la documentación y el sitio web oficial.

## ¿Por qué renunciar al maduro Docusaurus?

En el ecosistema de React, Docusaurus siempre ha sido la "respuesta estándar" para sitios de documentación. Es listo para usar, tiene complementos ricos y una comunidad activa. Sin embargo, con el aumento de las funciones de HagiCode, también sentimos sus limitaciones:

1.  **Cuellos de botella de rendimiento**: Docusaurus es esencialmente una React SPA (aplicación de una sola página). Incluso si escribe páginas puramente estáticas, el cliente necesita cargar el tiempo de ejecución de React y realizar la hidratación, lo cual es demasiado pesado para páginas de documentación simples.
2.  **Volumen de recursos**: Incluso si el contenido de la página es muy escaso, el volumen de JS empaquetado es relativamente fijo, lo cual no es amigable para los usuarios móviles y entornos con redes deficientes.
3.  **Insuficiente flexibilidad**: Aunque se puede extender, anhelamos un control más de bajo nivel en la personalización del proceso de construcción.

La aparición de Astro resuelve exactamente estos problemas. Proporciona una nueva "arquitectura de islas" (Islands Architecture): de manera predeterminada, Astro genera HTML estático con cero JavaScript, solo los componentes que necesitan interacción se "activan" y cargan JS. Esto significa que la mayor parte del contenido de nuestro sitio es HTML puro, extremadamente rápido.

## Estrategia central de migración: transición suave de arquitectura

La migración no es un simple copiar y pegar, sino un cambio de mentalidad. Cambiamos del "modo React completo" de Docusaurus al modo "Core + Islands" de Astro.

### 1. Refactorización del sistema de configuración

Primero, necesitamos pasar de `docusaurus.config.ts` a `astro.config.mjs`. Esto no es solo un cambio de nombre de archivo, sino una reescritura de la lógica de enrutamiento y construcción.

En Docusaurus, todo es un complemento; mientras que en Astro, todo es una integración. Necesitamos redefinir la ruta base del sitio, el modo de salida de construcción (estático vs SSR) y la estrategia de compresión de recursos.

**Antes de la migración:**
```typescript
// docusaurus.config.ts
export default {
  title: 'HagiCode',
  url: 'https://hagicode.com',
  baseUrl: '/',
  // ... más configuración
};
```

**Después de la migración:**
```javascript
// astro.config.mjs
import { defineConfig } from 'astro/config';
import react from '@astrojs/react';

export default defineConfig({
  integrations: [react()],
  site: 'https://hagicode.com',
  base: '/',
  // Configuración de optimización para recursos estáticos
  build: {
    inlineStylesheets: 'auto',
  },
});
```

### 2. Conservación y transformación de componentes React

Esta es la parte más dolorosa de la migración. Nuestro sitio actual tiene muchos componentes React (como el componente Tabs, resaltado de código, botones de retroalimentación, etc.). Desecharlos directamente es una pena, pero conservarlos todos provocaría una carga de JS excesiva.

HagiCode adoptó una estrategia de **hidratación progresiva**:
*   **Componentes puramente estáticos**: Para contenido de exhibición (como encabezados, pies de página, documentación de texto sin formato), reescribirlos como componentes Astro (archivos `.astro`), renderizados directamente como HTML en el momento de la construcción.
*   **Islas interactivas**: Para componentes que deben conservar interacción (como el conmutador de temas, cambio de Tabs, botones de copia en bloques de código), conservar la implementación de React y agregar las directivas `client:load` o `client:visible`.

Por ejemplo, nuestro componente Tabs comúnmente usado en la documentación:

```jsx
// src/components/Tabs.jsx
import { useState } from 'react';
import './Tabs.css'; // Importar estilos

export default function Tabs({ items }) {
  const [activeIndex, setActiveIndex] = useState(0);
  // ... lógica de estado
  return (
    <div className="tabs-wrapper">
      {/* lógica de renderizado */}
    </div>
  );
}
```

Al usarlo en Markdown, le decimos explícitamente a Astro: "este componente necesita JS":

```markdown
// src/content/docs/example.mdx
import Tabs from '../../components/Tabs.jsx';

<!-- Solo carga JS cuando el componente entra en la ventana -->
<Tabs client:visible items={...} />
```

De esta manera, los componentes interactivos fuera de la ventana no acapararán el ancho de banda, optimizando enormemente la velocidad de carga de la primera pantalla.

### 3. Adaptación del sistema de estilos: de CSS Modules a Scoped

Docusaurus admite CSS Modules de manera predeterminada, mientras que Astro promueve el uso de Scoped CSS (a través de etiquetas `<style>`). Ambas ideas centrales son aislar estilos, pero la sintaxis es diferente.

En la migración de HagiCode, desglosamos la mayoría de los CSS Modules complejos en estilos Scoped de Astro. En realidad, esto es algo bueno, porque en los archivos `.astro`, los estilos y las plantillas se escriben en el mismo archivo, lo que hace que el mantenimiento sea más intuitivo.

**Antes de la transformación:**
```css
/* Tabs.module.css */
.wrapper { background: var(--ifm-background-color); }
```

**Después de la transformación (Astro Scoped):**
```html
<!-- Tabs.astro -->
<div class="tabs-wrapper">
  <slot />
</div>

<style>
  .tabs-wrapper {
    /* Usar variables CSS directamente, adaptarse al tema */
    background: var(--bg-color);
    padding: 1rem;
  }
</style>
```

Al mismo tiempo, unificamos el sistema global de variables CSS, aprovechando la capacidad de感知 ambiental de Astro para garantizar que la transición del modo oscuro entre diferentes páginas sea fluida.

## Problemas prácticos y soluciones

En el proceso de migración real de HagiCode, encontramos muchos problemas. Aquí comparto algunos de los más típicos.

### 1. Dolor de cabeza con rutas y variables de entorno

HagiCode admite el despliegue en subrutas (como el despliegue en subdirectorios de GitHub Pages). En Docusaurus, maneja automáticamente `baseUrl`. Pero en Astro, al manejar enlaces de imágenes y solicitudes de API, necesitamos tener más cuidado.

Introdujimos un mecanismo de variables de entorno para gestionar de manera uniforme:

```javascript
// Manejar rutas en el script de construcción
const getBasePath = () => import.meta.env.VITE_SITE_BASE || '/';
```

Recuerde, no codifique rutas que comiencen con `/` en el código. En el entorno de desarrollo y producción, o después de configurar la ruta `base`, esto provocará que los recursos devuelvan 404.

### 2. Compatibilidad con scripts CommonJS

Nuestro sitio antiguo tenía algunos scripts de Node.js (para extraer automáticamente datos de Metrics, actualizar sitemaps, etc.), que estaban escritos en CommonJS (`require`). Astro y las herramientas de construcción modernas abrazan completamente los ES Modules (`import`/`export`).

Si también tiene scripts similares, recuerde refactorizarlos todos a ES Modules. Esta es la tendencia general, cámbielos antes para ahorrar preocupaciones.

```javascript
// Forma antigua
const fs = require('fs');

// Nueva forma
import fs from 'fs';
```

### 3. No olvide el SEO y las redirecciones

Los motores de búsqueda ya han indexado las páginas antiguas de Docusaurus de HagiCode. Si cambiamos directamente a Astro y la estructura de URL cambia, provocará muchos 404 y una gran caída de autoridad.

Configuramos reglas de redirección en Astro:

```javascript
// astro.config.mjs
export default defineConfig({
  redirects: {
    '/docs/old-path': '/docs/new-path',
    // Mapeo por lotes de enlaces antiguos a nuevos
  }
});
```
O manejarlo a nivel de configuración del servidor. Asegúrese de que los enlaces antiguos puedan redirigirse 301 a la nueva dirección, lo cual es crucial para el SEO.

## Resumen

Migrar de Docusaurus a Astro, para HagiCode, no es solo una actualización de framework, sino también una práctica del concepto "rendimiento primero".

**Nuestros logros:**
*   **Puntuación Lighthouse extrema**: Después de la migración, la puntuación de rendimiento del sitio HagiCode se acerca fácilmente a la puntuación perfecta.
*   **Velocidad de construcción más rápida**: La construcción incremental de Astro redujo a la mitad el tiempo de publicación de actualizaciones de nuestra documentación.
*   **Conservamos la flexibilidad**: A través de la arquitectura Islands, no sacrificamos ninguna función interactiva, y aún podemos usar React donde sea necesario.

Si también está manteniendo un sitio de documentación y sufre problemas de volumen de empaquetado o velocidad de carga, pruebe Astro. Aunque el proceso de migración requiere algo de cirugía (como cambiar el nombre de PCode a HagiCode, desenterrar los componentes uno por uno), lo que se gana es una experiencia de usuario fluida como la seda, absolutamente vale la pena.

El sistema de construcción compartido en este artículo es exactamente el plan que desarrollamos al encontrar problemas y optimizar durante el desarrollo de HagiCode. Si considera que este plan es valioso, significa que nuestra capacidad de ingeniería es bastante buena—entonces HagiCode mismo también vale la pena prestarle atención.
