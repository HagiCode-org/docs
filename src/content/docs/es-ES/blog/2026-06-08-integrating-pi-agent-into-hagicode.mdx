## El camino práctico para integrar Pi Agent en HagiCode

> Este artículo comparte nuestra experiencia práctica al integrar Pi agent como un punto de entrada de flujo de trabajo de primera clase en el proyecto HagiCode, incluyendo el diseño de arquitectura, implementación de componentes principales y detalles de adaptación de frontend y backend.

## Antecedentes

En realidad, todo comienza con esa pregunta: en el proyecto HagiCode Mono, aunque `repos/Hagicode.Libs` ya ha implementado el `PiProvider` reutilizable, `repos/hagicode-core` y `repos/web` aún no han elevado Pi a un Agent CLI de primera clase a nivel de proyecto. Es como tener buenos zapatos pero aún no haber atado los cordones: puedes caminar, pero sientes que falta algo.

El `PiProvider` existente se encuentra en `HagiCode.Libs/src/HagiCode.Libs.Providers/Pi/PiProvider.cs`, implementa un protocolo de comunicación CLI basado en JSON separado por líneas (`--mode json --print`), y admite gestión de sesiones, invocación de herramientas y salida en streaming. Este código está bien escrito, solo que está ahí esperando ser activado.

Esta situación crea una desconexión: las capacidades subyacentes de Pi están completas, pero la ruta de integración a nivel de proyecto (desde la configuración del usuario hasta el monitoreo de ejecución) está ausente. Es como haber pintado un buen cuadro pero no haberlo colgado en la pared para que otros lo aprecien. Por lo tanto, necesitamos integrar Pi como un nuevo provider activo `PiCli` en todo el sistema, convirtiéndolo en un punto de entrada de flujo de trabajo de primera clase como otros Agent CLI.

Después de todo, lo que queremos es una experiencia completa, no fragmentos dispersos.

## Sobre HagiCode

La solución compartida en este artículo proviene de nuestra experiencia práctica en el proyecto [HagiCode](https://hagicode.com). HagiCode es un proyecto de asistente de código inteligente, y durante el desarrollo necesitamos integrar múltiples proveedores de capacidades de IA. La solución de integración de Pi agent descrita en este artículo es precisamente un patrón de integración reutilizable que resumimos en nuestra práctica, y tiene valor de referencia para escenarios similares de integración de múltiples proveedores. La dirección de GitHub del proyecto es [github.com/HagiCode-org/site](https://github.com/HagiCode-org/site).

## Diseño de arquitectura

La integración del agente Pi adoptó el **patrón thin adapter**, en lugar de reimplementar el protocolo de proceso Pi en la capa core. Esta decisión de diseño no es realmente compleja, es simplemente pensar: ¿por qué reinventar la rueda?

Después de todo:

1. **Evitar implementación duplicada**: la lógica de construcción de parámetros, inicio de procesos, análisis JSON y manejo de errores en `PiProvider` ya está muy completa. La implementación duplicada crearía dos fuentes de comportamiento y dos matrices de prueba. Bueno, pero el mantenimiento sería problemático.
2. **Mantener consistencia**: mantener consistencia con los métodos de integración de CLI existentes como Kimi, Gemini, Reasonix, DeepAgents, todos delegando a implementaciones de la capa libs a través de thin adapters. Todos lo hacen así, simplemente sigue el camino.
3. **Separación de preocupaciones**: `hagicode-core` se centra en contratos de tiempo de ejecución y lógica de negocio, mientras que los detalles del proceso se delegan a `HagiCode.Libs`. Cada uno hace su trabajo, ¿no es hermoso?

Este diseño permite que HagiCode mantenga la capa core concisa mientras integra rápidamente nuevos proveedores de capacidades de IA. Después de todo, la simplicidad es belleza, y la eficiencia es dinero.

## Implementación de componentes principales

### Enumeración Provider y Factory

En `hagicode-core/src/PCode.Models/AIProviderType.cs` se agregó el valor de enumeración `PiCli = 13`:

```csharp
public enum AIProviderType
{
    ClaudeCodeCli = 0,
    // ... otros providers
    PiCli = 13,
}
```

Este valor de enumeración es la raíz de la identidad del provider, afectará la enumeración frontend `PCode_Models_AIProviderType.ts` generada por OpenAPI, la rama de creación de `AIProviderFactory`, y la lógica de análisis de Provider y juicio de provider activo. Registrar la nueva rama de creación en `AIProviderFactory`:

```csharp
case AIProviderType.PiCli:
    provider = new PiCliProvider(
        logger,
        configuration,
        providerFactory.GetRequiredService<ICliProvider<PiOptions>>(),
        providerFactory.GetService<IAgentCliRuntimeEnvironmentResolver>());
    break;
```

En realidad, este código no es nada especial, es solo una enumeración más un switch case. Solo que son importantes, como las notas en una partitura, agregadas una por una pueden interpretar una melodía completa.

### Implementación de Thin Adapter

`PiCliProvider.cs` es el thin adapter principal, implementa las interfaces `IAIProvider`, `IVersionedAIProvider` e `IAsyncDisposable`. A través del constructor recibe `ICliProvider<PiOptions>` (de `HagiCode.Libs`), mapea `AIRequest` / `ProviderConfiguration` a `PiOptions`, y luego delega operaciones como ejecución, ping y consulta de versión al provider subyacente.

La clave es manejar el flujo de eventos JSON específico de Pi, incluyendo eventos como `assistant.thought`, `assistant`, `terminal.completed`, etc. Estos eventos necesitan ser analizados y convertidos correctamente al formato estándar del sistema durante el proceso de salida en streaming. Es un poco como traducción, traducir un idioma a otro, el significado debe transmitirse con precisión.

### Preset de profesión principal

En `main-professions.yaml` se agregó la entrada de profesión principal `profession-pi`:

```yaml
- Id: "profession-pi"
  Name: "Pi"
  Family: "pi"
  Summary: "hero.professionCopy.primary.pi.summary"
  Icon: "executor-avatar:Pi"
  SourceLabel: "hero.professionCopy.sources.aiProvidersPiCli"
  ProviderType: "PiCli"
  SortOrder: 59
  DefaultEnabled: true
  DefaultParameters:
    binary: "pi"
    provider: "omniroute"
    thinking: "balanced"
```

Esto asegura que la instantánea del backend y el directorio fallback del frontend tengan una identidad Pi consistente, la configuración de Pi puede ser gestionada a través del editor Hero existente, y la adición de Pi no romperá la consumibilidad de las entradas de profesión principal existentes. Después de todo, no queremos desordenar las cosas originales al agregar una nueva función, eso sería perder lo general por lo particular.

### Registro de monitoreo

`AgentCliMonitoringRegistry` necesita agregar el descriptor de monitoreo de Pi, para que el sistema pueda analizar la ruta ejecutable, mostrar el nombre de la marca, realizar sondas de salud, y mostrar el estado de Pi en la barra de estado y detalles de salud:

```csharp
new AgentCliMonitoringDescriptor(
    CliId: "pi",
    DisplayName: "Pi",
    ProviderType: AIProviderType.PiCli,
    DisplayOrder: 13,
    Strategy: AgentCliMonitoringStrategy.Grain,
    NotConfiguredMessage: "Pi CLI is not configured or executable not found.",
    EnabledPaths: [],
    ExecutablePathConfigPaths: ["Hero:PrimaryProfessions:Pi:ExecutablePath"],
    DefaultExecutablePath: "pi")
```

El sistema de monitoreo es como el tablero de instrumentos, te dice cómo está funcionando el coche. El estado de Pi es claro a primera vista, para que los usuarios sepan si todo está normal.

## Adaptación del frontend

### Adaptación de tipo de ejecutor

El frontend necesita actualizar `executorTypeAdapter.ts`, agregando la lógica de reconocimiento de tipo de Pi:

```typescript
const PI_IDS = new Set<string>([
  PCode_Models_AIProviderType.PI_CLI,
  'Pi',
  'PiCli',
  'pi',
  'picli',
  'pi-cli',
]);

export function isPi(value: string): boolean {
  const normalized = normalize(value);
  const normalizedLower = normalized.toLowerCase();
  return PI_IDS.has(normalized)
    || normalizedLower.includes('pi-cli')
    || normalizedLower.includes('picli')
    || normalizedLower === 'pi';
}
```

Es como darle a Pi varios nombres, no importa cómo lo llames, sabe que te estás refiriendo a él. Después de todo, el nombre no es importante, lo importante es saber quién eres.

### Directorio Fallback Hero

En `hero.ts` agregar la entrada fallback de Pi, asegurando que incluso si los datos del backend no están cargados, el frontend pueda mostrar la configuración de Pi normalmente:

```typescript
{
  id: "profession-pi",
  name: "Pi",
  family: "pi",
  summary: "hero.professionCopy.primary.pi.summary",
  icon: "executor-avatar:Pi",
  sourceLabel: "hero.professionCopy.sources.aiProvidersPiCli",
  providerType: AIProviderType.PI_CLI,
  sortOrder: 59,
  isReadOnly: true,
  managedParameterKeys: {
    // keys de parámetros soportados en la primera versión
  },
  defaultParameters: {
    binary: "pi",
    provider: "omniroute",
    thinking: "balanced",
  },
}
```

Fallback es como un plan de respaldo, en caso de que el backend falle o los datos no se carguen, el frontend puede funcionar normalmente. Después de todo, nadie quiere estar desordenado cuando llegue el momento.

### Copias localizadas y formularios

Agregar traducciones relacionadas con Pi en `locales/*/common/{hero,settings}.yml`, y en `HeroCliEquipmentForm.tsx` agregar un bloque de campos de configuración para Pi, soportando campos binary, provider, thinking, sessionDirectory y interruptores de herramientas/sesión.

La primera versión de Pi solo expone campos mínimos necesarios, funciones complejas como allowlist/denylist de herramientas y editor de variables de entorno se explícitamente pospusieron a cambios posteriores. Después de todo, no se puede ser gordo de un bocado, es más rápido ir poco a poco.

## Ejemplo de configuración

### Configuración del backend

Configurar parámetros de Pi en `appsettings.yml`:

```yaml
Hero:
  PrimaryProfessions:
    Pi:
      ExecutablePath: /usr/local/bin/pi
      Provider: omniroute
      Thinking: balanced
      SessionDirectory: ~/.pi/sessions
      NoSession: false
      DisableAllTools: false
      DisableBuiltinTools: false
```

### Configuración del frontend

Configurar profesión Pi en el editor Hero:

```typescript
{
  id: "my-pi-profession",
  name: "My Pi Profession",
  family: "pi",
  providerType: AIProviderType.PI_CLI,
  primaryModel: {
    provider: "PiCli",
    model: "glm-4.7",
    providerSettings: {
      provider: "omniroute",
      thinking: "balanced",
      sessionDirectory: "/Users/username/.pi/sessions",
      noSession: false,
      disableAllTools: false,
      disableBuiltinTools: false,
    },
  },
}
```

Los archivos de configuración son como recetas, siguiéndolas puedes hacer buena comida. Solo que a veces, incluso siguiendo la receta, puedes quemar la comida... pero esta configuración es muy clara, no debería haber problema.

## Solución de problemas comunes

### No se encuentra el ejecutable Pi

Si el chequeo de salud devuelve "Pi executable was not found.", necesitas verificar si pi está en PATH, o confirmar si la ruta configurada es correcta. El método de solución es asegurar que `pi` esté instalado y en PATH, o configurar el `ExecutablePath` correcto en `appsettings.yml`.

Es como no encontrar la llave de casa, tienes que pensar si la pusiste en el lugar equivocado. En realidad, la solución es bastante simple, ya sea poner la llave de vuelta en su lugar original, o cambiar a una nueva cerradura.

### Campos de configuración no reconocidos

Si al inicio se lanza el error "PiCli runtime settings [...] are not supported", verifica si solo estás usando los campos de configuración soportados en la primera versión. Los campos soportados en la primera versión incluyen: `provider`, `thinking`, `sessionDirectory`, `noSession`, `disableAllTools`, `disableBuiltinTools`.

A veces es codicia, quieres demasiadas funciones, y el resultado es que el sistema no las soporta. En realidad, las funciones de la primera versión ya son suficientes, codicia excesiva lleva a una digestión difícil.

### No se puede seleccionar Pi en el frontend

Si no hay opción Pi en el editor Hero, verifica si has ejecutado `npm run generate:api` para regenerar la enumeración del frontend, si hay una entrada `profession-pi` en `hero.ts`, y si las copias localizadas se agregaron correctamente.

Solucionar problemas es como buscar objetos perdidos, tienes que ir paso a paso. Después de todo, buscar ciegamente no sirve, tienes que buscar con lógica.

## Mejores prácticas

1. **Usar el patrón thin adapter**: no reimplementes el protocolo de proceso en la capa core, delega al provider de la capa libs. Esto puede evitar implementación duplicada y mantener la consistencia del código. Después de todo, reinventar la rueda no solo es agotador, también es propenso a problemas.

2. **Mantener consistencia de nombres**: usar convenciones de nombres unificadas en frontend y backend, evitar confusión. Enumeración de provider usa `PiCli`, ID de CLI usa `"pi"`, nombre de visualización usa `"Pi"`. Si eliges un buen nombre, el costo de comunicación es menor.

3. **Priorizar el uso de presets**: la primera versión debe basarse en el preset `profession-pi`, en lugar de requerir que los usuarios configuren manualmente. Esto permite que los usuarios comiencen rápidamente y reduzca la complejidad de configuración. A los usuarios les gustan las cosas simples, déjenme las cosas complejas.

4. **Prestar atención a los mensajes de error**: asegurar que los mensajes de error sean claros y accionables, ayudando a los usuarios a localizar problemas rápidamente. Si los mensajes de error están escritos claramente, los usuarios no se volverán locos por un error.

5. **Compatibilidad de versiones**: considerar la estabilidad de serialización de los valores de enumeración `AIProviderType`, los cambios necesitan ser manejados con precaución. El valor de enumeración `AIProviderType.PiCli = 13` no puede modificarse fácilmente. Después de todo, cambiar este valor puede romper la compatibilidad hacia atrás, eso sería problemático.

## Resumen

A través del patrón thin adapter, integramos exitosamente Pi agent en el sistema HagiCode, convirtiéndolo en un punto de entrada de flujo de trabajo de primera clase como otros Agent CLI. La ventaja central de esta solución es:

- Evitó implementación duplicada, reutilizando el `PiProvider` existente en la capa libs
- Mantuvo un método de integración consistente con los providers existentes, reduciendo el costo de mantenimiento
- Implementó la cadena completa desde la configuración del usuario hasta el monitoreo de ejecución

La práctica de HagiCode demuestra que el patrón thin adapter es una solución efectiva para integrar proveedores de capacidades de IA. Nos permite soportar rápidamente nuevos agentes mientras mantenemos la estabilidad y mantenibilidad del sistema.

De hecho, hacer tecnología es así, encontrar un buen patrón, y luego reutilizarlo. Esto puede avanzar rápidamente sin perder la dirección. Es como caminar, encuentras un buen camino y sigues caminando, solo que a veces también te detienes a ver el paisaje...

Si también estás haciendo integración de capacidades de IA de múltiples proveedores, espero que esta solución te dé algunas inspiraciones. Si estás interesado en el proyecto HagiCode, bienvenido a venir a GitHub para intercambiar. Después de todo, la tecnología necesita más comunicación para progresar.
