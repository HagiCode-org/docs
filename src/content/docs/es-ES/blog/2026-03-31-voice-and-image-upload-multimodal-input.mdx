## Escribir es menos eficiente que hablar, hablar es menos eficiente que capturas de pantalla: prácticas de entrada multimodal para asistentes de código AI

> En realidad, cuando se trata de escribir código, escribir rápido tiene un límite. A veces, una sola frase se puede decir en segundos, pero requiere mucho teclear; a veces, una imagen explica todo claramente, pero requiere una pila de texto para describirlo. Este artículo habla sobre nuestras experiencias al hacer [HagiCode](https://hagicode.com) — reconocimiento de voz, carga de imágenes, bueno, solo queríamos hacer que el asistente de código AI fuera más fácil de usar, eso es todo.

## Antecedentes

Al hacer HagiCode, descubrimos un problema — o más bien, un problema que naturalmente surgió cuando los usuarios lo usaron más: depender solo de escribir a veces es bastante agotador.

Piénsalo, los usuarios interactúan con Agent, este es el escenario central. Pero cada vez tienes que sentarte frente al teclado y teclear como loco, ¿cómo decirlo?, la eficiencia realmente no es muy alta:

1. **Escribir es demasiado lento**: Algunos problemas complejos, errores, cosas de la interfaz, etc., tomarían media minuto escribirlos, pero verbalmente se pueden decir en diez segundos. Esta diferencia de tiempo es bastante frustrante.

2. **Las imágenes son más directas**: A veces la interfaz tiene un error, o quieres comparar un diseño, o quieres mostrar la estructura del código... "una imagen vale más que mil palabras", aunque la frase es vieja, la verdad no lo es. Dejar que la AI "vea" directamente el problema es mucho más claro que describirlo durante mucho tiempo.

3. **La interacción debería ser natural**: Los asistentes de AI actuales deberían admitir texto, voz, imágenes y otros métodos, ¿verdad? Los usuarios deberían poder usar lo que quieran, eso es natural, ¿no?

Así que pensamos, ¿por qué no agregar funciones de reconocimiento de voz y carga de imágenes a HagiCode para hacer que la operación del Agent sea más conveniente? Después de todo, permitir que los usuarios escriban menos caracteres también es bueno.

## Sobre HagiCode

Las soluciones compartidas en este artículo provienen de nuestra práctica en el proyecto [HagiCode](https://hagicode.com) — o más bien, experiencia explorada mientras tropezábamos constantemente.

HagiCode es un proyecto de asistente de código AI de código abierto, la idea es simple: usar tecnología AI para mejorar la eficiencia del desarrollo. Mientras lo hacíamos, descubrimos que la demanda de entrada multimodal de los usuarios es bastante fuerte — a veces decir una frase es más rápido que escribir un montón de palabras, a veces una captura de pantalla es más clara que describir durante mucho tiempo.

Estas demandas nos impulsaron hacia adelante, y finalmente tuvimos funciones como reconocimiento de voz y carga de imágenes. Los usuarios pueden interactuar con la AI de la manera más natural, esta sensación, es bastante buena.

## Análisis

### Desafíos técnicos del reconocimiento de voz

Al hacer la función de reconocimiento de voz, encontramos un problema bastante difícil: **la API de WebSocket del navegador no admite encabezados HTTP personalizados**.

Y el servicio de reconocimiento de voz que elegimos fue la API de reconocimiento de voz de Doubao de ByteDance. Esta API específicamente requiere pasar información de autenticación a través de encabezados HTTP, como `accessToken`, `secretKey`, etc. Ahora bien, surge la contradicción técnica:

```javascript
// La API de WebSocket del navegador no admite la siguiente forma
const ws = new WebSocket('wss://api.com/ws', {
  headers: {
    'Authorization': 'Bearer token'  // no admitido
  }
});
```

Las soluciones ante nosotros son aproximadamente dos:

1. **Esquema de parámetros de consulta URL**: poner la información de autenticación en la URL
   - La ventaja es que es simple de implementar
   - La desventaja es que las credenciales se exponen en el frontend, mala seguridad; y algunas APIs requieren obligatoriamente verificación de encabezados

2. **Esquema de proxy backend**: implementar un proxy WebSocket en el backend
   - La ventaja es que las credenciales se almacenan de forma segura en el backend; completamente compatible con los requisitos de la API
   - La desventaja es que la implementación es un poco más compleja

Finalmente elegimos el **esquema de proxy backend**. Después de todo, la seguridad es una línea que no se puede comprometer — en esto, nadie puede engañar a nadie.

### Requisitos funcionales de carga de imágenes

La función de carga de imágenes, nuestros requisitos son en realidad bastante simples:

1. **Múltiples métodos de carga**: hacer clic para seleccionar archivo, arrastrar para cargar, pegar desde portapapeles, al menos debe tener estos, ¿verdad?
2. **Validación de archivos**: límite de tipo (PNG, JPG, WebP, GIF), límite de tamaño (5-10MB), estas son operaciones básicas
3. **Experiencia de usuario**: progreso de carga, vista previa, indicaciones de error, al menos hay que dejar que la gente sepa qué está sucediendo
4. **Seguridad**: validación del lado del servidor, prevención de carga de archivos maliciosos, esto es algo grande

## Soluciones

### Reconocimiento de voz: arquitectura de proxy WebSocket

Diseñamos una solución de reconocimiento de voz de arquitectura de tres capas, ¿cómo decirlo?, encontramos un camino:

```
Browser WebSocket
       |
       | ws://backend/api/voice/ws
       | (binary audio)
       v
Backend Proxy
       |
       | wss://openspeech.bytedance.com/ (with auth header)
       v
Doubao API
```

**Implementación de componentes principales**:

1. **Procesador AudioWorklet del frontend**:

```javascript
class AudioProcessorWorklet extends AudioWorkletProcessor {
  process(inputs, outputs, parameters) {
    const input = inputs[0]?.[0];
    if (!input) return true;

    // Remuestrear a 16kHz (requerimiento de la API de Doubao)
    const samples = this.resampleAudio(input, 48000, 16000);

    // Acumular muestras en bloques de 500ms
    this.accumulatedSamples.push(...samples);

    if (this.accumulatedSamples.length >= 8000) {
      // Convertir a PCM de 16-bit y enviar
      const pcm = this.floatToPcm16(this.accumulatedSamples);
      this.port.postMessage({ type: 'audioData', data: pcm.buffer }, [pcm.buffer]);
      this.accumulatedSamples = [];
    }
    return true;
  }
}
```

2. **Procesador WebSocket del backend** (C#):

```csharp
[HttpGet("ws")]
public async Task GetWebSocket()
{
    if (HttpContext.WebSockets.IsWebSocketRequest)
    {
        await _webSocketHandler.HandleAsync(HttpContext);
    }
}
```

3. **Componente VoiceTextArea del frontend**:

```tsx
export const VoiceTextArea = forwardRef<HTMLTextAreaElement, VoiceTextAreaProps>(
```

### Carga de imágenes: componente de carga de múltiples formas

Hicimos un componente de carga de imágenes con funciones completas que admite tres métodos de carga, ¿cómo decirlo?, básicamente cubrimos todos los escenarios comunes de los usuarios.

**Características principales**:

1. **Tres métodos de carga**:

```tsx
// carga por clic
const handleClick = () => fileInputRef.current?.click();

// carga por arrastre
const handleDrop = (e: React.DragEvent) => {
  const file = e.dataTransfer.files?.[0];
  if (file) uploadFile(file);
};

// pegar desde portapapeles
const handlePaste = (e: ClipboardEvent) => {
  for (const item of Array.from(e.clipboardData?.items || [])) {
    if (item.type.startsWith('image/')) {
      const file = item.getAsFile();
      if (file) uploadFile(file);
    }
  }
};
```

2. **Validación del frontend**:

```tsx
const validateFile = (file: File): { valid: boolean; error?: string } => {
  if (!acceptedTypes.includes(file.type)) {
    return { valid: false, error: 'Only PNG, JPG, JPEG, WebP, and GIF images are allowed' };
  }
  if (file.size > maxSize) {
    return { valid: false, error: `Maximum file size is ${(maxSize / 1024 / 1024).toFixed(1)}MB` };
  }
  return { valid: true };
};
```

3. **Procesamiento de carga del backend** (TypeScript):

```typescript
export const Route = createFileRoute('/api/upload')({
  server: {
    handlers: {
      POST: async ({ request }) => {
        const formData = await request.formData();
        const file = formData.get('file') as File;

        // validar
        const validation = validateFile(file);
        if (!validation.isValid) {
          return Response.json({ error: validation.error }, { status: 400 });
        }

        // guardar archivo
        const uuid = uuidv4();
        const filePath = join(uploadDir, `${uuid}${extension}`);
        await writeFile(filePath, buffer);

        return Response.json({ url: `/uploaded/${today}/${uuid}${extension}` });
      }
    }
  }
});
```

## Guía de práctica

### Cómo usar el reconocimiento de voz

1. **Configurar el servicio de reconocimiento de voz**:
   - Entra a la página de configuración de reconocimiento de voz
   - Configura el `AppId` y `AccessToken` de Doubao Voice
   - (Opcional) Configura palabras clave para mejorar la precisión de reconocimiento de términos profesionales

2. **Usar en el campo de entrada**:
   - Haz clic en el icono de micrófono a la izquierda del campo de entrada
   - Comienza a hablar cuando veas la animación de forma de onda
   - Haz clic en el icono nuevamente para detener la grabación
   - El resultado del reconocimiento se insertará automáticamente en la posición del cursor

3. **Ejemplo de configuración de palabras clave**:

```text
TypeScript
React
useState
useEffect
```

### Cómo usar la carga de imágenes

1. **Métodos de carga**:
   - Haz clic en el botón de carga para seleccionar archivo
   - Arrastra directamente la imagen al área de carga
   - Usa `Ctrl+V` para pegar la captura de pantalla del portapapeles

2. **Formatos admitidos**: PNG, JPG, JPEG, WebP, GIF
3. **Límite de tamaño**: 5MB por defecto (configurable)

### Precauciones

1. **Reconocimiento de voz**:
   - Requiere permiso de micrófono
   - Se recomienda usar en un entorno tranquilo
   - La duración máxima de grabación admitida es de 300 segundos (configurable)

2. **Carga de imágenes**:
   - Solo admite formatos de imagen comunes
   - Presta atención al límite de tamaño de archivo
   - Las imágenes cargadas generarán automáticamente URL de vista previa

3. **Consideraciones de seguridad**:
   - Las credenciales de reconocimiento de voz se almacenan en el backend
   - La carga de imágenes tiene validación estricta del lado del servidor
   - Se recomienda usar HTTPS/WSS en entornos de producción

## Resumen

Después de agregar reconocimiento de voz y carga de imágenes, la experiencia de usuario de HagiCode realmente mejoró bastante. Los usuarios ahora pueden interactuar con la AI de una manera más natural — hablar en lugar de escribir, capturas de pantalla en lugar de describir. Esta sensación, ¿cómo decirlo?, es como finalmente encontrar una forma de comunicación más cómoda.

Al hacer esta función, encontramos el problema de que el WebSocket del navegador no admite encabezados personalizados, y finalmente lo resolvimos a través de la solución de proxy backend. Esta solución no solo garantiza la seguridad, sino que también sienta las bases para la integración posterior de otros servicios WebSocket que requieren autenticación — también se puede considerar como una ganancia inesperada.

El componente de carga de imágenes también es lo mismo, usamos múltiples métodos de carga para que los usuarios puedan elegir el más conveniente según el escenario. Ya sea clic, arrastre, o pegar directamente, todos pueden completar la carga rápidamente. Todos los caminos conducen a Roma, solo que algunos caminos son más fáciles de recorrer, otros son un poco más tortuosos.
