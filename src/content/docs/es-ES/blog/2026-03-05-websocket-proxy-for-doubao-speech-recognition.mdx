## Solucionando el problema de autenticación WebSocket en navegadores: práctica de solución proxy para reconocimiento de voz Doubao

> La API WebSocket del navegador no soporta headers HTTP personalizados, lo que plantea un desafío para los servicios de reconocimiento de voz que necesitan pasar información de autenticación a través de headers. Este artículo comparte cómo el proyecto HagiCode resolvió este problema mediante una solución de proxy backend, así como el proceso práctico desde playground hasta producción.


## Antecedentes

En realidad, al desarrollar la función de reconocimiento de voz para el proyecto HagiCode, también elegimos con plena confianza el servicio de reconocimiento de voz Doubao de ByteDance. El diseño inicial era muy simple: el frontend se conectaba directamente al servicio WebSocket de Doubao. ¿Qué tan difícil podría ser? ¿No se trata simplemente de establecer una conexión y transmitir algunos datos?

Pero, para nuestra sorpresa, la API de Doubao requiere pasar información de autenticación a través de headers HTTP, como accessToken, secretKey, etc. Esto fue un poco incómodo, ya que la API WebSocket del navegador simplemente no soporta establecer headers personalizados.

¿Qué se puede hacer si no es compatible?

En ese momento también estuvimos indecisos por un tiempo. Después de todo, había dos opciones ante nosotros:

1. Meter la información de autenticación en los parámetros de consulta de la URL: simple y bruto
2. Hacer una capa de proxy en el backend: parece un poco más problemático

Con el primer enfoque, las credenciales se expondrían directamente en el código frontend y el almacenamiento local. ¿Es esto seguro? Personalmente, no estoy muy de acuerdo. Además, algunas API deben usar validación por header, así que ese enfoque simplemente no funciona.

Finalmente, después de pensarlo, elegimos el segundo enfoque: implementar un proxy WebSocket en el backend. Por casualidad, esta solución se probó originalmente en nuestro entorno de pruebas playground, y solo después de confirmar su estabilidad se aplicó al entorno de producción. Después de todo, nadie quiere ser conejillo de indias en producción, entiendo eso al menos.

## Sobre HagiCode

La solución compartida en este artículo proviene de nuestra experiencia práctica en el proyecto [HagiCode](https://github.com/HagiCode-org/site).

HagiCode es un proyecto de asistente de código AI que admite funciones de interacción de voz. Es decir, debido a que necesitábamos llamar al servicio de reconocimiento de voz desde el frontend, nos encontramos con este problema de autenticación WebSocket, y de ahí surgió la solución posterior. A veces pienso que las dificultades no son del todo inútiles, al menos nos enseñaron a usar proxies, ¿no es así?

## Análisis de desafíos técnicos

### Limitaciones del WebSocket del navegador

La API WebSocket estándar parece realmente muy simple:

```javascript
const ws = new WebSocket('wss://example.com/ws');
```

Pero el problema está en la palabra "simple": solo pasa parámetros en la URL, y no puede establecer headers como las solicitudes HTTP:

```javascript
// Esto no es compatible con la API WebSocket
const ws = new WebSocket('wss://example.com/ws', {
  headers: {
    'Authorization': 'Bearer token'
  }
});
```

Mira, ¿a quién puedes reclamar? Para servicios como el reconocimiento de voz de Doubao que requieren autenticación por header, esta limitación es simplemente un obstáculo insuperable.

Bueno, ¿qué más se puede hacer?

### Decisiones de diseño de arquitectura

Al diseñar la solución, también lo pensamos mucho, sopesando una y otra vez.

**Decisión uno: elección del modo de proxy**

Comparamos dos enfoques:

| Enfoque | Ventajas | Desventajas | Decisión |
|---------|----------|-------------|----------|
| WebSocket nativo | Ligero, simple, reenvío directo | Requiere gestión manual de conexiones | Elegido |
| SignalR | Reconexión automática, tipado fuerte | Excesivamente complejo, dependencia adicional | No elegido |

Finalmente elegimos WebSocket nativo. Para ser honestos,主要是因为 es el más ligero y adecuado para reenvío simple de flujos binarios bidireccionales. Agregar SignalR sería como usar una cañonera para matar a un mosquito, y además aumentaría la latencia: ¿para qué molestarse?

**Decisión dos: estrategia de gestión de conexiones**

Adoptamos el modo de "una sesión por conexión": cada conexión WebSocket del frontend corresponde a una conexión independiente al backend de Doubao.

Las ventajas de este enfoque son obvias:
- Implementación simple, coincide con escenarios de uso típicos
- Fácil de depurar y solucionar problemas
- Aislamiento de recursos, evitando interferencias entre sesiones

En realidad, simplemente dicho: a veces lo simple y bruto es la mejor opción. Las soluciones complejas no son necesariamente mejores, las simples no son necesariamente peores.

**Decisión tres: almacenamiento de información de autenticación**

Las credenciales se almacenan en archivos de configuración del backend (appsettings.yml o variables de entorno) y se cargan mediante inyección de dependencias:

- Método de configuración simple, coincide con el modo de configuración del backend existente
- La información sensible no se expone al frontend
- Soporta configuración de múltiples entornos (desarrollo, pruebas, producción)

La sensación de seguridad es algo que debe haber. Después de todo, nadie quiere que sus credenciales estén por todas partes, ¿verdad?

### Diseño del flujo de datos

El flujo de datos general es así:

```
Frontend (navegador)
  │
  │ ws://backend/api/voice/ws
  │ WebSocket (binario)
  ▼
Backend (proxy)
  │
  │ wss://openspeech.bytedance.com/
  │ (con header de autenticación)
  ▼
API de Doubao
```

El proceso no es muy complicado, son solo estos pocos pasos:
1. El frontend se conecta al proxy backend a través de WebSocket
2. El proxy backend recibe los datos de audio y se conecta a la API de Doubao con headers
3. La API de Doubao devuelve el resultado de reconocimiento, el proxy lo reenvía al frontend
4. Transmisión de flujo bidireccional asincrónica durante todo el proceso

Todo parece tan natural, ¿no es así?

## Implementación de componentes principales

### 1. Configuración del endpoint WebSocket

```csharp
app.Map("/ws", async context =>
{
    if (context.WebSockets.IsWebSocketRequest)
    {
        // Leer configuración desde parámetros de consulta
        var appId = context.Request.Query["appId"];
        var accessToken = context.Request.Query["accessToken"];

        // Validar parámetros requeridos
        if (string.IsNullOrEmpty(appId) || string.IsNullOrEmpty(accessToken))
        {
            context.Response.StatusCode = 400;
            return;
        }

        // Aceptar conexión WebSocket
        using var webSocket = await context.WebSockets.AcceptWebSocketAsync();

        // Bucle de procesamiento de mensajes
        var buffer = new byte[4096];
        while (!webSocket.CloseStatus.HasValue)
        {
            var result = await webSocket.ReceiveAsync(buffer, CancellationToken.None);

            if (result.MessageType == WebSocketMessageType.Close)
            {
                await webSocket.CloseAsync(
                    result.CloseStatus.Value,
                    result.CloseStatusDescription,
                    CancellationToken.None);
                break;
            }

            // Procesar datos de audio
            await HandleAudioDataAsync(buffer, result.Count);
        }
    }
});
```

### 2. Gestión de sesiones

```csharp
public class DoubaoSessionManager : IDoubaoSessionManager
{
    private readonly ConcurrentDictionary<string, DoubaoSession> _sessions = new();

    public DoubaoSession CreateSession(string connectionId)
    {
        var session = new DoubaoSession(connectionId);
        _sessions[connectionId] = session;
        return session;
    }

    public async Task SendAudioAsync(string connectionId, byte[] audioData)
    {
        if (_sessions.TryGetValue(connectionId, out var session))
        {
            await session.SendAudioAsync(audioData);
        }
    }

    public void RemoveSession(string connectionId)
    {
        if (_sessions.TryRemove(connectionId, out var session))
        {
            session.Dispose();
        }
    }
}
```

Gestionar sesiones con ConcurrentDictionary, no hay que preocuparse por la seguridad de hilos. Cada conexión que entra crea una Session, y se limpia automáticamente al desconectarse: esto es más o menos lo que se llama "venir y pasar rápido".

## Diseño del protocolo de mensajes

El frontend y el backend usan mensajes de texto en formato JSON para control, y mensajes binarios para transmitir datos de audio.

Ejemplo de mensaje de control:

```json
{
    "type": "control",
    "messageId": "msg_123",
    "timestamp": "2026-03-03T10:00:00Z",
    "payload": {
        "command": "StartRecognition",
        "parameters": {
            "hotwordId": "hotword1",
            "boosting_table_id": "table123"
        }
    }
}
```

Ejemplo de resultado de reconocimiento:

```json
{
    "type": "result",
    "timestamp": "2026-03-03T10:00:03Z",
    "payload": {
        "text": "你好世界",
        "confidence": 0.95,
        "duration": 1500,
        "isFinal": true,
        "utterances": [
            {
                "text": "你好",
                "startTime": 0,
                "endTime": 800,
                "definite": true
            }
        ]

```

Este diseño separa las señales de control de los datos de audio, lo que hace que el procesamiento sea más claro. A veces dividir y conquistar es realmente un buen enfoque.

## Práctica de integración frontend

### Conexión WebSocket

```javascript
class DoubaoVoiceClient {
    constructor(config) {
        this.config = config;
        this.ws = null;
    }

    async connect() {
        const url = new URL(this.config.wsUrl);
        // Añadir parámetros de consulta
        Object.entries(this.config.params).forEach(([key, value]) => {
            url.searchParams.set(key, value);
        });

        this.ws = new WebSocket(url);

        return new Promise((resolve, reject) => {
            this.ws.onopen = () => {
                console.log('[DoubaoVoice] Connected');
                resolve();
            };

            this.ws.onmessage = (event) => {
                this._handleMessage(JSON.parse(event.data));
            };

            this.ws.onerror = reject;
        });
    }

    _handleMessage(message) {
        switch (message.type) {
            case 'status':
                this._handleStatus(message.payload);
                break;
            case 'result':
                this.onResult?.(message.payload);
                break;
            case 'error':
                console.error('[DoubaoVoice] Error:', message.payload);
                break;
        }
    }
}

// Ejemplo de uso
const client = new DoubaoVoiceClient({
    wsUrl: 'ws://localhost:5000/ws',
    params: {
        appId: 'your-app-id',
        accessToken: 'your-access-token',
        sampleRate: 16000,
        bitsPerSample: 16,
        channels: 1
    }
});
```

### Captura y envío de audio

Usar AudioWorkletNode para procesamiento de audio ofrece mejor rendimiento:

```javascript
// audio-worklet.js
class AudioProcessorWorklet extends AudioWorkletProcessor {
    process(inputs, outputs, parameters) {
        const input = inputs[0]?.[0];
        if (!input) return true;

        // Convertir a PCM de 16 bits
        const pcm = new Int16Array(input.length);
        for (let i = 0; i < input.length; i++) {
            pcm[i] = Math.max(-32768, Math.min(32767, input[i] * 32767));
        }

        this.port.postMessage({
            type: 'audioData',
            data: pcm.buffer
        }, [pcm.buffer]);

        return true;
    }
}

registerProcessor('audio-processor', AudioProcessorWorklet);

// Código del hilo principal
async function startAudioRecording() {
    const stream = await navigator.mediaDevices.getUserMedia({
        audio: {
            echoCancellation: true,
            noiseSuppression: true,
            autoGainControl: true,
            sampleRate: 48000
        }
    });

    const audioContext = new AudioContext();
    const audioSource = audioContext.createMediaStreamSource(stream);

    await audioContext.audioWorklet.addModule('/audio-worklet.js');
    const audioWorkletNode = new AudioWorkletNode(audioContext, 'audio-processor');

    audioWorkletNode.port.onmessage = (event) => {
        if (event.data.type === 'audioData' && ws?.readyState === WebSocket.OPEN) {
            ws.send(event.data.data); // Enviar datos binarios directamente
        }
    };

    audioSource.connect(audioWorkletNode);
}
```

AudioWorklet tiene mucho mejor rendimiento que ScriptProcessorNode, no hay problemas de tartamudeo de audio. En estos días, ¿quién quiere escuchar ese tipo de ruido chirriante?

## Consideraciones y mejores prácticas

### 1. Monitoreo de conexiones

- Generar registros de estado de sesión periódicamente para rastrear el ciclo de vida de las conexiones
- Monitorear la cantidad y duración de los segmentos de audio para identificar conexiones anómalas
- Registrar el estado de conexión con el servicio Doubao y las situaciones de reconexión

Estas son solo algunas operaciones básicas.

### 2. Manejo de errores

- Capturar y registrar todas las excepciones de WebSocket
- Usar IAsyncDisposable para asegurar la limpieza de recursos
- Implementar cierre elegante de conexiones y manejo de tiempos de espera

En resumen, la estabilidad es lo primero.

### 3. Requisitos de formato de audio

- Tasa de muestreo: 16000 Hz (recomendado) o 8000 Hz
- Profundidad de bits: 16-bit
- Canales: mono
- Codificación: PCM (raw)

Un formato incorrecto causará fallos en el reconocimiento o resultados muy pobres. Hay que respetar estas reglas.

### 4. Consideraciones de seguridad

- Las credenciales sensibles solo existen en la configuración del backend
- Implementar límites de conexiones para evitar agotamiento de recursos
- Usar HTTPS/WSS en entorno de producción

La seguridad es algo importante, procedamos con cuidado.

### 5. Optimización del rendimiento

- Usar operaciones asincrónicas para evitar bloqueos
- Ajustar adecuadamente el tamaño del búfer (por defecto 4096 bytes)
- Considerar estrategias de agrupación y reutilización de conexiones

Estas técnicas de optimización, úsalas si puedes.

## Sugerencias de despliegue

1. **Despliegue Docker**: empaquetar el servicio proxy en un contenedor para facilitar escalado y gestión
2. **Equilibrio de carga**: usar Nginx o Envoy como proxy inverso WebSocket
3. **Verificación de salud**: implementar mecanismo de latido para monitorear disponibilidad del servicio
4. **Agregación de registros**: enviar registros a un sistema centralizado (como ELK, Loki)

El despliegue, se puede decir que es simple y también complejo. Depende de la persona y la situación.

## Resumen

La solución de proxy WebSocket resuelve el problema fundamental de que la API WebSocket del navegador no soporta headers personalizados. En el proyecto HagiCode, esta solución se probó desde el playground hasta el despliegue en producción, demostrando su viabilidad y estabilidad.

Resumen de puntos clave:
- El proxy backend puede transmitir información de autenticación de manera segura
- WebSocket nativo es ligero y eficiente, adecuado para escenarios simples
- "Una sesión por conexión" simplifica la implementación y depuración
- El protocolo de mensajes frontend-backend separa señales de control y datos de audio

Si también estás desarrollando funciones que requieren autenticación WebSocket, espero que esta solución te inspire.

Si tienes alguna pregunta, bienvenido a discutir. Después de todo, la tecnología progresa a través del intercambio.
