## Hagicode.Libs: Práctica de ingeniería para la integración unificada de múltiples CLIs de asistentes de programación con IA

> Durante el desarrollo del proyecto HagiCode, necesitamos integrar simultáneamente los CLIs de múltiples asistentes de programación con IA como Claude Code, Codex, CodeBuddy, etc. Cada CLI tiene diferentes interfaces, parámetros y formatos de salida, y el código de integración repetido hacía que el proyecto fuera cada vez más difícil de mantener. Este artículo comparte cómo construimos una capa de abstracción unificada a través de la biblioteca HagiCode.Libs para resolver este problema de ingeniería, también podríamos decir que es la experiencia acumulada de los obstáculos que enfrentamos.

## Antecedentes

El mercado actual de asistentes de programación con IA está bastante animado. Además de Claude Code, está Codex de OpenAI, CodeBuddy de Zhipu, entre otros. Como proyecto de asistente de código con IA, HagiCode necesita integrar estas diferentes herramientas de CLI en múltiples subproyectos (escritorio, backend, web, etc.).

Al principio no era un problema grave, integrar un CLI era cuestión de unas pocas cientos de líneas de código. Pero a medida que aumentaba la cantidad de CLIs a soportar, las cosas empezaron a complicarse.

Cada CLI tiene diferentes formatos de parámetros de línea de comandos, diferentes requisitos de variables de entorno, y los formatos de salida son aún más variados: algunos producen JSON, otros JSON en streaming, otros simplemente texto plano. Además de los problemas de compatibilidad multiplataforma, los mecanismos de descubrimiento de ejecutables y gestión de procesos en Windows y sistemas Unix son completamente diferentes, lo que aumenta la redundancia del código. Al final, no es más que un poco de Ctrl+C, Ctrl+V, pero el mantenimiento se vuelve un dolor de cabeza.

Lo peor es que cada vez que había que agregar soporte para una nueva función de CLI, había que modificar el mismo código en varios proyectos. Este enfoque claramente no es sostenible a largo plazo: el código también tiene carácter, y si hay demasiada redundancia, también protesta.

## Sobre HagiCode

La solución que compartimos en este artículo proviene de nuestra experiencia práctica en el proyecto [HagiCode](https://hagicode.com). HagiCode es un proyecto de asistente de código con IA de código abierto que necesita mantener múltiples subproyectos: extensión de VSCode, servicio de IA de backend, cliente de escritorio multiplataforma, entre otros. Se podría decir que es precisamente este escenario complejo de múltiples lenguajes y plataformas lo que impulsó el nacimiento de HagiCode.Libs: fue algo forzado, pero así fue.

## Análisis: buscando puntos en común

Aunque estos CLIs de asistentes de programación con IA tienen sus propias características, desde una perspectiva técnica, tienen características comunes obvias:

**Patrones de interacción similares**: todos implican iniciar un proceso CLI, enviar prompts, recibir respuestas en streaming, analizar mensajes y finalmente terminar o continuar la sesión. Todo este flujo, en el fondo, está cortado por el mismo patrón.

**Necesidades de configuración similares**: todos necesitan autenticación con claves API, configuración de directorio de trabajo, selección de modelos, control de permisos de herramientas y gestión de sesiones. Al final, todos vivimos del API, la diferencia es solo el sabor.

**Desafíos multiplataforma consistentes**: todos necesitan resolver problemas como resolución de rutas de ejecutables (`claude` vs `claude.exe` vs `/usr/local/bin/claude`), inicio de procesos y manejo de variables de entorno, escape de comandos de shell y construcción de parámetros. Este tema multiplataforma, si hablamos mucho, son lágrimas: solo quienes han enfrentado estos obstáculos conocen las diferencias entre Windows y Unix.

Basándonos en estos análisis, necesitamos una capa de abstracción unificada que proporcione una interfaz consistente, encapsule la lógica de descubrimiento de CLI multiplataforma, maneje el análisis de salida en streaming, y al mismo tiempo admita tanto el uso con inyección de dependencias como sin DI. Pensarlo ya da dolor de cabeza, pero hay que enfrentarlo: después de todo, es nuestro propio proyecto, aunque tengamos que terminarlo llorando.

## Solución: HagiCode.Libs

Creamos **HagiCode.Libs**: un espacio de trabajo de bibliotecas .NET 10 ligero con licencia de código abierto MIT, ahora publicado en [GitHub](https://github.com/HagiCode-org/Hagicode.Libs). Aunque no es una obra maestra que conmueva al cielo y a la tierra, es bastante agradable para resolver problemas reales.

### Estructura del proyecto

```
HagiCode.Libs/
├── src/
│   ├── HagiCode.Libs.Core/           # Funcionalidad principal
│   │   ├── Discovery/                 # Descubrimiento de ejecutables CLI
│   │   ├── Process/                   # Gestión de procesos multiplataforma
│   │   ├── Transport/                 # Transporte de mensajes en streaming
│   │   └── Environment/               # Resolución del entorno de ejecución
│   ├── HagiCode.Libs.Providers/       # Implementaciones de proveedores
│   │   ├── ClaudeCode/                # Proveedor de Claude Code
│   │   ├── Codex/                     # Proveedor de Codex
│   │   └── Codebuddy/                 # Proveedor de CodeBuddy
│   ├── HagiCode.Libs.ConsoleTesting/  # Framework de pruebas
│   ├── HagiCode.Libs.ClaudeCode.Console/
│   ├── HagiCode.Libs.Codex.Console/
│   └── HagiCode.Libs.Codebuddy.Console/
└── tests/                             # Pruebas xUnit
```

### Objetivos de diseño

Al diseñar HagiCode.Libs, seguimos varios principios: después de todo, son la experiencia resumida de los obstáculos que enfrentamos:

**Cero dependencias de frameworks pesados**: no dependemos de ABP u otros frameworks grandes, manteniendo la biblioteca ligera. En estos días, menos dependencias significan menos problemas: ¿quién no ha sufrido del infierno de las dependencias?

**Soporte multiplataforma**: soporte nativo para Windows, macOS y Linux, sin necesidad de escribir código diferente para cada plataforma. Un código para todos, también está bien.

**Procesamiento en streaming**: utiliza flujos asíncronos para procesar la salida de CLI, más acorde con el patrón de programación moderno de .NET. Al fin y al cabo, los tiempos cambian, lo asíncrono es el camino.

**Integración flexible**: tanto soporta escenarios con inyección de dependencias como permite la instanciación directa. Para cada gusto, lo que sea más conveniente.

## Formas de uso

### A través de inyección de dependencias

Si tu proyecto ya está usando inyección de dependencias (como ASP.NET Core o un host genérico), puedes integrar directamente: también es como un hijo, aunque no muy grande, pero bastante obediente:

```csharp
using HagiCode.Libs.Providers;
using Microsoft.Extensions.DependencyInjection;

var services = new ServiceCollection();
services.AddHagiCodeLibs();

await using var provider = services.BuildServiceProvider();
var claude = provider.GetRequiredService<ICliProvider<ClaudeCodeOptions>>();

var options = new ClaudeCodeOptions
{
    ApiKey = "your-api-key",
    Model = "claude-sonnet-4-20250514"
};

await foreach (var message in claude.ExecuteAsync(options, "Hello, Claude!"))
{
    Console.WriteLine($"{message.Type}: {message.Content}");
}
```

### Instanciación directa

Si es un script simple o un escenario que no usa DI, también puedes crear una instancia directamente: en el fondo, es una cuestión de preferencia personal:

```csharp
var claude = new ClaudeCodeProvider();
var options = new ClaudeCodeOptions
{
    ApiKey = "sk-ant-xxx",
    Model = "claude-sonnet-4-20250514"
};

await foreach (var message in claude.ExecuteAsync(options, "帮我写一个快速排序"))
{
    // Procesar mensaje
}
```

Ambos métodos usan la misma implementación subyacente, puedes elegir la forma de integración apropiada según la situación real del proyecto. En este mundo no hay una respuesta estándar, lo que te conviene es lo mejor: aunque esta frase sea un cliché, realmente tiene su lógica.

## Compartiendo experiencia práctica

### 1. Consola de prueba dedicada

Cada proveedor tiene un proyecto de consola de prueba dedicado, lo que facilita la verificación independiente del efecto de integración: ¿cómo decirlo? Para las pruebas, o no se hacen o se hacen bien:

```bash
# Prueba de Claude Code
dotnet run --project src/HagiCode.Libs.ClaudeCode.Console -- --test-provider
dotnet run --project src/HagiCode.Libs.ClaudeCode.Console -- --test-all claude

# Prueba de CodeBuddy
dotnet run --project src/HagiCode.Libs.Codebuddy.Console -- --test-provider codebuddy-cli

# Prueba de Codex
dotnet run --project src/HagiCode.Libs.Codex.Console -- --test-provider codex-cli
```

Los escenarios de prueba cubren varios escenarios clave:
- **Ping**: verificación de salud, confirma que el CLI está disponible
- **Simple Prompt**: prueba básica de prompts
- **Complex Prompt**: prueba de conversaciones de múltiples rondas
- **Session Restore/Resume**: prueba de recuperación de sesión
- **Repository Analysis**: prueba de análisis de repositorios de código

Este diseño de consola de prueba independiente es especialmente útil durante la depuración, permitiendo localizar rápidamente si el problema está en la capa HagiCode.Libs o en el CLI mismo. La depuración, en el fondo, es ver dónde está el problema: con la dirección correcta, ya estás a mitad de camino.

### 2. Verificación CI/CD multiplataforma

La compatibilidad multiplataforma es uno de los objetivos principales de HagiCode.Libs. Configuramos un flujo de trabajo de GitHub Actions `.github/workflows/cli-discovery-cross-platform.yml` que ejecuta verificaciones reales de descubrimiento de CLI en tres plataformas: `ubuntu-latest`, `macos-latest` y `windows-latest`.

Esto asegura que cada cambio de código no rompa la compatibilidad multiplataforma. Durante el desarrollo local también puedes reproducir esto con los siguientes comandos: no puedes dejar que CI cargue con toda la culpa, tú también debes poder ejecutarlo localmente:

```bash
npm install --global @anthropic-ai/claude-code@2.1.79
HAGICODE_REAL_CLI_TESTS=1 dotnet test --filter "Category=RealCli"
```

### 3. Procesamiento de flujos de mensajes

HagiCode.Libs usa flujos asíncronos para procesar la salida de CLI, este enfoque es más acorde con el estilo de programación asíncrona moderno de .NET que los patrones tradicionales de callbacks o eventos: al fin y al cabo, este es el progreso tecnológico, nadie lo puede detener:

```csharp
public async IAsyncEnumerable<CliMessage> ExecuteAsync(
    TOptions options,
    string prompt,
    [EnumeratorCancellation] CancellationToken cancellationToken = default)
{
    // Iniciar proceso CLI
    // Analizar salida JSON en streaming
    // Generar secuencia CliMessage
}
```

Los tipos de mensajes incluyen:
- `user`: mensajes del usuario
- `assistant`: respuestas del asistente
- `tool_use`: llamadas a herramientas
- `result`: fin de sesión

Este diseño permite a quien llama manejar la salida en streaming de manera flexible, como visualización en tiempo real, procesamiento después del buffer, o reenvío a otros servicios. ¿Por qué preocuparse por el clima cuando tienes belleza? Lo importante es abrir la mente, cualquier forma de usar está bien.

### 4. Exploración de repositorios Git

El módulo HagiCode.Libs.Exploration proporciona funcionalidad de descubrimiento y verificación de estado de repositorios Git, lo cual es especialmente útil en escenarios de análisis de repositorios de código: esta también fue una funcionalidad forzada, ¿quién va a hacer que HagiCode necesite analizar repositorios de código?

```csharp
// Descubrir repositorios Git
var repositories = await GitRepositoryDiscovery.DiscoverAsync("/path/to/search");

// Obtener información del repositorio
var info = await GitRepository.GetInfoAsync(repoPath);
Console.WriteLine($"Branch: {info.Branch}, Remote: {info.RemoteUrl}");
Console.WriteLine($"Has uncommitted changes: {info.HasUncommittedChanges}");
```

La funcionalidad de análisis de código de HagiCode usa este módulo para identificar la estructura del proyecto y el estado de Git. Se podría decir que estamos aprovechando al máximo los recursos, también como criar un hijo, no fue en vano.

## Consideraciones importantes

Basándonos en nuestra práctica en el proyecto HagiCode, hay varios aspectos que necesitan atención especial: todo son problemas, problemas, problemas:

**Seguridad de claves API**: no codifiques las claves API en el código, usa variables de entorno o gestión de configuración. HagiCode.Libs soporta pasar configuración a través de objetos Options, facilitando la integración con varias fuentes de configuración. Después de todo, en cuanto a seguridad, no se puede ser demasiado cuidadoso.

**Bloqueo de versión de CLI**: en CI/CD bloqueamos versiones específicas (como `@anthropic-ai/claude-code@2.1.79`) para reducir la incertidumbre causada por el desvío de versiones. Durante el desarrollo local también se recomienda usar una versión fija. El tema de versiones, si hablas mucho, son lágrimas: sin una versión fija, en cuestión de minutos te enseña una lección.

**Clasificación de pruebas**: por defecto, las pruebas usan proveedores falsos para mantener determinismo y velocidad, las pruebas reales de CLI necesitan activarse explícitamente. Esto garantiza retroalimentación rápida de CI y, cuando es necesario, permite verificación en entorno real. Caminar por esta cuerda floja de equilibrio tampoco es fácil: rápido y estable siempre requieren compromisos.

**Gestión de sesiones**: diferentes CLIs tienen diferentes mecanismos de recuperación de sesión, Claude Code usa el directorio `.claude/` para almacenar sesiones, Codex y CodeBuddy tienen sus propias formas. Al usarlos, presta atención a sus respectivos documentos para entender el mecanismo específico de persistencia de sesiones. Esto también está bien, entender claramente nunca hace daño.
