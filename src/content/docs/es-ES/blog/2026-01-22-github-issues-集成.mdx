## Construyendo la integración con GitHub Issues desde cero: La práctica de conexión directa de HagiCode

> Este artículo registra el proceso completo de integración de GitHub Issues en la plataforma HagiCode. Exploraremos cómo implementar una autenticación OAuth segura y una sincronización eficiente de Issues manteniendo el backend ligero, mediante una arquitectura de "conexión directa desde el frontend + backend minimizado".

## Contexto: ¿Por qué integrar GitHub?

HagiCode, como una plataforma de desarrollo asistida por IA, tiene como valor central conectar ideas con implementaciones. Sin embargo, en el uso práctico, descubrimos que después de que los usuarios completaban un Proposal (propuesta) en HagiCode, a menudo necesitaban copiar manualmente el contenido a GitHub Issues para el seguimiento del proyecto.

Esto trae varios puntos de dolor evidentes:

1.  **Flujo de trabajo fragmentado**: Los usuarios necesitan alternar entre dos sistemas, lo que no solo resulta en una experiencia poco fluida, sino que también facilita la pérdida de información clave durante el proceso de copiar y pegar.
2.  **Colaboración inconveniente**: Otros miembros del equipo están acostumbrados a ver tareas en GitHub y no pueden ver directamente el progreso de las propuestas en HagiCode.
3.  **Trabajo repetitivo**: Cada vez que se actualiza una propuesta, hay que actualizar manualmente el Issue correspondiente en GitHub, aumentando costos de mantenimiento innecesarios.

Para resolver este problema, decidimos introducir la función de **GitHub Issues Integration**, conectando las sesiones de HagiCode con los repositorios de GitHub para实现 "sincronización con un clic".

## Selección técnica: Conexión directa desde el frontend vs Proxy backend

Al diseñar el plan de integración, teníamos dos caminos frente a nosotros: el tradicional "modo proxy backend" y el más agresivo "modo de conexión directa desde el frontend".

### Comparación de soluciones

En el tradicional **modo proxy backend**, todas las solicitudes del frontend deben pasar primero por nuestro backend, y luego el backend llama a la GitHub API. Aunque la lógica está centralizada, esto impone una carga significativa en el backend:

1.  **Backend inflado**: Es necesario escribir un encapsulamiento especial del cliente de GitHub API, y también manejar la máquina de estados compleja de OAuth.
2.  **Riesgo de Token**: El GitHub Token del usuario debe almacenarse en la base de datos del backend. Aunque puede cifrarse, de todos modos aumenta la superficie de riesgo de seguridad.
3.  **Costo de desarrollo**: Se requieren migraciones de base de datos para almacenar el Token, y también mantener un servicio de sincronización adicional.

El **modo de conexión directa desde el frontend** es mucho más ligero. En esta solución, solo usamos el backend para manejar el paso más sensible de "intercambio de claves" (OAuth callback). Después de obtener el Token, se almacena directamente en el localStorage del navegador. Operaciones posteriores como crear Issues y actualizar comentarios se realizan directamente desde el frontend mediante solicitudes HTTP a GitHub.

| Dimensión de comparación | Modo proxy backend | Modo de conexión directa desde el frontend |
| :--- | :--- | :--- |
| **Complejidad del backend** | Requiere servicio OAuth completo y cliente GitHub API | Solo requiere un endpoint de callback OAuth |
| **Gestión de Token** | Requiere almacenamiento cifrado en base de datos, riesgo de filtración | Almacenado en el navegador, solo visible para el usuario |
| **Costo de implementación** | Requiere migración de base de datos, desarrollo de múltiples servicios | Principalmente trabajo del frontend |
| **Experiencia de usuario** | Lógica unificada, pero la latencia del servidor puede ser ligeramente mayor | Respuesta extremadamente rápida, interacción directa con GitHub |

Considerando que buscamos una integración rápida y cambios mínimos en el backend, **finalmente adoptamos el "modo de conexión directa desde el frontend"**. Es como emitir un "pasaporte temporal" al navegador; después de obtener el documento, el navegador puede ir directamente a GitHub para hacer sus cosas, sin necesidad de buscar la aprobación del administrador del backend cada vez.

## Diseño central: Flujo de datos y seguridad

Después de determinar la arquitectura, necesitamos diseñar el flujo de datos específico. El núcleo de todo el proceso de sincronización radica en cómo obtener el Token de manera segura y utilizarlo de manera eficiente.

### Diagrama de arquitectura general

Todo el sistema puede abstraerse en tres roles: navegador (frontend), backend HagiCode, GitHub.

```text
+--------------+        +--------------+        +--------------+
|  Frontend    |        |    Backend   |        |    GitHub    |
|   React      |        |   ASP.NET    |        |    REST API  |
|  +--------+  |        |              |        |              |
|  |  OAuth |--+--------> /callback    |        |              |
|  |  flujo  |  |        |              |        |              |
|  +--------+  |        |              |        |              |
|              |        |              |        |              |
|  +--------+  |        |  +--------+  |        |  +--------+  |
|  |GitHub  |  +------------>Session |  +----------> Issues |  |
|  |API     |  |        |  |Metadata|  |        |  |        |  |
|  |directo |  |        |  +--------+  |        |  +--------+  |
|  +--------+  |        |              |        |              |
+--------------+        +--------------+        +--------------+
```

**El punto clave es**: solo un pequeño paso de OAuth (obtener code para cambiar por token) necesita pasar por el backend; el trabajo pesado posterior (crear Issue) lo hace el frontend directamente con GitHub.

### Explicación detallada del flujo de datos de sincronización

Cuando el usuario hace clic en el botón "Sync to GitHub" en la interfaz de HagiCode, ocurre una serie de acciones complejas:

```text
Usuario hace clic en "Sync to GitHub"
         │
         ▼
1. Frontend verifica localStorage para obtener GitHub Token
         │
         ▼
2. Formatea contenido del Issue (convierte Proposal a Markdown)
         │
         ▼
3. Frontend llama directamente a GitHub API para crear/actualizar Issue
         │
         ▼
4. Llama a HagiCode Backend API para actualizar Session.metadata (almacena Issue URL, etc.)
         │
         ▼
5. Backend transmite evento SessionUpdated a través de SignalR
         │
         ▼
6. Frontend recibe el evento, actualiza UI mostrando estado "sincronizado"
```

### Diseño de seguridad

Los problemas de seguridad son siempre lo más importante al integrar servicios de terceros. Hicimos las siguientes consideraciones:

1.  **Prevención de ataques CSRF**: Durante la redirección OAuth, se genera un parámetro `state` aleatorio y se almacena en sessionStorage. En el callback, se verifica estrictamente el state para evitar que las solicitudes sean falsificadas.
2.  **Aislamiento de almacenamiento de Token**: El Token solo se almacena en `localStorage` del navegador, aprovechando la política del mismo origen (Same-Origin Policy), solo los scripts de HagiCode pueden leerlo, evitando que las filtraciones de la base de datos del servidor afecten a los usuarios.
3.  **Límites de error**: Para errores comunes de GitHub API (como 401 Token expirado, 422 validación fallida, 429 límite de tasa), diseñamos una lógica de manejo de errores especializada para dar a los usuarios avisos amigables.

## Práctica: Detalles de implementación del código

Papel dicho es superficial, veamos cómo se implementa el código específico.

### 1. Cambios mínimos en el backend

El backend solo necesita hacer dos cosas: almacenar información de sincronización, manejar callback de OAuth.

**Cambios en la base de datos**
Solo necesitamos agregar una columna `Metadata` a la tabla `Sessions` para almacenar información extendida en formato JSON.

### 2. Flujo OAuth del frontend

Este es el punto de entrada de la conexión. Usamos el estándar Authorization Code Flow.

```typescript
// src/HagiCode.Client/src/services/githubOAuth.ts

// Generar URL de autorización y redirigir
export async function generateAuthUrl(): Promise<string> {
  const state = generateRandomString(); // Generar cadena aleatoria anti-CSRF
  sessionStorage.setItem('hagicode_github_state', state);
  
  const params = new URLSearchParams({
    client_id: clientId,
    redirect_uri: window.location.origin + '/settings?tab=github&oauth=callback',
    scope: ['repo', 'public_repo'].join(' '),
    state: state,
  });
  
  return `https://github.com/login/oauth/authorize?${params.toString()}`;
}

// Manejar intercambio de Code por Token en la página de callback
export async function exchangeCodeForToken(code: string, state: string): Promise<GitHubToken> {
  // 1. Verificar State para prevenir CSRF
  const savedState = sessionStorage.getItem('hagicode_github_state');
  if (state !== savedState) throw new Error('Invalid state parameter');

  // 2. Llamar a API del backend para intercambio de Token
  // Nota: esto debe pasar por el backend porque necesita ClientSecret, no puede exponerse en el frontend
  const response = await fetch('/api/GitHubOAuth/callback', {
    method: 'POST',
    headers: { 'Content-Type': 'application/json' },
    body: JSON.stringify({ code, state, redirectUri: window.location.origin + '/settings?tab=github&oauth=callback' }),
  });

  if (!response.ok) throw new Error('Failed to exchange token');
  
  const token = await response.json();
  
  // 3. Guardar en LocalStorage
  saveToken(token);
  return token;
}
```

### 3. Encapsulación del cliente GitHub API

Después de tener el Token, necesitamos una herramienta potente para llamar a GitHub API.

```typescript
// src/HagiCode.Client/src/services/githubApiClient.ts

const GITHUB_API_BASE = 'https://api.github.com';

// Encapsulación de solicitud central
async function githubApi<T>(endpoint: string, options: RequestInit = {}): Promise<T> {
  const token = localStorage.getItem('gh_token');
  if (!token) throw new Error('Not connected to GitHub');
  
  const response = await fetch(`${GITHUB_API_BASE}${endpoint}`, {
    ...options,
    headers: {
      ...options.headers,
      Authorization: `Bearer ${token}`,
      Accept: 'application/vnd.github.v3+json', // Especificar versión de API
    },
  });
  
  // Lógica de manejo de errores
  if (!response.ok) {
    if (response.status === 401) throw new Error('GitHub Token inválido, por favor vuelve a conectar');
    if (response.status === 403) throw new Error('Sin permiso para acceder al repositorio o excedió el límite de tasa');
    if (response.status === 422) throw new Error('Validación de Issue fallida, posible título duplicado');
    throw new Error(`GitHub API Error: ${response.statusText}`);
  }
  
  return response.json();
}

// Crear Issue
export async function createIssue(owner: string, repo: string, data: { title: string, body: string, labels: string[] }) {
  return githubApi(`/repos/${owner}/${repo}/issues`, {
    method: 'POST',
    body: JSON.stringify(data),
  });
}
```

### 4. Formateo de contenido y sincronización

El último paso es convertir los datos de la Session de HagiCode al formato de GitHub Issue. Es un poco como un trabajo de "traducción".

```typescript
// Convertir objeto Session a cadena Markdown
function formatIssueForSession(session: Session): string {
  let content = `# ${session.title}\n\n`;
  content += `**> HagiCode Session:** #${session.code}\n`;
  content += `**> Status:** ${session.status}\n\n`;
  content += `## Description\n\n${session.description || 'No description provided.'}\n\n`;
  
  // Si es tipo Proposal, agregar campos adicionales
  if (session.type === 'proposal') {
    content += `## Chief Complaint\n\n${session.chiefComplaint || ''}\n\n`;
    // Agregar un deep link, conveniente para saltar de GitHub a HagiCode
    content += `---\n\n**[View in HagiCode](hagicode://sessions/${session.id})**\n`;
  }
  
  return content;
}

// Lógica principal del botón de sincronización
const handleSync = async (session: Session) => {
  try {
    const repoInfo = parseRepositoryFromUrl(session.repoUrl); // Analizar URL del repositorio
    if (!repoInfo) throw new Error('Invalid repository URL');

    toast.loading('Sincronizando con GitHub...');
    
    // 1. Formatear contenido
    const issueBody = formatIssueForSession(session);
    
    // 2. Llamar a API
    const issue = await githubApiClient.createIssue(repoInfo.owner, repoInfo.repo, {
      title: `[HagiCode] ${session.title}`,
      body: issueBody,
      labels: ['hagicode', 'proposal', `status:${session.status}`],
    });
    
    // 3. Actualizar Session Metadata (guardar enlace del Issue)
    await SessionsService.patchApiSessionsSessionId(session.id, {
      metadata: {
        githubIssue: {
          owner: repoInfo.owner,
          repo: repoInfo.repo,
          issueNumber: issue.number,
          issueUrl: issue.html_url,
          syncedAt: new Date().toISOString(),
        }
      }
    });

    toast.success('¡Sincronización exitosa!');
  } catch (err) {
    console.error(err);
    toast.error('Sincronización fallida, por favor verifica Token o red');
  }
};
```

## Resumen y perspectivas

A través de esta solución de "conexión directa desde el frontend", implementamos una integración perfecta de GitHub Issues con la menor cantidad de código backend.

### Logros
1.  **Alta eficiencia de desarrollo**: Cambios mínimos en el backend, principalmente agregar un campo en la base de datos y una interfaz simple de callback OAuth, la mayor parte de la lógica se completa en el frontend.
2.  **Buena seguridad**: El Token no pasa por la base de datos del servidor, reduciendo el riesgo de filtración.
3.  **Excelente experiencia de usuario**: Las solicitudes se inician directamente desde el frontend, respuesta rápida, sin necesidad de pasar por el backend como intermediario.

### Consideraciones
En el deployment real, hay varios pozos a tener en cuenta:
- **Configuración de OAuth App**: Recuerda llenar la `Authorization callback URL` correcta en la configuración de GitHub OAuth App (generalmente `http://localhost:3000/settings?tab=github&oauth=callback`).
- **Límite de tasa**: GitHub API limita estrictamente las solicitudes no autenticadas, pero con Token suele ser suficiente (5000 veces/hora).
- **Análisis de URL**: Las URLs de Repo que los usuarios ingresan son variadas, recuerda que la expresión regular debe coincidir con sufijos `.git`, formatos SSH, etc.

### Mejoras futuras
Actualmente la función es sincronización unidireccional (HagiCode -> GitHub). En el futuro planeamos implementar sincronización bidireccional a través de GitHub Webhooks, por ejemplo, al cerrar un Issue en GitHub, el estado de la sesión en HagiCode también puede actualizarse automáticamente. Esto requiere que expongamos un endpoint de recepción de Webhook en el backend, que también es el próximo trabajo interesante a realizar.

Espero que este artículo te traiga un poco de inspiración para el desarrollo de integraciones de terceros. Si tienes preguntas, bienvenido a abrir un Issue en [HagiCode GitHub](https://github.com/HagiCode-org/site) para discutir.
