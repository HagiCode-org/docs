## Protección de código .NET en la práctica: desde ofuscación hasta protección por máquina virtual

> Este artículo presenta cómo implementar estrategias de protección de código de múltiples niveles en proyectos .NET, cubriendo desde la ofuscación básica hasta soluciones profesionales de protección por máquina virtual.

## Antecedentes

En el desarrollo de aplicaciones .NET, proteger el código central (como verificación de licencias, lógica de negocio, configuración sensible, etc.) contra la descompilación y el análisis inverso, bueno, es un tema inevitable. Con la maduración del ecosistema .NET, los desarrolladores tienen múltiples medios de protección de código, desde atributos de ofuscación integrados hasta herramientas profesionales de protección por virtualización, hay bastantes opciones.

Como un proyecto monorepo complejo y multilingüe, HagiCode incluye aplicaciones de escritorio, sistemas de construcción y funciones de gestión de licencias. El código implica inevitablemente lógica de verificación de licencias, configuración sensible (como claves de API, ID de producto) y lógica central de negocio, estas cosas deben estar bien protegidas, después de todo, nadie quiere que su trabajo sea visto fácilmente por otros.

Este artículo compartirá las soluciones de protección de código que realmente adoptamos en el proyecto HagiCode, resumiendo el proceso completo desde aprender por experiencia hasta optimizar, quizás pueda darte alguna inspiración.

## Acerca de HagiCode

[HagiCode](https://hagicode.com) es un proyecto de asistente de código AI de código abierto, comprometido a proporcionar a los desarrolladores una experiencia de asistencia de programación inteligente. El proyecto adopta una arquitectura monorepo, manteniendo simultáneamente múltiples componentes como extensiones de VSCode, servicios AI de backend, clientes de escritorio multiplataforma, etc. Esta complejidad multilingüe y multiplataforma hace que la protección del código sea un desafío de ingeniería inevitable, no hay más remedio, ¿quién hizo que el proyecto fuera tan complejo?

Las soluciones compartidas en este artículo son exactamente las que hemos aprendido por experiencia y optimizado durante el desarrollo de HagiCode. Si quieres saber cómo resolvimos estos problemas técnicos, continúa leyendo, quizás tengas algunas sorpresas.

## Contenido principal

### 1. Atributo de ofuscación integrado de Microsoft

.NET Framework proporciona un `[ObfuscationAttribute]` integrado, que es la marca de ofuscación de código más básica y comúnmente utilizada. Este atributo se encuentra en el espacio de nombres `System.Reflection` y puede realizar protección básica del código sin introducir herramientas de terceros, lo cual es bastante conveniente.

**Características principales**:
- Propiedad `Feature`: especifica las características de ofuscación, como `"ultra"` (alta ofuscación), `"all"` (toda la ofuscación)
- Propiedad `Exclude`: `true` indica excluir ofuscación, `false` indica aplicar ofuscación
- Se puede aplicar a clases, métodos, propiedades y otros miembros de tipo

En el proyecto HagiCode, puedes ver ejemplos de uso real:

```csharp
[Obfuscation(Feature = "ultra", Exclude = false)]
public async Task<LicenseValidationResult?> ValidateLicenseAsync(...)
```

Las **ventajas** de este método son bastante obvias:
- No requiere dependencias adicionales, viene integrado en .NET Framework, ahorra muchos problemas
- Puede ser reconocido y procesado por herramientas de ofuscación de terceros
- No aumenta significativamente el tamaño del ensamblado compilado

Sin embargo, también tiene limitaciones: solo sirve como marca, el efecto de ofuscación real depende de la implementación de la herramienta, no puede proporcionar el nivel de seguridad de protección por máquina virtual, bueno, después de todo, no fue creado para eso.

### 2. VMP (Virtual Machine Protection)

VMP es una herramienta profesional de protección de código que proporciona un alto nivel de protección compilando el código en instrucciones de máquina virtual. A diferencia de la simple ofuscación de nombres, VMP convierte realmente la lógica del código en una forma que no puede ser restaurada por descompiladores convencionales, lo cual es bastante impresionante.

**Clasificación de niveles de protección**:

| Nivel | Virtualización | Mutación | Anti-depuración | Cifrado de cadenas | Escenarios aplicables |
|------|---------------|---------|----------------|-------------------|---------------------|
| HIGH | full | high | Habilitado | Habilitado | Verificación de licencia, concurrencia de sesiones, constantes sensibles |
| MEDIUM | partial | medium | Habilitado | Habilitado | Lógica de negocio, modelos de dominio |
| LOW | none | low | Deshabilitado | Deshabilitado | Clases de utilidad, código no crítico |

El proyecto HagiCode define un sistema de atributos declarativo para marcar el código que necesita protección:

```csharp
// Protección de alta prioridad
[VmProtect(VmProtectionPriority.High, Reason = "Contains license verification logic")]
public class KeygenClient { ... }

// Excluir protección
[VmExclude(Reason = "Public API that must remain unchanged")]
public class PublicApi { ... }

// Protección por herencia
[VmProtect(Priority.High, ProtectDerived = true)]
public class BaseLicenseValidator { ... }
```

### 3. Estrategia de protección en tiempo de compilación

La protección VMP no solo es efectiva en tiempo de ejecución, sino que también debe procesarse automáticamente en el flujo de compilación, después de todo, hacerlo manualmente es demasiado problemático. El sistema de compilación de HagiCode soporta múltiples modos:

- **Modo nativo de Windows**: llama directamente a la herramienta VMProtect
- **Modo contenedor Docker de Linux**: ejecuta VMP en el contenedor (resuelve problemas de compatibilidad multiplataforma)
- **Escaneo de Attribute**: descubre automáticamente las marcas de protección en el código
- **Mecanismo de verificación**: confirma que la protección se ha aplicado correctamente

La combinación de estas funciones es bastante conveniente.

## Soluciones

### 1. Uso del atributo de ofuscación integrado de Microsoft

Aplica directamente `ObfuscationAttribute` en el código:

```csharp
using System.Reflection;

[Obfuscation(Feature = "ultra", Exclude = false)]
public class LicenseService
{
    [Obfuscation(Feature = "ultra", Exclude = false)]
    public async Task<bool> ValidateLicenseAsync(string key)
    {
        // Lógica de verificación de licencia
    }

    [Obfuscation(Feature = "flow", Exclude = false)]
    private string DecryptToken(string encrypted)
    {
        // Lógica de descifrado
    }
}
```

A veces es necesario permitir que los ensamblados de prueba accedan a miembros internos mientras se mantiene la seguridad del código de producción:

```csharp
// AssemblyInfo.cs
[assembly: InternalsVisibleTo("HagiCode.Application.Tests")]
[assembly: InternalsVisibleTo("DynamicProxyGenAssembly2")] // for Moq
```

Así es mucho más conveniente para las pruebas, después de todo, el código todavía debe probarse.

### 2. Definición de Attribute personalizado para protección VMP

Crea atributos de protección personalizados para controlar el comportamiento de VMP:

```csharp
using System;

[AttributeUsage(AttributeTargets.Class | AttributeTargets.Method | AttributeTargets.Property)]
public class VmProtectAttribute : Attribute
{
    public VmProtectionPriority Priority { get; set; }
    public string? Reason { get; set; }
    public bool ProtectDerived { get; set; }
}

[AttributeUsage(AttributeTargets.Class | AttributeTargets.Method | AttributeTargets.Property)]
public class VmExcludeAttribute : Attribute
{
    public string? Reason { get; set; }
}

public enum VmProtectionPriority
{
    None = 0,
    Low = 1,
    Medium = 2,
    High = 3
}
```

Los atributos que defines tú mismo son más fáciles de usar, después de todo, conoces tus necesidades.

### 3. Archivo de configuración VMP

```yaml
# vmp_config.yml
protection:
  priority_mode: "attribute"  # Prioridad basada en Attribute
  default_level: "medium"

tools:
  - name: "vmprotect"
    path: "C:\\Program Files\\VMProtect Ultimate\\VMProtect.exe"

protection_levels:
  high:
    virtualization: "full"
    mutation: "high"
    anti_debug: true
    anti_dump: true
    encrypt_strings: true
    encrypt_resources: true

  medium:
    virtualization: "partial"
    mutation: "medium"
    anti_debug: true
    encrypt_strings: true

  low:
    virtualization: "none"
    mutation: "low"
    anti_debug: false
```

Escribe la configuración con claridad, para que sea más fácil mantenerla más adelante.

## Guía práctica

### 1. Práctica de protección de componentes clave

Según la especificación code-protection de HagiCode, los siguientes componentes deben usar protección de prioridad HIGH:

```csharp
// Constantes de entorno de producción - deben cifrarse y estar protegidas por VMP
[VmProtect(VmProtectionPriority.High, Reason = "Production constants")]
public static class ProductionConstants
{
    // Accesor de cadenas cifradas, protegido por VMP
    [VmProtect(VmProtectionPriority.High)]
    public static string GetLicenseServerUrl(IOptions<LicenseOptions> options) => ...;
}

// Lógica de verificación de licencia
[VmProtect(VmProtectionPriority.High, Reason = "License verification logic")]
public class KeygenClient : IKeygenClient
{
    [Obfuscation(Feature = "ultra", Exclude = false)]
    public async Task<LicenseValidationResult?> ValidateLicenseAsync(...) { ... }
}

// Servicio de huella de máquina
[VmProtect(VmProtectionPriority.High)]
public class MachineFingerprintService : IMachineFingerprintService { ... }
```

El código clave debe estar protegido con énfasis, después de todo, si se filtra la lógica central, habrá problemas.

### 2. Descifrado en tiempo de ejecución de cadenas cifradas

Cifra cadenas en tiempo de compilación, descifra en tiempo de ejecución:

```csharp
public static class StringDecryption
{
    [VmProtect(VmProtectionPriority.High, Reason = "CRITICAL SECURITY")]
    public static string DecryptString(byte[] encryptedData, byte[] key, byte[] iv)
    {
        using var aes = Aes.Create();
        aes.Key = key;
        aes.IV = iv;

        using var decryptor = aes.CreateDecryptor();
        using var ms = new MemoryStream(encryptedData);
        using var cs = new CryptoStream(ms, decryptor, CryptoStreamMode.Read);
        using var reader = new StreamReader(cs);

        return reader.ReadToEnd();
    }
}

// Accesor de constantes de producción (carga diferida + caché)
public static class ProductionConstants
{
    private static string? _cachedLicenseServerUrl;

    public static string GetLicenseServerUrl(IOptions<LicenseOptions> options)
    {
        if (_cachedLicenseServerUrl == null)
        {
            var encrypted = GetEncryptedLicenseServerUrl();
#if DEBUG
            _cachedLicenseServerUrl = options.Value.PrimaryServer.Url;
#else
            _cachedLicenseServerUrl = StringDecryption.DecryptString(
                encrypted,
                GetEncryptionKey(),
                GetEncryptionIV());
#endif
        }
        return _cachedLicenseServerUrl;
    }
}
```

El cifrado de cadenas también es bastante importante, después de todo, la información sensible no puede dejarse en texto plano.

### 3. Verificación de protección VMP

Después de la compilación, debe verificarse que la protección se haya aplicado correctamente, de lo contrario, ¿cómo saber que la protección está vigente:

```csharp
// Ejemplo de script de verificación
public bool VerifyProtection(string assemblyPath)
{
    // 1. Verificar firma VMP
    var bytes = File.ReadAllBytes(assemblyPath);
    var vmpSignature = Encoding.ASCII.GetBytes("VMProtect");
    if (bytes.Any(b => vmpSignature.Contains(b)))
    {
        return true;
    }

    // 2. Verificar cambio de tamaño de archivo (después de la protección generalmente aumenta)
    var originalInfo = new FileInfo(assemblyPath.Replace(".dll", ".bak"));
    if (originalInfo.Exists)
    {
        var sizeRatio = (double)new FileInfo(assemblyPath).Length / originalInfo.Length;
        return sizeRatio > 1.1;
    }

    return false;
}
```

Siempre es bueno verificar, para no saber cuando hay un problema.

### 4. Precauciones

Hay algunos problemas a los que hay que prestar especial atención, después de todo, los aprendimos por experiencia:

1. **No ofusques todo el código**: las API públicas, definiciones de interfaces y clases DTO generalmente no necesitan protección, la ofuscación excesiva afectará el rendimiento y la depuración, el proyecto HagiCode sufrió esto, hay que prestar atención

2. **Protege los accesores de claves**: los métodos de obtención de claves de cifrado deben tener el mismo nivel o un nivel de protección superior que los datos cifrados, de lo contrario es inútil, tampoco tiene mucho sentido

3. **Equilibrio entre pruebas y producción**: las compilaciones DEBUG deben omitir el cifrado para facilitar la depuración del desarrollo, las compilaciones RELEASE deben habilitar la protección completa, recuerda usar compilación condicional `#if DEBUG` para distinguir, así es más conveniente para el desarrollo

4. **Consideraciones de entorno Docker**: ejecutar VMP en entornos Linux requiere una solución de contenedorización, asegurando la compatibilidad de las herramientas de protección, HagiCode usa la solución de contenedor Wine + VMP, los problemas multiplataforma se resuelven así

5. **La verificación es indispensable**: después de completar la compilación, debe verificarse que la protección sea exitosa, de lo contrario podría exponer código sensible, el código de verificación anterior sirve para esto, es mejor verificar

## Resumen

A través de esta estrategia de protección de múltiples niveles, HagiCode ha implementado un sistema completo de seguridad de código desde la ofuscación básica hasta la protección por máquina virtual:

- **Primer nivel**: usa `ObfuscationAttribute` para marcado básico, proporcionando sugerencias para herramientas de terceros
- **Segundo nivel**: declara intenciones y prioridades de protección a través de `VmProtectAttribute` personalizado
- **Tercer nivel**: la protección por máquina virtual VMP convierte el código clave en instrucciones de máquina virtual irreversibles
- **Cuarto nivel**: escaneo automático en tiempo de compilación para aplicar protección, verificación de resultados de protección

Este sistema puede resistir tanto herramientas comunes de descompilación como ataques de análisis inverso avanzados.
