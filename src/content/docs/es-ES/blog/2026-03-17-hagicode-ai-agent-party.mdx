## Creando un equipo de aventuras AI: Práctica de configuración de colaboración multi-Agent de HagiCode

> En el desarrollo de software moderno, un solo AI Agent ya es difícil de satisfacer necesidades complejas. ¿Cómo hacer que múltiples asistentes IA de diferentes empresas trabajen juntos en el mismo proyecto? Este artículo compartirá la solución de configuración de colaboración multi-Agent explorada por el proyecto HagiCode en el desarrollo real.

:::note[Nota sobre el contexto temporal]
Este artículo registra la práctica de configuración multi-Agent de la **versión publicada el 2026-03-17**. El alcance de soporte actual residente se basa en la configuración actual del backend: Claude Code, Codex, GitHub Copilot, OpenCode, Hermes, QoderCLI, Kiro, Kimi, Gemini, DeepAgents, Codebuddy; donde `IFlowCli` ahora se mantiene solo como un valor de compatibilidad histórica y no pertenece al alcance de soporte activo actual. ## Antecedentes

Creo que muchos desarrolladores han tenido esta experiencia: introducir asistentes IA en el proyecto para ayudar con la programación mejora la eficiencia. Pero a medida que las necesidades se vuelven cada vez más complejas, un AI Agent comienza a no ser suficiente: quieres que maneje múltiples tareas como revisión de código, generación de documentación, pruebas unitarias al mismo tiempo, y el resultado a menudo es descuidar una cosa por otra, con calidad de salida desigual.

Lo que es más doloroso es cuando intentas introducir múltiples asistentes IA, el problema se vuelve más complejo. Cada Agent tiene su propia forma de configuración, interfaz API y lógica de ejecución, e incluso pueden entrar en conflicto entre sí. Es como un equipo de fútbol, cada jugador es muy talentoso, pero nadie sabe cómo cooperar, y el resultado es un desastre.

El proyecto HagiCode también encontró el mismo problema durante el desarrollo. Como un proyecto complejo que involucra extensiones VSCode frontend, servicios AI backend y clientes de escritorio multiplataforma, en la **versión de 2026-03** necesitábamos conectar múltiples asistentes IA de diferentes empresas al mismo tiempo: Claude Code, Codex, CodeBuddy, iFlow, etc. Cómo hacer que coexistan armoniosamente en el mismo proyecto y desplieguen sus respectivas fortalezas se convirtió en un problema clave que debía resolverse.

Después de todo, ¿quién querría tratar con un grupo de IA peleando todos los días?

La solución compartida en este artículo es exactamente la práctica de configuración de colaboración multi-Agent que optimizamos en el proyecto HagiCode a través de prueba y error reales. Si también estás sufriendo por la colaboración de múltiples asistentes IA, creo que este artículo te dará algunas ideas. Quizás, después de todo, la situación de cada persona es diferente.

## Sobre HagiCode

[HagiCode](https://hagicode.com) es un proyecto de asistente de código IA que adopta el modo de "equipo de aventuras" de colaboración de múltiples motores IA. Dirección del proyecto: [github.com/HagiCode-org/site](https://github.com/HagiCode-org/site).

La solución de configuración multi-Agent compartida en este artículo es una de las tecnologías centrales que permiten a HagiCode mantener un desarrollo eficiente en proyectos complejos. No hay nada especial, es solo convertir un grupo de IA en un equipo de aventuras que pueda cooperar.

## Diseño de arquitectura multi-Agent de HagiCode

### De "luchar solo" a "trabajo en equipo"

En las primeras etapas del proyecto HagiCode, también intentamos usar solo un AI Agent para manejar todas las tareas. Pronto descubrimos que este enfoque tiene cuellos de botella obvios: diferentes tareas requieren diferentes enfoques de capacidad, algunas tareas requieren capacidades de comprensión de contexto más fuertes, otras requieren capacidades de modificación de código más precisas. Es difícil que un Agent sobresalga en todos los aspectos.

Esto nos hizo darnos cuenta de que debemos hacer que múltiples Agent trabajen juntos. Pero el problema es, ¿cómo hacer que productos IA de diferentes empresas coexistan pacíficamente en el mismo proyecto? Necesitábamos resolver varios problemas centrales:

1. **Complejidad de gestión de configuración**: cada Agent tiene diferentes métodos de configuración, interfaces API y modos de ejecución
2. **Unificación de protocolos de comunicación**: se necesita una forma estandarizada de intercambiar datos entre diferentes Agent
3. **Coordinación de división de tareas**: cómo asignar tareas razonablemente para que cada Agent despliegue sus fortalezas

Con estos problemas, comenzamos a diseñar la arquitectura multi-Agent de HagiCode. En realidad no es tan complicado, solo entendimos el enfoque.

### Vista general de la arquitectura

Después de múltiples iteraciones, la arquitectura que finalmente determinamos es esta:

```
┌─────────────────────────────────────────────────────────────────┐
│                    AIProviderFactory                             │
│  (Patrón de fábrica gestiona uniformemente todos los AI Provider)│
├─────────────────────────────────────────────────────────────────┤
│  ClaudeCodeCli  │  CodexCli  │  CodebuddyCli  │  IFlowCli    │
│  (Anthropic)   │  (OpenAI)  │  (Zhipu GLM)     │  (Zhipu)      │
└─────────────────────────────────────────────────────────────────┘
```

La idea central es: a través de una interfaz Provider unificada, diferentes AI Agent pueden ser gestionados por el mismo conjunto de código. Al mismo tiempo, use el patrón de fábrica para crear y configurar dinámicamente estos Provider, asegurando la escalibilidad y flexibilidad del sistema.

Es como la división del trabajo en la vida, cada persona tiene su propio rol, solo que aquí esta división del trabajo se convierte en arquitectura de código.

### Tipos de Agent y división de responsabilidades

Según la experiencia de uso real del proyecto HagiCode, asignamos diferentes responsabilidades a cada Agent:

| Agent | Proveedor | Modelo | Uso principal |
|-------|--------|------|---------|
| ClaudeCodeCli | Anthropic | glm-5-turbo | Generar propuestas técnicas y Proposals |
| CodexCli | OpenAI/Zed | gpt-5.4 | Ejecutar modificaciones de código precisas |
| CodebuddyCli | Zhipu | glm-4.7 | Optimizar descripciones de propuestas y documentación |
| IFlowCli | Zhipu | glm-4.7 | Archivar propuestas y registros históricos (configuración en ese momento; actualmente solo compatibilidad histórica) |
| OpenCodeCli | - | - | Edición de código general |
| GitHubCopilot | Microsoft | - | Programación asistida y autocompletado de código |

La lógica detrás de esta división es: cada Agent tiene su propio campo de especialización. Claude Code sobresale en comprender y analizar requisitos complejos, por lo que se encarga del diseño de soluciones en las primeras etapas; Codex es más preciso en modificaciones de código, adecuado para manejar tareas de implementación específicas; CodeBuddy tiene una alta relación calidad-precio, perfecto para optimizar documentación.

Después de todo, lo que te conviene es lo mejor, todos los caminos conducen a Roma, solo que algunos caminos son más fáciles de recorrer y otros son un poco más tortuosos.

## Mecanismo de configuración central

### Diseño de interfaz Provider unificada

Para permitir que diferentes AI Agent puedan ser gestionados de manera uniforme, primero necesitamos definir un conjunto de interfaces unificadas. HagiCode define esta interfaz:

```csharp
public interface IAIProvider
{
    // Interfaz Provider unificada
    Task<IAIProvider?> GetProviderAsync(AIProviderType providerType);
    Task<IAIProvider?> GetProviderAsync(string providerName, CancellationToken cancellationToken);
}
```

Esta interfaz parece simple, pero es la piedra angular de todo el sistema multi-Agent. A través de la interfaz unificada, podemos ignorar qué producto IA de qué compañía está en la parte inferior y llamarlo de la misma manera.

De hecho, esto es simplificar cosas complejas, después de todo, lo simple es bello.

### Implementación del patrón de fábrica Provider

Con una interfaz unificada, lo siguiente es cómo crear estas instancias de Provider. HagiCode usa el patrón de fábrica:

```csharp
private IAIProvider? CreateProvider(AIProviderType providerType, ProviderConfiguration config)
{
    return providerType switch
    {
        AIProviderType.ClaudeCodeCli =>
            ActivatorUtilities.CreateInstance<ClaudeCodeCliProvider>(_serviceProvider, Options.Create(config)),
        AIProviderType.CodebuddyCli =>
            ActivatorUtilities.CreateInstance<CodebuddyCliProvider>(_serviceProvider, Options.Create(config)),
        AIProviderType.CodexCli =>
            ActivatorUtilities.CreateInstance<CodexCliProvider>(_serviceProvider, Options.Create(config)),
        AIProviderType.IFlowCli =>
            ActivatorUtilities.CreateInstance<IFlowCliProvider>(_serviceProvider, Options.Create(config)),
        _ => null
    };
}
```

Aquí se usa `ActivatorUtilities.CreateInstance` de inyección de dependencias, que puede crear instancias de Provider dinámicamente en tiempo de ejecución e inyectar dependencias automáticamente. El beneficio de este diseño es: al agregar un nuevo tipo de Agent, solo necesita agregar la clase Provider correspondiente y luego agregar una rama de caso en el método de fábrica, sin necesidad de modificar el código existente.

Después de todo, ¿quién quiere modificar un montón de código antiguo cada vez que agrega una nueva función?

### Resolución de configuración dinámica

Para hacer la configuración más flexible, también implementamos un mecanismo de mapeo de tipos:

```csharp
public static AIProviderTypeExtensions
{
    private static readonly Dictionary<string, AIProviderType> _typeMap = new(
        StringComparer.OrdinalIgnoreCase)
    {
        ["ClaudeCodeCli"] = AIProviderType.ClaudeCodeCli,
        ["CodebuddyCli"] = AIProviderType.CodebuddyCli,
        ["CodexCli"] = AIProviderType.CodexCli,
        ["IFlowCli"] = AIProviderType.IFlowCli,
        // ... más mapeos de tipos
    };
}
```

El papel de esta tabla de mapeo es convertir nombres de Provider en forma de cadena a tipos de enumeración. De esta manera, los archivos de configuración pueden usar nombres de cadena intuitivos, mientras que el código interno usa enumeraciones seguras de tipos para procesar.

Después de todo, la configuración, cuanto más intuitiva mejor, ¿quién quiere recordar un montón de código complejo?

### Ejemplo de archivo de configuración

En uso real, solo necesita configurar en `appsettings.json`:

```yaml
AI:
  Providers:
    Providers:
      ClaudeCodeCli:
        Enabled: true
        Model: glm-5-turbo
        WorkingDirectory: /path/to/project
      CodebuddyCli:
        Enabled: true
        Model: glm-4.7
      CodexCli:
        Enabled: true
        Model: gpt-5.4
      IFlowCli:
        Enabled: true
        Model: glm-4.7
```

Cada Provider puede configurar independientemente interruptores, versiones de modelo, directorios de trabajo y otros parámetros. Este diseño garantiza flexibilidad y facilita la gestión y el mantenimiento.

De hecho, los archivos de configuración son como las opciones en la vida, puedes elegir activar o desactivar ciertas funciones, solo que las opciones en el código son más fáciles de lamentar.

## Flujo de tareas del equipo de aventuras

### El arte de la división de tareas

Con una arquitectura técnica unificada, lo siguiente es cómo hacer que múltiples Agent trabajen juntos. HagiCode diseñó un mecanismo de flujo de tareas que permite que diferentes Agent manejen tareas en diferentes etapas:

```
Creación de propuesta (usuario)
    │
    ▼
[Claude Code] ──generar propuesta──▶ Documento de propuesta
    │                        │
    │                        ▼
    │               [Codebuddy] ──optimizar descripción──▶ Propuesta optimizada
    │                        │
    │                        ▼
    │               [Codex] ──ejecutar modificación──▶ Cambio de código
    │                        │
    │                        ▼
    └───────────────▶ [iFlow] ──archivar──▶ Registro histórico
```

El beneficio de esta división es: cada Agent solo necesita concentrarse en las tareas en las que es bueno, no necesita "saberlo todo". Claude Code es responsable de generar propuestas desde cero, Codebuddy es responsable de hacer que las descripciones de las propuestas sean más claras, Codex es responsable de convertir las propuestas en cambios de código reales, e iFlow es responsable de archivar y guardar estos cambios.

De hecho, es como el trabajo en equipo en la vida, cada persona tiene su propio rol, y juntos pueden completar algo grande. Solo que los miembros del equipo aquí son IA.

### Puntos clave de práctica

En la operación real, resumimos las siguientes experiencias:

**1. La estrategia de selección de Agent es importante**

No asignes tareas al azar, sino asigna según las fortalezas de cada Agent:
- Generación de propuestas: usa Claude Code, porque tiene capacidades de comprensión de contexto más fuertes
- Ejecución de código: usa Codex, porque es más preciso en modificaciones de código
- Optimización de propuestas: usa Codebuddy, porque tiene una alta relación calidad-precio
- Archivo y almacenamiento: usa iFlow, porque es estable y confiable

Después de todo, dejar que la persona adecuada haga lo adecuado es una verdad eterna.

**2. El aislamiento de configuración asegura la estabilidad**

La configuración de cada Agent se gestiona de forma independiente, admite anulación de variables de entorno, y los directorios de trabajo también son independientes. De esta manera, un error de configuración de un Agent no afectará a otros Agent.

Es como los límites en la vida, cada persona tiene su propio espacio, y solo sin interferencia mutua se puede coexistir en armonía.

**3. Mecanismo de manejo de errores**

El fallo de un solo Agent no debería afectar el flujo general. Implementamos una estrategia de degradación: cuando un Agent falla en la ejecución, el sistema puede cambiar automáticamente a una solución de respaldo, o omitir directamente ese paso y continuar ejecutando las tareas posteriores. Al mismo tiempo, registros completos facilitan la resolución de problemas después del hecho.

Después de todo, nadie puede garantizar que nunca habrá errores, la clave es cómo manejar los errores. Es como la vida, siempre encontrarás contratiempos, lo importante es cómo superarlos.

**4. Monitoreo y observabilidad**

A través del protocolo ACP (nuestro protocolo de comunicación personalizado, basado en JSON-RPC 2.0), se puede rastrear el estado de ejecución de cada Agent. El aislamiento de sesiones asegura la seguridad de concurrencia, y el almacenamiento en caché dinámico optimiza el rendimiento.

Después de todo, las cosas invisibles son las más propensas a problemas, tener algo de monitoreo es mejor que estar a ciegas.

## Efectos reales y beneficios

Después de adoptar esta configuración de colaboración multi-Agent, la eficiencia de desarrollo del proyecto HagiCode mejoró significativamente. Específicamente:

1. **Capacidad de procesamiento de tareas duplicada**: antes, un Agent necesitaba manejar múltiples tareas al mismo tiempo, ahora puede procesarlas en paralelo, el rendimiento se ha más que duplicado
2. **Calidad de salida más estable**: cada Agent solo se concentra en las tareas en las que es bueno, la consistencia y calidad de los resultados de salida son más altas
3. **Costos de mantenimiento reducidos**: la interfaz y gestión de configuración unificadas hacen que todo el sistema sea más fácil de mantener y expandir
4. **Agregar nuevos Agent es simple**: si desea conectar un nuevo producto IA, solo necesita implementar la interfaz y agregar configuración, sin necesidad de modificar la lógica central

Esta solución no solo resolvió los problemas de HagiCode, sino que también demostró que la colaboración multi-Agent es de hecho una opción de arquitectura viable.

De hecho, el efecto es bastante obvio, solo que el proceso fue un poco complicado.

## Conclusión

Este artículo comparte la experiencia práctica del proyecto HagiCode en configuración de colaboración multi-Agent. Los puntos clave incluyen:

1. **Interfaz estandarizada**: a través de `IAIProvider` unificar el comportamiento de diferentes Agent, permitiendo que el código ignore qué producto de qué compañía está en la parte inferior
2. **Patrón de fábrica**: usar `ActivatorUtilities.CreateInstance` para crear instancias de Provider dinámicamente, soportando configuración en tiempo de ejecución e inyección de dependencias
3. **Protocolo unificado**: el protocolo ACP implementa comunicación estandarizada entre Agent, un mecanismo de comunicación bidireccional basado en JSON-RPC 2.0
4. **División de tareas**: asignar razonablemente tareas a diferentes Agent, dejándolos desplegar sus fortalezas, en lugar de intentar hacer que un Agent haga todo

Este diseño no solo resolvió el problema de "múltiples Agent peleando", sino que también logró la automatización y especialización del proceso de desarrollo a través del mecanismo de flujo de tareas del equipo de aventuras.
