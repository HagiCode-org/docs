## Desde cero: Cómo integrar Reasonix CLI en el sistema HagiCode

> Este artículo comparte la práctica técnica completa de integrar Reasonix CLI como un proveedor de agentes de primera clase en el sistema HagiCode, cubriendo el diseño de arquitectura de tres capas, decisiones técnicas clave y detalles de implementación en frontend y backend.

## Antecedentes

Reasonix CLI, para decirlo de alguna manera, es una cosa bastante interesante. Es una herramienta de asistente de código con IA basada en ACP (Agent Communication Protocol) que proporciona potentes capacidades de transmisión en streaming y gestión de sesiones. De hecho, en la capa HagiCode.Libs, ya hemos completado la implementación de bajo nivel, pero estos componentes aún se encuentran en un estado aislado, como perlas bonitas que aún no se han ensartado en un collar. Los usuarios no pueden usarlo a través de la selección de profesiones Hero, el flujo de ejecución de sesiones o el panel de monitoreo, lo cual es un poco lamentable.

El problema que enfrentamos es: ¿cómo elevar Reasonix al mismo nivel que Codex, Hermes y otros proveedores de agentes de primera clase, logrando un enrutamiento completo del backend y una visualización en el frontend? Esto no es tan simple como registrar un valor de enumeración, sino que requiere construir un flujo completo desde la abstracción de bajo nivel hasta la interfaz de usuario. Es como construir una casa, no puedes solo poner los cimientos, tienes que levantar las paredes y poner el techo.

El desafío de esta integración radica en que Reasonix, como una herramienta CLI local, tiene su propio carácter y temperamento. Por ejemplo, no necesita una cadena de conexión, todos los parámetros son configurados por el usuario en tiempo de ejecución; puede no estar instalado en absoluto, requiriendo un manejo de degradación elegante; es compatible con modelos de la serie anthropic, pero tiene sus propios parámetros específicos de ACP como effort, budget, etc. Es como una persona, con su propia forma única de hacer las cosas, no se puede forzar.

Después de un diseño cuidadoso de la arquitectura y múltiples rondas de discusión, finalmente adoptamos un esquema de arquitectura de tres capas claro, integrando exitosamente Reasonix en el sistema. Este esquema no solo resolvió el problema actual, sino que también proporcionó un patrón reutilizable para integraciones posteriores de proveedores CLI similares. De hecho, muchas cosas son así, una vez que encuentras el método correcto, el camino posterior es mucho más fácil.

## Contenido principal

### Diseño de arquitectura técnica

El sistema adopta una arquitectura de tres capas clara para separar las preocupaciones, cada capa tiene límites de responsabilidad claros:

**Capa HagiCode.Libs**: Esta capa ya está completada, proporcionando la abstracción del proveedor CLI y la implementación específica. Define la interfaz `ICliProvider<ReasonixOptions>`, implementa `ReasonixProvider` para manejar la transmisión en streaming ACP y la gestión de sesiones, al mismo tiempo que soporta parámetros como effort, budget, yolo, transcript, etc. La responsabilidad de esta capa es proporcionar capacidades de bajo nivel estables y reutilizables, sin involucrar ninguna lógica de negocio. Es como los cimientos de una casa, aunque no se ven, son muy importantes.

**Capa hagicode-core**: Este es el foco de nuestra integración actual. Es responsable de conectar la abstracción de bajo nivel con la interfaz unificada del sistema. El trabajo específico incluye registrar el valor de enumeración `AIProviderType.ReasonixCli = 12`, crear `ReasonixCliProvider` como un adaptador fino que conecta la capa Libs, implementar `ReasonixGrain` para manejar el estado de sesión y el flujo de ejecución, así como integrar el sistema Hero para el mapeo de parámetros y la gestión de configuración. El núcleo de esta capa es coordinar los componentes, construyendo el flujo de negocio completo. Es como las paredes portantes de una casa, conectando todas las partes.

**Capa web**: Responsable de mostrar y recopilar configuraciones para los usuarios. Necesitamos regenerar los tipos de OpenAPI para soportar nuevos valores de enumeración, implementar el mapeo de tipos visuales para que Reasonix tenga su propio icono y nombre para mostrar, crear formularios de configuración de parámetros CLI para que los usuarios puedan configurar varios parámetros, y agregar soporte multilingüe. El enfoque de esta capa es la experiencia del usuario y el diseño de interacción. Es como la decoración de una casa, si está bien hecha o no afecta directamente qué tan cómodo sea vivir.

Este diseño en capas permite que cada capa se concentre en sus propias responsabilidades, reduciendo la complejidad del sistema y facilitando el mantenimiento y la expansión posterior. De hecho, muchas veces, cuando las cosas se aclaran, se vuelven más simples.

### Decisiones técnicas clave

Durante el proceso de implementación, tomamos varias decisiones técnicas clave que tienen un impacto importante en la arquitectura final y la experiencia del usuario.

**Decisión uno: Usar un Grain dedicado**

Creamos un `ReasonixGrain : IReasonixGrain, IExecutorStreamGrain` independiente, en lugar de intentar reutilizar un Grain compartido. Esta decisión sigue el patrón establecido de los 11 proveedores existentes del sistema. Aunque puede parecer que hay cierta repetición de código, un Grain dedicado nos permite hacer un control refinado para las características de Reasonix, como su mecanismo de vinculación de sesiones específico y el mapeo de mensajes ACP. También definimos un DTO de respuesta vacío `ReasonixResponse` como un discriminador de tipo, aunque no contiene datos reales, juega un papel importante en el sistema de tipos. Es como cada persona tiene su propia habitación, incluso si está vacía, es su propio espacio.

**Decisión dos: No crear una clase Settings dedicada**

A diferencia de ciertos proveedores que necesitan una cadena de conexión, toda la configuración de Reasonix es establecida por el usuario en tiempo de ejecución, sin necesidad de verificación al inicio. Por lo tanto, no creamos una clase Settings dedicada, sino que almacenamos toda la configuración en el diccionario `AIProviderOptions.Providers[ReasonixCli].Settings`. Este patrón se mantiene consistente con otros proveedores CLI locales como Qoder, Kiro, Kimi, simplificando la estructura del código y evitando capas de abstracción innecesarias. Las claves de configuración compatibles incluyen: `effort`, `budgetUsd`, `transcriptPath`, `enableYolo`, `arguments`, `startupTimeoutMs`, `reasoning`. A veces, lo simple es mejor.

**Decisión tres: Monitoreo de salud de estrategia de proveedor**

Reasonix es un CLI instalado localmente por el usuario, puede no estar instalado en absoluto, o no estar en el PATH del sistema. En este caso, no deberíamos lanzar un error directamente, sino manejarlo con una degradación elegante. Usamos la estrategia `Provider`, verificando si el CLI está disponible a través de `CommandUtil.TryResolveExecutablePath`. Si la verificación falla, la interfaz mostrará "no disponible", pero no afectará otras partes del sistema. Este diseño hace que el sistema sea más robusto y también proporciona retroalimentación clara al usuario. Después de todo, nadie quiere que todo el sistema se caiga por un pequeño problema.

**Decisión cuatro: Clasificación del sistema económico**

En el sistema HagiCode, diferentes proveedores tienen diferentes clasificaciones del sistema económico. Decidimos que Reasonix use por defecto la clasificación del sistema económico `'claude'`, porque Reasonix en sí es compatible con modelos de la serie anthropic. Actualmente solo Codex y Copilot tienen clasificaciones del sistema económico dedicadas, otros proveedores reutilizan las clasificaciones existentes. Esto mantiene la simplicidad del sistema y maneja correctamente la facturación y las estadísticas de costos. Reutilizar también es sabiduría, no necesitas empezar todo desde cero.

**Decisión cinco: Compatibilidad de modelos**

Reasonix soporta múltiples modelos a través de la bandera `--model`, especialmente la serie anthropic. Agregamos mapeo de compatibilidad en `secondary-professions.index.json`, permitiendo que los usuarios seleccionen estos modelos en Reasonix. Este diseño respeta las capacidades de Reasonix y mantiene la consistencia del sistema, los usuarios no necesitan entender las diferencias de bajo nivel para usar varios modelos sin problemas. Los usuarios tampoco lo tienen fácil, es mejor dejarles las cosas simples.

**Implementación de Thin Adapter**

`ReasonixCliProvider` es el componente clave que conecta la capa Libs con la interfaz unificada del sistema:

```csharp
public sealed class ReasonixCliProvider : IAIProvider, IVersionedAIProvider, IAsyncDisposable
{
    private static readonly IReadOnlyList<string> SupportedSettingKeys =
    [
        "effort",
        "budgetUsd",
        "transcriptPath",
        "enableYolo",
        "arguments",
        "startupTimeoutMs",
        "reasoning"
    ];

    private readonly ICliProvider<ReasonixOptions> _provider;
    private readonly ConcurrentDictionary<string, string> _sessionBindings = new(StringComparer.Ordinal);

    public async IAsyncEnumerable<AIStreamingChunk> StreamCoreAsync(
        AIRequest request,
        string? sessionId = null,
        [EnumeratorCancellation] CancellationToken cancellationToken = default)
    {
        var options = BuildOptions(request, sessionId);
        await foreach (var message in _provider.StreamAsync(options, cancellationToken))
        {
            yield return MapToStreamingChunk(message);
        }
    }

    private ReasonixOptions BuildOptions(AIRequest request, string? sessionId)
    {
        return new ReasonixOptions
        {
            ExecutablePath = GetExecutablePath(),
            WorkingDirectory = GetWorkingDirectory(),
            Model = _config.Model,
            Effort = _config.Settings.GetValueOrDefault("effort", "medium"),
            Budget = _config.Settings.GetValueOrDefault("budgetUsd", 10.0),
            Yolo = _config.Settings.GetValueOrDefault("enableYolo", false),
            TranscriptPath = _config.Settings.GetValueOrDefault("transcriptPath"),
            Arguments = _config.Settings.GetValueOrDefault("arguments", ""),
            StartupTimeout = GetStartupTimeout(),
            EnvironmentVariables = _environmentVariables,
            CessionId = sessionId ?? GetCessionId()
        };
    }
}
```

Las responsabilidades clave de este adaptador son:
1. Validar parámetros de configuración, rechazando claves de configuración no compatibles
2. Mantener relaciones de vinculación de sesiones, soportando la recuperación de sesiones
3. Mapear mensajes ACP al formato unificado del sistema
4. Usar `ProviderErrorAutoRetryCoordinator` para implementar reintentos automáticos

Es como un traductor, traduciendo de un idioma a otro, asegurando que el significado sea preciso.

**Implementación de Orleans Grain**

`ReasonixGrain` es responsable de manejar el estado de sesión y el flujo de ejecución:

```csharp
public class ReasonixGrain : Grain, IReasonixGrain, IExecutorStreamGrain
{
    private readonly Dictionary<string, ExecutorToolLifecycleStatus> _toolLifecycleState =
        new(StringComparer.Ordinal);

    public IAsyncEnumerable<ReasonixResponse> ExecuteCommandStreamAsync(
        string command,
        string? heroId = null,
        CancellationToken token = default,
        string? executionMessageId = null,
        string? systemMessage = null,
        Dictionary<string, string>? requestSettings = null)
    {
        var request = BuildRequest(command, isEdit: false, heroId, executionMessageId, systemMessage, requestSettings);
        return SendAsync(request, heroId, token);
    }

    private async IAsyncEnumerable<ReasonixResponse> SendAsync(
        AIRequest request,
        string? heroId,
        [EnumeratorCancellation] CancellationToken token)
    {
        _cancellationTokenSource = new CancellationTokenSource();
        var linkedToken = CancellationTokenSource.CreateLinkedTokenSource(token, _cancellationTokenSource.Token);

        var provider = await ResolveReasonixProviderAsync(heroId);

        await foreach (var chunk in provider.StreamAsync(request, linkedToken.Token))
        {
            var response = BuildChunkResponse(chunk);
            yield return response;
        }
    }

    private async Task<IAIProvider> ResolveReasonixProviderAsync(string? heroId)
    {
        // Lógica de reserva de configuración consciente de Hero
        var config = await HeroProviderResolver.ResolveAsync(AIProviderType.ReasonixCli, heroId);
        return _aiProviderFactory.CreateProvider(AIProviderType.ReasonixCli, config);
    }
}
```

Las funciones principales de Grain incluyen:
1. Usar `[PersistentState("reasonix-interop")]` para mantener el estado de sesión
2. Implementar lógica de reserva de configuración consciente de Hero
3. Rastrear el estado del ciclo de vida de las herramientas
4. Soportar cadena de tokens de cancelación, asegurando que la ejecución pueda interrumpirse a tiempo

Es como un mayordomo, organizando las cosas de manera ordenada.

### Mejores prácticas y precauciones

Durante el proceso de implementación, resumimos algunas mejores prácticas y lugares a tener en cuenta.

**Validación de parámetros**

ReasonixCliProvider debe validar estrictamente los parámetros de configuración, rechazando claves de configuración no compatibles:

```csharp
public void ValidateConfigurationOverrides(Dictionary<string, string?> overrides)
{
    foreach (var key in overrides.Keys)
    {
        if (!SupportedSettingKeys.Contains(key))
        {
            throw new HeroProviderConfigurationException(
                $"Unsupported setting key '{key}' for Reasonix provider");
        }
    }
}
```

Esto puede evitar que los usuarios configuren parámetros incorrectos, evitando errores en tiempo de ejecución. Es como un portero, no se puede dejar que entren cosas que no deben entrar.

**Gestión de vinculación de sesiones**

Usar `ConcurrentDictionary` para gestionar vínculos de sesión, soportando la recuperación de sesiones:

```csharp
_sessionBindings[cessionId] = sessionId;

// Vincular sesión existente en solicitudes posteriores
if (_sessionBindings.TryGetValue(cessionId, out var boundSessionId))
{
    options.SessionId = boundSessionId;
}
```

Este diseño permite que los usuarios reanuden sesiones anteriores después de una interrupción, proporcionando una mejor experiencia. Es como recordar una historia, la próxima vez puedes seguirla contando.

**Manejo de degradación elegante**

El frontend debería verificar la disponibilidad del CLI y dar indicaciones amigables:

```typescript
const reasonixAvailable = await healthApi.checkCliAvailable('reasonix');
if (!reasonixAvailable) {
  showMessage('Reasonix CLI no está instalado, por favor configúrelo en el PATH del sistema');
}
```

No dejes que los usuarios encuentren errores inexplicables, verifica de antemano y da instrucciones claras. Después de todo, nadie quiere quedarse atascado inexplicablemente.
