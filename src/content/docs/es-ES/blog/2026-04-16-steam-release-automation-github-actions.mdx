## Cómo implementar la publicación automatizada en Steam con GitHub Actions

> Este artículo comparte la solución completa para implementar la publicación automatizada en Steam en el proyecto HagiCode Desktop, desde GitHub Release hasta la tubería de automatización completa de la plataforma Steam, incluyendo detalles técnicos clave como la autenticación Steam Guard y la carga de múltiples plataformas Depot.

## Antecedentes

El proceso de publicación en la plataforma Steam es en realidad bastante diferente del método tradicional de distribución de aplicaciones. Steam tiene su propio sistema completo de distribución de actualizaciones. Los desarrolladores deben cargar los artefactos de compilación en la red CDN de Steam a través de la herramienta SteamCMD, en lugar de simplemente lanzar un enlace de descarga como en otras plataformas.

El proyecto HagiCode Desktop planea lanzarse en la plataforma Steam, lo que también trae algunos nuevos desafíos a nuestro proceso de publicación:

1. Necesitamos convertir los artefactos de compilación existentes a un formato compatible con Steam
2. Debemos cargarlos en la plataforma Steam a través de la herramienta SteamCMD
3. También debemos manejar Steam Guard, esa cosa
4. Necesitamos soportar la carga de Depot para múltiples plataformas (Linux, Windows, macOS)
5. También queremos implementar el flujo automatizado desde GitHub Release hasta Steam

El proyecto ya había implementado anteriormente el "modo de versión portátil" (portable version mode), que permite a la aplicación detectar cargas de servicio fijas empaquetadas en el directorio extra. Nuestro objetivo es simplemente integrar este modo portátil con la distribución de Steam sin problemas.

## Diseño de Arquitectura

El núcleo de todo el proceso de publicación en Steam es un flujo de trabajo de GitHub Actions que divide todo el proceso en tres etapas principales:

```
┌─────────────────────────────────────────────────────────────┐
│ Flujo de trabajo de GitHub Actions (Publicación en Steam)    │
├─────────────────────────────────────────────────────────────┤
│ 1. Fase de preparación:                                     │
│    - Verificar código de portable-version                   │
│    - Descargar artefactos de compilación de GitHub Release  │
│    - Descomprimir y preparar directorios de contenido Steam │
│                                                             │
│ 2. Configuración de SteamCMD:                               │
│    - Instalar/reutilizar SteamCMD                           │
│    - Autenticarse usando Steam Guard                        │
│                                                             │
│ 3. Fase de publicación:                                     │
│    - Generar archivos de configuración Depot VDF            │
│    - Generar archivos de configuración App Build VDF        │
│    - Llamar a SteamCMD para cargar en Steam                 │
└─────────────────────────────────────────────────────────────┘
```

Las ventajas de este diseño, ¿cómo decirlo?:
- Reutiliza los artefactos de GitHub Release existentes, evitando compilaciones duplicadas, después de todo, ¿quién quiere trabajo duplicado?
- A través de ejecutores autoalojados para lograr aislamiento de seguridad, siempre es bueno tener una capa adicional de protección
- Soporta el cambio entre modo de vista previa y ramas de publicación oficial, un poco más flexible
- Manejo completo de errores y registro de logs, para no estar demasiado perdido cuando surjan problemas

## Implementación del Flujo de Trabajo

### Diseño de Parámetros de Activación

Nuestro flujo de trabajo soporta los siguientes parámetros clave:

```yaml
inputs:
  release:           # Etiqueta de publicación de Portable Version
    description: 'Etiqueta de versión a publicar (ej. v1.0.0)'
    required: true
  steam_preview:     # Si se genera una compilación de vista previa
    description: 'Si es modo vista previa'
    required: false
    default: 'false'
  steam_branch:      # Establecer como rama live de Steam
    description: 'Rama de Steam objetivo'
    required: false
    default: 'preview'
  steam_description: # Sobrescritura de descripción de compilación
    description: 'Descripción de compilación'
    required: false
```

### Configuración de Ejecutor Autoalojado

Por razones de seguridad, usamos ejecutores autoalojados con etiqueta steam:

```yaml
runs-on:
  - self-hosted
  - Linux
  - X64
  - steam
```

Esto asegura que la publicación en Steam se ejecute en un ejecutor dedicado, manteniendo el aislamiento seguro de credenciales sensibles. Después de todo, con la seguridad, es mejor prestar un poco más de atención.

### Control de Concurrencia

Para evitar interferencias entre publicaciones de la misma versión, configuramos el control de concurrencia:

```yaml
concurrency:
  group: portable-version-steam-${{ github.event.inputs.release }}
  cancel-in-progress: false
```

Tenga en cuenta que aquí configuramos `cancel-in-progress: false`, porque el proceso de publicación en Steam puede ser largo y no queremos cancelar una publicación en progreso debido a un nuevo disparador. Después de todo, publicar una versión no es fácil, hay que dejarla terminar, ¿no?

## Implementación de Scripts Principales

### Preparar Entrada de Publicación

El script `prepare-steam-release-input.mjs` es responsable de preparar la entrada necesaria para la publicación:

```javascript
// Descargar manifiesto de compilación e inventario de artefactos de GitHub Release
const buildManifest = await downloadBuildManifest(releaseTag);
const artifactInventory = await downloadArtifactInventory(releaseTag);

// Descargar paquetes comprimidos de cada plataforma
for (const platform of ['linux-x64', 'win-x64', 'osx-universal']) {
  const artifactUrl = getArtifactUrl(artifactInventory, platform);
  await downloadArtifact(artifactUrl, platform);
}

// Descomprimir en estructura de directorios de contenido Steam
await extractToSteamContent(sources, contentRoot);
```

### Autenticación Steam Guard

Steam requiere usar Steam Guard para proteger cuentas, implementamos un algoritmo de generación de código basado en secreto compartido:

```javascript
function generateSteamGuardCode(sharedSecret, timestamp = Date.now()) {
  const secret = decodeSharedSecret(sharedSecret);
  const time = Math.floor(timestamp / 1000 / 30);

  const timeBuffer = Buffer.alloc(8);
  timeBuffer.writeBigUInt64BE(BigInt(time));

  // Usar HMAC-SHA1 para generar código de un solo uso basado en tiempo
  const hash = crypto.createHmac('sha1', secret)
    .update(timeBuffer)
    .digest();

  // Convertir a código Steam Guard de 5 caracteres
  const code = steamGuardCode(hash);
  return code;
}
```

Esta implementación se basa en el mecanismo TOTP (Time-based One-Time Password) de Steam Guard, generando un nuevo código de verificación cada 30 segundos. Después de todo, con la seguridad, hay que usar métodos confiables.

### Generación de Configuración VDF

VDF (Valve Data Format) es el formato de configuración usado por Steam, necesitamos generar dos tipos de archivos VDF:

**Depot VDF** para configurar el contenido de cada plataforma:

```javascript
function buildDepotVdf(depotId, contentRoot) {
  return [
    '"DepotBuildConfig"',
    '{',
    `  "DepotID" "${escapeVdf(depotId)}"`,
    `  "ContentRoot" "${escapeVdf(contentRoot)}"`,
    '  "FileMapping"',
    '  {',
    '    "LocalPath" "*"',
    '    "DepotPath" "."',
    '    "recursive" "1"',
    '  }',
    '}'
  ].join('\n');
}
```

**App Build VDF** para configurar toda la compilación de la aplicación:

```javascript
function buildAppBuildVdf(appId, depotBuilds, description, setLive) {
  const vdf = [
    '"appbuild"',
    '{',
    `  "appid" "${appId}"`,
    `  "desc" "${escapeVdf(description)}"`,
    `  "contentroot" "${escapeVdf(contentRoot)}"`,
    '  "buildoutput" "build_output"',
    '  "depots"',
    '  {'
  ];

  for (const [depotId, depotVdfPath] of Object.entries(depotBuilds)) {
    vdf.push(`    "${depotId}" "${depotVdfPath}"`);
  }

  if (setLive) {
    vdf.push(`  }`);
    vdf.push(`  "setlive" "${setLive}"`);
  }

  vdf.push('}');
  return vdf.join('\n');
}
```

### Invocación de SteamCMD

Finalmente, llamando a SteamCMD para ejecutar la carga:

```javascript
await runCommand(steamcmdPath, [
  '+login', steamUsername, steamPassword, steamGuardCode,
  '+run_app_build', appBuildPath,
  '+quit'
]);
```

## Manejo de Multiplataforma Depot

Steam usa el sistema Depot para administrar contenido de diferentes plataformas, soportamos tres Depot principales:

| Plataforma | Identificador Depot | Soporte de Arquitectura |
|------------|---------------------|------------------------|
| Linux | `linux-x64` | x64_64 |
| Windows | `win-x64` | x64_64 |
| macOS | `osx-universal` | universal, x64_64, arm64 |

Cada Depot tiene su propio directorio de contenido y archivo de configuración VDF independiente, lo que asegura que los usuarios de diferentes plataformas solo descarguen el contenido que necesitan. Después de todo, el ancho de banda también es dinero, ahorrar un poco es un poco.

## Proceso de Publicación

### Paso 1: Preparar GitHub Release

Primero necesitas crear un GitHub Release en el repositorio portable-version, incluyendo:
- Paquetes comprimidos de cada plataforma
- Manifiesto de compilación (`{tag}.build-manifest.json`)
- Inventario de artefactos (`{tag}.artifact-inventory.json`)

### Paso 2: Activar Flujo de Publicación en Steam

Activa manualmente el flujo de trabajo a través de GitHub Actions, llenando los parámetros necesarios:
- `release`: Etiqueta de versión a publicar (ej. v1.0.0)
- `steam_branch`: Rama objetivo (ej. `preview` o `public`)
- `steam_preview`: Si es modo vista previa

### Paso 3: Ejecutar Automáticamente el Proceso de Publicación

El flujo de trabajo ejecutará automáticamente los siguientes pasos:

1. Descargar y descomprimir artefactos de GitHub Release
2. Instalar/actualizar SteamCMD
3. Generar archivos de configuración Steam VDF
4. Autenticarse usando Steam Guard
5. Cargar contenido a Steam CDN
6. Establecer rama especificada como live

Después de completar todo este proceso, se puede decir que hicimos todo lo que había que hacer.

## Mejores Prácticas

### Gestión de Autenticación Steam Guard

La primera ejecución requiere ingresar manualmente el código de Steam Guard, después se recomienda configurar el secreto compartido para generar código automáticamente. Esto evita tener que intervenir manualmente en cada publicación, después de todo, nadie quiere repetir la misma operación cada vez.

SteamCMD guardará el token de inicio de sesión, que se puede reutilizar posteriormente. Pero ten en cuenta la validez del token, cuando expire aún necesitas volver a autenticarte, no hay forma de evitarlo.

### Estructura de Directorios de Contenido

Asegura que la estructura de directorios de contenido de Steam sea correcta:

```
steam-content/
├── linux-x64/     # Contenido de plataforma Linux
├── win-x64/       # Contenido de plataforma Windows
└── osx-universal/ # Contenido binario universal de macOS
```

Cada directorio debe contener los archivos completos de la aplicación correspondientes a esa plataforma. Esto no tiene mucho más que decir, simplemente hay que hacer lo que se debe hacer.

### Uso del Modo Vista Previa

El modo vista previa no establece ninguna rama como live, adecuado para pruebas de verificación:

```yaml
if [ "$STEAM_PREVIEW_INPUT" = 'true' ]; then
  cmd+=(--preview)
fi
```

Esto permite cargar primero en la plataforma Steam para verificación, y después de confirmar que no hay problemas, cambiar a la rama oficial. Una capa adicional de verificación siempre es buena.

### Manejo de Errores y Registros

Los scripts incluyen manejo completo de errores y registro de logs:
- Verificar la existencia de GitHub Release
- Verificar archivos de metadatos requeridos
- Asegurar que el contenido de la plataforma exista
- Generar informe de resumen de GitHub Actions

Esta información es muy valiosa para depuración y auditoría, después de todo, cuando surgen problemas, tener una pista es mejor que estar completamente perdido.

### Gestión de Artefactos

El flujo de trabajo genera dos tipos de artefactos:
- `portable-steam-release-preparation-{tag}`: Metadatos de preparación de publicación
- `portable-steam-build-metadata-{tag}`: Metadatos de compilación de Steam

Estos artefactos se pueden usar para auditoría y depuración posteriores, se recomienda establecer el tiempo de retención en 30 días. Después de todo, tampoco ocupan mucho espacio, mantenerlos no estorba.
