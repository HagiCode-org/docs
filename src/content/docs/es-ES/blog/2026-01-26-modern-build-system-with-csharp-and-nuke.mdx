## Adiós al infierno de los scripts: Por qué elegimos C# para construir un sistema de compilación moderno

> Descubre cómo el proyecto HagiCode utiliza Nuke para implementar un flujo de automatización de compilación con seguridad de tipos, multiplataforma y altamente escalable, resolviendo completamente los problemas de mantenimiento de los scripts de compilación tradicionales.

## Antecedentes

En el largo viaje del desarrollo de software, la palabra "compilación" a menudo genera sentimientos encontrados. Lo que nos gusta es que con un solo clic, el código se convierte en producto, ese es el momento más fascinante para un programador; lo que nos disgusta es mantener ese lío de scripts de compilación, simplemente es una pesadilla.

En muchos proyectos, estamos acostumbrados a escribir scripts en Python, o usar archivos de configuración XML (imagina ese miedo dominado por `<property>`). Pero a medida que aumenta la complejidad del proyecto, especialmente en proyectos como HagiCode que implican desarrollo frontend y backend, multiplataforma y múltiples idiomas, los métodos de compilación tradicionales comienzan a quedar cortos. La lógica de los scripts está dispersa, falta verificación de tipos, el soporte del IDE es débil... Estos problemas son como pequeños agujeros que hacen que el equipo de desarrollo tropiece de vez en cuando.

Para resolver estos puntos problemáticos, en el proyecto HagiCode decidimos introducir **Nuke** — un sistema de compilación moderno basado en C#. No es solo una herramienta, es más bien una re conceptualización del proceso de compilación. Hoy, hablemos de por qué lo elegimos y cómo hizo que nuestra experiencia de desarrollo "despegara".

## Análisis central: ¿Por qué Nuke?

Puede que tengas dudas: "Vaya, hay tantos sistemas de compilación, como Make, Gradle, ¿incluso directamente usar scripts de Shell no funciona? ¿Por qué tiene que ser uno en C#?"

En realidad, esa es una buena pregunta. El encanto central de Nuke radica en que lleva las características del lenguaje de programación que más conocemos al mundo de los scripts de compilación.

### 1. Modularizar el proceso de compilación: El arte de los Target

La filosofía de diseño de Nuke es muy clara: **Todo es un objetivo**.

En los scripts tradicionales, podríamos escribir cientos de líneas de código que se ejecutan linealmente, con lógica compleja. En Nuke, descomponemos el proceso de compilación en `Target` (objetivos) independientes. Cada objetivo solo es responsable de una cosa, por ejemplo:

- `Clean`: Limpiar el directorio de salida
- `Restore`: Restaurar paquetes de dependencias
- `Compile`: Compilar código
- `Test`: Ejecutar pruebas unitarias

Este diseño se ajusta muy bien al principio de responsabilidad única. Como construir bloques, podemos combinar estos Target a voluntad. Más importante aún, Nuke nos permite definir las relaciones de dependencia entre los Target. Por ejemplo, si quieres `Test`, el sistema verificará automáticamente si primero ejecutaste `Compile`; si quieres `Compile`, naturalmente primero debes hacer `Restore`.

Este gráfico de dependencias no solo hace que la lógica sea más clara, sino que también mejora enormemente la eficiencia de ejecución, Nuke analizará automáticamente la ruta de ejecución óptima.

### 2. Seguridad de tipos: Adiós a la pesadilla de los errores ortográficos

Los amigos que han usado Python para escribir scripts de compilación seguramente han encontrado esta situación incómoda: el script se ejecutó durante cinco minutos, y finalmente informó un error diciendo que `Confi.guration` está mal escrito, o pasaste una cadena a un parámetro que debería ser un número.

La mayor ventaja de usar C# para escribir scripts de compilación es la **seguridad de tipos**. Esto significa:

- **Verificación en tiempo de compilación**: Mientras escribes código, el IDE te dirá dónde está el error, sin tener que esperar hasta el tiempo de ejecución para descubrirlo.
- **Refactorización sin preocupaciones**: Si quieres cambiar un nombre de variable o método, la función de refactorización del IDE lo resuelve con un clic, sin tener que buscar y reemplazar globalmente con miedo.
- **Sugerencias inteligentes**: El poderoso IntelliSense completará automáticamente el código, no necesitas revisar la documentación para recordar esas APIs oscuras.

### 3. Multiplataforma: Una experiencia de compilación unificada

Antes escribíamos `.bat` en Windows, `.sh` en Linux, y para que ambos fueran compatibles, también teníamos que escribir un script en Python. Ahora, dondequiera que .NET Core (ahora .NET 5+) pueda ejecutarse, Nuke puede ejecutarse.

Esto significa que ya sea que los miembros del equipo usen Windows, Linux o macOS, ya sea que usen Visual Studio, VS Code o Rider, todos ejecutan la misma lógica. Esto elimina enormemente los problemas causados por diferencias ambientales como "funciona en mi máquina".

### 4. Gestión de parámetros y configuración

Nuke proporciona un mecanismo de análisis de parámetros muy elegante. No necesitas analizar manualmente `string[] args`, solo necesitas definir una propiedad, agregar la característica `[Parameter]`, y Nuke manejará automáticamente el mapeo de argumentos de línea de comandos y archivos de configuración.

Por ejemplo, podemos definir fácilmente la configuración de compilación:

```csharp
[Parameter("Configuration to build - Default is 'Debug'")]
readonly Configuration BuildConfiguration = IsLocalBuild ? Configuration.Debug : Configuration.Release;

Target Compile => _ => _
    .DependsOn(Restore)
    .Executes(() =>
    {
        // 在这里使用 BuildConfiguration，它是类型安全的
        DotNetBuild(s => s
            .SetConfiguration(BuildConfiguration)
            .SetProjectFile(SolutionFile));
    });
```

Esta forma de escribir es tanto intuitiva como propensa a errores.

## Guía práctica: Cómo implementarlo en un proyecto

Hablar es barato, veamos cómo implementamos específicamente esta solución en el proyecto HagiCode.

### 1. Planificar la estructura del proyecto

No queríamos que los scripts de compilación contaminaran el directorio raíz del proyecto, ni queríamos que la estructura del directorio fuera tan profunda como algunos proyectos de Java. Entonces, colocamos todos los archivos de compilación relacionados con Nuke en la carpeta `nukeBuild/`.

Las ventajas de hacer esto son:
- El directorio raíz del proyecto se mantiene limpio.
- La lógica de compilación es cohesiva y fácil de administrar.
- Cuando se unen nuevos miembros, pueden ver de inmediato "oh, esta es la lógica relacionada con la compilación".

### 2. Diseñar una cadena de dependencia de Target clara

Al diseñar Target, seguimos un principio: **Atomicidad + Flujo de dependencia**.

Cada Target debe ser lo suficientemente pequeño y hacer solo una cosa. Por ejemplo, `Clean` solo se encarga de eliminar archivos, no hagas empaquetado en él.

El flujo de dependencia recomendado es más o menos así:

`Clean` -> `Restore` -> `Compile` -> `Test` -> `Pack`

Por supuesto, esto no es absoluto. Por ejemplo, si solo quieres ejecutar una prueba y no quieres empaquetar, Nuke te permite ejecutar directamente `nuke Test`, y manejará automáticamente los pasos previos de Restore y Compile.

### 3. Manejo de errores y registro completo

Lo que más temen los scripts de compilación? Es que la información del error no sea clara. Por ejemplo, la compilación falló, el registro solo muestra "Error: 1", lo cual es muy frustrante.

En Nuke, dado que podemos usar directamente el mecanismo de manejo de excepciones de C#, podemos capturar y reportar errores con mucha precisión.

```csharp
Target Publish => _ => _
    .DependsOn(Test)
    .Executes(() =>
    {
        try 
        {
            // 尝试发布到 NuGet
            DotNetNuGetPush(s => s
                .SetTargetPath(ArtifactPath)
                .SetSource("https://api.nuget.org/v3/index.json")
                .SetApiKey(ApiKey));
        }
        catch (Exception ex)
        {
            Log.Error($"发布失败了，兄弟们检查一下 Key 对不对: {ex.Message}");
            throw; // 确保构建进程以非零退出码结束
        }
    });
```

### 4. Pruebas de integración para garantizar la calidad

Los scripts de compilación también son código y también necesitan pruebas. Nuke nos permite escribir pruebas para el proceso de compilación, asegurando que cuando modifiquemos la lógica de compilación, no rompamos el proceso de publicación existente. Esto es especialmente importante en las canalizaciones de integración continua (CI).

## Resumen

Al introducir Nuke, el flujo de compilación de HagiCode se volvió sin precedentes. No es solo un reemplazo de herramienta, es una mejora del pensamiento de ingeniería.

**¿Qué obtuvimos?**
- **Mantenibilidad**: Código como configuración, lógica clara, los novatos también pueden comenzar rápidamente.
- **Estabilidad**: La verificación de tipos fuerte reduce más del 90% de los errores de bajo nivel.
- **Consistencia**: Una experiencia unificada multiplataforma, eliminando las diferencias ambientales.

Si antes escribir scripts de compilación era "explorando en la oscuridad", usar Nuke es como "caminar de noche con las luces encendidas". Si estás cansado de mantener esos lenguajes de scripting difíciles de depurar, por qué no intentas mover la lógica de compilación al mundo de C#, quizás descubras que la compilación también puede ser tan elegante.
