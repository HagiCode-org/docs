## Guía de integración múltiple CLI de Hagicode con GLM-5.1

:::note[Nota sobre el contexto temporal]
Este artículo registra el diseño de integración múltiple CLI y el mapeo de modelos en el momento del lanzamiento. Para el alcance de soporte actual, consulte [Instalación de CLI de AI Agent](/related-software-installation/ai-agent-cli/) y [Guía de instalación](/installation); el `IFlowCli` mencionado en el artículo solo representa la configuración histórica de ese momento y no pertenece al alcance de soporte activo actual.
:::

## Antecedentes

En el proyecto Hagicode, los usuarios pueden elegir múltiples herramientas CLI para impulsar el asistente de programación IA, incluyendo Claude Code CLI, GitHub Copilot, OpenCode CLI, Codebuddy CLI, Hermes CLI, entre otros. Estas herramientas CLI son herramientas de programación IA genéricas, pero a través de la capa de abstracción de Hagicode, pueden conectarse flexiblemente con diferentes proveedores de modelos IA.

Zhipu AI (ZAI) proporciona una interfaz compatible con la API de Anthropic Claude, lo que permite que estas herramientas CLI usen directamente los modelos de la serie GLM domésticos. Entre ellos, GLM-5.1 es el modelo de lenguaje grande más reciente lanzado por Zhipu, con mejoras significativas respecto a GLM-5.0.

## Arquitectura de abstracción CLI de Hagicode

Hagicode define 11 tipos de proveedores CLI a través del enum `AIProviderType`, cubriendo las herramientas CLI de programación IA principales:

```csharp
public enum AIProviderType
{
    ClaudeCodeCli = 0,    // Claude Code CLI
    CodexCli = 1,          // GitHub Copilot Codex
    GitHubCopilot = 2,     // GitHub Copilot
    CodebuddyCli = 3,     // Codebuddy CLI
    OpenCodeCli = 4,      // OpenCode CLI
    IFlowCli = 5,         // IFlow CLI
    HermesCli = 6,        // Hermes CLI
    QoderCli = 7,         // Qoder CLI
    KiroCli = 8,          // Kiro CLI
    KimiCli = 9,          // Kimi CLI
    GeminiCli = 10,       // Gemini CLI
}
```

Cada CLI tiene una configuración de parámetros de modelo correspondiente, soportando los parámetros `model` y `reasoning`:

```csharp
private static readonly IReadOnlyDictionary<AIProviderType, IReadOnlyList<string>> ManagedModelParameterKeysByProvider =

        [AIProviderType.KiroCli] = ["model", "reasoning"],
        [AIProviderType.GeminiCli] = ["model"],  // Gemini no soporta el parámetro reasoning
        // ...

```

## Sistema de soporte de modelos GLM

El Catálogo de Profesiones Secundarias de Hagicode define el soporte completo de la serie de modelos GLM:

| Model ID | Name | Default Reasoning | Compatible CLI Families |
|----------|------|-------------------|-------------------------|
| `glm-4.7` | GLM 4.7 | high | claude, codebuddy, hermes, qoder, kiro |
| `glm-5` | GLM 5 | high | claude, codebuddy, hermes, qoder, kiro |
| `glm-5-turbo` | GLM 5 Turbo | high | claude, codebuddy, hermes, qoder, kiro |
| `glm-5.0` | GLM 5.0 (Legacy) | high | claude, codebuddy, hermes, qoder, kiro |
| `glm-5.1` | GLM 5.1 | high | claude, codebuddy, hermes, qoder, kiro |

## Diferencias clave entre GLM-5.1 y GLM-5.0

A partir de la implementación de `AcpSessionModelBootstrapper.cs`, podemos ver claramente las diferencias entre GLM-5.1 y GLM-5.0:

### Implementación independiente de GLM-5.1

## Uso de GLM-5.1 en diferentes CLI

### 1. Claude Code CLI + GLM-5.1

Claude Code CLI es uno de los CLI principales de Hagicode, configurado a través del sistema de configuración Hero:

```json
{
  "primaryProfessionId": "profession-claude-code",
  "secondaryProfessionId": "secondary-glm-5-1",
  "model": "glm-5.1",
  "reasoning": "high"
}
```

Configuración correspondiente de `HeroEquipmentCatalogItem`:

```typescript
{
  id: 'secondary-glm-5-1',
  name: 'GLM 5.1',
  family: 'anthropic',
  kind: 'model',
  primaryFamily: 'claude',
  compatiblePrimaryFamilies: ['claude', 'codebuddy', 'hermes', 'qoder', 'kiro'],
  defaultParameters: {
    model: 'glm-5.1',
    reasoning: 'high'
  }
}
```

### 2. OpenCode CLI + GLM-5.1

OpenCode CLI es el CLI más flexible, soportando el formato `provider/model` para especificar cualquier modelo:

**Método 1: Usar prefijo de provider ZAI**
```json
{
  "primaryProfessionId": "profession-opencode",
  "model": "zai/glm-5.1",
  "reasoning": "high"
}
```

**Método 2: Usar directamente el ID del modelo**
```json
{
  "model": "glm-5.1"
}
```

**Método 3: Interfaz de configuración frontal**

En `HeroModelEquipmentForm.tsx`, OpenCode CLI tiene un mensaje de marcador de posición especial:

```tsx
const OPEN_CODE_MODEL_PLACEHOLDER = 'myprovider/glm-4.7';

const modelPlaceholder = primaryProviderType === PCode_Models_AIProviderType.OPEN_CODE_CLI
  ? OPEN_CODE_MODEL_PLACEHOLDER
  : 'gpt-5.4';
```

Los usuarios pueden ingresar:
```
zai/glm-5.1
glm-5.1
```

**Lógica de resolución de modelos de OpenCode CLI**:

```csharp
internal OpenCodeModelSelection? ResolveModelSelection(string? rawModel)
{
    var normalized = NormalizeOptionalValue(rawModel);
    if (normalized == null) return null;

    var slashIndex = normalized.IndexOf('/');
    if (slashIndex < 0)
    {
        // Sin barra: usar directamente el ID del modelo
        return new OpenCodeModelSelection {
            ProviderId = string.Empty,
            ModelId = normalized,
        };
    }

    // Con barra: analizar formato provider/model
    var providerId = normalized[..slashIndex].Trim();
    var modelId = normalized[(slashIndex + 1)..].Trim();

    return new OpenCodeModelSelection {
        ProviderId = providerId,
        ModelId = modelId,
    };
}
```

### 3. Codebuddy CLI + GLM-5.1

Codebuddy CLI tiene una lógica especial de procesamiento legacy:

```json
{
  "primaryProfessionId": "profession-codebuddy",
  "model": "glm-5.1",
  "reasoning": "high"
}
```

**Nota**: Codebuddy tiene una reserva especial para GLM-5.0, sin pasar por normalización legacy:

```csharp
return !string.Equals(providerName, "CodebuddyCli", StringComparison.OrdinalIgnoreCase)
       && string.Equals(normalizedModel, LegacyGlm5TurboModelValue, StringComparison.OrdinalIgnoreCase)
    ? Glm5TurboModelValue
    : normalizedModel;
// Cuando es CodebuddyCli, glm-5.0 no se normaliza a glm-5-turbo
```

## Ventajas de mejora de GLM-5.1

Comparado con GLM-5.0, GLM-5.1 tiene las siguientes mejoras significativas:

### 1. Mejor capacidad de razonamiento

Según la información oficial de lanzamiento de Zhipu, las mejoras de GLM-5.1 incluyen:

- **Mayor capacidad de comprensión de código**: análisis más preciso de estructuras de código complejas
- **Comprensión de contexto más larga**: soporta contextos de conversación más largos
- **Mejora en llamadas a herramientas**: mayor tasa de éxito en llamadas a herramientas MCP
- **Estabilidad de salida**: reduce la aleatoriedad y las alucinaciones

### 2. Compatibilidad completa con múltiples CLI

GLM-5.1 cubre todos los CLI principales soportados por Hagicode:

```
compatiblePrimaryFamilies: [
  "claude",      // Claude Code CLI
  "codebuddy",   // Codebuddy CLI
  "hermes",      // Hermes CLI
  "qoder",       // Qoder CLI
  "kiro"         // Kiro CLI
]
```

## Consideraciones importantes

### 1. Configuración de API Key

Asegúrese de configurar correctamente la variable de entorno `ANTHROPIC_AUTH_TOKEN`, que es la credencial necesaria para que todos los CLI se conecten a los modelos.

### 2. Disponibilidad de modelos

GLM-5.1 debe estar activado en el proveedor de modelos correspondiente:
- Plataforma Zhipu AI ZAI: soporte predeterminado
- Alibaba Cloud DashScope: puede requerir solicitud separada

### 3. Formato de OpenCode CLI

Al usar el formato `provider/model`, asegúrese de que el ID del provider sea correcto:
- Zhipu AI: `zai` o `zhipuai`
- Alibaba Cloud: `aliyun` o `dashscope`

### 4. Parámetro Reasoning

- Se recomienda usar el nivel `high` para obtener los mejores resultados de generación de código
- Gemini CLI no soporta el parámetro reasoning e ignorará automáticamente esta configuración

## Resumen

Hagicode, a través de una capa de abstracción unificada, implementa la integración flexible de GLM-5.1 con múltiples CLI. Los desarrolladores pueden elegir la herramienta CLI apropiada según sus preferencias y escenarios de uso, y usar el último modelo GLM-5.1 a través de una configuración simple.

GLM-5.1, como la versión más reciente del modelo de Zhipu, tiene mejoras obvias respecto a GLM-5.0:
- Identificador de versión independiente, sin carga legacy
- Mayor capacidad de razonamiento y comprensión de código
- Amplia compatibilidad con múltiples CLI
- Configuración flexible de niveles de razonamiento

A través de la configuración correcta de variables de entorno y equipamiento Hero, los jugadores pueden aprovechar al máximo el poder de GLM-5.1 en diferentes entornos CLI.
