## Creando un grupo de aventuras de IA: configuración práctica de colaboración multiagente de HagiCode

> En el desarrollo de software moderno, un solo agente de IA ya no puede satisfacer necesidades complejas. ¿Cómo hacer que varios asistentes de IA de diferentes empresas trabajen juntos en el mismo proyecto? Este artículo compartirá la solución de configuración de colaboración multiagente que el proyecto HagiCode ha explorado en el desarrollo real.

## Antecedentes

Creo que muchos desarrolladores han experimentado esto: el proyecto introdujo asistentes de IA para ayudar con la programación y la eficiencia definitivamente mejoró. Pero a medida que las necesidades se volvieron cada vez más complejas, un agente de IA comenzó a no ser suficiente: querías que manejara simultáneamente múltiples tareas como revisión de código, generación de documentación, pruebas unitarias, etc., y el resultado a menudo era cuidar un aspecto y perder otro, con calidad de producción desigual.

Lo que es aún más doloroso es que, cuando intentas introducir varios asistentes de IA, el problema se vuelve más complejo. Cada agente tiene su propia forma de configuración, interfaz API y lógica de ejecución, e incluso pueden entrar en conflicto entre sí. Es como un equipo de fútbol, cada jugador es muy capaz, pero nadie sabe cómo cooperar, y el resultado es un desastre.

El proyecto HagiCode también enfrentó el mismo problema durante su desarrollo. Como un proyecto complejo que involucra la extensión VSCode front-end, servicios de IA back-end y clientes de escritorio multiplataforma, necesitamos conectar simultáneamente varios asistentes de IA de diferentes empresas: Claude Code, Codex, CodeBuddy, iFlow, etc. Cómo hacer que coexistan armoniosamente en el mismo proyecto y desplieguen sus respectivas fortalezas se convirtió en un problema clave que debía resolverse.

En realidad, esto es aceptable, después de todo, ¿quién quiere lidiar todos los días con un grupo de IA que pelea entre sí?

La solución compartida en este artículo es precisamente la práctica de configuración de colaboración multiagente que el proyecto HagiCode ha explorado a través de pruebas y optimizaciones reales. Si también estás sufriendo por la colaboración de múltiples asistentes de IA, creo que este artículo te dará algunas inspiraciones. Quizás, después de todo, la situación de cada persona es diferente.

## Sobre HagiCode

[HagiCode](https://hagicode.com) es un proyecto de asistente de código de IA que utiliza el modo de "grupo de aventuras" con múltiples motores de IA trabajando juntos. Dirección del proyecto: [github.com/HagiCode-org/site](https://github.com/HagiCode-org/site).

La solución de configuración multiagente compartida en este artículo es precisamente una de las tecnologías centrales que permiten a HagiCode mantener un desarrollo eficiente en proyectos complejos. No hay nada especial, solo se trata de convertir un grupo de IA en un grupo de aventuras que pueda cooperar.

## Diseño de arquitectura multiagente de HagiCode

### De "luchar solo" a "trabajo en equipo"

En las primeras etapas del proyecto HagiCode, también intentamos usar solo un agente de IA para manejar todas las tareas. Pronto descubrimos que este enfoque tiene cuellos de botella obvios: diferentes tareas necesitan diferentes enfoques de capacidades, algunas tareas requieren capacidades de comprensión de contexto más fuertes, otras requieren capacidades de modificación de código más precisas. Es difícil que un agente sobresalga en todos los aspectos.

Esto nos hizo darnos cuenta de que debíamos hacer que varios agentes trabajaran juntos. Pero el problema era ¿cómo hacer que los productos de IA de diferentes empresas coexistan pacíficamente en el mismo proyecto? Necesitábamos resolver varios problemas centrales:

1. **Complejidad de gestión de configuración**: cada agente tiene diferentes formas de configuración, interfaces API y modos de ejecución
2. **Unificación del protocolo de comunicación**: se necesita una forma estandarizada de intercambiar datos entre diferentes agentes
3. **Coordinación de la división de tareas**: cómo asignar tareas razonablemente para que cada agente despliegue sus fortalezas

Con estos problemas, comenzamos a diseñar la arquitectura multiagente de HagiCode. En realidad no es tan compleja, solo se trata de entenderlo.

### Vista general de la arquitectura general

Después de múltiples iteraciones, la arquitectura que finalmente determinamos es esta:

```
┌─────────────────────────────────────────────────────────────────┐
│                    AIProviderFactory                             │
│  (Patrón de fábrica para gestionar uniformemente todos los AI Provider) │
├─────────────────────────────────────────────────────────────────┤
│  ClaudeCodeCli  │  CodexCli  │  CodebuddyCli  │  IFlowCli    │
│  (Anthropic)   │  (OpenAI)  │  (Zhipu GLM)     │  (Zhipu)      │
└─────────────────────────────────────────────────────────────────┘
```

La idea central es: a través de una interfaz Provider unificada, diferentes agentes de IA pueden ser gestionados por el mismo conjunto de código. Al mismo tiempo, se utiliza el patrón de fábrica para crear y configurar dinámicamente estos Providers, asegurando la escalabilidad y flexibilidad del sistema.

Es como la división del trabajo en la vida, cada persona tiene su propio rol, solo aquí esta división del trabajo se convierte en arquitectura de código.

### Tipos de Agent y división de responsabilidades

Según la experiencia real de uso del proyecto HagiCode, asignamos diferentes responsabilidades a cada Agente:

| Agente | Proveedor | Modelo | Uso principal |
|-------|--------|------|---------|
| ClaudeCodeCli | Anthropic | glm-5-turbo | Generar soluciones técnicas y propuestas |
| CodexCli | OpenAI/Zed | gpt-5.4 | Ejecutar modificaciones de código precisas |
| CodebuddyCli | Zhipu | glm-4.7 | Optimizar descripciones de propuestas y documentación |
| IFlowCli | Zhipu | glm-4.7 | Archivar propuestas e registros históricos |
| OpenCodeCli | - | - | Edición de código general |
| GitHubCopilot | Microsoft | - | Programación asistida y finalización de código |

La lógica detrás de esta división es: cada Agente tiene su propio campo de especialización. Claude Code sobresale en comprender y analizar requisitos complejos, por lo que se le encarga del diseño de soluciones en la etapa inicial; Codex es más preciso en modificaciones de código, adecuado para manejar tareas de implementación específicas; CodeBuddy tiene una alta relación calidad-precio, usado para optimizar la documentación es más que apropiado.

Después de todo, lo que te conviene es lo mejor, todos los caminos llevan a Roma, solo que algunos caminos son más fáciles de recorrer y otros son un poco más tortuosos.

## Mecanismo de configuración central

### Diseño de interfaz Provider unificada

Para permitir que diferentes agentes de IA sean gestionados uniformemente, primero debemos definir un conjunto de interfaces unificadas. En HagiCode se definió esta interfaz:

```csharp
public interface IAIProvider
{
    // Interfaz Provider unificada
    Task<IAIProvider?> GetProviderAsync(AIProviderType providerType);
    Task<IAIProvider?> GetProviderAsync(string providerName, CancellationToken cancellationToken);
}
```

Esta interfaz parece simple, pero es la piedra angular de todo el sistema multiagente. A través de una interfaz unificada, podemos ignorar qué producto de IA de qué compañía está en la parte inferior, y todos se llaman de la misma manera.

En realidad, esto solo está simplificando cosas complejas, después de todo, simple es hermoso.

### Implementación del patrón de fábrica Provider

Con una interfaz unificada, lo siguiente es cómo crear instancias de estos Providers. HagiCode utiliza el patrón de fábrica:

```csharp
private IAIProvider? CreateProvider(AIProviderType providerType, ProviderConfiguration config)
{
    return providerType switch
    {
        AIProviderType.ClaudeCodeCli =>
            ActivatorUtilities.CreateInstance<ClaudeCodeCliProvider>(_serviceProvider, Options.Create(config)),
        AIProviderType.CodebuddyCli =>
            ActivatorUtilities.CreateInstance<CodebuddyCliProvider>(_serviceProvider, Options.Create(config)),
        AIProviderType.CodexCli =>
            ActivatorUtilities.CreateInstance<CodexCliProvider>(_serviceProvider, Options.Create(config)),
        AIProviderType.IFlowCli =>
            ActivatorUtilities.CreateInstance<IFlowCliProvider>(_serviceProvider, Options.Create(config)),
        _ => null
    };
}
```

Aquí se usa `ActivatorUtilities.CreateInstance` de inyección de dependencias, que puede crear dinámicamente instancias Provider en tiempo de ejecución e inyectar automáticamente dependencias. El beneficio de este diseño es: al agregar un nuevo tipo de Agente, solo necesita agregar la clase Provider correspondiente, y luego agregar una rama case en el método de fábrica, sin necesidad de modificar el código existente.

Esto es aceptable, después de todo, ¿quién quiere cambiar un montón de código antiguo cada vez que agrega una nueva función?

### Análisis de configuración dinámica

Para hacer que la configuración sea más flexible, también implementamos un mecanismo de mapeo de tipos:

```csharp
public static AIProviderTypeExtensions
{
    private static readonly Dictionary<string, AIProviderType> _typeMap = new(
        StringComparer.OrdinalIgnoreCase)
    {
        ["ClaudeCodeCli"] = AIProviderType.ClaudeCodeCli,
        ["CodebuddyCli"] = AIProviderType.CodebuddyCli,
        ["CodexCli"] = AIProviderType.CodexCli,
        ["IFlowCli"] = AIProviderType.IFlowCli,
        // ... más mapeos de tipos
    };
}
```

El rol de esta tabla de mapeo es convertir nombres de Provider en forma de cadena a tipos enumerados. De esta manera, los archivos de configuración pueden usar nombres de cadena intuitivos, mientras que el código interno usa enumeraciones seguras de tipos para procesar.

Después de todo, la configuración es algo, cuanto más intuitivo mejor, ¿quién quiere memorizar un montón de código complejo?

### Ejemplo de archivo de configuración

En uso real, solo necesita configurar en `appsettings.json`:

```yaml
AI:
  Providers:
    Providers:
      ClaudeCodeCli:
        Enabled: true
        Model: glm-5-turbo
        WorkingDirectory: /path/to/project
      CodebuddyCli:
        Enabled: true
        Model: glm-4.7
      CodexCli:
        Enabled: true
        Model: gpt-5.4
      IFlowCli:
        Enabled: true
        Model: glm-4.7
```

Cada Provider puede configurar independientemente interruptores, versiones de modelo, directorios de trabajo y otros parámetros. Este diseño garantiza flexibilidad y facilita la gestión y el mantenimiento.

En realidad, los archivos de configuración son como las opciones en la vida, puedes elegir activar o desactivar ciertas funciones, solo que las elecciones en el código son más fáciles de lamentar.

## Flujo de tareas del grupo de aventuras

### El arte de la división de tareas

Con una arquitectura técnica unificada, lo siguiente es cómo hacer que varios agentes trabajen juntos. HagiCode diseñó un mecanismo de flujo de tareas para que diferentes agentes manelen tareas en diferentes etapas:

```
Creación de propuesta (usuario)
    │
    ▼
[Claude Code] ──generar propuesta──▶ Documento de propuesta
    │                        │
    │                        ▼
    │               [Codebuddy] ──optimizar descripción──▶ Propuesta optimizada
    │                        │
    │                        ▼
    │               [Codex] ──ejecutar modificación──▶ Cambios de código
    │                        │
    │                        ▼
    └───────────────▶ [iFlow] ──archivar──▶ Registros históricos
```

El beneficio de esta división es: cada Agente solo necesita concentrarse en las tareas en las que es bueno, sin necesidad de "saber todo". Claude Code es responsable de generar propuestas de la nada, Codebuddy es responsable de hacer que las descripciones de las propuestas sean más claras, Codex es responsable de convertir las propuestas en cambios de código reales, e iFlow es responsable de archivar y guardar estos cambios.

En realidad, esto es como el trabajo en equipo en la vida, cada persona tiene su propio rol, y juntos pueden completar algo grande. Solo que aquí los miembros del equipo son IA.

### Puntos clave de práctica

En la operación real, resumimos las siguientes experiencias:

**1. La estrategia de selección de Agent es muy importante**

No asignes tareas al azar, sino asigna según las fortalezas de cada Agente:
- Generación de propuestas: usa Claude Code, porque tiene capacidades de comprensión de contexto más fuertes
- Ejecución de código: usa Codex, porque es más preciso en modificaciones de código
- Optimización de propuestas: usa CodeBuddy, porque tiene una alta relación calidad-precio
- Archivo y almacenamiento: usa iFlow, porque es estable y confiable

Después de todo, dejar que la persona adecuada haga lo adecuado es una verdad inmutable.

**2. El aislamiento de configuración asegura la estabilidad**

La configuración de cada Agente se gestiona de forma independiente, admite anulación de variables de entorno, y los directorios de trabajo también son independientes entre sí. De esta manera, un error de configuración de un Agente no afectará a otros Agentes.

Es como los límites en la vida, cada persona tiene su propio espacio, y solo sin interferencias mutuas pueden coexistir en armonía.

**3. Mecanismo de manejo de errores**

El fallo de un solo Agente no debería afectar el flujo general. Implementamos una estrategia de degradación: cuando un Agente falla en la ejecución, el sistema puede cambiar automáticamente a una solución alternativa, o omitir directamente ese paso y continuar ejecutando tareas posteriores. Al mismo tiempo, el registro completo también facilita la investigación de problemas después del hecho.

Después de todo, nadie puede garantizar que nunca habrá errores, la clave es cómo manejar los errores. Es como la vida, siempre encontrarás contratiempos, lo importante es cómo superarlos.

**4. Monitoreo y observabilidad**

A través del protocolo ACP (nuestro protocolo de comunicación personalizado, basado en JSON-RPC 2.0), se puede rastrear el estado de ejecución de cada Agente. El aislamiento de sesiones asegura la seguridad de concurrencia, y el caché dinámico optimiza el rendimiento.

Después de todo, lo que no se ve es lo más propenso a problemas, tener algo de monitoreo es mejor que estar a ciegas.

## Efectos reales y beneficios

Después de adoptar esta configuración de colaboración multiagente, la eficiencia de desarrollo del proyecto HagiCode mejoró significativamente. Específicamente:

1. **Capacidad de procesamiento de tareas duplicada**: anteriormente un Agente necesitaba manejar múltiples tipos de tareas simultáneamente, ahora puede procesarlas en paralelo, el rendimiento se ha más que duplicado
2. **Calidad de salida más estable**: cada Agente solo se concentra en las tareas en las que es bueno, la consistencia y calidad de los resultados son más altas
3. **Costos de mantenimiento reducidos**: la interfaz y gestión de configuración unificadas hacen que todo el sistema sea más fácil de mantener y expandir
4. **Agregar nuevos Agent es simple**: si quieres conectar un nuevo producto de IA, solo necesitas implementar la interfaz y agregar configuración, sin necesidad de modificar la lógica central

Esta solución no solo resolvió los problemas del propio HagiCode, sino que también demostró que la colaboración multiagente es de hecho una opción de arquitectura viable.

En realidad, el efecto es bastante obvio, solo que el proceso es un poco tedioso.

## Resumen

Este artículo compartió la experiencia práctica del proyecto HagiCode en configuración de colaboración multiagente. Los puntos centrales incluyen:

1. **Interfaz estandarizada**: a través de `IAIProvider` unificar el comportamiento de diferentes Agent, permitiendo que el código ignore qué producto de qué compañía está en la parte inferior
2. **Patrón de fábrica**: usar `ActivatorUtilities.CreateInstance` para crear dinámicamente instancias Provider, admitiendo configuración en tiempo de ejecución e inyección de dependencias
3. **Protocolo unificado**: el protocolo ACP implementa comunicación estandarizada entre Agent, basado en el mecanismo de comunicación bidireccional JSON-RPC 2.0
4. **División de flujo de tareas**: asignar razonablemente tareas a diferentes Agent, dejándolos desplegar sus fortalezas, en lugar de intentar hacer que un Agente haga todo

Este diseño no solo resolvió el problema de "múltiples Agent peleando", sino que también a través del mecanismo de flujo de tareas del grupo de aventuras, logró la automatización y especialización del flujo de desarrollo.
