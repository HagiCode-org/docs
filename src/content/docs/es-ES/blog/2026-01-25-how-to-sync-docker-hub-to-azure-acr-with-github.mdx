## Implementación de la sincronización automatizada de imágenes de Docker Hub a Azure ACR

> Este artículo presenta cómo usar GitHub Actions y la herramienta image-syncer para lograr la sincronización automatizada de imágenes de Docker Hub a Azure Container Registry, resolviendo el problema de la velocidad lenta de acceso a Docker Hub en China y algunas regiones de Azure, mejorando la disponibilidad de imágenes y la eficiencia de implementación en el entorno de Azure.

## Antecedentes/Introducción

El proyecto HagiCode utiliza imágenes de Docker como componentes principales de tiempo de ejecución, con las imágenes principales alojadas en Docker Hub. Con el desarrollo del proyecto y el aumento de las necesidades de implementación en entornos de Azure, encontramos los siguientes puntos problemáticos:
- Velocidad lenta de extracción de imágenes, acceso restringido a Docker Hub en China y algunas regiones de Azure
- Riesgo de fallo de un solo punto al depender de una sola fuente de imágenes
- El uso de Azure Container Registry en entornos de Azure proporciona un mejor rendimiento de red y una experiencia de integración

Para resolver estos problemas, necesitamos establecer un mecanismo de sincronización de imágenes automatizado que sincronice periódicamente las imágenes de Docker Hub a Azure ACR, asegurando que los usuarios puedan obtener velocidades de extracción de imágenes más rápidas y una mayor disponibilidad en el entorno de Azure.

## Comparación de soluciones técnicas

Al desarrollar la solución, comparamos múltiples soluciones técnicas:

### 1. image-syncer (elección final)
- Sincronización incremental: solo sincroniza las capas de imágenes modificadas, reduciendo significativamente la transferencia de red
- Reanudación de interrupciones: puede reanudar la sincronización después de una interrupción de red
- Control de concurrencia: admite la configuración del número de subprocesos concurrentes, mejorando la eficiencia de sincronización de imágenes grandes
- Manejo de errores completo: mecanismo de reintegro incorporado para fallos (predeterminado 3 veces)
- Despliegue ligero: único archivo binario, sin dependencias
- Soporte multi-registro: compatible con Docker Hub, Azure ACR, Harbor, etc.

### 2. Docker CLI
- No admite sincronización incremental: cada vez necesita extraer el contenido completo de la imagen
- Baja eficiencia: gran cantidad de transferencia de red, largo tiempo
- Fácil de usar: usa comandos familiares docker pull/push

### 3. Azure CLI
- Alta complejidad: requiere configurar la autenticación de Azure CLI
- Limitaciones funcionales: la función az acr import es relativamente simple
- Integración nativa: buena integración con servicios de Azure

## Decisiones de diseño de arquitectura

### Decisión 1: Configurar la frecuencia de sincronización a diario a las 00:00 UTC
- Equilibra la frescura de las imágenes y el consumo de recursos
- Evita horas pico de negocio, reduciendo el impacto en otras operaciones
- Las imágenes de Docker Hub generalmente se actualizan después de la construcción diaria

### Decisión 2: Sincronizar todas las etiquetas de imágenes
- Mantener una consistencia completa con Docker Hub
- Proporcionar a los usuarios opciones flexibles de versiones
- Simplificar la lógica de sincronización, evitando reglas complejas de filtrado de etiquetas

### Decisión 3: Usar GitHub Secrets para almacenar información de autenticación
- Soporte nativo de GitHub Actions, alta seguridad
- Configuración simple, fácil de gestionar y mantener
- Admite control de acceso a nivel de repositorio

## Evaluación y mitigación de riesgos

### Riesgo 1: Filtración de información de autenticación de Azure ACR
- Usar almacenamiento cifrado de GitHub Secrets
- Rotación periódica de contraseñas de ACR
- Limitar los permisos del usuario de ACR a solo empujar
- Monitorear los registros de acceso de ACR

### Riesgo 2: Fallo de sincronización que cause inconsistencia de imágenes
- Mecanismo de sincronización incremental incorporado en image-syncer
- Reintento automático de fallos (predeterminado 3 veces)
- Registros de error detallados y notificaciones de fallo
- Función de reanudación de interrupciones

### Riesgo 3: Consumo excesivo de recursos
- Sincronización incremental reduce la transferencia de red
- Número de subprocesos concurrentes configurables (actualmente configurado en 10)
- Monitorear la cantidad y el tamaño de las imágenes sincronizadas
- Ejecutar sincronización en horas no pico

## Solución central

Adoptamos la solución automatizada de GitHub Actions + image-syncer para lograr la sincronización de imágenes de Docker Hub a Azure ACR.

## Pasos de implementación

### 1. Fase de preparación
- Crear o confirmar Azure Container Registry en Azure Portal
- Crear clave de acceso de ACR (nombre de usuario y contraseña)
- Confirmar permisos de acceso al repositorio de imágenes de Docker Hub

### 2. Configurar GitHub Secrets
Agrega los siguientes Secrets en la configuración del repositorio de GitHub:
- AZURE_ACR_USERNAME: Nombre de usuario de Azure ACR
- AZURE_ACR_PASSWORD: Contraseña de Azure ACR

### 3. Crear flujo de trabajo de GitHub Actions
Configura el flujo de trabajo en .github/workflows/sync-docker-acr.yml:
- Activación programada: todos los días a las 00:00 UTC
- Activación manual: soporta workflow_dispatch
- Activación adicional: activar al empujar a la rama publish (para sincronización rápida)

### 4. Flujo de ejecución del flujo de trabajo

| Orden | Participante | Acción | Descripción |
| --- | --- | --- | --- |
| 1 | GitHub Actions | Activar flujo de trabajo | Activado por programación, manual o empuje a rama `publish` |
| 2 | GitHub Actions → image-syncer | Descargar y ejecutar herramienta de sincronización | Entrar en fase de sincronización real |
| 3 | image-syncer → Docker Hub | Obtener manifest y lista de etiquetas de imagen | Leer metadatos del repositorio fuente |
| 4 | image-syncer → Azure ACR | Obtener información de imágenes existentes en repositorio destino | Determinar estado actual del lado destino |
| 5 | image-syncer | Comparar diferencias entre fuente y destino | Identificar capas de imagen que necesitan sincronización |
| 6 | image-syncer → Docker Hub | Extraer capas de imagen modificadas | Solo transmitir contenido que necesita actualización |
| 7 | image-syncer → Azure ACR | Empujar capas de imagen modificadas | Completar sincronización incremental |
| 8 | image-syncer → GitHub Actions | Devolver información estadística de sincronización | Incluyendo resultados, diferencias e información de errores |
| 9 | GitHub Actions | Registrar registros y subir artifact | Facilitar auditoría y solución de problemas posteriores |

## Implementación del flujo de trabajo de GitHub Actions

A continuación se muestra la configuración del flujo de trabajo real en ejecución (.github/workflows/sync-docker-acr.yml):

```yaml
name: Sync Docker Image to Azure ACR

on:
  schedule:
    - cron: "0 0 * * *" # Todos los días a las 00:00 UTC
  workflow_dispatch: # Activación manual
  push:
    branches: [publish]

permissions:
  contents: read

jobs:
  sync:
    runs-on: ubuntu-latest

    steps:
      - name: Checkout code
        uses: actions/checkout@v4

      - name: Download image-syncer
        run: |
          # Descargar archivo binario image-syncer
          wget https://github.com/AliyunContainerService/image-syncer/releases/download/v1.5.5/image-syncer-v1.5.5-linux-amd64.tar.gz
          tar -zxvf image-syncer-v1.5.5-linux-amd64.tar.gz
          chmod +x image-syncer

      - name: Create auth config
        run: |
          # Generar archivo de configuración de autenticación (formato YAML)
          cat > auth.yaml <<EOF
          hagicode.azurecr.io:
            username: "${{ secrets.AZURE_ACR_USERNAME }}"
            password: "${{ secrets.AZURE_ACR_PASSWORD }}"
          EOF

      - name: Create images config
        run: |
          # Generar archivo de configuración de sincronización de imágenes (formato YAML)
          cat > images.yaml <<EOF
          docker.io/newbe36524/hagicode: hagicode.azurecr.io/hagicode
          EOF

      - name: Run image-syncer
        run: |
          # Ejecutar sincronización (usando parámetros --auth y --images de la nueva versión)
          ./image-syncer --auth=./auth.yaml --images=./images.yaml --proc=10 --retries=3

      - name: Upload logs
        if: always()
        uses: actions/upload-artifact@v4
        with:
          name: sync-logs
          path: image-syncer-*.log
          retention-days: 7
```

## Descripción de la configuración

### 1. Condiciones de activación
- Activación programada: cron: "0 0 * * *" - Ejecutar todos los días a las 00:00 UTC
- Activación manual: workflow_dispatch - Permite a los usuarios ejecutar manualmente en la interfaz de usuario de GitHub
- Activación por empuje: push: branches: [publish] - Activar al empujar a la rama de publicación (para sincronización rápida)

### 2. Configuración de autenticación (auth.yaml)
```yaml
hagicode.azurecr.io:
  username: "${{ secrets.AZURE_ACR_USERNAME }}"
  password: "${{ secrets.AZURE_ACR_PASSWORD }}"
```

### 3. Configuración de sincronización de imágenes
```yaml
docker.io/newbe36524/hagicode: hagicode.azurecr.io/hagicode
```
Esta configuración indica sincronizar todas las etiquetas de docker.io/newbe36524/hagicode a hagicode.azurecr.io/hagicode

### 4. Parámetros de image-syncer
- --auth=./auth.yaml: Ruta del archivo de configuración de autenticación
- --images=./images.yaml: Ruta del archivo de configuración de sincronización de imágenes
- --proc=10: Número de subprocesos concurrentes es 10
- --retries=3: Reintentar 3 veces en caso de fallo

## Precauciones y mejores prácticas

### 1. Recomendaciones de seguridad
- Rotar periódicamente la contraseña de Azure ACR (se recomienda cada 90 días)
- Usar una cuenta de servicio dedicada de ACR, limitando los permisos a solo empujar
- Monitorear los registros de acceso de ACR, detectando accesos anormales a tiempo
- No generar información de autenticación en los registros
- No enviar información de autenticación al repositorio de código

### 2. Optimización del rendimiento
- Ajustar el parámetro --proc: ajusta el número concurrente según el ancho de banda de la red (se recomienda 5-20)
- Monitorear el tiempo de sincronización: si el tiempo de sincronización es demasiado largo, considera reducir el número concurrente
- Limpiar periódicamente los registros: establece un retention-days razonable (actualmente 7 días)

### 3. Solución de problemas

#### Problema 1: Fallo de autenticación
```
Error: failed to authenticate to hagicode.azurecr.io
```
Solución:
1. Verificar que GitHub Secrets esté configurado correctamente
2. Verificar que la contraseña de Azure ACR no haya expirado
3. Confirmar que los permisos de la cuenta de servicio de ACR sean correctos

#### Problema 2: Tiempo de espera de red
```
Error: timeout waiting for response
```
Solución:
1. Verificar la conexión de red
2. Reducir el número de subprocesos concurrentes (parámetro --proc)
3. Esperar a que se recupere la red y volver a activar el flujo de trabajo

#### Problema 3: Sincronización de imágenes incompleta
```
Warning: some tags failed to sync
```
Solución:
1. Verificar los registros de sincronización, identificar las etiquetas que fallaron
2. Activar manualmente el flujo de trabajo para resincronizar
3. Verificar que la imagen fuente de Docker Hub sea normal

### 4. Monitoreo y alertas
- Verificar periódicamente la página Actions, confirmar el estado de ejecución del flujo de trabajo
- Configurar notificaciones de GitHub para recibir alertas oportunas de fallos del flujo de trabajo
- Monitorear el uso de almacenamiento de Azure ACR
- Verificar periódicamente la coherencia de las etiquetas de las imágenes

## Preguntas frecuentes y soluciones

### P1: ¿Cómo sincronizar etiquetas específicas en lugar de todas las etiquetas?

Modifica el archivo de configuración images.yaml:
```yaml
# Solo sincronizar etiquetas latest y v1.0
docker.io/newbe36524/hagicode:latest: hagicode.azurecr.io/hagicode:latest
docker.io/newbe36524/hagicode:v1.0: hagicode.azurecr.io/hagicode:v1.0
```

### P2: ¿Cómo sincronizar múltiples repositorios de imágenes?

Agrega múltiples líneas de configuración en images.yaml:
```yaml
docker.io/newbe36524/hagicode: hagicode.azurecr.io/hagicode
docker.io/newbe36524/another-image: hagicode.azurecr.io/another-image
```

### P3: ¿Cómo reintentar después de un fallo de sincronización?
- Reintento automático: mecanismo de reintento incorporado de image-syncer (predeterminado 3 veces)
- Reintento manual: haz clic en Re-run all jobs en la página de GitHub Actions

### P4: ¿Cómo ver el progreso detallado de la sincronización?
- Ver registros en tiempo real en la página Actions
- Descargar el artifact sync-logs para ver el archivo de registro completo
- El archivo de registro contiene el estado de sincronización y la velocidad de transmisión de cada etiqueta

### P5: ¿Cuánto tiempo toma la sincronización?
- Sincronización completa inicial: según el tamaño de la imagen, generalmente se necesita 10-30 minutos
- Sincronización incremental: si el cambio de imagen es pequeño, generalmente 2-5 minutos
- El tiempo depende del ancho de banda de la red, el tamaño de la imagen y la configuración concurrente
