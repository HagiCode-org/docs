## Implementar la sincronización automática de imágenes de Docker Hub a Azure ACR

> Este artículo describe cómo utilizar GitHub Actions y la herramienta image-syncer para implementar la sincronización automática de imágenes de Docker Hub a Azure Container Registry, resolviendo el problema de la lentitud de acceso a Docker Hub desde China continental y ciertas regiones de Azure, mejorando la disponibilidad de las imágenes y la eficiencia de despliegue en entornos Azure.

## Contexto/Introducción

El proyecto HagiCode utiliza imágenes Docker como componente principal de tiempo de ejecución, con las imágenes principales alojadas en Docker Hub. Con el crecimiento del proyecto y la creciente necesidad de despliegues en entornos Azure, nos encontramos con los siguientes problemas:
- Velocidad de extracción de imágenes lenta, con acceso restringido a Docker Hub desde China continental y ciertas regiones de Azure
- Riesgo de punto único de fallo al depender de una única fuente de imágenes
- Mejor rendimiento de red e experiencia de integración al utilizar Azure Container Registry en entornos Azure

Para resolver estos problemas, necesitamos establecer un mecanismo de sincronización automática de imágenes que sincronice periódicamente las imágenes de Docker Hub a Azure ACR, asegurando que los usuarios puedan obtener velocidades de extracción más rápidas y mayor disponibilidad en entornos Azure.

## Comparación de soluciones técnicas

Al elaborar la solución, comparamos múltiples enfoques técnicos:

### 1. image-syncer (elección final)
- Sincronización incremental: solo sincroniza las capas de imágenes modificadas, reduciendo significativamente la transferencia de red
- Reanudación de interrupciones: puede reanudar la sincronización después de una interrupción de red
- Control de concurrencia: permite configurar el número de hilos concurrentes, mejorando la eficiencia de sincronización de imágenes grandes
- Manejo de errores robusto: mecanismo integrado de reintento en caso de fallo (3 intentos por defecto)
- Despliegue ligero: archivo binario único, sin dependencias
- Soporte multi-registro: compatible con Docker Hub, Azure ACR, Harbor, etc.

### 2. Docker CLI
- No soporta sincronización incremental: requiere extraer el contenido completo de la imagen cada vez
- Baja eficiencia: gran volumen de transferencia de red, tiempos prolongados
- Fácil de usar: utiliza los comandos familiares docker pull/push

### 3. Azure CLI
- Alta complejidad: requiere configurar la autenticación de Azure CLI
- Funcionalidad limitada: la función az acr import es relativamente básica
- Integración nativa: buena integración con los servicios de Azure

## Decisiones de diseño de arquitectura

### Decisión 1: frecuencia de sincronización configurada a diario a las 00:00 UTC
- Equilibrio entre frescura de las imágenes y consumo de recursos
- Evita las horas punta de actividad, reduciendo el impacto en otras operaciones
- Las imágenes de Docker Hub suelen actualizarse después de las compilaciones diarias

### Decisión 2: sincronizar todas las etiquetas de las imágenes
- Mantener la consistencia completa con Docker Hub
- Ofrecer a los usuarios una selección de versiones flexible
- Simplificar la lógica de sincronización, evitando reglas complejas de filtrado de etiquetas

### Decisión 3: utilizar GitHub Secrets para almacenar información de autenticación
- Soporte nativo de GitHub Actions, alta seguridad
- Configuración sencilla, fácil de gestionar y mantener
- Soporte de control de acceso a nivel de repositorio

## Evaluación y mitigación de riesgos

### Riesgo 1: filtración de credenciales de Azure ACR
- Almacenamiento cifrado mediante GitHub Secrets
- Rotación periódica de la contraseña de ACR
- Limitación de los permisos del usuario de ACR a solo push
- Monitoreo de los registros de acceso de ACR

### Riesgo 2: fallo de sincronización causando inconsistencia de imágenes
- image-syncer tiene un mecanismo integrado de sincronización incremental
- Reintento automático en caso de fallo (3 intentos por defecto)
- Registros de errores detallados y notificaciones de fallo
- Función de reanudación de interrupciones

### Riesgo 3: consumo excesivo de recursos
- La sincronización incremental reduce la transferencia de red
- Número configurable de hilos concurrentes (configurado actualmente en 10)
- Monitoreo de la cantidad y tamaño de las imágenes sincronizadas
- Ejecución de la sincronización en horas no punta

## Solución central

Adoptamos la solución automatizada de GitHub Actions + image-syncer para implementar la sincronización de imágenes desde Docker Hub a Azure ACR.

## Pasos de implementación

### 1. Fase de preparación
- Crear o confirmar Azure Container Registry en Azure Portal
- Crear las claves de acceso de ACR (nombre de usuario y contraseña)
- Confirmar los permisos de acceso al repositorio de imágenes de Docker Hub

### 2. Configurar GitHub Secrets
Añadir los siguientes Secrets en la configuración del repositorio de GitHub:
- AZURE_ACR_USERNAME: nombre de usuario de Azure ACR
- AZURE_ACR_PASSWORD: contraseña de Azure ACR

### 3. Crear el flujo de trabajo de GitHub Actions
Configurar el flujo de trabajo en .github/workflows/sync-docker-acr.yml:
- Activación programada: todos los días a las 00:00 UTC
- Activación manual: soporte para workflow_dispatch
- Activación adicional: al hacer push a la rama publish (para sincronización rápida)

### 4. Flujo de ejecución del flujo de trabajo

| Paso | Participante | Acción | Descripción |
| --- | --- | --- | --- |
| 1 | GitHub Actions | Activar el flujo de trabajo | Activado por programación, manualmente o por push a la rama `publish` |
| 2 | GitHub Actions → image-syncer | Descargar y ejecutar la herramienta de sincronización | Entrar en la fase de sincronización real |
| 3 | image-syncer → Docker Hub | Obtener el manifest y la lista de etiquetas de la imagen | Leer los metadatos del repositorio fuente |
| 4 | image-syncer → Azure ACR | Obtener información de las imágenes existentes en el repositorio destino | Determinar el estado actual del lado destino |
| 5 | image-syncer | Comparar diferencias entre fuente y destino | Identificar las capas de imágenes que necesitan sincronizarse |
| 6 | image-syncer → Docker Hub | Extraer las capas de imágenes modificadas | Transferir solo el contenido que necesita actualizarse |
| 7 | image-syncer → Azure ACR | Enviar las capas de imágenes modificadas | Completar la sincronización incremental |
| 8 | image-syncer → GitHub Actions | Devolver estadísticas de sincronización | Incluye resultados, diferencias e información de errores |
| 9 | GitHub Actions | Registrar logs y subir artefacto | Facilitar la auditoría y resolución de problemas posterior |

## Implementación del flujo de trabajo de GitHub Actions

A continuación se muestra la configuración del flujo de trabajo en ejecución real (.github/workflows/sync-docker-acr.yml):

```yaml
name: Sync Docker Image to Azure ACR

on:
  schedule:
    - cron: "0 0 * * *" # Todos los días a las 00:00 UTC
  workflow_dispatch: # Activación manual
  push:
    branches: [publish]

permissions:
  contents: read

jobs:
  sync:
    runs-on: ubuntu-latest

    steps:
      - name: Checkout code
        uses: actions/checkout@v4

      - name: Download image-syncer
        run: |
          # Descargar el binario de image-syncer
          wget https://github.com/AliyunContainerService/image-syncer/releases/download/v1.5.5/image-syncer-v1.5.5-linux-amd64.tar.gz
          tar -zxvf image-syncer-v1.5.5-linux-amd64.tar.gz
          chmod +x image-syncer

      - name: Create auth config
        run: |
          # Generar archivo de configuración de autenticación (formato YAML)
          cat > auth.yaml <<EOF
          hagicode.azurecr.io:
            username: "${{ secrets.AZURE_ACR_USERNAME }}"
            password: "${{ secrets.AZURE_ACR_PASSWORD }}"
          EOF

      - name: Create images config
        run: |
          # Generar archivo de configuración de sincronización de imágenes (formato YAML)
          cat > images.yaml <<EOF
          docker.io/newbe36524/hagicode: hagicode.azurecr.io/hagicode
          EOF

      - name: Run image-syncer
        run: |
          # Ejecutar la sincronización (usando los parámetros --auth e --images de la nueva versión)
          ./image-syncer --auth=./auth.yaml --images=./images.yaml --proc=10 --retries=3

      - name: Upload logs
        if: always()
        uses: actions/upload-artifact@v4
        with:
          name: sync-logs
          path: image-syncer-*.log
          retention-days: 7
```

## Explicación de la configuración

### 1. Condiciones de activación
- Activación programada: cron: "0 0 * * *" - se ejecuta todos los días a las 00:00 UTC
- Activación manual: workflow_dispatch - permite al usuario ejecutarlo manualmente desde la interfaz de GitHub
- Activación por push: push: branches: [publish] - se activa al hacer push a la rama de publicación (para sincronización rápida)

### 2. Configuración de autenticación (auth.yaml)
```yaml
hagicode.azurecr.io:
  username: "${{ secrets.AZURE_ACR_USERNAME }}"
  password: "${{ secrets.AZURE_ACR_PASSWORD }}"
```

### 3. Configuración de sincronización de imágenes
```yaml
docker.io/newbe36524/hagicode: hagicode.azurecr.io/hagicode
```
Esta configuración indica que se sincronizarán todas las etiquetas de docker.io/newbe36524/hagicode a hagicode.azurecr.io/hagicode

### 4. Parámetros de image-syncer
- --auth=./auth.yaml: ruta del archivo de configuración de autenticación
- --images=./images.yaml: ruta del archivo de configuración de sincronización de imágenes
- --proc=10: número de hilos concurrentes establecido en 10
- --retries=3: 3 reintentos en caso de fallo

## Notas y mejores prácticas

### 1. Recomendaciones de seguridad
- Rotar periódicamente la contraseña de Azure ACR (se recomienda cada 90 días)
- Utilizar una cuenta de servicio dedicada de ACR, con permisos limitados a solo push
- Monitorear los registros de acceso de ACR para detectar accesos anómalos de manera oportuna
- No imprimir información de autenticación en los registros
- No enviar información de autenticación al repositorio de código

### 2. Optimización del rendimiento
- Ajustar el parámetro --proc: ajustar la concurrencia según el ancho de banda de red (se recomienda 5-20)
- Monitorear el tiempo de sincronización: si es demasiado largo, considerar reducir la concurrencia
- Limpieza periódica de registros: configurar un retention-days razonable (actualmente 7 días)

### 3. Resolución de problemas

#### Problema 1: fallo de autenticación
```
Error: failed to authenticate to hagicode.azurecr.io
```
Solución:
1. Verificar que los GitHub Secrets estén configurados correctamente
2. Validar que la contraseña de Azure ACR no haya expirado
3. Confirmar que los permisos de la cuenta de servicio de ACR sean correctos

#### Problema 2: tiempo de espera de red agotado
```
Error: timeout waiting for response
```
Solución:
1. Verificar la conexión de red
2. Reducir el número de hilos concurrentes (parámetro --proc)
3. Esperar a que se recupere la red y volver a activar el flujo de trabajo

#### Problema 3: sincronización de imágenes incompleta
```
Warning: some tags failed to sync
```
Solución:
1. Revisar los registros de sincronización para identificar las etiquetas que fallaron
2. Activar manualmente el flujo de trabajo para volver a sincronizar
3. Verificar que la imagen fuente de Docker Hub esté funcionando correctamente

### 4. Monitoreo y alertas
- Revisar periódicamente la página de Actions para confirmar el estado de ejecución del flujo de trabajo
- Configurar notificaciones de GitHub para recibir alertas oportunas de fallos del flujo de trabajo
- Monitorear el uso de almacenamiento de Azure ACR
- Verificar periódicamente la consistencia de las etiquetas de las imágenes

## Preguntas frecuentes y soluciones

### P1: ¿Cómo sincronizar etiquetas específicas en lugar de todas?

Modificar el archivo de configuración images.yaml:
```yaml
# Sincronizar solo las etiquetas latest y v1.0
docker.io/newbe36524/hagicode:latest: hagicode.azurecr.io/hagicode:latest
docker.io/newbe36524/hagicode:v1.0: hagicode.azurecr.io/hagicode:v1.0
```

### P2: ¿Cómo sincronizar múltiples repositorios de imágenes?

Añadir múltiples líneas de configuración en images.yaml:
```yaml
docker.io/newbe36524/hagicode: hagicode.azurecr.io/hagicode
docker.io/newbe36524/another-image: hagicode.azurecr.io/another-image
```

### P3: ¿Cómo reintentar después de un fallo de sincronización?
- Reintento automático: image-syncer tiene un mecanismo de reintento integrado (3 intentos por defecto)
- Reintento manual: hacer clic en Re-run all jobs en la página de GitHub Actions

### P4: ¿Cómo ver el progreso detallado de la sincronización?
- Ver los registros en tiempo real en la página de Actions
- Descargar el artefacto sync-logs para ver el archivo de registros completo
- El archivo de registros contiene el estado de sincronización y la velocidad de transferencia de cada etiqueta

### P5: ¿Cuánto tiempo tarda la sincronización?
- Primera sincronización completa: según el tamaño de las imágenes, generalmente 10-30 minutos
- Sincronización incremental: si los cambios en las imágenes son pequeños, generalmente 2-5 minutos
- El tiempo depende del ancho de banda de red, el tamaño de las imágenes y la configuración de concurrencia
