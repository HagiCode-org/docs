## Guía de implementación de la función de palabras clave para el reconocimiento de voz de Doubao

> Este artículo presentará en detalle cómo implementar la función de soporte de palabras clave para el reconocimiento de voz de Doubao en el proyecto HagiCode. A través de dos métodos: palabras clave personalizadas y tabla de palabras clave de la plataforma, se mejora significativamente la precisión de reconocimiento de vocabulario en campos específicos.

## Antecedentes

La tecnología de reconocimiento de voz ha evolucionado durante tantos años, pero en realidad ha habido un problema que ha preocupado a los desarrolladores. Aunque los modelos de reconocimiento de voz generales pueden cubrir el lenguaje cotidiano, para términos profesionales, nombres de productos, nombres de personas y otras palabras, la precisión de reconocimiento siempre deja un poco que desear. Piénselo, los asistentes de voz en el campo médico necesitan reconocer con precisión "hipertensión", "diabetes", "enfermedad coronaria"; los sistemas legales necesitan capturar con precisión "causa de la acción", "alegato", "carga de la prueba"—en estos escenarios, ¿cómo decirlo?, los modelos generales también hacen lo mejor que pueden.

En el proyecto HagiCode, también nos enfrentamos al mismo desafío. Como asistente de código de IA multifuncional, HagiCode necesita manejar varios escenarios de reconocimiento de voz de términos técnicos. Sin embargo, la API de reconocimiento de voz de Doubao por defecto no puede satisfacer completamente nuestros requisitos de precisión para términos profesionales. En realidad, no es que Doubao no sea lo suficientemente bueno, es que cada campo tiene su propio sistema de terminología. Después de una investigación y exploración técnica, descubrimos que la API de reconocimiento de voz de Doubao en realidad proporciona una función de soporte de palabras clave. Con una configuración simple, se puede mejorar significativamente la precisión de reconocimiento de palabras clave específicas. Es algo así como, le dices en qué palabras debe prestar atención y ella las escuchará con más atención.

Lo que este artículo va a compartir es la solución completa para implementar la función de palabras clave de reconocimiento de voz de Doubao en el proyecto HagiCode. Dos modos, palabras clave personalizadas y tabla de palabras clave de la plataforma, ambos se pueden usar y también se pueden combinar. A través de esta solución, los desarrolladores pueden configurar palabras clave de manera flexible según los escenarios comerciales, permitiendo que el sistema de reconocimiento de voz "reconozca" aquellas palabras profesionales, raras pero críticas.

## Implementación central

### Análisis de los dos modos de palabras clave

La API de reconocimiento de voz de Doubao nos proporciona dos métodos de configuración de palabras clave, cada uno con sus propios escenarios de aplicación y ventajas únicas.

El **modo de palabras clave personalizadas** nos permite pasar directamente texto de palabras clave a través del campo `corpus.context`. Este método es muy adecuado para escenarios que necesitan configurar rápidamente un pequeño número de palabras clave, por ejemplo, cuando temporalmente necesitas reconocer un nombre de producto o un nombre de persona. En la implementación de HagiCode, analizamos el texto de palabras clave de múltiples líneas ingresado por el usuario en una lista de cadenas, y luego lo formateamos como una matriz `context_data` según los requisitos de la API de Doubao. ¿Cómo decirlo?, este método es muy directo, como decirle a la otra parte "debes prestar atención a estas palabras", y luego ella las presta.

El **modo de tabla de palabras clave de la plataforma** hace referencia a la tabla de palabras clave preconfigurada en la plataforma de autoaprendizaje de Doubao a través del campo `corpus.boosting_table_id`. Este método es adecuado para escenarios que necesitan administrar una gran cantidad de palabras clave. Podemos crear y mantener tablas de palabras clave en la plataforma de autoaprendizaje de Doubao, y luego referenciarlas a través del ID. Para proyectos como HagiCode que necesitan actualizar y mantener continuamente términos profesionales, este modo proporciona mejor manejabilidad. Después de todo, cuando hay muchas palabras clave, encontrar un lugar para administrarlas unificadamente siempre es mejor que tener que ingresarlas manualmente cada vez.

Lo interesante es que estos dos modos también se pueden combinar. La API de Doubao admite incluir tanto palabras clave personalizadas como el ID de la tabla de palabras clave de la plataforma en la misma solicitud, controlando la estrategia de combinación a través del parámetro `combine_mode`. Esta flexibilidad permite que HagiCode pueda manejar varias necesidades complejas de reconocimiento de términos profesionales. Esto también es bastante bueno, a veces combinar varios métodos puede dar mejores resultados.

### Definición de tipos y validación en frontend

En la implementación de frontend de HagiCode, definimos un conjunto completo de tipos de configuración de palabras clave y lógica de validación. Primero está la parte de definición de tipos:

```typescript
export interface HotwordConfig {
  contextText: string;           // texto de palabras clave de múltiples líneas
  boostingTableId: string;      // ID de la tabla de palabras clave de la plataforma Doubao
  combineMode: boolean;          // si se usa en combinación
}
```

Esta interfaz simple contiene todos los elementos de configuración de la función de palabras clave. Entre ellos, `contextText` es la parte que el usuario percibe más directamente—permitimos que los usuarios ingresen una frase de palabra clave por línea, lo cual es muy intuitivo. Después de todo, pedir a los usuarios que ingresen una palabra por línea siempre es mejor que pedirles que entiendan reglas de configuración complejas.

A continuación está la implementación de la función de validación. Considerando las limitaciones de la API de Doubao, formulamos reglas de validación estrictas: el texto de palabras clave puede tener hasta 100 líneas, cada línea hasta 50 caracteres, hasta 5000 caracteres en total; boosting_table_id hasta 200 caracteres, solo permite letras, números, guiones bajos y guiones. Estas limitaciones no son imaginadas por nosotros, sino basadas en los requisitos reales de la documentación oficial de Doubao. Después de todo, las limitaciones de la API son las limitaciones de la API, no tenemos más remedio que cumplirlas.

```typescript
export function validateContextText(contextText: string): HotwordValidationResult {
  if (!contextText || contextText.trim().length === 0) {
    return { isValid: true, errors: [] };
  }

  const lines = contextText.split('\n').filter(line => line.trim().length > 0);
  const errors: string[] = [];

  if (lines.length > 100) {
    errors.push(`El número de líneas de palabras clave no puede superar las 100, actualmente es ${lines.length} líneas`);
  }

  const totalChars = contextText.length;
  if (totalChars > 5000) {
    errors.push(`El número total de caracteres de palabras clave no puede superar los 5000, actualmente es ${totalChars}`);
  }

  for (let i = 0; i < lines.length; i++) {
    if (lines[i].length > 50) {
      errors.push(`La línea ${i + 1} de palabras clave supera el límite de 50 caracteres`);
    }
  }

  return { isValid: errors.length === 0, errors };
}

export function validateBoostingTableId(boostingTableId: string): HotwordValidationResult {
  if (!boostingTableId || boostingTableId.trim().length === 0) {
    return { isValid: true, errors: [] };
  }

  const errors: string[] = [];

  if (boostingTableId.length > 200) {
    errors.push(`boosting_table_id no puede superar los 200 caracteres, actualmente es ${boostingTableId.length}`);
  }

  if (!/^[a-zA-Z0-9_-]+$/.test(boostingTableId)) {
    errors.push('boosting_table_id solo puede contener letras, números, guiones bajos y guiones');
  }

  return { isValid: errors.length === 0, errors };
}
```

Estas funciones de validación se ejecutarán inmediatamente cuando el usuario configure las palabras clave, asegurando que los problemas se descubran en la etapa más temprana. Para la experiencia del usuario, este retorno inmediato es muy importante. Después de todo, saber dónde está el error mientras se ingresa siempre es mejor que descubrirlo después de enviar.

### Persistencia de configuración en frontend

En la implementación de frontend de HagiCode, elegimos usar localStorage del navegador para almacenar la configuración de palabras clave. Hay varias consideraciones detrás de esta decisión de diseño: primero, la configuración de palabras clave es una configuración muy personalizada, y diferentes usuarios pueden tener diferentes necesidades de campos profesionales; segundo, este método simplifica la implementación de backend, sin necesidad de tablas de base de datos adicionales e interfaces API; finalmente, después de que el usuario configura una vez en el navegador, el uso posterior puede cargar automáticamente, lo cual es muy conveniente. En realidad, para ser honesto, es para ahorrar trabajo.

```typescript
const HOTWORD_STORAGE_KEYS = {
  contextText: 'hotword-context-text',
  boostingTableId: 'hotword-boosting-table-id',
  combineMode: 'hotword-combine-mode',
} as const;

export const DEFAULT_HOTWORD_CONFIG: HotwordConfig = {
  contextText: '',
  boostingTableId: '',
  combineMode: false,
};

// cargar configuración de palabras clave
export function loadHotwordConfig(): HotwordConfig {
  const contextText = localStorage.getItem(HOTWORD_STORAGE_KEYS.contextText) || '';
  const boostingTableId = localStorage.getItem(HOTWORD_STORAGE_KEYS.boostingTableId) || '';
  const combineMode = localStorage.getItem(HOTWORD_STORAGE_KEYS.combineMode) === 'true';

  return { contextText, boostingTableId, combineMode };
}

// guardar configuración de palabras clave
export function saveHotwordConfig(config: HotwordConfig): void {
  localStorage.setItem(HOTWORD_STORAGE_KEYS.contextText, config.contextText);
  localStorage.setItem(HOTWORD_STORAGE_KEYS.boostingTableId, config.boostingTableId);
  localStorage.setItem(HOTWORD_STORAGE_KEYS.combineMode, String(config.combineMode));
}
```

La lógica de este código es muy simple y clara. Al cargar la configuración, se lee de localStorage; al guardar la configuración, se escribe en localStorage. También proporcionamos una configuración predeterminada para asegurar que el sistema pueda funcionar normalmente sin ninguna configuración. Después de todo, siempre tiene que haber un valor predeterminado.

En la implementación de backend de HagiCode, necesitamos agregar propiedades relacionadas con palabras clave en la clase de configuración del SDK. Considerando las características del lenguaje C# y los hábitos de uso, adoptamos `List<string>` para almacenar el contexto de palabras clave personalizadas:

```csharp
public class DoubaoVoiceConfig

    /// <summary>
```

El diseño de esta clase de configuración sigue el estilo simple consistente de HagiCode. `HotwordContexts` es un tipo de lista que acepta valores NULL, `BoostingTableId` es una cadena que acepta valores NULL, de modo que cuando no hay ninguna configuración de palabras clave, estas propiedades no afectarán la solicitud de ninguna manera. Después de todo, cuando no se usan, no deberían existir, eso se llama limpio.

### Lógica de construcción de Payload

La construcción del Payload es el núcleo de toda la función de palabras clave. Cuando tenemos la configuración de palabras clave, necesitamos formatearla en la estructura JSON correcta según los requisitos de la API de Doubao. Este proceso ocurre antes de que el SDK envíe la solicitud:

```csharp
private void AddCorpusToRequest(Dictionary<string, object> request)
{
    var corpus = new Dictionary<string, object>();

    // agregar palabras clave personalizadas
    if (Config.HotwordContexts != null && Config.HotwordContexts.Count > 0)
    {
        corpus["context"] = new Dictionary<string, object>
        {
            ["context_type"] = "dialog_ctx",
            ["context_data"] = Config.HotwordContexts
                .Select(text => new Dictionary<string, object> { ["text"] = text })
                .ToList()
        };
    }

    // agregar ID de tabla de palabras clave de la plataforma
    if (!string.IsNullOrEmpty(Config.BoostingTableId))
    {
        corpus["boosting_table_id"] = Config.BoostingTableId;
    }

    // solo agregar corpus a la solicitud cuando no esté vacío
    if (corpus.Count > 0)
    {
        request["corpus"] = corpus;
    }
}
```

Este código muestra cómo construir dinámicamente el campo corpus según la configuración. El punto clave es: solo cuando realmente existe una configuración de palabras clave, agregaremos el campo corpus. Este diseño asegura la compatibilidad con versiones anteriores—cuando no se configuran palabras clave, la estructura de la solicitud es exactamente la misma que antes. Después de todo, la compatibilidad es importante, no podemos arruinar la lógica anterior solo por agregar una función.

### Transmisión de parámetros WebSocket

Entre frontend y backend, los parámetros de palabras clave se transmiten a través de mensajes de control WebSocket. El diseño de HagiCode es: el frontend carga la configuración de palabras clave de localStorage cuando comienza la grabación, y luego la envía al backend a través de mensajes WebSocket.

```typescript
const controlMessage = {
  type: 'control',
  payload: {
    command: 'StartRecognition',
    contextText: 'hipertensión\ndiabetes\nenfermedad coronaria',
    boosting_table_id: 'medical_table',
    combineMode: false
  }
};
```

Aquí hay un detalle que debe tenerse en cuenta: el frontend pasa texto de múltiples líneas (separado por caracteres de nueva línea), y el backend necesita analizarlo. El WebSocket Handler del backend analizará estos parámetros y los pasará al SDK:

```csharp
private async Task HandleControlMessageAsync(
    string connectionId,
    DoubaoSession session,
    ControlMessage message)
{
    if (message.Payload is SessionControlRequest controlRequest)
    {
        // analizar parámetros de palabras clave
        string? contextText = controlRequest.ContextText;
        string? boostingTableId = controlRequest.BoostingTableId;
        bool? combineMode = controlRequest.CombineMode;

        // analizar texto de múltiples líneas en lista de palabras clave
        if (!string.IsNullOrEmpty(contextText))
        {
            var hotwords = contextText
                .Split('\n', StringSplitOptions.RemoveEmptyEntries)
                .Select(s => s.Trim())
                .Where(s => s.Length > 0)
                .ToList();

            session.HotwordContexts = hotwords;
        }

        session.BoostingTableId = boostingTableId;
    }
}
```

A través de este diseño, la transmisión de la configuración de palabras clave del frontend al backend se vuelve clara y eficiente. En realidad, no hay nada especial, es solo pasar capa por capa.

## Guía práctica

### Configurar palabras clave personalizadas

En uso real, configurar palabras clave personalizadas es muy simple. Abra la página de configuración de reconocimiento de voz de HagiCode y busque el área "Configuración de palabras clave". En el cuadro de entrada "Texto de palabras clave personalizadas", ingrese una frase de palabra clave por línea.

Por ejemplo, si está desarrollando una aplicación relacionada con lo médico, puede configurarla así:

```
hipertensión
diabetes
enfermedad coronaria
angina de pecho
infarto de miocardio
insuficiencia cardíaca
```

Después de guardar la configuración, cada vez que comience el reconocimiento de voz, estas palabras clave se pasarán automáticamente a la API de Doubao. Las pruebas reales muestran que después de configurar las palabras clave, la precisión de reconocimiento de los términos profesionales relacionados ha mejorado significativamente. ¿Cómo decirlo?, el efecto está ahí, al menos es mucho mejor que antes.

### Configurar tabla de palabras clave de la plataforma

Si necesita administrar una gran cantidad de palabras clave, o las palabras clave necesitan actualizarse con frecuencia, entonces el modo de tabla de palabras clave de la plataforma es más adecuado para usted. Primero necesita crear una tabla de palabras clave en la plataforma de autoaprendizaje de Doubao, obtener el boosting_table_id generado, y luego ingresar este ID en la página de configuración de HagiCode.

La plataforma de autoaprendizaje de Doubao proporciona funciones como importación por lotes de palabras clave, administración categorizada, etc., lo cual es muy práctico para equipos que necesitan administrar una gran cantidad de términos profesionales. Al administrar palabras clave a través de la plataforma, se puede lograr el mantenimiento centralizado y la actualización unificada de palabras clave. Después de todo, cuando hay muchas palabras clave, tener un lugar para administrarlas unificadamente siempre es mejor que tener que ingresarlas manualmente cada vez.

### Uso del modo combinado

En algunos escenarios complejos, es posible que necesite usar tanto palabras clave personalizadas como la tabla de palabras clave de la plataforma al mismo tiempo. En este caso, solo necesita configurar ambos tipos de palabras clave en HagiCode y activar el interruptor "Modo combinado".

En el modo combinado, la API de Doubao considerará ambas fuentes de palabras clave al mismo tiempo, y la precisión de reconocimiento suele ser mayor que usar cualquiera de las dos por separado. Sin embargo, debe tenerse en cuenta que el modo combinado aumentará la complejidad de la solicitud, y se recomienda decidir si activarlo después de pruebas reales. Después de todo, ¿vale la pena realmente aumentar la complejidad?, todavía tiene que depender del efecto real.

## Precauciones

Al implementar y usar la función de palabras clave, hay varios puntos que necesitan atención especial.

Primero están las limitaciones de caracteres. La API de Doubao tiene limitaciones estrictas sobre las palabras clave, incluyendo el número de líneas, el número de caracteres por línea, el número total de caracteres, etc. Si excede las limitaciones, la API devolverá un error. En la implementación de frontend de HagiCode, verificamos en la etapa de entrada del usuario a través de funciones de validación, evitando enviar configuraciones inválidas al backend. Después de todo, descubrir problemas temprano siempre es mejor que esperar a que la API devuelva un error.

En segundo lugar está el formato de boosting_table_id. Este campo solo permite letras, números, guiones bajos y guiones, y no permite espacios u otros caracteres especiales. Al crear una tabla de palabras clave en la plataforma de autoaprendizaje de Doubao, debe prestar atención a las convenciones de nomenclatura. En realidad, no es de extrañar, los requisitos de formato de la API siempre son relativamente estrictos.

Tercero está la compatibilidad con versiones anteriores. Los parámetros de palabras clave son completamente opcionales. Cuando no se configuran palabras clave, el sistema funciona exactamente igual que antes. Este diseño asegura que los usuarios existentes no se vean afectados de ninguna manera, y también facilita la migración y actualización graduales. Después de todo, no podemos arruinar la lógica anterior solo por agregar una función.

Finalmente está el manejo de errores. Cuando la configuración de palabras clave no es válida, la API de Doubao devolverá la información de error correspondiente. La implementación de HagiCode registrará registros detallados para facilitar a los desarrolladores la solución de problemas. Al mismo tiempo, el frontend también mostrará errores de validación en la interfaz para ayudar a los usuarios a corregir la configuración. Cuando el manejo de errores se hace bien, la experiencia del usuario naturalmente también es buena.
