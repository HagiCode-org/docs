## VSCode y code-server: Selección de soluciones de edición de código en el navegador

> Al crear capacidades de edición de código en el navegador, los desarrolladores enfrentan una elección clave: usar la funcionalidad oficial `code serve-web` de VSCode o adoptar la solución `code-server` impulsada por la comunidad. Esta elección no solo afecta la arquitectura técnica, sino que también se relaciona con el cumplimiento de licencias y la flexibilidad de implementación.

## Antecedentes

En realidad, hacer una selección técnica es un poco como elegir un camino en la vida. Una vez que eliges un camino, debes seguir por él, y cuando quieras cambiar, el costo es enorme.

En la era de la programación asistida por IA, la capacidad de edición de código en el navegador se vuelve cada vez más importante. Los usuarios esperan que, después de que el asistente de IA analice el código, puedan abrir inmediatamente el editor en la misma sesión del navegador para realizar modificaciones, sin cambiar de aplicación. Esta experiencia sin problemas, ¿cómo decirlo?, es como cuando lo piensas, está ahí—pero a veces simplemente no está.

Sin embargo, al implementar esta funcionalidad, los desarrolladores enfrentan una elección técnica clave: usar la funcionalidad oficial `code serve-web` de VSCode o adoptar la solución `code-server` impulsada por la comunidad.

Ambas soluciones tienen sus ventajas y desventajas, y elegir mal puede traer muchos problemas más adelante. Por ejemplo, problemas de licencia—descubrir que la licencia no es compliant después de que el producto se lanza, entonces es demasiado tarde. Es un poco como el amor, si no lo piensas claro al principio, al final descubres que los valores de dos personas son fundamentalmente incompatibles, el precio pagado es grande. Otro ejemplo es la forma de implementación—una solución funciona bien en el entorno de desarrollo, pero una vez que está en contenedores, varios problemas surgen, nadie quiere caer en estos agujeros, después de todo, caer en muchos agujeros, la gente se vuelve insensible.

Al implementar capacidades de edición de código en el navegador, estudiamos profundamente ambas soluciones y finalmente admitimos ambas en el diseño de arquitectura, pero por defecto priorizamos code-server.

## Diferencias de licencia (lo más crítico)

Esta es la diferencia más fundamental entre las dos soluciones, y también es el primer factor que consideramos al hacer la selección. Después de todo, en la selección técnica, el primer paso es pensar claramente sobre los riesgos legales, de lo contrario, si algo sucede más tarde, ¿a quién culpar?

**code-server**
- Licencia MIT, completamente open source
- Mantenido por la empresa Coder.com, comunidad activa
- Se puede usar comercialmente, modificar y distribuir libremente
- Sin restricciones de escenarios de uso

**VSCode `code serve-web`**
- Pertenece al producto Microsoft VSCode
- Utiliza la licencia de Microsoft (la licencia de VS Code tiene restricciones de uso comercial)
- Dirigido principalmente a desarrolladores individuales
- El implementación a nivel empresarial puede necesitar consideraciones de licencia comercial adicionales

Desde la perspectiva de licencias, code-server es más amigable para proyectos comerciales. Esto debe pensarse claramente en la etapa de planificación del producto, de lo contrario, cuando la escala crezca y luego migrar, el costo será enorme. Después de todo, la migración, es fácil de decir, difícil de hacer, quien lo experimenta lo sabe.

## Diferencias en la forma de implementación

Después de resolver el problema de la licencia, lo siguiente es la forma de implementación. Esto afecta directamente tus costos operativos y diseño de arquitectura, y también afecta indirectamente tu estado de ánimo diario—cuanto más simple es la implementación, mejor es el estado de ánimo, todos entienden esta verdad.

**code-server**
- Aplicación Node.js independiente, se puede implementar por separado
- Soporta múltiples fuentes de tiempo de ejecución:
  - Especificar directamente la ruta del ejecutable
  - Búsqueda en PATH del sistema
  - Detección automática del entorno NVM Node.js 22.x
- No requiere instalar VSCode escritorio en el servidor
- Implementación en contenedores más simple

**VSCode `code serve-web`**
- Debe depender del VSCode CLI instalado localmente
- Necesita un comando `code` disponible en la máquina local
- El sistema filtrará el contenedor CLI de VS Code Remote
- Diseñado principalmente para escenarios de desarrollo local

code-server es más adecuado para escenarios de implementación en servidor/contenedor. Si su producto necesita ejecutarse en Docker, o el entorno del usuario no tiene VSCode, entonces elija code-server. Después de todo, simple es hermoso, lo complejo propensa a problemas, problemas deben arreglarse, después de arreglar puede introducir nuevos problemas, este ciclo interminable, ¿quién quiere experimentarlo?

## Diferencias en parámetros funcionales

Las dos soluciones también tienen algunas diferencias en los parámetros funcionales, aunque no grandes, pero pueden traer algunos problemas en el uso real. Estos detalles son como pequeñas fricciones en la vida, no muchas, pero muchas se vuelven molestas.

| Característica | code-server | code serve-web |
|------|-------------|----------------|
| Ruta base pública | `/` (configurable) | `/vscode-server` (fijo) |
| Método de autenticación | parámetro `--auth`, soporta múltiples modos | `--connection-token` / `--without-connection-token` |
| Directorio de datos | `{DataDir}/code-server` | `{DataDir}/vscode-serve-web` |
| Telemetría | Deshabilitado por defecto `--disable-telemetry` | Depende de la configuración de VSCode |
| Verificación de actualizaciones | Se puede deshabilitar `--disable-update-check` | Depende de la configuración de VSCode |

Estas diferencias deben tenerse en cuenta especiales al integrar. Por ejemplo, diferentes rutas de URL significan que el código frontend necesita un procesamiento específico. Los desarrolladores saben que este tipo de detalles consume más tiempo, pero no hay otra manera, si no lo haces, no funciona.

## Diferencias en la detección de disponibilidad

Al implementar la funcionalidad de cambio de editor, la lógica de detección de disponibilidad también es diferente. Esta diferencia es como la forma en que las personas se llevan, a algunas personas les gusta ser directas, a otras les gusta ser sutiles y eupemísticas.

**code-server**
- Siempre se devuelve como implementación visible
- Incluso si no está disponible, se mostrará, indicando el estado `install-required`
- Soporta detección automática del entorno NVM Node.js 22.x

**code serve-web**
- Solo es visible cuando se detecta el CLI `code` local
- Si no está disponible, el frontend ocultará automáticamente esta opción
- Depende del estado de instalación local de VSCode

Esta diferencia afecta directamente la experiencia del usuario. La forma de code-server es más transparente, el usuario sabe que hay esta opción, simplemente aún no está instalada; la forma de code serve-web es más oculta, el usuario puede no saber que hay esta elección. ¿Qué manera es mejor? Depende del posicionamiento del producto, después de todo, la experiencia del usuario, no hay respuesta estándar, solo adecuada o no.

## Arquitectura de doble implementación de HagiCode

Después de un análisis profundo, el proyecto HagiCode adoptó una arquitectura de doble implementación, admitiendo ambas soluciones a nivel de arquitectura. No es que tengamos dificultades para seleccionar tecnología, sino que realmente hay necesidades reales. Después de todo, en el mundo técnico, no hay una elección absolutamente correcta, solo la elección más adecuada para ti.

### Predeterminado a code-server

```csharp
// La implementación activa predeterminada es code-server
// Si se guarda una activeImplementation explícita, intenta esa implementación primero
// Si la implementación solicitada no está disponible, el resolutor intentará la otra implementación
// Si se produce una reserva, se devuelve fallbackReason
```

Elegimos code-server por defecto, considerando principalmente la licencia y la flexibilidad de implementación. Pero para usuarios con un entorno VSCode local, code serve-web también es una buena opción. Después de todo, dar a los usuarios más opciones es siempre bueno, obligar a otros a aceptar una sola solución, no puedo hacer eso.

### Selector de implementación

`CodeServerImplementationResolver` es responsable uniformemente de:
- Selección de implementación durante el precalentamiento de inicio
- Selección de implementación durante la lectura de estado
- Selección de implementación durante la apertura del proyecto
- Selección de implementación durante la apertura de Vault

Este diseño permite que el sistema responda flexiblemente a diferentes escenarios, y los usuarios pueden elegir la implementación más adecuada según su entorno. Diseño flexible, se gasta un poco más de tiempo al principio, pero se ahorra preocupaciones más tarde, después de todo, nadie quiere cambiar código en todas partes.

### Reglas de adaptación frontend

```typescript
// cuando localCodeAvailable=false, no mostrar code serve-web
// cuando localCodeAvailable=true, mostrar configuración de code serve-web
```

El frontend muestra automáticamente las opciones disponibles según el entorno, evitando que los usuarios vean funcionalidades inutilizables y se confundan. Los usuarios confundidos vienen a preguntarte, preguntan mucho también te molestas, te molestas quieres cambiar el código, cambiar el código puede introducir bugs, este ciclo vicioso, quién quiere experimentarlo que lo experimente.

## Guía práctica

Después de tanta teoría, ¿qué debemos prestar atención al implementar realmente? Después de todo, la teoría está bien, la implementación no funciona, la práctica es el único estándar para probar la verdad.

### Recomendaciones de implementación Docker

Para implementación en contenedores, code-server es la mejor opción:

```dockerfile
# Usar directamente la imagen oficial de code-server
FROM codercom/code-server:latest

# O instalar a través de npm
RUN npm install -g code-server
```

Una capa lo resuelve, no necesita instalar VSCode adicionalmente. Simple es bueno, lo complejo propensa a errores, esta verdad se aplica en todas partes.

### Ejemplos de configuración

**Configuración de code-server**
```json
{
  "vscodeServer": {
    "enabled": true,
    "activeImplementation": "code-server",
    "codeServer": {
      "host": "0.0.0.0",
      "port": 8080,
      "executablePath": "",
      "authMode": "none"
    }
  }
}
```

**Configuración de code serve-web**
```json
{
  "vscodeServer": {
    "enabled": true,
    "activeImplementation": "serve-web",
    "serveWeb": {
      "host": "0.0.0.0",
      "port": 8080,
      "executablePath": "/usr/local/bin/code"
    }
  }
}
```

Configurar, la primera vez es problemático, una vez configurado, más tarde es fácil. Como la vida, invertir un poco más al principio, los días posteriores serán más fáciles.

### Diferencias en la construcción de URL

**code-server**
```
http://localhost:8080/?folder=/path/to/project&vscode-lang=es-ES
```

**code serve-web**
```
http://localhost:8080/vscode-server/?folder=/path/to/project&tkn=xxx&vscode-lang=es-ES
```

Preste atención a las diferencias en rutas y parámetros, debe procesarse por separado al integrar. Los detalles determinan el éxito o el fracaso, esto no es falso en absoluto, falta un parámetro, puede no abrir la página.

### Cambiar implementación

El sistema soporta cambio en tiempo de ejecución, al cambiar automáticamente detiene la implementación anterior:

```csharp
// VsCodeServerManager maneja automáticamente la exclusión mutua
// Al cambiar activeImplementation, la implementación anterior no continuará en segundo plano
```

Este diseño permite a los usuarios probar diferentes implementaciones en cualquier momento y encontrar la solución más adecuada para ellos. Después de todo, lo adecuado es lo mejor, las sugerencias de otros son solo para referencia, finalmente tienes que probarlo tú mismo.

### Monitoreo de estado

```typescript
const { settings, runtime } = await getVsCodeServerSettings();

// runtime.activeImplementation: "code-server" | "serve-web"
// runtime.fallbackReason: razón del cambio
// runtime.status: "running" | "starting" | "stopped" | "unhealthy"
```

Estado visible, solo así se sabe. Los usuarios pueden localizar rápidamente si es un problema del servidor o un problema de su propia operación cuando encuentran problemas. Cuando no conoces el estado, las personas fácilmente entran en pánico, en pánico fácilmente hacen juicios incorrectos, una vez que se inicia esta cadena, no se puede detener.

## Resumen

| Dimensión de comparación | code-server | code serve-web | Recomendado |
|---------|-------------|----------------|------|
| Licencia | MIT (amigable comercial) | Microsoft (con restricciones) | code-server |
| Flexibilidad de implementación | Implementación independiente | Depende de VSCode local | code-server |
| Aplicabilidad de servidor | Diseñado para servidores | Dirigido principalmente al desarrollo local | code-server |
| Contenedorización | Soporte nativo | Necesita instalar VSCode | code-server |
| Integridad funcional | Cercano a la versión de escritorio | Versión oficial completa | code serve-web |
| Actividad de mantenimiento | Comunidad activa | Oficial Microsoft | Cada uno tiene ventajas |

**Estrategia recomendada**: Priorizar el uso de code-server, considerar code serve-web cuando se necesite funcionalidad oficial completa y se tenga un entorno VSCode local.
