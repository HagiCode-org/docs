## De 120 segundos a 45 segundos: guía práctica para optimizar el rendimiento de compilación de Vite con Worker Threads

> Al trabajar con proyectos frontend de gran tamaño, la compilación de código para producción suele hacer que uno espere ansiosamente. Este artículo comparte cómo reducir el tiempo de la etapa de ofuscación de código en la compilación de Vite de 120 segundos a 45 segundos mediante Node.js Worker Threads, con detalles de implementación y lecciones aprendidas en el proyecto HagiCode.

## Contexto

En nuestra práctica de ingeniería frontend, a medida que el proyecto crece, los problemas de eficiencia de compilación se vuelven cada vez más evidentes. Especialmente en el flujo de compilación para producción, para proteger la lógica del código fuente, normalmente introducimos herramientas de ofuscación de JavaScript (como `javascript-obfuscator`). Aunque este paso es necesario, requiere una gran cantidad de cálculo y consume enormemente los recursos de la CPU.

En las primeras fases de desarrollo del proyecto **HagiCode**, nos enfrentamos a un cuello de botella de rendimiento muy problemático: el tiempo de compilación para producción se deterioraba rápidamente a medida que aumentaba la cantidad de código.

**Los problemas específicos eran**:
- Ejecución serial en un solo hilo para las tareas de ofuscación, con un solo núcleo de CPU al máximo y los demás núcleos inactivos
- El tiempo de compilación se disparaba de los 30 segundos iniciales a 110-120 segundos
- El flujo de verificación de compilación después de cada modificación del código era extremadamente largo, ralentizando gravemente la eficiencia de iteración del desarrollo
- En la pipeline de CI/CD, la etapa de compilación se convertía en la parte más lenta

**¿Por qué HagiCode tenía esta necesidad?**
HagiCode es un asistente de codificación inteligente impulsado por IA, cuya arquitectura frontend incluye módulos complejos de lógica de negocio e interacción con IA. Para garantizar la seguridad del código central, habilitamos obligatoriamente la ofuscación de alta intensidad en las publicaciones de producción. Ante una espera de compilación de casi dos minutos, decidimos realizar una optimización profunda del rendimiento del sistema de compilación.

## Análisis: buscando el punto de ruptura del cuello de botella de rendimiento

Antes de abordar el problema de rendimiento, necesitamos primero organizar nuestras ideas y determinar la solución técnica óptima.

### Decisión central: ¿por qué elegir Worker Threads?

Para implementar computación paralela en el entorno de Node.js, existen principalmente tres enfoques:
1. **child_process**: crear procesos hijos independientes
2. **Web Workers**: principalmente para el lado del navegador
3. **worker_threads**: soporte nativo multihilo de Node.js

Después de un análisis comparativo, HagiCode finalmente eligió **Worker Threads**, por las siguientes razones:
- **Sobrecarga de serialización cero**: los Worker Threads se encuentran en el mismo proceso y pueden compartir memoria a través de `SharedArrayBuffer` o transferencia de control, evitando los altos costos de serialización de la comunicación entre procesos.
- **Soporte nativo**: integrado en Node.js 12+, sin necesidad de introducir dependencias pesadas adicionales.
- **Contexto unificado**: la depuración y el registro son más convenientes que con los procesos hijos.

### Granularidad de tareas: ¿cómo dividir las tareas de ofuscación?

Ofuscar un archivo JS Bundle gigante es difícil de paralelizar (porque el código tiene dependencias), pero los artefactos de compilación de Vite están compuestos por múltiples **Chunks**. Esto nos da un límite natural de paralelización:

- **Independencia**: los diferentes Chunks generados por el empaquetado de Vite tienen sus dependencias desacopladas y pueden procesarse de forma paralela de manera segura.
- **Granularidad adecuada**: normalmente un proyecto tendrá entre 10 y 30 Chunks, una cantidad ideal para la programación paralela.
- **Fácil integración**: el gancho `generateBundle` del plugin de Vite nos permite interceptar y procesar estos Chunks antes de que se generen los archivos.

### Diseño de arquitectura

Diseñamos un sistema de procesamiento paralelo con cuatro componentes centrales:

1. **Task Splitter**: recorre el objeto bundle de Vite, filtra los archivos que no necesitan ofuscación (como vendor), y genera una cola de tareas.
2. **Worker Pool Manager**: gestiona el ciclo de vida de los Workers, responsable de la distribución, recuperación y reintento de tareas.
3. **Progress Reporter**: emite informes de progreso de compilación en tiempo real, eliminando la ansiedad de espera del usuario.
4. **ObfuscationWorker**: el hilo de trabajo que ejecuta la lógica de ofuscación propiamente dicha.

## Solución: implementación práctica

Basándonos en el análisis anterior, comenzamos a implementar este sistema de ofuscación paralela.

### 1. Configurar el plugin de Vite

Primero, integramos el plugin de ofuscación paralela en `vite.config.ts`. La configuración es muy intuitiva, solo se necesita especificar el número de Workers y las reglas de ofuscación.

```typescript
import { defineConfig } from 'vite'
import { parallelJavascriptObfuscator } from './buildTools/plugin'

export default defineConfig(({ mode }) => {
  const isProduction = mode === 'production'
  
  return {
    build: {
      rollupOptions: {
        ...(isProduction
          ? {
              plugins: [
                parallelJavascriptObfuscator({
                  enabled: true,
                  // Ajustar automáticamente según el número de núcleos de CPU, se recomienda dejar un núcleo para el hilo principal
                  workerCount: 4, 
                  retryAttempts: 3,
                  fallbackToMainThread: true, // Degradación automática a un solo hilo en caso de error
                  // Filtrar los vendor chunks, normalmente no es necesario ofuscar bibliotecas de terceros
                  isVendorChunk: (fileName: string) => fileName.includes('vendor-'),
                  obfuscationConfig: {
                    compact: true,
                    controlFlowFlattening: true,
                    deadCodeInjection: true,
                    disableConsoleOutput: true,
                    // ... más opciones de ofuscación
                  },
                }),
              ],
            }
          : {}),
      },
    },
  }
})
```

### 2. Implementar la lógica del Worker

El Worker es la unidad de ejecución de tareas. Necesitamos definir bien las estructuras de datos de entrada y salida.

**Nota**: aunque el código aquí es simple, hay algunos puntos a tener en cuenta. Por ejemplo, la verificación de nulidad de `parentPort`, y el manejo de errores. En la práctica de HagiCode, descubrimos que algunas sintaxis especiales de ES6 pueden causar que el ofuscador se bloquee, por lo que añadimos protección con `try-catch`.

```typescript
import { parentPort } from 'worker_threads'
import javascriptObfuscator from 'javascript-obfuscator'

export interface ObfuscationTask {
  chunkId: string
  code: string
  config: any
}

export interface ObfuscationResult {
  chunkId: string
  obfuscatedCode: string
  error?: string
}

// Escuchar las tareas enviadas desde el hilo principal
if (parentPort) {
  parentPort.on('message', async (task: ObfuscationTask) => {
    try {
      // Ejecutar la ofuscación
      const obfuscated = javascriptObfuscator.obfuscate(task.code, task.config)
      const result: ObfuscationResult = {
        chunkId: task.chunkId,
        obfuscatedCode: obfuscated.getObfuscatedCode(),
      }
      // Enviar el resultado de vuelta al hilo principal
      parentPort?.postMessage(result)
    } catch (error) {
      // Manejar la excepción, asegurando que el fallo de un solo Worker no bloquee toda la compilación
      const result: ObfuscationResult = {
        chunkId: task.chunkId,
        obfuscatedCode: '',
        error: error instanceof Error ? error.message : 'Unknown error',
      }
      parentPort?.postMessage(result)
    }
  })
}
```

### 3. Gestor del pool de Workers

Este es el núcleo de toda la solución. Necesitamos mantener un pool de Workers de tamaño fijo, utilizando una estrategia **FIFO (primero en entrar, primero en salir)** para programar las tareas.

```typescript
import { Worker } from 'worker_threads'
import os from 'os'

export class WorkerPool {
  private workers: Worker[] = []
  private taskQueue: Array<{
    task: ObfuscationTask
    resolve: (result: ObfuscationResult) => void
    reject: (error: Error) => void
  }> = []
  
  constructor(options: WorkerPoolOptions = {}) {
    // Por defecto número de núcleos - 1, dejando un poco de espacio para el hilo principal
    const workerCount = options.workerCount ?? Math.max(1, (os.cpus().length || 4) - 1)
    
    for (let i = 0; i < workerCount; i++) {
      this.createWorker()
    }
  }

  private createWorker() {
    const worker = new Worker('./worker.ts')
    
    worker.on('message', (result) => {
      // Después de completar la tarea, sacar la siguiente tarea de la cola
      const nextTask = this.taskQueue.shift()
      if (nextTask) {
        this.dispatchTask(worker, nextTask)
      } else {
        // Si no hay tareas pendientes, marcar el Worker como inactivo
        this.activeWorkers.delete(worker)
      }
    })
    
    this.workers.push(worker)
  }

  // Enviar tarea al pool
  public runTask(task: ObfuscationTask): Promise<ObfuscationResult> {
    return new Promise((resolve, reject) => {
      const job = { task, resolve, reject }
      const idleWorker = this.workers.find(w => !this.activeWorkers.has(w))
      
      if (idleWorker) {
        this.dispatchTask(idleWorker, job)
      } else {
        this.taskQueue.push(job)
      }
    })
  }

  private dispatchTask(worker: Worker, job: any) {
    this.activeWorkers.set(worker, job.task)
    worker.postMessage(job.task)
  }
}
```

### 4. Informe de progreso

Esperar es doloroso, especialmente cuando no se sabe cuánto falta. Añadimos un simple reportador de progreso que proporciona retroalimentación en tiempo real del estado actual.

```typescript
export class ProgressReporter {
  private completed = 0
  private readonly total: number
  private readonly startTime: number

  constructor(total: number) {
    this.total = total
    this.startTime = Date.now()
  }

  increment(): void {
    this.completed++
    this.report()
  }

  private report(): void {
    const now = Date.now()
    const elapsed = now - this.startTime
    const percentage = (this.completed / this.total) * 100
    
    // Estimación simple de ETA
    const avgTimePerChunk = elapsed / this.completed
    const remaining = (this.total - this.completed) * avgTimePerChunk

    console.log(
      `[Parallel Obfuscation] ${this.completed}/${this.total} chunks completed (${percentage.toFixed(1)}%) | ETA: ${(remaining / 1000).toFixed(1)}s`
    )
  }
}
```

## Práctica: resultados y lecciones aprendidas

Después de implementar esta solución, el rendimiento de compilación del proyecto HagiCode mejoró de inmediato de manera notable.

### Datos de referencia de rendimiento

Realizamos las pruebas en el siguiente entorno:
- CPU: Intel Core i7-12700K (12 núcleos / 20 hilos)
- RAM: 32GB DDR4
- Node.js: v18.17.0
- SO: Ubuntu 22.04

**Comparación de resultados**:
- **Un solo hilo (antes de la optimización)**: 118 segundos
- **4 Workers**: 55 segundos (mejora del **53%**)
- **8 Workers**: 48 segundos (mejora del **60%**)
- **12 Workers**: 45 segundos (mejora del **62%**)

Se puede observar que la mejora no es lineal. Cuando el número de Workers supera los 8, la magnitud de la mejora disminuye. Esto se debe principalmente a la uniformidad de la distribución de tareas y a los cuellos de botella del ancho de banda de memoria.

### Problemas comunes y soluciones

En el uso real de HagiCode, también nos encontramos con algunos problemas que compartimos aquí:

**P1: ¿El tiempo de compilación no ha disminuido notablemente, sino que se ha vuelto más lento?**
- **Causa**: la creación de Workers tiene su propio costo, o el número de Workers es excesivo causando cambios de contexto frecuentes.
- **Solución**: se recomienda establecer el número de Workers en `número de núcleos de CPU - 1`. Al mismo tiempo, verificar si hay algún Chunk especialmente grande (por ejemplo, > 5MB), estos archivos "gigantes" se convierten en el cuello de botella; se podría considerar optimizar la estrategia de división de código.

**P2: ¿Ocasionalmente los Workers se bloquean y la compilación falla?**
- **Causa**: ciertas sintaxis especiales de código pueden causar errores internos en el ofuscador.
- **Solución**: implementamos un **mecanismo de degradación automática**. Cuando el número de fallos consecutivos de un Worker alcanza un umbral, el plugin vuelve automáticamente al modo de un solo hilo, asegurando que la compilación no se interrumpa. Al mismo tiempo, se registra el nombre del archivo con error para facilitar su corrección específica posterior.

**P3: ¿Consumo excesivo de memoria (OOM)?**
- **Causa**: cada Worker necesita un espacio de memoria independiente para cargar el ofuscador y analizar el AST.
- **Solución**:
  - Reducir el número de Workers.
  - Aumentar el límite de memoria de Node.js: `NODE_OPTIONS="--max-old-space-size=4096" npm run build`.
  - Asegurarse de no mantener referencias a objetos grandes innecesarios dentro del Worker.

## Conclusión

A través de la introducción de Node.js Worker Threads, logramos reducir con éxito el tiempo de compilación para producción del proyecto HagiCode de 120 segundos a aproximadamente 45 segundos, mejorando enormemente la experiencia de desarrollo y la eficiencia de CI/CD.

El núcleo de esta solución radica en:
1. **División racional de tareas**: utilizar los Chunks de Vite como unidades de paralelización.
2. **Control de recursos**: usar un pool de Workers para evitar el agotamiento de recursos.
3. **Diseño tolerante a fallos**: el mecanismo de degradación automática asegura la estabilidad de la compilación.
