## Diseño de arquitectura

### Descripción general de la arquitectura en capas

La integración de C# con CodeBuddy adopta una arquitectura en capas clara, este diseño hace que las responsabilidades del código estén claras y el mantenimiento posterior también sea más conveniente:

```
┌─────────────────────────────────────────────┐
│           Capa de contrato Provider          │
│   Enumeración AIProviderType + métodos de extensión             │
├─────────────────────────────────────────────┤
│           Capa de fábrica Provider            │
│   Fábrica de inyección de dependencias AIProviderFactory             │
├─────────────────────────────────────────────┤
│           Capa de implementación Provider     │
│   Implementación concreta CodebuddyCliProvider              │
├─────────────────────────────────────────────┤
│           Capa de infraestructura ACP        │
│  ACPSessionManager / StdioAcpTransport      │
│  AcpRpcClient / AcpAgentClient              │
└─────────────────────────────────────────────┘
```

¿Cuál es el beneficio de esta estratificación? En pocas palabras, las capas no se molestan entre sí. Supongamos que luego necesitas cambiar un método de comunicación (por ejemplo, de stdio a WebSocket), solo necesitas modificar la capa inferior, el código de negocio superior no necesita cambios en absoluto. Después de todo, nadie quiere que un pequeño cambio afecte a todo, cambiar un método de comunicación y tener que modificar el código de negocio por mucho tiempo sería demasiado miserable.

### Análisis de componentes centrales

La **capa de contrato Provider** es la piedra angular de toda la arquitectura. Definimos la enumeración `AIProviderType`, donde `CodebuddyCli = 3` sirve como valor de enumeración, implementando el mapeo bidireccional entre cadena y enumeración a través de métodos de extensión. De esta manera, las cadenas en los archivos de configuración pueden convertirse convenientemente en enumeraciones, y al depurar las enumeraciones también pueden convertirse en cadenas para su salida. Esto también es, básicamente es una relación de mapeo, pero si se hace bien se ahorra preocupaciones.

La **capa de fábrica Provider** es responsable de crear instancias de Provider correspondientes según la configuración. Aquí se utiliza el mecanismo de inyección de dependencias de .NET, junto con `ActivatorUtilities.CreateInstance` para lograr la creación dinámica. El beneficio del patrón de fábrica es que al agregar un nuevo Provider solo necesitas agregar lógica de creación, sin modificar el código existente. Es como escribir artículos, si quieres agregar un nuevo capítulo, agrégalo, sin necesidad de reescribir todo lo anterior.

La **capa de implementación Provider** es donde realmente se trabaja. `CodebuddyCliProvider` implementa la interfaz `IAIProvider`, proporcionando dos formas de invocación: `ExecuteAsync` (no flujo) y `StreamAsync` (flujo).

La **capa de infraestructura ACP** es el soporte de comunicación de bajo nivel. Esta capa maneja todos los detalles del protocolo, incluyendo gestión de procesos, serialización de mensajes, análisis de respuestas, etc. Es como los cimientos de una casa, no importa qué tan hermosa sea la construcción superior, las cosas de abajo deben ser estables.

## Mecanismo de comunicación

### Modo de transmisión Stdio

CodeBuddy utiliza el modo **Stdio (entrada/salida estándar)** para comunicarse con procesos externos. El comando de inicio es simple:

```bash
codebuddy --acp
```

Luego realiza el intercambio de mensajes JSON-RPC a través de entrada/salida estándar. Las ventajas de este método son:

1. **Inicio rápido**: la comunicación de procesos locales no tiene latencia de red
2. **Configuración simple**: solo necesitas especificar la ruta del archivo ejecutable
3. **Aislamiento de entorno**: cada sesión es un proceso independiente, sin interferencia mutua

Durante el proceso de comunicación se admite la inyección de variables de entorno, las comunes incluyen:

- `CODEBUDDY_API_KEY`: autenticación de clave API
- `CODEBUDDY_INTERNET_ENVIRONMENT`: configuración de entorno de red

Es como, la comunicación entre personas, encontrar una manera conveniente, para poder hablar.

### Protocolo de mensajes

ACP se basa en el protocolo JSON-RPC 2.0, el formato del mensaje es más o menos así:

```json
// Mensaje de solicitud
{
  "jsonrpc": "2.0",
  "id": 1,
  "method": "agent/prompt",
  "params": {
    "prompt": "Ayúdame a escribir un algoritmo de ordenamiento",
    "sessionId": "session-123"
  }
}

// Mensaje de respuesta
{
  "jsonrpc": "2.0",
  "id": 1,
  "result": {
    "content": "Aquí está la respuesta de la AI..."
  }
}
```

En la implementación real, encapsulamos todos estos detalles del protocolo, el código de negocio de nivel superior solo necesita preocuparse por prompt y response. Esto también, si se encapsula bien, las personas que lo usan después se sentirán más cómodas.

### 2. Integración de fábrica Provider

Agrega la rama de creación de CodeBuddy en la clase de fábrica:

```csharp
// AIProviderFactory.cs
private IAIProvider? CreateProvider(AIProviderType providerType, ProviderConfiguration config)
{
    return providerType switch
    {
        AIProviderType.CodebuddyCli =>
            ActivatorUtilities.CreateInstance<CodebuddyCliProvider>(
                _serviceProvider,
                Options.Create(config)),
        // ... otros providers
        _ => throw new NotSupportedException($"Provider {providerType} not supported")
    };
}
```

Aquí se usa `ActivatorUtilities` de inyección de dependencias, que maneja automáticamente la inyección de parámetros del constructor, muy conveniente. Esto también, las cosas de .NET, si se usan correctamente se ahorran preocupaciones.

### 3. Implementación completa del Provider

A continuación está la implementación central de `CodebuddyCliProvider`, que incluye dos formas de invocación: flujo y no flujo:

```csharp
public class CodebuddyCliProvider : IAIProvider
{
    private readonly ILogger<CodebuddyCliProvider> _logger;
    private readonly IACPSessionManager _sessionManager;
    private readonly ProviderConfiguration _config;

    public string Name => "CodebuddyCli";
    public bool SupportsStreaming => true;
    public ProviderCapabilities Capabilities { get; }

    public CodebuddyCliProvider(
        ILogger<CodebuddyCliProvider> logger,
        IACPSessionManager sessionManager,
        IOptions<ProviderConfiguration> config)
    {
        _logger = logger;
        _sessionManager = sessionManager;
        _config = config.Value;

        // Define las capacidades del Provider actual
        Capabilities = new ProviderCapabilities
        {
            SupportsStreaming = true,
            SupportsTools = true,
            SupportsSystemMessages = true,
            SupportsArtifacts = false,
            MaxTokens = 8192
        };
    }

    // Invocación no flujo: espera a que todos los resultados se devuelvan juntos
    public async Task<AIResponse> ExecuteAsync(
        AIRequest request,
        CancellationToken cancellationToken = default)
    {
        // Crea sesión independiente para la solicitud
        var session = await _sessionManager.CreateSessionAsync(
            "CodebuddyCli",
            request.WorkingDirectory,
            cancellationToken,
            request.SessionId);

        try
        {
            var fullPrompt = BuildPrompt(request);
            await session.SendPromptAsync(fullPrompt, cancellationToken);

            var responseBuilder = new StringBuilder();
            var toolCalls = new List<AIToolCall>();

            // Recopila todos los bloques de respuesta
            await foreach (var chunk in StreamFromSession(session, cancellationToken))
            {
                if (!string.IsNullOrEmpty(chunk.Content))
                {
                    responseBuilder.Append(chunk.Content);
                }
                // Maneja llamadas a herramientas...
            }

            return new AIResponse
            {
                Content = AIResultContentSanitizer.SanitizeResultContent(
                    responseBuilder.ToString()),
                ToolCalls = toolCalls,
                Provider = Name,
                Model = string.Empty
            };
        }
        finally
        {
            // Libera recursos de sesión
            await session.DisposeAsync();
        }
    }

    // Invocación de flujo: devuelve bloques de respuesta en tiempo real
    public async IAsyncEnumerable<AIStreamingChunk> StreamAsync(
        AIRequest request,
        [EnumeratorCancellation] CancellationToken cancellationToken = default)
    {
        var session = await _sessionManager.CreateSessionAsync(
            "CodebuddyCli",
            request.WorkingDirectory,
            cancellationToken);

        try
        {
            var fullPrompt = BuildPrompt(request);
            await session.SendPromptAsync(fullPrompt, cancellationToken);

            await foreach (var chunk in StreamFromSession(session, cancellationToken))
            {
                yield return chunk;
            }
        }
        finally
        {
            await session.DisposeAsync();
        }
    }

    private async IAsyncEnumerable<AIStreamingChunk> StreamFromSession(
        IACPSession session,
        [EnumeratorCancellation] CancellationToken cancellationToken)
    {
        // Itera sobre todas las actualizaciones en la sesión
        await foreach (var notification in session.ReceiveUpdatesAsync(cancellationToken))
        {
            switch (notification.Update)
            {
                case AgentMessageChunkSessionUpdate agentMessage:
                    // Maneja bloques de contenido de texto
                    if (agentMessage.Content is AcpImp.TextContentBlock textContent)
                    {
                        yield return new AIStreamingChunk
                        {
                            Content = textContent.Text,
                            Type = StreamingChunkType.ContentDelta,
                            IsComplete = false
                        };
                    }
                    break;

                case ToolCallSessionUpdate toolCall:
                    // Maneja llamadas a herramientas
                    yield return new AIStreamingChunk
                    {
                        Content = string.Empty,
                        Type = StreamingChunkType.ToolCallDelta,
                        ToolCallDelta = new AIToolCallDelta
                        {
                            Id = toolCall.ToolCallId,
                            Name = toolCall.Kind.ToString(),
                            Arguments = toolCall.RawInput?.ToString()
                        }
                    };
                    break;

                case AcpImp.PromptCompletedSessionUpdate:
                    // Respuesta completada
                    yield break;
            }
        }
    }

    // Construye el prompt completo
    private string BuildPrompt(AIRequest request, string? embeddedCommandPrompt = null)
    {
        var sb = new StringBuilder();

        // Prompt de comando incrustado (si lo hay)
        if (!string.IsNullOrEmpty(embeddedCommandPrompt))
        {
            sb.AppendLine(embeddedCommandPrompt);
            sb.AppendLine();
        }

        // Mensaje del sistema
        if (!string.IsNullOrEmpty(request.SystemMessage))
        {
            sb.AppendLine(request.SystemMessage);
            sb.AppendLine();
        }

        // Prompt del usuario
        sb.Append(request.Prompt);
        return sb.ToString();
    }
}
```

Este código tiene varios puntos clave:

1. **Gestión de sesiones**: cada solicitud crea una sesión independiente, los recursos se liberan después de que se completa la solicitud. Esta es una experiencia aprendida de dificultades — si la reutilización de sesiones no se hace bien, es fácil aparecer problemas de contaminación de estado. Después de todo, lo que se usa debe limpiarse, de lo contrario la próxima persona que lo use tendrá problemas.

2. **Procesamiento de flujo**: `IAsyncEnumerable` permite que las respuestas se devuelvan mientras se generan, sin esperar a que se genere todo el contenido. Esto es particularmente importante para escenarios de texto largo, la experiencia del usuario será mucho mejor. Es como, las personas que esperan resultados tampoco quieren esperar siempre.

3. **Llamadas a herramientas**: CodeBuddy soporta llamadas a herramientas (Function Calling), manejadas a través de `ToolCallSessionUpdate`. Esta capacidad es crítica para tareas complejas de edición de código.

4. **Filtrado de contenido**: usa `AIResultContentSanitizer` para filtrar el contenido de bloques Think, manteniendo la salida limpia.

## Ejemplo de configuración

### Archivo de configuración

Agrega la configuración relacionada con CodeBuddy en `appsettings.json`:

```yaml
AI:
  # Provider usado por defecto
  DefaultProvider: "CodebuddyCli"

  # Configuración de Providers
  Providers:
    CodebuddyCli:
      Type: "CodebuddyCli"
      WorkingDirectory: "C:/projects/my-app"
      ExecutablePath: "C:/tools/codebuddy.cmd"

  # Configuraciones relacionadas con la plataforma
  PlatformConfigurations:
    CodebuddyCli:
      ExecutablePath: "C:/tools/codebuddy.cmd"
      Arguments: "--acp"
      StartupTimeoutMs: 5000
      EnvironmentVariables:
        CODEBUDDY_API_KEY: "${CODEBUDDY_API_KEY}"
        CODEBUDDY_INTERNET_ENVIRONMENT: "production"
```

## Resumen de experiencia práctica

### Registro de dificultades

Encontramos varios problemas típicos durante el proceso de implementación, los compartimos para que todos eviten desvíos. Después de todo, evitar las dificultades de otros es algo bueno:

1. **Problema de fugas de sesión**: al principio no liberábamos correctamente las sesiones, lo que provocó el agotamiento de recursos del proceso. La solución es usar `try-finally` para asegurar que cada solicitud libere recursos. Esto también, lo que se usa debe devolverse, de lo contrario ¿qué usarán los demás después?

2. **Paso de variables de entorno**: la sintaxis de variables de entorno es diferente en Windows y Linux, luego unificamos el uso de `Dictionary<string, string?>` para manejar. Multiplataforma, si unificas el estándar al principio, después te ahorras preocupaciones.

3. **Configuración de tiempo de espera**: el inicio de CLI necesita tiempo, configuramos un tiempo de espera de inicio de 5 segundos para evitar fallas en solicitudes rápidas. Todo debe tener un límite, demasiada prisa conduce al fracaso.

4. **Problema de codificación**: la codificación predeterminada en Windows puede causar caracteres confusos en chino, al iniciar el proceso especificamos explícitamente la codificación UTF-8. Si el chino no se muestra, es muy incómodo.

### Optimización de rendimiento

1. **Grupo de sesiones**: para solicitudes cortas frecuentes, considera implementar un grupo de sesiones para reutilizar procesos
2. **Caché de conexiones**: la clase de fábrica ya soporta el almacenamiento en caché de instancias de Provider
3. **Prioridad asíncrona**: usa programación asíncrona en todo momento, evita bloquear hilos

Rendimiento, si se puede optimizar, se optimiza, después de todo cuanto más tiempo espere el usuario, peor será la experiencia.
