## Mejores prácticas para la gestión de configuración personalizada del usuario

> Nada especial, solo comparto un enfoque de gestión de configuración. Desde la arquitectura hasta la práctica, espero que te ayude a evitar algunos problemas.

## Antecedentes

La gestión de configuración es algo que, aunque importante, también puede ser trivial. Es como esos pequeños hábitos en la vida: a algunos les gusta tomar café temprano por la mañana, a otros les gusta programar hasta tarde; estas preferencias aparentemente insignificantes en realidad definen quién eres. Lo mismo ocurre con la configuración personalizada del usuario: cambio de tema, selección de idioma, personalización de atajos. Cuando estas funciones están bien hechas, los usuarios sienten que el producto "los entiende" y la retención aumenta naturalmente.

Sin embargo, hacer que este sistema sea completo no es tan simple como parece. Gestión de versiones, compatibilidad con versiones anteriores, validación de datos, control de concurrencia, control de características DLC—estos problemas surgen uno tras otro como los problemas de la adolescencia y son inevitables. Durante el desarrollo de HagiCode, también fuimos atormentados por estos problemas, pero al final encontramos algunas soluciones.

## Sobre HagiCode

Este enfoque es en realidad lo que exploramos en HagiCode. HagiCode es un asistente de código de IA con muchas funciones: idioma de la interfaz, preferencias de idioma de IA, temas, reconocimiento de voz, notificaciones, acciones rápidas—hay muchas cosas que se pueden configurar. Precisamente porque los usuarios quieren tanto, tuvimos que crear este sistema.

Dirección del proyecto: [github.com/HagiCode-org/site](https://github.com/HagiCode-org/site)

## Diseño de arquitectura

### Arquitectura en capas

El sistema de gestión de configuración de HagiCode, ¿cómo decirlo?, está dividido en varias capas, cada una con su propia responsabilidad:

```
┌─────────────────────────────────────────┐
│         Frontend (React + Redux)        │
│  - Gestión de estado de configuración   │
│  - Renderizado de formularios UI        │
│  - Persistencia de configuración        │
│    por grupos                           │
└─────────────────────────────────────────┘
                    ↓ HTTP/REST API
┌─────────────────────────────────────────┐
│   Application Service Layer             │
│  - FrontendConfigAppService             │
│  - Procesamiento de lógica de negocio   │
│  - Control de permisos                  │
└─────────────────────────────────────────┘
                    ↓
┌─────────────────────────────────────────┐
│   Domain Layer (Config Store)           │
│  - FrontendConfigStore                  │
│  - Lectura/escritura de configuración   │
│  - Validación y normalización de datos  │
└─────────────────────────────────────────┘
                    ↓
┌─────────────────────────────────────────┐
│   Infrastructure Layer                  │
│  - Almacenamiento de archivos YAML       │
│  - ISystemManagedVaultService           │
│  - Control de concurrencia              │
│    (SemaphoreSlim)                      │
└─────────────────────────────────────────┘
```

Los beneficios de este enfoque en capas son bastante evidentes:
- **Responsabilidades claras**: cada capa gestiona sus propios asuntos sin interferir entre sí
- **Fácil de probar**: cada capa se puede probar individualmente, lo que hace que las modificaciones sean más seguras
- **Extensible**: cambiar el método de almacenamiento o modificar la API no afecta a otras capas

### Diseño de interfaces principales

La interfaz para el almacenamiento de configuración del backend está definida aproximadamente así:

```csharp
public interface IFrontendConfigStore
{
    // Obtener configuración completa del usuario
    Task<FrontendConfigStoreResult> GetAsync(CancellationToken cancellationToken = default);
    
    // Actualizar configuración (soporta actualización parcial)
    Task<FrontendConfigStoreResult> UpdateAsync(
        UpdateFrontendConfigRequestDto input,
        CancellationToken cancellationToken = default);
    
    // Gestión de estado de idioma de IA
    Task<FrontendConfigAiLanguageState> GetAiLanguageStateAsync(
        string userId,
        CancellationToken cancellationToken = default);
        
    Task<FrontendConfigAiLanguageState> SetAiLanguageAsync(
        string userId,
        string language,
        CancellationToken cancellationToken = default);
}
```

Los puntos clave de esta interfaz son bastante fáciles de entender:
- **Operaciones asíncronas**: todo es asíncrono, después de todo nadie quiere esperar
- **Tokens de cancelación**: las operaciones que duran demasiado tiempo se agotan, no deben seguir consumiendo recursos
- **Actualización parcial**: solo actualiza las partes que necesitan cambiarse, sin tener que revisar toda la configuración

Agrupar la configuración es como clasificar los asuntos de la vida: trabajo es trabajo, entretenimiento es entretenimiento, relaciones son relaciones. Mezclarlo todo crea desorden, pero clasificarlo correctamente facilita las cosas:
- **Actualización bajo demanda**: actualiza solo lo que necesitas cambiar, sin afectar todo lo demás
- **Control de permisos**: diferentes configuraciones pueden tener diferentes permisos, después de todo no todos pueden modificar cualquier cosa
- **Control DLC**: las funciones avanzadas pueden vincularse a DLC, para usar cosas buenas hay que pagar

## Validación y normalización de datos

### Ejemplo de normalización de idioma

La normalización de configuración, en pocas palabras, es mantener los datos consistentes. Tomemos como ejemplo la configuración de idioma:

```csharp
public static class FrontendConfigLanguageRules
{
    // Lista de idiomas de interfaz compatibles
    public static readonly HashSet<string> ValidUiLanguages = new(StringComparer.OrdinalIgnoreCase)
    {
        "zh-CN", "zh-Hant", "en-US", "ja-JP", "ko-KR", 
        "de-DE", "fr-FR", "es-ES", "pt-BR", "ru-RU"
    };
    
    public static string NormalizeUiLanguage(string? value)
    {
        // Manejo de valores vacíos: devuelve el idioma predeterminado
        if (string.IsNullOrWhiteSpace(value)) return "en-US";
        
        var normalized = value.Trim();
        
        // Manejo de alias: convierte alias comunes a códigos estándar
        normalized = normalized.ToLower() switch
        {
            "zh" or "chinese" or "cn" => "zh-CN",
            "en" or "english" => "en-US",
            "ja" or "japanese" => "ja-JP",
            "ko" or "korean" => "ko-KR",
            _ => normalized
        };
        
        // Manejo de variantes dialectales
        if (normalized.StartsWith("zh-Hans", StringComparison.OrdinalIgnoreCase))
            return "zh-CN";
        if (normalized.StartsWith("zh-TW", StringComparison.OrdinalIgnoreCase))
            return "zh-Hant";
            
        // Validar y devolver
        return ValidUiLanguages.Contains(normalized) ? normalized : "en-US";
    }
}
```

La normalización es como ordenar una habitación: cuando las cosas están desordenadas hay que organizarlas, de lo contrario al final ni siquiera encuentras lo que buscas:
1. **Valores predeterminados para valores vacíos**: da a los valores vacíos un valor predeterminado razonable, no puedes dejarlos vacíos
2. **Mapeo de alias**: alias y abreviaturas comunes se convierten uniformemente al formato estándar
3. **Normalización de dialectos**: las variantes dialectales se fusionan en códigos estándar, después de todo programar no es investigación dialectal
4. **Validación final**: asegura que el valor devuelto esté definitivamente en la lista válida, de lo contrario todo el trabajo fue en vano

## Control de características DLC

HagiCode soporta interruptores de características DLC, algunos elementos de configuración avanzados requieren comprar DLC para usarlos. Esto es bastante común en software comercial: funciones básicas gratuitas, para usar cosas buenas hay que pagar, después de todo los desarrolladores también necesitan comer.

### Verificación de acceso DLC

```csharp
private async Task<PreparedFrontendConfigUpdate> PrepareUpdateAsync(
    UpdateFrontendConfigRequestDto input,
    FrontendConfigStoreResult current,
    CancellationToken cancellationToken)
{
    // Verifica permisos de acceso DLC
    var accessState = await _grainFactory
        .GetDlcAccessStateGrain(TurboEngineDlcId)
        .GetAccessStateAsync();
        
    var blockedFields = new List<string>();
    
    if (!accessState.IsActive)
    {
        // DLC no activado, mantiene configuración actual
        if (input.GeneralSettings?.BrandingLogo != null)
        {
            blockedFields.Add("brandingLogo");
            // Mantiene valor antiguo
            input.GeneralSettings.BrandingLogo = current.Snapshot.GeneralSettings.BrandingLogo;
        }
        
        if (input.GeneralSettings?.BrandingTitle != null)
        {
            blockedFields.Add("brandingTitle");
            input.GeneralSettings.BrandingTitle = current.Snapshot.GeneralSettings.BrandingTitle;
        }
    }
    
    return new PreparedFrontendConfigUpdate(
        input,
        new FrontendConfigUpdateDiagnosticsDto
        {
            Status = blockedFields.Count > 0 ? "partially-applied" : "success",
            BlockedFields = blockedFields,
        });
}
```

## Gestión de estado frontend

### Redux Slice

El frontend usa Redux para gestionar el estado de configuración, también es bastante convencional:

```typescript
export const frontendConfigSlice = createSlice({
  name: 'frontendConfig',
  initialState,
  reducers: {
    setConfigStatus(state, action: PayloadAction<FrontendConfigStatus>) {
      state.status = action.payload;
    },
    updateConfigGroups(state, action: PayloadAction<Partial<FrontendConfigSnapshot>>) {
      // Fusiona actualizaciones de configuración
      Object.assign(state.snapshot, action.payload);
    },
  },
  extraReducers: (builder) => {
    builder
      .addCase(fetchConfig.pending, (state) => {
        state.status = 'loading';
      })
      .addCase(fetchConfig.fulfilled, (state, action) => {
        state.status = 'succeeded';
        state.snapshot = action.payload;
      })
      .addCase(fetchConfig.rejected, (state, action) => {
        state.status = 'failed';
        state.error = action.error.message;
      });
  },
});
```

    });
  },

## Control de concurrencia

Las operaciones de actualización de configuración deben garantizar seguridad de subprocesos, de lo contrario, si dos personas modifican la configuración al mismo tiempo, ¿qué se guarda al final? HagiCode usa `SemaphoreSlim` para el control de concurrencia, ¿cómo decirlo?, también es un truco común:

```csharp
private readonly SemaphoreSlim _semaphore = new(1, 1);

public async Task<FrontendConfigStoreResult> UpdateAsync(
    UpdateFrontendConfigRequestDto input,
    CancellationToken cancellationToken = default)
{
    await _semaphore.WaitAsync(cancellationToken);
    try
    {
        // Lee configuración actual
        var current = await GetAsync(cancellationToken);
        
        // Prepara actualización
        var prepared = await PrepareUpdateAsync(input, current, cancellationToken);
        
        // Escribe configuración
        await WriteConfigAsync(prepared.UpdatedConfig, cancellationToken);
        
        return new FrontendConfigStoreResult(prepared.UpdatedConfig, prepared.Diagnostics);
    }
    finally
    {
        _semaphore.Release();
    }
}
```

## Guía práctica

### Pasos para agregar nuevos elementos de configuración

Si quieres agregar nuevos elementos de configuración, sigue este orden:

1. **Agregar una propiedad al DTO del backend**

```csharp
public class FrontendConfigGeneralSettingsDto
{
    // ... propiedades existentes
    public string? NewFeatureEnabled { get; set; }
}
```

2. **Agregar algo de lógica de normalización**

```csharp
private static FrontendConfigGeneralSettingsDto NormalizeGeneralSettings(
    FrontendConfigGeneralSettingsDto settings)
{
    return new FrontendConfigGeneralSettingsDto
    {
        // ... propiedades existentes
        NewFeatureEnabled = NormalizeOptionalString(settings.NewFeatureEnabled),
    };
}
```

3. **Agregar un control de formulario al frontend**

```typescript
<SettingsCard
  icon={<Star className="h-5 w-5" />}
  title="Configuración de nuevas funciones"
  description="Controla el estado de habilitación de nuevas funciones"
>
  <NewFeatureToggle />
</SettingsCard>
```

4. **Actualizar definición de grupos de configuración** (si agregas un nuevo grupo)

```typescript
export const ALL_PERSISTABLE_FRONTEND_CONFIG_GROUPS = [
  // ... grupos existentes
  'newFeatureSettings',
] as const;
```

### Manejo de problemas comunes

1. **Pérdida de configuración**: mecanismos de respaldo y recuperación, siempre es mejor estar preparado
2. **Conflictos de concurrencia**: bloqueos optimistas o pesimistas, como SemaphoreSlim
3. **Problemas de rendimiento**: soporta actualizaciones parciales, no muevas todo el objeto de configuración cada vez
4. **Seguridad**: la configuración sensible (como claves API) debe almacenarse cifrada, de lo contrario es problemático si alguien la roba

## Conclusión

Un sistema de gestión de configuración completo, ¿cómo decirlo?, hay muchas cosas que considerar—estructura de datos, validación y normalización, gestión de versiones, control DLC, control de concurrencia, ninguna puede faltar. El enfoque de HagiCode funciona bastante estable en el entorno de producción, al menos puede satisfacer requisitos complejos de gestión de configuración.

Este artículo comparte algo de mi experiencia. Una buena gestión de configuración no solo mejora la experiencia del usuario, también reduce los costos de mantenimiento y hace que la iteración del producto sea más fácil.
