## Hagicode.Libs：複数の AI プログラミングアシスタント CLI を統合するエンジニアリング実践

> HagiCode プロジェクトの開発過程で、Claude Code、Codex、CodeBuddy など複数の AI プログラミングアシスタント CLI を同時に統合する必要がありました。各 CLI のインターフェース、パラメータ、出力形式はすべて異なり、重複した統合コードによりプロジェクトはますます保守困難になりました。本文では、HagiCode.Libs ライブラリを通じて統一された抽象レイヤーを構築し、このエンジニアリングの課題を解決した方法を共有します——これは私たちが踏んだ坑と、蓄積した少しの経験です。

## 背景

現在の AI プログラミングアシスタント市場は賑やかで、Claude Code 以外にも、OpenAI の Codex、智譜の CodeBuddy などがあります。AI コードアシスタントプロジェクトとして、HagiCode は複数のサブプロジェクト（デスクトップ版、バックエンド、Web など）でこれらの異なる CLI ツールを統合する必要があります。

最初は大きな問題ではありませんでしたが、CLI を 1 つ統合するのはせいぜい数百行コードを書く程度のことでした。ただ、対応する CLI が増えるにつれて、事態は面倒になり始めました——

各 CLI には異なるコマンドラインパラメータ形式があり、環境変数の要件も異なり、出力形式はさらに千差万別です—— JSON を出力するもの、ストリーミング JSON を出力するもの、単なるプレーンテキストを出力するものもあります。さらにクロスプラットフォーム互換性の問題も加わり、Windows と Unix システムでは実行ファイルの発見とプロセス管理の仕組みが完全に異なり、コードの重複度が高くなります。実際、これは単に Ctrl+C、Ctrl+V が少し多くなるだけですが、保守するとなると頭痛の種です。

最も頭痛なのは、新しい CLI 機能のサポートを追加するたびに、複数のプロジェクトで同じコードを修正する必要があることです。この方法は明らかに長期的な解決策ではありません——コードにも気があり、重複が多すぎると文句を言います。

## HagiCode について

本文で共有するソリューションは、[HagiCode](https://hagicode.com) プロジェクトでの実践経験から来ています。HagiCode はオープンソースの AI コードアシスタントプロジェクトで、フロントエンドの VSCode 拡張、バックエンドの AI サービス、クロスプラットフォームデスクトップクライアントなど複数のサブプロジェクトを同時に保守する必要があります。どういうわけか、まさにこの多言語、マルチプラットフォームの複雑なシナリオが、HagiCode.Libs の誕生を促しました——追い詰められて生まれたものでしょう。

## 分析：共通点を探す

これらの AI プログラミングアシスタント CLI はそれぞれ特徴がありますが、技術的な観点からは、明らかな共通の特徴が存在します：

**類似した対話モード**：すべて CLI プロセスを起動し、プロンプトを送信し、ストリーミング応答を受信し、メッセージを解析し、最後にセッションを終了または継続します——この一連の流れは、結局同じ型から刻まれたものです。

**類似した設定要件**：すべて API キー認証、作業ディレクトリ設定、モデル選択、ツール権限制御、セッション管理が必要です。結局、みんな API という飯を食っているので、違いは味が違う程度です。

**一貫したクロスプラットフォームの課題**：すべて実行ファイルパス解決（`claude` vs `claude.exe` vs `/usr/local/bin/claude`）、プロセス起動と環境変数処理、シェルコマンドエスケープとパラメータ構築などの問題を解決する必要があります。クロスプラットフォームのことは、言っても涙が出てくるだけです—— Windows と Unix の違いは、坑を踏んだ人だけが知っています。

これらの分析に基づき、一貫したインターフェースを提供する統一された抽象レイヤー、クロスプラットフォームの CLI 発見ロジックをカプセル化し、ストリーミング出力の解析を処理し、同時に依存性注入と非 DI シナリオでの使用をサポートする必要があります。考えるだけで頭が痛くなりますが、直面する必要があります——結局自分のプロジェクトなので、泣いてもやり遂げるしかありません。

## ソリューション：HagiCode.Libs

私たちは **HagiCode.Libs** を作成しました —— 軽量級の .NET 10 ライブラリワークスペースで、MIT オープンソースプロトコルを採用し、[GitHub](https://github.com/HagiCode-org/Hagicode.Libs) で公開されています。天地を驚かすような大作ではありませんが、実際の問題を解決するには、かなり良いものです。

### プロジェクト構造

```
HagiCode.Libs/
├── src/
│   ├── HagiCode.Libs.Core/           # コア機能
│   │   ├── Discovery/                 # CLI 実行ファイル発見
│   │   ├── Process/                   # クロスプラットフォームプロセス管理
│   │   ├── Transport/                 # ストリーミングメッセージ転送
│   │   └── Environment/               # ランタイム環境解決
│   ├── HagiCode.Libs.Providers/       # プロバイダー実装
│   │   ├── ClaudeCode/                # Claude Code プロバイダー
│   │   ├── Codex/                     # Codex プロバイダー
│   │   └── Codebuddy/                 # CodeBuddy プロバイダー
│   ├── HagiCode.Libs.ConsoleTesting/  # テストフレームワーク
│   ├── HagiCode.Libs.ClaudeCode.Console/
│   ├── HagiCode.Libs.Codex.Console/
│   └── HagiCode.Libs.Codebuddy.Console/
└── tests/                             # xUnit テスト
```

### 設計目標

HagiCode.Libs の設計時に、私たちはいくつかの原則に従いました——これも踏んだ坑からまとめた経験です：

**ゼロ重量級フレームワーク依存**：ABP やその他の大型フレームワークに依存せず、軽量級を保ちます。最近は、依存関係が少ないほど、トラブルも少ないです——誰も依存関係地獄に痛めつけられたことがないでしょうか。

**クロスプラットフォームサポート**：Windows、macOS、Linux のネイティブサポートで、異なるプラットフォーム向けに異なるコードを書く必要はありません。一つのコードでどこでも動くのも、良いものです。

**ストリーミング処理**：非同期ストリームを使用して CLI 出力を処理し、モダンな .NET のプログラミングパターンに適合させます。時代は変わり、非同期こそが王道です。

**柔軟な統合**：依存性注入シナリオをサポートし、直接インスタンス化して使用することもできます。好みは人それぞれ、便利な方法を使えばいいです。

## 使用方法

### 依存性注入を通じて

プロジェクトですでに依存性注入を使用している場合（ASP.NET Core または汎用ホストなど）、直接統合できます——これも子供のようなもので、大きくはありませんが、かなり素直です：

```csharp
using HagiCode.Libs.Providers;
using Microsoft.Extensions.DependencyInjection;

var services = new ServiceCollection();
services.AddHagiCodeLibs();

await using var provider = services.BuildServiceProvider();
var claude = provider.GetRequiredService<ICliProvider<ClaudeCodeOptions>>();

var options = new ClaudeCodeOptions
{
    ApiKey = "your-api-key",
    Model = "claude-sonnet-4-20250514"
};

await foreach (var message in claude.ExecuteAsync(options, "Hello, Claude!"))
{
    Console.WriteLine($"{message.Type}: {message.Content}");
}
```

### 直接インスタンス化

単純なスクリプトや DI を使用しないシナリオの場合、直接インスタンスを作成しても問題ありません——要するに個人の好みの問題です：

```csharp
var claude = new ClaudeCodeProvider();
var options = new ClaudeCodeOptions
{
    ApiKey = "sk-ant-xxx",
    Model = "claude-sonnet-4-20250514"
};

await foreach (var message in claude.ExecuteAsync(options, "クイックソートを書いてください"))
{
    // メッセージを処理
}
```

両方の方法は同じ一連の基盤実装を使用しており、プロジェクトの実際の状況に応じて適切な統合方法を選択できます。この世界に標準答案はなく、自分に合ったものが最適です——この言葉は陳腐ですが、確かにその通りです。

## 実践経験の共有

### 1. 専用テストコンソール

各プロバイダーには専用のテストコンソールプロジェクトがあり、統合効果を独立して検証するのに便利です——どういうわけか、テストというものは、やらないならいいのですが、やるなら徹底的にやるべきです：

```bash
# Claude Code テスト
dotnet run --project src/HagiCode.Libs.ClaudeCode.Console -- --test-provider
dotnet run --project src/HagiCode.Libs.ClaudeCode.Console -- --test-all claude

# CodeBuddy テスト
dotnet run --project src/HagiCode.Libs.Codebuddy.Console -- --test-provider codebuddy-cli

# Codex テスト
dotnet run --project src/HagiCode.Libs.Codex.Console -- --test-provider codex-cli
```

テストシナリオはいくつかの重要なシナリオをカバーしています：
- **Ping**：ヘルスチェック、CLI が利用可能か確認
- **Simple Prompt**：基本プロンプトテスト
- **Complex Prompt**：マルチターン対話テスト
- **Session Restore/Resume**：セッション回復テスト
- **Repository Analysis**：コードリポジトリ分析テスト

この独立したテストコンソール設計はデバッグ時に特に便利で、問題が HagiCode.Libs レイヤーにあるのか CLI 自体にあるのかを迅速に特定できます。デバッグというのは、要するに問題がどこにあるかを見ることです——方向が合えば、半分成功です。

### 2. クロスプラットフォーム CI/CD 検証

クロスプラットフォーム互換性は HagiCode.Libs の主要な目標の一つです。GitHub Actions ワークフロー `.github/workflows/cli-discovery-cross-platform.yml` を設定し、`ubuntu-latest`、`macos-latest`、`windows-latest` の 3 つのプラットフォームで実際の CLI 発見検証を実行します。

これにより、すべてのコード変更がクロスプラットフォーム互換性を破壊しないことを保証します。ローカル開発では以下のコマンドで再現できます—— CI にすべての責任を押し付けることはできません。自分のローカルでも実行できる必要があります：

```bash
npm install --global @anthropic-ai/claude-code@2.1.79
HAGICODE_REAL_CLI_TESTS=1 dotnet test --filter "Category=RealCli"
```

### 3. メッセージストリーム処理

HagiCode.Libs は非同期ストリームを使用して CLI 出力を処理し、この方法は従来のコールバックやイベントモードよりもモダンな .NET の非同期プログラミングスタイルに適合します——結局、これが技術の進歩であり、誰にも止められません：

```csharp
public async IAsyncEnumerable<CliMessage> ExecuteAsync(
    TOptions options,
    string prompt,
    [EnumeratorCancellation] CancellationToken cancellationToken = default)
{
    // CLI プロセスを起動
    // ストリーミング JSON 出力を解析
    // CliMessage シーケンスを生成
}
```

メッセージタイプには以下が含まれます：
- `user`：ユーザーメッセージ
- `assistant`：アシスタント応答
- `tool_use`：ツール呼び出し
- `result`：セッション終了

この設計により、呼び出し側はストリーミング出力を柔軟に処理できます。たとえば、リアルタイム表示、バッファリング後処理、または他のサービスへの転送などです。美しさを気にする必要はありません。重要なのは思考が開かれ、どう使っても構いません。

### 4. Git リポジトリ探索

HagiCode.Libs.Exploration モジュールは Git リポジトリの発見と状態チェック機能を提供し、これはコードリポジトリ分析シナリオで特に便利です——これも追い詰められて生まれた機能です。HagiCode がコードリポジトリを分析する必要があるからです：

```csharp
// Git リポジトリを発見
var repositories = await GitRepositoryDiscovery.DiscoverAsync("/path/to/search");

// リポジトリ情報を取得
var info = await GitRepository.GetInfoAsync(repoPath);
Console.WriteLine($"Branch: {info.Branch}, Remote: {info.RemoteUrl}");
Console.WriteLine($"Has uncommitted changes: {info.HasUncommittedChanges}");
```

HagiCode のコード分析機能は、プロジェクト構造と Git 状態を識別するためにこのモジュールを使用しています。物を有効に使っており、子供のようなもので、無駄に育てませんでした。

## 注意事項

HagiCode プロジェクトでの実践に基づき、特に注意が必要な箇所がいくつかあります——すべて問題、問題、問題です：

**API キーのセキュリティ**：API キーをコードにハードコードしないでください。環境変数または設定管理を使用してください。HagiCode.Libs は Options オブジェクトを通じて設定を渡すことをサポートしており、様々な設定ソースとの統合が便利です。セキュリティというのは、どれだけ注意してもしすぎることはありません。

**CLI バージョン固定**：CI/CD では、バージョン漂流による不確実性を減らすために特定のバージョン（`@anthropic-ai/claude-code@2.1.79` など）を固定しました。ローカル開発でも固定バージョンを使用することをお勧めします。バージョンのことは、言っても涙が出てくるだけです——バージョンを固定しないと、すぐに教えられます。

**テスト分類**：デフォルトのテストは偽のプロバイダーを使用して決定性と速度を保ち、実際の CLI テストは明示的に有効にする必要があります。これにより、CI の高速なフィードバックを保証し、必要なときに実際の環境検証を行うことができます。この平衡木を歩くのも簡単ではありません——速さと安定性、常にトレードオフが必要です。

**セッション管理**：異なる CLI のセッション回復メカニズムは異なります。Claude Code は `.claude/` ディレクトリを使用してセッションを保存し、Codex と CodeBuddy はそれぞれ独自の方法を持っています。使用時にはそれぞれのドキュメントを確認し、セッション永続化の具体的なメカニズムを理解する必要があります。これもまた、しっかり理解しておけば害はありません。

## まとめ

HagiCode.Libs は、HagiCode の開発過程で、複数の CLI 統合の重複エンジニアリング問題を解決するために構築された統一抽象レイヤーです。一貫したインターフェースを提供し、クロスプラットフォームの詳細をカプセル化し、柔軟な統合方法をサポートすることで、複数の AI プログラミングアシスタントを統合するエンジニアリング複雑さを大幅に削減しました。すべては薄れました——しかし経験は残りました。

もしプロジェクトで複数の AI CLI ツールを統合する必要がある場合、またはクロスプラットフォームプロセス管理、ストリーミングメッセージ処理に興味がある場合は、[GitHub](https://github.com/HagiCode-org/Hagicode.Libs) にぜひお越しください。プロジェクトは MIT オープンソースプロトコルを採用しており、コントリビューションとフィードバックを歓迎します。結局は縁です、来たのですから、ぜひ友達になりましょう。

本文で共有したソリューションは、HagiCode が実際に坑を踏み、実際に最適化したものです。これにはどうしようもありませんか？坑を踏むのは正常です——もしこのソリューションに価値を感じてくれたなら、私たちのエンジニアリング力もまあまあだということです。では HagiCode 自体も、注目する価値があるかもしれません。意外な惊喜があるかもしれません。
