## HagiCodeが総合エージェントコアにHermesを選んだ理由

> AI支援コーディングプラットフォームを構築する際、適切なエージェントコアの選択はシステムの能力の上限を直接決定します。結局、無理なことは無理——間違ったフレームワークを選ぶと、どうあがいてもしっくりきません。本文ではHagiCodeの技術選定における考察と、Hermes Agentの統合実践について共有します。

## 背景

AI支援コーディングをやるにあたって、最も頭を悩ませるのは下層のエージェントフレームワークの選択です。実は市場には選択肢もそれなりにあるんです、ただ——機能が単純すぎたり、デプロイが複雑すぎたり、拡張性が見劣りしたり。私たちが求めているのは、5ドルのVPSでも動いて、GPUクラスターにも接続できるソリューション。この要件は高いとも低いとも言えず、結構な人が諦めてしまいます。

しかし実際のところ、多くのいわゆる「全能エージェント」はクラウドでしか動かなかったり、ローカルデプロイの要求が法外ほど高かったりします。2週間かけて様々なソリューションを調査した後、私たちは大胆な決断を下しました：エージェントコアを全面的に作り直し、Hermesを総合エージェントの基礎エンジンとして採用する。

この決断がもたらしたすべては、あるいは運命だったのかもしれません。

本文で共有するソリューションは、[HagiCode](https://hagicode.com) プロジェクトの実践経験から来ています。HagiCodeはAI支援コーディングプラットフォームで、VSCode拡張、デスクトップクライアント、Webサービスを通じて、開発者にインテリジェントなコーディングアシスタントを提供します。もしかすると、あなたも似たようなツールを使ったことがあるかもしれません。ただ、何かが一息足りないと感じたことは——それも私たちは理解しています。

## なぜHagiCodeはHermesを必要とするのか

Hermesを詳しく紹介する前に、なぜHagiCodeがそのようなニーズを持つのかを説明しましょう。世の中のことは、あなたがどうなりたいと思ってもそうなれるとは限りません。やはり適切な理由が必要です。

AIコードアシスタントとして、HagiCodeは複数の使用シナリオを同時にサポートする必要があります：
- ローカル開発環境：開発者は自分のPC上で実行することを望み、データはローカル外に出ない——このご時世、データセキュリティは大したことではないとも言えるし、そうでもないとも言える
- チーム協業環境：小規模チームはサーバーにデプロイされたエージェントを共有できる——節約嘛、誰も大変なんです
- クラウド弾力拡張：複雑なタスクを処理する際、GPUクラスターに自動的に拡張できる——備えあれば憂いなし

この「両方欲しい」というニーズが、私たちの目をHermesに向けさせました。この選択が正しいかどうかは分かりませんが、当時は他に良い方法もありませんでした。

## Hermes Agentとは

Hermes Agentは**Nous Research**によって作られた自律AIエージェントです。Nous Researchに馴染みのない人もいるかもしれません——彼らはHermes、Nomos、Psychéなどのオープンソース大規模モデルを開発した研究所です。彼らもなかなか大変なんです、これほど多くの良いものを作っているのに、知っている人は多くないんです。

従来のIDEプログラミングアシスタントや単純なAPIチャットラッパーとは異なり、Hermesには一つの特徴があります：動作時間が長いほど、能力が強くなる。一度きりのタスク完了で終わりではなく、長時間の動作の中で継続的に学習し経験を蓄積できます。この点は人間に似ていますよね？

### コア機能

Hermesのいくつかのコア機能は、まさにHagiCodeのニーズに合致しています。偶然とは思いませんか？

これはHagiCodeがユーザーシナリオに応じて、最適なデプロイ方法を選択できることを意味します。個人ユーザーはローカルで、チームユーザーはサーバーデプロイ、複雑なタスクはGPUへ——一つのコードで完了。この世の中、一つでも省ければ省くものです。

**マルチプラットフォームメッセージゲートウェイ**
HermesはTelegram、Discord、Slack、WhatsAppなどのプラットフォームをネイティブでサポートしています。HagiCodeにとって、これは将来これらのチャネルのAIアシスタントを容易にサポートできることを意味します。結局、誰だって選択肢は多いほうがいいでしょう？

**豊富なツールシステム**
40以上の内蔵ツールに加え、MCP（Model Context Protocol）拡張機能。これはコードアシスタントにとって非常に重要です——シェルコマンドの実行、ファイルシステム操作、Git呼び出し、これらはすべてツールのサポートが必要です。ツールのないエージェントは、羽のない鳥のようなもの——飛びたくても飛べない。

**セッションをまたぐ記憶**
Hermesは永続的な記憶システムを持ち、FTS5全文検索で過去の会話を呼び出します。これによりエージェントは以前のコンテキストを記憶し、毎回「記憶喪失」にならずに済みます。時々私も記憶喪失になりたいです。何も考えたくない、でもそうはいかない。

## HagiCodeはHermesをどのように統合するか

「なぜ」が終わったので、次は「どうやって」を見てみましょう。分かったらやるべきです。考えるだけではダメです。

### Providerレイヤーの抽象化

HagiCodeのアーキテクチャでは、すべてのAI Providerが統一された `IAIProvider` インターフェースを実装します：

```csharp
public sealed class HermesCliProvider : IAIProvider, IVersionedAIProvider
{
    public ProviderCapabilities Capabilities { get; } = new ProviderCapabilities
    {
        SupportsStreaming = true,   // ストリーミング出力をサポート
        SupportsTools = true,       // ツール呼び出しをサポート
        SupportsSystemMessages = true, // システムプロンプトをサポート
        SupportsArtifacts = false
    };
}
```

この抽象化レイヤにより、HagiCodeは異なるAI Providerをシームレスに切り替えることができます。OpenAI、Claude、Hermesのいずれであっても、上層の呼び出し方は完全に同一です。要するに、楽ができるということです。

### ACP通信プロトコル

Hermesは**ACP (Agent Communication Protocol)** を使用して通信します。これはエージェント通信のために設計されたプロトコルで、主なメソッドは以下の通りです：

| メソッド | 説明 |
|------|------|
| `initialize` | 接続を初期化し、プロトコルバージョンとクライアント機能を取得 |
| `authenticate` | 認証を処理し、複数の認証方法をサポート |
| `session/new` | 新しいセッションを作成し、作業ディレクトリとMCPサーバーを設定 |
| `session/prompt` | プロンプトを送信し応答を取得 |

HagiCodeは `StdioAcpTransport` を通じてACP転送層を実装し、Hermesの子プロセスを起動して標準入出力で通信します。このことは複雑に聞こえますが、やるとなんとかなります——主に忍耐が必要です。

### 設定管理

`HermesPlatformConfiguration` クラスを通じて設定を管理します：

```csharp
public sealed class HermesPlatformConfiguration : IAcpPlatformConfiguration
{
    public string ExecutablePath { get; set; } = "hermes";
    public string Arguments { get; set; } = "acp";
    public int StartupTimeoutMs { get; set; } = 5000;
    public string ClientName { get; set; } = "HagiCode";
    public HermesAuthenticationConfiguration Authentication { get; set; }
    public HermesSessionDefaultsConfiguration SessionDefaults { get; set; }
}
```

`appsettings.json` でHermesを設定します：

```json
{
  "Providers": {
    "HermesCli": {
      "ExecutablePath": "hermes",
      "Arguments": "acp",
      "StartupTimeoutMs": 10000,
      "ClientName": "HagiCode",
      "Authentication": {
        "PreferredMethodId": "api-key",
        "MethodInfo": {
          "api-key": "your-api-key-here"
        }
      },
      "SessionDefaults": {
        "Model": "claude-sonnet-4-20250514",
        "ModeId": "default"
      }
    }
  }
}
```

設定というのは、見た目は簡単ですが、正しく調整するのには結構な手間がかかります。

### Orleans分散アーキテクチャ

HagiCodeはOrleansを使用して分散システムを構築しており、Hermes統合は以下のコンポーネントを通じて実現されます：

- **HermesGrain**：Orleans Grain実装、セッション実行を処理
- **HermesPlatformConfiguration**：プラットフォーム固有の設定
- **HermesAcpSessionAdapter**：ACPセッションアダプタ
- **HermesConsole**：専用の検証コンソール

Orleansという名前はなかなか良いですね、伝説のアリババ——このOrleansはあのOrleansではありませんが、名前が良いのは加点要素です。

### 完全な実行フロー

以下はHermes Providerのコア実行ロジックです：

```csharp
private async IAsyncEnumerable<AIStreamingChunk> StreamCoreAsync(
    AIRequest request,
    string? embeddedCommandPrompt,
    [EnumeratorCancellation] CancellationToken cancellationToken)
{
    // 1. 転送層を作成し、Hermes子プロセスを起動
    await using var transport = new StdioAcpTransport(
        platformConfiguration.GetExecutablePath(),
        platformConfiguration.GetArguments(),
        platformConfiguration.GetEnvironmentVariables(),
        platformConfiguration.GetStartupTimeout(),
        _loggerFactory.CreateLogger<StdioAcpTransport>());
    await transport.ConnectAsync(cancellationToken);

    // 2. 初期化し、プロトコルバージョンと認証方法を取得
    var initializeResult = await SendHermesRequestAsync(
        transport, nextRequestId++, "initialize",
        BuildInitializeParameters(platformConfiguration), cancellationToken);

    // 3. 認証を処理
    var authMethods = ParseAuthMethods(initializeResult);
    if (!isAuthenticated)
    {
        var methodId = platformConfiguration.Authentication.ResolveMethodId(authMethods);
        await SendHermesRequestAsync(transport, nextRequestId++, "authenticate", ...);
    }

    // 4. セッションを作成
    var newSessionResult = await SendHermesRequestAsync(
        transport, nextRequestId++, "session/new",
        BuildNewSessionParameters(platformConfiguration, workingDirectory, model), cancellationToken);
    var sessionId = ParseSessionId(newSessionResult);

    // 5. プロンプトを実行しストリーミング応答を収集
    await foreach (var payload in transport.ReceiveMessagesAsync(cancellationToken))
    {
        // session/update通知を処理し、ストリーミングチャンクに変換
        if (TryParseSessionNotification(root, out var notification))
        {
            if (_responseMapper.TryConvertToStreamingChunk(notification, out var chunk))
            {
                yield return chunk;
            }
        }
    }
}
```

コードも、見慣れればそんなものです。重要なのは発想ですよね？

### ヘルスチェック

Hermesサービスの可用性を保証するため、HagiCodeはヘルスチェックメカニズムを実装しています：

```csharp
public async Task<ProviderTestResult> PingAsync(CancellationToken cancellationToken = default)
{
    var response = await ExecuteAsync(
        new AIRequest
        {
            Prompt = "Reply with exactly PONG.",
            CessionId = null,
            AllowedTools = Array.Empty<string>(),
            WorkingDirectory = ResolveWorkingDirectory(null)
        },
        cancellationToken);

    var success = string.Equals(response.Content.Trim(), "PONG", StringComparison.OrdinalIgnoreCase);
    return new ProviderTestResult
    {
        ProviderName = Name,
        Success = success,
        ResponseTimeMs = stopwatch.ElapsedMilliseconds,
        ErrorMessage = success ? null : $"Unexpected Hermes ping response: '{response.Content}'."
    };
}
```

これがいわゆる「ヘルスチェック」でしょう。実は人間も同じです。たまに自分をチェックする必要があります——ただ通常、何をチェックすべきかは誰も教えてくれません。

## 実践上の注意事項

Hermes統合の過程で、事前に知っておくべき落とし穴がいくつかあります。このご時世、誰だっていくつかの穴に落ちたことがあるでしょう？

### 認証方法の設定

Hermesは複数の認証方法（API Key、Tokenなど）をサポートしており、実際のデプロイ状況に応じて選択する必要があります。設定ミスは接続失敗を引き起こしますが、エラーメッセージが直感的でない場合があります。時々エラーメッセージと実際の原因が十万八千里も違うことがあり、少しずつ調査する必要があります。

### MCPサーバーの設定

セッション作成時にMCPサーバーリストを設定し、Hermesが外部ツールを呼び出せるようにします。ただ注意が必要です：
- MCPサーバーアドレスはアクセス可能である必要がある
- タイムアウト時間を合理的に設定する
- サーバーが利用できない場合のデグレード処理

この世の中、防ぎようがありません。

### 作業ディレクトリの管理

各セッションでは作業ディレクトリを指定する必要があり、Hermesがプロジェクトファイルに正しくアクセスできるようにします。マルチプロジェクトシナリオでは、作業ディレクトリを動的に切り替える必要があります。言うのは簡単ですが、実際にやるには考慮すべき場合も結構あります。

### 応答の集約処理

Hermesの応答は `session/update` 通知と最終結果に分散している可能性があり、正しくマージして処理する必要があります。そうでないとコンテンツの欠落が発生します。このことでも私は損をしていません。徐々に慣れました。

### エラー処理戦略

実行時エラーは明確に返すべきで、他のProviderに静かにフォールバックすべきではありません。こうすることで、ユーザーはHermesに問題があることを知り、なぜか別のモデルに切り替わったという不思議な体験を避けられます。結局、適当に誤魔化すわけにはいかない。

## まとめ

HagiCodeが総合エージェントコアにHermesを選んだのは、思いつきの決断ではなく、実際のニーズと技術的特徴に基づいた慎重な選択です。この選択が正しいかどうか、今のところまだ早いですが、現時点では使い勝手は悪くありません。

Hermesが提供する柔軟なデプロイ能力により、HagiCodeは様々な使用シナリオに適応できます。強力なツールシステムとMCPサポートにより、AIアシスタントが本当に実用的な作業を行えるようになります。そしてACPプロトコルとProvider抽象化レイヤーにより、統合プロセス全体が明確で制御可能になります。

もしあなたがAIプロジェクトのエージェントフレームワークを選定しているなら、この記事が参考になれば幸いです。結局、正しい基礎アーキテクチャを選べば、その後の開発はずっと楽になります......
