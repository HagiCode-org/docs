## スクリプト地獄にさよなら：なぜ C# でモダンなビルドシステムを構築するのか

> HagiCode プロジェクトが Nuke を利用して、型安全でクロスプラットフォーム、かつ高度に拡張可能な自動ビルドプロセスを実現し、従来のビルドスクリプトの保守問題を完全に解決する方法を紹介します。

## 背景

ソフトウェア開発の長い旅路において、「ビルド」という言葉は愛憎入り混じるものです。愛されるのは、ワンクリックでコードが製品になる瞬間、それはプログラマーにとって最も魅力的な瞬間です。憎まれるのは、あの散らかったビルドスクリプトの山を保守することで、それはまさに悪夢です。

多くのプロジェクトで、私たちは Python でスクリプトを書いたり、XML 設定ファイルを使うことに慣れてきました（`<property>` に支配された恐怖を想像してください）。しかし、プロジェクトの複雑さが増すにつれて、特に HagiCode のようにフロントエンドとバックエンド、マルチプラットフォーム、多言語混合開発に関わるプロジェクトでは、従来のビルド方法は力不足になり始めました。スクリプトロジックの分散、型チェックの欠如、IDE サポートの弱さ... これらの問題は小さな穴のように、開発チームを時々つまずかせます。

これらの問題を解決するために、HagiCode プロジェクトでは **Nuke** —— C# に基づくモダンなビルドシステムを導入することにしました。それは単なるツールではなく、ビルドプロセスに対する再考のようなものです。今日は、なぜそれを選択したのか、そしてそれがどのようにして私たちの開発体験を「向上」させたのかについて話しましょう。

## 核心分析：なぜ Nuke なのか？

「ビルドシステムなんてたくさんあるじゃないか。Make、Gradle、あるいは直接 Shell スクリプトを使えばいいんじゃない？なんでわざわざ C# のものを使うの？」と疑問に思うかもしれません。

これは実は良い質問です。Nuke の核心的な魅力は、私たちが最も慣れ親しんだプログラミング言語の特性をビルドスクリプトの世界にもたらす点にあります。

### 1. ビルドプロセスのモジュール化：Target の芸術

Nuke の設計理念は非常に明確です：**すべてがターゲット**。

従来のスクリプトでは、数百行の線形実行コードを書くことがあり、ロジックが複雑に入り組んでいます。一方、Nuke では、ビルドプロセスを独立した `Target`（ターゲット）に分解します。各ターゲットは一つのことだけを担当します。例えば：

- `Clean`: 出力ディレクトリをクリア
- `Restore`: 依存パッケージを復元
- `Compile`: コードをコンパイル
- `Test`: ユニットテストを実行

この設計は単一責任原則に非常によく合致しています。ブロックを積み上げるように、これらの Target を自由に組み合わせることができます。さらに重要な的是、Nuke は Target 間の依存関係を定義できます。例えば、`Test` を実行したい場合、システムは自動的に最初に `Compile` が実行されたかチェックします。`Compile` を実行したい場合、当然ながら最初に `Restore` が必要です。

この依存関係グラフはロジックをより明確にするだけでなく、実行効率も大幅に向上させます。Nuke は最適な実行パスを自動的に分析します。

### 2. 型安全：スペルミスの悪夢にさよなら

Python でビルドスクリプトを書いたことがある友人は、このような厄介な状況に遭遇したことがあるでしょう：スクリプトが5分間実行され、最後に `Confi.guration` のスペルミス、あるいは文字列を数値であるべきパラメータに渡したというエラーが表示されます。

C# でビルドスクリプトを書く最大の利点は **型安全** です。これは以下を意味します：

- **コンパイル時チェック**：コードを入力しているときに、IDE がどこが間違っているかを教えてくれます。実行時になるまで待つ必要はありません。
- **リファクタリングの安心感**：変数名やメソッド名を変更したい場合、IDE のリファクタリング機能でワンクリックで完了します。グローバル検索と置換で怯える必要はありません。
- **インテリセンス**：強力な IntelliSense がコードを自動補完し、あの難解な API を覚えるためにドキュメントを調べる必要がありません。

### 3. クロスプラットフォーム：統一されたビルド体験

以前は Windows で `.bat` を書き、Linux で `.sh` を書き、両方を互換性を持たせるために Python スクリプトも書いていました。今では、.NET Core（現在 .NET 5+）が動作する場所ならどこでも、Nuke も動作します。

これは、チームメンバーが Windows、Linux、macOS のいずれを使用していても、Visual Studio、VS Code、Rider のいずれを使用していても、すべてが同じロジックを実行することを意味します。「私のマシンでは動く」といった環境の違いによる問題を大幅に排除できます。

### 4. パラメータと設定管理

Nuke は非常にエレガントなパラメータ解析メカニズムを提供しています。`string[] args` を手動で解析する必要はありません。プロパティを定義し、`[Parameter]` 属性を追加するだけで、Nuke がコマンドライン引数と設定ファイルのマッピングを自動処理します。

例えば、ビルド設定を簡単に定義できます：

```csharp
[Parameter("Configuration to build - Default is 'Debug'")]
readonly Configuration BuildConfiguration = IsLocalBuild ? Configuration.Debug : Configuration.Release;

Target Compile => _ => _
    .DependsOn(Restore)
    .Executes(() =>
    {
        // ここで BuildConfiguration を使用します。これは型安全です
        DotNetBuild(s => s
            .SetConfiguration(BuildConfiguration)
            .SetProjectFile(SolutionFile));
    });
```

この書き方は直感的で、エラーが発生しにくいです。

## 実践ガイド：プロジェクトでの導入方法

空論は国を誤り、実践は国を興す。HagiCode プロジェクトで、具体的にこのソリューションをどのように導入したかを見てみましょう。

### 1. プロジェクト構造の計画

ビルドスクリプトがプロジェクトのルートディレクトリを汚染したくはありませんし、一部の Java プロジェクトのようにディレクトリ構造が深すぎるものにもしたくありません。そこで、Nuke に関連するすべてのビルドファイルを `nukeBuild/` フォルダに統一して配置しました。

これを行う利点は次のとおりです：
- プロジェクトのルートディレクトリが清潔に保たれます。
- ビルドロジックが凝集され、管理が容易です。
- 新しいメンバーが参加したとき、「ああ、これはビルド関連のロジックだ」と一目でわかります。

### 2. 明確な Target 依存チェーンの設計

Target を設計する際、私たちは一つの原則に従いました：**原子化 + 依存フロー**。

各 Target は十分に小さく、一つのことだけを行うべきです。例えば、`Clean` はファイルの削除のみを担当し、その中でついでにパッケージ化をするべきではありません。

推奨される依存フローはおよそ次のようになります：

`Clean` -> `Restore` -> `Compile` -> `Test` -> `Pack`

もちろん、これは絶対ではありません。例えば、テストだけを実行してパッケージ化したくない場合、Nuke では `nuke Test` を直接実行でき、自動的に前段階の Restore と Compile ステップを処理します。

### 3. 完全なエラーハンドリングとログ

ビルドスクリプトで最も恐ろしいものは何ですか？それはエラーメッセージが不明確なことです。例えば、ビルドが失敗し、ログに "Error: 1" としか表示されないと、非常にイライラします。

Nuke では、C# の例外処理メカニズムを直接使用できるため、エラーを非常に正確にキャプチャしてレポートできます。

```csharp
Target Publish => _ => _
    .DependsOn(Test)
    .Executes(() =>
    {
        try 
        {
            // NuGet に公開しようとする
            DotNetNuGetPush(s => s
                .SetTargetPath(ArtifactPath)
                .SetSource("https://api.nuget.org/v3/index.json")
                .SetApiKey(ApiKey));
        }
        catch (Exception ex)
        {
            Log.Error($"公開に失敗しました。みんなキーが正しいかチェックしてね: {ex.Message}");
            throw; // ビルドプロセスが非ゼロの終了コードで終了することを確認
        }
    });
```

### 4. 統合テストで品質を保証

ビルドスクリプト自体もコードであり、テストも必要です。Nuke では、ビルドプロセスのテストを書くことができ、ビルドロジックを変更した後、既存の公開プロセスを破壊しないことを保証できます。これは継続的インテグレーション（CI）パイプラインで特に重要です。

## まとめ

Nuke の導入により、HagiCode のビルドプロセスはかつてないほどスムーズになりました。これは単なるツールの置き換えではなく、エンジニアリング思考の向上です。

**私たちが得たものは何ですか？**
- **保守性**：コード即設定、ロジックが明確で、新人もすぐに使いこなせます。
- **安定性**：強力な型チェックにより、90%以上の初歩的なエラーが削減されました。
- **一貫性**：クロスプラットフォームの統一されたエクスペリエンスにより、環境の差異が排除されました。

以前、ビルドスクリプトを書くことは「暗闇の中で手探り」だったとしたら、Nuke を使用することは「明かりをつけて夜道を歩く」ようなものです。デバッグが難しいスクリプト言語の保守にうんざりしているなら、ビルドロジックも C# の世界に移してみてはどうでしょうか。もしかすると、ビルドもこれほどエレガントになれるかもしれません。
