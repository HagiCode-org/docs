## Design.md：AI によるフロントエンド UI デザインの一貫性を保つソリューション

> AI 支援フロントエンド開発の時代において、AI が生成する UI の一貫性をどのように保つか？本文では、awesome-design-md に基づいてデザインギャラリーサイトを構築した実践経験と、構造化された design.md を作成して AI に規範的な UI デザインを指導する方法について共有します。

## 背景

AI でフロントエンドコードを書いたことのある友人なら、誰でも似たような経験を持っているはずです：同じページで、AI に何度か生成させると、毎回スタイルが異なります。ある時は角丸、ある時は角ばった角、ある時は間隔が 8px、ある時は 16px になり、同じボタンでさえ異なる会話では異なる見た目になります。

これは単なる個別の現象ではありません。AI 支援開発の普及に伴い、**AI が生成するフロントエンド UI の一貫性の欠如**は普遍的な問題になっています。異なる AI アシスタント、異なるプロンプト、さらには同じアシスタントでも異なる会話では、全く異なるスタイルのインターフェースデザインが生成されます。これは製品の反復開発に巨大なメンテナンスコストをもたらします。

問題の根源は実際には非常に単純です：**権威あるデザイン参考ドキュメント**が不足しています。従来の CSS スタイルファイルは開発者に「どう実装するか」を伝えることしかできませんが、「なぜこのようにデザインするのか」および「どのシーンでどのデザインパターンを使用するか」を完全に伝えることはできません。AI にとっては、デザイン仕様を理解するためにより明確な構造化された記述が必要です。

同時に、オープンソースコミュニティにはすでにいくつかの優れたリソースがあります。[VoltAgent/awesome-design-md](https://github.com/VoltAgent/awesome-design-md) プロジェクトは、多くの有名企業のデザインシステムドキュメントを収集しており、各ディレクトリには README.md、DESIGN.md、プレビュー HTML が含まれています。しかし、これらは上流リポジトリに分散しており、迅速に閲覧・比較することが困難です。

では、これらのリソースを統合して閲覧しやすいデザインギャラリーにし、同時に構造化された design.md を作成して AI に使用させることはできないでしょうか？

答えはイエスです。次に、私たちのソリューションを共有します。

## HagiCode について

本文で共有するソリューションは、[HagiCode](https://hagicode.com) プロジェクトでの実践経験に基づいています。HagiCode は AI 支援開発プラットフォームであり、開発プロセスにおいて、私たちも AI が生成する UI の一貫性がない問題に直面しました。この問題を解決するために、デザインギャラリーサイトを構築し、規範化された design.md を作成しました。本文はこのソリューションの要約です。

[GitHub - HagiCode-org/site](https://github.com/HagiCode-org/site)

## 分析

コードを書き始める前に、この問題のいくつかの技術的な課題を分解してみましょう。

### コンテンツソース管理：分散したデザインリソースをどのように統一する？

上流の awesome-design-md リポジトリには大量のデザインドキュメントが含まれていますが、それを私たちのプロジェクトに組み込む方法が必要です。

**ソリューション：git submodule を使用する**

```
awesome-design-md-site
└── vendor/awesome-design-md          # 上流リソース（git submodule）
```

这样做有几个好处：
- バージョン管理可能：特定の上流バージョンをロックできます
- オフラインビルド：ビルド時に外部 API をリクエストする必要がありません
- コンテンツレビュー：PR で具体的な変更を確認できます

### データ標準化：異なるドキュメント構造をどのように統一する？

異なる企業のデザインドキュメント構造は異なる場合があり、プレビューファイルが不足しているものや、命名が統一されていないものがあります。ビルド時に標準化処理を行う必要があります。

**ソリューション：ビルド時にスキャンして標準化されたエントリを生成する**

コアモジュールは `awesomeDesignCatalog.ts` であり、以下を担当します：
1. `vendor/awesome-design-md/design-md/*` ディレクトリをスキャンする
2. 各エントリに必須ファイル（README.md、DESIGN.md、少なくとも1つのプレビューファイル）が含まれているかを検証する
3. Markdown コンテンツを抽出して HTML としてレンダリングする
4. 標準化されたエントリデータを生成する

### 静的サイトアーキテクチャ：静的デプロイを維持しながら動的検索を提供するにはどうすればよいか？

デザインギャラリーなので、検索機能は必須です。しかし、Astro は静的サイトジェネレーターであり、リアルタイム検索をどのように実装するのでしょうか？

**ソリューション：React Island + URL クエリパラメータ同期**

```typescript
// src/components/gallery/SearchToolbar.tsx

export function SearchToolbar() {
  const [query, setQuery] = useState('');

  // URL 同期
  useEffect(() => {
    const params = new URLSearchParams(window.location.search);
    setQuery(params.get('q') || '');
  }, []);

  // リアルタイムフィルタリング
  const filtered = entries.filter(entry =>
    entry.searchText.includes(query)
  );

  return <input value={query} onChange={e => {
    setQuery(e.target.value);
    updateURL(e.target.value);
  }} />;
}
```

这样做的好处是保留了静态站点的可部署性（可以部署到任何静态托管服务），同时提供了即时过滤的用户体验。

### デザイン文書化：AI にデザイン仕様を理解させて遵守させるにはどうすればよいか？

これはソリューション全体のコアです。AI がデザイン仕様を理解して適用できるように、構造化された design.md を作成する必要があります。

**ソリューション：ClickHouse DESIGN.md の構造を参考にする**

[ClickHouse の DESIGN.md](https://github.com/ClickHouse/ClickHouse/blob/master/DESIGN.md) は良い参考になります。以下が含まれています：
- Visual Theme & Atmosphere

- Do's and Don'ts

私たちのアプローチは：**構造を参考にし、内容を書き直す**ことです。ClickHouse DESIGN.md の章構造を保持し、内容を独自のプロジェクトで実際に使用するデザイントークンとコンポーネント仕様に置き換えます。

## ソリューション

上記の分析に基づき、私たちのソリューションには 4 つのコアモジュールが含まれます。

### 1. コンテンツ取り込みパイプライン

これはシステム全体の基礎であり、上流リソースからコンテンツを抽出して標準化する責任を負います。

```typescript
// src/lib/content/awesomeDesignCatalog.ts

export async function scanSourceEntries(): Promise<DesignEntry[]> {
  const designDir = 'vendor/awesome-design-md/design-md';
  const entries: DesignEntry[] = [];

  for (const dir of await fs.readdir(designDir)) {
    const entryPath = path.join(designDir, dir);
    if (await isValidDesignEntry(entryPath)) {
      const entry = await normalizeDesignEntry(entryPath);
      entries.push(entry);
    }
  }

  return entries;
}

async function isValidDesignEntry(dir: string): Promise<boolean> {
  const requiredFiles = ['README.md', 'DESIGN.md'];
  for (const file of requiredFiles) {
    if (!(await fileExists(path.join(dir, file)))) {
      return false;
    }
  }
  return true;
}
```

### 2. ギャラリー閲覧インターフェース

ギャラリーインターフェースには 3 つの主要部分が含まれます：

**トップページ**：すべてのデザインエントリのカードグリッドを表示し、各カードには以下が含まれます：
- デザインエントリのタイトルと概要
- プレビュー画像（ある場合）
- クイック検索ハイライト

**詳細ページ**：単一のデザインエントリの完全な情報を集約して表示：
- README ドキュメント
- DESIGN ドキュメント
- プレビュー（明/暗テーマ切り替え対応）
- 隣接するエントリのナビゲーション

**ナビゲーション**：ギャラリーに戻る、隣接するエントリを閲覧することをサポート

トップページのギャラリーは高密度カードレイアウトを使用し、異なるソースの design.md エントリを統一された視覚フレームワークに並べて表示し、ブランドスタイル、ボタンパターン、タイポグラフィのリズムを迅速に比較するのに便利です。

![Awesome Design MD Gallery デザインカードグリッド](/article-images/design-md-consistent-ui/1775613199000-e652f7155dd5.png)

特定のエントリに入ると、詳細ページはデザインの概要とリアルタイムプレビューを同じページに配置し、ドキュメント、プレビュー、ソースコード間を行き来するコストを削減します。

### 3. 検索機能

検索機能はクライアント側フィルタリングに基づき、URL クエリパラメータを使用して状態を保持します：

```typescript
// src/components/gallery/SearchToolbar.tsx

function SearchToolbar({ entries }: { entries: DesignEntry[] }) {
  const [query, setQuery] = useState('');
  const [results, setResults] = useState(entries);

  useEffect(() => {
    const params = new URLSearchParams(window.location.search);
    const initialQuery = params.get('q') || '';
    setQuery(initialQuery);
    filterEntries(initialQuery);
  }, []);

  const filterEntries = (searchQuery: string) => {
    const filtered = entries.filter(entry =>
      entry.searchText.toLowerCase().includes(searchQuery.toLowerCase())
    );
    setResults(filtered);
  };

  const handleChange = (e: React.ChangeEvent<HTMLInputElement>) => {
    const value = e.target.value;
    setQuery(value);
    filterEntries(value);

    // URL を更新（ページリフレッシュをトリガーしない）
    const newUrl = value
      ? `${window.location.pathname}?q=${encodeURIComponent(value)}`
      : window.location.pathname;
    window.history.replaceState({}, '', newUrl);
  };

  return (
    <div className="search-toolbar">
      <input
        type="text"
        value={query}
        onChange={handleChange}
        placeholder="デザインエントリを検索..."
      />
      <span className="result-count">{results.length} 件の結果</span>
    </div>
  );
}
```

## 実践

ソリューションを理解したところで、具体的にどう実装するのでしょうか？

### 実装手順

**第一段階：サブモジュールの初期化**

```bash
# 上流リポジトリをサブモジュールとして追加
git submodule add https://github.com/VoltAgent/awesome-design-md.git vendor/awesome-design-md

# サブモジュールを初期化して更新
git submodule update --init --recursive
```

**第二段階：コンンテンツパイプラインの作成**

`awesomeDesignCatalog.ts` を実装し、以下を含みます：
- ファイルスキャンと検証ロジック
- Markdown レンダリング（Astro の組み込みレンダラーを使用）
- エントリデータの抽出

**第三段階：ギャラリー UI の構築**

Astro + React Islands を使用して作成：
- トップページギャラリーレイアウト（カードグリッド）
- デザインカードコンポーネント
- 検索ツールバー
- 詳細ページレイアウト

**第四段階：デザインドキュメントの作成**

ClickHouse DESIGN.md の構造に基づき、独自プロジェクトの実際のデザイントークンを入力します。README.md を更新し、design.md へのリンクを追加します。

### 注意事項

**セキュリティ**：Markdown レンダリングは安全でない HTML をフィルタリングする必要があります。Astro の組み込みレンダラーはデフォルトで script タグをフィルタリングしますが、XSS リスクに注意する必要があります。

**パフォーマンス**：大量の iframe プレビューはファーストスクリーンの読み込みに影響する可能性があります。`loading="lazy"` を使用してプレビューコンテンツを遅延読み込みすることをお勧めします。

**メンテナンス性**：design.md はコード実装と同期を保つ必要があります。CI でチェックを追加し、CSS 変数がドキュメントとコードで一致していることを確認することをお勧めします。

**アクセシビリティ**：カラーコントラストが WCAG AA 基準（少なくとも 4.5:1）に準拠していることを確認してください。

### AI 使用ガイド

design.md を作成した後、AI に実際に使用させるにはどうすればよいでしょうか？実用的なテクニックをいくつか紹介します：

**テクニック1：プロンプトで明確に参照する**

```
プロジェクトルートディレクトリの design.md ファイルを参照し、その中で定義されたデザイン仕様を使用して以下のコンポーネントを実装してください：
- ボタン：プライマリカラーを使用、角丸 8px
- カード：elevation-2 シャドウ階層を使用
```

**テクニック2：具体的な CSS 変数を参照するよう要求する**

```
ナビゲーションバーを実装し、以下を要求：
- 背景色に --color-bg-primary を使用
- ボーダーに --color-border-subtle を使用
- テキストに --text-color-primary を使用
```

**テクニック3：システムプロンプトに design.md コンテンツを含める**

AI ツールがカスタムシステムプロンプトをサポートしている場合、design.md のコアコンテンツを直接追加できます。

### テスト戦略

**コンテンツパイプラインテスト**：
- ファイル欠如シナリオ（README.md または DESIGN.md が不足）
- フォーマットエラーシナリオ（Markdown 解析失敗）
- 空ディレクトリシナリオ

**検索機能テスト**：
- 空結果処理
- 特殊文字（中国語、emoji など）
- URL 同期検証

**UI コンポーネントテスト**：
- 明/暗テーマ切り替え
- レスポンシブレイアウト
- プレビュー読み込み状態

### デプロイフロー

サブモジュールの更新とビルドデプロイを自動化することをお勧めします。上流リポジトリが更新されたときに CI フローを自動的にトリガーできます。

## まとめ

HagiCode が開発プロセスで遭遇した AI 生成 UI の一貫性がない問題は、本質的には構造化されたデザイン参考ドキュメントが不足していることです。デザインギャラリーサイトを構築し、規範化された design.md を作成することで、私たちはこの問題を正常に解決しました。

このソリューションのコア価値は以下の点にあります：
- **統一されたリソース**：分散したデザインシステムドキュメントを統合
- **構造化された仕様**：デザイン仕様を AI が理解できる形式で提示
- **継続的なメンテナンス**：git submodule を通じてコンテンツの更新を保持

もし AI 支援フロントエンド開発を使用しているなら、このソリューションを試してみることをお勧めします。構造化された design.md を作成することは、AI が生成するコードの一貫性を向上させるだけでなく、チーム内でデザイン仕様の統一を維持するのにも役立ちます。
