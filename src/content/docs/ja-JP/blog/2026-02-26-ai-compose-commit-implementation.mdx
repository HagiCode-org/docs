## AI Compose Commit：AI で Git コミットワークフローをインテリジェントに再構築

## Introduction

ソフトウェア開発プロセスにおいて、コードのコミットはプログラマーが毎日直面する日常業務です。しかし、こんな状況を経験したことはありませんか？1日の作業終了後に Git を開くと、数十の未ステージの変更ファイルがあり、それらをどのように合理的なコミットに整理すればよいか分からないという状況です。

従来の方法では、ファイルを手動でバッチごとにステージし、1つずつコミットし、コミットメッセージを記述するというプロセスで、時間がかかりやすくエラーも発生しやすいです。私たちはよくこの作業に多くの時間を浪費していました。結局のところ、疲れた夜にこのような些細なことにも悩みたくは誰もありませんから。

HagiCode プロジェクトでは、このワークフローを完全に変革することを目的とした新機能——AI Compose Commit を立ち上げました。この機能は AI が作業ディレクトリ内のすべての未コミット変更をインテリジェントに分析し、複数の論理的なコミットに自動的にグループ化して、規約に準拠したコミット操作を実行します。この記事では、この機能の実装原理、技術アーキテクチャ、そして実践で遭遇した課題と解決策について深く掘り下げて解説します。

## About HagiCode

この記事で共有するソリューションは、[HagiCode](https://hagicode.com) プロジェクトでの実践経験に基づいています。

## Background

### 従来の Git コミットの課題

Git はバージョン管理システムとして、開発者に強力なコード管理機能を提供します。しかし実際の使用では、コミット操作が開発プロセスのボトルネックになることがよくあります：

1. **手動グループ化の時間コスト**: 大量のファイル変更がある場合、開発者はファイルの内容を1つずつチェックし、どのファイルが同じ機能に属するかを判断する必要があり、多くの脳力を消費します
2. **コミットメッセージの品質のばらつき**: Conventional Commits 規約に準拠したコミットメッセージを記述するには経験とスキルが必要で、初心者はしばしば不規範なコミットを作成します
3. **マルチリポジトリ管理の複雑さ**: monorepo 環境では、リポジトリ間を切り替える必要があり、操作の複雑さが増します
4. **ワークフローの中断**: コードをコミットすると開発の思考が中断され、コーディング効率に影響します

これらの問題は大規模プロジェクトやチーム協力環境では特に顕著です。優れた開発ツールは、開発者が核心となるコーディング作業に集中できるようにし、煩雑なコミットプロセスに悩まされるべきではありません。

### AI 支援開発のトレンド

近年、AI 技術のソフトウェア開発分野への応用はますます広がっています。コード補完、エラー検出からドキュメントの自動生成まで、AI は開発のあらゆる段階に徐々に浸透しています。Git ワークフローの面では、コミットメッセージ生成機能を提供するツールは既にいくつか存在しますが、その多くは単一のコミットシナリオに限定されており、作業ディレクトリ全体の変更に対するインテリジェントな分析とグループ化機能が不足しています。

実は HagiCode も開発プロセスでこれらの課題に直面し、様々なツールを試しましたが、いずれも多かれ少なかれ限界がありました。機能が不十分か、ユーザーエクスペリエンスが良くないかでした。これが最終的に独自に AI Compose Commit 機能を実装することに決めた理由です。

HagiCode の AI Compose Commit 機能はまさにこのギャップを埋めるために生まれました。コミットメッセージを生成するだけでなく、ファイル分析からコミット実行までの全プロセスを完全に引き継ぎます。

## Problem

### 技術的課題

AI Compose Commit 機能の実装プロセスでは、複数の技術的課題に直面しました：

1. **ファイルセマンティクスの理解**: AI はファイル変更の意味的関係を理解し、どのファイルが同じ機能モジュールに属するかを判断する必要があります。これにはファイルの内容、ディレクトリ構造、変更のコンテキストを深く分析する必要があります。

2. **コミットグループ化戦略**: 合理的なグループ化基準をどう定義するか？機能別、モジュール別、それともファイルタイプ別？異なるプロジェクトには異なる戦略が適用される場合があります。

3. **リアルタイムフィードバックと非同期処理**: Git 操作は特に大量のファイルを処理する場合、長時間を要する場合があります。ユーザーエクスペリエンスを確保しながら複雑な操作を完了するにはどうすればよいでしょうか？

4. **マルチリポジトリサポート**: monorepo アーキテクチャでは、メインリポジトリとサブリポジトリ間で操作を正しくルーティングする必要があります。

5. **エラー処理とロールバック**: あるコミットが失敗した場合、すでに実行されたコミットをどう処理するか？ステージ済みのファイルをロールバックする必要があるか？

6. **コミットメッセージの一貫性**: 生成されたコミットメッセージはプロジェクトの既存のスタイルに合致し、過去のコミットのフォーマットと一貫性を保つ必要があります。

### パフォーマンスの考慮事項

AI が大量のファイル変更を処理すると、大幅な時間と計算リソースを消費します。私たちは以下の面で最適化を行う必要があります：

- 不要な AI 呼び出しを削減する
- ファイルコンテキストの構築方法を最適化する
- 効率的な Git 操作バッチ処理を実装する

これらの問題は HagiCode の実際の使用で本当に発生し、絶え間ない反復と最適化を通じて比較的完全なソリューションを見つけました。もし同様のツールを開発しているなら、私たちの経験が何らかのインスピレーションを与えられれば幸いです。

## Solution

### 全体アーキテクチャ設計

私たちは AI Compose Commit 機能を実装するために階層アーキテクチャを採用し、システムが良好な拡張性と保守性を持つようにしました：

#### 1. API 層（Web 層）

**GitController** は機能のエントリとして `POST /api/git/auto-compose-commit` エンドポイントを提供します。ユーザーエクスペリエンスを最適化するため、Fire-and-Forget 非同期モードを採用しました：

- クライアントがリクエストを発行すると、サーバーは直ちに HTTP 202 Accepted を返します
- 実際の AI 処理はバックグラウンドで非同期実行されます
- 処理完了後、SignalR を通じてクライアントに通知します

この設計により、AI 処理に数分かかっても、ユーザーは直ちに応答を得られ、システムが停滞していると感じることがありません。

#### 2. アプリケーションサービス層（Application 層）

**GitAppService** はコアビジネスロジックを担当します：

- リポジトリ検出：monorepo でのマルチリポジトリ管理をサポート
- ロック管理：並行操作による競合を防止
- ファイルステージング調整：AI 処理フローとの相互作用
- エラーロールバック：失敗シナリオでの状態復旧

#### 3. 分散コンピューティング層（Orleans Grains）

**AIGrain** は AI 操作のコア実行単位として、`IAIGrain` インターフェースの `AutoComposeCommitAsync` メソッドを実装しています：

```csharp
// AI 自動構成コミットのインターフェースメソッドを定義
// パラメータ説明:
// - projectId: プロジェクトの一意識別子
// - unstagedFiles: 未ステージファイルリスト、ファイルパスと状態情報を含む
// - projectPath: プロジェクトルートディレクトリパス（オプション）、プロジェクトコンテキストへのアクセスに使用
// 戻り値: 実行結果を含むレスポンスオブジェクト、成功/失敗ステータスと詳細情報を含む
[Alias("AutoComposeCommitAsync")]
[ResponseTimeout("00:20:00")] // 20 分タイムアウト、大規模変更セットの処理に適用
Task<AutoComposeCommitResponseDto> AutoComposeCommitAsync(
    string projectId,
    GitFileStatusDto[] unstagedFiles,
    string? projectPath = null);
```

このメソッドは大規模変更セットを処理するため、20 分のタイムアウト時間を設定しています。HagiCode の実際の使用では、一部のプロジェクトの単一変更で数百のファイルが関係する場合があり、より長い処理時間が必要になることが分かりました。

#### 4. AI サービス層

抽象的な `IAIService` インターフェースを通じて、AI サービスのプラグ可能なアーキテクチャを実現しました。現在は Claude Helper サービスを使用していますが、他の AI プロバイダーに容易に切り替えることができます。

### コア実装ロジック

#### ファイルコンテキスト構築

AI がインテリジェントな決定を下すには、各ファイルの状態を把握する必要があります。私たちは `BuildFileChangesXml` メソッドを通じてファイルコンテキストを構築します：

```csharp
/// <summary>
/// ファイル変更の XML 表現を構築し、AI に完全なファイルコンテキスト情報を提供する
/// </summary>
/// <param name="stagedFiles">ステージ済みファイルリスト、ファイルパス、状態、旧パス（リネーム操作用）を含む</param>
/// <returns>フォーマットされた XML 文字列、すべてのファイルのメタデータ情報を含む</returns>
private static string BuildFileChangesXml(GitFileStatusDto[] stagedFiles)
{
    var sb = new StringBuilder();
    sb.AppendLine("<files>");

    foreach (var file in stagedFiles)
    {
        sb.AppendLine("  <file>");
        // XML エスケープを使用して特殊文字が XML 構造を破壊しないようにする
        sb.AppendLine($"    <path>{System.Security.SecurityElement.Escape(file.Path)}</path>");
        sb.AppendLine($"    <status>{System.Security.SecurityElement.Escape(file.Status)}</status>");

        // ファイルリネームシナリオを処理し、旧パスを記録して AI が変更関係を理解できるようにする
        if (!string.IsNullOrEmpty(file.OldPath))
        {
            sb.AppendLine($"    <oldPath>{System.Security.SecurityElement.Escape(file.OldPath)}</oldPath>");
        }

        sb.AppendLine("  </file>");
    }

    sb.AppendLine("</files>");
    return sb.ToString();
}
```

この XML フォーマットのコンテキストには、ファイルパス、状態、旧パス（リネーム操作用）が含まれ、AI に完全なメタデータを提供します。構造化された XML フォーマットにより、AI が各ファイルの状態と変更タイプを正確に理解できるようにしました。

#### AI 権限管理

AI が直接 Git 操作を実行できるようにするため、包括的なツール権限を設定しました：

```csharp
// AI が使用できるツールセットを定義、ファイル操作と Git コマンド実行権限を含む
// Read/Write/Edit: ファイルの読み書き・編集能力
// Bash(git:*): すべての Git コマンドの実行権限
// その他の Bash コマンド: ファイル内容とディレクトリ構造を確認し、AI がコンテキストを理解するのを補助
var allowedTools = new[]
{
    "Read", "Write", "Edit",
    "Bash(git:*)", "Bash(cat:*)", "Bash(ls:*)", "Bash(find:*)",
    "Bash(grep:*)", "Bash(head:*)", "Bash(tail:*)", "Bash(wc:*)"
};

// 完全な AI リクエストオブジェクトを構築
var request = new AIRequest
{
    Prompt = prompt,                          // 完全な Prompt テンプレート、タスク命令と制約条件を含む
    WorkingDirectory = projectPath ?? GetTempDirectory(), // 作業ディレクトリ、AI が正しいプロジェクトコンテキストで実行することを保証
    AllowedTools = allowedTools,               // 使用可能なツールセット
    PermissionMode = PermissionMode.bypassPermissions, // 権限チェックをバイパスし、Git 操作の直接実行を許可
    LanguagePreference = languagePreference         // 言語プリファレンス設定、ユーザーの期待に合ったコミットメッセージを生成することを保証
};
```

ここでは `PermissionMode.bypassPermissions` モードを使用し、AI がユーザー確認なしで直接 Git コマンドを実行できるようにしています。これは機能設計のコアですが、同時に乱用を防ぐための厳格な入力検証も必要です。HagiCode の実際のデプロイでは、バックエンドのパラメータ検証とログ監視を通じて、この仕組みの安全性を保証しています。

#### コミット結果解析

AI 実行完了後、構造化された結果を返します。私たちは互換性を確保するために二重解析戦略を実装しました：

```csharp
/// <summary>
/// AI が返したコミット実行結果を解析、デリミタフォーマットと正規表現フォーマットをサポート
/// </summary>
/// <param name="aiResponse">AI が返した生のレスポンス内容</param>
/// <returns>解析後のコミット結果リスト、各結果はコミットハッシュと実行状態を含む</returns>
private List<CommitResultDto> ParseCommitExecutionResults(string aiResponse)
{
    var results = new List<CommitResultDto>();

    // デリミタ解析を優先（新フォーマット）、このフォーマットの方が明確で信頼性が高い
    if (aiResponse.Contains("---"))
    {
        logger.LogDebug("Using delimiter-based parsing for AI response");
        results = ParseDelimitedFormat(aiResponse);

        if (results.Count > 0)
        {
            return results; // 解析成功、結果を直接返す
        }

        logger.LogWarning("Delimiter-based parsing produced no results, falling back to regex");
    }
    else
    {
        logger.LogDebug("No delimiter found, using legacy regex-based parsing");
    }

    // 正規表現解析にフォールバック（旧フォーマット）、後方互換性を確保
    return ParseLegacyFormat(aiResponse);
}
```

デリミタフォーマットは `---` をコミット間の区切りとして使用し、フォーマットが明確で解析が容易です：

```
---
Commit 1: abc123def456
feat(auth): add user login functionality

Implement JWT-based authentication with login form and API endpoints.

Co-Authored-By: Hagicode <noreply@hagicode.com>
---
Commit 2: 789ghi012jkl
docs(readme): update installation instructions

Add new setup steps for Docker environment.

Co-Authored-By: Hagicode <noreply@hagicode.com>
---
```

このフォーマット設計により、解析がシンプルで信頼性が高くなり、人間の読みやすさも確保されています。

### ロック管理仕組み

並行操作による状態競合を防ぐため、リポジトリロック仕組みを実装しました：

```csharp
// リポジトリロックを取得、並行操作を防止
// パラメータ説明:
// - fullPath: リポジトリの完全パス、異なるリポジトリインスタンスを識別するために使用
// - requestedBy: リクエスト者識別子、追跡とログ記録に使用
await _autoComposeLockService.AcquireLockAsync(fullPath, requestedBy);

try
{
    // AI Compose Commit 操作を実行
    // この部分のコードは Orleans Grain のメソッドを呼び出し、実際の AI 処理と Git 操作を実行
    await aiGrain.AutoComposeCommitAsync(projectId, unstagedFiles, projectPath);
}
finally
{
    // ロックが確実に解放されるように、操作の成功または失敗に関わらず
    // finally ブロックを使用すると、例外状況でもロックを解放でき、デッドロックを回避できる
    await _autoComposeLockService.ReleaseLockAsync(fullPath);
}
```

ロックには 20 分のタイムアウトがあり、AI 操作のタイムアウト設定と一致しています。操作が失敗またはタイムアウトすると、システムは自動的にロックを解放し、永続的なブロックを回避します。HagiCode の実際の使用では、このロック仕組みが非常に重要であることが分かりました。特にチーム協力環境では、複数の開発者が同時に AI Compose Commit 操作をトリガーする可能性があるためです。

### SignalR リアルタイム通知

処理完了後、システムは SignalR を通じてフロントエンドに通知を送信します：

```csharp
/// <summary>
/// 自動構成コミット完了の通知を送信
/// </summary>
/// <param name="projectId">プロジェクト識別子、正しいクライアントに通知をルーティングするために使用</param>
/// <param name="totalCount">総コミット数、成功と失敗を含む</param>
/// <param name="successCount">成功したコミット数</param>
/// <param name="failureCount">失敗したコミット数</param>
/// <param name="success">全体操作の成功フラグ</param>
/// <param name="error">エラー情報（操作が失敗した場合）</param>
private async Task SendAutoComposeCommitNotificationAsync(
    string projectId,
    int totalCount,
    int successCount,
    int failureCount,
    bool success,
    string? error)
{
    try
    {
        // 詳細な実行結果を含む通知データ転送オブジェクトを構築
        var notification = new AutoComposeCommitCompletedDto
        {
            ProjectId = projectId,
            TotalCount = totalCount,
            SuccessCount = successCount,
            FailureCount = failureCount,
            Success = success,
            Error = error
        };

        // SignalR Hub を通じてすべての接続されたクライアントに通知をブロードキャスト
        await messageService.SendAutoComposeCommitCompletedAsync(notification);

        logger.LogInformation(
            "Auto compose commit notification sent for project {ProjectId}: {SuccessCount}/{TotalCount} succeeded",
            projectId, successCount, totalCount);
    }
    catch (Exception ex)
    {
        // 通知エラーを記録するが、主操作フローには影響しない
        // 通知失敗は全体操作の失敗原因となるべきではない
        logger.LogError(ex, "Failed to send auto compose commit notification for project {ProjectId}", projectId);
    }
}
```

フロントエンドは通知を受け取った後、UI を更新してコミット成功または失敗の状態を表示し、ユーザーエクスペリエンスを向上させます。このリアルタイムフィードバック仕組みは HagiCode の使用で良好なユーザーフィードバックを得ており、ユーザーは操作がいつ完了し、結果がどうだったかを明確に知ることができます。

## Implementation

### Prompt エンジニアリング設計

AI の振る舞いは Prompt によって完全に決まります。私たちは Auto Compose Commit の Prompt テンプレートを入念に設計しました。日本語版を例に挙げます（`auto-compose-commit.ja-JP.hbs`）：

#### 非インタラクティブモードサポート

Prompt の冒頭で非インタラクティブ実行モードをサポートしていることを明確に宣言しています。これは CI/CD と自動化スクリプトの重要な要件です：

```handlebars
**重要提示**:このプロンプトは非インタラクティブ環境（CI/CD、自動化スクリプトなど）で実行される場合があります。

**非インタラクティブモード**:
- AskUserQuestion やすべてのインタラクティブツールの使用を禁止
- ユーザー入力が必要な場合:
  - 合理的なデフォルト値を使用（コミットタイプは feat など）
  - オプションの確認ステップをスキップ
  - 行った仮定を記録
```

この設計により、AI Compose Commit 機能がインタラクティブな IDE 環境だけでなく、CI/CD パイプラインにも統合でき、完全自動化されたコミットフローを実現できます。

#### ブランチ保護仕組み

AI が危険な操作を実行するのを防ぐため、Prompt に厳格なブランチ保護ルールを追加しました：

```handlebars
**ブランチ保護**:
- すべてのブランチ切り替え操作（git checkout、git switch）を禁止
- すべての git commit コマンドは現在のブランチで実行する必要があります
- ブランチの作成、削除、リネームを禁止
- 未追跡ファイルや未ステージ変更を変更しないでください
- ブランチ切り替えが必要な操作では、実行せずにエラーを返すべきです
```

これらのルールは AI のツール使用範囲を制約することで、操作の安全性を確保します。HagiCode の実際のテストでは、これらの制約の有効性を検証しました。AI はブランチ切り替えが必要なシナリオに遭遇すると、危険な操作を実行せず、安全にエラーを返します。

#### インテリジェントグループ化決定木

Prompt にはファイルグループ化の決定ロジックが詳細に定義されています：

```handlebars
**ファイルグループ化決定木**:
├── 設定ファイル（package.json、tsconfig.json、.env など）ですか？
│   ├── はい → 独立コミット（タイプ：chore または build）
│   └── いいえ → 続行
├── ドキュメントファイル（README.md、*.md、docs/**）ですか？
│   ├── はい → 独立コミット（タイプ：docs）
│   └── いいえ → 続行
├── 同一機能に関連していますか？
│   ├── はい → 同一コミットにマージ
│   └── いいえ → それぞれコミット
└── モジュール間変更ですか？
    ├── はい → モジュール別にグループ化
    └── いいえ → 機能別にグループ化
```

この決定木は AI に明確なグループ化ロジックを提供し、生成されたコミットが意味的に合理的であることを保証します。HagiCode の実際の使用では、この決定木がほとんどの一般的なシナリオを処理でき、生成されるグループ化結果が開発者の期待に合うことが分かりました。

#### 履歴フォーマット一貫性解析

コミットメッセージをプロジェクト履歴と一貫性させるため、Prompt は AI に生成前に最近のコミット履歴を分析するよう要求します：

```handlebars
**履歴フォーマット一貫性**:コミットメッセージを生成する前に、現在のリポジトリのコミット履歴を分析して既存のスタイルに一致させる**必要があります**

1. git log -n 15 --pretty=format:"%H|%s|%b%n---%n" を使用して最近のコミット履歴を取得
2. コミットを分析して以下を識別:
   - 構造パターン：プロジェクトは複数段落を使用していますか？"Changes:" や "Capabilities:" セクションがありますか？
   - 言語パターン：コミットメッセージは英語、日本語、それとも混合？
   - 一般的なタイプ：最も頻繁に使用されるコミットタイプ（feat、fix、docs など）
   - 特殊フォーマット：Co-Authored-By 行はありますか？その他のプロジェクト特有の規約？
3. 検出されたパターンに従うコミットメッセージを生成
```

この分析により、AI が生成するコミットメッセージが唐突に見えず、プロジェクトのコミット履歴とスタイルが一貫していることを保証します。HagiCode の多言語プロジェクトでは、この機能が特に重要であり、プロジェクトのコミット履歴に基づいて自動的に適切な言語とフォーマットを選択できます。

#### Co-Authored-By 要件

各コミットには Co-Authored-By 情報を含める必要があります：

```handlebars
**重要**:各コミットには Co-Authored-By 情報を追加する必要があります
- 以下のフォーマットを使用：git commit -m "type(scope): subject" -m "" -m "Co-Authored-By: Hagicode <noreply@hagicode.com>"
- またはコミットメッセージに直接 Co-Authored-By 行を含める
```

これは貢献規約のためだけでなく、AI 支援のコミット履歴を追跡するためでもあります。HagiCode はこの要件を強制ルールとしており、AI が生成するすべてのコミットに明確なソース識別子が含まれることを保証しています。

### ワークフロー詳細解説

完全な AI Compose Commit ワークフローは以下の通りです：

1. **ユーザートリガー**: ユーザーが Git Status パネルまたは Quick Actions Zone で「AI Auto Compose Commit」ボタンをクリック
2. **API リクエスト**: フロントエンドが POST リクエストを `/api/git/auto-compose-commit` エンドポイントに送信
3. **即時レスポンス**: サーバーが HTTP 202 Accepted を返し、処理完了を待たない
4. **バックグラウンド処理**:
   - GitAppService がリポジトリロックを取得
   - AIGrain の AutoComposeCommitAsync メソッドを呼び出し
   - ファイルコンテキスト XML を構築
   - AI Prompt を実行し、AI に分析とコミット実行を指示
5. **AI 実行**:
   - Git コマンドを使用してすべての未ステージ変更を取得
   - ファイル内容を読み取り変更の性質を理解
   - 意味的関係に基づいてファイルをグループ化
   - 各グループに対して `git add` と `git commit` 操作を実行
6. **結果解析**: AI が返した実行結果を解析
7. **通知送信**: SignalR を通じてフロントエンドに通知
8. **ロック解放**: 成功または失敗に関わらず、リポジトリロックを解放

このフローの設計により、ユーザーは操作を開始した後、AI 処理の完了を待たずにすぐに他の作業を続けることができます。HagiCode のユーザーフィードバックによると、この非同期処理方式はワークフロー体験を大幅に向上させました。

### エラー処理仕組み

私たちは複数のエラー処理を実装しました：

#### 1. 入力検証

```csharp
// リクエストパラメータの有効性を検証、無効なリクエストがバックエンド処理ロジックに到達するのを防止
if (request.UnstagedFiles == null || request.UnstagedFiles.Count == 0)
{
    return BadRequest(new
    {
        message = "No unstaged files provided. Please make changes in the working directory first.",
        status = "validation_failed"
    });
}
```

#### 2. エラーロールバック

AI 処理中にエラーが発生した場合、システムはロールバック操作を実行し、ステージ済みのファイルのステージを解除して、不整合な状態が残らないようにします。この仕組みは HagiCode の実際の使用で何度も予期せぬ中断を救い、リポジトリ状態の完全性を保証しました。

#### 3. タイムアウト処理

20 分のタイムアウト設定により、長時間実行される操作がリソースを無期限にブロックしないようにしています。タイムアウト後、システムはロックを解放し、ユーザーに操作失敗を通知します。HagiCode の実際の使用では、大部分の操作が 2-5 分以内に完了し、超大規模変更セットを処理する場合にのみタイムアウト制限に近づくことが分かりました。

## Best Practices

### AI Compose Commit の使用ベストプラクティス

#### 1. 適切な使用タイミング

AI Compose Commit は以下のシナリオに最適です：

- 1日の作業終了後、複数のファイルの変更を一括処理
- リファクタリング操作後、複数の関連ファイルをそれぞれコミット
- 機能開発完了後、関連する変更をグループ化してコミット

以下のシナリオには適していません：

- 単一ファイルの迅速なコミット（通常のコミットを使用する方が速い）
- コミット内容を正確に制御する必要があるシナリオ
- 機密情報を含むコミット（手動レビューが必要）

#### 2. AI が生成したコミットをレビュー

AI のインテリジェントなグループ化は強力ですが、開発者は生成されたコミットをレビューする必要があります：

- コミットのグループ化が期待通りか確認
- コミットメッセージの正確性を検証
- 漏れや誤って含まれたファイルがないか確認

不合理なグループ化が見つかった場合、`git reset --soft HEAD~N` で取り消してから再グループ化できます。HagiCode の経験では、AI のグループ化がインテリジェントであっても、人的レビューには価値があります。特に重要な機能コミットではそうです。

#### 3. プロジェクト規約との連携

プロジェクトの Git 設定が Conventional Commits をサポートしていることを確認してください：

```bash
# commitlint をインストール
npm install -g @commitlint/cli @commitlint/config-conventional

# commitlint を設定
echo "module.exports = {extends: ['@commitlint/config-conventional']}" > commitlint.config.js
```

このようにすることで、CI/CD パイプラインでコミットメッセージフォーマットを検証し、AI Compose Commit が生成するフォーマットと一貫性を保つことができます。

### 類似機能実装の提案

プロジェクトで同様の AI 支援コミット機能を実装したい場合、以下が私たちの提案です：

#### 1. 小規模から開始

まず単一のコミットメッセージ生成を実装し、徐々にマルチコミットグループ化機能に拡張します。この方が検証と反復が容易です。HagiCode もこのパスに沿って徐々に機能を改良しました。初期バージョンは単一コミットのみをサポートし、後にマルチコミットのインテリジェントグループ化に拡張しました。

#### 2. 成熟した AI SDK を使用

AI 呼び出しロジックを自分で実装せず、既存の SDK を使用すると開発時間と潜在的なバグを削減できます。私たちは Claude Helper サービスを使用しており、安定したインターフェースと完全なエラー処理を提供しています。

#### 3. Prompt 設計を重視

Prompt の品質が AI 出力の品質を直接決定します。詳細な Prompt 設計には時間を投資しましょう：

- 明確なタスク記述
- 具体的な出力フォーマット要求
- 境界状況の処理ルール
- サンプル説明

HagiCode は Prompt 設計に多くの時間を投資しており、これは機能成功の重要な要素の 1 つです。

#### 4. 包括的なエラー処理を実装

AI 操作は様々な理由で失敗する可能性があります（ネットワーク問題、API 制限、コンテンツ審査など）。システムがこれらのエラーを優雅に処理し、意味のあるエラー情報を提供できるようにしてください。

#### 5. 手動介入仕組みを提供

完全自動化せず、ユーザーに制御権を残してください。グループ化結果の確認、グループ化調整、手動コミットメッセージ編集などのオプションを提供し、自動化と柔軟性のバランスを取ります。HagiCode は自動実装を実装しましたが、プレビューと調整の能力も保持しています。

### パフォーマンス最適化のヒント

#### 1. ファイルフィルタリング

ファイルコンテキスト構築時、AI 分析が不要なファイルをフィルタリングします：

```csharp
// 自動生成されたファイルと大きすぎるファイルをフィルタリング、AI 処理負担を軽減
var relevantFiles = stagedFiles
    .Where(f => !IsGeneratedFile(f.Path))
    .Where(f => !IsLargeFile(f.Path))
    .ToArray();
```

#### 2. 並列処理

複数の独立したリポジトリをサポートする場合、異なるリポジトリのコミットを並列処理して、全体の効率を向上させることができます。

#### 3. キャッシュ最適化

プロジェクトのコミット履歴分析結果をキャッシュし、毎回再分析することを避けます。設定ファイルに履歴フォーマットプリファレンスを保存し、AI 呼び出し回数を削減できます。

## Conclusion

AI Compose Commit 機能は、ソフトウェア開発ツールにおける AI 技術の深層応用を表しています。ファイル変更のインテリジェントな分析、自動グループ化コミット、規範的なコミットメッセージ生成を通じて、Git ワークフローの効率を大幅に向上させ、開発者が核心となるコーディング作業に集中できるようにします。

実装プロセスでは、いくつかの重要な経験を学びました：

1. **ユーザーフィードバックが鍵**: 初期バージョンでは同期待機方式を採用していましたが、ユーザーフィードバックで体験が良くないことが分かり、Fire-and-Forget モードに変更した後、満足度が大幅に向上しました
2. **Prompt 設計が品質を決定**: 入念に設計された Prompt は、複雑なアルゴリズムよりも AI 出力の品質を保証できます
3. **安全性は常に第一**: AI に Git コマンドを直接実行する権限を与えることで効率向上が得られますが、厳格な制約と検証と組み合わせる必要があります
4. **漸進的改善**: 単純なシナリオから開始し、徐々に複雑さを増す方が、すべての機能を一度に実装するよりも成功しやすいです

今後、AI Compose Commit 機能をさらに最適化する計画があります：

- コードレビュープロセスを統合し、コミット前に自動的にレビューをトリガー
- カスタムコミットメッセージテンプレートをサポートし、異なるプロジェクトの個別ニーズを満たす
