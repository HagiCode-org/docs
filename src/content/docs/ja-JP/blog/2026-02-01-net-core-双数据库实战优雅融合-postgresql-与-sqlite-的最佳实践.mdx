## .NET Core デュアルデータベース実践：PostgreSQL と SQLite の平和的共存

> モダンアプリケーション構築において、開発環境では軽量・利便性を、本番環境では高並行性と高可用性を求めることがよくあります。本記事では、.NET Core プロジェクトで PostgreSQL と SQLite を同時にエレガントにサポートし、「開発は SQLite、本番は PostgreSQL」というベストプラクティスを実現する方法を紹介します。


## 背景

ソフトウェア開発において、環境の差異はエンジニアリングチームを長年悩ませてきた課題の一つです。私たちが構築している **HagiCode** プラットフォームを例に挙げましょう。これは ASP.NET Core 10 と React に基づく AI 補助開発システムで、内部で Orleans を統合して分散状態管理を行っており、技術スタックはモダンでかなり複雑です。

プロジェクト初期、私たちは典型的なエンジニアリングの痛点に直面しました。開発者はローカル環境をすぐに使えるようにしたい一方で、重い PostgreSQL データベースのインストールと設定はしたくありません。しかし本番環境では、高並行性の書き込みと複雑な JSON クエリを処理する必要があり、軽量な SQLite では力不足です。

コードベースを統一しつつ、SQLite の携帯性を活かしつつ、PostgreSQL の強力なパフォーマンスも享受するにはどうすればよいのでしょうか。これが本記事で探る核心的な問いです。

## コアトピック 1：アーキテクチャ設計と統一抽象化

.NET Core でデュアルデータベースをサポートするための核となる考え方は、「具象実装ではなく抽象に依存する」ことです。データベースの選択をビジネスコードから切り離し、設定層に決定を委ねる必要があります。

### 設計アプローチ

1.  **統一インターフェース**：すべてのビジネスロジックは特定の `PostgreSqlDbContext` ではなく、`DbContext` 基底クラスまたはカスタムインターフェースに依存する必要があります。
2.  **設定駆動**：`appsettings.json` の設定項目を使用し、アプリケーション起動時に動的に読み込むデータベースプロバイダーを決定します。
3.  **機能分離**：PostgreSQL 固有の機能（JSONB など）に適応ロジックを追加し、SQLite でも正常に動作するようにします。

### コード実装：動的コンテキスト設定

ASP.NET Core の `Program.cs` では、`UseNpgsql` や `UseSqlite` をハードコードすべきではありません。代わりに設定を読み込み、動的に決定します。

まず、設定クラスを定義します。

```csharp
public class DatabaseSettings
{
    public const string SectionName = "Database";

    // データベースタイプ：PostgreSQL または SQLite
    public string DbType { get; set; } = "PostgreSQL";

    // 接続文字列
    public string ConnectionString { get; set; } = string.Empty;
}
```

次に、`Program.cs` で設定に基づいてサービスを登録します。

```csharp
// 設定を読み込む
var databaseSettings = builder.Configuration.GetSection(DatabaseSettings.SectionName).Get<DatabaseSettings>();

// DbContext を登録
builder.Services.AddDbContext<ApplicationDbContext>(options =>
{
    if (databaseSettings?.DbType?.ToLower() == "sqlite")
    {
        // SQLite 設定
        options.UseSqlite(databaseSettings.ConnectionString);

        // SQLite の並行書き込み制限の処理
        // 注意：本番環境では WAL モードの有効化をお勧めします
    }
    else
    {
        // PostgreSQL 設定（デフォルト）
        options.UseNpgsql(databaseSettings.ConnectionString, npgsqlOptions =>
        {
            // JSONB サポートを有効化。AI 会話記録の処理に非常に有用
            npgsqlOptions.UseJsonNet();
        });

        // 接続プールのリトライポリシーを設定
        options.EnableRetryOnFailure(3);
    }
});
```

## コアトピック 2：差異の処理とマイグレーション戦略

PostgreSQL と SQLite はどちらも SQL 標準をサポートしていますが、特定の機能と動作に大きな違いがあります。これらの差異を適切に処理しないと、「ローカルでは動くのに本番でエラーになる」という事態になりかねません。

### 1. JSON 型の処理

HagiCode では、大量のプロンプトと AI メタデータを保存する必要があり、通常 JSON カラムが関わります。
- **PostgreSQL**：ネイティブの `JSONB` 型を持ち、クエリパフォーマンスが優れています。
- **SQLite**：ネイティブの JSON 型がありません（新しいバージョンには JSON1 拡張がありますが、オブジェクトマッピングはまだ異なります）。通常は TEXT として保存されます。

**解決策**：
EF Core のエンティティマッピングで変換可能な型として設定します。

```csharp
protected override void OnModelCreating(ModelBuilder modelBuilder)
{
    base.OnModelCreating(modelBuilder);

    // エンティティの設定
    modelBuilder.Entity<PromptTemplate>(entity =>
    {
        entity.Property(e => e.Metadata)
              .HasColumnType("jsonb") // PG は jsonb を使用
              .HasConversion(
                  v => JsonSerializer.Serialize(v, (JsonSerializerOptions)null),
                  v => JsonSerializer.Deserialize<Dictionary<string, object>>(v, (JsonSerializerOptions)null)
              );
    });
}
```

SQLite を使用する場合、`HasColumnType("jsonb")` は無視されるか警告を発生します。しかし `HasConversion` が設定されているため、データは正しくシリアライズ・デシリアライズされ、TEXT フィールドに保存されることで互換性が確保されます。

### 2. マイグレーション戦略の分離

同じマイグレーションスクリプトを PostgreSQL と SQLite の両方で動作させようとしてはいけません。主キー生成戦略、インデックス構文などの違いにより、必ず失敗します。

**推奨プラクティス**：
2 つのマイグレーションブランチまたはプロジェクトを保守します。HagiCode の開発フローでは以下のように処理しています。

1.  **開発段階**：主に SQLite で作業します。`Add-Migration Init_Sqlite -OutputDir Migrations/Sqlite` を使用。
2.  **適応段階**：機能開発後、接続文字列を PostgreSQL に切り替え、`Add-Migration Init_Postgres -OutputDir Migrations/Postgres` を実行。
3.  **自動化スクリプト**：現在の環境変数に基づいて正しいマイグレーションを自動適用するシンプルな PowerShell または Bash スクリプトを作成。

```bash
# シンプルなデプロイロジックの擬似コード
if [ "$DATABASE_PROVIDER" = "PostgreSQL" ]; then
    dotnet ef database update --project Migrations.Postgres
else
    dotnet ef database update --project Migrations.Sqlite
fi
```

## コアトピック 3：HagiCode の実践経験のまとめ

**HagiCode** を単一データベースからデュアルデータベースサポートにリファクタリングする過程で、いくつかの落とし穴を経験し、皆さんが同じ間違いを避けられるよう重要な教訓をまとめました。

### 1. 並行性とトランザクションの違い

PostgreSQL はサーバー・クライアントアーキテクチャで、強力なトランザクション分離レベルを備えた高並行性書き込みをサポートします。SQLite はファイルロックを使用し、WAL モードを有効にしない限り書き込み操作がデータベースファイル全体をロックします。

**推奨事項**：
頻繁な書き込みを伴うビジネスロジック（例：ユーザーの編集状態のリアルタイム保存）を記述する際は、SQLite のロックモデルを考慮する必要があります。**HagiCode** の OpenSpec 協作モジュールの設計では、「書き込み前マージ」メカニズムを導入し、データベースへの直接書き込み頻度を減らすことで、両方のデータベースで良好なパフォーマンスを維持しました。

### 2. 接続文字列のライフサイクル管理

PostgreSQL の接続確立は比較的高コストで、接続プールに依存します。SQLite の接続は非常に軽量ですが、適切に解放しないとファイルロックが後続操作のタイムアウトを引き起こす可能性があります。

`Program.cs` で各データベースに応じた細かな調整が可能です。

```csharp
if (databaseSettings?.DbType?.ToLower() == "sqlite")
{
    // SQLite：接続を開いたままにするとパフォーマンスが向上しますが、ファイルロックに注意
    options.UseSqlite(connectionString, sqliteOptions =>
    {
        // コマンドタイムアウトの設定
        sqliteOptions.CommandTimeout(30);
    });
}
else
{
    // PG：接続プールを活用
    options.UseNpgsql(connectionString, npgsqlOptions =>
    {
        npgsqlOptions.MaxBatchSize(100);
        npgsqlOptions.CommandTimeout(30);
    });
}
```

### 3. テストカバレッジの重要性

多くの開発者（私たちのチームの初期メンバーを含む）がよく犯す間違いがあります。開発環境（通常は SQLite）でしか単体テストを実行しないことです。

HagiCode の CI/CD パイプラインでは、すべての Pull Request で PostgreSQL の統合テストを実行する GitHub Action ステップを強制的に組み込みました。

```yaml
# .github/workflows/test.yml の例
- name: Run Integration Tests (PostgreSQL)
  run: |
    docker-compose up -d db_postgres
    dotnet test --filter "Category=Integration"
```

これにより、SQL 構文の差異や大文字小文字の区別に関連する無数のバグを未然に防ぐことができました。

## まとめ

抽象化レイヤーと設定駆動の依存性注入を導入することで、**HagiCode** プロジェクトで PostgreSQL と SQLite の「二本柱」運用を成功裏に実現しました。これにより、PostgreSQL のインストールが不要になり、新規開発者の参入障壁が大幅に低下すると同時に、本番環境には強固なパフォーマンス保証を提供しました。

重要ポイントの振り返り：
1.  **抽象化が最優先**：ビジネスコードは具象的なデータベース実装に依存すべきではありません。
2.  **設定の分離**：開発と本番で別々の `appsettings.json` を使用します。
3.  **マイグレーションの分離**：一つの Migration をどこでも使おうとしてはいけません。
4.  **機能の段階的劣化**：SQLite では互換性を、PostgreSQL ではパフォーマンスを優先します。

このアーキテクチャパターンは HagiCode だけでなく、軽量な開発環境と重量級の本番環境のバランスを取る必要があるすべての .NET プロジェクトに適用できます。
