## 安価なクラウドサーバーを使ってダウンロード加速ディストリビューションサイトを構築する方法

> クラウドストレージのトラフィック料金は信じられないほど高く、国際アクセスはイライラするほど遅く、CDNの価格は躊躇するほど高い......これらの苦しみは、おそらく配信をやる人なら誰でもわかるでしょう。本文では、[HagiCode](https://hagicode.com) プロジェクトで探り出した低コストソリューションを紹介します。クラウドサーバー + Nginx キャッシュ層、コストは半分になり、速度は向上しました。これもほんの少しの慰めでしょう。

## 背景

インターネットにおいて、ダウンロード速度と安定性は、結局のところユーザー体験です。オープンソースでも商用でも、ユーザーに信頼できるダウンロード方法を提供する必要があります。

クラウドストレージ（Azure Blob Storage、AWS S3 など）から直接ファイルをダウンロードするのは、簡単に見えますが、実際には問題が少なくありません：

**ネットワーク遅延**：国や地域を越えたアクセスは、キーボードを叩きたくなるほど遅いです。ユーザーが待つ間に花が咲いてしまいます。体験が良くなるわけがありません。

**帯域幅コスト**：クラウドストレージの出口トラフィックは、見ていられないほど高いです。Azure Blob Storage に中国本土からアクセスすると、約 1GB あたり 0.5 元で、1ヶ月 1TB なら 500 元です。小規模チームにとって、この金額は多いとも少ないとも言えます。結局、誰のお金も風から吹いてきたものではありません。

**アクセス制限**：一部の地域では海外クラウドサービスへのアクセスが不安定で、時にはまったくアクセスできないこともあります。ユーザーがダウンロードしたいのにダウンロードできない、これは無力な話です。

**CDN コスト**：商用 CDN は確実に問題を解決できますが、価格も確実に高いです。小規模チームが使えるわけがありません。

では、お金を節約しながら使いやすい方法はないのでしょうか？ 実際にはあります。クラウドサーバー + リバースプロキシ + キャッシュ層、これだけで単純粗暴です。コストは約半分になり、速度も向上しました。これもほんの少しの慰めでしょう。

## HagiCode について

このソリューションは突然思いついたものではなく、[HagiCode](https://hagicode.com) プロジェクトで試行錯誤した経験です。

HagiCode は AI コードアシスタントで、サーバー側とデスクトップ側のダウンロードサービスをユーザーに提供する必要があります。開発者向けのツールとして、世界中のユーザーが高速かつ安定してダウンロードできることは、本来とても重要です。これが、低コストソリューションを考え出さなければならなかった理由でもあります——結局、誰のお金も風から吹いてきたものではありません。

このソリューションが価値があると思うなら、私たちのエンジニアリング力はまあまあということでしょう......そうであれば、HagiCode 自体も注目に値するのでは？

## アーキテクチャ設計

### 全体アーキテクチャの考え方

まず、全体のアーキテクチャ設計を見てみましょう：

```
ユーザーリクエスト
    ↓
DNS 解決
    ↓
┌─────────────────────────────────────┐
│   リバースプロキシ層 (Traefik/Bunker Web)   │ ← SSL 終了、ルーティング配信、セキュリティ保護
├─────────────────────────────────────┤
│   ポート: 80/443                       │
│   機能: 自動 Let's Encrypt 証明書      │
│         Host ルーティング                    │
└─────────────────────────────────────┘
    ↓
┌─────────────────────────────────────┐
│   キャッシュ層 (Nginx)                     │ ← ファイルキャッシュ、Gzip 圧縮
├─────────────────────────────────────┤
│   ポート: 8080(server) / 8081(desktop) │
│   キャッシュ戦略:                          │
│   - index.json: 1 時間               │
│   - 他のファイル: 7 日                   │
│   キャッシュサイズ: 1GB                      │
└─────────────────────────────────────┘
    ↓
┌─────────────────────────────────────┐
│   オリジンサーバー (Azure Blob Storage)         │ ← ファイルストレージ
└─────────────────────────────────────┘
```

このアーキテクチャの核心思想は、端的に言えば：**ユーザーとクラウドストレージの間に、キャッシュ層を追加するだけ**です。

ユーザーリクエストはまずクラウドサーバー上のリバースプロキシ層に到達し、その後 Nginx キャッシュ層が引き継ぎます。キャッシュにファイルがあれば？ 直接ユーザーに提供します。なければ？ クラウドストレージから取得し、ついでにローカルにも保存します。次回同じファイルにアクセスするときは、クラウドストレージを煩わせる必要がありません。実際には記憶のようなもので、覚えておけば次回はまた考えなくて済みます......

### なぜこのアーキテクチャを選択したのか？

**クラウドサーバーの利点**：
- コスト制御可能：Alibaba Cloud などが安価なクラウドサーバーを提供（1-2 コア 2GB 構成で月額約 50-100 元）
- 柔軟なデプロイ：リバースプロキシ、キャッシュ戦略を自由に設定可能
- 地理的な柔軟性：ユーザーに近いサーバーリージョンを選択可能
- 拡張性が高い：トラフィック需要に応じて構成をアップグレード可能

**リバースプロキシ + キャッシュアーキテクチャ**：
- オリジンサーバーの負荷軽減：ホットファイルをキャッシュし、クラウドストレージへのアクセスを削減
- コスト削減：クラウドサーバーのトラフィック料金はクラウドストレージの出口トラフィックよりはるかに低い
- 速度向上：近接アクセス、サーバーの帯域幅は通常クラウドストレージより優れている

**なぜキャッシュ層に Nginx を選択したのか？**

これは適当に選んだわけではなく、Nginx には確かに理由があります：

1. 高性能：Nginx リバースプロキシの性能は、業界で周知の事実です
2. キャッシュの成熟度：組み込みの proxy_cache 機能は安定して信頼性が高く、頻繁に問題を起こしません
3. リソース占有が低い：256MB メモリで動作し、サーバーにも優しい
4. 柔軟な設定：異なるファイルタイプに異なるキャッシュ戦略を設定でき、これは親切です

## リバースプロキシ層：Traefik vs Bunker Web

HagiCode のデプロイソリューションは、実際には 2 種類のリバースプロキシをサポートしています。それぞれに特徴があります：

| 方案 | 特徴 | 適用シナリオ |
|------|------|----------|
| **Traefik** | 軽量、自動 SSL、設定簡単 | 基本デプロイ、低トラフィックシナリオ |
| **Bunker Web** | 組み込み WAF、DDoS 防御、クローラー防御 | 高セキュリティ要件、高トラフィックシナリオ |

### Traefik：軽量の第一選択

Traefik はモダンな HTTP リバースプロキシとロードバランサーで、最大の特徴は設定が簡単で、Let's Encrypt 証明書を自動的に取得できることです。

初期デプロイまたはトラフィックが多くないシナリオでは、Traefik はかなりの良い選択です：
- リソース占有が少ない（1.5 CPU/512MB メモリで十分）
- SSL 証明書の自動設定、自分で心配する必要がない
- Docker ラベルベースのルーティング設定、これも便利

### Bunker Web：高セキュリティシナリオ

Bunker Web は Nginx ベースの Web アプリケーションファイアウォールで、セキュリティ保護がより包括的です。

いつ Bunker Web に切り替えることを検討できるでしょうか？ おそらくこれらの状況です：
- DDoS 攻撃を受けている（誰も遭遇したくないですが）
- ModSecurity 保护が必要
- クローラー防御機能が欲しい
- セキュリティに対してより高い要件がある

HagiCode は `switch-deployment.sh` スクリプトを提供しており、2 つのソリューション間で迅速に切り替えることができます：

```bash
# Bunker Web に切り替え
./switch-deployment.sh bunkerweb

# Traefik に切り戻し
./switch-deployment.sh traefik

# 現在のステータスを表示
./switch-deployment.sh status
```

スクリプトは自動的に事前チェック、ヘルスチェックを実行し、自動ロールバックもできます。切り替えプロセスも安全で信頼性が高く、切り替えている最中に落ちることはありません。

## Nginx キャッシュ層の設定

キャッシュ層はアーキテクチャ全体の核心であり、Nginx の設定が良いか悪いかで、キャッシュ効果は天と地の差があります。結局、これはかなり重要なことです。

### キャッシュパスの設定

```nginx
# キャッシュパス設定
proxy_cache_path /var/cache/nginx levels=1:2 keys_zone=azure_cache:10m
                   max_size=1g inactive=7d use_temp_path=off;
```

パラメータの説明：
- `levels=1:2`：キャッシュディレクトリ階層、2 階層ディレクトリ構造、ファイルアクセス効率を向上
- `keys_zone=azure_cache:10m`：キャッシュキー記憶領域、10MB で大量のキーを記憶するのに十分
- `max_size=1g`：最大キャッシュサイズ 1GB
- `inactive=7d`：キャッシュファイルが 7 日間アクセスされないと削除
- `use_temp_path=off`：キャッシュディレクトリに直接書き込み、パフォーマンスを向上

### 階層的キャッシュ戦略

異なるタイプのファイルには異なるキャッシュ戦略が必要です：

```nginx
# Server ダウンロードサービス
server {
    listen 8080;

    # index.json 短期キャッシュ（タイムリーな更新のため）
    location /index.json {
        proxy_cache azure_cache;
        proxy_cache_valid 200 1h;
        proxy_cache_key "$scheme$server_port$request_uri";
        add_header X-Cache-Status $upstream_cache_status;
        add_header Cache-Control "public, max-age=3600";

        # Azure OSS へのリバースプロキシ
        proxy_pass https://${SERVER_DL_HOST}/${SERVER_DL_CONTAINER}$uri?${SERVER_DL_SAS_TOKEN};
        proxy_ssl_server_name on;
        proxy_ssl_protocols TLSv1.2 TLSv1.3;
    }

    # インストーラーなどの静的ファイル長期キャッシュ
    location / {
        proxy_cache azure_cache;
        proxy_cache_valid 200 7d;
        proxy_cache_key "$scheme$server_port$request_uri";
        add_header X-Cache-Status $upstream_cache_status;
        add_header Cache-Control "public, max-age=604800";

        proxy_pass https://${SERVER_DL_HOST}/${SERVER_DL_CONTAINER}$uri?${SERVER_DL_SAS_TOKEN};
        proxy_ssl_server_name on;
        proxy_ssl_protocols TLSv1.2 TLSv1.3;
    }
}
```

**なぜこのように設計したのか？**

`index.json` はバージョンチェックファイルであり、タイムリーに更新する必要があるため、キャッシュ時間は 1 時間に設定しました。こうすることで、新しいバージョンをリリースした後、最大 1 時間でユーザーが更新を検出でき、それほど長くはありません。

インストーラーなどの静的ファイルは、そもそも変化が少なく、7 日間キャッシュすればオリジンサーバーへのアクセスを大幅に削減できます。更新が必要なときは手動でキャッシュをクリアすればよく、それほど面倒ではありません。

**X-Cache-Status レスポンスヘッダー**：

このレスポンスヘッダーでキャッシュヒット状況を確認でき、かなり便利です：
- `HIT`：キャッシュヒット
- `MISS`：キャッシュミス、オリジンサーバーから取得
- `EXPIRED`：キャッシュ有効期限切れ、オリジンサーバーから再取得
- `BYPASS`：キャッシュがバイパスされました

確認方法：
```bash
curl -I https://server.dl.hagicode.com/app.zip
```

## コスト分析

月間ダウンロードトラフィック 1TB を想定して、計算してみましょう：

| 方案 | トラフィック費用 | サーバー費用 | 合計 |
|------|----------|------------|------|
| 直接 Azure OSS | 約 ¥500 | ¥0 | ¥500 |
| クラウドサーバー + OSS (キャッシュヒット率 80%) | ¥100 + ¥80 | ¥60 | ¥240 |
| 商用 CDN | ¥300-500 | ¥0 | ¥300-500 |

**結論**：キャッシュ層で約 50% の配信コストを節約できます。

この計算はキャッシュヒット率 80% を想定しています。実際の状況では、ファイル更新頻度が高くない場合、ヒット率はさらに高くなる可能性があります。結局、これも自然なことです。

## デプロイ実践

### 環境準備

まず環境変数を設定します：

```bash
cd /path/to/hagicode_aliyun_deployment/docker
cp .env.example .env
vi .env  # Azure OSS SAS URL、Lark Webhook URL を入力
```

**重要**：`.env` ファイルには機密情報（SAS Token、Webhook URL）が含まれているため、バージョン管理に絶対にコミットしないでください。これは結局重要なことです。

### DNS 設定

以下の DNS A レコードを追加します。このステップを忘れないでください：
- `server.dl.hagicode.com` → サーバー IP
- `desktop.dl.hagicode.com` → サーバー IP

### サーバーの初期化

Ansible 自動化でサーバーを初期化します：

```bash
cd /path/to/hagicode_aliyun_deployment
ansible-playbook -i ./ansible/inventory/hosts.yml ./ansible/playbooks/init.yml
```

この Playbook は自動的にこれらを処理します：
- デプロイユーザーの作成
- Docker と Docker Compose のインストール
- SSH 鍵の設定
- ファイアウォールルールの設定

それほど複雑ではありません。結局、自動化は時間と労力を節約します。

### サービスのデプロイ

```bash
./deploy.sh
```

デプロイスクリプトはこれらを支援します：
- 環境設定のチェック
- 最新コードの取得
- Docker コンテナの起動
- ヘルスチェックの実行
- デプロイ通知の送信（Lark）

一つのコマンドで完了、便利です。

### デプロイの検証

```bash
# コンテナステータスのチェック
docker ps

# ダウンロードドメインのテスト
curl -I https://server.dl.hagicode.com/index.json
curl -I https://desktop.dl.hagicode.com/index.json
```

## 運用のヒント

### キャッシュ管理

キャッシュは、時々手入れする必要があります：

**キャッシュディスク使用量の確認**：
```bash
docker volume inspect docker_nginx-cache
du -sh /var/lib/docker/volumes/docker_nginx-cache/_data
```

**手動キャッシュクリア**：
```bash
./clear-cache.sh
```

または手動で実行。面倒ですが、効果的です：
```bash
docker exec nginx sh -c "rm -rf /var/cache/nginx/*"
docker restart nginx
```

### リソース制限

1 コア 2GB サーバーでは、リソース制限の設定は以下の通りです：

```yaml
services:
  traefik:
    deploy:
      resources:
        limits:
          cpus: '1.50'
          memory: 512M

  nginx:
    deploy:
      resources:
        limits:
          cpus: '0.50'
          memory: 256M
```

リソース使用状況を監視することも時々良いことです：
```bash
docker stats
```

### SAS Token セキュリティプラクティス

SAS Token は Azure Blob Storage へのアクセス証明書であり、漏洩すると笑い事ではありません：

- `.env` ファイルをバージョン管理にコミットしない（.gitignore に追加済み）
- SAS Token に適切な有効期限を設定（推奨 1 年）
- SAS Token 権限を制限（読み取りのみ）
- SAS Token を定期的にローテーション

### 監視アラート

HagiCode は Lark/飞书 Webhook 通知を統合しており、以下の状況で通知を送信できます：
- デプロイ成功/失敗
- キャッシュクリアステータス
- サービス異常

通知にはサーバー情報、タイムスタンプ、エラー詳細が含まれており、問題の迅速な特定もいくらか便利です。

### 高可用性拡張

単一サーバーで支えきれないときは、以下も検討できます：

1. **水平スケーリング**：複数ノードをデプロイし、DNS ラウンドロビンまたはロードバランサーで配信
2. **CDN 強化**：クラウドサーバーの前に CDN を接続し、アクセス速度をさらに向上
3. **キャッシュプリロード**：スクリプトを使って人気ファイルを事前にキャッシュにロード

## 注意事項

いくつか注意点を提醒します。結局、誰も問題に遭いたくありません：

1. **SSL 証明書**：Let's Encrypt にはレート制限があるため、頻繁にデプロイを切り替えないでください。そうしないと証明書を取得できない可能性があります
2. **キャッシュクリア**：重要なファイルを更新した後はキャッシュをクリアすることを忘れないでください。そうしないとユーザーが新しいバージョンをダウンロードできない可能性があります
3. **ログ管理**：定期的に Docker ログをクリアしてください。そうしないとディスクが満タンになり、問題になります
4. **バックアップ戦略**：Traefik acme.json、Bunker Web 設定などは、バックアップしておいた方が良いでしょう
5. **監視アラート**：Lark 通知を設定し、デプロイステータスをタイムリーに把握し、問題が発生しても迅速に反応できるようにします

## まとめ

クラウドサーバー + Nginx キャッシュ層、これだけで簡単です。HagiCode はこのソリューションを使用しており、コストはそれほど高くありません（サーバー費用は約 60-100 元/月）、効果はかなり良いです。核心的な利点は大概これらです：

- **コスト制御可能**：クラウドストレージや商用 CDN を直接使用するよりも、コストは約 50% 削減
- **柔軟なデプロイ**：Traefik か Bunker Web か、自分で選択
- **高い拡張性**：必要に応じて水平スケーリング可能、または CDN を追加
- **運用が簡単**：Shell スクリプト + Ansible、自動デプロイも便利

ファイル配信が必要な小規模チームや個人開発者にとって、このソリューションは試してみる価値があります。

HagiCode はこのアーキテクチャを使用して本番環境で一定期間安定して運用しており、世界中のユーザーのダウンロードでも大きな問題は発生していません。同様のソリューションを探しているなら、試してみる価値があるかもしれません。あなたにも役立つかもしれません。
