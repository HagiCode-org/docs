## 120秒から45秒へ：Worker Threads で Vite ビルドパフォーマンスを最適化する実践

> 大規模なフロントエンドプロジェクトでは、本番ビルドに時間がかかりすぎることがよくあります。本記事では、Node.js Worker Threads を活用して Vite ビルドのコード難読化の所要時間を 120 秒から 45 秒に短縮し、HagiCode プロジェクトでの実装の詳細と苦労話を詳しく紹介します。

## 背景

フロントエンドエンジニアリングの実践において、プロジェクト規模の拡大に伴い、ビルド効率の問題が徐々に顕在化してきました。特に本番ビルドプロセスでは、ソースコードのロジックを保護するために JavaScript 難読化ツール（`javascript-obfuscator` など）を導入することが一般的です。このステップは必要ですが、計算量が非常に大きく、CPU リソースを大量に消費します。

**HagiCode** プロジェクトの初期開発段階で、非常に厄介なパフォーマンスのボトルネックに直面しました：本番ビルド時間がコード量の増加に伴い急速に悪化したのです。

**具体的なペインポイントは以下の通りです：**
- シングルスレッドで直列に難読化タスクを実行し、CPU の1コアだけがフル稼働で他のコアがアイドル状態
- ビルド時間が当初の30秒から110〜120秒に急増
- コード変更後のビルド検証プロセスが極めて長く、開発イテレーション効率を著しく低下
- CI/CD パイプラインで、ビルドが最も時間のかかる部分に

**なぜ HagiCode にこのニーズがあるのか？**
HagiCode は AI 駆動のコードインテリジェントアシスタントで、そのフロントエンドアーキテクチャには複雑なビジネスロジックと AI インタラクションモジュールが含まれています。コアコードの安全性を確保するため、本番リリース時に高強度の難読化を強制的に有効にしています。2分に近いビルド待ち時間に直面し、ビルドシステムの深いパフォーマンス最適化に取り組むことを決定しました。

## 分析：パフォーマンスボトルネックの突破口を見つける

パフォーマンス問題の解決に取り組む前に、まず考えを整理し、最適な技術ソリューションを特定する必要があります。

### コア決定：なぜ Worker Threads を選んだのか？

Node.js 環境で並列計算を実現するには主に3つの方法があります：
1. **child_process**：独立した子プロセスを作成
2. **Web Workers**：主にブラウザ側で使用
3. **worker_threads**：Node.js のネイティブマルチスレッドサポート

比較分析の結果、HagiCode は **Worker Threads** を選択しました。理由は以下の通りです：
- **ゼロシリアライゼーションオーバーヘッド**：Worker Threads は同じプロセス内にあり、`SharedArrayBuffer` または制御権の転送を通じてメモリを共有でき、プロセス間通信の大きなシリアライゼーションコストを回避できます。
- **ネイティブサポート**：Node.js 12+ に組み込まれており、追加の重い依存関係は不要です。
- **コンテキストの統一**：サブプロセスよりもデバッグとロギングが便利です。

### タスクの粒度：難読化タスクをどう分割するか？

巨大な JS Bundle ファイル1つの難読化は並列化が難しい（コードに依存関係があるため）ですが、Vite のビルド成果物は複数の **Chunk** で構成されています。これが私たちに天然の並列境界を与えてくれます：

- **独立性**：Vite パッケージング後の異なる Chunk 間の依存関係は既に分離されており、安全に並列処理できます。
- **適切な粒度**：通常のプロジェクトには 10〜30 の Chunk があり、この規模は並列スケジューリングに非常に適しています。
- **統合の容易さ**：Vite プラグインの `generateBundle` フックにより、ファイル生成前にこれらの Chunk をインターセプトして処理できます。

### アーキテクチャ設計

4つのコアコンポーネントを含む並列処理システムを設計しました：

1. **Task Splitter**：Vite の bundle オブジェクトを反復処理し、難読化が不要なファイル（vendor など）をフィルタリングして、タスクキューを生成します。
2. **Worker Pool Manager**：Worker のライフサイクルを管理し、タスクの配布、回収、エラー時のリトライを担当します。
3. **Progress Reporter**：ビルド進捗をリアルタイムに出力し、ユーザーの待ち不安を軽減します。
4. **ObfuscationWorker**：実際に難読化ロジックを実行するワーカースレッド。

## 解決策：実践的なコーディングと実装

上記の分析に基づき、この並列難読化システムの実装を開始しました。

### 1. Vite プラグインの設定

まず、`vite.config.ts` で並列難読化プラグインを統合します。設定は非常に直感的で、Worker 数と難読化ルールを指定するだけです。

```typescript
import { defineConfig } from 'vite'
import { parallelJavascriptObfuscator } from './buildTools/plugin'

export default defineConfig(({ mode }) => {
  const isProduction = mode === 'production'

  return {
    build: {
      rollupOptions: {
        ...(isProduction
          ? {
              plugins: [
                parallelJavascriptObfuscator({
                  enabled: true,
                  // CPU コア数に基づいて自動調整。メインスレッド用に1コア残すことを推奨
                  workerCount: 4,
                  retryAttempts: 3,
                  fallbackToMainThread: true, // エラー時に自動的にシングルスレッドにフォールバック
                  // vendor chunk を除外。通常、サードパーティライブラリは難読化不要
                  isVendorChunk: (fileName: string) => fileName.includes('vendor-'),
                  obfuscationConfig: {
                    compact: true,
                    controlFlowFlattening: true,
                    deadCodeInjection: true,
                    disableConsoleOutput: true,
                    // ... その他の難読化オプション
                  },
                }),
              ],
            }
          : {}),
      },
    },
  }
})
```

### 2. Worker ロジックの実装

Worker はタスクを実行する単位です。入力と出力のデータ構造を明確に定義する必要があります。

**注意**：ここのコードはシンプルですが、いくつかの落とし穴に注意が必要です。例えば `parentPort` の null チェックやエラー処理です。HagiCode の実践では、特定の ES6 構文が難読化器のクラッシュを引き起こすことが判明したため、`try-catch` 保護を追加しました。

```typescript
import { parentPort } from 'worker_threads'
import javascriptObfuscator from 'javascript-obfuscator'

export interface ObfuscationTask {
  chunkId: string
  code: string
  config: any
}

export interface ObfuscationResult {
  chunkId: string
  obfuscatedCode: string
  error?: string
}

// メインスレッドからのタスクをリッスン
if (parentPort) {
  parentPort.on('message', async (task: ObfuscationTask) => {
    try {
      // 難読化を実行
      const obfuscated = javascriptObfuscator.obfuscate(task.code, task.config)
      const result: ObfuscationResult = {
        chunkId: task.chunkId,
        obfuscatedCode: obfuscated.getObfuscatedCode(),
      }
      // 結果をメインスレッドに送信
      parentPort?.postMessage(result)
    } catch (error) {
      // 例外を処理。単一の Worker のクラッシュがビルド全体をブロックしないようにする
      const result: ObfuscationResult = {
        chunkId: task.chunkId,
        obfuscatedCode: '',
        error: error instanceof Error ? error.message : 'Unknown error',
      }
      parentPort?.postMessage(result)
    }
  })
}
```

### 3. Worker プールマネージャー

これがソリューション全体のコアです。固定サイズの Worker プールを維持し、**FIFO（先入れ先出し）**戦略でタスクをスケジューリングする必要があります。

```typescript
import { Worker } from 'worker_threads'
import os from 'os'

export class WorkerPool {
  private workers: Worker[] = []
  private taskQueue: Array<{
    task: ObfuscationTask
    resolve: (result: ObfuscationResult) => void
    reject: (error: Error) => void
  }> = []

  constructor(options: WorkerPoolOptions = {}) {
    // デフォルトはコア数 - 1。メインスレッドに少し余裕を残す
    const workerCount = options.workerCount ?? Math.max(1, (os.cpus().length || 4) - 1)

    for (let i = 0; i < workerCount; i++) {
      this.createWorker()
    }
  }

  private createWorker() {
    const worker = new Worker('./worker.ts')

    worker.on('message', (result) => {
      // タスク完了後、キューから次のタスクを取り出す
      const nextTask = this.taskQueue.shift()
      if (nextTask) {
        this.dispatchTask(worker, nextTask)
      } else {
        // 待機中のタスクがない場合、Worker をアイドル状態にマーク
        this.activeWorkers.delete(worker)
      }
    })

    this.workers.push(worker)
  }

  // タスクをプールに送信
  public runTask(task: ObfuscationTask): Promise<ObfuscationResult> {
    return new Promise((resolve, reject) => {
      const job = { task, resolve, reject }
      const idleWorker = this.workers.find(w => !this.activeWorkers.has(w))

      if (idleWorker) {
        this.dispatchTask(idleWorker, job)
      } else {
        this.taskQueue.push(job)
      }
    })
  }

  private dispatchTask(worker: Worker, job: any) {
    this.activeWorkers.set(worker, job.task)
    worker.postMessage(job.task)
  }
}
```

### 4. 進捗報告

待つのは苦痛です。特にあとどれくらい待つか分からない時は。そこで、簡単な進捗レポーターを追加し、現在のステータスをリアルタイムでフィードバックするようにしました。

```typescript
export class ProgressReporter {
  private completed = 0
  private readonly total: number
  private readonly startTime: number

  constructor(total: number) {
    this.total = total
    this.startTime = Date.now()
  }

  increment(): void {
    this.completed++
    this.report()
  }

  private report(): void {
    const now = Date.now()
    const elapsed = now - this.startTime
    const percentage = (this.completed / this.total) * 100

    // シンプルな ETA 推定
    const avgTimePerChunk = elapsed / this.completed
    const remaining = (this.total - this.completed) * avgTimePerChunk

    console.log(
      `[Parallel Obfuscation] ${this.completed}/${this.total} chunks completed (${percentage.toFixed(1)}%) | ETA: ${(remaining / 1000).toFixed(1)}s`
    )
  }
}
```

## 実践：効果と苦労話

このソリューションを導入した後、HagiCode プロジェクトのビルドパフォーマンスは即座に向上しました。

### パフォーマンスベンチマークデータ

以下の環境でテストを実施しました：
- CPU：Intel Core i7-12700K (12 cores / 20 threads)
- RAM：32GB DDR4
- Node.js：v18.17.0
- OS：Ubuntu 22.04

**結果比較：**
- **シングルスレッド（最適化前）**：118秒
- **4 Workers**：55秒（**53%** 向上）
- **8 Workers**：48秒（**60%** 向上）
- **12 Workers**：45秒（**62%** 向上）

ご覧の通り、改善は線形ではありません。Worker 数が8を超えると、向上幅が小さくなります。これは主にタスク配布の均等性とメモリ帯域幅のボトルネックによるものです。

### よくある問題と解決策

HagiCode の実際の使用でいくつかの落とし穴に遭遇しました。ここで共有します：

**Q1: ビルド時間が明確に減らず、むしろ遅くなった？**
- **原因**：Worker 作成自体にオーバーヘッドがあるか、Worker 数が多すぎてコンテキストスイッチが頻発している可能性があります。
- **解決策**：Worker 数を `CPU コア数 - 1` に設定することを推奨します。また、特定の Chunk が特に大きい（> 5MB など）場合は、その「巨大な」ファイルがボトルネックになるので、コード分割戦略の最適化を検討してください。

**Q2: Worker が偶発的にクラッシュし、ビルドが失敗する？**
- **原因**：特定のコード構文が難読化器の内部エラーを引き起こす可能性があります。
- **解決策**：**自動フォールバックメカニズム**を実装しました。Worker の連続失敗回数が閾値に達すると、プラグインは自動的にシングルスレッドモードにフォールバックし、ビルドが中断しないようにします。同時にエラーのファイル名を記録し、後で针对性的に修正できるようにします。

**Q3: メモリ使用量が高すぎる（OOM）？**
- **原因**：各 Worker が難読化器と AST 解析をロードするために独立したメモリ空間を必要とします。
- **解決策**：
  - Worker 数を減らす。
  - Node.js のメモリ制限を増やす：`NODE_OPTIONS="--max-old-space-size=4096" npm run build`。
  - Worker 内部で不要な大きなオブジェクト参照を保持しないようにする。

## まとめ

Node.js Worker Threads を導入することで、HagiCode プロジェクトの本番ビルド時間を 120 秒から約 45 秒に短縮し、開発体験と CI/CD 効率を大幅に向上させました。

このソリューションのコアは以下の3点です：
1. **適切なタスク分割**：Vite の Chunk を並列ユニットとして活用。
2. **リソース制御**：Worker プールを使用してリソースの枯渇を回避。
3. **フォールトトレランス設計**：自動フォールバックメカニズムでビルドの安定性を確保。
