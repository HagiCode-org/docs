## Reactプロジェクトでエレガントに新規ユーザーガイドを実装：HagiCodeのdriver.js実践

> ユーザーが初めてあなたの製品を開いたとき、本当にどこから始めればよいか分かっているでしょうか？この記事では、HagiCodeプロジェクトでdriver.jsを使って新規ユーザーガイドを実装した経験についてお話しします。

## 背景

このようなシナリオに遭遇したことはありませんか？新規ユーザーがあなたの製品に登録し、ページを開いたとき、何をクリックすればよいのか、何をすべきのか分からず、右往左往している様子。開発者として、私たちは「ユーザーは自分で探検するだろう」と思いがちです。結局、人の好奇心は無限ですから。しかし現実は——大部分のユーザーは入口が見つからず、数分以内に静かに去っていきます。物語が突然始まり、自然に終わるように。

新規ユーザーガイドはこの問題を解決する重要な手段ですが、実装もそれほど簡単ではありません。良いガイドシステムには以下が必要です：

- ページ上の要素を正確に特定しハイライト表示できる
- 複数ステップのガイドフローをサポートする
- ユーザーの選択（完了/スキップ）を記憶できる
- ページのパフォーマンスと正常なインタラクションに影響を与えない
- コード構造が明確で、保守が容易

HagiCodeの開発過程でも、私たちは同じ課題に直面しました。HagiCodeはAIコードアシスタントプロジェクトであり、核心的なワークフローは「ユーザーが提案を作成 → AIが計画を生成 → ユーザーがレビュー → AIが実行」というOpenSpecプロセスです。この概念に初めて触れるユーザーにとって、このプロセスは全く新しいもので、素早く使いこなせるよう良いガイドが必要です。新しいことを習得するには、やはり少し時間が必要なのです。

## HagiCodeについて

この記事で共有するソリューションは、[HagiCode](https://hagicode.com)プロジェクトでの実践経験から来ています。HagiCodeはClaudeベースのAIコードアシスタントであり、OpenSpecワークフローを通じて開発者がより効率的にコードタスクを完了できるようにします。[GitHub](https://github.com/HagiCode-org/site)で私たちのオープンソースコードを確認できます。

## なぜdriver.jsを選択したのか

技術選定の段階で、いくつかの主流のガイドライブラリを評価しました。それぞれに特徴があります：

- **Intro.js**：機能は強力ですがサイズが大きく、スタイルのカスタマイズが比較的複雑
- **Shepherd.js**：API設計は優れていますが、私たちのシナリオにとっては少し「重い」
- **driver.js**：軽量、シンプル、APIが直感的で、Reactエコシステムをサポート

最終的に私たちはdriver.jsを選択しました。特別な理由があるわけではなく、主に以下の点を考慮しました：

1. **軽量**：コアライブラリのサイズが小さく、バンドルサイズを大幅に増加させない
2. **APIがシンプル**：設定項目が明確で直感的で、素早く使いこなせる
3. **柔軟性**：カスタム配置、スタイル、インタラクション動作をサポート
4. **動的インポート**：必要に応じてロードでき、初回画面のパフォーマンスに影響を与えない

技術選定において、最善なものはなく、最適なものだけです。

## 技術的実装

### コア設定

driver.jsの設定は非常に直感的です。以下はHagiCodeプロジェクトのコア設定です：

```typescript
import { driver } from 'driver.js';
import 'driver.js/dist/driver.css';

const newConversationDriver = driver({
  allowClose: true,           // ユーザーがガイドを閉じることを許可
  animate: true,              // アニメーション効果を有効化
  overlayClickBehavior: 'close', // オーバーレイクリックでガイドを閉じる
  disableActiveInteraction: false, // 要素のインタラクションを維持
  showProgress: false,        // プログレスバーを表示しない（カスタム進行管理があるため）
  steps: guideSteps           // ガイドステップ配列
});
```

これらの設定の背後にある考え方は以下の通りです：

- `allowClose: true` - ユーザーの選択を尊重し、ガイドの完了を強制しない
- `disableActiveInteraction: false` - 特定のステップではユーザーが実際に操作する必要がある（テキスト入力など）ため、インタラクションを無効にできない
- `overlayClickBehavior: 'close'` - ユーザーに高速な退出方法を提供する

### 状態管理

ガイド状態の永続化は鍵です——ページを更新するたびにガイドを再表示されたくありません。HagiCodeはlocalStorageを使ってガイド状態を管理します：

```typescript
export type GuideState = 'pending' | 'dismissed' | 'completed';

export interface UserGuideState {
  session: GuideState;
  detailGuides: Record<string, GuideState>;
}

// 状態を読み取る
export const getUserGuideState = (): UserGuideState => {
  const state = localStorage.getItem('userGuideState');
  return state ? JSON.parse(state) : { session: 'pending', detailGuides: {} };
};

// 状態を更新する
export const setUserGuideState = (state: UserGuideState) => {
  localStorage.setItem('userGuideState', JSON.stringify(state));
};
```

3つの状態を定義しました：

- `pending`：ガイド進行中、ユーザーがまだ完了またはスキップしていない
- `dismissed`：ユーザーが能動的にガイドを閉じた
- `completed`：ユーザーがすべてのステップを完了した

提案詳細ページのガイドについては、より粒度の細かい状態追跡もサポートしています（`detailGuides`ディクショナリを通じて）。なぜなら、一つの提案は複数の段階（ドラフト、レビュー、実行完了）を経る可能性があり、各段階で異なるガイドが必要だからです。物事の状態は常に変化するものです。

### ターゲット要素の配置

driver.jsはCSSセレクタを使ってターゲット要素を特定します。HagiCodeは`data-guide`カスタム属性を使ってガイドターゲットをマークするという規約を採用しています：

```typescript
const steps = [
  {
    element: '[data-guide="launch"]',
    popover: {
      title: '新しい会話を開始',
      description: 'ここをクリックして新しい会話セッションを作成します...'
    }
  }
];
```

コンポーネントでは次のように使用します：

```typescript
<button data-guide="launch" onClick={handleLaunch}>
  新しい会話
</button>
```

この方法の利点は：

- ビジネススタイルクラス名との競合を回避できる
- セマンティクスが明確で、この要素がガイドに関連していることが一目でわかる
- 統一された管理と保守が容易

### 動的インポートの最適化

ガイド機能は特定のシナリオでのみ必要（新規ユーザーの初回アクセスなど）なので、動的インポートを使って初期ロードパフォーマンスを最適化しています：

```typescript
const initNewUserGuide = async () => {
  // driver.jsを動的インポート
  const { driver } = await import('driver.js');
  await import('driver.js/dist/driver.css');

  // ガイドを初期化
  const newConversationDriver = driver({
    // ...設定
  });

  newConversationDriver.drive();
};
```

これにより、driver.jsとそのスタイルファイルは必要な時だけロードされ、初回画面のパフォーマンスに影響を与えません。結局、一時的に使わないもののために待つコストを支払いたい人はいないでしょう。

## ガイドフローの設計

HagiCodeは2つのガイドパスを実装し、ユーザーのコア使用シナリオをカバーしています。

### 会話ガイド（10ステップ）

このガイドは、会話の作成から最初の完全な提案の提出までの全プロセスを完了するのをユーザーに支援します：

1. **launch** - ガイドを開始し、「新しい会話」ボタンを紹介
2. **compose** - 入力ボックスにリクエストを入力するようガイド
3. **send** - 送信ボタンのクリックをガイド
4. **proposal-launch-readme** - README提案の作成をガイド
5. **proposal-compose-readme** - READMEリクエスト内容の編集をガイド
6. **proposal-submit-readme** - README提案の提出をガイド
7. **proposal-launch-agents** - AGENTS.md提案の作成をガイド
8. **proposal-compose-agents** - AGENTS.mdリクエストの編集をガイド
9. **proposal-submit-agents** - AGENTS.md提案の提出をガイド
10. **proposal-wait** - AIが処理中であることを説明し、しばらくお待ちください

このガイドの設計思想は：2つの実際の提案作成タスク（READMEとAGENTS.md）を通じて、ユーザーにHagiCodeのコアワークフローを実際に体験してもらうことです。結局、紙の上で得る知識は浅く、これを知るには自分で実践する必要があります。

### 提案詳細ガイド（3ステップ）

ユーザーが提案詳細ページに入るとき、提案の現在の状態に基づいて対応するガイドをトリガーします：

1. **drafting**（ドラフト段階）- ユーザーがAI生成計画を確認するようガイド
2. **reviewing**（レビュー段階）- ユーザーが計画を実行するようガイド
3. **executionCompleted**（完了段階）- ユーザーが計画をアーカイブするようガイド

このガイドの特徴は状態駆動であること——提案の実際の状態に基づいて動的にどのガイドステップを表示するかを決定します。物事は常に変化するものであり、ガイドもそれに応じて変化すべきです。

この段階では、ガイドがユーザーの注意を「計画を生成」という重要なアクションに集中させ、初めて詳細ページに入ったときに何をすべきかわからないことを防ぎます。

### 要素レンダリング再試行メカニズム

Reactアプリケーションでは、ガイドターゲット要素がまだレンダリングされていない可能性があります（非同期データロードの待機中など）。この状況を処理するために、HagiCodeは再試行メカニズムを実装しています：

```typescript
const waitForElement = (selector: string, maxRetries = 10, interval = 100) => {
  let retries = 0;
  return new Promise<HTMLElement>((resolve, reject) => {
    const checkElement = () => {
      const element = document.querySelector(selector) as HTMLElement;
      if (element) {
        resolve(element);
      } else if (retries < maxRetries) {
        retries++;
        setTimeout(checkElement, interval);
      } else {
        reject(new Error(`Element not found: ${selector}`));
      }
    };
    checkElement();
  });
};
```

ガイドを初期化する前にこの関数を呼び出し、ターゲット要素が存在することを確実にします。時には、少し待つ価値もあります。

## ベストプラクティスのまとめ

HagiCodeの実践経験に基づき、ここでいくつかの重要なベストプラクティスを共有します：

### 1. ガイドは「脱出可能」であるべき

ユーザーにガイドの完了を強制しないでください。一部のユーザーは探索型で、自分で試すことを好みます。明確な「スキップ」ボタンを提供し、ユーザーの選択を記憶して、次回は邪魔をしないようにします。美しい事物や人は、必ずしも所有する必要はなく、彼女が美しいままでいれば、自分で彼女の美しさを眺めていればいいのです。

### 2. ガイドの内容は簡潔で力強いべき

各ガイドステップは単一の目標に集中すべきです：

- **Title**：簡潔で明確、10文字以内
- **Description**：要点を突き、「これが何か」と「なぜ使うのか」をユーザーに伝える

長々とした説明を避けてください——ガイド段階のユーザーの注意力は限られています。話が多すぎると、かえって見たくなくなります。

### 3. セレクタは安定しているべき

頻繁に変化しない安定した要素マーク付け方法を使用してください。`data-guide`カスタム属性は良い選択です。クラス名やDOM構造に依存しないでください。これらはリファクタリングで容易に変化する可能性があります。コードは常に変化しますが、一部のものは可能な限り安定させるべきです。

### 4. ガイドをテストする

HagiCodeはガイド機能のために完全なテストケースを作成しました：

```typescript
describe('NewUserConversationGuide', () => {
  it('ガイド状態を正しく初期化するべき', () => {
    const state = getUserGuideState();
    expect(state.session).toBe('pending');
  });

  it('ガイド状態を正しく更新するべき', () => {
    setUserGuideState({ session: 'completed', detailGuides: {} });
    const state = getUserGuideState();
    expect(state.session).toBe('completed');
  });
});
```

テストにより、コードをリファクタリングするときにうっかりガイド機能を壊さないことを保証できます。結局、誰もコードを少し変更しただけで以前の機能を壊したくないでしょう。

### 5. パフォーマンスの最適化

- 動的インポートを使ってガイドライブラリの遅延ロードを行う
- ユーザーがすでにガイドを完了した後もガイドロジックを初期化しない
- ガイドアニメーションのパフォーマンス影響を考慮し、ローエンドデバイスではアニメーションをオフにする

パフォーマンスというものは、生活と同じで、節約すべきところは節約すべきです。

## まとめ

新規ユーザーガイドは製品のユーザーエクスペリエンスを向上させる重要な要素です。HagiCodeプロジェクトでは、driver.jsを使って完全なガイドシステムを構築し、会話作成から提案実行までの全ワークフローをカバーしました。

この記事の共有を通じて、私たちが伝えたい核心的な観点は：

1. **技術選定はニーズにマッチすべき**：driver.jsは最強ではありませんが、私たちにとって最適です
2. **状態管理が重要**：localStorageを使ってガイド状態を永続化し、繰り返しユーザーを邪魔しない
3. **ガイド設計は集中すべき**：各ステップで一つの問題を解決し、欲張らない
4. **コード構造は明確すべき**：ガイド設定、状態管理、UIロジックを分離し、保守を容易にする

自分のプロジェクトに新規ユーザーガイド機能を追加している場合、この記事の実践経験が役立つことを願っています。技術というものは、実は何の神秘的なものでもなく、試してみて、まとめて、徐々に良くなっていくものです......
