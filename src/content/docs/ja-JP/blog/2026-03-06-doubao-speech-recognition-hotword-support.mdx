## 豆包音声認識のホットワード機能実装ガイド

> 本記事では、HagiCodeプロジェクトで豆包音声認識のホットワードサポート機能を実装する方法を詳しく解説します。カスタムホットワードとプラットフォームホットワードテーブルの2つの方法により、特定分野の語彙の認識精度を大幅に向上させます。

## 背景

音声認識技術は長年発展してきましたが、実は開発者たちをずっと悩ませてきた問題があります。汎用音声認識モデルは日常的な言葉をカバーできますが、専門用語、製品名、人名などの単語に対しては、認識精度がいまひとつです。考えてみてください、医療分野の音声アシスタントが「高血圧」「糖尿病」「冠動脈疾患」を正確に認識する必要がある、法律システムが「案由」「答弁」「挙証責任」を的確に捉える必要がある——これらのシナリオでは、汎用モデルの表现はどうでしょうか、まあ頑張っていますよね。

HagiCodeプロジェクトでも、同じ課題に直面しました。多機能なAIコードアシスタントとして、HagiCodeは様々な技術用語の音声認識シナリオを処理する必要があります。しかし、豆包音声認識APIはデフォルトでは、専門用語の精度に関する私たちの要件を完全には満たしていませんでした。実のところ、豆包が不十分なわけではなく、各分野には独自の用語体系があるだけです。調査と技術的探求を経て、豆包音声認識APIが実際にはホットワードサポート機能を提供していることがわかりました。簡単な設定で、特定語彙の認識精度を大幅に向上させることができます。これはまるで、「注意すべき言葉」を教えると、それらの言葉をもっと注意深く聞くようになるようなものです。

本記事で共有するのは、HagiCodeプロジェクトで豆包音声認識のホットワード機能を実装する完全なソリューションです。カスタムホットワードとプラットフォームホットワードテーブルの2つのモードがあり、どちらも使用可能で、組み合わせて使用することもできます。このソリューションを通じて、開発者はビジネスシナリオに応じてホットワードを柔軟に設定し、音声認識システムに専門的で稀ですが極めて重要な語彙を「認識」させることができます。

## コア実装

### 2つのホットワードモードの解説

豆包音声認識APIは、2種類のホットワード設定方法を提供しており、それぞれに独自の適用シナリオと利点があります。

**カスタムホットワードモード**では、`corpus.context`フィールドを通じて直接ホットワードテキストを渡すことができます。この方法は、少量のホットワードを迅速に設定する必要があるシナリオに非常に適しています。例えば、一時的に特定の製品名や人名を認識する必要がある場合などです。HagiCodeの実装では、ユーザーが入力した複数行のホットワードテキストを文字列リストとして解析し、豆包APIの要件に従って`context_data`配列としてフォーマットします。どうでしょうか、この方法は非常に直接的で、相手に「これらの言葉に注意してください」と伝えると、それらの言葉に注意するようになるようなものです。

**プラットフォームホットワードテーブルモード**では、`corpus.boosting_table_id`フィールドを通じて豆包自己学習プラットフォームで事前設定されたホットワードテーブルを参照します。この方法は大量のホットワードを管理する必要があるシナリオに適しており、豆包自己学習プラットフォームでホットワードテーブルを作成・維持し、IDを通じて参照できます。HagiCodeのように専門用語を継続的に更新・維持する必要があるプロジェクトでは、このモードがより優れた管理性を提供します。何しろ、ホットワードが多くなったら、統一管理できる場所がある方が、毎回手動で入力するより良いです。

面白いことに、この2つのモードを組み合わせて使用することもできます。豆包APIは同じリクエストでカスタムホットワードとプラットフォームホットワードテーブルIDの両方を含むことをサポートし、`combine_mode`パラメータで組み合わせ戦略を制御します。この柔軟性により、HagiCodeは様々な複雑な専門用語認識ニーズに対応できます。これもなかなか良いもので、複数の方法を組み合わせると、効果がより良くなることもあります。

### フロントエンドの型定義と検証

HagiCodeのフロントエンド実装では、完全なホットワード設定型と検証ロジックを定義しました。まず型定義の部分から：

```typescript
export interface HotwordConfig {
  contextText: string;           // 複数行ホットワードテキスト
  boostingTableId: string;      // 豆包プラットフォームホットワードテーブルID
  combineMode: boolean;          // 組み合わせて使用するかどうか
}
```

このシンプルなインターフェースには、ホットワード機能のすべての設定項目が含まれています。その中で`contextText`はユーザーが最も直感的に感じる部分です——ユーザーが1行に1つのホットワードフレーズを入力できるようにし、この方法は非常に直感的です。何しろ、ユーザーに1行に1つの言葉を入力させる方が、複雑な設定ルールを理解させるより良いです。

次に検証関数の実装です。豆包APIの制限を考慮して、厳密な検証ルールを策定しました：ホットワードテキストは最大100行、1行最大50文字、合計最大5000文字；boosting_table_idは最大200文字、英数字、アンダースコア、ハイフンのみ許可されます。これらの制限は私たちが勝手に想像したものではなく、豆包公式ドキュメントの実際の要件に基づいています。何しろ、APIの制限はAPIの制限であり、従うしかありません。

```typescript
export function validateContextText(contextText: string): HotwordValidationResult {
  if (!contextText || contextText.trim().length === 0) {
    return { isValid: true, errors: [] };
  }

  const lines = contextText.split('\n').filter(line => line.trim().length > 0);
  const errors: string[] = [];

  if (lines.length > 100) {
    errors.push(`ホットワード行数は100行を超えてはいけません。現在は${lines.length}行です`);
  }

  const totalChars = contextText.length;
  if (totalChars > 5000) {
    errors.push(`ホットワード総文字数は5000を超えてはいけません。現在は${totalChars}です`);
  }

  for (let i = 0; i < lines.length; i++) {
    if (lines[i].length > 50) {
      errors.push(`第${i + 1}行ホットワードが50文字制限を超えています`);
    }
  }

  return { isValid: errors.length === 0, errors };
}

export function validateBoostingTableId(boostingTableId: string): HotwordValidationResult {
  if (!boostingTableId || boostingTableId.trim().length === 0) {
    return { isValid: true, errors: [] };
  }

  const errors: string[] = [];

  if (boostingTableId.length > 200) {
    errors.push(`boosting_table_idは200文字を超えてはいけません。現在は${boostingTableId.length}文字です`);
  }

  if (!/^[a-zA-Z0-9_-]+$/.test(boostingTableId)) {
    errors.push('boosting_table_idは英数字、アンダースコア、ハイフンのみ含めることができます');
  }

  return { isValid: errors.length === 0, errors };
}
```

これらの検証関数は、ユーザーがホットワードを設定するときにすぐに実行され、問題が最も早い段階で発見されるようにします。ユーザーエクスペリエンスにとって、この即時フィードバックは非常に重要です。何しろ、ユーザーが入力するときにどこが間違っているかを知る方が、送信後に発見するより良いです。

### フロントエンド設定の永続化

HagiCodeのフロントエンド実装では、ブラウザのlocalStorageを使用してホットワード設定を保存することを選択しました。この設計決定の背後にはいくつかの考慮事項があります：まず、ホットワード設定は非常に個人的な設定であり、異なるユーザーは異なる専門分野のニーズを持つ可能性があります。次に、この方法はバックエンドの実装を簡素化し、追加のデータベーステーブルとAPIインターフェースを必要としません。最後に、ユーザーはブラウザで一度設定すれば、その後の使用で自動的にロードされ、非常に便利です。要するに、面倒を省くということです。

```typescript
const HOTWORD_STORAGE_KEYS = {
  contextText: 'hotword-context-text',
  boostingTableId: 'hotword-boosting-table-id',
  combineMode: 'hotword-combine-mode',
} as const;

export const DEFAULT_HOTWORD_CONFIG: HotwordConfig = {
  contextText: '',
  boostingTableId: '',
  combineMode: false,
};

// ホットワード設定をロード
export function loadHotwordConfig(): HotwordConfig {
  const contextText = localStorage.getItem(HOTWORD_STORAGE_KEYS.contextText) || '';
  const boostingTableId = localStorage.getItem(HOTWORD_STORAGE_KEYS.boostingTableId) || '';
  const combineMode = localStorage.getItem(HOTWORD_STORAGE_KEYS.combineMode) === 'true';

  return { contextText, boostingTableId, combineMode };
}

// ホットワード設定を保存
export function saveHotwordConfig(config: HotwordConfig): void {
  localStorage.setItem(HOTWORD_STORAGE_KEYS.contextText, config.contextText);
  localStorage.setItem(HOTWORD_STORAGE_KEYS.boostingTableId, config.boostingTableId);
  localStorage.setItem(HOTWORD_STORAGE_KEYS.combineMode, String(config.combineMode));
}
```

このコードのロジックは非常にシンプルで明確です。設定をロードするときはlocalStorageから読み取り、保存するときはlocalStorageに書き込みます。また、デフォルト設定を提供し、設定がない場合でもシステムが正常に動作するようにしています。何しろ、デフォルト値が必要ですよね。

### バックエンドSDK設定拡張

HagiCodeのバックエンド実装では、SDK設定クラスにホットワード関連のプロパティを追加する必要があります。C#の言語特性と使用習慣を考慮して、カスタムホットワードコンテキストを保存するために`List<string>`を採用しました：

```csharp
public class DoubaoVoiceConfig
{
    /// <summary>
    /// アプリケーションID
    /// </summary>
    public string AppId { get; set; } = string.Empty;

    /// <summary>
    /// アクセストークン
    /// </summary>
    public string AccessToken { get; set; } = string.Empty;

    /// <summary>
    /// サービスURL
    /// </summary>
    public string ServiceUrl { get; set; } = string.Empty;

    /// <summary>
    /// カスタムホットワードコンテキストリスト
    /// </summary>
    public List<string>? HotwordContexts { get; set; }

    /// <summary>
    /// 豆包プラットフォームホットワードテーブルID
    /// </summary>
    public string? BoostingTableId { get; set; }
}
```

この設定クラスの設計は、HagiCodeの一貫したシンプルなスタイルに従っています。`HotwordContexts`はNULL可能なリスト型、`BoostingTableId`はNULL可能な文字列で、ホットワード設定がない場合、これらのプロパティはリクエストに影響を与えません。何しろ、使わないときは存在すべきではなく、それこそがきれいというものです。

### Payload構築ロジック

Payloadの構築はホットワード機能全体の中核です。ホットワード設定を取得した後、豆包APIの要件に従って正しいJSON構造にフォーマットする必要があります。このプロセスはSDKがリクエストを送信する前に発生します：

```csharp
private void AddCorpusToRequest(Dictionary<string, object> request)
{
    var corpus = new Dictionary<string, object>();

    // カスタムホットワードを追加
    if (Config.HotwordContexts != null && Config.HotwordContexts.Count > 0)
    {
        corpus["context"] = new Dictionary<string, object>
        {
            ["context_type"] = "dialog_ctx",
            ["context_data"] = Config.HotwordContexts
                .Select(text => new Dictionary<string, object> { ["text"] = text })
                .ToList()
        };
    }

    // プラットフォームホットワードテーブルIDを追加
    if (!string.IsNullOrEmpty(Config.BoostingTableId))
    {
        corpus["boosting_table_id"] = Config.BoostingTableId;
    }

    // corpusが空の場合のみリクエストに追加
    if (corpus.Count > 0)
    {
        request["corpus"] = corpus;
    }
}
```

このコードは、設定に基づいてcorpusフィールドを動的に構築する方法を示しています。重要な点は、実際にホットワード設定が存在する場合にのみ、corpusフィールドを追加することです。この設計は後方互換性を保証します——ホットワードを設定しない場合、リクエストの構造は以前と完全に一致します。何しろ、互換性は重要で、機能を追加するだけで以前のロジックを混乱させてはいけません。

### WebSocketパラメータ伝達

フロントエンドとバックエンドの間では、ホットワードパラメータはWebSocket制御メッセージを通じて伝達されます。HagiCodeの設計では、フロントエンドは録音を開始するときにlocalStorageからホットワード設定をロードし、WebSocketメッセージを通じてバックエンドに送信します。

```typescript
const controlMessage = {
  type: 'control',
  payload: {
    command: 'StartRecognition',
    contextText: '高血圧\n糖尿病\n冠動脈疾患',
    boosting_table_id: 'medical_table',
    combineMode: false
  }
};
```

ここで注意すべき詳細があります：フロントエンドが渡すのは複数行テキスト（改行文字で区切られ）で、バックエンドで解析する必要があります。バックエンドのWebSocketハンドラーはこれらのパラメータを解析し、SDKに渡します：

```csharp
private async Task HandleControlMessageAsync(
    string connectionId,
    DoubaoSession session,
    ControlMessage message)
{
    if (message.Payload is SessionControlRequest controlRequest)
    {
        // ホットワードパラメータを解析
        string? contextText = controlRequest.ContextText;
        string? boostingTableId = controlRequest.BoostingTableId;
        bool? combineMode = controlRequest.CombineMode;

        // 複数行テキストをホットワードリストに解析
        if (!string.IsNullOrEmpty(contextText))
        {
            var hotwords = contextText
                .Split('\n', StringSplitOptions.RemoveEmptyEntries)
                .Select(s => s.Trim())
                .Where(s => s.Length > 0)
                .ToList();

            session.HotwordContexts = hotwords;
        }

        session.BoostingTableId = boostingTableId;
    }
}
```

このような設計を通じて、フロントエンドからバックエンドへのホットワード設定の伝達が明確で効率的になります。何の特別なこともなく、ただ一層一層渡していくだけです。

## 実践ガイド

### カスタムホットワードの設定

実際の使用では、カスタムホットワードの設定は非常にシンプルです。HagiCodeの音声認識設定ページを開き、「ホットワード設定」エリアを見つけます。「カスタムホットワードテキスト」入力ボックスで、1行に1つのホットワードフレーズを入力します。

例えば、医療関連のアプリケーションを開発している場合、このように設定できます：

```
高血圧
糖尿病
冠動脈疾患
心绞痛
心肌梗死
心力衰竭
```

設定を保存した後、音声認識を開始するたびに、これらのホットワードは自動的に豆包APIに渡されます。実際のテストでは、ホットワードを設定した後、関連する専門用語の認識精度が明らかに向上しました。どうでしょうか、効果はやはりあり、少なくとも以前よりずっと良いです。

### プラットフォームホットワードテーブルの設定

大量のホットワードを管理する必要がある場合、またはホットワードを頻繁に更新する必要がある場合、プラットフォームホットワードテーブルモードの方が適しています。まず豆包自己学習プラットフォームでホットワードテーブルを作成し、生成されたboosting_table_idを取得してから、HagiCodeの設定ページでこのIDを入力する必要があります。

豆包自己学習プラットフォームはホットワードの一括インポート、分類管理などの機能を提供しており、大量の専門用語を管理する必要があるチームに非常に実用的です。プラットフォームを通じてホットワードを管理することで、ホットワードの集中管理と統一更新を実現できます。何しろ、ホットワードが多くなったら、統一管理できる場所がある方が、毎回手動で入力するより良いです。

### 組み合わせモードの使用

複雑なシナリオでは、カスタムホットワードとプラットフォームホットワードテーブルを同時に使用する必要があるかもしれません。この場合、HagiCodeで2種類のホットワードを同時に設定し、「組み合わせモード」スイッチを有効にするだけです。

組み合わせモードでは、豆包APIは両方のホットワードソースを同時に考慮し、認識精度は通常、どちらかを単独で使用するよりも高くなります。ただし、組み合わせモードはリクエストの複雑さを増すため、実際のテストを行ってから有効にするかどうかを決めることをお勧めします。何しろ、複雑さが増えたら、本当に価値があるかどうか、実際の効果を見る必要があります。

## 注意事項

ホットワード機能の実装と使用では、いくつかの点に特に注意する必要があります。

まずは文字制限です。豆包APIはホットワードに対して厳格な制限を持っており、行数、1行の文字数、総文字数などが含まれます。制限を超えると、APIはエラーを返します。HagiCodeのフロントエンド実装では、検証関数を通じてユーザー入力段階でチェックを行い、無効な設定がバックエンドに送られるのを防ぎます。何しろ、事前に問題を発見する方が、APIがエラーを返すのを待つより良いです。

次にboosting_table_idのフォーマットです。このフィールドは英数字、アンダースコア、ハイフンのみ許可され、スペースやその他の特殊文字は含めません。豆包自己学習プラットフォームでホットワードテーブルを作成するときは、命名規則に注意する必要があります。これも無理はありません、APIのフォーマット要件は常に比較的厳格です。

第三に後方互換性です。ホットワードパラメータは完全にオプションで、ホットワードを設定しない場合、システムの動作は以前と完全に一致します。この設計は、既存のユーザーに影響を与えず、段階的な移行とアップグレードを容易にします。何しろ、機能を追加するだけで以前のロジックを混乱させてはいけません。

最後はエラー処理です。ホットワード設定が無効な場合、豆包APIは対応するエラー情報を返します。HagiCodeの実装は詳細なログを記録し、開発者が問題を特定するのに役立ちます。同時に、フロントエンドもインターフェース上で検証エラーを表示し、ユーザーが設定を修正するのを助けます。エラー処理が適切に行われれば、ユーザーエクスペリエンスも自然と良くなります。

本記事の解説を通じて、HagiCodeプロジェクトで豆包音声認識のホットワード機能を実装する完全なソリューションを詳しく紹介しました。このソリューションは、ニーズ分析、技術選択からコード実装までのすべての段階をカバーし、開発者に参照可能な実践範例を提供します。

核心的な要点は次のように要約できます：第一に、豆包APIはカスタムホットワードとプラットフォームホットワードテーブルの2つのモードをサポートし、独立して使用することも組み合わせて使用することもできます；第二に、フロントエンドはlocalStorageを使用して設定を保存し、シンプルで効率的です；第三に、バックエンドはcorpusフィールドを動的に構築してホットワードパラメータを渡し、良好な後方互換性を維持します；第四に、完全な検証ロジックは設定の正確性を保証し、無効なリクエストを回避します。どうでしょうか、このソリューションは複雑ではなく、APIの要件に従っているだけです。
