## VSCodeとcode-server：ブラウザ側コード編集ソリューションの選定

> ブラウザ側のコード編集機能を構築する際、開発者は重要な選択に直面します：VSCode公式の`code serve-web`機能を使用するか、コミュニティ主導の`code-server`ソリューションを採用するか？この選択は技術アーキテクチャだけでなく、ライセンス準拠性とデプロイの柔軟性にも関係します。

## 背景

実は技術選定って、人生の道を選ぶのにちょっと似ています。ある道を選んだら、ずっとそれを歩き続けることになり、道を変えたくなった時のコストは相当なものになります。

AIアシストプログラミングの時代において、ブラウザ側のコード編集機能はますます重要になっています。ユーザーはAIアシスタントがコード分析を完了した後、同じブラウザセッションですぐにエディタを開いて修正できることを期待しており、アプリケーションを切り替える必要はありません。このシームレスな体験は、どう言えばいいでしょうか、あなたが求めた時に、そこにある――ただし、時々そこになくて困るものです。

しかし、この機能を実装する際、開発者は重要な技術的選択に直面します：VSCode公式の`code serve-web`機能を使用するか、コミュニティ主導の`code-server`ソリューションを採用するか？

これら2つのソリューションはそれぞれ一長一短があり、間違った選択をすると後で多くのトラブルをもたらす可能性があります。例えばライセンス問題――製品がリリースされてからライセンス不準拠だと気づいたら、もう遅いです。これは恋愛にちょっと似ていて、最初によく考えておかないと、結局二人の価値観が全く合わないと気づいて、支払ったコストは大きくなります。また、デプロイ方法――あるソリューションは開発環境ではうまく動いていたのに、コンテナに載せたらいろいろな問題が出る。このような穴、誰も踏みたくありません。毕竟坑踩多了，人也就麻了（穴を踏みすぎると、人は無感覚になる）。

## ライセンスの違い（最も重要）

これが2つのソリューションの最も根本的な違いであり、選定時に最初に考慮する要素でもあります。毕竟選定这事儿，第一步就要想清楚法律风险，不然以后出事了，怪谁呢？（選定の第一歩で法的リスクを明確にしておかないと、後で問題が起きた時に誰のせいにするんですか？）

**code-server**
- MITライセンス、完全オープンソース
- Coder.com社によって保守、コミュニティが活発
- 自由に商用利用、変更、配布可能
- 使用シーンの制限なし

**VSCode `code serve-web`**
- Microsoft VSCode製品の一部
- Microsoftのライセンスを使用（VS Codeのライセンスには商用利用の制限がある）
- 主に個人開発者向け
- エンタープライズデプロイでは追加の商用ライセンスの考慮が必要な場合がある

ライセンスの観点から、code-serverは商用プロジェクトによりフレンドリーです。この点は製品計画段階で明確にしておく必要があります。そうしないと、规模上来了再去迁移，那成本可就大了（規模が大きくなってから移行しようとすると、コストが大きくなります）。毕竟迁移这事儿，说起来容易，做起来难，谁经历过谁知道（移行なんて、言うのは簡単ですが実際は難しく、経験した人にしかわかりません）。

## デプロイ方法の違い

ライセンス問題が解決したら、次はデプロイ方法です。これは直接運用コストとアーキテクチャ設計に影響し、間接的に毎日の気分にも影響します――デプロイが簡単であればあるほど、気分は良くなります。この道理はみんなわかっています。

**code-server**
- 独立したNode.jsアプリケーション、単独でデプロイ可能
- 複数のランタイムソースをサポート：
  - 実行可能ファイルパスを直接指定
  - システムPATHで検索
  - NVM Node.js 22.x環境の自動検出
- サーバーにVSCodeデスクトップ版のインストールは不要
- コンテナ化デプロイがより簡単

**VSCode `code serve-web`**
- ローカルにインストールされたVSCode CLIに依存する必要がある
- ローカルマシンで利用可能な`code`コマンドが必要
- システムはVS Code Remote CLIラッパーをフィルタリングする
- 主にローカル開発シーン向けに設計されている

code-serverはサーバー/コンテナデプロイシーンにより適しています。製品がDockerで動く必要がある場合、またはユーザー環境にVSCodeがない場合、code-serverを選ぶのが正解です。毕竟简单就是美，复杂了容易出问题，出了问题还得修，修完了还可能引入新问题，这无穷无尽的循环，谁愿意经历呢？（シンプルは美であり、複雑だと問題が起きやすく、問題を直しても新しい問題が導入されるかもしれない。この無限ループ、誰が経験したいですか？）

## 機能パラメータの違い

2つのソリューションは機能パラメータにもいくつか違いがあり、大きくはないですが、実際の使用ではいくらかの面倒をもたらす可能性があります。これらの詳細は、生活の小さな摩擦のようなもので、多くはないですが、多くなると煩わしくなります。

| 特性 | code-server | code serve-web |
|------|-------------|----------------|
| 公開ベースパス | `/` (設定可能) | `/vscode-server` (固定) |
| 認証方式 | `--auth`パラメータ、複数のモードをサポート | `--connection-token` / `--without-connection-token` |
| データディレクトリ | `{DataDir}/code-server` | `{DataDir}/vscode-serve-web` |
| テレメトリ | デフォルトで無効 `--disable-telemetry` | VSCode設定に依存 |
| 更新チェック | 無効化可能 `--disable-update-check` | VSCode設定に依存 |

これらの違いは統合時に特に注意が必要です。例えばURLパスの違いは、フロントエンドコードが针对性的な処理をする必要があることを意味します。開発者なら誰でもわかりますが、このような詳細を処理するのが一番時間がかかりますが、仕方がありません。やらないと動きませんから。

## 可用性検出の違い

エディタ切り替え機能を実装する際、可用性検出のロジックも異なります。この違いは、人と人の付き合い方のようなもので、ストレートなのが好きな人もいれば、遠回しなのが好きな人もいます。

**code-server**
- 常に可視実装として返される
- 利用不可能でも表示され、`install-required`状態を提示
- NVM Node.js 22.x環境の自動検出をサポート

**code serve-web**
- ローカルの`code` CLIが検出された場合のみ可視
- 利用不可能な場合、フロントエンドは自動的にこのオプションを非表示にする
- ローカルVSCodeインストール状態に依存

この違いはユーザー体験に直接影響します。code-serverの方法はより透過的で、ユーザーはこのオプションがあることを知っており、まだインストールされていないだけです。code serve-webの方法はより隠れており、ユーザーはこの選択肢があることを知らないかもしれません。どちらの方法が良いですか？これは製品の位置付けによります。毕竟用户体验这事儿，没有标准答案，只有合适不合适（ユーザー体験なんて、標準答えはなく、適しているかいないかだけです）。

これだけ理論を話しましたが、実際のデプロイ時には何に注意すべきでしょうか？其实理论说得再好，落地不行也没用，毕竟实践是检验真理的唯一标准（理論がどんなに良くても、実行できなければ意味がありません。毕竟実践は真理を検証する唯一の基準です）。

### Dockerデプロイ推奨事項

コンテナ化デプロイの場合、code-serverがより良い選択です：

```dockerfile
# code-server公式イメージを直接使用
FROM codercom/code-server:latest

# またはnpmでインストール
RUN npm install -g code-server
```

これで1つで完了し、VSCodeを追加でインストールする必要はありません。简单就是好，复杂了容易出错，这道理放哪儿都适用（シンプルは良いことで、複雑だと間違いやすく、この道理はどこにでも適用されます）。

### 設定例

**code-server設定**
```json
{
  "vscodeServer": {
    "enabled": true,
    "activeImplementation": "code-server",
    "codeServer": {
      "host": "0.0.0.0",
      "port": 8080,
      "executablePath": "",
      "authMode": "none"
    }
  }
}
```

**code serve-web設定**
```json
{
  "vscodeServer": {
    "enabled": true,
    "activeImplementation": "serve-web",
    "serveWeb": {
      "host": "0.0.0.0",
      "port": 8080,
      "executablePath": "/usr/local/bin/code"
    }
  }
}
```

配置这事儿，第一次麻烦点，配好了后面就省心了（設定は最初は少し面倒ですが、設定すれば後は安心です）。就像生活一样，前期投入多一点，后面日子就好过一点（人生と同じで、前期に少し多く投入すれば、後の日子は少し楽になります）。

### URL構築の違い

**code-server**

### 実装の切り替え

システムはランタイム切り替えをサポートし、切り替え時に自動的に古い実装を停止します：

```csharp
// VsCodeServerManagerは相互排他を自動的に処理
// activeImplementationを切り替える時、古い実装はバックグラウンドで存続し続けない
```

この設計により、ユーザーはいつでも異なる実装を試して、自分に最も適したソリューションを見つけることができます。毕竟适合自己的才是最好的，别人的建议仅供参考，最终还得自己试（自分に適したものが最高で、他人の提案は参考にすぎず、結局は自分で試すしかありません）。

## まとめ

| 対比次元 | code-server | code serve-web | 推奨 |
|---------|-------------|----------------|------|
| ライセンス | MIT（商用フレンドリー） | Microsoft（制限あり） | code-server |
| デプロイ柔軟性 | 独立デプロイ | ローカルVSCodeに依存 | code-server |
| サーバー適合性 | サーバー向け設計 | 主にローカル開発向け | code-server |
| コンテナ化 | ネイティブサポート | VSCodeインストールが必要 | code-server |
| 機能完全性 | デスクトップ版に近い | 公式完全版 | code serve-web |
| 保守活力度 | コミュニティ活発 | Microsoft公式 | 各々長所 |

**推奨戦略**：優先的にcode-serverを使用し、完全な公式機能が必要でローカルVSCode環境がある場合、code serve-webを検討します。
