## 段階的開示：「 Less is More 」の理念で AI 製品の人間コンピュータインタラクションを改善する方法

> AI 製品の設計において、ユーザー入力の品質はしばしば出力の品質を左右します。本文では、HagiCode プロジェクトで実践している「段階的開示」インタラクションソリューションを紹介し、ステップバイステップのガイダンス、インテリジェントな補完、即座のフィードバックを通じて、ユーザーの簡潔で曖昧な入力を構造化された技術提案に変換し、人間コンピュータインタラクションの効率を大幅に向上させる方法を共有します。

## 背景

AI 製品を開発している皆さんなら、こんなシーンに遭遇したことがあるはずです：ユーザーがあなたのアプリを開き、意欲的に一行のニーズを入力したのに、AI が返した内容が全く見当違い。AI が賢くないわけではなく、ユーザーが提供した情報が少なすぎるだけです。結局のところ、心を読むなんて誰にもできません。

この現象は、HagiCode を開発する過程で特に顕著でした。HagiCode は AI 主導のコードアシスタントで、ユーザーは自然言語でニーズを記述して技術提案と会話を作成します。しかし実際に使ってみると、ユーザー入力には以下のような問題があることがわかりました：

- **入力品質のばらつき**：一部のユーザーは数文字しか入力せず、「ログイン最適化」「バグ修正」など、必要なコンテキストが欠けている
- **技術用語の不統一**：異なるユーザーが同じことを別の言葉で表現し、ある人は「フロントエンド」、ある人は「FE」と言う
- **構造化された情報の欠如**：プロジェクト背景、リポジトリ範囲、影響範囲といった重要情報がない
- **反復する問題**：同じタイプのニーズが繰り返し現れ、毎回最初から説明しなければならない

これらの問題がもたらす直接的な結果は：AI の理解が困難、生成された提案の品質が不安定、ユーザーエクスペリエンスが悪い。ユーザーは「この AI はダメだ」と思い、私たちも「たった一文で何が欲しいか当てろって？」と愤ります。

実際、これも仕方がありません。人と人との理解さえ時間がかかるのですから、ましてや機械においてはなおさらです。

これらの痛みを解決するため、私たちは大胆な決断を下しました：「段階的開示」の設計理念を導入して人間コンピュータインタラクションを改善するのです。この決断がもたらした変化は、あなたが想像するよりも大きいかもしれません。ただ、当時はこれほど効果があるとは思っていませんでしたが。

## HagiCode について

本文で共有するソリューションは、[HagiCode](https://hagicode.com) プロジェクトでの実践経験から来ています。HagiCode はオープンソースの AI コードアシスタントプロジェクトで、自然言語インタラクションを通じて開発者がコード作成、技術提案生成、コードレビューなどのタスクを完了できることを目的としています。プロジェクトアドレス：[github.com/HagiCode-org/site](https://github.com/HagiCode-org/site)。

この段階的開示ソリューションは、実際の開発過程で何度もの反復と最適化を経て要約したものです。このソリューションに価値を感じてくれたなら、私たちのエンジニアリング力は悪くないということ——ならば HagiCode 自体も注目に値するでしょう。結局、良いものは共有する価値があるのですから。

## 段階的開示とは

「段階的開示」は HCI（人間コンピュータインタラクション）分野に由来する設計原則で、核心的なアイデアはとてもシンプルです：**すべての情報とオプションを一度にユーザーに提示するのではなく、ユーザーの操作とニーズに応じて、必要なコンテンツを段階的に提示する**のです。

この原則は AI 製品に特に適しています。なぜなら AI インタラクションは本質的に段階的だからです——ユーザーが少し話し、AI が少し理解し、そして補充し、さらに理解を深める。人と人との交流のように、やはりゆっくり進める必要があります。結局、会ってすぐ心を開く人なんていませんよね？

具体的に HagiCode のシーンでは、私たちは 4 つの側面から段階的開示を実施しました：

### 1. 記述最適化メカニズム：AI あなたの代わりに言葉を明確にする

ユーザーが簡潔な記述を入力したとき、私たちは AI に直接理解させるのではなく、まず「記述最適化」プロセスをトリガーします。このプロセスの核心は「構造化出力」——ユーザーの自由テキストを標準フォーマットに変換することです。散らばった真珠をネックレスに通すように、見た目もそう混乱しません。

最適化された記述には、以下の標準セクションが含まれている必要があります：

- **背景**：問題の背景とコンテキスト
- **分析**：技術分析と思考プロセス
- **解決**：ソリューションと実装手順
- **実践**：実際のコード例と注意事項

同時に、ターゲットリポジトリ、パス、編集権限などの情報を表示する Markdown テーブルも自動生成し、AI の後続操作を容易にします。結局、明確な目次があれば物を探すのも便利です。

以下が実際のコード実装です：

```csharp
// ProposalDescriptionMemoryService.cs の中核メソッド
public async Task<string> OptimizeDescriptionAsync(
    string title,
    string description,
    string locale = "zh-CN",
    DescriptionOptimizationMemoryContext? memoryContext = null,
    CancellationToken cancellationToken = default)
{
    // クエリパラメータを構築
    var queryContext = BuildQueryContext(title, description);

    // 履歴コンテキストを検索
    var memoryContext = await RetrieveHistoricalContextAsync(queryContext, cancellationToken);

    // 構造化されたプロンプトを生成
    var prompt = await BuildOptimizationPromptAsync(
        title,
        description,
        memoryContext,
        cancellationToken);

    // AI を呼び出して最適化
    return await _aiService.CompleteAsync(prompt, cancellationToken);
}
```

このプロセスの鍵は「メモリ注入」にあります——私たちはプロジェクトの慣習、類似案例、ネガティブパターンなどの履歴コンテキストをプロンプトに注入し、AI が最適化時に過去の経験を参照できるようにします。結局、「失敗は成功の母」、過去の経験を無駄にするわけにはいきません。

**注意事項**：
- 現在の入力が履歴メモリより優先されることを保証し、ユーザーが明示的に指定した情報が上書きされないようにする
- HagIndex 参照は情報ソースとして扱われ、履歴案例によって変更されてはならない
- 低信頼度の修正提案は強制約として注入すべきでない

### 2. 音声入力機能：話す方がタイピングより自然

テキスト入力に加えて、音声入力もサポートしています。これは複雑なニーズを記述する際に特に便利です——考えてみてください、技術的なニーズを入力するのに数分かかるかもしれませんが、言うなら数十秒で終わります。結局、口の方が手より速いです。

音声入力の設計の重点は「状態管理」で、ユーザーは現在のシステムがどのような状態にあるかを明確に把握する必要があります。私たちは以下のいくつかの状態を定義しました：

- **アイドル**：システムが準備完了、録音を開始できる
- **上流待ち**：バックエンドサービスに接続中
- **録音中**：ユーザーの音声を録音中
- **処理中**：音声をテキストに変換中
- **エラー**：エラーが発生し、ユーザーの処理が必要

フロントエンドの状態モデルはだいたいこんな感じです：

```typescript
interface VoiceInputState {
  status: 'idle' | 'waiting-upstream' | 'recording' | 'processing' | 'error';
  duration: number;
  error?: string;
  deletedSet: Set<string>; // 削除された結果の指紋集合
}

// 録音開始時の状態遷移
const handleVoiceInputStart = async () => {
  // まず待機状態に入り、ローディングアニメーションを表示
  setState({ status: 'waiting-upstream' });

  // バックエンドの準備完了確認を待つ
  const isReady = await waitForBackendReady();
  if (!isReady) {
    setState({ status: 'error', error: 'バックエンドサービスが準備完了していません' });
    return;
  }

  // 録音を開始
  setState({ status: 'recording', startTime: Date.now() });
};

// 認識結果を処理
const handleRecognitionResult = (result: RecognitionResult) => {
  const fingerprint = normalizeFingerprint(result.text);

  // 削除されたかどうかを確認
  if (state.deletedSet.has(fingerprint)) {
    return; // 削除されたコンテンツをスキップ
  }

  // 結果をテキストボックスにマージ
  appendResult(result);
};
```

ここに细节があります：私たちは「指紋集合」を使って削除同期を管理しています。音声認識が複数の結果を返すとき、ユーザーはその一部を削除するかもしれません。私たちは削除されたコンテンツの指紋を保存し、後で同じ内容が再び現れたら自動的にスキップします。これはどの料理が嫌いかを覚えているようなもので、次回は注文しません。結局、誰も同じ問題で二度と悩みたくありません。

### 3. プロンプト管理システム：AI の「脳」を外部化

HagiCode には柔軟なプロンプト管理システムがあり、すべてのプロンプトはファイル形式で保存されます：

```
prompts/
├── metadata/
│   ├── optimize-description.zh-CN.json
│   └── optimize-description.en-US.json
└── templates/
    ├── optimize-description.zh-CN.hbs
    └── optimize-description.en-US.hbs
```

各プロンプトは二つの部分から構成されます：
- **メタデータファイル（.json）**：プロンプトのシナリオ、バージョン、パラメータなどの情報を定義
- **テンプレートファイル（.hbs）**：Handlebars 構文を使用する実際のプロンプト内容

メタデータファイルのフォーマットはこんな感じです：

```json
{
  "scenario": "optimize-description",
  "locale": "zh-CN",
  "version": "1.0.0",
  "syntax": "handlebars",
  "syntaxVersion": "1.0",
  "parameters": [
    {
      "name": "title",
      "type": "string",
      "required": true,
      "description": "提案タイトル"
    },
    {
      "name": "description",
      "type": "string",
      "required": true,
      "description": "元の記述"
    }
  ],
  "author": "HagiCode Team",
  "description": "ユーザー入力の技術提案記述を最適化",
  "lastModified": "2026-04-05",
  "tags": ["optimization", "nlp"]
}
```

テンプレートファイルは Handlebars 構文を使用し、パラメータ注入をサポートします：

```handlebars
あなたは技術提案の専門家です。

<task>
以下の情報に基づいて構造化された技術提案記述を生成してください。
</task>

<input>
<title>{{title}}</title>
<description>{{description}}</description>
{{#if memoryContext}}
<memory_context>
{{memoryContext}}
</memory_context>
{{/if}}
</input>

<output_format>
## 背景
[問題の背景とコンテキストを記述し、プロジェクト情報、リポジトリ範囲などを含める]

## 分析
[技術分析と思考プロセス、なぜこの変更が必要かを説明]

## 解決
[ソリューションと実装手順、キーコードの場所をリストアップ]

## 実践
[実際のコード例と注意事項]
</output_format>
```

この設計の利点は：
- プロンプトはコードのようにバージョン管理できる
- 多言語をサポートし、ユーザーの好みに応じて自動切替
- パラメータ化設計で、動的にコンテキストを注入可能
- 起動時に完全性を検証し、実行時エラーを回避

結局、脳の中のものを書き留めないと、いつ忘れるかわかりません。後で後悔するより、最初から記録をしっかりとっておく方がいいでしょう。

### 4. 段階的ウィザード：複雑なタスクを小さなステップに分解

複雑なタスク（例えば初回インストール設定）に対して、私たちはマルチステップウィザードの設計を使用しました。各ステップでは必要な情報のみを要求し、明確な進度表示を提供します。生活も同じです。一口で太った人にはなれません。一歩一歩進む方が逆に安全です。

ウィザードの状態モデル：

```typescript
interface WizardState {
  currentStep: number;           // 0-3、4 つのステップに対応
  steps: WizardStep[];
  canGoNext: boolean;
  canGoBack: boolean;
  isLoading: boolean;
  error: string | null;
}

interface WizardStep {
  id: number;
  title: string;
  description: string;
  completed: boolean;
}

// ステップナビゲーションロジック
const goToNextStep = () => {
  if (wizardState.currentStep < wizardState.steps.length - 1) {
    // 現在のステップの入力を検証
    if (validateCurrentStep()) {
      wizardState.currentStep++;
      wizardState.steps[wizardState.currentStep - 1].completed = true;
    }
  }
};

const goToPreviousStep = () => {
  if (wizardState.currentStep > 0) {
    wizardState.currentStep--;
  }
};
```

各ステップには独立した検証ロジックがあり、完了したステップには明確な視覚マークが付きます。キャンセル操作は確認ダイアログをポップアップし、ユーザーの誤操作による進度の喪失を防ぎます。結局、間違った道を進んでも戻れますが、道をすべて壊してしまったら本当に手の打ちようがありません。

## まとめ

HagiCode の段階的開示の実践を振り返ると、いくつかの中核原則を要約できます：

1. **ステップバイステップガイダンス**：複雑なタスクを小さなステップに分解し、各ステップで必要な情報のみを要求
2. **インテリジェント補完**：履歴コンテキストとプロジェクト知識を利用して情報を自動補完
3. **即座のフィードバック**：各操作には明確な視覚フィードバックと状態提示がある
4. **フォルトトレラントメカニズム**：ユーザーが取り消し、リセットできるようにし、エラーによる不可逆な損失を回避
5. **入力の多様化**：テキスト、音声など複数の入力方式をサポート

このソリューションは HagiCode での実際の効果として：ユーザー入力の平均長は 20 文字未満から構造化された 200-300 文字に向上し、AI が生成した提案の品質は大幅に向上し、ユーザー満足度も上がってきました。

実際、これも不思議ではありません。あなたが提供する情報が多ければ多いほど、AI の理解が正確になり、返された結果も自然と良くなります。これは人と人との交流也没什么两样。

もしあなたも AI 関連製品を開発しているなら、これらの経験が何らかのインスピレーションを与えてくれることを願っています。覚えておいてください：ユーザーは情報を提供したくないのではなく、あなたがまだ正しい質問をしていないだけです。段階的開示の中核は、質問する最適なタイミングと方法を見つけることですが、このタイミングと方法には、少しの忍耐を持って模索する必要があるだけです。
