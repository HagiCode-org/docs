## HagiCode における StreamJsonRpc の深い統合と実践

> 本記事では、HagiCode（旧 PCode）プロジェクトが Microsoft の StreamJsonRpc 通信ライブラリを統合して、独自の JSON-RPC 実装を置き換え、統合過程での技術的課題とアーキテクチャ上の課題を解決した方法を詳しく解説します。

## 背景

StreamJsonRpc は Microsoft が公式にメンテナンスする .NET および TypeScript 向けの JSON-RPC 通信ライブラリで、強力な型安全性、自動プロキシ生成、成熟した例外処理メカニズムで知られています。HagiCode プロジェクトでは、ACP (Agent Communication Protocol) を通じて外部 AI ツール（iflow CLI、OpenCode CLI など）と通信し、初期の独自 JSON-RPC 実装によるメンテナンスコストと潜在的なバグを排除するため、StreamJsonRpc の統合を決定しました。しかし、統合過程でストリーミング JSON-RPC 特有の課題に直面し、特にプロキシターゲットのバインディングとジェネリックパラメータの認識において問題が発生しました。

これらの課題を解決するため、大胆な決断を下しました：ビルドシステム全体を作り直すことです。この決断による変化は、皆さんが想像するよりも大きなものになるかもしれません——詳しくは後ほど説明します。

## 分析

現在のプロジェクトは ACP プロトコル統合の重要な段階にあり、以下の技術的課題とアーキテクチャ上の課題に直面しています：

### 1. 独自実装の限界
元の JSON-RPC 実装は `src/HagiCode.ClaudeHelper/AcpImp/` にあり、`JsonRpcEndpoint` や `ClientSideConnection` などのコンポーネントを含んでいます。この独自コードのメンテナンスコストは高く、進捗報告やキャンセルサポートなど成熟したライブラリの高度な機能が欠けています。

### 2. StreamJsonRpc 統合の障害
既存の `CallbackProxyTarget` パターンを StreamJsonRpc に移行しようとしたところ、`_rpc.AddLocalRpcTarget(target)` メソッドがプロキシモードで作成されたターゲットを認識できないことが判明しました。具体的には、StreamJsonRpc がジェネリック型 `T` のプロパティを RPC メソッドパラメータに自動的に分割できず、サーバー側がクライアントから発信されたメソッド呼び出しを正しく処理できないという問題です。

### 3. アーキテクチャ階層の混乱
既存の `ClientSideConnection` はトランスポート層（WebSocket/Stdio）、プロトコル層（JSON-RPC）、ビジネス層（ACP Agent インターフェース）が混在しており、責任の分離が不明確です。また、`AcpAgentCallbackRpcAdapter` のメソッドバインディングが欠落しています。

### 4. ログの欠如
WebSocket トランスポート層に生の JSON コンテンツのログ出力がなく、RPC 通信の問題をデバッグする際に、シリアライズの問題なのかネットワークの問題なのかを特定するのが困難です。

## 解決策

上記の問題に対し、以下の体系的なソリューションを採用しました。アーキテクチャのリファクタリング、ライブラリの統合、デバッグの強化の3つの次元から最適化を行っています。

### 1. StreamJsonRpc への完全移行

#### レガシーコードの削除
`JsonRpcEndpoint.cs`、`AgentSideConnection.cs`、および関連するカスタムシリアライゼーションコンバータ（`JsonRpcMessageJsonConverter` など）を削除しました。

#### 公式ライブラリの統合
`StreamJsonRpc` NuGet パッケージを導入し、その `JsonRpc` クラスを使用してコア通信ロジックを処理します。

#### トランスポート層の抽象化
`IAcpTransport` インターフェースを定義し、`WebSocket` と `Stdio` の両方のトランスポートモードを統一的に処理し、プロトコル層とトランスポート層の分離を確保します。

```csharp
// IAcpTransport インターフェース定義
public interface IAcpTransport
{
    Task SendAsync(string message, CancellationToken cancellationToken = default);
    Task<string> ReceiveAsync(CancellationToken cancellationToken = default);
    Task CloseAsync(CancellationToken cancellationToken = default);
}

// WebSocket トランスポート実装
public class WebSocketTransport : IAcpTransport
{
    private readonly WebSocket _webSocket;

    public WebSocketTransport(WebSocket webSocket)
    {
        _webSocket = webSocket;
    }

    // 送信・受信メソッドの実装
    // ...
}

// Stdio トランスポート実装
public class StdioTransport : IAcpTransport
{
    private readonly StreamReader _reader;
    private readonly StreamWriter _writer;

    public StdioTransport(StreamReader reader, StreamWriter writer)
    {
        _reader = reader;
        _writer = writer;
    }

    // 送信・受信メソッドの実装
    // ...
}
```

### 2. プロキシターゲット認識問題の修正

#### `CallbackProxyTarget` の分析
既存の動的プロキシ生成ロジックを検査し、StreamJsonRpc が認識できない根本原因を特定します（通常はプロキシオブジェクトが実際のメソッドシグネチャを公開していないか、StreamJsonRpc がサポートしないパラメータタイプを使用しているためです）。

#### パラメータ渡しのリファクタリング
ジェネリックプロパティを明示的な RPC メソッドパラメータに分割します。動的プロパティに依存するのではなく、具体的な Request/Response DTO（データ転送オブジェクト）を定義し、StreamJsonRpc がリフレクションを通じてメソッドシグネチャを正しく認識できるようにします。

```csharp
// 元のジェネリックプロパティ方式
public class CallbackProxyTarget<T>
{
    public Func<T, Task> Callback { get; set; }
}

// リファクタリング後の具体的メソッド方式
public class ReadTextFileRequest
{
    public string FilePath { get; set; }
}

public class ReadTextFileResponse
{
    public string Content { get; set; }
}

public interface IAcpAgentCallback
{
    Task<ReadTextFileResponse> ReadTextFileAsync(ReadTextFileRequest request);
    // その他のメソッド...
}
```

#### `AddLocalRpcTarget` の代わりに `Attach` を使用
一部の複雑なシナリオでは、`JsonRpc` オブジェクトを手動でプロキシして `RpcConnection` を処理する方が、ターゲットを直接追加するよりも柔軟な場合があります。

### 3. メソッドバインディングとログ強化の実装

#### `AcpAgentCallbackRpcAdapter` の実装
このコンポーネントが StreamJsonRpc のプロキシインターフェースを明示的に実装し、ACP プロトコルで定義されたメソッド（`ReadTextFileAsync` など）を StreamJsonRpc のコールバックハンドラーにマッピングするようにします。

#### ロギングの統合
WebSocket または Stdio のメッセージ処理パイプラインで、JSON-RPC リクエストとレスポンスの生テキストをインターセプトして記録します。`ILogger` を使用して、解析前とシリアライズ後に生のペイロードを出力し、フォーマットの問題をより簡単に診断できるようにします。

```csharp
// ログ強化トランスポートラッパー
public class LoggingAcpTransport : IAcpTransport
{
    private readonly IAcpTransport _innerTransport;
    private readonly ILogger<LoggingAcpTransport> _logger;

    public LoggingAcpTransport(IAcpTransport innerTransport, ILogger<LoggingAcpTransport> logger)
    {
        _innerTransport = innerTransport;
        _logger = logger;
    }

    public async Task SendAsync(string message, CancellationToken cancellationToken = default)
    {
        _logger.LogTrace("Sending message: {Message}", message);
        await _innerTransport.SendAsync(message, cancellationToken);
    }

    public async Task<string> ReceiveAsync(CancellationToken cancellationToken = default)
    {
        var message = await _innerTransport.ReceiveAsync(cancellationToken);
        _logger.LogTrace("Received message: {Message}", message);
        return message;
    }

    public async Task CloseAsync(CancellationToken cancellationToken = default)
    {
        _logger.LogDebug("Closing connection");
        await _innerTransport.CloseAsync(cancellationToken);
    }
}
```

### 4. アーキテクチャ階層のリファクタリング

#### トランスポート層（`AcpRpcClient`）
StreamJsonRpc 接続をカプセル化し、`InvokeAsync` 呼び出しと接続ライフサイクルの管理を担当します。

```csharp
public class AcpRpcClient : IDisposable
{
    private readonly JsonRpc _rpc;
    private readonly IAcpTransport _transport;

    public AcpRpcClient(IAcpTransport transport)
    {
        _transport = transport;
        _rpc = new JsonRpc(new StreamRpcTransport(transport));
        _rpc.StartListening();
    }

    public async Task<TResponse> InvokeAsync<TResponse>(string methodName, object parameters)
    {
        return await _rpc.InvokeAsync<TResponse>(methodName, parameters);
    }

    public void Dispose()
    {
        _rpc.Dispose();
        _transport.Dispose();
    }

    // StreamRpcTransport は IAcpTransport の StreamJsonRpc アダプター
    private class StreamRpcTransport : IDuplexPipe
    {
        // IDuplexPipe インターフェースの実装
        // ...
    }
}
```

#### プロトコル層（`IAcpAgentClient` / `IAcpAgentCallback`）
明確な client-to-agent と agent-to-client のインターフェースを定義します。`Func<IAcpAgent, IAcpClient>` のような循環依存のファクトリパターンを削除し、依存性の注入または直接のコールバック登録に置き換えます。

## 実践

StreamJsonRpc のベストプラクティスとプロジェクトの経験に基づき、実装時の重要な推奨事項を以下にまとめます：

### 1. 強い型の DTO がダイナミックオブジェクトより優れている
StreamJsonRpc のコアな利点は強い型付けにあります。`dynamic` や `JObject` を使用してパラメータを渡さないでください。各 RPC メソッドに対して明示的な C# POCO クラスをパラメータとして定義する必要があります。これにより、プロキシターゲットの認識問題が解決されるだけでなく、コンパイル時に型エラーを検出できます。

例：`CallbackProxyTarget` のジェネリックプロパティを `ReadTextFileRequest` や `WriteTextFileRequest` などの具象クラスに置き換えます。

### 2. メソッド名を明示的に宣言する
`[JsonRpcMethod]` 属性を使用して RPC メソッド名を明示的に指定し、デフォルトのメソッド名マッピングに依存しないようにします。これにより、PascalCase と camelCase などの命名スタイルの違いによる呼び出し失敗を防げます。

```csharp
public interface IAcpAgentCallback
{
    [JsonRpcMethod("readTextFile")]
    Task<ReadTextFileResponse> ReadTextFileAsync(ReadTextFileRequest request);

    [JsonRpcMethod("writeTextFile")]
    Task WriteTextFileAsync(WriteTextFileRequest request);
}
```

### 3. 接続状態コールバックを活用する
StreamJsonRpc は `JsonRpc.ConnectionLost` イベントを提供しています。プロセスの予期せぬ終了やネットワーク切断を処理するために、必ずこのイベントをリッスンしてください。これは Orleans の Grain 失敗検出のみに依存するよりも迅速です。

```csharp
_rpc.ConnectionLost += (sender, e) =>
{
    _logger.LogError("RPC connection lost: {Reason}", e.ToString());
    // 再接続ロジックの処理またはユーザーへの通知
};
```

### 4. ログを階層的に記録する
- **Trace レベル**：完全な JSON Request/Response の生データを記録
- **Debug レベル**：メソッドコールスタックとパラメータの概要を記録
- **注意**：ログに機密性の高い Authorization Token や大容量ファイルの Base64 エンコードが含まれないようにしてください

### 5. ストリーミング転送の特殊性を処理する
StreamJsonRpc は `IAsyncEnumerable` をネイティブにサポートしています。ACP のストリーミング Prompt レスポンスを実装する際、カスタムページネーションロジックではなく、`IAsyncEnumerable` を直接使用してください。これによりストリーミング処理のコード量が大幅に簡略化されます。

```csharp
public interface IAcpAgentCallback
{
    [JsonRpcMethod("streamText")]
    IAsyncEnumerable<string> StreamTextAsync(StreamTextRequest request);
}
```

### 6. アダプタパターン
`ACPSession` と `ClientSideConnection` を分離して維持します。`ACPSession` は Orleans の状態管理とビジネスロジック（メッセージのキューイングなど）に集中し、継承ではなくコンポジションを通じて StreamJsonRpc 接続オブジェクトを使用するようにします。

## まとめ

StreamJsonRpc の全面的な統合により、HagiCode プロジェクトは元の独自実装のメンテナンスコストの高さ、機能の限界、アーキテクチャ階層の混乱などの問題を解決しました。主な改善点は以下の通りです：

1. 強い型の DTO を動的プロパティに置き換え、コードの保守性と信頼性を向上
2. トランスポート層の抽象化とプロトコル層の分離を実装し、アーキテクチャの明確さを向上
3. ロギング機能を強化し、通信問題の診断を容易にした
4. ストリーミングサポートを導入し、ストリーミング処理の実装を簡略化

これらの改善により、HagiCode はより安定で効率的な通信基盤を獲得し、外部 AI ツールとのより良い対話を可能にし、将来の機能拡張のための強固な基盤を築きました。
