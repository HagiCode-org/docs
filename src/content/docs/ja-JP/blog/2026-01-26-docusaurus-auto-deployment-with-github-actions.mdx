## HagiCodeにGitHub Pages自動デプロイサポートを追加

> 本プロジェクトの初期コードネームはPCodeでしたが、現在は正式にHagiCodeに改名されました。本文では、プロジェクトに自動的な静的サイトデプロイ機能を導入し、コンテンツの公開を水を飲むように簡単にする方法を記録しています。

## 背景/はじめに

HagiCodeの開発プロセスにおいて、私たちは非常に現実的な問題に直面しました。ドキュメントや提案が増えるにつれて、これらのコンテンツを効率的に管理・表示する方法が急務となりました。静的サイトをホストするためにGitHub Pagesを導入することにしましたが、手動でのビルドとデプロイは本当に面倒でした——変更のたびにローカルでビルド・パッケージ化し、手動で`gh-pages`ブランチにプッシュする必要があります。これは効率が悪いだけでなく、エラーも発生しやすくなります。

この問題を解決するために（主に怠けたいために）、自動化されたデプロイプロセスが必要でした。本文では、HagiCodeプロジェクトにGitHub Actionsの自動デプロイサポートを追加する方法を詳しく記録し、コンテンツ作作に集中できるようにし、残りは自動化プロセスに任せられるようにします。

## 目標分析

手を動かす前に、このタスクで一体何をするのかを明確にする必要があります。毕竟（ここがタイポです、正しくは「やはり」です）研ぎ刃は薪を伐るのを妨げない（準備が大切）ということです。

### コア要件

1.  **自動ビルド**：コードが`main`ブランチにプッシュされたとき、自動的にビルドプロセスをトリガーします。
2.  **自動デプロイ**：ビルド成功後、生成された静的ファイルを自動的にGitHub Pagesにデプロイします。
3.  **環境の一貫性**：CI環境とローカルビルド環境の一貫性を確保し、「ローカルでは動くのに本番ではエラー」という厄介な状況を回避します。

### 技術選択

HagiCodeがDocusaurus（非常に人気のあるReact静的サイトジェネレーター）に基づいて構築されていることを考慮すると、GitHub Actionsを利用してこの目標を達成できます。

## GitHub Actionsワークフローの設定

GitHub ActionsはGitHubが提供するCI/CDサービスです。コードリポジトリにYAML形式のワークフローファイルを定義することで、様々な自動化タスクをカスタマイズできます。

### ワークフローファイルの作成

プロジェクトのルートディレクトリにある`.github/workflows`フォルダに新しい設定ファイルを作成する必要があります。例えば`deploy.yml`などです。フォルダが存在しない場合は、忘れずに手動で作成してください。

この設定ファイルのコアロジックは以下の通りです：

1.  **トリガー条件**：`main`ブランチの`push`イベントを監視します。
2.  **実行環境**：最新版のUbuntu。
3.  **ビルドステップ**：
    -   コードのチェックアウト
    -   Node.jsのインストール
    -   依存関係のインストール (`npm install`)
    -   静的ファイルのビルド (`npm run build`)
4.  **デプロイステップ**：公式の`action-gh-pages`を使用してビルド成果物を`gh-pages`ブランチにプッシュします。

### 重要な設定コード

以下が、私たちが最終的に採用した設定テンプレートです：

```yaml
name: Deploy to GitHub Pages

# トリガー条件：mainブランチにプッシュされたとき
on:
  push:
    branches:
      - main
    # 必要に応じてパスフィルタを追加できます。例：ドキュメントの変更時のみビルド
    # paths:
    #   - 'docs/**'
    #   - 'package.json'

# 権限の設定、GitHub Pagesへのデプロイに重要です
permissions:
  contents: read
  pages: write
  id-token: write

# 並行性制御：同一ブランチの古いビルドをキャンセル
concurrency:
  group: "pages"
  cancel-in-progress: false

jobs:
  build:
    runs-on: ubuntu-latest
    steps:
      - name: Checkout
        uses: actions/checkout@v4
        # 注意：fetch-depth: 0を設定しないと、ビルドバージョン番号が不正確になる可能性があります
        with:
          fetch-depth: 0

      - name: Setup Node
        uses: actions/setup-node@v4
        with:
          node-version: 20 # ローカル開発環境と一致させることを推奨
          cache: 'npm'     # キャッシュを有効にするとビルドプロセスが高速化されます

      - name: Install dependencies
        run: npm ci
        # npm installではなくnpm ciを使用します。CI環境向けに高速で厳密だからです

      - name: Build website
        run: npm run build
        env:
          # サイトのビルドに環境変数が必要な場合はここで設定
          # NODE_ENV: production
          # PUBLIC_URL: /your-repo-name
          
      - name: Upload artifact
        uses: actions/upload-pages-artifact@v3
        with:
          path: ./build # Docusaurusのデフォルト出力ディレクトリ

  deploy:
    environment:
      name: github-pages
      url: ${{ steps.deployment.outputs.page_url }}
    runs-on: ubuntu-latest
    needs: build
    steps:
      - name: Deploy to GitHub Pages
        id: deployment
        uses: actions/deploy-pages@v4
```

## 実装プロセスでの落とし穴

実際の操作では、いくつかの問題に遭遇しました。ここで共有することで、皆さんが回避できる（または解決策を事前に準備できる）ことを願っています。

### 1. GitHubトークンの権限問題

最初の設定時、デプロイで常に403 (Forbidden)エラーが発生していました。長時間調査した結果、GitHubのデフォルトの`GITHUB_TOKEN`にはPagesへの書き込み権限がないことが判明しました。

**解決策**：リポジトリの`Settings` -> `Actions` -> `General` -> `Workflow permissions`で、必ず**"Read and write permissions"**を選択してください。

### 2. ビルドディレクトリのパスエラー

Docusaurusはデフォルトでビルドされた静的ファイルを`build`ディレクトリに配置します。しかし、一部のプロジェクト（Create React Appのデフォルトは`build`、Viteのデフォルトは`dist`など）では設定が異なる場合があります。Actionsでファイルが見つからないエラーが発生した場合は、`docusaurus.config.js`で出力パスの設定を確認してください。

### 3. サブパス問題

リポジトリがユーザーホームページ（`username.github.io`）ではなく、プロジェクトホームページ（例：`username.github.io/project-name`）である場合、`baseUrl`を設定する必要があります。

`docusaurus.config.js`で：

```javascript
module.exports = {
  // ...
  url: 'https://hagicode.com', // あなたのHagicode URL
  baseUrl: '/', // ルートパスデプロイ
  // ...
};
```
これは見落とされがちです。設定が正しくないと、リソースパスが読み込めず、ページが真っ白になります。

## 成果の検証

すべての設定を完了してコードをプッシュした後、GitHubリポジトリの**Actions**タブで状況を確認できます。

黄色い円（ワークフロー実行中）が表示され、緑色になれば成功です！赤くなった場合は、クリックしてログを確認してください。通常、問題を特定できます（ほとんどの場合、スペルミスやパス設定の間違いです）。

ビルド成功後、`https://<あなたのユーザー名>.github.io/<リポジトリ名>/` にアクセスすると、新しくなったサイトを確認できます。

## まとめ

GitHub Actionsを導入することで、HagiCodeドキュメントサイトの自動デプロイを正常に実現しました。これにより、手動操作の時間を節約できるだけでなく、公開プロセスの標準化も保証されます。現在、どの仲間がドキュメントを更新しても、`main`ブランチにマージするだけで、数分後にオンラインで最新のコンテンツを確認できます。

**コアメリット**：
- **効率向上**：「手動パッケージ化・手動アップロード」から「コード即公開」へ。
- **エラー削減**：人的操作ミスの可能性を排除。
- **体験最適化**：開発者が面倒なデプロイプロセスに煩わされることなく、コンテンツの品質に集中できるように。

最初はCI/CDの設定が少し面倒かもしれません（特に様々な権限やパスの問題）、しかし一度きりの投資で長期的に大きな見返りがある作業です。すべての静的サイトプロジェクトで同様の自動化プロセスを導入することを強くお勧めします。
