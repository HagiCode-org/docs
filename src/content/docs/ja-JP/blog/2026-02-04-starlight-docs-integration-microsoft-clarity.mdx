## 技術選定と探索

最初、プロポーザル段階で複数の統合方法を議論しました。Starlight（Astro ベースのドキュメントフレームワーク）を使用しているので、最も直感的なアイデアは Astro のフックを活用することでした。

まず `astro.config.mjs` を変更し、ビルド時に Clarity スクリプトを注入することを試みました。この方法は全局をカバーできますが、柔軟性に欠けます。ユーザーの設定に基づいてスクリプトを動的にロード・アンロードできません。

ユーザー体験とプライバシー制御を考慮し、最終的に**コンポーネントオーバーライド**のソリューションを採用しました。Starlight は開発者が内部コンポーネントをオーバーライドすることを許可しており、`<footer>` や `<head>` のレンダリングロジックを引き継ぐことで、Clarity の読み込みタイミングを細かく制御できます。

ここで小さなエピソードがあります。当初 `StarlightWrapper.astro` という名前のレイアウトラッパーを作成しようとしました。しかし実際のデバッグで、Starlight のルーティングメカニズムはこのカスタム Wrapper を自動的に呼び出さないことがわかり、一部のページでスクリプトが無効になりました。これは典型的な「想定」による落とし穴の経験であり、**フレームワークのレンダリングフローを深く理解する必要があり、汎用的なフレームワークパターンを盲目に適用すべきではない**ということを教えてくれました。

## コアソリューション：Footer コンポーネントオーバーライド

Clarity スクリプトがすべてのページ（ドキュメントとブログを含む）で確実に読み込まれ、元のページ構造を破壊しないようにするために、Starlight の `Footer` コンポーネントをオーバーライドすることを選択しました。

### なぜ Footer なのか？

1.  **全局性**：Footer はほぼすべての標準ページに表示されます。
2.  **非侵襲性**：スクリプトを Footer 領域（実際には body の下部にレンダリング）に配置しても、ページのクリティカルレンダリングパス（LCP）をブロックせず、パフォーマンスへの影響が最小限です。
3.  **ロジックの集中化**：コンポーネント内部で Cookie 同意ロジックを一元的に処理できます。

`src/components/` ディレクトリにファイル `StarlightFooter.astro` を作成します。Starlight はこのファイルを自動的に認識し、デフォルトの Footer をオーバーライドします。

コアコードロジックは以下の通りです：

```astro
---
```

**重要点の解説**：
- `is:inline`：Astro にこの script タグ内の内容を処理せず、直接 HTML に出力するよう指示します。これはサードパーティの分析スクリプトにとって重要です。そうしないと、Astro のバンドル最適化によりスクリプトが無効になる可能性があります。
- `define:vars`：これは Astro 3+ の機能で、スコープ内に安全に変数を注入できます。
- `import.meta.env.PROD`：ローカル開発時（デバッグ目的を除く）に無効な統計データが生成されないようにし、データの純潔性を保ちます。

## 経験のまとめと落とし穴

このソリューションを HagiCode に展開する過程で、見落とされがちな詳細をいくつかまとめました：

1.  **`StarlightWrapper.astro` は罠です**：
    前述の通り、スクリプトを注入するためにグローバル Wrapper を作成しようとしないでください。これは Starlight では機能しません。素直に特定のコンポーネント（`StarlightFooter.astro` や `StarlightHead.astro` など）をオーバーライドするのが正解です。

2.  **スクリプト配置のパフォーマンス考慮**：
    Clarity はデータの正確性を確保するために `<head>` への配置を推奨していますが、ドキュメントサイトの場合、ファーストスクリーンの読み込み速度（LCP）は SEO とユーザー維持率に直接影響します。私たちは Footer（Body の下部）に配置することを選択しました。これにより「秒で離脱」するごく少数のユーザーデータが少し失われますが、より高速なページ読み込み体験が得られ、これは価値のあるトレードオフです。

3.  **開発環境の干渉**：
    必ず `import.meta.env.PROD` 判定を追加してください。開発モードでは、ページを頻繁に更新することになり、これにより大量の無意味なテストデータが生成され、Clarity ダッシュボードを汚染します。

## 効果の検証

展開完了後、Clarity コンソールでリアルタイムデータを確認できます。通常数分以内に、ユーザーのヒートマップと録画が表示されます。

HagiCode の場合、これらのデータから以下のことがわかりました：
- 多くのユーザーが「クイックスタート」セクションを繰り返し見ており、インストールガイドがまだ十分直感的ではない可能性があります。
- 「API リファレンス」ページの滞在時間が最も長く、コアユーザー群のニーズを裏付けています。

## まとめ

Microsoft Clarity の導入には、複雑なサーバー側改造も、重い SDK の導入も必要ありません。

Starlight のコンポーネントオーバーライドメカニズムを活用し、軽量な `StarlightFooter.astro` コンポーネントだけで、グローバルデータ統計を実現しました。この「マイクロ統合」方式は、コードの整洁さを保ちながら、ユーザー行動を洞察する能力を与えてくれます。

技術的なプロジェクトを運営している場合、特に **HagiCode** のようにドキュメントを継続的に反復改善する必要があるプロジェクトでは、Clarity の導入を強くお勧めします。データは、ユーザーの真の痛みがどこにあるかを教えてくれます。
