## ゼロから始める：Reasonix CLI を HagiCode システムに統合する方法

> 本記事では、Reasonix CLI をファーストクラス Agent Provider として HagiCode システムに統合する完全な技術実践を共有し、3層アーキテクチャ設計、主要な技術的決定事項、およびフロントエンド/バックエンドの実装詳細を網羅しています。

## 背景

Reasonix CLI は、言ってみれば面白いものです。これは ACP（Agent Communication Protocol）に基づいた AI コードアシスタントツールで、強力なストリーミング伝送とセッション管理機能を提供しています。実は HagiCode.Libs レイヤーでは、その基礎的な実装はすでに完了していますが、これらのコンポーネントはまだ孤立した状態にあり、きれいな真珠がまだネックレスに通されていないようなものです。ユーザーは Hero 職業選択、セッション実行チェーン、またはモニタリングパネルを通じてそれを使用することができず、これは少し残念です。

私たちが直面している問題は、Reasonix を Codex、Hermes と同等レベルのファーストクラス Agent Provider に昇格させ、完全なバックエンドルーティングとフロントエンド表示を実現するにはどうすればよいかということです。これは単に列挙値を登録するだけでは済まず、基礎的な抽象化からユーザーインターフェースまでの完全なチェーンを構築する必要があります。家を建てるのと同じで、基礎を打つだけでは終わらず、壁を積み上げ、屋根を被せる必要があります。

この統合の課題は、Reasonix がローカル CLI ツールとして独自の性格を持っていることです。例えば接続文字列は不要で、すべてのパラメータはユーザーの実行時設定です。そもそもインストールされていない可能性があり、優雅なデグラデーション処理が必要です。anthropic シリーズのモデルと互換性がありますが、独自の effort、budget などの ACP 固有パラメータもあります。これは人間と同じで、独自の物事への対処方法があり、無理強いはできません。

慎重なアーキテクチャ設計と複数回の議論を経て、最終的に明確な3層アーキテクチャ案を採用し、Reasonix をシステムに正常に統合しました。この案は現在の問題を解決しただけでなく、同様の CLI Provider の統合のための再利用可能なパターンも提供しています。実際、多くのことはこうです。正しい方法が見つかれば、その後の道はずっと歩みやすくなります。

## HagiCode について

本記事で共有する内容は、[HagiCode](https://hagicode.com) プロジェクトでの実践経験に基づいています。HagiCode はオープンソースの AI コードアシスタントプロジェクトで、開発者に強力なコード生成、リファクタリング、最適化機能を提供することを目指しています。開発過程で、様々な技術的課題に直面し、Reasonix をファーストクラス Agent Provider として統合することもその一つでした。本記事で共有する案に価値を感じていただけた場合、私たちのエンジニアリング実践は悪くないということですので、HagiCode 自体も注目に値するかもしれません。

## コア内容

### 技術アーキテクチャ設計

システムは関心の分離のために明確な3層アーキテクチャを採用し、各レイヤーには明確な責務境界があります：

**HagiCode.Libs レイヤー**：このレイヤーはすでに完了しており、CLI provider の抽象化と具体的な実装を提供しています。`ICliProvider<ReasonixOptions>` インターフェースを定義し、ACP ストリーミング伝送とセッション管理を処理する `ReasonixProvider` を実装し、effort、budget、yolo、transcript などのパラメータを同時にサポートしています。このレイヤーの責務は安定して再利用可能な基礎機能を提供することで、ビジネスロジックは関与しません。家の基礎のように、見えませんが非常に重要です。

**hagicode-core レイヤー**：これが今回の統合の重点です。基礎的な抽象化をシステムの統一インターフェースにブリッジすることがその役割です。具体的な作業には、`AIProviderType.ReasonixCli = 12` 列挙値の登録、Libs レイヤーにブリッジする `ReasonixCliProvider` を thin adapter として作成、セッション状態と実行フローを処理する `ReasonixGrain` の実装、および Hero システムとの統合によるパラメータマッピングと構成管理が含まれます。このレイヤーの核心は各コンポーネントを調整し、完全なビジネスチェーンを構築することです。家の耐力壁のように、各部分を接続します。

**web レイヤー**：ユーザーへの表示と構成の収集を担当します。新しい列挙値をサポートするために OpenAPI タイプを再生成し、Reasonix が独自のアイコンと表示名を持つように視覚タイプマッピングを実装し、ユーザーが各パラメータを設定できるように CLI パラメータ構成フォームを作成し、多言語サポートを追加する必要があります。このレイヤーの重点はユーザーエクスペリエンスとインタラクションデザインです。家の内装のように、良いかどうかが居住の快適さに直接影響します。

このようなレイヤー設計により、各レイヤーは独自の責務に集中し、システムの複雑さを軽減し、後続の保守と拡張も容易になります。実際、多くの場合、物事を明確に分けたほうが、かえってシンプルになります。

### 主要な技術的決定事項

実装過程で、最終的なアーキテクチャとユーザーエクスペリエンスに重要な影響を与えるいくつかの主要な技術的決定を行いました。

**決定事項1：専用 Grain の使用**

既存の共有 Grain を再利用しようとするのではなく、独立した `ReasonixGrain : IReasonixGrain, IExecutorStreamGrain` を作成しました。この決定は、システムの既存の11個の provider の既定パターンに従っています。コードの重複があるように見えますが、専用 Grain により Reasonix の特性に対して細かい制御が可能になります。例えば、その独自のセッションバインディングメカニズムと ACP メッセージマッピングなどです。また、空の応答 DTO `ReasonixResponse` を型の区別子として定義しましたが、実際のデータは含まれていませんが、型システムで重要な役割を果たしています。すべての人が自分の部屋を持っているように、空っぽでも、自分の空間です。

**決定事項2：専用 Settings クラスを作成しない**

接続文字列が必要な一部の Provider とは異なり、Reasonix のすべての構成はユーザーの実行時に設定されるため、起動時の検証は不要です。そのため、専用の Settings クラスを作成せず、すべての構成を `AIProviderOptions.Providers[ReasonixCli].Settings` 辞書に保存しました。このパターンは Qoder、Kiro、Kimi などの他のローカル CLI Provider と一致しており、コード構造を簡素化し、不必要な抽象レイヤーを回避しました。サポートされる設定キーには：`effort`、`budgetUsd`、`transcriptPath`、`enableYolo`、`arguments`、`startupTimeoutMs`、`reasoning` があります。時にはシンプルなほうが良いこともあります。

**決定事項3：Provider 戦略ヘルスモニタリング**

Reasonix はユーザーがローカルにインストールした CLI であり、インストールされていないか、システム PATH にない可能性があります。この場合、直接エラーを出すのではなく、優雅にデグラデーション処理を行うべきです。`Provider` 戦略を使用し、`CommandUtil.TryResolveExecutablePath` を通じて CLI が使用可能かどうかをチェックします。チェックに失敗した場合、UI は「使用不可」と表示されますが、システムの他の部分には影響しません。この設計によりシステムがより堅牢になり、ユーザーにも明確なフィードバックが提供されます。誰も小さな問題でシステム全体が停止することは望んでいません。

**決定事項4：経済システム分類**

HagiCode システムでは、異なる Provider が異なる経済システム分類を持っています。Reasonix がデフォルトで `'claude'` 経済システム分類を使用するように決定しました。これは Reasonix 自体が anthropic シリーズのモデルと互換性があるためです。現在、Codex と Copilot のみが専用の経済システム分類を持っており、他の Provider は既存の分類を再利用しています。これによりシステムの簡潔さを保ちつつ、課金とコスト統計を正しく処理できます。再利用も知恵で、すべてを一から始める必要はありません。

**決定事項5：モデル互換性**

Reasonix は `--model` フラグを通じて複数のモデル、特に anthropic シリーズをサポートしています。`secondary-professions.index.json` に互換性マッピングを追加し、ユーザーが Reasonix でこれらのモデルを選択できるようにしました。この設計は Reasonix の能力を尊重しつつ、システムの一貫性を保ち、ユーザーは基礎的な違いを理解する必要なく、各種モデルを円滑に使用できます。ユーザーも簡単ではないので、シンプルにしてあげましょう。

**Thin Adapter 実装**

`ReasonixCliProvider` は Libs レイヤーとシステムの統一インターフェースを接続する重要なコンポーネントです：

```csharp
public sealed class ReasonixCliProvider : IAIProvider, IVersionedAIProvider, IAsyncDisposable
{
    private static readonly IReadOnlyList<string> SupportedSettingKeys =
    [
        "effort",
        "budgetUsd",
        "transcriptPath",
        "enableYolo",
        "arguments",
        "startupTimeoutMs",
        "reasoning"
    ];

    private readonly ICliProvider<ReasonixOptions> _provider;
    private readonly ConcurrentDictionary<string, string> _sessionBindings = new(StringComparer.Ordinal);

    public async IAsyncEnumerable<AIStreamingChunk> StreamCoreAsync(
        AIRequest request,
        string? sessionId = null,
        [EnumeratorCancellation] CancellationToken cancellationToken = default)
    {
        var options = BuildOptions(request, sessionId);
        await foreach (var message in _provider.StreamAsync(options, cancellationToken))
        {
            yield return MapToStreamingChunk(message);
        }
    }

    private ReasonixOptions BuildOptions(AIRequest request, string? sessionId)
    {
        return new ReasonixOptions
        {
            ExecutablePath = GetExecutablePath(),
            WorkingDirectory = GetWorkingDirectory(),
            Model = _config.Model,
            Effort = _config.Settings.GetValueOrDefault("effort", "medium"),
            Budget = _config.Settings.GetValueOrDefault("budgetUsd", 10.0),
            Yolo = _config.Settings.GetValueOrDefault("enableYolo", false),
            TranscriptPath = _config.Settings.GetValueOrDefault("transcriptPath"),
            Arguments = _config.Settings.GetValueOrDefault("arguments", ""),
            StartupTimeout = GetStartupTimeout(),
            EnvironmentVariables = _environmentVariables,
            CessionId = sessionId ?? GetCessionId()
        };
    }
}
```

この adapter の主な責務は：
1. 構成パラメータを検証し、サポートされていない設定キーを拒否する
2. セッションバインディング関係を維持し、セッション回復をサポートする
3. ACP メッセージをシステムの統一形式にマッピングする
4. `ProviderErrorAutoRetryCoordinator` を使用して自動再試行を実装する

通訳のように、ある言語を別の言語に翻訳し、意味が正確であることを保証する必要があります。

**Orleans Grain 実装**

`ReasonixGrain` はセッション状態と実行フローの処理を担当します：

```csharp
public class ReasonixGrain : Grain, IReasonixGrain, IExecutorStreamGrain
{
    private readonly Dictionary<string, ExecutorToolLifecycleStatus> _toolLifecycleState =
        new(StringComparer.Ordinal);

    public IAsyncEnumerable<ReasonixResponse> ExecuteCommandStreamAsync(
        string command,
        string? heroId = null,
        CancellationToken token = default,
        string? executionMessageId = null,
        string? systemMessage = null,
        Dictionary<string, string>? requestSettings = null)
    {
        var request = BuildRequest(command, isEdit: false, heroId, executionMessageId, systemMessage, requestSettings);
        return SendAsync(request, heroId, token);
    }

    private async IAsyncEnumerable<ReasonixResponse> SendAsync(
        AIRequest request,
        string? heroId,
        [EnumeratorCancellation] CancellationToken token)
    {
        _cancellationTokenSource = new CancellationTokenSource();
        var linkedToken = CancellationTokenSource.CreateLinkedTokenSource(token, _cancellationTokenSource.Token);

        var provider = await ResolveReasonixProviderAsync(heroId);

        await foreach (var chunk in provider.StreamAsync(request, linkedToken.Token))
        {
            var response = BuildChunkResponse(chunk);
            yield return response;
        }
    }

    private async Task<IAIProvider> ResolveReasonixProviderAsync(string? heroId)
    {
        // Hero 対応の構成フォールバックロジック
        var config = await HeroProviderResolver.ResolveAsync(AIProviderType.ReasonixCli, heroId);
        return _aiProviderFactory.CreateProvider(AIProviderType.ReasonixCli, config);
    }
}
```

Grain の核心機能には：
1. `[PersistentState("reasonix-interop")]` を使用してセッション状態を維持する
2. Hero 対応の構成フォールバックロジックを実装する
3. ツールライフサイクル状態を追跡する
4. キャンセレーショントークンチェーンをサポートし、実行が適時に中断できるようにする

家事を整理する執事のように、物事を整然と管理します。

### ベストプラクティスと注意事項

実装過程で、いくつかのベストプラクティスと注意すべき点をまとめました。

**パラメータ検証**

ReasonixCliProvider は構成パラメータを厳密に検証し、サポートされていない設定キーを拒否する必要があります：

```csharp
public void ValidateConfigurationOverrides(Dictionary<string, string?> overrides)
{
    foreach (var key in overrides.Keys)
    {
        if (!SupportedSettingKeys.Contains(key))
        {
            throw new HeroProviderConfigurationException(
                $"Unsupported setting key '{key}' for Reasonix provider");
        }
    }
}
```

これにより、ユーザーが間違ったパラメータを構成するのを防ぎ、実行時エラーを回避できます。ゴールキーパーのように、入ってはいけないものを混ぜないようにする必要があります。

**セッションバインディング管理**

`ConcurrentDictionary` を使用してセッションバインディングを管理し、セッション回復をサポートします：

```csharp
_sessionBindings[cessionId] = sessionId;

// 後続のリクエストで既存のセッションをバインド
if (_sessionBindings.TryGetValue(cessionId, out var boundSessionId))
{
    options.SessionId = boundSessionId;
}
```

この設計により、ユーザーは中断後に以前のセッションを回復でき、より良いエクスペリエンスを提供できます。物語を記憶して、次回も続きを語れるようにするようなものです。

**優雅なデグラデーション処理**

フロントエンドは CLI の使用可能性をチェックし、フレンドリーなプロンプトを提供する必要があります：

```typescript
const reasonixAvailable = await healthApi.checkCliAvailable('reasonix');
if (!reasonixAvailable) {
  showMessage('Reasonix CLI がインストールされていません。システム PATH で構成してください');
}
```

ユーザーが不可解なエラーに遭遇しないように、事前にチェックして明確な指導を提供します。誰も不可解にブロックされることは望みません。

**テストカバレッジ**

完全なテストは品質保証の鍵です：

```csharp
[Fact]
public async Task ExecuteCommandStreamAsync_WithValidCommand_StreamsReasonixResponse()
{
    // Arrange
    var grain = _grainFactory.GetGrain<IReasonixGrain>("test-cession");
    var responses = new List<ReasonixResponse>();

    // Act
    await foreach (var response in grain.ExecuteCommandStreamAsync("help"))
    {
        responses.Add(response);
    }

    // Assert
    responses.Should().NotBeEmpty();
    responses.All(r => r.Kind == ExecutorResponseKind.Content).Should().BeTrue();
}
```

このような単体テストにより、核心機能が正常に動作しているかを検証し、回帰エラーを防ぐことができます。試験前に練習問題をするのと同じで、本当に習得したことを確認する必要があります。
