## HagiCode Soul プラットフォーム技術解析：ニーズの萌芽から独立プラットフォームへの進化

> 技術記事を書くことなんて、大したことじゃない。ただ踏んだ坑や回り道を整理しただけだ。誰にも若い時期があるものだ。本文では、HagiCode プロジェクトにおける Soul（AI Agent パーソナリティ設定システム）の設計理念、アーキテクチャの進化、核心技術の実装を深く解析し、独立プラットフォームを通じてより焦点を絞った Agent パーソナリティの作成・共有体験を提供する方法を探求する。

## 背景

AI Agent の開発実践において、私たちはよく一見シンプルだが極めて重要な問題に直面する：**異なる Agent に安定した独自の言語スタイルとパーソナリティを持たせるにはどうすればよいのか？**

この問題は言うのも無念だ。初期の HagiCode の Hero システムでは、異なる英雄（Agent インスタンス）は主に職業設定と汎用プロンプトに依存して表現方法を区別していた。この方式はいくつかの明らかな痛みをもたらし、やってきた友人なら共感できるだろう。

まず、言語スタイルの一貫性を保つのが難しい。同じ「開発エンジニア」役割でも、今日の返信はプロフェッショナルで厳密かもしれないが、明日の出力は気まぐれで散漫になるかもしれない。これはモデル自体の問題ではなく、出力スタイルを制約・誘導する独立したパーソナリティ設定層が欠如しているからだ。

次に、役割感が一般的に弱い。Agent の特徴を説明する際、「親切」「プロフェッショナル」「ユーモア」といった曖昧な形容詞しか使えず、これらの抽象的な記述を支える具体的な言語ルールがない。つまり、言うのは簡単だが、やるのは手詰まりだ。

第三に、パーソナリティ設定の再利用性がほぼゼロだ。精巧に設計した「猫少女ウェイター」の話し方を、別のビジネスシーンで再利用したい場合、ほぼ最初から設定し直す必要がある。美しい事物や人は、必ずしも占領する必要はないが、ただ再利用したいだけだ......しかし本当に難しい。

これらの実際的な問題を解決するため、私たちは **Soul** メカニズムを導入した——装備と説明から独立した**言語スタイル設定層**だ。Soul は Agent の話し癖、口調の好み、言葉の境界を定義でき、複数の英雄間で共有・再利用でき、Session の初回呼び出し時に自動的にシステムプロンプトに注入される。

ある人は「ただのプロンプト設定じゃないか」と思うかもしれない。しかし時には、問題の鍵は「できるかどうか」ではなく「どうやってよりエレガントにやるか」にある。Soul の能力が徐々に成熟するにつれ、私たちはそれが独立した発展の潜在力を持っていることに気づいた。専用の Soul プラットフォームは、ユーザーが Hero システムの他の機能に邪魔されず、様々な面白いパーソナリティ設定の作成・共有・閲覧により集中できるようにする。そこで、**soul.hagicode.com** 独立プラットフォームが誕生した。

## Soul プラットフォームの技術アーキテクチャ進化

Soul プラットフォームの発展は一足飛びではなく、三つの明確な段階を経た。この物語は突然始まり、自然に終わる。

### 第一段階：Hero 組み込み Soul 設定

最初期の Soul 実装は Hero ワークスペースの機能モジュールとして存在していた。Hero インターフェースに独立した SOUL 編集エリアを追加し、プリセット適用とテキスト微調整の二つの方式をサポートした。

プリセット適用は「プロフェッショナル開発エンジニア」「猫少女ウェイター」などの古典的なパーソナリティテンプレートから選択できる。テキスト微調整はプリセットベースで個性的な修正を行える。バックエンドの Hero エンティティはそれに応じて `Soul` フィールドを追加し、`SoulCatalogId` でソースを識別した。

この段階は「あるかないか」の問題を解決し、まだ子供のようによちよちと成長していた。しかし Soul コンテンツが豊になるにつれ、Hero システムと結合したアーキテクチャは限界を示し始めた。

### 第二段階：サイト内 Marketplace

より良い Soul の発見と再利用体験を提供するため、SOUL Marketplace ディレクトリページを構築し、閲覧、検索、詳細表示、お気に入り機能をサポートした。

この段階で、**50 組のメイン Catalog（基本役割）** と **10 組の直交ルール（表現方式）** の組み合わせ設計を導入した。メイン Catalog は Agent の核心役割を定義し、「霧港旅人」「夜航狩人」のような抽象的な役割設定だ。直交ルールは表現方式を定義し、「簡潔干練」「冗長親切」などの言語スタイル特徴だ。

50 × 10 = 500 の組み合わせ可能性が、ユーザーに豊富なパーソナリティ設定空間を提供する。この数量は多いようで少ないようだ、どう言えばよいか、「すべての道はローマに通ず」だが、ただ道の歩きやすさが違うだけだ。バックエンドは `catalog-sources.json` を通じて完全な SOUL ディレクトリを生成し、フロントエンドはこれらのディレクトリ項目を対話型カードリストとして表示する。

サイト内 Marketplace は良い過渡案だが、あくまで過渡に過ぎない。まだメインシステムに依存しており、Soul 機能だけを使いたいユーザーにとって、アクセスパスが深すぎる。結局、簡単なことをするために遠回りしたい人は誰もいないだろう。

### 第三段階：独立プラットフォーム分離

最終的に、Soul 能力を独立リポジトリ（`repos/soul`）に移行することを決定した。元メインシステムの Marketplace は外部ジャンプ誘導に変更し、新プラットフォームは **Builder-first** 設計理念を採用——デフォルトのトップページが作成ワークベンチであり、ユーザーがサイトを開いた瞬間に自分のパーソナリティ設定の作成を開始できる。

この段階の技術スタックも全面的にアップグレードした：Vite 8 + React 19 + TypeScript 5.9 の組み合わせを採用し、shadcn/ui コンポーネントシステムでデザイン言語を統一し、Tailwind CSS 4 のテーマ変数システムを導入した。フロントエンドエンジニアリングレベルの向上は、後続の機能反復のために強固な基盤を築いた。

すべては淡くなった......いや、すべてはまだ始まったばかりだ。

## 核心技術設計と実装

### 素材統合戦略

Soul プラットフォームの核心設計理念の一つは**ローカルファースト**だ。これはトップページがバックエンドなしで完全に実行可能であり、リモート素材が失敗してもページ进入をブロックしてはならないことを意味する。

これは大したことではなく、システム設計時にもう一歩考えただけだ。ローカルスナップショットをベースラインとし、リモートを強化とする。この考え方は製品がどのようなネットワーク条件下でも基本的な可用性を提供できるようにする。具体的な実装では、二層素材アーキテクチャを採用した：

```typescript
export async function loadBuilderMaterials(): Promise<BuilderMaterials> {
  const localMaterials = createLocalMaterials(snapshot)  // ローカルベースライン

  try {
    const inspirationFragments = await fetchMarketplaceItems()  // リモート強化
    return { ...localMaterials, inspirationFragments, remoteState: "ready" }
  } catch (error) {
    return { ...localMaterials, remoteState: "fallback" }  // エレガントなフォールバック
  }
}
```

ローカル素材はメインシステムドキュメントのビルド期スナップショットから来ており、50 組の基本役割と 10 組の表現ルールの完全なデータを含む。リモート素材はユーザーが公開した Soul から来ており、Marketplace API を通じて取得する。両者の結合は、ユーザーに公式テンプレートからコミュニティクリエイティブまでの完全な素材スペクトルを提供する。笑って落ちた涙を隠す......いや、何でもない、ただローカルにリモートを加えただけだ。

### Soul 碎片データモデル

Soul の核心データ抽象は **SoulFragment**（魂の欠片）だ：

```typescript
export type SoulFragment = {
  fragmentId: string
  group: "main-catalog" | "expression-rule" | "published-soul"
  title: string
  summary: string
  content: string
  keywords: string[]
  localized?: Partial<Record<AppLocale, LocalizedFragmentContent>>
  sourceRef: SoulFragmentSourceRef
  meta: SoulFragmentMeta
}
```

`group` フィールドは欠片のタイプを区別する：メインディレクトリは役割の核心を定義し、直交ルールは表現方式を定義し、ユーザーが公開した Soul は `published-soul` とマークされる。`localized` フィールドは多言語をサポートし、同じ欠片が異なる言語環境で異なるタイトルと説明を表示できるようにする。国際化設計は早めに、この言葉も私たちは使った。

Builder ドラフト状態はユーザーの現在の編集状態をカプセル化する：

```typescript
export type SoulBuilderDraft = {
  draftId: string
  name: string
  selectedMainFragmentId: string | null
  selectedRuleFragmentId: string | null
  inspirationSoulId: string | null
  mainSlotText: string
  ruleSlotText: string
  customPrompt: string
  previewText: string
  updatedAt: string
}
```

ユーザーがエディタで選択した各欠片のコンテンツは、対応する slot（スロット）に結合され、最終的なプレビューテキストを形成する。`mainSlotText` はメイン役割コンテンツに対応し、`ruleSlotText` は表現ルールコンテンツに対応し、`customPrompt` はユーザーの追加補足指令だ。

### プレビュー compilation メカニズム

プレビュー compilation は Soul Builder の核心機能で、ユーザーが選択した欠片とカスタムテキストをコピー可能なシステムプロンプトに組み立てる：

```typescript
export function compilePreview(
  draft: Pick<SoulBuilderDraft, "mainSlotText" | "ruleSlotText" | "customPrompt">,
  fragments: {
    mainFragment: SoulFragment | null
    ruleFragment: SoulFragment | null
    inspirationFragment: SoulFragment | null
  }
): PreviewCompilation {
  // 組み立てロジック：メイン役割 + 表現ルール + 霊感参考 + カスタムコンテンツ
}
```

compilation 結果は中央プレビューパネルに表示され、ユーザーは最終効果をリアルタイムで見ることができ、ワンクリックでクリップボードにコピーできる。この機能は言うのも簡単だろう。しかしシンプルなものほど最も実用的だ。

### フロントエンド状態管理

Soul Builder のフロントエンド状態管理は一つの重要原則に従う：**状態境界を明確に区分する**。具体的には、ドロワー状態は永続化せず、ドラフトに直接書き込まない。明確な Builder 操作のみが状態変更をトリガーする。

```typescript
// 領域状態
export function useSoulBuilder() {
  // 素材ロードとキャッシュ
  // スロット集約とプレビュー compilation
  // コピー動作とフィードバックメッセージ
  // Locale セーフな記述子
}

// 表示状態
export function useHomeEditorState() {
  // activeSlot, drawerSide, drawerOpen
  // デフォルトフォーカス動作
}
```

この分離は編集状態の安全性とインターフェースの応答速度を確保する。ドロワーの開閉は純粋な UI 交互であり、複雑な永続化ロジックをトリガーする必要はない。これは無駄話だ！いや、実際とても重要——UI 状態とビジネス状態を明確に区分し、UI 交互が核心データモデルを汚染するのを避ける。

### 単一ドローライフサイクル

Soul Builder は単一ドロワーモデルを採用：同時に一つのスロットドロワーのみを開くことを許可する。マスクレイヤーのクリック、ESC キーの押下、またはスロットの切り替えはすべて現在のドロワーを自動的に閉じる。この設計は状態管理を簡素化し、モバイルドロワー交互の一般的なモデルにも合致する。

ドロワーを閉じても現在の編集コンテンツはクリアされず、ユーザーが切り替えて戻ってくると、コンテキストが保持される。この「軽量」なドロワー設計は、ユーザー操作の中断感を避ける。結局、せっかく書いたものが、誤ってクリックしただけで全部消えるのを望む人は誰もいないだろう。

### 二言語サポートアーキテクチャ

国際化は Soul プラットフォームの重要な特徴だ。システム文案は完全に二言語切り替えをサポートし、ユーザードラフトテキストは言語切り替えによって決して書き換えられない——ドラフトテキスト自体がユーザーが自由に入力したコンテンツであり、システム翻訳を含まないからだ。

公式霊感カードは上流表示名を維持するが、ベストエフォートの英語要約を提供する。中国語名の Soul については、定義済みのマッピングルールを通じて英語版を生成する：

```typescript
// メイン役割英語名マッピング
const mainNameEnglishMap = {
  "雾港旅人": "Mistport Traveler",
  "夜航猎手": "Night Hunter",
  // ...
}

// 直交ルール英語名マッピング
const ruleNameEnglishMap = {
  "简洁干练": "Concise & Professional",
  "啰嗦亲切": "Verbose & Friendly",
  // ...
}
```

このマッピングテーブルはシンプルに見えるが、維持するには多くの労力が必要だ。結局 50 組のメイン役割と 10 組の直交ルールがあり、掛けると 500 の組み合わせ、この数量は大きくも小さくもない。

## バックエンドディレクトリ生成

Soul Catalog の一括生成はバックエンドで完了し、C# を使用して 50 × 10 = 500 の組み合わせの自動作成を実現した：

```csharp
foreach (var main in source.MainCatalogs)
{
    foreach (var orthogonal in source.OrthogonalCatalogs)
    {
        var catalogId = $"soul-{main.Index:00}-{orthogonal.Index:00}";
        var displayName = BuildNickname(main, orthogonal);
        var soulSnapshot = BuildSoulSnapshot(main, orthogonal);
        // データベースに書き込み...
    }
}
```

ニックネーム生成アルゴリズムはメイン役割名と表現ルール名を組み合わせ、想像力に富んだ Agent 代号を創造する：

```csharp
private static readonly string[] MainHandleRoots = [
    "雾港", "夜航", "零帧", "星渊", "霓虹", "断云", ...
];
private static readonly string[] OrthogonalHandleSuffixes = [
    "旅人", "猎手", "术师", "行者", "星使", ...
];
// 組み合わせ例：霧港旅人、夜航狩人、零帧術師...
```

Soul スナップショットの組み立ては固定されたテンプレートフォーマットに従い、メイン役割核心、標識特徴、表現ルール核心、出力制約を組み合わせる：

```csharp
private static string BuildSoulSnapshot(main, orthogonal) => string.Join('\n', [
    $"你的人设内核来自「{main.Name}」：{main.Core}",
    $"保持以下标志性语言特征：{main.Signature}",
    $"你的表达规则来自「{orthogonal.Name}」：{orthogonal.Core}",
    $"必须遵循这些输出约束：{orthogonal.Signature}"
]);
```

このテンプレート組み立ては退屈極まりない仕事だが、これらの退屈な仕事がなければ、面白い製品は来ない。

## プラットフォーム移行戦略

Soul がメインシステムから分離して独立プラットフォームになった後、私たちは重要な課題に直面した——既存のユーザーデータをどう処理するか。この問題はよくある——分離は簡単だが、移行は難しい。私たちは三つの保障措置を講じた：

**後方互換保障**。保存された Hero SOUL スナップショットは可視を維持し、履歴スナップショットは Marketplace ソース ID を失ってもプレビュー可能だ。これはユーザーの以前のすべての設定が失われないことを意味し、ただ表示位置が変化しただけだ。結局、せっかくの設定が、消えたら消えてだと言う人は誰もいないだろう。

**メインシステムインターフェース廃止**。サイト内 Marketplace API は 410 Gone ステータスコードを返し、移行プロンプトを添付し、ユーザーを soul.hagicode.com へ誘導する。

**Hero SOUL フォーム改造**。Hero Soul 編集エリアに移行プロンプトブロックを新規追加し、Soul プラットフォームが独立したことを明確にユーザーに伝え、ワンクリックジャンプボタンを提供する：

```typescript
// HeroSoulForm.tsx
<div className="rounded-2xl border border-orange-200/70 bg-orange-50/80 p-4">
  <div>{t('hero.soul.migrationTitle')}</div>
  <p>{t('hero.soul.migrationDescription')}</p>
  <Button onClick={onOpenSoulPlatform}>
    {t('hero.soul.openSoulPlatformAction')}
  </Button>
</div>
```

## 実践要点まとめ

Soul プラットフォームの全体開発プロセスを振り返ると、共有する価値のある実践経験がいくつかある。これも経験者の心得で、大した道理ではないが、ただ踏んだ坑だ。

**ローカルファーストの実行時仮定**。リモートデータに依存する機能を設計する際、常にネットワークが利用不可能であると仮定する。ローカルスナップショットをベースラインとし、リモートを強化とする。この考え方は製品がどのようなネットワーク条件下でも基本的な可用性を提供できるようにする。結局このご時世、ネットワークなんて、切れたら切れたで、誰にもわからない。

**状態境界の明確な区分**。UI 状態とビジネス状態を明確に区分し、UI 交互が核心データモデルを汚染するのを避ける。ドロワーの開閉は純粋な UI 状態であり、ドラフト永続化と混ぜる必要はない。

**国際化設計は早めに**。製品に国際化ニーズがある場合、データモデル設計段階で考慮するのが最善だ。`localized` フィールドはデータ構造の複雑度を増すが、後続の多言語コンテンツの維持コストは大幅に低下する。

**素材同期ワークフローの自動化**。Soul プラットフォームのローカル素材はメインシステムドキュメントから来ており、上流ドキュメントが更新された際、フロントエンドスナップショットに同期する仕組みが必要だ。私たちは `npm run materials:sync` スクリプトを設計してこのプロセスを自動化し、素材が常に上流と一致するようにした。

## 今後の展望

現在のアーキテクチャ設計に基づき、Soul プラットフォームは今後以下の発展方向を検討できる。これも浅はかな考えで、必ずしも正しくないが、参考までに。

**コミュニティ共有エコシステム**。ユーザーによるカスタム Soul のアップロードと共有をサポートし、評価、コメント、推薦メカニズムを追加し、優秀な Soul 設定がより多くの人に発見・使用されるようにする。結局、独りで楽しむよりみんなで楽しむ方が良い。

**マルチモーダル拡張**。文字スタイルだけでなく、音声スタイル設定、絵文字使用好み、コードスタイルとフォーマットルールなどもサポート可能。これは言うのは美しいが、やるのは......

**インテリジェントアシスタント**。使用シーンに基づいて自動的に Soul を推薦し、スタイル移行と融合を行い、異なる Soul の実際の効果を A/B テストさえする。美しさは天気の良し悪しを気にする必要があるか？ 試せばわかる。

**クロスプラットフォーム同期**。他の AI プラットフォームからパーソナリティ設定をインポートし、標準化された Soul エクスポートフォーマットを提供し、主流 Agent フレームワークと統合する。

## まとめ

本文では HagiCode Soul プラットフォームのニーズの萌芽から独立プラットフォームへの完全な進化プロセスを共有した。なぜ Soul メカニズムが必要かを探求し（Agent パーソナリティ一貫性問題の解決）、技術アーキテクチャの三つの発展段階を分析し（組み込み設定、サイト内 Marketplace、独立プラットフォーム）、核心のデータモデル、状態管理、プレビュー compilation、国際化設計を深く讲解し、プラットフォーム移行の実践経験を共有した。

Soul の本質は、ビジネスロジックから独立した**パーソナリティ設定層**だ。それは AI Agent の言語スタイルを定義可能、再利用可能、共有可能にする。技術的観点から、この設計は複雑ではないが、解決する問題は真实的で、広範なニーズがある。

もし AI Agent 製品を開発しているなら、パーソナリティ設定ソリューションが十分に柔軟か考えてみてはどうだろう。Soul プラットフォームの実践が何らかのインスピレーションを提供するかもしれない。

この情は追憶となるのを待つが、当時は既に惘然としていた。ある日、同様の問題に遭遇し、その時にこの記事が少し役に立てば、それで十分だ。
