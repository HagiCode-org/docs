## ゼロから構築する GitHub Issues 連携：HagiCode のフロントエンド直結実践

> 本記事では、HagiCode プラットフォームに GitHub Issues を連携する全プロセスを記録します。「フロントエンド直結 + バックエンド最小化」のアーキテクチャを採用し、バックエンドを軽量に保ちながら、安全な OAuth 認証と効率的な Issues 同期を実現する方法を探ります。

## 背景：なぜ GitHub を連携するのか

AI サポート開発プラットフォームとして、HagiCode のコアバリューはアイデアと実装を繋ぐことにあります。しかし実際の利用において、ユーザーが HagiCode で Proposal（提案）を完了した後、手動で内容を GitHub Issues にコピーしてプロジェクト管理を行う必要があることが分かりました。

これにはいくつかの明らかなペインポイントがあります：

1.  **ワークフローの分断**：ユーザーは2つのシステム間を行き来する必要があり、体験がスムーズでないだけでなく、コピー＆ペーストの過程で重要な情報が失われる可能性があります。
2.  **コラボレーションの不便さ**：チームの他のメンバーは GitHub でタスクを確認する習慣があり、HagiCode 内の提案の進捗を直接見ることができません。
3.  **重複作業**：提案が更新されるたびに、GitHub の対応する Issue を手動で更新する必要があり、不必要なメンテナンスコストが増加します。

この問題を解決するため、**GitHub Issues Integration** 機能を導入し、HagiCode セッションと GitHub リポジトリの接続を実現して「ワンクリック同期」を実現することにしました。

## 技術選定：フロントエンド直結 vs バックエンドプロキシ

連携方式を設計する際、目の前に2つの道がありました：従来の「バックエンドプロキシモデル」と、より革新的な「フロントエンド直結モデル」です。

### 方式の比較

従来の**バックエンドプロキシモデル**では、フロントエンドからのすべてのリクエストがまずバックエンドを経由し、バックエンドが GitHub API を呼び出します。ロジックは集中していますが、バックエンドに大きな負担をかけます：

1.  **バックエンドの肥大化**：専用の GitHub API クライアントラッパーを書き、複雑な OAuth ステートマシンを処理する必要があります。
2.  **Token リスク**：ユーザーの GitHub Token をバックエンドデータベースに保存する必要があります。暗号化は可能ですが、セキュリティリスクの面は増大します。
3.  **開発コスト**：Token を保存するためのデータベースマイグレーションと、追加の同期サービスのメンテナンスが必要です。

一方、**フロントエンド直結モデル**ははるかに軽量です。この方式では、最も機密性の高い「秘密鍵交換」ステップ（OAuth コールバック）にのみバックエンドを使用します。Token を取得した後、ブラウザの localStorage に直接保存します。その後の Issue 作成、コメント更新などの操作は、フロントエンドから直接 HTTP リクエストを GitHub に送信します。

| 比較項目 | バックエンドプロキシモデル | フロントエンド直結モデル |
| :--- | :--- | :--- |
| **バックエンドの複雑さ** | 完全な OAuth サービスと GitHub API クライアントが必要 | OAuth コールバックエンドポイントのみ |
| **Token 管理** | データベースに暗号化保存、漏洩リスクあり | ブラウザに保存、ユーザー自身のみ閲覧可能 |
| **実装コスト** | データベースマイグレーション、複数サービスの開発が必要 | 主にフロントエンドの作業 |
| **ユーザー体験** | ロジックは統一されているが、サーバーレイテンシがやや高い | GitHub と直接通信、極めて高速なレスポンス |

高速な連携とバックエンドの最小変更を実現するため、**最終的に「フロントエンド直結モデル」を採用しました**。これはブラウザに「一時通行証」を発行するようなもので、証を取得すれば、ブラウザは自分で GitHub に用事を済ませに行け、毎回バックエンド管理者の承認を得る必要がありません。

## コア設計：データフローとセキュリティ

アーキテクチャを決定した後、具体的なデータフローを設計する必要があります。同期プロセスのコアは、Token を安全に取得し、効率的に利用する方法です。

### 全体アーキテクチャ図

システム全体は3つの役割に抽象化できます：ブラウザ（フロントエンド）、HagiCode バックエンド、GitHub。

```text
+--------------+        +--------------+        +--------------+
| フロントエンド|        |   バックエンド|        |    GitHub    |
|  React       |        |   ASP.NET    |        |   REST API   |
|              |        |              |        |              |
|  +--------+  |        |              |        |              |
|  | OAuth  |--+--------> /callback    |        |              |
|  | フロー  |  |        |              |        |              |
|  +--------+  |        |              |        |              |
|              |        |              |        |              |
|  +--------+  |        |  +--------+  |        |  +--------+  |
|  |GitHub  |  +------------>Session |  +----------> Issues |  |
|  |API     |  |        |  |Metadata|  |        |  |        |  |
|  |直結    |  |        |  +--------+  |        |  +--------+  |
|  +--------+  |        |              |        |              |
+--------------+        +--------------+        +--------------+
```

**重要なポイント**：OAuth のほんの一部（code を token に交換する）だけがバックエンドを経由する必要があり、その後の重作業（Issue の作成）はすべてフロントエンドが GitHub と直接やり取りします。

### 同期データフローの詳細

ユーザーが HagiCode インターフェースの「Sync to GitHub」ボタンをクリックすると、一連の複雑なアクションが発生します：

```text
ユーザーが「Sync to GitHub」をクリック
         │
         ▼
1. フロントエンドが localStorage から GitHub Token を取得
         │
         ▼
2. Issue の内容をフォーマット（Proposal を Markdown に変換）
         │
         ▼
3. フロントエンドが直接 GitHub API を呼び出して Issue を作成/更新
         │
         ▼
4. HagiCode バックエンド API を呼び出して Session.metadata を更新（Issue URL などの情報を保存）
         │
         ▼
5. バックエンドが SignalR 経由で SessionUpdated イベントをブロードキャスト
         │
         ▼
6. フロントエンドがイベントを受信し、UI を「同期済み」状態に更新
```

### セキュリティ設計

セキュリティはサードパーティサービス連携において常に最優先事項です。以下の考慮事項を設けました：

1.  **CSRF 攻撃の防止**：OAuth リダイレクト時にランダムな `state` パラメータを生成し、sessionStorage に保存します。コールバック時に state を厳密に検証し、リクエストの偽造を防止します。
2.  **Token ストレージの分離**：Token はブラウザの `localStorage` のみに保存されます。同一オリジンポリシー（Same-Origin Policy）を利用し、HagiCode のスクリプトのみが読み取れるようにし、サーバーサイドデータベースの漏洩がユーザーに波及するのを防ぎます。
3.  **エラーバウンダリ**：GitHub API の一般的なエラー（401 Token 期限切れ、422 バリデーション失敗、429 レート制限など）に対して専用のエラー処理ロジックを設計し、ユーザーにフレンドリーなメッセージを表示します。

### 2. フロントエンド OAuth フロー

これが連携の入り口です。標準の Authorization Code Flow を使用します。

```typescript
// src/HagiCode.Client/src/services/githubOAuth.ts

// 認可 URL を生成してリダイレクト
export async function generateAuthUrl(): Promise<string> {
  const state = generateRandomString(); // CSRF 防止用のランダム文字列を生成
  sessionStorage.setItem('hagicode_github_state', state);

  const params = new URLSearchParams({
    client_id: clientId,
    redirect_uri: window.location.origin + '/settings?tab=github&oauth=callback',
    scope: ['repo', 'public_repo'].join(' '),
    state: state,
  });

  return `https://github.com/login/oauth/authorize?${params.toString()}`;
}

// コールバックページで Code を Token に交換
export async function exchangeCodeForToken(code: string, state: string): Promise<GitHubToken> {
  // 1. State を検証して CSRF を防止
  const savedState = sessionStorage.getItem('hagicode_github_state');
  if (state !== savedState) throw new Error('Invalid state parameter');

  // 2. バックエンド API を呼び出して Token を交換
  // 注意：ClientSecret をフロントエンドに公開できないため、必ずバックエンドを経由する必要がある
  const response = await fetch('/api/GitHubOAuth/callback', {
    method: 'POST',
    headers: { 'Content-Type': 'application/json' },
    body: JSON.stringify({ code, state, redirectUri: window.location.origin + '/settings?tab=github&oauth=callback' }),
  });

  if (!response.ok) throw new Error('Failed to exchange token');

  const token = await response.json();

  // 3. LocalStorage に保存
  saveToken(token);
  return token;
}
```

### 3. GitHub API クライアントラッパー

Token を取得した後、GitHub API を呼び出すための強力なツールが必要です。

```typescript
// src/HagiCode.Client/src/services/githubApiClient.ts

const GITHUB_API_BASE = 'https://api.github.com';

// コアリクエストラッパー
async function githubApi<T>(endpoint: string, options: RequestInit = {}): Promise<T> {
  const token = localStorage.getItem('gh_token');
  if (!token) throw new Error('Not connected to GitHub');

  const response = await fetch(`${GITHUB_API_BASE}${endpoint}`, {
    ...options,
    headers: {
      ...options.headers,
      Authorization: `Bearer ${token}`,
      Accept: 'application/vnd.github.v3+json', // API バージョンを指定
    },
  });

  // エラー処理ロジック
  if (!response.ok) {
    if (response.status === 401) throw new Error('GitHub Token が期限切れです。再接続してください');
    if (response.status === 403) throw new Error('このリポジトリへのアクセス権限がないか、レート制限を超えています');
    if (response.status === 422) throw new Error('Issue の検証に失敗しました。タイトルが重複している可能性があります');
    throw new Error(`GitHub API Error: ${response.statusText}`);
  }

  return response.json();
}

// Issue を作成
export async function createIssue(owner: string, repo: string, data: { title: string, body: string, labels: string[] }) {
  return githubApi(`/repos/${owner}/${repo}/issues`, {
    method: 'POST',
    body: JSON.stringify(data),
  });
}
```

### 4. コンテンツのフォーマットと同期

最後のステップは、HagiCode の Session データを GitHub Issue の形式に変換することです。これは翻訳作業に似ています。

```typescript
// Session オブジェクトを Markdown 文字列に変換
function formatIssueForSession(session: Session): string {
  let content = `# ${session.title}\n\n`;
  content += `**> HagiCode Session:** #${session.code}\n`;
  content += `**> Status:** ${session.status}\n\n`;
  content += `## Description\n\n${session.description || 'No description provided.'}\n\n`;

  // Proposal タイプの場合、追加フィールドを表示
  if (session.type === 'proposal') {
    content += `## Chief Complaint\n\n${session.chiefComplaint || ''}\n\n`;
    // GitHub から HagiCode に戻るためのディープリンクを追加
    content += `---\n\n**[View in HagiCode](hagicode://sessions/${session.id})**\n`;
  }

  return content;
}

// 同期ボタンクリック時のメインロジック
const handleSync = async (session: Session) => {
  try {
    const repoInfo = parseRepositoryFromUrl(session.repoUrl); // リポジトリ URL を解析
    if (!repoInfo) throw new Error('Invalid repository URL');

    toast.loading('GitHub へ同期中...');

    // 1. コンテンツをフォーマット
    const issueBody = formatIssueForSession(session);

    // 2. API を呼び出し
    const issue = await githubApiClient.createIssue(repoInfo.owner, repoInfo.repo, {
      title: `[HagiCode] ${session.title}`,
      body: issueBody,
      labels: ['hagicode', 'proposal', `status:${session.status}`],
    });

    // 3. Session Metadata を更新（Issue リンクを保存）
    await SessionsService.patchApiSessionsSessionId(session.id, {
      metadata: {
        githubIssue: {
          owner: repoInfo.owner,
          repo: repoInfo.repo,
          issueNumber: issue.number,
          issueUrl: issue.html_url,
          syncedAt: new Date().toISOString(),
        }
      }
    });

    toast.success('同期完了！');
  } catch (err) {
    console.error(err);
    toast.error('同期に失敗しました。Token またはネットワークを確認してください');
  }
};
```

## まとめと展望

この「フロントエンド直結」方式により、最小限のバックエンドコードで GitHub Issues のシームレスな連携を実現しました。

### 得られた成果
1.  **高い開発効率**：バックエンドの変更は最小限で、主にデータベースに1つのフィールド追加と、シンプルな OAuth コールバックエンドポイントのみ。大部分のロジックはフロントエンドで完了します。
2.  **高いセキュリティ**：Token がサーバーデータベースを経由しないため、漏洩リスクが低減されます。
3.  **優れたユーザー体験**：フロントエンドから直接リクエストを送信するため、レスポンスが速く、バックエンドを中継する必要がありません。

### 注意事項
実際のデプロイ時には、いくつかの落とし穴に注意が必要です：
- **OAuth App の設定**：GitHub OAuth App 設定で正しい `Authorization callback URL`（通常は `http://localhost:3000/settings?tab=github&oauth=callback`）を入力してください。
- **レート制限**：GitHub API は未認証リクエストの制限が厳しいですが、Token を使用すれば通常は十分です（5000回/時間）。
- **URL の解析**：ユーザーが入力するリポジトリ URL は多種多様です。正規表現で `.git` サフィックス、SSH 形式などにも対応できるようにしてください。

### 今後の拡張
現在の機能はまだ一方向同期（HagiCode → GitHub）です。将来的には GitHub Webhooks を通じた双方向同期を計画しています。例えば、GitHub で Issue を閉じると、HagiCode 側のセッションステータスも自動更新されるようにします。これにはバックエンドで Webhook 受信エンドポイントを公開する必要があり、これも次のステップとして取り組む予定です。
