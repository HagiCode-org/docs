## ゼロから構築する GitHub Issues 連携：HagiCode のフロントエンド直結実践

> この記事では、HagiCode プラットフォームにおける GitHub Issues 連携の全プロセスを記録します。「フロントエンド直結 + バックエンド最小化」のアーキテクチャを通じて、バックエンドの軽量化を保ちながら、安全な OAuth 認証と効率的な Issues 同期を実現する方法を探求します。

## 背景：なぜ GitHub を連携するのか？

HagiCode は AI 補助開発プラットフォームとして、アイデアと実装を繋ぐことを核心的価値としています。しかし実際の使用において、ユーザーが HagiCode で Proposal（提案）を完了した後、手動でコンテンツを GitHub Issues にコピーしてプロジェクト追跡を行う必要があることがわかりました。

これはいくつかの明らかな課題をもたらします：

1.  **ワークフローの分断**：ユーザーは2つのシステム間を行き来する必要があり、体験がスムーズでないだけでなく、コピー＆ペーストの過程で重要な情報が失われやすくなります。
2.  **協業の不便さ**：チームの他のメンバーは GitHub でタスクを確認することに慣れており、HagiCode 内の提案の進捗を直接確認できません。
3.  **重複労働**：提案が更新されるたびに、手動で GitHub の対応する Issue を更新する必要があり、不必要なメンテナンスコストが増加します。

この問題を解決するため、**GitHub Issues Integration** 機能を導入することにし、HagiCode セッションと GitHub リポジトリの接続を打通して、「ワンクリック同期」を実現することにしました。

## 技術選定：フロントエンド直結 vs バックエンドプロキシ

連携ソリューションを設計する際、私たちの前に2つの道がありました：従来の「バックエンドプロキシモード」と、より野心的な「フロントエンド直結モード」です。

### ソリューション比較

従来の**バックエンドプロキシモード**では、フロントエンドのすべてのリクエストがまず私たちのバックエンドを経由し、そこからバックエンドが GitHub API を呼び出します。これによりロジックは集中しますが、バックエンドに小さくない負担をもたらします：

1.  **バックエンドの肥大化**：専用の GitHub API クライアントカプセル化を作成する必要があり、OAuth の複雑なステートマシンも処理する必要があります。
2.  **Token のリスク**：ユーザーの GitHub Token をバックエンドデータベースに保存する必要があり、暗号化はできますが、セキュリティリスクの範囲は拡大します。
3.  **開発コスト**：Token を保存するためのデータベースマイグレーションが必要であり、追加の同期サービス一式を維持する必要があります。

一方、**フロントエンド直結モード**ははるかに軽量です。このソリューションでは、最も機密性の高い「鍵交換」フェーズ（OAuth callback）を処理するためにのみバックエンドを利用し、Token を取得した後、直接ブラウザの localStorage に保存します。その後の Issue 作成、コメント更新などの操作は、フロントエンドから直接 GitHub に HTTP リクエストを送信します。

| 比較次元 | バックエンドプロキシモード | フロントエンド直結モード |
| :--- | :--- | :--- |
| **バックエンド複雑度** | 完全な OAuth サービスと GitHub API クライアントが必要 | OAuth コールバックエンドポイントのみ |
| **Token 管理** | データベースに暗号化保存、漏洩リスクあり | ブラウザに保存、ユーザー自身のみ可視 |
| **実施コスト** | データベースマイグレーション、マルチサービス開発が必要 | 主にフロントエンド作業量 |
| **ユーザー体験** | ロジックは統一されているが、サーバーレイテンシがやや高い可能性 | 応答が極めて高速、GitHub と直接対話 |

私たちが求めているのは迅速な統合と最小限のバックエンド変更であることを考慮し、**最終的に「フロントエンド直結モード」を採用しました**。これはブラウザに「一時通行証」を発行するようなもので、証を取得した後、ブラウザは自分で GitHub の用事を済ませることができ、毎回バックエンド管理者の承認を求める必要がありません。

## コア設計：データフローとセキュリティ

アーキテクチャを確定した後、具体的なデータフローを設計する必要があります。同期プロセス全体の核心は、いかに安全に Token を取得し、効率的に活用するかにあります。

### 全体アーキテクチャ図

システム全体は3つの役割に抽象化できます：ブラウザ（フロントエンド）、HagiCode バックエンド、GitHub。

```text
+--------------+        +--------------+        +--------------+
|  フロントエンド  |        |    バックエンド      |        |    GitHub    |
|    React      |        |   ASP.NET    |        |    REST API  |
|  +--------+  |        |              |        |              |
|  |  OAuth |--+--------> /callback    |        |              |
|  |  フロー  |  |        |              |        |              |
|  +--------+  |        |              |        |              |
|              |        |              |        |              |
|  +--------+  |        |  +--------+  |        |  +--------+  |
|  |GitHub  |  +------------>Session |  +----------> Issues |  |
|  |API     |  |        |  |Metadata|  |        |  |        |  |
|  |直結    |  |        |  +--------+  |        |  +--------+  |
|  +--------+  |        |              |        |              |
+--------------+        +--------------+        +--------------+
```

**重要な点は**：OAuth の小さな一歩（code を token に交換）のみがバックエンドを経由する必要があり、その後の重労働（Issue 作成）はすべてフロントエンドが直接 GitHub と対話します。

### 同期データフローの詳細解説

ユーザーが HagiCode インターフェース上の「Sync to GitHub」ボタンをクリックすると、一連の複雑なアクションが発生します：

```text
ユーザーが「Sync to GitHub」をクリック
         │
         ▼
1. フロントエンドが localStorage をチェックして GitHub Token を取得
         │
         ▼
2. Issue コンテンツをフォーマット（Proposal を Markdown に変換）
         │
         ▼
3. フロントエンドが直接 GitHub API を呼び出して Issue を作成/更新
         │
         ▼
4. HagiCode バックエンド API を呼び出して Session.metadata を更新（Issue URL などの情報を保存）
         │
         ▼
5. バックエンドが SignalR を通じて SessionUpdated イベントをブロードキャスト
         │
         ▼
6. フロントエンドがイベントを受信し、UI を更新して「同期済み」状態を表示
```

### セキュリティ設計

セキュリティ問題は常にサードパーティサービス統合の最重要事項です。以下の考慮事項を行いました：

1.  **CSRF 攻撃対策**：OAuth リダイレクト時に、ランダムな `state` パラメータを生成して sessionStorage に保存します。コールバック時に state を厳密に検証し、リクエストの偽造を防止します。
2.  **Token 保存の分離**：Token はブラウザの `localStorage` にのみ保存され、同一生成元ポリシー（Same-Origin Policy）を利用して、HagiCode のスクリプトのみが読み取れるようにし、サーバーサイドデータベース漏洩の影響がユーザーに及ぶのを回避します。
3.  **エラー境界**：GitHub API の一般的なエラー（401 Token 有効期限切れ、422 検証失敗、429 レート制限）に対し、専用のエラー処理ロジックを設計し、ユーザーにフレンドリーなプロンプトを提供します。

### 2. フロントエンド OAuth フロー

これが接続の入口です。標準の Authorization Code Flow を使用します。

```typescript
// src/HagiCode.Client/src/services/githubOAuth.ts

// 認証 URL を生成してリダイレクト
export async function generateAuthUrl(): Promise<string> {
  const state = generateRandomString(); // CSRF 対策のランダム文字列を生成
  sessionStorage.setItem('hagicode_github_state', state);
  
  const params = new URLSearchParams({
    client_id: clientId,
    redirect_uri: window.location.origin + '/settings?tab=github&oauth=callback',
    scope: ['repo', 'public_repo'].join(' '),
    state: state,
  });
  
  return `https://github.com/login/oauth/authorize?${params.toString()}`;
}

// コールバックページで Code と Token を交換
export async function exchangeCodeForToken(code: string, state: string): Promise<GitHubToken> {
  // 1. State 検証で CSRF を防止
  const savedState = sessionStorage.getItem('hagicode_github_state');
  if (state !== savedState) throw new Error('Invalid state parameter');

  // 2. バックエンド API を呼び出して Token 交換
  // 注意：ここはバックエンドを経由する必要があります。ClientSecret が必要であり、フロントエンドに露出してはいけないため
  const response = await fetch('/api/GitHubOAuth/callback', {
    method: 'POST',
    headers: { 'Content-Type': 'application/json' },
    body: JSON.stringify({ code, state, redirectUri: window.location.origin + '/settings?tab=github&oauth=callback' }),
  });

  if (!response.ok) throw new Error('Failed to exchange token');
  
  const token = await response.json();
  
  // 3. LocalStorage に保存
  saveToken(token);
  return token;
}
```

### 3. GitHub API クライアントカプセル化

Token を取得した後、GitHub API を呼び出す強力なツールが必要です。

```typescript
// src/HagiCode.Client/src/services/githubApiClient.ts

const GITHUB_API_BASE = 'https://api.github.com';

// コアリクエストカプセル化
async function githubApi<T>(endpoint: string, options: RequestInit = {}): Promise<T> {
  const token = localStorage.getItem('gh_token');
  if (!token) throw new Error('Not connected to GitHub');
  
  const response = await fetch(`${GITHUB_API_BASE}${endpoint}`, {
    ...options,
    headers: {
      ...options.headers,
      Authorization: `Bearer ${token}`,
      Accept: 'application/vnd.github.v3+json', // API バージョンを指定
    },
  });
  
  // エラー処理ロジック
  if (!response.ok) {
    if (response.status === 401) throw new Error('GitHub Token が失効しました。再接続してください');
    if (response.status === 403) throw new Error('リポジトリへのアクセス権限がないか、レート制限を超えています');
    if (response.status === 422) throw new Error('Issue の検証に失敗しました。タイトルが重複している可能性があります');
    throw new Error(`GitHub API Error: ${response.statusText}`);
  }
  
  return response.json();
}

// Issue を作成
export async function createIssue(owner: string, repo: string, data: { title: string, body: string, labels: string[] }) {
  return githubApi(`/repos/${owner}/${repo}/issues`, {
    method: 'POST',
    body: JSON.stringify(data),
  });
}
```

### 4. コンテンツフォーマットと同期

最後のステップは、HagiCode の Session データを GitHub Issue のフォーマットに変換することです。これは少し「翻訳」作業のようなものです。

```typescript
// Session オブジェクトを Markdown 文字列に変換
function formatIssueForSession(session: Session): string {
  let content = `# ${session.title}\n\n`;
  content += `**> HagiCode Session:** #${session.code}\n`;
  content += `**> Status:** ${session.status}\n\n`;
  content += `## Description\n\n${session.description || 'No description provided.'}\n\n`;
  
  // Proposal タイプの場合、追加フィールドを含める
  if (session.type === 'proposal') {
    content += `## Chief Complaint\n\n${session.chiefComplaint || ''}\n\n`;
    // ディープリンクを追加し、GitHub から HagiCode に戻るのを容易にします
    content += `---\n\n**[View in HagiCode](hagicode://sessions/${session.id})**\n`;
  }
  
  return content;
}

// 同期ボタンクリック時のメインロジック
const handleSync = async (session: Session) => {
  try {
    const repoInfo = parseRepositoryFromUrl(session.repoUrl); // リポジトリ URL を解析
    if (!repoInfo) throw new Error('Invalid repository URL');

    toast.loading('GitHub に同期中...');
    
    // 1. コンテンツをフォーマット
    const issueBody = formatIssueForSession(session);
    
    // 2. API を呼び出し
    const issue = await githubApiClient.createIssue(repoInfo.owner, repoInfo.repo, {
      title: `[HagiCode] ${session.title}`,
      body: issueBody,
      labels: ['hagicode', 'proposal', `status:${session.status}`],
    });
    
    // 3. Session Metadata を更新（Issue リンクを保存）
    await SessionsService.patchApiSessionsSessionId(session.id, {
      metadata: {
        githubIssue: {
          owner: repoInfo.owner,
          repo: repoInfo.repo,
          issueNumber: issue.number,
          issueUrl: issue.html_url,
          syncedAt: new Date().toISOString(),
        }
      }
    });

    toast.success('同期に成功しました！');
  } catch (err) {
    console.error(err);
    toast.error('同期に失敗しました。Token またはネットワークを確認してください');
  }
};
```

## まとめと展望

この「フロントエンド直結」ソリューションを通じて、最小限のバックエンドコードで GitHub Issues のシームレスな統合を実現しました。

### 收获
1.  **開発効率が高い**：バックエンドの変更は極めて小さく、主にデータベースに1つのフィールドを追加し、単純な OAuth コールバックインターフェースを追加するだけで、大部分のロジックはフロントエンドで完了します。
2.  **セキュリティが良い**：Token はサーバーデータベースを経由せず、漏洩リスクを低減します。
3.  **ユーザー体験が良い**：フロントエンドから直接リクエストを開始し、応答速度が速く、バックエンド中継を経由する必要がありません。

### 注意事項
実際のデプロイ時には、いくつかの落とし穴に注意してください：
- **OAuth App 設定**：GitHub OAuth App 設定で正しい `Authorization callback URL`（通常は `http://localhost:3000/settings?tab=github&oauth=callback`）を入力することを忘れないでください。
- **レート制限**：GitHub API は未認証リクエストに対して厳しい制限がありますが、Token を使用した後は通常十分です（5000回/時間）。
- **URL 解析**：ユーザーが入力する Repo URL は千差万別であり、正規表現で `.git` サフィックス、SSH フォーマットなどの情况をマッチングすることを忘れないでください。

### 今後の拡張
現在の機能はまだ一方向同期（HagiCode -> GitHub）です。将来的には GitHub Webhooks を通じて双方向同期を実現する予定です。例えば、GitHub で Issue を閉じると、HagiCode 側のセッション状態も自動的に更新されるなどです。これにはバックエンドに Webhook 受信エンドポイントを公開する必要があり、これも次に行う興味深い作業です。
