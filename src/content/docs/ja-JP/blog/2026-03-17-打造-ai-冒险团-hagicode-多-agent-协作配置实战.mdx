## AIパーティーを構築する：HagiCode マルチエージェント協調設定の実践

> 現代のソフトウェア開発において、単一のAIエージェントだけでは複雑なニーズを満たすのが難しくなっています。異なる企業の複数のAIアシスタントをどのように同一プロジェクトで協調動作させるのか？本文では、HagiCodeプロジェクトの実際の開発で探索したマルチエージェント協調設定ソリューションを共有します。

## 背景

多くの開発者が次のような経験をしたことがあると思います：プロジェクトでAIアシスタントを導入してプログラミングを支援し、確かに効率が向上しました。しかしニーズがますます複雑になるにつれて、1つのAIエージェントでは不十分になり始めました——コードレビュー、ドキュメント生成、単体テストなどの複数のタスクを同時に処理させようとすると、往々にして一方を立てれば他方が立たずになり、出力品質がまちまちになります。

さらに頭痛の種なのは、複数のAIアシスタントを導入しようとすると、問題はより複雑になることです。各エージェントには独自の設定方法、APIインターフェース、実行ロジックがあり、互いに競合さえ生じます。これはサッカーチームのようなもので、各選手は優秀ですが、誰も連携方法を知らないため、結果はめちゃくちゃになってしまいます。

HagiCodeプロジェクトも開発過程で同様の悩みに直面しました。フロントエンドのVSCode拡張、バックエンドのAIサービス、クロスプラットフォームデスクトップクライアントを含む複雑なプロジェクトとして、異なる企業の複数のAIアシスタントと同時に接続する必要があります：Claude Code、Codex、CodeBuddy、iFlowなど。それらを同一プロジェクトで調和させて共存させ、それぞれの長所を発揮させるにはどうすればよいか、解決すべき重要な問題になりました。

まあこれはさておき、毎日喧嘩するAIの群れと付き合いたい人は誰もいないでしょう。

本文で共有するソリューションは、まさに私たちがHagiCodeプロジェクトで実際に試行錯誤し、実際に最適化したマルチエージェント協調設定の実践です。もしあなたも複数のAIアシスタントの協調に頭を抱えているなら、この記事が何らかのヒントを与えてくれるはずです。たぶんね、結局のところ人の状況はみんな違いますから。

## HagiCodeについて

[HagiCode](https://hagicode.com) はAIコードアシスタントプロジェクトで、複数のAIエンジンが協調動作する「パーティー」モードを採用しています。プロジェクトアドレス：[github.com/HagiCode-org/site](https://github.com/HagiCode-org/site)。

本文で共有するマルチエージェント設定ソリューションは、HagiCodeが複雑なプロジェクトで高効率な開発を維持できる核心技術の一つです。特に何もなく、ただAIの群れを連携できるパーティーに変えるだけです。

## HagiCodeのマルチエージェント設計

### 「単独奮闘」から「チーム協調」へ

HagiCodeプロジェクトの初期段階では、すべてのタスクを処理するために1つのAIエージェントのみを使用することを試みました。すぐに、この方法には明らかなボトルネックがあることがわかりました：異なるタスクには異なる能力の重点が必要で、あるタスクはより強いコンテキスト理解能力を必要とし、あるタスクはより正確なコード修正能力を必要とします。1つのエージェントがすべての面で優秀に働くのは困難です。

これは私たちに、複数のエージェントを協調動作させる必要があることを認識させました。しかし問題は、異なる企業のAI製品をどのように同一プロジェクトで平和的に共存させるかです。解決すべきいくつかの核心的な問題があります：

1. **設定管理の複雑性**：各エージェントには異なる設定方法、APIインターフェース、実行モードがある
2. **通信プロトコルの統一**：異なるエージェント間でデータ交換を行うための標準化された方法が必要
3. **タスク分担の調整**：各エージェントが長所を発揮できるように合理的にタスクを割り当てるにはどうすればよいか

これらの問題を抱えて、私たちはHagiCodeのマルチエージェント設計を開始しました。それほど複雑ではなく、ただ理解しただけです。

### 全体設計の概要

何度かの反復を経て、私たちが最終的に決定した設計は以下の通りです：

```
┌─────────────────────────────────────────────────────────────────┐
│                    AIProviderFactory                             │
│  (ファクトリーパターンですべてのAI Providerを統一管理)              │
├─────────────────────────────────────────────────────────────────┤
│  ClaudeCodeCli  │  CodexCli  │  CodebuddyCli  │  IFlowCli    │
│  (Anthropic)   │  (OpenAI)  │  (智谱 GLM)     │  (智谱)      │
└─────────────────────────────────────────────────────────────────┘
```

核心的なアイデアは：統一されたProviderインターフェースを通じて、異なるAIエージェントを同一のコードセットで管理できるようにすることです。同時にファクトリーパターンを使用してこれらのProviderを動的に作成・設定し、システムの拡張性と柔軟性を確保します。

これは生活における分業のようなもので、誰もが自分の役割を持っていますが、ここではこの分業をコード設計に変えただけです。

### エージェントのタイプと責任の分担

HagiCodeプロジェクトの実際の使用経験に基づいて、各エージェントに異なる責任を割り当てました：

| Agent | プロバイダ | モデル | 主な用途 |
|-------|--------|------|---------|
| ClaudeCodeCli | Anthropic | glm-5-turbo | 技術ソリューションと提案の生成 |
| CodexCli | OpenAI/Zed | gpt-5.4 | 正確なコード変更の実行 |
| CodebuddyCli | 智谱 | glm-4.7 | 提案記述とドキュメントの最適化 |
| IFlowCli | 智谱 | glm-4.7 | 提案と履歴のアーカイブ |
| OpenCodeCli | - | - | 汎用コード編集 |
| GitHubCopilot | Microsoft | - | プログラミング支援とコード補完 |

この分業の背後にある論理は：各エージェントには自分の得意な分野があるということです。Claude Codeは複雑なニーズの理解と分析に優れているため、初期段階の設計を担当します。Codexはコード変更でより正確なため、具体的な実装タスクの処理に適しています。CodeBuddyはコストパフォーマンスが高く、ドキュメントの最適化に使用するのに最適です。

結局のところ自分に合ったものが最高で、すべての道はローマに通じますが、ある道は少し歩きやすく、ある道は少し曲がりくねっているだけです。

## 核心設定メカニズム

### 統一されたProviderインターフェース設計

異なるAIエージェントを統一管理できるようにするには、まず統一されたインターフェースセットを定義する必要があります。HagiCodeではこのインターフェースを定義しています：

```csharp
public interface IAIProvider
{
    // 統一されたProviderインターフェース
    Task<IAIProvider?> GetProviderAsync(AIProviderType providerType);
    Task<IAIProvider?> GetProviderAsync(string providerName, CancellationToken cancellationToken);
}
```

このインターフェースは単純に見えますが、マルチエージェントシステム全体の基石です。統一されたインターフェースを通じて、下層がどの企業のAI製品であるかに関係なく、同じ方法で呼び出すことができます。

実はこれは複雑なことを単純化しただけで、結局シンプルこそが美です。

### Providerファクトリーパターンの実装

統一されたインターフェースができたので、次はこれらのProviderインスタンスを作成する方法です。HagiCodeではファクトリーパターンを使用しています：

```csharp
private IAIProvider? CreateProvider(AIProviderType providerType, ProviderConfiguration config)
{
    return providerType switch
    {
        AIProviderType.ClaudeCodeCli =>
            ActivatorUtilities.CreateInstance<ClaudeCodeCliProvider>(_serviceProvider, Options.Create(config)),
        AIProviderType.CodebuddyCli =>
            ActivatorUtilities.CreateInstance<CodebuddyCliProvider>(_serviceProvider, Options.Create(config)),
        AIProviderType.CodexCli =>
            ActivatorUtilities.CreateInstance<CodexCliProvider>(_serviceProvider, Options.Create(config)),
        AIProviderType.IFlowCli =>
            ActivatorUtilities.CreateInstance<IFlowCliProvider>(_serviceProvider, Options.Create(config)),
        _ => null
    };
}
```

ここでは依存性注入の `ActivatorUtilities.CreateInstance` を使用しており、実行時にProviderインスタンスを動的に作成し、自動的に依存項目を注入できます。この設計の利点は：新しいエージェントタイプを追加する際、対応するProviderクラスを追加し、ファクトリーメソッドにcase分岐を追加するだけで、既存のコードを変更する必要が全くありません。

まあこれはさておき、新しい機能を追加するたびに既存のコードを山ほど変更したい人は誰もいないでしょう。

### 動的設定解析

設定をより柔軟にするために、タイプマッピングメカニズムも実装しました：

```csharp
public static AIProviderTypeExtensions
{
    private static readonly Dictionary<string, AIProviderType> _typeMap = new(
        StringComparer.OrdinalIgnoreCase)
    {
        ["ClaudeCodeCli"] = AIProviderType.ClaudeCodeCli,
        ["CodebuddyCli"] = AIProviderType.CodebuddyCli,
        ["CodexCli"] = AIProviderType.CodexCli,
        ["IFlowCli"] = AIProviderType.IFlowCli,
        // ...さらに多くのタイプマッピング
    };
}
```

このマッピングテーブルの役割は、文字列形式のProvider名を列挙型に変換することです。これにより、設定ファイルは直感的な文字列名を使用でき、コード内部ではタイプセーフな列挙型を使用して処理します。

結局設定というものは、直感的であればあるほどよく、誰も複雑なコードを覚えたいとは思いません。

### 設定ファイルの例

実際の使用時には、`appsettings.json` で設定するだけです：

```yaml
AI:
  Providers:
    Providers:
      ClaudeCodeCli:
        Enabled: true
        Model: glm-5-turbo
        WorkingDirectory: /path/to/project
      CodebuddyCli:
        Enabled: true
        Model: glm-4.7
      CodexCli:
        Enabled: true
        Model: gpt-5.4
      IFlowCli:
        Enabled: true
        Model: glm-4.7
```

各Providerは独立してスイッチ、モデルバージョン、作業ディレクトリなどのパラメータを設定できます。この設計は柔軟性を保証すると同時に、管理と保守も容易です。

実は設定ファイルは人生の選択肢のようなもので、特定の機能を有効または無効にすることを選択できますが、コード内の選択肢の方が後悔しやすいだけです。

## パーティータスクフロー

### タスク分担の芸術

統一された技術設計ができたので、次は複数のエージェントをどのように協調動作させるかです。HagiCodeではタスクフローメカニズムを設計し、異なるエージェントが異なる段階のタスクを処理するようにしました：

```
提案作成 (ユーザー)
    │
    ▼
[Claude Code] ──提案生成──▶ 提案ドキュメント
    │                        │
    │                        ▼
    │               [Codebuddy] ──記述最適化──▶ 最適化された提案
    │                        │
    │                        ▼
    │               [Codex] ──変更実行──▶ コード変更
    │                        │
    │                        ▼
    └───────────────▶ [iFlow] ──アーカイブ──▶ 履歴記録
```

この分業の利点は：各エージェントは自分の得意なタスクに集中するだけで、「何でもできる」必要がないことです。Claude Codeはゼロから提案を生成し、Codebuddyは提案の記述をより明確にし、Codexは提案を実際のコード変更にし、iFlowはこれらの変更をアーカイブして保存します。

実はこれは生活におけるチームワークのようなもので、誰もが自分の役割を持っており、合わさって初めて大きなことができます。ただここのチームメンバーはAIです。

### 重要な実践ポイント

実際の運用では、以下の経験をまとめました：

**1. エージェント選択戦略が重要**

タスクを無造作に割り当てるのではなく、各エージェントの特長に基づいて割り当てます：
- 提案生成：Claude Codeを使用、より強いコンテキスト理解能力があるため
- コード実行：Codexを使用、コード変更でより正確なため
- 提案最適化：CodeBuddyを使用、コストパフォーマンスが高いため
- アーカイブ保存：iFlowを使用、安定して信頼性が高いため

結局のところ適切な人に適切なことをさせるのは、永遠に変わらない真理です。

**2. 設定の分離で安定性を確保**

各エージェントの設定は独立して管理され、環境変数による上書きをサポートし、作業ディレクトリも相互に独立しています。これにより、1つのエージェントの設定ミスが他のエージェントに影響しません。

これは生活における境界のようなもので、誰もが自分のスペースを持ち、互いに干渉しないことで調和して共存できます。

**3. エラー処理メカニズム**

単一のエージェントの失敗は全体のフローに影響すべきではありません。私たちはデグレード戦略を実装しました：あるエージェントの実行が失敗した場合、システムは自動的に代替ソリューションに切り替えるか、そのステップをスキップして後続のタスクを実行し続けます。同時に、完全なログ記録も事後の問題調査に役立ちます。

結局のところ誰も絶対に間違わないことを保証できません、重要なのはエラーをどう処理するかです。これは人生のようなもので、いつか挫折に遭遇しますが、重要なのはどうそこから立ち上がるかです。

**4. モニタリングと可観測性**

ACPプロトコル（私たちがカスタマイズした通信プロトコル、JSON-RPC 2.0に基づく）を通じて、各エージェントの実行状態を追跡できます。セッション分離は並行安全性を確保し、動的キャッシュはパフォーマンスを最適化します。

結局のところ見えないものが最も問題を起こしやすく、少しのモニタリングは何もないよりはましです。

## 実際の効果と利点

このセットのマルチエージェント協調設定を採用した後、HagiCodeプロジェクトの開発効率は明らかに向上しました。具体的には：

1. **タスク処理能力が倍増**：以前は1つのエージェントが複数のタスクを同時に処理する必要がありましたが、現在は並列処理ができ、スループットが2倍以上になりました
2. **出力品質がより安定**：各エージェントは自分の得意なタスクに集中するだけで、出力結果の一貫性と品質がより高くなりました
3. **保守コストの削減**：統一されたインターフェースと設定管理により、システム全体の保守と拡張が容易になりました
4. **エージェントの追加が簡単**：新しいAI製品を接続する場合、インターフェースを実装し、設定を追加するだけで、核心ロジックを変更する必要がありません

このソリューションはHagiCode自身の問題を解決しただけでなく、マルチエージェント協調が確実に実行可能な設計選択であることを証明しました。

実は効果はかなり明らかでしたが、プロセスは少し面倒でした。

## まとめ

本文では、HagiCodeプロジェクトのマルチエージェント協調設定に関する実践経験を共有しました。核心的なポイントは以下の通りです：

1. **標準化インターフェース**：`IAIProvider` を通じて異なるエージェントの動作を統一し、コードが下層がどの企業の製品であるかを気にせずに済むようにします
2. **ファクトリーパターン**：`ActivatorUtilities.CreateInstance` を使用してProviderインスタンスを動的に作成し、実行時設定と依存性注入をサポートします
3. **プロトコル統一**：ACPプロトコルはエージェント間の標準化通信を実現し、JSON-RPC 2.0に基づく双方向通信メカニズムです
4. **タスク分流**：異なるエージェントに合理的にタスクを割り当て、それぞれが長所を発揮できるようにし、1つのエージェントですべてのことをさせようとするのではなく

この設計は「マルチエージェントの喧嘩」の問題を解決しただけでなく、パーティーのタスクフローメカニズムを通じて、開発プロセスの自動化と専門化を実現しました。
