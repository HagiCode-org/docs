## Docker コンテナでAI CLIツールを実行する：ユーザー分離と永続化ボリュームの実践ガイド

> コンテナ化環境に Claude Code、Codex、OpenCode などのAIプログラミングツールを統合するのは、簡単そうに見えて実は奥が深いです。本記事では、HagiCode プロジェクトが Docker デプロイでどのようにユーザー権限、設定の永続化、バージョン管理などの核心的な課題を解決したかを深く解説し、落とし穴を回避できるようにお手伝いします。

## 背景

Docker コンテナ内でAIプログラミング CLI ツールを実行することにしたとき、最も直感的なアイデアは「コンテナは root で動くんだから、そのままインストールすればいいんじゃない？」かもしれません。しかし、この考え方は一見シンプルですが、解決しなければならない核心的な問題がいくつか隠されています。

まず、**セキュリティ制限が最初の障壁**です。Claude CLI を例にとると、root ユーザーでの実行を明確に禁止しています——これは強制的なセキュリティチェックで、root を検出すると起動を拒否します。「じゃあ `USER` 命令で切り替えればいいじゃん」と思うかもしれません。そんなに単純ではありません。コンテナの非 root ユーザーとホストマシンのユーザー権限の間にはまだマッピング問題があります。世の中の物事、そんなに簡単なわけがないですよね？

次に、**状態の永続化が2つ目の落とし穴**です。Claude Code はログインが必要で、Codex には独自の設定があり、OpenCode にもキャッシュディレクトリがあります。コンテナが再起動するたびに再設定するなら、この「自動化」は全く意味がありません。これらの設定をコンテナのライフサイクル外で永続化する必要があります。設定というのは記憶のようなもので、消えてしまったら結構萎しいものです。

3つ目の問題は**権限の一貫性**です。ホストマシンのユーザーが作成した設定ファイルを、コンテナ内のプロセスがアクセスできるか？UID/GID の不一致はファイル権限エラーを引き起こし、これは実際のデプロイで非常に一般的です。この問題、言われてみれば仕方がないことですが、解決策もないんです。

これらの問題は独立しているように見えますが、実は密接に関連しています。HagiCode プロジェクトは開発過程で実行可能な解決策を見つけ出しました。これから技術的な詳細と落とし穴の経験を詳しく共有します。

## なぜ単純に root を使ってはいけないのか？

ここに一般的な誤解があります：Docker コンテナはデフォルトで root として実行されるので、そのまま root でツールをインストールすればいいじゃん。そう考えると、Claude CLI は容赦なく最初の挫折を与えてくれます。

```bash
# 直接 root で Claude CLI を実行？ダメです
docker run --rm -it --user root myimage claude
# 出力: Error: This command cannot be run as root user
```

これは Claude CLI の硬いセキュリティ制限です。理由はシンプルです：これらの CLI ツールはユーザーの機微な設定（API トークン、ローカルキャッシュ、ユーザーが書いたスクリプトを実行する可能性もある）を読み書きします。root 権限でこれらのツールを実行すると、潜在的なリスクが大きすぎます。セキュリティというのは、どんなに慎重になっても慎重すぎることはありませんから。

では問題です：CLI のセキュリティ要件を満たしつつ、コンテナ管理の柔軟性を維持するにはどうすればいい？発想を変える必要があります——**実行時にユーザーを切り替えるのではなく、イメージ構築段階から専用ユーザーを作成する**のです。物事を別の角度から見ると、答えが自然と浮かび上がってくることもあります。

## 専用ユーザーの作成：名前を変えるだけじゃない

「じゃあ Dockerfile に `USER` 命令を1行追加すればいいんじゃない？」と思うかもしれません。これは確かに最もシンプルな解決策ですが、十分堅牢ではありません。シンプルなものは往々にしてエレガントではありませんよね？

### 静的作成 vs 動的マッピング

HagiCode のソリューションは UID 1000 の `hagicode` ユーザーを作成することです。この UID は通常、大多数のホストマシンのデフォルトユーザーに一致します：

```dockerfile
RUN groupadd -o -g 1000 hagicode && \
    useradd -o -u 1000 -g 1000 -s /bin/bash -m hagicode && \
    mkdir -p /home/hagicode/.claude && \
    chown -R hagicode:hagicode /home/hagicode
```

しかし、これはイメージに組み込まれたユーザーの問題しか解決していません。ホストマシンのユーザーが UID 1001 だったら？コンテナ起動時には動的マッピングもサポートする必要があります。

docker-entrypoint.sh での重要なロジック：

```bash
if [ -n "$PUID" ] && [ -n "$PGID" ]; then
    if ! id hagicode >/dev/null 2>&1; then
        groupadd -g "$PGID" hagicode
        useradd -u "$PUID" -g "$PGID" -s /bin/bash -m hagicode
    fi
fi
```

この設計の利点は**イメージ構築時にはデフォルトの UID 1000 を使用し、実行時には環境変数 PUID/PGID で動的に調整できる**ことです。ホストマシンのユーザーがどんな UID でも、設定ファイルの所有権に問題は発生しません。この設計、言われてみれば自然です。結局、柔軟性とデフォルト値の間でバランスを見つける必要があるだけです。

## 永続化ボリュームの設計哲学

AI CLI ツールごとに好まれる設定ディレクトリがあり、これらを一つ一つ対応させる必要があります：

| CLI ツール | コンテナ内パス | 名前付きボリューム |
|---------|-----------|--------|
| Claude | `/home/hagicode/.claude` | `claude-data` |
| Codex | `/home/hagicode/.codex` | `codex-data` |
| OpenCode | `/home/hagicode/.config/opencode` | `opencode-config-data` |

なぜ名前付きボリュームを使うのか、バインドマウントではなく？3つの理由があります：

1. **管理の簡素化**：名前付きボリュームは Docker が自動的にライフサイクルを管理し、ホストマシンのディレクトリを手動で作成する必要がありません
2. **権限の分離**：ボリュームの初期コンテンツはコンテナ内のユーザーによって作成され、ホストマシンの権限競合を回避します
3. **独立した移行**：ボリュームはコンテナから独立して存在でき、イメージをアップグレードしてもデータが失われません

docker-compose-builder-web は対応するボリューム設定を自動的に生成します：

```yaml
volumes:
  claude-data:
  codex-data:
  opencode-config-data:

services:
  hagicode:
    volumes:
      - claude-data:/home/hagicode/.claude
      - codex-data:/home/hagicode/.codex
      - opencode-config-data:/home/hagicode/.config/opencode
    user: "${PUID:-1000}:${PGID:-1000}"
```

ここでの `user` フィールドに注意してください：環境変数で PUID/PGID を注入し、コンテナプロセスがホストマシンのユーザーIDに一致して実行されるようにします。この詳細、実は結構重要です。権限問題が一度発生すると、調査するのも結構頭の痛いことになりますから。

## バージョン管理：ベイクされたバージョンと実行時上書き

Docker イメージのバージョンを固定することは再現性を保証する鍵です。しかし実際の開発では、新しいバージョンをテストしたり、緊急でバグを修正したりする必要がよくあります。毎回イメージを再構築するのは効率が悪すぎます。

HagiCode の戦略は**固定バージョンをデフォルト値とし、実行時の上書きを拡張機能とする**ことです。これはエンジニアリング実践の一種の妥協でもあります。安定性と柔軟性の間には必ず取舍選択が必要です。

Dockerfile.template で固定バージョン：

```dockerfile
USER hagicode
WORKDIR /home/hagicode

# npm グローバルインストールパスを設定
RUN mkdir -p /home/hagicode/.npm-global && \
    npm config set prefix '/home/hagicode/.npm-global'

# CLI ツールをインストール（固定バージョンを使用）
RUN npm install -g @anthropic-ai/claude-code@2.1.71 && \
    npm install -g @openai/codex@0.112.0 && \
    npm install -g opencode-ai@1.2.25 && \
    npm cache clean --force
```

docker-entrypoint.sh で実行時上書きをサポート：

```bash
install_cli_override_if_needed() {
    local package_name="$2"
    local override_version="$5"

    if [ -n "$override_version" ]; then
        gosu hagicode npm install -g "${package_name}@${override_version}"
    fi
}

# 使用例
install_cli_override_if_needed "" "@anthropic-ai/claude-code" "" "" "${CLAUDE_CODE_CLI_VERSION}"
```

こうすることで、イメージを再構築せずに環境変数で新しいバージョンをテストできます：

```bash
docker run -e CLAUDE_CODE_CLI_VERSION=2.2.0 myimage
```

この設計、実用的です。新しい機能をテストするたびにイメージを再構築したい人なんていないですからね。

## 自動設定注入

CLI ツールの手動設定以外に、一部のシーンでは設定の自動注入も必要です。最も典型的なのは API トークンです。

```bash
if [ -n "$ANTHROPIC_AUTH_TOKEN" ]; then
    mkdir -p /home/hagicode/.claude
    cat > /home/hagicode/.claude/settings.json <<EOF
{
  "env": {
    "ANTHROPIC_AUTH_TOKEN": "${ANTHROPIC_AUTH_TOKEN}"
  }
}
EOF
    chown -R hagicode:hagicode /home/hagicode/.claude
fi
```

ここで2つの点に注意が必要です：**機微な情報は環境変数で渡し、イメージにハードコードしない**こと；設定ファイルの所有権を正しく設定すること。さもないと CLI ツールが読み取れません。これは基本的なことですが、間違う人は結構多いです。

## ベストプラクティスと落とし穴回避ガイド

### 権限不一致の問題

これは最も踏みやすい落とし穴です。ホストマシンのユーザー UID が 1001 で、コンテナ内が 1000 だと、作成したファイルがお互いにアクセスできません。

```bash
# 正しい方法：コンテナをホストマシンのユーザーに合わせる
docker run \
    -e PUID=$(id -u) \
    -e PGID=$(id -g) \
    myimage
```

この問題、よくあることですが、初めて遭遇したときは結構萎しいです。

### コンテナ再起動後の設定消失

毎回再起動するたびに再ログインが必要になることに気づいたら、永続化ボリュームをマウントするのを忘れていないか確認してください：

```yaml
volumes:
  - claude-data:/home/hagicode/.claude
```

設定、苦労して設定したのに消えてしまったら、その感覚、どう言えばいいか、結構崩れ落ちますよね。

### バージョンアップグレードの正しい方法

実行中のコンテナで直接 `npm install -g` を実行しないでください。正しい方法は：

1. 環境変数を設定して上書きインストールをトリガーする
2. またはイメージを再構築する

```bash
# 方法1：実行時上書き
docker run -e CLAUDE_CODE_CLI_VERSION=2.2.0 myimage

# 方法2：再構築
docker build -t myimage:v2 .
```

「すべての道はローマに通ず」と言いますが、ある道は歩きやすく、ある道は少し曲がりくねっているだけです。

### セキュリティ強化チェックリスト

- API トークンは環境変数で渡し、イメージに書き込まない
- 設定ファイルは 600 権限に設定
- 常に非 root ユーザーでアプリケーションを実行
- 定期的に CLI バージョンを更新し、セキュリティ脆弱性を修正

セキュリティ、重要だと言われていますが、実際に実装する際、どれくらいの人がしっかりやっているのでしょうか？

## 新しい CLI ツールの拡張

将来的に新しい CLI ツールをサポートする場合、必要なのは3ステップだけです：

テンプレート化された設計により、拡張が簡単になり、コアロジックを変更する必要がありません。これは経験者からの心得です。大した理屈ではありませんが、踏んだ落とし穴から来たものです。

## まとめ

Docker コンテナで AI CLI ツールを実行する場合、核心的な課題は**ユーザー権限、設定の永続化、バージョン管理**の3次元にあります。HagiCode プロジェクトは専用ユーザーの作成、名前付きボリュームによる分離、環境変数による上書きを組み合わせたソリューションにより、安全で柔軟なデプロイアーキテクチャを実現しました。

重要な設計ポイント：

- **ユーザー分離**：イメージ構築段階から専用ユーザーを作成し、実行時には PUID/PGID 動的マッピングをサポート
- **永続化戦略**：各 CLI ツールが独立した名前付きボリュームに対応し、コンテナの再起動が設定に影響しない
- **バージョンの柔軟性**：デフォルト値を固定して再現性を確保し、実行時上書きでテスト能力を提供
- **自動設定**：環境変数による機微な設定の自動注入をサポート

このソリューションは HagiCode プロジェクトでしばらく安定して動作しており、同様のニーズを持つ開発者の方々の参考になれば幸いです。実はそれほど複雑ではなく、単なるエンジニアリング実践に過ぎません。
