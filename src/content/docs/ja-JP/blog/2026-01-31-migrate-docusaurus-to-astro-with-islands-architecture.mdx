## Docusaurus 3.x から Astro 5.x へ：HagiCode サイト移行実戦レビュー

> この記事では、HagiCode 公式サイトを Docusaurus 3.x から Astro 5.x に移行した全プロセスを振り返ります。Astro の Islands アーキテクチャを通じてパフォーマンスのボトルネックを解決しながら、既存の React コンポーネント資産を保持し、ビルド速度と読み込みパフォーマンスの両方を向上させる方法について深く掘り下げます。


## 背景

2026 年 1 月、HagiCode の公式サイトに「心臓移植手術」を行いました——コアフレームワークを Docusaurus 3.x から Astro 5.x に全面的に移行しました。これは衝動的な大規模リファクタリングではなく、熟慮された技術的決断です。

移行前、サイトの機能は完善していましたが、次第にいくつかの「富裕層病」が顕在化していました：ビルド产物のサイズが肥大化し、JavaScript の負荷が過剰で、複雑なドキュメントページではページ読み込み速度が理想的ではありませんでした。AI コードアシスタントプロジェクトとして、HagiCode は頻繁にドキュメントと機能紹介を更新する必要があり、ビルド効率はリリース速度に直接影響します。同時に、より多くの開発者がこのプロジェクトを見つけられるよう、サイトが検索エンジン（SEO）に優しいことを望んでいました。

これらの痛点を解決するため、私たちは大胆な決断を下しました：ビルドシステム全体を作り直し、Astro に移行すること。この決断がもたらす変化は、想像以上に大きなものでした——後ほど具体的に説明します。

## HagiCode について

この記事で共有するサイト移行ソリューションは、[HagiCode](https://github.com/HagiCode-org/site) プロジェクトでの実践経験に基づいています。

HagiCode は開発効率の向上に尽力する AI コードアシスタントで、コア機能の反復だけでなく、開発者体験も重視しています。今回のサイトリファクタリングも、ユーザーがドキュメントと公式サイトを閲覧する際に極限の読み込み速度を得られるようにするためです。

## なぜ成熟した Docusaurus を放棄したのか？

React エコシステムでは、Docusaurus は常にドキュメントサイトの「標準答え」でした。すぐに使えて、プラグインが豊富で、コミュニティが活発です。しかし、HagiCode の機能増加に伴い、私たちはその限界も感じ始めました：

1.  **パフォーマンスのボトルネック**：Docusaurus は本質的に React SPA（シングルページアプリケーション）です。純粋な静的ページを書いても、クライアント側で React ランタイムをロードし水和を行う必要があり、単純なドキュメントページには重すぎます。
2.  **リソースサイズ**：ページコンテンツが少なくても、バンドル後の JS サイズは比較的固定されており、モバイルユーザーやネットワーク環境が悪い環境には優しくありません。
3.  **柔軟性不足**：拡張は可能ですが、ビルドプロセスのカスタマイズにおいて、より低レベルの制御権を求めていました。

Astro の登場はまさにこれらの問題を解決しました。それは全新的な「アイランドアーキテクチャ」（Islands Architecture）を提供します：デフォルトでは、Astro はゼロ JavaScript の静的 HTML を生成し、インタラクションが必要なコンポーネントのみが「アクティブ化」され JS をロードします。これはサイトの大部分が純粋な HTML であり、非常に高速であることを意味します。

## 移行コア戦略：アーキテクチャのスムーズな移行

移行は単なるコピーペーストではなく、思考パターンの転換です。Docusaurus の「全 React モード」から Astro の「Core + Islands」モードに切り替えました。

### 1. 設定システムのリファクタリング

まず、`docusaurus.config.ts` から `astro.config.mjs` に移行する必要があります。これはファイル名の変化だけでなく、ルーティングとビルドロジックの書き直しでもあります。

Docusaurus ではすべてがプラグインですが、Astro ではすべてが統合です。サイトのベースパス、ビルド出力モード（静的 vs SSR）、およびリソース圧縮戦略を再定義する必要があります。

**移行前：**
```typescript
// docusaurus.config.ts
export default {
  title: 'HagiCode',
  url: 'https://hagicode.com',
  baseUrl: '/',
  // ... その他の設定
};
```

**移行後：**
```javascript
// astro.config.mjs
import { defineConfig } from 'astro/config';
import react from '@astrojs/react';

export default defineConfig({
  integrations: [react()],
  site: 'https://hagicode.com',
  base: '/',
  // 静的リソース向けの最適化設定
  build: {
    inlineStylesheets: 'auto',
  },
});
```

### 2. React コンポーネントの去留と改造

これが移行で最も頭痛の痛い部分です。既存のサイトには多くの React コンポーネント（Tabs コンポーネント、コードハイライト、フィードバックボタンなど）があります。直接捨てるには惜しいし、全部保持すると JS 負荷が過重になります。

HagiCode は**漸進的水和**戦略を採用しました：
*   **純粋な静的コンポーネント**：表示型コンテンツ（ヘッダー、フッター、純粋なテキストドキュメントなど）の場合、Astro コンポーネント（`.astro` ファイル）に書き換え、ビルド時に直接 HTML としてレンダリングします。
*   **インタラクティブなアイランド**：インタラクションを保持する必要があるコンポーネント（テーマスイッチャー、Tabs 切り替え、コードブロックコピーボタンなど）の場合、React 実装を保持し、`client:load` または `client:visible` ディレクティブを追加します。

例えば、私たちのドキュメントでよく使われる Tabs コンポーネント：

```jsx
// src/components/Tabs.jsx
import { useState } from 'react';
import './Tabs.css'; // スタイルのインポート

export default function Tabs({ items }) {
  const [activeIndex, setActiveIndex] = useState(0);
  // ... 状態ロジック
  return (
    <div className="tabs-wrapper">
      {/* レンダリングロジック */}
    </div>
  );
}
```

Markdown で使用する際、Astro に明確に伝えます：「このコンポーネントは JS が必要」：

```markdown
// src/content/docs/example.mdx
import Tabs from '../../components/Tabs.jsx';

<!-- コンポーネントがビューポートに入った時のみ JS をロード -->
<Tabs client:visible items={...} />
```

こうすることで、ビューポート外のインタラクティブコンポーネントが帯域幅を奪うことがなく、ファーストビューの読み込み速度が大幅に最適化されます。

### 3. スタイルシステムの適合：CSS Modules から Scoped へ

Docusaurus はデフォルトで CSS Modules をサポートし、Astro は Scoped CSS（`<style>` タグを通じて）の使用を推奨しています。両方の核心思想はスタイルの分離ですが、構文が異なります。

HagiCode の移行では、大部分の複雑な CSS Modules を Astro の Scoped スタイルに分解しました。これは実は良いことで、`.astro` ファイルではスタイルとテンプレートが同じファイルに書かれるため、メンテナンスがより直感的になります。

**改造前：**
```css
/* Tabs.module.css */
.wrapper { background: var(--ifm-background-color); }
```

**改造後 (Astro Scoped):**
```html
<!-- Tabs.astro -->
<div class="tabs-wrapper">
  <slot />
</div>

<style>
  .tabs-wrapper {
    /* CSS 変数を直接使用、テーマに適合 */
    background: var(--bg-color);
    padding: 1rem;
  }
</style>
```

同時に、グローバル CSS 変数システムを統一し、Astro の環境認知能力を活用して、ダークモードが異なるページ間でシームレスに切り替わるようにしました。

## 実践における落とし穴と解決策

HagiCode の実際の移行プロセスで、多くの落とし穴に遭遇しました。ここでは最も典型的ないくつかを共有します。

### 1. パスと環境変数の痛点

HagiCode はサブパスデプロイ（GitHub Pages のサブディレクトリへのデプロイなど）をサポートしています。Docusaurus では、`baseUrl` を自動的に処理します。しかし Astro では、画像リンクと API リクエストを処理する際、より注意が必要です。

統一管理するために環境変数メカニズムを導入しました：

```javascript
// ビルドスクリプトでパスを処理
const getBasePath = () => import.meta.env.VITE_SITE_BASE || '/';
```
必ず覚えておいてください、コード内で `/` で始まるパスをハードコードしないでください。開発環境と本番環境、または `base` パスを設定した後、これはリソースの 404 を引き起こします。

### 2. CommonJS スクリプトの互換性

私たちの旧サイトには、Node.js スクリプト（Metrics データの自動取得、sitemap の更新など）があり、それらは CommonJS (`require`) で書かれていました。Astro と現代のビルドツールは ES Modules (`import`/`export`) を全面的に採用しています。

同様のスクリプトがある場合、それらをすべて ES Modules にリファクタリングすることを忘れないでください。これは時代の趨勢であり、早めに改めれば早めに楽になります。

```javascript
// 旧方式
const fs = require('fs');

// 新方式
import fs from 'fs';
```

### 3. SEO とリダイレクトを忘れずに

検索エンジンはすでに HagiCode の旧 Docusaurus ページをインデックスしています。直接 Astro に切り替えると、URL 構造が変化し、大量の 404 が発生し、順位が大幅に下がります。

Astro でリダイレクトルールを設定しました：

```javascript
// astro.config.mjs
export default defineConfig({
  redirects: {
    '/docs/old-path': '/docs/new-path',
    // 旧リンクから新リンクへの一括マッピング
  }
});
```
またはサーバー設定レベルで処理します。旧リンクが新しいアドレスに 301 リダイレクトすることを保証する。これは SEO にとって非常に重要です。

## まとめ

Docusaurus から Astro への移行は、HagiCode にとって単なるフレームワークアップグレードではなく、「パフォーマンス優先」理念の実践でもあります。

**私たちの収穫：**
*   **極限の Lighthouse スコア**：移行後、HagiCode サイトのパフォーマンススコアは容易に満点に近づきました。
*   **より高速なビルド速度**：Astro の増分ビルドにより、ドキュメント更新のリリース時間が半分になりました。
*   **柔軟性の保持**：Islands アーキテクチャを通じて、インタラクティブ機能を一切犠牲にすることなく、必要な場所で依然として React を使用できます。

ドキュメント型サイトを保守していて、バンドルサイズや読み込み速度に悩まされている場合、Astro を試してみてください。移行プロセスには手術が必要ですが（PCode の名前を HagiCode に変更し、コンポーネントを一つずつ移すなど）、得られるのは絹のように滑らかなユーザー体験であり、絶対に価値があります。

この記事で共有したビルドシステムは、私たちが HagiCode を開発する過程で実際に踏み抜いた落とし穴と、実際に最適化したソリューションです。このソリューションに価値を感じてくれたなら、私たちのエンジニアリング力は悪くありません——では HagiCode 自体も注目に値します。
