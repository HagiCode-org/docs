## AI 冒険団を構築する：HagiCode のマルチ Agent 協調設定の実践

> 現代のソフトウェア開発において、単一の AI Agent では複雑なニーズを満たすのが難しくなっています。異なる企業の複数の AI アシスタントを同じプロジェクトで協調動作させるにはどうすればよいでしょうか？本文では HagiCode プロジェクトが実際の開発で探索したマルチ Agent 協調設定ソリューションを共有します。

:::note[時間的文脈の説明]
本文は **2026-03-17 リリース時** のマルチ Agent 設定の実践を記録しています。現在の常駐サポート範囲はバックエンドの現行設定が基準です：Claude Code、Codex、GitHub Copilot、OpenCode、Hermes、QoderCLI、Kiro、Kimi、Gemini、DeepAgents、Codebuddy；うち `IFlowCli` は現在歴史的互換性のためにのみ保持されており、現在のアクティブサポート範囲には属しません。現在のインストールとサポートの基準を確認したい場合は、[AI Agent CLI のインストール](/related-software-installation/ai-agent-cli/) と [インストールガイド](/installation) を参照してください。

## 背景

多くの開発者が次のような経験をしたことがあると思います：プロジェクトに AI アシスタントを導入してプログラミングを支援させると、確かに効率が向上します。しかしニーズがますます複雑になるにつれて、1 つの AI Agent では不足し始めます——コードレビュー、ドキュメント生成、単体テストなど複数のタスクを同時に処理させようとすると、往々にして一方を犠牲に他方を得る結果となり、出力品質がまちまちになります。

さらに頭が痛いのは、複数の AI アシスタントを導入しようとすると、問題がより複雑になることです。各 Agent には独自の設定方法、API インターフェース、実行ロジックがあり、互いに競合さえ生じます。これはサッカーチームのように、各選手は凄腕ですが、誰も連携方法を知らないため、結果的にめちゃくちゃなプレイになってしまうのと同じです。

HagiCode プロジェクトも開発過程で同様の悩みに遭遇しました。フロントエンドの VSCode 拡張、バックエンドの AI サービス、クロスプラットフォームデスクトップクライアントを含む複雑なプロジェクトとして、**2026-03 当時のバージョン**では異なる企業からの複数の AI アシスタント：Claude Code、Codex、CodeBuddy、iFlow などに同時に接続する必要がありました。これらを同じプロジェクトで調和させて共存させ、それぞれの長所を発揮させるにはどうすればよいかということが、解決すべき重要な問題になりました。

まあこれもそうですが、毎日喧嘩する AI のグループと付き合いたい人はいません。

本文で共有するソリューションは、私たちが HagiCode プロジェクトで実際に試行錯誤し、実際に最適化したマルチ Agent 協調設定の実践です。もし複数の AI アシスタントの連携で頭を抱えているなら、この記事が何らかのヒントを与えてくれるかもしれません。たぶんね、結局のところ人それぞれ状況が違いますから。

## HagiCode について

[HagiCode](https://hagicode.com) は、複数の AI エンジンが協調動作する「冒険団」モを採用した AI コードアシスタントプロジェクトです。プロジェクトアドレス：[github.com/HagiCode-org/site](https://github.com/HagiCode-org/site)。

本文で共有するマルチ Agent 設定ソリューションは、HagiCode が複雑なプロジェクトで効率的な開発を維持できるコア技術の一つです。何も特別なことはなく、単に一群の AI を連携できる冒険団にしただけです。

## HagiCode のマルチ Agent アーキテクチャ設計

### 「単独独闘」から「チーム協調」へ

HagiCode プロジェクトの初期段階では、すべてのタスクを処理するために 1 つの AI Agent のみを使用することも試みました。すぐに、この方法には明らかなボトルネックがあることがわかりました：タスクによって必要な能力の重点が異なり、あるタスクはより強いコンテキスト理解能力を必要とし、別のタスクはより正確なコード修正能力を必要とします。1 つの Agent がすべての面で優秀に振る舞うのは困難です。

これにより、複数の Agent を協調動作させる必要があることに気づきました。しかし問題は、異なる企業の AI 製品を同じプロジェクトで平和的に共存させるにはどうすればよいかということです。解決すべきいくつかのコア問題があります：

1. **設定管理の複雑性**：各 Agent には異なる設定方法、API インターフェース、実行モードがある
2. **通信プロトコルの統一**：異なる Agent 間でデータ交換を行うための標準化された方法が必要
3. **タスク分業の調整**：各 Agent の長所を発揮させるために、合理的にタスクを割り当てるにはどうすればよいか

これらの問題を抱えて、私たちは HagiCode のマルチ Agent アーキテクチャの設計を始めました。そんなに複雑ではなく、ただ理解しただけです。

### アーキテクチャ全体図

何度かの反復を経て、最終的に決定したアーキテクチャは次のとおりです：

```
┌─────────────────────────────────────────────────────────────────┐
│                    AIProviderFactory                             │
│  (ファクトリーモードですべての AI Provider を統一管理)              │
├─────────────────────────────────────────────────────────────────┤
│  ClaudeCodeCli  │  CodexCli  │  CodebuddyCli  │  IFlowCli    │
│  (Anthropic)   │  (OpenAI)  │  (智谱 GLM)     │  (智谱)      │
└─────────────────────────────────────────────────────────────────┘
```

コアの考え方は、統一された Provider インターフェースを通じて、異なる AI Agent を同一のコードセットで管理できるようにすることです。同時にファクトリーモードを使用してこれらの Provider を動的に作成・設定し、システムの拡張性と柔軟性を確保します。

これは生活の中の分業のようなもので、誰も自分の役割を持っていますが、ここではこの分業をコードアーキテクチャにしただけです。

### Agent タイプと役割分担

HagiCode プロジェクトの実際の使用経験に基づいて、各 Agent に異なる役割を割り当てました：

| Agent | プロバイダ | モデル | 主な用途 |
|-------|--------|------|---------|
| ClaudeCodeCli | Anthropic | glm-5-turbo | 技術ソリューションと提案の生成 |
| CodexCli | OpenAI/Zed | gpt-5.4 | 正確なコード修正の実行 |
| CodebuddyCli | 智谱 | glm-4.7 | 提案記述とドキュメントの最適化 |
| IFlowCli | 智谱 | glm-4.7 | 提案と履歴のアーカイブ（当時の設定；現在は歴史的互換のみ） |
| OpenCodeCli | - | - | 汎用コード編集 |
| GitHubCopilot | Microsoft | - | プログラミング支援とコード補完 |

この分業の背後にある論理は、各 Agent には自分が得意とする分野があるということです。Claude Code は複雑なニーズの理解と分析に優れているため、初期のソリューション設計を担当します。Codex はコード修正により正確であるため、具体的な実装タスクの処理に適しています。CodeBuddy はコストパフォーマンスが高く、ドキュメントの最適化に使用するのに最適です。

結局自分に適したものが最高で、すべての道はローマに通じますが、ある道は少し歩きやすく、ある道は少し曲がりくねっているだけです。

## コア設定メカニズム

### 統一された Provider インターフェース設計

異なる AI Agent を統一管理できるようにするには、まず統一されたインターフェースセットを定義する必要があります。HagiCode ではこのインターフェースを定義しました：

```csharp
public interface IAIProvider
{
    // 統一された Provider インターフェース
    Task<IAIProvider?> GetProviderAsync(AIProviderType providerType);
    Task<IAIProvider?> GetProviderAsync(string providerName, CancellationToken cancellationToken);
}
```

このインターフェースは単純に見えますが、マルチ Agent システム全体の基石です。統一されたインターフェースを通じて、下層がどの企業の AI 製品であるかにかかわらず、同じ方法で呼び出すことができます。

実際これは複雑なことを単純化しただけであり、結局シンプルこそが美です。

### Provider ファクトリーモードの実装

統一されたインターフェースがあれば、次はこれらの Provider インスタンスを作成する方法です。HagiCode ではファクトリーモードを使用しています：

```csharp
private IAIProvider? CreateProvider(AIProviderType providerType, ProviderConfiguration config)
{
    return providerType switch
    {
        AIProviderType.ClaudeCodeCli =>
            ActivatorUtilities.CreateInstance<ClaudeCodeCliProvider>(_serviceProvider, Options.Create(config)),
        AIProviderType.CodebuddyCli =>
            ActivatorUtilities.CreateInstance<CodebuddyCliProvider>(_serviceProvider, Options.Create(config)),
        AIProviderType.CodexCli =>
            ActivatorUtilities.CreateInstance<CodexCliProvider>(_serviceProvider, Options.Create(config)),
        AIProviderType.IFlowCli =>
            ActivatorUtilities.CreateInstance<IFlowCliProvider>(_serviceProvider, Options.Create(config)),
        _ => null
    };
}
```

ここでは依存性注入の `ActivatorUtilities.CreateInstance` を使用しており、実行時に Provider インスタンスを動的に作成し、依存項目を自動的に注入できます。この設計の利点は、新しい Agent タイプを追加する場合、対応する Provider クラスを追加し、ファクトリーメソッドに case 分岐を追加するだけであり、既存のコードを完全に修正する必要がないことです。

これもそうですが、結局新しい機能を追加するたびに既存のコードを大量に修正したい人はいません。

### 動的設定解析

設定をより柔軟にするために、タイプマッピングメカニズムも実装しました：

```csharp
public static AIProviderTypeExtensions
{
    private static readonly Dictionary<string, AIProviderType> _typeMap = new(
        StringComparer.OrdinalIgnoreCase)
    {
        ["ClaudeCodeCli"] = AIProviderType.ClaudeCodeCli,
        ["CodebuddyCli"] = AIProviderType.CodebuddyCli,
        ["CodexCli"] = AIProviderType.CodexCli,
        ["IFlowCli"] = AIProviderType.IFlowCli,
        // ...その他のタイプマッピング
    };
}
```

このマッピングテーブルの役割は、文字列形式の Provider 名を列挙型に変換することです。これにより、設定ファイルは直感的な文字列名を使用でき、コード内部ではタイプセーフな列挙型を使用して処理できます。

結局設定のようなものは直感的であればあるほどよく、誰も複雑なコードを覚えたいとは思いません。

### 設定ファイルの例

実際に使用する際は、`appsettings.json` で設定するだけです：

```yaml
AI:
  Providers:
    Providers:
      ClaudeCodeCli:
        Enabled: true
        Model: glm-5-turbo
        WorkingDirectory: /path/to/project
      CodebuddyCli:
        Enabled: true
        Model: glm-4.7
      CodexCli:
        Enabled: true
        Model: gpt-5.4
      IFlowCli:
        Enabled: true
        Model: glm-4.7
```

各 Provider は独立してオン/オフ、モデルバージョン、作業ディレクトリなどのパラメータを設定できます。この設計は柔軟性を保証すると同時に、管理と保守にも便利です。

実際、設定ファイルは人生の選択肢のようなもので、特定の機能をオンまたはオフにすることができますが、コード内の選択は後悔しやすいだけです。

## 冒険団タスクフロー

### タスク分業の芸術

統一された技術アーキテクチャができれば、次は複数の Agent を協調動作させる方法です。HagiCode はタスクフローメカニズムを設計し、異なる Agent が異なる段階のタスクを処理するようにしました：

```
提案作成 (ユーザー)
    │
    ▼
[Claude Code] ──提案生成──▶ 提案ドキュメント
    │                        │
    │                        ▼
    │               [Codebuddy] ──記述最適化──▶ 最適化された提案
    │                        │
    │                        ▼
    │               [Codex] ──修正実行──▶ コード変更
    │                        │
    │                        ▼
    └───────────────▶ [iFlow] ──アーカイブ──▶ 履歴記録
```

この分業の利点は、各 Agent が自分が得意とするタスクに集中するだけで、「何でもできる」必要がないことです。Claude Code はゼロから提案を生成することを担当し、Codebuddy は提案記述をより明確にすることを担当し、Codex は提案を実際のコード変更にすることを担当し、iFlow はこれらの変更をアーカイブ保存することを担当します。

実際これは生活の中のチームワークのようなもので、誰も自分の役割を持っており、合わせて初めて大きなことを成し遂げることができます。ただここのチームメンバーが AI であるだけです。

### 重要な実践の要点

実際の運用では、以下の経験をまとめました：

**1. Agent 選択戦略が重要**

タスクを無闇に割り当てるのではなく、各 Agent の長所に基づいて割り当てる必要があります：
- 提案生成：より強力なコンテキスト理解能力があるため Claude Code を使用
- コード実行：コード修正により正確であるため Codex を使用
- 提案最適化：コストパフォーマンスが高いため Codebuddy を使用
- アーカイブ保存：安定して信頼性が高いため iFlow を使用

結局適切な人に適切なことをさせるのは、永遠に変わらない真理です。

**2. 設定の分離が安定性を確保**

各 Agent の設定は独立して管理され、環境変数による上書きをサポートし、作業ディレクトリも相互に独立しています。これにより、ある Agent の設定エラーが他の Agent に影響しません。

これは生活の中の境界のようなもので、誰も自分の空間を持っており、互いに干渉しないからこそ調和して共存できます。

**3. エラー処理メカニズム**

単一の Agent の失敗が全体のフローに影響すべきではありません。私たちは降格戦略を実装しました：ある Agent の実行が失敗した場合、システムは自動的に予備案に切り替えるか、そのステップを直接スキップして後続のタスクを実行し続けることができます。同時に、完全なログ記録も事後の問題調査に便利です。

結局誰も絶対に間違わないことを保証できませんが、重要なのはエラーをどう処理するかです。これは人生と同じで、必ず挫折に遭遇しますが、重要なのはどう立ち直るかです。

**4. モニタリングと可観測性**

ACP プロトコル（私たちがカスタマイズした通信プロトコル、JSON-RPC 2.0 に基づく）を通じて、各 Agent の実行状態を追跡できます。セッション分離は並行安全を確保し、動的キャッシュはパフォーマンスを最適化します。

結局見えないものが最も問題を起こしやすく、モニタリングがあるよりは真っ暗な方がいいです。

## 実際の効果と利点

このマルチ Agent 協調設定を採用した後、HagiCode プロジェクトの開発効率は明らかに向上しました。具体的には：

1. **タスク処理能力が倍増**：以前は 1 つの Agent が複数のタスクを同時に処理する必要がありましたが、現在は並列処理でき、スループットが倍以上になりました
2. **出力品質がより安定**：各 Agent が自分が得意とするタスクに集中するため、出力結果の一貫性と品質がより高くなりました
3. **保守コストの削減**：統一されたインターフェースと設定管理により、システム全体の保守と拡張が容易になりました
4. **Agent の追加が簡単**：新しい AI 製品を接続する場合、インターフェースを実装し、設定を追加するだけであり、コアロジックを修正する必要がありません

このソリューションは HagiCode 自身の問題を解決しただけでなく、マルチ Agent 協調が実際に実行可能なアーキテクチャ選択であることを証明しました。

実際、効果はかなり明らかですが、プロセスは少し面倒でした。

## まとめ

本文では HagiCode プロジェクトのマルチ Agent 協調設定に関する実践経験を共有しました。コア要点は以下の通りです：

1. **標準化されたインターフェース**：`IAIProvider` を通じて異なる Agent の動作を統一し、コードが下層がどの企業の製品であるかを気にしなくてもよいようにします
2. **ファクトリーモード**：`ActivatorUtilities.CreateInstance` を使用して Provider インスタンスを動的に作成し、実行時設定と依存性注入をサポートします
3. **プロトコルの統一**：ACP プロトコルは Agent 間の標準化された通信を実現し、JSON-RPC 2.0 に基づく双方向通信メカニズムです
4. **タスクの分流**：異なる Agent に合理的にタスクを割り当て、それぞれが長所を発揮できるようにし、1 つの Agent ですべてのことをさせようとするのではなく

この設計は「複数の Agent が喧嘩する」問題を解決しただけでなく、冒険団のタスクフローメカニズムを通じて、開発プロセスの自動化と専門化を実現しました。

もし複数の AI アシスタントの導入を検討しているなら、本文が何らかの参考になることを願っています。もちろん、プロジェクトの状況は異なり、具体的なソリューションは実際の状況に応じて調整する必要があります。結局、どこにでも適用できるソリューションはなく、自分に適したものが最高です。

美しい事物や人は必ずしも所有する必要はなく、彼女がまだ美しい限り、自分で彼女の美しさを好好と見ていればいいだけです。技術ソリューションも同様で、自分に適したものが最高です......
