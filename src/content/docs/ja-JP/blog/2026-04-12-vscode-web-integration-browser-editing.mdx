## ブラウザで素早くコードを編集：VSCode Web統合の実践

> AIがコードの分析を終えた後、どのようにブラウザですぐにエディターを開いて修正できるのか？本文ではHagiCodeプロジェクトでcode-serverを統合した実践経験を共有し、AIアシスタントとコード編集エクスペリエンスのシームレスな接続を実現します。

## 背景

AI支援プログラミングの時代において、開発者は頻繁にコードを素早く確認・編集する必要があります。従来の開発フローは、デスクトップIDEでプロジェクトを開き、ファイルを特定し、編集して保存するというものです。しかし、このフローはあるシナリオでは、どこか違和感があります。

シナリオ1：リモート開発。HagiCodeのようなAIアシスタントを使用する場合、バックエンドはリモートサーバーやコンテナで実行されている可能性があり、ローカルからプロジェクトファイルに直接アクセスできません。コードを確認・修正するたびにSSHなどで接続する必要があり、体験が断片化されます。まるで会いたい人がいるのに、厚いガラスを隔てて見えるだけ触れられないような感じです。

シナリオ2：クイックプレビュー。AIアシスタントがコードを分析した後、ユーザーは特定のファイルを素早く浏览したり、小幅な修正を加えたいだけかもしれません。完全なデスクトップIDEを起動するのは重厚すぎ、ブラウザ内の軽量エディタの方が「クイック確認」のニーズに合っています。結局のところ、一目見るために大げさなことはしたくないものです。

シナリオ3：クロスデバイス協業。異なるデバイスで作業する場合、ブラウザ内のエディタは統一されたアクセス入口を提供し、各デバイスで開発環境を設定する必要がありません。これは省力的で、人生は短いので、無駄な労働は避けたいものです。

これらの課題を解決するため、HagiCodeプロジェクトではVSCode Webを統合しました。AIアシスタントとコード編集エクスペリエンスをシームレスに接続—AIがコード分析を終えた後、ユーザーは同じブラウザセッションですぐにエディタを開いて修正でき、アプリを切り替える必要がありません。この体験は、言ってみれば、あなたが望むときに、そこに在るかのようなものです。

## HagiCodeについて

本文で共有するソリューションは、[HagiCode](https://hagicode.com)プロジェクトでの実践経験から来ています。HagiCodeはAI駆動のコードアシスタントであり、自然言語交互を通じて開発効率を向上させることを目的としています。開発過程で、ユーザーはAI分析とコード編集の間で頻繁に切り替える必要があることがわかり、これがエディタをブラウザに直接統合する方法を探るきっかけとなりました。

プロジェクトアドレス：[github.com/HagiCode-org/site](https://github.com/HagiCode-org/site)

## 技術選択：なぜcode-serverなのか？

数あるVSCode Webソリューションの中で、私たちはcode-serverを選択しました。この選択には、いくつかの考慮があります：

機能が完全。code-serverはVSCodeのWeb版であり、デスクトップ版の大部分の機能をサポートしています。拡張システム、スマート補完、デバッグなどが含まれます。これはユーザーがブラウザでもデスクトップ版に近い編集体験を得られることを意味します。結局のところ、機能で妥協したい人はいません。

展開が柔軟。code-serverはスタンドアロンサービスとして実行でき、Dockerコンテナ展開もサポートしており、HagiCodeのアーキテクチャに適合します。私たちのバックエンドはC#で記述され、フロントエンドはReactで、REST APIを通じてcode-serverサービスと通信します。これはブロック遊びのようなもので、各ブロックには自分の場所があります。

認証が安全。code-serverは組み込みのconnection-tokenメカニズムを備えており、不正アクセスを防ぎます。各セッションには一意のトークンがあり、認証されたユーザーのみがエディタにアクセスできることを保証します。安心感というのは、あって初めて重要さがわかるものです。

## アーキテクチャ設計

HagiCodeのVSCode Web統合は、フロントエンドとバックエンドを分離したアーキテクチャ設計を採用しています。

### フロントエンドサービス層

フロントエンドは`vscodeServerService.ts`を通じてバックエンドとの交互をカプセル化しています：

```typescript
// プロジェクトを開く
export async function openProjectInCodeServer(
  id: string,
  currentInterfaceLanguage?: string,
): Promise<VsCodeServerLaunchResponseDto>

// vaultを開く
export async function openVaultInCodeServer(
  id: string,
  path?: string,
  currentInterfaceLanguage?: string,
): Promise<VsCodeServerLaunchResponseDto>
```

この2つのメソッドの違いは、`openProjectInCodeServer`はプロジェクト全体を開くためのもので、`openVaultInCodeServer`はVaultの特定パスを開くためのものです。MonoSpecsマルチリポジトリプロジェクトの場合、システムは自動的にワークスペースファイルを作成します。役割分担が明確で、各自が自分の仕事をすれば十分です。

### バックエンドサービス層

バックエンドの`VaultAppService.cs`はコアロジックを実装しています：

```csharp
public async Task<VsCodeServerLaunchResponseDto> OpenInCodeServerAsync(
    string id,
    string? relativePath = null,
    string? currentInterfaceLanguage = null,
    CancellationToken cancellationToken = default)
{
    // 1. 設定を取得し有効かチェック
    var settings = await _vsCodeServerSettingsService.GetResolvedSettingsAsync(cancellationToken);
    if (!settings.Enabled) {
        throw new BusinessException(VsCodeServerErrorCodes.Disabled, "VSCode Server is disabled.");
    }

    // 2. vaultを取得し起動ディレクトリを解析
    var vault = await RequireVaultAsync(id, cancellationToken);
    var launchDirectory = ResolveLaunchDirectory(vault, relativePath);

    // 3. code-serverが実行されていることを保証しランタイム情報を取得
    var runtime = await _vsCodeServerManager.EnsureStartedAsync(settings, cancellationToken);

    // 4. 言語設定を解析
    var language = _vsCodeServerSettingsService.ResolveLaunchLanguage(
        settings.Language,
        currentInterfaceLanguage);

    // 5. 起動URLを構築
    return new VsCodeServerLaunchResponseDto {
        LaunchUrl = AppendQueryString(runtime.BaseUrl, new Dictionary<string, string?> {
            ["folder"] = launchDirectory,
            ["tkn"] = runtime.ConnectionToken,
            ["vscode-lang"] = language,
        }),
        ConnectionToken = runtime.ConnectionToken,
        OpenMode = "folder",
        Runtime = VsCodeServerSettingsService.MapRuntime(
            await _vsCodeServerManager.GetRuntimeSnapshotAsync(cancellationToken)),
    };
}
```

このメソッドの責務は明確です：設定チェック、パス解析、サービス起動、URL構築。そのうち`ResolveLaunchDirectory`メソッドはパスセキュリティチェックを行い、パストラバーサル攻撃を防ぎます。コードは詩のようなもので、各行には自分のリズムがあります。

### 自動ランタイム管理

バックエンドは`VsCodeServerManager`を通じてcode-serverプロセスを管理します：

- プロセス状態チェック
- 停止したサービスの自動起動
- ランタイムスナップショット返却（ポート、プロセスID、起動時間など）

この設計により、システムはcode-serverのライフサイクルを自動的に処理でき、ユーザーは手動でサービスプロセスを管理する必要がありません。人生は既に十分複雑なので、自動化できるものは自動化しましょう。

### 言語追従メカニズム

HagiCodeは多言語インターフェースをサポートしており、code-serverもこの設定に追従する必要があります。システムは3つの言語モードをサポートしています：

- `follow`：現在のインターフェース言語に追従
- `zh-CN`：中国語固定
- `en-US`：英語固定

URLパラメータ`vscode-lang`を通じてcode-serverに渡し、エディタ言語がHagiCodeインターフェースと一貫していることを保証します。言語は統一されてこそ心地よいものです。

### MonoSpecsマルチリポジトリワークスペース

MonoSpecsプロジェクト（複数のサブリポジトリを含むmono-repo）の場合、システムは自動的に`.code-workspace`ファイルを作成します：

```csharp
private async Task<string> CreateWorkspaceFileAsync(Project project, Guid projectId)
{
    var folders = await ResolveWorkspaceFoldersAsync(project.Path);
    var workspaceDocument = new {
        folders = folders.Select(path => new { path }).ToArray(),
    };
    // workspaceファイルを生成...
}
```

このように一つのcode-serverインスタンスで複数のサブリポジトリを同時に編集でき、大規模なmono-repoプロジェクトに非常に実用的です。複数のリポジトリを一つのウィンドウで編集するのは、一つの本に複数の物語があるようなものです。

## フロントエンド統合

HagiCodeフロントエンドはReact + TypeScriptを使用しており、code-server統合の手順はそれほど複雑ではありません。

### クイックアクションボタン

プロジェクトカードにCode Serverボタンを追加します：

```tsx
// QuickActionsZone.tsx
<Button
  size="sm"
  variant="default"
  onClick={() => onAction({ type: 'open-code-server' })}
>
  <Globe className="h-3 w-3 mr-1" />
  <span className="text-xs">{t('project.openCodeServer')}</span>
</Button>
```

このボタンは開くアクションをトリガーし、バックエンドAPIを呼び出して起動URLを取得します。一つのボタンが一つのアクションを担い、シンプルで直接的です。

### 開くアクションの処理

```tsx
const handleAction = async (action: ProjectAction) => {
  if (action.type === 'open-code-server') {
    const response = await openProjectInCodeServer(project.id, i18n.language);
    window.open(response.launchUrl, '_blank', 'noopener,noreferrer');
  }
};
```

`window.open`を使用して新しいタブでcode-serverを開き、`noopener,noreferrer`パラメータが安全性を保証します。セキュリティに対しては、どれだけ注意してもしすぎることはありません。

### Vault編集入口

Vaultリストに同様の編集ボタンを追加します：

```tsx
const handleEditVault = async (vault: VaultItemDto) => {
  const response = await openVaultInCodeServer(vault.id);
  window.open(response.launchUrl, '_blank', 'noopener,noreferrer');
};
```

プロジェクトとVaultは同じ開く方式を使用し、交互の一貫性を保っています。一貫性の重要性は、機能自体に劣りません。

## URL構築ロジック

code-serverのURLフォーマットも、ちょっとしたこだわりがあります：

**フォルダモード**：
```
http://{host}:{port}/?folder={path}&tkn={token}&vscode-lang={lang}
```

**ワークスペースモード**：
```
http://{host}:{port}/?workspace={workspacePath}&tkn={token}&vscode-lang={lang}
```

`tkn`は接続トークンであり、code-server起動時に自動生成され、アクセスの安全性を保証します。`vscode-lang`パラメータはエディタインターフェース言語を制御します。これらのパラメータはそれぞれ役割があり、一つも欠けられません。

## 使用シナリオ

### シナリオ1：AI支援コードレビュー

ユーザーはHagiCodeと対話し、AIはプロジェクトコードを分析して潜在的な問題を発見します。ユーザーは「Open in Code Server」ボタンをクリックしてブラウザで直接エディタを開き、問題ファイルを確認して修正し、その後HagiCodeに戻って対話を続けます。プロセス全体がブラウザで完了し、アプリを切り替える必要がありません。この感覚は、言ってみれば、水流のようにスムーズです。

### シナリオ2：Vault学習資料編集

ユーザーはあるオープンソースプロジェクトを学習するVaultを作成し、`docs/`ディレクトリに学習ノートを追加したいと考えます。code-serverを通じてブラウザで直接Markdownファイルを編集し、保存後HagiCodeは更新されたノート内容を同期的に読み取ることができます。これは個人のナレッジベースを構築するのに非常に有用です。知識は蓄積すればするほど価値が高まります。

### シナリオ3：MonoSpecsマルチリポジトリ開発

MonoSpecsプロジェクトは複数のサブリポジトリを含み、code-serverは自動的にマルチフォルダワークスペースを作成します。ユーザーはブラウザで複数のリポジトリのコードを同時に編集でき、修正後それぞれのGitリポジトリにコミットします。この作業方式は、リポジトリ間をまたぐ修正が必要なシナリオに特に適しています。複数のリポジトリを同時に修正するのは、複数のことを同時に処理するようなもので、ちょっとしたコツが必要です。

## セキュリティ考慮

code-server統合を実装する際、セキュリティは重点的に注意すべき問題です。結局のところ、セキュリティは問題が起きてからでは遅いものです。

### Connection Token

connection-tokenはランダムに生成され、漏洩すべきではありません。HTTPS環境下で使用することをお勧めし、トークンが中間者に傍受されるのを防ぎます。機密情報は保護しておくに越したことはありません。

### ファイルパスセキュリティ

バックエンドはパストラバーサルチェックを実装しています：

```csharp
private static string ResolveLaunchDirectory(VaultRegistryEntry vault, string? relativePath)
{
    var vaultRoot = EnsureTrailingSeparator(Path.GetFullPath(vault.PhysicalPath));
    var combinedPath = Path.GetFullPath(Path.Combine(vaultRoot, relativePath ?? "."));
    if (!combinedPath.StartsWith(vaultRoot, StringComparison.OrdinalIgnoreCase))
    {
        throw new BusinessException(VaultRelativePathTraversalCode, "Relative path traversal detected.");
    }
    return combinedPath;
}
```

このコードは、ユーザーが`../`などの方法でvaultディレクトリ外のファイルにアクセスできないことを保証します。境界チェックは、やらないよりやる方が良いものです。

### 権限制御

code-serverプロセスは適切なユーザー権限で実行され、システムの機密ファイルへのアクセスを避けます。専用ユーザーでcode-serverサービスを実行することをお勧めします。権限制御は、あるべきものはあるべきです。

## パフォーマンス最適化

code-serverはサーバーリソースを消費します。以下は最適化の提案です：

- CPU/メモリ使用を監視し、必要に応じてリソース制限を調整
- 大規模プロジェクトではタイムアウト時間の増加が必要な場合があります
- セッションタイムアウト自動クリーンアップを実装し、リソースを解放
- キャッシュ使用を検討し、重複計算を削減

HagiCodeはランタイム状態監視APIを提供しており、フロントエンドは`getVsCodeServerSettings()`を通じて現在の状態を取得できます：

```typescript
const { settings, runtime } = await getVsCodeServerSettings();
// runtime.status: 'disabled' | 'stopped' | 'starting' | 'running' | 'unhealthy'
// runtime.baseUrl: "http://localhost:8080"
// runtime.processId: 12345
```

この設計により、ユーザーはcode-serverの健全状態を明確に理解でき、問題発生時に迅速に位置特定できます。状態が可視化されてこそ、安心感があります。

## ユーザーエクスペリエンスの詳細

実装過程で、ユーザーエクスペリエンスに影響するいくつかの詳細を発見し、特別に注目する価値があります：

初めてcode-serverを開くとき、起動待ちが必要な場合があり、この時間は数秒から半分ほどまで様々です。フロントエンドでローディング状態を表示し、システムが処理中であることをユーザーに知らせることをお勧めします。待機することは、フィードバックがあれば良いものです。

ブラウザがポップアップをブロックする可能性があり、ユーザーに手動で許可するよう促す必要があります。HagiCodeは初回開く時にガイド情報を表示し、ユーザーにブラウザ権限の設定方法を伝えます。ユーザーエクスペリエンスは、まさにこうした詳細の中で体現されます。

ランタイム状態（起動中/実行中/エラー）を表示することをお勧めします。これによりユーザーは問題に遭遇したとき、サーバー側の問題なのか自分の操作問題なのかを知ることができます。問題がどこにあるか知っていれば、少なくとも安心できます。

## 設定例

code-serverの設定はそれほど複雑ではありません：

```json
{
  "vscodeServer": {
    "enabled": true,
    "host": "0.0.0.0",
    "port": 8080,
    "language": "follow"
  }
}
```

`enabled`は機能スイッチを制御し、`host`と`port`は待ち受けアドレスを指定し、`language`は言語モードを設定します。これらの設定はUIインターフェースから修正でき、リアルタイムで有効になります。シンプルなものこそ、最も使いやすいものです。

## まとめ

HagiCodeのVSCode Web統合は、エレガントなソリューションを提供します：AIアシスタントとコード編集エクスペリエンスをシームレスに接続します。ブラウザでcode-serverを統合することで、ユーザーはAIの分析結果に素早く応答し、同じブラウザセッションで分析から編集までの完全なプロセスを完了できます。

このソリューションのいくつかの重要な利点：一つ目は統一エクスペリエンスで、プロジェクトとVaultは同じ開く方式を使用します。二つ目はマルチリポジトリサポートで、MonoSpecsプロジェクトは自動的にワークスペースを作成します。三つ目は安全制御で、ランタイム状態監視とパスセキュリティチェックです。
