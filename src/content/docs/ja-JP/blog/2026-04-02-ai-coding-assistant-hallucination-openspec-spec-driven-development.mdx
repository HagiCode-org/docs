## AI コーディングアシスタントの幻覚問題：OpenSpec で仕様駆動開発を実現する方法

> AI コーディングアシスタントは強力ですが、実際の要件に合わない、プロジェクト規約に違反するコードを生成することがよくあります。この記事では、HagiCode プロジェクトが OpenSpec プロセスを通じて「仕様駆動開発」を実現し、構造化的な提案メカニズムで AI 幻覚リスクを大幅に削減する方法を紹介します。

## 背景

GitHub Copilot や ChatGPT でコードを書いたことのある人は、こんな経験があるかもしれません：AI が生成したコードは見た目は素晴らしいですが、実際に使ってみると問題だらけです。プロジェクト内のコンポーネントを間違って使っていたり、チームのコーディング規約を無視していたり、存在しない前提に基づいて大段のロジックを書いたりしているかもしれません。

これがいわゆる「AI 幻覚」問題です—プログラミング分野では、一見合理的に見えるが実際にはプロジェクトの実情に合わないコードを生成することとして表れます。

実はこの問題もかなり厄介です。AI コーディングアシスタントがますます普及し、この問題も深刻化しています。結局のところ、AI はプロジェクトの歴史、アーキテクチャ決定、コーディング規約への理解が不足しており、自由度が高すぎると「創造的に」実際の状況に合わないコードを生成しやすくなります。これは文章を書くのに似ていて、ルールがなければ天馬空空のように書いてしまいがちですが、実際はそうではありません。

これらの痛点を解決するために、私たちは大胆な決定を下しました：AI をより賢くしようとするのではなく、「規約」という檻をはめることにしました。この決定がもたらす変化は、想像以上に大きいかもしれません—後で詳しく説明します。

## HagiCode について

この記事で共有するソリューションは、[HagiCode](https://hagicode.com) プロジェクトでの実践経験から来ています。HagiCode はオープンソースの AI コードアシスタントプロジェクトで、構造化的なエンジニアリング実践を通じて AI プログラミングの实际问题を解決することに尽力しています。

## AI 幻覚の根源

解決策を深く掘り下げる前に、まず問題がどこにあるかを見てみましょう。敵を知り己を知れば百戦危うからず、という言葉を AI に当てはめるのも、なんだか面白いですね。

### コンテキストの欠如

AI モデルは訓練時に公開コードリポジトリを使用しますが、あなたのプロジェクトには独自の歴史、約束事、アーキテクチャ決定があります。これらの「暗黙的知識」は AI が直接取得できないため、生成されたコードはプロジェクトの実情と脱節しがちです。

これも全部 AI のせいとは言えません。結局、あなたのプロジェクトにいたこともないし、あんたたちの江湖の決まりなんて知るわけがないでしょう？ 新しく来た実習生みたいなもので、ルールを知らないのも普通ですが、代償は少し大きいかもしれません。

### 自由度が高すぎる

AI に「ユーザー認証機能を実装して」と頼むと、どんな形式のコードでも生成するかもしれません。明確な制約がないと、AI はプロジェクトの要件ではなく、自分の「合理的だと思われる」方法で実装しようとします。

これはプロジェクト規約を学んだことのない人に自由に発揮させるようなもので、問題が出ないはずがありません。実際、無責任なわけではなく、そもそも責任が何かを知らないだけです。

### 検証の欠如

AI がコードを生成した後、構造化的なレビュープロセスを経ていないと、間違った前提に基づいたコードが直接コードリポジトリに入ってしまいます。テストや本番環境で問題が発覚するまで待つと、代償が大きすぎます。

これは羊を逃してから柵を修理するようなもので、羊がもう全部逃げてしまってからでは、柵を修理しても意味がありません。この道理は誰でも分かりますが、実際に行うとなると、面倒に感じてしまいます。結局、事態が悪化する前は、誰も余分な時間を費やしたくないものです。

## OpenSpec：仕様駆動開発の答え

HagiCode は OpenSpec をソリューションとして選択しました。核心的なアイデアは：**すべてのコード変更は構造化された提案プロセスを通過しなければならない**、抽象的なアイデアを実行可能な実施計画に変換するということです。

これは高大上に聞こえますが、実際には AI がコードを書く前に、まず要件文書を書くようにするだけです。結局、準備すれば立ち、準備しなければ廃する、古人の言葉は決して嘘ではありません。

### OpenSpec とは

OpenSpec は npm ベースのコマンドラインツール（`@fission-ai/openspec`）で、標準的な提案ファイル構造と検証メカニズムを定義しています。簡単に言えば、AI にコードを書く前に「要件文書を書く」ようにさせるものです。

### 3ステッププロセスで幻覚を防ぐ

OpenSpec は3ステップのプロセスを通じて提案品質を確保します：

**ステップ1：提案の初期化** - セッション状態を Openspecing に設定
**ステップ2：中間処理** - Openspecing 状態を維持し、アーティファクトを段階的に改善
**ステップ3：提案の完了** - Reviewing 状態に変換

この設計には巧妙な詳細があります：最初のステップでは `ProposalGenerationStart` タイプを使用し、完了しても状態変換がトリガーされません。これにより、多段プロセス全体が完了する前に、早すぎてレビュー段階に入るのを防ぎます。

実はこの詳細も面白いです。料理のように、火加減が足らないのに蓋を開けると、絶対にうまくいきません。忍耐強く、一歩一歩進めてこそ、最後に良い料理が作れます。

```csharp
// HagiCode プロジェクトでの実装
public enum MessageAssociationType
{
    ProposalGeneration = 2,
    ProposalExecution = 3,

    /// <summary>
    /// 3ステップ提案生成プロセスの開始をマーク
    /// 完了しても Reviewing 状態には変換されない
    /// </summary>
    ProposalGenerationStart = 5
}
```

### 標準化されたファイル構造

各 OpenSpec 提案は同じディレクトリ構造に従います：

```
openspec/
├── changes/                    # アクティブおよびアーカイブされた変更
│   ├── {change-name}/
│   │   ├── proposal.md        # 提案記述
│   │   ├── design.md          # 設計文書
│   │   ├── specs/             # 技術仕様
│   │   └── tasks.md           # 実行可能タスクリスト
│   └── archive/               # アーカイブされた変更
└── specs/                     # 独立した仕様ライブラリ
```

HagiCode プロジェクトの統計によると、現在 4000+ のアーカイブ変更と 15 万行以上の仕様ファイルがあります。これらの歴史的な蓄積は AI に従うべきルールを提供するだけでなく、チームにも貴重な知識ベースを提供しています。

これは古人が残した典籍のようなもので、読めば読むほど自然に道理が悟れます。ただ、今この典籍は竹簡に書かれているのではなく、ファイルに保存されているだけです。

### 多層検証メカニズム

システムは提案品質を確保するために多層検証を実装しています：

```csharp
// 必須ファイルの存在を検証
ValidateProposalFiles()

// 実行前提条件を検証
ValidateExecuteAsync()

// 開始条件を検証
ValidateStartAsync()

// アーカイブ条件を検証
ValidateArchiveAsync()

// 提案名フォーマット検証（kebab-case）
ValidateNameFormat()
```

これらの検証は何層にもわたってチェックする門番のようなもので、真に合格した提案だけが通過できます。面倒に見えますが、ひどいコードがコードリポジトリに入るよりはいいでしょう？

### Prompt テンプレート制約

HagiCode で AI が実行する際、事前定義された Handlebars テンプレートを使用します。これらのテンプレートには明確なステップガイダンスと保護措置が含まれています。例えば：

- ユーザーの意図を理解していない場合は継続を禁止
- 検証されていないコードの生成を禁止
- 名前が無効な場合は再提供を要求
- 変更が既に存在する場合は、再作成ではなく継続コマンドの使用を提案

この「足枷をつけて踊る」方式は、逆に AI をニーズの理解と規約に準拠したコード生成に集中させます。制約は必ずしも悪いことではありません。結局、自由すぎると混乱しやすいものです。

## 実践：プロジェクトで OpenSpec を使用する方法

### インストールと初期化

```bash
npm install -g @fission-ai/openspec@1
openspec --version  # インストールを検証
```

プロジェクトルートディレクトリに `openspec/` フォルダ構造が自動的に作成されます。

このステップは特に説明することもありません。ツールのインストールなので、誰でも分かります。ただ、`@fission-ai/openspec@1` というバージョンを使うのを覚えておいてください。新しいバージョンにはバグがあるかもしれません。結局、安定性がすべてです。

### 提案の作成

HagiCode の対話インターフェースで、ショートカットコマンドを使用します：

```
/opsx:new
```

または、変更名と対象リポジトリを指定します：

```
/opsx:new "add-user-auth" --repos "repos/web"
```

提案を作成することは、文章を書くときにアウトラインを列挙するのと同じで、アウトラインがあれば後で書きやすくなります。ただ、多くの人は直接書き始め、書き終わってから考えが通じていないことに気づき、それこそ頭痛の種です。

### アーティファクトの生成

`/opsx:continue` を使用して必要なアーティファクトを段階的に生成します：

**proposal.md** - 変更の目的と範囲を記述
```markdown
# Proposal: Add User Authentication

## Why
現在のシステムはユーザー認証機能が不足しており、機密 API を保護できません。

## What Changes
- JWT 認証ミドルウェアを追加
- ログイン/登録 API を実装
- フロントエンド統合を更新
```

**design.md** - 詳細な技術設計
```markdown
# Design: Add User Authentication

## Context
現在公開 API を使用しており、誰でもアクセス可能...

## Decisions
1. JWT を選択、Session ではなく...
2. HS256 アルゴリズムを使用...

## Risks
- トークン漏洩リスク...
- 緩和措置...
```

**specs/** - 技術仕様とテストシナリオ
```markdown
# user-auth Specification

## Requirements

### Requirement: JWT Token Generation
システムは HS256 アルゴリズムを使用して JWT トークンを生成しなければなりません。

#### Scenario: Valid login
- WHEN ユーザーが有効な認証情報を提供
- THEN システムは有効な JWT トークンを返さなければなりません
```

**tasks.md** - 実行可能なタスクリスト
```markdown
# Tasks: Add User Authentication

## 1. Backend Changes
- [ ] 1.1 AuthController を作成
- [ ] 1.2 JWT ミドルウェアを実装
- [ ] 1.3 単体テストを追加
```

これらのアーティファクトは文章を書くときの下書きのようなもので、下書きが良く書ければ、本文も自然とスムーズになります。ただ、多くの人は下書きを書くのが好きではなく、時間の無駄だと感じていますが、実際には下書きこそが考えを整理するのに最適な場所です。

### レビューと適用

すべてのアーティファクトが完了したら：

```
/opsx:apply
```

AI はすべてのコンテキストファイルを読み取り、tasks.md のリストに従ってタスクを段階的に実行します。この時点ではっきりした規約があるため、生成されるコードの品質はずっと高くなります。

実はこの段階まで来れば、ことは半分以上成し遂げられています。明確なタスクリストがあれば、後は手順通りに実行するだけです。ただ、多くの人が前のステップをスキップして直接ここに来るため、品質は当然保証できません。

### アーカイブ

変更完了後：

```
/opsx:archive
```

完了した変更を `archive/` ディレクトリに移動し、後で参照・再利用しやすくします。

アーカイブはかなり重要です。書き終わった文章をきちんと片付けるのと同じです。後で似たような問題に遭遇したとき、以前の記録を振り返れば、答えが見つかるかもしれません。ただ、多く人は面倒でやりませんが、実際にはこれらの蓄積こそが最も貴重な財産です。

## 注意事項とベストプラクティス

### 提案命名規約

kebab-case 形式を使用し、文字で始まり、小文字、数字、ハイフンのみを含みます：
- ✅ `add-user-auth`
- ❌ `AddUserAuth`
- ❌ `add--user-auth`

命名規約は、大したことではないと言われるかもしれませんが、統一されているのは結構良いことです。結局、コードにおいて consistency は重要ですが、多く人は気にしていません。

### 一般的なエラーを回避

1. **3ステッププロセスのステップ1で間違ったタイプを使用する** - 状態が早すぎて変換される
2. **最後のステップで状態変換をトリガーするのを忘れる** - Openspecing 状態で詰まる
3. **レビューをスキップして直接実行する** - すべてのアーティファクトが完全か検証すべき

これらのエラーは初心者が犯しやすいものです。ベテランは自然に回避方法を知っています。初心者もいつかはベテランになります。遠回りしてもいいですが、あまりに遠回りしすぎないことを願っています。

### 多変更管理

OpenSpec は複数の提案を同時に管理することをサポートしており、大規模な機能を扱う際に特に便利です：

```bash
# すべてのアクティブな変更を表示
openspec list

# 特定の変更に切り替え
openspec apply "add-user-auth"

# 変更状態を表示
openspec status --change "add-user-auth"
```

多変更管理は、複数の記事を同時に書くようなもので、少しの技術と忍耐が必要です。ただ、慣れれば大丈夫です。結局、人間は何にでも適応できます。

### セッション状態マシンの理解

状態変換を理解すると問題のデバッグに役立ちます：

```
Init → Drafting → Openspecing → Reviewing → Executing → ExecutionCompleted → Completed → Archived
```

- **Openspecing**：計画生成中
- **Reviewing**：レビュー中（アーティファクトを繰り返し修正可能）
- **Executing**：実行中（tasks.md を適用）

状態マシンは、要するに一連のルールです。ルールは時には面倒ですが、それ以上に有用です。結局、ルールがなければ円も方形も作れません、と古人はずっと前に言っていました。

## まとめ

OpenSpec プロセスを通じて、HagiCode プロジェクトは AI 幻覚問題の解決において顕著な成果を上げました：

1. **幻覚を削減** - AI は構造化された規約に従う必要があり、自由にコードを生成できません
2. **品質を向上** - 多層検証により変更がプロジェクト標準に準拠することを確保
3. **協業を加速** - アーカイブされた変更は後続開発に参考を提供
4. **追跡可能性** - 各変更には完全な提案、設計、仕様、タスク記録があります

このソリューションは AI を賢くするのではなく、「規約」という檻をはめるものです。実践が証明しているように、足枷をつけて踊る方が、逆に良く踊れます。

実はこの道理は簡単で、制約は必ずしも悪いことではありません。文章を書くように、フォーマットの制約があれば、良いものを書きやすくなります。ただ、多く人は制約を嫌い、創造力を制限されていると感じていますが、実際には創造力も土壌があればこそ花を咲かせ実をつけることができます。

もし AI コーディングアシスタントを使用していて、似たような問題に遭遇したことがあるなら、OpenSpec を試してみてください。仕様駆動開発は一見ステップが増えるように見えますが、これらの事前投資はコード品質と保守効率で数倍のリターンを得られます。

結局、遅いことは時に速いことです。ただ、多く人はこの道理を理解していません......
