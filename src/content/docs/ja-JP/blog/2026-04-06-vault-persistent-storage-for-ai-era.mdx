## AI時代にプロジェクトを学ぶ方法：Vaultクロスプロジェクト永続化ストレージシステム

> AI支援開発時代において、AIアシスタントに学習リソースをより良く理解させるには？HagiCodeプロジェクトはVaultシステムを通じて統一的な、AIが理解可能な知識ストレージ抽象化レイヤーを実現し、プロジェクト学習の効率を大幅に向上させました。

## 背景

AI時代において、開発者が新しい技術やアーキテクチャを学ぶ方法は深く変化しています。「プロジェクトを学ぶ」——つまり優れたオープンソースプロジェクトのコード、アーキテクチャ、設計パターンを深く研究し学ぶこと——はすでに効率的な学習方法となっています。伝統的な本を読んだりビデオを見たりすることに比べ、高品質なオープンソースプロジェクトを直接読み実行することで、実世界のエンジニアリング実践をより速く理解できます。

しかし、この学習方法も多くの課題に直面しています。

学習資料が分散しすぎています。ノートはObsidianにあり、コードリポジトリは様々なフォルダに散らばっており、AIアシスタントの対話履歴はまた別のデータサイロとなっています。AIにプロジェクトの分析を手伝ってもらいたいとき、コードスニペットを手動でコピーし、コンテキストを整理する必要があり、このプロセスはかなり面倒です。

実はもっと面倒なのはコンテキストの断絶です。AIアシスタントはローカルの学習リソースに直接アクセスできず、対話のたびに背景情報を再提供する必要があります。また、学習しているコードリポジトリは速く更新され、手動同期はエラーになりやすく、複数の学習プロジェクト間での知識共有も困難です。

これらの問題は本質的に「データサイロ」によって引き起こされています。統一されたストレージ抽象化レイヤーがあり、AIアシスタントがすべての学習リソースを理解してアクセスできれば、問題は解決します。

## HagiCodeについて

この記事で共有するVaultシステムは、私たちが[HagiCode](https://hagicode.com)の開発過程で実践した解決策です。HagiCodeはAIコードアシスタントプロジェクトであり、日常開発で様々なオープンソースプロジェクトを学習・参照する必要がよくあります。AIアシスタントにこれらの学習リソースをより良く理解させるために、Vaultクロスプロジェクト永続化ストレージシステムを設計しました。

このソリューションはHagiCodeですでに実際に検証されています。もし同様の知識管理の問題に直面しているなら、これらの経験が何らかのインスピレーションを与えられることを願っています。毕竟、踩れた坑は、後の人に何かを残すべきですから。

## Vaultシステム設計理念

Vaultシステムの核心思想はとてもシンプルです：統一的な、AIが理解可能な知識ストレージ抽象化レイヤーを作成する。実装の観点からは、システムにはいくつかの重要な特徴があります。

### マルチタイプサポート

システムは4つのvaultタイプをサポートし、それぞれ異なる使用シナリオに対応します：

```typescript
// folder：汎用フォルダタイプ
export const DEFAULT_VAULT_TYPE = 'folder';

// coderef：コードプロジェクトの学習専用タイプ
export const CODEREF_VAULT_TYPE = 'coderef';

// obsidian：Obsidianノートソフトとの統合
export const OBSIDIAN_VAULT_TYPE = 'obsidian';

// system-managed：システム自動管理のvault
export const SYSTEM_MANAGED_VAULT_TYPE = 'system-managed';
```

その中で**coderef**タイプはHagiCodeで最もよく使われます。それはコードプロジェクトの学習専用に設計され、標準化されたディレクトリ構造とAIが読み取り可能なメタデータ記述を提供します。

### 永続化ストレージメカニズム

VaultのレジストリはJSON形式で永続化され、アプリケーション再起動後も設定が利用可能であることを保証します：

```csharp
public class VaultRegistryStore : IVaultRegistryStore
{
    private readonly string _registryFilePath;

    public VaultRegistryStore(IConfiguration configuration, ILogger<VaultRegistryStore> logger)
    {
        var dataDir = configuration["DataDir"] ?? "./data";
        var absoluteDataDir = Path.IsPathRooted(dataDir)
            ? dataDir
            : Path.GetFullPath(Path.Combine(Directory.GetCurrentDirectory(), dataDir));

        _registryFilePath = Path.Combine(absoluteDataDir, "personal-data", "vaults", "registry.json");
    }
}
```

この設計の利点はシンプルで信頼性が高いことです。JSON形式は人間が読みやすく、デバッグや手動修正に便利です。ファイルシステムストレージはデータベースの複雑さを回避し、システムの依存を低減します。やはり、時にはシンプルな方が最高なこともあります。

### AIコンテキスト統合

最も重要なのは、システムがvault情報をAI提案のコンテキストに自動注入できることです：

```typescript
export function buildTargetVaultsText(
  vaults: VaultForText[],
  template: VaultPromptTemplate = DEFAULT_VAULT_PROMPT_TEMPLATE,
): string {
  const readOnlyVaults = vaults.filter((vault) => vault.accessType === 'read');
  const editableVaults = vaults.filter((vault) => vault.accessType === 'write');

  if (readOnlyVaults.length === 0 && editableVaults.length === 0) {
    return '';
  }

  const sections = [
    buildVaultSection(readOnlyVaults, template.reference),
    buildVaultSection(editableVaults, template.editable),
  ].filter(Boolean);

  return `\n\n### ${template.heading}\n\n${sections.join('\n')}`;
}
```

こうして重要な機能を実現します：AIアシスタントは利用可能な学習リソースを自動的に理解し、ユーザーが手動でコンテキストを提供する必要がありません。これはある種の黙契と言えるでしょう。

## CodeRef Vaultの標準化構造

coderefタイプのvaultに対し、HagiCodeは標準化されたディレクトリ構造を提供します：

```
my-coderef-vault/
├── index.yaml          # vaultメタデータ記述
├── AGENTS.md           # AIアシスタントの操作ガイド
├── docs/               # 学習ノートとドキュメントの保存
└── repos/              # Gitサブモジュールを通じて学習するコードリポジトリを管理
```

vaultを作成するとき、システムはこの構造を自動的に初期化します：

```csharp
private async Task EnsureCodeRefStructureAsync(
    string vaultName,
    string physicalPath,
    ICollection<VaultBootstrapDiagnosticDto> diagnostics,
    CancellationToken cancellationToken)
{
    Directory.CreateDirectory(physicalPath);

    var indexPath = Path.Combine(physicalPath, CodeRefIndexFileName);
    var docsPath = Path.Combine(physicalPath, CodeRefDocsDirectoryName);
    var reposPath = Path.Combine(physicalPath, CodeRefReposDirectoryName);

    // 標準ディレクトリ構造を作成
    if (!Directory.Exists(docsPath))
    {
        Directory.CreateDirectory(docsPath);
    }

    if (!Directory.Exists(reposPath))
    {
        Directory.CreateDirectory(reposPath);
    }

    // AGENTS.mdガイドを作成
    await EnsureCodeRefAgentsDocumentAsync(physicalPath, cancellationToken);

    // index.yamlメタデータを作成
    await WriteCodeRefIndexDocumentAsync(indexPath, mergedDocument, cancellationToken);
}
```

この構造の設計にも考慮があります：

- **docs/**ディレクトリは学習ノートを保存し、Markdown形式でコードへの理解、アーキテクチャ分析、落とし穴の経験などを記録できます
- **repos/**ディレクトリはGitサブモジュールを通じて学習するリポジトリを管理し、コードを直接コピーしません。こうしてコードの同期を維持し、スペースも節約できます
- **index.yaml**はvaultのメタデータを含み、AIアシスタントがこのvaultの用途と内容を素早く理解できます
- **AGENTS.md**はAIアシスタント専用に書かれたガイドであり、このvaultの内容をどう処理するかを説明します

こうして組織すれば、AIもあなたの考えをより容易に理解できるかもしれません。

## システム管理の自動初期化

手動でvaultを作成することに加え、HagiCodeはシステム自動管理のvaultもサポートします：

```csharp
public async Task<IReadOnlyList<VaultRegistryEntry>> EnsureAllSystemManagedVaultsAsync(
    CancellationToken cancellationToken = default)
{
    var definitions = GetAllResolvedDefinitions();
    var entries = new List<VaultRegistryEntry>(definitions.Count);

    foreach (var definition in definitions)
    {
        entries.Add(await EnsureResolvedSystemManagedVaultAsync(definition, cancellationToken));
    }

    return entries;
}
```

システムは以下のvaultを自動的に作成・管理します：

- **hagiprojectdata**：プロジェクトデータストレージ、プロジェクトの設定と状態を保存するため
- **personaldata**：個人データストレージ、ユーザーの偏好設定を保存するため
- **hbsprompt**：プロンプトテンプレートライブラリ、よく使うAIプロンプトを管理するため

これらのvaultはシステム起動時に自動的に初期化され、ユーザーが手動で設定する必要はありません。やはり、あることはシステムに任せれば良く、人間が心配する必要はありません。

## アクセス制御メカニズム

重要な設計の1つはアクセス制御です。システムはvaultを2つのアクセスタイプに分けます：

```typescript
export interface VaultForText {
  id: string;
  name: string;
  type: string;
  physicalPath: string;
  accessType: 'read' | 'write';  // キー：読み取り専用と編集可能を区別
}
```

- **reference（読み取り専用）**：AIは分析と理解のみに使用し、内容を変更できません。参照するオープンソースプロジェクト、ドキュメントなどに適用
- **editable（編集可能）**：AIはタスクの必要性に応じて内容を変更できます。ノート、下書きなどに適用

この区別は重要です。AIにどの内容が「読み取り専用参照」で、どれが「手を加えてよいもの」かを知らせ、誤操作のリスクを回避します。やはり、自分の心血がうっかり消されてしまうのは誰も望みません。

## 実戦：Vaultの作成と使用

原理を見たところで、実際にどう使うか見てみましょう。

### CodeRef Vaultの作成

以下は完全なフロントエンド呼び出し例です：

```typescript
const createCodeRefVault = async () => {
  const response = await VaultService.postApiVaults({
    requestBody: {
      name: "React Learning Vault",
      type: "coderef",
      physicalPath: "/Users/developer/vaults/react-learning",
      gitUrl: "https://github.com/facebook/react.git"
    }
  });

  // システムは自動的に：
  // 1. Reactリポジトリをvault/repos/reactにクローン
  // 2. ノート用のdocs/ディレクトリを作成
  // 3. index.yamlメタデータを生成
  // 4. AGENTS.mdガイドファイルを作成

  return response;
};
```

このAPI呼び出しは一連の操作を完了します：ディレクトリ構造の作成、Gitサブモジュールの初期化、メタデータファイルの生成など。基本情報を提供するだけで、残りはシステムに任せます。実はこれも結構安心です。

## ベストプラクティスと注意事項

Vaultシステムを使用する過程で、私たちはいくつかの経験をまとめました。

### パス安全

システムはパスを厳格に検証し、パストラバーサル攻撃を防ぎます：

```csharp
private static string ResolveFilePath(string vaultRoot, string relativePath)
{
    var rootPath = EnsureTrailingSeparator(Path.GetFullPath(vaultRoot));
    var combinedPath = Path.GetFullPath(Path.Combine(rootPath, relativePath));
    if (!combinedPath.StartsWith(rootPath, StringComparison.OrdinalIgnoreCase))
    {
        throw new BusinessException(VaultRelativePathTraversalCode,
            "Vault file paths must stay inside the registered vault root.");
    }
    return combinedPath;
}
```

これは重要です。カスタムvaultパスの場合、パスが許可された範囲内にあることを確認する必要があります。そうしないとシステムは操作を拒否します。安全というものは、どれだけ強調しても過言ではありません。

### Gitサブモジュール管理

CodeRef vaultはGitサブモジュールの使用を推奨し、コードを直接コピーしません：

```csharp
private static string BuildCodeRefAgentsContent()
{
    return """
    # CodeRef Vault Guide

    Repositories under `repos/` should be maintained through Git submodules
    rather than copied directly into the vault root.

    Keep this structure stable so assistants and tools can understand the vault quickly.
    """ + Environment.NewLine;
}
```

こうすることにはいくつかの利点があります：コードを上流と同期させ、ディスクスペースを節約し、複数バージョンのコード管理を容易にします。やはり、誰も同じものを何度も何度も繰り返しダウンロードしたいとは思いません。

### ファイルプレビュー制限

パフォーマンス問題を防ぐため、システムはファイルサイズとタイプを制限します：

```csharp
private const int FileEnumerationLimit = 500;
private const int PreviewByteLimit = 256 * 1024;  // 256KB
```

vaultに大量のファイルや超大ファイルが含まれている場合、プレビュー機能のパフォーマンスに影響する可能性があります。この場合、バッチ処理や専用の検索ツールの使用を検討できます。やはり、あるものは大きすぎると、かえって処理しにくいものです。

### 診断情報

vaultを作成するとき診断情報が返され、デバッグに役立ちます：

```csharp
List<VaultBootstrapDiagnosticDto> bootstrapDiagnostics = [];

if (IsCodeRefVaultType(normalizedType))
{
    bootstrapDiagnostics = await EnsureCodeRefBootstrapAsync(
        normalizedName,
        normalizedPhysicalPath,
        normalizedGitUrl,
        cancellationToken);
}
```

作成が失敗した場合、診断情報を確認して具体的な原因を知ることができます。エラーが起きたら診断情報を見る、これは問題を解決する1つの方法です。

## まとめ

Vaultシステムは統一されたストレージ抽象化レイヤーを通じて、AI時代のプロジェクト学習の中核的な痛点を解決します：

- **知識集中管理**：すべての学習リソースが1か所に集中し、あちこちに散らばりません
- **AIコンテキスト自動注入**：AIアシスタントは利用可能な学習リソースを自動的に理解し、手動でコンテキストを提供する必要がありません
- **クロスプロジェクト知識再利用**：複数の学習プロジェクト間で知識を共有・再利用できます
- **標準化ディレクトリ構造**：一貫したディレクトリ構造を提供し、学習コストを低減します

このソリューションはHagiCodeプロジェクトですでに実際に検証されています。もしAI支援開発関連のツールを作っていたり、同様の知識管理問題に直面していたりするなら、これらの経験が何らかの参考になることを願っています。

実は技術ソリューションの価値はどれだけ複雑かにあるのではなく、実際の問題を解決できるかどうかにあります。Vaultシステムの核心思想はとてもシンプル——統一された、AIが理解可能な知識ストレージレイヤーを確立するだけです。しかし、まさにこのシンプルな抽象化により、私たちの開発効率はかなり向上しました。

時には、シンプルな方が最高なこともあります。やはり、複雑なものにはもっと多くの落とし穴が隠れていることが多いです......
