## Skillsbase を使って独自の Skills コレクションリポジトリを維持する理由

> 笑い話かもしれませんが、AI プログラミング時代が到来し、手元の Agent Skills はますます増えていますが、それに伴う煩わしさも増えています。この記事では、私たちがどのように skillsbase を使ってこれらの問題を解決しているかについて話したいと思います。

## 背景

AI プログラミング時代において、開発者はますます多くの Agent Skills を維持する必要があります——これらは Claude Code、OpenCode、Cursor などのコーディングアシスタントの機能を拡張するための再利用可能な命令セットです。しかし、スキルの数が増加するにつれて、現実的な問題が徐々に浮き彫りになってきました：

実際、大きな問題というわけではありません。ただ、物が多くなると、管理が面倒になるだけです。

### スキルが分散して保存され、管理コストが高い

- ローカルスキルが複数の場所に散らばっています：`~/.agents/skills/`、`~/.claude/skills/`、`~/.codex/skills/.system/` など
- 異なる場所に命名の競合が存在する可能性があります（例：`skill-creator` がユーザーディレクトリとシステムディレクトリの両方に存在する場合など）
- 統一された管理エントリーが不足しており、バックアップと移行が困難です

これはかなり煩わしいです。時々、あるスキルが一体どこにあるのか自分でも分からなくなり、物を失くしたようで、探すのが大変です。

### 標準化されたメンテナンスプロセスの欠如

- スキルの手動コピーはエラーが発生しやすく、ソースを追跡するのが困難です
- 統一された検証メカニズムがなく、スキルリポジトリの完全性を保証できません
- チームコラボレーションでは、スキルコレクションを同期して共有するのが困難です

手動操作ですから、常にエラーが発生しやすいです。結局のところ、人の記憶力には限界があり、そんなに多くの物がどこから来たのか覚えていられません。

### 再現性の要件を満たせない

- 開発マシンを変更する際、すべてのスキルを再設定する必要があります
- CI/CD 環境では、スキルリポジトリを自動的に検証して同期することができません

パソコンを変えるたびに最初からやり直す必要があります。この感覚は、どう言えばよいでしょうか、引っ越しのように面倒です。毎回新しい環境に再適応し、すべての物を再設定する必要があります。

これらの問題点を解決するために、私たちは多くのソリューションを試しました：手動コピーからスクリプトの自動化まで、ディレクトリを直接管理するからグローバルインストールしてから回収するまで。各ソリューションにはそれぞれの欠陥があり、一貫性を保証できないか、環境を汚染するか、CI で使用するのが困難です。

実際、かなりの遠回りをしました。

最終的に、私たちはよりエレガントなソリューション——skillsbase を見つけました。このソリューションの核心思想は：まずローカルにインストールして検証し、次に構造を変換してリポジトリに書き込み、最後に一時ファイルをアンインストールします。こうすることで、リポジトリの内容と実際のインストール結果が一致することを保証しながら、グローバル環境を汚染しません。

言うのは簡単ですが、実際には多くの穴を踏んでからようやく編み出しました。

## HagiCode について

この記事で共有するソリューションは、[HagiCode](https://hagicode.com) プロジェクトでの実践経験から来ています。

HagiCode は AI コードアシスタントプロジェクトであり、開発プロセスにおいて、様々なコーディング機能を拡張するために大量の Agent Skills を維持する必要があります。まさにこれらの実際のニーズが、スキルリポジトリを標準化して管理するためのツールである skillsbase を開発するきっかけとなりました。

実際、この物は突然思いついたものではなく、すべて追い込まれた結果です。スキルが増えれば、自然と管理が必要になります。管理の過程で問題に遭遇すれば、自然と解決が必要になります。一歩一歩、今日に至りました。

HagiCode に興味がある場合は、[公式サイトで詳細を確認する](https://hagicode.com)か、[GitHub](https://github.com/HagiCode-org/site)でソースコードを確認してください。

## 分析

### 技術的な課題

保守可能なスキルコレクションリポジトリを構築するには、以下の核心的な問題を解決する必要があります：

1. **名前空間の競合の統一**：複数のソースに同名のスキルが存在する場合、どのように上書きを回避するか？
2. **ソースの追跡可能性**：各スキルのソースをどのように記録し、後続の更新と監査を容易にするか？
3. **同期と検証**：リポジトリの内容と実際のインストール結果が一致していることをどのように保証するか？
4. **自動化統合**：CI/CD プロセスと統合し、自動同期と検証を実現するにはどうすればよいか？

これらの問題は一見簡単に見えますが、どれも頭が痛くなります。しかし、何をするにしても難しくはないでしょうか？

### 設計のトレードオフ

**ソリューション1：ディレクトリを直接コピー**

長所：実装が簡単
短所：`skills` CLI の実際のインストール結果との整合性を保証できない

このソリューションは、正直なところ、私たちも考えました。ただ後になって、CLI インストール時にいくつかの前処理ロジックがある可能性があり、直接コピーするとそれがスキップされることが分かりました。結果として、コピーした物と実際にインストールした物が異なり、これが問題となります。

**ソリューション2：グローバルインストール後に回収**

長所：インストールプロセスを検証できる
短所：実行環境を汚染し、CI とローカルの結果の一貫性を保つのが困難

このソリューションはさらに悪いです。グローバルインストールは環境を汚染します。さらに厄介なのは、CI 環境とローカル環境の一貫性を保つのが難しく、「ローカルでは動くが、CI では失敗する」という問題につながります。この感覚、誰か分かりますか？

**ソリューション3：ローカルインストール → 変換 → アンインストール（最終ソリューション）**

これは skillsbase が採用するソリューションです：

- まず `npx skills` を通じてスキルを一時的な場所にインストールします
- ディレクトリ構造を変換し、ソースメタデータを追加します
- ターゲットリポジトリに書き込みます
- 最後に一時ファイルをアンインストールします

このソリューションは、リポジトリの内容と実際の消費者のインストール結果が一致することを保証しながら、グローバル環境を汚染せず、変換プロセスを標準化でき、べき等操作をサポートします。

実際、このソリューションも最初から思い付いたものではなく、試行錯誤を繰り返した結果、自然と何が実行可能で何が不可能か分かってきました。

### アーキテクチャの決定

| 決定項目 | 選択 | 理由 |
|--------|------|------|
| ランタイム | Node.js ESM | ビルドステップが不要、`.mjs` でファイルシステムのオーケストレーションを完了できる |
| 設定形式 | YAML (`sources.yaml`) | 可読性が高く、手動メンテナンスをサポート |
| 命名戦略 | 名前空間プレフィックス | ユーザースキルは元の名前を保持し、システムスキルには `system-` プレフィックスを追加 |
| ワークフロー | `add` でマニフェストを変更 → `sync` で同期を実行 | 単一の同期エンジン、ルールの二重実装を回避 |
| ファイル管理 | 管理対象ファイルの識別 | コメントヘッダーを追加し、安全な上書きをサポート |

これらの決定は、結局のところ一つの目標のためです：物事を簡単にすること。結局のところ、シンプルさが王道です。

## 解決

### CLI アーキテクチャ

`skillsbase` CLI は4つのコアコマンドを提供します：

```text
skillsbase
├── init          # リポジトリ構造を初期化
├── sync          # スキルコンテンツを同期
├── add           # 新しいスキルを追加
└── github_action # GitHub Actions 設定を生成
```

コマンドは多くありませんが、十分です。結局のところ、ツールというのは、使えれば十分です。

### コアワークフロー

```
┌─────────────┐    ┌─────────────┐    ┌─────────────┐    ┌─────────────┐
│    init     │───▶│    add      │───▶│    sync     │───▶│github_action│
│  リポジトリ初期化 │    │  ソース追加   │    │  コンテンツ同期 │    │  CI 生成    │
└─────────────┘    └─────────────┘    └─────────────┘    └─────────────┘
```

一歩一歩進め、急いではいけません。

### 同期プロセスの設計

```
sources.yaml → ソース解析 → npx skills インストール → 構造変換 → skills/ に書き込み → 一時ファイルをアンインストール
                              ↓
                        .skill-source.json (ソースメタデータ)
```

このプロセスの設計はかなり明確です。少なくとも私が見るとき、各ステップで何をしているのか理解できます。

### リポジトリ構造

```
repos/skillsbase/
├── sources.yaml              # ソースマニフェスト（単一の真実のソース）
├── skills/                   # スキルディレクトリ
│   ├── frontend-design/      # ユーザースキル
│   ├── skill-creator/        # ユーザースキル
│   └── system-skill-creator/ # システムスキル（プレフィックス付き）
├── scripts/
│   ├── sync-skills.mjs       # 同期スクリプト
│   └── validate-skills.mjs   # 検証スクリプト
├── docs/
│   └── maintainer-workflow.md # メンテナー向けドキュメント
└── .github/
    ├── workflows/
    │   └── skills-sync.yml   # CI ワークフロー
    └── actions/
        └── skillsbase-sync/
            └── action.yml     # 再利用可能な Action
```

ファイルは少し多めですが、まあ良しとしましょう。結局のところ、組織構造が明確になれば、メンテナンスも便利になります。

## 実践

### スキルリポジトリの初期化

```bash
# 1. 空のリポジトリを作成
mkdir repos/myskills && cd repos/myskills
git init

# 2. skillsbase を使用して初期化
npx skillsbase init

# 出力：
# [1/4] create manifest ................. done
# [2/4] create scripts .................. done
# [3/4] create docs ..................... done
# [4/4] create github workflow .......... done
#
# next: skillsbase add <skill-name>
```

このステップでは大量のファイルが生成されますが、心配しないでください。すべて自動生成されます。次にスキルの追加を開始できます。

### スキルの追加

```bash
# 単一のスキルを追加（自動的に同期が実行されます）
npx skillsbase add frontend-design --source vercel-labs/agent-skills

# ローカルソースに追加
npx skillsbase add documentation-writer --source /home/user/.agents/skills

# 出力：
# source: first-party ......... updated
# target: skills/frontend-design ... synced
# status: 1 skill added, 0 removed
```

スキルの追加は非常に簡単で、コマンド一つで十分です。ただ、時々予期しない状況に遭遇することがあります。例えば、ネットワークが悪い、権限の問題など。ただ、これらはすべて些細なことなので、焦らず進めてください。

### スキルの同期

```bash
# 同期を実行（すべてのソースを照合）
npx skillsbase sync

# ドリフトのチェックのみ（ファイルは変更しない）
npx skillsbase sync --check

# 欠落したソースを許可（CI シナリオ）
npx skillsbase sync --allow-missing-sources
```

同期時、システムは `sources.yaml` で定義されたソースをすべてチェックし、`skills/` ディレクトリの内容と照合します。差分があれば更新し、差分がなければスキップします。こうすることで「設定を変更したがファイルが変わっていない」という問題を回避できます。

### CI 設定の生成

```bash
# workflow を生成
npx skillsbase github_action --kind workflow

# action を生成
npx skillsbase github_action --kind action

# すべてを生成
npx skillsbase github_action --kind all
```

CI 設定も自動生成されます。ただ、トリガー条件、実行環境などの詳細を自分で調整する必要があります。ただ、これらも難しくありません。

### sources.yaml 設定例

```yaml
# スキルルートディレクトリの設定
skillsRoot: skills/
metadataFile: .skill-source.json

# ソース定義
sources:
  # ファーストパーティ：ローカルユーザースキル
  first-party:
    type: local
    path: /home/user/.agents/skills
    naming: original  # 元の名前を保持
    includes:
      - documentation-writer
      - frontend-design
      - skill-creator

  # システム：システム提供スキル
  system:
    type: local
    path: /home/user/.codex/skills/.system
    naming: prefix-system  # system- プレフィックスを追加
    includes:
      - imagegen
      - openai-docs
      - skill-creator  # system-skill-creator になります

  # リモート：サードパーティリポジトリ
  vercel:
    type: remote
    url: vercel-labs/agent-skills
    naming: original
    includes:
      - web-design-guidelines
```

この設定ファイルはシステム全体の核心です。すべてのソースがここで定義され、ここを変更すると、次回の同期時に有効になります。したがって、これは「単一の真実のソース」と言えます。

### .skill-source.json メタデータ例

```json
{
  "source": "first-party",
  "originalPath": "/home/user/.agents/skills/documentation-writer",
  "originalName": "documentation-writer",
  "targetName": "documentation-writer",
  "syncedAt": "2026-04-07T00:00:00.000Z",
  "version": "unknown"
}
```

各スキルディレクトリの下にこのファイルがあり、そのソース情報を記録しています。こうすれば、後で問題が発生したとき、どこから来たのか、いつ同期されたのかを迅速に特定できます。

### セキュリティと検証

```bash
# リポジトリ構造を検証
node scripts/validate-skills.mjs

# skills CLI を使用して検証
npx skills add . --list

# 更新をチェック
npx skills check
```

検証は、重要と言えば重要、不要と言えば不要です。ただ、保険のために、時々実行しても悪くありません。結局のところ、何か予期しないことが起こるかわからないではありませんか。

### GitHub Actions の統合

```yaml
# .github/workflows/skills-sync.yml
name: Skills Sync

on:
  push:
    paths:
      - 'sources.yaml'
      - 'skills/**'
  workflow_dispatch:

jobs:
  validate:
    runs-on: ubuntu-latest
    steps:
      - uses: actions/checkout@v4
      - uses: actions/setup-node@v4
        with:
          node-version: '20'
      - name: Validate repository
        run: |
          npx skills add . --list
          node scripts/validate-skills.mjs
      - name: Sync check
        run: npx skillsbase sync --check
```

CI と統合すると、`sources.yaml` または `skills/` ディレクトリを変更するたびに、検証が自動的にトリガーされます。こうすれば、「ローカルで変更したが同期を忘れた」という問題を回避できます。

### ベストプラクティス

1. **命名競合の処理**：システムスキルには一律 `system-` プレフィックスを追加します。こうすることで、すべてのスキルを保持しながら、命名競合を回避できます。
2. **べき等操作**：すべてのコマンドは繰り返し実行をサポートしており、`sync` を何度実行しても副作用は発生しません。これは CI で特に重要です。
3. **管理対象ファイル**：生成されたファイルには `# Managed by skillsbase CLI` コメントが含まれており、識別と管理に便利です。これらのファイルは安全に上書きでき、手動での変更は保持されません。
4. **非インタラクティブモード**：CI 環境はデフォルトで決定論的な動作を使用し、インタラクティブなプロンプトによって中断されることはありません。すべての設定は `sources.yaml` を通じて宣言されます。
5. **ソースの追跡可能性**：各スキルには `.skill-source.json` があり、ソース情報を記録しているため、問題が発生したときに迅速に特定できます。

### チームコラボレーション

```bash
# チームメンバーが共有スキルライブラリをインストール
npx skills add your-org/myskills -g --all

# ローカルクローンで検証
git clone https://github.com/your-org/myskills.git
cd myskills
npx skills add . --list
```

Git を使用してスキルリポジトリを管理することで、チームメンバーはスキルコレクションを簡単に同期でき、全員が同じバージョンのツールと設定を使用していることを保証できます。

これはチームコラボレーションで特に有用で、「自分の環境では動くが、相手の環境では動かない」という状況を回避できます。結局のところ、環境が統一されれば、問題は半減します。

## まとめ

`skillsbase` を使用してスキルコレクションリポジトリを維持する核心的価値は以下の通りです：

- **安全性**：ソース検証、競合検出、管理対象ファイルの保護
- **保守性**：統一されたエントリー、べき等操作、設定はドキュメント
- **標準化**：統一されたディレクトリ構造、命名規則、メタデータ形式
- **自動化**：CI/CD 統合、自動同期、自動検証

このソリューションにより、開発者は npm パッケージを管理するように独自の Agent Skills を管理し、再現可能、共有可能、保守可能なスキルリポジトリシステムを実現できます。

この記事で共有したツールとプロセスは、まさに私たちが HagiCode を開発する過程で実際に穴を踏み、実際に最適化したソリューションです。このソリューションに価値があると感じる場合、私たちのエンジニアリング実践の方向が正しいことを意味します——では、HagiCode 自体も注目する価値があります。

結局のところ、良いツールはより多くの人に使われるべきです。
