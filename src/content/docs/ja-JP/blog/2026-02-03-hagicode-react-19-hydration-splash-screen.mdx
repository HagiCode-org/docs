## HagiCode のために 12 種類の究極の起動体験をデザインする：ミニマルからサイバーパンクまで

> React 19 アプリケーションのダウンロードと Hydration の一瞬の隙間は、ユーザーにブランドの個性を感じてもらう黄金の機会です。この記事では、HagiCode プロジェクトで HTML/CSS/JS に基づいて構築した完全な起動スタイルシステムを共有します。


## 背景

HagiCode は ASP.NET Core 10 と React 19 (Vite) に基づくモダンなアプリケーションとして、フロントエンドとバックエンドを分離して展開するアーキテクチャを採用しています。フロントエンドのビルド成果物はバックエンドの `wwwroot/` ディレクトリに配置され、ASP.NET Core によってホストされます。

しかし、このアーキテクチャには古典的なユーザー体験の痛みがあります。ユーザーが Web ページにアクセスすると、ブラウザはまず HTML を読み込み、次に巨大な JS バンドルをダウンロードし、最後に React が Hydration（注水）を実行する必要があります。この数百ミリ秒から数秒の「真空期間」で、ユーザーは真っ白な画面、または活気のない静的ページに直面することになります。

この隙間を埋め、HagiCode のブランド個性を注入するために、`index.html` のインラインコードのみに基づいた起動スタイルシステムを設計する必要があります。

## HagiCode について

この記事で共有する起動画面のデザイン案は、[HagiCode](https://github.com/HagiCode-org/site) プロジェクトでの実践経験から来ています。AI コードアシスタントとして、HagiCode はコード生成の効率だけでなく、開発者の視覚体験も同様に重視しています。この起動システムは、究極のフロントエンドパフォーマンスを追求する過程で生まれた産物です。

## 中核的な課題とアーキテクチャ設計

デザインに着手する前に、まず技術的制約を明確にする必要があります。`index.html` でインライン実装するということは、React 自体のバンドルを除いて、外部の CSS または JS ファイルを一切読み込めないことを意味します。

### 技術的制約の分析

1.  **ゼロ依存の原則**：すべてのスタイルは `<style>` タグ内に、ロジックは `<script>` タグ内に記述する必要があります。
2.  **防御的 CSS**：React アプリがマウントされた後、グローバルスタイルが起動画面に汚染されるのを防ぐため、高優先度の ID プレフィックス（`#boot-screen` など）ですべての起動スタイルをラップすることにしました。
3.  **パフォーマンス優先**：アニメーションは可能な限り CSS の `transform` と `opacity` を使用し、再配置をトリガーしてメインスレッドをブロックしないようにします。
4.  **視覚的一貫性**：色とフォントは HagiCode の Tailwind 設定と一致している必要があります。

### アーキテクチャパターン：Shell & Injector

**バリアントパターン**を採用しました。中核的なロジックは即時実行関数（IIFE）にカプセル化され、具体的なレンダリングロジックは設定項目として注入されます。これにより、DOM 操作ロジックを繰り返し記述することなく、単純な設定で異なるスタイルを切り替えることができます。

以下が中核的なアーキテクチャコードです：

```html
<!-- index.html にインライン -->
<div id="boot-root"></div>

<script>
(function() {
  const BootSequence = {
    config: {
      theme: 'terminal', // 'minimal', 'skeleton', 'code-rain' などに設定可能
      color: '#3b82f6'   // ブランドカラー
    },
    
    // 中核的なライフサイクル
    init() {
      this.render();
      this.listenForMount();
    },

    // 選択されたスタイルをレンダリング
    render() {
      const root = document.getElementById('boot-root');
      if (this.variants[this.config.theme]) {
        root.innerHTML = this.variants[this.config.theme].render();
      }
    },

    // React のマウント成功を監視し、優雅に終了
    listenForMount() {
      window.addEventListener('hagicode:ready', () => {
        const screen = document.getElementById('boot-root');
        // まずフェードアウトし、その後 DOM を削除してチラつきを回避
        screen.style.opacity = '0';
        screen.style.transition = 'opacity 0.3s ease';
        setTimeout(() => screen.remove(), 300);
      });
    },

    // 12種類のスタイルの実装ロジックがここに集中
    variants: {
      // ...具体的な実装は以下を参照
    }
  };

  BootSequence.init();
})();
</script>
```

## 12 種類の起動スタイルデザインリスト

この 12 種類のスタイルを 6 つのカテゴリーに分け、異なるシーンと美的ニーズを満たすようにしました。

### A. ミニマリズム

> 「 Less is More（より少ないことは、より豊かなこと）」。究極の読み込み速度を追求するシーンのために、最も軽量なソリューションを提供します。

#### 1. Minimalist Dot（ミニマルドット）
画面の中心にはシンプルな点が一つだけあり、呼吸アニメーションと連動します。
*   **実装**：CSS `@keyframes` で scale と opacity を制御します。
*   **適用**：ページを絶対的にクリーンに保つ必要があるあらゆる場面。

#### 2. Brand Reveal（ブランド揭示）
SVG の `stroke-dasharray` アニメーションを通じて、手描きのように HagiCode のロゴのラインを描き、その後テキストをフェードインさせます。
*   **テクニック**：SVG パスアニメーションを使用し、非常に質感があります。

### B. スケルトン画面模倣

> 「目を欺く芸術」。実際の UI レイアウトをシミュレートすることで、ユーザーにページがすでに半分読み込まれたと感じさせます。

#### 3. Sidebar Chat Skeleton（サイドバースケルトン画面）
これが最も実用的なものかもしれません。HTML で React コンポーネント `Sidebar` と `ChatInput` と全く同じレイアウトを手動で構築し、灰色のストライプアニメーションを上書きします。
*   **価値**：React の hydration が完了すると、スケルトン画面が瞬時にリアルなコンポーネントに変わり、ユーザーはほとんど切り替えを感じません。

#### 4. Card Stack Skeleton（カードスタック）
提案カードの読み込み時のスタックアニメーションをシミュレートし、3D 変換を使用してカードをわずかに浮遊させます。

### C. 抽象とアート

> HagiCode のギークの遺伝子を展示します。

#### 5. Geometric Morph（幾何学的変形）
画面の中心に幾何学的な形状（正方形）をレンダリングし、時間の経過とともに滑らかに円、三角形に変形し、最後にロゴになります。
*   **技術**：CSS `border-radius` の滑らかな遷移。

#### 6. Code Rain（コードの雨）
『マトリックス』に敬意を表します。JetBrains Mono フォントを使用し、背景に淡い文字の流れを落とします。
*   **注意**：パフォーマンスのため、文字の流れは小さな領域に制限するか、リフレッシュ頻度を下げる必要があります。

#### 7. Neon Pulse（ネオンパルス）
サイバーパンクスタイルの発光リングで、`box-shadow` の多重重ね合わせを使用して強い発光感を生み出します。

### D. ブランドとテーマ

> システムを「生きている」感じにさせます。

#### 8. Seasonal Theme（季節テーマ）
これは動的ローダーです。現在の日付に基づいて祝日（春節、クリスマスなど）を判断し、対応する SVG アニメーションを読み込みます。
*   **例**：春節の際、画面の下部で赤い提灯がゆらりと揺れます。

#### 9. Gradient Flow（グラデーションフロー）
背景には HagiCode のブランドカラーの流体グラデーションを使用し、`background-size` と `background-position` のアニメーションと組み合わせて、オーロラのような流動感を作り出します。

### E. 技術感

> 開発者に敬意を表します。

#### 10. Terminal Boot（ターミナル起動）
コンソール出力をシミュレートします。コードが一行一行高速にスクロールします：
```text
> Initializing HagiCode Core...
> Loading models...
> Connecting to neural network...
```
これはすべての開発者に親近感を与えます。

#### 11. Progress Bar（ミニマルプログレスバー）
画面の上部には細いプログレスバーが一本あり、右側にパーセンテージが表示されます。実際のダウンロード進捗を取得することはできませんが、タイマーを使用して「信頼できる」読み込みプロセスをシミュレートできます（最初の 80% は高速、最後の 20% は減速）。

### F. 創造性

#### 12. Pixel Assembly（ピクセルアッセンブリ）
これは非常に興味深いアイデアです。画面にはいくつかのブロックが散らばっており、それらが中心に集まり、徐々に HagiCode のロゴアイコンを組み上げます。コードの構築プロセスを象徴しています。

## ベストプラクティスと落とし穴のまとめ

HagiCode の実際の開発では、いくつかの極めて重要な実践的な詳細をまとめました。

### 1. 防御的 CSS は必須
プレフィックスを書かない手抜きはしないでください。かつて、起動画面のスタイルに ID 制限を加えなかったことがあり、React マウント後のグローバル `div` スタイルが予期せず起動画面に影響し、レイアウトが崩壊することがありました。
**経験**：すべての CSS セレクターは `#boot-screen` の下に配置し、`!important` を使用して優先度を上げます（起動画面の CSS のみ）。

### 2. 優雅な遷移
React のマウントが成功した後、起動画面の DOM を直接 `remove()` しないでください。
**正しい方法**：
1.  React が `window.dispatchEvent(new Event('hagicode:ready'))` をトリガーします。
2.  起動画面がイベントを検知し、まず `opacity: 0` を設定します。
3.  300ms（CSS トランジション時間）待ち、ユーザーに見えなくなったことを確認してから `.remove()` を実行します。

### 3. テーマ変数の同期
起動画面のカラーコードは `index.html` にハードコードされています。Tailwind のメインカラーを変更した場合、ここも同期して変更する必要があります。
**最適化案**：Vite のビルドスクリプトで、単純なプラグインを作成し、`tailwind.config.js` を読み取ってカラーバリアブルを `index.html` のテンプレート変数に注入し、単一データソースを実現します。

### 4. フォントのプリロード
起動画面は通常ブランドフォントを使用する必要がありますが、フォントの読み込みが遅いと、FOUT (Flash of Unstyled Text) が発生します。
**解決策**：`<head>` に `<link rel="preload" href="/fonts/JetBrainsMono.woff2" as="font" type="font/woff2" crossorigin>` を追加します。これは体験を向上させる低コストで高リターンの手段です。

### 5. パフォーマンス監視
`index.html` の下部に `performance.mark('boot-start')` を注入し、React のマウント成功時に `boot-end` をマークします。
**意義**：Application Insights を通じてこれらのデータを収集することで、起動画面がユーザーの知覚待ち時間（Perceived Loading Time）をどの程度短縮したかを実際に確認できます。データによると、優れたスケルトン画面はユーザーの「低速ネットワーク」に対する許容度を 50% 以上向上させます。

## まとめ

良い起動画面は単なる「待機時の装飾」ではなく、製品とユーザーの最初のインタラクションの握手シグナルです。HagiCode プロジェクトでは、この **Variants パターン** に基づいた起動システムにより、異なる祝日や異なるバージョン間でスタイルを柔軟に切り替えることができ、製品の面白さと専門性を大幅に高めています。

この記事で共有したソリューションは完全にネイティブ Web 標準に基づいており、重い依存関係は一切導入していません。これはまさに HagiCode が「軽量かつ強力」を追求する体現です。このソリューションに価値を感じていただけたなら、HagiCode リポジトリでソースコードの実装を見たり、創造的なデザインを貢献したりしてください！
