## Webインターフェースで直接DESIGN.mdを編集：アイデアから実装へ

> MonoSpecs プロジェクト管理システムでは、DESIGN.md がプロジェクトのアーキテクチャ設計と技術的決定を担っています。しかし、従来の編集方法ではユーザーは外部エディタに切り替える必要があり、この断絶したワークフローは、まあ言うなら、詩を読んでいる時に突然中断されるようなものです——インスピレーションも消え、気分も失せてしまいます。本文では、HagiCode プロジェクトで実践したソリューションを紹介します：Web インターフェースで直接 DESIGN.md を編集し、オンラインデザインサイトからテンプレートをインポートする機能をサポートします。結局のところ、誰が一気呵成に成し遂げる心地よさを好まないでしょうか？

## 背景

`DESIGN.md` はプロジェクト設計ドキュメントの中核として、アーキテクチャ設計、技術的決定、実装ガイドなどの重要な情報を担っています。しかし、従来の編集方法では、ユーザーは VS Code などの外部エディタに切り替え、物理パスを手動で特定してから編集する必要があります。このプロセスは複雑とは言えませんが、数回繰り返すうちに、人間も疲れてしまうものです。

具体的な問題は以下の側面で表れています：

- **ワークフローの断絶**：ユーザーは Web 管理インターフェースとローカルエディタ間を頻繁に切り替える必要があり、ワークフローの連続性が損なわれます——音楽を聴いている時に突然ネットが切れたようなもの、リズムが全て乱れてしまいます。
- **再利用の困難**：デザインサイトは豊富なデザインテンプレートライブラリを公開していますが、プロジェクト編集プロセスに直接統合できません。優れたものがあるのに使えない、これは確かに少し残念な感じです。
- **体験の欠如**：「プレビュー-選択-インポート」のクローズドループが欠けており、ユーザーは手動でコピペする必要があり、エラーリスクが増加します。手動操作の回数が増えれば、エラーの機会も自然と増えます。
- **協力の障害**：設計ドキュメントとコード実装の同期維持が高摩擦になり、チーム協力の効率が阻害されます。チーム協力はもともと容易ではありません、なぜさらなる障害を追加するのでしょうか？

これらの課題を解決するため、Web インターフェースで DESIGN.md の直接編集機能を実装し、オンラインデザインサイトからワンクリックでテンプレートをインポートできるようにすることを決めました。これは天地を揺るがすような決定ではありませんが、ただ開発体験をもう少しスムーズにしたいだけです。

## HagiCode について

本文で紹介するソリューションは、[HagiCode](https://hagicode.com) プロジェクトでの実践経験から来ています。HagiCode は AI 駆動のコードアシスタントプロジェクトであり、開発プロセスにおいて、プロジェクトの設計ドキュメントを頻繁に維持する必要があります。チームがより効率的に協力できるように、私たちはこのオンライン編集・インポートソリューションを探索し、実装しました。特に変わったことはありません、ただ問題に遭遇し、解決策を考えただけです。

## 技術ソリューション

### 全体アーキテクチャ

このソリューションは、フロントエンドとバックエンドを分離したSame-Originプロキシアーキテクチャを採用しており、主に以下のいくつかの層で構成されています。このアーキテクチャの設計は、まあ言うなら「各司其職」の四文字に過ぎません：

**1. フロントエンドエディタ層**
```typescript
// コアコンポーネント：DesignMdManagementDrawer
// 位置：repos/web/src/components/project/DesignMdManagementDrawer.tsx
// 機能：編集、保存、バージョン競合検出、インポートプロセスを担当
```

**2. バックエンドサービス層**
```csharp
// コアサービス：ProjectAppService.DesignMd
// 位置：repos/hagicode-core/src/PCode.Application/ProjectAppService.DesignMd.cs
// 機能：パス解決、ファイル読み書き、バージョン管理
```

**3. Same-Origin プロキシ層**
```csharp
// プロキシサービス：ProjectAppService.DesignMdSiteIndex
// 位置：repos/hagicode-core/src/PCode.Application/ProjectAppService.DesignMdSiteIndex.cs
// 機能：デザインサイトリソースのプロキシ、プレビュー画像キャッシュ、セキュリティ検証
```

### 重要な技術的決定

#### 決定 1：グローバルドロワーモード

単一のグローバルドロワーを採用し、`layoutSlice` で状態を管理することで、クロスビュー（classic/kanban）で一貫した体験を実現しました。この方法により、ユーザーがどのビューでエディタを開いても、統一されたインタラクション体験を得られるようにしています。結局、一貫した体験はユーザーをより快適に感じさせ、ビューを変えても方向性を失わなくて済みます。

#### 決定 2：プロジェクトスコープ API

DESIGN.md 関連インターフェースを `ProjectController` の下に配置し、既存のプロジェクト権限境界を再利用することで、独立したコントローラを新規追加する複雑さを回避しました。この設計の利点は、権限管理がより明確になり、RESTful のリソース編成原則にも適合していることです。時には、新規作成よりも再利用の方が意味があります、そう思いませんか？

#### 決定 3：バージョン競合検出

ファイルシステムの `LastWriteTimeUtc` に基づいて opaque version を派生させ、軽量な楽観的同時実行制御を実現しました。複数のユーザーが同時に同一ファイルを編集する場合、システムは競合を検出し、ユーザーにリフレッシュを促します。この設計は、ユーザーの編集操作をブロックすることなく、データの一貫性を保証します——人間関係における境界感のように、過度に疎遠でもなく、越境でもない、そんな感じです。

#### 決定 4：Same-Origin プロキシモード

`IHttpClientFactory` を通じて外部デザインサイトリソースをプロキシすることで、クロスオリジン問題と SSRF リスクを回避しました。この設計はセキュリティを保証すると同時に、フロントエンド呼び出しも簡素化します。セキュリティというのは、いくらやっても足りないものです。結局、データセキュリティは健康のようなもので、失ってから後悔しても遅いのです。

## コア実装

### 1. DESIGN.md の直接編集

#### バックエンド実装

バックエンドは主にパス解決、ファイル読み書き、バージョン管理を担当します。これらの作業は基本的ですが不可欠であり、家の基礎のようなものです：

```csharp
// パス解決とセキュリティ検証
private Task<string> ResolveDesignDocumentDirectoryAsync(string projectPath, string? repositoryPath)
{
    if (string.IsNullOrWhiteSpace(repositoryPath))
    {
        return Task.FromResult(Path.GetFullPath(projectPath));
    }
    return ValidateSubPathAsync(projectPath, repositoryPath);
}

// バージョン番号生成（ファイルシステムタイムスタンプとサイズに基づく）
private static string BuildDesignDocumentVersion(string path)
{
    var fileInfo = new FileInfo(path);
    fileInfo.Refresh();
    return string.Create(
        CultureInfo.InvariantCulture,
        $"{fileInfo.LastWriteTimeUtc.Ticks:x}-{fileInfo.Length:x}");
}
```

バージョン番号の設計も実に面白いものです。ファイルの最終更新時刻とサイズを使って一意のバージョン識別子を生成します。これなら軽量で信頼性が高く、追加のバージョンデータベースを維持する必要がありません。シンプルなものは、往々にしてより有効です、そう思いませんか？

#### フロントエンド実装

フロントエンドはダーティ状態検出と保存ロジックを実装しました。この設計により、ユーザーは自分の変更が保存されているかを常に知ることができ、「万一紛失したらどうしよう」という不安を軽減します：

```typescript
// ダーティ状態検出と保存ロジック
const [draft, setDraft] = useState('');
const [savedDraft, setSavedDraft] = useState('');
const isDirty = draft !== savedDraft;

const handleSave = useCallback(async () => {
    const result = await saveProjectDesignMdDocument({
        ...activeTarget,
        content: draft,
        expectedVersion: document.version, // 楽観的同時実行制御
    });
    setSavedDraft(draft); // 保存済み状態を更新
}, [activeTarget, document, draft]);
```

この実装では、2 つの状態を維持しています：`draft` は現在編集中の内容、`savedDraft` は保存された内容です。両者を比較することで、未保存の変更があるかを判断します。この設計はシンプルですが、安心感を与えてくれます。結局、誰が苦労して書いたものが突然消えるのを望むでしょうか？

#### インデックスデータ形式

デザインサイトのインデックスファイルは、利用可能な全テンプレートを定義します。このインデックスがあれば、ユーザーはレストランで注文するのように、簡単に望みのテンプレートを選択できます：

```json
{
  "entries": [
    {
      "slug": "linear.app",
      "title": "Linear Inspired Design System",
      "summary": "AI 製品 / ダーク感",
      "detailUrl": "/designs/linear.app/",
      "designDownloadUrl": "/designs/linear.app/DESIGN.md",
      "previewLightImageUrl": "...",
      "previewDarkImageUrl": "..."
    }
  ]
}
```

各エントリは、テンプレートの基本情報とダウンロードリンクを含みます。バックエンドはこのインデックスから利用可能なテンプレートリストを読み取り、ユーザーに選択肢として提示します。この設計により、選択が直感的になり、暗闇の中で手探りする必要がなくなります。

#### Same-Origin プロキシ実装

セキュリティを保証するため、バックエンドはデザインサイトへのアクセスに対して厳格な検証を行います。セキュリティというのは、いくら注意しても足りないものです：

```csharp
// 安全な slug 検証
private static readonly Regex SafeDesignSiteSlugRegex =
    new("^[A-Za-z0-9](?:[A-Za-z0-9._-]{0,127})$", RegexOptions.Compiled);

private static string NormalizeDesignSiteSlug(string slug)
{
    var normalizedSlug = slug?.Trim() ?? string.Empty;
    if (!IsSafeDesignSiteSlug(normalizedSlug))
    {
        throw new BusinessException(
            ProjectDesignSiteIndexErrorCodes.InvalidSlug,
            "Design site slug must be a single safe path segment.");
    }
    return normalizedSlug;
}

// プレビュー画像キャッシュ（OS 一時ディレクトリ）
private static string ComputePreviewCacheKey(string slug, string theme, string previewUrl)
{
    var raw = $"{slug}|{theme}|{previewUrl}";
    var bytes = SHA256.HashData(Encoding.UTF8.GetBytes(raw));
    return Convert.ToHexString(bytes).ToLowerInvariant();
}
```

ここでは 2 つのことを行っています：一是正規表現で slug のフォーマットを厳格に検証し、パストラバーサル攻撃を防止します。二是プレビュー画像をキャッシュし、外部サイトへのリクエスト圧力を軽減します。前者は防御、後者は最適化、どちらも欠かせません。

### 3. 完全なインポートプロセス

```typescript
// 1. インポートドロワーを開く
const handleRequestImportDrawer = useCallback(() => {
    setIsImportDrawerOpen(true);
}, []);

// 2. 選択してインポート
const handleImportRequest = useCallback((entry) => {
    if (isDirty) {
        setPendingImportEntry(entry);
        setConfirmMode('import'); // 上書き確認
        return;
    }
    void executeImport(entry);
}, [isDirty]);

// 3. インポート実行
const executeImport = useCallback(async (entry) => {
    const result = await getProjectDesignMdSiteImportDocument(
        activeTarget.projectId,
        entry.slug
    );
    setDraft(result.content); // エディタテキストのみ置換、自動保存はしない
    setIsImportDrawerOpen(false);
}, [activeTarget?.projectId]);
```

インポートプロセスの設計は「ユーザー確認」の原則に従っています：インポート後はエディタの内容のみを更新し、自動保存はしません。ユーザーはインポートされた内容を確認し、問題なければ手動で保存できます。結局、ユーザーが自分の書いたものに最終的な決定権を持つべきです、そう思いませんか？

## 実践例

### シナリオ 1：プロジェクトルートディレクトリ DESIGN.md 作成

DESIGN.md が存在しない場合、バックエンドは仮想ドキュメント状態を返します。この設計により、フロントエンドは「ファイルが存在しない」場合を特別に処理する必要がなく、統一された API インターフェースがコードロジックを簡素化します：

```csharp
return new ProjectDesignDocumentDto
{
    Path = targetPath,
    Exists = false,  // 仮想ドキュメント状態
    Content = string.Empty,
    Version = null
};

// 初回保存時に自動的にファイルを作成
public async Task<SaveProjectDesignDocumentResultDto> SaveDesignDocumentAsync(...)
{
    Directory.CreateDirectory(targetDirectory);
    await File.WriteAllTextAsync(targetPath, input.Content);
    return new SaveProjectDesignDocumentResultDto { Created = !exists };
}
```

この設計の利点は、フロントエンドが「ファイルが存在しない」場合を特別に処理する必要がなく、統一された API インターフェースがコードロジックを簡素化することです。時には、複雑性をバックエンドに隠すことで、フロントエンドはよりリラックスしてユーザー体験に集中できます。

### シナリオ 2：デザインサイトからテンプレートをインポート

ユーザーがインポートドロワーで「Linear」デザインテンプレートを選択すると、システムはバックエンドプロキシを通じて DESIGN.md の内容を取得します。全体のプロセスはユーザーにとって透過的であり、彼らはテンプレートを選択するだけで、システムが自動的にすべてのネットワークリクエストとデータ変換を処理します：

```typescript
// 1. システムはバックエンドプロキシを通じて DESIGN.md の内容を取得
GET /api/project/{id}/design-md/site-index/linear.app

// 2. バックエンドは slug を検証し、上流からコンテンツを取得
var entry = FindDesignSiteEntry(catalog, "linear.app");
using var upstreamResponse = await httpClient.SendAsync(request);
var content = await upstreamResponse.Content.ReadAsStringAsync();

// 3. フロントエンドがエディタテキストを置換
setDraft(result.content);
// ユーザーが確認した後、手動でディスクに保存
```

全体のプロセスはユーザーにとって透過的であり、彼らはテンプレートを選択するだけで、システムが自動的にすべてのネットワークリクエストとデータ変換を処理します。ユーザーは背後の複雑さを気にする必要がありません。これが私たちが追求する体験です——シンプルだが、強力です。

### シナリオ 3：バージョン競合処理

複数のユーザーが同時に同一 DESIGN.md を編集する場合、システムはバージョン競合を検出します。この楽観的同時実行制御メカニズムは、データの一貫性を保証すると同時に、ユーザーの編集操作をブロックしません：

```csharp
if (!string.Equals(currentVersion, expectedVersion, StringComparison.Ordinal))
{
    throw new BusinessException(
        ProjectDesignDocumentErrorCodes.VersionConflict,
        $"DESIGN.md at '{targetPath}' changed on disk.");
}
```

フロントエンドはこのエラーをキャッチし、ユーザーにプロンプトを表示します：

```typescript
// フロントエンドがユーザーにリフレッシュして再試行を促す
<Alert>
    <AlertTitle>バージョン競合</AlertTitle>
    <AlertDescription>
        ファイルが他のプロセスによって変更されました。最新バージョンをリフレッシュしてから再試行してください。
    </AlertDescription>
</Alert>
```

この楽観的同時実行制御メカニズムは、データの一貫性を保証すると同時に、ユーザーの編集操作をブロックしません。競合は避けられませんが、少なくともユーザーに何が起きたかを知らせることができ、黙って変更を失うことはありません。

## 注意事項とベストプラクティス

### 1. パスセキュリティ

常に `repositoryPath` を検証し、パストラバーサル攻撃を防止します。セキュリティというのは、いくらやっても足りないものです：

```csharp
// 常に repositoryPath を検証し、パストラバーサル攻撃を防止
return ValidateSubPathAsync(projectPath, repositoryPath);
// "../" や絶対パスなどの危険な入力を拒否
```

### 2. キャッシュ戦略

プレビュー画像は 24 時間キャッシュし、最大 160 ファイルまで。適度なキャッシュはパフォーマンスを向上させますが、やりすぎも良くありません。結局、バランスこそが鍵です：

```csharp
// プレビュー画像を 24 時間キャッシュ、最大 160 ファイル
private static readonly TimeSpan PreviewCacheTtl = TimeSpan.FromHours(24);
private const int PreviewCacheMaxFiles = 160;
// 定期的に期限切れキャッシュをクリーンアップ
```

### 3. エラーハンドリング

上流サイトが利用できない場合のフォールバック処理。このエレガントなフォールバック設計により、外部依存関係が利用できない場合でも、コア編集機能は正常に動作します。結局、一つの外部サービスが落ちたからといって、システム全体が麻痺するべきではありません：

```typescript
// 上流サイトが利用できない場合のフォールバック処理
try {
    const catalog = await getProjectDesignMdSiteImportCatalog(projectId);
} catch (error) {
    toast.error(t('project.designMd.siteImport.feedback.catalogLoadFailed'));
    // メイン編集ドロワーは依然として利用可能
}
```

このエレガントなフォールバック設計により、外部依存関係が利用できない場合でも、コア編集機能は正常に動作します。システムはレジリエンスを持つべきであり、問題に遭遇したらすぐに倒れるべきではありません。

### 4. ユーザー体験最適化

インポート前の上書き確認、インポート後の自動保存なし。ユーザーは自分の操作に制御権を持つべきであり、システムが勝手に判断すべきではありません：

```typescript
// インポート前に上書きを確認
if (isDirty) {
    setConfirmMode('import');
    return;
}

// インポート後は自動保存せず、ユーザーに確認させる
setDraft(result.content); // ドラフトのみ更新
// ユーザーが確認した後、Save をクリックして初めてディスクに書き込む
```

### 5. パフォーマンス考慮事項

HTTP クライアントファクトリを使用し、過剰な接続作成を回避します。リソース管理は、目立たないかもしれませんが、きちんとやれば予期せぬ効果をもたらします：

```csharp
// HTTP クライアントファクトリを使用し、過剰な接続作成を回避
private const string DesignSiteProxyClientName = "ProjectDesignSiteProxy";
private static readonly TimeSpan DesignSiteProxyTimeout = TimeSpan.FromSeconds(8);
```

## 拡張提案

1. **Markdown 強化**：現在は基本的な `Textarea` を使用していますが、CodeMirror にアップグレードして構文ハイライトとショートカットキーをサポートすることを検討できます。エディタの体験が良くなれば、ドキュメントを書く気分も良くなります。
2. **プレビューモード**：Markdown のリアルタイムプレビューを追加し、編集体験を向上させます。WYSIWYG は、いつも人々により多くの自信を与えます。
3. **差分マージ**：単純な全文置換ではなく、インテリジェントなマージアルゴリズムを実装します。競合は避けられませんが、少なくとも競合処理のプロセスをそれほど苦痛にしなくて済みます。
4. **ローカルキャッシュ**：design.json をデータベースにキャッシュし、外部サイトへの依存を減らします。依存が少なければ少ないほど、システムはより安定します。これは単純な理屈です。

## まとめ

HagiCode プロジェクトでは、フロントエンドとバックエンドの協力により、完全な DESIGN.md オンライン編集・インポートソリューションを実現しました。このソリューションの核心的な価値は以下の通りです：

- **効率向上**：ツールを切り替える必要がなく、統一された Web インターフェースで設計ドキュメントの編集とインポートを完了できます。節約した時間で、より意義のあることができます。
- **ハードル低下**：デザインテンプレートをワンクリックでインポートでき、新規プロジェクトを素早く開始できます。開始が容易であればあるほど、継続する可能性も高くなります。
- **安全で信頼性が高い**：パス検証、バージョン競合検出、エレガントなフォールバックなどのメカニズムにより、システムの安定稼働を保証します。安定が基礎であり、安定がなければ、すべては空談です。
- **ユーザー体験**：グローバルドロワー、ダーティ状態検出、確認ダイアログなどのディテールがインタラクション体験を磨き上げました。「細部が成败を決める」という言葉は、ユーザー体験に特に適用されます。

このソリューションはすでに HagiCode プロジェクトで実際に稼働しており、チームの設計ドキュメント管理の課題を解決しました。もしあなたも同様の問題に直面しているなら、この記事が何らかのインスピレーションを与えられれば幸いです。実のところ、何も高尚な理論はありません。ただ問題に遭遇し、解決策を考えただけです。
