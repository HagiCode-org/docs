## 背景

時が経つのは本当に早く、大規模言語モデルの発展は春の竹のように急速に伸びています。かつては「コードを書けるAI」だけで喜んでいた私たちですが、今やマルチモデルの協調とマルチツールの統合の時代となりました。これは面白いことでしょうか？ まあ、開発者が必要としているのはツールそのものではなく、異なるシーンに適応し、柔軟に切り替えられる余裕なのかもしれません。

HagiCodeはAI支援コーディングプラットフォームとして、最近2つの大きな出来事を迎えました。一是智譜AIのGLM-5.1モデルの全面的な統合、二是Gemini CLIが正式に10番目にサポートされるAgent CLIとなったことです。これらは大きくも小さくもありませんが、プラットフォームの完善にとってはとにかく良いことです。

GLM-5.1は智譜AIの最新のフラッグシップモデルであり、GLM-5.0と比較して推論能力が強化され、コードの理解が深まり、ツールの呼び出しがよりスムーズになりました。さらに重要なことは、画像入力をサポートする初めてのGLMモデルであるということです。これはどういうことでしょうか？ ユーザーが直接スクリーンショットを撮ってAIに問題を見せることができ、面倒な説明をする必要がないということです。この便利さは使えばわかります。

同時に、HagiCodeはHagiCode.Libs.Providersアーキテクチャを通じて、Gemini CLIを正常に統合しました。これは10番目のAgent CLIです。正直なところ、ここまで来られたのは少しの達成感があるだけです。

特筆すべきは、HagiCodeの画像アップロード機能により、ユーザーが直接スクリーンショットを撮ってAIと交流できることです。GLM 4.7バージョンを実行していても、プラットフォームは良好に動作し、プロジェクトの多くの重要な構築作業を完了するのを助けてきました。GLM-5.1については？ 自然にさらに進化するでしょう。

## マルチCLIアーキテクチャ設計

HagiCodeの中核的な利点の1つは、統一された抽象層を通じて多様なAIプログラミングCLIツールをサポートすることです。この設計の利点は、まあそんなものです。新しいものが入ってきて、古いものが残って、コードも乱れない。結局、誰もが生活がこうであってほしいと思うでしょう？

### AIProviderType列挙型

プラットフォームは`AIProviderType`列挙型を通じてサポートされるCLIプロバイダータイプを定義しています：

```csharp
public enum AIProviderType
{
    ClaudeCodeCli = 0,    // Claude Code CLI
    CodexCli = 1,          // GitHub Copilot Codex
    GitHubCopilot = 2,     // GitHub Copilot
    CodebuddyCli = 3,     // Codebuddy CLI
    OpenCodeCli = 4,      // OpenCode CLI
    IFlowCli = 5,         // IFlow CLI
    HermesCli = 6,        // Hermes CLI
    QoderCli = 7,         // Qoder CLI
    KiroCli = 8,          // Kiro CLI
    KimiCli = 9,          // Kimi CLI
    GeminiCli = 10,       // Gemini CLI (新規追加)
}
```

見てわかるように、Gemini CLIは10番目のメンバーとしてこの大家族に参加しました。各CLIには独自の特徴と適用シーンがあり、ユーザーは自分のニーズに応じて柔軟に選択できます。結局、すべての道はローマに通じていますが、ある道は少し歩きやすく、ある道は少し曲がりくねっているだけです。

### Providerアーキテクチャ

HagiCode.Libs.Providersは統一されたProviderインターフェースを提供し、各CLIの統合を標準化して簡潔にします。Gemini CLIを例にすると：

```csharp
public class GeminiProvider : ICliProvider<GeminiOptions>
{
    private static readonly string[] DefaultExecutableCandidates = ["gemini", "gemini-cli"];
    private const string ManagedBootstrapArgument = "--acp";

    public string Name => "gemini";
    public bool IsAvailable => _executableResolver.ResolveFirstAvailablePath(DefaultExecutableCandidates) is not null;
}
```

この設計の利点は：
- 新しいCLIの統合はProviderクラスを1つ実装するだけで済む
- 統一されたライフサイクル管理とセッションプーリング
- 自動化されたエイリアス解決と実行可能ファイルの検索

要するに、この設計は実際には複雑なことを単純化し、生活をもう少し楽にしているだけです。

### Provider Registry

Provider Registryはエイリアスのマッピングと登録を自動的に処理します：

```csharp
if (provider is GeminiProvider)
{
    registry.Register(provider.Name, provider, ["gemini-cli"]);
    continue;
}
```

これはユーザーが`gemini`または`gemini-cli`の2つの方法でGemini CLIを呼び出すことができ、システムが自動的に識別することを意味します。これは友達が多いのと似ていて、一部は大名で呼ばれ、一部はあだ名で呼ばれますが、どれも彼なので、どのように呼んでも構いません。

## GLM-5.1モデルサポート

GLM-5.1は智譜AIの最新のフラッグシップモデルであり、HagiCodeはその全面的なサポートを完了しました。

### Secondary Professions Catalog

HagiCodeはSecondary Professions Catalogを通じてサポートされるすべてのモデルを管理しています。以下はGLMシリーズの設定です：

| Model ID | Name | SupportsImage | Compatible CLI Families |
|----------|------|----------------|-------------------------|
| `glm-4.7` | GLM 4.7 | - | claude, codebuddy, hermes, qoder, kiro |
| `glm-5` | GLM 5 | - | claude, codebuddy, hermes, qoder, kiro |
| `glm-5-turbo` | GLM 5 Turbo | - | claude, codebuddy, hermes, qoder, kiro |
| `glm-5.0` | GLM 5.0 (Legacy) | - | claude, codebuddy, hermes, qoder, kiro |
| `glm-5.1` | GLM 5.1 | **true** | claude, codebuddy, hermes, qoder, kiro |

GLM-5.1の重要な特性は以下のように要約できます：
- 独立したバージョン識別子、legacyの荷物がない
- 画像入力をサポートする初めてのGLMモデル
- より強力な推論能力とコード理解
- 広範なマルチCLI互換性

### GLM-5.1 vs GLM-5.0

コードレベルから見ると、GLM-5.1とGLM-5.0の重要な違い：

```csharp
// GLM-5.0 (Legacy) - 特別な保持ロジックがある
private const string Glm50CodebuddySecondaryProfessionId = "secondary-glm-5-codebuddy";
private const string Glm50CodebuddyModelValue = "glm-5.0";

// GLM-5.1 - 独立した新しいモデル識別子
private const string Glm51SecondaryProfessionId = "secondary-glm-5-1";
private const string Glm51ModelValue = "glm-5.1";
```

GLM-5.0には「Legacy」マークがあり、後方互換性のために保持された旧バージョンの識別子です。一方、GLM-5.1は全く新しい独立したバージョンであり、歴史的な荷物はありません。これは一部の人々が常に過去に生きているのと似ていますが、ある人は軽装で出発し、より速く進むだけです。

### GLM-5.1の設定

HagiCodeでGLM-5.1を使用する設定例：

```json
{
  "primaryProfessionId": "profession-claude-code",
  "secondaryProfessionId": "secondary-glm-5-1",
  "model": "glm-5.1",
  "reasoning": "high"
}
```

## 画像アップロード機能

HagiCodeの画像サポートはSecondaryProfessionの`SupportsImage`属性を通じて実装されています：

```csharp
public class HeroSecondaryProfessionSettingDto
```

Secondary Professions Catalogでは、GLM-5.1の設定は以下の通りです：

```json

  "supportsImage": true
```

これはユーザーが直接スクリーンショットをアップロードしてAIに分析させることができることを意味します。例えば：
- エラーメッセージのスクリーンショット
- UIインターフェースの問題
- データ可視化チャート
- コード実行結果

もう手動で問題を説明する必要はなく、直接スクリーンショットを撮るだけで済みます。この機能の便利さは使えばわかります。結局、あることは、何度も言うより一目見る方が良いです。

## Gemini CLI統合

Gemini CLIは10番目のAgent CLIとして、標準のProviderアーキテクチャを通じてHagiCodeに統合されました。

### 設定オプション

Gemini CLIは豊富な設定オプションをサポートしています：

```csharp
public class GeminiOptions
{
    public string? ExecutablePath { get; set; }
    public string? WorkingDirectory { get; set; }
    public string? SessionId { get; set; }
    public string? Model { get; set; }
    public string? AuthenticationMethod { get; set; }
    public string? AuthenticationToken { get; set; }
    public Dictionary<string, string?> AuthenticationInfo { get; set; }
    public Dictionary<string, string?> EnvironmentVariables { get; set; }
    public string[] ExtraArguments { get; set; }
    public TimeSpan? StartupTimeout { get; set; }
    public CliPoolSettings? PoolSettings { get; set; }
}
```

これらのオプションは基本設定から高度な機能までカバーしており、ユーザーは自分のニーズに応じて柔軟に設定できます。結局、誰のニーズも違うので、柔軟であることはいつも良いことです。

### ACP通信プロトコル

Gemini CLIはACP（Agent Communication Protocol）通信プロトコルをサポートしており、これはHagiCodeの統一されたCLI通信標準です。ACPを通じて、異なるCLIは一貫した方法でプラットフォームと対話でき、統合作業が大幅に簡素化されます。要するに、複雑なことを統一して、誰もが少し楽になるようにしているだけです。

## マルチCLI統合のベストプラクティス

HagiCode.Libs.Providersは統一された登録と使用メカニズムを提供します：

```csharp
services.AddHagiCodeLibs();

var gemini = serviceProvider.GetRequiredService<ICliProvider<GeminiOptions>>();
var codebuddy = serviceProvider.GetRequiredService<ICliProvider<CodebuddyOptions>>();
var hermes = serviceProvider.GetRequiredService<ICliProvider<HermesOptions>>();
```

この依存性注入の設計により、各CLIの使用が非常に簡潔になり、単体テストとモックも便利です。結局、コードをきれいに書くことは、自分自身への責任でもあります。

### 注意事項

実際の使用では、いくつかの点に注意する必要があります：

1. **API Key設定**：`ANTHROPIC_AUTH_TOKEN`を正しく設定してください。そうしないとモデルを呼び出すことができません
2. **モデルの利用可能性**：GLM-5.1は対応するモデルプロバイダーで権限を開通する必要があります
3. **画像機能**：`supportsImage: true`をサポートするモデルのみが画像アップロード機能を使用できます
4. **CLIインストール**：Gemini CLIを使用する前に、`gemini`または`gemini-cli`がシステムPATHにあることを確認してください

これらは小さなことですが、小さなことをうまく処理しないと、大きなことになるかもしれません。したがって、やはり注意する必要があります。
