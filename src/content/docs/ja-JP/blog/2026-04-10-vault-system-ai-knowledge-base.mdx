## Vaultシステムで構築するAI時代のクロスプロジェクトナレッジベース

> プロジェクト模倣学習法が主流になりつつありますが、学習資料が分散し、コンテキストが断片化する課題により、AIアシスタントが最大価値を発揮できていません。本文ではHagiCodeプロジェクトのVaultシステム設計を紹介します——統一されたストレージ抽象化レイヤーを通じて、AIアシスタントがすべての学習リソースを理解・アクセスできるようにし、真のクロスプロジェクト知識再利用を実現します。

## 背景

実は、AI時代において、私たちが新しい技術を学ぶ方法は静かに変化しています。伝統的な読書や動画視聴も重要ですが、「プロジェクト模倣」——優れたオープンソースプロジェクトのコード、アーキテクチャ、設計パターンを深く研究・学習すること——が確実に効率的になっています。高品質なオープンソースプロジェクトを直接実行・修正することで、実世界のエンジニアリング実践を最速で理解できます。

ただ、この方法は新たな課題ももたらします。

**学習資料が分散しすぎています**。ノートはObsidian、コードリポジトリは各フォルダに散らばり、AIアシスタントの対話履歴は独立したデータサイロになっています。AIにあるプロジェクトの分析を依頼するたびに、コードスニペットを手動でコピーし、コンテキストを整理する必要があり、プロセスが非常に面倒です。

**コンテキストが頻繁に途切れます**。AIアシスタントはローカル学習リソースに直接アクセスできず、会話のたびに背景情報を再提供する必要があります。模倣したコードリポジトリは更新が速く、手動同期はエラーを起こしやすいです。さらに悪いことに、複数の学習プロジェクト間で知識を共有しにくい——Aプロジェクトで学んだ設計パターンを、AIがBプロジェクトを処理する際に全く知らないのです。

これらの問題の本質は「データサイロ」です。統一されたストレージ抽象化レイヤーがあり、AIアシスタントがすべての学習リソースを理解・アクセスできれば、問題は解決します。

これらの課題を解決するため、HagiCode開発時に重要な設計決定を行いました：Vaultシステムを統一ナレッジストレージ抽象化レイヤーとして構築することです。この決定による変化は、想像以上に大きいかもしれません——後ほど詳しく説明します。

## HagiCodeについて

本文で共有するソリューションは、[HagiCode](https://hagicode.com)プロジェクトでの実践経験に基づいています。HagiCodeはOpenSpecワークフローに基づくAIコードアシスタントで、その核心理念はAIに「話す」だけでなく「実行」させること——コードリポジトリを直接操作し、コマンドを実行し、テストを実行できるようにすることです。GitHub：[github.com/HagiCode-org/site](https://github.com/HagiCode-org/site)

開発過程で、AIアシスタントが頻繁にユーザーの様々な学習リソースにアクセスする必要があることがわかりました：コードリポジトリ、ノートドキュメント、設定ファイルなど。毎回ユーザーに手動で提供させる必要があるなら、体験が最悪です。これがVaultシステムの設計を促しました。

## コア設計

### マルチタイプサポート

HagiCodeのVaultシステムは4つのタイプをサポートし、それぞれ異なる使用シナリオに対応します：

| タイプ | 用途 | 典型的シナリオ |
|------|------|----------|
| `folder` | 汎用フォルダタイプ | 一時学習資料、下書き |
| `coderef` | コードプロジェクトの模倣専用 | あるオープンソースプロジェクトを体系的に学習 |
| `obsidian` | Obsidianノートソフトウェアとの統合 | 既存ノートライブラリの再利用 |
| `system-managed` | システム自動管理 | プロジェクト設定、プロンプトテンプレートなど |

そのうち`coderef`タイプはHagiCodeで最も一般的に使用され、コードプロジェクトの模倣のために標準化されたディレクトリ構造とAI可読なメタデータ記述を提供します。なぜこのタイプを専用設計したのか？ オープンソースプロジェクトを模倣するのは単に「コードをダウンロードする」だけでなく、コード自体、学習ノート、設定ファイルなど複数のコンテンツを同時に管理する必要があるからです。`coderef`はこれらをすべて規格化しました。

### 永続化ストレージメカニズム

VaultのレジストリはJSON形式でファイルシステムに永続化されます：

```csharp
_registryFilePath = Path.Combine(absoluteDataDir, "personal-data", "vaults", "registry.json");
```

この設計は一見シンプルに見えますが、実は熟慮されたものです：

**シンプルで信頼性が高い**。JSON形式は人間が読め、デバッグや手動修正に便利です。システムに問題が発生した場合、ファイルを直接開いて状態を確認し、手動で修復さえできます——これは開発段階で特に有用です。

**依存関係の削減**。ファイルシステムストレージはデータベースの複雑さを回避します。追加のデータベースサービスのインストールと設定が不要で、システム複雑度と保守コストを削減できます。

**並行安全性**。`SemaphoreSlim`を使用してマルチスレッド安全性を確保します。AIコードアシスタントのシナリオでは、複数の操作が同時にvaultレジストリにアクセスする可能性があり、適切な並行制御が必要です。

### AIコンテキスト統合

システムの中核能力は、vault情報をAIプロンププトのコンテキストに自動的に注入できることです：

```typescript
export function buildTargetVaultsText(
  vaults: VaultForText[],
  template: VaultPromptTemplate = DEFAULT_VAULT_PROMPT_TEMPLATE,
): string {
  const readOnlyVaults = vaults.filter((vault) => vault.accessType === 'read');
  const editableVaults = vaults.filter((vault) => vault.accessType === 'write');

  const sections = [
    buildVaultSection(readOnlyVaults, template.reference),
    buildVaultSection(editableVaults, template.editable),
  ].filter(Boolean);

  return `\n\n### ${template.heading}\n\n${sections.join('\n')}`;
}
```

こうしてAIアシスタントは利用可能な学習リソースを自動的に理解でき、ユーザーが毎回手動でコンテキストを提供する必要がありません。この設計により、HagiCodeの体験が非常に自然になります——AIに「Reactの同時実行レンダリングを分析して」と伝えると、AIは自動的に以前登録したReact学習vaultを見つけ、コードを何度も貼り付ける必要がありません。

### アクセス制御メカニズム

システムはvaultを2つのアクセスタイプに分類します：

- **reference（読み取り専用）**：AIは分析・理解のみに使用し、コンテンツを変更できません
- **editable（編集可能）**：AIはタスクに応じてコンテンツを変更できます

この区分により、AIはどのコンテンツが「読み取り専用参照」で、どのコンテンツが「修正可能」かを知り、誤操作リスクを回避できます。例えば、学習材料としてオープンソースプロジェクトのvaultを登録した場合、AIがコードを勝手に修正してほしくない——`reference`とマークします。しかし、自分のプロジェクトvaultなら`editable`とマークし、AIにコード修正を手伝わせることができます。

## 実践ガイド

### CodeRef Vaultの標準化構造

coderefタイプのvaultについて、システムは標準化されたディレクトリ構造を提供します：

```
my-coderef-vault/
├── index.yaml          # vaultメタデータ記述
├── AGENTS.md           # AIアシスタントの操作ガイド
├── docs/               # 学習ノートとドキュメントの保存
└── repos/              # Gitサブモジュールを通じて模倣したコードリポジトリを管理
```

この構造の設計理念は何でしょうか？

**docs/** は学習ノートを保存し、Markdown形式でコードの理解、アーキテクチャ分析、トラブルシューティング経験を記録します。これらのノートは自分だけでなく、AIも読めます——関連タスクを処理する際に自動的に参照されます。

**repos/** はGitサブモジュールを通じて模倣したリポジトリを管理し、コードを直接コピーしません。これには2つの利点があります：1つ目は上流との同期を維持し、`git submodule update`で最新コードを取得できます。2つ目はスペースを節約し、複数のvaultが同じリポジトリの異なるバージョンを参照できます。

**index.yaml** はvaultのメタデータを含み、AIアシスタントが用途とコンテンツを素早く理解できます。vaultの「自己紹介」を書くようなもので、AIは初見で何のためのものかを知ります。

**AGENTS.md** はAIアシスタント向けのガイドで、vault内のコンテンツを処理する方法を説明します。ここでAIに「このプロジェクトを分析する際はパフォーマンス最適化関連のコードに重点を置く」や「テストファイルを修正しないで」と伝えることができます。

### Vaultの作成と使用

CodeRef vaultの作成は簡単です：

```typescript
const createCodeRefVault = async () => {
  const response = await VaultService.postApiVaults({
    requestBody: {
      name: "React Learning Vault",
      type: "coderef",
      physicalPath: "/Users/developer/vaults/react-learning",
      gitUrl: "https://github.com/facebook/react.git"
    }
  });

  // システムは自動的に：
  // 1. Reactリポジトリをvault/repos/reactにクローン
  // 2. ノート用のdocs/ディレクトリを作成
  // 3. index.yamlメタデータを生成
  // 4. AGENTS.mdガイドファイルを作成

  return response;
};
```

そしてAIプロンププトでこのvaultを参照します：

```typescript
const proposal = composeProposalChiefComplaint({
  chiefComplaint: "Reactの同時実行レンダリングメカニズムを分析してください",
  repositories: [
    { id: "react", gitUrl: "https://github.com/facebook/react.git" }
  ],
  vaults: [
    {
      id: "react-learning",
      name: "React Learning Vault",
      type: "coderef",
      physicalPath: "/vaults/react-learning",
      accessType: "read"  // AIは読み取りのみ、修正不可
    }
  ],
  quickRequestText: "fiberアーキテクチャとscheduler実装に重点を置く"
});
```

### 典型的使用シナリオ

**シナリオ1：オープンソースプロジェクトの体系的学習**

CodeRef vaultを作成し、Gitサブモジュールを通じてターゲットリポジトリを管理し、`docs/`ディレクトリに学習ノートを記録します。AIはコードとノートに同時にアクセスでき、より正確な分析を提供できます。あるモジュールを学習する際に書いたノートは、AIが関連コードを分析する際に自動的に参照されます——「アシスタント」が以前の思考を覚えているようなものです。

**シナリオ2：Obsidianノートライブラリの再利用**

既にObsidianでノートを管理している場合、既存のvaultをHagiCodeに登録するだけで済みます。AIはナレッジベースに直接アクセスでき、手動でのコピー＆ペーストが不要です。この機能は特に実用的で、多くの人々が何年もかけて蓄積したノートライブラリを持っており、接続後、AIは知識体系を「読む」ことができます。

**シナリオ3：クロスプロジェクト知識再利用**

複数のAIプロンププトが同じvaultを参照でき、知識のクロスプロジェクト再利用を実現します。例えば「デザインパターン学習vault」を作成し、各種デザインパターンのノートとコード例を記録します。どのプロジェクトを分析しても、AIはこのvault内のコンテンツを参照できます——知識を重複して蓄積する必要がありません。

### パス安全メカニズム

システムはパスを厳格に検証し、パストラバーサル攻撃を防止します：

```csharp
private static string ResolveFilePath(string vaultRoot, string relativePath)
{
    var rootPath = EnsureTrailingSeparator(Path.GetFullPath(vaultRoot));
    var combinedPath = Path.GetFullPath(Path.Combine(rootPath, relativePath));
    if (!combinedPath.StartsWith(rootPath, StringComparison.OrdinalIgnoreCase))
    {
        throw new BusinessException(VaultRelativePathTraversalCode,
            "Vault file paths must stay inside the registered vault root.");
    }
    return combinedPath;
}
```

これにより、すべてのファイル操作がvaultのルートディレクトリ範囲内で行われ、悪意あるパスアクセスを防止します。安全には手抜きができません。AIアシスタントがファイルシステムを操作する場合、境界を明確にする必要があります。

## 注意事項

HagiCode Vaultシステムを使用する際、特別な注意が必要な点がいくつかあります：

1. **パスの安全性**：カスタムパスが許可範囲内にあることを確認してください。そうでない場合、システムは操作を拒否します。これは誤操作と潜在的なセキュリティリスクを防ぐためです。

2. **Gitサブモジュール管理**：CodeRef vaultはコードを直接コピーするよりもGitサブモジュールの使用を推奨します。利点は前述の通り——同期維持、スペース節約。ただ、サブモジュールには独自の使用方法があり、初回使用時は少し慣れが必要かもしれません。

3. **ファイルプレビュー制限**：システムはファイルサイズ（256KB）と数（500個）を制限し、超大ファイルは分割処理が必要です。この制限はパフォーマンス考慮のためです。超大ファイルに遭遇した場合、手動分割や他の方法で処理できます。

4. **診断情報**：vault作成時に診断情報が返され、失敗時のデバッグに使用できます。問題に遭遇した場合、まず診断情報を確認してください。ほとんどの場合、手がかりが見つかります。

## まとめ

HagiCodeのVaultシステムは本質的にシンプルですが深刻な問題を解決しています：AIアシスタントにローカルナレッジリソースを理解・使用させる方法を。

統一されたストレージ抽象化レイヤー、標準化されたディレクトリ構造、自動化されたコンテキスト注入を通じて、「一度登録、どこでも再利用」のナレッジ管理方式を実現しました。vaultを作成すると、学習ノート、コードリポジトリ、ドキュメント資料のいずれでも、AIが自動的にアクセス・理解できます。

この設計による体験向上は明白です。もうコードスニペットの手動コピー、背景情報の繰り返し説明は不要——AIアシスタントはプロジェクト状況を真に理解している同僚のように、既存のナレッジに基づいてより価値のある支援を提供できます。

本文で共有したVaultシステムは、HagiCode開発過程で実際に試行錯誤し、実際に最適化したソリューションです。この設計に価値を感じるなら、エンジニアリング実力は悪くありません——那么HagiCode自体も注目に値するでしょう。
