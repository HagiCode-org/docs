## Résoudre le problème d'authentification WebSocket dans les navigateurs : pratique de la solution proxy pour la reconnaissance vocale Doubao

> L'API WebSocket du navigateur ne prend pas en charge les en-têtes HTTP personnalisés, ce qui pose un défi pour les services de reconnaissance vocale qui nécessitent la transmission d'informations d'authentification via des en-têtes. Cet article partage comment le projet HagiCode a résolu ce problème grâce à une solution de proxy backend, ainsi que le processus de pratique de l'environnement de playground à la production.


## Contexte

En fait, lors du développement de la fonction de reconnaissance vocale du projet HagiCode, nous avons également choisi avec confiance le service de reconnaissance vocale Doubao de ByteDance. La conception initiale était très simple - le frontend se connecte directement au service WebSocket de Doubao. Qu'est-ce qu'il y a de difficile ? Il suffit d'établir une connexion et de transmettre des données, non ?

Cependant, à notre grande surprise, l'API de Doubao exige la transmission d'informations d'authentification via des en-têtes HTTP, comme accessToken, secretKey, etc. Cela devient un peu gênant car l'API WebSocket du navigateur ne prend pas du tout en charge la définition d'en-têtes personnalisés.

Que faire lorsque ce n'est pas pris en charge ?

À ce moment-là, nous avons aussi hésité un moment. Après tout, deux choix s'offraient à nous :

1. Mettre les informations d'authentification dans les paramètres de requête de l'URL - simple et brutal
2. Faire une couche de proxy dans le backend - semble un peu plus compliqué

La première solution expose directement les identifiants dans le code frontend et le stockage local. Est-ce sûr ? Franchement, je ne suis pas vraiment d'accord. De plus, certaines API doivent utiliser la vérification par en-tête, ce qui ne fonctionne tout simplement pas.

Finalement, après réflexion, nous avons choisi la deuxième solution - implémenter un proxy WebSocket dans le backend. C'est d'ailleurs une coïncidence, cette solution a été initialement testée dans notre environnement de playground, et ensuite confirmée comme stable avant d'être appliquée à l'environnement de production. Après tout, personne ne veut servir de cobaye en production, je comprends quand même ça.

## Analyse des défis techniques

### Les limites du WebSocket dans le navigateur

L'API WebSocket standard semble vraiment très simple :

```javascript
const ws = new WebSocket('wss://example.com/ws');
```

Mais le problème réside dans le mot "simple" - elle ne transmet des paramètres que dans l'URL, et ne peut pas définir de headers comme une requête HTTP :

```javascript
// Ceci n'est pas pris en charge dans l'API WebSocket
const ws = new WebSocket('wss://example.com/ws', {
  headers: {
    'Authorization': 'Bearer token'
  }
});
```

Vous voyez, à qui se plaindre ? Pour des services comme la reconnaissance vocale Doubao qui nécessitent une authentification par en-tête, cette limite est simplement un obstacle infranchissable.

Bon, bon, et alors ?

### Décisions de conception architecturale

Lors de la conception de la solution, nous avons réfléchi longuement, pesé le pour et le contre.

**Décision un : choix du mode proxy**

Nous avons comparé deux solutions :

| Solution | Avantages | Inconvénients | Décision |
|------|------|------|------|
| WebSocket natif | Léger, simple, transfert direct | Gestion manuelle des connexions | Choisir |
| SignalR | Reconnexion automatique, typage fort | Trop complexe, dépendance supplémentaire | Ne pas choisir |

Nous avons finalement choisi le WebSocket natif. Pour être honnête, c'est parce qu'il est le plus léger et convient au transfert de flux binaires bidirectionnels simples. Ajouter SignalR, c'est vraiment utiliser un marteau-piqueur pour casser une noix, et cela augmenterait la latence - à quoi bon ?

**Décision deux : stratégie de gestion des connexions**

Nous avons adopté le mode "une session par connexion" - chaque connexion WebSocket frontend correspond à une connexion backend Doubao indépendante.

Les avantages sont évidents :
- Implémentation simple, conforme aux scénarios d'utilisation typiques
- Facile à déboguer et dépanner
- Isolement des ressources, évitant les interférences entre sessions

En fait, pour faire simple - parfois la solution simple et brute est le meilleur choix. Les solutions complexes ne sont pas nécessairement bonnes, les solutions simples ne sont pas nécessairement mauvaises.

**Décision trois : stockage des informations d'authentification**

Les identifiants sont stockés dans les fichiers de configuration backend (appsettings.yml ou variables d'environnement) et chargés par injection de dépendances :

- Mode de configuration simple, conforme au mode de configuration backend existant
- Les informations sensibles ne sont pas exposées au frontend
- Prise en charge de la configuration multi-environnements (développement, test, production)

La sécurité, il faut en avoir. Après tout, personne ne veut voir ses identifiants partout, n'est-ce pas ?

### Conception du flux de données

Le flux de données global est comme ceci :

```
Frontend (navigateur)
  │
  │ ws://backend/api/voice/ws
  │ WebSocket (binaire)
  ▼
Backend (proxy)
  │
  │ wss://openspeech.bytedance.com/
  │ (avec en-têtes d'authentification)
  ▼
API Doubao
```

Le processus n'est pas compliqué, juste ces quelques étapes :
1. Le frontend se connecte au proxy backend via WebSocket
2. Le proxy backend reçoit les données audio et se connecte à l'API Doubao avec des en-têtes
3. L'API Doubao renvoie les résultats de reconnaissance, le proxy les transmet au frontend
4. Transmission de flux bidirectionnel asynchrone tout au long du processus

Tout semble si naturel, non ?

## Implémentation des composants principaux

### 1. Configuration du point de terminaison WebSocket

```csharp
app.Map("/ws", async context =>
{
    if (context.WebSockets.IsWebSocketRequest)
    {
        // Lire la configuration depuis les paramètres de requête
        var appId = context.Request.Query["appId"];
        var accessToken = context.Request.Query["accessToken"];

        // Valider les paramètres requis
        if (string.IsNullOrEmpty(appId) || string.IsNullOrEmpty(accessToken))
        {
            context.Response.StatusCode = 400;
            return;
        }

        // Accepter la connexion WebSocket
        using var webSocket = await context.WebSockets.AcceptWebSocketAsync();

        // Boucle de traitement des messages
        var buffer = new byte[4096];
        while (!webSocket.CloseStatus.HasValue)
        {
            var result = await webSocket.ReceiveAsync(buffer, CancellationToken.None);

            if (result.MessageType == WebSocketMessageType.Close)
            {
                await webSocket.CloseAsync(
                    result.CloseStatus.Value,
                    result.CloseStatusDescription,
                    CancellationToken.None);
                break;
            }

            // Traiter les données audio
            await HandleAudioDataAsync(buffer, result.Count);
        }
    }
});
```

### 2. Gestion des sessions

```csharp
public class DoubaoSessionManager : IDoubaoSessionManager
{
    private readonly ConcurrentDictionary<string, DoubaoSession> _sessions = new();

    public DoubaoSession CreateSession(string connectionId)
    {
        var session = new DoubaoSession(connectionId);
        _sessions[connectionId] = session;
        return session;
    }

    public async Task SendAudioAsync(string connectionId, byte[] audioData)
    {
        if (_sessions.TryGetValue(connectionId, out var session))
        {
            await session.SendAudioAsync(audioData);
        }
    }

    public void RemoveSession(string connectionId)
    {
        if (_sessions.TryRemove(connectionId, out var session))
        {
            session.Dispose();
        }
    }
}
```

Utiliser ConcurrentDictionary pour gérer les sessions, la sécurité des threads n'est plus un souci. Chaque connexion entrante crée une Session, nettoyage automatique à la déconnexion - c'est大概 ce qu'on appelle "venir en hâte, partir en hâte".

## Pratique d'intégration frontend

### Connexion WebSocket

```javascript
class DoubaoVoiceClient {
    constructor(config) {
        this.config = config;
        this.ws = null;
    }

    async connect() {
        const url = new URL(this.config.wsUrl);
        // Ajouter des paramètres de requête
        Object.entries(this.config.params).forEach(([key, value]) => {
            url.searchParams.set(key, value);
        });

        this.ws = new WebSocket(url);

        return new Promise((resolve, reject) => {
            this.ws.onopen = () => {
                console.log('[DoubaoVoice] Connected');
                resolve();
            };

            this.ws.onmessage = (event) => {
                this._handleMessage(JSON.parse(event.data));
            };

            this.ws.onerror = reject;
        });
    }

    _handleMessage(message) {
        switch (message.type) {
            case 'status':
                this._handleStatus(message.payload);
                break;
            case 'result':
                this.onResult?.(message.payload);
                break;
            case 'error':
                console.error('[DoubaoVoice] Error:', message.payload);
                break;
        }
    }
}

// Exemple d'utilisation
const client = new DoubaoVoiceClient({
    wsUrl: 'ws://localhost:5000/ws',
    params: {
        appId: 'your-app-id',
        accessToken: 'your-access-token',
        sampleRate: 16000,
        bitsPerSample: 16,
        channels: 1
    }
});
```

### Acquisition et envoi audio

Utiliser AudioWorkletNode pour le traitement audio offre de meilleures performances :

```javascript
// audio-worklet.js
class AudioProcessorWorklet extends AudioWorkletProcessor {
    process(inputs, outputs, parameters) {
        const input = inputs[0]?.[0];
        if (!input) return true;

        // Convertir en PCM 16-bit
        const pcm = new Int16Array(input.length);
        for (let i = 0; i < input.length; i++) {
            pcm[i] = Math.max(-32768, Math.min(32767, input[i] * 32767));
        }

        this.port.postMessage({
            type: 'audioData',
            data: pcm.buffer
        }, [pcm.buffer]);

        return true;
    }
}

registerProcessor('audio-processor', AudioProcessorWorklet);

// Code du thread principal
async function startAudioRecording() {
    const stream = await navigator.mediaDevices.getUserMedia({
        audio: {
            echoCancellation: true,
            noiseSuppression: true,
            autoGainControl: true,
            sampleRate: 48000
        }
    });

    const audioContext = new AudioContext();
    const audioSource = audioContext.createMediaStreamSource(stream);

    await audioContext.audioWorklet.addModule('/audio-worklet.js');
    const audioWorkletNode = new AudioWorkletNode(audioContext, 'audio-processor');

    audioWorkletNode.port.onmessage = (event) => {
        if (event.data.type === 'audioData' && ws?.readyState === WebSocket.OPEN) {
            ws.send(event.data.data); // Envoyer directement les données binaires
        }
    };

    audioSource.connect(audioWorkletNode);
}
```

AudioWorklet a bien meilleures performances que ScriptProcessorNode, pas de problèmes d'interruption audio. De nos jours, qui veut encore écouter ce genre de bruit grésillant ?

## Précautions et meilleures pratiques

### 1. Surveillance des connexions

- Sortir régulièrement des journaux d'état de session pour faciliter le suivi du cycle de vie des connexions
- Surveiller le nombre et la durée des segments audio pour identifier les connexions anormales
- Enregistrer l'état de connexion et les reconnexions avec le service Doubao

Ce sont juste quelques opérations de base.

### 2. Gestion des erreurs

- Capturer et enregistrer toutes les exceptions WebSocket
- Utiliser IAsyncDisposable pour assurer le nettoyage des ressources
- Implémenter une fermeture gracieuse des connexions et une gestion des délais d'attente

En résumé, la stabilité avant tout.

### 3. Exigences de format audio

- Taux d'échantillonnage : 16000 Hz (recommandé) ou 8000 Hz
- Profondeur de bits : 16-bit
- Canaux : mono
- Encodage : PCM (raw)

Un format incorrect entraînera un échec de reconnaissance ou de très mauvais résultats. Ces règles doivent être respectées.

### 4. Considérations de sécurité

- Les identifiants sensibles ne sont stockés que dans la configuration backend
- Mettre en œuvre des limites de connexions pour éviter l'épuisement des ressources
- Utiliser HTTPS/WSS en production

La sécurité n'est pas une mince affaire, procédons avec prudence.

### 5. Optimisation des performances

- Utiliser des opérations asynchrones pour éviter les blocages
- Ajuster judicieusement la taille du tampon (4096 octets par défaut)
- Envisager des stratégies de pool et de réutilisation de connexions

Utilisez ces techniques d'optimisation autant que possible.

## Recommandations de déploiement

1. **Déploiement Docker** : Empaquetez le service proxy dans un conteneur pour faciliter la mise à l'échelle et la gestion
2. **Équilibrage de charge** : Utilisez Nginx ou Envoy pour le proxy inverse WebSocket
3. **Vérification de santé** : Implémentez un mécanisme de heartbeat pour surveiller la disponibilité du service
4. **Agrégation de journaux** : Envoyez les journaux vers un système de journalisation centralisé (comme ELK, Loki)

Le déploiement, c'est simple quand c'est simple, compliqué quand c'est compliqué. C'est selon la personne et la situation.
