## Protection de code .NET en pratique : de l'obfuscation à la protection par machine virtuelle

> Cet article présente comment mettre en œuvre une stratégie de protection de code multiniveau dans des projets .NET, couvrant toutes les solutions de l'obfuscation de base à la protection professionnelle par machine virtuelle.

## Contexte

Dans le développement d'applications .NET, protéger le code central (comme la vérification de licences, la logique métier, les configurations sensibles, etc.) contre la décompilation et l'analyse inverse est, comment dire, un sujet incontournable. Avec la maturation de l'écosystème .NET, les développeurs disposent de plusieurs moyens de protection de code, des attributs d'obfuscation intégrés aux outils de protection par virtualisation professionnels, le choix est plutôt vaste.

En tant que projet monorepo complexe multilingue, HagiCode comprend des applications de bureau, des systèmes de build et des fonctionnalités de gestion de licences. Le code implique inévitablement une logique de vérification de licences, des configurations sensibles (comme les clés API, les ID de produit) ainsi que la logique métier principale, ces choses doivent être bien protégées, après tout personne ne veut voir son travail facilement dévoilé.

Cet article partagera les solutions de protection de code que nous avons réellement adoptées dans le projet HagiCode, résumant le processus complet des difficultés rencontrées à l'optimisation, ce qui pourrait peut-être vous inspirer.

## À propos de HagiCode

[HagiCode](https://hagicode.com) est un projet d'assistant de code IA open source, s'efforçant de fournir une expérience de programmation intelligente aux développeurs. Le projet adopte une architecture monorepo, tout en maintenant plusieurs composants tels que l'extension VSCode, les services IA backend, les clients de bureau multiplateformes, etc. Cette complexité multilingue et multiplateforme rend la protection du code un défi d'ingénierie incontournable, pas le choix, puisque le projet est si complexe.

La solution partagée dans cet article est précisément ce que nous avons réellement développé et optimisé lors du développement de HagiCode. Si vous souhaitez comprendre comment nous avons résolu ces problèmes techniques, continuez à lire, vous aurez peut-être des surprises.

## Contenu principal

### 1. Attribut d'obfuscation intégré de Microsoft

.NET Framework fournit un `[ObfuscationAttribute]` intégré, qui est la balise d'obfuscation de code la plus basique et la plus couramment utilisée. Cet attribut se trouve dans l'espace de noms `System.Reflection` et peut protéger le code de base sans introduire d'outils tiers, ce qui est plutôt pratique.

**Caractéristiques principales** :
- Propriété `Feature` : spécifie les caractéristiques d'obfuscation, comme `"ultra"` (obfuscation élevée), `"all"` (obfuscation complète)
- Propriété `Exclude` : `true` signifie exclure l'obfuscation, `false` signifie appliquer l'obfuscation
- Peut être appliqué aux classes, méthodes, propriétés et autres membres de type

Dans le projet HagiCode, on peut voir un exemple d'utilisation réelle :

```csharp
[Obfuscation(Feature = "ultra", Exclude = false)]
public async Task<LicenseValidationResult?> ValidateLicenseAsync(...)
```

Les **avantages** de cette approche sont assez évidents :
- Aucune dépendance supplémentaire requise, intégré à .NET Framework, ce qui économise beaucoup d'efforts
- Peut être reconnu et traité par des outils d'obfuscation tiers
- N'augmente pas significativement la taille de l'assemblage compilé

Cependant, il a aussi des limitations : il ne s'agit que d'une balise, l'effet d'obfuscation réel dépend de la mise en œuvre de l'outil, il ne peut pas fournir un niveau de sécurité de protection par machine virtuelle, tant pis, après tout il n'a pas été créé pour cela.

### 2. VMP (Virtual Machine Protection)

VMP est un outil de protection de code professionnel qui fournit un niveau de protection élevé en compilant le code en instructions de machine virtuelle. Contrairement à l'obfuscation de noms simple, VMP convertit réellement la logique de code sous une forme qui ne peut pas être restaurée par les décompilateurs classiques, ce qui est plutôt impressionnant.

**Classification des niveaux de protection** :

| Niveau | Virtualisation | Mutation | Anti-débogage | Chiffrement des chaînes | Scénarios d'utilisation |
|--------|----------------|----------|---------------|-------------------------|-------------------------|
| HIGH | full | high | Activé | Activé | Vérification de licence, concurrence de session, constantes sensibles |
| MEDIUM | partial | medium | Activé | Activé | Logique métier, modèles de domaine |
| LOW | none | low | Désactivé | Désactivé | Classes utilitaires, code non critique |

Le projet HagiCode définit un système d'attributs déclaratif pour marquer le code nécessitant une protection :

```csharp
// Protection haute priorité
[VmProtect(VmProtectionPriority.High, Reason = "Contains license verification logic")]
public class KeygenClient { ... }

// Exclusion de protection
[VmExclude(Reason = "Public API that must remain unchanged")]
public class PublicApi { ... }

// Protection par héritage
[VmProtect(Priority.High, ProtectDerived = true)]
public class BaseLicenseValidator { ... }
```

### 3. Stratégie de protection au moment de la build

La protection VMP ne fonctionne pas seulement au moment de l'exécution, elle doit également être traitée automatiquement dans le processus de build, après tout le faire manuellement est trop fastidieux. Le système de build de HagiCode prend en charge plusieurs modes :

- **Mode natif Windows** : appelle directement l'outil VMProtect
- **Mode conteneur Docker Linux** : exécute VMP dans un conteneur (résout les problèmes de compatibilité multiplateforme)
- **Analyse d'attributs** : découvre automatiquement les balises de protection dans le code
- **Mécanisme de vérification** : confirme que la protection a été appliquée avec succès

Ces fonctionnalités combinées sont plutôt rassurantes.

## Solutions

### 1. Utilisation de l'attribut d'obfuscation intégré de Microsoft

Appliquer directement `ObfuscationAttribute` dans le code :

```csharp
using System.Reflection;

[Obfuscation(Feature = "ultra", Exclude = false)]
public class LicenseService
{
    [Obfuscation(Feature = "ultra", Exclude = false)]
    public async Task<bool> ValidateLicenseAsync(string key)
    {
        // Logique de vérification de licence
    }

    [Obfuscation(Feature = "flow", Exclude = false)]
    private string DecryptToken(string encrypted)
    {
        // Logique de déchiffrement
    }
}
```

Parfois, il est nécessaire de permettre aux assemblages de test d'accéder aux membres internes tout en maintenant la sécurité du code de production :

```csharp
// AssemblyInfo.cs
[assembly: InternalsVisibleTo("HagiCode.Application.Tests")]
[assembly: InternalsVisibleTo("DynamicProxyGenAssembly2")] // for Moq
```

Ainsi, les tests sont beaucoup plus pratiques, après tout le code doit être testé.

### 2. Définition d'attributs personnalisés pour la protection VMP

Créer des attributs de protection personnalisés pour contrôler le comportement de VMP :

```csharp
using System;

[AttributeUsage(AttributeTargets.Class | AttributeTargets.Method | AttributeTargets.Property)]
public class VmProtectAttribute : Attribute
{
    public VmProtectionPriority Priority { get; set; }
    public string? Reason { get; set; }
    public bool ProtectDerived { get; set; }
}

[AttributeUsage(AttributeTargets.Class | AttributeTargets.Method | AttributeTargets.Property)]
public class VmExcludeAttribute : Attribute
{
    public string? Reason { get; set; }
}

public enum VmProtectionPriority
{
    None = 0,
    Low = 1,
    Medium = 2,
    High = 3
}
```

Les attributs que l'on définit soi-même sont plus agréables à utiliser, car on comprend ses propres besoins.

### 3. Fichier de configuration VMP

```yaml
# vmp_config.yml
protection:
  priority_mode: "attribute"  # Priorité basée sur les attributs
  default_level: "medium"

tools:
  - name: "vmprotect"
    path: "C:\\Program Files\\VMProtect Ultimate\\VMProtect.exe"

protection_levels:
  high:
    virtualization: "full"
    mutation: "high"
    anti_debug: true
    anti_dump: true
    encrypt_strings: true
    encrypt_resources: true

  medium:
    virtualization: "partial"
    mutation: "medium"
    anti_debug: true
    encrypt_strings: true

  low:
    virtualization: "none"
    mutation: "low"
    anti_debug: false
```

Une configuration claire facilite également la maintenance ultérieure.

## Guide pratique

### 1. Pratique de protection des composants clés

Selon la spécification code-protection de HagiCode, les composants suivants doivent utiliser une protection de priorité HIGH :

```csharp
// Constantes de production - doivent être chiffrées et protégées par VMP
[VmProtect(VmProtectionPriority.High, Reason = "Production constants")]
public static class ProductionConstants
{
    // Accesseur de chaîne chiffrée, protégé par VMP
    [VmProtect(VmProtectionPriority.High)]
    public static string GetLicenseServerUrl(IOptions<LicenseOptions> options) => ...;
}

// Logique de vérification de licence
[VmProtect(VmProtectionPriority.High, Reason = "License verification logic")]
public class KeygenClient : IKeygenClient
{
    [Obfuscation(Feature = "ultra", Exclude = false)]
    public async Task<LicenseValidationResult?> ValidateLicenseAsync(...) { ... }
}

// Service d'empreinte machine
[VmProtect(VmProtectionPriority.High)]
public class MachineFingerprintService : IMachineFingerprintService { ... }
```

Le code clé doit être protégé de manière ciblée, après tout la fuite de la logique principale serait problématique.

### 2. Déchiffrement à l'exécution de chaînes chiffrées

Chiffrer les chaînes au moment de la build, déchiffrer à l'exécution :

```csharp
public static class StringDecryption
{
    [VmProtect(VmProtectionPriority.High, Reason = "CRITICAL SECURITY")]
    public static string DecryptString(byte[] encryptedData, byte[] key, byte[] iv)
    {
        using var aes = Aes.Create();
        aes.Key = key;
        aes.IV = iv;

        using var decryptor = aes.CreateDecryptor();
        using var ms = new MemoryStream(encryptedData);
        using var cs = new CryptoStream(ms, decryptor, CryptoStreamMode.Read);
        using var reader = new StreamReader(cs);

        return reader.ReadToEnd();
    }
}

// Accesseur de constantes de production (chargement différé + cache)
public static class ProductionConstants
{
    private static string? _cachedLicenseServerUrl;

    public static string GetLicenseServerUrl(IOptions<LicenseOptions> options)
    {
        if (_cachedLicenseServerUrl == null)
        {
            var encrypted = GetEncryptedLicenseServerUrl();
#if DEBUG
            _cachedLicenseServerUrl = options.Value.PrimaryServer.Url;
#else
            _cachedLicenseServerUrl = StringDecryption.DecryptString(
                encrypted,
                GetEncryptionKey(),
                GetEncryptionIV());
#endif
        }
        return _cachedLicenseServerUrl;
    }
}
```

Le chiffrement des chaînes est aussi assez important, après tout les informations sensibles ne doivent pas être en clair.

### 3. Vérification de la protection VMP

Après la build, il faut vérifier que la protection a été appliquée avec succès, sinon comment savoir si la protection est efficace :

```csharp
// Exemple de script de vérification
public bool VerifyProtection(string assemblyPath)
{
    // 1. Vérifier la signature VMP
    var bytes = File.ReadAllBytes(assemblyPath);
    var vmpSignature = Encoding.ASCII.GetBytes("VMProtect");
    if (bytes.Any(b => vmpSignature.Contains(b)))
    {
        return true;
    }

    // 2. Vérifier le changement de taille de fichier (après protection, augmente généralement)
    var originalInfo = new FileInfo(assemblyPath.Replace(".dll", ".bak"));
    if (originalInfo.Exists)
    {
        var sizeRatio = (double)new FileInfo(assemblyPath).Length / originalInfo.Length;
        return sizeRatio > 1.1;
    }

    return false;
}
```

C'est toujours bien de vérifier, pour éviter des problèmes dont on ne serait pas conscient.

### 4. Points d'attention

Il y a quelques pièges à noter particulièrement, puisque nous les avons tous rencontrés :

1. **N'obfusquez pas tout le code** : les API publiques, les définitions d'interface, les classes DTO n'ont généralement pas besoin d'être protégées, une obfuscation excessive affectera les performances et le débogage, le projet HagiCode a souffert de ce problème, il faut faire attention

2. **Protégez les accesseurs de clés** : les méthodes d'obtention des clés de chiffrement doivent bénéficier du même niveau ou d'un niveau de protection supérieur que les données chiffrées, sinon c'est inefficace, cela n'a plus beaucoup de sens

3. **Équilibre entre test et production** : les builds DEBUG doivent sauter le chiffrement pour faciliter le développement et le débogage, les builds RELEASE activent la protection complète, rappelez-vous d'utiliser la compilation conditionnelle `#if DEBUG` pour faire la distinction, cela facilite aussi le développement

4. **Considérations d'environnement Docker** : exécuter VMP dans un environnement Linux nécessite une solution conteneurisée, assurez la compatibilité des outils de protection, HagiCode utilise une solution de conteneur Wine + VMP, les problèmes multiplateforme sont ainsi résolus

5. **La vérification est indispensable** : après la build, il faut vérifier que la protection a réussi, sinon cela pourrait exposer du code sensible, le code de vérification précédent remplit ce rôle, mieux vaut vérifier

## Résumé

Grâce à cette stratégie de protection multicouche, HagiCode a réalisé un système complet de sécurité du code allant de l'obfuscation de base à la protection par machine virtuelle :

- **Première couche** : utiliser `ObfuscationAttribute` pour le marquage de base, fournir des indications aux outils tiers
- **Deuxième couche** : déclarer l'intention de protection et la priorité via `VmProtectAttribute` personnalisé
- **Troisième couche** : la protection par machine virtuelle VMP convertit le code clé en instructions de machine virtuelle irréversibles
- **Quatrième couche** : analyse et application automatiques de la protection au moment de la build, vérification des résultats de protection

Cette solution permet à la fois de résister aux outils de décompilation classiques et de contrer les attaques d'analyse inverse avancées. Si vous développez également des applications .NET nécessitant une protection du code, j'espère que cela pourra vous donner une référence, même une petite inspiration suffit.
