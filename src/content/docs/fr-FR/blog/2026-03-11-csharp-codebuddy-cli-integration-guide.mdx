## Guide pratique d'intégration de CodeBuddy CLI dans les backend C#

## Conception de l'architecture

### Vue d'ensemble de l'architecture en couches

L'intégration entre C# et CodeBuddy adopte une architecture en couches claire. Cette conception permet de séparer clairement les responsabilités du code, ce qui facilite également la maintenance ultérieure :

```
┌─────────────────────────────────────────────┐
│           Couche de contrat Provider         │
│   Énumération AIProviderType + méthodes d'extension             │
├─────────────────────────────────────────────┤
│           Couche de fabrique Provider         │
│   Fabrique d'injection de dépendances AIProviderFactory             │
├─────────────────────────────────────────────┤
│           Couche d'implémentation Provider   │
│   Implémentation concrète CodebuddyCliProvider              │
├─────────────────────────────────────────────┤
│           Couche d'infrastructure ACP        │
│  ACPSessionManager / StdioAcpTransport      │
│  AcpRpcClient / AcpAgentClient              │
└─────────────────────────────────────────────┘
```

Quel est l'avantage de ce découpage en couches ? En termes simples, les couches ne s'interfèrent pas entre elles. Supposons que vous deviez changer de mode de communication (par exemple, passer de stdio à WebSocket), vous n'avez qu'à modifier la couche la plus basse, le code métier supérieur reste complètement inchangé. Après tout, personne ne veut que changer une communication oblige à modifier tout le code métier pendant des heures, ce serait trop cruel.

### Analyse des composants principaux

La **couche de contrat Provider** est la pierre angulaire de toute l'architecture. Nous définissons l'énumération `AIProviderType`, où `CodebuddyCli = 3` sert de valeur d'énumération, et réalisons le mappage bidirectionnel entre chaînes et énumérations via des méthodes d'extension. Ainsi, les chaînes des fichiers de configuration peuvent être facilement converties en énumérations, et lors du débogage, les énumérations peuvent être converties en chaînes pour la sortie. C'est tout, c'est juste une relation de mappage, mais une fois bien fait, ça fait gagner du temps.

La **couche de fabrique Provider** est responsable de créer des instances Provider correspondantes en fonction de la configuration. Ici, nous utilisons le mécanisme d'injection de dépendances de .NET, avec `ActivatorUtilities.CreateInstance` pour réaliser la création dynamique. L'avantage du pattern de fabrique est que lors de l'ajout d'un nouveau Provider, il suffit d'ajouter la logique de création, sans modifier le code existant. C'est comme écrire un article, si vous voulez ajouter un nouveau chapitre, ajoutez-le, sans avoir à réécrire tout ce qui précède.

La **couche d'implémentation Provider** est l'endroit où le vrai travail se fait. `CodebuddyCliProvider` implémente l'interface `IAIProvider`, offrant deux modes d'appel : `ExecuteAsync` (non-streaming) et `StreamAsync` (streaming).

La **couche d'infrastructure ACP** constitue le support de communication de bas niveau. Cette couche traite tous les détails du protocole, y compris la gestion des processus, la sérialisation des messages, l'analyse des réponses, etc. Comme les fondations d'une maison, peu importe la beauté de ce qui est au-dessus, ce qui est dessous doit être stable.

## Mécanisme de communication

### Mode de transport Stdio

CodeBuddy utilise le mode **Stdio (entrée/sortie standard)** pour communiquer avec les processus externes. La commande de démarrage est simple :

```bash
codebuddy --acp
```

Ensuite, les messages JSON-RPC sont échangés via l'entrée/sortie standard. Les avantages de cette approche sont :

1. **Démarrage rapide** : la communication entre processus locaux n'a pas de latence réseau
2. **Configuration simple** : il suffit de spécifier le chemin de l'exécutable
3. **Isolation de l'environnement** : chaque session est un processus indépendant, sans interférence

Le processus de communication prend en charge l'injection de variables d'environnement, notamment :

- `CODEBUDDY_API_KEY` : authentification par clé API
- `CODEBUDDY_INTERNET_ENVIRONMENT` : configuration de l'environnement réseau

C'est comme, pour la communication entre personnes, il faut trouver un moyen pratique pour pouvoir parler.

### Protocole de messages

ACP est basé sur le protocole JSON-RPC 2.0, le format des messages ressemble à ceci :

```json
// Message de requête
{
  "jsonrpc": "2.0",
  "id": 1,
  "method": "agent/prompt",
  "params": {
    "prompt": "Aide-moi à écrire un algorithme de tri",
    "sessionId": "session-123"
  }
}

// Message de réponse
{
  "jsonrpc": "2.0",
  "id": 1,
  "result": {
    "content": "Voici la réponse de l'IA..."
  }
}
```

Dans l'implémentation réelle, nous avons encapsulé tous ces détails de protocole, le code métier de niveau supérieur n'a qu'à se concentrer sur le prompt et la réponse. C'est tout, une bonne encapsulation rend l'utilisation plus confortable pour les suivants.

### 2. Intégration de la fabrique Provider

Ajoutons la branche de création CodeBuddy dans la classe de fabrique :

```csharp
// AIProviderFactory.cs
private IAIProvider? CreateProvider(AIProviderType providerType, ProviderConfiguration config)
{
    return providerType switch
    {
        AIProviderType.CodebuddyCli =>
            ActivatorUtilities.CreateInstance<CodebuddyCliProvider>(
                _serviceProvider,
                Options.Create(config)),
        // ... autres providers
        _ => throw new NotSupportedException($"Provider {providerType} not supported")
    };
}
```

Ici, nous utilisons `ActivatorUtilities` de l'injection de dépendances, qui gère automatiquement l'injection des paramètres du constructeur, très pratique. C'est tout, avec .NET, quand c'est bien utilisé, ça fait gagner du temps.

### 3. Implémentation complète du Provider

Voici l'implémentation principale de `CodebuddyCliProvider`, incluant les deux modes d'appel streaming et non-streaming :

```csharp
public class CodebuddyCliProvider : IAIProvider
{
    private readonly ILogger<CodebuddyCliProvider> _logger;
    private readonly IACPSessionManager _sessionManager;
    private readonly ProviderConfiguration _config;

    public string Name => "CodebuddyCli";
    public bool SupportsStreaming => true;
    public ProviderCapabilities Capabilities { get; }

    public CodebuddyCliProvider(
        ILogger<CodebuddyCliProvider> logger,
        IACPSessionManager sessionManager,
        IOptions<ProviderConfiguration> config)
    {
        _logger = logger;
        _sessionManager = sessionManager;
        _config = config.Value;

        // Définir les capacités du Provider actuel
        Capabilities = new ProviderCapabilities
        {
            SupportsStreaming = true,
            SupportsTools = true,
            SupportsSystemMessages = true,
            SupportsArtifacts = false,
            MaxTokens = 8192
        };
    }

    // Appel non-streaming : attendre que tous les résultats soient renvoyés ensemble
    public async Task<AIResponse> ExecuteAsync(
        AIRequest request,
        CancellationToken cancellationToken = default)
    {
        // Créer une session indépendante pour la requête
        var session = await _sessionManager.CreateSessionAsync(
            "CodebuddyCli",
            request.WorkingDirectory,
            cancellationToken,
            request.SessionId);

        try
        {
            var fullPrompt = BuildPrompt(request);
            await session.SendPromptAsync(fullPrompt, cancellationToken);

            var responseBuilder = new StringBuilder();
            var toolCalls = new List<AIToolCall>();

            // Collecter tous les blocs de réponse
            await foreach (var chunk in StreamFromSession(session, cancellationToken))
            {
                if (!string.IsNullOrEmpty(chunk.Content))
                {
                    responseBuilder.Append(chunk.Content);
                }
                // Traitement des appels d'outils...
            }

            return new AIResponse
            {
                Content = AIResultContentSanitizer.SanitizeResultContent(
                    responseBuilder.ToString()),
                ToolCalls = toolCalls,
                Provider = Name,
                Model = string.Empty
            };
        }
        finally
        {
            // Libérer les ressources de la session
            await session.DisposeAsync();
        }
    }

    // Appel streaming : renvoyer les blocs de réponse en temps réel
    public async IAsyncEnumerable<AIStreamingChunk> StreamAsync(
        AIRequest request,
        [EnumeratorCancellation] CancellationToken cancellationToken = default)
    {
        var session = await _sessionManager.CreateSessionAsync(
            "CodebuddyCli",
            request.WorkingDirectory,
            cancellationToken);

        try
        {
            var fullPrompt = BuildPrompt(request);
            await session.SendPromptAsync(fullPrompt, cancellationToken);

            await foreach (var chunk in StreamFromSession(session, cancellationToken))
            {
                yield return chunk;
            }
        }
        finally
        {
            await session.DisposeAsync();
        }
    }

    private async IAsyncEnumerable<AIStreamingChunk> StreamFromSession(
        IACPSession session,
        [EnumeratorCancellation] CancellationToken cancellationToken)
    {
        // Parcourir toutes les mises à jour de la session
        await foreach (var notification in session.ReceiveUpdatesAsync(cancellationToken))
        {
            switch (notification.Update)
            {
                case AgentMessageChunkSessionUpdate agentMessage:
                    // Traiter les blocs de contenu texte
                    if (agentMessage.Content is AcpImp.TextContentBlock textContent)
                    {
                        yield return new AIStreamingChunk
                        {
                            Content = textContent.Text,
                            Type = StreamingChunkType.ContentDelta,
                            IsComplete = false
                        };
                    }
                    break;

                case ToolCallSessionUpdate toolCall:
                    // Traiter les appels d'outils
                    yield return new AIStreamingChunk
                    {
                        Content = string.Empty,
                        Type = StreamingChunkType.ToolCallDelta,
                        ToolCallDelta = new AIToolCallDelta
                        {
                            Id = toolCall.ToolCallId,
                            Name = toolCall.Kind.ToString(),
                            Arguments = toolCall.RawInput?.ToString()
                        }
                    };
                    break;

                case AcpImp.PromptCompletedSessionUpdate:
                    // Réponse terminée
                    yield break;
            }
        }
    }

    // Construire le prompt complet
    private string BuildPrompt(AIRequest request, string? embeddedCommandPrompt = null)
    {
        var sb = new StringBuilder();

        // Prompt de commande intégré (si présent)
        if (!string.IsNullOrEmpty(embeddedCommandPrompt))
        {
            sb.AppendLine(embeddedCommandPrompt);
            sb.AppendLine();
        }

        // Message système
        if (!string.IsNullOrEmpty(request.SystemMessage))
        {
            sb.AppendLine(request.SystemMessage);
            sb.AppendLine();
        }

        // Prompt utilisateur
        sb.Append(request.Prompt);
        return sb.ToString();
    }
}
```

Ce code présente plusieurs points clés :

1. **Gestion des sessions** : chaque requête crée une session indépendante, les ressources sont libérées après la fin de la requête. C'est une expérience acquise à travers les pièges — si la réutilisation des sessions n'est pas bien faite, il est facile d'avoir des problèmes de pollution d'état. Après tout, une fois utilisé, il faut ranger proprement, sinon le prochain utilisateur aura des problèmes.

2. **Traitement streaming** : `IAsyncEnumerable` permet de renvoyer la réponse pendant qu'elle est générée, sans attendre que tout le contenu soit généré. C'est particulièrement important pour les scénarios de texte long, l'expérience utilisateur sera bien meilleure. Comme ça, ceux qui attendent le résultat n'ont pas à attendre inutilement.

3. **Appels d'outils** : CodeBuddy prend en charge les appels d'outils (Function Calling), traités via `ToolCallSessionUpdate`. Cette capacité est essentielle pour les tâches complexes d'édition de code.

4. **Filtrage du contenu** : utilise `AIResultContentSanitizer` pour filtrer le contenu des blocs Think, gardant la sortie propre.

```csharp
public void ConfigureModule(IServiceCollection context)

    context.Services.AddSingleton<IAcpToAIResponseMapper, AcpToAIResponseMapper>();
}
```

## Exemples de configuration

### Fichier de configuration

Ajoutez la configuration liée à CodeBuddy dans `appsettings.json` :

```yaml
AI:
  # Provider utilisé par défaut
  DefaultProvider: "CodebuddyCli"

  # Configuration des Providers
  Providers:
    CodebuddyCli:
      Type: "CodebuddyCli"
      WorkingDirectory: "C:/projects/my-app"
      ExecutablePath: "C:/tools/codebuddy.cmd"

  # Configuration spécifique à la plateforme
  PlatformConfigurations:
    CodebuddyCli:
      ExecutablePath: "C:/tools/codebuddy.cmd"
      Arguments: "--acp"
      StartupTimeoutMs: 5000
      EnvironmentVariables:
        CODEBUDDY_API_KEY: "${CODEBUDDY_API_KEY}"
        CODEBUDDY_INTERNET_ENVIRONMENT: "production"
```

## Synthèse de l'expérience pratique

### Enregistrement des pièges rencontrés

Nous avons rencontré plusieurs pièges typiques lors de la mise en œuvre, partagés ici pour vous aider à éviter les détours. Après tout, éviter les pièges des autres est une bonne chose :

1. **Problème de fuite de sessions** : au début, nous ne libérions pas correctement les sessions, ce qui entraînait l'épuisement des ressources de processus. La solution consiste à utiliser `try-finally` pour garantir que chaque requête libérera les ressources. C'est tout, ce qui a été utilisé doit être remis en place, sinon que vont utiliser les suivants ?

2. **Transmission des variables d'environnement** : la syntaxe des variables d'environnement est différente entre Windows et Linux, nous avons ensuite unifié l'utilisation de `Dictionary<string, string?>` pour le traitement. Pour la multiplateforme, il faut unifier les normes dès le début, et ensuite on gagne du temps.

3. **Configuration du délai d'attente** : le démarrage du CLI prend du temps, nous avons défini un délai de démarrage de 5 secondes pour éviter l'échec des requêtes rapides. Il faut savoir mesurer dans tout, trop de précipitation et rien ne se fait.

4. **Problème d'encodage** : l'encodage par défaut sur Windows peut provoquer des caractères illisibles pour le chinois, nous avons spécifié explicitement l'encodage UTF-8 lors du démarrage du processus. Ne pas pouvoir afficher le chinois, ce serait vraiment désagréable.

### Optimisation des performances

1. **Pool de sessions** : pour les requêtes courtes fréquentes, envisagez d'implémenter un pool de sessions pour réutiliser les processus
2. **Cache de connexions** : la classe de fabrique prend déjà en charge le cache des instances Provider
3. **Priorité à l'asynchrone** : utilisation entièrement de la programmation asynchrone pour éviter de bloquer les threads

Les performances, quand on peut optimiser, on optimise, car plus l'utilisateur attend, pire est l'expérience.
