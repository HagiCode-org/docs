## Concevoir 12 expériences de démarrage exceptionnelles pour HagiCode : du minimalisme au cyberpunk

> Le court intervalle entre le téléchargement et l'hydratation d'une application React 19 est une fenêtre d'or pour permettre aux utilisateurs de percevoir la personnalité de la marque. Cet article partage notre expérience dans le projet HagiCode, basé sur un système complet de styles de démarrage construit avec HTML/CSS/JS.


## Contexte

En tant qu'application moderne basée sur ASP.NET Core 10 et React 19 (Vite), HagiCode adopte une architecture de déploiement séparée entre le frontend et le backend. Les artefacts frontend sont empaquetés et placés dans le répertoire `wwwroot/` du backend pour être hébergés par ASP.NET Core.

Cependant, cette architecture présente un point de douleur classique en termes d'expérience utilisateur : lorsqu'un utilisateur accède à la page web, le navigateur doit d'abord charger le HTML, puis télécharger l'énorme bundle JS, et enfin React exécute l'hydratation. Pendant ces quelques centaines de millisecondes à plusieurs secondes de "période de vide", l'utilisateur fait face à une page blanche ou à une page statique sans vie.

Pour combler cet intervalle et injecter la personnalité de la marque HagiCode, nous devions concevoir un système de styles de démarrage entièrement basé sur du code en ligne dans `index.html`.

## À propos de HagiCode

Le plan de conception d'écran de démarrage partagé dans cet article provient de notre expérience pratique dans le projet [HagiCode](https://github.com/HagiCode-org/site). En tant qu'assistant de code IA, HagiCode se concentre non seulement sur l'efficacité de la génération de code, mais accorde également une grande importance à l'expérience visuelle des développeurs. Ce système de démarrage est précisément le produit de notre quête de performances frontend ultimes.

## Défis centraux et conception de l'architecture

Avant de commencer la conception, nous devons d'abord clarifier les contraintes techniques. Puisque nous devons l'implémenter en ligne dans `index.html`, cela signifie que nous ne pouvons charger aucun fichier CSS ou JS externe (à l'exception du bundle React lui-même).

### Analyse des contraintes techniques

1.  **Principe de zéro dépendance** : Tous les styles doivent être écrits dans des balises `<style>`, et la logique dans des balises `<script>`.
2.  **CSS défensif** : Pour éviter que les styles globaux ne polluent l'écran de démarrage après le montage de l'application React, nous avons décidé d'utiliser des préfixes d'ID haute priorité (comme `#boot-screen`) pour envelopper tous les styles de démarrage.
3.  **Performance prioritaire** : Les animations doivent utiliser autant que possible CSS `transform` et `opacity` pour éviter de déclencher des reflows et garantir de ne pas bloquer le thread principal.
4.  **Cohérence visuelle** : Les couleurs et les polices doivent rester cohérentes avec la configuration Tailwind de HagiCode.

### Modèle d'architecture : Shell & Injector

Nous avons adopté un **modèle de variante**. La logique centrale est encapsulée dans une fonction immédiatement exécutée (IIFE), et la logique de rendu spécifique est injectée en tant qu'éléments de configuration. Cela nous permet de basculer entre différents styles par une simple configuration sans avoir à répéter la logique de manipulation du DOM.

Voici le code de l'architecture centrale :

```html
<!-- Inline dans index.html -->
<div id="boot-root"></div>

<script>
(function() {
  const BootSequence = {
    config: {
      theme: 'terminal', // Peut être configuré comme 'minimal', 'skeleton', 'code-rain', etc.
      color: '#3b82f6'   // Couleur de la marque
    },
    
    // Cycle de vie central
    init() {
      this.render();
      this.listenForMount();
    },

    // Rend le style actuellement sélectionné
    render() {
      const root = document.getElementById('boot-root');
      if (this.variants[this.config.theme]) {
        root.innerHTML = this.variants[this.config.theme].render();
      }
    },

    // Écoute le succès du montage React, sortie élégante
    listenForMount() {
      window.addEventListener('hagicode:ready', () => {
        const screen = document.getElementById('boot-root');
        // D'abord faire disparaître en fondu, puis supprimer le DOM pour éviter les scintillements
        screen.style.opacity = '0';
        screen.style.transition = 'opacity 0.3s ease';
        setTimeout(() => screen.remove(), 300);
      });
    },

    // La logique d'implémentation des 12 styles est concentrée ici
    variants: {
      // ... implémentation spécifique voir ci-dessous
    }
  };

  BootSequence.init();
})();
</script>
```

## Liste de 12 styles de démarrage

Nous avons divisé ces 12 styles en six grandes catégories pour répondre à différents scénarios et besoins esthétiques.

### A. Minimalisme

> "Moins c'est plus". Pour les scénarios qui poursuivent une vitesse de chargement ultime, nous fournissons la solution la plus légère.

#### 1. Minimalist Dot (Point minimaliste respirant)
Un seul point simple au centre de l'écran, avec une animation de respiration.
*   **Implémentation** : CSS `@keyframes` contrôlant scale et opacity.
*   **Applicable** : Toute occasion où la page doit rester absolument propre.

#### 2. Brand Reveal (Révélation de la marque)
Via l'animation SVG `stroke-dasharray`, simule le dessin à la main des lignes du logo HagiCode, suivi d'un fondu du texte.
*   **Technique** : Utilisation de l'animation de chemin SVG, très texturé.

### B. Mimétisme d'écran squelette

> "L'art de tromper l'œil". En simulant la vraie disposition de l'interface utilisateur, donnant aux utilisateurs l'impression que la page est déjà chargée à moitié.

#### 3. Sidebar Chat Skeleton (Écran squelette de barre latérale et chat)
C'est probablement le plus pratique. Nous construisons manuellement avec HTML une disposition identique aux composants React `Sidebar` et `ChatInput`, et couvrons avec une animation de rayures grises.
*   **Valeur** : Lorsque l'hydratation React est terminée, l'écran squelette se transforme instantanément en composants réels, et l'utilisateur ne ressent presque pas la transition.

#### 4. Card Stack Skeleton (Pile de cartes)
Simule l'effet d'empilement lors du chargement des cartes de propositions, utilisant des transformations 3D pour faire flotter légèrement les cartes.

### C. Abstrait et artistique

> Montrer les gènes de geek de HagiCode.

#### 5. Geometric Morph (Morphose géométrique)
Rend une forme géométrique (carré) au centre de l'écran, qui se transforme progressivement en cercle, puis en triangle, et enfin devient le logo.
*   **Technique** : Transition fluide de CSS `border-radius`.

#### 6. Code Rain (Pluie de code)
Hommage à "Matrix". Utilise la police JetBrains Mono, faisant tomber un flux de caractères léger en arrière-plan.
*   **Attention** : Pour les performances, le flux de caractères doit être limité à une petite zone ou réduire la fréquence de rafraîchissement.

#### 7. Neon Pulse (Pulsation néon)
Un cercle lumineux de style cyberpunk, utilisant le superposition multiple de `box-shadow` pour produire une forte sensation de luminescence.

### D. Marque et thème

> Rendre le système "vivant".

#### 8. Seasonal Theme (Thème saisonnier)
C'est un chargeur dynamique. Détermine les vacances en fonction de la date actuelle (comme le Nouvel An chinois, Noël), charge l'animation SVG correspondante.
*   **Exemple** : Pendant le Nouvel An chinois, des lanternes rouges oscillent doucement en bas de l'écran.

#### 9. Gradient Flow (Flux de dégradé)
Le fond utilise un flux fluide des couleurs de la marque HagiCode, combiné avec l'animation de `background-size` et `background-position`, créant une sensation de流动 comme des aurores boréales.

### E. Sensation technique

> Hommage aux développeurs.

#### 10. Terminal Boot (Démarrage terminal)
Simule la sortie de la console. Les lignes de code défilent rapidement :
```text
> Initializing HagiCode Core...
> Loading models...
> Connecting to neural network...
```
Cela rendra chaque développeur se sentir familier.

#### 11. Progress Bar (Barre de progression minimaliste)
Une fine barre de progression en haut de l'écran, avec le pourcentage affiché à droite. Bien que nous ne puissions pas obtenir la vraie progression du téléchargement, nous pouvons utiliser une minuterie pour simuler un processus de chargement "crédible" (rapide pour les 80% premiers, ralentissement pour les 20% derniers).

### F. Créativité

#### 12. Pixel Assembly (Assemblage de pixels)
C'est une idée très intéressante. Des carrés sont dispersés sur l'écran, ils convergent vers le centre, et assemblent progressivement l'icône du logo HagiCode. Symbolisant le processus de construction du code.

## Meilleures pratiques et résumé des pièges

Dans le développement réel de HagiCode, nous avons résumé certains détails de pratique cruciaux.

### 1. Le CSS défensif est indispensable
Ne soyez pas paresseux et n'oubliez pas d'écrire les préfixes. Une fois, nous n'avons pas limité les styles de l'écran de démarrage avec des ID, ce qui a entraîné que les styles globaux `div` après le montage React ont affecté par inadvertance l'écran de démarrage, provoquant une rupture de la disposition.
**Expérience** : Tous les sélecteurs CSS sont suspendus sous `#boot-screen`, et utilisez `!important` pour augmenter la priorité (seulement dans le CSS de l'écran de démarrage).

### 2. Transition élégante
Après le succès du montage React, ne `remove()` pas directement le DOM de l'écran de démarrage.
**Bonne pratique** :
1.  React déclenche `window.dispatchEvent(new Event('hagicode:ready'))`.
2.  L'écran de démarrage écoute l'événement, définit d'abord `opacity: 0`.
3.  Attendez 300ms (temps de transition CSS), assurez-vous que l'utilisateur ne voit plus, puis exécutez `.remove()`.

### 3. Synchronisation des variables de thème
Le code couleur de l'écran de démarrage est écrit en dur dans `index.html`. Si nous modifions la couleur principale de Tailwind, nous devons la modifier ici également.
**Solution d'optimisation** : Dans le script de construction Vite, écrivez un plugin simple, lisez `tailwind.config.js` et injectez les variables de couleur dans les variables de modèle de `index.html`, réalisant une source unique de données.

### 4. Préchargement des polices
L'écran de démarrage doit généralement utiliser la police de la marque, mais si le chargement de la police est lent, il y aura un FOUT (Flash of Unstyled Text).
**Solution** : Ajoutez `<link rel="preload" href="/fonts/JetBrainsMono.woff2" as="font" type="font/woff2" crossorigin>` dans `<head>`. C'est un moyen à faible coût et à rendement élevé pour améliorer l'expérience.

### 5. Surveillance des performances
Nous avons injecté `performance.mark('boot-start')` en bas de `index.html`, et marqué `boot-end` lorsque le montage React réussit.
**Signification** : En collectant ces données via Application Insights, nous pouvons voir réellement dans quelle mesure l'écran de démarrage réduit le temps d'attente perçu par l'utilisateur (Perceived Loading Time). Les données montrent qu'un excellent écran squelette peut augmenter la tolérance des utilisateurs aux "réseaux lents" de plus de 50%.

## Conclusion

Un bon écran de démarrage n'est pas seulement une "décoration pendant l'attente", c'est le signal de poignée de main de la première interaction entre le produit et l'utilisateur. Dans le projet HagiCode, ce système de démarrage basé sur le **mode Variants** nous permet de basculer flexibles entre différents styles pour différentes vacances et différentes versions, améliorant grandement le plaisir et le professionnalisme du produit.

La solution partagée dans cet article est entièrement basée sur les normes Web natives, sans introduire de dépendances lourdes, ce qui est exactement le reflet de la poursuite de "léger et puissant" par HagiCode.
