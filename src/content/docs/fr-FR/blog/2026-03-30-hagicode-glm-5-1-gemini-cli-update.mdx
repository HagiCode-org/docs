## Prise en charge complète de GLM-5.1 et intégration de Gemini CLI : l'évolution multi-modèle de HagiCode

> Cet article présente les mises à jour récentes de la plateforme HagiCode - la prise en charge complète du modèle GLM-5.1 de Zhipu AI, ainsi que l'intégration réussie de Gemini CLI en tant que dixième Agent CLI. Ces deux mises à jour renforcent davantage les capacités multi-modèles et l'écosystème multi-CLI de la plateforme.

:::note[Note sur le contexte temporel]
Cet article documente l'état d'extension multi-CLI au moment de la publication. ## Contexte

Le temps passe vite, le développement des grands modèles de langage progresse comme le bambou au printemps, grimpant rapidement. Auparavant, nous acclamions encore "une IA capable d'écrire du code", aujourd'hui nous sommes déjà à l'époque de la collaboration multi-modèles et de la fusion multi-outils. Est-ce intéressant ? Peut-être, après tout, ce dont les développeurs ont besoin n'a jamais été seulement les outils eux-mêmes, mais une capacité d'adaptation à différents scénarios et de bascule flexible avec aisance.

En tant que plateforme de programmation assistée par IA, HagiCode a récemment connu deux événements majeurs : premièrement, l'intégration complète du modèle GLM-5.1 de Zhipu AI, et deuxièmement, Gemini CLI devenant officiellement le dixième Agent CLI supporté. Ces deux événements, ni grands ni petits, sont finalement une bonne chose pour l'amélioration de la plateforme.

GLM-5.1 est le dernier modèle flagship de Zhipu AI. Par rapport à GLM-5.0, ses capacités de raisonnement sont plus fortes, sa compréhension du code plus profonde, et ses appels d'outils plus fluides. Plus important encore, c'est le premier modèle GLM à prendre en charge les entrées d'images - qu'est-ce que cela signifie ? Cela signifie que les utilisateurs peuvent directement faire des captures d'écran pour montrer le problème à l'IA, sans avoir à décrire péniblement. Cette commodité, on la comprend après l'avoir utilisée.

Pendant ce temps, HagiCode a intégré avec succès Gemini CLI via l'architecture HagiCode.Libs.Providers. C'est le dixième Agent CLI, honnêtement, arriver à ce stade procure un certain sentiment d'accomplissement.

Il convient de mentionner que la fonction de téléchargement d'images de HagiCode permet aux utilisateurs de communiquer directement avec l'IA par captures d'écran. Même en exécutant la version GLM 4.7, la plateforme fonctionne toujours bien et a déjà aidé le projet à terminer de nombreux travaux de construction importants. Quant à GLM-5.1 ? Ce sera naturelment un pas de plus.

## À propos de HagiCode

La solution partagée dans cet article provient de notre expérience pratique dans le projet [HagiCode](https://hagicode.com). HagiCode est une plateforme de programmation assistée par IA open source, visant à fournir aux développeurs un assistant de programmation IA flexible et puissant grâce à une architecture multi-modèles et multi-CLI. Adresse du projet : [github.com/HagiCode-org/site](https://github.com/HagiCode-org/site)

## Conception d'architecture multi-CLI

L'un des avantages principaux de HagiCode est de prendre en charge plusieurs outils CLI de programmation IA différents via une couche d'abstraction unifiée. Les avantages de cette conception, pour le dire simplement, sont : les nouvelles choses peuvent entrer, les anciennes peuvent rester, et le code ne devient pas désordonné. Après tout, tout le monde souhaite que la vie soit ainsi, non ?

### Énumération AIProviderType

La plateforme définit les types de fournisseurs CLI supportés via l'énumération `AIProviderType` :

```csharp
public enum AIProviderType
{
    ClaudeCodeCli = 0,    // Claude Code CLI
    CodexCli = 1,          // GitHub Copilot Codex
    GitHubCopilot = 2,     // GitHub Copilot
    CodebuddyCli = 3,     // Codebuddy CLI
    OpenCodeCli = 4,      // OpenCode CLI
    IFlowCli = 5,         // IFlow CLI
    HermesCli = 6,        // Hermes CLI
    QoderCli = 7,         // Qoder CLI
    KiroCli = 8,          // Kiro CLI
    KimiCli = 9,          // Kimi CLI
    GeminiCli = 10,       // Gemini CLI (nouveau)
}
```

On peut voir que Gemini CLI a rejoint cette grande famille en tant que dixième membre. Chaque CLI a des caractéristiques uniques et des scénarios d'application appropriés, et les utilisateurs peuvent choisir flexiblement selon leurs besoins. Après tout, tous les chemins mènent à Rome, certains sont juste plus faciles à parcourir, d'autres un peu plus sinueux.

### Architecture Provider

HagiCode.Libs.Providers fournit une interface Provider unifiée, rendant l'intégration de chaque CLI standard et concise. Prenons l'exemple de Gemini CLI :

```csharp
public class GeminiProvider : ICliProvider<GeminiOptions>
{
    private static readonly string[] DefaultExecutableCandidates = ["gemini", "gemini-cli"];
    private const string ManagedBootstrapArgument = "--acp";

    public string Name => "gemini";
    public bool IsAvailable => _executableResolver.ResolveFirstAvailablePath(DefaultExecutableCandidates) is not null;
}
```

Les avantages de cette conception sont :
- L'intégration de nouvelles CLI nécessite seulement d'implémenter une classe Provider
- Gestion unifiée du cycle de vie et mise en pool des sessions
- Résolution automatique des alias et recherche des exécutables

Pour le dire simplement, cette conception consiste en fait à simplifier les choses complexes et à rendre la vie un peu plus facile.

### Provider Registry

Le Provider Registry gère automatiquement le mappage et l'enregistrement des alias :

```csharp
if (provider is GeminiProvider)
{
    registry.Register(provider.Name, provider, ["gemini-cli"]);
    continue;
}
```

Cela signifie que les utilisateurs peuvent utiliser `gemini` ou `gemini-cli` pour appeler Gemini CLI, le système reconnaîtra automatiquement. C'est comme quand vous avez beaucoup d'amis, certains appellent par le grand nom, d'autres par le surnom, c'est toujours lui, on peut appeler comme on veut.

## Prise en charge du modèle GLM-5.1

GLM-5.1 est le dernier modèle flagship de Zhipu AI, et HagiCode a terminé sa prise en charge complète.

### Catalogue des professions secondaires

HagiCode gère tous les modèles supportés via le Secondary Professions Catalog. Voici la configuration de la série GLM :

| ID du modèle | Nom | Prise en charge des images | Familles CLI compatibles |
|--------------|-----|----------------------------|--------------------------|
| `glm-4.7` | GLM 4.7 | - | claude, codebuddy, hermes, qoder, kiro |
| `glm-5` | GLM 5 | - | claude, codebuddy, hermes, qoder, kiro |
| `glm-5-turbo` | GLM 5 Turbo | - | claude, codebuddy, hermes, qoder, kiro |
| `glm-5.0` | GLM 5.0 (Legacy) | - | claude, codebuddy, hermes, qoder, kiro |
| `glm-5.1` | GLM 5.1 | **true** | claude, codebuddy, hermes, qoder, kiro |

Les caractéristiques clés de GLM-5.1 peuvent être résumées comme suit :
- Identificateur de version indépendant, sans fardeau legacy
- Premier modèle GLM à prendre en charge les entrées d'images
- Capacités de raisonnement plus fortes et compréhension du code
- Large compatibilité multi-CLI

### GLM-5.1 vs GLM-5.0

Du point de vue du code, la différence clé entre GLM-5.1 et GLM-5.0 :

```csharp
// GLM-5.0 (Legacy) - a une logique de rétention spéciale
private const string Glm50CodebuddySecondaryProfessionId = "secondary-glm-5-codebuddy";
private const string Glm50CodebuddyModelValue = "glm-5.0";

// GLM-5.1 - nouvel identificateur de modèle indépendant
private const string Glm51SecondaryProfessionId = "secondary-glm-5-1";
private const string Glm51ModelValue = "glm-5.1";
```

GLM-5.0 porte la marque "Legacy", un identificateur d'ancienne version conservé pour la rétrocompatibilité. Tandis que GLM-5.1 est une toute nouvelle version indépendante, sans aucun bagage historique. C'est comme certaines personnes, toujours vivant dans le passé ; et d'autres, partant léger, allant plus vite.

### Configuration de GLM-5.1

Exemple de configuration pour utiliser GLM-5.1 dans HagiCode :

```json
{
  "primaryProfessionId": "profession-claude-code",
  "secondaryProfessionId": "secondary-glm-5-1",
  "model": "glm-5.1",
  "reasoning": "high"

```

## Fonction de téléchargement d'images

La prise en charge des images de HagiCode est implémentée via la propriété `SupportsImage` de SecondaryProfession :

```csharp
public class HeroSecondaryProfessionSettingDto
{
    public bool SupportsImage { get; set; }
}
```

Dans le Secondary Professions Catalog, la configuration de GLM-5.1 est la suivante :

```json
{
  "id": "secondary-glm-5-1",
  "supportsImage": true
}
```

Cela signifie que les utilisateurs peuvent directement télécharger des captures d'écran pour que l'IA les analyse, par exemple :
- Captures d'écran de messages d'erreur
- Problèmes d'interface utilisateur
- Graphiques de visualisation de données
- Résultats d'exécution de code

Plus besoin de décrire manuellement le problème, il suffit de faire une capture d'écran - la commodité de cette fonction se comprend après l'avoir utilisée. Après tout, pour certaines choses, voir vaut mieux que mille mots.

## Intégration de Gemini CLI

Gemini CLI, en tant que dixième Agent CLI, est intégré à HagiCode via l'architecture Provider standard.

### Options de configuration

Gemini CLI prend en charge de nombreuses options de configuration :

```csharp
public class GeminiOptions
{
    public string? ExecutablePath { get; set; }
    public string? WorkingDirectory { get; set; }
    public string? SessionId { get; set; }
    public string? Model { get; set; }
    public string? AuthenticationMethod { get; set; }
    public string? AuthenticationToken { get; set; }
    public Dictionary<string, string?> AuthenticationInfo { get; set; }
    public Dictionary<string, string?> EnvironmentVariables { get; set; }
    public string[] ExtraArguments { get; set; }
    public TimeSpan? StartupTimeout { get; set; }
    public CliPoolSettings? PoolSettings { get; set; }
}
```

Ces options couvrent tous les aspects, de la configuration de base aux fonctionnalités avancées, et les utilisateurs peuvent configurer flexiblement selon leurs besoins. Après tout, les besoins de chacun sont différents, pouvoir être flexible est toujours une bonne chose.

### Protocole de communication ACP

Gemini CLI prend en charge le protocole de communication ACP (Agent Communication Protocol), qui est le standard de communication CLI unifié de HagiCode. Via ACP, différents CLI peuvent interagir avec la plateforme de manière cohérente, simplifiant grandement le travail d'intégration. Pour le dire simplement, il s'agit d'uniformiser les choses complexes pour que tout le monde puisse se simplifier la vie.

## Meilleures pratiques d'intégration multi-CLI

HagiCode.Libs.Providers fournit un mécanisme d'enregistrement et d'utilisation unifié :

```csharp
services.AddHagiCodeLibs();

var gemini = serviceProvider.GetRequiredService<ICliProvider<GeminiOptions>>();
var codebuddy = serviceProvider.GetRequiredService<ICliProvider<CodebuddyOptions>>();
var hermes = serviceProvider.GetRequiredService<ICliProvider<HermesOptions>>();
```

Cette conception d'injection de dépendances rend l'utilisation de chaque CLI très concise, et facilite également les tests unitaires et les simulations. Après tout, écrire du code proprement est aussi une responsabilité envers soi-même.

### Précautions

Dans l'utilisation réelle, il y a plusieurs points à noter :

1. **Configuration de la clé API** : Assurez-vous de configurer correctement `ANTHROPIC_AUTH_TOKEN`, sinon vous ne pourrez pas appeler le modèle
2. **Disponibilité du modèle** : GLM-5.1 doit être activé auprès du fournisseur de modèles correspondant
3. **Fonctionnalité d'image** : Seuls les modèles prenant en charge `supportsImage: true` peuvent utiliser la fonction de téléchargement d'images
4. **Installation de CLI** : Avant d'utiliser Gemini CLI, assurez-vous que `gemini` ou `gemini-cli` est dans le PATH du système

Ce sont des petites choses, mais si les petites choses ne sont pas bien gérées, elles peuvent devenir de grandes choses. Il faut donc faire attention.
