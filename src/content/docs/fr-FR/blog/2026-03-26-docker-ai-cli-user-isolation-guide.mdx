## Exécuter des outils CLI IA dans des conteneurs Docker : guide pratique de l'isolation utilisateur et des volumes persistants

> Intégrer des outils de codage IA comme Claude Code, Codex et OpenCode dans des environnements conteneurisés semble simple, mais il y a des complexités cachées partout. Cet article examine en profondeur la façon dont le projet HagiCode résout les défis fondamentaux des déploiements Docker, notamment les permissions utilisateur, la persistance de la configuration et la gestion des versions, afin que vous puissiez éviter les pièges courants.

## Contexte

Quand nous avons décidé d'exécuter des outils CLI de codage IA dans des conteneurs Docker, la pensée la plus intuitive était probablement : « Les conteneurs ne tournent-ils pas en root ? Pourquoi ne pas tout installer directement et en finir ? » En réalité, cette idée en apparence simple cache plusieurs problèmes fondamentaux qu'il faut résoudre.

Premièrement, **les restrictions de sécurité sont le premier obstacle**. Prenons Claude CLI comme exemple : il interdit explicitement l'exécution en tant qu'utilisateur root. C'est un contrôle de sécurité obligatoire, et si root est détecté, il refuse de démarrer. Vous pourriez penser : ne puis-je pas simplement changer d'utilisateur avec la directive `USER` ? Ce n'est pas si simple. Il reste un problème de mappage entre l'utilisateur non-root à l'intérieur du conteneur et les permissions utilisateur sur la machine hôte.

Deuxièmement, **la persistance de l'état est le deuxième piège**. Claude Code nécessite une connexion, Codex a sa propre configuration, et OpenCode a aussi un répertoire de cache. Si vous devez tout reconfigurer à chaque redémarrage du conteneur, toute l'idée d'« automatisation » perd son sens. Nous avons besoin que ces configurations persistent au-delà du cycle de vie du conteneur.

Le troisième problème est la **cohérence des permissions**. Les processus à l'intérieur du conteneur peuvent-ils accéder aux fichiers de configuration créés par l'utilisateur hôte ? Les inadéquations UID/GID provoquent souvent des erreurs de permissions de fichiers, et c'est extrêmement courant dans les déploiements réels.

Ces problèmes peuvent sembler indépendants, mais en pratique ils sont étroitement liés. Pendant le développement de HagiCode, nous avons progressivement élaboré une solution pratique. Ensuite, je partagerai les détails techniques et les leçons apprises de ces pièges.

HagiCode est une plateforme de programmation assistée par IA open source qui intègre plusieurs assistants de codage IA mainstream, notamment Claude Code, Codex et OpenCode. En tant que projet nécessitant un déploiement multiplateforme et hautement disponible, HagiCode doit résoudre l'ensemble des défis liés au déploiement conteneurisé.

Si vous trouvez la solution technique de cet article précieuse, c'est un signe que HagiCode a quelque chose de réel à offrir en matière de pratique d'ingénierie. Dans ce cas, le [site officiel de HagiCode](https://hagicode.com) et le [dépôt GitHub](https://github.com/HagiCode-org/site) valent tous deux la peine d'être suivis.

## Pourquoi ne pas simplement utiliser root ?

Il y a ici un malentendu courant : les conteneurs Docker tournent en root par défaut, alors pourquoi ne pas installer les outils en root ? Si vous pensez ainsi, Claude CLI vous montrera rapidement le contraire.

```bash
# Exécuter Claude CLI directement en root ? Non.
docker run --rm -it --user root myimage claude
# Sortie : Error: This command cannot be run as root user
```

C'est une restriction de sécurité stricte dans Claude CLI. La raison est simple : ces outils CLI lisent et écrivent des configurations utilisateur sensibles, y compris des tokens API, des caches locaux et même des scripts écrits par l'utilisateur. Les exécuter avec des privilèges root introduit trop de risques.

La question devient donc : comment satisfaire les exigences de sécurité du CLI tout en gardant la gestion des conteneurs flexible ? Nous devons changer notre façon de penser : **au lieu de changer d'utilisateur à l'exécution, créer un utilisateur dédié pendant la phase de build de l'image**.

## Créer un utilisateur dédié : bien plus que changer un nom

Vous pourriez penser qu'ajouter une seule ligne `USER` au Dockerfile suffit. C'est effectivement l'approche la plus simple, mais elle n'est pas assez robuste.

### Création statique vs mappage dynamique

L'approche de HagiCode est de créer un utilisateur `hagicode` avec UID 1000, qui correspond généralement à l'utilisateur par défaut sur la plupart des machines hôtes :

```dockerfile
RUN groupadd -o -g 1000 hagicode && \
    useradd -o -u 1000 -g 1000 -s /bin/bash -m hagicode && \
    mkdir -p /home/hagicode/.claude && \
    chown -R hagicode:hagicode /home/hagicode
```

Mais cela ne résout que l'utilisateur intégré dans l'image. Que se passe-t-il si l'utilisateur hôte a l'UID 1001 ? Vous avez toujours besoin de supporter le mappage dynamique au démarrage du conteneur.

docker-entrypoint.sh contient la logique clé :

```bash
if [ -n "$PUID" ] && [ -n "$PGID" ]; then
    if ! id hagicode >/dev/null 2>&1; then
        groupadd -g "$PGID" hagicode
        useradd -u "$PUID" -g "$PGID" -s /bin/bash -m hagicode
    fi
fi
```

L'avantage de cette conception est clair : **utiliser l'UID 1000 par défaut au moment du build de l'image, puis ajuster dynamiquement à l'exécution via les variables d'environnement `PUID` et `PGID`**. Quel que soit l'UID de l'utilisateur hôte, la propriété des fichiers de configuration reste correcte.

## La philosophie de conception des volumes persistants

Chaque outil CLI IA a son propre répertoire de configuration préféré, ils doivent donc être mappés un par un :

| Outil CLI | Chemin dans le conteneur | Volume nommé |
|---------|-----------|--------|
| Claude | `/home/hagicode/.claude` | `claude-data` |
| Codex | `/home/hagicode/.codex` | `codex-data` |
| OpenCode | `/home/hagicode/.config/opencode` | `opencode-config-data` |

Pourquoi utiliser des volumes nommés plutôt que des bind mounts ? Trois raisons :

1.  **Gestion plus simple** : les volumes nommés sont gérés automatiquement par Docker, vous n'avez donc pas besoin de créer les répertoires hôtes manuellement.
2.  **Isolation des permissions** : le contenu initial des volumes est créé par l'utilisateur à l'intérieur du conteneur, évitant les conflits de permissions avec l'hôte.
3.  **Migration indépendante** : les volumes peuvent exister indépendamment des conteneurs, les données ne sont donc pas perdues lors de la mise à niveau des images.

docker-compose-builder-web génère automatiquement la configuration de volumes correspondante :

```yaml
volumes:
  claude-data:
  codex-data:
  opencode-config-data:

services:
  hagicode:
    volumes:
      - claude-data:/home/hagicode/.claude
      - codex-data:/home/hagicode/.codex
      - opencode-config-data:/home/hagicode/.config/opencode
    user: "${PUID:-1000}:${PGID:-1000}"
```

Faites attention au champ `user` ici : `PUID` et `PGID` sont injectés via des variables d'environnement pour garantir que les processus à l'intérieur du conteneur s'exécutent avec une identité correspondant à l'utilisateur hôte. Ce détail compte car les problèmes de permissions sont douloureux à déboguer une fois qu'ils apparaissent.

## Gestion des versions : versions intégrées avec remplacement à l'exécution

Épingler les versions des images Docker est essentiel pour la reproductibilité. Mais en développement réel, nous avons souvent besoin de tester une version plus récente ou de corriger un bug en urgence. Si nous devions reconstruire l'image à chaque fois, le workflow serait bien trop inefficace.

La stratégie de HagiCode est des **versions fixes par défaut, avec des remplacements à l'exécution comme mécanisme d'extension**. C'est un compromis d'ingénierie pragmatique entre stabilité et flexibilité.

Dockerfile.template éping les versions ici :

```dockerfile
USER hagicode
WORKDIR /home/hagicode

# Configure the global npm install path
RUN mkdir -p /home/hagicode/.npm-global && \
    npm config set prefix '/home/hagicode/.npm-global'

# Install CLI tools using pinned versions
RUN npm install -g @anthropic-ai/claude-code@2.1.71 && \
    npm install -g @openai/codex@0.112.0 && \
    npm install -g opencode-ai@1.2.25 && \
    npm cache clean --force
```

docker-entrypoint.sh supporte les remplacements à l'exécution :

```bash
install_cli_override_if_needed() {
    local package_name="$2"
    local override_version="$5"

    if [ -n "$override_version" ]; then
        gosu hagicode npm install -g "${package_name}@${override_version}"
    fi
}

# Example usage
install_cli_override_if_needed "" "@anthropic-ai/claude-code" "" "" "${CLAUDE_CODE_CLI_VERSION}"
```

Cela vous permet de tester une nouvelle version via une variable d'environnement sans reconstruire l'image :

```bash
docker run -e CLAUDE_CODE_CLI_VERSION=2.2.0 myimage
```

Cette conception est pratique car personne ne veut reconstruire une image à chaque fois qu'il teste une nouvelle fonctionnalité.

## Injection automatique de la configuration

En plus de configurer manuellement les outils CLI, certains scénarios nécessitent une injection automatique de la configuration. L'exemple le plus typique est un token API.

```bash
if [ -n "$ANTHROPIC_AUTH_TOKEN" ]; then
    mkdir -p /home/hagicode/.claude
    cat > /home/hagicode/.claude/settings.json <<EOF
{
  "env": {
    "ANTHROPIC_AUTH_TOKEN": "${ANTHROPIC_AUTH_TOKEN}"
  }
}
EOF
    chown -R hagicode:hagicode /home/hagicode/.claude
fi
```

Deux choses comptent ici : **passer les informations sensibles via des variables d'environnement plutôt que de les coder en dur dans l'image**, et s'assurer que la propriété des fichiers de configuration est correctement définie, sinon les outils CLI ne pourront pas les lire.

## Bonnes pratiques et liste de contrôle des pièges

### Problèmes d'inadéquation des permissions

C'est le piège le plus facile dans lequel tomber. L'utilisateur hôte a l'UID 1001, tandis que le conteneur utilise 1000, les fichiers créés d'un côté ne peuvent donc pas être accessés de l'autre.

```bash
# Approche correcte : faire correspondre le conteneur à l'utilisateur hôte
docker run \
    -e PUID=$(id -u) \
    -e PGID=$(id -g) \
    myimage
```

Ce problème est très courant et peut être frustrant la première fois que vous le rencontrez.

### La configuration disparaît après le redémarrage du conteneur

Si vous vous retrouvez à vous reconnecter après chaque redémarrage, vérifiez si vous avez oublié de monter un volume persistant :

```yaml
volumes:
  - claude-data:/home/hagicode/.claude
```

Rien n'est plus frustrant que de configurer soigneusement une configuration pour la voir disparaître.

### La bonne façon de mettre à jour les versions

N'exécutez pas `npm install -g` directement dans un conteneur en cours d'exécution. Les approches correctes sont :

1. Définir une variable d'environnement pour déclencher l'installation de remplacement.
2. Ou reconstruire l'image.

```bash
# Option 1 : remplacement à l'exécution
docker run -e CLAUDE_CODE_CLI_VERSION=2.2.0 myimage

# Option 2 : reconstruire l'image
docker build -t myimage:v2 .
```

Il y a plus d'un chemin pour aller à Rome, mais certains sont plus lisses que d'autres.

### Liste de contrôle de renforcement de la sécurité

- Passer les tokens API via des variables d'environnement plutôt que de les écrire dans l'image.
- Définir les permissions des fichiers de configuration à `600`.
- Toujours exécuter l'application en tant qu'utilisateur non-root.
- Mettre à jour régulièrement les versions des CLI pour corriger les vulnérabilités de sécurité.

La sécurité est toujours importante, mais le vrai défi est de l'appliquer systématiquement en pratique.

## Étendre le support pour de nouveaux outils CLI

Si vous souhaitez supporter un nouvel outil CLI à l'avenir, il n'y a que trois étapes :

1.  **Dockerfile.template** : ajouter l'étape d'installation.
2.  **docker-entrypoint.sh** : ajouter la logique de remplacement de version.
3.  **docker-compose-builder-web** : ajouter le mappage de volume persistant.

Cette conception basée sur des templates rend l'extension simple sans modifier la logique principale.

## Conclusion

L'exécution d'outils CLI IA dans des conteneurs Docker implique trois défis fondamentaux : **les permissions utilisateur, la persistance de la configuration et la gestion des versions**. En combinant des utilisateurs dédiés, l'isolation par volumes nommés et les remplacements basés sur des variables d'environnement, le projet HagiCode a construit une architecture de déploiement à la fois sécurisée et flexible.

Points de conception clés :

- **Isolation utilisateur** : créer un utilisateur dédié pendant la phase de build de l'image, avec un support à l'exécution pour le mappage dynamique `PUID`/`PGID`.
- **Stratégie de persistance** : chaque outil CLI obtient son propre volume nommé, les redémarrages n'affectent donc pas la configuration.
- **Flexibilité des versions** : des valeurs par défaut fixes garantissent la reproductibilité, tandis que les remplacements à l'exécution offrent une marge pour les tests.
- **Configuration automatisée** : les configurations sensibles peuvent être injectées automatiquement via des variables d'environnement.

Cette solution fonctionne de manière stable dans le projet HagiCode depuis un certain temps, et j'espère qu'elle offrira des points de référence utiles aux développeurs ayant des besoins similaires.
