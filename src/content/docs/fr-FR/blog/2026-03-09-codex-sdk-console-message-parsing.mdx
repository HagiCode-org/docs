## Mécanisme de flux d'événements

### Concepts de base

Codex SDK utilise la méthode `thread.runStreamed()` pour renvoyer un itérateur d'événements asynchrone :

```typescript
import { Codex } from '@openai/codex-sdk';

const client = new Codex({
  apiKey: process.env.CODEX_API_KEY,
  baseUrl: process.env.CODEX_BASE_URL,
});

const thread = client.startThread({
  workingDirectory: '/path/to/project',
  skipGitRepoCheck: false,
});

const { events } = await thread.runStreamed('your prompt here', {
  outputSchema: {
    type: 'object',
    properties: {
      output: { type: 'string' },
      status: { type: 'string', enum: ['ok', 'action_required'] },
    },
    required: ['output', 'status'],
  },
});

for await (const event of events) {
  // Traiter chaque événement
}
```

### Explication des types d'événements

| Type d'événement | Description | Données clés |
|------------------|-------------|-------------|
| `thread.started` | Démarrage du thread réussi | `thread_id` |
| `item.updated` | Mise à jour du contenu du message | `item.text` |
| `item.completed` | Message complété | `item.text` |
| `turn.completed` | Exécution complétée | `usage` (utilisation de tokens) |
| `turn.failed` | Échec de l'exécution | `error.message` |
| `error` | Événement d'erreur | `message` |

Dans des projets réels, le composant exécuteur de HagiCode est précisément basé sur ces types d'événements. Nous devons traiter chaque événement avec précision pour assurer la fluidité de l'expérience utilisateur. C'est comme dans une relation, chaque détail doit être traité avec soin, sinon comment peut-il y avoir un bon résultat ?

## Implémentation de l'analyse de messages

### Extraction du contenu des messages

Le contenu des messages est extrait via la fonction de gestion d'événements :

```typescript
private handleThreadEvent(event: ThreadEvent, onMessage: (content: string) => void): void {
  // Traiter uniquement les événements de mise à jour et de complétion de messages
  if (event.type !== 'item.updated' && event.type !== 'item.completed') {
    return;
  }

  // Traiter uniquement le type de message agent
  if (event.item.type !== 'agent_message') {
    return;
  }

  // Extraire le contenu texte
  onMessage(event.item.text);
}
```

Points clés :
- Traiter uniquement les événements `item.updated` et `item.completed`
- Traiter uniquement le contenu de type `agent_message`
- Le contenu du message se trouve dans le champ `event.item.text`

### Analyse de la sortie structurée

Codex prend en charge la sortie structurée JSON, spécifiez le format de retour via le paramètre `outputSchema` :

```typescript
const DEFAULT_OUTPUT_SCHEMA = {
  type: 'object',
  properties: {
    output: { type: 'string' },
    status: { type: 'string', enum: ['ok', 'action_required'] },
  },
  required: ['output', 'status'],
  additionalProperties: false,
} as const;
```

La fonction d'analyse tentera d'analyser le JSON, si elle échoue elle renverra le texte brut - c'est comme la vie, parfois vous voulez une réponse parfaite, mais la réalité vous donne souvent une réponse floue, qu'il faut digérer lentement par soi-même.

```typescript
function toStructuredOutput(raw: string): StructuredOutput {
  try {
    const parsed = JSON.parse(raw) as Partial<StructuredOutput>;
    if (typeof parsed.output === 'string') {
      return {
        output: parsed.output,
        status: parsed.status === 'action_required' ? 'action_required' : 'ok',
      };
    }
  } catch {
    // Échec de l'analyse JSON, retour au texte brut
  }

  return {
    output: raw,
    status: 'ok',
  };
}
```

### Processus complet de gestion des événements

```typescript
private async runWithStreaming(
  thread: Thread,
  input: CodexStageExecutionInput
): Promise<{ output: string; usage: Usage | null }> {
  const abortController = new AbortController();
  const timeoutHandle = setTimeout(() => {
    abortController.abort();
  }, Math.max(1000, input.timeoutMs));

  let latestMessage = '';
  let usage: Usage | null = null;
  let emittedLength = 0;

  try {
    const { events } = await thread.runStreamed(input.prompt, {
      outputSchema: DEFAULT_OUTPUT_SCHEMA,
      signal: abortController.signal,
    });

    for await (const event of events) {
      // Traiter le contenu du message
      this.handleThreadEvent(event, (nextContent) => {
        const delta = nextContent.slice(emittedLength);
        if (delta.length > 0) {
          emittedLength = nextContent.length;
          input.callbacks?.onChunk?.(delta);  // Callback streaming
        }
        latestMessage = nextContent;
      });

      // Traiter différentes données selon le type d'événement
      if (event.type === 'thread.started') {
        this.threadId = event.thread_id;
      } else if (event.type === 'turn.completed') {
        usage = event.usage;
      } else if (event.type === 'turn.failed') {
        throw new CodexExecutorError('gateway_unavailable', event.error.message, true);
      } else if (event.type === 'error') {
        throw new CodexExecutorError('gateway_unavailable', event.message, true);
      }
    }
  } catch (error) {
    if (abortController.signal.aborted) {
      throw new CodexExecutorError(
        'upstream_timeout',
        `Codex stage timed out after ${input.timeoutMs}ms`,
        true
      );
    }
    throw error;
  } finally {
    clearTimeout(timeoutHandle);
  }

  const structured = toStructuredOutput(latestMessage);
  return { output: structured.output, usage };
}
```

## Stratégies de gestion des erreurs

### Mappage des codes d'erreur

Mapper aux codes d'erreur spécifiques selon les caractéristiques de l'erreur pour faciliter le traitement de niveau supérieur :

```typescript
function mapError(error: unknown): CodexExecutorError {
  if (error instanceof CodexExecutorError) {
    return error;
  }

  const message = error instanceof Error ? error.message : String(error);
  const normalized = message.toLowerCase();

  // Erreurs d'authentification - non réessayer
  if (normalized.includes('401') ||
      normalized.includes('403') ||
      normalized.includes('api key') ||
      normalized.includes('auth')) {
    return new CodexExecutorError('auth_invalid', message, false);
  }

  // Limitation de débit - réessayer
  if (normalized.includes('429') || normalized.includes('rate limit')) {
    return new CodexExecutorError('rate_limited', message, true);
  }

  // Erreurs de timeout - réessayer
  if (normalized.includes('timeout') || normalized.includes('aborted')) {
    return new CodexExecutorError('upstream_timeout', message, true);
  }

  // Erreur par défaut
  return new CodexExecutorError('gateway_unavailable', message, true);
}
```

### Définition des types d'erreurs

```typescript
export type CodexErrorCode =
  | 'auth_invalid'      // Échec d'authentification
  | 'upstream_timeout'  // Timeout amont
  | 'rate_limited'      // Limitation de débit
  | 'gateway_unavailable'; // Passerelle indisponible

export class CodexExecutorError extends Error {
  readonly code: CodexErrorCode;
  readonly retryable: boolean;

  constructor(code: CodexErrorCode, message: string, retryable: boolean) {
    super(message);
    this.name = 'CodexExecutorError';
    this.code = code;
    this.retryable = retryable;
```

## Configuration du répertoire de travail et de l'environnement

### Validation du répertoire de travail

Codex SDK exige que le répertoire de travail soit un dépôt Git valide - c'est comme dans la vie, il faut toujours avoir une racine, une origine, sinon comment être serein ?

```typescript
export function validateWorkingDirectory(
  workingDirectory: string,
  skipGitRepoCheck: boolean
): void {
  const resolvedWorkingDirectory = path.resolve(workingDirectory);

  if (!existsSync(resolvedWorkingDirectory)) {
    throw new CodexExecutorError(
      'gateway_unavailable',
      'Working directory does not exist.',
      false
    );
  }

  if (!statSync(resolvedWorkingDirectory).isDirectory()) {
    throw new CodexExecutorError(
      'gateway_unavailable',
      'Working directory is not a directory.',
      false
    );
  }

  if (skipGitRepoCheck) {
    return;
  }

  const gitDir = path.join(resolvedWorkingDirectory, '.git');
  if (!existsSync(gitDir)) {
    throw new CodexExecutorError(
      'gateway_unavailable',
      'Working directory is not a git repository.',
      false
    );
  }
}
```

### Chargement des variables d'environnement

Codex SDK doit charger les variables d'environnement depuis le Shell de connexion pour garantir que l'Agent IA peut accéder aux commandes système :

```typescript
function parseEnvironmentOutput(output: Buffer): Record<string, string> {
  const parsed: Record<string, string> = {};

  for (const entry of output.toString('utf8').split('\0')) {
    if (!entry) continue;

    const separatorIndex = entry.indexOf('=');
    if (separatorIndex <= 0) continue;

    const key = entry.slice(0, separatorIndex);
    const value = entry.slice(separatorIndex + 1);
    if (key.length > 0) {
      parsed[key] = value;
    }
  }

  return parsed;
}

function tryLoadEnvironmentFromShell(shellPath: string): Record<string, string> | null {
  const result = spawnSync(shellPath, ['-ilc', 'env -0'], {
    env: process.env,
    stdio: ['ignore', 'pipe', 'pipe'],
    timeout: 5000,
  });

  if (result.error || result.status !== 0) {
    return null;
  }

  return parseEnvironmentOutput(result.stdout);
}
```

## Exemple d'utilisation complet

### Utilisation de base

Dans le projet HagiCode, nous utilisons la méthode suivante pour initialiser le client Codex et exécuter des tâches :

```typescript
import { Codex } from '@openai/codex-sdk';

async function executeWithCodex(prompt: string, workingDir: string) {
  const client = new Codex({
    apiKey: process.env.CODEX_API_KEY,
    env: { PATH: process.env.PATH },
  });

  const thread = client.startThread({
    workingDirectory: workingDir,
  });

  const { events } = await thread.runStreamed(prompt);

  let result = '';
  for await (const event of events) {
    if (event.type === 'item.updated' && event.item.type === 'agent_message') {
      result = event.item.text;
    }
    if (event.type === 'turn.completed') {
      console.log('Token usage:', event.usage);
    }
  }

  // Tenter d'analyser la sortie JSON
  try {
    const parsed = JSON.parse(result);
    return parsed.output;
  } catch {
    return result;
  }
}
```

### Implémentation complète avec mécanisme de réessai

```typescript
export class CodexSdkExecutor {
  private readonly config: CodexRuntimeConfig;
  private readonly client: Codex;
  private threadId: string | null = null;

  async executeStage(input: CodexStageExecutionInput): Promise<CodexStageExecutionResult> {
    const maxAttempts = Math.max(1, this.config.retryCount + 1);
    let attempt = 0;
    let lastError: CodexExecutorError | null = null;

    while (attempt < maxAttempts) {
      attempt += 1;

      try {
        const thread = this.getThread(input.workingDirectory);
        const { output, usage } = await this.runWithStreaming(thread, input);

        return {
          output,
          usage,
          threadId: this.threadId!,
          attempts: attempt,
          latencyMs: Date.now() - startedAt,
        };
      } catch (error) {
        const mappedError = mapError(error);
        lastError = mappedError;

        // Erreur non réessai ou nombre maximum de réessais atteint
        if (!mappedError.retryable || attempt >= maxAttempts) {
          throw mappedError;
        }

        // Attendre avant de réessayer
        await new Promise(resolve => setTimeout(resolve, 1000 * attempt));
      }
    }

    throw lastError!;
  }
}
```

## Meilleures pratiques

### 1. Exigences du répertoire de travail

- Assurer que le répertoire de travail est un dépôt Git valide
- Spécifier explicitement avec la variable d'environnement `PROJECT_ROOT`
- Définir `CODEX_SKIP_GIT_REPO_CHECK=true` pour ignorer la vérification lors du développement

### 2. Configuration des variables d'environnement

- Transmettre les variables d'environnement nécessaires via un mécanisme de liste blanche
- Utiliser le Shell de connexion pour charger l'environnement complet
- Éviter de transmettre des informations sensibles

### 3. Timeout et réessai

- Définir un délai d'expiration raisonnable selon la complexité de la tâche
- Implémenter un retour exponentiel pour les erreurs réessaiables
- Enregistrer le nombre et la raison des réessais

### 4. Gestion des erreurs

- Distinguer les erreurs réessaiables et non réessaiables
- Fournir des messages d'erreur clairs et des suggestions
- Codes d'erreur unifiés pour faciliter le traitement de niveau supérieur

### 5. Sortie streaming

- Implémenter des callbacks de sortie incrémentale pour améliorer l'expérience utilisateur
- Gérer correctement les mises à jour incrémentales des messages
- Enregistrer l'utilisation des tokens pour l'analyse des coûts
