## De 120 secondes à 45 secondes : guide pratique pour optimiser les performances de build Vite avec Worker Threads

> Lors du traitement de grands projets frontend, le build de code en environnement de production peut sembler interminable. Cet article partage comment réduire le temps de l'étape d'obfuscation de code dans le build Vite de 120 secondes à 45 secondes grâce aux Worker Threads Node.js, avec une présentation détaillée de la mise en œuvre dans le projet HagiCode et des leçons apprises.

## Contexte

Dans notre pratique d'ingénierie frontend, avec l'agrandissement des projets, le problème de l'efficacité de build est devenu de plus en plus apparent. En particulier dans le processus de build de production, pour protéger la logique du code source, nous introduisons généralement des outils d'obfuscation JavaScript (comme `javascript-obfuscator`). Bien que nécessaire, cette étape est très gourmande en calculs et consomme énormément de ressources CPU.

Dans les premiers stades de développement du projet **HagiCode**, nous avons rencontré un goulot d'étranglement de performances très problématique : le temps de build de production se dégradait rapidement avec l'augmentation du volume de code.

**Problèmes spécifiques** :
- Exécution séquentielle mono-thread des tâches d'obfuscation, un seul cœur CPU à 100 %, les autres cœurs au repos
- Le temps de build est passé de 30 secondes initialement à 110-120 secondes
- Le processus de validation de build après chaque modification de code était extrêmement long, ralentissant considérablement l'efficacité des itérations de développement
- Dans le pipeline CI/CD, l'étape de build est devenue la plus chronophage

## Analyse : trouver la clé pour résoudre le goulot d'étranglement

Avant de résoudre les problèmes de performances, nous devons clarifier notre réflexion et déterminer la solution technique optimale.

### Décision clé : pourquoi choisir Worker Threads ?

Il existe principalement trois approches pour le calcul parallèle dans l'environnement Node.js :
1. **child_process** : création de processus enfants indépendants
2. **Web Workers** : principalement pour le côté navigateur
3. **worker_threads** : support natif multithread de Node.js

Après analyse comparative, HagiCode a finalement choisi les **Worker Threads**, pour les raisons suivantes :
- **Zéro surcoût de sérialisation** : les Worker Threads se trouvent dans le même processus et peuvent partager la mémoire via `SharedArrayBuffer` ou le transfert de contrôle, évitant les coûts élevés de sérialisation de la communication inter-processus.
- **Support natif** : intégré dans Node.js 12+, sans dépendances supplémentaires lourdes.
- **Contexte unifié** : le débogage et la journalisation sont plus pratiques qu'avec les processus enfants.

### Granularité des tâches : comment diviser les tâches d'obfuscation ?

Obfusquer un énorme fichier JS Bundle est difficile à paralléliser (car le code a des dépendances), mais les artefacts de build de Vite sont composés de multiples **Chunks**. Cela nous donne une frontière de parallélisme naturelle :

- **Indépendance** : les différents Chunks produits par Vite ont leurs dépendances découplées et peuvent être traités en parallèle en toute sécurité.
- **Granularité appropriée** : un projet typique aura 10-30 Chunks, un nombre parfaitement adapté à l'ordonnancement parallèle.
- **Intégration facile** : le hook `generateBundle` du plugin Vite nous permet d'intercepter et de traiter ces Chunks avant la génération des fichiers.

### Conception de l'architecture

Nous avons conçu un système de traitement parallèle avec quatre composants principaux :

1. **Task Splitter** : parcourt l'objet bundle de Vite, filtre les fichiers qui n'ont pas besoin d'obfuscation (comme les vendors), et génère une file de tâches.
2. **Worker Pool Manager** : gère le cycle de vie des Workers, responsable de la distribution, de la récupération et du réessai en cas d'erreur.
3. **Progress Reporter** : affiche la progression du build en temps réel pour réduire l'anxiété d'attente.
4. **ObfuscationWorker** : thread de travail exécutant réellement la logique d'obfuscation.

## Solution : codage pratique et mise en œuvre

Sur la base de l'analyse ci-dessus, nous avons commencé à implémenter ce système d'obfuscation parallèle.

### 1. Configuration du plugin Vite

```typescript
import { defineConfig } from 'vite'
import { parallelJavascriptObfuscator } from './buildTools/plugin'

export default defineConfig(({ mode }) => {
  const isProduction = mode === 'production'

  return {
    build: {
      rollupOptions: {
        ...(isProduction
          ? {
              plugins: [
                parallelJavascriptObfuscator({
                  enabled: true,
                  workerCount: 4,
                  retryAttempts: 3,
                  fallbackToMainThread: true,
                  isVendorChunk: (fileName: string) => fileName.includes('vendor-'),
                  obfuscationConfig: {
                    compact: true,
                    controlFlowFlattening: true,
                    deadCodeInjection: true,
                    disableConsoleOutput: true,
                  },
                }),
              ],
            }
          : {}),
      },
    },
  }
})
```

### 2. Implémentation de la logique Worker

```typescript
import { parentPort } from 'worker_threads'
import javascriptObfuscator from 'javascript-obfuscator'

export interface ObfuscationTask {
  chunkId: string
  code: string
  config: any
}

export interface ObfuscationResult {
  chunkId: string
  obfuscatedCode: string
  error?: string
}

if (parentPort) {
  parentPort.on('message', async (task: ObfuscationTask) => {
    try {
      const obfuscated = javascriptObfuscator.obfuscate(task.code, task.config)
      const result: ObfuscationResult = {
        chunkId: task.chunkId,
        obfuscatedCode: obfuscated.getObfuscatedCode(),
      }
      parentPort?.postMessage(result)
    } catch (error) {
      const result: ObfuscationResult = {
        chunkId: task.chunkId,
        obfuscatedCode: '',
        error: error instanceof Error ? error.message : 'Unknown error',
      }
      parentPort?.postMessage(result)
    }
  })
}
```

### 3. Gestionnaire de pool de Workers

```typescript
import { Worker } from 'worker_threads'
import os from 'os'

export class WorkerPool {
  private workers: Worker[] = []
  private taskQueue: Array<{
    task: ObfuscationTask
    resolve: (result: ObfuscationResult) => void
    reject: (error: Error) => void
  }> = []

  constructor(options: WorkerPoolOptions = {}) {
    const workerCount = options.workerCount ?? Math.max(1, (os.cpus().length || 4) - 1)
    for (let i = 0; i < workerCount; i++) {
      this.createWorker()
    }
  }

  private createWorker() {
    const worker = new Worker('./worker.ts')
    worker.on('message', (result) => {
      const nextTask = this.taskQueue.shift()
      if (nextTask) {
        this.dispatchTask(worker, nextTask)
      } else {
        this.activeWorkers.delete(worker)
      }
    })
    this.workers.push(worker)
  }

  public runTask(task: ObfuscationTask): Promise<ObfuscationResult> {
    return new Promise((resolve, reject) => {
      const job = { task, resolve, reject }
      const idleWorker = this.workers.find(w => !this.activeWorkers.has(w))
      if (idleWorker) {
        this.dispatchTask(idleWorker, job)
      } else {
        this.taskQueue.push(job)
      }
    })
  }

  private dispatchTask(worker: Worker, job: any) {
    this.activeWorkers.set(worker, job.task)
    worker.postMessage(job.task)
  }
}
```

### 4. Rapport de progression

```typescript
export class ProgressReporter {
  private completed = 0
  private readonly total: number
  private readonly startTime: number

  constructor(total: number) {
    this.total = total
    this.startTime = Date.now()
  }

  increment(): void {
    this.completed++
    this.report()
  }

  private report(): void {
    const now = Date.now()
    const elapsed = now - this.startTime
    const percentage = (this.completed / this.total) * 100
    const avgTimePerChunk = elapsed / this.completed
    const remaining = (this.total - this.completed) * avgTimePerChunk

    console.log(
      `[Parallel Obfuscation] ${this.completed}/${this.total} chunks completed (${percentage.toFixed(1)}%) | ETA: ${(remaining / 1000).toFixed(1)}s`
    )
  }
}
```

## Résultats et leçons apprises

Après le déploiement de cette solution, les performances de build du projet HagiCode se sont améliorées de manière significative.

### Données de référence

Testé dans l'environnement suivant :
- CPU : Intel Core i7-12700K (12 cœurs / 20 threads)
- RAM : 32 Go DDR4
- Node.js : v18.17.0
- OS : Ubuntu 22.04

**Comparaison des résultats** :
- **Mono-thread (avant optimisation)** : 118 secondes
- **4 Workers** : 55 secondes (amélioration de **53 %**)
- **8 Workers** : 48 secondes (amélioration de **60 %**)
- **12 Workers** : 45 secondes (amélioration de **62 %**)

On constate que les gains ne sont pas linéaires. Au-delà de 8 Workers, l'amélioration diminue, principalement en raison de l'uniformité de la distribution des tâches et du goulot d'étranglement de la bande passante mémoire.

### Questions fréquentes et solutions

**Q1 : Le temps de build n'a pas diminué, il a même augmenté ?**
- **Cause** : la création de Workers a un coût en soi, ou le nombre de Workers est trop élevé, entraînant des changements de contexte fréquents.
- **Solution** : réglez le nombre de Workers à `nombre de cœurs CPU - 1`. Vérifiez aussi s'il y a un Chunk particulièrement gros (> 5 Mo).

**Q2 : Les Workers plantent occasionnellement ?**
- **Cause** : certaines syntaxes de code spécifiques peuvent provoquer des erreurs internes dans l'obfuscateur.
- **Solution** : nous avons implémenté un **mécanisme de dégradation automatique**. Lorsque le nombre d'échecs consécutifs atteint un seuil, le plugin revient automatiquement en mode mono-thread.

**Q3 : Consommation mémoire excessive (OOM) ?**
- **Solution** : réduisez le nombre de Workers. Augmentez la limite mémoire de Node.js : `NODE_OPTIONS="--max-old-space-size=4096" npm run build`.

## Conclusion

En introduisant les Worker Threads Node.js, nous avons réussi à réduire le temps de build de production du projet HagiCode de 120 secondes à environ 45 secondes, améliorant considérablement l'expérience de développement et l'efficacité du CI/CD.

Les points clés de cette solution :
1. **Division judicieuse des tâches** : utilisation des Chunks de Vite comme unités de parallélisation.
2. **Contrôle des ressources** : utilisation d'un pool de Workers pour éviter l'épuisement des ressources.
3. **Conception tolérante aux pannes** : mécanisme de dégradation automatique assurant la stabilité du build.
