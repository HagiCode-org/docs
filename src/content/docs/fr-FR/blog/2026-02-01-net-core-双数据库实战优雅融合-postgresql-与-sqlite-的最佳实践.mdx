## .NET Core double base de données en pratique : laissez PostgreSQL et SQLite coexister pacifiquement

> Lors de la construction d'applications modernes, nous sommes souvent confrontés à ce compromis : les environnements de développement veulent quelque chose de léger et pratique, tandis que les environnements de production exigent une haute concurrence et une haute disponibilité. Cet article partage comment supporter élégamment à la fois PostgreSQL et SQLite dans un projet .NET Core et mettre en œuvre la bonne pratique « SQLite pour le développement, PostgreSQL pour la production ».

## Contexte

Dans le développement logiciel, les différences entre environnements ont toujours été l'un des problèmes les plus difficiles pour les équipes d'ingénierie. Prenons l'exemple de la plateforme **HagiCode** que nous construisons : c'est un système de développement assisté par IA basé sur ASP.NET Core 10 et React, avec Orleans intégré en interne pour la gestion d'état distribué. La stack est moderne et assez sophistiquée.

Au début du projet, nous avons rencontré un problème d'ingénierie classique : les développeurs voulaient que l'environnement local fonctionne hors de la boîte, sans avoir à installer et configurer une base de données PostgreSQL lourde. Mais en production, nous devions gérer des écritures à haute concurrence et des requêtes JSON complexes, et c'est précisément là que SQLite, malgré sa légèreté, commence à montrer ses limites.

Comment conserver une base de code unique tout en permettant à l'application de bénéficier de la portabilité de SQLite comme une application de bureau, et aussi d'exploiter les performances puissantes de PostgreSQL comme un service de niveau entreprise ? C'est la question centrale que cet article explore.

## Sujet principal 1 : conception de l'architecture et abstraction unifiée

Pour supporter deux bases de données dans .NET Core, l'idée clé est de dépendre des abstractions plutôt que des implémentations concrètes. Nous devons séparer la sélection de la base de données du code métier et laisser la couche de configuration décider.

### Approche de conception

1.  **Interface unifiée** : toute la logique métier devrait dépendre de la classe de base `DbContext` ou d'interfaces personnalisées, plutôt que d'un `PostgreSqlDbContext` spécifique.
2.  **Pilotée par la configuration** : utilisez les éléments de configuration dans `appsettings.json` pour décider dynamiquement quel provider de base de données charger au démarrage de l'application.
3.  **Isolation des fonctionnalités** : ajoutez une logique d'adaptation pour les capacités spécifiques à PostgreSQL, telles que JSONB, afin que l'application puisse encore se dégrader gracieusement sur SQLite.

### Implémentation du code : configuration dynamique du contexte

Dans le `Program.cs` d'ASP.NET Core, nous ne devrions pas coder en dur `UseNpgsql` ou `UseSqlite`. Nous devrions plutôt lire la configuration et décider dynamiquement.

D'abord, définissons la classe de configuration :

```csharp
public class DatabaseSettings
{
    public const string SectionName = "Database";
    
    // Database type: PostgreSQL or SQLite
    public string DbType { get; set; } = "PostgreSQL"; 
    
    // Connection string
    public string ConnectionString { get; set; } = string.Empty;
}
```

Puis enregistrons le service dans `Program.cs` en fonction de la configuration :

```csharp
// Read configuration
var databaseSettings = builder.Configuration.GetSection(DatabaseSettings.SectionName).Get<DatabaseSettings>();

// Register DbContext
builder.Services.AddDbContext<ApplicationDbContext>(options =>
{
    if (databaseSettings?.DbType?.ToLower() == "sqlite")
    {
        // SQLite configuration
        options.UseSqlite(databaseSettings.ConnectionString);
        
        // Handling SQLite's concurrent write limitations
        // Note: in production, enabling WAL mode is recommended to improve concurrency performance
    }
    else
    {
        // PostgreSQL configuration (default)
        options.UseNpgsql(databaseSettings.ConnectionString, npgsqlOptions =>
        {
            // Enable JSONB support, which is very useful when handling AI conversation records
            npgsqlOptions.UseJsonNet(); 
        });
        
        // Configure connection pool retry policy
        options.EnableRetryOnFailure(3);
    }
});
```

## Sujet principal 2 : gestion des différences et stratégie de migration

Bien que PostgreSQL et SQLite supportent tous les deux la norme SQL, ils diffèrent significativement dans leurs capacités et comportements spécifiques. Si ces différences ne sont pas gérées avec soin, on peut facilement se retrouver dans la situation embarrassante où tout fonctionne localement mais échoue après le déploiement.

### 1. Gestion des types JSON

Dans HagiCode, nous devons stocker une grande quantité de prompts et de métadonnées IA, ce qui implique généralement des colonnes JSON.
- **PostgreSQL** : possède un type natif `JSONB` avec d'excellentes performances de requête.
- **SQLite** : ne possède pas de type JSON natif (les versions plus récentes incluent l'extension JSON1, mais le mapping d'objets diffère toujours), les données sont donc généralement stockées en TEXT.

**Solution** :
Dans le mapping d'entités EF Core, nous le configurons comme un type convertible.

```csharp
protected override void OnModelCreating(ModelBuilder modelBuilder)
{
    base.OnModelCreating(modelBuilder);

    // Configure entity
    modelBuilder.Entity<PromptTemplate>(entity => 
    {
        entity.Property(e => e.Metadata)
              .HasColumnType("jsonb") // PG uses jsonb
              .HasConversion(
                  v => JsonSerializer.Serialize(v, (JsonSerializerOptions)null),
                  v => JsonSerializer.Deserialize<Dictionary<string, object>>(v, (JsonSerializerOptions)null)
              ); 
    });
}
```

Lorsque SQLite est utilisé, `HasColumnType("jsonb")` peut être ignoré ou déclencher un avertissement. Cependant, grâce à la configuration de `HasConversion`, les données sont toujours sérialisées et désérialisées correctement sous forme de chaînes stockées dans un champ TEXT, assurant ainsi la compatibilité.

### 2. Séparation des stratégies de migration

N'essayez jamais de faire fonctionner un seul ensemble de scripts Migration pour PostgreSQL et SQLite en même temps. Les différences dans les stratégies de génération de clés primaires, la syntaxe des index et autres détails de base de données provoqueront inévitablement des échecs.

**Bonne pratique** :
Maintenez deux branches ou projets de migration. Dans le workflow de développement de HagiCode, voici comment nous procédons :

1.  **Phase de développement** : travaillez principalement avec SQLite. Utilisez `Add-Migration Init_Sqlite -OutputDir Migrations/Sqlite`.
2.  **Phase d'adaptation** : après avoir développé une fonctionnalité, changez la chaîne de connexion vers PostgreSQL et exécutez `Add-Migration Init_Postgres -OutputDir Migrations/Postgres`.
3.  **Scripts d'automatisation** : écrivez un script PowerShell ou Bash simple pour appliquer automatiquement la bonne migration en fonction de la variable d'environnement actuelle.

```bash
# Pseudocode for simple deployment logic
if [ "$DATABASE_PROVIDER" = "PostgreSQL" ]; then
    dotnet ef database update --project Migrations.Postgres
else
    dotnet ef database update --project Migrations.Sqlite
fi
```

## Sujet principal 3 : leçons apprises de HagiCode en production

Lors de la refactorisation de **HagiCode** d'un modèle à base de données unique vers un support double base de données, nous avons rencontré quelques pièges et tiré des leçons importantes qui pourraient vous aider à éviter les mêmes erreurs.

### 1. Différences de concurrence et de transactions

PostgreSQL utilise une architecture client-serveur et prend en charge les écritures à haute concurrence avec des niveaux d'isolation de transaction puissants. SQLite utilise le verrouillage de fichiers, les opérations d'écriture verrouillent donc l'ensemble du fichier de base de données sauf si le mode WAL est activé.

**Recommandation** :
Lors de l'écriture de logique métier impliquant des écritures fréquentes, comme la sauvegarde en temps réel de l'état d'édition d'un utilisateur, vous devez prendre en compte le modèle de verrouillage de SQLite. Lors de la conception du module de collaboration OpenSpec dans **HagiCode**, nous avons introduit un mécanisme de « fusion avant écriture » pour réduire la fréquence des écritures directes en base de données, nous permettant ainsi de maintenir de bonnes performances sur les deux bases de données.

### 2. Gestion du cycle de vie des chaînes de connexion

L'établissement d'une connexion PostgreSQL est relativement coûteux et dépend du pooling de connexions. Les connexions SQLite sont très légères, mais si elles ne sont pas libérées rapidement, les verrous de fichiers peuvent provoquer des délais d'attente pour les opérations ultérieures.

Dans `Program.cs`, nous pouvons affiner le comportement pour chaque base de données :

```csharp
if (databaseSettings?.DbType?.ToLower() == "sqlite")
{
    // SQLite: keeping connections open can improve performance, but watch out for file locks
    options.UseSqlite(connectionString, sqliteOptions =>
    {
        // Set command timeout
        sqliteOptions.CommandTimeout(30);
    });
}
else
{
    // PG: make full use of connection pooling
    options.UseNpgsql(connectionString, npgsqlOptions =>
    {
        npgsqlOptions.MaxBatchSize(100);
        npgsqlOptions.CommandTimeout(30);
    });
}
```

### 3. L'importance de la couverture de tests

Beaucoup de développeurs, y compris certains premiers membres de notre équipe, ont tendance à faire une erreur courante : exécuter les tests unitaires uniquement dans l'environnement de développement, qui est généralement SQLite.

Dans le pipeline CI/CD de HagiCode, nous avons imposé une étape GitHub Actions pour s'assurer que chaque pull request exécute des tests d'intégration PostgreSQL.

```yaml
# Example snippet from .github/workflows/test.yml
- name: Run Integration Tests (PostgreSQL)
  run: |
    docker-compose up -d db_postgres
    dotnet test --filter "Category=Integration"
```

Cela nous a permis de détecter d'innombrables bugs liés aux différences de syntaxe SQL et à la sensibilité à la casse.

## Conclusion

Grâce à l'introduction d'une couche d'abstraction et de l'injection de dépendances pilotée par la configuration, nous avons réussi à mettre en œuvre une configuration double voie PostgreSQL et SQLite dans le projet **HagiCode**. Cela a non seulement considérablement abaissé la barrière d'entrée pour les nouveaux développeurs en supprimant le besoin d'installer PostgreSQL, mais a également fourni de solides garanties de performance pour la production.

Pour résumer les points clés :
1.  **Abstraction d'abord** : le code métier ne devrait pas dépendre d'implémentations concrètes de base de données.
2.  **Configuration séparée** : utilisez des fichiers `appsettings.json` différents pour le développement et la production.
3.  **Migrations séparées** : n'essayez pas de faire fonctionner un seul ensemble de Migration partout.
4.  **Dégradation des fonctionnalités** : privilégiez la compatibilité dans SQLite et les performances dans PostgreSQL.

Ce modèle d'architecture n'est pas seulement adapté à HagiCode, mais à tout projet .NET qui doit trouver un équilibre entre un développement léger et une production robuste.
