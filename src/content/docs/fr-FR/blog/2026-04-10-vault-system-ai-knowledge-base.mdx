## Construire une base de connaissances inter-projets pour l'ère de l'IA avec le système Vault

> La méthode d'apprentissage par copie de projet devient dominante, mais la dispersion des matériaux d'apprentissage et la rupture de contexte limitent la valeur des assistants IA. Cet article présente la conception du système Vault du projet HagiCode—à travers une couche d'abstraction de stockage unifiée, permettant aux assistants IA de comprendre et d'accéder à toutes les ressources d'apprentissage, réalisant ainsi une véritable réutilisation des connaissances inter-projets.

## Contexte

En réalité, à l'ère de l'IA, notre façon d'apprendre de nouvelles technologies évolue silencieusement. La lecture traditionnelle et le visionnage de vidéos restent importants, mais la "copie de projet"—étude approfondie du code, de l'architecture et des modèles de conception de projets open source excellents—devient de plus en plus efficace. Exécuter et modifier directement des projets open source de haute qualité vous permet de comprendre le plus rapidement possible les pratiques d'ingénierie du monde réel.

Cependant, cette approche apporte aussi de nouveaux défis.

**Les matériaux d'apprentissage sont trop dispersés**. Les notes peuvent se trouver dans Obsidian, les dépôts de code sont éparpillés dans différents dossiers, et l'historique des conversations des assistants IA constitue un autre silo de données indépendant. Chaque fois que l'aide de l'IA est nécessaire pour analyser un projet, il faut copier manuellement des extraits de code et organiser le contexte, un processus assez fastidieux.

**Le contexte est souvent rompu**. Les assistants IA ne peuvent pas accéder directement aux ressources d'apprentissage locales, et chaque conversation nécessite de fournir à nouveau des informations contextuelles. Les dépôts de code copiés se mettent à jour rapidement, et la synchronisation manuelle est sujette aux erreurs. Pire encore, il est difficile de partager des connaissances entre plusieurs projets d'apprentissage—ce qui a été appris dans le projet A, l'IA l'ignore complètement lorsqu'elle traite le projet B.

L'essence de ces problèmes est le "silos de données". S'il existait une couche d'abstraction de stockage unifiée permettant aux assistants IA de comprendre et d'accéder à toutes les ressources d'apprentissage, le problème serait résolu.

Pour résoudre ces points sensibles, nous avons pris une décision de conception clé lors du développement de HagiCode : construire un système Vault comme couche d'abstraction de stockage de connaissances unifiée. Les changements apportés par cette décision pourraient être plus importants qu'on ne l'imagine—we'll see.

## À propos de HagiCode

La solution présentée dans cet article provient de l'expérience pratique du projet [HagiCode](https://hagicode.com). HagiCode est un assistant de code IA basé sur le flux de travail OpenSpec, dont le concept central est de permettre à l'IA de non seulement "parler", mais surtout "agir"—capable de manipuler directement des dépôts de code, d'exécuter des commandes et de lancer des tests. GitHub : [github.com/HagiCode-org/site](https://github.com/HagiCode-org/site)

Au cours du développement, nous avons découvert que les assistants IA doivent fréquemment accéder aux diverses ressources d'apprentissage des utilisateurs : dépôts de code, documents de notes, fichiers de configuration, etc. Si l'utilisateur doit les fournir manuellement à chaque fois, l'expérience serait trop médiocre. Cela nous a amenés à concevoir le système Vault.

## Conception centrale

### Support multi-types

Le système Vault de HagiCode prend en charge quatre types, correspondant à différents scénarios d'utilisation :

| Type | Usage | Scénario typique |
|------|-------|------------------|
| `folder` | Type de dossier générique | Matériaux d'apprentissage temporaires, brouillons |
| `coderef` | Spécialisé pour les projets de code copiés | Apprentissage systématique d'un projet open source |
| `obsidian` | Intégration avec le logiciel de notes Obsidian | Réutilisation des bases de notes existantes |
| `system-managed` | Gestion automatique par le système | Configuration de projet, modèles de prompts, etc. |

Le type `coderef` est le plus couramment utilisé dans HagiCode, fournissant une structure de répertoires standardisée et des descriptions de métadonnées lisibles par l'IA pour les projets de code copiés. Pourquoi concevoir spécifiquement ce type ? Parce que copier un projet open source n'est pas un simple "téléchargement de code"—il faut gérer simultanément le code lui-même, les notes d'apprentissage, les fichiers de configuration, etc., et `coderef` normalise tout cela.

### Mécanisme de stockage persistant

Le registre Vault est stocké de manière persistante au format JSON sur le système de fichiers :

```csharp
_registryFilePath = Path.Combine(absoluteDataDir, "personal-data", "vaults", "registry.json");
```

Cette conception semble simple, mais elle a été réfléchie :

**Simple et fiable**. Le format JSON est lisible par l'homme, facilitant le débogage et les modifications manuelles. Lorsque des problèmes surviennent dans le système, on peut directement ouvrir le fichier pour vérifier l'état, voire le réparer manuellement—ce qui est particulièrement utile pendant le développement.

**Réduction des dépendances**. Le stockage sur système de fichiers évite la complexité des bases de données. Aucune installation et configuration supplémentaires de service de base de données ne sont nécessaires, réduisant la complexité du système et les coûts de maintenance.

**Sécurité de concurrence**. Utilisation de `SemaphoreSlim` pour assurer la sécurité multi-thread. Dans le scénario d'un assistant de code IA, plusieurs opérations peuvent accéder simultanément au registre Vault, nécessitant un contrôle de concurrence approprié.

### Intégration du contexte IA

La capacité centrale du système réside dans sa capacité à injecter automatiquement les informations Vault dans le contexte des propositions IA :

```typescript
export function buildTargetVaultsText(
  vaults: VaultForText[],
  template: VaultPromptTemplate = DEFAULT_VAULT_PROMPT_TEMPLATE,
): string {
  const readOnlyVaults = vaults.filter((vault) => vault.accessType === 'read');
  const editableVaults = vaults.filter((vault) => vault.accessType === 'write');

  const sections = [
    buildVaultSection(readOnlyVaults, template.reference),
    buildVaultSection(editableVaults, template.editable),
  ].filter(Boolean);

  return `\n\n### ${template.heading}\n\n${sections.join('\n')}`;
}
```

Ainsi, les assistants IA peuvent comprendre automatiquement les ressources d'apprentissage disponibles, sans que l'utilisateur n'ait à fournir manuellement le contexte à chaque fois. Cette conception rend l'expérience HagiCode particulièrement naturelle—dites à l'IA "aide-moi à analyser le rendu concurrent de React", et l'IA trouvera automatiquement le Vault d'apprentissage React précédemment enregistré, au lieu de coller le code encore et encore.

### Mécanisme de contrôle d'accès

Le système divise les Vaults en deux types d'accès :

- **reference (lecture seule)** : L'IA ne l'utilise que pour l'analyse et la compréhension, ne peut pas modifier le contenu
- **editable (modifiable)** : L'IA peut modifier le contenu selon les besoins de la tâche

Cette distinction permet à l'IA de savoir quel contenu est "référence en lecture seule" et quel contenu peut être "modifié", évitant les risques d'erreurs d'opération. Par exemple, si vous enregistrez un Vault d'un projet open source comme matériel d'apprentissage, vous ne voulez certainement pas que l'IA modifie le code à l'intérieur—marquez-le alors comme `reference`. Mais s'il s'agit de votre propre projet Vault, vous pouvez le marquer comme `editable`, permettant à l'IA de modifier le code.

## Guide pratique

### Structure normalisée des CodeRef Vaults

Pour les Vaults de type coderef, le système fournit une structure de répertoires standardisée :

```
my-coderef-vault/
├── index.yaml          # description des métadonnées Vault
├── AGENTS.md           # guide d'opération pour l'assistant IA
├── docs/               # stockage des notes d'apprentissage et documents
└── repos/              # gestion des dépôts de code copiés via sous-modules Git
```

Quelle est la philosophie de conception de cette structure ?

**docs/** stocke les notes d'apprentissage, enregistrant la compréhension du code, l'analyse de l'architecture et l'expérience des problèmes rencontrés au format Markdown. Ces notes ne sont pas seulement pour vous, l'IA peut aussi les lire—elle y fera référence automatiquement lors du traitement de tâches connexes.

**repos/** gère les dépôts copiés via des sous-modules Git, plutôt que de copier directement le code. Cela présente deux avantages : premièrement, maintenir la synchronisation avec l'amont, un `git submodule update` suffit pour obtenir le dernier code ; deuxièmement, économiser l'espace, plusieurs Vaults peuvent référencer différentes versions du même dépôt.

**index.yaml** contient les métadonnées du Vault, permettant à l'assistant IA de comprendre rapidement son but et son contenu. C'est comme écrire une "auto-présentation" pour le Vault, l'IA sait ce qu'il fait au premier contact.

**AGENTS.md** est un guide écrit spécifiquement pour les assistants IA, expliquant comment traiter le contenu dans le Vault. Vous pouvez dire à l'IA ici : "lors de l'analyse de ce projet, concentrez-vous sur le code lié à l'optimisation des performances" ou "ne modifiez pas les fichiers de test".

### Création et utilisation de Vaults

Créer un CodeRef Vault est simple :

```typescript
const createCodeRefVault = async () => {
  const response = await VaultService.postApiVaults({
    requestBody: {
      name: "React Learning Vault",
      type: "coderef",
      physicalPath: "/Users/developer/vaults/react-learning",
      gitUrl: "https://github.com/facebook/react.git"
    }
  });

  // Le système automatiquement :
  // 1. Clone le dépôt React vers vault/repos/react
  // 2. Crée le répertoire docs/ pour les notes
  // 3. Génère les métadonnées index.yaml
  // 4. Crée le fichier de guide AGENTS.md

  return response;
};
```

Ensuite, référencez ce Vault dans une proposition IA :

```typescript
const proposal = composeProposalChiefComplaint({
  chiefComplaint: "Aide-moi à analyser le mécanisme de rendu concurrent de React",
  repositories: [
    { id: "react", gitUrl: "https://github.com/facebook/react.git" }
  ],
  vaults: [
    {
      id: "react-learning",
      name: "React Learning Vault",
      type: "coderef",
      physicalPath: "/vaults/react-learning",
      accessType: "read"  // L'IA ne peut que lire, ne peut pas modifier
    }
  ],
  quickRequestText: "Concentrez-vous sur l'architecture fiber et l'implémentation du scheduler"
});
```

### Scénarios d'utilisation typiques

**Scénario un : Apprentissage systématique de projets open source**

Créez un CodeRef Vault, gérez le dépôt cible via des sous-modules Git, enregistrez les notes d'apprentissage dans le répertoire `docs/`. L'IA peut accéder simultanément au code et aux notes, fournissant une analyse plus précise. Les notes écrites lors de l'apprentissage d'un module seront automatiquement référencées par l'IA lors de l'analyse du code connexe ultérieur—comme avoir un "assistant" qui se souvient des réflexions précédentes.

**Scénario deux : Réutilisation de la base de notes Obsidian**

Si vous utilisez déjà Obsidian pour gérer vos notes, enregistrez simplement votre Vault existant dans HagiCode. L'IA peut accéder directement à la base de connaissances, sans copier-coller manuel. Cette fonction est particulièrement pratique, beaucoup de gens ont accumulé des notes pendant des années, et après l'intégration, l'IA peut "lire" et comprendre le système de connaissances.

**Scénario trois : Réutilisation des connaissances inter-projets**

Plusieurs propositions IA peuvent référencer le même Vault, réalisant la réutilisation des connaissances inter-projets. Par exemple, créez un "Vault d'apprentissage des modèles de conception", enregistrez les notes et exemples de code de divers modèles de conception. Quelle que soit l'analyse de projet, l'IA peut référencer le contenu de ce Vault—les connaissances n'ont pas besoin d'être accumulées à plusieurs reprises.

### Mécanisme de sécurité des chemins

Le système vérifie strictement les chemins pour prévenir les attaques de traversée de chemin :

```csharp
private static string ResolveFilePath(string vaultRoot, string relativePath)
{
    var rootPath = EnsureTrailingSeparator(Path.GetFullPath(vaultRoot));
    var combinedPath = Path.GetFullPath(Path.Combine(rootPath, relativePath));
    if (!combinedPath.StartsWith(rootPath, StringComparison.OrdinalIgnoreCase))
    {
        throw new BusinessException(VaultRelativePathTraversalCode,
            "Vault file paths must stay inside the registered vault root.");
    }
    return combinedPath;
}
```

Cela garantit que toutes les opérations de fichiers restent dans la portée du répertoire racine Vault, prévenant l'accès malveillant aux chemins. La sécurité ne peut pas être négligée, l'assistant IA doit opérer sur le système de fichiers, les limites doivent être clairement définies.

## Points d'attention

Lors de l'utilisation du système Vault HagiCode, quelques points nécessitent une attention particulière :

1. **Sécurité des chemins** : Assurez-vous que les chemins personnalisés sont dans la plage autorisée, sinon le système rejettera l'opération. C'est pour prévenir les erreurs d'opération et les risques de sécurité potentiels.

2. **Gestion des sous-modules Git** : CodeRef Vault recommande d'utiliser des sous-modules Git plutôt que de copier directement le code. Les avantages ont été mentionnés précédemment—maintenir la synchronisation, économiser l'espace. Cependant, les sous-modules ont leur propre mode d'utilisation, une familiarisation peut être nécessaire lors de la première utilisation.

3. **Limites de prévisualisation des fichiers** : Le système limite la taille des fichiers (256KB) et le nombre (500), les fichiers ultra-volumineux doivent être traités par lots. Cette limite est pour des considérations de performance, si vous rencontrez des fichiers ultra-volumineux, vous pouvez les diviser manuellement ou utiliser d'autres méthodes de traitement.

4. **Informations de diagnostic** : La création d'un Vault renvoie des informations de diagnostic, qui peuvent être utilisées pour le débogage en cas d'échec. En cas de problème, vérifiez d'abord les informations de diagnostic, dans la plupart des cas, vous pourrez trouver des indices.

## Résumé

Le système Vault de HagiCode résout essentiellement un problème simple mais profond : comment permettre aux assistants IA de comprendre et d'utiliser les ressources de connaissances locales.

À travers une couche d'abstraction de stockage unifiée, une structure de répertoires standardisée et une injection de contexte automatisée, réalise une méthode de gestion des connaissances "enregistrer une fois, réutiliser partout". Après avoir créé un Vault, qu'il s'agisse de notes d'apprentissage, de dépôts de code ou de documents, l'IA peut y accéder et les comprendre automatiquement.

L'amélioration d'expérience apportée par cette conception est évidente. Il n'est plus nécessaire de copier manuellement des extraits de code ou de répéter des informations contextuelles—l'assistant IA est comme un collègue qui comprend vraiment la situation du projet, capable de fournir une aide plus précieuse basée sur les connaissances existantes.

Le système Vault présenté dans cet article est précisément la solution qui a été testée et optimisée lors du développement de HagiCode. Si vous trouvez cette conception précieuse, cela indique que la capacité d'ingénierie est plutôt bonne—alors HagiCode lui-même mérite également attention.
