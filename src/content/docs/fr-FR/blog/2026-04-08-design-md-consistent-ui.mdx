## Design.md : Une solution pour une conception UI front-end cohérente par l'IA

> À l'ère du développement front-end assisté par l'IA, comment maintenir la cohérence des UI générées par l'IA ? Cet article partage notre expérience pratique basée sur awesome-design-md pour construire une galerie de design, et comment créer un design.md structuré pour guider l'IA dans une conception UI normalisée.

## Contexte

Ceux qui ont utilisé l'IA pour écrire du code front-end ont probablement vécu une expérience similaire : pour la même page, en demandant à l'IA de la générer plusieurs fois, le style est différent à chaque fois. Parfois les coins sont arrondis, parfois carrés, parfois l'espacement est de 8px, parfois de 16px, et même le même bouton peut avoir un aspect différent dans différentes conversations.

Ce n'est pas un phénomène isolé. Avec la popularisation du développement assisté par l'IA, **le manque de cohérence des UI front-end générées par l'IA** est devenu un problème répandu. Différents assistants IA, différents prompts, et même le même assistant dans différentes conversations produisent des conceptions d'interface très variées. Cela entraîne d'énormes coûts de maintenance pour l'itération produit.

La racine du problème est en fait très simple : l'absence d'un **document de référence de design faisant autorité**. Les fichiers CSS traditionnels ne peuvent indiquer aux développeurs que "comment implémenter", mais ne peuvent pas transmettre complètement "pourquoi concevoir ainsi" et "quels modèles de design utiliser dans quels scénarios". Pour l'IA, elle a encore plus besoin d'une description structurée claire pour comprendre les spécifications de design.

En parallèle, la communauté open source dispose déjà de certaines excellentes ressources. Le projet [VoltAgent/awesome-design-md](https://github.com/VoltAgent/awesome-design-md) collecte des documents de système de design de nombreuses entreprises bien connues, chaque répertoire contenant README.md, DESIGN.md et un HTML de prévisualisation. Mais ceux-ci sont dispersés dans les dépôts amont, ce qui les rend difficiles à consulter et comparer rapidement.

Alors, est-il possible d'intégrer ces ressources pour créer une galerie de design facile à consulter, tout en consolidant un design.md structuré pour l'IA ?

La réponse est oui. Voici notre solution.

Pour résoudre ce problème, nous avons construit un site de galerie de design et créé un design.md normalisé, cet article étant le résumé de cette solution.

## Analyse

Avant de commencer à écrire du code, analysons quelques défis techniques de ce problème.

### Gestion des sources de contenu : comment unifier les ressources de design dispersées ?

Le dépôt amont awesome-design-md contient de nombreux documents de design, mais nous avons besoin d'un moyen de l'intégrer dans notre projet.

**Solution : utiliser git submodule**

```
awesome-design-md-site
└── vendor/awesome-design-md          # Ressources amont (git submodule)
```

Cela présente plusieurs avantages :
- Version contrôlable : peut verrouiller une version amont spécifique
- Construction hors ligne : pas besoin de requêter des API externes lors de la construction
- Révision du contenu : peut voir les changements spécifiques dans les PR

### Standardisation des données : comment unifier différentes structures de documents ?

Les structures de documents de design de différentes entreprises peuvent être différentes, certaines manquent de fichiers de prévisualisation, certaines ont des noms non normalisés. Nous devons effectuer un traitement de standardisation lors de la construction.

**Solution : analyse à la construction et génération d'entrées normalisées**

Le module central est `awesomeDesignCatalog.ts`, responsable de :
1. Scanner le répertoire `vendor/awesome-design-md/design-md/*`
2. Vérifier que chaque entrée contient les fichiers requis (README.md, DESIGN.md, au moins un fichier de prévisualisation)
3. Extraire et rendre le contenu Markdown en HTML
4. Générer des données d'entrée normalisées

### Architecture de site statique : comment fournir une recherche dynamique tout en maintenant le déploiement statique ?

Puisqu'il s'agit d'une galerie de design, la fonction de recherche est indispensable. Mais Astro est un générateur de site statique, comment implémenter une recherche en temps réel ?

**Solution : îlot React + synchronisation des paramètres de requête URL**

```typescript
// src/components/gallery/SearchToolbar.tsx

export function SearchToolbar() {
  const [query, setQuery] = useState('');

  // Synchronisation URL
  useEffect(() => {
    const params = new URLSearchParams(window.location.search);
    setQuery(params.get('q') || '');
  }, []);

  // Filtrage en temps réel
  const filtered = entries.filter(entry =>
    entry.searchText.includes(query)
  );

  return <input value={query} onChange={e => {
    setQuery(e.target.value);
    updateURL(e.target.value);
  }} />;
}
```

L'avantage de cette approche est qu'elle préserve la capacité de déploiement de sites statiques (peut être déployé sur n'importe quel service d'hébergement statique), tout en offrant une expérience utilisateur de filtrage instantané.

### Documentation de design : comment faire comprendre et respecter les spécifications de design par l'IA ?

C'est le cœur de toute la solution. Nous devons créer un design.md structuré que l'IA puisse comprendre et appliquer nos spécifications de design.

**Solution : s'inspirer de la structure DESIGN.md de ClickHouse**

Le [DESIGN.md de ClickHouse](https://github.com/ClickHouse/ClickHouse/blob/master/DESIGN.md) est une excellente référence, il contient :
- Thème Visuel & Atmosphère
- Palette de Couleurs & Rôles
- Règles de Typographie
- Styles de Composants
- Principes de Mise en Page
- Profondeur & Élévation
- Do's and Don'ts
- Comportement Responsive
- Guide de Prompt Agent

Notre approche est : **structure de référence, contenu réécrit**. Conserver la structure des chapitres du DESIGN.md de ClickHouse, mais remplacer le contenu par les tokens de design et spécifications de composants réellement utilisés dans notre projet.

## Solution

Basée sur l'analyse ci-dessus, notre solution comprend quatre modules principaux.

### 1. Pipeline d'ingestion de contenu

C'est la base de tout le système, responsable de l'extraction et de la standardisation du contenu à partir des ressources amont.

```typescript
// src/lib/content/awesomeDesignCatalog.ts

export async function scanSourceEntries(): Promise<DesignEntry[]> {
  const designDir = 'vendor/awesome-design-md/design-md';
  const entries: DesignEntry[] = [];

  for (const dir of await fs.readdir(designDir)) {
    const entryPath = path.join(designDir, dir);
    if (await isValidDesignEntry(entryPath)) {
      const entry = await normalizeDesignEntry(entryPath);
      entries.push(entry);
    }
  }

  return entries;
}

async function isValidDesignEntry(dir: string): Promise<boolean> {
  const requiredFiles = ['README.md', 'DESIGN.md'];
  for (const file of requiredFiles) {
    if (!(await fileExists(path.join(dir, file)))) {
      return false;
    }
  }
  return true;
}
```

### 2. Interface de navigation de la galerie

L'interface de la galerie comprend trois parties principales :

**Page d'accueil** : affiche une grille de cartes de toutes les entrées de design, chaque carte contenant :
- Titre et résumé de l'entrée de design
- Image de prévisualisation (si disponible)
- Mise en évidence de la recherche rapide

**Page de détails** : affiche agrégée toutes les informations complètes d'une entrée de design :
- Document README
- Document DESIGN
- Prévisualisation (support du basculement thème clair/sombre)
- Navigation vers les entrées adjacentes

**Navigation** : supporte le retour à la galerie, la navigation des entrées adjacentes

La galerie de la page d'accueil utilise une disposition de cartes à haute densité, étalant les entrées design.md de différentes sources dans un cadre visuel unifié, facilitant la comparaison rapide des styles de marque, des modèles de boutons et du rythme de mise en page.

En entrant dans une entrée spécifique, la page de détails place le résumé de design et la prévisualisation en temps réel sur la même page, réduisant le coût de basculement entre documents, prévisualisations et code source.

### 3. Fonction de recherche

La fonction de recherche est basée sur le filtrage côté client, utilisant les paramètres de requête URL pour maintenir l'état :

```typescript
// src/components/gallery/SearchToolbar.tsx

function SearchToolbar({ entries }: { entries: DesignEntry[] }) {
  const [query, setQuery] = useState('');
  const [results, setResults] = useState(entries);

  useEffect(() => {
    const params = new URLSearchParams(window.location.search);
    const initialQuery = params.get('q') || '';
    setQuery(initialQuery);
    filterEntries(initialQuery);
  }, []);

  const filterEntries = (searchQuery: string) => {
    const filtered = entries.filter(entry =>
      entry.searchText.toLowerCase().includes(searchQuery.toLowerCase())
    );
    setResults(filtered);
  };

  const handleChange = (e: React.ChangeEvent<HTMLInputElement>) => {
    const value = e.target.value;
    setQuery(value);
    filterEntries(value);

    // Mettre à jour l'URL (sans déclencher de rafraîchissement de page)
    const newUrl = value
      ? `${window.location.pathname}?q=${encodeURIComponent(value)}`
      : window.location.pathname;
    window.history.replaceState({}, '', newUrl);
  };

  return (
    <div className="search-toolbar">
      <input
        type="text"
        value={query}
        onChange={handleChange}
        placeholder="Rechercher des entrées de design..."
      />
      <span className="result-count">{results.length} résultats</span>
    </div>
  );
}
```

## Pratique

Après avoir compris la solution, comment la mettre en œuvre concrètement ?

### Étapes de mise en œuvre

**Première étape : initialiser le sous-module**

```bash
# Ajouter le dépôt amont comme sous-module
git submodule add https://github.com/VoltAgent/awesome-design-md.git vendor/awesome-design-md

# Initialiser et mettre à jour le sous-module
git submodule update --init --recursive
```

**Deuxième étape : créer le pipeline de contenu**

Implémenter `awesomeDesignCatalog.ts`, y compris :
- Logique de scan et validation de fichiers
- Rendu Markdown (en utilisant le moteur de rendu intégré d'Astro)
- Extraction des données d'entrée

**Troisième étape : construire l'UI de la galerie**

Utiliser Astro + React Islands pour créer :
- Mise en page de la galerie de la page d'accueil (grille de cartes)
- Composant de carte de design
- Barre d'outils de recherche
- Mise en page de la page de détails

**Quatrième étape : rédiger le document de design**

Basé sur la structure DESIGN.md de ClickHouse, remplir avec les tokens de design réels du projet. Mettre à jour README.md, ajouter un lien vers design.md.

### Points d'attention

**Sécurité** : Le rendu Markdown nécessite de filtrer le HTML non sécurisé. Le moteur de rendu intégré d'Astro filtre par défaut les balises script, mais il faut toujours faire attention aux risques XSS.

**Performance** : Un grand nombre d'iframes de prévisualisation peut affecter le chargement initial. Il est recommandé d'utiliser `loading="lazy"` pour retarder le chargement du contenu de prévisualisation.

**Maintenabilité** : design.md doit rester synchronisé avec l'implémentation du code. Il est recommandé d'ajouter des vérifications dans CI pour garantir que les variables CSS sont cohérentes entre la documentation et le code.

**Accessibilité** : Assurer que le contraste des couleurs respecte les standards WCAG AA (au moins 4.5:1).

### Guide d'utilisation IA

Après avoir créé design.md, comment faire en sorte que l'IA l'utilise réellement ? Voici quelques astuces pratiques :

**Astuce 1 : référencer explicitement dans les prompts**

```
Veuillez vous référer au fichier design.md à la racine du projet et utiliser les spécifications de design définies pour implémenter les composants suivants :
- Bouton : utiliser la couleur primary, coins arrondis 8px
- Carte : utiliser le niveau d'ombre elevation-2
```

**Astuce 2 : demander à l'IA de référencer des variables CSS spécifiques**

```
Implémenter une barre de navigation, exigeant :
- Couleur d'arrière-plan utiliser --color-bg-primary
- Bordure utiliser --color-border-subtle
- Texte utiliser --text-color-primary
```

**Astuce 3 : inclure le contenu design.md dans le prompt système**

Si votre outil IA prend en charge les prompts système personnalisés, vous pouvez directement ajouter le contenu principal de design.md.

### Stratégie de test

**Tests du pipeline de contenu** :
- Scénarios de fichiers manquants (absence de README.md ou DESIGN.md)
- Scénarios d'erreurs de format (échec de l'analyse Markdown)
- Scénarios de répertoires vides

**Tests de la fonction de recherche** :
- Gestion des résultats vides
- Caractères spéciaux (comme chinois, emoji)
- Vérification de la synchronisation URL

**Tests des composants UI** :
- Basculement thème clair/sombre
- Mise en page responsive
- État de chargement de la prévisualisation

### Processus de déploiement

```bash
# 1. Mettre à jour le sous-module vers la dernière version
git submodule update --remote

# 2. Reconstruire le site
npm run build

# 3. Déployer les ressources statiques
npm run deploy
```

Il est recommandé d'automatiser la mise à jour des sous-modules et la construction/déploiement, pouvant déclencher automatiquement le processus CI lors de la mise à jour du dépôt amont.

## Résumé

Le problème d'incohérence des UI générées par l'IA rencontré par HagiCode au cours du développement est essentiellement un manque de documentation de référence de design structurée. En construisant un site de galerie de design et en créant un design.md normalisé, nous avons résolu ce problème avec succès.

La valeur principale de cette solution réside dans :
- **Ressources unifiées** : intégrer les documents de système de design dispersés
- **Spécifications structurées** : présenter les spécifications de design sous une forme compréhensible par l'IA
- **Maintenance continue** : maintenir le contenu à jour via git submodule

Si vous utilisez également le développement front-end assisté par l'IA, nous vous recommandons d'essayer cette solution. Créer un design.md structuré peut non seulement améliorer la cohérence du code généré par l'IA, mais aussi aider l'équipe interne à maintenir l'uniformité des spécifications de design.
