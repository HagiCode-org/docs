## Le chemin vers l'intégration de Pi Agent dans HagiCode

> Cet article partage notre expérience pratique de l'intégration de Pi agent comme point d'entrée de flux de travail de premier ordre dans le projet HagiCode, y compris la conception de l'architecture, la mise en œuvre des composants principaux et les détails d'adaptation frontend et backend.

## Contexte

Tout commence par cette question : dans le projet HagiCode Mono, bien que `repos/Hagicode.Libs` ait déjà implémenté `PiProvider` réutilisable, `repos/hagicode-core` et `repos/web` n'ont pas encore élevé Pi au rang d'Agent CLI de premier ordre au niveau du projet. C'est comme avoir de bonnes chaussures sans avoir noué les lacets, on peut marcher mais il manque toujours quelque chose.

Le `PiProvider` existant se trouve dans `HagiCode.Libs/src/HagiCode.Libs.Providers/Pi/PiProvider.cs`, il implémente le protocole de communication CLI basé sur JSON séparé par des lignes (`--mode json --print`), supporte la gestion de sessions, les appels d'outils et la sortie en streaming. Ce code est bien écrit, mais il reste là, dormant.

Cette situation crée une rupture : les capacités sous-jacentes de Pi sont complètes, mais le chaînage au niveau du projet (de la configuration utilisateur à la surveillance de l'exécution) est manquant. C'est comme avoir peint un bon tableau sans l'accrocher au mur pour que les gens puissent l'apprécier. Il faut donc intégrer Pi comme nouveau provider actif `PiCli` dans tout le système, pour en faire un point d'entrée de flux de travail de premier ordre comme les autres Agent CLI.

Après tout, ce que nous voulons, c'est une expérience complète, pas des fragments dispersés.

## À propos de HagiCode

La solution partagée dans cet article provient de notre expérience pratique dans le projet [HagiCode](https://hagicode.com). HagiCode est un projet d'assistant de code intelligent, et au cours du développement nous devons intégrer plusieurs fournisseurs de capacités AI. La solution d'intégration de Pi agent décrite ici est précisément un modèle d'intégration réutilisable que nous avons résumé dans notre pratique, qui présente une valeur de référence pour des scénarios similaires d'intégration multi-fournisseurs. L'adresse GitHub du projet est [github.com/HagiCode-org/site](https://github.com/HagiCode-org/site).

## Conception de l'architecture

L'intégration de Pi agent utilise le **mode thin adapter**, plutôt que de réimplémenter le protocole de processus Pi au niveau core. Cette décision de conception n'est pas vraiment complexe, c'est juste qu'on s'est dit : pourquoi réinventer la roue ?

Après tout :

1. **Éviter la duplication de mise en œuvre** : la logique de construction de paramètres, de démarrage de processus, d'analyse JSON et de gestion des erreurs dans `PiProvider` est déjà très complète, la duplication créerait deux sources de comportement et deux matrices de tests. Cela dit, la maintenance serait pénible.
2. **Maintenir la cohérence** : garder une cohérence avec les méthodes d'intégration des CLI existants comme Kimi, Gemini, Reasonix, DeepAgents, tous délèguent via thin adapter à l'implémentation de la couche libs. Tout le monde fait comme ça, suivons le mouvement.
3. **Séparation des préoccupations** : `hagicode-core` se concentre sur les contrats d'exécution et la logique métier, les détails de processus sont confiés à `HagiCode.Libs`. Chacun son travail, n'est-ce pas parfait ?

Cette conception permet à HagiCode de maintenir la simplicité de la couche core tout en pouvant intégrer rapidement de nouveaux fournisseurs de capacités AI. Après tout, la simplicité est beauté, l'efficacité c'est de l'argent.

## Mise en œuvre des composants principaux

### Énumération Provider et Factory

Dans `hagicode-core/src/PCode.Models/AIProviderType.cs`, la valeur d'énumération `PiCli = 13` a été ajoutée :

```csharp
public enum AIProviderType
{
    ClaudeCodeCli = 0,
    // ... autres providers
    PiCli = 13,
}
```

Cette valeur d'énumération est à la racine de l'identité du provider, elle affecte l'énumération frontend `PCode_Models_AIProviderType.ts` générée par OpenAPI, la branche de création de `AIProviderFactory`, ainsi que la logique d'analyse du provider et de jugement des providers actifs. Dans `AIProviderFactory`, une nouvelle branche de création est enregistrée :

```csharp
case AIProviderType.PiCli:
    provider = new PiCliProvider(
        logger,
        configuration,
        providerFactory.GetRequiredService<ICliProvider<PiOptions>>(),
        providerFactory.GetService<IAgentCliRuntimeEnvironmentResolver>());
    break;
```

En fait, ces codes n'ont rien de spécial, c'est juste une énumération et un switch case. Mais ils sont importants, comme les notes sur une partition, ajoutées une par une pour jouer un morceau complet.

### Mise en œuvre du Thin Adapter

`PiCliProvider.cs` est le thin adapter principal, il implémente les interfaces `IAIProvider`, `IVersionedAIProvider` et `IAsyncDisposable`. Via le constructeur, il reçoit `ICliProvider<PiOptions>` (depuis `HagiCode.Libs`), mappe `AIRequest` / `ProviderConfiguration` vers `PiOptions`, puis délègue l'exécution, le ping, la requête de version et autres opérations au provider sous-jacent.

La clé est de gérer le flux d'événements JSON spécifique à Pi, y compris les événements `assistant.thought`, `assistant`, `terminal.completed`, etc. Ces événements doivent être correctement analysés et convertis au format standard du système lors de la sortie en streaming. C'est un peu comme de la traduction, traduire une langue dans une autre, le sens doit être transmis correctement.

### Préconfiguration de la profession principale

Dans `main-professions.yaml`, l'entrée de profession principale `profession-pi` a été ajoutée :

```yaml
- Id: "profession-pi"
  Name: "Pi"
  Family: "pi"
  Summary: "hero.professionCopy.primary.pi.summary"
  Icon: "executor-avatar:Pi"
  SourceLabel: "hero.professionCopy.sources.aiProvidersPiCli"
  ProviderType: "PiCli"
  SortOrder: 59
  DefaultEnabled: true
  DefaultParameters:
    binary: "pi"
    provider: "omniroute"
    thinking: "balanced"
```

Cela garantit que le snapshot backend et le répertoire fallback frontend ont une identité Pi cohérente, permet de gérer la configuration Pi via l'éditeur Hero existant, et l'ajout de Pi ne brise pas la consommabilité des entrées de professions principales existantes. Après tout, nous ne voulons pas déranger les choses existantes en ajoutant une nouvelle fonctionnalité, ce serait perdre beaucoup pour gagner peu.

### Registre de surveillance

`AgentCliMonitoringRegistry` doit ajouter le descriptor de surveillance de Pi, permettant au système d'analyser le chemin exécutable, d'afficher le nom de marque, d'effectuer des sondages de santé, et d'afficher l'état de Pi dans la barre d'état et les détails de santé :

```csharp
new AgentCliMonitoringDescriptor(
    CliId: "pi",
    DisplayName: "Pi",
    ProviderType: AIProviderType.PiCli,
    DisplayOrder: 13,
    Strategy: AgentCliMonitoringStrategy.Grain,
    NotConfiguredMessage: "Pi CLI is not configured or executable not found.",
    EnabledPaths: [],
    ExecutablePathConfigPaths: ["Hero:PrimaryProfessions:Pi:ExecutablePath"],
    DefaultExecutablePath: "pi")
```

Le système de surveillance est comme un tableau de bord, vous dit comment la voiture roule. L'état de Pi est clair d'un coup d'œil, donc les utilisateurs savent si tout va bien.

## Adaptation frontend

### Adaptation du type d'exécuteur

Le frontend doit mettre à jour `executorTypeAdapter.ts`, ajouter la logique de reconnaissance de type pour Pi :

```typescript
const PI_IDS = new Set<string>([
  PCode_Models_AIProviderType.PI_CLI,
  'Pi',
  'PiCli',
  'pi',
  'picli',
  'pi-cli',
]);

export function isPi(value: string): boolean {
  const normalized = normalize(value);
  const normalizedLower = normalized.toLowerCase();
  return PI_IDS.has(normalized)
    || normalizedLower.includes('pi-cli')
    || normalizedLower.includes('picli')
    || normalizedLower === 'pi';
}
```

C'est comme donner plusieurs noms à Pi, peu importe comment vous l'appelez, il sait que vous l'appelez. Après tout, le nom n'est pas important, ce qui compte c'est de savoir qui vous êtes.

### Répertoire Fallback Hero

Dans `hero.ts`, ajouter l'entrée fallback pour Pi, garantissant que même si les données backend ne sont pas chargées, le frontend peut afficher normalement la configuration Pi :

```typescript
{
  id: "profession-pi",
  name: "Pi",
  family: "pi",
  summary: "hero.professionCopy.primary.pi.summary",
  icon: "executor-avatar:Pi",
  sourceLabel: "hero.professionCopy.sources.aiProvidersPiCli",
  providerType: AIProviderType.PI_CLI,
  sortOrder: 59,
  isReadOnly: true,
  managedParameterKeys: {
    // clés de paramètres supportées par la première version
  },
  defaultParameters: {
    binary: "pi",
    provider: "omniroute",
    thinking: "balanced",
  },
}
```

Fallback est comme un plan de secours, au cas où le backend tombe ou que les données ne se chargent pas, le frontend peut fonctionner normalement. Après tout, personne ne veut être paniqué à ce moment-là.

### Localisation et formulaire

Dans `locales/*/common/{hero,settings}.yml`, ajouter les traductions liées à Pi, et dans `HeroCliEquipmentForm.tsx`, ajouter un bloc de champs de configuration pour Pi, supportant les champs binary, provider, thinking, sessionDirectory et les commutateurs d'outils/session.

La première version de Pi n'expose que les champs minimum nécessaires, les fonctionnalités complexes comme les listes allow/deny d'outils et l'éditeur de variables d'environnement sont explicitement reportées aux modifications ultérieures. Après tout, on ne peut pas tout manger d'un coup, il vaut mieux y aller doucement.

## Exemples de configuration

### Configuration backend

Dans `appsettings.yml`, configurer les paramètres Pi :

```yaml
Hero:
  PrimaryProfessions:
    Pi:
      ExecutablePath: /usr/local/bin/pi
      Provider: omniroute
      Thinking: balanced
      SessionDirectory: ~/.pi/sessions
      NoSession: false
      DisableAllTools: false
      DisableBuiltinTools: false
```

### Configuration frontend

Dans l'éditeur Hero, configurer la profession Pi :

```typescript
{
  id: "my-pi-profession",
  name: "My Pi Profession",
  family: "pi",
  providerType: AIProviderType.PI_CLI,
  primaryModel: {
    provider: "PiCli",
    model: "glm-4.7",
    providerSettings: {
      provider: "omniroute",
      thinking: "balanced",
      sessionDirectory: "/Users/username/.pi/sessions",
      noSession: false,
      disableAllTools: false,
      disableBuiltinTools: false,
    },
  },
}
```

Les fichiers de configuration sont comme des recettes, suivez-les et vous ferez un bon plat. Parfois, même en suivant la recette, on peut brûler le plat... mais cette configuration est très claire, il ne devrait pas y avoir de problème.

## Résolution des problèmes courants

### Exécutable Pi introuvable

Si le contrôle de santé retourne "Pi executable was not found.", il faut vérifier si pi est dans le PATH, ou confirmer que le chemin configuré est correct. La solution est de s'assurer que `pi` est installé et dans le PATH, ou de configurer le bon `ExecutablePath` dans `appsettings.yml`.

C'est comme ne pas trouver la clé de la maison, il faut réfléchir si on l'a mise au mauvais endroit. En fait, la solution est assez simple, soit remettre la clé à sa place, soit changer la serrure.

### Champs de configuration non reconnus

Si au démarrage l'erreur "PiCli runtime settings [...] are not supported" est lancée, vérifier si seuls les champs de configuration supportés par la première version sont utilisés. Les champs supportés par la première version incluent : `provider`, `thinking`, `sessionDirectory`, `noSession`, `disableAllTools`, `disableBuiltinTools`.

Parfois c'est juste de l'avidité, trop de fonctionnalités voulues, résultat le système ne supporte pas. En fait, les fonctionnalités de la première version sont déjà suffisantes, trop en prendre à la fois fait mal.

### Impossible de sélectionner Pi dans le frontend

Si l'éditeur Hero n'a pas d'option Pi, vérifier si `npm run generate:api` a été exécuté pour régénérer l'énumération frontend, si `hero.ts` a l'entrée `profession-pi`, et si les textes de localisation ont été correctement ajoutés.

Résoudre les problèmes est comme chercher un objet perdu, il faut y aller étape par étape. Après tout, chercher aveuglément ne mène nulle part, il faut chercher avec logique.

## Bonnes pratiques

1. **Utiliser le mode thin adapter** : ne pas réimplémenter le protocole de processus au niveau core, déléguer au provider de la couche libs. Cela évite la duplication de mise en œuvre et maintient la cohérence du code. Après tout, réinventer la roue est fatigant et sujet à erreurs.

2. **Maintenir la cohérence des noms** : utiliser des conventions de dénomination unifiées entre frontend et backend, éviter la confusion. L'énumération Provider utilise `PiCli`, l'ID CLI utilise `"pi"`, le nom d'affichage utilise `"Pi"`. De bons noms réduisent les coûts de communication.

3. **Prioriser les préconfigurations** : la première version devrait être basée sur la préconfiguration `profession-pi`, plutôt que de demander à l'utilisateur de configurer manuellement. Cela permet aux utilisateurs de démarrer rapidement, réduit la complexité de configuration. Les utilisateurs aiment les choses simples, les complexes qu'ils me laissent gérer.

4. **Se concentrer sur les messages d'erreur** : s'assurer que les messages d'erreur sont clairs et actionnables, aidant les utilisateurs à localiser rapidement les problèmes. Des messages d'erreur clairs empêchent les utilisateurs de paniquer à cause d'une erreur.

5. **Compatibilité des versions** : compte tenu de la stabilité de sérialisation des valeurs d'énumération `AIProviderType`, les modifications doivent être traitées avec prudence. La valeur d'énumération `AIProviderType.PiCli = 13` ne peut pas être facilement modifiée. Après tout, changer cette valeur pourrait briser la compatibilité ascendante, ce serait problématique.

## Résumé

Grâce au mode thin adapter, nous avons réussi à intégrer Pi agent dans le système HagiCode, en faisant un point d'entrée de flux de travail de premier ordre comme les autres Agent CLI. Les avantages principaux de cette solution sont :

- Évité la duplication de mise en œuvre, réutilisé le `PiProvider` existant de la couche libs
- Maintenu une cohérence avec les méthodes d'intégration des providers existants, réduit les coûts de maintenance
- Réalisé le chaînage complet de la configuration utilisateur à la surveillance de l'exécution

Cette pratique de HagiCode prouve que le mode thin adapter est une solution efficace pour intégrer des fournisseurs de capacités AI. Il nous permet de supporter rapidement de nouveaux agents tout en maintenant la stabilité et la maintenabilité du système.

Faire de la technologie, c'est comme ça : trouver un bon modèle, puis le réutiliser. Cela permet d'avancer rapidement sans se perdre. Comme marcher, trouver une bonne route, continuer à marcher, juste s'arrêter parfois pour admirer le paysage...

Si vous faites aussi de l'intégration de capacités AI multi-fournisseurs, j'espère que cette solution vous donnera des inspirations. Si vous êtes intéressé par le projet HagiCode, venez échanger sur GitHub. Après tout, la technologie, c'est en échangeant qu'on progresse.
