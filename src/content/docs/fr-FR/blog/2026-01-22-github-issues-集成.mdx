## Construire une intégration GitHub Issues à partir de zéro : la pratique de connexion frontend directe de HagiCode

> Cet article documente le processus complet d'intégration de GitHub Issues dans la plateforme HagiCode. Nous explorerons comment, grâce à l'architecture "connexion frontend directe + backend minimisé", nous pouvons maintenir un backend léger tout en assurant une authentification OAuth sécurisée et une synchronisation efficace des Issues.

## Contexte : Pourquoi intégrer GitHub ?

HagiCode, en tant que plateforme de développement assistée par IA, a pour valeur centrale de connecter les idées et leur réalisation. Mais dans l'utilisation pratique, nous avons constaté qu'après avoir complété une Proposition (Proposal) dans HagiCode, les utilisateurs doivent souvent copier manuellement le contenu dans GitHub Issues pour le suivi de projet.

Cela présente plusieurs points de douleur évidents :

1.  **Workflow fragmenté** : Les utilisateurs doivent basculer entre deux systèmes, une expérience non seulement peu fluide mais aussi susceptible de perdre des informations critiques lors du copier-coller.
2.  **Collaboration difficile** : Les autres membres de l'équipe ont l'habitude de consulter les tâches sur GitHub et ne peuvent pas voir directement la progression des propositions dans HagiCode.
3.  **Travail répétitif** : Chaque fois qu'une proposition est mise à jour, il faut mettre à jour manuellement l'Issue correspondante sur GitHub, augmentant inutilement les coûts de maintenance.

Pour résoudre ce problème, nous avons décidé d'introduire la fonctionnalité **GitHub Issues Integration**, connectant les sessions HagiCode aux dépôts GitHub pour réaliser une "synchronisation en un clic".

## Choix technologique : Connexion frontend directe vs Proxy backend

Lors de la conception de la solution d'intégration, deux chemins s'offraient à nous : le traditionnel "mode proxy backend" et le plus radical "mode connexion frontend directe".

### Comparaison des solutions

Dans le traditionnel **mode proxy backend**, toutes les requêtes du frontend doivent d'abord passer par notre backend, qui ensuite appelle l'API GitHub. Bien que la logique soit centralisée, cela alourdit considérablement le backend :

1.  **Backend gonflé** : Nécessite d'écrire un encapsuleur client dédié pour l'API GitHub et de gérer la machine d'état complexe d'OAuth.
2.  **Risque Token** : Le GitHub Token de l'utilisateur doit être stocké dans la base de données du backend ; bien qu'il puisse être chiffré, cela augmente nonetheless la surface de risque de sécurité.
3.  **Coût de développement** : Nécessite une migration de base de données pour stocker les Tokens et la maintenance d'un service de synchronisation supplémentaire.

Le **mode connexion frontend directe** est quant à lui beaucoup plus léger. Dans cette solution, nous n'utilisons le backend que pour traiter l'étape la plus sensible de "l'échange de clés" (OAuth callback). Une fois le Token obtenu, il est stocké directement dans le localStorage du navigateur. Les opérations ultérieures de création d'Issue, de mise à jour de commentaires, etc., sont effectuées directement par le frontend via des requêtes HTTP vers GitHub.

| Dimension de comparaison | Mode proxy backend | Mode connexion frontend directe |
| :--- | :--- | :--- |
| **Complexité backend** | Nécessite un service OAuth complet et un client API GitHub | Uniquement un endpoint de callback OAuth |
| **Gestion des Tokens** | Stockage chiffré dans la base de données, risque de fuite | Stockage dans le navigateur, visible uniquement par l'utilisateur |
| **Coût de mise en œuvre** | Migration de base de données, développement multi-services | Principalement du travail frontend |
| **Expérience utilisateur** | Logique unifiée, mais latence serveur potentiellement plus élevée | Réponse extrêmement rapide, interaction directe avec GitHub |

Considérant notre objectif d'intégration rapide et de modifications backend minimales, **nous avons finalement adopté le "mode connexion frontend directe"**. C'est comme délivrer un "pass temporaire" au navigateur ; une fois le passeport obtenu, le navigateur peut aller s'occuper de ses affaires sur GitHub sans avoir à demander l'approbation de l'administrateur backend à chaque fois.

## Conception centrale : Flux de données et sécurité

Après avoir déterminé l'architecture, nous devions concevoir le flux de données spécifique. Le cœur du processus de synchronisation réside dans la manière d'obtenir le Token de manière sécurisée et de l'utiliser efficacement.

### Diagramme d'architecture globale

L'ensemble du système peut être abstrait en trois rôles : le navigateur (frontend), le backend HagiCode, GitHub.

```text
+--------------+        +--------------+        +--------------+
|  Frontend    |        |    Backend   |        |    GitHub    |
|   React      |        |   ASP.NET    |        |    REST API  |
|  +--------+  |        |              |        |              |
|  |  OAuth |--+--------> /callback    |        |              |
|  |  flux  |  |        |              |        |              |
|  +--------+  |        |              |        |              |
|              |        |              |        |              |
|  +--------+  |        |  +--------+  |        |  +--------+  |
|  |GitHub  |  +------------>Session |  +----------> Issues |  |
|  |API     |  |        |  |Metadata|  |        |  |        |  |
|  |connexion|  |        |  +--------+  |        |  +--------+  |
|  |directe |  |        |              |        |              |
|  +--------+  |        |              |        |              |
+--------------+        +--------------+        +--------------+
```

**Le point clé** : seule une petite étape d'OAuth (obtenir le code pour échanger contre le token) doit passer par le backend ; par la suite, les tâches lourdes (créer l'Issue) sont directement gérées par le frontend avec GitHub.

### Explication détaillée du flux de synchronisation

Lorsque l'utilisateur clique sur le bouton "Sync to GitHub" dans l'interface HagiCode, une série d'actions complexes se déclenchent :

```text
Utilisateur clique "Sync to GitHub"
         │
         ▼
1. Le frontend vérifie localStorage pour obtenir le GitHub Token
         │
         ▼
2. Formater le contenu de l'Issue (convertir la Proposition en Markdown)
         │
         ▼
3. Le frontend appelle directement l'API GitHub pour créer/mettre à jour l'Issue
         │
         ▼
4. Appeler l'API backend HagiCode pour mettre à jour Session.metadata (stocker l'URL de l'Issue, etc.)
         │
         ▼
5. Le backend diffuse l'événement SessionUpdated via SignalR
         │
         ▼
6. Le frontend reçoit l'événement, met à jour l'UI pour afficher l'état "synchronisé"
```

### Conception de la sécurité

La sécurité est toujours primordiale lors de l'intégration de services tiers. Nous avons pris les mesures suivantes en compte :

1.  **Protection contre les attaques CSRF** : Lors de la redirection OAuth, générer un paramètre `state` aléatoire et le stocker dans sessionStorage. Vérifier strictement le state lors du callback pour empêcher les requêtes falsifiées.
2.  **Isolement du stockage des Tokens** : Les Tokens sont stockés uniquement dans le `localStorage` du navigateur. Tirant parti de la politique de même origine (Same-Origin Policy), seuls les scripts HagiCode peuvent les lire, évitant que les fuites de base de données côté serveur n'affectent les utilisateurs.
3.  **Limites d'erreur** : Pour les erreurs courantes de l'API GitHub (comme 401 Token expiré, 422 échec de validation, 429 limitation de débit), une logique de gestion d'erreur dédiée a été conçue pour fournir des indications conviviales aux utilisateurs.

## Pratique : Détails d'implémentation du code

Théorie et pratique sont deux choses différentes, voyons comment le code est concrètement implémenté.

### 1. Modifications minimales du backend

Le backend n'a besoin de faire que deux choses : stocker les informations de synchronisation et traiter le callback OAuth.

**Modifications de la base de données**
Nous devons simplement ajouter une colonne `Metadata` à la table `Sessions` pour stocker des informations étendues au format JSON.

```sql
-- Ajouter la colonne metadata à la table Sessions
ALTER TABLE "Sessions" ADD COLUMN "Metadata" text NULL;
```

**Définition des entités et DTO**


### 2. Flux OAuth frontend

C'est le point d'entrée de la connexion. Nous utilisons le standard Authorization Code Flow.

```typescript
// src/HagiCode.Client/src/services/githubOAuth.ts

// Générer l'URL d'autorisation et rediriger
export async function generateAuthUrl(): Promise<string> {
  const state = generateRandomString(); // Générer une chaîne aléatoire anti-CSRF
  sessionStorage.setItem('hagicode_github_state', state);
  
  const params = new URLSearchParams({
    client_id: clientId,
    redirect_uri: window.location.origin + '/settings?tab=github&oauth=callback',
    scope: ['repo', 'public_repo'].join(' '),
    state: state,
  });
  
  return `https://github.com/login/oauth/authorize?${params.toString()}`;
}

// Dans la page de callback, traiter l'échange Code contre Token
export async function exchangeCodeForToken(code: string, state: string): Promise<GitHubToken> {
  // 1. Vérifier State pour prévenir CSRF
  const savedState = sessionStorage.getItem('hagicode_github_state');
  if (state !== savedState) throw new Error('Invalid state parameter');

  // 2. Appeler l'API backend pour l'échange de Token
  // Note : cela doit passer par le backend car ClientSecret est nécessaire et ne peut pas être exposé au frontend
  const response = await fetch('/api/GitHubOAuth/callback', {
    method: 'POST',
    headers: { 'Content-Type': 'application/json' },
    body: JSON.stringify({ code, state, redirectUri: window.location.origin + '/settings?tab=github&oauth=callback' }),
  });

  if (!response.ok) throw new Error('Failed to exchange token');
  
  const token = await response.json();
  
  // 3. Stocker dans LocalStorage
  saveToken(token);
  return token;
}
```

### 3. Encapsulation du client API GitHub

Une fois le Token obtenu, nous avons besoin d'un outil puissant pour appeler l'API GitHub.

```typescript
// src/HagiCode.Client/src/services/githubApiClient.ts

const GITHUB_API_BASE = 'https://api.github.com';

// Encapsulation des requêtes principales
async function githubApi<T>(endpoint: string, options: RequestInit = {}): Promise<T> {
  const token = localStorage.getItem('gh_token');
  if (!token) throw new Error('Not connected to GitHub');
  
  const response = await fetch(`${GITHUB_API_BASE}${endpoint}`, {
    ...options,
    headers: {
      ...options.headers,
      Authorization: `Bearer ${token}`,
      Accept: 'application/vnd.github.v3+json', // Spécifier la version de l'API
    },
  });
  
  // Logique de gestion des erreurs
  if (!response.ok) {
    if (response.status === 401) throw new Error('GitHub Token invalide, veuillez vous reconnecter');
    if (response.status === 403) throw new Error('Accès non autorisé au dépôt ou limite de débit dépassée');
    if (response.status === 422) throw new Error('Échec de la validation de l\'Issue, le titre est peut-être en double');
    throw new Error(`GitHub API Error: ${response.statusText}`);
  }
  
  return response.json();
}

// Créer une Issue
export async function createIssue(owner: string, repo: string, data: { title: string, body: string, labels: string[] }) {
  return githubApi(`/repos/${owner}/${repo}/issues`, {
    method: 'POST',
    body: JSON.stringify(data),
  });
}
```

### 4. Formatage du contenu et synchronisation

La dernière étape consiste à convertir les données de Session HagiCode au format GitHub Issue. C'est un peu comme un travail de "traduction".

```typescript
// Convertir l'objet Session en chaîne Markdown
function formatIssueForSession(session: Session): string {
  let content = `# ${session.title}\n\n`;
  content += `**> HagiCode Session:** #${session.code}\n`;
  content += `**> Status:** ${session.status}\n\n`;
  content += `## Description\n\n${session.description || 'No description provided.'}\n\n`;
  
  // Si c'est de type Proposal, ajouter des champs supplémentaires
  if (session.type === 'proposal') {
    content += `## Chief Complaint\n\n${session.chiefComplaint || ''}\n\n`;
    // Ajouter un deep lien pour faciliter le retour de GitHub vers HagiCode
    content += `---\n\n**[View in HagiCode](hagicode://sessions/${session.id})**\n`;
  }
  
  return content;
}

// Logique principale du clic sur le bouton de synchronisation
const handleSync = async (session: Session) => {
  try {
    const repoInfo = parseRepositoryFromUrl(session.repoUrl); // Analyser l'URL du dépôt
    if (!repoInfo) throw new Error('Invalid repository URL');

    toast.loading('Synchronisation vers GitHub...');
    
    // 1. Formater le contenu
    const issueBody = formatIssueForSession(session);
    
    // 2. Appeler l'API
    const issue = await githubApiClient.createIssue(repoInfo.owner, repoInfo.repo, {
      title: `[HagiCode] ${session.title}`,
      body: issueBody,
      labels: ['hagicode', 'proposal', `status:${session.status}`],
    });
    
    // 3. Mettre à jour Session Metadata (enregistrer le lien de l'Issue)
    await SessionsService.patchApiSessionsSessionId(session.id, {
      metadata: {
        githubIssue: {
          owner: repoInfo.owner,
          repo: repoInfo.repo,
          issueNumber: issue.number,
          issueUrl: issue.html_url,
          syncedAt: new Date().toISOString(),
        }
      }
    });

    toast.success('Synchronisation réussie !');
  } catch (err) {
    console.error(err);
    toast.error('Échec de la synchronisation, veuillez vérifier le Token ou le réseau');
  }
};
```

## Conclusion et perspectives

Grâce à cette solution "connexion frontend directe", nous avons réalisé une intégration transparente de GitHub Issues avec un minimum de code backend.

### Réussites
1.  **Haute efficacité de développement** : modifications backend minimales, principalement l'ajout d'un champ en base de données et d'une interface simple de callback OAuth, la majeure partie de la logique étant effectuée au frontend.
2.  **Bonne sécurité** : Les Tokens ne traversent pas la base de données du serveur, réduisant les risques de fuite.
3.  **Excellente expérience utilisateur** : requêtes directes depuis le frontend, réponse rapide, pas besoin de passer par le backend.

### Points d'attention
Lors du déploiement en production, attention à quelques pièges :
- **Configuration OAuth App** : N'oubliez pas de remplir correctement l'`URL de callback d'autorisation` dans les paramètres de l'OAuth App GitHub (généralement `http://localhost:3000/settings?tab=github&oauth=callback`).
- **Limitation de débit** : L'API GitHub est assez stricte sur les requêtes non authentifiées, mais avec un Token c'est généralement suffisant (5000 requêtes/heure).
- **Analyse d'URL** : Les URL de dépôt saisies par les utilisateurs sont variées, n'oubliez pas que les regex doivent correspondre au suffixe `.git`, au format SSH, etc.

### Améliorations futures
La fonctionnalité actuelle est encore une synchronisation unidirectionnelle (HagiCode -> GitHub). À l'avenir, nous prévoyons de réaliser une synchronisation bidirectionnelle via GitHub Webhooks ; par exemple, fermer une Issue sur GitHub mettrait automatiquement à jour l'état de la session HagiCode. Cela nécessite d'exposer un endpoint de réception Webhook dans le backend, ce qui est la prochaine étape intéressante à réaliser.
