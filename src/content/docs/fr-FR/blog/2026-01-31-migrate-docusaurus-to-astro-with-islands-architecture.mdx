## De Docusaurus 3.x à Astro 5.x : Analyse pratique de la migration du site HagiCode

> Cet article examine le processus complet de migration du site officiel de HagiCode de Docusaurus 3.x vers Astro 5.x. Nous explorerons en profondeur comment résoudre les goulots d'étranglement de performance grâce à l'architecture Islands d'Astro, tout en préservant nos assets de composants React existants, pour améliorer à la fois la vitesse de construction et les performances de chargement.


## Contexte

En janvier 2026, nous avons effectué une "opération de transplantation cardiaque" sur le site officiel de HagiCode - une migration complète du framework central de Docusaurus 3.x vers Astro 5.x. Il ne s'agissait pas d'une refonte impulsive, mais d'une décision technique réfléchie.

Avant la migration, notre site était fonctionnellement complet, mais commençait à montrer des signes de "maladie de richesse" : volume des artefacts de construction gonflé, charge JavaScript excessive, et vitesse de chargement des pages insuffisante pour les pages de documentation complexes. En tant que projet d'assistant de code IA, HagiCode nécessite des mises à jour fréquentes de la documentation et des présentations de fonctionnalités, et l'efficacité de la construction impacte directement la vitesse de publication. De plus, nous souhaitions que notre site soit plus favorable aux moteurs de recherche (SEO) afin que davantage de développeurs découvrent ce projet.

Pour résoudre ces points de douleur, nous avons pris une décision audacieuse : reconstruire entièrement le système de construction et migrer vers Astro. Les changements apportés par cette décision pourraient être plus importants que vous ne l'imaginez - j'en parlerai en détail plus tard.

## À propos de HagiCode

La solution de migration de site présentée dans cet article provient de notre expérience pratique dans le projet [HagiCode](https://github.com/HagiCode-org/site).

HagiCode est un assistant de code IA dédié à l'amélioration de l'efficacité de développement. Nous nous concentrons non seulement sur l'itération des fonctionnalités principales, mais accordons également une grande importance à l'expérience développeur. Cette refonte du site vise également à offrir aux utilisateurs une vitesse de chargement extrême lorsqu'ils naviguent dans la documentation et le site officiel.

## Pourquoi abandonner Docusaurus mature ?

Dans l'écosystème React, Docusaurus a toujours été la "réponse standard" pour les sites de documentation. Il est prêt à l'emploi, riche en plugins, et sa communauté est active. Cependant, avec l'augmentation des fonctionnalités de HagiCode, nous avons également ressenti ses limites :

1.  **Goulots d'étranglement de performance** : Docusaurus est essentiellement un React SPA (application monopage). Même pour écrire des pages purement statiques, le client doit charger le runtime React et effectuer l'hydratation, ce qui est trop lourd pour de simples pages de documentation.
2.  **Volume des ressources** : Même si le contenu de la page est minimal, le volume JS après empaquetage reste relativement fixe, ce qui n'est pas idéal pour les utilisateurs mobiles et les environnements réseau médiocres.
3.  **Flexibilité insuffisante** : Bien que l'extension soit possible, nous souhaitions un contrôle plus bas niveau sur le processus de construction.

L'apparition d'Astro résout parfaitement ces problèmes. Il offre une nouvelle "architecture Islands" : par défaut, Astro génère du HTML statique avec zéro JavaScript, seuls les composants nécessitant une interaction sont "activés" et chargent du JS. Cela signifie que la majorité du contenu de notre site est du HTML pur, extrêmement rapide.

## Stratégie centrale de migration : transition architecturale en douceur

La migration n'est pas un simple copier-coller, mais un changement de paradigme. Nous sommes passés du "mode tout React" de Docusaurus au mode "Core + Islands" d'Astro.

### 1. Refonte du système de configuration

D'abord, nous devons passer de `docusaurus.config.ts` à `astro.config.mjs`. Il ne s'agit pas seulement d'un changement de nom de fichier, mais d'une réécriture de la logique de routage et de construction.

Dans Docusaurus, tout est plugin ; dans Astro, tout est intégration. Nous devons redéfinir le chemin de base du site, le mode de sortie de construction (statique vs SSR) et les stratégies de compression des ressources.

**Avant migration :**
```typescript
// docusaurus.config.ts
export default {
  title: 'HagiCode',
  url: 'https://hagicode.com',
  baseUrl: '/',
  // ... plus de configuration
};
```

**Après migration :**
```javascript
// astro.config.mjs
import { defineConfig } from 'astro/config';
import react from '@astrojs/react';

export default defineConfig({
  integrations: [react()],
  site: 'https://hagicode.com',
  base: '/',
  // Configuration d'optimisation pour les ressources statiques
  build: {
    inlineStylesheets: 'auto',
  },
});
```

### 2. Conservation et transformation des composants React

C'était la partie la plus difficile de la migration. Notre site actuel possède de nombreux composants React (comme les composants Tabs, la coloration syntaxique, les boutons de feedback, etc.). Les jeter serait dommage, mais tous les conserver entraînerait une charge JS excessive.

HagiCode a adopté une **stratégie d'hydratation progressive** :
*   **Composants purement statiques** : Pour le contenu de présentation (comme l'en-tête, le pied de page, la documentation en texte brut), réécrits en composants Astro (fichiers `.astro`), rendus directement en HTML au moment de la construction.
*   **Îlots interactifs** : Pour les composants qui doivent conserver l'interactivité (comme le sélecteur de thème, les onglets, les boutons de copie des blocs de code), conservez l'implémentation React et ajoutez les directives `client:load` ou `client:visible`.

Par exemple, notre composant Tabs couramment utilisé dans la documentation :

```jsx
// src/components/Tabs.jsx
import { useState } from 'react';
import './Tabs.css'; // Import des styles

export default function Tabs({ items }) {
  const [activeIndex, setActiveIndex] = useState(0);
  // ... logique d'état
  return (
    <div className="tabs-wrapper">
      {/* logique de rendu */}
    </div>
  );
}
```

Lors de l'utilisation dans Markdown, nous indiquons explicitement à Astro : "ce composant a besoin de JS" :

```markdown
// src/content/docs/example.mdx
import Tabs from '../../components/Tabs.jsx';

<!-- Le JS n'est chargé que lorsque le composant entre dans la vue -->
<Tabs client:visible items={...} />
```

Ainsi, les composants interactifs hors de la vue n'accaparent pas la bande passante, optimisant considérablement la vitesse de chargement de l'écran initial.

### 3. Adaptation du système de styles : de CSS Modules à Scoped

Docusaurus prend en charge CSS Modules par défaut, tandis qu'Astro privilégie l'utilisation de Scoped CSS (via les balises `<style>`). Les deux idées centrales sont l'isolement des styles, mais la syntaxe diffère.

Dans la migration de HagiCode, nous avons décomposé la plupart des CSS Modules complexes en styles Scoped d'Astro. C'est en fait une bonne chose, car dans les fichiers `.astro`, les styles et les modèles sont écrits dans le même fichier, ce qui les rend plus intuitifs à maintenir.

**Avant transformation :**
```css
/* Tabs.module.css */
.wrapper { background: var(--ifm-background-color); }
```

**Après transformation (Astro Scoped) :**
```html
<!-- Tabs.astro -->
<div class="tabs-wrapper">
  <slot />
</div>

<style>
  .tabs-wrapper {
    /* Utilisation directe des variables CSS, adaptation du thème */
    background: var(--bg-color);
    padding: 1rem;
  }
</style>
```

En parallèle, nous avons unifié le système global de variables CSS, en utilisant les capacités de détection d'environnement d'Astro pour assurer une transition sans faille du mode sombre entre les différentes pages.

## Pièges rencontrés en pratique et solutions

Lors du processus de migration réel de HagiCode, nous avons rencontré de nombreux obstacles. Voici quelques-uns des plus typiques à partager.

### 1. Points de douleur des chemins et variables d'environnement

HagiCode prend en charge le déploiement de sous-chemins (comme le déploiement dans un sous-répertoire de GitHub Pages). Dans Docusaurus, il gère automatiquement `baseUrl`. Mais dans Astro, nous devons être plus prudents lors du traitement des liens d'images et des requêtes API.

Nous avons introduit un mécanisme de variables d'environnement pour une gestion unifiée :

```javascript
// Gérer les chemins dans le script de construction
const getBasePath = () => import.meta.env.VITE_SITE_BASE || '/';
```

Rappelez-vous, ne codez pas en dur les chemins commençant par `/` dans le code. Dans les environnements de développement et de production, ou après avoir configuré le chemin `base`, cela entraînera des ressources 404.

### 2. Compatibilité des scripts CommonJS

Notre ancien site comportait quelques scripts Node.js (pour récupérer automatiquement les données Metrics, mettre à jour le sitemap, etc.), écrits en CommonJS (`require`). Astro et les outils de construction modernes embrassent pleinement les ES Modules (`import`/`export`).

Si vous avez des scripts similaires, n'oubliez pas de les refactoriser tous en ES Modules. C'est la tendance générale, refactorisez-les tôt pour gagner de la tranquillité d'esprit.

```javascript
// Ancienne méthode
const fs = require('fs');

// Nouvelle méthode
import fs from 'fs';
```

### 3. N'oubliez pas le SEO et les redirections

Les moteurs de recherche ont déjà indexé les anciennes pages Docusaurus de HagiCode. Si vous passez directement à Astro et que la structure des URL change, cela entraînera de nombreux 404 et une chute significative de l'autorité.

Nous avons configuré des règles de redirection dans Astro :

```javascript
// astro.config.mjs
export default defineConfig({
  redirects: {
    '/docs/old-path': '/docs/new-path',
    // Mappage批量 des anciens liens vers les nouveaux liens
  }
});
```
Ou gérez-les au niveau de la configuration du serveur. Assurez-vous que les anciens liens peuvent rediriger 301 vers les nouvelles adresses, ce qui est crucial pour le SEO.

## Conclusion

Migrer de Docusaurus à Astro, pour HagiCode, n'est pas seulement une mise à niveau de framework, mais aussi une pratique de la philosophie "performance d'abord".

**Ce que nous avons gagné :**
*   **Scores Lighthouse extrêmes** : Après la migration, les scores de performance du site HagiCode atteignent facilement près de 100.
*   **Vitesse de construction plus rapide** : La construction incrémentale d'Astro a réduit de moitié le temps de publication de nos mises à jour de documentation.
*   **Flexibilité préservée** : Grâce à l'architecture Islands, nous n'avons sacrifié aucune fonctionnalité interactive et pouvons toujours utiliser React là où c'est nécessaire.

Si vous maintenez également un site de type documentation et que vous souffrez du volume de l'empaquetage ou de la vitesse de chargement, essayez Astro. Bien que le processus de migration nécessite quelques "chirurgies" (comme changer le nom de PCode en HagiCode, extraire les composants un par un), cela en vaut la peine pour une expérience utilisateur fluide.

Le système de construction présenté dans cet article est précisément la solution que nous avons affinée en rencontrant et en optimisant les problèmes réels lors du développement de HagiCode. Si vous trouvez cette solution précieuse, cela montre que nos capacités d'ingénierie sont plutôt bonnes - alors HagiCode lui-même mérite également une attention.
