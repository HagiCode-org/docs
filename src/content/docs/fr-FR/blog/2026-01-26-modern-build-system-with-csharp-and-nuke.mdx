## Adieu l'enfer des scripts : pourquoi nous avons choisi C# pour un système de build moderne

> Découvrez comment le projet HagiCode utilise Nuke pour construire un workflow de build automatisé sûr au niveau des types, multiplateforme et hautement extensible, résolvant complètement les problèmes de maintenance des scripts de build traditionnels.

## Contexte

Dans le long voyage du développement logiciel, le mot « build » suscite souvent des sentiments mitigés. On l'aime, car en un clic, le code devient un produit — c'est le moment le plus fascinant pour un programmeur. On le déteste, car maintenir cette pile de scripts de build désordonnés est un véritable cauchemar.

Dans beaucoup de projets, nous sommes habitués à écrire des scripts en Python, ou à utiliser des fichiers de configuration XML. Mais avec l'augmentation de la complexité des projets, surtout ceux comme HagiCode qui impliquent le développement frontend et backend, multiplateforme et multilingue, les méthodes de build traditionnelles commencent à montrer leurs limites. Logique de scripts dispersée, absence de vérification de types, support IDE faible... ces problèmes sont autant de petits pièges qui font trébucher l'équipe de développement.

Pour résoudre ces problèmes, dans le projet HagiCode, nous avons décidé d'introduire **Nuke** — un système de build moderne basé sur C#. Ce n'est pas seulement un outil, c'est plutôt une réflexion nouvelle sur le processus de build. Aujourd'hui, nous allons parler de pourquoi nous l'avons choisi, et comment il a fait « décoller » notre expérience de développement.

## Analyse approfondie : pourquoi Nuke ?

Vous vous dites peut-être : « Tiens, il y a tellement de systèmes de build, comme Make, Gradle, pourquoi ne pas utiliser directement des scripts Shell ? Pourquoi s'embêter avec du C# ? »

C'est en fait une bonne question. Le charme de Nuke réside dans le fait qu'il apporte les caractéristiques du langage de programmation que nous connaissons le mieux dans le monde des scripts de build.

### 1. Modulariser le processus de build : l'art des Targets

La philosophie de conception de Nuke est très claire : **tout est une cible**.

Dans les scripts traditionnels, on peut écrire des centaines de lignes de code s'exécutant linéairement, avec une logique complexe. Dans Nuke, nous décomposons le processus de build en `Target` (cibles) indépendants. Chaque cible ne fait qu'une seule chose, par exemple :

- `Clean` : nettoyer le répertoire de sortie
- `Restore` : restaurer les dépendances
- `Compile` : compiler le code
- `Test` : exécuter les tests unitaires

Cette conception est très conforme au principe de responsabilité unique. Comme des briques de construction, nous pouvons combiner librement ces Targets. Plus important encore, Nuke nous permet de définir les dépendances entre les Targets. Par exemple, si vous voulez `Test`, le système vérifie automatiquement si vous avez d'abord exécuté `Compile` ; pour `Compile`, il faut d'abord `Restore`.

Ce graphe de dépendances non seulement clarifie la logique, mais améliore aussi considérablement l'efficacité d'exécution, Nuke analysant automatiquement le chemin d'exécution optimal.

### 2. Sécurité de type : adieu aux cauchemars de fautes de frappe

Ceux qui ont écrit des scripts de build en Python ont sûrement rencontré cette situation embarrassante : le script tourne pendant cinq minutes, puis se bloque avec une erreur disant que `Confi.guration` est mal orthographié.

Le plus grand avantage d'écrire des scripts de build en C# est la **sécurité de type**. Cela signifie :

- **Vérification à la compilation** : pendant que vous codez, l'IDE vous indique les erreurs, sans avoir à attendre l'exécution.
- **Refactoring sans souci** : si vous voulez changer un nom de variable ou de méthode, le refactoring de l'EDI s'en charge en un clic.
- **Complétion intelligente** : IntelliSense complète automatiquement le code, sans avoir à consulter la documentation pour mémoriser des API obscures.

### 3. Multiplateforme : une expérience de build unifiée

Autrefois, on écrivait des `.bat` sous Windows et des `.sh` sous Linux, et pour assurer la compatibilité des deux, il fallait encore écrire un script Python. Maintenant, partout où .NET (5+) peut tourner, Nuke peut tourner.

Cela signifie que quel que soit le système d'exploitation de l'équipe — Windows, Linux ou macOS — tout le monde exécute la même logique. Cela élimine considérablement les problèmes de différences d'environnement de type « ça marche sur ma machine ».

### 4. Gestion des paramètres et de la configuration

Nuke offre un mécanisme très élégant d'analyse des paramètres. Vous n'avez pas besoin d'analyser manuellement `string[] args` : il suffit de définir une propriété avec l'attribut `[Parameter]`, et Nuke gère automatiquement le mappage des arguments de ligne de commande et des fichiers de configuration.

```csharp
[Parameter("Configuration to build - Default is 'Debug'")]
readonly Configuration BuildConfiguration = IsLocalBuild ? Configuration.Debug : Configuration.Release;

Target Compile => _ => _
    .DependsOn(Restore)
    .Executes(() =>
    {
        DotNetBuild(s => s
            .SetConfiguration(BuildConfiguration)
            .SetProjectFile(SolutionFile));
    });
```

## Guide pratique : comment mettre en œuvre dans un projet

Voyons comment nous avons mis en œuvre cette solution dans le projet HagiCode.

### 1. Planification de la structure du projet

Nous avons placé tous les fichiers de build liés à Nuke dans le dossier `nukeBuild/`.

Les avantages :
- Le répertoire racine du projet reste propre.
- La logique de build est cohérente et facile à gérer.
- Les nouveaux membres peuvent voir d'un coup d'œil que c'est la logique de build.

### 2. Conception d'une chaîne de dépendances de Targets claire

Nous avons suivi un principe : **atomisation + flux de dépendances**.

Chaque Target doit être suffisamment petit et ne faire qu'une seule chose. Le flux de dépendances recommandé :

`Clean` -> `Restore` -> `Compile` -> `Test` -> `Pack`

Nuke permet d'exécuter directement `nuke Test`, et il gérera automatiquement les étapes préalables de Restore et Compile.

### 3. Gestion des erreurs et journaux de build

Dans Nuke, nous pouvons capturer et signaler les erreurs très précisément grâce au mécanisme de gestion des exceptions de C#.

```csharp
Target Publish => _ => _
    .DependsOn(Test)
    .Executes(() =>
    {
        try
        {
            DotNetNuGetPush(s => s
                .SetTargetPath(ArtifactPath)
                .SetSource("https://api.nuget.org/v3/index.json")
                .SetApiKey(ApiKey));
        }
        catch (Exception ex)
        {
            Log.Error($"La publication a échoué, vérifiez la clé : {ex.Message}");
            throw;
        }
    });
```
