## Créer une équipe d'aventuriers IA : Configuration pratique de la collaboration multi-Agent chez HagiCode

> Dans le développement logiciel moderne, un seul IA Agent ne suffit plus pour répondre aux besoins complexes. Comment faire collaborer plusieurs assistants IA de différentes entreprises sur le même projet ? Cet article partagera la solution de configuration multi-Agent explorée par le projet HagiCode dans son développement pratique.

## Contexte

Beaucoup de développeurs ont probablement vécu cette expérience : un projet intègre un assistant IA pour aider à la programmation, et l'efficacité augmente effectivement. Mais à mesure que les besoins deviennent plus complexes, un IA Agent ne suffit plus - vous voulez qu'il traite simultanément plusieurs tâches comme la revue de code, la génération de documentation, les tests unitaires, et le résultat est souvent qu'il sacrifie une chose pour une autre, avec une qualité de sortie inégale.

Ce qui est encore plus frustrant, c'est que lorsque vous essayez d'introduire plusieurs assistants IA, les problèmes deviennent plus complexes. Chaque Agent a sa propre méthode de configuration, son interface API et sa logique d'exécution, et peuvent même entrer en conflit les uns avec les autres. C'est comme une équipe de football où chaque joueur est très fort, mais personne ne sait comment collaborer, résultant en un jeu chaotique.

Le projet HagiCode a rencontré le même problème lors de son développement. En tant que projet complexe impliquant une extension VSCode frontend, un service IA backend, et un client desktop multi-plateforme, nous devions dans la **version de 2026-03** connecter simultanément plusieurs assistants IA de différentes entreprises : Claude Code, Codex, CodeBuddy, iFlow, et autres. Comment faire en sorte qu'ils coexistent harmonieusement dans le même projet et exploitent leurs forces respectives est devenu un problème clé à résoudre.

Après tout, personne ne veut traiter quotidiennement avec une bande d'IA qui se battent entre elles.

La solution partagée dans cet article est précisément la pratique de configuration multi-Agent que le projet HagiCode a affinée à travers des essais et erreurs réels. Si vous êtes également préoccupé par la collaboration multi-assistants IA, cet article devrait vous donner quelques inspirations. Peut-être, après tout chaque situation est différente.

## À propos de HagiCode

[HagiCode](https://hagicode.com) est un projet d'assistant de code IA, adoptant le mode "équipe d'aventuriers" où plusieurs moteurs IA travaillent en collaboration. Adresse du projet : [github.com/HagiCode-org/site](https://github.com/HagiCode-org/site).

La solution de configuration multi-Agent partagée dans cet article est précisément l'une des technologies clés permettant à HagiCode de maintenir un développement efficace dans des projets complexes. Rien de spécial, juste transformer un groupe d'IA en une équipe d'aventuriers capable de collaborer.

## Conception de l'architecture multi-Agent de HagiCode

### Du "combat solo" au "travail d'équipe"

Dans les premières phases du projet HagiCode, nous avons aussi essayé d'utiliser un seul IA Agent pour traiter toutes les tâches. Très vite, nous avons découvert que cette approche présentait des goulots d'étranglement évidents : différentes tâches nécessitent différentes capacités, certaines nécessitant une meilleure compréhension du contexte, d'autres des modifications de code plus précises. Un Agent difficilement exceller dans tous les aspects.

Cela nous a fait réaliser que nous devions faire travailler plusieurs Agents en collaboration. Mais le problème était : comment faire coexister pacifiquement des produits IA de différentes entreprises dans le même projet ? Nous devions résoudre plusieurs problèmes fondamentaux :

1. **Complexité de la gestion de configuration** : Chaque Agent a différentes méthodes de configuration, interfaces API et modes d'exécution
2. **Unification du protocole de communication** : Nécessite une méthode standardisée pour l'échange de données entre différents Agents
3. **Coordination de la répartition des tâches** : Comment répartir raisonnablement les tâches pour que chaque Agent exploite ses forces

Avec ces problèmes à l'esprit, nous avons commencé à concevoir l'architecture multi-Agent de HagiCode. Ce n'était pas si compliqué, nous avons juste compris les choses.

### Aperçu de l'architecture globale

Après plusieurs itérations, l'architecture que nous avons finalement déterminée est la suivante :

```
┌─────────────────────────────────────────────────────────────────┐
│                    AIProviderFactory                             │
│  (Pattern Factory pour gérer uniformément tous les AI Provider)  │
├─────────────────────────────────────────────────────────────────┤
│  ClaudeCodeCli  │  CodexCli  │  CodebuddyCli  │  IFlowCli    │
│  (Anthropic)   │  (OpenAI)  │  (智谱 GLM)     │  (智谱)      │
└─────────────────────────────────────────────────────────────────┘
```

L'idée centrale est : à travers une interface Provider unifiée, différents IA Agents peuvent être gérés par le même ensemble de code. Simultanément, l'utilisation du pattern Factory pour créer et configurer dynamiquement ces Providers assure l'extensibilité et la flexibilité du système.

C'est comme la division du travail dans la vie, chacun a son rôle, seulement ici nous avons transformé cette division en architecture de code.

### Types d'Agent et division des responsabilités

Basé sur l'expérience d'utilisation réelle du projet HagiCode, nous avons assigné différentes responsabilités à chaque Agent :

| Agent | Fournisseur | Modèle | Usage principal |
|-------|--------|------|---------|
| ClaudeCodeCli | Anthropic | glm-5-turbo | Générer des solutions techniques et des propositions |
| CodexCli | OpenAI/Zed | gpt-5.4 | Exécuter des modifications de code précises |
| CodebuddyCli | 智谱 | glm-4.7 | Optimiser les descriptions de propositions et la documentation |
| IFlowCli | 智谱 | glm-4.7 | Archiver les propositions et les enregistrements historiques (configuration de l'époque ; actuellement compatibilité historique uniquement) |
| OpenCodeCli | - | - | Édition de code générale |
| GitHubCopilot | Microsoft | - | Assistance à la programmation et complétion de code |

La logique derrière cette division est : chaque Agent a son domaine d'expertise. Claude Code excelle dans la compréhension et l'analyse de besoins complexes, donc il est responsable de la conception de solutions initiale ; Codex est plus précis dans les modifications de code, convenant pour traiter des tâches d'implémentation spécifiques ; CodeBuddy offre un bon rapport qualité-prix, parfait pour optimiser la documentation.

Après tout, ce qui vous convient est le meilleur, tous les chemins mènent à Rome, certains chemins sont plus faciles, d'autres un peu plus détournés, c'est tout.

## Mécanisme de configuration central

### Conception de l'interface Provider unifiée

Pour permettre à différents IA Agents d'être gérés uniformément, il faut d'abord définir une interface unifiée. HagiCode a défini cette interface :

```csharp
public interface IAIProvider
{
    // Interface Provider unifiée
    Task<IAIProvider?> GetProviderAsync(AIProviderType providerType);
    Task<IAIProvider?> GetProviderAsync(string providerName, CancellationToken cancellationToken);
}
```

Cette interface semble simple, mais elle est la pierre angulaire de tout le système multi-Agent. À travers cette interface unifiée, nous pouvons ignorer quel produit IA de quelle entreprise est dans la couche inférieure, et tous sont appelés de la même manière.

En fait, cela simplifie juste les choses complexes, après tout la simplicité est la beauté.

### Implémentation du pattern Factory Provider

Avec l'interface unifiée, voici comment créer ces instances Provider. HagiCode utilise le pattern Factory :

```csharp
private IAIProvider? CreateProvider(AIProviderType providerType, ProviderConfiguration config)
{
    return providerType switch
    {
        AIProviderType.ClaudeCodeCli =>
            ActivatorUtilities.CreateInstance<ClaudeCodeCliProvider>(_serviceProvider, Options.Create(config)),
        AIProviderType.CodebuddyCli =>
            ActivatorUtilities.CreateInstance<CodebuddyCliProvider>(_serviceProvider, Options.Create(config)),
        AIProviderType.CodexCli =>
            ActivatorUtilities.CreateInstance<CodexCliProvider>(_serviceProvider, Options.Create(config)),
        AIProviderType.IFlowCli =>
            ActivatorUtilities.CreateInstance<IFlowCliProvider>(_serviceProvider, Options.Create(config)),
        _ => null
    };
}
```

Ici, nous utilisons `ActivatorUtilities.CreateInstance` de l'injection de dépendances, qui peut créer dynamiquement des instances Provider au runtime et injecter automatiquement les dépendances. L'avantage de cette conception est : lors de l'ajout d'un nouveau type d'Agent, il suffit d'ajouter la classe Provider correspondante, puis d'ajouter une branche case dans la méthode Factory, sans avoir besoin de modifier le code existant.

C'est tout, après tout qui veut modifier un tas d'ancien code à chaque fois qu'on ajoute une nouvelle fonctionnalité.

### Analyse de configuration dynamique

Pour rendre la configuration plus flexible, nous avons également implémenté un mécanisme de mappage de types :

```csharp
public static AIProviderTypeExtensions
{
    private static readonly Dictionary<string, AIProviderType> _typeMap = new(
        StringComparer.OrdinalIgnoreCase)
    {
        ["ClaudeCodeCli"] = AIProviderType.ClaudeCodeCli,
        ["CodebuddyCli"] = AIProviderType.CodebuddyCli,
        ["CodexCli"] = AIProviderType.CodexCli,
        ["IFlowCli"] = AIProviderType.IFlowCli,
        // ...plus de mappages de types
    };
}
```

Le rôle de cette table de mappage est de convertir les noms Provider sous forme de chaînes en types énumération. Ainsi, les fichiers de configuration peuvent utiliser des noms de chaînes intuitifs, tandis que le code interne utilise des énumérations de type sûr pour le traitement.

Après tout pour la configuration, plus c'est intuitif mieux c'est, qui veut mémoriser un tas de code complexe.

### Exemple de fichier de configuration

En utilisation réelle, il suffit de configurer dans `appsettings.json` :

```yaml
AI:
  Providers:
    Providers:
      ClaudeCodeCli:
        Enabled: true
        Model: glm-5-turbo
        WorkingDirectory: /path/to/project
      CodebuddyCli:
        Enabled: true
        Model: glm-4.7
      CodexCli:
        Enabled: true
        Model: gpt-5.4
      IFlowCli:
        Enabled: true
        Model: glm-4.7
```

Chaque Provider peut être configuré indépendamment avec des paramètres comme l'interrupteur, la version du modèle, le répertoire de travail, etc. Cette conception assure à la fois la flexibilité et facilite la gestion et la maintenance.

En fait, les fichiers de configuration sont comme les options de la vie, vous pouvez choisir d'activer ou désactiver certaines fonctions, seulement les choix dans le code sont plus facilement regrettables.

## Flux de travail des tâches de l'équipe d'aventuriers

### L'art de la division des tâches

Avec une architecture technique unifiée, voici comment faire collaborer plusieurs Agents. HagiCode a conçu un mécanisme de flux de tâches, permettant à différents Agents de traiter des tâches à différentes étapes :

```
Création de proposition (utilisateur)
    │
    ▼
[Claude Code] ──Générer proposition──▶ Document de proposition
    │                        │
    │                        ▼
    │               [Codebuddy] ──Optimiser description──▶ Proposition optimisée
    │                        │
    │                        ▼
    │               [Codex] ──Exécuter modifications──▶ Changements de code
    │                        │
    │                        ▼
    └───────────────▶ [iFlow] ──Archiver──▶ Enregistrements historiques
```

L'avantage de cette division est : chaque Agent se concentre uniquement sur les tâches qu'il maîtrise, sans avoir besoin de "tout savoir". Claude Code est responsable de générer des propositions à partir de zéro, Codebuddy de rendre les descriptions de propositions plus claires, Codex de transformer les propositions en changements de code réels, et iFlow d'archiver et sauvegarder ces changements.

En fait, c'est comme le travail d'équipe dans la vie, chacun a son rôle, et ensemble ils peuvent accomplir de grandes choses. Seulement ici les membres de l'équipe sont des IA.

### Points clés de la pratique

Dans le fonctionnement réel, nous avons résumé les points d'expérience suivants :

**1. La stratégie de sélection des Agent est très importante**

Il ne s'agit pas de distribuer les tâches au hasard, mais de les assigner en fonction des forces de chaque Agent :
- Génération de propositions : Utiliser Claude Code, car il a une meilleure capacité de compréhension du contexte
- Exécution de code : Utiliser Codex, car il est plus précis dans les modifications de code
- Optimisation de propositions : Utiliser CodeBuddy, car il offre un bon rapport qualité-prix
- Archivage et stockage : Utiliser iFlow, car il est stable et fiable

Après tout, faire faire la bonne chose à la bonne personne, c'est une vérité éternelle.

**2. L'isolement de la configuration assure la stabilité**

La configuration de chaque Agent est gérée indépendamment, prend en charge la substitution par variables d'environnement, et les répertoires de travail sont également indépendants. Ainsi, une erreur de configuration d'un Agent n'affectera pas les autres Agents.

C'est comme les limites dans la vie, chacun a son propre espace, et ne pas s'interférer permet de coexister harmonieusement.

**3. Mécanisme de gestion des erreurs**

L'échec d'un Agent ne devrait pas affecter le flux global. Nous avons implémenté une stratégie de dégradation : lorsqu'un Agent échoue, le système peut automatiquement basculer vers une solution de secours, ou directement sauter cette étape et continuer à exécuter les tâches suivantes. Simultanément, des enregistrements de journaux complets facilitent le dépannage ultérieur.

Après tout personne ne peut garantir de ne jamais faire d'erreur, la clé est comment gérer les erreurs. C'est comme la vie, on rencontrera toujours des revers, l'important est comment en sortir.

**4. Surveillance et observabilité**

À travers le protocole ACP (notre protocole de communication personnalisé, basé sur JSON-RPC 2.0), nous pouvons suivre l'état d'exécution de chaque Agent. L'isolement des sessions assure la sécurité concurrentielle, et le cache dynamique optimise les performances.

Après tout ce qu'on ne voit pas est le plus sujet à problèmes, un peu de surveillance vaut mieux que d'être dans le noir complet.

## Résultats et bénéfices réels

Après avoir adopté cette configuration de collaboration multi-Agent, l'efficacité du développement du projet HagiCode s'est nettement améliorée. Plus spécifiquement :

1. **Capacité de traitement des tâches doublée** : Auparavant, un Agent devait traiter simultanément plusieurs types de tâches, maintenant il peut les traiter en parallèle, le débit a plus que doublé
2. **Qualité de sortie plus stable** : Chaque Agent se concentre uniquement sur les tâches qu'il maîtrise, la cohérence et la qualité des résultats sont supérieures
3. **Coût de maintenance réduit** : L'interface et la gestion de configuration unifiées rendent l'ensemble du système plus facile à maintenir et étendre
4. **Ajout d'Agent simple** : Pour intégrer un nouveau produit IA, il suffit d'implémenter l'interface et d'ajouter la configuration, sans modifier la logique centrale

Cette solution résout non seulement les problèmes propres à HagiCode, mais prouve également que la collaboration multi-Agent est vraiment un choix architectural viable.

En fait, les effets sont assez évidents, seulement le processus était un peu tortueux.

## Résumé

Cet article a partagé l'expérience pratique du projet HagiCode en matière de configuration de collaboration multi-Agent. Les points clés incluent :

1. **Interface standardisée** : À travers `IAIProvider` unifier le comportement de différents Agents, permettant au code d'ignorer quel produit de quelle entreprise est dans la couche inférieure
2. **Pattern Factory** : Utiliser `ActivatorUtilities.CreateInstance` pour créer dynamiquement des instances Provider, prenant en charge la configuration runtime et l'injection de dépendances
3. **Protocole unifié** : Le protocole ACP réalise une communication standardisée entre Agents, un mécanisme de communication bidirectionnel basé sur JSON-RPC 2.0
4. **Division du flux de tâches** : Répartir raisonnablement les tâches entre différents Agents, leur permettant d'exploiter leurs forces, plutôt que d'essayer de faire faire tout à un seul Agent

Cette conception résout non seulement le problème des "multi-Agent qui se battent", mais réalise également l'automatisation et la spécialisation du flux de développement à travers le mécanisme de flux de tâches de l'équipe d'aventuriers.
