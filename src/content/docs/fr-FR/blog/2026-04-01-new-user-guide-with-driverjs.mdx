## Implémentation élégante du guide pour les nouveaux utilisateurs dans les projets React : La pratique driver.js de HagiCode

> Lorsque les utilisateurs ouvrent votre produit pour la première fois, savent-ils vraiment par où commencer ? Cet article explore notre expérience avec driver.js pour le guide des nouveaux utilisateurs dans le projet HagiCode - une approche que nous espérons inspirante.

## Contexte

Avez-vous déjà rencontré cette situation : un nouvel utilisateur s'inscrit à votre produit, ouvre la page et paraît perdu, regardant partout sans savoir où cliquer ni que faire. En tant que développeurs, nous pensons souvent que les utilisateurs "exploreront par eux-mêmes", après tout, la curiosité humaine est infinie. Mais la réalité est différente - la plupart des utilisateurs partiront discrètement en quelques minutes parce qu'ils ne trouvent pas le point d'entrée, comme une histoire qui commence soudainement et finit naturellement.

Le guide pour les nouveaux utilisateurs est un moyen important de résoudre ce problème, mais sa mise en œuvre n'est pas si simple. Un bon système de guide doit :

- Être capable de localiser précisément les éléments de la page et de les mettre en évidence
- Prendre en charge les processus de guide multi-étapes
- Être capable de mémoriser les choix de l'utilisateur (terminé/ignoré)
- Ne pas affecter les performances de la page et les interactions normales
- Avoir une structure de code claire, facile à maintenir

Lors du développement de HagiCode, nous avons rencontré le même défi. HagiCode est un projet d'assistant de code AI, le workflow principal est "utilisateur crée un proposal → AI génère un plan → utilisateur révise → AI exécute" - un processus OpenSpec. Pour les utilisateurs qui découvrent ce concept pour la première fois, ce processus est entièrement nouveau, il doit y avoir un bon guide pour les aider à démarrer rapidement. Après tout, les nouvelles choses ont toujours besoin d'un peu de temps.

## À propos de HagiCode

La solution présentée dans cet article provient de notre expérience pratique dans le projet [HagiCode](https://hagicode.com). HagiCode est un assistant de code AI basé sur Claude qui aide les développeurs à accomplir les tâches de code plus efficacement via le workflow OpenSpec. Vous pouvez consulter notre code open source sur [GitHub](https://github.com/HagiCode-org/site).

## Pourquoi choisir driver.js

Lors de la phase de sélection technique, nous avons évalué plusieurs bibliothèques de guide principales, comment dire, chacune a ses propres caractéristiques :

- **Intro.js** : Puissant mais volumineux, la personnalisation des styles est relativement complexe
- **Shepherd.js** : Très bonne conception de l'API, mais un peu "lourd" pour notre scénario
- **driver.js** : Léger, concis, API intuitive, et prend en charge l'écosystème React

Finalement, nous avons choisi driver.js, en fait, pas de raison particulière, principalement basé sur les considérations suivantes :

1. **Léger** : La bibliothèque principale est petite, n'augmente pas significativement la taille du bundle
2. **API concise** : Les options de configuration sont claires et intuitives, facile à prendre en main
3. **Flexibilité** : Prend en charge le positionnement, les styles et les comportements d'interaction personnalisés
4. **Import dynamique** : Peut être chargé à la demande, n'affecte pas les performances de l'écran d'accueil

Le choix technique, en fait, il n'y a pas de meilleur, seulement le plus approprié.

## Implémentation technique

### Configuration principale

La configuration de driver.js est très intuitive, voici la configuration principale du projet HagiCode :

```typescript
import { driver } from 'driver.js';
import 'driver.js/dist/driver.css';

const newConversationDriver = driver({
  allowClose: true,           // Permettre à l'utilisateur de fermer le guide
  animate: true,              // Activer les effets d'animation
  overlayClickBehavior: 'close', // Cliquer sur la superposition pour fermer le guide
  disableActiveInteraction: false, // Garder les éléments interactifs
  showProgress: false,        // Ne pas afficher la barre de progression (nous avons une gestion de progression personnalisée)
  steps: guideSteps           // Tableau des étapes du guide
});
```

Les considérations derrière ces configurations sont :

- `allowClose: true` - Respecter le choix de l'utilisateur, ne pas forcer l'achèvement du guide, après tout, forcer ne donne rien de bon
- `disableActiveInteraction: false` - Certaines étapes nécessitent une action réelle de l'utilisateur (comme saisir du texte), donc l'interaction ne peut pas être désactivée
- `overlayClickBehavior: 'close'` - Donner à l'utilisateur un moyen de sortie rapide

### Gestion de l'état

La persistance de l'état du guide est essentielle - nous ne voulons pas répéter le guide à chaque actualisation de la page, c'est assez ennuyeux. HagiCode utilise localStorage pour gérer l'état du guide :

```typescript
export type GuideState = 'pending' | 'dismissed' | 'completed';

export interface UserGuideState {
  session: GuideState;
  detailGuides: Record<string, GuideState>;
}

// Lire l'état
export const getUserGuideState = (): UserGuideState => {
  const state = localStorage.getItem('userGuideState');
  return state ? JSON.parse(state) : { session: 'pending', detailGuides: {} };
};

// Mettre à jour l'état
export const setUserGuideState = (state: UserGuideState) => {
  localStorage.setItem('userGuideState', JSON.stringify(state));
};
```

Nous avons défini trois états :

- `pending` : Le guide est en cours, l'utilisateur n'a pas encore terminé ou ignoré
- `dismissed` : L'utilisateur a fermé activement le guide
- `completed` : L'utilisateur a terminé toutes les étapes

Pour le guide de la page de détails du proposal, nous prenons également en charge un suivi d'état plus granulaire (via le dictionnaire `detailGuides`), car un proposal peut passer par plusieurs phases (brouillon, révision, exécution terminée), chaque phase nécessitant un guide différent. Après tout, l'état des choses est toujours en évolution.

### Localisation des éléments cibles

driver.js utilise des sélecteurs CSS pour localiser les éléments cibles. HagiCode adopte une convention : utiliser l'attribut personnalisé `data-guide` pour marquer les cibles du guide :

```typescript
const steps = [
  {
    element: '[data-guide="launch"]',
    popover: {
      title: 'Démarrer une nouvelle conversation',
      description: 'Cliquez ici pour créer une nouvelle session de conversation...'
    }
  }
];
```

Utilisé dans le composant comme ceci :

```typescript
<button data-guide="launch" onClick={handleLaunch}>
  Nouvelle conversation
</button>
```

Les avantages de cette approche sont :

- Éviter les conflits avec les noms de classes de style métier
- Sémantique claire, on voit immédiatement que cet élément est lié au guide
- Facilite la gestion et la maintenance unifiées

### Optimisation de l'import dynamique

Parce que la fonctionnalité de guide n'est nécessaire que dans des scénarios spécifiques (comme la première visite d'un nouvel utilisateur), nous utilisons l'import dynamique pour optimiser les performances de chargement initial :

```typescript
const initNewUserGuide = async () => {
  // Import dynamique de driver.js
  const { driver } = await import('driver.js');
  await import('driver.js/dist/driver.css');

  // Initialiser le guide
  const newConversationDriver = driver({
    // ...configuration
  });

  newConversationDriver.drive();
};
```

Ainsi, driver.js et ses fichiers de style ne seront chargés que lorsque nécessaire, n'affectant pas les performances de l'écran d'accueil. Après tout, qui veut payer le prix d'attendre pour quelque chose qui n'est pas encore utilisé ?

## Conception du processus de guide

HagiCode a implémenté deux chemins de guide, couvrant les scénarios d'utilisation principaux des utilisateurs.

### Guide de conversation (10 étapes)

Ce guide aide les utilisateurs à accomplir l'ensemble du processus, de la création de conversation à la soumission du premier proposal complet :

1. **launch** - Démarrer le guide, présenter le bouton "Nouvelle conversation"
2. **compose** - Guider l'utilisateur à saisir une demande dans la zone de saisie
3. **send** - Guider à cliquer sur le bouton d'envoi
4. **proposal-launch-readme** - Guider à créer un proposal README
5. **proposal-compose-readme** - Guider à éditer le contenu de la demande README
6. **proposal-submit-readme** - Guider à soumettre le proposal README
7. **proposal-launch-agents** - Guider à créer un proposal AGENTS.md
8. **proposal-compose-agents** - Guider à éditer la demande AGENTS.md
9. **proposal-submit-agents** - Guider à soumettre le proposal AGENTS.md
10. **proposal-wait** - Expliquer que l'AI est en train de traiter, veuillez patienter

L'idée de conception de ce guide est : à travers deux tâches de création de proposal réelles (README et AGENTS.md), laisser les utilisateurs expérimenter personnellement le workflow principal de HagiCode. Après tout, ce qu'on apprend sur le papier reste superficiel, il faut le pratiquer pour vraiment le comprendre.

### Guide de détails du proposal (3 étapes)

Lorsque l'utilisateur entre dans la page de détails du proposal, le guide correspondant est déclenché en fonction de l'état actuel du proposal :

1. **drafting** (Phase de brouillon) - Guider l'utilisateur à consulter le plan généré par l'AI
2. **reviewing** (Phase de révision) - Guider l'utilisateur à exécuter le plan
3. **executionCompleted** (Phase terminée) - Guider l'utilisateur à archiver le plan

La particularité de ce guide est qu'il est piloté par l'état - détermine dynamiquement quelle étape du guide afficher en fonction de l'état réel du proposal. Les choses changent toujours, le guide devrait aussi changer.

### Mécanisme de réessai de rendu des éléments

Dans les applications React, les éléments cibles du guide peuvent ne pas encore avoir fini de s'afficher (par exemple, en attendant le chargement de données asynchrones). Pour gérer cette situation, HagiCode a implémenté un mécanisme de réessai :

```typescript
const waitForElement = (selector: string, maxRetries = 10, interval = 100) => {
  let retries = 0;
  return new Promise<HTMLElement>((resolve, reject) => {
    const checkElement = () => {
      const element = document.querySelector(selector) as HTMLElement;
      if (element) {
        resolve(element);
      } else if (retries < maxRetries) {
        retries++;
        setTimeout(checkElement, interval);
      } else {
        reject(new Error(`Element not found: ${selector}`));
      }
    };
    checkElement();
  });
};
```

Appelez cette fonction avant d'initialiser le guide pour vous assurer que l'élément cible existe déjà. Parfois, attendre un peu plus en vaut la peine.

## Résumé des meilleures pratiques

Basé sur l'expérience pratique de HagiCode, voici quelques meilleures pratiques clés à partager :

### 1. Le guide devrait être "escapable"

Ne forcez pas les utilisateurs à terminer le guide. Certains utilisateurs sont de type explorateur, ils préfèrent explorer par eux-mêmes. Fournissez un bouton "Ignorer" clair et mémorisez le choix de l'utilisateur, ne les dérangez plus la prochaine fois. Après tout, les belles choses ou les belles personnes, on n'a pas besoin de les posséder, tant qu'elles restent belles, il suffit de bien admirer leur beauté.

### 2. Le contenu du guide doit être concis et percutant

Chaque étape du guide devrait se concentrer sur un seul objectif :

- **Titre** : Court et clair, pas plus de 10 caractères
- **Description** : Direct au point, dire à l'utilisateur "c'est quoi" et "pourquoi l'utiliser"

Évitez les explications longues - l'attention des utilisateurs pendant la phase de guide est très limitée. Si on dit trop, personne ne voudra lire.

### 3. Les sélecteurs doivent être stables

Utilisez une méthode de marquage d'éléments stable, qui ne change pas fréquemment. L'attribut personnalisé `data-guide` est un bon choix, évitez de dépendre des noms de classe ou de la structure DOM, car ceux-ci changent facilement lors des refactorisations. Le code change toujours, mais certaines choses devraient rester stables autant que possible.

### 4. Testez votre guide

HagiCode a écrit des cas de test complets pour la fonctionnalité de guide :

```typescript
describe('NewUserConversationGuide', () => {
  it('devrait correctement initialiser l\'état du guide', () => {
    const state = getUserGuideState();
    expect(state.session).toBe('pending');
  });

  it('devrait correctement mettre à jour l\'état du guide', () => {
    setUserGuideState({ session: 'completed', detailGuides: {} });
    const state = getUserGuideState();
    expect(state.session).toBe('completed');
  });
});
```

Les tests peuvent garantir que la fonctionnalité de guide ne sera pas accidentellement cassée lors de la refactorisation du code. Après tout, personne ne veut casser les fonctionnalités précédentes en modifiant un peu de code.

### 5. Optimisation des performances

- Utilisez l'import dynamique pour charger la bibliothèque de guide en différé
- Évitez d'initialiser la logique du guide après que l'utilisateur a terminé le guide
- Considérez l'impact sur les performances des animations du guide, les animations peuvent être désactivées sur les appareils bas de gamme

La performance, c'est comme la vie, il faut économiser là où il faut.

## Conclusion

Le guide pour les nouveaux utilisateurs est un élément important pour améliorer l'expérience utilisateur du produit. Dans le projet HagiCode, nous avons utilisé driver.js pour construire un système de guide complet, couvrant l'ensemble du workflow de la création de conversation à l'exécution du proposal.

À travers le partage de cet article, nous espérons transmettre les points de vue clés suivants :

1. **Le choix technique doit correspondre aux besoins** : driver.js n'est pas le plus puissant, mais pour nous, c'est le plus approprié
2. **La gestion de l'état est essentielle** : Utilisez localStorage pour rendre persistant l'état du guide, évitez de déranger répétitivement les utilisateurs
3. **La conception du guide doit être focalisée** : Chaque étape résout un problème, ne soyez pas gourmand
4. **La structure du code doit être claire** : Séparez la configuration du guide, la gestion de l'état et la logique UI, facilitant la maintenance

Si vous ajoutez une fonctionnalité de guide pour les nouveaux utilisateurs à votre propre projet, nous espérons que l'expérience pratique de cet article pourra vous aider. En fait, la technologie, ce n'est pas mystérieux, essayez plus, résumez plus, et ça ira progressivement mieux......
