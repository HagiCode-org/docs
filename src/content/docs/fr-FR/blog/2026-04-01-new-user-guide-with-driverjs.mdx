## Construire un onboarding élégant pour les nouveaux utilisateurs dans les projets React : la pratique de driver.js chez HagiCode

> Quand les utilisateurs ouvrent votre produit pour la première fois, savent-ils vraiment par où commencer ? Dans cet article, je veux parler un peu de la façon dont nous avons utilisé driver.js pour l'onboarding des nouveaux utilisateurs dans le projet HagiCode. Considérez cela comme un petit exemple pratique pour lancer la discussion.

## Contexte

Vous êtes-vous déjà trouvé dans cette situation ? Un nouvel utilisateur s'inscrit à votre produit, ouvre la page, et a immédiatement l'air perdu. Il regarde autour de lui, sans savoir sur quoi cliquer ou quoi faire ensuite. En tant que développeurs, nous supposons souvent que les utilisateurs vont simplement « explorer par eux-mêmes » car, après tout, la curiosité humaine est sans limites. Mais la réalité est différente : la plupart des utilisateurs quittent silencieusement en quelques minutes s'ils ne trouvent pas le bon point d'entrée, comme si l'histoire commençait soudainement et se terminait tout aussi naturellement.

L'onboarding des nouveaux utilisateurs est un moyen important de résoudre ce problème, mais le construire correctement n'est pas si simple. Un bon système d'onboarding doit :

- Localiser précisément les éléments de la page et les mettre en évidence
- Prendre en charge les flux d'onboarding en plusieurs étapes
- Se souvenir du choix de l'utilisateur (terminé ou ignoré)
- Éviter d'affecter les performances de la page et l'interaction normale
- Garder la structure du code claire et facile à maintenir

Lors du développement de HagiCode, nous avons rencontré le même défi. HagiCode est un assistant de codage IA, et son flux de travail principal est un workflow OpenSpec qui ressemble à ceci : « l'utilisateur crée une Proposal -> l'IA génère un plan -> l'utilisateur le révise -> l'IA l'exécute. » Pour les utilisateurs qui découvrent ce concept pour la première fois, cette séquence de flux de travail est complètement nouvelle. S'il n'y a pas de guidage, ils se retrouvent face à l'écran sans savoir quoi faire.

C'est pourquoi nous avons introduit driver.js comme solution d'onboarding.

## Pourquoi driver.js ?

Sur le marché, il existe de nombreuses bibliothèques de guidage (intro.js, shepherd.js, react-joyride, etc.), mais nous avons choisi driver.js pour plusieurs raisons :

1. **Léger** : le bundle est petit, avec un impact minimal sur les performances.
2. **Sans dépendances de framework** : fonctionne avec React, Vue, ou du JavaScript pur.
3. **API intuitive** : configuration simple, courbe d'apprentissage douce.
4. **Support du mode sombre** : essentiel pour HagiCode qui a un thème sombre par défaut.
5. **Activement maintenu** : la communauté est active et les bugs sont corrigés rapidement.

## Mise en œuvre

La mise en œuvre suit un modèle en trois étapes :

1. **Définir les étapes du guide** : identifier les éléments clés de l'interface et créer des étapes de guidage correspondantes.
2. **Gérer l'état de l'onboarding** : suivre si l'utilisateur a déjà terminé ou ignoré le guide.
3. **Intégration avec le cycle de vie React** : déclencher le guide au bon moment.

## Conclusion

L'onboarding des nouveaux utilisateurs est un investissement qui en vaut la peine. Dans HagiCode, l'utilisation de driver.js nous a permis de construire un système de guidage élégant avec un effort minimal. Si vous développez également un produit avec un flux de travail complexe, l'onboarding est un domaine qui mérite votre attention.
