## Ajouter le support de déploiement automatique GitHub Pages à HagiCode

> Ce projet avait initialement le nom de code PCode, et a maintenant été officiellement renommé HagiCode. Cet article documente l'ajout de capacités de déploiement automatisé de sites statiques au projet, rendant la publication de contenu aussi simple que de boire un verre d'eau.

## Contexte/Introduction

Au cours du développement de HagiCode, nous avons rencontré un problème très concret : à mesure que les documents et propositions augmentaient, la gestion et l'affichage efficaces de ces contenus sont devenus une priorité urgente. Nous avons décidé d'utiliser GitHub Pages pour héberger notre site statique, mais la construction et le déploiement manuels étaient vraiment trop fastidieux — chaque modification nécessitait une construction locale, un empaquetage, puis une poussée manuelle vers la branche `gh-pages`. Non seulement cela était inefficace, mais cela entraînait également facilement des erreurs.

Pour résoudre ce problème (principalement pour être paresseux), nous avions besoin d'un processus de déploiement automatisé. Cet article documentera en détail comment ajouter le support de déploiement automatisé GitHub Actions au projet HagiCode, nous permettant de nous concentrer uniquement sur la création de contenu, le reste étant confié au processus automatisé.

## Analyse des objectifs

Avant de commencer, nous devons d'abord clarifier ce que cette tâche doit réellement accomplir. Après tout, aiguiser sa hache ne retardera pas le travail de bûcheron.

### Besoins fondamentaux

1.  **Construction automatisée** : Lorsque le code est poussé vers la branche `main`, le processus de construction se déclenche automatiquement.
2.  **Déploiement automatique** : Une fois la construction réussie, déploie automatiquement les fichiers statiques générés vers GitHub Pages.
3.  **Cohérence de l'environnement** : Garantir que l'environnement CI et l'environnement de construction local sont cohérents, évitant l'embarras de "fonctionne en local, échoue en ligne".

### Sélection technique

Considérant que HagiCode est basé sur Docusaurus (un générateur de sites statiques React très populaire), nous pouvons utiliser GitHub Actions pour atteindre cet objectif.

## Configuration du workflow GitHub Actions

GitHub Actions est le service CI/CD fourni par GitHub. En définissant des fichiers de workflow au format YAML dans le référentiel de code, nous pouvons personnaliser diverses tâches automatisées.

### Création du fichier de workflow

Nous devons créer un nouveau fichier de configuration dans le dossier `.github/workflows` à la racine du projet, par exemple appelé `deploy.yml`. Si le dossier n'existe pas, n'oubliez pas de le créer manuellement d'abord.

La logique principale de ce fichier de configuration est la suivante :

1.  **Condition de déclenchement** : Écouter les événements `push` de la branche `main`.
2.  **Environnement d'exécution** : Dernière version d'Ubuntu.
3.  **Étapes de construction** :
    -   Extraire le code
    -   Installer Node.js
    -   Installer les dépendances (`npm install`)
    -   Construire les fichiers statiques (`npm run build`)
4.  **Étapes de déploiement** : Utiliser `action-gh-pages` officiellement fourni pour pousser les artefacts de construction vers la branche `gh-pages`.

### Code de configuration clé

Voici le modèle de configuration que nous avons finalement adopté :

```yaml
name: Deploy to GitHub Pages

# Condition de déclenchement : lors d'une poussée vers la branche main
on:
  push:
    branches:
      - main
    # Peut ajouter un filtrage de chemins si nécessaire, par exemple construire uniquement lors de modifications de documents
    # paths:
    #   - 'docs/**'
    #   - 'package.json'

# Définir les permissions, important pour le déploiement vers GitHub Pages
permissions:
  contents: read
  pages: write
  id-token: write

# Contrôle de concurrence : annule les anciennes constructions de la même branche
concurrency:
  group: "pages"
  cancel-in-progress: false

jobs:
  build:
    runs-on: ubuntu-latest
    steps:
      - name: Checkout
        uses: actions/checkout@v4
        # Note : doit définir fetch-depth: 0, sinon peut entraîner un numéro de version de construction inexact
        with:
          fetch-depth: 0

      - name: Setup Node
        uses: actions/setup-node@v4
        with:
          node-version: 20 # Recommandé de rester cohérent avec l'environnement de développement local
          cache: 'npm'     # Activer le cache peut accélérer le processus de construction

      - name: Install dependencies
        run: npm ci
        # Utiliser npm ci au lieu de npm install, car c'est plus rapide et plus strict, adapté aux environnements CI

      - name: Build website
        run: npm run build
        env:
          # Si la construction de votre site nécessite des variables d'environnement, configurez-les ici
          # NODE_ENV: production
          # PUBLIC_URL: /your-repo-name
          
      - name: Upload artifact
        uses: actions/upload-pages-artifact@v3
        with:
          path: ./build # Répertoire de sortie par défaut de Docusaurus

  deploy:
    environment:
      name: github-pages
      url: ${{ steps.deployment.outputs.page_url }}
    runs-on: ubuntu-latest
    needs: build
    steps:
      - name: Deploy to GitHub Pages
        id: deployment
        uses: actions/deploy-pages@v4
```

## Problèmes rencontrés lors de la mise en œuvre

Dans la pratique, nous avons rencontré quelques problèmes. Je les partage ici dans l'espoir que vous puissiez les éviter (ou préparer à l'avance des solutions).

### 1. Problème d'autorisation GitHub Token

Au début de la configuration, le déploiement rapportait toujours une erreur 403 (Forbidden). Après avoir cherché longtemps, nous avons découvert que le `GITHUB_TOKEN` par défaut de GitHub n'avait pas la permission d'écriture pour Pages.

**Solution** : Dans `Settings` -> `Actions` -> `General` -> `Workflow permissions` du référentiel, vous devez absolument choisir **"Read and write permissions"**.

### 2. Chemin de répertoire de construction incorrect

Docusaurus place par défaut les fichiers statiques construits dans le répertoire `build`. Cependant, certains projets (par exemple Create React App utilise par défaut `build`, Vite utilise par défaut `dist`) peuvent avoir des configurations différentes. Si Actions rapporte une erreur de fichier introuvable, n'oubliez pas de vérifier la configuration du chemin de sortie dans `docusaurus.config.js`.

### 3. Problème de sous-chemin

Si votre référentiel n'est pas une page utilisateur (c'est-à-dire pas `username.github.io`), mais une page de projet (comme `username.github.io/project-name`), vous devez configurer `baseUrl`.

Dans `docusaurus.config.js` :

```javascript
module.exports = {
  // ...
  url: 'https://hagicode.com', // Votre URL Hagicode
  baseUrl: '/', // Déploiement sur le chemin racine
  // ...
};
```
Ce point est facilement négligé. Une configuration incorrecte entraînera un écran blanc lors de l'ouverture de la page, car les chemins des ressources ne peuvent pas être chargés.

## Vérification des résultats

Après avoir configuré tout et poussé le code, nous pouvons aller à l'onglet **Actions** du référentiel GitHub pour regarder.

Vous verrez un cercle jaune (le workflow est en cours d'exécution), qui devient vert signifie succès ! S'il devient rouge, cliquez pour voir les journaux, vous pouvez généralement identifier le problème (la plupart du temps c'est une faute de frappe ou une configuration de chemin incorrecte).

Une fois la construction réussie, visitez `https://<votre-nom-d-utilisateur>.github.io/<nom-du-référentiel>/` pour voir le tout nouveau site.

## Résumé

En introduisant GitHub Actions, nous avons réussi à mettre en œuvre le déploiement automatisé du site de documentation HagiCode. Cela permet non seulement de gagner du temps sur les opérations manuelles, mais plus important encore, garantit la standardisation du processus de publication. Maintenant, quel que soit le collègue qui met à jour la documentation, dès qu'il fusionne dans la branche `main`, le contenu le plus récent peut être vu en ligne en quelques minutes.

**Avantages fondamentaux** :
- **Amélioration de l'efficacité** : De "l'empaquetage manuel, du téléchargement manuel" à "le code est la publication".
- **Réduction des erreurs** : Élimine la possibilité d'erreurs opérationnelles humaines.
- **Optimisation de l'expérience** : Permet aux développeurs de se concentrer davantage sur la qualité du contenu plutôt que d'être gênés par des processus de déploiement fastidieux.

Bien que la configuration CI/CD soit un peu fastidieuse au début (surtout les divers problèmes d'autorisations et de chemins), c'est un investissement unique avec un retour à long terme énorme. Il est fortement recommandé que tous les projets de sites statiques intègrent des processus automatisés similaires.
