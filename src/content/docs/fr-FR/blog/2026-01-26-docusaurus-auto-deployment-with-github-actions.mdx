## Ajouter le support de déploiement automatique GitHub Pages à HagiCode

> Le nom de code initial de ce projet était PCode, il a été officiellement renommé HagiCode. Cet article documente comment nous avons introduit la capacité de déploiement automatisé de sites statiques, rendant la publication de contenu aussi simple que de boire de l'eau.

## Contexte / Introduction

Au cours du développement de HagiCode, nous avons rencontré un problème très concret : avec de plus en plus de documentation et de propositions, la gestion et l'affichage efficaces de ces contenus sont devenus une priorité. Nous avons décidé d'utiliser GitHub Pages pour héberger notre site statique, mais le build et le déploiement manuels étaient vraiment trop fastidieux — chaque modification nécessitait un build local, un packaging, puis une poussée manuelle vers la branche `gh-pages`. Non seulement cela était inefficace, mais cela entraînait aussi facilement des erreurs.

Pour résoudre ce problème (principalement pour pouvoir être paresseux), nous avions besoin d'un processus de déploiement automatisé. Cet article documente en détail comment nous avons ajouté le support de déploiement automatique via GitHub Actions au projet HagiCode, afin que nous puissions nous concentrer uniquement sur la création de contenu, en laissant le reste au processus automatisé.

## Analyse des objectifs

Avant de passer à l'action, nous devons clarifier ce que cette tâche doit accomplir. Après tout, bien affûter sa scie ne gâte pas le coup de bois.

### Besoins fondamentaux

1.  **Build automatisé** : lorsque le code est poussé vers la branche `main`, le processus de build se déclenche automatiquement.
2.  **Déploiement automatique** : après un build réussi, les fichiers statiques générés sont automatiquement déployés sur GitHub Pages.
3.  **Cohérence d'environnement** : s'assurer que l'environnement CI est cohérent avec l'environnement de build local, pour éviter la situation embarrassante où « ça marche en local mais pas en production ».

### Choix technique

HagiCode étant basé sur Docusaurus (un générateur de sites statiques React très populaire), nous pouvons utiliser GitHub Actions pour atteindre cet objectif.

## Configuration du workflow GitHub Actions

GitHub Actions est le service CI/CD fourni par GitHub. En définissant des fichiers de workflow au format YAML dans le dépôt de code, nous pouvons personnaliser diverses tâches automatisées.

### Création du fichier de workflow

Nous devons créer un nouveau fichier de configuration dans le dossier `.github/workflows` à la racine du projet, par exemple `deploy.yml`. Si le dossier n'existe pas, n'oubliez pas de le créer manuellement d'abord.

La logique principale de ce fichier de configuration est la suivante :

1.  **Condition de déclenchement** : écouter l'événement `push` sur la branche `main`.
2.  **Environnement d'exécution** : dernière version d'Ubuntu.
3.  **Étapes de build** :
    -   Extraction du code
    -   Installation de Node.js
    -   Installation des dépendances (`npm install`)
    -   Build des fichiers statiques (`npm run build`)
4.  **Étapes de déploiement** : utilisation de `action-gh-pages` officiel pour pousser les artefacts de build vers la branche `gh-pages`.

### Code de configuration clé

Voici le modèle de configuration que nous avons finalement adopté :

```yaml
name: Deploy to GitHub Pages

# Condition de déclenchement : lors d'une poussée vers la branche main
on:
  push:
    branches:
      - main

# Définition des permissions, important pour le déploiement sur GitHub Pages
permissions:
  contents: read
  pages: write
  id-token: write

# Contrôle de concurrence : annuler les anciens builds de la même branche
concurrency:
  group: "pages"
  cancel-in-progress: false

jobs:
  build:
    runs-on: ubuntu-latest
    steps:
      - name: Checkout
        uses: actions/checkout@v4
        with:
          fetch-depth: 0

      - name: Setup Node
        uses: actions/setup-node@v4
        with:
          node-version: 20
          cache: 'npm'

      - name: Install dependencies
        run: npm ci

      - name: Build website
        run: npm run build

      - name: Upload artifact
        uses: actions/upload-pages-artifact@v3
        with:
          path: ./build

  deploy:
    environment:
      name: github-pages
      url: ${{ steps.deployment.outputs.page_url }}
    runs-on: ubuntu-latest
    needs: build
    steps:
      - name: Deploy to GitHub Pages
        id: deployment
        uses: actions/deploy-pages@v4
```

## Pièges rencontrés lors de la mise en œuvre

Dans la pratique, nous avons rencontré quelques problèmes que nous partageons ici pour que vous puissiez les éviter.

### 1. Problème de permissions du GitHub Token

Au début de la configuration, le déploiement échouait toujours avec une erreur 403 (Forbidden). Après une longue recherche, nous avons découvert que le `GITHUB_TOKEN` par défaut de GitHub n'avait pas la permission d'écriture sur Pages.

**Solution** : dans `Settings` -> `Actions` -> `General` -> `Workflow permissions` du dépôt, veillez à sélectionner **« Read and write permissions »**.

### 2. Erreur de chemin du répertoire de build

Docusaurus place par défaut les fichiers statiques buildés dans le répertoire `build`. Mais certains projets peuvent avoir une configuration différente. Si Actions signale une erreur de fichier introuvable, vérifiez le chemin de sortie dans `docusaurus.config.js`.

### 3. Problème de sous-chemin

Si votre dépôt n'est pas une page utilisateur (c'est-à-dire pas `username.github.io`), mais une page de projet (par exemple `username.github.io/project-name`), vous devez configurer `baseUrl`.

Dans `docusaurus.config.js` :

```javascript
module.exports = {
  // ...
  url: 'https://hagicode.com',
  baseUrl: '/',
  // ...
};
```
Ce point est facilement négligé ; une configuration incorrecte entraînera un écran blanc, car les chemins des ressources ne pourront pas être chargés.

## Vérification des résultats

Après avoir configuré tout cela et poussé le code, nous pouvons aller sur l'onglet **Actions** du dépôt GitHub pour observer.

Vous verrez un cercle jaune (le workflow est en cours d'exécution) ; s'il devient vert, c'est que c'est réussi ! S'il devient rouge, cliquez dessus pour consulter les journaux — généralement vous pourrez identifier le problème.

Après un build réussi, rendez-vous sur `https://<votre-nom-d-utilisateur>.github.io/<nom-du-dépôt>/` pour voir le tout nouveau site.

## Conclusion

En introduisant GitHub Actions, nous avons réalisé avec succès le déploiement automatisé du site de documentation HagiCode. Cela permet non seulement de gagner du temps sur les opérations manuelles, mais surtout de garantir la standardisation du processus de publication.

**Bénéfices clés** :
- **Gain d'efficacité** : passer de « packaging manuel, téléchargement manuel » à « le code est la publication ».
- **Réduction des erreurs** : élimination de la possibilité d'erreurs de manipulation humaine.
- **Amélioration de l'expérience** : permettre aux développeurs de se concentrer sur la qualité du contenu plutôt que d'être gênés par des processus de déploiement fastidieux.
