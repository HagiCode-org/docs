## Modifier rapidement le code dans le navigateur : Pratiques d'intégration VSCode Web

> Après l'analyse du code par l'IA, comment ouvrir immédiatement l'éditeur dans le navigateur pour effectuer des modifications ? Cet article partage l'expérience pratique de l'intégration de code-server dans le projet HagiCode, réalisant une connexion transparente entre l'assistant IA et l'expérience d'édition de code.

## Contexte

À l'ère de la programmation assistée par IA, les développeurs ont souvent besoin de visualiser et de modifier rapidement du code. Le processus de développement traditionnel consiste à : ouvrir le projet dans un IDE de bureau, localiser le fichier, modifier, enregistrer. Mais dans certains scénarios, ce processus semble quelque peu inadéquat.

Scénario 1 : développement à distance. Lors de l'utilisation d'un assistant IA comme HagiCode, le backend peut fonctionner sur un serveur distant ou dans un conteneur, et l'accès local aux fichiers du projet n'est pas possible. Chaque fois que vous souhaitez visualiser ou modifier du code, vous devez vous connecter via SSH ou d'autres moyens, ce qui crée une expérience fragmentée. C'est comme vouloir voir quelqu'un, mais être séparé par une épaisse vitre, on peut voir mais pas toucher.

Scénario 2 : prévisualisation rapide. Après que l'assistant IA a analysé le code, l'utilisateur peut simplement vouloir parcourir rapidement un fichier ou apporter des modifications mineures. Lancer un IDE de bureau complet semble trop lourd, un éditeur léger dans le navigateur correspond mieux au besoin de "visualisation rapide". Après tout, qui voudrait déployer autant d'efforts juste pour jeter un coup d'œil ?

Scénario 3 : collaboration multi-appareils. Lorsque l'on travaille sur différents appareils, l'éditeur dans le navigateur offre un point d'accès unifié, sans avoir besoin de configurer un environnement de développement sur chaque appareil. C'est plutôt pratique, après tout, la vie est courte, pourquoi se répéter.

Pour résoudre ces points de douleur, nous avons intégré VSCode Web dans le projet HagiCode. Permettant une connexion transparente entre l'assistant IA et l'expérience d'édition de code - après que l'IA a analysé le code, les utilisateurs peuvent immédiatement ouvrir l'éditeur dans la même session de navigateur pour apporter des modifications, sans changer d'application. Cette expérience, comment dire, c'est comme quand vous en avez besoin, elle est là.

## À propos de HagiCode

La solution partagée dans cet article provient de notre expérience pratique dans le projet [HagiCode](https://hagicode.com). HagiCode est un assistant de code piloté par l'IA, visant à améliorer l'efficacité du développement par une interaction en langage naturel. Au cours du développement, nous avons découvert que les utilisateurs avaient souvent besoin de basculer rapidement entre l'analyse IA et l'édition de code, ce qui nous a poussés à explorer comment intégrer directement l'éditeur dans le navigateur.

Adresse du projet : [github.com/HagiCode-org/site](https://github.com/HagiCode-org/site)

## Choix technique : pourquoi code-server ?

Parmi les nombreuses solutions VSCode Web, nous avons choisi code-server. Ce choix repose en fait sur plusieurs considérations :

Fonctionnalités complètes. code-server est la version Web de VSCode, prenant en charge la plupart des fonctionnalités de la version de bureau, y compris le système d'extensions, les suggestions intelligentes, le débogage, etc. Cela signifie que les utilisateurs peuvent également obtenir une expérience d'édition proche de la version de bureau dans le navigateur. Après tout, qui veut faire des compromis sur les fonctionnalités ?

Déploiement flexible. code-server peut fonctionner comme un service indépendant, prend également en charge le déploiement en conteneur Docker, ce qui correspond à l'architecture de HagiCode. Notre backend est écrit en C#, le frontend est React, et communique avec le service code-server via REST API. C'est comme des blocs de construction, chaque bloc a sa place.

Authentification sécurisée. code-server intègre un mécanisme de connection-token pour empêcher l'accès non autorisé. Chaque session a un token unique, garantissant que seuls les utilisateurs autorisés peuvent accéder à l'éditeur. La sécurité, on ne réalise son importance que quand on l'a.

## Conception de l'architecture

L'intégration VSCode Web de HagiCode adopte une conception d'architecture avec séparation frontend-backend.

### Couche de service frontend

Le frontend encapsule l'interaction avec le backend via `vscodeServerService.ts` :

```typescript
// Ouvrir un projet
export async function openProjectInCodeServer(
  id: string,
  currentInterfaceLanguage?: string,
): Promise<VsCodeServerLaunchResponseDto>

// Ouvrir un vault
export async function openVaultInCodeServer(
  id: string,
  path?: string,
  currentInterfaceLanguage?: string,
): Promise<VsCodeServerLaunchResponseDto>
```

La différence entre ces deux méthodes est que : `openProjectInCodeServer` est utilisé pour ouvrir l'ensemble du projet, tandis que `openVaultInCodeServer` est utilisé pour ouvrir un chemin spécifique du Vault. Pour les projets multi-dépôts MonoSpecs, le système crée automatiquement des fichiers d'espace de travail. Division claire du travail, chacun fait sa part, c'est suffisant.

### Couche de service backend

Le `VaultAppService.cs` du backend implémente la logique principale :

```csharp
public async Task<VsCodeServerLaunchResponseDto> OpenInCodeServerAsync(
    string id,
    string? relativePath = null,
    string? currentInterfaceLanguage = null,
    CancellationToken cancellationToken = default)
{
    // 1. Obtenir les paramètres et vérifier si activé
    var settings = await _vsCodeServerSettingsService.GetResolvedSettingsAsync(cancellationToken);
    if (!settings.Enabled) {
        throw new BusinessException(VsCodeServerErrorCodes.Disabled, "VSCode Server is disabled.");
    }

    // 2. Obtenir le vault et résoudre le répertoire de lancement
    var vault = await RequireVaultAsync(id, cancellationToken);
    var launchDirectory = ResolveLaunchDirectory(vault, relativePath);

    // 3. S'assurer que code-server fonctionne et obtenir les informations d'exécution
    var runtime = await _vsCodeServerManager.EnsureStartedAsync(settings, cancellationToken);

    // 4. Résoudre les paramètres de langue
    var language = _vsCodeServerSettingsService.ResolveLaunchLanguage(
        settings.Language,
        currentInterfaceLanguage);

    // 5. Construire l'URL de lancement
    return new VsCodeServerLaunchResponseDto {
        LaunchUrl = AppendQueryString(runtime.BaseUrl, new Dictionary<string, string?> {
            ["folder"] = launchDirectory,
            ["tkn"] = runtime.ConnectionToken,
            ["vscode-lang"] = language,
        }),
        ConnectionToken = runtime.ConnectionToken,
        OpenMode = "folder",
        Runtime = VsCodeServerSettingsService.MapRuntime(
            await _vsCodeServerManager.GetRuntimeSnapshotAsync(cancellationToken)),
    };
}
```

Les responsabilités de cette méthode sont très claires : vérifier les paramètres, résoudre les chemins, démarrer le service, construire l'URL. La méthode `ResolveLaunchDirectory` effectue des vérifications de sécurité des chemins pour empêcher les attaques de traversée de chemins. Le code est comme de la poésie, chaque ligne a son propre rythme.

### Gestion automatique du runtime

Le backend gère le processus code-server via `VsCodeServerManager` :

- Vérifier l'état du processus
- Démarrer automatiquement les services arrêtés
- Retourner un instantané du runtime (port, ID de processus, heure de démarrage, etc.)

Cette conception permet au système de gérer automatiquement le cycle de vie de code-server, les utilisateurs n'ont pas besoin de gérer manuellement les processus du service. Après tout, la vie est déjà assez compliquée, automatisons ce qui peut l'être.

### Mécanisme de suivi de langue

HagiCode prend en charge les interfaces multilingues, et code-server doit également suivre ce paramètre. Le système prend en charge trois modes de langue :

- `follow` : suivre la langue de l'interface actuelle
- `zh-CN` : chinois fixe
- `en-US` : anglais fixe

Le paramètre d'URL `vscode-lang` est transmis à code-server pour garantir que la langue de l'éditeur reste cohérente avec l'interface HagiCode. La langue, quand elle est unifiée, c'est plus confortable.

### Espace de travail multi-dépôts MonoSpecs

Pour les projets MonoSpecs (mono-repo contenant plusieurs sous-dépôts), le système crée automatiquement un fichier `.code-workspace` :

```csharp
private async Task<string> CreateWorkspaceFileAsync(Project project, Guid projectId)
{

```

Cela permet d'éditer plusieurs sous-dépôts simultanément dans une seule instance code-server, très pratique pour les grands projets mono-repo. Plusieurs dépôts dans une seule fenêtre, comme plusieurs histoires dans un même livre.

## Intégration frontend

Le frontend HagiCode utilise React + TypeScript, l'intégration de code-server n'est pas très compliquée.

### Bouton d'action rapide

Ajouter un bouton Code Server dans la carte du projet :

```tsx
// QuickActionsZone.tsx
<Button
  size="sm"
  variant="default"
  onClick={() => onAction({ type: 'open-code-server' })}
>
  <Globe className="h-3 w-3 mr-1" />
  <span className="text-xs">{t('project.openCodeServer')}</span>
</Button>
```

Ce bouton déclenche l'action d'ouverture, appelant l'API backend pour obtenir l'URL de lancement. Un bouton, portant une action, simple et direct.

### Gestion de l'action d'ouverture

```tsx
const handleAction = async (action: ProjectAction) => {
  if (action.type === 'open-code-server') {
    const response = await openProjectInCodeServer(project.id, i18n.language);
    window.open(response.launchUrl, '_blank', 'noopener,noreferrer');
  }
};
```

Utiliser `window.open` pour ouvrir code-server dans un nouvel onglet, les paramètres `noopener,noreferrer` garantissent la sécurité. En matière de sécurité, on ne peut jamais être trop prudent.

### Point d'entrée d'édition Vault

Ajouter un bouton d'édition similaire dans la liste des Vaults :

```tsx
const handleEditVault = async (vault: VaultItemDto) => {
  const response = await openVaultInCodeServer(vault.id);
  window.open(response.launchUrl, '_blank', 'noopener,noreferrer');
};
```

Les projets et les Vaults utilisent la même méthode d'ouverture, maintenant la cohérence de l'interaction. La cohérence est aussi importante que la fonctionnalité elle-même.

## Logique de construction de l'URL

Le format de l'URL de code-server est en fait assez spécifique :

**Mode dossier** :
```
http://{host}:{port}/?folder={path}&tkn={token}&vscode-lang={lang}
```

**Mode workspace** :
```
http://{host}:{port}/?workspace={workspacePath}&tkn={token}&vscode-lang={lang}
```

Où `tkn` est le token de connexion, généré automatiquement à chaque démarrage de code-server, garantissant la sécurité de l'accès. Le paramètre `vscode-lang` contrôle la langue de l'interface de l'éditeur. Ces paramètres ont chacun leur utilité, aucun n'est superflu.

## Scénarios d'utilisation

### Scénario 1 : Révision de code assistée par IA

L'utilisateur dialogue avec HagiCode, l'IA analyse le code du projet et découvre des problèmes potentiels. L'utilisateur clique sur le bouton "Open in Code Server" pour ouvrir directement l'éditeur dans le navigateur, visualiser et corriger les fichiers problématiques, puis retourne à HagiCode pour continuer la conversation. L'ensemble du processus se termine dans le navigateur, sans changer d'application. Cette sensation, comment dire, c'est aussi fluide que l'eau qui coule.

### Scénario 2 : Édition de matériel d'apprentissage Vault

L'utilisateur a créé un Vault pour apprendre un projet open source et souhaite ajouter des notes d'apprentissage dans le répertoire `docs/`. En éditant directement les fichiers Markdown dans le navigateur via code-server, après enregistrement, HagiCode peut lire de manière synchrone le contenu mis à jour des notes. C'est très utile pour construire une base de connaissances personnelle. Le savoir, plus on l'accumule, plus il a de la valeur.

Les projets MonoSpecs contiennent plusieurs sous-dépôts, code-server crée automatiquement des espaces de travail multi-dossiers. Les utilisateurs peuvent éder simultanément le code de plusieurs dépôts dans le navigateur, et après modification, soumettre à leurs dépôts Git respectifs. Cette méthode de travail est particulièrement adaptée aux scénarios nécessitant des modifications inter-dépôts. Modifier plusieurs dépôts ensemble, c'est comme gérer plusieurs tâches simultanément, ça demande un peu de technique.

## Considérations de sécurité

Lors de la mise en œuvre de l'intégration code-server, la sécurité est un problème qui nécessite une attention particulière. Après tout, en matière de sécurité, quand il y a un problème, il est trop tard.

### Connection Token

Le connection-token est généré aléatoirement et ne doit pas être divulgué. Il est recommandé de l'utiliser dans un environnement HTTPS pour empêcher le token d'être intercepté par un attaquant de type homme du milieu. Les informations sensibles, mieux vaut les protéger.

### Sécurité des chemins de fichiers

Le backend implémente des vérifications de traversée de chemins :

```csharp
private static string ResolveLaunchDirectory(VaultRegistryEntry vault, string? relativePath)
{
    var vaultRoot = EnsureTrailingSeparator(Path.GetFullPath(vault.PhysicalPath));
    var combinedPath = Path.GetFullPath(Path.Combine(vaultRoot, relativePath ?? "."));
    if (!combinedPath.StartsWith(vaultRoot, StringComparison.OrdinalIgnoreCase))
    {
        throw new BusinessException(VaultRelativePathTraversalCode, "Relative path traversal detected.");
    }
    return combinedPath;
}
```

Ce code garantit que les utilisateurs ne peuvent pas accéder aux fichiers en dehors du répertoire vault via `../` etc. Les vérifications de limites, mieux vaut les faire que ne pas les faire.

### Contrôle des permissions

Le processus code-server s'exécute avec des permissions utilisateur appropriées, évitant d'accéder aux fichiers sensibles du système. Il est recommandé d'utiliser un utilisateur dédié pour exécuter le service code-server. Le contrôle des permissions, c'est quelque chose qu'il faut avoir.

## Optimisation des performances

code-server consomme des ressources du serveur, voici quelques suggestions d'optimisation :

- Surveiller l'utilisation du CPU/de la mémoire, ajuster les limites de ressources si nécessaire
- Les grands projets peuvent nécessiter d'augmenter le temps d'expiration
- Implémenter le nettoyage automatique des sessions expirées pour libérer des ressources
- Envisager d'utiliser le cache pour réduire les calculs répétitifs

HagiCode fournit une API de surveillance de l'état du runtime, le frontend peut obtenir l'état actuel via `getVsCodeServerSettings()` :

```typescript
const { settings, runtime } = await getVsCodeServerSettings();
// runtime.status: 'disabled' | 'stopped' | 'starting' | 'running' | 'unhealthy'
// runtime.baseUrl: "http://localhost:8080"
// runtime.processId: 12345
```

Cette conception permet aux utilisateurs de comprendre clairement l'état de santé de code-server et de localiser rapidement les problèmes lorsqu'ils surviennent. Un état visible, on sait à quoi s'attendre.

## Détails de l'expérience utilisateur

Au cours de la mise en œuvre, nous avons découvert quelques détails qui affectent l'expérience utilisateur et méritent une attention particulière :

La première ouverture de code-server peut nécessiter d'attendre le démarrage, ce temps peut varier de quelques secondes à une demi-minute. Il est recommandé d'afficher un état de chargement dans le frontend pour que l'utilisateur sache que le système traite. Attendre, tant qu'il y a un retour, ça va.

Le navigateur peut bloquer les pop-ups, il faut inviter l'utilisateur à les autoriser manuellement. HagiCode affiche des informations de guide lors de la première ouverture, indiquant à l'utilisateur comment configurer les permissions du navigateur. L'expérience utilisateur, c'est dans ces détails que ça se manifeste.

Il est recommandé d'afficher l'état du runtime (démarrage/en cours d'exécution/erreur), afin que les utilisateurs sachent s'il s'agit d'un problème serveur ou de leur propre fonctionnement lorsqu'ils rencontrent un problème. Savoir où est le problème, au moins on sait à quoi s'attendre.

## Exemple de configuration

La configuration de code-server n'est pas très compliquée :

```json
{
  "vscodeServer": {
    "enabled": true,
    "host": "0.0.0.0",
    "port": 8080,
    "language": "follow"
  }
}
```

`enabled` contrôle l'activation de la fonctionnalité, `host` et `port` spécifient l'adresse d'écoute, `language` définit le mode de langue. Ces configurations peuvent être modifiées via l'interface UI et prennent effet en temps réel. Les choses simples sont souvent les plus utiles.
