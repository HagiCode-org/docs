## Éditer DESIGN.md directement dans l'interface Web : de l'idée à la réalisation

> Dans le système de gestion de projet MonoSpecs, DESIGN.md contient l'architecture de conception et les décisions techniques du projet. Mais la méthode d'édition traditionnelle oblige les utilisateurs à basculer vers un éditeur externe, ce flux fragmenté, comment dire, c'est comme si l'inspiration disparaissait soudainement en lisant un poème — l'inspiration partie, l'humeur aussi. Cet article partage la solution que nous avons pratiquée dans le projet HagiCode : éditer DESIGN.md directement dans l'interface Web et prendre en charge l'importation de modèles depuis le site de conception en ligne. Après tout, qui n'aime pas le sentiment d'accomplir une tâche d'un trait ?

## Contexte

`DESIGN.md`, en tant que support central du document de conception de projet, contient des informations clés telles que l'architecture de conception, les décisions techniques et les directives d'implémentation. Cependant, la méthode d'édition traditionnelle oblige les utilisateurs à basculer vers un éditeur externe (comme VS Code) et à localiser manuellement le chemin physique avant de pouvoir éditer. Ce processus n'est pas complexe en soi, mais après quelques répétitions, on finit par s'épuiser.

Les problèmes spécifiques se manifestent dans les aspects suivants :

- **Flux fragmenté** : Les utilisateurs doivent fréquemment basculer entre l'interface Web de gestion et l'éditeur local, ce qui brise la continuité du flux de travail — comme lorsque la musique s'arrête soudainement pendant l'écoute, le rythme est complètement perturbé.
- **Difficulté de réutilisation** : Le site de conception a déjà publié une riche bibliothèque de modèles de conception, mais elle ne peut pas être intégrée directement dans le processus d'édition de projet. Il y a de bonnes ressources, mais on ne peut pas les utiliser, c'est vraiment dommage.
- **Expérience incomplète** : Il manque la boucle fermée "prévisualisation-sélection-importation", les utilisateurs doivent copier-coller manuellement, ce qui augmente le risque d'erreurs. Plus il y a d'opérations manuelles, plus il y a d'opportunités d'erreurs.
- **Obstacles à la collaboration** : La maintenance synchronisée des documents de conception et de l'implémentation du code devient à fort frottement, entravant l'efficacité de la collaboration d'équipe. La collaboration d'équipe n'est déjà pas facile, pourquoi ajouter ces résistances ?

Pour résoudre ces points de douleur, nous avons décidé d'implémenter la capacité d'édition directe de DESIGN.md dans l'interface Web, avec support pour l'importation en un clic de modèles depuis le site de conception en ligne. Ce n'est pas une décision révolutionnaire, juste une volonté de rendre l'expérience de développement plus fluide.

## À propos de HagiCode

La solution partagée dans cet article provient de notre expérience pratique dans le projet [HagiCode](https://hagicode.com). HagiCode est un projet d'assistant de code piloté par l'IA, et lors du développement, nous devons fréquemment maintenir les documents de conception du projet. Pour permettre à l'équipe de collaborer plus efficacement, nous avons exploré et implémenté cette solution d'édition en ligne et d'importation. En fait, il n'y a rien de spécial, juste rencontré un problème et trouvé un moyen de le résoudre.

## Solution technique

### Architecture globale

Cette solution adopte une architecture de proxy homologue avec séparation frontend-backend, principalement composée des couches suivantes. La conception de cette architecture, pour le dire simplement, se résume à "chacun son rôle" :

**1. Couche d'éditeur frontend**
```typescript
// Composant principal : DesignMdManagementDrawer
// Emplacement : repos/web/src/components/project/DesignMdManagementDrawer.tsx
// Fonction : Porte l'édition, la sauvegarde, la détection des conflits de version, le flux d'importation
```

**2. Couche de service backend**
```csharp
// Service principal : ProjectAppService.DesignMd
// Emplacement : repos/hagicode-core/src/PCode.Application/ProjectAppService.DesignMd.cs
// Fonction : Résolution de chemin, lecture/écriture de fichiers, gestion de version
```

**3. Couche de proxy homologue**
```csharp
// Service de proxy : ProjectAppService.DesignMdSiteIndex
// Emplacement : repos/hagicode-core/src/PCode.Application/ProjectAppService.DesignMdSiteIndex.cs
// Fonction : Proxy des ressources du site de conception, cache des images de prévisualisation, validation de sécurité
```

### Décisions techniques clés

#### Décision 1 : Mode tiroir global

Adopter un tiroir global unique plutôt qu'une superposition locale, en gérant l'état via `layoutSlice`, réalisant une expérience cohérente à travers les vues (classic/kanban). Cette approche garantit que les utilisateurs obtiennent une expérience d'interaction unifiée quelle que soit la vue dans laquelle ils ouvrent l'éditeur. Après tout, une expérience cohérente permet aux utilisateurs de se sentir plus à l'aise, sans se perdre en changeant de vue.

#### Décision 2 : API à portée de projet

Placer les interfaces liées à DESIGN.md sous `ProjectController`, en réutilisant les limites de permission de projet existantes, évitant la complexité d'ajouter un contrôleur indépendant. L'avantage de cette conception est une gestion des permissions plus claire et conforme aux principes d'organisation des ressources RESTful. Parfois, réutiliser est plus significatif que recréer, n'est-ce pas ?

#### Décision 3 : Détection des conflits de version

Dériver une version opaque basée sur `LastWriteTimeUtc` du système de fichiers, réalisant un contrôle optimiste de concurrence léger. Lorsque plusieurs utilisateurs éditent simultanément le même fichier, le système peut détecter les conflits et inviter les utilisateurs à actualiser. Cette conception ne bloque pas les opérations d'édition des utilisateurs tout en garantissant la cohérence des données — comme le sens des limites dans les relations interpersonnelles, ni trop éloigné, ni empiétant.

#### Décision 4 : Mode de proxy homologue

Proxy les ressources du site de conception externe via `IHttpClientFactory`, évitant les problèmes CORS et les risques SSRF. Cette conception garantit à la fois la sécurité et simplifie les appels frontend. La sécurité, on n'en fait jamais assez, après tout, la sécurité des données est comme la santé, on ne le regrette que quand c'est perdu.

## Implémentation principale

### 1. Éditer directement DESIGN.md

#### Implémentation backend

Le backend est principalement responsable de la résolution de chemin, de la lecture/écriture de fichiers et de la gestion de version. Ce travail, bien que fondamental, est indispensable, comme les fondations d'une maison :

```csharp
// Résolution de chemin et validation de sécurité
private Task<string> ResolveDesignDocumentDirectoryAsync(string projectPath, string? repositoryPath)
{
    if (string.IsNullOrWhiteSpace(repositoryPath))
    {
        return Task.FromResult(Path.GetFullPath(projectPath));
    }
    return ValidateSubPathAsync(projectPath, repositoryPath);
}

// Génération de numéro de version (basé sur l'horodatage et la taille du système de fichiers)
private static string BuildDesignDocumentVersion(string path)
{
    var fileInfo = new FileInfo(path);
    fileInfo.Refresh();
    return string.Create(
        CultureInfo.InvariantCulture,
        $"{fileInfo.LastWriteTimeUtc.Ticks:x}-{fileInfo.Length:x}");
}
```

La conception du numéro de version est assez intéressante, nous utilisons la dernière heure de modification et la taille du fichier pour générer un identifiant de version unique. C'est à la fois léger et fiable, sans avoir besoin de maintenir une base de données de version supplémentaire. Les choses simples sont souvent plus efficaces, n'est-ce pas ?

#### Implémentation frontend

Le frontend implémente la détection d'état modifié et la logique de sauvegarde. Cette conception permet aux utilisateurs de savoir à tout moment si leurs modifications sont sauvegardées, réduisant l'anxiété de "et si je perds tout" :

```typescript
// Détection d'état modifié et logique de sauvegarde
const [draft, setDraft] = useState('');
const [savedDraft, setSavedDraft] = useState('');
const isDirty = draft !== savedDraft;

const handleSave = useCallback(async () => {
    const result = await saveProjectDesignMdDocument({
        ...activeTarget,
        content: draft,
        expectedVersion: document.version, // Contrôle optimiste de concurrence
    });
    setSavedDraft(draft); // Mettre à jour l'état sauvegardé
}, [activeTarget, document, draft]);
```

Dans cette implémentation, nous maintenons deux états : `draft` est le contenu actuellement édité, `savedDraft` est le contenu sauvegardé. En comparant les deux, nous déterminons s'il y a des modifications non sauvegardées. Cette conception, bien que simple, peut donner une tranquillité d'esprit, après tout qui veut voir son dur labeur disparaître soudainement ?

### 2. Importer des fichiers de conception depuis en ligne

#### Structure des répertoires

```
repos/index/
└── src/data/public/design.json    # Index des modèles de conception

repos/awesome-design-md-site/
├── vendor/awesome-design-md/       # Modèles de conception en amont
│   └── design-md/
│       ├── clickhouse/
│       │   └── DESIGN.md
│       ├── linear/
│       │   └── DESIGN.md
│       └── ...
└── src/lib/content/
    └── awesomeDesignCatalog.ts     # Pipeline de contenu
```

#### Format des données d'index

Le fichier d'index du site de conception définit tous les modèles disponibles. Avec cet index, les utilisateurs peuvent facilement choisir le modèle qu'ils souhaitent, comme commander au restaurant :

```json
{
  "entries": [
    {
      "slug": "linear.app",
      "title": "Linear Inspired Design System",
      "summary": "Produit IA / Thème sombre",
      "detailUrl": "/designs/linear.app/",
      "designDownloadUrl": "/designs/linear.app/DESIGN.md",
      "previewLightImageUrl": "...",
      "previewDarkImageUrl": "..."
    }
  ]
}
```

Chaque entrée contient les informations de base et le lien de téléchargement du modèle. Le backend lit la liste des modèles disponibles à partir de cet index, puis l'affiche aux utilisateurs pour sélection. Cette conception rend le choix intuitif, plutôt que de tâtonner dans le noir.

#### Implémentation du proxy homologue

Pour garantir la sécurité, le backend a effectué des validations strictes sur l'accès au site de conception. La sécurité, on ne peut jamais être trop prudent :

```csharp
// Validation sécurisée du slug
private static readonly Regex SafeDesignSiteSlugRegex =
    new("^[A-Za-z0-9](?:[A-Za-z0-9._-]{0,127})$", RegexOptions.Compiled);

private static string NormalizeDesignSiteSlug(string slug)
{
    var normalizedSlug = slug?.Trim() ?? string.Empty;
    if (!IsSafeDesignSiteSlug(normalizedSlug))
    {
        throw new BusinessException(
            ProjectDesignSiteIndexErrorCodes.InvalidSlug,
            "Design site slug must be a single safe path segment.");
    }
    return normalizedSlug;
}

// Cache des images de prévisualisation (répertoire temporaire OS)
private static string ComputePreviewCacheKey(string slug, string theme, string previewUrl)
{
    var raw = $"{slug}|{theme}|{previewUrl}";
    var bytes = SHA256.HashData(Encoding.UTF8.GetBytes(raw));
    return Convert.ToHexString(bytes).ToLowerInvariant();
}
```

Ici nous avons fait deux choses : d'abord, nous utilisons une expression régulière pour valider strictement le format du slug, prévenant les attaques de traversaison de chemin ; ensuite, nous mettons en cache les images de prévisualisation, réduisant la pression sur le site externe. Le premier est la protection, le second est l'optimisation, les deux sont indispensables.

### 3. Flux d'importation complet

```typescript
// 1. Ouvrir le tiroir d'importation
const handleRequestImportDrawer = useCallback(() => {
    setIsImportDrawerOpen(true);
}, []);

// 2. Sélectionner et importer
const handleImportRequest = useCallback((entry) => {
    if (isDirty) {
        setPendingImportEntry(entry);
        setConfirmMode('import'); // Confirmation de remplacement
        return;
    }
    void executeImport(entry);
}, [isDirty]);

// 3. Exécuter l'importation
const executeImport = useCallback(async (entry) => {
    const result = await getProjectDesignMdSiteImportDocument(
        activeTarget.projectId,
        entry.slug
    );
    setDraft(result.content); // Remplacer seulement le texte de l'éditeur, ne pas sauvegarder automatiquement
    setIsImportDrawerOpen(false);
}, [activeTarget?.projectId]);
```

La conception du flux d'importation suit le principe de "confirmation utilisateur" : après l'importation, seul le contenu de l'éditeur est mis à jour, sans sauvegarde automatique. Les utilisateurs peuvent vérifier le contenu importé et sauvegarder manuellement après confirmation. Après tout, les utilisateurs devraient avoir le dernier mot sur ce qu'ils écrivent, n'est-ce pas ?

## Exemples pratiques

### Scénario 1 : Création de DESIGN.md dans le répertoire racine du projet

Lorsque DESIGN.md n'existe pas, le backend retourne un état de document virtuel. Cette conception évite au frontend de gérer le cas "fichier inexistant", l'interface API unifiée simplifiant la logique du code :

```csharp
return new ProjectDesignDocumentDto
{
    Path = targetPath,
    Exists = false,  // État de document virtuel
    Content = string.Empty,
    Version = null
};

// Création automatique du fichier lors de la première sauvegarde
public async Task<SaveProjectDesignDocumentResultDto> SaveDesignDocumentAsync(...)
{
    Directory.CreateDirectory(targetDirectory);
    await File.WriteAllTextAsync(targetPath, input.Content);
    return new SaveProjectDesignDocumentResultDto { Created = !exists };
}
```

L'avantage de cette conception est que le frontend n'a pas besoin de gérer spécialement le cas "fichier inexistant", l'interface API unifiée simplifiant la logique du code. Parfois, cacher la complexité dans le backend permet au frontend de se concentrer plus aisément sur l'expérience utilisateur.

### Scénario 2 : Importer un modèle depuis le site de conception

Lorsque l'utilisateur sélectionne le modèle de conception "Linear" dans le tiroir d'importation, le système récupère le contenu DESIGN.md via le proxy backend. L'ensemble du processus est transparent pour l'utilisateur, il lui suffit de sélectionner le modèle, le système traitant automatiquement toutes les requêtes réseau et conversions de données :

```typescript
// 1. Le système récupère le contenu DESIGN.md via le proxy backend
GET /api/project/{id}/design-md/site-index/linear.app

// 2. Le backend valide le slug et récupère le contenu depuis l'amont
var entry = FindDesignSiteEntry(catalog, "linear.app");
using var upstreamResponse = await httpClient.SendAsync(request);
var content = await upstreamResponse.Content.ReadAsStringAsync();

// 3. Le frontend remplace le texte de l'éditeur
setDraft(result.content);
// L'utilisateur vérifie puis sauvegarde manuellement sur le disque
```

L'ensemble du processus est transparent pour l'utilisateur, il lui suffit de sélectionner le modèle, le système traitant automatiquement toutes les requêtes réseau et conversions de données. Les utilisateurs n'ont pas besoin de se soucier de la complexité sous-jacente, c'est l'expérience que nous poursuivons — simple, mais puissant.

### Scénario 3 : Gestion des conflits de version

Lorsque plusieurs utilisateurs éditent simultanément le même DESIGN.md, le système détecte le conflit de version. Ce mécanisme de contrôle optimiste de concurrence garantit la cohérence des données tout en ne bloquant pas les opérations d'édition des utilisateurs :

```csharp
if (!string.Equals(currentVersion, expectedVersion, StringComparison.Ordinal))
{
    throw new BusinessException(
        ProjectDesignDocumentErrorCodes.VersionConflict,
        $"DESIGN.md at '{targetPath}' changed on disk.");
}
```

Le frontend capture cette erreur et invite l'utilisateur :

```typescript
// Le frontend invite l'utilisateur à actualiser et réessayer
<Alert>
    <AlertTitle>Conflit de version</AlertTitle>
    <AlertDescription>
        Le fichier a été modifié par un autre processus. Veuillez actualiser à la dernière version et réessayer.
    </AlertDescription>
</Alert>
```

Ce mécanisme de contrôle optimiste de concurrence garantit la cohérence des données tout en ne bloquant pas les opérations d'édition des utilisateurs. Les conflits sont inévitables, mais au moins nous pouvons informer les utilisateurs de ce qui s'est passé, plutôt que de perdre silencieusement les modifications.

## Précautions et meilleures pratiques

### 1. Sécurité des chemins

Toujours valider `repositoryPath` pour prévenir les attaques de traversaison de chemin. La sécurité, on n'en fait jamais assez :

```csharp
// Toujours valider repositoryPath pour prévenir les attaques de traversaison de chemin
return ValidateSubPathAsync(projectPath, repositoryPath);
// Rejeter "../", chemins absolus et autres entrées dangereuses
```

### 2. Stratégie de cache

Cache des images de prévisualisation pendant 24 heures, maximum 160 fichiers. Un cache modéré améliore les performances, mais ne faut pas exagérer, après tout l'équilibre est la clé :

```csharp
// Cache des images de prévisualisation 24 heures, maximum 160 fichiers
private static readonly TimeSpan PreviewCacheTtl = TimeSpan.FromHours(24);
private const int PreviewCacheMaxFiles = 160;
// Nettoyage périodique du cache expiré
```

### 3. Gestion des erreurs

Dégradation gracieuse lorsque le site amont est indisponible. Cette conception de dégradation gracieuse garantit que même si les dépendances externes sont indisponibles, la fonction d'édition principale fonctionne toujours normalement. Après tout, ce n'est pas parce qu'un service externe est en panne que tout le système doit être paralysé :

```typescript
// Dégradation gracieuse lorsque le site amont est indisponible
try {
    const catalog = await getProjectDesignMdSiteImportCatalog(projectId);
} catch (error) {
    toast.error(t('project.designMd.siteImport.feedback.catalogLoadFailed'));
    // Le tiroir d'édition principal reste disponible
}
```

Cette conception de dégradation gracieuse garantit que même si les dépendances externes sont indisponibles, la fonction d'édition principale fonctionne toujours normalement. Le système devrait être résilient, pas s'effondrer au premier problème.

### 4. Optimisation de l'expérience utilisateur

Confirmation avant remplacement lors de l'importation, pas de sauvegarde automatique après importation. Les utilisateurs devraient avoir le contrôle sur leurs opérations, pas que le système décide à leur place :

```typescript
// Confirmation avant remplacement lors de l'importation
if (isDirty) {
    setConfirmMode('import');
    return;
}

// Pas de sauvegarde automatique après importation, confirmation par l'utilisateur
setDraft(result.content); // Mettre à jour seulement le brouillon
// L'utilisateur vérifie puis clique sur Save pour écrire réellement sur le disque
```

### 5. Considérations de performance

Utiliser la factory de clients HTTP pour éviter de créer trop de connexions. La gestion des ressources, bien que semblant mineure, peut apporter des effets inattendus si bien faite :

```csharp
// Utiliser la factory de clients HTTP pour éviter de créer trop de connexions
private const string DesignSiteProxyClientName = "ProjectDesignSiteProxy";
private static readonly TimeSpan DesignSiteProxyTimeout = TimeSpan.FromSeconds(8);
```

## Suggestions d'extension

1. **Amélioration Markdown** : Actuellement utilisant `Textarea` basique, envisager une mise à niveau vers CodeMirror pour prendre en charge la coloration syntaxique et les raccourcis clavier. Une meilleure expérience d'éditeur rend la rédaction de documents plus agréable.
2. **Mode prévisualisation** : Ajouter une prévisualisation Markdown en temps réel pour améliorer l'expérience d'édition. WYSIWYG, ça donne toujours plus de confiance.
3. **Fusion intelligente** : Implémenter un algorithme de fusion intelligent plutôt qu'un simple remplacement complet du texte. Les conflits sont inévitables, mais au moins le processus de gestion des conflits peut être moins douloureux.
4. **Cache local** : Mettre en cache design.json dans la base de données pour réduire la dépendance au site externe. Moins il y a de dépendances, plus le système est stable, c'est une logique simple.

## Conclusion

Dans le projet HagiCode, nous avons implémenté une solution complète d'édition en ligne et d'importation DESIGN.md grâce à la collaboration frontend-backend. La valeur principale de cette solution réside dans :

- **Amélioration de l'efficacité** : Pas besoin de changer d'outil, éditer et importer des documents de conception dans une interface Web unifiée. Le temps économisé peut être utilisé pour des tâches plus significatives.
- **Réduction des barrières** : Importation en un clic des modèles de conception, les nouveaux projets peuvent démarrer rapidement. Plus le démarrage est facile, plus grandes sont les chances de persévérer.
- **Sécurité et fiabilité** : Les mécanismes de validation de chemin, détection des conflits de version, dégradation gracieuse garantissent le fonctionnement stable du système. La stabilité est la base, sans elle, tout n'est que discours.
- **Expérience utilisateur** : Tiroir global, détection d'état modifié, boîtes de dialogue de confirmation, etc., polissent l'expérience d'interaction. Les détails font la différence, cette phrase s'applique particulièrement à l'expérience utilisateur.
