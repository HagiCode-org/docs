## Créer une équipe d'aventure IA : pratique de configuration de la collaboration multi-Agent de HagiCode

> Dans le développement logiciel moderne, un seul Agent IA a du mal à répondre aux besoins complexes. Comment faire travailler ensemble plusieurs assistants IA de différentes entreprises sur le même projet ? Cet article partagera la solution de configuration de collaboration multi-Agent explorée par le projet HagiCode lors du développement réel.

## Contexte

De nombreux développeurs ont probablement vécu cette expérience : l'introduction d'assistants IA dans un projet pour aider à la programmation améliore vraiment l'efficacité. Mais à mesure que les besoins deviennent de plus en plus complexes, un Agent IA ne suffit plus - vous voulez qu'il traite simultanément plusieurs tâches telles que la révision de code, la génération de documentation, les tests unitaires, etc., et le résultat est souvent de sacrifier une chose pour une autre, avec une qualité de sortie inégale.

Ce qui est encore plus frustrant, c'est que lorsque vous essayez d'introduire plusieurs assistants IA, les problèmes deviennent plus complexes. Chaque Agent a sa propre méthode de configuration, son interface API et sa logique d'exécution, et ils peuvent même entrer en conflit les uns avec les autres. C'est comme une équipe de football où chaque joueur est très fort, mais personne ne sait comment collaborer, résultant en un jeu chaotique.

Le projet HagiCode a rencontré les mêmes difficultés lors de son développement. En tant que projet complexe impliquant une extension VSCode frontend, un service IA backend et un client desktop multiplateforme, nous devions connecter simultanément plusieurs assistants IA de différentes entreprises : Claude Code, Codex, CodeBuddy, iFlow, etc. Comment faire en sorte qu'ils coexistent harmonieusement dans le même projet et jouent chacun leurs forces est devenu un problème clé à résoudre.

Après tout, qui veut avoir affaire quotidiennement à un groupe d'IA qui se battent ?

La solution partagée dans cet article est précisément la pratique de configuration de collaboration multi-Agent que le projet HagiCode a affinée à travers des essais et erreurs réels. Si vous êtes aussi préoccupé par la collaboration de plusieurs assistants IA, cet article devrait vous donner quelques idées. Peut-être, après tout, chaque situation est différente.

## À propos de HagiCode

[HagiCode](https://hagicode.com) est un projet d'assistant de code IA qui adopte le mode "équipe d'aventure" avec plusieurs moteurs IA travaillant ensemble. Adresse du projet : [github.com/HagiCode-org/site](https://github.com/HagiCode-org/site).

La solution de configuration multi-Agent partagée dans cet article est l'une des technologies clés permettant à HagiCode de maintenir un développement efficace dans des projets complexes. Rien de spécial, juste transformer un groupe d'IA en une équipe d'aventure capable de collaborer.

## Conception de l'architecture multi-Agent de HagiCode

### Du "combat solo" au "travail d'équipe"

Dans les premiers stades du projet HagiCode, nous avons aussi essayé d'utiliser un seul Agent IA pour traiter toutes les tâches. Très vite, nous avons découvert que cette approche présentait des goulots d'étranglement évidents : différentes tâches nécessitent différentes capacités, certaines nécessitant une meilleure compréhension du contexte, d'autres des modifications de code plus précises. Un Agent a du mal à exceller dans tous les domaines.

Cela nous a fait réaliser que nous devions faire travailler plusieurs Agents ensemble. Mais le problème était : comment faire coexister pacifiquement des produits IA de différentes entreprises dans le même projet ? Nous devions résoudre plusieurs problèmes clés :

1. **Complexité de la gestion de la configuration** : chaque Agent a différentes méthodes de configuration, interfaces API et modes d'exécution
2. **Unification du protocole de communication** : besoin d'une méthode normalisée pour l'échange de données entre différents Agents
3. **Coordination de la répartition des tâches** : comment répartir raisonnablement les tâches pour que chaque Agent exploite ses points forts

Avec ces problèmes à l'esprit, nous avons commencé à concevoir l'architecture multi-Agent de HagiCode. Ce n'était pas si complexe finalement, nous avons juste compris les choses.

### Aperçu de l'architecture globale

Après plusieurs itérations, l'architecture que nous avons finalement déterminée est la suivante :

```
┌─────────────────────────────────────────────────────────────────┐
│                    AIProviderFactory                             │
│  (Mode usine pour gérer uniformément tous les AI Provider)       │
├─────────────────────────────────────────────────────────────────┤
│  ClaudeCodeCli  │  CodexCli  │  CodebuddyCli  │  IFlowCli    │
│  (Anthropic)   │  (OpenAI)  │  (Zhipu GLM)     │  (Zhipu)      │
└─────────────────────────────────────────────────────────────────┘
```

L'idée centrale est : grâce à l'interface Provider unifiée, différents Agents IA peuvent être gérés par le même ensemble de code. Simultanément, l'utilisation du mode usine pour créer et configurer dynamiquement ces Providers assure l'extensibilité et la flexibilité du système.

C'est comme la division du travail dans la vie, chacun a son propre rôle, nous avons juste transformé cette division du travail en architecture de code.

### Types d'Agents et division des responsabilités

Basée sur l'expérience réelle d'utilisation du projet HagiCode, nous avons assigné différentes responsabilités à chaque Agent :

| Agent | Fournisseur | Modèle | Usage principal |
|-------|--------|------|---------|
| ClaudeCodeCli | Anthropic | glm-5-turbo | Générer des solutions techniques et des propositions |
| CodexCli | OpenAI/Zed | gpt-5.4 | Exécuter des modifications de code précises |
| CodebuddyCli | Zhipu | glm-4.7 | Optimiser les descriptions de propositions et la documentation |
| IFlowCli | Zhipu | glm-4.7 | Archiver les propositions et les enregistrements historiques |
| OpenCodeCli | - | - | Édition de code générale |
| GitHubCopilot | Microsoft | - | Programmation assistée et complétion de code |

La logique derrière cette division est : chaque Agent a son propre domaine d'expertise. Claude Code excelle dans la compréhension et l'analyse des besoins complexes, il est donc responsable de la conception de la solution initiale ; Codex est plus précis dans les modifications de code, convient pour traiter les tâches d'implémentation spécifiques ; CodeBuddy offre un bon rapport qualité-prix, idéal pour optimiser la documentation.

Après tout, ce qui vous convient est le mieux, tous les chemins mènent à Rome, certains sont juste plus faciles à parcourir, d'autres un peu plus détournés.

## Mécanisme de configuration central

### Conception de l'interface Provider unifiée

Pour permettre une gestion unifiée de différents Agents IA, nous devons d'abord définir un ensemble d'interfaces unifiées. HagiCode a défini cette interface :

```csharp
public interface IAIProvider
{
    // Interface Provider unifiée
    Task<IAIProvider?> GetProviderAsync(AIProviderType providerType);
    Task<IAIProvider?> GetProviderAsync(string providerName, CancellationToken cancellationToken);
}
```

Cette interface semble simple, mais elle est la pierre angulaire de tout le système multi-Agent. Grâce à l'interface unifiée, nous pouvons ignorer quel produit IA de quelle entreprise est sous-jacent, et tous être appelés de la même manière.

En fait, cela simplifie juste les choses complexes, après tout, le simple est beau.

### Implémentation du mode usine Provider

Avec une interface unifiée, la prochaine étape est de créer des instances de ces Providers. HagiCode utilise le mode usine :

```csharp
private IAIProvider? CreateProvider(AIProviderType providerType, ProviderConfiguration config)
{
    return providerType switch
    {
        AIProviderType.ClaudeCodeCli =>
            ActivatorUtilities.CreateInstance<ClaudeCodeCliProvider>(_serviceProvider, Options.Create(config)),
        AIProviderType.CodebuddyCli =>
            ActivatorUtilities.CreateInstance<CodebuddyCliProvider>(_serviceProvider, Options.Create(config)),
        AIProviderType.CodexCli =>
            ActivatorUtilities.CreateInstance<CodexCliProvider>(_serviceProvider, Options.Create(config)),
        AIProviderType.IFlowCli =>
            ActivatorUtilities.CreateInstance<IFlowCliProvider>(_serviceProvider, Options.Create(config)),
        _ => null
    };
}
```

Ici, nous utilisons `ActivatorUtilities.CreateInstance` de l'injection de dépendances, qui peut créer dynamiquement des instances Provider au moment de l'exécution et injecter automatiquement les dépendances. L'avantage de cette conception est : lors de l'ajout d'un nouveau type d'Agent, il suffit d'ajouter la classe Provider correspondante, puis d'ajouter une branche case dans la méthode usine, sans besoin de modifier le code existant.

C'est tout, après tout, qui veut modifier une pile de vieux code à chaque fois qu'une nouvelle fonctionnalité est ajoutée.

### Analyse de configuration dynamique

Pour rendre la configuration plus flexible, nous avons également implémenté un mécanisme de mappage de types :

```csharp
public static AIProviderTypeExtensions
{
    private static readonly Dictionary<string, AIProviderType> _typeMap = new(
        StringComparer.OrdinalIgnoreCase)
    {
        ["ClaudeCodeCli"] = AIProviderType.ClaudeCodeCli,
        ["CodebuddyCli"] = AIProviderType.CodebuddyCli,
        ["CodexCli"] = AIProviderType.CodexCli,
        ["IFlowCli"] = AIProviderType.IFlowCli,
        // ... plus de mappages de types
    };
}
```

Le rôle de cette table de mappage est de convertir les noms de Provider sous forme de chaînes en types énumération. Ainsi, les fichiers de configuration peuvent utiliser des noms de chaînes intuitifs, tandis que le code interne utilise des énumérations de type sûr pour le traitement.

Après tout, pour la configuration, plus c'est intuitif, mieux c'est, qui veut mémoriser une pile de code complexe.

### Exemple de fichier de configuration

En utilisation réelle, il suffit de configurer dans `appsettings.json` :

```yaml
AI:
  Providers:
    Providers:
      ClaudeCodeCli:
        Enabled: true
        Model: glm-5-turbo
        WorkingDirectory: /path/to/project
      CodebuddyCli:
        Enabled: true
        Model: glm-4.7
      CodexCli:
        Enabled: true
        Model: gpt-5.4
      IFlowCli:
        Enabled: true
        Model: glm-4.7
```

Chaque Provider peut être configuré indépendamment avec des paramètres tels que l'interrupteur, la version du modèle et le répertoire de travail. Cette conception garantit à la fois la flexibilité et facilite la gestion et la maintenance.

En fait, les fichiers de configuration sont comme les options de la vie, vous pouvez choisir d'activer ou de désactiver certaines fonctions, seulement les choix dans le code sont plus faciles à regretter.

## Flux de travail des tâches de l'équipe d'aventure

### L'art de la division des tâches

Avec une architecture technique unifiée, l'étape suivante est de faire travailler plusieurs Agents ensemble. HagiCode a conçu un mécanisme de flux de travail permettant à différents Agents de traiter des tâches à différentes étapes :

```
Création de proposition (utilisateur)
    │
    ▼
[Claude Code] ──Générer proposition──▶ Document de proposition
    │                        │
    │                        ▼
    │               [Codebuddy] ──Optimiser description──▶ Proposition optimisée
    │                        │
    │                        ▼
    │               [Codex] ──Exécuter modifications──▶ Changements de code
    │                        │
    │                        ▼
    └───────────────▶ [iFlow] ──Archiver──▶ Enregistrements historiques
```

L'avantage de cette division est : chaque Agent n'a besoin de se concentrer que sur les tâches qu'il maîtrise, sans avoir besoin de "tout savoir". Claude Code est responsable de générer des propositions à partir de zéro, Codebuddy de rendre les descriptions de propositions plus claires, Codex de transformer les propositions en changements de code réels, et iFlow d'archiver et sauvegarder ces changements.

En fait, c'est comme le travail d'équipe dans la vie, chacun a son propre rôle, et ensemble on peut accomplir de grandes choses. Seulement ici, les membres de l'équipe sont des IA.

### Points clés de la pratique

Dans le fonctionnement réel, nous avons résumé les expériences suivantes :

**1. La stratégie de sélection des Agents est importante**

Il ne s'agit pas de répartir les tâches au hasard, mais de les répartir en fonction des points forts de chaque Agent :
- Génération de propositions : utiliser Claude Code, car il a une meilleure capacité de compréhension du contexte
- Exécution de code : utiliser Codex, car il est plus précis dans les modifications de code
- Optimisation de propositions : utiliser CodeBuddy, car il offre un bon rapport qualité-prix
- Archivage et stockage : utiliser iFlow, car il est stable et fiable

Après tout, laisser les bonnes personnes faire les bonnes choses, c'est un principe éternel.

**2. L'isolement de la configuration assure la stabilité**

La configuration de chaque Agent est gérée indépendamment, prend en charge la substitution par variables d'environnement, et les répertoires de travail sont également indépendants. Ainsi, une erreur de configuration d'un Agent n'affectera pas les autres Agents.

C'est comme les frontières dans la vie, chacun a son propre espace, et ne pas s'interférer mutuellement permet de coexistence harmonieuse.

**3. Mécanisme de gestion des erreurs**

L'échec d'un Agent ne devrait pas affecter le flux global. Nous avons implémenté une stratégie de dégradation : lorsqu'un Agent échoue, le système peut automatiquement basculer vers une solution de secours, ou directement sauter cette étape et continuer à exécuter les tâches suivantes. En même temps, une journalisation complète facilite le dépannage ultérieur.

Après tout, personne ne peut garantir de ne jamais faire d'erreur, la clé est comment gérer les erreurs. C'est comme la vie, on rencontrera toujours des revers, l'important est comment en sortir.

**4. Surveillance et observabilité**

À travers le protocole ACP (notre protocole de communication personnalisé, basé sur JSON-RPC 2.0), nous pouvons suivre l'état d'exécution de chaque Agent. L'isolement des sessions assure la sécurité des accès simultanés, et le cache dynamique optimise les performances.

Après tout, ce qu'on ne voit pas est le plus sujet à problèmes, avoir un peu de surveillance vaut mieux que d'être dans le noir complet.

## Résultats et bénéfices réels

Après avoir adopté cette configuration de collaboration multi-Agent, l'efficacité du développement du projet HagiCode s'est nettement améliorée. Concrètement :

1. **Capacité de traitement des tâches doublée** : auparavant, un Agent devait traiter simultanément plusieurs types de tâches, maintenant il peut les traiter en parallèle, le débit a plus que doublé
2. **Qualité de sortie plus stable** : chaque Agent se concentre uniquement sur les tâches qu'il maîtrise, la cohérence et la qualité des résultats sont plus élevées
3. **Coût de maintenance réduit** : l'interface et la gestion de configuration unifiées rendent l'ensemble du système plus facile à maintenir et à étendre
4. **Ajout d'Agents simple** : pour intégrer un nouveau produit IA, il suffit d'implémenter l'interface et d'ajouter la configuration, sans besoin de modifier la logique centrale

Cette solution a non seulement résolu les propres problèmes de HagiCode, mais a également prouvé que la collaboration multi-Agent est vraiment un choix d'architecture viable.

En fait, les résultats sont assez évidents, seulement le processus était un peu tortueux.

## Résumé

Cet article partage l'expérience pratique du projet HagiCode en matière de configuration de collaboration multi-Agent. Les points clés incluent :

1. **Interface standardisée** : à travers `IAIProvider` unifier le comportement de différents Agents, permettant au code d'ignorer quel produit est sous-jacent
2. **Mode usine** : utiliser `ActivatorUtilities.CreateInstance` pour créer dynamiquement des instances Provider, prenant en charge la configuration au moment de l'exécution et l'injection de dépendances
3. **Unification de protocole** : le protocole ACP réalise une communication normalisée entre Agents, mécanisme de communication bidirectionnel basé sur JSON-RPC 2.0
4. **Division du flux de tâches** : répartir raisonnablement les tâches entre différents Agents, leur permettant d'exprimer leurs points forts, plutôt que d'essayer de faire faire tout à un seul Agent

Cette conception a non seulement résolu le problème des "plusieurs Agents qui se battent", mais a également réalisé l'automatisation et la spécialisation du processus de développement grâce au mécanisme de flux de travail de l'équipe d'aventure.
