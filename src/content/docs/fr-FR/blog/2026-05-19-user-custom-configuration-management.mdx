## Meilleures pratiques de gestion de configuration personnalisée par l'utilisateur

> Rien de spécial, juste le partage d'une solution de gestion de configuration. De l'architecture à la pratique, on espère que cela vous aidera à éviter certains pièges.

## Contexte

La gestion de configuration, c'est important et fastidieux à la fois. Comme les petites habitudes de la vie — certains aiment le café le matin, d'autres aiment coder tard la nuit, ces préférences apparemment insignifiantes définissent en fait discrètement qui vous êtes. Les configurations personnalisées des utilisateurs sont similaires : le changement de thème, le choix de la langue, la personnalisation des raccourcis — lorsque ces fonctionnalités sont bien faites, les utilisateurs sentent que le produit "les comprend", et la fidélisation augmente naturellement.

Cependant, rendre ce système complet n'est pas aussi simple qu'on pourrait le penser. La gestion des versions, la compatibilité descendante, la validation des données, le contrôle de la concurrence, le contrôle des fonctionnalités DLC — ces problèmes surgissent les uns après les autres comme les soucis de l'adolescence, impossibles à éviter. Durant le développement de HagiCode, nous avons aussi été tourmentés par ces problèmes, mais heureusement nous avons fini par comprendre certains principes.

## À propos de HagiCode

Cette solution, en fait, a été explorée dans HagiCode. HagiCode est un assistant de code IA avec beaucoup de fonctionnalités — langue de l'interface, préférences de langue IA, thème, reconnaissance vocale, notifications, actions rapides, il y a beaucoup de choses à configurer. C'est précisément parce que les utilisateurs veulent tant de choses que nous avons dû créer ce système.

Adresse du projet : [github.com/HagiCode-org/site](https://github.com/HagiCode-org/site)

## Conception de l'architecture

### Architecture en couches

Le système de gestion de configuration de HagiCode, comment dire, est divisé en plusieurs couches, chacune ayant son rôle :

```
┌─────────────────────────────────────────┐
│         Frontend (React + Redux)        │
│  - Gestion de l'état de configuration   │
│  - Rendu des formulaires UI             │
│  - Persistance des configurations par groupe │
└─────────────────────────────────────────┘
                    ↓ HTTP/REST API
┌─────────────────────────────────────────┐
│   Application Service Layer             │
│  - FrontendConfigAppService             │
│  - Traitement de la logique métier      │
│  - Contrôle des permissions             │
└─────────────────────────────────────────┘
                    ↓
┌─────────────────────────────────────────┐
│   Domain Layer (Config Store)           │
│  - FrontendConfigStore                  │
│  - Lecture/écriture de configuration    │
│  - Validation et normalisation des données │
└─────────────────────────────────────────┘
                    ↓
┌─────────────────────────────────────────┐
│   Infrastructure Layer                  │
│  - Stockage de fichiers YAML            │
│  - ISystemManagedVaultService           │
│  - Contrôle de la concurrence (SemaphoreSlim) │
└─────────────────────────────────────────┘
```

Les avantages de cette approche en couches sont en fait assez évidents :
- **Responsabilités claires** : chaque couche gère ses propres affaires, sans interférence
- **Facile à tester** : chaque couche peut être testée individuellement, les modifications sont plus sûres
- **Extensible flexiblement** : changer le mode de stockage ou modifier l'API, les autres couches continuent de fonctionner

### Conception des interfaces principales

L'interface du stockage de configuration backend est définie environ comme ceci :

```csharp
public interface IFrontendConfigStore
{
    // Obtenir la configuration complète de l'utilisateur
    Task<FrontendConfigStoreResult> GetAsync(CancellationToken cancellationToken = default);
    
    // Mettre à jour la configuration (supporte les mises à jour partielles)
    Task<FrontendConfigStoreResult> UpdateAsync(
        UpdateFrontendConfigRequestDto input,
        CancellationToken cancellationToken = default);
    
    // Gestion de l'état de langue IA
    Task<FrontendConfigAiLanguageState> GetAiLanguageStateAsync(
        string userId,
        CancellationToken cancellationToken = default);
        
    Task<FrontendConfigAiLanguageState> SetAiLanguageAsync(
        string userId,
        string language,
        CancellationToken cancellationToken = default);
}
```

Les points clés de cette interface sont en fait assez compréhensibles :
- **Opérations asynchrones** : tout est asynchrone, après tout personne n'aime attendre
- **Jeton d'annulation** : si l'operation dure trop longtemps, elle expire, ne reste pas bloqué éternellement
- **Mise à jour partielle** : met seulement à jour ce qui doit être changé, sans avoir à tout revoir

## Validation et normalisation des données

### Exemple de normalisation de langue

La normalisation de configuration, en gros, c'est garder les données cohérentes. Prenons l'exemple de la configuration de langue :

```csharp
public static class FrontendConfigLanguageRules
{
    // Liste des langues d'interface prises en charge
    public static readonly HashSet<string> ValidUiLanguages = new(StringComparer.OrdinalIgnoreCase)
    {
        "zh-CN", "zh-Hant", "en-US", "ja-JP", "ko-KR", 
        "de-DE", "fr-FR", "es-ES", "pt-BR", "ru-RU"
    };
    
    public static string NormalizeUiLanguage(string? value)
    {
        // Gestion des valeurs vides : retourner la langue par défaut
        if (string.IsNullOrWhiteSpace(value)) return "en-US";
        
        var normalized = value.Trim();
        
        // Gestion des alias : convertir les alias courants en codes standard
        normalized = normalized.ToLower() switch
        {
            "zh" or "chinese" or "cn" => "zh-CN",
            "en" or "english" => "en-US",
            "ja" or "japanese" => "ja-JP",
            "ko" or "korean" => "ko-KR",
            _ => normalized
        };
        
        // Gestion des variantes dialectales
        if (normalized.StartsWith("zh-Hans", StringComparison.OrdinalIgnoreCase))
            return "zh-CN";
        if (normalized.StartsWith("zh-TW", StringComparison.OrdinalIgnoreCase))
            return "zh-Hant";
            
        // Vérifier et retourner
        return ValidUiLanguages.Contains(normalized) ? normalized : "en-US";
    }
}
```

La normalisation, en fait, c'est comme ranger sa chambre — quand les choses sont en désordre, il faut les organiser, sinon à la fin on ne se retrouve même plus :
1. **Valeur de secours pour les vides** : donner une valeur par défaut raisonnable aux valeurs vides, ne peut pas les laisser vides
2. **Mapping d'alias** : alias courants, abréviations, convertir uniformément en format standard
3. **Unification des dialectes** : les variantes dialectales sont fusionnées dans des codes standard, après tout écrire du code n'est pas faire de recherche dialectale
4. **Vérification finale** : s'assurer que la valeur retournée est dans la liste valide, sinon tout le travail est inutile

Les numéros de version, en fait, c'est pour la compatibilité descendante — les données des anciens utilisateurs ne peuvent pas être perdues à cause d'une mise à niveau de version :

```csharp
    };
```

## Contrôle des fonctionnalités DLC

HagiCode prend en charge les commutateurs de fonctionnalités DLC, certains éléments de configuration avancés doivent être achetés pour être utilisés. C'est assez courant dans les logiciels commerciaux — fonctionnalités de base gratuites, pour utiliser de bonnes choses il faut payer, après tout les développeurs doivent aussi manger.

### Vérification d'accès DLC

```csharp
private async Task<PreparedFrontendConfigUpdate> PrepareUpdateAsync(
    UpdateFrontendConfigRequestDto input,
    FrontendConfigStoreResult current,
    CancellationToken cancellationToken)
{
    // Vérifier les permissions d'accès DLC
    var accessState = await _grainFactory
        .GetDlcAccessStateGrain(TurboEngineDlcId)
        .GetAccessStateAsync();
        
    var blockedFields = new List<string>();
    
    if (!accessState.IsActive)
    {
        // DLC non activé, conserver la configuration actuelle
        if (input.GeneralSettings?.BrandingLogo != null)
        {
            blockedFields.Add("brandingLogo");
            // Conserver l'ancienne valeur
            input.GeneralSettings.BrandingLogo = current.Snapshot.GeneralSettings.BrandingLogo;
        }
        
        if (input.GeneralSettings?.BrandingTitle != null)
        {
            blockedFields.Add("brandingTitle");
            input.GeneralSettings.BrandingTitle = current.Snapshot.GeneralSettings.BrandingTitle;
        }
    }
    
    return new PreparedFrontendConfigUpdate(
        input,
        new FrontendConfigUpdateDiagnosticsDto
        {
            Status = blockedFields.Count > 0 ? "partially-applied" : "success",
            BlockedFields = blockedFields,
        });
}
```

## Gestion de l'état frontend

### Redux Slice

Le frontend utilise Redux pour gérer l'état de configuration, en fait c'est aussi assez classique :

```typescript
export const frontendConfigSlice = createSlice({
  name: 'frontendConfig',
  initialState,
  reducers: {
    setConfigStatus(state, action: PayloadAction<FrontendConfigStatus>) {
      state.status = action.payload;
    },
    updateConfigGroups(state, action: PayloadAction<Partial<FrontendConfigSnapshot>>) {
      // Fusionner les mises à jour de configuration
      Object.assign(state.snapshot, action.payload);
    },
  },
  extraReducers: (builder) => {
    builder
      .addCase(fetchConfig.pending, (state) => {
        state.status = 'loading';
      })
      .addCase(fetchConfig.fulfilled, (state, action) => {
        state.status = 'succeeded';
        state.snapshot = action.payload;
      })
      .addCase(fetchConfig.rejected, (state, action) => {
        state.status = 'failed';
        state.error = action.error.message;
      });
  },
});
```

### Encapsulation du service de configuration

```typescript
export const frontendConfigService = {
```

## Contrôle de la concurrence

Les opérations de mise à jour de configuration doivent garantir la sécurité des threads, sinon si deux personnes modifient la configuration en même temps, laquelle sera sauvegardée à la fin ? HagiCode utilise `SemaphoreSlim` pour le contrôle de la concurrence, comment dire, c'est aussi une tactique assez courante :

```csharp
private readonly SemaphoreSlim _semaphore = new(1, 1);

public async Task<FrontendConfigStoreResult> UpdateAsync(
    UpdateFrontendConfigRequestDto input,
    CancellationToken cancellationToken = default)
{
    await _semaphore.WaitAsync(cancellationToken);
    try
    {
        // Lire la configuration actuelle
        var current = await GetAsync(cancellationToken);
        
        // Préparer la mise à jour
        var prepared = await PrepareUpdateAsync(input, current, cancellationToken);
        
        // Écrire la configuration
        await WriteConfigAsync(prepared.UpdatedConfig, cancellationToken);
        
        return new FrontendConfigStoreResult(prepared.UpdatedConfig, prepared.Diagnostics);
    }
    finally
    {
        _semaphore.Release();
    }
}
```

## Guide pratique

### Étapes pour ajouter un nouvel élément de configuration

Pour ajouter un nouvel élément de configuration, suivez simplement cet ordre :

1. **Ajouter une propriété au DTO backend**

```csharp
public class FrontendConfigGeneralSettingsDto
{
    // ... propriétés existantes
    public string? NewFeatureEnabled { get; set; }
}
```

2. **Ajouter un peu de logique de normalisation**

```csharp
private static FrontendConfigGeneralSettingsDto NormalizeGeneralSettings(
    FrontendConfigGeneralSettingsDto settings)
{
    return new FrontendConfigGeneralSettingsDto
    {
        // ... propriétés existantes
        NewFeatureEnabled = NormalizeOptionalString(settings.NewFeatureEnabled),
    };
}
```

3. **Ajouter un contrôle de formulaire frontend**

```typescript
<SettingsCard

  description="Contrôler l'état d'activation des nouvelles fonctionnalités"

  <NewFeatureToggle />

```

4. **Mettre à jour la définition des groupes de configuration** (si vous ajoutez un nouveau groupe)

```typescript
export const ALL_PERSISTABLE_FRONTEND_CONFIG_GROUPS = [
  // ... groupes existants
  'newFeatureSettings',
] as const;
```

### Traitement des problèmes courants

1. **Perte de configuration** : mécanismes de sauvegarde et de restauration, il faut toujours avoir une solution de secours
2. **Conflits de concurrence** : verrouillage optimiste ou pessimiste, comme SemaphoreSlim
3. **Problèmes de performance** : prendre en charge les mises à jour partielles, ne pas transporter tout l'objet de configuration à chaque fois
4. **Sécurité** : les configurations sensibles (comme les clés API) doivent être stockées chiffrées, sinon si elles sont volées c'est problématique
