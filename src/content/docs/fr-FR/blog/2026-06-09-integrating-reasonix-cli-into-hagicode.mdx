## De zéro : Comment intégrer Reasonix CLI dans le système HagiCode

> Cet article partage la pratique technique complète de l'intégration de Reasonix CLI en tant que premier Agent Provider dans le système HagiCode, couvrant la conception d'architecture à trois niveaux, les décisions techniques clés et les détails de mise en œuvre front-end et back-end.

## Contexte

Reasonix CLI, pour le dire simplement, est un outil assez intéressant. C'est un outil d'assistant de code basé sur ACP (Agent Communication Protocol) qui fournit des capacités puissantes de streaming et de gestion de session. En fait, au niveau HagiCode.Libs, nous avons déjà terminé son implémentation de bas niveau, mais ces composants restent dans un état isolé, comme de belles perles qui n'ont pas encore été enfilées en collier. Les utilisateurs ne peuvent pas l'utiliser via la sélection de profession Hero, le flux d'exécution de session ou le panneau de surveillance, ce qui est dommage.

Le problème auquel nous sommes confrontés est : comment élever Reasonix au niveau d'un premier Agent Provider comparable à Codex et Hermes, en réalisant un routage back-end complet et un affichage front-end ? Ce n'est pas quelque chose qui peut être résolu en enregistrant simplement une valeur d'énumération, mais nécessite de construire une chaîne complète de l'abstraction de bas niveau à l'interface utilisateur. C'est comme construire une maison, on ne peut pas se contenter de couler les fondations, il faut aussi construire les murs et poser le toit.

Le défi de cette intégration est que Reasonix, en tant qu'outil CLI local, a sa propre personnalité et son propre caractère. Par exemple, il n'a pas besoin de chaîne de connexion, tous les paramètres sont configurés par l'utilisateur au moment de l'exécution ; il peut ne pas être installé du tout, ce qui nécessite une gestion de dégradation élégante ; il est compatible avec la série de modèles anthropic, mais a ses propres paramètres spécifiques à ACP comme effort, budget, etc. C'est comme une personne, avec sa propre façon unique de faire les choses, on ne peut pas y aller de force.

Après une conception d'architecture minutieuse et plusieurs discussions, nous avons finalement adopté une solution d'architecture claire à trois niveaux, intégrant avec succès Reasonix dans le système. Cette solution résout non seulement le problème immédiat, mais fournit également un modèle réutilisable pour l'intégration future de CLI Provider similaires. En fait, beaucoup de choses sont comme ça, une fois qu'on trouve la bonne méthode, le reste devient beaucoup plus facile.

## À propos de HagiCode

La solution partagée dans cet article provient de notre expérience pratique dans le projet [HagiCode](https://hagicode.com). HagiCode est un projet d'assistant de code IA open source qui vise à fournir aux développeurs des capacités puissantes de génération, de refactoring et d'optimisation de code. Au cours du développement, nous avons rencontré divers défis techniques, et l'intégration de Reasonix en tant que premier Agent Provider en est un exemple. Si vous trouvez que la solution partagée dans cet article a de la valeur, cela signifie que notre pratique d'ingénierie est plutôt bonne, et HagiCode lui-même mérite également votre attention.

## Contenu principal

### Conception de l'architecture technique

Le système adopte une architecture claire à trois niveaux pour séparer les préoccupations, chaque niveau ayant des limites de responsabilités claires :

**Niveau HagiCode.Libs** : Ce niveau est déjà terminé, fournissant l'abstraction et l'implémentation concrète du CLI provider. Il définit l'interface `ICliProvider<ReasonixOptions>`, implémente `ReasonixProvider` pour gérer le streaming ACP et la gestion de session, tout en supportant les paramètres effort, budget, yolo, transcript, etc. La responsabilité de ce niveau est de fournir des capacités de bas niveau stables et réutilisables, sans impliquer aucune logique métier. C'est comme les fondations d'une maison, invisibles mais très importantes.

**Niveau hagicode-core** : C'est le point central de notre intégration. Il est responsable de connecter l'abstraction de bas niveau à l'interface unifiée du système. Le travail spécifique comprend l'enregistrement de la valeur d'énumération `AIProviderType.ReasonixCli = 12`, la création de `ReasonixCliProvider` comme thin adapter connectant le niveau Libs, l'implémentation de `ReasonixGrain` pour gérer l'état de session et le flux d'exécution, ainsi que l'intégration du système Hero pour le mappage de paramètres et la gestion de configuration. Le cœur de ce niveau est de coordonner les composants et de construire la chaîne métier complète. C'est comme les murs porteurs d'une maison, connectant toutes les parties.

**Niveau web** : Responsable de l'affichage et de la collecte de configuration pour les utilisateurs. Nous devons régénérer les types OpenAPI pour supporter la nouvelle valeur d'énumération, implémenter le mappage de type visuel pour que Reasonix ait sa propre icône et son propre nom d'affichage, créer le formulaire de configuration des paramètres CLI pour que les utilisateurs puissent configurer chaque paramètre, et ajouter le support multilingue. Le point clé de ce niveau est l'expérience utilisateur et la conception d'interaction. C'est comme la décoration d'une maison, la qualité affecte directement le confort de vie.

Cette conception à niveaux permet à chaque niveau de se concentrer sur ses responsabilités, réduit la complexité du système et facilite la maintenance et l'extension ultérieures. En fait, beaucoup de fois, bien séparer les choses les rend plus simples.

### Décisions techniques clés

Au cours de la mise en œuvre, nous avons pris plusieurs décisions techniques clés qui ont eu un impact important sur l'architecture finale et l'expérience utilisateur.

**Décision 1 : Utiliser un Grain dédié**

Nous avons créé un `ReasonixGrain : IReasonixGrain, IExecutorStreamGrain` indépendant au lieu d'essayer de réutiliser un Grain partagé. Cette décision suit le modèle établi des 11 providers existants du système. Bien qu'il puisse y avoir une certaine duplication de code, un Grain dédié nous permet de faire un contrôle précis basé sur les caractéristiques de Reasonix, comme son mécanisme de liaison de session unique et le mappage des messages ACP. Nous avons également défini un DTO de réponse vide `ReasonixResponse` comme discriminateur de type, bien qu'il ne contienne pas de données réelles, il joue un rôle important dans le système de types. Comme chacun a sa propre chambre, même vide, c'est son propre espace.

**Décision 2 : Ne pas créer de classe Settings dédiée**

Contrairement à certains providers qui nécessitent une chaîne de connexion, toute la configuration de Reasonix est définie par l'utilisateur au moment de l'exécution et n'a pas besoin d'être validée au démarrage. Par conséquent, nous n'avons pas créé de classe Settings dédiée, mais avons stocké toute la configuration dans le dictionnaire `AIProviderOptions.Providers[ReasonixCli].Settings`. Ce modèle est cohérent avec d'autres providers CLI locaux comme Qoder, Kiro, Kimi, ce qui simplifie la structure du code et évite les couches d'abstraction inutiles. Les clés de paramètres supportées incluent : `effort`, `budgetUsd`, `transcriptPath`, `enableYolo`, `arguments`, `startupTimeoutMs`, `reasoning`. Parfois, plus c'est simple, mieux c'est.

**Décision 3 : Surveillance de santé de la stratégie Provider**

Reasonix est un CLI installé localement par l'utilisateur, il peut ne pas être installé du tout ou ne pas être dans le PATH du système. Dans ce cas, nous ne devrions pas directement signaler une erreur, mais plutôt gérer la dégradation élégamment. Nous avons utilisé la stratégie `Provider` pour vérifier si le CLI est disponible via `CommandUtil.TryResolveExecutablePath`. Si la vérification échoue, l'interface affichera "non disponible", mais n'affectera pas les autres parties du système. Cette conception rend le système plus robuste et donne aux utilisateurs un retour clair. Après tout, personne ne veut que le système entier plante à cause d'un petit problème.

**Décision 4 : Classification du système économique**

Dans le système HagiCode, différents providers ont des classifications de système économique différentes. Nous avons décidé que Reasonix utilise par défaut la classification de système économique `'claude'`, car Reasonix lui-même est compatible avec la série de modèles anthropic. Actuellement, seuls Codex et Copilot ont des classifications de système économique dédiées, les autres providers réutilisent les classifications existantes. Cela maintient la simplicité du système tout en traitant correctement la facturation et les statistiques de coûts. La réutilisation est aussi une sagesse, il n'est pas nécessaire de tout recommencer à zéro.

**Décision 5 : Compatibilité des modèles**

Reasonix supporte plusieurs modèles via le drapeau `--model`, en particulier la série anthropic. Nous avons ajouté un mappage de compatibilité dans `secondary-professions.index.json` pour permettre aux utilisateurs de sélectionner ces modèles dans Reasonix. Cette conception respecte les capacités de Reasonix tout en maintenant la cohérence du système, les utilisateurs peuvent utiliser divers modèles en douceur sans comprendre les différences de bas niveau. Ce n'est pas facile pour les utilisateurs, simplifions les choses pour eux.

**Implémentation du Thin Adapter**

`ReasonixCliProvider` est le composant clé connectant le niveau Libs et l'interface unifiée du système :

```csharp
public sealed class ReasonixCliProvider : IAIProvider, IVersionedAIProvider, IAsyncDisposable
{
    private static readonly IReadOnlyList<string> SupportedSettingKeys =
    [
        "effort",
        "budgetUsd",
        "transcriptPath",
        "enableYolo",
        "arguments",
        "startupTimeoutMs",
        "reasoning"
    ];

    private readonly ICliProvider<ReasonixOptions> _provider;
    private readonly ConcurrentDictionary<string, string> _sessionBindings = new(StringComparer.Ordinal);

    public async IAsyncEnumerable<AIStreamingChunk> StreamCoreAsync(
        AIRequest request,
        string? sessionId = null,
        [EnumeratorCancellation] CancellationToken cancellationToken = default)
    {
        var options = BuildOptions(request, sessionId);
        await foreach (var message in _provider.StreamAsync(options, cancellationToken))
        {
            yield return MapToStreamingChunk(message);
        }
    }

    private ReasonixOptions BuildOptions(AIRequest request, string? sessionId)
    {
        return new ReasonixOptions
        {
            ExecutablePath = GetExecutablePath(),
            WorkingDirectory = GetWorkingDirectory(),
            Model = _config.Model,
            Effort = _config.Settings.GetValueOrDefault("effort", "medium"),
            Budget = _config.Settings.GetValueOrDefault("budgetUsd", 10.0),
            Yolo = _config.Settings.GetValueOrDefault("enableYolo", false),
            TranscriptPath = _config.Settings.GetValueOrDefault("transcriptPath"),
            Arguments = _config.Settings.GetValueOrDefault("arguments", ""),
            StartupTimeout = GetStartupTimeout(),
            EnvironmentVariables = _environmentVariables,
            CessionId = sessionId ?? GetCessionId()
        };
    }
}
```

Les responsabilités clés de cet adapter sont :
1. Valider les paramètres de configuration, rejeter les clés de paramètres non supportées
2. Maintenir les relations de liaison de session, supporter la reprise de session
3. Mapper les messages ACP au format unifié du système
4. Utiliser `ProviderErrorAutoRetryCoordinator` pour implémenter la réessai automatique

Comme un traducteur, traduire d'une langue à une autre tout en garantissant la précision du sens.

**Implémentation du Grain Orleans**

`ReasonixGrain` est responsable de gérer l'état de session et le flux d'exécution :

```csharp
public class ReasonixGrain : Grain, IReasonixGrain, IExecutorStreamGrain
{
    private readonly Dictionary<string, ExecutorToolLifecycleStatus> _toolLifecycleState =
        new(StringComparer.Ordinal);

    public IAsyncEnumerable<ReasonixResponse> ExecuteCommandStreamAsync(
        string command,
        string? heroId = null,
        CancellationToken token = default,
        string? executionMessageId = null,
        string? systemMessage = null,
        Dictionary<string, string>? requestSettings = null)
    {
        var request = BuildRequest(command, isEdit: false, heroId, executionMessageId, systemMessage, requestSettings);
        return SendAsync(request, heroId, token);
    }

    private async IAsyncEnumerable<ReasonixResponse> SendAsync(
        AIRequest request,
        string? heroId,
        [EnumeratorCancellation] CancellationToken token)
    {
        _cancellationTokenSource = new CancellationTokenSource();
        var linkedToken = CancellationTokenSource.CreateLinkedTokenSource(token, _cancellationTokenSource.Token);

        var provider = await ResolveReasonixProviderAsync(heroId);

        await foreach (var chunk in provider.StreamAsync(request, linkedToken.Token))
        {
            var response = BuildChunkResponse(chunk);
            yield return response;
        }
    }

    private async Task<IAIProvider> ResolveReasonixProviderAsync(string? heroId)
    {
        // Logique de repli de configuration consciente de Hero
        var config = await HeroProviderResolver.ResolveAsync(AIProviderType.ReasonixCli, heroId);
        return _aiProviderFactory.CreateProvider(AIProviderType.ReasonixCli, config);
    }
}
```

Les fonctions clés du Grain incluent :
1. Utiliser `[PersistentState("reasonix-interop")]` pour maintenir l'état de session
2. Implémenter la logique de repli de configuration consciente de Hero
3. Suivre l'état du cycle de vie des outils
4. Supporter la chaîne de jetons d'annulation, garantissant que l'exécution peut être interrompue en temps opportun

C'est comme un intendant, organisant les choses de manière ordonnée.

### Meilleures pratiques et points d'attention

Au cours de la mise en œuvre, nous avons résumé certaines meilleures pratiques et des points à surveiller.

**Validation des paramètres**

ReasonixCliProvider doit valider strictement les paramètres de configuration et rejeter les clés de paramètres non supportées :

```csharp
public void ValidateConfigurationOverrides(Dictionary<string, string?> overrides)
{
    foreach (var key in overrides.Keys)
    {
        if (!SupportedSettingKeys.Contains(key))
        {
            throw new HeroProviderConfigurationException(
                $"Unsupported setting key '{key}' for Reasonix provider");
        }
    }
}
```

Cela peut empêcher les utilisateurs de configurer des paramètres incorrects et éviter les erreurs d'exécution. Comme un gardien de but, on ne peut pas laisser entrer ce qui ne devrait pas entrer.

**Gestion de la liaison de session**

Utiliser `ConcurrentDictionary` pour gérer les liaisons de session et supporter la reprise de session :

```csharp
_sessionBindings[cessionId] = sessionId;

// Lier une session existante dans les demandes ultérieures
if (_sessionBindings.TryGetValue(cessionId, out var boundSessionId))
{
    options.SessionId = boundSessionId;
}
```

Cette conception permet aux utilisateurs de reprendre la session précédente après une interruption, offrant une meilleure expérience. Comme se souvenir d'une histoire, on peut continuer à la raconter la prochaine fois.

**Gestion de dégradation élégante**

Le front-end doit vérifier la disponibilité du CLI et donner un indice amical :

```typescript
const reasonixAvailable = await healthApi.checkCliAvailable('reasonix');
if (!reasonixAvailable) {
  showMessage('Reasonix CLI n'est pas installé, veuillez le configurer dans le PATH du système');
}
```

Ne laissez pas les utilisateurs rencontrer des erreurs mystérieuses, vérifiez à l'avance et donnez des indications claires. Après tout, personne ne veut être bloqué sans raison.

**Couverture de tests**

Des tests complets sont la clé de l'assurance qualité :


Ces tests unitaires peuvent vérifier si les fonctions clés fonctionnent normalement et éviter les erreurs de régression. Comme faire des exercices avant un examen, il faut s'assurer qu'on a vraiment maîtrisé.
