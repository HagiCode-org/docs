## Comment utiliser un serveur cloud bon marché pour construire une station de distribution de téléchargement accélérée

> Le trafic de stockage cloud est outrageusement cher, l'accès transfrontalier est lentement frustrant, et le prix des CDN est dissuasif...... Ceux qui font de la distribution comprennent ces douleurs. Cet article partage une solution à faible coût que nous avons développée dans le projet [HagiCode](https://hagicode.com). Serveur cloud + couche de cache Nginx, les coûts ont diminué de moitié, la vitesse a augmenté, c'est aussi une petite consolation.

## Contexte

Sur Internet, la vitesse de téléchargement et la stabilité sont finalement une question d'expérience utilisateur. Open source ou commercial, il faut toujours fournir aux utilisateurs une méthode de téléchargement fiable.

Télécharger directement des fichiers depuis le stockage cloud (comme Azure Blob Storage, AWS S3) semble simple, mais en réalité il y a assez peu de problèmes :

**Latence réseau** : L'accès transfrontalier et transrégional est lent au point de vouloir casser son clavier. Les utilisateurs attendent que les fleurs se fanent, l'expérience peut-elle être bonne ?

**Coût de bande passante** : Le trafic de sortie du stockage cloud est douloureusement cher. Azure Blob Storage accessible depuis la Chine continentale coûte environ 0,5 yuan par GB, 1TB par mois fait 500 yuans. Pour les petites équipes, cette somme n'est pas énorme mais pas négligeable, après tout l'argent ne tombe pas du ciel.

**Restrictions d'accès** : Dans certaines régions, l'accès aux services cloud étrangers est instable, parfois carrément inaccessible. Les utilisateurs ne peuvent pas télécharger, c'est assez frustrant en soi.

**Coût CDN** : Les CDN commerciaux peuvent effectivement résoudre le problème, mais le prix est vraiment beau. Les petites équipes peuvent-elles se permettre cela ?

Y a-t-il un moyen à la fois économique et facile à utiliser ? En fait, oui. Serveur cloud + proxy inverse + couche de cache, aussi simple et brutal que cela. Les coûts ont diminué d'environ moitié, la vitesse a augmenté, c'est aussi une petite consolation.

## Conception de l'architecture

### Idée globale de l'architecture

Regardons d'abord la conception globale de l'architecture :

```
Requête utilisateur
    ↓
Résolution DNS
    ↓
┌─────────────────────────────────────┐
│   Couche de proxy inverse (Traefik/Bunker Web)   │ ← Terminaison SSL, distribution de routage, protection de sécurité
├─────────────────────────────────────┤
│   Port: 80/443                       │
│   Fonction: certificat Let's Encrypt automatique      │
│         Routage Host                    │
└─────────────────────────────────────┘
    ↓
┌─────────────────────────────────────┐
│   Couche de cache (Nginx)                     │ ← Cache de fichiers, compression Gzip
├─────────────────────────────────────┤
│   Port: 8080(server) / 8081(desktop) │
│   Stratégie de cache:                          │
│   - index.json: 1 heure               │
│   - Autres fichiers: 7 jours                   │
│   Taille du cache: 1GB                      │
└─────────────────────────────────────┘
    ↓
┌─────────────────────────────────────┐
│   Site source (Azure Blob Storage)         │ ← Stockage de fichiers
└─────────────────────────────────────┘
```

L'idée centrale de cette architecture, pour dire la vérité, est : **ajouter une couche de cache entre l'utilisateur et le stockage cloud**.

La requête de l'utilisateur arrive d'abord à la couche de proxy inverse sur le serveur cloud, puis la couche de cache Nginx prend le relais. Le fichier est dans le cache ? Donner directement à l'utilisateur. Non ? Aller chercher une copie sur le stockage cloud, et enregistrer une copie localement en passant. La prochaine fois que vous accéderez au même fichier, vous n'aurez plus à déranger le stockage cloud. En fait, c'est comme la mémoire, certaines choses une fois mémorisées, la prochaine fois vous n'aurez plus à faire d'efforts pour y penser......

### Pourquoi choisir cette architecture ?

**Avantages du serveur cloud** :
- Coût contrôlable : Alibaba Cloud et autres proposent des serveurs cloud bon marché (configuration 1-2 cœurs 2GB, frais mensuels d'environ 50-100 yuans)
- Déploiement flexible : configuration libre du proxy inverse et des stratégies de cache
- Emplacement géographique flexible : possibilité de choisir une zone de serveur proche des utilisateurs
- Forte évolutivité : possibilité de mettre à niveau la configuration en fonction des besoins de trafic

**Architecture proxy inverse + cache** :
- Réduit la pression sur le site source : met en cache les fichiers hot, réduit l'accès au stockage cloud
- Réduit les coûts : les frais de trafic du serveur cloud sont bien inférieurs au trafic de sortie du stockage cloud
- Améliore la vitesse : accès proche, la bande passante du serveur est généralement meilleure que celle du stockage cloud

**Pourquoi choisir Nginx comme couche de cache ?**

Ce n'est pas un choix arbitraire, après tout Nginx a ses raisons :

1. Haute performance : la performance du proxy inverse Nginx est reconnue dans l'industrie
2. Cache mature : fonction proxy_cache intégrée, stable et fiable, ne pose pas de problèmes fréquents
3. Faible occupation des ressources : peut fonctionner avec 256MB de mémoire, convient également au serveur
4. Configuration flexible : différents types de fichiers peuvent définir différentes stratégies de cache, c'est assez attentionné

## Couche de proxy inverse : Traefik vs Bunker Web

La solution de déploiement de HagiCode prend en charge en fait deux proxys inversés, disons, chacun a ses caractéristiques :

| Solution | Caractéristiques | Scénario applicable |
|------|------|----------|
| **Traefik** | Léger, SSL automatique, configuration simple | Déploiement de base, scénario à faible trafic |
| **Bunker Web** | WAF intégré, anti-DDoS, anti-crawlers | Exigences de sécurité élevées, scénario à fort trafic |

### Traefik : premier choix léger

Traefik est un proxy inverse HTTP moderne et équilibreur de charge, la plus grande caractéristique est la configuration simple, et peut automatiquement obtenir des certificats Let's Encrypt.

Pour le déploiement initial ou les scénarios à faible trafic, Traefik est un assez bon choix :
- Faible occupation des ressources (1.5 CPU/512MB de mémoire suffit)
- Configuration automatique des certificats SSL, pas besoin de s'en soucier
- Configuration de routage basée sur les étiquettes Docker, assez pratique

### Bunker Web : scénario haute sécurité

Bunker Web est un pare-feu d'application Web basé sur Nginx, la protection de sécurité est plus complète.

Quand peut-on envisager de passer à Bunker Web ? Probablement dans ces situations :
- Attaque DDoS subie (bien que personne ne souhaite rencontrer cela)
- Nécessité d'une protection ModSecurity
- Fonctionnalité anti-crawlers souhaitée
- Exigences plus élevées en matière de sécurité

HagiCode fournit le script `switch-deployment.sh`, permettant de basculer rapidement entre les deux solutions :

```bash
# Basculer vers Bunker Web
./switch-deployment.sh bunkerweb

# Revenir à Traefik
./switch-deployment.sh traefik

# Voir l'état actuel
./switch-deployment.sh status
```

Le script effectuera automatiquement des pré-vérifications, des contrôles de santé, et peut revenir automatiquement en arrière, le processus de basculement est également sûr et fiable, ne plantera pas en changeant.

## Configuration de la couche de cache Nginx

La couche de cache est le cœur de toute l'architecture, une bonne configuration Nginx fait une différence énorme dans l'effet du cache. Après tout, c'est assez critique.

### Configuration du chemin de cache

```nginx
# Configuration du chemin de cache
proxy_cache_path /var/cache/nginx levels=1:2 keys_zone=azure_cache:10m
                   max_size=1g inactive=7d use_temp_path=off;
```

Description des paramètres :
- `levels=1:2` : niveaux de répertoire de cache, structure de répertoire à 2 niveaux, améliore l'efficacité d'accès aux fichiers
- `keys_zone=azure_cache:10m` : zone de stockage des clés de cache, 10MB suffisent pour stocker un grand nombre de clés
- `max_size=1g` : taille maximale du cache 1GB
- `inactive=7d` : les fichiers de cache non accédés pendant 7 jours sont supprimés
- `use_temp_path=off` : écriture directe dans le répertoire de cache, améliore les performances

### Stratégie de cache hiérarchique

Différents types de fichiers ont besoin de différentes stratégies de cache :

```nginx
# Service de téléchargement Server
server {
    listen 8080;

    # Cache à court terme pour index.json (facilite les mises à jour rapides)
    location /index.json {
        proxy_cache azure_cache;
        proxy_cache_valid 200 1h;
        proxy_cache_key "$scheme$server_port$request_uri";
        add_header X-Cache-Status $upstream_cache_status;
        add_header Cache-Control "public, max-age=3600";

        # Proxy inverse vers Azure OSS
        proxy_pass https://${SERVER_DL_HOST}/${SERVER_DL_CONTAINER}$uri?${SERVER_DL_SAS_TOKEN};
        proxy_ssl_server_name on;
        proxy_ssl_protocols TLSv1.2 TLSv1.3;
    }

    # Cache à long terme pour les fichiers statiques comme les programmes d'installation
    location / {
        proxy_cache azure_cache;
        proxy_cache_valid 200 7d;
        proxy_cache_key "$scheme$server_port$request_uri";
        add_header X-Cache-Status $upstream_cache_status;
        add_header Cache-Control "public, max-age=604800";

        proxy_pass https://${SERVER_DL_HOST}/${SERVER_DL_CONTAINER}$uri?${SERVER_DL_SAS_TOKEN};
        proxy_ssl_server_name on;
        proxy_ssl_protocols TLSv1.2 TLSv1.3;
    }
}
```

**Pourquoi concevoir ainsi ?**

`index.json` est le fichier de vérification de version, doit être mis à jour rapidement, donc le temps de cache est défini à 1 heure. Ainsi après la publication d'une nouvelle version, les utilisateurs peuvent détecter la mise à jour dans 1 heure au maximum, ce n'est pas trop long.

Les fichiers statiques comme les programmes d'installation changent rarement, les mettre en cache pendant 7 jours peut réduire considérablement l'accès au site source. Quand une mise à jour est nécessaire, il suffit de nettoyer manuellement le cache, ce n'est pas trop gênant.

**En-tête de réponse X-Cache-Status** :

Cet en-tête de réponse vous permet de voir le taux de réussite du cache, c'est aussi assez utile :
- `HIT` : cache atteint
- `MISS` : cache manqué, récupéré depuis le site source
- `EXPIRED` : cache expiré, récupéré à nouveau depuis le site source
- `BYPASS` : court-circuit du cache

Méthode de visualisation :
```bash
curl -I https://server.dl.hagicode.com/app.zip
```

## Analyse des coûts

En supposant un trafic de téléchargement mensuel de 1TB, calculons les coûts :

| Solution | Frais de trafic | Frais de serveur | Total |
|------|----------|------------|------|
| Azure OSS direct | Environ ¥500 | ¥0 | ¥500 |
| Serveur cloud + OSS (taux de réussite de cache 80%) | ¥100 + ¥80 | ¥60 | ¥240 |
| CDN commercial | ¥300-500 | ¥0 | ¥300-500 |

**Conclusion** : la couche de cache peut économiser environ 50% des coûts de distribution.

Ce calcul suppose un taux de réussite de cache de 80%, dans la situation réelle, si la fréquence de mise à jour des fichiers n'est pas élevée, le taux de réussite peut être encore plus élevé, après tout c'est aussi naturel.

## Pratique de déploiement

### Préparation de l'environnement

Tout d'abord, configurez les variables d'environnement :

```bash
cd /path/to/hagicode_aliyun_deployment/docker
cp .env.example .env
vi .env  # Remplissez l'URL SAS Azure OSS, l'URL Webhook Lark
```

**Important** : le fichier `.env` contient des informations sensibles (SAS Token, Webhook URL), ne le soumettez jamais au contrôle de version, c'est assez critique.

### Configuration DNS

Ajoutez les enregistrements DNS A suivants, n'oubliez pas cette étape :
- `server.dl.hagicode.com` → IP du serveur
- `desktop.dl.hagicode.com` → IP du serveur

### Initialisation du serveur

Utilisez Ansible pour initialiser automatiquement le serveur :

```bash
cd /path/to/hagicode_aliyun_deployment
ansible-playbook -i ./ansible/inventory/hosts.yml ./ansible/playbooks/init.yml
```

Ce Playbook résoudra automatiquement ces points :
- Créer un utilisateur de déploiement
- Installer Docker et Docker Compose
- Configurer les clés SSH
- Configurer les règles de pare-feu

Pas trop complexe, après tout l'automatisation fait gagner du temps et des efforts.

### Déploiement du service

```bash
./deploy.sh
```

Le script de déploiement vous aidera à faire :
- Vérifier la configuration de l'environnement
- Tirer le dernier code
- Démarrer les conteneurs Docker
- Exécuter les contrôles de santé
- Envoyer des notifications de déploiement (Lark)

Une commande pour tout faire, assez pratique.

## Astuces d'exploitation

### Gestion du cache

Le cache doit être occasionnellement entretenu :

**Voir l'utilisation du disque de cache** :
```bash
docker volume inspect docker_nginx-cache
du -sh /var/lib/docker/volumes/docker_nginx-cache/_data
```

**Nettoyer manuellement le cache** :
```bash
./clear-cache.sh
```

Ou exécuter manuellement, bien que un peu plus gênant, mais efficace :
```bash
docker exec nginx sh -c "rm -rf /var/cache/nginx/*"
docker restart nginx
```

### Limites de ressources

Sur un serveur 1 cœur 2GB, la configuration des limites de ressources est la suivante :

```yaml
services:
  traefik:
    deploy:
      resources:
        limits:
          cpus: '1.50'
          memory: 512M

  nginx:
    deploy:
      resources:
        limits:
          cpus: '0.50'
          memory: 256M
```

Surveiller l'utilisation des ressources, regarder occasionnellement c'est aussi bien :
```bash
docker stats
```

### Bonnes pratiques de sécurité SAS Token

Le SAS Token est les informations d'identification pour accéder à Azure Blob Storage, les fuites ne sont pas une plaisanterie :

- Le fichier `.env` n'est pas soumis au contrôle de version (déjà dans .gitignore)
- Le SAS Token définit un délai d'expiration approprié (recommandé 1 an)
- Limiter les autorisations du SAS Token (lecture seule)
- Rotation régulière du SAS Token

### Surveillance et alertes

HagiCode intègre les notifications Webhook Lark/Feishu, permettant d'envoyer des notifications dans les situations suivantes :
- Succès/échec du déploiement
- État de nettoyage du cache
- Anomalie de service

Les notifications contiennent des informations sur le serveur, l'horodatage, les détails de l'erreur, permettant de localiser rapidement les problèmes.

### Extension haute disponibilité

Quand un seul serveur ne peut plus supporter, vous pouvez aussi envisager :

1. **Extension horizontale** : déployer plusieurs nœuds, distribuer par rotation DNS ou équilibreur de charge
2. **Support CDN** : ajouter un CDN devant le serveur cloud, améliorer encore la vitesse d'accès
3. **Préchauffage du cache** : utiliser des scripts pour charger à l'avance les fichiers populaires dans le cache

## Précautions

Quelques points à rappeler, après tout personne ne veut rencontrer des problèmes :

1. **Certificat SSL** : Let's Encrypt a des limites de taux, ne basculez pas fréquemment les déploiements, sinon vous pourriez ne pas obtenir de certificat
2. **Nettoyage du cache** : n'oubliez pas de nettoyer le cache après avoir mis à jour des fichiers importants, sinon les utilisateurs pourraient ne pas télécharger la nouvelle version
3. **Gestion des journaux** : nettoyez régulièrement les journaux Docker, sinon le disque sera plein et ce sera gênant
4. **Stratégie de sauvegarde** : Traefik acme.json, configuration Bunker Web, il vaut mieux les sauvegarder
5. **Surveillance et alertes** : configurez les notifications Lark, connaître l'état du déploiement à temps, réagir rapidement en cas de problème

Serveur cloud + couche de cache Nginx, aussi simple que cela. HagiCode utilise cette solution, les coûts ne sont pas élevés (frais de serveur environ 60-100 yuans/mois), l'effet est assez bon. Les avantages principaux sont environ ceux-ci :

- **Coût contrôlable** : par rapport à l'utilisation directe du stockage cloud ou d'un CDN commercial, les coûts ont diminué d'environ 50%
