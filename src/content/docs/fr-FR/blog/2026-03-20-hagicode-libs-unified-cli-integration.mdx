## Hagicode.Libs : Pratiques d'ingénierie pour l'intégration unifiée de multiples assistants de programmation IA en CLI

> Lors du développement du projet HagiCode, nous devions intégrer simultanément plusieurs CLI d'assistants de programmation IA comme Claude Code, Codex, CodeBuddy, etc. Chaque CLI ayant des interfaces, des paramètres et des formats de sortie différents, le code d'intégration répétitif rendait le projet de plus en plus difficile à maintenir. Cet article partage notre expérience sur la construction d'une couche d'abstraction unifiée via la bibliothèque HagiCode.Libs pour résoudre ce problème d'ingénierie — on peut dire que ce sont les obstacles sur lesquels nous avons trébuché et l'expérience que nous avons accumulée.

## Contexte

Le marché actuel des assistants de programmation IA est assez animé. Outre Claude Code, il y a Codex d'OpenAI, CodeBuddy de Zhipu, etc. En tant que projet d'assistant de code IA, HagiCode doit intégrer ces différents outils CLI dans plusieurs sous-projets (bureau, backend, Web, etc.).

Au début, le problème n'était pas grave, intégrer une CLI n'était que quelques centaines de lignes de code. Mais à mesure que nous devions supporter de plus en plus de CLI, les choses ont commencé à devenir compliquées —  

Chaque CLI a un format de paramètres de ligne de commande différent, les exigences de variables d'environnement ne sont pas les mêmes, et les formats de sortie sont encore plus variés — certains sortent du JSON, d'autres du JSON en flux, d'autres du texte brut. En plus des problèmes de compatibilité multiplateforme, les mécanismes de découverte d'exécutables et de gestion des processus sont complètement différents entre Windows et les systèmes Unix, et la répétition du code devient de plus en plus élevée. En fait, ce n'est rien de plus que quelques Ctrl+C, Ctrl+V supplémentaires, mais la maintenance devient un véritable casse-tête.

Le plus casse-tête, c'est que chaque fois qu'il faut ajouter le support d'une nouvelle fonctionnalité CLI, il faut modifier le même code dans plusieurs projets. Cette approche n'est clairement pas une solution durable — le code a aussi son caractère, et s'il est trop répété, il finira par vous faire des siennes.

## À propos de HagiCode

La solution partagée dans cet article provient de notre expérience pratique dans le projet [HagiCode](https://hagicode.com). HagiCode est un projet d'assistant de code IA open source qui doit maintenir simultanément plusieurs sous-projets : l'extension VSCode frontend, le service IA backend, le client desktop multiplateforme, etc. Comment dire, c'est précisément ce scénario complexe multilingue et multiplateforme qui a favorisé la naissance de HagiCode.Libs — on peut dire que c'est né de la nécessité, soit.

## Analyse : Chercher les points communs

Bien que ces CLI d'assistants de programmation IA aient chacune leurs caractéristiques, d'un point de vue technique, elles présentent des traits communs évidents :

**Modes d'interaction similaires** : toutes lancent un processus CLI, envoient un prompt, reçoivent une réponse en flux, analysent les messages, et enfin la session se termine ou continue — ce processus, au fond, est coulé dans le même moule.

**Besoins de configuration similaires** : toutes nécessitent une authentification par clé API, la configuration du répertoire de travail, le choix du modèle, le contrôle des permissions des outils, la gestion des sessions. Après tout, tout le monde vit de l'API, la différence est juste une question de goût.

**Défis multiplateforme cohérents** : toutes doivent résoudre les problèmes de résolution de chemin d'exécutable (`claude` vs `claude.exe` vs `/usr/local/bin/claude`), le démarrage de processus et le traitement des variables d'environnement, l'échappement des commandes Shell et la construction des paramètres. Ces histoires multiplateforme, plus on en parle, plus on pleure — les différences entre Windows et Unix, seul ceux qui ont trébuché comprennent.

Sur la base de ces analyses, nous avions besoin d'une couche d'abstraction unifiée pour fournir une interface cohérente, encapsuler la logique de découverte CLI multiplateforme, traiter l'analyse des sorties en flux, tout en supportant l'injection de dépendances et l'utilisation sans DI. C'est le genre de chose qui fait mal à la tête juste d'y penser, mais il faut y faire face — après tout, c'est notre propre projet, on doit finir même en pleurant.

## Solution : HagiCode.Libs

Nous avons créé **HagiCode.Libs** — un espace de travail de bibliothèque .NET 10 léger, utilisant la licence open source MIT, maintenant publié sur [GitHub](https://github.com/HagiCode-org/Hagicode.Libs). Bien que ce ne soit pas une œuvre spectaculaire, elle résout des problèmes réels, ce qui est plutôt satisfaisant.

### Structure du projet

```
HagiCode.Libs/
├── src/
│   ├── HagiCode.Libs.Core/           # Fonctionnalités principales
│   │   ├── Discovery/                 # Découverte d'exécutables CLI
│   │   ├── Process/                   # Gestion de processus multiplateforme
│   │   ├── Transport/                 # Transport de messages en flux
│   │   └── Environment/               # Résolution d'environnement d'exécution
│   ├── HagiCode.Libs.Providers/       # Implémentations de fournisseurs
│   │   ├── ClaudeCode/                # Fournisseur Claude Code
│   │   ├── Codex/                     # Fournisseur Codex
│   │   └── Codebuddy/                 # Fournisseur CodeBuddy
│   ├── HagiCode.Libs.ConsoleTesting/  # Framework de test
│   ├── HagiCode.Libs.ClaudeCode.Console/
│   ├── HagiCode.Libs.Codex.Console/
│   └── HagiCode.Libs.Codebuddy.Console/
└── tests/                             # Tests xUnit
```

### Objectifs de conception

Dans la conception de HagiCode.Libs, nous avons suivi plusieurs principes — après tout, ce sont aussi des obstacles sur lesquels nous avons trébuché et l'expérience résumée :

**Zéro dépendance de framework lourd** : pas de dépendance à ABP ou d'autres frameworks volumineux, rester léger. De nos jours, moins il y a de dépendances, moins il y a de problèmes — qui n'a jamais souffert de l'enfer des dépendances ?

**Support multiplateforme** : support natif pour Windows, macOS, Linux, pas besoin d'écrire différents codes pour différentes plateformes. Un seul code pour tout le monde, c'est plutôt bien.

**Traitement en flux** : utiliser des flux asynchrones pour traiter les sorties CLI, plus conforme au modèle de programmation .NET moderne. Après tout, les temps changent, l'asynchrone est roi.

**Intégration flexible** : supporte à la fois les scénarios d'injection de dépendances et permet l'instanciation directe. Chacun ses préférences, utilisez ce qui est le plus pratique.

## Mode d'utilisation

### Via l'injection de dépendances

Si votre projet utilise déjà l'injection de dépendances (comme ASP.NET Core ou l'hôte générique), vous pouvez l'intégrer directement — c'est un peu comme un enfant, bien que pas grand, mais plutôt sage :

```csharp
using HagiCode.Libs.Providers;
using Microsoft.Extensions.DependencyInjection;

var services = new ServiceCollection();
services.AddHagiCodeLibs();

await using var provider = services.BuildServiceProvider();
var claude = provider.GetRequiredService<ICliProvider<ClaudeCodeOptions>>();

var options = new ClaudeCodeOptions
{
    ApiKey = "your-api-key",
    Model = "claude-sonnet-4-20250514"
};

await foreach (var message in claude.ExecuteAsync(options, "Hello, Claude!"))
{
    Console.WriteLine($"{message.Type}: {message.Content}");
}
```

### Instanciation directe

Pour les scripts simples ou les scénarios sans DI, la création directe d'instance fonctionne aussi — pour dire la vérité, c'est une question de préférence personnelle :

```csharp
var claude = new ClaudeCodeProvider();
var options = new ClaudeCodeOptions
{
    ApiKey = "sk-ant-xxx",
    Model = "claude-sonnet-4-20250514"
};

await foreach (var message in claude.ExecuteAsync(options, "帮我写一个快速排序"))
{
    // Traiter les messages
}
```

Les deux méthodes utilisent la même implémentation sous-jacente, vous pouvez choisir la méthode d'intégration appropriée selon la situation réelle du projet. Il n'y a pas de réponse standard dans ce monde, ce qui vous convient est le meilleur — bien que ce soit un peu cliché, c'est vraiment le cas.

## Partage d'expérience pratique

### 1. Consoles de test dédiées

Chaque fournisseur a un projet de console de test dédié, facilitant la vérification indépendante de l'intégration — comment dire, pour les tests, soit on ne le fait pas, soit on le fait à fond :

```bash
# Test Claude Code
dotnet run --project src/HagiCode.Libs.ClaudeCode.Console -- --test-provider
dotnet run --project src/HagiCode.Libs.ClaudeCode.Console -- --test-all claude

# Test CodeBuddy
dotnet run --project src/HagiCode.Libs.Codebuddy.Console -- --test-provider codebuddy-cli

# Test Codex
dotnet run --project src/HagiCode.Libs.Codex.Console -- --test-provider codex-cli
```

Les scénarios de test couvrent plusieurs scénarios clés :
- **Ping** : vérification de santé, confirmer que la CLI est disponible
- **Simple Prompt** : test de prompt basique
- **Complex Prompt** : test de conversation multi-tours
- **Session Restore/Resume** : test de reprise de session
- **Repository Analysis** : test d'analyse de codebase

Cette conception de console de test indépendante est particulièrement utile lors du débogage, permettant de localiser rapidement si le problème se situe au niveau HagiCode.Libs ou dans la CLI elle-même. Le débogage, pour dire la vérité, c'est juste voir où est le problème — une fois la bonne direction trouvée, on a déjà réussi à moitié.

### 2. Vérification CI/CD multiplateforme

La compatibilité multiplateforme est l'un des objectifs principaux de HagiCode.Libs. Nous avons configuré un workflow GitHub Actions `.github/workflows/cli-discovery-cross-platform.yml` qui exécute une vérification réelle de découverte CLI sur trois plateformes : `ubuntu-latest`, `macos-latest`, `windows-latest`.

Cela garantit que chaque changement de code ne brise pas la compatibilité multiplateforme. Lors du développement local, vous pouvez également reproduire avec les commandes suivantes — on ne peut pas laisser CI porter toute la responsabilité, il faut aussi pouvoir l'exécuter localement :

```bash
npm install --global @anthropic-ai/claude-code@2.1.79
HAGICODE_REAL_CLI_TESTS=1 dotnet test --filter "Category=RealCli"
```

### 3. Traitement des flux de messages

HagiCode.Libs utilise des flux asynchrones pour traiter les sorties CLI, cette approche est plus conforme au style de programmation asynchrone .NET moderne que les modèles traditionnels de callback ou d'événements — en fin de compte, c'est le progrès technologique, personne ne peut l'arrêter :

```csharp
public async IAsyncEnumerable<CliMessage> ExecuteAsync(
    TOptions options,
    string prompt,
    [EnumeratorCancellation] CancellationToken cancellationToken = default)
{
    // Démarrer le processus CLI
    // Analyser la sortie JSON en flux
    // Générer la séquence CliMessage
}
```

Les types de messages incluent :
- `user` : message utilisateur
- `assistant` : réponse de l'assistant
- `tool_use` : appel d'outil
- `result` : fin de session

Cette conception permet à l'appelant de traiter flexiblement les sorties en flux, comme l'affichage en temps réel, le traitement après tamponnage, ou le transfert vers d'autres services. Pourquoi se soucier de la météo ? L'important c'est que l'esprit s'ouvre, toutes les utilisations sont possibles.

### 4. Exploration de dépôt Git

Le module HagiCode.Libs.Exploration offre des fonctionnalités de découverte et de vérification d'état de dépôts Git, particulièrement utile dans les scénarios d'analyse de codebase — c'est aussi une fonctionnalité née de la nécessité, puisque HagiCode a besoin d'analyser les codebases :

```csharp
// Découvrir les dépôts Git
var repositories = await GitRepositoryDiscovery.DiscoverAsync("/path/to/search");

// Obtenir les informations du dépôt
var info = await GitRepository.GetInfoAsync(repoPath);
Console.WriteLine($"Branch: {info.Branch}, Remote: {info.RemoteUrl}");
Console.WriteLine($"Has uncommitted changes: {info.HasUncommittedChanges}");
```

La fonction d'analyse de code de HagiCode utilise ce module pour identifier la structure du projet et l'état Git. On peut dire que c'est une utilisation optimale, c'est aussi comme un enfant, pas élevé en vain.

## Points d'attention

Basé sur notre pratique dans le projet HagiCode, il y a plusieurs points à particulièrement attentionner — tout est problème, problème, problème :

**Sécurité des clés API** : ne codez pas en dur les clés API dans le code, utilisez des variables d'environnement ou la gestion de configuration. HagiCode.Libs supporte la transmission de configuration via l'objet Options, facilitant l'intégration de diverses sources de configuration. Après tout, pour la sécurité, on ne peut jamais être trop prudent.

**Verrouillage de version CLI** : dans CI/CD, nous verrouillons des versions spécifiques (comme `@anthropic-ai/claude-code@2.1.79`) pour réduire l'incertitude due à la dérive de version. Lors du développement local, il est recommandé d'utiliser également une version fixe. Les histoires de version, plus on en parle, plus on pleure — sans version fixe, ça vous apprend la vie en quelques secondes.

**Catégorisation des tests** : par défaut, les tests utilisent de faux fournisseurs pour maintenir le déterminisme et la vitesse, les tests CLI réels nécessitent une activation explicite. Cela garantit à la fois un retour rapide de CI et permet une vérification en environnement réel lorsque nécessaire. Marcher sur cet équilibre n'est pas facile — rapidité et stabilité nécessitent toujours des compromis.

**Gestion des sessions** : les mécanismes de reprise de session diffèrent entre les CLI, Claude Code utilise le répertoire `.claude/` pour stocker les sessions, Codex et CodeBuddy ont chacun leur propre méthode. Lors de l'utilisation, veuillez consulter leurs documentations respectives pour comprendre le mécanisme spécifique de persistance des sessions. C'est comme ça, comprendre clairement ne fait jamais de mal.

En fournissant une interface cohérente, en encapsulant les détails multiplateforme, et en supportant des méthodes d'intégration flexibles, elle réduit considérablement la complexité d'ingénierie d'intégration de plusieurs assistants de programmation IA.
