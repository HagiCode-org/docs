## Réaliser la synchronisation automatique d'images de Docker Hub vers Azure ACR

> Cet article explique comment utiliser GitHub Actions et l'outil image-syncer pour automatiser la synchronisation d'images de Docker Hub vers Azure Container Registry, résolvant le problème de lenteur d'accès à Docker Hub en Chine continentale et dans certaines régions Azure, tout en améliorant la disponibilité des images et l'efficacité de déploiement dans les environnements Azure.

## Contexte / Introduction

Le projet HagiCode utilise des images Docker comme composants d'exécution principaux, les images principales étant hébergées sur Docker Hub. Avec l'évolution du projet et la croissance des besoins de déploiement Azure, nous avons rencontré les problèmes suivants :
- La lenteur de récupération des images, car l'accès à Docker Hub est limité en Chine continentale et dans certaines régions Azure
- La dépendance à une source d'images unique crée un risque de point de défaillance unique
- L'utilisation d'Azure Container Registry dans les environnements Azure offre de meilleures performances réseau et une meilleure expérience d'intégration

Pour résoudre ces problèmes, nous devons établir un mécanisme de synchronisation automatique d'images qui synchronise régulièrement les images de Docker Hub vers Azure ACR, afin de garantir que les utilisateurs bénéficient de vitesses de récupération d'images plus rapides et d'une plus grande disponibilité dans les environnements Azure.

## Comparaison des solutions techniques

Lors de la définition de la solution, nous avons comparé plusieurs approches techniques :

### 1. image-syncer (choix final)
- Synchronisation incrémentale : synchronise uniquement les couches d'images modifiées, réduisant considérablement le transfert réseau
- Reprise après interruption : la synchronisation peut reprendre après une interruption réseau
- Contrôle de la concurrence : prend en charge la configuration du nombre de threads concurrents pour améliorer l'efficacité de synchronisation des grandes images
- Gestion des erreurs robuste : mécanisme de réessai intégré en cas d'échec (3 fois par défaut)
- Déploiement léger : un seul binaire sans dépendances
- Prise en charge multi-registries : compatible avec Docker Hub, Azure ACR, Harbor, etc.

### 2. Docker CLI
- Pas de prise en charge de la synchronisation incrémentale : chaque exécution nécessite de récupérer le contenu complet de l'image
- Efficacité réduite : volume de transfert réseau important et temps d'exécution long
- Simple et facile à utiliser : repose sur les commandes familières `docker pull` / `docker push`

### 3. Azure CLI
- Complexité plus élevée : nécessite la configuration de l'authentification Azure CLI
- Limitations fonctionnelles : `az acr import` est relativement limité
- Intégration native : s'intègre bien avec les services Azure

## Décisions de conception de l'architecture

### Décision 1 : définir la fréquence de synchronisation à quotidienne à 00:00 UTC
- Équilibre entre la fraîcheur des images et la consommation de ressources
- Évite les heures de pointe d'activité et réduit l'impact sur les autres opérations
- Les images Docker Hub sont généralement mises à jour après les builds quotidiens

### Décision 2 : synchroniser toutes les étiquettes d'images
- Maintient une cohérence complète avec Docker Hub
- Offre des choix de version flexibles aux utilisateurs
- Simplifie la logique de synchronisation en évitant des règles de filtrage d'étiquettes complexes

### Décision 3 : stocker les identifiants dans GitHub Secrets
- Prise en charge native de GitHub Actions avec une forte sécurité
- Configuration simple, facile à gérer et à maintenir
- Prise en charge du contrôle d'accès au niveau du dépôt

## Évaluation et atténuation des risques

### Risque 1 : fuite des identifiants Azure ACR
- Utiliser GitHub Secrets pour le stockage chiffré
- Faire tourner régulièrement les mots de passe ACR
- Limiter les permissions du compte ACR à la poussée uniquement
- Surveiller les journaux d'accès ACR

### Risque 2 : échecs de synchronisation entraînant une incohérence des images
- image-syncer inclut un mécanisme de synchronisation incrémentale intégré
- Réessai automatique en cas d'échec (3 fois par défaut)
- Journaux d'erreurs détaillés et notifications d'échec
- Fonctionnalité de reprise après interruption

### Risque 3 : consommation excessive de ressources
- La synchronisation incrémentale réduit le transfert réseau
- Concurrence configurable (réglée à `10` dans la configuration actuelle)
- Surveiller le nombre et la taille des images synchronisées
- Exécuter la synchronisation pendant les heures creuses

## Solution principale

Nous adoptons une solution automatisée GitHub Actions + image-syncer pour synchroniser les images de Docker Hub vers Azure ACR.

## Étapes de mise en œuvre

### 1. Phase de préparation
- Créer ou confirmer un Azure Container Registry dans le portail Azure
- Créer des identifiants d'accès ACR (nom d'utilisateur et mot de passe)
- Confirmer les droits d'accès au dépôt d'images Docker Hub

### 2. Configuration des GitHub Secrets
Ajoutez les secrets suivants dans les paramètres du dépôt GitHub :
- AZURE_ACR_USERNAME : nom d'utilisateur Azure ACR
- AZURE_ACR_PASSWORD : mot de passe Azure ACR

### 3. Création du workflow GitHub Actions
Configurez le workflow dans `.github/workflows/sync-docker-acr.yml` :
- Déclencheur planifié : tous les jours à 00:00 UTC
- Déclencheur manuel : prend en charge `workflow_dispatch`
- Déclencheur supplémentaire : exécution lors d'une poussée sur la branche `publish` (pour une synchronisation rapide)

### 4. Flux d'exécution du workflow

| Séquence | Participant | Action | Description |
| --- | --- | --- | --- |
| 1 | GitHub Actions | Déclencher le workflow | Déclenché par la planification, l'exécution manuelle ou une poussée sur la branche `publish` |
| 2 | GitHub Actions → image-syncer | Télécharger et exécuter l'outil de synchronisation | Entrée dans la phase de synchronisation réelle |
| 3 | image-syncer → Docker Hub | Récupérer les manifests et la liste des étiquettes d'images | Lire les métadonnées du dépôt source |
| 4 | image-syncer → Azure ACR | Récupérer les informations d'images existantes du dépôt cible | Déterminer l'état actuel côté cible |
| 5 | image-syncer | Comparer les différences entre source et cible | Identifier les couches d'images à synchroniser |
| 6 | image-syncer → Docker Hub | Récupérer les couches d'images modifiées | Transférer uniquement le contenu nécessitant une mise à jour |
| 7 | image-syncer → Azure ACR | Pousser les couches d'images modifiées | Compléter la synchronisation incrémentale |
| 8 | image-syncer → GitHub Actions | Retourner les statistiques de synchronisation | Inclut les résultats, les différences et les informations d'erreur |
| 9 | GitHub Actions | Enregistrer les journaux et télécharger les artefacts | Utile pour l'audit et le dépannage ultérieurs |

## Implémentation du workflow GitHub Actions

Voici la configuration de workflow réellement utilisée (`.github/workflows/sync-docker-acr.yml`) :

```yaml
name: Sync Docker Image to Azure ACR

on:
  schedule:
    - cron: "0 0 * * *" # Tous les jours à 00:00 UTC
  workflow_dispatch: # Déclencheur manuel
  push:
    branches: [publish]

permissions:
  contents: read

jobs:
  sync:
    runs-on: ubuntu-latest

    steps:
      - name: Checkout code
        uses: actions/checkout@v4

      - name: Download image-syncer
        run: |
          # Télécharger le binaire image-syncer
          wget https://github.com/AliyunContainerService/image-syncer/releases/download/v1.5.5/image-syncer-v1.5.5-linux-amd64.tar.gz
          tar -zxvf image-syncer-v1.5.5-linux-amd64.tar.gz
          chmod +x image-syncer

      - name: Create auth config
        run: |
          # Générer le fichier de configuration d'authentification (format YAML)
          cat > auth.yaml <<EOF
          hagicode.azurecr.io:
            username: "${{ secrets.AZURE_ACR_USERNAME }}"
            password: "${{ secrets.AZURE_ACR_PASSWORD }}"
          EOF

      - name: Create images config
        run: |
          # Générer le fichier de configuration de synchronisation d'images (format YAML)
          cat > images.yaml <<EOF
          docker.io/newbe36524/hagicode: hagicode.azurecr.io/hagicode
          EOF

      - name: Run image-syncer
        run: |
          # Exécuter la synchronisation (avec les nouveaux paramètres --auth et --images)
          ./image-syncer --auth=./auth.yaml --images=./images.yaml --proc=10 --retries=3

      - name: Upload logs
        if: always()
        uses: actions/upload-artifact@v4
        with:
          name: sync-logs
          path: image-syncer-*.log
          retention-days: 7
```

## Détails de la configuration

### 1. Conditions de déclenchement
- Déclencheur planifié : `cron: "0 0 * * *"` — s'exécute tous les jours à 00:00 UTC
- Déclencheur manuel : `workflow_dispatch` — permet aux utilisateurs de l'exécuter manuellement dans l'interface GitHub
- Déclencheur de poussée : `push: branches: [publish]` — déclenché lorsqu'une poussée est effectuée sur la branche publish (pour une synchronisation rapide)

### 2. Configuration d'authentification (`auth.yaml`)
```yaml
hagicode.azurecr.io:
  username: "${{ secrets.AZURE_ACR_USERNAME }}"
  password: "${{ secrets.AZURE_ACR_PASSWORD }}"
```

### 3. Configuration de synchronisation d'images
```yaml
docker.io/newbe36524/hagicode: hagicode.azurecr.io/hagicode
```
Cette configuration signifie synchroniser toutes les étiquettes de `docker.io/newbe36524/hagicode` vers `hagicode.azurecr.io/hagicode`

### 4. Paramètres image-syncer
- `--auth=./auth.yaml` : chemin du fichier de configuration d'authentification
- `--images=./images.yaml` : chemin du fichier de configuration de synchronisation d'images
- `--proc=10` : définir la concurrence à `10`
- `--retries=3` : réessayer `3` fois en cas d'échec

## Remarques et bonnes pratiques

### 1. Recommandations de sécurité
- Faire tourner régulièrement les mots de passe Azure ACR (recommandé tous les 90 jours)
- Utiliser un compte de service ACR dédié avec des permissions de poussée uniquement
- Surveiller les journaux d'accès ACR pour détecter les accès anormaux
- Ne pas afficher les identifiants dans les journaux
- Ne pas commiter les identifiants dans le dépôt de code

### 2. Optimisation des performances
- Ajuster le paramètre `--proc` : régler la concurrence en fonction de la bande passante réseau (recommandé `5-20`)
- Surveiller le temps de synchronisation : s'il est trop long, envisagez de réduire la concurrence
- Nettoyer régulièrement les journaux : définir une valeur `retention-days` raisonnable (`7` jours dans la configuration actuelle)

### 3. Dépannage

#### Problème 1 : échec de l'authentification
```
Error: failed to authenticate to hagicode.azurecr.io
```
Solution :
1. Vérifier que les GitHub Secrets sont correctement configurés
2. Vérifier que le mot de passe Azure ACR n'a pas expiré
3. Confirmer que les permissions du compte de service ACR sont correctes

#### Problème 2 : délai d'attente réseau
```
Error: timeout waiting for response
```
Solution :
1. Vérifier la connectivité réseau
2. Réduire le nombre de threads concurrents (paramètre `--proc`)
3. Attendre la récupération du réseau et déclencher à nouveau le workflow

#### Problème 3 : synchronisation d'images incomplète
```
Warning: some tags failed to sync
```
Solution :
1. Consulter les journaux de synchronisation pour identifier les étiquettes ayant échoué
2. Déclencher manuellement le workflow pour synchroniser à nouveau
3. Vérifier que l'image source sur Docker Hub fonctionne correctement

### 4. Surveillance et alertes
- Vérifier régulièrement la page `Actions` pour confirmer l'état d'exécution des workflows
- Configurer les notifications GitHub pour recevoir rapidement les alertes d'échec de workflow
- Surveiller l'utilisation du stockage Azure ACR
- Vérifier régulièrement la cohérence des étiquettes d'images

## Questions fréquentes et solutions

### Q1 : Comment synchroniser des étiquettes spécifiques au lieu de toutes les étiquettes ?

Modifiez le fichier de configuration `images.yaml` :
```yaml
# Synchroniser uniquement les étiquettes latest et v1.0
docker.io/newbe36524/hagicode:latest: hagicode.azurecr.io/hagicode:latest
docker.io/newbe36524/hagicode:v1.0: hagicode.azurecr.io/hagicode:v1.0
```

### Q2 : Comment synchroniser plusieurs dépôts d'images ?

Ajoutez plusieurs lignes dans `images.yaml` :
```yaml
docker.io/newbe36524/hagicode: hagicode.azurecr.io/hagicode
docker.io/newbe36524/another-image: hagicode.azurecr.io/another-image
```

### Q3 : Comment réessayer après un échec de synchronisation ?
- Réessai automatique : image-syncer inclut un mécanisme de réessai intégré (`3` fois par défaut)
- Réessai manuel : cliquez sur `Re-run all jobs` sur la page GitHub Actions

### Q4 : Comment consulter la progression détaillée de la synchronisation ?
- Consulter les journaux en temps réel sur la page `Actions`
- Télécharger l'artefact `sync-logs` pour voir le fichier journal complet
- Le fichier journal contient l'état de synchronisation et la vitesse de transfert pour chaque étiquette

### Q5 : Combien de temps prend la synchronisation ?
- Synchronisation complète initiale : généralement `10-30` minutes selon la taille des images
- Synchronisation incrémentale : généralement `2-5` minutes si les modifications d'images sont minimes
- Le temps dépend de la bande passante réseau, de la taille des images et des paramètres de concurrence


```yaml
- name: Generate report
  if: always()

    echo "## Sync Report" >> $GITHUB_STEP_SUMMARY

    echo "- Sync time: ${{ steps.sync.outputs.duration }}" >> $GITHUB_STEP_SUMMARY
```
