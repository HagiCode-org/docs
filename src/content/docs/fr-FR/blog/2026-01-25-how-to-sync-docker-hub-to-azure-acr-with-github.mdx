## Implémenter la synchronisation automatique des images de Docker Hub vers Azure ACR

> Cet article présente comment utiliser GitHub Actions et l'outil image-syncer pour synchroniser automatiquement les images de Docker Hub vers Azure Container Registry, résolvant les problèmes de lenteur d'accès à Docker Hub en Chine et dans certaines régions Azure, améliorant ainsi la disponibilité des images et l'efficacité du déploiement dans les environnements Azure.

## Contexte/Introduction

Le projet HagiCode utilise des images Docker comme composant d'exécution principal, avec les images principales hébergées sur Docker Hub. Avec le développement du projet et l'augmentation des besoins de déploiement dans les environnements Azure, nous avons rencontré les problèmes suivants :
- Vitesse de tirage d'images lente, Docker Hub étant restreint en accès en Chine et dans certaines régions Azure
- Risque de panne de point unique en dépendant d'une seule source d'images
- Meilleures performances réseau et expérience d'intégration avec Azure Container Registry dans les environnements Azure

Pour résoudre ces problèmes, nous devons établir un mécanisme de synchronisation d'images automatisé, synchronisant régulièrement les images de Docker Hub vers Azure ACR, garantissant que les utilisateurs peuvent obtenir des vitesses de tirage d'images plus rapides et une meilleure disponibilité dans les environnements Azure.

## Comparaison des solutions techniques

Lors de l'élaboration de la solution, nous avons comparé plusieurs approches techniques :

### 1. image-syncer (choix final)
- Synchronisation incrémentale : synchronise uniquement les couches d'images modifiées, réduisant considérablement le transfert réseau
- Reprise après interruption : peut reprendre la synchronisation après une interruption réseau
- Contrôle de la concurrence : supporte la configuration du nombre de threads parallèles, améliorant l'efficacité de la synchronisation des grandes images
- Gestion des erreurs complète : mécanisme de réessai en cas d'échec intégré (3 fois par défaut)
- Déploiement léger : fichier binaire unique, sans dépendances
- Support multi-registres : compatible avec Docker Hub, Azure ACR, Harbor, etc.

### 2. Docker CLI
- Ne supporte pas la synchronisation incrémentale : doit tirer le contenu complet de l'image à chaque fois
- Efficacité faible : transfert réseau important, temps long
- Simple à utiliser : utilise les commandes familières docker pull/push

### 3. Azure CLI
- Complexité élevée : nécessite la configuration de l'authentification Azure CLI
- Limitations fonctionnelles : la fonction az acr import est relativement limitée
- Intégration native : bonne intégration avec les services Azure

## Décisions de conception de l'architecture

### Décision 1 : Fréquence de synchronisation définie à quotidiennement à UTC 00:00
- Équilibre entre la fraîcheur des images et la consommation de ressources
- Évite les périodes de pointe d'activité, réduisant l'impact sur d'autres opérations
- Les images Docker Hub sont généralement mises à jour après la construction quotidienne

### Décision 2 : Synchroniser toutes les étiquettes d'images
- Maintenir une cohérence complète avec Docker Hub
- Fournir aux utilisateurs un choix flexible de versions
- Simplifier la logique de synchronisation, éviter les règles complexes de filtrage d'étiquettes

### Décision 3 : Utiliser GitHub Secrets pour stocker les informations d'authentification
- Support natif de GitHub Actions, sécurité élevée
- Configuration simple, facile à gérer et maintenir
- Support du contrôle d'accès au niveau du dépôt

## Évaluation des risques et atténuation

### Risque 1 : Divulgation des informations d'authentification Azure ACR
- Stockage chiffré avec GitHub Secrets
- Rotation régulière des mots de passe ACR
- Limitation des permissions de l'utilisateur ACR à push uniquement
- Surveillance des journaux d'accès ACR

### Risque 2 : Échec de synchronisation entraînant une incohérence des images
- Mécanisme de synchronisation incrémentale intégré à image-syncer
- Réessai automatique en cas d'échec (3 fois par défaut)
- Journaux d'erreur détaillés et notifications d'échec
- Fonction de reprise après interruption

### Risque 3 : Consommation de ressources excessive
- Synchronisation incrémentale réduisant le transfert réseau
- Nombre de threads parallèles configurables (actuellement défini à 10)
- Surveillance du nombre et de la taille des images synchronisées
- Exécution de la synchronisation hors des heures de pointe

## Solution principale

Nous adoptons la solution d'automatisation GitHub Actions + image-syncer pour réaliser la synchronisation d'images de Docker Hub vers Azure ACR.

## Étapes de mise en œuvre

### 1. Phase de préparation
- Créer ou confirmer Azure Container Registry dans le portail Azure
- Créer une clé d'accès ACR (nom d'utilisateur et mot de passe)
- Confirmer les droits d'accès au dépôt d'images Docker Hub

### 2. Configuration des GitHub Secrets
Ajoutez les Secrets suivants dans les paramètres du dépôt GitHub :
- AZURE_ACR_USERNAME : Nom d'utilisateur Azure ACR
- AZURE_ACR_PASSWORD : Mot de passe Azure ACR

### 3. Créer le workflow GitHub Actions
Configurez le workflow dans .github/workflows/sync-docker-acr.yml :
- Déclenchement programmé : tous les jours à UTC 00:00
- Déclenchement manuel : support de workflow_dispatch
- Déclenchement supplémentaire : déclenché lors du push vers la branche publish (pour synchronisation rapide)

### 4. Flux d'exécution du workflow

| Ordre | Participant | Action | Description |
| --- | --- | --- | --- |
| 1 | GitHub Actions | Déclencher le workflow | Déclenché par programmation, manuellement ou push vers la branche `publish` |
| 2 | GitHub Actions → image-syncer | Télécharger et exécuter l'outil de synchronisation | Entrer dans la phase de synchronisation effective |
| 3 | image-syncer → Docker Hub | Obtenir le manifeste et la liste des étiquettes | Lire les métadonnées du dépôt source |
| 4 | image-syncer → Azure ACR | Obtenir les informations des images existantes | Déterminer l'état actuel côté cible |
| 5 | image-syncer | Comparer les différences source et cible | Identifier les couches d'images à synchroniser |
| 6 | image-syncer → Docker Hub | Tirer les couches d'images modifiées | Transférer uniquement le contenu à mettre à jour |
| 7 | image-syncer → Azure ACR | Pousser les couches d'images modifiées | Compléter la synchronisation incrémentale |
| 8 | image-syncer → GitHub Actions | Retourner les statistiques de synchronisation | Incluant les résultats, différences et informations d'erreur |
| 9 | GitHub Actions | Enregistrer les journaux et télécharger l'artifact | Faciliter l'audit et le dépannage ultérieur |

## Implémentation du workflow GitHub Actions

Voici la configuration du workflow réellement exécutée (.github/workflows/sync-docker-acr.yml) :

```yaml
name: Sync Docker Image to Azure ACR

on:
  schedule:
    - cron: "0 0 * * *" # Tous les jours à UTC 00:00
  workflow_dispatch: # Déclenchement manuel
  push:
    branches: [publish]

permissions:
  contents: read

jobs:
  sync:
    runs-on: ubuntu-latest

    steps:
      - name: Checkout code
        uses: actions/checkout@v4

      - name: Download image-syncer
        run: |
          # Télécharger le binaire image-syncer
          wget https://github.com/AliyunContainerService/image-syncer/releases/download/v1.5.5/image-syncer-v1.5.5-linux-amd64.tar.gz
          tar -zxvf image-syncer-v1.5.5-linux-amd64.tar.gz
          chmod +x image-syncer

      - name: Create auth config
        run: |
          # Générer le fichier de configuration d'authentification (format YAML)
          cat > auth.yaml <<EOF
          hagicode.azurecr.io:
            username: "${{ secrets.AZURE_ACR_USERNAME }}"
            password: "${{ secrets.AZURE_ACR_PASSWORD }}"
          EOF

      - name: Create images config
        run: |
          # Générer le fichier de configuration de synchronisation d'images (format YAML)
          cat > images.yaml <<EOF
          docker.io/newbe36524/hagicode: hagicode.azurecr.io/hagicode
          EOF

      - name: Run image-syncer
        run: |
          # Exécuter la synchronisation (utilisant les nouveaux paramètres --auth et --images)
          ./image-syncer --auth=./auth.yaml --images=./images.yaml --proc=10 --retries=3

      - name: Upload logs
        if: always()
        uses: actions/upload-artifact@v4
        with:
          name: sync-logs
          path: image-syncer-*.log
          retention-days: 7
```

## Explication de la configuration

### 1. Conditions de déclenchement
- Déclenchement programmé : cron: "0 0 * * *" - Exécuté tous les jours à UTC 00:00
- Déclenchement manuel : workflow_dispatch - Permet aux utilisateurs d'exécuter manuellement dans l'interface GitHub
- Déclenchement par push : push: branches: [publish] - Déclenché lors du push vers la branche publish (pour synchronisation rapide)

### 2. Configuration d'authentification (auth.yaml)
```yaml
hagicode.azurecr.io:
  username: "${{ secrets.AZURE_ACR_USERNAME }}"
  password: "${{ secrets.AZURE_ACR_PASSWORD }}"
```

### 3. Configuration de synchronisation d'images (images.yaml)
```yaml
docker.io/newbe36524/hagicode: hagicode.azurecr.io/hagicode
```
Cette configuration indique de synchroniser toutes les étiquettes de docker.io/newbe36524/hagicode vers hagicode.azurecr.io/hagicode

### 4. Paramètres image-syncer
- --auth=./auth.yaml : Chemin du fichier de configuration d'authentification
- --images=./images.yaml : Chemin du fichier de configuration de synchronisation d'images
- --proc=10 : Nombre de threads parallèles à 10
- --retries=3 : Réessai 3 fois en cas d'échec

## Précautions et meilleures pratiques

### 1. Recommandations de sécurité
- Faire une rotation régulière du mot de passe Azure ACR (recommandé tous les 90 jours)
- Utiliser un compte de service ACR dédié, limiter les permissions à push uniquement
- Surveiller les journaux d'accès ACR, détecter rapidement les accès anormaux
- Ne pas afficher les informations d'authentification dans les journaux
- Ne pas commiter les informations d'authentification dans le dépôt de code

### 2. Optimisation des performances
- Ajuster le paramètre --proc : ajuster le nombre de parallèles en fonction de la bande passante réseau (recommandé 5-20)
- Surveiller le temps de synchronisation : si le temps de synchronisation est trop long, envisager de réduire le nombre de parallèles
- Nettoyer régulièrement les journaux : définir un retention-days raisonnable (actuellement 7 jours)

### 3. Dépannage

#### Problème 1 : Échec d'authentification
```
Error: failed to authenticate to hagicode.azurecr.io
```
Solution :
1. Vérifier que les GitHub Secrets sont correctement configurés
2. Vérifier que le mot de passe Azure ACR n'a pas expiré
3. Confirmer que les permissions du compte de service ACR sont correctes

#### Problème 2 : Timeout réseau
```
Error: timeout waiting for response
```
Solution :
1. Vérifier la connexion réseau
2. Réduire le nombre de threads parallèles (paramètre --proc)
3. Attendre la récupération du réseau et redéclencher le workflow

#### Problème 3 : Synchronisation d'images incomplète
```
Warning: some tags failed to sync
```
Solution :
1. Vérifier les journaux de synchronisation, identifier les étiquettes échouées
2. Déclencher manuellement le workflow pour resynchroniser
3. Vérifier que l'image source Docker Hub est normale

### 4. Surveillance et alertes
- Vérifier régulièrement la page Actions, confirmer l'état d'exécution du workflow
- Configurer les notifications GitHub, obtenir rapidement les notifications d'échec du workflow
- Surveiller l'utilisation du stockage Azure ACR
- Vérifier régulièrement la cohérence des étiquettes d'images

## Questions fréquentes et solutions

### Q1 : Comment synchroniser des étiquettes spécifiques plutôt que toutes les étiquettes ?

Modifiez le fichier de configuration images.yaml :
```yaml
# Synchroniser uniquement les étiquettes latest et v1.0
docker.io/newbe36524/hagicode:latest: hagicode.azurecr.io/hagicode:latest
docker.io/newbe36524/hagicode:v1.0: hagicode.azurecr.io/hagicode:v1.0
```

### Q2 : Comment synchroniser plusieurs dépôts d'images ?

Ajoutez plusieurs lignes de configuration dans images.yaml :
```yaml
docker.io/newbe36524/hagicode: hagicode.azurecr.io/hagicode
docker.io/newbe36524/another-image: hagicode.azurecr.io/another-image
```

### Q3 : Comment réessayer après un échec de synchronisation ?
- Réessai automatique : mécanisme de réessai intégré à image-syncer (3 fois par défaut)
- Réessai manuel : cliquez sur Re-run all jobs dans la page GitHub Actions

### Q4 : Comment voir la progression détaillée de la synchronisation ?
- Voir les journaux en temps réel sur la page Actions
- Télécharger l'artifact sync-logs pour voir le fichier de journal complet
- Le fichier de journal contient l'état de synchronisation et la vitesse de transfert de chaque étiquette

### Q5 : Combien de temps prend la synchronisation ?
- Première synchronisation complète : selon la taille de l'image, généralement 10-30 minutes
- Synchronisation incrémentale : si les modifications de l'image sont faibles, généralement 2-5 minutes
- Le temps dépend de la bande passante réseau, de la taille de l'image et des paramètres de parallélisme
