## VSCode et code-server : choix de solutions d'édition de code dans le navigateur

> Lors de la construction de capacités d'édition de code côté navigateur, les développeurs sont confrontés à un choix critique : utiliser la fonction officielle `code serve-web` de VSCode ou adopter la solution `code-server` pilotée par la communauté ? Ce choix affecte non seulement l'architecture technique, mais aussi la conformité des licences et la flexibilité du déploiement.

## Contexte

En réalité, faire un choix technologique ressemble un peu au choix d'un chemin de vie. Une fois que vous avez choisi une route, vous devez la suivre jusqu'au bout, et si vous voulez changer de route, le coût devient énorme.

À l'ère de la programmation assistée par l'IA, la capacité d'édition de code côté navigateur devient de plus en plus importante. Les utilisateurs s'attendent à ce qu'après que l'assistant IA ait analysé le code, ils puissent immédiatement ouvrir l'éditeur dans la même session de navigateur pour apporter des modifications, sans avoir à changer d'application. Cette expérience transparente, comment dire, c'est comme quand vous y pensez, elle est là - mais parfois elle ne l'est tout simplement pas.

Cependant, dans la mise en œuvre de cette fonctionnalité, les développeurs sont confrontés à un choix technique critique : utiliser la fonction officielle `code serve-web` de VSCode ou adopter la solution `code-server` pilotée par la communauté ?

Ces deux solutions ont chacune leurs avantages et inconvénients, et un mauvais choix pourrait causer beaucoup de problèmes plus tard. Par exemple, les problèmes de licence - découvrir que la licence n'est pas conforme une fois le produit en ligne, il serait alors trop tard. C'est un peu comme dans une relation amoureuse, si on ne réfléchit pas clairement au début, on découvre à la fin que les valeurs des deux personnes sont fondamentalement incompatibles, et le prix à payer est élevé. Un autre exemple est le mode de déploiement - une solution fonctionne très bien dans l'environnement de développement, mais une fois dans un conteneur, tous sorts de problèmes surviennent. Personne ne veut tomber dans ces pièges, après tout, après en avoir trop rencontrés, on finit par s'y habituer.

## À propos de HagiCode

La solution partagée dans cet article provient de notre expérience pratique dans le projet [HagiCode](https://hagicode.com). HagiCode est un assistant de code piloté par l'IA. Lors de la mise en œuvre de la capacité d'édition de code côté navigateur, nous avons étudié en profondeur ces deux solutions, et finalement dans la conception de l'architecture, nous prenons en charge les deux, mais par défaut nous privilégions code-server.

Adresse du projet : [github.com/HagiCode-org/site](https://github.com/HagiCode-org/site)

## Différences de licence (le plus critique)

C'est la différence la plus fondamentale entre les deux solutions, et c'est aussi le premier facteur que nous avons considéré lors de notre choix. Après tout, pour un choix technologique, la première étape consiste à bien comprendre les risques juridiques, sinon quand des problèmes surviennent plus tard, de qui se plaindre ?

**code-server**
- Licence MIT, entièrement open source
- Maintenu par la société Coder.com, communauté active
- Peut être librement utilisé à des fins commerciales, modifié et distribué
- Aucune restriction sur les scénarios d'utilisation

**VSCode `code serve-web`**
- Fait partie du produit Microsoft VSCode
- Utilise la licence Microsoft (la licence VS Code a des restrictions d'utilisation commerciale)
- Principalement destiné aux développeurs individuels
- Le déploiement au niveau de l'entreprise peut nécessiter une autorisation commerciale supplémentaire

Du point de vue de la licence, code-server est plus favorable aux projets commerciaux. Ce point doit être clarifié dès la phase de planification du produit, sinon lorsque l'échelle augmentera et que vous devrez migrer, le coût sera énorme. Après tout, la migration, c'est facile à dire, mais difficile à faire, ceux qui l'ont vécu le savent.

## Différences de mode de déploiement

Une fois le problème de la licence résolu, vient ensuite le mode de déploiement. Cela affecte directement vos coûts d'exploitation et votre conception d'architecture, et affecte indirectement votre humeur quotidienne - plus le déploiement est simple, meilleure est l'humeur, tout le monde le comprend.

**code-server**
- Application Node.js indépendante, peut être déployée seule
- Prend en charge plusieurs sources d'exécution :
  - Spécifier directement le chemin de l'exécutable
  - Recherche dans le PATH du système
  - Détection automatique de l'environnement NVM Node.js 22.x
- Aucune installation de la version bureau de VSCode requise sur le serveur
- Déploiement en conteneur plus simple

**VSCode `code serve-web`**
- Doit dépendre de VSCode CLI installé localement
- Nécessite une commande `code` disponible sur la machine locale
- Le système filtrera le wrapper VS Code Remote CLI
- Principalement conçu pour les scénarios de développement local

code-server est plus adapté aux scénarios de déploiement sur serveur/conteneur. Si votre produit doit fonctionner dans Docker, ou si l'environnement de l'utilisateur n'a pas VSCode, alors choisissez code-server. Après tout, simple est beau, compliqué conduit facilement à des problèmes, et une fois les problèmes apparus, il faut les réparer, et après réparation, de nouveaux problèmes peuvent être introduits, ce cycle sans fin, qui voudrait le vivre ?

## Différences de paramètres fonctionnels

Les deux solutions présentent également certaines différences dans les paramètres fonctionnels, bien que mineures, elles peuvent causer des problèmes dans l'utilisation réelle. Ces détails ressemblent aux petits frottements de la vie, pas beaucoup, mais quand ils s'accumulent, ils deviennent ennuyeux.

| Caractéristique | code-server | code serve-web |
|------|-------------|----------------|
| Chemin de base public | `/` (configurable) | `/vscode-server` (fixe) |
| Méthode d'authentification | Paramètre `--auth`, prend en charge plusieurs modes | `--connection-token` / `--without-connection-token` |
| Répertoire de données | `{DataDir}/code-server` | `{DataDir}/vscode-serve-web` |
| Télémétrie | Désactivée par défaut `--disable-telemetry` | Dépend des paramètres VSCode |
| Vérification des mises à jour | Peut être désactivée `--disable-update-check` | Dépend des paramètres VSCode |

Ces différences nécessitent une attention particulière lors de l'intégration. Par exemple, la différence des chemins URL signifie que le code frontal doit être traité spécifiquement. Les développeurs le savent, ce genre de détails prend le plus de temps à traiter, mais il n'y a pas le choix, si on ne le fait pas, ça ne marche pas.

## Différences de détection de disponibilité

Dans la mise en œuvre de la fonction de basculement d'éditeur, la logique de détection de disponibilité est également différente. Cette différence ressemble à la façon dont les gens interagissent, certains aiment aller droit au but, d'autres préfèrent être subtils et implicites.

**code-server**
- Toujours renvoyé comme implémentation visible
- Même si indisponible, il s'affiche et invite avec l'état `install-required`
- Prend en charge la détection automatique de l'environnement NVM Node.js 22.x

**code serve-web**
- Visible uniquement si le CLI `code` local est détecté
- Si indisponible, le frontend masquera automatiquement cette option
- Dépend de l'état d'installation VSCode local

Cette différence affecte directement l'expérience utilisateur. L'approche de code-server est plus transparente, les utilisateurs savent qu'il y a cette option, elle n'est juste pas encore installée ; l'approche de code serve-web est plus cachée, les utilisateurs peuvent même ne pas savoir qu'il y a ce choix. Quelle approche est la meilleure ? Cela dépend du positionnement du produit, après tout pour l'expérience utilisateur, il n'y a pas de réponse standard, seulement ce qui convient ou non.

## Architecture double implémentation de HagiCode

Après une analyse approfondie, le projet HagiCode a adopté une architecture double implémentation, prenant en charge les deux solutions au niveau de l'architecture. Ce n'est pas que nous avons du mal à choisir technologiquement, mais il y a vraiment des besoins réels. Après tout, dans le monde technologique, il n'y a pas de choix absolument correct, seulement celui qui vous convient le mieux.

### Choix par défaut de code-server

```csharp
// L'implémentation active par défaut est code-server
// Si un activeImplementation explicite est enregistré, essayez d'abord cette implémentation
// Si l'implémentation demandée n'est pas disponible, le résolveur essaiera l'autre implémentation
// En cas de repli, renvoie fallbackReason
```

Nous choisissons par défaut code-server, principalement en considérant la licence et la flexibilité du déploiement. Mais pour les utilisateurs avec un environnement VSCode local, code serve-web est aussi un bon choix. Après tout, donner aux utilisateurs un choix supplémentaire est toujours une bonne chose, forcer les autres à accepter une solution unique, c'est quelque chose que je ne peux pas faire.

### Sélecteur d'implémentation

`CodeServerImplementationResolver` est uniformément responsable de :
- Sélection de l'implémentation lors du préchauffage au démarrage
- Sélection de l'implémentation lors de la lecture de l'état
- Sélection de l'implémentation lors de l'ouverture du projet
- Sélection de l'implémentation lors de l'ouverture du Vault

Cette conception permet au système de répondre flexibly à différents scénarios, et les utilisateurs peuvent choisir l'implémentation la plus appropriée selon leur environnement. La conception flexible prend un peu plus de temps au début, mais fait gagner du temps plus tard, après tout personne ne veut modifier du code partout.

### Règles d'adaptation frontend

```typescript
// Quand localCodeAvailable=false, ne pas afficher code serve-web
// Quand localCodeAvailable=true, afficher la configuration code serve-web
```

Le frontend affiche automatiquement les options disponibles selon l'environnement, évitant que les utilisateurs ne voient des fonctionnalités inutilisables et soient confus. Si les utilisateurs sont confus, ils viennent vous demander, trop de questions deviennent ennuyeux, trop d'ennui fait vouloir modifier le code, modifier le code peut introduire des bugs, ce cercle vicieux, qui veut le vivre ?

## Guide pratique

Après avoir tant parlé de théorie, à quoi faut-il faire attention lors du déploiement réel ? En fait, peu importe à quel point la théorie est bonne, si la mise en œuvre ne marche pas, ça ne sert à rien, après tout la pratique est le seul critère de vérité.

### Recommandations de déploiement Docker

Pour le déploiement en conteneur, code-server est le meilleur choix :

```dockerfile
# Utiliser directement l'image officielle code-server
FROM codercom/code-server:latest

# Ou installer via npm
RUN npm install -g code-server
```

Une seule couche suffit, pas besoin d'installer VSCode supplémentaire. Simple est bon, compliqué conduit facilement à des erreurs, ce principe s'applique partout.

### Exemples de configuration

**Configuration code-server**
```json
{
  "vscodeServer": {
    "enabled": true,
    "activeImplementation": "code-server",
    "codeServer": {
      "host": "0.0.0.0",
      "port": 8080,
      "executablePath": "",
      "authMode": "none"
    }
  }
}
```

**Configuration code serve-web**
```json
{
  "vscodeServer": {
    "enabled": true,
    "activeImplementation": "serve-web",
    "serveWeb": {
      "host": "0.0.0.0",
      "port": 8080,
      "executablePath": "/usr/local/bin/code"
    }
  }
}
```

La configuration, la première fois c'est un peu ennuyeux, une fois configurée, ensuite ça fait gagner du temps. Comme dans la vie, investir un peu plus au début, rend la vie plus facile ensuite.

### Différences de construction d'URL

**code-server**
```
http://localhost:8080/?folder=/path/to/project&vscode-lang=fr-FR
```

**code serve-web**
```
http://localhost:8080/vscode-server/?folder=/path/to/project&tkn=xxx&vscode-lang=fr-FR
```

Notez les différences de chemin et de paramètres, ils doivent être traités séparément lors de l'intégration. Les détails décident du succès ou de l'échec, ce n'est pas du tout faux, il manque un paramètre, et la page peut ne pas s'ouvrir.

### Changer d'implémentation

Le système prend en charge le changement d'exécution, lors du changement l'ancienne implémentation s'arrête automatiquement :

```csharp
// VsCodeServerManager gère automatiquement l'exclusion mutuelle
// Lors du changement d'activeImplementation, l'ancienne implémentation ne continue pas à fonctionner en arrière-plan
```

Cette conception permet aux utilisateurs d'essayer différentes implémentations à tout moment, trouvant la solution qui leur convient le mieux. Après tout, ce qui vous convient est le meilleur, les suggestions des autres ne sont que des références, finalement vous devez essayer vous-même.

### Surveillance de l'état

```typescript
const { settings, runtime } = await getVsCodeServerSettings();

// runtime.activeImplementation: "code-server" | "serve-web"
// runtime.fallbackReason: raison du basculement
// runtime.status: "running" | "starting" | "stopped" | "unhealthy"
```

État visible, on sait ce qui se passe. Les utilisateurs peuvent rapidement localiser s'il s'agit d'un problème serveur ou de leur propre opération lorsqu'ils rencontrent des problèmes. Quand on ne connaît pas l'état, on panique facilement, quand on panique, on fait facilement des erreurs de jugement, une fois cette chaîne démarrée, elle ne s'arrête plus.

## Résumé

| Dimension de comparaison | code-server | code serve-web | Recommandé |
|---------|-------------|----------------|------|
| Licence | MIT (favorable au commercial) | Microsoft (restrictions) | code-server |
| Flexibilité de déploiement | Déploiement indépendant | Dépend de VSCode local | code-server |
| Adéquation au serveur | Conçu pour les serveurs | Principalement pour le développement local | code-server |
| Conteneurisation | Support natif | Nécessite d'installer VSCode | code-server |
| Complétude fonctionnelle | Proche de la version bureau | Version officielle complète | code serve-web |
| Activité de maintenance | Communauté active | Officiel Microsoft | Chacun ses avantages |

**Stratégie recommandée** : Prioriser l'utilisation de code-server, envisager code serve-web lorsque des fonctionnalités officielles complètes sont nécessaires et qu'un environnement VSCode local est disponible.
