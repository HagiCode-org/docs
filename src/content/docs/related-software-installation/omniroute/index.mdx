本页面向第一次接触 OmniRoute 的 HagiCode 用户：先用最短路径把本地服务跑起来，再验证 `http://localhost:20128/v1` 可用，最后把 HagiCode 或其他 OpenAI 兼容工具接过去。

:::tip[推荐路径]
首版文档只把**最稳定的推荐路径**放在前面：`npm install -g omniroute`、`omniroute`、Dashboard `http://localhost:20128`、OpenAI 兼容 Base URL `http://localhost:20128/v1`。Docker、VPS 和更高级的部署方式放在后面，作为可选路径。
:::

## 安装前准备

- 本机已安装 Node.js。建议先完成 [安装 Node.js](/related-software-installation/nodejs/installation)。
- 预留可访问的本地端口 `20128`，避免被其他服务占用。
- 准备至少一个你要通过 OmniRoute 统一接入的上游模型提供方账号。
- 计划接入 HagiCode、CLI 或 SDK 时，先把“Dashboard 地址”和“API Base URL”区分清楚：
  - Dashboard：`http://localhost:20128`
  - OpenAI 兼容 Base URL：`http://localhost:20128/v1`

## 本地安装与启动

推荐先走全局 npm 安装路径：

```bash
npm install -g omniroute
omniroute
```

启动后，按下面的顺序检查：

1. 打开浏览器访问 `http://localhost:20128`，确认 Dashboard 可以正常加载。
2. 在 Dashboard 中进入 `Endpoints` 页面，创建或复制一个 API Key。
3. 记下稍后要提供给工具的 Base URL：`http://localhost:20128/v1`。

:::note[为什么先看 Dashboard]
首次接入时，Dashboard 是确认 OmniRoute 已启动、Provider 已连接、API Key 已生成的最快路径。后续工具调用如果失败，先回到 Dashboard 看运行状态，通常比直接排查 SDK 配置更快。
:::

## 启动后验证

### 验证一：确认模型目录可读

先把刚才在 `Endpoints` 页面拿到的 Key 放进环境变量：

```bash
export OMNIROUTE_API_KEY="替换为你在 Endpoints 页面复制的 Key"
```

然后请求 `/v1/models`：

```bash
curl http://localhost:20128/v1/models \
  -H "Authorization: Bearer ${OMNIROUTE_API_KEY}"
```

如果返回模型列表，说明 OpenAI 兼容端点和鉴权已经可用。

### 验证二：发一次最小聊天请求

```bash
curl http://localhost:20128/v1/chat/completions \
  -H "Authorization: Bearer ${OMNIROUTE_API_KEY}" \
  -H "Content-Type: application/json" \
  -d '{
    "model": "替换为 /v1/models 返回的模型 ID",
    "messages": [
      { "role": "user", "content": "Reply with ok." }
    ]
  }'
```

这一步的重点不是提示词，而是确认你选择的模型已经能从 OmniRoute 正常路由出去。

## 应用接入

### 你真正要填的四个值

| 项目 | 值 | 说明 |
| --- | --- | --- |
| Dashboard | `http://localhost:20128` | 用于查看运行状态、Provider、Endpoints。 |
| Base URL | `http://localhost:20128/v1` | 给 OpenAI 兼容客户端、CLI 或 SDK 使用。 |
| API Key | 来自 Dashboard `Endpoints` | 不要随便写占位符；先从 Dashboard 复制可用 Key。 |
| Model | 从 `/v1/models` 返回结果中选择 | 先请求模型目录，再把真实模型 ID 填到客户端。 |

### OpenAI SDK 最小示例

```ts
import OpenAI from "openai";

const client = new OpenAI({
  baseURL: "http://localhost:20128/v1",
  apiKey: process.env.OMNIROUTE_API_KEY,
});

const response = await client.chat.completions.create({
  model: process.env.OMNIROUTE_MODEL ?? "replace-with-a-model-id-from-v1-models",
  messages: [{ role: "user", content: "Reply with ok." }],
});

console.log(response.choices[0]?.message?.content);
```

这类示例的关键只有两点：`baseURL` 指向 OmniRoute，`model` 使用 `/v1/models` 已经返回过的真实模型 ID。

### HagiCode 接入示例

如果你想把 HagiCode 也接到同一个 OmniRoute 实例，建议按下面两层配置理解：

1. **工具层**：在你要使用的 Agent CLI 或 OpenAI 兼容客户端里，填写：
   - Base URL：`http://localhost:20128/v1`
   - API Key：从 Dashboard `Endpoints` 页面复制
2. **HagiCode 共享集成层**：在 HagiCode 的配置面板中打开 `OmniRoute 集成` 选项卡，只填写 OmniRoute 的**基础 URL**，也就是 `http://localhost:20128`。

HagiCode 现有集成会基于这个基础 URL 做健康探测，并读取 OmniRoute 的 `/v1/models` 目录。也就是说，在 HagiCode 这一层不要手动把地址写成 `/v1/models`，系统会自动规范化。

如果你还没有准备好本地 Agent CLI，可以先看 [AI Agent CLI 安装](/related-software-installation/ai-agent-cli/)。

## 可选部署路径

推荐先把本地 npm 路径跑通，再考虑下面这些可选方案：

- **Docker**：适合已经有容器化习惯、希望把 OmniRoute 独立运行的环境。
- **VPS / 远端主机**：适合想把 OmniRoute 放到常驻节点，再让多台设备共用同一个 OpenAI 兼容端点的情况。
- **更高级的网关能力**：例如 MCP、A2A 或更复杂的 Provider 编排，应以 OmniRoute 上游文档为准，不建议在首次接入阶段同步处理。

:::note[什么时候再看可选路径]
只有在本地 `npm install -g omniroute` 路径已经验证成功之后，再迁移到 Docker 或远端部署。否则你很难判断问题出在 OmniRoute 本身、Provider 配置，还是网络与容器层。
:::

## 最佳实践

- 先把本地推荐路径跑通，再扩展到 Docker、VPS 或团队共享实例。
- 把 `/v1/models` 当成模型 ID 的唯一事实来源，不要凭记忆手填模型名。
- 区分 `http://localhost:20128` 和 `http://localhost:20128/v1`，前者偏运维与观察，后者给客户端请求使用。
- 如果 HagiCode 显示模型目录陈旧，优先回到 Dashboard 检查运行状态，再重新请求 `/v1/models`。
- 首次接入时尽量只改一个变量，例如先确认 API Key 正确，再切换模型或 Provider。
