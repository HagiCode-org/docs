## CodeBuddy 简介

这篇页面分成两部分：

1. 先按 CodeBuddy 官方文档完成 CLI 安装
2. 再补充 HagiCode 当前接入所需的 ACP 校验与平台配置

这样可以把“官方安装步骤”和“HagiCode 集成步骤”分开，避免把我们自己的表达覆盖掉官方原本的安装说明。

## 适用范围与官方来源

- **适用对象**：希望先把 CodeBuddy CLI 装好，再在 HagiCode 中启用它的用户
- **最后核对日期**：2026-03-13

## 先按官方文档完成安装

### 官方安装前提

根据 CodeBuddy 官方安装页，使用包管理器安装时需要：

- Node.js `18.20` 或更高版本
- 可用的终端环境，例如 PowerShell、Terminal、bash 或 zsh
- 正常的网络连接

如果你不想依赖 Node.js，官方安装页还提供了 **Native Binary（Beta）** 路径。

### 官方安装方式 1：包管理器

官方安装页提供了 `npm`、`pnpm`、`yarn`、`bun` 与 Homebrew 多种方式。最常见的命令如下：

```bash
npm install -g @tencent-ai/codebuddy-code
```

```bash
pnpm add -g @tencent-ai/codebuddy-code
```

```bash
brew install Tencent-CodeBuddy/tap/codebuddy-code
```

安装完成后，官方建议运行：

```bash
codebuddy --version
```

**预期结果**：终端返回版本号，说明 `codebuddy` 命令已经可以使用。

### 官方安装方式 2：原生二进制（Beta）

CodeBuddy 官方安装页还提供原生二进制安装方式，特点是：

- 单文件可执行，不依赖额外运行时
- 不需要 Node.js
- 自动更新体验更完整

官方给出的安装脚本为：

```bash
curl -fsSL https://copilot.tencent.com/cli/install.sh | bash
```

Windows PowerShell 对应脚本为：

```powershell
irm https://copilot.tencent.com/cli/install.ps1 | iex
```

官方安装页说明，原生二进制当前仍处于 **Beta** 阶段，但已经覆盖：

- macOS（Apple Silicon 与 Intel）
- Linux（arm64 / x86_64）
- Windows（x86_64）

如果执行后命令仍不可用，官方文档建议检查 PATH；macOS / Linux 通常需要确认 `$HOME/.local/bin` 已加入 PATH。

### 官方更新与常见恢复动作

官方安装页还给出了这两个后续命令：

```bash
codebuddy update
```

```bash
npm install -g @tencent-ai/codebuddy-code
```

前者用于按当前安装方式自动更新，后者是在包管理器安装场景下的手动恢复方法。

## HagiCode 额外需要的 ACP 校验

上面的内容已经足够完成官方意义上的安装；但如果你要在 HagiCode 中使用 CodeBuddy，还需要额外确认 ACP 能启动。

### 1) 验证 ACP 启动方式

CodeBuddy 官方 ACP 文档说明，外部工具可通过以下方式启动 CodeBuddy：

```bash
codebuddy --acp
```

在 HagiCode 场景中，我们建议你至少先确认本机对 `--acp` 参数没有报错。若需要快速做参数探测，也可以使用：

```bash
codebuddy --acp --help
```

### 2) 如使用 API Key，按官方 ACP 文档补环境变量

官方 ACP 文档给出的环境变量为：

```bash
export CODEBUDDY_API_KEY="your-api-key"
export CODEBUDDY_INTERNET_ENVIRONMENT="ioa"
```

Windows PowerShell 可写为：

```powershell
$env:CODEBUDDY_API_KEY = "your-api-key"
$env:CODEBUDDY_INTERNET_ENVIRONMENT = "ioa"
```

:::note
官方 ACP 文档特别说明：CodeBuddy `0.3.3.21` 及以上版本使用 `ioa`；更早的版本使用 `internal`。如果你的 CLI 版本较旧，请不要继续使用我们之前写错的 `public` 值。
:::

### 3) 验证 HagiCode 已识别 CodeBuddy

建议按下面顺序检查：

1. 先按官方安装页完成安装，并运行 `codebuddy --version`
2. 再按官方 ACP 文档确认 `codebuddy --acp` 可以启动
3. 启动 HagiCode，检查健康检查、Provider 状态页或启动日志
4. 确认没有出现“找不到 `codebuddy`”或 ACP 启动失败的错误
5. 发起一次最小请求，确认 `CodebuddyCli` 可以返回结果

## 常见问题排查

| 问题现象 | 更贴近官方文档的判断方式 | 建议处理步骤 |
|---|---|---|
| `codebuddy: command not found` | 官方安装页把这类问题归为 PATH 未生效 | 重新打开终端；检查 PATH；原生二进制场景确认 `$HOME/.local/bin` 或 Windows 用户目录下的安装路径已加入 PATH |
| `codebuddy --acp` 无法启动 | 官方安装页只覆盖安装，ACP 需额外参考官方 ACP 文档 | 先确认 CLI 已安装成功，再核对当前版本是否支持 ACP，并复查启动参数 |
| API Key 模式鉴权失败 | ACP 相关环境变量不完整或 `CODEBUDDY_INTERNET_ENVIRONMENT` 值不对 | 补齐 `CODEBUDDY_API_KEY`；`0.3.3.21` 及以上版本使用 `ioa`，旧版本使用 `internal` |
| HagiCode 检测不到 CodeBuddy | 终端里可用，但 HagiCode 启动时没有正确继承本机环境 | 先在同一台机器上确认 `codebuddy --version` 和 `codebuddy --acp` 正常，再检查 HagiCode 启动日志 |
| 包管理器更新失败 | 官方安装页建议重新执行安装命令 | 先运行 `codebuddy update`；失败后重新执行对应包管理器安装命令 |
