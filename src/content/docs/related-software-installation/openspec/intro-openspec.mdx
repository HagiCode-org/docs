# OpenSpec 介绍

OpenSpec 是一个规范驱动的开发工具，帮助团队通过结构化的提案和规范管理软件开发流程。本文将全面介绍 OpenSpec 的核心概念、工作流程和使用方法。

## 什么是 OpenSpec？

OpenSpec 是一个命令行工具，用于管理软件开发过程中的提案（Proposals）、规范（Specifications）和变更（Changes）。它将开发流程标准化，确保每个功能从想法到实现都经过充分的设计和审查。

### 核心理念

OpenSpec 基于以下核心理念：

- **规范先行**：在编写代码之前，先定义清晰的规范
- **结构化协作**：通过标准化的文档格式促进团队协作
- **可追溯性**：记录从提案到实现的完整过程
- **质量保证**：通过规范验证确保代码符合设计意图

### 主要功能

OpenSpec 提供以下主要功能：

1. **提案管理**：创建、审查和跟踪功能提案
2. **规范定义**：编写和维护技术规范文档
3. **变更追踪**：记录所有经过审查的变更历史
4. **规范验证**：确保代码变更符合预定义的规范
5. **团队协作**：支持多人协作开发和审查流程

## OpenSpec 工作流程

OpenSpec 的典型工作流程包括以下几个阶段：

### 1. 创建提案（Proposal）

当有新功能或重大变更需求时，首先创建一个提案：

```bash
openspec proposal create
```

提案包含：
- **问题描述**：要解决的问题或需求
- **解决方案**：提议的解决方案概述
- **影响范围**：对现有系统的影响
- **替代方案**：考虑过的其他方案

### 2. 编写规范（Specification）

提案通过后，编写详细的技术规范：

```bash
openspec spec create
```

规范包含：
- **技术设计**：详细的技术实现方案
- **接口定义**：API、数据结构等接口设计
- **实现细节**：关键实现逻辑和算法
- **测试计划**：如何验证实现的正确性

### 3. 创建变更（Change）

开始实现时，创建一个变更来跟踪开发过程：

```bash
openspec change create
```

变更包含：
- **关联规范**：该变更实现的规范
- **实现任务**：具体的开发任务列表
- **进度跟踪**：任务完成状态
- **代码变更**：实际的代码修改

### 4. 验证和归档

完成开发后，验证变更是否符合规范，然后归档：

```bash
openspec change validate
openspec change archive
```

## OpenSpec 目录结构

OpenSpec 在项目根目录下创建 `openspec/` 目录，结构如下：

```
openspec/
├── proposals/           # 提案目录
│   ├── PROPOSAL-001.md
│   └── PROPOSAL-002.md
├── specs/              # 规范目录
│   ├── SPEC-001.md
│   └── SPEC-002.md
├── changes/            # 变更目录
│   ├── feature-name/
│   │   ├── proposal.md
│   │   ├── design.md
│   │   ├── specs.md
│   │   └── tasks.md
│   └── another-feature/
└── archived/           # 已归档的变更
    └── feature-name/
```

### 目录说明

- **proposals/**：存放所有提案文档，每个提案一个独立的 Markdown 文件
- **specs/**：存放技术规范文档，与提案对应
- **changes/**：存放正在进行的变更，每个变更一个子目录
- **archived/**：存放已完成并归档的变更

## 核心命令

### 提案相关命令

```bash
# 创建新提案
openspec proposal create

# 列出所有提案
openspec proposal list

# 查看提案详情
openspec proposal view <proposal-id>

# 批准提案
openspec proposal approve <proposal-id>
```

### 规范相关命令

```bash
# 创建新规范
openspec spec create

# 列出所有规范
openspec spec list

# 查看规范详情
openspec spec view <spec-id>

# 更新规范
openspec spec update <spec-id>
```

### 变更相关命令

```bash
# 创建新变更
openspec change create

# 列出所有变更
openspec change list

# 查看变更详情
openspec change view <change-name>

# 添加任务
openspec change add-task <change-name>

# 验证变更
openspec change validate <change-name>

# 归档变更
openspec change archive <change-name>
```

### 通用命令

```bash
# 查看版本
openspec --version

# 查看帮助
openspec --help

# 查看特定命令帮助
openspec <command> --help
```

## 在 HagiCode 项目中使用 OpenSpec

### 项目集成

在 HagiCode 项目中，OpenSpec 已集成到开发工作流中：

1. **初始化检查**：启动向导会检查 OpenSpec 是否安装
2. **提案驱动开发**：所有新功能都需要通过 OpenSpec 提案流程
3. **代码审查**：变更需要符合 OpenSpec 规范才能合并
4. **文档同步**：OpenSpec 文档与代码实现保持同步

### 版本要求

HagiCode 项目要求使用 **OpenSpec >=1.0.0 <2.0.0** 版本范围。

安装命令：

```bash
npm install -g @fission-ai/openspec@1
```

### 工作流示例

以下是在 HagiCode 项目中使用 OpenSpec 的完整示例：

#### 步骤 1：创建提案

```bash
cd /path/to/hagicode-project
openspec proposal create
```

在交互式提示中输入：
- 提案标题：添加用户认证功能
- 问题描述：系统需要用户登录和权限管理
- 解决方案：实现基于 JWT 的认证系统

#### 步骤 2：编写规范

提案批准后，创建详细规范：

```bash
openspec spec create
```

在规范中定义：
- 认证流程设计
- API 接口定义
- 数据库表结构
- 安全考虑

#### 步骤 3：创建变更

开始实现：

```bash
openspec change create
```

输入变更名称：`add-user-authentication`

#### 步骤 4：添加任务

```bash
openspec change add-task add-user-authentication
```

添加具体任务：
- 实现用户注册接口
- 实现登录接口
- 实现 JWT 中间件
- 编写单元测试

#### 步骤 5：实现和验证

完成代码后，验证变更：

```bash
openspec change validate add-user-authentication
```

#### 步骤 6：归档

验证通过后，归档变更：

```bash
openspec change archive add-user-authentication
```

## 最佳实践

### 提案编写

1. **清晰的问题陈述**：明确要解决的问题，避免模糊描述
2. **考虑替代方案**：列出并比较不同的解决方案
3. **评估影响**：分析对现有系统的影响和风险
4. **保持简洁**：提案应该简明扼要，详细内容留给规范

### 规范编写

1. **详细但不冗余**：提供足够的细节，但避免过度设计
2. **使用图表**：用流程图、架构图等辅助说明
3. **定义接口**：清晰定义所有公共接口和数据结构
4. **考虑边界情况**：列出异常情况和错误处理

### 变更管理

1. **小步迭代**：将大功能拆分为多个小变更
2. **及时更新**：保持任务状态和文档的同步
3. **代码审查**：变更完成后进行代码审查
4. **及时归档**：完成的变更及时归档，保持工作目录整洁

### 团队协作

1. **定期审查**：定期审查提案和规范，确保团队共识
2. **文档先行**：先完成文档再开始编码
3. **保持沟通**：在提案和规范中记录讨论和决策
4. **版本控制**：将 OpenSpec 文档纳入版本控制

## 常见使用场景

### 场景 1：添加新功能

1. 创建提案描述新功能需求
2. 团队讨论并批准提案
3. 编写详细的技术规范
4. 创建变更并实现功能
5. 验证并归档变更

### 场景 2：重构现有代码

1. 创建提案说明重构原因和目标
2. 在规范中定义重构方案和步骤
3. 创建变更并逐步重构
4. 确保测试覆盖，验证重构结果
5. 归档变更

### 场景 3：修复 Bug

对于简单 Bug：
- 可以直接修复，不需要完整的 OpenSpec 流程

对于复杂 Bug：
1. 创建提案分析 Bug 根因
2. 在规范中定义修复方案
3. 创建变更并实现修复
4. 添加测试防止回归
5. 归档变更

### 场景 4：技术债务管理

1. 创建提案列出技术债务项
2. 在规范中定义改进计划
3. 创建变更逐步偿还技术债务
4. 跟踪进度和效果
5. 归档完成的改进

## 与其他工具集成

### Git 集成

OpenSpec 与 Git 无缝集成：

```bash
# OpenSpec 文档应纳入版本控制
git add openspec/
git commit -m "Add OpenSpec proposal for feature X"
```

### CI/CD 集成

在 CI/CD 流程中验证 OpenSpec 规范：

```yaml
# .github/workflows/openspec.yml
name: OpenSpec Validation
on: [pull_request]
jobs:
  validate:
    runs-on: ubuntu-latest
    steps:
      - uses: actions/checkout@v2
      - name: Install OpenSpec
        run: npm install -g @fission-ai/openspec@1
      - name: Validate changes
        run: openspec change validate
```

### IDE 集成

推荐使用支持 Markdown 的编辑器编辑 OpenSpec 文档：
- VS Code + Markdown 插件
- JetBrains IDEs + Markdown 支持
- Vim/Neovim + Markdown 插件

## 故障排除

### 常见问题

#### 问题：OpenSpec 命令未找到

**解决方案**：
1. 确认 OpenSpec 已安装：`npm list -g @fission-ai/openspec`
2. 检查 PATH 环境变量是否包含 npm 全局 bin 目录
3. 重新安装：`npm install -g @fission-ai/openspec@1`

#### 问题：无法创建提案或变更

**解决方案**：
1. 确认当前目录是项目根目录
2. 检查是否有 `openspec/` 目录，如没有则创建
3. 确认有写入权限

#### 问题：验证失败

**解决方案**：
1. 检查变更目录下的文档是否完整
2. 确认所有必需的任务都已完成
3. 查看详细错误信息：`openspec change validate <change-name> --verbose`

#### 问题：归档失败

**解决方案**：
1. 确保变更已通过验证
2. 检查 `openspec/archived/` 目录是否存在
3. 确认没有未提交的代码变更

## 进阶使用

### 自定义模板

OpenSpec 支持自定义提案和规范模板：

```bash
# 在项目根目录创建模板目录
mkdir -p .openspec/templates

# 创建自定义提案模板
cat > .openspec/templates/proposal.md << 'EOF'
# Proposal: {{title}}

## Problem Statement
{{problem}}

## Proposed Solution
{{solution}}

## Custom Section
Your custom content here
EOF
```

### 配置文件

在项目根目录创建 `.openspec.json` 配置文件：

```json
{
  "version": "1.0",
  "templates": {
    "proposal": ".openspec/templates/proposal.md",
    "spec": ".openspec/templates/spec.md"
  },
  "validation": {
    "requireTests": true,
    "requireDocs": true
  },
  "hooks": {
    "pre-archive": "npm test"
  }
}
```

### 钩子（Hooks）

OpenSpec 支持在特定操作前后执行自定义脚本：

```json
{
  "hooks": {
    "pre-create": "scripts/check-permissions.sh",
    "post-archive": "scripts/notify-team.sh"
  }
}
```
