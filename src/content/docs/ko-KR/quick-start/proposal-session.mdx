제안 세션은 요청이 너무 중요해서 단일 자유 형식 프롬프트로 처리할 수 없을 때 적합한 시작점입니다. HagiCode는 즉시 편집하는 대신 먼저 요청을 목표, 범위, 작업 및 검증 기준으로 전환한 다음 실행으로 이동합니다.

## 전제조건

제안 세션을 시작하기 전에 다음 사항을 확인하세요.

- 완료 [데스크탑 설치](/ko-KR/installation/desktop)
- 완료 [마법사 설정](/ko-KR/quick-start/wizard-setup) 하나 이상의 프로젝트를 만들었습니다.
- 변경 사항을 설명하기에 충분한 프로젝트 컨텍스트 그렇지 않다면 다음부터 시작하세요 [대화 세션 만들기](/ko-KR/quick-start/conversation-session)

## 제안 세션의 목적

제안 세션은 다음과 같은 경우에 가장 효과적입니다.

- 편집을 시작하기 전에 변경 내용에 대한 명확한 범위가 필요합니다.
- 여러 저장소, 모듈 또는 결과물이 관련됩니다.
- 작업은 검토 가능한 단계로 나누어야 합니다.
- 추론 및 실행 추적이 나중에 계속 표시되기를 원합니다.

저장소 이해나 간단한 토론만 필요한 경우 대화 세션이 더 빠릅니다. 구조와 추적성이 필요한 경우 여기에서 계속하세요.

## 워크플로우 개요

현재 제안 흐름은 다음과 같이 요약될 수 있습니다.

| 무대 | 사용자 조치 | 시스템 결과 |
| --- | --- | --- |
| 제안 만들기 | 열기 `New Idea` 서랍을 열고 요청을 설명하세요. | 제안은 명시적인 프로젝트 및 저장소 범위로 시작됩니다. |
| 구조를 확인하세요 | 세션 세부정보 보기 및 워크플로 상태 검토 | AI는 심층 실행 전에 목표, 단계 및 상태를 설명합니다. |
| 진행 상황 추적 | 세션 보드를 사용하여 보류, 활성 및 보관된 항목을 모니터링합니다. | 여러 제안을 더 쉽게 관리할 수 있습니다. |
| 결과 검토 | 완료된 세션 보기로 돌아가기 | 다시 시작하는 대신 커밋 메모와 세션 기록을 재사용할 수 있습니다. |

## 1단계: 변경 사항 정의 `New Idea` 서랍

현재 제안 진입점은 `New Idea` 서랍. 짧은 문자 요청을 받아들이는 것 이상의 역할을 합니다. 또한 요청을 구체적인 프로젝트 및 저장소 범위로 강제 적용합니다.

![프로젝트 선택, 저장소 범위 및 요청 입력이 포함된 새로운 아이디어 서랍](../../img/screenshots/shared/create-proposal-drawser/original.png)

다음 영역을 검토하는 것부터 시작하세요.

- **프로젝트 선택기**: 요청을 소유한 프로젝트를 확인하세요.
- **저장소 범위**: 경계 내에 있는 저장소를 선택하세요.
- **미리보기 영역**: 시스템이 대상 범위로 처리할 내용을 확인합니다.
- **요청 상자**: 자연어의 실제 변화를 설명합니다.

작업이 문서, 프런트엔드, 백엔드와 함께 관련되는 경우 실행 중에 늦게 추가하는 대신 여기에서 해당 경계를 정의하세요.

## 2단계: 제안 세부정보 보기에서 목표, 단계, 상태 검토

HagiCode는 생성 후 바로 편집으로 넘어가지 않습니다. 먼저 워크플로 상태, 제안 내용 및 세션 컨텍스트를 함께 유지하는 세부 정보 보기를 엽니다.

![워크플로 단계, 콘텐츠 패널 및 기록 사이드바가 포함된 제안 세션 세부정보 보기](../../img/screenshots/shared/classic-proposal-view/original.png)

이 화면은 제안 세션이 일반 대화 세션과 다른 이유를 보여줍니다.

- 중앙 패널에는 채팅만 있는 것이 아닙니다. 제안 관련 콘텐츠를 계속 표시합니다.
- 워크플로 스테퍼는 현재 상태를 명시적으로 만듭니다.
- 세션 목록 및 세부 정보 영역은 장기 실행 작업을 위한 컨텍스트를 보존합니다.

여기에서 목표, 범위 또는 작업 분류가 여전히 잘못된 것으로 보이면 계속 진행하기 전에 제안을 수정하세요. 이것이 이 워크플로의 핵심입니다.

## 3단계: 세션 보드에서 여러 제안 추적

프로젝트에 여러 제안, 여러 실행 라운드 또는 보류 중인 작업과 완료된 작업이 혼합되어 있는 경우 세션 보드가 가장 명확한 개요가 됩니다.

![보류 중, 진행 중 및 보관됨 열이 있는 세션 칸반 보기](../../img/screenshots/shared/kanban-view/original.png)

이 보기는 특히 다음 세 가지에 유용합니다.

- 아직 시작되지 않은 제안 확인
- 여러 활성 세션의 속도 비교
- 작업 공간을 계속 읽을 수 있도록 완료된 작업을 보관합니다.

일상적인 흐름에 여러 병렬 스레드가 포함된 경우 보드 보기는 한 번에 하나의 세부 정보 보기를 여는 것보다 더 나은 경우가 많습니다.

## 4단계: 완료된 실행 상태 다시 확인

제안이 끝날 무렵에는 일반적으로 단순한 "완료" 표시 이상의 것이 필요합니다. 또한 변경된 사항, 커밋 요약이 어떻게 구성되었는지, 동일한 컨텍스트가 다음 라운드까지 계속될 수 있는지 검토해야 합니다.

![커밋 메모, 주요 작업 및 세션 기록이 포함된 완료된 세션 보기](../../img/screenshots/shared/classic-view/original.png)

이 완료 상태 보기는 다음 두 가지 후속 작업에 유용합니다.

- 생성된 커밋 노트와 주요 실행 결과 검토
- 처음부터 모든 것을 다시 빌드하는 대신 동일한 세션 컨텍스트에서 계속

즉, 제안 세션은 일회성 생성기가 아닙니다. 지속 가능한 작업 흐름 기록입니다.

## 제안 세션을 선호하는 경우

다음과 같은 경우 먼저 제안 세션을 선택하세요.

- 변경 사항이 복잡하고 AI가 너무 일찍 즉석에서 작업하는 것을 원하지 않습니다.
- 작업은 나중에 검토되거나 다른 사람들과 공유됩니다.
- 여러 저장소, 모듈 또는 역할이 관련됩니다.
- 실행 기록과 근거를 재사용 가능하게 유지하고 싶습니다.

작업이 여전히 작고 탐구적인 경우 일반적으로 대화 세션이 더 가벼운 경로입니다.
