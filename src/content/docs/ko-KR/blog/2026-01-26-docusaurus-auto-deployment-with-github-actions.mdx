## HagiCode에 GitHub Pages 자동 배포 지원 추가

> 이 프로젝트의 초기 코드명은 PCode였으며, 이제 HagiCode로 공식적으로 개명되었습니다. 이 문서에서는 프로젝트에 자동화된 정적 사이트 배포 기능을 도입하여 콘텐츠 게시를 물 마시듯 간단하게 만드는 방법을 기록합니다.

## 배경/서론

HagiCode 개발 과정에서 우리는 매우 현실적인 문제에 직면했습니다. 문서와 제안이 점점越来越多아짐에 따라 이러한 콘텐츠를 효율적으로 관리하고 표시하는 것이 시급한 과제가 되었습니다. 우리는 GitHub Pages를 도입하여 정적 사이트를 호스팅하기로 결정했지만, 수동으로 빌드하고 배포하는 것이 정말 번거로웠습니다 — 매번 변경할 때마다 로컬에서 빌드하고 패키징한 다음 수동으로 `gh-pages` 브랜치에 푸시해야 했습니다. 이는 효율성이 낮을 뿐만 아니라 오류가 발생하기 쉬웠습니다.

이 문제를 해결하기 위해 (주로 게으름을 피하기 위해), 우리는 자동화된 배포 프로세스가 필요했습니다. 이 문서에서는 HagiCode 프로젝트에 GitHub Actions 자동 배포 지원을 추가하는 방법을 자세히 기록하여, 우리가 콘텐츠 창작에만 집중하고 나머지는 자동화 프로세스에 맡길 수 있도록 합니다.

> 안녕, 우리가 만들고 있는 것을 소개할게

우리는 **HagiCode**를 개발하고 있습니다 — AI 기반 코드 지능 어시스턴트로, 개발 경험을 더 스마트하고 편리하며 재미있게 만들어줍니다.

## 목표 분석

시작하기 전에 이번 작업이 정확히 무엇을 하는지 먼저 명확히 해야 합니다.毕竟 (여기서 오타가 났는데,毕竟이어야 함) 도끼를 갈면 나무를 베는데 지체되지 않는다죠.

### 핵심 요구사항

1.  **자동화된 빌드**: 코드가 `main` 브랜치에 푸시될 때 자동으로 빌드 프로세스를 트리거합니다.
2.  **자동 배포**: 빌드 성공 후 생성된 정적 파일을 GitHub Pages에 자동으로 배포합니다.
3.  **환경 일관성**: CI 환경과 로컬 빌드 환경이 일치하는지 확인하여 "로컬에서는 실행되지만 온라인에서는 오류가 발생"하는 당황스러운 상황을 피합니다.

### 기술 선택

HagiCode는 Docusaurus(매우 인기 있는 React 정적 사이트 생성기)를 기반으로 구축되었으므로 GitHub Actions를 활용하여 이 목표를 달성할 수 있습니다.

## GitHub Actions 워크플로우 구성

GitHub Actions는 GitHub가 제공하는 CI/CD 서비스입니다. 코드 저장소에서 YAML 형식의 워크플로우 파일을 정의하여 다양한 자동화 작업을 사용자 정의할 수 있습니다.

### 워크플로우 파일 생성

프로젝트 루트 디렉터리의 `.github/workflows` 폴더에 `deploy.yml`과 같은 새 구성 파일을 만들어야 합니다. 폴더가 없으면 먼저 수동으로 만드세요.

이 구성 파일의 핵심 논리는 다음과 같습니다:

1.  **트리거 조건**: `main` 브랜치의 `push` 이벤트를 수신합니다.
2.  **실행 환경**: 최신 버전의 Ubuntu.
3.  **빌드 단계**:
    -   코드 체크아웃
    -   Node.js 설치
    -   종속성 설치 (`npm install`)
    -   정적 파일 빌드 (`npm run build`)
4.  **배포 단계**: 공식적으로 제공되는 `action-gh-pages`를 사용하여 빌드 산출물을 `gh-pages` 브랜치로 푸시합니다.

### 핵심 구성 코드

다음은 우리가 최종적으로 채택한 구성 템플릿입니다:

```yaml
name: Deploy to GitHub Pages

# 트리거 조건: main 브랜치에 푸시될 때
on:
  push:
    branches:
      - main
    # 필요에 따라 경로 필터링을 추가할 수 있음, 예: 문서 변경시에만 빌드
    # paths:
    #   - 'docs/**'
    #   - 'package.json'

# 권한 설정, GitHub Pages 배포에 중요
permissions:
  contents: read
  pages: write
  id-token: write

# 동시성 제어: 동일 브랜치의 이전 빌드 취소
concurrency:
  group: "pages"
  cancel-in-progress: false

jobs:
  build:
    runs-on: ubuntu-latest
    steps:
      - name: Checkout
        uses: actions/checkout@v4
        # 주의: fetch-depth: 0을 설정해야 함, 그렇지 않으면 빌드 버전 번호가 부정확할 수 있음
        with:
          fetch-depth: 0

      - name: Setup Node
        uses: actions/setup-node@v4
        with:
          node-version: 20 # 로컬 개발 환경과 일치하는 것을 권장
          cache: 'npm'     # 캐시 활성화로 빌드 프로세스 가속화

      - name: Install dependencies
        run: npm ci
        # npm install 대신 npm ci 사용, 더 빠르고 엄격하며 CI 환경에 적합

      - name: Build website
        run: npm run build
        env:
          # 사이트 빌드에 환경 변수가 필요한 경우 여기에 구성
          # NODE_ENV: production
          # PUBLIC_URL: /your-repo-name
          
      - name: Upload artifact
        uses: actions/upload-pages-artifact@v3
        with:
          path: ./build # Docusaurus 기본 출력 디렉터리

  deploy:
    environment:
      name: github-pages
      url: ${{ steps.deployment.outputs.page_url }}
    runs-on: ubuntu-latest
    needs: build
    steps:
      - name: Deploy to GitHub Pages
        id: deployment
        uses: actions/deploy-pages@v4
```

## 구현 과정에서의 함정

실제 작업에서 우리는 몇 가지 문제에 직면했습니다. 여기에서 공유하여 여러분이 피할 수 있도록 (또는 미리 해결책을 준비할 수 있도록) 하겠습니다.

### 1. GitHub Token 권한 문제

처음 구성할 때 배포가 항상 403 (Forbidden) 오류를 보고했습니다. 오랫동안 조사한 결과 GitHub 기본 `GITHUB_TOKEN`에는 Pages에 쓰기 권한이 없다는 것을 발견했습니다.

**해결 방법**: 저장소의 `Settings` -> `Actions` -> `General` -> `Workflow permissions`에서 반드시 **"Read and write permissions"**를 선택하세요.

### 2. 빌드 디렉터리 경로 오류

Docusaurus는 기본적으로 빌드된 정적 파일을 `build` 디렉터리에 둡니다. 그러나 일부 프로젝트(예: Create React App 기본값은 `build`, Vite 기본값은 `dist`)는 구성이 다를 수 있습니다. Actions에서 파일을 찾을 수 없다는 오류가 발생하면 `docusaurus.config.js`에서 출력 경로 구성을 확인하세요.

### 3. 하위 경로 문제

저장소가 사용자 페이지(즉, `username.github.io`가 아님)가 아니라 프로젝트 페이지(예: `username.github.io/project-name`)인 경우 `baseUrl`을 구성해야 합니다.

`docusaurus.config.js`에서:

```javascript
module.exports = {
  // ...
  url: 'https://hagicode.com', // Hagicode URL
  baseUrl: '/', // 루트 경로 배포
  // ...
};
```
이 부분은 쉽게 간과되기 쉽습니다. 구성이 올바르지 않으면 리소스 경로를 로드할 수 없어 페이지가 열리면 흰 화면만 나타납니다.

## 결과 검증

모든 것을 구성하고 코드를 푸시한 후 GitHub 저장소의 **Actions** 탭으로 이동하여 볼 수 있습니다.

노란색 원(워크플로우 실행 중)이 보이고, 녹색으로 변하면 성공입니다! 빨간색으로 변하면 클릭하여 로그를 확인하면 일반적으로 문제를 찾을 수 있습니다(대부분 오타나 경로 구성이 올바르지 않음).

빌드 성공 후 `https://<사용자명>.github.io/<저장소명>/`에 접속하면 새로운 사이트를 볼 수 있습니다.

## 요약

GitHub Actions를 도입하여 우리는 HagiCode 문서 사이트의 자동화된 배포를 성공적으로 구현했습니다. 이는 수동 작업 시간을 절약할 뿐만 아니라 게시 프로세스의 표준화를 보장합니다. 이제 어떤 팀원이 문서를 업데이트하든 `main` 브랜치에 병합하기만 하면 몇 분 내에 온라인에서 최신 콘텐츠를 볼 수 있습니다.

**핵심 이점**:
- **효율성 향상**: "수동 패키징, 수동 업로드"에서 "코드即게시"로.
- **오류 감소**: 인적 조작 실수 가능성 제거.
- **경험 최적화**: 개발자가 복잡한 배포 프로세스에 방해받는 대신 콘텐츠 품질에 집중할 수 있도록 함.

CI/CD 구성은 처음에 약간 번거로울 수 있지만(특히 다양한 권한 및 경로 문제), 일회성 투자이며 장기적으로 큰 보상이 있는 작업입니다. 모든 정적 사이트 프로젝트에 유사한 자동화 프로세스를 도입하는 것을 강력히 권장합니다.
