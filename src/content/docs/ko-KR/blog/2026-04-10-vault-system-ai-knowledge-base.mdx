## Vault 시스템으로 AI 시대의 프로젝트 간 지식 베이스 구축

> 모작 프로젝트 학습법이 주류가 되고 있지만, 학습 자료가 분산되고 컨텍스트가 단절되는 문제로 AI 어시스턴트가 최대의 가치를 발휘하지 못합니다. 이 글에서는 HagiCode 프로젝트의 Vault 시스템 설계를 소개합니다 — 통합된 저장소 추상화 계층을 통해 AI 어시스턴트가 모든 학습 리소스를 이해하고 접근할 수 있게 하여, 진정한 프로젝트 간 지식 재사용을 실현합니다.

## 배경

사실 AI 시대에 우리는 새로운 기술을 배우는 방식이 조용히 변화하고 있습니다. 전통적인 독서, 비디오 시청도 중요하지만, "모작 프로젝트" — 우수한 오픈 소스 프로젝트의 코드, 아키텍처, 디자인 패턴을 깊이 연구하고 학습하는 것 — 이 점점 더 효율적입니다. 고품질 오픈 소스 프로젝트를 직접 실행하고 수정하면 실제 엔지니어링 실무를 가장 빠르게 이해할 수 있습니다.

하지만 이 방식도 새로운 문제를 가져옵니다.

**학습 자료가 너무 분산되어 있습니다**. 메모는 Obsidian에 있고, 코드 저장소는 여러 폴더에 흩어져 있으며, AI 어시스턴트의 대화 기록은 또 다른 독립된 데이터 사일로입니다. AI의 도움으로 특정 프로젝트를 분석할 때마다 코드 조각을 수동으로 복사하고 컨텍스트를 정리해야 하므로 과정이 상당히 번거롭습니다.

**컨텍스트가 자주 끊깁니다**. AI 어시스턴트는 로컬 학습 리소스에 직접 접근할 수 없어 매번 대화할 때마다 배경 정보를 다시 제공해야 합니다. 모작한 코드 저장소는 빠르게 업데이트되어 수동 동기화는 오류가 발생하기 쉽습니다. 더 나쁜 것은 여러 학습 프로젝트 간에 지식을 공유하기 어렵다는 것입니다 — A 프로젝트에서 배운 디자인 패턴은 AI가 B 프로젝트를 처리할 때 전혀 알지 못합니다.

이러한 문제의 본질은 "데이터 사일로"입니다. 통합된 저장소 추상화 계층이 있어 AI 어시스턴트가 모든 학습 리소스를 이해하고 접근할 수 있다면 문제가 쉽게 해결됩니다.

이러한 문제점을 해결하기 위해 HagiCode를 개발하면서 핵심 설계 결정을 내렸습니다: 통합 지식 저장소 추상화 계층으로 Vault 시스템을 구축합니다. 이 결정이 가져온 변화는 상상보다 클 수 있습니다 — 나중에 자세히 설명하겠습니다.

## HagiCode 소개

이 글에서 공유하는 솔루션은 [HagiCode](https://hagicode.com) 프로젝트의 실제 경험에서 나온 것입니다. HagiCode는 OpenSpec 워크플로우 기반의 AI 코드 어시스턴트로, 핵심 개념은 AI가 "말"하는 것뿐만 아니라 "행동"할 수 있게 하는 것입니다 — 코드 저장소를 직접 조작하고, 명령을 실행하며, 테스트를 실행할 수 있습니다. GitHub: [github.com/HagiCode-org/site](https://github.com/HagiCode-org/site)

개발 과정에서 AI 어시스턴트가 사용자의 다양한 학습 리소스에 자주 접근해야 한다는 것을 발견했습니다: 코드 저장소, 메모 문서, 구성 파일 등. 매번 사용자가 수동으로 제공해야 한다면 경험이 너무 나쁩니다. 이것이 Vault 시스템을 설계하게 된 동기가 되었습니다.

## 핵심 설계

### 다중 유형 지원

HagiCode의 Vault 시스템은 네 가지 유형을 지원하며 각각 다른 사용 시나리오에 해당합니다:

| 유형 | 용도 | 일반적인 시나리오 |
|------|------|----------|
| `folder` | 일반 폴더 유형 | 임시 학습 자료, 초안 |
| `coderef` | 모작 코드 프로젝트 전용 | 특정 오픈 소스 프로젝트를 체계적으로 학습 |
| `obsidian` | Obsidian 메모 소프트웨어 통합 | 기존 메모 라이브러리 재사용 |
| `system-managed` | 시스템 자동 관리 | 프로젝트 구성, 프롬프트 템플릿 등 |

其中 `coderef` 유형은 HagiCode에서 가장 자주 사용되며, 모작 코드 프로젝트를 위해 표준화된 디렉터리 구조와 AI가 읽을 수 있는 메타데이터 설명을 제공합니다. 왜 이 유형을 특별히 설계했을까요? 오픈 소스 프로젝트를 모작하는 것은 단순히 "코드를 다운로드"하는 것이 아니라 코드 자체, 학습 메모, 구성 파일 등 다양한 콘텐츠를 동시에 관리해야 하기 때문에 `coderef`가 이 모든 것을 표준화합니다.

### 영구 저장소 메커니즘

Vault의 레지스트리는 JSON 형식으로 파일 시스템에 영구 저장됩니다:

```csharp
_registryFilePath = Path.Combine(absoluteDataDir, "personal-data", "vaults", "registry.json");
```

이 설계는 간단해 보이지만 사실 깊은 고민 끝에 나온 것입니다:

**간단하고 신뢰할 수 있습니다**. JSON 형식은 인간이 읽을 수 있어 디버깅과 수동 수정에 편리합니다. 시스템에 문제가 발생하면 파일을 직접 열어 상태를 볼 수 있고 심지어 수동으로 수정할 수도 있습니다 — 이는 개발 단계에서 특히 유용합니다.

**종속성 감소**. 파일 시스템 저장소는 데이터베이스의 복잡성을 피합니다. 데이터베이스 서비스를 별도로 설치하고 구성할 필요가 없어 시스템 복잡도와 유지 관리 비용이 줄어듭니다.

**동시성 안전**. `SemaphoreSlim`을 사용하여 다중 스레드 안전성을 보장합니다. AI 코드 어시스턴트 시나리오에서는 여러 작업이 동시에 vault 레지스트리에 접근할 수 있으므로 동시성 제어를 잘 해야 합니다.

### AI 컨텍스트 통합

시스템의 핵심 능력은 vault 정보를 AI 제안의 컨텍스트에 자동으로 주입할 수 있다는 것입니다:

```typescript
export function buildTargetVaultsText(
  vaults: VaultForText[],
  template: VaultPromptTemplate = DEFAULT_VAULT_PROMPT_TEMPLATE,
): string {
  const readOnlyVaults = vaults.filter((vault) => vault.accessType === 'read');
  const editableVaults = vaults.filter((vault) => vault.accessType === 'write');

  const sections = [
    buildVaultSection(readOnlyVaults, template.reference),
    buildVaultSection(editableVaults, template.editable),
  ].filter(Boolean);

  return `\n\n### ${template.heading}\n\n${sections.join('\n')}`;
}
```

이렇게 하면 AI 어시스턴트가 사용 가능한 학습 리소스를 자동으로 이해하여 사용자가 매번 수동으로 컨텍스트를 제공할 필요가 없습니다. 이 설계로 HagiCode의 경험이 특히 자연스러워졌습니다 — AI에게 "React의 동시성 렌더링을 분석해 줘"라고 말하면 AI가 이전에 등록된 React 학습 vault를 자동으로 찾아 코드를 반복해서 붙여넣을 필요가 없습니다.

### 접근 제어 메커니즘

시스템은 vault를 두 가지 접근 유형으로 나눕니다:

- **reference(읽기 전용)**: AI는 분석 및 이해에만 사용하며 콘텐츠를 수정할 수 없습니다
- **editable(편집 가능)**: AI는 작업 필요에 따라 콘텐츠를 수정할 수 있습니다

이러한 구분은 AI가 어떤 콘텐츠가 "읽기 전용 참조"이고 어떤 콘텐츠가 "수정할 수 있는 것"인지 알게 하여 오작업 위험을 피합니다. 예를 들어 학습 자료로 오픈 소스 프로젝트의 vault를 등록했다면 AI가 코드를 마음대로 수정하는 것을 원하지 않을 것입니다 — `reference`로 표시하면 됩니다. 하지만 자신의 프로젝트 vault라면 `editable`로 표시하여 AI가 코드 수정을 도와줄 수 있습니다.

## 실용 가이드

### CodeRef Vault의 표준화된 구조

coderef 유형의 vault에 대해 시스템은 표준화된 디렉터리 구조를 제공합니다:

```
my-coderef-vault/
├── index.yaml          # vault 메타데이터 설명
├── AGENTS.md           # AI 어시스턴트의 운영 가이드
├── docs/               # 학습 메모 및 문서 보관
└── repos/              # Git 하위 모듈을 통해 모작한 코드 저장소 관리
```

이 구조의 설계 철학은 무엇일까요?

**docs/** 에는 학습 메모를 보관하며 Markdown 형식으로 코드에 대한 이해, 아키텍처 분석, 문제 해결 경험을 기록합니다. 이러한 메모는 자신만 보는 것이 아니라 AI도 읽을 수 있습니다 — 관련 작업을 처리할 때 자동으로 참조합니다.

**repos/** 는 Git 하위 모듈을 통해 모작한 저장소를 관리하며 코드를 직접 복사하지 않습니다. 이렇게 하면 두 가지 이점이 있습니다:一是 업스트림과 동기화를 유지하고 `git submodule update` 하나로 최신 코드를 받을 수 있으며,二是 공간을 절약하고 여러 vault가 동일한 저장소의 다른 버전을 참조할 수 있습니다.

**index.yaml** 에는 vault의 메타데이터가 포함되어 AI 어시스턴트가 용도와 콘텐츠를 빠르게 이해할 수 있습니다. vault에 "자기소개"를 쓴 것과 같아 AI가 처음 보면 무엇을 위한 것인지 알게 됩니다.

**AGENTS.md** 는 AI 어시스턴트를 위해 작성된 가이드로 vault의 콘텐츠를 처리하는 방법을 설명합니다. 여기서 AI에게 "이 프로젝트를 분석할 때 성능 최적화 관련 코드에 중점을 둬" 또는 "테스트 파일을 수정하지 마"라고 말할 수 있습니다.

### Vault 생성 및 사용

CodeRef vault를 만드는 것은 간단합니다:

```typescript
const createCodeRefVault = async () => {
  const response = await VaultService.postApiVaults({
    requestBody: {
      name: "React Learning Vault",
      type: "coderef",
      physicalPath: "/Users/developer/vaults/react-learning",
      gitUrl: "https://github.com/facebook/react.git"
    }
  });

  // 시스템이 자동으로:
  // 1. React 저장소를 vault/repos/react에 클론
  // 2. 메모용 docs/ 디렉터리 생성
  // 3. index.yaml 메타데이터 생성
  // 4. AGENTS.md 가이드 파일 생성

  return response;
};
```

그런 다음 AI 제안에서 이 vault를 참조합니다:

```typescript
const proposal = composeProposalChiefComplaint({
  chiefComplaint: "帮我分析 React 的并发渲染机制",
  repositories: [
    { id: "react", gitUrl: "https://github.com/facebook/react.git" }
  ],
  vaults: [
    {
      id: "react-learning",
      name: "React Learning Vault",
      type: "coderef",
      physicalPath: "/vaults/react-learning",
      accessType: "read"  // AI는 읽기만 가능, 수정 불가
    }
  ],
  quickRequestText: "重点关注 fiber 架构和 scheduler 实现"
});
```

### 일반적인 사용 시나리오

**시나리오 1: 오픈 소스 프로젝트 체계적 학습**

CodeRef vault를 만들고 Git 하위 모듈을 통해 대상 저장소를 관리하며 `docs/` 디렉터리에 학습 메모를 기록합니다. AI는 코드와 메모에 동시에 접근하여 더 정확한 분석을 제공할 수 있습니다. 특정 모듈을 학습할 때 작성한 메모는 AI가 이후 관련 코드를 분석할 때 자동으로 참조합니다 — 마치 "조수"가 이전의 생각을 기억하는 것과 같습니다.

**시나리오 2: Obsidian 메모 라이브러리 재사용**

이미 Obsidian으로 메모를 관리하고 있다면 기존 vault를 HagiCode에 직접 등록하기만 하면 됩니다. AI는 지식 베이스에 직접 접근할 수 있으며 수동으로 복사하여 붙여넣을 필요가 없습니다. 이 기능은 특히 실용적이며 많은 사람들이 수년 동안 축적한 메모 라이브러리가 있으며 연결하면 AI가 지식 체계를 "읽"을 수 있습니다.

**시나리오 3: 프로젝트 간 지식 재사용**

여러 AI 제안이 동일한 vault를 참조하여 지식의 프로젝트 간 재사용을 실현할 수 있습니다. 예를 들어 "디자인 패턴 학습 vault"를 만들고 다양한 디자인 패턴의 메모와 코드 예제를 기록했습니다. 어떤 프로젝트를 분석하든 AI는 이 vault의 내용을 참조할 수 있습니다 — 지식을 반복해서 축적할 필요가 없습니다.

### 경로 보안 메커니즘

시스템은 경로를 엄격하게 검증하여 경로 순회 공격을 방지합니다:

```csharp
private static string ResolveFilePath(string vaultRoot, string relativePath)
{
    var rootPath = EnsureTrailingSeparator(Path.GetFullPath(vaultRoot));
    var combinedPath = Path.GetFullPath(Path.Combine(rootPath, relativePath));
    if (!combinedPath.StartsWith(rootPath, StringComparison.OrdinalIgnoreCase))
    {
        throw new BusinessException(VaultRelativePathTraversalCode,
            "Vault file paths must stay inside the registered vault root.");
    }
    return combinedPath;
}
```

이것은 모든 파일 작업이 vault의 루트 디렉터리 범위 내에 있도록 보장하며 악의적인 경로 접근을 방지합니다. 보안은 소홀히 할 수 없으며 AI 어시스턴트가 파일 시스템을 조작하려면 경계를 명확히 해야 합니다.

## 주의 사항

HagiCode Vault 시스템을 사용할 때 몇 가지 사항에特别注意해야 합니다:

1. **경로 보안**: 사용자 정의 경로가 허용된 범위 내에 있는지 확인하십시오. 그렇지 않으면 시스템이 작업을 거부합니다. 이는 오작업과 잠재적인 보안 위험을 방지하기 위한 것입니다.

2. **Git 하위 모듈 관리**: CodeRef vault는 코드를 직접 복사하는 대신 Git 하위 모듈을 사용하는 것이 좋습니다. 앞서 말한 이점 — 동기화 유지, 공간 절약 — 이 있습니다. 단 하위 모듈만의 사용 방식이 있어 처음 사용할 때 익숙해질 필요가 있습니다.

3. **파일 미리보기 제한**: 시스템은 파일 크기(256KB)와 수량(500개)을 제한하며 초대형 파일은 일괄 처리해야 합니다. 이 제한은 성능 고려사항이며 초대형 파일을 만나면 수동으로 분할하거나 다른 방식으로 처리할 수 있습니다.

4. **진단 정보**: vault를 만들면 진단 정보가 반환되며 실패 시 디버깅에 사용할 수 있습니다. 문제가 발생하면 먼저 진단 정보를 확인하십시오. 대부분의 경우 단서를 찾을 수 있습니다.

## 요약

HagiCode의 Vault 시스템은 본질적으로 간단하지만 깊은 문제를 해결합니다: AI 어시스턴트가 로컬 지식 리소스를 이해하고 사용하게 하는 방법.

통합된 저장소 추상화 계층, 표준화된 디렉터리 구조, 자동화된 컨텍스트 주입을 통해 "한 번 등록하면 어디서든 재사용"하는 지식 관리 방식을 실현했습니다. vault를 만든 후 학습 메모, 코드 저장소, 문서 자료 등 AI가 자동으로 접근하고 이해할 수 있습니다.

이 설계가 가져온 경험 향상은 분명합니다. 더 이상 코드 조각을 수동으로 복사하거나 배경 정보를 반복해서 설명할 필요가 없습니다 — AI 어시스턴트는 프로젝트 상황을 진정으로 이해하는 동료처럼 기존 지식을 기반으로 더 가치 있는 도움을 제공할 수 있습니다.

이 글에서 공유한 Vault 시스템은 HagiCode를 개발하면서 실제로 문제를 겪고 실제로 최적화한 솔루션입니다. 이 설계가 가치 있다고 느껴진다면 엔지니어링 역량이 괜찮은 것입니다 — 그렇다면 HagiCode 자체도 주목할 가치가 있습니다.
