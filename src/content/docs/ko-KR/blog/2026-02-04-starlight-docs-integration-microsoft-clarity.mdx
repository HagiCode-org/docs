## 기술 선정 및 탐색

처음에 우리는 Proposal 단계에서 여러 통합 방식을 논의했습니다. Starlight(Astro 기반 문서 프레임워크)를 사용하고 있으므로 가장 직관적인 생각은 Astro의 Hooks를 활용하는 것이었습니다.

우선 `astro.config.mjs`를 수정하여 빌드 시 Clarity 스크립트를 주입하려고 시도했습니다. 이 방식은 전체 범위를 보장할 수 있지만 유연성이 부족했습니다. 사용자 설정에 따라 스크립트를 동적으로 로드하거나 언로드할 수 없었습니다.

사용자 경험과 개인정보 보호 통제를 고려하여 최종적으로 **구성 요소 재정의** 솔루션을 채택하기로 결정했습니다. Starlight는 개발자가 내부 구성 요소를 재정의할 수 있게 하므로 `<footer>` 또는 `<head>` 렌더링 로직을 인수하여 Clarity 로드 타이밍을 세밀하게 제어할 수 있습니다.

작은 에피소드가 있습니다. 원래 `StarlightWrapper.astro`라는 레이아웃 래퍼를 만들려고 했습니다. 하지만 실제 디버깅中发现 Starlight의 라우팅 메커니즘은 이 사용자 정의 Wrapper를 자동으로 호출하지 않아 일부 페이지에서 스크립트가 작동하지 않았습니다. 이것은 전형적인 "당연히 그렇겠지"라는 함정 경험으로, 우리가 **프레임워크의 렌더링 흐름을 깊이 이해해야 하지 맹목적으로 일반 프레임워크 패턴을 적용하면 안 된다는 점**을 상기시켜 주었습니다.

## 핵심 솔루션: Footer 구성 요소 재정의

Clarity 스크립트가 모든 페이지(문서 및 블로그 포함)에서 로드되고 기존 페이지 구조를 파기하지 않도록 Starlight의 `Footer` 구성 요소를 재정의하기로 선택했습니다.

### 왜 Footer인가요?

1.  **전역성**: Footer는 거의 모든 표준 페이지에 나타납니다.
2.  **비침투성**: 스크립트를 Footer 영역(실제로는 body 하단에 렌더링)에 배치하면 중요한 렌더링 경로(LCP)를 차단하지 않아 성능에 미치는 영향이 최소화됩니다.
3.  **논리 중앙화**: 구성 요소 내부에서 Cookie 동의 로직을 통합적으로 처리할 수 있습니다.

**핵심 포인트 분석**:
- `is:inline`: Astro에게 이 script 태그 내의 콘텐츠를 처리하지 말고 HTML로 직접 출력하도록 지시합니다. 이는 서드파티 통계 스크립트에 매우 중요합니다. 그렇지 않으면 Astro의 번들링 최적화로 인해 스크립트가 작동하지 않을 수 있습니다.
- `define:vars`: Astro 3+ 기능으로 범위 내에서 변수를 안전하게 주입할 수 있습니다.
- `import.meta.env.PROD`: 로컬 개발 시(디버깅 목적이 아닌 한) 무효 통계를 생성하지 않도록 하여 데이터를 깨끗하게 유지합니다.

### 고급: 개인정보 보호 규정 준수 및 Cookie 제어

코드만 추가하는 것은 충분하지 않습니다. 특히 GDPR 관할 지역에서는 사용자의 선택을 존중해야 합니다.

HagiCode의 접근 방식은 간단한 스위치를 제공하는 것입니다. 전체 기능의 Cookie Banner는 아니지만 순수 전시용 기술 문서 사이트의 경우 일반적으로 "필수" 또는 "통계" 유형의 Cookie에 속하므로 개인정보 처리방침을 통해 알리고 기본적으로 활성화하거나 Footer에서 개인정보 설정 페이지로 링크할 수 있습니다.

## 경험 요약 및 함정

이 솔루션을 HagiCode에 적용하는 과정에서 우리는 몇 가지容易被忽视하는 세부 사항을 요약했습니다:

1.  **`StarlightWrapper.astro`는 함정입니다**:
    앞서 말한 것처럼 스크립트를 주입하기 위해 전역 Wrapper를 만들려고 시도하지 마세요. Starlight에서는 작동하지 않습니다. 특정 구성 요소(예: `StarlightFooter.astro` 또는 `StarlightHead.astro`)를 재정의하는 것이 정석입니다.

2.  **스크립트 위치의 성능 고려**:
    Clarity는 데이터 정확성을 위해 `<head>`에 배치할 것을 권장하지만 문서 사이트의 경우 첫 화면 로드 속도(LCP)는 SEO와 사용자 리텐션에 직접적인 영향을 미칩니다. 우리는 Footer(Body 하단)에 배치하기로 선택했는데, 이는 극소수의 "초기 이탈" 사용자 데이터를 약간 잃을 수 있지만 더 빠른 페이지 로드 경험을 얻는 대가로 가치 있는 트레이드오프입니다.

3.  **개발 환경의 간섭**:
    반드시 `import.meta.env.PROD` 판단을 추가해야 합니다. 개발 모드에서 페이지를 자주 새로고침하면 의미 없는 테스트 데이터가 대량으로 생성되어 Clarity 대시보드를 오염시킵니다.

## 효과 검증

배포 완료 후 Clarity 콘솔에서 실시간 데이터를 확인할 수 있습니다. 일반적으로 몇 분 내에 사용자의 히트맵과 녹화를 볼 수 있습니다.

HagiCode의 경우 이 데이터를 통해 우리는 다음을 발견했습니다:
- 많은 사용자가 "빠른 시작" 장을 반복해서 보는데, 이는 우리의 설치 가이드가 충분히 직관적이지 않다는 것을 의미합니다.
- "API 참조" 페이지의 체류 시간이 가장 길어 핵심 사용자 그룹의 수요를 확인했습니다.

## 요약

Microsoft Clarity 통합은 복잡한 서버 측 개조도 필요 없고 무거운 SDK 도입도 필요하지 않습니다.

Starlight의 구성 요소 재정의 메커니즘을 활용하여 단 하나의 가벼운 `StarlightFooter.astro` 구성 요소로 전역 데이터 통계를 구현했습니다. 이러한 "마이크로 통합" 방식은 코드의 깔끔함을 보장할 뿐만 아니라 사용자 행동을 통찰하는 능력도 부여합니다.

기술 프로젝트를 운영 중이시라면, 특히 **HagiCode**처럼 문서를 지속적으로 반복해야 하는 프로젝트라면 Clarity 통합을 강력히 권장합니다. 데이터가 사용자의 진짜 통증 지점이 어디인지 알려줄 것입니다.
