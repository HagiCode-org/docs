## Docker Hub에서 Azure ACR로 자동화된 이미지 동기화 구현

> 이 문서에서는 GitHub Actions와 image-syncer 도구를 사용하여 Docker Hub 이미지를 Azure Container Registry로 자동 동기화하는 방법을 소개합니다. 이는 국내 및 일부 Azure 지역에서 Docker Hub 액세스 속도가 느린 문제를 해결하고, 이미지 가용성과 Azure 환경의 배포 효율성을 향상시킵니다.

## 배경/서론

HagiCode 프로젝트는 Docker 이미지를 핵심 런타임 구성 요소로 사용하며, 주요 이미지는 Docker Hub에 호스팅됩니다. 프로젝트 발전과 Azure 환경 배포 요구사항 증가에 따라 다음과 같은 문제점에 직면했습니다:
- 이미지 풀 속도가 느리고, Docker Hub에 대한 국내 및 일부 Azure 지역 액세스가 제한됨
- 단일 이미지 소스 의존으로 인한 단일 실패점(SPOF) 위험
- Azure 환경에서 Azure Container Registry를 사용하면 더 나은 네트워크 성능과 통합 경험을 얻을 수 있음

이러한 문제를 해결하기 위해 Docker Hub 이미지를 정기적으로 Azure ACR로 동기화하는 자동화된 메커니즘을 구축하여, 사용자가 Azure 환경에서 더 빠른 이미지 풀 속도와 더 높은 가용성을 보장할 수 있도록 해야 했습니다.

## 구현 단계

### 1. 준비 단계
- Azure Portal에서 Azure Container Registry 생성 또는 확인
- ACR 액세스 키 생성(사용자 이름 및 암호)
- Docker Hub 이미지 리포지토리 액세스 권한 확인

### 2. GitHub Secrets 구성
GitHub 리포지토리 설정에서 다음 Secrets 추가:
- AZURE_ACR_USERNAME: Azure ACR 사용자 이름
- AZURE_ACR_PASSWORD: Azure ACR 암호

### 3. GitHub Actions 워크플로우 생성
.github/workflows/sync-docker-acr.yml에 워크플로우 구성:
- 예약 트리거: 매일 UTC 00:00
- 수동 트리거: workflow_dispatch 지원
- 추가 트리거: publish 분기 푸시 시 트리거(빠른 동기화용)

### 4. 워크플로우 실행 흐름

| 순서 | 참여자 | 동작 | 설명 |
| --- | --- | --- | --- |
| 1 | GitHub Actions | 워크플로우 트리거 | 예약, 수동 또는 `publish` 분기 푸시로 트리거 |
| 2 | GitHub Actions → image-syncer | 동기화 도구 다운로드 및 실행 | 실제 동기화 단계 진입 |
| 3 | image-syncer → Docker Hub | 이미지 manifest 및 태그 목록 가져오기 | 소스 리포지토리 메타데이터 읽기 |
| 4 | image-syncer → Azure ACR | 대상 리포지토리 기존 이미지 정보 가져오기 | 대상 측 현재 상태 확인 |
| 5 | image-syncer | 소스와 대상 간 차이 비교 | 동기화가 필요한 이미지 레이어 식별 |
| 6 | image-syncer → Docker Hub | 변경된 이미지 레이어 풀 | 업데이트가 필요한 콘텐츠만 전송 |
| 7 | image-syncer → Azure ACR | 변경된 이미지 레이어 푸시 | 증분 동기화 완료 |
| 8 | image-syncer → GitHub Actions | 동기화 통계 정보 반환 | 결과, 차이점 및 오류 정보 포함 |
| 9 | GitHub Actions | 로그 기록 및 artifact 업로드 | 향후 감사 및 문제 해결 용이 |

## GitHub Actions 워크플로우 구현

실제 실행 중인 워크플로우 구성입니다(.github/workflows/sync-docker-acr.yml):

```yaml
name: Sync Docker Image to Azure ACR

on:
  schedule:
    - cron: "0 0 * * *" # 매일 UTC 00:00
  workflow_dispatch: # 수동 트리거
  push:
    branches: [publish]

permissions:
  contents: read

jobs:
  sync:
    runs-on: ubuntu-latest

    steps:
      - name: Checkout code
        uses: actions/checkout@v4

      - name: Download image-syncer
        run: |
          # image-syncer 바이너리 파일 다운로드
          wget https://github.com/AliyunContainerService/image-syncer/releases/download/v1.5.5/image-syncer-v1.5.5-linux-amd64.tar.gz
          tar -zxvf image-syncer-v1.5.5-linux-amd64.tar.gz
          chmod +x image-syncer

      - name: Create auth config
        run: |
          # 인증 구성 파일 생성 (YAML 형식)
          cat > auth.yaml <<EOF
          hagicode.azurecr.io:
            username: "${{ secrets.AZURE_ACR_USERNAME }}"
            password: "${{ secrets.AZURE_ACR_PASSWORD }}"
          EOF

      - name: Create images config
        run: |
          # 이미지 동기화 구성 파일 생성 (YAML 형식)
          cat > images.yaml <<EOF
          docker.io/newbe36524/hagicode: hagicode.azurecr.io/hagicode
          EOF

      - name: Run image-syncer
        run: |
          # 동기화 실행 (새 버전 --auth 및 --images 매개변수 사용)
          ./image-syncer --auth=./auth.yaml --images=./images.yaml --proc=10 --retries=3

      - name: Upload logs
        if: always()
        uses: actions/upload-artifact@v4
        with:
          name: sync-logs
          path: image-syncer-*.log
          retention-days: 7
```

## 구성 설명

### 1. 트리거 조건
- 예약 트리거: cron: "0 0 * * *" - 매일 UTC 00:00 실행
- 수동 트리거: workflow_dispatch - GitHub UI에서 사용자가 수동으로 실행 가능
- 푸시 트리거: push: branches: [publish] - 게시 분기 푸시 시 트리거(빠른 동기화용)

### 2. 인증 구성 (auth.yaml)
```yaml
hagicode.azurecr.io:
  username: "${{ secrets.AZURE_ACR_USERNAME }}"
  password: "${{ secrets.AZURE_ACR_PASSWORD }}"
```

### 3. 이미지 동기화 구성
```yaml
docker.io/newbe36524/hagicode: hagicode.azurecr.io/hagicode
```
이 구성은 docker.io/newbe36524/hagicode의 모든 태그를 hagicode.azurecr.io/hagicode로 동기화함을 나타냅니다.

### 4. image-syncer 매개변수
- --auth=./auth.yaml: 인증 구성 파일 경로
- --images=./images.yaml: 이미지 동기화 구성 파일 경로
- --proc=10: 동시 스레드 수 10
- --retries=3: 실패 시 3회 재시도

## 주의사항 및 모범 사례

### 1. 보안 권장사항
- Azure ACR 암호 정기 교체(권장: 90일마다)
- 전용 ACR 서비스 계정 사용, 권한을 푸시 전용으로 제한
- ACR 액세스 로그 모니터링, 비정상 액세스 즉시 발견
- 로그에 인증 정보 출력 금지
- 인증 정보를 코드 리포지토리에 커밋 금지

### 2. 성능 최적화
- --proc 매개변수 조정: 네트워크 대역폭에 따라 동시성 수 조정(권장: 5-20)
- 동기화 시간 모니터링: 동기화 시간이 너무 길면 동시성 수 감소 고려
- 로그 정기 정리: 합리적인 retention-days 설정(현재 7일)

### 3. 문제 해결

#### 문제 1: 인증 실패
```
Error: failed to authenticate to hagicode.azurecr.io
```
해결 방법:
1. GitHub Secrets가 올바르게 구성되었는지 확인
2. Azure ACR 암호가 만료되었는지 확인
3. ACR 서비스 계정 권한이 올바른지 확인

#### 문제 2: 네트워크 시간 초과
```
Error: timeout waiting for response
```
해결 방법:
1. 네트워크 연결 확인
2. 동시 스레드 수 감소(--proc 매개변수)
3. 네트워크 복구 시까지 기다린 후 워크플로우 다시 트리거

#### 문제 3: 이미지 동기화 불완전
```
Warning: some tags failed to sync
```
해결 방법:
1. 동기화 로그 확인, 실패한 태그 식별
2. 수동으로 워크플로우 트리거하여 재동기화
3. Docker Hub 소스 이미지가 정상인지 확인

### 4. 모니터링 및 알림
- Actions 페이지 정기 확인, 워크플로우 실행 상태 확인
- GitHub 알림 설정, 워크플로우 실패 알림 즉시 수신
- Azure ACR 저장소 사용량 모니터링
- 이미지 태그 일관성 정기 확인

## 자주 묻는 질문 및 해결 방법

### Q1: 모든 태그가 아닌 특정 태그만 동기화하려면 어떻게 하나요?

images.yaml 구성 파일 수정:
```yaml
# latest 및 v1.0 태그만 동기화
docker.io/newbe36524/hagicode:latest: hagicode.azurecr.io/hagicode:latest
docker.io/newbe36524/hagicode:v1.0: hagicode.azurecr.io/hagicode:v1.0
```

### Q2: 여러 이미지 리포지토리를 동기화하려면 어떻게 하나요?

images.yaml에 여러 줄 구성 추가:
```yaml
docker.io/newbe36524/hagicode: hagicode.azurecr.io/hagicode
docker.io/newbe36524/another-image: hagicode.azurecr.io/another-image
```

### Q3: 동기화 실패 후 재시도하려면 어떻게 하나요?
- 자동 재시도: image-syncer 내장 재시도 메커니즘(기본값 3회)
- 수동 재시도: GitHub Actions 페이지에서 Re-run all jobs 클릭

### Q4: 동기화의 상세 진행 상황을 어떻게 보나요?
- Actions 페이지에서 실시간 로그 확인
- sync-logs artifact 다운로드하여 전체 로그 파일 확인
- 로그 파일은 각 태그의 동기화 상태와 전송 속도 포함

### Q5: 동기화하는 데 얼마나 걸리나요?
- 초기 전체 동기화: 이미지 크기에 따라 일반적으로 10-30분 소요
- 증분 동기화: 이미지 변경이 작으면 일반적으로 2-5분 소요
- 시간은 네트워크 대역폭, 이미지 크기, 동시성 설정에 따라 달라짐

```yaml
- name: Filter tags

    echo "docker.io/newbe36524/hagicode:v* : hagicode.azurecr.io/hagicode:v*" > images.yaml
```

### 3. 동기화 통계 보고서 추가
```yaml
- name: Generate report
  if: always()
  run: |
    echo "## Sync Report" >> $GITHUB_STEP_SUMMARY
    echo "- Total tags: $(grep -c 'synced' image-syncer-*.log)" >> $GITHUB_STEP_SUMMARY
    echo "- Sync time: ${{ steps.sync.outputs.duration }}" >> $GITHUB_STEP_SUMMARY
```

## 요약

이 문서에서 소개한 방법을 통해 Docker Hub에서 Azure ACR로의 자동화된 이미지 동기화를 성공적으로 구현했습니다. 이 솔루션은 GitHub Actions의 예약 트리거 및 수동 트리거 기능과 image-syncer의 증분 동기화 및 오류 처리 메커니즘을 활용하여 이미지의 적시 동기화와 일관성을 보장합니다.

또한 보안 모범 사례, 성능 최적화, 문제 해결 등에 관한 내용을 논의하여 사용자가 이 동기화 메커니즘을 더 잘 관리하고 유지할 수 있도록 도왔습니다. 이 문서가 Azure 환경에서 Docker 이미지를 배포해야 하는 개발자에게 가치 있는 참고 자료가 되기를 바랍니다.
