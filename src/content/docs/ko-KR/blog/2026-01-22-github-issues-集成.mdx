## 처음부터 GitHub Issues 통합 구축하기: HagiCode의 프론트엔드 직접 연결 실천

> 이 글에서는 HagiCode 플랫폼에 GitHub Issues를 통합하는 전체 과정을 기록합니다. "프론트엔드 직접 연결 + 백엔드 최소화" 아키텍처를 통해 백엔드를 가볍게 유지하면서 안전한 OAuth 인증과 효율적인 Issues 동기화를 구현하는 방법을 살펴봅니다.

## 배경: 왜 GitHub를 통합하는가?

AI 보조 개발 플랫폼인 HagiCode의 핵심 가치는 아이디어와 구현을 연결하는 데 있습니다. 하지만 실제 사용에서 사용자가 HagiCode에서 Proposal(제안)을 완료한 후, 프로젝트 추적을 위해 GitHub Issues에 내용을 수동으로 복사해야 하는 경우가 많았습니다.

이로 인해 몇 가지 뚜렷한 문제점이 발생했습니다:

1.  **워크플로우 단절**: 사용자가 두 시스템 간을 오가야 합니다. 경험이 매끄럽지 않고, 복사 붙여넣기 과정에서 핵심 정보가 누락되기 쉽습니다.
2.  **협업 불편**: 팀의 다른 구성원은 GitHub에서 작업을 확인하는 데 익숙하며, HagiCode 내의 제안 진행 상황을 직접 볼 수 없습니다.
3.  **반복 작업**: 제안이 업데이트될 때마다 GitHub에서 해당 Issue를 수동으로 업데이트해야 하며, 불필요한 유지보수 비용이 발생합니다.

이 문제를 해결하기 위해 **GitHub Issues Integration** 기능을 도입하여 HagiCode 세션과 GitHub 저장소의 연결을打通하고 "원클릭 동기화"를 구현하기로 했습니다.

## 기술 선택: 프론트엔드 직접 연결 vs 백엔드 프록시

통합 방안을 설계할 때, 전통적인 "백엔드 프록시 모델"과 더 적극적인 "프론트엔드 직접 연결 모델" 두 가지 선택지가 있었습니다.

### 방안 비교

전통적인 **백엔드 프록시 모델**에서는 프론트엔드의 모든 요청이 먼저 백엔드를 거친 후 백엔드가 GitHub API를 호출합니다. 논리는 집중되지만 백엔드에 적지 않은 부담을 줍니다:

1.  **백엔드 비대화**: 전용 GitHub API 클라이언트 래퍼를 작성해야 하고 OAuth의 복잡한 상태 머신도 처리해야 합니다.
2.  **토큰 위험**: 사용자의 GitHub 토큰을 백엔드 데이터베이스에 저장해야 합니다. 암호화가 가능하지만 보안 위험 표면이 증가합니다.
3.  **개발 비용**: 토큰 저장을 위한 데이터베이스 마이그레이션과 추가적인 동기화 서비스 유지보수가 필요합니다.

반면 **프론트엔드 직접 연결 모델**은 훨씬 가볍습니다. 이 방안에서는 백엔드를 가장 민감한 "키 교환" 단계(OAuth 콜백)에만 사용합니다. 토큰을 획득한 후에는 브라우저의 localStorage에 직접 저장합니다. 이후 Issue 생성, 댓글 업데이트 등의 작업은 프론트엔드에서 직접 HTTP 요청을 GitHub로 보냅니다.

| 비교 차원 | 백엔드 프록시 모델 | 프론트엔드 직접 연결 모델 |
| :--- | :--- | :--- |
| **백엔드 복잡도** | 완전한 OAuth 서비스와 GitHub API 클라이언트 필요 | OAuth 콜백 엔드포인트만 필요 |
| **토큰 관리** | 데이터베이스에 암호화 저장, 유출 위험 존재 | 브라우저에 저장, 사용자만 볼 수 있음 |
| **구현 비용** | 데이터베이스 마이그레이션, 다중 서비스 개발 필요 | 주로 프론트엔드 작업 |
| **사용자 경험** | 논리가 통합되지만 서버 지연이 약간 높을 수 있음 | GitHub와 직접 상호작용하여 매우 빠른 응답 |

빠른 통합과 최소한의 백엔드 변경을 원했기 때문에, **최종적으로 "프론트엔드 직접 연결 모델"을 선택**했습니다. 브라우저에 "임시 통행증"을 발급하는 것과 같습니다. 통행증을 받은 후 브라우저가 직접 GitHub에서 업무를 처리할 수 있고, 매번 백엔드 관리자의 승인을 받을 필요가 없습니다.

## 핵심 설계: 데이터 흐름과 보안

아키텍처를 확정한 후 구체적인 데이터 흐름을 설계해야 했습니다. 동기화 프로세스의 핵심은 토큰을 안전하게 획득하고 효율적으로 활용하는 방법입니다.

### 전체 아키텍처 다이어그램

전체 시스템은 세 가지 역할로 추상화할 수 있습니다: 브라우저(프론트엔드), HagiCode 백엔드, GitHub.

```text
+--------------+        +--------------+        +--------------+
|  프론트엔드   |        |    백엔드     |        |    GitHub    |
|   React      |        |   ASP.NET    |        |   REST API   |
|              |        |              |        |              |
|  +--------+  |        |              |        |              |
|  |  OAuth |--+--------> /callback    |        |              |
|  |  흐름  |  |        |              |        |              |
|  +--------+  |        |              |        |              |
|              |        |              |        |              |
|  +--------+  |        |  +--------+  |        |  +--------+  |
|  |GitHub  |  +------------>Session |  +----------> Issues |  |
|  |API     |  |        |  |Metadata|  |        |  |        |  |
|  |직접연결|  |        |  +--------+  |        |  +--------+  |
|  +--------+  |        |              |        |              |
+--------------+        +--------------+        +--------------+
```

**핵심 포인트**: OAuth의 작은 한 단계(code를 토큰으로 교환)만 백엔드를 거치고, 그 이후의 무거운 작업(Issue 생성)은 모두 프론트엔드가 GitHub와 직접 통신합니다.

### 동기화 데이터 흐름 상세

사용자가 HagiCode 인터페이스에서 "Sync to GitHub" 버튼을 클릭하면 일련의 복잡한 동작이 발생합니다:

```text
사용자가 "Sync to GitHub" 클릭
         │
         ▼
1. 프론트엔드가 localStorage에서 GitHub 토큰 확인
         │
         ▼
2. Issue 내용 포맷팅 (Proposal을 Markdown으로 변환)
         │
         ▼
3. 프론트엔드가 GitHub API를 직접 호출하여 Issue 생성/업데이트
         │
         ▼
4. HagiCode 백엔드 API를 호출하여 Session.metadata 업데이트 (Issue URL 등 저장)
         │
         ▼
5. 백엔드가 SignalR을 통해 SessionUpdated 이벤트 브로드캐스트
         │
         ▼
6. 프론트엔드가 이벤트를 수신하고 UI를 업데이트하여 "동기화 완료" 상태 표시
```

### 보안 설계

보안은 타사 서비스 통합에서 항상 최우선입니다. 다음 사항을 고려했습니다:

1.  **CSRF 공격 방어**: OAuth 리다이렉트 시 무작위 `state` 매개변수를 생성하여 sessionStorage에 저장합니다. 콜백 시 state를 엄격하게 검증하여 요청 위조를 방지합니다.
2.  **토큰 저장 격리**: 토큰은 브라우저의 `localStorage`에만 저장됩니다. 동일 출처 정책(Same-Origin Policy)을 활용하여 HagiCode 스크립트만 읽을 수 있으며, 서버 측 데이터베이스 유출이 사용자에게 영향을 주지 않습니다.
3.  **오류 경계**: GitHub API의 일반적인 오류(401 토큰 만료, 422 검증 실패, 429 속도 제한 등)에 대해 전용 오류 처리 로직을 설계하여 사용자에게 친절한 피드백을 제공합니다.

### 2. 프론트엔드 OAuth 흐름

연결의 진입점입니다. 표준 Authorization Code Flow를 사용합니다.

```typescript
// src/HagiCode.Client/src/services/githubOAuth.ts

// 인증 URL 생성 및 리다이렉트
export async function generateAuthUrl(): Promise<string> {
  const state = generateRandomString(); // CSRF 방지 무작위 문자열 생성
  sessionStorage.setItem('hagicode_github_state', state);

  const params = new URLSearchParams({
    client_id: clientId,
    redirect_uri: window.location.origin + '/settings?tab=github&oauth=callback',
    scope: ['repo', 'public_repo'].join(' '),
    state: state,
  });

  return `https://github.com/login/oauth/authorize?${params.toString()}`;
}

// 콜백 페이지에서 Code를 Token으로 교환 처리
export async function exchangeCodeForToken(code: string, state: string): Promise<GitHubToken> {
  // 1. State 검증으로 CSRF 방지
  const savedState = sessionStorage.getItem('hagicode_github_state');
  if (state !== savedState) throw new Error('Invalid state parameter');

  // 2. 백엔드 API를 호출하여 토큰 교환
  // 참고: ClientSecret이 필요하므로 반드시 백엔드를 거쳐야 합니다 (프론트엔드에 노출 불가)
  const response = await fetch('/api/GitHubOAuth/callback', {
    method: 'POST',
    headers: { 'Content-Type': 'application/json' },
    body: JSON.stringify({ code, state, redirectUri: window.location.origin + '/settings?tab=github&oauth=callback' }),
  });

  if (!response.ok) throw new Error('Failed to exchange token');

  const token = await response.json();

  // 3. LocalStorage에 저장
  saveToken(token);
  return token;
}
```

### 3. GitHub API 클라이언트 래퍼

토큰을 확보한 후 GitHub API를 호출할 강력한 도구가 필요합니다.

```typescript
// src/HagiCode.Client/src/services/githubApiClient.ts

const GITHUB_API_BASE = 'https://api.github.com';

// 핵심 요청 래퍼
async function githubApi<T>(endpoint: string, options: RequestInit = {}): Promise<T> {
  const token = localStorage.getItem('gh_token');
  if (!token) throw new Error('Not connected to GitHub');

  const response = await fetch(`${GITHUB_API_BASE}${endpoint}`, {
    ...options,
    headers: {
      ...options.headers,
      Authorization: `Bearer ${token}`,
      Accept: 'application/vnd.github.v3+json', // API 버전 지정
    },
  });

  // 오류 처리 로직
  if (!response.ok) {
    if (response.status === 401) throw new Error('GitHub 토큰이 만료되었습니다. 다시 연결해 주세요.');
    if (response.status === 403) throw new Error('이 저장소에 대한 접근 권한이 없거나 속도 제한을 초과했습니다.');
    if (response.status === 422) throw new Error('Issue 검증에 실패했습니다. 제목이 중복되었을 수 있습니다.');
    throw new Error(`GitHub API Error: ${response.statusText}`);
  }

  return response.json();
}

// Issue 생성
export async function createIssue(owner: string, repo: string, data: { title: string, body: string, labels: string[] }) {
  return githubApi(`/repos/${owner}/${repo}/issues`, {
    method: 'POST',
    body: JSON.stringify(data),
  });
}
```

### 4. 콘텐츠 포맷팅과 동기화

마지막 단계는 HagiCode의 Session 데이터를 GitHub Issue 형식으로 변환하는 것입니다. 이는 "번역" 작업과 유사합니다.

```typescript
// Session 객체를 Markdown 문자열로 변환
function formatIssueForSession(session: Session): string {
  let content = `# ${session.title}\n\n`;
  content += `**> HagiCode Session:** #${session.code}\n`;
  content += `**> Status:** ${session.status}\n\n`;
  content += `## Description\n\n${session.description || 'No description provided.'}\n\n`;

  // Proposal 유형인 경우 추가 필드 포함
  if (session.type === 'proposal') {
    content += `## Chief Complaint\n\n${session.chiefComplaint || ''}\n\n`;
    // GitHub에서 HagiCode로 돌아갈 수 있는 딥링크 추가
    content += `---\n\n**[View in HagiCode](hagicode://sessions/${session.id})**\n`;
  }

  return content;
}

// 동기화 버튼 클릭 시 메인 로직
const handleSync = async (session: Session) => {
  try {
    const repoInfo = parseRepositoryFromUrl(session.repoUrl); // 저장소 URL 파싱
    if (!repoInfo) throw new Error('Invalid repository URL');

    toast.loading('GitHub에 동기화 중...');

    // 1. 콘텐츠 포맷팅
    const issueBody = formatIssueForSession(session);

    // 2. API 호출
    const issue = await githubApiClient.createIssue(repoInfo.owner, repoInfo.repo, {
      title: `[HagiCode] ${session.title}`,
      body: issueBody,
      labels: ['hagicode', 'proposal', `status:${session.status}`],
    });

    // 3. Session Metadata 업데이트 (Issue 링크 저장)
    await SessionsService.patchApiSessionsSessionId(session.id, {
      metadata: {
        githubIssue: {
          owner: repoInfo.owner,
          repo: repoInfo.repo,
          issueNumber: issue.number,
          issueUrl: issue.html_url,
          syncedAt: new Date().toISOString(),
        }
      }
    });

    toast.success('동기화가 완료되었습니다!');
  } catch (err) {
    console.error(err);
    toast.error('동기화에 실패했습니다. 토큰이나 네트워크를 확인해 주세요.');
  }
};
```

## 요약 및 전망

이 "프론트엔드 직접 연결" 방안을 통해 최소한의 백엔드 코드로 GitHub Issues의 원활한 통합을 구현했습니다.

### 성과
1.  **높은 개발 효율**: 백엔드 변경이 매우 적습니다. 주로 데이터베이스에 하나의 필드를 추가하고 간단한 OAuth 콜백 엔드포인트를 구현하는 것으로, 대부분의 로직이 프론트엔드에서 완료됩니다.
2.  **우수한 보안**: 토큰이 서버 데이터베이스를 거치지 않아 유출 위험이 감소합니다.
3.  **좋은 사용자 경험**: 프론트엔드에서 직접 요청을 보내므로 응답 속도가 빠르고 백엔드 중계가 필요 없습니다.

### 주의사항
실제 배포 시 몇 가지 주의점이 있습니다:
- **OAuth App 설정**: GitHub OAuth App 설정에 올바른 `Authorization callback URL`을 입력해야 합니다 (일반적으로 `http://localhost:3000/settings?tab=github&oauth=callback`).
- **속도 제한**: GitHub API는 인증되지 않은 요청에 대해 엄격한 제한을 적용하지만, 토큰 사용 시 일반적으로 충분합니다 (시간당 5000회).
- **URL 파싱**: 사용자가 입력하는 저장소 URL은 다양하므로 정규식이 `.git` 접미사, SSH 형식 등의 경우를 처리하도록 해야 합니다.

### 향후 개선
현재 기능은 단방향 동기화(HagiCode -> GitHub)입니다. 향후 GitHub Webhooks를 통해 양방향 동기화를 구현할 계획입니다. 예를 들어 GitHub에서 Issue를 닫으면 HagiCode 측의 세션 상태도 자동으로 업데이트될 수 있습니다. 이를 위해서는 백엔드에 Webhook 수신 엔드포인트를 노출해야 하며, 이는 다음에 해야 할 흥미로운 작업입니다.
