## 처음부터 GitHub Issues 통합 구축하기: HagiCode의 프론트엔드 직접 연결实践

> 이 문서는 HagiCode 플랫폼에서 GitHub Issues 통합의 전체 과정을 기록합니다. "프론트엔드 직접 연결 + 백엔드 최소화" 아키텍처를 통해 백엔드를 가볍게 유지하면서 안전한 OAuth 인증과 효율적인 Issues 동기화를 구현하는 방법에 대해 탐구합니다.

## 배경: GitHub를 통합하는 이유는?

HagiCode는 AI 보조 개발 플랫폼으로서 핵심 가치는 아이디어와 구현을 연결하는 것입니다. 하지만 실제 사용에서 사용자가 HagiCode에서 Proposal(제안)을 완료한 후 수동으로 내용을 GitHub Issues에 복사하여 프로젝트 추적을 해야 하는 경우가 많다는 것을 발견했습니다.

이로 인해 몇 가지 명확한 문제점이 발생했습니다:

1.  **워크플로우 단절**: 사용자는 두 시스템 간을 오가야 하며, 경험이 원활하지 않을 뿐만 아니라 복사 및 붙여넣기 과정에서 핵심 정보가 손실되기 쉽습니다.
2.  **협업 불편**: 팀의 다른 멤버들은 GitHub에서 작업을 확인하는 데 익숙하며, HagiCode의 제안 진행 상황을 직접 볼 수 없습니다.
3.  **반복 작업**: 제안이 업데이트될 때마다 수동으로 GitHub에서 해당 Issue를 업데이트해야 하며, 불필요한 유지 관리 비용이 증가합니다.

이 문제를 해결하기 위해 우리는 **GitHub Issues Integration** 기능을 도입하기로 결정했으며, HagiCode 세션과 GitHub 저장소의 연결을 연결하여 "원 클릭 동기화"를 구현했습니다.

## 기술 선택: 프론트엔드 직접 연결 vs 백엔드 프록시

통합 솔루션을 설계할 때 우리 앞에는 두 가지 길이 있었습니다: 전통적인 "백엔드 프록시 모드"와 더 과감한 "프론트엔드 직접 연결 모드".

### 솔루션 비교

전통적인 **백엔드 프록시 모드**에서 프론트엔드의 모든 요청은 먼저 우리 백엔드를 거쳐야 하며, 백엔드가 GitHub API를 호출합니다. 이것은 논리가 집중되지만 백엔드에 적지 않은 부담을 줍니다:

1.  **백엔드 비대**: 전용 GitHub API 클라이언트 캡슐화를 작성해야 하며 OAuth의 복잡한 상태 머신도 처리해야 합니다.
2.  **Token 위험**: 사용자의 GitHub Token은 백엔드 데이터베이스에 저장해야 하며 암호화할 수 있지만 보안 위험 면적이 증가합니다.
3.  **개발 비용**: Token을 저장하기 위한 데이터베이스 마이그레이션이 필요하며 추가적인 동기화 서비스를 유지 관리해야 합니다.

반면 **프론트엔드 직접 연결 모드**는 훨씬 가볍습니다. 이 솔루션에서 우리는 가장 민감한 "키 교환" 단계(OAuth 콜백)를 처리하기 위해 백엔드만 사용하며 Token을 얻은 후 브라우저의 localStorage에 직접 저장합니다. 이후 Issue 생성, 댓글 업데이트 등의 작업은 프론트엔드에서 GitHub로 HTTP 요청을 직접 보냅니다.

| 비교 차원 | 백엔드 프록시 모드 | 프론트엔드 직접 연결 모드 |
| :--- | :--- | :--- |
| **백엔드 복잡도** | 완전한 OAuth 서비스와 GitHub API 클라이언트 필요 | OAuth 콜백 엔드포인트만 필요 |
| **Token 관리** | 데이터베이스에 암호화 저장, 유출 위험 있음 | 브라우저에 저장, 사용자만 볼 수 있음 |
| **구현 비용** | 데이터베이스 마이그레이션, 다중 서비스 개발 필요 | 주로 프론트엔드 작업량 |
| **사용자 경험** | 논리가 통합되지만 서버 지연이 약간 높을 수 있음 | 매우 빠른 응답, GitHub와 직접 상호작용 |

빠른 통합과 백엔드 변경 최소화를 고려하여 **우리는 최종적으로 "프론트엔드 직접 연결 모드"를 채택했습니다**. 이것은 브라우저에 "임시 통행증"을 발급하는 것과 같습니다. 증서를 받은 후 브라우저는 직접 GitHub로 일을 처리할 수 있으며 매번 백엔드 관리자의 승인을 요청할 필요가 없습니다.

## 핵심 설계: 데이터 흐름과 보안

아키텍처를 확정한 후 구체적인 데이터 흐름을 설계해야 합니다. 전체 동기화 프로세스의 핵심은 Token을 안전하게 얻고 효율적으로 활용하는 방법입니다.

### 전체 아키텍처 다이어그램

전체 시스템은 세 가지 역할로 추상화할 수 있습니다: 브라우저(프론트엔드), HagiCode 백엔드, GitHub.

```text
+--------------+        +--------------+        +--------------+
|  프론트엔드 React  |        |    백엔드      |        |    GitHub    |
|              |        |   ASP.NET    |        |    REST API  |
|  +--------+  |        |              |        |              |
|  |  OAuth |--+--------> /callback    |        |              |
|  |  흐름  |  |        |              |        |              |
|  +--------+  |        |              |        |              |
|              |        |              |        |              |
|  +--------+  |        |  +--------+  |        |  +--------+  |
|  |GitHub  |  +------------>Session |  +----------> Issues |  |
|  |API     |  |        |  |Metadata|  |        |  |        |  |
|  |직렬    |  |        |  +--------+  |        |  +--------+  |
|  +--------+  |        |              |        |              |
+--------------+        +--------------+        +--------------+
```

**핵심은**: OAuth의 작은 단계(code를 token으로 교환)만 백엔드를 거치며, 이후의 거친 일(Issue 생성)은 모두 프론트엔드가 GitHub와 직접 거래합니다.

### 동기화 데이터 흐름 상세 설명

사용자가 HagiCode 인터페이스의 "Sync to GitHub" 버튼을 클릭하면 일련의 복잡한 작업이 발생합니다:

```text
사용자 클릭 "Sync to GitHub"
         │
         ▼
1. 프론트엔드 localStorage 확인하여 GitHub Token 획득
         │
         ▼
2. Issue 내용 포맷(Proposal을 Markdown으로 변환)
         │
         ▼
3. 프론트엔드에서 GitHub API를 직접 호출하여 Issue 생성/업데이트
         │
         ▼
4. HagiCode 백엔드 API 호출하여 Session.metadata 업데이트 (Issue URL 등 정보 저장)
         │
         ▼
5. 백엔드가 SignalR을 통해 SessionUpdated 이벤트 브로드캐스트
         │
         ▼
6. 프론트엔드가 이벤트를 수신하고 UI를 업데이트하여 "동기화됨" 상태 표시
```

### 보안 설계

보안 문제는 항상 타사 서비스 통합의 가장 중요한 요소입니다. 우리는 다음 사항을 고려했습니다:

1.  **CSRF 공격 방지**: OAuth 리다이렉트时 무작위 `state` 매개변수를 생성하고 sessionStorage에 저장합니다. 콜백时 state를 엄격하게 검증하여 요청이 위조되는 것을 방지합니다.
2.  **Token 저장 격리**: Token은 브라우저의 `localStorage`에만 저장되며 동일 출처 정책(Same-Origin Policy)을 활용하여 HagiCode 스크립트만 읽을 수 있으며 서버 측 데이터베이스 유출이 사용자에게 영향을 주는 것을 방지합니다.
3.  **오류 경계**: GitHub API의 일반적인 오류(401 Token 만료, 422 검증 실패, 429 속도 제한)에 대해 전용 오류 처리 논리를 설계하여 사용자에게 친근한 프롬프트를 제공합니다.

### 2. 프론트엔드 OAuth 흐름

이것은 연결의 진입점입니다. 우리는 표준 Authorization Code Flow를 사용합니다.

```typescript
// src/HagiCode.Client/src/services/githubOAuth.ts

// 권한 URL 생성 및 리다이렉트
export async function generateAuthUrl(): Promise<string> {
  const state = generateRandomString(); // CSRF 방지 무작위 문자열 생성
  sessionStorage.setItem('hagicode_github_state', state);
  
  const params = new URLSearchParams({
    client_id: clientId,
    redirect_uri: window.location.origin + '/settings?tab=github&oauth=callback',
    scope: ['repo', 'public_repo'].join(' '),
    state: state,
  });
  
  return `https://github.com/login/oauth/authorize?${params.toString()}`;
}

// 콜백 페이지에서 Code로 Token 교환 처리
export async function exchangeCodeForToken(code: string, state: string): Promise<GitHubToken> {
  // 1. State 확인하여 CSRF 방지
  const savedState = sessionStorage.getItem('hagicode_github_state');
  if (state !== savedState) throw new Error('Invalid state parameter');

  // 2. 백엔드 API 호출하여 Token 교환
  // 주의: 여기는 백엔드를 거쳐야 합니다. ClientSecret이 필요하며 프론트엔드에 노출할 수 없습니다
  const response = await fetch('/api/GitHubOAuth/callback', {
    method: 'POST',
    headers: { 'Content-Type': 'application/json' },
    body: JSON.stringify({ code, state, redirectUri: window.location.origin + '/settings?tab=github&oauth=callback' }),
  });

  if (!response.ok) throw new Error('Failed to exchange token');
  
  const token = await response.json();
  
  // 3. LocalStorage에 저장
  saveToken(token);
  return token;
}
```

### 3. GitHub API 클라이언트 캡슐화

Token이 있으면 GitHub API를 호출할 수 있는 강력한 도구가 필요합니다.

```typescript
// src/HagiCode.Client/src/services/githubApiClient.ts

const GITHUB_API_BASE = 'https://api.github.com';

// 핵심 요청 캡슐화
async function githubApi<T>(endpoint: string, options: RequestInit = {}): Promise<T> {
  const token = localStorage.getItem('gh_token');
  if (!token) throw new Error('Not connected to GitHub');
  
  const response = await fetch(`${GITHUB_API_BASE}${endpoint}`, {
    ...options,
    headers: {
      ...options.headers,
      Authorization: `Bearer ${token}`,
      Accept: 'application/vnd.github.v3+json', // API 버전 지정
    },
  });
  
  // 오류 처리 논리
  if (!response.ok) {
    if (response.status === 401) throw new Error('GitHub Token이 만료되었습니다. 다시 연결해주세요');
    if (response.status === 403) throw new Error('저장소에 대한 권한이 없거나 속도 제한을 초과했습니다');
    if (response.status === 422) throw new Error('Issue 검증에 실패했습니다. 제목이 중복일 수 있습니다');
    throw new Error(`GitHub API Error: ${response.statusText}`);
  }
  
  return response.json();
}

// Issue 생성
export async function createIssue(owner: string, repo: string, data: { title: string, body: string, labels: string[] }) {
  return githubApi(`/repos/${owner}/${repo}/issues`, {
    method: 'POST',
    body: JSON.stringify(data),
  });
}
```

### 4. 내용 포맷팅 및 동기화

마지막 단계는 HagiCode의 Session 데이터를 GitHub Issue 형식으로 변환하는 것입니다. 이것은 "번역" 작업과 비슷합니다.

```typescript
// Session 객체를 Markdown 문자열로 변환
function formatIssueForSession(session: Session): string {
  let content = `# ${session.title}\n\n`;
  content += `**> HagiCode Session:** #${session.code}\n`;
  content += `**> Status:** ${session.status}\n\n`;
  content += `## Description\n\n${session.description || 'No description provided.'}\n\n`;
  
  // Proposal 유형인 경우 추가 필드 포함
  if (session.type === 'proposal') {
    content += `## Chief Complaint\n\n${session.chiefComplaint || ''}\n\n`;
    // GitHub에서 HagiCode로 돌아갈 수 있는 딥 링크 추가
    content += `---\n\n**[View in HagiCode](hagicode://sessions/${session.id})**\n`;
  }
  
  return content;
}

// 동기화 버튼 클릭의 주요 논리
const handleSync = async (session: Session) => {
  try {
    const repoInfo = parseRepositoryFromUrl(session.repoUrl); // 저장소 URL 파싱
    if (!repoInfo) throw new Error('Invalid repository URL');

    toast.loading('GitHub에 동기화 중...');
    
    // 1. 내용 포맷
    const issueBody = formatIssueForSession(session);
    
    // 2. API 호출
    const issue = await githubApiClient.createIssue(repoInfo.owner, repoInfo.repo, {
      title: `[HagiCode] ${session.title}`,
      body: issueBody,
      labels: ['hagicode', 'proposal', `status:${session.status}`],
    });
    
    // 3. Session Metadata 업데이트 (Issue 링크 저장)
    await SessionsService.patchApiSessionsSessionId(session.id, {
      metadata: {
        githubIssue: {
          owner: repoInfo.owner,
          repo: repoInfo.repo,
          issueNumber: issue.number,
          issueUrl: issue.html_url,
          syncedAt: new Date().toISOString(),
        }
      }
    });

    toast.success('동기화 성공!');
  } catch (err) {
    console.error(err);
    toast.error('동기화 실패. Token 또는 네트워크를 확인해주세요');
  }
};
```

## 요약 및 전망

이 "프론트엔드 직접 연결" 솔루션을 통해 우리는 최소한의 백엔드 코드로 GitHub Issues의 원활한 통합을 구현했습니다.

### 수확
1.  **높은 개발 효율성**: 백엔드 변경이 매우 적으며 주로 데이터베이스에 필드 하나를 추가하고 간단한 OAuth 콜백 인터페이스를 추가하는 것이며 대부분의 논리는 프론트엔드에서 완료됩니다.
2.  **우수한 보안**: Token이 서버 데이터베이스를 거치지 않아 유출 위험이 줄어듭니다.
3.  **우수한 사용자 경험**: 프론트엔드에서 직접 요청을发起하여 응답 속도가 빠르며 백엔드 중계를 거칠 필요가 없습니다.

### 주의 사항
실제 배포시 몇 가지 함정에 주의해야 합니다:
- **OAuth App 설정**: GitHub OAuth App 설정에 올바른 `Authorization callback URL`을 기억하세요(일반적으로 `http://localhost:3000/settings?tab=github&oauth=callback`).
- **속도 제한**: GitHub API는 인증되지 않은 요청에 대해 엄격한 제한이 있지만 Token을 사용한 후에는 일반적으로 충분합니다(시간당 5000회).
- **URL 파싱**: 사용자가 입력하는 Repo URL은 다양하며 정규식이 `.git` 접미사, SSH 형식 등을 일치시키도록 해야 합니다.

### 향후 강화
현재 기능은 여전히 단방향 동기화(HagiCode -> GitHub)입니다. 향후 우리는 GitHub Webhooks를 통해 양방향 동기화를 계획하며 예를 들어 GitHub에서 Issue를 닫으면 HagiCode 쪽의 세션 상태도 자동으로 업데이트될 수 있습니다. 이것은 백엔드에 Webhook 수신 엔드포인트를 노출해야 하며 다음 단계에서 할 흥미로운 작업입니다.
