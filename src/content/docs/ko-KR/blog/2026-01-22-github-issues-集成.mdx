## 처음부터 GitHub Issues 통합 구축하기: HagiCode의 프론트엔드 직접 연결 실천

> 이 글에서는 HagiCode 플랫폼에 GitHub Issues를 통합하는 전체 과정을 기록합니다. "프론트엔드 직접 연결 + 백엔드 최소화" 아키텍처를 통해 백엔드를 가볍게 유지하면서 안전한 OAuth 인증과 효율적인 Issues 동기화를 구현하는 방법을 다룹니다.

## 배경: 왜 GitHub를 통합해야 할까요?

HagiCode는 AI 보조 개발 플랫폼으로, 핵심 가치는 아이디어와 구현을 연결하는 데 있습니다. 하지만 실제 사용에서 사용자가 HagiCode에서 Proposal(제안)을 완료한 후, 수동으로 내용을 GitHub Issues에 복사하여 프로젝트를 추적해야 한다는 것을 발견했습니다.

이는 몇 가지 명확한 문제점을 가져왔습니다:

1.  **워크플로우 단절**: 사용자가 두 시스템 사이를 오가야 하므로 경험이 원활하지 않으며, 복사 붙여넣기 과정에서 핵심 정보가 누락되기 쉽습니다.
2.  **협업 불편**: 팀의 다른 멤버들은 GitHub에서 작업을 확인하는 데 익숙하므로, HagiCode의 제안 진행 상황을 직접 볼 수 없습니다.
3.  **중복 작업**: 제안이 업데이트될 때마다 GitHub에서 해당 Issue를 수동으로 업데이트해야 하므로 불필요한 유지보수 비용이 발생합니다.

이 문제를 해결하기 위해 **GitHub Issues Integration** 기능을 도입하여 HagiCode 세션과 GitHub 저장소의 연결을打通하고, "원클릭 동기화"를 구현하기로 결정했습니다.

## HagiCode에 대하여

**스마트** — AI가 전 과정을 보조하여 아이디어에서 코드까지 코딩 효율을 수배로 향상시킵니다. **편리** — 멀티스레드 병렬 작업으로 리소스를 충분히 활용하고 개발 흐름을 원활하게 합니다. **재미** — 게임화 메커니즘과 성취 시스템으로 코딩이 더 이상 지루하지 않고 성취감을 느낄 수 있습니다.

프로젝트가 빠르게 반복 개발 중이며, 기술 글쓰기, 지식 관리 또는 AI 보조 개발에 관심이 있으시다면 [GitHub](https://github.com/HagiCode-org/site)를 방문해 보세요~

## 기술 선택: 프론트엔드 직접 연결 vs 백엔드 프록시

통합 솔루션을 설계할 때, 전통적인 "백엔드 프록시 모델"과 더 적극적인 "프론트엔드 직접 연결 모델" 두 가지 선택지가 있었습니다.

### 솔루션 비교

전통적인 **백엔드 프록시 모델**에서는 프론트엔드의 모든 요청이 먼저 백엔드를 거쳐야 하고, 백엔드가 GitHub API를 호출합니다. 논리는 집중되지만 백엔드에 적지 않은 부담을 줍니다:

1.  **백엔드 비대화**: 전용 GitHub API 클라이언트 캡슐화를 작성해야 하며, OAuth의 복잡한 상태 머신도 처리해야 합니다.
2.  **토큰 위험**: 사용자의 GitHub Token을 백엔드 데이터베이스에 저장해야 하므로 암호화할 수 있지만 보안 위험을 증가시킵니다.
3.  **개발 비용**: Token을 저장하기 위한 데이터베이스 마이그레이션이 필요하고, 추가적인 동기화 서비스도 유지해야 합니다.

반면 **프론트엔드 직접 연결 모델**은 훨씬 가볍습니다. 이 솔루션에서는 가장 민감한 "키 교환" 단계(OAuth callback)만 백엔드에서 처리하고, Token을 획득한 후에는 브라우저의 localStorage에 직접 저장합니다. 이후 Issue 생성, 댓글 업데이트 등의 작업은 프론트엔드에서 직접 HTTP 요청을 GitHub로 보냅니다.

| 비교 차원 | 백엔드 프록시 모델 | 프론트엔드 직접 연결 모델 |
| :--- | :--- | :--- |
| **백엔드 복잡도** | 전체 OAuth 서비스와 GitHub API 클라이언트 필요 | OAuth 콜백 엔드포인트만 필요 |
| **토큰 관리** | 데이터베이스에 암호화 저장, 유출 위험 | 브라우저에 저장, 사용자만 접근 가능 |
| **구현 비용** | 데이터베이스 마이그레이션, 다중 서비스 개발 필요 | 주로 프론트엔드 작업량 |
| **사용자 경험** | 논리가 통일되지만 서버 지연이 약간 높을 수 있음 | GitHub과 직접 상호작용하여 응답이 매우 빠름 |

빠른 통합과 백엔드 변경 최소화를 고려하여 **최종적으로 "프론트엔드 직접 연결 모델"을 채택했습니다**. 이는 브라우저에 "임시 통행증"을 발급하는 것과 같아서, 통행증을 받은 후 브라우저가 직접 GitHub에 가서 업무를 처리할 수 있고, 매번 백엔드 관리자의 승인을 받을 필요가 없습니다.

## 핵심 설계: 데이터 흐름과 보안

아키텍처를 확정한 후, 구체적인 데이터 흐름을 설계해야 합니다. 전체 동기화 프로세스의 핵심은 Token을 안전하게 획득하고 효율적으로 활용하는 것입니다.

### 전체 아키텍처 다이어그램

전체 시스템은 세 가지 역할로 추상화할 수 있습니다: 브라우저(프론트엔드), HagiCode 백엔드, GitHub.

```text
+--------------+        +--------------+        +--------------+
|  前端 React  |        |    后端      |        |    GitHub    |
|              |        |   ASP.NET    |        |    REST API  |
|  +--------+  |        |              |        |              |
|  |  OAuth |--+--------> /callback    |        |              |
|  |  流程  |  |        |              |        |              |
|  +--------+  |        |              |        |              |
|              |        |              |        |              |
|  +--------+  |        |  +--------+  |        |  +--------+  |
|  |GitHub  |  +------------>Session |  +----------> Issues |  |
|  |API     |  |        |  |Metadata|  |        |  |        |  |
|  |直连    |  |        |  +--------+  |        |  +--------+  |
|  +--------+  |        |              |        |              |
+--------------+        +--------------+        +--------------+
```

**핵심 포인트는** OAuth의 작은 단계(code를 토큰으로 교환)만 백엔드를 거치고, 이후의 무거운 작업(Issue 생성)은 모두 프론트엔드가 GitHub과 직접 통신한다는 것입니다.

### 동기화 데이터 흐름 상세

사용자가 HagiCode 인터페이스의 "Sync to GitHub" 버튼을 클릭하면 일련의 복잡한 동작이 발생합니다:

```text
用户点击 "Sync to GitHub"
         │
         ▼
1. 前端检查 localStorage 获取 GitHub Token
         │
         ▼
2. 格式化 Issue 内容（将 Proposal 转换为 Markdown）
         │
         ▼
3. 前端直接调用 GitHub API 创建/更新 Issue
         │
         ▼
4. 调用 HagiCode 后端 API 更新 Session.metadata (存储 Issue URL 等信息)
         │
         ▼
5. 后端通过 SignalR 广播 SessionUpdated 事件
         │
         ▼
6. 前端接收事件，更新 UI 显示"已同步"状态
```

### 보안 설계

보안 문제는 항상 서드파티 서비스 통합의 가장 중요한 부분입니다. 다음 사항을 고려했습니다:

1.  **CSRF 공격 방지**: OAuth 리디렉트 시 무작위 `state` 매개변수를 생성하여 sessionStorage에 저장합니다. 콜백 시 state를 엄격하게 검증하여 요청이 위조되지 않았는지 확인합니다.
2.  **토큰 저장 격리**: Token은 브라우저의 `localStorage`에만 저장되며, 동일 출처 정책(Same-Origin Policy)을 활용하여 HagiCode 스크립트만 읽을 수 있으므로 서버 측 데이터베이스 유출이 사용자에게 영향을 주지 않습니다.
3.  **오류 경계**: GitHub API의 일반적인 오류(예: 401 Token 만료, 422 유효성 검사 실패, 429 속도 제한)에 대해 전용 오류 처리 로직을 설계하여 사용자에게 친절한 안내를 제공합니다.

### 2. 프론트엔드 OAuth 흐름

이것이 연결의 진입점입니다. 표준 Authorization Code Flow를 사용합니다.

```typescript
// src/HagiCode.Client/src/services/githubOAuth.ts

// 生成授权 URL 并跳转
export async function generateAuthUrl(): Promise<string> {
  const state = generateRandomString(); // 生成防 CSRF 的随机串
  sessionStorage.setItem('hagicode_github_state', state);
  
  const params = new URLSearchParams({
    client_id: clientId,
    redirect_uri: window.location.origin + '/settings?tab=github&oauth=callback',
    scope: ['repo', 'public_repo'].join(' '),
    state: state,
  });
  
  return `https://github.com/login/oauth/authorize?${params.toString()}`;
}

// 在回调页面处理 Code 换取 Token
export async function exchangeCodeForToken(code: string, state: string): Promise<GitHubToken> {
  // 1. 验证 State 防止 CSRF
  const savedState = sessionStorage.getItem('hagicode_github_state');
  if (state !== savedState) throw new Error('Invalid state parameter');

  // 2. 调用后端 API 进行 Token 交换
  // 注意：这里必须经过后端，因为需要 ClientSecret，不能暴露在前端
  const response = await fetch('/api/GitHubOAuth/callback', {
    method: 'POST',
    headers: { 'Content-Type': 'application/json' },
    body: JSON.stringify({ code, state, redirectUri: window.location.origin + '/settings?tab=github&oauth=callback' }),
  });

  if (!response.ok) throw new Error('Failed to exchange token');
  
  const token = await response.json();
  
  // 3. 存入 LocalStorage
  saveToken(token);
  return token;
}
```

### 3. GitHub API 클라이언트 캡슐화

Token이 있으면 GitHub API를 호출할 강력한 도구가 필요합니다.

```typescript
// src/HagiCode.Client/src/services/githubApiClient.ts

const GITHUB_API_BASE = 'https://api.github.com';

// 核心请求封装
async function githubApi<T>(endpoint: string, options: RequestInit = {}): Promise<T> {
  const token = localStorage.getItem('gh_token');
  if (!token) throw new Error('Not connected to GitHub');
  
  const response = await fetch(`${GITHUB_API_BASE}${endpoint}`, {
    ...options,
    headers: {
      ...options.headers,
      Authorization: `Bearer ${token}`,
      Accept: 'application/vnd.github.v3+json', // 指定 API 版本
    },
  });
  
  // 错误处理逻辑
  if (!response.ok) {
    if (response.status === 401) throw new Error('GitHub Token 失效，请重新连接');
    if (response.status === 403) throw new Error('无权访问该仓库或超出速率限制');
    if (response.status === 422) throw new Error('Issue 验证失败，可能标题重复');
    throw new Error(`GitHub API Error: ${response.statusText}`);
  }
  
  return response.json();
}

// 创建 Issue
export async function createIssue(owner: string, repo: string, data: { title: string, body: string, labels: string[] }) {
  return githubApi(`/repos/${owner}/${repo}/issues`, {
    method: 'POST',
    body: JSON.stringify(data),
  });
}
```

### 4. 콘텐츠 포매팅 및 동기화

마지막 단계는 HagiCode의 Session 데이터를 GitHub Issue 형식으로 변환하는 것입니다. 이것은 일종의 "번역" 작업과 비슷합니다.

```typescript
// 将 Session 对象转换为 Markdown 字符串
function formatIssueForSession(session: Session): string {
  let content = `# ${session.title}\n\n`;
  content += `**> HagiCode Session:** #${session.code}\n`;
  content += `**> Status:** ${session.status}\n\n`;
  content += `## Description\n\n${session.description || 'No description provided.'}\n\n`;
  
  // 如果是 Proposal 类型，添加额外字段
  if (session.type === 'proposal') {
    content += `## Chief Complaint\n\n${session.chiefComplaint || ''}\n\n`;
    // 添加一个深链接，方便从 GitHub 跳回 HagiCode
    content += `---\n\n**[View in HagiCode](hagicode://sessions/${session.id})**\n`;
  }
  
  return content;
}

// 点击同步按钮的主逻辑
const handleSync = async (session: Session) => {
  try {
    const repoInfo = parseRepositoryFromUrl(session.repoUrl); // 解析仓库 URL
    if (!repoInfo) throw new Error('Invalid repository URL');

    toast.loading('正在同步到 GitHub...');
    
    // 1. 格式化内容
    const issueBody = formatIssueForSession(session);
    
    // 2. 调用 API
    const issue = await githubApiClient.createIssue(repoInfo.owner, repoInfo.repo, {
      title: `[HagiCode] ${session.title}`,
      body: issueBody,
      labels: ['hagicode', 'proposal', `status:${session.status}`],
    });
    
    // 3. 更新 Session Metadata (保存 Issue 链接)
    await SessionsService.patchApiSessionsSessionId(session.id, {
      metadata: {
        githubIssue: {
          owner: repoInfo.owner,
          repo: repoInfo.repo,
          issueNumber: issue.number,
          issueUrl: issue.html_url,
          syncedAt: new Date().toISOString(),
        }
      }
    });

    toast.success('同步成功！');
  } catch (err) {
    console.error(err);
    toast.error('同步失败，请检查 Token 或网络');
  }
};
```

## 요약 및 전망

이 "프론트엔드 직접 연결" 솔루션을 통해 최소한의 백엔드 코드로 GitHub Issues의 원활한 통합을 구현했습니다.

### 성과
1.  **높은 개발 효율**: 백엔드 변경이 매우 적고, 주로 데이터베이스에 필드 하나를 추가하고 간단한 OAuth 콜백 인터페이스를 구현하는 것이며, 대부분의 논리는 프론트엔드에서 완료됩니다.
2.  **우수한 보안**: Token이 서버 데이터베이스를 거치지 않으므로 유출 위험이 감소합니다.
3.  **좋은 사용자 경험**: 프론트엔드에서 직접 요청을 보내므로 응답 속도가 빠르고 백엔드 중계가 필요 없습니다.

### 주의 사항
실제 배포 시 몇 가지 주의할 점이 있습니다:
- **OAuth App 설정**: GitHub OAuth App 설정에서 올바른 `Authorization callback URL`을 입력해야 합니다(일반적으로 `http://localhost:3000/settings?tab=github&oauth=callback`).
- **속도 제한**: GitHub API는 인증되지 않은 요청에 대해 엄격한 제한을 적용하지만, Token 사용 시 일반적으로 충분합니다(시간당 5000회).
- **URL 파싱**: 사용자가 입력하는 Repo URL은 다양하므로, 정규식이 `.git` 접미사, SSH 형식 등을 일치시켜야 합니다.

### 향후 개선
현재 기능은 단방향 동기화(HagiCode -> GitHub)입니다. 향후 GitHub Webhooks를 통해 양방향 동기화를 구현할 계획입니다. 예를 들어 GitHub에서 Issue를 닫으면 HagiCode의 세션 상태도 자동으로 업데이트됩니다. 이를 위해 백엔드에 Webhook 수신 엔드포인트를 노출해야 하며, 이것이 다음 단계에서 할 흥미로운 작업입니다.

이 글이 서드파티 통합 개발에 영감을 주기를 바랍니다! 질문이 있으시면 [HagiCode GitHub](https://github.com/HagiCode-org/site)에 Issue를 올려 토론해 주세요.
