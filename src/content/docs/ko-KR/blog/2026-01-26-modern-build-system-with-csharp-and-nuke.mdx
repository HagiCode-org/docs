## 스크립트 지옥과 작별: 왜 C#로 현대화된 빌드 시스템을 구축하기로 했는가

> HagiCode 프로젝트가 Nuke를 활용하여 유형 안전, 크로스 플랫폼, 고도로 확장 가능한 자동화 빌드 프로세스를 구현하고, 전통적인 빌드 스크립트의 유지보수 문제를 완전히 해결한 방법을 공개합니다.

## 배경

소프트웨어 개발의 긴 여정에서 "빌드"라는 단어는 사랑과 증오를 동시에 불러일으킵니다. 사랑받는 이유는 원클릭으로 코드가 제품이 되는 순간이 프로그래머에게 가장 매력적인 순간이기 때문이고, 증오받는 이유는 어질러진 빌드 스크립트를 유지하는 것이 악몽과 같기 때문입니다.

많은 프로젝트에서 Python으로 스크립트를 작성하거나 XML 구성 파일을 사용하는 데 익숙합니다(`<property>`가 지배하던 그 두려움을 상상해 보세요). 하지만 프로젝트 복잡도가 높아지면서, 특히 HagiCode처럼 프론트엔드와 백엔드, 다중 플랫폼, 다중 언어 혼합 개발을 포함하는 프로젝트에서는 전통적인 빌드 방식이 한계에 직면하기 시작합니다. 스크립트 로직이 분산되고, 유형 검사가 부족하며, IDE 지원이 약합니다... 이러한 문제들은 개발 팀이 자주 넘어지는 작은 함정들입니다.

이러한 문제를 해결하기 위해 HagiCode 프로젝트에서는 **Nuke** — C# 기반의 현대화된 빌드 시스템을 도입하기로 결정했습니다. 이는 단순한 도구가 아니라 빌드 프로세스에 대한 근본적인 재고입니다. 오늘은 왜 이것을 선택했는지, 그리고 어떻게 개발 경험을 "이륙"시켰는지 이야기해 보겠습니다.

## 핵심 분석: 왜 Nuke인가?

"빌드 시스템이那么多 있는데, Make, Gradle, 또는 Shell 스크립트를 직접 사용하면 안 되나? 왜 하필 C#으로 해야 하나?"라고 의아하게 생각하실 수 있습니다.

이는 좋은 질문입니다. Nuke의 핵심 매력은 우리가 가장 익숙한 프로그래밍 언어 기능을 빌드 스크립트의 세계로 가져온다는 데 있습니다.

### 1. 빌드 프로세스 모듈화: Target의 기술

Nuke의 설계 철학은 매우 명확합니다: **모든 것은 Target입니다**.

전통적인 스크립트에서는 수백 줄의 선형 실행 코드를 작성하여 로직이 복잡하게 얽혀 있습니다. 반면 Nuke에서는 빌드 프로세스를 독립적인 `Target`(목표)으로 분해합니다. 각 Target은 하나의 작업만 담당합니다:

- `Clean`: 출력 디렉토리 정리
- `Restore`: 의존성 패키지 복원
- `Compile`: 코드 컴파일
- `Test`: 단위 테스트 실행

이 설계는 단일 책임 원칙에 매우 부합합니다. 블록 조립처럼 이러한 Target을 자유롭게 조합할 수 있습니다. 더 중요한 것은 Nuke가 Target 간의 종속성 관계를 정의할 수 있다는 점입니다. 예를 들어, `Test`를 실행하려면 시스템이 `Compile`을 먼저 실행했는지 자동으로 확인하고, `Compile`을 원하면 당연히 `Restore`를 먼저 해야 합니다.

이러한 종속성 그래프는 로직을 더 명확하게 만들 뿐만 아니라 실행 효율을 크게 높이며, Nuke가 최적의 실행 경로를 자동으로 분석합니다.

### 2. 유형 안전: 오타의 악몽과 작별

Python으로 빌드 스크립트를 작성해 본 분들은 이런 당혹스러운 상황을 겪어 보셨을 것입니다. 스크립트가 5분 동안 실행된 후 `Confi.guration` 오타가 있다는 오류가 발생하거나, 숫자여야 할 매개변수에 문자열을 전달한 경우입니다.

C#으로 빌드 스크립트를 작성하는 가장 큰 장점은 **유형 안전**입니다. 이는 다음을 의미합니다:

- **컴파일 시 검사**: 코드를 작성할 때 IDE가 오류를 알려주므로 런타임까지 기다릴 필요가 없습니다.
- **리팩토링无忧**: 변수명이나 메서드명을 변경하려면 IDE의 리팩토링 기능으로 원클릭 해결됩니다. 전역 검색 및 바꾸기로 조심스럽게 작업할 필요가 없습니다.
- **스마트 제안**: 강력한 IntelliSense가 코드를 자동 완성하므로 문서를 뒤져 낯선 API를 기억할 필요가 없습니다.

### 3. 크로스 플랫폼: 통일된 빌드 경험

이전에는 Windows에서 `.bat`, Linux에서 `.sh`를 작성하고, 두 플랫폼을 모두 지원하기 위해 Python 스크립트를 작성해야 했습니다. 이제 .NET Core(현 .NET 5+)가 실행되는 곳이라면 어디서나 Nuke를 실행할 수 있습니다.

즉, 팀원이 Windows, Linux 또는 macOS를 사용하든, Visual Studio, VS Code 또는 Rider를 사용하든 모두 동일한 로직을 실행합니다. 이는 "내 컴퓨터에서는 되는데"라는 환경 차이로 인한 문제를 크게 제거합니다.

### 4. 매개변수 및 구성 관리

Nuke는 매우 우아한 매개변수 파싱 메커니즘을 제공합니다. `string[] args`를 수동으로 파싱할 필요 없이, 속성을 정의하고 `[Parameter]` 특성을 추가하기만 하면 Nuke가 명령줄 매개변수와 구성 파일의 매핑을 자동으로 처리합니다.

예를 들어, 빌드 구성을 쉽게 정의할 수 있습니다:

```csharp
[Parameter("Configuration to build - Default is 'Debug'")]
readonly Configuration BuildConfiguration = IsLocalBuild ? Configuration.Debug : Configuration.Release;

Target Compile => _ => _
    .DependsOn(Restore)
    .Executes(() =>
    {
        // BuildConfiguration을 사용합니다. 유형 안전합니다
        DotNetBuild(s => s
            .SetConfiguration(BuildConfiguration)
            .SetProjectFile(SolutionFile));
    });
```

이 작성 방식은 직관적이고 오류가 발생하기 어렵습니다.

## 실천 가이드: 프로젝트에 어떻게 적용하는가

말만 하지 말고 실천합시다. HagiCode 프로젝트에서 이 방안을 어떻게 구체적으로 적용했는지 살펴보겠습니다.

### 1. 프로젝트 구조 기획

빌드 스크립트가 프로젝트 루트 디렉토리를 오염시키는 것을 원하지 않았고, 일부 Java 프로젝트처럼 디렉토리 구조가 너무 깊어지는 것도 원하지 않았습니다. 따라서 Nuke 관련 모든 빌드 파일을 `nukeBuild/` 폴더에 통합하여 배치했습니다.

이렇게 하면 다음과 같은 이점이 있습니다:
- 프로젝트 루트 디렉토리가 깔끔하게 유지됩니다.
- 빌드 로직이 응집되어 관리하기 편합니다.
- 새로운 팀원이 합류할 때 "아, 이것이 빌드 관련 로직이구나"라고 한눈에 알 수 있습니다.

### 2. 명확한 Target 종속성 체인 설계

Target을 설계할 때 하나의 원칙을 따랐습니다: **원자화 + 종속성 흐름**.

각 Target은 충분히 작고 하나의 작업만 수행해야 합니다. 예를 들어 `Clean`은 파일 삭제만 담당하고, 그 안에서 패키징을 겸해서 해서는 안 됩니다.

권장되는 종속성 흐름은 대략 다음과 같습니다:

`Clean` -> `Restore` -> `Compile` -> `Test` -> `Pack`

물론 이것은 절대적인 것은 아닙니다. 예를 들어 테스트만 실행하고 패키징을 하고 싶지 않다면, Nuke에서 `nuke Test`를 직접 실행할 수 있으며, 사전에 필요한 Restore 및 Compile 단계를 자동으로 처리합니다.

### 3. 완비된 오류 처리와 로깅

빌드 스크립트가 가장 두려워하는 것은 무엇일까요? 바로 불명확한 오류 메시지입니다. 예를 들어 빌드가 실패했는데 로그에 "Error: 1"만 표시되면 매우 답답합니다.

Nuke에서는 C#의 예외 처리 메커니즘을 직접 사용할 수 있으므로, 오류를 정밀하게 캡처하고 보고할 수 있습니다.

```csharp
Target Publish => _ => _
    .DependsOn(Test)
    .Executes(() =>
    {
        try
        {
            // NuGet에 게시 시도
            DotNetNuGetPush(s => s
                .SetTargetPath(ArtifactPath)
                .SetSource("https://api.nuget.org/v3/index.json")
                .SetApiKey(ApiKey));
        }
        catch (Exception ex)
        {
            Log.Error($"게시에 실패했습니다. 키가 올바른지 확인해 주세요: {ex.Message}");
            throw; // 빌드 프로세스가 비정상 종료 코드로 종료되도록 보장
        }
    });
```

### 4. 통합 테스트로 품질 보장

빌드 스크립트 자체도 코드이므로 테스트가 필요합니다. Nuke는 빌드 프로세스에 대한 테스트를 작성할 수 있게 하여, 빌드 로직을 수정한 후 기존 게시 프로세스가 손상되지 않도록 보장합니다. 이는 지속적 통합(CI) 파이프라인에서 특히 중요합니다.

## 요약

Nuke를 도입함으로써 HagiCode의 빌드 프로세스는 전례 없이 원활해졌습니다. 이는 단순한 도구 교체가 아니라 엔지니어링 사고의 향상입니다.

**우리가 얻은 것은?**
- **유지보수성**: 코드即구성, 로직이 명확하여 신규 팀원도 빠르게 익힐 수 있습니다.
- **안정성**: 강유형 검사로 90% 이상의 사소한 오류를 감소시켰습니다.
- **일관성**: 크로스 플랫폼의 통일된 경험으로 환경 차이를 제거했습니다.

이전에 빌드 스크립트를 작성하는 것이 "어둠 속에서 더듬는 것"이었다면, Nuke를 사용하는 것은 "불을 켜고 밤길을 걷는 것"과 같습니다. 디버깅하기 어려운 스크립트 언어의 유지보수에 지치셨다면, 빌드 로직을 C#의 세계로 옮겨보세요. 빌드도 이렇게 우아할 수 있다는 것을 발견하실 수 있습니다.
