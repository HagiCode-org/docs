## 스크립트 지옥 작별: C#으로 현대적 빌드 시스템을 구축하기로 결정한 이유

> HagiCode 프로젝트가 Nuke를 활용하여 타입 안전성, 크로스 플랫폼, 고도로 확장 가능한 자동화 빌드 프로세스를 구현하고, 전통적인 빌드 스크립트의 유지 관리 문제를 완전히 해결하는 방법을 공개합니다.

## 배경

소프트웨어 개발의 긴 여정에서 "빌드"라는 단어는 사랑과 증오가 공존하는 감정을 불러일으킵니다. 사랑하는 이유는 버튼 하나로 코드가 제품으로 변하는 순간, 그것은 프로그래머에게 가장 매력적인 순간이기 때문입니다. 증오하는 이유는 그 지저분한 빌드 스크립트들을 유지 관리하는 것이 악몽과 같기 때문입니다.

많은 프로젝트에서 우리는 Python으로 스크립트를 작성하거나 XML 구성 파일을 사용하는 데 익숙해져 있습니다 (`<property>`에 지배되던 공포를 상상해 보세요). 하지만 프로젝트 복잡도가 높아짐에 따라, 특히 HagiCode처럼 프론트엔드와 백엔드, 다중 플랫폼, 다중 언어 혼합 개발이 포함된 프로젝트에서는 전통적인 빌드 방식이 역부족이라는 것이 드러나기 시작했습니다. 스크립트 로직이 분산되어 있고, 타입 검사가 부족하며, IDE 지원이 약합니다... 이러한 문제들은 작은 구덩이처럼 개발 팀이 수시로 걸려 넘어지게 만듭니다.

이러한 문제점들을 해결하기 위해 HagiCode 프로젝트에서 우리는 **Nuke**를 도입하기로 결정했습니다. 이것은 단순한 도구가 아니라 빌드 프로세스에 대한 재고와도 같습니다. 오늘 우리는 왜 이것을 선택했는지, 그리고 어떻게 우리의 개발 경험을 "날아오르게" 했는지에 대해 이야기해 보겠습니다.

## HagiCode 소개

> 안녕하세요, 우리가 만들고 있는 것을 소개합니다

우리는 **HagiCode**를 개발하고 있습니다. 이것은 AI 기반 코드 지능 어시스턴트로, 개발 경험을 더 스마트하고, 더 편리하며, 더 흥미롭게 만들어 줍니다.

**스마트** — AI가 전 과정을 지원하여 아이디어부터 코드까지, 코딩 효율성을 수배로 향상시킵니다. **편리** — 멀티스레드 동시 작업으로 자원을 충분히 활용하고, 개발 프로세스가 원활합니다. **흥미** — 게이미피케이션 메커니즘과 성취 시스템으로 코딩이 지루하지 않고 성취감이 넘칩니다.

## 핵심 분석: 왜 Nuke인가?

아마 "빌드 시스템이那么多은데, Make, Gradle, 심지어 Shell 스크립트로 직접 하면 안 돼? 왜 굳이 C#을 써야 하지?"라고 생각하고 계실 겁니다.

이것은 정말 좋은 질문입니다. Nuke의 핵심 매력은 가장 익숙한 프로그래밍 언어 특성을 빌드 스크립트의 세계로 가져왔다는 것입니다.

### 1. 빌드 프로세스 모듈화: Target의 예술

Nuke의 디자인 철학은 매우 명확합니다: **모든 것은 대상(Target)**입니다.

전통적인 스크립트에서는 수백 줄의 선형 실행 코드를 작성할 수 있으며, 로직이 복잡하게 얽혀 있습니다. 하지만 Nuke에서는 빌드 프로세스를 독립적인 `Target`(대상)으로 분해합니다. 각 대상은 한 가지 일만 담당합니다. 예를 들어:

- `Clean`: 출력 디렉터리 정리
- `Restore`: 종속성 패키지 복원
- `Compile`: 코드 컴파일
- `Test`: 단위 테스트 실행

이 디자인은 단일 책임 원칙에 매우 부합합니다. 블록을 쌓듯이 이러한 Target을 자유롭게 조합할 수 있습니다. 더 중요한 것은 Nuke가 Target 간의 종속성을 정의할 수 있다는 것입니다. 예를 들어, `Test`를 원하면 시스템이 자동으로 먼저 `Compile`을 실행했는지 확인하고, `Compile`을 원하면 자연스럽게 먼저 `Restore`를 해야 합니다.

이러한 종속성 그래프는 로직을 더 명확하게 만들 뿐만 아니라 실행 효율성을 크게 높입니다. Nuke는 자동으로 최적의 실행 경로를 분석합니다.

### 2. 타입 안전성: 오타의 악몽 작별

Python으로 빌드 스크립트를 작성해 본 친구들은 분명 이러한 당황스러운 상황을 겪었을 겁니다: 스크립트가 5분 동안 실행되고 마지막에 `Confi.guration` 철자가 틀렸다는 오류가 발생하거나, 숫자여야 하는 매개변수에 문자열을 전달했습니다.

C#으로 빌드 스크립트를 작성하는 가장 큰 장점은 **타입 안전성**입니다. 이것은 다음을 의미합니다:

- **컴파일 타임 검사**: 코드를 입력할 때 IDE가 어디가 잘못되었는지 알려주므로, 런타임이 될 때까지 기다릴 필요가 없습니다.
- **리팩토링无忧**: 변수나 메서드 이름을 바꾸고 싶다면 IDE의 리팩토링 기능으로 한 번에 해결할 수 있으며, 전역 검색 및 바꾸기로 조마조마할 필요가 없습니다.
- **스마트 완성**: 강력한 IntelliSense가 자동으로 코드를 완성하므로, 낯선 API를 문서에서 찾아서 기억할 필요가 없습니다.

### 3. 크로스 플랫폼: 통일된 빌드 경험

이전에는 Windows에서 `.bat`를 작성하고 Linux에서 `.sh`를 작성했으며, 두 플랫폼 모두와 호환되도록 하려면 Python 스크립트를 작성해야 했습니다. 이제 .NET Core(현재 .NET 5+)가 실행되는 모든 곳에서 Nuke가 실행됩니다.

이것은 팀 구성원이 Windows, Linux 또는 macOS를 사용하든, Visual Studio, VS Code 또는 Rider를 사용하든 상관없이 모두가 동일한 로직을 실행한다는 것을 의미합니다. 이것은 "내 컴퓨터에서는 실행되는데"와 같은 환경 차이로 인한 문제를 크게 줄여줍니다.

### 4. 매개변수 및 구성 관리

Nuke는 매우 우아한 매개변수 분석 메커니즘을 제공합니다. `string[] args`를 수동으로 분석할 필요가 없으며, 속성을 정의하고 `[Parameter]` 특성을 추가하기만 하면 Nuke가 자동으로 명령줄 매개변수와 구성 파일의 매핑을 처리합니다.

예를 들어, 빌드 구성을 쉽게 정의할 수 있습니다:

```csharp
[Parameter("Configuration to build - Default is 'Debug'")]
readonly Configuration BuildConfiguration = IsLocalBuild ? Configuration.Debug : Configuration.Release;

Target Compile => _ => _
    .DependsOn(Restore)
    .Executes(() =>
    {
        // 여기서 BuildConfiguration을 사용합니다. 타입 안전합니다.
        DotNetBuild(s => s
            .SetConfiguration(BuildConfiguration)
            .SetProjectFile(SolutionFile));
    });
```

이러한 작성 방식은 직관적이고 오류가 발생하기 어렵습니다.

## 실무 가이드: 프로젝트에 적용하는 방법

빈말은 국가를 망치고, 실천은 국가를 번성하게 합니다. HagiCode 프로젝트에서 이 솔루션을 구체적으로 어떻게 적용했는지 살펴보겠습니다.

### 1. 프로젝트 구조 계획

우리는 빌드 스크립트가 프로젝트 루트 디렉터리를 오염시키는 것을 원하지 않았고, 일부 Java 프로젝트처럼 디렉터리 구조가 깊어지는 것도 원하지 않았습니다. 따라서 Nuke와 관련된 모든 빌드 파일을 `nukeBuild/` 폴더에 통합하여 배치했습니다.

이렇게 하면 다음과 같은 이점이 있습니다:
- 프로젝트 루트 디렉터리가 깔끔하게 유지됩니다.
- 빌드 로직이 응집되어 있고 관리하기 쉽습니다.
- 새로운 팀원이 합류하면 한눈에 "아, 이것은 빌드 관련 로직이구나"라고 볼 수 있습니다.

### 2. 명확한 Target 종속성 체인 디자인

Target을 디자인할 때 우리는 하나의 원칙을 따랐습니다: **원자화 + 종속성 흐름**.

각 Target은 충분히 작아야 하며 한 가지 일만 해야 합니다. 예를 들어 `Clean`은 파일 삭제만 담당하고, 그 안에서 패키징을 해서는 안 됩니다.

권장되는 종속성 흐름은 대략 다음과 같습니다:

`Clean` -> `Restore` -> `Compile` -> `Test` -> `Pack`

물론 이것은 절대적인 것은 아닙니다. 예를 들어 테스트만 실행하고 패키징하고 싶지 않다면 `nuke Test`를 직접 실행할 수 있으며, Nuke가 자동으로 사전의 Restore 및 Compile 단계를 처리합니다.

### 3. 완벽한 오류 처리 및 로깅

빌드 스크립트가 가장 두려워하는 것은 무엇일까요? 바로 명확하지 않은 오류 메시지입니다. 예를 들어 빌드가 실패했는데 로그에 "Error: 1"만 표시되면 정말 답답합니다.

Nuke에서는 C# 예외 처리 메커니즘을 직접 사용할 수 있으므로 오류를 매우 정확하게 포착하고 보고할 수 있습니다.

```csharp
Target Publish => _ => _
    .DependsOn(Test)
    .Executes(() =>
    {
        try 
        {
            // NuGet에 게시하려고 시도합니다.
            DotNetNuGetPush(s => s
                .SetTargetPath(ArtifactPath)
                .SetSource("https://api.nuget.org/v3/index.json")
                .SetApiKey(ApiKey));
        }
        catch (Exception ex)
        {
            Log.Error($"게시가 실패했습니다. 형님들 키가 올바른지 확인해 주세요: {ex.Message}");
            throw; // 빌드 프로세스가 0이 아닌 종료 코드로 종료되도록 합니다.
        }
    });
```

### 4. 품질 보장을 위한 통합 테스트

빌드 스크립트 자체도 코드이므로 테스트가 필요합니다. Nuke는 빌드 프로세스에 대한 테스트를 작성할 수 있게 하므로, 빌드 로직을 수정한 후 기존 릴리스 프로세스를 파기하지 않도록 할 수 있습니다. 이것은 지속적 통합(CI) 파이프라인에서 특히 중요합니다.

## 요약

Nuke를 도입함으로써 HagiCode의 빌드 프로세스는 전례 없이 원활해졌습니다. 이것은 단순한 도구 교체가 아니라 엔지니어링 사고의 향상입니다.

**우리가 얻은 것은 무엇인가요?**
- **유지 관리성**: 코드가 곧 구성이며, 로직이 명확하고 신규 팀원도 빠르게 시작할 수 있습니다.
- **안정성**: 강력한 타입 검사로 90% 이상의 사소한 오류가 줄어듭니다.
- **일관성**: 크로스 플랫폼의 통일된 경험으로 환경 차이를 제거했습니다.

예전에 빌드 스크립트를 작성하는 것이 "어둠 속에서 더듬는 것"과 같았다면, Nuke를 사용하는 것은 "불을 켜고 밤길을 걷는 것"과 같습니다. 디버깅하기 어려운 스크립트 언어 유지 관리에 지쳤다면 빌드 로직을 C# 세상으로 가져와 보세요. 아마 빌드도 이렇게 우아할 수 있다는 것을 발견할 수도 있습니다.
