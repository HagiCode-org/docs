## 사용자 정의 구성 관리 모범 사례

> 별거 없네요, 그냥 구성 관리 방법을 공유하는 것뿐입니다. 아키텍처부터 실천까지, 뭐랄까, 덜 헤매는 데 도움이 되기를 바랍니다.

## 배경

구성 관리라는 건, 중요하면 중요하고 사소하면 사소합니다. 마치 삶의 작은 습관들과 같아요——아침 일찍 커피를 마시는 사람을 좋아하는 사람, 밤새 코드를 작성하는 것을 좋아하는 사람, 이看似 사소한 선호도는 사실 당신이 누구인지를 조용히 정의하고 있습니다. 사용자 정의 구성도 마찬가지로, 테마 전환, 언어 선택, 단축키 사용자 정의, 이 기능들을 잘하면 사용자는 이 제품이 "나를 이해한다"고 느끼게 되고, 자연스럽게 충성도가 올라갑니다.

하지만 이 시스템을 완벽하게 만드는 것도 상상보다 간단하지 않습니다. 버전 관리,向后 호환성, 데이터 검증, 동시성 제어, DLC 기능 게이트——이 문제들은 사춘기의烦恼처럼 하나씩 나타나고 피할 수 없습니다. 우리는 HagiCode의 개발 과정에서도 이 문제들로 고통받았지만, 다행히 마지막에는 약간의 요령을 터득했습니다.

## HagiCode에 대하여

이 방안은 사실 우리가 HagiCode에서摸索해낸 것입니다. HagiCode는 AI 코드助手로, 기능이 상당히 많습니다——인터페이스 언어, AI 언어 선호도, 테마, 음성 인식, 알림, 빠른 작업, 구성할 수 있는 것이 적지 않습니다. 사용자가 원하는 것이 너무 많기 때문에 우리는 이 시스템을 만들 수밖에 없었습니다.

프로젝트 주소：[github.com/HagiCode-org/site](https://github.com/HagiCode-org/site)

## 아키텍처 설계

### 계층형 아키텍처

HagiCode의 구성 관리 시스템, 뭐랄까, 몇 개의 계층으로 나누어져 있고 각자의 역할을 수행합니다：

```
┌─────────────────────────────────────────┐
│         Frontend (React + Redux)        │
│  - 구성 상태 관리                          │
│  - UI 양식 렌더링                           │
│  - 그룹별 구성持久化                        │
└─────────────────────────────────────────┘
                    ↓ HTTP/REST API
┌─────────────────────────────────────────┐
│   Application Service Layer             │
│  - FrontendConfigAppService             │
│  - 비즈니스 로직 처리                      │
│  - 권한 제어                               │
└─────────────────────────────────────────┘
                    ↓
┌─────────────────────────────────────────┐
│   Domain Layer (Config Store)           │
│  - FrontendConfigStore                  │
│  - 구성 읽기/쓰기                         │
│  - 데이터 검증及정규화                     │
└─────────────────────────────────────────┘
                    ↓
┌─────────────────────────────────────────┐
│   Infrastructure Layer                  │
│  - YAML 파일 저장                         │
│  - ISystemManagedVaultService           │
│  - 동시성 제어 (SemaphoreSlim)           │
└─────────────────────────────────────────┘
```

이렇게 계층화하는 장점도 상당히 명확합니다：
- **책임 명확**：각 계층은 각자의 일을 관리하고 서로 방해하지 않습니다
- **테스트 용이**：각 계층을 개별적으로 테스트할 수 있고 수정할 때도 안심할 수 있습니다
- **유연한 확장**：저장 방식을 바꾸거나 API를 수정해도 다른 계층은 여전히 작동합니다

### 핵심 인터페이스 설계

백엔드 구성 저장소 인터페이스는 대략 다음과 같이 정의됩니다：

```csharp
public interface IFrontendConfigStore
{
    // 사용자 전체 구성 가져오기
    Task<FrontendConfigStoreResult> GetAsync(CancellationToken cancellationToken = default);
    
    // 구성 업데이트（부분 업데이트 지원）
    Task<FrontendConfigStoreResult> UpdateAsync(
        UpdateFrontendConfigRequestDto input,
        CancellationToken cancellationToken = default);
    
    // AI 언어 상태 관리
    Task<FrontendConfigAiLanguageState> GetAiLanguageStateAsync(
        string userId,
        CancellationToken cancellationToken = default);
        
    Task<FrontendConfigAiLanguageState> SetAiLanguageAsync(
        string userId,
        string language,
        CancellationToken cancellationToken = default);
}
```

이 인터페이스의 몇 가지 핵심 포인트도 이해하기 쉽습니다：
- **비동기 작업**：모두 비동기이며, 아무도 기다리고 싶지 않기 때문입니다
- **취소 토큰**：작업이 너무 오래 걸리면 시간 초과되고 계속 기다리지 마세요
- **부분 업데이트**：필요한 부분만 업데이트하고 전체 구성을 뒤집을 필요가 없습니다

- **DLC 게이트**：고급 기능은 DLC와 바인딩할 수 있으며 좋은 것을 사용하려면 비용을 지불해야 합니다

## 데이터 검증及정규화

### 언어 정규화 예시

구성 정규화,说白了就是让数据保持一致。언어 구성을 예로 들면：

```csharp
public static class FrontendConfigLanguageRules
{
    // 지원되는 인터페이스 언어 목록
    public static readonly HashSet<string> ValidUiLanguages = new(StringComparer.OrdinalIgnoreCase)
    {
        "zh-CN", "zh-Hant", "en-US", "ja-JP", "ko-KR", 
        "de-DE", "fr-FR", "es-ES", "pt-BR", "ru-RU"
    };
    
    public static string NormalizeUiLanguage(string? value)
    {
        // 공백 값 처리：기본 언어 반환
        if (string.IsNullOrWhiteSpace(value)) return "en-US";
        
        var normalized = value.Trim();
        
        // 별칭 처리：일반적인 별칭을 표준 코드로 변환
        normalized = normalized.ToLower() switch
        {
            "zh" or "chinese" or "cn" => "zh-CN",
            "en" or "english" => "en-US",
            "ja" or "japanese" => "ja-JP",
            "ko" or "korean" => "ko-KR",
            _ => normalized
        };
        
        // 방언变体 처리
        if (normalized.StartsWith("zh-Hans", StringComparison.OrdinalIgnoreCase))
            return "zh-CN";
        if (normalized.StartsWith("zh-TW", StringComparison.OrdinalIgnoreCase))
            return "zh-Hant";
            
        // 검증 및 반환
        return ValidUiLanguages.Contains(normalized) ? normalized : "en-US";
    }
}
```

정규화란, 사실 방을 정리하는 것과 같습니다——물건이混乱해지면 정리해야 하고 그렇지 않으면 마지막에는 자신도 찾을 수 없습니다：
1. **공백 값 백업**：공백 값에 합리적인 기본값을 제공하고 비워둘 수는 없습니다
2. **별칭 매핑**：일반적인 별칭, 약어를统一하여 표준 형식으로 변환
3. **방언归一**：방언变体를 표준 코드로 병합, 코드 작성은 방언 연구가 아니기 때문입니다
4. **최종 검증**：반환된 값이 반드시 유효한 목록에 있는지 확인하고 그렇지 않으면白忙了

## DLC 기능 게이트

HagiCode는 DLC 기능 스위치를 지원하며 일부 고급 구성 항목은 DLC를 구매해야 사용할 수 있습니다。이것은 상업화 소프트웨어에서 상당히 일반적입니다——기본 기능은 무료이고 좋은 것을 사용하려면 돈을 내야 합니다,毕竟개발자도 먹고 살아야 하니까요。

### DLC 액세스 검사

```csharp
private async Task<PreparedFrontendConfigUpdate> PrepareUpdateAsync(
    UpdateFrontendConfigRequestDto input,
    FrontendConfigStoreResult current,
    CancellationToken cancellationToken)
{
    // DLC 액세스 권한 확인
    var accessState = await _grainFactory
        .GetDlcAccessStateGrain(TurboEngineDlcId)
        .GetAccessStateAsync();
        
    var blockedFields = new List<string>();
    
    if (!accessState.IsActive)
    {
        // DLC가 활성화되지 않은 경우 현재 구성 유지
        if (input.GeneralSettings?.BrandingLogo != null)
        {
            blockedFields.Add("brandingLogo");
            // 이전 값 유지
            input.GeneralSettings.BrandingLogo = current.Snapshot.GeneralSettings.BrandingLogo;
        }
        
        if (input.GeneralSettings?.BrandingTitle != null)
        {
            blockedFields.Add("brandingTitle");
            input.GeneralSettings.BrandingTitle = current.Snapshot.GeneralSettings.BrandingTitle;
        }
    }
    
    return new PreparedFrontendConfigUpdate(
        input,
        new FrontendConfigUpdateDiagnosticsDto
        {
            Status = blockedFields.Count > 0 ? "partially-applied" : "success",
            BlockedFields = blockedFields,
        });
}
```

## 프론트엔드 상태 관리

### Redux Slice

프론트엔드는 Redux로 구성 상태를 관리하며, 사실 상당히常规的입니다：

```typescript
export const frontendConfigSlice = createSlice({
  name: 'frontendConfig',
  initialState,
  reducers: {
    setConfigStatus(state, action: PayloadAction<FrontendConfigStatus>) {
      state.status = action.payload;
    },
    updateConfigGroups(state, action: PayloadAction<Partial<FrontendConfigSnapshot>>) {
      // 구성 업데이트 병합
      Object.assign(state.snapshot, action.payload);
    },
  },
  extraReducers: (builder) => {
    builder
      .addCase(fetchConfig.pending, (state) => {
        state.status = 'loading';
      })
      .addCase(fetchConfig.fulfilled, (state, action) => {
        state.status = 'succeeded';
        state.snapshot = action.payload;
      })
      .addCase(fetchConfig.rejected, (state, action) => {
        state.status = 'failed';
        state.error = action.error.message;
      });
  },
});
```

## 동시성 제어

구성 업데이트 작업은 반드시 스레드 안전을 보장해야 합니다, 그렇지 않으면 두 사람이 동시에 구성을 수정하면 마지막에 저장되는 것은 누구의 것일까요？HagiCode는 `SemaphoreSlim`으로 동시성 제어를 하며, 뭐랄까,也算个常见的招了：

```csharp
private readonly SemaphoreSlim _semaphore = new(1, 1);

public async Task<FrontendConfigStoreResult> UpdateAsync(
    UpdateFrontendConfigRequestDto input,
    CancellationToken cancellationToken = default)
{
    await _semaphore.WaitAsync(cancellationToken);
    try
    {
        // 현재 구성 읽기
        var current = await GetAsync(cancellationToken);
        
        // 업데이트 준비
        var prepared = await PrepareUpdateAsync(input, current, cancellationToken);
        
        // 구성 쓰기
        await WriteConfigAsync(prepared.UpdatedConfig, cancellationToken);
        
        return new FrontendConfigStoreResult(prepared.UpdatedConfig, prepared.Diagnostics);
    }
    finally
    {
        _semaphore.Release();
    }
}
```

### 일반적인 문제 처리

1. **구성 손실**：백업及복구 메커니즘,总得有备无患
2. **동시성 충돌**：낙관적 잠금 또는 비관적 잠금, 예：SemaphoreSlim
3. **성능 문제**：부분 업데이트 지원,每次整个구성 객체를 이리저리 옮기지 마세요
4. **보안**：민감한 구성（예：API 키）은 반드시 암호화 저장, 그렇지 않으면 도난당하면麻烦了

## 요약

완벽한 구성 관리 시스템은, 뭐랄까, 고려해야 할 것이 상당히 많습니다——데이터 구조, 검증정규화, 버전 관리, DLC 게이트, 동시성 제어, 하나도 빠질 수 없습니다。HagiCode의 이 방안은生产환경에서 상당히 안정적으로 실행되며, 최소한 복잡한 구성 관리 요구를 충족할 수 있습니다。

이 문서를 여기까지 쓰면서, 자신의 약간의 경험을 공유했습니다。좋은 구성 관리는 사용자 경험이 좋을 뿐만 아니라 유지 관리 비용도 낮출 수 있고 제품 반복도 더省心罢了。
