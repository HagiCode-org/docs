## HagiCode에서 StreamJsonRpc의 심층 통합과 실천

> 이 글에서는 HagiCode(구 PCode) 프로젝트가 Microsoft의 StreamJsonRpc 통신 라이브러리를 성공적으로 통합하여 기존의 커스텀 JSON-RPC 구현을 대체하고, 통합 과정에서의 기술적 문제점과 아키텍처 과제를 해결한 방법을 상세히 소개합니다.

## 배경

StreamJsonRpc는 Microsoft에서 공식적으로 유지보수하는 .NET 및 TypeScript용 JSON-RPC 통신 라이브러리로, 강력한 유형 안전성, 자동 프록시 생성 및 성숙한 예외 처리 메커니즘으로 유명합니다. HagiCode 프로젝트에서는 ACP (Agent Communication Protocol)를 통해 외부 AI 도구(iflow CLI, OpenCode CLI 등)와 통신하기 위해, 그리고 초기 커스텀 JSON-RPC 구현으로 인한 유지보수 비용과 잠재적 버그를 제거하기 위해 StreamJsonRpc 통합을 결정했습니다. 그러나 통합 과정에서 스트리밍 JSON-RPC 특유의 과제에 직면했으며, 특히 프록시 대상 바인딩과 제네릭 매개변수 식별 문제가 있었습니다.

이러한 문제를 해결하기 위해 대담한 결정을 내렸습니다: 전체 빌드 시스템을 완전히 재설계합니다. 이 결정으로 인한 변화는 상상보다 클 것입니다 — 자세한 내용은 나중에 설명하겠습니다.

## 분석

현재 프로젝트는 ACP 프로토콜 통합의 핵심 단계에 있으며, 다음과 같은 기술적 문제점과 아키텍처 과제에 직면해 있습니다:

### 1. 커스텀 구현의 한계
기존 JSON-RPC 구현은 `src/HagiCode.ClaudeHelper/AcpImp/`에 위치하며, `JsonRpcEndpoint` 및 `ClientSideConnection` 등의 컴포넌트를 포함합니다. 이 커스텀 코드를 유지하는 비용이 높고, 성숙한 라이브러리의 고급 기능(진행률 보고, 취소 지원 등)이 부족합니다.

### 2. StreamJsonRpc 통합 장애
기존 `CallbackProxyTarget` 패턴을 StreamJsonRpc로 마이그레이션하려고 시도할 때, `_rpc.AddLocalRpcTarget(target)` 메서드가 프록시 패턴으로 생성된 대상을 인식하지 못하는 것을 발견했습니다. 구체적으로 StreamJsonRpc가 제네릭 유형 `T`의 속성을 RPC 메서드 매개변수로 자동 분할하지 못하여, 서버 측에서 클라이언트가 시작한 메서드 호출을 올바르게 처리할 수 없습니다.

### 3. 아키텍처 계층 혼란
기존 `ClientSideConnection`은 전송 계층(WebSocket/Stdio), 프로토콜 계층(JSON-RPC) 및 비즈니스 계층(ACP Agent 인터페이스)을 혼합하여 책임이 불명확하고, `AcpAgentCallbackRpcAdapter` 메서드 바인딩 누락 문제가 존재합니다.

### 4. 로깅 부재
WebSocket 전송 계층에 원시 JSON 콘텐츠의 로그 출력이 없어, RPC 통신 문제를 디버깅할 때 직렬화 문제인지 네트워크 문제인지 파악하기 어렵습니다.

## 해결

위의 문제에 대해 아키텍처 재구성, 라이브러리 통합 및 디버깅 강화 세 가지 차원에서 최적화하는 체계적인 해결 방안을 채택했습니다:

### 1. StreamJsonRpc로의 완전 마이그레이션

#### 기존 코드 제거
`JsonRpcEndpoint.cs`, `AgentSideConnection.cs` 및 관련 커스텀 직렬화 변환기(`JsonRpcMessageJsonConverter` 등)를 삭제합니다.

#### 공식 라이브러리 통합
`StreamJsonRpc` NuGet 패키지를 도입하고, `JsonRpc` 클래스를 활용하여 핵심 통신 로직을 처리합니다.

#### 전송 계층 추상화
`IAcpTransport` 인터페이스를 정의하여 `WebSocket` 및 `Stdio` 두 가지 전송 모드를 통합적으로 처리하고, 프로토콜 계층과 전송 계층의 결합을 해제합니다.

```csharp
// IAcpTransport 인터페이스 정의
public interface IAcpTransport
{
    Task SendAsync(string message, CancellationToken cancellationToken = default);
    Task<string> ReceiveAsync(CancellationToken cancellationToken = default);
    Task CloseAsync(CancellationToken cancellationToken = default);
}

// WebSocket 전송 구현
public class WebSocketTransport : IAcpTransport
{
    private readonly WebSocket _webSocket;

    public WebSocketTransport(WebSocket webSocket)
    {
        _webSocket = webSocket;
    }

    // 송수신 메서드 구현
    // ...
}

// Stdio 전송 구현
public class StdioTransport : IAcpTransport
{
    private readonly StreamReader _reader;
    private readonly StreamWriter _writer;

    public StdioTransport(StreamReader reader, StreamWriter writer)
    {
        _reader = reader;
        _writer = writer;
    }

    // 송수신 메서드 구현
    // ...
}
```

### 2. 프록시 대상 식별 문제 수정

#### `CallbackProxyTarget` 분석
기존 동적 프록시 생성 로직을 검사하여 StreamJsonRpc가 인식하지 못하는 근본 원인을 확인합니다(일반적으로 프록시 객체가 실제 메서드 서명을 공개하지 않거나, StreamJsonRpc가 지원하지 않는 매개변수 유형을 사용하기 때문입니다).

#### 매개변수 전달 재구성
제네릭 속성을 명확한 RPC 메서드 매개변수로 분할합니다. 동적 속성에 의존하는 대신 구체적인 Request/Response DTO(데이터 전송 객체)를 정의하여 StreamJsonRpc가 리플렉션을 통해 메서드 서명을 올바르게 식별할 수 있도록 합니다.

```csharp
// 기존 제네릭 속성 방식
public class CallbackProxyTarget<T>
{
    public Func<T, Task> Callback { get; set; }
}

// 재구성 후 구체적 메서드 방식
public class ReadTextFileRequest
{
    public string FilePath { get; set; }
}

public class ReadTextFileResponse
{
    public string Content { get; set; }
}

public interface IAcpAgentCallback
{
    Task<ReadTextFileResponse> ReadTextFileAsync(ReadTextFileRequest request);
    // 기타 메서드...
}
```

#### `AddLocalRpcTarget` 대신 `Attach` 사용
일부 복잡한 시나리오에서는 `JsonRpc` 객체를 수동으로 프록시하고 `RpcConnection`을 처리하는 것이 대상을 직접 추가하는 것보다 더 유연할 수 있습니다.

### 3. 메서드 바인딩 및 로깅 강화 구현

#### `AcpAgentCallbackRpcAdapter` 구현
이 컴포넌트가 StreamJsonRpc의 프록시 인터페이스를 명시적으로 구현하여, ACP 프로토콜에서 정의한 메서드(예: `ReadTextFileAsync`)를 StreamJsonRpc의 콜백 핸들러에 매핑합니다.

#### 로깅 통합
WebSocket 또는 Stdio의 메시지 처리 파이프라인에서 JSON-RPC 요청 및 응답의 원시 텍스트를 가로채고 기록합니다. `ILogger`를 활용하여 파싱 전과 직렬화 후에 원시 payload를 출력하여 형식 오류를 조사할 수 있도록 합니다.

```csharp
// 로깅 강화 전송 래퍼
public class LoggingAcpTransport : IAcpTransport
{
    private readonly IAcpTransport _innerTransport;
    private readonly ILogger<LoggingAcpTransport> _logger;

    public LoggingAcpTransport(IAcpTransport innerTransport, ILogger<LoggingAcpTransport> logger)
    {
        _innerTransport = innerTransport;
        _logger = logger;
    }

    public async Task SendAsync(string message, CancellationToken cancellationToken = default)
    {
        _logger.LogTrace("Sending message: {Message}", message);
        await _innerTransport.SendAsync(message, cancellationToken);
    }

    public async Task<string> ReceiveAsync(CancellationToken cancellationToken = default)
    {
        var message = await _innerTransport.ReceiveAsync(cancellationToken);
        _logger.LogTrace("Received message: {Message}", message);
        return message;
    }

    public async Task CloseAsync(CancellationToken cancellationToken = default)
    {
        _logger.LogDebug("Closing connection");
        await _innerTransport.CloseAsync(cancellationToken);
    }
}
```

### 4. 아키텍처 계층 재구성

#### 전송 계층 (`AcpRpcClient`)
StreamJsonRpc 연결을 캡슐화하고, `InvokeAsync` 및 연결 수명 주기 관리를 담당합니다.

```csharp
public class AcpRpcClient : IDisposable
{
    private readonly JsonRpc _rpc;
    private readonly IAcpTransport _transport;

    public AcpRpcClient(IAcpTransport transport)
    {
        _transport = transport;
        _rpc = new JsonRpc(new StreamRpcTransport(transport));
        _rpc.StartListening();
    }

    public async Task<TResponse> InvokeAsync<TResponse>(string methodName, object parameters)
    {
        return await _rpc.InvokeAsync<TResponse>(methodName, parameters);
    }

    public void Dispose()
    {
        _rpc.Dispose();
        _transport.Dispose();
    }

    // StreamRpcTransport는 IAcpTransport에 대한 StreamJsonRpc 어댑터입니다
    private class StreamRpcTransport : IDuplexPipe
    {
        // IDuplexPipe 인터페이스 구현
        // ...
    }
}
```

#### 프로토콜 계층 (`IAcpAgentClient` / `IAcpAgentCallback`)
명확한 client-to-agent 및 agent-to-client 인터페이스를 정의하고, `Func<IAcpAgent, IAcpClient>`와 같은 순환 종속성 팩토리 패턴을 제거하고 의존성 주입 또는 직접 콜백 등록을 사용합니다.

## 실천

StreamJsonRpc의 모범 사례와 프로젝트 경험을 바탕으로, 구현 과정에서의 핵심 권장 사항은 다음과 같습니다:

### 1. 강유형 DTO가 동적 객체보다 우선
StreamJsonRpc의 핵심 장점은 강유형에 있습니다. `dynamic` 또는 `JObject`를 사용하여 매개변수를 전달하지 마세요. 각 RPC 메서드에 대해 명확한 C# POCO 클래스를 매개변수로 정의해야 합니다. 이는 프록시 대상 식별 문제를 해결할 뿐만 아니라 컴파일 시 유형 오류를 발견할 수 있습니다.

예: `CallbackProxyTarget`의 제네릭 속성을 `ReadTextFileRequest` 및 `WriteTextFileRequest` 등의 구체적인 클래스로 대체합니다.

### 2. 메서드 이름을 명시적으로 선언
`[JsonRpcMethod]` 특성을 사용하여 RPC 메서드 이름을 명시적으로 지정하고, 기본 메서드 이름 매핑에 의존하지 마세요. 이는 명명 스타일 차이(PascalCase vs camelCase 등)로 인한 호출 실패를 방지할 수 있습니다.

```csharp
public interface IAcpAgentCallback
{
    [JsonRpcMethod("readTextFile")]
    Task<ReadTextFileResponse> ReadTextFileAsync(ReadTextFileRequest request);

    [JsonRpcMethod("writeTextFile")]
    Task WriteTextFileAsync(WriteTextFileRequest request);
}
```

### 3. 연결 상태 콜백 활용
StreamJsonRpc는 `JsonRpc.ConnectionLost` 이벤트를 제공합니다. 프로세스의 예기치 않은 종료나 네트워크 연결 끊김을 처리하기 위해 반드시 이 이벤트를 수신해야 합니다. 이는 Orleans의 Grain 실패 감지에만 의존하는 것보다 더 시기적절합니다.

```csharp
_rpc.ConnectionLost += (sender, e) =>
{
    _logger.LogError("RPC connection lost: {Reason}", e.ToString());
    // 재연결 로직 처리 또는 사용자에게 알림
};
```

### 4. 로깅 계층 기록
- **Trace 수준**: 완전한 JSON Request/Response 원문 기록
- **Debug 수준**: 메서드 호출 스택 및 매개변수 요약 기록
- **주의**: 로그에 민감한 Authorization Token이나 대용량 파일 콘텐츠의 Base64 인코딩이 포함되지 않도록 하세요.

### 5. 스트리밍 전송의 특수성 처리
StreamJsonRpc는 `IAsyncEnumerable`을 기본적으로 지원합니다. ACP의 스트리밍 Prompt 응답을 구현할 때는 커스텀 페이지네이션 로직 대신 `IAsyncEnumerable`을 직접 사용해야 합니다. 이는 스트리밍 처리 코드를 크게 단순화할 수 있습니다.

```csharp
public interface IAcpAgentCallback
{
    [JsonRpcMethod("streamText")]
    IAsyncEnumerable<string> StreamTextAsync(StreamTextRequest request);
}
```

### 6. 어댑터 패턴 (Adapter Pattern)
`ACPSession`과 `ClientSideConnection`의 분리를 유지합니다. `ACPSession`은 Orleans의 상태 관리와 비즈니스 로직(예: 메시지 큐잉)에 집중하고, 상속이 아닌 조합 방식으로 StreamJsonRpc 연결 객체를 사용해야 합니다.

## 요약

StreamJsonRpc를 전면적으로 통합함으로써 HagiCode 프로젝트는 기존 커스텀 구현의 높은 유지보수 비용, 기능적 한계 및 아키텍처 계층 혼란 등의 문제를 성공적으로 해결했습니다. 핵심 개선 사항은 다음과 같습니다:

1. 강유형 DTO를 도입하여 동적 속성을 대체하고 코드의 유지보수성과 신뢰성을 향상시켰습니다
2. 전송 계층 추상화와 프로토콜 계층 분리를 구현하여 아키텍처의 명확성을 높였습니다
3. 로깅 기능을 강화하여 통신 문제 조사를 용이하게 했습니다
4. 스트리밍 전송 지원을 도입하여 스트리밍 처리 구현을 단순화했습니다

이러한 개선은 HagiCode에 더 안정적이고 효율적인 통신 기반을 제공하여 외부 AI 도구와 더 잘 상호작용할 수 있도록 하며, 향후 기능 확장을 위한 탄탄한 기반을 마련했습니다.
