## Docker 컨테이너에서 AI CLI 도구 실행: 사용자 격리 및 지속 볼륨 실무 가이드

> Claude Code, Codex, OpenCode와 같은 AI 코딩 도구를 컨테이너화된 환경에 통합하는 것은 간단해 보이지만, 숨겨진 복잡성이 곳곳에 존재합니다. 이 글에서는 HagiCode 프로젝트가 Docker 배포의 핵심 과제를 어떻게 해결하는지 심층적으로 다루며, 사용자 권한, 구성 지속성, 버전 관리를 포함하여 일반적인 함정을 피할 수 있도록 안내합니다.

## 배경

Docker 컨테이너 내에서 AI 코딩 CLI 도구를 실행하기로 결정했을 때, 가장 직관적인 생각은 아마 이것이었을 것입니다: "컨테이너는 기본적으로 root 아닌가요? 모든 것을 직접 설치하고 끝내면 되지 않겠습니까?" 실제로 그 단순해 보이는 생각에는 반드시 해결해야 하는 몇 가지 핵심 문제가 숨어 있습니다.

첫째, **보안 제한이 첫 번째 관문**입니다. Claude CLI를 예로 들면, root 사용자로 실행하는 것을 명시적으로 금지합니다. 이는 강제 보안 검사이며, root가 감지되면 시작을 거부합니다. `USER` 지시어로 사용자를 전환하면 되지 않겠냐고 생각할 수 있습니다. 그렇게 간단하지 않습니다. 컨테이너 내의 비root 사용자와 호스트 머신의 사용자 권한 사이에 여전히 매핑 문제가 있습니다.

둘째, **상태 지속성이 두 번째 함정**입니다. Claude Code는 로그인이 필요하고, Codex는 자체 구성을 가지며, OpenCode도 캐시 디렉토리를 가집니다. 컨테이너가 재시작될 때마다 모든 것을 다시 구성해야 한다면, "자동화"라는 아이디어 자체가 의미를 잃게 됩니다. 이러한 구성이 컨테이너의 수명 주기를 넘어서 지속되어야 합니다.

세 번째 문제는 **권한 일관성**입니다. 컨테이너 내부의 프로세스가 호스트 사용자가 생성한 구성 파일에 접근할 수 있습니까? UID/GID 불일치는 종종 파일 권한 오류를 일으키며, 실제 배포에서 극히 흔합니다.

이러한 문제는 독립적으로 보이지만, 실제로는 긴밀하게 연결되어 있습니다. HagiCode의 개발 과정에서 우리는 점차 실용적인 해결책을 마련했습니다. 다음으로 기술적 세부 사항과 그 함정에서 얻은 교훈을 공유하겠습니다.

## 왜 root를 그냥 사용하면 안 될까요?

여기에 흔한 오해가 있습니다. Docker 컨테이너는 기본적으로 root로 실행되므로, 도구를 root로 설치하면 되지 않겠습니까? 그렇게 생각하신다면 Claude CLI가 금방 그렇지 않다는 것을 가르쳐 줄 것입니다.

```bash
# root로 Claude CLI를 직접 실행? 안 됩니다.
docker run --rm -it --user root myimage claude
# 출력: Error: This command cannot be run as root user
```

이것은 Claude CLI의 강제 보안 제한입니다. 이유는 간단합니다. 이 CLI 도구들은 API 토큰, 로컬 캐시, 사용자가 작성한 스크립트까지 민감한 사용자 구성을 읽고 씁니다. root 권한으로 실행하면 위험이 너무 큽니다.

그렇다면 질문은 이것이 됩니다. CLI의 보안 요구사항을 충족하면서 컨테이너 관리의 유연성을 어떻게 유지할 것인가? 우리는 생각하는 방식을 바꿔야 합니다: **런타임에 사용자를 전환하는 대신, 이미지 빌드 단계에서 전용 사용자를 생성**합니다.

## 전용 사용자 생성: 이름 변경 그 이상

Dockerfile에 `USER` 한 줄을 추가하는 것만으로 충분하다고 생각할 수 있습니다. 실제로 가장 간단한 접근 방식이지만, 충분히 견고하지는 않습니다.

### 정적 생성 vs 동적 매핑

HagiCode의 접근 방식은 UID 1000을 가진 `hagicode` 사용자를 생성하는 것입니다. 이는 대부분의 호스트 머신에서 기본 사용자와 일치합니다:

```dockerfile
RUN groupadd -o -g 1000 hagicode && \
    useradd -o -u 1000 -g 1000 -s /bin/bash -m hagicode && \
    mkdir -p /home/hagicode/.claude && \
    chown -R hagicode:hagicode /home/hagicode
```

하지만 이것은 이미지 내의 내장 사용자만 해결합니다. 호스트 사용자가 UID 1001이라면 어떻게 됩니까? 여전히 컨테이너 시작 시 동적 매핑을 지원해야 합니다.

docker-entrypoint.sh에 핵심 로직이 포함되어 있습니다:

```bash
if [ -n "$PUID" ] && [ -n "$PGID" ]; then
    if ! id hagicode >/dev/null 2>&1; then
        groupadd -g "$PGID" hagicode
        useradd -u "$PUID" -g "$PGID" -s /bin/bash -m hagicode
    fi
fi
```

이 설계의 장점은 명확합니다: **이미지 빌드 시 기본 UID 1000을 사용하고, 런타임에 `PUID` 및 `PGID` 환경 변수를 통해 동적으로 조정**합니다. 호스트 사용자의 UID가 무엇이든, 구성 파일의 소유권은 올바르게 유지됩니다.

## 지속 볼륨의 설계 철학

각 AI CLI 도구는 자체적으로 선호하는 구성 디렉토리를 가지므로, 하나씩 매핑해야 합니다:

| CLI 도구 | 컨테이너 내 경로 | 명명된 볼륨 |
|---------|-----------|--------|
| Claude | `/home/hagicode/.claude` | `claude-data` |
| Codex | `/home/hagicode/.codex` | `codex-data` |
| OpenCode | `/home/hagicode/.config/opencode` | `opencode-config-data` |

왜 바인드 마운트 대신 명명된 볼륨을 사용할까요? 세 가지 이유가 있습니다:

1. **관리가 간단합니다**: 명명된 볼륨은 Docker에 의해 자동으로 관리되므로, 호스트 디렉토리를 수동으로 생성할 필요가 없습니다.
2. **권한 격리**: 볼륨의 초기 콘텐츠는 컨테이너 내부의 사용자에 의해 생성되므로, 호스트와의 권한 충돌을 피합니다.
3. **독립적인 마이그레이션**: 볼륨은 컨테이너와 독립적으로 존재할 수 있으므로, 이미지 업그레이드 시 데이터가 손실되지 않습니다.

docker-compose-builder-web은 해당 볼륨 구성을 자동으로 생성합니다:

```yaml
volumes:
  claude-data:
  codex-data:
  opencode-config-data:

services:
  hagicode:
    volumes:
      - claude-data:/home/hagicode/.claude
      - codex-data:/home/hagicode/.codex
      - opencode-config-data:/home/hagicode/.config/opencode
    user: "${PUID:-1000}:${PGID:-1000}"
```

여기서 `user` 필드에 주의하십시오. `PUID`와 `PGID`는 환경 변수를 통해 주입되어 컨테이너 내부의 프로세스가 호스트 사용자와 일치하는 ID로 실행되도록 합니다. 이 세부 사항이 중요합니다. 권한 문제는 한 번 발생하면 디버깅하기 고통스럽기 때문입니다.

## 버전 관리: 내장 버전과 런타임 재정의

Docker 이미지 버전을 고정하는 것은 재현성에 필수적입니다. 하지만 실제 개발에서는 종종 더 새로운 버전을 테스트하거나 긴급하게 버그를 수정해야 합니다. 매번 이미지를 재빌드해야 한다면 워크플로우가 너무 비효율적입니다.

HagiCode의 전략은 **고정 버전을 기본으로, 런타임 재정의를 확장 메커니즘으로** 사용하는 것입니다. 안정성과 유연성 사이의 실용적인 엔지니어링 타협입니다.

Dockerfile.template는 버전을 다음과 같이 고정합니다:

```dockerfile
USER hagicode
WORKDIR /home/hagicode

# 글로벌 npm 설치 경로 구성
RUN mkdir -p /home/hagicode/.npm-global && \
    npm config set prefix '/home/hagicode/.npm-global'

# 고정 버전으로 CLI 도구 설치
RUN npm install -g @anthropic-ai/claude-code@2.1.71 && \
    npm install -g @openai/codex@0.112.0 && \
    npm install -g opencode-ai@1.2.25 && \
    npm cache clean --force
```

docker-entrypoint.sh는 런타임 재정의를 지원합니다:

```bash
install_cli_override_if_needed() {
    local package_name="$2"
    local override_version="$5"

    if [ -n "$override_version" ]; then
        gosu hagicode npm install -g "${package_name}@${override_version}"
    fi
}

# 사용 예시
install_cli_override_if_needed "" "@anthropic-ai/claude-code" "" "" "${CLAUDE_CODE_CLI_VERSION}"
```

이렇게 하면 환경 변수를 통해 새 버전을 테스트할 수 있으며, 이미지를 재빌드할 필요가 없습니다:

```bash
docker run -e CLAUDE_CODE_CLI_VERSION=2.2.0 myimage
```

이 설계는 실용적입니다. 새로운 기능을 테스트할 때마다 이미지를 재빌드하고 싶은 사람은 아무도 없으니까요.

## 자동 구성 주입

CLI 도구를 수동으로 구성하는 것 외에도, 일부 시나리오에서는 자동 구성 주입이 필요합니다. 가장 전형적인 예가 API 토큰입니다.

```bash
if [ -n "$ANTHROPIC_AUTH_TOKEN" ]; then
    mkdir -p /home/hagicode/.claude
    cat > /home/hagicode/.claude/settings.json <<EOF
{
  "env": {
    "ANTHROPIC_AUTH_TOKEN": "${ANTHROPIC_AUTH_TOKEN}"
  }
}
EOF
    chown -R hagicode:hagicode /home/hagicode/.claude
fi
```

여기서 두 가지가 중요합니다: **민감한 정보를 환경 변수를 통해 전달하고 이미지에 하드코딩하지 않으며**, 구성 파일의 소유권을 올바르게 설정해야 합니다. 그렇지 않으면 CLI 도구가 읽을 수 없습니다.

## 모범 사례 및 함정 체크리스트

### 권한 불일치 문제

가장 빠지기 쉬운 함정입니다. 호스트 사용자가 UID 1001이고 컨테이너는 1000을 사용하면, 한쪽에서 생성한 파일을 다른 쪽에서 접근할 수 없습니다.

```bash
# 올바른 방법: 컨테이너를 호스트 사용자와 일치시키기
docker run \
    -e PUID=$(id -u) \
    -e PGID=$(id -g) \
    myimage
```

이 문제는 매우 흔하며, 처음 마주치면 꽤 답답합니다.

### 컨테이너 재시작 후 구성이 사라지는 문제

매번 재시작할 때마다 다시 로그인해야 한다면, 지속 볼륨을 마운트했는지 확인해 보십시오:

```yaml
volumes:
  - claude-data:/home/hagicode/.claude
```

신중하게 구성한 것이 사라지는 것보다 더 답답한 일은 없습니다.

### 올바른 버전 업그레이드 방법

실행 중인 컨테이너 내에서 직접 `npm install -g`를 실행하지 마십시오. 올바른 방법은:

1. 환경 변수를 설정하여 재정의 설치를 트리거합니다.
2. 또는 이미지를 재빌드합니다.

```bash
# 방법 1: 런타임 재정의
docker run -e CLAUDE_CODE_CLI_VERSION=2.2.0 myimage

# 방법 2: 이미지 재빌드
docker build -t myimage:v2 .
```

로마로 가는 길은 하나가 아니지만, 어떤 길은 다른 길보다 더 부드럽습니다.

### 보안 강화 체크리스트

- API 토큰을 이미지에 작성하는 대신 환경 변수를 통해 전달합니다.
- 구성 파일 권한을 `600`으로 설정합니다.
- 항상 비root 사용자로 애플리케이션을 실행합니다.
- 정기적으로 CLI 버전을 업데이트하여 보안 취약점을 수정합니다.

보안은 항상 중요하지만, 실제로 일관되게 시행하는 것이 진정한 과제입니다.

## 새로운 CLI 도구 지원 확장

향후 새로운 CLI 도구를 지원하려면 세 단계만 필요합니다:

1. **Dockerfile.template**: 설치 단계를 추가합니다.
2. **docker-entrypoint.sh**: 버전 재정의 로직을 추가합니다.
3. **docker-compose-builder-web**: 지속 볼륨 매핑을 추가합니다.

이 템플릿 기반 설계는 핵심 로직을 변경하지 않고도 확장을 간단하게 만듭니다.

## 결론

Docker 컨테이너에서 AI CLI 도구를 실행하는 것에는 세 가지 핵심 과제가 있습니다: **사용자 권한, 구성 지속성, 버전 관리**. 전용 사용자, 명명된 볼륨 격리, 환경 변수 기반 재정의를 결합하여 HagiCode 프로젝트는 안전하고 유연한 배포 아키텍처를 구축했습니다.

핵심 설계 포인트:

- **사용자 격리**: 이미지 빌드 단계에서 전용 사용자를 생성하고, 런타임에 동적 `PUID`/`PGID` 매핑을 지원합니다.
- **지속성 전략**: 각 CLI 도구는 자체 명명된 볼륨을 가지므로, 재시작해도 구성에 영향이 없습니다.
- **버전 유연성**: 고정 기본값은 재현성을 보장하고, 런타임 재정의는 테스트를 위한 여지를 제공합니다.
- **자동화된 구성**: 민감한 구성을 환경 변수를 통해 자동으로 주입할 수 있습니다.

이 해결책은 HagiCode 프로젝트에서 이미 한동안 안정적으로 실행되고 있으며, 비슷한 요구를 가진 개발자에게 유용한 참고 포인트가 되기를 바랍니다.
