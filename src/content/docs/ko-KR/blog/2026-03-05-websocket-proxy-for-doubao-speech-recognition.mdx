## 브라우저 WebSocket 인증 난제 해결: 두바오 음성 인식의 프록시 솔루션 실천

> 브라우저 WebSocket API는 커스텀 HTTP 헤더 설정을 지원하지 않으므로, 헤더를 통해 인증 정보를 전달해야 하는 음성 인식 서비스에 도전 과제가 됩니다. 이 글에서는 HagiCode 프로젝트에서 백엔드 프록시 솔루션을 통해 이 문제를 해결하는 방법과 playground에서 프로덕션 환경까지의 실천 과정을 공유합니다.


## 배경

사실 HagiCode 프로젝트의 음성 인식 기능을 개발할 때, 자신감을 가지고 바이트댄스의 두바오 음성 인식 서비스를 선택했습니다. 초기 설계는 아주 간단했습니다 — 프론트엔드에서 두바오 WebSocket 서비스에 직접 연결. 이게 뭐가 어려운가요? 연결 하나 만들고 데이터 좀 보내는 것뿐인데.

하지만 예상치 못한 일이 — 두바오 API는 HTTP 헤더를 통해 인증 정보(accessToken, secretKey 등)를 전달하도록 요구합니다. 이것은 좀 난감했습니다. 브라우저의 WebSocket API는 커스텀 헤더 설정을 전혀 지원하지 않거든요.

지원하지 않으면 어떡합니까?

그때도 한참 고민했습니다. 선택지는 두 가지:

1. 인증 정보를 URL 쿼리 매개변수에 넣기 — 간단하고 직관적
2. 백엔드에 프록시 레이어를 만들기 — 좀 더 복잡해 보이지만

첫 번째 방안은 자격 증명이 프론트엔드 코드와 로컬 저장소에 직접 노출됩니다. 안전할까요? 동의하기 어렵습니다. 게다가 일부 API는 반드시 헤더 인증을 요구하므로 아예 통하지 않습니다.

결국 두 번째 방안을 선택했습니다 — 백엔드에 WebSocket 프록시를 구현합니다. 우연히도 이 솔루션은 처음 우리 playground에서 검증되었고, 안정성이 확인된 후 프로덕션 환경에 적용되었습니다. 프로덕션에서 실험체가 되고 싶지는 않으니까요.

## HagiCode에 대하여

이 글에서 공유하는 솔루션은 [HagiCode](https://github.com/HagiCode-org/site) 프로젝트의 실천 경험에서 왔습니다.

HagiCode는 AI 코드 어시스턴트 프로젝트로 음성 인터랙션 기능을 지원합니다. 프론트엔드에서 음성 인식 서비스를 호출해야 하므로 이 WebSocket 인증 문제에 직면했고, 그 결과 이 솔루션이 탄생했습니다. 때때로 생각해 보면, 어려움도 전혀 쓸모없는 것은 아닙니다. 적어도 프록시 사용법을 배웠으니까요, 그렇죠?

## 기술적 도전 분석

### 브라우저 WebSocket의 제한

표준 WebSocket API는 정말 간단해 보입니다:

```javascript
const ws = new WebSocket('wss://example.com/ws');
```

하지만 문제는 바로 "간단하다"는 데 있습니다 — URL에서만 매개변수를 전달하고, HTTP 요청처럼 헤더를 설정할 수 없습니다:

```javascript
// WebSocket API에서는 지원되지 않습니다
const ws = new WebSocket('wss://example.com/ws', {
  headers: {
    'Authorization': 'Bearer token'
  }
});
```

두바오 음성 인식처럼 헤더 인증이 필요한 서비스에서 이 제한은 정말 넘기 힘든 장벽입니다.

### 아키텍처 설계 결정

솔루션을 설계할 때도 많이 고민하고 저울질했습니다.

**결정 1: 프록시 모델 선택**

두 가지 방안을 비교했습니다:

| 방안 | 장점 | 단점 | 결정 |
|------|------|------|------|
| 네이티브 WebSocket | 경량, 간단, 직접 전달 | 연결 관리를 수동으로 처리해야 함 | 선택 |
| SignalR | 자동 재연결, 강력한 타입 | 과도하게 복잡, 추가 종속성 | 미선택 |

결국 네이티브 WebSocket을 선택했습니다. 솔직히 말하면 가장 가벼워서 간단한 양방향 바이너리 스트림 전달에 적합하기 때문입니다. SignalR을 추가하는 것은 확실히 과한이며 지연을 증가시킬 것입니다.

**결정 2: 연결 관리 전략**

"연결당 단일 세션" 모델을 채택했습니다 — 각 프론트엔드 WebSocket 연결이 독립적인 두바오 백엔드 연결에 대응합니다.

이렇게 하면:
- 구현이 간단하고 전형적인 사용 시나리오에 부합합니다
- 디버깅 및 장애 조사가 용이합니다
- 리소스가 격리되어 세션 간 서로 간섭하지 않습니다

간단하고 직관적인 것이 때로는 최선의 선택입니다.

**결정 3: 인증 정보 저장**

자격 증명은 백엔드 구성 파일(appsettings.yml 또는 환경 변수)에 저장되고 의존성 주입을 통해 로드됩니다:

- 구성 방식이 간단하고 기존 백엔드 구성 패턴에 부합합니다
- 민감한 정보가 프론트엔드에 노출되지 않습니다
- 다중 환경 구성(개발, 테스트, 프로덕션)을 지원합니다

보안은 반드시 필요합니다. 자격 증명이 여기저기 흩어지는 것은 아무도 원하지 않으니까요.

### 데이터 흐름 설계

전체 데이터 흐름은 다음과 같습니다:

```
前端 (浏览器)
  │
  │ ws://backend/api/voice/ws
  │ WebSocket (二进制)
  ▼
后端 (代理)
  │
  │ wss://openspeech.bytedance.com/
  │ (带认证 header)
  ▼
豆包 API
```

프로세스도 복잡하지 않습니다:
1. 프론트엔드가 WebSocket으로 백엔드 프록시에 연결
2. 백엔드 프록시가 오디오 데이터를 수신하고 헤더가 포함된 방식으로 두바오 API에 연결
3. 두바오 API가 인식 결과를 반환하고 프록시가 프론트엔드로 전달
4. 전 과정 비동기 양방향 스트리밍 전송

## 핵심 컴포넌트 구현

### 1. WebSocket 엔드포인트 구성

```csharp
app.Map("/ws", async context =>
{
    if (context.WebSockets.IsWebSocketRequest)
    {
        var appId = context.Request.Query["appId"];
        var accessToken = context.Request.Query["accessToken"];

        if (string.IsNullOrEmpty(appId) || string.IsNullOrEmpty(accessToken))
        {
            context.Response.StatusCode = 400;
            return;
        }

        using var webSocket = await context.WebSockets.AcceptWebSocketAsync();

        var buffer = new byte[4096];
        while (!webSocket.CloseStatus.HasValue)
        {
            var result = await webSocket.ReceiveAsync(buffer, CancellationToken.None);

            if (result.MessageType == WebSocketMessageType.Close)
            {
                await webSocket.CloseAsync(
                    result.CloseStatus.Value,
                    result.CloseStatusDescription,
                    CancellationToken.None);
                break;
            }

            await HandleAudioDataAsync(buffer, result.Count);
        }
    }
});
```

### 2. 세션 관리

```csharp
public class DoubaoSessionManager : IDoubaoSessionManager
{
    private readonly ConcurrentDictionary<string, DoubaoSession> _sessions = new();

    public DoubaoSession CreateSession(string connectionId)
    {
        var session = new DoubaoSession(connectionId);
        _sessions[connectionId] = session;
        return session;
    }

    public async Task SendAudioAsync(string connectionId, byte[] audioData)
    {
        if (_sessions.TryGetValue(connectionId, out var session))
        {
            await session.SendAudioAsync(audioData);
        }
    }

    public void RemoveSession(string connectionId)
    {
        if (_sessions.TryRemove(connectionId, out var session))
        {
            session.Dispose();
        }
    }
}
```

ConcurrentDictionary로 세션을 관리하므로 스레드 안전성에 대해 걱정할 필요가 없습니다. 각 연결이 들어오면 Session을 생성하고, 연결이 끊어지면 자동 정리됩니다.

프론트엔드와 백엔드 간에 JSON 형식의 텍스트 메시지로 제어를 하고, 바이너리 메시지로 오디오 데이터를 전송합니다.

이 설계는 제어 신호와 오디오 데이터를 분리하여 처리를 더 명확하게 합니다.

## 프론트엔드 연동 실천

### WebSocket 연결

```javascript
class DoubaoVoiceClient {
    constructor(config) {
        this.config = config;
        this.ws = null;
    }

    async connect() {
        const url = new URL(this.config.wsUrl);
        Object.entries(this.config.params).forEach(([key, value]) => {
            url.searchParams.set(key, value);
        });

        this.ws = new WebSocket(url);

        return new Promise((resolve, reject) => {
            this.ws.onopen = () => {
                console.log('[DoubaoVoice] Connected');
                resolve();
            };

            this.ws.onmessage = (event) => {
                this._handleMessage(JSON.parse(event.data));
            };

            this.ws.onerror = reject;
        });
    }

    _handleMessage(message) {
        switch (message.type) {
            case 'status':
                this._handleStatus(message.payload);
                break;
            case 'result':
                this.onResult?.(message.payload);
                break;
            case 'error':
                console.error('[DoubaoVoice] Error:', message.payload);
                break;
        }
    }
}

const client = new DoubaoVoiceClient({
    wsUrl: 'ws://localhost:5000/ws',
    params: {
        appId: 'your-app-id',
        accessToken: 'your-access-token',
        sampleRate: 16000,
        bitsPerSample: 16,
        channels: 1
    }
});
```

### 오디오 수집 및 전송

AudioWorkletNode로 오디오를 처리하면 성능이 더 좋습니다:

```javascript
class AudioProcessorWorklet extends AudioWorkletProcessor {
    process(inputs, outputs, parameters) {
        const input = inputs[0]?.[0];
        if (!input) return true;

        const pcm = new Int16Array(input.length);
        for (let i = 0; i < input.length; i++) {
            pcm[i] = Math.max(-32768, Math.min(32767, input[i] * 32767));
        }

        this.port.postMessage({
            type: 'audioData',
            data: pcm.buffer
        }, [pcm.buffer]);

        return true;
    }
}

registerProcessor('audio-processor', AudioProcessorWorklet);

async function startAudioRecording() {
    const stream = await navigator.mediaDevices.getUserMedia({
        audio: {
            echoCancellation: true,
            noiseSuppression: true,
            autoGainControl: true,
            sampleRate: 48000
        }
    });

    const audioContext = new AudioContext();
    const audioSource = audioContext.createMediaStreamSource(stream);

    await audioContext.audioWorklet.addModule('/audio-worklet.js');
    const audioWorkletNode = new AudioWorkletNode(audioContext, 'audio-processor');

    audioWorkletNode.port.onmessage = (event) => {
        if (event.data.type === 'audioData' && ws?.readyState === WebSocket.OPEN) {
            ws.send(event.data.data);
        }
    };

    audioSource.connect(audioWorkletNode);
}
```

AudioWorklet은 ScriptProcessorNode보다 성능이 훨씬 좋아 오디오 끊김 현상이 없습니다.

## 주의 사항 및 모범 사례

### 1. 연결 모니터링

- 정기적으로 세션 상태 로그를 출력하여 연결 수명 주기를 추적합니다
- 오디오 세그먼트 수와 지속 시간을 모니터링하여 비정상 연결을 식별합니다
- 두바오 서비스의 연결 상태 및 재연결 상황을 기록합니다

### 2. 오류 처리

- 모든 WebSocket 예외를 캡처하고 기록합니다
- IAsyncDisposable을 사용하여 리소스 정리를 보장합니다
- 우아한 연결 종료 및 시간 초과 처리를 구현합니다

### 3. 오디오 형식 요구 사항

- 샘플링 레이트: 16000 Hz(권장) 또는 8000 Hz
- 비트 깊이: 16-bit
- 채널: 모노
- 인코딩: PCM (raw)

형식이 올바르지 않으면 인식 실패 또는 매우 나쁜 결과가 발생합니다.

### 4. 보안 고려 사항

- 민감한 자격 증명은 백엔드 구성에만 저장합니다
- 리소스 고갈을 방지하기 위해 연결 수 제한을 구현합니다
- 프로덕션 환경에서는 HTTPS/WSS를 사용합니다

### 5. 성능 최적화

- 비동기 작업을 사용하여 차단을 방지합니다
- 버퍼 크기를 적절히 조정합니다(기본 4096 바이트)
- 연결 풀 및 재사용 전략을 고려합니다

## 배포 권장 사항

1. **Docker 배포**: 프록시 서비스를 컨테이너로 패키징하여 확장 및 관리를 용이하게 합니다
2. **로드 밸런싱**: Nginx 또는 Envoy로 WebSocket 리버스 프록시를 구성합니다
3. **헬스 체크**: 하트비트 메커니즘을 구현하여 서비스 가용성을 모니터링합니다
4. **로그 집계**: 로그를 중앙 집중식 로그 시스템(예: ELK, Loki)으로 전송합니다

## 요약

WebSocket 프록시 솔루션은 브라우저 WebSocket API가 커스텀 헤더를 지원하지 않는 근본적인 문제를 해결했습니다. HagiCode 프로젝트에서 이 솔루션은 playground 검증에서 프로덕션 환경 배포까지 그 실현 가능성과 안정성을 입증했습니다.

핵심 포인트 요약:
- 백엔드 프록시를 통해 인증 정보를 안전하게 전달할 수 있습니다
- 네이티브 WebSocket은 경량 고효율로 간단한 시나리오에 적합합니다
- "연결당 단일 세션"은 구현과 디버깅을 단순화합니다
- 프론트엔드-백엔드 메시지 프로토콜 분리로 제어 신호와 오디오 데이터를 분리합니다

WebSocket 인증이 필요한 기능을 개발 중이시라면 이 솔루션이 영감을 주기를 바랍니다.
