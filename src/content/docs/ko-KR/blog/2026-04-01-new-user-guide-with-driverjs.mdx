## React 프로젝트에서 우아하게 신규 사용자 온보딩 구현하기: HagiCode의 driver.js 실천

> 사용자가 처음 당신의 제품을 열었을 때, 그들은 정말로 어디서 시작해야 할지 알고 있을까요? 이 글에서는 HagiCode 프로젝트에서 driver.js로 신규 사용자 온보딩을 구현한 이야기를 나누어 봅니다.

## 배경

다음과 같은 상황을 겪어보신 적이 있나요? 신규 사용자가 당신의 제품에 가입하고, 페이지를 연 후에 멍하니 서 있고, 이리저리 둘러보면서 어디를 클릭해야 할지, 무엇을 해야 할지 모릅니다. 개발자로서 우리는 사용자가 "스스로 탐색"할 것이라고 생각합니다.毕竟 사람의 호기심은 무한하기 때문입니다. 하지만 현실은——대부분의 사용자는 몇 분 안에 진입점을 찾지 못해 조용히 떠납니다. 마치 이야기가 갑자기 시작되었지만 자연스럽게 끝나는 것처럼.

신규 사용자 온보딩은 이 문제를 해결하는 중요한 수단이지만, 구현하기가 그렇게 간단하지는 않습니다. 좋은 온보딩 시스템은 다음이 필요합니다:

- 페이지 요소를 정확하게定位하고 하이라이트 표시할 수 있어야 합니다
- 다단계 온보딩 프로세스를 지원해야 합니다
- 사용자의 선택(완료/건너뛰기)을 기억할 수 있어야 합니다
- 페이지 성능과 정상적인 상호작용에 영향을 주지 않아야 합니다
- 코드 구조가 명확하고 유지보수가 쉬워야 합니다

HagiCode를 개발하는 과정에서 우리도 같은 도전에 직면했습니다. HagiCode는 AI 코드 어시스턴트 프로젝트로, 핵심 워크플로우는 "사용자가 제안 생성 → AI가 계획 생성 → 사용자가 검토 → AI가 실행"과 같은 OpenSpec 프로세스입니다. 이 개념을 처음 접하는 사용자에게 이 프로세스는 완전히 새로운 것이며, 빠르게 익숙해질 수 있도록 좋은 온보딩이 필수적입니다.毕竟 새로운 것은 항상 약간의 시간이 필요합니다.

## HagiCode에 대해

이 글에서 공유하는 솔루션은 [HagiCode](https://hagicode.com) 프로젝트에서의 실천 경험에서 왔습니다. HagiCode는 Claude 기반의 AI 코드 어시스턴트로, OpenSpec 워크플로우를 통해 개발자가 더 효율적으로 코드 작업을 완료하도록 돕습니다. [GitHub](https://github.com/HagiCode-org/site)에서 우리의 오픈 소스 코드를 확인할 수 있습니다.

## 왜 driver.js를 선택했나

기술 선정 단계에서 우리는 몇 개의 주요 온보딩 라이브러리를 평가했습니다. 솔직히 말하면 각각의 특징이 있습니다:

- **Intro.js**: 강력한 기능但 큰 볼륨, 스타일 커스터마이징이 상대적으로 복잡함
- **Shepherd.js**: 좋은 API 디자인, 하지만 우리 시나리오에는 약간 "무거움"
- **driver.js**: 경량, 간결, 직관적인 API, React 생태계 지원

최종적으로 driver.js를 선택했습니다. 특별한 이유는 없고 주로 다음 고려사항에 기반합니다:

1. **경량**: 핵심 라이브러리 볼륨이 작아 번들 크기를 크게 증가시키지 않습니다
2. **간결한 API**: 설정 항목이 명확하고 직관적이며 빠르게 익힐 수 있습니다
3. **유연성**: 커스텀定位, 스타일 및 상호작용 동작을 지원합니다
4. **동적 가져오기**: 필요에 따라 로드하여 첫 화면 성능에 영향을 주지 않습니다

선정这件事, 사실 최고는 없고 가장 적합한 것만 있을 뿐입니다.

## 기술 구현

### 핵심 구성

driver.js의 구성은 매우 직관적입니다. 다음은 HagiCode 프로젝트의 핵심 구성입니다:

```typescript
import { driver } from 'driver.js';
import 'driver.js/dist/driver.css';

const newConversationDriver = driver({
  allowClose: true,           // 사용자가 온보딩을 닫을 수 있도록 허용
  animate: true,              // 애니메이션 효과 활성화
  overlayClickBehavior: 'close', // 오버레이 클릭 시 온보딩 닫기
  disableActiveInteraction: false, // 요소 상호작용 유지
  showProgress: false,        // 진행률 표시 안 함(우리는 커스텀 진행률 관리가 있음)
  steps: guideSteps           // 온보딩 단계 배열
});
```

이러한 구성 뒤의 고려사항은:

- `allowClose: true` - 사용자 선택을 존중하고 온보딩 완료를 강요하지 않습니다.毕竟 억지로 만든 것은 달지 않습니다
- `disableActiveInteraction: false` - 특정 단계에서 사용자의 실제 조작(예: 텍스트 입력)이 필요하므로 상호작용을 비활성화할 수 없습니다
- `overlayClickBehavior: 'close'` - 사용자에게 빠른 종료 방법을 제공합니다

### 상태 관리

온보딩 상태의 지속성은 핵심입니다——우리는 페이지를 새로 고칠 때마다 다시 온보딩하는 것을 원하지 않습니다. 그것은 꽤 성가신 일입니다. HagiCode는 localStorage를 사용하여 온보딩 상태를 관리합니다:

```typescript
export type GuideState = 'pending' | 'dismissed' | 'completed';

export interface UserGuideState {
  session: GuideState;
  detailGuides: Record<string, GuideState>;
}

// 상태 읽기
export const getUserGuideState = (): UserGuideState => {
  const state = localStorage.getItem('userGuideState');
  return state ? JSON.parse(state) : { session: 'pending', detailGuides: {} };
};

// 상태 업데이트
export const setUserGuideState = (state: UserGuideState) => {
  localStorage.setItem('userGuideState', JSON.stringify(state));
};
```

우리는 세 가지 상태를 정의했습니다:

- `pending`: 온보딩 진행 중, 사용자가 아직 완료하거나 건너뛰지 않음
- `dismissed`: 사용자가 온보딩을主动으로 닫음
- `completed`: 사용자가 모든 단계를 완료함

제안 상세 페이지의 온보딩의 경우, 더 세분화된 상태 추적도 지원합니다(`detailGuides` 사전을 통해). 하나의 제안은 여러 단계(초안, 검토, 실행 완료)를 거칠 수 있으며, 각 단계는 다른 온보딩이 필요합니다.毕竟 일의 상태는 항상 변화합니다.

### 대상 요소定位

driver.js는 CSS 선택자를 사용하여 대상 요소를定位합니다. HagiCode는 `data-guide` 커스텀 속성을 사용하여 온보딩 대상을 표시하는 약속을 채택했습니다:

```typescript
const steps = [
  {
    element: '[data-guide="launch"]',
    popover: {
      title: '새 대화 시작',
      description: '여기를 클릭하여 새 대화 세션을 만듭니다...'
    }
  }
];
```

컴포넌트에서는 다음과 같이 사용합니다:

```typescript
<button data-guide="launch" onClick={handleLaunch}>
  새 대화
</button>
```

이 접근 방식의 장점은:

- 비즈니스 스타일 클래스 이름과의 충돌을 피합니다
- 의미가 명확하여 이 요소가 온보딩과 관련되어 있다는 것을 한눈에 알 수 있습니다
- 통합 관리와 유지보수에 용이합니다

### 동적 가져오기 최적화

온보딩 기능은 특정 시나리오에서만 필요합니다(예: 신규 사용자가 처음 방문할 때), 우리는 동적 가져오기를 사용하여 초기 로딩 성능을 최적화합니다:

```typescript
const initNewUserGuide = async () => {
  // driver.js 동적 가져오기
  const { driver } = await import('driver.js');
  await import('driver.js/dist/driver.css');

  // 온보딩 초기화
  const newConversationDriver = driver({
    // ...구성
  });

  newConversationDriver.drive();
};
```

이렇게 하면 driver.js와 해당 스타일 파일이 필요할 때만 로드되어 첫 화면 성능에 영향을 주지 않습니다.毕竟 누구도 당분간 사용하지 않을 것을 위해 기다리는 대가를 지불하고 싶어 하지 않습니다.

## 온보딩 프로세스 디자인

HagiCode는 두 가지 온보딩 경로를 구현하여 사용자의 핵심 사용 시나리오를覆盖합니다.

### 세션 온보딩(10단계)

이 온보딩은 사용자가 대화 생성부터 첫 번째 완전한 제안 제출까지의 전체 프로세스를 완료하도록 돕습니다:

1. **launch** - 온보딩 시작, "새 대화" 버튼 소개
2. **compose** - 입력란에 요청을 입력하도록 안내
3. **send** - 전송 버튼 클릭 안내
4. **proposal-launch-readme** - README 제안 생성 안내
5. **proposal-compose-readme** - README 요청 내용 편집 안내
6. **proposal-submit-readme** - README 제안 제출 안내
7. **proposal-launch-agents** - AGENTS.md 제안 생성 안내
8. **proposal-compose-agents** - AGENTS.md 요청 편집 안내
9. **proposal-submit-agents** - AGENTS.md 제안 제출 안내
10. **proposal-wait** - AI가 처리 중이니 잠시 기다려 주십시오 안내

이 온보딩의 디자인 아이디어는: 두 개의 실제 제안 생성 작업(README 및 AGENTS.md)을 통해 사용자가 직접 HagiCode의 핵심 워크플로우를 경험하게 하는 것입니다.毕竟纸上得来终觉浅，绝知此事要躬行입니다.

### 제안 상세 온보딩(3단계)

사용자가 제안 상세 페이지에 들어갈 때, 제안의 현재 상태에 따라 해당 온보딩을 트리거합니다:

1. **drafting**(초안 단계) - 사용자가 AI가 생성한 계획을 확인하도록 안내
2. **reviewing**(검토 단계) - 사용자가 계획을 실행하도록 안내
3. **executionCompleted**(완료 단계) - 사용자가 계획을 보관하도록 안내

이 온보딩의 특징은 상태驱动입니다——제안의 실제 상태에 따라 동적으로 어떤 온보딩 단계를 표시할지 결정합니다. 사물은 항상 변화하므로 온보딩도 따라 변화해야 합니다.

이 단계에서 온보딩은 사용자의 주의를 "계획 생성"이라는 핵심 동작에 집중시켜 처음 상세 페이지에 들어갔을 때 무엇을 먼저 해야 할지 모르는 것을 방지합니다.

### 요소 렌더링 재시도 메커니즘

React 애플리케이션에서 온보딩 대상 요소가 아직 렌더링되지 않았을 수 있습니다(예: 비동기 데이터 로드 대기 중). 이 상황을 처리하기 위해 HagiCode는 재시도 메커니즘을 구현했습니다:

```typescript
const waitForElement = (selector: string, maxRetries = 10, interval = 100) => {
  let retries = 0;
  return new Promise<HTMLElement>((resolve, reject) => {
    const checkElement = () => {
      const element = document.querySelector(selector) as HTMLElement;
      if (element) {
        resolve(element);
      } else if (retries < maxRetries) {
        retries++;
        setTimeout(checkElement, interval);
      } else {
        reject(new Error(`Element not found: ${selector}`));
      }
    };
    checkElement();
  });
};
```

온보딩 초기화 전에 이 함수를 호출하여 대상 요소가 이미 존재하는지 확인합니다. 때로는 좀 더 기다리는 것도 가치가 있습니다.

## 모범 사례 요약

HagiCode의 실천 경험을 바탕으로 몇 가지 핵심 모범 사례를 공유합니다:

### 1. 온보딩은 "탈출 가능한" 것이어야 합니다

사용자에게 온보딩 완료를 강요하지 마세요. 일부 사용자는 탐색형이며 스스로 탐색하는 것을 더 선호합니다. 명확한 "건너뛰기" 버튼을 제공하고 사용자의 선택을 기억하여 다시 방해하지 마세요.毕竟 아름다운 것 또는 사람은 반드시 소유할 필요는 없으며, 그녀가 여전히 아름답다면 자신이 그녀의 아름다움을 잘 바라보면 됩니다.

### 2. 온보딩 내용은 간결하고 강력해야 합니다

각 온보딩 단계는 단일 목표에 집중해야 합니다:

- **Title**: 간결하고 명확하며 10자 이내
- **Description**: 핵심을 찌르고, 사용자에게 "이것은 무엇인가"와 "왜 사용해야 하는가"를 알려줍니다

장황한 설명을 피하세요——사용자는 온보딩 단계에서 주의력이 매우 제한적입니다. 말이 많으면 아무도 보고 싶어 하지 않습니다.

### 3. 선택자는 안정적이어야 합니다

안정적이고 자주 변하지 않는 요소 표시 방식을 사용하세요. `data-guide` 커스텀 속성은 좋은 선택입니다. 클래스 이름이나 DOM 구조에 의존하지 마세요. 이것들은 리팩토링 중에 쉽게 변할 수 있습니다. 코드는 항상 변하지만, 일부 것들은 가능한 한 안정적이어야 합니다.

### 4. 온보딩 테스트

HagiCode는 온보딩 기능을 위해 완전한 테스트 케이스를 작성했습니다:

```typescript
describe('NewUserConversationGuide', () => {
  it('온보딩 상태를 올바르게 초기화해야 함', () => {
    const state = getUserGuideState();
    expect(state.session).toBe('pending');
  });

  it('온보딩 상태를 올바르게 업데이트해야 함', () => {
    setUserGuideState({ session: 'completed', detailGuides: {} });
    const state = getUserGuideState();
    expect(state.session).toBe('completed');
  });
});
```

테스트는 코드를 리팩토링할 때 온보딩 기능을 실수로 파괴하지 않도록 보장할 수 있습니다.毕竟 누구도 코드를 조금 수정하여 이전 기능을 망가뜨리고 싶어 하지 않습니다.

### 5. 성능 최적화

- 동적 가져오기를 사용하여 온보딩 라이브러리를 지연 로드합니다
- 사용자가 이미 온보딩을 완료한 후에도 온보딩 로직을 초기화하지 않도록 합니다
- 온보딩 애니메이션의 성능 영향을 고려하고, 저사양 장치에서는 애니메이션을 끌 수 있습니다

성능这个东西, 삶과 같습니다. 아껴야 할 곳은 아껴야 합니다.

## 요약

신규 사용자 온보딩은 제품 사용자 경험을 향상시키는 중요한环节입니다. HagiCode 프로젝트에서 우리는 driver.js를 사용하여 세션 생성부터 제안 실행까지 전체 워크플로우를覆盖하는 완전한 온보딩 시스템을 구축했습니다.

이 글의 공유를 통해 우리가 전달하고 싶은 핵심 관점은 다음과 같습니다:

1. **기술 선정은 요구사항과 일치해야 합니다**: driver.js는 가장 강력하지는 않지만 우리에게는 가장 적합합니다
2. **상태 관리가 핵심입니다**: localStorage로 온보딩 상태를 지속화하여 사용자에게 반복적인 방해를 피합니다
3. **온보딩 디자인은 집중해야 합니다**: 각 단계는 하나의 문제를 해결하고, 너무 욕심내지 마세요
4. **코드 구조는 명확해야 합니다**: 온보딩 구성, 상태 관리 및 UI 로직을 분리하여 유지보수를 용이하게 합니다

자신의 프로젝트에 신규 사용자 온보딩 기능을 추가 중이라면, 이 글의 실천 경험이 도움이 되기를 바랍니다. 사실 기술这个东西, 별로 신비로운 것은 없습니다. 더 많이 시도하고, 더 많이 요약하면, 점차 좋아집니다......
