## React 프로젝트에서 우아한 신규 사용자 온보딩 구축하기: HagiCode의 driver.js 실천

> 사용자가 제품을 처음 열었을 때, 정말로 어디서 시작해야 할지 알고 있을까요? 이 글에서는 HagiCode 프로젝트에서 driver.js를 사용하여 신규 사용자 온보딩을 구현한 방법을 이야기해 보려 합니다. 작은 실천 예시로 생각해 주시면 감사하겠습니다.

## 배경

이런 상황을 겪어 본 적이 있으신가요? 새 사용자가 제품에 가입하고, 페이지를 열자마자 당황한 표정입니다. 이리저리 둘러보면서 무엇을 클릭해야 할지, 다음에 무엇을 해야 할지 모릅니다. 개발자로서 우리는 종종 사용자가 "스스로 탐색할" 것이라고 가정합니다.毕竟 인간의 호기심은 무한하니까요. 하지만 현실은 다릅니다: 올바른 진입점을 찾지 못하면 대부분의 사용자는 몇 분 안에 조용히 떠나갑니다.

신규 사용자 온보딩은 이 문제를 해결하는 중요한 방법이지만, 잘 구축하는 것은 그렇게 간단하지 않습니다. 좋은 온보딩 시스템은 다음을 필요로 합니다:

- 페이지 요소를 정밀하게 위치 지정하고 하이라이트합니다
- 다단계 온보딩 흐름을 지원합니다
- 사용자의 선택(완료 또는 건너뛰기)을 기억합니다
- 페이지 성능과 정상적인 상호작용에 영향을 주지 않습니다
- 코드 구조를 명확하고 유지보수하기 쉽게 유지합니다

HagiCode를 구축하면서 우리도 같은 과제에 직면했습니다. HagiCode는 AI 코딩 어시스턴트로, 핵심 워크플로우는 OpenSpec 워크플로우입니다: "사용자가 Proposal을 생성 -> AI가 계획을 생성 -> 사용자가 검토 -> AI가 실행". 이 개념을 처음 접하는 사용자에게 워크플로우는 완전히 새롭기 때문에, 빠르게 시작할 수 있도록 탄탄한 온보딩이 필요합니다. 새로운 것에 익숙해지는 데는 항상 약간의 시간이 걸립니다.

## HagiCode에 대하여

이 글에서 공유하는 접근 방식은 [HagiCode](https://hagicode.com) 프로젝트의 실무 경험에서 비롯되었습니다. HagiCode는 Claude 기반의 AI 코딩 어시스턴트로, OpenSpec 워크플로우를 통해 개발자가 코딩 작업을 더 효율적으로 완료하도록 돕습니다. [GitHub](https://github.com/HagiCode-org/site)에서 오픈소스 코드를 확인할 수 있습니다.

## driver.js를 선택한 이유

기술 평가 단계에서 여러 주류 온보딩 라이브러리를 검토했습니다. 각각 장점이 있었습니다:

- **Intro.js**: 강력하지만 비교적 크고, 스타일 커스터마이징이 다소 복잡합니다
- **Shepherd.js**: 잘 설계된 API이지만 우리의 사용 사례에는 약간 "무거운" 편입니다
- **driver.js**: 가볍고 간결하며 직관적인 API로, React 생태계에서 잘 동작합니다

결국 driver.js를 선택했습니다. 특별히 극적인 이유는 없습니다. 선택은 주로 다음 고려 사항에서 비롯되었습니다:

1. **가벼움**: 코어 라이브러리가 작아 번들 크기를 크게 증가시키지 않습니다
2. **간단한 API**: 구성이 명확하고 직관적이어서 쉽게 익힐 수 있습니다
3. **유연함**: 커스텀 위치 지정, 스타일링, 상호작용 동작을 지원합니다
4. **동적 임포트**: 필요에 따라 로드되어 첫 화면 성능에 영향을 주지 않습니다

기술 선택에는 보편적으로 최선인 답이 거의 없습니다. 보통 가장 잘 맞는 옵션이 있을 뿐입니다.

## 기술 구현

### 핵심 구성

driver.js는 매우 직관적인 구성 모델을 가지고 있습니다. HagiCode 프로젝트에서 사용하는 핵심 구성은 다음과 같습니다:

```typescript
import { driver } from 'driver.js';
import 'driver.js/dist/driver.css';

const newConversationDriver = driver({
  allowClose: true,           // 사용자가 가이드를 닫을 수 있도록 허용
  animate: true,              // 애니메이션 활성화
  overlayClickBehavior: 'close', // 오버레이 클릭 시 가이드 닫기
  disableActiveInteraction: false, // 요소 상호작용 유지
  showProgress: false,        // 진행률 표시줄 미표시 (직접 관리)
  steps: guideSteps           // 가이드 단계 배열
});
```

이 설정의 배경은 다음과 같습니다:

- `allowClose: true` - 사용자의 선택을 존중하고 가이드 완료를 강제하지 않습니다
- `disableActiveInteraction: false` - 일부 단계에서는 실제 사용자 동작이 필요하므로(예: 입력), 상호작용을 비활성화할 수 없습니다
- `overlayClickBehavior: 'close'` - 사용자에게 빠른 종료 방법을 제공합니다

### 상태 관리

온보딩 상태를 지속하는 것은 매우 중요합니다. 페이지를 새로고침할 때마다 가이드를 다시 시작하면 빠르게 짜증스러워집니다. HagiCode는 `localStorage`를 사용하여 가이드 상태를 관리합니다:

```typescript
export type GuideState = 'pending' | 'dismissed' | 'completed';

export interface UserGuideState {
  session: GuideState;
  detailGuides: Record<string, GuideState>;
}

// 상태 읽기
export const getUserGuideState = (): UserGuideState => {
  const state = localStorage.getItem('userGuideState');
  return state ? JSON.parse(state) : { session: 'pending', detailGuides: {} };
};

// 상태 업데이트
export const setUserGuideState = (state: UserGuideState) => {
  localStorage.setItem('userGuideState', JSON.stringify(state));
};
```

세 가지 상태를 정의합니다:

- `pending`: 가이드가 아직 진행 중이며, 사용자가 완료하거나 건너뛰지 않은 상태
- `dismissed`: 사용자가 가이드를 적극적으로 닫은 상태
- `completed`: 사용자가 모든 단계를 완료한 상태

Proposal 상세 페이지 온보딩의 경우, `detailGuides` 맵을 통해 더 세밀한 상태 추적도 지원합니다. 하나의 Proposal이 여러 단계(초안, 검토, 실행 완료)를 거칠 수 있고, 각 단계마다 다른 안내가 필요하기 때문입니다. 상황은 항상 변하고, 온보딩도 이를 반영해야 합니다.

### 대상 요소 선택

driver.js는 CSS 선택자를 사용하여 대상 요소를 위치 지정합니다. HagiCode는 간단한 관례를 따릅니다: 사용자 정의 `data-guide` 속성으로 온보딩 대상을 표시합니다:

```typescript
const steps = [
  {
    element: '[data-guide="launch"]',
    popover: {
      title: '새 대화 시작',
      description: '여기를 클릭하여 새 대화 세션을 생성합니다...'
    }
  }
];
```

컴포넌트에서는 다음과 같습니다:

```typescript
<button data-guide="launch" onClick={handleLaunch}>
  새 대화
</button>
```

이 방식의 장점은 다음과 같습니다:

- 비즈니스 스타일링 클래스 이름과의 충돌을 피합니다
- 의미가 명확하여 온보딩과 관련된 요소임을 즉시 알 수 있습니다
- 일관되게 관리하고 유지보수하기 쉽습니다

### 동적 임포트 최적화

온보딩 기능은 사용자의 첫 방문과 같은 특정 시나리오에서만 필요하므로, 동적 임포트를 사용하여 초기 로딩 성능을 최적화합니다:

```typescript
const initNewUserGuide = async () => {
  // driver.js 동적 임포트
  const { driver } = await import('driver.js');
  await import('driver.js/dist/driver.css');

  // 가이드 초기화
  const newConversationDriver = driver({
    // ...구성
  });

  newConversationDriver.drive();
};
```

이렇게 하면 driver.js와 스타일시트가 필요할 때만 로드되어 첫 화면 성능에 영향을 주지 않습니다. 아직 필요하지도 않은 것을 기다리는 것을 좋아하는 사람은 많지 않습니다.

## 온보딩 흐름 설계

HagiCode는 사용자의 핵심 시나리오를 커버하는 두 가지 온보딩 경로를 구현합니다.

### 세션 온보딩 (10단계)

이 온보딩 경로는 사용자가 대화 생성에서 첫 번째 완전한 Proposal 제출까지 전체 흐름을 완료하도록 돕습니다:

1. **launch** - 가이드 시작 및 "새 대화" 버튼 소개
2. **compose** - 사용자가 입력 상자에 요청을 입력하도록 안내
3. **send** - 사용자가 전송 버튼을 클릭하도록 안내
4. **proposal-launch-readme** - 사용자가 README Proposal을 생성하도록 안내
5. **proposal-compose-readme** - 사용자가 README 요청 내용을 편집하도록 안내
6. **proposal-submit-readme** - 사용자가 README Proposal을 제출하도록 안내
7. **proposal-launch-agents** - 사용자가 `AGENTS.md` Proposal을 생성하도록 안내
8. **proposal-compose-agents** - 사용자가 `AGENTS.md` 요청을 편집하도록 안내
9. **proposal-submit-agents** - 사용자가 `AGENTS.md` Proposal을 제출하도록 안내
10. **proposal-wait** - AI가 처리 중이며 잠시 기다려야 한다는 것을 설명

이 경로의 아이디어는 사용자가 `README`용과 `AGENTS.md`용 두 가지 실제 Proposal 생성 작업을 통해 HagiCode의 핵심 워크플로우를 직접 경험하게 하는 것입니다. 워크플로우에 대해 듣는 것과 직접 경험하는 것은 큰 차이가 있습니다.

### Proposal 상세 온보딩 (3단계)

사용자가 Proposal 상세 페이지에 들어가면 HagiCode는 Proposal의 현재 상태에 따라 해당 가이드를 트리거합니다:

1. **drafting** (초안 단계) - 사용자가 AI가 생성한 계획을 검토하도록 안내
2. **reviewing** (검토 단계) - 사용자가 계획을 실행하도록 안내
3. **executionCompleted** (완료 단계) - 사용자가 계획을 아카이브하도록 안내

이 가이드의 특징은 상태 기반이라는 것입니다: Proposal의 실제 상태에 따라 표시할 온보딩 단계를 동적으로 결정합니다. 상황은 변하고, 온보딩도 변해야 합니다.

### 요소 렌더링 재시도 메커니즘

React 애플리케이션에서 대상 온보딩 요소가 아직 렌더링을 완료하지 않았을 수 있습니다(예: 비동기 데이터가 아직 로딩 중인 경우). 이를 처리하기 위해 HagiCode는 재시도 메커니즘을 구현합니다:

```typescript
const waitForElement = (selector: string, maxRetries = 10, interval = 100) => {
  let retries = 0;
  return new Promise<HTMLElement>((resolve, reject) => {
    const checkElement = () => {
      const element = document.querySelector(selector) as HTMLElement;
      if (element) {
        resolve(element);
      } else if (retries < maxRetries) {
        retries++;
        setTimeout(checkElement, interval);
      } else {
        reject(new Error(`Element not found: ${selector}`));
      }
    };
    checkElement();
  });
};
```

가이드를 초기화하기 전에 이 함수를 호출하여 대상 요소가 이미 존재하는지 확인합니다. 때로는 조금 더 기다리는 것이 가치 있습니다.

## 모범 사례 요약

HagiCode의 실무 경험을 바탕으로 몇 가지 핵심 모범 사례를 공유합니다:

### 1. 온보딩은 쉽게 종료할 수 있어야 합니다

사용자에게 온보딩 완료를 강제하지 마십시오. 일부 사용자는 탐험가 성향이 있어 스스로 알아내는 것을 선호합니다. 명확한 "건너뛰기" 버튼을 제공하고, 그 선택을 기억하며, 다음번에는 다시 방해하지 마십시오.

### 2. 온보딩 내용은 짧고 날카롭게 유지

각 온보딩 단계는 단일 목표에 집중해야 합니다:

- **제목**: 짧고 명확하게, 이상적으로 몇 단어 이내
- **설명**: 핵심을 직접 전달하고, 이것이 무엇이며 왜 중요한지 알려줍니다

장황한 설명을 피하십시오. 온보딩 중에는 사용자의 주의력이 제한되어 있으며, 말이 많을수록 읽을 가능성이 낮아집니다.

### 3. 선택자는 안정적으로 유지

자주 변경되지 않는 안정적인 요소 마커를 사용하십시오. 사용자 정의 `data-guide` 속성이 좋은 선택입니다. 클래스 이름이나 DOM 구조에 의존하지 마십시오. 리팩토링 중에 쉽게 변경될 수 있기 때문입니다. 코드는 항상 변하지만, 일부 앵커는 가능하면 안정적으로 유지되어야 합니다.

### 4. 온보딩을 테스트하십시오

HagiCode는 온보딩 기능에 대한 완전한 테스트 케이스를 포함합니다:

```typescript
describe('NewUserConversationGuide', () => {
  it('should initialize guide state correctly', () => {
    const state = getUserGuideState();
    expect(state.session).toBe('pending');
  });

  it('should update guide state correctly', () => {
    setUserGuideState({ session: 'completed', detailGuides: {} });
    const state = getUserGuideState();

  });
});
```

테스트는 리팩토링이 온보딩 동작을 실수로 손상시키지 않도록 보장합니다. 작은 코드 변경이 이전에 작동하던 기능을 조용히 망가뜨리는 것을 아무도 원하지 않습니다.

### 5. 성능 최적화

- 동적 임포트를 사용하여 온보딩 라이브러리를 지연 로드합니다
- 사용자가 이미 가이드를 완료한 후에는 온보딩 로직을 초기화하지 않습니다
- 애니메이션의 성능 영향을 고려하고, 필요하면 저사양 기기에서 비활성화합니다

성능은 인생의 많은 것과 마찬가지로 약간의 신중한 예산 편성이 필요합니다.

## 결론

신규 사용자 온보딩은 제품 사용자 경험을 향상하는 중요한 부분입니다. HagiCode 프로젝트에서는 driver.js를 사용하여 세션 생성부터 Proposal 실행까지 전체 워크플로우를 커버하는 완전한 온보딩 시스템을 구축했습니다.

이 글을 통해 공유하고자 하는 핵심 포인트는 다음과 같습니다:

1. **기술 선택은 실제 요구에 맞아야 합니다**: driver.js가 가장 강력한 옵션은 아니지만 우리에게 가장 잘 맞습니다
2. **상태 관리가 중요합니다**: `localStorage`를 사용하여 온보딩 상태를 지속하고 사용자에게 반복적으로 방해하지 마십시오
3. **온보딩 설계는 집중적으로 유지해야 합니다**: 각 단계는 하나의 문제만 해결해야 합니다
4. **코드 구조는 명확하게 유지해야 합니다**: 온보딩 구성, 상태 관리, UI 로직을 분리하여 유지보수를 쉽게 만듭니다

자체 프로젝트에 신규 사용자 온보딩을 추가 중이시라면, 이 글의 실무 경험이 도움이 되기를 바랍니다. 이런 종류의 기술은 특별히 신비로운 것이 아닙니다. 계속 시도하고, 배운 것을 계속 요약하면, 점차 쉬워집니다.
