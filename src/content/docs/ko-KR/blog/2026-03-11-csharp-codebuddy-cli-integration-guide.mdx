## C# 백엔드 CodeBuddy CLI 통합 실전 가이드

> 이 문서에서는 C# 백엔드 프로젝트에 CodeBuddy CLI를 통합하여 AI 프로그래밍 어시스턴트 기능을 구현하는 완전한 솔루션을 상세히 소개합니다.

:::note[시간대 컨텍스트 설명]
이 문서에서 논의하는 것은 발행 당시의 CodeBuddy 통합 설계와 provider 구조입니다. 현재 상시 지원 범위는 [AI Agent CLI 설치](/related-software-installation/ai-agent-cli/) 및 [설치 가이드](/installation)를 기준으로 하며; `IFlowCli`는 이제 역사적 호환성 값으로만 유지되며, 현재 활성 지원 범위에 속하지 않습니다.
:::

## 배경

현대 AI 코드 어시스턴트 개발에서 단일 AI Provider는 복잡하고 변화무쌍한 개발 시나리오를 충족하기 어렵습니다. 이것은 마치, 인생의 길이 멀다면 어느 한 방향만 고집할 수 없는 것과 같지 않을까요? HagiCode는 다기능 AI 프로그래밍 어시스턴트로 더 나은 사용자 경험을 제공하기 위해 여러 AI Provider를 지원해야 합니다.毕竟, 사용자의 선택권은 충분히 보장되어야 합니다. 2026년 초, 프로젝트는 C# 백엔드에서 CodeBuddy의 ACP(Agent Communication Protocol) 통합 기능을 복원하는 핵심 결정에 직면했습니다.

이전에 프로젝트에서 CodeBuddy 연결이 구현된 적이 있었지만, 관련 코드는 한 번의 리팩토링 과정에서 제거되었습니다. 사실 불평할 것도 없습니다, 코드 이터레이션인데, 잊혀지는 것들은 항상 있게 마련입니다. 이번 기술 솔루션의 목표는 이 기능을 완전히 복원하고 아키텍처를 최적화하여 더욱 견고하고 유지보수 가능하게 만드는 것입니다.

만약 당신도 자신의 프로젝트에 다양한 AI 프로그래밍 어시스턴트를 연결하는 것을 고려하고 있다면, 아래의 솔루션이 약간의 영감을 줄 수도 있습니다 - 이것은 우리가 무수한 함정을 밟은 후에 요약한 경험입니다. 당신이 좀 더 돌아가는 길을 덜 걷게 할 수 있다면, 제가 한 좋은 일이라고 할 수 있겠네요.

## HagiCode에 대하여

이 문서에서 공유하는 솔루션은 [HagiCode](https://hagicode.com) 프로젝트에서의 실천 경험에서 나온 것입니다. HagiCode는 다양한 AI Provider와 크로스 플랫폼 실행을 지원하는 오픈 소스 AI 코드 어시스턴트 프로젝트입니다. 다양한 사용자의 선호를 만족시키기 위해 우리는 다양한 AI 프로그래밍 어시스턴트를 유연하게 전환할 수 있어야 하며, 이것이 이 문서에서 소개할 CodeBuddy 통합 솔루션의 계기가 되었습니다.

HagiCode는 모듈형 설계를 채택하며, AI Provider는 플러그인 가능한 컴포넌트로, 이러한 아키텍처는 우리가 새로운 AI 지원을 쉽게 추가할 수 있게 하며, 기존 기능에 영향을 주지 않습니다. 이 정도는 그만두고, 설계라는 건 처음에 잘해두면 나중에 마음이 편합니다. 만약 당신이 우리의 기술 아키텍처에 관심이 있다면 GitHub에서 완전한 소스 코드를 확인할 수 있습니다.

## 아키텍처 설계

### 계층형 아키텍처 개요

C#과 CodeBuddy의 연결은 명확한 계층형 아키텍처를 채택하며, 이러한 설계는 코드 책임이 명확하게 분리되어 후기 유지보수에 더 편리합니다:

```
┌─────────────────────────────────────────────┐
│           Provider 계층                      │
│   AIProviderType 열거형 + 확장 메서드         │
├─────────────────────────────────────────────┤
│           Provider 팩토리 계층                │
│   AIProviderFactory 의존성 주입 팩토리        │
├─────────────────────────────────────────────┤
│           Provider 구현 계층                  │
│   CodebuddyCliProvider 구체적 구현            │
├─────────────────────────────────────────────┤
│           ACP 인프라 계층                     │
│  ACPSessionManager / StdioAcpTransport      │
│  AcpRpcClient / AcpAgentClient              │
└─────────────────────────────────────────────┘
```

이러한 계층화의 장점은 무엇일까요? 간단히 말하면 각 계층 사이에 서로 방해하지 않는다는 것입니다. 예를 들어 나중에 다른 통신 방식으로 바꾸려고 한다면(예를 들어 stdio에서 WebSocket으로), 가장 아래 계층만 수정하면 되고 위의 비즈니스 코드는 완전히 건드릴 필요가 없습니다.毕竟, 누구도 하나를 건드려 전체에 영향을 받고 싶지 않으며, 통신 방식을 바꾸려고 반나절 비즈니스 코드를 수정하는 것은 너무 비참합니다.

### 핵심 컴포넌트 분석

**Provider 계층**은 전체 아키텍처의 기초입니다. 우리는 `AIProviderType` 열거형을 정의했으며, 여기서 `CodebuddyCli = 3`을 열거형 값으로 사용하고, 확장 메서드를 통해 문자열과 열거형 간의 양방향 매핑을 구현합니다. 이렇게 하면 설정 파일의 문자열을 열거형으로 매우 편리하게 변환할 수 있으며, 디버깅 시 열거형을 문자열로 출력할 수도 있습니다. 이 정도는 그만두고, 사실 매핑 관계일 뿐이지만, 잘해두면 편합니다.

**Provider 팩토리 계층**은 설정에 따라 해당 Provider 인스턴스를 생성하는 책임을 집니다. 여기서는 .NET의 의존성 주입 메커니즘을 사용하며, `ActivatorUtilities.CreateInstance`와 결합하여 동적 생성을 구현합니다. 팩토리 패턴의 장점은 새 Provider를 추가할 때 생성 로직만 추가하면 되며 기존 코드를 수정할 필요가 없다는 것입니다. 이것은 글쓰기와 비슷합니다, 새 장을 추가하려면 새 장을 추가하면 되며 앞의 내용을 모두 다시 쓸 필요가 없습니다.

**Provider 구현 계층**은 실제로 일하는 곳입니다. `CodebuddyCliProvider`는 `IAIProvider` 인터페이스를 구현하며 `ExecuteAsync`(비스트리밍)와 `StreamAsync`(스트리밍) 두 가지 호출 방식을 제공합니다.

**ACP 인프라 계층**은 통신의 하부 지지입니다. 이 계층은 프로세스 관리, 메시지 직렬화, 응답 구문 분석 등 모든 프로토콜 세부 사항을 처리합니다. 마치 집의 기초와 같아서 위가 아무리 아름다워도 아래가 안정적이어야 합니다.

## 통신 메커니즘

### Stdio 전송 모드

CodeBuddy는 **Stdio(표준 입출력)** 방식으로 외부 프로세스와 통신합니다. 시작 명령은 매우 간단합니다:

```bash
codebuddy --acp
```

그런 다음 표준 입출력을 통해 JSON-RPC 메시지 교환을 합니다. 이 방식의 장점은 다음과 같습니다:

1. **빠른 시작**: 로컬 프로세스 통신은 네트워크 지연이 없습니다
2. **간단한 설정**: 실행 파일 경로만 지정하면 됩니다
3. **환경 격리**: 각 세션은 독립 프로세스로 서로 영향을 주지 않습니다

통신 과정에서 환경 변수 주입을 지원하며, 일반적으로 사용되는 것은 다음과 같습니다:

- `CODEBUDDY_API_KEY`: API 키 인증
- `CODEBUDDY_INTERNET_ENVIRONMENT`: 네트워크 환경 설정

이것은 마치, 사람과 사람 사이의 소통에서 편리한 방식을 찾아야 대화할 수 있는 것과 같습니다.

### 메시지 프로토콜

ACP는 JSON-RPC 2.0 프로토콜을 기반으로 하며, 메시지 형식은 대략 이렇습니다:

```json
// 요청 메시지
{
  "jsonrpc": "2.0",
  "id": 1,
  "method": "agent/prompt",
  "params": {
    "prompt": "정렬 알고리즘을 작성해주세요",
    "sessionId": "session-123"
  }
}

// 응답 메시지
{
  "jsonrpc": "2.0",
  "id": 1,
  "result": {
    "content": "여기는 AI의 응답..."
  }
}
```

실제 구현에서 우리는 이러한 프로토콜 세부 사항을 모두 캡슐화했으며, 상위 비즈니스 코드는 prompt와 response에만 집중하면 됩니다. 이 정도는 그만두고, 캡슐화를 잘해두면 나중에 사용하는 사람이 편합니다.

## 핵심 구현

### 1. Provider 계약 복원

먼저 열거형 파일에서 CodeBuddy 유형을 복원합니다:

```csharp
// PCode.Models/AIProviderType.cs
public enum AIProviderType
{
    ClaudeCodeCli = 0,
    CodexCli = 1,
    GitHubCopilot = 2,
    CodebuddyCli = 3,  // 이 열거형 값 복원
    OpenCodeCli = 4,
    IFlowCli = 5,
}
```

그런 다음 확장 메서드에서 문자열 매핑을 추가하여 설정 파일에서 문자열로 Provider를 지정할 수 있게 합니다:

```csharp
// AIProviderTypeExtensions.cs
private static readonly Dictionary<string, AIProviderType> _typeMap = new(
    StringComparer.OrdinalIgnoreCase)
{
    ["CodebuddyCli"] = AIProviderType.CodebuddyCli,
    ["Codebuddy"] = AIProviderType.CodebuddyCli,
    ["codebuddy"] = AIProviderType.CodebuddyCli,
    // ... 다른 provider의 매핑
};
```

### 2. Provider 팩토리 통합

팩토리 클래스에서 CodeBuddy의 생성 분기를 추가합니다:

```csharp
// AIProviderFactory.cs
private IAIProvider? CreateProvider(AIProviderType providerType, ProviderConfiguration config)
{
    return providerType switch
    {
        AIProviderType.CodebuddyCli =>
            ActivatorUtilities.CreateInstance<CodebuddyCliProvider>(
                _serviceProvider,
                Options.Create(config)),
        // ... 다른 provider
        _ => throw new NotSupportedException($"Provider {providerType} not supported")
    };
}
```

여기서는 의존성 주입의 `ActivatorUtilities`를 사용했으며, 이것은 자동으로 생성자 매개변수 주입을 처리하므로 매우 편리합니다. 이 정도는 그만두고, .NET의 것은 올바르게 사용하면 편합니다.

### 3. 완전한 Provider 구현

다음은 `CodebuddyCliProvider`의 핵심 구현으로 스트리밍 및 비스트리밍 두 가지 호출 방식을 포함합니다:

```csharp
public class CodebuddyCliProvider : IAIProvider
{
    private readonly ILogger<CodebuddyCliProvider> _logger;
    private readonly IACPSessionManager _sessionManager;
    private readonly ProviderConfiguration _config;

    public string Name => "CodebuddyCli";
    public bool SupportsStreaming => true;
    public ProviderCapabilities Capabilities { get; }

    public CodebuddyCliProvider(
        ILogger<CodebuddyCliProvider> logger,
        IACPSessionManager sessionManager,
        IOptions<ProviderConfiguration> config)
    {
        _logger = logger;
        _sessionManager = sessionManager;
        _config = config.Value;

        // 현재 Provider의 기능 정의
        Capabilities = new ProviderCapabilities
        {
            SupportsStreaming = true,
            SupportsTools = true,
            SupportsSystemMessages = true,
            SupportsArtifacts = false,
            MaxTokens = 8192
        };
    }

    // 비스트리밍 호출: 모든 결과가 함께 반환될 때까지 기다립니다
    public async Task<AIResponse> ExecuteAsync(
        AIRequest request,
        CancellationToken cancellationToken = default)
    {
        // 요청을 위한 독립 세션 생성
        var session = await _sessionManager.CreateSessionAsync(
            "CodebuddyCli",
            request.WorkingDirectory,
            cancellationToken,
            request.SessionId);

        try
        {
            var fullPrompt = BuildPrompt(request);
            await session.SendPromptAsync(fullPrompt, cancellationToken);

            var responseBuilder = new StringBuilder();
            var toolCalls = new List<AIToolCall>();

            // 모든 응답 청크 수집
            await foreach (var chunk in StreamFromSession(session, cancellationToken))
            {
                if (!string.IsNullOrEmpty(chunk.Content))
                {
                    responseBuilder.Append(chunk.Content);
                }
                // 도구 호출 처리...
            }

            return new AIResponse
            {
                Content = AIResultContentSanitizer.SanitizeResultContent(
                    responseBuilder.ToString()),
                ToolCalls = toolCalls,
                Provider = Name,
                Model = string.Empty
            };
        }
        finally
        {
            // 세션 리소스 해제
            await session.DisposeAsync();
        }
    }

    // 스트리밍 호출: 실시간으로 응답 청크를 반환합니다
    public async IAsyncEnumerable<AIStreamingChunk> StreamAsync(
        AIRequest request,
        [EnumeratorCancellation] CancellationToken cancellationToken = default)
    {
        var session = await _sessionManager.CreateSessionAsync(
            "CodebuddyCli",
            request.WorkingDirectory,
            cancellationToken);

        try
        {
            var fullPrompt = BuildPrompt(request);
            await session.SendPromptAsync(fullPrompt, cancellationToken);

            await foreach (var chunk in StreamFromSession(session, cancellationToken))
            {
                yield return chunk;
            }
        }
        finally
        {
            await session.DisposeAsync();
        }
    }

    private async IAsyncEnumerable<AIStreamingChunk> StreamFromSession(
        IACPSession session,
        [EnumeratorCancellation] CancellationToken cancellationToken)
    {
        // 세션의 모든 업데이트를 순회
        await foreach (var notification in session.ReceiveUpdatesAsync(cancellationToken))
        {
            switch (notification.Update)
            {
                case AgentMessageChunkSessionUpdate agentMessage:
                    // 텍스트 내용 청크 처리
                    if (agentMessage.Content is AcpImp.TextContentBlock textContent)
                    {
                        yield return new AIStreamingChunk
                        {
                            Content = textContent.Text,
                            Type = StreamingChunkType.ContentDelta,
                            IsComplete = false
                        };
                    }
                    break;

                case ToolCallSessionUpdate toolCall:
                    // 도구 호출 처리
                    yield return new AIStreamingChunk
                    {
                        Content = string.Empty,
                        Type = StreamingChunkType.ToolCallDelta,
                        ToolCallDelta = new AIToolCallDelta
                        {
                            Id = toolCall.ToolCallId,
                            Name = toolCall.Kind.ToString(),
                            Arguments = toolCall.RawInput?.ToString()
                        }
                    };
                    break;

                case AcpImp.PromptCompletedSessionUpdate:
                    // 응답 완료
                    yield break;
            }
        }
    }

    // 완전한 프롬프트 구성
    private string BuildPrompt(AIRequest request, string? embeddedCommandPrompt = null)
    {
        var sb = new StringBuilder();

        // 임베드된 명령 프롬프트(있는 경우)
        if (!string.IsNullOrEmpty(embeddedCommandPrompt))
        {
            sb.AppendLine(embeddedCommandPrompt);
            sb.AppendLine();
        }

        // 시스템 메시지
        if (!string.IsNullOrEmpty(request.SystemMessage))
        {
            sb.AppendLine(request.SystemMessage);
            sb.AppendLine();
        }

        // 사용자 prompt
        sb.Append(request.Prompt);
        return sb.ToString();
    }
}
```

이 코드에는 몇 가지 핵심 사항이 있습니다:

1. **세션 관리**: 각 요청은 독립 세션을 생성하며 요청 완료 후 리소스를 해제합니다. 이것은 함정을 밟고 나온 경험입니다 - 세션 재사용이 잘못되면 상태 오염 문제가 발생하기 쉽습니다.毕竟, 사용한 후에는 정리해야 하며 그렇지 않으면 다음 사용하는 사람이 불편합니다.

2. **스트리밍 처리**: `IAsyncEnumerable`는 응답이 생성되는 대로 반환할 수 있으며 모든 내용이 생성될 때까지 기다릴 필요가 없습니다. 이것은 긴 텍스트 시나리오에서 특히 중요하며 사용자 경험이 훨씬 좋아집니다. 마치, 결과를 기다리는 사람도 계속 기다리고 싶지 않은 것과 같습니다.

3. **도구 호출**: CodeBuddy는 도구 호출(Function Calling)을 지원하며 `ToolCallSessionUpdate`를 통해 처리합니다. 이 기능은 복잡한 코드 편집 작업에 매우 중요합니다.

4. **내용 필터링**: `AIResultContentSanitizer`를 사용하여 Think 블록 내용을 필터링하고 출력을 깔끔하게 유지합니다.

### 4. 의존성 주입 설정

모듈 등록에서 관련 서비스를 추가합니다:

```csharp
// PCodeClaudeHelperModule.cs
public void ConfigureModule(IServiceCollection context)
{
    // Provider 등록
    context.Services.AddTransient<CodebuddyCliProvider>();

    // ACP 인프라 등록
    context.Services.AddSingleton<IACPSessionManager, ACPSessionManager>();
    context.Services.AddSingleton<IAcpPlatformConfigurationResolver, AcpPlatformConfigurationResolver>();
    context.Services.AddSingleton<IAIRequestToAcpMapper, AIRequestToAcpMapper>();
    context.Services.AddSingleton<IAcpToAIResponseMapper, AcpToAIResponseMapper>();
}
```

## 설정 예시

### 설정 파일

`appsettings.json`에서 CodeBuddy 관련 설정을 추가합니다:

```yaml
AI:
  # 기본 사용 Provider
  DefaultProvider: "CodebuddyCli"

  # Provider 설정
  Providers:
    CodebuddyCli:
      Type: "CodebuddyCli"
      WorkingDirectory: "C:/projects/my-app"
      ExecutablePath: "C:/tools/codebuddy.cmd"

  # 플랫폼 관련 설정
  PlatformConfigurations:
    CodebuddyCli:
      ExecutablePath: "C:/tools/codebuddy.cmd"
      Arguments: "--acp"
      StartupTimeoutMs: 5000
      EnvironmentVariables:
        CODEBUDDY_API_KEY: "${CODEBUDDY_API_KEY}"
        CODEBUDDY_INTERNET_ENVIRONMENT: "production"
```

### 설정 모델

해당하는 설정 모델 정의:

```csharp
public class CodebuddyPlatformConfiguration : IAcpPlatformConfiguration
{
    public string ProviderName => "CodebuddyCli";
    public AcpTransportType TransportType => AcpTransportType.Stdio;

    public string ExecutablePath { get; set; } = "codebuddy";
    public string Arguments { get; set; } = "--acp";
    public int StartupTimeoutMs { get; set; } = 5000;

    public Dictionary<string, string?>? EnvironmentVariables { get; set; }
}
```

## 실천 경험 요약

### 함정 기록

우리는 구현 과정에서 몇 가지 전형적인 함정에 직면했으며 공유하여 모두가 돌아가는 길을 줄이기를 바랍니다.毕竟, 남의 함정은 자신이 피할 수 있다면 좋은 일입니다:

1. **세션 누출 문제**: 처음에는 세션을 올바르게 해제하지 않아 프로세스 리소스가 고갈되었습니다. 해결 방법은 `try-finally`를 사용하여 모든 요청이 리소스를 해제하도록 보장하는 것입니다. 이 정도는 그만두고, 사용한 물건은 돌려줘야 하며 그렇지 않으면 다음 사람이 무엇을 사용합니까.

2. **환경 변수 전달**: Windows와 Linux의 환경 변수 문법이 다르며 나중에 `Dictionary<string, string?>`를 사용하여 처리하도록 통일했습니다. 크로스 플랫폼은 처음에 통일된 규범을 정해두면 나중에 편합니다.

3. **시간 초과 설정**: CLI 시작에는 시간이 필요하며 5초의 시작 시간 초과를 설정하여 빠른 요청 실패를 방지합니다. 모든 일에는 정도가 필요하며 너무 급하면 오히려 일이 이루어지지 않습니다.

4. **인코딩 문제**: Windows에서 기본 인코딩은 한자 깨짐 현상을 일으킬 수 있으며 프로세스 시작 시 명시적으로 UTF-8 인코딩을 지정해야 합니다. 한자가 표시되지 않으면 너무 불편합니다.

### 성능 최적화

1. **세션 풀**: 빈번한 짧은 요청의 경우 세션 풀을 구현하여 프로세스를 재사용할 수 있습니다
2. **연결 캐시**: 팩토리 클래스는 이미 Provider 인스턴스 캐시를 지원합니다
3. **비동기 우선**: 전 과정에서 비동기 프로그래밍을 사용하며 스레드 차단을 피합니다

성능 같은 것은 최적화할 수 있으면 최적화해야 하며毕竟 사용자가 오래 기다릴수록 경험은 나빠집니다.

## 요약

이 문서에서는 C# 백엔드 CodeBuddy CLI 통합의 완전한 솔루션을 상세히 소개했으며 아키텍처 설계부터 구체적 구현까지 전체 과정을 다루었습니다. 계층형 아키텍처 설계를 통해 우리는 프로토콜 세부 사항과 비즈니스 로직을 분리하여 코드를 더 명확하고 유지보수 가능하게 만들었습니다.

핵심 요점 회고:

- Provider 계층, 팩토리 계층, 구현 계층, 인프라 계층의 계층형 아키텍처 채택
- JSON-RPC over Stdio 방식을 사용한 프로세스 간 통신
- 의존성 주입을 통한 유연한 설정 및 확장 구현
- 스트리밍 및 비스트리밍 두 가지 호출 방식 제공

이 솔루션은 CodeBuddy뿐만 아니라 새 AI Provider 추가도 동일한 패턴을 따릅니다. 만약 당신도 비슷한 다중 AI Provider 통합을 하고 있다면 이 문서가 일부 참고가 되기를 바랍니다. 사실, 글쓰기와 코드 작성은 같으며 공유하여 다른 사람을 도울 수 있다면 헛되지 않습니다.
