## 처음부터: Reasonix CLI를 HagiCode 시스템에 통합하는 방법

> 이 글은 Reasonix CLI를 일등급 Agent Provider로 HagiCode 시스템에 통합하는 완전한 기술 실천을 공유합니다. 3계층 아키텍처 설계, 주요 기술 결정 및 프론트엔드/백엔드 구현 세부사항을 다룹니다.

## 배경

Reasonix CLI, 말해보자면 꽤 흥미로운 것입니다. ACP(Agent Communication Protocol)를 기반으로 하는 AI 코드 도구로 강력한 스트리밍 및 세션 관리 능력을 제공합니다. 사실 HagiCode.Libs 계층에서는 이미 하위 레벨 구현을 완료했지만, 이러한 구성요소들은 고립된 상태에 있습니다. 마치 예쁜 진주들이 목걸이로 엮어지지 않은 것과 같습니다. 사용자는 Hero 직업 선택, 세션 실행 체인 또는 모니터링 패널을 통해 이것을 사용할 수 없습니다. 이는 다소 아쉬운 일입니다.

우리가 직면한 문제는 다음과 같습니다: Reasonix를 Codex, Hermes와 동등한 일등급 Agent Provider로 승격시켜 완전한 백엔드 라우팅과 프론트엔드 표시를 구현하는 방법은 무엇인가? 이것은 단순히 열거형 값을 등록하는 것으로 끝날 수 있는 일이 아니라, 하위 레벨 추상화부터 사용자 인터페이스까지 완전한 체인을 구축해야 합니다. 마치 집을 짓는 것과 같습니다. 단지 기초만 놓는 것으로 끝낼 수 없고, 벽을 쌓고 지붕을 올려야 합니다.

이 통합의 도전 과제는 Reasonix가 로컬 CLI 도구로서 자신만의 성격과 기질을 가지고 있다는 점입니다. 예를 들어 연결 문자열이 필요하지 않고, 모든 매개변수는 사용자가 런타임에 구성합니다. 아직 설치되지 않았을 수 있으므로 우아하게 저하 처리해야 합니다. anthropic 시리즈 모델과 호환되지만, effort, budget 등 ACP 전용 매개변수가 있습니다. 이것은 마치 각자 독특한 대처 방식을 가진 사람과 같아서 강요할 수 없습니다.

신중한 아키텍처 설계와 여러 라운드의 논의를 거쳐, 우리는 결국 명확한 3계층 아키텍처 방안을 채택하여 Reasonix를 시스템에 성공적으로 통합했습니다. 이 방안은 현재의 문제를 해결했을 뿐만 아니라, 이후 유사한 CLI Provider 통합을 위한 재사용 가능한 패턴도 제공했습니다. 사실 많은 일들이 이렇습니다. 올바른 방법을 찾으면 뒤의 길이 훨씬 쉬워집니다.

## HagiCode 소개

이 글에서 공유하는 방안은 [HagiCode](https://hagicode.com) 프로젝트에서의 실천 경험에서 왔습니다. HagiCode는 개발자에게 강력한 코드 생성, 리팩토링 및 최적화 능력을 제공하는 오픈 소스 AI 코드 도구 프로젝트입니다. 개발 과정에서 우리는 다양한 기술 도전 과제에 직면했고, Reasonix를 일등급 Agent Provider로 통합하는 것이 그중 하나였습니다. 이 글에서 공유하는 방안이 가치 있다고 느끼신다면, 우리의 엔지니어링 실천이 아주 나쁘지 않다는 것이고, HagiCode 그 자체도 주목할 가치가 있습니다.

## 핵심 내용

### 기술 아키텍처 설계

시스템은 관심사 분리를 위해 명확한 3계층 아키텍처를 채택했으며, 각 계층에는 명확한 책임 경계가 있습니다:

**HagiCode.Libs 계층**: 이 계층은 이미 완료되었으며, CLI provider 추상화와 구체적 구현을 제공합니다. `ICliProvider<ReasonixOptions>` 인터페이스를 정의하고, ACP 스트리밍 및 세션 관리를 처리하는 `ReasonixProvider`를 구현했습니다. 동시에 effort, budget, yolo, transcript 등 매개변수를 지원합니다. 이 계층의 책임은 안정적이고 재사용 가능한 하위 레벨 능력을 제공하는 것이며, 비즈니스 로직과는 관련이 없습니다. 마치 집의 기초와 같습니다. 보이지 않지만 매우 중요합니다.

**hagicode-core 계층**: 이것이 우리가 이번 통합의 초점입니다. 하위 레벨 추상화를 시스템의 통일된 인터페이스에 연결하는 책임을 담당합니다. 구체적인 작업은 `AIProviderType.ReasonixCli = 12` 열거형 값을 등록하고, Libs 계층을 연결하는 thin 어댑터로서 `ReasonixCliProvider`를 생성하고, 세션 상태와 실행 흐름을 처리하는 `ReasonixGrain`을 구현하며, Hero 시스템을 통합하여 매개변수 매핑 및 구성 관리를 수행하는 것입니다. 이 계층의 핵심은 각 구성요소를 조정하여 완전한 비즈니스 체인을 구축하는 것입니다. 마치 집의 내력 벽처럼 각 부분을 연결합니다.

**web 계층**: 사용자에게 표시하고 구성을 수집하는 책임을 담당합니다. 새 열거형 값을 지원하도록 OpenAPI 타입을 다시 생성해야 하고, Reasonix가 자신만의 아이콘과 표시 이름을 갖도록 시각적 타입 매핑을 구현하며, 사용자가 각 매개변수를 구성할 수 있도록 CLI 매개변수 구성 양식을 생성하고, 다국어 지원을 추가해야 합니다. 이 계층의 초점은 사용자 경험과 상호작용 설계입니다. 마치 집의 인테리어와 같습니다. 잘하느냐가 거주의 편안함에 직접적인 영향을 미칩니다.

이러한 분할 설계는 각 계층이 자신의 책임에 집중하게 하여 시스템의 복잡도를 낮추고, 유지보수와 확장에도 용이합니다. 사실 많은 경우 일을 분명히 구분하는 것이 더 간단합니다.

### 주요 기술 결정

구현 과정에서 우리는 최종 아키텍처와 사용자 경험에 중요한 영향을 미치는 몇 가지 주요 기술 결정을 내렸습니다.

**결정 1: 전용 Grain 사용**

우리는 공유 Grain을 재사용하려고 시도하는 대신 독립적인 `ReasonixGrain : IReasonixGrain, IExecutorStreamGrain`을 생성했습니다. 이 결정은 시스템 기존의 11개 provider의 확립된 패턴을 따랐습니다. 일부 코드 중복이 발생할 수 있지만, 전용 Grain은 우리가 Reasonix의 특성에 맞는 정밀한 제어를 할 수 있게 합니다. 예를 들어 고유한 세션 바인딩 메커니즘과 ACP 메시지 매핑과 같은 것들입니다. 또한 타입 구분자로서 역할을 하는 빈 응답 DTO `ReasonixResponse`를 정의했습니다. 실제 데이터를 포함하지 않지만 타입 시스템에서 중요한 역할을 합니다. 마치 모든 사람이 자신만의 방을 갖는 것과 같습니다. 비어 있더라도 자신만의 공간입니다.

**결정 2: 전용 Settings 클래스 생성 안 함**

연결 문자열이 필요한 일부 Provider와 달리 Reasonix의 모든 구성은 사용자가 런타임에 설정하므로 시작 시 검증이 필요하지 않습니다. 따라서 우리는 전용 Settings 클래스를 생성하지 않고, 모든 구성을 `AIProviderOptions.Providers[ReasonixCli].Settings` 사전에 저장했습니다. 이 패턴은 Qoder, Kiro, Kimi와 같은 다른 로컬 CLI Provider와 일치하여 코드 구조를 단순화하고 불필요한 추상화 계층을 피했습니다. 지원하는 설정 키는 다음을 포함합니다: `effort`, `budgetUsd`, `transcriptPath`, `enableYolo`, `arguments`, `startupTimeoutMs`, `reasoning`. 때로는 단순한 것이 더 낫습니다.

**결정 3: Provider 전략 건강 모니터링**

Reasonix는 사용자가 로컬에 설치하는 CLI이므로 설치되지 않았거나 시스템 PATH에 없을 수 있습니다. 이 경우 우리는 직접 오류를 보고하는 대신 우아하게 저하 처리해야 합니다. `Provider` 전략을 사용하여 `CommandUtil.TryResolveExecutablePath`로 CLI 사용 가능성을 확인했습니다. 확인이 실패하면 UI에 "사용 불가능"으로 표시되지만 시스템의 다른 부분에는 영향을 주지 않습니다. 이 설계는 시스템을 더 견고하게 만들고 사용자에게 명확한 피드백을 제공합니다. 아무도 작은 문제로 인해 전체 시스템이 멈추기를 원하지 않습니다.

**결정 4: 경제 시스템 분류**

HagiCode 시스템에서 다른 Provider는 다른 경제 시스템 분류를 가집니다. 우리는 Reasonix가 기본적으로 `'claude'` 경제 시스템 분류를 사용하도록 결정했습니다. Reasonix 자체가 anthropic 시리즈 모델과 호환되기 때문입니다. 현재는 Codex와 Copilot만 전용 경제 시스템 분류를 가지며, 다른 Provider는 기존 분류를 재사용합니다. 이렇게 하면 시스템의 단순성을 유지하면서 청구 및 비용 통계를 올바르게 처리할 수 있습니다. 재사용도 지혜입니다. 모든 것을 처음부터 할 필요는 없습니다.

**결정 5: 모델 호환성**

Reasonix는 `--model` 플래그를 통해 여러 모델, 특히 anthropic 시리즈를 지원합니다. 우리는 `secondary-professions.index.json`에 호환성 매핑을 추가하여 사용자가 Reasonix에서 이러한 모델을 선택할 수 있게 했습니다. 이 설계는 Reasonix의 능력을 존중하면서 시스템의 일관성을 유지합니다. 사용자는 하위 레벨 차이를 이해하지 않고도 다양한 모델을 원활하게 사용할 수 있습니다. 사용자도 쉽지 않습니다. 그들을 조금 더 단순하게 해주세요.

### 백엔드 구현 세부사항

백엔드 구현은 몇 가지 주요 부분으로 나뉘며, 각 부분에는 독특한 기술 포인트가 있습니다.

**열거형 및 타입 등록**

먼저 시스템에 새 Provider 타입을 등록해야 합니다:

```csharp
// AIProviderType.cs
public enum AIProviderType
{
    // ... 다른 provider
    ReasonixCli = 12,
}

// AIProviderTypeExtensions.cs
private static readonly Dictionary<string, AIProviderType> _typeMap = new()
{
    // ... 다른 매핑
    ["Reasonix"] = AIProviderType.ReasonixCli,
    ["reasonix"] = AIProviderType.ReasonixCli,
    ["reasonix-cli"] = AIProviderType.ReasonixCli,
    ["ReasonixCli"] = AIProviderType.ReasonixCli,
};
```

이 열거형 값은 다른 동시 변경 사항과 조정하여 충돌을 피해야 합니다. 우리는 다음 사용 가능한 번호인 12를 선택했습니다. 마치 줄 서기와 같습니다. 선후 순서가 있어야 합니다.

**Thin Adapter 구현**

`ReasonixCliProvider`는 Libs 계층과 시스템 통합 인터페이스를 연결하는 핵심 구성요소입니다:

```csharp
public sealed class ReasonixCliProvider : IAIProvider, IVersionedAIProvider, IAsyncDisposable
{
    private static readonly IReadOnlyList<string> SupportedSettingKeys =
    [
        "effort",
        "budgetUsd",
        "transcriptPath",
        "enableYolo",
        "arguments",
        "startupTimeoutMs",
        "reasoning"
    ];

    private readonly ICliProvider<ReasonixOptions> _provider;
    private readonly ConcurrentDictionary<string, string> _sessionBindings = new(StringComparer.Ordinal);

    public async IAsyncEnumerable<AIStreamingChunk> StreamCoreAsync(
        AIRequest request,
        string? sessionId = null,
        [EnumeratorCancellation] CancellationToken cancellationToken = default)
    {
        var options = BuildOptions(request, sessionId);
        await foreach (var message in _provider.StreamAsync(options, cancellationToken))
        {
            yield return MapToStreamingChunk(message);
        }
    }

    private ReasonixOptions BuildOptions(AIRequest request, string? sessionId)
    {
        return new ReasonixOptions
        {
            ExecutablePath = GetExecutablePath(),
            WorkingDirectory = GetWorkingDirectory(),
            Model = _config.Model,
            Effort = _config.Settings.GetValueOrDefault("effort", "medium"),
            Budget = _config.Settings.GetValueOrDefault("budgetUsd", 10.0),
            Yolo = _config.Settings.GetValueOrDefault("enableYolo", false),
            TranscriptPath = _config.Settings.GetValueOrDefault("transcriptPath"),
            Arguments = _config.Settings.GetValueOrDefault("arguments", ""),
            StartupTimeout = GetStartupTimeout(),
            EnvironmentVariables = _environmentVariables,
            CessionId = sessionId ?? GetCessionId()
        };
    }
}
```

이 어댑터의 주요 책임은 다음과 같습니다:
1. 구성 매개변수를 검증하고 지원하지 않는 설정 키를 거부합니다
2. 세션 바인딩 관계를 유지하고 세션 복구를 지원합니다
3. ACP 메시지를 시스템의 통합 형식으로 매핑합니다
4. `ProviderErrorAutoRetryCoordinator`를 사용하여 자동 재시도를 구현합니다

마치 통역관과 같습니다. 한 언어를 다른 언어로 번역하면서 의미가 정확해야 합니다.

**Orleans Grain 구현**

`ReasonixGrain`은 세션 상태와 실행 흐름을 처리하는 책임을 담당합니다:

```csharp
public class ReasonixGrain : Grain, IReasonixGrain, IExecutorStreamGrain
{
    private readonly Dictionary<string, ExecutorToolLifecycleStatus> _toolLifecycleState =
        new(StringComparer.Ordinal);

    public IAsyncEnumerable<ReasonixResponse> ExecuteCommandStreamAsync(
        string command,
        string? heroId = null,
        CancellationToken token = default,
        string? executionMessageId = null,
        string? systemMessage = null,
        Dictionary<string, string>? requestSettings = null)
    {
        var request = BuildRequest(command, isEdit: false, heroId, executionMessageId, systemMessage, requestSettings);
        return SendAsync(request, heroId, token);
    }

    private async IAsyncEnumerable<ReasonixResponse> SendAsync(
        AIRequest request,
        string? heroId,
        [EnumeratorCancellation] CancellationToken token)
    {
        _cancellationTokenSource = new CancellationTokenSource();
        var linkedToken = CancellationTokenSource.CreateLinkedTokenSource(token, _cancellationTokenSource.Token);

        var provider = await ResolveReasonixProviderAsync(heroId);

        await foreach (var chunk in provider.StreamAsync(request, linkedToken.Token))
        {
            var response = BuildChunkResponse(chunk);
            yield return response;
        }
    }

    private async Task<IAIProvider> ResolveReasonixProviderAsync(string? heroId)
    {
        // Hero 인식 구성 대체 로직
        var config = await HeroProviderResolver.ResolveAsync(AIProviderType.ReasonixCli, heroId);
        return _aiProviderFactory.CreateProvider(AIProviderType.ReasonixCli, config);
    }
}
```

Grain의 핵심 기능은 다음을 포함합니다:
1. `[PersistentState("reasonix-interop")]`를 사용하여 세션 상태를 유지합니다
2. Hero 인식 구성 대체 로직을 구현합니다
3. 도구 라이프사이클 상태를 추적합니다
4. 취소 토큰 체인을 지원하여 실행이 적시에 중단되도록 합니다

마치 집사처럼 일을 잘 정리합니다.

**Hero 시스템 통합**

Hero 시스템은 HagiCode의 직업 구성 시스템입니다. 우리는 이 시스템에 Reasonix를 통합해야 합니다:

```csharp
// HeroAppService.cs
// Family 추론
AIProviderType.ReasonixCli => "reasonix"

// 관리되는 CLI 매개변수
ManagedCliParameterKeysByProvider[AIProviderType.ReasonixCli] = 
    ["binary", "effort", "budgetUsd", "transcriptPath", "enableYolo", "arguments", "startupTimeoutMs"];

// 관리되는 모델 매개변수
ManagedModelParameterKeysByProvider[AIProviderType.ReasonixCli] = 
    ["model", "reasoning"];
```

직업 디렉토리 구성 파일 `main-professions.yaml`에서:

```yaml
- Id: "profession-reasonix"
  Name: "Reasonix"
  Family: "reasonix"
  Summary: "hero.professionCopy.primary.reasonix.summary"
  Icon: "executor-avatar:Reasonix"
  SourceLabel: "hero.professionCopy.sources.aiProvidersReasonixCli"
  ProviderType: "ReasonixCli"
  SortOrder: 130
  DefaultEnabled: true
  DefaultParameters:
    binary: "reasonix"
    effort: "medium"
    enableYolo: "false"
    startupTimeoutMs: "15000"
```

이렇게 구성하면 사용자는 Hero 구성 인터페이스에서 Reasonix 옵션을 보고 개인화된 설정을 할 수 있습니다. 마치 사람에게 호적을 등록하는 것과 같습니다. 신분이 있어야 사회에서 정상적으로 생활할 수 있습니다.

### 프론트엔드 구현 세부사항

프론트엔드 구현은 주로 사용자 상호작용과 표시를 담당하며, 역시 몇 가지 주요 부분으로 나뉩니다.

**타입 생성 및 시각적 매핑**

먼저 OpenAPI 타입을 다시 생성해야 합니다:


```typescript

      return 'Reasonix';

```

이렇게 하면 Reasonix가 자신만의 시각적 타입을 가지게 되고, 해당 아이콘과 스타일을 표시할 수 있습니다. 마치 모든 사람이 자신만의 신분증 사진을 가지는 것과 같습니다.

**구성 양식 구현**

**다국어 지원**

### 모범 사례 및 주의사항

구현 과정에서 우리는 몇 가지 모범 사례와 주의해야 할 점을 정리했습니다.

**매개변수 검증**

ReasonixCliProvider는 구성 매개변수를 엄격히 검증하고 지원하지 않는 설정 키를 거부해야 합니다:

```csharp
public void ValidateConfigurationOverrides(Dictionary<string, string?> overrides)
{
    foreach (var key in overrides.Keys)
    {
        if (!SupportedSettingKeys.Contains(key))
        {
            throw new HeroProviderConfigurationException(
                $"Unsupported setting key '{key}' for Reasonix provider");
        }
    }
}
```

이렇게 하면 사용자가 잘못된 매개변수를 구성하는 것을 방지하고 런타임 오류를 피할 수 있습니다. 마치 골키퍼처럼 들어오지 말아야 할 것이 들어오지 못하게 합니다.

**세션 바인딩 관리**

`ConcurrentDictionary`를 사용하여 세션 바인딩을 관리하고 세션 복구를 지원합니다:

```csharp
_sessionBindings[cessionId] = sessionId;

// 후속 요청에서 기존 세션 바인딩
if (_sessionBindings.TryGetValue(cessionId, out var boundSessionId))
{
    options.SessionId = boundSessionId;
}
```

이 설계는 사용자가 중단된 후 이전 세션을 복구할 수 있게 하여 더 나은 경험을 제공합니다. 마치 이야기를 기억했다가 다음에 계속 이어가는 것과 같습니다.

**우아한 저하 처리**

프론트엔드는 CLI 사용 가능성을 확인하고 친절한 프롬프트를 제공해야 합니다:

```typescript
const reasonixAvailable = await healthApi.checkCliAvailable('reasonix');
if (!reasonixAvailable) {
  showMessage('Reasonix CLI가 설치되지 않았습니다. 시스템 PATH에 구성해주세요.');
}
```

사용자가 납득할 수 없는 오류를 만나지 않도록 미리 확인하고 명확한 안내를 제공하세요. 아무도 명확하지 않은 이유로 막히기를 원하지 않습니다.

**테스트 커버리지**

완벽한 테스트는 품질 보증의 핵심입니다:

```csharp
[Fact]
public async Task ExecuteCommandStreamAsync_WithValidCommand_StreamsReasonixResponse()
{
    // Arrange
    var grain = _grainFactory.GetGrain<IReasonixGrain>("test-cession");
    var responses = new List<ReasonixResponse>();

    // Act
    await foreach (var response in grain.ExecuteCommandStreamAsync("help"))
    {
        responses.Add(response);
    }

    // Assert
    responses.Should().NotBeEmpty();
    responses.All(r => r.Kind == ExecutorResponseKind.Content).Should().BeTrue();
}
```

이러한 단위 테스트는 핵심 기능이 정상적으로 작동하는지 확인하고 회귀 오류를 방지할 수 있습니다. 마치 시험 전에 연습 문제를 푸는 것과 같습니다. 정말로 마스터했는지 확인해야 합니다.

## 요약

이번 Reasonix 통합 실천을 통해 우리는 로컬 CLI 도구를 시스템의 일등급 Agent Provider로 성공적으로 승격시켰습니다. 전체 과정에서 우리는 확립된 아키텍처 패턴을 따르고 합리적인 기술 결정을 내려, 결국 기능이 완전하고 사용자 경험이 양호한 통합 방안을 구현했습니다.

이 방안의 핵심 가치는 다음과 같습니다:
1. 명확한 3계층 아키텍처는 관심사를 분리하여 복잡도를 낮췄습니다
2. 전용 Grain과 thin 어댑터 설계는 유연성을 유지했습니다
3. 우아한 저하와 건강 모니터링은 사용자 경험을 향상시켰습니다
4. 완벽한 매개변수 검증과 세션 관리는 신뢰성을 보장했습니다

다른 유사한 CLI Provider 통합에 대해 이 방안은 재사용 가능한 패턴을 제공합니다. 우리는 이번 실천이 다른 개발자에게 도움이 되기를 바라며, HagiCode 프로젝트에서 경험을 교환하셔도 좋습니다.

사실 많은 일들이 이렇습니다. 시작에는 어려워 보이지만 방법만 맞으면 단계별로 나아가면 결국 해결할 수 있습니다. 산을 오르는 것과 같습니다. 산꼭대기가 멀어 보이지만 포기하지 않으면 결국 올라갈 수 있습니다.
