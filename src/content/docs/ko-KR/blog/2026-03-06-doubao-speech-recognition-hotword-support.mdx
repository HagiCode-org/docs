## 더우바오 음성 인식 핵심어 기능 구현 가이드

> 이 문서에서는 HagiCode 프로젝트에서 더우바오 음성 인식의 핵심어 지원 기능을 구현하는 방법을 자세히 설명합니다. 사용자 정의 핵심어와 플랫폼 핵심어 표 두 가지 방식을 통해 특정 분야 어휘의 인식 정확도를 크게 향상시킬 수 있습니다.

## 배경

음성 인식 기술이 이렇게 발전했지만, 사실 하나의 문제가 개발자들을 계속 괴롭혀왔습니다. 일반 음성 인식 모델은 일상적인 언어를 커버할 수 있지만, 전문 용어, 제품 이름, 사람 이름 같은 단어에 대해서는 인식 정확도가 조금 부족합니다. 생각해보세요, 의료 분야의 음성 비서는 "고혈압", "당뇨병", "관상동맥 질환"을 정확히 인식해야 하고, 법률 시스템은 "사유", "답변", "입증 책임"을 정확히 포착해야 합니다. 이런 시나리오에서 일반 모델의 성능은, 뭐, 나름대로 최선을 다하고 있습니다.

HagiCode 프로젝트에서도 우리는 같은 도전에 직면했습니다. 다기능 AI 코드 어시스턴트로서 HagiCode는 다양한 기술 용어의 음성 인식 시나리오를 처리해야 합니다. 그러나 더우바오 음성 인식 API는 기본적으로 우리의 전문 용어 정확도 요구사항을 완전히 충족시키지 못했습니다. 사실 더우바오가 부족한 게 아니라, 각 분야마다 자신만의 용어 체계가 있을 뿐입니다. 조사와 기술 탐색을 거쳐 더우바오 음성 인식 API가 실제로 핵심어 지원 기능을 제공한다는 것을 발견했고, 간단한 설정만으로 특정 단어의 인식 정확도를 크게 향상시킬 수 있습니다. 이것은 마치 당신이 주의해야 할 단어를 알려주면, 그 단어들을 더 주의 깊게 듣는 것과 같습니다.

이 문서에서 공유하는 것은 HagiCode 프로젝트에서 더우바오 음성 인식 핵심어 기능을 구현하는 완전한 솔루션입니다. 사용자 정의 핵심어와 플랫폼 핵심어 표 두 가지 모드를 모두 사용할 수 있고, 조합해서 사용할 수도 있습니다. 이 솔루션을 통해 개발자는 비즈니스 시나리오에 따라 핵심어를 유연하게 구성하여 음성 인식 시스템이 전문적이고 희귀하지만 매우 중요한 어휘를 "인식"하도록 할 수 있습니다.

## 핵심 구현

### 두 가지 핵심어 모드 분석

더우바오 음성 인식 API는 두 가지 핵심어 구성 방식을 제공하며, 각 방식은 고유한 응용 시나리오와 장점이 있습니다.

**사용자 정의 핵심어 모드**는 `corpus.context` 필드를 통해 핵심어 텍스트를 직접 전달할 수 있습니다. 이 방식은 소수의 핵심어를 빠르게 구성해야 하는 시나리오에 매우 적합하며, 예를 들어 특정 제품 이름이나 사람 이름을 일시적으로 인식해야 할 때 유용합니다. HagiCode 구현에서는 사용자가 입력한 여러 줄의 핵심어 텍스트를 문자열 목록으로 구문 분석한 다음 더우바오 API의 요구 형식에 따라 `context_data` 배열로 포맷합니다. 솔직히 말하면, 이 방식은 매우 직관적입니다. 상대방에게 "이 단어들을 주의해달라"고 말하는 것과 같습니다.

**플랫폼 핵심어 표 모드**는 `corpus.boosting_table_id` 필드를 통해 더우바오 자체 학습 플랫폼에서 미리 구성된 핵심어 표를 참조합니다. 이 방식은 대량의 핵심어를 관리해야 하는 시나리오에 적합하며, 더우바오 자체 학습 플랫폼에서 핵심어 표를 생성 및 유지 관리한 다음 ID를 통해 참조할 수 있습니다. HagiCode처럼 전문 용어를 지속적으로 업데이트하고 유지 관리해야 하는 프로젝트의 경우 이 모드는 더 나은 관리 용이성을 제공합니다. 아무래도 핵심어가 많아지면 통합 관리할 곳이 있는 것이 매번 수동으로 입력하는 것보다 낫습니다.

흥미롭게도 이 두 모드는 조합해서 사용할 수도 있습니다. 더우바오 API는 동일한 요청에서 사용자 정의 핵심어와 플랫폼 핵심어 표 ID를 동시에 포함할 수 있으며, `combine_mode` 매개변수를 통해 조합 전략을 제어합니다. 이러한 유연성 덕분에 HagiCode는 다양한 복잡한 전문 용어 인식 요구사항에 대처할 수 있습니다. 이것도 꽤 좋습니다. 때로는 여러 방식을 조합하면 효과가 더 좋을 수 있습니다.

### 프론트엔드 타입 정의 및 검증

HagiCode의 프론트엔드 구현에서는 완전한 핵심어 구성 타입과 검증 논리를 정의했습니다. 먼저 타입 정의 부분부터 살펴보겠습니다:

```typescript
export interface HotwordConfig {
  contextText: string;           // 여러 줄 핵심어 텍스트
  boostingTableId: string;      // 더우바오 플랫폼 핵심어 표 ID
  combineMode: boolean;          // 조합 사용 여부
}
```

이 간단한 인터페이스는 핵심어 기능의 모든 구성 항목을 포함합니다. 그중 `contextText`는 사용자가 가장 직관적으로 느끼는 부분입니다. 우리는 사용자가 한 줄에 하나의 핵심어 구문을 입력하도록 허용하며, 이 방식은 매우 직관적입니다. 아무래도 사용자에게 한 줄에 한 단어씩 입력하게 하는 것이 복잡한 구성 규칙을 이해하게 하는 것보다 낫습니다.

다음은 검증 함수 구현입니다. 더우바오 API의 제한사항을 고려하여 엄격한 검증 규칙을制定了습니다: 핵심어 텍스트는 최대 100줄, 각 줄은 최대 50자, 총 최대 5000자입니다. boosting_table_id는 최대 200자이며 영문자, 숫자, 밑줄, 하이픈만 허용됩니다. 이러한 제한사항은 우리가 임의로 상상한 것이 아니라 더우바오 공식 문서의 실제 요구사항을 기반으로 합니다. 아무래도 API의 제한은 API의 제한이고, 우리도 따를 수밖에 없습니다.

```typescript
export function validateContextText(contextText: string): HotwordValidationResult {
  if (!contextText || contextText.trim().length === 0) {
    return { isValid: true, errors: [] };
  }

  const lines = contextText.split('\n').filter(line => line.trim().length > 0);
  const errors: string[] = [];

  if (lines.length > 100) {
    errors.push(`핵심어 줄 수는 100줄을 초과할 수 없습니다. 현재 ${lines.length}줄`);
  }

  const totalChars = contextText.length;
  if (totalChars > 5000) {
    errors.push(`핵심어 총 문자 수는 5000을 초과할 수 없습니다. 현재 ${totalChars}`);
  }

  for (let i = 0; i < lines.length; i++) {
    if (lines[i].length > 50) {
      errors.push(`${i + 1}번째 줄 핵심어가 50자 제한을 초과했습니다`);
    }
  }

  return { isValid: errors.length === 0, errors };
}

export function validateBoostingTableId(boostingTableId: string): HotwordValidationResult {
  if (!boostingTableId || boostingTableId.trim().length === 0) {
    return { isValid: true, errors: [] };
  }

  const errors: string[] = [];

  if (boostingTableId.length > 200) {
    errors.push(`boosting_table_id는 200자를 초과할 수 없습니다. 현재 ${boostingTableId.length}자`);
  }

  if (!/^[a-zA-Z0-9_-]+$/.test(boostingTableId)) {
    errors.push('boosting_table_id는 영문자, 숫자, 밑줄, 하이픈만 포함할 수 있습니다');
  }

  return { isValid: errors.length === 0, errors };
}
```

이러한 검증 함수는 사용자가 핵심어를 구성할 때 즉시 실행되어 문제가 가장 초기 단계에서 발견되도록 합니다. 사용자 경험에 있어 이러한 즉시 피드백은 매우 중요합니다. 아무래도 사용자가 입력할 때 어디가 틀렸는지 아는 것이 제출한 후에 발견하는 것보다 낫습니다.

### 프론트엔드 구성 영구 저장

HagiCode의 프론트엔드 구현에서는 브라우저의 localStorage를 사용하여 핵심어 구성을 저장하기로 선택했습니다. 이 설계 결정 뒤에는 몇 가지 고려사항이 있습니다: 첫째, 핵심어 구성은 매우 개인화된 설정이며 다른 사용자는 다른 전문 분야 요구사항이 있을 수 있습니다. 둘째, 이 방식은 백엔드 구현을 단순화하며 추가 데이터베이스 테이블과 API 인터페이스가 필요하지 않습니다. 마지막으로, 사용자가 브라우저에서 한 번 구성하면 이후 사용 시 자동으로 로드되어 매우 편리합니다. 솔직히 말해서, 그냥 편하려고 그런 것입니다.

```typescript
const HOTWORD_STORAGE_KEYS = {
  contextText: 'hotword-context-text',
  boostingTableId: 'hotword-boosting-table-id',
  combineMode: 'hotword-combine-mode',
} as const;

export const DEFAULT_HOTWORD_CONFIG: HotwordConfig = {
  contextText: '',
  boostingTableId: '',
  combineMode: false,
};

// 핵심어 구성 불러오기
export function loadHotwordConfig(): HotwordConfig {
  const contextText = localStorage.getItem(HOTWORD_STORAGE_KEYS.contextText) || '';
  const boostingTableId = localStorage.getItem(HOTWORD_STORAGE_KEYS.boostingTableId) || '';
  const combineMode = localStorage.getItem(HOTWORD_STORAGE_KEYS.combineMode) === 'true';

  return { contextText, boostingTableId, combineMode };
}

// 핵심어 구성 저장
export function saveHotwordConfig(config: HotwordConfig): void {
  localStorage.setItem(HOTWORD_STORAGE_KEYS.contextText, config.contextText);
  localStorage.setItem(HOTWORD_STORAGE_KEYS.boostingTableId, config.boostingTableId);
  localStorage.setItem(HOTWORD_STORAGE_KEYS.combineMode, String(config.combineMode));
}
```

이 코드의 논리는 매우 간단하고 명확합니다. 구성을 불러올 때 localStorage에서 읽고, 저장할 때 localStorage에 씁니다. 또한 기본 구성을 제공하여 구성이 전혀 없을 때도 시스템이 정상적으로 작동하도록 합니다. 아무래도 기본값은 있어야 합니다.

### 백엔드 SDK 구성 확장

HagiCode의 백엔드 구현에서는 SDK 구성 클래스에 핵심어 관련 속성을 추가해야 합니다. C#의 언어 특성과 사용 습관을 고려하여 `List<string>`을 사용하여 사용자 정의 핵심어 컨텍스트를 저장합니다:

```csharp
public class DoubaoVoiceConfig
{
    /// <summary>
    /// 애플리케이션 ID
    /// </summary>
    public string AppId { get; set; } = string.Empty;

    /// <summary>
    /// 액세스 토큰
    /// </summary>
    public string AccessToken { get; set; } = string.Empty;

    /// <summary>
    /// 서비스 URL
    /// </summary>
    public string ServiceUrl { get; set; } = string.Empty;

    /// <summary>
    /// 사용자 정의 핵심어 컨텍스트 목록
    /// </summary>
    public List<string>? HotwordContexts { get; set; }

    /// <summary>
    /// 더우바오 플랫폼 핵심어 표 ID
    /// </summary>
    public string? BoostingTableId { get; set; }
}
```

이 구성 클래스의 설계는 HagiCode의 일관된 간결한 스타일을 따릅니다. `HotwordContexts`는 nullable 목록 타입이고 `BoostingTableId`는 nullable 문자열이므로 핵심어 구성이 전혀 없을 때 이러한 속성은 요청에 영향을 주지 않습니다. 아무래도 사용하지 않을 때는 존재하지 않아야 깔끔합니다.

### Payload 구성 논리

Payload 구성은 전체 핵심어 기능의 핵심입니다. 핵심어 구성이 있으면 더우바오 API의 요구에 따라 올바른 JSON 구조로 포맷해야 합니다. 이 과정은 SDK가 요청을 보내기 전에 발생합니다:

```csharp
private void AddCorpusToRequest(Dictionary<string, object> request)
{
    var corpus = new Dictionary<string, object>();

    // 사용자 정의 핵심어 추가
    if (Config.HotwordContexts != null && Config.HotwordContexts.Count > 0)
    {
        corpus["context"] = new Dictionary<string, object>
        {
            ["context_type"] = "dialog_ctx",
            ["context_data"] = Config.HotwordContexts
                .Select(text => new Dictionary<string, object> { ["text"] = text })
                .ToList()
        };
    }

    // 플랫폼 핵심어 표 ID 추가
    if (!string.IsNullOrEmpty(Config.BoostingTableId))
    {
        corpus["boosting_table_id"] = Config.BoostingTableId;
    }

    // corpus가 비어 있을 때만 요청에 추가
    if (corpus.Count > 0)
    {
        request["corpus"] = corpus;
    }
}
```

이 코드는 구성에 따라 corpus 필드를 동적으로 구성하는 방법을 보여줍니다. 핵심은 실제로 핵심어 구성이 있을 때만 corpus 필드를 추가한다는 것입니다. 이 설계는 하위 호환성을 보장합니다. 핵심어를 구성하지 않으면 요청 구조가 이전과 완전히 동일합니다. 아무래도 호환성이 중요합니다. 기능을 추가했다고 이전 논리를 망칠 수는 없습니다.

### WebSocket 매개변수 전달

프론트엔드와 백엔드 사이에서 핵심어 매개변수는 WebSocket 제어 메시지를 통해 전달됩니다. HagiCode의 설계는 프론트엔드가 녹음을 시작할 때 localStorage에서 핵심어 구성을 불러온 다음 WebSocket 메시지를 통해 백엔드로 전송하는 것입니다.

```typescript
const controlMessage = {
  type: 'control',
  payload: {
    command: 'StartRecognition',
    contextText: '고혈압\n당뇨병\n관상동맥 질환',
    boosting_table_id: 'medical_table',
    combineMode: false
  }
};
```

여기 주의해야 할 세부 사항이 있습니다: 프론트엔드는 여러 줄 텍스트(줄 바꿈 문자로 구분)를 전달하고 백엔드에서 구문 분석해야 합니다. 백엔드 WebSocket 핸들러는 이러한 매개변수를 구문 분석하여 SDK에 전달합니다:

```csharp
private async Task HandleControlMessageAsync(
    string connectionId,
    DoubaoSession session,
    ControlMessage message)
{
    if (message.Payload is SessionControlRequest controlRequest)
    {
        // 핵심어 매개변수 구문 분석
        string? contextText = controlRequest.ContextText;
        string? boostingTableId = controlRequest.BoostingTableId;
        bool? combineMode = controlRequest.CombineMode;

        // 여러 줄 텍스트를 핵심어 목록으로 구문 분석
        if (!string.IsNullOrEmpty(contextText))
        {
            var hotwords = contextText
                .Split('\n', StringSplitOptions.RemoveEmptyEntries)
                .Select(s => s.Trim())
                .Where(s => s.Length > 0)
                .ToList();

            session.HotwordContexts = hotwords;
        }

        session.BoostingTableId = boostingTableId;
    }
}
```

이러한 설계를 통해 핵심어 구성이 프론트엔드에서 백엔드로 전달되는 것이 명확하고 효율적입니다. 사실 특별한 것은 없고 그냥 한 층 한 층 전달하는 것입니다.

## 실습 가이드

### 사용자 정의 핵심어 구성

실제 사용에서 사용자 정의 핵심어 구성은 매우 간단합니다. HagiCode의 음성 인식 설정 페이지를 열고 "핵심어 구성" 영역을 찾으세요. "사용자 정의 핵심어 텍스트" 입력 상자에 한 줄에 하나씩 핵심어 구문을 입력합니다.

예를 들어 의료 관련 애플리케이션을 개발 중이라면 다음과 같이 구성할 수 있습니다:

```
고혈압
당뇨병
관상동맥 질환
협심증
심근 경색
심부전
```

구성을 저장하면 음성 인식을 시작할 때마다 이러한 핵심어가 자동으로 더우바오 API로 전달됩니다. 실제 테스트에 따르면 핵심어를 구성한 후 관련 전문 용어의 인식 정확도가 현저히 향상되었습니다. 솔직히 말해서, 효과는 있습니다. 적어도 이전보다 훨씬 낫습니다.

### 플랫폼 핵심어 표 구성

대량의 핵심어를 관리해야 하거나 핵심어를 자주 업데이트해야 한다면 플랫폼 핵심어 표 모드가 더 적합합니다. 먼저 더우바오 자체 학습 플랫폼에서 핵심어 표를 생성하고 생성된 boosting_table_id를 가져온 다음 HagiCode 설정 페이지에서 이 ID를 입력해야 합니다.

더우바오 자체 학습 플랫폼은 핵심어의 대량 가져오기, 분류 관리 등의 기능을 제공하며 대량의 전문 용어를 관리해야 하는 팀에게 매우 실용적입니다. 플랫폼을 통해 핵심어를 관리하면 핵심어의 중앙 유지 관리와 통합 업데이트를 실현할 수 있습니다. 아무래도 핵심어가 많아지면 통합 관리할 곳이 있는 것이 매번 수동으로 입력하는 것보다 낫습니다.

### 조합 모드 사용

일부 복잡한 시나리오에서는 사용자 정의 핵심어와 플랫폼 핵심어 표를 동시에 사용해야 할 수 있습니다. 이 경우 HagiCode에서 두 가지 핵심어를 모두 구성하고 "조합 모드" 스위치를 켜기만 하면 됩니다.

조합 모드에서 더우바오 API는 두 가지 핵심어 소스를 동시에 고려하며 인식 정확도는 일반적으로 어느 하나를 단독으로 사용할 때보다 높습니다. 그러나 조합 모드는 요청의 복잡도를 증가시키므로 실제 테스트 후 사용 여부를 결정하는 것이 좋습니다. 아무래도 복잡도가 증가하면 정말 가치가 있는지 실제 효과를 봐야 합니다.

### 코드 통합 예제

HagiCode 프로젝트에 핵심어 기능을 통합하는 것은 매우 간단합니다. 다음은 일반적으로 사용되는 코드 조각입니다:

```typescript
import {
  loadHotwordConfig,
  saveHotwordConfig,
  validateHotwordConfig,
  parseContextText,
  getEffectiveHotwordMode,
  type HotwordConfig
} from '@/types/hotword';

// 구성 불러오기 및 검증
const config = loadHotwordConfig();
const validation = validateHotwordConfig(config);

if (!validation.isValid) {
  console.error('핵심어 구성 검증 실패:', validation.errors);
  return;
}

// 핵심어 텍스트 구문 분석
const hotwords = parseContextText(config.contextText);
console.log('구문 분석된 핵심어:', hotwords);

// 유효한 핵심어 모드 가져오기
const mode = getEffectiveHotwordMode(config);
console.log('현재 핵심어 모드:', mode);
```

백엔드 사용도 간단합니다:

```csharp
var config = new DoubaoVoiceConfig
{
    AppId = "your_app_id",
    AccessToken = "your_access_token",
    ServiceUrl = "wss://openspeech.bytedance.com/api/v3/sauc/bigmodel_async",

    // 사용자 정의 핵심어 구성

    {
        "고혈압",
        "당뇨병",
        "관상동맥 질환"

var client = new DoubaoVoiceClient(config, logger);
await client.ConnectAsync();
await client.SendFullClientRequest();
```

## 주의사항

핵심어 기능을 구현하고 사용할 때 몇 가지에 특히 주의해야 합니다.

첫째, 문자 제한입니다. 더우바오 API는 핵심어에 대해 줄 수, 각 줄의 문자 수, 총 문자 수 등 엄격한 제한이 있습니다. 제한을 초과하면 API가 오류를 반환합니다. HagiCode의 프론트엔드 구현에서는 사용자 입력 단계에서 검증 함수를 통해 검사하여 잘못된 구성이 백엔드로 전송되는 것을 방지합니다. 아무래도 문제를 조기에 발견하는 것이 API 오류를 기다리는 것보다 낫습니다.

둘째, boosting_table_id의 형식입니다. 이 필드는 영문자, 숫자, 밑줄, 하이픈만 허용하며 공백이나 다른 특수 문자는 포함할 수 없습니다. 더우바오 자체 학습 플랫폼에서 핵심어 표를 생성할 때 명명 규칙에 주의해야 합니다. 사실 이것도 어쩔 수 없습니다. API는 형식 요구사항이 항상 엄격합니다.

셋째, 하위 호환성입니다. 핵심어 매개변수는 완전히 선택 사항이며 핵심어를 구성하지 않으면 시스템 작동 방식이 이전과 완전히 동일합니다. 이 설계는 기존 사용자가 영향을 받지 않도록 하고 점진적인 마이그레이션과 업그레이드를 용이하게 합니다. 아무래도 기능을 추가했다고 이전 논리를 망칠 수는 없습니다.

마지막으로 오류 처리입니다. 핵심어 구성이 잘못되면 더우바오 API는 해당 오류 정보를 반환합니다. HagiCode 구현은 개발자가 문제를 해결할 수 있도록 자세한 로그를 기록합니다. 동시에 프론트엔드도 인터페이스에서 검증 오류를 표시하여 사용자가 구성을 수정하도록 돕습니다. 오류 처리를 잘하면 사용자 경험도 자연스럽게 좋아집니다.
