## ImgBin CLI 도구 설계: HagiCode 이미지 자산 관리 솔루션

> 이 글에서는 자동화 실행 가능한 이미지 자산 파이프라인을 처음부터 구축하는 방법을 소개하며, CLI 도구 설계, Provider Adapter 아키텍처, 메타데이터 관리 전략을 다룹니다.

## 배경

사실 이미지 자산 관리라는 일이 이렇게 오랫동안 우리를 고민하게 할 줄은 몰랐습니다.

HagiCode 프로젝트 개발 과정에서 우리는 겉보기에는 간단해 보이지만 실제로는 매우 까다로운 문제에 직면했습니다. 이미지 자산의 생성과 관리입니다. 말하자면 사춘기의 그 일들과 같습니다. 표면적으로는 평온하지만, 속으로는 출렁이는 것입니다.

프로젝트 문서와 마케팅 소재가 많아지면서 많은 삽화가 필요해졌습니다. 이 삽화들은 AI로 생성해야 하는 것, 기존 소재 라이브러리에서 선택해야 하는 것, 기존 이미지를 AI로 인식하고 자동 태깅해야 하는 것 등이 있습니다. 문제는 이러한 작업이 오랫동안 산발적인 스크립트와 수동 작업으로 완료되어 왔다는 것입니다. 이미지를 생성할 때마다 수동으로 스크립트를 실행하고, 메타데이터를 수동으로 정리하고, 썸네일을 수동으로 생성해야 했습니다. 그것도 그렇다 치지만, 이 산발적인 것들이 여기저기 흩어져 있어 찾을 때 찾을 수 없고, 쓸 때 쓸 수 없다는 것이 핵심입니다.

구체적인 문제점은 다음과 같습니다:

1. **통합된 진입점 부족**: 이미지 생성 로직이 다양한 스크립트에 분산되어 있어 일괄 실행이 불가능합니다
2. **메타데이터 누락**: 생성된 이미지에 통일된 metadata.json이 없어 검색 및 추적이 불가능합니다
3. **높은 수동 정리 비용**: 이미지의 제목, 태그 등을 모두 수동으로 일일이 정리해야 하며 효율이 낮습니다
4. **자동화 불가**: CI/CD 프로세스에서 자동으로 삽화를 생성하고 싶다면? 방법이 없습니다

그냥 무시하자고도 생각했지만,毕竟 프로젝트는 해야 하니까요. 피할 수 없다면 해결 방법을 찾아야 합니다. 그래서 우리는 ImgBin을 "산발적인 스크립트"에서 자동화 실행 가능한 이미지 자산 파이프라인으로 업그레이드하기로 결정했습니다.毕竟 어떤 일은 도피도 해결책이 아닙니다.

## HagiCode에 대하여

이 글에서 공유하는 솔루션은 [HagiCode](https://hagicode.com) 프로젝트에서의 실천 경험에서 비롯되었습니다. HagiCode는 AI 코드 어시스턴트 프로젝트로, VSCode 확장, 백엔드 AI 서비스, 크로스 플랫폼 데스크톱 클라이언트 등 다양한 컴포넌트를 동시에 유지 관리합니다. 이러한 다국어, 다중 플랫폼의 복잡한 시나리오에서 이미지 자산의 규범적 관리는 개발 효율 향상의 핵심이 되었습니다.

말하자면 이것도 HagiCode 성장 과정의 작은烦恼입니다. 모든 프로젝트에는 이런 때가 있습니다. 보기에 대단하지 않은 작은 문제가 사람을 반나절 고생하게 하는 경우입니다.

HagiCode의 빌드 시스템은 TypeScript + Node.js 생태계를 채택하므로, ImgBin도 자연스럽게 동일한 기술 스택을 선택하여 전체 프로젝트의 기술 일관성을 보장했습니다.毕竟 익숙한 것을 사용하고 있어서 다른 것으로 바꾸는 것도 번거롭습니다.

## 핵심 설계

### 전체 아키텍처

ImgBin은 레이어드 아키텍처를 채택하여 CLI 명령, 애플리케이션 서비스, 서드파티 API 어댑터 및 인프라 레이어를 명확하게 분리합니다:

```
컴포넌트 계층 구조
├── CLI Entry (cli.ts)              전역 매개변수 파싱, 명령 라우팅
├── Commands (commands/*)           generate | batch | annotate | thumbnail
├── Application Services            job-runner | metadata | thumbnail | asset-writer
├── Provider Adapters               image-api-provider | vision-api-provider
└── Infrastructure Layer            config | logger | paths | schema
```

이러한 레이어링 설계의 장점은 각 레이어의 책임이 명확하고, 테스트 시 외부 의존성을 쉽게 mock할 수 있다는 것입니다. 즉, 각자 자기 일을 하고 서로 방해하지 않아서 문제가 발생하면 원인을 찾기 쉽다는 것입니다.

### 단일 자산 디렉토리 모델

ImgBin은 "하나의 자산, 하나의 디렉토리" 모델을 채택하며, 이미지를 생성할 때마다 다음과 같은 구조를 생성합니다:

```
library/
└── 2026-03/
    └── orange-dashboard/
        ├── original.png      # 원본 이미지
        ├── thumbnail.webp    # 512x512 썸네일
        └── metadata.json     # 구조화된 메타데이터
```

이 모델의 장점은 다음과 같습니다:

1. **자기 포함**: 각 자산의 모든 파일이 동일한 디렉토리에 있어 이동 및 백업이 편리합니다
2. **추적 가능**: metadata.json을 통해 이미지의 생성 시간, 사용된 prompt, 모델 등의 정보를 추적할 수 있습니다
3. **확장 가능**: 향후 더 많은 변형(예: 다양한 크기의 썸네일)을 추가하려면 동일한 디렉토리에 파일을 추가하면 됩니다

아름다운 것은 반드시 소유할 필요가 없습니다. 그녀가 여전히 아름답다면, 그 아름다움을 잘 바라보면 됩니다. 말이 좀 멀리 갔지만, 이치는 이것입니다. 이미지를 함께 모아두면 보기에도 좋고 찾기에도 편합니다.

### 메타데이터 계층 저장

metadata.json은 전체 시스템의 핵심으로, 계층 저장 전략을 채택하여 세 가지 유형의 필드를 구분합니다:

```json
{
  "schemaVersion": 2,
  "assetId": "orange-dashboard",
  "slug": "orange-dashboard",
  "title": "Orange Dashboard",
  "tags": ["dashboard", "hero", "orange"],

  "source": { "type": "generated" },

  "paths": {
    "assetDir": "library/2026-03/orange-dashboard",
    "original": "original.png",
    "thumbnail": "thumbnail.webp"
  },

  "generated": {
    "prompt": "orange dashboard for docs hero",
    "provider": "azure-openai-image-api",
    "model": "gpt-image-1.5"
  },

  "recognized": {
    "title": "Orange Dashboard",
    "tags": ["dashboard", "ui", "orange"],
    "description": "A modern orange dashboard with charts and metrics"
  },

  "status": {
    "generation": "succeeded",
    "recognition": "succeeded",
    "thumbnail": "succeeded"
  },

  "timestamps": {
    "createdAt": "2026-03-11T04:01:19.570Z",
    "updatedAt": "2026-03-11T04:02:09.132Z"
  }
}
```

- **generated**: 이미지 생성 시의 원본 정보 기록. 사용된 prompt, 제공자, 모델 등
- **recognized**: AI 인식 결과. 자동 생성된 제목, 태그, 설명
- **manual**: 수동 정리 결과. 이 영역의 데이터는 우선순위가 가장 높으며 AI 인식으로 덮어쓰지 않습니다

이러한 계층 전략은 이전의 핵심 모순을 해결했습니다. AI 인식 결과와 수동 정리 결과 중 어느 것이 우선인가? 답은 수동이 우선이며, AI 인식은 보조일 뿐입니다. 이것도 깨달았습니다. 어떤 것들은 기계는终究 기계이며, 결국 사람이 관리해야 합니다.

## Provider Adapter 패턴

ImgBin의 또 다른 핵심 설계는 Provider Adapter 패턴입니다. 외부 API를 통일된 인터페이스로 추상화하여 AI 서비스 제공자를 변경하더라도 비즈니스 로직을 수정할 필요가 없습니다.

말하자면 감정과 같습니다. 외모가 어떻게 변하든 중요한 것은 내면의 그것이 변하지 않는다는 것입니다. 인터페이스가 정해지면 내부 구현은 어떻게 바꿔도 됩니다.

### Image Generation Provider

```typescript
interface ImageGenerationProvider {
  // 이미지 생성, 이미지의 Buffer 반환
  generate(options: GenerateOptions): Promise<Buffer>;

  // 지원되는 모델 목록 획득
  getSupportedModels(): Promise<string[]>;
}

interface GenerateOptions {
  prompt: string;
  model?: string;
  size?: '1024x1024' | '1792x1024' | '1024x1792';
  quality?: 'standard' | 'hd';
  format?: 'png' | 'webp' | 'jpeg';
}
```

### Vision Recognition Provider

```typescript
interface VisionRecognitionProvider {
  // 이미지 콘텐츠 인식, 구조화된 메타데이터 반환
  recognize(imageBuffer: Buffer): Promise<RecognitionResult>;

  // 지원되는 모델 목록 획득
  getSupportedModels(): Promise<string[]>;
}

interface RecognitionResult {
  title?: string;
  tags: string[];
  description?: string;
  confidence: number;
}
```

이러한 인터페이스 설계의 장점은 다음과 같습니다:

1. **테스트 가능**: 단위 테스트 시 mock provider를 전달할 수 있어 외부 API를 실제로 호출할 필요가 없습니다
2. **확장 가능**: 새 provider를 추가하려면 인터페이스만 구현하면 되며 호출자 코드를 수정할 필요가 없습니다
3. **교체 가능**: 프로덕션 환경에서는 Azure OpenAI를, 테스트 환경에서는 로컬 모델을 사용하며, 설정만 전환하면 됩니다

웃음으로 떨어지는 눈물을 숨기고, 울음으로 마비되었는지 시험하려 하다 보니 — 때로는 프로젝트가 이렇습니다. 겉보기에는 API를 바꾼 것이지만, 실제로는 내면의 그 로직이 전혀 변하지 않았으니 두려울 것이 없습니다.

## CLI 명령 설계

ImgBin은 네 가지 핵심 명령을 제공하여 다양한 사용 시나리오를 충족합니다:

### generate: 단일 이미지 생성

```bash
# 가장 간단한 사용법
imgbin generate --prompt "orange dashboard for docs hero"

# 동시에 썸네일과 AI 태깅 생성
imgbin generate --prompt "orange dashboard" --annotate --thumbnail

# 출력 디렉토리 지정
imgbin generate --prompt "orange dashboard" --output ./library
```

### batch: 일괄 작업

일괄 작업은 YAML 또는 JSON manifest 파일로 정의하며, CI/CD 프로세스에서 사용하기 적합합니다:

```yaml
# assets/jobs/launch.yaml
defaults:
  annotate: true
  thumbnail: true
  libraryRoot: ./library

jobs:
  - prompt: "orange dashboard hero"
    slug: orange-dashboard
    tags: [dashboard, hero, orange]

  - prompt: "pricing grid for docs"
    slug: pricing-grid
    tags: [pricing, grid, docs]
```

실행 명령:

```bash
imgbin batch assets/jobs/launch.yaml
```

일괄 작업 설계는 실패 격리를 지원합니다. manifest에서 항목별로 처리하며, 단일 항목 실패가 다른 작업에 영향을 주지 않습니다. `--dry-run`으로 미리보기만 할 수도 있습니다.

이것도 그렇고, 중요한 것은 어디가 성공했고 어디가 실패했는지 알려준다는 것입니다. 어떤 일은 실패해도 어떻게 실패했는지도 모릅니다.

### annotate: AI 태깅

기존 이미지에 AI 인식을 수행하여 제목, 태그, 설명을 자동 생성합니다:

```bash
# 단일 이미지 태깅
imgbin annotate ./library/2026-03/orange-dashboard

# 전체 디렉토리 일괄 태깅
imgbin annotate ./library/2026-03/
```

### thumbnail: 썸네일 생성

기존 이미지에 썸네일을 보충 생성합니다:

```bash
# 썸네일 생성
imgbin thumbnail ./library/2026-03/orange-dashboard
```

## 일괄 작업 Manifest 설계

일괄 작업의 manifest는 유연한 설정을 지원하며, 기본값을 통일 설정할 수 있고 개별 작업에서도 덮어쓸 수 있습니다:

```yaml
# 전역 기본값
defaults:
  annotate: true        # 기본적으로 AI 태깅 활성화
  thumbnail: true       # 기본적으로 썸네일 생성
  libraryRoot: ./library
  model: gpt-image-1.5

jobs:
  # 최소 설정, prompt만 제공
  - prompt: "first image"

  # 완전한 설정
  - prompt: "second image"
    slug: custom-slug
    tags: [tag1, tag2]
    annotate: false     # 이 작업은 AI 태깅을 수행하지 않음
    model: dall-e-3    # 이 작업은 다른 모델 사용
```

실행 시 ImgBin은 작업을 순차적으로 처리하며, 각 작업의 결과는 해당 metadata.json에 기록됩니다. 특정 작업이 실패하더라도 다른 작업에 영향을 주지 않습니다. 작업 완료 후 요약 보고서를 출력합니다:

```
✓ orange-dashboard (succeeded)
✓ pricing-grid (succeeded)
✗ hero-banner (failed: API rate limit exceeded)

2/3 succeeded, 1 failed
```

어떤 일은 서두르지 못합니다. 하나씩 하면 오히려 확실합니다. 이것이 아마도 일괄 작업의 철학일 것입니다.

## 환경 변수 설정

ImgBin은 환경 변수를 통해 유연한 설정을 지원합니다:

```bash
# ImgBin 작업 디렉토리
IMGBIN_WORKDIR=/path/to/imgbin

# 실행 파일 경로 (스크립트에서 호출 시)
IMGBIN_EXECUTABLE=/path/to/imgbin/dist/cli.js

# 자산 라이브러리 루트 디렉토리
IMGBIN_LIBRARY_ROOT=./.imgbin-library

# Azure OpenAI 설정 (Azure provider를 사용하는 경우)
AZURE_OPENAI_ENDPOINT=https://your-resource.openai.azure.com/
AZURE_OPENAI_API_KEY=your-api-key
AZURE_OPENAI_IMAGE_DEPLOYMENT=gpt-image-1
```

설정이란 중요하기도 하고 중요하지 않기도 합니다.毕竟 편한 대로 하면 됩니다. 자신에게 맞는 것이 가장 좋습니다.

## 구현 요점

구현 과정에서 다음과 같은 핵심 포인트를 요약했습니다:

### Provider 인터페이스 설계

인터페이스 정의는 명확하고 완전해야 하며, 입력 매개변수, 반환값, 오류 처리를 포함해야 합니다. 동기 및 비동기 두 가지 호출 방식을 동시에 제공하여 다양한 시나리오에서 사용하기 편리하게 하는 것을 권장합니다.

이것도 경험자의 조언입니다.毕竟 인터페이스는 한번 정하면 다시 고치기 싫고 번거롭습니다.

### 실패 처리 전략

일괄 작업에서 특정 항목이 실패하면 다음을 수행해야 합니다:

1. 상세한 오류 정보를 별도의 로그 파일에 기록
2. 다른 작업을 계속 실행하고 전체 프로세스를 중단하지 않음
3. 최종적으로 0이 아닌 종료 코드를 반환하여 작업 실패를 표시
4. 요약 보고서에서 각 작업의 실행 결과를 명확하게 표시

어떤 일은 실패하면 실패한 것입니다. 도피도 소용없으니 대답하게 인정하고 해결 방법을 생각하는 것이 낫습니다. 이 이치는 프로젝트와 인생이 같습니다.

### 메타데이터 병합 전략

인식 결과는 기본적으로 `recognized` 영역에 기록되고, 수동으로 수정한 필드는 `manual` 표시가 있습니다. 메타데이터 업데이트 시 "추가만 하고 삭제하지 않는" 전략을 채택합니다. 명시적으로 `--force` 매개변수를 전달하지 않는 한 기존의 수동 정리 결과를 덮어쓰지 않습니다.

이것도 깨달았습니다. 어떤 것은 놓치면 놓친 것이고, 덮어쓰면 사라집니다. 보존하는 것이 좋습니다.毕竟 기록 자체도 일종의 아름다움입니다.

### 디렉토리 생성 원자성

`fs.mkdir({ recursive: true })`를 사용하여 디렉토리 생성의 원자성을 보장하고, 동시 시나리오에서의 경쟁 조건을 방지합니다.

이것이 아마 소위 말하는 안정감일 것입니다. 안정할 때는 안정하고, 빠를 때는 빠르게. 미적지근하지도, 주저하지도 않습니다.

## 요약

ImgBin은 HagiCode 프로젝트 이미지 자산 관리의 핵심 도구로, 다음과 같은 설계로 우리가 직면한 문제를 해결했습니다:

1. **통일된 진입점**: CLI 명령이 생성, 태깅, 썸네일 등 모든 작업을 포괄합니다
2. **메타데이터 기반**: 각 자산에 완전한 metadata.json이 있어 검색 및 추적을 지원합니다
3. **Provider Adapter**: 유연한 외부 API 추상화로 테스트 및 확장이 용이합니다
4. **일괄 작업 지원**: CI/CD 프로세스에서 자동으로 일괄 이미지 생성을 실행할 수 있습니다
