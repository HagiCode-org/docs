## GLM-5.1 완전 지원 및 Gemini CLI 통합: HagiCode의 다중 모델 진화 여정

> 이 글에서는 HagiCode 플랫폼의 두 가지 주요 최근 업데이트를 소개합니다: 지푸AI GLM-5.1 모델의 완전 지원과 Gemini CLI를 열 번째 Agent CLI로 성공적으로 통합한 것입니다. 이 두 업데이트는 함께 플랫폼의 다중 모델 기능과 다중 CLI 생태계를 더욱 강화합니다.

:::note[시점 안내]
이 글은 발행 당시의 다중 CLI 확장 진행 상황을 기록합니다. 현재 항상 지원되는 범위는 [AI Agent CLI 설치](/en/related-software-installation/ai-agent-cli/) 및 [설치 가이드](/en/installation)를 참조하십시오. 여기서 언급된 `IFlowCli`는 당시의 역사적 구성만 나타내며, 현재 활성 지원 범위에는 포함되지 않습니다.
:::

## 배경

시간은 정말 빠르게 흐릅니다. 대형 언어 모델의 발전은 봄에 돋아나는 대나무처럼 치솟고 있습니다. 얼마 전까지만 해도 "코드를 쓸 수 있는 AI"에 환호했는데, 이제는 이미 다중 모델 협업과 다중 도구 통합의 시대에 접어들었습니다. 흥미롭습니까? 아마도. 결국 개발자에게 필요한 것은 도구 자체가 아니라, 다양한 시나리오에 적응하고 필요할 때 유연하게 전환하는 편리함입니다.

AI 보조 코딩 플랫폼으로서 HagiCode는 최근 두 가지 중요한 발전을 맞이했습니다. 첫째, 지푸AI의 GLM-5.1 모델 완전 통합. 둘째, Gemini CLI를 열 번째 지원 Agent CLI로 공식 추가. 이 두 업데이트가 세상을 뒤흔들 정도는 아니지만, 플랫폼의 지속적인 성숙에는 의심할 여지 없이 좋은 소식입니다.

GLM-5.1은 지푸AI의 최신 플래그십 모델입니다. GLM-5.0과 비교하여 더 강력한 추론, 더 깊은 코드 이해, 더 원활한 도구 호출을 제공합니다. 더 중요한 것은 이미지 입력을 지원하는 첫 번째 GLM 모델이라는 점입니다. 이것이 의미하는 바는 무엇일까요? 사용자가 AI에게 문제를 말로 힘겹게 설명하는 대신 스크린샷을 직접 볼 수 있다는 것입니다. 한 번 그 편리함을 경험하면 즉시 그 가치를 이해하게 됩니다.

동시에, `HagiCode.Libs.Providers` 아키텍처를 통해 HagiCode는 Gemini CLI를 플랫폼에 성공적으로 통합했습니다. 이것이 열 번째 Agent CLI입니다. 솔직히 여기까지 오면 약간의 성취감이 듭니다.

HagiCode의 이미지 업로드 기능 덕분에 사용자는 스크린샷을 통해 AI와 직접 소통할 수 있습니다. GLM 4.7을 실행 중에도 플랫폼은 잘 작동하며 이미 여러 중요한 빌드 작업을 완료하는 데 도움이 되었습니다. GLM-5.1은 당연히 한 단계 더 나아갑니다.

## HagiCode 소개

이 글에서 공유하는 접근 방식은 [HagiCode](https://hagicode.com) 프로젝트의 실무 경험에서 나온 것입니다. HagiCode는 다중 모델, 다중 CLI 아키텍처를 통해 개발자에게 유연하고 강력한 AI 프로그래밍 어시스턴트를 제공하기 위해 설계된 오픈소스 AI 보조 코딩 플랫폼입니다. 프로젝트 저장소: [github.com/HagiCode-org/site](https://github.com/HagiCode-org/site)

## 다중 CLI 아키텍처 설계

HagiCode의 핵심 장점 중 하나는 통합 추상화 계층을 통해 여러 AI 프로그래밍 CLI 도구를 지원하는 것입니다. 이 설계의 장점은 사실 매우 간단합니다. 새로운 도구가 들어오고, 이전 도구는 남으며, 코드베이스가 혼란에 빠지지 않습니다. 공정하게 말하면, 모든 사람이 삶이 이렇게 작동하기를 바랄 것입니다.

### AIProviderType 열거형

플랫폼은 `AIProviderType` 열거형을 통해 지원되는 CLI 제공자 유형을 정의합니다:

```csharp
public enum AIProviderType
{
    ClaudeCodeCli = 0,    // Claude Code CLI
    CodexCli = 1,          // GitHub Copilot Codex
    GitHubCopilot = 2,     // GitHub Copilot
    CodebuddyCli = 3,     // Codebuddy CLI
    OpenCodeCli = 4,      // OpenCode CLI
    IFlowCli = 5,         // IFlow CLI
    HermesCli = 6,        // Hermes CLI
    QoderCli = 7,         // Qoder CLI
    KiroCli = 8,          // Kiro CLI
    KimiCli = 9,          // Kimi CLI
    GeminiCli = 10,       // Gemini CLI (신규)
}
```

보시다시피 Gemini CLI가 열 번째 멤버로 이 패밀리에 합류했습니다. 각 CLI는 고유한 특성과 사용 시나리오를 가지고 있으므로, 사용자는 필요에 따라 유연하게 선택할 수 있습니다. 로마로 가는 길은 많으며, 어떤 길은 다른 길보다 더 쉽습니다.

### Provider 아키텍처

`HagiCode.Libs.Providers`는 통합 Provider 인터페이스를 제공하여 각 CLI 통합을 표준화하고 간결하게 만듭니다. Gemini CLI를 예로 들어보겠습니다:

```csharp
public class GeminiProvider : ICliProvider<GeminiOptions>
{
    private static readonly string[] DefaultExecutableCandidates = ["gemini", "gemini-cli"];
    private const string ManagedBootstrapArgument = "--acp";

    public string Name => "gemini";
    public bool IsAvailable => _executableResolver.ResolveFirstAvailablePath(DefaultExecutableCandidates) is not null;
}
```

이 설계의 장점은:
- 새로운 CLI를 통합하려면 하나의 Provider 클래스만 구현하면 됩니다
- 통합 수명 주기 관리 및 세션 풀링
- 자동화된 별칭 해석 및 실행 파일 검색

쉽게 말해 이 설계는 복잡한 것을 더 단순하게 만들고 삶을 조금 더 편하게 합니다.

### Provider 레지스트리

Provider 레지스트리는 별칭 매핑 및 등록을 자동으로 처리합니다:

```csharp
if (provider is GeminiProvider)
{
    registry.Register(provider.Name, provider, ["gemini-cli"]);
    continue;
}
```

즉, 사용자는 `gemini` 또는 `gemini-cli`로 Gemini CLI를 호출할 수 있으며, 시스템이 자동으로 인식합니다. 공식 이름과 별명을 모두 가진 친구와 비슷합니다. 어느 쪽으로 부르든 사람들은 누구인지 알 수 있습니다.

## GLM-5.1 모델 지원

GLM-5.1은 지푸AI의 최신 플래그십 모델이며, HagiCode는 이에 대한 완전한 지원을 완료했습니다.

### Secondary Professions 카탈로그

HagiCode는 Secondary Professions 카탈로그를 통해 지원되는 모든 모델을 관리합니다. GLM 시리즈의 구성은 다음과 같습니다:

| 모델 ID | 이름 | SupportsImage | 호환 CLI 패밀리 |
|----------|------|----------------|-------------------------|
| `glm-4.7` | GLM 4.7 | - | claude, codebuddy, hermes, qoder, kiro |
| `glm-5` | GLM 5 | - | claude, codebuddy, hermes, qoder, kiro |
| `glm-5-turbo` | GLM 5 Turbo | - | claude, codebuddy, hermes, qoder, kiro |
| `glm-5.0` | GLM 5.0 (Legacy) | - | claude, codebuddy, hermes, qoder, kiro |
| `glm-5.1` | GLM 5.1 | **true** | claude, codebuddy, hermes, qoder, kiro |

GLM-5.1의 핵심 특징은 다음과 같이 요약될 수 있습니다:
- 레거시 짐 없는 독립 버전 식별자
- 이미지 입력을 지원하는 첫 번째 GLM 모델
- 더 강력한 추론 및 코드 이해
- 광범위한 다중 CLI 호환성

### GLM-5.1 vs GLM-5.0

코드 수준에서 GLM-5.1과 GLM-5.0의 핵심 차이는 다음과 같습니다:

```csharp
// GLM-5.0 (Legacy) - 특수 유지 로직 포함
private const string Glm50CodebuddySecondaryProfessionId = "secondary-glm-5-codebuddy";
private const string Glm50CodebuddyModelValue = "glm-5.0";

// GLM-5.1 - 독립적인 새 모델 식별자
private const string Glm51SecondaryProfessionId = "secondary-glm-5-1";
private const string Glm51ModelValue = "glm-5.1";
```

GLM-5.0은 하위 호환성을 위해 유지된 이전 버전 식별자이므로 "Legacy" 라벨을 가집니다. 반면 GLM-5.1은 역사적 짐이 없는 완전히 새로운 독립 버전입니다. 어떤 것은 과거에 머물고, 어떤 것은 더 가볍고 더 빠르게 나아갑니다.

### GLM-5.1 구성

HagiCode에서 GLM-5.1을 사용하는 구성 예시입니다:

```json
{
  "primaryProfessionId": "profession-claude-code",
  "secondaryProfessionId": "secondary-glm-5-1",
  "model": "glm-5.1",
  "reasoning": "high"
}
```

## 이미지 업로드 기능

HagiCode의 이미지 지원은 `SecondaryProfession`의 `SupportsImage` 속성을 통해 구현됩니다:

```csharp
public class HeroSecondaryProfessionSettingDto
{
    public bool SupportsImage { get; set; }
}
```

Secondary Professions 카탈로그에서 GLM-5.1 구성은 다음과 같습니다:

```json
{
  "id": "secondary-glm-5-1",
  "supportsImage": true
}
```

즉, 사용자는 스크린샷을 직접 업로드하여 AI 분석을 받을 수 있습니다. 예를 들어:
- 오류 메시지의 스크린샷
- UI 화면의 문제
- 데이터 시각화 차트
- 코드 실행 결과

더 이상 모든 것을 수동으로 설명할 필요가 없습니다. 스크린샷을 업로드하기만 하면 됩니다. 이 기능의 편리함은 한 번 사용해 보면 명확해집니다. 때로는 한 번 보는 것이 긴 설명보다 더 많은 것을 말해줍니다.

## Gemini CLI 통합

열 번째 Agent CLI로서 Gemini CLI는 표준 Provider 아키텍처를 통해 HagiCode에 통합됩니다.

### 구성 옵션

Gemini CLI는 다양한 구성 옵션을 지원합니다:

```csharp
public class GeminiOptions
{
    public string? ExecutablePath { get; set; }
    public string? WorkingDirectory { get; set; }
    public string? SessionId { get; set; }
    public string? Model { get; set; }
    public string? AuthenticationMethod { get; set; }
    public string? AuthenticationToken { get; set; }
    public Dictionary<string, string?> AuthenticationInfo { get; set; }
    public Dictionary<string, string?> EnvironmentVariables { get; set; }
    public string[] ExtraArguments { get; set; }
    public TimeSpan? StartupTimeout { get; set; }
    public CliPoolSettings? PoolSettings { get; set; }
}
```

이 옵션은 기본 설정부터 고급 기능까지 모두 포함하며, 사용자가 자신의 필요에 맞게 CLI를 구성할 수 있는 유연성을 제공합니다. 모든 사람의 워크플로우가 다르므로, 약간의 유연성은 항상 환영받습니다.

### ACP 통신 프로토콜

Gemini CLI는 ACP(Agent Communication Protocol)를 지원하며, 이는 HagiCode의 통합 CLI 통신 표준입니다. ACP를 통해 다양한 CLI가 일관된 방식으로 플랫폼과 상호작용할 수 있어, 통합 작업이 크게 간소화됩니다. 쉽게 말해 복잡한 부분을 표준화하여 모두가 더 쉽게 일할 수 있도록 합니다.

## HagiCode 자체 빌드 실무

실제 실무를 말하자면, 가장 좋은 예는 HagiCode 플랫폼 자체의 빌드 워크플로입니다. HagiCode의 개발 프로세스는 이미 AI 기능을 충분히 활용하고 있습니다.

### GLM 4.7에서도 잘 작동합니다

HagiCode의 플랫폼 설계는 잘 최적화되어 있어, GLM 4.7에서도 여전히 좋은 개발 경험을 제공합니다. 플랫폼은 이미 여러 중요한 빌드 프로젝트를 완료하는 데 도움이 되었으며, 다음을 포함합니다:
- 여러 CLI Provider 통합
- 이미지 업로드 기능 구현
- 문서 생성 및 콘텐츠 게시

이것은 사실 좋은 일입니다. 모든 사람이 항상 최신 것을 필요로 하는 것은 아닙니다. 본인에게 가장 잘 맞는 것이 가장 중요합니다.

### GLM-5.1은 더 적은 노력으로 더 많은 것을 제공합니다

GLM-5.1로 업그레이드하면 이러한 기능이 더욱 강화됩니다:
- 더 강력한 코드 이해로, 왕복 통신 감소
- 더 정확한 의존성 분석으로, 즉시 올바른 방향을 가리킴
- 더 효율적인 오류 진단으로, 문제를 더 빠르게 찾아냄
- 이미지 입력 지원으로, 문제 설명이 가속화됨

자전거에서 자동차로 바꾸는 것과 비슷합니다. 같은 목적지에 도달할 수 있지만, 속도와 편안함이 다릅니다.

## 다중 CLI 통합 모범 사례

`HagiCode.Libs.Providers`는 등록 및 사용을 위한 통합 메커니즘을 제공합니다:

```csharp
services.AddHagiCodeLibs();

var gemini = serviceProvider.GetRequiredService<ICliProvider<GeminiOptions>>();
var codebuddy = serviceProvider.GetRequiredService<ICliProvider<CodebuddyOptions>>();
var hermes = serviceProvider.GetRequiredService<ICliProvider<HermesOptions>>();
```

이 의존성 주입 설계는 다양한 CLI 간의 사용을 매우 간결하게 만들며, 단위 테스트와 모킹도 더 편리합니다. 깨끗한 코드는 자신에게 책임을 다하는 방식입니다.

### 참고 사항

실제 사용 시 몇 가지 주의사항이 있습니다:

1. **API 키 구성**: `ANTHROPIC_AUTH_TOKEN`이 올바르게 설정되어 있는지 확인하십시오. 그렇지 않으면 모델을 호출할 수 없습니다
2. **모델 가용성**: GLM-5.1은 해당 모델 제공자에서 활성화해야 합니다
3. **이미지 기능**: `supportsImage: true`인 모델만 이미지 업로드를 사용할 수 있습니다
4. **CLI 설치**: Gemini CLI를 사용하기 전에 `gemini` 또는 `gemini-cli`가 시스템 PATH에 있는지 확인하십시오

이것들은 작은 세부 사항이지만, 작은 세부 사항을 잘못 처리하면 큰 문제가 될 수 있으므로 주의할 가치가 있습니다.

## 결론

GLM-5.1의 완전한 지원과 Gemini CLI의 성공적인 통합을 통해 HagiCode는 다중 모델, 다중 CLI AI 프로그래밍 플랫폼으로서의 역량을 더욱 강화합니다. 이러한 업데이트는 사용자에게 더 많은 선택지를 제공할 뿐만 아니라, HagiCode의 선구적인 아키텍처와 확장성을 보여줍니다.

GLM-5.1의 이미지 지원과 HagiCode의 스크린샷 업로드 기능을 결합하면 AI가 "이미지에서 이해"할 수 있게 되어, 문제 설명의 비용이 크게 줄어듭니다. 그리고 10개의 CLI를 지원함으로써, 사용자는 자신의 환경설정과 시나리오에 가장 잘 맞는 AI 프로그래밍 어시스턴트를 유연하게 선택할 수 있습니다. 더 많은 선택지는 거의 항상 좋은 일입니다.

가장 중요한 것은 HagiCode 자체의 빌드 실무가 플랫폼이 GLM 4.7에서도 이미 잘 실행되고 복잡한 작업을 완료할 수 있음을 증명하며, GLM-5.1로 업그레이드하면 개발 효율을 더욱 향상시킬 수 있다는 점입니다. 삶도 종종 그렇습니다. 항상 절대 최고가 필요한 것이 아니라, 본인에게 맞는 것이 필요할 뿐입니다. 물론, 본인에게 맞는 것이 더 좋아질 수 있다면 더할 나위 없이 좋습니다.

다중 모델, 다중 CLI AI 프로그래밍 플랫폼에 관심이 있으시다면 HagiCode를 사용해 보십시오. 오픈소스, 무료이며, 여전히 진화하고 있습니다. 사용해 보는 데 아무런 비용이 들지 않으며, 정확히 필요한 것이 될 수도 있습니다.
