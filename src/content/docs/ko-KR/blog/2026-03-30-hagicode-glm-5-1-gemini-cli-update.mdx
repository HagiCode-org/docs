## GLM-5.1 완전 지원 및 Gemini CLI 통합: HagiCode의 다중 모델 진화之路

> 本文介绍了 HagiCode 平台近期的重要更新——智谱 AI GLM-5.1 模型的全面支持，以及 Gemini CLI 作为第十个 Agent CLI 的成功集成。这两项更新进一步强化了平台的多模型能力和多 CLI 生态。

## 배경

시간이 정말 빠르네요, 대형 언어 모델의 발전은 마치 봄의 대나무처럼 쑥쑥 자라납니다. 한때 우리는 "코드를 작성할 수 있는 AI"를 위해 환호했지만, 이제는 이미 다중 모델 협업, 다중 도물 융합의 시대가 되었습니다. 이게 재미있나요? 어쩌면 그렇겠죠,毕竟 개발자에게 필요한 것은 도구 그 자체가 아니라, 다른 시나리오에 적응하고 유연하게 전환할 수 있는 여유니까요.

HagiCode는 AI 보조 코딩 플랫폼으로서, 최근 두 가지 중요한 일을 맞이했습니다: 하나는 Zhipu AI의 GLM-5.1 모델이 완전히接入된 것이고, 다른 하나는 Gemini CLI가 공식적으로 열 번째 지원되는 Agent CLI가 된 것입니다. 이 두 가지 일은 크다면 크고 작다면 작지만, 플랫폼의 완성을 위해서는 어쨌든 좋은 일입니다.

GLM-5.1은 Zhipu AI의 최신 플래그십 모델로, GLM-5.0에 비해 추론 능력이 더 강해지고, 코드 이해가 더 깊어지며, 도구 호출이 더 원활해졌습니다. 더 중요한 것은 이미지 입력을 지원하는 첫 번째 GLM 모델이라는 점입니다 - 이게 무엇을 의미할까요? 사용자가 직접 스크린샷을 찍어 AI에게 문제를 보여줄 수 있고, 더 이상 힘들게 설명할 필요가 없다는 것입니다. 이 편리함은 써보면 압니다.

한편, HagiCode는 HagiCode.Libs.Providers 아키텍처를 통해 Gemini CLI를 성공적으로 통합했습니다. 이것은 열 번째 Agent CLI입니다. 솔직히 여기까지 올 수 있어서 약간의 성취감이 느껴집니다.

특히 언급할 만한 것은, HagiCode의 이미지 업로드 기능이 사용자가 직접 스크린샷을 찍어 AI와 소통할 수 있게 한다는 것입니다. GLM 4.7 버전을 실행하더라도 플랫폼은 여전히 원활하게 실행되며, 이미 프로젝트의 많은 중요한 빌드 작업을 완료하는 데 도움이 되었습니다. GLM-5.1은 어떨까요? 당연히 더 한층 발전할 것입니다.

## HagiCode에 대하여

本文分享的方案来自我们在 [HagiCode](https://hagicode.com) 项目中的实践经验。HagiCode 是一个开源的 AI 辅助编码平台，旨在通过多模型、多 CLI 的架构设计，为开发者提供灵活、强大的 AI 编程助手。项目地址：[github.com/HagiCode-org/site](https://github.com/HagiCode-org/site)

## 다중 CLI 아키텍처 설계

HagiCode의 핵심 장점 중 하나는 통합된 추상화 계층을 통해 다양한 AI 프로그래밍 CLI 도구를 지원하는 것입니다. 이 설계의 장점은, 솔직히 말하면 그런 것입니다: 새로운 것이 들어오고, 오래된 것이 남고, 코드가 엉망이 되지 않습니다.毕竟, 누구나 삶이 이랬으면 좋겠죠?

### AIProviderType 열거형

플랫폼은 `AIProviderType` 열거형을 통해 지원되는 CLI 공급자 유형을 정의합니다:

```csharp
public enum AIProviderType
{
    ClaudeCodeCli = 0,    // Claude Code CLI
    CodexCli = 1,          // GitHub Copilot Codex
    GitHubCopilot = 2,     // GitHub Copilot
    CodebuddyCli = 3,     // Codebuddy CLI
    OpenCodeCli = 4,      // OpenCode CLI
    IFlowCli = 5,         // IFlow CLI
    HermesCli = 6,        // Hermes CLI
    QoderCli = 7,         // Qoder CLI
    KiroCli = 8,          // Kiro CLI
    KimiCli = 9,          // Kimi CLI
    GeminiCli = 10,       // Gemini CLI (新增)
}
```

보시다시피, Gemini CLI는 열 번째 멤버로 이 대가족에 합류했습니다. 각 CLI는 고유한 특징과 적용 가능한 시나리오가 있으며, 사용자는 자신의 필요에 따라 유연하게 선택할 수 있습니다.毕竟, 모든 길은 로마로 통하지만, 어떤 길은 조금 더 걷기 좋고, 어떤 길은 조금 더 구불구불할 뿐입니다.

### Provider 아키텍처

HagiCode.Libs.Providers는 통합된 Provider 인터페이스를 제공하여 각 CLI의 통합을 규범적이고 간결하게 만듭니다. Gemini CLI를 예로 들면:

```csharp
public class GeminiProvider : ICliProvider<GeminiOptions>
{
    private static readonly string[] DefaultExecutableCandidates = ["gemini", "gemini-cli"];
    private const string ManagedBootstrapArgument = "--acp";

    public string Name => "gemini";
    public bool IsAvailable => _executableResolver.ResolveFirstAvailablePath(DefaultExecutableCandidates) is not null;
}
```

이 설계의 장점은:
- 새로운 CLI 통합은 하나의 Provider 클래스를 구현하기만 하면 됩니다
- 통합된 수명 주기 관리 및 세션 풀링
- 자동화된 별칭 확인 및 실행 파일 찾기

솔직히 말하면, 이 설계는 복잡한 일을 간단하게 만드는 것, 즉 삶을 조금 더 편하게 만드는 것입니다.

### Provider Registry

Provider Registry는 별칭 매핑 및 등록을 자동으로 처리합니다:

```csharp
if (provider is GeminiProvider)
{
    registry.Register(provider.Name, provider, ["gemini-cli"]);
    continue;
}
```

这意味着 사용자는 `gemini` 또는 `gemini-cli` 두 가지 방식으로 Gemini CLI를 호출할 수 있으며, 시스템이 자동으로 인식합니다. 이것은 친구가 많은 것과 같습니다. 어떤 사람은 대명으로 부르고, 어떤 사람은 소명으로 부르지만, 어차피 그 사람입니다. 어떻게 부르든 상관없습니다.

## GLM-5.1 모델 지원

GLM-5.1은 Zhipu AI의 최신 플래그십 모델로, HagiCode는 이미 이에 대한 완전한 지원을 완료했습니다.

### Secondary Professions Catalog

HagiCode는 Secondary Professions Catalog를 통해 지원되는 모든 모델을 관리합니다. 다음은 GLM 시리즈의 구성입니다:

| Model ID | Name | SupportsImage | Compatible CLI Families |
|----------|------|----------------|-------------------------|
| `glm-4.7` | GLM 4.7 | - | claude, codebuddy, hermes, qoder, kiro |
| `glm-5` | GLM 5 | - | claude, codebuddy, hermes, qoder, kiro |
| `glm-5-turbo` | GLM 5 Turbo | - | claude, codebuddy, hermes, qoder, kiro |
| `glm-5.0` | GLM 5.0 (Legacy) | - | claude, codebuddy, hermes, qoder, kiro |
| `glm-5.1` | GLM 5.1 | **true** | claude, codebuddy, hermes, qoder, kiro |

GLM-5.1의 핵심 특징을 요약하면:
- 독립적인 버전 식별, legacy 짐이 없음
- 이미지 입력을 지원하는 첫 번째 GLM 모델
- 더 강한 추론 능력 및 코드 이해
- 광범위한 다중 CLI 호환성

### GLM-5.1 vs GLM-5.0

코드 관점에서 GLM-5.1과 GLM-5.0의 핵심 차이점:

```csharp
// GLM-5.0 (Legacy) - 특별한 보존 논리가 있음
private const string Glm50CodebuddySecondaryProfessionId = "secondary-glm-5-codebuddy";
private const string Glm50CodebuddyModelValue = "glm-5.0";

// GLM-5.1 - 독립적인 새 모델 식별
private const string Glm51SecondaryProfessionId = "secondary-glm-5-1";
private const string Glm51ModelValue = "glm-5.1";
```

GLM-5.0은 "Legacy" 표시가 있으며, 하위 호환성을 위해 남겨둔 오래된 버전 식별자입니다. 반면 GLM-5.1은 완전히 새로운 독립 버전으로, 어떠한 역사적 짐도 없습니다. 이것은 어떤 사람은 항상 과거에 살고, 어떤 사람은 가볍게 출발하여 더 빨리 나아가는 것과 같습니다.

### GLM-5.1 구성

HagiCode에서 GLM-5.1을 사용하는 구성 예:

```json
{
  "primaryProfessionId": "profession-claude-code",
  "secondaryProfessionId": "secondary-glm-5-1",
  "model": "glm-5.1",
  "reasoning": "high"
}
```

## 이미지 업로드 기능

HagiCode의 이미지 지원은 SecondaryProfession의 `SupportsImage` 속성을 통해 구현됩니다:

```csharp
public class HeroSecondaryProfessionSettingDto
{
    public bool SupportsImage { get; set; }
}
```

Secondary Professions Catalog에서 GLM-5.1의 구성은 다음과 같습니다:

```json
{
  "id": "secondary-glm-5-1",
  "supportsImage": true
}
```

这意味着 사용자는 직접 스크린샷을 업로드하여 AI가 분석하게 할 수 있습니다. 예를 들어:
- 오류 메시지의 스크린샷
- UI 인터페이스의 문제
- 데이터 시각화 차트
- 코드 실행 결과

더 이상 수동으로 문제를 설명할 필요가 없으며, 직접 스크린샷을 찍으면 됩니다 - 이 기능의 편리함은 써보면 압니다.毕竟, 어떤 일은 백 마디 말보다 한 번 보는 것이 낫습니다.

## Gemini CLI 통합

Gemini CLI는 열 번째 Agent CLI로서 표준 Provider 아키텍처를 통해 HagiCode에 통합되었습니다.

### 구성 옵션

Gemini CLI는 풍부한 구성 옵션을 지원합니다:

```csharp
public class GeminiOptions
{
    public string? ExecutablePath { get; set; }
    public string? WorkingDirectory { get; set; }
    public string? SessionId { get; set; }
    public string? Model { get; set; }
    public string? AuthenticationMethod { get; set; }
    public string? AuthenticationToken { get; set; }
    public Dictionary<string, string?> AuthenticationInfo { get; set; }
    public Dictionary<string, string?> EnvironmentVariables { get; set; }
    public string[] ExtraArguments { get; set; }
    public TimeSpan? StartupTimeout { get; set; }
    public CliPoolSettings? PoolSettings { get; set; }
}
```

이러한 옵션은 기본 구성에서 고급 기능에 이르기까지 모든 측면을 다루며, 사용자는 자신의 필요에 따라 유연하게 구성할 수 있습니다.毕竟, 모든 사람의 필요는 다릅니다. 유연할 수록 항상 좋습니다.

### ACP 통신 프로토콜

Gemini CLI는 ACP (Agent Communication Protocol) 통신 프로토콜을 지원하며, 이것은 HagiCode의 통합된 CLI 통신 표준입니다. ACP를 통해 다른 CLI는 일관된 방식으로 플랫폼과 상호 작용할 수 있으며, 통합 작업이 크게 간소화됩니다. 솔직히 말하면, 복잡한 일을 통일하여 모두가 조금 더 쉬워지게 하는 것입니다.

## HagiCode 자체 빌드 실천

실천에 대해 말하자면, 가장 좋은 예는 HagiCode 플랫폼 자체의 빌드 프로세스입니다. HagiCode의 개발 과정은 이미 AI 능력을 충분히 활용했습니다:

### GLM 4.7만 사용해도 잘 실행됨

HagiCode 플랫폼의 설계가 비교적 최적화되어 있어 GLM 4.7을 사용하더라도 좋은 개발 경험을 얻을 수 있습니다. 플랫폼은 이미 다음과 같은 여러 중요한 빌드 프로젝트 완료를 도왔습니다:
- 다중 CLI Provider 통합
- 이미지 업로드 기능 구현
- 문서 생성 및 콘텐츠 게시

사실 이것도 꽤 좋습니다.毕竟 모든 사람이 최신 것을 사용할 필요는 없습니다. 자신에게 맞는 것이 가장 좋은 것입니다.

### GLM-5.1으로 배가됨

GLM-5.1로 업그레이드한 후 이러한 능력은 더욱 강화됩니다:
- 더 강한 코드 이해 능력, 왕복 소통 감소
- 더 정확한 종속성 분석, 한 번에 올바른 방향 지정
- 더 효율적인 오류 진단, 빠른 문제 위치 파악
- 이미지 입력 지원, 문제 설명 가속화

이것은 자전거에서 자동차로 바꾸는 것과 같습니다. 도착할 수 있는 곳은 같지만, 속도와 편안함이 다를 뿐입니다.

## 다중 CLI 통합 모범 사례

HagiCode.Libs.Providers는 통합된 등록 및 사용 메커니즘을 제공합니다:

```csharp
services.AddHagiCodeLibs();

var gemini = serviceProvider.GetRequiredService<ICliProvider<GeminiOptions>>();
var codebuddy = serviceProvider.GetRequiredService<ICliProvider<CodebuddyOptions>>();
var hermes = serviceProvider.GetRequiredService<ICliProvider<HermesOptions>>();
```

이러한 종속성 주입 설계는 각 CLI의 사용을 매우 간결하게 만들며, 단위 테스트 및 모의도 편리합니다.毕竟, 코드를 깔끔하게 작성하는 것은 자신에게도 책임입니다.

### 주의 사항

실제 사용에서 몇 가지 주의해야 할 곳이 있습니다:

1. **API Key 구성**: `ANTHROPIC_AUTH_TOKEN`이 올바르게 설정되어 있는지 확인하세요. 그렇지 않으면 모델을 호출할 수 없습니다
2. **모델 가용성**: GLM-5.1은 해당 모델 공급자에서 권한을 개통해야 합니다
3. **이미지 기능**: `supportsImage: true`를 지원하는 모델만 이미지 업로드 기능을 사용할 수 있습니다
4. **CLI 설치**: Gemini CLI를 사용하기 전에 `gemini` 또는 `gemini-cli`가 시스템 PATH에 있는지 확인하세요

이것들은 모두 사소한 일이지만, 사소한 일을 잘 처리하지 못하면 큰일이 될 수도 있습니다. 그러니 주의해야 합니다.

## 요약

GLM-5.1의 완전한 지원과 Gemini CLI의 성공적인 통합을 통해 HagiCode는 다중 모델, 다중 CLI AI 프로그래밍 플랫폼으로서의 능력을 더욱 강화했습니다. 이러한 업데이트는 사용자에게 더 많은 선택지를 제공할 뿐만 아니라 HagiCode의 아키텍처 설계에서의 전망성과 확장성도 보여줍니다.

GLM-5.1의 이미지 지원 능력은 HagiCode의 스크린샷 업로드 기능과 결합하여 "그림을 보고 말하기"를 가능하게 합니다 - 문제 설명의 비용을 크게 줄여줍니다. 그리고 10개 CLI의 지원은 사용자가 자신의 선호도와 시나리오에 따라 가장 적합한 AI 프로그래밍 도우미를 유연하게 선택할 수 있음을 의미합니다.毕竟, 선택이 많으면 항상 좋은 일입니다.

가장 중요한 것은 HagiCode 플랫폼 자체의 빌드 실천이 증명했다는 점입니다: GLM 4.7을 사용하더라도 플랫폼은 잘 실행되고 복잡한 작업을 완료할 수 있습니다. GLM-5.1로 업그레이드한 후 개발 효율성은 더욱 향상될 것입니다. 이것은 인생과 같습니다. 반드시 최고를 추구할 필요는 없습니다. 자신에게 맞는 것이 좋습니다. 물론 자신에게 맞는 것을 더 잘하게 된다면 더 좋습니다.

다중 모델, 다중 CLI의 AI 프로그래밍 플랫폼에 관심이 있다면 HagiCode를 시도해 보세요 - 오픈 소스, 무료, 끊임없이 진화합니다. 어차피 시도해 보는 것은 비용이 들지 않습니다.万一 정말 당신에게 맞다면요?
