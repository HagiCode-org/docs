## CLI 호출에서 SDK 통합으로: .NET 프로젝트에서의 GitHub Copilot 모범 사례

> 명령줄 호출에서 공식 SDK 통합으로의 업그레이드 여정, 말하자면 꽤 경험이라 할 만하죠. 오늘은 HagiCode 프로젝트에서 겪은 시행착오와 배운 점을 공유해 드리겠습니다.

## 배경

GitHub Copilot SDK가 2025년에 정식으로 출시된 후, 우리는 이를 AI 기능 레이어에 통합하기 시작했습니다. 그 전에는 프로젝트가 주로 Copilot CLI 명령줄 도구를 직접 호출하여 GitHub Copilot 기능을 사용했는데, 이 방식에는 몇 가지 명확한 문제가 있었습니다.

- **프로세스 관리 복잡성**: CLI 프로세스의 수명 주기, 시작 시간 초과 및 프로세스 정리를 수동으로 관리해야 합니다. 아무래도 프로세스라는 게, 말하자면 갑자기崩溃하는 경우도 있고 별다른 예고도 없으니까요.
- **이벤트 처리 불완전**: 원시 CLI 호출에서는 모델 추론 과정과 도구 실행의 세분화된 이벤트를 포착하기 어렵습니다. 결과만 볼 수 있고 생각하는 과정은 볼 수 없는 것과 같습니다.
- **세션 관리 어려움**: 효과적인 세션 재사용 및 복구 메커니즘이 부족하여 매번 다시 시작해야 합니다. 생각해보면 꽤 피곤한 일이죠.
- **호환성 문제**: CLI 매개변수가 자주 업데이트되어 매개변수 호환성 로직을 지속적으로 유지 관리해야 합니다. 이건 풍차와 싸우는 것과 다를 바 없습니다.

이러한 문제들은 일상 개발에서 점차 드러났으며, 특히 모델 추론 과정(thinking)과 도구 실행 상태를 실시간으로 추적해야 할 때 CLI 호출의 한계가 특히 명확했습니다. 우리도 어느 정도 깨달았습니다. 더 기본적이고 완전한 통합 방식이 필요하다는 것을요.毕竟, 모든 길은 로마로 통하지만, 어떤 길은 조금 더 쉽고, 어떤 길은 조금 더 굽이쳐 있을 뿐입니다.

## HagiCode에 대해

이 글에서 공유하는 솔루션은 [HagiCode](https://hagicode.com) 프로젝트에서의 실제 경험에서 나왔습니다. HagiCode는 오픈 소스 AI 코드 도우미 프로젝트로, 개발 과정에서 우리는 GitHub Copilot의 다양한 기능을 깊이 통합해야 했습니다. 기본적인 코드 완성부터 복잡한 다단계 대화와 도구 호출까지. 이러한 실제 요구 사항이 우리를 CLI 호출에서 공식 SDK 통합으로 업그레이드하게 만들었습니다.

만약 이 글의 실제 솔루션에 관심이 있다면, 우리의 엔지니어링 실무가 도움이 될 수 있다는 뜻입니다. 그렇다면 HagiCode 프로젝트 자체도 주목해 볼 만합니다. 아마 글 마지막에서 프로젝트에 대한 더 많은 정보와 링크를 발견하게 될지도 모르겠습니다. 누가 알겠습니까......

## 아키텍처 설계

프로젝트는 CLI 호출 문제를 해결하기 위해 계층형 아키텍처를 채택했습니다.

```
┌─────────────────────────────────────────────────────────┐
│  hagicode-core (Orleans Grains + AI Provider Layer)    │
│  - CopilotAIProvider: AIRequest를 CopilotOptions로 변환 │
│  - GitHubCopilotGrain: Orleans 분산 실행 인터페이스            │
└─────────────────────────────────────────────────────────┘
                          ↓
┌─────────────────────────────────────────────────────────┐
│  HagiCode.Libs (Shared Provider Layer)                 │
│  - CopilotProvider: CLI Provider 인터페이스 구현               │
│  - ICopilotSdkGateway: SDK 호출 추상                     │
│  - GitHubCopilotSdkGateway: SDK 세션 관리 및 이벤트 분배     │
└─────────────────────────────────────────────────────────┘
                          ↓
┌─────────────────────────────────────────────────────────┐
│  GitHub Copilot SDK (Official .NET SDK)                │
│  - CopilotClient: SDK 클라이언트                            │
│  - CopilotSession: 세션 관리                             │
│  - SessionEvent: 이벤트 스트림                                 │
└─────────────────────────────────────────────────────────┘
```

이러한 계층 설계가 가져오는 기술적 이점들은, 사실 꽤 실용적입니다.

1. **관심사 분리**: 핵심 비즈니스 로직과 SDK 구현 세부 사항의 분리——毕竟, 각 레이어는 각자의 일을 하고, 서로 간섭하지 않습니다.
2. **테스트 가능성**: `ICopilotSdkGateway` 인터페이스를 통해 단위 테스트를 쉽게 수행할 수 있어, 테스트도 그렇게 번거롭지 않습니다.
3. **재사용성**: HagiCode.Libs는 여러 프로젝트에서 참조할 수 있어, 한 번 작성하면 여러 곳에서 사용합니다.
4. **유지 관리성**: SDK 업그레이드는 Gateway 레이어만 수정하면 되고, 상위 코드는 건드리지 않아도 되어, 아주 좋습니다.

## 핵심 구현

### 인증 흐름

인증은 SDK 통합의 첫 단계이자 가장 중요한 단계입니다——毕竟, 문도 못 들어가면 다음 이야기는 없으니까요. 우리는 유연한 인증 구성을 설계하여 여러 인증 소스를 지원합니다.

```csharp
// CopilotProvider.cs - 인증 소스 구성
public class CopilotOptions
{
    public bool UseLoggedInUser { get; set; } = true;
    public string? GitHubToken { get; set; }
    public string? CliUrl { get; set; }
}

// SDK 요청으로 변환
return new CopilotSdkRequest(
    GitHubToken: options.AuthSource == CopilotAuthSource.GitHubToken
        ? options.GitHubToken
        : null,
    UseLoggedInUser: options.AuthSource != CopilotAuthSource.GitHubToken
);
```

이 설계의 장점은, 사실 꽤 명확합니다.

- 로그인된 사용자 모드(토큰 불필요) 지원, 데스크톱 환경에 적합——사용자가 자신의 계정으로 로그인하면 됩니다.
- GitHub 토큰 모드 지원, 서버 측 배포에 적용——통일 관리도 편리합니다.
- Copilot CLI URL 재정의 지원, 기업 프록시 구성에 편리——기업 환경이니까, 항상 특별한 규칙이 있기 마련입니다.

실제 사용에서 이러한 유연한 인증 방식은 다양한 배포 시나리오의 구성 작업을 크게 단순화했습니다. 데스크톱에서는 사용자 자신의 Copilot 로그인 상태를 사용할 수 있고, 서버에서는 토큰을 통해 통일 관리할 수 있습니다.怎么说呢, 각자 필요한 것을 취하는 것일 뿐입니다.

### 이벤트 스트림 처리

SDK의 가장 강력한 기능 중 하나는 완전한 이벤트 스트림 포착能力입니다. 우리는 다양한 SDK 이벤트를 실시간으로 처리할 수 있는 이벤트 분배 시스템을 구현했습니다——毕竟, 과정을 아는 것과 결과만 아는 것은 느낌이 다릅니다.

```csharp
// GitHubCopilotSdkGateway.cs - 이벤트 분배 핵심 로직
internal static SessionEventDispatchResult DispatchSessionEvent(
    SessionEvent evt, bool sawDelta)
{
    switch (evt)
    {
        case AssistantReasoningEvent reasoningEvent:
            // 모델 추론 과정 포착
            events.Add(new CopilotSdkStreamEvent(
                CopilotSdkStreamEventType.ReasoningDelta,
                Content: reasoningEvent.Data.Content));
            break;

        case ToolExecutionStartEvent toolStartEvent:
            // 도구 호출 시작 포착
            events.Add(new CopilotSdkStreamEvent(
                CopilotSdkStreamEventType.ToolExecutionStart,
                ToolName: toolStartEvent.Data.ToolName,
                ToolCallId: toolStartEvent.Data.ToolCallId));
            break;

        case ToolExecutionCompleteEvent toolCompleteEvent:
            // 도구 호출 완료 및 결과 포착
            events.Add(new CopilotSdkStreamEvent(
                CopilotSdkStreamEventType.ToolExecutionEnd,
                Content: ExtractToolExecutionContent(toolCompleteEvent)));
            break;

        default:
            // 처리되지 않은 이벤트는 RawEvent로 보존
            events.Add(new CopilotSdkStreamEvent(
                CopilotSdkStreamEventType.RawEvent,
                RawEventType: evt.GetType().Name));
            break;
    }
}
```

이 구현이 가져오는 가치는,怎么说呢:

- **모델 추론 과정 완전 포착**(thinking): 사용자가 AI의 생각 과정을 볼 수 있고, 최종 결과뿐만 아니라——마치 답을 아는 것보다 어떻게 생각해냈는지 아는 것이 더 중요합니다.
- **도구 실행 상태 실시간 추적**: 어떤 도구가 실행 중인지, 언제 완료되는지, 어떤 결과를 반환했는지 알 수 있습니다.
- **이벤트 손실 없음**: RawEvent 메커니즘에 대한 fallback을 통해 모든 이벤트가 기록되도록 보장하며, 아무것도 놓치지 않습니다.

HagiCode의 실제 사용에서 이러한 세분화된 이벤트는 사용자가 AI의 작동 과정을 더 깊이 이해할 수 있게 해줍니다. 특히 복잡한 작업을 디버깅할 때——이건 꽤 유용합니다.

### CLI 호환성 처리

CLI 호출에서 SDK로 마이그레이션한 후 우리는 일부 기존 CLI 매개변수가 SDK에서 더 이상 적용되지 않는다는 것을 발견했습니다. 하위 호환성을 유지하기 위해 매개변수 필터링 시스템을 구현했습니다——毕竟, 구성을 사용할 수 없는 것도 꽤 골치 아프니까요.

```csharp
// CopilotCliCompatibility.cs - 매개변수 필터링
private static readonly Dictionary<string, string> RejectedFlags = new()
{
    ["--headless"] = "지원되지 않는 시작 매개변수",
    ["--model"] = "SDK 네이티브 필드를 통해 전달",
    ["--prompt"] = "SDK 네이티브 필드를 통해 전달",
    ["--interactive"] = "공급자가 대화형 관리",
};

public static CopilotCliArgumentBuildResult BuildCliArgs(CopilotOptions options)
{
    // 지원되지 않는 매개변수 필터링, 호환 매개변수 유지
    // 진단 정보 생성
}
```

이렇게 하는 장점:

- 호환되지 않는 CLI 매개변수를 자동으로 필터링하여 런타임 오류 방지——프로그램崩溃하는 것은 장난이 아닙니다.
- 명확한 오류 진단 정보를 생성하여 개발자가 문제를 빠르게 찾을 수 있도록 돕습니다.
- SDK 안정성을 보장하며 CLI 매개변수 변경의 영향을 받지 않습니다.

업그레이드 과정에서 이러한 호환성 처리 메커니즘은 원활한 전환을 도왔습니다. 이전 구성 파일은 계속 사용할 수 있으며 진단 정보에 따라 점진적으로 조정하면 됩니다——점진적인 과정이라 할 수 있겠습니다.

### 런타임 풀링

Copilot SDK의 세션 생성 비용이 높아서 세션을 자주 생성하고 삭제하면 성능에 영향을 미칩니다. 우리는 세션 풀 관리 시스템을 구현했습니다——마치 물탱크의 물처럼, 다 쓰고 나서 다시 채우는 것보다 남겨두었다가 다음에 계속 쓰는 게 낫습니다.

```csharp
// CopilotProvider.cs - 세션 풀 관리
await using var lease = await _poolCoordinator.AcquireCopilotRuntimeAsync(
    request,
    async ct => await _gateway.CreateRuntimeAsync(sdkRequest, ct),
    cancellationToken);

if (lease.IsWarmLease)
{
    // 기존 세션 재사용
    yield return CreateSessionReusedMessage();
}

await foreach (var eventData in lease.Entry.Resource.SendPromptAsync(...))
{
    yield return MapEvent(eventData);
}
```

세션 풀링의 장점:

- **세션 재사용**: 동일한 sessionId의 요청은 기존 세션을 재사용할 수 있어 시작 오버헤드를 줄입니다.
- **세션 복구 지원**: 네트워크 중단 후 이전 세션 상태를 복구할 수 있습니다——毕竟 네트워크라는 건, 누가 항상 안정적이라고 보장하겠습니까.
- **자동 풀 관리**: 만료된 세션을 자동으로 정리하여 리소스 누수를 방지합니다.

HagiCode의 실제 사용에서 세션 풀링은 응답 속도를 크게 향상시켰습니다. 특히 연속 대화를 처리할 때 효과가 명확합니다——이러한 향상은 느낄 수 있습니다.

### Orleans 통합

HagiCode는 Orleans를 분산 프레임워크로 사용하며, 우리는 Copilot SDK를 Orleans Grain에 통합했습니다——분산이라는 건, 말하면 복잡하지만 사용해보면 꽤 편합니다.

```csharp
// GitHubCopilotGrain.cs - 분산 실행
public async IAsyncEnumerable<GitHubCopilotResponse> ExecuteCommandStreamAsync(
    string command,
    CancellationToken token = default)
{
    var provider = await aiProviderFactory.GetProviderAsync(AIProviderType.GitHubCopilot);

    await foreach (var chunk in provider.SendMessageAsync(request, null, token))
    {
        // 통합된 응답 형식으로 매핑
        yield return BuildChunkResponse(chunk, startedAt);
    }
}
```

Orleans 통합의 장점:

- **통합된 AI Provider 추상**: 다른 AI 공급자로 쉽게 전환할 수 있습니다——오늘은 이걸 쓰고, 내일은 저걸 쓰는 것도 꽤 유연합니다.
- **다중 테넌트 격리**: 다른 사용자의 Copilot 세션이 서로 격리되어, 서로 간섭하지 않습니다.
- **지속적 세션 상태**: 세션 상태는 서버 재시작 후에도 복구할 수 있으며, 재시작해도 데이터 손실을 걱정하지 않습니다.

대량의 동시 요청을 처리해야 하는 시나리오에서 Orleans의 분산 능력은 훌륭한 확장성을 제공합니다——毕竟, 단일 머신이 감당할 수 없을 때는 분산에 의존할 수밖에 없습니다.

### 사용 시 주의사항

실제 사용에서 우리는 주의해야 할 몇 가지 사항을 요약했습니다——어떤 것들은 시행착오를 통해 얻은 경험입니다.

**시작 시간 초과 구성**: Copilot CLI를 처음 시작하는 데 오랜 시간이 걸리므로 `StartupTimeout`을 최소 30초로 설정하는 것이 좋습니다. 처음 로그인하는 경우 더 오랜 시간이 필요할 수 있습니다——毕竟 처음 로그인은 검증이 필요하니까, 어쩔 수 없습니다.

**권한 관리**: 프로덕션 환경에서는 `AllowAllTools: true` 사용을 피하세요. `AllowedTools` 화이트리스트를 사용하여 사용 가능한 도구를 제어하고, `DeniedTools` 블랙리스트를 사용하여 위험한 작업을 금지하세요. 이렇게 하면 AI가 위험한 명령을 실행하는 것을 효과적으로 방지할 수 있습니다——보안이라는 건, 조심하는 게 항상 맞습니다.

**세션 관리**: 동일한 `sessionId`의 요청은 자동으로 세션을 재사용합니다. 세션 상태는 `ProviderSessionId`를 통해 지속됩니다. 취소 작업은 `CancellationTokenSource`를 통해 전달됩니다——세션 관리를 잘 하면 경험도 자연스럽게 좋아집니다.

**진단 출력**: 호환되지 않는 CLI 매개변수는 `diagnostic` 유형 메시지를 생성합니다. 원본 SDK 이벤트는 `event.raw` 유형으로 보존됩니다. 오류 메시지에는 분류(시작 시간 초과, 매개변수 호환되지 않음 등)가 포함되어 문제 해결에 편리합니다——문제가 생겼을 때 빠르게 찾을 수 있는 것도 일종의 위안입니다.

### 모범 사례

실제 경험을 바탕으로 여기서 몇 가지 모범 사례를 공유합니다——어떻게 보면 요약이라 할 수 있겠습니다.

**1. 도구 화이트리스트 사용**

```csharp
var request = new AIRequest
{
    Prompt = "이 파일을 분석해",
    AllowedTools = new[] { "Read", "Grep", "Bash(git:*)" }
};
```

화이트리스트를 통해 명시적으로 허용된 도구를 지정하여 AI가 예기치 않은 작업을 실행하는 것을 방지하세요. 특히 쓰기 권한이 있는 도구(예: Edit)의 경우 특히 주의해야 합니다——毕竟 데이터베이스를 삭제하는 일은 누구도 겪고 싶지 않습니다.

**2. 합리적인 시간 초과 설정**

```csharp
options.Timeout = 3600;  // 1시간
options.StartupTimeout = 60;  // 1분
```

작업의 복잡도에 따라 적절한 시간 초과를 설정하세요. 너무 짧으면 작업이 중단될 수 있고, 너무 길면 응답 없는 요청을 기다리며 리소스를 낭비할 수 있습니다——모든 것은 적당히, 지나치면 오히려 손해입니다.

**3. 세션 재사용 활성화**

```csharp
options.SessionId = "my-session-123";
```

관련 작업에 동일한 sessionId를 설정하면 이전 세션 컨텍스트를 재사용하여 응답 속도를 높일 수 있습니다——컨텍스트라는 건, 때로는 꽤 중요합니다.

**4. 스트리밍 응답 처리**

```csharp
await foreach (var chunk in provider.StreamAsync(request))
{
    switch (chunk.Type)
    {
        case StreamingChunkType.ThinkingDelta:
            // 추론 과정 처리
            break;
        case StreamingChunkType.ToolCallDelta:
            // 도구 호출 처리
            break;
        case StreamingChunkType.ContentDelta:
            // 텍스트 출력 처리
            break;
    }
}
```

스트리밍 응답은 AI의 처리 진행 상황을 실시간으로 표시하여 사용자 경험을 향상시킬 수 있습니다. 특히 시간이 오래 걸리는 작업의 경우 실시간 피드백이 매우 중요합니다——진행률 표시줄을 보는 것이 그냥 기다리는 것보다 낫습니다.

**5. 오류 처리 및 재시도**

```csharp
try
{
    await foreach (var chunk in provider.StreamAsync(request))
    {
        // 응답 처리
    }
}
catch (CopilotSessionException ex)
{
    // 세션 예외 처리
    logger.LogError(ex, "Copilot session failed");
    // 예외 유형에 따라 재시도 여부 결정
}
```

적절한 오류 처리 및 재시도 메커니즘은 시스템 안정성을 높일 수 있습니다——아무도 프로그램이 영원히 오류가 없다고 보장할 수 없으니, 오류가 나면 잘 처리하면 됩니다.

## 요약

CLI 호출에서 SDK 통합으로의 업그레이드는 HagiCode 프로젝트에 상당한 가치를 가져왔습니다——怎么说呢, 이번 업그레이드는 꽤 가치 있었습니다.

- **안정성 향상**: SDK는 더 안정적인 인터페이스를 제공하며 CLI 버전 변경의 영향을 받지 않습니다——매일 버전 업데이트를 걱정할 필요가 없습니다.
- **기능 완전성**: 추론 과정과 도구 실행 상태를 포함한 완전한 이벤트 스트림을 포착할 수 있습니다——과정과 결과를 모두 볼 수 있습니다.
- **개발 효율성**: 타입 안전 SDK 인터페이스로 개발이 더 효율적이며 런타임 오류를 줄입니다——타입 검사가 있으면 마음이 편합니다.
- **사용자 경험**: 실시간 이벤트 피드백으로 사용자가 AI의 작동 과정을 더 명확하게 이해할 수 있습니다——그것이 무엇을 생각하고 있는지 아는 것이 아무것도 모르는 것보다 낫습니다.

이번 업그레이드는 단순한 기술 솔루션 교체가 아니라 전체 AI 기능 레이어 아키텍처의 최적화였습니다. 계층 설계와 추상 인터페이스를 통해 더 나은 유지 관리성과 확장성을 얻었습니다——아키텍처가 잘 되어 있으면 다음 일이 더 쉬워집니다.

만약 GitHub Copilot를 .NET 프로젝트에 통합하는 것을 고려 중이라면, 이 글의 실제 경험이 도움이 되어 더 많은 우회로를 피할 수 있기를 바랍니다. 공식 SDK는 CLI 호출보다 더 안정적이고 완전하며, 이해하고 익히는 데 시간을 투자할 가치가 있습니다——毕竟, 올바른 도구는 일을 절반하는 데 도움이 되고, 이 말도 근거가 없는 것은 아닙니다.
