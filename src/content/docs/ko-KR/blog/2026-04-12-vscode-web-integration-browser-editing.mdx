## 브라우저에서 빠르게 코드 편집하기: VSCode Web 통합 실践

> AI가 코드 분석을 완료한 후, 즉시 브라우저에서 편집기를 열어 수정하는 방법은? HagiCode 프로젝트에서 code-server를 통합한 실천 경험을 공유하며, AI 어시스턴트와 코드 편집 경험의 매끄러운 연결을 구현합니다.

## 배경

AI 지원 프로그래밍 시대에 개발자들은 자주 빠르게 코드를 보고 편집해야 합니다. 전통적인 개발 프로세스는: 데스크톱 IDE에서 프로젝트를 열고, 파일을 찾고, 편집하고, 저장합니다. 하지만 이 프로세스는 어떤 시나리오에서는 어딘가 어색한 느낌이 듭니다.

시나리오 1: 원격 개발. HagiCode와 같은 AI 어시스턴트를 사용할 때, 백엔드는 원격 서버나 컨테이너에서 실행될 수 있으며, 로컬에서 프로젝트 파일에 직접 접근할 수 없습니다. 코드를 보거나 수정할 때마다 SSH나 다른 방법으로 연결해야 하며, 경험이 단절됩니다. 이런 느낌은 마치 어떤 사람을 만나고 싶은데 두꺼운 유리를 사이에 두고 보이지만 만질 수 없는 것과 같습니다.

시나리오 2: 빠른 미리보기. AI 어시스턴트가 코드를 분석한 후, 사용자는 단순히 특정 파일을 빠르게浏览하거나 작은 수정을 하고 싶을 수 있습니다. 완전한 데스크톱 IDE를 시작하는 것은 무겁고, 브라우저 내의 가벼운 편집기가 "빠른 보기" 요구에 더 적합합니다.毕竟, 누가 한번 보기 위해 큰일을 하고 싶겠습니까?

시나리오 3: 크로스 디바이스 협업. 다른 디바이스에서 작업할 때, 브라우저의 편집기는 통합된 접근 입구를 제공하며, 모든 디바이스에 개발 환경을 설정할 필요가 없습니다. 이 또한 편리합니다.毕竟 인생은 짧고, 왜 반복 노동을 해야 합니까?

이러한痛点을 해결하기 위해 HagiCode 프로젝트에 VSCode Web을 통합했습니다. AI 어시스턴트와 코드 편집 경험을 매끄럽게 연결합니다—AI가 코드 분석을 완료한 후, 사용자는 즉시 같은 브라우저 세션에서 편집기를 열어 수정할 수 있으며, 애플리케이션 전환 없이. 이런 경험, 말하자면 당신이 생각할 때, 그게 거기에 있는 것 같습니다.

## HagiCode에 대하여

이 글에서 공유하는 방안은 [HagiCode](https://hagicode.com) 프로젝트에서의 실천 경험에서 비롯되었습니다. HagiCode는 AI 기반 코드 어시스턴트로, 자연어 상호작용을 통해 개발 효율을 높이는 것을 목표로 합니다. 개발 과정에서 우리는 사용자가 자주 AI 분석과 코드 편집 사이를 빠르게 전환해야 한다는 것을 발견했고, 이는 편집기를 브라우저에 직접 통합하는 방법을 탐색하게 했습니다.

프로젝트 주소: [github.com/HagiCode-org/site](https://github.com/HagiCode-org/site)

## 기술 선정: 왜 code-server인가?

수많은 VSCode Web 솔루션 중에서 우리는 code-server를 선택했습니다. 이 선택은 사실 몇 가지 고려가 있습니다:

기능 완전. code-server는 VSCode의 웹 버전으로, 데스크톱 버전의 대부분 기능을 지원하며, 확장 시스템, 스마트 프롬프트, 디버깅 등을 포함합니다. 즉 사용자가 브라우저에서도 데스크톱 버전에 가까운 편집 경험을 얻을 수 있습니다.毕竟, 누가 기능에서 타협하고 싶겠습니까?

배포 유연. code-server는 독립 서비스로 실행될 수 있으며, Docker 컨테이너화 배포도 지원하여 HagiCode의 아키텍처와 부합합니다. 우리의 백엔드는 C#으로 작성되었고, 프론트엔드는 React이며, REST API를 통해 code-server 서비스와 통신합니다. 이것은 블록을 쌓는 것과 같습니다. 각 블록은 자신의 위치가 있습니다.

인증 안전. code-server는 내장 connection-token 메커니즘으로 무단 접근을 방지합니다. 각 세션은 고유 token을 가지며, 권한 있는 사용자만 편집기에 접근할 수 있도록 합니다. 안전감은 있어야 중요성을 압니다.

## 아키텍처 설계

HagiCode의 VSCode Web 통합은 프론트엔드와 백엔드 분리 아키텍처 설계를 채택합니다.

### 프론트엔드 서비스 계층

프론트엔드는 `vscodeServerService.ts`를 통해 백엔드와의 상호작용을 캡슐화합니다:

```typescript
// 프로젝트 열기
export async function openProjectInCodeServer(
  id: string,
  currentInterfaceLanguage?: string,
): Promise<VsCodeServerLaunchResponseDto>

// vault 열기
export async function openVaultInCodeServer(
  id: string,
  path?: string,
  currentInterfaceLanguage?: string,
): Promise<VsCodeServerLaunchResponseDto>
```

이 두 메서드의 차이점은: `openProjectInCodeServer`는 전체 프로젝트를 열기 위해 사용되고, `openVaultInCodeServer`는 Vault의 특정 경로를 열기 위해 사용됩니다. MonoSpecs 다중 저장소 프로젝트의 경우, 시스템이 자동으로 작업 영역 파일을 생성합니다. 분명한 분업, 각자 자신의 일을 잘하면 됩니다.

### 백엔드 서비스 계층

백엔드의 `VaultAppService.cs`는 핵심 논리를 구현합니다:

```csharp
public async Task<VsCodeServerLaunchResponseDto> OpenInCodeServerAsync(
    string id,
    string? relativePath = null,
    string? currentInterfaceLanguage = null,
    CancellationToken cancellationToken = default)
{
    // 1. 설정을 가져오고 활성화 여부 확인
    var settings = await _vsCodeServerSettingsService.GetResolvedSettingsAsync(cancellationToken);
    if (!settings.Enabled) {
        throw new BusinessException(VsCodeServerErrorCodes.Disabled, "VSCode Server is disabled.");
    }

    // 2. vault를 가져오고 시작 디렉토리 해석
    var vault = await RequireVaultAsync(id, cancellationToken);
    var launchDirectory = ResolveLaunchDirectory(vault, relativePath);

    // 3. code-server 실행 및 런타임 정보 확인
    var runtime = await _vsCodeServerManager.EnsureStartedAsync(settings, cancellationToken);

    // 4. 언어 설정 해석
    var language = _vsCodeServerSettingsService.ResolveLaunchLanguage(
        settings.Language,
        currentInterfaceLanguage);

    // 5. 시작 URL 구축
    return new VsCodeServerLaunchResponseDto {
        LaunchUrl = AppendQueryString(runtime.BaseUrl, new Dictionary<string, string?> {
            ["folder"] = launchDirectory,
            ["tkn"] = runtime.ConnectionToken,
            ["vscode-lang"] = language,
        }),
        ConnectionToken = runtime.ConnectionToken,
        OpenMode = "folder",
        Runtime = VsCodeServerSettingsService.MapRuntime(
            await _vsCodeServerManager.GetRuntimeSnapshotAsync(cancellationToken)),
    };
}
```

이 메서드의 책임은 명확합니다: 설정 확인, 경로 해석, 서비스 시작, URL 구축. 그중 `ResolveLaunchDirectory` 메서드는 경로 안전 검사를 수행하며, 경로 순회 공격을 방지합니다. 코드는 시와 같습니다. 모든 줄이 자신의 운율이 있습니다.

### 자동 런타임 관리

백엔드는 `VsCodeServerManager`를 통해 code-server 프로세스를 관리합니다:

- 프로세스 상태 확인
- 중단된 서비스 자동 시작
- 런타임 스냅샷 반환(포트, 프로세스 ID, 시작 시간 등)

이 설계는 시스템이 code-server의 수명 주기를 자동으로 처리할 수 있게 하며, 사용자가 수동으로 서비스 프로세스를 관리할 필요가 없습니다.毕竟 인생은 이미 충분히 복잡하니, 자동화할 수 있는 건 자동화하세요.

### 언어 팔로우 메커니즘

HagiCode는 다국어 인터페이스를 지원하며, code-server도 이 설정을 따라야 합니다. 시스템은 세 가지 언어 모드를 지원합니다:

- `follow`: 현재 인터페이스 언어 팔로우
- `zh-CN`: 고정 중국어
- `en-US`: 고정 영어

URL 매개변수 `vscode-lang`을 통해 code-server에 전달되며, 편집기 언어가 HagiCode 인터페이스와 일치하도록 합니다. 언어는 통일되어야 편안합니다.

### MonoSpecs 다중 저장소 작업 영역

MonoSpecs 프로젝트(여러 하위 저장소를 포함하는 mono-repo)의 경우, 시스템이 자동으로 `.code-workspace` 파일을 생성합니다:

```csharp
private async Task<string> CreateWorkspaceFileAsync(Project project, Guid projectId)
{
    var folders = await ResolveWorkspaceFoldersAsync(project.Path);
    var workspaceDocument = new {
        folders = folders.Select(path => new { path }).ToArray(),
    };
    // workspace 파일 생성...
}
```

이렇게 하나의 code-server 인스턴스에서 여러 하위 저장소를 동시에 편집할 수 있으며, 대형 mono-repo 프로젝트에 매우 실용적입니다. 여러 저장소가 한 창에 있는 것은 여러 이야기가 같은 책에 있는 것과 같습니다.

## 프론트엔드 통합

HagiCode 프론트엔드는 React + TypeScript를 사용하며, code-server 통합 단계는 복잡하지 않습니다.

### 빠른 작업 버튼

프로젝트 카드에 Code Server 버튼 추가:

```tsx
// QuickActionsZone.tsx
<Button
  size="sm"
  variant="default"
  onClick={() => onAction({ type: 'open-code-server' })}
>
  <Globe className="h-3 w-3 mr-1" />
  <span className="text-xs">{t('project.openCodeServer')}</span>
</Button>
```

이 버튼은 열기 동작을 트리거하며, 백엔드 API를 호출하여 시작 URL을 가져옵니다. 하나의 버튼이 하나의 동작을 담당하며, 간단하고 직접적입니다.

### 열기 동작 처리

```tsx
const handleAction = async (action: ProjectAction) => {
  if (action.type === 'open-code-server') {
    const response = await openProjectInCodeServer(project.id, i18n.language);
    window.open(response.launchUrl, '_blank', 'noopener,noreferrer');
  }
};
```

`window.open`을 사용하여 새 탭에서 code-server를 열며, `noopener,noreferrer` 매개변수는 안전성을 보장합니다. 안전은 아무리 주의해도 지나치지 않습니다.

### Vault 편집 입구

Vault 목록에 유사한 편집 버튼 추가:

```tsx
const handleEditVault = async (vault: VaultItemDto) => {
  const response = await openVaultInCodeServer(vault.id);
  window.open(response.launchUrl, '_blank', 'noopener,noreferrer');
};
```

프로젝트와 Vault는 같은 열기 방식을 사용하며, 상호작용의 일관성을 유지합니다. 일관성의 중요성은 기능 그 자체 못지않습니다.

## URL 구축 논리

code-server의 URL 형식은 사실 약간 주의가 필요합니다:

**폴더 모드**:
```
http://{host}:{port}/?folder={path}&tkn={token}&vscode-lang={lang}
```

**작업 영역 모드**:
```
http://{host}:{port}/?workspace={workspacePath}&tkn={token}&vscode-lang={lang}
```

그중 `tkn`은 연결 token으로, code-server 시작할 때마다 자동으로 생성되며, 접근 안전을 보장합니다. `vscode-lang` 매개변수는 편집기 인터페이스 언어를 제어합니다. 이러한 매개변수는 각자의 쓰임이 있으며, 없어서는 안 됩니다.

## 사용 시나리오

### 시나리오 1: AI 지원 코드 검토

사용자가 HagiCode와 대화하며, AI가 프로젝트 코드를 분석하고 잠재적 문제를 발견합니다. 사용자가 "Open in Code Server" 버튼을 클릭하여 브라우저에서 직접 편집기를 열고, 문제 파일을 보고 수정한 후, HagiCode로 돌아가 대화를 계속합니다. 전체 프로세스가 브라우저에서 완료되며, 애플리케이션 전환 없이. 이런 느낌은 말하자면 물이 흐르는 것처럼 매끄럽습니다.

### 시나리오 2: Vault 학습 자료 편집

사용자가 오픈 소스 프로젝트 학습용 Vault를 만들고, `docs/` 디렉토리에 학습 노트를 추가하고 싶어 합니다. code-server를 통해 브라우저에서 직접 Markdown 파일을 편집하고, 저장 후 HagiCode가 업데이트된 노트 내용을 동기화하여 읽을 수 있습니다. 이것은 개인 지식 베이스 구축에 매우 유용합니다. 지식은 쌓을수록 가치가 있습니다.

### 시나리오 3: MonoSpecs 다중 저장소 개발

MonoSpecs 프로젝트는 여러 하위 저장소를 포함하며, code-server가 자동으로 다중 폴더 작업 영역을 생성합니다. 사용자가 브라우저에서 여러 저장소의 코드를 동시에 편집할 수 있으며, 수정 후 각각의 Git 저장소에 제출합니다. 이런 작업 방식은 특히 저장소 간 수정이 필요한 시나리오에 적합합니다. 여러 저장소를 함께 수정하는 것은 동시에 여러 일을 처리하는 것과 같습니다. 약간의 기술이 필요합니다.

## 안전성 고려

code-server 통합을 구현할 때, 안전성은 중점적으로 주의해야 할 문제입니다.毕竟, 안전은 문제가 생기면 늦습니다.

### Connection Token

connection-token은 무작위로 생성되며, 누출되어서는 안 됩니다. HTTPS 환경에서 사용을 권장하며, man-in-the-middle이 token을 가로채는 것을 방지합니다. 민감 정보는 보호하는 것이 좋습니다.

### 파일 경로 안전

백엔드는 경로 순회 검사를 구현합니다:

```csharp
private static string ResolveLaunchDirectory(VaultRegistryEntry vault, string? relativePath)
{
    var vaultRoot = EnsureTrailingSeparator(Path.GetFullPath(vault.PhysicalPath));
    var combinedPath = Path.GetFullPath(Path.Combine(vaultRoot, relativePath ?? "."));
    if (!combinedPath.StartsWith(vaultRoot, StringComparison.OrdinalIgnoreCase))
    {
        throw new BusinessException(VaultRelativePathTraversalCode, "Relative path traversal detected.");
    }
    return combinedPath;
}
```

이 코드는 사용자가 `../` 등의 방식으로 vault 디렉토리 외부의 파일에 접근할 수 없도록 합니다. 경계 검사는 하는 게 안 하는 게 낫습니다.

### 권한 제어

code-server 프로세스는 적절한 사용자 권한으로 실행되며, 시스템 민감 파일 접근을 피합니다. 전용 사용자로 code-server 서비스를 실행하는 것을 권장합니다. 권한 제어는 있어야 합니다.

## 성능 최적화

code-server는 서버 리소스를 소모하며, 다음은 일부 최적화 제안입니다:

- CPU/메모리 사용 모니터링, 필요 시 리소스 제한 조정
- 대형 프로젝트는 시간 초과 증가가 필요할 수 있음
- 세션 시간 초과 자동 정리 구현, 리소스 해제
- 캐시 사용으로 반복 계산 감소 고려

HagiCode는 런타임 상태 모니터링 API를 제공하며, 프론트엔드는 `getVsCodeServerSettings()`를 통해 현재 상태를 가져올 수 있습니다:

```typescript
const { settings, runtime } = await getVsCodeServerSettings();
// runtime.status: 'disabled' | 'stopped' | 'starting' | 'running' | 'unhealthy'
// runtime.baseUrl: "http://localhost:8080"
// runtime.processId: 12345
```

이 설계는 사용자가 code-server의 건강 상태를 명확히 알 수 있게 하며, 문제 발생 시 빠르게 위치를 파악합니다. 상태가 보여야 마음이 편합니다.

## 사용자 경험 세부 사항

구현 과정에서 우리는 사용자 경험에 영향을 미치는 일부 세부 사항을 발견했으며, 특별히 주의할 가치가 있습니다:

처음 code-server를 열면 시작을 기다려야 할 수 있으며, 이 시간은 몇 초에서 반분까지 다양할 수 있습니다. 프론트엔드에서 로딩 상태를 표시하는 것을 권장하며, 사용자가 시스템이 처리 중임을 알게 합니다. 기다림은 피드백이 있으면 됩니다.

브라우저가 팝업을 차단할 수 있으며, 사용자가 수동으로 허용하도록 안내해야 할 수 있습니다. HagiCode는 처음 열 때 안내 정보를 표시하며, 브라우저 권한을 설정하는 방법을 알려줍니다. 사용자 경험은 바로 이 세부 사항에서 반영됩니다.

런타임 상태(시작 중/실행 중/오류)를 표시하는 것을 권장하며, 이렇게 하면 사용자가 문제를 만났을 때 서버 측 문제인지 자신의 작업 문제인지 알 수 있습니다. 문제가 어디에 있는지 알면 최마 마음이 편합니다.

## 구성 예시

code-server의 구성은 복잡하지 않습니다:

```json
{
  "vscodeServer": {
    "enabled": true,
    "host": "0.0.0.0",
    "port": 8080,
    "language": "follow"
  }
}
```

`enabled`는 기능 스위치를 제어하고, `host`와 `port`는 수신 주소를 지정하며, `language`는 언어 모드를 설정합니다. 이러한 구성은 UI 인터페이스를 통해 수정할 수 있으며, 실시간으로 유효합니다. 간단한 것이 종종 가장 잘 쓰입니다.

## 요약

HagiCode의 VSCode Web 통합은 우아한 솔루션을 제공합니다: AI 어시스턴트와 코드 편집 경험을 매끄럽게 연결합니다. 브라우저에 code-server를 통합함으로써, 사용자는 AI의 분석 결과에 빠르게 응답할 수 있으며, 같은 브라우저 세션에서 분석부터 편집까지의 완전한 프로세스를 완료할 수 있습니다.

이 방안의 몇 가지 핵심 장점: 첫째, 통합 경험, 프로젝트와 Vault는 같은 열기 방식을 사용합니다. 둘째, 다중 저장소 지원, MonoSpecs 프로젝트는 자동으로 작업 영역을 생성합니다. 셋째, 안전하고 제어 가능, 런타임 상태 모니터링과 경로 안전 검사.

이 글에서 공유하는 방안은 HagiCode가 실제 개발에서 요약한 것입니다. 이 방안이 가치 있다고 생각하면, 우리의 엔지니어링 실천이 나쁘지 않다는 것을 의미합니다—그렇다면 HagiCode 자체도 주목할 가치가 있습니다.毕竟, 좋은 것은 더 많은 사람이 볼 자격이 있습니다.
