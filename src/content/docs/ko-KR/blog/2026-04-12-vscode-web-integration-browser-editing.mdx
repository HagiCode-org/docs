## 브라우저에서 빠른 코드 편집: VSCode 웹 통합 실전

> AI가 코드 분석을 마친 후, 브라우저에서 즉시 에디터를 열고 변경을 시작하려면 어떻게 해야 할까요? 이 글에서는 HagiCode 프로젝트에서 code-server를 통합하여 AI 어시스턴트와 코드 편집 경험 간의 매끄러운 연결을 만든 실무 경험을 공유합니다.

## 배경

AI 보조 프로그래밍 시대에 개발자는 코드를 빠르게 검사하고 편집해야 하는 경우가 많습니다. 전통적인 워크플로우는 간단합니다. 데스크톱 IDE에서 프로젝트를 열고, 파일을 찾고, 편집하고, 저장합니다. 하지만 어떤 상황에서는 이 흐름이 항상 약간 어색하게 느껴집니다.

시나리오 하나: 원격 개발. HagiCode와 같은 AI 어시스턴트를 사용할 때 백엔드가 원격 서버나 컨테이너 내에서 실행될 수 있으며, 로컬 머신에서 프로젝트 파일에 직접 접근할 수 없습니다. 코드를 검사하거나 수정할 때마다 SSH 또는 다른 방법으로 연결해야 하며, 경험이 단절됩니다. 두꺼운 유리를 통해 누군가를 만나려는 것과 같습니다. 보이지만 닿을 수 없습니다.

시나리오 둘: 빠른 미리보기. AI 어시스턴트가 코드를 분석한 후 사용자가 파일을 빠르게 탐색하거나 작은 변경을 하고 싶을 수 있습니다. 전체 데스크톱 IDE를 실행하는 것은 무겁게 느껴지지만, 가벼운 인브라우저 에디터는 "빠르게 보기"의 필요에 더 잘 맞습니다. 한 번 보기 위해 전체 도구 체인을 동원하고 싶은 사람이 어디 있겠습니까?

시나리오 셋: 크로스 디바이스 협업. 여러 기기에서 작업할 때 브라우저 기반 에디터는 통합된 접근점을 제공하며, 모든 머신에 개발 환경을 구성할 필요가 없습니다. 그것만으로도 많은 번거로움이 줄어듭니다. 인생은 짧습니다. 같은 설정 작업을 반복할 필요가 있겠습니까?

이러한 문제점들을 해결하기 위해 HagiCode 프로젝트에 VSCode 웹을 통합했습니다. 이를 통해 AI 어시스턴트와 코드 편집 경험이 매끄럽게 연결됩니다. AI가 코드를 분석한 후 사용자는 동일한 브라우저 세션에서 즉시 에디터를 열고 변경할 수 있으며, 애플리케이션을 전환할 필요가 없습니다. 필요할 때 바로 거기에 있는 경험입니다.

## HagiCode 소개

이 글에서 공유하는 해결책은 [HagiCode](https://hagicode.com) 프로젝트에서의 실무 경험에서 나온 것입니다. HagiCode는 자연어 상호 작용을 통해 개발 효율을 높이도록 설계된 AI 기반 코딩 어시스턴트입니다. 개발 과정에서 사용자가 AI 분석과 코드 편집 사이를 빠르게 전환해야 한다는 것을 발견했으며, 이것이 에디터를 브라우저에 직접 통합하는 방법을 탐색하게 된 계기입니다.

## 기술 선택: 왜 code-server인가?

여러 VSCode 웹 솔루션 중에서 code-server를 선택했습니다. 그 결정 뒤에는 몇 가지 구체적인 이유가 있었습니다.

기능 완전성. code-server는 VSCode의 웹 버전이며 확장 시스템, 스마트 제안, 디버깅 등 대부분의 데스크톱 기능을 지원합니다. 즉, 사용자가 브라우저에서 데스크톱 버전에 매우 가까운 편집 경험을 얻을 수 있습니다. 누가 기능을 타협하고 싶겠습니까?

유연한 배포. code-server는 독립적인 서비스로 실행할 수 있으며 Docker 기반 배포도 지원합니다. HagiCode의 아키텍처에 잘 맞습니다. 백엔드는 C#으로 작성되고, 프론트엔드는 React를 사용하며, 둘 다 REST API를 통해 code-server 서비스와 통신합니다. 블록으로 건축하는 것과 같습니다. 각 조각이 제자리에 있습니다.

안전한 인증. code-server에는 내장된 `connection-token` 메커니즘이 포함되어 있어 승인되지 않은 접근을 방지합니다. 각 세션에는 고유한 토큰이 있으므로 승인된 사용자만 에디터를 열 수 있습니다. 보안은 가질 때만 완전히 이해할 수 있는 것 중 하나입니다.

## 아키텍처 설계

HagiCode의 VSCode 웹 통합은 프론트엔드-백엔드 분리 아키텍처를 사용합니다.

### 프론트엔드 서비스 레이어

프론트엔드는 `vscodeServerService.ts`를 통해 백엔드와의 상호 작용을 래핑합니다.

```typescript
// 프로젝트 열기
export async function openProjectInCodeServer(
  id: string,
  currentInterfaceLanguage?: string,
): Promise<VsCodeServerLaunchResponseDto>

// Vault 열기
export async function openVaultInCodeServer(
  id: string,
  path?: string,
  currentInterfaceLanguage?: string,
): Promise<VsCodeServerLaunchResponseDto>
```

이 두 메서드의 차이는 간단합니다. `openProjectInCodeServer`는 전체 프로젝트를 열고, `openVaultInCodeServer`는 Vault 내의 특정 경로를 엽니다. MonoSpecs 다중 저장소 프로젝트의 경우 시스템이 자동으로 워크스페이스 파일을 생성합니다. 각 부분이 자기 일을 잘하면 명확한 책임 분할만으로 충분합니다.

### 백엔드 서비스 레이어

백엔드 `VaultAppService.cs`는 핵심 로직을 구현합니다.

```csharp
public async Task<VsCodeServerLaunchResponseDto> OpenInCodeServerAsync(
    string id,
    string? relativePath = null,
    string? currentInterfaceLanguage = null,
    CancellationToken cancellationToken = default)
{
    // 1. 설정을 가져오고 기능이 활성화되어 있는지 확인
    var settings = await _vsCodeServerSettingsService.GetResolvedSettingsAsync(cancellationToken);
    if (!settings.Enabled) {
        throw new BusinessException(VsCodeServerErrorCodes.Disabled, "VSCode Server is disabled.");
    }

    // 2. Vault를 가져오고 실행 디렉토리 해석
    var vault = await RequireVaultAsync(id, cancellationToken);
    var launchDirectory = ResolveLaunchDirectory(vault, relativePath);

    // 3. code-server가 실행 중인지 확인하고 런타임 정보 가져오기
    var runtime = await _vsCodeServerManager.EnsureStartedAsync(settings, cancellationToken);

    // 4. 언어 설정 해석
    var language = _vsCodeServerSettingsService.ResolveLaunchLanguage(
        settings.Language,
        currentInterfaceLanguage);

    // 5. 실행 URL 구성
    return new VsCodeServerLaunchResponseDto {
        LaunchUrl = AppendQueryString(runtime.BaseUrl, new Dictionary<string, string?> {
            ["folder"] = launchDirectory,
            ["tkn"] = runtime.ConnectionToken,
            ["vscode-lang"] = language,
        }),
        ConnectionToken = runtime.ConnectionToken,
        OpenMode = "folder",
        Runtime = VsCodeServerSettingsService.MapRuntime(
            await _vsCodeServerManager.GetRuntimeSnapshotAsync(cancellationToken)),
    };
}
```

이 메서드는 매우 명확한 책임을 가집니다. 설정 확인, 경로 해석, 서비스 시작, URL 구성. 그 중 `ResolveLaunchDirectory` 메서드는 경로 순회 공격을 방지하기 위해 경로 보안 검사를 수행합니다. 모든 줄에 목적이 있을 때 코드는 시처럼 느껴질 수 있습니다.

### 자동 런타임 관리

백엔드는 `VsCodeServerManager`를 통해 code-server 프로세스를 관리합니다.

- 프로세스 상태 확인
- 중지된 서비스 자동 시작
- 포트, 프로세스 ID, 시작 시간과 같은 런타임 스냅샷 반환

이 설계를 통해 시스템이 code-server 수명 주기를 자동으로 처리하므로 사용자가 서비스 프로세스를 수동으로 관리할 필요가 없습니다. 삶은 이미 충분히 복잡합니다. 자동화할 수 있는 것은 자동화해야 합니다.

### 언어 동기화

HagiCode는 다국어 인터페이스를 지원하며, code-server도 이 설정을 따라야 합니다. 시스템은 세 가지 언어 모드를 지원합니다.

- `follow`: 현재 인터페이스 언어를 따름
- `zh-CN`: 중국어로 고정
- `en-US`: 영어로 고정

이 설정은 `vscode-lang` URL 매개변수를 통해 code-server에 전달되어 에디터 언어가 HagiCode 인터페이스와 일치하도록 합니다. 언어는 통일되었을 때 가장 좋습니다.

### MonoSpecs 다중 저장소 워크스페이스

MonoSpecs 프로젝트의 경우 하나의 모노레포 내에 여러 하위 저장소가 포함되어 있으며, 시스템이 자동으로 `.code-workspace` 파일을 생성합니다.

```csharp
private async Task<string> CreateWorkspaceFileAsync(Project project, Guid projectId)
{
    var folders = await ResolveWorkspaceFoldersAsync(project.Path);
    var workspaceDocument = new {
        folders = folders.Select(path => new { path }).ToArray(),
    };
    // 워크스페이스 파일 생성...
}
```

이를 통해 동일한 code-server 인스턴스에서 여러 하위 저장소를 편집할 수 있으며, 이는 특히 대형 모노레포 프로젝트에 실용적입니다. 한 창에 여러 저장소가 있는 것은 한 책에 여러 이야기가 모인 것과 같습니다.

## 프론트엔드 통합

HagiCode 프론트엔드는 React + TypeScript를 사용하며, code-server 통합은 특별히 복잡하지 않습니다.

### 빠른 실행 버튼

프로젝트 카드에 Code Server 버튼을 추가합니다.

```tsx
// QuickActionsZone.tsx
<Button
  size="sm"
  variant="default"
  onClick={() => onAction({ type: 'open-code-server' })}
>
  <Globe className="h-3 w-3 mr-1" />
  <span className="text-xs">{t('project.openCodeServer')}</span>
</Button>
```

이 버튼은 열기 동작을 트리거하고 백엔드 API를 호출하여 실행 URL을 가져옵니다. 하나의 버튼, 하나의 동작, 직접적이고 간단합니다.

### 열기 동작 처리

```tsx
const handleAction = async (action: ProjectAction) => {
  if (action.type === 'open-code-server') {
    const response = await openProjectInCodeServer(project.id, i18n.language);
    window.open(response.launchUrl, '_blank', 'noopener,noreferrer');
  }
};
```

`window.open`을 사용하여 code-server를 새 탭에서 엽니다. `noopener,noreferrer` 매개변수는 추가 보안을 제공합니다. 보안에 있어서는 너무 조심하는 것은 없습니다.

### Vault 편집 진입점

Vault 목록에 유사한 편집 버튼을 추가합니다.

```tsx
const handleEditVault = async (vault: VaultItemDto) => {
  const response = await openVaultInCodeServer(vault.id);
  window.open(response.launchUrl, '_blank', 'noopener,noreferrer');
};
```

프로젝트와 Vault는 동일한 열기 메커니즘을 사용하여 상호 작용의 일관성을 유지합니다. 일관성은 기능 자체만큼이나 중요합니다.

## URL 구성 로직

code-server의 URL 형식에는 주의할 만한 몇 가지 세부 사항이 있습니다.

**폴더 모드**:
```
http://{host}:{port}/?folder={path}&tkn={token}&vscode-lang={lang}
```

**워크스페이스 모드**:
```
http://{host}:{port}/?workspace={workspacePath}&tkn={token}&vscode-lang={lang}
```

여기서 `tkn`은 연결 토큰입니다. code-server가 시작될 때마다 자동으로 생성되어 안전한 접근을 보장합니다. `vscode-lang` 매개변수는 에디터 UI 언어를 제어합니다. 이러한 매개변수 각각이 역할을 가지고 있습니다.

## 사용 시나리오

### 시나리오 1: AI 보조 코드 리뷰

사용자가 HagiCode와 대화하고, AI가 프로젝트 코드를 분석하여 잠재적 문제를 발견한 다음, 사용자가 "Code Server에서 열기" 버튼을 클릭하여 브라우저에서 직접 에디터를 열고, 영향을 받는 파일을 검사하고, 수정한 다음 HagiCode로 돌아가 대화를 계속합니다. 전체 흐름이 브라우저에서 이루어지며 애플리케이션 전환이 필요 없습니다. 흐르는 물이 매끄러운 것처럼 매끄럽습니다.

### 시나리오 2: Vault에서 학습 자료 편집

사용자가 오픈소스 프로젝트를 연구하기 위한 Vault를 만들고, `docs/` 디렉토리 아래에 학습 메모를 추가하고 싶어 합니다. code-server를 사용하면 브라우저에서 Markdown 파일을 직접 편집하고, 저장한 다음 HagiCode가 업데이트된 메모를 즉시 읽을 수 있도록 할 수 있습니다. 이것은 개인 지식 베이스를 구축하는 데 특히 유용합니다. 지식은 축적할수록 더 가치 있어집니다.

### 시나리오 3: MonoSpecs 다중 저장소 개발

MonoSpecs 프로젝트에는 여러 하위 저장소가 포함되어 있으며, code-server는 자동으로 다중 폴더 워크스페이스를 생성합니다. 브라우저에서 사용자는 여러 저장소의 코드를 한 번에 편집한 다음 변경 사항을 각각의 Git 저장소에 커밋할 수 있습니다. 이 워크플로우는 여러 저장소에 걸쳐야 하는 변경에 특히 잘 맞습니다. 여러 저장소를 동시에 편집하는 것은 여러 작업을 동시에 처리하는 것과 같이 약간의 기술이 필요합니다.

## 보안 고려 사항

code-server 통합을 구현할 때 보안은 특별한 주의가 필요합니다. 보안이 잘못되면 항상 너무 늦게 알게 됩니다.

### 연결 토큰

`connection-token`은 무작위로 생성되며 노출되어서는 안 됩니다. 중간자 공격으로 토큰이 가로채이는 것을 방지하기 위해 HTTPS 하에서 사용하는 것이 가장 좋습니다. 민감한 정보는 제대로 보호할 가치가 있습니다.

### 파일 경로 보안

백엔드는 경로 순회 검사를 구현합니다.

```csharp
private static string ResolveLaunchDirectory(VaultRegistryEntry vault, string? relativePath)
{
    var vaultRoot = EnsureTrailingSeparator(Path.GetFullPath(vault.PhysicalPath));
    var combinedPath = Path.GetFullPath(Path.Combine(vaultRoot, relativePath ?? "."));
    if (!combinedPath.StartsWith(vaultRoot, StringComparison.OrdinalIgnoreCase))
    {
        throw new BusinessException(VaultRelativePathTraversalCode, "Relative path traversal detected.");
    }
    return combinedPath;
}
```

이 코드는 사용자가 `../` 또는 유사한 패턴을 사용하여 Vault 디렉토리 외부의 파일에 접근할 수 없도록 보장합니다. 경계 검사는 건너뛰는 것보다 항상 하는 것이 낫습니다.

### 권한 제어

code-server 프로세스는 적절한 사용자 권한으로 실행되어야 하며, 중요한 시스템 파일에 접근할 수 없어야 합니다. code-server 서비스는 전용 사용자로 실행하는 것이 가장 좋습니다. 권한 제어는 항상 유지해야 할 기본 중 하나입니다.

## 성능 최적화

code-server는 서버 리소스를 소비하므로 몇 가지 최적화 제안이 있습니다.

- CPU 및 메모리 사용량 모니터링, 필요에 따라 리소스 제한 조정
- 큰 프로젝트는 더 긴 시간 초과가 필요할 수 있습니다
- 자동 세션 시간 초과 정리를 구현하여 리소스 해제
- 캐싱을 고려하여 반복 계산 감소

HagiCode는 런타임 상태 모니터링 API를 제공하며, 프론트엔드는 `getVsCodeServerSettings()`를 호출하여 현재 상태를 검색할 수 있습니다.

```typescript
const { settings, runtime } = await getVsCodeServerSettings();
// runtime.status: 'disabled' | 'stopped' | 'starting' | 'running' | 'unhealthy'
// runtime.baseUrl: "http://localhost:8080"
// runtime.processId: 12345
```

이 설계를 통해 사용자는 code-server의 상태를 명확하게 이해하고 문제가 발생하면 빠르게 원인을 파악할 수 있습니다. 상태가 보일 때 사람들은 더 많은 통제감을 느낍니다.

## 사용자 경험 세부 사항

구현 과정에서 사용자 경험에 눈에 띄게 영향을 미치는 몇 가지 세부 사항을 발견했습니다.

code-server를 처음 열면 시작 대기가 필요할 수 있으며, 그 지연은 몇 초에서 최대 30초까지 다양합니다. 프론트엔드에 로딩 상태를 표시하여 시스템이 여전히 작동 중임을 사용자가 알도록 하는 것이 좋습니다. 피드백이 있으면 기다리기가 더 쉽습니다.

브라우저가 팝업을 차단할 수 있으므로 사용자에게 수동으로 허용하도록 안내해야 합니다. 첫 번째 실행 시 HagiCode는 필요한 브라우저 권한을 부여하는 방법을 설명하는 안내를 표시합니다. 사용자 경험은 종종 이러한 작은 세부 사항에 있습니다.

시작 중, 실행 중, 오류와 같은 런타임 상태를 표시하는 것도 좋은 아이디어입니다. 문제가 발생했을 때 사용자가 문제가 서버 측인지 자신의 조작 때문인지 빠르게 파악할 수 있습니다. 문제가 어디에 있는지 아는 것만으로도 시작점이 생깁니다.

## 설정 예시

code-server의 설정은 복잡하지 않습니다.

```json
{
  "vscodeServer": {
    "enabled": true,
    "host": "0.0.0.0",
    "port": 8080,
    "language": "follow"
  }
}
```

`enabled`는 기능 켜기/끄기를 제어하고, `host`와 `port`는 수신 주소를 정의하며, `language`는 언어 모드를 설정합니다. 이 설정은 UI를 통해 수정할 수 있으며 즉시 적용됩니다. 간단한 것이 종종 가장 사용하기 쉽습니다.

## 결론

HagiCode의 VSCode 웹 통합은 우아한 해결책을 제공합니다. AI 어시스턴트와 코드 편집 경험을 매끄럽게 연결합니다. code-server를 브라우저에 통합함으로써 사용자는 AI 분석 결과에 즉시 조치하고, 동일한 브라우저 세션에서 분석부터 편집까지 전체 흐름을 완료할 수 있습니다.

이 해결책은 몇 가지 핵심 장점을 가집니다. 통합된 경험 (프로젝트와 Vault가 동일한 열기 메커니즘을 사용), 다중 저장소 지원 (MonoSpecs 프로젝트가 자동으로 워크스페이스를 생성), 제어 가능한 보안 (런타임 상태 모니터링 및 경로 안전 검사).

이 글에서 공유한 접근 방식은 HagiCode가 실제 개발 작업에서 정제한 것입니다. 이 해결책이 가치 있다고 생각하신다면, 우리의 엔지니어링 실무가 올바른 방향이며, HagiCode 자체도 더 주목할 가치가 있다는 좋은 신호입니다. 좋은 도구는 더 많은 사람들이 볼 가치가 있습니다.
