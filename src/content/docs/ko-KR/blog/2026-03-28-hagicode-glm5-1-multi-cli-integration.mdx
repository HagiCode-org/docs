## Hagicode와 GLM-5.1 다중 CLI 통합 가이드

:::note[시간적 맥락 설명]
이 문서는 릴리스 당시의 다중 CLI 연결 설계와 모델 매핑을 기록한 것입니다. 현재 지원 범위는 [AI Agent CLI 설치](/related-software-installation/ai-agent-cli/) 및 [설치 가이드](/installation)를 기준으로 하며, 문서의 `IFlowCli`는 당시의 역사적 구성만 나타내며 현재 활성 지원 범위에 속하지 않습니다.
:::

## 배경

Hagicode 프로젝트에서 사용자는 Claude Code CLI, GitHub Copilot, OpenCode CLI, Codebuddy CLI, Hermes CLI 등 다양한 CLI 도구를 선택하여 AI 프로그래밍 어시스턴트를 구동할 수 있습니다. 이러한 CLI 도구는 본래 범용 AI 프로그래밍 도구이지만, Hagicode의 추상화 계층을 통해 다양한 AI 모델 공급자에게 유연하게 연결할 수 있습니다.

지푸 AI(ZAI)는 Anthropic Claude API와 호환되는 인터페이스를 제공하여 이러한 CLI 도구가 국산 GLM 시리즈 모델을 직접 사용할 수 있게 합니다. 그중 GLM-5.1은 지푸에서 최신으로 출시한 대규모 언어 모델로, GLM-5.0에 비해 현저한 개선이 있습니다.

## Hagicode의 CLI 추상화 아키텍처

Hagicode는 `AIProviderType` 열거형을 통해 11가지 CLI 공급자 유형을 정의하여 주요 AI 프로그래밍 CLI 도구를 포함합니다:

```csharp
public enum AIProviderType
{
    ClaudeCodeCli = 0,    // Claude Code CLI
    CodexCli = 1,          // GitHub Copilot Codex
    GitHubCopilot = 2,     // GitHub Copilot
    CodebuddyCli = 3,     // Codebuddy CLI
    OpenCodeCli = 4,      // OpenCode CLI
    IFlowCli = 5,         // IFlow CLI
    HermesCli = 6,        // Hermes CLI
    QoderCli = 7,         // Qoder CLI
    KiroCli = 8,          // Kiro CLI
    KimiCli = 9,          // Kimi CLI
    GeminiCli = 10,       // Gemini CLI
}
```

각 CLI는 해당 모델 매개변수 구성이 있으며 `model` 및 `reasoning` 매개변수를 지원합니다:

```csharp
private static readonly IReadOnlyDictionary<AIProviderType, IReadOnlyList<string>> ManagedModelParameterKeysByProvider =
    new Dictionary<AIProviderType, IReadOnlyList<string>>
    {
        [AIProviderType.ClaudeCodeCli] = ["model", "reasoning"],
        [AIProviderType.CodexCli] = ["model", "reasoning"],
        [AIProviderType.OpenCodeCli] = ["model", "reasoning"],
        [AIProviderType.HermesCli] = ["model", "reasoning"],
        [AIProviderType.CodebuddyCli] = ["model", "reasoning"],
        [AIProviderType.QoderCli] = ["model", "reasoning"],
        [AIProviderType.KiroCli] = ["model", "reasoning"],
        [AIProviderType.GeminiCli] = ["model"],  // Gemini는 reasoning 매개변수를 지원하지 않음
        // ...
    };
```

## GLM 모델 지원 체계

Hagicode의 Secondary Professions Catalog는 완전한 GLM 시리즈 모델 지원을 정의합니다:

| Model ID | Name | Default Reasoning | Compatible CLI Families |
|----------|------|-------------------|-------------------------|
| `glm-4.7` | GLM 4.7 | high | claude, codebuddy, hermes, qoder, kiro |
| `glm-5` | GLM 5 | high | claude, codebuddy, hermes, qoder, kiro |
| `glm-5-turbo` | GLM 5 Turbo | high | claude, codebuddy, hermes, qoder, kiro |
| `glm-5.0` | GLM 5.0 (Legacy) | high | claude, codebuddy, hermes, qoder, kiro |
| `glm-5.1` | GLM 5.1 | high | claude, codebuddy, hermes, qoder, kiro |

## GLM-5.1 vs GLM-5.0의 핵심 차이점

`AcpSessionModelBootstrapper.cs` 구현에서 GLM-5.1과 GLM-5.0의 차이를 명확히 볼 수 있습니다:

### GLM-5.1의 독립 구현

GLM-5.1은 독립적인 새 모델 식별자이며, 레거시 처리 논리가 전혀 없습니다:

```csharp
private const string Glm51ModelValue = "glm-5.1";
```

Secondary Professions Catalog의 정의:

```json
{
  "id": "secondary-glm-5-1",
  "name": "GLM 5.1",
  "family": "anthropic",
  "summary": "hero.professionCopy.secondary.glm51.summary",
  "sourceLabel": "hero.professionCopy.sources.aiSharedAnthropicModel",
  "sortOrder": 64,
  "supportsImage": true,
  "compatiblePrimaryFamilies": [
    "claude",
    "codebuddy",
    "hermes",
    "qoder",
    "kiro"
  ],
  "defaultParameters": {
    "model": "glm-5.1",
    "reasoning": "high"
  }
}
```

## 다양한 CLI에서 GLM-5.1 사용

### 1. Claude Code CLI + GLM-5.1

Claude Code CLI는 Hagicode의 핵심 CLI 중 하나로, Hero 구성 시스템을 통해 설정합니다:

```json
{
  "primaryProfessionId": "profession-claude-code",
  "secondaryProfessionId": "secondary-glm-5-1",
  "model": "glm-5.1",
  "reasoning": "high"
}
```

해당 `HeroEquipmentCatalogItem` 구성:

```typescript
{
  id: 'secondary-glm-5-1',
  name: 'GLM 5.1',
  family: 'anthropic',
  kind: 'model',
  primaryFamily: 'claude',
  compatiblePrimaryFamilies: ['claude', 'codebuddy', 'hermes', 'qoder', 'kiro'],
  defaultParameters: {
    model: 'glm-5.1',
    reasoning: 'high'
  }
}
```

### 2. OpenCode CLI + GLM-5.1

OpenCode CLI는 가장 유연한 CLI로, `provider/model` 형식을 사용하여 임의의 모델을 지정할 수 있습니다:

**방법 1: ZAI provider 접두사 사용**
```json
{
  "primaryProfessionId": "profession-opencode",
  "model": "zai/glm-5.1",
  "reasoning": "high"
}
```

**방법 2: 모델 ID 직접 사용**
```json
{
  "model": "glm-5.1"
}
```

**방법 3: 프론트엔드 구성 인터페이스**

`HeroModelEquipmentForm.tsx`에서 OpenCode CLI에는 특수한 플레이스홀더 힌트가 있습니다:

```tsx
const OPEN_CODE_MODEL_PLACEHOLDER = 'myprovider/glm-4.7';

const modelPlaceholder = primaryProviderType === PCode_Models_AIProviderType.OPEN_CODE_CLI
  ? OPEN_CODE_MODEL_PLACEHOLDER
  : 'gpt-5.4';
```

사용자는 다음을 입력할 수 있습니다:
```
zai/glm-5.1
glm-5.1
```

**OpenCode CLI의 모델解析 로직**:

```csharp
internal OpenCodeModelSelection? ResolveModelSelection(string? rawModel)
{
    var normalized = NormalizeOptionalValue(rawModel);
    if (normalized == null) return null;

    var slashIndex = normalized.IndexOf('/');
    if (slashIndex < 0)
    {
        // 슬래시 없음: 모델 ID를 직접 사용
        return new OpenCodeModelSelection {
            ProviderId = string.Empty,
            ModelId = normalized,
        };
    }

    // 슬래시 있음: provider/model 형식解析
    var providerId = normalized[..slashIndex].Trim();
    var modelId = normalized[(slashIndex + 1)..].Trim();

    return new OpenCodeModelSelection {
        ProviderId = providerId,
        ModelId = modelId,
    };
}
```

### 3. Codebuddy CLI + GLM-5.1

Codebuddy CLI에는 특수한 레거시 처리 논리가 있습니다:

```json
{
  "primaryProfessionId": "profession-codebuddy",
  "model": "glm-5.1",
  "reasoning": "high"
}
```

**참고**: Codebuddy는 GLM-5.0에 대해 특수 예약을 하며, 레거시 정규화를 수행하지 않습니다:

```csharp
return !string.Equals(providerName, "CodebuddyCli", StringComparison.OrdinalIgnoreCase)
       && string.Equals(normalizedModel, LegacyGlm5TurboModelValue, StringComparison.OrdinalIgnoreCase)
    ? Glm5TurboModelValue
    : normalizedModel;
// CodebuddyCli일 때 glm-5.0은 glm-5-turbo로 정규화되지 않음
```

## GLM-5.1의 개선 이점

GLM-5.0에 비해 GLM-5.1은 다음과 같은 현저한 개선이 있습니다:

### 1. 더 나은 추론 능력

지푸 공식 발표 정보에 따르면 GLM-5.1의 개선사항은 다음과 같습니다:

- **더 강력한 코드 이해 능력**: 복잡한 코드 구조의 해석이 더 정확함
- **더 긴 컨텍스트 이해**: 더 긴 대화 컨텍스트 지원
- **도구 호출 강화**: MCP 도구 호출 성공률 향상
- **출력 안정성**: 무작위성과 환각 감소

### 2. 다중 CLI 전면 호환

GLM-5.1은 Hagicode가 지원하는 모든 주요 CLI를 포괄합니다:

```
compatiblePrimaryFamilies: [
  "claude",      // Claude Code CLI
  "codebuddy",   // Codebuddy CLI
  "hermes",      // Hermes CLI
  "qoder",       // Qoder CLI
  "kiro"         // Kiro CLI
]
```

## 주의사항

### 1. API Key 구성

모든 CLI가 모델에 연결하는 데 필요한 자격 증명인 `ANTHROPIC_AUTH_TOKEN` 환경 변수를 올바르게 설정했는지 확인하세요.

### 2. 모델 가용성

GLM-5.1은 해당 모델 공급자에서 개통해야 합니다:
- 지푸 AI ZAI 플랫폼은 기본 지원
- 알리클라우드 DashScope는 별도 신청이 필요할 수 있음

### 3. OpenCode CLI 형식

`provider/model` 형식을 사용할 때 provider ID가 올바른지 확인하세요:
- 지푸 AI: `zai` 또는 `zhipuai`
- 알리클라우드: `aliyun` 또는 `dashscope`

### 4. Reasoning 매개변수

- 최적의 코드 생성 효과를 얻으려면 `high` 수준 사용을 권장합니다
- Gemini CLI는 reasoning 매개변수를 지원하지 않으며 이 구성을 자동으로 무시합니다

## 요약

Hagicode는 통합된 추상화 계층을 통해 GLM-5.1과 다양한 CLI의 유연한 통합을 구현했습니다. 개발자는 자신의 선호도와 사용 시나리오에 따라 적절한 CLI 도구를 선택하고 간단한 구성을 통해 최신 GLM-5.1 모델을 사용할 수 있습니다.

지푸의 최신 모델 버전인 GLM-5.1은 GLM-5.0에 비해 명확한 개선이 있습니다:
- 독립적인 버전 식별자, 레거시 부담 없음
- 더 강력한 추론 능력과 코드 이해
- 광범위한 다중 CLI 호환성
- 유연한 추론 수준 구성

올바른 환경 변수와 Hero 장비 구성을 통해 플레이어는 다양한 CLI 환경에서 GLM-5.1의 강력한 능력을 충분히 발휘할 수 있습니다.

- 로컬 데스크톱 경험을 선호한다면 Desktop 진입점에서 바로 시작: [hagicode.com/desktop/](https://hagicode.com/desktop/)

Kimi, Claude Code, OpenCode와 같은 CLI와 GLM-5.1을 동일한 추상화 계층에 놓고 비교할 때 모델 전환, 매개변수 매핑 및 엔지니어링 경계에 대한 많은 질문이 한눈에 명확해질 것입니다.
