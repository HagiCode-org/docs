## Hagicode 결합 GLM-5.1 다중 CLI 통합 가이드

:::note[시간적 맥락 설명]
이 글은 발행 당시의 다중 CLI 접근 설계 및 모델 매핑을 기록합니다. 현재 상주 지원 범위는 [AI Agent CLI 설치](/related-software-installation/ai-agent-cli/) 및 [설치 가이드](/installation)를 기준으로 합니다. 본문의 `IFlowCli`은 당시의 역사적 설정만 대표하며, 현재 활성 지원 범위에 속하지 않습니다.
:::

## 배경

Hagicode 프로젝트에서 사용자는 Claude Code CLI, GitHub Copilot, OpenCode CLI, Codebuddy CLI, Hermes CLI 등 여러 CLI 도구를 선택하여 AI 프로그래밍 어시스턴트를 구동할 수 있습니다. 이러한 CLI 도구 자체는 범용 AI 프로그래밍 도구이지만, Hagicode의 추상화 레이어를 통해 다양한 AI 모델 제공자에 유연하게 접속할 수 있습니다.

智譜 AI(ZAI)는 Anthropic Claude API와 호환되는 인터페이스를 제공하여, 이러한 CLI 도구가 국산 GLM 시리즈 모델을 직접 사용할 수 있게 합니다. 그중 GLM-5.1은 智譜가 최근 출시한 대형 언어 모델로, GLM-5.0에 비해 상당한 개선이 있습니다.

## Hagicode의 CLI 추상 아키텍처

Hagicode는 `AIProviderType` 열거형으로 11종 CLI 제공자 유형을 정의하여, 주류 AI 프로그래밍 CLI 도구를 포괄합니다:

```csharp
public enum AIProviderType
{
    ClaudeCodeCli = 0,    // Claude Code CLI
    CodexCli = 1,          // GitHub Copilot Codex
    GitHubCopilot = 2,     // GitHub Copilot
    CodebuddyCli = 3,     // Codebuddy CLI
    OpenCodeCli = 4,      // OpenCode CLI
    IFlowCli = 5,         // IFlow CLI
    HermesCli = 6,        // Hermes CLI
    QoderCli = 7,         // Qoder CLI
    KiroCli = 8,          // Kiro CLI
    KimiCli = 9,          // Kimi CLI
    GeminiCli = 10,       // Gemini CLI
}
```

각 CLI에는 해당하는 모델 매개변수 설정이 있으며, `model` 및 `reasoning` 매개변수를 지원합니다:

```csharp
private static readonly IReadOnlyDictionary<AIProviderType, IReadOnlyList<string>> ManagedModelParameterKeysByProvider =
    new Dictionary<AIProviderType, IReadOnlyList<string>>
    {
        [AIProviderType.ClaudeCodeCli] = ["model", "reasoning"],
        [AIProviderType.CodexCli] = ["model", "reasoning"],
        [AIProviderType.OpenCodeCli] = ["model", "reasoning"],
        [AIProviderType.HermesCli] = ["model", "reasoning"],
        [AIProviderType.CodebuddyCli] = ["model", "reasoning"],
        [AIProviderType.QoderCli] = ["model", "reasoning"],
        [AIProviderType.KiroCli] = ["model", "reasoning"],
        [AIProviderType.GeminiCli] = ["model"],  // Gemini은 reasoning 매개변수를 지원하지 않음
        // ...
    };
```

## GLM 모델 지원 체계

Hagicode의 Secondary Professions Catalog에는 완전한 GLM 시리즈 모델 지원이 정의되어 있습니다:

| Model ID | 이름 | 기본 Reasoning | 호환 CLI 패밀리 |
|----------|------|-------------------|-------------------------|
| `glm-4.7` | GLM 4.7 | high | claude, codebuddy, hermes, qoder, kiro |
| `glm-5` | GLM 5 | high | claude, codebuddy, hermes, qoder, kiro |
| `glm-5-turbo` | GLM 5 Turbo | high | claude, codebuddy, hermes, qoder, kiro |
| `glm-5.0` | GLM 5.0 (Legacy) | high | claude, codebuddy, hermes, qoder, kiro |
| `glm-5.1` | GLM 5.1 | high | claude, codebuddy, hermes, qoder, kiro |

## GLM-5.1 vs GLM-5.0의 핵심 차이

`AcpSessionModelBootstrapper.cs`의 구현에서 GLM-5.1과 GLM-5.0의 차이를 명확하게 볼 수 있습니다:

### GLM-5.1의 독립 구현

GLM-5.1은 독립적인 새 모델 식별자이며, legacy 처리 로직이 없습니다:

```csharp

```

Secondary Professions Catalog의 정의:

## 다양한 CLI에서 GLM-5.1 사용

### 1. Claude Code CLI + GLM-5.1

Claude Code CLI는 Hagicode의 핵심 CLI 중 하나이며, Hero 설정 시스템을 통해 구성합니다:

```json
{
  "primaryProfessionId": "profession-claude-code",
  "secondaryProfessionId": "secondary-glm-5-1",
  "model": "glm-5.1",
  "reasoning": "high"
}
```

해당 `HeroEquipmentCatalogItem` 설정:

```typescript
{
  id: 'secondary-glm-5-1',
  name: 'GLM 5.1',
  family: 'anthropic',
  kind: 'model',
  primaryFamily: 'claude',
  compatiblePrimaryFamilies: ['claude', 'codebuddy', 'hermes', 'qoder', 'kiro'],
  defaultParameters: {
    model: 'glm-5.1',
    reasoning: 'high'
  }
}
```

### 2. OpenCode CLI + GLM-5.1

OpenCode CLI는 가장 유연한 CLI로, `provider/model` 형식으로 임의의 모델을 지정할 수 있습니다:

**방식 1: ZAI provider 접두사 사용**
```json
{
  "primaryProfessionId": "profession-opencode",
  "model": "zai/glm-5.1",
  "reasoning": "high"
}
```

**방식 2: 모델 ID 직접 사용**
```json
{
  "model": "glm-5.1"
}
```

**방식 3: 프론트엔드 설정 화면**

`HeroModelEquipmentForm.tsx`에서 OpenCode CLI에는 특수 플레이스홀더 힌트가 있습니다:

```tsx
const OPEN_CODE_MODEL_PLACEHOLDER = 'myprovider/glm-4.7';

const modelPlaceholder = primaryProviderType === PCode_Models_AIProviderType.OPEN_CODE_CLI
  ? OPEN_CODE_MODEL_PLACEHOLDER
  : 'gpt-5.4';
```

사용자는 다음을 입력할 수 있습니다:
```
zai/glm-5.1
glm-5.1
```

**OpenCode CLI의 모델 파싱 로직**:

```csharp
internal OpenCodeModelSelection? ResolveModelSelection(string? rawModel)
{
    var normalized = NormalizeOptionalValue(rawModel);
    if (normalized == null) return null;

    var slashIndex = normalized.IndexOf('/');
    if (slashIndex < 0)
    {
        // 슬래시 없음: 모델 ID를 직접 사용
        return new OpenCodeModelSelection {
            ProviderId = string.Empty,
            ModelId = normalized,
        };
    }

    // 슬래시 있음: provider/model 형식 파싱
    var providerId = normalized[..slashIndex].Trim();
    var modelId = normalized[(slashIndex + 1)..].Trim();

    return new OpenCodeModelSelection {
        ProviderId = providerId,
        ModelId = modelId,
    };
}
```

### 3. Codebuddy CLI + GLM-5.1

Codebuddy CLI에는 특수한 legacy 처리 로직이 있습니다:

```json
{
  "primaryProfessionId": "profession-codebuddy",
  "model": "glm-5.1",
  "reasoning": "high"
}
```

**주의**: Codebuddy는 GLM-5.0에 대해 특수 보유가 있으며, legacy normalization을 따르지 않습니다:

```csharp
return !string.Equals(providerName, "CodebuddyCli", StringComparison.OrdinalIgnoreCase)
       && string.Equals(normalizedModel, LegacyGlm5TurboModelValue, StringComparison.OrdinalIgnoreCase)
    ? Glm5TurboModelValue
    : normalizedModel;
// CodebuddyCli일 때, glm-5.0은 glm-5-turbo로 정규화되지 않음
```

## GLM-5.1의 개선 장점

GLM-5.0에 비해 GLM-5.1은 다음과 같은 현저한 개선이 있습니다:

### 1. 더 나은 추론 능력

智譜 공식 발표 정보에 따르면 GLM-5.1의 개선에는 다음이 포함됩니다:

- **더 강한 코드 이해 능력**: 복잡한 코드 구조의 파싱이 더 정확합니다
- **더 긴 컨텍스트 이해**: 더 긴 대화 컨텍스트를 지원합니다
- **도구 호출 향상**: MCP 도구 호출의 성공률이 향상되었습니다
- **출력 안정성**: 무작위성과 환각 감소

### 2. 다중 CLI 전면 호환

GLM-5.1은 Hagicode가 지원하는 모든 주류 CLI를 포괄합니다:

```
compatiblePrimaryFamilies: [
  "claude",      // Claude Code CLI
  "codebuddy",   // Codebuddy CLI
  "hermes",      // Hermes CLI
  "qoder",       // Qoder CLI
  "kiro"         // Kiro CLI
]
```

## 주의사항

### 1. API Key 설정

`ANTHROPIC_AUTH_TOKEN` 환경 변수를 올바르게 설정했는지 확인하십시오. 이것은 모든 CLI가 모델에 연결하는 데 필요한 자격 증명입니다.

### 2. 모델 가용성

GLM-5.1은 해당 모델 제공자에서 개통해야 합니다:
- 智譜 AI ZAI 플랫폼은 기본적으로 지원합니다
- 알리바바 클라우드 DashScope는 별도로 신청해야 할 수 있습니다

### 3. OpenCode CLI 형식

`provider/model` 형식을 사용할 때 provider ID가 올바른지 확인하십시오:
- 智譜 AI: `zai` 또는 `zhipuai`
- 알리바바 클라우드: `aliyun` 또는 `dashscope`

### 4. Reasoning 매개변수

- 최상의 코드 생성 효과를 위해 `high` 수준을 사용하는 것을 권장합니다
- Gemini CLI는 reasoning 매개변수를 지원하지 않으며, 이 설정은 자동으로 무시됩니다

## 요약

Hagicode는 통일된 추상화 레이어를 통해 GLM-5.1과 여러 CLI의 유연한 통합을 실현했습니다. 개발자는 자신의 선호와 사용 시나리오에 따라 적절한 CLI 도구를 선택하고, 간단한 설정으로 최신 GLM-5.1 모델을 사용할 수 있습니다.

GLM-5.1은 智譜의 최신 모델 버전으로, GLM-5.0에 비해 명확한 개선이 있습니다:
- 독립적인 버전 식별자, legacy 짐을 없음
- 더 강한 추론 능력과 코드 이해
- 광범위한 다중 CLI 호환성
- 유연한 추론 수준 설정

올바른 환경 변수와 Hero 장비를 구성하면, 플레이어는 다양한 CLI 환경에서 GLM-5.1의 강력한 능력을 충분히 발�할 수 있습니다.

## HagiCode 계속 체험하기

GLM-5.1, 다중 CLI 오케스트레이션 및 Hagiode 설정 체계를 실제로 구동하려면 다음 진입점에서 시작할 수 있습니다:

- GitHub에서 메인 프로젝트와 최신 구현 진행 상황 확인: [github.com/HagiCode-org/site](https://github.com/HagiCode-org/site)
- 공식 웹사이트에서 제품 로드맵, 능력 경계 및 설치 진입점 확인: [hagicode.com](https://hagicode.com)
- Docker Compose 문서에 따라 원클릭 배포를 완료한 후 모델 및 CLI를 전환하며 실제 검증: [docs.hagicode.com/installation/docker-compose](https://docs.hagicode.com/installation/docker-compose)
- 로컬 데스크톱 경험을 선호하신다면 Desktop 진입점에서 시작할 수 있습니다: [hagicode.com/desktop/](https://hagicode.com/desktop/)

Kimi, Claude Code, OpenCode 같은 CLI와 GLM-5.1을 동일한 추상화 레이어에 놓고 비교하면, 모델 전환, 매개변수 매핑 및 엔지니어링 경계에 관한 많은 문제가 한 번에 명확해집니다.
