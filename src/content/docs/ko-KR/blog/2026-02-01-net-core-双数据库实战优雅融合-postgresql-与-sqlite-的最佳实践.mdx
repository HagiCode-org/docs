## .NET Core 이중 데이터베이스 실전: PostgreSQL과 SQLite의 평화로운 공존

> 현대 애플리케이션을 구축할 때 우리는 종종 이러한 선택에 직면합니다: 개발 환경은 가볍고 편리한 것을 원하고, 프로덕션 환경은 높은 동시성과 높은 가용성이 필요합니다. 이 글에서는 .NET Core 프로젝트에서 PostgreSQL과 SQLite를 동시에 우아하게 지원하여 "개발에는 SQLite, 프로덕션에는 PG"를 실현하는 모범 사례를 공유합니다.


## 배경

소프트웨어 개발에서 환경 차별화는 항상 개발 팀을 괴롭히는 난제 중 하나입니다. 우리가 구축 중인 **HagiCode** 플랫폼을 예로 들면, 이것은 ASP.NET Core 10과 React를 기반으로 하는 AI 보조 개발 시스템으로, 내부에 Orleans가 통합되어 분산 상태 관리를 수행하며, 기술 스택이 상당히 현대적이고 복잡합니다.

프로젝트 초기에 우리는 전형적인 엔지니어링 문제에 직면했습니다: 개발자들은 로컬 환경이 "설치만 하면 바로 사용 가능"하기를 원하며, 무거운 PostgreSQL 데이터베이스를 설치하고 구성하는 것을 원하지 않았습니다. 하지만 프로덕션 환경에서는 높은 동시성 쓰기와 복잡한 JSON 쿼리를 처리해야 했고, 이때 가벼운 SQLite는 역부족이었습니다.

코드베이스의 통일을 전제로 애플리케이션이 클라이언트 소프트웨어처럼 SQLite의 휴대성을 활용하면서도 엔터프라이즈 서비스처럼 PostgreSQL의 강력한 성능을 발휘하게 하려면 어떻게 해야 할까요? 이것이 이 글에서 탐구할 핵심 문제입니다.

## HagiCode에 대해

이 글에서 공유하는 이중 데이터베이스 적용 방안은 우리의 **HagiCode** 프로젝트에서의 실전 경험에서 직접 비롯되었습니다. HagiCode는 AI 프롬프트 관리와 OpenSpec 워크플로우가 통합된 차세대 개발 플랫폼입니다. 개발자의 경험과 프로덕션 환경의 안정성을 모두 고려하기 위해 우리는 이 효과적인 아키텍처 패턴을 탐색해냈습니다.

프로젝트의 전모를 알아보시려면 GitHub 저장소를 방문하세요: [HagiCode-org/site](https://github.com/HagiCode-org/site).

## 핵심 내용 1: 아키텍처 설계와 통합 추상화

.NET Core에서 이중 데이터베이스 지원을 구현하려면 핵심 아이디어는 "구체적 구현이 아닌 추상화에 의존하는 것"입니다. 데이터베이스 선택권을 비즈니스 코드에서 분리하여 구성 계층에 맡겨야 합니다.

### 설계 아이디어

1.  **통합 인터페이스**: 모든 비즈니스 로직은 `DbContext` 기본 클래스나 사용자 정의 인터페이스에 의존해야 하며, 구체적인 `PostgreSqlDbContext`가 아닙니다.
2.  **구성 기반**: `appsettings.json`의 구성 항목을 통해 애플리케이션 시작 시 로드할 데이터베이스 공급자를 동적으로 결정합니다.
3.  **기능 격리**: PostgreSQL 고유 기능(예: JSONB)에 대해 적용 처리를 수행하여 SQLite에서도降级 실행되도록 합니다.

### 코드 구현: 동적 컨텍스트 구성

ASP.NET Core의 `Program.cs`에서 `UseNpgsql`이나 `UseSqlite`를 하드코딩하면 안 됩니다. 대신 구성을 읽어 동적으로 결정해야 합니다.

먼저 구성 클래스를 정의합니다:

```csharp
public class DatabaseSettings
{
    public const string SectionName = "Database";
    
    // 데이터베이스 유형: PostgreSQL 또는 SQLite
    public string DbType { get; set; } = "PostgreSQL"; 
    
    // 연결 문자열
    public string ConnectionString { get; set; } = string.Empty;
}
```

그런 다음 `Program.cs`에서 구성에 따라 서비스를 등록합니다:

```csharp
// 구성 읽기
var databaseSettings = builder.Configuration.GetSection(DatabaseSettings.SectionName).Get<DatabaseSettings>();

// DbContext 등록
builder.Services.AddDbContext<ApplicationDbContext>(options =>
{
    if (databaseSettings?.DbType?.ToLower() == "sqlite")
    {
        // SQLite 구성
        options.UseSqlite(databaseSettings.ConnectionString);
        
        // SQLite의 동시 쓰기 제한 처리
        // 참고: 프로덕션 환경에서는 동시 성능 향상을 위해 WAL 모드를 켜는 것이 좋습니다
    }
    else
    {
        // PostgreSQL 구성(기본값)
        options.UseNpgsql(databaseSettings.ConnectionString, npgsqlOptions =>
        {
            // JSONB 지원 활성화, 이것은 AI 대화 기록을 처리할 때 매우 유용합니다
            npgsqlOptions.UseJsonNet(); 
        });
        
        // 연결 풀 재시도 전략 구성
        options.EnableRetryOnFailure(3);
    }
});
```

## 핵심 내용 2: 차이점 처리와 마이그레이션 전략

PostgreSQL과 SQLite는 모두 SQL 표준을 지원하지만, 구체적인 기능과 동작에는 상당한 차이가 있습니다. 이러한 차이점을 잘 처리하지 않으면 "로컬에서는 작동하지만 배포하면 오류가 발생"하는 당황스러운 상황이 발생할 수 있습니다.

### 1. JSON 유형의 처리

HagiCode에서 우리는 많은 프롬프트와 AI 메타데이터를 저장해야 하며, 이것은 보통 JSON 열을 포함합니다.
- **PostgreSQL**: 원시 `JSONB` 유형이 있어 쿼리 성능이 매우 우수합니다.
- **SQLite**: 원시 JSON 유형이 없습니다(최신 버전에는 JSON1 확장이 있지만 객체 매핑에는 여전히 차이가 있으며), 보통 TEXT로 저장됩니다.

**솔루션**:
EF Core의 엔티티 매핑에서 변환 가능한 유형으로 구성합니다.

```csharp
protected override void OnModelCreating(ModelBuilder modelBuilder)
{
    base.OnModelCreating(modelBuilder);

    // 엔티티 구성
    modelBuilder.Entity<PromptTemplate>(entity => 
    {
        entity.Property(e => e.Metadata)
              .HasColumnType("jsonb") // PG는 jsonb 사용
              .HasConversion(
                  v => JsonSerializer.Serialize(v, (JsonSerializerOptions)null),
                  v => JsonSerializer.Deserialize<Dictionary<string, object>>(v, (JsonSerializerOptions)null)
              ); 
    });
}
```

SQLite를 사용할 때 `HasColumnType("jsonb")`는 무시되거나 경고가 생성되지만, `HasConversion`이 구성되어 있으므로 데이터는 올바르게 직렬화 및 역직렬화되어 TEXT 필드에 저장되어 호환성이 보장됩니다.

### 2. 마이그레이션 전략의 분리

동일한 Migration 스크립트가 PG와 SQLite를 모두 적용하도록 하려고 시도하지 마십시오. 기본 키 생성 전략, 인덱스 구문 등의 차이로 인해 필연적으로 실패합니다.

**권장 사항**:
두 개의 마이그레이션 분기 또는 프로젝트를 유지 관리합니다. HagiCode의 개발 흐름에서 우리는 다음과 같이 처리합니다:

1.  **개발 단계**: 주로 SQLite에서 작업합니다. `Add-Migration Init_Sqlite -OutputDir Migrations/Sqlite`를 사용합니다.
2.  **적용 단계**: 기능 개발을 완료한 후 연결 문자열을 PostgreSQL로 전환하고 `Add-Migration Init_Postgres -OutputDir Migrations/Postgres`를 실행합니다.
3.  **자동화 스크립트**: 현재 환경 변수에 따라 해당 마이그레이션을 자동으로 적용하는 간단한 PowerShell 또는 Bash 스크립트를 작성합니다.

```bash
# 간단한 배포 논리 의사 코드
if [ "$DATABASE_PROVIDER" = "PostgreSQL" ]; then
    dotnet ef database update --project Migrations.Postgres
else
    dotnet ef database update --project Migrations.Sqlite
fi
```

## 핵심 내용 3: HagiCode의 실전 경험 요약

**HagiCode**를 단일 데이터베이스에서 이중 데이터베이스 지원으로 리팩토링하는 과정에서 우리는 몇 가지 문제를 겪었고 핵심 경험도 요약했습니다. 여러분이 같은 문제를 겪지 않기를 바랍니다.

### 1. 동시성과 트랜잭션의 차이

PostgreSQL은 서버-클라이언트 아키텍처로 높은 동시성 쓰기를 지원하며 트랜잭션 격리 수준이 매우 강력합니다. 반면 SQLite는 파일 잠금 메커니즘으로 쓰기 작업이 전체 데이터베이스 파일을 잠급니다(WAL 모드가 활성화된 경우 제외).

**제안**:
빈번한 쓰기가 관련된 비즈니스 로직(예: 사용자의 편집 상태를 실시간으로 저장)을 작성할 때 SQLite의 잠금 메커니즘을 반드시 고려해야 합니다. **HagiCode**의 OpenSpec 협업 모듈을 설계할 때 우리는 "쓰기 전 병합" 메커니즘을 도입하여 데이터베이스의 직접 쓰기 빈도를 줄여 두 데이터베이스에서 모두 높은 성능을 유지할 수 있었습니다.

### 2. 연결 문자열의 수명 주기 관리

PostgreSQL 연결 생성 비용이 상대적으로 높아 연결 풀에 의존합니다. 반면 SQLite 연결은 매우 가볍지만 제때 해제하지 않으면 파일 잠금으로 인해 후속 작업이 시간 초과될 수 있습니다.

`Program.cs`에서 다른 데이터베이스에 대해 정밀한 조정을 할 수 있습니다:

```csharp
if (databaseSettings?.DbType?.ToLower() == "sqlite")
{
    // SQLite: 연결을 유지하면 성능이 향상되지만 파일 잠금에 주의해야 합니다
    options.UseSqlite(connectionString, sqliteOptions =>
    {
        // 명령 시간 초과 설정
        sqliteOptions.CommandTimeout(30);
    });
}
else
{
    // PG: 연결 풀 활용
    options.UseNpgsql(connectionString, npgsqlOptions =>
    {
        npgsqlOptions.MaxBatchSize(100);
        npgsqlOptions.CommandTimeout(30);
    });
}
```

### 3. 테스트 커버리지의 중요성

많은 개발자들(우리 팀 초기 구성원 포함)이 쉽게 범하는 실수는 개발 환경(보통 SQLite)에서만 단위 테스트를 실행하는 것입니다.

우리는 HagiCode의 CI/CD 파이프라인에 GitHub Action 단계를 강제로 추가하여 모든 Pull Request가 PostgreSQL 통합 테스트를 통과하도록 했습니다.

```yaml
# .github/workflows/test.yml 예시 조각
- name: Run Integration Tests (PostgreSQL)
  run: |
    docker-compose up -d db_postgres
    dotnet test --filter "Category=Integration"
```

이것은 SQL 구문 차이, 대소문자 구분에 대한 수많은 버그를 차단하는 데 도움이 되었습니다.

## 요약

추상화 계층과 구성 기반 의존성 주입을 통해 **HagiCode** 프로젝트에서 PostgreSQL과 SQLite의 "이중 궤도" 실행을 성공적으로 구현했습니다. 이것은 새로운 개발자의 진입 장벽을 크게 낮출 뿐만 아니라(PG를 설치할 필요 없음), 프로덕션 환경에도 견고한 성능 보장을 제공합니다.

핵심 내용을 되돌아보면:
1.  **추상화 최우선**: 비즈니스 코드는 구체적인 데이터베이스 구현에 의존하지 않습니다.
2.  **구성 분리**: 개발과 프로덕션은 다른 `appsettings.json`을 사용합니다.
3.  **마이그레이션 분리**: 하나의 Migration으로 모든 것을 처리하려고 하지 마십시오.
4.  **기능 降级**: SQLite에서는 호환성을 우선으로 하고, PostgreSQL에서는 성능을 우선으로 합니다.

이 아키텍처 패턴은 HagiCode뿐만 아니라 가벼운 개발과 무거운 프로덕션 사이에서 균형을 찾아야 하는 모든 .NET 프로젝트에 적용할 수 있습니다.
