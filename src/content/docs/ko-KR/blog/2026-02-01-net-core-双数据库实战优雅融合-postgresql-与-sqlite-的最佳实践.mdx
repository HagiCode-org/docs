## .NET Core 듀얼 데이터베이스 실전: PostgreSQL과 SQLite의 평화로운 공존

> 현대 애플리케이션을 구축할 때 자주 직면하는 선택입니다: 개발 환경에서는 가볍고 편리한 것을 원하고, 프로덕션 환경에서는 높은 동시성과 고가용성을 필요로 합니다. 이 글에서는 .NET Core 프로젝트에서 PostgreSQL과 SQLite를 동시에 우아하게 지원하여 "개발에는 SQLite, 프로덕션에는 PG"를 실현하는 모범 사례를 공유합니다.


## 배경

소프트웨어 개발에서 환경 차별화는 항상 개발 팀을 괴롭히는 난제 중 하나입니다. 우리가 구축 중인 **HagiCode** 플랫폼을 예로 들면, 이는 ASP.NET Core 10과 React 기반의 AI 보조 개발 시스템으로, Orleans를 통합하여 분산 상태 관리를 수행하며, 기술 스택이 매우 현대적이고 복잡합니다.

프로젝트 초기에 전형적인 엔지니어링 문제에 직면했습니다: 개발자는 로컬 환경에서 "기본 제공(Out of the box)" 상태를 원하며, 무거운 PostgreSQL 데이터베이스의 설치와 구성을 원하지 않습니다. 하지만 프로덕션 환경에서는 높은 동시성 쓰기와 복잡한 JSON 쿼리를 처리해야 하며, 이때 가벼운 SQLite는 역부족입니다.

코드베이스의 통일성을 유지하면서 애플리케이션이 클라이언트 소프트웨어처럼 SQLite의 이식성을 활용할 수 있고, 동시에 엔터프라이즈급 서비스처럼 PostgreSQL의 강력한 성능을 발휘할 수 있도록 하는 방법이 이 글의 핵심 주제입니다.

## HagiCode 소개

이 글에서 공유하는 듀얼 데이터베이스 어댑터 방안은 **HagiCode** 프로젝트에서의 실전 경험에서 직접 비롯되었습니다. HagiCode는 AI 프롬프트 관리와 OpenSpec 워크플로우를 통합한 차세대 개발 플랫폼입니다. 개발자 경험과 프로덕션 환경의 안정성을 모두 고려하기 위해 이 효과적인 아키텍처 패턴을 탐색했습니다.

GitHub 저장소를 방문하여 프로젝트 전체를 확인해 주세요: [HagiCode-org/site](https://github.com/HagiCode-org/site).

## 핵심 내용 1: 아키텍처 설계와 통합 추상화

.NET Core에서 듀얼 데이터베이스 지원을 구현하려면 핵심 사상은 "구체적인 구현이 아닌 추상화에 의존하는 것"입니다. 데이터베이스 선택권을 비즈니스 코드에서 분리하여 구성 레이어에 맡겨야 합니다.

### 설계 아이디어

1.  **통합 인터페이스**: 모든 비즈니스 로직은 구체적인 `PostgreSqlDbContext`가 아닌 `DbContext` 기본 클래스 또는 커스텀 인터페이스에 의존해야 합니다.
2.  **구성 구동**: `appsettings.json`의 구성 항목을 통해 애플리케이션 시작 시 로드할 데이터베이스 공급자를 동적으로 결정합니다.
3.  **기능 격리**: PostgreSQL 고유 기능(예: JSONB)에 대한 어댑터 처리를 수행하여 SQLite에서도 저하 모드로 실행할 수 있도록 보장합니다.

### 코드 구현: 동적 컨텍스트 구성

ASP.NET Core의 `Program.cs`에서 `UseNpgsql` 또는 `UseSqlite`를 하드코딩하지 않아야 합니다. 대신 구성을 읽어 동적으로 결정해야 합니다.

먼저 구성 클래스를 정의합니다:

```csharp
public class DatabaseSettings
{
    public const string SectionName = "Database";

    // 데이터베이스 유형: PostgreSQL 또는 SQLite
    public string DbType { get; set; } = "PostgreSQL";

    // 연결 문자열
    public string ConnectionString { get; set; } = string.Empty;
}
```

그런 다음 `Program.cs`에서 구성에 따라 서비스를 등록합니다:

```csharp
// 구성 읽기
var databaseSettings = builder.Configuration.GetSection(DatabaseSettings.SectionName).Get<DatabaseSettings>();

// DbContext 등록
builder.Services.AddDbContext<ApplicationDbContext>(options =>
{
    if (databaseSettings?.DbType?.ToLower() == "sqlite")
    {
        // SQLite 구성
        options.UseSqlite(databaseSettings.ConnectionString);

        // SQLite의 동시 쓰기 제한 처리
        // 주의: 프로덕션 환경에서는 WAL 모드를 활성화하여 동시 성능을 향상시키는 것을 권장합니다
    }
    else
    {
        // PostgreSQL 구성 (기본값)
        options.UseNpgsql(databaseSettings.ConnectionString, npgsqlOptions =>
        {
            // JSONB 지원 활성화, AI 대화 기록 처리에 매우 유용합니다
            npgsqlOptions.UseJsonNet();
        });

        // 연결 풀 재연결 전략 구성
        options.EnableRetryOnFailure(3);
    }
});
```

## 핵심 내용 2: 차이점 처리와 마이그레이션 전략

PostgreSQL과 SQLite는 모두 SQL 표준을 지원하지만 구체적인 기능과 동작에서 유의미한 차이가 있습니다. 이러한 차이를 제대로 처리하지 않으면 "로컬에서는 되는데 배포하면 오류가 발생"하는 당혹스러운 상황이 발생할 수 있습니다.

### 1. JSON 유형의 처리

HagiCode에서는 많은 양의 프롬프트와 AI 메타데이터를 저장해야 하며, 이는 일반적으로 JSON 컬럼을 포함합니다.
- **PostgreSQL**: 네이티브 `JSONB` 유형을 보유하며 쿼리 성능이 매우 우수합니다.
- **SQLite**: 네이티브 JSON 유형이 없습니다(신규 버전에는 JSON1 확장이 있지만 객체 매핑에서 여전히 차이가 있으며), 일반적으로 TEXT로 저장됩니다.

**해결 방법**:
EF Core의 엔티티 매핑에서 변환 가능한 유형으로 구성합니다.

```csharp
protected override void OnModelCreating(ModelBuilder modelBuilder)
{
    base.OnModelCreating(modelBuilder);

    // 엔티티 구성
    modelBuilder.Entity<PromptTemplate>(entity =>
    {
        entity.Property(e => e.Metadata)
              .HasColumnType("jsonb") // PG는 jsonb 사용
              .HasConversion(
                  v => JsonSerializer.Serialize(v, (JsonSerializerOptions)null),
                  v => JsonSerializer.Deserialize<Dictionary<string, object>>(v, (JsonSerializerOptions)null)
              );
    });
}
```

SQLite를 사용할 때 `HasColumnType("jsonb")`는 무시되거나 경고를 생성하지만, `HasConversion`이 구성되어 있으므로 데이터는 올바르게 직렬화 및 역직렬화되어 TEXT 필드에 저장되어 호환성을 보장합니다.

### 2. 마이그레이션 전략의 분리

동일한 Migration 스크립트 세트를 PG와 SQLite에 모두 적용하려고 시도하지 마세요. 기본 키 생성 전략, 인덱스 문법 등의 차이로 인해 반드시 실패합니다.

**권장 실천**:
두 개의 마이그레이션 분기 또는 프로젝트를 유지합니다. HagiCode의 개발 흐름에서는 다음과 같이 처리합니다:

1.  **개발 단계**: 주로 SQLite에서 작업합니다. `Add-Migration Init_Sqlite -OutputDir Migrations/Sqlite`를 사용합니다.
2.  **적응 단계**: 일정 기능 개발 후 연결 문자열을 PostgreSQL로 전환하고 `Add-Migration Init_Postgres -OutputDir Migrations/Postgres`를 실행합니다.
3.  **자동화 스크립트**: 현재 환경 변수에 따라 해당 마이그레이션을 자동으로 적용하는 간단한 PowerShell 또는 Bash 스크립트를 작성합니다.

```bash
# 간단한 배포 로직 의사코드
if [ "$DATABASE_PROVIDER" = "PostgreSQL" ]; then
    dotnet ef database update --project Migrations.Postgres
else
    dotnet ef database update --project Migrations.Sqlite
fi
```

## 핵심 내용 3: HagiCode의 실전 경험 요약

**HagiCode**를 단일 데이터베이스에서 듀얼 데이터베이스 지원으로 리팩토링하는 과정에서 몇 가지 시행착오를 겪었으며, 여러분이 이를 피할 수 있도록 핵심 경험을 요약합니다.

### 1. 동시성과 트랜잭션의 차이

PostgreSQL은 서버-클라이언트 아키텍처로 높은 동시성 쓰기를 지원하며 트랜잭션 격리 수준이 매우 강력합니다. 반면 SQLite는 파일 잠금 메커니즘으로 쓰기 작업이 전체 데이터베이스 파일을 잠급니다(WAL 모드 활성화 시 제외).

**권장사항**:
빈번한 쓰기를 포함하는 비즈니스 로직을 작성할 때(예: 사용자 편집 상태의 실시간 저장) 반드시 SQLite의 잠금 메커니즘을 고려해야 합니다. **HagiCode**의 OpenSpec 협업 모듈 설계 시 "쓰기 전 병합" 메커니즘을 도입하여 데이터베이스 직접 쓰기 빈도를 줄여 두 데이터베이스 모두에서 높은 성능을 유지했습니다.

### 2. 연결 문자열의 수명 주기 관리

PostgreSQL은 연결 설정 비용이 높고 연결 풀에 의존합니다. SQLite 연결은 매우 가볍지만, 적시에 해제하지 않으면 파일 잠금으로 인해 후속 작업 시간 초과가 발생할 수 있습니다.

`Program.cs`에서 다양한 데이터베이스에 대해 정밀한 조정을 수행할 수 있습니다:

```csharp
if (databaseSettings?.DbType?.ToLower() == "sqlite")
{
    // SQLite: 연결을 유지하면 성능이 향상되지만 파일 잠금에 주의해야 합니다
    options.UseSqlite(connectionString, sqliteOptions =>
    {
        sqliteOptions.CommandTimeout(30);
    });
}
else
{
    // PG: 연결 풀 활용
    options.UseNpgsql(connectionString, npgsqlOptions =>
    {
        npgsqlOptions.MaxBatchSize(100);
        npgsqlOptions.CommandTimeout(30);
    });
}
```

### 3. 테스트 커버리지의 중요성

많은 개발자(우리 팀의 초기 멤버 포함)가 쉽게 범하는 실수가 있습니다: 개발 환경(일반적으로 SQLite)에서만 단위 테스트를 실행합니다.

HagiCode의 CI/CD 파이프라인에서는 모든 Pull Request가 PostgreSQL 통합 테스트를 통과하도록 GitHub Action 단계를 강제로 추가했습니다.

```yaml
# .github/workflows/test.yml 예시 조각
- name: Run Integration Tests (PostgreSQL)
  run: |
    docker-compose up -d db_postgres
    dotnet test --filter "Category=Integration"
```

이것은 SQL 문법 차이, 대소문자 민감성에 관한 버그를 수없이 차단했습니다.

## 요약

추상화 계층과 구성 구동 의존성 주입을 도입함으로써 **HagiCode** 프로젝트에서 PostgreSQL과 SQLite의 "이중 궤도" 실행을 성공적으로 구현했습니다. 이는 새로운 개발자의 진입 장벽을 크게 낮추고(PG 설치 불필요), 프로덕션 환경에 확실한 성능 보장을 제공합니다.

핵심 포인트를 되짚어 보겠습니다:
1.  **추상화 최우선**: 비즈니스 코드는 구체적인 데이터베이스 구현에 의존하지 않습니다.
2.  **구성 분리**: 개발과 프로덕션에서 각각 다른 `appsettings.json`을 사용합니다.
3.  **마이그레이션 분리**: 하나의 Migration으로 모든 것을 처리하려고 시도하지 마세요.
4.  **기능 저하**: SQLite에서는 호환성을 우선으로 하고, PostgreSQL에서는 성능을 우선으로 합니다.

이 아키텍처 패턴은 HagiCode뿐만 아니라 가벼운 개발과 무거운 프로덕션 사이의 균형을 찾아야 하는 모든 .NET 프로젝트에 적용할 수 있습니다.
