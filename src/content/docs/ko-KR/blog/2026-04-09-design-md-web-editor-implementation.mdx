## 웹 인터페이스에서 DESIGN.md 직접 편집하기: 아이디어에서 구현까지

> MonoSpecs 프로젝트 관리 시스템에서 `DESIGN.md`는 프로젝트의 아키텍처 설계와 기술 결정을 담고 있습니다. 하지만 전통적인 편집 워크플로우는 사용자가 외부 에디터로 전환해야 합니다. 그 단절된 경험은 시를 읽다가 중간에 방해받는 것과 같습니다. 영감도, 분위기도 사라집니다. 이 글에서는 HagiCode 프로젝트에서 실제로 적용한 해결책을 공유합니다. 웹 인터페이스에서 `DESIGN.md`를 직접 편집하고, 온라인 디자인 사이트에서 템플릿을 가져오는 기능을 지원합니다. 모든 것을 한 번의 흐름으로 완성하는 느낌, 누가 좋아하지 않겠습니까?

## 배경

프로젝트 디자인 문서의 핵심 매개체로서 `DESIGN.md`는 아키텍처 설계, 기술 결정, 구현 지침과 같은 핵심 정보를 담고 있습니다. 하지만 전통적인 편집 방식은 사용자가 VS Code와 같은 외부 에디터로 전환하여 물리적 경로를 수동으로 찾은 다음 파일을 편집해야 합니다. 특별히 복잡하지는 않지만, 몇 번 반복하면 지치게 됩니다.

문제는 주로 다음과 같이 나타납니다.

- **단절된 워크플로우**: 사용자는 웹 관리 인터페이스와 로컬 에디터 사이를 계속 전환해야 하며, 워크플로우의 연속성이 끊어집니다. 음악을 듣다가 중간에 끊기는 것과 같습니다.
- **재사용 곤란**: 디자인 사이트에는 이미 풍부한 디자인 템플릿 라이브러리가 게시되어 있지만, 프로젝트 편집 워크플로우에 직접 통합할 수 없습니다. 좋은 것이 존재하지만, 필요한 곳에서 사용할 수 없습니다.
- **경험 루프 부재**: "미리보기-선택-가져오기"의 폐쇄 루프가 없어 사용자가 수동으로 복사하고 붙여넣어야 하며, 실수 위험이 증가합니다.
- **협업 마찰**: 디자인 문서와 코드 구현을 동기화 상태로 유지하는 것이 마찰이 큰 프로세스가 되어 팀 효율성에 영향을 미칩니다.

이러한 문제점들을 해결하기 위해 웹 인터페이스에서 `DESIGN.md`에 대한 직접 편집 지원을 추가하고, 온라인 디자인 사이트에서 원클릭 템플릿 가져오기를 허용하기로 결정했습니다. 대단한 결정은 아니었습니다. 그저 개발 경험을 더 부드럽게 만들고 싶었을 뿐입니다.

## HagiCode 소개

이 글에서 공유하는 해결책은 [HagiCode](https://hagicode.com) 프로젝트에서의 실무 경험에서 나온 것입니다. HagiCode는 AI 기반 코딩 어시스턴트 프로젝트이며, 개발 과정에서 프로젝트 디자인 문서를 자주 유지 관리해야 합니다. 팀이 더 효율적으로 협업할 수 있도록 온라인 편집 및 가져오기 해결책을 탐색하고 구현했습니다. 신비로운 것은 없습니다. 문제에 직면하고 해결 방법을 찾아낸 것뿐입니다.

## 기술 해결책

### 전체 아키텍처

이 해결책은 동일 출처 프록시를 사용하는 프론트엔드-백엔드 분리 아키텍처를 사용하며, 주로 다음 레이어로 구성됩니다. 실제로 이 설계는 "각 부분이 자기 일을 하도록" 요약할 수 있습니다.

**1. 프론트엔드 에디터 레이어**
```typescript
// 핵심 컴포넌트: DesignMdManagementDrawer
// 위치: repos/web/src/components/project/DesignMdManagementDrawer.tsx
// 책임: 편집, 저장, 버전 충돌 감지 및 가져오기 흐름 처리
```

**2. 백엔드 서비스 레이어**
```csharp
// 핵심 서비스: ProjectAppService.DesignMd
// 위치: repos/hagicode-core/src/PCode.Application/ProjectAppService.DesignMd.cs
// 책임: 경로 해석, 파일 읽기/쓰기 및 버전 관리
```

**3. 동일 출처 프록시 레이어**
```csharp
// 프록시 서비스: ProjectAppService.DesignMdSiteIndex
// 위치: repos/hagicode-core/src/PCode.Application/ProjectAppService.DesignMdSiteIndex.cs
// 책임: 디자인 사이트 리소스 프록시, 미리보기 이미지 캐싱 및 보안 검증
```

### 핵심 기술 결정

#### 결정 1: 글로벌 드로어 패턴

단일 글로벌 드로어를 사용하여 로컬 팝업 레이어 대신 `layoutSlice`를 통해 상태를 관리합니다. 이를 통해 사용자는 뷰(`classic` 및 `kanban`) 전반에 걸쳐 일관된 경험을 얻을 수 있습니다. 사용자가 어떤 뷰에서 에디터를 열든 동일한 상호 작용 모델을 제공합니다. 일관된 경험은 사람들이 뷰를 전환할 때 혼란스러워하지 않게 합니다.

#### 결정 2: 프로젝트 범위 API

`DESIGN.md` 관련 엔드포인트를 `ProjectController` 아래에 배치하여 기존 프로젝트 권한 경계를 재사용하고, 별도의 컨트롤러를 추가하는 복잡성을 피합니다. 이로써 권한 처리가 더 명확해지고 RESTful 리소스 조직과도 일치합니다. 때로는 처음부터 새로 만드는 것보다 재사용이 더 의미 있습니다.

#### 결정 3: 버전 충돌 감지

파일 시스템의 `LastWriteTimeUtc`에서 불투명한 버전을 도출하여 경량의 낙관적 동시성 제어를 구현합니다. 여러 사용자가 동시에 같은 파일을 편집할 때 시스템이 충돌을 감지하고 사용자에게 새로고침을 요청할 수 있습니다. 이 설계는 편집을 차단하지 않으면서도 데이터 일관성을 보호합니다.

#### 결정 4: 동일 출처 프록시 패턴

`IHttpClientFactory`를 사용하여 외부 디자인 사이트 리소스를 프록시하여, 크로스 오리진 문제와 SSRF 위험을 모두 피합니다. 시스템 보안을 유지하면서 프론트엔드 호출도 단순화합니다. 보안에 있어서는 너무 조심하는 것은 없습니다.

## 핵심 구현

### 1. DESIGN.md 직접 편집

#### 백엔드 구현

백엔드는 주로 경로 해석, 파일 읽기/쓰기 및 버전 관리를 담당합니다. 이러한 작업은 기본적이지만 필수적입니다. 집의 기초와 같습니다.

```csharp
// 경로 해석 및 보안 검증
private Task<string> ResolveDesignDocumentDirectoryAsync(string projectPath, string? repositoryPath)
{
    if (string.IsNullOrWhiteSpace(repositoryPath))
    {
        return Task.FromResult(Path.GetFullPath(projectPath));
    }
    return ValidateSubPathAsync(projectPath, repositoryPath);
}

// 버전 생성 (파일 시스템 타임스탬프 및 크기 기반)
private static string BuildDesignDocumentVersion(string path)
{
    var fileInfo = new FileInfo(path);
    fileInfo.Refresh();
    return string.Create(
        CultureInfo.InvariantCulture,
        $"{fileInfo.LastWriteTimeUtc.Ticks:x}-{fileInfo.Length:x}");
}
```

버전 설계는 그 단순함에서 흥미롭습니다. 파일의 마지막 수정 시간과 크기를 사용하여 고유한 버전 식별자를 생성합니다. 경량이고 신뢰할 수 있으며, 추가적인 버전 데이터베이스를 유지할 필요가 없습니다. 간단한 해결책이 종종 가장 효과적입니다.

#### 프론트엔드 구현

프론트엔드에서는 더티 상태 감지와 저장 로직을 구현합니다. 이 설계는 사용자가 변경 사항이 저장되었는지 이해하도록 도와주며, "잃어버리면 어떡하지?"라는 불안감을 줄여줍니다.

```typescript
// 더티 상태 감지 및 저장 로직
const [draft, setDraft] = useState('');
const [savedDraft, setSavedDraft] = useState('');
const isDirty = draft !== savedDraft;

const handleSave = useCallback(async () => {
    const result = await saveProjectDesignMdDocument({
        ...activeTarget,
        content: draft,
        expectedVersion: document.version, // 낙관적 동시성 제어
    });
    setSavedDraft(draft); // 저장된 상태 업데이트
}, [activeTarget, document, draft]);
```

이 구현에서는 두 가지 상태를 유지합니다. `draft`는 현재 편집 중인 콘텐츠이고, `savedDraft`는 저장된 콘텐츠입니다. 둘을 비교하면 저장되지 않은 변경 사항이 있는지 알 수 있습니다. 설계는 간단하지만 사람들에게 안심을 줍니다. 열심히 작업한 것이 사라지기를 원하는 사람은 없으니까요.

#### 인덱스 데이터 형식

디자인 사이트의 인덱스 파일은 사용 가능한 모든 템플릿을 정의합니다. 이 인덱스가 있으면 사용자는 메뉴에서 주문하는 것처럼 쉽게 원하는 템플릿을 선택할 수 있습니다.

```json
{
  "entries": [
    {
      "slug": "linear.app",
      "title": "Linear Inspired Design System",
      "summary": "AI Product / Dark Feel",
      "detailUrl": "/designs/linear.app/",
      "designDownloadUrl": "/designs/linear.app/DESIGN.md",
      "previewLightImageUrl": "...",
      "previewDarkImageUrl": "..."
    }
  ]
}
```

각 항목에는 템플릿의 기본 정보와 다운로드 링크가 포함되어 있습니다. 백엔드는 이 인덱스에서 사용 가능한 템플릿 목록을 읽어 사용자가 선택할 수 있도록 표시합니다. 선택이 직관적이어서 어둠 속에서 더듬을 필요가 없습니다.

#### 동일 출처 프록시 구현

보안을 위해 백엔드는 디자인 사이트에 대한 접근을 엄격하게 검증합니다. 보안에 있어서는 너무 조심하는 것은 없습니다.

```csharp
// 안전한 슬러그 검증
private static readonly Regex SafeDesignSiteSlugRegex =
    new("^[A-Za-z0-9](?:[A-Za-z0-9._-]{0,127})$", RegexOptions.Compiled);

private static string NormalizeDesignSiteSlug(string slug)
{
    var normalizedSlug = slug?.Trim() ?? string.Empty;
    if (!IsSafeDesignSiteSlug(normalizedSlug))
    {
        throw new BusinessException(
            ProjectDesignSiteIndexErrorCodes.InvalidSlug,
            "Design site slug must be a single safe path segment.");
    }
    return normalizedSlug;
}

// 미리보기 이미지 캐싱 (OS 임시 디렉토리)
private static string ComputePreviewCacheKey(string slug, string theme, string previewUrl)
{
    var raw = $"{slug}|{theme}|{previewUrl}";
    var bytes = SHA256.HashData(Encoding.UTF8.GetBytes(raw));
    return Convert.ToHexString(bytes).ToLowerInvariant();
}
```

여기서 두 가지를 합니다. 첫째, 정규식으로 슬러그 형식을 엄격하게 검증하여 경로 순회 공격을 방지합니다. 둘째, 미리보기 이미지를 캐싱하여 외부 사이트의 부하를 줄입니다. 전자는 보호이고, 후자는 최적화이며, 둘 다 중요합니다.

### 3. 전체 가져오기 흐름

```typescript
// 1. 가져오기 드로어 열기
const handleRequestImportDrawer = useCallback(() => {
    setIsImportDrawerOpen(true);
}, []);

// 2. 선택 및 가져오기
const handleImportRequest = useCallback((entry) => {
    if (isDirty) {
        setPendingImportEntry(entry);
        setConfirmMode('import'); // 덮어쓰기 확인
        return;
    }
    void executeImport(entry);
}, [isDirty]);

// 3. 가져오기 실행
const executeImport = useCallback(async (entry) => {
    const result = await getProjectDesignMdSiteImportDocument(
        activeTarget.projectId,
        entry.slug
    );
    setDraft(result.content); // 에디터 텍스트만 교체, 자동 저장하지 않음
    setIsImportDrawerOpen(false);
}, [activeTarget?.projectId]);
```

가져오기 흐름은 "사용자 확인" 원칙을 따릅니다. 가져오기 후 에디터 콘텐츠만 업데이트되며, 자동으로 저장되지 않습니다. 사용자는 가져온 콘텐츠를 검토하고 확인 후 수동으로 저장할 수 있습니다. 최종 결정은 사용자의 손에 있어야 합니다.

## 실용 예시

### 시나리오 1: 프로젝트 루트에 DESIGN.md 생성

`DESIGN.md`가 존재하지 않을 때, 백엔드는 가상 문서 상태를 반환합니다. 이렇게 하면 프론트엔드에서 "파일이 존재하지 않음" 경우에 대한 특별한 처리를 피할 수 있으며, 통합된 API가 코드 로직을 단순화합니다.

```csharp
return new ProjectDesignDocumentDto
{
    Path = targetPath,
    Exists = false,  // 가상 문서 상태
    Content = string.Empty,
    Version = null
};

// 첫 번째 저장 시 파일 자동 생성
public async Task<SaveProjectDesignDocumentResultDto> SaveDesignDocumentAsync(...)
{
    Directory.CreateDirectory(targetDirectory);
    await File.WriteAllTextAsync(targetPath, input.Content);
    return new SaveProjectDesignDocumentResultDto { Created = !exists };
}
```

이 설계의 이점은 프론트엔드에서 누락된 파일에 대한 특별 케이스 로직이 필요하지 않다는 것입니다. 그 복잡성을 백엔드에 숨김으로써 프론트엔드는 사용자 경험에 더 집중할 수 있습니다.

### 시나리오 2: 디자인 사이트에서 템플릿 가져오기

사용자가 가져오기 드로어에서 "Linear" 디자인 템플릿을 선택하면, 시스템은 백엔드 프록시를 통해 `DESIGN.md` 콘텐츠를 가져옵니다. 전체 과정은 사용자에게 투명합니다. 템플릿을 선택하기만 하면 시스템이 네트워크 요청과 데이터 변환을 자동으로 처리합니다.

```typescript
// 1. 시스템이 백엔드 프록시를 통해 DESIGN.md 콘텐츠를 가져옵니다
GET /api/project/{id}/design-md/site-index/linear.app

// 2. 백엔드가 슬러그를 검증하고 업스트림에서 콘텐츠를 가져옵니다
var entry = FindDesignSiteEntry(catalog, "linear.app");
using var upstreamResponse = await httpClient.SendAsync(request);
var content = await upstreamResponse.Content.ReadAsStringAsync();

// 3. 프론트엔드가 에디터 텍스트를 교체합니다
setDraft(result.content);
// 사용자가 검토한 후 수동으로 디스크에 저장합니다
```

전체 흐름은 사용자에게 투명합니다. 템플릿을 선택하기만 하면 시스템이 네트워킹과 변환을 뒤에서 처리합니다. 이것이 우리가 원하는 경험입니다. 간단하지만 강력합니다.

### 시나리오 3: 버전 충돌 처리

여러 사용자가 동시에 같은 `DESIGN.md`를 편집할 때, 시스템이 버전 충돌을 감지합니다. 이 낙관적 동시성 제어 메커니즘은 사용자의 편집을 차단하지 않으면서 데이터 일관성을 유지합니다.

```csharp
if (!string.Equals(currentVersion, expectedVersion, StringComparison.Ordinal))
{
    throw new BusinessException(
        ProjectDesignDocumentErrorCodes.VersionConflict,
        $"DESIGN.md at '{targetPath}' changed on disk.");
}
```

프론트엔드는 이 오류를 감지하고 사용자에게 알립니다.

```typescript
// 프론트엔드가 사용자에게 새로고침 및 재시도를 요청합니다
<Alert>
    <AlertTitle>버전 충돌</AlertTitle>
    <AlertDescription>
        파일이 다른 프로세스에 의해 수정되었습니다. 최신 버전을 새로고침한 후 다시 시도해 주세요.
    </AlertDescription>
</Alert>
```

이 낙관적 동시성 제어 메커니즘은 사용자가 작업하는 동안 데이터 일관성을 유지합니다. 충돌은 피할 수 없지만, 적어도 사용자는 무엇이 일어났는지 알아야 합니다. 변경 사항이 조용히 사라지는 것보다 낫습니다.

## 주의 사항 및 모범 사례

### 1. 경로 보안

항상 `repositoryPath`를 검증하여 경로 순회 공격을 방지해야 합니다. 보안에 있어서는 너무 많이 하는 것은 없습니다.

```csharp
// 항상 repositoryPath를 검증하여 경로 순회 공격을 방지합니다
return ValidateSubPathAsync(projectPath, repositoryPath);
// "../" 및 절대 경로와 같은 위험한 입력 거부
```

### 2. 캐시 전략

미리보기 이미지를 24시간 동안 캐시하며, 최대 160개의 파일을 유지합니다. 적절한 캐싱은 성능을 향상시키지만, 균형도 중요합니다.

```csharp
// 미리보기 이미지를 24시간 동안 캐시, 최대 160개 파일
private static readonly TimeSpan PreviewCacheTtl = TimeSpan.FromHours(24);
private const int PreviewCacheMaxFiles = 160;
// 정기적으로 만료된 캐시 정리
```

### 3. 오류 처리

업스트림 사이트를 사용할 수 없을 때 우아하게 저하됩니다. 이 설계는 외부 종속성이 실패하더라도 핵심 편집 기능이 정상적으로 작동하도록 보장합니다.

```typescript
// 업스트림 사이트를 사용할 수 없을 때 우아하게 저하
try {
    const catalog = await getProjectDesignMdSiteImportCatalog(projectId);
} catch (error) {
    toast.error(t('project.designMd.siteImport.feedback.catalogLoadFailed'));
    // 주 편집 드로어는 계속 사용 가능
}
```

이 우아한 저하는 외부 종속성을 사용할 수 없을 때도 핵심 편집 기능이 계속 작동하도록 보장합니다. 시스템은 무언가 잘못되자마자 무너지는 것이 아니라 회복력이 있어야 합니다.

### 4. 사용자 경험 최적화

가져오기 전에 덮어쓰기를 확인하고, 가져오기 후 자동 저장하지 않습니다. 사용자가 자신의 행동을 통제할 수 있도록 해야 합니다.

```typescript
// 가져오기 전 덮어쓰기 확인
if (isDirty) {
    setConfirmMode('import');
    return;
}

// 가져오기 후 자동 저장하지 않음. 사용자가 확인하도록 합니다
setDraft(result.content); // 드래프트만 업데이트
// 콘텐츠는 사용자가 검토하고 저장을 클릭한 후에 디스크에 기록됩니다
```

### 5. 성능 고려 사항

HTTP 클라이언트 팩토리를 사용하여 너무 많은 연결 생성을 피합니다. 리소스 관리는 작아 보일 수 있지만, 잘하면 큰 차이를 만들 수 있습니다.

```csharp
// HTTP 클라이언트 팩토리를 사용하여 너무 많은 연결 생성을 피합니다
private const string DesignSiteProxyClientName = "ProjectDesignSiteProxy";
```

## 제안 확장

1. **Markdown 향상**: 현재 기본 `Textarea`를 사용하지만, 구문 강조 및 키보드 단축키를 위해 CodeMirror로 업그레이드할 수 있습니다. 에디터가 더 좋아지면 문서를 작성하는 것도 더 좋아집니다.
2. **미리보기 모드**: 실시간 Markdown 미리보기를 추가하여 편집 경험을 향상시킵니다. WYSIWYG는 항상 사람들에게 더 큰 자신감을 줍니다.
3. **Diff 병합**: 단순한 전체 텍스트 교체 대신 지능적인 병합 알고리즘을 구현합니다. 충돌은 피할 수 없지만, 충돌 해결 과정이 고통스러울 필요는 없습니다.
4. **로컬 캐싱**: `design.json`을 데이터베이스에 캐싱하여 외부 사이트에 대한 의존성을 줄입니다. 시스템의 종속성이 적을수록 더 안정적입니다.

HagiCode 프로젝트에서 프론트엔드-백엔드 협업을 통해 `DESIGN.md`에 대한 완전한 온라인 편집 및 가져오기 해결책을 구현했습니다. 이 해결책의 핵심 가치는 다음과 같습니다.

- **더 높은 효율성**: 도구 전환이 불필요하며, 편집 및 디자인 문서 가져오기를 하나의 통합된 웹 인터페이스에서 수행할 수 있습니다.
- **더 낮은 진입 장벽**: 원클릭 디자인 템플릿 가져오기로 새 프로젝트를 빠르게 시작할 수 있습니다.

- **더 나은 사용자 경험**: 글로벌 드로어, 더티 상태 감지 및 확인 대화 상자가 전체 상호 작용 경험을 정교하게 합니다.
