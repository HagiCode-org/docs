## Web 인터페이스에서 DESIGN.md 직접 편집하기: 아이디어부터 구현까지

> MonoSpecs 프로젝트 관리 시스템에서 DESIGN.md는 프로젝트의 아키텍처 설계와 기술적 결정을 담고 있습니다. 하지만 전통적인 편집 방식은 사용자가 외부 편집기로 전환해야만 했는데, 이런 단절된 프로세스는 마치 시를 읽다가 갑자기 중단된 것과 같습니다. 영감도 사라지고 기분도 망가지죠. 이 글에서는 HagiCode 프로젝트에서 실천한 우리의 솔루션을 공유합니다: Web 인터페이스에서 DESIGN.md를 직접 편집하고, 온라인 디자인 사이트에서 템플릿을 가져오는 기능을 지원합니다. 한 번에 완성되는 느낌, 싫어하는 사람이 있을까요?

## 배경

`DESIGN.md`는 프로젝트 설계 문서의 핵심 매체로서 아키텍처 설계, 기술적 결정, 구현 가이드 등 핵심 정보를 담고 있습니다. 그러나 전통적인 편집 방식은 사용자가 외부 편집기(예: VS Code)로 전환하여 물리적 경로를 수동으로 찾은 후 편집해야 했습니다. 이 프로세스가 복잡하다고는 할 수 없지만, 몇 번 반복하다 보면 사람도 지치게 됩니다.

구체적인 문제는 다음과 같은 몇 가지 측면에서 나타납니다:

- **프로세스 단절**: Web 관리 인터페이스와 로컬 편집기 사이를 자주 전환해야 하며, 워크플로우의 연속성이 깨집니다. 마치 음악을 듣다가 갑자기 인터넷이 끊긴 것처럼, 리듬이 완전히 깨지죠.
- **재사용 어려움**: 디자인 사이트에는 이미 풍부한 디자인 템플릿 라이브러리가 게시되어 있지만, 프로젝트 편집 프로세스에 직접 통합할 수 없습니다. 좋은 것이 있는데 쓸 수 없다는 것, 확실히 아쉬운 일입니다.
- **경험 부재**: "미리보기-선택-가져오기"의闭环이 부족하여 사용자가 수동으로 복사하여 붙여넣어야 하며, 오류 위험이 증가합니다. 수동 작업이 많으면 많을수록 오류가 발생할 기회도 naturally 많아집니다.
- **협업 장애**: 설계 문서와 코드 구현의 동기화 유지 관리가 마찰이 높고, 팀 협업 효율을 방해합니다. 팀 협업이 본래 쉽지 않은데, 왜 더 많은 저항을 추가하는 걸까요?

이러한 문제점을 해결하기 위해 우리는 Web 인터페이스에서 DESIGN.md의 직접 편집 기능을 구현하고, 온라인 디자인 사이트에서 원클릭으로 템플릿을 가져오는 기능을 지원하기로 결정했습니다. 이거 별로 대단한 결정은 아니지만, 개발 경험을 조금 더 원활하게 만들고 싶었을 뿐입니다.

## HagiCode에 대해

이 글에서 공유하는 솔루션은 [HagiCode](https://hagicode.com) 프로젝트에서의 실천 경험에서 나왔습니다. HagiCode는 AI 기반 코드 어시스턴트 프로젝트로, 개발 과정에서 프로젝트의 설계 문서를 자주 유지 관리해야 합니다. 팀이 더 효율적으로 협업할 수 있도록 우리는 이 온라인 편집 및 가져오기 솔루션을 탐색하고 구현했습니다. 사실 특별한 건 없고, 문제에 부딪혔을 때 해결 방법을 찾았을 뿐입니다.

### 핵심 기술적 결정

#### 결정 1: 전역 드로어 모드

단일 전역 드로어를 사용하고 로컬 팝업을 사용하지 않으며, `layoutSlice`로 상태를 관리하여 뷰(classic/kanban) 간의 일관된 경험을 실현합니다. 이 방식은 사용자가 어떤 뷰에서든 편집기를 열 때 통일된 상호작용 경험을 얻을 수 있도록 보장합니다. 일관된 경험은 사용자가 더 편안하게 느끼게 하며, 뷰를 바꿨다고 해서 길을 잃지 않게 합니다.

#### 결정 2: 프로젝트 범위 API

DESIGN.md 관련 인터페이스를 `ProjectController` 아래에 배치하고 기존 프로젝트 권한 경계를 재사용하여, 독립 컨트롤러를 새로 추가하는 복잡성을 피했습니다. 이렇게 설계的好处은 권한 관리가 더 명확하고 RESTful 리소스 구성 원칙에도 부합합니다. 때로는 재사용이 새로 만드는 것보다 더 의미 있지 않을까요?

#### 결정 3: 버전 충돌 감지

파일 시스템 `LastWriteTimeUtc`를 기반으로 불투명 버전을 파생하여 가벼운 낙관적 동시성 제어를 실현했습니다. 여러 사용자가 동시에 같은 파일을 편집할 때 시스템은 충돌을 감지하고 사용자에게 새로고침을 안내합니다. 이 설계는 사용자의 편집 작업을 차단하지 않으면서도 데이터 일관성을 보장합니다. 마치 대인 관계에서의 경계감과 같아서, 과하게 거리를 두지도 않고 넘지도 않습니다.

#### 결정 4: 동일 출처 프록시 모드

`IHttpClientFactory`를 통해 외부 디자인 사이트 리소스를 프록시하여 CORS 문제와 SSRF 위험을 피했습니다. 이 설계는 보안을 보장하면서도 프론트엔드 호출을 단순화합니다. 보안은 아무리 많이 해도 지나치지 않습니다. 데이터 보안은 건강과 같아서, 잃어버린 후에야 후회하면 늦습니다.

## 핵심 구현

### 1. DESIGN.md 직접 편집

#### 백엔드 구현

백엔드는 주로 경로 해석, 파일 읽기/쓰기 및 버전 관리를 담당합니다. 이 작업들은 기초적이지만 필수적입니다. 마치 집의 기초와 같습니다:

```csharp
// 경로 해석 및 보안 검증
private Task<string> ResolveDesignDocumentDirectoryAsync(string projectPath, string? repositoryPath)
{
    if (string.IsNullOrWhiteSpace(repositoryPath))
    {
        return Task.FromResult(Path.GetFullPath(projectPath));
    }
    return ValidateSubPathAsync(projectPath, repositoryPath);
}

// 버전 번호 생성(파일 시스템 타임스탬프 및 크기 기반)
private static string BuildDesignDocumentVersion(string path)
{
    var fileInfo = new FileInfo(path);
    fileInfo.Refresh();
    return string.Create(
        CultureInfo.InvariantCulture,
        $"{fileInfo.LastWriteTimeUtc.Ticks:x}-{fileInfo.Length:x}");
}
```

버전 번호 설계도 사실 꽤 흥미롭습니다. 우리는 파일의 마지막 수정 시간과 크기를 사용하여 고유한 버전 식별자를 생성합니다. 이렇게 하면 가볍고 신뢰할 수 있으며, 추가 버전 데이터베이스를 유지할 필요가 없습니다. 간단한 것이 종종 더 효과적이지 않을까요?

#### 프론트엔드 구현

프론트엔드는 더티 상태 감지와 저장 로직을 구현했습니다. 이 설계는 사용자가 자신의 수정 사항이 저장되었는지 언제든지 알 수 있게 하여 "만약에 잃어버리면 어떡하지?"라는 불안을 줄여줍니다:

```typescript
// 더티 상태 감지 및 저장 로직
const [draft, setDraft] = useState('');
const [savedDraft, setSavedDraft] = useState('');
const isDirty = draft !== savedDraft;

const handleSave = useCallback(async () => {
    const result = await saveProjectDesignMdDocument({
        ...activeTarget,
        content: draft,
        expectedVersion: document.version, // 낙관적 동시성 제어
    });
    setSavedDraft(draft); // 저장된 상태 업데이트
}, [activeTarget, document, draft]);
```

이 구현에서 우리는 두 가지 상태를 유지합니다: `draft`는 현재 편집 중인 내용, `savedDraft`는 이미 저장된 내용입니다. 둘을 비교하여 저장되지 않은 수정 사항이 있는지 판단합니다. 이 설계는 간단하지만 안심감을 줍니다. 고생해서 작성한 내용이 갑자기 사라지는 건 누구도 원하지 않으니까요.

#### 인덱스 데이터 형식

디자인 사이트의 인덱스 파일은 사용 가능한 모든 템플릿을 정의합니다. 이 인덱스가 있으면 사용자는 레스토랑에서 주문하듯 쉽게 원하는 템플릿을 선택할 수 있습니다:

```json
{
  "entries": [
    {
      "slug": "linear.app",
      "title": "Linear Inspired Design System",
      "summary": "AI 제품 / 다크 감성",
      "detailUrl": "/designs/linear.app/",
      "designDownloadUrl": "/designs/linear.app/DESIGN.md",
      "previewLightImageUrl": "...",
      "previewDarkImageUrl": "..."
    }
  ]
}
```

각 항목은 템플릿의 기본 정보와 다운로드 링크를 포함합니다. 백엔드는 이 인덱스에서 사용 가능한 템플릿 목록을 읽은 다음 사용자가 선택할 수 있도록 표시합니다. 이 설계는 선택을 직관적으로 만들어 어둠 속에서摸索하는 것이 아니라 명확하게 보여줍니다.

#### 동일 출처 프록시 구현

보안을 보장하기 위해 백엔드는 디자인 사이트 접근에 대해 엄격한 검증을 수행했습니다. 보안, 아무리小心해도 지나치지 않습니다:

```csharp
// 안전한 slug 검증
private static readonly Regex SafeDesignSiteSlugRegex =
    new("^[A-Za-z0-9](?:[A-Za-z0-9._-]{0,127})$", RegexOptions.Compiled);

private static string NormalizeDesignSiteSlug(string slug)
{
    var normalizedSlug = slug?.Trim() ?? string.Empty;
    if (!IsSafeDesignSiteSlug(normalizedSlug))
    {
        throw new BusinessException(
            ProjectDesignSiteIndexErrorCodes.InvalidSlug,
            "Design site slug must be a single safe path segment.");
    }
    return normalizedSlug;
}

// 미리보기 이미지 캐싱(OS 임시 디렉터리)
private static string ComputePreviewCacheKey(string slug, string theme, string previewUrl)
{
    var raw = $"{slug}|{theme}|{previewUrl}";
    var bytes = SHA256.HashData(Encoding.UTF8.GetBytes(raw));
    return Convert.ToHexString(bytes).ToLowerInvariant();
}
```

여기서 우리는 두 가지를 했습니다: 하나는 정규 표현식으로 slug 형식을 엄격하게 검증하여 경로 순회 공격을 방지하는 것이고, 다른 하나는 미리보기 이미지를 캐싱하여 외부 사이트에 대한 요청 압력을 줄이는 것입니다. 전자는 방어, 후자는 최적화로서, 둘 다 필수적일 뿐입니다.

### 3. 완전한 가져오기 프로세스

```typescript
// 1. 가져오기 드로어 열기
const handleRequestImportDrawer = useCallback(() => {
    setIsImportDrawerOpen(true);
}, []);

// 2. 선택 및 가져오기
const handleImportRequest = useCallback((entry) => {
    if (isDirty) {
        setPendingImportEntry(entry);
        setConfirmMode('import'); // 덮어쓰기 확인
        return;
    }
    void executeImport(entry);
}, [isDirty]);

// 3. 가져오기 실행
const executeImport = useCallback(async (entry) => {
    const result = await getProjectDesignMdSiteImportDocument(
        activeTarget.projectId,
        entry.slug
    );
    setDraft(result.content); // 편집기 텍스트만 교체하고 자동 저장하지 않음
    setIsImportDrawerOpen(false);
}, [activeTarget?.projectId]);
```

가져오기 프로세스 설계는 "사용자 확인" 원칙을 따릅니다: 가져온 후에 편집기 내용만 업데이트하고 자동 저장하지 않습니다. 사용자는 가져온 내용을 검사하고 확인 후 수동으로 저장할 수 있습니다.毕竟, 사용자는 자신이 작성한 것에 최종 결정권을 가져야 하지 않을까요?

## 실천 예시

### 시나리오 1: 프로젝트 루트 디렉터리 DESIGN.md 생성

DESIGN.md가 없을 때 백엔드는 가상 문서 상태를 반환합니다. 이 설계는 프론트엔드가 "파일 없음" 상황을 특별히 처리할 필요가 없게 하며, 통일된 API 인터페이스가 코드 로직을 단순화합니다:

```csharp
return new ProjectDesignDocumentDto
{
    Path = targetPath,
    Exists = false,  // 가상 문서 상태
    Content = string.Empty,
    Version = null
};

// 처음 저장할 때 파일 자동 생성
public async Task<SaveProjectDesignDocumentResultDto> SaveDesignDocumentAsync(...)
{
    Directory.CreateDirectory(targetDirectory);
    await File.WriteAllTextAsync(targetPath, input.Content);
    return new SaveProjectDesignDocumentResultDto { Created = !exists };
}
```

이 설계的好处은 프론트엔드가 "파일 없음" 상황을 특별히 처리할 필요가 없다는 점입니다. 통일된 API 인터페이스가 코드 로직을 단순화합니다. 때로는 복잡성을 백엔드에 숨기면 프론트엔드가 사용자 경험에 더 집중할 수 있습니다.

### 시나리오 2: 디자인 사이트에서 템플릿 가져오기

사용자가 가져오기 드로어에서 "Linear" 디자인 템플릿을 선택하면 시스템은 백엔드 프록시를 통해 DESIGN.md 내용을 가져옵니다. 전체 프로세스는 사용자에게 투명하며, 그들은 템플릿을 선택하기만 하면 시스템이 자동으로 모든 네트워크 요청과 데이터 변환을 처리합니다:

```typescript
// 1. 시스템은 백엔드 프록시를 통해 DESIGN.md 내용을 가져옵니다
GET /api/project/{id}/design-md/site-index/linear.app

// 2. 백엔드는 slug를 검증하고 업스트림에서 내용을 가져옵니다
var entry = FindDesignSiteEntry(catalog, "linear.app");
using var upstreamResponse = await httpClient.SendAsync(request);
var content = await upstreamResponse.Content.ReadAsStringAsync();

// 3. 프론트엔드는 편집기 텍스트를 교체합니다
setDraft(result.content);
// 사용자가 검사한 후 수동으로 디스크에 저장합니다
```

전체 프로세스는 사용자에게 투명하며, 그들은 템플릿을 선택하기만 하면 시스템이 자동으로 모든 네트워크 요청과 데이터 변환을 처리합니다. 사용자는背后的 복잡성에 신경 쓸 필요가 없습니다.这就是 우리가 추구하는 경험—간단하지만 강력합니다.

### 시나리오 3: 버전 충돌 처리

여러 사용자가 동시에 같은 DESIGN.md를 편집할 때 시스템은 버전 충돌을 감지합니다. 이 낙관적 동시성 제어 메커니즘은 데이터 일관성을 보장하면서도 사용자의 편집 작업을 차단하지 않습니다:

```csharp
if (!string.Equals(currentVersion, expectedVersion, StringComparison.Ordinal))
{
    throw new BusinessException(
        ProjectDesignDocumentErrorCodes.VersionConflict,
        $"DESIGN.md at '{targetPath}' changed on disk.");
}
```

프론트엔드는 이 오류를 포착하고 사용자에게 안내합니다:

```typescript
// 프론트엔드 사용자에게 새로고침하고 다시 시도하라는 안내
<Alert>
    <AlertTitle>버전 충돌</AlertTitle>
    <AlertDescription>
        파일이 다른 프로세스에서 수정되었습니다. 최신 버전으로 새로고침한 후 다시 시도하세요.
    </AlertDescription>
</Alert>
```

이 낙관적 동시성 제어 메커니즘은 데이터 일관성을 보장하면서도 사용자의 편집 작업을 차단하지 않습니다. 충돌은 피할 수 없지만 최소한 사용자에게 무슨 일이 일어났는지 알려주어 조용히 수정 사항이 사라지지 않게 할 수 있습니다.

## 주의 사항 및 모범 사례

### 1. 경로 보안

항상 `repositoryPath`를 검증하여 경로 순회 공격을 방지하세요. 보안은 아무리 많이 해도 지나치지 않습니다:

```csharp
// 항상 repositoryPath를 검증하여 경로 순회 공격 방지
return ValidateSubPathAsync(projectPath, repositoryPath);
// "../", 절대 경로 등 위험한 입력 거부
```

### 2. 캐싱 전략

미리보기 이미지는 24시간, 최대 160개 파일까지 캐시합니다. 적절한 캐싱은 성능을 높이지만 과도하면 안 됩니다.毕竟 균형이 핵심입니다:

```csharp
// 미리보기 이미지 캐시 24시간, 최대 160개 파일
private static readonly TimeSpan PreviewCacheTtl = TimeSpan.FromHours(24);
private const int PreviewCacheMaxFiles = 160;
// 정기적으로 만료된 캐시 정리
```

### 3. 오류 처리

업스트림 사이트를 사용할 수 없을 때는 폴백 처리합니다. 이 우아한 폴백 설계는 외부 종속성을 사용할 수 없더라도 핵심 편집 기능이 정상적으로 작동하도록 보장합니다.毕竟, 외부 서비스 하나가 죽었다고 해서 전체 시스템이 마비되어서는 안 됩니다:

```typescript
// 업스트림 사이트를 사용할 수 없을 때 폴백 처리
try {
    const catalog = await getProjectDesignMdSiteImportCatalog(projectId);
} catch (error) {
    toast.error(t('project.designMd.siteImport.feedback.catalogLoadFailed'));
    // 주요 편집 드로어는 여전히 사용 가능
}
```

이 우아한 폴백 설계는 외부 종속성을 사용할 수 없더라도 핵심 편집 기능이 정상적으로 작동하도록 보장합니다. 시스템은 탄력적이어야 하지 문제에 부딪히면 바로 넘어지면 안 됩니다.

### 4. 사용자 경험 최적화

가져오기 전에 덮어쓰기 확인, 가져온 후 자동 저장하지 않음. 사용자는 자신의 작업에 제어권을 가져야 하지 시스템이 임의로 결정해서는 안 됩니다:

```typescript
// 가져오기 전에 덮어쓰기 확인
if (isDirty) {
    setConfirmMode('import');
    return;
}

// 가져온 후 자동 저장하지 않고 사용자 확인
setDraft(result.content); //草稿만 업데이트
// 사용자가 검사한 후 Save를 클릭해야 실제로 디스크에 쓰기
```

### 5. 성능 고려

HTTP 클라이언트 팩토리를 사용하여 너무 많은 연결 생성을 피하세요. 리소스 관리는 보잘것없어 보이지만 잘하면 예상치 못한 효과를 줄 수 있습니다:

```csharp
// HTTP 클라이언트 팩토리 사용, 너무 많은 연결 생성 방지
private const string DesignSiteProxyClientName = "ProjectDesignSiteProxy";
private static readonly TimeSpan DesignSiteProxyTimeout = TimeSpan.FromSeconds(8);
```

## 확장 제안

1. **Markdown 향상**: 현재 기본 `Textarea`를 사용 중이지만, CodeMirror로 업그레이드하여 구문 강조와 바로가기 키를 지원할 수 있습니다. 편집기 경험이 좋아지면 문서를 작성하는 기분도 좋아집니다.
2. **미리보기 모드**: Markdown 실시간 미리보기를 추가하여 편집 경험을 높이세요. WYSIWYG은 항상 더 많은自信을 줍니다.
3. **차이 병합**: 지능형 병합 알고리즘을 구현하여 간단한 전체 텍스트 교체가 아닌 방식으로. 충돌은 피할 수 없지만 최소한 처리 프로세스를 덜 고통스럽게 만들 수 있습니다.
4. **로컬 캐싱**: design.json을 데이터베이스에 캐싱하여 외부 사이트에 대한 종속성을 줄이세요. 종속성이 적을수록 시스템이 더 안정적입니다. 이것은 간단한 진리입니다.

## 요약

HagiCode 프로젝트에서 우리는 프론트엔드-백엔드 협업을 통해 완전한 DESIGN.md 온라인 편집 및 가져오기 솔루션을 구현했습니다. 이 솔루션의 핵심 가치는 다음과 같습니다:

- **효율성 향상**: 도구 전환 없이 통일된 Web 인터페이스에서 설계 문서 편집 및 가져오기를 완료합니다. 절약한 시간으로 더 의미 있는 일을 할 수 있습니다.
- **진입 장벽 낮춤**: 디자인 템플릿을 원클릭으로 가져오고 새 프로젝트를 빠르게 시작할 수 있습니다. 시작하기 쉬울수록坚持下去 가능성이 커집니다.
- **안전하고 신뢰할 수 있음**: 경로 검증, 버전 충돌 감지, 우아한 폴백 등 메커니즘이 시스템 안정적인 실행을 보장합니다. 안정성은 기반입니다. 안정성이 없으면 모든 것이 공담입니다.
- **사용자 경험**: 전역 드로어, 더티 상태 감지, 확인 대화 상자 등 세부 사항이 상호작용 경험을 다듬었습니다. 세부 사항이 성패를 결정합니다. 특히 사용자 경험에서는.

이 솔루션은 HagiCode 프로젝트에서 실제로 실행되었으며 팀의 설계 문서 관리 문제를 해결했습니다. 만약 당신도 비슷한 문제에 직면해 있다면 이 글이 당신에게 영감을 주기를 바랍니다. 사실 고심한 이론은 없고 문제에 부딪혔을 때 해결 방법을 찾았을 뿐입니다.
