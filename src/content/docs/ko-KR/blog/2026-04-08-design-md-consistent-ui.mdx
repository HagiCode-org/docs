## Design.md: AI 기반 프론트엔드 UI 디자인 일관성을 위한 해결책

> AI 보조 프론트엔드 개발 시대에 AI가 생성한 UI의 일관성을 어떻게 유지할 수 있을까요? 이 글에서는 awesome-design-md 기반 디자인 갤러리 사이트 구축 실무 경험을 공유하고, AI가 표준화된 UI 디자인을 수행하도록 안내하는 구조화된 `design.md`를 만드는 방법을 소개합니다.

## 배경

AI를 사용하여 프론트엔드 코드를 작성해 본 사람이라면 아마 비슷한 경험을 했을 것입니다. AI에게 동일한 페이지를 여러 번 생성하도록 요청하면, 매번 다른 스타일로 나옵니다. 때로는 모서리가 둥글고, 때로는 각져 있습니다. 간격이 8px일 때도 있고, 16px가 될 때도 있습니다. 심지어 동일한 버튼도 대화마다 다르게 보일 수 있습니다.

이것은 고립된 문제가 아닙니다. AI 보조 개발이 보편화되면서 **AI가 생성한 프론트엔드 UI의 일관성 부족**은 널리 퍼진 문제가 되었습니다. 다른 AI 어시스턴트, 다른 프롬프트, 심지어 동일한 어시스턴트도 다른 대화에서 극적으로 다른 인터페이스 디자인을 만들어냅니다. 이는 제품 반복 과정에서 막대한 유지 관리 비용을 초래합니다.

근본 원인은 사실 간단합니다. **권위 있는 디자인 참조 문서**가 없기 때문입니다. 전통적인 CSS 스타일시트는 개발자에게 "어떻게 구현하는지"를 알려줄 수 있지만, "왜 이렇게 디자인되었는지"나 "어떤 시나리오에서 어떤 디자인 패턴을 사용해야 하는지"를 완전히 전달할 수는 없습니다. AI에게는 명확하고 구조화된 설명이 디자인 규칙을 이해하는 데 더욱 중요합니다.

동시에, 오픈소스 커뮤니티에는 이미 훌륭한 리소스들이 있습니다. [VoltAgent/awesome-design-md](https://github.com/VoltAgent/awesome-design-md) 프로젝트는 많은 잘 알려진 기업의 디자인 시스템 문서를 수집합니다. 각 디렉토리에는 `README.md`, `DESIGN.md`, 미리보기 HTML이 포함되어 있습니다. 하지만 이 모든 것이 업스트림 저장소에 흩어져 있어 빠르게 탐색하고 비교하기 어렵습니다.

그렇다면 이러한 리소스를 탐색하기 쉬운 디자인 갤러리로 통합하면서, AI가 사용할 수 있는 구조화된 `design.md`도 정리할 수 있을까요?

대답은 '예'입니다. 다음으로 우리의 접근 방식을 설명하겠습니다.

## 분석

코드를 작성하기 전에 이 문제 뒤의 기술적 과제를 분석해 보겠습니다.

### 소스 콘텐츠 관리: 흩어진 디자인 리소스를 어떻게 통합합니까?

업스트림 awesome-design-md 저장소에는 많은 수의 디자인 문서가 포함되어 있지만, 이를 우리 프로젝트로 가져오는 방법이 필요합니다.

**해결책: git 서브모듈 사용**

```
awesome-design-md-site
└── vendor/awesome-design-md          # 업스트림 리소스 (git 서브모듈)
```

이렇게 하면 여러 가지 이점이 있습니다:
- 버전 관리: 특정 업스트림 버전을 고정할 수 있습니다
- 오프라인 빌드: 빌드 시 외부 API에 요청할 필요가 없습니다
- 콘텐츠 검토: 특정 변경 사항이 PR에서 볼 수 있습니다

### 데이터 정규화: 다른 문서 구조를 어떻게 표준화합니까?

각 기업의 디자인 문서 구조가 다릅니다. 일부는 미리보기 파일이 누락되어 있고, 일부는 명명이 일관되지 않습니다. 빌드 과정에서 정규화해야 합니다.

**해결책: 빌드 시 스캔 및 정규화된 항목 생성**

핵심 모듈은 `awesomeDesignCatalog.ts`로, 다음을 담당합니다:
1. `vendor/awesome-design-md/design-md/*` 디렉토리 스캔
2. 각 항목에 필수 파일(`README.md`, `DESIGN.md`, 최소 하나의 미리보기 파일)이 포함되어 있는지 검증
3. Markdown 콘텐츠를 추출하여 HTML로 렌더링
4. 정규화된 항목 데이터 생성

### 정적 사이트 아키텍처: 완전히 정적인 상태에서 동적 검색을 어떻게 제공합니까?

디자인 갤러리이므로 검색은 필수입니다. 하지만 Astro는 정적 사이트 생성기이므로 실시간 검색을 어떻게 구현할 수 있을까요?

**해결책: React 아일랜드 + URL 쿼리 매개변수 동기화**

```typescript
// src/components/gallery/SearchToolbar.tsx

export function SearchToolbar() {

```

이 접근 방식의 장점은 정적 사이트의 배포 가능성을 유지한다는 것입니다. 즉, 모든 정적 호스팅 서비스에 배포할 수 있으면서도 즉각적인 필터링 경험을 제공합니다.

### 디자인 문서화: AI가 디자인 표준을 이해하고 따르도록 어떻게 돕습니까?

이것은 전체 해결책의 핵심입니다. AI가 이해하고 적용할 수 있는 구조화된 `design.md`를 만들어야 합니다.

**해결책: ClickHouse `DESIGN.md`의 구조를 차용**

[ClickHouse의 `DESIGN.md`](https://github.com/ClickHouse/ClickHouse/blob/master/DESIGN.md)는 훌륭한 참고 자료입니다. 다음을 포함합니다:
- 비주얼 테마 및 분위기
- 색상 팔레트 및 역할
- 타이포그래피 규칙
- 컴포넌트 스타일
- 레이아웃 원칙
- 깊이 및 입체감
- Do's and Don'ts
- 반응형 동작
- Agent 프롬프트 가이드

우리의 접근 방식은 **구조를 재사용하고, 콘텐츠를 재작성**하는 것입니다. ClickHouse `DESIGN.md`의 섹션 구조를 유지하면서, 콘텐츠를 자체 프로젝트의 실제 디자인 토큰과 컴포넌트 규칙으로 대체합니다.

## 해결책

위의 분석을 바탕으로, 우리의 해결책은 네 가지 핵심 모듈로 구성됩니다.

### 1. 콘텐츠 수집 파이프라인

이것은 전체 시스템의 기반으로, 업스트림 리소스에서 콘텐츠를 추출하고 정규화합니다.

```typescript
// src/lib/content/awesomeDesignCatalog.ts

export async function scanSourceEntries(): Promise<DesignEntry[]> {
  const designDir = 'vendor/awesome-design-md/design-md';
  const entries: DesignEntry[] = [];

  for (const dir of await fs.readdir(designDir)) {
    const entryPath = path.join(designDir, dir);
    if (await isValidDesignEntry(entryPath)) {
      const entry = await normalizeDesignEntry(entryPath);
      entries.push(entry);
    }
  }

  return entries;
}

async function isValidDesignEntry(dir: string): Promise<boolean> {
  const requiredFiles = ['README.md', 'DESIGN.md'];
  for (const file of requiredFiles) {
    if (!(await fileExists(path.join(dir, file)))) {
      return false;
    }
  }
  return true;
}
```

### 2. 갤러리 탐색 인터페이스

갤러리 인터페이스는 세 가지 주요 부분으로 구성됩니다.

**홈페이지**: 모든 디자인 항목의 카드 그리드를 표시하며, 각 카드에는 다음이 포함됩니다:
- 디자인 항목 제목 및 요약
- 미리보기 이미지 (있는 경우)
- 빠른 검색 하이라이팅

**상세 페이지**: 단일 디자인 항목의 전체 정보를 집계합니다:
- `README` 문서
- `DESIGN` 문서
- 라이트/다크 테마 전환이 가능한 미리보기
- 인접 항목 탐색

**탐색**: 갤러리로 돌아가기 및 인접 항목 탐색 지원

홈페이지 갤러리는 고밀도 카드 레이아웃을 사용하여, 여러 소스의 `design.md` 항목을 통합된 시각적 프레임워크로 평면화하여 팀이 브랜드 스타일, 버튼 패턴, 타이포그래피 리듬을 빠르게 비교할 수 있도록 합니다.

![Awesome Design MD Gallery design card grid](/article-images/design-md-consistent-ui/1775613199000-e652f7155dd5.png)

특정 항목을 열면 상세 페이지에서 디자인 요약과 라이브 미리보기를 같은 페이지에 배치하여 문서, 미리보기, 소스 코드를 번갈아 보는 비용을 줄여줍니다.

### 3. 검색 기능

검색 기능은 클라이언트 측 필터링을 기반으로 하며, 상태는 URL 쿼리 매개변수를 통해 보존됩니다:

```typescript
// src/components/gallery/SearchToolbar.tsx

function SearchToolbar({ entries }: { entries: DesignEntry[] }) {
  const [query, setQuery] = useState('');
  const [results, setResults] = useState(entries);

  useEffect(() => {
    const params = new URLSearchParams(window.location.search);
    const initialQuery = params.get('q') || '';
    setQuery(initialQuery);
    filterEntries(initialQuery);
  }, []);

  const filterEntries = (searchQuery: string) => {
    const filtered = entries.filter(entry =>
      entry.searchText.toLowerCase().includes(searchQuery.toLowerCase())
    );
    setResults(filtered);
  };

  const handleChange = (e: React.ChangeEvent<HTMLInputElement>) => {
    const value = e.target.value;
    setQuery(value);
    filterEntries(value);

    // 페이지 새로고침 없이 URL 업데이트
    const newUrl = value
      ? `${window.location.pathname}?q=${encodeURIComponent(value)}`
      : window.location.pathname;
    window.history.replaceState({}, '', newUrl);
  };

  return (
    <div className="search-toolbar">
      <input
        type="text"
        value={query}
        onChange={handleChange}
        placeholder="Search design entries..."
      />
      <span className="result-count">{results.length} results</span>
    </div>
  );
}
```

### 4. 디자인 참조 문서 (`design.md`)

이것은 전체 해결책의 핵심 산출물입니다. 프로젝트 루트에 다음 구조로 `design.md`를 만듭니다:

AI가 소비하는 원본 `design.md` 콘텐츠 외에도, `README`와 `DESIGN` 문서를 동일한 읽기 인터페이스에 배치하여 사람들이 교정하고, 스니펫을 복사하고, 미리보기 결과와 비교하기 쉽게 합니다.

## 실습

해결책을 소개했으니, 실제로 어떻게 구현할까요?

### 구현 단계

**1단계: 서브모듈 초기화**

```bash
# 업스트림 저장소를 서브모듈로 추가
git submodule add https://github.com/VoltAgent/awesome-design-md.git vendor/awesome-design-md

# 서브모듈 초기화 및 업데이트
git submodule update --init --recursive
```

**2단계: 콘텐츠 파이프라인 생성**

`awesomeDesignCatalog.ts`를 구현합니다:
- 파일 스캔 및 검증 로직
- Astro의 내장 렌더러를 사용한 Markdown 렌더링
- 항목 데이터 추출

**3단계: 갤러리 UI 구축**

Astro + React 아일랜드를 사용하여 생성합니다:
- 홈페이지 갤러리 레이아웃 (카드 그리드)
- 디자인 카드 컴포넌트
- 검색 툴바
- 상세 페이지 레이아웃

**4단계: 디자인 문서 작성**

ClickHouse `DESIGN.md`의 구조를 기반으로, 자체 프로젝트의 실제 디자인 토큰을 채웁니다. `README.md`를 업데이트하고 `design.md`에 대한 링크를 추가합니다.

### 주의 사항

**보안**: Markdown 렌더링에는 안전하지 않은 HTML 필터링이 필요합니다. Astro의 내장 렌더러는 기본적으로 `script` 태그를 필터링하지만, XSS 위험에 대해 주의해야 합니다.

**성능**: 많은 수의 iframe 미리보기는 첫 페인트 성능에 영향을 줄 수 있습니다. 미리보기 콘텐츠를 지연 로드하려면 `loading="lazy"`를 사용하는 것이 좋습니다.

**유지 관리성**: `design.md`는 코드 구현과 동기화되어야 합니다. CI 검사를 추가하여 CSS 변수가 문서와 코드 간에 일치하는지 확인하는 것이 좋습니다.

**접근성**: 색상 대비가 WCAG AA 표준(최소 4.5:1)을 충족하는지 확인하세요.

### AI 사용 가이드

`design.md`를 만든 후, AI가 실제로 사용하도록 하려면 어떻게 해야 할까요? 다음은 몇 가지 실용적인 팁입니다.

**팁 1: 프롬프트에서 명시적으로 참조**

```
프로젝트 루트의 design.md 파일을 참조하여, 거기에 정의된 디자인 규칙을 사용하여 다음 컴포넌트를 구현하세요:
- 버튼: 기본 색상에 8px border-radius 사용
- 카드: elevation-2 그림자 수준 사용
```

**팁 2: AI에게 특정 CSS 변수를 참조하도록 요구**

```
다음 요구 사항을 충족하는 내비게이션 바를 구현하세요:
- 배경색에 --color-bg-primary 사용
- 테두리에 --color-border-subtle 사용
- 텍스트에 --text-color-primary 사용
```

**팁 3: 시스템 프롬프트에 `design.md` 내용 포함**

AI 도구가 사용자 정의 시스템 프롬프트를 지원한다면, `design.md`의 핵심 콘텐츠를 직접 추가할 수 있습니다.

### 테스트 전략

**콘텐츠 파이프라인 테스트**:
- 파일 누락 시나리오 (`README.md` 또는 `DESIGN.md` 누락)
- 형식 오류 시나리오 (Markdown 파싱 실패)
- 빈 디렉토리 시나리오

**검색 기능 테스트**:
- 빈 결과 처리
- 중국어 및 이모지와 같은 특수 문자
- URL 동기화 검증

**UI 컴포넌트 테스트**:
- 라이트/다크 테마 전환
- 반응형 레이아웃
- 미리보기 로딩 상태

### 배포 워크플로우

```bash
# 1. 서브모듈을 최신 버전으로 업데이트
git submodule update --remote

# 2. 사이트 재빌드
npm run build

# 3. 정적 에셋 배포
npm run deploy
```

서브모듈 업데이트와 빌드 및 배포를 자동화하는 것이 좋습니다. 그러면 업스트림 저장소가 업데이트될 때마다 CI가 자동으로 트리거됩니다.

## 결론

HagiCode가 개발 과정에서 직면한 AI 생성 UI의 불일치는, 본질적으로 구조화된 디자인 참조 문서의 부재로 인한 것이었습니다. 디자인 갤러리 사이트를 구축하고 표준화된 `design.md`를 만들어 이 문제를 성공적으로 해결했습니다.

이 해결책의 핵심 가치는 다음과 같습니다:
- **통합된 리소스**: 흩어진 디자인 시스템 문서를 하나로 모음
- **구조화된 표준**: 디자인 규칙을 AI가 이해할 수 있는 형식으로 표현
- **지속적인 유지 관리**: `git submodule`을 통해 콘텐츠를 최신 상태로 유지

AI를 사용하여 프론트엔드 개발을 하고 계시다면, 이 접근 방식은 시도해 볼 가치가 있습니다. 구조화된 `design.md`를 만드는 것은 AI가 생성한 코드의 일관성을 높일 뿐만 아니라, 팀 내부에서 통합된 디자인 표준을 유지하는 데도 도움이 됩니다.
