## Design.md: AI가 일관되게 프론트엔드 UI를 설계하도록 하는 솔루션

> AI 보조 프론트엔드 개발 시대, AI가 생성한 UI의 일관성을 유지하려면 어떻게 해야 할까요? 이 글에서는 awesome-design-md를 기반으로 디자인 갤러리 사이트를 구축한实践经验과 AI가 규범적인 UI 설계를 수행하도록 안내하는 구조화된 design.md를 만드는 방법을 공유합니다.

## 배경

AI로 프론트엔드 코드를 작성해 본 분들은 비슷한 경험을 해보셨을 것입니다. 같은 페이지를 AI에게 여러 번 생성하게 하면 매번 스타일이 다릅니다. 때로는 둥근 모서리, 때로는 각진 모서리, 간격은 8px일 때도 있고 16px으로 변할 때도 있으며, 심지어 같은 버튼이 다른 대화에서도 다르게 보입니다.

이는 단순히 개별적인 현상이 아닙니다. AI 보조 개발이 보편화됨에 따라 **AI가 생성한 프론트엔드 UI의 일관성 부족**은 이미 일반적인 문제가 되었습니다. 서로 다른 AI 어시스턴트, 서로 다른 프롬프트, 심지어 같은 어시스턴트가 다른 대화에서도 완전히 다른 스타일의 인터페이스 설계를 생성합니다. 이는 제품 반복에 막대한 유지보수 비용을 초래합니다.

문제의 근원은 사실 간단합니다. **권위 있는 디자인 참조 문서**가 부족하기 때문입니다. 전통적인 CSS 스타일 파일은 개발자에게 "어떻게 구현하는지"만 알려줄 뿐, "왜 이렇게 설계했는지"와 "어떤 시나리오에서 어떤 디자인 패턴을 사용하는지"를 완전하게 전달할 수 없습니다. AI에게는 더욱 명확한 구조화된 설명이 필요하여 설계 규범을 이해할 수 있습니다.

동시에 오픈소스 커뮤니티에는 이미 좋은 리소스들이 있습니다. [VoltAgent/awesome-design-md](https://github.com/VoltAgent/awesome-design-md) 프로젝트는 유명 회사들의 대량의 디자인 시스템 문서를 수집했으며, 각 디렉토리에는 README.md, DESIGN.md 및 미리보기 HTML이 포함되어 있습니다. 하지만 이 모든 것이 상류 저장소에 분산되어 있어 빠르게 조회하고 비교하기 어렵습니다.

그렇다면 이러한 리소스를 통합하여 편리하게 조회할 수 있는 디자인 갤러리로 만들면서, AI가 사용할 수 있는 구조화된 design.md를 정리할 수 있을까요?

답은 긍정적입니다. 다음으로 우리의 솔루션을 공유하겠습니다.

## 분석

코드를 작성하기 전에 먼저 이 문제의 몇 가지 기술적 도전을 분석해 보겠습니다.

### 콘텐츠 소스 관리: 분산된 디자인 리소스를 어떻게 통합할까요?

상류의 awesome-design-md 저장소에는 대량의 디자인 문서가 포함되어 있지만, 우리는 이를 우리 프로젝트에 포함시키는 방법이 필요합니다.

**솔루션: git submodule 사용**

```
awesome-design-md-site
└── vendor/awesome-design-md          # 상류 리소스 (git submodule)
```

이렇게 하면 다음과 같은 이점이 있습니다:
- 버전 관리 가능: 특정 상류 버전을 고정할 수 있습니다
- 오프라인 빌드: 빌드 시 외부 API를 요청할 필요가 없습니다
- 콘텐츠 검토: PR에서 구체적인 변경 사항을 볼 수 있습니다

### 데이터 표준화: 다른 문서 구조를 어떻게 통일할까요?

다른 회사의 디자인 문서 구조는 다를 수 있으며, 일부는 미리보기 파일이 부족하고 일부는 명명이 통일되지 않았습니다. 우리는 빌드 시에 표준화 처리를 수행해야 합니다.

**솔루션: 빌드 시 스캔 및 표준화된 항목 생성**

핵심 모듈은 `awesomeDesignCatalog.ts`로 다음을 담당합니다:
1. `vendor/awesome-design-md/design-md/*` 디렉토리 스캔
2. 각 항목에 필수 파일(README.md, DESIGN.md, 최소 하나의 미리보기 파일)이 포함되어 있는지 검증
3. Markdown 콘텐츠를 추출하고 HTML로 렌더링
4. 표준화된 항목 데이터 생성

### 정적 사이트 아키텍처: 정적 배포를 유지하면서 동적 검색을 어떻게 제공할까요?

디자인 갤러리이므로 검색 기능은 필수입니다. 하지만 Astro는 정적 사이트 생성기인데 실시간 검색을 어떻게 구현할까요?

**솔루션: React island + URL 쿼리 매개변수 동기화**

## 솔루션

위의 분석을 기반으로 우리의 솔루션은 4개의 핵심 모듈을 포함합니다.

### 1. 콘텐츠 수집 파이프라인

시스템의 기초로 상류 리소스에서 콘텐츠를 추출하고 표준화합니다.

```typescript
// src/lib/content/awesomeDesignCatalog.ts

export async function scanSourceEntries(): Promise<DesignEntry[]> {
  const designDir = 'vendor/awesome-design-md/design-md';
  const entries: DesignEntry[] = [];

  for (const dir of await fs.readdir(designDir)) {
    const entryPath = path.join(designDir, dir);
    if (await isValidDesignEntry(entryPath)) {
      const entry = await normalizeDesignEntry(entryPath);
      entries.push(entry);
    }
  }

  return entries;
}

async function isValidDesignEntry(dir: string): Promise<boolean> {
  const requiredFiles = ['README.md', 'DESIGN.md'];
  for (const file of requiredFiles) {
    if (!(await fileExists(path.join(dir, file)))) {
      return false;
    }
  }
  return true;
}
```

### 2. 갤러리 브라우징 인터페이스

갤러리 인터페이스는 3개의 주요 부분을 포함합니다:

**홈페이지**: 모든 디자인 항목의 카드 그리드를 표시하며 각 카드는 다음을 포함합니다:
- 디자인 항목 제목 및 소개
- 미리보기 이미지(있는 경우)
- 빠른 검색 강조

**상세 페이지**: 단일 디자인 항목의 완전한 정보를 집계하여 표시:
- README 문서
- DESIGN 문서
- 미리보기(밝은/어두운 테마 전환 지원)
- 인접 항목 탐색

**탐색**: 갤러리로 돌아가기, 인접 항목 탐색 지원

홈페이지 갤러리는 고밀도 카드 레이아웃을 사용하여 다른 출처의 design.md 항목을 통일된 시각 프레임에 평면적으로 배치하여 브랜드 스타일, 버튼 패턴 및 타이포그래피 리듬을 빠르게 비교하기 편리하게 합니다.

![Awesome Design MD Gallery 디자인 카드 그리드](/article-images/design-md-consistent-ui/1775613199000-e652f7155dd5.png)

구체적인 항목에 들어간 후 상세 페이지는 디자인 요약과 실시간 미리보기를 동일한 페이지에 배치하여 문서, 미리보기 및 소스 코드 사이를 오가는 비용을 줄여줍니다.

### 3. 검색 기능

검색 기능은 클라이언트 측 필터링을 기반으로 하며 URL 쿼리 매개변수를 사용하여 상태를 유지합니다:

```typescript
// src/components/gallery/SearchToolbar.tsx

function SearchToolbar({ entries }: { entries: DesignEntry[] }) {
  const [query, setQuery] = useState('');
  const [results, setResults] = useState(entries);

  useEffect(() => {
    const params = new URLSearchParams(window.location.search);
    const initialQuery = params.get('q') || '';
    setQuery(initialQuery);
    filterEntries(initialQuery);
  }, []);

  const filterEntries = (searchQuery: string) => {
    const filtered = entries.filter(entry =>
      entry.searchText.toLowerCase().includes(searchQuery.toLowerCase())
    );
    setResults(filtered);
  };

  const handleChange = (e: React.ChangeEvent<HTMLInputElement>) => {
    const value = e.target.value;
    setQuery(value);
    filterEntries(value);

    // URL 업데이트(페이지 새로고침 트리거 안 함)
    const newUrl = value
      ? `${window.location.pathname}?q=${encodeURIComponent(value)}`
      : window.location.pathname;
    window.history.replaceState({}, '', newUrl);
  };

  return (
    <div className="search-toolbar">
      <input
        type="text"
        value={query}
        onChange={handleChange}
        placeholder="디자인 항목 검색..."
      />
      <span className="result-count">{results.length}개 결과</span>
    </div>
  );
}
```

AI가 소비하는 원시 `design.md` 콘텐츠 외에도 README와 DESIGN 두 문서를 동일한 읽기 인터페이스에 넣어 사람이 교정하고, 조각을 복사하며 미리보기 결과와 대조하기 편리하게 합니다.

## 실천

솔루션을 이해한 후 구체적으로 어떻게 실행할까요?

### 실행 단계

**첫 번째 단계: 하위 모듈 초기화**

```bash
# 상류 저장소를 하위 모듈로 추가
git submodule add https://github.com/VoltAgent/awesome-design-md.git vendor/awesome-design-md

# 하위 모듈 초기화 및 업데이트
git submodule update --init --recursive
```

**두 번째 단계: 콘텐츠 파이프라인 생성**

`awesomeDesignCatalog.ts`를 구현합니다:
- 파일 스캔 및 검증 로직
- Markdown 렌더링(Astro의 내장 렌더러 사용)
- 항목 데이터 추출

**세 번째 단계: 갤러리 UI 구축**

Astro + React Islands를 사용하여 생성:
- 홈페이지 갤러리 레이아웃(카드 그리드)
- 디자인 카드 컴포넌트
- 검색 도구 모음
- 상세 페이지 레이아웃

**네 번째 단계: 디자인 문서 작성**

ClickHouse DESIGN.md 구조를 기반으로 프로젝트의 실제 디자인 토큰을 채웁니다. README.md를 업데이트하고 design.md에 대한 링크를 추가합니다.

### 주의 사항

**보안**: Markdown 렌더링은 안전하지 않은 HTML을 필터링해야 합니다. Astro의 내장 렌더러는 기본적으로 script 태그를 필터링하지만 여전히 XSS 위험에 주의해야 합니다.

**성능**: 대량의 iframe 미리보기는 첫 화면 로딩에 영향을 줄 수 있습니다. `loading="lazy"`를 사용하여 미리보기 콘텐츠를 지연 로드하는 것이 좋습니다.

**유지보수성**: design.md는 코드 구현과 동기화를 유지해야 합니다. CI에 검사를 추가하여 CSS 변수가 문서와 코드에서 일치하는지 확인하는 것이 좋습니다.

**접근성**: 색상 대비가 WCAG AA 표준(최소 4.5:1)을 충족하는지 확인하세요.

### AI 사용 가이드

design.md를 만든 후 AI가 정말로 사용하게 하려면 어떻게 할까요? 여기 몇 가지 실용적인 팁이 있습니다:

**팁 1: 프롬프트에서 명확하게 참조**

```
프로젝트 루트 디렉토리의 design.md 파일을 참조하여 그 안에 정의된 디자인 규범을 사용하여 다음 컴포넌트를 구현하세요:
- 버튼: primary色调 사용, 둥근 모서리 8px
- 카드: elevation-2 그림자 계층 사용
```

**팁 2: AI에 구체적인 CSS 변수 참조 요구**

```
탐색 모음을 구현하려면 다음이 필요합니다:
- 배경색은 --color-bg-primary 사용
- 테두리는 --color-border-subtle 사용
- 텍스트는 --text-color-primary 사용
```

**팁 3: 시스템 프롬프트에 design.md 콘텐츠 포함**

AI 도구가 사용자 정의 시스템 프롬프트를 지원하는 경우 design.md의 핵심 콘텐츠를 직접 추가할 수 있습니다.

### 테스트 전략

**콘텐츠 파이프라인 테스트**:
- 파일 누락 시나리오(README.md 또는 DESIGN.md 누락)
- 형식 오류 시나리오(Markdown 구문 분석 실패)
- 빈 디렉토리 시나리오

**검색 기능 테스트**:
- 빈 결과 처리
- 특수 문자(중국어, emoji 등)
- URL 동기화 검증

**UI 컴포넌트 테스트**:
- 밝은/어두운 테마 전환
- 반응형 레이아웃
- 미리보기 로딩 상태

### 배포 프로세스

```bash
# 1. 하위 모듈을 최신 버전으로 업데이트
git submodule update --remote

# 2. 사이트 재구축
npm run build

# 3. 정적 리소스 배포
npm run deploy
```

하위 모듈 업데이트와 구축 배포를 자동화하는 것이 좋습니다. 상류 저장소가 업데이트될 때 자동으로 CI 프로세스를 트리거할 수 있습니다.

## 요약

HagiCode가 개발 과정에서 직면한 AI 생성 UI 불일치 문제는 본질적으로 구조화된 디자인 참조 문서가 부족하기 때문입니다. 디자인 갤러리 사이트를 구축하고 규범적인 design.md를 만들어 우리는 이 문제를 성공적으로 해결했습니다.

이 솔루션의 핵심 가치는 다음과 같습니다:
- **통일된 리소스**: 분산된 디자인 시스템 문서 통합
- **구조화된 규범**: 디자인 규범을 AI가 이해할 수 있는 형태로 제시
- **지속적 유지보수**: git submodule을 통해 콘텐츠 업데이트 유지

AI 보조 프론트엔드 개발을 사용 중이라면 이 솔루션을 시도해 보세요. 구조화된 design.md를 만들면 AI가 생성한 코드의 일관성을 높일 뿐만 아니라 팀 내부에서도 디자인 규범의 통일을 유지하는 데 도움이 됩니다.
