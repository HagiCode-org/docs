## AI 모험단 만들기: HagiCode 다중 에이전트 협업 구성 실전

> 현대 소프트웨어 개발에서 단일 AI 에이전트로는 이미 복잡한 요구사항을 충족하기 어렵습니다. 다른 회사의 여러 AI 도우미를 동일한 프로젝트에서 협업하게 하려면 어떻게 해야 할까요? 이 글에서는 HagiCode 프로젝트가 실제 개발에서 탐색한 다중 에이전트 협업 구성 방안을 공유합니다.

이 글은 **2026-03-17 발표 시점**의 다중 에이전트 구성 실천을 기록한 것입니다. 현재 상시 지원 범위는 백엔드 현행 구성을 기준으로 합니다: Claude Code, Codex, GitHub Copilot, OpenCode, Hermes, QoderCLI, Kiro, Kimi, Gemini, DeepAgents, Codebuddy. 그중 `IFlowCli`는 현재 역사적 호환성 값으로만 보존되며, 현재 활성 지원 범위에 속하지 않습니다. 현재 설치 및 지원 기준을 보려면 [AI Agent CLI 설치](/related-software-installation/ai-agent-cli/)와 [설치 가이드](/installation)를 참조하세요.

## 배경

많은 개발자가 이런 경험을 했을 것입니다: 프로젝트에서 AI 도우미를 도입해 프로그래밍을 보조하니 효율성이确实 높아졌습니다. 하지만 요구사항이 점점 복잡해지면서, 하나의 AI 에이전트로는 부족하게 되었습니다—코드 리뷰, 문서 생성, 단위 테스트 등 여러 작업을 동시에 처리하게 하려면 결과적으로 이것저것 처리하느라 출력 품질이 들쭉날쭉합니다.

더 골치 아픈 것은, 여러 AI 도우미를 도입하려고 시도할 때 문제가 더 복잡해진다는 점입니다. 각 에이전트는 자신만의 구성 방식, API 인터페이스, 실행 로직이 있어 서로 충돌까지 일어납니다. 마치 축구팀처럼 각 선수는 모두 훌륭하지만 어떻게 협업해야 할지 아는 사람이 없어서 결국 엉망이 되는 것과 같습니다.

HagiCode 프로젝트도 개발 과정에서 동일한困扰를 겪었습니다. 프론트엔드 VSCode 확장, 백엔드 AI 서비스, 크로스 플랫폼 데스크톱 클라이언트를 포함하는 복잡한 프로젝트로서, 우리는 **2026-03 당시 버전**에서 다른 회사의 여러 AI 도우미와 동시에 연결해야 했습니다: Claude Code, Codex, CodeBuddy, iFlow 등. 어떻게 그들을 동일한 프로젝트에서 조화롭게 공존시키고 각자의 장점을 발휘하게 할 것인가가 반드시 해결해야 할 핵심 문제가 되었습니다.

사실 그럴 수도 있습니다. 매일 싸우는 AI들을 상대하고 싶은 사람은 아무도 없으니까요.

이 글에서 공유하는 방안은 HagiCode 프로젝트에서 실제로 시행착오를 겪고 실제로 최적화한 다중 에이전트 협업 구성 실천입니다. 다중 AI 도우미 협업으로 골머리를 앓고 있다면, 이 글이 도움이 될 것입니다. 아마도요, 모든 사람의 상황이 다르니까요.

## HagiCode에 대하여

[HagiCode](https://hagicode.com)는 여러 AI 엔진이 협동하는 "모험단" 모델을 채택한 AI 코드 도우미 프로젝트입니다. 프로젝트 주소: [github.com/HagiCode-org/site](https://github.com/HagiCode-org/site).

이 글에서 공유하는 다중 에이전트 구성 방안은 HagiCode가 복잡한 프로젝트에서 효율적인 개발을 유지할 수 있는 핵심 기술 중 하나입니다. 특별한 것은 없습니다. 그저一群的 AI를 협동할 수 있는 모험단으로 만들 뿐입니다.

## HagiCode의 다중 에이전트 아키텍처 설계

### "단독 행동"에서 "팀 협업"으로

HagiCode 프로젝트 초기에 우리도 하나의 AI 에이전트로 모든 작업을 처리하려고 시도했습니다.很快 우리는 이 방식이 명확한 병목이 있다는 것을 발견했습니다: 다른 작업은 다른 능력 중점이 필요하고, 어떤 작업은 더 강한 문맥 이해 능력이 필요하고, 어떤 것은 더 정확한 코드 수정 능력이 필요합니다. 하나의 에이전트가 모든 면에서 뛰어나게 수행하기는 어렵습니다.

이것은 우리가 여러 에이전트가 협동으로 일해야 한다는 것을 깨닫게 했습니다. 하지만 문제는, 다른 회사의 AI 제품을 동일한 프로젝트에서 평화롭게 공존하게 어떻게 만들 것인가입니다? 우리는 몇 가지 핵심 문제를 해결해야 했습니다:

1. **구성 관리 복잡성**: 각 에이전트는 다른 구성 방식, API 인터페이스 및 실행 모드가 있습니다
2. **통신 프로토콜 통일**: 다른 에이전트 간의 데이터 교환을 위한 표준화된 방식이 필요합니다
3. **작업 분업 조정**: 각 에이전트가 장점을 발휘하도록 합리적으로 작업을 분배하는 방법

이러한 문제들을 안고 우리는 HagiCode의 다중 에이전트 아키텍처 설계를 시작했습니다. 사실 그렇게 복잡하지 않습니다. 그저 생각을 정리했을 뿐입니다.

### 전체 아키텍처 개요

여러 번의 반복을 거쳐 우리가 최종적으로 확정한 아키텍처는 다음과 같습니다:

```
┌─────────────────────────────────────────────────────────────────┐
│                    AIProviderFactory                             │
│  (공장 모델로 모든 AI Provider를 통일 관리)                         │
├─────────────────────────────────────────────────────────────────┤
│  ClaudeCodeCli  │  CodexCli  │  CodebuddyCli  │  IFlowCli    │
│  (Anthropic)   │  (OpenAI)  │  (Zhipu GLM)     │  (Zhipu)      │
└─────────────────────────────────────────────────────────────────┘
```

핵심 아이디어는: 통일된 Provider 인터페이스를 통해 다른 AI 에이전트가 동일한 코드로 관리될 수 있게 하는 것입니다. 동시에 공장 모델을 사용해 동적으로 Provider를 생성하고 구성하여 시스템의 확장성과 유연성을 보장합니다.

이것은 생활에서의 분업과 같습니다.每个人都有自己的 역할，여기서는 이러한 분업을 코드 아키텍처로 만들었을 뿐입니다.

### 에이전트 유형과 책임 분업

HagiCode 프로젝트의 실제 사용 경험에 따라, 우리는 각 에이전트에 다른 책임을 할당했습니다:

| Agent | 제공자 | 모델 | 주요 용도 |
|-------|--------|------|---------|
| ClaudeCodeCli | Anthropic | glm-5-turbo | 기술 방안 및 Proposal 생성 |
| CodexCli | OpenAI/Zed | gpt-5.4 | 정확한 코드 수정 실행 |
| CodebuddyCli | Zhipu | glm-4.7 | 제안 설명 및 문서 최적화 |
| IFlowCli | Zhipu | glm-4.7 | 제안 및历史记录 아카이빙 (당시 구성; 현재는 역사적 호환성만) |
| OpenCodeCli | - | - | 일반 코드 편집 |
| GitHubCopilot | Microsoft | - | 프로그래밍 보조 및 코드 완성 |

이러한 분업 배후의 논리는: 각 에이전트는 자신이 잘하는 분야가 있습니다. Claude Code는 복잡한 요구사항의 이해와 분석에서 뛰어나서 초기 방안 설계를 담당하게 하고; Codex는 코드 수정에서 더 정확해서 구체적인 구현 작업 처리에 적합하고; CodeBuddy는 가성비가 높아서 문서 최적화에 사용하기에 더욱 적합합니다.

자신에게 맞는 것이 최고인 것입니다. 모든 길은 로마로 통하지만, 어떤 길은 더 쉽고, 어떤 길은 조금 더 굽이굽이할 뿐입니다.

## 핵심 구성 메커니즘

### 통일된 Provider 인터페이스 설계

다른 AI 에이전트를 통일적으로 관리하게 하려면 먼저 통일된 인터페이스 세트를 정의해야 합니다. HagiCode에서 이 인터페이스를 정의했습니다:

```csharp
public interface IAIProvider
{
    // 통일된 Provider 인터페이스
    Task<IAIProvider?> GetProviderAsync(AIProviderType providerType);
    Task<IAIProvider?> GetProviderAsync(string providerName, CancellationToken cancellationToken);
}
```

이 인터페이스는 간단해 보이지만, 전체 다중 에이전트 시스템의 기초입니다. 통일된 인터페이스를 통해 우리는 하층이 어느 회사의 AI 제품인지 무시하고 동일한 방식으로 호출할 수 있습니다.

사실 이것은 복잡한 것을 단순화한 것입니다. 간단한 것이 아름다운 것입니다.

### Provider 공장 모델 구현

통일된 인터페이스가 있으니, 다음은 이러한 Provider 인스턴스를 어떻게 생성하는가입니다. HagiCode는 공장 모델을 사용했습니다:

```csharp
private IAIProvider? CreateProvider(AIProviderType providerType, ProviderConfiguration config)
{
    return providerType switch
    {
        AIProviderType.ClaudeCodeCli =>
            ActivatorUtilities.CreateInstance<ClaudeCodeCliProvider>(_serviceProvider, Options.Create(config)),
        AIProviderType.CodebuddyCli =>
            ActivatorUtilities.CreateInstance<CodebuddyCliProvider>(_serviceProvider, Options.Create(config)),
        AIProviderType.CodexCli =>
            ActivatorUtilities.CreateInstance<CodexCliProvider>(_serviceProvider, Options.Create(config)),
        AIProviderType.IFlowCli =>
            ActivatorUtilities.CreateInstance<IFlowCliProvider>(_serviceProvider, Options.Create(config)),
        _ => null
    };
}
```

여기서 의존성 주입의 `ActivatorUtilities.CreateInstance`를 사용했는데, 실행 시간에 동적으로 Provider 인스턴스를 생성하고 자동으로 의존성 항목을 주입할 수 있습니다. 이 설계의 장점은: 새 에이전트 유형을 추가할 때, 해당 Provider 클래스만 추가하고 공장 메서드에서 case 분기를 하나만 추가하면 되며, 기존 코드를 전혀 수정할 필요가 없습니다.

그럴 수도 있습니다. 매번 새 기능을 추가할 때마다一堆의旧代码를 수정하고 싶은 사람은 아무도 없으니까요.

### 동적 구성 분석

구성을 더 유연하게 하기 위해 우리는 유형 매핑 메커니즘도 구현했습니다:

```csharp
public static AIProviderTypeExtensions
{
    private static readonly Dictionary<string, AIProviderType> _typeMap = new(
        StringComparer.OrdinalIgnoreCase)
    {
        ["ClaudeCodeCli"] = AIProviderType.ClaudeCodeCli,
        ["CodebuddyCli"] = AIProviderType.CodebuddyCli,
        ["CodexCli"] = AIProviderType.CodexCli,
        ["IFlowCli"] = AIProviderType.IFlowCli,
        // ...더 많은 유형 매핑
    };
}
```

이 매핑 테이블의 역할은 문자열 형식의 Provider 이름을 열거형 유형으로 변환하는 것입니다. 이렇게 하면 구성 파일은 직관적인 문자열 이름을 사용할 수 있고, 코드 내부에서는 유형 안전 열거형으로 처리합니다.

구성은 직관적일수록 좋습니다.谁愿意记一堆复杂的代码呢.

### 구성 파일 예시

실제 사용할 때는 `appsettings.json`에서 구성만 하면 됩니다:

```yaml
AI:
  Providers:
    Providers:
      ClaudeCodeCli:
        Enabled: true
        Model: glm-5-turbo
        WorkingDirectory: /path/to/project
      CodebuddyCli:
        Enabled: true
        Model: glm-4.7
      CodexCli:
        Enabled: true
        Model: gpt-5.4
      IFlowCli:
        Enabled: true
        Model: glm-4.7
```

각 Provider는 독립적으로 스위치, 모델 버전, 작업 디렉터리 등 매개변수를 구성할 수 있습니다. 이 설계는 유연성을 보장할 뿐만 아니라 관리와 유지 보수에도 편리합니다.

사실 구성 파일은 인생의 선택과 같습니다. 어떤 기능을 켜거나 끌 수 있습니다.只是代码里的选择更容易后悔罢了.

## 모험단 작업 흐름

### 작업 분업의 예술

통일된 기술 아키텍처가 있으니, 다음은 여러 에이전트를 협동으로 일하게 하는 것입니다. HagiCode는 작업 흐름 메커니즘을 설계해 다른 에이전트가 다른 단계의 작업을 처리하게 합니다:

```
제안 생성 (사용자)
    │
    ▼
[Claude Code] ──제안 생성──▶ 제안 문서
    │                        │
    │                        ▼
    │               [Codebuddy] ──설명 최적화──▶ 최적화된 제안
    │                        │
    │                        ▼
    │               [Codex] ──수정 실행──▶ 코드 변경
    │                        │
    │                        ▼
    └───────────────▶ [iFlow] ──아카이빙──▶历史记录
```

이러한 분업의 장점은: 각 에이전트는 자신이 잘하는 작업에만 집중하면 되고 "모든 것을 잘할" 필요가 없습니다. Claude Code는 없에서 있게 제안을 생성하고, Codebuddy는 제안 설명을 더 명확하게 만들고, Codex는 제안을 실제 코드 변경으로 만들고, iFlow는 이러한 변경을 아카이빙하고 보존합니다.

사실 이것은 생활에서의 팀 협업과 같습니다.每个人都有自己的 역할，합쳐야만 큰 일을 완성할 수 있습니다。只是这里的团队成员是 AI而已.

### 핵심 실천 요점

실제 실행에서 우리는 다음과 같은 몇 가지 경험을 요약했습니다:

**1. 에이전트 선택 전략이 중요합니다**

작업을 무작위로 분배하는 것이 아니라, 각 에이전트의 장점에 따라 분배합니다:
- 제안 생성: Claude Code 사용, 더 강한 문맥 이해 능력이 있기 때문
- 코드 실행: Codex 사용, 코드 수정에서 더 정확하기 때문
- 제안 최적화: Codebuddy 사용, 가성비가 높기 때문
- 아카이빙 저장: iFlow 사용, 안정적이고 신뢰할 수 있기 때문

적합한 사람에게 적합한 일을 시키는 것,这是千古不变的道理.

**2. 구성 격리로 안정성 보장**

각 에이전트의 구성은 독립적으로 관리되고, 환경 변수 덮어쓰기를 지원하며, 작업 디렉터리도 서로 독립적입니다. 이렇게 하면 한 에이전트의 구성 오류가 다른 에이전트에 영향을 주지 않습니다.

이것은 생활에서의 경계와 같습니다.每个人都有自己的 공간，서로 방해하지 않아야 조화롭게 공존할 수 있습니다.

**3. 오류 처리 메커니즘**

단일 에이전트 실패가 전체 흐름에 영향을 주어서는 안 됩니다. 우리는 강화 전략을 구현했습니다: 특정 에이전트 실행 실패 시 시스템은 자동으로 예비 방안으로 전환하거나 해당 단계를 건너뛰고 후속 작업을 계속 실행할 수 있습니다. 동시에 완전한 로그 기록으로 사후 문제 해결에 편리합니다.

누구도 결코 오류가 발생하지 않는다고 보장할 수 없습니다. 핵심은 오류를 어떻게 처리하는가입니다. 이것은 인생과 같습니다.挫折을 만나겠지만，중요한 것은 어떻게 빠져나오는가입니다.

**4. 모니터링과 관찰 가능성**

ACP 프로토콜(우리가 사용자 정의한 통신 프로토콜, JSON-RPC 2.0 기반)을 통해 각 에이전트의 실행 상태를 추적할 수 있습니다. 세션 격리는 동시성 안전을 보장하고, 동적 캐시는 성능을 최적화합니다.

보이지 않는 것이最容易出问题，모니터링이 좀 있는 것이 눈앞이 캄캄한 것보다 낫습니다.

## 실제 효과와 수익

이 다중 에이전트 협업 구성을 채택한 후 HagiCode 프로젝트의 개발 효율성은 현저하게 향상되었습니다. 구체적으로 다음과 같이 나타납니다:

1. **작업 처리 능력 배증**: 이전 하나의 에이전트가 여러 작업을 동시에 처리해야 했는데, 이제는 병렬로 처리할 수 있어 처리량이 2배 이상 향상되었습니다
2. **출력 품질이 더 안정적**: 각 에이전트는 자신이 잘하는 작업에만 집중하고 출력 결과의 일관성과 품질이 더 높습니다
3. **유지 보수 비용 감소**: 통일된 인터페이스와 구성 관리로 전체 시스템이 더 쉽게 유지 보수되고 확장됩니다
4. **새 에이전트 추가 간단**: 새 AI 제품을 접속하려면 인터페이스만 구현하고 구성을 추가하면 되며 핵심 논리를 수정할 필요가 없습니다

이 방안은 HagiCode 자체의 문제를 해결했을 뿐만 아니라, 다중 에이전트 협업이 실행 가능한 아키텍처 선택임도 증명했습니다.

사실 효과는 상당히 명확합니다。只是过程有点折腾罢了.

## 요약

이 글에서는 HagiCode 프로젝트가 다중 에이전트 협업 구성 방면의 실제 경험을 공유했습니다. 핵심 요점은 다음과 같습니다:

1. **표준화 인터페이스**: `IAIProvider`를 통해 다른 에이전트의 행동을 통일하고, 코드가 하층이 어느 회사의 제품인지 무시할 수 있게 합니다
2. **공장 모델**: `ActivatorUtilities.CreateInstance`를 사용해 Provider 인스턴스를 동적으로 생성하고 실행 시간 구성 및 의존성 주입을 지원합니다
3. **프로토콜 통일**: ACP 프로토콜은 에이전트 간의 표준화된 통신을 구현하고 JSON-RPC 2.0 기반의 양방향 통신 메커니즘입니다
4. **작업 분류**: 합리적으로 작업을 다른 에이전트에 분배하고 그들이 각자 장점을 발휘하게 하며, 하나의 에이전트가 모든 것을 하려고 시도하지 않습니다

이 설계는 "다중 에이전트 싸움" 문제를 해결했을 뿐만 아니라, 모험단의 작업 흐름 메커니즘을 통해 개발 흐름의 자동화와 전문화를 실현했습니다.

다중 AI 도우미 도입을 고려하고 있다면 이 글이 참고가 되기를 바랍니다. 물론 각 프로젝트의 상황이 다르므로 구체적인 방안은 실제 상황에 따라 조정해야 합니다. 모든 곳에 적용되는 방안은 없습니다. 자신에게 맞는 것이 최고입니다.

아름다운 것 또는 사람，반드시 점유할 필요는 없습니다，그녀가 여전히 아름다우면，자신이 그녀의 아름다움을 잘 보면 됩니다。기술 방안도 마찬가지입니다，자신에게 맞는 것이，최고입니다......
