## AI 모험단 구축: HagiCode 다중 Agent 협업 설정 실전

> 현대 소프트웨어 개발에서 단일 AI Agent로는 복잡한 요구를 충족하기 어렵습니다. 서로 다른 회사의 여러 AI 어시스턴트가 동일한 프로젝트에서 협업하게 하려면 어떻게 해야 할까요? 이 글에서는 HagiCode 프로젝트가 실제 개발에서 탐색한 다중 Agent 협업 설정 솔루션을 공유합니다.

:::note[시간적 맥락 설명]
이 글은 **2026-03-17 발행 시**의 다중 Agent 설정 실천을 기록합니다. 현재 상주 지원 범위는 백엔드 현행 설정을 기준으로 합니다: Claude Code, Codex, GitHub Copilot, OpenCode, Hermes, QoderCLI, Kiro, Kimi, Gemini, DeepAgents, Codebuddy. 그중 `IFlowCli`는 현재 역사적 호환성 값으로만 보존되며, 현재 활성 지원 범위에 속하지 않습니다. 현재 설치 및 지원 범위를 보려면 [AI Agent CLI 설치](/related-software-installation/ai-agent-cli/) 및 [설치 가이드](/installation)를 참조하십시오.
:::

## 배경

많은 개발자들이 이런 경험을 해보셨을 것입니다. 프로젝트에 AI 어시스턴트를 도입하여 프로그래밍을 보조하니 효율이 확실히 올랐습니다. 하지만 요구가 점점 복잡해지면서 하나의 AI Agent로는 부족해졌습니다. 코드 리뷰, 문서 생성, 단위 테스트 등 여러 작업을 동시에 처리하려 하면 결과적으로 하나를 놓치거나 출력 품질이 들쭉날쭉해집니다.

더 골치 아픈 것은 여러 AI 어시스턴트를 도입하려고 하면 문제가 더 복잡해진다는 것입니다. 각 Agent는 자체적인 설정 방식, API 인터페이스 및 실행 로직을 가지고 있으며, 서로 간에 충돌까지 발생할 수 있습니다. 한 축구팀과 같습니다. 각 선수가 뛰어나지만 누구도 어떻게 협력해야 할지 모르면 경기가 엉망이 됩니다.

HagiCode 프로젝트도 개발 과정에서 같은困扰를 겪었습니다. 프론트엔드 VSCode 확장, 백엔드 AI 서비스, 크로스 플랫폼 데스크톱 클라이언트를 포함하는 복잡한 프로젝트로서, **2026-03의 당시 버전**에서는 서로 다른 회사의 여러 AI 어시스턴트를 동시에 연결해야 했습니다: Claude Code, Codex, CodeBuddy, iFlow 등. 이들이 동일한 프로젝트에서 조화롭게 공존하며 각자의 장점을 발휘하게 하는 것이 반드시 해결해야 할 핵심 문제였습니다.

사실 이것도 그렇고,毕竟 매일 싸우는 AI 무리와打交道하고 싶은 사람은 없겠죠.

이 글에서 공유하는 솔루션은 HagiCode 프로젝트에서 실제로 함정을 밟고 최적화한 다중 Agent 협업 설정 실천입니다. 다중 AI 어시스턴트 협업으로 골치 아프신 분들에게 이 글이 영감을 줄 수 있을 것이라고 믿습니다. 아마도,毕竟 각자의 상황이 다르니까요.

## HagiCode에 대하여

[HagiCode](https://hagicode.com)는 다중 AI 엔진이 협업하는 "모험단" 모드를 채택한 AI 코드 어시스턴트 프로젝트입니다. 프로젝트 주소: [github.com/HagiCode-org/site](https://github.com/HagiCode-org/site).

이 글에서 공유하는 다중 Agent 설정 솔루션은 HagiCode가 복잡한 프로젝트에서 효율적인 개발을 유지할 수 있는 핵심 기술 중 하나입니다. 별것도 아닙니다. 그냥 여러 AI를 협력할 수 있는 모험단으로 만들었을 뿐입니다.

## HagiCode의 다중 Agent 아키텍처 설계

### "독불장군"에서 "팀 협업"으로

HagiCode 프로젝트 초기에는 하나의 AI Agent로 모든 작업을 처리하려고도 했습니다. 곧 이 방식에는 명확한 병목이 있다는 것을 발견했습니다. 다양한 작업에는 다른 능력 측면이 필요하며, 어떤 작업은 더 강한 컨텍스트 이해 능력이, 어떤 작업은 더 정밀한 코드 수정 능력이 필요합니다. 하나의 Agent가 모든 면에서 뛰어나게 수행하기는 어렵습니다.

이것은 여러 Agent가 협업해야 한다는 것을 깨닫게 했습니다. 하지만 문제는 어떻게 서로 다른 회사의 AI 제품을 동일한 프로젝트에서 평화롭게 공존시킬 것인가입니다. 우리는 몇 가지 핵심 문제를 해결해야 했습니다:

1. **설정 관리 복잡성**: 각 Agent는 다른 설정 방식, API 인터페이스 및 실행 모드를 가집니다
2. **통신 프로토콜 통일**: 서로 다른 Agent 간에 데이터 교환을 위한 표준화된 방법이 필요합니다
3. **작업 분담 조정**: 각 Agent가 장점을 발휘하도록 합리적으로 작업을 분배하는 방법

이러한 문제를 안고 HagiCode의 다중 Agent 아키텍처 설계를 시작했습니다. 사실 그렇게 복잡하지는 않았습니다. 그냥 이해했을 뿐입니다.

### 전체 아키텍처 개요

여러 번의 반복 끝에 최종적으로 확정된 아키텍처는 다음과 같습니다:

```
┌─────────────────────────────────────────────────────────────────┐
│                    AIProviderFactory                             │
│  (팩토리 패턴으로 모든 AI Provider 통합 관리)                     │
├─────────────────────────────────────────────────────────────────┤
│  ClaudeCodeCli  │  CodexCli  │  CodebuddyCli  │  IFlowCli    │
│  (Anthropic)   │  (OpenAI)  │  (智谱 GLM)     │  (智谱)      │
└─────────────────────────────────────────────────────────────────┘
```

핵심 아이디어는 통일된 Provider 인터페이스를 통해 서로 다른 AI Agent를 동일한 코드로 관리할 수 있게 하는 것입니다. 동시에 팩토리 패턴을 사용하여 이러한 Provider를 동적으로 생성 및 구성하여 시스템의 확장성과 유연성을 보장합니다.

이것은 삶의 분업과 같습니다. 각자 자신의 역할이 있으며, 여기서는 이러한 분업을 코드 아키텍처로 만든 것뿐입니다.

### Agent 유형 및 책임 분담

HagiCode 프로젝트의 실제 사용 경험에 따라 각 Agent에 다른 책임을 할당했습니다:

| Agent | 제공자 | 모델 | 주요 용도 |
|-------|--------|------|---------|
| ClaudeCodeCli | Anthropic | glm-5-turbo | 기술 방안 및 Proposal 생성 |
| CodexCli | OpenAI/Zed | gpt-5.4 | 정밀한 코드 수정 실행 |
| CodebuddyCli | 智谱 | glm-4.7 | 제안 설명 및 문서 최적화 |
| IFlowCli | 智谱 | glm-4.7 | 제안 및 기록 아카이빙 (당시 설정; 현재 역사적 호환만) |
| OpenCodeCli | - | - | 범용 코드 편집 |
| GitHubCopilot | Microsoft | - | 보조 프로그래밍 및 코드 완성 |

이러한 분담의 배후 논리는 각 Agent에게 자신이 능숙한 영역이 있다는 것입니다. Claude Code는 복잡한 요구를 이해하고 분석하는 데 뛰어나므로 초기 방안 설계를 담당합니다. Codex는 코드 수정에서 더 정밀하므로 구체적인 구현 작업에 적합합니다. CodeBuddy는 가성비가 높아 문서 최적화에 적합합니다.

毕竟 자신에게 맞는 것이 가장 좋습니다. 모든 길이 로마로 통하지만, 어떤 길은 조금 더 걷기 편하고, 어떤 길은 조금 더 우회합니다.

## 핵심 설정 메커니즘

### 통일된 Provider 인터페이스 설계

서로 다른 AI Agent를 통합 관리하려면 먼저 통일된 인터페이스를 정의해야 합니다. HagiCode에서는 이 인터페이스를 정의했습니다:

```csharp
public interface IAIProvider
{
    // 통일된 Provider 인터페이스
    Task<IAIProvider?> GetProviderAsync(AIProviderType providerType);
    Task<IAIProvider?> GetProviderAsync(string providerName, CancellationToken cancellationToken);
}
```

이 인터페이스는 간단해 보이지만 전체 다중 Agent 시스템의 기초입니다. 통일된 인터페이스를 통해 하위가 어느 회사의 AI 제품이든 동일한 방식으로 호출할 수 있습니다.

사실 이것은 복잡한 것을 단순하게 만든 것입니다.毕竟 단순함이 아름다움입니다.

### Provider 팩토리 패턴 구현

통일된 인터페이스가 있으면 다음은 이 Provider 인스턴스를 어떻게 생성할 것인가입니다. HagiCode는 팩토리 패턴을 사용했습니다:

```csharp
private IAIProvider? CreateProvider(AIProviderType providerType, ProviderConfiguration config)
{
    return providerType switch
    {
        AIProviderType.ClaudeCodeCli =>
            ActivatorUtilities.CreateInstance<ClaudeCodeCliProvider>(_serviceProvider, Options.Create(config)),
        AIProviderType.CodebuddyCli =>
            ActivatorUtilities.CreateInstance<CodebuddyCliProvider>(_serviceProvider, Options.Create(config)),
        AIProviderType.CodexCli =>
            ActivatorUtilities.CreateInstance<CodexCliProvider>(_serviceProvider, Options.Create(config)),
        AIProviderType.IFlowCli =>
            ActivatorUtilities.CreateInstance<IFlowCliProvider>(_serviceProvider, Options.Create(config)),
        _ => null
    };
}
```

여기서는 의존성 주입의 `ActivatorUtilities.CreateInstance`를 사용하여 런타임에 Provider 인스턴스를 동적으로 생성하고 의존성을 자동 주입합니다. 이 설계의 장점은 새 Agent 유형을 추가할 때 해당 Provider 클래스를 추가하고 팩토리 메서드에 case 분기를 하나 추가하면 되며, 기존 코드를 전혀 수정할 필요가 없다는 것입니다.

이것도 그렇고,毕竟 매번 새 기능을 추가할 때마다 기존 코드를 한 무더기 수정하고 싶은 사람은 없겠죠.

### 동적 설정 파싱

설정을 더 유연하게 하기 위해 유형 매핑 메커니즘도 구현했습니다:

```csharp
public static AIProviderTypeExtensions
{
    private static readonly Dictionary<string, AIProviderType> _typeMap = new(
        StringComparer.OrdinalIgnoreCase)
    {
        ["ClaudeCodeCli"] = AIProviderType.ClaudeCodeCli,
        ["CodebuddyCli"] = AIProviderType.CodebuddyCli,
        ["CodexCli"] = AIProviderType.CodexCli,
        ["IFlowCli"] = AIProviderType.IFlowCli,
        // ...더 많은 유형 매핑
    };
}
```

이 매핑 테이블의 역할은 문자열 형식의 Provider 이름을 열거형 유형으로 변환하는 것입니다. 이렇게 하면 설정 파일에서 직관적인 문자열 이름을 사용할 수 있고, 코드 내부에서는 유형 안전한 열거형으로 처리합니다.

毕竟 설정은 직관적일수록 좋습니다. 복잡한 코드를 기억하고 싶은 사람은 없겠죠.

### 설정 파일 예제

실제 사용 시 `appsettings.json`에 설정하기만 하면 됩니다:

```yaml
AI:
  Providers:
    Providers:
      ClaudeCodeCli:
        Enabled: true
        Model: glm-5-turbo
        WorkingDirectory: /path/to/project
      CodebuddyCli:
        Enabled: true
        Model: glm-4.7
      CodexCli:
        Enabled: true
        Model: gpt-5.4
      IFlowCli:
        Enabled: true
        Model: glm-4.7
```

각 Provider는 독립적으로 스위치, 모델 버전, 작업 디렉토리 등의 매개변수를 설정할 수 있습니다. 이 설계는 유연성을 보장하면서 관리 및 유지보수를 용이하게 합니다.

사실 설정 파일은 인생의 선택과 같습니다. 특정 기능을 켜거나 끌 수 있지만, 코드에서의 선택은 후회하기가 더 쉽습니다.

## 모험단 작업 흐름

### 작업 분담의 예술

통일된 기술 아키텍처가 있으면 다음은 여러 Agent가 어떻게 협업할 것인가입니다. HagiCode는 작업 흐름 메커니즘을 설계하여 서로 다른 Agent가 다른 단계의 작업을 처리하도록 했습니다:

```
제안 생성 (사용자)
    │
    ▼
[Claude Code] ──제안 생성──▶ 제안 문서
    │                        │
    │                        ▼
    │               [Codebuddy] ──설명 최적화──▶ 최적화된 제안
    │                        │
    │                        ▼
    │               [Codex] ──수정 실행──▶ 코드 변경
    │                        │
    │                        ▼
    └───────────────▶ [iFlow] ──아카이빙──▶ 기록
```

이러한 분담의 장점은 각 Agent가 자신이 능숙한 작업에만 집중하고 "모든 것을 할 수 있을" 필요가 없다는 것입니다. Claude Code는 무에서 유를 창조하여 제안을 생성하고, Codebuddy는 제안 설명을 더 명확하게 만들고, Codex는 제안을 실제 코드 변경으로 변환하며, iFlow는 이러한 변경을 아카이빙하여 보존합니다.

사실 이것은 삶의 팀워크와 같습니다. 각자 자신의 역할이 있고, 합쳐야 큰일을 완성할 수 있습니다. 여기서 팀원이 AI일 뿐입니다.

### 핵심 실천 요점

실제 운영에서 다음과 같은 경험을 요약했습니다:

**1. Agent 선택 전략이 중요합니다**

무작위로 작업을 할당하는 것이 아니라 각 Agent의 장점에 따라 할당해야 합니다:
- 제안 생성: Claude Code 사용. 더 강한 컨텍스트 이해 능력이 있으므로
- 코드 실행: Codex 사용. 코드 수정에서 더 정밀하므로
- 제안 최적화: Codebuddy 사용. 가성비가 높으므로
- 아카이빙 저장: iFlow 사용. 안정적이고 신뢰할 수 있으므로

毕竟 적합한 사람에게 적합한 일을 시키는 것은千古不变의 이치입니다.

**2. 설정 격리로 안정성 보장**

각 Agent의 설정은 독립적으로 관리되며, 환경 변수 덮어쓰기를 지원하고, 작업 디렉토리도 서로 독립적입니다. 이렇게 하면 한 Agent의 설정 오류가 다른 Agent에 영향을 주지 않습니다.

이것은 삶의 경계와 같습니다. 각자 자신의 공간이 있고, 서로 방해하지 않아야 조화롭게 공존할 수 있습니다.

**3. 오류 처리 메커니즘**

단일 Agent 실패가 전체 프로세스에 영향을 주어서는 안 됩니다. 우리는 저하 전략을 구현했습니다. 특정 Agent 실행 실패 시 시스템이 자동으로 백업 방안으로 전환하거나 해당 단계를 건너뛰고 후속 작업을 계속 실행합니다. 동시에 완전한 로그 기록으로 사후 문제 해결을 용이하게 합니다.

毕竟 누구도 영원히 오류가 없다고 보장할 수 없습니다. 중요한 것은 오류를 어떻게 처리하는가입니다. 인생과 같습니다. 항상 좌절을 만나지만, 중요한 것은 어떻게 빠져나오는가입니다.

**4. 모니터링 및 가시성**

ACP 프로토콜(우리가 정의한 JSON-RPC 2.0 기반의 통신 프로토콜)을 통해 각 Agent의 실행 상태를 추적할 수 있습니다. 세션 격리는 동시성 안전을 보장하고, 동적 캐시는 성능을 최적화합니다.

毕竟 보이지 않는 것이 가장 문제가 되기 쉽습니다. 약간의 모니터링이라도 아무것도 모르는 것보다 낫습니다.

## 실제 효과 및 수익

이 다중 Agent 협업 설정을 채택한 후 HagiCode 프로젝트의 개발 효율이 현저히 향상되었습니다. 구체적으로:

1. **작업 처리 능력 배증**: 이전에는 하나의 Agent가 여러 작업을 동시에 처리해야 했지만, 이제는 병렬로 처리하여 처리량이 두 배 이상 증가했습니다
2. **출력 품질 안정화**: 각 Agent가 자신이 능숙한 작업에만 집중하여 출력 결과의 일관성과 품질이 더 높아졌습니다
3. **유지보수 비용 감소**: 통일된 인터페이스와 설정 관리로 전체 시스템의 유지보수 및 확장이 더 쉬워졌습니다
4. **신규 Agent 추가 간소화**: 새로운 AI 제품을 연결하려면 인터페이스 구현, 설정 추가만 하면 되며 핵심 로직을 수정할 필요가 없습니다

이 솔루션은 HagiCode 자체의 문제를 해결했을 뿐만 아니라, 다중 Agent 협업이 실현 가능한 아키텍처 선택임을 증명했습니다.

사실 효과는 꽤 명확합니다. 다만 과정이 좀 번거로웠을 뿐입니다.

## 요약

이 글에서는 HagiCode 프로젝트의 다중 Agent 협업 설정 실천 경험을 공유했습니다. 핵심 요점은 다음과 같습니다:

1. **표준화된 인터페이스**: `IAIProvider`를 통해 서로 다른 Agent의 행동을 통일하여 코드가 하위가 어느 회사의 제품인지 무시할 수 있게 합니다
2. **팩토리 패턴**: `ActivatorUtilities.CreateInstance`를 사용하여 Provider 인스턴스를 동적으로 생성하고 런타임 설정 및 의존성 주입을 지원합니다
3. **프로토콜 통일**: ACP 프로토콜로 Agent 간의 표준화된 통신을 구현합니다. JSON-RPC 2.0 기반의 양방향 통신 메커니즘입니다
4. **작업 분류**: 합리적으로 작업을 서로 다른 Agent에 분배하여 각자의 장점을 발휘하게 합니다. 하나의 Agent가 모든 것을 하려고 시도하는 것이 아니라

이 설계는 "다중 Agent 충돌" 문제를 해결했을 뿐만 아니라 모험단의 작업 흐름 메커니즘을 통해 개발 프로세스의 자동화 및 전문화를 실현했습니다.

여러 AI 어시스턴트 도입을 고려 중이시라면 이 글이 참고가 되기를 바랍니다. 물론 각 프로젝트의 상황이 다르므로 구체적인 솔루션은 실제 상황에 따라 조정해야 합니다.毕竟 만국에 통용되는 솔루션은 없으며, 자신에게 맞는 것이 가장 좋습니다.

아름다운 것은 반드시 소유할 필요가 없습니다. 그녀가 여전히 아름답다면, 그 아름다움을 잘 바라보면 됩니다. 기술 솔루션도 마찬가지입니다. 자신에게 맞는 것이 가장 좋습니다......
