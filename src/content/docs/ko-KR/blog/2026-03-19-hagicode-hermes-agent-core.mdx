## HagiCode가 Hermes를 종합 Agent 코어로 선택한 이유

> AI 보조 코딩 플랫폼을 구축할 때 적절한 Agent 코어를 선택하는 것은 시스템 능력의 한계를 직접 결정합니다.毕竟 어떤 일은 억지로 할 수 없습니다. 잘못된 프레임워크를 선택하면 아무리 애써도 제대로 되지 않습니다. 이 글에서는 HagiCode의 기술 선택에서의 고민과 Hermes Agent의 통합 실천을 공유합니다.

## 배경

AI 보조 코딩을 하면서 가장 골치 아픈 것은 기본 Agent 프레임워크를 선택하는 것입니다. 사실 시중에 선택할 수 있는 솔루션도 꽤 많습니다. 다만 — 어떤 것은 기능이 너무 단순하고, 어떤 것은 배포가 너무 복잡하며, 어떤 것은 확장성이 부족합니다. 우리가 원하는 것은 5달러 VPS에서도 실행할 수 있고, GPU 클러스터에도 연결할 수 있는 솔루션입니다. 이 요구가 높다고도 할 수 없고, 낮다고도 할 수 없지만, 적지 않은 사람들이 포기했습니다.

하지만 실제 상황은 많은 이른바 "만능 Agent"가 클라우드에서만 실행되거나 로컬 배포 요구사항이 비현실적으로 높다는 것입니다. 2주 동안 다양한 솔루션을 조사한 후 대담한 결정을 내렸습니다. 전체 Agent 코어를 뒤엎고 Hermes를 종합 Agent의 기본 엔진으로 채택합니다.

이 결정이 가져온 모든 것은 아마 필연이었을 것입니다.

## HagiCode에 대하여

이 글에서 공유하는 솔루션은 [HagiCode](https://hagicode.com) 프로젝트에서의 실천 경험에서 비롯되었습니다. HagiCode는 VSCode 확장, 데스크톱 클라이언트 및 웹 서비스를 통해 개발자에게 지능적인 코딩 어시스턴트를 제공하는 AI 보조 코딩 플랫폼입니다. 비슷한 도구를 사용해 보셨을 수도 있지만, 항상 뭔가 한 끗 부족하다는 느낌이 드셨을 것입니다. 이것도 이해합니다.

- GitHub: [github.com/HagiCode-org/site](https://github.com/HagiCode-org/site)
- 공식 웹사이트: [hagicode.com](https://hagicode.com)

## HagiCode에 Hermes가 필요한 이유

Hermes를 자세히 소개하기 전에, HagiCode에 왜 이러한 요구가 있는지 먼저 이야기해 보겠습니다. 세상의 일은 원하는 대로 되지 않는 경우가 많고, 항상 적절한 이유가 필요합니다.

AI 코드 어시스턴트로서 HagiCode는 여러 사용 시나리오를 동시에 지원해야 합니다:
- 로컬 개발 환경: 개발자가 자신의 컴퓨터에서 실행하기를 원하며, 데이터는 로컬을 벗어나지 않습니다. 이 시대에 데이터 보안은 크다고도 작다고도 할 수 없습니다
- 팀 협업 환경: 소규모 팀은 서버에 배포된 Agent를 공유할 수 있습니다. 비용을 절감해야 합니다. 모두가 쉽지 않으니까요
- 클라우드 탄력적 확장: 복잡한 작업 처리 시 GPU 클러스터로 자동 확장합니다. 만일의 사태에 대비해야 합니다

이러한 "이것도 저것도"의 요구로 인해 우리는 Hermes에 주목했습니다. 이 선택이 맞는지 모르겠습니다. 당시에는 더 나은 방법이 없었습니다.

## Hermes Agent란 무엇인가

Hermes Agent는 **Nous Research**가 만든 자율 AI Agent입니다. Nous Research에 익숙하지 않은 분도 있을 것입니다. Hermes, Nomos, Psyché 등 오픈소스 대형 언어 모델을 개발한 연구소입니다. 말하자면 그들도 꽤 힘든 일을 했습니다.这么多 좋은 것을 만들었는데 아는 사람이 많지 않으니까요.

전통적인 IDE 프로그래밍 어시스턴트나 단순한 API 채팅 래퍼와 달리 Hermes에는 특징이 있습니다. 실행 시간이 길수록 능력이 강해집니다. 일회성으로 작업을 완료하고 끝나는 것이 아니라, 장시간 실행 중 지속적으로 학습하고 경험을 축적합니다. 이 점은 사람과 비슷하지 않습니까?

### 핵심 특성

Hermes의 몇 가지 핵심 특성이 HagiCode의 요구와 정확히 일치합니다. 얼마나 우연한 일입니까?

이것은 HagiCode가 사용자 시나리오에 따라 가장 적절한 배포 방식을 선택할 수 있음을 의미합니다. 개인 사용자는 로컬에서, 팀 사용자는 서버에 배포하고, 복잡한 작업은 GPU에서 — 한 세트의 코드로 해결합니다. 이 시대에는 한 가지 일이라도 줄이는 것이 좋습니다.

**다중 플랫폼 메시지 게이트웨이**
Hermes는 Telegram, Discord, Slack, WhatsApp 등의 플랫폼을 기본 지원합니다. HagiCode에게 이것은 미래에 이러한 채널의 AI 어시스턴트를 쉽게 지원할 수 있음을 의미합니다.毕竟 누가 더 많은 길을 원하지 않겠습니까?

**풍부한 도구 시스템**
40개 이상의 내장 도구와 MCP(Model Context Protocol) 확장 능력. 이것은 코드 어시스턴트에게 매우 중요합니다. 셸 명령 실행, 파일 시스템 조작, Git 호출 등 모두 도구 지원이 필요합니다. 도구가 없는 Agent는 날개 없는 새와 같습니다. 날고 싶어도 날 수 없습니다.

**크로스 세션 메모리**
Hermes는 FTS5 전체 텍스트 검색으로 과거 대화를 검색하는 지속적 메모리 시스템을 가지고 있습니다. 이것은 Agent가 이전 컨텍스트를 기억하여 매번 "기억 상실"하지 않게 합니다. 때로는 저도 기억 상실하고 아무것도 생각하지 않고 싶지만, 할 수가 없습니다.

## HagiCode가 Hermes를 통합하는 방법

"왜"를 이야기했으니, 이제 "어떻게"를 살펴보겠습니다. 어떤 일은 이해하면 행동해야 합니다. 생각만 하고 실행하지 않는 것은 안 됩니다.

### Provider 레이어 추상화

HagiCode의 아키텍처에서 모든 AI Provider는 통일된 `IAIProvider` 인터페이스를 구현합니다:

```csharp
public sealed class HermesCliProvider : IAIProvider, IVersionedAIProvider
{
    public ProviderCapabilities Capabilities { get; } = new ProviderCapabilities
    {
        SupportsStreaming = true,   // 스트리밍 출력 지원
        SupportsTools = true,       // 도구 호출 지원
        SupportsSystemMessages = true, // 시스템 프롬프트 지원
        SupportsArtifacts = false
    };
}
```

이 추상 레이어 덕분에 HagiCode는 OpenAI, Claude, Hermes 등 다양한 AI Provider를 원활하게 전환할 수 있으며, 상위 호출 방식이 완전히 일치합니다. 말하자면 일이 줄어듭니다.

### ACP 통신 프로토콜

Hermes는 **ACP (Agent Communication Protocol)**를 사용하여 통신합니다. 이것은 Agent 통신을 위해 특별히 설계된 프로토콜이며, 주요 메서드는 다음과 같습니다:

| 메서드 | 설명 |
|--------|------|
| `initialize` | 연결 초기화, 프로토콜 버전 및 클라이언트 능력 획득 |
| `authenticate` | 인증 처리, 다양한 인증 방법 지원 |
| `session/new` | 새 세션 생성, 작업 디렉토리 및 MCP 서버 설정 |
| `session/prompt` | 프롬프트 전송 및 응답 획득 |

HagiCode는 `StdioAcpTransport`를 통해 ACP 전송 레이어를 구현하고, Hermes 하위 프로세스를 시작하여 표준 입출력으로 통신합니다. 이 일은 복잡하게 들리지만 실제로는 괜찮습니다. 주로 인내심이 필요합니다.

### 설정 관리

`HermesPlatformConfiguration` 클래스를 통해 설정을 관리합니다:

```csharp
public sealed class HermesPlatformConfiguration : IAcpPlatformConfiguration
{
    public string ExecutablePath { get; set; } = "hermes";
    public string Arguments { get; set; } = "acp";
    public int StartupTimeoutMs { get; set; } = 5000;
    public string ClientName { get; set; } = "HagiCode";
    public HermesAuthenticationConfiguration Authentication { get; set; }
    public HermesSessionDefaultsConfiguration SessionDefaults { get; set; }
}
```

`appsettings.json`에서 Hermes를 설정합니다:

```json
{
  "Providers": {
    "HermesCli": {
      "ExecutablePath": "hermes",
      "Arguments": "acp",
      "StartupTimeoutMs": 10000,
      "ClientName": "HagiCode",
      "Authentication": {
        "PreferredMethodId": "api-key",
        "MethodInfo": {
          "api-key": "your-api-key-here"
        }
      },
      "SessionDefaults": {
        "Model": "claude-sonnet-4-20250514",
        "ModeId": "default"
      }
    }
  }
}
```

설정이란 것은 보기에 간단하지만, 제대로 조정하려면 약간의 노력이 필요합니다.

### Orleans 분산 아키텍처

HagiCode는 Orleans를 사용하여 분산 시스템을 구축하며, Hermes 통합은 다음 컴포넌트를 통해 구현됩니다:

- **HermesGrain**: Orleans Grain 구현, 세션 실행 처리
- **HermesPlatformConfiguration**: 플랫폼별 설정
- **HermesAcpSessionAdapter**: ACP 세션 어댑터
- **HermesConsole**: 전용 검증 콘솔

Orleans라는 이름은 참 좋습니다. 전설의 알리바바 — 비록 이 Orleans가 저 Orleans가 아니지만, 이름이 좋으면 항상 가산점입니다.

### 완전한 실행 흐름

다음은 Hermes Provider의 핵심 실행 로직입니다:

```csharp
private async IAsyncEnumerable<AIStreamingChunk> StreamCoreAsync(
    AIRequest request,
    string? embeddedCommandPrompt,
    [EnumeratorCancellation] CancellationToken cancellationToken)
{
    // 1. 전송 레이어 생성, Hermes 하위 프로세스 시작
    await using var transport = new StdioAcpTransport(
        platformConfiguration.GetExecutablePath(),
        platformConfiguration.GetArguments(),
        platformConfiguration.GetEnvironmentVariables(),
        platformConfiguration.GetStartupTimeout(),
        _loggerFactory.CreateLogger<StdioAcpTransport>());
    await transport.ConnectAsync(cancellationToken);

    // 2. 초기화, 프로토콜 버전 및 인증 방법 획득
    var initializeResult = await SendHermesRequestAsync(
        transport, nextRequestId++, "initialize",
        BuildInitializeParameters(platformConfiguration), cancellationToken);

    // 3. 인증 처리
    var authMethods = ParseAuthMethods(initializeResult);
    if (!isAuthenticated)
    {
        var methodId = platformConfiguration.Authentication.ResolveMethodId(authMethods);
        await SendHermesRequestAsync(transport, nextRequestId++, "authenticate", ...);
    }

    // 4. 세션 생성
    var newSessionResult = await SendHermesRequestAsync(
        transport, nextRequestId++, "session/new",
        BuildNewSessionParameters(platformConfiguration, workingDirectory, model), cancellationToken);
    var sessionId = ParseSessionId(newSessionResult);

    // 5. 프롬프트 실행 및 스트리밍 응답 수집
    await foreach (var payload in transport.ReceiveMessagesAsync(cancellationToken))
    {
        // session/update 알림 처리, 스트리밍 청크로 변환
        if (TryParseSessionNotification(root, out var notification))
        {
            if (_responseMapper.TryConvertToStreamingChunk(notification, out var chunk))
            {
                yield return chunk;
            }
        }
    }
}
```

코드는 많이 보면 그런 것입니다. 중요한 것은 사고방식입니다, 맞죠?

### 상태 점검

Hermes 서비스의 가용성을 보장하기 위해 HagiCode는 상태 점검 메커니즘을 구현했습니다:

```csharp
public async Task<ProviderTestResult> PingAsync(CancellationToken cancellationToken = default)
{
    var response = await ExecuteAsync(
        new AIRequest
        {
            Prompt = "Reply with exactly PONG.",
            CessionId = null,
            AllowedTools = Array.Empty<string>(),
            WorkingDirectory = ResolveWorkingDirectory(null)
        },
        cancellationToken);

    var success = string.Equals(response.Content.Trim(), "PONG", StringComparison.OrdinalIgnoreCase);
    return new ProviderTestResult
    {
        ProviderName = Name,
        Success = success,
        ResponseTimeMs = stopwatch.ElapsedMilliseconds,
        ErrorMessage = success ? null : $"Unexpected Hermes ping response: '{response.Content}'."
    };
}
```

이것이 소위 "상태 점검"입니다. 사실 사람도 마찬가지입니다. 가끔 자신을 점검해야 합니다. 다만 보통은 누가 우리에게 무엇을 점검해야 하는지 알려주지 않습니다.

## 실천 시 주의사항

Hermes 통합 과정에서 미리 알아두면 좋은 몇 가지 함정이 있습니다. 이 시대에 함정을 밟아본 적 없는 사람이 어디 있겠습니까?

### 인증 방법 설정

Hermes는 여러 인증 방법(API Key, Token 등)을 지원하며, 실제 배포 상황에 따라 선택해야 합니다. 설정 오류는 연결 실패를 초래하지만, 오류 메시지가 직관적이지 않을 수 있습니다. 때로는 오류 메시지가 실제 원인과十万八千里 떨어져 있어 천천히 조사해야 합니다.

### MCP 서버 설정

세션 생성 시 MCP 서버 목록을 구성하여 Hermes가 외부 도구를 호출할 수 있게 합니다. 다만 주의해야 할 점이 있습니다:
- MCP 서버 주소에 접근 가능해야 합니다
- 타임아웃 시간을 합리적으로 설정해야 합니다
- 서버를 사용할 수 없을 때의 저하 처리

이 세상은 방어하기 어렵습니다.

### 작업 디렉토리 관리

각 세션에는 작업 디렉토리를 지정해야 하며, Hermes가 프로젝트 파일에 올바르게 접근할 수 있도록 해야 합니다. 다중 프로젝트 시나리오에서는 작업 디렉토리를 동적으로 전환해야 합니다. 말은 간단하지만, 고려해야 할 경우가 꽤 많습니다.

### 응답 집계 처리

Hermes의 응답은 `session/update` 알림과 최종 결과에 분산될 수 있으며, 올바르게 병합 처리하지 않으면 콘텐츠 손실이 발생할 수 있습니다. 이것도 많이 손해를 본 적이 있습니다. 천천히 좋아졌습니다.

### 오류 처리 전략

런타임 오류는 명확하게 반환해야 하며, 조용히 다른 Provider로 폴백해서는 안 됩니다. 그래야 사용자가 Hermes에 문제가 있다는 것을 알 수 있고, 알 수 없는 다른 모델로 바뀐 것이 아닙니다.毕竟 일을 대충 처리하는 것은 이런 식이 아닙니다.

## 요약

HagiCode가 Hermes를 종합 Agent 코어로 선택한 것은 즉흥적인 결정이 아니라 실제 요구와 기술적 특성에 기반한 신중한 선택입니다. 이 선택이 맞는지 지금 말하기는 이르지만, 현재 사용하기에는 꽤 편합니다.

Hermes가 제공하는 유연한 배포 능력으로 HagiCode는 다양한 사용 시나리오에 적응할 수 있습니다. 강력한 도구 시스템과 MCP 지원으로 AI 어시스턴트가 진정으로 실질적인 일을 할 수 있습니다. ACP 프로토콜과 Provider 추상 레이어는 전체 통합 과정을 명확하고 통제 가능하게 만듭니다.

AI 프로젝트를 위해 Agent 프레임워크를 선택 중이시라면 이 글이 참고가 되기를 바랍니다.毕竟, 올바른 기본 아키텍처를 선택하면 후속 개발이 훨씬 쉬워집니다......
