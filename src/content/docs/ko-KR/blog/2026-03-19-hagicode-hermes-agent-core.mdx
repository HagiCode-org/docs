## HagiCode가 Hermes를 종합 Agent 코어로 선택한 이유

> AI 보조 코딩 플랫폼을 구축할 때, 적절한 Agent 코어를 선택하는 것은 시스템 능력의 한계를 직접 결정합니다.毕竟有些事情，勉强不来——选错了框架，怎么折腾都不得劲。本文分享 HagiCode 在技术选型中的思考，以及 Hermes Agent 的集成实践。

## 배경

AI 보조 코딩을 하는 일에서 가장 골치 아픈 것은 기본 Agent 프레임워크를 선택하는 것입니다. 사실 시장에 선택 가능한 솔루션도 꽤 많지만, 어떤 것은 기능이 너무 단순하고, 어떤 것은 배포가 너무 복잡하며, 어떤 것은 확장성이 부족합니다. 우리는 5달러 VPS에서도 실행되고 GPU 클러스터에도 접속할 수 있는 솔루션을 원했습니다. 요구사항이 높지도 않지만, 꽤 많은 사람들이 그만두기도 했습니다.

하지만 실제 상황은 많은 소위 '전능 Agent'가 클라우드에서만 실행되거나 로컬 배포 요구사항이 비현실적으로 높다는 것입니다. 2주간 다양한 솔루션을 조사한 후, 우리는 대담한 결정을 내렸습니다: 전체 Agent 코어를 처음부터 다시 만들고 Hermes를 종합 Agent의 기본 엔진으로 채택합니다.

이 결정이 가져온 모든 것은 어쩌면 운명일지도 모릅니다.

## HagiCode에 대해

이 글에서 공유하는 솔루션은 [HagiCode](https://hagicode.com) 프로젝트의 실천 경험에서 나온 것입니다. HagiCode는 AI 보조 코딩 플랫폼으로, VSCode 확장, 데스크톱 클라이언트 및 웹 서비스를 통해 개발자에게 지능형 코딩 도우미를 제공합니다. 아마도 비슷한 도구를 사용해 보셨을 수도 있지만, 항상 뭔가 부족하다고 느끼셨을 것입니다—이것도 이해합니다.

- GitHub: [github.com/HagiCode-org/site](https://github.com/HagiCode-org/site)
- 공식 웹사이트: [hagicode.com](https://hagicode.com)

## HagiCode에 Hermes가 필요한 이유

Hermes에 대해 자세히 소개하기 전에, HagiCode가 왜 이러한 요구사항을 가지게 되었는지 먼저 이야기해 보겠습니다. 세상일은 생각한 대로 되지 않는 경우가 많습니다.

AI 코드 도우미로서 HagiCode는 다양한 사용 시나리오를 동시에 지원해야 합니다:
- 로컬 개발 환경: 개발자는 자신의 컴퓨터에서 실행되고 데이터가 로컬을 벗어나지 않기를 원합니다—요즘, 데이터 보안은 크지도 작지도 않은 문제입니다
- 팀 협업 환경: 소규모 팀은 서버에 배포된 Agent를 공유할 수 있습니다—돈을 절약해야 하니, 누구나 쉽지 않습니다
- 클라우드 탄력적 확장: 복잡한 작업을 처리할 때 GPU 클러스터로 자동 확장될 수 있습니다—준비된 것이 항상 좋습니다

이러한 '이것도 저것도' 요구사항으로 우리의 시선은 Hermes로 향했습니다. 이 선택이 맞는지는 모르겠지만, 당시에는 다른 더 나은 방법이 없었습니다.

## Hermes Agent란 무엇인가

Hermes Agent는 **Nous Research**가 만든 자율 AI Agent입니다. Nous Research에 대해 익숙하지 않은 분들도 계실 것입니다—그들은 Hermes, Nomos, Psyché 등 오픈소스 대규모 모델을 개발한 연구소입니다. 말하자면 그들도 꽤 힘들게 했고,这么多 좋은 것들을 만들었는데 알려지지 않은 것도 안타까운 일입니다.

전통적인 IDE 프로그래밍 도우미나 단순한 API 채팅 래퍼와 달리, Hermes는 한 가지 특징이 있습니다: 실행 시간이 길수록 능력이 강해집니다. 일회성으로 작업을 완료하는 것이 아니라 장시간 실행 중에 지속적으로 학습하고 경험을 축적할 수 있습니다. 이 점도 사람과 비슷하지 않습니까?

### 핵심 기능

Hermes의 핵심 기능들은 HagiCode의 요구사항과 정확히 일치합니다. 얼마나 우연한 일입니까?

이것은 HagiCode가 사용자 시나리오에 따라 가장 적합한 배포 방식을 선택할 수 있다는 것을 의미합니다. 개인 사용자는 로컬에서 실행하고, 팀 사용자는 서버에 배포하며, 복잡한 작업은 GPU로—하나의 코드로 해결합니다. 이 세상에서, 일을 줄일 수 있으면 줄이는 것이 좋습니다.

**다중 플랫폼 메시지 게이트웨이**
Hermes는 Telegram, Discord, Slack, WhatsApp 등 플랫폼을 기본적으로 지원합니다. HagiCode에게 이는 미래에 이러한 채널의 AI 도우미를 쉽게 지원할 수 있다는 것을 의미합니다.毕竟谁不想多几条路呢？

**풍부한 도구 시스템**
40개 이상의 내장 도구와 MCP(모델 컨텍스트 프로토콜) 확장 능력을 갖추고 있습니다. 이것은 코드 도우미에게 너무나 중요합니다—shell 명령 실행, 파일 시스템 조작, Git 호출 등은 모두 도구 지원이 필요합니다. 도구가 없는 Agent는 날개가 없는 새와 같습니다—날고 싶어도 날 수 없습니다.

**세션 간 메모리**
Hermes는 FTS5 전체 텍스트 검색으로 과거 대화를 검색하는 지속 메모리 시스템을 가지고 있습니다. 이것은 Agent가 이전 컨텍스트를 기억할 수 있고 매번 '기억상실'이 되지 않게 합니다. 때로는 저도 잊어버리고 싶지만, 하기가 어렵습니다.

## HagiCode가 Hermes를 통합하는 방법

'왜'를 설명했으니 '어떻게'를 살펴보겠습니다. 어떤 일은 이해하면 바로 행동해야 합니다, 생각만 하고 실행하지 않는 것도 일이 아닙니다.

### Provider 계층 추상화

HagiCode 아키텍처에서 모든 AI Provider는 통합된 `IAIProvider` 인터페이스를 구현합니다:

```csharp
public sealed class HermesCliProvider : IAIProvider, IVersionedAIProvider
{
    public ProviderCapabilities Capabilities { get; } = new ProviderCapabilities
    {
        SupportsStreaming = true,   // 스트리밍 출력 지원
        SupportsTools = true,       // 도구 호출 지원
        SupportsSystemMessages = true, // 시스템 프롬프트 지원
        SupportsArtifacts = false
    };
}
```

이 추상화 계층을 통해 HagiCode는 OpenAI, Claude 또는 Hermes와 상관없이 상위 호출 방식이 완전히 동일하게 다양한 AI Provider를 원활하게 전환할 수 있습니다. 말하자면, 일을 줄이는 것입니다.

### ACP 통신 프로토콜

Hermes는 **ACP (Agent Communication Protocol)**를 사용하여 통신합니다. 이것은 Agent 통신을 위해 특별히 설계된 프로토콜로, 주요 방법은 다음과 같습니다:

| 메서드 | 설명 |
|------|------|
| `initialize` | 연결을 초기화하고 프로토콜 버전과 클라이언트 기능을 가져옵니다 |
| `authenticate` | 인증을 처리하고 다양한 인증 방법을 지원합니다 |
| `session/new` | 새 세션을 만들고 작업 디렉터리와 MCP 서버를 설정합니다 |
| `session/prompt` | 프롬프트를 보내고 응답을 가져옵니다 |

HagiCode는 `StdioAcpTransport`를 통해 ACP 전송 계층을 구현하고, Hermes 하위 프로세스를 시작하여 표준 입출력으로 통신합니다. 이 일은 복잡하게 들리지만, 실행하는 것은 괜찮습니다—주로 인내심이 필요합니다.

### 구성 관리

`HermesPlatformConfiguration` 클래스를 통해 구성을 관리합니다:

```csharp
public sealed class HermesPlatformConfiguration : IAcpPlatformConfiguration
{
    public string ExecutablePath { get; set; } = "hermes";
    public string Arguments { get; set; } = "acp";
    public int StartupTimeoutMs { get; set; } = 5000;
    public string ClientName { get; set; } = "HagiCode";
    public HermesAuthenticationConfiguration Authentication { get; set; }
    public HermesSessionDefaultsConfiguration SessionDefaults { get; set; }
}
```

`appsettings.json`에서 Hermes를 구성합니다:

```json
{
  "Providers": {
    "HermesCli": {
      "ExecutablePath": "hermes",
      "Arguments": "acp",
      "StartupTimeoutMs": 10000,
      "ClientName": "HagiCode",
      "Authentication": {
        "PreferredMethodId": "api-key",
        "MethodInfo": {
          "api-key": "your-api-key-here"
        }
      },
      "SessionDefaults": {
        "Model": "claude-sonnet-4-20250514",
        "ModeId": "default"
      }
    }
  }
}
```

구성은 간단해 보이지만, 정확하게 조정하려면 상당한 시간이 걸립니다.

### Orleans 분산 아키텍처

HagiCode는 Orleans를 사용하여 분산 시스템을 구축하고 Hermes 통합은 다음 구성 요소를 통해 구현됩니다:

- **HermesGrain**: Orleans Grain 구현, 세션 실행 처리
- **HermesPlatformConfiguration**: 플랫폼별 구성
- **HermesAcpSessionAdapter**: ACP 세션 어댑터
- **HermesConsole**: 전용 검증 콘솔

Orleans라는 이름은 참 좋습니다. 전설의 알라딘—비록 이 Orleans는 저 Orleans가 아니지만, 좋은 이름은 항상 가산점입니다.

### 전체 실행 흐름

다음은 Hermes Provider의 핵심 실행 논리입니다:

```csharp
private async IAsyncEnumerable<AIStreamingChunk> StreamCoreAsync(
    AIRequest request,
    string? embeddedCommandPrompt,
    [EnumeratorCancellation] CancellationToken cancellationToken)
{
    // 1. 전송 계층을 만들고 Hermes 하위 프로세스 시작
    await using var transport = new StdioAcpTransport(
        platformConfiguration.GetExecutablePath(),
        platformConfiguration.GetArguments(),
        platformConfiguration.GetEnvironmentVariables(),
        platformConfiguration.GetStartupTimeout(),
        _loggerFactory.CreateLogger<StdioAcpTransport>());
    await transport.ConnectAsync(cancellationToken);

    // 2. 초기화, 프로토콜 버전과 인증 방법 가져오기
    var initializeResult = await SendHermesRequestAsync(
        transport, nextRequestId++, "initialize",
        BuildInitializeParameters(platformConfiguration), cancellationToken);

    // 3. 인증 처리
    var authMethods = ParseAuthMethods(initializeResult);
    if (!isAuthenticated)
    {
        var methodId = platformConfiguration.Authentication.ResolveMethodId(authMethods);
        await SendHermesRequestAsync(transport, nextRequestId++, "authenticate", ...);
    }

    // 4. 세션 만들기
    var newSessionResult = await SendHermesRequestAsync(
        transport, nextRequestId++, "session/new",
        BuildNewSessionParameters(platformConfiguration, workingDirectory, model), cancellationToken);
    var sessionId = ParseSessionId(newSessionResult);

    // 5. 프롬프트 실행 및 스트리밍 응답 수집
    await foreach (var payload in transport.ReceiveMessagesAsync(cancellationToken))
    {
        // session/update 알림 처리, 스트리밍 청크로 변환
        if (TryParseSessionNotification(root, out var notification))
        {
            if (_responseMapper.TryConvertToStreamingChunk(notification, out var chunk))
            {
                yield return chunk;
            }
        }
    }
}
```

코드를 많이 보면 그게 그거라고 느껴집니다. 중요한 것은 사고방식입니다.

### 상태 확인

Hermes 서비스의 가용성을 보장하기 위해 HagiCode는 상태 확인 메커니즘을 구현했습니다:

```csharp
public async Task<ProviderTestResult> PingAsync(CancellationToken cancellationToken = default)
{
    var response = await ExecuteAsync(
        new AIRequest
        {
            Prompt = "Reply with exactly PONG.",
            CessionId = null,
            AllowedTools = Array.Empty<string>(),
            WorkingDirectory = ResolveWorkingDirectory(null)
        },
        cancellationToken);

    var success = string.Equals(response.Content.Trim(), "PONG", StringComparison.OrdinalIgnoreCase);
    return new ProviderTestResult
    {
        ProviderName = Name,
        Success = success,
        ResponseTimeMs = stopwatch.ElapsedMilliseconds,
        ErrorMessage = success ? null : $"Unexpected Hermes ping response: '{response.Content}'."
    };
}
```

이것이 소위 '상태 확인'일 것입니다. 사실 사람도 마찬가지입니다. 항상 때때로 자신을 점검해야 합니다—단, 보통은 우리가 무엇을 점검해야 할지 알려주지 않습니다.

## 실제 주의사항

Hermes 통합 과정에서 미리 알아두어야 할 함정들이 있습니다. 요즘 누구나 몇 번쯤 함정에 빠져보지 않았습니까?

### 인증 방법 구성

Hermes는 다양한 인증 방법(API Key, Token 등)을 지원하므로 실제 배포 상황에 따라 선택해야 합니다. 구성이 잘못되면 연결이 실패하지만 오류 메시지가 명확하지 않을 수 있습니다. 때로는 오류 메시지가 실제 원인과 전혀 다를 수 있어 점진적으로 확인해야 합니다.

### MCP 서버 구성

세션 생성 시 MCP 서버 목록을 구성하여 Hermes가 외부 도구를 호출하도록 할 수 있습니다. 하지만 주의해야 합니다:
- MCP 서버 주소는 접근 가능해야 합니다
- 제한 시간을 합리적으로 설정해야 합니다
- 서버를 사용할 수 없을 때의 대처 방안이 필요합니다

이 세상에서, 방어는 불가능합니다.

### 작업 디렉터리 관리

각 세션은 작업 디렉터리를 지정해야 하므로 Hermes가 프로젝트 파일에 올바르게 접근할 수 있습니다. 다중 프로젝트 시나리오에서는 작업 디렉터리를 동적으로 전환해야 합니다. 간단해 들리지만 고려해야 할 상황도 많습니다.

### 응답 집계 처리

Hermes의 응답은 `session/update` 알림과 최종 결과에 분산될 수 있으므로 올바르게 병합 처리해야 합니다. 그렇지 않으면 콘텐츠 손실이 발생할 수 있습니다. 이 일로 저도 많이 고생했습니다.

### 오류 처리 전략

런타임 오류는 명확하게 반환해야 하며 다른 Provider로 조용히 대체하면 안 됩니다. 그래야 사용자가 Hermes에 문제가 있다는 것을 알고 이상하게 다른 모델로 바뀐 것이 아님을 알 수 있습니다.

## 요약

HagiCode가 Hermes를 종합 Agent 코어로 선택한 것은 충동적인 결정이 아니라 실제 요구사항과 기술 특성을 기반으로 한 신중한 선택이었습니다. 이 선택이 맞았는지는 지금 말하기 이르지만, 현재 사용하기에 꽤 편안합니다.

Hermes의 유연한 배포 능력으로 HagiCode는 다양한 사용 시나리오에 적응할 수 있습니다. 강력한 도구 시스템과 MCP 지원으로 AI 도우미가 실제로 일을 할 수 있으며, ACP 프로토콜과 Provider 추상화 계층은 전체 통합 과정을 명확하고 제어 가능하게 만듭니다.

AI 프로젝트를 위해 Agent 프레임워크를 선택 중이라면 이 글이 도움이 되기를 바랍니다.毕竟，选对底层架构，后续开发会轻松很多......
