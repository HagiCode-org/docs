## AI 코딩 도우미의 환각 문제: OpenSpec으로 명세 기반 개발 구현하기

> AI 코딩 도우미는 강력하지만, 종종 실제 요구사항에 맞지 않고 프로젝트 규칙을 위반하는 코드를 생성합니다. 이 글에서는 HagiCode 프로젝트가 OpenSpec 프로세스를 통해 "명세 기반 개발"을 구현하고, 구조화된 제안 메커니즘으로 AI 환각 위험을 획기적으로 줄이는 방법을 공유합니다.

## 배경

GitHub Copilot이나 ChatGPT로 코드를 작성해 본 경험이 있는 분들은 다음과 같은 경험을 해보셨을 것입니다: AI가 생성한 코드는 멀짝하게 보이지만, 실제로 사용해 보면 문제가 투성이입니다. 프로젝트의 어떤 컴포넌트를 잘못 사용했거나, 팀의 코딩 규칙을 무시했거나, 존재하지 않는 가정을 바탕으로 긴 로직을 작성했을 수 있습니다.

이것이 소위 "AI 환각" 문제입니다—프로그래밍 영역에서는 그럴듯해 보이지만 실제로는 프로젝트의 실제 상황에 맞지 않는 코드를 생성하는 형태로 나타납니다.

사실 이 일도 참 답답합니다. AI 코딩 도우미가 점점 보편화되면서 이 문제도 더 심각해지고 있습니다. 결국 AI는 프로젝트 역사, 아키텍처 결정, 코딩 규칙에 대한 이해가 부족하고, 자유도가 너무 높으면 실제와 맞지 않는 코드를 "창의적으로" 생성하기 쉽습니다. 이것은 글쓰기와 비슷합니다. 법칙 없이 글을 쓰면 천방지축으로 쓰기 쉽지만, 실제로는 그렇지 않습니다.

이러한 문제점을 해결하기 위해 우리는 대담한 결정을 내렸습니다: AI를 더 똑똑하게 만드는 것이 아니라, 그것에 "규칙"이라는 채울 씌우는 것입니다. 이 결정이 가져온 변화는 상상보다 훨씬 클 수 있습니다—나중에 구체적으로 설명하겠습니다.

## HagiCode에 대하여

이 글에서 공유하는 솔루션은 [HagiCode](https://hagicode.com) 프로젝트의 실제 경험에서 나왔습니다. HagiCode는 구조화된 엔지니어링 실천을 통해 AI 프로그래밍의 실제 문제를 해결하는 데 전념하는 오픈 소스 AI 코드 도우미 프로젝트입니다.

## AI 환각의 근원

솔루션을 자세히 살펴보기 전에, 문제가 어디서 발생하는지 먼저 살펴봅시다. 적을 알고 나를 알면 백전백승이라는 말이죠. 이 말을 AI에 적용해보니 꽤 재미있네요.

### 컨텍스트 부족

AI 모델은 공개 코드 저장소로 학습되었지만, 귀하의 프로젝트에는 자체 역사, 약속, 아키텍처 결정이 있습니다. 이러한 "암묵적 지식"은 AI가 직접 얻을 수 없으므로 생성된 코드는 종종 프로젝트 실제 상황과 동떨어집니다.

이것을 전적으로 AI 탓으로 돌릴 수도 없습니다. 결국 그것은 귀하의 프로젝트에 있어본 적이 없으니, 귀하의 까다로운 규칙을 어찌 알겠습니까? 마치 새로 온 인턴처럼, 규칙을 모르는 것은 정상입니다. 다만 대가가 조금 클 뿐입니다.

### 과도한 자유도

AI에게 "사용자 인증 기능을 구현해 도와줘"라고 물으면, 어떤 형태의 코드든 생성할 수 있습니다. 명확한 제약이 없으면 AI는 자신이 "합리적"이라고 생각하는 방식으로 구현하고, 귀하의 프로젝트 요구사항에 따르지 않습니다.

이것은 프로젝트 규칙을 배워본 적 없는 사람에게 자유롭게 발휘하도록 하는 것과 같습니다. 문제가 없을 리가 없죠. 사실 그것이 책임감이 없는 것은 아니라, 처음부터 책임이 무엇인지 모를 뿐입니다.

### 검증 부족

AI가 코드를 생성한 후, 구조화된 검토 프로세스를 거치지 않으면 잘못된 가정에 기반한 코드가 바로 코드 저장소에 들어갑니다. 테스트나 심지어 프로덕션 환경에서 문제를 발견하면 대가가 너무 큽니다.

이것은 양을 잃고 외양간을 고치는 것과 같습니다. 다만 양이 이미 다 달아났는데 외양간을 고쳐서 무엇에 쓰겠습니까? 이 도리는 누구나 알지만, 정말로 실행하려면 또 귀찮게 느껴집니다. 결국 일이 나빠지기 전에는 누구도 더 많은 시간을 쓰고 싶어 하지 않습니다.

## OpenSpec: 명세 기반 개발의 답

HagiCode는 OpenSpec을 솔루션으로 선택했고, 핵심 아이디어는 **모든 코드 변경은 구조화된 제안 프로세스를 거쳐야 한다**는 것입니다. 추상적인 아이디어를 실행 가능한 계획으로 전환하는 것입니다.

이 말은 거창하지만, 사실 AI가 코드를 작성하기 전에 먼저 요구사항 문서를 작성하게 하는 것입니다. 결국 모든 일은 준비하면 성립하고, 준비하지 않으면 실패합니다. 옛사람들의 말은 결코 틀리지 않습니다.

### OpenSpec이란 무엇인가

OpenSpec은 npm 기반 명령줄 도구(`@fission-ai/openspec`)로, 표준화된 제안 파일 구조와 검증 메커니즘을 정의합니다. 간단히 말해, AI가 코드를 작성하기 전에 먼저 "요구사항 문서를 작성"하게 하는 것입니다.

### 3단계 프로세스로 환각 방지

OpenSpec은 3단계 프로세스를 통해 제안 품질을 보장합니다.

**Step 1: 제안 초기화** - 세션 상태를 Openspecing으로 설정
**Step 2: 중간 처리** - Openspecing 상태를 유지하며 점진적으로 산출물 완성
**Step 3: 제안 완료** - Reviewing 상태로 전환

이 디자인에는 아주 교묘한 디테일이 있습니다: 첫 번째 단계에서는 `ProposalGenerationStart` 타입을 사용하고, 완료해도 상태 전환이 트리거되지 않습니다. 이는 전체 다단계 프로세스가 완료되기 전에 너무 빨리 검토 단계로 들어가지 않도록 보장합니다.

사실 이 디테일도 꽤 재미있습니다. 요리를 하는 것과 같습니다. 불이 제대로 오기 전에 뚜껑을 열면, 아무것도 제대로 만들 수 없습니다. 인내심을 가지고 단계별로 진행해야만 마침내 좋은 요리를 만들 수 있습니다.

```csharp
// HagiCode 프로젝트의 구현
public enum MessageAssociationType
{
    ProposalGeneration = 2,
    ProposalExecution = 3,

    /// <summary>
    /// 3단계 제안 생성 프로세스의 시작을 표시
    /// 완료해도 Reviewing 상태로 전환되지 않음
    /// </summary>
    ProposalGenerationStart = 5
}
```

### 표준화된 파일 구조

모든 OpenSpec 제안은 동일한 디렉토리 구조를 따릅니다.

```
openspec/
├── changes/                    # 활성 및 보관된 변경사항
│   ├── {change-name}/
│   │   ├── proposal.md        # 제안 설명
│   │   ├── design.md          # 설계 문서
│   │   ├── specs/             # 기술 명세
│   │   └── tasks.md           # 실행 가능 작업 목록
│   └── archive/               # 보관된 변경사항
└── specs/                     # 독립적인 명세 저장소
```

HagiCode 프로젝트의 통계에 따르면 현재 4000개 이상의 보관된 변경사항과 15만 줄 이상의 명세 파일이 있습니다. 이러한 역사적 축적은 AI가 따라야 할 규칙을 제공할 뿐만 아니라 팀에 귀중한 지식베이스도 제공합니다.

이것은 옛사람들이 남긴 전적과 같습니다. 많이 읽으면 자연스럽게 요령을 터득할 수 있습니다. 다만 지금은 이 전적이 대나무 쪽에 쓰여 있는 것이 아니라 파일에 저장되어 있을 뿐입니다.

### 다층 검증 메커니즘

시스템은 제안 품질을 보장하기 위해 다층 검증을 구현합니다.

```csharp
// 필수 파일 존재 검증
ValidateProposalFiles()

// 실행 전제조건 검증
ValidateExecuteAsync()

// 시작 조건 검증
ValidateStartAsync()

// 보관 조건 검증
ValidateArchiveAsync()

// 제안 이름 형식 검증(kebab-case)
ValidateNameFormat()
```

이러한 검증은 단계별로 확인하는 문지기와 같습니다. 정말로 자격 있는 제안만 통과할 수 있습니다. 번거로워 보이지만, 나쁜 코드가 코드 저장소에 들어가는 것보다는 낫지 않을까요?

### 프롬프트 템플릿 제약

HagiCode에서 AI는 실행할 때 미리 정의된 Handlebars 템플릿을 사용하며, 이 템플릿에는 명확한 단계 지침과 보호 조치가 포함되어 있습니다. 예를 들어:

- 사용자 의도를 이해하지 못했을 때 계속 금지
- 검증되지 않은 코드 생성 금지
- 이름이 유효하지 않을 때 다시 제공하도록 요구
- 변경사항이 이미 존재하는 경우 다시 만드는 대신 계속 명령 사용 제안

이러한 "발에 족쇄를 차고 춤을 추는" 방식은 오히려 AI를 요구사항 이해와 규칙에 맞는 코드 생성에 더 집중하게 합니다. 사실 제약은 반드시 나쁜 것은 아닙니다. 결국 자유가 지나치면 혼란스러워지기 쉽습니다.

## 실천: 프로젝트에서 OpenSpec 사용 방법

### 설치 및 초기화

```bash
npm install -g @fission-ai/openspec@1
openspec --version  # 설치 확인
```

프로젝트 루트 디렉토리에 `openspec/` 폴더 구조가 자동으로 생성됩니다.

이 단계는 사실 설명할 것도 없습니다. 도구를 설치하는 것이니 다들 아실 겁니다. 다만 `@fission-ai/openspec@1` 버전을 사용하세요. 새 버전에는 문제가 있을 수 있습니다. 결국 안정성이 최우선입니다.

### 제안 생성

HagiCode의 대화형 인터페이스에서 빠른 명령을 사용합니다.

```
/opsx:new
```

또는 변경사항 이름과 대상 저장소를 지정합니다.

```
/opsx:new "add-user-auth" --repos "repos/web"
```

제안을 만드는 일은 글을 쓸 때 개요를 짜는 것과 같습니다. 개요가 있으면 뒤에 쓰기가 쉬워집니다. 다만 많은 사람들이 바로 쓰기 시작하고, 절반쯤 써서 생각이 통하지 않는다는 것을 발견하면 그때가 정말头疼입니다.

### 산출물 생성

`/opsx:continue`를 사용하여 필요한 산출물을 점진적으로 생성합니다.

**proposal.md** - 변경사항의 목적과 범위 설명
```markdown
# Proposal: Add User Authentication

## Why
현재 시스템에는 사용자 인증 기능이 없어 민감한 API를 보호할 수 없습니다.

## What Changes
- JWT 인증 미들웨어 추가
- 로그인/등록 API 구현
- 프론트엔드 통합 업데이트
```

**design.md** - 상세한 기술 설계
```markdown
# Design: Add User Authentication

## Context
현재 공개 API를 사용하여 누구나 접근할 수 있습니다...

## Decisions
1. JWT를 Session이 아닌 선택...
2. HS256 알고리즘 사용...

## Risks
- 토큰 유출 위험...
- 완화 조치...
```

**specs/** - 기술 명세 및 테스트 시나리오
```markdown
# user-auth Specification

## Requirements

### Requirement: JWT Token Generation
시스템은 HS256 알고리즘을 사용하여 JWT 토큰을 생성해야 합니다(SHALL).

#### Scenario: Valid login
- WHEN 사용자가 유효한 자격증명을 제공하면
- THEN 시스템은 유효한 JWT 토큰을 반환해야 합니다(SHALL)
```

**tasks.md** - 실행 가능한 작업 목록
```markdown
# Tasks: Add User Authentication

## 1. Backend Changes
- [ ] 1.1 AuthController 생성
- [ ] 1.2 JWT 미들웨어 구현
- [ ] 1.3 단위 테스트 추가
```

이러한 산출물은 사실 글을 쓸 때 초안과 같습니다. 초안이 잘 작성되면 본문이 자연스럽게流畅해집니다. 다만 많은 사람들이 초안 작성을 싫어하고 시간 낭비라고 생각하지만, 사실 초안이 생각을 정리하는 가장 좋은 곳입니다.

### 검토 및 적용

모든 산출물을 완료한 후:

```
/opsx:apply
```

AI는 모든 컨텍스트 파일을 읽고 tasks.md의 목록에 따라 단계별로 작업을 실행합니다. 이때 명확한 규칙이 있으므로 생성된 코드 품질이 훨씬 높습니다.

사실 이 단계까지 오면 일은 이미 절반 완성된 것입니다. 명확한 작업 목록이 있으면 나머지는 절차대로 실행하는 것입니다. 다만 많은 사람들이 앞 단계를 건너뛰고 바로 여기로 오니, 품질이 자연스럽게 보장되기 어렵습니다.

### 보관

변경사항을 완료한 후:

```
/opsx:archive
```

완료된 변경사항을 `archive/` 디렉토리로 이동하여 나중에 조회 및 재사용에 편리하게 합니다.

보관은 꽤 중요합니다. 완성된 글을 잘 정리하는 것과 같습니다. 나중에 비슷한 문제를 만나면 이전 기록을 찾아보면 답이 있을 수 있습니다. 다만 많은 사람들이 귀찮아서 하지 않지만, 사실 이러한 축적이 가장 귀중한 자산입니다.

## 주의사항 및 모범 사례

### 제안 명명 규칙

kebab-case 형식을 사용하고, 문자로 시작하며 소문자, 숫자, 하이픈만 포함합니다:
- ✅ `add-user-auth`
- ❌ `AddUserAuth`
- ❌ `add--user-auth`

명명 규칙은 사실 별것 아니지만, 통일하는 것이 좋습니다. 결국 코드에서 consistency가 중요합니다. 다만 많은 사람들이 신경 쓰지 않습니다.

### 일반적인 오류 방지

1. **3단계 프로세스의 단계 1에서 잘못된 타입 사용** - 상태가 너무 빨리 전환됨
2. **마지막 단계에서 상태 전환 트리거를 잊음** - Openspecing 상태에 갇힘
3. **검토 없이 바로 실행** - 모든 산출물이 완전한지 먼저 검증해야 함

이러한 오류는 초보자가 범하기 쉽습니다. 베테랑은 자연스럽게 피하는 방법을 압니다. 다만 초보자는 언젠가 베테랑이 되는 날이 오고, 돌아가는 길을 가더라도 너무 많이 돌아가지 않기를 바랄 뿐입니다.

### 다중 변경사항 관리

OpenSpec은 여러 제안을 동시에 관리할 수 있습니다. 이는 대규모 기능을 처리할 때 특히 유용합니다.

```bash
# 모든 활성 변경사항 보기
openspec list

# 특정 변경사항으로 전환
openspec apply "add-user-auth"

# 변경사항 상태 보기
openspec status --change "add-user-auth"
```

다중 변경사항 관리는 여러 글을 동시에 쓰는 것과 같습니다. 약간의 기술과 인내심이 필요합니다. 다만 익숙해지면 괜찮습니다. 결국 사람은 적응할 수 있습니다.

### 세션 상태 머신 이해

상태 전환을 이해하면 문제 해결에 도움이 됩니다.

```
Init → Drafting → Openspecing → Reviewing → Executing → ExecutionCompleted → Completed → Archived
```

- **Openspecing**: 계획 생성 중
- **Reviewing**: 검토 중(산출물 반복 수정 가능)
- **Executing**: 실행 중(tasks.md 적용)

상태 머신은 말하자면 일련의 규칙입니다. 규칙은 때때로 귀찮지만 더 자주는 유용합니다. 결국 규칙 없이는 방원이 없다는 말을 옛사람들이 이미 했습니다.

## 요약

OpenSpec 프로세스를 통해 HagiCode 프로젝트는 AI 환각 문제 해결에서 눈에 띄는 성과를 거두었습니다.

1. **환각 감소** - AI는 구조화된 규칙을 따라야 하며, 임의로 코드를 생성할 수 없습니다
2. **품질 향상** - 다층 검증으로 변경사항이 프로젝트 표준을 준수하도록 보장합니다
3. **협업 가속화** - 보관된 변경사항은 후속 개발에 참고 자료를 제공합니다
4. **추적 가능성** - 모든 변경사항은 완전한 제안, 설계, 명세, 작업 기록을 갖습니다

이 솔루션은 AI를 똑똑하게 만드는 것이 아니라, 그것에 "규칙"이라는 채울 씌우는 것입니다. 실천으로 증명했듯이, 발에 족쇄를 차고 춤을 추면 오히려 더 잘 춥니다.

사실 이 이치도 간단합니다. 제약이 반드시 나쁜 것은 아닙니다. 글쓰기와 같습니다. 형식의 제약이 있으면 오히려 좋은 것을 쓰기 쉽습니다. 다만 많은 사람들은 제약을 싫어하고 창의력을 제한한다고 생각하지만, 사실 창의력도 흙이 있어야 꽃을 피우고 열매를 맺을 수 있습니다.

AI 코딩 도우미를 사용하고 있고 비슷한 문제를 겪은 적이 있다면 OpenSpec을 시도해 보세요. 명세 기반 개발은 단계가 더 많아 보일 수 있지만, 이러한 초기 투자는 코드 품질과 유지 보수 효율성에서 수배의 보상을 받게 됩니다.

결국 천천히 가는 것이 때로는 더 빠릅니다. 다만 많은 사람들이 이 이치를 이해하지 못할 뿐입니다......
