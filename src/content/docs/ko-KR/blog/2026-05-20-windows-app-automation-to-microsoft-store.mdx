## Windows 앱 자동 Microsoft Store 등록 자동화 실천

> 버전 파싱부터 Store 배포까지 전 프로세스 자동화를 통해 번거로운 수동 작업을告别하세요.

## 배경

Electron 앱을 Microsoft Store에 등록하려는 경우 다음과 같은 문제에 직면할 수 있습니다. Store는 분리된 설치 프로세스를 지원하지 않습니다. 데스크톱 앱과 서버 페이로드를 하나의 완전한 AppX/MSIX 패키지로 패키징해야 합니다.

이것으로 끝이 아닙니다. 문제는 그 다음입니다. 새 버전을 출시할 때마다 다음을 수행해야 합니다:

1. 데스크톱 및 서버 버전이 업데이트되었는지 확인
2. 해당 태그에서 코드 체크아웃
3. 서버 페이로드 다운로드 및 주입
4. MSIX 패키지 빌드
5. Microsoft Store에 수동 업로드
6. 스토어 정보 및 가격 구성

모든 단계를 수동으로 수행해야 한다면 정말 번거로울 것입니다. 게다가 실수하기 쉽고, 어떤 단계를 수행했고 어떤 단계를 수행하지 않았는지 기억하기 어렵습니다.

사실 이것은 누구의 탓도 아닙니다. 수동 작업은 누락되기 쉽기 때문입니다. 하지만 우리는 매번 이렇게 번거로움을 겪고 싶지 않아서 전체 프로세스를 자동으로 실행하도록 결정했습니다.

## HagiCode 소개

이 문서에서 공유하는 솔루션은 [HagiCode](https://hagicode.com) 프로젝트의 실제 경험에서 비롯되었습니다. AI 코드 어시스턴트로서 HagiCode는 데스크톱 및 웹 클라이언트를 제공하며 다양한 배포 채널을 지원해야 합니다. Windows Store 자동 등록을 구현하는 과정에서 우리는 완전한 자동화 솔루션을 정리했습니다.

말하자면, 이것은 의외의 수확이기도 합니다. 원래는 시간을 조금 절약하기 위함이었는데, 마지막에는 이런 것들이 만들어졌습니다.

## 기술 아키텍처 분석

이 문제는 실제로 여러 기술 레벨의 조율을 포함합니다. 우리는 이것을 5개 레이어로 나눌 수 있습니다:

### 버전 조율 레이어

먼저 새 버전을 출시해야 할 때를 알아야 합니다. Azure 인덱스(빌드 산출물을 저장하는 데 사용하는 blob 저장소)에서 데스크톱 및 서버 컴포넌트의 최신 버전을 파싱한 다음 새 Store 패키지를 생성해야 하는지 판단해야 합니다.

이것은 마치 자신에게 묻는 것과 같습니다: 지금 이 작업을 해야 할까요?

### 작업 공간 관리 레이어

AppX 빌드는 소스 코드 수준의 구성과 런타임 레이아웃에 의존하므로, 준비된 빌드 산출물을 단순히 재패키징할 수 없습니다. 데스크톱 리포지토리의 특정 태그에서 코드를 체크아웃하여 빌드에 사용되는 소스 코드 상태가 올바른지 확인해야 합니다.

결국 소스 코드가 올바르지 않으면 뒤에서 아무리 노력해도 소용이 없습니다.

### 런타임 패키징 레이어

이것은 핵심 부분입니다. 데스크톱 앱의 패키징 레이아웃에 서버 페이로드를 주입해야 합니다. 이렇게 하면 앱이 시작될 때 패키징된 런타임을 감지하고 Steam 모드(오프라인 모드)로 진입합니다.

이 단계를 제대로 하지 않으면 앞의 모든 노력이 헛수고가 됩니다.

### 빌드 출력 레이어

electron-builder를 사용하여 Store 요구 사항을 충족하는 MSIX 패키지를 생성합니다. AppX 빌드에는 Windows SDK가 필요하므로 이 단계는 Windows 환경에서 실행해야 합니다.

어떤 작업은 올바른 장소에서 해야 하고, 장소를 바꾸면 안 됩니다.

### 출시 배포 레이어

마지막으로 GitHub Releases 및 Microsoft Store에 출시합니다. GitHub Release는 백업 및 버전 추적으로, Microsoft Store는 최종 사용자를 대상으로 합니다.

모든 준비가 완료되었습니다. 이제 마지막 단계만 남았습니다.

## 전체 아키텍처 설계

```
┌─────────────────────────────────────────────────────────────┐
│                    package-release workflow                  │
├─────────────────────────────────────────────────────────────┤
│  ┌─────────────────┐    ┌─────────────────┐                │
│  │ resolve_plan    │───▶│   build         │                │
│  │ (버전 파싱 및 건너뛰기)│    │   (MSIX 빌드)    │                │
│  └─────────────────┘    └────────┬────────┘                │
│         │                       │                          │
│         ▼                       ▼                          │
│  ┌─────────────────┐    ┌─────────────────┐                │
│  │ skip_summary    │    │   publish       │                │
│  │ (건너뛰기 보고서)│    │   (출시)         │                │
│  └─────────────────┘    └────────┬────────┘                │
│                                 │                          │
│                                 ▼                          │
│                        ┌─────────────────┐                 │
```

전체 프로세스는 GitHub Actions에 의해 구동되며, 정기 트리거(4시간마다) 또는 수동 트리거가 가능합니다. 수동 트리거 시 특정 데스크톱 버전과 서버 버전을 지정하거나 강제 재빌드를 할 수 있습니다.

사실 4시간에 한 번씩도 자주는 아닙니다. 코드 업데이트는 항상 빠르고 느림이 있지만, 이렇게 하는 것이 더 안전할 뿐입니다.

## 구현 상세 설명

### 1. 빌드 계획 파싱

첫 번째 단계는 새 버전을 빌드해야 하는지 확인하는 것입니다. 데스크톱 및 서버 컴포넌트의 현재 버전을 파싱한 다음 해당 출시가 이미 존재하는지 확인해야 합니다.

```javascript
// scripts/resolve-dispatch-build-plan.mjs
export async function resolveDispatchBuildPlan({
  eventName,
  eventPayload,
  desktopAzureSasUrl,
  serverAzureSasUrl,
}) {
  // 트리거 입력 파싱
  const trigger = normalizeTriggerInputs({ eventName, eventPayload });
  
  // Azure 인덱스에서 데스크톱 및 서버 버전 파싱
  const [desktopRelease, serverRelease] = await Promise.all([
    resolveIndexRelease({ 
      azureSasUrl: desktopAzureSasUrl,
      platformId: 'win-x64' 
    }),
    resolveIndexRelease({ 
      azureSasUrl: serverAzureSasUrl,
      platformId: 'win-x64' 
    })
  ]);
  
  // 출시 태그 생성
  const desktopTag = normalizeGitTag(desktopRelease.version);
  const releaseTag = deriveStoreReleaseTag(desktopRelease.version, serverRelease.version);
  // 예: desktop-v1.2.3-server-v4.5.6
  
  // 이미 존재하는지 확인(중복 빌드 방지)
  const existingRelease = await findReleaseByTag(packerRepository, releaseTag);
  const shouldBuild = !existingRelease || trigger.forceRebuild;
  
  return {
    release: { tag: releaseTag, exists: Boolean(existingRelease) },
    build: { shouldBuild, skipReason },
    upstream: { desktop: { tag: desktopTag }, server }
  };
}
```

이 단계의 핵심은 건너뛰기 로직입니다. 동일한 버전 조합이 이미 빌드된 경우 전체 빌드 프로세스를 다시 실행할 필요가 없습니다. 이렇게 하면 CI 비용과 시간을 절약할 수 있습니다.

같은 작업을 반복하는 것은 아무 의미가 없습니다. 결국 시간과 리소스는 한정되어 있습니다.

### 2. 작업 공간 준비

빌드해야 한다고 결정한 후 깨끗한 빌드 환경을 준비해야 합니다. 우리는 git worktree를 사용하여 특정 태그에서 코드를 체크아웃합니다.

```javascript
// scripts/prepare-packaging-workspace.mjs
export async function preparePackagingWorkspace({
  planPath,
  platformId,
  workspacePath,
  desktopSourcePath
}) {
  // git worktree를 사용하여 특정 태그에서 데스크톱 코드 체크아웃
  await runCommand('git', [
    '-C', resolvedDesktopSourcePath,
    'worktree', 'add', '--detach',
    desktopWorkspace,
    `refs/tags/${plan.upstream.desktop.tag}`
  ]);
  
  // 작업 공간 검증
  const validation = await validateDesktopWorkspace({
    desktopWorkspace,
    storePackageConfig
  });
  
  // 작업 공간 매니페스트 생성
  const workspaceManifest = {
    desktopWorkspace,
    runtimeInjectionRoot: validation.runtimeRoot,
    desktopTag: plan.upstream.desktop.tag,
    desktopRef,
  };
  
  return workspaceManifest;
}
```

worktree를 사용하면 주 작업 디렉토리에 영향을 주지 않고 빌드를 병렬로 실행할 수 있으며, 빌드 완료 후 쉽게 정리할 수 있습니다.

결국 빌드 때문에 주 작업 디렉토리가 엉망이 되는 것은 원하지 않습니다.

### 3. 서버 페이로드 주입

이 단계는 서버 페이로드를 다운로드하여 올바른 위치에 주입합니다.

```javascript
// scripts/stage-server-payload.mjs
export async function stageServerPayload({
  planPath,
  workspacePath,
  platformId,
  azureSasUrl
}) {
  // Azure 인덱스에서 서버 페이로드 다운로드
  const assetSource = resolveAssetDownloadUrl({ asset, sasUrl: azureSasUrl });
  await downloadFromSource({ 
    sourceUrl: assetSource, 
    destinationPath: downloadPath 
  });
  
  // 압축 해제 및 검증
  await extractArchive(downloadPath, extractionPath);
  const runtimeRoot = await resolveRuntimeRoot(extractionPath);
  const validation = await validateServerPayloadRoot(runtimeRoot, platformId);
  
  // 패키징 런타임 레이아웃에 주입
  // 대상 경로는 resources/portable-fixed/current
  // 이 경로는 AppX 내의 extra/portable-fixed/current에 매핑됩니다
  await copyDir(runtimeRoot, targetPath);
}
```

핵심은 경로 매핑입니다. `resources/portable-fixed/current`는 AppX의 `extra/portable-fixed/current`로 패키징되므로 앱 시작 시 로컬 런타임을 감지하고 오프라인 모드로 진입합니다.

경로는 틀리면 안 됩니다. 조금만 어긋나도 찾을 수 없습니다.

### 4. MSIX 빌드

준비된 작업 공간과 서버 페이로드가 있으면 MSIX 패키지를 빌드할 수 있습니다.

```javascript
// scripts/build-appx.mjs
export async function buildAppx({
  planPath,
  workspacePath,
  platformId
}) {
  // Store 특정 electron-builder 구성 오버레이 생성
  const overlayConfig = await writeStoreElectronBuilderConfig({
    desktopWorkspace: workspaceManifest.desktopWorkspace,
    sourceConfigPath: storePackageConfig.desktop.electronBuilderConfigPath,
    outputConfigPath: 'electron-builder.store.yml'
  });
  
  // 데스크톱 빌드 명령 실행
  await runShellCommand(
    buildDesktopStoreCommand(overlayConfig.outputPath, desktopScripts),
    workspaceManifest.desktopWorkspace
  );
  
  // MSIX 출력 수집
  const storeOutputs = await findStoreOutputs(pkgDirectory);
  const artifactPath = path.join(
    workspaceManifest.outputDirectory, 
    artifactFileName
  );
  await copySingleFile(primaryOutput, artifactPath);
}
```

electron-builder 구성에는 패키지 ID, 게시자 표시 이름 등 Store 특정 메타데이터가 포함되어야 합니다. 이 정보는 Store 제출 시 사용됩니다.

구성이 올바르지 않으면 패키지를 만들 수 없습니다. 더 이상 말할 것이 없습니다.

### 5. GitHub에 출시

빌드가 완료되면 GitHub Releases에 산출물을 출시해야 합니다.

```javascript
// scripts/publish-release.mjs
export async function publishRelease({
  planPath,
  artifactsDir,
  outputDir,
  forceDryRun
}) {
  // 출시 산출물 매니페스트 구축
  const publicationArtifacts = await buildPublicationArtifacts({
    plan,
    artifactsDir,
    outputDir
  });
  
  if (!dryRun) {
    // GitHub Release 생성 또는 업데이트
    const releaseResult = await upsertReleaseNotes(
      plan.release.repository,
      plan.release.tag,
      token,
      { name, body }
    );
    
    // 자산 업로드
    for (const upload of publicationArtifacts.uploads) {
      await uploadReleaseAsset({
        release: releaseResult.release,
        filePath: upload.filePath,
        fileName: upload.fileName
      });
    }
  }
}
```

GitHub Release는 버전 추적 및 백업용으로, Store 출시가 실패해도 빌드 산출물을 보존할 수 있습니다.

항상 비상구를 남겨두는 것이 좋습니다. 언젠가 되돌아가야 한다면?

### 6. Microsoft Store에 출시

마지막 단계는 Microsoft Store CLI를 사용하여 스토어에 출시하는 것입니다.

```yaml
# .github/workflows/package-release.yml
publish_store:
  runs-on: windows-latest
  steps:
    - name: Configure Microsoft Store CLI
      uses: microsoft/microsoft-store-apppublisher@v1.2
    
    - name: Publish MSIX packages to Store
      shell: pwsh
      run: |
        msstore reconfigure --tenantId $env:AZURE_AD_TENANT_ID ...
        msstore publish "$($package.FullName)" -id $env:MICROSOFT_STORE_PRODUCT_ID
```

이 단계는 Windows runner에서 실행해야 합니다. Microsoft Store CLI는 Windows 환경이 필요하기 때문입니다.

어떤 작업은 올바른 장소에서 해야 하고, 환경을 바꾸면 안 됩니다.

## 실제 가이드

### 구성 파일

Store 패키지 구성 파일은 패키지 ID 및 빌드 설정을 정의합니다:

```json
{
  "packageIdentity": {
    "displayName": "Hagicode",
    "publisherDisplayName": "newbe36524",
    "publisher": "CN=8B6C8A94-AAE5-4C8B-9202-A29EA42B042F",
    "identityName": "newbe36524.Hagicode",
    "backgroundColor": "transparent",
    "languages": ["en-US", "zh-CN", "zh-Hant"]
  },
  "supportedWindowsTargets": ["win-x64"],
  "desktop": {
    "submodulePath": "inputs/hagicode-desktop",
    "electronBuilderConfigPath": "electron-builder.yml",
    "buildScript": "build:appx",
    "runtimeInjectionPath": "resources/portable-fixed/current"
  }
}
```

구성은 한 번 수정하면 골치 아프므로, 한 번에 올바르게 하는 것이 좋습니다.

### 워크플로우 트리거

정기 트리거 또는 수동 트리거가 가능합니다:

```yaml
on:
  schedule:
    - cron: '0 */4 * * *'  # 4시간마다 실행
  workflow_dispatch:
    inputs:
      desktop_version:
        description: '데스크톱 버전 선택기'
        required: false
      server_version:
        description: '서버 버전 선택기'
        required: false
      force_rebuild:
        description: '이미 존재하더라도 강제 재빌드'
        type: boolean
      dry_run:
        description: '빌드만 하고 출시하지 않음'
        type: boolean
```

자동이든 수동이든, 결국 실행되면 됩니다.

### 주의 사항

1. **Git Tag 요구 사항**: 데스크톱 리포지토리에는 출시 버전에 해당하는 태그(예: v1.2.3)가 포함되어야 합니다. 그렇지 않으면 빌드가 실패합니다

2. **Azure SAS URL**: 인덱스에 액세스하려면 `DESKTOP_AZURE_SAS_URL` 및 `SERVER_AZURE_SAS_URL` 환경 변수를 구성해야 합니다

3. **Microsoft Store 자격 증명**: 다음 키를 구성해야 합니다:
   - `AZURE_AD_APPLICATION_CLIENT_ID`
   - `AZURE_AD_APPLICATION_SECRET`
   - `AZURE_AD_TENANT_ID`
   - `SELLER_ID`
   - `MICROSOFT_STORE_PRODUCT_ID`

4. **런타임 검증**: 서버 페이로드에 필수 런타임 파일이 포함되어야 합니다. 그렇지 않으면 패키징이 실패합니다

5. **Windows Runner**: AppX 빌드 및 Store 출시는 Windows runner에서 실행해야 합니다

6. **중복 검출**: 계획 실행은 동일한 release 태그가 존재하는지 확인하여 불필요한 빌드를 방지합니다

우리는 이러한 문제들을 모두 겪어보았으므로, 알려드립니다. 결국 누구나 반복해서 실수하고 싶지는 않을 것입니다.

## 요약

Windows Store 자동 등록은 복잡해 보이지만 독립적인 단계로 나누면 각 단계는 어렵지 않습니다. 핵심은:
- git worktree로 빌드 환경 관리
- 서버 페이로드를 올바른 경로에 주입
- electron-builder로 MSIX 패키지 생성
- Store CLI로 최종 출시 완료

이 솔루션은 HagiCode 프로젝트에서 몇 달 동안 안정적으로 실행되었으며, 첫 번째 자격 증명 구성 및 설정을 제외하고 기본적으로 "완전 자동화" 목표를 달성했습니다. 후속 버전 출시에는 인간의 개입이 필요하지 않습니다.

사실 많은 일들이 이렇습니다. 어려워 보이지만 해보면 그렇게 어렵지 않습니다. 단지 약간의 인내와 시도가 필요할 뿐입니다.

비슷한 자동화 작업을 하고 있다면 이 문서가 참고가 되기를 바랍니다. 물론 각 프로젝트의 상황은 다르므로 구체적인 요구에 따라 솔루션을 조정해야 할 수 있습니다.

결국 로마로 가는 길은 많습니다. 어떤 길은 조금 더 쉽고, 어떤 길은 조금 더 구불구불할 뿐입니다......
