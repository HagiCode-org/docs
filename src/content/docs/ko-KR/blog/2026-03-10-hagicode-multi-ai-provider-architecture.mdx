## HagiCode 플랫폼의 다중 AI Provider 아키텍처 실천

> 이 글에서는 Orleans Grain 아키텍처 하에서 통일된 IAIProvider 인터페이스를 통해 iflow와 OpenCode 두 가지 AI 도구를 통합하는 기술 솔루션을 공유하며, WebSocket과 HTTP 두 가지 통신 방식의 구현 차이를 상세히 비교합니다.

:::note[시간적 맥락 설명]
이 글은 발행 당시의 provider 아키텍처 실천을 기록합니다. 현재 상주 지원 범위는 [AI Agent CLI 설치](/related-software-installation/ai-agent-cli/) 및 [설치 가이드](/installation)를 기준으로 합니다. `IFlowCli`는 현재 역사적 호환성 값으로만 보존되며, 현재 활성 지원 범위에 속하지 않습니다.
:::

## 배경

사실 별것도 아닙니다. HagiCode를 만들면서 꽤 실제적인 문제에 직면했습니다. 사용자들이 서로 다른 AI 도구를 사용하고 싶어 한다는 것입니다. 이상할 것도 없죠,毕竟每个人都有자기만의 습관이 있습니다. Claude Code를 좋아하는 사람도 있고, GitHub Copilot을 애정하는 사람도 있으며, 어떤 팀은 자체적으로 개발한 도구를 사용하기도 합니다.

처음의 솔루션도 아주 단순무식했습니다. 각 AI 도구에 전용 대응 코드를 작성하는 것이었습니다. 하지만 나중에 문제가 생겼습니다. 코드가 온통 if-else로 가득 찼고, 한 곳을 수정하면 여기저기 테스트해야 했으며, 새 도구를 연결하려면 새로운 로직을 다시 작성해야 했습니다. 생각만 해도 피곤합니다.

나중에 깨달았습니다. 통일된 IAIProvider 인터페이스를 만들어서 모든 AI 제공자의 능력을 추상화하는 것이 낫겠다고요. 그렇게 하면 하위 계층에서 어떤 도구를 사용하든 상위 계층에서는 동일한 호출 방식이 되지 않겠습니까? 얼마나 좋습니까?

최근 프로젝트에 두 개의 새로운 도구를 연결하려고 합니다: iflow와 OpenCode. 둘 다 ACP 프로토콜을 지원하지만 통신 방식이 다릅니다. iflow는 WebSocket을, OpenCode는 HTTP API를 사용합니다. 이것도 일종의 시험이죠. 통일된 인터페이스 아래에서 두 가지 다른 통신 모드를 적응시켜야 합니다. 하지만 생각해 보면 꽤 흥미롭습니다.

## HagiCode에 대하여

이 글에서 공유하는 솔루션은 [HagiCode](https://hagicode.com) 프로젝트에서의 실천 경험에서 비롯되었습니다. HagiCode는 Orleans Grain 아키텍처 기반의 AI 보조 개발 플랫폼으로, 통일된 IAIProvider 인터페이스를 통해 다양한 AI 제공자와 통합하여 사용자가 자신이 좋아하는 AI 도구를 유연하게 선택할 수 있도록 합니다.

## 아키텍처 설계

### 통일된 인터페이스 추상화

먼저, IAIProvider 인터페이스를 정의하여 모든 AI 제공자가 구현해야 하는 능력을 추상화합니다:

```csharp
public interface IAIProvider
{
    string Name { get; }
    bool SupportsStreaming { get; }
    ProviderCapabilities Capabilities { get; }

    Task<AIResponse> ExecuteAsync(AIRequest request, CancellationToken cancellationToken = default);
    IAsyncEnumerable<AIStreamingChunk> StreamAsync(AIRequest request, CancellationToken cancellationToken = default);
    Task<ProviderTestResult> PingAsync(CancellationToken cancellationToken = default);
    IAsyncEnumerable<AIStreamingChunk> SendMessageAsync(AIRequest request, string? embeddedCommandPrompt = null, CancellationToken cancellationToken = default);
}
```

이 인터페이스에는 몇 가지 핵심 메서드가 있습니다:
- ExecuteAsync: 일회성 AI 요청 실행
- StreamAsync: 스트리밍으로 응답을 얻어 실시간 표시 지원
- PingAsync: 상태 점검, provider 사용 가능 여부 확인
- SendMessageAsync: 메시지 전송, 임베디드 명령 지원

### IFlowCliProvider: WebSocket 기반 구현

iflow는 WebSocket을 사용하여 ACP 통신을 하며, 전체 아키텍처는 다음과 같습니다:

```
IFlowCliProvider → ACPSessionManager → WebSocketAcpTransport → iflow CLI
                ↓
         동적 포트 할당 + 프로세스 관리
```

핵심 흐름도 꽤 간단합니다:
1. ACPSessionManager가 ACP 세션 생성 및 관리를 담당
2. WebSocketAcpTransport가 WebSocket 통신 처리
3. 동적 포트를 할당하고 `iflow --experimental-acp --port <port>`로 iflow 프로세스 시작
4. IAIRequestToAcpMapper와 IAcpToAIResponseMapper로 요청/응답 변환

핵심 코드를 살펴보겠습니다:

```csharp
private async IAsyncEnumerable<AIStreamingChunk> StreamCoreAsync(
    AIRequest request,
    string? embeddedCommandPrompt,
    [EnumeratorCancellation] CancellationToken cancellationToken)
{
    // 작업 디렉토리 해석
    var resolvedWorkingDirectory = ResolveWorkingDirectory(request);
    var effectiveRequest = ApplyEmbeddedCommandPrompt(request, embeddedCommandPrompt);

    // ACP 세션 생성
    await using var session = await _sessionManager.CreateSessionAsync(
        Name,
        resolvedWorkingDirectory,
        cancellationToken,
        request.SessionId);

    // 프롬프트 전송
    var prompt = _requestMapper.ToPromptString(effectiveRequest);
    var promptResponse = await session.SendPromptAsync(prompt, cancellationToken);

    // 스트리밍 응답 수신
    await foreach (var notification in session.ReceiveUpdatesAsync(cancellationToken))
    {
        if (_responseMapper.TryConvertToStreamingChunk(notification, out var chunk))
        {
            if (chunk.Type == StreamingChunkType.Metadata && chunk.IsComplete)
            {
                yield return chunk;
                yield break;
            }
            yield return chunk;
        }
    }
}
```

여기에는 몇 가지 설계상의 주의점이 있으며, 약간의 소감도 있습니다:
- `await using`으로 세션이 올바르게 해제되도록 하여 리소스 누수 방지.毕竟 리소스는 사용하지 않으면 자연으로 돌려보내야 합니다
- 스트리밍 응답은 `IAsyncEnumerable`로 반환되어 자연스럽게 비동기 스트림을 지원
- Metadata 유형의 청크에서 완료 여부를 판단하여 응답을 완전히 수신하도록 보장

### OpenCodeCliProvider: HTTP API 기반 구현

OpenCode는 HTTP API 방식으로 서비스를 제공하며, 아키텍처가 약간 다릅니다:

```plaintext
OpenCodeCliProvider → OpenCodeRuntimeManager → OpenCodeClient → OpenCode HTTP API
                      ↓
                OpenCodeProcessManager → opencode 프로세스 관리
```

OpenCode의 특징은 SQLite 데이터베이스로 세션 바인딩 관계를 영속화하여 세션 복구와 프롬프트 응답 복구를 지원하는 것입니다. 꽤 세심한 설계입니다:

```csharp
private async Task<OpenCodePromptExecutionResult> ExecutePromptAsync(
    AIRequest request,
    string? embeddedCommandPrompt,
    CancellationToken cancellationToken)
{
    var prompt = BuildPrompt(request, embeddedCommandPrompt);
    var resolvedWorkingDirectory = ResolveWorkingDirectory(request.WorkingDirectory);
    var client = await _runtimeManager.GetClientAsync(resolvedWorkingDirectory, cancellationToken);
    var bindingSessionId = request.SessionId;
    var boundSession = TryGetBinding(bindingSessionId, resolvedWorkingDirectory);

    // 이미 바인딩된 세션 사용 시도
    if (boundSession is not null)
    {
        try
        {
            return await PromptSessionAsync(
                client,
                boundSession,
                BuildPromptRequest(request, prompt, CreatePromptMessageId()),
                request.Model ?? _settings.Model,
                cancellationToken);
        }
        catch (OpenCodeApiException ex) when (IsStaleBinding(ex))
        {
            // 세션 만료, 바인딩 제거
            RemoveBinding(bindingSessionId);
        }
    }

    // 새 세션 생성
    var session = await client.Session.CreateAsync(new OpenCodeSessionCreateRequest
    {
        Title = BuildSessionTitle(request)
    }, cancellationToken);

    BindSession(bindingSessionId, session.Id, resolvedWorkingDirectory);
    return await PromptSessionAsync(client, session.Id, ...);
}
```

이 구현에는 몇 가지 하이라이트가 있으며, 혹은 몇 가지 흥미로운 점이 있습니다:
- 세션 바인딩 메커니즘: 동일한 SessionId는 OpenCode 세션을 재사용하여 중복 생성 방지, 리소스 낭비 감소
- 만료 처리: 세션 만료를 감지하면 자동으로 바인딩 정리. 옛 것이 가지 않으면 새 것이 오지 않습니다
- 데이터베이스 영속화: SQLite에 바인딩 관계를 저장하여 재부팅 후에도 유효. 어떤 것들은 기억에 남는 것이 기억에 남는 것입니다

### 두 가지 방식의 비교

| 측면 | IFlowCliProvider | OpenCodeCliProvider |
|------|-----------------|---------------------|
| 통신 방식 | WebSocket (ACP) | HTTP API |
| 프로세스 관리 | ACPSessionManager | OpenCodeProcessManager |
| 포트 할당 | 동적 포트 | 포트 없음 (HTTP 사용) |
| 세션 관리 | ACPSession | OpenCodeSession |
| 영속화 | 메모리 캐시 | SQLite 데이터베이스 |
| 시작 명령 | iflow --experimental-acp --port | opencode |
| 지연 | 더 낮음 (장기 연결) | 상대적으로 높음 (HTTP 요청) |

어떤 방식을 선택할지는 주로 요구사항에 따라 다릅니다. WebSocket은 실시간성이 높은 시나리오에 적합하고, HTTP API는 더 간단하고 디버깅하기 쉽습니다. 길을 선택하는 것과 같습니다. 어떤 길은 조금 빠르고, 어떤 길은 조금 걷기 편합니다.

## 실천 가이드

### Provider 설정

먼저 설정 파일에서 두 provider를 활성화합니다:

```yaml
AI:
  Providers:
    IFlowCli:
      Type: "IFlowCli"
      Enabled: true
      ExecutablePath: "iflow"
      Model: null
      WorkingDirectory: null
    OpenCodeCli:
      Type: "OpenCodeCli"
      Enabled: true
      ExecutablePath: "opencode"
      Model: "anthropic/claude-sonnet-4"
      WorkingDirectory: null

OpenCode:
  Enabled: true
  BaseUrl: "http://localhost:38376"
  ExecutablePath: "opencode"
  StartupTimeoutSeconds: 30
  RequestTimeoutSeconds: 120
```

### IFlowCliProvider 사용

```csharp
// Factory를 통해 provider 획득
var provider = await _providerFactory.GetProviderAsync(AIProviderType.IFlowCli);

// AI 요청 실행
var request = new AIRequest
{
    Prompt = "이 함수를 리팩토링해 주세요",
    WorkingDirectory = "/path/to/project",
    Model = "claude-sonnet-4"
};

// 전체 응답 획득
var response = await provider.ExecuteAsync(request, cancellationToken);
Console.WriteLine(response.Content);

// 또는 스트리밍 응답 사용
await foreach (var chunk in provider.StreamAsync(request, cancellationToken))
{
    if (chunk.Type == StreamingChunkType.ContentDelta)
    {
        Console.Write(chunk.Content);
    }
}
```

### OpenCodeCliProvider 사용

```csharp
// Factory를 통해 provider 획득
var provider = await _providerFactory.GetProviderAsync(AIProviderType.OpenCodeCli);

var request = new AIRequest
{
    Prompt = "이 오류를 분석해 주세요",
    WorkingDirectory = "/path/to/project",
    Model = "anthropic/claude-sonnet-4"
};

var response = await provider.ExecuteAsync(request, cancellationToken);
Console.WriteLine(response.Content);
```

### 상태 점검

시작 또는 사용 전에 provider가 사용 가능한지 먼저 확인할 수 있습니다:

```csharp
var iflowResult = await iflowProvider.PingAsync(cancellationToken);
if (!iflowResult.Success)
{
    Console.WriteLine($"IFlow 사용 불가: {iflowResult.ErrorMessage}");
    return;
}

var openCodeResult = await openCodeProvider.PingAsync(cancellationToken);
if (!openCodeResult.Success)
{
    Console.WriteLine($"OpenCode 사용 불가: {openCodeResult.ErrorMessage}");
    return;
}
```

### 임베디드 명령 지원

두 provider 모두 임베디드 명령을 지원합니다. 예를 들어 `/file:xxx`와 같은 명령입니다:

```csharp
var request = new AIRequest
{
    Prompt = "이 파일의 문제를 분석해 주세요",
    SystemMessage = "당신은 코드 분석 전문가입니다"
};

await foreach (var chunk in provider.SendMessageAsync(
    request,
    embeddedCommandPrompt: "/file:src/main.cs",
    cancellationToken))
{
    Console.Write(chunk.Content);
}
```

## 주의사항 및 베스트 프랙티스

### 리소스 관리

IFlow는 WebSocket 장기 연결을 사용하므로 리소스 관리에 특별히 주의해야 합니다:
- `await using`으로 세션이 올바르게 해제되도록 하고, 사용하지 않으면 놓아줍니다
- 취소 작업은 프로세스 정리를 트리거합니다
- ACPSessionManager는 최대 세션 수 제한을 지원합니다

OpenCode의 프로세스 관리는 상대적으로 간단하며, OpenCodeRuntimeManager가 자동으로 처리하여 꽤 편합니다.

### 오류 처리

두 provider 모두 완전한 오류 처리를 갖추고 있습니다:
- IFlow의 오류는 ACP 세션 업데이트를 통해 전파됩니다
- OpenCode의 오류는 OpenCodeApiException으로 발생합니다
- 호출자에서 이러한 예외를 캡처하고 처리하는 것을 권장합니다.毕竟 오류는 항상 발생합니다

### 성능 고려사항

- IFlow의 WebSocket 통신은 HTTP보다 더 낮은 지연을 가집니다
- OpenCode의 세션 재사용은 HTTP 요청 오버헤드를 줄일 수 있습니다
- Factory의 캐싱 메커니즘은 provider의 중복 생성을 방지합니다
- 높은 동시성 시나리오에서는 프로세스 수와 연결 수 제한에 주의해야 합니다

### 설정 검증

시작 시 실행 파일 경로를 검증하지만, 런타임에도 문제가 발생할 수 있습니다. PingAsync는 좋은 도구로, 설정이 올바른지 검증할 수 있습니다:

```csharp
// 시작 시 확인
var provider = await _providerFactory.GetProviderAsync(providerType);
var result = await provider.PingAsync(cancellationToken);
if (!result.Success)
{
    _logger.LogError("Provider {ProviderType} 사용 불가: {Error}", providerType, result.ErrorMessage);
}
```

## 요약

이 글에서는 HagiCode 플랫폼이 iflow와 OpenCode 두 가지 AI 도구를 통합할 때의 기술 솔루션을 공유했습니다. 통일된 IAIProvider 인터페이스를 통해 서로 다른 통신 방식(WebSocket과 HTTP)의 적응을 실현하면서 상위 계층 호출의 일관성을 유지했습니다.

핵심 아이디어는 사실 꽤 간단합니다:
1. 통일된 인터페이스 추상화 정의
2. 다양한 구현에 대해 적응 레이어 생성
3. 팩토리 패턴을 통한 통합 관리

이렇게 하면 확장성이 좋아집니다. 새로운 AI 도구를 연결하려면 IAIProvider 인터페이스만 구현하면 되고, 기존 코드를 많이 변경할 필요가 없습니다. 블록 놀이와 같습니다. 통일된 인터페이스가 있으면 어떻게 조립해도 됩니다.

다중 AI 도구 통합을 하고 계신다면 이 글이 도움이 되기를 바랍니다. 하지만 다시 생각해 보면, 기술이란 사람들에게 도움이 되면 되는 것이고, 다른 것은 너무 신경 쓰지 않아도 됩니다......
