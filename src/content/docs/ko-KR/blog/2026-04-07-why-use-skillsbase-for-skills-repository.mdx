## Skillsbase로 자신의 Skills 컬렉션 저장소를 관리하는 이유

> 말하자면 꽤 웃긴 일입니다. AI 프로그래밍 시대가 도래하여 우리手里的 Agent Skills가 점점 많아지고 있지만,随之而来的麻烦也越来越多입니다. 이 글에서는 우리가 skillsbase를 사용하여这些问题를 어떻게 해결하는지 이야기해보려고 합니다.

## 배경

AI 프로그래밍 시대에 개발자는 점점 더 많은 Agent Skills를 유지 관리해야 합니다. 이는 Claude Code, OpenCode, Cursor 등 코딩 도우미의 기능을 확장하는 재사용 가능한 명령 세트입니다. 그러나 기술 수량이 증가함에 따라 현실적인 문제가 점차 드러나고 있습니다:

사실 큰 문제라고 할 수도 없고, 물건이 많아지면 관리하기 번거로워질 뿐입니다.

### 기술이 분산되어 저장되고 관리 비용이 높음

- 로컬 기술이 여러 위치에 분산되어 있음: `~/.agents/skills/`, `~/.claude/skills/`, `~/.codex/skills/.system/` 등
- 다른 위치에 이름 충돌이 존재할 수 있음(예: `skill-creator`가 사용자 디렉토리와 시스템 디렉토리에 동시에 존재)
- 통합된 관리 진입점이 부족하여 백업 및 마이그레이션이 어려움

이 점이 꽤 성가십니다. 때로는 어떤 기술이 어디에 있는지 모르고, 마치 잃어버린 물건처럼 찾기가 힘듭니다.

### 표준화된 유지 관리 프로세스 부족

- 수동으로 기술을 복사하면 오류가 발생하기 쉽고 출처를 추적하기 어려움
- 통합된 검증 메커니즘이 없어 기술 저장소의 무결성을 보장할 수 없음
- 팀 협업 시 기술 세트를 동기화하고 공유하기 어려움

수동 작업은 항상 실수하기 쉽습니다. 결국 인간의 기억력은 한계가 있어, 누가那么多东西都是从哪来的呢를 기억할 수 있을까요?

### 재현성 요구사항 충족 불가

- 개발 머신을 변경할 때 모든 기술을 다시 구성해야 함
- CI/CD 환경에서 기술 저장소를 자동으로 검증하고 동기화할 수 없음

컴퓨터를 바꾸면 다시 처음부터 해야 하는데, 이感觉은怎么说呢，就像搬家一样麻烦입니다。每次都得重新适应新的环境，重新配置所有东西。

이러한 pain points를 해결하기 위해 우리는 여러 가지 방안을 시도했습니다: 수동 복사에서 스크립트 자동화까지, 직접 디렉토리 관리에서 전역 설치 후 재활용까지. 각 방안에는各自的缺陷이 있어，일관성을 보장할 수 없거나，환경을 오염시키거나，CI에서 사용하기 어렵습니다。

其实也是走了不少弯路입니다。

최종적으로 우리는 더 우아한 솔루션을 찾았습니다——skillsbase입니다. 이 솔루션의 핵심 아이디어는: 먼저 로컬에 설치하여 검증하고，그 다음 구조를 변환하여 저장소에 기록한 후，마지막으로 임시 파일을卸载합니다.这样既能确保仓库内容与实际安装结果一致，又不会污染全局环境。

말하자면 간단하지만, 꽤 많은坑을 밟아서才玩磨出来罢了。

## HagiCode에 대하여

이 글에서 공유하는 솔루션은 [HagiCode](https://hagicode.com) 프로젝트에서의 실천 경험에서 나온 것입니다.

HagiCode는 AI 코드 도우미 프로젝트로, 개발 과정에서 다양한 코딩 능력을 확장하기 위해大量의 Agent Skills를 유지 관리해야 합니다.正是这些实际需求，促使我们开发了 skillsbase 这套工具来规范化管理技能仓库。

其实这东西也不是凭空想出来的，都是被逼的。기술이 많아지면，자연히 관리가 필요합니다。관리的过程中遇到问题，자연히 해결이 필요합니다。一步一步，就走到今天了。

HagiCode에 관심이 있으시다면 [官网了解更多](https://hagicode.com)를 방문하거나 [GitHub](https://github.com/HagiCode-org/site)에서 소스 코드를 확인할 수 있습니다.

## 분석

### 기술적 도전

유지 관리 가능한 기술 컬렉션 저장소를 구축하려면 다음 핵심 문제를 해결해야 합니다:

1. **통합된命名空间冲突**：여러 소스에同名 기술이 있을 때，어떻게 덮어쓰기를 방지합니까?
2. **출처 추적 가능성**：각 기술의 출처를 어떻게 기록하여，이후 업데이트와 감사를 용이하게 합니까?
3. **동기화 및 검증**：저장소 내용이 실제 설치 결과와 일치하는지 어떻게 보장합니까?
4. **자동화 통합**：CI/CD 프로세스와 통합하여，자동 동기화 및 검증을 어떻게 구현합니까?

이 문제들은 간단해 보이지만，每一个都够头疼的입니다。不过话说回来，做什么事不难呢？

### 설계 trade-off

**방안 1: 직접 디렉토리 복사**

장점: 구현이 간단함
단점: `skills` CLI 실제 설치 결과와 일치함을 보장할 수 없음

이 방안은，说真的，우리도 생각했습니다。只是后来发现，CLI 설치할 때 일부 전처리 로직이 있을 수 있고，직접 복사하면 건너뜁니다。结果是，복사한 것과 실제 설치한 것이 다르고，这就有问题了。

**방안 2: 전역 설치 후 재활용**

장점: 설치 과정을 검증할 수 있음
단점: 실행 환경을 오염시키고，CI와 로컬 결과를 일관되게 유지하기 어려움

이 방안은 더 나쁩니다。전역 설치는，환경을 오염시킵니다。更麻烦的是，CI 환경과 로컬 환경을 일관되게 유지하기 어려워，"本地能跑，CI 失败" 문제를 초래합니다。这种感觉，谁懂啊？

**방안 3: 로컬 설치 → 변환 →卸载（최종 방안）**

이것은 skillsbase가 채택한 방안입니다:

- 먼저 `npx skills`를 통해 기술을 임시 위치에 설치
- 디렉토리 구조를 변환하고 출처 메타데이터 추가
- 대상 저장소에 기록
- 마지막으로 임시 파일卸载

이 방안은 저장소 내용이 실제 소비자 설치 결과와 일치함을 보장하면서，전역 환경을 오염시키지 않고，변환 과정은 표준화할 수 있으며，멱등 연산을 지원합니다。

其实这个方案也不是一开始就想到的，只是试错试多了，自然就知道什么可行、什么不可行了。

### 아키텍처 결정

| 결정 항목 | 선택 | 이유 |
|--------|------|------|
| 런타임 | Node.js ESM | 빌드 단계 불필요，`.mjs`로 파일 시스템 오케스트레이션 충분 |
| 구성 형식 | YAML (`sources.yaml`) | 가독성이 높고，인력 유지 보수 지원 |
| 명명 전략 | 네임스페이스 접두사 | 사용자 기술은原名 유지，시스템 기술은 `system-` 접두사 추가 |
| 워크플로 | `add`修改清单 → `sync`执行同步 | 단일 동기화 엔진，규칙 이중 구현 방지 |
| 파일 관리 | 관리 파일 식별 | 주석 헤더 추가，안전한 덮어쓰기 지원 |

이러한 결정은，说到底都是为了一个目标：让事情变得简单입니다。毕竟，简单才是王道。

## 해결

### CLI 아키텍처

`skillsbase` CLI는 4개의 핵심 명령을 제공합니다:

```text
skillsbase
├── init          # 저장소 구조 초기화
├── sync          # 기술 내용 동기화
├── add           # 새 기술 추가
└── github_action # GitHub Actions 구성 생성
```

명령은 많지 않지만，그래도 충분합니다。毕竟，工具这东西，够用就好。

### 핵심 워크플로

```
┌─────────────┐    ┌─────────────┐    ┌─────────────┐    ┌─────────────┐
│    init     │───▶│    add      │───▶│    sync     │───▶│github_action│
│  저장소 초기화 │    │  소스 추가   │    │  내용 동기화  │    │  CI 생성    │
└─────────────┘    └─────────────┘    └─────────────┘    └─────────────┘
```

一步一步来，急不得。

### 동기화 프로세스 설계

```
sources.yaml → 解析来源 → npx skills 설치 → 구조 변환 → skills/에 기록 → 임시 파일卸载
                              ↓
                        .skill-source.json (출처 메타데이터)
```

이 프로세스 설계는还算清晰입니다。至少我自己看的时候，能明白每一步在做什么。

### 저장소 구조

```
repos/skillsbase/
├── sources.yaml              # 소스清单（단일 진실 소스）
├── skills/                   # 기술 디렉토리
│   ├── frontend-design/      # 사용자 기술
│   ├── skill-creator/        # 사용자 기술
│   └── system-skill-creator/ # 시스템 기술（접두사 포함）
├── scripts/
│   ├── sync-skills.mjs       # 동기화 스크립트
│   └── validate-skills.mjs   # 검증 스크립트
├── docs/
│   └── maintainer-workflow.md # 유지 관리자 문서
└── .github/
    ├── workflows/
    │   └── skills-sync.yml   # CI 워크플로
    └── actions/
        └── skillsbase-sync/
            └── action.yml     # 재사용 가능한 Action
```

파일이 좀 많지만，그래도 괜찮습니다。毕竟，조직 구조가清晰해지면，유지 관리也方便。

## 실천

### 기술 저장소 초기화

```bash
# 1. 빈 저장소 생성
mkdir repos/myskills && cd repos/myskills
git init

# 2. skillsbase로 초기화
npx skillsbase init

# 출력:
# [1/4] create manifest ................. done
# [2/4] create scripts .................. done
# [3/4] create docs ..................... done
# [4/4] create github workflow .......... done
#
# next: skillsbase add <skill-name>
```

이 단계에서一堆文件이 생성되지만，걱정하지 마세요，모두 자동으로 생성된 것입니다。다음으로 기술 추가를 시작할 수 있습니다.

### 기술 추가

```bash
# 단일 기술 추가（자동으로 동기화 실행）
npx skillsbase add frontend-design --source vercel-labs/agent-skills

# 로컬 소스에 추가
npx skillsbase add documentation-writer --source /home/user/.agents/skills

# 출력:
# source: first-party ......... updated
# target: skills/frontend-design ... synced
# status: 1 skill added, 0 removed
```

기술 추가는挺简单的，한 명령으로 충분합니다。只是때로는一些意外情况을 만날 수 있고，예를 들어 네트워크가 안 좋거나，권한 문제之类的입니다。不过这些都是小事，慢慢来。

### 기술 동기화

```bash
# 동기화 실행（모든 소스 대조）
npx skillsbase sync

# 드리프트 여부만 확인（파일 수정 안 함）
npx skillsbase sync --check

# 누락된 소스 허용（CI 시나리오）
npx skillsbase sync --allow-missing-sources
```

동기화할 때，시스템은 `sources.yaml`에 정의된 소스를 모두 한 번씩 검사하고，`skills/` 디렉토리의 내용과 대조합니다。차이가 있으면 업데이트하고，없으면 건너뜁니다。这样就不会出现"配置改了但文件没变"的问题了。

### CI 구성 생성

```bash
# workflow 생성
npx skillsbase github_action --kind workflow

# action 생성
npx skillsbase github_action --kind action

# 전체 생성
npx skillsbase github_action --kind all
```

CI 구성也자동으로 생성됩니다。只是일부 세부 사항을직접 조정해야 하고，예를 들어 트리거 조건，실행 환경之类的입니다。不过这些都不难。

### sources.yaml 구성 예시

```yaml
# 기술 루트 디렉토리 구성
skillsRoot: skills/
metadataFile: .skill-source.json

# 소스 정의
sources:
  # 제1자：로컬 사용자 기술
  first-party:
    type: local
    path: /home/user/.agents/skills
    naming: original  #原名 유지
    includes:
      - documentation-writer
      - frontend-design
      - skill-creator

  # 시스템：시스템 제공 기술
  system:
    type: local
    path: /home/user/.codex/skills/.system
    naming: prefix-system  # system- 접두사 추가
    includes:
      - imagegen
      - openai-docs
      - skill-creator  # system-skill-creator이 됨

  # 원격：제3자 저장소
  vercel:
    type: remote
    url: vercel-labs/agent-skills
    naming: original
    includes:
      - web-design-guidelines
```

이 구성 파일은 전체 시스템의 핵심입니다。모든 소스가 여기에 정의되어 있고，여기를 수정하면，다음 동기화时生效합니다。所以说，这算是一个"단일 진실 소스"。

### .skill-source.json 메타데이터 예시

```json
{
  "source": "first-party",
  "originalPath": "/home/user/.agents/skills/documentation-writer",
  "originalName": "documentation-writer",
  "targetName": "documentation-writer",
  "syncedAt": "2026-04-07T00:00:00.000Z",
  "version": "unknown"
}
```

각 기술 디렉토리에는 이 파일이 있어，그 출처 정보를 기록합니다。这样以后出问题的时候，能快速定位是从哪来的、什么时候同步的。

### 안전 및 검증

```bash
# 저장소 구조 검증
node scripts/validate-skills.mjs

# skills CLI로 검증
npx skills add . --list

# 업데이트 확인
npx skills check
```

검증这东西，说重要也重要，说没必要也没必要입니다。不过为了保险起见，偶尔跑一下也无妨입니다。毕竟，谁知道会不会有什么意外呢？

### GitHub Actions 통합

```yaml
# .github/workflows/skills-sync.yml
name: Skills Sync

on:
  push:
    paths:
      - 'sources.yaml'
      - 'skills/**'
  workflow_dispatch:

jobs:
  validate:
    runs-on: ubuntu-latest
    steps:
      - uses: actions/checkout@v4
      - uses: actions/setup-node@v4
        with:
          node-version: '20'
      - name: Validate repository
        run: |
          npx skills add . --list
          node scripts/validate-skills.mjs
      - name: Sync check
        run: npx skillsbase sync --check
```

CI 통합后，每次 `sources.yaml` 또는 `skills/` 디렉토리를 수정하면，자동으로 검증이 트리거됩니다。这样就不会出现"本地改了忘了同步"的问题了。

### 모범 사례

1. **이름 충돌 처리**：시스템 기술은 통일하여 `system-` 접두사를 추가합니다。这样既能保留所有技能，又能避免命名冲突。
2. **멱등 연산**：모든 명령은 반복 실행을 지원하며，여러 번 `sync`를 실행해도 부작용이 없습니다。这点在 CI 里特别重要。
3. **관리 파일**：생성된 파일은 `# Managed by skillsbase CLI` 주석을 포함하며，식별 및 관리에 편리합니다。这些文件可以安全覆盖，手工修改不会被保留。
4. **비대화형 모드**：CI 환경은 기본적으로 결정론적 동작을 사용하며，대화형 프롬프트로 인해 중단되지 않습니다。所有配置都通过 `sources.yaml` 声明。
5. **출처 추적 가능**：각 기술에는 `.skill-source.json`이 있어 출처 정보를 기록하며，出问题的时候能快速定位。

### 팀 협업

```bash
# 팀원이 공유 기술 라이브러리 설치
npx skills add your-org/myskills -g --all

# 로컬 클론 검증
git clone https://github.com/your-org/myskills.git
cd myskills
npx skills add . --list
```

Git으로 기술 저장소를 관리함으로써，팀원은 기술 세트를 쉽게 동기화할 수 있으며，모두가 동일한 버전의 도구와 구성을 사용하는지 확인할 수 있습니다.

这点在团队协作里特别有用，不会出现"我这边能跑你那边不行"的情况입니다。毕竟，환경이 통일되면，문제가 절반 줄어듭니다。

## 요약

`skillsbase`로 기술 컬렉션 저장소를 관리하는 핵심 가치는 다음과 같습니다:

- **안전성**：출처 검증，충돌 감지，관리 파일 보호
- **유지 관리성**：통합 진입점，멱등 연산，구성即문서
- **표준화**：통합된 디렉토리 구조，명명 규범，메타데이터 형식
- **자동화**：CI/CD 통합，자동 동기화，자동 검증

이 솔루션을 통해 개발자는 npm 패키지를 관리하듯 자신의 Agent Skills를 관리할 수 있으며，재현 가능，공유 가능，유지 관리 가능한 기술 저장소 시스템을 실현할 수 있습니다.
