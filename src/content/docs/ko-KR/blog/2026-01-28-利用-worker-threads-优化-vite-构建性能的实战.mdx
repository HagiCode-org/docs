## 120초에서 45초로: Worker Threads로 Vite 빌드 성능 최적화 실전

> 대규모 프론트엔드 프로젝트를 처리할 때, 프로덕션 환경의 코드 빌드는 종종 참을 수 없을 정도로 오래 걸립니다. 이 글에서는 Node.js Worker Threads를 사용하여 Vite 빌드의 코드 난독화 단계 소요 시간을 120초에서 45초로 줄이는 방법을 공유하고, HagiCode 프로젝트의 구현 세부 사항과 시행착오 경험을 자세히 소개합니다.

## 배경

프론트엔드 엔지니어링 실무에서 프로젝트 규모가 확대됨에 따라 빌드 효율성 문제가 점차 두드러졌습니다. 특히 프로덕션 환경 빌드 프로세스에서 소스 코드 로직을 보호하기 위해 JavaScript 난독화 도구(예: `javascript-obfuscator`)를 도입하는 경우가 많습니다. 이 단계는 필수적이지만 계산량이 매우 크고 CPU 리소스를 극도로 소모합니다.

**HagiCode** 프로젝트의 초기 개발 단계에서 우리는 매우 까다로운 성능 병목 현상에 직면했습니다: 프로덕션 빌드 시간이 코드량 증가에 따라 급격히 악화되었습니다.

**구체적인 문제점은 다음과 같습니다**:
- 단일 스레드 직렬 실행으로 난독화 작업 처리, CPU 단일 코어만 풀 가동되고 다른 코어는 유휴 상태
- 빌드 시간이 초기 30초에서 110-120초로 급증
- 코드 수정 후 빌드 검증 프로세스가 매우 길어져 개발 반복 효율이 크게 저하됨
- CI/CD 파이프라인에서 빌드 단계가 가장 시간이 오래 걸리는 부분으로 됨

**HagiCode에 이런 요구가 생긴 이유는 무엇인가요?**
HagiCode는 AI 기반 코드 지능 어시스턴트로, 복잡한 비즈니스 로직과 AI 상호작용 모듈을 포함한 프론트엔드 아키텍처를 가지고 있습니다. 핵심 코드의 보안을 보장하기 위해 프로덕션 배포 시 고강도 난독화를 강제로 적용합니다. 2분에 달하는 빌드 대기 시간에 직면하여 우리는 빌드 시스템에 대한 심도 있는 성능 최적화를 결심했습니다.

## 분석: 성능 병목 지점의 돌파구 찾기

성능 문제를 해결하기 전에 먼저 개념을 정리하고 최적의 기술 솔루션을 결정해야 합니다.

### 핵심 결정: Worker Threads를 선택한 이유는?

Node.js 환경에서 병렬 계산을 구현하는 주요 세 가지 솔루션이 있습니다:
1. **child_process**: 독립적인 자식 프로세스 생성
2. **Web Workers**: 주로 브라우저 측용
3. **worker_threads**: Node.js 네이티브 멀티스레드 지원

비교 분석 결과 HagiCode는 **Worker Threads**를 선택했습니다. 이유는 다음과 같습니다:
- **제로 직렬화 오버헤드**: Worker Threads는 동일한 프로세스에 있어 `SharedArrayBuffer` 또는 제어권 이전 방식으로 메모리를 공유할 수 있어 프로세스 간 통신의 큰 직렬화 비용을 피할 수 있습니다.
- **네이티브 지원**: Node.js 12+ 버전에 내장되어 있어 추가적인 무거운 의존성 도입이 필요 없습니다.
- **통합된 컨텍스트**: 자식 프로세스보다 디버깅과 로깅이 더 편리합니다.

### 작업 단위: 난독화 작업을 어떻게 분할할까?

거대한 JS Bundle 파일을 난독화하는 것은 병렬 처리가 어렵습니다(코드에 의존 관계가 있기 때문). 하지만 Vite의 빌드 산출물은 여러 **Chunk**로 구성되어 있습니다. 이것이 우리에게 자연스러운 병렬 경계를 제공합니다:

- **독립성**: Vite로 패키징된 다른 Chunk 간의 의존 관계가 이미 해결되어 안전하게 병렬 처리할 수 있습니다.
- **적절한 단위**: 일반적으로 프로젝트는 10-30개의 Chunk를 가지며, 이 수준은 병렬 스케줄링에 매우 적합합니다.
- **통합 용이성**: Vite 플러그인의 `generateBundle` 훅을 사용하여 파일 생성 전에 이러한 Chunk를 가로채서 처리할 수 있습니다.

### 아키텍처 설계

우리는 네 가지 핵심 구성 요소를 포함하는 병렬 처리 시스템을 설계했습니다:

1. **Task Splitter**: Vite의 번들 객체를 순회하며 난독화가 필요 없는 파일(예: vendor)을 필터링하여 작업 대기열을 생성합니다.
2. **Worker Pool Manager**: Worker의 생명주기를 관리하며 작업 분배, 회수, 오류 재시도를 담당합니다.
3. **Progress Reporter**: 빌드 진행 상황을 실시간으로 출력하여 사용자의 대기 불안을 해소합니다.
4. **ObfuscationWorker**: 실제 난독화 로직을 실행하는 작업 스레드입니다.

## 해결: 실전 코딩 및 구현

위의 분석을 바탕으로 이 병렬 난독화 시스템을 구현하기 시작했습니다.

### 1. Vite 플러그인 구성

먼저 `vite.config.ts`에 병렬 난독화 플러그인을 통합했습니다. 구성은 매우 직관적이며 Worker 수와 난독화 규칙만 지정하면 됩니다.

```typescript
import { defineConfig } from 'vite'
import { parallelJavascriptObfuscator } from './buildTools/plugin'

export default defineConfig(({ mode }) => {
  const isProduction = mode === 'production'
  
  return {
    build: {
      rollupOptions: {
        ...(isProduction
          ? {
              plugins: [
                parallelJavascriptObfuscator({
                  enabled: true,
                  // CPU 코어 수에 따라 자동 조정, 메인 스레드를 위해 하나의 코어를 비워두는 것을 권장
                  workerCount: 4, 
                  retryAttempts: 3,
                  fallbackToMainThread: true, // 오류 시 자동으로 단일 스레드로 강등
                  // vendor chunk 필터링, 일반적으로 서드파티 라이브러리는 난독화할 필요 없음
                  isVendorChunk: (fileName: string) => fileName.includes('vendor-'),
                  obfuscationConfig: {
                    compact: true,
                    controlFlowFlattening: true,
                    deadCodeInjection: true,
                    disableConsoleOutput: true,
                    // ... 더 많은 난독화 옵션
                  },
                }),
              ],
            }
          : {}),
      },
    },
  }
})
```

### 2. Worker 로직 구현

Worker는 작업을 실행하는 단위입니다. 입력과 출력의 데이터 구조를 잘 정의해야 합니다.

**주의**: 여기의 코드는 간단하지만 몇 가지 함정에 주의해야 합니다. 예를 들어 `parentPort`의 null 검사와 오류 처리가 있습니다. HagiCode 실무에서 우리는 일부 특수한 ES6 문법이 난독화기 충돌을 일으킬 수 있어 `try-catch` 보호를 추가했습니다.

```typescript
import { parentPort } from 'worker_threads'
import javascriptObfuscator from 'javascript-obfuscator'

export interface ObfuscationTask {
  chunkId: string
  code: string
  config: any
}

export interface ObfuscationResult {
  chunkId: string
  obfuscatedCode: string
  error?: string
}

// 메인 스레드에서 보낸 작업 수신
if (parentPort) {
  parentPort.on('message', async (task: ObfuscationTask) => {
    try {
      // 난독화 실행
      const obfuscated = javascriptObfuscator.obfuscate(task.code, task.config)
      const result: ObfuscationResult = {
        chunkId: task.chunkId,
        obfuscatedCode: obfuscated.getObfuscatedCode(),
      }
      // 결과를 메인 스레드로 전송
      parentPort?.postMessage(result)
    } catch (error) {
      // 예외 처리, 단일 Worker 충돌이 전체 빌드를 차단하지 않도록 보장
      const result: ObfuscationResult = {
        chunkId: task.chunkId,
        obfuscatedCode: '',
        error: error instanceof Error ? error.message : 'Unknown error',
      }
      parentPort?.postMessage(result)
    }
  })
}
```

### 3. Worker 풀 관리자

이 솔루션의 핵심입니다. 고정 크기의 Worker 풀을 유지하고 **FIFO(선입선출)** 전략으로 작업을 스케줄링해야 합니다.

```typescript
import { Worker } from 'worker_threads'
import os from 'os'

export class WorkerPool {
  private workers: Worker[] = []
  private taskQueue: Array<{
    task: ObfuscationTask
    resolve: (result: ObfuscationResult) => void
    reject: (error: Error) => void
  }> = []
  
  constructor(options: WorkerPoolOptions = {}) {
    // 기본값은 코어 수 - 1, 메인 스레드를 위해 약간의 여유 공간을 남겨둠
    const workerCount = options.workerCount ?? Math.max(1, (os.cpus().length || 4) - 1)
    
    for (let i = 0; i < workerCount; i++) {
      this.createWorker()
    }
  }

  private createWorker() {
    const worker = new Worker('./worker.ts')
    
    worker.on('message', (result) => {
      // 작업 완료 후 대기열에서 다음 작업 가져오기
      const nextTask = this.taskQueue.shift()
      if (nextTask) {
        this.dispatchTask(worker, nextTask)
      } else {
        // 처리할 작업이 없으면 Worker를 유휴 상태로 표시
        this.activeWorkers.delete(worker)
      }
    })
    
    this.workers.push(worker)
  }

  // 풀에 작업 제출
  public runTask(task: ObfuscationTask): Promise<ObfuscationResult> {
    return new Promise((resolve, reject) => {
      const job = { task, resolve, reject }
      const idleWorker = this.workers.find(w => !this.activeWorkers.has(w))
      
      if (idleWorker) {
        this.dispatchTask(idleWorker, job)
      } else {
        this.taskQueue.push(job)
      }
    })
  }

  private dispatchTask(worker: Worker, job: any) {
    this.activeWorkers.set(worker, job.task)
    worker.postMessage(job.task)
  }
}
```

### 4. 진행 상황 보고

기다림은 고통스럽습니다, 특히 얼마나 더 기다려야 할지 모를 때 더욱 그렇습니다. 우리는 간단한 진행 상황 보고기를 추가하여 현재 상태를 실시간으로 피드백합니다.

```typescript
export class ProgressReporter {
  private completed = 0
  private readonly total: number
  private readonly startTime: number

  constructor(total: number) {
    this.total = total
    this.startTime = Date.now()
  }

  increment(): void {
    this.completed++
    this.report()
  }

  private report(): void {
    const now = Date.now()
    const elapsed = now - this.startTime
    const percentage = (this.completed / this.total) * 100
    
    // 간단한 ETA 추정
    const avgTimePerChunk = elapsed / this.completed
    const remaining = (this.total - this.completed) * avgTimePerChunk

    console.log(
      `[Parallel Obfuscation] ${this.completed}/${this.total} chunks completed (${percentage.toFixed(1)}%) | ETA: ${(remaining / 1000).toFixed(1)}s`
    )
  }
}
```

## 실천: 효과 및 시행착오

이 솔루션을 배포한 후 HagiCode 프로젝트의 빌드 성능은 즉각적인 향상을 보았습니다.

### 성능 벤치마크 데이터

우리는 다음 환경에서 테스트를 진행했습니다:
- CPU: Intel Core i7-12700K (12 cores / 20 threads)
- RAM: 32GB DDR4
- Node.js: v18.17.0
- OS: Ubuntu 22.04

**결과 비교**:
- **단일 스레드(최적화 전)**: 118초
- **4 Workers**: 55초 (**53%** 향상)
- **8 Workers**: 48초 (**60%** 향상)
- **12 Workers**: 45초 (**62%** 향상)

수익은 선형적이지 않다는 것을 알 수 있습니다. Worker 수가 8개를 초과하면 향상 폭이 줄어듭니다. 이는 주로 작업 분배의 균일성과 메모리 대역폭 병목에 의해 제한됩니다.

### 일반적인 문제 및 해결 방법

HagiCode 실제 사용 중 일부 함정에도 직면했는데, 여기서 공유합니다:

**Q1: 빌드 시간이 명확히 줄어들지 않고 오히려 느려진 경우?**
- **원인**: Worker 생성 자체에 오버헤드가 있거나 Worker 수가 너무 많아 설정되어 컨텍스트 전환이 빈번합니다.
- **해결**: Worker 수를 `CPU 코어 수 - 1`로 설정하는 것을 권장합니다. 동시에 단일 Chunk가 특히 큰지(예: > 5MB) 확인해보세요. 이러한 "거대" 파일은 병목이 될 수 있으며 코드 분할 전략을 최적화하는 것이 좋습니다.

**Q2: 간혹 Worker 충돌이 발생하여 빌드 실패?**
- **원인**: 특정 코드 문법이 난독화기 내부 오류를 일으킬 수 있습니다.
- **해결**: 우리는 **자동 강등 메커니즘**을 구현했습니다. Worker가 연속 실패 횟수가 임계값에 도달하면 플러그인이 자동으로 단일 스레드 모드로 돌아가 빌드가 중단되지 않도록 합니다. 동시에 오류 파일 이름을 기록하여 추후 대응 수정이 용이하도록 합니다.

**Q3: 메모리 사용량이 너무 높음(OOM)?**
- **원인**: 각 Worker는 난독화기를 로드하고 AST를 구문 분석하기 위해 독립적인 메모리 공간이 필요합니다.
- **해결**:
  - Worker 수를 줄이세요.
  - Node.js 메모리 제한을 늘리세요: `NODE_OPTIONS="--max-old-space-size=4096" npm run build`.
  - Worker 내부에 불필요한 대형 객체 참조를 유지하지 않도록 하세요.

## 요약

Node.js Worker Threads를 도입하여 우리는 HagiCode 프로젝트의 프로덕션 빌드 시간을 120초에서 약 45초로 성공적으로 줄였으며, 개발 경험과 CI/CD 효율성을 크게 향상했습니다.

이 솔루션의 핵심은 다음과 같습니다:
1. **합리적인 작업 분할**: Vite의 Chunk를 병렬 단위로 활용합니다.
2. **리소스 제어**: Worker 풀을 사용하여 리소스 고갈을 방지합니다.
3. **내결함성 설계**: 자동 강등 메커니즘으로 빌드 안정성을 보장합니다.
