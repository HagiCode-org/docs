## AI Compose Commit: AI를 활용한 Git 커밋 워크플로우 지능형 리팩토링

## Introduction

소프트웨어 개발 과정에서 코드 커밋은 개발자가 매일 마주하는 일상적인 작업입니다. 하지만 다음과 같은 상황을 경험해보셨나요? 하루 일과가 끝난 후 Git을 열어서 수십 개의 스테이지되지 않은 수정 파일을 보았지만, 어떻게 논리적인 커밋으로组织해야 할지 모르는 상황에요.

전통적인 방식은 파일을 수동으로 일괄 스테이징하고, 개별적으로 커밋하고, 커밋 메시지를 작성하는 것인데, 이 과정은 시간이 오래 걸리고 오류가 발생하기 쉽습니다. 우리는 종종 이 일에 상당한 시간을 낭비했는데, 아무도 이미 피로한 밤에 이런 사소한 일로 스트레스받고 싶지 않기 때문입니다.

우리는 HagiCode 프로젝트에서 AI Compose Commit이라는 새로운 기능을 출시했습니다. 이 기능은 이 워크플로우를 완전히 혁신하는 것을 목표로 합니다. AI를 통해 작업 공간의 모든 커밋되지 않은 변경 사항을 지능적으로 분석하고, 자동으로 여러 논리적 커밋으로 그룹화하며, 규칙을 준수하는 커밋 작업을 실행합니다. 이 문서에서는 이 기능의 구현 원리, 기술 아키텍처, 그리고 실무에서 겪은 과제와 해결 방안에 대해 심층적으로 다룹니다.

## About HagiCode

이 문서에서 공유하는 방안은 [HagiCode](https://hagicode.com) 프로젝트에서의 실무 경험에서 나왔습니다.

## Background

### 전통적 Git 커밋의 문제점

Git은 버전 관리 시스템으로서 개발자에게 강력한 코드 관리 능력을 제공합니다. 하지만 실제 사용에서 커밋 작업은 종종 개발 프로세스의 병목이 됩니다:

1. **수동 그룹화 시간 소요**: 대량의 파일 변경이 있을 때 개발자는 파일 내용을 개별적으로 확인하고 어떤 것이 동일한 기능에 속하는지 판단해야 하며, 이는 상당한 인지적 노력이 필요합니다.
2. **커밋 메시지 품질 저하**: Conventional Commits 규칙을 준수하는 커밋 메시지를 작성하려면 경험과 기술이 필요하며, 초보자들은 종종 비표준 커밋을 작성합니다.
3. **다중 저장소 관리 복잡**: monorepo 환경에서는 여러 저장소 간에 전환해야 하므로 작업 복잡도가 증가합니다.
4. **워크플로우 중단**: 코드 커밋은 개발 사고를 방해하고 코딩 효율에 영향을 미칩니다.

이러한 문제는 대형 프로젝트와 팀 협업 환경에서 특히 두드러집니다. 우수한 개발 도구는 개발자가 핵심 코딩 작업에 집중할 수 있게 해야 하며, 번거로운 커밋 프로세스에 방해받지 않아야 합니다.

### AI 보조 개발의 동향

최근 AI 기술은 소프트웨어 개발 분야에서 점점 더 널리 적용되고 있습니다. 코드 완성, 오류 감지, 자동 문서 생성에서 AI는 개발의各个环节에 점진적으로 침투하고 있습니다. Git 워크플로우 측면에서는 일부 도구가 커밋 메시지 생성 기능을 제공하지만, 대부분 단일 커밋 시나리오에 국한되어 전체 작업 공간 변경에 대한 지능적 분석과 그룹화 능력이 부족합니다.

사실 HagiCode도 개발 과정에서 이러한 문제점을 겪었고, 여러 도구를 시도했지만 어느 정도 제한이 있었습니다. 기능이 충분하지 않거나 사용자 경험이 충분하지 않았습니다. 이것이 우리가 결국 직접 AI Compose Commit 기능을 구현하기로 결정한 이유입니다.

HagiCode의 AI Compose Commit 기능은 바로 이러한 공백을 메우기 위해 탄생했습니다. 단순히 커밋 메시지를 생성하는 것이 아니라, 파일 분석부터 커밋 실행까지 전체 프로세스를 완전히 담당합니다.

## Problem

### 기술적 과제

AI Compose Commit 기능을 구현하는 과정에서 우리는 여러 기술적 과제에 직면했습니다:

1. **파일 의미 이해**: AI는 파일 변경의 의미적 관계를 이해하고 어떤 파일이 동일한 기능 모듈에 속하는지 판단해야 합니다. 이는 파일 내용, 디렉터리 구조 및 변경의 맥락을 심층적으로 분석해야 합니다.

2. **커밋 그룹화 전략**: 합리적인 그룹화 기준을 어떻게 정의할까요? 기능별, 모듈별, 아니면 파일 유형별로? 다른 프로젝트는 다른 전략이 적용될 수 있습니다.

3. **실시간 피드백과 비동기 처리**: Git 작업은 특히 대량의 파일을 처리할 때 상당한 시간이 걸릴 수 있습니다. 사용자 경험을 보장하면서 복잡한 작업을 어떻게 완료할까요?

4. **다중 저장소 지원**: monorepo 아키텍처에서 메인 저장소와 하위 저장소 간에 작업을 올바르게 라우팅해야 합니다.

5. **오류 처리 및 롤백**: 특정 커밋이 실패하면 이미 실행된 커밋을 어떻게 처리할까요? 이미 스테이징된 파일을 롤백해야 할까요?

6. **커밋 메시지 일관성**: 생성된 커밋 메시지는 프로젝트의 기존 스타일을 준수하고 과거 커밋의 형식 일관성을 유지해야 합니다.

### 성능 고려사항

AI가 대량의 파일 변경을 처리하면 상당한 시간과 컴퓨팅 리소스를 소비합니다. 우리는 다음 측면에서 최적화해야 했습니다:

- 불필요한 AI 호출 줄이기
- 파일 컨텍스트 구성 방식 최적화
- 효율적인 Git 작업 일괄 처리 구현

이러한 문제는 HagiCode의 실제 사용에서 실제로 발생했으며, 우리는 지속적인 반복과 최적화를 통해 상대적으로 완벽한 해결책을 찾았습니다. 유사한 도구를 개발 중이라면 우리의 경험이 영감을 줄 수 있기를 바랍니다.

## Solution

### 전체 아키텍처 설계

우리는 AI Compose Commit 기능을 구현하기 위해 계층형 아키텍처를 채택하여 시스템이 좋은 확장성과 유지 관리성을 갖도록 보장했습니다:

#### 1. API 계층(Web 계층)

**GitController**는 `POST /api/git/auto-compose-commit` 엔드포인트를 제공하며 기능 진입점으로 작동합니다. 사용자 경험을 최적화하기 위해 우리는 Fire-and-Forget 비동기 모드를 채택했습니다:

- 클라이언트가 요청을 보내면 서버는 즉시 HTTP 202 Accepted를 반환합니다.
- 실제 AI 처리는 백그라운드에서 비동기적으로 실행됩니다.
- 처리가 완료되면 SignalR을 통해 클라이언트에 알립니다.

이 설계는 AI 처리가 몇 분이 걸리더라도 사용자가 즉시 응답을 받아 시스템이 멈춘 것처럼 느끼지 않도록 보장합니다.

#### 2. 애플리케이션 서비스 계층(Application 계층)

**GitAppService**는 핵심 비즈니스 로직을 담당합니다:

- 저장소 감지: monorepo에서 다중 저장소 관리 지원
- 잠금 관리: 동시 작업으로 인한 충돌 방지
- 파일 스테이징 조정: AI 처리 프로세스와의 상호 작용
- 오류 롤백: 실패 시나리오에서 상태 복구 처리

#### 3. 분산 컴퓨팅 계층(Orleans Grains)

**AIGrain**은 AI 작업의 핵심 실행 단위로 작동하며 `IAIGrain` 인터페이스의 `AutoComposeCommitAsync` 메서드를 구현합니다:

```csharp
// AI 자동 조합 커밋의 인터페이스 메서드 정의
// 매개변수 설명:
// - projectId: 프로젝트 고유 식별자
// - unstagedFiles: 스테이지되지 않은 파일 목록, 파일 경로 및 상태 정보 포함
// - projectPath: 프로젝트 루트 디렉터리 경로(선택 사항), 프로젝트 컨텍스트 접근용
// 반환값: 실행 결과를 포함한 응답 객체, 성공/실패 상태 및 상세 정보 포함
[Alias("AutoComposeCommitAsync")]
[ResponseTimeout("00:20:00")] // 20분 타임아웃, 대형 변경 집합 처리용
Task<AutoComposeCommitResponseDto> AutoComposeCommitAsync(
    string projectId,
    GitFileStatusDto[] unstagedFiles,
    string? projectPath = null);
```

이 메서드는 대형 변경 집합을 처리하기 위해 20분의 타임아웃을 설정했습니다. HagiCode의 실제 사용에서 일부 프로젝트의 단일 변경 사항은 수백 개의 파일을 포함할 수 있으며 더 긴 처리 시간이 필요합니다.

#### 4. AI 서비스 계층

추상적인 `IAIService` 인터페이스를 통해 우리는 AI 서비스의 플러그인 가능 아키텍처를 구현했습니다. 현재 Claude Helper 서비스를 사용하지만 다른 AI 제공업체로 쉽게 전환할 수 있습니다.

### 핵심 구현 로직

#### 파일 컨텍스트 구축

AI는 지능적인 의사결정을 내리기 위해 각 파일의 상태를 알아야 합니다. 우리는 `BuildFileChangesXml` 메서드를 통해 파일 컨텍스트를 구축합니다:

```csharp
/// <summary>
/// AI에 완전한 파일 컨텍스트 정보를 제공하기 위해 파일 변경의 XML 표현을 구축합니다
/// </summary>
/// <param name="stagedFiles">스테이징된 파일 목록, 파일 경로, 상태 및 이전 경로(이름 변경 작업용) 포함</param>
/// <returns>모든 파일의 메타데이터 정보를 포함한 형식화된 XML 문자열</returns>
private static string BuildFileChangesXml(GitFileStatusDto[] stagedFiles)
{
    var sb = new StringBuilder();
    sb.AppendLine("<files>");

    foreach (var file in stagedFiles)
    {
        sb.AppendLine("  <file>");
        // XML 이스케이프를 사용하여 특수 문자가 XML 구조를 파괴하지 않도록 합니다
        sb.AppendLine($"    <path>{System.Security.SecurityElement.Escape(file.Path)}</path>");
        sb.AppendLine($"    <status>{System.Security.SecurityElement.Escape(file.Status)}</status>");

        // 파일 이름 변경 시나리오 처리, AI가 변경 관계를 이해할 수 있도록 이전 경로 기록
        if (!string.IsNullOrEmpty(file.OldPath))
        {
            sb.AppendLine($"    <oldPath>{System.Security.SecurityElement.Escape(file.OldPath)}</oldPath>");
        }

        sb.AppendLine("  </file>");
    }

    sb.AppendLine("</files>");
    return sb.ToString();
}
```

이 XML 형식의 컨텍스트는 파일 경로, 상태 및 이전 경로(이름 변경 작업용)를 포함하며 AI에 완전한 메타데이터를 제공합니다. 구조화된 XML 형식을 통해 우리는 AI가 각 파일의 상태와 변경 유형을 정확하게 이해할 수 있도록 보장했습니다.

#### AI 권한 관리

AI가 직접 Git 작업을 실행할 수 있도록 우리는 포괄적인 도구 권한을 구성했습니다:

```csharp
// AI가 사용할 수 있는 도구 집합 정의, 파일 작업 및 Git 명령 실행 권한 포함
// Read/Write/Edit: 파일 읽기/쓰기/편집 능력
// Bash(git:*): 모든 Git 명령 실행 권한
// 기타 Bash 명령: 파일 내용 및 디렉터리 구조 보기, AI가 컨텍스트를 이해하는 것을 보조
var allowedTools = new[]
{
    "Read", "Write", "Edit",
    "Bash(git:*)", "Bash(cat:*)", "Bash(ls:*)", "Bash(find:*)",
    "Bash(grep:*)", "Bash(head:*)", "Bash(tail:*)", "Bash(wc:*)"
};

// 완전한 AI 요청 객체 구축
var request = new AIRequest
{
    Prompt = prompt,                          // 완전한 프롬프트 템플릿, 작업 지시사항 및 제약 조건 포함
    WorkingDirectory = projectPath ?? GetTempDirectory(), // 작업 디렉터리, AI가 올바른 프로젝트 컨텍스트에서 실행되도록 보장
    AllowedTools = allowedTools,               // 사용 가능한 도구 집합
    PermissionMode = PermissionMode.bypassPermissions, // 권한 검사 우회, Git 작업을 직접 실행할 수 있도록 허용
    LanguagePreference = languagePreference         // 언어 기본 설정, 사용자 기대에 부합하는 커밋 메시지 생성 보장
};
```

여기서는 `PermissionMode.bypassPermissions` 모드를 사용하여 AI가 사용자 확인 없이 직접 Git 명령을 실행할 수 있도록 허용합니다. 이것은 기능 설계의 핵심이지만 동시에 엄격한 입력 검증이 필요하여 오용을 방지합니다. HagiCode는 실제 배포에서 백엔드 매개변수 검증과 로그 모니터링을 통해 이 메커니즘의 보안을 보장했습니다.

#### 커밋 결과 구문 분석

AI 실행이 완료되면 구조화된 결과를 반환합니다. 우리는 호환성을 보장하기 위해 이중 구문 분석 전략을 구현했습니다:

```csharp
/// <summary>
/// AI가 반환한 커밋 실행 결과를 구문 분석합니다. 구분 기호 형식과 정규식 형식을 지원합니다
/// </summary>
/// <param name="aiResponse">AI가 반환한 원시 응답 내용</param>
/// <returns>구문 분석된 커밋 결과 목록, 각 결과는 커밋 해시 및 실행 상태 포함</returns>
private List<CommitResultDto> ParseCommitExecutionResults(string aiResponse)
{
    var results = new List<CommitResultDto>();

    // 구분 기호 구문 분석(새 형식)을 우선 사용, 이 형식이 더 명확하고 신뢰할 수 있습니다
    if (aiResponse.Contains("---"))
    {
        logger.LogDebug("Using delimiter-based parsing for AI response");
        results = ParseDelimitedFormat(aiResponse);

        if (results.Count > 0)
        {
            return results; // 구문 분석 성공, 결과를 직접 반환
        }

        logger.LogWarning("Delimiter-based parsing produced no results, falling back to regex");
    }
    else
    {
        logger.LogDebug("No delimiter found, using legacy regex-based parsing");
    }

    // 정규식 구문 분석(이전 형식)으로 대체, 하위 호환성 보장
    return ParseLegacyFormat(aiResponse);
}
```

구분 기호 형식은 `---`를 커밋 간 구분자로 사용하며 형식이 명확하고 구문 분석하기 쉽습니다:

```
---
Commit 1: abc123def456
feat(auth): add user login functionality

Implement JWT-based authentication with login form and API endpoints.

Co-Authored-By: Hagicode <noreply@hagicode.com>
---
Commit 2: 789ghi012jkl
docs(readme): update installation instructions

Add new setup steps for Docker environment.

Co-Authored-By: Hagicode <noreply@hagicode.com>
---
```

이 형식 설계는 구문 분석을 간단하고 신뢰할 수 있게 만들며 인간이 읽기에도 명확합니다.

### 잠금 관리 메커니즘

동시 작업으로 인한 상태 충돌을 방지하기 위해 우리는 저장소 잠금 메커니즘을 구현했습니다:

```csharp
// 저장소 잠금 획득, 동시 작업 방지
// 매개변수 설명:
// - fullPath: 저장소의 전체 경로, 다른 저장소 인스턴스 식별용
// - requestedBy: 요청자 식별자, 추적 및 로그 기록용
await _autoComposeLockService.AcquireLockAsync(fullPath, requestedBy);

try
{
    // AI Compose Commit 작업 실행
    // 이 부분의 코드는 Orleans Grain 메서드를 호출하여 실제 AI 처리 및 Git 작업을 실행합니다
    await aiGrain.AutoComposeCommitAsync(projectId, unstagedFiles, projectPath);
}
finally
{
    // 잠금이 해제되도록 보장, 작업 성공 또는 실패 여부와 무관
    // finally 블록은 예외 상황에서도 잠금을 해제할 수 있어 교착 상태 방지
    await _autoComposeLockService.ReleaseLockAsync(fullPath);
}
```

잠금은 AI 작업의 타임아웃 설정과 일치하는 20분의 타임아웃을 가집니다. 작업이 실패하거나 타임아웃되면 시스템은 자동으로 잠금을 해제하여 영구적인 차단을 방지합니다. HagiCode의 실제 사용에서 이 잠금 메커니즘은 매우 중요하다는 것을 발견했으며, 특히 팀 협업 환경에서 여러 개발자가 동시에 AI Compose Commit 작업을 트리거할 때 중요합니다.

### SignalR 실시간 알림

처리가 완료되면 시스템은 SignalR을 통해 프론트엔드에 알림을 보냅니다:

```csharp
/// <summary>
/// 자동 조합 커밋 완료 알림을 보냅니다
/// </summary>
/// <param name="projectId">프로젝트 식별자, 올바른 클라이언트로 알림 라우팅용</param>
/// <param name="totalCount">총 커밋 수, 성공 및 실패 포함</param>
/// <param name="successCount">성공한 커밋 수</param>
/// <param name="failureCount">실패한 커밋 수</param>
/// <param name="success">전체 작업 성공 여부 플래그</param>
/// <param name="error">오류 정보(작업 실패 시)</param>
private async Task SendAutoComposeCommitNotificationAsync(
    string projectId,
    int totalCount,
    int successCount,
    int failureCount,
    bool success,
    string? error)
{
    try
    {
        // 상세한 실행 결과를 포함한 알림 데이터 전송 객체 구축
        var notification = new AutoComposeCommitCompletedDto
        {
            ProjectId = projectId,
            TotalCount = totalCount,
            SuccessCount = successCount,
            FailureCount = failureCount,
            Success = success,
            Error = error
        };

        // SignalR Hub를 통해 모든 연결된 클라이언트에 알림 브로드캐스트
        await messageService.SendAutoComposeCommitCompletedAsync(notification);

        logger.LogInformation(
            "Auto compose commit notification sent for project {ProjectId}: {SuccessCount}/{TotalCount} succeeded",
            projectId, successCount, totalCount);
    }
    catch (Exception ex)
    {
        // 알림 오류를 기록하지만 주요 작업 프로세스에는 영향을 주지 않음
        // 알림 실패가 전체 작업 실패를 초래해서는 안 됩니다
        logger.LogError(ex, "Failed to send auto compose commit notification for project {ProjectId}", projectId);
    }
}
```

프론트엔드는 알림을 받으면 UI를 업데이트하고 커밋 성공 또는 실패 상태를 표시하여 사용자 경험을 향상시킵니다. HagiCode 사용에서 이 실시간 피드백 메커니즘은 좋은 사용자 피드백을 받았으며, 사용자는 작업이 언제 완료되었는지 결과가 어떤지 명확하게 알 수 있습니다.

## Implementation

### Prompt 엔지니어링 설계

AI의 동작은 완전히 Prompt에 의해 결정됩니다. 우리는 Auto Compose Commit의 Prompt 템플릿을 신중하게 설계했습니다. 중국어 버전을 예로 들면(`auto-compose-commit.zh-CN.hbs`):

#### 비대화형 모드 지원

Prompt 시작 부분에 비대화형 실행 모드 지원을 명확히 선언합니다. 이는 CI/CD 및 자동화 스크립트의 핵심 요구사항입니다:

```handlebars
**중요 알림**: 이 프롬프트는 비대화형 환경(CI/CD, 자동화 스크립트 등)에서 실행될 수 있습니다.

**비대화형 모드**:
- AskUserQuestion 또는 모든 대화형 도구 사용 금지
- 사용자 입력이 필요한 경우:
  - 합리적인 기본값 사용(예: 커밋 유형에 feat 사용)
  - 선택적 확인 단계 건너뛰기
  - 가정된 내용 기록
```

이 설계는 AI Compose Commit 기능이 대화형 IDE 환경에서만 사용되는 것이 아니라 CI/CD 프로세스에도 통합되어 완전히 자동화된 커밋 프로세스를 실현할 수 있도록 보장합니다.

#### 분기 보호 메커니즘

AI가 위험한 작업을 실행하는 것을 방지하기 위해 우리는 Prompt에 엄격한 분기 보호 규칙을 추가했습니다:

```handlebars
**분기 보호**:
- 모든 분기 전환 작업(git checkout, git switch) 실행 금지
- 모든 git commit 명령은 현재 분기에서 실행해야 함
- 분기 생성, 삭제 또는 이름 변경 금지
- 추적되지 않은 파일 또는 스테이지되지 않은 변경 사항 수정 금지
- 작업을 완료하려면 분기 전환이 필요한 경우 실행 대신 오류 반환
```

이러한 규칙은 AI의 도구 사용 범위를 제한하여 작업의 안전성을 보장합니다. HagiCode의 실제 테스트에서 이러한 제약 조건의 유효성을 검증했으며, AI는 분기 전환이 필요한 시나리오를 만나면 안전하게 오류를 반환하는 것이 위험한 작업을 실행하는 것입니다.

#### 지능형 그룹화 의사결정 트리

Prompt에는 파일 그룹화의 의사결정 로직이 상세히 정의되어 있습니다:

```handlebars
**파일 그룹화 의사결정 트리**:
├── 구성 파일인가(package.json, tsconfig.json, .env 등)?
│   ├── 예 → 독립 커밋(유형:chore 또는 build)
│   └── 아니오 → 계속
├── 문서 파일인가(README.md, *.md, docs/**)?
│   ├── 예 → 독립 커밋(유형:docs)
│   └── 아니오 → 계속
├── 동일한 기능과 관련이 있는가?
│   ├── 예 → 동일한 커밋에 병합
│   └── 아니오 → 개별 커밋
└── 모듈 간 변경인가?
    ├── 예 → 모듈별 그룹화
    └── 아니오 → 기능별 그룹화
```

이 의사결정 트리는 AI에 명확한 그룹화 로직을 제공하여 생성된 커밋이 의미론적 합리성을 준수하도록 보장합니다. HagiCode의 실제 사용에서 이 의사결정 트리는 대부분의 일반적인 시나리오를 처리할 수 있으며 생성된 그룹화 결과는 개발자의 기대에 부합합니다.

#### 과거 형식 일관성 분석

커밋 메시지를 프로젝트 과거와 일치시키기 위해 Prompt는 AI가 생성하기 전에 최근 커밋 과거를 분석하도록 요구합니다:

```handlebars
**과거 형식 일관성**: 커밋 메시지를 생성하기 전에 현재 저장소의 커밋 과거를 분석하여 기존 스타일과 일치시켜야 합니다

1. git log -n 15 --pretty=format:"%H|%s|%b%n---%n"을 사용하여 최근 커밋 과거 가져오기
2. 커밋을 분석하여 식별:
   - 구조 패턴: 프로젝트가 다단락을 사용하는가? "Changes:" 또는 "Capabilities:" 섹션이 있는가?
   - 언어 패턴: 커밋 메시지가 영어, 중국어 또는 혼합인가?
   - 일반적인 유형: 가장 자주 사용되는 커밋 유형(feat, fix, docs 등)은 무엇인가?
   - 특수 형식: Co-Authored-By 행이 있는가? 기타 프로젝트 특정 규칙?
3. 감지된 패턴을 따르는 커밋 메시지 생성
```

이 분석은 AI가 생성한 커밋 메시지가 돌출되지 않고 프로젝트의 커밋 과거와 스타일을 일치시키도록 보장합니다. HagiCode의 다국어 프로젝트에서 이 기능은 특히 중요하며 프로젝트의 커밋 과거에 따라 적절한 언어와 형식을 자동으로 선택할 수 있습니다.

#### Co-Authored-By 요구사항

각 커밋은 Co-Authored-By 정보를 포함해야 합니다:

```handlebars
**중요**: 각 커밋은 Co-Authored-By 정보를 추가해야 합니다
- 다음 형식 사용:git commit -m "type(scope): subject" -m "" -m "Co-Authored-By: Hagicode <noreply@hagicode.com>"
- 또는 커밋 메시지에 Co-Authored-By 행을 직접 포함
```

이것은 기여 규칙을 위한 것일 뿐만 아니라 AI 보조 커밋 과거를 추적하기 위한 것입니다. HagiCode는 이 요구사항을 강제 규칙으로 만들어 모든 AI가 생성한 커밋이 명확한 출처 식별자를 갖도록 보장합니다.

### 워크플로우 상세 설명

완전한 AI Compose Commit 워크플로우는 다음과 같습니다:

1. **사용자 트리거**: 사용자가 Git Status 패널 또는 Quick Actions Zone에서 "AI Auto Compose Commit" 버튼 클릭
2. **API 요청**: 프론트엔드가 POST 요청을 `/api/git/auto-compose-commit` 엔드포인트로 전송
3. **즉시 응답**: 서버가 HTTP 202 Accepted를 반환, 처리 완료를 기다리지 않음
4. **백그라운드 처리**:
   - GitAppService가 저장소 잠금 획득
   - AIGrain의 AutoComposeCommitAsync 메서드 호출
   - 파일 컨텍스트 XML 구축
   - AI Prompt 실행, AI가 분석하고 커밋 실행
5. **AI 실행**:
   - Git 명령을 사용하여 모든 스테이지되지 않은 변경 사항 가져오기
   - 파일 내용을 읽어 변경 성질 이해
   - 의미적 관계에 따라 파일 그룹화
   - 각 그룹에 대해 `git add` 및 `git commit` 작업 실행
6. **결과 구문 분석**: AI가 반환한 실행 결과 구문 분석
7. **알림 전송**: SignalR을 통해 프론트엔드에 알림
8. **잠금 해제**: 성공 또는 실패 여부와 무관하게 저장소 잠금 해제

이 프로세스 설계는 사용자가 작업을 시작한 후 즉시 다른 작업을 계속할 수 있고 AI 처리 완료를 기다릴 필요가 없도록 보장합니다. HagiCode 사용자 피드백에 따르면 이 비동기 처리 방식은 워크플로우 경험을 크게 향상시킵니다.

우리는 다중 레벨의 오류 처리를 구현했습니다:

#### 1. 입력 검증

// 요청 매개변수의 유효성을 검증하여 유효하지 않은 요청이 백엔드 처리 로직에 도달하는 것을 방지

#### 2. 오류 롤백

AI 처리 과정에서 오류가 발생하면 시스템은 롤백 작업을 실행하여 이미 스테이징된 파일을 스테이징 해제하고 불일치 상태가 남지 않도록 합니다. 이 메커니즘은 HagiCode의 실제 사용에서 여러 번의 예기치 않은 중단을 구하여 저장소 상태의 무결성을 보장했습니다.

#### 3. 타임아웃 처리

20분의 타임아웃 설정은 장기 실행 작업이 리소스를 무기한 차단하지 않도록 보장합니다. 타임아웃 후 시스템은 잠금을 해제하고 사용자에게 작업 실패를 알립니다. HagiCode의 실제 사용에서 대부분의 작업은 2-5분 내에 완료될 수 있으며 초대형 변경 집합을 처리할 때만 타임아웃 제한에 근접합니다.

## Best Practices

### AI Compose Commit 사용 모범 사례

#### 1. 적절한 사용 시기

AI Compose Commit은 다음 시나리오에 가장 적합합니다:

- 하루 일과가 끝난 후 여러 파일의 변경 사항을 일괄 처리
- 리팩토링 작업 후 여러 관련 파일을 개별적으로 커밋해야 할 때
- 기능 개발 완료 후 관련 변경 사항을 그룹화하여 커밋해야 할 때

다음 시나리오에는 적합하지 않습니다:

- 단일 파일의 빠른 커밋(일반 커밋을 직접 사용하는 것이 더 빠름)
- 커밋 내용을 정확하게 제어해야 하는 시나리오
- 민감한 정보를 포함하는 커밋(수동 검토 필요)

#### 2. AI가 생성한 커밋 검토

AI 지능형 그룹화가 강력하지만 개발자는 여전히 생성된 커밋을 검토해야 합니다:

- 커밋 그룹화가 기대에 부합하는지 확인
- 커밋 메시지의 정확성 검증
- 누락되거나 잘못 포함된 파일이 없는지 확인

불합리한 그룹화가 발견되면 `git reset --soft HEAD~N`을 사용하여 취소한 후 다시 그룹화할 수 있습니다. HagiCode의 경험에 따르면 AI 그룹화가 지능적이더라도 수동 검토는 여전히 가치가 있으며 특히 중요한 기능 커밋에서 그렇습니다.

#### 3. 프로젝트 규칙과 함께 사용

프로젝트의 Git 구성이 Conventional Commits를 지원하도록 하세요:

```bash
# commitlint 설치
npm install -g @commitlint/cli @commitlint/config-conventional

# commitlint 구성
echo "module.exports = {extends: ['@commitlint/config-conventional']}" > commitlint.config.js
```

이렇게 하면 CI/CD 프로세스에서 커밋 메시지 형식을 검증하고 AI Compose Commit이 생성한 형식과 일치시킬 수 있습니다.

### 유사한 기능 구현 제안

프로젝트에서 유사한 AI 보조 커밋 기능을 구현하려는 경우 다음은 우리의 제안입니다:

#### 1. 소규모로 시작

단일 커밋 메시지 생성을 먼저 구현한 다음 점진적으로 다중 커밋 그룹화 기능으로 확장합니다. 이렇게 하면 검증과 반복이更容易합니다. HagiCode도 이 경로를 따라 점진적으로 기능을 개선했으며 초기 버전은 단일 커밋만 지원하다가 나중에 다중 커밋 지능형 그룹화로 확장했습니다.

#### 2. 성숙한 AI SDK 사용

AI 호출 로직을 직접 구현하지 말고 기존 SDK를 사용하면 개발 시간과 잠재적 버그를 줄일 수 있습니다. 우리는 안정적인 인터페이스와 완전한 오류 처리를 제공하는 Claude Helper 서비스를 사용했습니다.

#### 3. Prompt 설계 중시

Prompt의 품질은 AI 출력의 품질을 직접 결정합니다. 상세한 Prompt를 설계하는 데 시간을 투자하세요:

- 명확한 작업 설명
- 구체적인 출력 형식 요구사항
- 경계 상황 처리 규칙
- 예시 설명

HagiCode는 Prompt 설계에 상당한 시간을 투자했으며 이것이 기능 성공의 핵심 요인 중 하나입니다.

#### 4. 포괄적인 오류 처리 구현

AI 작업은 다양한 이유로 실패할 수 있습니다(네트워크 문제, API 속도 제한, 콘텐츠 검토 등). 시스템이 이러한 오류를 우아하게 처리하고 의미 있는 오류 정보를 제공할 수 있도록 하세요.

#### 5. 수동 개입 메커니즘 제공

완전히 자동화하지 말고 사용자에게 제어권을 남겨주세요. 그룹화 결과 보기, 그룹화 조정, 커밋 메시지 수동 편집 등의 옵션을 제공하여 자동화와 유연성의 균형을 맞추세요. HagiCode는 자동 실행을 구현했지만 여전히 미리보기 및 조정 기능을 유지합니다.

### 성능 최적화 팁

#### 1. 파일 필터링

파일 컨텍스트를 구축할 때 AI 분석이 필요 없는 파일을 필터링하세요:

```csharp
// 자동 생성된 파일과 너무 큰 파일을 필터링하여 AI 처리 부담 줄이기
var relevantFiles = stagedFiles
    .Where(f => !IsGeneratedFile(f.Path))
    .Where(f => !IsLargeFile(f.Path))
    .ToArray();
```

#### 2. 병렬 처리

여러 독립 저장소를 지원하는 경우 병렬로 다른 저장소의 커밋을 처리하여 전체 효율성을 높일 수 있습니다.

#### 3. 캐시 최적화

프로젝트 커밋 과거 분석 결과를 캐시하여 매번 다시 분석하지 않도록 합니다. 구성 파일에 과거 형식 기본 설정을 저장하여 AI 호출 횟수를 줄일 수 있습니다.

## Conclusion

AI Compose Commit 기능은 소프트웨어 개발 도구에서 AI 기술의 심층 적용을 대표합니다. 지능적인 파일 변경 분석, 자동 그룹화 커밋, 표준화된 커밋 메시지 생성을 통해 Git 워크플로우의 효율성을 크게 향상시키고 개발자가 핵심 코딩 작업에 더 집중할 수 있도록 합니다.

구현 과정에서 우리는 몇 가지 중요한 경험을 얻었습니다:

1. **사용자 피드백이 핵심**: 초기 버전은 동기 대기 방식을 채택했고 사용자 피드백은 경험이 좋지 않았으며 Fire-and-Forget 모드로 변경한 후 만족도가 크게 향상되었습니다
2. **Prompt 설계가 품질을 결정**: 신중하게 설계된 Prompt가 복잡한 알고리즘보다 AI 출력 품질을 더 잘 보장합니다
3. **안전이 항상 최우선**: AI에 Git 명령을 직접 실행할 수 있는 권한을 부여하는 것은 효율성 향상을 가져왔지만 반드시 엄격한 제약과 검증과 함께 해야 합니다
4. **점진적 개선**: 간단한 시나리오부터 시작하여 점진적으로 복잡도를 높이는 것이 모든 기능을 한 번에 구현하는 것보다 성공하기更容易합니다

미래에 우리는 AI Compose Commit 기능을 더욱 최적화할 계획입니다:

- 더 많은 커밋 그룹화 전략 지원(시간별, 개발자별 등)
- 코드 검토 프로세스 통합, 커밋 전 자동 검토 트리거
- 사용자 정의 커밋 메시지 템플릿 지원, 다른 프로젝트의 개인화된 요구 충족

이 문서에서 공유한 방안이 가치 있다고 생각하신다면 HagiCode도 사용해보고 실제 개발에서 이 기능의 효과를 경험해보세요.毕竟 실천은 진리를 검증하는 유일한 표준입니다.
