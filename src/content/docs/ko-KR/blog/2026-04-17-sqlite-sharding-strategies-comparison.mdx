## SQLite 샤딩 솔루션 실전: 세 가지 샤딩 전략의 심층 비교

## 배경

고성능 애플리케이션을 구축할 때 단일 파일 SQLite 데이터베이스는 매우 현실적인 문제에 직면하게 됩니다. 사용자 수와 데이터량이 늘어나면 다음과 같은 상황들이 줄지어 발생합니다:

- 쓰기 작업이 대기열을 형성하기 시작하고, 응답 시간이 눈에 띄게 길어집니다
- 쿼리 성능이 데이터 증가와 함께 하락합니다
- 다중 스레드 액세스 시 "database is locked" 오류가 자주 발생합니다

많은 사람들의 첫 반응은 PostgreSQL이나 MySQL로 직접 마이그레이션하는 것입니다. 이 작업은 문제를 해결할 수 있지만 배포 복잡성이 급격히 증가합니다. 더 가벼운 솔루션이 없을까요?

답은 샤딩입니다. 결국 엔지니어링 문제는 엔지니어링 방법으로 해결해야 합니다. 데이터를 여러 SQLite 파일로 분산시켜 동시성 능력과 쿼리 성능을 크게 향상시키면서 SQLite의 가벼운 특성을 유지할 수 있습니다.

## 세 가지 샤딩 솔루션 개요

HagiCode 코드베이스 분석을 통해 우리는 다양한 비즈니스 시나리오에 맞는 세 가지 SQLite 샤딩 솔루션을 발견했습니다:

1. **Session Message 샤딩 저장**: AI 대화 메시지 저장, 특징은 고빈도 쓰기, Session 기반 격리 쿼리
2. **Orleans Grain 샤딩 저장**: 분산 프레임워크 상태 지속성, 특징은 노드 간 액세스, 결정론적 라우팅 필요
3. **Hero History 샤딩 저장**: 게임화 시스템 히스토리 이벤트 기록, 특징은 이벤트 소싱, 마이그레이션 호환성 필요

비즈니스 시나리오는 다르지만, 세 가지 모두 동일한 핵심 설계 원칙을 따릅니다:

- **결정론적 라우팅**: 비즈니스 ID에서 직접 샤드를 계산, 메타데이터 테이블 불필요
- **투명한 액세스**: 상층이 통합 인터페이스를 통해 작동, 샤딩 존재를 인식하지 못함
- **독립 저장**: 각 샤드는 완전히 독립적인 SQLite 파일
- **동시성 최적화**: WAL 모드 + busy_timeout으로 잠금 경쟁 감소

많은 사람들이 묻습니다: 왜 범용 샤딩 솔루션을 만들지 않나요? 이 질문은 매우 실질적입니다. 우리는 결론부터 말하겠습니다: 엔지니어링에는 만능 솔루션이 없고 현재 비즈니스 시나리오에 가장 적합한 솔루션만 있을 뿐입니다. 다음으로 이 세 가지 솔루션의 구체적인 구현을 심층 비교해 보겠습니다.

## 샤딩 전략 비교

### 샤드 수량 및 명명 규칙

| 측면 | Session Message | Orleans Grain | Hero History |
|------|----------------|---------------|--------------|
| **샤드 수량** | 256 (16²) | 100 | 10 |
| **명명 규칙** | 16진수 (00-ff) | 10진수 (00-99) | 10진수 (0-9) |
| **저장 디렉토리** | `DataDir/messages/` | `DataDir/orleans/grains/` | `DataDir/hero-history/` |
| **파일명 패턴** | `{shard}.db` | `grains-{shard}.db` | `{shard}.db` |

샤드 수량 차이가 왜 이렇게 큰가요? 이것은 비즈니스 특징에 달려 있습니다. 다시 말해, 모델은 변하고, 도구는 변하고, 워크플로우는 업그레이드되지만 엔지니어링의 기본은 항상 거기에 있습니다: 먼저 무슨 문제를 해결할지 명확히 해야 합니다.

- **Session Message**은 256개 샤드를 사용합니다. 대화 메시지의 쓰기 빈도가 가장 높고 부하를 분산시키기 위해 더 많은 샤드가 필요하기 때문입니다
- **Orleans Grain**은 100개 샤드를 사용하여 동시성 성능과 관리 복잡성의 균형을 맞춥니다
- **Hero History**는 10개 샤드만 사용합니다. 히스토리 이벤트 쓰기 빈도가 낮고 마이그레이션 비용을 고려해야 하기 때문입니다

### 라우팅 알고리즘 차이

라우팅 알고리즘은 샤딩 솔루션의 핵심으로 데이터가 각 샤드에 어떻게 분산되는지 결정합니다. 세 가지 솔루션은 다양한 라우팅 전략을 사용합니다:

```csharp
// Session Message: GUID 마지막 두 자리 16진수
var normalized = Guid.Parse(sessionId.Value).ToString("N").ToLowerInvariant();
return normalized[^2..];  // 마지막 두 자리 16진수 문자 취득

// Orleans Grain: 숫자 추출 후 마지막 두 자리 모듈로
var digits = ExtractDigits(grainId);  // 모든 숫자 추출
var lastTwoDigits = (digits[^2] * 10) + digits[^1];
return lastTwoDigits % shardCount;

// Hero History: 마지막 문자 ASCII 값 모듈로
return heroId[^1] % 10;
```

**설계 아이디어 분석**:

- **Session Message**의 ID는 GUID이며, 16진수로 변환 후 마지막 두 자리를 취하면 균등하게 분산된 256개 샤드를 얻을 수 있습니다
- **Orleans Grain**의 ID 형식은 통일되지 않아 문자와 숫자를 포함할 수 있으므로 모든 숫자를 추출한 후 모듈로를 취합니다
- **Hero History**의 ID는 문자열이며 마지막 문자의 ASCII 값으로 직접 모듈로를 취합니다. 간단하지만 분산이 균등하지 않을 수 있습니다

**핵심 포인트**: 어떤 알고리즘을 사용하든 동일한 ID가 항상 동일한 샤드에 매핑되도록 보장해야 합니다. 이것은 분산 시스템의 가장 기본적인 요구사항이며, 그렇지 않으면 데이터 불일치가 발생합니다. 결국 라우팅이 불안정하면 모든 노력이 무의미해집니다.

### 초기화 전략 차이

| 측면 | Session Message | Orleans Grain | Hero History |
|------|----------------|---------------|--------------|
| **초기화 시점** | 요청 시 지연 로딩 | 시작 시 전체 병렬 초기화 | 요청 시 지연 로딩 |
| **동시성 제어** | `Lazy<Task>` 중복 초기화 방지 | `Parallel.ForEachAsync` | `Lazy<Task>` 중복 초기화 방지 |

**왜 Orleans Grain은 시작 시 전체 초기화를 선택했나요?**

Orleans는 분산 프레임워크이며 Grain은 임의 노드에 스케줄링될 수 있습니다. 런타임에 샤드 파일이 없음을 발견하면 요청 실패로 이어집니다. 시작 시 전체 초기화는 시작 시간을 연장하지만 런타임 안정성을 보장할 수 있습니다. 실행 가능한 것은 시작일 뿐이며, 유지 가능해야 진짜 실력입니다.

**지연 로딩의 장점**:

Session Message와 Hero History의 경우 지연 로딩을 사용하면 시작 시간을 줄일 수 있으며, 실제로 특정 샤드에 액세스해야 할 때만 파일을 생성하고 Schema를 초기화합니다. `Lazy<Task>`를 사용하면 동시 초기화 시 경쟁 상태를 방지할 수 있습니다. 이 설계는 간단해 보이지만 실제 프로젝트에서 많은 불필요한 문제를 줄여줍니다.

### Schema 설계 특징

세 가지 솔루션의 Schema 설계는 각자의 비즈니스 특징을 반영합니다:

**Session Message**:
- Event Sourcing 패턴 지원(이벤트 테이블 + 스냅샷 테이블)
- 메시지 내용 블록 하위 테이블 포함(MessageContentBlocks)
- 압축 및 압축 표시 필드 포함, 후속 최적화 지원

**Orleans Grain**:
- 최소 설계: 단일 테이블 GrainState
- JSON 직렬화 저장 상태
- ETag 낙관적 동시성 제어

**Hero History**:
- 타임라인 쿼리 최적화 인덱스
- DedupeKey 유일 제약으로 중복 방지
- 다양한 이벤트 유형과 상태 지원

이러한 설계에서 Schema 설계는 범용성을 추구하기보다 비즈니스 요구사항에 밀접하게 부합해야 함을 알 수 있습니다. Orleans Grain의 간단한 설계는 직렬화된 상태만 저장하면 되고 복잡한 쿼리 능력이 필요하지 않기 때문입니다. 이것은 허황된 이론이 아니라 엔지니어링입니다. 이름을 너무 크게 짓는 것보다 먼저 이것이 팀에서 두 반복기 동안 살아남을 수 있는지 살펴보세요.

### 동시성 구성 비교

세 가지 솔루션은 모두 동일한 SQLite 동시성 최적화 구성을 사용합니다:

```sql
PRAGMA journal_mode=WAL;      -- 쓰기 전 로그 모드
PRAGMA synchronous=NORMAL;     -- 지속성 오버헤드 감소
PRAGMA busy_timeout=5000;      -- 5초 대기
PRAGMA foreign_keys=ON;        -- 외래 키 제약
```

**WAL 모드의 장점**:

전통적인 롤백 로그 모드는 쓰기 시 잠금 경쟁을 발생시키는 반면 WAL 모드는 읽기와 쓰기 동시 실행을 허용합니다. 이는 대용량 데이터 시나리오에서 성능을 크게 향상시킬 수 있습니다. 많은 사람들이 이 구성을 모르지만 사실 생각보다 훨씬 중요합니다.

**synchronous=NORMAL의 절충**:

FULL로 설정하면 최고 안전성을 보장할 수 있지만 성능이 크게 저하됩니다. NORMAL 모드는 안전성과 성능 간의 균형을 취하며 대부분의 애플리케이션에 적합한 선택입니다. 이 구성으로 너무 오래 고민할 필요 없습니다, NORMAL이면 충분합니다.

## 샤딩 전략 선택 방법

HagiCode 세 가지 솔루션 분석을 기반으로 다음 의사 결정 매트릭스를 요약할 수 있습니다:

```
고처리량 시나리오 → 더 많은 샤드(예: Message용 256)
간단한 유지보수   → 더 적은 샤드(예: Hero History용 10)
숫자 ID 위주     → 모듈로 알고리즘(Orleans Grain)
GUID 위주        → 16진수 접미사(Session Message)
문자열 ID        → ASCII 모듈로(Hero History)
```

**샤드 수량 선택 경험값**:

- 너무 적음(< 10): 동시성 향상 제한, 샤딩 의미 미미
- 너무 많음(> 1000): 파일 관리 복잡, 연결 풀 오버헤드 큼
- 경험값: 10-100개 샤드는 대부분 시나리오에 적합
- 극고동시성 시나리오: 256개 샤드 고려 가능

이것을 데모만 보면 정말 쉽게 빠져들 수 있습니다. 하지만 한번 프로덕션 환경에 들어가면 비용을 하나씩 계산해야 합니다. 많은 문제가 불가능한 것이 아니라 비용을 명확히 계산하지 못했을 뿐입니다.

## 실천 가이드

### 표준화된 샤딩 라우터 구현

```csharp
public interface IShardResolver<TId>
{
    string ResolveShardKey(TId id);
}

// 16진수 샤딩(GUID용)
public class HexSuffixShardResolver : IShardResolver<string>
{
    private readonly int _suffixLength;

    public HexSuffixShardResolver(int suffixLength = 2)
    {
        _suffixLength = suffixLength;
    }

    public string ResolveShardKey(string id)
    {
        var normalized = id.Replace("-", "").ToLowerInvariant();
        return normalized[^_suffixLength..];
    }
}

// 숫자 모듈로 샤딩(순수 숫자 ID용)
public class NumericModuloShardResolver : IShardResolver<long>
{
    private readonly int _shardCount;

    public NumericModuloShardResolver(int shardCount)
    {
        _shardCount = shardCount;
    }

    public string ResolveShardKey(long id)
    {
        return (id % _shardCount).ToString("D2");
    }
}
```

### 통합 연결 팩토리 패턴

```csharp
public class ShardedConnectionFactory<TOptions>
{
    private readonly ConcurrentDictionary<string, Lazy<Task>> _initializationTasks = new();
    private readonly TOptions _options;
    private readonly IShardSchemaInitializer _initializer;

    public ShardedConnectionFactory(
        TOptions options,
        IShardSchemaInitializer initializer)
    {
        _options = options;
        _initializer = initializer;
    }

    public async Task<TDbContext> CreateAsync(string shardKey, CancellationToken ct)
    {
        var connectionString = BuildConnectionString(shardKey);

        // Lazy<Task>를 사용하여 동시 초기화 방지
        var initTask = _initializationTasks.GetOrAdd(
            connectionString,
            _ => new Lazy<Task>(() => InitializeShardAsync(connectionString, ct))
        );

        await initTask.Value;
        return CreateDbContext(connectionString);
    }

    private async Task InitializeShardAsync(string connectionString, CancellationToken ct)
    {
        await _initializer.InitializeAsync(connectionString, ct);
    }

    private string BuildConnectionString(string shardKey)
    {
        var shardPath = Path.Combine(_options.BaseDirectory, $"{shardKey}.db");
        return $"Data Source={shardPath}";
    }

    private TDbContext CreateDbContext(string connectionString)
    {
        // 구체적인 ORM에 따라 DbContext 생성
        return Activator.CreateInstance(typeof(TDbContext), connectionString) as TDbContext;
    }
}
```

### Schema 초기화 모범 사례

```csharp
public class SqliteShardInitializer : IShardSchemaInitializer
{
    public async Task InitializeAsync(string connectionString, CancellationToken ct)
    {
        await using var connection = new SqliteConnection(connectionString);
        await connection.OpenAsync(ct);

        // 동시성 최적화 구성
        await connection.ExecuteAsync("""
            PRAGMA journal_mode=WAL;
            PRAGMA synchronous=NORMAL;
            PRAGMA busy_timeout=5000;
            PRAGMA foreign_keys=ON;
        """);

        // 테이블 구조 생성
        await connection.ExecuteAsync("""
            CREATE TABLE IF NOT EXISTS Entities (
                Id TEXT PRIMARY KEY,
                CreatedAt TEXT NOT NULL,
                UpdatedAt TEXT NOT NULL,
                Data TEXT NOT NULL,
                ETag TEXT
            );
        """);

        // 인덱스 생성
        await connection.ExecuteAsync("""
            CREATE INDEX IF NOT EXISTS IX_Entities_CreatedAt
            ON Entities(CreatedAt DESC);

            CREATE INDEX IF NOT EXISTS IX_Entities_UpdatedAt
            ON Entities(UpdatedAt DESC);
        """);
    }
}
```

### 주의사항

**1. 라우팅 안정성**

라우팅 알고리즘은 동일한 ID가 항상 동일한 샤드에 매핑되도록 보장해야 합니다. 랜덤 또는 시간 관련 계산을 피하고 알고리즘에 가변 매개변수를 도입하지 마세요.

**2. 샤드 수량 선택**

샤드 수량은 설계 단계에서 결정해야 하며 후기 수정은 매우 어렵습니다. 다음을 고려해야 합니다:
- 현재 및 미래 동시성
- 단일 샤드 관리 비용
- 데이터 마이그레이션 복잡성

**3. 마이그레이션 고려**

Hero History 솔루션은 완전한 마이그레이션 경로를 보여줍니다:

1. 새 샤딩 저장 인프라 구축
2. 마이그레이션 서비스를 구현하여 메인 DB 데이터를 샤드로 복사
3. 마이그레이션 후 쿼리 호환성 검증
4. 샤드로 읽기/쓰기 경로 전환
5. 메인 DB 구 테이블 정리

샤딩 솔루션 설계 시 미래 마이그레이션 요구사항을 고려해야 합니다. Talk is cheap. Show me the code지만 코드만으로는 부족하며 완전한 마이그레이션 경로도 필요합니다. 한 번의 성공은 시스템이 아니며 지속적인 성공이 시스템입니다.

**4. 모니터링 및 운영**

- 각 샤드의 크기 분포를 모니터링하여 데이터 스큐를及时发现
- 샤드 핫스팟에 대한 알람을 설정하여 단일 샤드 병목 방지
- WAL 파일 크기를 정기적으로 확인하여 디스크 공간 과다 점유 방지
- 샤드 상태 확인 메커니즘 구축

**5. 테스트 커버리지**

- 경계 조건 테스트(빈 ID, 특수 문자, 초장 ID)
- 라우팅 결정론적 검증, 동일한 ID가 항상 동일한 샤드에 매핑되는지 확인
- 동시 쓰기 스트레스 테스트, 잠금 경쟁이 효과적으로 완화되는지 검증
- 마이그레이션 테스트, 데이터 무결성과 일관성 보장

## 요약

HagiCode 프로젝트의 세 가지 SQLite 샤딩 솔루션 비교를 통해 다음을 알 수 있습니다:

1. **만능 솔루션은 없습니다**: 다른 비즈니스 시나리오는 다른 샤딩 전략이 필요합니다
2. **핵심 원칙은 범용입니다**: 결정론적 라우팅, 투명 액세스, 독립 저장, 동시성 최적화
3. **설계는 미래를 향해야 합니다**: 마이그레이션 경로와 운영 비용 고려

프로젝트가 SQLite를 사용 중이고 동시성 병목에 직면하기 시작했다면 이 글이 몇 가지 아이디어를 제공할 수 있기를 바랍니다. 대형 데이터베이스로 급하게 마이그레이션할 필요 없습니다. 때로는 적절한 샤딩 솔루션으로 문제를 해결할 수 있습니다.

물론 샤딩은 은탄환이 아닙니다. 샤딩 솔루션을 선택하기 전에 먼저 확인하세요:
- 단일 테이블 쿼리 성능을 최적화했습니다
- 적절한 인덱스를 사용했습니다
- WAL 모드를 활성화했습니다

이러한 최적화를 모두 완료한 후에도 여전히 성능 병목이 존재할 때만 샤딩 도입을 고려하세요. 간단한 일을 잘할 수 있는 것 자체가 능력입니다.
