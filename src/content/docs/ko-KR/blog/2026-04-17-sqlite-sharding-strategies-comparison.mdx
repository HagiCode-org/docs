## SQLite 샤딩 실무: 세 가지 샤딩 전략의 심층 비교

> 단일 파일 SQLite 데이터베이스가 동시성 병목에 직면하면 어떻게 돌파할까요? HagiCode 프로젝트의 세 가지 SQLite 샤딩 접근 방식을 다양한 시나리오에서 공유하여, 올바른 샤딩 전략을 선택하는 방법을 이해하는 데 도움을 드립니다.

## 배경

고성능 애플리케이션을 구축할 때 단일 파일 SQLite 데이터베이스는 매우 실용적인 문제에 직면합니다. 사용자 수와 데이터 볼륨이 증가하면 다음 문제들이 줄줄이 나타납니다.

- 쓰기 작업이 큐에 쌓이고 응답 시간이 눈에 띄게 증가합니다
- 데이터 볼륨이 증가함에 따라 쿼리 성능이 저하됩니다
- 멀티스레드 접근에서 빈번한 `database is locked` 오류가 발생합니다

많은 사람들이 본능적으로 묻습니다. PostgreSQL이나 MySQL로 바로 마이그레이션하면 되지 않을까요? 그것도 문제를 해결할 수 있지만 배포 복잡성이 급격히 높아집니다. 더 가벼운 옵션은 없을까요?

대답은 샤딩입니다. 결국 공학적 문제는 공학적 방법으로 해결해야 합니다. 데이터를 여러 SQLite 파일에 분산시키면 SQLite의 경량 특성을 유지하면서 동시성과 쿼리 성능을 크게 향상시킬 수 있습니다.

## 세 가지 샤딩 접근 방식 개요

HagiCode 코드베이스를 분석하여 다양한 비즈니스 시나리오에 맞는 세 가지 SQLite 샤딩 접근 방식을 식별했습니다.

1. **세션 메시지 샤딩 저장**: AI 대화 메시지 저장용, 고빈도 쓰기와 세션 기반 격리 쿼리가 특징
2. **Orleans Grain 샤딩 저장**: 분산 프레임워크의 상태 지속성용, 크로스 노드 접근과 결정론적 라우팅 필요가 특징
3. **히어로 히스토리 샤딩 저장**: 게이미피케이션 시스템의 과거 이벤트 기록용, 이벤트 소싱과 마이그레이션 호환성 필요가 특징

비즈니스 시나리오는 다르지만 세 가지 모두 동일한 핵심 설계 원칙을 따릅니다.

- **결정론적 라우팅**: 비즈니스 ID에서 샤드를 직접 계산하며, 메타데이터 테이블이 필요 없습니다
- **투명한 접근**: 상위 계층은 통합 인터페이스를 사용하며 하위 샤드를 인식하지 못합니다
- **독립 저장**: 각 샤드는 완전히 독립적인 SQLite 파일입니다
- **동시성 최적화**: WAL 모드와 `busy_timeout`으로 잠금 경합을 줄입니다

많은 사람이 묻습니다. 왜 하나의 범용 샤딩 솔루션을 만들지 않나요? 매우 실용적인 질문이며, 결론은 간단합니다. 공학에서는 범용 솔루션이 없으며 현재 비즈니스 시나리오에 가장 잘 맞는 솔루션만 있을 뿐입니다. 다음으로 이 세 가지 접근 방식의 구체적인 구현을 심층적으로 비교하겠습니다.

## 샤딩 전략 비교

### 샤드 수 및 명명 규칙

| 항목 | 세션 메시지 | Orleans Grain | 히어로 히스토리 |
|------|----------------|---------------|--------------|
| **샤드 수** | 256 (16²) | 100 | 10 |
| **명명 규칙** | 16진수 (00-ff) | 10진수 (00-99) | 10진수 (0-9) |
| **저장 디렉토리** | `DataDir/messages/` | `DataDir/orleans/grains/` | `DataDir/hero-history/` |
| **파일명 패턴** | `{shard}.db` | `grains-{shard}.db` | `{shard}.db` |

샤드 수에 이렇게 큰 차이가 있는 이유는 무엇일까요? 비즈니스 특성에 따라 다릅니다. 다시 말해 모델은 변하고 도구는 발전하며 워크플로우는 업그레이드되지만, 공학의 기본은 변하지 않습니다. 먼저 실제로 해결하려는 문제를 이해하는 것입니다.

- **세션 메시지**는 256개의 샤드를 사용합니다. 대화 메시지의 쓰기 빈도가 가장 높고 부하를 분산시키기 위해 더 많은 샤드가 필요하기 때문입니다
- **Orleans Grain**은 100개의 샤드를 사용하여 동시성 성능과 운영 복잡성의 균형을 맞춥니다
- **히어로 히스토리**는 10개의 샤드만 사용합니다. 과거 이벤트 쓰기 빈도가 낮고 마이그레이션 비용을 고려해야 하기 때문입니다

### 라우팅 알고리즘 차이

라우팅 알고리즘은 샤딩 방식의 핵심입니다. 데이터가 샤드에 어떻게 분산되는지를 결정합니다. 세 가지 접근 방식은 다른 라우팅 전략을 사용합니다.

```csharp
// 세션 메시지: GUID의 마지막 두 자리 16진수
var normalized = Guid.Parse(sessionId.Value).ToString("N").ToLowerInvariant();
return normalized[^2..];  // 마지막 두 자리 16진수를 가져옵니다

// Orleans Grain: 숫자를 추출한 다음 마지막 두 자리를 샤드 수로 모듈로
var digits = ExtractDigits(grainId);  // 모든 숫자 추출
var lastTwoDigits = (digits[^2] * 10) + digits[^1];
return lastTwoDigits % shardCount;

// 히어로 히스토리: 마지막 문자의 ASCII 값을 10으로 모듈로
return heroId[^1] % 10;
```

**설계 분석**:

- **세션 메시지** ID는 GUID입니다. 16진수로 변환한 후 마지막 두 자리를 가져오면 256개의 샤드에 균등하게 분산됩니다
- **Orleans Grain** ID는 일관된 형식이 없으며 문자와 숫자가 모두 포함될 수 있으므로 모든 숫자를 추출한 다음 모듈로를 사용합니다
- **히어로 히스토리** ID는 문자열이므로 마지막 문자의 ASCII 값을 직접 모듈로 사용합니다. 간단하지만 분포가 덜 균일할 수 있습니다

**핵심 포인트**: 어떤 알고리즘을 사용하든 동일한 ID는 항상 동일한 샤드에 매핑되어야 합니다. 이것은 분산 시스템에서 가장 기본적인 요구 사항 중 하나입니다. 그렇지 않으면 데이터 불일치는 피할 수 없습니다. 라우팅이 불안정하면 다른 모든 노력이 무로 돌아갑니다.

### 초기화 전략 차이

| 항목 | 세션 메시지 | Orleans Grain | 히어로 히스토리 |
|------|----------------|---------------|--------------|
| **초기화 시점** | 온디맨드 지연 로드 | 시작 시 전체 병렬 초기화 | 온디맨드 지연 로드 |
| **동시성 제어** | `Lazy<Task>`로 중복 초기화 방지 | `Parallel.ForEachAsync` | `Lazy<Task>`로 중복 초기화 방지 |

**왜 Orleans Grain은 시작 시 전체 초기화를 선택했을까요?**

Orleans는 분산 프레임워크이므로 Grain이 어떤 노드에든 스케줄될 수 있습니다. 런타임에 샤드 파일이 누락된 것을 발견하면 요청이 실패할 수 있습니다. 시작 시 전체 초기화는 시작 시간을 연장하지만 런타임 안정성을 보장합니다. 실행하는 것만이 시작일 뿐이며, 유지 관리 가능하게 만드는 것이 진정한 기술입니다.

**지연 로딩의 장점**:

세션 메시지와 히어로 히스토리의 경우 지연 로딩은 시작 시간을 줄입니다. 파일과 스키마는 샤드가 실제로 필요할 때만 생성됩니다. `Lazy<Task>`를 사용하면 동시 초기화 중의 경쟁 상태도 방지합니다. 설계는 간단해 보이지만 실제 프로젝트에서 많은 불필요한 문제를 피합니다.

### 스키마 설계 특징

세 가지 접근 방식의 스키마 설계는 각각의 비즈니스 특성을 반영합니다.

**세션 메시지**:
- Event Sourcing 모델 지원 (이벤트 테이블 + 스냅샷 테이블)
- 메시지 콘텐츠 블록용 자식 테이블 포함 (`MessageContentBlocks`)
- 압축 및 압축 플래그 필드를 포함하여 향후 최적화 지원

**Orleans Grain**:
- 미니멀리스트 설계: 단일 `GrainState` 테이블
- 상태를 직렬화된 JSON으로 저장
- ETag 기반 낙관적 동시성 제어 사용

**히어로 히스토리**:
- 타임라인 쿼리 최적화 인덱스
- 고유한 `DedupeKey` 제약으로 중복 방지
- 여러 이벤트 유형 및 상태 지원

이러한 설계는 스키마 설계가 비즈니스 요구 사항에 긴밀하게 맞춰야 하며, 범용성을 좇아서는 안 된다는 것을 보여줍니다. Orleans Grain이 간단한 이유는 직렬화된 상태만 저장하면 되고 복잡한 쿼리 기능이 필요하지 않기 때문입니다. 이것은 신비주의가 아닙니다. 공학입니다. 실제 팀에서 두 번의 반복을 견뎌낼 수 있는지 확인하기 전에는 거창한 이름을 붙이지 마세요.

### 동시성 구성 비교

세 가지 접근 방식 모두 동일한 SQLite 동시성 최적화 설정을 사용합니다.

```sql
PRAGMA journal_mode=WAL;      -- Write-ahead logging 모드
PRAGMA synchronous=NORMAL;     -- 지속성 오버헤드 감소
PRAGMA busy_timeout=5000;      -- 5초 대기
PRAGMA foreign_keys=ON;        -- 외래 키 제약
```

**WAL 모드의 장점**:

전통적인 롤백 저널 모드는 쓰기 중 잠금 경합을 일으키지만, WAL 모드는 읽기와 쓰기를 동시에 진행할 수 있게 합니다. 대용량 데이터 시나리오에서 이것은 성능을 크게 향상시킬 수 있습니다. 많은 개발자가 이 설정을 간과하지만 생각보다 훨씬 더 중요합니다.

**`synchronous=NORMAL`의 트레이드오프**:

`FULL`로 설정하면 최대 안전성을 제공하지만 성능이 크게 저하됩니다. `NORMAL`은 안전성과 성능 사이의 균형을 맞추며 대부분의 애플리케이션에 올바른 선택입니다. 이것에 대해 너무 생각할 필요는 없습니다. `NORMAL`이 충분합니다.

## 샤딩 전략 선택 방법

HagiCode의 세 가지 접근 방식에 대한 분석을 바탕으로 다음 결정 매트릭스를 요약할 수 있습니다.

```
고처리량 시나리오 -> 더 많은 샤드 (예: 메시지는 256개 사용)
간단한 유지 관리   -> 더 적은 샤드 (예: 히어로 히스토리는 10개 사용)
주로 숫자 ID       -> 모듈로 알고리즘 (Orleans Grain)
주로 GUID         -> 16진수 접미사 (세션 메시지)
문자열 ID         -> ASCII 모듈로 (히어로 히스토리)
```

**샤드 수 선택 규칙**:

- 너무 적음 (`< 10`): 동시성 향상이 제한되어 샤딩의 의미가 줄어듭니다
- 너무 많음 (`> 1000`): 파일 관리가 복잡해지고 연결 풀 오버헤드가 증가합니다
- 경험칙: 10~100개의 샤드가 대부분의 시나리오에 맞습니다
- 극도로 높은 동시성 시나리오: 256개의 샤드를 고려할 수 있습니다

데모만 보면 쉽게 흥분할 수 있습니다. 하지만 프로덕션에 진입하면 모든 비용을 신중하게 계산해야 합니다. 많은 것이 불가능한 것이 아니라 정직하게 가격이 매겨지지 않은 것뿐입니다.

## 실용 가이드

### 표준화된 샤드 리졸버 구현

```csharp
public interface IShardResolver<TId>
{
    string ResolveShardKey(TId id);
}

// 16진수 샤딩 (GUID용)
public class HexSuffixShardResolver : IShardResolver<string>
{
    private readonly int _suffixLength;

    public HexSuffixShardResolver(int suffixLength = 2)
    {
        _suffixLength = suffixLength;
    }

    public string ResolveShardKey(string id)
    {
        var normalized = id.Replace("-", "").ToLowerInvariant();
        return normalized[^_suffixLength..];
    }
}

// 숫자 모듈로 샤딩 (순수 숫자 ID용)
public class NumericModuloShardResolver : IShardResolver<long>
{
    private readonly int _shardCount;

    public NumericModuloShardResolver(int shardCount)
    {
        _shardCount = shardCount;
    }

    public string ResolveShardKey(long id)
    {
        return (id % _shardCount).ToString("D2");
    }
}
```

### 통합 연결 팩토리 패턴

```csharp
public class ShardedConnectionFactory<TOptions>
{
    private readonly ConcurrentDictionary<string, Lazy<Task>> _initializationTasks = new();
    private readonly TOptions _options;
    private readonly IShardSchemaInitializer _initializer;

    public ShardedConnectionFactory(
        TOptions options,
        IShardSchemaInitializer initializer)
    {
        _options = options;
        _initializer = initializer;
    }

    public async Task<TDbContext> CreateAsync(string shardKey, CancellationToken ct)
    {
        var connectionString = BuildConnectionString(shardKey);

        // Lazy<Task>를 사용하여 동시 초기화 방지
        var initTask = _initializationTasks.GetOrAdd(
            connectionString,
            _ => new Lazy<Task>(() => InitializeShardAsync(connectionString, ct))
        );

        await initTask.Value;
        return CreateDbContext(connectionString);
    }

    private async Task InitializeShardAsync(string connectionString, CancellationToken ct)
    {
        await _initializer.InitializeAsync(connectionString, ct);
    }

    private string BuildConnectionString(string shardKey)
    {
        var shardPath = Path.Combine(_options.BaseDirectory, $"{shardKey}.db");
        return $"Data Source={shardPath}";
    }

    private TDbContext CreateDbContext(string connectionString)
    {
        // 특정 ORM에 따라 DbContext 생성
        return Activator.CreateInstance(typeof(TDbContext), connectionString) as TDbContext;
    }
}
```

### 스키마 초기화 모범 사례

```csharp
public class SqliteShardInitializer : IShardSchemaInitializer
{
    public async Task InitializeAsync(string connectionString, CancellationToken ct)
    {
        await using var connection = new SqliteConnection(connectionString);
        await connection.OpenAsync(ct);

        // 동시성 최적화 설정
        await connection.ExecuteAsync("""
            PRAGMA journal_mode=WAL;
            PRAGMA synchronous=NORMAL;
            PRAGMA busy_timeout=5000;
            PRAGMA foreign_keys=ON;
        """);

        // 테이블 스키마 생성
        await connection.ExecuteAsync("""
            CREATE TABLE IF NOT EXISTS Entities (
                Id TEXT PRIMARY KEY,
                CreatedAt TEXT NOT NULL,
                UpdatedAt TEXT NOT NULL,
                Data TEXT NOT NULL,
                ETag TEXT
            );
        """);

        // 인덱스 생성
        await connection.ExecuteAsync("""
            CREATE INDEX IF NOT EXISTS IX_Entities_CreatedAt
            ON Entities(CreatedAt DESC);

            CREATE INDEX IF NOT EXISTS IX_Entities_UpdatedAt
            ON Entities(UpdatedAt DESC);
        """);
    }
}
```

### 핵심 고려 사항

**1. 라우팅 안정성**

라우팅 알고리즘은 동일한 ID가 항상 동일한 샤드에 매핑되도록 보장해야 합니다. 무작위 또는 시간 종속적 계산을 피하고, 변경 가능한 매개변수를 알고리즘에 도입하지 마세요.

**2. 샤드 수 선택**

샤드 수는 설계 단계에서 결정해야 합니다. 나중에 변경하는 것은 극도로 어렵습니다. 다음을 고려하세요.
- 현재 및 미래의 동시성 볼륨
- 각 샤드의 관리 비용
- 데이터 마이그레이션의 복잡성

**3. 마이그레이션 계획**

히어로 히스토리 접근 방식은 완전한 마이그레이션 경로를 보여줍니다.

1. 새로운 샤딩 저장소 인프라 구축
2. 기본 데이터베이스에서 샤드로 데이터를 복사하는 마이그레이션 서비스 구현
3. 마이그레이션 후 쿼리 호환성 검증
4. 읽기/쓰기 경로를 샤드로 전환
5. 기본 데이터베이스의 레거시 테이블 정리

샤딩 방식을 설계할 때 미래의 마이그레이션 요구 사항을 고려해야 합니다. 말은 쉽지만 코드를 보여주세요. 하지만 코드만으로는 충분하지 않습니다. 완전한 마이그레이션 경로도 필요합니다. 일회성 성공은 시스템이 아닙니다. 지속적인 성공이 시스템입니다.

**4. 모니터링 및 운영**

- 샤드 간 크기 분포를 모니터링하여 데이터 치우침을 조기에 감지하세요
- 샤드 핫스팟에 대한 알림을 설정하여 단일 샤드가 병목이 되는 것을 방지하세요
- WAL 파일 크기를 정기적으로 검사하여 과도한 디스크 사용을 피하세요
- 샤드 상태 확인 메커니즘을 확립하세요

**5. 테스트 커버리지**

- 빈 ID, 특수 문자, 과도하게 긴 ID와 같은 경계 조건을 테스트하세요
- 라우팅 결정론성을 검증하여 동일한 ID가 항상 동일한 샤드에 매핑되는지 확인하세요
- 동시 쓰기 스트레스 테스트를 실행하여 잠금 경합이 효과적으로 줄어드는지 확인하세요
- 마이그레이션 테스트를 실행하여 데이터 무결성과 일관성을 보장하세요

## 요약

HagiCode 프로젝트의 세 가지 SQLite 샤딩 접근 방식을 비교하면 다음을 알 수 있습니다.

1. **범용 솔루션은 없습니다**: 다른 비즈니스 시나리오에는 다른 샤딩 전략이 필요합니다
2. **핵심 원칙은 공유됩니다**: 결정론적 라우팅, 투명한 접근, 독립 저장, 동시성 최적화
3. **설계는 미래를 향해야 합니다**: 마이그레이션 경로와 운영 비용을 고려하세요

프로젝트에서 SQLite를 사용하고 동시성 병목에 직면하기 시작했다면 이 글이 몇 가지 유용한 아이디어를 제공하기를 바랍니다. 무거운 데이터베이스로 마이그레이션할 필요는 없습니다. 때로는 올바른 샤딩 전략으로 문제를 해결하기에 충분합니다.

물론 샤딩은 만병통치약이 아닙니다. 샤딩 전략을 선택하기 전에 먼저 다음을 확인하세요.
- 단일 테이블 쿼리 성능을 이미 최적화했는지
- 적절한 인덱스를 이미 추가했는지
- WAL 모드를 이미 활성화했는지

이러한 최적화를 모두 마친 후에도 성능 병목이 남아 있다면 샤딩을 도입하는 것을 고려하세요. 간단한 것을 잘하는 것 자체가 능력입니다.

때로는 한 번 하는 것이 열 번 설명하는 것보다 낫습니다. 여기서부터는 엔지니어링 결과가 말해주도록 하겠습니다.
