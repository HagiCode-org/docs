## 저렴한 클라우드 서버로 다운로드 가속 배포 스테이션 구축 방법

> 클라우드 스토리지의 트래픽 비용은 터무니없이 비싸고, 해외 액세스는 느려서 미치겠고, CDN 가격은 포기하게 만들어요...... 이런 고통은 아마 배포를 해본 사람들은 다 알 거예요. 이 글에서는 [HagiCode](https://hagicode.com) 프로젝트에서摸索해낸 저비용 솔루션을 공유합니다. 클라우드 서버 + Nginx 캐시 계층으로 비용은 절반 줄고 속도는 올라갔습니다. 그래도 작은 위안이 되네요.

## 배경

인터넷이니까 다운로드 속도와 안정성은 결국 사용자 경험입니다. 오픈소스든 상용이든 사용자에게 믿을 수 있는 다운로드 방식을 제공해야 합니다.

클라우드 스토리지(예: Azure Blob Storage, AWS S3)에서 직접 파일을 다운로드하는 것은 간단해 보이지만 실제로는 문제가 꽤 많습니다:

**네트워크 지연**: 국가 간, 지역 간 액세스는 느려서 키보드를 부수고 싶어집니다. 사용자가 기다리다 지쳐버리는데 경험이 좋을 리가요?

**대역폭 비용**: 클라우드 스토리지의 아웃바운드 트래픽은 비싸서 마음이 아픕니다. Azure Blob Storage를 중국 본토에서 액세스하면 대략 GB당 0.5위안, 한 달 1TB면 500위안입니다. 작은 팀에게는 이 돈이 많다고도 하고 적다고도 할 수 없죠. 그래도 누구의 돈이든 공짜로 온 건 아닙니다.

**액세스 제한**: 일부 지역에서 해외 클라우드 서비스 액세스는 때로는 좋다가 때로는 안 되다가, 아예 액세스 못 할 때도 있습니다. 사용자가 다운로드하려고 해도 못 하는 상황, 정말 답답하죠.

**CDN 비용**: 상용 CDN은 확실히 문제를 해결할 수 있지만 가격도 확실히 비쌉니다. 작은 팀이 어디 감히 쓰겠어요?

그럼 돈도 절약하고 쓸만한 방법이 없을까요? 사실 있습니다. 클라우드 서버 + 리버스 프록시 + 캐시 계층, 이렇게 간단하고 직관적입니다. 비용은 대략 절반 줄고 속도는 올라갔습니다. 그래도 작은 위안이 되네요.

## 아키텍처 설계

### 전체 아키텍처 아이디어

먼저 전체 아키텍처 설계를 볼까요:

```
사용자 요청
    ↓
DNS 해석
    ↓
┌─────────────────────────────────────┐
│   리버스 프록시 계층 (Traefik/Bunker Web)   │ ← SSL 종료, 라우팅 분배, 보안 방어
├─────────────────────────────────────┤
│   포트: 80/443                       │
│   기능: 자동 Let's Encrypt 인증서      │
│         Host 라우팅                    │
└─────────────────────────────────────┘
    ↓
┌─────────────────────────────────────┐
│   캐시 계층 (Nginx)                     │ ← 파일 캐시, Gzip 압축
├─────────────────────────────────────┤
│   포트: 8080(server) / 8081(desktop) │
│   캐시 전략:                          │
│   - index.json: 1시간               │
│   - 다른 파일: 7일                   │
│   캐시 크기: 1GB                      │
└─────────────────────────────────────┘
    ↓
┌─────────────────────────────────────┐
│   원본 서버 (Azure Blob Storage)         │ ← 파일 저장소
└─────────────────────────────────────┘
```

이 아키텍처의 핵심 아이디어는, 간단히 말해서: **사용자와 클라우드 스토리지 사이에 캐시 계층을 하나 추가하는 것**입니다.

사용자 요청이 먼저 클라우드 서버의 리버스 프록시 계층으로 가고, 그다음 Nginx 캐시 계층이接管합니다. 캐시에 파일이 있나요? 사용자에게 직접 줍니다. 없나요? 클라우드 스토리지에서 한 벌 가져와서 현지에 한 벌 저장합니다. 다음에 같은 파일을 액세스하면 클라우드 스토리지를 귀찮게 할 필요가 없습니다. 사실 기억과 비슷합니다. 어떤 것들을 기억해두면 다음에는 다시 힘들게 생각할 필요가 없죠......

### 왜 이 아키텍처를 선택했나?

**클라우드 서버 장점**:
- 비용 통제 가능: 알리바바 클라우드 등에서 저렴한 클라우드 서버 제공(1-2코어 2GB 구성 월 요금 약 50-100위안)
- 배포 유연: 리버스 프록시, 캐시 전략을 자유롭게 구성 가능
- 지리적 위치 유연: 사용자에게 가까운 서버 지역 선택 가능
- 확장성 강: 트래픽 요구에 따라 구성 업그레이드 가능

**리버스 프록시 + 캐시 아키텍처**:
- 원본 서버 압력 감소: 핫스팟 파일 캐시로 클라우드 스토리지 액세스 감소
- 비용 절감: 클라우드 서버 트래픽 비용은 클라우드 스토리지 아웃바운드 트래픽보다 훨씬 낮음
- 속도 향상: 근접 액세스, 서버 대역폭은 보통 클라우드 스토리지보다 우수

**왜 캐시 계층으로 Nginx를 선택했나?**

이것도 그냥 선택한 게 아니라, 역시 Nginx에게는 그만한 이유가 있습니다:

1. 고성능: Nginx 리버스 프록시의 성능은 업계에서 주목받는 바 있습니다
2. 캐시 성숙: 내장된 proxy_cache 기능은 안정적이고 신뢰할 수 있어서 자동 문제가 생기지 않습니다
3. 자원 점유 낮음: 256MB 메모리면 실행 가능해서 서버에도 친숙합니다
4. 구성 유연: 다른 파일 유형에 다른 캐시 전략 설정 가능, 이 점은 꽤 마음에 듭니다

## 리버스 프록시 계층: Traefik vs Bunker Web

HagiCode의 배포 솔루션은 사실 두 가지 리버스 프록시를 지원합니다. 말하자면 각각 특징이 있습니다:

| 솔루션 | 특징 | 적용 시나리오 |
|------|------|----------|
| **Traefik** | 경량, 자동 SSL, 구성 간단 | 기초 배포, 저트래픽 시나리오 |
| **Bunker Web** | 내장 WAF, DDoS 방어, 크롤러 방어 | 높은 보안 요구, 높은 트래픽 시나리오 |

### Traefik: 경량 선택

Traefik은 현대 HTTP 리버스 프록시와 로드 밸런서로, 가장 큰 특징은 구성이 간단하고 Let's Encrypt 인증서도 자동으로 처리해줍니다.

초기 배포나 트래픽이 많지 않은 시나리오에서 Traefik은 꽤 좋은 선택입니다:
- 자원 점유가 많지 않습니다(1.5 CPU/512MB 메모리면 충분)
- SSL 인증서 자동 구성, 직접 신경 쓸 필요 없음
- 라우팅 구성이 Docker 라벨 기반이라 편리합니다

### Bunker Web: 높은 보안 시나리오

Bunker Web은 Nginx 기반 웹 애플리케이션 방화벽으로, 보안 방어가 더 포괄적입니다.

언제 Bunker Web으로 전환을 고려할까요? 대략 이런 상황입니다:
- DDoS 공격 받을 때(물론 아무도 만나고 싶지 않겠지만)
- ModSecurity 방어 필요
- 크롤러 방어 기능 원함
- 보안에 더 높은 요구가 있을 때

HagiCode는 `switch-deployment.sh` 스크립트를 제공해서 두 솔루션 사이를 빠르게 전환할 수 있습니다:

```bash
# Bunker Web로 전환
./switch-deployment.sh bunkerweb

# Traefik으로 전환
./switch-deployment.sh traefik

# 현재 상태 보기
./switch-deployment.sh status
```

스크립트는 자동으로 사전 검사, 건강 검사를 하고 자동 롤백도 가능합니다. 전환 과정도 안전적이고 신뢰할 수 있어서, 전환하다 죽는 일은 없습니다.

## Nginx 캐시 계층 구성

캐시 계층은 전체 아키텍처의 핵심입니다. Nginx 구성을 잘하냐 못하냐에 따라 캐시 효과가 천지차입니다. 아주 중요한 일입니다.

### 캐시 경로 구성

```nginx
# 캐시 경로 구성
proxy_cache_path /var/cache/nginx levels=1:2 keys_zone=azure_cache:10m
                   max_size=1g inactive=7d use_temp_path=off;
```

매개변수 설명:
- `levels=1:2`: 캐시 디렉토리 계층, 2단계 디렉토리 구조로 파일 액세스 효율 향상
- `keys_zone=azure_cache:10m`: 캐시 키 저장 영역, 10MB면 대량의 키 저장에 충분
- `max_size=1g`: 최대 캐시 크기 1GB
- `inactive=7d`: 캐시 파일이 7일 동안 액세스되지 않으면 삭제
- `use_temp_path=off`: 캐시 디렉토리에 직접 쓰기로 성능 향상

### 등급별 캐시 전략

다른 유형의 파일은 다른 캐시 전략이 필요합니다:

```nginx
# Server 다운로드 서비스
server {
    listen 8080;

    # index.json 단기 캐시(적시 업데이트를 위해)
    location /index.json {
        proxy_cache azure_cache;
        proxy_cache_valid 200 1h;
        proxy_cache_key "$scheme$server_port$request_uri";
        add_header X-Cache-Status $upstream_cache_status;
        add_header Cache-Control "public, max-age=3600";

        # Azure OSS로 리버스 프록시
        proxy_pass https://${SERVER_DL_HOST}/${SERVER_DL_CONTAINER}$uri?${SERVER_DL_SAS_TOKEN};
        proxy_ssl_server_name on;
        proxy_ssl_protocols TLSv1.2 TLSv1.3;
    }

    # 설치 패키지 등 정적 파일 장기 캐시
    location / {
        proxy_cache azure_cache;
        proxy_cache_valid 200 7d;
        proxy_cache_key "$scheme$server_port$request_uri";
        add_header X-Cache-Status $upstream_cache_status;
        add_header Cache-Control "public, max-age=604800";

        proxy_pass https://${SERVER_DL_HOST}/${SERVER_DL_CONTAINER}$uri?${SERVER_DL_SAS_TOKEN};
        proxy_ssl_server_name on;
        proxy_ssl_protocols TLSv1.2 TLSv1.3;
    }
}
```

**왜 이렇게 설계했나?**

`index.json`은 버전 확인 파일이라 제때 업데이트해야 하므로 캐시 시간을 1시간으로 설정했습니다. 이렇게 하면 새 버전을 출시한 후 최대 1시간이면 사용자가 업데이트를 감지할 수 있고, 너무 길지도 않습니다.

설치 패키지 같은 정적 파일은 변화가 원래 적으므로 7일 캐시하면 원본 서버 액세스를 크게 줄일 수 있습니다. 업데이트가 필요할 때 수동으로 캐시를 한 번 정리하면 되고, 그렇게 번거롭지도 않습니다.

**X-Cache-Status 응답 헤더**:

이 응답 헤더로 캐시 적중 상황을 볼 수 있어서 꽤 유용합니다:
- `HIT`: 캐시 적중
- `MISS`: 캐시 미적중, 원본 서버에서 가져옴
- `EXPIRED`: 캐시 만료, 원본 서버에서 다시 가져옴
- `BYPASS`: 캐시 우회

보는 방법:
```bash
curl -I https://server.dl.hagicode.com/app.zip
```

## 배포 실천

먼저 환경 변수를 구성합니다:

```bash
cd /path/to/hagicode_aliyun_deployment/docker

```

**중요**: `.env` 파일에는 민감한 정보( SAS Token, Webhook URL)가 포함되어 있으므로 절대 버전 컨트롤에 커밋하지 마세요. 이건 꽤 중요한 일입니다.

다음 DNS A 레코드를 추가하는 것을 잊지 마세요:
- `server.dl.hagicode.com` → 서버 IP
- `desktop.dl.hagicode.com` → 서버 IP

### 서버 초기화

Ansible으로 서버 자동 초기화:

```bash

ansible-playbook -i ./ansible/inventory/hosts.yml ./ansible/playbooks/init.yml
```

이 Playbook은 자동으로 이런 것들을 처리합니다:

너무 복잡하지 않습니다. 역시 자동화는 시간도 노력도 절약합니다.

### 캐시 관리

캐시도 가끔 관리해야 합니다:

**캐시 디스크 사용량 보기**:
```bash
docker volume inspect docker_nginx-cache
du -sh /var/lib/docker/volumes/docker_nginx-cache/_data
```

**수동 캐시 제거**:
```bash
./clear-cache.sh
```

또는 수동으로 실행하는 것도 가능합니다. 좀 번거롭지만 효과는 있습니다:
```bash
docker exec nginx sh -c "rm -rf /var/cache/nginx/*"
docker restart nginx
```

### 자원 제한

1코어 2GB 서버에서 자원 제한 구성은 다음과 같습니다:

```yaml
services:
  traefik:
    deploy:
      resources:
        limits:
          cpus: '1.50'
          memory: 512M

  nginx:
    deploy:
      resources:
        limits:
          cpus: '0.50'
          memory: 256M
```

자원 사용 상황을 모니터링하는 것도 가끔 하는 게 좋습니다:
```bash
docker stats
```

### SAS Token 보안 실천

SAS Token은 Azure Blob Storage 액세스 자격 증명입니다. 유출되면 큰일입니다:

- `.env` 파일을 버전 컨트롤에 커밋하지 않기(이미 .gitignore에 있음)
- SAS Token에 적절한 만료 시간 설정(권장 1년)
- SAS Token 권한 제한(읽기만)
- 정기적으로 SAS Token 교체

### 모니터링 알림

HagiCode는 Lark/飞书 Webhook 알림을 통합해서 다음 상황에서 알림을 보낼 수 있습니다:
- 배포 성공/실패
- 캐시 제거 상태
- 서비스 이상

알림에는 서버 정보, 타임스탬프, 오류 세부 정보가 포함되어 있어서 문제를 빠르게定位하기 편합니다.

### 고가용성 확장

단일 서버로 버틸 수 없을 때도 다음을 고려할 수 있습니다:

1. **수평 확장**: 여러 노드 배포, DNS 라운드 로빈 또는 로드 밸런싱으로 분배
2. **CDN 추가**: 클라우드 서버 앞에 CDN 연결로 액세스 속도 더 향상
3. **캐시 예열**: 스크립트로 인기 있는 파일을 미리 캐시에 로드

## 주의 사항

몇 가지는 꼭 알려드려야 합니다. 아무도 이상한 일을 만나고 싶지 않으니까요:

1. **SSL 인증서**: Let's Encrypt에는 속도 제한이 있으니 자주 배포를 전환하지 마세요. 아니면 인증서를 신청하지 못할 수도 있습니다
2. **캐시 제거**: 중요한 파일을 업데이트한 후에는 캐시를 제거하는 것을 기억하세요. 아니면 사용자가 새 버전을 다운로드하지 못할 수도 있습니다
3. **로그 관리**: 정기적으로 Docker 로그를 제거하세요. 아니면 디스크가 가득 차면 곤란합니다
4. **백업 전략**: Traefik acme.json, Bunker Web 구성 같은 건 백업하는 게 좋습니다
5. **모니터링 알림**: 飞书 알림을 구성해서 배포 상황을 제때 알고, 문제가 생겨도 빠르게 대응하세요

## 요약

클라우드 서버 + Nginx 캐시 계층, 이렇게 간단합니다. HagiCode는 이 솔루션으로 비용이 높지 않고(서버 비용 대략 60-100위안/월), 효과는 꽤 좋습니다. 핵심 장점은 대략 이런 것들입니다:

- **비용 통제 가능**: 클라우드 스토리지나 상용 CDN을 직접 사용하는 것보다 비용이 대략 50% 감소
- **배포 유연**: Traefik이든 Bunker Web이든 직접 선택
- **확장성 강**: 필요하면 수평 확장하거나 CDN 추가 가능
- **운영 간단**: Shell 스크립트 + Ansible으로 자동화 배포도 편리

파일 배포가 필요한 작은 팀과 개인 개발자에게는 이 솔루션을 시도해 볼 만합니다.

HagiCode는 이 아키텍처로 생산 환경에서 일정 기간 안정적으로 실행했고, 전 세계 사용자 다운로드도 큰 문제가 없었습니다. 당신도 비슷한 솔루션을 찾고 있다면 시도해 보세요. 아마 도움이 될 겁니다.

## 기술 스택 요약

마지막으로 사용된 기술을 정리하면 그럭저럭 설명이 됩니다:

| 구성 요소 | 선택 | 용도 |
|------|------|------|
| 클라우드 서버 | 알리바바 클라우드 ECS | 기초 실행 환경 |
| 리버스 프록시 | Traefik / Bunker Web | SSL 종료, 라우팅, 보안 방어 |
| 캐시 계층 | Nginx | 리버스 프록시 캐시, Gzip 압축 |
| 파일 저장소 | Azure Blob Storage | 파일 원본 서버 |
| 컨테이너화 | Docker Compose | 서비스 오케스트레이션 |
| 자동화 | Ansible | 서버 구성 관리 |
| 알림 | Lark/飞书 Webhook | 배포 상태 알림 |
