## Hermes Agent 통합 실무: 프로토콜부터 프로덕션까지

> ACP 프로토콜 적응, 세션 풀 관리, 프론트엔드/백엔드 계약 동기화를 포함한 HagiCode의 Hermes Agent 통합 전체 경험을 공유합니다.

## 배경

AI 보조 코딩 플랫폼인 HagiCode를 구축하면서 팀은 로컬에서 실행할 수 있고 클라우드로도 확장할 수 있는 Agent 프레임워크를 통합해야 했습니다. 연구 끝에 Nous Research의 Hermes Agent를 범용 Agent 기능의 기본 엔진으로 선택했습니다.

솔직히 기술 선택은 특별히 어렵지도 특별히 쉽지도 않습니다. 시장에 강력한 Agent 프레임워크가 많이 있지만, Hermes는 ACP 프로토콜과 도구 시스템이 HagiCode의 까다로운 요구 사항인 로컬 개발, 팀 협업, 클라우드 확장에 특히 잘 맞아 돋보였습니다. 하지만 Hermes를 실제 프로덕션 시스템에 통합하려면 긴 공학적 문제 목록을 해결해야 했습니다. 이 부분은 결코 사소하지 않았습니다.

HagiCode의 스택은 Orleans를 사용하여 분산 시스템을 구축하며, 프론트엔드는 React + TypeScript로 구축됩니다. Hermes를 통합하려면 아키텍처 일관성을 유지하면서 Hermes를 ClaudeCode 및 OpenCode와 동등한 최우선 실행자로 만들어야 했습니다. 말로는 간단해 보이지만 구현은 항상 진짜 이야기를 합니다.

이 글에서는 HagiCode에 Hermes Agent를 통합한 실무 경험을 공유하며, 비슷한 요구 사항에 직면한 팀에 유용한 참고 자료가 되기를 바랍니다. 한 번 함정에 빠졌다면 다른 사람이 같은 함정에 빠지게 할 이유가 없으니까요.

## HagiCode 소개

이 글에서 설명하는 해결책은 [HagiCode](https://hagicode.com) 프로젝트에서의 실무 작업에서 나온 것입니다. HagiCode는 AI 기반 코딩 지원 플랫폼으로 여러 AI Provider에 대한 통합 접근 및 관리를 지원합니다. Hermes Agent 통합 과정에서 새로운 Agent 유형이 기존 시스템에 매끄럽게 연결될 수 있도록 범용 Provider 추상화 레이어를 설계했습니다.

HagiCode에 관심이 있으시다면 [GitHub](https://github.com/HagiCode-org/site)를 방문하여 자세한 정보를 확인해 보세요. 관심을 가지는 사람이 많을수록 동력이 더 강해집니다.

## 아키텍처 설계

### 계층형 설계 접근

HagiCode의 Hermes 통합은 명확한 계층형 아키텍처를 사용하며, 각 계층은 자체 책임에 집중합니다.

**백엔드 핵심 계층**
- `HermesCliProvider`: `IAIProvider` 인터페이스를 구현하여 통합 AI Provider 진입점 역할
- `HermesPlatformConfiguration`: Hermes 실행 파일 경로, 인수, 인증 및 관련 설정 관리
- `ICliProvider<HermesOptions>`: 하위 프로세스 수명 주기를 처리하는 HagiCode.Libs에서 제공하는 저수준 CLI 추상화

**전송 계층**
- `StdioAcpTransport`: 표준 입력 및 출력을 통해 Hermes ACP 하위 프로세스와 통신
- ACP 프로토콜 메서드: `initialize`, `authenticate`, `session/new`, `session/prompt`

**런타임 계층**
- `HermesGrain`: 분산 세션 실행을 처리하는 Orleans Grain 구현
- `CliAcpSessionPool`: ACP 하위 프로세스를 재사용하여 빈번한 시작 오버헤드를 피하는 세션 풀

**프론트엔드 계층**
- `ExecutorAvatar`: Hermes 시각적 식별 및 아이콘
- `executorTypeAdapter`: Provider 유형 매핑 로직
- SignalR 실시간 메시징: 메시지 스트림 전체에서 Hermes 식별 일관성 유지

이 계층형 설계를 통해 각 계층이 독립적으로 발전할 수 있습니다. 예를 들어 나중에 WebSocket과 같은 새로운 전송 메커니즘을 추가하려면 전송 계층만 수정하면 됩니다. 하나의 전송이 변경된다고 해서 전체 시스템을 뒤집을 필요는 없습니다.

### 통합 인터페이스 추상화

모든 AI Provider는 `IAIProvider` 인터페이스를 구현하며, 이것은 HagiCode 아키텍처의 핵심 설계 선택 중 하나입니다.

```csharp
public interface IAIProvider
{
    string Name { get; }
    ProviderCapabilities Capabilities { get; }

    IAsyncEnumerable<AIStreamingChunk> StreamAsync(
        AIRequest request,
        CancellationToken cancellationToken = default);

    Task<AIResponse> ExecuteAsync(
        AIRequest request,
        CancellationToken cancellationToken = default);
}
```

`HermesCliProvider`는 이 인터페이스를 구현하며 `ClaudeCodeProvider`, `OpenCodeProvider` 등과 동등한 위치에 있습니다. 이 설계의 이점은 다음과 같습니다.

1. **대체 가능성**: Provider 전환이 상위 비즈니스 로직에 영향을 주지 않습니다
2. **테스트 가능성**: Provider를 쉽게 모킹하여 단위 테스트를 수행할 수 있습니다
3. **확장성**: 새 Provider를 추가하려면 인터페이스를 구현하기만 하면 됩니다

결국 인터페이스는 규칙과 비슷합니다. 규칙이 자리를 잡으면 모두가 조화롭게 공존하고 각자의 강점을 발휘하며 서로를 방해하지 않을 수 있습니다. 거기에는 일종의 우아함이 있습니다.

## 핵심 구현

### Provider 계층 구현

`HermesCliProvider`는 전체 통합의 핵심입니다. 단일 AI 호출을 완료하는 데 필요한 여러 구성 요소를 조정합니다.

```csharp
public sealed class HermesCliProvider : IAIProvider, IVersionedAIProvider
{
    private readonly ICliProvider<LibsHermesOptions> _provider;
    private readonly ConcurrentDictionary<string, string> _sessionBindings;

    public ProviderCapabilities Capabilities { get; } = new()
    {
        SupportsStreaming = true,
        SupportsTools = true,
        SupportsSystemMessages = true,
        SupportsArtifacts = false
    };

    public async IAsyncEnumerable<AIStreamingChunk> StreamAsync(
        AIRequest request,
        [EnumeratorCancellation] CancellationToken cancellationToken = default)
    {
        // 1. 세션 바인딩 키 해석
        var bindingKey = ResolveBindingKey(request.CessionId);

        // 2. 세션 풀을 통해 Hermes 세션을 가져오거나 생성
        var options = new HermesOptions
        {
            ExecutablePath = _platformConfiguration.ExecutablePath,
            Arguments = _platformConfiguration.Arguments,
            SessionId = _sessionBindings.TryGetValue(bindingKey, out var sessionId) ? sessionId : null,
            WorkingDirectory = request.WorkingDirectory,
            Model = request.Model
        };

        // 3. 실행 및 스트리밍 응답 수집
        await foreach (var message in _provider.ExecuteAsync(options, request.Prompt, cancellationToken))
        {
            // 4. ACP 메시지를 AIStreamingChunk로 매핑
            if (_responseMapper.TryConvertToStreamingChunk(message, out var chunk))
            {
                yield return chunk;
            }
        }
    }
}
```

여기서 몇 가지 설계 포인트가 특히 중요합니다.

1. **세션 바인딩**: `CessionId`를 사용하여 여러 요청을 동일한 Hermes 하위 프로세스에 바인딩하여 다중 턴 대화에서 컨텍스트 연속성을 유지합니다
2. **응답 매핑**: Hermes ACP 메시지 형식을 통합된 `AIStreamingChunk` 형식으로 변환합니다
3. **스트리밍 지원**: `IAsyncEnumerable`을 사용하여 진정한 스트리밍 응답을 지원합니다

세션 바인딩은 인간 관계와 약간 비슷합니다. 연결이 한 번 설정되면 이후의 통신에 컨텍스트가 있으므로 매번 처음부터 시작할 필요가 없습니다. 물론 그 관계도 유지해야 합니다.

### ACP 프로토콜 적응

Hermes는 ACP (Agent Communication Protocol)를 사용하며, 이는 전통적인 HTTP API와 다릅니다. ACP는 표준 입력 및 출력 기반의 프로토콜이며, 몇 가지 특성이 있습니다.

1. **시작 마커**: Hermes 프로세스가 시작된 후 `//ready` 마커를 출력합니다
2. **동적 인증**: 인증 방법이 고정되어 있지 않으며 프로토콜을 통해 협상해야 합니다
3. **세션 재사용**: 설정된 세션은 `SessionId`를 통해 재사용됩니다
4. **분할 응답**: 완전한 응답이 여러 `session/update` 알림으로 분할될 수 있습니다

HagiCode는 `StdioAcpTransport`를 통해 이러한 특성을 처리합니다.

```csharp
public class StdioAcpTransport
{
    public async Task InitializeAsync(CancellationToken cancellationToken)
    {
        // //ready 마커 대기
        var readyLine = await _outputReader.ReadLineAsync(cancellationToken);
        if (readyLine != "//ready")
        {
            throw new InvalidOperationException("Hermes did not send ready signal");
        }

        // initialize 요청 전송
        await SendRequestAsync(new
        {
            jsonrpc = "2.0",
            id = 1,
            method = "initialize",
            @params = new
            {
                protocolVersion = "2024-11-05",
                capabilities = new { },
                clientInfo = new { name = "HagiCode", version = "1.0.0" }
            }
        }, cancellationToken);
    }
}
```

프로토콜은 사람 간의 상호 이해와 약간 비슷합니다. 그 이해가 형성되면 통신이 훨씬 더 원활해집니다. 구축하는 데 시간이 걸릴 뿐입니다.

### 세션 풀 관리

Hermes 하위 프로세스를 자주 시작하는 것은 비용이 많이 들므로 세션 풀 메커니즘을 구현했습니다.

```csharp
services.AddSingleton(static _ =>
{
    var registry = new CliProviderPoolConfigurationRegistry();
    registry.Register("hermes", new CliPoolSettings
    {
        MaxActiveSessions = 50,
        IdleTimeout = TimeSpan.FromMinutes(10)
    });
    return registry;
});
```

핵심 세션 풀 매개변수:
- `MaxActiveSessions`: 동시성 한도를 제어하여 리소스 고갈을 방지합니다
- `IdleTimeout`: 시작 비용과 메모리 사용량의 균형을 맞추는 유휴 시간 초과

실제로 다음을 발견했습니다.
1. 유휴 시간 초과가 너무 짧으면 세션이 자주 재시작되고, 너무 길면 메모리가 계속 점유됩니다
2. 동시성 한도는 실제 부하에 맞게 조정해야 합니다. 너무 높게 설정하면 시스템이 느려질 수 있습니다
3. 세션 풀 활용률을 모니터링하여 매개변수를 제때 조정해야 합니다

이것은 삶의 많은 선택과 비슷합니다. 너무 공격적이면 문제가 생기고, 너무 보수적이면 기회를 놓칩니다. 목표는 적절한 균형을 찾는 것입니다.

## 프론트엔드 통합

### 유형 매핑

프론트엔드는 Hermes Provider를 올바르게 식별하고 해당 시각적 요소를 표시해야 합니다.

```typescript
// executorTypeAdapter.ts
export const resolveExecutorVisualTypeFromProviderType = (
  providerType: PCode_Models_AIProviderType | null | undefined
): ExecutorVisualType => {
  switch (providerType) {
    case PCode_Models_AIProviderType.HERMES_CLI:
      return 'Hermes';
    default:
      return 'Unknown';
  }
};
```

결국 아름다운 것은 아름다운 표현을 받을 자격이 있습니다. 그 아름다움이 실제로 보이도록 만드는 것은 프론트엔드 장인정신에 달려 있습니다.

### 계약 동기화

프론트엔드와 백엔드는 OpenAPI 생성을 통해 계약을 일치시킵니다. 백엔드는 `AIProviderType` 열거형을 정의합니다.

```csharp
public enum AIProviderType
{
    Unknown,
    ClaudeCode,
    OpenCode,
    HermesCli  // 새로 추가됨
}
```

프론트엔드는 OpenAPI를 통해 해당 TypeScript 유형을 생성하여 열거형 값이 일치하도록 보장합니다. 이것이 프론트엔드에 `Unknown`이 표시되는 것을 방지하는 핵심입니다.

계약은 약속과 비슷합니다. 한 번 합의하면 지켜야 합니다. 그렇지 않으면 `Unknown`과 같은 당혹스러운 상황에 처하게 됩니다.

## 설정 관리

Hermes 설정은 `appsettings.json`을 통해 관리됩니다.

```json
{
  "Providers": {
    "HermesCli": {
      "ExecutablePath": "hermes",
      "Arguments": "acp",
      "StartupTimeoutMs": 10000,
      "ClientName": "HagiCode",
      "Authentication": {
        "PreferredMethodId": "api-key",
        "MethodInfo": {
          "api-key": "your-api-key-here"
        }
      },
      "SessionDefaults": {
        "Model": "claude-sonnet-4-20250514",
        "ModeId": "default"
      }
    }
  }
}
```

이 설정 기반 설계는 유연성을 제공합니다.
- 실행 파일 경로를 재정의할 수 있어 개발 및 테스트에 편리합니다
- 시작 인수를 사용자 정의하여 다른 Hermes 버전에 맞출 수 있습니다
- 인증 정보를 구성하여 여러 인증 방법을 지원할 수 있습니다

설정은 삶의 객관식 문제와 비슷합니다. 충분한 옵션이 있으면 보통 맞는 것이 하나 있습니다. 하지만 너무 많은 옵션은 자체적인 결정 피로를 만들 수 있습니다.

## 실무 경험

### 상태 확인

신뢰할 수 있는 Provider를 구축하려면 포괄적인 상태 확인이 필요합니다.

```csharp
public async Task<ProviderTestResult> PingAsync(CancellationToken cancellationToken = default)
{
    var response = await ExecuteAsync(new AIRequest
    {
        Prompt = "Reply with exactly PONG.",
        CessionId = null,
        AllowedTools = Array.Empty<string>(),
        WorkingDirectory = ResolveWorkingDirectory(null)
    }, cancellationToken);

    var success = string.Equals(response.Content.Trim(), "PONG", StringComparison.OrdinalIgnoreCase);
    return new ProviderTestResult
    {
        ProviderName = Name,
        Success = success,
        ResponseTimeMs = stopwatch.ElapsedMilliseconds,
        ErrorMessage = success ? null : $"Unexpected Hermes ping response: '{response.Content}'."
    };
}
```

상태 확인 시 주의할 점:
1. 간단한 테스트 케이스를 사용하고 지나치게 복잡한 시나리오를 피하세요
2. 합리적인 시간 초과 값을 설정하세요
3. 응답 시간을 기록하여 성능 분석을 지원하세요

사람에게 건강 검진이 필요하듯 시스템에도 상태 확인이 필요합니다. 문제를 일찍 발견할수록 수정이 쉬워집니다.

### 검증 도구

HagiCode는 Hermes 통합 검증을 위한 전용 콘솔을 제공합니다.

```bash
# 기본 검증
HagiCode.Libs.Hermes.Console --test-provider

# 전체 검증 (저장소 분석 포함)
HagiCode.Libs.Hermes.Console --test-provider-full --repo .

# 사용자 정의 실행 파일
HagiCode.Libs.Hermes.Console --test-provider-full --executable /path/to/hermes
```

이 도구는 개발 중에 극도로 유용합니다. 통합이 올바른지 빠르게 검증할 수 있습니다. 문제가 표면화된 후에야 테스트하는 것을 아무도 원하지 않으니까요.

### 일반적인 문제 처리

**인증 실패**
- `Authentication.PreferredMethodId`가 Hermes가 실제로 지원하는 인증 방법과 일치하는지 확인하세요
- API Key 또는 Bearer Token과 같은 인증 정보 형식이 올바른지 확인하세요

**세션 시간 초과**
- `StartupTimeoutMs` 값을 늘리세요
- MCP 서버 접근 가능성을 확인하세요
- 시스템 리소스 사용률을 검토하세요

**불완전한 응답**
- `session/update` 알림과 최종 결과가 올바르게 집계되는지 확인하세요
- 스트리밍 처리의 취소 로직을 확인하세요
- 오류 처리가 완전한지 검증하세요

**프론트엔드에 Unknown 표시**
- OpenAPI 생성에 `HermesCli` 열거형 값이 이미 포함되어 있는지 확인하세요
- 유형 매핑이 올바른지 확인하세요
- 브라우저 캐시를 지우고 유형을 다시 생성하세요

문제는 항상 존재합니다. 나타났을 때 중요한 것은 당황하지 않는 것입니다. 단계별로 원인을 추적하면 대부분 해결할 수 있습니다.

### 성능 최적화 제안

1. **세션 풀 사용**: ACP 하위 프로세스를 재사용하여 시작 오버헤드를 줄입니다
2. **시간 초과 적절히 설정**: 메모리 사용량과 시작 비용의 균형을 맞추세요
3. **세션 ID 재사용**: 배치 작업에 동일한 `CessionId`를 사용하세요
4. **MCP 필요에 따라 구성**: 불필요한 도구 호출을 피하세요

성능은 일상 생활의 효율성과 비슷합니다. 잘 맞으면 적은 것으로 더 많은 것을 이루고, 잘못되면 노력은 배가되고 결과는 줄어듭니다. 그 "적절한" 지점을 찾는 데는 경험과 운이 모두 필요합니다.

## 결론

Hermes Agent를 프로덕션 시스템에 통합하려면 여러 차원의 문제를 고려해야 합니다.

1. **아키텍처**: 통합 Provider 인터페이스를 설계하고 대체 가능한 컴포넌트 아키텍처를 구현합니다
2. **프로토콜**: 시작 마커 및 동적 인증과 같은 ACP 특정 동작을 올바르게 처리합니다
3. **성능**: 세션 풀을 통해 리소스를 재사용하고 시작 비용과 메모리 사용량의 균형을 맞춥니다
4. **프론트엔드**: 계약 동기화를 보장하고 일관된 시각적 경험을 제공합니다

HagiCode의 경험은 좋은 계층형 설계와 설정 기반 구현을 통해 복잡한 Agent 시스템을 기존 아키텍처에 매끄럽게 통합할 수 있음을 보여줍니다.

이러한 원칙들은 말로 설명하면 간단해 보이지만 실제 구현은 항상 여러 종류의 문제를 도입합니다. 괜찮습니다. 문제가 해결되면 경험이 되고, 해결되지 않으면 교훈이 됩니다. 어느 쪽이든 가치가 있습니다.

아름다운 것은 소유할 필요가 없습니다. 아름다운 채로 남아 있는 한 조용히 그 아름다움을 감상하는 것만으로 충분합니다. 기술도 비슷합니다. 시스템을 더 좋게 만드는 데 도움이 된다면 특정 프레임워크나 프로토콜은 사람들이 때때로 생각하는 것보다 훨씬 덜 중요합니다.
