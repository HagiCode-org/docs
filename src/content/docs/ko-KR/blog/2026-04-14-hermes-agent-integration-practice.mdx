## Hermes Agent 통합 실무: 프로토콜에서 프로덕션까지

> HagiCode에 Hermes Agent를 통합한 전체 실무 경험을 공유합니다. ACP 프로토콜 어댑터, 세션 풀 관리, 프론트엔드-백엔드 계약 동기화 등 핵심 경험을 포함합니다.

## 배경

AI 보조 코딩 플랫폼 HagiCode를 구축하는 과정에서, 팀은 로컬에서 실행되면서도 클라우드로 확장할 수 있는 Agent 프레임워크를 통합해야 했습니다. 조사 끝에 Nous Research의 Hermes Agent가 종합 Agent의 기본 엔진으로 선택되었습니다.

사실 선정 이야기는, 어렵지도 쉽지도 않습니다. 시장에서 쓸 만한 Agent 프레임워크가 적지 않지만, Hermes의 ACP 프로토콜과 도구 시스템은 확실히 무언가가 있어서 HagiCode의 "이것도 저것도" 요구 시나리오에 딱 맞았습니다——로컬 개발, 팀 협업, 클라우드 확장. 하지만 Hermes를 진짜 프로덕션 시스템에 통합하려면 일련의 엔지니어링 문제를 해결해야 해서, 장난이 아닙니다.

HagiCode의 기술 스택은 Orleans를 기반으로 분산 시스템을 구축하고, 프론트엔드는 React + TypeScript를 사용합니다. Hermes를 통합하려면 기존 아키텍처의 통일성을 유지하면서 Hermes를 ClaudeCode, OpenCode 등과 병행하는 "1등 시민" 실행기로 만들어야 합니다. 말은 쉽지만, 실제로는 그렇게죠 뭐.

이 글에서는 HagiCode 프로젝트에서 Hermes Agent를 통합한 실무 경험을 공유합니다. 비슷한 요구에 직면한 팀에 참고가 되기를 바랍니다. 결국 밟은 함정은 다른 사람이 또 밟을 필요가 없습니다.

## HagiCode에 대하여

이 글에서 공유하는 방안은 [HagiCode](https://hagicode.com) 프로젝트의 실무 경험에서 왔습니다. HagiCode는 AI 기반의 코딩 보조 플랫폼으로, 다양한 AI Provider의 통합 접속과 관리를 지원합니다. Hermes Agent를 통합하는 과정에서 우리는 일반적인 Provider 추상화 계층을 설계해서 새로운 Agent 유형이 기존 시스템에 매끄럽게 접속할 수 있게 했습니다.

HagiCode에 관심이 있다면 [GitHub](https://github.com/HagiCode-org/site)에서 더 많은 정보를 보세요. 여러 사람이 보면 힘이 더 커질 뿐입니다.

## 아키텍처 설계

### 계층 설계 아이디어

HagiCode의 Hermes 통합은 명확한 계층 아키텍처를 사용하고, 각 계층은 각자의 역할을 담당합니다:

**백엔드 핵심 계층**
- `HermesCliProvider`: `IAIProvider` 인터페이스를 구현하고 통합 AI Provider 진입점으로 역할
- `HermesPlatformConfiguration`: Hermes 실행 파일 경로, 매개변수, 인증 등 구성 관리
- `ICliProvider<HermesOptions>`: HagiCode.Libs가 제공하는 하위 수준 CLI 추상화, 하위 프로세스 수명 주기 처리

**전송 계층**
- `StdioAcpTransport`: 표준 입출력을 통해 Hermes ACP 하위 프로세스와 통신
- ACP 프로토콜 메서드: `initialize`, `authenticate`, `session/new`, `session/prompt`

**런타임 계층**
- `HermesGrain`: Orleans Grain 구현, 분산 세션 실행 처리
- `CliAcpSessionPool`: 세션 풀, ACP 하위 프로세스 재사용, 잦은 시작 오버헤드 방지

**프론트엔드 계층**
- `ExecutorAvatar`: Hermes 시각적 식별자와 아이콘
- `executorTypeAdapter`: Provider 유형 매핑 로직
- SignalR 실시간 메시지 전달: 메시지 흐름에서 Hermes 신원의 일관성 유지

이런 계층 설계는 각 계층이 독립적으로 발전할 수 있게 합니다. 예를 들어 나중에 새로운 전송 방식(WebSocket 등)을 추가하려면 전송 계층만 수정하면 됩니다. 결국 전송 방식 하나 바꾸려고 시스템 전체를 뒤집고 싶은 사람은 없습니다. 너무 힘드니까요.

### 통합 인터페이스 추상화

모든 AI Provider는 `IAIProvider` 인터페이스를 구현합니다. 이것은 HagiCode 아키텍처의 핵심 설계입니다:

```csharp
public interface IAIProvider
{
    string Name { get; }
    ProviderCapabilities Capabilities { get; }

    IAsyncEnumerable<AIStreamingChunk> StreamAsync(
        AIRequest request,
        CancellationToken cancellationToken = default);

    Task<AIResponse> ExecuteAsync(
        AIRequest request,
        CancellationToken cancellationToken = default);
}
```

`HermesCliProvider`는 이 인터페이스를 구현하고 `ClaudeCodeProvider`, `OpenCodeProvider` 등과 동등한 위치에 있습니다. 이 설계의 장점:

1. **교체 가능성**: Provider 전환이 상위 비즈니스 로직에 영향을 주지 않음
2. **테스트 가능성**: Provider를 쉽게 Mock해서 단위 테스트 가능
3. **확장 가능성**: 새 Provider는 인터페이스만 구현하면 됨

결국 인터페이스라는 건 규칙과 같습니다. 규칙이 있어야 모두가 조화롭게 지내고, 각자 장점을 발휘하며 서로 방해하지 않습니다. 이건 일종의 아름다움이 아닐까요?

## 핵심 구현

### Provider 계층 구현

`HermesCliProvider`는 통합의 핵심으로, 각 컴포넌트를 조정해서 한 번의 AI 호출을 완성합니다:

```csharp
public sealed class HermesCliProvider : IAIProvider, IVersionedAIProvider
{
    private readonly ICliProvider<LibsHermesOptions> _provider;
    private readonly ConcurrentDictionary<string, string> _sessionBindings;

    public ProviderCapabilities Capabilities { get; } = new()
    {
        SupportsStreaming = true,
        SupportsTools = true,
        SupportsSystemMessages = true,
        SupportsArtifacts = false
    };

    public async IAsyncEnumerable<AIStreamingChunk> StreamAsync(
        AIRequest request,
        [EnumeratorCancellation] CancellationToken cancellationToken = default)
    {
        // 1. 세션 바인딩 키 해석
        var bindingKey = ResolveBindingKey(request.CessionId);

        // 2. 세션 풀을 통해 Hermes 세션을 가져오거나 생성
        var options = new HermesOptions
        {
            ExecutablePath = _platformConfiguration.ExecutablePath,
            Arguments = _platformConfiguration.Arguments,
            SessionId = _sessionBindings.TryGetValue(bindingKey, out var sessionId) ? sessionId : null,
            WorkingDirectory = request.WorkingDirectory,
            Model = request.Model
        };

        // 3. 실행하고 스트리밍 응답 수집
        await foreach (var message in _provider.ExecuteAsync(options, request.Prompt, cancellationToken))
        {
            // 4. ACP 메시지를 AIStreamingChunk로 매핑
            if (_responseMapper.TryConvertToStreamingChunk(message, out var chunk))
            {
                yield return chunk;
            }
        }
    }
}
```

여기 몇 가지 핵심 설계 포인트가 있습니다:

1. **세션 바인딩**: `CessionId`를 통해 여러 요청을 같은 Hermes 하위 프로세스에 바인딩해서 다중 라운드 대화의 컨텍스트 연속성 구현
2. **응답 매핑**: Hermes ACP 메시지 형식을 통합 `AIStreamingChunk` 형식으로 변환
3. **스트리밍 처리**: `IAsyncEnumerable`을 사용해서 진짜 스트리밍 응답 지원

사실 세션 바인딩은 사람 사이의 관계와 같습니다. 일단 연결이建立되면 후속 교류에는 컨텍스트가 있어서 매번 처음부터 시작할 필요가 없습니다. 다만 이 관계를 잘 유지해야 합니다. 그렇지 않으면 끊어지면 끊어진 것입니다.

### ACP 프로토콜 어댑터

Hermes는 ACP(Agent Communication Protocol) 프로토콜을 사용합니다. 전통적인 HTTP API와 다릅니다. ACP는 표준 입출력 기반의 프로토콜로 몇 가지 특징이 있습니다:

1. **시작 마커**: Hermes 프로세스가 시작하면 `//ready` 마커를 출력
2. **동적 인증**: 인증 메서드가 고정되어 있지 않고 프로토콜 협상을 통해 필요
3. **세션 재사용**: `SessionId`를 통해 이미建立된 세션 재사용
4. **응답 분산**: 완전한 응답이 여러 `session/update` 알림에 분산될 수 있음

HagiCode는 `StdioAcpTransport`를 통해 이런 특징을 처리합니다:

```csharp
public class StdioAcpTransport
{
    public async Task InitializeAsync(CancellationToken cancellationToken)
    {
        // //ready 마커 대기
        var readyLine = await _outputReader.ReadLineAsync(cancellationToken);
        if (readyLine != "//ready")
        {
            throw new InvalidOperationException("Hermes did not send ready signal");
        }

        // initialize 요청 전송
        await SendRequestAsync(new
        {
            jsonrpc = "2.0",
            id = 1,
            method = "initialize",
            @params = new
            {
                protocolVersion = "2024-11-05",
                capabilities = new { },
                clientInfo = new { name = "HagiCode", version = "1.0.0" }
            }
        }, cancellationToken);
    }
}
```

프로토콜이라는 건 사람 사이의 묵默과 같습니다. 묵默이 있으면 소통이 훨씬顺畅해집니다. 다만 묵默을建立하는 데 시간이 필요하고, 적응은 누구나 피할 수 없습니다.

### 세션 풀 관리

Hermes 하위 프로세스를 자주 시작하는 오버헤드가 커서 세션 풀 메커니즘을 구현했습니다:

```csharp
services.AddSingleton(static _ =>
{
    var registry = new CliProviderPoolConfigurationRegistry();
    registry.Register("hermes", new CliPoolSettings
    {
        MaxActiveSessions = 50,
        IdleTimeout = TimeSpan.FromMinutes(10)
    });
    return registry;
});
```

세션 풀의 핵심 매개변수:
- `MaxActiveSessions`: 동시성 상한 제어, 자원 고갈 방지
- `IdleTimeout`: 유휴 시간 초과, 시작 비용과 메모리 점유의 균형

실무에서 발견한 점:
1. 유휴 시간 초과를 너무 짧게 설정하면 잦은 재시작이 발생하고, 너무 길게 설정하면 메모리를 점유
2. 동시성 상한은 실제 부하에 따라 조정해야 하고, 너무 크면 시스템 버벅거릴 수 있음
3. 세션 풀 사용 상황을 모니터링해서 적시에 매개변수 조정 필요

이건 인생의 많은 선택과 같습니다. 너무 급진하면 문제가 생기고, 너무 보수면 기회를 놓칩니다. 균형점을 찾을 뿐입니다.

## 프론트엔드 통합

### 유형 매핑

프론트엔드는 Hermes Provider를 정확히 식별하고 해당 시각 요소를 표시해야 합니다:

```typescript
// executorTypeAdapter.ts
export const resolveExecutorVisualTypeFromProviderType = (
  providerType: PCode_Models_AIProviderType | null | undefined
): ExecutorVisualType => {
  switch (providerType) {
    case PCode_Models_AIProviderType.HERMES_CLI:
      return 'Hermes';
    default:
      return 'Unknown';
  }
};
```

결국 아름다운 것도 아름다운 표현이 있어야 합니다. 다만 이 아름다움이 사람들에게 보이려면 프론트엔드 동료들의 노력이 필요합니다.

### 계약 동기화

프론트엔드와 백엔드는 OpenAPI 생성을 통해 계약 일관성을 유지합니다. 백엔드는 `AIProviderType` 열거형을 정의합니다:

```csharp
public enum AIProviderType
{
    Unknown,
    ClaudeCode,
    OpenCode,
    HermesCli  // 새로 추가
}
```

프론트엔드는 OpenAPI 생성을 통해 해당 TypeScript 유형을 만들고 열거값 일관성을 보장합니다. 이건 프론트엔드에 "Unknown"이 표시되지 않게 하는 핵심입니다.

계약이란 건 약속과 같습니다. 말한 대로 해야지, 그렇지 않으면 "Unknown" 같은 곤란한 상황이 생깁니다.

## 구성 관리

Hermes의 구성은 `appsettings.json`을 통해 관리합니다:

```json
{
  "Providers": {
    "HermesCli": {
      "ExecutablePath": "hermes",
      "Arguments": "acp",
      "StartupTimeoutMs": 10000,
      "ClientName": "HagiCode",
      "Authentication": {
        "PreferredMethodId": "api-key",
        "MethodInfo": {
          "api-key": "your-api-key-here"
        }
      },
      "SessionDefaults": {
        "Model": "claude-sonnet-4-20250514",
        "ModeId": "default"
      }
    }
  }
}
```

이런 구성 주도 설계는 유연성을 가져옵니다:
- 실행 파일 경로를 덮어써서 개발 테스트에 편리함
- 시작 매개변수를 사용자 정의해서 다른 버전의 Hermes에 적용
- 인증 정보를 구성해서 다양한 인증 방식 지원

구성이란 건 인생의 선택 문제와 같습니다. 충분한 옵션을 주면 자기에게 맞는 것을 찾을 수 있습니다. 다만 때로는 옵션이 너무 많아서 선택 장애가 생기기도 합니다.

## 실무 경험

### 건강 검사

신뢰할 수 있는 Provider를 구현하려면 완비된 건강 검사가 필요합니다:

```csharp
public async Task<ProviderTestResult> PingAsync(CancellationToken cancellationToken = default)
{
    var response = await ExecuteAsync(new AIRequest
    {
        Prompt = "Reply with exactly PONG.",
        CessionId = null,
        AllowedTools = Array.Empty<string>(),
        WorkingDirectory = ResolveWorkingDirectory(null)
    }, cancellationToken);

    var success = string.Equals(response.Content.Trim(), "PONG", StringComparison.OrdinalIgnoreCase);
    return new ProviderTestResult
    {
        ProviderName = Name,
        Success = success,
        ResponseTimeMs = stopwatch.ElapsedMilliseconds,
        ErrorMessage = success ? null : $"Unexpected Hermes ping response: '{response.Content}'."
    };
}
```

건강 검사 시 주의해야 할 점:
1. 간단한 테스트 케이스 사용, 복잡한 시나리오 피하기
2. 합리적인 시간 초과 설정
3. 응답 시간 기록, 성능 분석에 편리함

사람에게 건강 검진이 필요하듯 시스템에도 건강 검사가 필요합니다. 일찍 발견하고 일찍 치료해야 나중에 큰 문제가 생기지 않습니다.

### 검증 도구

HagiCode는 Hermes 통합 검증을 위한 전용 콘솔을 제공합니다:

```bash
# 기본 검증
HagiCode.Libs.Hermes.Console --test-provider

# 전체 제품군(창고 분석 포함)
HagiCode.Libs.Hermes.Console --test-provider-full --repo .

# 사용자 정의 실행 파일
HagiCode.Libs.Hermes.Console --test-provider-full --executable /path/to/hermes
```

이 도구는 개발 과정에서 매우 유용해서 통합이 올바른지 빠르게 검증할 수 있습니다. 결국 문제를 발견했을 때 테스트를 생각하고 싶은 사람은 없습니다.

### 일반적인 문제 처리

**인증 실패**
- `Authentication.PreferredMethodId`와 Hermes가 실제로 지원하는 인증 메서드가 일치하는지 확인
- 인증 정보 형식이 올바른지 확인(API Key, Bearer Token 등)

**세션 시간 초과**
- `StartupTimeoutMs` 값 증가
- MCP 서버 접근 가능성 확인
- 시스템 자원 사용 상황 확인

**응답 불완전**
- `session/update` 알림과 최종 결과를 올바르게 집계하는지 확인
- 스트리밍 처리 취소 로직 확인
- 오류 처리가 완전한지 확인

**프론트엔드 Unknown 표시**
- OpenAPI 생성에 `HermesCli` 열거값이 포함되어 있는지 확인
- 유형 매핑이 올바른지 확인
- 브라우저 캐시를 지우고 유형 재생성 확인

문제는 항상 있습니다. 다만 문제에 직면했을 때 당황하지 말고 천천히 원인을 찾으면 해결할 수 있습니다. 결국 방법은困难보다 많습니다.

### 성능 최적화 제안

1. **세션 풀 사용**: ACP 하위 프로세스 재사용, 시작 오버헤드 감소
2. **합리적인 시간 초과 설정**: 메모리와 시작 비용의 균형
3. **세션 ID 재사용**: 일괄 작업은 같은 `CessionId` 사용
4. **필요에 따른 MCP 구성**: 불필요한 도구 호출 방지

성능이란 건 생활의 효율성과 같습니다. 잘하면事半功倍, 잘못하면事倍功半. 다만 그 "옳은" 점을 찾으려면 경험과 운이 필요합니다.

## 요약

프로덕션 시스템에 Hermes Agent를 통합하려면 여러 층면의 문제를 고려해야 합니다:

1. **아키텍처 층면**: 통합 Provider 인터페이스 설계, 교체 가능한 컴포넌트 아키텍처 구현
2. **프로토콜 층면**: ACP 프로토콜의 특수성을 올바르게 처리(시작 마커, 동적 인증 등)
3. **성능 층면**: 세션 풀을 통해 자원 재사용, 시작 비용과 메모리 점유의 균형
4. **프론트엔드 층면**: 계약 동기화 보장, 일관된 시각 경험 제공

HagiCode의 실무는 좋은 계층 설계와 구성 주도를 통해 복잡한 Agent 시스템을 기존 아키텍처에 매끄럽게 통합할 수 있음을 보여줍니다.

사실 이런 원리들은 말로는 다 간단합니다. 다만 진짜로 실행할 때 갖가지 문제에 직면합니다. 상관없습니다. 문제가 해결되면 경험이고, 해결되지 않으면 교훈입니다. 둘 다 가치 있는 것입니다.

아름다운 것 또는 사람은 반드시 점유할 필요가 없습니다. 그녀가 여전히 아름다우면 자신이 그녀의 아름다움을 잘 보면 됩니다. 기술도 마찬가지입니다. 시스템을 더 낫게 만들 수 있다면 어떤 프레임워크, 어떤 프로토콜을 사용하는지는 사실 그렇게 중요하지 않습니다......
