## 점진적 공개: "Less is More" 개념으로 AI 제품의 HCI 개선하기

> AI 제품 설계에서 사용자 입력의 품질은 종종 출력의 품질을 결정합니다. 이 문서는 HagiCode 프로젝트에서 실천한 "점진적 공개" 인터랙션 솔루션을 공유합니다. 단계적 가이드, 스마트 자동 완성 및 즉시 피드백을 통해 사용자의 간단하고 모호한 입력을 구조화된 기술 제안으로 변환하여 HCI 효율성을 크게 향상시킵니다.

## 배경

AI 제품을 만드는 분들이라면 다음과 같은 상황을 겪어보셨을 것입니다. 사용자가 애플리케이션을 열고 흥미진진하게 요구사항 한 줄을 입력했는데, AI가 반환한 내용이 전혀 상관없는 것입니다. AI가 똑똑하지 않은 것이 아니라, 사용자가 제공한 정보가 너무 적습니다.毕竟 속마음을 추측하는 일은 누구도 잘할 수 없습니다.

이러한 현상은 우리가 HagiCode를 개발하는 과정에서 특히 두드러졌습니다. HagiCode는 AI 기반 코드 어시스턴트로, 사용자가 자연어로 요구사항을 설명하여 기술 제안과 대화를 생성합니다. 하지만 실제 사용에서 우리는 사용자 입력에 다음과 같은 문제가 있다는 것을 발견했습니다.

- **입력 품질의 편차**: 일부 사용자는 "로그인 최적화", "버그 수정"과 같은 몇 글자만 입력하고 필요한 컨텍스트가 부족합니다.
- **기술 용어의 불일치**: 다른 사용자가 같은 일을 다른 단어로 표현합니다. 일부는 "프론트엔드"라고 하고 일부는 "FE"라고 합니다.
- **구조화된 정보 부족**: 프로젝트 배경, 저장소 범위, 영향 범위와 같은 핵심 정보가 없습니다.
- **반복적인 문제**: 동일한 유형의 요구사항이 반복적으로 나타나며 매번 처음부터 설명해야 합니다.

이러한 문제로 인한 직접적인 결과는 AI 이해의 어려움, 생성된 제안의 품질 불안정, 사용자 경험 저하입니다. 사용자는 "이 AI는 별로네"라고 생각하고, 우리도 억울합니다. 한 문장만 주고 무엇을 원하는지 어떻게 추측하냐고요?

사실 방법이 없습니다. 사람 간의 이해에도 시간이 필요한데, 하물며 기계이겠습니까?

이러한 문제점들을 해결하기 위해 우리는 대담한 결정을 내렸습니다. "점진적 공개" 설계 개념을 도입하여 HCI를 개선합니다. 이 결정이 가져온 변화는 상상보다 훨씬 클 수 있지만, 당시에는 그렇게 효과적일 줄은 몰랐습니다.

## HagiCode에 대해

이 문서에서 공유하는 솔루션은 [HagiCode](https://hagicode.com) 프로젝트에서의 실천 경험에서 비롯되었습니다. HagiCode는 오픈소스 AI 코드 어시스턴트 프로젝트로, 자연어 인터랙션을 통해 개발자가 코드 작성, 기술 제안 생성, 코드 검토 등의 작업을 완료하도록 돕는 것을 목표로 합니다. 프로젝트 주소: [github.com/HagiCode-org/site](https://github.com/HagiCode-org/site).

이 점진적 공개 솔루션은 실제 개발 과정에서 여러 번의 반복과 최적화를 거쳐 요약된 것입니다. 이 솔루션이 가치 있다고 생각하신다면 우리의 엔지니어링 역량이 꽤 괜찮다는 뜻입니다. 그렇다면 HagiCode 자체도 주목할 만합니다.毕竟 좋은 것은 공유할 가치가 있습니다.

## 점진적 공개란 무엇인가

"점진적 공개"(Progressive Disclosure)는 HCI(인간-컴퓨터 상호작용) 분야에서 유래된 설계 원칙으로, 핵심 아이디어는 매우 간단합니다. **모든 정보와 옵션을 한 번에 사용자에게 표시하지 않고, 사용자의 작업과 요구에 따라 필요한 내용을 점진적으로 표시합니다**.

이 원칙은 AI 제품에 특히 적합합니다. AI 인터랙션은 본질적으로 점진적입니다. 사용자가 조금 말하면 AI가 조금 이해하고, 그런 다음 조금 보충하고, 더 많이 이해합니다. 사람 간의 의사소통과 같습니다. 천천히 해야 합니다.毕竟 첫 만남에서 마음을 다 열어 보이는 사람은 없습니다.

HagiCode 시나리오에서 우리는 네 가지 측면에서 점진적 공개를 구현했습니다.

### 1. 설명 최적화 메커니즘: AI가 말을 명확하게 해드립니다

사용자가 간단한 설명을 입력할 때 우리는 AI가 직접 이하게 하지 않고 먼저 "설명 최적화" 프로세스를 트리거합니다. 이 프로세스의 핵심은 "구조화된 출력"입니다. 사용자의 자유 텍스트를 표준 형식으로 변환합니다. 마치 흩어진 진주를 목걸이로 엮는 것과 같습니다. 그렇게 보면 그렇게 복잡하지 않습니다.

최적화된 설명에는 다음 표준 섹션이 포함되어야 합니다.

- **배경**: 문제 배경 및 컨텍스트
- **분석**: 기술 분석 및 사고 과정
- **해결**: 솔루션 및 구현 단계
- **실습**: 실제 코드 예제 및 주의 사항

동시에 우리는 자동으로 Markdown 테이블을 생성하여 대상 저장소, 경로, 편집 권한 등의 정보를 표시하고 AI의 후속 작업을 용이하게 합니다.毕竟 명확한 목차가 있으면 물건을 찾기도 편합니다.

다음은 실제 코드 구현입니다.

```csharp
// ProposalDescriptionMemoryService.cs의 핵심 메서드
public async Task<string> OptimizeDescriptionAsync(
    string title,
    string description,
    string locale = "zh-CN",
    DescriptionOptimizationMemoryContext? memoryContext = null,
    CancellationToken cancellationToken = default)
{
    // 쿼리 매개변수 빌드
    var queryContext = BuildQueryContext(title, description);

    // 기록 컨텍스트 검색
    var memoryContext = await RetrieveHistoricalContextAsync(queryContext, cancellationToken);

    // 구조화된 프롬프트 생성
    var prompt = await BuildOptimizationPromptAsync(
        title,
        description,
        memoryContext,
        cancellationToken);

    // AI를 호출하여 최적화
    return await _aiService.CompleteAsync(prompt, cancellationToken);
}
```

이 프로세스의 핵심은 "기억 주입"입니다. 우리는 프로젝트 관행, 유사 사례, 부정적 패턴 등의 기록 컨텍스트를 프롬프트에 주입하여 AI가 최적화할 때 과거 경험을 참고할 수 있도록 합니다.毕竟 한 번 배우면 한 번 자라나고, 과거의 경험을 헛되이 낭비해서는 안 되지 않습니까?

**주의 사항**:
- 현재 입력이 기록보다 우선되도록 하여 사용자가 명시적으로 지정한 정보가 덮어쓰이지 않도록 합니다.
- HagIndex 참조는 사실 출처로 사용해야 하며 기록 사례에 의해 수정되어서는 안 됩니다.
- 낮은 신뢰도의 수정 제안은 강력한 제약 조건으로 주입해서는 안 됩니다.

### 2. 음성 입력 기능: 말하기가 입력보다 더 자연스럽습니다

텍스트 입력 외에도 우리는 음성 입력을 지원합니다. 이는 복잡한 요구사항을 설명할 때 특히 유용합니다. 생각해 보세요, 기술 요구사항을 입력하는 데 몇 분이 걸릴 수 있지만 말하는 데는 수십 초면 됩니다.毕竟 입은 항상 손보다 빠릅니다.

음성 입력 설계의 핵심은 "상태 관리"입니다. 사용자는 현재 시스템이 어떤 상태에 있는지 명확히 알아야 합니다. 우리는 다음과 같은 몇 가지 상태를 정의했습니다.

- **유휴**: 시스템이 준비되었으며 녹음을 시작할 수 있습니다.
- **업스트림 대기 중**: 백엔드 서비스에 연결 중입니다.
- **녹음 중**: 사용자 음성을 녹음 중입니다.
- **처리 중**: 음성을 텍스트로 변환 중입니다.
- **오류**: 오류가 발생했으며 사용자가 처리해야 합니다.

프론트엔드의 상태 모델은 대략 다음과 같습니다.

```typescript
interface VoiceInputState {
  status: 'idle' | 'waiting-upstream' | 'recording' | 'processing' | 'error';
  duration: number;
  error?: string;
  deletedSet: Set<string>; // 삭제된 결과의 지문 집합
}

// 녹음 시작 시 상태 전환
const handleVoiceInputStart = async () => {
  // 먼저 대기 상태로 들어가 로딩 애니메이션 표시
  setState({ status: 'waiting-upstream' });

  // 백엔드 준비 확인 대기
  const isReady = await waitForBackendReady();
  if (!isReady) {
    setState({ status: 'error', error: '백엔드 서비스가 준비되지 않았습니다' });
    return;
  }

  // 녹음 시작
  setState({ status: 'recording', startTime: Date.now() });
};

// 인식 결과 처리
const handleRecognitionResult = (result: RecognitionResult) => {
  const fingerprint = normalizeFingerprint(result.text);

  // 삭제되었는지 확인
  if (state.deletedSet.has(fingerprint)) {
    return; // 삭제된 콘텐츠 건너뛰기
  }

  // 텍스트 상자에 결과 병합
  appendResult(result);
};
```

여기에 세부 사항이 있습니다. 우리는 "지문 집합"을 사용하여 삭제 동기화를 관리합니다. 음성 인식이 여러 결과를 반환할 때 사용자는 그중 일부를 삭제할 수 있습니다. 우리는 삭제된 콘텐츠의 지문을 저장하고 나중에 동일한 콘텐츠가 나타나면 자동으로 건너뜁니다. 이는 어떤 음식을 좋아하지 않는지 기억하는 것과 같습니다. 다음에는 다시 주문하지 않을 것입니다.毕竟 아무도 같은 문제로 두 번 괴롭힘받기를 원하지 않습니다.

### 3. 프롬프트 관리 시스템: AI의 "뇌"를 외부에 배치합니다

HagiCode에는 유연한 프롬프트 관리 시스템이 있으며 모든 프롬프트는 파일 형태로 저장됩니다.

```
prompts/
├── metadata/
│   ├── optimize-description.zh-CN.json
│   └── optimize-description.en-US.json
└── templates/
    ├── optimize-description.zh-CN.hbs
    └── optimize-description.en-US.hbs
```

각 프롬프트는 두 부분으로 구성됩니다.
- **메타데이터 파일(.json)**: 프롬프트의 시나리오, 버전, 매개변수 등의 정보를 정의합니다.
- **템플릿 파일(.hbs)**: Handlebars 구문을 사용하는 실제 프롬프트 콘텐츠

메타데이터 파일의 형식은 다음과 같습니다.

```json
{
  "scenario": "optimize-description",
  "locale": "zh-CN",
  "version": "1.0.0",
  "syntax": "handlebars",
  "syntaxVersion": "1.0",
  "parameters": [
    {
      "name": "title",
      "type": "string",
      "required": true,
      "description": "제안 제목"
    },
    {
      "name": "description",
      "type": "string",
      "required": true,
      "description": "원래 설명"
    }
  ],
  "author": "HagiCode Team",
  "description": "사용자 입력의 기술 제안 설명 최적화",
  "lastModified": "2026-04-05",
  "tags": ["optimization", "nlp"]
}
```

템플릿 파일은 Handlebars 구문을 사용하며 매개변수 주입을 지원합니다.

```handlebars
당신은 기술 제안 전문가입니다.

<task>
다음 정보를 기반으로 구조화된 기술 제안 설명을 생성하세요.
</task>

<input>
<title>{{title}}</title>
<description>{{description}}</description>
{{#if memoryContext}}
<memory_context>
{{memoryContext}}
</memory_context>
{{/if}}
</input>

<output_format>
## 배경
[문제 배경 및 컨텍스트를 설명하세요. 프로젝트 정보, 저장소 범위 등 포함]

## 분석
[기술 분석 및 사고 과정, 이 변경이 필요한 이유 설명]

## 해결
[솔루션 및 구현 단계, 핵심 코드 위치 나열]

## 실습
[실제 코드 예제 및 주의 사항]
</output_format>
```

이 설계의 장점은 다음과 같습니다.
- 프롬프트는 코드와 같이 버전 관리할 수 있습니다.
- 다국어를 지원하며 사용자偏好에 따라 자동으로 전환됩니다.
- 매개변수화된 설계로 동적으로 컨텍스트를 주입할 수 있습니다.
- 시작 시 완전성을 검증하여 런타임 오류를 방지합니다.

毕竟 머릿속에 있는 것을 적어두지 않으면 언제 잊어버릴지 모르고, 나중에 후회하는 것보다 처음부터 기록을 잘하는 것이 낫습니다.

### 4. 점진적 마법사: 복잡한 작업을 작은 단계로 나눕니다

복잡한 작업(예: 첫 설치 구성)의 경우 우리는 다단계 마법사 설계를 사용했습니다. 각 단계는 필요한 정보만 요청하고 명확한 진행 표시를 제공합니다. 삶도 그렇습니다. 한입에 뚱뚹해질 수 없고 단계별로 하는 것이 더 안전합니다.

마법사의 상태 모델:

```typescript
interface WizardState {
  currentStep: number;           // 0-3, 4단계에 해당
  steps: WizardStep[];
  canGoNext: boolean;
  canGoBack: boolean;
  isLoading: boolean;
  error: string | null;
}

interface WizardStep {
  id: number;
  title: string;
  description: string;
  completed: boolean;
}

// 단계 탐색 로직
const goToNextStep = () => {
  if (wizardState.currentStep < wizardState.steps.length - 1) {
    // 현재 단계의 입력 유효성 검사
    if (validateCurrentStep()) {
      wizardState.currentStep++;
      wizardState.steps[wizardState.currentStep - 1].completed = true;
    }
  }
};

const goToPreviousStep = () => {
  if (wizardState.currentStep > 0) {
    wizardState.currentStep--;
  }
};
```

각 단계에는 독립적인 유효성 검사 논리가 있으며 완료된 단계에는 명확한 시각적 표시가 있습니다. 취소 작업은 확인 대화 상자를 표시하여 사용자의 실수로 진행 상황이 손실되는 것을 방지합니다.毕竟 잘못된 길로 가면 돌아갈 수 있지만 길을 모두 철거하면 정말 방법이 없습니다.

## 요약

HagiCode의 점진적 공개 실천을 되돌아보면 몇 가지 핵심 원칙을 요약할 수 있습니다.

1. **단계적 가이드**: 복잡한 작업을 작은 단계로 나누고 각 단계는 필요한 정보만 요청합니다.
2. **스마트 자동 완성**: 기록 컨텍스트와 프로젝트 지식을 활용하여 자동으로 정보를 완성합니다.
3. **즉시 피드백**: 각 작업에는 명확한 시각적 피드백과 상태 표시가 있습니다.
4. **오류 처리 메커니즘**: 사용자가 실행 취소 및 재설정할 수 있으며 오류로 인한 되돌릴 수 없는 손실을 방지합니다.
5. **다양한 입력**: 텍스트, 음성 등 다양한 입력 방법을 지원합니다.

HagiCode에서 이 솔루션의 실제 효과는 사용자 입력의 평균 길이가 20자 미만에서 구조화된 200-300자로 증가했고 AI가 생성한 제안의 품질이 크게 향상되었으며 사용자 만족도도 따라서 상승했습니다.

사실 이것은 이상한 일이 아닙니다.毕竟 제공하는 정보가 많을수록 AI가 더 정확하게 이해하고 반환된 결과가 자연스럽게 더 좋아집니다. 이는 사람 간의 의사소통과 다르지 않습니다.

AI 관련 제품을 만드는 경우 이러한 경험이 영감을 주기를 바랍니다. 기억하세요. 사용자는 정보를 주고 싶지 않은 것이 아니라 아직 올바른 질문을 하지 않았을 뿐입니다. 점진적 공개의 핵심은 질문의 최적의 타이밍과 방법을 찾는 것이지만 이 타이밍과 방법은 약간의 인내로摸索해야 합니다.
