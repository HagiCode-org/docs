## Docusaurus 3.x에서 Astro 5.x로: HagiCode 사이트 마이그레이션 실전 복기

> 이 문서에서는 HagiCode 공식 웹사이트를 Docusaurus 3.x에서 Astro 5.x로 마이그레이션한 전체 과정을 되돌아봅니다. Astro의 Islands 아키텍처를 통해 성능 병목을 해결하고 기존 React 컴포넌트 자산을 유지하여 빌드 속도와 로딩 성능을 동시에 향상시키는 방법을 심층적으로 다룹니다.


## 배경

2026년 1월, 우리는 HagiCode 공식 사이트에 대해 "심장 이식 수술"을 수행했습니다. 즉, 핵심 프레임워크를 Docusaurus 3.x에서 Astro 5.x로 전면 마이그레이션했습니다. 이것은 충동적인 대규모 리팩토링이 아니라 신중한 기술적 선택이었습니다.

마이그레이션 전, 우리 사이트는 기능이 완비되었지만 점차 일부 "부자병"을 드러내기 시작했습니다. 빌드 산물 볼륨이 비대해지고 JavaScript 부하가 과도하며, 복잡한 문서 페이지에서 페이지 로딩 속도가 이상적이지 않았습니다. AI 코드 어시스턴트 프로젝트인 HagiCode는 문서와 기능 소개를 자주 업데이트해야 하므로 빌드 효율성이 출시 속도에 직접적인 영향을 미칩니다. 동시에 더 많은 개발자가 이 프로젝트를 발견할 수 있도록 사이트가 검색 엔진(SEO)에 더 친화적이기를 희망했습니다.

이러한 문제점을 해결하기 위해 우리는 대담한 결정을 내렸습니다. 전체 빌드 시스템을 뒤엎고 Astro로 마이그레이션했습니다. 이 결정이 가져온 변화는 상상보다 훨씬 클 수 있습니다. 나중에 구체적으로 설명하겠습니다.

## HagiCode 소개

이 문서에서 공유하는 사이트 마이그레이션 방안은 [HagiCode](https://github.com/HagiCode-org/site) 프로젝트에서의 실제 경험에서 비롯되었습니다.

HagiCode는 개발 효율성 향상에 전념하는 AI 코드 어시스턴트로, 핵심 기능 반복뿐만 아니라 개발자 경험도 중요하게 생각합니다. 이번 사이트 리팩토링도 사용자가 문서와 공식 웹사이트를 탐색할 때 극한의 로딩 속도를 얻을 수 있도록 하기 위함입니다.

## 왜 성숙한 Docusaurus를 포기했을까요?

React 생태계에서 Docusaurus는 항상 문서 사이트의 "표준 답변"이었습니다. 설치 즉시 사용 가능하고 플러그인이 풍부하며 커뮤니티가 활발합니다. 하지만 HagiCode 기능이 증가함에 따라 우리도 그 한계를 느꼈습니다.

1.  **성능 병목**: Docusaurus는 본질적으로 React SPA(단일 페이지 애플리케이션)입니다. 순수 정적 페이지를 작성하더라도 클라이언트는 React 런타임을 로드하고 수화(hydration)를 수행해야 하므로 간단한 문서 페이지에는 너무 무겁습니다.
2.  **리소스 볼륨**: 페이지 콘텐츠가 적더라도 패키징된 JS 볼륨은 상대적으로 고정되어 있어 모바일 사용자와 네트워크가 열악한 환경에 충분히 친화적이지 않습니다.
3.  **유연성 부족**: 확장은 가능하지만 빌드 프로세스 사용자 정의에 대해 우리는 더 하위 수준의 제어권을 갖고 싶었습니다.

Astro의 등장으로 이러한 문제들이 해결되었습니다. 새로운 "섬 아키텍처"(Islands Architecture)를 제공합니다. 기본적으로 Astro는 JavaScript가 없는 정적 HTML을 생성하며, 상호작용이 필요한 컴포넌트만 "활성화"되어 JS를 로드합니다. 즉, 사이트의 대부분 콘텐츠가 순수 HTML이므로 속도가 매우 빠릅니다.

## 마이그레이션 핵심 전략: 아키텍처의 원활한 전환

마이그레이션은 단순한 복사 붙여넣기가 아니라 사고방식의 전환입니다. 우리는 Docusaurus의 "전체 React 모드"에서 Astro의 "Core + Islands" 모드로 전환했습니다.

### 1. 구성 시스템의 리팩토링

먼저 `docusaurus.config.ts`에서 `astro.config.mjs`로 전환해야 합니다. 이것은 단순한 파일 이름 변경이 아니라 라우팅과 빌드 로직의 재작성입니다.

Docusaurus에서는 모든 것이 플러그인이지만, Astro에서는 모든 것이 통합(integration)입니다. 사이트의 기본 경로, 빌드 출력 모드(정적 vs SSR) 및 리소스 압축 전략을 다시 정의해야 합니다.

**마이그레이션 전:**
```typescript
// docusaurus.config.ts
export default {
  title: 'HagiCode',
  url: 'https://hagicode.com',
  baseUrl: '/',
  // ... 더 많은 구성
};
```

**마이그레이션 후:**
```javascript
// astro.config.mjs
import { defineConfig } from 'astro/config';
import react from '@astrojs/react';

export default defineConfig({
  integrations: [react()],
  site: 'https://hagicode.com',
  base: '/',
  // 정적 리소스를 위한 최적화 구성
  build: {
    inlineStylesheets: 'auto',
  },
});
```

### 2. React 컴포넌트의 폐지와 개조

이것이 마이그레이션에서 가장 골치 아픈 부분입니다. 기존 사이트에는 많은 React 컴포넌트(예: Tabs 컴포넌트, 코드 하이라이팅, 피드백 버튼 등)가 있습니다. 모두 버리기엔 아깝고, 모두 보존하면 JS 부하가 과도해집니다.

HagiCode는 **점진적 수화(hydration)** 전략을 채택했습니다.
*   **순수 정적 컴포넌트**: 전시형 콘텐츠(예: 헤더, 푸터, 순수 텍스트 문서)의 경우 Astro 컴포넌트(`.astro` 파일)로 다시 작성하여 빌드 시 HTML로 직접 렌더링합니다.
*   **인터랙티브 섬**: 상호작용을 유지해야 하는 컴포넌트(예: 테마 전환기, Tabs 전환, 코드 블록 복사 버튼)의 경우 React 구현을 유지하고 `client:load` 또는 `client:visible` 지시문을 추가합니다.

예를 들어, 문서에서 자주 사용하는 Tabs 컴포넌트:

```jsx
// src/components/Tabs.jsx
import { useState } from 'react';
import './Tabs.css'; // 스타일 도입

export default function Tabs({ items }) {
  const [activeIndex, setActiveIndex] = useState(0);
  // ... 상태 로직
  return (
    <div className="tabs-wrapper">
      {/* 렌더링 로직 */}
    </div>
  );
}
```

Markdown에서 사용할 때 Astro에게 명확하게 알립니다. "이 컴포넌트는 JS가 필요합니다":

```markdown
// src/content/docs/example.mdx
import Tabs from '../../components/Tabs.jsx';

<!-- 컴포넌트가 뷰포트에 진입할 때만 JS 로드 -->
<Tabs client:visible items={...} />
```

이렇게 하면 뷰포트 내에 없는 인터랙티브 컴포넌트가 대역폭을 선점하지 않으므로 첫 화면 로딩 속도가 크게 최적화됩니다.

### 3. 스타일 시스템 적응: CSS Modules에서 Scoped로

Docusaurus는 기본적으로 CSS Modules를 지원하고, Astro는 Scoped CSS(`<style>` 태그 통해) 사용을 선호합니다. 두 핵심 아이디어는 모두 스타일 격리이지만 구문이 다릅니다.

HagiCode 마이그레이션에서 우리는 대부분의 복잡한 CSS Modules을 Astro의 Scoped 스타일로 분해했습니다. 이것은 사실 좋은 일입니다. `.astro` 파일에서 스타일과 템플릿을 같은 파일에 작성하므로 유지 관리가 더 직관적입니다.

**개조 전:**
```css
/* Tabs.module.css */
.wrapper { background: var(--ifm-background-color); }
```

**개조 후 (Astro Scoped):**
```html
<!-- Tabs.astro -->
<div class="tabs-wrapper">
  <slot />
</div>

<style>
  .tabs-wrapper {
    /* CSS 변수 직접 사용, 테마 적응 */
    background: var(--bg-color);
    padding: 1rem;
  }
</style>
```

동시에 Astro의 환경 인식 능력을 활용하여 전역 CSS 변수 시스템을 통일하고 다크 모드가 다른 페이지 간에 원활하게 전환되도록 보장했습니다.

## 실제 과정의 함정과 해결 방안

HagiCode의 실제 마이그레이션 과정에서 우리는 많은 함정에 빠졌습니다. 그중 가장 전형적인 몇 가지를 공유합니다.

### 1. 경로与环境 변수의 고통

HagiCode는 하위 경로 배포(예: GitHub Pages 하위 디렉터리에 배포)를 지원합니다. Docusaurus에서는 `baseUrl`을 자동으로 처리합니다. 하지만 Astro에서는 이미지 링크와 API 요청을 처리할 때 더 주의해야 합니다.

우리는 환경 변수 메커니즘을 도입하여 통합 관리합니다:

```javascript
// 빌드 스크립트에서 경로 처리
const getBasePath = () => import.meta.env.VITE_SITE_BASE || '/';
```

명심하세요. 코드에 `/`로 시작하는 경로를 하드코딩하지 마세요. 개발 환경과 프로덕션 환경, 또는 `base` 경로를 구성한 후 이는 리소스 404를 초래할 수 있습니다.

### 2. CommonJS 스크립트 호환성

이전 사이트에는 일부 Node.js 스크립트(Metrics 데이터 자동 수집, sitemap 업데이트 등용)가 있으며, CommonJS(`require`)로 작성되었습니다. Astro와 현대 빌드 도구는 ES Modules(`import`/`export`)를 전면적으로 수용합니다.

유사한 스크립트가 있는 경우 모두 ES Modules로 리팩토링하십시오. 이것은 대세이므로 조기에 수정하는 것이 좋습니다.

```javascript
// 구방식
const fs = require('fs');

// 새방식
import fs from 'fs';
```

### 3. SEO와 리디렉션 잊지 마세요

검색 엔진은 이미 HagiCode의 이전 Docusaurus 페이지를 수집했습니다. Astro로 직접 전환하면 URL 구조가 변경되어 대량의 404가 발생하고 권한이 크게 떨어집니다.

우리는 Astro에서 리디렉션 규칙을 구성했습니다:

```javascript
// astro.config.mjs
export default defineConfig({
  redirects: {
    '/docs/old-path': '/docs/new-path',
    // 이전 링크를 새 링크로 일괄 매핑
  }
});
```
또는 서버 구성 수준에서 처리합니다. 이전 링크가 새 주소로 301 리디렉션되도록 하는 것이 SEO에 매우 중요합니다.

## 요약

Docusaurus에서 Astro로 마이그레이션한 것은 HagiCode에게 단순한 프레임워크 업그레이드가 아니라 "성능 우선" 이념의 실천이었습니다.

**우리의 수확:**
*   **극한의 Lighthouse 점수**: 마이그레이션 후 HagiCode 사이트의 성능 점수가 쉽게 만점에 가까워졌습니다.
*   **더 빠른 빌드 속도**: Astro의 증분 빌드로 문서 업데이트의 게시 시간이 절반으로 단축되었습니다.
*   **유연성 유지**: Islands 아키텍처를 통해 우리는 모든 인터랙티브 기능을 희생하지 않고 필요한 곳에 여전히 React를 사용할 수 있습니다.

문서형 사이트를 유지 관리하고 있고 패키징 볼륨이나 로딩 속도로 고통받고 있다면 Astro를 시도해 보세요. 마이그레이션 과정에는 수술이 필요하지만(예: PCode의 이름을 HagiCode로 변경하고 컴포넌트를 하나씩 가져오는 등), 그 대가로 비단 같이 부드러운 사용자 경험을 얻을 수 있으므로 확실히 가치가 있습니다.

이 문서에서 공유하는 빌드 시스템은 우리가 HagiCode를 개발하는 과정에서 실제로 함정을 밟고 실제로 최적화한 방안입니다. 이 방안이 가치 있다고 생각하신다면 우리의 엔지니어링 역량이 괜찮다는 뜻입니다. 그렇다면 HagiCode 자체도 주목할 가치가 있습니다.
