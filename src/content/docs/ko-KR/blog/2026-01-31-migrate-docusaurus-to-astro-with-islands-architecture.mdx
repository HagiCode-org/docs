## Docusaurus 3.x에서 Astro 5.x로: HagiCode 사이트 마이그레이션 실전 복기

> 이 글에서는 HagiCode 공식 웹사이트를 Docusaurus 3.x에서 Astro 5.x로 완전히 마이그레이션한 전체 과정을 복기합니다. Astro의 Islands 아키텍처를 통해 성능 병목을 해결하면서 기존 React 컴포넌트 자산을 보존하여 빌드 속도와 로딩 성능을 동시에 향상시키는 방법을 심층적으로 논의합니다.


## 배경

2026년 1월, 우리는 HagiCode 공식 사이트에 "심장 이식 수술"을 실시했습니다 — 핵심 프레임워크를 Docusaurus 3.x에서 Astro 5.x로 완전히 마이그레이션했습니다. 이는 충동적인 대규모 리팩토링이 아니라 신중한 기술적 선택이었습니다.

마이그레이션 전, 사이트는 기능적으로 완비되어 있었지만 점차 몇 가지 "부자병"이 드러났습니다: 빌드 산출물의 부피가 비대하고, JavaScript 부하가 과도하며, 복잡한 문서 페이지에서 페이지 로딩 속도가 이상적이지 않았습니다. AI 코드 어시스턴트 프로젝트인 HagiCode는 문서와 기능 소개를 자주 업데이트해야 하므로 빌드 효율이 직접적으로 게시 속도에 영향을 미칩니다. 동시에 더 많은 개발자가 이 프로젝트를 발견할 수 있도록 사이트를 검색엔진(SEO)에 더 친화적으로 만들고자 했습니다.

이러한 문제를 해결하기 위해 대담한 결정을 내렸습니다: 전체 빌드 시스템을 완전히 재설계하고 Astro로 마이그레이션합니다. 이 결정으로 인한 변화는 상상보다 클 것입니다 — 자세한 내용은 나중에 설명하겠습니다.

## HagiCode 소개

이 글에서 공유하는 사이트 마이그레이션 방안은 [HagiCode](https://github.com/HagiCode-org/site) 프로젝트에서의 실천 경험에서 비롯되었습니다.

HagiCode는 개발 효율 향상에 전념하는 AI 코드 어시스턴트로, 핵심 기능의 반복뿐만 아니라 개발자 경험도 동등하게 중시합니다. 이번 사이트 리팩토링도 사용자가 문서와 공식 웹사이트를 탐색할 때 극한의 로딩 속도를 경험할 수 있도록 하기 위한 것입니다.

## 왜 성숙한 Docusaurus를 포기했는가?

React 생태계에서 Docusaurus는 항상 문서 사이트의 "표준 답안"이었습니다. 기본 제공 기능이 풍부하고, 플러그인이 다양하며, 커뮤니티가 활성화되어 있습니다. 하지만 HagiCode 기능이 증가함에 따라 그 한계도 느꼈습니다:

1.  **성능 병목**: Docusaurus는 본질적으로 React SPA(단일 페이지 애플리케이션)입니다. 순수 정적 페이지를 작성하더라도 클라이언트는 React 런타임을 로드하고 하이드레이션을 수행해야 합니다. 이는 단순한 문서 페이지에는 너무 무겁습니다.
2.  **리소스 부피**: 페이지 콘텐츠가 적더라도 번들링된 JS 부피는 상대적으로 고정되어 있어, 모바일 사용자와 네트워크 환경이 열악한 곳에 충분히 친화적이지 않습니다.
3.  **유연성 부족**: 확장이 가능하지만 빌드 프로세스의 커스터마이징에 있어 더 하위 수준의 제어권을 원했습니다.

Astro의 등장은 정확히 이러한 문제들을 해결했습니다. 새로운 "아일랜드 아키텍처"(Islands Architecture)를 제공합니다: 기본적으로 Astro는 JavaScript가 없는 정적 HTML을 생성하고, 상호작용이 필요한 컴포넌트만 "활성화"되어 JS를 로드합니다. 이는 사이트의 대부분 콘텐츠가 순수 HTML이 되어 매우 빠르다는 것을 의미합니다.

## 마이그레이션 핵심 전략: 아키텍처의 부드러운 전환

마이그레이션은 단순한 복사 붙여넣기가 아니라 사고방식의 전환입니다. Docusaurus의 "전체 React 모드"에서 Astro의 "Core + Islands" 모드로 전환했습니다.

### 1. 구성 시스템의 재구성

먼저 `docusaurus.config.ts`에서 `astro.config.mjs`로 전환해야 합니다. 이는 파일 이름 변경뿐만 아니라 라우팅 및 빌드 로직의 재작성입니다.

Docusaurus에서는 모든 것이 플러그인이고, Astro에서는 모든 것이 통합(Integration)입니다. 사이트의 기본 경로, 빌드 출력 모드(정적 vs SSR) 및 리소스 압축 전략을 재정의해야 합니다.

**마이그레이션 전:**
```typescript
// docusaurus.config.ts
export default {
  title: 'HagiCode',
  url: 'https://hagicode.com',
  baseUrl: '/',
  // ... 더 많은 구성
};
```

**마이그레이션 후:**
```javascript
// astro.config.mjs
import { defineConfig } from 'astro/config';
import react from '@astrojs/react';

export default defineConfig({
  integrations: [react()],
  site: 'https://hagicode.com',
  base: '/',
  // 정적 리소스 최적화 구성
  build: {
    inlineStylesheets: 'auto',
  },
});
```

### 2. React 컴포넌트의 보존과 개조

이것이 마이그레이션에서 가장 골치 아픈 부분입니다. 기존 사이트에는 많은 React 컴포넌트(예: Tabs 컴포넌트, 코드 하이라이트, 피드백 버튼 등)가 있습니다. 모두 버리기 아깝고, 모두 유지하면 JS 부하가 과도해집니다.

HagiCode는 **점진적 하이드레이션** 전략을 채택했습니다:
*   **순수 정적 컴포넌트**: 전시형 콘텐츠(예: 헤더, 푸터, 순수 텍스트 문서)에 대해 Astro 컴포넌트(`.astro` 파일)로 재작성하여 빌드 시 직접 HTML로 렌더링합니다.
*   **인터랙티브 아일랜드**: 상호작용을 유지해야 하는 컴포넌트(예: 테마 전환기, Tabs 전환, 코드 블록 복사 버튼)는 React 구현을 유지하고 `client:load` 또는 `client:visible` 지시문을 추가합니다.

예를 들어, 문서에서 자주 사용하는 Tabs 컴포넌트:

```jsx
// src/components/Tabs.jsx
import { useState } from 'react';
import './Tabs.css'; // 스타일 도입

export default function Tabs({ items }) {
  const [activeIndex, setActiveIndex] = useState(0);
  // ... 상태 로직
  return (
    <div className="tabs-wrapper">
      {/* 렌더링 로직 */}
    </div>
  );
}
```

Markdown에서 사용할 때 Astro에게 "이 컴포넌트는 JS가 필요합니다"라고 명확히 알립니다:

```markdown
// src/content/docs/example.mdx
import Tabs from '../../components/Tabs.jsx';

<!-- 컴포넌트가 뷰포트에 들어올 때만 JS 로드 -->
<Tabs client:visible items={...} />
```

이렇게 하면 뷰포트 외부의 인터랙티브 컴포넌트가 대역폭을 선점하지 않아 첫 화면 로딩 속도가 크게 최적화됩니다.

### 3. 스타일 시스템의 적응: CSS Modules에서 Scoped로

Docusaurus는 기본적으로 CSS Modules를 지원하고, Astro는 Scoped CSS(`<style>` 태그)를 권장합니다. 두 가지 모두 스타일 격리라는 핵심 사상을 공유하지만 문법이 다릅니다.

HagiCode 마이그레이션에서 대부분의 복잡한 CSS Modules를 Astro의 Scoped 스타일로 분해했습니다. 이는 사실 좋은 일입니다. `.astro` 파일에서 스타일과 템플릿이 같은 파일에 작성되어 유지보수가 더 직관적입니다.

**개조 전:**
```css
/* Tabs.module.css */
.wrapper { background: var(--ifm-background-color); }
```

**개조 후 (Astro Scoped):**
```html
<!-- Tabs.astro -->
<div class="tabs-wrapper">
  <slot />
</div>

<style>
  .tabs-wrapper {
    /* CSS 변수를 직접 사용하여 테마에 적응 */
    background: var(--bg-color);
    padding: 1rem;
  }
</style>
```

동시에 글로벌 CSS 변수 시스템을 통일하고, Astro의 환경 감지 능력을 활용하여 다크 모드가 다른 페이지 간에 원활하게 전환되도록 보장했습니다.

## 실천 중의 함정과 해결 방안

HagiCode의 실제 마이그레이션 과정에서 적지 않은 함정에 직면했습니다. 여기서 가장 전형적인 몇 가지를 공유합니다.

### 1. 경로와 환경 변수의 고통

HagiCode는 서브패스 배포를 지원합니다(예: GitHub Pages의 하위 디렉토리에 배포). Docusaurus에서는 `baseUrl`을 자동으로 처리하지만, Astro에서는 이미지 링크와 API 요청을 처리할 때 더 주의해야 합니다.

환경 변수 메커니즘을 도입하여 통합 관리했습니다:

```javascript
// 빌드 스크립트에서 경로 처리
const getBasePath = () => import.meta.env.VITE_SITE_BASE || '/';
```

반드시 기억하세요: 코드에 `/`로 시작하는 경로를 하드코딩하지 마세요. 개발 환경과 프로덕션 환경, 또는 `base` 경로를 구성한 후에 리소스 404가 발생할 수 있습니다.

### 2. CommonJS 스크립트의 호환성

기존 사이트에는 일부 Node.js 스크립트(Metrics 데이터 자동 수집, sitemap 업데이트 등용)가 있으며, CommonJS (`require`)로 작성되었습니다. Astro와 현대 빌드 도구는 ES Modules (`import`/`export`)를 전면적으로 수용합니다.

유사한 스크립트가 있다면 모두 ES Modules로 리팩토링하세요. 이는 대세이며, 일찍 바꿀수록 일찍 편해집니다.

```javascript
// 기존 방식
const fs = require('fs');

// 새 방식
import fs from 'fs';
```

### 3. SEO와 리다이렉트를 잊지 마세요

검색엔진이 이미 HagiCode의 기존 Docusaurus 페이지를 인덱싱했습니다. Astro로 바로 전환하면 URL 구조가 변경되어 대량의 404가 발생하고 권위가 크게 하락합니다.

Astro에서 리다이렉트 규칙을 구성했습니다:

```javascript
// astro.config.mjs
export default defineConfig({
  redirects: {
    '/docs/old-path': '/docs/new-path',
    // 기존 링크를 새 링크로 일괄 매핑
  }
});
```

또는 서버 구성 수준에서 처리합니다. 기존 링크가 새 주소로 301 리다이렉트되도록 보장하세요. 이는 SEO에 매우 중요합니다.

## 요약

Docusaurus에서 Astro로 마이그레이션하는 것은 HagiCode에게 단순한 프레임워크 업그레이드가 아니라 "성능 우선" 이념의 실천이었습니다.

**우리의 수확:**
*   **극한의 Lighthouse 점수**: 마이그레이션 후 HagiCode 사이트의 성능 점수가 쉽게 만점에 가까워졌습니다.
*   **더 빠른 빌드 속도**: Astro의 증분 빌드로 문서 업데이트의 게시 시간이 절반으로 단축되었습니다.
*   **유연성 보존**: Islands 아키텍처를 통해 어떠한 인터랙션 기능도 희생하지 않고, 필요한 곳에서 여전히 React를 사용할 수 있습니다.

문서형 사이트를 유지보수하고 있고, 번들 부피나 로딩 속도로 고민 중이라면 Astro를 시도해 보세요. 마이그레이션 과정에 약간의 수술이 필요하지만(예: PCode의 이름을 HagiCode로 변경하고 컴포넌트를 하나씩 가져오는 것), 그 대가로 실크처럼 부드러운 사용자 경험을 얻을 수 있으며, 충분히 가치가 있습니다.

이 글에서 공유하는 빌드 시스템은 HagiCode 개발 과정에서 실제로 시행착오를 겪으며 최적화한 방안입니다. 이 방안이 가치 있다고 생각되신다면, HagiCode 자체도 주목할 가치가 있습니다.
