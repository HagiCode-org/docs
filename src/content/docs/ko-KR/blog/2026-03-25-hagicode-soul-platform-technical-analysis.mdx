## HagiCode Soul 플랫폼 기술 해석: 니즈 시작부터 독립 플랫폼으로의 진화 과정

> 사실 기술 글을 쓰는 일은 대단한 게 아니야, 겪어본 함정과 우회한 돌아가는 길들을 정리하는 것일 뿐이지. 누구나 젊었던 시절이 있잖아, 그렇지? 이 글에서는 HagiCode 프로젝트의 Soul（AI Agent 인격 구성 시스템）의 설계 이념, 아키텍처 진화와 핵심 기술 구현을 심층적으로 해석하고, 독립 플랫폼을 통해 어떻게 더 집중된 Agent 인격 생성과 공유 경험을 제공할 수 있는지 탐구합니다.

## 배경

AI Agent 개발 실무에서 우리는 종종 단순해 보이지만 실제로는 매우 중요한 문제에 직면합니다: **어떻게 다른 Agent가 안정적이고 독특한 언어 스타일과 인격 특징을 가질 수 있게 할까요?**

이 문제는 말하자니 참 딱한 문제입니다. 초기 HagiCode의 Hero 시스템에서는 다른 영웅(Agent 인스턴스)이 주로 직업 구성과 일반 프롬프트로 표현 방식을 구분했습니다. 이 방식은 몇 가지 명확한 고통점을 가져왔는데, 해본 사람들은 공감할 수 있을 겁니다.

우선, 언어 스타일을 일관되게 유지하기 어렵습니다. 같은 "개발 엔지니어" 역할이라도 오늘의 응답은 전문적이고 엄격할 수 있고, 내일의 출력은 다시随意해질 수 있습니다. 이것은 모델 자체의 문제가 아니라, 출력 스타일을 제약하고 안내하는 독립적인 인격 구성 계층이 부족하기 때문입니다.

둘째, 역할감이 전반적으로 약합니다. 우리가 Agent의 특징을 묘사할 때 종종 "친절", "전문", "유머" 같은 모호한 형용사만 사용할 수 있을 뿐, 이러한 추상적인 묘사를 뒷받침하는 구체적인 언어 규칙이 없습니다. 솔직히 말하면, 말만 좋지 실제로는 방법이 없습니다.

셋째, 인격 구성의 재사용성이 거의 0입니다. 우리가 "고양이 메이드 웨이터"의 말하기 스타일을 정교하게 설계했다고 가정하고, 다른 비즈니스 시나리오에서 이 표현 방식을 재사용하려면 처음부터 구성해야 합니다. 아름다운 것 또는 사람, 점유할 필요는 없고 재사용만 하고 싶을 뿐인데......하지만 정말 어렵습니다.

바로 이러한 실제 문제를 해결하기 위해 우리는 **Soul** 메커니즘을 도입했습니다——장비와 묘사와 독립된 **언어 스타일 구성 계층**. Soul은 Agent의 말하기 습관, 어조 선호도와 용어 경계를 정의할 수 있고, 여러 영웅 간에 공유 재사용이 가능하며, Session 최초 호출 시 시스템 프롬프트에 자동 주입될 수 있습니다.

어떤 사람들은 이것이 그저 그런 거라고 생각할 수도 있습니다, 그냥 몇 개의 프롬프트를 구성하는 것에 불과하죠. 하지만 때로는 문제의 핵심이 할 수 있느냐가 아니라, 어떻게 더 우아하게 할 수 있느냐에 있습니다. Soul 능력이 점차 성숙해짐에 따라 우리는 이미 독립적으로 발전할 잠재력을 가지고 있음을 깨달았습니다. 전문적인 Soul 플랫폼은 사용자가 더 집중적으로 다양한 흥미로운 인격 구성을 생성, 공유 및 탐색할 수 있게 하고, Hero 시스템의 다른 기능의 간섭을 받지 않게 합니다. 그래서, **soul.hagicode.com** 독립 플랫폼이 탄생했습니다.

## Soul 플랫폼의 기술 아키텍처 진화

Soul 플랫폼의 발전은 단숨에 이루어진 것이 아니라 세 가지 명확한 단계를 거쳤습니다. 이 이야기는 갑자기 시작되어 자연스럽게 끝납니다.

### 첫 번째 단계: Hero 내장 Soul 구성

가장 초기 Soul 구현은 Hero 워크스페이스의 기능 모듈로 존재했습니다. 우리는 Hero 인터페이스에 독립적인 SOUL 편집 영역을 추가하여 프리셋 적용과 텍스트 미세 조정 두 가지 방식을 지원했습니다.

프리셋 적용은 사용자가 "전문 개발 엔지니어", "고양이 메이드 웨이터" 같은 클래식 인격 템플릿에서 선택할 수 있게 합니다. 텍스트 미세 조정은 사용자가 프리셋 기반으로 개인화된 수정을 할 수 있게 합니다. 백엔드 Hero 엔티티는 그에 따라 `Soul` 필드를 추가하고 `SoulCatalogId`로 출처를 식별했습니다.

이 단계는 "있느냐 없느냐"의 문제를 해결했고, 아직 어린아이라 며칠살이 비틀거리며 성장하고 있었습니다. 하지만 Soul 콘텐츠가 점점 풍부해짐에 따라 Hero 시스템과 결합된 아키텍처는 한계를 드러내기 시작했습니다.

### 두 번째 단계: 사내 Marketplace

더 나은 Soul 발견과 재사용 경험을 제공하기 위해 우리는 SOUL Marketplace 카탈로그 페이지를 구축하여 탐색, 검색, 상세 보기 및 즐겨찾기 기능을 지원했습니다.

이 단계에서 우리는 **50개의 주요 Catalog（기본 역할）**와 **10개의 직교 규칙（표현 방식）**의 조합 설계를 도입했습니다. 주요 Catalog는 Agent의 핵심 인격을 정의합니다. 예를 들어 "안개 항구 여행자", "야간 항해 사냥꾼" 같은 추상적인 역할 설정입니다. 직교 규칙은 표현 방식을 정의합니다. 예를 들어 "간결하고 능숙함", "수다롭고 친절함" 같은 언어 스타일 특징입니다.

50 × 10 = 500개의 조합 가능성은 사용자에게 풍부한 인격 구성 공간을 제공했습니다. 이 숫자는 많다고도 하고 적다고도 하니, 어쩌면 말이죠. 모든 길은 로마로 통하지만, 어떤 길은 조금 더 쉽게 갈 수 있을 뿐입니다. 백엔드는 `catalog-sources.json`을 통해 완전한 SOUL 카탈로그를 생성하고, 프론트엔드는 이러한 카탈로그 항목을 상호 작용 가능한 카드 목록으로 표현합니다.

사내 Marketplace는 좋은 과도기 솔루션이지만, 여전히 과도기일 뿐입니다. 여전히 주 시스템에 종속되어 있어 Soul 기능만 사용하려는 사용자에게는 접근 경로가 여전히 너무 깊습니다. 아무도 크게 돌아서야 간단한 일을 하고 싶지 않겠죠?

### 세 번째 단계: 독립 플랫폼 분리

결국 우리는 Soul 능력을 독립 저장소(`repos/soul`)로 마이그레이션하기로 결정했고, 원 주 시스템의 Marketplace는 외부 점프 안내로 변경되었으며, 새 플랫폼은 **Builder-first** 설계 이념을 채택했습니다——기본 홈페이지가 생성 워크벤치이며, 사용자가 웹사이트를 여는 순간부터 자신의 인격 구성 생성을 시작할 수 있습니다.

이 단계의 기술 스택도 전면적으로 업그레이드되었습니다: Vite 8 + React 19 + TypeScript 5.9 조합을 채택하고, shadcn/ui 컴포넌트 시스템으로 디자인 언어를 통일하며, Tailwind CSS 4의 테마 변수 시스템을 도입했습니다. 프론트엔드 엔지니어링 수준의 향상은 후속 기능 반복을 위한 견고한 기반을 마련했습니다.

모든 것이 옅어졌습니다......아니, 이제 막 시작입니다.

## 핵심 기술 설계와 구현

### 소재 통합 전략

Soul 플랫폼의 핵심 설계 이념 중 하나는 **로컬 우선**입니다. 즉, 홈페이지는 백엔드 없이도 완전히 실행 가능해야 하며, 원격 소재 실패 시 페이지 진입을 차단해서는 안 됩니다.

사실 이것도 대단한 건 아니야, 설계 시스템에서 한 단계 더 고려했을 뿐입니다. 로컬 스냅샷을 기준으로, 원격을 강화로 사용하는 이 생각은 제품이 어떤 네트워크 조건에서도 기본적인 사용 가능성을 제공하게 합니다. 구체적으로 우리는 두 계층의 소재 아키텍처를 채택했습니다:

```typescript
export async function loadBuilderMaterials(): Promise<BuilderMaterials> {
  const localMaterials = createLocalMaterials(snapshot)  // 로컬 기준

  try {
    const inspirationFragments = await fetchMarketplaceItems()  // 원격 강화
    return { ...localMaterials, inspirationFragments, remoteState: "ready" }
  } catch (error) {
    return { ...localMaterials, remoteState: "fallback" }  // 우아한 저하
  }
}
```

로컬 소재는 주 시스템 문서의 빌드 기간 스냅샷에서 오며, 50개의 기본 역할과 10개의 표현 규칙의 완전한 데이터를 포함합니다. 원격 소재는 사용자가 게시한 Soul에서 오며 Marketplace API를 통해 가져옵니다. 두 가지의 결합은 사용자에게 공식 템플릿부터 커뮤니티 창의성까지 완전한 소재 스펙트럼을 제공합니다. 웃음으로 흘러내린 눈물을 위장하려고......아니, 사실 아무것도 아니야, 그냥 로컬 더하기 원격일 뿐입니다.

### Soul 조각 데이터 모델

Soul의 핵심 데이터 추상화는 **SoulFragment**（영혼 조각）입니다:

```typescript
export type SoulFragment = {
  fragmentId: string
  group: "main-catalog" | "expression-rule" | "published-soul"
  title: string
  summary: string
  content: string
  keywords: string[]
  localized?: Partial<Record<AppLocale, LocalizedFragmentContent>>
  sourceRef: SoulFragmentSourceRef
  meta: SoulFragmentMeta
}
```

`group` 필드는 조각의 유형을 구분합니다: 주 카탈로그는 역할 커널을 정의하고, 직교 규칙은 표현 방식을 정의하며, 사용자가 게시한 Soul은 `published-soul`로 표시됩니다. `localized` 필드는 다국어를 지원하여 같은 조각이 다른 언어 환경에서 다른 제목과 묘사를 표현할 수 있게 합니다. 국제화 설계는 일찍이 해야 하는데, 우리도 이 말을 사용하게 되었습니다.

Builder 초안 상태는 사용자의 현재 편집 상태를 캡슐화합니다:

```typescript
export type SoulBuilderDraft = {
  draftId: string
  name: string
  selectedMainFragmentId: string | null
  selectedRuleFragmentId: string | null
  inspirationSoulId: string | null
  mainSlotText: string
  ruleSlotText: string
  customPrompt: string
  previewText: string
  updatedAt: string
}
```

사용자가 편집기에서 선택한 각 조각의 내용은 해당 slot（슬롯）에 연결되어 최종 미리보기 텍스트를 형성합니다. `mainSlotText`는 주 역할 콘텐츠에 해당하고, `ruleSlotText`는 표현 규칙 콘텐츠에 해당하며, `customPrompt`는 사용자의 추가 보충 지시입니다.

### 미리보기 컴파일 메커니즘

미리보기 컴파일은 Soul Builder의 핵심 기능으로, 사용자가 선택한 조각과 사용자 정의 텍스트를 복사 가능한 시스템 프롬프트로 조립합니다:

```typescript
export function compilePreview(
  draft: Pick<SoulBuilderDraft, "mainSlotText" | "ruleSlotText" | "customPrompt">,
  fragments: {
    mainFragment: SoulFragment | null
    ruleFragment: SoulFragment | null
    inspirationFragment: SoulFragment | null
  }
): PreviewCompilation {
  // 조립 논리: 주 역할 + 표현 규칙 + 영감 참고 + 사용자 정의 콘텐츠
}
```

컴파일 결과는 중앙 미리보기 패널에 표시되며, 사용자는 실시간으로 최종 효과를 보고 원 클릭으로 클립보드에 복사할 수 있습니다. 이 기능도 말하면 간단하지, 그렇지? 하지만 간단한 것들이 종종 가장 실용적입니다.

### 프론트엔드 상태 관리

Soul Builder의 프론트엔드 상태 관리는 중요한 원칙을 따릅니다: **상태 경계 명확하게 구분**. 구체적으로 말하면, 서랍 상태는 지속화되지 않고 초안에 직접 쓰지 않습니다. 명확한 Builder 작업만 상태 변경을 트리거합니다.

```typescript
// 도메인 상태（useSoulBuilder）
export function useSoulBuilder() {
  // 소재 로드와 캐시
  // 슬롯 집계와 미리보기 컴파일
  // 복사 행동과 피드백 메시지
  // Locale 안전한 설명자
}

// 표현 상태（useHomeEditorState）
export function useHomeEditorState() {
  // activeSlot, drawerSide, drawerOpen
  // 기본 포커스 동작
}
```

이 분리는 편집 상태의 안전성과 인터페이스의 응답 속도를 보장합니다. 서랍 열기 닫기는 순수한 UI 상호 작용으로 복잡한 지속화 논리를 트리거할 필요가 없습니다. 이것은 당연한 말이야! 아니, 사실 매우 중요합니다——인터페이스 상태와 비즈니스 상태를 명확히 구분하여 UI 상호 작용이 핵심 데이터 모델을 오염시키는 것을 피해야 합니다.

### 단일 서랍 수명 주기

Soul Builder는 단일 서랍 모드를 채택합니다: 동시에 하나의 슬롯 서랍만 열 수 있습니다. 마스크 레이어 클릭, ESC 키 누르기 또는 슬롯 전환은 현재 서랍을 자동으로 닫습니다. 이 설계는 상태 관리를 단순화하고 모바일 서랍 상호 작용의 일반적인 패턴에도 부합합니다.

서랍 닫기는 현재 편집 내용을 지우지 않으며, 사용자가 다시 전환하면 컨텍스트가 보존됩니다. 이 "가벼운" 서랍 설계는 사용자 작업의 중단감을 피합니다. 아무도 고생해서 작성한 내용을 실수로 잘못 클릭해서 전부 잃고 싶지 않겠죠?

### 이중 언어 지원 아키텍처

국제화는 Soul 플랫폼의 중요한 특징입니다. 시스템 문안은 완전히 이중 언어 전환을 지원하고, 사용자 초안 텍스트는 언어 전환으로 인해 절대 다시 쓰이지 않습니다——왜냐하면 초안 텍스트 자체가 사용자가 자유롭게 입력한 내용이기 때문에 시스템 번역과 관련이 없습니다.

공식 영감 카드(Marketplace Soul)는 상류 표시 이름을 유지하지만 최선의 영어 요약을 제공합니다. 중국어 이름의 Soul에 대해 우리는 미리 정의된 매핑 규칙을 통해 영어 버전을 생성합니다:

```typescript
// 주 역할 영어 이름 매핑
const mainNameEnglishMap = {
  "雾港旅人": "Mistport Traveler",
  "夜航猎手": "Night Hunter",
  // ...
}

// 직교 규칙 영어 이름 매핑
const ruleNameEnglishMap = {
  "简洁干练": "Concise & Professional",
  "啰嗦亲切": "Verbose & Friendly",
  // ...
}
```

이 매핑 테이블도 간단해 보이지만, 잘 유지하려면 적지 않은心思가 필요합니다. 50개의 주 역할과 10개의 직교 규칙이 있으니, 곱하면 500개의 조합이고, 이 숫자는 크다고도 하고 작다고도 합니다.

## 백엔드 카탈로그 생성

Soul Catalog의 대량 생성은 백엔드에서 완료되며, C#으로 50 × 10 = 500개의 조합 자동 생성을 구현했습니다:

```csharp
foreach (var main in source.MainCatalogs)
{
    foreach (var orthogonal in source.OrthogonalCatalogs)
    {
        var catalogId = $"soul-{main.Index:00}-{orthogonal.Index:00}";
        var displayName = BuildNickname(main, orthogonal);
        var soulSnapshot = BuildSoulSnapshot(main, orthogonal);
        // 데이터베이스에 쓰기...
    }
}
```

닉네임 생성 알고리즘은 주 역할 이름과 표현 규칙 이름을 결합하여 풍부한 상상력의 Agent 코드를 만듭니다:

```csharp
private static readonly string[] MainHandleRoots = [
    "雾港", "夜航", "零帧", "星渊", "霓虹", "断云", ...
];
private static readonly string[] OrthogonalHandleSuffixes = [
    "旅人", "猎手", "术师", "行者", "星使", ...
];
// 조합 예시：雾港旅人、夜航猎手、零帧术师...
```

Soul 스냅샷 조립은 고정된 템플릿 형식에 따라 주 역할 커널, 상징적 특징, 표현 규칙 커널과 출력 제약을 조합합니다:

```csharp
private static string BuildSoulSnapshot(main, orthogonal) => string.Join('\n', [
    $"你的人设内核来自「{main.Name}」：{main.Core}",
    $"保持以下标志性语言特征：{main.Signature}",
    $"你的表达规则来自「{orthogonal.Name}」：{orthogonal.Core}",
    $"必须遵循这些输出约束：{orthogonal.Signature}"
]);
```

이 템플릿 조립도 따지고 보면 지루한 일이지만, 이런 지루한 작업 없이 어떻게 흥미로운 제품이 나올까요?

## 플랫폼 마이그레이션 전략

Soul이 주 시스템에서 독립 플랫폼으로 분리된 후, 우리는 기존 사용자 데이터를 어떻게 처리할지라는 중요한 과제에 직면했습니다. 이 문제도 흔한 일이야——분리는 쉽고 마이그레이션은 어렵습니다. 우리는 세 가지 보장 조치를 취했습니다:

**후방 호환성 보장**. 저장된 Hero SOUL 스냅샷은 가시성을 유지하며, 과거 스냅샷은 Marketplace 출처 ID를 잃더라도 여전히 미리볼 수 있습니다. 즉 사용자의 이전 모든 구성이 손실되지 않고, 표시 위치만 변경되었습니다. 아무도 고생해서 구성한 것이 말 그대로 사라지길 원치 않겠죠.

**주 시스템 인터페이스 폐기**. 사내 Marketplace API는 410 Gone 상태 코드를 반환하고 마이그레이션 안내를 첨부하여 사용자가 soul.hagicode.com을 방문하도록 안내합니다.

**Hero SOUL 양식 개조**. Hero Soul 편집 영역에 마이그레이션 안내 블록을 추가하여 Soul 플랫폼이 이미 독립되었음을 명확히 알리고 원 클릭 점프 버튼을 제공합니다:

```typescript
// HeroSoulForm.tsx
<div className="rounded-2xl border border-orange-200/70 bg-orange-50/80 p-4">
  <div>{t('hero.soul.migrationTitle')}</div>
  <p>{t('hero.soul.migrationDescription')}</p>
  <Button onClick={onOpenSoulPlatform}>
    {t('hero.soul.openSoulPlatformAction')}
  </Button>
</div>
```

## 실무 요점 요약

Soul 플랫폼의 전체 개발 과정을 회고하면, 공유할 만한 몇 가지 실무 경험이 있습니다. 이것도 경험자의 작은心得, 대단한 도리는 아니고 그저 밟은 함정일 뿐입니다.

**로컬 우선의 런타임 가정**. 원격 데이터에 의존하는 기능을 설계할 때, 항상 네트워크를 사용할 수 없다고 가정해야 합니다. 로컬 스냅샷을 기준으로, 원격을 강화로 사용하는 이 생각은 제품이 어떤 네트워크 조건에서도 기본적인 사용 가능성을 제공하게 합니다. 요즘은 네트워크라는 게 말이죠, 끊어지면 끊어지는 거니 누구도 장담할 수 없습니다.

**상태 경계 명확하게 구분**. 인터페이스 상태와 비즈니스 상태를 명확히 구분하여 UI 상호 작용이 핵심 데이터 모델을 오염시키는 것을 피해야 합니다. 서랍 스위치는 순수한 UI 상태로 초안 지속화와 섞일 필요가 없습니다.

**국제화 설계는 일찍이**. 제품에 국제화 요구가 있다면 데이터 모델 설계 단계에서 고려하는 것이 가장 좋습니다. `localized` 필드는 데이터 구조의 복잡도를 높이지만, 후속 다국어 콘텐츠 유지 비용이 크게 줄어듭니다.

**소재 동기화 워크플로우는 자동화**. Soul 플랫폼의 로컬 소재는 주 시스템 문서에서 오므로, 상류 문서가 업데이트될 때 프론트엔드 스냅샷으로 동기화하는 메커니즘이 필요합니다. 우리는 `npm run materials:sync` 스크립트를 설계하여 이 과정을 자동화하고 소재가 항상 상류와 일치하도록 보장합니다.

## 미래 전망

현재 아키텍처 설계를 기반으로 Soul 플랫폼은 미래에 다음 발전 방향을 고려할 수 있습니다. 이것도 그저 얕은 생각일 뿐, 반드시 맞는 것은 아니니, 투석인견으로 삼아주세요.

**커뮤니티 공유 생태계**. 사용자가 사용자 정의 Soul을 업로드하고 공유할 수 있게 지원하고, 평점, 댓글 및 추천 메커니즘을 추가하여 우수한 Soul 구성이 더 많은 사람에게 발견되고 사용되게 합니다. 혼자 즐기는 것보다 함께 즐기는 게 낫습니다.

**다중 모달 확장**. 텍스트 스타일 외에도 음성 스타일 구성, 이모티콘 사용 선호도, 코드 스타일과 포맷팅 규칙 등 차원을 고려할 수 있습니다. 이 일은 말만 좋지 실제로는......

**지능형 보조**. 사용 시나리오를 기반으로 Soul을 자동 추천하고, 스타일 마이그레이션과 융합, 심지어 다른 Soul의 실제 효과를 A/B 테스트합니다. 아름다움은 날씨가 맑든 흐리든 상관없지 않아요? 시도해 보면 알게 됩니다.

**크로스 플랫폼 동기화**. 다른 AI 플랫폼에서 인격 구성을 가져오고, 표준화된 Soul 내보내기 형식을 제공하며, 주류 Agent 프레임워크와 통합합니다.

## 요약

이 글에서는 HagiCode Soul 플랫폼이 니즈 시작부터 독립 플랫폼까지의 완전한 진화 과정을 공유했습니다. 우리는 왜 Soul 메커니즘이 필요한지(Agent 인격 일관성 문제 해결), 기술 아키텍처의 세 가지 발전 단계(내장 구성, 사내 Marketplace, 독립 플랫폼)를 분석했으며, 핵심 데이터 모델, 상태 관리, 미리보기 컴파일 및 국제화 설계를 심층적으로 설명하고 플랫폼 마이그레이션의 실무 경험을 공유했습니다.

Soul의 본질은 비즈니스 로직과 독립적인 **인격 구성 계층**입니다. 그것은 AI Agent의 언어 스타일을 정의 가능, 재사용 가능, 공유 가능하게 만듭니다. 기술적 관점에서 이 설계는 복잡하지 않지만, 그것이 해결하는 문제는 진실하고 광범위한 요구가 있습니다.

당신도 AI Agent 제품을 개발 중이라면 당신의 인격 구성 솔루션이 충분히 유연한지 고려해 보세요. Soul 플랫폼의 실무가 당신에게 영감을 줄 수 있을 것입니다.

이 정은 추억이 될 수 있지만, 그때는 이미 망연했습니다. 어느 날 당신도 유사한 문제에 직면하게 될 것이고, 그때 이 글이 조금이라도 도움이 된다면, 그것으로 충분합니다.
