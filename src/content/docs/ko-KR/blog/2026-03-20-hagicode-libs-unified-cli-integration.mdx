## Hagicode.Libs: 다중 AI 프로그래밍 어시스턴트 CLI 통합 엔지니어링 실천

> HagiCode 프로젝트 개발 과정에서 Claude Code, Codex, CodeBuddy 등 여러 AI 프로그래밍 어시스턴트 CLI를 동시에 통합해야 했습니다. 각 CLI의 인터페이스, 매개변수, 출력 형식이 모두 달라서 중복되는 통합 코드로 인해 프로젝트가 점점 유지보수하기 어려워졌습니다. 이 글에서는 HagiCode.Libs 라이브러리를 통해 통일된 추상화 레이어를 구축하여 이 엔지니어링 문제를 해결한 방법을 공유합니다. 우리가 밟은 함정에서 축적된 약간의 경험입니다.

## 배경

요즘 AI 프로그래밍 어시스턴트 시장은 꽤 활기가 넘칩니다. Claude Code 외에도 OpenAI의 Codex, 智谱의 CodeBuddy 등이 있습니다. AI 코드 어시스턴트 프로젝트로서 HagiCode는 여러 하위 프로젝트(데스크톱, 백엔드, 웹 등)에서 이러한 다양한 CLI 도구를 통합해야 합니다.

처음에는 문제가 크지 않았습니다. CLI 하나를 통합하는 것은 수백 줄의 코드에 불과했습니다. 다만 지원해야 할 CLI가越来越多아지면서 상황이 번거로워지기 시작했습니다.

각 CLI는 다른 명령행 매개변수 형식을 가지고 있고, 환경 변수 요구사항도 다르며, 출력 형식은 더욱 다양합니다. JSON을 출력하는 것, 스트리밍 JSON을 출력하는 것, 순수 텍스트를 출력하는 것 등입니다. 게다가 크로스 플랫폼 호환성 문제까지 있습니다. Windows와 Unix 시스템의 실행 파일 발견 및 프로세스 관리 메커니즘이 완전히 다르고, 코드 중복도가 점점 높아졌습니다. 사실 이것은 Ctrl+C, Ctrl+V가 좀 더 많았을 뿐이지만, 유지보수는 정말 골치 아픕니다.

가장 골치 아픈 것은 새로운 CLI 기능 지원을 추가할 때마다 여러 프로젝트에서 같은 코드를 수정해야 한다는 것입니다. 이 방식은 분명 장기적인 해결책이 아닙니다. 코드에도 기분이 있어서, 너무 많이 반복되면 불만을 제기합니다.

## HagiCode에 대하여

이 글에서 공유하는 솔루션은 [HagiCode](https://hagicode.com) 프로젝트에서의 실천 경험에서 비롯되었습니다. HagiCode는 오픈소스 AI 코드 어시스턴트 프로젝트로, 프론트엔드 VSCode 확장, 백엔드 AI 서비스, 크로스 플랫폼 데스크톱 클라이언트 등 여러 하위 프로젝트를 동시에 유지 관리해야 합니다. 이러한 다국어, 다중 플랫폼의 복잡한 시나리오가 HagiCode.Libs의 탄생을 촉진했습니다. 억지로 만든 셈입니다.

## 분석: 공통점 찾기

비록 이러한 AI 프로그래밍 어시스턴트 CLI는 각자의 특징이 있지만, 기술적 관점에서 명확한 공통 특징이 존재합니다:

**유사한 상호작용 패턴**: CLI 프로세스를 시작하고, 프롬프트를 전송하며, 스트리밍 응답을 수신하고, 메시지를 파싱하며, 마지막으로 세션이 종료되거나 계속되는 일련의 프로세스입니다. 이 흐름은 결국 같은 틀에서 나온 것입니다.

**유사한 설정 요구사항**: 모두 API 키 인증, 작업 디렉토리 설정, 모델 선택, 도구 권한 제어, 세션 관리가 필요합니다.毕竟 모두 API를 먹고사는 사람들이니, 차이는 맛의 차이일 뿐입니다.

**일관된 크로스 플랫폼 과제**: 모두 실행 파일 경로 파싱(`claude` vs `claude.exe` vs `/usr/local/bin/claude`), 프로세스 시작 및 환경 변수 처리, 셸 명령 이스케이프 및 매개변수 구성 등의 문제를 해결해야 합니다. 이 크로스 플랫폼 문제는 이야기하면 눈물이 납니다. Windows와 Unix의 차이는 밟아본 사람만이 압니다.

이러한 분석을 기반으로 우리는 일관된 인터페이스를 제공하는 통일된 추상화 레이어, 크로스 플랫폼 CLI 발견 로직을 캡슐화하고, 스트리밍 출력 파싱을 처리하며, 의존성 주입과 비 DI 시나리오를 모두 지원하는 것이 필요했습니다. 생각만 해도 머리가 아프지만, 직면해야 합니다.毕竟 자기 프로젝트이니 울면서라도 끝내야 합니다.

## 솔루션: HagiCode.Libs

우리는 **HagiCode.Libs**를 만들었습니다. 경량급 .NET 10 라이브러리 워크스페이스로, MIT 오픈소스 라이선스를 채택했으며, [GitHub](https://github.com/HagiCode-org/Hagicode.Libs)에 공개되어 있습니다. 세상을 놀라게 할 대작은 아니지만, 실제 문제를 해결하는 데는 꽤 쓸모가 있습니다.

### 프로젝트 구조

```
HagiCode.Libs/
├── src/
│   ├── HagiCode.Libs.Core/           # 핵심 기능
│   │   ├── Discovery/                 # CLI 실행 파일 발견
│   │   ├── Process/                   # 크로스 플랫폼 프로세스 관리
│   │   ├── Transport/                 # 스트리밍 메시지 전송
│   │   └── Environment/               # 런타임 환경 파싱
│   ├── HagiCode.Libs.Providers/       # 제공자 구현
│   │   ├── ClaudeCode/                # Claude Code 제공자
│   │   ├── Codex/                     # Codex 제공자
│   │   └── Codebuddy/                 # CodeBuddy 제공자
│   ├── HagiCode.Libs.ConsoleTesting/  # 테스트 프레임워크
│   ├── HagiCode.Libs.ClaudeCode.Console/
│   ├── HagiCode.Libs.Codex.Console/
│   └── HagiCode.Libs.Codebuddy.Console/
└── tests/                             # xUnit 테스트
```

### 설계 목표

HagiCode.Libs를 설계할 때 우리는 몇 가지 원칙을 따랐습니다.毕竟 모두 밟은 함정에서 요약한 경험입니다:

**무거운 프레임워크 의존성 없음**: ABP 또는 다른 대형 프레임워크에 의존하지 않고 경량급을 유지합니다. 이 시대에 의존성이 적을수록 문제가 적습니다.毕竟 누가 의존성 지옥에 당한 적 없겠습니까.

**크로스 플랫폼 지원**: Windows, macOS, Linux 네이티브 지원, 다른 플랫폼용 코드를 따로 작성할 필요가 없습니다. 한 세트의 코드로 모든 곳을 커버합니다. 꽤 좋습니다.

**스트리밍 처리**: 비동기 스트림으로 CLI 출력을 처리하여 현대 .NET 프로그래밍 패턴에 더 부합합니다.毕竟 시대가 변하고, 비동기가 왕입니다.

**유연한 통합**: 의존성 주입 시나리오를 지원하면서 직접 인스턴스화 사용도 허용합니다. 취향은 사람마다 다르니, 편한 대로 하시면 됩니다.

## 사용 방법

### 의존성 주입을 통한 통합

프로젝트가 이미 의존성 주입을 사용 중이라면(예: ASP.NET Core 또는 제너릭 호스트), 직접 통합할 수 있습니다. 약간 작지만 꽤 잘 behaved하는 아이입니다:

```csharp
using HagiCode.Libs.Providers;
using Microsoft.Extensions.DependencyInjection;

var services = new ServiceCollection();
services.AddHagiCodeLibs();

await using var provider = services.BuildServiceProvider();
var claude = provider.GetRequiredService<ICliProvider<ClaudeCodeOptions>>();

var options = new ClaudeCodeOptions
{
    ApiKey = "your-api-key",
    Model = "claude-sonnet-4-20250514"
};

await foreach (var message in claude.ExecuteAsync(options, "Hello, Claude!"))
{
    Console.WriteLine($"{message.Type}: {message.Content}");
}
```

### 직접 인스턴스화

간단한 스크립트이거나 DI를 사용하지 않는 시나리오의 경우, 직접 인스턴스를 생성해도 됩니다. 개인 취향의 문제입니다:

```csharp
var claude = new ClaudeCodeProvider();
var options = new ClaudeCodeOptions
{
    ApiKey = "sk-ant-xxx",
    Model = "claude-sonnet-4-20250514"
};

await foreach (var message in claude.ExecuteAsync(options, "퀵 정렬을 작성해 주세요"))
{
    // 메시지 처리
}
```

두 방식 모두 동일한 하위 구현을 사용합니다. 프로젝트의 실제 상황에 따라 적절한 통합 방식을 선택할 수 있습니다. 이 세상에는 정답이 없습니다. 자신에게 맞는 것이 가장 좋습니다. 약간 진부한 말이지만, 사실 그렇습니다.

## 실천 경험 공유

### 1. 전용 테스트 콘솔

각 제공자에는 전용 테스트 콘솔 프로젝트가 있어 독립적인 통합 효과 검증에 편리합니다. 테스트는 하지 않거나 제대로 하세요:

```bash
# Claude Code 테스트
dotnet run --project src/HagiCode.Libs.ClaudeCode.Console -- --test-provider
dotnet run --project src/HagiCode.Libs.ClaudeCode.Console -- --test-all claude

# CodeBuddy 테스트
dotnet run --project src/HagiCode.Libs.Codebuddy.Console -- --test-provider codebuddy-cli

# Codex 테스트
dotnet run --project src/HagiCode.Libs.Codex.Console -- --test-provider codex-cli
```

테스트 시나리오는 몇 가지 핵심 상황을 포괄합니다:
- **Ping**: 상태 점검, CLI 사용 가능 여부 확인
- **Simple Prompt**: 기본 프롬프트 테스트
- **Complex Prompt**: 다중 대화 테스트
- **Session Restore/Resume**: 세션 복구 테스트
- **Repository Analysis**: 코드베이스 분석 테스트

이러한 독립적인 테스트 콘솔 설계는 디버깅 시 특히 유용합니다. 문제가 HagiCode.Libs 레이어에 있는지 CLI 자체에 있는지 빠르게 찾을 수 있습니다. 디버깅은 문제가 어디에 있는지 파악하는 것입니다. 방향이 맞으면 절반은 성공입니다.

### 2. 크로스 플랫폼 CI/CD 검증

크로스 플랫폼 호환성은 HagiCode.Libs의 핵심 목표 중 하나입니다. GitHub Actions 워크플로 `.github/workflows/cli-discovery-cross-platform.yml`을 구성하여 `ubuntu-latest`, `macos-latest`, `windows-latest` 세 플랫폼에서 실제 CLI 발견 검증을 실행합니다.

이것은 모든 코드 변경이 크로스 플랫폼 호환성을 깨뜨리지 않도록 보장합니다. 로컬 개발 시에도 다음 명령으로 재현할 수 있습니다.毕竟 CI에 모든 책임을 전가할 수는 없습니다. 로컬에서도 실행할 수 있어야 합니다:

```bash
npm install --global @anthropic-ai/claude-code@2.1.79
HAGICODE_REAL_CLI_TESTS=1 dotnet test --filter "Category=RealCli"
```

### 3. 메시지 스트림 처리

HagiCode.Libs는 비동기 스트림으로 CLI 출력을 처리하며, 이 방식은 전통적인 콜백이나 이벤트 패턴보다 현대 .NET의 비동기 프로그래밍 스타일에 더 부합합니다. 이것이 기술의 진보이며, 아무도 막을 수 없습니다:

```csharp
public async IAsyncEnumerable<CliMessage> ExecuteAsync(
    TOptions options,
    string prompt,
    [EnumeratorCancellation] CancellationToken cancellationToken = default)
{
    // CLI 프로세스 시작
    // 스트리밍 JSON 출력 파싱
    // CliMessage 시퀀스 생성
}
```

메시지 유형은 다음과 같습니다:
- `user`: 사용자 메시지
- `assistant`: 어시스턴트 응답
- `tool_use`: 도구 호출
- `result`: 세션 종료

이 설계는 호출자가 스트리밍 출력을 유연하게 처리할 수 있게 합니다. 예를 들어 실시간 표시, 버퍼링 후 처리 또는 다른 서비스로의 전달 등입니다. 아름다움을 굳이 맑은 날 흐린 날로 나눌 필요는 없습니다. 중요한 것은 사고가 열린 것이고, 어떻게 쓰든 다 됩니다.

### 4. Git 저장소 탐색

HagiCode.Libs.Exploration 모듈은 Git 저장소 발견 및 상태 확인 기능을 제공하며, 코드베이스 분석 시나리오에서 특히 유용합니다. 이것도 억지로 만든 기능입니다.毕竟 HagiCode는 코드베이스를 분석해야 하니까요:

```csharp
// Git 저장소 발견
var repositories = await GitRepositoryDiscovery.DiscoverAsync("/path/to/search");

// 저장소 정보 획득
var info = await GitRepository.GetInfoAsync(repoPath);
Console.WriteLine($"Branch: {info.Branch}, Remote: {info.RemoteUrl}");
Console.WriteLine($"Has uncommitted changes: {info.HasUncommittedChanges}");
```

HagiCode의 코드 분석 기능은 이 모듈을 사용하여 프로젝트 구조와 Git 상태를 식별합니다. 쓸모를 다한 것이며, 기른 보람이 있습니다.

## 주의사항

HagiCode 프로젝트에서의 실천을 기반으로 몇 가지 특별히 주의해야 할 점이 있습니다:

**API 키 보안**: API 키를 코드에 하드코딩하지 마세요. 환경 변수나 설정 관리를 사용하십시오. HagiCode.Libs는 Options 객체를 통해 설정을 전달하는 것을 지원하며, 다양한 설정 소스와의 통합이 용이합니다.毕竟 보안은 아무리 조심해도 지나치지 않습니다.

**CLI 버전 고정**: CI/CD에서는 버전 드리프트의 불확실성을 줄이기 위해 특정 버전(예: `@anthropic-ai/claude-code@2.1.79`)을 고정했습니다. 로컬 개발 시에도 고정 버전을 사용하는 것을 권장합니다. 버전 이야기는 눈물이 납니다. 고정하지 않으면 순식간에 교육을 시킵니다.

**테스트 분류**: 기본 테스트는 가짜 제공자를 사용하여 결정성과 속도를 유지하고, 실제 CLI 테스트는 명시적으로 활성화해야 합니다. 이렇게 하면 CI의 빠른 피드백을 보장하면서 필요할 때 실제 환경 검증을 수행할 수 있습니다. 이 균형 잡기는 쉽지 않습니다. 빠름과 안정 사이에는 항상 트레이드오프가 있습니다.

**세션 관리**: 다른 CLI의 세션 복구 메커니즘은 다릅니다. Claude Code는 `.claude/` 디렉토리를 사용하여 세션을 저장하고, Codex와 CodeBuddy는 각자의 방식이 있습니다. 사용 시 각각의 문서를 확인하여 세션 영속화의 구체적인 메커니즘을 이해해야 합니다.毕竟 제대로 알아두면 나쁠 것은 없습니다.

## 요약

HagiCode.Libs는 HagiCode 개발 과정에서 다중 CLI 통합의 반복 엔지니어링 문제를 해결하기 위해 구축한 통일된 추상화 레이어입니다. 일관된 인터페이스 제공, 크로스 플랫폼 세부 사항 캡슐화, 유연한 통합 방식 지원을 통해 여러 AI 프로그래밍 어시스턴트를 통합하는 엔지니어링 복잡도를 크게 낮추었습니다. 모든 것이 옅어졌습니다. 하지만 경험은 남았습니다.

프로젝트에서 여러 AI CLI 도구를 통합해야 하거나 크로스 플랫폼 프로세스 관리, 스트리밍 메시지 처리에 관심이 있으시다면 [GitHub](https://github.com/HagiCode-org/Hagicode.Libs)를 방문해 주세요. 프로젝트는 MIT 오픈소스 라이선스를 채택하며, 기여와 피드백을 환영합니다.终究은 한 번의 인연이니, 이렇게 왔으니 친구를 맺어봅시다.

이 글에서 공유한 솔루션은 HagiCode가 실제로 함정을 밟고 최적화한 것입니다. 어떻겠습니까. 함정은 정상입니다. 이 솔루션이 가치 있다고 생각하신다면, 우리의 엔지니어링 실력이 꽤 괜찮다는 뜻이겠죠. 그렇다면 HagiCode 자체도 주목할 만합니다. 혹시 놀라움이 있을지도 모릅니다.
