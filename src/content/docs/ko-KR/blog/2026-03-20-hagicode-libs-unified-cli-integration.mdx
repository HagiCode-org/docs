## Hagicode.Libs: 여러 AI 프로그래밍 도우미 CLI 통합을 위한 통합 엔지니어링 관행

> HagiCode 프로젝트를 개발하는 과정에서 Claude Code, Codex, CodeBuddy 등 여러 AI 프로그래밍 도우미 CLI를 동시에 통합해야 했습니다. 각 CLI의 인터페이스, 매개변수, 출력 형식이 모두 달랐고, 반복되는 통합 코드로 인해 프로젝트가 점점 더 유지 관리하기 어려워졌습니다. 이 글에서는 HagiCode.Libs 라이브러리를 통해 통합 추상화 계층을 구축하여 이 엔지니어링 문제를 해결한 방법을 공유합니다. 우리가 겪었던 문제들과 축적된 경험일 뿐입니다.

## 배경

현재 AI 프로그래밍 도우미 시장은 Claude Code 외에도 OpenAI의 Codex, 지푸의 CodeBuddy 등으로 꽤 활기찹니다. AI 코드 도우미 프로젝트인 HagiCode는 여러 하위 프로젝트(데스크톱, 백엔드, 웹 등)에서 이러한 다양한 CLI 도구를 통합해야 합니다.

처음에는 문제가 크지 않았습니다. CLI 하나를 통합하는 것은 수백 줄의 코드면 충분했습니다. 하지만 지원해야 할 CLI가越来越多아지면서 상황이 복잡해지기 시작했습니다.

각 CLI에는 다른 명령줄 매개변수 형식이 있고, 환경 변수 요구 사항도 다르며, 출력 형식은 더욱 다양합니다. 어떤 것은 JSON을 출력하고, 어떤 것은 스트리밍 JSON을 출력하며, 어떤 것은 순수 텍스트를 출력합니다. 여기에 크로스플랫폼 호환성 문제까지 더해지면, Windows와 Unix 시스템의 실행 파일 발견 및 프로세스 관리 메커니즘이 완전히 달라 코드 중복도가越来越 높아집니다. 사실 이것도 그저 Ctrl+C, Ctrl+V가 조금 더 많을 뿐이지만, 유지 관리는 정말 골치 아픈 일입니다.

가장 골치 아픈 것은 새로운 CLI 기능 지원을 추가할 때마다 여러 프로젝트에서 동일한 코드를 수정해야 한다는 점입니다. 이 방식은 분명 장기적인 해결책이 아닙니다. 코드에도 기분이 있고, 중복이 너무 많으면 고장을 일으킵니다.

## HagiCode 소개

이 글에서 공유하는 솔루션은 [HagiCode](https://hagicode.com) 프로젝트에서의 실무 경험에서 비롯되었습니다. HagiCode는 오픈소스 AI 코드 도우미 프로젝트로, 프론트엔드 VSCode 확장, 백엔드 AI 서비스, 크로스플랫폼 데스크톱 클라이언트 등 여러 하위 프로젝트를 동시에 유지 관리해야 합니다. 이러한 다언어, 다중 플랫폼의 복잡한 시나리오가 HagiCode.Libs의 탄생을 촉진했습니다. 억지로 만들어진 것이긴 합니다만.

## 분석: 공통점 찾기

이러한 AI 프로그래밍 도우미 CLI는 각각 특징이 있지만, 기술적 관점에서 보면 명확한 공통 특징이 있습니다.

**유사한 상호 작용 패턴**: 모두 CLI 프로세스를 시작하고, 프롬프트를 보내고, 스트리밍 응답을 받고, 메시지를 구문 분석하며, 마지막으로 세션이 종료되거나 계속됩니다. 이 전체 프로세스는 본질적으로 동일한 틀에서 나온 것입니다.

**유사한 구성 요구 사항**: 모두 API 키 인증, 작업 디렉터리 설정, 모델 선택, 도구 권한 제어, 세션 관리가 필요합니다. 모두 API로 밥을 먹는 것이라 차이는 기호일 뿐입니다.

**일관된 크로스플랫폼 과제**: 모두 실행 파일 경로 구문 분석(`claude` vs `claude.exe` vs `/usr/local/bin/claude`), 프로세스 시작 및 환경 변수 처리, 셸 명령 이스케이프 및 매개변수 구성 등의 문제를 해결해야 합니다. 크로스플랫폼 문제는 말하면 눈물이 나옵니다. Windows와 Unix의 차이는 직접 겪어본 사람만이 압니다.

이러한 분석을 바탕으로 일관된 인터페이스를 제공하고, 크로스플랫폼 CLI 발견 논리를 캡슐화하고, 스트리밍 출력 구문 분석을 처리하며, 동시에 의존성 주입과 비 DI 시나리오 사용을 지원하는 통합 추상화 계층이 필요했습니다. 생각만 해도 머리가 아프지만 직면해야 합니다. 결국 자신의 프로젝트니까 울면서도 완성해야 합니다.

## 솔루션: HagiCode.Libs

우리는 **HagiCode.Libs**를 만들었습니다. 이것은 경량의 .NET 10 라이브러리 작업 공간으로, MIT 오픈소스 라이선스를 채택했으며 [GitHub](https://github.com/HagiCode-org/Hagicode.Libs)에 게시되었습니다. 세상을 뒤집을 만한 대작은 아니지만 실제 문제를 해결하는 데는 꽤 유용합니다.

### 프로젝트 구조

```
HagiCode.Libs/
├── src/
│   ├── HagiCode.Libs.Core/           # 핵심 기능
│   │   ├── Discovery/                 # CLI 실행 파일 발견
│   │   ├── Process/                   # 크로스플랫폼 프로세스 관리
│   │   ├── Transport/                 # 스트리밍 메시지 전송
│   │   └── Environment/               # 런타임 환경 구문 분석
│   ├── HagiCode.Libs.Providers/       # 공급자 구현
│   │   ├── ClaudeCode/                # Claude Code 공급자
│   │   ├── Codex/                     # Codex 공급자
│   │   └── Codebuddy/                 # CodeBuddy 공급자
│   ├── HagiCode.Libs.ConsoleTesting/  # 테스트 프레임워크
│   ├── HagiCode.Libs.ClaudeCode.Console/
│   ├── HagiCode.Libs.Codex.Console/
│   └── HagiCode.Libs.Codebuddy.Console/
└── tests/                             # xUnit 테스트
```

### 설계 목표

HagiCode.Libs를 설계할 때 우리는 몇 가지 원칙을 따랐습니다. 이 모든 것이 겪었던 문제에서 나온 경험입니다.

**무거운 프레임워크 의존성 없음**: ABP나 기타 대형 프레임워크에 의존하지 않고 경량을 유지합니다. 이 시대에 의존성이 적을수록 문제가 적습니다. 의존성 지옥에서 고생해본 사람이면 다 압니다.

**크로스플랫폼 지원**: Windows, macOS, Linux를 기본 지원하며, 다른 플랫폼을 위해 다른 코드를 작성할 필요가 없습니다. 한 코드로 모든 곳에서 작동하는 것도 좋습니다.

**스트리밍 처리**: 비동기 스트림을 사용하여 CLI 출력을 처리하며, 현대 .NET 프로그래밍 패턴에 더 적합합니다. 시대는 변하고 비동기가 왕입니다.

**유연한 통합**: 의존성 주입 시나리오와 직접 인스턴스화 사용을 모두 지원합니다. 각자 좋아하는 대로, 편한 대로 사용하면 됩니다.

## 사용 방법

### 의존성 주입을 통한 통합

프로젝트에서 이미 의존성 주입을 사용 중이라면(예: ASP.NET Core 또는 일반 호스트), 직접 통합할 수 있습니다. 작은 녀석이긴 하지만 꽤 얌전합니다:

```csharp
using HagiCode.Libs.Providers;
using Microsoft.Extensions.DependencyInjection;

var services = new ServiceCollection();
services.AddHagiCodeLibs();

await using var provider = services.BuildServiceProvider();
var claude = provider.GetRequiredService<ICliProvider<ClaudeCodeOptions>>();

var options = new ClaudeCodeOptions
{
    ApiKey = "your-api-key",
    Model = "claude-sonnet-4-20250514"
};

await foreach (var message in claude.ExecuteAsync(options, "Hello, Claude!"))
{
    Console.WriteLine($"{message.Type}: {message.Content}");
}
```

### 직접 인스턴스화

간단한 스크립트이거나 DI를 사용하지 않는 시나리오라면 직접 인스턴스를 만들어도 됩니다. 결국 개인 취향의 문제입니다:

```csharp
var claude = new ClaudeCodeProvider();
var options = new ClaudeCodeOptions
{
    ApiKey = "sk-ant-xxx",
    Model = "claude-sonnet-4-20250514"
};

await foreach (var message in claude.ExecuteAsync(options, "帮我写一个快速排序"))
{
    // 메시지 처리
}
```

두 방식 모두 동일한 기본 구현을 사용합니다. 프로젝트 실제 상황에 따라 적절한 통합 방식을 선택할 수 있습니다. 이 세상에 정답은 없고 자신에게 맞는 것이 최고입니다. 이 말은 약간 진부하지만 사실입니다.

## 실무 경험 공유

### 1. 전용 테스트 콘솔

각 공급자는 독립적인 통합 효과 검증을 위한 전용 테스트 콘솔 프로젝트가 있습니다. 테스트는 하지 않거나 하면 제대로 하는 것입니다:

```bash
# Claude Code 테스트
dotnet run --project src/HagiCode.Libs.ClaudeCode.Console -- --test-provider
dotnet run --project src/HagiCode.Libs.ClaudeCode.Console -- --test-all claude

# CodeBuddy 테스트
dotnet run --project src/HagiCode.Libs.Codebuddy.Console -- --test-provider codebuddy-cli

# Codex 테스트
dotnet run --project src/HagiCode.Libs.Codex.Console -- --test-provider codex-cli
```

테스트 시나리오는 몇 가지 핵심 시나리오를 포함합니다:
- **Ping**: 상태 확인, CLI 사용 가능 여부 확인
- **Simple Prompt**: 기본 프롬프트 테스트
- **Complex Prompt**: 다중 대화 테스트
- **Session Restore/Resume**: 세션 복원 테스트
- **Repository Analysis**: 코드베이스 분석 테스트

이러한 독립적인 테스트 콘솔 디자인은 디버깅할 때 특히 유용하며, HagiCode.Libs 계층에 문제가 있는지 CLI 자체에 문제가 있는지 빠르게 찾을 수 있습니다. 디버깅은 결국 문제가 어디에 있는지 파악하는 것입니다. 방향이 맞으면 절반은 성공입니다.

### 2. 크로스플랫폼 CI/CD 검증

크로스플랫폼 호환성은 HagiCode.Libs의 핵심 목표 중 하나입니다. 우리는 GitHub Actions 워크플로 `.github/workflows/cli-discovery-cross-platform.yml`을 구성하여 `ubuntu-latest`, `macos-latest`, `windows-latest` 세 플랫폼에서 실제 CLI 발견 검증을 실행합니다.

이렇게 하면 모든 코드 변경이 크로스플랫폼 호환성을 깨뜨리지 않도록 보장합니다. 로컬 개발 시 다음 명령으로 재현할 수도 있습니다. 모든 것을 CI에 맡길 수는 없습니다. 로컬에서도 실행할 수 있어야 합니다:

```bash
npm install --global @anthropic-ai/claude-code@2.1.79
HAGICODE_REAL_CLI_TESTS=1 dotnet test --filter "Category=RealCli"
```

### 3. 메시지 스트림 처리

HagiCode.Libs는 비동기 스트림을 사용하여 CLI 출력을 처리합니다. 이 방식은 기존 콜백이나 이벤트 패턴보다 현대 .NET 비동기 프로그래밍 스타일에 더 적합합니다. 결국 기술의 진보이며 아무도 막을 수 없습니다:

```csharp
public async IAsyncEnumerable<CliMessage> ExecuteAsync(
    TOptions options,
    string prompt,
    [EnumeratorCancellation] CancellationToken cancellationToken = default)
{
    // CLI 프로세스 시작
    // 스트리밍 JSON 출력 구문 분석
    // CliMessage 시퀀스 생성
}
```

메시지 유형은 다음을 포함합니다:
- `user`: 사용자 메시지
- `assistant`: 도우미 응답
- `tool_use`: 도구 호출
- `result`: 세션 종료

이 디자인은 호출자가 스트리밍 출력을 유연하게 처리할 수 있게 합니다. 예를 들어 실시간 표시, 버퍼링 후 처리 또는 다른 서비스로 전달할 수 있습니다. 날씨가 어떤지는 중요하지 않습니다. 중요한 것은 사고가 열린 것입니다. 어떻게 사용하든 됩니다.

### 4. Git 저장소 탐색

HagiCode.Libs.Exploration 모듈은 Git 저장소 발견 및 상태 확인 기능을 제공합니다. 이는 코드베이스 분석 시나리오에서 특히 유용합니다. 이것도 억지로 만든 기능입니다. HagiCode가 코드베이스를 분석해야 하니까요:

```csharp
// Git 저장소 발견
var repositories = await GitRepositoryDiscovery.DiscoverAsync("/path/to/search");

// 저장소 정보 가져오기
var info = await GitRepository.GetInfoAsync(repoPath);
Console.WriteLine($"Branch: {info.Branch}, Remote: {info.RemoteUrl}");
Console.WriteLine($"Has uncommitted changes: {info.HasUncommittedChanges}");
```

HagiCode의 코드 분석 기능은 이 모듈을 사용하여 프로젝트 구조와 Git 상태를 식별합니다. 제 몫을 하는 셈입니다. 키운 보람이 있습니다.

## 주의 사항

HagiCode 프로젝트에서의 실무 경험을 바탕으로 몇 가지 주의해야 할 점이 있습니다. 모두 문제, 문제, 문제입니다:

**API 키 보안**: API 키를 코드에 하드코딩하지 말고 환경 변수나 구성 관리를 사용하세요. HagiCode.Libs는 Options 개체를 통해 구성을 전달하는 것을 지원하므로 다양한 구성 소스와 통합하기 쉽습니다. 보안은 아무리 조심해도 지나치지 않습니다.

**CLI 버전 고정**: CI/CD에서 우리는 버전 드리프트로 인한 불확실성을 줄이기 위해 특정 버전(예: `@anthropic-ai/claude-code@2.1.79`)을 고정했습니다. 로컬 개발 시에도 고정 버전을 사용하는 것이 좋습니다. 버전 문제는 말하면 눈물이 나옵니다. 버전을 고정하지 않으면 바로 교육시킵니다.

**테스트 분류**: 기본 테스트는 결정성과 속도를 유지하기 위해 가짜 공급자를 사용하며, 실제 CLI 테스트는 명시적으로 활성화해야 합니다. 이렇게 하면 CI의 빠른 피드백을 보장하면서 필요할 때 실제 환경 검증을 할 수 있습니다. 이 균형을 유지하는 것도 쉽지 않습니다. 빠름과 안정은 항상 트레이드오프가 필요합니다.

**세션 관리**: 다른 CLI의 세션 복원 메커니즘은 다릅니다. Claude Code는 `.claude/` 디렉터리를 사용하여 세션을 저장하고 Codex와 CodeBuddy는 각자의 방식이 있습니다. 사용할 때 각 문서를 확인하여 세션 지속성의 구체적인 메커니즘을 이해해야 합니다. 이것도 알아두면 나쁠 것은 없습니다.

## 요약

HagiCode.Libs는 HagiCode를 개발하는 과정에서 다중 CLI 통합의 반복되는 엔지니어링 문제를 해결하기 위해 구축한 통합 추상화 계층입니다. 일관된 인터페이스를 제공하고 크로스플랫폼 세부 정보를 캡슐화하며 유연한 통합 방식을 지원함으로써 여러 AI 프로그래밍 도우미 통합의 엔지니어링 복잡성을 크게 줄였습니다. 모든 것이 희미해졌지만 경험은 남았습니다.

프로젝트에서 여러 AI CLI 도구를 통합해야 하거나 크로스플랫폼 프로세스 관리, 스트리밍 메시지 처리에 관심이 있다면 [GitHub](https://github.com/HagiCode-org/Hagicode.Libs)에서 둘러보세요. 프로젝트는 MIT 오픈소스 라이선스를 채택하며 기여와 피드백을 환영합니다. 결국 인연입니다. 왔으니 친구를 맺어보세요.

이 글에서 공유한 솔루션은 HagiCode가 실제로 겪은 문제를 통해 실제로 최적화된 것입니다. 어쩌겠습니까? 문제를 겪는 것은 정상입니다. 이 솔루션이 가치 있다고 생각한다면 우리의 엔지니어링 실력이 그럭저럭 괜찮다는 뜻입니다. 그렇다면 HagiCode 자체도 주목할 만합니다.说不定惊喜이 있습니다.
