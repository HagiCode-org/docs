## HagiCode Desktop 하이브리드 배포 아키텍처 분석: P2P로 대용량 파일 다운로드 가속하기

> 실제로 이 글은 오랫동안 미루다가 겨우 써낸 거라, 잘 쓴지도 모르겠네요. 기술 글이라는 게, 쓰는 건 쉬워도 제맛을 살리기는 어려운 법이죠. 그래도 에이, 그만두죠. 대문장가도 아닌데, 신의 필체도 없고, 그냥 이 조잡한 글이나 다 써낸 거니까요.

## 배경

데스크톱 앱 개발을 하는 팀이라면, 이른바 늦든 빠르든 한 번쯤은 마주하게 되는 골치 아픈 문제가 있습니다: 대용량 파일을 어떻게 배포할 것인가?

이 얘기가 참 속상하게도 이렇습니다. 전통적인 HTTP/HTTPS 직링크 다운로드는, 파일 크기가 작고 사용자가 많지 않을 때는 사실 꽤 버틸만합니다——마치 소년 시절의 감정처럼, 단순하고 순수하고, 별일 없죠. 하지만 아, 시간이라는 게 정말 무정해서, 프로젝트가 계속 발전하면서 설치 패키지는 점점 커지고: Desktop 쪽 ZIP 패키지, 포터블 패키지(portable package), Web 배포 아카이브... 문제가 서서히 드러나기 시작합니다:

- **다운로드 속도가 원본 서버 대역폭에 제한됨**: 단일 서버의 대역폭이 아무리 높아도, 모두가 동시에 다운로드하면 버티기 어렵습니다. 이게 무엇과 같을까요? 한 사람을 좋아하지만, 그 사람의 마음은 그만큼밖에 안 되고, 이미 다른 사람으로 가득 차 있어서, 아무리 노력해도 들어갈 수 없는 것과 같습니다.
- **이어받기 능력이 사실상 제로**: HTTP 다운로드가 끊기면 처음부터 다시 시작해야 하고, 시간 낭비는 둘째치고 대역폭까지 낭비입니다. 아름다움이 굳이 맑음과 흐림을 따질 필요는 없지만, 안타깝게도 하늘은 뜻대로 되지 않습니다.
- **원본 서버 부하 심각**: 모든 트래픽이 중앙 서버로 쏠리면서, 대역폭 비용이 미친 듯이 오르고, 확장성도 문제가 됩니다. 이게 바로 소위 중앙집중식의 딜레마죠——모든 걸 한 지점에 누르면, 결국 무너지게 되어 있습니다.

HagiCode Desktop 프로젝트도 예외가 아닙니다. 저희가 배포 시스템을 설계할 때, 이런 생각을 했어요: 기존 `index.json` 제어 플레인을 바꾸지 않으면서, 하이브리드 배포 솔루션을 만들 수 있을까? P2P 네트워크의 분산 특성을 활용해 다운로드를 가속하면서도, HTTP 폴백은 유지해서, 기업 네트워크 같은 제약된 환경에서도 가용성을 보장할 수 있는.

이 결정이 가져온 변화는, 당신이 상상하는 것보다 훨씬 클 겁니다——서두르지 마세요, 아래에서 자세히 설명하겠습니다. 뭐, 어떤 일들은 말해야만 이해받을 수 있으니까요.

Desktop 쪽의 하이브리드 배포 아키텍처는 바로 HagiCode가 실제 운영에서 삽질하면서 최적화해 나온 솔루션입니다. 혹시 이런 걸 왜 쓰냐고 물어볼 분도 계실 것 같습니다. 사실 별 의미 없습니다. 그저 이 솔루션이 가치가 있다면, 저희가 엔지니어링 실무에서 좀 노하우가 있다는 뜻이니까요——그렇다면 HagiCode 자체도 한 번쯤 눈여겨볼 만하다, 그 정도입니다.

프로젝트의 GitHub 주소는 [HagiCode-org/site](https://github.com/HagiCode-org/site)입니다. 관심 있으시면 일단 스타 찍어두시고. 아름다운 건, 보관될 가치가 있으니까요.

## 핵심 설계 사상: P2P 우선, HTTP 폴백

결론부터 말하면, 하이브리드 배포의 핵심 사상은 한 문장으로 요약할 수 있습니다: **P2P 우선, HTTP 폴백**.

이 솔루션의 핵심은 「하이브리드」라는 두 글자에 있습니다. 그냥 BitTorrent을 던져서 끝나는 게 아니라, 두 가지 다운로드 방식이 협동하고 서로의 장단점을 보완하도록 만드는 겁니다:

- **P2P 네트워크**는 분산 가속을 제공하고, 다운로드하는 사람이 많을수록 노드가 많아지고 속도가 빨라집니다. 이게 무엇과 같을까요? 우리나 저나 다 소년 시절의 그 사람이었던 그 시절을 떠올려 보세요, 마음속에 빛이 있으면, 세상이 다 밝아진다고 느껴지는 것과 같습니다.
- **WebSeed/HTTP 폴백**은 가용성을 보장하고, 기업 방화벽, 내부망 환경에서도 정상적으로 다운로드할 수 있습니다. 아무 곳이나你想进就能进的 곳은 아닙니다.
- **제어 플레인은 단순하게 유지**하고, `index.json`의 핵심 로직을 바꾸지 않고, 선택적 메타데이터 필드만 추가합니다. 단순한 게 뭐가 나쁘죠? 복잡한 일을 많이 하다 보면, 가끔 단순하게 하는 것도 꽤 좋습니다.

이렇게 하는 장점이 뭘까요? 사용자가 체감하는 건 「다운로드가 더 빨라졌다」는 거고, 엔지니어링 팀은 P2P의 복잡성에 너무 많은 대가를 치를 필요가 없습니다——아무래도 BT 프로토콜 자체가 이미 충분히 성숙하니까, 저희도 바퀴를 다시 발명할 생각은 없습니다.

## 아키텍처 설계

### 계층형 아키텍처 개요

먼저 전체 아키텍처 다이어그램을 하나 보여드릴게요, 거시적인 인상을 드리게:

```
┌─────────────────────────────────────┐
│     Renderer (UI 층)                │
├─────────────────────────────────────┤
│     IPC/Preload (브리지 층)            │
├─────────────────────────────────────┤
│   VersionManager (버전 관리)         │
├─────────────────────────────────────┤
│ HybridDownloadCoordinator (조정 층)  │
│  ├── DistributionPolicyEvaluator   │
│  ├── DownloadEngineAdapter         │
│  ├── CacheRetentionManager         │
│  └── SHA256 Verifier               │
├─────────────────────────────────────┤
│   WebTorrent (다운로드 엔진)             │
└─────────────────────────────────────┘
```

이 다이어그램에서 알 수 있듯이, 전체 시스템은 계층형으로 설계되어 있습니다. 왜 이렇게 세분화했을까요? 주로 **테스트 가능성과 교체 가능성**을 위해서입니다. 사실 사람 사는 것도 이 원리와 비슷합니다——일을 명확히 나누고, 각자의 역할을 하면, 세상이 단순해지죠.

- **UI 층**은 다운로드 진행률, 공유 가속 스위치 표시를 담당합니다——이게 겉모습이고
- **조정 층**이 핵심이고, 정책 평가, 엔진 어댑터, 캐시 관리, 무결성 검증을 포함합니다——이게 핵심이고
- **엔진 층**은 구체적인 다운로드 구현을 캡슐화하고, 현재는 WebTorrent를 사용합니다——이게 도구입니다.

엔진 층을 `DownloadEngineAdapter` 인터페이스로 추상화해서, 나중에 다른 BT 엔진으로 바꾸거나 사이드카 프로세스를 만들어도, 별로 어렵지 않게 돌아갑니다. 아무래도 한 나무에 목매고 싶은 사람은 없고, 코드 세계도 마찬가지입니다.

### 제어 플레인과 데이터 플레인 분리

HagiCode Desktop은 `index.json`을 유일한 제어 플레인으로 유지하는데, 이 설계가 아주 중요합니다. 제어 플레인은 버전 발견, 채널 선택, 중앙집중식 정책을 담당하고, 데이터 플레인이 실제 파일을 다운로드하는 곳입니다.

`index.json`에 추가된 필드는 선택적입니다:

```json
{
  "asset": {
    "torrentUrl": "https://cdn.example.com/app.torrent",
    "infoHash": "abc123...",
    "webSeeds": [
      "https://cdn.example.com/app.zip",
      "https://backup.example.com/app.zip"
    ],
    "sha256": "def456...",
    "directUrl": "https://cdn.example.com/app.zip"
  }
}
```

이 필드들은 모두 선택적이고, 누락되면 전통적인 HTTP 다운로드 모드로 폴백합니다. 이렇게 설계한 장점은 **하위 호환성**이고, 구버전 클라이언트는 전혀 영향을 받지 않습니다. 세상은 변하지만, 어떤 건 변하면 안 됩니다——변하면 되돌릴 수 없으니까요.

### 정책 기반 의사결정

모든 파일이 P2P 배포할 가치가 있는 건 아닙니다. 사실 세상 일 대부분이 그렇습니다——모든 걸 다 한 번 싸워볼 필요는 없고, 어떤 건 안 맞으면 안 맞는 거죠, 한 걸음 물러나면 하늘이 뚫립니다.

DistributionPolicyEvaluator는 정책을 평가하고, 다음 조건을 만족하는 파일만 하이브리드 다운로드를 활성화합니다:

1. **소스 타입은 반드시 HTTP index여야 함**: GitHub 직접 다운로드나 로컬 폴더 소스는 이 방식을 쓰지 않습니다. 아무래도 모든 길이 P2P에 적합한 건 아닙니다.
2. **파일 크기는 반드시 ≥ 100MB여야 함**: 작은 파일에 P2P를 쓰면 오버헤드가 더 큽니다. 감정도 마찬가지입니다, 어떤 일은 너무 사소해서 크게 일 벌일 필요가 없죠.
3. **완전한 하이브리드 메타데이터를 갖춰야 함**: torrentUrl, webSeeds, sha256 하나도 빠짐없이. 하나라도 빠지면 안 되고, 이게 규칙입니다.
4. **latest desktop 패키지와 web 배포 패키지로 제한**: 구버전은 전통적인 방식으로 하면 됩니다. 새 사람이 웃고, 옛 사람이 우는데, 굳이 왜 그럴까요?

```typescript
class DistributionPolicyEvaluator {
  evaluate(version: Version, settings: SharingAccelerationSettings): HybridDownloadPolicy {
    // 소스 타입 검사
    if (version.sourceType !== 'http-index') {
      return { useHybrid: false, reason: 'not-http-index' };
    }

    // 메타데이터 무결성 검사
    if (!version.hybrid) {
      return { useHybrid: false, reason: 'not-eligible' };
    }

    // 활성화 여부 검사
    if (!settings.enabled) {
      return { useHybrid: false, reason: 'shared-disabled' };
    }

    // 자산 타입 검사(latest desktop/web 패키지만)
    if (!version.hybrid.isLatestDesktopAsset && !version.hybrid.isLatestWebAsset) {
      return { useHybrid: false, reason: 'latest-only' };
    }

    return { useHybrid: true, reason: 'shared-enabled' };
  }
}
```

이렇게 하는 장점은, **시스템 동작을 예측할 수 있다**는 겁니다. 개발자든 사용자든, 어떤 파일이 P2P로 가고 어떤 건 아닌지 명확히 알 수 있습니다. 기대 관리만 잘되면, 마음도 안정됩니다.

## 핵심 구현

### 타입 정의 체계

먼저 타입 정의를 볼까요, 이게 전체 시스템의 기초입니다. 사실 타입 정의라는 게, 사물에 성격을 부여하는 것과 같습니다——한번 정해지면, 그 뒤의 길이 편해지죠.

```typescript
// 하이브리드 배포 메타데이터
interface HybridDistributionMetadata {
  torrentUrl?: string;      // 토렌트 파일 URL
  infoHash?: string;        // InfoHash
  webSeeds: string[];       // WebSeed 리스트
  sha256?: string;          // 파일 해시
  directUrl?: string;       // HTTP 직링크(폴백용)
  eligible: boolean;        // 하이브리드 배포 조건 부합 여부
  thresholdBytes: number;   // 임계값(바이트)
  assetKind: VersionAssetKind;
  isLatestDesktopAsset: boolean;
  isLatestWebAsset: boolean;
}

// 공유 가속 설정
interface SharingAccelerationSettings {
  enabled: boolean;           // 메인 스위치
  uploadLimitMbps: number;    // 업로드 속도 제한
  cacheLimitGb: number;       // 캐시 상한
  retentionDays: number;      // 보관 일수
  hybridThresholdMb: number;  // 하이브리드 배포 임계값
  onboardingChoiceRecorded: boolean;
}

// 다운로드 진행률
interface VersionDownloadProgress {
  current: number;
  total: number;
  percentage: number;
  stage: VersionInstallStage;  // queued, downloading, backfilling, verifying, extracting, completed, error
  mode: VersionDownloadMode;   // http-direct, shared-acceleration, source-fallback
  peers?: number;              // 연결된 노드 수
  p2pBytes?: number;          // P2P로 가져온 바이트 수
  fallbackBytes?: number;      // 폴백으로 가져온 바이트 수
  verified?: boolean;          // 검증 여부
}
```

타입 정의가 명확하니, 뒤의 구현은 자연스럽게 이어집니다. 아마 이게 소위 「잘 시작하면 반은 성공이다」라는 말이나 다름없겠네요, 좀 촌스럽긴 해도.

### 핵심 조정기

HybridDownloadCoordinator는 전체 다운로드 프로세스의 오케스트레이터이고, 정책 평가, 엔진 실행, SHA256 검증, 캐시 관리를 조정합니다. 말하자면 복잡한데, 사실 핵심 로직은 그냥 몇 단계고, 인생과 아주 비슷합니다——겉보기에 복잡해 보이지만, 실체를 들춰보면 그게 전부입니다.

```typescript
class HybridDownloadCoordinator {
  async download(
    version: Version,
    cachePath: string,
    packageSource: PackageSource,
    onProgress?: DownloadProgressCallback,
  ): Promise<HybridDownloadResult> {
    // 1. 정책 평가: 하이브리드 다운로드 사용 여부
    const policy = this.policyEvaluator.evaluate(version, settings);

    // 2. 다운로드 실행
    if (policy.useHybrid) {
      await this.engine.download(version, cachePath, settings, onProgress);
    } else {
      await packageSource.downloadPackage(version, cachePath, onProgress);
    }

    // 3. SHA256 검증(강력한门槛)
    const verified = await this.verify(version, cachePath, onProgress);
    if (!verified) {
      await this.cacheRetentionManager.discard(version.id, cachePath);
      throw new Error(`sha256 verification failed for ${version.id}`);
    }

    // 4. 신뢰할 수 있는 캐시로 표시, 제어된 시딩 시작
    await this.cacheRetentionManager.markTrusted({
      versionId: version.id,
      cachePath,
      cacheSize,
    }, settings);

    return { cachePath, policy, verified };
  }
}
```

여기에 핵심 포인트가 하나 있습니다: **SHA256 검증은 강력한门槛**입니다. 다운로드한 파일은 반드시 검증을 통과해야만 설치 프로세스로 들어갈 수 있습니다. 검증 실패는 캐시를 폐기하고, 「잘못된 파일을 다운로드해서 설치에 문제가 생기는」상황을 방지합니다.

이게 무엇과 같을까요? 바로 신뢰라는 것——한번 배신당하면, 다시 재건하기가 정말 어렵습니다. 그래서 처음부터 문턱을 제대로 세워두는 겁니다.

### 다운로드 엔진 추상화

DownloadEngineAdapter는 추상 인터페이스이고, 엔진이 반드시 구현해야 할 메서드를 정의합니다:

```typescript
interface DownloadEngineAdapter {
  download(
    version: Version,
    destinationPath: string,
    settings: SharingAccelerationSettings,
    onProgress?: (progress: VersionDownloadProgress) => void,
  ): Promise<void>;

  stopAll(): Promise<void>;
}
```

V1 구현은 WebTorrent를 기반으로 하고, InProcessTorrentEngineAdapter에 캡슐화되어 있습니다:

```typescript
class InProcessTorrentEngineAdapter implements DownloadEngineAdapter {
  async download(...) {
    const client = this.getClient(settings);  // 업로드 속도 제한 적용
    const torrent = client.add(torrentId, {
      path: path.dirname(destinationPath),
      destroyStoreOnDestroy: false,
      maxWebConns: 8,
    });

    // WebSeed 추가
    torrent.on('ready', () => {
      for (const seed of hybrid.webSeeds) {
        torrent.addWebSeed(seed);
      }
      if (hybrid.directUrl) {
        torrent.addWebSeed(hybrid.directUrl);
      }
    });

    // 진행률 보고 - P2P와 폴백 구분
    torrent.on('download', () => {
      const hasP2PPeer = torrent.wires.some(w => w.type !== 'webSeed');
      const mode = hasP2PPeer ? 'shared-acceleration' : 'source-fallback';
      // ... 진행률 보고
    });
  }
}
```

엔진의 플러그인 가능한 설계는 미래의 최적화를 단순하게 만듭니다. 예를 들어 V2는 엔진을 헬퍼 프로세스에서 실행해서, 메인 프로세스 크래시 위험을 피할 수 있습니다. 아무래도 한 쥐똥이 한 솥의 죽을 망치지 않게 하고 싶은 거고, 코드 세계도 그렇고, 인생도 마찬가지입니다.

### 진행률 보고의 모드 구분

UI 층에서 사용자가 가장 관심있는 건 「지금 P2P로 다운로드 중인가 HTTP 폴백 중인가」입니다. InProcessTorrentEngineAdapter는 `torrent.wires`의 타입을 검사해서 판단합니다:

```typescript
const hasP2PPeer = torrent.wires.some((wire) => wire.type !== 'webSeed');
const hasFallbackWire = torrent.wires.some((wire) => wire.type === 'webSeed');

const mode = hasP2PPeer ? 'shared-acceleration'
         : hasFallbackWire ? 'source-fallback'
         : 'shared-acceleration';

const stage = hasP2PPeer ? 'downloading'
           : hasFallbackWire ? 'backfilling'
           : 'downloading';
```

이 로직은 단순해 보이지만, 사용자 경험의 핵심입니다. 사용자는 현재 「공유 가속」인지 「폴백 보충」인지 명확히 볼 수 있고, 마음이 편해집니다. 사실 사람 사이도 마찬가지입니다——투명하면 다들 편안합니다.

### SHA256 스트리밍 검증

무결성 검증은 Node.js의 crypto 모듈을 사용하고, 스트리밍 해시 계산을 해서, 전체 파일을 메모리에 로드하는 것을 피합니다:

```typescript
private async computeSha256(filePath: string): Promise<string> {
  const hash = createHash('sha256');
  await new Promise<void>((resolve, reject) => {
    const stream = fs.createReadStream(filePath);
    stream.on('data', (chunk) => hash.update(chunk));
    stream.on('error', reject);
    stream.on('end', resolve);
  });
  return hash.digest('hex').toLowerCase();
}
```

이 구현은 대용량 파일에 특히 친절합니다. 생각해 보세요, 2GB 설치 패키지를 다운로드하고, 전체 파일을 메모리에 읽어서 검증하면, 메모리 점유가 얼마나 무서워질까요? 스트리밍 처리가 이 문제를 완벽하게 해결합니다.

이게 감정과 비슷하지 않나요? 어떤 건, 한 번에 전부를 가질 필요가 없고, 조금씩 가져오는 게 더 낫습니다.

## 제품화 포장

기술 솔루션이 아무리 좋아도, 사용자 경험이 안 좋으면 소용없습니다. HagiCode Desktop은 제품화에 적지 않은 일을 했습니다. 기술은 뼈대 안의 일이고, 제품은 가죽인데, 가죽이 안 좋으면, 뼈대가 아무리 좋아도 아무도 더 보려고 하지 않습니다.

### BT 용어 숨기기

대부분의 사용자는 BitTorrent가 뭔지, InfoHash가 뭔지 모릅니다. 그래서 제품 차원에서 「공유 가속」이라는 의미를 사용합니다:

- 기능은 「P2P 다운로드」가 아니라 「공유 가加速」라고 부릅니다
- 설정 항목은 「시딩」이 아니라 「업로드 속도 제한」이라고 합니다
- 진행률은 「WebSeed 폴백」이 아니라 「폴백 보충」이라고 표시합니다

이렇게 하면 용어의 인지 부담이 줄어듭니다. 사실 말하는 것도 예술이고, 간단하게 하면 다들 편합니다.

### 첫 번째 안내에서 기본 활성화

새 사용자가 처음 데스크톱을 사용할 때, 공유 가속 기능을 소개하는 안내 페이지를 하나 보게 됩니다:

> 다운로드 속도를 높이기 위해, 다운로드하는 동안 이미 다운로드한 파일의 일부를 다른 사용자와 공유합니다. 이 과정은 완전히 선택 사항이고, 설정에서 언제든지 끌 수 있습니다.

기본값은 켜져 있지만, 명확한 취소入口를 제공합니다. 기업 사용자가 필요 없으면, 안내에서 바로 끄면 됩니다. 선택권은 사용자에게 있고, 누구도 강요받는 건 싫어합니다.

### 사용자 제어 가능한 매개변수

설정 페이지는 세 가지 조정 가능한 매개변수를 제공합니다:

| 매개변수 | 기본값 | 설명 |
|------|--------|------|
| 업로드 속도 제한 | 2 MB/s | 상향 대역폭 과도 점유 방지 |
| 캐시 상한 | 10 GB | 디스크 공간 점유 제어 |
| 보관 일수 | 7 일 | 이 시간을 초과하면 자동으로 캐시 정리 |

이 매개변수들은 합리적인 기본값을 가지고 있고, 일반 사용자는 수정할 필요가 없고, 고급 사용자는 자신의 네트워크 환경에 따라 조정할 수 있습니다. 아무래도 식구들은 각자 다르니까, 좀 자유를 주는 게 좋습니다.

## 핵심 설계 결정

전체 솔루션을 회고해 보면, 몇 가지 핵심 결정이 언급할 만합니다:

### 엔진을 메인 프로세스 내에 두기(V1)

왜 처음부터 사이드카/헬퍼 프로세스를 안 만들었을까요? 이유는 간단합니다: **빠른 출시**. 메인 프로세스 내 솔루션은 개발 주기가 짧고, 디버깅이 편해서, 먼저 기능을 돌려놓고, 나중에 안정성 최적화를 고려합니다.

물론, 이 결정에는 대가가 있습니다: 엔진 크래시가 메인 프로세스에 영향을 줍니다. 그래서 어댑터 경계와 타임아웃 제어로 이 문제를 완화합니다. 동시에 마이그레이션 경로를 예약해 두고, V2는 쉽게 독립 프로세스로 마이그레이션할 수 있습니다.

이게 젊은 시절의 우리와 비슷하지 않나요? 일단 타고 보고, 뒤의 일은 나중에 다시 생각합니다. 아무래도 너무 많이 생각하면 발을 떼기 어려울 때가 있습니다.

### 무결성 검증으로 SHA256 사용

MD5나 CRC32를 안 쓰고 SHA256을 쓰는 건, SHA256이 더 안전하기 때문입니다. MD5와 CRC32의 충돌 비용이 너무 낮고, 만약 누가 악의적으로 가짜 설치 패키지를 만들면,后果는 상상하기 어렵습니다. SHA256의 계산 비용이 좀 크지만, 보안성은 그 대가를 치를 만합니다.

신뢰라는 건, 만들기 어렵고, 무너지는 건 찰나입니다. 그래서 안전을 선택할 수 있을 때는, 그 비용을 아끼지 마세요.

### HTTP index에만 활성화

GitHub 다운로드, 로컬 폴더 소스 같은 시나리오는 하이브리드 배포를 쓰지 않습니다. 이건 기술 제한이 아니라, **복잡화 피하기**입니다. BT 프로토콜은 사설 네트워크에서 가치가 별로 없고, 불필요한 코드 복잡성을 늘립니다.

어떤 서클은 억지로 섞일 필요가 없습니다. 이치가 이렇게 간단합니다.

## 실전 포인트

### 설정 정규화

SharingAccelerationSettingsStore에서 모든 수치는 경계 검사와 정규화를 해야 합니다:

```typescript
private normalize(settings: SharingAccelerationSettings): SharingAccelerationSettings {
  return {
    enabled: Boolean(settings.enabled),
    uploadLimitMbps: this.clampNumber(settings.uploadLimitMbps, 1, 200, DEFAULT_SETTINGS.uploadLimitMbps),
    cacheLimitGb: this.clampNumber(settings.cacheLimitGb, 1, 500, DEFAULT_SETTINGS.cacheLimitGb),
    retentionDays: this.clampNumber(settings.retentionDays, 1, 90, DEFAULT_SETTINGS.retentionDays),
    hybridThresholdMb: DEFAULT_SETTINGS.hybridThresholdMb,  // 고정값, 사용자가 수정하지 못하게
    onboardingChoiceRecorded: Boolean(settings.onboardingChoiceRecorded),
  };
}

private clampNumber(value: number, min: number, max: number, fallback: number): number {
  if (!Number.isFinite(value)) {
    return fallback;
  }
  return Math.min(max, Math.max(min, Math.round(value)));
}
```

이렇게 하면 사용자가 수동으로 설정 파일을 수정해서 이상한 값이 들어가는 것을 방지할 수 있습니다. 아무래도 사용자가 어떤 이상한 숫자를 입력할지 모르고, 저도 설정 스크린샷을 보고 싶지 않은데, 방법이 없네요.

### 캐시 LRU 정리

CacheRetentionManager.prune() 메서드는 만료 및 초과 캐시를 정리합니다. 정리 정책은 LRU(가장 최근에 사용하지 않은)입니다:

```typescript
const records = [...this.listRecords()]
  .sort((left, right) =>
    new Date(left.lastUsedAt).getTime() - new Date(right.lastUsedAt).getTime()
  );

// 상한 초과 시 정리할 때, 가장 오랫동안 사용하지 않은 것부터 삭제
while (totalBytes > maxBytes && retainedEntries.length > 0) {
  const evicted = records.find((record) => retainedEntries.includes(record.versionId));
  retainedEntries.splice(retainedEntries.indexOf(evicted.versionId), 1);
  removedEntries.push(evicted.versionId);
  totalBytes -= evicted.cacheSize;
  await fs.rm(evicted.cachePath, { force: true });
}
```

이 로직은 디스크 공간이 합리적으로 사용되면서, 사용자가 아직 필요할 수도 있는历史版本을 보존합니다. 아무래도 자주 안 쓰지만, 잃어버리면 아까운 것들도 있고, 사람은 원래 그런 거니까요.

### 즉시 시딩 중지 구현

사용자가 공유 가속 스위치를 끄면, 즉시 시딩을 중단하고 토렌트 클라이언트를 파괴해야 합니다:

```typescript
async disableSharingAcceleration(): Promise<void> {
  this.settingsStore.updateSettings({ enabled: false });
  await this.cacheRetentionManager.stopAllSeeding();  // 시딩 중지
  await this.engine.stopAll();  // 토렌트 클라이언트 파괴
}
```

사용자가 기능을 끄면, 더 이상 어떤 P2P 리소스도 점유하면 안 되고, 이건 기본적인 제품 예의입니다. 더 이상 사랑하지 않으면, 시원하게 놓아주고, 질질 끌지 마세요.

## 위험과 트레이드오프

세상에 완벽한 솔루션은 없고, 하이브리드 배포도 예외가 아닙니다. 주요 트레이드오프 포인트는 다음과 같습니다:

**사이드카보다 크래시 격리가 약함**: V1은 메인 프로세스 내 엔진을 사용하고, 엔진 크래시가 메인 프로세스에 영향을 줍니다. 이건 어댑터 경계와 타임아웃 제어로 완화하지만, 근본적인 해결책은 아닙니다. V2는 헬퍼 프로세스 마이그레이션 경로를 계획했습니다. 아무래도 초보자가 도로에 나가면, 좀 수업료를 내야 합니다.

**기본 활성화로 인한 리소스 점유**: 기본 2 MB/s 업로드, 10 GB 캐시, 7 일 보존은, 사용자 머신에 일정 리소스 소모가 있습니다. 안내 설명과 설정 투명도로 사용자 기대를 관리합니다. 아무래도 공짜 점심은 없고, 얻는 게 있으면 포기하는 게 있습니다.

**기업 네트워크 호환성**: WebSeed/HTTPS 자동 폴백은 기업 네트워크下的 가용성을 보장하지만, P2P 가속 효과는 할인됩니다. 이건 설계上的取舍이고, 가용성을 우선합니다. 아무래도 어떤 일은 빠른 것보다 더 중요한 게, 예를 들어 안정성.

**메타데이터 하위 호환성**: 모든 새 필드는 선택적이고, 누락 시 HTTP 모드로 폴백합니다. 구버전 클라이언트는 전혀 영향을 받지 않고, 업그레이드 경로가 매끄럽습니다. 아무래도 업그레이드할 때마다 한 번씩 터지는 건, 너무 자극적입니다.

이 글은 HagiCode Desktop 프로젝트의 하이브리드 배포 아키텍처를 상세히 분석하고, 요약하면 다음 몇 가지 핵심 포인트가 있습니다:

1. **아키텍처 계층화**: 제어 플레인과 데이터 플레인 분리, 엔진을 플러그인 가능한 인터페이스로 추상화, 테스트와 확장에 용이합니다. 분명한 분업은 효율이 높습니다.

2. **정책 기반**: 모든 파일이 P2P로 가는 게 아니라, 조건을 만족하는 대용량 파일만 하이브리드 배포를 활성화합니다. 억지로 맞춘 건 맛이 없고, 적합한 게 제일 중요합니다.

3. **무결성 검증**: SHA256을 강력한门槛으로, 스트리밍 계산으로 메모리 문제를 피합니다. 신뢰 구축이 쉽지 않으니, 쓸 때마다 소중히 하세요.

4. **제품화 포장**: BT 용어를 숨기고 「공유 가속」 의미를 사용, 첫 안내에서 기본 활성화. 말하는 것도 예술이고, 간단하게 하면 다들 편합니다.

5. **사용자 제어**: 업로드 속도 제한, 캐시 상한, 보관 일수 등 조정 가능한 매개변수를 제공합니다. 선택권은 사용자에게 있고, 누구도 강요받는 건 싫어합니다.

이 솔루션은 이미 HagiCode Desktop 프로젝트에落地되어 실시되었고, 실제 효과가 어떤지, 설치体验 후 피드백 환영합니다. 이론은 이론이고, 실천이야 진리를 검증하는 유일한 표준입니다.
