## HagiCode Desktop 하이브리드 배포 아키텍처 해설: P2P가 대용량 파일 다운로드를 가속하는 방법

> 이 글을 오래 참았다가 마침내 썼습니다. 읽기 좋은지는 아직 확신할 수 없습니다. 기술 글은 쓰기는 쉽지만, 정말 흥미롭게 만들기는 어렵습니다. 그래도 저는 위대한 문장가가 아니므로, 그냥 이 소박한 설명을 적어두겠습니다.

## 배경

데스크톱 애플리케이션을 구축하는 팀은 모두 sooner or later 같은 두통을 겪게 됩니다. 대용량 파일을 어떻게 배포할 것인가요?

이것은 난감한 문제입니다. 전통적인 HTTP/HTTPS 직접 다운로드는 파일이 작고 사용자 수가 제한적일 때는 견딜 수 있습니다. 하지만 시간은 결코 자비롭지 않습니다. 프로젝트가 계속 성장함에 따라 설치 패키지도 함께 커집니다. Desktop ZIP 패키지, 포터블 패키지, 웹 배포 아카이브 등. 그러면 문제들이 표면으로 떠오르기 시작합니다:

- **다운로드 속도는 오리진 대역폭에 제한됩니다**: 단일 서버의 대역폭이 아무리 넓어도, 모두가 동시에 다운로드하면 버겁습니다.
- **이어받기 지원이 거의 없습니다**: HTTP 다운로드가 중단되면 처음부터 다시 시작해야 하는 경우가 많습니다. 시간과 대역폭 모두 낭비입니다.
- **오리진 서버가 모든 압력을 받습니다**: 모든 트래픽이 중앙 서버로 향하고, 대역폭 비용은 계속 상승하며, 확장성은 진정한 문제가 됩니다.

HagiCode Desktop 프로젝트도 예외가 아니었습니다. 배포 시스템을 설계할 때, 우리는 계속 자문했습니다. 기존 `index.json` 제어 평면을 변경하지 않고 하이브리드 배포 방식을 도입할 수 있을까요? 즉, P2P 네트워크의 분산 특성을 활용하여 다운로드를 가속하면서, 기업 네트워크와 같은 제한된 환경에서도 시스템이 사용 가능하도록 HTTP 오리진 폴백을 유지할 수 있을까요?

그 결정의 영향은 예상보다 컸습니다. 단계별로 살펴보겠습니다.

## HagiCode 소개

이 글에서 공유하는 접근 방식은 [HagiCode](https://hagicode.com) 프로젝트의 실제 경험에서 나온 것입니다. HagiCode는 개발 팀의 엔지니어링 효율을 높이는 데 집중하는 오픈소스 AI 코딩 어시스턴트 프로젝트입니다. 프로젝트는 프론트엔드, 백엔드, 데스크톱 런처, 문서, 빌드 파이프라인, 서버 배포를 포함한 여러 서브시스템에 걸쳐 있습니다.

Desktop 하이브리드 배포 아키텍처는 HagiCode가 실제 운영 경험과 반복적인 최적화를 통해 다듬은 바로 그런 해결책입니다. 이 설계가 유용하다면, HagiCode 자체도 주목할 가치가 있다는 뜻일 것입니다.

프로젝트의 GitHub 저장소는 [HagiCode-org/site](https://github.com/HagiCode-org/site)입니다. 관심이 있으시다면 Star를 누르고 나중에 다시 보세요.

## 핵심 설계 철학: P2P 우선, HTTP 폴백

하이브리드 배포 모델의 본질은 한 문장으로 요약할 수 있습니다: **P2P 우선, HTTP 폴백**.

핵심은 "하이브리드"라는 단어에 있습니다. 이것은 단순히 BitTorrent를 추가하고 끝내는 것이 아닙니다. 두 전달 방식이 함께 작동하고 서로를 보완하도록 만드는 것입니다:

- **P2P 네트워크**는 분산 가속을 제공합니다. 다운로드하는 사람이 많을수록 피어가 많이 참여하고, 전송이 더 빨라집니다.
- **WebSeed/HTTP 폴백**은 가용성을 보장하여, 기업 방화벽 및 내부 네트워크 환경에서도 다운로드가 작동합니다.
- **제어 평면은 단순하게 유지됩니다.** `index.json`의 핵심 로직은 변경하지 않고, 몇 가지 선택적 메타데이터 필드만 추가합니다.

실제 이점은 단순합니다. 사용자는 "다운로드가 더 빨라졌다"고 느끼고, 엔지니어링 팀은 너무 많은 추가 복잡성을 감당하지 않아도 됩니다. BT 프로토콜은 이미 성숙했으며, 바퀴를 재발명할 이유가 거의 없습니다.

## 아키텍처 설계

### 계층화 아키텍처 개요

전체 아키텍처 다이어그램으로 시작하여 높은 수준의 멘탈 모델을 구축해 보겠습니다:

```
┌─────────────────────────────────────┐
│     Renderer (UI 계층)             │
├─────────────────────────────────────┤
│     IPC/Preload (브릿지 계층)      │
├─────────────────────────────────────┤
│   VersionManager (버전 관리자)  │
├─────────────────────────────────────┤
│ HybridDownloadCoordinator (조정)  │
│  ├── DistributionPolicyEvaluator    │
│  ├── DownloadEngineAdapter          │
│  ├── CacheRetentionManager          │
│  └── SHA256 Verifier                │
├─────────────────────────────────────┤
│   WebTorrent (다운로드 엔진)      │
└─────────────────────────────────────┘
```

다이어그램에서 보듯이, 시스템은 계층화 설계를 사용합니다. 이렇게 명확하게 책임을 분리하는 이유는 간단합니다: **테스트 가능성과 교체 가능성**입니다.

- **UI 계층**은 다운로드 진행 상황과 공유 가속 토글을 표시합니다. 표면입니다.
- **조정 계층**이 핵심입니다. 정책 평가, 엔진 적응, 캐시 관리 및 무결성 검증을 포함합니다.
- **엔진 계층**은 구체적인 다운로드 구현을 캡슐화합니다. 현재 WebTorrent를 사용합니다.

엔진 계층은 `DownloadEngineAdapter` 인터페이스 뒤에 추상화되어 있습니다. 나중에 다른 BT 엔진으로 교체하거나 구현을 사이드카 프로세스로 이동하려면 훨씬 쉬워집니다.

### 제어 평면과 데이터 평면의 분리

HagiCode Desktop은 `index.json`을 유일한 제어 평면으로 유지하며, 이 설계가 중요합니다. 제어 평면은 버전 발견, 채널 선택 및 중앙 집중식 정책을 담당하고, 데이터 평면은 실제 파일 전송이 일어나는 곳입니다.

`index.json`에 추가된 새 필드는 선택적입니다:

```json
{
  "asset": {
    "torrentUrl": "https://cdn.example.com/app.torrent",
    "infoHash": "abc123...",
    "webSeeds": [
      "https://cdn.example.com/app.zip",
      "https://backup.example.com/app.zip"
    ],
    "sha256": "def456...",
    "directUrl": "https://cdn.example.com/app.zip"
  }
}
```

이 필드들은 모두 선택적입니다. 누락되면 클라이언트는 전통적인 HTTP 다운로드 모드로 폴백합니다. 이 설계의 장점은 **하위 호환성**입니다. 이전 클라이언트는 전혀 영향을 받지 않습니다.

### 정책 기반 결정

모든 파일이 P2P를 통해 배포할 가치가 있는 것은 아닙니다.

`DistributionPolicyEvaluator`는 정책을 평가합니다. 다음 조건을 모두 충족하는 파일만 하이브리드 다운로드를 사용합니다:

1. **소스 유형이 HTTP 인덱스여야 합니다**: 직접 GitHub 다운로드나 로컬 폴더 소스는 이 경로를 사용하지 않습니다.
2. **파일 크기가 최소 100MB 이상이어야 합니다**: 더 작은 파일의 경우 P2P의 오버헤드가 이점보다 큽니다.
3. **완전한 하이브리드 메타데이터가 있어야 합니다**: `torrentUrl`, `webSeeds`, `sha256`이 모두 필요합니다.
4. **최신 데스크톱 패키지와 웹 배포 패키지만 해당합니다**: 이전 버전은 전통적인 배포 경로를 계속 사용합니다.

```typescript
class DistributionPolicyEvaluator {
  evaluate(version: Version, settings: SharingAccelerationSettings): HybridDownloadPolicy {
    // 소스 유형 확인
    if (version.sourceType !== 'http-index') {
      return { useHybrid: false, reason: 'not-http-index' };
    }

    // 메타데이터 완전성 확인
    if (!version.hybrid) {
      return { useHybrid: false, reason: 'not-eligible' };
    }

    // 기능 활성화 여부 확인
    if (!settings.enabled) {
      return { useHybrid: false, reason: 'shared-disabled' };
    }

    // 에셋 유형 확인 (최신 데스크톱/웹 패키지만)
    if (!version.hybrid.isLatestDesktopAsset && !version.hybrid.isLatestWebAsset) {
      return { useHybrid: false, reason: 'latest-only' };
    }

    return { useHybrid: true, reason: 'shared-enabled' };
  }
}
```

이로써 시스템은 **예측 가능한 동작**을 갖게 됩니다. 개발자와 사용자 모두 어떤 파일이 P2P를 사용하고 어떤 파일이 그렇지 않은지 명확히 이해할 수 있습니다.

## 핵심 구현

### 타입 정의 시스템

타입 정의부터 시작하겠습니다. 전체 시스템의 기초가 되기 때문입니다.

```typescript
// 하이브리드 배포 메타데이터
interface HybridDistributionMetadata {
  torrentUrl?: string;      // 토렌트 파일 URL
  infoHash?: string;        // InfoHash
  webSeeds: string[];       // WebSeed 목록
  sha256?: string;          // 파일 해시
  directUrl?: string;       // HTTP 직접 링크 (오리진 폴백용)
  eligible: boolean;        // 하이브리드 배포 적용 여부
  thresholdBytes: number;   // 바이트 단위 임계값
  assetKind: VersionAssetKind;
  isLatestDesktopAsset: boolean;
  isLatestWebAsset: boolean;
}

// 공유 가속 설정
interface SharingAccelerationSettings {
  enabled: boolean;           // 마스터 스위치
  uploadLimitMbps: number;    // 업로드 대역폭 제한
  cacheLimitGb: number;       // 캐시 제한
  retentionDays: number;      // 보존 기간
  hybridThresholdMb: number;  // 하이브리드 배포 임계값
  onboardingChoiceRecorded: boolean;
}

// 다운로드 진행 상황
interface VersionDownloadProgress {
  current: number;
  total: number;
  percentage: number;
  stage: VersionInstallStage;  // queued, downloading, backfilling, verifying, extracting, completed, error
  mode: VersionDownloadMode;   // http-direct, shared-acceleration, source-fallback
  peers?: number;              // 연결된 피어 수
  p2pBytes?: number;           // P2P에서 수신한 바이트
  fallbackBytes?: number;      // 폴백에서 수신한 바이트
  verified?: boolean;          // 검증 완료 여부
}
```

타입 시스템이 명확해지면 나머지 구현은 자연스럽게 따라옵니다.

### 핵심 코디네이터

`HybridDownloadCoordinator`는 전체 다운로드 워크플로를 조율합니다. 정책 평가, 엔진 실행, SHA256 검증 및 캐시 관리를 조정합니다.

```typescript
class HybridDownloadCoordinator {
  async download(
    version: Version,
    cachePath: string,
    packageSource: PackageSource,
    onProgress?: DownloadProgressCallback,
  ): Promise<HybridDownloadResult> {
    // 1. 정책 평가: 하이브리드 다운로드를 사용해야 합니까?
    const policy = this.policyEvaluator.evaluate(version, settings);

    // 2. 다운로드 실행
    if (policy.useHybrid) {
      await this.engine.download(version, cachePath, settings, onProgress);
    } else {
      await packageSource.downloadPackage(version, cachePath, onProgress);
    }

    // 3. SHA256 검증 (하드 게이트)
    const verified = await this.verify(version, cachePath, onProgress);
    if (!verified) {
      await this.cacheRetentionManager.discard(version.id, cachePath);
      throw new Error(`sha256 verification failed for ${version.id}`);
    }

    // 4. 신뢰할 수 있는 캐시로 표시 및 제어된 시딩 시작
    await this.cacheRetentionManager.markTrusted({
      versionId: version.id,
      cachePath,
      cacheSize,
    }, settings);

    return { cachePath, policy, verified };
  }
}
```

여기에 특히 중요한 점이 있습니다: **SHA256 검증은 하드 게이트입니다.** 다운로드한 파일은 설치 흐름에 들어가기 전에 검증을 통과해야 합니다. 검증에 실패하면 캐시가 폐기되어, 잘못된 파일이 설치 문제를 일으키지 않도록 합니다.

### 다운로드 엔진 추상화

`DownloadEngineAdapter`는 모든 엔진이 구현해야 하는 메서드를 정의하는 추상 인터페이스입니다:

```typescript
interface DownloadEngineAdapter {
  download(
    version: Version,
    destinationPath: string,
    settings: SharingAccelerationSettings,
    onProgress?: (progress: VersionDownloadProgress) => void,
  ): Promise<void>;

  stopAll(): Promise<void>;
}
```

V1 구현은 WebTorrent를 기반으로 하며, `InProcessTorrentEngineAdapter`에 래핑되어 있습니다:

```typescript
class InProcessTorrentEngineAdapter implements DownloadEngineAdapter {
  async download(...) {
    const client = this.getClient(settings);  // 업로드 속도 제한 적용
    const torrent = client.add(torrentId, {
      path: path.dirname(destinationPath),
      destroyStoreOnDestroy: false,
      maxWebConns: 8,
    });

    // WebSeed 소스 추가
    torrent.on('ready', () => {
      for (const seed of hybrid.webSeeds) {
        torrent.addWebSeed(seed);
      }
      if (hybrid.directUrl) {
        torrent.addWebSeed(hybrid.directUrl);
      }
    });

    // 진행 상황 보고 - P2P와 오리진 폴백 구분
    torrent.on('download', () => {
      const hasP2PPeer = torrent.wires.some(w => w.type !== 'webSeed');
      const mode = hasP2PPeer ? 'shared-acceleration' : 'source-fallback';
      // ... 진행 상황 보고
    });
  }
}
```

플러그인 가능한 엔진 설계는 향후 최적화를 훨씬 쉽게 만듭니다. 예를 들어 V2에서는 엔진을 헬퍼 프로세스에서 실행하여 엔진 충돌 시 메인 프로세스가 중단되지 않도록 할 수 있습니다.

### 진행 상황 보고 모드 구분

UI 계층에서 사용자가 가장 신경 쓰는 것은 간단합니다. "현재 P2P로 다운로드 중인가요, 아니면 HTTP 폴백으로 다운로드 중인가요?" `InProcessTorrentEngineAdapter`는 `torrent.wires` 내부의 유형을 확인하여 이를 판단합니다:

```typescript
const hasP2PPeer = torrent.wires.some((wire) => wire.type !== 'webSeed');
const hasFallbackWire = torrent.wires.some((wire) => wire.type === 'webSeed');

const mode = hasP2PPeer ? 'shared-acceleration'
         : hasFallbackWire ? 'source-fallback'
         : 'shared-acceleration';

const stage = hasP2PPeer ? 'downloading'
           : hasFallbackWire ? 'backfilling'
           : 'downloading';
```

로직은 간단해 보이지만, 사용자 경험의 핵심 부분입니다. 사용자는 현재 상태가 "공유 가속"인지 "오리진 백필링"인지 명확히 볼 수 있어, 동작을 더 쉽게 이해할 수 있습니다.

### SHA256 스트리밍 검증

무결성 검증은 Node.js의 `crypto` 모듈을 사용하여 스트리밍 방식으로 해시를 계산하며, 전체 파일을 메모리에 로드하는 것을 피합니다:

```typescript
private async computeSha256(filePath: string): Promise<string> {
  const hash = createHash('sha256');
  await new Promise<void>((resolve, reject) => {
    const stream = fs.createReadStream(filePath);
    stream.on('data', (chunk) => hash.update(chunk));
    stream.on('error', reject);
    stream.on('end', resolve);
  });
  return hash.digest('hex').toLowerCase();
}
```

이 구현은 대용량 파일에 특히 친화적입니다. 2GB 설치 패키지를 다운로드한 다음 검증하기 위해 전체를 메모리에 로드하는 것을 상상해 보십시오. 스트리밍이 이를 깔끔하게 해결합니다.

## 데이터 흐름

전체 데이터 흐름은 다음과 같습니다:


## 제품화

아무리 좋은 기술 설계라도 사용자 경험이 나쁘면 무용지물입니다. HagiCode Desktop은 이 기능을 제품화하는 데 상당한 노력을 투자했습니다.

### BT 용어 숨기기

대부분의 사용자는 BitTorrent나 InfoHash가 무엇인지 모릅니다. 그래서 제품 수준에서는 "공유 가속"이라는 문구로 기능을 표현합니다:

- 기능 이름은 P2P 다운로드가 아니라 "공유 가속"입니다.
- 설정 이름은 시딩이 아니라 "업로드 제한"입니다.
- 진행 상태 라벨은 WebSeed 폴백이 아니라 "오리진 백필링"입니다.

이는 용어의 인지 부하를 낮추고 기능을 더 쉽게 수용할 수 있게 합니다.

### 첫 실행 마법사에서 기본 활성화

새 사용자가 데스크톱 앱을 처음 실행하면, 공유 가속을 소개하는 마법사 페이지가 표시됩니다:

> 다운로드 속도를 향상시키기 위해, 본인의 다운로드가 진행되는 동안 이미 다운로드한 부분을 다른 사용자와 공유합니다. 이는 완전히 선택 사항이며, 언제든 설정에서 끌 수 있습니다.

기본적으로 활성화되지만, 사용자에게 명확한 옵트아웃 방법이 제공됩니다. 기업 사용자가 원하지 않으면 온보딩 중에 비활성화할 수 있습니다.

### 사용자 제어 매개변수

설정 페이지에는 세 가지 조정 가능한 매개변수가 표시됩니다:

| 매개변수 | 기본값 | 설명 |
|------|--------|------|
| 업로드 제한 | 2 MB/s | 과도한 업스트림 대역폭 사용 방지 |
| 캐시 제한 | 10 GB | 디스크 공간 소비 제어 |
| 보존 일수 | 7일 | 이 기간 후 자동으로 이전 캐시 정리 |

이 매개변수들은 모두 합리적인 기본값을 가집니다. 대부분의 사용자는 변경할 필요가 없으며, 고급 사용자는 자신의 네트워크 환경에 따라 조정할 수 있습니다.

## 핵심 설계 결정

전체 해결책을 돌아보며, 몇 가지 설계 결정이 언급할 가치가 있습니다.

### 엔진을 메인 프로세스에서 실행 (V1)

왜 처음부터 사이드카나 헬퍼 프로세스를 시작하지 않았을까요? 이유는 간단합니다: **빨리 출시하기 위해**. 인프로세스 설계는 개발 주기가 짧고 디버깅이 더 쉽습니다. 첫 번째 우선순위는 기능을 실행하게 만드는 것이고, 안정성은 그 다음입니다.

물론 이 결정에는 대가가 있습니다. 엔진이 충돌하면 메인 프로세스에 영향을 미칠 수 있습니다. 어댑터 경계와 시간 초과 제어를 통해 위험을 줄이며, V2에서 별도 프로세스로 더 쉽게 마이그레이션할 수 있도록 마이그레이션 경로도 열어두었습니다.

### 무결성 검증으로 SHA256 사용

MD5나 CRC32 대신 SHA256을 사용합니다. SHA256이 더 안전하기 때문입니다. MD5와 CRC32의 충돌 비용이 너무 낮습니다. 악의적으로 가짜 설치 패키지를 만들면 결과가 심각할 수 있습니다. SHA256은 계산 비용이 더 많지만, 보안 이득은 그만한 가치가 있습니다.

### HTTP 인덱스 소스에만 활성화

GitHub 다운로드 및 로컬 폴더 소스와 같은 시나리오에서는 하이브리드 배포를 사용하지 않습니다. 이것은 기술적 제한이 아니라 **불필요한 복잡성을 피하기 위한 것**입니다. BT 프로토콜은 사설 네트워크 시나리오에서 제한된 가치를 추가하며, 코드 복잡성만 증가시킵니다.

## 실무 참고 사항

### 설정 정규화

`SharingAccelerationSettingsStore` 내에서 모든 숫자 값은 범위 검사 및 정규화를 거쳐야 합니다:

```typescript
private normalize(settings: SharingAccelerationSettings): SharingAccelerationSettings {
  return {
    enabled: Boolean(settings.enabled),
    uploadLimitMbps: this.clampNumber(settings.uploadLimitMbps, 1, 200, DEFAULT_SETTINGS.uploadLimitMbps),
    cacheLimitGb: this.clampNumber(settings.cacheLimitGb, 1, 500, DEFAULT_SETTINGS.cacheLimitGb),
    retentionDays: this.clampNumber(settings.retentionDays, 1, 90, DEFAULT_SETTINGS.retentionDays),
    hybridThresholdMb: DEFAULT_SETTINGS.hybridThresholdMb,  // 고정 값, 사용자 구성 불가
    onboardingChoiceRecorded: Boolean(settings.onboardingChoiceRecorded),
  };
}

private clampNumber(value: number, min: number, max: number, fallback: number): number {
  if (!Number.isFinite(value)) {
    return fallback;
  }
  return Math.min(max, Math.max(min, Math.round(value)));
}
```

이렇게 하면 사용자가 수동으로 구성 파일을 유효하지 않은 값으로 편집하는 것을 방지합니다.

### 캐시 LRU 정리

`CacheRetentionManager.prune()`은 만료되었거나 크기가 초과된 캐시 항목을 정리합니다. 정리 전략은 LRU(가장 최근에 사용되지 않은 항목 우선)를 사용합니다:

```typescript
const records = [...this.listRecords()]
  .sort((left, right) =>
    new Date(left.lastUsedAt).getTime() - new Date(right.lastUsedAt).getTime()
  );

// 한도를 초과하면 가장 오래된 항목을 먼저 제거
while (totalBytes > maxBytes && retainedEntries.length > 0) {
  const evicted = records.find((record) => retainedEntries.includes(record.versionId));
  retainedEntries.splice(retainedEntries.indexOf(evicted.versionId), 1);
  removedEntries.push(evicted.versionId);
  totalBytes -= evicted.cacheSize;
  await fs.rm(evicted.cachePath, { force: true });
}
```

이 로직은 디스크 공간이 효율적으로 사용되도록 하면서, 사용자가 여전히 필요할 수 있는 과거 버전을 보존합니다.

### 즉시 시딩 중지 동작

사용자가 공유 가속을 끄면, 앱은 즉시 시딩을 중지하고 토렌트 클라이언트를 파기해야 합니다:

```typescript
async disableSharingAcceleration(): Promise<void> {
  this.settingsStore.updateSettings({ enabled: false });
  await this.cacheRetentionManager.stopAllSeeding();  // 시딩 중지
  await this.engine.stopAll();  // 토렌트 클라이언트 파기
}
```

사용자가 기능을 비활성화하면, 제품은 더 이상 P2P 리소스를 소비해서는 안 됩니다. 기본적인 제품 예의입니다.

## 위험 및 트레이드오프

완벽한 해결책은 없으며, 하이브리드 배포도 예외가 아닙니다. 주요 트레이드오프는 다음과 같습니다:

**사이드카보다 충돌 격리가 약함**: V1은 인프로세스 엔진을 사용하므로, 엔진 충돌이 메인 프로세스에 영향을 미칠 수 있습니다. 어댑터 경계와 시간 초과 제어는 위험을 줄이지만, 근본적인 해결책은 아닙니다. V2에는 헬퍼 프로세스로의 마이그레이션 경로가 계획되어 있습니다.

**기본 활성화 시 리소스 사용**: 2MB/s 업로드, 10GB 캐시, 7일 보존의 기본 설정은 일정한 머신 리소스를 소비합니다. 사용자의 기대는 온보딩 카피와 투명한 설정을 통해 관리됩니다.

**기업 네트워크 호환성**: 자동 WebSeed/HTTPS 폴백은 기업 네트워크에서의 가용성을 보존하지만, P2P의 가속 이점을 줄일 수 있습니다. 가용성을 우선시하는 의도적인 트레이드오프입니다.

**하위 호환 가능한 메타데이터**: 모든 새 필드는 선택적입니다. 누락되면 시스템은 HTTP 모드로 폴백합니다. 이전 클라이언트는 전혀 영향을 받지 않으므로, 업그레이드가 원활합니다.

이 글에서는 HagiCode Desktop 프로젝트에서 사용된 하이브리드 배포 아키텍처를 살펴보았습니다. 핵심 요점은 다음과 같습니다:

1. **계층화 아키텍처**: 제어 평면과 데이터 평면이 분리되고, 엔진은 플러그인 가능한 인터페이스 뒤에 추상화되어 테스트와 확장이 더 쉽습니다.

2. **정책 기반 동작**: 모든 파일이 P2P를 사용하는 것은 아닙니다. 하이브리드 배포는 필요한 조건을 충족하는 대용량 파일에만 활성화됩니다.

3. **무결성 검증**: SHA256이 하드 게이트로 작동하며, 스트리밍 검증이 메모리 압력을 피합니다.

4. **제품화된 표현**: BT 용어는 "공유 가속"이라는 문구 뒤에 숨겨지고, 기능은 온보딩 중 기본적으로 활성화됩니다.

5. **사용자 제어**: 업로드 제한, 캐시 제한, 보존 일수 등의 매개변수는 사용자가 조정할 수 있습니다.
