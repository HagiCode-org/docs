## VSCode와 code-server: 브라우저 기반 코드 편집 솔루션 선택하기

> 브라우저 기반 코드 편집 기능을 구축할 때 개발자는 핵심적인 선택에 직면합니다. VSCode의 공식 `code serve-web` 기능을 사용할 것인지, 커뮤니티 주도의 `code-server` 솔루션을 채택할 것인지? 이 결정은 기술 아키텍처뿐만 아니라 라이선스 준수와 배포 유연성에도 영향을 미칩니다.

## 배경

기술 선택은 인생에서 길을 선택하는 것과 비슷합니다. 하나를 선택하면 보통 계속 걸어가야 하며, 나중에 전환하면 비용이 매우 커질 수 있습니다.

AI 보조 프로그래밍 시대에 브라우저 기반 코드 편집은 점점 더 중요해지고 있습니다. 사용자는 AI 어시스턴트가 코드 분석을 마친 후 동일한 브라우저 세션에서 즉시 에디터를 열고 애플리케이션을 전환하지 않고 변경할 수 있기를 기대합니다. 그런 매끄러운 경험은 필요할 때 그저 거기에 있어야 합니다.

하지만 이 기능을 구현할 때 개발자는 중요한 기술적 선택에 직면합니다. VSCode의 공식 `code serve-web` 기능을 사용해야 할까요, 아니면 커뮤니티 주도의 `code-server` 솔루션을 사용해야 할까요?

각 옵션에는 고유한 장점과 트레이드오프가 있으며, 잘못 선택하면 나중에 많은 문제가 발생할 수 있습니다. 라이선스가 한 예입니다. 출시 후에 제품이 라이선스를 준수하지 않는다는 것을 발견하면 이미 너무 늦습니다. 배포도 마찬가지입니다. 개발에서는 완벽하게 작동하지만 컨테이너로 이동하면 모든 종류의 문제가 발생할 수 있습니다. 이러한 함정은 팀이 피하고자 하는 것들입니다.

## HagiCode 소개

이 글에서 공유하는 접근 방식은 [HagiCode](https://hagicode.com) 프로젝트에서의 실무 경험에서 나온 것입니다. HagiCode는 AI 기반 코딩 어시스턴트입니다. 브라우저 기반 코드 편집을 구현하면서 두 솔루션을 심층적으로 연구했으며, 궁극적으로 두 솔루션을 모두 지원하는 아키텍처를 설계하면서 code-server를 기본값으로 선택했습니다.

프로젝트 저장소: [github.com/HagiCode-org/site](https://github.com/HagiCode-org/site)

## 라이선스 차이 (가장 중요)

이것은 두 솔루션 간의 가장 근본적인 차이이며, 평가 시 가장 먼저 고려한 요소입니다. 기술적 선택을 할 때 법적 위험을 미리 이해하는 것이 중요합니다.

**code-server**
- MIT 라이선스, 완전한 오픈소스
- Coder.com에서 유지 관리하며 활발한 커뮤니티 보유
- 상업적 사용, 수정, 배포가 자유롭습니다
- 사용 시나리오에 제한이 없습니다

**VSCode `code serve-web`**
- Microsoft VSCode 제품의 일부
- Microsoft의 라이선스 사용 (VS Code 라이선스에는 상업적 사용 제한이 포함됩니다)
- 주로 개인 개발자를 위한 것입니다
- 엔터프라이즈 배포에는 추가적인 상업적 라이선스 검토가 필요할 수 있습니다

라이선스 관점에서 code-server는 상업적 프로젝트에 더 친화적입니다. 이것은 제품 기획 단계에서 생각해야 할 사항입니다. 나중에 마이그레이션하면 비용이 매우 커질 수 있기 때문입니다.

## 배포 차이

라이선스가 정리되면 다음은 배포입니다. 이것은 운영 비용과 아키텍처 설계에 직접적인 영향을 미칩니다.

**code-server**
- 독립적인 Node.js 애플리케이션으로 독립적으로 배포할 수 있습니다
- 여러 런타임 소스를 지원합니다:
  - 실행 파일 경로를 직접 지정
  - 시스템 PATH를 통해 검색
  - NVM Node.js 22.x 환경 자동 감지
- 서버에 VSCode 데스크톱 애플리케이션을 설치할 필요가 없습니다
- 컨테이너에서 더 쉽게 배포할 수 있습니다

**VSCode `code serve-web`**
- 로컬에 설치된 VSCode CLI에 의존해야 합니다
- 호스트 머신에서 사용 가능한 `code` 명령어가 필요합니다
- 시스템이 VS Code Remote CLI 래퍼를 필터링합니다
- 주로 로컬 개발 시나리오용으로 설계되었습니다

code-server는 서버 및 컨테이너 배포 시나리오에 더 적합합니다. 제품이 Docker에서 실행되어야 하거나 사용자가 VSCode를 설치하지 않은 경우, code-server가 일반적으로 올바른 선택입니다.

## 기능 매개변수 차이

두 솔루션은 몇 가지 기능 매개변수에서도 차이가 있습니다. 차이가 크지는 않지만 실제 사용에서 통합 마찰을 일으킬 수 있습니다.

| 기능 | code-server | code serve-web |
|------|-------------|----------------|
| 공개 기본 경로 | `/` (구성 가능) | `/vscode-server` (고정) |
| 인증 | 여러 모드의 `--auth` 매개변수 | `--connection-token` / `--without-connection-token` |
| 데이터 디렉토리 | `{DataDir}/code-server` | `{DataDir}/vscode-serve-web` |
| 원격 분석 | `--disable-telemetry`로 기본 비활성화 | VSCode 설정에 따름 |
| 업데이트 확인 | `--disable-update-check`로 비활성화 가능 | VSCode 설정에 따름 |

이러한 차이는 통합 시 특별한 주의가 필요합니다. 예를 들어 다른 URL 경로는 프론트엔드 코드에서 전용 처리가 필요함을 의미합니다.

## 가용성 감지 차이

에디터 전환을 구현할 때 가용성 감지 로직도 다릅니다.

**code-server**
- 항상 표시 구현으로 반환됩니다
- 사용 불가능한 경우에도 `install-required` 상태와 함께 표시됩니다
- NVM Node.js 22.x 환경 자동 감지를 지원합니다

**code serve-web**
- 로컬 `code` CLI가 감지된 경우에만 표시됩니다
- 사용 불가능한 경우 프론트엔드가 이 옵션을 자동으로 숨깁니다
- 로컬 VSCode 설치 상태에 따라 다릅니다

이 차이는 사용자 경험에 직접적인 영향을 미칩니다. code-server는 더 투명합니다. 사용자가 옵션을 보고 설치가 아직 필요하다는 것을 이해할 수 있습니다. code serve-web은 더 숨겨집니다. 사용자가 옵션이 존재하는지조차 모를 수 있습니다. 어느 접근 방식이 더 나은지는 제품 포지셔닝에 따라 다릅니다.

## HagiCode의 이중 구현 아키텍처

심층 분석 후 HagiCode 프로젝트는 아키텍처 수준에서 두 솔루션을 모두 지원하는 이중 구현 아키텍처를 채택했습니다.

### code-server를 기본값으로

```csharp
// 기본 활성 구현은 code-server입니다
// 명시적인 activeImplementation이 저장된 경우 해당 구현을 먼저 시도합니다
// 요청된 구현을 사용할 수 없으면 리졸버가 다른 구현을 시도합니다
// 폴백이 발생하면 fallbackReason을 반환합니다
```

기본적으로 code-server를 선택한 것은 주로 라이선스와 배포 유연성 때문입니다. 하지만 이미 로컬 VSCode 환경이 있는 사용자에게 code serve-web도 훌륭한 옵션입니다.

### 구현 선택기

`CodeServerImplementationResolver`는 다음을 통합하는 데 책임이 있습니다.
- 시작 시 워밍업 중 구현 선택
- 상태 읽기 시 구현 선택
- 프로젝트 열기 시 구현 선택
- Vault 열기 시 구현 선택

이 설계를 통해 시스템은 다양한 시나리오에 유연하게 대응할 수 있으며, 사용자는 자신의 환경에 가장 잘 맞는 구현을 선택할 수 있습니다.

### 프론트엔드 적응 규칙

```typescript
// localCodeAvailable=false인 경우 code serve-web을 표시하지 않음
// localCodeAvailable=true인 경우 code serve-web 구성을 표시
```

프론트엔드는 환경에 따라 사용 가능한 옵션을 자동으로 표시하므로, 사용자가 사용할 수 없는 기능으로 혼란스러워하지 않습니다.

## 실용 가이드

이론은 충분합니다. 실제 배포 시 무엇에 주의해야 할까요? 결국 구현이 중요합니다.

### Docker 배포 권장

컨테이너화된 배포의 경우 code-server가 더 나은 선택입니다.

```dockerfile
# 공식 code-server 이미지를 직접 사용
FROM codercom/code-server:latest

# 또는 npm을 통해 설치
RUN npm install -g code-server
```

이것은 추가적인 VSCode 설치 없이 한 레이어로 문제를 해결합니다.

### 설정 예시

**code-server 설정**
```json
{
  "vscodeServer": {
    "enabled": true,
    "activeImplementation": "code-server",
    "codeServer": {
      "host": "0.0.0.0",
      "port": 8080,
      "executablePath": "",
      "authMode": "none"
    }
  }
}
```

**code serve-web 설정**
```json
{
  "vscodeServer": {
    "enabled": true,
    "activeImplementation": "serve-web",
    "serveWeb": {
      "host": "0.0.0.0",
      "port": 8080,
      "executablePath": "/usr/local/bin/code"
    }
  }
}
```

설정은 처음에 약간 번거로울 수 있지만, 한 번 자리를 잡으면 유지 관리가 훨씬 쉬워집니다.

### URL 구성 차이

**code-server**
```
http://localhost:8080/?folder=/path/to/project&vscode-lang=zh-CN
```

**code serve-web**
```
http://localhost:8080/vscode-server/?folder=/path/to/project&tkn=xxx&vscode-lang=zh-CN
```

경로와 매개변수의 차이에 주의하세요. 통합 시 별도로 처리해야 합니다.

### 구현 전환

시스템은 런타임 전환을 지원하며 전환 시 이전 구현을 자동으로 중지합니다.

```csharp
// VsCodeServerManager가 상호 배제를 자동으로 처리합니다
// activeImplementation을 전환할 때 이전 구현이 백그라운드에서 계속 실행되지 않습니다
```

이 설계를 통해 사용자는 언제든지 다른 구현을 시도하고 가장 잘 맞는 옵션을 찾을 수 있습니다.

### 상태 모니터링

```typescript
const { settings, runtime } = await getVsCodeServerSettings();

// runtime.activeImplementation: "code-server" | "serve-web"
// runtime.fallbackReason: 전환 이유
// runtime.status: "running" | "starting" | "stopped" | "unhealthy"
```

상태가 보이면 사용자는 문제가 서버 측에서 비롯된 것인지 자신의 조작 때문인지 빠르게 판단할 수 있습니다.

## 결론

| 비교 차원 | code-server | code serve-web | 권장 |
|---------|-------------|----------------|------|
| 라이선스 | MIT (상업 친화적) | Microsoft (제한적) | code-server |
| 배포 유연성 | 독립 배포 | 로컬 VSCode 의존 | code-server |
| 서버 적합성 | 서버용 설계 | 주로 로컬 개발용 | code-server |
| 컨테이너화 | 네이티브 지원 | VSCode 설치 필요 | code-server |
| 기능 완전성 | 데스크톱 버전에 근접 | 공식 완전 버전 | code serve-web |
| 유지 관리 활동 | 활발한 커뮤니티 | Microsoft 공식 유지 관리 | 각각 장점이 있음 |

**권장 전략**: 먼저 code-server를 사용하고, 공식 기능이 완전히 필요하며 로컬 VSCode 환경이 이미 있는 경우 code serve-web을 고려하세요.
