## VSCode와 code-server: 브라우저 기반 코드 편집 솔루션 선정

> 브라우저 기반 코드 편집 기능을 구축할 때 개발자는 핵심 선택에 직면합니다: VSCode 공식 `code serve-web` 기능을 사용할 것인가, 아니면 커뮤니티 주도의 `code-server` 솔루션을 채택할 것인가? 이 선택은 기술 아키텍처뿐만 아니라 라이선스 준수와 배포 유연성에도 영향을 미칩니다.

## 배경

사실 기술 선정이라는 건 인생의 길을 선택하는 것과 비슷합니다. 한 길을 선택하면 계속 걸어가야 하고, 길을 바꾸고 싶을 때는 비용이 크게 듭니다.

AI 보조 프로그래밍 시대에 브라우저 기반 코드 편집 기능은 점점 더 중요해지고 있습니다. 사용자는 AI 어시스턴트가 코드 분석을 마친 후 즉시 같은 브라우저 세션에서 에디터를 열어 수정할 수 있기를 기대하며, 애플리케이션을 전환할 필요가 없습니다. 이러한 원활한 경험은, 말하자면, 원할 때 거기에 있다는 것——그런데 가끔은 하필 그곳에 없습니다.

그러나 이 기능을 구현할 때 개발자는 핵심적인 기술 선정에 직면합니다: VSCode 공식 `code serve-web` 기능을 사용할 것인가, 아니면 커뮤니티 주도의 `code-server` 솔루션을 채택할 것인가?

두 솔루션은 각자 장단점이 있으며, 잘못 선택하면 나중에 상당한 문제가 발생할 수 있습니다. 예를 들어 라이선스 문제——제품이 출시된 후에야 라이선스가 비준수라는 것을 발견하면 이미 늦습니다. 이건 연애와 비슷합니다. 처음에 제대로 생각하지 않았다가 나중에야 두 사람의 가치관이 완전히 맞지 않는다는 것을 발견하면 대가가 큽니다. 또 배포 방식도 마찬가지입니다. 어떤 솔루션은 개발 환경에서는 잘 실행되지만 컨테이너에 올리면 다양한 문제가 발생합니다. 이런 함정은 누구도 밟고 싶지 않습니다.毕竟 함정을 많이 밟으면 사람도 마비가 되니까요.

## HagiCode 소개

이 글에서 공유하는 솔루션은 [HagiCode](https://hagicode.com) 프로젝트에서의 실천 경험에서 나온 것입니다. HagiCode는 AI 기반 코드 어시스턴트로, 브라우저 기반 코드 편집 기능을 구현할 때 우리는 두 솔루션을 깊이 연구했습니다. 최종적으로 아키텍처 설계에서 둘 다 지원하지만 기본적으로 code-server를 우선 선택합니다.

프로젝트 주소: [github.com/HagiCode-org/site](https://github.com/HagiCode-org/site)

## 라이선스 차이 (가장 중요)

이것은 두 솔루션의 가장 근본적인 차이이며, 우리가 선정할 때 가장 먼저 고려한 요소입니다. 기술 선정은 첫 단계에서 법적 위험을 명확히 해야 합니다. 그렇지 않으면 나중에 문제가 발생했을 때 누구를 탓할까요?

**code-server**
- MIT 라이선스, 완전히 오픈 소스
- Coder.com 회사에서 유지 관리, 커뮤니티 활동적
- 자유롭게 상업적으로 사용, 수정 및 배포 가능
- 사용 시나리오 제한 없음

**VSCode `code serve-web`**
- Microsoft VSCode 제품의 일부
- Microsoft 라이선스 사용 (VS Code 라이선스에는 상업적 사용 제한 있음)
- 주로 개인 개발자 사용을 대상
- 엔터프라이즈 배포는 추가적인 상업적 라이선스 고려가 필요할 수 있음

라이선스 관점에서 code-server는 상업 프로젝트에 더 친화적입니다. 이 점은 제품 기획 단계에서 명확히 해야 합니다. 그렇지 않으면 규모가 커진 후에 마이그레이션하려면 비용이 큽니다.毕竟 마이그레이션은 말하기는 쉽지만 하기는 어렵습니다. 경험해 본 사람만 압니다.

## 배포 방식 차이

라이선스 문제를 해결한 후 다음은 배포 방식입니다. 이것은 운영 비용과 아키텍처 설계에 직접적으로 영향을 미치며, 간접적으로 매일의 기분에도 영향을 줍니다——배포가 간단할수록 기분이 좋아지는 건 다 아는 사실입니다.

**code-server**
- 독립적인 Node.js 애플리케이션, 단독 배포 가능
- 여러 런타임 소스 지원:
  - 실행 파일 경로 직접 지정
  - 시스템 PATH查找
  - NVM Node.js 22.x 환경 자동 감지
- 서버에 VSCode 데스크톱 버전 설치 불필요
- 컨테이너화 배포가 더 간단

**VSCode `code serve-web`**
- 로컬에 설치된 VSCode CLI에 의존해야 함
- 로컬 시스템에서 사용 가능한 `code` 명령 필요
- 시스템이 VS Code Remote CLI 래퍼를 필터링함
- 주로 로컬 개발 시나리오용으로 설계됨

code-server는 서버/컨테이너 배포 시나리오에 더 적합합니다. 제품이 Docker에서 실행되어야 하거나 사용자 환경에 VSCode가 없다면 code-server를 선택하세요.毕竟 간단한 것이 아름답습니다. 복잡하면 문제가 발생하기 쉽고, 문제가 발생하면 고쳐야 하고, 고치고 나면 새로운 문제가 도입될 수 있습니다. 이 무한한 루프를 누가 경험하고 싶을까요?

## 기능 매개변수 차이

두 솔루션은 기능 매개변수에도 약간의 차이가 있습니다. 비록 크지 않지만 실제 사용에서 약간의 문제를 일으킬 수 있습니다. 이러한 세부 사항은 생활의 작은 마찰과 같습니다. 많지 않지만 많아지면 짜증이 납니다.

| 특성 | code-server | code serve-web |
|------|-------------|----------------|
| 공개 기본 경로 | `/` (구성 가능) | `/vscode-server` (고정) |
| 인증 방식 | `--auth` 매개변수, 다양한 모드 지원 | `--connection-token` / `--without-connection-token` |
| 데이터 디렉터리 | `{DataDir}/code-server` | `{DataDir}/vscode-serve-web` |
| 원격 측정 | 기본적으로 비활성화 `--disable-telemetry` | VSCode 설정에 의존 |
| 업데이트 확인 | 비활성화 가능 `--disable-update-check` | VSCode 설정에 의존 |

이러한 차이는 통합할 때 특별히 주의해야 합니다. 예를 들어 URL 경로가 다르다는 것은 프론트엔드 코드에서 대상 처리를 해야 한다는 것을 의미합니다. 개발을 해본 사람들은 다 압니다. 이런 세부 사항을 처리하는 데 시간이 가장 많이 걸리지만, 어쩔 수 없습니다. 하지 않으면 실행되지 않으니까요.

## 가용성 감지 차이

에디터 전환 기능을 구현할 때 가용성 감지 로직도 다릅니다. 이러한 차이는 사람과 사람 사이의相处方式과 같습니다. 어떤 사람은 솔직하게 대하는 것을 좋아하고, 어떤 사람은 은근하고 우아한 것을 좋아합니다.

**code-server**
- 항상 표시되는 구현으로 반환됨
- 사용할 수 없어도 표시되며 `install-required` 상태를 표시함
- NVM Node.js 22.x 환경 자동 감지 지원

**code serve-web**
- 로컬 `code` CLI가 감지될 때만 표시됨
- 사용할 수 없는 경우 프론트엔드에서 자동으로 이 옵션을 숨김
- 로컬 VSCode 설치 상태에 의존

이러한 차이는 사용자 경험에 직접적으로 영향을 미칩니다. code-server의 방식은 더 투명하며, 사용자는 이 옵션이 있다는 것을 알고 있지만 아직 설치하지 않은 것입니다. code serve-web의 방식은 더 은밀하며, 사용자는 이 선택지가 있다는 것을 모를 수도 있습니다. 어떤 방식이 더 좋을까요? 이것은 제품 포지셔닝에 달려 있습니다.毕竟 사용자 경험에는 정답이 없고 적합한지 아닌지만 있을 뿐입니다.

## HagiCode의 이중 구현 아키텍처

심층 분석 끝에 HagiCode 프로젝트는 이중 구현 아키텍처를 채택하여 아키텍처 수준에서 두 솔루션을 모두 지원합니다. 이것이 우리가 기술 선정에 어려움을 겪는 것이 아니라 실제로 실제 요구사항이 있기 때문입니다.毕竟 기술 세계에는 절대적으로 올바른 선택은 없고 자신에게 가장 적합한 선택만 있습니다.

### 기본 선택 code-server

```csharp
// 기본 활성 구현은 code-server입니다.
// 명시적인 activeImplementation이 저장된 경우 해당 구현을 우선 시도합니다.
// 요청된 구현을 사용할 수 없으면 확인자는 다른 구현을 시도합니다.
// 폴백이 발생하면 fallbackReason을 반환합니다.
```

우리는 기본적으로 code-server를 선택합니다. 주요 고려 사항은 라이선스와 배포 유연성입니다. 하지만 로컬 VSCode 환경이 있는 사용자에게는 code serve-web도 좋은 선택입니다.毕竟 사용자에게 더 많은 선택지를 주는 것은 좋은 일입니다. 타인에게 단일 솔루션을 강요하는 일은 저는 할 수 없습니다.

### 구현 선택기

`CodeServerImplementationResolver`는 통일하여 다음을 책임집니다:
- 시작 예열 시 구현 선택
- 상태 읽기 시 구현 선택
- 프로젝트 열기 시 구현 선택
- Vault 열기 시 구현 선택

이러한 설계는 시스템이 다양한 시나리오에 유연하게 대응할 수 있게 하며, 사용자는 자신의 환경에 따라 가장 적합한 구현을 선택할 수 있습니다. 유연한 설계는 초기에 시간을 더 쓰지만 후기에는 마음이 편합니다.毕竟 누구도 여기저기 코드를 수정하고 싶지 않습니다.

### 프론트엔드 적응 규칙

```typescript
// localCodeAvailable=false일 때 code serve-web 표시 안 함
// localCodeAvailable=true일 때 code serve-web 구성 표시
```

프론트엔드는 환경에 따라 사용 가능한 옵션을 자동으로 표시하여 사용자가 사용할 수 없는 기능을 보고 혼란스러워하지 않도록 합니다. 사용자가 혼란스러워하면 당신에게 오고, 많이 물어보면 당신도 짜증이 나고, 짜증이 나면 코드를 수정하고 싶어지고, 코드를 수정하면 버그가 도입될 수 있습니다. 이 악순환을 누가 경험하고 싶을까요?

## 실천 가이드

이렇게 이론을 많이 말했지만 실제 배포할 때는 무엇을 주의해야 할까요? 사실 이론이 아무리 좋아도落地不行也无用입니다.毕竟 실천이 진리를 검증하는 유일한 표준입니다.

### Docker 배포 권장사항

컨테이너화 배포의 경우 code-server가 더 나은 선택입니다:

```dockerfile
# code-server 공식 이미지 직접 사용
FROM codercom/code-server:latest

# 또는 npm을 통해 설치
RUN npm install -g code-server
```

이렇게 한 레이어로 해결되며 VSCode를 추가로 설치할 필요가 없습니다. 간단한 것이 좋습니다. 복잡하면 오류가 발생하기 쉽습니다. 이 원칙은 어디든 적용됩니다.

### 구성 예제

**code-server 구성**
```json
{
  "vscodeServer": {
    "enabled": true,
    "activeImplementation": "code-server",
    "codeServer": {
      "host": "0.0.0.0",
      "port": 8080,
      "executablePath": "",
      "authMode": "none"
    }
  }
}
```

**code serve-web 구성**
```json
{
  "vscodeServer": {
    "enabled": true,
    "activeImplementation": "serve-web",
    "serveWeb": {
      "host": "0.0.0.0",
      "port": 8080,
      "executablePath": "/usr/local/bin/code"
    }
  }
}
```

구성은 처음에는 번거롭지만 잘 구성하면 나중에 마음이 편합니다. 삶과 마찬가지입니다. 초기에 더 투자하면 나중에 생활이 더 좋아집니다.

### URL 빌드 차이

**code-server**
```
http://localhost:8080/?folder=/path/to/project&vscode-lang=ko-KR
```

**code serve-web**
```
http://localhost:8080/vscode-server/?folder=/path/to/project&tkn=xxx&vscode-lang=ko-KR
```

경로와 매개변수의 차이에 주의하며 통합할 때 각각 처리해야 합니다. 세부 사항이 성패를 좌우합니다. 이 말은 틀림없습니다. 매개변수가 하나缺少되면 페이지를 열 수 없습니다.

### 구현 전환

시스템은 런타임 전환을 지원하며 전환 시 자동으로 오래된 구현을 중지합니다:

```csharp
// VsCodeServerManager가 자동으로 상호 배제를 처리합니다.
// activeImplementation을 전환할 때 오래된 구현은 백그라운드에서 계속 실행되지 않습니다.
```

이러한 설계는 사용자가 언제든지 다른 구현을 시도하고 자신에게 가장 적합한 솔루션을 찾을 수 있게 합니다.毕竟 자신에게 적합한 것이 가장 좋습니다. 다른 사람의 제안은 참고만 할 뿐 결국 자신이 시도해야 합니다.

### 상태 모니터링

```typescript
const { settings, runtime } = await getVsCodeServerSettings();

// runtime.activeImplementation: "code-server" | "serve-web"
// runtime.fallbackReason: 전환 이유
// runtime.status: "running" | "starting" | "stopped" | "unhealthy"
```

상태가 보여야 마음이 편합니다. 사용자는 문제가 발생했을 때 서버 문제인지 자신의 조작 문제인지 빠르게定位할 수 있습니다. 상태를 모를 때 사람은 당황하기 쉽고, 당황하면 잘못된 판단을 하기 쉽습니다. 이 사슬이 한번 시작되면 멈출 수 없습니다.
