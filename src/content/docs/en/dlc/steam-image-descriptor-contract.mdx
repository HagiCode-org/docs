Steam application, DLC, and bundle visuals should come from the Index Steam payload. Consumers read image descriptors instead of hardcoding asset paths in UI components.

This contract currently covers three product keys:

- `hagicode`: the main Steam application.
- `turbo-engine`: Turbo Engine DLC.
- `hagicode-plus`: the official Steam bundle.

## Payload URL

Default endpoint:

```text
https://index.hagicode.com/steam/index.json
```

`repos/web` can override this URL with `VITE_STEAM_INDEX_URL` for testing, staging, or temporary Index source changes.

## Top-Level Shape

The payload must expose these top-level fields:

```json
{
  "applications": [],
  "bundles": [],
  "images": {}
}
```

Consumers validate these fields at the service boundary:

- `applications` must be an array for main app and DLC records.
- `bundles` must be an array for bundle records.
- `images` must be an object for product-keyed shared image descriptors.

## Image Descriptor

Every descriptor uses the same contract:

```json
{
  "src": "https://cdn.example.invalid/steam/hagicode-wide.jpg",
  "variant": "wide-capsule",
  "alt": "Hagicode Steam wide capsule artwork",
  "width": 1920,
  "height": 1080
}
```

Fields:

- `src`: required image URL or published asset path.
- `variant`: required layout role.
- `alt`: optional but recommended accessible image text.
- `width` / `height`: optional dimensions for future layout decisions.

Common variants:

| Variant | Purpose | Consumer layout |
| --- | --- | --- |
| `store-capsule` | Compact store/product card | Small card or thumbnail |
| `wide-capsule` | Wide promotional art | Default primary image |
| `library-capsule` | Tall library artwork | Thumbnail or alternate primary image |
| `hero` | Large banner artwork | Hero or large header |

Unknown variants must not break consumers. They may be ignored or rendered with a generic layout.

## Application And DLC Records

Applications and DLC records live in `applications`:

```json
{
  "key": "turbo-engine",
  "displayName": "Hagicode: Turbo Engine DLC",
  "kind": "dlc",
  "parentKey": "hagicode",
  "storeAppId": 4635480,
  "storeUrl": "https://store.steampowered.com/app/4635480/Hagicode__Turbo_Engine/",
  "images": []
}
```

Rules:

- `key` should be stable, lowercase, and used as the consumer lookup key.
- DLC records should preserve `parentKey` for UI context.
- `images` is backward-compatible; older clients can ignore it.
- If record-level `images` is empty, consumers may read descriptors from top-level `images[key]`.

## Bundle Records

Bundle records live in `bundles`:

```json
{
  "key": "hagicode-plus",
  "displayName": "Hagicode Plus",
  "storeBundleId": 73989,
  "storeUrl": "https://store.steampowered.com/bundle/73989/Hagicode_Plus/",
  "includedApplicationKeys": ["hagicode", "turbo-engine"],
  "images": []
}
```

Rules:

- Bundle images describe the bundle itself and do not replace `includedApplicationKeys`.
- Consumers should prefer bundle-specific images.
- If a bundle has no valid images, the product name, Steam link, and included application metadata must remain visible.

## Accessibility

Every rendered image needs accessible text:

- Use descriptor `alt` when present.
- Otherwise generate deterministic text such as `Hagicode wide capsule image`.
- If an image fails to load, replace the broken media with a branded fallback that keeps product name and Steam CTA visible.

## Maintenance Flow

When adding or changing Steam images:

1. Upload or register the image asset in the Index publishing source.
2. Update `/steam/index.json` record-level `images` or top-level `images[key]`.
3. Verify `src`, `variant`, and optional `alt` match this contract.
4. `repos/web` caches parsed payloads for the current browser page lifetime; refresh or HTTP cache revalidation picks up new data.
5. `repos/site` is a static Astro site. This implementation uses a static descriptor snapshot with visible fallback states; sync the snapshot after Index publishes final image URLs.
6. `repos/docs` documents the contract and maintenance process only.

## Implementation Scope

This change uses these implementation boundaries:

- `repos/index` is reference-only for this change and is not edited here.
- `repos/web` is the runtime consumer for dynamic loading, caching, normalization, and gallery rendering.
- `repos/site` uses a static-generation-safe descriptor snapshot and visible fallback rendering.
- `repos/docs` records the field contract, accessibility expectations, compatibility rules, and maintenance flow.

The `images` field is additive, so older clients can ignore it while newer clients render Steam visual resources.
