## Automating Docker Hub to Azure ACR Image Sync

> This article explains how to use GitHub Actions and the image-syncer tool to automate image synchronization from Docker Hub to Azure Container Registry, addressing slow Docker Hub access in mainland China and some Azure regions while improving image availability and deployment efficiency in Azure environments.

## Background / Introduction

The HagiCode project uses Docker images as core runtime components, with its primary images hosted on Docker Hub. As the project has grown and Azure deployment needs have increased, we ran into several pain points:
- Slow image pulls, because Docker Hub access is constrained in mainland China and some Azure regions
- A single image source introduces single-point-of-failure risk
- Using Azure Container Registry in Azure environments provides better network performance and integration

To address these issues, we needed an automated image synchronization mechanism to regularly sync Docker Hub images to Azure ACR, ensuring that users get faster image pull speeds and higher availability when deploying in Azure environments.

## Technical Solution Comparison

When defining the solution, we compared multiple technical approaches:

### 1. image-syncer (Final Choice)
- Incremental sync: Only synchronizes changed image layers, significantly reducing network transfer
- Resume support: Synchronization can continue after network interruptions
- Concurrency control: Supports configurable concurrent worker counts to improve large image sync efficiency
- Robust error handling: Built-in retry mechanism for failures (3 retries by default)
- Lightweight deployment: Single binary with no dependencies
- Multi-registry support: Compatible with Docker Hub, Azure ACR, Harbor, and more

### 2. Docker CLI
- No incremental sync support: Must pull the full image content every time
- Lower efficiency: Large network transfer volume and longer execution time
- Simple to use: Relies on familiar `docker pull` and `docker push` commands

### 3. Azure CLI
- Higher complexity: Requires Azure CLI authentication setup
- Functional limitations: `az acr import` has a relatively narrow feature set
- Native integration: Integrates well with Azure services

## Architecture Design Decisions

### Decision 1: Set the sync schedule to daily at 00:00 UTC
- Balances image freshness with resource consumption
- Avoids business peak hours and reduces interference with other operations
- Docker Hub images are typically updated after daily builds

### Decision 2: Sync all image tags
- Maintains full consistency with Docker Hub
- Gives users flexible version choices
- Simplifies sync logic by avoiding complex tag filtering rules

### Decision 3: Store credentials in GitHub Secrets
- Natively supported by GitHub Actions with strong security
- Simple to configure, manage, and maintain
- Supports repository-level access control

## Risk Assessment and Mitigation

### Risk 1: Azure ACR credential leakage
- Store credentials securely in GitHub Secrets
- Rotate ACR passwords regularly
- Restrict ACR user permissions to push-only
- Monitor ACR access logs

### Risk 2: Sync failures causing image inconsistency
- image-syncer includes a built-in incremental sync mechanism
- Automatic retry on failure (3 times by default)
- Detailed error logs and failure notifications
- Resume capability after interruption

### Risk 3: Excessive resource consumption
- Incremental sync reduces network transfer
- Configurable concurrency level (currently set to 10)
- Monitor the number and size of synchronized images
- Run synchronization during off-peak hours

## Core Solution

We use an automated GitHub Actions + image-syncer workflow to synchronize images from Docker Hub to Azure ACR.

## Implementation Steps

### 1. Preparation
- Create or confirm an Azure Container Registry in the Azure Portal
- Create ACR access credentials (username and password)
- Verify access permissions for the Docker Hub image repository

### 2. Configure GitHub Secrets
Add the following Secrets in the GitHub repository settings:
- `AZURE_ACR_USERNAME`: Azure ACR username
- `AZURE_ACR_PASSWORD`: Azure ACR password

### 3. Create the GitHub Actions Workflow
Configure the workflow in `.github/workflows/sync-docker-acr.yml`:
- Scheduled trigger: every day at 00:00 UTC
- Manual trigger: supports `workflow_dispatch`
- Additional trigger: runs on pushes to the `publish` branch for fast synchronization

### 4. Workflow Execution Flow

| Order | Participant | Action | Description |
| --- | --- | --- | --- |
| 1 | GitHub Actions | Trigger workflow | Triggered by schedule, manual dispatch, or a push to the `publish` branch |
| 2 | GitHub Actions → image-syncer | Download and execute the sync tool | Enters the actual synchronization phase |
| 3 | image-syncer → Docker Hub | Fetch image manifests and tag list | Reads metadata from the source repository |
| 4 | image-syncer → Azure ACR | Fetch existing image information from the target repository | Determines the current state of the target side |
| 5 | image-syncer | Compare source and target differences | Identifies the image layers that need to be synchronized |
| 6 | image-syncer → Docker Hub | Pull changed image layers | Transfers only the content that needs updating |
| 7 | image-syncer → Azure ACR | Push changed image layers | Completes incremental synchronization |
| 8 | image-syncer → GitHub Actions | Return synchronization statistics | Includes results, differences, and error details |
| 9 | GitHub Actions | Record logs and upload artifacts | Useful for later auditing and troubleshooting |

## GitHub Actions Workflow Implementation

Below is the actual workflow configuration in use (`.github/workflows/sync-docker-acr.yml`):

```yaml
name: Sync Docker Image to Azure ACR

on:
  schedule:
    - cron: "0 0 * * *" # Every day at 00:00 UTC
  workflow_dispatch: # Manual trigger
  push:
    branches: [publish]

permissions:
  contents: read

jobs:
  sync:
    runs-on: ubuntu-latest

    steps:
      - name: Checkout code
        uses: actions/checkout@v4

      - name: Download image-syncer
        run: |
          # Download the image-syncer binary
          wget https://github.com/AliyunContainerService/image-syncer/releases/download/v1.5.5/image-syncer-v1.5.5-linux-amd64.tar.gz
          tar -zxvf image-syncer-v1.5.5-linux-amd64.tar.gz
          chmod +x image-syncer

      - name: Create auth config
        run: |
          # Generate the authentication config file (YAML format)
          cat > auth.yaml <<EOF
          hagicode.azurecr.io:
            username: "${{ secrets.AZURE_ACR_USERNAME }}"
            password: "${{ secrets.AZURE_ACR_PASSWORD }}"
          EOF

      - name: Create images config
        run: |
          # Generate the image sync config file (YAML format)
          cat > images.yaml <<EOF
          docker.io/newbe36524/hagicode: hagicode.azurecr.io/hagicode
          EOF

      - name: Run image-syncer
        run: |
          # Run the synchronization (using the newer --auth and --images arguments)
          ./image-syncer --auth=./auth.yaml --images=./images.yaml --proc=10 --retries=3

      - name: Upload logs
        if: always()
        uses: actions/upload-artifact@v4
        with:
          name: sync-logs
          path: image-syncer-*.log
          retention-days: 7
```

## Configuration Details

### 1. Trigger Conditions
- Scheduled trigger: `cron: "0 0 * * *"` runs every day at 00:00 UTC
- Manual trigger: `workflow_dispatch` allows users to run the workflow manually from the GitHub UI
- Push trigger: `push: branches: [publish]` runs when the publish branch receives a push for faster synchronization

### 2. Authentication Configuration (`auth.yaml`)
```yaml
hagicode.azurecr.io:
  username: "${{ secrets.AZURE_ACR_USERNAME }}"
  password: "${{ secrets.AZURE_ACR_PASSWORD }}"
```

### 3. Image Sync Configuration
```yaml
docker.io/newbe36524/hagicode: hagicode.azurecr.io/hagicode
```
This configuration means all tags from `docker.io/newbe36524/hagicode` are synchronized to `hagicode.azurecr.io/hagicode`.

### 4. image-syncer Parameters
- `--auth=./auth.yaml`: path to the authentication config file
- `--images=./images.yaml`: path to the image sync config file
- `--proc=10`: sets the number of concurrent workers to 10
- `--retries=3`: retries failures 3 times

## Notes and Best Practices

### 1. Security Recommendations
- Rotate Azure ACR passwords regularly (recommended every 90 days)
- Use a dedicated ACR service account and restrict it to push-only permissions
- Monitor ACR access logs to detect abnormal access promptly
- Do not output credentials in logs
- Do not commit credentials to the code repository

### 2. Performance Optimization
- Tune the `--proc` parameter: adjust concurrency according to available bandwidth (recommended 5-20)
- Monitor synchronization duration: if it takes too long, consider reducing concurrency
- Clean up logs regularly: set a reasonable `retention-days` value (currently 7 days)

### 3. Troubleshooting

#### Issue 1: Authentication failure
```
Error: failed to authenticate to hagicode.azurecr.io
```
Solution:
1. Check whether GitHub Secrets are configured correctly
2. Verify whether the Azure ACR password has expired
3. Confirm that the ACR service account permissions are correct

#### Issue 2: Network timeout
```
Error: timeout waiting for response
```
Solution:
1. Check the network connection
2. Reduce the number of concurrent workers (the `--proc` parameter)
3. Wait for network recovery and rerun the workflow

#### Issue 3: Incomplete image synchronization
```
Warning: some tags failed to sync
```
Solution:
1. Check the synchronization logs and identify the failed tags
2. Manually rerun the workflow to sync again
3. Verify that the source image on Docker Hub is healthy

### 4. Monitoring and Alerts
- Regularly check the Actions page to confirm workflow run status
- Configure GitHub notifications to receive workflow failure alerts promptly
- Monitor Azure ACR storage usage
- Regularly verify image tag consistency

## Frequently Asked Questions and Solutions

### Q1: How do I sync specific tags instead of all tags?

Modify the `images.yaml` configuration file:
```yaml
# Only sync the latest and v1.0 tags
docker.io/newbe36524/hagicode:latest: hagicode.azurecr.io/hagicode:latest
docker.io/newbe36524/hagicode:v1.0: hagicode.azurecr.io/hagicode:v1.0
```

### Q2: How do I sync multiple image repositories?

Add multiple lines to `images.yaml`:
```yaml
docker.io/newbe36524/hagicode: hagicode.azurecr.io/hagicode
docker.io/newbe36524/another-image: hagicode.azurecr.io/another-image
```

### Q3: How do I retry after synchronization fails?
- Automatic retry: image-syncer has a built-in retry mechanism (3 times by default)
- Manual retry: click `Re-run all jobs` on the GitHub Actions page

### Q4: How can I view detailed synchronization progress?
- View real-time logs on the Actions page
- Download the `sync-logs` artifact to inspect the full log file
- The log file includes synchronization status and transfer speed for each tag

### Q5: How long does synchronization take?
- Initial full synchronization: typically 10-30 minutes depending on image size
- Incremental synchronization: typically 2-5 minutes when image changes are small
- Total time depends on network bandwidth, image size, and concurrency settings
