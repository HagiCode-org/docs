This page is for first-time HagiCode users who want a fast OmniRoute setup path: start the local service, verify that `http://localhost:20128/v1` works, and then point HagiCode or any other OpenAI-compatible tool at it.

:::tip[Recommended path]
This first version keeps the most stable path first: `npm install -g omniroute`, `omniroute`, Dashboard at `http://localhost:20128`, and the OpenAI-compatible Base URL at `http://localhost:20128/v1`. Docker, VPS, and more advanced deployment modes stay in the optional section.
:::

## Prerequisites

- Node.js is already installed. Complete [Installing Node.js](/en/related-software-installation/nodejs/installation) first if needed.
- Local port `20128` is available and not already occupied.
- You have at least one upstream model provider account that you plan to route through OmniRoute.
- Before integrating HagiCode, a CLI, or an SDK, keep these two addresses separate:
  - Dashboard: `http://localhost:20128`
  - OpenAI-compatible Base URL: `http://localhost:20128/v1`

## Install and run locally

Start with the recommended global npm path:

```bash
npm install -g omniroute
omniroute
```

After the service starts, validate it in this order:

1. Open `http://localhost:20128` in a browser and confirm that the Dashboard loads.
2. Go to the `Endpoints` page in the Dashboard and create or copy an API key.
3. Note the Base URL you will hand to tools later: `http://localhost:20128/v1`.

:::note[Why start with the Dashboard]
For the first setup, the Dashboard is the fastest way to confirm that OmniRoute is running, your providers are connected, and an API key exists. If a later client call fails, checking the Dashboard first is usually faster than debugging SDK code immediately.
:::

## Validate the running service

### Validation 1: confirm the model catalog is readable

Put the key from the `Endpoints` page into an environment variable:

```bash
export OMNIROUTE_API_KEY="replace-with-the-key-you-copied-from-the-Endpoints-page"
```

Then request `/v1/models`:

```bash
curl http://localhost:20128/v1/models \
  -H "Authorization: Bearer ${OMNIROUTE_API_KEY}"
```

If you get a model list back, the OpenAI-compatible endpoint and auth flow are working.

### Validation 2: send a minimal chat request

```bash
curl http://localhost:20128/v1/chat/completions \
  -H "Authorization: Bearer ${OMNIROUTE_API_KEY}" \
  -H "Content-Type: application/json" \
  -d '{
    "model": "replace-with-a-model-id-from-v1-models",
    "messages": [
      { "role": "user", "content": "Reply with ok." }
    ]
  }'
```

The goal here is not prompt quality. It is to prove that your chosen model can already be routed through OmniRoute successfully.

## Application integration

### The four values you actually need

| Item | Value | Notes |
| --- | --- | --- |
| Dashboard | `http://localhost:20128` | Use it to inspect runtime state, providers, and endpoints. |
| Base URL | `http://localhost:20128/v1` | Use it in OpenAI-compatible clients, CLIs, or SDKs. |
| API Key | From Dashboard `Endpoints` | Copy a real key from the Dashboard instead of inventing a placeholder. |
| Model | Pick one from `/v1/models` | Query the model catalog first, then paste a real model ID into the client. |

### Minimal OpenAI SDK example

```ts
import OpenAI from "openai";

const client = new OpenAI({
  baseURL: "http://localhost:20128/v1",
  apiKey: process.env.OMNIROUTE_API_KEY,
});

const response = await client.chat.completions.create({
  model: process.env.OMNIROUTE_MODEL ?? "replace-with-a-model-id-from-v1-models",
  messages: [{ role: "user", content: "Reply with ok." }],
});

console.log(response.choices[0]?.message?.content);
```

The important part is simple: point `baseURL` at OmniRoute, and use a real model ID that already appeared in `/v1/models`.

### HagiCode-oriented integration example

If you also want HagiCode to use the same OmniRoute instance, think about it as two layers:

1. **Tool layer**: In the Agent CLI or OpenAI-compatible client that will send requests, set:
   - Base URL: `http://localhost:20128/v1`
   - API Key: copied from the Dashboard `Endpoints` page
2. **Shared HagiCode integration layer**: In the HagiCode configuration panel, open the `OmniRoute Integration` tab and enter only the OmniRoute **base URL**, which is `http://localhost:20128`.

The current HagiCode integration uses that base URL for health checks and to refresh the OmniRoute `/v1/models` catalog. In other words, do not manually append `/v1/models` on the HagiCode side, because the system normalizes that path automatically.

If your local Agent CLI setup is not ready yet, continue with [AI Agent CLI Installation](/en/related-software-installation/ai-agent-cli/).

## Optional deployment paths

Get the local npm path working first, then consider these optional routes:

- **Docker**: good when you already prefer containerized services and want OmniRoute isolated.
- **VPS or remote host**: good when you want one always-on OpenAI-compatible endpoint shared across devices.
- **More advanced gateway features**: for MCP, A2A, or more complex provider orchestration, use the upstream OmniRoute docs as the source of truth instead of expanding your first setup session.

:::note[When to move to an optional path]
Only move to Docker or remote hosting after the local `npm install -g omniroute` path is already validated. Otherwise you will have a harder time separating OmniRoute issues from container, network, or host-level issues.
:::

## Best practices

- Validate the local recommended path first, then move to Docker, VPS, or shared team instances.
- Treat `/v1/models` as the single source of truth for model IDs instead of typing model names from memory.
- Keep `http://localhost:20128` and `http://localhost:20128/v1` distinct: the first is for operations and observation, the second is for client requests.
- If HagiCode shows a stale model catalog, go back to the Dashboard first and then re-run `/v1/models`.
- On a first setup, change one variable at a time, such as verifying the API key before changing the model or provider.
