## About OpenCode

This page intentionally splits the workflow into two parts:

1. install OpenCode CLI using the official OpenCode instructions
2. then validate the runtime and provider baseline that HagiCode currently expects

That keeps OpenCode installation issues separate from HagiCode integration issues.

> Note: as of March 21, 2026, HagiCode marks OpenCode as a docs-only primary profession in the UI. The app links to this page instead of offering the one-click install flow used for Claude Code, Codex, GitHub Copilot, and Codebuddy.

## Finish the official OpenCode installation first

### Common install methods

Based on the current `repos/opencode/README.md`, common OpenCode CLI install paths include:

```bash
# Official install script
curl -fsSL https://opencode.ai/install | bash
```

```bash
# npm
npm i -g opencode-ai@latest
```

```bash
# Homebrew
brew install anomalyco/tap/opencode
```

```powershell
# Windows
scoop install opencode
```

```powershell
# Windows
choco install opencode
```

If you use Arch Linux, mise, nix, or another supported path, keep following the options listed in the upstream OpenCode README.

### Minimal post-install check

After installation, run:

```bash
opencode --version
```

**Expected result**: the terminal prints a version string and confirms that `opencode` is available on PATH.

### Optional: desktop app

The OpenCode README also documents a desktop app:

```bash
brew install --cask opencode-desktop
```

For HagiCode integration, the CLI is enough. The desktop app is optional.

## HagiCode's current OpenCode integration baseline

In the current repository, HagiCode treats OpenCode as a dedicated primary profession with both frontend and backend assumptions.

### Default primary profession parameters

`repos/web/src/types/hero.ts` currently defines these OpenCode defaults:

```yaml
binary: opencode
runtimeModel: anthropic/claude-sonnet-4
managedRuntimeModel: glm-4.7
requestTimeoutSeconds: 3600
```

That means:

- the CLI executable is expected to be `opencode`
- the current primary runtime model defaults to `anthropic/claude-sonnet-4`
- HagiCode also keeps a managed runtime marker for `glm-4.7`
- longer-running requests are expected compared with lighter providers

### Backend / runtime configuration reference

The local HagiCode architecture article shows this OpenCode example:

```yaml
AI:
  Providers:
    OpenCodeCli:
      Type: "OpenCodeCli"
      Enabled: true
      ExecutablePath: "opencode"
      Model: "anthropic/claude-sonnet-4"
      WorkingDirectory: null

OpenCode:
  Enabled: true
  BaseUrl: "http://localhost:38376"
  ExecutablePath: "opencode"
  StartupTimeoutSeconds: 30
  RequestTimeoutSeconds: 120
```

From that baseline, HagiCode currently expects at least:

- the `OpenCodeCli` provider is enabled
- `opencode` can be found locally
- the OpenCode runtime can start correctly
- the local HTTP endpoint is reachable from HagiCode

## Recommended validation order

Use this sequence to reduce ambiguity:

1. complete the official OpenCode installation
2. run `opencode --version`
3. start OpenCode once outside HagiCode and confirm the CLI initializes cleanly
4. return to HagiCode and inspect provider status, startup logs, or health checks
5. send one small request and verify `OpenCodeCli` returns content

## Common troubleshooting

| Symptom | More likely cause | Suggested resolution |
|---|---|---|
| `opencode: command not found` | The CLI is installed but PATH is not active in the current environment | Restart the terminal and confirm the install directory is on PATH |
| HagiCode cannot start OpenCode | HagiCode cannot resolve `ExecutablePath: "opencode"` | First verify `opencode --version` on the same machine, then inspect HagiCode startup environment |
| OpenCode runtime does not respond | The local runtime failed to start or the configured `BaseUrl` is unreachable | Confirm OpenCode starts outside HagiCode, then re-check endpoint and port settings |
| Requests keep timing out | The request exceeds the current runtime window or the model responds too slowly | Try a smaller request first, then inspect timeout settings if needed |
| Model behavior is unexpected | `Model`, `runtimeModel`, and `managedRuntimeModel` are not aligned | Compare your current values against the HagiCode baseline and standardize them |
