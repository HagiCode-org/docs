## About CodeBuddy

This page intentionally separates two concerns:

1. follow the official CodeBuddy installation guide to get the CLI onto the machine
2. add the extra ACP validation that HagiCode currently needs

That keeps the installation wording aligned with the official docs instead of replacing it with our own summary.

## Scope and Official Sources

- **Target**: users who want a working CodeBuddy CLI first, and then want to enable it in HagiCode
- **Last verified**: 2026-03-13
- **Official installation guide**: [CodeBuddy CLI Installation](https://www.codebuddy.ai/docs/cli/installation)
- **Official ACP guide**: [CodeBuddy ACP Overview](https://www.codebuddy.ai/docs/cli/acp/overview)
- **HagiCode integration baseline**: `repos/hagicode-core` currently uses `ExecutablePath: "codebuddy"` and `Arguments: "--acp"`

## Complete the official installation first

### Official prerequisites

According to the official installation page, package-manager installs require:

- Node.js `18.20` or newer
- a working terminal such as PowerShell, Terminal, bash, or zsh
- normal network connectivity

If you do not want to depend on Node.js, the official page also documents a **Native Binary (Beta)** path.

### Official install option 1: package managers

The official page lists `npm`, `pnpm`, `yarn`, `bun`, and Homebrew. The most common commands are:

```bash
npm install -g @tencent-ai/codebuddy-code
```

```bash
pnpm add -g @tencent-ai/codebuddy-code
```

```bash
brew install Tencent-CodeBuddy/tap/codebuddy-code
```

After installation, the official page asks you to verify with:

```bash
codebuddy --version
```

**Expected result**: the terminal prints a version string, which confirms that the `codebuddy` command is available.

### Official install option 2: native binary (Beta)

The official installation page also provides a native-binary path with these benefits:

- a single executable with no extra runtime dependency
- no Node.js requirement
- a better auto-update flow

The official install scripts are:

```bash
curl -fsSL https://copilot.tencent.com/cli/install.sh | bash
```

```powershell
irm https://copilot.tencent.com/cli/install.ps1 | iex
```

The official page says the current Beta covers:

- macOS (Apple Silicon and Intel)
- Linux (`arm64` / `x86_64`)
- Windows (`x86_64`)

If the command still is not available afterwards, the official doc recommends checking PATH. On macOS and Linux that usually means confirming `$HOME/.local/bin` is included.

### Official update and recovery actions

The installation page also gives these follow-up commands:

```bash
codebuddy update
```

```bash
npm install -g @tencent-ai/codebuddy-code
```

The first command performs an update based on the current install method. The second is the manual fallback for package-manager installs.

## Extra ACP validation required by HagiCode

The steps above are enough to finish the official installation. For HagiCode, you also need to confirm that ACP startup works.

### 1) Validate ACP startup

The official ACP guide says external tools can launch CodeBuddy like this:

```bash
codebuddy --acp
```

For HagiCode, we at least recommend confirming that the local CLI does not reject the ACP flag. If you want a lighter probe first, you can also try:

```bash
codebuddy --acp --help
```

### 2) If you use API-key auth, apply the official ACP env vars

The official ACP guide documents these environment variables:

```bash
export CODEBUDDY_API_KEY="your-api-key"
export CODEBUDDY_INTERNET_ENVIRONMENT="ioa"
```

In PowerShell:

```powershell
$env:CODEBUDDY_API_KEY = "your-api-key"
$env:CODEBUDDY_INTERNET_ENVIRONMENT = "ioa"
```

:::note
The official ACP guide explicitly says CodeBuddy `0.3.3.21` and later use `ioa`, while older versions use `internal`. That is why this page no longer tells you to use `public`.
:::

### 3) Confirm HagiCode can detect CodeBuddy

Recommended validation order:

1. finish the official installation and run `codebuddy --version`
2. confirm `codebuddy --acp` can start
3. start HagiCode and inspect its health check, provider status page, or startup logs
4. confirm there is no missing `codebuddy` or ACP startup error
5. send a small test request and verify `CodebuddyCli` returns a response

## Troubleshooting

| Symptom | Best interpretation based on the official docs | Suggested resolution |
|---|---|---|
| `codebuddy: command not found` | The official install completed, but PATH did not take effect | Restart the terminal, re-check PATH, and for native-binary installs confirm `$HOME/.local/bin` or the Windows user install directory is on PATH |
| `codebuddy --acp` fails | The install doc covers installation only; ACP behavior is described separately in the ACP guide | First verify the CLI installation itself, then re-check ACP support and startup arguments |
| API-key auth fails | ACP env vars are incomplete or `CODEBUDDY_INTERNET_ENVIRONMENT` is wrong | Set `CODEBUDDY_API_KEY`; use `ioa` for `0.3.3.21+` and `internal` for older versions |
| HagiCode cannot detect CodeBuddy | HagiCode starts without the same working environment as your terminal session | First confirm `codebuddy --version` and `codebuddy --acp` work on the same machine, then inspect HagiCode startup logs |
| Package-manager updates fail | The official install page recommends reinstalling through the package manager | Try `codebuddy update` first, then rerun the package-manager install command |
