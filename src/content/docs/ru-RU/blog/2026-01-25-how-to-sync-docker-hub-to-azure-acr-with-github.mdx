## Реализация автоматической синхронизации образов из Docker Hub в Azure ACR

> В этой статье описывается, как использовать GitHub Actions и утилиту image-syncer для автоматической синхронизации образов Docker Hub с Azure Container Registry. Это решает проблему медленного доступа к Docker Hub в Китае и некоторых регионах Azure, повышая доступность образов и эффективность развертывания в среде Azure.

## Предпосылки/Введение

Проект HagiCode использует Docker-образы в качестве основных компонентов среды выполнения, основные образы размещены на Docker Hub. С развитием проекта и ростом потребностей в развертывании в среде Azure мы столкнулись со следующими проблемами:
- Медленная скорость загрузки образов, доступ к Docker Hub ограничен в Китае и некоторых регионах Azure
- Зависимость от единственного источника образов создаёт риск единой точки отказа
- Использование Azure Container Registry в среде Azure обеспечивает лучшую производительность сети и интеграцию

Для решения этих проблем нам необходимо создать автоматический механизм синхронизации образов, который будет регулярно синхронизировать образы из Docker Hub в Azure ACR, гарантируя, что пользователи смогут получить более высокую скорость загрузки и доступность в среде Azure.

## Сравнение технических решений

При разработке решения мы сравнили несколько технических подходов:

### 1. image-syncer (окончательный выбор)
- Инкрементальная синхронизация: синхронизируются только изменённые слои образов, что значительно сокращает сетевую передачу
- Возобновление передачи: возможность возобновить синхронизацию после прерывания сети
- Управление параллелизмом: поддержка настройки количества параллельных потоков, повышение эффективности для больших образов
- Полная обработка ошибок: встроенный механизм повторных попыток при сбоях (по умолчанию 3 раза)
- Лёгкое развертывание: один бинарный файл, без зависимостей
- Поддержка нескольких реестров: совместим с Docker Hub, Azure ACR, Harbour и др.

### 2. Docker CLI
- Нет поддержки инкрементальной синхронизации: каждый раз нужно загружать полное содержимое образа
- Низкая эффективность: большой объём сетевой передачи, долгое время
- Простота использования: использование знакомых команд docker pull/push

### 3. Azure CLI
- Высокая сложность: требуется настройка аутентификации Azure CLI
- Функциональные ограничения: функция az acr import относительно ограничена
- Нативная интеграция: хорошая интеграция со службами Azure

## Решения по архитектуре

### Решение 1: Частота синхронизации установлена на ежедневно в 00:00 UTC
- Баланс между свежестью образов и потреблением ресурсов
- Избегание пиковых часов нагрузки, уменьшение влияния на другие операции
- Образы Docker Hub обычно обновляются после ежедневной сборки

### Решение 2: Синхронизация всех тегов образов
- Полное соответствие с Docker Hub
- Предоставление пользователям гибкого выбора версий
- Упрощение логики синхронизации, избегание сложных правил фильтрации тегов

### Решение 3: Использование GitHub Secrets для хранения аутентификационной информации
- Нативная поддержка GitHub Actions, высокая безопасность
- Простая настройка, лёгкое управление и обслуживание
- Поддержка контроля доступа на уровне репозитория

## Оценка рисков и смягчение

### Риск 1: Утечка аутентификационной информации Azure ACR
- Использование шифрования GitHub Secrets для хранения
- Регулярная ротация паролей ACR
- Ограничение прав пользователя ACR только на push
- Мониторинг журналов доступа ACR

### Риск 2: Сбой синхронизации приводит к несоответствию образов
- Встроенный механизм инкрементальной синхронизации image-syncer
- Автоматический повтор при сбоях (по умолчанию 3 раза)
- Подробные журналы ошибок и уведомления о сбоях
- Функция возобновления передачи

### Риск 3: Чрезмерное потребление ресурсов
- Инкрементальная синхронизация сокращает сетевую передачу
- Настраиваемое количество параллельных потоков (в настоящее время установлено 10)
- Мониторинг количества и размера синхронизируемых образов
- Запуск синхронизации в непиковые часы

## Основное решение

Мы используем автоматическое решение GitHub Actions + image-syncer для реализации синхронизации образов из Docker Hub в Azure ACR.

## Этапы реализации

### 1. Подготовительный этап
- Создание или подтверждение Azure Container Registry в Azure Portal
- Создание ключа доступа ACR (имя пользователя и пароль)
- Подтверждение прав доступа к репозиторию образов Docker Hub

### 2. Настройка GitHub Secrets
В настройках репозитория GitHub добавьте следующие Secrets:
- AZURE_ACR_USERNAME: имя пользователя Azure ACR
- AZURE_ACR_PASSWORD: пароль Azure ACR

### 3. Создание рабочего процесса GitHub Actions
Настройте рабочий процесс в .github/workflows/sync-docker-acr.yml:
- Плановый запуск: каждый день в 00:00 UTC
- Ручной запуск: поддержка workflow_dispatch
- Дополнительный запуск: при push в ветку publish (для быстрой синхронизации)

### 4. Процесс выполнения рабочего процесса

| Порядок | Участник | Действие | Описание |
| --- | --- | --- | --- |
| 1 | GitHub Actions | Запуск рабочего процесса | По расписанию, вручную или при push в ветку `publish` |
| 2 | GitHub Actions → image-syncer | Загрузка и выполнение утилиты синхронизации | Вход в фактический этап синхронизации |
| 3 | image-syncer → Docker Hub | Получение manifest и списка тегов | Чтение метаданных исходного репозитория |
| 4 | image-syncer → Azure ACR | Получение информации о существующих образах в целевом репозитории | Определение текущего состояния на целевой стороне |
| 5 | image-syncer | Сравнение различий между источником и целью | Идентификация слоёв образов, требующих синхронизации |
| 6 | image-syncer → Docker Hub | Загрузка изменённых слоёв образа | Передача только содержимого, которое нужно обновить |
| 7 | image-syncer → Azure ACR | Отправка изменённых слоёв образа | Завершение инкрементальной синхронизации |
| 8 | image-syncer → GitHub Actions | Возврат статистической информации о синхронизации | Включает результаты, различия и информацию об ошибках |
| 9 | GitHub Actions | Запись журналов и загрузка артефактов | Удобство последующего аудита и устранения неполадок |

## Реализация рабочего процесса GitHub Actions

Ниже приведена фактическая конфигурация рабочего процесса (.github/workflows/sync-docker-acr.yml):

```yaml
name: Sync Docker Image to Azure ACR

on:
  schedule:
    - cron: "0 0 * * *" # Каждый день в 00:00 UTC
  workflow_dispatch: # Ручной запуск
  push:
    branches: [publish]

permissions:
  contents: read

jobs:
  sync:
    runs-on: ubuntu-latest

    steps:
      - name: Checkout code
        uses: actions/checkout@v4

      - name: Download image-syncer
        run: |
          # Загрузка бинарного файла image-syncer
          wget https://github.com/AliyunContainerService/image-syncer/releases/download/v1.5.5/image-syncer-v1.5.5-linux-amd64.tar.gz
          tar -zxvf image-syncer-v1.5.5-linux-amd64.tar.gz
          chmod +x image-syncer

      - name: Create auth config
        run: |
          # Генерация файла аутентификации (формат YAML)
          cat > auth.yaml <<EOF
          hagicode.azurecr.io:
            username: "${{ secrets.AZURE_ACR_USERNAME }}"
            password: "${{ secrets.AZURE_ACR_PASSWORD }}"
          EOF

      - name: Create images config
        run: |
          # Генерация файла конфигурации синхронизации образов (формат YAML)
          cat > images.yaml <<EOF
          docker.io/newbe36524/hagicode: hagicode.azurecr.io/hagicode
          EOF

      - name: Run image-syncer
        run: |
          # Выполнение синхронизации (использование новых параметров --auth и --images)
          ./image-syncer --auth=./auth.yaml --images=./images.yaml --proc=10 --retries=3

      - name: Upload logs
        if: always()
        uses: actions/upload-artifact@v4
        with:
          name: sync-logs
          path: image-syncer-*.log
          retention-days: 7
```

## Описание конфигурации

### 1. Условия запуска
- Плановый запуск: cron: "0 0 * * *" — выполнение каждый день в 00:00 UTC
- Ручной запуск: workflow_dispatch — позволяет пользователям запускать вручную через GitHub UI
- Запуск при push: push: branches: [publish] — запуск при push в ветку публикации (для быстрой синхронизации)

### 2. Конфигурация аутентификации (auth.yaml)
```yaml
hagicode.azurecr.io:
  username: "${{ secrets.AZURE_ACR_USERNAME }}"
  password: "${{ secrets.AZURE_ACR_PASSWORD }}"
```

### 3. Конфигурация синхронизации образов (images.yaml)
```yaml
docker.io/newbe36524/hagicode: hagicode.azurecr.io/hagicode
```
Эта конфигурация означает синхронизацию всех тегов docker.io/newbe36524/hagicode в hagicode.azurecr.io/hagicode

### 4. Параметры image-syncer
- --auth=./auth.yaml: путь к файлу конфигурации аутентификации
- --images=./images.yaml: путь к файлу конфигурации синхронизации образов
- --proc=10: количество параллельных потоков равно 10
- --retries=3: 3 повторных попытки при сбое

## Меры предосторожности и лучшие практики

### 1. Рекомендации по безопасности
- Регулярная ротация пароля Azure ACR (рекомендуется каждые 90 дней)
- Использование выделенной учётной записи ACR с ограничением прав только на push
- Мониторинг журналов доступа ACR, своевременное обнаружение аномального доступа
- Не выводите аутентификационную информацию в журналы
- Не коммитьте аутентификационную информацию в репозиторий кода

### 2. Оптимизация производительности
- Настройка параметра --proc: настройте количество параллельных потоков в зависимости от пропускной способности сети (рекомендуется 5-20)
- Мониторинг времени синхронизации: если время синхронизации слишком велико, рассмотрите возможность уменьшения количества параллельных потоков
- Регулярная очистка журналов: установка разумного retention-days (в настоящее время 7 дней)

### 3. Устранение неполадок

#### Проблема 1: Ошибка аутентификации
```
Error: failed to authenticate to hagicode.azurecr.io
```
Решение:
1. Проверьте, правильно ли настроены GitHub Secrets
2. Проверьте, не истёк ли срок действия пароля Azure ACR
3. Подтвердите правильность прав учётной записи службы ACR

#### Проблема 2: Тайм-аут сети
```
Error: timeout waiting for response
```
Решение:
1. Проверьте сетевое подключение
2. Уменьшите количество параллельных потоков (параметр --proc)
3. После восстановления сети перезапустите рабочий процесс вручную

#### Проблема 3: Неполная синхронизация образа
```
Warning: some tags failed to sync
```
Решение:
1. Проверьте журналы синхронизации, определите неудачные теги
2. Вручную запустите рабочий процесс для повторной синхронизации
3. Проверьте, нормально ли работает исходный образ Docker Hub

### 4. Мониторинг и оповещения
- Регулярно проверяйте страницу Actions, подтверждайте статус выполнения рабочего процесса
- Настройте уведомления GitHub для своевременного получения уведомлений о сбоях рабочего процесса
- Мониторьте использование хранилища Azure ACR
- Регулярно проверяйте соответствие тегов образов

## Частые вопросы и решения

### Q1: Как синхронизировать определённые теги вместо всех тегов?

Измените файл конфигурации images.yaml:
```yaml
# Синхронизировать только теги latest и v1.0
docker.io/newbe36524/hagicode:latest: hagicode.azurecr.io/hagicode:latest
docker.io/newbe36524/hagicode:v1.0: hagicode.azurecr.io/hagicode:v1.0
```

### Q2: Как синхронизировать несколько репозиториев образов?

Добавьте несколько строк конфигурации в images.yaml:
```yaml
docker.io/newbe36524/hagicode: hagicode.azurecr.io/hagicode
docker.io/newbe36524/another-image: hagicode.azurecr.io/another-image
```

### Q3: Как повторить попытку после неудачной синхронизации?
- Автоматический повтор: встроенный механизм повторных попыток image-syncer (по умолчанию 3 раза)
- Ручной повтор: нажмите Re-run all jobs на странице GitHub Actions

### Q4: Как просмотреть подробный прогресс синхронизации?
- Просмотр журналов в реальном времени на странице Actions
- Скачайте artifact sync-logs для просмотра полного файла журнала
- Файл журнала содержит статус синхронизации и скорость передачи для каждого тега

### Q5: Сколько времени занимает синхронизация?
- Первая полная синхронизация: в зависимости от размера образа, обычно 10-30 минут
- Инкрементальная синхронизация: если изменения образа незначительны, обычно 2-5 минут
- Время зависит от пропускной способности сети, размера образа и настроек параллелизма
