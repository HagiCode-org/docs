## Автоматизация синхронизации образов из Docker Hub в Azure ACR

> В этой статье описывается, как с помощью GitHub Actions и утилиты image-syncer реализовать автоматическую синхронизацию Docker Hub образов в Azure Container Registry, решая проблему медленного доступа к Docker Hub в материковом Китае и некоторых регионах Azure, а также повышая доступность образов и эффективность развёртывания в среде Azure.

## Предпосылки / Введение

Проект HagiCode использует Docker-образы в качестве основных компонентов среды выполнения, при этом основные образы размещены на Docker Hub. По мере развития проекта и роста потребностей в развёртывании в Azure мы столкнулись со следующими проблемами:
- Медленная загрузка образов, так как доступ к Docker Hub ограничен в материковом Китае и некоторых регионах Azure
- Зависимость от единственного источника образов создаёт риск единой точки отказа
- Использование Azure Container Registry в среде Azure обеспечивает лучшую сетевую производительность и интеграцию

Для решения этих проблем нам необходимо создать автоматизированный механизм синхронизации образов, регулярно синхронизирующий образы из Docker Hub в Azure ACR, чтобы обеспечить пользователям более высокую скорость загрузки и лучшую доступность в среде Azure.

## Сравнение технических решений

При разработке решения мы сравнили несколько технических подходов:

### 1. image-syncer (итоговый выбор)
- Инкрементальная синхронизация: синхронизируются только изменённые слои образа, что значительно сокращает сетевой трафик
- Поддержка докачки: синхронизация может быть возобновлена после сетевых сбоев
- Управление параллелизмом: поддерживается настраиваемое количество параллельных потоков для повышения эффективности синхронизации больших образов
- Надёжная обработка ошибок: встроенный механизм повторных попыток (по умолчанию 3)
- Легковесное развёртывание: один бинарный файл без зависимостей
- Поддержка нескольких реестров: совместимость с Docker Hub, Azure ACR, Harbor и другими

### 2. Docker CLI
- Нет поддержки инкрементальной синхронизации: каждый раз требуется загрузка полного содержимого образа
- Низкая эффективность: большой объём сетевого трафика и длительное время выполнения
- Простота использования: использует знакомые команды `docker pull` / `docker push`

### 3. Azure CLI
- Высокая сложность: требует настройки аутентификации Azure CLI
- Функциональные ограничения: `az acr import` относительно ограничен в возможностях
- Нативная интеграция: хорошо интегрируется со службами Azure

## Архитектурные решения

### Решение 1: Частота синхронизации — ежедневно в 00:00 UTC
- Баланс между свежестью образов и потреблением ресурсов
- Избегание пиковых часов нагрузки, снижение влияния на другие операции
- Образы Docker Hub обычно обновляются после ежедневных сборок

### Решение 2: Синхронизация всех тегов образа
- Поддержание полной согласованности с Docker Hub
- Гибкий выбор версий для пользователей
- Упрощение логики синхронизации за счёт отказа от сложных правил фильтрации тегов

### Решение 3: Хранение учётных данных в GitHub Secrets
- Нативная поддержка GitHub Actions с высоким уровнем безопасности
- Простота настройки, удобство управления и обслуживания
- Поддержка контроля доступа на уровне репозитория

## Оценка рисков и их смягчение

### Риск 1: Утечка учётных данных Azure ACR
- Использование GitHub Secrets для зашифрованного хранения
- Регулярная ротация паролей ACR
- Ограничение прав учётной записи ACR только на push
- Мониторинг журналов доступа ACR

### Риск 2: Сбои синхронизации, приводящие к несогласованности образов
- image-syncer включает встроенный механизм инкрементальной синхронизации
- Автоматический повтор при сбоях (по умолчанию 3 раза)
- Подробные журналы ошибок и уведомления о сбоях
- Поддержка докачки

### Риск 3: Чрезмерное потребление ресурсов
- Инкрементальная синхронизация сокращает сетевой трафик
- Настраиваемое количество параллельных потоков (в текущей настройке — 10)
- Мониторинг количества и размера синхронизируемых образов
- Запуск синхронизации в непиковые часы

## Основное решение

Мы используем автоматизированное решение на базе GitHub Actions + image-syncer для синхронизации образов из Docker Hub в Azure ACR.

## Порядок реализации

### 1. Подготовительный этап
- Создание или подтверждение Azure Container Registry в Azure Portal
- Создание учётных данных доступа ACR (имя пользователя и пароль)
- Подтверждение прав доступа к репозиторию образов Docker Hub

### 2. Настройка GitHub Secrets
Добавьте следующие секреты в настройках репозитория GitHub:
- AZURE_ACR_USERNAME: имя пользователя Azure ACR
- AZURE_ACR_PASSWORD: пароль Azure ACR

### 3. Создание рабочего процесса GitHub Actions
Настройте рабочий процесс в `.github/workflows/sync-docker-acr.yml`:
- Плановый запуск: каждый день в 00:00 UTC
- Ручной запуск: поддержка `workflow_dispatch`
- Дополнительный триггер: запуск при push в ветку `publish` (для быстрой синхронизации)

### 4. Порядок выполнения рабочего процесса

| Шаг | Участник | Действие | Описание |
| --- | --- | --- | --- |
| 1 | GitHub Actions | Запуск рабочего процесса | По расписанию, вручную или при push в ветку `publish` |
| 2 | GitHub Actions → image-syncer | Загрузка и запуск инструмента синхронизации | Переход к фактической синхронизации |
| 3 | image-syncer → Docker Hub | Получение манифеста образа и списка тегов | Чтение метаданных исходного репозитория |
| 4 | image-syncer → Azure ACR | Получение информации о существующих образах в целевом репозитории | Определение текущего состояния целевой стороны |
| 5 | image-syncer | Сравнение различий между источником и целью | Идентификация слоёв образа, требующих синхронизации |
| 6 | image-syncer → Docker Hub | Загрузка изменённых слоёв образа | Передача только обновлённого содержимого |
| 7 | image-syncer → Azure ACR | Отправка изменённых слоёв образа | Завершение инкрементальной синхронизации |
| 8 | image-syncer → GitHub Actions | Возврат статистики синхронизации | Включает результаты, различия и информацию об ошибках |
| 9 | GitHub Actions | Запись журналов и загрузка артефактов | Для последующего аудита и устранения неполадок |

## Реализация рабочего процесса GitHub Actions

Ниже представлена фактическая конфигурация рабочего процесса (`.github/workflows/sync-docker-acr.yml`):

```yaml
name: Sync Docker Image to Azure ACR

on:
  schedule:
    - cron: "0 0 * * *" # Каждый день в 00:00 UTC
  workflow_dispatch: # Ручной запуск
  push:
    branches: [publish]

permissions:
  contents: read

jobs:
  sync:
    runs-on: ubuntu-latest

    steps:
      - name: Checkout code
        uses: actions/checkout@v4

      - name: Download image-syncer
        run: |
          # Загрузка бинарного файла image-syncer
          wget https://github.com/AliyunContainerService/image-syncer/releases/download/v1.5.5/image-syncer-v1.5.5-linux-amd64.tar.gz
          tar -zxvf image-syncer-v1.5.5-linux-amd64.tar.gz
          chmod +x image-syncer

      - name: Create auth config
        run: |
          # Генерация файла аутентификации (формат YAML)
          cat > auth.yaml <<EOF
          hagicode.azurecr.io:
            username: "${{ secrets.AZURE_ACR_USERNAME }}"
            password: "${{ secrets.AZURE_ACR_PASSWORD }}"
          EOF

      - name: Create images config
        run: |
          # Генерация файла конфигурации синхронизации образов (формат YAML)
          cat > images.yaml <<EOF
          docker.io/newbe36524/hagicode: hagicode.azurecr.io/hagicode
          EOF

      - name: Run image-syncer
        run: |
          # Выполнение синхронизации (с использованием новых параметров --auth и --images)
          ./image-syncer --auth=./auth.yaml --images=./images.yaml --proc=10 --retries=3

      - name: Upload logs
        if: always()
        uses: actions/upload-artifact@v4
        with:
          name: sync-logs
          path: image-syncer-*.log
          retention-days: 7
```

## Описание конфигурации

### 1. Условия запуска
- Плановый запуск: `cron: "0 0 * * *"` — выполняется каждый день в 00:00 UTC
- Ручной запуск: `workflow_dispatch` — позволяет пользователям запускать вручную через интерфейс GitHub
- Запуск по push: `push: branches: [publish]` — срабатывает при push в ветку publish (для быстрой синхронизации)

### 2. Конфигурация аутентификации (`auth.yaml`)
```yaml
hagicode.azurecr.io:
  username: "${{ secrets.AZURE_ACR_USERNAME }}"
  password: "${{ secrets.AZURE_ACR_PASSWORD }}"
```

### 3. Конфигурация синхронизации образов
```yaml
docker.io/newbe36524/hagicode: hagicode.azurecr.io/hagicode
```
Эта конфигурация означает синхронизацию всех тегов из `docker.io/newbe36524/hagicode` в `hagicode.azurecr.io/hagicode`

### 4. Параметры image-syncer
- `--auth=./auth.yaml`: путь к файлу конфигурации аутентификации
- `--images=./images.yaml`: путь к файлу конфигурации синхронизации образов
- `--proc=10`: количество параллельных потоков — 10
- `--retries=3`: количество повторных попыток при сбоях — 3

## Замечания и лучшие практики

### 1. Рекомендации по безопасности
- Регулярно ротируйте пароли Azure ACR (рекомендуется каждые 90 дней)
- Используйте выделенную сервисную учётную запись ACR с правами только на push
- Мониторьте журналы доступа ACR для своевременного обнаружения аномалий
- Не выводите учётные данные в журналах
- Не коммитьте учётные данные в репозиторий кода

### 2. Оптимизация производительности
- Настройте параметр `--proc`: подбирайте параллелизм в зависимости от пропускной способности сети (рекомендуется 5–20)
- Мониторьте время синхронизации: если оно слишком велико, рассмотрите снижение параллелизма
- Регулярно очищайте журналы: устанавливайте разумное значение `retention-days` (в текущей настройке — 7 дней)

### 3. Устранение неполадок

#### Проблема 1: Ошибка аутентификации
```
Error: failed to authenticate to hagicode.azurecr.io
```
Решение:
1. Проверьте правильность настройки GitHub Secrets
2. Убедитесь, что пароль Azure ACR не истёк
3. Подтвердите правильность прав сервисной учётной записи ACR

#### Проблема 2: Таймаут сети
```
Error: timeout waiting for response
```
Решение:
1. Проверьте сетевое подключение
2. Уменьшите количество параллельных потоков (параметр `--proc`)
3. Дождитесь восстановления сети и перезапустите рабочий процесс

#### Проблема 3: Неполная синхронизация образов
```
Warning: some tags failed to sync
```
Решение:
1. Проверьте журналы синхронизации, чтобы выявить теги с ошибками
2. Запустите рабочий процесс вручную для повторной синхронизации
3. Убедитесь, что исходный образ на Docker Hub доступен

### 4. Мониторинг и оповещения
- Регулярно проверяйте страницу `Actions` для подтверждения статуса выполнения рабочего процесса
- Настройте уведомления GitHub для оперативного получения информации о сбоях
- Мониторьте использование хранилища Azure ACR
- Регулярно проверяйте согласованность тегов

## Часто задаваемые вопросы и решения

### В1: Как синхронизировать определённые теги вместо всех?

Измените файл конфигурации `images.yaml`:
```yaml
# Синхронизация только тегов latest и v1.0
docker.io/newbe36524/hagicode:latest: hagicode.azurecr.io/hagicode:latest
docker.io/newbe36524/hagicode:v1.0: hagicode.azurecr.io/hagicode:v1.0
```

### В2: Как синхронизировать несколько репозиториев образов?

Добавьте несколько строк в `images.yaml`:
```yaml
docker.io/newbe36524/hagicode: hagicode.azurecr.io/hagicode
docker.io/newbe36524/another-image: hagicode.azurecr.io/another-image
```

### В3: Как повторить попытку после сбоя синхронизации?
- Автоматический повтор: image-syncer включает встроенный механизм повторных попыток (по умолчанию 3)
- Ручной повтор: нажмите `Re-run all jobs` на странице GitHub Actions

### В4: Как просмотреть подробный прогресс синхронизации?
- Просмотр журналов в реальном времени на странице `Actions`
- Скачайте артефакт `sync-logs` для просмотра полного файла журнала
- Файл журнала содержит статус синхронизации и скорость передачи для каждого тега

### В5: Сколько времени занимает синхронизация?
- Первоначальная полная синхронизация: обычно 10–30 минут в зависимости от размера образа
- Инкрементальная синхронизация: обычно 2–5 минут при небольших изменениях
- Время зависит от пропускной способности сети, размера образа и настроек параллелизма
