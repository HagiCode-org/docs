## Руководство по реализации функции горячего слова в распознавании речи Doubao

> В этой статье подробно описывается, как реализовать поддержку горячих слов в распознавании речи Doubao в проекте HagiCode. С помощью пользовательских горячих слов и таблиц горячих слов платформы можно значительно повысить точность распознавания словарного запаса в определённых областях.

## Предпосылки

Технологии распознавания речи развиваются уже много лет, но одна проблема всё ещё беспокоит разработчиков. Хотя общие модели распознавания речи могут охватить повседневную лексику, для профессиональных терминов, названий продуктов, имён людей точность распознавания всегда немного недостаточна. Подумайте: голосовым ассистентам в медицинской области необходимо точно распознавать "гипертония", "диабет", "ишемическая болезнь сердца"; юридическим системам нужно точно отслеживать "основание иска", "возражение", "бремя доказывания" — в этих сценариях общие модели показывают себя, так сказать, как могут.

В проекте HagiCode мы столкнулись с той же проблемой. Как многофункциональный ИИ-помощник по коду, HagiCode должен обрабатывать различные сценарии распознавания речи технических терминов. Однако API распознавания речи Doubao по умолчанию не может полностью удовлетворить наши требования к точности профессиональных терминов. На самом деле, Doubao не плох, просто каждая область имеет свою собственную систему терминов. После ряда исследований и технических экспериментов мы обнаружили, что API распознавания речи Doubao фактически предоставляет функцию поддержки горячих слов — достаточно простой конфигурации, чтобы значительно повысить точность распознавания определённых слов. Это немного похоже на то, что вы говорите ему, на какие слова обратить внимание, и он будет более внимательно слушать эти слова.

В этой статье я поделюсь полным решением по реализации функции горячих слов в распознавании речи Doubao в проекте HagiCode. Два режима — пользовательские горячие слова и таблица горячих слов платформы — можно использовать как по отдельности, так и в комбинации. С помощью этого решения разработчики могут гибко настраивать горячие слова в соответствии со сценариями бизнеса, позволяя системе распознавания речи "знать" те профессиональные, редкие, но важнейшие слова.

## Основная реализация

### Анализ двух режимов горячих слов

API распознавания речи Doubao предоставляет нам два способа настройки горячих слов, каждый из которых имеет свои уникальные сценарии применения и преимущества.

**Режим пользовательских горячих слов** позволяет нам напрямую передавать текст горячих слов через поле `corpus.context`. Этот способ非常适合 для сценариев, требующих быстрой настройки небольшого количества горячих слов, например, когда временно необходимо распознать название продукта или имя человека. В реализации HagiCode мы анализируем многострочный текст горячих слов, введённый пользователем, в список строк, а затем форматируем его в массив `context_data` в соответствии с требованиями API Doubao. Как сказать, этот способ очень прямой — словно вы говорите собеседнику "обрати внимание на эти слова", и он обращает внимание.

**Режим таблицы горячих слов платформы** ссылается на предварительно настроенную таблицу горячих слов платформы самобучения Doubao через поле `corpus.boosting_table_id`. Этот способ подходит для сценариев, требующих управления большим количеством горячих слов — мы можем создавать и поддерживать таблицы горячих слов на платформе самобучения Doubao, а затем ссылаться на них по ID. Для таких проектов, как HagiCode, которым необходимо постоянно обновлять и поддерживать профессиональную терминологию, этот режим обеспечивает лучшую управляемость. Ведь когда горячих слов становится много, найти место для централизованного управления определённо лучше, чем каждый раз вручную вводить их.

Интересно, что эти два режима также можно использовать в комбинации. API Doubao поддерживает одновременное включение пользовательских горячих слов и ID таблицы горячих слов платформы в одном запросе, с управлением стратегией комбинации через параметр `combine_mode`. Такая гибкость позволяет HagiCode справляться с различными сложными потребностями распознавания профессиональной терминологии. Это тоже хорошо — иногда комбинация нескольких способов может дать лучший эффект.

### Определение типов и проверка на фронтенде

Во фронтенд-реализации HagiCode мы определили полный набор типов конфигурации горячих слов и логики проверки. Сначала часть определения типов:

```typescript
export interface HotwordConfig {
  contextText: string;           // Многострочный текст горячих слов
  boostingTableId: string;      // ID таблицы горячих слов платформы Doubao
  combineMode: boolean;          // Использовать ли комбинацию
}
```

Этот простой интерфейс содержит все элементы конфигурации функции горячих слов. 其中 `contextText` — это наиболее интуитивно понятная для пользователя часть — мы позволяем пользователю вводить по одной фразе горячего слова на строку, что очень интуитивно. Ведь позволить пользователю вводить по одному слову в строку определённо лучше, чем заставлять его понимать сложные правила конфигурации.

Далее идёт реализация функций проверки. С учётом ограничений API Doubao мы разработали строгие правила проверки: текст горячих слов — максимум 100 строк, каждая строка — максимум 50 символов, всего — максимум 5000 символов; boosting_table_id — максимум 200 символов, разрешены только буквы, цифры, подчёркивания и дефисы. Эти ограничения не придуманы нами, а основаны на фактических требованиях официальной документации Doubao. Ведь ограничения API — это ограничения API, мы ничего не можем поделать,只好 соблюдать.

```typescript
export function validateContextText(contextText: string): HotwordValidationResult {
  if (!contextText || contextText.trim().length === 0) {
    return { isValid: true, errors: [] };
  }

  const lines = contextText.split('\n').filter(line => line.trim().length > 0);
  const errors: string[] = [];

  if (lines.length > 100) {
    errors.push(`Количество строк горячих слов не может превышать 100, текущее значение: ${lines.length}`);
  }

  const totalChars = contextText.length;
  if (totalChars > 5000) {
    errors.push(`Общее количество символов горячих слов не может превышать 5000, текущее значение: ${totalChars}`);
  }

  for (let i = 0; i < lines.length; i++) {
    if (lines[i].length > 50) {
      errors.push(`Строка ${i + 1} горячих слов превышает ограничение в 50 символов`);
    }
  }

  return { isValid: errors.length === 0, errors };
}

export function validateBoostingTableId(boostingTableId: string): HotwordValidationResult {
  if (!boostingTableId || boostingTableId.trim().length === 0) {
    return { isValid: true, errors: [] };
  }

  const errors: string[] = [];

  if (boostingTableId.length > 200) {
    errors.push(`boosting_table_id не может превышать 200 символов, текущее значение: ${boostingTableId.length}`);
  }

  if (!/^[a-zA-Z0-9_-]+$/.test(boostingTableId)) {
    errors.push('boosting_table_id может содержать только буквы, цифры, подчёркивания и дефисы');
  }

  return { isValid: errors.length === 0, errors };
}
```

Эти функции проверки выполняются немедленно, когда пользователь настраивает горячие слова, обеспечивая обнаружение проблем на самом раннем этапе. Для пользовательского опыта такая немедленная обратная связь очень важна. Ведь пользователю лучше узнать, где ошибка, при вводе, чем после отправки.

### Постоянство конфигурации фронтенда

Во фронтенд-реализации HagiCode мы выбрали использование localStorage браузера для хранения конфигурации горячих слов. За этим дизайном есть несколько соображений: во-первых, конфигурация горячих слов — это очень персонализированная настройка, у разных пользователей могут быть разные потребности в профессиональных областях; во-вторых, этот способ упрощает серверную реализацию, не требуя дополнительных таблиц базы данных и API-интерфейсов; наконец, пользователь настраивает один раз в браузере, а последующее использование автоматически загружается, что очень удобно. Говоря прямо, это просто экономит усилия.

```typescript
const HOTWORD_STORAGE_KEYS = {
  contextText: 'hotword-context-text',
  boostingTableId: 'hotword-boosting-table-id',
  combineMode: 'hotword-combine-mode',
} as const;

export const DEFAULT_HOTWORD_CONFIG: HotwordConfig = {
  contextText: '',
  boostingTableId: '',
  combineMode: false,
};

// Загрузка конфигурации горячих слов
export function loadHotwordConfig(): HotwordConfig {
  const contextText = localStorage.getItem(HOTWORD_STORAGE_KEYS.contextText) || '';
  const boostingTableId = localStorage.getItem(HOTWORD_STORAGE_KEYS.boostingTableId) || '';
  const combineMode = localStorage.getItem(HOTWORD_STORAGE_KEYS.combineMode) === 'true';

  return { contextText, boostingTableId, combineMode };
}

// Сохранение конфигурации горячих слов
export function saveHotwordConfig(config: HotwordConfig): void {
  localStorage.setItem(HOTWORD_STORAGE_KEYS.contextText, config.contextText);
  localStorage.setItem(HOTWORD_STORAGE_KEYS.boostingTableId, config.boostingTableId);
  localStorage.setItem(HOTWORD_STORAGE_KEYS.combineMode, String(config.combineMode));
}
```

Логика этого кода очень проста и понятна. При загрузке конфигурации читаем из localStorage, при сохранении — записываем в localStorage. Мы также предоставляем конфигурацию по умолчанию, обеспечивая нормальную работу системы при отсутствии какой-либо конфигурации. Ведь должна же быть какая-то конфигурация по умолчанию.

### Расширение конфигурации SDK на бэкенде

В серверной реализации HagiCode нам нужно добавить свойства, связанные с горячими словами, в класс конфигурации SDK. С учётом особенностей языка C# и привычек использования мы использовали `List<string>` для хранения пользовательского контекста горячих слов:

```csharp
public class DoubaoVoiceConfig
{
    /// <summary>
    /// ID приложения
    /// </summary>
    public string AppId { get; set; } = string.Empty;

    /// <summary>
    /// Токен доступа
    /// </summary>
    public string AccessToken { get; set; } = string.Empty;

    /// <summary>
    /// URL службы
    /// </summary>
    public string ServiceUrl { get; set; } = string.Empty;

    /// <summary>
    /// Список пользовательских контекстов горячих слов
    /// </summary>
    public List<string>? HotwordContexts { get; set; }

    /// <summary>
    /// ID таблицы горячих слов платформы Doubao
    /// </summary>
    public string? BoostingTableId { get; set; }
}
```

Дизайн этого класса конфигурации следует постоянному простому стилю HagiCode. `HotwordContexts` — это тип nullable-списка, `BoostingTableId` — nullable-строка, так что при отсутствии какой-либо конфигурации горячих слов эти свойства не будут влиять на запрос. Ведь когда не используется, не должно существовать — это и называется чистотой.

### Логика построения Payload

Построение Payload — это ядро всей функции горячих слов. Когда у нас есть конфигурация горячих слов, нужно отформатировать её в правильную структуру JSON в соответствии с требованиями API Doubao. Этот процесс происходит перед отправкой запроса SDK:

```csharp
private void AddCorpusToRequest(Dictionary<string, object> request)
{
    var corpus = new Dictionary<string, object>();

    // Добавление пользовательских горячих слов
    if (Config.HotwordContexts != null && Config.HotwordContexts.Count > 0)
    {
        corpus["context"] = new Dictionary<string, object>
        {
            ["context_type"] = "dialog_ctx",
            ["context_data"] = Config.HotwordContexts
                .Select(text => new Dictionary<string, object> { ["text"] = text })
                .ToList()
        };
    }

    // Добавление ID таблицы горячих слов платформы
    if (!string.IsNullOrEmpty(Config.BoostingTableId))
    {
        corpus["boosting_table_id"] = Config.BoostingTableId;
    }

    // Добавляем в запрос только если corpus не пустой
    if (corpus.Count > 0)
    {
        request["corpus"] = corpus;
    }
}
```

Этот код показывает, как динамически строить поле corpus на основе конфигурации. Ключевой момент: мы добавляем поле corpus только если действительно существует конфигурация горячих слов. Этот дизайн обеспечивает обратную совместимость — при отсутствии настройки горячих слов структура запроса полностью соответствует предыдущей. Ведь совместимость важна, нельзя добавлять функцию и ломать предыдущую логику.

### Передача параметров WebSocket

Между фронтендом и бэкендом параметры горячих слов передаются через управляющие сообщения WebSocket. Дизайн HagiCode: фронтенд загружает конфигурацию горячих слов из localStorage при начале записи, а затем отправляет её бэкенду через сообщение WebSocket.

```typescript
const controlMessage = {
  type: 'control',
  payload: {
    command: 'StartRecognition',
    contextText: 'гипертония\nдиабет\nишемическая болезнь сердца',
    boosting_table_id: 'medical_table',
    combineMode: false
  }
};
```

Здесь есть деталь, которую нужно отметить: фронтенд передаёт многострочный текст (разделённый символами новой строки), бэкенд должен его разобрать. WebSocket Handler бэкенда разбирает эти параметры и передаёт их SDK:

```csharp
private async Task HandleControlMessageAsync(
    string connectionId,
    DoubaoSession session,
    ControlMessage message)
{
    if (message.Payload is SessionControlRequest controlRequest)
    {
        // Разбор параметров горячих слов
        string? contextText = controlRequest.ContextText;
        string? boostingTableId = controlRequest.BoostingTableId;
        bool? combineMode = controlRequest.CombineMode;

        // Разбор многострочного текста в список горячих слов
        if (!string.IsNullOrEmpty(contextText))
        {
            var hotwords = contextText
                .Split('\n', StringSplitOptions.RemoveEmptyEntries)
                .Select(s => s.Trim())
                .Where(s => s.Length > 0)
                .ToList();

            session.HotwordContexts = hotwords;
        }

        session.BoostingTableId = boostingTableId;
    }
}
```

С помощью такого дизайна передача конфигурации горячих слов с фронтенда на бэкенд становится понятной и эффективной. На самом деле ничего особенного, просто передаём слой за слоем.

## Практическое руководство

### Настройка пользовательских горячих слов

В реальном использовании настройка пользовательских горячих слов очень проста. Откройте страницу настроек распознавания речи HagiCode, найдите область "Конфигурация горячих слов". В поле ввода "Текст пользовательских горячих слов" вводите по одной фразе горячего слова на строку.

Например, если вы разрабатываете медицинское приложение, можно настроить так:

```
гипертония
диабет
ишемическая болезнь сердца
стенокардия
инфаркт миокарда
сердечная недостаточность
```

После сохранения конфигурации при каждом начале распознавания речи эти горячие слова будут автоматически передаваться в API Doubao. Фактические тесты показывают, что после настройки горячих слов точность распознавания соответствующих профессиональных терминов значительно повысилась. Как сказать, эффект есть, по крайней мере лучше, чем раньше.

### Настройка таблицы горячих слов платформы

Если вам нужно управлять большим количеством горячих слов или горячие слова требуют частого обновления, то режим таблицы горячих слов платформы вам больше подходит. Сначала нужно создать таблицу горячих слов на платформе самобучения Doubao, получить сгенерированный boosting_table_id, а затем ввести этот ID на странице настроек HagiCode.

Платформа самобучения Doubao предоставляет такие функции, как массовый импорт горячих слов, классификационное управление и т. д., что очень практично для команд, которым нужно управлять большим количеством профессиональной терминологии. Управление горячими словами через платформу позволяет реализовать централизованное обслуживание и единое обновление горячих слов. Ведь когда горячих слов становится много, иметь место для централизованного управления определённо лучше, чем каждый раз вручную вводить.

### Использование комбинированного режима

В некоторых сложных сценариях вам может понадобиться использовать одновременно пользовательские горячие слова и таблицу горячих слов платформы. В этом случае просто настройте оба типа горячих слов в HagiCode и включите переключатель "Комбинированный режим".

В комбинированном режиме API Doubao одновременно учитывает оба источника горячих слов, точность распознавания обычно выше, чем при отдельном использовании любого из них. Однако нужно отметить, что комбинированный режим увеличивает сложность запроса, рекомендуется решить, включать ли его, после фактического тестирования. Ведь увеличение сложности — действительно ли это стоит, нужно смотреть на фактический эффект.

### Примеры интеграции кода

Интеграция функции горячих слов в проект HagiCode очень проста. Ниже приведены некоторые часто используемые фрагменты кода:

```typescript
import {
  loadHotwordConfig,
  saveHotwordConfig,
  validateHotwordConfig,
  parseContextText,
  getEffectiveHotwordMode,
  type HotwordConfig
} from '@/types/hotword';

// Загрузка и проверка конфигурации
const config = loadHotwordConfig();
const validation = validateHotwordConfig(config);

if (!validation.isValid) {
  console.error('Проверка конфигурации горячих слов не удалась:', validation.errors);
  return;
}

// Разбор текста горячих слов
const hotwords = parseContextText(config.contextText);
console.log('Распознанные горячие слова:', hotwords);

// Получение эффективного режима горячих слов
const mode = getEffectiveHotwordMode(config);
console.log('Текущий режим горячих слов:', mode);
```

## Меры предосторожности

При реализации и использовании функции горячих слов есть несколько моментов, которые требуют особого внимания.

Во-первых, ограничения символов. API Doubao имеет строгие ограничения на горячие слова, включая количество строк, количество символов в каждой строке, общее количество символов и т. д. Если превышены ограничения, API вернёт ошибку. Во фронтенд-реализации HagiCode мы проверяем на этапе ввода пользователя с помощью функций проверки, избегая отправки недействительной конфигурации на бэкенд. Ведь ранняя обнаружение проблемы определённо лучше, чем ждать ошибки API.

Во-вторых, формат boosting_table_id. Это поле допускает только буквы, цифры, подчёркивания и дефисы, не допускает пробелов или других специальных символов. При создании таблицы горячих слов на платформе самобучения Doubao нужно обратить внимание на соглашения об именовании. Это неудивительно, требования API к формату всегда более строгие.

В-третьих, обратная совместимость. Параметры горячих слов полностью необязательны, при отсутствии настройки горячих слов система работает так же, как раньше. Этот дизайн гарантирует, что существующие пользователи не будут затронуты, а также облегчает постепенную миграцию и обновление. Ведь нельзя добавлять функцию и ломать предыдущую логику.

Наконец, обработка ошибок. Когда конфигурация горячих слов недействительна, API Doubao возвращает соответствующую информацию об ошибке. Реализация HagiCode записывает подробные журналы, что помогает разработчикам устранять неполадки. В то же время фронтенд также отображает ошибки проверки в интерфейсе, помогая пользователям исправить конфигурацию. Хорошая обработка ошибок естественно приводит к хорошему пользовательскому опыту.
