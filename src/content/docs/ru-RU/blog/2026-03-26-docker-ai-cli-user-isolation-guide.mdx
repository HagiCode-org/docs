## Запуск AI CLI инструментов в Docker контейнерах: Практическое руководство по изоляции пользователей и постоянным томам

> Интеграция инструментов AI программирования таких как Claude Code, Codex, OpenCode в контейнеризированной среде кажется простой, но на деле скрывает множество нюансов. В этой статье мы подробно разберём, как проект HagiCode в Docker-развертывании решает ключевые задачи с правами пользователей, персистентностью конфигурации, управлением версиями и другими фундаментальными вызовами, помогая вам избежать распространённых ошибок.

## Предпосылки

Когда мы решили запускать AI CLI инструменты программирования внутри Docker контейнеров, самой интуитивной мыслью могло быть: "Контейнеры же работают под root, просто установи и всё?" На самом деле, эта мысль кажется простой, но за ней скрываются несколько фундаментальных проблем, которые необходимо решить.

Во-первых, **ограничения безопасности — первое препятствие**. Например, Claude CLI явно запрещает запуск от имени пользователя root — это обязательная проверка безопасности, при обнаружении root он отказывается запускаться. Можно подумать: "А что, если я переключусь с помощью инструкции `USER`?" Всё не так просто, между пользователями контейнера и правами пользователей хост-машины существуют проблемы отображения. Ведь в этом мире, что может быть таким простым?

Во-вторых, **персистентность состояния — вторая ловушка**. Claude Code требует входа в систему, Codex имеет собственную конфигурацию, OpenCode также имеет каталог кэша. Если при каждой перезагрузке контейнера заново настраивать, то эта "автоматизация" теряет всякий смысл. Нам нужно сделать так, чтобы эти конфигурации сохранялись за пределами жизненного цикла контейнера. Конфигурация — это как память: исчезнет — и это довольно удручает.

Третья проблема — **согласованность прав доступа**. Может ли процесс внутри контейнера получить доступ к файлам конфигурации, созданным пользователем хост-машины? Несовпадение UID/GID приведёт к ошибкам прав доступа к файлам, что очень часто встречается в реальном развертывании. Эта проблема довольно раздражает, но ничего не поделаешь.

Эти проблемы кажутся независимыми, но на самом деле тесно взаимосвязаны. В процессе разработки проекта HagiCode мы постепенно нашли работоспособное решение, далее я подробно поделюсь техническими деталями и опытом преодоления проблем.

## Почему нельзя просто использовать root?

Здесь есть распространённое заблуждение: Docker контейнеры по умолчанию работают под root, так что можно просто установить инструменты под root. Если так подумать, Claude CLI безжалостно преподнесёт вам урок.

```bash
# Запуск Claude CLI напрямую от root? Нет
docker run --rm -it --user root myimage claude
# Вывод: Error: This command cannot be run as root user
```

Это жёсткое ограничение безопасности Claude CLI. Причина проста: эти CLI инструменты читают и записывают чувствительную конфигурацию пользователей, включая API Token, локальный кэш и даже могут выполнять скрипты, написанные пользователями. Запуск этих инструментов с правами root несёт слишком большие потенциальные риски. Ведь к безопасности нельзя относиться слишком осторожно.

Тогда возникает вопрос: как удовлетворить требованиям безопасности CLI и одновременно сохранить гибкость управления контейнерами? Нам нужно сменить угол зрения — **не переключать пользователя во время выполнения, а создавать выделенного пользователя уже на этапе сборки образа**. Иногда, если посмотреть на проблему под другим углом, ответ сам собой emerges.

## Создание выделенного пользователя: не просто смена имени

Можно подумать: "Добавлю строку с инструкцией `USER` в Dockerfile — и всё?" Это действительно самое простое решение, но оно недостаточно надёжно. Простые вещи часто недостаточно элегантны, не так ли?

### Статическое создание vs динамическое отображение

Решение HagiCode — создать пользователя `hagicode` с UID 1000, этот UID обычно соответствует большинству пользователей хост-машины по умолчанию:

```dockerfile
RUN groupadd -o -g 1000 hagicode && \
    useradd -o -u 1000 -g 1000 -s /bin/bash -m hagicode && \
    mkdir -p /home/hagicode/.claude && \
    chown -R hagicode:hagicode /home/hagicode
```

Но это решает только проблему встроенного в образ пользователя. А если пользователь хост-машины имеет UID 1001? При запуске контейнера нужно поддерживать динамическое отображение.

Ключевая логика в docker-entrypoint.sh:

```bash
if [ -n "$PUID" ] && [ -n "$PGID" ]; then
    if ! id hagicode >/dev/null 2>&1; then
        groupadd -g "$PGID" hagicode
        useradd -u "$PUID" -g "$PGID" -s /bin/bash -m hagicode
    fi
fi
```

Преимущество такой конструкции: **при сборке образа используется UID по умолчанию 1000, во время выполнения можно динамически настраивать через переменные окружения PUID/PGID**. Какой бы ни был UID пользователя хост-машины, владение файлами конфигурации не будет проблематичным. Такая конструкция вполне естественна, ведь нужно найти баланс между гибкостью и значениями по умолчанию.

## Философия дизайна постоянных томов

Каждый AI CLI инструмент имеет свои предпочитаемые каталоги конфигурации, что требует точного соответствия:

| CLI инструмент | Путь внутри контейнера | Именованный том |
|----------------|------------------------|-----------------|
| Claude | `/home/hagicode/.claude` | `claude-data` |
| Codex | `/home/hagicode/.codex` | `codex-data` |
| OpenCode | `/home/hagicode/.config/opencode` | `opencode-config-data` |

Почему используются именованные тома вместо привязочных монтирований? Три причины:

1. **Упрощение управления**: именованные тома автоматически управляются Docker, нет необходимости вручную создавать каталоги на хост-машине
2. **Изоляция прав**: начальное содержимое тома создаётся пользователем внутри контейнера, что избегает конфликтов прав хост-машины
3. **Независимая миграция**: тома могут существовать независимо от контейнеров, при обновлении образа данные не теряются

docker-compose-builder-web автоматически генерирует соответствующую конфигурацию томов:

```yaml
volumes:
  claude-data:
  codex-data:
  opencode-config-data:

services:
  hagicode:
    volumes:
      - claude-data:/home/hagicode/.claude
      - codex-data:/home/hagicode/.codex
      - opencode-config-data:/home/hagicode/.config/opencode
    user: "${PUID:-1000}:${PGID:-1000}"
```

Обратите внимание на поле `user`: через переменные окружения внедряются PUID/PGID, обеспечивая запуск процессов контейнера от имени пользователя, соответствующего хост-машине. Эта деталь довольно важна, ведь проблемы с правами, если возникают, отслеживать довольно раздражает.

## Управление версиями: фиксированные версии и переопределение во время выполнения

Фиксация версий Docker образа — ключ к гарантируемой воспроизводимости. Но в реальной разработке часто нужно протестировать новые версии или экстренно исправить ошибку. Если каждый раз пересобирать образ, эффективность слишком низкая.

Стратегия HagiCode — **фиксированные версии как значения по умолчанию, переопределение во время выполнения как расширенная возможность**. Это своего рода компромисс в инженерной практике, между стабильностью и гибкостью всегда нужно делать выбор.

Фиксированные версии в Dockerfile.template:

```dockerfile
USER hagicode
WORKDIR /home/hagicode

# Конфигурация пути глобальной установки npm
RUN mkdir -p /home/hagicode/.npm-global && \
    npm config set prefix '/home/hagicode/.npm-global'

# Установка CLI инструментов (с фиксированными версиями)
RUN npm install -g @anthropic-ai/claude-code@2.1.71 && \
    npm install -g @openai/codex@0.112.0 && \
    npm install -g opencode-ai@1.2.25 && \
    npm cache clean --force
```

Поддержка переопределения во время выполнения в docker-entrypoint.sh:

```bash
install_cli_override_if_needed() {
    local package_name="$2"
    local override_version="$5"

    if [ -n "$override_version" ]; then
        gosu hagicode npm install -g "${package_name}@${override_version}"
    fi
}

# Пример использования
install_cli_override_if_needed "" "@anthropic-ai/claude-code" "" "" "${CLAUDE_CODE_CLI_VERSION}"
```

Таким образом, без пересборки образа можно протестировать новые версии через переменные окружения:

```bash
docker run -e CLAUDE_CODE_CLI_VERSION=2.2.0 myimage
```

Эта конструкция довольно практична, ведь кто захочет каждый раз при тестировании новых функций пересобирать образ?

## Автоматическое внедрение конфигурации

Кроме ручной конфигурации CLI инструментов, в некоторых сценариях нужно автоматически внедрять конфигурацию. Самый типичный случай — это API Token.

```bash
if [ -n "$ANTHROPIC_AUTH_TOKEN" ]; then
    mkdir -p /home/hagicode/.claude
    cat > /home/hagicode/.claude/settings.json <<EOF
{
  "env": {
    "ANTHROPIC_AUTH_TOKEN": "${ANTHROPIC_AUTH_TOKEN}"
  }
}
EOF
    chown -R hagicode:hagicode /home/hagicode/.claude
fi
```

Здесь нужно обратить внимание на два момента: **чувствительная информация передаётся через переменные окружения, не прописывается в образ**; права владения файлом конфигурации должны быть установлены правильно, иначе CLI инструмент не сможет прочитать. Это довольно базовая вещь, но довольно много людей делают неправильно.

## Лучшие практики и руководство по избежанию проблем

### Проблема несоответствия прав доступа

Это самая распространённая ловушка. UID пользователя хост-машины 1001, внутри контейнера 1000, созданные файлы не могут быть взаимно доступны.

```bash
# Правильный подход: позволить контейнеру соответствовать пользователю хост-машины
docker run \
    -e PUID=$(id -u) \
    -e PGID=$(id -g) \
    myimage
```

Эта проблема довольно распространённа, но при первой встрече довольно удручает.

### Потеря конфигурации после перезагрузки контейнера

Если вы обнаружите, что при каждой перезагрузке нужно снова входить, проверьте, не забыли ли примонтировать персистентные тома:

```yaml
volumes:
  - claude-data:/home/hagicode/.claude
```

Конфигурация — такая вещь: с трудом настроишь, исчезнет — и чувство, как бы это сказать, довольно угнетающее.

### Правильная стратегия обновления версии

Не выполняйте `npm install -g` напрямую в работающем контейнере. Правильный подход:

1. Установить переменную окружения для запуска переопределяющей установки
2. Или пересобрать образ

```bash
# Способ 1: переопределение во время выполнения
docker run -e CLAUDE_CODE_CLI_VERSION=2.2.0 myimage

# Способ 2: пересборка
docker build -t myimage:v2 .
```

Все дороги ведут в Рим, только некоторые дороги проще, некоторые немного извилистее.

### Чек-лист усиления безопасности

- API Token передаётся через переменные окружения, не записывается в образ
- Файлы конфигурации с правами 600
- Всегда запускать приложения от имени не-root пользователя
- Регулярно обновлять версии CLI, исправлять уязвимости безопасности

Безопасность — вещь важная, но сколько людей действительно могут хорошо внедрить её на практике?

## Расширение на новые CLI инструменты

Если в будущем нужно будет поддерживать новые CLI инструменты, потребуется только три шага:

1. **Dockerfile.template**: добавить шаги установки
2. **docker-entrypoint.sh**: добавить логику переопределения версии
3. **docker-compose-builder-web**: добавить отображение персистентного тома

Шаблонный дизайн делает расширение простым, не нужно менять основную логику. Это тоже своего рода опыт прошедшего путь, не великие принципы, просто пережитые проблемы.

## Заключение

Запуск AI CLI инструментов в Docker контейнерах, ключевые проблемы лежат в трёх измерениях: **права пользователей, персистентность конфигурации, управление версиями**. Проект HagiCode через комбинацию решений с созданием выделенного пользователя, изоляцией именованных томов, переопределением переменных окружения реализовал архитектуру развертывания, безопасную и гибкую.

Ключевые моменты дизайна:

- **Изоляция пользователей**: создание выделенного пользователя на этапе сборки образа, поддержка динамического отображения PUID/PGID во время выполнения
- **Стратегия персистентности**: каждый CLI инструмент соответствует независимому именованному тому, перезагрузка контейнера не влияет на конфигурацию
- **Гибкость версий**: фиксированные значения по умолчанию обеспечивают воспроизводимость, переопределение во время выполнения предоставляет возможность тестирования
- **Автоматическая конфигурация**: поддержка автоматического внедрения чувствительной конфигурации через переменные окружения

Это решение стабильно работает в проекте HagiCode уже некоторое время, надеюсь, оно будет полезно разработчикам с аналогичными потребностями. На самом деле не так уж и сложно, просто инженерная практика.
