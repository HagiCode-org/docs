## Design.md: решение для обеспечения согласованного проектирования пользовательского интерфейса с помощью ИИ

> В эпоху ИИ-поддерживаемой фронтенд-разработки, как обеспечить согласованность пользовательского интерфейса, генерируемого ИИ? В этой статье мы делимся нашим практическим опытом создания сайта галереи дизайна на основе awesome-design-md, а также тем, как создавать структурированные файлы design.md для управления ИИ в создании стандартизированного пользовательского интерфейса.

## Предыстория

Те, кто использовал ИИ для написания фронтенд-кода, вероятно, сталкивались с похожим опытом: одна и та же страница, сгенерированная ИИ несколько раз, каждый раз имеет разный стиль. Иногда это скругленные углы, иногда прямые, иногда отступ 8px, а иногда 16px, даже одна и та же кнопка в разных диалогах выглядит по-разному.

Это не единичный случай. С распространением ИИ-поддерживаемой разработки **отсутствие согласованности в пользовательском интерфейсе, генерируемом ИИ**, стало повсеместной проблемой. Разные ИИ-помощники, разные подсказки, даже один и тот же помощник в разных диалогах создают совершенно разные по стилю интерфейсы. Это приносит огромные расходы на обслуживание при итерации продукта.

Корень проблемы на самом деле прост: отсутствие **авторитетного справочного документа по дизайну**. Традиционные файлы стилей CSS могут только сказать разработчикам "как реализовать", но не могут полностью передать "почему так спроектировано" и "в каких сценариях использовать какой шаблон дизайна". А для ИИ ему нужно более четкое структурированное описание для понимания спецификаций дизайна.

В то же время в сообществе открытого исходного кода уже есть некоторые хорошие ресурсы. Проект [VoltAgent/awesome-design-md](https://github.com/VoltAgent/awesome-design-md) собрал большую документацию по системам дизайна известных компаний, каждый каталог содержит README.md, DESIGN.md и предварительный просмотр HTML. Но всё это разбросано по вышестоящим репозиториям, трудно быстро просматривать и сравнивать.

А можно ли объединить эти ресурсы и сделать удобную для просмотра галерею дизайна, а также沉淀ить структурированный design.md для использования ИИ?

Ответ положительный. Далее поделюсь нашим решением.

## Анализ

Прежде чем писать код, давайте разберем несколько технических задач этой проблемы.

### Управление источниками контента: как объединить разбросанные ресурсы дизайна?

Вышестоящий репозиторий awesome-design-md содержит большую документацию по дизайну, но нам нужен способ включить её в наш проект.

**Решение: использовать git submodule**

```
awesome-design-md-site
└── vendor/awesome-design-md          # Вышестоящие ресурсы (git submodule)
```

Это имеет несколько преимуществ:
- Контролируемая версия: можно зафиксировать конкретную вышестоящую версию
- Офлайн-сборка: не нужно запрашивать внешние API при сборке
- Проверка контента: можно увидеть конкретные изменения в PR

### Стандартизация данных: как унифицировать разные структуры документов?

Структуры документов дизайна разных компаний могут отличаться, некоторые отсутствуют файлы предварительного просмотра, некоторые именования не согласованы. Нам нужно выполнить стандартизацию при сборке.

**Решение: сканирование при сборке и генерация стандартизированных пунктов**

Основной модуль - `awesomeDesignCatalog.ts`, отвечающий за:
1. Сканирование каталога `vendor/awesome-design-md/design-md/*`
2. Проверка каждого пункта на наличие обязательных файлов (README.md, DESIGN.md, хотя бы один файл предварительного просмотра)
3. Извлечение и рендеринг содержимого Markdown в HTML
4. Генерация стандартизированных данных пунктов

### Архитектура статического сайта: как предоставить динамический поиск, сохраняя статическое развертывание?

Поскольку это галерея дизайна, функция поиска обязательна. Но Astro - это генератор статических сайтов, как реализовать поиск в реальном времени?

**Решение: React island + синхронизация параметров запроса URL**

```typescript
// src/components/gallery/SearchToolbar.tsx

export function SearchToolbar() {
  const [query, setQuery] = useState('');

  // Синхронизация URL
  useEffect(() => {
    const params = new URLSearchParams(window.location.search);
    setQuery(params.get('q') || '');
  }, []);

  const filtered = entries.filter(entry =>

  return <input value={query} onChange={e => {
    setQuery(e.target.value);
    updateURL(e.target.value);
  }} />;
}
```

Преимущество этого подхода в том, что он сохраняет возможность развертывания статического сайта (можно развернуть на любом хостинге статических файлов), обеспечивая при этом пользовательский опыт мгновенной фильтрации.

### Документирование дизайна: как заставить ИИ понимать и соблюдать спецификации дизайна?

Это ядро всего решения. Нам нужно создать структурированный design.md, чтобы ИИ мог понимать и применять наши спецификации дизайна.

**Решение: заимствовать структуру DESIGN.md от ClickHouse**

[DESIGN.md от ClickHouse](https://github.com/ClickHouse/ClickHouse/blob/master/DESIGN.md) - хороший справочник, он содержит:
- Visual Theme & Atmosphere
- Color Palette & Roles
- Typography Rules
- Component Stylings
- Layout Principles
- Depth & Elevation
- Do's and Don'ts
- Responsive Behavior
- Agent Prompt Guide

Наш подход: **структура как справочник, содержимое переписано**. Сохраняем структуру разделов DESIGN.md от ClickHouse, но заменяем содержимое на фактически используемые в нашем проекте дизайн-токены и спецификации компонентов.

## Решение

На основе вышеуказанного анализа наше решение содержит четыре основных модуля.

### 1. Конвейер получения контента

Это основа всей системы, отвечающая за извлечение и стандартизацию контента из вышестоящих ресурсов.

```typescript
// src/lib/content/awesomeDesignCatalog.ts

export async function scanSourceEntries(): Promise<DesignEntry[]> {
  const designDir = 'vendor/awesome-design-md/design-md';
  const entries: DesignEntry[] = [];

  for (const dir of await fs.readdir(designDir)) {
    const entryPath = path.join(designDir, dir);
    if (await isValidDesignEntry(entryPath)) {
      const entry = await normalizeDesignEntry(entryPath);
      entries.push(entry);
    }
  }

  return entries;
}

async function isValidDesignEntry(dir: string): Promise<boolean> {
  const requiredFiles = ['README.md', 'DESIGN.md'];
  for (const file of requiredFiles) {
    if (!(await fileExists(path.join(dir, file)))) {
      return false;
    }
  }
  return true;
}
```

### 2. Интерфейс галереи

Интерфейс галереи включает три основные части:

**Главная страница**: отображает сетку карточек всех пунктов дизайна, каждая карточка содержит:
- Название и резюме пункта дизайна
- Изображение предварительного просмотра (если есть)
- Быстрое выделение поиска

**Страница деталей**: агрегированное отображение полной информации одного пункта дизайна:
- Документация README
- Документация DESIGN
- Предварительный просмотр (поддержка переключения светлой/темной темы)
- Навигация по соседним пунктам

**Навигация**: поддержка возврата в галерею, просмотр соседних пунктов

Главная страница галереи использует плотную карточную компоновку, раскладывая пункты design.md из разных источников в единой визуальной структуре, что удобно для быстрого сравнения стилей бренда, паттернов кнопок и типографического ритма.

После входа в конкретный пункт, страница деталей помещает резюме дизайна и предварительный просмотр в реальном времени на одной странице, снижая затраты на переключение между документацией, предварительным просмотром и исходным кодом.

### 3. Функция поиска

Функция поиска основана на клиентской фильтрации, используя параметры запроса URL для сохранения состояния:

```typescript
// src/components/gallery/SearchToolbar.tsx

function SearchToolbar({ entries }: { entries: DesignEntry[] }) {
  const [query, setQuery] = useState('');
  const [results, setResults] = useState(entries);

  useEffect(() => {
    const params = new URLSearchParams(window.location.search);
    const initialQuery = params.get('q') || '';
    setQuery(initialQuery);
    filterEntries(initialQuery);
  }, []);

  const filterEntries = (searchQuery: string) => {
    const filtered = entries.filter(entry =>
      entry.searchText.toLowerCase().includes(searchQuery.toLowerCase())
    );
    setResults(filtered);
  };

  const handleChange = (e: React.ChangeEvent<HTMLInputElement>) => {
    const value = e.target.value;
    setQuery(value);
    filterEntries(value);

    // Обновление URL (без обновления страницы)
    const newUrl = value
      ? `${window.location.pathname}?q=${encodeURIComponent(value)}`
      : window.location.pathname;
    window.history.replaceState({}, '', newUrl);
  };

  return (
    <div className="search-toolbar">
      <input
        type="text"
        value={query}
        onChange={handleChange}
        placeholder="Поиск пунктов дизайна..."
      />
      <span className="result-count">{results.length} результатов</span>
    </div>
  );
}
```

## Практика

Поняв решение, как конкретно реализовать?

### Шаги реализации

**Первый шаг: инициализация подмодуля**

```bash
# Добавление вышестоящего репозитория как подмодуля
git submodule add https://github.com/VoltAgent/awesome-design-md.git vendor/awesome-design-md

# Инициализация и обновление подмодуля
git submodule update --init --recursive
```

**Второй шаг: создание конвейера контента**

Реализация `awesomeDesignCatalog.ts`, включая:
- Логику сканирования и проверки файлов
- Рендеринг Markdown (используя встроенный рендерер Astro)
- Извлечение данных пунктов

**Третий шаг: создание UI галереи**

Использование Astro + React Islands для создания:
- Макет главной страницы галереи (сетка карточек)
- Компонент карточки дизайна
- Панель инструментов поиска
- Макет страницы деталей

**Четвертый шаг: написание документации по дизайну**

На основе структуры DESIGN.md от ClickHouse заполнить фактически используемые дизайн-токены вашего проекта. Обновите README.md, добавьте ссылку на design.md.

### Меры предосторожности

**Безопасность**: при рендеринге Markdown нужно фильтровать небезопасный HTML. Встроенный рендерер Astro по умолчанию фильтрует теги script, но всё равно нужно обращать внимание на риски XSS.

**Производительность**: большое количество iframe предварительного просмотра может повлиять на начальную загрузку. Рекомендуется использовать `loading="lazy"` для отложенной загрузки контента предварительного просмотра.

**Обслуживаемость**: design.md должен быть синхронизирован с реализацией кода. Рекомендуется добавить проверки в CI, чтобы убедиться, что CSS переменные согласованы в документации и коде.

**Доступность**: убедитесь, что контрастность цветов соответствует стандарту WCAG AA (минимум 4.5:1).

### Руководство по использованию ИИ

После создания design.md, как заставить ИИ действительно использовать его? Вот несколько полезных советов:

**Совет 1: явно указать в подсказке**

```
Пожалуйста, обратитесь к файлу design.md в корневом каталоге проекта, используйте определенные спецификации дизайна для реализации следующих компонентов:
- Кнопка: используйте primary оттенок, скругленные углы 8px
- Карточка: используйте уровень тени elevation-2
```

**Совет 2: потребовать от ИИ ссылаться на конкретные CSS переменные**

```
Реализуйте навигационную панель, требуя:
- Цвет фона использовать --color-bg-primary
- Граница использовать --color-border-subtle
- Текст использовать --text-color-primary
```

**Совет 3: включить содержимое design.md в системную подсказку**

Если ваш инструмент ИИ поддерживает пользовательские системные подсказки, вы можете напрямую добавить основное содержимое design.md.

### Стратегия тестирования

**Тестирование конвейера контента**:
- Сценарии отсутствия файлов (отсутствие README.md или DESIGN.md)
- Сценарии ошибок формата (сбой анализа Markdown)
- Сценарии пустых каталогов

**Тестирование функции поиска**:
- Обработка пустых результатов
- Специальные символы (например, китайские иероглифы, эмодзи)
- Проверка синхронизации URL

**Тестирование UI компонентов**:
- Переключение светлой/темной темы
- Адаптивный макет
- Состояние загрузки предварительного просмотра

## Резюме

Проблема несогласованности пользовательского интерфейса, генерируемого ИИ, с которой столкнулся HagiCode в процессе разработки, по сути заключается в отсутствии структурированного справочного документа по дизайну. Создав сайт галереи дизайна и стандартизированный design.md, мы успешно решили эту проблему.

Основная ценность этого решения:
- **Унификация ресурсов**: объединение разрозненной документации по системам дизайна
- **Структурированные спецификации**: представление спецификаций дизайна в форме, понятной для ИИ
- **Постоянное обслуживание**: сохранение актуальности контента через git submodule

Если вы также используете ИИ для поддержки фронтенд-разработки, рекомендуйте попробовать это решение. Создание структурированного design.md не только повысит согласованность кода, генерируемого ИИ, но и поможет команде поддерживать единство спецификаций дизайна.
