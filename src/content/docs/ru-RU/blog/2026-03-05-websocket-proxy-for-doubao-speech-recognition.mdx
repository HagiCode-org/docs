## Решение проблем аутентификации WebSocket в браузере: практический паттерн прокси для распознавания речи Doubao

> API браузерного WebSocket не поддерживает пользовательские HTTP-заголовки, что создаёт проблему для служб распознавания речи, требующих аутентификационных данных в заголовках. В этой статье мы делимся тем, как решили эту проблему в проекте HagiCode с помощью паттерна серверного прокси, и как подход развивался от экспериментов до промышленного использования.


## Предпосылки

Когда мы начали создавать распознавание речи для проекта HagiCode, мы уверенно выбрали службу распознавания речи Doubao от ByteDance. Исходный дизайн был простым: позволить фронтенду напрямую подключаться к WebSocket-службе Doubao. Как сложно это может быть? Просто открыть соединение и отправить данные, верно?

Затем последовал сюрприз: API Doubao требует передачи аутентификационной информации через HTTP-заголовки, включая такие параметры, как `accessToken` и `secretKey`. Это сразу стало неудобным, поскольку браузерный WebSocket API просто не поддерживает установку пользовательских заголовков.

Что же делать, когда браузер не позволяет их отправить?

На тот момент мы взвесили два варианта:

1. Поместить учётные данные в параметры URL-запроса — просто и грубо
2. Добавить слой прокси на бэкенде — на первый взгляд больше работы

Первый вариант напрямую раскрывает учётные данные в коде фронтенда и локальном хранилище. Насколько это безопасно? Меня это не устраивало. Кроме того, некоторые API требуют заголовочной проверки, поэтому этот подход даже нецелесообразен.

В конечном счёте мы выбрали второй вариант: реализовать WebSocket-прокси на бэкенде. Как ни странно, этот паттерн был сначала проверен в нашей экспериментальной среде, и только после подтверждения его стабильности мы перенесли его в продакшен. Ведь никто не хочет, чтобы продакшен служил лабораторией для экспериментов.

## О HagiCode

HagiCode — это проект AI-ассистента программирования с поддержкой голосового взаимодействия. Поскольку нам нужно было вызывать службу распознавания речи с фронтенда, мы непосредственно столкнулись с этой проблемой аутентификации WebSocket, что привело нас к решению, описанному здесь. Иногда такие технические препятствия вызывают разочарование, но они также заставляют изучать паттерны, которые впоследствии оказываются полезными.

## Анализ технических вызовов

### Ограничения браузерного WebSocket

Стандартный WebSocket API выглядит замечательно просто:

```javascript
const ws = new WebSocket('wss://example.com/ws');
```

Но именно в этой простоте кроется проблема — он передаёт параметры только через URL и не может устанавливать заголовки так, как это делает HTTP-запрос:

```javascript
// This is not supported in the WebSocket API
const ws = new WebSocket('wss://example.com/ws', {
  headers: {
    'Authorization': 'Bearer token'
  }
});
```

И это основная проблема. Для таких служб, как распознавание речи Doubao, зависящих от заголовочной аутентификации, это ограничение является непреодолимым препятствием.

Приняв это ограничение, архитектура должна измениться.

### Решения по архитектурному дизайну

При проектировании решения мы тщательно сравнили компромиссы.

**Решение 1: выбор паттерна прокси**

Мы сравнили два подхода:

| Вариант | Преимущества | Недостатки | Решение |
|------|------|------|------|
| Нативный WebSocket | Лёгкий, простой, прямая пересылка | Управление подключениями нужно обрабатывать вручную | Выбран |
| SignalR | Автоматическое переподключение, строгая типизация | Избыточная сложность, дополнительные зависимости | Отклонён |

В конечном счёте мы выбрали нативный WebSocket. Честно говоря, это был самый лёгкий вариант и лучше подходил для простой двусторонней пересылки двоичных потоков. Использование SignalR показалось бы избыточным для этого сценария, и это могло бы добавить дополнительную задержку.

**Решение 2: стратегия управления подключениями**

Мы приняли модель «одно подключение — одна сессия» — каждое фронтенд-подключение WebSocket отображается на собственное независимое подключение к бэкенду Doubao.

Преимущества очевидны:
- Прост в реализации и соответствует стандартному паттерну использования
- Проще отлаживать и устранять неполадки
- Хорошая изоляция ресурсов, предотвращающая взаимное влияние сессий

Простыми словами: прямое решение иногда является лучшим. Сложность не делает дизайн автоматически лучше.

**Решение 3: хранение аутентификационных данных**

Учётные данные хранятся в файлах конфигурации бэкенда (`appsettings.yml` или переменных окружения) и загружаются через внедрение зависимостей:

- Простая модель конфигурации, соответствующая существующим соглашениям бэкенда
- Конфиденциальные данные никогда не попадают на фронтенд
- Поддерживает многосредовую настройку для разработки, тестирования и продакшена

Такой уровень разделения важен. Никто не хочет, чтобы учётные данные попадали туда, где им не место.

### Дизайн потока данных

Общий поток данных выглядит следующим образом:

```
Frontend (browser)
  │
  │ ws://backend/api/voice/ws
  │ WebSocket (binary)
  ▼
Backend (proxy)
  │
  │ wss://openspeech.bytedance.com/
  │ (with authentication headers)
  ▼
Doubao API
```

Сам поток несложен:
1. Фронтенд подключается к бэкенд-прокси через WebSocket
2. Бэкенд-прокси получает аудиоданные и подключается к API Doubao с аутентифицированными заголовками
3. API Doubao возвращает результаты распознавания, а прокси пересылает их на фронтенд
4. Весь процесс остаётся полностью асинхронным с двусторонней потоковой передачей

Когда обязанности чётко разделены, дизайн становится вполне естественным.

## Реализация основных компонентов

### 1. Конфигурация WebSocket-конечной точки

```csharp
app.Map("/ws", async context =>
{
    if (context.WebSockets.IsWebSocketRequest)
    {
        // Read configuration from query parameters
        var appId = context.Request.Query["appId"];
        var accessToken = context.Request.Query["accessToken"];

        // Validate required parameters
        if (string.IsNullOrEmpty(appId) || string.IsNullOrEmpty(accessToken))
        {
            context.Response.StatusCode = 400;
            return;
        }

        // Accept the WebSocket connection
        using var webSocket = await context.WebSockets.AcceptWebSocketAsync();

        // Message handling loop
        var buffer = new byte[4096];
        while (!webSocket.CloseStatus.HasValue)
        {
            var result = await webSocket.ReceiveAsync(buffer, CancellationToken.None);

            if (result.MessageType == WebSocketMessageType.Close)
            {
                await webSocket.CloseAsync(
                    result.CloseStatus.Value,
                    result.CloseStatusDescription,
                    CancellationToken.None);
                break;
            }

            // Process audio data
            await HandleAudioDataAsync(buffer, result.Count);
        }
    }
});
```

### 2. Управление сессиями

```csharp
public class DoubaoSessionManager : IDoubaoSessionManager
{
    private readonly ConcurrentDictionary<string, DoubaoSession> _sessions = new();

    public DoubaoSession CreateSession(string connectionId)
    {
        var session = new DoubaoSession(connectionId);
        _sessions[connectionId] = session;
        return session;
    }

    public async Task SendAudioAsync(string connectionId, byte[] audioData)
    {
        if (_sessions.TryGetValue(connectionId, out var session))
        {
            await session.SendAudioAsync(audioData);
        }
    }

    public void RemoveSession(string connectionId)
    {
        if (_sessions.TryRemove(connectionId, out var session))
        {
            session.Dispose();
        }
    }
}
```

Использование `ConcurrentDictionary` для управления сессиями означает, что потоковая безопасность в значительной степени обеспечивается. Каждое входящее подключение получает свою сессию, а очистка происходит автоматически при отключении.

## Дизайн протокола сообщений

Фронтенд и бэкенд используют текстовые JSON-сообщения для управления, а двоичные сообщения передают аудиоданные.

Пример управляющего сообщения:

```json
{
    "type": "control",
    "messageId": "msg_123",
    "timestamp": "2026-03-03T10:00:00Z",
    "payload": {
        "command": "StartRecognition",
        "parameters": {
            "hotwordId": "hotword1",
            "boosting_table_id": "table123"

    }

```

Пример результата распознавания:

```json
{
    "type": "result",
    "timestamp": "2026-03-03T10:00:03Z",
    "payload": {
        "text": "Hello world",
        "confidence": 0.95,
        "duration": 1500,
        "isFinal": true,

            {
                "text": "Hello",
```

Этот дизайн разделяет управляющие сигналы и аудиопосылки, что делает систему более понятной и удобной для реализации. Чистое разделение обязанностей часто является правильным решением.

## Интеграция с фронтендом на практике

### WebSocket-подключение

```javascript
class DoubaoVoiceClient {
    constructor(config) {
        this.config = config;
        this.ws = null;
    }

    async connect() {
        const url = new URL(this.config.wsUrl);
        // Add query parameters
        Object.entries(this.config.params).forEach(([key, value]) => {
            url.searchParams.set(key, value);
        });

        this.ws = new WebSocket(url);

        return new Promise((resolve, reject) => {
            this.ws.onopen = () => {
                console.log('[DoubaoVoice] Connected');
                resolve();
            };

            this.ws.onmessage = (event) => {
                this._handleMessage(JSON.parse(event.data));
            };

            this.ws.onerror = reject;
        });
    }

    _handleMessage(message) {
        switch (message.type) {
            case 'status':
                this._handleStatus(message.payload);
                break;
            case 'result':
                this.onResult?.(message.payload);
                break;
            case 'error':
                console.error('[DoubaoVoice] Error:', message.payload);
                break;
        }
    }
}

// Usage example
const client = new DoubaoVoiceClient({
    wsUrl: 'ws://localhost:5000/ws',
    params: {
        appId: 'your-app-id',
        accessToken: 'your-access-token',
        sampleRate: 16000,
        bitsPerSample: 16,
        channels: 1
    }
});
```

### Захват и передача аудио

Использование `AudioWorkletNode` для обработки аудио обеспечивает лучшую производительность:

```javascript
// audio-worklet.js
class AudioProcessorWorklet extends AudioWorkletProcessor {
    process(inputs, outputs, parameters) {
        const input = inputs[0]?.[0];
        if (!input) return true;

        // Convert to 16-bit PCM
        const pcm = new Int16Array(input.length);
        for (let i = 0; i < input.length; i++) {
            pcm[i] = Math.max(-32768, Math.min(32767, input[i] * 32767));
        }

        this.port.postMessage({
            type: 'audioData',
            data: pcm.buffer
        }, [pcm.buffer]);

        return true;
    }
}

registerProcessor('audio-processor', AudioProcessorWorklet);

// Main thread code
async function startAudioRecording() {
    const stream = await navigator.mediaDevices.getUserMedia({
        audio: {
            echoCancellation: true,
            noiseSuppression: true,
            autoGainControl: true,
            sampleRate: 48000
        }
    });

    const audioContext = new AudioContext();
    const audioSource = audioContext.createMediaStreamSource(stream);

    await audioContext.audioWorklet.addModule('/audio-worklet.js');
    const audioWorkletNode = new AudioWorkletNode(audioContext, 'audio-processor');

    audioWorkletNode.port.onmessage = (event) => {
        if (event.data.type === 'audioData' && ws?.readyState === WebSocket.OPEN) {
            ws.send(event.data.data); // Send binary data directly
        }
    };

    audioSource.connect(audioWorkletNode);
}
```

`AudioWorklet` работает значительно лучше, чем `ScriptProcessorNode`, и избегает проблем с прерывистостью аудио, которые часто возникают при использовании более старых путей обработки.

## Замечания и лучшие практики

### 1. Мониторинг подключений

- Регулярно записывайте состояние сессии для отслеживания полного жизненного цикла подключения
- Мониторьте количество и длительность аудиосегментов для обнаружения аномальных подключений
- Записывайте статус подключения и поведение переподключения к службе Doubao

Это базовые операционные практики, но они имеют реальное значение в продакшене.

### 2. Обработка ошибок

- Перехватывайте и записывайте все исключения WebSocket
- Используйте `IAsyncDisposable` для обеспечения очистки ресурсов
- Реализуйте корректное завершение подключений и обработку тайм-аутов

Короче говоря, отдавайте приоритет надёжности.

### 3. Требования к формату аудио

- Частота дискретизации: 16000 Гц (рекомендуется) или 8000 Гц
- Разрядность: 16-bit
- Каналы: моно
- Кодирование: PCM (без сжатия)

Если формат неверен, распознавание может завершиться ошибкой или качество значительно снизится. Эти требования важны.

### 4. Соображения о безопасности

- Храните конфиденциальные учётные данные только в конфигурации бэкенда
- Устанавливайте ограничения подключений для предотвращения исчерпания ресурсов
- Используйте HTTPS/WSS в продакшене

Безопасность никогда не является мелочью.

### 5. Оптимизация производительности

- Используйте асинхронные операции для избежания блокировок
- Настройте размеры буферов по необходимости (по умолчанию: 4096 байт)
- Рассмотрите стратегии объединения и повторного использования подключений

Примените эти оптимизации там, где они имеют смысл для вашей рабочей нагрузки.

## Рекомендации по развёртыванию

1. **Развёртывание в Docker**: упакуйте прокси-сервис как контейнер для удобного масштабирования и управления
2. **Балансировка нагрузки**: используйте Nginx или Envoy как обратный прокси для WebSocket-трафика
3. **Проверки работоспособности**: реализуйте проверки на основе пульса для мониторинга доступности службы
4. **Агрегация журналов**: отправляйте журналы в централизованную систему журналирования, такую как ELK или Loki

Развёртывание может быть простым или сложным в зависимости от команды и окружения, поэтому адаптируйтесь соответствующим образом.

## Итоги

Паттерн WebSocket-прокси решает основную проблему: браузерный WebSocket API не поддерживает пользовательские заголовки. В проекте HagiCode этот паттерн доказал свою целесообразность и стабильность при переходе от экспериментальной проверки к продакшен-развёртыванию.

Ключевые выводы:
- Бэкенд-прокси может безопасно передавать аутентификационную информацию

- Модель «одно подключение — одна сессия» упрощает как реализацию, так и отладку
- Протокол фронтенд-бэкенд должен разделять управляющие сигналы и аудиоданные

Если вы создаёте функцию, также зависящую от аутентификации WebSocket, надеюсь, этот паттерн послужит вам полезной отправной точкой.
