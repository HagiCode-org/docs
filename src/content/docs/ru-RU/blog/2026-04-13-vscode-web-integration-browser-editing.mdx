## VSCode и code-server: выбор решения для редактирования кода в браузере

> При создании возможностей редактирования кода в браузере разработчики сталкиваются с ключевым выбором: использовать официальную функцию `code serve-web` от VSCode или применить решение `code-server`, управляемое сообществом? Этот выбор влияет не только на техническую архитектуру, но и на соответствие лицензионным соглашениям и гибкость развёртывания.

## Предыстория

Дело технического выбора немного похоже на выбор жизненного пути. Выбрав один путь, приходится идти по нему дальше, а стоимость смены пути становится высокой.

В эпохе ИИ-поддержки программирования возможности редактирования кода в браузере становятся всё более важными. Пользователи ожидают, что после анализа кода ИИ-ассистентом они смогут немедленно открыть редактор в той же сессии браузера для внесения изменений, без переключения приложений. Такой бесшовный опыт, знаете ли, как будто он там именно тогда, когда он нужен — только иногда его именно в этот момент и нет рядом.

Однако при реализации этой функциональности разработчики сталкиваются с ключевым техническим выбором: использовать официальную функцию `code serve-web` от VSCode или применить решение `code-server`, управляемое сообществом?

У каждого из этих подходов есть свои плюсы и минусы, и неправильный выбор может создать немало проблем на более позднем этапе. Например, вопросы лицензирования — когда продукт уже запущен, обнаруживается, что лицензия не соответствует требованиям, и тогда уже поздно. Это немного похоже на отношения: изначально не продумав, потом обнаруживаешь, что ценности двух людей совершенно не совпадают, и цена тогда становится высокой. Или способ развёртывания — какое-то решение отлично работает в среде разработки, а в контейнере начинаются различные проблемы; такие ямы никто не хочет наступать,毕竟 наступаешь на ямы много, человек и онемевает.

## О HagiCode

Решение, представленное в этой статье, основано на нашем практическом опыте в проекте [HagiCode](https://hagicode.com). HagiCode — это ИИ-ассистент для программирования; при реализации возможностей редактирования кода в браузере мы глубоко изучили оба подхода и в архитектурном проекте одновременно поддерживаем оба, но по умолчанию приоритет отдаём code-server.

Адрес проекта: [github.com/HagiCode-org/site](https://github.com/HagiCode-org/site)

## Различия в лицензиях (наиболее критично)

Это самое фундаментальное различие между двумя подходами и первый фактор, который мы учитываем при выборе. Ведь при выборе первым делом нужно ясно понять юридические риски, иначе потом при возникновении проблем, на кого кого винишь?

**code-server**
- Лицензия MIT, полностью открытый исходный код
- Поддерживается компанией Coder.com, активное сообщество
- Можно свободно использовать в коммерческих целях, модифицировать и распространять
- Без ограничений на сценарии использования

**VSCode `code serve-web`**
- Является частью продукта Microsoft VSCode
- Использует лицензию Microsoft (лицензия VS Code имеет ограничения на коммерческое использование)
- В основном ориентирован на индивидуальных разработчиков
- Для корпоративного развёртывания может потребоваться дополнительное рассмотрение коммерческого лицензирования

С точки зрения лицензии code-server более дружелюбен к коммерческим проектам. Это нужно ясно понять на этапе планирования продукта, иначе когда масштаб вырастет, стоимость миграции станет высокой. Ведь миграция — это легко сказать, но сложно сделать, тот, кто прошёл через это, знает.

## Различия в способах развёртывания

После решения вопросов лицензирования следующим шагом становится способ развёртывания. Это напрямую влияет на ваши операционные расходы и архитектурное проектирование, а также косвенно влияет на ежедневное настроение — чем проще развёртывание, тем лучше настроение, этот принцип все понимают.

**code-server**
- Самостоятельное приложение Node.js, можно развёртывать отдельно
- Поддерживает различные источники среды выполнения:
  - Прямое указание пути к исполняемому файлу
  - Поиск в системном PATH
  - Автоматическое обнаружение среды NVM Node.js 22.x
- На сервере не требуется установка десктопной версии VSCode
- Контейнерное развёртывание проще

**VSCode `code serve-web`**
- Обязательно зависит от локально установленного VSCode CLI
- Требует доступную команду `code` на данной машине
- Система будет фильтровать обёртку VS Code Remote CLI
- В основном разработан для сценариев локальной разработки

code-server больше подходит для сценариев развёртывания на серверах/в контейнерах. Если ваш продукт должен работать в Docker или в среде пользователя нет VSCode, то выбирайте code-server. Ведь простота — это красота, сложность легко приводит к проблемам, проблемы нужно исправлять, после исправления можно внести новые проблемы, этот бесконечный цикл, кто хочет его переживать?

## Различия в функциональных параметрах

У двух подходов есть некоторые различия в функциональных параметрах. Хотя они невелики, на практике могут создать некоторые неудобства. Эти детали похожи на мелкие трения в жизни, немного, но когда их много, это раздражает.

| Характеристика | code-server | code serve-web |
|------|-------------|----------------|
| Базовый путь | `/` (настраиваемый) | `/vscode-server` (фиксированный) |
| Способ аутентификации | параметр `--auth`, поддерживает несколько режимов | `--connection-token` / `--without-connection-token` |
| Каталог данных | `{DataDir}/code-server` | `{DataDir}/vscode-serve-web` |
| Телеметрия | по умолчанию отключена `--disable-telemetry` | зависит от настроек VSCode |
| Проверка обновлений | можно отключить `--disable-update-check` | зависит от настроек VSCode |

При интеграции нужно обратить особое внимание на эти различия. Например, различие URL-путей означает, что frontend-код требует специальной обработки. Разработчики знают, что обработка таких деталей отнимает больше всего времени, но иначе не работает, если не сделаешь.

## Различия в проверке доступности

При реализации функции переключения редакторов логика проверки доступности также отличается. Это различие похоже на способы взаимодействия между людьми — одни любят прямоту, другие предпочитают более сдержанный подход.

**code-server**
- Всегда возвращается как видимая реализация
- Даже если недоступен, отображается, указывая статус `install-required`
- Поддерживает автоматическое обнаружение среды NVM Node.js 22.x

**code serve-web**
- Виден только при обнаружении локального CLI `code`
- Если недоступен, frontend автоматически скрывает этот вариант
- Зависит от статуса локальной установки VSCode

Это различие напрямую влияет на пользовательский опыт. Подход code-server более прозрачен — пользователь знает об этом варианте, просто ещё не установил; подход code serve-web более скрыт — пользователь может даже не знать о таком выборе. Какой подход лучше? Это зависит от позиционирования продукта,毕竟 пользовательский опыт не имеет стандартного ответа, только подходящий или неподходящий.

## Двухреализационная архитектура HagiCode

После глубокого анализа проект HagiCode принял двухреализационную архитектуру, одновременно поддерживающую оба подхода на архитектурном уровне. Это не потому что у нас синдром трудности выбора технологий, а есть реальная потребность. Ведь в техническом мире нет абсолютно правильного выбора, есть только наиболее подходящий для себя.

### Выбор по умолчанию code-server

```csharp
// Реализация по умолчанию - code-server
// Если сохранена явная activeImplementation, сначала пытается использовать эту реализацию
// Если запрошенная реализация недоступна, парсер пытается другую реализацию
// При отката возвращает fallbackReason
```

Мы по умолчанию выбираем code-server, в основном учитываем лицензию и гибкость развёртывания. Но для пользователей с локальной средой VSCode code serve-web также является неплохим выбором. Ведь предоставить пользователю ещё один выбор, в целом хорошо, заставлять других принимать единственное решение, это я не могу сделать.

### Селектор реализации

`CodeServerImplementationResolver` единообразно отвечает за:
- выбор реализации при запуске прогрева
- выбор реализации при чтении состояния
- выбор реализации при открытии проекта
- выбор реализации при открытии Vault

Эта конструкция позволяет системе гибко реагировать на различные сценарии, пользователи могут выбирать наиболее подходящую реализацию в соответствии со своей средой. Гибкое проектирование на раннем этапе тратит больше времени, но позже экономит усилия, ведь никто не хочет везде менять код.

### Правила адаптации frontend

```typescript
// когда localCodeAvailable=false, не отображать code serve-web
// когда localCodeAvailable=true, отображать конфигурацию code serve-web
```

Frontend автоматически отображает доступные варианты в соответствии со средой, чтобы пользователи не видели недоступные функции и не путались. Пользователи путаются, приходят спрашивать вас, спрашивают много, вы тоже раздражаетесь, раздражаетесь много и хотите изменить код, изменение кода может внести bug, этот порочный цикл, кто хочет переживать, тот и переживает.

## Практическое руководство

Много теории было сказано, но на что обратить внимание при фактическом развёртывании? Ведь как бы хороша ни была теория, если она не работает на практике, это бесполезно — практика — единственный критерий истины.

### Рекомендации по Docker развёртыванию

Для контейнерного развёртывания code-server является лучшим выбором:

```dockerfile
# Непосредственное использование официального образа code-server
FROM codercom/code-server:latest

# Или установка через npm
RUN npm install -g code-server
```

Таким образом одним слоем решается проблема, дополнительная установка VSCode не требуется. Простота — это хорошо, сложность легко приводит к ошибкам, этот принцип применим везде.

### Пример конфигурации

**Конфигурация code-server**
```json
{
  "vscodeServer": {
    "enabled": true,
    "activeImplementation": "code-server",
    "codeServer": {
      "host": "0.0.0.0",
      "port": 8080,
      "executablePath": "",
      "authMode": "none"
    }
  }
}
```

**Конфигурация code serve-web**
```json
{
  "vscodeServer": {
    "enabled": true,
    "activeImplementation": "serve-web",
    "serveWeb": {
      "host": "0.0.0.0",
      "port": 8080,
      "executablePath": "/usr/local/bin/code"
    }
  }
}
```

Конфигурация — это дело, в первый раз немного хлопотно, после настройки потом будет легко. Это как жизнь, больше инвестиций на раннем этапе, потом дни будут легче.

### Различия в построении URL

**code-server**
```
http://localhost:8080/?folder=/path/to/project&vscode-lang=zh-CN
```

**code serve-web**
```
http://localhost:8080/vscode-server/?folder=/path/to/project&tkn=xxx&vscode-lang=zh-CN
```

Обратите внимание на различия в путях и параметрах, при интеграции нужно обрабатывать отдельно. Детали решают успех или неудачу, это совсем не преувеличение, нехватает одного параметра, возможно не удастся открыть страницу.

### Переключение реализации

Система поддерживает переключение во время выполнения, при переключении автоматически останавливается старая реализация:

```csharp
// VsCodeServerManager автоматически обрабатывает взаимное исключение
// При переключении activeImplementation старая реализация не продолжает работать в фоновом режиме
```

Эта конструкция позволяет пользователям в любое время пробовать различные реализации и найти наиболее подходящее для себя решение. Ведь наиболее подходящее для себя является лучшим, предложения других — только для справки, в конечном итоге нужно пробовать самому.

### Мониторинг состояния

```typescript
const { settings, runtime } = await getVsCodeServerSettings();

// runtime.activeImplementation: "code-server" | "serve-web"
// runtime.fallbackReason: причина переключения
// runtime.status: "running" | "starting" | "stopped" | "unhealthy"
```

Видимость состояния, и тогда есть понимание. Пользователи при возникновении проблем могут быстро определить, это проблема сервера или их собственная операция. Не зная состояния, человек легко паникует, паника легко приводит к неправильным суждениям, эта цепочка一旦 запускается, не останавливается.

## Заключение

| Измерение сравнения | code-server | code serve-web | Рекомендация |
|---------|-------------|----------------|------|
| Лицензия | MIT (дружелюбна к коммерции) | Microsoft (ограничения) | code-server |
| Гибкость развёртывания | Самостоятельное развёртывание | Зависит от локального VSCode | code-server |
| Пригодность для серверов | Специально разработан для серверов | В основном для локальной разработки | code-server |
| Контейнеризация | Нативная поддержка | Требует установки VSCode | code-server |
| Полнота функциональности | Близко к десктопной версии | Официальная полная версия | code serve-web |
| Активность поддержки | Активное сообщество | Официально Microsoft | У каждого свои преимущества |

**Рекомендуемая стратегия**: приоритетное использование code-server, при необходимости полных официальных функций и наличии локальной среды VSCode рассматривать code serve-web.

Решение, представленное в этой статье, обобщено HagiCode из фактической разработки. Если вы считаете это решение ценным, это указывает на то, что наша инженерная практика неплохая — тогда сам HagiCode также заслуживает внимания. Ведь обмен — это интересно, когда есть взаимность, только вывод без ввода, в целом не долгосрочное решение.
