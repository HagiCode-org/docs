## Добавление автоматического развертывания GitHub Pages для HagiCode

> Ранний кодовый номер этого проекта — PCode, теперь официально переименованный в HagiCode. В этой статье записано, как внедрить возможности автоматического развертывания статических сайтов для проекта, чтобы публикация контента стала такой же простой, как пить воду.

## Предпосылки/Введение

В процессе разработки HagiCode мы столкнулись с очень реальной проблемой: по мере увеличения количества документов и предложений, как эффективно управлять и отображать этот контент стало насущной задачей. Мы решили внедрить GitHub Pages для размещения нашего статического сайта, но ручное построение и развертывание действительно слишком хлопотно — каждое изменение требует локальной сборки, упаковки, а затем ручной отправки в ветку `gh-pages`. Это не только неэффективно, но и подвержено ошибкам.

Чтобы решить эту проблему (в основном для лени), нам нужен набор автоматизированного процесса развертывания. В этой статье подробно описано, как добавить поддержку автоматического развертывания GitHub Actions для проекта HagiCode, чтобы мы могли сосредоточиться только на создании контента, а остальное оставить автоматизированному процессу.

## Анализ целей

Прежде чем начать, мы должны сначала уточнить, что именно нужно сделать в этой задаче. Ведь (здесь опечатка, должно быть毕竟) заточить топор не помешает рубке дров.

### Основные требования

1.  **Автоматическая сборка**: Когда код отправляется в ветку `main`, автоматически запускается процесс сборки.
2.  **Автоматическое развертывание**: После успешной сборки автоматически развертывает сгенерированные статические файлы на GitHub Pages.
3.  **Согласованность среды**: Обеспечение согласованности среды CI и локальной среды сборки, избегая неловкой ситуации "локально работает, онлайн ошибка".

### Выбор технологий

Учитывая, что HagiCode построен на Docusaurus (очень популярный генератор статических сайтов на React), мы можем использовать GitHub Actions для достижения этой цели.

## Настройка рабочего процесса GitHub Actions

GitHub Actions — это CI/CD сервис, предоставляемый GitHub. Определяя файлы рабочего процесса в формате YAML в хранилище кода, мы можем настроить различные автоматические задачи.

### Создание файла рабочего процесса

Нам нужно создать новый файл конфигурации в папке `.github/workflows` в корневом каталоге проекта, например `deploy.yml`. Если папка не существует, не забудьте сначала создать её вручную.

Основная логика этого файла конфигурации следующая:

1.  **Условие запуска**: Отслеживать события `push` ветки `main`.
2.  **Среда выполнения**: Последняя версия Ubuntu.
3.  **Шаги сборки**:
    -   Проверка кода
    -   Установка Node.js
    -   Установка зависимостей (`npm install`)
    -   Сборка статических файлов (`npm run build`)
4.  **Шаги развертывания**: Использование официально предоставленного `action-gh-pages` для отправки результатов сборки в ветку `gh-pages`.

### Ключевой код конфигурации

Ниже приведен шаблон конфигурации, который мы в конечном итоге приняли:

```yaml
name: Deploy to GitHub Pages

# Условие запуска: при отправке в ветку main
on:
  push:
    branches:
      - main
    # Можно добавить фильтрацию путей по необходимости, например, сборка только при изменении документов
    # paths:
    #   - 'docs/**'
    #   - 'package.json'

# Установка разрешений, это важно для развертывания на GitHub Pages
permissions:
  contents: read
  pages: write
  id-token: write

# Конкурентный контроль: отмена старых сборок той же ветки
concurrency:
  group: "pages"
  cancel-in-progress: false

jobs:
  build:
    runs-on: ubuntu-latest
    steps:
      - name: Checkout
        uses: actions/checkout@v4
        # Примечание: необходимо установить fetch-depth: 0, иначе может привести к неточному номеру версии сборки
        with:
          fetch-depth: 0

      - name: Setup Node
        uses: actions/setup-node@v4
        with:
          node-version: 20 # Рекомендуется сохранять согласованность с локальной средой разработки
          cache: 'npm'     # Включение кеша может ускорить процесс сборки

      - name: Install dependencies
        run: npm ci
        # Используйте npm ci вместо npm install, потому что это быстрее и строже, подходит для CI среды

      - name: Build website
        run: npm run build
        env:
          # Если сборка вашего сайта требует переменных среды, настройте здесь
          # NODE_ENV: production
          # PUBLIC_URL: /your-repo-name
          
      - name: Upload artifact
        uses: actions/upload-pages-artifact@v3
        with:
          path: ./build # Каталог вывода Docusaurus по умолчанию

  deploy:
    environment:
      name: github-pages
      url: ${{ steps.deployment.outputs.page_url }}
    runs-on: ubuntu-latest
    needs: build
    steps:
      - name: Deploy to GitHub Pages
        id: deployment
        uses: actions/deploy-pages@v4
```

## Проблемы в процессе реализации

В реальной работе мы столкнулись с некоторыми проблемами, здесь поделись ими, надеюсь, вы сможете их обойти (или заранее подготовить решения).

### 1. Проблема с разрешениями GitHub Token

В начале конфигурации развертывание всегда сообщало об ошибке 403 (Forbidden). Долго проверяли, прежде чем обнаружили, что у GitHub по умолчанию `GITHUB_TOKEN` нет разрешения на запись в Pages.

**Решение**: В репозитории `Settings` -> `Actions` -> `General` -> `Workflow permissions`, обязательно выберите **"Read and write permissions"**.

### 2. Ошибка пути к каталогу сборки

Docusaurus по умолчанию помещает собранные статические файлы в каталог `build`. Но некоторые проекты (например, Create React App по умолчанию `build`, Vite по умолчанию `dist`) могут иметь другую конфигурацию. Если в Actions появляется ошибка, что файл не найден, не забудьте проверить конфигурацию пути вывода в `docusaurus.config.js`.

### 3. Проблема с подпутями

Если ваш репозиторий не является пользовательской главной страницей (т.е. не `username.github.io`), а главной страницей проекта (например, `username.github.io/project-name`), вам нужно настроить `baseUrl`.

В `docusaurus.config.js`:

```javascript
module.exports = {
  // ...
  url: 'https://hagicode.com', // Ваш Hagicode URL
  baseUrl: '/', // Развертывание по корневому пути
  // ...
};
```
Это легко упустить, неправильная конфигурация приведет к тому, что страница открывается пустой, потому что пути ресурсов не загружаются.

## Проверка результатов

После настройки всех параметров и отправки кода мы можем пойти на вкладку **Actions** репозитория GitHub, чтобы посмотреть.

Вы увидите желтый круг (рабочий процесс выполняется), он станет зеленым, когда успешно! Если он станет красным, нажмите, чтобы просмотреть журнал, обычно можно выяснить проблему (в основном это опечатки или неправильная конфигурация путей).

После успешной сборки, посетите `https://<ваше имя пользователя>.github.io/<имя репозитория>/`, чтобы увидеть новый сайт.

## Итоги

Внедрив GitHub Actions, мы успешно реализовали автоматическое развертывание сайта документации HagiCode. Это не только сэкономило время на ручные операции, но и, что более важно, обеспечило стандартизацию процесса публикации. Теперь, какой бы коллега ни обновил документ, как только он сольется с веткой `main`, через несколько минут можно будет увидеть последний контент онлайн.

**Ключевые преимущества**:
- **Повышение эффективности**: От "ручной упаковки, ручной загрузки" к "код публикуется".
- **Снижение ошибок**: Устранена возможность человеческих ошибок.
- **Оптимизация опыта**: Позволяет разработчикам больше сосредоточиться на качестве контента, а не на утомительном процессе развертывания.

Хотя настройка CI/CD в начале немного хлопотна (особенно с различными разрешениями и путями), это разовая инвестиция с долгосрочной высокой отдачей. Настоятельно рекомендуется всем проектам статических сайтов внедрять аналогичные автоматические процессы.
