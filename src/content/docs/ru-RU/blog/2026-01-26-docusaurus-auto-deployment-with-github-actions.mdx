## Добавление автоматического развёртывания на GitHub Pages в HagiCode

> Раннее кодовое имя проекта — PCode, теперь он официально переименован в HagiCode. В этой статье описано, как мы внедрили автоматическое развёртывание статического сайта, чтобы публикация контента стала такой же простой, как пить воду.

## Предпосылки / Введение

В процессе разработки HagiCode мы столкнулись с весьма практической проблемой: по мере роста количества документации и предложений, эффективное управление и отображение этого контента стало крайне важным. Мы решили использовать GitHub Pages для хостинга нашего статического сайта, но ручная сборка и развёртывание были слишком обременительными. Каждое изменение требовало локальной сборки, упаковки и ручного push в ветку `gh-pages`. Это было не только неэффективно, но и подвержено ошибкам.

Для решения этой проблемы (главным образом, из-за лени) нам понадобился автоматизированный рабочий процесс развёртывания. В этой статье подробно описано, как мы добавили поддержку автоматического развёртывания на базе GitHub Actions в проект HagiCode, чтобы мы могли сосредоточиться на создании контента, а остальное доверить автоматизации.

## Анализ целей

Прежде чем приступить к работе, нам нужно чётко определить, что именно должна выполнить эта задача. В конце концов, точиль топор не мешает рубить дрова.

### Основные требования

1.  **Автоматическая сборка**: автоматический запуск процесса сборки при push кода в ветку `main`.
2.  **Автоматическое развёртывание**: после успешной сборки автоматическое развёртывание сгенерированных статических файлов на GitHub Pages.
3.  **Согласованность окружения**: обеспечение соответствия CI-окружения локальному окружению сборки, чтобы избежать неловкой ситуации «работает локально, падает на проде».

### Выбор технологий

Поскольку HagiCode построен на Docusaurus (очень популярном генераторе статических сайтов на React), мы можем использовать GitHub Actions для достижения этой цели.

## Настройка рабочего процесса GitHub Actions

GitHub Actions — это CI/CD сервис, предоставляемый GitHub. Определяя файлы рабочих процессов в формате YAML внутри репозитория, мы можем настраивать различные задачи автоматизации.

### Создание файла рабочего процесса

Нам нужно создать новый файл конфигурации в папке `.github/workflows` в корне проекта, например `deploy.yml`. Если папка не существует, не забудьте создать её вручную.

Основная логика этого файла конфигурации выглядит следующим образом:

1.  **Условие запуска**: отслеживание событий `push` в ветку `main`.
2.  **Среда выполнения**: последняя версия Ubuntu.
3.  **Шаги сборки**:
    -   Получение кода
    -   Установка Node.js
    -   Установка зависимостей (`npm install`)
    -   Сборка статических файлов (`npm run build`)
4.  **Шаг развёртывания**: использование официального `action-gh-pages` для推送 артефактов сборки в ветку `gh-pages`.

### Основной код конфигурации

Ниже представлен шаблон конфигурации, который мы в итоге приняли:

```yaml
name: Deploy to GitHub Pages

# Условие запуска: при push в ветку main
on:
  push:
    branches:
      - main
    # При необходимости можно добавить фильтрацию по путям, например сборку только при изменении документации
    # paths:
    #   - 'docs/**'
    #   - 'package.json'

# Настройка прав, что важно для развёртывания на GitHub Pages
permissions:
  contents: read
  pages: write
  id-token: write

# Управление параллелизмом: отмена старых сборок той же ветки
concurrency:
  group: "pages"
  cancel-in-progress: false

jobs:
  build:
    runs-on: ubuntu-latest
    steps:
      - name: Checkout
        uses: actions/checkout@v4
        # Примечание: необходимо установить fetch-depth: 0, иначе номер версии сборки может быть неточным
        with:
          fetch-depth: 0

      - name: Setup Node
        uses: actions/setup-node@v4
        with:
          node-version: 20 # Рекомендуется согласовывать с локальным окружением разработки
          cache: 'npm'     # Включение кэша ускоряет процесс сборки

      - name: Install dependencies
        run: npm ci
        # Используйте npm ci вместо npm install, так как он быстрее, строже и лучше подходит для CI

      - name: Build website
        run: npm run build
        env:
          # Если для сборки сайта требуются переменные окружения, настройте их здесь
          # NODE_ENV: production
          # PUBLIC_URL: /your-repo-name

      - name: Upload artifact
        uses: actions/upload-pages-artifact@v3
        with:
          path: ./build # Директория вывода Docusaurus по умолчанию

  deploy:
    environment:
      name: github-pages
      url: ${{ steps.deployment.outputs.page_url }}
    runs-on: ubuntu-latest
    needs: build
    steps:
      - name: Deploy to GitHub Pages
        id: deployment
        uses: actions/deploy-pages@v4
```

## Подводные камни при реализации

На практике мы столкнулись с несколькими проблемами. Делимся ими здесь в надежде, что все смогут их избежать или хотя бы заранее подготовить решения.

### 1. Проблемы с правами GitHub Token

При первоначальной настройке развёртывание постоянно завершалось с ошибкой 403 (Forbidden). После долгих поисков мы обнаружили, что стандартный `GITHUB_TOKEN` GitHub не имеет прав на запись в Pages.

**Решение**: в настройках репозитория `Settings` -> `Actions` -> `General` -> `Workflow permissions` обязательно выберите **«Read and write permissions»**.

### 2. Неверный путь к директории сборки

По умолчанию Docusaurus помещает собранные статические файлы в директорию `build`. Однако в некоторых проектах конфигурация может отличаться. Например, Create React App по умолчанию использует `build`, а Vite — `dist`. Если в Actions появляется ошибка о невозможности найти файлы, проверьте конфигурацию пути вывода в `docusaurus.config.js`.

### 3. Проблемы с подпутями

Если ваш репозиторий — это не пользовательская страница (т.е. не `username.github.io`), а страница проекта (например, `username.github.io/project-name`), вам необходимо настроить `baseUrl`.

В `docusaurus.config.js`:

```javascript
module.exports = {
  // ...
  url: 'https://hagicode.com', // URL вашего HagiCode
  baseUrl: '/', // Развёртывание в корневом пути
  // ...
};
```
Эта деталь легко упускается из виду. При неправильной настройке страница может отображаться как белый экран, так как пути к ресурсам не разрешаются.

## Проверка результата

После настройки всего и push кода мы можем перейти на вкладку **Actions** в репозитории GitHub и наблюдать за процессом.

Вы увидите жёлтый кружок (рабочий процесс выполняется). Когда он станет зелёным — значит успех. Если красным — перейдите в журналы для проверки проблемы. Обычно там можно найти причину, и в большинстве случаев это опечатка или неверная конфигурация пути.

После успешной сборки перейдите по адресу `https://<ваше-имя-пользователя>.github.io/<имя-репозитория>/`, и вы увидите ваш новый сайт.

## Заключение

С помощью внедрения GitHub Actions мы успешно реализовали автоматическое развёртывание документационного сайта HagiCode. Это не только сэкономило время на ручных операциях, но и, что более важно, стандартизировало процесс публикации. Теперь, независимо от того, какой участник обновил документацию, достаточно слить изменения в ветку `main`, и через несколько минут свежий контент появится онлайн.

**Ключевые преимущества**:
- **Высокая эффективность**: от «ручной упаковки и загрузки» к «код — это релиз».
- **Меньше ошибок**: исключается возможность ошибок ручных операций.
- **Лучший опыт**: позволяет разработчикам сосредоточиться на качестве контента, а не отвлекаться на рутинные шаги развёртывания.

Хотя настройка CI/CD может показаться утомительной вначале (особенно из-за проблем с правами и путями), это разовое вложение с огромной долгосрочной отдачей. Настоятельно рекомендуем всем проектам статических сайтов внедрить аналогичный автоматизированный рабочий процесс.
