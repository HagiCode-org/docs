## Дизайн архитектуры

### Обзор слоистой архитектуры

Интеграция C# с CodeBuddy использует чёткую слоистую архитектуру. Такой дизайн разделяет обязанности кода, что делает последующее обслуживание более удобным:

```
┌─────────────────────────────────────────────┐
│           Уровень контракта Provider         │
│   Перечисление AIProviderType + методы расширения             │
├─────────────────────────────────────────────┤
│           Уровень фабрики Provider           │
│   Фабрика внедрения зависимостей AIProviderFactory             │
├─────────────────────────────────────────────┤
│           Уровень реализации Provider        │
│   Конкретная реализация CodebuddyCliProvider              │
├─────────────────────────────────────────────┤
│           Уровень инфраструктуры ACP         │
│  ACPSessionManager / StdioAcpTransport      │
│  AcpRpcClient / AcpAgentClient              │
└─────────────────────────────────────────────┘
```

В чём好处 такой слоистости? Проще говоря, слои не мешают друг другу. Допустим, позже нужно изменить способ связи (например, с stdio на WebSocket), вам нужно изменить только самый нижний слой, а бизнес-код сверху полностью менять не нужно. В конце концов, никто не хочет, чтобы одно изменение затрагивало всё - менять способ связи и полдня переписывать бизнес-код, это уже слишком.

### Разбор основных компонентов

**Уровень контракта Provider** является краеугольным камнем всей архитектуры. Мы определили перечисление `AIProviderType`, где `CodebuddyCli = 3` как значение перечисления, а через методы расширения реализовали двустороннее отображение между строками и перечислениями. Таким образом, строки в файле конфигурации можно удобно преобразовать в перечисления, а при отладке перечисления также можно преобразовать в строки для вывода. Это ещё нормально - по сути это отношение отображения, но если сделать хорошо, то станет удобнее.

**Уровень фабрики Provider** отвечает за создание соответствующих экземпляров Provider на основе конфигурации. Здесь используется механизм внедрения зависимостей .NET в сочетании с `ActivatorUtilities.CreateInstance` для динамического создания. Преимущество паттерна фабрики в том, что при добавлении нового Provider нужно только добавить логику создания, не изменяя существующий код. Это похоже на написание статьи - если хотите добавить новую главу, просто добавьте новую главу, не нужно переписывать всё предыдущее.

**Уровень реализации Provider** - это место, где происходит реальная работа. `CodebuddyCliProvider` реализует интерфейс `IAIProvider`, предоставляя два способа вызова: `ExecuteAsync` (потоковый) и `StreamAsync` (потоковый).

**Уровень инфраструктуры ACP** является нижележащей поддержкой связи. Этот уровень обрабатывает все детали протокола, включая управление процессами, сериализацию сообщений, синтаксический анализ ответов и т.д. Как фундамент дома - whatever beautiful above, the foundation must be solid.

## Механизм связи

### Режим передачи Stdio

CodeBuddy использует способ **Stdio (стандартный ввод/вывод)** для связи с внешними процессами. Команда запуска очень проста:

```bash
codebuddy --acp
```

Затем происходит обмен сообщениями JSON-RPC через стандартный ввод/вывод. Преимущества этого способа:

1. **Быстрый запуск**：нет задержки сети при локальной межпроцессной связи
2. **Простая конфигурация**：нужно только указать путь к исполняемому файлу
3. **Изоляция окружения**：каждая сессия - независимый процесс, не влияют друг на друга

В процессе связи поддерживается внедрение переменных окружения, обычно используемых:

- `CODEBUDDY_API_KEY`：аутентификация по ключу API
- `CODEBUDDY_INTERNET_ENVIRONMENT`：конфигурация сетевого окружения

Это как общение между людьми - найдите удобный способ, и можно будет разговаривать.

### Протокол сообщений

ACP основан на протоколе JSON-RPC 2.0, формат сообщений примерно такой:

```json
// Сообщение запроса
{
  "jsonrpc": "2.0",
  "id": 1,
  "method": "agent/prompt",
  "params": {
    "prompt": "Помоги написать алгоритм сортировки",
    "sessionId": "session-123"
  }
}

// Сообщение ответа
{
  "jsonrpc": "2.0",
  "id": 1,
  "result": {
    "content": "Здесь ответ ИИ..."
  }
}
```

В реальной реализации мы инкапсулировали все детали протокола, бизнес-код верхнего уровня должен заботиться только о prompt и response. Это ещё нормально - если инкапсулировать хорошо, последующим людям будет удобнее пользоваться.

### 2. Интеграция фабрики Provider

Добавьте ветвь создания CodeBuddy в классе фабрики:

```csharp
// AIProviderFactory.cs
private IAIProvider? CreateProvider(AIProviderType providerType, ProviderConfiguration config)
{
    return providerType switch
    {
        AIProviderType.CodebuddyCli =>
            ActivatorUtilities.CreateInstance<CodebuddyCliProvider>(
                _serviceProvider,
                Options.Create(config)),
```

Здесь используется `ActivatorUtilities` внедрения зависимостей, он автоматически обрабатывает внедрение параметров конструктора, очень удобно. Это ещё нормально - если использовать .NET правильно, будет удобно.

### 3. Полная реализация Provider

Ниже приведена основная реализация `CodebuddyCliProvider`, включающая два способа вызова: потоковый и непотоковый:

```csharp
public class CodebuddyCliProvider : IAIProvider
{
    private readonly ILogger<CodebuddyCliProvider> _logger;
    private readonly IACPSessionManager _sessionManager;
    private readonly ProviderConfiguration _config;

    public string Name => "CodebuddyCli";
    public bool SupportsStreaming => true;
    public ProviderCapabilities Capabilities { get; }

    public CodebuddyCliProvider(
        ILogger<CodebuddyCliProvider> logger,
        IACPSessionManager sessionManager,
        IOptions<ProviderConfiguration> config)
    {
        _logger = logger;
        _sessionManager = sessionManager;
        _config = config.Value;

        // Определяем возможности текущего Provider
        Capabilities = new ProviderCapabilities
        {
            SupportsStreaming = true,
            SupportsTools = true,
            SupportsSystemMessages = true,
            SupportsArtifacts = false,
            MaxTokens = 8192
        };
    }

    // Непотоковый вызов: ждать всех результатов вместе
    public async Task<AIResponse> ExecuteAsync(
        AIRequest request,
        CancellationToken cancellationToken = default)
    {
        // Создать независимую сессию для запроса
        var session = await _sessionManager.CreateSessionAsync(
            "CodebuddyCli",
            request.WorkingDirectory,
            cancellationToken,
            request.SessionId);

        try
        {
            var fullPrompt = BuildPrompt(request);
            await session.SendPromptAsync(fullPrompt, cancellationToken);

            var responseBuilder = new StringBuilder();
            var toolCalls = new List<AIToolCall>();

            // Собрать все блоки ответов
            await foreach (var chunk in StreamFromSession(session, cancellationToken))
            {
                if (!string.IsNullOrEmpty(chunk.Content))
                {
                    responseBuilder.Append(chunk.Content);
                }
                // Обработка вызовов инструментов...
            }

            return new AIResponse
            {
                Content = AIResultContentSanitizer.SanitizeResultContent(
                    responseBuilder.ToString()),
                ToolCalls = toolCalls,
                Provider = Name,
                Model = string.Empty
            };
        }
        finally
        {
            // Освободить ресурсы сессии
            await session.DisposeAsync();
        }
    }

    // Потоковый вызов: возвращать блоки ответов в реальном времени
    public async IAsyncEnumerable<AIStreamingChunk> StreamAsync(
        AIRequest request,
        [EnumeratorCancellation] CancellationToken cancellationToken = default)
    {
        var session = await _sessionManager.CreateSessionAsync(
            "CodebuddyCli",
            request.WorkingDirectory,
            cancellationToken);

        try
        {
            var fullPrompt = BuildPrompt(request);
            await session.SendPromptAsync(fullPrompt, cancellationToken);

            await foreach (var chunk in StreamFromSession(session, cancellationToken))
            {
                yield return chunk;
            }
        }
        finally
        {
            await session.DisposeAsync();
        }
    }

    private async IAsyncEnumerable<AIStreamingChunk> StreamFromSession(
        IACPSession session,
        [EnumeratorCancellation] CancellationToken cancellationToken)
    {
        // Перебрать все обновления в сессии
        await foreach (var notification in session.ReceiveUpdatesAsync(cancellationToken))
        {
            switch (notification.Update)
            {
                case AgentMessageChunkSessionUpdate agentMessage:
                    // Обработать блоки текстового содержимого
                    if (agentMessage.Content is AcpImp.TextContentBlock textContent)
                    {
                        yield return new AIStreamingChunk
                        {
                            Content = textContent.Text,
                            Type = StreamingChunkType.ContentDelta,
                            IsComplete = false
                        };
                    }
                    break;

                case ToolCallSessionUpdate toolCall:
                    // Обработать вызов инструмента
                    yield return new AIStreamingChunk
                    {
                        Content = string.Empty,
                        Type = StreamingChunkType.ToolCallDelta,
                        ToolCallDelta = new AIToolCallDelta
                        {
                            Id = toolCall.ToolCallId,
                            Name = toolCall.Kind.ToString(),
                            Arguments = toolCall.RawInput?.ToString()
                        }
                    };
                    break;

                case AcpImp.PromptCompletedSessionUpdate:
                    // Ответ завершён
                    yield break;
            }
        }
    }

    // Построить полный подсказку
    private string BuildPrompt(AIRequest request, string? embeddedCommandPrompt = null)
    {
        var sb = new StringBuilder();

        // Встроенная подсказка команды (если есть)
        if (!string.IsNullOrEmpty(embeddedCommandPrompt))
        {
            sb.AppendLine(embeddedCommandPrompt);
            sb.AppendLine();
        }

        // Системное сообщение
        if (!string.IsNullOrEmpty(request.SystemMessage))
        {
            sb.AppendLine(request.SystemMessage);
            sb.AppendLine();
        }

        // Пользовательский prompt
        sb.Append(request.Prompt);
        return sb.ToString();
    }
}
```

Этот код имеет несколько ключевых моментов:

1. **Управление сессиями**：каждый запрос создаёт независимую сессию, после завершения запроса ресурсы освобождаются. Это опыт, полученный на ямах - если повторное использование сессии сделано плохо, легко возникает проблема загрязнения состояния. В конце концов, после использования нужно убрать, иначе следующему пользователю будет неудобно.

2. **Потоковая обработка**：`IAsyncEnumerable` позволяет возвращать ответ по мере генерации, не дожидаясь генерации всего содержимого. Это особенно важно для сценариев с длинным текстом, пользовательский опыт будет намного лучше. Как, человек, который ждёт результата, тоже не хочет просто ждать.

3. **Вызов инструментов**：CodeBuddy поддерживает вызов инструментов (Function Calling), обрабатывается через `ToolCallSessionUpdate`. Эта возможность критически важна для сложных задач редактирования кода.

4. **Фильтрация содержимого**：используйте `AIResultContentSanitizer` для фильтрации содержимого блоков Think, сохраняя вывод чистым.

## Резюме практического опыта

### Запись о ямах

В процессе реализации мы столкнулись с несколькими типичными ямами, делимся ими, чтобы вы могли избежать обходных путей. В конце концов, чужие ямы, которые можно обойти, - это уже хорошо:

1. **Проблема утечки сессий**：сначала не правильно освобождали сессии, что приводило к исчерпанию ресурсов процесса. Решение - использовать `try-finally`, чтобы убедиться, что каждый запрос освободит ресурсы. Это ещё нормально - использованные вещи нужно вернуть, иначе что будут использовать следующие.

2. **Передача переменных окружения**：синтаксис переменных окружения в Windows и Linux разный, затем унифицировали использование `Dictionary<string, string?>` для обработки. Кроссплатформенность - если сразу унифицировать стандарты, потом будет удобнее.

3. **Конфигурация таймаута**：запуск CLI требует времени, установили 5-секундный таймаут запуска, чтобы избежать сбоя быстрых запросов. Во всём нужна мера, слишком спешить не стоит - наоборот, не получится сделать дело.

4. **Проблема кодировки**：кодировка по умолчанию на Windows может привести к искажению китайских символов, при запуске процесса явно укажите кодировку UTF-8. Если китайский не отображается, это так неприятно.

### Оптимизация производительности

1. **Пул сессий**：для частых коротких запросов можно рассмотреть реализацию пула сессий для повторного использования процессов
2. **Кэширование соединений**：класс фабрики уже поддерживает кэширование экземпляров Provider
3. **Приоритет асинхронности**：полностью используйте асинхронное программирование, избегайте блокировки потоков

Производительность - что можно оптимизировать, то оптимизируйте, ведь чем дольше ждёт пользователь, тем хуже опыт.

- Использование слоистой архитектуры уровня контракта Provider, уровня фабрики, уровня реализации и уровня инфраструктуры
- Использование JSON-RPC over Stdio для межпроцессной связи
- Реализация гибкой конфигурации и расширения через внедрение зависимостей
- Предоставление двух способов вызова: потокового и непотокового

Это решение применимо не только к CodeBuddy, добавление новых AI Provider также следует тому же шаблону. Если вы тоже занимаетесь подобной интеграцией нескольких AI Provider, надеюсь, эта статья даст вам некоторую ссылку. На самом деле, писать статьи и писать код одно и то же - если поделиться и помочь кому-то, то не зря писали.
