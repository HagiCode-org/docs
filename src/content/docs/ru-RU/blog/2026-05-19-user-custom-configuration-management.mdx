## Лучшие практики управления пользовательскими настройками

> Ничего особенного, просто делюсь набором решений для управления конфигурацией. От архитектуры до практики — надеюсь, это поможет вам избежать некоторых проблем.

## Предыстория

Управление конфигурацией — вещь и важная, и мелочная. Как мелкие привычки в жизни — кто-то любит утром пить кофе, кто-то предпочитает ночную кодинг-сессию. Эти, казалось бы, незначительные предпочтения на самом деле тихо определяют, кто вы есть. Пользовательские настройки — то же самое: смена темы, выбор языка, настройка горячих клавиш. Когда эти функции сделаны хорошо, пользователи чувствуют, что продукт «их понимает», и лояльность растёт.

Только вот, сделать эту систему полноценной не так просто, как кажется. Управление версиями, обратная совместимость, валидация данных, контроль параллелизма, функциональные флаги DLC — эти проблемы возникают одна за другой, как подростковые тревоги, и их не обойти. В процессе разработки HagiCode мы тоже мучились с этими проблемами, но в конце концов кое-что поняли.

## О HagiCode

Это решение мы придумали в процессе работы над HagiCode. HagiCode — это AI-помощник по коду с множеством функций — язык интерфейса, языковые предпочтения ИИ, темы, распознавание речи, уведомления, быстрые действия. Настроить можно многое. Именно потому, что пользователям нужно было слишком много, нам пришлось создать эту систему.

Адрес проекта: [github.com/HagiCode-org/site](https://github.com/HagiCode-org/site)

## Архитектурный дизайн

### Слоистая архитектура

Система управления конфигурацией HagiCode,怎么说呢, просто разделена на несколько слоев, каждый со своими обязанностями:

```
┌─────────────────────────────────────────┐
│         Frontend (React + Redux)        │
│  - Управление состоянием конфигурации    │
│  - Рендеринг UI форм                     │
│  - Персистентность по группам            │
└─────────────────────────────────────────┘
                    ↓ HTTP/REST API
┌─────────────────────────────────────────┐
│   Application Service Layer             │
│  - FrontendConfigAppService             │
│  - Обработка бизнес-логики               │
│  - Контроль разрешений                   │
└─────────────────────────────────────────┘
                    ↓
┌─────────────────────────────────────────┐
│   Domain Layer (Config Store)           │
│  - FrontendConfigStore                  │
│  - Чтение/запись конфигурации            │
│  - Валидация и нормализация данных       │
└─────────────────────────────────────────┘
                    ↓
┌─────────────────────────────────────────┐
│   Infrastructure Layer                  │
│  - YAML файловое хранилище               │
│  - ISystemManagedVaultService           │
│  - Контроль параллелизма (SemaphoreSlim)│
└─────────────────────────────────────────┘
```

Преимущества такого слоя довольно очевидны:
- **Чёткое разделение обязанностей**: каждый слой занимается своим делом, не мешая другим
- **Простота тестирования**: каждый слой можно тестировать отдельно, изменения вносить безопаснее
- **Гибкое расширение**: хотите сменить способ хранения или изменить API — остальные слои продолжают работать

### Дизайн основных интерфейсов

Интерфейс для бэкенд-хранилища конфигурации примерно такой:

```csharp
public interface IFrontendConfigStore
{
    // Получить полную конфигурацию пользователя
    Task<FrontendConfigStoreResult> GetAsync(CancellationToken cancellationToken = default);
    
    // Обновить конфигурацию (поддерживает частичное обновление)
    Task<FrontendConfigStoreResult> UpdateAsync(
        UpdateFrontendConfigRequestDto input,
        CancellationToken cancellationToken = default);
    
    // Управление состоянием AI-языка
    Task<FrontendConfigAiLanguageState> GetAiLanguageStateAsync(
        string userId,
        CancellationToken cancellationToken = default);
        
    Task<FrontendConfigAiLanguageState> SetAiLanguageAsync(
        string userId,
        string language,
        CancellationToken cancellationToken = default);
}
```

Несколько ключевых моментов этого интерфейса тоже довольно понятны:
- **Асинхронные операции**: всё асинхронно, никто не любит ждать
- **Токены отмены**: слишком долгие операции истекают по таймауту, не виснут бесконечно
- **Частичные обновления**: обновляется только нужная часть, не надо перебирать всю конфигурацию

## Структура данных конфигурации

### Дизайн групп конфигурации

HagiCode разделяет конфигурацию по функциям на группы, каждая группа может обновляться независимо:

```csharp
public class FrontendConfigSnapshotDto
{
    // Общие настройки
    public FrontendConfigGeneralSettingsDto GeneralSettings { get; set; }
    
    // Конфигурация AI-языка
    public FrontendConfigAILanguageDto AiLanguage { get; set; }
    
    // Область действия проекта
    public FrontendConfigProjectScopeDto ProjectScope { get; set; }
    
    // Язык интерфейса
    public string UiLanguage { get; set; }
    
    // Тема
    public string Theme { get; set; }
    
    // Распознавание речи
    public FrontendConfigVoiceRecognitionDto VoiceRecognition { get; set; }
    
    // Настройки уведомлений
    public FrontendConfigNotificationsDto Notifications { get; set; }
    
    // Сортировка сессий
    public FrontendConfigSessionSortingDto SessionSorting { get; set; }
    
    // Быстрые действия
    public FrontendConfigQuickActionsDto QuickActions { get; set; }
    
    // Диалоги подтверждения
    public FrontendConfigConfirmDialogDto ConfirmDialog { get; set; }
    
    // Пресеты сессий
    public FrontendConfigSessionPresetsDto SessionPresets { get; set; }
    
    // Конфигурация иконок проектов
    public FrontendConfigProjectIconConfigDto ProjectIconConfig { get; set; }
    
    // Общие комментарии
    public FrontendConfigCommonCommentsDto CommonComments { get; set; }
}
```

Группировка конфигурации — это как классификация бытовых мелочей — работа отдельно, развлечения отдельно, отношения отдельно. Если смешать всё вместе — будет хаос, если чётко разделить — станет легче:
- **Обновление по требованию**: меняешь что-то одно — обновляется только это, не трогаешь всё остальное
- **Контроль разрешений**: разные конфигурации могут иметь разные разрешения,毕竟不是谁都能乱动的
- **Функциональные флаги DLC**: продвинутые функции можно привязать к DLC, чтобы пользоваться хорошими вещами нужно платить

## Валидация и нормализация данных

### Пример нормализации языка

Нормализация конфигурации,说白了就是让数据保持一致。 На примере языковой конфигурации:

```csharp
public static class FrontendConfigLanguageRules
{
    // Список поддерживаемых языков интерфейса
    public static readonly HashSet<string> ValidUiLanguages = new(StringComparer.OrdinalIgnoreCase)
    {
        "zh-CN", "zh-Hant", "en-US", "ja-JP", "ko-KR", 
        "de-DE", "fr-FR", "es-ES", "pt-BR", "ru-RU"
    };
    
    public static string NormalizeUiLanguage(string? value)
    {
        // Обработка пустых значений: возврат языка по умолчанию
        if (string.IsNullOrWhiteSpace(value)) return "en-US";
        
        var normalized = value.Trim();
        
        // Обработка псевдонимов: преобразование общих псевдонимов в стандартные коды
        normalized = normalized.ToLower() switch
        {
            "zh" or "chinese" or "cn" => "zh-CN",
            "en" or "english" => "en-US",
            "ja" or "japanese" => "ja-JP",
            "ko" or "korean" => "ko-KR",
            _ => normalized
        };
        
        // Обработка диалектных вариантов
        if (normalized.StartsWith("zh-Hans", StringComparison.OrdinalIgnoreCase))
            return "zh-CN";
        if (normalized.StartsWith("zh-TW", StringComparison.OrdinalIgnoreCase))
            return "zh-Hant";
            
        // Проверка и возврат
        return ValidUiLanguages.Contains(normalized) ? normalized : "en-US";
    }
}
```

Нормализация — это как уборка комнаты — если вещи беспорядочно лежат, нужно их разложить, иначе в конце сам себя не найдёшь:
1. **Защита от пустых значений**: даём пустым значениям разумное значение по умолчанию, нельзя оставлять их пустыми
2. **Маппинг псевдонимов**: общие псевдонимы, сокращения равномерно преобразуем в стандартный формат
3. **Унификация диалектов**: диалектные варианты объединяем в стандартные коды,毕竟写代码不是做方言研究
4. **Финальная проверка**: гарантируем, что возвращаемое значение точно в списке допустимых, иначе зря старались

## Функциональные флаги DLC

HagiCode поддерживает функциональные переключатели DLC, некоторые продвинутые настройки доступны только после покупки DLC. Это обычная практика в коммерческом ПО — базовые функции бесплатны, чтобы пользоваться хорошими вещами нужно платить,毕竟 разработчики тоже должны есть.

### Проверка доступа DLC

```csharp
private async Task<PreparedFrontendConfigUpdate> PrepareUpdateAsync(
    UpdateFrontendConfigRequestDto input,
    FrontendConfigStoreResult current,
    CancellationToken cancellationToken)
{
    // Проверка разрешений DLC
    var accessState = await _grainFactory
        .GetDlcAccessStateGrain(TurboEngineDlcId)
        .GetAccessStateAsync();
        
    var blockedFields = new List<string>();
    
    if (!accessState.IsActive)
    {
        // DLC не активирован, сохраняем текущую конфигурацию
        if (input.GeneralSettings?.BrandingLogo != null)
        {
            blockedFields.Add("brandingLogo");
            // Сохраняем старое значение
            input.GeneralSettings.BrandingLogo = current.Snapshot.GeneralSettings.BrandingLogo;
        }
        
        if (input.GeneralSettings?.BrandingTitle != null)
        {
            blockedFields.Add("brandingTitle");
            input.GeneralSettings.BrandingTitle = current.Snapshot.GeneralSettings.BrandingTitle;
        }
    }
    
    return new PreparedFrontendConfigUpdate(
        input,
        new FrontendConfigUpdateDiagnosticsDto
        {
            Status = blockedFields.Count > 0 ? "partially-applied" : "success",
            BlockedFields = blockedFields,
        });
}
```

## Управление состоянием фронтенда

### Redux Slice

На фронтенде используется Redux для управления состоянием конфигурации, тоже довольно стандартно:

```typescript
export const frontendConfigSlice = createSlice({
  name: 'frontendConfig',
  initialState,
  reducers: {
    setConfigStatus(state, action: PayloadAction<FrontendConfigStatus>) {
      state.status = action.payload;
    },
    updateConfigGroups(state, action: PayloadAction<Partial<FrontendConfigSnapshot>>) {
      // Слияние обновлений конфигурации
      Object.assign(state.snapshot, action.payload);
    },
  },
  extraReducers: (builder) => {
    builder
      .addCase(fetchConfig.pending, (state) => {
        state.status = 'loading';
      })
      .addCase(fetchConfig.fulfilled, (state, action) => {
        state.status = 'succeeded';
        state.snapshot = action.payload;
      })
      .addCase(fetchConfig.rejected, (state, action) => {
        state.status = 'failed';
        state.error = action.error.message;
      });
  },
});
```

```typescript
      body: requestBody,
      mediaType: 'application/json',

    group: PersistableFrontendConfigGroup,

    return this.updateConfig({
      configGroup: group,
      value: value,
    });
  },
};
```

## Контроль параллелизма

Операции обновления конфигурации должны быть потокобезопасными, иначе если два человека одновременно меняют конфигурацию, что сохранится в итоге? HagiCode использует `SemaphoreSlim` для контроля параллелизма,怎么说呢, тоже довольно распространённый приём:

```csharp
private readonly SemaphoreSlim _semaphore = new(1, 1);

public async Task<FrontendConfigStoreResult> UpdateAsync(
    UpdateFrontendConfigRequestDto input,
    CancellationToken cancellationToken = default)
{
    await _semaphore.WaitAsync(cancellationToken);
    try
    {
        // Чтение текущей конфигурации
        var current = await GetAsync(cancellationToken);
        
        // Подготовка обновления
        var prepared = await PrepareUpdateAsync(input, current, cancellationToken);
        
        // Запись конфигурации
        await WriteConfigAsync(prepared.UpdatedConfig, cancellationToken);
        
        return new FrontendConfigStoreResult(prepared.UpdatedConfig, prepared.Diagnostics);
    }
    finally
    {
        _semaphore.Release();
    }
}
```

## Практическое руководство

### Шаги по добавлению новой настройки

Если хотите добавить новую настройку, следуйте этому порядку:

1. **Добавить свойство в бэкенд DTO**

```csharp
public class FrontendConfigGeneralSettingsDto
{
    // ... существующие свойства
    public string? NewFeatureEnabled { get; set; }
}
```

2. **Добавить логику нормализации**

```csharp
private static FrontendConfigGeneralSettingsDto NormalizeGeneralSettings(
    FrontendConfigGeneralSettingsDto settings)
{
    return new FrontendConfigGeneralSettingsDto
    {
        // ... существующие свойства
        NewFeatureEnabled = NormalizeOptionalString(settings.NewFeatureEnabled),
    };
}
```

3. **Добавить элемент управления формы на фронтенде**

```typescript
<SettingsCard
  icon={<Star className="h-5 w-5" />}
  title="Настройки новой функции"
  description="Управление состоянием включения новой функции"
>
  <NewFeatureToggle />
</SettingsCard>
```

4. **Обновить определение групп конфигурации**（если добавляете новую группу）

```typescript
export const ALL_PERSISTABLE_FRONTEND_CONFIG_GROUPS = [
  // ... существующие группы
  'newFeatureSettings',
] as const;
```

### Обработка распространённых проблем

1. **Потеря конфигурации**: механизмы резервного копирования и восстановления, всегда要有备无患
2. **Конфликты параллелизма**: оптимистическая или пессимистическая блокировка, например SemaphoreSlim
3. **Проблемы производительности**: поддержка частичных обновлений, не нужно каждый раз перемещать весь объект конфигурации туда-сюда
4. **Безопасность**: чувствительные конфигурации (например, API-ключи) должны храниться зашифрованными, иначе если их украдут — будет проблема

Это решение HagiCode в продакшене работает довольно стабильно, по крайней мере может удовлетворять сложные потребности управления конфигурацией.
