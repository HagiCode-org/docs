## Построение межпроектной базы знаний эпохи ИИ с помощью системы Vault

> Метод обучения путём копирования проектов становится мейнстримом, но разрозненность учебных материалов и разрывы контекста мешают ИИ-ассистентам проявить себя максимально эффективно. В этой статье рассказывается о проектировании системы Vault в проекте HagiCode — с помощью единого уровня абстракции хранилища ИИ-ассистент получает возможность понимать и доступать все учебные ресурсы, реализуя настоящую межпроектную повторное использование знаний.

## Предыстория

На самом деле, в эпоху ИИ способ обучения новым технологиям незаметно меняется. Традиционное чтение книг и просмотр видео, конечно, важны, но «копирование проектов» — глубокое изучение и изучение кода, архитектуры и шаблонов проектирования превосходных проектов с открытым исходным кодом — действительно становится всё более эффективным. Непосредственный запуск и изменение высококачественных проектов с открытым исходным кодом позволяет вам быстрее всего понять инженерную практику реального мира.

Однако этот подход также приносит новые проблемы.

**Учебные материалы слишком разрозненны**. Заметки могут быть в Obsidian, репозитории кода разбросаны по разным папкам, а история диалогов ИИ-ассистента — это ещё один изолированный остров данных. Каждый раз, когда нужна помощь ИИ для анализа проекта, приходится вручную копировать фрагменты кода и упорядочивать контекст — процесс довольно утомительный.

**Контекст часто прерывается**. ИИ-ассистент не может напрямую доступать локальные учебные ресурсы, и каждый диалог требует повторного предоставления фоновой информации. Репозитории кода для копирования обновляются быстро, и ручная синхронизация подвержена ошибкам. Хуже того, сложно обмениваться знаниями между несколькими учебными проектами — шаблоны проектирования, изученные в проекте A, ИИ при обработке проекта B совершенно не знает.

Суть этих проблем — «изоляция данных». Если бы существовал единый уровень абстракции хранилища, позволяющий ИИ-ассистенту понимать и доступать все учебные ресурсы, проблема была бы легко решена.

Для решения этих проблем при разработке HagiCode мы приняли ключевое проектное решение: построить систему Vault как единый уровень абстракции хранения знаний. Изменения, которые принесло это решение, могут быть больше, чем можно представить — об этом подробнее позже.

## О HagiCode

Решение, которым делится эта статья, основано на практическом опыте проекта [HagiCode](https://hagicode.com). HagiCode — это ИИ-ассистент кода на основе рабочего процесса OpenSpec; его основная концепция — позволить ИИ не только «говорить», но и «делать» — напрямую оперировать репозиториями кода, выполнять команды, запускать тесты. GitHub: [github.com/HagiCode-org/site](https://github.com/HagiCode-org/site)

В процессе разработки мы обнаружили, что ИИ-ассистенту требуется частый доступ к различным учебным ресурсам пользователя: репозиториям кода, документам с заметками, файлам конфигурации и т. д. Если каждый раз просить пользователя предоставлять их вручную, опыт будет слишком плохим. Это побудило создать систему Vault.

## Основной дизайн

### Поддержка нескольких типов

Система Vault в HagiCode поддерживает четыре типа, соответствующие разным сценариям использования:

| Тип | Назначение | Типичные сценарии |
|------|------|----------|
| `folder` | Универсальный тип папки | Временные учебные материалы, черновики |
| `coderef` | Специально для копирования проектов кода | Систематическое изучение определённого проекта с открытым исходным кодом |
| `obsidian` | Интеграция с программой заметок Obsidian | Повторное использование существующей базы заметок |
| `system-managed` | Автоматическое управление системой | Конфигурация проекта, шаблоны промптов и т. д. |

Тип `coderef` — наиболее часто используемый в HagiCode; он обеспечивает стандартизированную структуру каталогов и понятные ИИ описания метаданных для копирования проектов кода. Почему для этого типа было спроектировано отдельно? Потому что копирование проекта с открытым исходным кодом — это не просто «скачать код»; нужно одновременно управлять самим кодом, учебными заметками, файлами конфигурации и т. д.; `coderef` всё это стандартизирует.

### Механизм персистентного хранилища

Реестр Vault персистентно хранится в файловой системе в формате JSON:

```csharp
_registryFilePath = Path.Combine(absoluteDataDir, "personal-data", "vaults", "registry.json");
```

Эта конструкция кажется простой, но продуманной:

**Простота и надёжность**. Формат JSON читаем человеком, удобен для отладки и ручного изменения. Когда в системе возникают проблемы, можно напрямую открыть файл для проверки состояния и даже вручную исправить — это особенно полезно на этапе разработки.

**Снижение зависимостей**. Хранение в файловой системе позволяет избежать сложности базы данных. Не нужно дополнительно устанавливать и настраивать службы базы данных, что снижает сложность системы и стоимость обслуживания.

**Потокобезопасность**. Использование `SemaphoreSlim` гарантирует безопасность многопоточности. В сценарии ИИ-ассистента кода несколько операций могут одновременно доступать реестр vault, и необходимо обеспечить контроль параллелизма.

### Интеграция с контекстом ИИ

Основная способность системы заключается в автоматическом внедрении информации о vault в контекст ИИ-предложения:

```typescript
export function buildTargetVaultsText(
  vaults: VaultForText[],
  template: VaultPromptTemplate = DEFAULT_VAULT_PROMPT_TEMPLATE,
): string {
  const readOnlyVaults = vaults.filter((vault) => vault.accessType === 'read');
  const editableVaults = vaults.filter((vault) => vault.accessType === 'write');

  const sections = [
    buildVaultSection(readOnlyVaults, template.reference),
    buildVaultSection(editableVaults, template.editable),
  ].filter(Boolean);

  return `\n\n### ${template.heading}\n\n${sections.join('\n')}`;
}
```

Так ИИ-ассистент может автоматически понять доступные учебные ресурсы, не требуя от пользователя вручную предоставлять контекст каждый раз. Этот дизайн делает опыт HagiCode особенно естественным — скажите ИИ «помоги мне проанализировать параллельный рендеринг React», ИИ автоматически найдёт ранее зарегистрированный обучающий vault React, вместо того чтобы копировать код снова и снова.

### Механизм контроля доступа

Система делит vault на два типа доступа:

- **reference（только чтение）**：ИИ используется только для анализа и понимания, не может изменять содержимое
- **editable（редактируемый）**：ИИ может изменять содержимое в зависимости от задач

Это различие позволяет ИИ знать, какой контент «только для чтения», а какой «можно изменять», избегая риска ошибочных операций. Например, если vault проекта с открытым исходным кодом зарегистрирован как учебный материал, вы точно не хотите, чтобы ИИ случайно изменял код внутри — тогда пометьте его как `reference`. Но если это ваш собственный проект vault, можно пометить его как `editable`, и ИИ поможет изменить код.

## Практическое руководство

### Стандартизированная структура CodeRef Vault

Для vault типа coderef система предоставляет стандартизированную структуру каталогов:

```
my-coderef-vault/
├── index.yaml          # описание метаданных vault
├── AGENTS.md           # руководство по операциям для ИИ-ассистента
├── docs/               # хранение учебных заметок и документов
└── repos/              # управление репозиториями кода для копирования через Git-сабмодули
```

В чём заключается концепция дизайна этой структуры?

**docs/** хранит учебные заметки, в формате Markdown записывая понимание кода, анализ архитектуры, опыт с проблемами. Эти заметки не только для вас, ИИ тоже может их прочитать — при выполнении соответствующих задач они будут автоматически ссылаться.

**repos/** управляет копируемыми репозиториями через Git-сабмодули, а не напрямую копирует код. Это даёт два преимущества: во-первых, поддерживает синхронизацию с апстримом — один `git submodule update` позволяет получить последний код; во-вторых, экономит место — несколько vault могут ссылаться на разные версии одного и того же репозитория.

**index.yaml** содержит метаданные vault, позволяя ИИ-ассистенту быстро понять назначение и содержимое. Это как «самопредставление» vault; ИИ при первой встрече сразу понимает, для чего это нужно.

**AGENTS.md** — это руководство, написанное специально для ИИ-ассистента, объясняющее, как обрабатывать содержимое vault. Здесь можно сказать ИИ: «при анализе этого проекта сосредоточьтесь на коде, связанном с оптимизацией производительности» или «не изменяйте тестовые файлы».

### Создание и использование Vault

Создать CodeRef vault очень просто:

```typescript
const createCodeRefVault = async () => {
  const response = await VaultService.postApiVaults({
    requestBody: {
      name: "React Learning Vault",
      type: "coderef",
      physicalPath: "/Users/developer/vaults/react-learning",
      gitUrl: "https://github.com/facebook/react.git"
    }
  });

  // Система автоматически:
  // 1. Клонировать репозиторий React в vault/repos/react
  // 2. Создать каталог docs/ для заметок
  // 3. Сгенерировать метаданные index.yaml
  // 4. Создать файл руководства AGENTS.md

  return response;
};
```

Затем сослаться на этот vault в ИИ-предложении:

```typescript
const proposal = composeProposalChiefComplaint({
  chiefComplaint: "Помоги мне проанализировать механизм параллельного рендеринга React",
  repositories: [
    { id: "react", gitUrl: "https://github.com/facebook/react.git" }
  ],
  vaults: [
    {
      id: "react-learning",
      name: "React Learning Vault",
      type: "coderef",
      physicalPath: "/vaults/react-learning",
      accessType: "read"  // ИИ может только читать, не может изменять
    }
  ],
  quickRequestText: "Сосредоточьтесь на архитектуре fiber и реализации scheduler"
});
```

### Типичные сценарии использования

**Сценарий 1: Систематическое изучение проектов с открытым исходным кодом**

Создайте CodeRef vault, управляйте целевым репозиторием через Git-сабмодули, записывайте учебные заметки в каталоге `docs/`. ИИ может одновременно доступать код и заметки, обеспечивая более точный анализ. Заметки, написанные при изучении определённого модуля, ИИ будет автоматически ссылаться при последующем анализе соответствующего кода — как если бы «ассистент» запомнил предыдущие размышления.

**Сценарий 2: Повторное использование базы заметок Obsidian**

Если вы уже используете Obsidian для управления заметками, просто зарегистрируйте существующий vault в HagiCode. ИИ может напрямую доступать базу знаний, не требуя ручного копирования и вставки. Эта функция особенно полезна — у многих людей есть накопленные за годы базы заметок, после подключения ИИ сможет «прочитать» систему знаний.

**Сценарий 3: Межпроектное повторное использование знаний**

Несколько ИИ-предложений могут ссылаться на один и тот же vault, реализуя межпроектное повторное использование знаний. Например, если создать «vault для изучения шаблонов проектирования» с записями заметок и примеров кода для различных шаблонов проектирования, ИИ может ссылаться на содержимое этого vault при анализе любого проекта — знание не нужно накапливать повторно.

### Механизм безопасности путей

Система строго проверяет пути для предотвращения атак обхода пути:

```csharp
private static string ResolveFilePath(string vaultRoot, string relativePath)
{
    var rootPath = EnsureTrailingSeparator(Path.GetFullPath(vaultRoot));
    var combinedPath = Path.GetFullPath(Path.Combine(rootPath, relativePath));
    if (!combinedPath.StartsWith(rootPath, StringComparison.OrdinalIgnoreCase))
    {
        throw new BusinessException(VaultRelativePathTraversalCode,
            "Vault file paths must stay inside the registered vault root.");
    }
    return combinedPath;
}
```

Это обеспечивает, что все файловые операции выполняются в пределах корневого каталога vault, предотвращая доступ по вредоносным путям. Безопасность здесь нельзя недооценивать — ИИ-ассистент должен оперировать файловой системой, и необходимо чётко обозначить границы.

## Меры предосторожности

При использовании системы HagiCode Vault есть несколько моментов, на которые нужно обратить особое внимание:

1. **Безопасность путей**: Убедитесь, что пользовательский путь находится в разрешённом диапазоне, иначе система откажет в операции. Это для предотвращения ошибочных операций и потенциальных рисков безопасности.

2. **Управление Git-сабмодулями**: Для CodeRef vault рекомендуется использовать Git-сабмодули, а не напрямую копировать код. Преимущества обсуждались ранее — поддержание синхронизации, экономия места. Однако сабмодули имеют свой собственный способ использования, и при первом использовании может потребоваться ознакомиться.

3. **Ограничения на предварительный просмотр файлов**: Система ограничивает размер файла (256 КБ) и количество (500 штук), сверхбольшие файлы нужно обрабатывать партиями. Это ограничение связано с соображениями производительности; если встречаются сверхбольшие файлы, можно вручную разбить их или обработать другим способом.

4. **Диагностическая информация**: Создание vault возвращает диагностическую информацию, которую можно использовать для отладки при сбое. При возникновении проблем сначала посмотрите на диагностическую информацию — в большинстве случаев можно найти подсказки.

## Заключение

Система Vault HagiCode по сути решает простую, но глубокую проблему: как позволить ИИ-ассистенту понимать и использовать локальные ресурсы знаний.

Благодаря единому уровню абстракции хранилища, стандартизированной структуре каталогов, автоматическому внедрению контекста реализован способ управления знаниями «одноразовая регистрация, повсеместное повторное использование». После создания vault, будь то учебные заметки, репозитории кода или документы, ИИ может автоматически доступать и понимать их.

Улучшение опыта, которое приносит этот дизайн, очевидно. Больше не нужно вручную копировать фрагменты кода и повторно объяснять фоновую информацию — ИИ-ассистент, как коллега, который действительно понимает ситуацию с проектом, может предоставить более ценную помощь на основе существующих знаний.

Система Vault, которой делится эта статья, — это решение, которое было получено в процессе реального преодоления проблем и оптимизации при разработке HagiCode. Если вы считаете этот дизайн ценным, это указывает на неплохие инженерные навыки — тогда и сам HagiCode заслуживает внимания.
