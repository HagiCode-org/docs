## Практика автоматизации публикации Windows-приложений в Microsoft Store

> Как реализовать полную автоматизацию процесса от парсинга версий до публикации в Store, избавившись от трудоёмких ручных операций.

## Предыстория

Если у вас есть Electron-приложение, которое вы хотите опубликовать в Microsoft Store, вы, вероятно, столкнётесь с такой проблемой: Store не поддерживает раздельную установку — вам нужно упаковать десктопное приложение и серверную нагрузку в один полный пакет AppX/MSIX.

С этим ещё можно смириться. Но проблемы не заканчиваются на этом. При каждом выпуске новой версии вам нужно:

1. Проверить обновление версий десктопа и сервера
2. Выгрузить код из соответствующего тега
3. Скачать и внедрить серверную нагрузку
4. Собрать пакет MSIX
5. Вручную загрузить в Microsoft Store
6. Настроить информацию о магазине и ценообразование

Если каждый шаг выполнять вручную, это слишком утомительно. Кроме того, легко допустить ошибку — не всегда вспомнишь, какой шаг был выполнен, а какой нет.

На самом деле это не вина никого — ручные операции по своей природе подвержены ошибкам. Но мы просто не хотели каждый раз проходить через это, поэтому решили彻底 решить проблему — автоматизировать весь процесс.

## Анализ технической архитектуры

Эта проблема на самом деле涉及到 координацию нескольких технических уровней. Мы можем разделить её на пять уровней:

### Уровень координации версий

Сначала нужно определить, когда требуется выпустить новую версию. Нам нужно извлечь из индекса Azure (blob-хранилище, которое мы используем для хранения артефактов сборки) последние версии десктопных и серверных компонентов, а затем определить, нужно ли генерировать новый пакет для Store.

Это как если спросить себя: пора ли делать это сейчас?

### Уровень управления рабочим пространством

Сборка AppX зависит от конфигурации на уровне исходного кода и структуры среды выполнения, поэтому нельзя просто перепаковать существующие артефакты сборки. Нам нужно выгрузить код из определённого тега в репозитории десктопа, чтобы гарантировать правильность состояния исходного кода, используемого для сборки.

Ведь если исходный код неверный, все последующие усилия будут напрасны.

### Уровень упаковки среды выполнения

Это основная часть. Нам нужно внедрить серверную нагрузку в структуру упаковки десктопного приложения. Таким образом при запуске приложение обнаружит упакованную среду выполнения и перейдёт в режим Steam (автономный режим).

Если этот шаг выполнить неправильно, все предыдущие усилия будут напрасны.

### Уровень вывода сборки

Использование electron-builder для генерации пакета MSIX, соответствующего требованиям Store. Этот шаг нужно выполнять в среде Windows, так как для сборки AppX требуется Windows SDK.

Некоторые вещи нужно делать в правильном месте, в другом они не сработают.

### Уровень публикации и распространения

Наконец, публикация в GitHub Releases и Microsoft Store. GitHub Release служит для резервного копирования и отслеживания версий, а Microsoft Store ориентирован на конечных пользователей.

Всё готово, остался только последний рывок.

│         │                       │                          │
│         ▼                       ▼                          │
│  ┌─────────────────┐    ┌─────────────────┐                │
│  │ skip_summary    │    │   publish       │                │
│  │ (отчёт о пропуске)       │    │   (публикация)         │                │
│  └─────────────────┘    └────────┬────────┘                │
│                                 │                          │
│                                 ▼                          │

Весь процесс управляется GitHub Actions и может запускаться по расписанию (каждые 4 часа) или вручную. При ручном запуске можно указать конкретные версии десктопа и сервера или принудительно пересобрать.

Собственно, один раз в 4 часа — это не слишком часто,毕竟 обновления кода всегда бывают быстрее или медленнее, просто так надёжнее.

## Детали реализации

### 1. Парсинг плана сборки

Первый шаг — определить, нужно ли собирать новую версию. Нам нужно извлечь текущие версии десктопных и серверных компонентов, а затем проверить, существует ли уже соответствующий релиз.

```javascript
// scripts/resolve-dispatch-build-plan.mjs
export async function resolveDispatchBuildPlan({
  eventName,
  eventPayload,
  desktopAzureSasUrl,
  serverAzureSasUrl,
}) {
  // Парсинг входных данных триггера
  const trigger = normalizeTriggerInputs({ eventName, eventPayload });
  
  // Извлечение версий десктопа и сервера из индекса Azure
  const [desktopRelease, serverRelease] = await Promise.all([
    resolveIndexRelease({ 
      azureSasUrl: desktopAzureSasUrl,
      platformId: 'win-x64' 
    }),
    resolveIndexRelease({ 
      azureSasUrl: serverAzureSasUrl,
      platformId: 'win-x64' 
    })
  ]);
  
  // Генерация тега релиза
  const desktopTag = normalizeGitTag(desktopRelease.version);
  const releaseTag = deriveStoreReleaseTag(desktopRelease.version, serverRelease.version);
  // Например: desktop-v1.2.3-server-v4.5.6
  
  // Проверка существования (избегание дублирования сборки)
  const existingRelease = await findReleaseByTag(packerRepository, releaseTag);
  const shouldBuild = !existingRelease || trigger.forceRebuild;
  
  return {
    release: { tag: releaseTag, exists: Boolean(existingRelease) },
    build: { shouldBuild, skipReason },
    upstream: { desktop: { tag: desktopTag }, server }
  };
}
```

Ключевой момент этого шага — логика пропуска: если комбинация тех же версий уже собрана, нет необходимости запускать полный процесс сборки снова. Это экономит затраты и время CI.

Повторять одно и то же, наверное, не имеет смысла. Ведь время и ресурсы ограничены.

### 2. Подготовка рабочего пространства

После определения необходимости сборки нужно подготовить чистую среду сборки. Мы используем git worktree для выгрузки кода из определённого тега.

```javascript
// scripts/prepare-packaging-workspace.mjs
export async function preparePackagingWorkspace({
  planPath,
  platformId,
  workspacePath,
  desktopSourcePath
}) {
  // Использование git worktree для выгрузки десктопного кода из определённого тега
  await runCommand('git', [
    '-C', resolvedDesktopSourcePath,
    'worktree', 'add', '--detach',
    desktopWorkspace,
    `refs/tags/${plan.upstream.desktop.tag}`
  ]);
  
  // Проверка рабочего пространства
  const validation = await validateDesktopWorkspace({
    desktopWorkspace,
    storePackageConfig
  });
  
  // Создание манифеста рабочего пространства
  const workspaceManifest = {
    desktopWorkspace,
    runtimeInjectionRoot: validation.runtimeRoot,
    desktopTag: plan.upstream.desktop.tag,
    desktopRef,
  };
  
  return workspaceManifest;
}
```

Преимущество использования worktree в том, что это не влияет на основную рабочую директорию, сборки могут выполняться параллельно, а после завершения сборки легко очиститься.

Ведь никто не хочет, чтобы основная рабочая директория была в беспорядке из-за сборки.

### 3. Внедрение серверной нагрузки

Этот шаг загружает и внедряет серверную нагрузку в правильное место.

```javascript
// scripts/stage-server-payload.mjs
export async function stageServerPayload({
  planPath,
  workspacePath,
  platformId,
  azureSasUrl
}) {
  // Загрузка серверной нагрузки из индекса Azure
  const assetSource = resolveAssetDownloadUrl({ asset, sasUrl: azureSasUrl });
  await downloadFromSource({ 
    sourceUrl: assetSource, 
    destinationPath: downloadPath 
  });
  
  // Распаковка и проверка
  await extractArchive(downloadPath, extractionPath);
  const runtimeRoot = await resolveRuntimeRoot(extractionPath);
  const validation = await validateServerPayloadRoot(runtimeRoot, platformId);
  
  // Внедрение в структуру среды выполнения упаковки
  // Целевой путь — resources/portable-fixed/current
  // Этот путь будет отображён в AppX как extra/portable-fixed/current
  await copyDir(runtimeRoot, targetPath);
}
```

Ключевой момент — отображение путей: `resources/portable-fixed/current` будет упакован в `extra/portable-fixed/current` в AppX, так что при запуске приложение обнаружит локальную среду выполнения и перейдёт в автономный режим.

Пути — вещь такая, где любая ошибка недопустима. Малейшее отклонение — и ничего не найдётся.

### 4. Сборка MSIX

С подготовленным рабочим пространством и серверной нагрузкой можно собрать пакет MSIX.

```javascript
// scripts/build-appx.mjs
export async function buildAppx({
  planPath,
  workspacePath,
  platformId
}) {
  // Генерация переопределения конфигурации electron-builder для Store
  const overlayConfig = await writeStoreElectronBuilderConfig({
    desktopWorkspace: workspaceManifest.desktopWorkspace,
    sourceConfigPath: storePackageConfig.desktop.electronBuilderConfigPath,
    outputConfigPath: 'electron-builder.store.yml'
  });
  
  // Запуск команды сборки десктопа
  await runShellCommand(
    buildDesktopStoreCommand(overlayConfig.outputPath, desktopScripts),
    workspaceManifest.desktopWorkspace
  );
  
  // Сбор выходных данных MSIX
  const storeOutputs = await findStoreOutputs(pkgDirectory);
  const artifactPath = path.join(
    workspaceManifest.outputDirectory, 
    artifactFileName
  );
  await copySingleFile(primaryOutput, artifactPath);
}
```

Конфигурация electron-builder должна включать метаданные, специфичные для Store, такие как идентификатор пакета, отображаемое имя издателя и т.д. Эта информация будет использоваться при отправке в Store.

Если конфигурация неверна, пакет не соберётся. Тут нечего добавить.

### 5. Публикация в GitHub

После завершения сборки нужно опубликовать артефакты в GitHub Releases.

```javascript
// scripts/publish-release.mjs
export async function publishRelease({
  planPath,
  artifactsDir,
  outputDir,
  forceDryRun
}) {
  // Создание манифеста артефактов публикации
  const publicationArtifacts = await buildPublicationArtifacts({
    plan,
    artifactsDir,
    outputDir
  });
  
  if (!dryRun) {
    // Создание или обновление GitHub Release
    const releaseResult = await upsertReleaseNotes(
      plan.release.repository,
      plan.release.tag,
      token,
      { name, body }
    );
    
    // Загрузка активов
    for (const upload of publicationArtifacts.uploads) {
      await uploadReleaseAsset({
        release: releaseResult.release,
        filePath: upload.filePath,
        fileName: upload.fileName
      });
    }
  }
}
```

GitHub Release служит для отслеживания версий и резервного копирования — даже если публикация в Store не удастся, артефакты сборки сохранятся.

Всегда хорошо иметь запасной вариант. Вдруг когда-нибудь понадобится откатиться?

### 6. Публикация в Microsoft Store

Последний шаг — публикация в магазине с помощью Microsoft Store CLI.

```yaml
# .github/workflows/package-release.yml
publish_store:
  runs-on: windows-latest
  steps:
    - name: Configure Microsoft Store CLI
      uses: microsoft/microsoft-store-apppublisher@v1.2
    
    - name: Publish MSIX packages to Store
      shell: pwsh
      run: |
        msstore reconfigure --tenantId $env:AZURE_AD_TENANT_ID ...
        msstore publish "$($package.FullName)" -id $env:MICROSOFT_STORE_PRODUCT_ID
```

Этот шаг нужно выполнять на Windows runner, так как Microsoft Store CLI требует среду Windows.

Некоторые вещи нужно делать в правильном месте, в другой среде они не сработают.

## Практическое руководство

### Файл конфигурации

Файл конфигурации пакета Store определяет идентификатор пакета и параметры сборки:

```json

    "publisherDisplayName": "newbe36524",
    "publisher": "CN=8B6C8A94-AAE5-4C8B-9202-A29EA42B042F",
    "identityName": "newbe36524.Hagicode",
    "backgroundColor": "transparent",
    "languages": ["en-US", "zh-CN", "zh-Hant"]
  },
  "supportedWindowsTargets": ["win-x64"],
  "desktop": {

    "buildScript": "build:appx",

  }

```

Конфигурация — вещь такая, что изменить один раз уже достаточно головной боли, поэтому лучше сразу сделать правильно.

### Триггеры рабочего процесса

Можно запускать по расписанию или вручную:

```yaml
on:
  schedule:
    - cron: '0 */4 * * *'  # Запуск каждые 4 часа
  workflow_dispatch:
    inputs:
      desktop_version:
        description: 'Селектор версии Desktop'

        description: 'Принудительная пересборка даже если существует'
        type: boolean
      dry_run:
        description: 'Только сборка, без публикации'
        type: boolean
```

Автоматически или вручную — неважно, главное, чтобы в итоге работало.

### Важные замечания

1. **Требования к Git Tag**: В репозитории десктопа должны быть теги, соответствующие версии релиза (например, v1.2.3), иначе сборка не удастся

2. **Azure SAS URL**: Нужно настроить переменные среды `DESKTOP_AZURE_SAS_URL` и `SERVER_AZURE_SAS_URL` для доступа к индексу

3. **Учётные данные Microsoft Store**: Нужно настроить следующие ключи:
   - `AZURE_AD_APPLICATION_CLIENT_ID`
   - `AZURE_AD_APPLICATION_SECRET`
   - `AZURE_AD_TENANT_ID`
   - `SELLER_ID`
   - `MICROSOFT_STORE_PRODUCT_ID`

4. **Проверка среды выполнения**: Серверная нагрузка должна содержать необходимые файлы среды выполнения, иначе упаковка не удастся

5. **Windows Runner**: Сборка AppX и публикация в Store должны выполняться на Windows runner

6. **Обнаружение дубликатов**: При плановом запуске проверяется наличие того же тега релиза, чтобы избежать ненужных сборок

Эти ямы мы, наверное, все прошли один раз, поэтому написали для напоминания. Ведь никто не хочет повторять ошибки?

## Заключение

Автоматическая публикация в Windows Store кажется сложной, но после разбиения на независимые шаги каждый из них несложен. Ключевые моменты:
- Использовать git worktree для управления средой сборки
- Внедрять серверную нагрузку в правильный путь
- Использовать electron-builder для генерации пакетов MSIX
- Использовать Store CLI для финальной публикации

Это решение стабильно работает в проекте HagiCode уже несколько месяцев, в основном достигнув цели «полной автоматизации» — за исключением первой настройки учётных данных и параметров, последующие релизы версий не требуют вмешательства человека.

На самом деле многие дела такие — кажутся сложными, а на деле всё просто. Нужно только немного терпения и попыток.

Если вы тоже занимаетесь подобной автоматизацией, надеюсь, эта статья даст вам некоторые идеи. Конечно, у каждого проекта свои особенности, возможно, потребуется настроить решение в соответствии с конкретными потребностями.
