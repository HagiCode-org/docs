## Элегантная реализация新手-пользователя в проектах React: опыт HagiCode с driver.js

> Когда пользователь впервые открывает ваш продукт, действительно ли он знает, с чего начать? Эта статья рассказывает о том, как мы использовали driver.js для新手-пользователя в проекте HagiCode.

## 背景

Вы когда-нибудь сталкивались с такой ситуацией: новый пользователь регистрируется в вашем продукте, открывает страницу и выглядит растерянным, оглядывается по сторонам, не зная, куда нажать и что делать. Как разработчики, мы всегда думаем, что пользователи будут "исследовать сами", ведь любопытство человека безгранично. Но реальность такова — большинство пользователей тихо уйдут через несколько минут, не найдя входа, как будто история началась внезапно и закончилась естественно.

新手-пользователя — это важное средство решения этой проблемы, хотя её реализация не так проста. Хорошая система наведения должна:

- Уметь точно定位-элементы страницы и выделять их
- Поддерживать многошаговые наводящие процессы
- Уметь запоминать выбор пользователя (завершено/пропущено)
- Не влиять на производительность страницы и нормальное взаимодействие
- Иметь清晰ную структуру кода, легко поддерживать

В процессе разработки HagiCode мы столкнулись с теми же проблемами. HagiCode — это проект AI-ассистента для кода, основной рабочий процесс — "пользователь создаёт предложение → AI генерирует план → пользователь审核 → AI выполняет" — такой процесс OpenSpec. Для пользователей, впервые столкнувшихся с этой концепцией, этот процесс совершенно новый, поэтому необходимо хорошее наведение, чтобы помочь им быстро начать работу. Ведь новое всегда требует времени.

## О HagiCode

Решение, описанное в этой статье, основано на нашем практическом опыте в проекте [HagiCode](https://hagicode.com). HagiCode — это AI-ассистент для кода на основе Claude, который помогает разработчикам более эффективно выполнять задачи с помощью рабочего процесса OpenSpec. Вы можете просмотреть наш открытый код на [GitHub](https://github.com/HagiCode-org/site).

## Почему выбран driver.js

На этапе выбора технологий мы оценили несколько основных библиотек наведения. У каждой есть свои особенности:

- **Intro.js**：мощный функционал, но большой объём,相对-сложно настроить стили
- **Shepherd.js**：хорошее проектирование API, но для нашего сценария немного "тяжеловесно"
- **driver.js**：лёгкий, лаконичный, интуитивный API, поддерживает экосистему React

В итоге мы выбрали driver.js, особых причин нет, в основном из следующих соображений:

1. **Лёгкий вес**：небольшой объём основной библиотеки, не значительно увеличивает объём пакета
2. **Простой API**：ясные и интуитивные параметры конфигурации, быстрое начало
3. **Гибкость**：поддержка пользовательского定位ения, стилей и поведения взаимодействия
4. **Динамический импорт**：может загружаться по требованию, не влияет на производительность первого экрана

Выбор технологий — на самом деле нет лучшего, только наиболее подходящий.

## Техническая реализация

### Основная конфигурация

Конфигурация driver.js очень интуитивна, ниже приведена основная конфигурация в проекте HagiCode:

```typescript
import { driver } from 'driver.js';
import 'driver.js/dist/driver.css';

const newConversationDriver = driver({
  allowClose: true,           // разрешить пользователю关闭 наведение
  animate: true,              // включить эффекты анимации
  overlayClickBehavior: 'close', // клик по遮罩层关闭 наведение
  disableActiveInteraction: false, // сохранить элементы интерактивными
  showProgress: false,        // не отображать индикатор прогресса (у нас есть пользовательское управление прогрессом)
  steps: guideSteps           // массив наводящих шагов
});
```

Соображения за этой конфигурацией:

- `allowClose: true` — уважать выбор пользователя, не принудительно завершать наведение, ведь насильно не мил
- `disableActiveInteraction: false` — некоторые шаги требуют реальных действий пользователя (например, ввод текста), поэтому нельзя отключать взаимодействие
- `overlayClickBehavior: 'close'` — дать пользователю быстрый способ выхода

### Управление состоянием

Persistency состояния наведения — это ключевой момент — мы не хотим, чтобы при каждом обновлении страницы наведение повторялось, это довольно раздражает. HagiCode использует localStorage для управления состоянием наведения:

```typescript
export type GuideState = 'pending' | 'dismissed' | 'completed';

export interface UserGuideState {
  session: GuideState;
  detailGuides: Record<string, GuideState>;
}

// чтение состояния
export const getUserGuideState = (): UserGuideState => {
  const state = localStorage.getItem('userGuideState');
  return state ? JSON.parse(state) : { session: 'pending', detailGuides: {} };
};

// обновление состояния
export const setUserGuideState = (state: UserGuideState) => {
  localStorage.setItem('userGuideState', JSON.stringify(state));
};
```

Мы определили три состояния:

- `pending`：наведение в процессе, пользователь ещё не завершил и не пропустил
- `dismissed`：пользователь主动 закрыл наведение
- `completed`：пользователь завершил все шаги

Для наведения на странице деталей предложения мы также поддерживаем более детальное отслеживание состояния (через словарь `detailGuides`), потому что предложение может проходить несколько этапов (черновик,审核, выполнение завершено), на каждом этапе требуется разное наведение. Ведь состояние вещей всегда меняется.

### 定ение целевых элементов

driver.js использует CSS-селекторы для定位ения целевых элементов. HagiCode использует соглашение: пользовательский атрибут `data-guide` для标记 наводящих целей:

```typescript
const steps = [
  {
    element: '[data-guide="launch"]',
    popover: {
      title: 'Начать新ный диалог',
      description: 'Нажмите здесь, чтобы создать新ную сессию диалога...'
    }
  }
];
```

Использование в компоненте:

```typescript
<button data-guide="launch" onClick={handleLaunch}>
  Создать диалог
</button>
```

Преимущества этого подхода:

- Избегать конфликтов с бизнес-стилями
- Ясная семантика, сразу видно, что этот элемент связан с наведением
- Удобно для унифицированного управления и поддержки

### Оптимизация динамического импорта

Поскольку функция наведения нужна только в определённых сценариях (например, первый доступ нового пользователя), мы используем динамический импорт для оптимизации производительности начальной загрузки:

```typescript
const initNewUserGuide = async () => {
  // динамический импорт driver.js
  const { driver } = await import('driver.js');
  await import('driver.js/dist/driver.css');

  // инициализация наведения
  const newConversationDriver = driver({
    // ...конфигурация
  });

  newConversationDriver.drive();
};
```

Таким образом driver.js и его файлы стилей загружаются только при необходимости, не влияя на производительность первого экрана. Ведь никто не хочет платить за то, что временно не используется.

## Дизайн наводящего процесса

HagiCode реализует два наводящих пути, покрывающих основные сценарии использования пользователями.

### Наведение по сессиям (10 шагов)

Это наведение помогает пользователю выполнить весь процесс от создания диалога до提交 первого полного предложения:

1. **launch** — запуск наведения, введение кнопки "Создать диалог"
2. **compose** — наведение пользователя вводить запрос в поле ввода
3. **send** — наведение нажать кнопку отправки
4. **proposal-launch-readme** — наведение создать предложение README
5. **proposal-compose-readme** — наведение редактировать содержимое запроса README
6. **proposal-submit-readme** — наведение提交 предложение README
7. **proposal-launch-agents** — наведение создать предложение AGENTS.md
8. **proposal-compose-agents** — наведение редактировать запрос AGENTS.md
9. **proposal-submit-agents** — наведение提交 предложение AGENTS.md
10. **proposal-wait** — объяснение, что AI обрабатывает, подождите

Идея этого наведения: через две реальные задачи создания предложений (README и AGENTS.md) позволить пользователям лично体验 основной рабочий процесс HagiCode. Ведь纸上得来终觉浅, 绝知此事要躬行.

### Наведение по деталям предложения (3 шага)

Когда пользователь входит на страницу деталей предложения, соответствующее наведение вызывается в зависимости от текущего состояния предложения:

1. **drafting** (этап черновика) — наведение пользователя просмотреть план, созданный AI
2. **reviewing** (этап审核) — наведение пользователя выполнить план
3. **executionCompleted** (этап завершения) — наведение пользователя архивировать план

Особенность этого наведения — управление состоянием — динамическое определение, какой наводящий шаг отображать, на основе фактического состояния предложения. Вещи всегда меняются, наведение тоже должно меняться.

### Механизм повтора рендеринга элементов

В приложениях React целевые элементы наведения могут быть ещё не отрендерены (например, ожидание загрузки асинхронных данных). Для обработки этой ситуации HagiCode реализует механизм повтора:

```typescript
const waitForElement = (selector: string, maxRetries = 10, interval = 100) => {
  let retries = 0;
  return new Promise<HTMLElement>((resolve, reject) => {
    const checkElement = () => {
      const element = document.querySelector(selector) as HTMLElement;
      if (element) {
        resolve(element);
      } else if (retries < maxRetries) {
        retries++;
        setTimeout(checkElement, interval);
      } else {
        reject(new Error(`Element not found: ${selector}`));
      }
    };
    checkElement();
  });
};
```

Перед инициализацией наведения вызовите эту функцию, чтобы убедиться, что целевой элемент уже существует. Иногда стоит подождать.

## Резюме лучших практик

На основе практического опыта HagiCode, здесь поделюсь несколькими ключевыми лучшими практиками:

### 1. Наведение должно быть "отменяемым"

Не принуждайте пользователя завершать наведение. Некоторые пользователи — исследовательского типа, они предпочитают сами исследовать. Предоставьте четкую кнопку "пропустить" и запомните выбор пользователя, в следующий раз не беспокоить. Ведь, красивые вещи или люди, не обязательно обладать, пока она красивая, хорошо сам смотреть на её красоту.

### 2. Содержание наведения должно быть кратким и сильным

Каждый наводящий шаг должен фокусироваться на единственной цели:

- **Title**：краткий и четкий, не более 10 символов
- **Description**：прямо к сути, расскажите пользователю "что это" и "зачем использовать"

Избегайте длинных объяснений — внимание пользователя на этапе наведения очень ограничено. Много слов, наоборот, никто не хочет смотреть.

### 3. Селекторы должны быть стабильными

Используйте стабильный, не часто меняющийся способ标记ения элементов. Пользовательский атрибут `data-guide` — хороший выбор, избегайте зависимости от имён классов или структуры DOM, потому что они легко меняются при рефакторинге. Код всегда меняется, но некоторые вещи должны по возможности сохранять стабильность.

### 4. Тестируйте своё наведение

HagiCode написал полные тестовые случаи для функции наведения:

```typescript
describe('NewUserConversationGuide', () => {
  it('должен правильно инициализировать состояние наведения', () => {
    const state = getUserGuideState();
    expect(state.session).toBe('pending');
  });

  it('должен правильно обновить состояние наведения', () => {
    setUserGuideState({ session: 'completed', detailGuides: {} });
    const state = getUserGuideState();
    expect(state.session).toBe('completed');
  });
});
```

Тесты могут гарантировать, что при рефакторинге кода не случайно не разрушить функцию наведения. Ведь никто не хочет, чтобы изменение кода сломало предыдущую функцию.

### 5. Оптимизация производительности

- Используйте динамический импорт для отложенной загрузки библиотеки наведения
- Избегайте инициализации логики наведения после того, как пользователь завершил наведение
- Рассмотрите влияние на производительность анимации наведения, на低端 устройствах можно отключить анимацию

Производительность — это как жизнь, где нужно экономить, там нужно экономить.

## Резюме

Наведение нового пользователя — важный элемент для повышения пользовательского опыта продукта. В проекте HagiCode мы использовали driver.js для создания полной системы наведения, покрывающей весь рабочий процесс от создания сессии до выполнения предложения.

Через эту статью мы хотим передать основные идеи:

1. **Выбор технологий должен соответствовать требованиям**：driver.js не самый мощный, но для нас самый подходящий
2. **Управление состоянием очень важно**：используйте localStorage для persistency состояния наведения, избегайте повторного беспокойства пользователя
3. **Дизайн наведения должен быть сфокусированным**：каждый шаг решает одну проблему, не будьте жадными
4. **Структура кода должна быть четкой**：разделение конфигурации наведения, управления состоянием и логики UI, удобно поддерживать
