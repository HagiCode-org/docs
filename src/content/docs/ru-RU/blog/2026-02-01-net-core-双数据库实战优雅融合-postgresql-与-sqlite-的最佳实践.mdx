## .NET Core двойная база данных на практике: позвольте PostgreSQL и SQLite сосуществовать мирно

> При создании современных приложений мы часто сталкиваемся с выбором: среда разработки хочет быть легкой и удобной, а производственная среда требует высокой concurrency и высокой доступности. В этой статье мы поделимся, как элегантно одновременно поддерживать PostgreSQL и SQLite в проектах .NET Core, реализуя лучшие практики "SQLite для разработки, PG для производства".


## Контекст

В разработке программного обеспечения различие сред всегда было одной из проблем, беспокоящих команды разработчиков. На примере создаваемой нами платформы **HagiCode** — это система вспомогательной разработки на основе AI, построенная на ASP.NET Core 10 и React, внутри интегрирована Orleans для распределенного управления состоянием, технологический стек довольно современный и сложный.

На начальном этапе проекта мы столкнулись с типичной инженерной проблемой: разработчики хотят, чтобы локальная среда работала "из коробки", не желая устанавливать и настраивать тяжелую базу данных PostgreSQL; но в производственной среде нам нужно обрабатывать высокую concurrency записи и сложные JSON запросы, в это время легковесный SQLite кажется недостаточно мощным.

Как при сохранении единой кодовой базы позволить приложению использовать портативность SQLite как клиентское программное обеспечение, а также использовать мощную производительность PostgreSQL как enterprise сервис? Это основной вопрос, который мы рассмотрим в этой статье.

## Основное содержание один: архитектурный дизайн и единая абстракция

Для реализации поддержки двойной базы данных в .NET Core основная идея — "зависеть от абстракции, а не от конкретной реализации". Нам нужно отделить право выбора базы данных от бизнес-кода и передать его уровню конфигурации.

### Идея дизайна

1.  **Единый интерфейс**: Вся бизнес-логика должна зависеть от базового класса `DbContext` или пользовательского интерфейса, а не от конкретного `PostgreSqlDbContext`.
2.  **На основе конфигурации**: Через элементы конфигурации в `appsettings.json` при запуске приложения динамически решать, какой поставщик базы данных загружать.
3.  **Изоляция характеристик**: Адаптировать специфичные для PostgreSQL функции (например, JSONB), обеспечить возможность снижения функционала в SQLite.

### Реализация кода: динамическая конфигурация контекста

В `Program.cs` ASP.NET Core мы не должны жестко кодировать `UseNpgsql` или `UseSqlite`. Напротив, мы должны читать конфигурацию для динамического принятия решения.

Сначала определим класс конфигурации:

```csharp
public class DatabaseSettings
{
    public const string SectionName = "Database";
    
    // Тип базы данных: PostgreSQL или SQLite
    public string DbType { get; set; } = "PostgreSQL"; 
    
    // Строка подключения
    public string ConnectionString { get; set; } = string.Empty;
}
```

Затем, в `Program.cs` регистрируем сервисы в соответствии с конфигурацией:

```csharp
// Чтение конфигурации
var databaseSettings = builder.Configuration.GetSection(DatabaseSettings.SectionName).Get<DatabaseSettings>();

// Регистрация DbContext
builder.Services.AddDbContext<ApplicationDbContext>(options =>
{
    if (databaseSettings?.DbType?.ToLower() == "sqlite")
    {
        // Конфигурация SQLite
        options.UseSqlite(databaseSettings.ConnectionString);
        
        // Обработка ограничений конкурентной записи SQLite
        // Примечание: в производственной среде рекомендуется включить режим WAL для повышения производительности конкурентности
    }
    else
    {
        // Конфигурация PostgreSQL (по умолчанию)
        options.UseNpgsql(databaseSettings.ConnectionString, npgsqlOptions =>
        {
            // Включение поддержки JSONB, это очень полезно при обработке записей AI диалогов
            npgsqlOptions.UseJsonNet(); 
        });
        
        // Настройка стратегии переподключения пула подключений
        options.EnableRetryOnFailure(3);
    }
});
```

## Основное содержание два: обработка различий и стратегия миграции

Хотя PostgreSQL и SQLite поддерживают стандарт SQL, в конкретных характеристиках и поведении существуют значительные различия. Если не обработать эти различия должным образом, скорее всего возникнет неловкая ситуация "локально работает, после выкладки ошибка".

### 1. Обработка типа JSON

В HagiCode нам нужно хранить большое количество промптов и AI метаданных, что обычно связано со столбцами JSON.
- **PostgreSQL**: Имеет нативный тип `JSONB`, отличная производительность запросов.
- **SQLite**: Нет нативного типа JSON (в новых версиях есть расширение JSON1, но в отображении объектов все еще есть различия), обычно хранится как TEXT.

**Решение**:
В сопоставлении сущностей EF Core мы настраиваем его как конвертируемый тип.

```csharp
protected override void OnModelCreating(ModelBuilder modelBuilder)
{
    base.OnModelCreating(modelBuilder);

    // Конфигурация сущности
    modelBuilder.Entity<PromptTemplate>(entity => 
    {
        entity.Property(e => e.Metadata)
              .HasColumnType("jsonb") // PG использует jsonb
              .HasConversion(
                  v => JsonSerializer.Serialize(v, (JsonSerializerOptions)null),
                  v => JsonSerializer.Deserialize<Dictionary<string, object>>(v, (JsonSerializerOptions)null)
              ); 
    });
}
```

При использовании SQLite, хотя `HasColumnType("jsonb")` будет проигнорировано или вызовет предупреждение, но благодаря настройке `HasConversion`, данные будут правильно сериализованы и десериализованы в строки и сохранены в поле TEXT, тем самым обеспечивая совместимость.

### 2. Разделение стратегии миграции

Никогда не пытайтесь заставить один и тот же сценарий Migration одновременно адаптироваться к PG и SQLite. Из-за различий в стратегии генерации первичных ключей, синтаксисе индексов и т.д. это неизбежно приведет к неудаче.

**Рекомендуемая практика**:
Поддерживайте две ветви миграции или проекта. В потоке разработки HagiCode мы обрабатываем это так:

1.  **Этап разработки**: В основном работаем под SQLite. Используем `Add-Migration Init_Sqlite -OutputDir Migrations/Sqlite`.
2.  **Этап адаптации**: После завершения разработки части функционала переключаем строку подключения на PostgreSQL, выполняем `Add-Migration Init_Postgres -OutputDir Migrations/Postgres`.
3.  **Автоматизированный скрипт**: Напишите простой PowerShell или Bash скрипт, автоматически применяющий соответствующую миграцию в соответствии с текущей переменной окружения.

```bash
# Псевдокод простой логики развертывания
if [ "$DATABASE_PROVIDER" = "PostgreSQL" ]; then
    dotnet ef database update --project Migrations.Postgres
else
    dotnet ef database update --project Migrations.Sqlite
fi
```

## Основное содержание три: резюме практического опыта HagiCode

В процессе рефакторинга **HagiCode** от одной базы данных к поддержке двойной базы данных мы наступили на некоторые ямы, а также подвели некоторые ключевые опыты, надеясь помочь всем их избежать.

### 1. Различие конкурентности и транзакций

PostgreSQL — это архитектура клиент-сервер, поддерживает высокую конкурентность записи, очень мощные уровни изоляции транзакций. А SQLite использует механизм файловых блокировок, операции записи блокируют весь файл базы данных (если не включен режим WAL).

**Рекомендация**:
При написании бизнес-логики, связанной с частыми записями (например, реальное сохранение статуса редактирования пользователя), обязательно учитывайте механизм блокировок SQLite. При проектировании модуля OpenSpec в **HagiCode** мы ввели механизм "предварительного слияния записи", уменьшив частоту прямой записи в базу данных, тем самым поддерживая высокую производительность в обеих базах данных.

### 2. Управление жизненным циклом строки подключения

Создание подключения PostgreSQL имеет высокую стоимость, зависит от пула подключений. А подключение SQLite очень легкое, но если своевременно не释放, файловая блокировка может привести к тайм-ауту последующих операций.

В `Program.cs` мы можем делать точную настройку для разных баз данных:

```csharp
if (databaseSettings?.DbType?.ToLower() == "sqlite")
{
    // SQLite: поддержание подключения открытым может повысить производительность, но обратите внимание на файловую блокировку
    options.UseSqlite(connectionString, sqliteOptions =>
    {
        // Установка тайм-аута команды
        sqliteOptions.CommandTimeout(30);
    });
}
else
{
    // PG: использование пула подключений
    options.UseNpgsql(connectionString, npgsqlOptions =>
    {
        npgsqlOptions.MaxBatchSize(100);
        npgsqlOptions.CommandTimeout(30);
    });
}
```

### 3. Важность тестового покрытия

Многие разработчики (включая ранних членов нашей команды) склонны совершать ошибку: запускать модульные тесты только в среде разработки (обычно SQLite).

В конвейере CI/CD HagiCode мы принудительно добавили шаг GitHub Action, чтобы каждый Pull Request проходил интеграционные тесты PostgreSQL.

```yaml
# Пример фрагмента .github/workflows/test.yml
- name: Run Integration Tests (PostgreSQL)
  run: |
    docker-compose up -d db_postgres
    dotnet test --filter "Category=Integration"
```

Это помогло нам перехватить бесчисленное количество Bug, связанных с различиями синтаксиса SQL, чувствительностью к регистру.

## Заключение

Благодаря введению абстрактного уровня и управляемого конфигурацией внедрения зависимостей, в проекте **HagiCode** мы успешно реализовали "двухполосную" работу PostgreSQL и SQLite. Это не только значительно снизило порог входа для новых разработчиков (не нужно устанавливать PG), но и обеспечило надежную гарантию производительности для производственной среды.

Вспомним ключевые моменты:
1.  **Абстракция превыше всего**: Бизнес-код не зависит от конкретной реализации базы данных.
2.  **Разделение конфигурации**: Разработка и производство используют разные `appsettings.json`.
3.  **Разделение миграции**: Не пытайтесь использовать один Migration для всех.
4.  **Снижение характеристик**: В SQLite совместимость имеет приоритет, в PostgreSQL производительность имеет приоритет.

Эта архитектурная модель применима не только к HagiCode, но и к любому проекту .NET, который ищет баланс между легковесной разработкой и тяжелым производством.
