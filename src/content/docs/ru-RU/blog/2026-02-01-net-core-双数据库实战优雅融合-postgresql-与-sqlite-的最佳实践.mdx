## .NET Core: двойная база данных на практике — пусть PostgreSQL и SQLite сосуществуют мирно

> При создании современных приложений мы часто сталкиваемся с выбором: среда разработки хочет быть лёгкой и удобной, а продакшен требует высокой параллельности и доступности. В этой статье описано, как элегантно поддерживать PostgreSQL и SQLite в проекте .NET Core, реализуя лучшую практику «SQLite для разработки, PostgreSQL для продакшена».

## Предпосылки

В разработке программного обеспечения различия между окружениями всегда были одной из самых сложных проблем для инженерных команд. Возьмём платформу **HagiCode**, которую мы создаём: это система AI-ассистированной разработки на базе ASP.NET Core 10 и React с внутренней интеграцией Orleans для распределённого управления состоянием. Стек современный и довольно сложный.

На раннем этапе проекта мы столкнулись с классической инженерной проблемой: разработчики хотели, чтобы локальное окружение работало «из коробки», без необходимости установки и настройки тяжёлой PostgreSQL. Но в продакшене нам нужно было обрабатывать высокопараллельные записи и сложные JSON-запросы, где легковесный SQLite уже не справлялся.

Как сохранить единую кодовую базу, позволяя приложению использовать портативность SQLite как десктопное приложение и одновременно задействовать мощную производительность PostgreSQL как корпоративный сервис? Это основной вопрос, который мы рассматриваем в этой статье.

## О HagiCode

Подход к адаптации двойной базы данных, описанный в этой статье, основан на нашем практическом опыте в проекте **HagiCode**. HagiCode — это платформа разработки следующего поколения, интегрирующая управление AI-промптами и рабочий процесс OpenSpec. Именно для балансировки впечатлений разработчика и стабильности продакшена мы выработали эту проверенную архитектурную модель.

Заходите в наш репозиторий GitHub, чтобы увидеть проект целиком: [HagiCode-org/site](https://github.com/HagiCode-org/site).

## Тема 1: архитектурный дизайн и единая абстракция

Для поддержки двух баз данных в .NET Core ключевая идея — «зависеть от абстракций, а не от конкретных реализаций». Нам нужно отделить выбор базы данных от бизнес-кода и передать решение на уровень конфигурации.

### Подход к проектированию

1.  **Единый интерфейс**: вся бизнес-логика должна зависеть от базового класса `DbContext` или пользовательских интерфейсов, а не от конкретного `PostgreSqlDbContext`.
2.  **Управляемый конфигурацией**: использование элементов конфигурации в `appsettings.json` для динамического определения провайдера базы данных при запуске приложения.
3.  **Изоляция возможностей**: добавление логики адаптации для специфичных функций PostgreSQL (таких как JSONB), чтобы приложение могло корректно деградировать на SQLite.

### Реализация: динамическая конфигурация контекста

В `Program.cs` ASP.NET Core не следует хардкодить `UseNpgsql` или `UseSqlite`. Вместо этого нужно читать конфигурацию и принимать решение динамически.

Сначала определим класс конфигурации:

```csharp
public class DatabaseSettings
{
    public const string SectionName = "Database";

    // Тип базы данных: PostgreSQL или SQLite
    public string DbType { get; set; } = "PostgreSQL";

    // Строка подключения
    public string ConnectionString { get; set; } = string.Empty;
}
```

Затем зарегистрируем сервис в `Program.cs` на основе конфигурации:

```csharp
// Чтение конфигурации
var databaseSettings = builder.Configuration.GetSection(DatabaseSettings.SectionName).Get<DatabaseSettings>();

// Регистрация DbContext
builder.Services.AddDbContext<ApplicationDbContext>(options =>
{
    if (databaseSettings?.DbType?.ToLower() == "sqlite")
    {
        // Конфигурация SQLite
        options.UseSqlite(databaseSettings.ConnectionString);

        // Обработка ограничений параллельной записи SQLite
        // Примечание: в продакшене рекомендуется включить WAL-режим для повышения производительности
    }
    else
    {
        // Конфигурация PostgreSQL (по умолчанию)
        options.UseNpgsql(databaseSettings.ConnectionString, npgsqlOptions =>
        {
            // Включение поддержки JSONB, очень полезно при обработке записей AI-разговоров
            npgsqlOptions.UseJsonNet();
        });

        // Конфигурация стратегии повторных попыток пула соединений
        options.EnableRetryOnFailure(3);
    }
});
```

## Тема 2: обработка различий и стратегия миграций

Хотя PostgreSQL и SQLite поддерживают стандарт SQL, они существенно различаются в специфических возможностях и поведении. Без правильной обработки этих различий легко оказаться в ситуации «работает локально, падает на проде».

### 1. Обработка JSON-типов

В HagiCode нам нужно хранить большое количество промптов и AI-метаданных, что обычно involves JSON-колонки.
- **PostgreSQL**: имеет нативный тип `JSONB` с отличной производительностью запросов.
- **SQLite**: не имеет нативного JSON-типа (в новых версиях есть расширение JSON1, но маппинг объектов различается), данные обычно хранятся как TEXT.

**Решение**:
В маппинге сущностей EF Core настраиваем преобразуемый тип.

```csharp
protected override void OnModelCreating(ModelBuilder modelBuilder)
{
    base.OnModelCreating(modelBuilder);

    modelBuilder.Entity<PromptTemplate>(entity =>
    {
        entity.Property(e => e.Metadata)
              .HasColumnType("jsonb") // PG использует jsonb
              .HasConversion(
                  v => JsonSerializer.Serialize(v, (JsonSerializerOptions)null),
                  v => JsonSerializer.Deserialize<Dictionary<string, object>>(v, (JsonSerializerOptions)null)
              );
    });
}
```

При использовании SQLite `HasColumnType("jsonb")` может быть проигнорирован или вызвать предупреждение. Однако благодаря `HasConversion` данные корректно сериализуются и десериализуются как строки, хранящиеся в TEXT-поле, обеспечивая совместимость.

### 2. Разделение стратегий миграций

Никогда не пытайтесь заставить один набор скриптов Migration работать одновременно для PostgreSQL и SQLite. Различия в стратегиях генерации первичных ключей, синтаксисе индексов и других деталях неизбежно приведут к сбоям.

**Рекомендуемая практика**:
Поддерживайте две ветки или проекта миграций. В рабочем процессе HagiCode это обрабатывается так:

1.  **Этап разработки**: основная работа с SQLite. Используем `Add-Migration Init_Sqlite -OutputDir Migrations/Sqlite`.
2.  **Этап адаптации**: после разработки функции переключаем строку подключения на PostgreSQL и выполняем `Add-Migration Init_Postgres -OutputDir Migrations/Postgres`.
3.  **Скрипты автоматизации**: простой PowerShell или Bash-скрипт для автоматического применения соответствующей миграции на основе переменной окружения.

```bash
# Псевдокод простой логики развёртывания
if [ "$DATABASE_PROVIDER" = "PostgreSQL" ]; then
    dotnet ef database update --project Migrations.Postgres
else
    dotnet ef database update --project Migrations.Sqlite
fi
```

## Тема 3: уроки из практики HagiCode

При рефакторинге **HagiCode** от одной базы данных к двойной поддержке мы набили несколько шишек и собрали важные уроки.

### 1. Различия в параллельности и транзакциях

PostgreSQL использует клиент-серверную архитектуру и поддерживает высокопараллельные записи с мощными уровнями изоляции транзакций. SQLite использует файловые блокировки, поэтому операции записи блокируют весь файл базы данных (если не включён WAL-режим).

**Рекомендация**:
При написании бизнес-логики, включающей частые записи (например, реальное сохранение состояния редактирования пользователя), обязательно учитывайте модель блокировок SQLite. При проектировании модуля OpenSpec-сотрудничества в **HagiCode** мы ввели механизм «слияние перед записью» для снижения частоты прямых записей в БД, что позволило поддерживать хорошую производительность на обеих базах данных.

### 2. Управление жизненным циклом строк подключения

Установление соединения PostgreSQL относительно дорого и зависит от пула подключений. Соединения SQLite очень легковесны, но если их своевременно не освобождать, файловые блокировки могут привести к таймаутам последующих операций.

В `Program.cs` можно точно настроить поведение для каждой базы данных:

```csharp
if (databaseSettings?.DbType?.ToLower() == "sqlite")
{
    // SQLite: поддержание открытых подключений повышает производительность, но следите за файловыми блокировками
    options.UseSqlite(connectionString, sqliteOptions =>
    {
        sqliteOptions.CommandTimeout(30);
    });
}
else
{
    // PG: полное использование пула подключений
    options.UseNpgsql(connectionString, npgsqlOptions =>
    {
        npgsqlOptions.MaxBatchSize(100);
        npgsqlOptions.CommandTimeout(30);
    });
}
```

### 3. Важность покрытия тестами

Многие разработчики (включая ранних членов нашей команды) часто допускают одну ошибку: запускают модульные тесты только в среде разработки (обычно SQLite).

В CI/CD-конвейере HagiCode мы принудительно добавили шаг GitHub Actions, обеспечивающий прохождение интеграционных тестов PostgreSQL для каждого Pull Request.

```yaml
# Пример из .github/workflows/test.yml
- name: Run Integration Tests (PostgreSQL)
  run: |
    docker-compose up -d db_postgres
    dotnet test --filter "Category=Integration"
```

Это помогло нам перехватить бесчисленное количество ошибок, связанных с различиями синтаксиса SQL и чувствительностью к регистру.

## Заключение

Внедрив абстракцию и конфигурационно-управляемое внедрение зависимостей, мы успешно реализовали «двухколейную» систему PostgreSQL и SQLite в проекте **HagiCode**. Это не только значительно снизило порог входа для новых разработчиков (не нужно устанавливать PG), но и обеспечило надёжные гарантии производительности для продакшена.

Краткий итог:
1.  **Абстракция прежде всего**: бизнес-код не должен зависеть от конкретной реализации базы данных.
2.  **Раздельные конфигурации**: разные `appsettings.json` для разработки и продакшена.
3.  **Раздельные миграции**: не пытайтесь использовать один набор Migration для всех случаев.
4.  **Деградация возможностей**: в SQLite приоритет совместимости, в PostgreSQL — производительности.

Эта архитектурная модель подходит не только для HagiCode, но и для любого .NET-проекта, которому нужен баланс между лёгкой разработкой и мощным продакшеном.
