## От 120 секунд до 45 секунд: оптимизация производительности сборки Vite с помощью Worker Threads

> При работе с крупными фронтенд-проектами сборка кода для продакшена часто заставляет ждать мучительно долго. В этой статье мы поделимся, как с помощью Node.js Worker Threads удалось сократить время этапа обфускации кода при сборке Vite со 120 секунд до 45 секунд, и подробно опишем детали реализации и опыт преодоления проблем в проекте HagiCode.

## Предпосылки

В нашей практике фронтенд-инжиниринга, по мере масштабирования проектов, проблемы эффективности сборки становятся всё более заметными. Особенно в процессе сборки для продакшена, для защиты логики исходного кода мы обычно внедряем инструменты обфускации JavaScript (например, `javascript-obfuscator`). Этот шаг хотя и необходим, но требует огромных вычислительных ресурсов и крайне потребляет ресурсы CPU.

На раннем этапе разработки проекта **HagiCode** мы столкнулись с очень сложной производительной узкой точкой: время продакшен-сборки быстро ухудшалось по мере увеличения объёма кода.

**Конкретные проблемы**:
- Однопоточное последовательное выполнение задач обфускации, одно ядро CPU загружено на 100%, остальные ядра простаивают
- Время сборки выросло с первоначальных 30 секунд до 110-120 секунд
- Процесс проверки сборки после каждого изменения кода крайне длительный, серьёзно замедляющий эффективность итераций разработки
- В конвейере CI/CD этап сборки стал самой времязатратной частью

**Почему в HagiCode возникла такая потребность?**
HagiCode — это AI-ассистент для написания кода, его фронтенд-архитектура содержит сложную бизнес-логику и модули AI-взаимодействия. Для обеспечения безопасности основного кода мы при продакшен-релизе принудительно включили высокоинтенсивную обфускацию. Столкнувшись с ожиданием сборки длиной в две минуты, мы решили провести глубокую оптимизацию производительности системы сборки.

## Анализ: поиск точки прорыва для производительности

Прежде чем приступать к решению проблем производительности, нам нужно систематизировать мысли и определить оптимальное техническое решение.

### Ключевое решение: почему выбраны Worker Threads?

В среде Node.js для реализации параллельных вычислений есть три основных схемы:
1. **child_process**: создание независимых дочерних процессов
2. **Web Workers**: в основном используется в браузере
3. **worker_threads**: нативная поддержка многопоточности в Node.js

После сравнительного анализа HagiCode в итоге выбрал **Worker Threads** по следующим причинам:
- **Нулевые накладные расходы сериализации**: Worker Threads находятся в одном процессе, могут совместно использовать память через `SharedArrayBuffer` или передачу прав управления, избегая больших затрат на сериализацию при межпроцессном взаимодействии.
- **Нативная поддержка**: встроена в Node.js версии 12+, не требует введения дополнительных тяжёлых зависимостей.
- **Единый контекст**: отладка и логирование удобнее, чем в дочерних процессах.

### Гранулярность задач: как разделить задачи обфускации?

Обфускация огромного JS Bundle файла трудно поддаётся распараллеливанию (потому что код имеет зависимости), но артефакты сборки Vite состоят из нескольких **Chunk**. Это даёт нам естественную границу параллелизма:

- **Независимость**: зависимости между разными Chunk после упаковки Vite уже разъединены, можно безопасно обрабатывать параллельно.
- **Умеренная гранулярность**: обычно в проекте будет 10-30 Chunk, этот порядок величины идеально подходит для параллельного планирования.
- **Лёгкая интеграция**: хук `generateBundle` плагина Vite позволяет перехватывать и обрабатывать эти Chunk до генерации файлов.

### Архитектурный дизайн

Мы разработали систему параллельной обработки из четырёх основных компонентов:

1. **Task Splitter**: обход bundle объекта Vite, фильтрация файлов не требующих обфускации (например, vendor), генерация очереди задач.
2. **Worker Pool Manager**: управление жизненным циклом Worker, ответственный за распределение, возврат и повторную попытку при ошибках задач.
3. **Progress Reporter**: реальный вывод прогресса сборки, устраняющий тревожность ожидания пользователя.
4. **ObfuscationWorker**: рабочий поток, выполняющий логику обфускации.

## Решение: практическое кодирование и внедрение

На основе вышеприведённого анализа мы начали реализовывать эту систему параллельной обфускации.

### 1. Конфигурация плагина Vite

Сначала мы интегрировали плагин параллельной обфускации в `vite.config.ts`. Конфигурация очень интуитивна, нужно только указать количество Worker и правила обфускации.

```typescript
import { defineConfig } from 'vite'
import { parallelJavascriptObfuscator } from './buildTools/plugin'

export default defineConfig(({ mode }) => {
  const isProduction = mode === 'production'
  
  return {
    build: {
      rollupOptions: {
        ...(isProduction
          ? {
              plugins: [
                parallelJavascriptObfuscator({
                  enabled: true,
                  // Автоматическая настройка по количеству ядер CPU, рекомендуется оставить одно ядро для основного потока
                  workerCount: 4, 
                  retryAttempts: 3,
                  fallbackToMainThread: true, // При ошибке автоматическое понижение до однопоточного режима
                  // Фильтрация vendor chunk, обычно第三方 библиотеки не требуют обфускации
                  isVendorChunk: (fileName: string) => fileName.includes('vendor-'),
                  obfuscationConfig: {
                    compact: true,
                    controlFlowFlattening: true,
                    deadCodeInjection: true,
                    disableConsoleOutput: true,
                    // ... больше опций обфускации
                  },
                }),
              ],
            }
          : {}),
      },
    },
  }
})
```

### 2. Реализация логики Worker

Worker — это единица выполнения задач. Нам нужно определить структуры данных для ввода и вывода.

**Внимание**: хотя этот код прост, есть несколько моментов, на которые нужно обратить внимание. Например, проверка на null для `parentPort` и обработка ошибок. В практике HagiCode мы обнаружили, что некоторый специальный синтаксис ES6 может привести к крашу обфускатора, поэтому добавили защиту через `try-catch`.

```typescript
import { parentPort } from 'worker_threads'
import javascriptObfuscator from 'javascript-obfuscator'

export interface ObfuscationTask {
  chunkId: string
  code: string
  config: any
}

export interface ObfuscationResult {
  chunkId: string
  obfuscatedCode: string
  error?: string
}

// Слушание задач от основного потока
if (parentPort) {
  parentPort.on('message', async (task: ObfuscationTask) => {
    try {
      // Выполнение обфускации
      const obfuscated = javascriptObfuscator.obfuscate(task.code, task.config)
      const result: ObfuscationResult = {
        chunkId: task.chunkId,
        obfuscatedCode: obfuscated.getObfuscatedCode(),
      }
      // Отправка результата обратно в основной поток
      parentPort?.postMessage(result)
    } catch (error) {
      // Обработка исключений, гарантируем, что краш отдельного Worker не заблокирует всю сборку
      const result: ObfuscationResult = {
        chunkId: task.chunkId,
        obfuscatedCode: '',
        error: error instanceof Error ? error.message : 'Unknown error',
      }
      parentPort?.postMessage(result)
    }
  })
}
```

### 3. Менеджер пула Worker

Это ядро всей схемы. Нам нужно поддерживать пул Worker фиксированного размера, используя стратегию **FIFO (first in, first out)** для планирования задач.

```typescript
import { Worker } from 'worker_threads'
import os from 'os'

export class WorkerPool {
  private workers: Worker[] = []
  private taskQueue: Array<{
    task: ObfuscationTask
    resolve: (result: ObfuscationResult) => void
    reject: (error: Error) => void
  }> = []
  
  constructor(options: WorkerPoolOptions = {}) {
    // По умолчанию количество ядер - 1, оставляем основному потоку немного места
    const workerCount = options.workerCount ?? Math.max(1, (os.cpus().length || 4) - 1)
    
    for (let i = 0; i < workerCount; i++) {
      this.createWorker()
    }
  }

  private createWorker() {
    const worker = new Worker('./worker.ts')
    
    worker.on('message', (result) => {
      // После завершения задачи берём следующую из очереди
      const nextTask = this.taskQueue.shift()
      if (nextTask) {
        this.dispatchTask(worker, nextTask)
      } else {
        // Если нет ожидающих задач, отмечаем Worker как свободный
        this.activeWorkers.delete(worker)
      }
    })
    
    this.workers.push(worker)
  }

  // Отправка задачи в пул
  public runTask(task: ObfuscationTask): Promise<ObfuscationResult> {
    return new Promise((resolve, reject) => {
      const job = { task, resolve, reject }
      const idleWorker = this.workers.find(w => !this.activeWorkers.has(w))
      
      if (idleWorker) {
        this.dispatchTask(idleWorker, job)
      } else {
        this.taskQueue.push(job)
      }
    })
  }

  private dispatchTask(worker: Worker, job: any) {
    this.activeWorkers.set(worker, job.task)
    worker.postMessage(job.task)
  }
}
```

### 4. Отчёт о прогрессе

Ожидание мучительно, особенно когда не знаешь, сколько ещё ждать. Мы добавили простой репортёр прогресса, в реальном времени отражающий текущее состояние.

```typescript
export class ProgressReporter {
  private completed = 0
  private readonly total: number
  private readonly startTime: number

  constructor(total: number) {
    this.total = total
    this.startTime = Date.now()
  }

  increment(): void {
    this.completed++
    this.report()
  }

  private report(): void {
    const now = Date.now()
    const elapsed = now - this.startTime
    const percentage = (this.completed / this.total) * 100
    
    // Простая оценка ETA
    const avgTimePerChunk = elapsed / this.completed
    const remaining = (this.total - this.completed) * avgTimePerChunk

    console.log(
      `[Parallel Obfuscation] ${this.completed}/${this.total} chunks completed (${percentage.toFixed(1)}%) | ETA: ${(remaining / 1000).toFixed(1)}s`
    )
  }
}
```

## Практика: результаты и проблемы

После внедрения этой схемы производительность сборки проекта HagiCode немедленно значительно улучшилась.

### Бенчмарк производительности

Мы проводили тестирование в следующей среде:
- CPU: Intel Core i7-12700K (12 cores / 20 threads)
- RAM: 32GB DDR4
- Node.js: v18.17.0
- OS: Ubuntu 22.04

**Сравнение результатов**:
- **Один поток (до оптимизации)**: 118 секунд
- **4 Workers**: 55 секунд (улучшение на **53%**)
- **8 Workers**: 48 секунд (улучшение на **60%**)
- **12 Workers**: 45 секунд (улучшение на **62%**)

Видно, что выигрыш не линеен. Когда количество Worker превышает 8,幅度 улучшения уменьшается. Это в основном ограничено равномерностью распределения задач и узким местом пропускной способности памяти.

### Частые проблемы и решения

В реальном использовании HagiCode мы также столкнулись с некоторыми проблемами, здесь делимся с вами:

**Q1: Время сборки не значительно сократилось, а наоборот замедлилось?**
- **Причина**: создание Worker само по себе имеет накладные расходы, или количество Worker установлено слишком много, что приводит к частому переключению контекста.
- **Решение**: рекомендуется устанавливать количество Worker как `количество ядер CPU - 1`. Одновременно проверьте, нет ли отдельного Chunk особенно большого (например, > 5MB), такие "гигантские" файлы станут узким местом, можно рассмотреть оптимизацию стратегии разделения кода.

**Q2: Иногда Worker падает, сборка не удаётся?**
- **Причина**: некоторый специальный синтаксис кода может привести к внутренней ошибке обфускатора.
- **Решение**: мы реализовали **механизм автоматического понижения**. Когда количество последовательных отказов Worker достигает порога, плагин автоматически откатывается в однопоточный режим, обеспечивая непрерывность сборки. Одновременно записывается имя ошибочного файла для удобства последующего целенаправленного исправления.

**Q3: Слишком высокое потребление памяти (OOM)?**
- **Причина**: каждому Worker требуется независимое пространство памяти для загрузки обфускатора и разбора AST.
- **Решение**:
  - Уменьшить количество Worker.
  - Увеличить лимит памяти Node.js: `NODE_OPTIONS="--max-old-space-size=4096" npm run build`.
  - Убедиться, что внутри Worker не удерживаются ненужные ссылки на большие объекты.
