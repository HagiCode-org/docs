## Полное руководство по разбору сообщений консоли Codex SDK

> В этой статье подробно описывается механизм потоков событий Codex SDK, разбор типов сообщений и лучшие практики в реальных проектах, помогая разработчикам быстро освоить ключевые навыки службы выполнения AI.

## Предыстория

На самом деле, при создании службы выполнения AI на основе Codex SDK мы неизбежно сталкиваемся с такой проблемой: как обрабатывать те потоковые сообщения событий, которые возвращает Codex. Эти сообщения содержат важные данные: состояние выполнения, выходное содержимое, информацию об ошибках — как те неразрывные чувства юности, которые нужно тщательно обдумать.

Как часть проекта HagiCode, нам нужно реализовать надёжный исполнитель в сценарии AI-помощника по коду. Именно поэтому мы решили глубоко изучить механизм потоков событий Codex SDK — ведь только понимая, как работают底层ные сообщения, можно создать по-настоящему enterprise-платформу выполнения AI. Это как в любви — если не понимать мысли другого, как двигаться дальше?

Codex SDK — это SDK для программирования, выпущенный OpenAI, который возвращает результаты выполнения через поток событий (Event Stream). В отличие от традиционной модели запрос-ответ, Codex использует потоковые события, что позволяет нам:

- Получать прогресс выполнения в реальном времени
- Своевременно обрабатывать ошибочные ситуации
- Получать детальную статистику использования токенов
- Поддерживать сложные задачи с длительным временем выполнения

Понимание этих типов событий и их правильный разбор довольно важны для реализации полнофункционального AI-исполнителя. В конце концов, никто не хочет иметь дело с чёрным ящиком.

## Механизм потоков событий

### Базовые концепции

Codex SDK использует метод `thread.runStreamed()` для возврата асинхронного итератора событий:

```typescript
import { Codex } from '@openai/codex-sdk';

const client = new Codex({
  apiKey: process.env.CODEX_API_KEY,
  baseUrl: process.env.CODEX_BASE_URL,
});

const thread = client.startThread({
  workingDirectory: '/path/to/project',
  skipGitRepoCheck: false,
});

const { events } = await thread.runStreamed('your prompt here', {
  outputSchema: {
    type: 'object',
    properties: {
      output: { type: 'string' },
      status: { type: 'string', enum: ['ok', 'action_required'] },
    },
    required: ['output', 'status'],
  },
});

for await (const event of events) {
  // Обработка каждого события
}
```

### Подробное описание типов событий

| Тип события | Описание | Ключевые данные |
|---------|------|----------|
| `thread.started` | Поток успешно запущен | `thread_id` |
| `item.updated` | Обновление содержимого сообщения | `item.text` |
| `item.completed` | Сообщение завершено | `item.text` |
| `turn.completed` | Выполнение завершено | `usage` (использование токенов) |
| `turn.failed` | Выполнение не удалось | `error.message` |
| `error` | Событие ошибки | `message` |

В реальных проектах компонент исполнителя HagiCode построен именно на основе этих типов событий. Нам нужна точная обработка каждого типа событий, чтобы обеспечить плавность пользовательского опыта. Это как в отношениях — каждая деталь требует внимательного отношения, иначе как можно добиться хорошего результата?

## Реализация разбора сообщений

### Извлечение содержимого сообщений

Содержимое сообщений извлекается через функцию обработки событий:

```typescript
private handleThreadEvent(event: ThreadEvent, onMessage: (content: string) => void): void {
  // Обрабатываем только события обновления и завершения сообщений
  if (event.type !== 'item.updated' && event.type !== 'item.completed') {
    return;
  }

  // Обрабатываем только тип сообщений агента
  if (event.item.type !== 'agent_message') {
    return;
  }

  // Извлекаем текстовое содержимое
  onMessage(event.item.text);
}
```

Ключевые моменты:
- Обрабатываем только события `item.updated` и `item.completed`
- Обрабатываем только содержимое типа `agent_message`
- Содержимое сообщения находится в поле `event.item.text`

### Разбор структурированного вывода

Codex поддерживает JSON структурированный вывод, формат возвращаемых данных указывается через параметр `outputSchema`:

```typescript
const DEFAULT_OUTPUT_SCHEMA = {
  type: 'object',
  properties: {
    output: { type: 'string' },
    status: { type: 'string', enum: ['ok', 'action_required'] },
  },
  required: ['output', 'status'],
  additionalProperties: false,
} as const;
```

Функция разбора попытается разобрать JSON, в случае неудачи вернёт исходный текст — это как в жизни: иногда вы хотите идеальный ответ, но реальность часто даёт вам смутный ответ, с которым можно только медленно разобраться самому.

```typescript
function toStructuredOutput(raw: string): StructuredOutput {
  try {
    const parsed = JSON.parse(raw) as Partial<StructuredOutput>;
    if (typeof parsed.output === 'string') {
      return {
        output: parsed.output,
        status: parsed.status === 'action_required' ? 'action_required' : 'ok',
      };
    }
  } catch {
    // Разбор JSON не удался, откат к исходному тексту
  }

  return {
    output: raw,
    status: 'ok',
  };
}
```

### Полный процесс обработки событий

```typescript
private async runWithStreaming(
  thread: Thread,
  input: CodexStageExecutionInput
): Promise<{ output: string; usage: Usage | null }> {
  const abortController = new AbortController();
  const timeoutHandle = setTimeout(() => {
    abortController.abort();
  }, Math.max(1000, input.timeoutMs));

  let latestMessage = '';
  let usage: Usage | null = null;
  let emittedLength = 0;

  try {
    const { events } = await thread.runStreamed(input.prompt, {
      outputSchema: DEFAULT_OUTPUT_SCHEMA,
      signal: abortController.signal,
    });

    for await (const event of events) {
      // Обработка содержимого сообщения
      this.handleThreadEvent(event, (nextContent) => {
        const delta = nextContent.slice(emittedLength);
        if (delta.length > 0) {
          emittedLength = nextContent.length;
          input.callbacks?.onChunk?.(delta);  // Потоковый обратный вызов
        }
        latestMessage = nextContent;
      });

      // Обработка разных данных в зависимости от типа события
      if (event.type === 'thread.started') {
        this.threadId = event.thread_id;
      } else if (event.type === 'turn.completed') {
        usage = event.usage;
      } else if (event.type === 'turn.failed') {
        throw new CodexExecutorError('gateway_unavailable', event.error.message, true);
      } else if (event.type === 'error') {
        throw new CodexExecutorError('gateway_unavailable', event.message, true);
      }
    }
  } catch (error) {
    if (abortController.signal.aborted) {
      throw new CodexExecutorError(
        'upstream_timeout',
        `Codex stage timed out after ${input.timeoutMs}ms`,
        true
      );
    }
    throw error;
  } finally {
    clearTimeout(timeoutHandle);
  }

  const structured = toStructuredOutput(latestMessage);
  return { output: structured.output, usage };
}
```

## Стратегии обработки ошибок

### Сопоставление кодов ошибок

Сопоставление характеристик ошибок с конкретными кодами ошибок для удобства обработки на верхнем уровне:

```typescript
function mapError(error: unknown): CodexExecutorError {
  if (error instanceof CodexExecutorError) {
    return error;
  }

  const message = error instanceof Error ? error.message : String(error);
  const normalized = message.toLowerCase();

  // Ошибки аутентификации - не повторять
  if (normalized.includes('401') ||
      normalized.includes('403') ||
      normalized.includes('api key') ||
      normalized.includes('auth')) {
    return new CodexExecutorError('auth_invalid', message, false);
  }

  // Ограничение скорости - можно повторить
  if (normalized.includes('429') || normalized.includes('rate limit')) {
    return new CodexExecutorError('rate_limited', message, true);
  }

  // Ошибки тайм-аута - можно повторить
  if (normalized.includes('timeout') || normalized.includes('aborted')) {
    return new CodexExecutorError('upstream_timeout', message, true);
  }

  // Ошибка по умолчанию
  return new CodexExecutorError('gateway_unavailable', message, true);
}
```

### Определение типов ошибок

```typescript
export type CodexErrorCode =
  | 'auth_invalid'      // Сбой аутентификации
  | 'upstream_timeout'  // Тайм-аут upstream
  | 'rate_limited'      // Ограничение скорости
  | 'gateway_unavailable'; // Шлюз недоступен

export class CodexExecutorError extends Error {
  readonly code: CodexErrorCode;
  readonly retryable: boolean;

  constructor(code: CodexErrorCode, message: string, retryable: boolean) {
    super(message);
    this.name = 'CodexExecutorError';
    this.code = code;
    this.retryable = retryable;
  }
}
```

## Рабочий каталог и конфигурация окружения

### Проверка рабочего каталога

Codex SDK требует, чтобы рабочий каталог был действительным репозиторием Git — это как быть человеком: всегда должен быть корень, происхождение, иначе как быть спокойным?

```typescript
export function validateWorkingDirectory(
  workingDirectory: string,
  skipGitRepoCheck: boolean
): void {
  const resolvedWorkingDirectory = path.resolve(workingDirectory);

  if (!existsSync(resolvedWorkingDirectory)) {
    throw new CodexExecutorError(
      'gateway_unavailable',
      'Working directory does not exist.',
      false
    );
  }

  if (!statSync(resolvedWorkingDirectory).isDirectory()) {
    throw new CodexExecutorError(
      'gateway_unavailable',
      'Working directory is not a directory.',
      false
    );
  }

  if (skipGitRepoCheck) {
    return;
  }

  const gitDir = path.join(resolvedWorkingDirectory, '.git');
  if (!existsSync(gitDir)) {
    throw new CodexExecutorError(
      'gateway_unavailable',
      'Working directory is not a git repository.',
      false
    );
  }
}
```

### Загрузка переменных окружения

Codex SDK должен загружать переменные окружения из логин-оболочки, чтобы AI Agent мог иметь доступ к системным командам:

```typescript
function parseEnvironmentOutput(output: Buffer): Record<string, string> {
  const parsed: Record<string, string> = {};

  for (const entry of output.toString('utf8').split('\0')) {
    if (!entry) continue;

    const separatorIndex = entry.indexOf('=');
    if (separatorIndex <= 0) continue;

    const key = entry.slice(0, separatorIndex);
    const value = entry.slice(separatorIndex + 1);
    if (key.length > 0) {
      parsed[key] = value;
    }
  }

  return parsed;
}

function tryLoadEnvironmentFromShell(shellPath: string): Record<string, string> | null {
  const result = spawnSync(shellPath, ['-ilc', 'env -0'], {
    env: process.env,
    stdio: ['ignore', 'pipe', 'pipe'],
    timeout: 5000,
  });

  if (result.error || result.status !== 0) {
    return null;
  }

  return parseEnvironmentOutput(result.stdout);
}

export function createExecutorEnvironment(
  envOverrides: Record<string, string> = {}
): Record<string, string> {
  // Загрузка переменных окружения логин-оболочки
  const consoleEnv = loadConsoleEnvironmentFromShell();

  return {
    ...process.env,
    ...consoleEnv,
    ...envOverrides,
  };
}
```

## Полный пример использования

### Базовое использование

В проекте HagiCode мы используем следующий способ для инициализации клиента Codex и выполнения задач:

```typescript
import { Codex } from '@openai/codex-sdk';

async function executeWithCodex(prompt: string, workingDir: string) {
  const client = new Codex({
    apiKey: process.env.CODEX_API_KEY,
    env: { PATH: process.env.PATH },
  });

  const thread = client.startThread({
    workingDirectory: workingDir,
  });

  const { events } = await thread.runStreamed(prompt);

  let result = '';
  for await (const event of events) {
    if (event.type === 'item.updated' && event.item.type === 'agent_message') {
      result = event.item.text;
    }
    if (event.type === 'turn.completed') {
      console.log('Token usage:', event.usage);
    }
  }

  // Попытка разобрать JSON вывод
  try {
    const parsed = JSON.parse(result);
    return parsed.output;
  } catch {
    return result;
  }
}
```

### Полная реализация с механизмом повторных попыток

```typescript
export class CodexSdkExecutor {
  private readonly config: CodexRuntimeConfig;
  private readonly client: Codex;
  private threadId: string | null = null;

  async executeStage(input: CodexStageExecutionInput): Promise<CodexStageExecutionResult> {
    const maxAttempts = Math.max(1, this.config.retryCount + 1);
    let attempt = 0;
    let lastError: CodexExecutorError | null = null;

    while (attempt < maxAttempts) {
      attempt += 1;

      try {
        const thread = this.getThread(input.workingDirectory);
        const { output, usage } = await this.runWithStreaming(thread, input);

        return {
          output,
          usage,
          threadId: this.threadId!,
          attempts: attempt,
          latencyMs: Date.now() - startedAt,
        };
      } catch (error) {
        const mappedError = mapError(error);
        lastError = mappedError;

        // Не повторяемая ошибка или достигнуто максимальное количество попыток
        if (!mappedError.retryable || attempt >= maxAttempts) {
          throw mappedError;
        }

        // Повтор после ожидания
        await new Promise(resolve => setTimeout(resolve, 1000 * attempt));
      }
    }

    throw lastError!;
  }
}
```

## Лучшие практики

### 1. Требования к рабочему каталогу

- Убедитесь, что рабочий каталог является действительным репозиторием Git
- Используйте переменную окружения `PROJECT_ROOT` для явного указания
- При разработке и отладке можно установить `CODEX_SKIP_GIT_REPO_CHECK=true` для пропуска проверки

### 2. Конфигурация переменных окружения

- Передавайте необходимые переменные окружения через механизм белого списка
- Используйте логин-оболочку для загрузки полного окружения
- Избегайте передачи конфиденциальной информации

### 3. Тайм-аут и повторы

- Устанавливайте разумное время тайм-аута в зависимости от сложности задачи
- Реализуйте экспоненциальную задержку для повторяемых ошибок
- Записывайте количество попыток и причины

### 4. Обработка ошибок

- Различайте повторяемые и неповторяемые ошибки
- Предоставляйте чёткую информацию об ошибках и рекомендации
- Унифицируйте коды ошибок для удобства обработки на верхнем уровне

### 5. Потоковый вывод

- Реализуйте обратный вызов инкрементного вывода для улучшения пользовательского опыта
- Правильно обрабатывайте инкрементные обновления сообщений
- Записывайте использование токенов для анализа затрат
