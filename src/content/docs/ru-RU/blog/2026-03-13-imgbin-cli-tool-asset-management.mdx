## Дизайн ImgBin CLI: решение HagiCode для управления изображениями

> В этой статье рассказывается, как с нуля создать автоматизируемый конвейер для управления изображениями, включая дизайн CLI-инструмента, архитектуру Provider Adapter и стратегии управления метаданными.

## Предпосылки

Честно говоря, мы не ожидали, что управление изображениями заставит нас так долго мучиться.

В процессе разработки проекта HagiCode мы столкнулись с проблемой, которая казалась простой, но оказалась крайне сложной: генерация и управление изображениями. Это похоже на подростковые переживания —表面上风平浪静，暗地里波澜起伏.

С ростом проектной документации и маркетинговых материалов потребовалось много иллюстраций. Некоторые из них должны быть сгенерированы ИИ, некоторые — выбраны из существующей библиотеки материалов, а还有一些需要对现有图片进行 AI 识别并自动标注。Проблема в том，что эта работа долгое время выполнялась с помощью разрозненных скриптов и ручных операций —每次生成一张图片，都需要手动执行脚本、手动整理元数据、手动生成缩略图。Это было бы ещё полбеды，но главное，что эти разрозненные элементы разбросаны повсюду，хочешь найти — не найдёшь，хочешь использовать — не можешь.

К конкретным проблемам относятся:

1. **Отсутствие единой точки входа**: логика генерации изображений разбросана по разным скриптам，о пакетном выполнении не может быть и речи
2. **Отсутствие метаданных**: после генерации изображений нет единого metadata.json，невозможно выполнить поиск и отслеживание
3. **Высокая стоимость ручной обработки**: заголовки и теги изображений требуют ручной обработки，низкая эффективность
4. **Невозможность автоматизации**: хотите автоматически генерировать иллюстрации в CI/CD процессах? Нет шансов

Мы тоже думали просто забить на это，но毕竟还是要做项目的嘛。Раз уйти от этого нельзя，надо想办法 решить.Поэтому мы решили обновить ImgBin с «разрозненных скриптов» до автоматизируемого конвейера управления изображениями.毕竟有些事儿，逃避也不是办法。

## О HagiCode

Решение，поделенное в этой статье，основано на нашем практическом опыте в проекте [HagiCode](https://hagicode.com).HagiCode — это проект ИИ-помощника для кодирования，одновременно поддерживающий VSCode расширение、бэкенд ИИ-сервис、кроссплатформенный десктопный клиент и другие компоненты.В такой сложной многоязычной и мультиплатформенной среде стандартизированное управление изображениями стало ключевым элементом повышения эффективности разработки.

怎么说呢，这也算是 HagiCode 成长过程中的一个小小烦恼吧.У каждого проекта бывают такие моменты，когда на вид незначительные проблемы могут заставить мучиться полдня.

Система сборки HagiCode использует экосистему TypeScript + Node.js，поэтому ImgBin также естественным образом выбрал тот же технологический стек，обеспечивая технологическую целостность всего проекта.毕竟都用习惯了，换别的也嫌麻烦嘛。

## Основной дизайн

### Общая архитектура

ImgBin использует слоистую архитектуру，чётко разделяющую CLI-команды，прикладные службы，адаптеры сторонних API и уровень инфраструктуры:

```
Иерархия компонентов
├── CLI Entry (cli.ts)              разбор глобальных параметров, маршрутизация команд
├── Commands (commands/*)           generate | batch | annotate | thumbnail
├── Application Services            job-runner | metadata | thumbnail | asset-writer
├── Provider Adapters               image-api-provider | vision-api-provider
└── Infrastructure Layer            config | logger | paths | schema
```

Преимущество такой слоистой конструкции в том，что обязанности каждого слоя чётко определены，а при тестировании можно легко мокать внешние зависимости.На самом деле это просто даёт каждому заниматься своим делом，не мешая друг другу，так что при проблемах легко найти причину，не так ли?

### Модель одно-каталогового актива

ImgBin использует модель «один актив — один каталог», при генерации каждого изображения создаётся следующая структура:

```
library/
└── 2026-03/
    └── orange-dashboard/
        ├── original.png      # исходное изображение
        ├── thumbnail.webp    # миниатюра 512x512
        └── metadata.json     # структурированные метаданные
```

Преимущества этой модели:

1. **Самодостаточность**: все файлы каждого актива находятся в одном каталоге，удобно мигрировать и резервировать
2. **Отслеживаемость**: через metadata.json можно отследить время генерации，использованный prompt，модель и другую информацию
3. **Расширяемость**: в будущем，если нужно добавить больше вариантов (например，миниатюры разных размеров)，просто нужно добавить файлы в тот же каталог

美的事物或人，不一定要占有，只要她还是美的，自己好好看着她的美就好了.Хотя это звучит немного далеко，но смысл такой — изображения вместе，выглядят уютно，и искать удобно.

### Слоистое хранение метаданных

metadata.json — это ядро всей системы，оно использует стратегию слоистого хранения，разделяя три типа полей:

```json
{
  "schemaVersion": 2,
  "assetId": "orange-dashboard",
  "slug": "orange-dashboard",
  "title": "Orange Dashboard",
  "tags": ["dashboard", "hero", "orange"],

  "source": { "type": "generated" },

  "paths": {
    "assetDir": "library/2026-03/orange-dashboard",
    "original": "original.png",
    "thumbnail": "thumbnail.webp"
  },

  "generated": {
    "prompt": "orange dashboard for docs hero",
    "provider": "azure-openai-image-api",
    "model": "gpt-image-1.5"
  },

  "recognized": {
    "title": "Orange Dashboard",
    "tags": ["dashboard", "ui", "orange"],
    "description": "A modern orange dashboard with charts and metrics"
  },

  "status": {
    "generation": "succeeded",
    "recognition": "succeeded",
    "thumbnail": "succeeded"
  },

  "timestamps": {
    "createdAt": "2026-03-11T04:01:19.570Z",
    "updatedAt": "2026-03-11T04:02:09.132Z"
  }
}
```

- **generated**:记录原始ную информацию при генерации изображения，например использованный prompt，поставщика，модель и т.д.
- **recognized**: результаты ИИ-распознавания，например автоматически сгенерированные заголовки，теги，описания
- **manual**: результаты ручной обработки，данные в этой зоне имеют наивысший приоритет и не будут перезаписаны ИИ-распознаванием

Эта слоистая стратегия решила нашу предыдущую ключевую дилемму: что имеет приоритет — результаты ИИ-распознавания или ручной обработки? Ответ — человек имеет приоритет，ИИ-распознавание только помогает.这事儿也想明白了——有些东西嘛，机器终究是机器，终究还是得人来把关.

## Паттерн Provider Adapter

Другим ключевым элементом дизайна ImgBin является паттерн Provider Adapter.Мы абстрагируем внешние API в единый интерфейс，так что даже при смене поставщика ИИ-услуг не нужно изменять бизнес-логику.

怎么说呢，这就跟感情一样——外表怎么变不重要，重要的是内心那套东西不变.Интерфейс определён，внутреннюю реализацию можно менять как угодно.

### Image Generation Provider

```typescript
interface ImageGenerationProvider {
  // Генерация изображения, возврат Buffer изображения
  generate(options: GenerateOptions): Promise<Buffer>;

  // Получение списка поддерживаемых моделей
  getSupportedModels(): Promise<string[]>;
}

interface GenerateOptions {
  prompt: string;
  model?: string;
  size?: '1024x1024' | '1792x1024' | '1024x1792';
  quality?: 'standard' | 'hd';
  format?: 'png' | 'webp' | 'jpeg';
}
```

### Vision Recognition Provider

```typescript
interface VisionRecognitionProvider {
  // Распознавание содержимого изображения, возврат структурированных метаданных
  recognize(imageBuffer: Buffer): Promise<RecognitionResult>;

  // Получение списка поддерживаемых моделей
  getSupportedModels(): Promise<string[]>;
}

interface RecognitionResult {
  title?: string;
  tags: string[];
  description?: string;
  confidence: number;
}
```

Преимущества такого дизайна интерфейса:

1. **Тестируемость**: при модульном тестировании можно передавать mock provider，не нужно вызывать реальные внешние API
2. **Расширяемость**: добавление нового provider требует только реализации интерфейса，не нужно изменять код вызывающей стороны
3. **Заменимость**: в продакшене использовать Azure OpenAI，в тестовой среде — локальную модель，просто нужно переключить конфигурацию

想笑来伪装自己掉下的泪，想哭来试探自己麻痹了没——иногда при разработке проектов так бывает，表面上 меняешь API，а фактически внутренняя логика вообще не изменилась，так что нечего бояться.

## Дизайн CLI-команд

ImgBin предоставляет четыре основные команды，покрывающие различные сценарии использования:

### generate: генерация одного изображения

```bash
# Самое простое использование
imgbin generate --prompt "orange dashboard for docs hero"

# Одновременная генерация миниатюры и ИИ-аннотации
imgbin generate --prompt "orange dashboard" --annotate --thumbnail

# Указание выходного каталога
imgbin generate --prompt "orange dashboard" --output ./library
```

### batch: пакетные задачи

Пакетные задачи определяются через YAML или JSON manifest файл，подходит для использования в CI/CD процессах:

```yaml
# assets/jobs/launch.yaml
defaults:
  annotate: true
  thumbnail: true
  libraryRoot: ./library

jobs:
  - prompt: "orange dashboard hero"
    slug: orange-dashboard
    tags: [dashboard, hero, orange]

  - prompt: "pricing grid for docs"
    slug: pricing-grid
    tags: [pricing, grid, docs]
```

Команда выполнения:

```bash
imgbin batch assets/jobs/launch.yaml
```

Дизайн пакетных задач поддерживает изоляцию сбоев: обрабатывает по одному элементу в manifest，сбой одного элемента не влияет на другие задачи.Можно использовать `--dry-run` для предварительного просмотра без фактического выполнения.

这也就罢了，关键是它还能告诉你哪儿成功了哪儿失败了，不像某些事儿，失败了都不知道怎么失败的.

### annotate: ИИ-аннотация

Выполняет ИИ-распознавание существующих изображений，автоматически генерирует заголовки，теги，описания:

```bash
# Аннотация одного изображения
imgbin annotate ./library/2026-03/orange-dashboard

# Пакетная аннотация всего каталога
imgbin annotate ./library/2026-03/
```

### thumbnail: генерация миниатюр

Дополняет существующие изображения миниатюрами:

```bash
# Генерация миниатюры
imgbin thumbnail ./library/2026-03/orange-dashboard
```

## Дизайн Manifest для пакетных задач

Manifest пакетных задач поддерживает гибкую конфигурацию，значения по умолчанию можно установить единообразно，отдельные задачи также могут переопределять:

```yaml
# Глобальные значения по умолчанию
defaults:
  annotate: true        # по умолчанию включить ИИ-аннотацию
  thumbnail: true       # по умолчанию генерировать миниатюры
  libraryRoot: ./library
  model: gpt-image-1.5

jobs:
  # Минимальная конфигурация, только prompt
  - prompt: "first image"

  # Полная конфигурация
  - prompt: "second image"
    slug: custom-slug
    tags: [tag1, tag2]
    annotate: false     # эта задача не выполняет ИИ-аннотацию
    model: dall-e-3    # эта задача использует другую модель
```

При выполнении ImgBin обрабатывает задачи одну за другой，результат каждой задачи записывается в соответствующий metadata.json，даже если какая-то задача не удастся，это не повлияет на другие задачи.После завершения задач выводится сводный отчёт:

```
✓ orange-dashboard (succeeded)
✓ pricing-grid (succeeded)
✗ hero-banner (failed: API rate limit exceeded)

2/3 succeeded, 1 failed
```

有些事儿吧，急也急不来，一个一个来，反而踏实.Это，возможно，и есть философия пакетных задач.

## Конфигурация переменных среды

ImgBin поддерживает гибкую конфигурацию через переменные среды:

```bash
# Рабочий каталог ImgBin
IMGBIN_WORKDIR=/path/to/imgbin

# Путь к исполняемому файлу (для вызова в скриптах)
IMGBIN_EXECUTABLE=/path/to/imgbin/dist/cli.js

# Корневой каталог библиотеки активов
IMGBIN_LIBRARY_ROOT=./.imgbin-library

# Конфигурация Azure OpenAI (если используется Azure provider)
AZURE_OPENAI_ENDPOINT=https://your-resource.openai.azure.com/
AZURE_OPENAI_API_KEY=your-api-key
AZURE_OPENAI_IMAGE_DEPLOYMENT=gpt-image-1
```

配置这东西，说重要也重要，说不重要也不重要.毕竟怎么舒服怎么来嘛，适合自己的才是最好的.

## Ключевые моменты реализации

В процессе реализации мы обобщили следующие ключевые моменты:

### Дизайн Provider-интерфейса

Определение интерфейса должно быть чётким и полным，включая входные параметры，возвращаемые значения，обработку ошибок.Рекомендуется одновременно предоставлять синхронный и асинхронный способы вызова，удобно для разных сценариев использования.

这也算是过来人的一点经验吧，毕竟接口这东西，定好了就不想再改，麻烦.

### Стратегия обработки сбоев

При сбое какой-либо задачи в пакете следует:

1. Записать подробную информацию об ошибке в отдельный файл журнала
2. Продолжить выполнение других задач，не прерывая весь процесс
3. В конечном итоге вернуть ненулевой код выхода，указывающий на наличие неудачных задач
4. Чётко отобразить результат выполнения каждой задачи в сводном отчёте

有些事儿失败了就是失败了，逃避也没用，不如大大方方承认，然后想办法解决.这道理，做项目和做人是一样的.

### Стратегия слияния метаданных

Результаты распознавания по умолчанию записываются в область `recognized`，ручные изменения имеют метку `manual`.При обновлении метаданных используется стратегия «только добавление，не удаление»: если не显式 передан параметр `--force`，существующие результаты ручной обработки не перезаписываются.

这事儿也想明白了——有些东西啊，错过了就是错过了，覆盖了也就没了.还是保留着比较好，毕竟记录本身也是一种美.

### Атомарность создания каталогов

Используйте `fs.mkdir({ recursive: true })` для обеспечения атомарности создания каталогов，избегайте竞态条件 в параллельных сценариях.

这大概就是所谓的安全感吧——该稳稳该快快，不拖泥带水，也不瞻前顾后.

## Заключение

ImgBin как ключевой инструмент управления изображениями в проекте HagiCode решил наши проблемы через следующий дизайн:

1. **Единая точка входа**: CLI-команды покрывают все операции，включая генерацию，аннотацию，миниатюры
2. **Управление на основе метаданных**: каждый актив имеет полный metadata.json，поддерживает поиск и отслеживание
3. **Provider Adapter**: гибкая абстракция внешних API，удобно для тестирования и расширения
4. **Поддержка пакетных задач**: в CI/CD процессах можно автоматически выполнять пакетную генерацию изображений
