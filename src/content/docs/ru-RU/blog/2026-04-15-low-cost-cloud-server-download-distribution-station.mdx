## Как создать недорогую распределительную станцию для ускорения загрузок с помощью дешевого облачного сервера

> Трафик облачного хранения настолько дорог, что это невыносимо, трансграничный доступ настолько медленный, что это сводит с ума, а цены на CDN настолько высоки, что от них отказываются... Эти боли, вероятно, знакомы всем, кто занимается распространением. В этой статье я поделюсь набором недорогих решений, которые мы разработали в проекте [HagiCode](https://hagicode.com). Облачный сервер + уровень кэширования Nginx, затраты снизились вдвое, а скорость повысилась — это небольшое утешение.

## Предпосылки

В интернете скорость загрузки и стабильность — в конечном итоге пользовательский опыт. Будь то open source или коммерческие проекты, пользователям всегда нужен надежный способ загрузки.

Прямая загрузка файлов из облачного хранилища (например, Azure Blob Storage, AWS S3) кажется простой, но на самом деле проблем还真 немало:

**Сетевая задержка**: трансграничный и трансрегиональный доступ настолько медленный, что хочется разбить клавиатуру. Пользователи ждут так долго, что experience уже не может быть хорошим.

**Стоимость пропускной способности**: исходящий трафик облачного хранилища настолько дорог, что это больно. Доступ к Azure Blob Storage из материкового Китая стоит около 0,5 юаня за ГБ, 1 ТБ в месяц — это 500 юаней. Для небольших команд эти деньги скажешь не много, не мало, но毕竟谁的钱都不是大风刮来的.

**Ограничения доступа**: в некоторых регионах доступ к зарубежным облачным сервисам иногда работает, иногда нет, а иногда вообще недоступен. Пользователи не могут скачать — это уже беспомощно.

**Стоимость CDN**: коммерческие CDN действительно могут решить проблему, но и цена действительно красивая. Какие небольшие команды могут позволить себе?

Тогда есть ли способ сэкономить и при этом хорошо пользоваться? На самом деле есть. Облачный сервер + обратный прокси + уровень кэширования — просто и грубо. Затраты снизились примерно вдвое, а скорость выросла — это тоже небольшое утешение.

## Архитектурный дизайн

### Общая идея архитектуры

Сначала посмотрим на общую архитектурную design:

```
Запрос пользователя
    ↓
Разрешение DNS
    ↓
┌─────────────────────────────────────┐
│   Уровень обратного прокси (Traefik/Bunker Web)   │ ← Прекращение SSL, маршрутизация, защита безопасности
├─────────────────────────────────────┤
│   Порт: 80/443                       │
│   Функция: автоматические сертификаты Let's Encrypt      │
│         Маршрутизация по Host                    │
└─────────────────────────────────────┘
    ↓
┌─────────────────────────────────────┐
│   Уровень кэширования (Nginx)                     │ ← Кэширование файлов, сжатие Gzip
├─────────────────────────────────────┤
│   Порт: 8080(server) / 8081(desktop) │
│   Стратегия кэширования:                          │
│   - index.json: 1 час               │
│   - другие файлы: 7 дней                   │
│   Размер кэша: 1ГБ                      │
└─────────────────────────────────────┘
    ↓
┌─────────────────────────────────────┐
│   Исходный сервер (Azure Blob Storage)         │ ← Хранилище файлов
└─────────────────────────────────────┘
```

Основная идея этой архитектуры, проще говоря: **между пользователем и облачным хранилищем добавить слой кэширования**.

Запрос пользователя сначала поступает на уровень обратного прокси облачного сервера, затем берет на себя уровень кэширования Nginx. Есть файл в кэше? Сразу отдаем пользователю. Нет? Идем в облачное хранилище за копией, заодно сохраняем локальную копию. В следующий раз при доступе к тому же файлу не нужно беспокоить облачное хранилище. На самом деле это как память — что-то запомнил, в следующий раз не нужно снова напрягаться, чтобы вспомнить...

### Почему выбрана эта архитектура?

**Преимущества облачного сервера**:
- Контролируемые затраты: Alibaba Cloud и другие предоставляют дешевые облачные серверы (конфигурация 1-2 ядра 2ГБ стоит около 50-100 юаней в месяц)
- Гибкое развертывание: можно свободно настроить обратный прокси, стратегию кэширования
- Гибкое географическое расположение: можно выбрать серверный регион, близкий к пользователям
- Масштабируемость: можно обновить конфигурацию в соответствии с требованиями трафика

**Архитектура обратный прокси + кэширование**:
- Снижение нагрузки на исходный сервер: кэширование горячих файлов, снижение доступа к облачному хранилищу
- Снижение затрат: стоимость трафика облачного сервера намного ниже, чем исходящий трафик облачного хранилища
- Повышение скорости: близкий доступ, пропускная способность сервера обычно лучше, чем у облачного хранилища

**Почему выбран Nginx в качестве уровня кэширования?**

Это не случайно выбрано,毕竟 у Nginx действительно есть свои причины:

1. Высокая производительность: производительность обратного прокси Nginx признана в индустрии
2. Зрелое кэширование: встроенная функция proxy_cache, стабильная и надежная, не будет часто создавать проблемы
3. Низкое использование ресурсов: может работать с 256МБ памяти, дружелюбно к серверу
4. Гибкая настройка: для разных типов файлов можно установить разные стратегии кэширования, это довольно мило

## Уровень обратного прокси: Traefik против Bunker Web

Схема развертывания HagiCode на самом деле поддерживает два типа обратных прокси, скажем так, у каждого свои особенности:

| Схема | Особенности | Применимые сценарии |
|------|------|----------|
| **Traefik** | Легкий вес, автоматический SSL, простая настройка | Базовое развертывание, сценарии с низким трафиком |
| **Bunker Web** | Встроенный WAF, защита от DDoS, защита от сканеров | Высокие требования к безопасности, сценарии с высоким трафиком |

### Traefik: лучший выбор для легкого веса

Traefik — это современный обратный прокси HTTP и балансировщик нагрузки, самая большая особенность — простая настройка, еще можно автоматически получить сертификаты Let's Encrypt.

Для начального развертывания или сценариев с небольшим трафиком Traefik — довольно хороший выбор:
- невысокое использование ресурсов (достаточно 1.5 CPU/512МБ памяти)
- автоматическая настройка SSL-сертификатов, не нужно беспокоиться самому
- настройка маршрутизации на основе меток Docker, тоже удобно

### Bunker Web: сценарии с высокой безопасностью

Bunker Web — это веб-application firewall на основе Nginx, безопасность более полная.

Когда можно рассмотреть переключение на Bunker Web? Примерно в этих случаях:
- подвергается DDoS-атаке (хотя никто не хочет встретить)
- нужна защита ModSecurity
- нужна функция защиты от сканеров
- более высокие требования к безопасности

HagiCode предоставляет скрипт `switch-deployment.sh`, который можно быстро переключить между двумя схемами:

```bash
# Переключиться на Bunker Web
./switch-deployment.sh bunkerweb

# Переключиться обратно на Traefik
./switch-deployment.sh traefik

# Просмотреть текущий статус
./switch-deployment.sh status
```

Скрипт автоматически выполнит предварительную проверку, проверку работоспособности, еще может автоматически откатиться, процесс переключения безопасен и надежен, не скажешь, что при переключении все повиснет.

## Настройка уровня кэширования Nginx

Уровень кэширования — это ядро всей архитектуры, хорошо ли настроен Nginx, эффект кэширования совершенно другой.毕竟 это довольно критично.

### Настройка пути кэширования

```nginx
# Настройка пути кэширования
proxy_cache_path /var/cache/nginx levels=1:2 keys_zone=azure_cache:10m
                   max_size=1g inactive=7d use_temp_path=off;
```

Описание параметров:
- `levels=1:2`: уровень каталога кэша, 2-уровневая структура каталога, повышение эффективности доступа к файлам
- `keys_zone=azure_cache:10m`: область хранения ключей кэша, 10МБ достаточно для хранения большого количества ключей
- `max_size=1g`: максимальный размер кэша 1ГБ
- `inactive=7d`: кэшированный файл удаляется, если не accessed 7 дней
- `use_temp_path=off`: прямой запись в каталог кэша, повышение производительности

### Стратегия分级ного кэширования

Разные типы файлов требуют разных стратегий кэширования:

```nginx
# Сервис загрузки Server
server {
    listen 8080;

    # Краткосрочное кэширование index.json (для своевременного обновления)
    location /index.json {
        proxy_cache azure_cache;
        proxy_cache_valid 200 1h;
        proxy_cache_key "$scheme$server_port$request_uri";
        add_header X-Cache-Status $upstream_cache_status;
        add_header Cache-Control "public, max-age=3600";

        # Обратный прокси в Azure OSS
        proxy_pass https://${SERVER_DL_HOST}/${SERVER_DL_CONTAINER}$uri?${SERVER_DL_SAS_TOKEN};
        proxy_ssl_server_name on;
        proxy_ssl_protocols TLSv1.2 TLSv1.3;
    }

    # Долгосрочное кэширование статических файлов, таких как установочные пакеты
    location / {
        proxy_cache azure_cache;
        proxy_cache_valid 200 7d;
        proxy_cache_key "$scheme$server_port$request_uri";
        add_header X-Cache-Status $upstream_cache_status;
        add_header Cache-Control "public, max-age=604800";

        proxy_pass https://${SERVER_DL_HOST}/${SERVER_DL_CONTAINER}$uri?${SERVER_DL_SAS_TOKEN};
        proxy_ssl_server_name on;
        proxy_ssl_protocols TLSv1.2 TLSv1.3;
    }
}
```

**Почему так спроектировано?**

`index.json` — это файл проверки версии, нужно своевременно обновлять, поэтому время кэширования установлено на 1 час. Таким образом, после выпуска новой версии максимум через 1 час пользователи могут обнаружить обновление, не слишком долго.

Статические файлы, такие как установочные пакеты,本来就 редко меняются, кэширование на 7 дней может значительно снизить доступ к исходному серверу. Когда нужно обновить, вручную очистите кэш, не слишком хлопотно.

**Заголовок ответа X-Cache-Status**:

Этот заголовок ответа позволяет вам посмотреть, как обстоят дела с попаданием в кэш, тоже довольно полезно:
- `HIT`: попадание в кэш
- `MISS`: промах кэша, получение с исходного сервера
- `EXPIRED`: кэш истек, повторное получение с исходного сервера
- `BYPASS`: кэш обойден

Метод просмотра:
```bash
curl -I https://server.dl.hagicode.com/app.zip
```

## Анализ затрат

Предположим, что ежемесячный трафик загрузки составляет 1ТБ, давайте посчитаем:

| Схема | Стоимость трафика | Стоимость сервера | Итого |
|------|----------|------------|------|
| Прямое Azure OSS | около ¥500 | ¥0 | ¥500 |
| Облачный сервер + OSS (коэффициент попадания в кэш 80%) | ¥100 + ¥80 | ¥60 | ¥240 |
| Коммерческий CDN | ¥300-500 | ¥0 | ¥300-500 |

**Вывод**: уровень кэширования может сэкономить примерно 50% затрат на распространение.

Этот расчет предполагает, что коэффициент попадания в кэш составляет 80%. В реальной ситуации, если частота обновления файлов невысокая, коэффициент попадания может быть еще выше,毕竟 это тоже естественно.

## Практика развертывания

### Подготовка окружения

Сначала настройте переменные окружения:

```bash
cd /path/to/hagicode_aliyun_deployment/docker
cp .env.example .env
vi .env  # Введите SAS-URL Azure OSS, URL Webhook Lark
```

**Важно**: файл `.env` содержит конфиденциальную информацию (SAS Token, Webhook URL), никогда не commit в систему контроля версий, это довольно критично.

### Настройка DNS

Добавьте следующие A-записи DNS, не забудьте этот шаг:
- `server.dl.hagicode.com` → IP сервера
- `desktop.dl.hagicode.com` → IP сервера

### Инициализация сервера

Используйте Ansible для автоматической инициализации сервера:

```bash
cd /path/to/hagicode_aliyun_deployment
ansible-playbook -i ./ansible/inventory/hosts.yml ./ansible/playbooks/init.yml
```

Этот Playbook автоматически сделает следующее:
- создаст пользователя развертывания
- установит Docker и Docker Compose
- настроит SSH-ключи
- настроит правила брандмауэра

Не слишком сложно,毕竟 автоматизация — это экономия времени и сил.

### Развертывание сервиса

```bash
./deploy.sh
```

Скрипт развертывания поможет вам сделать следующее:
- проверить конфигурацию окружения
- pull последний код
- запустить Docker контейнеры
- выполнить проверку работоспособности
- отправить уведомление о развертывании (Lark)

Одна команда搞定, тоже удобно.

## Советы по эксплуатации

### Управление кэшем

Кэш — это вещь, иногда нужно и ухаживать:

**Просмотр использования диска кэша**:
```bash
docker volume inspect docker_nginx-cache
du -sh /var/lib/docker/volumes/docker_nginx-cache/_data
```

**Ручная очистка кэша**:
```bash
./clear-cache.sh
```

Или выполнить вручную, хотя麻烦一点, но тоже работает:
```bash
docker exec nginx sh -c "rm -rf /var/cache/nginx/*"
docker restart nginx
```

### Ограничения ресурсов

На сервере с 1 ядром 2ГБ конфигурация ограничений ресурсов следующая:

```yaml
services:
  traefik:
    deploy:
      resources:
        limits:
          cpus: '1.50'
          memory: 512M

  nginx:
    deploy:
      resources:
        limits:
          cpus: '0.50'
          memory: 256M
```

Мониторинг использования ресурсов, иногда посмотреть тоже неплохо:
```bash
docker stats
```

### Практика безопасности SAS Token

SAS Token — это учетные данные для доступа к Azure Blob Storage, утечка — это не шутка:

- файл `.env` не commit в систему контроля версий (уже в .gitignore)
- SAS Token установить соответствующее время истечения (рекомендуется 1 год)
- ограничить разрешения SAS Token (только чтение)
- регулярно rotate SAS Token

### Мониторинг и оповещения

HagiCode интегрировал уведомления Lark/飞书 Webhook, может отправлять уведомления в следующих случаях:
- успешное/неуспешное развертывание
- статус очистки кэша
- аномалия сервиса

Уведомления включают информацию о сервере, timestamp, детали ошибок, быстрое定位 проблемы тоже удобнее.

### Высокодоступное расширение

Когда один сервер больше не выдерживает, можно рассмотреть:

1. **Горизонтальное масштабирование**: развернуть несколько узлов, распределение через DNS round-robin или балансировку нагрузки
2. **Поддержка CDN**: подключить CDN перед облачным сервером,进一步提升 скорость доступа
3. **Предварительный нагрев кэша**: использовать скрипт для предварительной загрузки популярных файлов в кэш

## Меры предосторожности

Несколько моментов все же нужно напомнить,毕竟 никто не хочет встретить проблемы:

1. **SSL-сертификаты**: Let's Encrypt имеет ограничения скорости, не часто переключайте развертывание, иначе можете не подать заявку на сертификат
2. **Очистка кэша**: после обновления важных файлов не забудьте очистить кэш, иначе пользователи могут не загрузить новую версию
3. **Управление журналами**: регулярно очищайте журналы Docker, иначе диск заполнится — это проблема
4. **Стратегия резервного копирования**: Traefik acme.json, конфигурация Bunker Web и т. д., все же лучше сделать резервную копию
5. **Мониторинг и оповещения**: настройте уведомления Lark, своевременно узнавайте статус развертывания, при возникновении проблем можно быстро реагировать

## Резюме

Облачный сервер + уровень кэширования Nginx, просто и ясно. HagiCode использует эту схему, затраты не слишком высоки (стоимость сервера около 60-100 юаней/месяц), эффект还挺不错的. Основные преимущества примерно следующие:

- **Контролируемые затраты**: по сравнению с прямым использованием облачного хранилища или коммерческого CDN затраты снизились примерно на 50%
- **Гибкое развертывание**: Traefik или Bunker Web, выбирайте сами
- **Масштабируемость**: если нужно можно масштабировать по горизонтали, или добавить CDN
- **Простая эксплуатация**: Shell-скрипты + Ansible, автоматическое развертывание тоже удобно

Для небольших команд и индивидуальных разработчиков, которым нужно распространение файлов, эта схема может быть попробована.

HagiCode использует эту архитектуру в производственной среде стабильно в течение некоторого времени, глобальные пользователи скачивают без серьезных проблем. Если вы тоже ищете похожее решение,不妨 попробуйте,说不定 вам тоже поможет.
