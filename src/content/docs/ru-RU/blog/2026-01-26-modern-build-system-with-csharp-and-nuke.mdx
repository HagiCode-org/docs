## Прощай, ад скриптов: Почему мы выбрали C# для создания современной системы сборки

> Раскрываем, как проект HagiCode использует Nuke для реализации типобезопасного, кроссплатформенного и высокорасширяемого процесса автоматизированной сборки, окончательно решая проблемы обслуживания традиционных сборочных скриптов.

## Предыстория

В долгом путешествии разработки программного обеспечения слово "сборка" часто вызывает смешанные чувства. Любовь — один клик, и код превращается в продукт, это самый очаровательный момент для программиста; ненависть — поддержание этой кучи беспорядочных скриптов сборки — просто кошмар.

Во многих проектах мы привыкли писать скрипты на Python или использовать файлы конфигурации XML (представьте тот ужас, который支配 `<property>`). Но с ростом сложности проекта, особенно в таких как HagiCode, который включает разработку внешнего и внутреннего интерфейса, мультиплатформенность и смешанное программирование на нескольких языках, традиционные методы сборки начинают показывать несостоятельность. Логика скриптов разбросана, отсутствует проверка типов, слабая поддержка IDE... Эти проблемы как маленькие ямы, в которые команда разработки постоянно спотыкается.

Для решения этих болей в проекте HagiCode мы решили внедрить **Nuke** — современную систему сборки на основе C#. Это не просто инструмент, скорее переосмысление процесса сборки. Сегодня давайте поговорим о том, почему мы выбрали его и как он сделал наш опыт разработки "взлётным".

## Основной анализ: Почему Nuke?

Вы можете подумать: "Ой, систем сборки так много, например Make, Gradle, можно даже просто использовать Shell скрипты, зачем обязательно использовать C#?"

Это на самом деле хороший вопрос. Основная прелесть Nuke заключается в том, что он приносит самые знакомые характеристики языка программирования в мир скриптов сборки.

### 1. Модулизация процесса сборки: Искусство Target

Философия дизайна Nuke очень ясна: **Всё является целью**.

В традиционных скриптах мы можем написать сотни строк линейно выполняемого кода со сложной логикой. В Nuke мы разбиваем процесс сборки на независимые `Target` (цели). Каждая цель отвечает только за одну вещь, например:

- `Clean`: Очистка выходных каталогов
- `Restore`: Восстановление пакетов зависимостей
- `Compile`: Компиляция кода
- `Test`: Запуск модульных тестов

Этот дизайн очень соответствует принципу единственной ответственности. Как кубики Lego, мы можем свободно комбинировать эти Target. Что более важно, Nuke позволяет нам определять отношения зависимостей между Target. Например, если вы хотите `Test`, система автоматически проверит, выполнили ли вы сначала `Compile`; чтобы выполнить `Compile`, естественно нужно сначала `Restore`.

Этот граф зависимостей не только делает логику более ясной, но и значительно повышает эффективность выполнения, Nuke автоматически анализирует оптимальный путь выполнения.

### 2. Типобезопасность: Прощай, кошмар опечаток

Друзья, использовавшие Python для написания скриптов сборки, наверняка сталкивались с таким неловкостью: скрипт выполняется пять минут, в конце ошибка говорит `Confi.guration` с опечаткой, или передали строку параметру, который должен быть числом.

Самое большое преимущество написания скриптов сборки на C# — это **типобезопасность**. Это означает:

- **Проверка во время компиляции**: Когда вы пишете код, IDE сразу скажет вам, где ошибка, не нужно ждать времени выполнения.
- **Рефакторинг без забот**: Если вы хотите изменить имя переменной или метода, рефакторинг IDE сделает это одним кликом, не нужно глобальный поиск и замена с опасением.
- **Умные подсказки**: Мощный IntelliSense автоматически дополняет код, не нужно листать документацию для запоминания редких API.

### 3. Кроссплатформенность: Единый опыт сборки

Раньше писали `.bat` на Windows, `.sh` на Linux, для совместимости обоих приходилось писать ещё и Python скрипт. Теперь, там где может работать .NET Core (сейчас .NET 5+), там может работать Nuke.

Это означает, что независимо от того, используют ли члены команды Windows, Linux или macOS, используют ли Visual Studio, VS Code или Rider, все выполняют один и тот же набор логики. Это значительно устраняет проблемы, вызванные различиями окружения типа "у меня на машине работает".

### 4. Управление параметрами и конфигурацией

Nuke предоставляет очень элегантный механизм разбора параметров. Вам не нужно вручную разбирать `string[] args`, просто определите свойство, добавьте характеристику `[Parameter]`, Nuke автоматически обработает отображение параметров командной строки и файлов конфигурации.

Например, мы можем легко определить конфигурацию сборки:

```csharp
[Parameter("Configuration to build - Default is 'Debug'")]
readonly Configuration BuildConfiguration = IsLocalBuild ? Configuration.Debug : Configuration.Release;

Target Compile => _ => _
    .DependsOn(Restore)
    .Executes(() =>
    {
        // Здесь используем BuildConfiguration, он типобезопасный
        DotNetBuild(s => s
            .SetConfiguration(BuildConfiguration)
            .SetProjectFile(SolutionFile));
    });
```

Этот способ написания интуитивно понятен и не подвержен ошибкам.

## Практическое руководство: Как внедрить в проект

Пустые разговоры вредны стране, реальные дела процветают. Давайте посмотрим, как в проекте HagiCode конкретно внедряется это решение.

### 1. Планирование структуры проекта

Мы не хотим, чтобы скрипты сборки загрязняли корневой каталог проекта, и не хотим делать как некоторые Java проекты с непостижимой структурой каталогов. Поэтому мы поместили все файлы сборки, связанные с Nuke, в единую папку `nukeBuild/`.

Преимущества этого:
- Корневой каталог проекта остаётся чистым.
- Логика сборки когерентна, удобна для управления.
- Когда новые участники присоединяются, сразу видят "ага, это логика, связанная со сборкой".

### 2. Проектирование чёткой цепочки зависимостей Target

При проектировании Target мы следуем принципу: **атомизация + поток зависимостей**.

Каждый Target должен быть достаточно мал, делать только одну вещь. Например `Clean` только удаляет файлы, не нужно в нём заодно делать упаковку.

Рекомендуемый поток зависимостей примерно такой:

`Clean` -> `Restore` -> `Compile` -> `Test` -> `Pack`

Конечно, это не абсолютно. Например, если вы хотите только запустить тест, не нужно упаковывать, Nuke позволяет вам напрямую выполнить `nuke Test`, он автоматически обработает предыдущие шаги Restore и Compile.

### 3. Совершенная обработка ошибок и логирование

Чего больше всего боятся скрипты сборки? Неоднозначные сообщения об ошибках. Например, сборка не удалась, лог только показывает "Error: 1", это очень бесит.

В Nuke, так как мы можем напрямую использовать механизм обработки исключений C#, можно очень точно перехватывать и сообщать об ошибках.

```csharp
Target Publish => _ => _
    .DependsOn(Test)
    .Executes(() =>
    {
        try 
        {
            // Пытаемся опубликовать в NuGet
            DotNetNuGetPush(s => s
                .SetTargetPath(ArtifactPath)
                .SetSource("https://api.nuget.org/v3/index.json")
                .SetApiKey(ApiKey));
        }
        catch (Exception ex)
        {
            Log.Error($"Публикация не удалась, ребята проверьте, правильно ли Key: {ex.Message}");
            throw; // Убедиться, что процесс сборки завершается с ненулевым кодом выхода
        }
    });
```

### 4. Интеграционные тесты для гарантии качества

Скрипты сборки сами по себе тоже код, тоже нуждаются в тестах. Nuke позволяет нам писать тесты для процесса сборки, чтобы гарантировать, что после изменения логики сборки мы не разрушим существующий процесс публикации. Это особенно важно в конвейере непрерывной интеграции (CI).

## Заключение

С внедрением Nuke процесс сборки HagiCode стал беспрецедентно плавным. Это не просто замена инструмента, а повышение инженерного мышления.

**Что мы получили?**
- **Поддерживаемость**: код как конфигурация, логика ясна, новые люди тоже могут быстро начать.
- **Стабильность**: сильная типизация сократила более 90% простых ошибок.
- **Согласованность**: единый кроссплатформенный опыт, устранил различия окружения.

Если раньше написание скриптов сборки было "поиском в темноте", то использование Nuke как "ходьба ночью с включённым светом". Если вам надоело поддерживать эти трудно отлаживаемые скриптовые языки, попробуйте перенести логику сборки в мир C#, может быть вы обнаружите, что сборка тоже может быть такой элегантной.
