## Практика шардирования SQLite: глубокое сравнение трёх стратегий шардирования

> Когда однофайловый SQLite сталкивается с проблемами конкурентности, как нам решить эту проблему? В этой статье мы делимся тремя различными сценариями шардирования SQLite в проекте HagiCode, помогая вам понять, как выбрать подходящую стратегию шардирования.

Всем привет, я продюсер HagiCode Юй Кунь.

## Предыстория

При создании высокопроизводительных приложений однофайловая база данных SQLite сталкивается с весьма реальными проблемами. С ростом количества пользователей и объёма данных эти ситуации начинают накапливаться:

- Операции записи начинают вставать в очередь, время ответа визуально увеличивается
- Производительность запросов падает с ростом данных
- При многопоточном доступе часто возникает ошибка "database is locked"

Первая реакция многих: не стоит ли сразу мигрировать на PostgreSQL или MySQL? Хотя это решит проблему, сложность развертывания резко возрастёт. Есть ли более лёгкое решение?

Ответ: шардирование. В конечном счёте, инженерные проблемы требуют инженерных методов решения. Распределяя данные по нескольким файлам SQLite, можно значительно повысить конкурентную способность и производительность запросов, сохраняя при этом лёгкость SQLite.

## Обзор трёх схем шардирования

После анализа кодовой базы HagiCode мы обнаружили три схемы шардирования SQLite для различных бизнес-сценариев:

1. **Шардирование сообщений сессий (Session Message)**: хранение сообщений AI-диалогов, характерная черта — частые записи, изолированные запросы на основе сессий
2. **Шардирование Orleans Grain**: персистентность состояния распределённого фреймворка, характерная черта — межузловой доступ, необходимость детерминированной маршрутизации
3. **Шардирование Hero History**: запись исторических событий геймифицированной системы, характерная черта — событие溯源, необходимость совместимости миграции

Хотя бизнес-сценарии различаются, все три следуют одним и тем же основным принципам проектирования:

- **Детерминированная маршрутизация**: прямой расчёт шарда из бизнес-ID, без метаданных таблиц
- **Прозрачный доступ**: верхний уровень работает через единый интерфейс, не замечая существование шардов
- **Независимое хранение**: каждый шард — полностью независимый файл SQLite
- **Оптимизация конкурентности**: режим WAL + busy_timeout снижают конкуренцию за блокировки

Многие спросят: почему не сделать универсальную схему шардирования? Этот вопрос очень практичный, сразу перейдём к выводу: в инженерии нет универсальных решений, есть только решения, максимально соответствующие текущему бизнес-сценарию. Далее мы глубоко сравним конкретную реализацию этих трёх схем.

## Сравнение стратегий шардирования

### Количество и правила именования шардов

| Аспект | Session Message | Orleans Grain | Hero History |
|------|----------------|---------------|--------------|
| **Количество шардов** | 256 (16²) | 100 | 10 |
| **Правило именования** | 16-ричное (00-ff) | 10-ричное (00-99) | 10-ричное (0-9) |
| **Каталог хранения** | `DataDir/messages/` | `DataDir/orleans/grains/` | `DataDir/hero-history/` |
| **Шаблон имени файла** | `{shard}.db` | `grains-{shard}.db` | `{shard}.db` |

Почему количество шардов так сильно различается? Это зависит от характеристик бизнеса. Другими словами, модели меняются, инструменты обновляются, рабочие процессы улучшаются, но инженерная база остаётся: сначала нужно понять, какую проблему вы решаете.

- **Session Message** использует 256 шардов, потому что частота записей диалоговых сообщений самая высокая, нужно больше шардов для распределения нагрузки
- **Orleans Grain** использует 100 шардов, балансируя между производительностью конкурентности и сложностью управления
- **Hero History** использует только 10 шардов, потому что частота записей исторических событий ниже, и нужно учесть стоимость миграции

### Различия в алгоритмах маршрутизации

Алгоритм маршрутизации — это ядро схемы шардирования, он определяет, как данные распределяются по шардам. Три схемы используют различные стратегии маршрутизации:

```csharp
// Session Message: последние две цифры GUID в 16-ричной системе
var normalized = Guid.Parse(sessionId.Value).ToString("N").ToLowerInvariant();
return normalized[^2..];  // берём последние два 16-ричных символа

// Orleans Grain: извлекаем две последние цифры и берём модуль
var digits = ExtractDigits(grainId);  // извлекаем все цифры
var lastTwoDigits = (digits[^2] * 10) + digits[^1];
return lastTwoDigits % shardCount;

// Hero History: модуль значения ASCII последнего символа
return heroId[^1] % 10;
```

**Анализ дизайна**:

- **Session Message**: ID — это GUID, после преобразования в 16-ричную систему берём последние две цифры, можно получить равномерно распределённые 256 шардов
- **Orleans Grain**: формат ID не единый, может содержать буквы и цифры, поэтому извлекаем все цифры и берём модуль
- **Hero History**: ID — это строка, напрямую берём модуль значения ASCII последнего символа, просто, но распределение может быть недостаточно равномерным

**Ключевой момент**: независимо от того, какой алгоритм используется, необходимо гарантировать, что один и тот же ID всегда отображается на один и тот же шард. Это базовое требование в распределённых системах, иначе это приведёт к несогласованности данных. В конечном счёте, нестабильная маршрутизация сводит все усилия к нулю.

### Различия в стратегиях инициализации

| Аспект | Session Message | Orleans Grain | Hero History |
|------|----------------|---------------|--------------|
| **Момент инициализации** | Ленивая загрузка по требованию | Полная параллельная инициализация при запуске | Ленивая загрузка по требованию |
| **Контроль конкурентности** | `Lazy<Task>` для предотвращения повторной инициализации | `Parallel.ForEachAsync` | `Lazy<Task>` для предотвращения повторной инициализации |

**Почему Orleans Grain выбирает полную инициализацию при запуске?**

Потому что Orleans — это распределённый фреймворк, Grain может быть запланирован на любом узле. Если во время выполнения обнаруживается, что файл шарда не существует, это приведёт к отказу запроса. Полная инициализация при запуске, хотя и увеличивает время запуска, но гарантирует стабильность во время выполнения. Просто запустить — это только начало, поддерживать —这才是本事.

**Преимущества ленивой загрузки**:

Для Session Message и Hero History ленивая загрузка может уменьшить время запуска, файл создаётся и схема инициализируется только при реальном необходимости доступа к шарду. Использование `Lazy<Task>` может предотвратить состояние гонки при параллельной инициализации. Этот дизайн кажется простым, но в реальном проекте может сэкономить много лишних проблем.

### Особенности дизайна схемы

Дизайн схемы трёх решений отражает их бизнес-характеристики:

**Session Message**:
- Поддержка паттерна Event Sourcing (таблица событий + таблица снимков)
- Включает подтаблицу блоков содержимого сообщений (MessageContentBlocks)
- Имеет поля сжатия и отметки сжатия, поддерживает последующую оптимизацию

**Orleans Grain**:
- Простейший дизайн: одна таблица GrainState
- Хранение состояния в формате JSON
- Оптимистичный контроль конкурентности ETag

**Hero History**:
- Индексы для оптимизации запросов временной шкалы
- Уникальное ограничение DedupeKey для предотвращения дублирования
- Поддержка различных типов событий и состояний

Из этих дизайнов можно видеть, что дизайн схемы должен тесно соответствовать бизнес-требованиям, а не преследовать универсальность. Простой дизайн Orleans Grain именно потому, что ему нужно хранить только сериализованное состояние, без сложных возможностей запросов. 这不是玄学，是工程。Не спешите давать слишком большие имена, сначала посмотрите, переживет ли这个东西 две итерации в команде.

### Сравнение конфигурации конкурентности

Все три схемы используют одинаковую конфигурацию оптимизации конкурентности SQLite:

```sql
PRAGMA journal_mode=WAL;      -- режим журналирования перед записью
PRAGMA synchronous=NORMAL;     -- снижение затрат на персистентность
PRAGMA busy_timeout=5000;      -- 5 секунд ожидания занятости
PRAGMA foreign_keys=ON;        -- ограничения внешних ключей
```

**Преимущества режима WAL**:

Традиционный режим журнала отката создаёт конкуренцию за блокировки при записи, тогда как режим WAL позволяет читать и писать параллельно. Это может значительно повысить производительность в сценариях с большими данными. Многие не знают об этой конфигурации, на самом деле она важнее, чем вы думаете.

**Компромисс synchronous=NORMAL**:

Установка на FULL может гарантировать максимальную безопасность, но значительно снизит производительность. Режим NORMAL достигает баланса между безопасностью и производительностью, для большинства приложений это подходящий выбор. Эту конфигурацию не нужно долго мучиться, NORMAL достаточно.

## Как выбрать стратегию шардирования

На основе анализа трёх решений HagiCode мы можем обобщить следующую матрицу решений:

```
Сценарии с высокой пропускной способностью → больше шардов (например, Message использует 256)
Простота обслуживания → меньше шардов (например, Hero History использует 10)
В основном цифровые ID → алгоритм модуля (Orleans Grain)
В основном GUID → 16-ричный суффикс (Session Message)
Строковые ID → модуль ASCII (Hero History)
```

**Опытные значения выбора количества шардов**:

- Слишком мало (< 10): ограниченный прирост конкурентности, шардирование мало смысла
- Слишком много (> 1000): сложное управление файлами, большие накладные расходы пула соединений
- Опытные значения: 10-100 шардов适用于大多数 сценариев
- Сценарии с экстремальной конкурентностью: можно рассмотреть 256 шардов

Если вы смотрите только на демо, это легко может вызвать энтузиазм; но как только попадёте в производственную среду, счёт нужно считать清楚.Многие проблемы не могут быть сделаны, просто не посчитали цену.

## Практическое руководство

### Реализация стандартного роутера шардов

```csharp
public interface IShardResolver<TId>
{
    string ResolveShardKey(TId id);
}

// 16-ричное шардирование (适用于 GUID)
public class HexSuffixShardResolver : IShardResolver<string>
{
    private readonly int _suffixLength;

    public HexSuffixShardResolver(int suffixLength = 2)
    {
        _suffixLength = suffixLength;
    }

    public string ResolveShardKey(string id)
    {
        var normalized = id.Replace("-", "").ToLowerInvariant();
        return normalized[^_suffixLength..];
    }
}

// Шардирование по модулю чисел (适用于 чисто цифровых ID)
public class NumericModuloShardResolver : IShardResolver<long>
{
    private readonly int _shardCount;

    public NumericModuloShardResolver(int shardCount)
    {
        _shardCount = shardCount;
    }

    public string ResolveShardKey(long id)
    {
        return (id % _shardCount).ToString("D2");
    }
}
```

### Паттерн единой фабрики соединений

```csharp
public class ShardedConnectionFactory<TOptions>
{
    private readonly ConcurrentDictionary<string, Lazy<Task>> _initializationTasks = new();
    private readonly TOptions _options;
    private readonly IShardSchemaInitializer _initializer;

    public ShardedConnectionFactory(
        TOptions options,
        IShardSchemaInitializer initializer)
    {
        _options = options;
        _initializer = initializer;
    }

    public async Task<TDbContext> CreateAsync(string shardKey, CancellationToken ct)
    {
        var connectionString = BuildConnectionString(shardKey);

        // Используем Lazy<Task> для предотвращения параллельной инициализации
        var initTask = _initializationTasks.GetOrAdd(
            connectionString,
            _ => new Lazy<Task>(() => InitializeShardAsync(connectionString, ct))
        );

        await initTask.Value;
        return CreateDbContext(connectionString);
    }

    private async Task InitializeShardAsync(string connectionString, CancellationToken ct)
    {
        await _initializer.InitializeAsync(connectionString, ct);
    }

    private string BuildConnectionString(string shardKey)
    {
        var shardPath = Path.Combine(_options.BaseDirectory, $"{shardKey}.db");
        return $"Data Source={shardPath}";
    }

    private TDbContext CreateDbContext(string connectionString)
    {
        // Создаём DbContext на основе конкретного ORM
        return Activator.CreateInstance(typeof(TDbContext), connectionString) as TDbContext;
    }
}
```

### Лучшие практики инициализации схемы

```csharp
public class SqliteShardInitializer : IShardSchemaInitializer
{
    public async Task InitializeAsync(string connectionString, CancellationToken ct)
    {
        await using var connection = new SqliteConnection(connectionString);
        await connection.OpenAsync(ct);

        // Конфигурация оптимизации конкурентности
        await connection.ExecuteAsync("""
            PRAGMA journal_mode=WAL;
            PRAGMA synchronous=NORMAL;
            PRAGMA busy_timeout=5000;
            PRAGMA foreign_keys=ON;
        """);

        // Создаём структуру таблицы
        await connection.ExecuteAsync("""
            CREATE TABLE IF NOT EXISTS Entities (
                Id TEXT PRIMARY KEY,
                CreatedAt TEXT NOT NULL,
                UpdatedAt TEXT NOT NULL,
                Data TEXT NOT NULL,
                ETag TEXT
            );
        """);

        // Создаём индексы
        await connection.ExecuteAsync("""
            CREATE INDEX IF NOT EXISTS IX_Entities_CreatedAt
            ON Entities(CreatedAt DESC);

            CREATE INDEX IF NOT EXISTS IX_Entities_UpdatedAt
            ON Entities(UpdatedAt DESC);
        """);
    }
}
```

### Ключевые моменты внимания

**1. Стабильность маршрутизации**

Алгоритм маршрутизации должен гарантировать, что один и тот же ID всегда отображается на один и тот же шард. Избегайте использования случайных или зависящих от времени вычислений, а также не вводите изменяемые параметры в алгоритм.

**2. Выбор количества шардов**

Количество шардов должно быть определено на этапе проектирования, последующее изменение очень сложно. Нужно учесть:
- Текущий и будущий объём конкурентности
- Стоимость управления одним шардом
- Сложность миграции данных

**3. Учёт миграции**

Решение Hero History демонстрирует полный путь миграции:

1. Создаём инфраструктуру шардированного хранения
2. Реализуем сервис миграции для копирования данных из главной базы в шарды
3. Проверяем совместимость запросов после миграции
4. Переключаем пути чтения-записи на шарды
5. Очищаем старые таблицы главной базы

При проектировании схемы шардирования нужно учесть будущие потребности миграции. Talk is cheap. Show me the code，но только код недостаточно, вам ещё нужен полный путь миграции.Один успех — не система, постоянный успех — это система.

**4. Мониторинг и эксплуатация**

- Мониторьте распределение размеров каждого шарда, своевременно обнаруживайте неравномерность данных
- Установите сигнализацию для обнаружения горячих точек шардов, избегайте, чтобы один шард стал узким местом
- Регулярно проверяйте размер файлов WAL, предотвращайте чрезмерное занятие дискового пространства
- Создайте механизм проверки здоровья шардов

**5. Покрытие тестированием**

- Тестируйте граничные условия (пустой ID, специальные символы, сверхдлинный ID)
- Проверяйте детерминизм маршрутизации, убедитесь, что один и тот же ID всегда отображается на один и тот же шард
- Тесты на нагрузку параллельной записи, проверяйте, что конкуренция за блокировки эффективно снижена
- Тесты миграции, убедитесь в целостности и согласованности данных

## Заключение

Сравнивая три схемы шардирования SQLite в проекте HagiCode, мы видим:

1. **Нет универсального решения**: различные бизнес-сценарии требуют различных стратегий шардирования
2. **Основные принципы универсальны**: детерминированная маршрутизация, прозрачный доступ, независимое хранение, оптимизация конкурентности
3. **Дизайн должен быть ориентирован на будущее**: учитывайте путь миграции и стоимость эксплуатации

Если ваш проект использует SQLite и начинает сталкиваться с проблемами конкурентности, надеюсь, эта статья предоставит вам некоторые идеи. Не нужно торопиться мигрировать на тяжёлые базы данных, иногда подходящая схема шардирования может решить проблему.

Конечно, шардирование — не серебряная пуля. Прежде чем выбрать схему шардирования, убедитесь:
- Вы уже оптимизировали производительность запросов одной таблицы
- Вы уже использовали подходящие индексы
- Вы уже включили режим WAL

Только когда все эти оптимизации выполнены, но проблемы с производительностью сохраняются, следует рассмотреть введение шардирования. Способность делать простые вещи хорошо — это уже само по себе способность.
