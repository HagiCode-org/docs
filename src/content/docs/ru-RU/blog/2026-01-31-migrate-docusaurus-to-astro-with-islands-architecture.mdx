## С Docusaurus 3.x на Astro 5.x: ретроспектива миграции сайта HagiCode

> В этой статье мы анализируем полный процесс миграции официального сайта HagiCode с Docusaurus 3.x на Astro 5.x. Мы подробно рассмотрим, как Islands-архитектура Astro помогла нам решить проблемы производительности, сохранив при этом существующие React-компоненты и обеспечив улучшения как скорости сборки, так и загрузки страниц.

## Предпосылки

В январе 2026 года мы провели «пересадку сердца» официального сайта HagiCode — полностью мигрировали основной фреймворк с Docusaurus 3.x на Astro 5.x. Это был не импульсивный рефакторинг, а тщательно обдуманное техническое решение.

До миграции наш сайт был функционально завершён, но начал проявлять некоторые «богатые проблемы»: раздутые артефакты сборки, избыточная JavaScript-нагрузка и менее чем идеальная скорость загрузки на сложных документационных страницах. Как проект AI-ассистента программирования, HagiCode нуждается в частых обновлениях документации и описания функций, поэтому эффективность сборки напрямую влияет на скорость публикации. Одновременно мы хотели сделать сайт более дружелюбным для поисковых систем (SEO), чтобы больше разработчиков могли обнаружить проект.

Для решения этих проблем мы приняли смелое решение: полностью перестроить систему на Astro. Последствия этого решения могут быть больше, чем вы ожидаете — об этом подробнее ниже.

## О HagiCode

Подход к миграции сайта, описанный в этой статье, основан на нашем практическом опыте в проекте [HagiCode](https://github.com/HagiCode-org/site).

HagiCode — это AI-ассистент программирования, направленный на повышение эффективности разработки. Мы заботимся не только об итерации основных функций, но и о впечатлении разработчика. Этот рефакторинг сайта также предназначен для предоставления пользователям максимально быстрого опыта при просмотре документации и официального сайта.

## Почему мы отказались от зрелого Docusaurus?

В экосистеме React Docusaurus давно был «стандартным ответом» для документационных сайтов. Он работает из коробки, имеет богатый плагин-экосистему и активное сообщество. Но по мере роста функций HagiCode мы также ощутили его ограничения:

1.  **Узкие места производительности**: Docusaurus по сути является React SPA (одностраничным приложением). Даже если вы пишете только статические страницы, клиенту всё равно нужно загружать React runtime и выполнять гидратацию, что излишне тяжело для простых документационных страниц.
2.  **Большой размер ресурсов**: даже при минимальном содержимом страницы размер связанного JS остаётся относительно фиксированным. Это неоптимально для мобильных пользователей и при плохом сетевом соединении.
3.  **Ограниченная гибкость**: хотя Docusaurus расширяем, мы хотели больше низкоуровневого контроля над конвейером сборки.

Astro появился как раз вовремя для решения этих проблем. Он представил новую «Islands-архитектуру»: по умолчанию Astro генерирует статический HTML с нулевым JavaScript, и только компоненты, требующие интерактивности, «активируются» и загружают JS. Это означает, что большая часть нашего сайта становится чистым HTML и загружается крайне быстро.

## Основная стратегия миграции: плавный архитектурный переход

Миграция — это не просто копирование и вставка. Она требует смены образа мышления. Мы перешли от модели «всё на React» Docusaurus к модели «Core + Islands» Astro.

### 1. Перестройка системы конфигурации

Сначала нужно было перейти от `docusaurus.config.ts` к `astro.config.mjs`. Это не просто переименование файла, а перезапись логики маршрутизации и сборки.

В Docusaurus всё — это плагины. В Astro всё — это интеграции. Нам нужно было переопределить базовый путь сайта, режим вывода сборки (статический vs SSR) и стратегию оптимизации ресурсов.

**До миграции:**
```typescript
// docusaurus.config.ts
export default {
  title: 'HagiCode',
  url: 'https://hagicode.com',
  baseUrl: '/',
  // ... дополнительная конфигурация
};
```

**После миграции:**
```javascript
// astro.config.mjs
import { defineConfig } from 'astro/config';
import react from '@astrojs/react';

export default defineConfig({
  integrations: [react()],
  site: 'https://hagicode.com',
  base: '/',
  // Настройки оптимизации статических ресурсов
  build: {
    inlineStylesheets: 'auto',
  },
});
```

### 2. Что сохранить, а что переделать в React-компонентах

Это была самая болезненная часть миграции. На нашем существующем сайте было много React-компонентов: Tabs, подсветка кода, кнопка обратной связи и другие. Отказаться от них — жалко, но оставлять всё — означает слишком тяжёлую JS-нагрузку.

HagiCode принял стратегию **прогрессивной гидратации**:
*   **Чисто статические компоненты**: для отображаемого контента (заголовки, подвалы, простые текстовые документы) мы переписали их как Astro-компоненты (`.astro`-файлы), отрендерив их в HTML на этапе сборки.
*   **Интерактивные острова**: для компонентов, которые должны оставаться интерактивными (переключатель тем, переключение вкладок, кнопка копирования кода), мы сохранили React-реализацию и добавили директивы `client:load` или `client:visible`.

Например, наш часто используемый компонент Tabs в документации:

```jsx
// src/components/Tabs.jsx
import { useState } from 'react';
import './Tabs.css'; // Импорт стилей

export default function Tabs({ items }) {
  const [activeIndex, setActiveIndex] = useState(0);
  // ... логика состояния
  return (
    <div className="tabs-wrapper">
      {/* Логика рендеринга */}
    </div>
  );
}
```

При использовании в Markdown мы явно указываем Astro: «Этому компоненту нужен JS»:

```markdown
// src/content/docs/example.mdx
import Tabs from '../../components/Tabs.jsx';

<!-- Загружаем JS только при входе компонента в область видимости -->
<Tabs client:visible items={...} />
```

Таким образом, интерактивные компоненты за пределами видимой области не конкурируют за пропускную способность, что значительно улучшает скорость загрузки первого экрана.

### 3. Адаптация системы стилей: от CSS Modules к Scoped CSS

Docusaurus по умолчанию поддерживает CSS Modules, а Astro поощряет Scoped CSS через тег `<style>`. Основная идея обоих — изоляция стилей, но синтаксис различается.

При миграции HagiCode мы преобразовали большинство сложных CSS Modules в scoped-стили Astro. На самом деле это оказалось к лучшему, поскольку в `.astro`-файлах стили и шаблоны находятся в одном файле, что делает обслуживание более наглядным.

**До рефакторинга:**
```css
/* Tabs.module.css */
.wrapper { background: var(--ifm-background-color); }
```

**После рефакторинга (Astro Scoped):**
```html
<!-- Tabs.astro -->
<div class="tabs-wrapper">
  <slot />
</div>

<style>
  .tabs-wrapper {
    /* Используем CSS-переменные напрямую для адаптации к теме */
    background: var(--bg-color);
    padding: 1rem;
  }
</style>
```

Одновременно мы унифицировали систему глобальных CSS-переменных и использовали возможности Astro по определению окружения для обеспечения плавного переключения тёмного режима между страницами.

## Подводные камни на практике и их решение

В процессе реальной миграции HagiCode мы столкнулись с рядом проблем. Вот несколько наиболее типичных.

### 1. Проблемы с путями и переменными окружения

HagiCode поддерживает развёртывание по подпутям, например в подкаталоге GitHub Pages. В Docusaurus `baseUrl` обрабатывается автоматически. В Astro же нужно быть осторожнее при работе с ссылками на изображения и API-запросами.

Мы ввели механизм переменных окружения для унифицированного управления:

```javascript
// Обработка путей в скрипте сборки
const getBasePath = () => import.meta.env.VITE_SITE_BASE || '/';
```

Обязательно не хардкодьте пути, начинающиеся с `/`. В разработке и проде, а также после настройки `base`-пути, это может привести к 404 для ресурсов.

### 2. Совместимость CommonJS-скриптов

На нашем старом сайте были Node.js-скрипты (для автоматического сбора данных метрик, обновления sitemap и т.д.), написанные на CommonJS (`require`). Astro и современные инструменты сборки полностью перешли на ES Modules (`import`/`export`).

Если у вас есть аналогичные скрипты, не забудьте рефакторить их все в ES Modules. Это общее направление экосистемы, и чем раньше вы это сделаете, тем меньше проблем будет позже.

```javascript
// Старый способ
const fs = require('fs');

// Новый способ
import fs from 'fs';
```

### 3. Не забудьте о SEO и перенаправлениях

Поисковые системы уже проиндексировали старые страницы HagiCode на Docusaurus. Если переключиться на Astro напрямую и изменится структура URL, это приведёт к множеству 404 и значительному падению ранжирования.

Мы настроили правила перенаправлений в Astro:

```javascript
// astro.config.mjs
export default defineConfig({
  redirects: {
    '/docs/old-path': '/docs/new-path',
    // Массовое сопоставление старых ссылок с новыми
  }
});
```

Или можно обработать это на уровне конфигурации сервера. Убедитесь, что старые ссылки перенаправляются по 301 на новые адреса — это критически важно для SEO.

## Заключение

Для HagiCode миграция с Docusaurus на Astro была не просто обновлением фреймворка, но и практической реализацией философии «производительность прежде всего».

**Что мы получили:**
*   **Отличные оценки Lighthouse**: после миграции оценка производительности сайта HagiCode легко приблизилась к максимуму.
*   **Более быстрая сборка**: инкрементальная сборка Astro сократила время публикации обновлений документации вдвое.
*   **Сохранённая гибкость**: благодаря Islands-архитектуре мы не пожертвовали ни одной интерактивной функцией и по-прежнему можем использовать React там, где это необходимо.

Если вы также поддерживаете документационный сайт и боретесь с размером бандла или скоростью загрузки, Astro определённо стоит попробовать. Хотя процесс миграции требует некоторых усилий, гладкий пользовательский опыт, который вы получаете взамен, безусловно, оправдывает их.
