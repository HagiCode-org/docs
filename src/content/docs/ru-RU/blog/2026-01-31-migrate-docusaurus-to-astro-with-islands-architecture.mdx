## От Docusaurus 3.x к Astro 5.x: практический анализ миграции сайта HagiCode

> В этой статье мы анализируем полный процесс миграции официального сайта HagiCode с Docusaurus 3.x на Astro 5.x. Мы подробно рассмотрим, как с помощью архитектуры Islands от Astro решить проблемы с производительностью, сохранив при этом существующие React-компоненты и достигнув двойного улучшения скорости сборки и загрузки.


## Предыстория

В январе 2026 года мы провели операцию "пересадки сердца" на официальном сайте HagiCode — полностью мигрировали основную инфраструктуру с Docusaurus 3.x на Astro 5.x. Это не было импульсивным масштабным рефакторингом, а продуманным техническим решением.

До миграции наш сайт, хотя и был функционально полным, постепенно проявлял некоторые "болезни богатства": громоздкие сборочные артефакты, чрезмерная нагрузка JavaScript, и недостаточная скорость загрузки страниц на сложных документационных страницах. Как проект AI-код-ассистента, HagiCode требует частого обновления документации и описаний функций, и эффективность сборки напрямую влияет на скорость публикации. В то же время мы хотели, чтобы сайт был более дружелюбен к поисковым системам (SEO), чтобы больше разработчиков могли обнаружить этот проект.

Для решения этих проблем мы приняли смелое решение: полностью перестроить систему сборки и мигрировать на Astro. Изменения, принесшие этим решением, оказались больше, чем можно представить — об этом я расскажу подробнее позже.

## О HagiCode

План миграции сайта, представленный в этой статье, основан на нашем практическом опыте в проекте [HagiCode](https://github.com/HagiCode-org/site).

HagiCode — это AI-код-ассистент, посвящённый повышению эффективности разработки. Мы уделяем внимание не только итерации основных функций, но и опыту разработчиков. Этот рефакторинг сайта также направлен на то, чтобы пользователи получали экстремальную скорость загрузки при просмотре документации и официального сайта.

## Почему отказаться от зрелого Docusaurus?

В экосистеме React Docusaurus всегда был "стандартным ответом" для документационных сайтов. Он работает из коробки, имеет богатые плагины и активное сообщество. Однако с ростом функциональности HagiCode мы также почувствовали его ограничения:

1.  **Проблемы производительности**: Docusaurus по существу является React SPA (одностраничным приложением). Даже если вы пишете чисто статические страницы, клиенту всё равно нужно загружать runtime React и выполнять гидратацию, что слишком тяжело для простых документационных страниц.
2.  **Объём ресурсов**: Даже при минимальном содержимом страницы объём упакованного JS относительно фиксирован, что недостаточно дружелюбно для мобильных пользователей и сред с плохой сетью.
3.  **Недостаточная гибкость**: Хотя расширение возможно, в настройке процесса сборки мы хотели иметь более низкоуровневый контроль.

Появление Astro как раз решило эти проблемы. Оно предоставляет совершенно новую "архитектуру островов" (Islands Architecture): по умолчанию Astro генерирует статический HTML с нулевым JavaScript, и только компоненты, требующие взаимодействия, "активируются" и загружают JS. Это означает, что большая часть нашего сайта — чистый HTML с экстремальной скоростью.

## Основная стратегия миграции: плавный переход архитектуры

Миграция — это не простое копирование и вставка, а смена мышления. Мы переключились с "полностью React режима" Docusaurus на режим "Core + Islands" Astro.

### 1. Рефакторинг конфигурационной системы

Во-первых, нам нужно перейти от `docusaurus.config.ts` к `astro.config.mjs`. Это не просто изменение имени файла, а перезапись логики маршрутизации и сборки.

В Docusaurus всё — это плагины; в Astro всё — это интеграции. Нам нужно переопределить базовый путь сайта, режим вывода сборки (статический vs SSR) и стратегию сжатия ресурсов.

**До миграции:**
```typescript
// docusaurus.config.ts
export default {
  title: 'HagiCode',
  url: 'https://hagicode.com',
  baseUrl: '/',
  // ... больше конфигурации
};
```

**После миграции:**
```javascript
// astro.config.mjs
import { defineConfig } from 'astro/config';
import react from '@astrojs/react';

export default defineConfig({
  integrations: [react()],
  site: 'https://hagicode.com',
  base: '/',
  // Конфигурация оптимизации для статических ресурсов
  build: {
    inlineStylesheets: 'auto',
  },
});
```

### 2. Сохранение и модификация React компонентов

Это самая сложная часть миграции. На нашем текущем сайте много React компонентов (например, компонент Tabs, подсветка кода, кнопка обратной связи и т.д.). Прямо выбросить их жалко, а полностью сохранить приведёт к чрезмерной нагрузке JS.

HagiCode采用了**渐进式水合**策略：
*   **Чисто статические компоненты**: для презентационного контента (например, заголовок, подвал, чисто текстовая документация) перезаписаны как Astro компоненты (`.astro` файлы), которые при сборке напрямую рендерятся в HTML.
*   **Интерактивные острова**: для компонентов, которые должны сохранить взаимодействие (например, переключатель темы, переключение Tabs, кнопка копирования блока кода), сохранена реализация React и добавлены директивы `client:load` или `client:visible`.

Например, наш часто используемый компонент Tabs в документации:

```jsx
// src/components/Tabs.jsx
import { useState } from 'react';
import './Tabs.css'; // Импорт стилей

export default function Tabs({ items }) {
  const [activeIndex, setActiveIndex] = useState(0);
  // ... логика состояния
  return (
    <div className="tabs-wrapper">
      {/* Логика рендеринга */}
    </div>
  );
}
```

При использовании в Markdown мы явно говорим Astro: "этому компоненту нужен JS":

```markdown
// src/content/docs/example.mdx
import Tabs from '../../components/Tabs.jsx';

<!-- JS загружается только когда компонент попадает в область видимости -->
<Tabs client:visible items={...} />
```

Таким образом, интерактивные компоненты вне области видимости не занимают пропускную способность, что значительно оптимизирует скорость загрузки первого экрана.

### 3. Адаптация системы стилей: от CSS Modules к Scoped

Docusaurus по умолчанию поддерживает CSS Modules, а Astro продвигает использование Scoped CSS (через теги `<style>`). Основная идея обоих — изоляция стилей, но синтаксис разный.

При миграции HagiCode мы разобрали большинство сложных CSS Modules на стили Astro Scoped. Это на самом деле хорошо, потому что в файлах `.astro` стили и шаблон пишутся в одном файле, и обслуживание становится более интуитивным.

**До модификации:**
```css
/* Tabs.module.css */
.wrapper { background: var(--ifm-background-color); }
```

**После модификации (Astro Scoped):**
```html
<!-- Tabs.astro -->
<div class="tabs-wrapper">
  <slot />
</div>

<style>
  .tabs-wrapper {
    /* Непосредственно используем CSS переменные, адаптируемся к теме */
    background: var(--bg-color);
    padding: 1rem;
  }
</style>
```

В то же время мы унифицировали глобальную систему CSS переменных, используя способность Astro к восприятию окружения, чтобы обеспечить бесшовное переключение тёмного режима между разными страницами.

## Практические проблемы и решения

В процессе реальной миграции HagiCode мы столкнулись со множеством проблем, здесь поделюсь несколькими самыми типичными.

### 1. Проблемы с путями и переменными окружения

HagiCode поддерживает развёртывание по подпутям (например, развёртывание в подкаталог GitHub Pages). В Docusaurus он автоматически обрабатывает `baseUrl`. Но в Astro при обработке ссылок на изображения и API запросов нужно быть более осторожным.

Мы ввели механизм переменных окружения для унифицированного управления:

```javascript
// Обработка путей в скрипте сборки
const getBasePath = () => import.meta.env.VITE_SITE_BASE || '/';
```

Помните: не хардкодьте пути, начинающиеся с `/`, в коде. В средах разработки и производства, или после настройки пути `base`, это приведёт к ошибке 404 ресурсов.

### 2. Совместимость скриптов CommonJS

На нашем старом сайте есть некоторые скрипты Node.js (для автоматического сбора данных Metrics, обновления sitemap и т.д.), они написаны на CommonJS (`require`). Astro и современные инструменты сборки полностью переходят на ES Modules (`import`/`export`).

Если у вас есть похожие скрипты, не забудьте полностью переработать их на ES Modules. Это общая тенденция, раньше переделаете — раньше сэкономите нервы.

```javascript
// Старый способ
const fs = require('fs');

// Новый способ
import fs from 'fs';
```

### 3. Не забудьте о SEO и перенаправлениях

Поисковые системы уже проиндексировали старые страницы Docusaurus HagiCode. Если напрямую переключиться на Astro и изменится структура URL, это приведёт к множеству 404 и значительному падению веса.

Мы настроили правила перенаправления в Astro:

```javascript
// astro.config.mjs
export default defineConfig({
  redirects: {
    '/docs/old-path': '/docs/new-path',
    // Пакетное отображение старых ссылок на новые
  }
});
```
Или обработать на уровне конфигурации сервера. Убедитесь, что старые ссылки могут 301 перенаправляться на новые адреса, это критически важно для SEO.

## Заключение

Миграция с Docusaurus на Astro для HagiCode — это не просто обновление фреймворка, а практика концепции "производительность прежде всего".

**Наш результат:**
*   **Экстремальные оценки Lighthouse**: после миграции оценка производительности сайта HagiCode легко приближается к полному баллу.
*   **Более высокая скорость сборки**: инкрементальная сборка Astro сократила время публикации обновлений документации вдвое.
*   **Сохранена гибкость**: благодаря архитектуре Islands мы не пожертвовали никакими интерактивными функциями и по-прежнему можем использовать React там, где это необходимо.

Если вы тоже поддерживаете документационный сайт и страдаете от объёма упаковки или скорости загрузки, попробуйте Astro. Хотя процесс миграции требует некоторых операций (например, изменение имени PCode на HagiCode, перенос компонентов по одному), взамен вы получите шелковисто плавный пользовательский опыт, что определённо стоит того.

Система сборки, описанная в этой статье, — это именно то решение, которое мы разработали на практике, наступая на грабли и оптимизируя в процессе разработки HagiCode. Если вы считаете, что这套 план ценен, значит, наши инженерные способности неплохи — тогда сам HagiCode заслуживает внимания.
