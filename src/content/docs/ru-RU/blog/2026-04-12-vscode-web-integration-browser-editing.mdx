## Быстрое редактирование кода в браузере: практика интеграции VSCode Web

> После анализа кода ИИ, как немедленно открыть редактор в браузере для внесения изменений? В этой статье делится опытом интеграции code-server в проекте HagiCode, реализуя бесшовное соединение между ИИ-помощником и опытом редактирования кода.

## Предыстория

В эпоху ИИ-поддерживаемого программирования разработчикам часто нужно быстро просматривать и редактировать код. Традиционный процесс разработки: открыть проект в настольной IDE, найти файл, отредактировать, сохранить. Но этот процесс в некоторых сценариях как-то кажется не совсем подходящим.

Сценарий первый: удалённая разработка. При использовании ИИ-помощника, такого как HagiCode, бэкенд может работать на удалённом сервере или в контейнере, и локально нельзя напрямую получить доступ к файлам проекта. Каждый раз для просмотра или изменения кода нужно подключаться через SSH или другими способами, опыт разделён. Это похоже на то, как если вы хотите увидеть человека, но через толстое стекло — видно, но не коснёшься.

Сценарий второй: быстрый просмотр. После анализа кода ИИ-помощником пользователь может просто хотеть быстро просмотреть какой-то файл или внести небольшие изменения. Запуск полной настольной IDE кажется тяжеловесным, лёгкий редактор в браузере больше соответствует требованию "быстрого просмотра". Ведь, кто хочет поднимать шум ради одного взгляда?

Сценарий третий: совместная работа на разных устройствах. При работе на разных устройствах редактор в браузере предоставляет единый вход доступа, не нужно настраивать среду разработки на каждом устройстве. Это тоже экономит время,毕竟 жизнь коротка, зачем повторять труд.

Для решения этих болей мы интегрировали VSCode Web в проект HagiCode. Позволили ИИ-помощнику и опыту редактирования кода соединиться бесшовно — после анализа кода ИИ пользователь может немедленно открыть редактор в той же сессии браузера для внесения изменений, без переключения приложений. Этот опыт, как бы это сказать, как когда вам нужно, оно там.

## О HagiCode

Решение, которым делится эта статья, из нашего практического опыта в проекте [HagiCode](https://hagicode.com). HagiCode — это ИИ-управляемый помощник кода, призванный повысить эффективность разработки через взаимодействие на естественном языке. В процессе разработки мы обнаружили, что пользователям часто нужно быстро переключаться между ИИ-анализом и редактированием кода, что побудило нас исследовать, как напрямую интегрировать редактор в браузер.

Адрес проекта: [github.com/HagiCode-org/site](https://github.com/HagiCode-org/site)

## Выбор технологий: почему code-server?

Из множества решений VSCode Web мы выбрали code-server. Этот выбор на самом деле имеет несколько соображений:

Полнофункциональный. code-server — это веб-версия VSCode, поддерживающая большую часть функций настольной версии, включая систему расширений, интеллектуальные подсказки, отладку и т.д. Это означает, что пользователи в браузере также могут получить опыт редактирования, близкий к настольной версии. Ведь, кто хочет жертвовать функциями?

Гибкое развёртывание. code-server может работать как отдельная служба, также поддерживает контейнерное развёртывание Docker, соответствует архитектуре HagiCode. Наш бэкенд написан на C#, фронтенд — React, общается со службой code-server через REST API. Это как конструктор лего, каждый блок имеет своё место.

Безопасная аутентификация. code-server имеет встроенный механизм connection-token, предотвращающий несанкционированный доступ. Каждая сессия имеет уникальный токен, гарантирующий, что только авторизованные пользователи могут получить доступ к редактору. Чувство безопасности, когда есть, тогда знаешь, что это важно.

## Архитектурный дизайн

Интеграция VSCode Web в HagiCode использует архитектурный дизайн с разделением фронтенда и бэкенда.

### Слой службы фронтенда

Фронтенд через `vscodeServerService.ts` инкапсулирует взаимодействие с бэкендом:

```typescript
// Открыть проект
export async function openProjectInCodeServer(
  id: string,
  currentInterfaceLanguage?: string,
): Promise<VsCodeServerLaunchResponseDto>

// Открыть vault
export async function openVaultInCodeServer(
  id: string,
  path?: string,
  currentInterfaceLanguage?: string,
): Promise<VsCodeServerLaunchResponseDto>
```

Разница между этими двумя методами: `openProjectInCodeServer` используется для открытия всего проекта, а `openVaultInCodeServer` используется для открытия определённого пути Vault. Для проектов MonoSpecs с несколькими репозиториями система автоматически создаст файл рабочего пространства. Чёткое разделение труда, каждый делает своё дело — этого достаточно.

### Слой службы бэкенда

Бэкенд `VaultAppService.cs` реализует основную логику:

```csharp
public async Task<VsCodeServerLaunchResponseDto> OpenInCodeServerAsync(
    string id,
    string? relativePath = null,
    string? currentInterfaceLanguage = null,
    CancellationToken cancellationToken = default)
{
    // 1. Получить настройки и проверить, включено ли
    var settings = await _vsCodeServerSettingsService.GetResolvedSettingsAsync(cancellationToken);
    if (!settings.Enabled) {
        throw new BusinessException(VsCodeServerErrorCodes.Disabled, "VSCode Server is disabled.");
    }

    // 2. Получить vault и разрешить каталог запуска
    var vault = await RequireVaultAsync(id, cancellationToken);
    var launchDirectory = ResolveLaunchDirectory(vault, relativePath);

    // 3. Убедиться, что code-server работает и получить информацию о runtime
    var runtime = await _vsCodeServerManager.EnsureStartedAsync(settings, cancellationToken);

    // 4. Разрешить настройки языка
    var language = _vsCodeServerSettingsService.ResolveLaunchLanguage(
        settings.Language,
        currentInterfaceLanguage);

    // 5. Построить URL запуска
    return new VsCodeServerLaunchResponseDto {
        LaunchUrl = AppendQueryString(runtime.BaseUrl, new Dictionary<string, string?> {
            ["folder"] = launchDirectory,
            ["tkn"] = runtime.ConnectionToken,
            ["vscode-lang"] = language,
        }),
        ConnectionToken = runtime.ConnectionToken,
        OpenMode = "folder",
        Runtime = VsCodeServerSettingsService.MapRuntime(
            await _vsCodeServerManager.GetRuntimeSnapshotAsync(cancellationToken)),
    };
}
```

Обязанности этого метода очень ясны: проверить настройки, разрешить путь, запустить службу, построить URL. Среди них метод `ResolveLaunchDirectory` выполняет проверку безопасности пути, предотвращая атаки обхода пути. Код как стихи, каждая строка имеет свой ритм.

### Автоматическое управление runtime

Бэкенд через `VsCodeServerManager` управляет процессом code-server:

- Проверка состояния процесса
- Автоматический запуск остановленных служб
- Возврат снимка runtime (порт, ID процесса, время запуска и т.д.)

Этот дизайн позволяет системе автоматически обрабатывать жизненный цикл code-server, пользователям не нужно вручную управлять процессами службы. Ведь, жизнь уже достаточно сложна, что можно автоматизировать — автоматизируйте.

### Механизм следования языку

HagiCode поддерживает многоязычный интерфейс, code-server также должен следовать этой настройке. Система поддерживает три языковых режима:

- `follow`: следовать текущему языку интерфейса
- `zh-CN`: фиксированный китайский
- `en-US`: фиксированный английский

Параметр URL `vscode-lang` передаётся в code-server, обеспечивая соответствие языка редактора интерфейсу HagiCode. Язык — это такая вещь, когда унифицирован, тогда комфортно.

### Рабочее пространство MonoSpecs с несколькими репозиториями

Для проектов MonoSpecs (mono-repo, содержащий несколько подрепозиториев) система автоматически создаст файл `.code-workspace`:

```csharp
private async Task<string> CreateWorkspaceFileAsync(Project project, Guid projectId)
{
    var folders = await ResolveWorkspaceFoldersAsync(project.Path);
    var workspaceDocument = new {
        folders = folders.Select(path => new { path }).ToArray(),
    };
    // Генерация файла workspace...
}
```

Это позволяет одновременно редактировать несколько подрепозиториев в одном экземпляре code-server, очень полезно для крупных проектов mono-repo. Несколько репозиториев в одном окне, как несколько историй в одной книге.

## Интеграция фронтенда

Фронтенд HagiCode использует React + TypeScript, шаги интеграции code-server не сложны.

### Кнопка быстрого действия

Добавить кнопку Code Server в карточку проекта:

```tsx
// QuickActionsZone.tsx
<Button
  size="sm"
  variant="default"
  onClick={() => onAction({ type: 'open-code-server' })}
>
  <Globe className="h-3 w-3 mr-1" />
  <span className="text-xs">{t('project.openCodeServer')}</span>
</Button>
```

Эта кнопка вызовет действие открытия, вызовет бэкенд API для получения URL запуска. Одна кнопка, несёт одно действие — просто и прямо.

### Обработка действия открытия

```tsx
const handleAction = async (action: ProjectAction) => {
  if (action.type === 'open-code-server') {
    const response = await openProjectInCodeServer(project.id, i18n.language);
    window.open(response.launchUrl, '_blank', 'noopener,noreferrer');
  }
};
```

Используя `window.open` для открытия code-server в новой вкладке, параметры `noopener,noreferrer` обеспечивают безопасность. Безопасность — это такая вещь, как бы осторожно ни было, не лишнее.

### Точка входа в редактирование Vault

В списке Vault добавьте аналогичную кнопку редактирования:

```tsx
const handleEditVault = async (vault: VaultItemDto) => {
  const response = await openVaultInCodeServer(vault.id);
  window.open(response.launchUrl, '_blank', 'noopener,noreferrer');
};
```

Проект и Vault используют одинаковый способ открытия, сохраняя согласованность взаимодействия. Важность согласованности не уступает самой функциональности.

## Логика построения URL

Формат URL code-server на самом деле тоже имеет свои нюансы:

**Режим папки**:
```
http://{host}:{port}/?folder={path}&tkn={token}&vscode-lang={lang}
```

**Режим рабочего пространства**:
```
http://{host}:{port}/?workspace={workspacePath}&tkn={token}&vscode-lang={lang}
```

Где `tkn` — это токен подключения, автоматически генерируемый при каждом запуске code-server, обеспечивающий безопасность доступа. Параметр `vscode-lang` управляет языком интерфейса редактора. Эти параметры各有 свои назначения, ни один нельзя пропустить.

## Сценарии использования

### Сценарий первый: ИИ-поддерживаемая проверка кода

Пользователь разговаривает с HagiCode, ИИ анализирует код проекта и обнаруживает потенциальные проблемы. Пользователь нажимает кнопку "Open in Code Server", чтобы напрямую открыть редактор в браузере, просмотреть проблемные файлы и исправить, затем вернуться в HagiCode для продолжения диалога. Весь процесс завершается в браузере, без переключения приложений. Это чувство, как бы это сказать, как текущая вода — плавно.

### Сценарий второй: Редактирование учебных материалов Vault

Пользователь создал Vault для изучения какого-то открытого проекта, хочет добавить учебные заметки в каталог `docs/`. Через code-server напрямую редактировать файлы Markdown в браузере, после сохранения HagiCode может синхронно читать обновлённое содержание заметок. Это очень полезно для создания личной базы знаний. Знания — такая вещь, чем больше накапливается, тем более ценны.

### Сценарий третий: Разработка MonoSpecs с несколькими репозиториями

Проект MonoSpecs содержит несколько подрепозиториев, code-server автоматически создаёт рабочее пространство с несколькими папками. Пользователь может одновременно редактировать код нескольких репозиториев в браузере, после изменения отправлять в各自的 репозитории Git. Этот способ работы особенно подходит для сценариев, требующих изменений между репозиториями. Изменение нескольких репозиториев вместе, как одновременная обработка нескольких дел — требует некоторых навыков.

## Вопросы безопасности

При реализации интеграции code-server безопасность — это проблема, требующая особого внимания. Ведь, с безопасностью, когда возникнут проблемы — поздно.

### Connection Token

connection-token генерируется случайно, не должен утекать. Рекомендуется использовать в среде HTTPS, чтобы предотвратить перехват токена посредником. Чувствительная информация — лучше защитить.

### Безопасность пути к файлу

Бэкенд реализует проверку обхода пути:

```csharp
private static string ResolveLaunchDirectory(VaultRegistryEntry vault, string? relativePath)
{
    var vaultRoot = EnsureTrailingSeparator(Path.GetFullPath(vault.PhysicalPath));
    var combinedPath = Path.GetFullPath(Path.Combine(vaultRoot, relativePath ?? "."));
    if (!combinedPath.StartsWith(vaultRoot, StringComparison.OrdinalIgnoreCase))
    {
        throw new BusinessException(VaultRelativePathTraversalCode, "Relative path traversal detected.");
    }
    return combinedPath;
}
```

Этот код гарантирует, что пользователи не могут получить доступ к файлам вне каталога vault через `../` и другие способы. Проверка границ — такая вещь, лучше сделать, чем не делать.

### Контроль доступа

Процесс code-server работает с соответствующими правами пользователя, избегая доступа к чувствительным файлам системы. Рекомендуется использовать выделенного пользователя для запуска службы code-server. Контроль доступа — что должно быть, то должно быть.

## Оптимизация производительности

code-server будет потреблять ресурсы сервера, вот некоторые рекомендации по оптимизации:

- Мониторинг использования CPU/памяти, при необходимости отрегулировать ограничения ресурсов
- Крупные проекты могут потребовать увеличения времени ожидания
- Реализовать автоматическую очистку по тайм-ауту сессии, освобождение ресурсов
- Рассмотреть использование кеша для уменьшения повторных вычислений

HagiCode предоставляет API мониторинга состояния runtime, фронтенд может получить текущее состояние через `getVsCodeServerSettings()`:

```typescript
const { settings, runtime } = await getVsCodeServerSettings();
// runtime.status: 'disabled' | 'stopped' | 'starting' | 'running' | 'unhealthy'
// runtime.baseUrl: "http://localhost:8080"
// runtime.processId: 12345
```

Этот дизайн позволяет пользователям ясно понимать состояние здоровья code-server и быстро определить местоположение при возникновении проблем. Когда состояние видно, тогда есть уверенность.

## Детали пользовательского опыта

В процессе реализации мы обнаружили некоторые детали, влияющие на пользовательский опыт, заслуживающие особого внимания:

При первом открытии code-server может потребоваться подождать запуска, это время может варьироваться от нескольких секунд до полминуты. Рекомендуется отображать состояние загрузки на фронтенде, чтобы пользователи знали, что система обрабатывает. Ожидание — дело такое, когда есть обратная связь — нормально.

Браузер может блокировать всплывающие окна, нужно указать пользователям вручную разрешить. HagiCode при первом открытии отобразит руководство, сообщающее пользователям, как настроить разрешения браузера. Пользовательский опыт — именно в этих деталях проявляется.

Рекомендуется отображать состояние runtime (запускается/работает/ошибка), чтобы пользователи при возникновении проблем могли знать, это проблема сервера или их собственная операция. Знать, где проблема — по крайней мере есть уверенность.

## Пример конфигурации

Конфигурация code-server не сложна:

```json
{
  "vscodeServer": {
    "enabled": true,
    "host": "0.0.0.0",
    "port": 8080,
    "language": "follow"
  }
}
```

`enabled` управляет переключателем функции, `host` и `port` указывают адрес прослушивания, `language` устанавливает языковой режим. Эти конфигурации можно изменить через интерфейс UI, вступают в силу в реальном времени. Простые вещи часто самые полезные.

## Заключение

Интеграция VSCode Web в HagiCode предоставляет элегантное решение: позволяет ИИ-помощнику и опыту редактирования кода соединиться бесшовно. Через интеграцию code-server в браузере пользователи могут быстро реагировать на результаты анализа ИИ, завершать полный процесс от анализа до редактирования в одной сессии браузера.

Несколько ключевых преимуществ этого решения: во-первых, унифицированный опыт, проект и Vault используют одинаковый способ открытия; во-вторых, поддержка нескольких репозиториев, проекты MonoSpecs автоматически создают рабочее пространство; в-третьих, безопасно и управляемо, мониторинг состояния runtime и проверка безопасности пути.

Решение, которым делится эта статья, суммировано HagiCode в реальной разработке. Если вы считаете, что это решение имеет ценность, это означает, что наша инженерная практика неплохая — тогда сам HagiCode также заслуживает внимания. Ведь хорошие вещи заслуживают того, чтобы их видело больше людей.
