## HagiCode.Libs: Инженерная практика унифицированной интеграции нескольких AI-помощников по программированию CLI

> В процессе разработки проекта HagiCode нам необходимо одновременно интегрировать несколько CLI AI-помощников по программированию, таких как Claude Code, Codex, CodeBuddy. Интерфейсы, параметры и форматы вывода каждого CLI различаются, а повторяющийся код интеграции делает проект всё труднее поддерживать. В этой статье мы поделимся тем, как через библиотеку HagiCode.Libs построить унифицированный уровень абстракции для решения этой инженерной проблемы — это можно назвать и ямами, по которым мы наступили, и накопленным опытом.

## Предпосылки

Современный рынок AI-помощников по программированию довольно оживлённый: помимо Claude Code, есть Codex от OpenAI, CodeBuddy от Zhipu и другие. Как проект AI-помощника по коду, HagiCode должен интегрировать эти различные CLI-инструменты в нескольких подпроектах (десктопная версия, бэкенд, веб и др.).

Сначала проблема была не столь серьёзной, интеграция одного CLI — это всего несколько сотен строк кода. Но по мере роста количества поддерживаемых CLI дела начали осложняться —

У каждого CLI свой формат параметров командной строки, свои требования к переменным окружения, а форматы вывода и вовсе самые разнообразные — одни выводят JSON, другие потоковый JSON, а третьи — просто обычный текст. Добавьте сюда проблемы совместимости на разных платформах: механизмы обнаружения исполняемых файлов и управления процессами в Windows и Unix-системах совершенно разные, степень дублирования кода растёт. На самом деле, это всего лишь немного больше Ctrl+C, Ctrl+V, но поддерживать такое — головная боль.

Самая большая головная боль — каждый раз при добавлении поддержки новой функции CLI приходится изменять один и тот же код в нескольких проектах. Этот подход явно не является долгосрочным решением — код тоже имеет характер, при чрезмерном повторении он也开始 капризничать.

## О HagiCode

Решение, описанное в этой статье, основано на нашем опыте работы над проектом [HagiCode](https://hagicode.com). HagiCode — это проект AI-помощника по коду с открытым исходным кодом, который должен одновременно поддерживать несколько подпроектов: расширение VSCode, AI-сервис на бэкенде, кроссплатформенный десктопный клиент и т.д. Как сказать, именно эта сложная многоязычная и мультиплатформенная среда способствовала рождению HagiCode.Libs — можно сказать, вынужденная мера, да и ладно.

## Анализ: поиск общих черт

Хотя у этих AI-помощников по программированию CLI есть свои особенности, с технической точки зрения у них есть очевидные общие черты:

**Похожие режимы взаимодействия**: все запускают процесс CLI, отправляют промпты, принимают потоковые ответы, анализируют сообщения и в итоге завершают или продолжают сессию — этот процесс, по сути, вылеплен по одной модели.

**Похожие потребности в конфигурации**: всем нужна аутентификация по API-ключу, настройка рабочего каталога, выбор модели, управление правами на инструменты, управление сессиями. Ведь все зарабатывают на API, разница лишь в предпочтениях.

**Одинаковые кроссплатформенные вызовы**: всем нужно разрешить обнаружение путей к исполняемым файлам (`claude` vs `claude.exe` vs `/usr/local/bin/claude`), запуск процессов и обработку переменных окружения, экранирование команд оболочки и сборку параметров. О кроссплатформенности чем больше говоришь, тем больше слёз — только тот, кто наступал на эти грабли, знает разницу между Windows и Unix.

Основываясь на этом анализе, нам нужен унифицированный уровень абстракции, который предоставит согласованный интерфейс, инкапсулирует логику обнаружения CLI на разных платформах, обработку анализа потокового вывода, а также поддерживает использование с внедрением зависимостей и без DI. Об этом думать — уже головная боль, но面对还是要面对 — ведь это свой проект, хочешь не хочешь, а доделывать придётся.

## Решение: HagiCode.Libs

Мы создали **HagiCode.Libs** — лёгкое рабочее пространство библиотек .NET 10 по лицензии MIT с открытым исходным кодом, теперь опубликованное на [GitHub](https://github.com/HagiCode-org/Hagicode.Libs). Хотя это не какое-то грандиозное произведение, способное удивить мир, но решение реальных проблем — довольно приятно.

### Структура проекта

```
HagiCode.Libs/
├── src/
│   ├── HagiCode.Libs.Core/           # Основная функциональность
│   │   ├── Discovery/                 # Обнаружение исполняемых файлов CLI
│   │   ├── Process/                   # Кроссплатформенное управление процессами
│   │   ├── Transport/                 # Транспорт потоковых сообщений
│   │   └── Environment/               # Разработка среды выполнения
│   ├── HagiCode.Libs.Providers/       # Реализации поставщиков
│   │   ├── ClaudeCode/                # Поставщик Claude Code
│   │   ├── Codex/                     # Поставщик Codex
│   │   └── Codebuddy/                 # Поставщик CodeBuddy
│   ├── HagiCode.Libs.ConsoleTesting/  # Тестовый фреймворк
│   ├── HagiCode.Libs.ClaudeCode.Console/
│   ├── HagiCode.Libs.Codex.Console/
│   └── HagiCode.Libs.Codebuddy.Console/
└── tests/                             # Тесты xUnit
```

### Цели проектирования

При проектировании HagiCode.Libs мы следовали нескольким принципам — ведь это тоже ямы, по которым мы наступали, и накопленный опыт:

**Нулевая зависимость от тяжёлых фреймворков**: не зависит от ABP или других крупных фреймворков, сохраняет лёгкость. В наше время меньше зависимостей — меньше проблем — кто же не страдал от ада зависимостей?

**Кроссплатформенная поддержка**: нативная поддержка Windows, macOS, Linux, не нужно писать разный код для разных платформ. Один код для всех платформ — тоже неплохо.

**Потоковая обработка**: использование асинхронных потоков для обработки вывода CLI, более соответствует современному стилю программирования .NET. Ведь времена меняются, асинхронность — это царский путь.

**Гибкая интеграция**: поддерживает как сценарии с внедрением зависимостей, так и прямое создание экземпляров. На вкус и цвет, как говорится, товарищей нет, выбирай удобное.

## Способы использования

### Через внедрение зависимостей

Если ваш проект уже использует внедрение зависимостей (например, ASP.NET Core или универсальный хост), можно напрямую интегрировать — это тоже своего рода ребёнок, хотя небольшой, но довольно послушный:

```csharp
using HagiCode.Libs.Providers;
using Microsoft.Extensions.DependencyInjection;

var services = new ServiceCollection();
services.AddHagiCodeLibs();

await using var provider = services.BuildServiceProvider();
var claude = provider.GetRequiredService<ICliProvider<ClaudeCodeOptions>>();

var options = new ClaudeCodeOptions
{
    ApiKey = "your-api-key",
    Model = "claude-sonnet-4-20250514"
};

await foreach (var message in claude.ExecuteAsync(options, "Hello, Claude!"))
{
    Console.WriteLine($"{message.Type}: {message.Content}");
}
```

### Прямое создание экземпляра

Если это простой скрипт или сценарий без использования DI, можно напрямую создать экземпляр — говоря прямо, дело вкуса:

```csharp
var claude = new ClaudeCodeProvider();
var options = new ClaudeCodeOptions
{
    ApiKey = "sk-ant-xxx",
    Model = "claude-sonnet-4-20250514"
};

await foreach (var message in claude.ExecuteAsync(options, "帮我写一个快速排序"))
{
    // Обработка сообщения
}
```

Оба способа используют один и тот же базовый слой реализации, вы можете выбрать подходящий способ интеграции в зависимости от реальной ситуации проекта. В этом мире нет стандартного ответа, подходящее для вас — лучшее — хотя это высказывание немного банально, но действительно так и есть.

## Обмен практическим опытом

### 1. Специализированная тестовая консоль

У каждого поставщика есть специализированный проект тестовой консоли для независимой проверки эффекта интеграции — как сказать, тестирование: либо не делай, либо делай полностью:

```bash
# Тест Claude Code
dotnet run --project src/HagiCode.Libs.ClaudeCode.Console -- --test-provider
dotnet run --project src/HagiCode.Libs.ClaudeCode.Console -- --test-all claude

# Тест CodeBuddy
dotnet run --project src/HagiCode.Libs.Codebuddy.Console -- --test-provider codebuddy-cli

# Тест Codex
dotnet run --project src/HagiCode.Libs.Codex.Console -- --test-provider codex-cli
```

Тестовые сценарии охватывают несколько ключевых сценариев:
- **Ping**: проверка работоспособности, подтверждение доступности CLI
- **Simple Prompt**: базовый тест промптов
- **Complex Prompt**: тест многоходовых диалогов
- **Session Restore/Resume**: тест восстановления сессии
- **Repository Analysis**: тест анализа репозитория кода

Такой дизайн независимой тестовой консоли особенно полезен при отладке, можно быстро определить, находится ли проблема на уровне HagiCode.Libs или в самом CLI. Отладка, говоря прямо, заключается в том, чтобы понять, где проблема — правильное направление — уже половина успеха.

### 2. Проверка кроссплатформенного CI/CD

Кроссплатформенная совместимость — одна из ключевых целей HagiCode.Libs. Мы настроили рабочий процесс GitHub Actions `.github/workflows/cli-discovery-cross-platform.yml`, который выполняет реальную проверку обнаружения CLI на трёх платформах: `ubuntu-latest`, `macos-latest`, `windows-latest`.

Это гарантирует, что каждое изменение кода не нарушит кроссплатформенную совместимость. При локальной разработке также можно воспроизвести следующими командами — нельзя позволить CI брать на себя всю вину, локально тоже должно работать:

```bash
npm install --global @anthropic-ai/claude-code@2.1.79
HAGICODE_REAL_CLI_TESTS=1 dotnet test --filter "Category=RealCli"
```

### 3. Обработка потока сообщений

HagiCode.Libs использует асинхронные потоки для обработки вывода CLI, этот способ больше соответствует современному асинхронному стилю программирования .NET, чем традиционные колбэки или события — в конечном счёте, это прогресс технологий, которому никто не может помешать:

```csharp
public async IAsyncEnumerable<CliMessage> ExecuteAsync(
    TOptions options,
    string prompt,
    [EnumeratorCancellation] CancellationToken cancellationToken = default)
{
    // Запуск процесса CLI
    // Анализ потокового вывода JSON
    // Генерация последовательности CliMessage
}
```

Типы сообщений включают:
- `user`: пользовательское сообщение
- `assistant`: ответ помощника
- `tool_use`: вызов инструмента
- `result`: завершение сессии

Такой дизайн позволяет вызывающей стороне гибко обрабатывать потоковый вывод, например, отображать в реальном времени, обрабатывать после буферизации или перенаправлять в другие сервисы. Красота何必在乎 погоду? Важно, что мышление открыто, можно использовать как угодно.

### 4. Исследование Git-репозитория

Модуль HagiCode.Libs.Exploration предоставляет функции обнаружения Git-репозиториев и проверки статуса, это особенно полезно в сценариях анализа репозиториев кода — это тоже вынужденная функция, ведь HagiCode нужно анализировать репозитории кода:

```csharp
// Обнаружение Git-репозиториев
var repositories = await GitRepositoryDiscovery.DiscoverAsync("/path/to/search");

// Получение информации о репозитории
var info = await GitRepository.GetInfoAsync(repoPath);
Console.WriteLine($"Branch: {info.Branch}, Remote: {info.RemoteUrl}");
Console.WriteLine($"Has uncommitted changes: {info.HasUncommittedChanges}");
```

Функция анализа кода HagiCode использует этот модуль для идентификации структуры проекта и статуса Git. Можно сказать, это с пользой, можно сказать, ребёнок, которого растили не зря.

## Меры предосторожности

Основываясь на нашем опыте в проекте HagiCode, есть несколько моментов, на которые нужно обратить особое внимание — всё дело, дело, дело:

**Безопасность API-ключей**: не hardcode'ите API-ключи в код, используйте переменные окружения или управление конфигурацией. HagiCode.Libs поддерживает передачу конфигурации через объект Options, удобно интегрировать различные источники конфигурации. Ведь в вопросах безопасности, как бы осторожно ни было, это не лишнее.

**Блокировка версии CLI**: В CI/CD мы заблокировали конкретную версию (например, `@anthropic-ai/claude-code@2.1.79`), чтобы снизить неопределённость из-за дрейфа версий. При локальной разработке также рекомендуется использовать фиксированную версию. О версиях чем больше говоришь, тем больше слёз — без фиксированной версии, за минуту научишь жизни.

**Классификация тестов**: по умолчанию тесты используют фейковых поставщиков для сохранения детерминизма и скорости, тесты с реальными CLI требуют явного включения. Это обеспечивает быструю обратную связь от CI, а при необходимости позволяет проводить проверку в реальной среде. Ходить по этому канату тоже непросто — быстро и стабильно всегда нужно чем-то жертвовать.

**Управление сессиями**: Механизмы восстановления сессии разных CLI различаются: Claude Code использует каталог `.claude/` для хранения сессий, Codex и CodeBuddy имеют свои собственные способы. При использовании нужно обратить внимание на документацию каждого, чтобы понять конкретный механизм персистентности сессий. Это также ладно, разобраться никогда не повредит.

HagiCode.Libs — это унифицированный уровень абстракции, который мы создали в процессе разработки HagiCode для решения проблемы повторяющейся инженерной работы при интеграции нескольких CLI. Он предоставляет согласованный интерфейс, инкапсулирует детали разных платформ, поддерживает гибкие способы интеграции, значительно снижая инженерную сложность интеграции нескольких AI-помощников по программированию.
