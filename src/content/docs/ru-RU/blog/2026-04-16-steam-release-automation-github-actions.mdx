## Как реализовать автоматическую публикацию в Steam с помощью GitHub Actions

> В этой статье делится полное решение реализации автоматической публикации в Steam в проекте HagiCode Desktop, полностью автоматизированный процесс от GitHub Release до платформы Steam, включая аутентификацию Steam Guard, загрузку мультиплатформенных Depot и другие ключевые технические детали.

## Предыстория

Процесс публикации на платформе Steam на самом деле довольно сильно отличается от традиционных способов распространения приложений. У Steam есть своя полноценная система обновления и распространения, разработчикам нужно загружать сборки в сеть CDN Steam через инструмент SteamCMD, а не просто бросать ссылку для скачивания, как на других платформах.

Проект HagiCode Desktop планирует выйти на платформу Steam, что принесло новые вызовы в наш процесс публикации:

1. Нужно преобразовать существующие сборки в формат, совместимый со Steam
2. Необходимо загрузить их на платформу Steam через инструмент SteamCMD
3. Также нужно обрабатывать аутентификацию Steam Guard
4. Необходимо поддерживать загрузку Depot для нескольких платформ (Linux, Windows, macOS)
5. Нужно реализовать автоматизированный переход от GitHub Release к Steam

Проект ранее уже реализовал "режим переносимой версии" (portable version mode), позволяющий приложению обнаруживать фиксированную сервисную нагрузку, упакованную в каталоге extra. Наша цель - обеспечить бесшовную интеграцию этого режима переносимой версии с распространением Steam.

## О HagiCode

Решение, представленное в этой статье, основано на нашем практическом опыте в проекте [HagiCode](https://hagicode.com). HagiCode - это проект AI-помощника по коду с поддержкой настольного запуска, и мы продвигаем работу по выходу на платформу Steam, поэтому нам нужно создать надёжный процесс автоматической публикации.

## Архитектурное проектирование

В основе всего процесса публикации Steam лежит рабочий процесс GitHub Actions, который делит весь процесс на три основных этапа:

```
┌─────────────────────────────────────────────────────────────┐
│ GitHub Actions Workflow (Steam Release)                      │
├─────────────────────────────────────────────────────────────┤
│ 1. Этап подготовки:                                         │
│    - Выгрузка кода portable-version                         │
│    - Загрузка сборок из GitHub Release                      │
│    - Распаковка и подготовка каталога содержимого Steam     │
│                                                             │
│ 2. Настройка SteamCMD:                                      │
│    - Установка/повторное использование SteamCMD             │
│    - Аутентификация через Steam Guard                       │
│                                                             │
│ 3. Этап публикации:                                         │
│    - Генерация файлов конфигурации Depot VDF                │
│    - Генерация файлов конфигурации App Build VDF            │
│    - Вызов SteamCMD для загрузки в Steam                    │
└─────────────────────────────────────────────────────────────┘
```

Преимущества этой конструкции, так скажем:
- Повторное использование существующих артефактов GitHub Release, избегание повторных сборок - ведь кто хочет повторять одну и ту же работу
- Обеспечение изоляции безопасности через самоуправляемые раннеры, дополнительный уровень защиты всегда полезен
- Поддержка переключения между режимом предварительного просмотра и официальной публикации ветки, немного гибкости
- Полная обработка ошибок и ведение журнала, чтобы не слишком теряться при возникновении проблем

## Реализация рабочего процесса

### Проектирование триггерных параметров

Наш рабочий процесс поддерживает следующие ключевые параметры:

```yaml
inputs:
  release:           # Тег выпуска Portable Version
    description: 'Тег версии для публикации (например, v1.0.0)'
    required: true
  steam_preview:     # Создавать ли предварительную сборку
    description: 'Режим предварительного просмотра'
    required: false
    default: 'false'
  steam_branch:      # Установка ветки Steam как live
    description: 'Целевая ветка Steam'
    required: false
    default: 'preview'
  steam_description: # Переопределение описания сборки
    description: 'Описание сборки'
    required: false
```

### Конфигурация самоуправляемого раннера

Из соображений безопасности мы используем самоуправляемые раннеры с меткой steam:

```yaml
runs-on:
  - self-hosted
  - Linux
  - X64
  - steam
```

Это обеспечивает выполнение публикации Steam на выделенном раннере с сохранением безопасной изоляции конфиденциальных учётных данных. Ведь безопасность - это то, к чему стоит относиться внимательнее.

### Управление конкурентностью

Чтобы избежать взаимного влияния публикаций одной и той же версии, мы настроили управление конкурентностью:

```yaml
concurrency:
  group: portable-version-steam-${{ github.event.inputs.release }}
  cancel-in-progress: false
```

Обратите внимание, что здесь установлено `cancel-in-progress: false`, поскольку процесс публикации Steam может быть длительным, и мы не хотим отменять выполняющуюся публикацию из-за нового триггера. Ведь публикация версии тоже не простая задача, нужно дать ей возможность завершиться.

## Реализация основных скриптов

### Подготовка входных данных для публикации

Скрипт `prepare-steam-release-input.mjs` отвечает за подготовку входных данных, необходимых для публикации:

```javascript
// Загрузка манифеста сборки и манифеста артефактов GitHub Release
const buildManifest = await downloadBuildManifest(releaseTag);
const artifactInventory = await downloadArtifactInventory(releaseTag);

// Загрузка архивов для каждой платформы
for (const platform of ['linux-x64', 'win-x64', 'osx-universal']) {
  const artifactUrl = getArtifactUrl(artifactInventory, platform);
  await downloadArtifact(artifactUrl, platform);
}

// Распаковка в структуру каталогов содержимого Steam
await extractToSteamContent(sources, contentRoot);
```

### Аутентификация Steam Guard

Steam требует использования Steam Guard для защиты учётных записей, мы реализовали алгоритм генерации кода на основе общего ключа:

```javascript
function generateSteamGuardCode(sharedSecret, timestamp = Date.now()) {
  const secret = decodeSharedSecret(sharedSecret);
  const time = Math.floor(timestamp / 1000 / 30);

  const timeBuffer = Buffer.alloc(8);
  timeBuffer.writeBigUInt64BE(BigInt(time));

  // Использование HMAC-SHA1 для генерации одноразового кода на основе времени
  const hash = crypto.createHmac('sha1', secret)
    .update(timeBuffer)
    .digest();

  // Преобразование в 5-символьный код Steam Guard
  const code = steamGuardCode(hash);
  return code;
}
```

Эта реализация основана на механизме TOTP (Time-based One-Time Password) Steam Guard, генерирующем новый код подтверждения каждые 30 секунд. Ведь безопасность - это то, что нужно делать надёжными способами.

### Генерация конфигурации VDF

VDF (Valve Data Format) - это формат конфигурации, используемый Steam, нам нужно сгенерировать два типа VDF-файлов:

**Depot VDF** для конфигурации содержимого каждой платформы:

```javascript
function buildDepotVdf(depotId, contentRoot) {
  return [
    '"DepotBuildConfig"',
    '{',
    `  "DepotID" "${escapeVdf(depotId)}"`,
    `  "ContentRoot" "${escapeVdf(contentRoot)}"`,
    '  "FileMapping"',
    '  {',
    '    "LocalPath" "*"',
    '    "DepotPath" "."',
    '    "recursive" "1"',
    '  }',
    '}'
  ].join('\n');
}
```

**App Build VDF** для конфигурации всей сборки приложения:

```javascript
function buildAppBuildVdf(appId, depotBuilds, description, setLive) {
  const vdf = [
    '"appbuild"',
    '{',
    `  "appid" "${appId}"`,
    `  "desc" "${escapeVdf(description)}"`,
    `  "contentroot" "${escapeVdf(contentRoot)}"`,
    '  "buildoutput" "build_output"',
    '  "depots"',
    '  {'
  ];

  for (const [depotId, depotVdfPath] of Object.entries(depotBuilds)) {
    vdf.push(`    "${depotId}" "${depotVdfPath}"`);
  }

  if (setLive) {
    vdf.push(`  }`);
    vdf.push(`  "setlive" "${setLive}"`);
  }

  vdf.push('}');
  return vdf.join('\n');
}
```

### Вызов SteamCMD

Наконец, загрузка выполняется путём вызова SteamCMD:

```javascript
await runCommand(steamcmdPath, [
  '+login', steamUsername, steamPassword, steamGuardCode,
  '+run_app_build', appBuildPath,
  '+quit'
]);
```

Этот шаг - последний прыжок всего процесса, после него всё завершается......

## Обработка мультиплатформенных Depot

Steam использует систему Depot для управления содержимым разных платформ, мы поддерживаем три основных Depot:

| Платформа | Идентификатор Depot | Поддержка архитектур |
|-----|-----------|---------|
| Linux | `linux-x64` | x64_64 |
| Windows | `win-x64` | x64_64 |
| macOS | `osx-universal` | universal, x64_64, arm64 |

Каждый Depot имеет независимый каталог содержимого и файл конфигурации VDF, что гарантирует, что пользователи разных платформ загружают только необходимое им содержимое. Ведь трафик тоже деньги, сэкономить немного - это хорошо.

## Процесс публикации

### Шаг 1: Подготовка GitHub Release

Сначала нужно создать GitHub Release в репозитории portable-version, включая:
- Архивы для каждой платформы
- Манифест сборки (`{tag}.build-manifest.json`)
- Манифест артефактов (`{tag}.artifact-inventory.json`)

### Шаг 2: Запуск рабочего процесса публикации Steam

Вручную запустите рабочий процесс через GitHub Actions, заполнив необходимые параметры:
- `release`: Тег версии для публикации (например, v1.0.0)
- `steam_branch`: Целевая ветка (например, `preview` или `public`)
- `steam_preview`: Режим предварительного просмотра

### Шаг 3: Автоматическое выполнение процесса публикации

Рабочий процесс автоматически выполнит следующие шаги:

1. Загрузка и распаковка артефактов GitHub Release
2. Установка/обновление SteamCMD
3. Генерация файлов конфигурации Steam VDF
4. Аутентификация через Steam Guard
5. Загрузка содержимого в Steam CDN
6. Установка указанной ветки как live

После прохождения всего этого процесса всё, что нужно сделать, сделано.

## Лучшие практики

### Управление аутентификацией Steam Guard

При первом запуске необходимо вручную ввести код Steam Guard, после чего рекомендуется настроить автоматическую генерацию кода с помощью общего ключа. Это позволяет избежать необходимости ручного вмешательства при каждой публикации, ведь никто не хочет повторять одни и те же действия каждый раз.

SteamCMD сохранит токен входа, который можно будет повторно использовать. Но обратите внимание на срок действия токена, после истечения срока всё равно нужно будет повторно аутентифицироваться, ничего не поделаешь.

### Структура каталога содержимого

Убедитесь, что структура каталога содержимого Steam правильна:

```
steam-content/
├── linux-x64/     # Содержимое для Linux
├── win-x64/       # Содержимое для Windows
└── osx-universal/ # Универсальное содержимое для macOS
```

В каждом каталоге должны находиться полные файлы приложения для соответствующей платформы. В этом тоже нет ничего особенного, просто нужно делать как нужно.

### Использование режима предварительного просмотра

Режим предварительного просмотра не устанавливает ни одной ветки как live, подходит для тестирования и проверки:

```yaml
if [ "$STEAM_PREVIEW_INPUT" = 'true' ]; then
  cmd+=(--preview)
fi
```

这样可以先上传到 Steam 平台进行验证，确认无误后再切换到正式分支。Дополнительный уровень проверки всегда полезен.

### Обработка ошибок и ведение журнала

Скрипты включают полную обработку ошибок и ведение журнала:
- Проверка существования GitHub Release
- Проверка необходимых файлов метаданных
- Обеспечение существования содержимого платформы
- Генерация сводного отчёта GitHub Actions

Эти информации очень ценны для отладки и аудита, ведь при возникновении проблем наличие线索 лучше, чем полная растерянность.

### Управление артефактами

Рабочий процесс генерирует два типа артефактов:
- `portable-steam-release-preparation-{tag}`: Метаданные подготовки публикации
- `portable-steam-build-metadata-{tag}`: Метаданные сборки Steam

Эти артефакты можно использовать для последующего аудита и отладки, время хранения рекомендуется установить на 30 дней. В любом случае они не занимают много места, можно и оставить.

## Практическое применение

В проекте HagiCode этот процесс автоматической публикации успешно работает уже несколько версий. Весь процесс от GitHub Release до платформы Steam полностью автоматизирован, без необходимости ручного вмешательства.

Это значительно повысило нашу эффективность и надёжность публикации. Ранее ручная публикация одной версии занимала более 30 минут, теперь весь процесс занимает всего несколько минут. Время - это то, что стоит сэкономить.

Более важно то, что автоматизированный процесс уменьшает возможность человеческих ошибок, каждая публикация следует стандартизированному процессу, и результаты более предсказуемы. Ведь повторяющиеся задачи лучше доверить машинам, и людям станет легче.
