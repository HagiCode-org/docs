## От вызовов CLI до интеграции SDK: лучшие практики использования GitHub Copilot в проектах .NET

> Путь обновления от вызовов командной строки до официальной интеграции SDK — это тоже своего рода опыт, и сегодня мы поделимся проблемами, с которыми мы столкнулись в проекте HagiCode, и тем, чему мы научились.

## Предпосылки

После официального выпуска GitHub Copilot SDK в 2025 году мы начали интегрировать его в слой AI-возможностей. До этого проект в основном использовал возможности GitHub Copilot через прямые вызовы инструментов командной строки Copilot CLI, но у этого подхода было несколько очевидных проблем:

- **Сложное управление процессами**: необходимо вручную управлять жизненным циклом процессов CLI, временем ожидания запуска и очисткой процессов — в конце концов, процессы могут аварийно завершиться в любой момент, без предварительного предупреждения
- **Неполная обработка событий**: оригинальные вызовы CLI затрудняют захват детальных событий процесса логического вывода модели и выполнения инструментов, как если бы можно было видеть только результат, но не процесс размышлений
- **Сложное управление сеансами**: отсутствие эффективных механизмов повторного использования и восстановления сеансов, каждый раз приходится начинать заново, что тоже довольно утомительно
- **Проблемы совместимости**: параметры CLI часто обновляются, необходимо постоянно поддерживать логику совместимости параметров, что подобно битве с ветряными мельницами

Эти проблемы постепенно проявлялись в повседневной разработке, особенно когда требовалось отслеживать в реальном времени процесс логического вывода модели (thinking) и статус выполнения инструментов — ограничения вызовов CLI становились особенно очевидными. Мы тоже в итоге поняли, что нам нужен более низкоуровневый и полный способ интеграции — ведь все дороги ведут в Рим, просто некоторые дороги легче, а другие немного более извилисты.

## О HagiCode

Решение, представленное в этой статье, основано на нашем практическом опыте в проекте [HagiCode](https://hagicode.com). HagiCode — это проект с открытым исходным кодом AI-помощника по коду, и в процессе разработки нам нужно было глубоко интегрировать различные возможности GitHub Copilot — от базового автодополнения кода до сложного многоэтапного диалога и вызова инструментов. Эти реальные потребности подтолкнули нас обновиться от вызовов CLI до официальной интеграции SDK.

Если вас интересует практическое решение, представленное в этой статье, это означает, что наша инженерная практика может быть вам полезна — тогда сам проект HagiCode тоже заслуживает внимания. Возможно, в конце статьи вы discover больше информации и ссылок о проекте, кто знает...

Технические преимущества этого слоистого дизайна на самом деле довольно практичны:

1. **Разделение задач**: основная бизнес-логика отделена от деталей реализации SDK — ведь каждый слой делает свое дело, воды не мешают
2. **Тестируемость**: через интерфейс `ICopilotSdkGateway` можно легко проводить модульное тестирование, тестирование тоже не так уж затруднительно
3. **Повторное использование**: HagiCode.Libs может использоваться в нескольких проектах, пишется один раз, используется в нескольких местах
4. **Обслуживаемость**: обновления SDK требуют модификации только слоя Gateway, верхний код менять не нужно, очень красиво

## Основная реализация

### Процесс аутентификации

Аутентификация — это первый и самый важный шаг интеграции SDK — ведь если даже не войти в дверь, о последнем не может быть и речи. Мы спроектировали гибкую конфигурацию аутентификации, поддерживающую несколько источников аутентификации:

```csharp
// CopilotProvider.cs - конфигурация источников аутентификации
public class CopilotOptions
{
    public bool UseLoggedInUser { get; set; } = true;
    public string? GitHubToken { get; set; }
    public string? CliUrl { get; set; }
}

// Преобразование в запрос SDK
return new CopilotSdkRequest(
    GitHubToken: options.AuthSource == CopilotAuthSource.GitHubToken
        ? options.GitHubToken
        : null,
    UseLoggedInUser: options.AuthSource != CopilotAuthSource.GitHubToken
);
```

Преимущества этого дизайна на самом деле тоже довольно очевидны:

- Поддержка режима вошедшего пользователя (без токена), подходящего для сценариев десктопа — пользователи просто входят под своим аккаунтом
- Поддержка режима GitHub Token, подходящего для серверного развертывания — унифицированное управление тоже удобно
- Поддержка переопределения URL Copilot CLI, удобного для конфигурации корпоративного прокси — в корпоративной среде всегда есть какие-то особые правила

В реальном использовании этот гибкий способ аутентификации значительно упрощает работу по конфигурации в различных сценариях развертывания. Десктоп может использовать статус входа пользователя в Copilot, серверная часть может осуществлять унифицированное управление через токены. Как бы то ни было, каждый берет то, что ему нужно.

### Обработка потока событий

Одной из самых мощных возможностей SDK является полный захват потока событий. Мы реализовали систему диспетчеризации событий, способную обрабатывать в реальном времени различные события SDK — ведь знание процесса и только результата, чувство всё-таки разное:

```csharp
// GitHubCopilotSdkGateway.cs - основная логика диспетчеризации событий
internal static SessionEventDispatchResult DispatchSessionEvent(
    SessionEvent evt, bool sawDelta)
{
    switch (evt)
    {
        case AssistantReasoningEvent reasoningEvent:
            // Захват процесса логического вывода модели
            events.Add(new CopilotSdkStreamEvent(
                CopilotSdkStreamEventType.ReasoningDelta,
                Content: reasoningEvent.Data.Content));
            break;

        case ToolExecutionStartEvent toolStartEvent:
            // Захват начала вызова инструмента
            events.Add(new CopilotSdkStreamEvent(
                CopilotSdkStreamEventType.ToolExecutionStart,
                ToolName: toolStartEvent.Data.ToolName,
                ToolCallId: toolStartEvent.Data.ToolCallId));
            break;

        case ToolExecutionCompleteEvent toolCompleteEvent:
            // Захват завершения вызова инструмента и результата
            events.Add(new CopilotSdkStreamEvent(
                CopilotSdkStreamEventType.ToolExecutionEnd,
                Content: ExtractToolExecutionContent(toolCompleteEvent)));
            break;

        default:
            // Необработанные события сохраняются как RawEvent
            events.Add(new CopilotSdkStreamEvent(
                CopilotSdkStreamEventType.RawEvent,
                RawEventType: evt.GetType().Name));
            break;
    }
}
```

Ценность этой реализации, как бы то ни было:

- **Полный захват процесса логического вывода модели** (thinking): пользователи могут видеть процесс размышлений AI, а не только конечный результат — как знание ответа не так хорошо, как знание того, как он был получен
- **Отслеживание статуса выполнения инструментов в реальном времени**: знание того, какие инструменты запущены, когда завершены, какие результаты возвращены
- **Нулевая потеря событий**: через механизм fallback к RawEvent, обеспечивается запись всех событий, ничего не пропадает

В реальном использовании HagiCode эти детальные события позволяют пользователям глубже понимать процесс работы AI, особенно при отладке сложных задач — это все-таки somewhat полезно.

### Обработка совместимости с CLI

После миграции от вызовов CLI к SDK мы обнаружили, что некоторые原有 параметры CLI больше не применимы в SDK. Для поддержания обратной совместимости мы реализовали систему фильтрации параметров — ведь если старая конфигурация не работает, это тоже довольно головная боль:

```csharp
public static CopilotCliArgumentBuildResult BuildCliArgs(CopilotOptions options)
{
    // Фильтрация неподдерживаемых параметров, сохранение совместимых параметров
    // Генерация диагностической информации
}
```

Преимущества этого подхода:

- Автоматическая фильтрация несовместимых параметров CLI, избежание ошибок выполнения — сбой программы не шутка
- Генерация четкой диагностической информации об ошибках, помощь разработчикам в быстром определении проблем
- Гарантия стабильности SDK, независимость от изменений параметров CLI

В процессе обновления этот механизм обработки совместимости помог нам плавно перейти, старые файлы конфигурации все еще могут использоваться, нужно только постепенно настраивать в соответствии с диагностической информацией — это тоже своего рода постепенный процесс.

### Пулинг во время выполнения

Стоимость создания сеансов в Copilot SDK довольно высока, частое создание и уничтожение сеансов влияет на производительность. Мы реализовали систему управления пулом сеансов — как вода в бассейне, лучше оставить для следующего использования, чем каждый раз наполнять заново:

```csharp
// CopilotProvider.cs - управление пулом сеансов
await using var lease = await _poolCoordinator.AcquireCopilotRuntimeAsync(
    request,
    async ct => await _gateway.CreateRuntimeAsync(sdkRequest, ct),
    cancellationToken);

if (lease.IsWarmLease)
{
    // Повторное использование существующего сеанса
    yield return CreateSessionReusedMessage();
}

await foreach (var eventData in lease.Entry.Resource.SendPromptAsync(...))
{
    yield return MapEvent(eventData);
}
```

Преимущества пулинга сеансов:

- **Повторное использование сеансов**: запросы с одинаковым sessionId могут повторно использовать существующий сеанс, уменьшая накладные расходы на запуск
- **Поддержка восстановления сеансов**: после прерывания сети можно восстановить предыдущее состояние сеанса — ведь кто может гарантировать, что сеть всегда стабильна
- **Автоматическое управление пулом**: автоматическая очистка истекших сеансов, избежание утечки ресурсов

В реальном использовании HagiCode пулинг сеансов значительно повысил скорость отклика, особенно при обработке непрерывных диалогов эффект очевиден — такое улучшение всё-таки можно почувствовать.

### Интеграция с Orleans

HagiCode использует Orleans как распределенный фреймворк, мы интегрировали Copilot SDK в Orleans Grain — распределенные системы, звучат сложно, но на деле тоже довольно удобно:

```csharp
// GitHubCopilotGrain.cs - распределенное выполнение
public async IAsyncEnumerable<GitHubCopilotResponse> ExecuteCommandStreamAsync(
    string command,
    CancellationToken token = default)
{
    var provider = await aiProviderFactory.GetProviderAsync(AIProviderType.GitHubCopilot);

    await foreach (var chunk in provider.SendMessageAsync(request, null, token))
    {
        // Отображение в унифицированный формат ответа
        yield return BuildChunkResponse(chunk, startedAt);
    }
}
```

Преимущества интеграции с Orleans:

- **Унифицированная абстракция AI Provider**: можно легко переключаться между различными AI-провайдерами — сегодня используешь этот, завтра тот, тоже довольно гибко
- **Изоляция мультизантности**: Copilot-сеансы разных пользователей изолированы друг от друга, воды не мешают
- **Сохранение состояния сеансов**: состояние сеансов может быть восстановлено после перезапуска сервера, перезапуск не страшен потери данных

Для сценариев, требующих обработки большого количества одновременных запросов, распределенные возможности Orleans обеспечивают хорошую масштабируемость — ведь когда одиночный сервер не справляется, приходится полагаться на распределенные системы.

### Меры предосторожности при использовании

В реальном использовании мы подытожили некоторые моменты, на которые стоит обратить внимание — некоторые из них получены из опыта на собственных ошибках:

**Конфигурация времени ожидания запуска**: первый запуск Copilot CLI требует значительного времени, рекомендуется устанавливать `StartupTimeout` не менее 30 секунд. Если это первый вход, может потребоваться больше времени — ведь при первом входе все равно нужно пройти проверку, ничего не поделаешь.

**Управление разрешениями**: в производственной среде избегайте использования `AllowAllTools: true`. Используйте белый список `AllowedTools` для управления доступными инструментами, черный список `DeniedTools` для запрещения опасных операций. Это может эффективно предотвратить выполнение AI опасных команд — в безопасности лучше быть осторожным.

**Управление сеансами**: запросы с одинаковым `sessionId` автоматически повторно используют сеанс. Состояние сеанса сохраняется через `ProviderSessionId`. Отмена операций передается через `CancellationTokenSource` — хорошее управление сеансами означает хороший пользовательский опыт.

**Диагностический вывод**: несовместимые параметры CLI генерируют сообщения типа `diagnostic`. Оригинальные события SDK сохраняются как тип `event.raw`. Сообщения об ошибках включают классификацию (тайм-аут запуска, несовместимость параметров и т. д.), что удобно для устранения неполадок — возможность быстрого определения проблемы при возникновении проблемы тоже является своего рода утешением.

### Лучшие практики

Основываясь на нашем реальном опыте, здесь мы поделимся некоторыми лучшими практиками — своего рода резюме:

**1. Использование белого списка инструментов**

```csharp
var request = new AIRequest
{
    Prompt = "Анализировать этот файл",
    AllowedTools = new[] { "Read", "Grep", "Bash(git:*)" }
};
```

Через белый список явно укажите разрешенные инструменты, избегайте случайных операций AI. Особенно для инструментов с правами записи (например, Edit), нужно быть особенно осторожным — ведь удаление базы данных, никто не хочет переживать.

**2. Установка разумного времени ожидания**

```csharp
options.Timeout = 3600;  // 1 час
options.StartupTimeout = 60;  // 1 минута
```

Устанавливайте подходящее время ожидания в соответствии со сложностью задачи. Слишком короткое может привести к прерыванию задачи, слишком длинное может привести к waste ресурсов, ожидающих запросов без ответа — во всем нужна мера, перебор хуже недостатка.

**3. Включение повторного использования сеансов**

```csharp
options.SessionId = "my-session-123";
```

Установка одинакового sessionId для связанных задач может повторно использовать предыдущий контекст сеанса, повышая скорость отклика — контекст иногда довольно важен.

**4. Обработка потоковых ответов**

```csharp
await foreach (var chunk in provider.StreamAsync(request))
{
    switch (chunk.Type)
    {
        case StreamingChunkType.ThinkingDelta:
            // Обработка процесса логического вывода
            break;
        case StreamingChunkType.ToolCallDelta:
            // Обработка вызова инструмента
            break;
        case StreamingChunkType.ContentDelta:
            // Обработка текстового вывода
            break;
    }
}
```

Потоковые ответы могут отображать в реальном времени прогресс обработки AI, повышая пользовательский опыт. Особенно для длительных задач реальная обратная связь очень важна — смотреть на полосу прогресса лучше, чем просто ждать.

**5. Обработка ошибок и повторные попытки**

```csharp
try
{
    await foreach (var chunk in provider.StreamAsync(request))
    {
        // Обработка ответа
    }
}
catch (CopilotSessionException ex)
{
    // Обработка исключения сеанса
    logger.LogError(ex, "Copilot session failed");
    // Решение о повторной попытке в зависимости от типа исключения
}
```

Надлежащая обработка ошибок и механизм повторных попыток могут повысить стабильность системы — никто не может гарантировать, что программа никогда не выдаст ошибок, главное — хорошо обрабатывать ошибки, когда они возникают.

## Заключение

Обновление от вызовов CLI до интеграции SDK принесло значительную ценность проекту HagiCode — как бы то ни было, это обновление того стоило:

- **Повышение стабильности**: SDK предоставляет более стабильный интерфейс, не подверженный изменениям версии CLI — не нужно каждый день беспокоиться об обновлениях версии
- **Полнота функциональности**: возможность захватывать полный поток событий, включая процесс логического вывода и статус выполнения инструментов — можно видеть и процесс, и результат
- **Эффективность разработки**: типобезопасный интерфейс SDK делает разработку более эффективной, уменьшает количество ошибок выполнения — с проверкой типов спокойнее на душе
- **Пользовательский опыт**: обратная связь о событиях в реальном времени позволяет пользователям более четко понимать процесс работы AI — знать, о чем он думает, лучше, чем ничего не знать

Это обновление не просто замена технического решения, но и оптимизация всей архитектуры слоя AI-возможностей. Через слоистый дизайн и абстрактные интерфейсы мы получили лучшую обслуживаемость и масштабируемость — когда архитектура сделана хорошо, дальнейшие дела легче.

Если вы рассматриваете возможность интеграции GitHub Copilot в ваш проект .NET, надеюсь, практический опыт из этой статьи поможет вам избежать некоторых ошибок. Официальный SDK действительно более стабилен и полон, чем вызовы CLI, стоит потратить время на понимание и освоение — ведь правильный инструмент может сделать дело вдвое быстрее, это не безосновательно.
