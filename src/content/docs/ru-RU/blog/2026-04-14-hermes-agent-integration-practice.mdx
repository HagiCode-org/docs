## Практика интеграции Hermes Agent: от протокола до продакшена

> Делится полным опытом интеграции Hermes Agent в HagiCode, включая адаптацию протокола ACP, управление пулом сессий, синхронизацию контрактов между фронтендом и бэкендом и другие ключевые опыты.

## Предыстория

В процессе создания платформы AI-поддержки кодирования HagiCode команде потребовалось интегрировать Agent-фреймворк, способный работать как локально, так и масштабироваться в облаке. После исследований Hermes Agent от Nous Research был выбран в качестве базового движка для комплексного Agent.

На самом деле выбор фреймворка — задача не из простых, но и не слишком сложная. Ведь на рынке есть немало достойных Agent-фреймворков, просто протокол ACP и инструментальная система Hermes действительно впечатляют, что идеально соответствует требованиям HagiCode «и то, и это» — локальная разработка, командная работа и облачное масштабирование. Но чтобы по-настоящему интегрировать Hermes в продакшн-систему, необходимо решить ряд инженерных проблем, и это уж точно не шутка.

Технологический стек HagiCode построен на Orleans для создания распределённых систем, фронтенд использует React + TypeScript. Интеграция Hermes требует сохранения единства существующей архитектуры, одновременно делая Hermes «первоклассным» исполнителем наряду с ClaudeCode, OpenCode и другими. Сказать-то легко, а сделать... ну, примерно так.

В этой статье мы делимся опытом интеграции Hermes Agent в проект HagiCode, надеясь, что это будет полезно командам с аналогичными требованиями. Ведь ошибки, которые мы совершили, необязательно повторять другим.

## О HagiCode

Решение, описанное в этой статье, основано на нашем опыте в проекте [HagiCode](https://hagicode.com). HagiCode — это платформа AI-поддержки кодирования, поддерживающая единый доступ и управление различными AI Provider. В процессе интеграции Hermes Agent мы разработали универсальный уровень абстракции Provider, позволяющий новым типам Agent бесшовно подключаться к существующей системе.

Если вас интересует HagiCode, добро пожаловать на [GitHub](https://github.com/HagiCode-org/site) для получения дополнительной информации. Чем больше людей посмотрит, тем больше сил.

## Архитектурный дизайн

### Концепция слоистой архитектуры

Интеграция Hermes в HagiCode использует чёткую слоистую архитектуру, где каждый слой выполняет свои функции:

**Базовый уровень бэкенда**
- `HermesCliProvider`: Реализует интерфейс `IAIProvider`, как единый вход AI Provider
- `HermesPlatformConfiguration`: Управляет путём к исполняемому файлу Hermes, параметрами, аутентификацией и другими конфигурациями
- `ICliProvider<HermesOptions>`: Базовая CLI-абстракция от HagiCode.Libs, обрабатывает жизненный цикл дочерних процессов

**Транспортный уровень**
- `StdioAcpTransport`: Общается с дочерним процессом Hermes ACP через стандартный ввод/вывод
- Методы протокола ACP: `initialize`, `authenticate`, `session/new`, `session/prompt`

**Уровень выполнения**
- `HermesGrain`: Реализация Orleans Grain, обрабатывает выполнение распределённых сессий
- `CliAcpSessionPool`: Пул сессий, повторно использует дочерние процессы ACP, избегая частых запусков

**Фронтенд-уровень**
- `ExecutorAvatar`: Визуальная идентификация и иконка Hermes
- `executorTypeAdapter`: Логика отображения типов Provider
- SignalR для обмена сообщениями в реальном времени: поддерживает согласованность идентичности Hermes в потоке сообщений

Такая слоистая архитектура позволяет каждому уровню развиваться независимо. Например, в будущем для добавления нового способа передачи (например, WebSocket) потребуется изменить только транспортный уровень. Ведь кто захочет перебирать всю систему только из-за изменения способа передачи? Утомительно же.

### Единая абстракция интерфейса

Все AI Provider реализуют интерфейс `IAIProvider` — это ключевой дизайн архитектуры HagiCode:

```csharp
public interface IAIProvider
{
    string Name { get; }
    ProviderCapabilities Capabilities { get; }

    IAsyncEnumerable<AIStreamingChunk> StreamAsync(
        AIRequest request,
        CancellationToken cancellationToken = default);

    Task<AIResponse> ExecuteAsync(
        AIRequest request,
        CancellationToken cancellationToken = default);
}
```

`HermesCliProvider` реализует этот интерфейс и находится на равных с `ClaudeCodeProvider`, `OpenCodeProvider` и другими. Преимущества этого дизайна:

1. **Заменимость**: Смена Provider не влияет на бизнес-логику верхнего уровня
2. **Тестируемость**: Можно легко создать Mock Provider для модульного тестирования
3. **Расширяемость**: Добавление нового Provider требует только реализации интерфейса

В конце концов, интерфейсы — это как правила. С правилами все могут гармонично сосуществовать, проявляя свои сильные стороны и не мешая друг другу. Разве это не прекрасно?

## Основная реализация

### Реализация уровня Provider

`HermesCliProvider` — это ядро всей интеграции, отвечающее за координацию различных компонентов для выполнения одного AI-вызова:

```csharp
public sealed class HermesCliProvider : IAIProvider, IVersionedAIProvider
{
    private readonly ICliProvider<LibsHermesOptions> _provider;
    private readonly ConcurrentDictionary<string, string> _sessionBindings;

    public ProviderCapabilities Capabilities { get; } = new()
    {
        SupportsStreaming = true,
        SupportsTools = true,
        SupportsSystemMessages = true,
        SupportsArtifacts = false
    };

    public async IAsyncEnumerable<AIStreamingChunk> StreamAsync(
        AIRequest request,
        [EnumeratorCancellation] CancellationToken cancellationToken = default)
    {
        // 1. Разрешить ключ привязки сессии
        var bindingKey = ResolveBindingKey(request.CessionId);

        // 2. Получить или создать сессию Hermes через пул сессий
        var options = new HermesOptions
        {
            ExecutablePath = _platformConfiguration.ExecutablePath,
            Arguments = _platformConfiguration.Arguments,
            SessionId = _sessionBindings.TryGetValue(bindingKey, out var sessionId) ? sessionId : null,
            WorkingDirectory = request.WorkingDirectory,
            Model = request.Model
        };

        // 3. Выполнить и собрать потоковый ответ
        await foreach (var message in _provider.ExecuteAsync(options, request.Prompt, cancellationToken))
        {
            // 4. Отобразить сообщения ACP в AIStreamingChunk
            if (_responseMapper.TryConvertToStreamingChunk(message, out var chunk))
            {
                yield return chunk;
            }
        }
    }
}
```

Здесь есть несколько ключевых моментов дизайна:

1. **Привязка сессии**: Через `CessionId` несколько запросов привязываются к одному дочернему процессу Hermes, реализуя непрерывность контекста многоуровневого диалога
2. **Отображение ответов**: Преобразование формата сообщений Hermes ACP в унифицированный формат `AIStreamingChunk`
3. **Потоковая обработка**: Использование `IAsyncEnumerable` для поддержки настоящих потоковых ответов

На самом деле привязка сессии похожа на отношения между людьми. Как только установлена связь, последующее общение имеет контекст, не нужно начинать с нуля каждый раз. Только эту связь нужно поддерживать, иначе она может прерваться.

### Адаптация протокола ACP

Hermes использует протокол ACP (Agent Communication Protocol), отличный от традиционного HTTP API. ACP — это протокол на основе стандартного ввода/вывода с несколькими особенностями:

1. **Маркер запуска**: После запуска процесса Hermes выводит маркер `//ready`
2. **Динамическая аутентификация**: Метод аутентификации не фиксирован, требуется согласование через протокол
3. **Повторное использование сессии**: Повторное использование установленной сессии через `SessionId`
4. **Распределённые ответы**: Полный ответ может быть распределён по нескольким уведомлениям `session/update`

HagiCode обрабатывает эти особенности через `StdioAcpTransport`:

```csharp
public class StdioAcpTransport
{
    public async Task InitializeAsync(CancellationToken cancellationToken)
    {
        // Ожидание маркера //ready
        var readyLine = await _outputReader.ReadLineAsync(cancellationToken);
        if (readyLine != "//ready")
        {
            throw new InvalidOperationException("Hermes did not send ready signal");
        }

        // Отправка запроса initialize
        await SendRequestAsync(new
        {
            jsonrpc = "2.0",
            id = 1,
            method = "initialize",
            @params = new
            {
                protocolVersion = "2024-11-05",
                capabilities = new { },
                clientInfo = new { name = "HagiCode", version = "1.0.0" }
            }
        }, cancellationToken);
    }
}
```

Протокол похож на взаимопонимание между людьми. С взаимопониманием общение становится намного плавнее. Только для установления взаимопонимания требуется время, и адаптация неизбежна.

### Управление пулом сессий

Частый запуск дочерних процессов Hermes связан с большими накладными расходами, поэтому мы реализовали механизм пула сессий:

```csharp
services.AddSingleton(static _ =>
{
    var registry = new CliProviderPoolConfigurationRegistry();
    registry.Register("hermes", new CliPoolSettings
    {
        MaxActiveSessions = 50,
        IdleTimeout = TimeSpan.FromMinutes(10)
    });
    return registry;
});
```

Ключевые параметры пула сессий:
- `MaxActiveSessions`: Контроль верхнего предела параллелизма, избежание истощения ресурсов
- `IdleTimeout`: Тайм-аут простоя, баланс между стоимостью запуска и использованием памяти

На практике мы обнаружили:
1. Слишком короткий тайм-аут простоя приводит к частым перезапускам, слишком долгий — занимает память
2. Верхний предел параллелизма нужно настроить в соответствии с реальной нагрузкой, слишком большое значение может вызвать зависание системы
3. Необходимо мониторить использование пула сессий для своевременной настройки параметров

Это похоже на многие жизненные выборы: слишком агрессивный подход ведёт к проблемам, слишком осторожный — упускает возможности. Просто найдите баланс.

### Синхронизация контрактов

Фронтенд и бэкенд поддерживают согласованность контрактов через генерацию OpenAPI. Бэкенд определяет перечисление `AIProviderType`:

```csharp
public enum AIProviderType
{
    Unknown,
    ClaudeCode,
    OpenCode,
    HermesCli  // Новое
}
```

Фронтенд генерирует соответствующий тип TypeScript через OpenAPI, обеспечивая согласованность значений перечисления. Это ключ к избежанию отображения «Unknown» на фронтенде.

Контракт похож на обещание. Что договорились, то и выполняйте, иначе возникнет неловкая ситуация с «Unknown».

## Управление конфигурацией

Конфигурация Hermes управляется через `appsettings.json`:

```json
{
  "Providers": {
    "HermesCli": {
      "ExecutablePath": "hermes",
      "Arguments": "acp",
      "StartupTimeoutMs": 10000,
      "ClientName": "HagiCode",
      "Authentication": {
        "PreferredMethodId": "api-key",
        "MethodInfo": {
          "api-key": "your-api-key-here"
        }
      },
      "SessionDefaults": {
        "Model": "claude-sonnet-4-20250514",
        "ModeId": "default"
      }
    }
  }
}
```

Этот дизайн, управляемый конфигурацией, обеспечивает гибкость:
- Можно переопределить путь к исполняемому файлу для удобства разработки и тестирования
- Можно настроить параметры запуска для адаптации к разным версиям Hermes
- Можно настроить информацию аутентификации для поддержки различных методов аутентификации

Конфигурация похожа на жизненные выборы. Достаточно вариантов, и всегда можно найти подходящий. Только иногда слишком много вариантов вызывает затруднения при выборе.

## Практический опыт

### Проверка работоспособности

Реализация надёжного Provider требует полной проверки работоспособности:

```csharp
public async Task<ProviderTestResult> PingAsync(CancellationToken cancellationToken = default)
{
    var response = await ExecuteAsync(new AIRequest
    {
        Prompt = "Reply with exactly PONG.",
        CessionId = null,
        AllowedTools = Array.Empty<string>(),
        WorkingDirectory = ResolveWorkingDirectory(null)
    }, cancellationToken);

    var success = string.Equals(response.Content.Trim(), "PONG", StringComparison.OrdinalIgnoreCase);
    return new ProviderTestResult
    {
        ProviderName = Name,
        Success = success,
        ResponseTimeMs = stopwatch.ElapsedMilliseconds,
        ErrorMessage = success ? null : $"Unexpected Hermes ping response: '{response.Content}'."
    };
}
```

Проверка работоспособности требует внимания:
1. Используйте простые тестовые случаи, избегайте сложных сценариев
2. Установите разумное время ожидания
3. Записывайте время ответа для анализа производительности

Как людям нужны медицинские осмотры, так и системы нуждаются в проверке работоспособности. Раннее обнаружение и раннее лечение, чтобы потом не было больших проблем.

### Обработка распространённых проблем

**Сбой аутентификации**
- Проверьте соответствие `Authentication.PreferredMethodId` методам аутентификации, фактически поддерживаемым Hermes
- Подтвердите правильность формата информации аутентификации (API Key, Bearer Token и т. д.)

**Тайм-аут сессии**
- Увеличьте значение `StartupTimeoutMs`
- Проверьте достижимость MCP-сервера
- Просмотрите использование системных ресурсов

**Неполный ответ**
- Убедитесь в правильной агрегации уведомлений `session/update` и конечного результата
- Проверьте логику отмены потоковой обработки
- Проверьте полноту обработки ошибок

**Фронтенд отображает Unknown**
- Подтвердите, что генерация OpenAPI включает значение перечисления `HermesCli`
- Проверьте правильность отображения типов
- Очистите кеш браузера и перегенерируйте типы

Проблемы всегда будут. Только когда сталкиваетесь с проблемой, не паникуйте, медленно ищите причину, и решение найдётся. Ведь решений всегда больше, чем проблем.

### Рекомендации по оптимизации производительности

1. **Используйте пул сессий**: Повторное использование дочерних процессов ACP, снижение накладных расходов на запуск
2. **Разумно установите тайм-ауты**: Баланс между памятью и стоимостью запуска
3. **Повторно используйте идентификаторы сессий**: Пакетные задачи используют один и тот же `CessionId`
4. **Настройте MCP по требованию**: Избегайте ненужных вызовов инструментов

Производительность похожа на эффективность в жизни. Сделал правильно —事半功倍, сделал неправильно —事倍功半. Только чтобы найти ту «правильную» точку, нужны опыт и удача.

## Заключение

Интеграция Hermes Agent в продакшн-систему требует рассмотрения проблем на нескольких уровнях:

1. **Архитектурный уровень**: Разработка единого интерфейса Provider, реализация заменяемой компонентной архитектуры
2. **Уровень протокола**: Правильная обработка особенностей протокола ACP, таких как маркеры запуска, динамическая аутентификация и т. д.
3. **Уровень производительности**: Повторное использование ресурсов через пул сессий, баланс между стоимостью запуска и использованием памяти
4. **Уровень фронтенда**: Обеспечение синхронизации контрактов, предоставление согласованного визуального опыта

Практика HagiCode показывает, что с хорошим слоистым дизайном и конфигурационным управлением сложные Agent-системы можно бесшовно интегрировать в существующую архитектуру.

На самом деле эти принципы кажутся простыми, только при реальном выполнении всегда встречаются различные проблемы. Но это нормально, решённые проблемы становятся опытом, нерешённые — уроками, и всё имеет ценность.

Прекрасные вещи или люди не обязательно должны быть в вашем владении, пока они остаются прекрасными, просто любуйтесь их красотой. То же самое с технологиями: пока система становится лучше, какой фреймворк или протокол использовать — не так уж и важно......
