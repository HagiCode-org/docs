## Глубокая интеграция и практическое использование StreamJsonRpc в HagiCode

> В этой статье подробно описано, как проект HagiCode (ранее PCode) успешно интегрировал библиотеку коммуникации StreamJsonRpc от Microsoft для замены оригинальной пользовательской реализации JSON-RPC, а также как были решены технические проблемы и архитектурные вызовы, возникшие в процессе интеграции.

## Предпосылки

StreamJsonRpc — это официально поддерживаемая Microsoft библиотека JSON-RPC для .NET и TypeScript, известная своей сильной типобезопасностью, автоматической генерацией прокси и зрелым механизмом обработки исключений. В проекте HagiCode было принято решение об интеграции StreamJsonRpc для взаимодействия с внешними AI-инструментами, такими как iflow CLI и OpenCode CLI, через ACP (Agent Communication Protocol), а также для устранения затрат на обслуживание и потенциальных ошибок, связанных с ранней пользовательской реализацией JSON-RPC. Однако в процессе интеграции возникли специфические для потокового JSON-RPC вызовы, особенно при обработке привязки прокси-целей и распознавании обобщённых параметров.

Для решения этих проблем мы приняли смелое решение: полностью перестроить систему сборки с нуля. Последствия этого решения могут быть больше, чем вы ожидаете, и я расскажу о них подробно.

## Анализ

Текущий проект находится на критическом этапе интеграции протокола ACP и сталкивается со следующими техническими проблемами и архитектурными вызовами:

### 1. Ограничения пользовательской реализации
Оригинальная реализация JSON-RPC находится в `src/HagiCode.ClaudeHelper/AcpImp/` и включает такие компоненты, как `JsonRpcEndpoint` и `ClientSideConnection`. Поддержка этой пользовательской кодовой базы обходится дорого, и ей не хватает расширенных возможностей зрелой библиотеки, таких как отчётность о прогрессе и поддержка отмены.

### 2. Препятствия интеграции StreamJsonRpc
При попытке миграции существующего паттерна `CallbackProxyTarget` в StreamJsonRpc было обнаружено, что метод `_rpc.AddLocalRpcTarget(target)` не может распознать цели, созданные через паттерн прокси. В частности, StreamJsonRpc не мог автоматически разделить свойства обобщённого типа `T` на параметры RPC-метода, из-за чего серверная сторона не могла корректно обрабатывать вызовы методов, инициированные клиентом.

### 3. Путаница в архитектурном разделении
Существующий `ClientSideConnection` смешивает транспортный уровень (WebSocket/Stdio), уровень протокола (JSON-RPC) и бизнес-уровень (интерфейс ACP Agent), что приводит к нечёткому разделению ответственности. Также наблюдается отсутствие привязки методов в `AcpAgentCallbackRpcAdapter`.

### 4. Отсутствие журналирования
На транспортном уровне WebSocket отсутствует журналирование исходного содержимого JSON, что затрудняет определение при отладке RPC, вызвана ли проблема сериализацией или сетью.

## Решение

Для решения описанных проблем мы применили систематическое решение, оптимизируя по трём направлениям: архитектурный рефакторинг, интеграция библиотеки и улучшение отладки.

### 1. Полная миграция на StreamJsonRpc

#### Удаление устаревшего кода
Удалены `JsonRpcEndpoint.cs`, `AgentSideConnection.cs` и связанные пользовательские сериализаторы, такие как `JsonRpcMessageJsonConverter`.

#### Интеграция официальной библиотеки
Добавлен NuGet-пакет `StreamJsonRpc` с использованием его класса `JsonRpc` для обработки основной логики коммуникации.

#### Абстракция транспортного уровня
Определён интерфейс `IAcpTransport` для унифицированной обработки режимов передачи `WebSocket` и `Stdio`, обеспечивая разделение уровня протокола от транспортного уровня.

```csharp
// Определение интерфейса IAcpTransport
public interface IAcpTransport
{
    Task SendAsync(string message, CancellationToken cancellationToken = default);
    Task<string> ReceiveAsync(CancellationToken cancellationToken = default);
    Task CloseAsync(CancellationToken cancellationToken = default);
}

// Реализация WebSocket-транспорта
public class WebSocketTransport : IAcpTransport
{
    private readonly WebSocket _webSocket;

    public WebSocketTransport(WebSocket webSocket)
    {
        _webSocket = webSocket;
    }

    // Реализация методов отправки и приёма
    // ...
}

// Реализация Stdio-транспорта
public class StdioTransport : IAcpTransport
{
    private readonly StreamReader _reader;
    private readonly StreamWriter _writer;

    public StdioTransport(StreamReader reader, StreamWriter writer)
    {
        _reader = reader;
        _writer = writer;
    }

    // Реализация методов отправки и приёма
    // ...
}
```

### 2. Исправление проблемы распознавания прокси-целей

#### Анализ `CallbackProxyTarget`
Инспекция существующей логики генерации динамического прокси для определения корневой причины, по которой StreamJsonRpc не может его распознать. В большинстве случаев это происходит из-за того, что прокси-объект не раскрывает публично фактические сигнатуры методов или использует типы параметров, не поддерживаемые StreamJsonRpc.

#### Рефакторинг передачи параметров
Разделение обобщённых свойств на явные параметры RPC-методов. Вместо зависимости от динамических свойств определены конкретные DTO запросов/ответов, чтобы StreamJsonRpc мог корректно распознавать сигнатуры методов через рефлексию.

```csharp
// Исходный подход с обобщёнными свойствами
public class CallbackProxyTarget<T>
{
    public Func<T, Task> Callback { get; set; }
}

// Рефакторинговый подход с конкретными методами
public class ReadTextFileRequest
{
    public string FilePath { get; set; }
}

public class ReadTextFileResponse
{
    public string Content { get; set; }
}

public interface IAcpAgentCallback
{
    Task<ReadTextFileResponse> ReadTextFileAsync(ReadTextFileRequest request);
    // Другие методы...
}
```

#### Использование `Attach` вместо `AddLocalRpcTarget`
В некоторых сложных сценариях ручное проксирование объекта `JsonRpc` и обработка `RpcConnection` может быть более гибким, чем прямое добавление локальной цели.

### 3. Реализация привязки методов и усиленное журналирование

#### Реализация `AcpAgentCallbackRpcAdapter`
Обеспечено явное реализование интерфейса прокси StreamJsonRpc этим компонентом с маппингом методов, определённых протоколом ACP (таких как `ReadTextFileAsync`), на обработчики обратных вызовов StreamJsonRpc.

#### Интеграция журналирования
Перехват и запись исходного текста JSON-RPC запросов и ответов в конвейере обработки сообщений WebSocket или Stdio. Использование `ILogger` для вывода исходного payload до парсинга и после сериализации для упрощения диагностики проблем форматирования.

```csharp
// Обёртка транспорта с усиленным журналированием
public class LoggingAcpTransport : IAcpTransport
{
    private readonly IAcpTransport _innerTransport;
    private readonly ILogger<LoggingAcpTransport> _logger;

    public LoggingAcpTransport(IAcpTransport innerTransport, ILogger<LoggingAcpTransport> logger)
    {
        _innerTransport = innerTransport;
        _logger = logger;
    }

    public async Task SendAsync(string message, CancellationToken cancellationToken = default)
    {
        _logger.LogTrace("Sending message: {Message}", message);
        await _innerTransport.SendAsync(message, cancellationToken);
    }

    public async Task<string> ReceiveAsync(CancellationToken cancellationToken = default)
    {
        var message = await _innerTransport.ReceiveAsync(cancellationToken);
        _logger.LogTrace("Received message: {Message}", message);
        return message;
    }

    public async Task CloseAsync(CancellationToken cancellationToken = default)
    {
        _logger.LogDebug("Closing connection");
        await _innerTransport.CloseAsync(cancellationToken);
    }
}
```

### 4. Рефакторинг архитектурного разделения

#### Транспортный уровень (`AcpRpcClient`)
Обёртка соединения StreamJsonRpc, отвечающая за вызовы `InvokeAsync` и управление жизненным циклом соединения.

```csharp
public class AcpRpcClient : IDisposable
{
    private readonly JsonRpc _rpc;
    private readonly IAcpTransport _transport;

    public AcpRpcClient(IAcpTransport transport)
    {
        _transport = transport;
        _rpc = new JsonRpc(new StreamRpcTransport(transport));
        _rpc.StartListening();
    }

    public async Task<TResponse> InvokeAsync<TResponse>(string methodName, object parameters)
    {
        return await _rpc.InvokeAsync<TResponse>(methodName, parameters);
    }

    public void Dispose()
    {
        _rpc.Dispose();
        _transport.Dispose();
    }

    // StreamRpcTransport — это адаптер StreamJsonRpc для IAcpTransport
    private class StreamRpcTransport : IDuplexPipe
    {
        // Реализация интерфейса IDuplexPipe
        // ...
    }
}
```

#### Уровень протокола (`IAcpAgentClient` / `IAcpAgentCallback`)
Определение чётких интерфейсов клиент-агент и агент-клиент. Удалён циклический фабричный паттерн `Func<IAcpAgent, IAcpClient>` и заменён на внедрение зависимостей или прямую регистрацию обратных вызовов.

## Практика

На основе лучших практик StreamJsonRpc и проектного опыта приведены следующие ключевые рекомендации по реализации:

### 1. Строго типизированные DTO лучше динамических объектов
Основное преимущество StreamJsonRpc заключается в строгой типизации. Не используйте `dynamic` или `JObject` для передачи параметров. Вместо этого определяйте явные C# POCO-классы как параметры для каждого RPC-метода. Это не только решает проблему распознавания прокси-целей, но и позволяет обнаруживать ошибки типов на этапе компиляции.

Пример: замена обобщённых свойств в `CallbackProxyTarget` на конкретные классы, такие как `ReadTextFileRequest` и `WriteTextFileRequest`.

### 2. Явное объявление имён методов
Используйте атрибут `[JsonRpcMethod]` для явного указания имён RPC-методов вместо зависимости от маппинга по умолчанию. Это предотвращает сбои вызовов, вызванные различиями в стиле именования, такими как PascalCase vs camelCase.

```csharp
public interface IAcpAgentCallback
{
    [JsonRpcMethod("readTextFile")]
    Task<ReadTextFileResponse> ReadTextFileAsync(ReadTextFileRequest request);

    [JsonRpcMethod("writeTextFile")]
    Task WriteTextFileAsync(WriteTextFileRequest request);
}
```

### 3. Использование обратных вызовов состояния соединения
StreamJsonRpc предоставляет событие `JsonRpc.ConnectionLost`. Обязательно отслеживайте его для обработки неожиданных завершений процессов или разрывов сети — это более оперативно, чем reliance только на обнаружение сбоев Grain Orleans.

```csharp
_rpc.ConnectionLost += (sender, e) =>
{
    _logger.LogError("RPC connection lost: {Reason}", e.ToString());
    // Обработка логики переподключения или уведомление пользователя
};
```

### 4. Послойное журналирование
- **Уровень Trace**: запись полного исходного JSON-запроса/ответа.
- **Уровень Debug**: запись стеков вызовов методов и сводки параметров.
- **Примечание**: убедитесь, что журналы не содержат чувствительных Authorization-токенов или Base64-кодированного содержимого больших файлов.

### 5. Обработка специфики потоковой передачи
StreamJsonRpc нативно поддерживает `IAsyncEnumerable`. При реализации потоковых prompt-ответов для ACP используйте `IAsyncEnumerable` напрямую вместо пользовательской логики постраничного обхода. Это значительно упрощает код для потоковой обработки.

```csharp
public interface IAcpAgentCallback
{
    [JsonRpcMethod("streamText")]
    IAsyncEnumerable<string> StreamTextAsync(StreamTextRequest request);
}
```

### 6. Паттерн адаптера
Сохраняйте разделение `ACPSession` и `ClientSideConnection`. `ACPSession` должен сосредоточиться на управлении состоянием Orleans и бизнес-логике (например, постановке сообщений в очередь), используя объект соединения StreamJsonRpc через композицию, а не наследование.

## Заключение

Посредством комплексной интеграции StreamJsonRpc проект HagiCode успешно решил проблемы высокого уровня обслуживания, функциональных ограничений и путаницы в архитектурном разделении оригинальной пользовательской реализации. Ключевые улучшения включают:

1. Замену динамических свойств на строго типизированные DTO, повышая удобство поддержки и надёжность
2. Реализацию абстракции транспортного уровня и разделения уровня протокола, повышая ясность архитектуры
3. Усиление возможностей журналирования для упрощения диагностики коммуникационных проблем
4. Внедрение поддержки потоковой передачи для упрощения реализации потоковой обработки

Эти улучшения обеспечивают HagiCode более стабильную и эффективную коммуникационную основу, позволяя лучше взаимодействовать с внешними AI-инструментами и закладывая прочный фундамент для будущего расширения функциональности.
