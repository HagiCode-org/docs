## Практический путь интеграции Pi Agent в HagiCode

> В этой статье мы делимся практическим опытом интеграции Pi agent как первоклассной точки входа рабочего процесса в проекте HagiCode, включая архитектурное проектирование, реализацию ключевых компонентов и детали адаптации фронтенда и бэкенда.

## Контекст

Всё началось с одного вопроса: в проекте HagiCode Mono, хотя в `repos/Hagicode.Libs` уже был реализован повторно используемый `PiProvider`, в `repos/hagicode-core` и `repos/web` Pi ещё не был повышен до проектного уровня первоклассного Agent CLI. Это похоже на то, как если бы у вас была отличная обувь, но вы бы ещё не завязали шнурки - можно ходить, но чего-то не хватает.

Существующий `PiProvider` расположен в `HagiCode.Libs/src/HagiCode.Libs.Providers/Pi/PiProvider.cs`, он реализует протокол CLI на основе JSON с разделением строк (`--mode json --print`), поддерживает управление сессиями, вызовы инструментов и потоковый вывод. Этот код написан очень хорошо, просто он пока лежит там и не активирован.

Такая ситуация создала разрыв: базовые возможности Pi полны, но цепочка проектного уровня интеграции (от пользовательской конфигурации до мониторинга выполнения) отсутствует. Это похоже на то, как если бы вы нарисовали хорошую картину, но не повесили бы её на стену, чтобы другие могли её оценить. Поэтому нужно интегрировать Pi как нового активного провайдера `PiCli` во всю систему, чтобы он стал первоклассной точкой входа рабочего процесса, такой же, как и другие Agent CLI.

В конце концов, мы хотим полный опыт, а не разрозненные фрагменты.

## О HagiCode

Решение, описанное в этой статье, основано на нашем практическом опыте в проекте [HagiCode](https://hagicode.com). HagiCode - это проект интеллектуального помощника по коду, и в процессе разработки нам нужно интегрировать нескольких поставщиков AI-возможностей. Решение по интеграции Pi agent, описанное в этой статье, - это именно такой повторно используемый паттерн интеграции, который мы обобщили на практике, и он имеет справочное значение для аналогичных сценариев интеграции с несколькими провайдерами. Адрес GitHub проекта - [github.com/HagiCode-org/site](https://github.com/HagiCode-org/site).

## Архитектурное проектирование

Интеграция Pi agent использует **thin adapter pattern**, а не повторно реализует протокол процесса Pi на уровне core. Это решение не является особенно сложным - просто подумали, зачем заново изобретать колесо?

毕竟：

1. **Избегание повторной реализации**: Логика построения параметров, запуска процессов, разбора JSON и обработки ошибок в `PiProvider` уже очень полная, повторная реализация создаст два источника поведения и две матрицы тестов. Это ладно, но поддержка будет очень проблематичной.
2. **Поддержание согласованности**: Существующие CLI, такие как Kimi, Gemini, Reasonix, DeepAgents и т.д., все используют одинаковый способ интеграции через thin adapter, делегируя реализации на уровне libs. Все так делают, просто следуйте.
3. **Разделение ответственности**: `hagicode-core` фокусируется на контрактах времени выполнения и бизнес-логике, детали процессов передаются `HagiCode.Libs` для обработки. Каждый делает свою работу, разве это не прекрасно?

Это проектирование позволяет HagiCode быстро интегрировать новых поставщиков AI-возможностей, сохраняя простоту основного уровня. В конце концов, простота - это красота, эффективность - это деньги.

## Реализация ключевых компонентов

### Перечисление Provider и Factory

В `hagicode-core/src/PCode.Models/AIProviderType.cs` добавлено новое значение перечисления `PiCli = 13`:

```csharp
public enum AIProviderType
{
    ClaudeCodeCli = 0,
    // ... другие provider
    PiCli = 13,
}
```

Это значение перечисления является источником идентичности провайдера, оно повлияет на генерируемое OpenAPI перечисление `PCode_Models_AIProviderType.ts` на фронтенде, ветку создания в `AIProviderFactory`, а также логику анализа провайдера и определения активного провайдера. В `AIProviderFactory` регистрируется новая ветка создания:

```csharp
case AIProviderType.PiCli:
    provider = new PiCliProvider(
        logger,
        configuration,
        providerFactory.GetRequiredService<ICliProvider<PiOptions>>(),
        providerFactory.GetService<IAgentCliRuntimeEnvironmentResolver>());
    break;
```

На самом деле этот код не является чем-то особенным, просто перечисление плюс case в switch. Просто они очень важны, как ноты на нотном стане, добавляясь по одной, они могут сыграть полную мелодию.

### Реализация Thin Adapter

`PiCliProvider.cs` - это основной thin adapter, он реализует интерфейсы `IAIProvider`, `IVersionedAIProvider` и `IAsyncDisposable`. Через конструктор он получает `ICliProvider<PiOptions>` (из `HagiCode.Libs`), отображает `AIRequest` / `ProviderConfiguration` на `PiOptions`, затем делегирует выполнение, ping, запрос версии и другие операции базовому провайдеру.

Ключевым моментом является обработка специфического для Pi потока событий JSON, включая события `assistant.thought`, `assistant`, `terminal.completed` и т.д. Эти события должны быть правильно разобраны и преобразованы в стандартный формат системы в процессе потокового вывода. Это немного похоже на перевод, нужно передать смысл точно, переводя с одного языка на другой.

### Предустановка основной профессии

В `main-professions.yaml` добавлена запись основной профессии `profession-pi`:

```yaml
- Id: "profession-pi"
  Name: "Pi"
  Family: "pi"
  Summary: "hero.professionCopy.primary.pi.summary"
  Icon: "executor-avatar:Pi"
  SourceLabel: "hero.professionCopy.sources.aiProvidersPiCli"
  ProviderType: "PiCli"
  SortOrder: 59
  DefaultEnabled: true
  DefaultParameters:
    binary: "pi"
    provider: "omniroute"
    thinking: "balanced"
```

Это гарантирует, что снимок бэкенда и каталог фронтенда fallback имеют согласованную идентичность Pi, конфигурацию Pi можно управлять через существующий редактор Hero, и добавление Pi не нарушит потребляемость существующих записей основной профессии. В конце концов, мы не хотим, чтобы добавление новой функции испортило существующие вещи, так будет больше потерь, чем пользы.

### Реестр мониторинга

`AgentCliMonitoringRegistry` должен добавить дескриптор мониторинга для Pi, чтобы система могла анализировать исполняемый путь, отображать название бренда, выполнять проверки работоспособности и отображать статус Pi в строке состояния и деталях работоспособности:

```csharp
new AgentCliMonitoringDescriptor(
    CliId: "pi",
    DisplayName: "Pi",
    ProviderType: AIProviderType.PiCli,
    DisplayOrder: 13,
    Strategy: AgentCliMonitoringStrategy.Grain,
    NotConfiguredMessage: "Pi CLI is not configured or executable not found.",
    EnabledPaths: [],
    ExecutablePathConfigPaths: ["Hero:PrimaryProfessions:Pi:ExecutablePath"],
    DefaultExecutablePath: "pi")
```

Система мониторинга похожа на приборную панель, она показывает, как работает машина. Статус Pi виден с первого взгляда, так что пользователи могут знать, всё ли в порядке.

## Адаптация фронтенда

### Адаптация типа исполнителя

Фронтенд должен обновить `executorTypeAdapter.ts`, добавить логику распознавания типа Pi:

```typescript
const PI_IDS = new Set<string>([
  PCode_Models_AIProviderType.PI_CLI,
  'Pi',
  'PiCli',
  'pi',
  'picli',
  'pi-cli',
]);

export function isPi(value: string): boolean {
  const normalized = normalize(value);
  const normalizedLower = normalized.toLowerCase();
  return PI_IDS.has(normalized)
    || normalizedLower.includes('pi-cli')
    || normalizedLower.includes('picli')
    || normalizedLower === 'pi';
}
```

Это похоже на то, как если бы вы дали Pi несколько имён, как бы вы его ни называли, он знает, что вы зовёте его. В конце концов, неважно, как вы его называете, важно знать, кто вы.

### Каталог Fallback Hero

В `hero.ts` добавлена запись fallback для Pi, чтобы фронтенд мог нормально отображать конфигурацию Pi, даже если данные бэкенда не загружены:

```typescript
{
  id: "profession-pi",
  name: "Pi",
  family: "pi",
  summary: "hero.professionCopy.primary.pi.summary",
  icon: "executor-avatar:Pi",
  sourceLabel: "hero.professionCopy.sources.aiProvidersPiCli",
  providerType: AIProviderType.PI_CLI,
  sortOrder: 59,
  isReadOnly: true,
  managedParameterKeys: {
    // Параметры key, поддерживаемые в первой версии
  },
  defaultParameters: {
    binary: "pi",
    provider: "omniroute",
    thinking: "balanced",
  },
}
```

Fallback похож на запасной план, если бэкенд упадёт или данные не загрузятся, фронтенд всё равно сможет работать нормально. В конце концов, никто не хочет потом суетиться.

### Локализованные тексты и формы

В `locales/*/common/{hero,settings}.yml` добавлены переводы, связанные с Pi, и в `HeroCliEquipmentForm.tsx` добавлен блок полей конфигурации для Pi, поддерживающий поля binary, provider, thinking, sessionDirectory и переключатели инструментов/сессий.

Первая версия Pi раскрывает только минимально необходимые поля, сложные функции, такие как редакторы allowlist/denylist инструментов и переменных среды, явно отложены до последующих изменений. В конце концов, не стоит пытаться сделать всё сразу, медленно - быстрее.

## Примеры конфигурации

### Конфигурация бэкенда

Настройка параметров Pi в `appsettings.yml`:

```yaml
Hero:
  PrimaryProfessions:
    Pi:
      ExecutablePath: /usr/local/bin/pi
      Provider: omniroute
      Thinking: balanced
      SessionDirectory: ~/.pi/sessions
      NoSession: false
      DisableAllTools: false
      DisableBuiltinTools: false
```

### Конфигурация фронтенда

Настройка профессии Pi в редакторе Hero:

```typescript
{
  id: "my-pi-profession",
  name: "My Pi Profession",
  family: "pi",
  providerType: AIProviderType.PI_CLI,
  primaryModel: {
    provider: "PiCli",
    model: "glm-4.7",
    providerSettings: {
      provider: "omniroute",
      thinking: "balanced",
      sessionDirectory: "/Users/username/.pi/sessions",
      noSession: false,
      disableAllTools: false,
      disableBuiltinTools: false,
    },
  },
}
```

Файлы конфигурации похожи на рецепты, следуйте им, и вы получите хорошее блюдо. Просто иногда, даже следуя рецепту, можно испортить блюдо... 不过这次的配置倒是很清晰，应该没什么问题。

## Проверка и тестирование

### Проверка бэкенда

В тестах проверяем предустановку основной профессии:

```csharp
var snapshot = await presetProvider.GetSnapshotAsync();
var piProfession = snapshot.FindById("profession-pi");
piProfession.ShouldNotBeNull();
piProfession.ProviderType.ShouldBe(AIProviderType.PiCli);
piProfession.Family.ShouldBe("pi");
```

Также нужно проверить доступность Pi и проверки работоспособности:

```bash
# Проверка исполняемого файла Pi
which pi
pi --version

# Проверка регистрации провайдера бэкенда
curl http://localhost:35168/api/health/agent-cli/pi
```

Тестирование похожа на экзамен, только после сдачи можно сказать, что вы действительно знаете. Просто иногда, сдача экзамена не означает, что вы действительно понимаете, но по крайней мере показывает, что вы можете правильно отвечать на вопросы.

### Проверка фронтенда

```typescript
// Проверка разбора типа и отображаемого имени
expect(resolveExecutorVisualType('pi-cli')).toBe('Pi');
expect(resolveExecutorVisualType(PCode_Models_AIProviderType.PI_CLI)).toBe('Pi');

expect(piFallback?.providerType).toBe(AIProviderType.PI_CLI);
```

Эти тестовые примеры охватывают основные логические пути, гарантируя, что Pi может быть правильно идентифицирован и отображён. В конце концов, то, что показывается пользователям, не должно содержать ошибок.

## Устранение общих проблем

### Исполняемый файл Pi не найден

Если проверка работоспособности возвращает "Pi executable was not found.", нужно проверить, есть ли pi в PATH, или подтвердить, что настроенный путь правильный. Решение - убедиться, что `pi` установлен и находится в PATH, или настроить правильный `ExecutablePath` в `appsettings.yml`.

Это похоже на то, когда не можешь найти ключи от дома, нужно подумать, не положил ли ты их не в то место. На самом деле решение довольно простое: либо вернуть ключи на прежнее место, либо поменять замок.

### Поля конфигурации не распознаются

Если при запуске возникает ошибка "PiCli runtime settings [...] are not supported", проверьте, используются ли только поля конфигурации, поддерживаемые в первой версии. Поля, поддерживаемые в первой версии: `provider`, `thinking`, `sessionDirectory`, `noSession`, `disableAllTools`, `disableBuiltinTools`.

Иногда это просто жадность, хочется слишком много функций, а система их не поддерживает. На самом деле функциональности первой версии достаточно, жадность не ведёт к хорошему.

### Невозможно выбрать Pi на фронтенде

Если в редакторе Hero нет опции Pi, проверьте, был ли запущен `npm run generate:api` для повторной генерации перечислений фронтенда, есть ли запись `profession-pi` в `hero.ts`, и правильно ли добавлены локализованные тексты.

Устранение проблем похоже на поиск потерянных предметов, нужно делать это шаг за шагом. В конце концов, слепой поиск не приведёт к результату, нужно искать логически.

## Лучшие практики

1. **Использование thin adapter pattern**: Не повторно реализуйте протокол процесса на уровне core, делегируйте провайдеру уровня libs. Это позволяет избежать повторной реализации и сохранить согласованность кода. В конце концов, повторное изобретение колеса не только утомительно, но и легко приводит к проблемам.

2. **Поддержание согласованности именования**: Используйте единые соглашения об именовании на фронтенде и бэкенде, чтобы избежать путаницы. Перечисление провайдера использует `PiCli`, CLI ID использует `"pi"`, отображаемое имя использует `"Pi"`. Хорошие имена снижают стоимость коммуникации.

3. **Приоритет использования предустановок**: Первая версия должна основываться на предустановке `profession-pi`, а не требовать от пользователя ручной настройки. Это позволяет пользователям быстро начать работу и снизить сложность настройки. Пользователи любят простые вещи, сложные - пускай приходят ко мне.

4. **Внимание к сообщениям об ошибках**: Убедитесь, что сообщения об ошибках ясны и actionable, помогая пользователям быстро находить проблемы. Чёткие сообщения об ошибках не заставят пользователей сходить с ума из-за одной ошибки.

5. **Совместимость версий**: Учитывая стабильность сериализации значений перечисления `AIProviderType`, изменения нужно обрабатывать осторожно. Значение перечисления `AIProviderType.PiCli = 13` не может быть легко изменено. В конце концов, изменение этого значения может нарушить обратную совместимость, это будет проблематично.

## Заключение

Используя thin adapter pattern, мы успешно интегрировали Pi agent в систему HagiCode, сделав его первоклассной точкой входа рабочего процесса, такой же, как и другие Agent CLI. Ключевые преимущества этого решения:

- Избегание повторной реализации, повторное использование существующего `PiProvider` на уровне libs
- Поддержание согласованного способа интеграции с существующими провайдерами, снижение стоимости поддержки
- Реализация полной цепочки от пользовательской конфигурации до мониторинга выполнения

Практика HagiCode доказала, что thin adapter pattern - это эффективное решение для интеграции поставщиков AI-возможностей. Это позволяет нам быстро поддерживать новых агентов, сохраняя стабильность и поддерживаемость системы.

На самом деле работа с технологией именно такова: найти хороший паттерн, затем повторно использовать его. Так можно быстро продвигаться вперёд, не теряя направления. Это похоже на ходьбу: нашёл хорошую дорогу и продолжай идти, только иногда останавливайся, чтобы полюбоваться пейзажем...

Если вы также занимаетесь интеграцией AI-возможностей с несколькими провайдерами, надеюсь, это решение даст вам вдохновение. Если вам интересен проект HagiCode, добро пожаловать на GitHub для общения. В конце концов, технологии - это то, что развивается при общении.
