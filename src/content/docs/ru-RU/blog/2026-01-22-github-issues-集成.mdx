## Построение интеграции с GitHub Issues с нуля: подход HagiCode с прямым подключением фронтенда

> В этой статье описан полный процесс интеграции GitHub Issues в платформу HagiCode. Мы рассмотрим, как с помощью архитектуры «прямое подключение фронтенда + минимальный бэкенд» реализовать безопасную аутентификацию OAuth и эффективную синхронизацию Issues, сохраняя при этом легковесность бэкенда.

## Предпосылки: зачем интегрировать GitHub?

Как платформа AI-ассистированной разработки, основная ценность HagiCode заключается в соединении идей с реализацией. Но на практике мы обнаружили, что после завершения Proposal в HagiCode пользователям часто приходится вручную копировать содержимое в GitHub Issues для отслеживания проекта.

Это создаёт несколько очевидных проблем:

1.  **Разрыв рабочего процесса**: пользователям приходится переключаться между двумя системами. Опыт работы неудобен, и ключевая информация легко теряется при копировании и вставке.
2.  **Сложности в совместной работе**: другие члены команды привыкли проверять задачи на GitHub и не могут напрямую видеть прогресс предложений в HagiCode.
3.  **Повторяющийся ручной труд**: при каждом обновлении предложения кто-то должен вручную обновлять соответствующий Issue на GitHub, что добавляет ненужные затраты на обслуживание.

Для решения этой проблемы мы решили внедрить функцию **GitHub Issues Integration**, связав сессии HagiCode с репозиториями GitHub для обеспечения «синхронизации в один клик».

## Технический выбор: прямое подключение фронтенда vs проксирование через бэкенд

При проектировании подхода к интеграции перед нами были два варианта: традиционная «модель проксирования через бэкенд» и более радикальная «модель прямого подключения фронтенда».

### Сравнение подходов

В традиционной **модели проксирования через бэкенд** каждый запрос от фронтенда должен сначала проходить через наш бэкенд, который затем вызывает GitHub API. Это централизует логику, но также создаёт значительную нагрузку на бэкенд:

1.  **Раздутый бэкенд**: пришлось бы написать отдельный клиент-обёртку для GitHub API и обрабатывать сложный конечный автомат OAuth.
2.  **Риски с токеном**: GitHub-токен пользователя пришлось бы хранить в базе данных бэкенда. Даже с шифрованием это увеличивает поверхность атаки.
3.  **Затраты на разработку**: потребовались бы миграции базы данных для хранения токенов и дополнительный сервис синхронизации для поддержки.

**Модель прямого подключения фронтенда** значительно легче. В этом подходе мы используем бэкенд только для самого чувствительного этапа «обмена секретами» (OAuth callback). После получения токена мы сохраняем его непосредственно в `localStorage` браузера. Последующие операции, такие как создание Issues и обновление комментариев, отправляются напрямую из фронтенда в GitHub по HTTP.

| Измерение | Модель проксирования через бэкенд | Модель прямого подключения фронтенда |
| :--- | :--- | :--- |
| **Сложность бэкенда** | Требуется полный OAuth-сервис и GitHub API клиент | Нужен только endpoint OAuth callback |
| **Управление токенами** | Должен храниться зашифрованным в БД, есть риск утечки | Хранится в браузере, доступен только пользователю |
| **Затраты на реализацию** | Требуются миграции БД и разработка нескольких сервисов | В основном работа на фронтенде |
| **Пользовательский опыт** | Централизованная логика, но задержка сервера может быть выше | Мгновенный отклик при прямом взаимодействии с GitHub |

Поскольку мы стремились к быстрой интеграции и минимальным изменениям бэкенда, **мы в конечном итоге выбрали «модель прямого подключения фронтенда»**. Это как выдача браузеру «временного пропуска» — получив пропуск, браузер может самостоятельно обращаться к GitHub, не запрашивая каждый раз разрешения у администратора бэкенда.

## Основной дизайн: поток данных и безопасность

Определив архитектуру, нам нужно было спроектировать конкретный поток данных. Суть процесса синхронизации — как безопасно получить токен и эффективно его использовать.

### Общая схема архитектуры

Всю систему можно абстрагировать в три роли: браузер (фронтенд), бэкенд HagiCode и GitHub.

```text
+--------------+        +--------------+        +--------------+
|  Frontend    |        |   Backend    |        |   GitHub     |
|  React       |        |   ASP.NET    |        |   REST API   |
|              |        |              |        |              |
|  +--------+  |        |              |        |              |
|  | OAuth  |--+--------> /callback    |        |              |
|  | Flow   |  |        |              |        |              |
|  +--------+  |        |              |        |              |
|              |        |              |        |              |
|  +--------+  |        |  +--------+  |        |  +--------+  |
|  | GitHub |  +------------> Session |  +----------> Issues |  |
|  | API    |  |        |  |Metadata|  |        |  |        |  |
|  | Direct |  |        |  +--------+  |        |  +--------+  |
|  +--------+  |        |              |        |              |
+--------------+        +--------------+        +--------------+
```

**Ключевой момент**: только один маленький шаг OAuth (обмен code на токен) должен проходить через бэкенд. После этого основная работа (создание Issues) выполняется напрямую между фронтендом и GitHub.

### Подробный разбор потока синхронизации данных

Когда пользователь нажимает кнопку «Sync to GitHub» в интерфейсе HagiCode, происходит серия сложных действий:

```text
Пользователь нажимает "Sync to GitHub"
         │
         ▼
1. Фронтенд проверяет localStorage на наличие GitHub-токена
         │
         ▼
2. Форматирование содержимого Issue (преобразование Proposal в Markdown)
         │
         ▼
3. Фронтенд напрямую вызывает GitHub API для создания/обновления Issue
         │
         ▼
4. Вызов API бэкенда HagiCode для обновления Session.metadata (сохранение URL Issue и другой информации)
         │
         ▼
5. Бэкенд транслирует событие SessionUpdated через SignalR
         │
         ▼
6. Фронтенд получает событие и обновляет UI для отображения состояния «Синхронизировано»
```

### Дизайн безопасности

Безопасность всегда приоритет при интеграции сторонних сервисов. Мы учли следующее:

1.  **Защита от CSRF-атак**: генерация случайного параметра `state` при OAuth-перенаправлении и сохранение его в `sessionStorage`. Строгая проверка state при обратном вызове для предотвращения подделки запросов.
2.  **Изолированное хранение токена**: токен хранится только в `localStorage` браузера. Благодаря политике Same-Origin, только скрипты HagiCode могут его читать, что исключает риск утечки из серверной базы данных.
3.  **Обработка ошибок**: для распространённых ошибок GitHub API (таких как 401 — истёкший токен, 422 — ошибка валидации, 429 — ограничение скорости) предусмотрена специальная логика обработки, обеспечивающая дружелюбные сообщения для пользователя.

### 2. OAuth-поток на фронтенде

Это точка входа для подключения. Мы используем стандартный Authorization Code Flow.

```typescript
// src/HagiCode.Client/src/services/githubOAuth.ts

// Генерация URL авторизации и перенаправление
export async function generateAuthUrl(): Promise<string> {
  const state = generateRandomString(); // Генерация случайной строки для защиты от CSRF
  sessionStorage.setItem('hagicode_github_state', state);

  const params = new URLSearchParams({
    client_id: clientId,
    redirect_uri: window.location.origin + '/settings?tab=github&oauth=callback',
    scope: ['repo', 'public_repo'].join(' '),
    state: state,
  });

  return `https://github.com/login/oauth/authorize?${params.toString()}`;
}

// Обработка обмена code на токен на странице обратного вызова
export async function exchangeCodeForToken(code: string, state: string): Promise<GitHubToken> {
  // 1. Проверка state для защиты от CSRF
  const savedState = sessionStorage.getItem('hagicode_github_state');
  if (state !== savedState) throw new Error('Invalid state parameter');

  // 2. Вызов API бэкенда для обмена токена
  // Примечание: это должно проходить через бэкенд, так как ClientSecret нельзя раскрывать на фронтенде
  const response = await fetch('/api/GitHubOAuth/callback', {
    method: 'POST',
    headers: { 'Content-Type': 'application/json' },
    body: JSON.stringify({ code, state, redirectUri: window.location.origin + '/settings?tab=github&oauth=callback' }),
  });

  if (!response.ok) throw new Error('Failed to exchange token');

  const token = await response.json();

  // 3. Сохранение в LocalStorage
  saveToken(token);
  return token;
}
```

### 3. Клиент-обёртка для GitHub API

Получив токен, нам нужен надёжный инструмент для вызова GitHub API.

```typescript
// src/HagiCode.Client/src/services/githubApiClient.ts

const GITHUB_API_BASE = 'https://api.github.com';

// Основная обёртка запросов
async function githubApi<T>(endpoint: string, options: RequestInit = {}): Promise<T> {
  const token = localStorage.getItem('gh_token');
  if (!token) throw new Error('Not connected to GitHub');

  const response = await fetch(`${GITHUB_API_BASE}${endpoint}`, {
    ...options,
    headers: {
      ...options.headers,
      Authorization: `Bearer ${token}`,
      Accept: 'application/vnd.github.v3+json', // Указание версии API
    },
  });

  // Логика обработки ошибок
  if (!response.ok) {
    if (response.status === 401) throw new Error('GitHub Token истёк, пожалуйста, переподключитесь');
    if (response.status === 403) throw new Error('Нет доступа к репозиторию или превышен лимит скорости');
    if (response.status === 422) throw new Error('Ошибка валидации Issue, возможно, заголовок дублируется');
    throw new Error(`GitHub API Error: ${response.statusText}`);
  }

  return response.json();
}

// Создание Issue
export async function createIssue(owner: string, repo: string, data: { title: string, body: string, labels: string[] }) {
  return githubApi(`/repos/${owner}/${repo}/issues`, {
    method: 'POST',
    body: JSON.stringify(data),
  });
}
```

### 4. Форматирование содержимого и синхронизация

Последний шаг — преобразование данных сессии HagiCode в формат GitHub Issue. Это похоже на работу «переводчика».

```typescript
// Преобразование объекта Session в строку Markdown
function formatIssueForSession(session: Session): string {
  let content = `# ${session.title}\n\n`;
  content += `**> HagiCode Session:** #${session.code}\n`;
  content += `**> Status:** ${session.status}\n\n`;
  content += `## Description\n\n${session.description || 'No description provided.'}\n\n`;

  // Если это тип Proposal, добавляем дополнительные поля
  if (session.type === 'proposal') {
    content += `## Chief Complaint\n\n${session.chiefComplaint || ''}\n\n`;
    // Добавляем глубокую ссылку для перехода из GitHub обратно в HagiCode
    content += `---\n\n**[View in HagiCode](hagicode://sessions/${session.id})**\n`;
  }

  return content;
}

// Основная логика при нажатии кнопки синхронизации
const handleSync = async (session: Session) => {
  try {
    const repoInfo = parseRepositoryFromUrl(session.repoUrl); // Разбор URL репозитория
    if (!repoInfo) throw new Error('Invalid repository URL');

    toast.loading('Синхронизация с GitHub...');

    // 1. Форматирование содержимого
    const issueBody = formatIssueForSession(session);

    // 2. Вызов API
    const issue = await githubApiClient.createIssue(repoInfo.owner, repoInfo.repo, {
      title: `[HagiCode] ${session.title}`,
      body: issueBody,
      labels: ['hagicode', 'proposal', `status:${session.status}`],
    });

    // 3. Обновление Session Metadata (сохранение ссылки на Issue)
    await SessionsService.patchApiSessionsSessionId(session.id, {
      metadata: {
        githubIssue: {
          owner: repoInfo.owner,
          repo: repoInfo.repo,
          issueNumber: issue.number,
          issueUrl: issue.html_url,
          syncedAt: new Date().toISOString(),
        }
      }
    });

    toast.success('Синхронизация успешна!');
  } catch (err) {
    console.error(err);
    toast.error('Ошибка синхронизации, проверьте Token или сеть');
  }
};
```

## Итоги и перспективы

С помощью подхода «прямое подключение фронтенда» мы реализовали бесшовную интеграцию с GitHub Issues при минимальном коде на бэкенде.

### Основные достижения
1.  **Высокая эффективность разработки**: изменения на бэкенде минимальны — в основном одно дополнительное поле в базе данных и простой endpoint OAuth callback. Большая часть логики реализована на фронтенде.
2.  **Хорошая безопасность**: токен не проходит через серверную базу данных, что снижает риск утечки.
3.  **Отличный пользовательский опыт**: запросы инициируются напрямую с фронтенда, поэтому скорость отклика высокая и не требуется проксирование через бэкенд.

### На что обратить внимание
При реальном развёртывании есть несколько подводных камней:
- **Настройки OAuth App**: не забудьте указать правильный `Authorization callback URL` в настройках GitHub OAuth App (обычно `http://localhost:3000/settings?tab=github&oauth=callback`).
- **Ограничения скорости**: GitHub API строго ограничивает неавторизованные запросы, но с токеном квоты обычно достаточно (5000 запросов/час).
- **Разбор URL**: пользователи вводят самые разные URL репозиториев, поэтому убедитесь, что ваше регулярное выражение обрабатывает суффиксы `.git`, SSH-форматы и аналогичные случаи.

### Дальнейшие улучшения
Текущая функция обеспечивает одностороннюю синхронизацию (HagiCode -> GitHub). В будущем мы планируем реализовать двустороннюю синхронизацию через GitHub Webhooks. Например, если Issue закрыт на GitHub, состояние сессии на стороне HagiCode также могло бы обновляться автоматически. Для этого нам потребуется создать endpoint для приёма вебхуков на бэкенде — это будет интересный следующий шаг.
