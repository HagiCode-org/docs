## Создание интеграции с GitHub Issues с нуля: практика прямого подключения фронтенда в HagiCode

> В этой статье записан полный процесс интеграции GitHub Issues в платформу HagiCode. Мы обсудим, как реализовать безопасную аутентификацию OAuth и эффективную синхронизацию Issues при сохранении лёгкости бэкенда через архитектуру «прямое подключение фронтенда + минимальный бэкенд».

## Предпосылки: зачем интегрировать GitHub?

HagiCode как платформа AI-поддержки разработки имеет своей основной ценностью соединение идей с реализацией. Однако на практике мы обнаружили, что после завершения Proposal (предложения) в HagiCode пользователям часто необходимо вручную копировать содержимое в GitHub Issues для отслеживания проекта.

Это создаёт несколько очевидных проблем:

1.  **Разрыв рабочего процесса**: Пользователям нужно переключаться между двумя системами, что не только нарушает плавность опыта, но и легко приводит к потере ключевой информации в процессе копирования и вставки.
2.  **Неудобство совместной работы**: Другие участники команды привыкли просматривать задачи на GitHub и не могут напрямую видеть прогресс предложений в HagiCode.
3.  **Дублирующая работа**: При каждом обновлении предложения необходимо вручную обновлять соответствующий Issue на GitHub, что увеличивает ненужные затраты на обслуживание.

Для решения этой проблемы мы решили внедрить функцию **GitHub Issues Integration**, соединив сессии HagiCode с репозиториями GitHub для реализации «синхронизации в один клик».

## Выбор технологий: прямое подключение фронтенда vs бэкенд-прокси

При разработке решения интеграции перед нами было два пути: традиционный «режим бэкенд-прокси» и более радикальный «режим прямого подключения фронтенда».

### Сравнение решений

В традиционном **режиме бэкенд-прокси** все запросы фронтенда сначала проходят через наш бэкенд, а затем бэкенд вызывает GitHub API. Хотя логика концентрирована, это создаёт значительную нагрузку на бэкенд:

1.  **Раздутый бэкенд**: Нужно написать специальную инкапсуляцию клиента GitHub API, а также обрабатывать сложный автомат состояний OAuth.
2.  **Риск Token**: GitHub Token пользователя должен храниться в базе данных бэкенда, хотя можно зашифровать, но это увеличивает поверхность безопасности.
3.  **Стоимость разработки**: Нужно миграцию базы данных для хранения Token, а также поддерживать дополнительный сервис синхронизации.

А **режим прямого подключения фронтенда** намного легче. В этом решении мы используем бэкенд только для обработки самого чувствительного этапа «обмена ключами» (OAuth callback), после получения Token он сразу сохраняется в localStorage браузера. Последующие операции, такие как создание Issue, обновление комментариев, выполняются фронтендом через прямые HTTP-запросы к GitHub.

| Измерение сравнения | Режим бэкенд-прокси | Режим прямого подключения фронтенда |
| :--- | :--- | :--- |
| **Сложность бэкенда** | Нужен полный OAuth сервис и клиент GitHub API | Требуется только одна конечная точка OAuth callback |
| **Управление Token** | Нужно зашифрованное хранение в базе данных, есть риск утечки | Хранится в браузере, виден только пользователю |
| **Стоимость внедрения** | Нужна миграция базы данных, разработка нескольких сервисов | В основном работа фронтенда |
| **Пользовательский опыт** | Логика единая, но задержка сервера может быть немного выше | Extremely быстрый ответ, прямое взаимодействие с GitHub |

Учитывая, что нам нужна быстрая интеграция и минимальные изменения бэкенда, **в итоге мы приняли «режим прямого подключения фронтенда»**. Это похоже на выдачу браузеру «временного пропуска», после получения пропуска браузер может сам идти в GitHub заниматься делами, не нужно каждый раз просить разрешения у администратора бэкенда.

## Основной дизайн: поток данных и безопасность

После определения архитектуры нам нужно спроектировать конкретный поток данных. Основной ключ всего процесса синхронизации заключается в том, как безопасно получить Token и эффективно использовать его.

### Схема общей архитектуры

Всю систему можно абстрагировать в три роли: браузер (фронтенд), бэкенд HagiCode, GitHub.

```text
+--------------+        +--------------+        +--------------+
|  Фронтенд    |        |    Бэкенд    |        |    GitHub    |
|   React      |        |   ASP.NET    |        |    REST API  |
|  +--------+  |        |              |        |              |
|  |  OAuth |--+--------> /callback    |        |              |
|  |  поток |  |        |              |        |              |
|  +--------+  |        |              |        |              |
|              |        |              |        |              |
|  +--------+  |        |  +--------+  |        |  +--------+  |
|  |GitHub  |  +------------>Session |  +----------> Issues |  |
|  |API     |  |        |  |Metadata|  |        |  |        |  |
|  |прямое  |  |        |  +--------+  |        |  +--------+  |
|  |подключ.| |        |              |        |              |
|  +--------+  |        |              |        |              |
+--------------+        +--------------+        +--------------+
```

**Ключевой момент в том**, что только маленький шаг OAuth (получение code для обмена на token) должен пройти через бэкенд, после этого тяжёлая работа (создание Issue) выполняется фронтендом напрямую с GitHub.

### Подробное описание потока данных синхронизации

Когда пользователь нажимает кнопку «Sync to GitHub» в интерфейсе HagiCode, происходит серия сложных действий:

```text
Пользователь нажимает "Sync to GitHub"
         │
         ▼
1. Фронтенд проверяет localStorage для получения GitHub Token
         │
         ▼
2. Форматирование содержимого Issue (преобразование Proposal в Markdown)
         │
         ▼
3. Фронтенд напрямую вызывает GitHub API для создания/обновления Issue
         │
         ▼
4. Вызов API бэкенда HagiCode для обновления Session.metadata (хранение информации о URL Issue и т. д.)
         │
         ▼
5. Бэкенд транслирует событие SessionUpdated через SignalR
         │
         ▼
6. Фронтенд получает событие, обновляет UI, отображая статус «синхронизировано»
```

### Дизайн безопасности

Вопросы безопасности всегда являются самой важной частью интеграции сторонних сервисов. Мы сделали следующие соображения:

1.  **Защита от CSRF-атак**: При перенаправлении OAuth генерируется случайный параметр `state` и сохраняется в sessionStorage. При обратном вызове строго проверяется state для предотвращения подделки запросов.
2.  **Изоляция хранения Token**: Token хранится только в `localStorage` браузера, используя политику того же источника (Same-Origin Policy), только скрипты HagiCode могут читать, избегая утечки базы данных на стороне сервера, затрагивающей пользователей.
3.  **Граница ошибок**: Для общих ошибок GitHub API (например, 401 Token истёк, 422 проверка не удалась, 429 ограничение скорости) разработана специальная логика обработки ошибок, дающая пользователю дружественные подсказки.

## Практика: детали реализации кода

Бумага получает поверхностно, давайте посмотрим, как реализован конкретный код.

### 1. Минимальные изменения бэкенда

Бэкенд нужно делать только две вещи: хранить информацию синхронизации, обрабатывать обратный вызов OAuth.

**Изменения базы данных**
Нам нужно только добавить столбец `Metadata` в таблицу `Sessions` для хранения расширенной информации в формате JSON.

### 2. Поток OAuth на фронтенде

Это вход подключения. Мы используем стандартный Authorization Code Flow.

```typescript
// src/HagiCode.Client/src/services/githubOAuth.ts

// Генерация URL авторизации и перенаправление
export async function generateAuthUrl(): Promise<string> {
  const state = generateRandomString(); // Генерация случайной строки для защиты от CSRF
  sessionStorage.setItem('hagicode_github_state', state);
  
  const params = new URLSearchParams({
    client_id: clientId,
    redirect_uri: window.location.origin + '/settings?tab=github&oauth=callback',
    scope: ['repo', 'public_repo'].join(' '),
    state: state,
  });
  
  return `https://github.com/login/oauth/authorize?${params.toString()}`;
}

// Обработка обмена Code на Token на странице обратного вызова
export async function exchangeCodeForToken(code: string, state: string): Promise<GitHubToken> {
  // 1. Проверка State для защиты от CSRF
  const savedState = sessionStorage.getItem('hagicode_github_state');
  if (state !== savedState) throw new Error('Invalid state parameter');

  // 2. Вызов API бэкенда для обмена Token
  // Примечание: здесь нужно пройти через бэкенд, так как нужен ClientSecret, нельзя раскрывать на фронтенде
  const response = await fetch('/api/GitHubOAuth/callback', {
    method: 'POST',
    headers: { 'Content-Type': 'application/json' },
    body: JSON.stringify({ code, state, redirectUri: window.location.origin + '/settings?tab=github&oauth=callback' }),
  });

  if (!response.ok) throw new Error('Failed to exchange token');
  
  const token = await response.json();
  
  // 3. Сохранение в LocalStorage
  saveToken(token);
  return token;
}
```

### 3. Инкапсуляция клиента GitHub API

После получения Token нам нужен мощный инструмент для вызова GitHub API.

```typescript
// src/HagiCode.Client/src/services/githubApiClient.ts

const GITHUB_API_BASE = 'https://api.github.com';

// Инкапсуляция основного запроса
async function githubApi<T>(endpoint: string, options: RequestInit = {}): Promise<T> {
  const token = localStorage.getItem('gh_token');
  if (!token) throw new Error('Not connected to GitHub');
  
  const response = await fetch(`${GITHUB_API_BASE}${endpoint}`, {
    ...options,
    headers: {
      ...options.headers,
      Authorization: `Bearer ${token}`,
      Accept: 'application/vnd.github.v3+json', // Указание версии API
    },
  });
  
  // Логика обработки ошибок
  if (!response.ok) {
    if (response.status === 401) throw new Error('GitHub Token недействителен, пожалуйста, переподключитесь');
    if (response.status === 403) throw new Error('Нет прав доступа к репозиторию или превышено ограничение скорости');
    if (response.status === 422) throw new Error('Проверка Issue не удалась, возможно, дубликат заголовка');
    throw new Error(`GitHub API Error: ${response.statusText}`);
  }
  
  return response.json();
}

// Создание Issue
export async function createIssue(owner: string, repo: string, data: { title: string, body: string, labels: string[] }) {
  return githubApi(`/repos/${owner}/${repo}/issues`, {
    method: 'POST',
    body: JSON.stringify(data),
  });
}
```

### 4. Форматирование содержимого и синхронизация

Последний шаг — преобразование данных Session HagiCode в формат GitHub Issue. Это немного похоже на работу «переводчика».

```typescript
// Преобразование объекта Session в строку Markdown
function formatIssueForSession(session: Session): string {
  let content = `# ${session.title}\n\n`;
  content += `**> HagiCode Session:** #${session.code}\n`;
  content += `**> Status:** ${session.status}\n\n`;
  content += `## Description\n\n${session.description || 'No description provided.'}\n\n`;
  
  // Если типа Proposal, добавление дополнительных полей
  if (session.type === 'proposal') {
    content += `## Chief Complaint\n\n${session.chiefComplaint || ''}\n\n`;
    // Добавление глубокой ссылки для удобства перехода из GitHub обратно в HagiCode
    content += `---\n\n**[View in HagiCode](hagicode://sessions/${session.id})**\n`;
  }
  
  return content;
}

// Основная логика при нажатии кнопки синхронизации
const handleSync = async (session: Session) => {
  try {
    const repoInfo = parseRepositoryFromUrl(session.repoUrl); // Разбор URL репозитория
    if (!repoInfo) throw new Error('Invalid repository URL');

    toast.loading('Синхронизация с GitHub...');
    
    // 1. Форматирование содержимого
    const issueBody = formatIssueForSession(session);
    
    // 2. Вызов API
    const issue = await githubApiClient.createIssue(repoInfo.owner, repoInfo.repo, {
      title: `[HagiCode] ${session.title}`,
      body: issueBody,
      labels: ['hagicode', 'proposal', `status:${session.status}`],
    });
    
    // 3. Обновление Session Metadata (сохранение ссылки Issue)
    await SessionsService.patchApiSessionsSessionId(session.id, {
      metadata: {
        githubIssue: {
          owner: repoInfo.owner,
          repo: repoInfo.repo,
          issueNumber: issue.number,
          issueUrl: issue.html_url,
          syncedAt: new Date().toISOString(),
        }
      }
    });

    toast.success('Синхронизация успешна!');
  } catch (err) {
    console.error(err);
    toast.error('Синхронизация не удалась, проверьте Token или сеть');
  }
};
```

## Заключение и перспективы

Через это решение «прямого подключения фронтенда» мы реализовали бесшовную интеграцию GitHub Issues с минимальным кодом бэкенда.

### Достижения
1.  **Высокая эффективность разработки**: Изменения бэкенда минимальны, в основном добавление одного поля в базу данных и простого интерфейса обратного вызова OAuth, большая часть логики выполняется на фронтенде.
2.  **Хорошая безопасность**: Token не проходит через базу данных сервера, снижая риск утечки.
3.  **Отличный пользовательский опыт**: Прямые запросы с фронтенда, быстрый ответ, не нужно проходить через транзит бэкенда.

### Меры предосторожности
При фактическом развертывании есть несколько ям, на которые нужно обратить внимание:
- **Настройки OAuth App**: Не забудьте заполнить правильный `Authorization callback URL` в настройках GitHub OAuth App (обычно `http://localhost:3000/settings?tab=github&oauth=callback`).
- **Ограничение скорости**: GitHub API строго ограничивает неаутентифицированные запросы, но после использования Token обычно достаточно (5000 раз/час).
- **Разбор URL**: URL репозитория, вводимый пользователем, очень разнообразен, не забудьте, что регулярное выражение должно соответствовать суффиксу `.git`, формату SSH и т. д.

### Последующие улучшения
Текущая функция всё ещё односторонняя синхронизация (HagiCode -> GitHub). В будущем мы планируем реализовать двустороннюю синхронизацию через GitHub Webhooks, например, закрытие Issue в GitHub, статус сессии в HagiCode также может автоматически обновляться. Это требует от нас раскрыть конечную точку приёма Webhook на бэкенде, это также следующая интересная работа.
