## С нуля: Как интегрировать Reasonix CLI в систему HagiCode

> В этой статье мы делимся полным техническим опытом интеграции Reasonix CLI как Agent Provider первого уровня в систему HagiCode, включая проектирование трёхуровневой архитектуры, ключевые технические решения и детали реализации на фронтенде и бэкенде.

## Предпосылки

Reasonix CLI — это довольно интересная штука. Это инструмент AI-ассистента кода, основанный на ACP (Agent Communication Protocol), который предоставляет мощные возможности стриминга и управления сессиями. В слое HagiCode.Libs мы уже реализовали его базовую функциональность, но эти компоненты всё ещё находятся в изолированном состоянии — как красивые жемчужины, ещё не нанизанные на нить. Пользователи не могут использовать его через выбор профессии Hero, цепочку выполнения сессий или панель мониторинга, что довольно жаль.

Проблема, с которой мы столкнулись: как поднять Reasonix до уровня Agent Provider первого уровня, равного Codex, Hermes и другим, реализуя полную маршрутизацию на бэкенде и отображение на фронтенде? Это не так просто, как зарегистрировать значение перечисления, — нужно построить полную цепочку от базовой абстракции до пользовательского интерфейса. Как при строительстве дома — нельзя просто заложить фундамент, нужно возвести стены и покрыть крышей.

Сложность этой интеграции в том, что Reasonix как локальный CLI-инструмент имеет свой характер и особенности. Например, он не требует строки подключения, все параметры конфигурируются пользователем во время выполнения; он может быть вообще не установлен, что требует элегантной обработки деградации; он совместим с моделями серии anthropic, но имеет свои уникальные ACP-параметры вроде effort, budget и других. Это как человек — свой уникальный подход к делам, к нему нельзя подойти грубо.

После тщательного архитектурного проектирования и нескольких раундов обсуждений мы в итоге采用了清晰ную трёхуровневую архитектуру, которая успешно интегрировала Reasonix в систему. Это решение не только решило текущую проблему, но и предоставило повторяемый шаблон для интеграции подобных CLI Provider в будущем. Многие вещи именно такие — как только найдёшь правильный метод, дальнейший путь становится намного проще.

## О HagiCode

Решение, которым мы делимся, основано на нашем практическом опыте в проекте [HagiCode](https://hagicode.com). HagiCode — это проект с открытым исходным кодом AI-ассистента для программирования, направленный на предоставление разработчикам мощных возможностей генерации, рефакторинга и оптимизации кода. В процессе разработки мы столкнулись с различными техническими вызовами, и интеграция Reasonix как Agent Provider первого уровня стала одним из них. Если вы считаете, что описанное нами решение имеет ценность, то это значит, что наша инженерная практика неплохая, и сам HagiCode заслуживает внимания.

## Основное содержание

### Проектирование технической архитектуры

Система использует чёткую трёхуровневую архитектуру для разделения ответственности, каждый уровень имеет чёткие границы обязанностей:

**Слой HagiCode.Libs**: Этот уровень уже завершён, он предоставляет абстракцию и конкретную реализацию CLI provider. Он определяет интерфейс `ICliProvider<ReasonixOptions>`, реализует `ReasonixProvider` для обработки ACP-стриминга и управления сессиями, а также поддерживает параметры вроде effort, budget, yolo, transcript и других. Обязанность этого уровня — предоставлять стабильную, повторяемую базовую функциональность, не затрагивая бизнес-логику. Как фундамент дома — хотя его не видно, он очень важен.

**Слой hagicode-core**: Это ключевой момент нашей текущей интеграции. Он отвечает за соединение базовой абстракции с унифицированным интерфейсом системы. Конкретная работа включает регистрацию значения перечисления `AIProviderType.ReasonixCli = 12`, создание `ReasonixCliProvider` в качестве thin adapter для соединения со слоем Libs, реализацию `ReasonixGrain` для обработки состояния сессии и потока выполнения, а также интеграцию с системой Hero для отображения параметров и управления конфигурацией. Суть этого уровня — координация компонентов и построение полной цепочки бизнес-логики. Как несущие стены дома, соединяющие все части.

**Слой web**: Отвечает за отображение пользователю и сбор конфигурации. Нам нужно заново сгенерировать типы OpenAPI для поддержки новых значений перечисления, реализовать визуальное отображение типов, чтобы Reasonix имел свою иконку и отображаемое имя, создать форму конфигурации параметров CLI, чтобы пользователи могли настраивать различные параметры, а также добавить поддержку многоязычности. Фокус этого уровня — пользовательский опыт и дизайн взаимодействия. Как отделка дома — качество напрямую влияет на комфорт проживания.

Такое слоистое проектирование позволяет каждому уровню концентрироваться на своих обязанностях, снижает сложность системы и упрощает последующее сопровождение и расширение. Во многих случаях чёткое разделение вещей упрощает их.

### Ключевые технические решения

В процессе реализации мы приняли несколько ключевых технических решений, которые оказали существенное влияние на финальную архитектуру и пользовательский опыт.

**Решение первое: Использование специализированного Grain**

Мы создали независимый `ReasonixGrain : IReasonixGrain, IExecutorStreamGrain`, вместо попытки повторного использования какого-то общего Grain. Это решение соответствует установленному образцу существующих 11 provider в системе. Хотя это может привести к некоторому дублированию кода, специализированный Grain позволяет нам проводить точный контроль для особенностей Reasonix, например, его уникального механизма привязки сессий и отображения ACP-сообщений. Мы также определили пустой DTO ответа `ReasonixResponse` в качестве идентификатора типа — хотя он не содержит фактических данных, он играет важную роль в системе типов. Как у каждого человека своя комната — даже если пустая, это своё пространство.

**Решение второе: Не создавать специализированный класс Settings**

В отличие от некоторых Provider, требующих строки подключения, вся конфигурация Reasonix задаётся пользователем во время выполнения и не требует проверки при запуске. Поэтому мы не создавали специализированный класс Settings, а хранили всю конфигурацию в словаре `AIProviderOptions.Providers[ReasonixCli].Settings`. Этот шаблон соответствует другим локальным CLI Provider таким как Qoder, Kiro, Kimi, упрощает структуру кода и избегает ненужных уровней абстракции. Поддерживаемые ключи настроек: `effort`, `budgetUsd`, `transcriptPath`, `enableYolo`, `arguments`, `startupTimeoutMs`, `reasoning`. Иногда простое — это лучше.

**Решение третье: Мониторинг здоровья стратегии Provider**

Reasonix — это CLI, устанавливаемый локально пользователем, он может быть вообще не установлен или отсутствовать в системном PATH. В таких случаях нам не следует напрямую выдавать ошибку, а нужно элегантно обработать деградацию. Мы используем стратегию `Provider`, проверяя доступность CLI через `CommandUtil.TryResolveExecutablePath`. Если проверка не проходит, UI отображает "недоступен", но это не влияет на другие части системы. Такой подход делает систему более надёжной и даёт пользователю чёткую обратную связь.毕竟 никто не хочет, чтобы вся система упала из-за маленькой проблемы.

**Решение четвёртое: Классификация экономической системы**

В системе HagiCode у разных Provider разные классификации экономической системы. Мы решили, чтобы Reasonix по умолчанию использовал классификацию `'claude'`, так как сам Reasonix совместим с моделями серии anthropic. В настоящее время только у Codex и Copilot есть специализированные классификации экономической системы, другие Provider повторно используют существующие классификации. Это сохраняет простоту системы и правильно обрабатывает биллинг и статистику затрат. Повторное использование — это тоже мудрость, не нужно начинать с нуля для всего.

**Решение пятое: Совместимость моделей**

Reasonix через флаг `--model` поддерживает различные модели, особенно серии anthropic. Мы добавили отображение совместимости в `secondary-professions.index.json`, что позволяет пользователям выбирать эти модели в Reasonix. Этот дизайн уважает возможности Reasonix и сохраняет согласованность системы — пользователи могут использовать различные модели, не понимая разницы на низком уровне. Пользователям тоже не легко, пусть всё будет проще.

**Реализация Thin Adapter**

`ReasonixCliProvider` — это ключевой компонент, соединяющий слой Libs с унифицированным интерфейсом системы:

```csharp
public sealed class ReasonixCliProvider : IAIProvider, IVersionedAIProvider, IAsyncDisposable
{
    private static readonly IReadOnlyList<string> SupportedSettingKeys =
    [
        "effort",
        "budgetUsd",
        "transcriptPath",
        "enableYolo",
        "arguments",
        "startupTimeoutMs",
        "reasoning"
    ];

    private readonly ICliProvider<ReasonixOptions> _provider;
    private readonly ConcurrentDictionary<string, string> _sessionBindings = new(StringComparer.Ordinal);

    public async IAsyncEnumerable<AIStreamingChunk> StreamCoreAsync(
        AIRequest request,
        string? sessionId = null,
        [EnumeratorCancellation] CancellationToken cancellationToken = default)
    {
        var options = BuildOptions(request, sessionId);
        await foreach (var message in _provider.StreamAsync(options, cancellationToken))
        {
            yield return MapToStreamingChunk(message);
        }
    }

    private ReasonixOptions BuildOptions(AIRequest request, string? sessionId)
    {
        return new ReasonixOptions
        {
            ExecutablePath = GetExecutablePath(),
            WorkingDirectory = GetWorkingDirectory(),
            Model = _config.Model,
            Effort = _config.Settings.GetValueOrDefault("effort", "medium"),
            Budget = _config.Settings.GetValueOrDefault("budgetUsd", 10.0),
            Yolo = _config.Settings.GetValueOrDefault("enableYolo", false),
            TranscriptPath = _config.Settings.GetValueOrDefault("transcriptPath"),
            Arguments = _config.Settings.GetValueOrDefault("arguments", ""),
            StartupTimeout = GetStartupTimeout(),
            EnvironmentVariables = _environmentVariables,
            CessionId = sessionId ?? GetCessionId()
        };
    }
};
```

Ключевые обязанности этого adapter:
1. Проверка параметров конфигурации, отклонение неподдерживаемых ключей настроек
2. Поддержка связей сессий, возможность восстановления сессий
3. Отображение ACP-сообщений в унифицированный формат системы
4. Использование `ProviderErrorAutoRetryCoordinator` для автоматического повторения

Как переводчик — переводит один язык на другой, при этом должен гарантировать точность смысла.

**Реализация Orleans Grain**

`ReasonixGrain` отвечает за обработку состояния сессии и потока выполнения:

```csharp
public class ReasonixGrain : Grain, IReasonixGrain, IExecutorStreamGrain
{
    private readonly Dictionary<string, ExecutorToolLifecycleStatus> _toolLifecycleState =
        new(StringComparer.Ordinal);

    public IAsyncEnumerable<ReasonixResponse> ExecuteCommandStreamAsync(
        string command,
        string? heroId = null,
        CancellationToken token = default,
        string? executionMessageId = null,
        string? systemMessage = null,
        Dictionary<string, string>? requestSettings = null)
    {
        var request = BuildRequest(command, isEdit: false, heroId, executionMessageId, systemMessage, requestSettings);
        return SendAsync(request, heroId, token);
    }

    private async IAsyncEnumerable<ReasonixResponse> SendAsync(
        AIRequest request,
        string? heroId,
        [EnumeratorCancellation] CancellationToken token)
    {
        _cancellationTokenSource = new CancellationTokenSource();
        var linkedToken = CancellationTokenSource.CreateLinkedTokenSource(token, _cancellationTokenSource.Token);

        var provider = await ResolveReasonixProviderAsync(heroId);

        await foreach (var chunk in provider.StreamAsync(request, linkedToken.Token))
        {
            var response = BuildChunkResponse(chunk);
            yield return response;
        }
    }

    private async Task<IAIProvider> ResolveReasonixProviderAsync(string? heroId)
    {
        // Логика откатов конфигурации с учётом Hero
        var config = await HeroProviderResolver.ResolveAsync(AIProviderType.ReasonixCli, heroId);
        return _aiProviderFactory.CreateProvider(AIProviderType.ReasonixCli, config);
    }
}
```

Основные функции Grain:
1. Использование `[PersistentState("reasonix-interop")]` для поддержания состояния сессии
2. Реализация логики откатов конфигурации с учётом Hero
3. Отслеживание состояния жизненного цикла инструментов
4. Поддержка цепочки токенов отмены, обеспечивающая своевременное прерывание выполнения

Как управляющий — всё упорядочено и чётко.

### Лучшие практики и внимание

В процессе реализации мы суммировали несколько лучших практик и моментов, требующих внимания.

**Проверка параметров**

ReasonixCliProvider должен строго проверять параметры конфигурации и отклонять неподдерживаемые ключи настроек:

```csharp
public void ValidateConfigurationOverrides(Dictionary<string, string?> overrides)
{
    foreach (var key in overrides.Keys)
    {
        if (!SupportedSettingKeys.Contains(key))
        {
            throw new HeroProviderConfigurationException(
                $"Unsupported setting key '{key}' for Reasonix provider");
        }
    }
}
```

这样可以防止用户配置错误的参数，避免运行时错误。 Как вратарь — нельзя пропускать лишнее.

**Управление привязкой сессий**

Использование `ConcurrentDictionary` для управления привязками сессий, поддержка восстановления сессий:

```csharp
_sessionBindings[cessionId] = sessionId;

// В последующих запросах привязка существующей сессии
if (_sessionBindings.TryGetValue(cessionId, out var boundSessionId))
{
    options.SessionId = boundSessionId;
}
```

Этот дизайн позволяет пользователям восстанавливать предыдущие сессии после прерывания, обеспечивая лучший опыт. Как запоминание истории — можно продолжить рассказ в следующий раз.

**Элегантная обработка деградации**

Фронтенд должен проверять доступность CLI и давать дружелюбные подсказки:

```typescript
const reasonixAvailable = await healthApi.checkCliAvailable('reasonix');
if (!reasonixAvailable) {
  showMessage('Reasonix CLI не установлен, пожалуйста настройте его в системном PATH');
}
```

Не стоит заставлять пользователей сталкиваться с непонятными ошибками — заранее проверяйте и давайте чёткие инструкции.毕竟 никто не хочет быть внезапно застрявшим.

**Покрытие тестами**
