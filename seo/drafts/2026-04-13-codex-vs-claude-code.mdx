如果你只想先得到一个可执行的结论，可以这样看：在 HagiCode 当前文档语境里，**Codex 和 Claude Code 都属于当前支持且可走一键安装的 Agent CLI**。如果你想沿着现有默认引导先跑通工作流，Claude Code 更容易对照当前向导示例；如果你更关心把一个 CLI 放进多 Agent 分工里的“精准代码修改”位置，仓库里的历史实践文章给 Codex 的定位会更清晰。

但这篇文章只讨论 **HagiCode 仓库已经写明的事实**，不做泛互联网层面的输赢判断。所以更准确的说法不是“谁更强”，而是“你现在更适合先落地哪一个”。

## 这篇对比基于哪些仓库事实

先看当前公开文档里能确认的基础盘：

- [安装指南](/installation) 明确把 Claude Code 和 Codex 都列进当前活跃支持范围
- [AI Agent CLI 安装](/related-software-installation/ai-agent-cli/) 把两者都标为 `当前支持` 和 `一键安装`
- [初始化向导页面](/quick-start/wizard-setup) 的默认示例使用 Claude Code，但同页也明确说明这只是默认示例，不是唯一支持项

也就是说，**当前支持状态** 这件事上，两者并没有谁在文档里被降级处理。真正能拉开差异的，是当前文档里对上手路径、工作流角色和容器化使用的描述方式。

## 当前文档里能确认的关键差别

### 1. 上手入口的呈现方式不同

在 [AI Agent CLI 安装](/related-software-installation/ai-agent-cli/) 里，Claude Code 和 Codex 都是标准入口，差别并不大，都是当前支持、都可走一键安装。

但在 [初始化向导页面](/quick-start/wizard-setup) 中，默认 Agent 示例选的是 `Claude Code` 与 `GLM 5.1`。这会带来一个很现实的体验差异：如果你是第一次接触 HagiCode，想尽量照着现有截图和默认流程理解系统，Claude Code 会更顺手一些。

这里必须补一句限制条件：**默认示例不等于唯一推荐。** 同一页已经明确写了，Codex 仍然属于当前活跃支持范围。

### 2. 历史实践文章给两者分工不同

如果你把视角从“安装入口”切到“多 Agent 协作”，仓库里已有更具体的角色描述。

在 [打造 AI 冒险团：HagiCode 多 Agent 协作配置实战](/blog/2026-03-17-hagicode-ai-agent-party) 这篇文章里，作者先加了一条时间语境说明：这篇内容记录的是 2026-03-17 当时的实践，当前支持口径仍应以安装页为准。基于这个前提，文中给出的分工是：

- Claude Code：更适合生成技术方案和 proposal
- Codex：更适合执行精准的代码修改

同一篇文章还解释了背后的理由：Claude Code 更偏向复杂需求理解和前期分析，Codex 更偏向具体实现与精确修改。

这组信息对“怎么选”非常有用，但也要读对：它是**历史实践中的工作流偏好**，不是一条永久不变的官方硬规则。

### 3. 容器化使用时，两者的运维关注点不同

如果你的目标不是本地先试，而是准备放进 Docker 或团队环境，那么差别会更具体。

在 [Docker 容器中运行 AI CLI 工具：用户隔离与持久化卷实战指南](/blog/2026-03-26-docker-ai-cli-user-isolation-guide) 里，仓库已经写明：Claude Code、Codex、OpenCode 这类工具都需要把各自配置目录持久化，不然容器重启后配置会丢失。

文中给出的目录是：

| CLI | 容器内配置目录 | 建议卷名 |
| --- | --- | --- |
| Claude Code | `/home/hagicode/.claude` | `claude-data` |
| Codex | `/home/hagicode/.codex` | `codex-data` |

这意味着在容器化部署里，你不能把它们简单看成“差不多的同类项”。至少从配置持久化角度看，它们就需要分别管理。

此外，这篇 Docker 文章还明确举例说明了 **Claude CLI 在容器里不能直接以 root 运行**。对 Codex，文档重点落在配置持久化目录，而不是 root 限制本身。所以如果你的第一落地场景就是容器，Claude Code 这边的运行用户和权限处理需要更早纳入设计。

### 4. 模型评测能提供参考，但不能直接代替 CLI 对比

仓库里还有一页 [Hagicode 中使用模型的对比和评价](/llm-guide/model-comparison-evaluation)。这里面给出了团队在真实任务中对 GPT 5.3 Codex、GPT 5.4、GLM 5、GLM 4.7、MiniMax M2.5 等模型的感性评估。

从这页能确认的事实是：

- GPT 5.3 Codex 与 GPT 5.4 在完成任务质量和工程输出质量上评价较高
- 文中明确说这是一份模型层面的项目内实测总结

所以这页能帮助你建立一个判断：如果你的选择会强依赖 Codex 相关模型能力，仓库内部的模型评价对它是偏正面的。但它**不能直接证明 Codex CLI 的所有体验都优于 Claude Code**，因为这页评测的对象本来就不是 CLI 产品本身。

## 按场景选择，哪一种更适合你

### 如果你想先沿着当前默认引导把系统跑通

优先看 Claude Code 会更自然。

理由不是“它比 Codex 更高级”，而是因为当前文档中的默认示例和向导截图都直接使用 Claude Code。对于第一次接触 HagiCode 的人，这能减少“文档说的是 A，我实际装的是 B”的理解偏差。

适合这类人：

- 第一次接触 HagiCode
- 更在意先跑通默认路径
- 希望跟着现有向导页面理解 Agent 配置逻辑

### 如果你已经有前置方案流程，只差一个执行修改工具

优先考虑 Codex 会更贴近仓库里的历史实践分工。

原因很直接：在那篇多 Agent 协作文章里，Codex 被放在“执行精准代码修改”的位置。这种定位对于已经把“需求理解”和“方案生成”交给别的 Agent 或别的步骤的人，会比较有吸引力。

适合这类人：

- 已经有 proposal 或方案产出链路
- 更看重精准代码修改
- 准备把不同 CLI 拆成明确分工，而不是只选一个包打天下

### 如果你准备直接上 Docker 或团队环境

这时不要先问“谁更强”，而要先问“谁更省运维成本”。

从仓库现有文章看，更实用的判断是：

- 两者都需要各自独立的持久化目录
- Claude Code 还要更早处理非 root 运行约束
- Codex 至少要把 `.codex` 目录单独保留下来

如果你的团队已经有现成的非 root 容器规范，Claude Code 的额外约束未必是问题；如果你现在只是想先把配置和数据路径管理清楚，那把 `.claude` 和 `.codex` 分开设计会比争论“先装谁”更重要。

## 一个保守但实用的决策方法

如果你现在还拿不准，可以按这个顺序判断：

1. 你是不是想先跟着当前默认文档路径学习？如果是，先上 Claude Code。
2. 你是不是已经有提案或方案链路，只需要一个执行修改的 CLI？如果是，优先试 Codex。
3. 你是不是准备直接进 Docker 或多人协作环境？如果是，先把权限、持久化卷和配置目录方案设计清楚，再决定先落地哪个。

如果以上三步你都看完还是不确定，一个更稳妥的路线是：**先按默认示例落地一个，再把另一个补成分工角色。** 这也更符合仓库现有多 Agent 文章的思路，而不是强迫自己在第一天就做“唯一正确选择”。

## FAQ

### Claude Code 是默认示例，是否代表 Codex 不推荐？

不是。

当前仓库页面已经明确写了，默认示例只是默认示例，不是唯一支持项。Codex 仍然在当前活跃支持范围内，并且同样被列为一键安装入口。

### Codex 和 Claude Code 现在都属于当前支持范围吗？

是。

[安装指南](/installation) 和 [AI Agent CLI 安装](/related-software-installation/ai-agent-cli/) 都把两者列在当前支持范围内。

### 在 Docker 场景里，两者都需要独立持久化目录吗？

需要。

仓库里的容器化文章已经明确给出不同目录：Claude Code 用 `/home/hagicode/.claude`，Codex 用 `/home/hagicode/.codex`。

### GPT 5.3 Codex 的高评分，能直接说明 Codex CLI 更强吗？

不能直接这样下结论。

那一页评测讨论的是模型在 HagiCode 真实任务里的表现，不是完整 CLI 产品体验。它可以提供参考信号，但不能替代 CLI 对比本身。
